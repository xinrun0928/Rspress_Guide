## 一、为什么需要策略模式

### if-else 的问题

```java
public class OrderProcessor {
    public double calculateDiscount(Order order) {
        if ("VIP".equals(order.getType())) {
            return order.getAmount() * 0.8; // VIP 8折
        } else if ("SVIP".equals(order.getType())) {
            return order.getAmount() * 0.7; // SVIP 7折
        } else if ("NEW_USER".equals(order.getType())) {
            return order.getAmount() * 0.9; // 新用户 9折
        } else if ("FLASH_SALE".equals(order.getType())) {
            return order.getAmount() * 0.5; // 秒杀 5折
        } else {
            return order.getAmount(); // 普通用户不打折
        }
    }
}
```

**问题**：
- 新增折扣类型需要修改代码
- 违反开闭原则
- 代码臃肿，难以维护
- 难以单独测试每种折扣算法

## 二、策略模式实现

### 核心结构

```java
// 策略接口
public interface DiscountStrategy {
    double calculate(Order order);
}

// 具体策略实现
public class VipDiscount implements DiscountStrategy {
    @Override
    public double calculate(Order order) {
        return order.getAmount() * 0.8;
    }
}

public class SvipDiscount implements DiscountStrategy {
    @Override
    public double calculate(Order order) {
        return order.getAmount() * 0.7;
    }
}

public class NewUserDiscount implements DiscountStrategy {
    @Override
    public double calculate(Order order) {
        return order.getAmount() * 0.9;
    }
}

public class FlashSaleDiscount implements DiscountStrategy {
    @Override
    public double calculate(Order order) {
        return order.getAmount() * 0.5;
    }
}

public class NoDiscount implements DiscountStrategy {
    @Override
    public double calculate(Order order) {
        return order.getAmount();
    }
}
```

### 使用策略模式

```java
public class OrderProcessor {
    private DiscountStrategy discountStrategy;

    // 注入策略
    public void setDiscountStrategy(DiscountStrategy discountStrategy) {
        this.discountStrategy = discountStrategy;
    }

    public double calculateDiscount(Order order) {
        return discountStrategy.calculate(order);
    }
}

// 使用
public class Client {
    public static void main(String[] args) {
        OrderProcessor processor = new OrderProcessor();

        Order vipOrder = new Order("VIP", 1000);
        processor.setDiscountStrategy(new VipDiscount());
        System.out.println("VIP订单折扣: " + processor.calculateDiscount(vipOrder));

        Order flashOrder = new Order("FLASH_SALE", 1000);
        processor.setDiscountStrategy(new FlashSaleDiscount());
        System.out.println("秒杀订单折扣: " + processor.calculateDiscount(flashOrder));
    }
}
```

## 三、策略模式 + 工厂模式

### 工厂管理策略

```java
public class DiscountStrategyFactory {
    private static final Map<String, DiscountStrategy> STRATEGIES = new HashMap<>();

    static {
        STRATEGIES.put("VIP", new VipDiscount());
        STRATEGIES.put("SVIP", new SvipDiscount());
        STRATEGIES.put("NEW_USER", new NewUserDiscount());
        STRATEGIES.put("FLASH_SALE", new FlashSaleDiscount());
        STRATEGIES.put("NORMAL", new NoDiscount());
    }

    public static DiscountStrategy getStrategy(String type) {
        DiscountStrategy strategy = STRATEGIES.get(type);
        if (strategy == null) {
            return new NoDiscount();
        }
        return strategy;
    }
}
```

### 简化后的使用

```java
public class OrderProcessor {
    public double calculateDiscount(Order order) {
        // 工厂获取策略，策略执行计算
        DiscountStrategy strategy = DiscountStrategyFactory.getStrategy(order.getType());
        return strategy.calculate(order);
    }
}
```

### 结构对比

```
不使用策略模式：
    OrderProcessor
         │
         └── calculateDiscount()
              └── if-else 硬编码

使用策略模式：
    OrderProcessor
         │
         ├── StrategyFactory (获取策略)
         │      └── Map<type, Strategy>
         │
         └── DiscountStrategy (执行策略)
                ├── VipDiscount
                ├── SvipDiscount
                └── FlashSaleDiscount
```

## 四、策略模式的优点

| 优点 | 说明 |
|------|------|
| 符合开闭原则 | 新增策略不修改现有代码 |
| 消除 if-else | 策略由类表示，可独立管理 |
| 提高复用性 | 策略可以在多个场景复用 |
| 易于测试 | 每个策略可单独测试 |

---

## 五、策略模式 vs 工厂模式

| 对比 | 策略模式 | 工厂模式 |
|------|----------|----------|
| 目的 | 选择算法/行为 | 创建对象 |
| 关注点 | what to do | how to create |
| 调用时机 | 运行时动态选择 | 通常编译时确定 |
| 组合使用 | 工厂负责获取策略 | 策略负责执行逻辑 |

## 六、Spring 整合策略模式

### 使用 @Autowired 注入

```java
@Service
public class OrderService {
    private final Map<String, DiscountStrategy> strategyMap;

    @Autowired
    public OrderService(Map<String, DiscountStrategy> strategyMap) {
        this.strategyMap = strategyMap;
    }

    public double calculateDiscount(Order order) {
        DiscountStrategy strategy = strategyMap.get(order.getType());
        if (strategy == null) {
            strategy = strategyMap.get("NORMAL");
        }
        return strategy.calculate(order);
    }
}
```

### 使用 @PostConstruct 初始化

```java
@Service
public class PaymentService {
    private final Map<String, PaymentStrategy> paymentStrategies = new HashMap<>();

    @Autowired
    private List<PaymentStrategy> strategyList;

    @PostConstruct
    public void init() {
        strategyList.forEach(s -> paymentStrategies.put(s.getType(), s));
    }

    public void pay(Order order) {
        PaymentStrategy strategy = paymentStrategies.get(order.getPaymentType());
        if (strategy == null) {
            throw new IllegalArgumentException("不支持的支付方式");
        }
        strategy.pay(order);
    }
}
```

## 七、Lambda 表达式优化

### JDK 8+ 写法

```java
public class DiscountStrategyFactory {
    private static final Map<String, DiscountStrategy> STRATEGIES = new HashMap<>();

    static {
        STRATEGIES.put("VIP", order -> order.getAmount() * 0.8);
        STRATEGIES.put("SVIP", order -> order.getAmount() * 0.7);
        STRATEGIES.put("NEW_USER", order -> order.getAmount() * 0.9);
        STRATEGIES.put("FLASH_SALE", order -> order.getAmount() * 0.5);
        STRATEGIES.put("NORMAL", order -> order.getAmount());
    }

    public static DiscountStrategy getStrategy(String type) {
        return STRATEGIES.getOrDefault(type, order -> order.getAmount());
    }
}
```

## 十、常见面试陷阱

> **⚠️ 陷阱 1**：策略模式会产生大量类
>
> 确实如此，但这些类职责单一、易于测试和维护。

> **⚠️ 陷阱 2**：策略模式适合短期项目
>
> 相反，策略模式更适合长期维护的项目，因为新增策略不需要修改现有代码。
