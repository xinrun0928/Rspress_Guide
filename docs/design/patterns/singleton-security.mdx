## 一、序列化破坏单例

### 问题演示

```java
public class SerializableSingleton implements Serializable {
    private static final long serialVersionUID = 1L;
    private static final SerializableSingleton INSTANCE = new SerializableSingleton();

    private SerializableSingleton() {}

    public static SerializableSingleton getInstance() {
        return INSTANCE;
    }
}

// 测试序列化
@Test
public void testSerialization() throws Exception {
    SerializableSingleton s1 = SerializableSingleton.getInstance();

    // 序列化
    ByteArrayOutputStream baos = new ByteArrayOutputStream();
    ObjectOutputStream oos = new ObjectOutputStream(baos);
    oos.writeObject(s1);
    byte[] data = baos.toByteArray();

    // 反序列化
    ByteArrayInputStream bais = new ByteArrayInputStream(data);
    ObjectInputStream ois = new ObjectInputStream(bais);
    SerializableSingleton s2 = (SerializableSingleton) ois.readObject();

    System.out.println(s1 == s2); // false ❌ 单例被破坏！
}
```

### 原因分析

反序列化时，Java 通过反射调用 `ObjectInputStream` 的 `readObject` 方法创建新对象：

```
反序列化流程：
1. ObjectInputStream.readObject()
2. 调用 ObjectStreamClass.lookup()
3. 通过反射创建新对象
4. 填充字段值
5. 返回新对象实例
```

### 解决方案：readResolve 方法

```java
public class SafeSingleton implements Serializable {
    private static final long serialVersionUID = 1L;
    private static final SafeSingleton INSTANCE = new SafeSingleton();

    private SafeSingleton() {}

    public static SafeSingleton getInstance() {
        return INSTANCE;
    }

    /**
     * 反序列化时返回单例，而不是创建新对象
     * Java 会在反序列化完成后调用此方法
     */
    private Object readResolve() {
        return INSTANCE;
    }
}
```

### readResolve 原理

```java
// ObjectInputStream 源码简化版
public class ObjectInputStream {
    public Object readObject() {
        // 1. 读取类信息
        ObjectStreamClass desc = readClassDesc();

        // 2. 创建新对象（通过反射）
        Object obj = desc.forClass().newInstance(); // 创建新实例

        // 3. 调用 readResolve（如果存在）
        if (obj instanceof Serializable && obj.getClass().isAnonymousClass()) {
            // ...
        }

        // 4. 检查是否有 readResolve 方法
        Method readResolve = desc.getReadResolveMethod();
        if (readResolve != null) {
            return readResolve.invoke(obj); // 返回 readResolve 的返回值
        }

        return obj;
    }
}
```

## 二、反射攻击与防护

### 攻击方式

```java
public class VulnerableSingleton {
    private static VulnerableSingleton INSTANCE;

    private VulnerableSingleton() {}

    public static VulnerableSingleton getInstance() {
        if (INSTANCE == null) {
            INSTANCE = new VulnerableSingleton();
        }
        return INSTANCE;
    }
}

// 攻击代码
@Test
public void testReflectionAttack() throws Exception {
    Constructor<VulnerableSingleton> constructor =
        VulnerableSingleton.class.getDeclaredConstructor();
    constructor.setAccessible(true); // 突破私有访问权限

    VulnerableSingleton s1 = constructor.newInstance();
    VulnerableSingleton s2 = constructor.newInstance();

    System.out.println(s1 == s2); // false ❌ 单例被破坏！
}
```

### 攻击原理

```
反射攻击流程：
1. 获取私有构造函数（getDeclaredConstructor）
2. 设置可访问标志（setAccessible(true)）
3. 调用构造函数创建新实例（newInstance）
4. 得到新的对象引用
```

### 防护措施

#### 方法一：构造函数检查

```java
public class ProtectedSingleton {
    private static volatile ProtectedSingleton INSTANCE;

    private ProtectedSingleton() {
        // 关键：在构造函数中检查是否已创建实例
        synchronized (ProtectedSingleton.class) {
            if (INSTANCE != null) {
                throw new RuntimeException("单例已被创建，禁止通过反射创建新实例！");
            }
        }
    }

    public static ProtectedSingleton getInstance() {
        if (INSTANCE == null) {
            synchronized (ProtectedSingleton.class) {
                if (INSTANCE == null) {
                    INSTANCE = new ProtectedSingleton();
                }
            }
        }
        return INSTANCE;
    }
}
```

#### 方法二：双重检查 + 防护

```java
public class DoubleProtectedSingleton {
    private static volatile DoubleProtectedSingleton INSTANCE;

    private DoubleProtectedSingleton() {
        // 防止通过反射多次创建
        synchronized (DoubleProtectedSingleton.class) {
            if (INSTANCE != null) {
                throw new RuntimeException("单例已被创建，禁止反射攻击！");
            }
        }
    }

    public static DoubleProtectedSingleton getInstance() {
        if (INSTANCE == null) {
            synchronized (DoubleProtectedSingleton.class) {
                if (INSTANCE == null) {
                    INSTANCE = new DoubleProtectedSingleton();
                }
            }
        }
        return INSTANCE;
    }
}
```

#### 方法三：使用 Enum

```java
public enum EnumSingleton {
    INSTANCE;

    private final SomeService service;

    EnumSingleton() {
        this.service = new SomeService();
    }

    public SomeService getService() {
        return service;
    }
}

// 尝试通过反射创建（会失败）
@Test
public void testEnumReflection() throws Exception {
    Constructor<EnumSingleton> constructor =
        EnumSingleton.class.getDeclaredConstructor();
    // setAccessible(true) 后调用 newInstance() 会抛出异常
    // UnsupportedOperationException: Cannot reflectively create enum objects
}
```

## 三、克隆破坏单例

### 问题演示

```java
public class CloneableSingleton implements Cloneable, Serializable {
    private static final CloneableSingleton INSTANCE = new CloneableSingleton();

    private CloneableSingleton() {}

    public static CloneableSingleton getInstance() {
        return INSTANCE;
    }

    @Override
    protected Object clone() throws CloneNotSupportedException {
        return super.clone(); // 返回新对象
    }
}

@Test
public void testCloneAttack() throws Exception {
    CloneableSingleton s1 = CloneableSingleton.getInstance();
    CloneableSingleton s2 = (CloneableSingleton) s1.clone();

    System.out.println(s1 == s2); // false ❌ 单例被破坏！
}
```

### 防护措施

```java
public class SafeCloneableSingleton implements Cloneable, Serializable {
    private static final SafeCloneableSingleton INSTANCE = new SafeCloneableSingleton();

    private SafeCloneableSingleton() {}

    public static SafeCloneableSingleton getInstance() {
        return INSTANCE;
    }

    @Override
    protected Object clone() throws CloneNotSupportedException {
        // 返回单例，而不是创建新对象
        return INSTANCE;
    }

    private Object readResolve() {
        return INSTANCE;
    }
}
```

## 四、多种攻击方式汇总

| 攻击方式 | 破坏原理 | 防护措施 |
|----------|----------|----------|
| 序列化 | 反序列化创建新对象 | `readResolve()` 返回单例 |
| 反射 | 通过反射调用私有构造函数 | 构造函数中检查 `INSTANCE != null` |
| 克隆 | `clone()` 方法创建新对象 | 重写 `clone()` 返回单例 |
| 反序列化 | 同序列化 | 同上 |

## 五、完美单例实现

### 推荐方案一：登记式单例（静态内部类增强）

```java
public class RegistrySingleton implements Serializable {
    private static final long serialVersionUID = 1L;

    private RegistrySingleton() {
        // 反射防护
        if (getInstance() != null) {
            throw new RuntimeException("单例已存在，禁止反射攻击！");
        }
    }

    private static class Holder {
        private static final RegistrySingleton INSTANCE = new RegistrySingleton();
    }

    public static RegistrySingleton getInstance() {
        return Holder.INSTANCE;
    }

    // 序列化安全
    private Object readResolve() {
        return getInstance();
    }
}
```

### 推荐方案二：ThreadLocal 单例（线程唯一）

```java
public class ThreadLocalSingleton {
    private static final ThreadLocal<ThreadLocalSingleton> INSTANCE =
        ThreadLocal.withInitial(ThreadLocalSingleton::new);

    private ThreadLocalSingleton() {}

    public static ThreadLocalSingleton getInstance() {
        return INSTANCE.get(); // 每个线程获取到的是该线程的唯一实例
    }
}
```

### 推荐方案三：枚举单例（最佳实践）

```java
public enum PerfectSingleton {
    INSTANCE;

    private final Connection connection;
    private final SomeService service;

    PerfectSingleton() {
        // 初始化资源
        this.connection = createConnection();
        this.service = new SomeService();
    }

    private Connection createConnection() {
        // 初始化数据库连接
        return null;
    }

    public Connection getConnection() {
        return connection;
    }

    public SomeService getService() {
        return service;
    }
}
```

## 六、线程安全的深层理解

### 双重检查锁定详解

```java
public class DCLSingleton {
    private static volatile DCLSingleton INSTANCE;

    private DCLSingleton() {}

    public static DCLSingleton getInstance() {
        // 第一次检查：避免不必要的同步
        // 大部分情况下 INSTANCE 已创建，无需进入 synchronized 块
        if (INSTANCE == null) {
            synchronized (DCLSingleton.class) {
                // 进入同步块后再次检查
                // 防止多个线程同时进入 synchronized 块
                if (INSTANCE == null) {
                    INSTANCE = new DCLSingleton();
                }
            }
        }
        return INSTANCE;
    }
}
```

### volatile 的两个作用

1. **防止指令重排序**：保证 `new DCLSingleton()` 的三步按正确顺序执行
2. **保证可见性**：一个线程对 `INSTANCE` 的写入对其他线程可见

### 无 volatile 的问题

```java
// 如果没有 volatile，指令重排序可能导致以下场景：
INSTANCE = new DCLSingleton();
// 可能被重排序为：
1. 分配内存
2. 赋值给 INSTANCE（此时 INSTANCE != null，但对象未构造）
3. 调用构造函数

// 线程 B 可能看到非 null 但未构造完成的 INSTANCE
```

## 八、与 Spring 的结合

### Spring 如何保证单例 Bean 安全

```java
// Spring 默认单例模式
@Component
public class UserService {
    private int counter = 0;

    public void increment() {
        counter++;
    }
}

// 问题是：counter 是实例变量，多线程访问会有问题！
// Spring 保证的是「单例 Bean」，不是「线程安全」

// 解决方案：
// 1. 使用 ThreadLocal
// 2. 使用无状态 Bean
// 3. 显式加锁
```

### Spring 的单例陷阱

```java
@Component
public class StatefulService {
    private int count = 0; // ⚠️ 危险！这个变量是共享的

    public void increment() {
        count++; // ⚠️ 非原子操作，有线程安全问题
    }
}

// 如果期望每个请求独立计数，应该：
// 1. 使用方法参数传递
// 2. 使用 ThreadLocal
// 3. 使用请求作用域 Bean
```

## 九、实战代码模板

```java
/**
 * 线程安全、防序列化、防反射的最优单例实现
 * 推荐在生产环境使用
 */
public final class BestSingleton implements Serializable {
    private static final long serialVersionUID = 1L;

    private BestSingleton() {
        // 反射防护
        if (INSTANCE != null) {
            throw new IllegalStateException("单例已存在！");
        }
    }

    private static volatile BestSingleton INSTANCE;

    public static BestSingleton getInstance() {
        if (INSTANCE == null) {
            synchronized (BestSingleton.class) {
                if (INSTANCE == null) {
                    INSTANCE = new BestSingleton();
                }
            }
        }
        return INSTANCE;
    }

    // 保证序列化安全
    private Object readResolve() {
        return getInstance();
    }
}
```

## 十、常见面试陷阱

> **⚠️ 陷阱 1**：认为 volatile 只是为了性能优化
>
> 很多人只知道「DCL 加 volatile 是为了避免不必要的同步」，却不知道 volatile 的核心作用是「禁止指令重排序」。

> **⚠️ 陷阱 2**：忽略 clone 和反序列化
>
> 面试中问到单例安全，只说「加 synchronized」或「加 volatile」，忽略了对序列化、克隆、反射的讨论。

> **⚠️ 陷阱 3**：过度设计
>
> 不是所有场景都需要「完美单例」。对于内部工具类，使用简单的饿汉式即可。
