## 一、单一职责原则（SRP）

**定义**：一个类应该只有一个引起它变化的原因。

### 为什么要单一职责？

假设你写了一个 `UserService`，里面既包含用户CRUD，又包含日志记录，还包含权限校验：

```java title="违反SRP的示例"
public class UserService {
    // 用户管理
    public void saveUser(User user) { ... }
    public void deleteUser(Long id) { ... }

    // 日志记录
    public void log(String message) { ... }

    // 权限校验
    public boolean hasPermission(Long userId, String action) { ... }
}
```

**问题**：
- 用户管理逻辑变化时，可能影响到日志记录代码
- 测试用例需要同时覆盖三种功能
- 难以复用用户管理代码（因为它和日志、权限耦合）

### 正确的做法

```java title="符合SRP的示例"
// 纯粹的领域对象
public class User {
    private Long id;
    private String name;
    // getter/setter
}

// 单一职责：用户服务
public class UserService {
    public void saveUser(User user) { ... }
    public void deleteUser(Long id) { ... }
}

// 单一职责：日志服务
public class Logger {
    public void log(String message) { ... }
}

// 单一职责：权限服务
public class PermissionService {
    public boolean hasPermission(Long userId, String action) { ... }
}
```

### 面试追问

> **第一层**：什么是单一职责原则？
>
> **第二层**：如何判断一个类是否违反了SRP？
>
> **第三层**：SRP 和微服务拆分有什么关系？

**💡 加分回答**：SRP 的本质是「变化的原因」，不是「功能数量」。一个类有多个 public 方法不一定是违反 SRP，关键看这些方法是否会因为同一个原因变化。

## 二、开闭原则（OCP）

**定义**：软件实体应该对扩展开放，对修改关闭。

### 核心思想

不修改现有代码，通过继承、组合、接口实现等方式扩展新功能。

```java title="违反OCP：每次加新图形都要修改计算器"
public class AreaCalculator {
    public double calculate(Object shape) {
        if (shape instanceof Circle) {
            // 计算圆形面积
        } else if (shape instanceof Rectangle) {
            // 计算矩形面积
        } else if (shape instanceof Triangle) {
            // 计算三角形面积
        }
        // 每次加新图形都要改这里！
    }
}
```

```java title="符合OCP：使用多态扩展新图形"
public interface Shape {
    double area();
}

public class Circle implements Shape {
    private double radius;
    @Override
    public double area() { return Math.PI * radius * radius; }
}

public class Rectangle implements Shape {
    private double width, height;
    @Override
    public double area() { return width * height; }
}

// 新增三角形？只需添加新类，无需修改 calculate 方法
public class Triangle implements Shape {
    private double base, height;
    @Override
    public double area() { return 0.5 * base * height; }
}

public class AreaCalculator {
    public double calculate(Shape shape) {
        return shape.area(); // 对扩展开放，对修改关闭
    }
}
```

### 面试追问

> **第一层**：开闭原则的「开」和「闭」分别指什么？
>
> **第二层**：如何做到对扩展开放、对修改关闭？
>
> **第三层**：OCP 和策略模式有什么关系？

**💡 加分回答**：OCP 是所有设计模式的最终目标。装饰器模式、策略模式、模板方法模式都是 OCP 的具体实现。

## 三、里氏替换原则（LSP）

**定义**：子类可以扩展父类的功能，但不能改变原有父类函数的行为。

### 两个关键规则

1. **子类重写方法时，参数要"放大"**（使用更宽泛的类型）
2. **子类的方法返回值要"缩小"或相同"**（返回更具体的类型）

### 违反LSP的例子

```java title="违反LSP：子类改变了父类行为"
public class Rectangle {
    protected double width;
    protected double height;

    public void setWidth(double width) { this.width = width; }
    public void setHeight(double height) { this.height = height; }
}

public class Square extends Rectangle {
    // 正方形的长和宽必须相等
    @Override
    public void setWidth(double side) {
        this.width = side;
        this.height = side;
    }

    @Override
    public void setHeight(double side) {
        this.width = side;
        this.height = side;
    }
}

// 测试代码
public class Test {
    public static void resize(Rectangle r, double width, double height) {
        r.setWidth(width);
        r.setHeight(height);
        // 期望：width 和 height 被分别设置
        // 但如果传入 Square，长宽相等，行为变了！
    }
}
```

### 正确的设计

```java title="符合LSP：正方形不应该继承 Rectangle"
public interface Shape {
    double area();
}

public class Rectangle implements Shape {
    private double width, height;
    public void setWidth(double w) { this.width = w; }
    public void setHeight(double h) { this.height = h; }
    @Override
    public double area() { return width * height; }
}

public class Square implements Shape {
    private double side;
    public void setSide(double s) { this.side = s; }
    @Override
    public double area() { return side * side; }
}
```

### 面试追问

> **第一层**：里氏替换原则的「替换」是什么意思？
>
> **第二层**：为什么说「正方形不是矩形」？
>
> **第三层**：LSP 和多态有什么区别？

**💡 加分回答**：LSP 是 SOLID 原则中最容易被忽略的一个。违反了 LSP，往往意味着你的继承关系设计有问题。

## 四、接口隔离原则（ISP）

**定义**：客户端不应该依赖它不需要的接口。

### 违反ISP的例子

```java title="胖接口导致实现类被迫实现不需要的方法"
public interface Animal {
    void fly();      // 鸟需要，但狗不需要
    void swim();     // 鱼需要，但鸟不需要
    void run();      // 狗需要，但鱼不需要
}

public class Dog implements Animal {
    @Override
    public void fly() { throw new UnsupportedOperationException(); } // 尴尬
    @Override
    public void swim() { throw new UnsupportedOperationException(); }
    @Override
    public void run() { /* 真正需要的 */ }
}
```

### 正确的设计：接口按职责拆分

```java title="按职责拆分的细粒度接口"
public interface Flyable {
    void fly();
}

public interface Swimmable {
    void swim();
}

public interface Runnable {
    void run();
}

// 鸟只需要飞
public class Bird implements Flyable {
    @Override
    public void fly() { /* 飞 */ }
}

// 狗只需要跑
public class Dog implements Runnable {
    @Override
    public void run() { /* 跑 */ }
}

// 鸭子既会飞又会游
public class Duck implements Flyable, Swimmable {
    @Override
    public void fly() { /* 飞 */ }
    @Override
    public void swim() { /* 游 */ }
}
```

### 面试追问

> **第一层**：接口隔离原则和单一职责原则有什么区别？
>
> **第二层**：为什么说 Spring 中的接口都遵循了 ISP？
>
> **第三层**：如何判断接口是否太"胖"？

**💡 加分回答**：ISP 和 SRP 本质相同，区别在于 SRP 约束的是类，ISP 约束的是接口。接口应该「小而专」，而不是「大而全」。

## 五、依赖倒置原则（DIP）

**定义**：
- 高层模块不应该依赖低层模块，两者都应该依赖抽象
- 抽象不应该依赖细节，细节应该依赖抽象

### 违反DIP的例子

```java title="高层模块直接依赖低层模块"
public class OrderService {
    private MySQLRepository repository = new MySQLRepository(); // 直接依赖具体实现

    public void save(Order order) {
        repository.save(order);
    }
}

public class MySQLRepository {
    public void save(Order order) { /* MySQL 存储 */ }
}
```

**问题**：
- 要换数据库？改代码
- 无法单独测试 OrderService（强依赖 MySQL）
- 违反开闭原则

### 符合DIP的例子

```java title="依赖倒置：通过接口解耦"
public interface OrderRepository {
    void save(Order order);
}

public class OrderService {
    private OrderRepository repository; // 依赖抽象

    public OrderService(OrderRepository repository) {
        this.repository = repository;
    }

    public void save(Order order) {
        repository.save(order);
    }
}

// 低层实现
public class MySQLRepository implements OrderRepository {
    @Override
    public void save(Order order) { /* MySQL 存储 */ }
}

public class RedisRepository implements OrderRepository {
    @Override
    public void save(Order order) { /* Redis 缓存 */ }
}
```

### Spring 的 DI 容器

Spring 的 `@Autowired` 注入正是 DIP 的最佳实践：

```java
@Service
public class OrderService {
    private final OrderRepository repository;

    @Autowired
    public OrderService(OrderRepository repository) { // 依赖抽象
        this.repository = repository;
    }
}
```

### 面试追问

> **第一层**：什么是依赖倒置？为什么要「倒置」？
>
> **第二层**：DIP 和控制反转（IOC）有什么关系？
>
> **第三层**：Spring 的依赖注入有哪几种方式？

**💡 加分回答**：依赖倒置是框架设计的核心思想。Spring、Guice、Dagger 都是 DIP 的实现。面试中能说出「好莱坞原则：Don't call us, we'll call you」会加分。

## 六、合成复用原则（CRP）

**定义**：优先使用组合/聚合，而不是继承。

### 继承 vs 组合

```java title="继承：耦合度高，违反OCP"
public class A {
    public void methodA() { /* ... */ }
}

// 使用继承：子类获得了父类的所有方法
public class B extends A {
    public void methodB() {
        methodA(); // 继承了父类方法
    }
}
```

```java title="组合：耦合度低，更灵活"
public class A {
    public void methodA() { /* ... */ }
}

public class B {
    private A a = new A(); // 组合

    public void methodB() {
        a.methodA(); // 调用组合对象的方法
    }
}
```

### 何时用继承，何时用组合？

| 场景 | 推荐方式 |
|------|----------|
| **is-a** 关系（如 Dog is Animal） | 继承 |
| **has-a** 关系（如 Car has Engine） | 组合 |
| 需要动态替换组件 | 组合 |
| 需要扩展行为 | 继承 |
| 需要多态 | 接口 + 继承/组合 |

### 组合的灵活性

```java title="组合可以运行时替换组件"
public class Car {
    private Engine engine;

    public Car(Engine engine) { // 运行时注入不同发动机
        this.engine = engine;
    }

    public void start() {
        engine.start();
    }
}

// 可以注入不同类型的发动机
Engine electric = new ElectricEngine();
Engine diesel = new DieselEngine();

Car car1 = new Car(electric);
Car car2 = new Car(diesel);
```

> **第一层**：合成复用原则和里氏替换原则的区别？
>
> **第二层**：为什么说「组合优于继承」？
>
> **第三层**：在 Spring 中，哪些地方体现了组合复用原则？

**💡 加分回答**：GoF 的设计模式中，大量使用了组合而不是继承。例如装饰器模式、策略模式、代理模式都是组合的应用。

## 七、六大原则对比总结

| 原则 | 关注点 | 关键词 | 常见违反场景 |
|------|--------|--------|--------------|
| SRP | 类的职责 | 变化原因 | 一个类干太多事 |
| OCP | 扩展性 | 对扩展开放 | 修改现有代码加功能 |
| LSP | 继承正确性 | 可替换性 | 子类改变父类行为 |
| ISP | 接口粒度 | 瘦接口 | 胖接口强迫实现不需要方法 |
| DIP | 依赖方向 | 依赖抽象 | 高层依赖低层 |
| CRP | 复用方式 | 组合优先 | 滥用继承 |

## 八、SOLID 原则的综合应用

### 从问题到原则

```
业务需求变化频繁
    ↓
需要扩展，不能修改现有代码
    ↓
开闭原则（OCP）
    ↓
抽象出稳定的接口
    ↓
依赖倒置原则（DIP）
    ↓
接口职责要单一
    ↓
单一职责（SRP）+ 接口隔离（ISP）
    ↓
子类必须能替换父类
    ↓
里氏替换原则（LSP）
    ↓
多用组合，少用继承
    ↓
合成复用原则（CRP）
```

### Spring 框架中的 SOLID

```java
// SRP：Spring MVC 中 Controller、Service、Repository 各司其职
// OCP：Spring 的扩展点（BeanPostProcessor）设计
// LSP：Spring 的事务传播行为
// ISP：Spring Data 的 Repository 接口拆分
// DIP：Spring 的依赖注入
// CRP：Spring 的组合优于继承（如 JdbcTemplate）
```

## 常见面试陷阱

> **⚠️ 陷阱 1**：把设计模式当成银弹
>
> 设计模式不是越多越好。过度设计比设计不足更危险。引入模式之前，先问自己：这个模式解决了什么问题？引入了什么复杂度？

> **⚠️ 陷阱 2**：混淆 SRP 和 ISP
>
> SRP 约束类，ISP 约束接口。一个类可能有多个职责但不胖接口（因为类可以组合多个接口）。

> **⚠️ 陷阱 3**：死记硬背原则定义
>
> 面试官不关心你能否背出定义，而关心你能否举出实际例子。准备 2-3 个你项目中违反原则后踩坑的真实案例。

## 加分回答

**💡 Spring Boot 自动配置中的 OCP**

Spring Boot 的 `Condition` 接口就是 OCP 的完美体现：

```java
// 定义扩展点
@FunctionalInterface
public interface Condition {
    boolean matches(ConditionContext context, AnnotatedTypeMetadata metadata);
}

// 用户自定义条件，无需修改 Spring 框架代码
public class MySQLCondition implements Condition {
    @Override
    public boolean matches(ConditionContext context, AnnotatedTypeMetadata metadata) {
        return context.getEnvironment().containsProperty("spring.datasource.url");
    }
}
```

## 延伸思考

> 为什么说 SOLID 原则是一把双刃剑？

1. **好处**：提高代码的可维护性、可扩展性、可测试性
2. **代价**：增加类的数量，提高理解成本
3. **平衡**：在项目复杂度和发展预期之间找到平衡

**💡 实战建议**：
- 新项目或核心模块：严格遵循 SOLID
- 小工具或一次性脚本：不必过度设计
- 遗留系统：逐步重构，不要一次性大改
