## 一、为什么需要单例模式

有些对象在整个应用中只需要一个实例：数据库连接池、配置管理器、日志记录器、线程池。

**不使用单例的问题**：

```java
// 每次调用都创建新对象
public class ConfigManager {
    public ConfigManager() {
        // 每次 new 都会重新加载配置文件
        this.config = loadFromFile();
    }
}

// 调用方
ConfigManager m1 = new ConfigManager();
ConfigManager m2 = new ConfigManager();
// m1 != m2，可能导致配置不一致
```

## 二、饿汉式（Eager Initialization）

### 基础写法

```java
public class HungrySingleton {
    // 类加载时就创建实例
    private static final HungrySingleton INSTANCE = new HungrySingleton();

    private HungrySingleton() {
        // 私有构造函数
    }

    public static HungrySingleton getInstance() {
        return INSTANCE;
    }
}
```

### 静态代码块写法

```java
public class HungrySingletonStatic {
    private static final HungrySingletonStatic INSTANCE;

    static {
        INSTANCE = new HungrySingletonStatic();
    }

    private HungrySingletonStatic() {}

    public static HungrySingletonStatic getInstance() {
        return INSTANCE;
    }
}
```

### 优缺点分析

| 维度 | 评价 |
|------|------|
| 线程安全 | ✅ 安全，JVM 保证类加载时线程安全 |
| 延迟加载 | ❌ 不是懒加载，类加载时就创建 |
| 执行效率 | ✅ 高，没有同步开销 |
| 资源浪费 | ❌ 如果从未使用，会浪费内存 |

### 适用场景

- 对象创建开销小，或者一定会被使用
- 需要保证实例在类加载时就被初始化

## 三、懒汉式（Lazy Initialization）

### 基础写法（线程不安全）

```java
public class LazySingletonUnsafe {
    private static LazySingletonUnsafe INSTANCE;

    private LazySingletonUnsafe() {}

    public static LazySingletonUnsafe getInstance() {
        if (INSTANCE == null) {
            INSTANCE = new LazySingletonUnsafe(); // 线程不安全！
        }
        return INSTANCE;
    }
}
```

### ⚠️ 线程安全问题分析

```
线程 A                    线程 B
   │                          │
   ├─ if (INSTANCE == null)   │
   │   ├─ true                │
   │   │                      ├─ if (INSTANCE == null)
   │   ├─ 创建实例 A           │    ├─ true
   │   │                      ├─ 创建实例 B
   │   │                      │
   └─ return A           vs    └─ return B
                                    │
                               A != B ❌
```

**⚠️ 陷阱**：在高并发场景下，两个线程可能同时进入 `if` 判断，创建出两个不同的实例。

### 同步方法写法

```java
public class LazySingletonSyncMethod {
    private static LazySingletonSyncMethod INSTANCE;

    private LazySingletonSyncMethod() {}

    // synchronized 保证线程安全
    public static synchronized LazySingletonSyncMethod getInstance() {
        if (INSTANCE == null) {
            INSTANCE = new LazySingletonSyncMethod();
        }
        return INSTANCE;
    }
}
```

| 维度 | 评价 |
|------|------|
| 线程安全 | ✅ 安全 |
| 延迟加载 | ✅ 懒加载 |
| 执行效率 | ❌ 每次调用都要获取锁，开销大 |

## 四、双重检查锁定（DCL）

### 标准写法

```java
public class DCLSingleton {
    // volatile 防止指令重排序
    private static volatile DCLSingleton INSTANCE;

    private DCLSingleton() {
        // 防止反射攻击：单例被创建后，再 new 会抛异常
        if (INSTANCE != null) {
            throw new RuntimeException("单例已被创建，禁止通过反射创建新实例");
        }
    }

    public static DCLSingleton getInstance() {
        if (INSTANCE == null) {              // 第一次检查：避免不必要的同步
            synchronized (DCLSingleton.class) { // 同步
                if (INSTANCE == null) {         // 第二次检查：确保只创建一次
                    INSTANCE = new DCLSingleton();
                }
            }
        }
        return INSTANCE;
    }
}
```

### DCL 为什么需要 volatile

`INSTANCE = new DCLSingleton()` 不是原子操作，它分为三步：

```
1. 分配内存
2. 调用构造函数
3. 将引用赋值给 INSTANCE
```

**问题**：JIT 编译器可能进行指令重排序，导致顺序变成：

```
1. 分配内存
2. 将引用赋值给 INSTANCE  ← 先赋值
3. 调用构造函数           ← 后构造
```

**⚠️ 线程 B 的问题**：

```java
// 线程 A 执行 new DCLSingleton()
INSTANCE = new DCLSingleton();  // 重排序后：先赋值，后构造

// 线程 B 执行 getInstance()
if (INSTANCE != null) {         // 此时 INSTANCE != null
    return INSTANCE;            // 但对象还未构造完成！
}
```

**volatile 的作用**：
- 禁止指令重排序
- 保证可见性（线程 A 的修改对线程 B 可见）

### DCL 优缺点

| 维度 | 评价 |
|------|------|
| 线程安全 | ✅ 安全 |
| 延迟加载 | ✅ 懒加载 |
| 执行效率 | ✅ 高，首次创建后无需同步 |
| 代码复杂度 | 中等 |

## 五、静态内部类（Initialization-on-demand holder）

### 写法

```java
public class StaticInnerClassSingleton {
    private StaticInnerClassSingleton() {
        if (Inner.INSTANCE != null) {
            throw new RuntimeException("禁止通过反射创建");
        }
    }

    private static class Inner {
        private static final StaticInnerClassSingleton INSTANCE =
            new StaticInnerClassSingleton();
    }

    public static StaticInnerClassSingleton getInstance() {
        return Inner.INSTANCE;
    }
}
```

### 为什么线程安全

JVM 的类加载机制保证了线程安全：

```
1. 外部类加载时，不会加载内部类
2. 只有调用 getInstance() 时，才会加载内部类
3. 类加载过程是线程安全的（JVM 内部有锁）
```

### 优缺点

| 维度 | 评价 |
|------|------|
| 线程安全 | ✅ 安全（JVM 类加载机制） |
| 延迟加载 | ✅ 懒加载 |
| 执行效率 | ✅ 高 |
| 序列化安全 | ⚠️ 需要额外处理 |

## 六、枚举单例（Effective Java 推荐）

### 写法

```java
public enum EnumSingleton {
    INSTANCE;

    private String data;

    public void setData(String data) {
        this.data = data;
    }

    public String getData() {
        return data;
    }
}

// 使用
EnumSingleton instance = EnumSingleton.INSTANCE;
```

### 为什么推荐枚举单例

1. **线程安全**：JVM 保证枚举实例的创建是线程安全的
2. **防止反射攻击**：枚举的构造函数由 JVM 调用，外部无法调用
3. **防止序列化问题**：枚举序列化由 JVM 特殊处理
4. **简洁**：代码量最少

### 枚举单例的底层原理

```java
// 编译后的 EnumSingleton 大致等价于
public class EnumSingleton extends Enum<EnumSingleton> {
    public static final EnumSingleton INSTANCE = new EnumSingleton();

    private EnumSingleton() {
        super("INSTANCE", 0);
    }
}
```

## 八、线程安全与序列化问题

### 序列化攻击

```java
public class SerializableSingleton implements Serializable {
    private static final long serialVersionUID = 1L;
    private static final SerializableSingleton INSTANCE = new SerializableSingleton();

    private SerializableSingleton() {}

    public static SerializableSingleton getInstance() {
        return INSTANCE;
    }
}

// 反序列化测试
ObjectOutputStream oos = new ObjectOutputStream(new FileOutputStream("singleton.ser"));
oos.writeObject(SerializableSingleton.getInstance());
oos.close();

ObjectInputStream ois = new ObjectInputStream(new FileInputStream("singleton.ser"));
SerializableSingleton s1 = SerializableSingleton.getInstance();
SerializableSingleton s2 = (SerializableSingleton) ois.readObject();
// s1 != s2 ❌ 单例被破坏！
```

### 解决方案：添加 `readResolve` 方法

```java
public class SerializableSingleton implements Serializable {
    private static final long serialVersionUID = 1L;
    private static final SerializableSingleton INSTANCE = new SerializableSingleton();

    private SerializableSingleton() {}

    public static SerializableSingleton getInstance() {
        return INSTANCE;
    }

    // 反序列化时返回单例，而不是创建新对象
    private Object readResolve() {
        return INSTANCE;
    }
}
```

## 九、反射攻击与防护

### 反射攻击原理

```java
// 通过反射创建新实例
Constructor<Singleton> constructor = Singleton.class.getDeclaredConstructor();
constructor.setAccessible(true);
Singleton s1 = constructor.newInstance();
Singleton s2 = constructor.newInstance();
// s1 != s2 ❌ 单例被破坏！
```

### 防护措施

```java
public class ProtectedSingleton {
    private static volatile ProtectedSingleton INSTANCE;

    private ProtectedSingleton() {
        // 在构造函数中检查
        if (INSTANCE != null) {
            throw new RuntimeException("单例已被创建，禁止反射攻击");
        }
    }

    public static ProtectedSingleton getInstance() {
        if (INSTANCE == null) {
            synchronized (ProtectedSingleton.class) {
                if (INSTANCE == null) {
                    INSTANCE = new ProtectedSingleton();
                }
            }
        }
        return INSTANCE;
    }
}
```

**⚠️ 缺陷**：这种防护对枚举无效（枚举构造函数只能由 JVM 调用）。

## 十一、Spring 中的单例

Spring 默认使用单例模式管理 Bean：

```java
// Spring 配置
@Configuration
public class AppConfig {
    @Bean
    public UserService userService() {
        return new UserService();
    }
}

// Spring 默认作用域：singleton
// 每次获取的都是同一个实例
UserService s1 = context.getBean(UserService.class);
UserService s2 = context.getBean(UserService.class);
// s1 == s2 ✅
```

## 常见面试陷阱

> **⚠️ 陷阱 1**：只说结论，不懂原理
>
> 面试官问「DCL 为什么加 volatile」，很多候选人回答「为了线程安全」就结束了。但真正理解的人应该说出：防止指令重排序、保证可见性。

> **⚠️ 陷阱 2**：忽略序列化问题
>
> 很多人知道单例的写法，但不知道序列化会破坏单例。需要补充「readResolve」方法。

> **⚠️ 陷阱 3**：认为 enum 单例完美无缺
>
> 枚举单例虽然安全，但在某些场景下不适合：比如需要懒加载、需要继承其他类。

```java
// 使用 Mockito 模拟单例
@Test
public void testSingleton() {

    }
}
```
