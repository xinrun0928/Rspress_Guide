## 一、为什么需要模板方法模式

### 重复代码的问题

```java
// 订单处理流程 A
public class OrderProcessorA {
    public void process(Order order) {
        // 1. 验证订单（相同）
        validate(order);

        // 2. 计算价格（不同）
        double price = calculatePriceA(order);

        // 3. 保存订单（相同）
        saveOrder(order, price);

        // 4. 发送通知（不同）
        sendNotificationA(order);
    }
}

// 订单处理流程 B
public class OrderProcessorB {
    public void process(Order order) {
        // 1. 验证订单（相同）
        validate(order);

        // 2. 计算价格（不同）
        double price = calculatePriceB(order);

        // 3. 保存订单（相同）
        saveOrder(order, price);

        // 4. 发送通知（不同）
        sendNotificationB(order);
    }
}
```

**问题**：
- 大量重复代码（validate、saveOrder）
- 难以统一修改流程
- 违反 DRY 原则

## 二、模板方法模式实现

### 核心思想

定义一个算法的骨架，将某些步骤延迟到子类中实现。

```java
// 抽象基类：定义模板方法
public abstract class OrderProcessorTemplate {

    // 模板方法：定义算法骨架（final 防止被子类重写）
    public final void process(Order order) {
        // 步骤1：验证订单（公共方法）
        validate(order);

        // 步骤2：计算价格（抽象方法，子类实现）
        double price = calculatePrice(order);

        // 步骤3：保存订单（公共方法）
        saveOrder(order, price);

        // 步骤4：发送通知（抽象方法，子类实现）
        sendNotification(order);
    }

    // 公共方法：所有子类共享
    private void validate(Order order) {
        System.out.println("验证订单: " + order.getId());
        if (order.getAmount() <= 0) {
            throw new IllegalArgumentException("订单金额必须大于0");
        }
    }

    private void saveOrder(Order order, double price) {
        System.out.println("保存订单，金额: " + price);
        order.setFinalPrice(price);
    }

    // 抽象方法：子类必须实现
    protected abstract double calculatePrice(Order order);

    protected abstract void sendNotification(Order order);
}

// 具体实现 A
public class OrderProcessorA extends OrderProcessorTemplate {
    @Override
    protected double calculatePrice(Order order) {
        System.out.println("使用算法A计算价格");
        return order.getAmount() * 0.9;
    }

    @Override
    protected void sendNotification(Order order) {
        System.out.println("发送短信通知");
    }
}

// 具体实现 B
public class OrderProcessorB extends OrderProcessorTemplate {
    @Override
    protected double calculatePrice(Order order) {
        System.out.println("使用算法B计算价格");
        return order.getAmount() * 0.8 - 10;
    }

    @Override
    protected void sendNotification(Order order) {
        System.out.println("发送邮件通知");
    }
}

// 使用
public class Client {
    public static void main(String[] args) {
        Order order = new Order(1L, 1000);

        OrderProcessorTemplate processorA = new OrderProcessorA();
        processorA.process(order);

        System.out.println("---");

        OrderProcessorTemplate processorB = new OrderProcessorB();
        processorB.process(order);
    }
}
```

## 三、钩子方法

### 什么是钩子方法

钩子方法提供默认实现，子类可以选择覆盖。

```java
public abstract class DataProcessorTemplate {

    // 模板方法
    public final void process() {
        connect();

        if (needCache()) {  // 钩子方法：可选步骤
            loadCache();
        }

        processData();

        if (needLogging()) {  // 钩子方法
            log();
        }

        close();
    }

    // 抽象方法（必须实现）
    protected abstract void processData();

    // 公共方法
    private void connect() {
        System.out.println("建立连接");
    }

    private void close() {
        System.out.println("关闭连接");
    }

    private void loadCache() {
        System.out.println("加载缓存");
    }

    // 钩子方法：子类可以选择覆盖
    protected boolean needCache() {
        return false; // 默认不缓存
    }

    protected boolean needLogging() {
        return true; // 默认需要日志
    }

    protected void log() {
        System.out.println("记录日志");
    }
}

// 子类A：需要缓存
public class CachedProcessor extends DataProcessorTemplate {
    @Override
    protected void processData() {
        System.out.println("处理数据（带缓存）");
    }

    @Override
    protected boolean needCache() {
        return true;
    }
}

// 子类B：不需要缓存
public class SimpleProcessor extends DataProcessorTemplate {
    @Override
    protected void processData() {
        System.out.println("处理数据（简单）");
    }
}
```

## 四、模板方法 vs 策略模式

| 对比 | 模板方法模式 | 策略模式 |
|------|--------------|----------|
| 继承 vs 组合 | 使用继承 | 使用组合 |
| 侧重点 | 定义算法骨架 | 选择算法实现 |
| 扩展方式 | 子类重写方法 | 注入不同策略对象 |
| 控制权 | 父类控制流程 | 客户端控制 |
| 符合 OCP | ✅ 部分符合 | ✅ 完全符合 |

```java
// 模板方法：子类决定"怎么做"，但"什么时候做"由父类控制
public final void templateMethod() {  // 父类控制流程
    step1();  // 子类实现
    step2();  // 子类实现
}

// 策略：客户端决定"用哪个策略"
public void execute(Strategy strategy) {
    strategy.execute();  // 完全由策略控制
}
```

## 五、Spring 中的模板方法模式

### JdbcTemplate

```java
// Spring 的 JdbcTemplate 是模板方法模式的经典应用
public class JdbcTemplate {
    // 模板方法
    public <T> T execute(StatementCallback<T> action) {
        Connection conn = null;
        Statement stmt = null;
        try {
            conn = getConnection();
            stmt = conn.createStatement();
            return action.doInStatement(stmt);  // 回调：具体操作由客户端实现
        } finally {
            closeStatement(stmt);
            releaseConnection(conn);
        }
    }
}

// 使用示例
 jdbcTemplate.execute(new StatementCallback<Integer>() {
    @Override
    public Integer doInStatement(Statement stmt) throws SQLException {
        return stmt.executeUpdate("INSERT INTO user VALUES(...)");
    }
});

// Lambda 简化
jdbcTemplate.execute(stmt -> stmt.executeUpdate("INSERT INTO user VALUES(...)"));
```

### Spring 的事务模板

```java
public class TransactionTemplate {
    public <T> T execute(TransactionCallback<T> action) {
        TransactionStatus status = transactionManager.getTransaction(null);
        try {
            T result = action.doInTransaction(status);
            transactionManager.commit(status);
            return result;
        } catch (Exception e) {
            transactionManager.rollback(status);
            throw e;
        }
    }
}

// 使用
transactionTemplate.execute(status -> {
    userDao.save(user);
    orderDao.save(order);
    return null;
});
```

## 六、实战应用

### 1. 数据导出

```java
public abstract class AbstractDataExporter {
    // 模板方法（final 防止重写）
    public final void export(String fileName) {
        // 1. 准备数据
        List<?> data = fetchData();

        // 2. 创建文件
        OutputStream out = createFile(fileName);

        // 3. 写入数据（子类实现）
        writeData(out, data);

        // 4. 关闭文件
        closeFile(out);

        // 5. 后续处理（钩子方法）
        afterExport(fileName);
    }

    protected List<?> fetchData() {
        return Collections.emptyList();
    }

    protected void afterExport(String fileName) {
        // 默认空实现
    }

    // 抽象方法：子类必须实现
    protected abstract OutputStream createFile(String fileName);
    protected abstract void writeData(OutputStream out, List<?> data) throws IOException;
    protected abstract void closeFile(OutputStream out);
}

// Excel 导出
public class ExcelExporter extends AbstractDataExporter {
    @Override
    protected OutputStream createFile(String fileName) {
        return new FileOutputStream(fileName + ".xlsx");
    }

    @Override
    protected void writeData(OutputStream out, List<?> data) {
        // Excel 写入逻辑
    }

    @Override
    protected void closeFile(OutputStream out) {
        out.close();
    }

    @Override
    protected void afterExport(String fileName) {
        System.out.println("Excel导出完成: " + fileName + ".xlsx");
    }
}
```

### 2. 缓存处理

```java
public abstract class AbstractCacheTemplate<K, V> {

    public V get(K key) {
        // 1. 从缓存获取
        V value = getFromCache(key);
        if (value != null) {
            return value;
        }

        // 2. 缓存未命中，从数据源获取
        value = loadFromSource(key);

        // 3. 存入缓存
        if (value != null) {
            putToCache(key, value);
        }

        return value;
    }

    // 抽象方法
    protected abstract V getFromCache(K key);
    protected abstract V loadFromSource(K key);
    protected abstract void putToCache(K key, V value);

    // 钩子方法：缓存过期时间
    protected Duration getExpireDuration() {
        return Duration.ofMinutes(30);
    }
}
```

## 八、常见面试陷阱

> **⚠️ 陷阱 1**：模板方法用 abstract 还是 final
>
> 模板方法（定义骨架的方法）应该是 `final`，防止子类修改流程。抽象方法（子类实现的方法）应该是 `abstract`。

> **⚠️ 陷阱 2**：忽视钩子方法
>
> 钩子方法可以让子类更灵活地选择是否执行某个步骤，是模板方法的重要扩展点。
