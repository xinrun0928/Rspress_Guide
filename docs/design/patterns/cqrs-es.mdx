# CQRS与Event Sourcing

## 一个银行账户的诡异余额

2021年，我们团队遇到一个匪夷所思的 bug：用户 A 的账户余额显示是 10000 元，但流水记录显示总存入 50000 元、总取出 30000 元。理论上应该是 20000 元。

排查了整整一周，发现根因：**数据被直接修改了**。

开发人员在紧急修复时执行了：

```sql
UPDATE account SET balance = 10000 WHERE id = 123;
```

但这笔修改没有记录在流水表里。

**这正是 Event Sourcing 要解决的问题：把"发生了什么"记录下来，而不是只记录"当前状态是什么"。**

## 二、CQRS：命令查询职责分离🔴

### 2.1 核心思想

```
传统模式：     同一个数据模型处理读和写
CQRS：          写模型和读模型分离

命令（Command）：修改数据的操作 → 写模型
查询（Query）：读取数据的操作 → 读模型
```

### 2.2 简单实现

```java
// 写模型：聚合根
@Aggregate
class BankAccount {
    private Money balance;
    private List<Transaction> transactions;

    public void deposit(Money amount) {
        this.balance = this.balance.add(amount);
        // 记录事件
        registerEvent(new MoneyDepositedEvent(this.id, amount, Instant.now()));
    }

    public void withdraw(Money amount) {
        if (this.balance.compareTo(amount) < 0) {
            throw new InsufficientBalanceException();
        }
        this.balance = this.balance.subtract(amount);
        registerEvent(new MoneyWithdrawnEvent(this.id, amount, Instant.now()));
    }
}

// 读模型：DTO
record AccountBalanceDTO(
    Long accountId,
    Money balance,
    Money availableBalance,
    LocalDateTime lastUpdated
) {}

record AccountStatementDTO(
    Long accountId,
    List<TransactionDTO> transactions,
    Money totalDeposits,
    Money totalWithdrawals,
    Money netChange
) {}
```

### 2.3 同步机制

```java
// 事件监听器：同步更新读模型
@EventListener
class AccountProjectionUpdater {
    @Autowired
    private JdbcTemplate jdbcTemplate;

    @Async
    @EventListener
    public void handleMoneyDeposited(MoneyDepositedEvent event) {
        // 更新余额
        jdbcTemplate.update(
            "UPDATE account_balance SET balance = balance + ? WHERE account_id = ?",
            event.getAmount(), event.getAccountId()
        );

        // 更新交易统计
        jdbcTemplate.update(
            "UPDATE account_stats SET total_deposits = total_deposits + ? WHERE account_id = ?",
            event.getAmount(), event.getAccountId()
        );
    }
}
```

## 三、Event Sourcing：事件溯源🔴

### 3.1 核心思想

```
传统模式：
余额 = 10000（当前状态）

Event Sourcing：
存入 10000 → 余额 = 10000
取出 5000  → 余额 = 5000
存入 10000 → 余额 = 15000

从事件重新计算当前状态，而不是直接存储当前状态
```

### 3.2 实现

```java
// 事件存储
@Entity
class AccountEvent {
    @Id
    private Long id;
    private Long accountId;
    private String eventType;  // DEPOSIT, WITHDRAW
    private Money amount;
    private Instant timestamp;
    private int version;       // 乐观锁版本
}

// 事件溯源聚合
class BankAccountES {
    private Long id;
    private Money balance;
    private List<AccountEvent> uncommittedEvents = new ArrayList<>();

    public void deposit(Money amount) {
        if (amount.compareTo(Money.ZERO) <= 0) {
            throw new IllegalArgumentException("Amount must be positive");
        }

        this.balance = this.balance.add(amount);

        // 记录事件，而不是直接修改状态
        uncommittedEvents.add(new AccountEvent(
            this.id, "DEPOSIT", amount, Instant.now()
        ));
    }

    public void withdraw(Money amount) {
        if (this.balance.compareTo(amount) < 0) {
            throw new InsufficientBalanceException();
        }

        this.balance = this.balance.subtract(amount);

        uncommittedEvents.add(new AccountEvent(
            this.id, "WITHDRAW", amount, Instant.now()
        ));
    }

    // 从事件重建状态
    public void rebuildFromEvents(List<AccountEvent> events) {
        this.balance = Money.ZERO;
        for (AccountEvent event : events) {
            applyEvent(event);
        }
    }

    private void applyEvent(AccountEvent event) {
        switch (event.getType()) {
            case "DEPOSIT":
                this.balance = this.balance.add(event.getAmount());
                break;
            case "WITHDRAW":
                this.balance = this.balance.subtract(event.getAmount());
                break;
        }
    }

    public List<AccountEvent> getUncommittedEvents() {
        return new ArrayList<>(uncommittedEvents);
    }

    public void markEventsCommitted() {
        uncommittedEvents.clear();
    }
}
```

### 3.3 快照优化

// 定期创建快照，避免从头重建
@Entity
class AccountSnapshot {
    private Long accountId;
    private Money balance;
    private int lastEventVersion;
    private Instant createdAt;
}

// 从快照恢复
class SnapshotRepository {
    public BankAccountES restore(Long accountId) {
        // 1. 获取最近的快照
        AccountSnapshot snapshot = findLatestSnapshot(accountId);

        // 2. 获取快照之后的事件
        List<AccountEvent> events = eventStore.findEventsAfter(
            accountId, snapshot.getLastEventVersion()
        );

        // 3. 重建聚合
        BankAccountES account = new BankAccountES(accountId);
        account.rebuildFromEvents(events);

        return account;
    }
}

## 四、CQRS + Event Sourcing 组合🟡

### 4.1 完整架构

```
命令端：
用户操作 → Command → Aggregate 处理 → 事件发布
                                    ↓
                                  Event Store（持久化）
                                    ↓
                              Event Bus
                                    ↓
读取端：
Event Bus → Projection → Read Model（物化视图）
                ↓
              Query Handler → API
```

### 4.2 Spring + Axon Framework 实现

```java
// 聚合根
@Aggregate
class BankAccountAggregate {
    @AggregateIdentifier
    private String id;

    @AggregateMember
    private Money balance = Money.ZERO;

    @CommandHandler
    public void handle(DepositCommand command) {
        apply(new MoneyDepositedEvent(id, command.getAmount()));
    }

    @EventSourcingHandler
    public void on(MoneyDepositedEvent event) {
        this.balance = this.balance.add(event.getAmount());
    }
}

// 查询处理器
@QueryHandler
class AccountQueryHandler {
    @QueryHandler
    public AccountView handle(GetAccountQuery query) {
        return viewStore.findById(query.getAccountId());
    }
}

// 事件监听器
@EventHandler
class AccountEventHandler {
    @EventHandler
    public void on(MoneyDepositedEvent event) {
        AccountView view = viewStore.findById(event.getId());
        view.setBalance(view.getBalance().add(event.getAmount()));
        viewStore.save(view);
    }
}
```

## 五、优势与挑战🟡

### 5.1 Event Sourcing 的优势

| 优势 | 说明 |
|------|------|
| 完整审计 | 所有操作都有记录，可以追溯任意时刻的状态 |
| 时间旅行 | 可以重放事件，还原历史状态 |
| 高并发写 | 事件追加模式，无锁写入 |
| 多视图 | 同一事件源生成多个读模型 |
| 调试友好 | 可以重放事件来重现问题 |

### 5.2 Event Sourcing 的挑战

| 挑战 | 说明 |
|------|------|
| 事件膨胀 | 历史数据越来越多，读取变慢 |
| 事件变更 | 如果业务规则变了，历史事件怎么处理？ |
| 一致性 | 事件发布和写入之间的一致性 |
| 查询复杂度 | 无法直接查询当前状态，需要重建 |
| 学习曲线 | 开发人员需要理解事件溯源 |

### 5.3 事件版本管理

```java
// 事件升级：处理历史事件格式变更
@SagaEventHandler(associationProperty = "accountId")
class AccountSagaEventHandler {

    @SagaEventHandler
    public void handle(AccountCreatedEvent event, AggregateLifecycle lifecycle) {
        // 处理事件
    }

    // 同一方法可以处理不同版本的事件
    @SagaEventHandler(eventType = "AccountCreatedEvent_v1")
    public void handleV1(AccountCreatedEventV1 event) {
        AccountCreatedEventV2 upgraded = new AccountCreatedEventV2(
            event.getId(),
            event.getOwner(),
            event.getInitialBalance(),
            "USD" // 默认货币
        );
        lifecycle.apply(upgraded);
    }
}
```

【架构权衡】
CQRS + Event Sourcing 是强大的组合，但代价是系统复杂度大幅增加。只在确实需要审计追溯、时间旅行、多视图同步的场景才值得使用。简单业务用这套架构是过度设计。
