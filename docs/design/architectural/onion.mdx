## 一、核心思想

洋葱架构（Onion Architecture）由 Jeffrey Palermo 提出，强调**核心业务逻辑在最中心**，外部依赖层层向外。

**核心原则**：依赖只能由外向内，内层不知道外层的存在。

## 四、代码实现

### 1. 核心域（Domain Layer）

```java
// 领域实体
public class Order {
    private Long id;
    private Long userId;
    private OrderStatus status;
    private List<OrderItem> items;

    // 领域方法
    public void addItem(OrderItem item) {
        if (status != OrderStatus.DRAFT) {
            throw new IllegalStateException("订单已锁定，无法添加商品");
        }
        this.items.add(item);
    }

    public void pay() {
        if (status != OrderStatus.DRAFT) {
            throw new IllegalStateException("订单状态不正确");
        }
        this.status = OrderStatus.PAID;
    }

    public Money getTotalAmount() {
        return items.stream()
            .map(OrderItem::getSubtotal)
            .reduce(Money.ZERO, Money::add);
    }
}

// 值对象
public class Money {
    private final BigDecimal amount;
    private final String currency;

    public Money add(Money other) {
        if (!this.currency.equals(other.currency)) {
            throw new IllegalArgumentException("货币类型不一致");
        }
        return new Money(this.amount.add(other.amount), this.currency);
    }
}

// 仓储接口（定义在核心层）
public interface OrderRepository {
    Order save(Order order);
    Optional<Order> findById(Long id);
}
```

### 2. 应用服务（Application Layer）

```java
// 应用服务 / 用例
public class CreateOrderUseCase {
    private final OrderRepository orderRepository;
    private final PaymentService paymentService;
    private final EventPublisher eventPublisher;

    public CreateOrderUseCase(OrderRepository orderRepository,
                              PaymentService paymentService,
                              EventPublisher eventPublisher) {
        this.orderRepository = orderRepository;
        this.paymentService = paymentService;
        this.eventPublisher = eventPublisher;
    }

    public Order execute(CreateOrderCommand command) {
        // 1. 创建订单
        Order order = new Order();
        order.setUserId(command.getUserId());

        for (ItemDTO item : command.getItems()) {
            OrderItem orderItem = new OrderItem();
            orderItem.setProductId(item.getProductId());
            orderItem.setQuantity(item.getQuantity());
            order.addItem(orderItem);
        }

        // 2. 保存订单
        Order savedOrder = orderRepository.save(order);

        // 3. 发布领域事件
        eventPublisher.publish(new OrderCreatedEvent(savedOrder));

        return savedOrder;
    }
}

// 命令对象
public class CreateOrderCommand {
    private Long userId;
    private List<ItemDTO> items;
    // getters/setters
}
```

### 3. 适配器层（Adapters Layer）

```java
// REST 控制器
@RestController
@RequestMapping("/api/orders")
public class OrderController {
    private final CreateOrderUseCase createOrderUseCase;

    @PostMapping
    public ResponseEntity<OrderDTO> createOrder(@RequestBody CreateOrderRequest request) {
        CreateOrderCommand command = toCommand(request);
        Order order = createOrderUseCase.execute(command);
        return ResponseEntity.ok(toDTO(order));
    }
}

// Spring Data JPA 适配器
@Repository
public class JpaOrderRepository implements OrderRepository {
    @Autowired
    private SpringDataOrderRepository jpaRepository;

    @Override
    public Order save(Order order) {
        return jpaRepository.save(order);
    }

    @Override
    public Optional<Order> findById(Long id) {
        return jpaRepository.findById(id);
    }
}
```

### 4. 基础设施层（Infrastructure Layer）

```java
@Configuration
public class DependencyInjection {
    @Bean
    public OrderRepository orderRepository(JpaOrderRepository jpaRepo) {
        return jpaRepo;  // 适配器注入
    }

    @Bean
    public CreateOrderUseCase createOrderUseCase(OrderRepository repo,
                                                  PaymentService payment,
                                                  EventPublisher events) {
        return new CreateOrderUseCase(repo, payment, events);
    }
}
```

## 五、洋葱架构 vs 六边形架构

| 对比 | 洋葱架构 | 六边形架构 |
|------|----------|------------|
| 核心理念 | 依赖向内 | 端口与适配器 |
| 分层 | 4-5 层同心圆 | 六边形 |
| 依赖规则 | 只能外层依赖内层 | 端口隔离 |
| 适用场景 | DDD 项目 | 复杂业务系统 |

---

## 六、核心优势

| 优势 | 说明 |
|------|------|
| 高度内聚 | 核心业务逻辑集中 |
| 低耦合 | 依赖单一方向 |
| 可测试性 | 核心可完全独立测试 |
| 可替换性 | 外层可随时替换 |
| DDD 友好 | 与领域驱动设计天然契合 |

## 八、实践建议

| 建议 | 说明 |
|------|------|
| 从核心开始 | 先写领域模型，再写外层 |
| 严格遵守依赖规则 | 用架构测试验证 |
| 使用 DI 容器 | Spring 支持依赖注入 |
| 单元测试优先 | 核心层测试不需要外层依赖 |
