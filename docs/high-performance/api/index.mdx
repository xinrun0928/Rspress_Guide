# 接口优化与最佳实践

你的接口响应时间是 500ms，用户抱怨体验差。

你加了个索引，响应时间降到了 100ms。用户说：还行吧。

隔壁团队的接口同样功能，响应时间只要 20ms。

差距在哪里？**有时候，不只是 SQL 的问题——是整个接口设计的思路。**

从网络传输到序列化，从连接池到缓存，接口性能优化是一个系统工程。本模块涵盖接口优化的核心知识点，助你打造高性能 API。

## 接口性能优化四板斧

### 第一斧：减少数据传输

网络是接口响应时间的大敌。一次 HTTP 请求，往返延迟可能就要 10-50ms。

优化思路：

```java
// 优化前：返回全量字段
public User getUser(Long id) {
    return userRepository.findById(id);
}
// 问题：用户头像 2MB、详细信息 50 个字段，用户只要名字

// 优化后：按需返回字段
public UserSummary getUserSummary(Long id) {
    return userRepository.findById(id)
        .map(User::toSummary);  // 只返回必要字段
}

// 优化三板斧：
// 1. 字段裁剪：只返回必要的字段
// 2. 压缩传输：开启 GZIP，500KB → 50KB
// 3. 懒加载：大字段按需加载
```

### 第二斧：减少网络请求

一次接口调用，可能触发 N 次数据库查询、N+1 问题、循环查库。

优化思路：

```java
// 优化前：N+1 查询
List<Order> orders = orderRepository.findByUserId(userId);
for (Order order : orders) {
    Product product = productRepository.findById(order.getProductId());
    // 100 个订单 = 100 次额外查询
}

// 优化后：批量查询
List<Order> orders = orderRepository.findByUserId(userId);
Set<Long> productIds = orders.stream()
    .map(Order::getProductId)
    .collect(Collectors.toSet());
Map<Long, Product> productMap = productRepository.findByIds(productIds)
    .stream().collect(Collectors.toMap(Product::getId, p -> p));

// 100 个订单 = 2 次查询（1 次订单 + 1 次产品）
```

### 第三斧：善用缓存

接口响应时间 100ms，80ms 在查库。缓存可以把这 80ms 变成 1ms。

```java
// 缓存模式：Cache-Aside
public User getUser(Long id) {
    // 1. 先查缓存
    User user = cache.get("user:" + id);
    if (user != null) {
        return user;
    }

    // 2. 缓存没有，查数据库
    user = userRepository.findById(id);

    // 3. 写入缓存
    cache.set("user:" + id, user, Duration.ofMinutes(10));

    return user;
}
```

### 第四斧：异步处理

不是所有操作都需要同步完成。日志、统计、通知——这些都可以异步处理。

```java
// 优化前：同步处理所有操作
public void createOrder(Order order) {
    orderRepository.save(order);        // 10ms
    inventoryService.deduct(order);    // 50ms
    messageService.send(order);         // 20ms
    notificationService.notify(order);  // 30ms
    // 总计：110ms
}

// 优化后：只同步核心操作
public void createOrder(Order order) {
    orderRepository.save(order);        // 10ms
    inventoryService.deduct(order);    // 50ms
    // 总计：60ms（节省 50ms）

    // 消息和通知异步处理
    CompletableFuture.runAsync(() -> {
        messageService.send(order);
        notificationService.notify(order);
    });
}
```

## 常见性能瓶颈与解决方案

| 瓶颈 | 症状 | 解决方案 |
|-----|------|---------|
| 网络延迟 | 跨机房调用慢 | 就近访问、CDN、合并请求 |
| 数据库慢查询 | 接口响应时间不稳定 | 索引优化、SQL 审查、读写分离 |
| 序列化耗时 | JSON 解析慢 | Protobuf、Kryo、Hessian |
| 连接数不足 | 并发高时超时 | 连接池调优、HTTP/2 |
| GC 停顿 | 偶发卡顿 | JVM 调优、对象池化 |

## 学习路径建议

### 入门：理解接口性能瓶颈

建议先学习**HTTP 连接复用**和**接口响应压缩**。

这两个是最容易上手的优化手段，几乎不需要改业务代码。

### 进阶：掌握缓存与批处理

深入学习**连接池配置**和**幂等性设计**。

连接池配置是基础，幂等性是保障——两者结合，才能构建高并发、高可用的接口。

### 高级：系统性优化

学习**CDN 加速**和**API 网关优化**。

这是架构层面的优化，需要结合业务场景和系统架构来设计。

## 延伸思考

接口性能优化有个「边际效应递减」原则：

**优化前 10% 的问题，投入产出比最高；越往后优化，代价越大，收益越小。**

所以，优化要有优先级：

1. 先优化网络（延迟最低的果子）
2. 再优化数据库（收益明显）
3. 最后考虑架构调整（代价最大）

**记住：不要过度优化。先让接口「能用」，再让它「好用」，最后再考虑「极致」。**
