# 数据库全景指南

想象一下，你正在设计一个系统。用户信息存在哪？订单数据放哪里？日志文件呢？热点缓存呢？时序监控数据呢？

不同的数据，有不同的特点；不同的业务，有不同的需求。于是，数据库的江湖应运而生——MySQL、Redis、MongoDB、HBase、PostgreSQL、Oracle、TiDB...每一种数据库都有自己的定位和擅长的领域。

本系列将带你系统性地了解各类数据库，从关系型到 NoSQL，从传统到分布式，从OLTP到OLAP，构建完整的数据库知识图谱。

## 数据爆炸时代：我们为什么需要这么多数据库？

2000 年前后，大多数系统只需要一种数据库：Oracle 或者 SQL Server。那时候数据量小、并发低、需求简单，一台小型机就能支撑整个业务。

但时代变了。

2010 年后，互联网爆发。用户量从百万到亿级，日活从几万到千万。每秒数万次请求，每秒数万次数据库读写。传统的关系型数据库开始力不从心。

与此同时，业务场景也在分化：

- 电商大促时，热点商品被疯狂访问，需要缓存
- 社交平台的用户关系、feed 流，需要图数据库或文档数据库
- IoT 设备每秒钟上报数据，需要时序数据库
- 日志、监控、用户行为，需要列式存储

**没有一个数据库能解决所有问题。** 这就是为什么我们需要这么多种数据库。

## 数据库的分类体系

现代数据库可以按照多个维度进行分类：

### 按数据模型分类


## CAP 定理与数据库选型

分布式数据库领域有一个著名的 CAP 定理：

> **一个分布式系统最多只能同时满足一致性（Consistency）、可用性（Availability）、分区容错性（Partition tolerance）中的两个。**

```
           CAP 定理
    ┌─────────────┴─────────────┐
    │                           │
 Consistency              Availability
   (一致性)                 (可用性)
    │                           │
    │   ┌───────────────────────┤
    │   │                       │
    └────┼───────────────────────┘
         │
    Partition Tolerance
       (分区容错性)
       
       必须满足
```

这意味着在网络分区发生时，我们必须在一致性和可用性之间做出选择：

- **CP 数据库**：优先保证一致性。牺牲可用性，意味着在网络分区时可能无法服务。典型代表：Zookeeper、HBase。
- **AP 数据库**：优先保证可用性。牺牲一致性，意味着可能读到过期数据。典型代表：Cassandra、Eureka。

:::tip 理解误区
CAP 定理并不是说在正常情况下我们必须在 C 和 A 之间权衡。它描述的是**网络分区发生时的行为**。在正常网络环境下，我们可以同时获得良好的一致性和可用性。
:::

## 关系型数据库：坚实的老大哥

关系型数据库（RDBMS）是数据库领域的奠基石。即使 NoSQL 大行其道的今天，它仍然是大多数业务系统的首选。

### MySQL：开源世界的王者

MySQL 是全球使用最广泛的开源关系型数据库。它的发展历程就是一部互联网数据库的进化史：

- 1995 年，MySQL AB 发布首个版本
- 2008 年，被 Sun Microsystems 收购
- 2010 年，Oracle 收购 Sun，MySQL 进入 Oracle 时代

MySQL 的成功秘诀在于：**足够简单、足够快、足够稳、足够免费**。

```bash
# MySQL 的部署极其简单
docker run -d --name mysql \
  -e MYSQL_ROOT_PASSWORD=secret \
  -p 3306:3306 \
  mysql:8.0
```

MySQL 的 InnoDB 存储引擎是面试中的高频考点：B+ 树索引、MVCC 并发控制、两阶段锁协议、行锁与表锁——每一个点都可能成为深挖的方向。

### PostgreSQL：功能最强大的开源数据库

如果说 MySQL 是"够用就好"，那 PostgreSQL 就是"应有尽有"。

PostgreSQL 不仅是关系型数据库，还支持：

- JSON/JSONB 存储（半结构化数据）
- 数组类型
- 范围类型
- GIS 地理信息（PostGIS 扩展）
- 自定义类型和函数
- 物化视图、窗口函数、CTE
- 全文检索

```sql
-- PostgreSQL 的 JSONB 查询
SELECT * FROM orders
WHERE metadata -> 'customer' ->> 'vip' = 'true'
  AND created_at >= NOW() - INTERVAL '30 days';
```

PostgreSQL 的 MVCC 实现与 MySQL 不同，使用的是"每事务一行版本"（XID）机制，而不是 MySQL 的 Undo 链 + ReadView 方式。

### Oracle：企业级市场的霸主

Oracle 数据库是商业数据库领域的绝对霸主，在金融、电信、政府等关键行业占据主导地位。

Oracle 的优势在于：

- **稳定性**：经过几十年打磨，故障率极低
- **功能完整**：PL/SQL、 flashback、Data Guard、RAC，功能应有尽有
- **性能优化**：先进的优化器、成本模型、执行计划调优
- **生态完善**：配套工具、开发支持、技术服务

但 Oracle 的价格也是"霸主级别"的——一套企业版 license 可能高达数十万美元。这正是开源数据库崛起的契机。

## NoSQL 数据库：打破关系模型的边界

NoSQL（Not Only SQL）并非"不要 SQL"，而是"不仅仅 SQL"。它解决的是关系型数据库在某些场景下的局限性。

### Redis：内存数据库的性能神话

Redis 是 Key-Value 存储的代表作，但它的能力远超简单的键值存储。

**Redis 的五大数据类型：**

- `String`：缓存、计数器、分布式锁
- `List`：最新消息列表、消息队列
- `Hash`：对象存储、表数据缓存
- `Set`：标签系统、好友关系、去重
- `ZSet`：排行榜、延迟队列、有序消息

Redis 的单线程模型是面试高频题。很多人质疑：单线程不会成为瓶颈吗？

答案在于：**Redis 的瓶颈从来不是 CPU，而是内存和网络 I/O**。单线程避免了锁竞争，反而让它更加高效。

```bash
# Redis 的 QPS 测试
redis-benchmark -n 100000 -c 50
# ====== 100000 requests in 1.23 seconds =====
# 81300.81 requests per second
```

### MongoDB：文档型数据库的灵活之美

MongoDB 是文档型 NoSQL 的代表，它的核心概念是**文档（Document）**。

```json
// MongoDB 文档示例：一个用户对象
{
  "_id": ObjectId("..."),
  "name": "张三",
  "email": "zhangsan@example.com",
  "profile": {
    "age": 28,
    "city": "北京",
    "interests": ["读书", "旅游", "摄影"]
  },
  "orders": [
    { "orderId": "ORD001", "amount": 299.00 },
    { "orderId": "ORD002", "amount": 599.00 }
  ]
}
```

对比 MySQL 的表结构设计，MongoDB 的文档模型更加灵活。字段可以随时增减，无需 ALTER TABLE。对于 Schema 变化频繁的业务（如用户画像、电商商品属性），MongoDB 是更好的选择。

### HBase：海量数据的列式存储

HBase 是构建在 HDFS 之上的 NoSQL 列式数据库，专为海量数据存储而设计。

如果说 MySQL 的一张表能存百万级数据，那 HBase 的一张表能存**万亿级数据**。

HBase 的核心特性：

- **海量存储**：PB 级数据轻松存储
- **高并发**：支持每秒数十万次读写
- **强一致性**：CAP 中选择 CP，保证数据一致
- **自动分区**：数据自动 split，无需人工干预
- **Hadoop 生态**：与 HDFS、Hive、Spark 深度集成

HBase 的架构设计决定了它适合**写多读少、跨度查询**的场景，比如监控数据、日志数据、历史订单。

## 时序数据库：数据驱动的新物种

随着 IoT 物联网、监控系统、金融行情的爆发，时序数据呈指数级增长。时序数据库（TSDB）专门为这类数据而生。

### 时序数据的特点

时序数据有四个核心概念：

1. **时间线（Time Series）**：同一个测量指标的所有数据点序列
2. **度量（Metric）**：测量的物理量，如 CPU 使用率、股票价格
3. **标签（Tag）**：描述数据的维度，如主机名、服务名、区域
4. **数据点（Data Point）**：时间戳 + 数值的组合

```
┌──────────────┬───────────────┬─────────────────────┐
│   时间戳      │   标签        │     值              │
│  timestamp   │   tags        │     value           │
├──────────────┼───────────────┼─────────────────────┤
│ 1700000000   │ host=server1  │ cpu=75.5            │
│ 1700000005   │ host=server1  │ cpu=78.2            │
│ 1700000010   │ host=server1  │ cpu=72.1            │
│ 1700000000   │ host=server2  │ cpu=45.0            │
└──────────────┴───────────────┴─────────────────────┘
```

时序数据库的独特设计，使得它能够高效处理这类数据：

- **数据按时间排序写入**，写入性能极高
- **数据按时间范围查询**，范围查询极快
- **数据压缩存储**，节省存储空间
- **数据保留策略**，自动过期清理

### 主流时序数据库对比

| 特性 | InfluxDB | TimescaleDB | TDengine | Prometheus |
| --- | --- | --- | --- | --- |
| 数据模型 | measurements/tags/fields | hypertables | supertables | metrics/labels |
| 查询语言 | InfluxQL/Flux | SQL | TAOS SQL | PromQL |
| 部署方式 | 单机/集群 | PostgreSQL 扩展 | 单机/集群 | 联邦 |
| 压缩算法 | TSM | Chunk 压缩 | 行列混合压缩 | 4字节压缩 |
| 许可 | MIT | Apache | AGPL | Apache |

## 数据库横向对比

面对如此多的数据库选择，我们如何做出决策？

| 场景 | 推荐选择 | 理由 |
| --- | --- | --- |
| Web 应用核心数据 | MySQL/PostgreSQL | 成熟稳定，生态完善 |
| 高并发缓存 | Redis | 内存级性能，丰富数据结构 |
| Schema 频繁变化 | MongoDB | 灵活文档模型 |
| 海量日志/监控 | HBase/ClickHouse | 列式存储，高压缩比 |
| IoT 时序数据 | InfluxDB/TDengine | 专为时序优化 |
| 金融级事务 | Oracle/TiDB | 高可用，强一致性 |
| 全文搜索 | Elasticsearch | 倒排索引，模糊查询 |
| 图关系分析 | Neo4j | 图遍历高效 |

## 总结

数据库是软件系统的基础设施，选型正确事半功倍，选型错误后患无穷。

本系列的目标是帮助你：

- **理解各类数据库的定位和特点**，在项目中做出正确的选型决策
- **深入理解核心原理**，在面试中从容应对，在工作中解决实际问题
- **构建完整的数据库知识体系**，从 CRUD 到架构设计，从单库优化到分布式存储

不同的数据库适合不同的场景，没有银弹，只有最适合的选择。深入理解每种数据库的设计哲学，才能在纷繁复杂的技术选型中做出明智的决策。
