# 审计插件与安全增强

数据库是数据的金库，审计就是金库的监控摄像头。它记录「谁在什么时间做了什么」，在数据泄露事件发生时提供溯源能力，在合规审计时提供证据链。

## 审计不是可选项

很多团队对审计不屑一顾：「我们是小公司，没人会攻击」。这种想法很危险。

审计的价值不止是「防坏人」，还有：

- **内部风险**：员工误删数据、恶意操作权限范围内的数据
- **合规要求**：等保三级、金融行业监管、GDPR，都要求操作日志
- **故障排查**：数据库抖动时，审计日志能帮你定位是哪个 SQL 惹的祸
- **性能分析**：慢查询可能是某个定时任务没优化，审计日志能定位

审计日志是数据库安全的最后一道防线，有了它，即使防线被突破，也能知道发生了什么。

---

## MariaDB 审计插件

MariaDB 提供官方的审计插件 `server_audit`，安装简单：

```sql
-- 安装审计插件
INSTALL SONAME 'server_audit';

-- 查看审计插件状态
SHOW PLUGINS;

-- 验证已加载
SELECT * FROM information_schema.PLUGINS WHERE PLUGIN_NAME = 'SERVER_AUDIT';
```

安装后，审计日志默认写入 `server_audit.log` 文件（在数据目录下）。

## 基础配置

审计插件的行为由系统变量控制：

```sql
-- 查看当前配置
SHOW VARIABLES LIKE 'server_audit%';

-- 推荐配置
SET GLOBAL server_audit_logging = 'ON';
SET GLOBAL server_audit_output_type = 'FILE';  -- 或 SYSLOG
SET GLOBAL server_audit_events = 'CONNECT,QUERY,TABLE';
SET GLOBAL server_audit_file_path = '/var/log/mariadb/audit.log';
SET GLOBAL server_audit_file_rotate_size = 10000000;  -- 10MB
SET GLOBAL server_audit_file_rotations = 10;  -- 保留 10 个文件
```

`server_audit_events` 是核心配置项：

| 值 | 记录内容 |
|---|---|
| `CONNECT` | 连接/断开连接事件 |
| `QUERY` | 所有 SQL 语句及结果 |
| `TABLE` | 表的读写操作（SELECT、INSERT、UPDATE、DELETE） |
| `QUERY_DDL` | DDL 语句（CREATE、DROP、ALTER） |
| `QUERY_DML` | DML 语句（INSERT、UPDATE、DELETE） |
| `QUERY_DCL` | DCL 语句（GRANT、REVOKE） |

**生产环境推荐**：`CONNECT,QUERY_DDL,QUERY_DML,TABLE`。记录所有连接、所有写操作、所有表操作，但不必记录所有 SELECT 查询（除非是安全要求极高的场景）。

## 解读审计日志

审计日志格式示例：

```log
20240315 14:32:01,localhost,mariauser,Query,connect,0,
  "SELECT * FROM users WHERE id = 1"
20240315 14:32:05,localhost,mariauser,Query,0,
  "UPDATE users SET email='hacker@evil.com' WHERE id = 1"
20240315 14:32:10,192.168.1.100,appuser,Query,,
  "DELETE FROM orders WHERE status = 'pending' AND created_at < '2023-01-01'"
```

每行格式：`时间戳,主机,用户,类型,操作,返回码,SQL语句`

注意最后一行的 `DELETE FROM orders`——如果没有审计，你根本不知道这个批量删除是谁执行的、什么时候执行的。

## 高级配置：按用户过滤

生产环境中，你可能不想记录所有用户的所有操作：

```sql
-- 不记录某些内部用户
SET GLOBAL server_audit_excl_users = 'backup_user,monitor_user';

-- 只记录特定用户
SET GLOBAL server_audit_incl_users = 'app_user';
```

同时配置 `incl_users` 和 `excl_users` 时，`excl_users` 优先级更高。

## 安全加固：不只是审计

审计是事后的，安全加固是事前的。MariaDB 提供多层安全机制：

### 1. 密码策略

```sql
-- 安装密码验证插件（强制复杂密码）
INSTALL SONAME 'simple_password_check';

-- 配置密码策略
SET GLOBAL simple_password_check_digits = 1;      -- 至少一位数字
SET GLOBAL simple_password_check_minimal_len = 12; -- 最小长度
SET GLOBAL simple_password_check_mixed_case = 1;   -- 大小写混合
SET GLOBAL simple_password_check_special = 1;     -- 特殊字符
```

### 2. 强制的 SSL 连接

```sql
-- 配置 MariaDB 强制 SSL
SET GLOBAL require_secure_transport = ON;

-- 查看连接是否加密
SHOW STATUS LIKE 'SSL_cipher';
```

### 3. 角色基础访问控制（RBAC）

MariaDB 10.5+ 支持角色：

```sql
-- 创建角色
CREATE ROLE 'app_read', 'app_write', 'app_admin';

-- 授予角色权限
GRANT SELECT ON app_db.* TO 'app_read';
GRANT SELECT, INSERT, UPDATE, DELETE ON app_db.* TO 'app_write';
GRANT ALL ON app_db.* TO 'app_admin';

-- 将角色分配给用户
GRANT 'app_read' TO 'app_user'@'%';
GRANT 'app_write' TO 'service_account'@'%';

-- 用户激活角色
SET DEFAULT ROLE 'app_read' FOR 'app_user'@'%';
```

### 4. 列级和行级安全

通过视图实现行级安全：

```sql
-- 只让用户看到自己的数据
CREATE VIEW user_orders AS
SELECT *
FROM orders
WHERE user_id = CURRENT_USER();  -- 需要配合插件实现
```

MariaDB 企业版提供更强大的数据脱敏和行级安全功能。

## 日志管理

审计日志会持续增长，需要妥善管理：

**1. 日志轮转**

审计插件支持自动轮转，配置 `server_audit_file_rotate_size` 和 `server_audit_file_rotations` 即可。

**2. 归档到日志系统**

```sql
-- 输出到 syslog
SET GLOBAL server_audit_output_type = 'SYSLOG';
SET GLOBAL server_audit_syslog_facility = 'LOG_LOCAL0';
SET GLOBAL server_audit_syslog_priority = 'LOG_INFO';
```

配合 `rsyslog` 将日志发送到 ELK、Splunk 等日志分析平台。

**3. 定期清理**

```bash
# 清理超过 90 天的审计日志
find /var/log/mariadb -name "server_audit.*" -mtime +90 -delete
```

## 审计与性能

审计不可避免地带来性能开销：

| 配置 | 性能影响 | 适用场景 |
|---|---|---|
| 记录所有 QUERY | 高（20-30%） | 极高安全要求 |
| 只记录 DDL/DML + TABLE | 中（5-10%） | 生产推荐 |
| 只记录 CONNECT + DDL | 低（2-3%） | 资源紧张 |

如果性能敏感，优先记录写操作（INSERT/UPDATE/DELETE）和 DDL，SELECT 查询除非必要可以不记录。

---

## 总结

审计是数据库安全体系的重要组成部分：

- **审计插件**记录所有连接和操作，提供事后溯源能力
- **密码策略、SSL、RBAC**提供事前防御机制
- **日志管理**确保审计系统本身不会成为瓶颈

<Badge text="合规要求" type="warning" /> 如果你的系统需要满足等保三级、PCI-DSS、GDPR 等合规要求，操作审计日志是**强制要求**。提前规划，不要等到审计时才发现日志缺失。
