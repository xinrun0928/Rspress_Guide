# 窗口函数与 CTE

写 SQL 久了，你会发现很多查询场景让人头疼：算累计和要嵌套子查询，算排名要自联接，按组内排名更要三层嵌套。有没有更优雅的方式？

窗口函数和 CTE（公共表表达式）就是答案。

---

## 窗口函数：分组之外的聚合

### 核心概念

窗口函数的本质是：**在「不改变行数」的前提下，对行周围的数据执行计算**。

这听起来有点抽象，对比一下就清楚了：

```sql
-- 传统聚合：行数减少
SELECT department, SUM(salary) FROM employees GROUP BY department;

-- 窗口聚合：行数不变
SELECT name, department, salary,
       SUM(salary) OVER (PARTITION BY department) as dept_total
FROM employees;
```

第二种写法，SUM 变成了窗口函数，`OVER` 定义了窗口的范围。每个员工行都保留，同时多了一列部门工资总额。

---

## 常用窗口函数

### 聚合类窗口函数

所有标准聚合函数都可以作为窗口函数使用：

```sql
SELECT
    order_date,
    amount,
    -- 部门内累计
    SUM(amount) OVER (PARTITION BY DATE_FORMAT(order_date, '%Y-%m')) as monthly_total,
    -- 累计平均值
    AVG(amount) OVER (ORDER BY order_date ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) as running_avg,
    -- 占总额比例
    amount / SUM(amount) OVER () * 100 as pct_of_total
FROM orders;
```

`ROWS BETWEEN` 定义了窗口的物理范围：

| 语法 | 说明 |
|---|---|
| `UNBOUNDED PRECEDING` | 窗口从第一行开始 |
| `CURRENT ROW` | 当前行 |
| `n PRECEDING` | 前 n 行 |
| `n FOLLOWING` | 后 n 行 |
| `UNBOUNDED FOLLOWING` | 窗口到最后一行为止 |

### 排名类窗口函数

这是窗口函数最常用的场景：

```sql
SELECT
    name,
    department,
    salary,
    -- 组内排名（相同值排名相同，跳过后续名次）
    RANK() OVER (PARTITION BY department ORDER BY salary DESC) as dense_rank,
    -- 组内排名（相同值排名相同，不跳过名次）
    ROW_NUMBER() OVER (PARTITION BY department ORDER BY salary DESC) as row_num,
    -- 组内排名（相同值排名相同，完整跳过）
    DENSE_RANK() OVER (PARTITION BY department ORDER BY salary DESC) as dense_rank,
    -- 百分位排名
    PERCENT_RANK() OVER (PARTITION BY department ORDER BY salary DESC) as pct_rank
FROM employees;
```

三种排名函数的区别：

| 函数 | 1, 2, 2, 4 | 适用场景 |
|---|---|---|
| ROW_NUMBER | 1, 2, 3, 4 | 每行需要唯一编号 |
| RANK | 1, 2, 2, 4 | 允许并列，需要「完整名次」 |
| DENSE_RANK | 1, 2, 2, 3 | 允许并列，名次连续 |

### 偏移类窗口函数

用于获取「上下行」的数据：

```sql
SELECT
    order_date,
    amount,
    -- 上一个订单金额
    LAG(amount, 1) OVER (ORDER BY order_date) as prev_amount,
    -- 上一个订单日期
    LAG(order_date, 1) OVER (ORDER BY order_date) as prev_date,
    -- 下一个订单金额
    LEAD(amount, 1) OVER (ORDER BY order_date) as next_amount,
    -- 与上一单的差值
    amount - LAG(amount, 1) OVER (ORDER BY order_date) as amount_diff
FROM orders;
```

`LAG` 和 `LEAD` 是计算环比、同比的利器。电商场景下计算「上一个月的销售额」，过去七天的 DAU 变化，都靠这两个函数。

## CTE：让复杂查询可读

### 基本语法

CTE（Common Table Expression，公共表表达式）本质上是「命名子查询」，让 SQL 可读性大幅提升：

```sql
WITH monthly_sales AS (
    SELECT
        DATE_FORMAT(order_date, '%Y-%m') as month,
        SUM(amount) as total
    FROM orders
    GROUP BY month
)
SELECT
    month,
    total,
    total - LAG(total, 1) OVER (ORDER BY month) as mom_diff
FROM monthly_sales
ORDER BY month;
```

没有 CTE，这段 SQL 要么嵌套两层子查询，要么拆成两个临时表。CTE 让逻辑清晰分层。

### 递归 CTE

递归 CTE 是 MariaDB 10.2+ 的特性，用于处理层级结构：

```sql
-- 递归查询组织架构
WITH RECURSIVE org_tree AS (
    -- 基础用例：CEO
    SELECT id, name, manager_id, 1 as level
    FROM employees
    WHERE manager_id IS NULL

    UNION ALL

    -- 递归用例：下属
    SELECT e.id, e.name, e.manager_id, ot.level + 1
    FROM employees e
    JOIN org_tree ot ON e.manager_id = ot.id
)
SELECT * FROM org_tree ORDER BY level, name;
```

递归 CTE 特别适合：组织架构、菜单树、路径规划、账单追溯等场景。

## 典型应用场景

### 场景一：计算七日留存

```sql
WITH user_actions AS (
    SELECT user_id, DATE(action_time) as action_date
    FROM events
    WHERE action = 'login'
),
user_first_day AS (
    SELECT user_id, MIN(action_date) as first_date
    FROM user_actions
    GROUP BY user_id
),
retention AS (
    SELECT
        DATEDIFF(a.action_date, f.first_date) as day_num,
        COUNT(DISTINCT a.user_id) as users
    FROM user_actions a
    JOIN user_first_day f ON a.user_id = f.user_id
    GROUP BY day_num
)
SELECT
    day_num,
    users,
    ROUND(users * 100.0 / (SELECT users FROM retention WHERE day_num = 0), 2) as retention_rate
FROM retention;
```

### 场景二：计算用户最近 N 次购买

```sql
SELECT *
FROM (
    SELECT
        o.*,
        ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY order_date DESC) as rn
    FROM orders o
) ranked
WHERE rn `<=` 3;  -- 取每个用户最近三次订单
```

## 性能考量

窗口函数在 MariaDB 10.3+ 有显著优化，但仍有需要注意的点：

**1. 大数据量慎用递归 CTE**

递归 CTE 如果没有合理的终止条件，可能无限循环。每次递归都是一次查询，数据量大时要评估执行计划。

```sql
EXPLAIN WITH RECURSIVE cnt(x) AS (
    SELECT 1
    UNION ALL
    SELECT x + 1 FROM cnt WHERE x `<` 1000000
)
SELECT * FROM cnt;  -- 生成 100 万行，谨慎执行
```

**2. 窗口分区要建索引**

如果按时间窗口查询大量数据，在 `OVER (ORDER BY date)` 的列上建索引能大幅提升性能。

**3. 避免在窗口函数中嵌套子查询**

```sql
-- 低效
SELECT *,
       (SELECT SUM(amount) FROM orders WHERE date `<=` o.date) as cumulative
FROM orders o;

-- 高效
SELECT *,
       SUM(amount) OVER (ORDER BY date) as cumulative
FROM orders;
```

## 总结

窗口函数和 CTE 是现代 SQL 的标配，它们让复杂查询变得可读、可维护：

- **窗口函数**：在保持行数不变的前提下，计算聚合值、排名、偏移量
- **CTE**：将复杂查询分解为命名子查询，递归 CTE 还能处理层级数据

MariaDB 对这两个特性的支持非常完善（10.2+），而且性能优秀。如果你还在用子查询和自联接实现排名、累计功能，是时候迁移到窗口函数了。

<Badge text="提示" type="tip" /> 学习窗口函数最好的方式是模仿。从简单场景开始：计算每行的累计和、组内排名。掌握这些基础后，再探索更复杂的窗口定义。
