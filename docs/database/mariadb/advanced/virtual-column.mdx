# 虚拟列与序列

传统的数据表设计，要求每个列都对应一个存储值。但有时，你需要的「列」是由其他列计算出来的——比如总价等于单价乘以数量、年龄由出生日期推算。这些「计算出来」的列，是虚拟列的用武之地。

序列则是 AUTO_INCREMENT 的升级版，解决自增主键在分布式场景下的局限性。

## 虚拟列：从「存储」到「计算」

### 基本概念

虚拟列（Virtual Column）的值由表达式计算得出，不存储在磁盘上。定义语法：

```sql
CREATE TABLE orders (
    id INT PRIMARY KEY,
    unit_price DECIMAL(10,2),
    quantity INT,
    total_price DECIMAL(10,2) AS (unit_price * quantity),  -- 虚拟列
    status VARCHAR(20),
    is_active BOOLEAN AS (status = 'active')  -- 布尔虚拟列
);
```

`AS (表达式)` 定义了虚拟列的计算方式。MariaDB 在查询时自动计算，不需要你手动维护。

虚拟列有两种类型：

| 类型 | 说明 | 存储 |
|---|---|---|
| VIRTUAL | 每次查询时计算 | 不占用磁盘空间 |
| PIRTUAL | 插入/更新时计算并存入磁盘 | 占用磁盘空间 |

默认是 VIRTUAL。PSTORED 的优势是查询更快，代价是占用更多存储空间，且修改源数据时需要更新虚拟列。

### 为虚拟列建索引

这是虚拟列的杀手级特性——**可以对虚拟列建立索引**：

```sql
CREATE TABLE users (
    id INT PRIMARY KEY,
    name VARCHAR(50),
    email VARCHAR(100),
    domain VARCHAR(50) AS (SUBSTRING_INDEX(email, '@', -1)) VIRTUAL,
    INDEX idx_domain (domain)  -- 在虚拟列上建索引
);

-- 查询邮件域名的效率大幅提升
SELECT * FROM users WHERE domain = 'gmail.com';
```

这个例子展示了虚拟列 + 索引的威力：如果按邮箱后缀查询，每次都要 `SUBSTRING_INDEX()` 全表扫描；有了虚拟列索引，查询变成索引查找，性能提升几十倍。

## 物化列：预计算的空间换时间

当虚拟列的表达式很复杂，查询频率又极高时，可以考虑物化列（STORED VIRTUAL COLUMN）：

```sql
CREATE TABLE sales (
    id INT PRIMARY KEY,
    sale_date DATE,
    amount DECIMAL(10,2),
    quarter TINYINT AS (EXTRACT(QUARTER FROM sale_date)) STORED,
    year_val SMALLINT AS (EXTRACT(YEAR FROM sale_date)) STORED,
    INDEX idx_quarter_year (quarter, year_val)
);
```

`STORED` 关键字告诉 MariaDB：将计算结果存入磁盘。查询时直接读存储的值，省去计算开销。

**适用场景**：
- 表达式很复杂，计算成本高
- 查询频率远高于写入频率
- 有足够的磁盘空间

**不适用场景**：
- 写入频繁，计算成本反而成为瓶颈
- 存储空间紧张

## 序列：超越 AUTO_INCREMENT

### AUTO_INCREMENT 的局限性

`AUTO_INCREMENT` 是 MySQL/MariaDB 传统的自增主键方案，但在分布式场景下有三个明显问题：

**问题一：多实例主键冲突**。两个数据库实例都从 1 开始自增，合并数据时主键必然冲突。

**问题二：无法共享**。多个表需要共用同一个序列（如订单号统一编号），AUTO_INCREMENT 无法实现。

**问题三：步长固定**。自增步长默认是 1，无法改成其他值。

### 序列的解决方案

MariaDB 10.0 引入了序列（Sequence），解决了以上所有问题：

```sql
-- 创建序列
CREATE SEQUENCE seq_order START WITH 1000 INCREMENT BY 1;

-- 使用序列获取下一个值
SELECT NEXT VALUE FOR seq_order;  -- 返回 1000
SELECT NEXT VALUE FOR seq_order;  -- 返回 1001

-- 在表定义中引用序列
CREATE TABLE orders (
    order_id INT DEFAULT (NEXT VALUE FOR seq_order),
    customer_id INT,
    amount DECIMAL(10,2)
);
```

### 序列的高级特性

**1. 循环序列**

```sql
CREATE SEQUENCE seq_session
    START WITH 1000
    INCREMENT BY 1
    MAXVALUE 9999
    CYCLE;  -- 到达上限后从头开始
```

**2. 预分配序列值**

```sql
-- 一次获取多个序列值
SELECT NEXT VALUE FOR seq_order, NEXT VALUE FOR seq_order, NEXT VALUE FOR seq_order;
-- 返回: 1000, 1001, 1002

-- 或者用数组语法
SELECT * FROM seq_order LIMIT 3 WITH VALUES;
```

**3. 查询序列状态**

```sql
SELECT * FROM seq_order;

-- 输出:
-- | seq_name   | next_value | min_value | max_value | increment | cycle |
-- | seq_order  | 1000       | 1         | 9223372036854775807 | 1        | 0     |
```

## 组合使用：虚拟列 + 序列

虚拟列和序列可以组合使用，实现复杂的数据建模：

```sql
-- 业务编码规则：年份(4位) + 序号(6位)
CREATE SEQUENCE seq_business START WITH 1 INCREMENT BY 1;

CREATE TABLE business_orders (
    id INT PRIMARY KEY,
    order_year SMALLINT AS (EXTRACT(YEAR FROM create_time)) STORED,
    order_seq BIGINT DEFAULT (NEXT VALUE FOR seq_business),
    order_code VARCHAR(20) AS (CONCAT(order_year, LPAD(order_seq, 6, '0'))) STORED,
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    amount DECIMAL(10,2)
);
```

现在 `order_code` 列自动生成为 `202401000001` 这样的格式，你不需要在应用层拼接字符串。

## 注意事项与限制

### 虚拟列的限制

虚拟列的表达式必须是**确定性**的——相同输入必须产生相同输出。下列写法不允许：

```sql
-- 错误：NOW() 每次调用返回值不同
CREATE TABLE t (
    id INT,
    now_col DATETIME AS (NOW())
);

-- 错误：RAND() 是非确定性的
CREATE TABLE t (
    id INT,
    random_val INT AS (FLOOR(RAND() * 100))
);
```

### 序列的限制

- 序列是 MariaDB 独有的，迁移到 MySQL/PostgreSQL 需要修改
- 序列值不会回滚，即使事务回滚，已分配的序列值也不会复用
- 序列与 `LAST_INSERT_ID()` 不兼容，批量插入时需注意

---

## 总结

虚拟列和序列是 MariaDB 10.0+ 引入的重要特性，它们改变了数据建模的方式：

- **虚拟列**：将计算逻辑下推到数据库层，支持索引优化，适合「查询频率高、表达式复杂」的场景
- **序列**：提供比 AUTO_INCREMENT 更灵活的主键方案，适合分布式系统和跨表共享场景

<Badge text="实践" type="tip" /> 在设计表结构时，先问：这个列能否由其他列计算得出？如果是，优先考虑虚拟列而非存储冗余数据。数据一致性比一点点存储空间更值钱。
