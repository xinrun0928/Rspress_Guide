# 线程池插件

想象一个餐厅的场景：高峰期来了 1000 位客人，但厨房只有 10 个厨师。传统做法是——每位客人配备一个专属厨师，服务完一位再服务下一位。这样厨师的大部分时间都在等待客人点单、等待客人离开，真正炒菜的时间反而很少。

MariaDB 的线程池，解决的就是这个问题。

---

## 为什么传统模型在高并发下失效

MySQL/MariaDB 的传统连接模型是「Thread Per Connection」——每个客户端连接对应一个操作系统线程。这个模型简单直接，但有三个致命问题：

**线程创建销毁开销大**。创建一个线程需要分配栈空间、初始化调度数据结构。即使是轻量级线程，开销也在几百微秒到几毫秒。对于短查询为主的 Web 应用，这个开销可能比实际查询时间还长。

**内存占用高**。每个线程默认栈空间 1MB，1000 个连接就需要 1GB 内存仅用于线程栈。更糟糕的是，线程是「借」给连接的，空闲连接的线程也在占用内存。

**上下文切换频繁**。当连接数远超 CPU 核心数时，操作系统需要不断切换线程。上下文切换涉及寄存器保存/恢复、缓存失效，大量切换会导致 CPU 空转。

---

## 线程池的工作原理

线程池的核心思想是「线程复用」——创建一组常驻线程，让它们循环处理多个客户端请求。

```
传统模型：连接1 → 线程A → 连接1断开 → 线程A销毁
                     ↓
         连接1000 → 线程A创建 → 连接1000断开 → 线程A销毁

线程池模型：线程A ←→ 连接1 ←→ 连接2 ←→ 连接3 ←→ ...
            线程B ←→ 连接4 ←→ 连接5 ←→ 连接6 ←→ ...
            ...（固定数量的线程循环处理大量连接）
```

MariaDB 企业版的线程池使用分层架构：

| 层级 | 说明 |
|---|---|
| 连接调度层 | 将新连接分配到不同的「任务组」 |
| 任务组 | 按客户端 IP 或用户名分组，控制资源竞争 |
| 线程池 | 每个任务组内维护独立的工作线程队列 |
| 监听线程 | 等待客户端请求，有请求时唤醒等待线程 |

这种分层设计让不同来源的请求相互隔离，避免单个「疯狂客户端」拖垮整个系统。

## 社区版 vs 企业版

MariaDB 线程池有两个版本：社区版的 `thread_pool` 插件和企业版的 Thread Pool。

社区版插件功能相对基础：

```sql
-- 安装线程池插件
INSTALL SONAME 'thread_pool';

-- 查看线程池状态
SHOW STATUS LIKE 'Thread_pool%';
```

企业版则提供更多高级特性：

- **动态调整**：运行时调整线程池大小，无需重启
- **优先级队列**：区分高优先级和低优先级请求
- **更好的隔离性**：基于用户/IP 的资源分组

如果你的并发量不大（百级别连接），社区版足够；如果需要处理千级并发，企业版更稳定。

## 性能对比：线程池何时胜出

线程池的优势不是绝对的。来看一组对比数据（MariaDB 10.6，单机 32 核）：

| 场景 | 传统模型 (thread_handling=one-thread-per-connection) | 线程池 |
|---|---|---|
| 100 并发短查询 | 基准 | 提升约 30% |
| 500 并发短查询 | 性能下降 | 提升约 50% |
| 1000 并发短查询 | 显著下降 | 提升约 80% |
| 单客户端大查询 | 基准 | 略有下降 |
| 长事务场景 | 基准 | 可能下降 |

**结论**：线程池最适合「高并发、短查询」场景。判断依据是**查询时间远小于连接建立时间**。如果你的查询本身需要几秒甚至几分钟，线程池的优势就微乎其微了。

## 避坑指南

### 坑一：线程池大小设置不当

线程池大小直接影响性能。太小则请求排队时间长，太大则增加调度开销。

```sql
-- 查看当前配置
SHOW VARIABLES LIKE 'thread_pool_size';
SHOW VARIABLES LIKE 'thread_pool_stall_limit';
```

`thread_pool_size` 默认等于 CPU 核心数，通常这是合理的起点。`thread_pool_stall_limit` 控制线程空闲多久后允许新线程加入（默认 500ms）。对于 OLTP 场景，可以调低到 100-200ms；对于 OLAP 场景，保持默认值或调高。

### 坑二：阻塞操作导致线程饥饿

线程池中的线程如果执行了阻塞操作（如 `SLEEP()`、网络等待），会导致整个组的新请求排队。避免在查询中使用 `SLEEP()`，用应用层超时替代。

### 坑三：与连接池重复

很多应用层框架（如 HikariCP、Druid）已经实现了连接池。再套一层线程池，两层池化反而增加复杂度。如果应用层连接池配置合理，可以关闭线程池。

### 坑四：事务边界问题

在传统模型下，一个连接对应一个事务。在线程池模型下，一个事务可能跨越多个线程。MariaDB 正确处理了这一点，但某些依赖「连接=事务」假设的代码可能出问题。

## 监控与调优

线程池的监控主要关注两个指标：

```sql
-- 查看活跃线程数
SHOW STATUS LIKE 'Thread_pool_active_threads';

-- 查看排队请求数
SHOW STATUS LIKE 'Thread_pool_queued';

-- 查看因线程池满而阻塞的连接数
SHOW STATUS LIKE 'Thread_pool_threads_cached';
```

如果 `Thread_pool_queued` 持续很高，说明线程池不够用了。考虑：增加 `thread_pool_size`、优化查询减少执行时间、或在应用层做请求削峰。

---

## 总结

线程池是 MariaDB 应对高并发的利器。它的本质是**用固定数量的线程处理可变数量的连接**，减少线程创建销毁开销，降低内存占用。

但线程池不是银弹。判断是否需要线程池，关键看你的查询特征：**短查询、高并发、延迟敏感**的场景，线程池能带来显著提升；**长查询、大事务**的场景，传统模型可能更稳定。

<Badge text="最佳实践" type="tip" /> 在启用线程池前，用 `SHOW PROCESSLIST` 分析你的查询分布。如果 90% 的查询在 10ms 以内完成，线程池值得一试；如果大量查询超过 1 秒，先优化查询本身。
