# MariaDB 面试汇总

MariaDB 作为 MySQL 的重要分支，在面试中经常被问到与 MySQL 的对比、Galera 集群、存储引擎选型等问题。本章节汇总高频面试题，帮你快速梳理核心知识点。

---

## 一、基础概念类

### Q1：MariaDB 和 MySQL 是什么关系？为什么会有 MariaDB？

MariaDB 是 MySQL 的一个社区分支，由 MySQL 创始人 Michael "Monty" Widenius 在 2010 年创建。

**历史背景**：
- 2008 年，Sun Microsystems 收购 MySQL AB
- 2010 年，Oracle 收购 Sun Microsystems，获得 MySQL 控制权
- Monty 担心 MySQL 走向闭源，创建了 MariaDB 作为"保险"

**命名来源**：Monty 的小女儿名叫 Maria，数据库名字取自她。

**兼容性**：MariaDB 被设计为 MySQL 的替代品，保持 API 和协议层面的完全兼容，数据文件和命令高度兼容。

---

### Q2：MariaDB 默认使用什么存储引擎？

MariaDB 10.2+ 默认使用 InnoDB（实际上是 XtraDB，它是 InnoDB 的增强版）。

MariaDB 对 InnoDB 做了大量改进：
- XtraDB 存储引擎作为默认引擎
- 更好的并发性能
- 改进的崩溃恢复
- 增强的缓冲池管理

---

### Q3：MariaDB 有哪些独特的存储引擎？

| 引擎 | 特点 | 适用场景 |
|---|---|---|
| XtraDB | InnoDB 的增强版 | 默认引擎，通用场景 |
| Aria | MyISAM 的增强版，崩溃安全 | 读密集型、内部表 |
| TokuDB | 分形树索引，高压缩 | 大数据量归档 |
| MyRocks | RocksDB 引擎，高写入压缩 | 日志、时序数据 |
| CONNECT | 支持异构数据源 | CSV、XML、ODBC |
| Spider | 分片存储引擎 | 分库分表 |

## 二、Galera Cluster 相关

### Q4：什么是 Galera Cluster？它和 MySQL 主从复制有什么区别？

**Galera Cluster** 是 MariaDB 提供的一种同步多主集群解决方案，多个节点可以同时接受读写请求。

| 对比项 | MySQL 主从复制 | Galera Cluster |
|---|---|---|
| 复制模式 | 异步 | 同步 |
| 写入能力 | 单主写入 | 多主写入 |
| 一致性 | 最终一致 | 强一致 |
| 故障转移 | 需要手动切换 | 自动故障转移 |
| 延迟影响 | 从库可能延迟 | 无延迟问题 |

**Galera 的核心原理**：
1. Write-Set 传播：节点提交事务时，将 Write-Set 广播给所有节点
2. 认证（Certify）：所有节点对 Write-Set 进行冲突检测
3. 应用（Apply）：通过的事务在所有节点应用

---

### Q5：Galera 的 SST 和 IST 是什么？

**SST（State Snapshot Transfer）**：全量状态同步

当新节点加入或 IST 不可用时触发，SST 会将整个数据库的全量数据传输给新节点。

常见 SST 方法：
- `mysqldump`：逻辑备份，最慢但最通用
- `xtrabackup`：物理备份，速度较快
- `rsync`：物理文件复制，速度最快

**IST（Incremental State Transfer）**：增量状态同步

当新节点加入且集群状态还在缓存中时触发，只传输缺失的 Write-Set。

**选择依据**：
- IST 要求捐赠节点缓存足够长时间，配置项 `gcache.size`
- 大数据量场景优先选择 xtrabackup SST

---

### Q6：Galera Cluster 有哪些局限性？

1. **网络依赖**：节点间网络质量直接影响性能
2. **写放大**：所有节点都需要验证写入，可能降低写入性能
3. **表锁限制**：DDL 操作使用 `wsrep_replicate_myisam=ON` 才有更好的多主支持
4. **自增主键**：需要配置 `auto_increment_offset` 避免冲突
5. **大规模集群**：超过 10 个节点的集群性能会下降

## 三、存储引擎类

### Q7：XtraDB 和 InnoDB 有什么区别？

XtraDB 是 Percona 对 InnoDB 的增强版本，被 MariaDB 采用作为默认存储引擎。

**主要改进**：
- **自适应哈希索引**：改进的实现，更高效
- **页面压缩**：支持 Compressed Page 特性
- **恢复性能**：改进的崩溃恢复速度
- **扩展性**：支持更多并发连接
- **缓冲池管理**：改进的 LRU 算法

**向后兼容**：XtraDB 完全兼容 InnoDB，可以使用 InnoDB 的工具和命令。

---

### Q8：TokuDB 有什么特点？为什么在大数据量场景下表现好？

TokuDB 使用分形树（Fractal Tree）索引作为底层数据结构。

**核心优势**：
1. **高写入性能**：分形树的插入复杂度是 O(log n)，且不会像 B+ 树那样产生大量页面分裂
2. **高压缩比**：支持压缩算法，压缩比可达 10:1 甚至更高
3. **在线 DDL**：支持在线添加索引，不阻塞读写
4. **大数据量支持**：单表可支持数十 TB 数据

**适用场景**：
- 日志类、监控类时序数据
- 历史数据归档
- 写入密集型工作负载

---

### Q9：MyRocks 存储引擎有什么特点？

MyRocks 是基于 RocksDB（Facebook 开发）的存储引擎。

**核心特点**：
1. **LSM-Tree 结构**：写入友好，所有写入先进入 MemTable，然后批量刷盘
2. **极高压缩比**：比 InnoDB 节省 50%-80% 存储空间
3. **写入放大控制**：相比普通 LSM-Tree 有更好的写入放大控制
4. **顺序写入优化**： compaction 过程主要顺序读写

**适用场景**：
- 写入密集型（日志、CDN）
- 存储空间敏感型应用
- 需要高压缩比的场景

## 四、高级特性类

### Q10：MariaDB 的线程池是什么？和 MySQL 企业版线程池有什么区别？

MariaDB 的线程池通过 `thread_pool` 插件实现。

**工作原理**：
- 使用固定数量的线程组（默认 CPU 核心数）
- 每个线程组有自己的任务队列
- 线程在组内动态分配任务
- 空闲线程可跨组获取任务

**与 MySQL 企业版的区别**：
| 对比项 | MariaDB 线程池 | MySQL 企业版线程池 |
|---|---|---|
| 开源 | 完全开源 | 商业版专属 |
| 实现 | thread_pool 插件 | 插件形式 |
| 优先级队列 | 支持 | 支持 |
| 突发查询 | 动态线程创建 | 固定线程 |

**适用场景**：高并发短查询、高并发 OLTP 场景。

---

### Q11：MariaDB 的窗口函数和 MySQL 8.0 相比如何？

MariaDB 10.2+ 支持窗口函数，与 MySQL 8.0+ 相比语法相近。

**支持的窗口函数**：
- `ROW_NUMBER()`、`RANK()`、`DENSE_RANK()`
- `SUM()`、`AVG()`、`COUNT()`
- `LAG()`、`LEAD()`
- `FIRST_VALUE()`、`LAST_VALUE()`
- `CUME_DIST()`、`NTILE()`

**使用示例**：

```sql
-- 计算每个部门的工资排名
SELECT 
    dept,
    name,
    salary,
    RANK() OVER (PARTITION BY dept ORDER BY salary DESC) as dept_rank,
    SUM(salary) OVER (PARTITION BY dept) as dept_total
FROM employees;
```

---

### Q12：CONNECT 存储引擎支持哪些数据类型？

CONNECT 引擎是 MariaDB 独有的，可以将外部数据源当作表来查询。

**支持的数据格式**：
| 类型 | 说明 | 示例 |
|---|---|---|
| DOS | 定长文本文件 | CSV 文件 |
| FIX | 定长格式文件 | 固定宽度文件 |
| BIN | 二进制文件 | C 结构体 |
| XML | XML 格式 | SOAP/配置文件 |
| JSON | JSON 格式 | REST API |
| ODBC | ODBC 数据源 | 其他数据库 |
| JDBC | JDBC 数据源 | Java 数据源 |
| MYSQL | MySQL/MariaDB | 远程表 |
| PROXY | 代理表 | 连接池 |

## 五、JSON 与新特性类

### Q13：MariaDB 的 JSON 支持和 MySQL 相比有什么差异？

MariaDB 10.2+ 引入了 JSON 函数，但实现方式与 MySQL 有所不同。

**MySQL 方式**：提供 `JSON` 数据类型，有专用的 JSON 函数。

**MariaDB 方式**：主要通过虚拟列（Virtual Column）实现类似功能。

```sql
-- MariaDB：使用虚拟列
CREATE TABLE users (
    data JSON,
    name VARCHAR(100) AS (data->>'$.name') VIRTUAL,
    email VARCHAR(100) AS (data->>'$.email') VIRTUAL
);
```

**差异对比**：
| 特性 | MySQL | MariaDB |
|---|---|---|
| JSON 数据类型 | 原生支持 | 支持（10.2+） |
| 虚拟列 | 支持 | 更强大 |
| JSON 函数 | 丰富 | 较少 |
| 索引支持 | 函数索引 | 虚拟列索引 |

---

### Q14：MariaDB 10.6 引入的 System-Versioned Tables 是什么？

系统版本表可以自动保存数据的历史版本。

```sql
CREATE TABLE accounts (
    id INT PRIMARY KEY,
    balance DECIMAL(10,2),
    name VARCHAR(100)
) WITH SYSTEM VERSIONING;

-- 修改数据
UPDATE accounts SET balance = 200 WHERE id = 1;

-- 查询历史版本
SELECT * FROM accounts FOR SYSTEM_TIME AS OF '2024-01-01 00:00:00';

-- 查询所有历史版本
SELECT * FROM accounts FOR SYSTEM_TIME BETWEEN '2024-01-01' AND '2024-12-31';
```

**应用场景**：
- 审计追踪
- 误操作恢复
- 时间点数据分析

## 六、复制与高可用类

### Q15：MariaDB 的 GTID 复制和 MySQL 有什么区别？

GTID（Global Transaction Identifier）是全局事务 ID，MariaDB 和 MySQL 都支持。

**MariaDB GTID 特点**：
- 默认开启
- 支持 `binlog_gtid_pos` 自动定位
- 与 Galera 完全兼容

**关键配置**：

```ini
# MariaDB GTID 配置
gtid_domain_id = 1  # 域 ID，多 master 场景必需
gtid_strict_mode = ON
```

**与 MySQL GTID 的区别**：
| 特性 | MySQL GTID | MariaDB GTID |
|---|---|---|
| 格式 | `server_uuid:transaction_id` | `domain_id:server_id:transaction_id` |
| 多源复制 | 需额外配置 | 域 ID 自然支持 |
| Galera 兼容 | 不兼容 | 完全兼容 |
