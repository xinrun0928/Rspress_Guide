# MariaDB 概述与 MySQL 分支发展史

2010 年，一个开源数据库社区的分叉改变了大半个数据库世界的格局。这一年，MySQL 的创始人 Michael "Monty" Widenius 在 Oracle 收购 MySQL 后，宣布创建 MariaDB——一个与 MySQL 完全兼容但走向不同的开源数据库分支。

十年后的今天，MariaDB 已经成为全球最流行的开源数据库之一，被 Wikimedia、Google、Red Hat 等巨头广泛采用。理解 MariaDB，不仅要会用，更要理解它背后的历史和哲学。

---

## MySQL 的前世今生

### 起源：一个芬兰项目的诞生

MySQL 的故事要从 1994 年说起。Michael "Monty" Widenius 和 David Axmark 在芬兰创建了 MySQL AB 公司，开发了一款名为 MySQL 的关系型数据库。

**为什么叫 MySQL？** 有两种说法：
- Monty 的女儿名字叫 My
- `my` 是瑞典语中常见的词缀

MySQL 的设计目标很明确：**简单、快速、可靠**。它没有追求功能的完备，而是专注于 Web 应用场景下最常见的需求：快速的读写性能。

### 快速发展：从开源新星到行业标准

2001 年，MySQL 3.23 版本发布，开始支持 InnoDB 存储引擎，事务支持成为可能。

2004 年，MySQL 4.1 引入了子查询和 R-tree 索引，功能进一步完善。

2005 年，MySQL 5.0 发布，增加了存储过程、触发器、视图、事务支持（InnoDB 成为默认引擎）、分布式事务（MySQL 5.0 企业版）。

2008 年，MySQL 5.1 发布，增加了表分区、事件调度器、基于行的复制。

### Oracle 收购：社区的担忧

2008 年，Sun Microsystems 以 10 亿美元收购 MySQL AB。

2010 年，Oracle 以 74 亿美元收购 Sun Microsystems，从而获得了 MySQL 的控制权。

**社区的担忧并非空穴来风**：
- Oracle 同时拥有 MySQL 和 Oracle Database（商业数据库巨头）
- MySQL 社区版的发展速度明显放缓
- 部分核心开发者离职

## MariaDB 的诞生

### Monty 的决定

Oracle 收购 MySQL 后，Monty 决定创建一个 MySQL 的分支。这个分支有两个核心原则：

1. **永远保持开源**：代码永远不会闭源
2. **完全兼容 MySQL**：现有 MySQL 应用可以零成本迁移

Monty 联合瑞典软件公司 SkySQL（后与 MariaDB 合并），于 2010 年发布了 MariaDB 5.1——MySQL 5.1 的二进制替代品。

**命名 MariaDB 的由来**：
- Monty 的小女儿名叫 Maria
- "Maria" 是"My"的下一个版本（A -> My -> Maria -> ...）
- 这是 Monty 一贯的命名风格——用他女儿的名字命名项目

### MariaDB Foundation：独立的中立机构

2012 年，MariaDB Foundation 成立，成为 MariaDB 项目的独立管理机构。

**Foundation 的职责**：
- 持有 MariaDB 商标
- 确保代码开源
- 管理社区贡献
- 与各发行版集成（Debian、Ubuntu、Red Hat 等）

**独立性保证**：Oracle 无法收购或控制 MariaDB Foundation，保证了 MariaDB 的长期独立发展。

## MariaDB 与 MySQL 的兼容性

### 为什么兼容性如此重要

MariaDB 的核心设计哲学是"兼容第一"。对于企业来说，迁移成本是选择数据库的关键考量。如果迁移需要修改代码、重新测试、停机切换，那迁移成本将高得难以承受。

MariaDB 做到了：

- **协议兼容**：MySQL 客户端可以直接连接 MariaDB
- **数据文件兼容**：MySQL 的数据目录可以转换为 MariaDB
- **SQL 语法兼容**：大部分 MySQL 语法在 MariaDB 中可直接使用
- **工具兼容**：mysqldump、mysql client 等工具通用

### 兼容性层级

| 层级 | 描述 | 兼容性程度 |
|---|---|---|
| 客户端协议 | MySQL 客户端可以连接 MariaDB | 完全兼容 |
| SQL 语法 | CREATE TABLE、SELECT 等语句 | 大部分兼容 |
| 存储引擎 | InnoDB/XtraDB、MyISAM 等 | 兼容 |
| 复制协议 | MySQL 主从可以复制到 MariaDB | 单向兼容 |
| 系统变量 | `show variables` 大部分相同 | 大部分兼容 |

### 迁移的实际考量

尽管兼容性很高，迁移仍需注意：

1. **存储引擎差异**：MySQL 的 InnoDB 和 MariaDB 的 XtraDB 虽有差异，但数据文件可互换
2. **函数差异**：部分 MySQL 特有函数在 MariaDB 中可能行为不同
3. **版本对应**：建议参考官方兼容性矩阵，选择合适的 MariaDB 版本
4. **测试验证**：迁移前务必在测试环境验证所有功能

## MariaDB 的核心演进

### 版本号与 MySQL 的对应关系

MariaDB 的版本号策略很有趣：它与 MySQL 的版本号有对应关系，但又独立发展。

| MariaDB 版本 | MySQL 对应版本 | 发布年份 |
|---|---|---|
| 5.1 | 5.1 | 2010 |
| 5.2 | 5.1 + 特性 | 2010 |
| 5.3 | 5.1 + 特性 | 2011 |
| 10.0 | 5.6 | 2012 |
| 10.1 | 5.6 + 特性 | 2014 |
| 10.2 | 5.7 | 2017 |
| 10.3 | 5.7 + 特性 | 2017 |
| 10.4 | 8.0 | 2019 |
| 10.5 | 8.0 | 2020 |
| 10.6 | 8.0 | 2021 |
| 10.7 | 8.0 | 2022 |
| 10.8 | 8.0 | 2022 |
| 10.9 | 8.0 | 2022 |
| 10.10 | 8.0 | 2023 |
| 10.11 | 8.0 LTS | 2023 |
| 11.0 | 8.0 + 新特性 | 2023 |
| 11.1+ | 8.0 + 新特性 | 2024 |

**版本策略**：
- MariaDB 10.x 系列是长期支持版（LTS）
- MariaDB 11.x 是最新系列，兼容 MySQL 8.0

### 里程碑版本回顾

**MariaDB 5.5（2012）**：第一个广泛使用的版本，继承 MySQL 5.5 的稳定性

**MariaDB 10.0（2012）**：跨版本跳跃，表明 MariaDB 不再跟随 MySQL 版本号

**MariaDB 10.2（2017）**：引入窗口函数、CTE（公共表表达式），功能大幅增强

**MariaDB 10.3（2017）**：引入系统版本表、原子 DDL、SQUENCE 存储引擎

**MariaDB 10.5（2020）**：引入 AliSQL 的查询优化、Instant ADD COLUMN

**MariaDB 10.6（2021）**：LTS 版本，性能和稳定性大幅提升

**MariaDB 11.0（2023）**：支持窗口函数增强、JSON TABLE 等

## MariaDB 的使用现状

### 谁在用 MariaDB

**互联网巨头**：
- **Wikipedia/Wikimedia**：全球最大的百科全书网站，使用 MariaDB 替代 MySQL
- **Google**：在部分内部服务中使用 MariaDB
- **Red Hat**：Fedora、CentOS Stream 默认数据库
- **腾讯云/阿里云**：提供 MariaDB 云数据库服务

**发行版支持**：
- Debian 和 Ubuntu 将 MariaDB 作为默认 MySQL 包
- Red Hat Enterprise Linux 7+ 提供 MariaDB
- SUSE Linux Enterprise Server 提供 MariaDB

### MariaDB 与 MySQL 的市场定位

**MySQL（Oracle）**：
- 定位企业级市场
- 强调 Oracle 数据库的互补性
- MySQL Enterprise Edition 提供商业插件

**MariaDB**：
- 定位开源社区和中小企业
- 强调与 PostgreSQL 的竞争
- MariaDB Enterprise 提供企业级支持

## 面试追问方向

### 为什么要问 MariaDB 的历史？

面试官问这个问题，往往想考察：

1. **技术视野**：你是否了解数据库领域的开源生态
2. **技术选型能力**：你是否理解兼容性的价值
3. **风险意识**：你是否考虑过供应商锁定的问题

### 可能的追问

- **Oracle 为什么要收购 MySQL？对开源社区有什么影响？**
  - Oracle 需要补齐开源数据库短板
  - MySQL 社区分裂促成了 MariaDB 的崛起
  - 推动了 PostgreSQL 的发展

- **既然 MariaDB 兼容 MySQL，为什么不直接用 MySQL？**
  - 开源保证：MariaDB Foundation 保证了代码永远开源
  - 创新速度：MariaDB 在某些功能上领先 MySQL
  - 社区治理：没有单一公司的商业利益驱动

---

## 总结

MariaDB 的诞生是开源世界对商业化风险的一次集体回应。Monty 用他女儿的名字命名了这个项目，也用他的坚持保证了 MariaDB 的开源基因。

理解 MariaDB，不能只看技术，还要理解它的价值观：**兼容性第一、永远开源、社区驱动**。

这些价值观体现在 MariaDB 的每一个设计决策中——为什么选择 XtraDB 而非自研引擎，为什么强调 MySQL 兼容性，为什么成立独立的 Foundation。
