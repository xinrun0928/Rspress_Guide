# Galera Cluster 原理

如果你问 DBA 什么是 Galera Cluster，大多数人会回答「多主集群」。这个答案没错，但不完整。Galera Cluster 的精妙在于它解决了分布式数据库最核心的问题：**如何在多个节点间保证数据一致性的同时，不牺牲可用性？**

这个问题困扰了分布式系统领域几十年。Galera 交出了一份优雅的答卷。

---

## 分布式一致性的困境

在说 Galera 之前，必须理解它要解决的问题。

### CAP 定理的约束

Eric Brewer 提出的 CAP 定理说：分布式系统最多同时满足一致性（Consistency）、可用性（Availability）、分区容错（Partition tolerance）中的两个。

```
        C
       /\
      /  \
     /    \
    A------P
```

- **一致性（C）**：所有节点看到相同的数据
- **可用性（A）**：每个请求都能收到响应
- **分区容错（P）**：网络分区时系统仍能运行

现实网络中，分区不可避免。CAP 定理告诉我们：要么选择 CP（一致性优先），要么选择 AP（可用性优先）。

### 主从复制的「伪一致性」

MySQL/MariaDB 的传统主从复制是**异步**的：

1. 主库执行事务，写入 binlog
2. 主库返回客户端「执行成功」
3. 从库异步拉取 binlog，回放

问题在于：步骤 2 和 3 之间有延迟。如果主库在步骤 2 之后崩溃，从库可能还没有这条数据。如果此时发生主备切换，新主库会丢失这条数据。

这是**最终一致性**，不是强一致性。对于电商库存、金融交易，这种不一致是灾难性的。

### Galera 的设计目标

Galera Cluster 追求的是：**同步复制 + 强一致性 + 高可用**。

实现方式是**乐观并发控制**——先执行，提交时再检查是否有冲突。如果有冲突，回滚。这与 MVCC 的思想一脉相承。

## Galera 的核心机制

### 1. 复制的四个阶段

Galera 的复制不是简单地将事务发到其他节点。它有一个精妙的多阶段流程：

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│   阶段一    │    │   阶段二    │    │   阶段三    │    │   阶段四    │
│  Local IO   │ -> │ Certification│ -> │ Replication │ -> │  Applier   │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
```

**阶段一：Local IO（本地执行）**

客户端发送事务到某个节点。该节点执行事务，写入 InnoDB 的 redo log、undo log。这个阶段与普通单机 MariaDB 无异。

此时，事务**并未提交**，而是在一个「预提交」状态。

**阶段二：Certification（认证）**

节点生成全局唯一的事务 ID（GTID），将事务内容（write set）广播给集群所有节点。

每个节点收到 write set 后，执行 **certification check**：

- 检查本节点上是否有与该 write set 冲突的已提交事务
- 如果有冲突，标记该事务为「冲突」，发起节点将回滚事务
- 如果无冲突，标记为「可提交」

这个过程像是一场「投票」。只有多数节点都同意，这个事务才能提交。

**阶段三：Replication（复制）**

通过 Galera 的总线协议（Galera Replication Protocol），write set 被复制到所有节点。所有节点按相同顺序写入本地 binlog。

注意：这一步只是「复制」，还没有「应用」。

**阶段四：Applier（回放）**

各节点独立回放 write set，提交事务。此时，所有节点的数据达成一致。

### 2. 乐观并发控制：冲突检测的核心

Galera 的冲突检测基于 **key-based conflict detection**。

每个 write set 记录了本次事务修改的所有行：

```c
struct write_set {
    uint64_t gtid;                    // 全局事务 ID
    uint64_t local_seqno;            // 节点本地序号
    std::vector<galera::KeyData> keys;  // 修改的所有 key
    std::vector<galera::WriteData> data; // 修改的所有 value
}
```

当节点 A 的事务 T1 和节点 B 的事务 T2 同时修改同一行 key 时：

- 节点 A 先广播 T1 的 write set
- 节点 B 收到后，发现 T2 也修改了这个 key
- B 将 T1 标记为「潜在冲突」
- T2 的 certification 通过后，A 发现 T1 与 T2 冲突，回滚 T1

这就是 **多主写入但不冲突** 的原理：并发写入时，只有第一个完成 certification 的事务能提交，其他冲突事务被回滚。

### 3. 全局事务 ID（GTID）

GTID 是 Galera 实现强一致性的关键。每个事务有一个全局唯一标识：

```
UUID:sequence_number
例：3e65790e-6b7a-11ec-9c31-0242ac110005:1234
```

- `UUID`：集群中每个节点唯一的标识
- `sequence_number`：该节点本地递增的序号

GTID 有两个重要作用：

**保证复制顺序一致**：所有节点按 GTID 顺序应用 write set，确保没有乱序。

**支持增量状态转移**：新节点加入时，只需同步 GTID 之后的增量数据，不需要全量同步。

### 4. 多数派投票（Quorum）

Galera 的高可用依赖于「多数派」机制。

当网络分区发生时，每个分区独立运行。但只有获得「多数票」的分区才能继续接受写入：

```
节点数：3
Quorum = floor(3/2) + 1 = 2

分区场景：
- 分区 A：节点 1、2（2票） → 继续工作
- 分区 B：节点 3（1票）   → 停止写入，只读
```

如果某个分区没有多数派，它会进入「非主」状态，拒绝写入，防止脑裂（Split Brain）。

Quorum 计算公式：

```
Quorum = floor(n/2) + 1  (n 为节点总数)
```

| 节点数 | Quorum | 容忍故障数 |
|---|---|---|
| 1 | 1 | 0 |
| 2 | 2 | 0（没有容错能力，不推荐） |
| 3 | 2 | 1 |
| 5 | 3 | 2 |
| 7 | 4 | 3 |

这就是为什么 Galera 推荐**奇数节点**部署。偶数节点不会提升容错能力，反而增加复杂度。

## SST 与 IST：节点同步的艺术

当新节点加入集群，或者节点需要同步数据时，Galera 提供了两种同步方式：

### State Snapshot Transfer（SST）：全量同步

SST 将整个数据库状态从一个节点复制到新节点。有三种实现：

| 方法 | 说明 | 备份锁 |
|---|---|---|
| mysqldump | 最通用，跨版本兼容 | 支持（低版本不支持） |
| rsync | 最快，文件级同步 | 不支持 |
| xtrabackup | 推荐方式，Percona 工具 | 支持 |

生产环境推荐 **xtrabackup SST**，因为它：
- 支持在线同步，不阻塞业务
- 支持增量备份
- 有成熟的运维生态

```bash
# xtrabackup SST 配置示例
wsrep_sst_method=xtrabackup-v2
wsrep_sst_auth=root:password
```

### Incremental State Transfer（IST）：增量同步

当节点短暂离群（如重启）又重新加入时，如果离群时间短，数据变更不多，IST 比 SST 更高效。

IST 的工作原理：

1. 新节点报告自己最后一个已同步的 GTID
2. 供体节点检查自己的 write set cache（gcache）
3. 如果需要的 write set 还在 cache 中，复制增量即可
4. 如果 cache 已清理，则退化为 SST

```
gcache.size = 128MB  # 默认缓存大小
# 缓存越大，IST 成功率越高
```

**生产建议**：合理设置 `gcache.size`，让短暂离群的节点能用 IST 快速恢复。

## Galera 的性能特性

### 写入性能的代价

Galera 的同步机制带来了额外的延迟：

```
单机事务延迟 ≈ 事务执行时间
Galera 事务延迟 ≈ 事务执行时间 + 广播时间 + 最慢节点的 certification + 网络 RTT
```

典型的单机房 Galera 集群，单事务延迟增加约 2-5ms。对于高并发短事务，这个开销可能影响吞吐量。

### 写集的优化

Galera 使用 **write set replication** 来优化性能：

- 事务广播的不是 SQL，而是 key-value 对
- 冲突检测基于 key，不需要重放 SQL
- 减少了网络传输量

```sql
-- 查看 write set 统计
SHOW STATUS LIKE 'wsrep%';
```

关键指标：

| 指标 | 含义 |
|---|---|
| `wsrep_local_send_queue_avg` | 平均发送队列长度，>0 说明网络有瓶颈 |
| `wsrep_cert_deps_distance` | 认证平均依赖距离，越大说明并发越好 |
| `wsrep_flow_control_paused` | 流控暂停时间比例，>0 说明节点跟不上 |

### 流控（Flow Control）

如果某个节点性能较差，处理速度跟不上，会拖慢整个集群。Galera 实现了流控机制：

```
节点处理速度 < 写入速度
       ↓
节点申请「暂停」写入
       ↓
其他节点感知到暂停，减缓写入
       ↓
当节点追上后，恢复写入
```

流控保证了集群不会因为单节点瓶颈而崩溃，但也会导致写入吞吐量下降。

**调优建议**：

```ini
# wsrep.cnf
wsrep_slave_threads = 16      # 从 applier 线程数，默认等于 CPU 核心数
wsrep_cert_log_size = 0       # 关闭 cert log，减少内存压力
gcache.size = 1GB              # 适当增加 gcache，提升 IST 成功率
```

## Galera 的局限性

Galera 不是万能的，以下场景需要特别注意：

### 1. 大事务问题

Galera 的 write set 需要全部节点同时确认。超过 50MB 的大事务可能导致：

- 节点间网络拥塞
- certification 时间过长
- 其他小事务被阻塞

**解决方案**：拆分为小事务，或使用 `wsrep_max_ws_rows` 和 `wsrep_max_ws_size` 限制事务大小。

### 2. 写入放大

由于所有节点都要执行相同的事务，如果应用层有大量短连接、短事务，Galera 的广播开销会放大。

**解决方案**：使用连接池，减少连接建立销毁；合并小事务。

### 3. 节点地理分布

Galera 的强一致性依赖低延迟网络。跨机房、跨地域部署时，网络延迟可能导致性能急剧下降。

**解决方案**：同机房部署 Galera，跨机房用异步复制。

### 4. AUTO_INCREMENT 的局限

Galera 环境下，每个节点独立生成自增 ID。但由于 certification 机制，最终只有部分 ID 被使用，不会冲突，但会有「空洞」。

**解决方案**：使用 `auto_increment_offset` 为不同节点设置不同起始值，或使用 Sequence。

---

## 总结

Galera Cluster 的核心是**基于乐观并发控制的同步多主复制**：

1. **本地执行**：事务先在本地执行，不影响性能
2. **广播 write set**：生成 write set，广播到所有节点
3. **Certification 检查**：每个节点检查冲突，少数服从多数
4. **同步回放**：无冲突的事务在所有节点同步提交

这种设计实现了「强一致性的同步复制」与「多主可写入」的统一。但代价是：写入性能受限于最慢节点，不适合跨地域部署，大事务需要拆分。

<Badge text="原理总结" type="tip" /> 理解 Galera 的关键是理解四个字：**乐观并发**。它假设大多数情况下没有冲突，先干再说，冲突了再回滚。这比悲观锁（先锁住再执行）的性能好很多，但需要应用层容忍事务回滚。
