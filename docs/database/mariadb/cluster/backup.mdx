# 备份恢复

数据库备份是「最后的安全网」。无论你的集群多可靠、系统多稳定，备份永远是不可替代的防线。本章讲解 MariaDB 各种备份方案，以及 Galera 集群环境下的特殊注意事项。

## 备份的基本分类

备份分为两大类：**物理备份**和**逻辑备份**。

| 类型 | 原理 | 优点 | 缺点 |
|---|---|---|---|
| 物理备份 | 复制数据文件 | 速度快，适合大数据库 | 只能恢复到备份时刻 |
| 逻辑备份 | 导出 SQL 语句 | 可跨版本，恢复灵活 | 速度慢，占用空间大 |

MariaDB 两种都支持，根据场景选择。

## 逻辑备份：mysqldump

### 基本用法

```bash
# 备份单个数据库
mysqldump -u root -p dbname > dbname.sql

# 备份所有数据库
mysqldump -u root -p --all-databases > all.sql

# 备份指定表
mysqldump -u root -p dbname users orders > tables.sql
```

### 生产级备份脚本

```bash title="backup_mysqldump.sh"
#!/bin/bash
set -e

DATE=$(date +%Y%m%d_%H%M%S)
BACKUP_DIR="/backup/mysql"
DB_HOST="localhost"
DB_USER="root"
DB_PASS="password"
DB_NAME="app_db"

# 创建备份目录
mkdir -p ${BACKUP_DIR}

# 备份并压缩
mysqldump -h${DB_HOST} -u${DB_USER} -p${DB_PASS} \
    --single-transaction \
    --routines \
    --triggers \
    --events \
    ${DB_NAME} | gzip > ${BACKUP_DIR}/${DB_NAME}_${DATE}.sql.gz

# 删除 7 天前的备份
find ${BACKUP_DIR} -name "*.sql.gz" -mtime +7 -delete

# 记录备份日志
echo "[$(date)] Backup completed: ${DB_NAME}_${DATE}.sql.gz" >> /var/log/backup.log
```

**参数说明**：

| 参数 | 作用 |
|---|---|
| `--single-transaction` | 快照备份，备份期间不锁表（仅 InnoDB） |
| `--routines` | 包含存储过程和函数 |
| `--triggers` | 包含触发器 |
| `--events` | 包含事件调度器 |

### Galera 集群备份

在 Galera 集群中，**只需要备份一个节点**，不需要同时备份所有节点：

```bash
# 在任意一个节点上执行
mysqldump -u root -p \
    --single-transaction \
    --gtid \
    --master-data=2 \
    app_db > backup.sql
```

`--gtid` 选项确保恢复时能定位到正确的 GTID。

## 物理备份：xtrabackup

### 为什么选择 xtrabackup

mysqldump 是逻辑备份，适合小数据库。对于 GB/TB 级别的大数据库，xtrabackup 是更好的选择：

- **速度快**：复制 InnoDB 数据文件，不需要重放 SQL
- **在线备份**：不影响业务
- **增量备份**：支持只备份变化的数据
- **Galera 兼容**：完美支持 Galera SST

xtrabackup 是 Percona 开发的工具，MariaDB 官方推荐使用。

### 安装 xtrabackup

```bash
# CentOS/RHEL
yum install percona-xtrabackup

# Debian/Ubuntu
apt install percona-xtrabackup
```

### 全量备份

```bash
# 创建备份目录
mkdir -p /backup/xtrabackup/full

# 执行备份
xtrabackup --backup \
    --user=root \
    --password=password \
    --target-dir=/backup/xtrabackup/full/$(date +%Y%m%d)
```

### 增量备份

```bash
# 基于上一次全量备份，创建增量备份
xtrabackup --backup \
    --user=root \
    --password=password \
    --target-dir=/backup/xtrabackup/incr/$(date +%Y%m%d) \
    --incremental-basedir=/backup/xtrabackup/full/20240101
```

增量备份的原理是记录 Page Change Log，只备份变化的文件。

### 备份策略

```
周全量 + 日增量
├── 周一：全量备份 (100GB)
├── 周二：增量 1 (5GB)
├── 周三：增量 2 (6GB)
├── 周四：增量 3 (4GB)
├── 周五：增量 4 (8GB)
├── 周六：增量 5 (3GB)
└── 周日：增量 6 (5GB)
```

这样，备份窗口只需每周一次全量，其他时间用增量，极大减少备份对业务的影响。

## Galera SST 备份配置

Galera 集群的新节点加入时，通过 SST 从供体节点拉取全量数据。xtrabackup 是 SST 的推荐方式。

### 配置 xtrabackup SST

```ini title="/etc/mysql/mariadb.conf.d/galera.cnf"
[mysqld]
wsrep_sst_method = xtrabackup-v2
wsrep_sst_auth = "root:password"
sst_extra_args = "--compress"
```

### 备份锁的问题

xtrabackup SST 在备份时会短暂锁表（FLUSH TABLES WITH READ LOCK）。对于大数据库，这可能影响业务。

MariaDB 10.5+ 支持**无锁备份**：

```sql
-- 执行备份前，先执行
FLUSH NO_WRITE_TO_BINLOG TABLES;

-- 然后 xtrabackup 执行 SST
```

## 恢复操作

### 逻辑备份恢复

```bash
# 恢复单个数据库
gunzip < dbname_20240101.sql.gz | mysql -u root -p dbname

# 或
mysql -u root -p dbname < dbname_20240101.sql
```

### 物理备份恢复

物理备份恢复分为两步：**准备（prepare）**和**恢复（restore）**。

**1. 准备备份**

```bash
# 对备份进行崩溃恢复准备
xtrabackup --prepare --target-dir=/backup/xtrabackup/full/20240101

# 增量备份的恢复准备
xtrabackup --prepare --target-dir=/backup/xtrabackup/full/20240101 \
    --incremental-dir=/backup/xtrabackup/incr/20240102
```

**2. 恢复数据**

```bash
# 停止 MariaDB
systemctl stop mariadb

# 清空数据目录（谨慎！）
rm -rf /var/lib/mysql/*

# 恢复数据
xtrabackup --copy-back --target-dir=/backup/xtrabackup/full/20240101

# 设置权限
chown -R mysql:mysql /var/lib/mysql

# 启动 MariaDB
systemctl start mariadb
```

### Galera 集群恢复

Galera 集群的恢复需要特殊处理：

**场景一：单节点数据损坏**

如果只有一台节点损坏，最简单的方式是删除该节点，让它重新 SST 加入：

```bash
# 修复的节点上
systemctl stop mariadb
rm -rf /var/lib/mysql/*

# 重新加入集群
systemctl start mariadb
```

**场景二：整个集群需要恢复**

```bash
# 1. 停止所有节点
systemctl stop mariadb

# 2. 在第一个节点恢复数据
xtrabackup --copy-back --target-dir=/backup/xtrabackup/full/20240101

# 3. 确保数据目录为空
rm -rf /var/lib/mysql/*

# 4. 第一个节点重新初始化
mysqld --wsrep_new_cluster &

# 5. 逐个启动其他节点
systemctl start mariadb
```

## 备份验证

备份不验证，等于没有备份。恢复测试必须定期执行：

```bash title="test_restore.sh"
#!/bin/bash
# 恢复测试脚本（恢复到一个临时目录）

TEST_DIR="/backup/test_restore"
BACKUP_FILE=$1

# 准备测试环境
mkdir -p ${TEST_DIR}
docker run -d --name mysql_restore_test -e MYSQL_ROOT_PASSWORD=test \
    -v ${TEST_DIR}:/var/lib/mysql mariadb:latest

# 解压并恢复
gunzip < ${BACKUP_FILE} | mysql -h localhost -u root -ptest

# 验证数据
mysql -h localhost -u root -ptest -e "SELECT COUNT(*) FROM app_db.users;"

# 清理
docker stop mysql_restore_test
docker rm mysql_restore_test
rm -rf ${TEST_DIR}
```

<Badge text="黄金法则" type="warning" /> **备份三原则**：

1. **可恢复**：定期测试恢复流程
2. **有副本**：备份文件异地存储
3. **不过期**：根据数据重要性设定保留周期

## 备份存储策略

### 本地备份 + 远程备份

```bash
# 本地备份
xtrabackup --backup --target-dir=/backup/local/$(date +%Y%m%d)

# 同步到远程
rsync -avz /backup/local/ root@remote-server:/backup/mysql/

# 删除本地 7 天前的备份
find /backup/local -type d -mtime +7 -exec rm -rf {} \;
```

### 云对象存储

```bash
# 备份到 S3 兼容存储
apt install s3cmd
s3cmd configure
s3cmd put /backup/local/$(date +%Y%m%d).tar.gz s3://my-backup-bucket/mysql/
```

### 备份加密

备份文件包含完整数据，必须加密：

```bash
# 使用 GPG 加密
gpg --encrypt --recipient backup@example.com \
    /backup/local/backup.sql.gz

# 恢复时解密
gpg --decrypt backup.sql.gz.gpg | mysql -u root -p
```

## 总结

MariaDB 的备份恢复体系完整且成熟：

- **mysqldump**：逻辑备份，适合小数据库，灵活性高
- **xtrabackup**：物理备份，适合大数据库，性能好
- **Galera 备份**：只需备份一个节点，SST 自动同步
- **恢复验证**：备份必须定期测试，否则不可信
