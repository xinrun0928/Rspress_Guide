# 节点加入与故障恢复

Galera Cluster 的强大不仅在于多主写入，更在于**故障自愈**。当节点崩溃、节点离群、新节点加入时，Galera 能自动处理，无需人工干预。本章将深入讲解这些场景的原理和操作。

## 节点的生命周期

一个 Galera 节点的完整生命周期：

```
初始化 -> 加入集群 -> 工作状态 -> 离群（可选） -> 重新加入（可选） -> 退出
```

理解每个阶段的转换条件，是运维 Galera 的基础。

### 集群状态机

Galera 节点有四种状态：

| 状态 | 说明 |
|---|---|
| `OPEN` | 节点刚启动，还没连接集群 |
| `PRIMARY` | 已连接集群，是合法成员 |
| `JOINING` | 正在加入集群（SST/IST 同步中） |
| `SYNCED` | 完全同步，正常工作 |

```
OPEN -> PRIM (连接成功)
PRIM -> JOIN (发起加入)
JOIN -> SYNC (同步完成)
SYNC -> OPEN (重启)
SYNC -> JOIN (重新加入)
```

状态转换由集群协议控制，应用层不应该手动干预。

## 新节点加入

新节点加入有两种场景：**新集群初始化**和**现有集群扩容**。

### 场景一：新集群初始化

第一个节点的启动稍有不同：

```bash
# 初始化第一个节点
mysqld --wsrep_new_cluster &
```

`--wsrep_new_cluster` 告诉 MariaDB，这是一个新集群的种子节点。

```sql
-- 查看集群成员
SHOW STATUS LIKE 'wsrep_cluster_size';
-- 输出：1

SHOW STATUS LIKE 'wsrep_cluster_status';
-- 输出：PRIMARY
```

### 场景二：扩容加入

后续节点启动时，会自动发现集群：

```ini
# /etc/mysql/mariadb.conf.d/galera.cnf
[mysqld]
wsrep_cluster_address = 'gcomm://192.168.1.10,192.168.1.11,192.168.1.12'
wsrep_node_address = '192.168.1.13'
wsrep_node_name = 'node4'
```

节点启动后，流程如下：

```
1. 读取 cluster_address，找到现有节点
2. 向现有节点发起 JOIN 请求
3. 现有节点投票，同意新节点加入
4. 触发 SST 或 IST，开始数据同步
5. 同步完成后，进入 SYNCED 状态
```

### SST vs IST：如何选择

节点加入时选择 SST 还是 IST，取决于以下因素：

| 因素 | SST | IST |
|---|---|---|
| 数据差异 | 差异大或完全空白 | 差异小（短暂离群） |
| 离群时间 | 长时间（cache 已清空） | 短时间（cache 有效） |
| 同步速度 | 慢（全量复制） | 快（增量复制） |
| 对业务影响 | 较大（备份锁） | 较小（无锁） |

**判断逻辑**：

```sql
-- 查看节点状态
SHOW STATUS LIKE 'wsrep_local_state_comment';
-- 输出：Synced 或 Joining

-- 查看供体节点信息
SHOW STATUS LIKE 'wsrep_incoming_addresses';
```

## 节点故障与恢复

### 故障类型

Galera 环境中，节点故障分为两类：

**1. 优雅关闭**

```bash
systemctl stop mariadb
mysqladmin shutdown
```

优雅关闭时，节点会通知集群「我要离群」，集群立即将其从成员列表中移除。其他节点不受影响。

**2. 非优雅关闭（崩溃）**

节点突然断电、OOM kill、进程崩溃。集群需要通过心跳检测来发现这个故障：

```ini
# 故障检测配置
evs.suspect_timeout = PT5S      # 5秒内无心跳，标记为可疑
evs.inactive_timeout = PT15S    # 15秒内无心跳，标记为离群
evs.consensus_timeout = PT3S    # 投票超时
```

### Quorum 与脑裂防护

当节点故障导致分区时，Quorum 机制防止脑裂：

```
正常状态：3 节点，Quorum = 2

故障场景：节点 3 崩溃
分区 A：节点 1、2（2票 >= Quorum）-> 继续工作
分区 B：节点 3（1票 < Quorum）-> 停止写入
```

节点 3 被标记为 `non-Primary`，拒绝所有写入：

```sql
SHOW STATUS LIKE 'wsrep_cluster_status';
-- 输出：non-Primary

-- 尝试写入会报错
ERROR 1047 (08S01): WSREP has not yet prepared node for application use
```

### 故障节点重新加入

故障节点修复后，重新加入集群：

```bash
systemctl start mariadb
```

流程：

```
1. 节点启动，进入 OPEN 状态
2. 连接集群，发现自己不在成员列表
3. 请求重新加入
4. 如果是 IST 适用（gcache 有效），执行 IST
5. 如果 IST 不可用（差距太大），执行 SST
6. 同步完成后，进入 SYNCED
```

**IST 的关键**：节点的 `gcache` 保存了最近的 write set。如果差距在 gcache 范围内，可以增量同步；否则需要全量 SST。

## 节点移除与缩容

### 手动移除节点

如果需要永久移除某个节点：

```sql
-- 在目标节点上执行
SET GLOBAL wsrep_desync = ON;
SET GLOBAL wsrep_ready = OFF;
```

```bash
# 然后关闭 MariaDB
systemctl stop mariadb
```

节点离群后，集群剩余节点的 Quorum 自动重新计算。

### 缩容注意事项

**警告**：不要将集群缩容到只剩 1 个节点——单节点无法构成高可用。

| 原节点数 | 缩容后 | 可行性 |
|---|---|---|
| 3 | 2 | 可行，但无容错能力 |
| 3 | 1 | 不推荐 |
| 5 | 3 | 推荐 |

缩容操作应该在低峰期进行，并确保集群状态健康。

## 常见故障与处理

### 故障一：SST 失败

SST 是最容易出问题的环节。常见原因：

**原因一：认证失败**

```log
WSREP_SST: [ERROR] xtrabackup failed
WSREP_SST: [ERROR] Failed to SST for some reason
```

检查供体节点的认证配置：

```ini
# 确保认证信息一致
wsrep_sst_auth = root:password
```

**原因二：网络超时**

```log
WSREP: Could not open channel for 'node4'
WSREP: Handshake with remote end failed
```

检查：
- 防火墙是否放行 4567 端口
- 节点间网络是否可达
- `evs.connect_timeout` 是否合理

**原因三：磁盘空间不足**

SST 需要临时存储复制数据。确保供体节点有足够磁盘空间。

### 故障二：节点长期处于 JOINING

如果节点卡在 JOINING 状态很长时间，可能是：

1. **数据量太大**：TB 级数据的 SST 需要很长时间，耐心等待
2. **网络瓶颈**：检查 `wsrep_local_send_queue_avg`
3. **供体节点性能差**：考虑换一个供体

```sql
-- 监控同步进度
SHOW STATUS LIKE 'wsrep%';
```

### 故障三：Quorum 丢失

当超过半数的节点同时故障时，会发生 Quorum 丢失：

```sql
SHOW STATUS LIKE 'wsrep_cluster_status';
-- 输出：non-Primary

SHOW STATUS LIKE 'wsrep_cluster_size';
-- 可能只有 1 或 0
```

**恢复步骤**：

```bash
# 方法一：强制启动为单节点集群（谨慎使用，可能丢数据）
mysqld --wsrep_ignore_apply_errors=6 --wsrep_cluster_address='dummy://' &
```

```bash
# 方法二：手动指定集群成员重建
mysqld --wsrep_cluster_address='gcomm://192.168.1.10,192.168.1.11' &
```

**警告**：强制单节点启动可能导致数据不一致。务必在操作前评估数据损失风险。

## 运维最佳实践

### 1. 监控关键指标

```sql
-- 集群健康状态
SHOW STATUS LIKE 'wsrep_cluster_status';  -- 必须是 Primary
SHOW STATUS LIKE 'wsrep_cluster_size';     -- 节点数
SHOW STATUS LIKE 'wsrep_connected';         -- 是否连接

-- 同步状态
SHOW STATUS LIKE 'wsrep_local_state_comment';  -- Synced/Joining
SHOW STATUS LIKE 'wsrep_local_recv_queue';      -- 接收队列
SHOW STATUS LIKE 'wsrep_local_send_queue';      -- 发送队列

-- 性能
SHOW STATUS LIKE 'wsrep_flow_control_paused';  -- 流控比例
SHOW STATUS LIKE 'wsrep_cert_deps_distance';    -- 并发能力
```

### 2. 合理配置 gcache

```ini
wsrep_provider_options = 'gcache.size=2G'
```

gcache 越大，IST 成功率越高。但也不要设得太大，浪费内存。

### 3. 使用负载均衡

不要让应用直连单个 Galera 节点。使用 HAProxy 或 MaxScale：

```
应用 -> HAProxy -> Galera 节点 1
                    -> Galera 节点 2
                    -> Galera 节点 3
```

这样任何节点故障，HAProxy 自动剔除，应用无感知。

---

## 总结

Galera 的节点管理是自动化的，但运维仍需理解其原理：

- **新节点加入**：通过 SST（全量）或 IST（增量）同步数据
- **故障检测**：心跳 + Quorum 投票，防止脑裂
- **故障恢复**：节点重启后自动追赶同步
- **缩容**：谨慎操作，确保剩余节点满足 Quorum

<Badge text="运维口诀" type="tip" /> 监控看 `cluster_status`（Primary）和 `local_state`（Synced）；缩容不缩到 1；SST 失败先查认证和端口。
