# 集群总览

当单机数据库无法支撑业务的流量压力，或者你需要更高的可用性保障时，集群化是必然选择。MariaDB 提供了完整的集群解决方案，从主从复制到 Galera 多主集群，覆盖了不同层级的需求。

## 为什么需要集群

单机数据库有三个天花板：

**容量天花板**。单机磁盘空间有限，当数据量达到 TB 级别时，单机存储成为瓶颈。虽然可以加磁盘，但单机的 IOPS 有物理上限。

**性能天花板**。所有读写请求都落在同一台机器上。CPU、内存、磁盘、网络，任意一个达到瓶颈都会拖垮整个系统。垂直扩展（升级硬件）有上限，水平扩展才是终局。

**可用性天花板**。单机故障意味着服务中断。即使你做了 RAID、UPS，仍然无法避免计划内维护导致的停机。

集群解决的是：能不能用的问题（高可用）、够不够快的问题（水平扩展）。

## MariaDB 的集群方案矩阵

MariaDB 提供多种集群方案，各有侧重：

| 方案 | 一致性 | 可用性 | 写入扩展 | 部署复杂度 |
|---|---|---|---|---|
| 主从复制 | 异步，最终一致 | 读故障转移 | 无 | 低 |
| 主主复制 | 异步，可能冲突 | 写故障转移 | 需应用协调 | 中 |
| Galera Cluster | 同步，强一致 | 多主写入 | 只读扩展 | 中高 |
| MaxScale + Galera | 同步 + 读写分离 | 自动故障转移 | 读扩展 | 高 |

没有银弹。每种方案都是 CAP 定理的权衡，选择取决于你的业务需求。

---

## 主从复制：最基础的集群形态

主从复制是 MariaDB 自带的功能，配置简单，运维成本低。

原理很简单：

1. 主库将写操作记录到 binlog
2. 从库连接主库，拉取 binlog 并回放
3. 从库数据与主库保持同步（异步）

**优点**：配置简单，对应用透明，支持跨机房容灾。

**缺点**：异步复制有延迟，主库故障时可能丢失数据；无法扩展写入能力。

主从复制适合「读多写少」的场景。通过读写分离，从库承担读请求，主库专注写入。

---

## Galera Cluster：真正的多主集群

Galera Cluster 是 MariaDB 的明星特性。它基于 Galera 库实现同步多主复制，多个节点可以同时接受写入，数据一致性由 Galera 协议保证。

**与主从复制的核心区别**：

- 主从复制是异步的，从库可能落后主库
- Galera 是同步的，任意节点写入成功，数据立即在所有节点可见

**优点**：
- 真正的强一致性
- 任意节点可写，无需应用层协调
- 节点故障自动切换，不影响服务

**缺点**：
- 写入性能受限于最慢节点
- 网络延迟敏感，跨地域部署需谨慎
- 存储空间必须一致（建议共享存储）

Galera 适合「对一致性要求高、写性能需求中等」的场景，如金融、电商核心交易系统。

---

## MaxScale：智能路由与高可用

MaxScale 是 MariaDB Corporation 开发的数据库代理，它位于应用和数据库之间，提供：

- **读写分离**：自动将 SELECT 路由到从库
- **连接池**：复用数据库连接，减少连接开销
- **自动故障转移**：主库故障时自动切换
- **查询路由**：按规则将不同查询路由到不同节点

MaxScale 本身是高可用的，支持主备部署。配合 Galera Cluster，构建完整的数据库高可用架构。

---

## 选择合适的方案

选择集群方案，核心是回答三个问题：

**问题一：你能接受多大的一致性延迟？**

如果业务要求强一致（如金融交易），选 Galera。如果允许最终一致（如日志分析），主从复制足够。

**问题二：写入压力有多大？**

如果写入是瓶颈，Galera 多主模式能分担负载（需应用层处理冲突）。如果读是瓶颈，MaxScale 读写分离更简单。

**问题三：运维能力如何？**

Galera + MaxScale 的组合功能强大，但运维复杂度也高。如果团队运维能力有限，从主从复制开始，逐步演进更稳妥。
