# 主从复制与 GTID

Galera Cluster 功能强大，但需要专用基础设施（Galera 库、多数派投票）。对于很多场景，传统的主从复制仍然是最实用的选择——配置简单、运维成熟、资源开销低。

本章讲解 MariaDB 的主从复制机制，以及 GTID 如何解决传统复制的痛点。

---

## 复制的原理

### 复制的三个线程

MariaDB 主从复制的核心是三个线程的协作：

```
主库                       从库
┌─────────┐              ┌─────────┐
│ Binlog  │─── dump ───>│ I/O    │  线程 1
│  Dump   │              │ Thread │──────┐
└─────────┘              └─────────┘      │
                                        │
                                   ┌─────┴─────┐
                                   │ Relay Log │
                                   └─────┬─────┘
                                         │
                                   ┌─────┴─────┐
                                   │ SQL      │  线程 2
                                   │ Thread   │───> 执行
                                   └─────────┘
```

**1. 主库的 Binlog Dump 线程**

从库连接主库时，主库启动 `Binlog Dump` 线程。该线程读取 binlog，将事件发送给从库的 I/O 线程。

**2. 从库的 I/O 线程**

从库接收到事件后，写入本地的 `relay log`。I/O 线程负责「拉取」和「存储」。

**3. 从库的 SQL 线程**

SQL 线程读取 relay log，执行其中的事件。SQL 线程是单线程的，这是传统复制的主要瓶颈。

### 复制的基本配置

**主库配置**：

```ini
[mysqld]
server-id = 1
log_bin = /var/log/mysql/mysql-bin
binlog_format = ROW
expire_logs_days = 7
```

**从库配置**：

```ini
[mysqld]
server-id = 2
relay_log = /var/log/mysql/mysql-relay-bin
read_only = ON
log_slave_updates = ON
```

**建立复制关系**：

```sql
-- 主库：创建复制账号
CREATE USER 'repl'@'%' IDENTIFIED BY 'password';
GRANT REPLICATION SLAVE ON *.* TO 'repl'@'%';

-- 从库：指定主库
CHANGE MASTER TO
    MASTER_HOST = '192.168.1.10',
    MASTER_USER = 'repl',
    MASTER_PASSWORD = 'password',
    MASTER_LOG_FILE = 'mysql-bin.000001',
    MASTER_LOG_POS = 107;

START SLAVE;
```

## GTID：复制位点的革命

### 传统复制的痛点

传统复制依赖 `FILE + POS`（文件名 + 位置）来指定从哪个点开始复制。这有几个问题：

**问题一：位点容易混乱**。主库切换时，需要知道新主库当前的 binlog 位置。如果记录错误，从库可能复制重复数据或漏复制数据。

**问题二：人工维护成本高**。每次主库重启、应用 crash，都要人工确认位点。

**问题三：无法判断数据一致性**。从库是否和主库一致？无法通过位点判断。

### GTID 的解决方案

GTID（Global Transaction Identifier）在 MariaDB 10.0+ 引入，每个事务都有一个全局唯一标识：

```
UUID:sequence_number
例如：3e65790e-6b7a-11ec-9c31-0242ac110005:1234
```

- `UUID`：主库的唯一标识
- `sequence_number`：该主库上的事务序号

有了 GTID，从库不再需要记录「文件名 + 位置」，只需要记录「我执行到第几个 GTID」。

### GTID 复制配置

```ini
[mysqld]
server-id = 2
gtid_domain_id = 2              # 复制域 ID（多主复制时区分来源）
log_slave_updates = ON          # GTID 必须开启此选项

# GTID 自动定位（关键！）
gtid_pos_autoengines = ''
gtid_strict_mode = ON           # 严格模式，防止 GTID 错误
```

**从库配置复制**：

```sql
-- 使用 GTID 自动定位
CHANGE MASTER TO
    MASTER_HOST = '192.168.1.10',
    MASTER_USER = 'repl',
    MASTER_PASSWORD = 'password',
    MASTER_USE_GTID = current_pos;

START SLAVE;
```

现在，从库会自动找到正确的复制起点，无需关心 binlog 文件和位置。

### GTID 的高级特性

**1. 自动跳过错误**

```sql
SET GLOBAL sql_slave_skip_counter = 1;
-- 或
SET GLOBAL gtid_slave_pos = 'uuid:N';  -- 跳过指定 GTID
```

**2. 多源复制**

MariaDB 10.0+ 支持多源复制，一个从库可以同时复制多个主库：

```sql
-- 连接到主库 1
CHANGE MASTER TO
    MASTER_HOST = '192.168.1.10',
    MASTER_USER = 'repl',
    MASTER_PASSWORD = 'password',
    MASTER_USE_GTID = current_pos,
    MASTER_LOG_FILE = ''
FOR CHANNEL 'source-1';

-- 连接到主库 2
CHANGE MASTER TO
    MASTER_HOST = '192.168.1.11',
    MASTER_USER = 'repl',
    MASTER_PASSWORD = 'password',
    MASTER_USE_GTID = current_pos,
    MASTER_LOG_FILE = ''
FOR CHANNEL 'source-2';

START SLAVE FOR CHANNEL 'source-1';
START SLAVE FOR CHANNEL 'source-2';
```

**3. GTID 切换**

当主库故障，需要切换到从库时：

```sql
-- 新主库（原来的从库）
STOP SLAVE;
RESET MASTER;  -- 变成主库

-- 其他从库
STOP SLAVE;
CHANGE MASTER TO
    MASTER_HOST = '新主库IP',
    MASTER_USER = 'repl',
    MASTER_PASSWORD = 'password',
    MASTER_USE_GTID = current_pos;
START SLAVE;
```

GTID 让切换变得简单，不需要手动查找 binlog 位置。

## 复制延迟与优化

### 延迟的原因

主从复制的 SQL 线程是单线程执行的。当主库并发写入很高时，从库可能严重滞后。

```sql
-- 查看延迟
SHOW SLAVE STATUS\G
Seconds_Behind_Master: 3600  -- 延迟 1 小时
```

延迟的原因：

| 原因 | 说明 |
|---|---|
| 大事务 | 一个事务修改 100 万行，从库需要较长时间 |
| 从库硬件差 | 从库 CPU、磁盘比主库差 |
| 从库负载高 | 从库同时服务读请求，影响复制 |
| 网络延迟 | 主从网络带宽不足 |

### 并行复制

MariaDB 10.0+ 引入了多线程并行复制：

```ini
# 配置并行复制
slave_parallel_threads = 8
slave_parallel_mode = optimistic  # optimistic 或 conservative
```

并行复制的模式：

| 模式 | 说明 | 适用场景 |
|---|---|---|
| `database` | 按数据库并行 | 多库写入 |
| `optimistic` | 乐观并行，冲突时回退单线程 | 写入冲突少 |
| `conservative` | 保守并行，保证不冲突 | 写入冲突多 |

### 优化建议

```sql
-- 从库关闭不必要的日志
log_bin = OFF
log_slave_updates = ON  -- 仍需开启，记录复制历史

-- 从库使用更好的磁盘
-- SSD 显著提升 relay log 回放速度

-- 调整 innodb_flush_log_at_trx_commit
SET GLOBAL innodb_flush_log_at_trx_commit = 2;
-- 牺牲一点持久性，换取写入性能
```

## 复制安全

### 半同步复制

异步复制的问题是：主库提交后，不保证从库已经接收。半同步复制（Semi-sync Replication）在主库提交时增加一个等待：

```sql
-- 主库安装插件
INSTALL PLUGIN rpl_semi_sync_master SONAME 'semisync_master.so';

-- 从库安装插件
INSTALL PLUGIN rpl_semi_sync_slave SONAME 'semisync_slave.so';

-- 启用
SET GLOBAL rpl_semi_sync_master_enabled = ON;
SET GLOBAL rpl_semi_sync_slave_enabled = ON;

-- 配置超时
SET GLOBAL rpl_semi_sync_master_timeout = 1000;  -- 1000ms
```

半同步复制的流程：

```
主库等待至少一个从库确认接收 -> 主库提交 -> 返回客户端
```

这确保了至少一个从库不会丢失数据。

### 复制过滤

可以在主库或从库端过滤复制的数据库/表：

```sql
-- 主库端（不推荐，会丢失 binlog）
binlog_do_db = app_db
binlog_ignore_db = test

-- 从库端（推荐）
replicate_do_db = app_db
replicate_ignore_db = test
replicate_do_table = app_db.orders
replicate_wild_do_table = app_db.order_items%
```

**警告**：从库端过滤可能导致 relay log 与主库 binlog 不一致。不要在从库上执行 `RESET SLAVE`，否则 relay log 丢失，复制会出错。

## 监控复制状态

```sql
-- 完整复制状态
SHOW SLAVE STATUS\G

-- 关键指标
-- Slave_IO_Running: I/O 线程是否运行
-- Slave_SQL_Running: SQL 线程是否运行
-- Seconds_Behind_Master: 延迟秒数
-- Relay_Log_Pos: relay log 位置
-- Exec_Master_Log_Pos: 已执行的 binlog 位置

-- GTID 模式
SHOW STATUS LIKE 'gtid%';
-- Slave_Pos: 当前 GTID 位置
-- Executed_Gtid_Set: 已执行的 GTID 集合
```

## 总结

主从复制是 MariaDB 高可用架构的基础：

- **传统复制**：依赖 binlog + 文件位置，简单但运维复杂
- **GTID 复制**：每个事务全局唯一，自动定位，切换主库更安全
- **并行复制**：多线程回放，解决从库延迟问题
- **半同步复制**：保证至少一个从库收到数据

GTID 是现代复制的标配。如果你还在用 `FILE + POS`，建议迁移到 GTID。迁移过程很简单：

```sql
SET GLOBAL gtid_pos_autoengines = '';
SET GLOBAL gtid_strict_mode = ON;
CHANGE MASTER TO MASTER_USE_GTID = current_pos;
START SLAVE;
```

<Badge text="最佳实践" type="tip" /> 主从复制的核心监控指标是 `Seconds_Behind_Master`。如果延迟持续增长，检查：从库 IO wait、网络延迟、SQL 线程 CPU 使用率。
