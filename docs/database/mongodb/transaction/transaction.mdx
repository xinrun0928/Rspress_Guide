# MongoDB 多文档事务

「MongoDB 事务怎么用？和关系型数据库的事务有什么区别？」

这是从关系型数据库转向 MongoDB 的开发者常问的问题。在 MongoDB 4.0 之前，MongoDB 只支持单文档原子性；4.0 之后，MongoDB 正式支持多文档事务，让复杂业务场景的数据一致性成为可能。

本文深入讲解 MongoDB 多文档事务的使用方法和注意事项。

## 事务基础

### 什么是多文档事务

多文档事务（Multi-Document Transaction）允许在多个文档、多个集合甚至多个数据库之间执行一组操作，这些操作要么全部成功，要么全部回滚。

```javascript
// MongoDB 事务的基本结构
const session = db.getMongo().startSession();

session.startTransaction({
  readConcern: { level: "snapshot" },
  writeConcern: { w: "majority" }
});

try {
  const ordersCollection = session.getDatabase("shop").orders;
  const inventoryCollection = session.getDatabase("shop").inventory;

  // 操作1：创建订单
  ordersCollection.insertOne({
    customer_id: ObjectId("..."),
    items: [{ product_id: ObjectId("..."), quantity: 2 }],
    total: 200,
    status: "pending"
  });

  // 操作2：扣减库存
  inventoryCollection.updateOne(
    { product_id: ObjectId("..."), quantity: { $gte: 2 } },
    { $inc: { quantity: -2 } }
  );

  // 提交事务
  session.commitTransaction();
} catch (error) {
  // 回滚事务
  session.abortTransaction();
  throw error;
} finally {
  session.endSession();
}
```

### 事务 API

#### startTransaction

开启一个新事务。

```javascript
session.startTransaction({
  readConcern: { level: "snapshot" },
  writeConcern: { w: "majority" }
});
```

| 选项 | 可选值 | 说明 |
|---|---|---|
| `readConcern` | `local`、`majority`、`snapshot` | 读取隔离级别 |
| `writeConcern` | `w: 1`、`w: majority`、`w: <number>` | 写入确认级别 |
| `maxTimeMS` | 毫秒数 | 事务超时时间 |

#### commitTransaction

提交事务，将所有修改持久化。

```javascript
session.commitTransaction();
```

#### abortTransaction

中止并回滚事务，撤销所有未提交的修改。

```javascript
session.abortTransaction();
```

### Node.js 驱动示例

```javascript
const { MongoClient } = require("mongodb");

async function transferFunds() {
  const client = new MongoClient("mongodb://localhost:27017");
  await client.connect();

  const session = client.startSession();

  try {
    await session.withTransaction(async () => {
      const accounts = client.db("bank").collection("accounts");

      // 扣款
      await accounts.updateOne(
        { _id: "account_a", balance: { $gte: 100 } },
        { $inc: { balance: -100 } },
        { session }
      );

      // 存款
      await accounts.updateOne(
        { _id: "account_b" },
        { $inc: { balance: 100 } },
        { session }
      );
    });

    console.log("转账成功");
  } catch (error) {
    console.log("转账失败，已回滚:", error.message);
  } finally {
    await session.endSession();
    await client.close();
  }
}
```

### withTransaction 封装

`withTransaction` 是最推荐的事务使用方式，它自动处理重试逻辑。

```javascript
await session.withTransaction(async () => {
  // 所有操作都在事务中执行
  await collection1.insertOne({ ... }, { session });
  await collection2.updateOne({ ... }, { ... }, { session });
}, {
  readPreference: "primary",
  readConcern: { level: "snapshot" },
  writeConcern: { w: "majority" }
});
```

## 事务的使用场景

### 适合使用事务的场景

```javascript
// 1. 跨集合的数据一致性
// 订单 + 库存 + 积分
async function createOrder(orderData) {
  const session = startSession();
  await session.withTransaction(async () => {
    // 创建订单
    const order = await orders.insertOne(orderData, { session });

    // 扣减库存
    await inventory.updateOne(
      { product_id: orderData.product_id },
      { $inc: { quantity: -orderData.quantity } },
      { session }
    );

    // 增加积分
    await users.updateOne(
      { _id: orderData.customer_id },
      { $inc: { points: orderData.total } },
      { session }
    );
  });
}

// 2. 账户转账
// 扣款 + 存款
async function transfer(from, to, amount) {
  await session.withTransaction(async () => {
    await accounts.updateOne(
      { _id: from, balance: { $gte: amount } },
      { $inc: { balance: -amount } },
      { session }
    );
    await accounts.updateOne(
      { _id: to },
      { $inc: { balance: amount } },
      { session }
    );
  });
}

// 3. 用户注册 + 创建初始数据
async function registerUser(userData) {
  await session.withTransaction(async () => {
    const user = await users.insertOne(userData, { session });
    await wallets.insertOne(
      { user_id: user.insertedId, balance: 0 },
      { session }
    );
    await preferences.insertOne(
      { user_id: user.insertedId, theme: "light" },
      { session }
    );
  });
}
```

## 事务的隔离级别

### readConcern 选项

| 级别 | 说明 | 适用场景 |
|---|---|---|
| `local` | 读取本地最新的数据，可能未同步到多数节点 | 性能优先，允许少量不一致 |
| `majority` | 读取被多数节点确认的数据 | 强一致性，推荐使用 |
| `snapshot` | 读取事务开始时的快照 | 事务内部读取，RR 隔离级别 |

```javascript
// 推荐配置
session.startTransaction({
  readConcern: { level: "snapshot" },   // 事务内读取一致
  writeConcern: { w: "majority" }       // 写入需多数节点确认
});
```

### writeConcern 选项

| 选项 | 说明 | 特点 |
|---|---|---|
| `w: 1` | 仅主节点确认 | 最快，但不保证持久性 |
| `w: majority` | 多数节点确认 | 推荐，平衡性能和安全 |
| `w: <number>` | 指定节点数确认 | 特定场景 |

```javascript
// 强一致配置
session.startTransaction({
  readConcern: { level: "snapshot" },
  writeConcern: { w: "majority", j: true, wtimeout: 5000 }
});
```

## 事务的限制

### 硬性限制

| 限制项 | 值 | 说明 |
|---|---|---|
| 事务超时 | 60 秒（默认） | 可通过 `maxTimeMS` 配置 |
| 事务内操作数 | 无硬性限制 | 受内存和超时限制 |
| 嵌套事务 | 不支持 | 事务内不能开启新事务 |

### 性能注意事项

```javascript
// 1. 事务不宜过大
// ❌ 事务包含大量操作
await session.withTransaction(async () => {
  for (const item of largeArray) {
    await collections.updateOne(...);
  }
});

// ✅ 分批处理
for (const batch of chunk(largeArray, 100)) {
  await session.withTransaction(async () => {
    for (const item of batch) {
      await collections.updateOne(...);
    }
  });
}

// 2. 避免长时间持锁
// ❌ 事务内包含网络请求
await session.withTransaction(async () => {
  await collection.updateOne(...);
  const response = await httpClient.get(...);  // 可能很慢
  await collection.updateOne(...);
});

// ✅ 事务内只做数据库操作
```

## 事务与副本集

### 副本集内事务

副本集内的事务直接在主节点执行，通过 oplog 保证复制。

```javascript
const session = replicaSet.getClient().startSession();

await session.withTransaction(async () => {
  // 所有操作在主节点执行
  // oplog 记录所有操作
  // 从节点异步复制
});
```

## 面试实战

**面试官问**：「MongoDB 的事务和 MySQL 的事务有什么区别？」

**深入回答**：

> 核心区别在于**设计理念**：
>
> 1. **使用方式**：MySQL 的事务是默认开启的，MongoDB 需要显式开启事务
>
> 2. **性能开销**：MongoDB 的文档模型使大多数操作不需要跨文档事务，只有必要时才使用
>
> 3. **隔离级别**：MySQL 有 4 个标准隔离级别，MongoDB 主要使用 snapshot
>
> 4. **分布式支持**：MongoDB 分片集群的事务是两阶段提交，比单节点事务复杂

**面试官问**：「MongoDB 事务有什么性能影响？」

**深入回答**：

> 事务的性能影响：
>
> 1. **锁的开销**：事务期间会持有锁，过长的事务会阻塞其他操作
>
> 2. **网络开销**：分片事务需要在多个节点间协调
>
> 3. **存储开销**：事务日志需要额外的磁盘空间
>
> 优化建议：
>
> - 保持事务简短
> - 避免在事务内进行网络请求
> - 使用合适的 writeConcern
> - 考虑业务层面的一致性设计

**面试官问**：「MongoDB 单文档原子性是怎么实现的？」

**深入回答**：

> MongoDB 单文档原子性的实现机制：
>
> 1. **WiredTiger 的文档级锁**：WiredTiger 存储引擎使用乐观并发控制，文档级别的锁保证了单文档操作的原子性
>
> 2. **MMAPv1 的集合级锁**：MMAPv1 引擎使用集合级锁，但单文档操作仍然是原子的
>
> 3. **Write Concern**：配合 `w: majority` 可以确保写入被复制到多个节点
>
> 4. **Read Concern**：配合 `readConcern` 可以控制读取的一致性视图

## 总结

MongoDB 多文档事务的使用要点：

- **开启事务**：`startSession()` + `startTransaction()` 或 `withTransaction()`
- **提交**：`commitTransaction()`
- **回滚**：`abortTransaction()`
- **推荐配置**：`readConcern: snapshot` + `writeConcern: majority`
- **性能注意**：保持事务简短，避免大事务

事务是保障数据一致性的重要工具，但应合理使用，避免过度依赖事务而影响性能。
