# MongoDB 事务隔离级别

「MongoDB 事务的隔离级别是什么？能不能防止脏读？」

这是理解 MongoDB 事务一致性的关键问题。MongoDB 的事务隔离级别与关系型数据库类似，但实现机制有所不同。

本文深入讲解 MongoDB 的事务隔离级别和隔离保证。

## 隔离级别概述

### MongoDB 的隔离级别

MongoDB 支持三种 readConcern 级别：

| 级别 | 说明 | 等价于 SQL |
|---|---|---|
| `local` | 读取本地最新的数据 | READ UNCOMMITTED |
| `snapshot` | 读取快照数据，事务内一致 | REPEATABLE READ |
| `majority` | 读取被多数节点确认的数据 | SERIALIZABLE（读） |

### 与 SQL 标准的对比

| SQL 隔离级别 | 脏读 | 不可重复读 | 幻读 |
|---|---|---|---|
| READ UNCOMMITTED | 可能 | 可能 | 可能 |
| READ COMMITTED | 不可能 | 可能 | 可能 |
| REPEATABLE READ | 不可能 | 不可能 | 可能 |
| SERIALIZABLE | 不可能 | 不可能 | 不可能 |

## readConcern 详解

### local（读未提交）

读取本地节点最新的数据，可能是尚未同步到多数节点的数据。

```javascript
// local 级别
db.orders.find({}, { readConcern: { level: "local" } })

// 特点：
// - 读取延迟最低
// - 可能读到未持久化的数据
// - 主节点故障可能丢失数据
// - 不推荐用于重要读取
```

:::warning local 级别风险

`local` 级别可能读到：
- 尚未写入多数节点的确认数据
- 已写入但未同步到从节点的数据
- 回滚段中的未提交数据

仅在可以容忍少量数据丢失的场景使用。

:::

### majority（读已提交）

读取被多数节点确认的数据，保证数据不会被回滚。

```javascript
// majority 级别
db.orders.find({}, { readConcern: { level: "majority" } })

// 特点：
// - 读取被 w: majority 确认的数据
// - 不会读到被回滚的数据
// - 可能读到旧版本的数据
// - 推荐用于需要一致性的读取
```

:::tip majority 级别保证

`majority` 级别保证：
- 读取的数据已经在多数节点确认
- 读取的数据不会被后续操作回滚
- 不保证读取到最新版本，但保证读取一致性

:::

### snapshot（快照读）

读取事务开始时的数据快照，事务内多次读取结果一致。

```javascript
// snapshot 级别（仅在事务内可用）
session.startTransaction({
  readConcern: { level: "snapshot" }
});

// 特点：
// - 读取事务开始时的数据快照
// - 事务内多次读取结果相同
// - 不会出现不可重复读
// - 事务隔离效果最强
```

## 隔离级别与事务

### 事务内的 readConcern

在事务内，只能使用 `snapshot` 和 `local` 级别。

```javascript
// ✅ 正确：事务内使用 snapshot
session.startTransaction({
  readConcern: { level: "snapshot" }
});

// ✅ 正确：事务内使用 local
session.startTransaction({
  readConcern: { level: "local" }
});

// ❌ 错误：事务内不能使用 majority
session.startTransaction({
  readConcern: { level: "majority" }  // 不支持
});
```

### readConcern + writeConcern 组合

| 组合 | 说明 | 使用场景 |
|---|---|---|
| `{ rc: "snapshot", wc: "majority" }` | 最强一致性保证 | 金融、转账等关键业务 |
| `{ rc: "majority", wc: "majority" }` | 多数确认读写 | 需要复制保证的场景 |
| `{ rc: "local", wc: "1" }` | 本地读写，最快 | 日志、可丢失的数据 |

```javascript
// 强一致事务配置
const session = client.startSession();
session.startTransaction({
  readConcern: { level: "snapshot" },
  writeConcern: { w: "majority", wtimeout: 5000 }
});
```

## MVCC 与隔离实现

### WiredTiger 的 MVCC

MongoDB 使用 WiredTiger 存储引擎的 MVCC（多版本并发控制）实现快照隔离。

```
MVCC 工作原理：

时刻 T1：事务开始
┌─────────────────────────────────────────┐
│  快照数据（事务可见）                      │
│  ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐       │
│  │ V1  │ │ V2  │ │ V3  │ │ V4  │       │
│  └─────┘ └─────┘ └─────┘ └─────┘       │
└─────────────────────────────────────────┘

时刻 T2：事务提交，新版本写入
┌─────────────────────────────────────────┐
│  新快照数据                               │
│  ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐│
│  │ V1  │ │ V2  │ │ V3  │ │ V4  │ │ V5* ││
│  └─────┘ └─────┘ └─────┘ └─────┘ └─────┘│
│                                    新版本 │
└─────────────────────────────────────────┘

* V5 对当前事务不可见，直到事务提交
```

### 版本链与回滚

```javascript
// WiredTiger 的文档版本链
// _id: 1, name: "张三"  (版本5)
//   ↓ roll_ptr
// _id: 1, name: "李四"  (版本4)
//   ↓ roll_ptr
// _id: 1, name: "王五"  (版本3)
//   ...

// 事务读取时：
// 1. 读取事务开始时的快照
// 2. 通过 roll_ptr 链找到对应版本
// 3. 返回快照版本的数据
```

## 脏读与防止

### 脏读场景

脏读：读取到其他事务未提交的修改。

```javascript
// 场景：事务A 修改数据但未提交

// 事务A
sessionA.startTransaction();
await db.users.updateOne(
  { _id: 1 },
  { $set: { balance: 0 } }
);
// 未提交，balance 仍为 100

// 事务B（不同的 session）
const user = await db.users.findOne({ _id: 1 });
console.log(user.balance);  // ?

// 不同 readConcern 的结果：
// - local: 可能读到 0（脏读）
// - majority: 读到 100（旧值）
// - snapshot: 在事务中读到 100（一致）
```

### 防止脏读

```javascript
// ✅ 使用 snapshot 隔离
session.startTransaction({
  readConcern: { level: "snapshot" }
});

// ✅ 使用 majority 读
db.users.findOne(
  { _id: 1 },
  { readConcern: { level: "majority" } }
);

// ✅ 使用事务
await session.withTransaction(async () => {
  // 事务内的所有读取都是 snapshot 隔离
  const user = await db.users.findOne({ _id: 1 }, { session });
  return user;
});
```

## 不可重复读与防止

### 不可重复读场景

不可重复读：同一事务内多次读取同一数据，结果不同。

```javascript
// 场景：事务A 两次读取同一数据，中间事务B修改并提交

// 事务A
const user1 = await db.users.findOne({ _id: 1 });
console.log(user1.balance);  // 第一次读：100

// 事务B（在事务A之外）
await db.users.updateOne(
  { _id: 1 },
  { $set: { balance: 200 } }
);

// 事务A
const user2 = await db.users.findOne({ _id: 1 });
console.log(user2.balance);  // 第二次读：?

// 不同隔离级别的结果：
// - local: 可能读到 200（不可重复读）
// - majority: 可能读到 200
// - snapshot（在事务中）：读到 100（可重复读）
```

### 防止不可重复读

```javascript
// ✅ 使用事务 + snapshot
await session.withTransaction(async () => {
  const user1 = await db.users.findOne({ _id: 1 }, { session });
  // ... 其他操作 ...
  const user2 = await db.users.findOne({ _id: 1 }, { session });
  // user1 === user2，事务内一致
});
```

## 幻读与防止

### 幻读场景

幻读：同一事务内两次查询，结果集不同（新增了记录）。

```javascript
// 场景：事务A 两次查询，中间事务B插入了新数据

// 事务A
const orders1 = await db.orders.find({ customer_id: 1 });
console.log(orders1.length);  // 第一次：5条

// 事务B
await db.orders.insertOne({ customer_id: 1, amount: 100 });

// 事务A
const orders2 = await db.orders.find({ customer_id: 1 });
console.log(orders2.length);  // 第二次：6条（幻读）
```

### 防止幻读

```javascript
// ✅ 使用 SERIALIZABLE 隔离（通过快照）
// MongoDB 的 snapshot 隔离可以防止幻读
await session.withTransaction(async () => {
  const snapshot = await db.orders.find({ customer_id: 1 }).toArray();
  // 整个查询结果集是快照，不会有新增
});

// ✅ 使用分布式锁
// 在应用层实现序列化
const lock = await acquireLock("customer:1:orders");
try {
  await session.withTransaction(async () => {
    // 临界区操作
  });
} finally {
  await releaseLock(lock);
}
```

## 隔离级别选择

### 选择原则

| 场景 | 推荐 readConcern | 推荐 writeConcern |
|---|---|---|
| 金融交易 | `snapshot` | `majority` |
| 订单处理 | `majority` | `majority` |
| 用户配置读取 | `majority` | `1` |
| 日志写入 | `local` | `1` |
| 批量统计 | `majority` | `1` |

### 配置示例

```javascript
// 金融场景：最强一致性
const txnOptions = {
  readConcern: { level: "snapshot" },
  writeConcern: { w: "majority", wtimeout: 10000 }
};

// 普通业务：平衡一致性和性能
const normalOptions = {
  readConcern: { level: "majority" },
  writeConcern: { w: "majority" }
};

// 实时数据：高性能优先
const fastOptions = {
  readConcern: { level: "local" },
  writeConcern: { w: 1 }
};
```

## 面试实战

**面试官问**：「MongoDB 的 snapshot 隔离级别和 MySQL 的 REPEATABLE READ 有什么区别？」

**深入回答**：

> 核心区别：
>
> 1. **实现机制**：MySQL REPEATABLE READ 通过 MVCC + Next-Key Lock 实现；MongoDB snapshot 通过 WiredTiger 的 MVCC 实现
>
> 2. **幻读处理**：MySQL REPEATABLE READ 可能出现幻读（通过 Gap Lock 缓解）；MongoDB snapshot 不会出现幻读
>
> 3. **锁粒度**：MySQL InnoDB 是行级锁；MongoDB WiredTiger 是文档级锁
>
> 4. **索引访问**：MySQL 通过索引条件判断锁范围；MongoDB 通过索引确定文档位置

**面试官问**：「readConcern: majority 和 readConcern: snapshot 怎么选？」

**深入回答**：

> 选择依据：
>
> 1. **是否在事务内**：
>    - 事务内：必须用 `snapshot`
>    - 非事务：只能用 `local` 或 `majority`
>
> 2. **一致性要求**：
>    - 最高一致性：snapshot + majority
>    - 强一致性：majority
>    - 最终一致性：local
>
> 3. **性能要求**：
>    - snapshot 需要事务，开销最大
>    - majority 需要多数节点确认，有一定延迟
>    - local 只读本地，最快

## 总结

MongoDB 事务隔离级别要点：

- **local**：读取本地最新数据，可能脏读
- **majority**：读取多数节点确认的数据，不会读到被回滚的数据
- **snapshot**：读取事务开始时的快照，事务内一致（仅事务内可用）
- **MVCC**：通过 WiredTiger 的 MVCC 实现快照隔离
- **writeConcern**：配合 `majority` 获得最强一致性保证

理解隔离级别，是正确使用 MongoDB 事务的基础。
