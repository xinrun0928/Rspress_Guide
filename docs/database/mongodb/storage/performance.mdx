# MongoDB 常见性能瓶颈与优化

「MongoDB 越来越慢，怎么排查？」

性能问题是生产环境最常见的问题之一。MongoDB 的性能瓶颈通常来自几个方面：索引缺失、查询模式不当、内存不足、写入压力过大等。

本文介绍常见的性能瓶颈和优化方法。

## 性能瓶颈概览

| 瓶颈类型 | 症状 | 排查方法 |
|---|---|---|
| 索引缺失 | 查询慢、全表扫描 | `explain()` 分析 |
| 内存不足 | Cache 命中率低 | `serverStatus().wiredTiger.cache` |
| 锁竞争 | 并发性能差 | `currentOp()` 查看锁等待 |
| 网络瓶颈 | 延迟高 | `serverStatus().network` |
| 磁盘 IO | 读写慢 | `mongostat` 查看 IO |

## 索引相关瓶颈

### 症状

```javascript
// 1. 查询越来越慢
// 2. docsExamined 远大于 nreturned
// 3. COLLSCAN（全表扫描）出现在 explain 中
```

### 排查方法

```javascript
// 使用 explain 分析查询
db.orders.find({ customer_id: "C001" }).explain("executionStats")

// 输出示例
{
  "queryPlanner": {
    "winningPlan": {
      "stage": "COLLSCAN",  // ❌ 全表扫描
      "filter": { "customer_id": "C001" }
    },
    "rejectedPlans": []
  },
  "executionStats": {
    "executionTimeMillis": 1500,
    "totalDocsExamined": 1000000,  // 检查了 100 万文档
    "nreturned": 10                // 只返回 10 条
  }
}

// 正确情况：使用 IXSCAN
{
  "queryPlanner": {
    "winningPlan": {
      "stage": "IXSCAN",  // ✅ 索引扫描
      "indexName": "customer_id_1"
    }
  },
  "executionStats": {
    "executionTimeMillis": 5,
    "totalDocsExamined": 10,
    "nreturned": 10
  }
}
```

### 解决方案

```javascript
// 1. 添加索引
db.orders.createIndex({ customer_id: 1 })

// 2. 复合索引（根据查询模式）
db.orders.createIndex({ customer_id: 1, created_at: -1 })

// 3. 覆盖索引（查询只返回索引字段）
db.orders.createIndex(
  { customer_id: 1, created_at: 1 },
  { name: "customer_date_idx" }
)

// 4. 删除无用索引
db.orders.dropIndex("unused_index")
```

### 索引优化建议

<Badge text="重要" type="warning" /> 索引不是越多越好，每个索引都会增加写入开销。

```javascript
// 索引优化原则：

// 1. 优先为常用查询添加索引
//    监控 Profiler 找出慢查询，然后加索引

// 2. 复合索引字段顺序
//    等值字段在前，范围字段在后
//    查询: { status: "active", age: { $gte: 18 } }
//    索引: { status: 1, age: 1 }

// 3. 覆盖索引
//    如果查询只返回索引字段，不需要回表
//    查询: { customer_id: "C001" }
//    投影: { _id: 0, customer_id: 1, created_at: 1 }
//    索引: { customer_id: 1, created_at: 1 }

// 4. 定期清理无用索引
db.collection.aggregate([{ $indexStats: {} }])
// 找出使用次数为 0 的索引
```

## 内存相关瓶颈

### 症状

```javascript
// 1. Cache 使用率接近 100%
db.serverStatus().wiredTiger.cache['percentage of cache in use']
// 长期 > 90%

// 2. 淘汰页数增加
db.serverStatus().wiredTiger.cache.pages_evicted
// 持续很高

// 3. 查询变慢（Cache miss）
```

### 排查方法

```javascript
// 1. 查看 Cache 使用
db.serverStatus().wiredTiger.cache

// 2. 查看内存趋势
// mongostat 每秒查看 dirty 和 used
mongostat --host localhost:27017 1

// 3. 查看慢查询
db.system.profile.find({ millis: { $gt: 1000 } })
```

### 解决方案

```javascript
// 1. 增加物理内存
// 最直接的解决方案

// 2. 调整 Cache 大小
// mongod.conf
storage:
  wiredTiger:
    engineConfig:
      cacheSizeGB: 16  # 增加 Cache

// 3. 优化查询，减少数据加载
// 避免返回大字段
db.orders.find(
  { customer_id: "C001" },
  { _id: 1, amount: 1, status: 1 }  # 只返回必要字段
)

// 4. 使用分片分散数据
// 热数据分散到多个分片的 Cache
```

## 锁竞争瓶颈

### 症状

```javascript
// 1. 并发性能差
// 单请求快，多请求慢

// 2. 写入队列增加
// mongostat 显示 qrw > 0

// 3. 锁等待超时
// error: "LockTimeout"
```

### 排查方法

```javascript
// 1. 查看当前锁等待操作
db.currentOp({
  "waitingForLock": true
})

// 2. 查看长时间运行的操作
db.currentOp({
  "active": true,
  "secs_running": { $gt: 10 }
})

// 3. 查看锁统计
db.serverStatus().locks
```

### 解决方案

```javascript
// 1. 使用事务时保持简短
// ❌ 事务包含大量操作
await session.withTransaction(async () => {
  for (const item of largeArray) {
    await db.collection.updateOne(...);
  }
});

// ✅ 小事务 + 分批处理

// 2. 避免长时间持锁
// ❌ 事务内包含网络请求
await session.withTransaction(async () => {
  await db.collection.updateOne(...);
  const response = await httpClient.get(...);  // 阻塞
  await db.collection.updateOne(...);
});

// 3. 优化查询，减少锁持有时间

// 4. 使用分片，分散锁竞争
sh.shardCollection("shop.orders", { customer_id: "hashed" })
```

## 写入瓶颈

### 症状

```javascript
// 1. 写入延迟增加
// 2. 写入队列增加
// 3. Journal 写入压力高
```

### 排查方法

```javascript
// 1. mongostat 查看写入
mongostat --host localhost:27017 1
// insert/update/delete 是否持续很高

// 2. 查看慢写入
db.system.profile.find({ op: "insert" }).sort({ millis: -1 }).limit(10)

// 3. 查看索引写入开销
db.collection.getIndexes()
// 索引越多，写入越慢
```

### 解决方案

```javascript
// 1. 批量写入
// ❌ 逐条插入
for (const doc of docs) {
  await db.collection.insertOne(doc);
}

// ✅ 批量插入
await db.collection.insertMany(docs, { ordered: false })

// 2. 使用无序写入
await db.collection.insertMany(docs, { ordered: false })
// 无序写入可以并行处理

// 3. 减少索引
// 只保留必要的索引
db.collection.createIndex({ a: 1 })  // 必要
db.collection.createIndex({ b: 1 })  // 评估是否必要

// 4. 使用 writeConcern 权衡
// ❌ 每次写入都等待多数节点确认
await db.collection.insertOne(doc, { writeConcern: { w: "majority", j: true } })

// ✅ 根据业务需求选择
// 日志：w: 0
// 普通数据：w: 1
// 重要数据：w: "majority"
```

## 查询模式优化

### 常见问题

```javascript
// 1. 字段类型不匹配
// 集合字段是 Number，查询用字符串
db.collection.find({ amount: "100" })  // ❌

db.collection.find({ amount: NumberLong("100") })  // ✅

// 2. 正则表达式全匹配
db.collection.find({ email: /.*@example\.com/ })  // ❌ 无法使用索引

db.collection.find({ email: /zhangsan@example\.com/ })  // 部分优化

db.collection.find({ email: "zhangsan@example.com" })  // ✅ 使用索引

// 3. $exists 查询
db.collection.find({ optional_field: { $exists: true } })  // 可能无法使用索引

// 4. 深度嵌套查询
db.collection.find({ "address.city.name": "北京" })  // 较慢

// 可以考虑反范式化设计
```

### 分页优化

```javascript
// ❌ 错误：skip 大数据量很慢
const page10 = await db.collection.find().skip(100000).limit(10)

// ✅ 正确：基于游标分页
// 上一页最后一条
const lastDoc = await db.collection.findOne({ user_id: "U001" });
const page2 = await db.collection.find({
  user_id: "U001",
  _id: { $gt: lastDoc._id }
}).limit(10);

// ✅ 正确：基于时间分页
const page2 = await db.collection.find({
  user_id: "U001",
  created_at: { $lt: lastTimestamp }
```

## 面试实战

**面试官问**：「MongoDB 查询慢怎么排查？」

**深入回答**：

> 排查步骤：
>
> 1. **使用 explain 分析**：找出是 COLLSCAN 还是 IXSCAN
>
> 2. **检查索引**：常用查询是否有索引
>
> 3. **检查内存**：Cache 是否够用
>
> 4. **检查锁**：是否有锁竞争
>
> 5. **分析 Profiler**：找出慢查询模式

**面试官问**：「如何避免 MongoDB 全表扫描？」

**深入回答**：

> 方法：
>
> 1. **为查询条件添加索引**：确保查询字段有索引
>
> 2. **使用覆盖索引**：查询和投影都只包含索引字段
>
> 3. **避免正则全匹配**：正则要从头匹配才能使用索引
>
> 4. **避免类型不匹配**：查询字段类型要与存储类型一致
>
> 5. **使用 explain 验证**：确认查询使用索引

## 总结

性能优化要点：

- **索引优化**：添加必要索引，删除无用索引
- **内存优化**：合理配置 Cache，监控使用率
- **锁优化**：保持事务简短，避免长时间持锁
- **写入优化**：批量操作，减少索引
- **查询优化**：避免全表扫描，使用分页
- **分片优化**：选择合适的分片键

性能优化是一个持续的过程，需要不断监控和调整。
