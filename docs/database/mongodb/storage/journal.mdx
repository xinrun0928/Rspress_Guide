# MongoDB Journal 日志与崩溃恢复

Journal 是 WiredTiger 的预写日志（WAL），记录所有写操作，确保在崩溃后能够恢复数据。

## Journal 概述

### 什么是 Journal

Journal 是 WiredTiger 的写前日志（Write-Ahead Log），在数据写入磁盘前先记录日志，确保数据持久性。

```
Journal 工作原理：

写入流程（启用 Journal）：

┌─────────────────────────────────────────────────┐
│ 1. 写入 Journal                                 │
│    { op: "update", doc_id: 1, new_value: ... } │
│    立即写入 Journal 文件                         │
└─────────────────────────────────────────────────┘
                        │
                        ▼
┌─────────────────────────────────────────────────┐
│ 2. 写入 Cache                                   │
│    修改内存中的数据页                             │
│    标记为 dirty                                 │
└─────────────────────────────────────────────────┘
                        │
                        ▼
┌─────────────────────────────────────────────────┐
│ 3. Checkpoint (定期)                            │
│    将脏页写入数据文件                             │
│    清空 Journal 文件                             │
└─────────────────────────────────────────────────┘

崩溃恢复流程：

┌─────────────────────────────────────────────────┐
│ 1. MongoDB 启动                                  │
│    发现上次 Checkpoint                         │
└─────────────────────────────────────────────────┘
                        │
                        ▼
┌─────────────────────────────────────────────────┐
│ 2. 读取 Checkpoint                              │
│    恢复 Checkpoint 时的数据状态                  │
└─────────────────────────────────────────────────┘
                        │
                        ▼
┌─────────────────────────────────────────────────┐
│ 3. 重放 Journal                                 │
│    应用 Checkpoint 后的 Journal 操作             │
│    恢复到崩溃前的最新状态                         │
└─────────────────────────────────────────────────┘
```

### Journal vs Checkpoint

| 特性 | Journal | Checkpoint |
|---|---|---|
| 作用 | 记录所有写操作 | 将内存数据写入磁盘 |
| 时机 | 每个写操作 | 定期（默认 60 秒） |
| 数据量 | 增量记录 | 全量快照 |
| 用途 | 崩溃恢复 | 数据持久化 |

## Journal 配置

### 启用 Journal

```javascript
// Journal 默认启用
// MongoDB 3.2+ 默认启用 Journal

// 禁用 Journal（不推荐！）
// mongod.conf
storage:
  journal:
    enabled: false  // 禁用 Journal

// 注意：禁用 Journal 后，崩溃可能丢失数据
```

### Journal 参数

```javascript
// mongod.conf
storage:
  journal:
    enabled: true
    commitIntervalMs: 100      # Journal 写入间隔（毫秒）
    commitRatio: 0.1           # 脏页比例阈值

// 参数说明：
// - commitIntervalMs: Journal 写入间隔，默认 100ms
//   越小越安全，但写入开销越大
// - commitRatio: 脏页比例达到此值时强制提交
```

### 日志格式

```javascript
// Journal 记录内容
// 每个记录包含：
// - 操作类型（insert、update、delete）
// - 命名空间（数据库.集合）
// - 文档 ID
// - 修改前后的数据

// 示例
{
  lsn: 123456,              // 日志序列号
  op: "update",            // 操作类型
  ns: "shop.orders",       // 命名空间
  docid: ObjectId("..."),  // 文档 ID
  pre: { value: 100 },      // 修改前的值
  post: { value: 150 }     // 修改后的值
}
```

## 崩溃恢复

### 恢复过程

```javascript
// 崩溃恢复步骤：

// 1. 检测崩溃
// MongoDB 启动时检测 Journal 文件

// 2. 读取 Checkpoint
// 找到最近的 Checkpoint 标记

// 3. 重放 Journal
// 从 Checkpoint 之后开始重放所有 Journal 记录

// 4. 完成恢复
// 数据恢复到崩溃前的状态

// 查看恢复日志
// tail -f /var/log/mongodb/mongod.log
// [initandlisten] Recovering from journal
// [initandlisten] Replaying journal records
// [initandlisten] Recovery complete
```

### 恢复时间

```javascript
// 恢复时间因素：

// 1. Journal 文件大小
//    Journal 越大，恢复时间越长

// 2. Checkpoint 间隔
//    Checkpoint 间隔越小，Journal 越小，恢复越快

// 3. 数据量
//    需要重放的操作越多，恢复越慢

// 4. 硬件性能
//    磁盘 IO 速度影响恢复速度

// 查看恢复统计
db.serverStatus().wiredTiger.txn
```

### 数据一致性保证

```javascript
// Journal 提供的数据保证：

// 1. 原子性
//    Journal 记录是原子的，一个记录要么全部写入，要么全部不写

// 2. 持久性
//    Journal 写入成功 = 数据已持久化
//    即使数据文件损坏，也可以从 Journal 恢复

// 3. 一致性
//    Journal 重放后，数据恢复到一致状态
```

## Write Concern 与 Journal

### j 选项

```javascript
// writeConcern 中的 j 选项控制 Journal 确认

// j: false（默认）
// 不等待 Journal 写入
db.collection.insertOne(
  { data: "value" },
  { writeConcern: { w: 1, j: false } }
)

// j: true
// 等待 Journal 写入成功
db.collection.insertOne(
  { data: "value" },
  { writeConcern: { w: 1, j: true } }
)

// 注意：j: true 会增加写入延迟
```

### 组合使用

```javascript
// 组合策略

// 1. 最高持久性（最慢）
{ w: "majority", j: true }
// 需要多数节点确认 + Journal 写入成功

// 2. 高持久性（较快）
{ w: "majority", j: false }
// 多数节点确认即可，不强制 Journal

// 3. 普通模式（快）
{ w: 1, j: false }
// 主节点确认即可

// 4. 快速模式（最快）
{ w: 1, j: false }
// 最快，但不保证持久性
```

### 选择建议

| 场景 | writeConcern | 说明 |
|---|---|---|
| 金融交易 | `{ w: "majority", j: true }` | 最高持久性 |
| 订单处理 | `{ w: "majority", j: false }` | 平衡性能和安全 |
| 实时数据 | `{ w: 1, j: true }` | 主节点持久性 |
| 日志 | `{ w: 0, j: false }` | 不关心持久性 |

## 性能影响

### Journal 对性能的影响

```javascript
// Journal 写入开销：

// 1. 写入延迟
// Journal 写入会增加写入延迟
// 默认 100ms 间隔，可调整

// 2. 磁盘空间
// Journal 需要磁盘空间
// 默认 3 个日志文件循环使用

// 3. 磁盘 IO
// 频繁的 Journal 写入
// 建议 Journal 使用独立磁盘
```

### 优化建议

```javascript
// Journal 性能优化：

// 1. 使用高速磁盘
// SSD 存储 Journal
storage:
  dbPath: /var/lib/mongodb        # 数据文件
  journal:
    enabled: true
# 建议 Journal 和数据文件使用不同磁盘

// 2. 调整提交间隔
// 对延迟不敏感的场景可以增大间隔
storage:
  journal:
    commitIntervalMs: 200  # 200ms 提交一次

// 3. 批量写入
// 使用批量操作减少 Journal 次数
db.collection.insertMany(docs, { writeConcern: { j: true } })
```

## 常见问题

### Journal 文件过大

```javascript
// 问题：Journal 文件占用过多磁盘空间

// 原因：
// - Checkpoint 间隔过长
// - 写入量大但 Checkpoint 不及时

// 解决方案：
// 1. 减小 Checkpoint 间隔
storage:
  wiredTiger:
    engineConfig:
      checkpoint:
        wait: 30  # 30 秒 Checkpoint 一次

// 2. 清理旧的 Journal 文件（停机维护）
```

### Journal 写入失败

```javascript
// 问题：Journal 写入失败导致 MongoDB 崩溃

// 可能原因：
// - 磁盘空间不足
// - Journal 所在磁盘损坏
// - 权限问题

// 解决方案：
// 1. 检查磁盘空间
df -h

// 2. 检查 Journal 目录权限
ls -la /var/lib/mongodb/journal/

// 3. 移动 Journal 到其他磁盘
```

## 总结

Journal 要点：

- **作用**：预写日志，保证数据持久性
- **工作原理**：写操作先记录到 Journal，然后写入数据文件
- **恢复机制**：通过 Checkpoint + Journal 实现崩溃恢复
- **writeConcern**：配合 `j: true` 可获得最高持久性
- **性能影响**：Journal 写入会增加延迟，可优化配置

正确配置 Journal 是保证 MongoDB 数据安全的关键。
