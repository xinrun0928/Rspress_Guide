# MongoDB 内存管理

「MongoDB WiredTiger Cache 是什么？占多少内存？」

理解 MongoDB 的内存管理对于性能调优至关重要。WiredTiger Cache 是 MongoDB 内存消耗的主要来源，合理配置可以显著提升性能。

## WiredTiger Cache

### Cache 概述

WiredTiger Cache 是存储引擎的内存缓存，用于缓存热数据，减少磁盘 IO。

```javascript
// WiredTiger Cache 配置

// 默认：占可用 RAM 的 50%（减去 1GB 或系统预留）
// 最大：取决于系统可用内存

// 查看 Cache 使用情况
db.serverStatus().wiredTiger.cache
```

### Cache 配置参数

```javascript
// mongod.conf
storage:
  wiredTiger:
    engineConfig:
      cacheSizeGB: 8  # Cache 大小 8GB

// 计算公式
// cacheSize = (总 RAM - 1GB) * 50%
// 例如：16GB RAM → 7.5GB Cache
```

### Cache 结构

```
WiredTiger Cache 分布：

┌─────────────────────────────────────────────────┐
│              WiredTiger Cache                    │
│                  (例如 8GB)                       │
├─────────────────────────────────────────────────┤
│                                                   │
│  ┌─────────────────────────────────────────┐   │
│  │         应用数据 (Leaf Pages)             │   │
│  │  热数据缓存，减少磁盘读取                    │   │
│  │  约 90% Cache 空间                       │   │
│  └─────────────────────────────────────────┘   │
│                                                   │
│  ┌─────────────────────────────────────────┐   │
│  │         索引数据 (Internal Pages)         │   │
│  │  索引树结构缓存                            │   │
│  │  约 10% Cache 空间                       │   │
│  └─────────────────────────────────────────┘   │
│                                                   │
│  ┌─────────────────────────────────────────┐   │
│  │         脏页 (Dirty Pages)               │   │
│  │  已修改待刷盘的数据                         │   │
│  │  约 10-20% Cache 空间                    │   │
│  └─────────────────────────────────────────┘   │
│                                                   │
└─────────────────────────────────────────────────┘
```

## Cache 淘汰策略

### 淘汰阈值

<Badge text="重要" type="warning" /> WiredTiger 的淘汰阈值会影响写入性能和内存使用。

```javascript
// 淘汰阈值配置
// mongod.conf
storage:
  wiredTiger:
    engineConfig:
      evictionTarget: 80    # 淘汰开始阈值（%）
      evictionTrigger: 95   # 强制淘汰阈值（%）
      evictionDirTrigger: 80  # 目录淘汰阈值

// 说明：
// - 内存使用达到 80%：开始淘汰干净页
// - 内存使用达到 95%：强制淘汰所有页
// - 磁盘空间达到 80%：停止写入
```

### 淘汰过程

```javascript
// 淘汰过程

// 1. 触发条件
//    Cache 使用率达到 evictionTrigger (默认 95%)

// 2. 淘汰策略
//    - 优先淘汰干净页（未修改）
//    - 按 LRU（最近最少使用）顺序淘汰
//    - 后台线程执行，不阻塞请求

// 3. 脏页处理
//    - 脏页需要先刷盘
//    - Checkpoint 会主动刷脏页
//    - 减少脏页比例

// 查看淘汰统计
db.serverStatus().wiredTiger.cache
```

## 连接与内存

### 连接内存开销

```javascript
// 每个连接都会消耗内存

// 连接内存配置
// mongod.conf
net:
  maxIncomingConnections: 65536  # 最大连接数

// 连接内存估算
// 每个连接约消耗 1-2MB 内存
// 10000 连接 → 约 10-20GB 内存开销

// 建议：连接数 + Cache 大小 < 总 RAM 的 80%
```

### 操作内存

```javascript
// 排序、聚合等操作需要内存

// 内存限制配置
// mongod.conf
operationProfiling:
  slowOpThresholdMs: 100  # 慢查询阈值

// sort 操作内存限制
// 默认 32MB，可通过 pipeline 调整
db.collection.aggregate([
  { $sort: { created_at: -1 } }
], { allowDiskUse: true })  // 允许使用磁盘

// 查看操作内存使用
db.serverStatus().wiredTiger.cache
```

## 内存配置最佳实践

### 配置原则

```javascript
// 内存配置原则：

// 1. Cache 大小
//    - 建议：总 RAM 的 50% - 1GB
//    - 最小：1GB
//    - 最大：根据系统可用内存

// 2. 连接数
//    - 根据应用需求设置
//    - 考虑每个连接 1-2MB 内存

// 3. 操作内存
//    - 排序、聚合需要内存
//    - 启用 allowDiskUse 避免内存溢出

// 4. 预留系统内存
//    - 系统需要内存用于其他进程
//    - 建议预留 2-4GB
```

### 推荐配置

```javascript
// 场景1：16GB RAM 服务器
storage:
  wiredTiger:
    engineConfig:
      cacheSizeGB: 7  # 16GB - 1GB 预留 = 15GB * 50% ≈ 7GB
net:
  maxIncomingConnections: 5000  # 5000 * 2MB = 10GB
# 总计约 17GB，小于 16GB，建议降低连接数

// 场景2：32GB RAM 服务器
storage:
  wiredTiger:
    engineConfig:
      cacheSizeGB: 15  # 32GB - 2GB = 30GB * 50% ≈ 15GB
net:
  maxIncomingConnections: 10000
# 总计约 35GB，建议增加 RAM 或降低配置

// 场景3：64GB RAM 服务器
storage:
  wiredTiger:
    engineConfig:
      cacheSizeGB: 30  # 留更多给系统
net:
  maxIncomingConnections: 20000
```

## OOM 问题处理

### 内存不足症状

```javascript
// 内存不足的表现：

// 1. 淘汰率升高
db.serverStatus().wiredTiger.cache.pages_evicted > 1000/秒

// 2. 写入变慢
// 大量脏页需要刷盘

// 3. 连接被拒绝
// 内存不足无法处理新连接

// 4. MongoDB 被 OOM Killer 杀掉
// 系统内存不足，OOM Killer 杀死进程
```

### 解决方案

```javascript
// 1. 增加物理内存
// 最直接的解决方案

// 2. 调整 Cache 大小
// mongod.conf
storage:
  wiredTiger:
    engineConfig:
      cacheSizeGB: 4  # 减小 Cache

// 3. 限制连接数
// mongod.conf
net:
  maxIncomingConnections: 3000  # 减少连接数

// 4. 优化查询
// 减少大结果集查询
// 使用索引覆盖
// 避免全表扫描

// 5. 启用磁盘操作
db.collection.aggregate(pipeline, { allowDiskUse: true })

// 6. 分片
// 水平扩展，分担内存压力
```

## 面试实战

**面试官问**：「MongoDB Cache 和系统内存是什么关系？」

**深入回答**：

> 内存关系：
>
> 1. **WiredTiger Cache**：默认占 RAM 的 50%
>
> 2. **连接内存**：每个连接约 1-2MB
>
> 3. **操作内存**：排序、聚合等操作
>
> 4. **系统预留**：约 1GB 用于系统进程
>
> 公式：`Cache + 连接内存 + 操作内存 + 系统 < 总 RAM`

**面试官问**：「如何判断 MongoDB 内存是否充足？」

**深入回答**：

> 判断方法：
>
> 1. **Cache 使用率**：`db.serverStatus().wiredTiger.cache.percentage of cache in use`
>    - 长期超过 80%：内存可能不足
>
> 2. **淘汰率**：`pages_evicted` 持续很高：Cache 不够用
>
> 3. **磁盘 IO**：频繁读取磁盘：Cache 命中率低
>
> 4. **慢查询增加**：内存不足导致性能下降

## 总结

内存管理要点：

- **WiredTiger Cache**：默认占 RAM 50%，缓存热数据
- **淘汰阈值**：80% 开始淘汰，95% 强制淘汰
- **连接内存**：每个连接约 1-2MB
- **操作内存**：排序、聚合需要内存，可使用磁盘
- **监控指标**：`pages_evicted`、`percentage of cache in use`

合理配置内存是 MongoDB 性能调优的关键。
