# WiredTiger 存储引擎原理

「MongoDB 的 WiredTiger 存储引擎是怎么工作的？」

理解 WiredTiger 的原理，对于性能调优和故障排查至关重要。大多数人知道 WiredTiger 是 MongoDB 的默认存储引擎，但追问下去——「B+ 树和 LSM 树有什么区别？」「MVCC 是怎么实现的？」「Checkpoint 和 Journal 是什么关系？」「Cache 是怎么管理的？」——能答上来的就很少了。

本文深入讲解 WiredTiger 存储引擎的原理和实现。

## WiredTiger 概述

### 什么是 WiredTiger

WiredTiger 是 MongoDB 3.2+ 默认的存储引擎，提供了高性能、高压缩比和并发控制能力。

```
WiredTiger 核心特性：

┌──────────────────────────────────────────────┐
│              WiredTiger                      │
├──────────────────────────────────────────────┤
│                                               │
│  ┌─────────────┐    ┌─────────────────┐     │
│  │  B+ 树索引   │    │  LSM 树索引     │     │
│  │ (默认)      │    │ (可选)         │     │
│  └─────────────┘    └─────────────────┘     │
│                                               │
│  ┌─────────────────────────────────────┐    │
│  │        MVCC 多版本并发控制             │    │
│  │  支持快照隔离，实现事务并发读写        │    │
│  └─────────────────────────────────────┘    │
│                                               │
│  ┌─────────────────────────────────────┐    │
│  │        文档级锁                       │    │
│  │  支持细粒度并发，替代 MMAPv1 的集合锁  │    │
│  └─────────────────────────────────────┘    │
│                                               │
│  ┌─────────────────────────────────────┐    │
│  │        内置压缩                       │    │
│  │  snappy / zstd / zlib              │    │
│  └─────────────────────────────────────┘    │
│                                               │
└──────────────────────────────────────────────┘
```

### WiredTiger vs MMAPv1

| 特性 | WiredTiger | MMAPv1（已废弃） |
|---|---|---|
| 锁粒度 | 文档级锁 | 集合级锁 |
| 并发能力 | 高 | 低 |
| 内存管理 | WT Cache | OS 文件缓存 |
| 压缩 | 支持 | 不支持 |
| Index | B+ 树 | B 树 |
| 适用场景 | 通用场景 | 大数据集 |

## B+ 树索引

### B+ 树结构

WiredTiger 默认使用 B+ 树作为索引结构。

```
WiredTiger B+ 树结构：

┌─────────────────────────────────────────────────┐
│                     根节点                        │
│              [key → page pointer]                │
└─────────────────────────────────────────────────┘
                        │
        ┌───────────────┼───────────────┐
        ▼               ▼               ▼
┌───────────────┐ ┌───────────────┐ ┌───────────────┐
│    内部节点     │ │    内部节点     │ │    内部节点     │
│[key1 → ptr1]   │ │[key4 → ptr4]   │ │[key7 → ptr7]  │
│[key2 → ptr2]   │ │[key5 → ptr5]   │ │               │
│[key3 → ptr3]   │ │[key6 → ptr6]   │ │               │
└───────┬───────┘ └───────┬───────┘ └───────┬───────┘
        │                 │                 │
        ▼                 ▼                 ▼
┌─────────────────────────────────────────────────┐
│                   叶子节点（链表连接）             │
│ ┌──────────┐ ┌──────────┐ ┌──────────┐        │
│ │ key: k1  │ │ key: k2  │ │ key: k3  │        │
│ │ val: ptr │ │ val: ptr │ │ val: ptr │        │
│ │ next ────┼─┼── next ──┼─┼── next  │        │
│ └──────────┘ └──────────┘ └──────────┘        │
└─────────────────────────────────────────────────┘
```

### B+ 树与 B 树的区别

| 特性 | B 树 | B+ 树 |
|---|---|---|
| 数据存储 | 所有节点存储数据 | 只有叶子节点存储数据 |
| 范围查询 | 需要中序遍历 | 叶子节点链表遍历 |
| 查询效率 | 取决于数据位置 | 稳定 O(log n) |
| 磁盘 IO | 不稳定 | 稳定 |

### 页面结构

```javascript
// WiredTiger 页面类型

// 1. 内部页面（Internal Page）
// 存储索引键和子页面指针
{
  page_type: "internal",
  items: [
    { key: 30, child: 0xFF35 },
    { key: 60, child: 0xFF42 }

}
```

## MVCC 多版本并发控制

### MVCC 原理

WiredTiger 使用 MVCC 实现并发控制，每个事务看到的是数据的快照版本。

```
MVCC 工作原理：

时间线：
T1 ────── T2 ────── T3 ────── T4 ────── T5 ────── T6
  │         │         │         │         │         │
  │    事务A开始      │         │         │         │
  │    读取: v1       │         │         │         │
  │         │         │         │         │         │
  │         │    事务B修改      │         │         │
  │         │    v1 → v2        │         │         │
  │         │         │         │         │         │
  │         │    事务A再次读取      │         │
  │         │    仍然读取 v1       │         │
  │         │    (快照隔离)        │         │
  │         │         │         │         │         │
  │         │         │    事务A提交    │         │
  │         │         │    事务B提交    │         │
  │         │         │         │    清理旧版本    │

版本链：
v1 ← v2 ← v3 ← ...  ← 最新版本
 ↑                  ↑
 事务A读取           事务B读取
 (快照版本)          (快照版本)
```

### 版本存储

```javascript
// WiredTiger 的文档版本存储

// 每个文档包含隐藏的版本信息
// 通过 _record 的时间戳字段判断可见性

// 文档结构（内部表示）
{
  _id: ObjectId("..."),
  name: "张三",
  age: 25,
  __wt: {
    start_ts: Timestamp(100, 1),   // 事务开始时间
    start_txn: 12345,              // 事务 ID
    stop_ts: Timestamp(200, 0)     // 停止时间（0 表示未删除）
  }
}
```

### 快照隔离

```javascript
// 事务内的快照隔离

// 事务A（readConcern: snapshot）
sessionA.startTransaction({ readConcern: { level: "snapshot" } });

// 事务B（在事务A之外）
db.collection.updateOne({ _id: 1 }, { $set: { value: "new" } });

// 事务A
const doc = await db.collection.findOne({ _id: 1 });
// 读取到的是事务A开始时的快照版本
// 不会看到事务B的修改
```

## Checkpoint 检查点

### Checkpoint 机制

Checkpoint 将内存中的数据持久化到磁盘，确保数据一致性。

```
Checkpoint 流程：

┌─────────────────────────────────────────────────┐
│                  WT Cache                        │
│  ┌─────────────────────────────────────────┐   │
│  │           Dirty Pages (脏页)              │   │
│  │  包含已修改但未写入磁盘的数据              │   │
│  └─────────────────────────────────────────┘   │
│                      │                           │
│                      │ Checkpoint                │
│                      ▼                           │
│  ┌─────────────────────────────────────────┐   │
│  │            数据文件 (.wt)                 │   │
│  │  Checkpoint 保证磁盘数据一致性             │   │
│  └─────────────────────────────────────────┘   │
│                                                   │
│  ┌─────────────────────────────────────────┐   │
│  │            Journal (WAL)                 │   │
│  │  记录所有写操作，支持崩溃恢复              │   │
│  └─────────────────────────────────────────┘   │
└─────────────────────────────────────────────────┘
```

### Checkpoint 配置

```javascript
// mongod.conf
storage:
  wiredTiger:
    engineConfig:
      checkpoint:
        wait: 60          # Checkpoint 间隔（秒）
        logSize: 100      # Journal 文件大小阈值（MB）
```

### 恢复过程

```javascript
// 崩溃恢复流程：

// 1. MongoDB 启动
// 2. 读取最新的 Checkpoint
// 3. 从 Checkpoint 恢复数据文件
// 4. 重放 Journal 中的未Checkpoint操作
// 5. 完成恢复

// 查看检查点信息
db.adminCommand({ replSetGetStatus: 1 })
```

## 文档级锁

### 锁的实现

WiredTiger 使用乐观锁实现文档级并发控制。

```javascript
// 文档锁示意

// 事务A 锁定文档1
┌─────────────────────────────────────────────────┐
│ 文档1: X-Lock (排他锁)                           │
│ 事务A: 正在修改文档1                              │
│ 事务B: 等待文档1释放锁...                         │
└─────────────────────────────────────────────────┘

// 事务A 提交后
┌─────────────────────────────────────────────────┐
│ 文档1: Unlocked                                  │
│ 事务B: 获取锁，继续操作                            │
└─────────────────────────────────────────────────┘
```

### 锁等待配置

```javascript
// 配置锁超时
// mongod.conf
storage:
  wiredTiger:
    engineConfig:
      lockTimeoutMS: 5000  # 锁等待超时（毫秒）

// 查看锁状态
db.serverStatus().locks
```

### WT Cache 结构


## 面试实战

**面试官问**：「WiredTiger 的 MVCC 是怎么实现的？」

**深入回答**：

> WiredTiger MVCC 的实现机制：
>
> 1. **版本时间戳**：每个文档版本都有 `start_ts` 和 `stop_ts`
>
> 2. **快照隔离**：事务开始时获取一个全局时间戳，读取该时间戳之前的数据版本
>
> 3. **写入时复制**：修改数据时创建新版本，旧版本保留
>
> 4. **垃圾回收**：后台线程定期清理不再需要的旧版本
>
> 5. **可见性判断**：根据事务时间戳和版本时间戳判断可见性

**面试官问**：「WiredTiger 和 InnoDB 有什么区别？」

**深入回答**：

> 主要区别：
>
> 1. **锁粒度**：WiredTiger 是文档级锁，InnoDB 是行级锁
>
> 2. **索引结构**：WiredTiger 默认 B+ 树（支持 LSM），InnoDB 只有 B+ 树
>
> 3. **内存管理**：WiredTiger 有独立的 Cache 管理，InnoDB 使用 Buffer Pool
>
> 4. **压缩**：WiredTiger 内置压缩，InnoDB 需要第三方方案
>
> 5. **适用场景**：WiredTiger 更适合大文档、高并发场景

## 总结

WiredTiger 核心要点：

- **B+ 树**：默认索引结构，叶子节点链表连接
- **MVCC**：多版本并发控制，快照隔离
- **Checkpoint**：将内存数据持久化到磁盘
- **Journal**：WAL 日志，保证崩溃恢复
- **文档级锁**：乐观并发控制
- **Cache**：默认占 RAM 50%，管理内存数据

理解 WiredTiger 原理是 MongoDB 性能调优的基础。
