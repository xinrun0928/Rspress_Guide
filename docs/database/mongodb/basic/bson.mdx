# BSON 数据类型

MongoDB 存储的数据格式叫 **BSON**——Binary JSON 的缩写。顾名思义，BSON 是 JSON 的二进制表示，但它比 JSON 支持更多数据类型。

理解 BSON 数据类型，是写好 MongoDB 查询和设计文档模型的基础。

## BSON 与 JSON 的区别

| 特性 | JSON | BSON |
|---|---|---|
| 可读性 | 人类可读 | 二进制，需要解析 |
| 大小 | 较大 | 更紧凑 |
| 数据类型 | 有限（string、number、boolean、null、array、object） | 丰富（多了日期、二进制、ObjectId 等） |
| 性能 | 解析慢 | 解析快 |

## 数据类型一览

### 基础类型

```javascript
// String（字符串）
{ name: "张三" }

// Number（数字）
// BSON 支持多种数字类型
{ age: 25 }                    // 默认是 float64（64位浮点）
{ age: NumberInt(25) }         // 32位整数
{ big_number: NumberLong("123456789") }  // 64位整数
{ price: NumberDecimal("3.14159") }      // 高精度Decimal

// Boolean（布尔）
{ isActive: true, isDeleted: false }

// Null（空值）
{ deletedAt: null }

// Array（数组）
{ tags: ["Java", "MongoDB", "Redis"] }

// Object（嵌套文档）
{ profile: { city: "北京", level: 10 } }
```

:::tip 数字类型选择

- 大多数场景直接用默认数字即可
- 需要精确金额计算时用 `NumberDecimal`
- 整数运算用 `NumberLong` 避免精度问题
- 旧代码中可能看到 `NumberInteger`（别名 `int`）
:::

### 日期和时间

```javascript
// 日期类型
{ createdAt: ISODate("2024-01-15T10:30:00Z") }

// 时间戳（毫秒）
{ timestamp: new Timestamp() }

// 获取当前时间
{ now: new Date() }
{ now: ISODate() }

// Date 的内部表示
// 存储为 64 位整数：距离 Unix 纪元的毫秒数
// 范围：约公元 2900 亿年到负数十亿年
```

:::warning JavaScript Date 的坑

JavaScript 的 `Date` 对象和 MongoDB 的 `Date` 类型不完全一致：

```javascript
// 在 mongo shell 中
new Date()       // 返回 BSON Date（带时区信息）
ISODate()        // 同上，推荐使用

// 创建的日期默认是 UTC
// 在中国时区显示会少 8 小时
```
:::

### ObjectId

MongoDB 的默认主键类型，全局唯一：

```javascript
// ObjectId 结构
{ _id: ObjectId("507f1f77bcf86cd799439011") }

// ObjectId 的 12 字节构成
// [0-3]  时间戳（4字节）
// [4-6]  机器标识（3字节）
// [7-8]  进程ID（2字节）
// [9-11] 计数器（3字节）

// ObjectId 的方法
ObjectId().str                    // 转字符串
ObjectId().getTimestamp()         // 获取时间戳
ObjectId().toString()
```

```javascript
// 查询中比较 ObjectId
db.users.findOne({ _id: ObjectId("507f1f77bcf86cd799439011") })

// 字符串和 ObjectId 不能直接比较
// ❌ 错误
db.users.findOne({ _id: "507f1f77bcf86cd799439011" })

// ✅ 正确
db.users.findOne({ _id: ObjectId("507f1f77bcf86cd799439011") })
```

:::tip ObjectId 的时间戳

可以直接从 ObjectId 提取创建时间：

```javascript
// 提取 ObjectId 对应的日期
db.users.findOne({ _id: ObjectId("507f1f77bcf86cd799439011") })._id.getTimestamp()
// ISODate("2024-01-15T10:30:00Z")
```
:::

### 二进制数据

```javascript
// Binary（二进制）
// 第一个参数是 Buffer/二进制数据，第二个是子类型
{ file: BinData(0, "base64编码的二进制") }

// 常用子类型
// 0: 通用二进制
// 1: 函数
// 2: 旧版 UUID
// 3: UUID
// 4: MD5

// Node.js 中使用
db.files.insertOne({
  name: "avatar.png",
  data: Buffer.from("二进制内容"),
  type: 0
})
```

### 其他类型

```javascript
// 正则表达式
{ email: /^[a-z]+@[a-z]+\.[a-z]+$/i }

// JavaScript 代码（不推荐，性能差）
{ $where: "this.age > 25" }

// DBRef（跨集合引用，类似外键）
{ ref: DBRef("users", ObjectId("...")) }

// Symbol（符号，用于特定语言）
{ symbol: Symbol("custom") }

// MinKey / MaxKey（比较极值）
{ minValue: MinKey }   // 比任何值都小
{ maxValue: MaxKey }   // 比任何值都大
```

## 类型查询

### 按类型查询

```javascript
// 查询某字段为特定类型
// 类型对照表：2=String, 16=Int, 18=Long, 1=Double, 8=Boolean, 10=Null, ...
db.users.find({ age: { $type: "int" } })

// 查询某字段不存在或为 null
db.users.find({ nickname: { $in: [null], $exists: true } })

// 查询某字段不存在
db.users.find({ nickname: { $exists: false } })

// 查询字段类型
db.users.find({ age: { $type: "string" } })  // age 是字符串的文档
```

## 数据类型的比较规则

BSON 数据有严格的比较顺序：

```
MinKey < Null < Numbers (Int, Long, Double, Decimal) < Strings
< Objects < Arrays < BinData < ObjectId < Boolean < Date
< Timestamp < Regular Expression < MaxKey
```

```javascript
// 数字内部比较：Integer < Long < Double < Decimal
// 字符串比较：按 Unicode 码点（字典序）

// 查询 age < "30" 是什么意思？
// 字符串 "30" > "张三" 因为 "3" 的码点小于中文字符
db.users.find({ name: { $lt: "30" } })  // 这会匹配什么？
```

数组中混合不同类型时，比较按上述规则进行：

## 面试实战

**面试官问**：「ObjectId 是怎么保证全局唯一的？」

**深入回答**：

> ObjectId 的 12 字节由四部分组成：
>
> 1. **时间戳（4字节）**：精确到秒，保证时间上有序
> 2. **机器标识（3字节）**：机器名/IP 的哈希，保证不同机器生成的 ID 不同
> 3. **进程ID（2字节）**：保证同一机器不同进程生成的 ID 不同
> 4. **计数器（3字节）**：每秒内递增，保证同一秒同一进程生成多个 ID 也不同
>
> 这四重保障加上 3 字节的容量（1600万/秒），保证了 ObjectId 的全局唯一性。

**面试官问**：「为什么要用 ObjectId 而不是自增整数作为主键？」

**深入回答**：

> 1. **分布式友好**：自增 ID 需要中心节点分配，容易成为瓶颈；ObjectId 可以本地生成
> 2. **写入分布**：ObjectId 包含时间戳，新文档的 ID 自然更大，在分片环境下写入更分散
> 3. **信息量**：从 ObjectId 可以直接提取创建时间
> 4. **安全**：UUID/ObjectId 不暴露业务量级
>
> 缺点：ObjectId 比整数更大（12字节 vs 4/8字节），且无序（时间戳在高位但不是严格单调）。

## 总结

BSON 数据类型要点：

- **基础类型**：String、Number（多种精度）、Boolean、Null、Array、Object
- **特殊类型**：Date、ObjectId、Binary、Regex、Timestamp
- **类型安全**：MongoDB 不强制类型转换，插入什么类型就存储什么类型
- **类型比较**：有严格的排序规则，混合类型查询时需注意
- **ObjectId**：MongoDB 默认主键，包含时间戳，保证全局唯一
