# MongoDB 与 SQL 概念对比

从关系型数据库迁移到 MongoDB，最难的不是学语法，而是**换思维**。

SQL 是「表格思维」——一切皆表，关联靠 JOIN。MongoDB 是「文档思维」——数据天然嵌入，关联靠引用。两种模型各有优劣，关键是用对场景。

## 概念对照表

| SQL 概念 | MongoDB 概念 | 说明 |
|---|---|---|
| Database | Database | 数据库 |
| Table | Collection | 表 vs 集合 |
| Row | Document | 行 vs 文档 |
| Column | Field | 列 vs 字段 |
| Primary Key | `_id` | 主键默认是 `_id` |
| Index | Index | 索引 |
| JOIN | `$lookup` / 嵌入 | 关联查询 |
| Transaction | Transaction | 事务（4.0+） |
| Foreign Key | DBRef（少用） | 外键引用 |

## 数据库操作对比

### 创建

```sql
-- SQL
CREATE DATABASE myapp;
USE myapp;
CREATE TABLE users (id INT PRIMARY KEY, name VARCHAR(50), age INT);
```

```javascript
// MongoDB
// 数据库：自动创建，不需要显式声明
use myapp

// 集合：插入文档时自动创建
db.users.insertOne({
  name: "张三",
  age: 25
})
```

:::tip MongoDB 的无模式特性

MongoDB 的集合不需要预先定义字段。文档可以有完全不同的结构：

```javascript
// 同一集合中可以插入不同结构的文档
db.users.insertOne({ name: "张三", age: 25 })
db.users.insertOne({ name: "李四", email: "li@example.com", phone: "13800138000" })
db.users.insertOne({ profile: { city: "北京", level: 5 } })
```

这就是「无模式」（Schema-less），灵活但也要谨慎使用。
:::

## CRUD 操作对比

### 插入数据

```sql
-- SQL
INSERT INTO users (name, age, email) VALUES ('张三', 25, 'zhang@example.com');
```

```javascript
// MongoDB
db.users.insertOne({
  name: "张三",
  age: 25,
  email: "zhang@example.com"
})
```

### 查询数据

```sql
-- SQL
-- 基本查询
SELECT * FROM users;

-- 条件查询
SELECT name, age FROM users WHERE age > 25;

-- AND/OR
SELECT * FROM users WHERE age > 25 AND city = '北京';

-- 排序分页
SELECT * FROM users ORDER BY age DESC LIMIT 10 OFFSET 20;
```

```javascript
// MongoDB
// 基本查询
db.users.find()

// 投影（选择字段）
db.users.find(
  { age: { $gt: 25 } },
  { name: 1, age: 1, _id: 0 }
)

// AND/OR
db.users.find({
  $and: [
    { age: { $gt: 25 } },
    { city: "北京" }
  ]
})

// 排序分页
db.users.find()
  .sort({ age: -1 })
  .limit(10)
  .skip(20)
```

### 更新数据

```sql
-- SQL
UPDATE users SET age = 26 WHERE name = '张三';

-- 递增
UPDATE users SET views = views + 1 WHERE _id = 1;
```

```javascript
// MongoDB
db.users.updateOne(
  { name: "张三" },
  { $set: { age: 26 } }
)

// 原子递增
db.users.updateOne(
  { _id: 1 },
  { $inc: { views: 1 } }
)
```

### 删除数据

```sql
-- SQL
DELETE FROM users WHERE age < 18;
```

```javascript
// MongoDB
db.users.deleteMany({ age: { $lt: 18 } })
```

## 聚合操作对比

### 计数

```sql
-- SQL
SELECT COUNT(*) FROM users WHERE age > 25;
SELECT COUNT(DISTINCT city) FROM users;
```

```javascript
// MongoDB
db.users.countDocuments({ age: { $gt: 25 } })

// 去重计数
db.users.distinct("city").length
// 或聚合管道
db.users.aggregate([
  { $group: { _id: "$city" } },
  { $count: "total" }
])
```

### 分组统计

```sql
-- SQL
SELECT city, COUNT(*) as cnt, AVG(age) as avg_age
FROM users
GROUP BY city
HAVING COUNT(*) > 10;
```

```javascript
// MongoDB
db.users.aggregate([
  { $group: {
      _id: "$city",
      count: { $sum: 1 },
      avg_age: { $avg: "$age" }
  }},
  { $match: { count: { $gt: 10 } }}
])
```

### 表关联

```sql
-- SQL
SELECT u.name, o.order_id, o.total
FROM users u
JOIN orders o ON u.id = o.user_id
WHERE u.city = '北京';
```

```javascript
// MongoDB $lookup（类似于 LEFT JOIN）
db.users.aggregate([
  { $match: { city: "北京" } },
  { $lookup: {
      from: "orders",           // 关联的集合
      localField: "_id",        // 本集合的字段
      foreignField: "user_id",  // 关联集合的字段
      as: "orders"              // 输出字段名
  }}
])
```

:::tip $lookup vs JOIN 的区别

`$lookup` 是**管道阶段**，可以串联多个 `lookup`，但它只是查询，不支持嵌套条件的预过滤。复杂的 JOIN 场景还是需要应用层处理或 MapReduce。
:::

## 数据建模对比

### SQL：规范化设计

```sql
-- 用户表
CREATE TABLE users (
  id INT PRIMARY KEY,
  name VARCHAR(50),
  city_id INT,
  FOREIGN KEY (city_id) REFERENCES cities(id)
);

-- 城市表
CREATE TABLE cities (
  id INT PRIMARY KEY,
  name VARCHAR(50)
);

-- 订单表
CREATE TABLE orders (
  id INT PRIMARY KEY,
  user_id INT,
  total DECIMAL(10,2),
  created_at TIMESTAMP
);
```

### MongoDB：嵌入 vs 引用

```javascript
// 方案一：嵌入（适合一对少/有限数据）
// 一个用户文档中嵌入地址信息
{
  _id: ObjectId("..."),
  name: "张三",
  address: {
    city: "北京",
    district: "朝阳区",
    street: "建国路"
  }
}

// 方案二：引用（适合多/增长数据）
// 订单独立成集合，通过 user_id 引用
// 用户文档
{ _id: ObjectId("..."), name: "张三" }

// 订单文档
{ _id: ObjectId("..."), user_id: ObjectId("..."), total: 100.00 }
```

### 嵌入 vs 引用的选择原则

| 场景 | 推荐方式 | 原因 |
|---|---|---|
| 一对一，字段不常更新 | 嵌入 | 一次查询获取全部数据 |
| 一对多，数据量有限 | 嵌入 | 避免 N+1 查询 |
| 一对多，数据量无限 | 引用 | 避免文档过大 |
| 多对多 | 引用 | 数据分散，需要 JOIN |
| 需要独立查询/更新 | 引用 | 嵌入无法单独查询子文档 |

```javascript
// ❌ 错误：嵌入无限增长的数据
{ user: "张三", orders: [...无限增长...] }

// ✅ 正确：引用独立集合
{ user: "张三" }  // users 集合
{ user_id: "...", order: "..." }  // orders 集合
```

## 索引对比

```sql
-- SQL
-- 单字段索引
CREATE INDEX idx_age ON users(age);

-- 复合索引
CREATE INDEX idx_age_city ON users(age, city);

-- 唯一索引
CREATE UNIQUE INDEX idx_email ON users(email);
```

```javascript
// MongoDB
// 单字段索引
db.users.createIndex({ age: 1 })

// 复合索引
db.users.createIndex({ age: 1, city: 1 })

// 唯一索引
db.users.createIndex({ email: 1 }, { unique: true })

// 其他索引类型
db.users.createIndex({ name: "text" })           // 文本索引
db.users.createIndex({ location: "2dsphere" })   // 地理空间索引
```

## 面试实战

**面试官问**：「MongoDB 为什么不需要预定义表结构？」

**深入回答**：

> MongoDB 是无模式（Schema-less）数据库，集合不强制文档结构相同。这带来了灵活性：
>
> - 快速迭代：字段可以随时增删，不用 ALTER TABLE
> - 应对多样化数据：同一集合可存不同结构
>
> 但无模式不等于无设计。无节制的无模式会导致：
> - 查询逻辑复杂化
> - 数据质量难以保证
> - 索引设计困难
>
> 实践中，建议在应用层定义 Schema 约束，用 MongoDB Schema Validation 或应用代码校验。

**面试官问**：「什么时候用 MongoDB 而不是 MySQL？」

**深入回答**：

> 选择 MongoDB 的场景：
> - 数据结构灵活多变
> - 需要水平扩展
> - 一对多/嵌套数据，JOIN 需求少
> - 高写入吞吐量
> - 需要地理空间查询
>
> 选择 MySQL 的场景：
> - 强事务需求（ACID）
> - 高度规范化，数据高度关联
> - 复杂的 JOIN 查询
> - 需要强一致性保证

## 总结

MongoDB 与 SQL 的核心差异：

- **模式**：MongoDB 无模式，SQL 强类型
- **关联**：MongoDB 用嵌入/引用，SQL 用 JOIN
- **事务**：SQL 原生 ACID，MongoDB 4.0+ 支持多文档事务
- **扩展**：MongoDB 天然水平扩展，SQL 扩展有限
- **选择**：没有绝对优劣，只有场景匹配
