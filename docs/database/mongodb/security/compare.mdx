# 数据库选型：MongoDB vs MySQL vs Redis

「我们该用什么数据库？」

这个问题几乎每个技术团队都会遇到。MongoDB、MySQL、Redis——三个都是业界主流的数据库，但它们的定位完全不同。选错了，轻则性能低下，重则项目失败。

今天我们不聊"哪个最好"，而是聊"什么时候该用哪个"。

## 三种数据库的本质定位

在选型之前，先搞清楚三种数据库的基因：

| 数据库 | 类型 | 设计哲学 | 最适合场景 |
|---|---|---|---|
| MySQL | 关系型数据库 | ACID + 强一致性 | 结构化数据、复杂查询、事务场景 |
| MongoDB | 文档型数据库 | 灵活 Schema + 水平扩展 | 灵活数据结构、半结构化数据 |
| Redis | 内存数据库 | 极致读写性能 | 缓存、计数器、实时数据 |

这三种数据库并不是非此即彼的关系。很多成熟架构会同时使用三者：**Redis 做缓存，MySQL 做持久化，MangoDB 做灵活扩展。**

## 场景对比：事务需求

### MySQL：强事务场景的首选

如果你的业务**必须保证 ACID**，MySQL 是最稳妥的选择：

```sql
-- 转账场景：必须原子性完成
BEGIN;
UPDATE accounts SET balance = balance - 1000 WHERE user_id = 1;
UPDATE accounts SET balance = balance + 1000 WHERE user_id = 2;
COMMIT;

-- 任何一步失败，全部回滚
```

**适合场景**：
- 金融系统：账户余额、交易记录
- 电商核心链路：订单、库存扣减
- ERP/CRM：业务数据需要严格一致性

### MongoDB：单文档事务，跨集合需要权衡

MongoDB 4.0+ 支持多文档事务，4.2+ 支持分片集群事务：

```javascript
// MongoDB 事务示例
session = db.getMongo().startSession();
session.startTransaction();

try {
    session.getDatabase("ecommerce").orders.insertOne({
        userId: "user123",
        items: [{ productId: "prod001", quantity: 2 }]
    });
    session.getDatabase("ecommerce").inventory.updateOne(
        { productId: "prod001" },
        { $inc: { stock: -2 } }
    );
    session.commitTransaction();
} catch (e) {
    session.abortTransaction();
}
```

**但需要注意**：MongoDB 的事务性能开销比 MySQL 大，在高频写入场景下可能成为瓶颈。

**不适合场景**：
- 强一致性要求的金融交易
- 跨多个集合的复杂关联查询
- 需要存储过程/触发器的业务逻辑

### Redis：不支持传统事务，但有替代方案

Redis 的事务是**乐观锁模式**（MULTI/EXEC），不支持回滚：

```redis
MULTI
INCR order:count
HSET order:1001 user_id "user123"
EXEC
-- 如果 HSET 失败，INCR 不会回滚
```

Redis 通过 **Lua 脚本** 和 **Watch** 实现原子性操作：

```redis
-- Watch 实现乐观锁
WATCH order:count
count = GET order:count
MULTI
SET order:count 999
EXEC  -- 如果 watch 的 key 被其他客户端修改，EXEC 返回 nil
```

**适合场景**：
- 乐观锁控制的计数器
- Lua 脚本实现的原子业务逻辑

## 场景对比：灵活 Schema

### MongoDB：Schema-less 的真正优势

MongoDB 的文档模型让数据结构变化变得简单：

```javascript
// 用户文档 v1
{
    _id: "user001",
    name: "张三",
    email: "zhang@example.com"
}

// 用户文档 v2：增加地址字段，无需修改表结构
{
    _id: "user001",
    name: "张三",
    email: "zhang@example.com",
    addresses: [
        { city: "北京", district: "朝阳区", street: "建国路" }
    ]
}

// 用户文档 v3：不同用户可以有完全不同的字段
{
    _id: "user002",
    name: "李四",
    profile: { hobby: "摄影", level: "professional" }
    // 没有 email 字段，MongoDB 完全接受
}
```

**适合场景**：
- 快速迭代的产品：字段频繁变化
- 日志/内容管理：不同文档结构差异大
- 多租户系统：每个租户有自己的字段扩展

### MySQL：结构化数据的王者

MySQL 的强类型让数据质量更有保障：

```sql
-- 定义严格的表结构
CREATE TABLE users (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(50) NOT NULL,
    email VARCHAR(100) NOT NULL UNIQUE,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 字段类型强制约束，不允许随意扩展
INSERT INTO users (name, email) VALUES ('张三', 'zhang@example.com');
-- 后续想加 "addresses" 字段？必须 ALTER TABLE
```

**适合场景**：
- 数据结构稳定的业务系统
- 需要强类型校验的数据
- 报表和数据分析（复杂 SQL 查询）

### Redis：KV 存储，不关心结构

Redis 对数据结构完全"无感"：

```redis
SET user:001 "{\"name\":\"张三\",\"age\":30}"
SET user:002 "just a string"
-- Redis 不关心值是什么格式
```

**适合场景**：
- Session 存储
- 临时数据/中间结果
- 配置信息

## 场景对比：高并发缓存

### Redis：缓存场景的不二之选

Redis 的内存存储带来了极致的性能：

```redis
# 读取性能对比（理论值）
Redis: ~100,000+ QPS
MongoDB: ~10,000-50,000 QPS（取决于数据量和索引）
MySQL: ~5,000-20,000 QPS（取决于查询复杂度）

# Redis 缓存模式
GET product:12345          # 缓存命中
SETEX product:12345 3600 '{"name":"商品","price":99}'  # 1小时过期
```

**Redis 缓存的典型模式**：

```redis
# 先查缓存
result = GET user:12345
if result == nil:
    # 缓存未命中，查数据库
    result = MySQL.query("SELECT * FROM users WHERE id = 12345")
    # 写入缓存
    SETEX user:12345 1800 result

return result
```

**适合场景**：
- 热点数据缓存（商品详情、用户信息）
- 计数器（点赞数、浏览量）
- Session 管理
- 分布式锁

### MongoDB：缓存层也能胜任

MongoDB 可以作为中等规模的缓存层：

```javascript
// MongoDB 缓存模式：适合需要复杂查询的缓存
db.product_cache.createIndex({ "productId": 1 }, { expireAfterSeconds: 3600 });

db.product_cache.insertOne({
    productId: "prod001",
    data: { name: "商品", price: 99 },
    createdAt: new Date()
});
```

**但不适合**：
- 极高并发场景（Redis 更擅长）
- 简单的 KV 存取

### MySQL：别忘了 Query Cache（历史教训）

MySQL 曾经有 Query Cache，但在 MySQL 8.0 被移除，原因：

- 并发写入时全局锁竞争
- 缓存失效频繁
- 内存管理不够高效

**结论**：MySQL 不适合做缓存老老实实做持久化存储。

## 选型决策树

```
你的业务需要强事务吗？
├── 是 → MySQL
└── 否 → 数据结构是否经常变化？
    ├── 是 → MongoDB
    └── 否 → 需要缓存层吗？
        ├── 是 → Redis + MySQL
        └── 否 → MySQL
```

## 实战架构：三层数据库架构

很多大型互联网公司采用三层架构：

```
┌─────────────────────────────────────────┐
│            应用层                        │
└─────────────────┬───────────────────────┘
                  │
        ┌─────────┴─────────┐
        │                   │
   ┌────▼────┐        ┌─────▼─────┐
   │  Redis  │        │  MongoDB  │
   │  缓存层  │        │  灵活数据  │
   └────┬────┘        └─────┬─────┘
        │                   │
        └─────────┬─────────┘
                  │
           ┌──────▼──────┐
           │    MySQL    │
           │   持久化层   │
           └─────────────┘
```

| 层级 | 数据库 | 存储内容 |
|---|---|---|
| 缓存层 | Redis | Session、热点数据、计数器 |
| 灵活数据层 | MongoDB | 用户生成内容、日志、变化频繁的数据 |
| 持久化层 | MySQL | 核心业务数据、交易数据、报表 |

## 总结

选型没有银弹，关键是**匹配业务场景**：

- **强事务 + 结构化数据**：MySQL
- **灵活 Schema + 快速迭代**：MongoDB
- **高并发缓存 + 简单存取**：Redis
- **复杂系统**：三者组合使用

下次选型时，先问自己三个问题：数据要不要事务？结构会不会频繁变？访问并发有多高？答案自然浮现。
