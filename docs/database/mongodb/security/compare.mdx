# MongoDB vs MySQL vs Redis 选型对比

「MongoDB、MySQL、Redis 各有什么特点？什么时候用哪个？」

这是技术选型时最常见的问题。三种数据库各有特点，适合不同的业务场景。

## 核心对比

### 基本特性对比

| 特性 | MongoDB | MySQL | Redis |
|---|---|---|---|
| 类型 | 文档数据库 | 关系型数据库 | 内存数据库 |
| 数据模型 | 文档/JSON | 表（行列） | Key-Value |
| 存储格式 | BSON | 行 | 多种数据结构 |
| 扩展方式 | 水平扩展（分片） | 垂直扩展/读写分离 | 水平扩展（集群） |
| 事务支持 | 单文档原子性 + 多文档事务 | 完整 ACID 事务 | Lua 脚本原子操作 |
| 索引 | B+ 树、哈希、文本等 | B+ 树 | 哈希、跳表 |

### 性能对比

| 指标 | MongoDB | MySQL | Redis |
|---|---|---|---|
| 写入性能 | 高 | 中 | 极高 |
| 读取性能 | 高 | 高 | 极高 |
| 复杂查询 | 支持聚合管道 | SQL 强大 | 有限 |
| 内存需求 | 中等 | 中等 | 高（内存数据库） |
| 磁盘占用 | 较大（压缩） | 中等 | 较小 |

## 数据模型对比

### MySQL：关系型模型

```sql
-- 规范化设计
CREATE TABLE users (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(100),
    email VARCHAR(100)
);

CREATE TABLE orders (
    id INT PRIMARY KEY AUTO_INCREMENT,
    user_id INT,
    total DECIMAL(10,2),
    FOREIGN KEY (user_id) REFERENCES users(id)
);

CREATE TABLE order_items (
    id INT PRIMARY KEY AUTO_INCREMENT,
    order_id INT,
    product_id INT,
    quantity INT,
    FOREIGN KEY (order_id) REFERENCES orders(id)
);

-- 查询需要 JOIN
SELECT u.name, SUM(oi.quantity * p.price) as total
FROM users u
JOIN orders o ON u.id = o.user_id
JOIN order_items oi ON o.id = oi.order_id
JOIN products p ON oi.product_id = p.id
WHERE u.id = 1
GROUP BY u.id, u.name;
```

### MongoDB：文档模型

```javascript
// 嵌入式设计
{
  _id: ObjectId("..."),
  name: "张三",
  email: "zhangsan@example.com",
  orders: [
    {
      order_id: "A001",
      total: 500,
      items: [
        { product_id: "P001", product_name: "商品A", quantity: 2, price: 100 },
        { product_id: "P002", product_name: "商品B", quantity: 3, price: 100 }
      ]
    }
  ]
}

// 查询简单
db.users.findOne(
  { _id: ObjectId("...") },
  { "orders.items": 1 }
)
```

### Redis：键值模型

```redis
# 字符串
SET user:1:name "张三"
GET user:1:name

# 哈希
HSET user:1 name "张三" email "zhangsan@example.com"
HGETALL user:1

# 列表
LPUSH user:1:orders "order:A001"
LRANGE user:1:orders 0 -1

# 集合
SADD user:1:tags "Java" "MongoDB" "Redis"
SMEMBERS user:1:tags
```

## 适用场景

### MongoDB 适用场景

```javascript
// 1. 内容管理 / CMS
// 文档结构灵活，适合多变的字段
{
  title: "文章标题",
  content: "文章内容",
  author: "作者",
  tags: ["技术", "MongoDB"],
  metadata: {
    views: 1000,
    likes: 50,
    comments: 10
  }
}

// 2. 用户画像
// 字段多，结构灵活
{
  user_id: "U001",
  preferences: {
    language: "zh-CN",
    theme: "dark",
    notifications: true
  },
  behavior: [
    { action: "login", timestamp: ISODate(...) },
    { action: "purchase", product: "P001" }
  ]
}

// 3. IoT 数据
// 写入量大，需要水平扩展
{
  device_id: "D001",
  timestamp: ISODate(...),
  temperature: 25.5,
  humidity: 60,
  location: { lat: 39.9, lng: 116.4 }
}

// 4. 实时分析
// 聚合管道支持复杂分析
db.events.aggregate([
  { $match: { type: "click" } },
  { $group: { _id: "$page", count: { $sum: 1 } } }
])
```

### MySQL 适用场景

```sql
-- 1. 金融系统
-- 需要强一致性，完整事务支持
BEGIN TRANSACTION;
  UPDATE accounts SET balance = balance - 100 WHERE id = 1;
  UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;

-- 2. ERP / CRM 系统
-- 复杂业务逻辑，强关联数据
SELECT o.*, c.name, e.name as sales_person
FROM orders o
JOIN customers c ON o.customer_id = c.id
JOIN employees e ON o.sales_id = e.id
WHERE c.region = '华北';

-- 3. 报表系统
-- 复杂 SQL 查询，多表关联
SELECT
  DATE_FORMAT(o.created_at, '%Y-%m') as month,
  c.category,
  SUM(o.amount) as total
FROM orders o
JOIN products p ON o.product_id = p.id
JOIN categories c ON p.category_id = c.id
GROUP BY month, c.category
ORDER BY total DESC;

-- 4. 订单系统
-- 需要严格的外键约束
CREATE TABLE order_items (
  order_id INT NOT NULL,
  product_id INT NOT NULL,
  quantity INT NOT NULL,
  PRIMARY KEY (order_id, product_id),
  FOREIGN KEY (order_id) REFERENCES orders(id),
  FOREIGN KEY (product_id) REFERENCES products(id)
);
```

### Redis 适用场景

```redis
# 1. 缓存
SET user:session:12345 "token_value" EX 3600
GET user:session:12345

# 2. 会话存储
HSET session:abc123 user_id 1 name "张三" login_at "2024-01-01"
EXPIRE session:abc123 86400

# 3. 排行榜
ZADD leaderboard 1000 "player1"
ZADD leaderboard 2000 "player2"
ZADD leaderboard 1500 "player3"
ZRANGE leaderboard 0 9 WITHSCORES

# 4. 分布式锁
SET lock:order:12345 "locked" NX EX 30
# Lua 脚本释放锁
EVAL "if redis.call('get', KEYS[1]) == ARGV[1] then return redis.call('del', KEYS[1]) else return 0 end" 1 lock:order:12345 "locked"

# 5. 消息队列
LPUSH queue:tasks "task:1"
BRPOP queue:tasks 0

# 6. 实时计数器
INCR page:views:20240101
INCR article:123:likes
```

## 架构设计对比

### 存储模式

```
MySQL：规范化存储
┌─────────┐     ┌─────────┐     ┌─────────┐
│  Users   │────▶│ Orders  │────▶│ Items   │
│ (用户表)  │     │ (订单表) │     │ (订单项) │
└─────────┘     └─────────┘     └─────────┘

MongoDB：反规范化/嵌入式
┌─────────────────────────────────┐
│          Order 文档               │
│  { user: {...}, items: [...] }  │
└─────────────────────────────────┘

Redis：缓存 + 主存储
┌──────────┐      ┌──────────┐
│  Redis    │ ←─── │ MongoDB  │
│  (缓存)   │      │  (主存储) │
└──────────┘      └──────────┘
```

### 分片策略

```javascript
// MongoDB 分片键选择
sh.shardCollection("app.orders", { customer_id: "hashed" })

// MySQL 分库分表（中间件）
// MyCAT / ShardingSphere

// Redis 集群槽分配
// 16384 个槽自动分配到 6 个节点
// cluster nodes
// 每个键通过 CRC16(key) % 16384 确定槽
```

## 选型决策树

```
选型决策树：

1. 需要强一致性事务？
   │
   ├─ 是 → MySQL
   │
   └─ 否 → 继续判断
       │
       2. 数据结构复杂度？
           │
           ├─ 复杂查询/多表关联 → MySQL
           │
           └─ 简单/文档结构 → MongoDB
               │
               3. 主要用途？
                   │
                   ├─ 缓存/会话/计数器 → Redis
                   │
                   ├─ 主数据存储 → MongoDB
                   │
                   └─ 两者结合 → Redis + MongoDB
```

## 面试实战

**面试官问**：「MongoDB 和 MySQL 的区别是什么？」

**深入回答**：

> 核心区别：
>
> 1. **数据模型**：MySQL 是关系型，表之间通过外键关联；MongoDB 是文档型，数据可以嵌入
>
> 2. **事务**：MySQL 有完整的 ACID 事务；MongoDB 单文档原子性 + 多文档事务（4.0+）
>
> 3. **查询能力**：MySQL SQL 强大，支持复杂 JOIN；MongoDB 聚合管道强大，但 JOIN 有限
>
> 4. **扩展性**：MySQL 垂直扩展为主；MongoDB 原生支持水平扩展（分片）

**面试官问**：「什么时候用 Redis，什么时候用 MongoDB？」

**深入回答**：

> 选择原则：
>
> 1. **Redis**：缓存、会话、计数器、分布式锁、排行榜、消息队列
>
> 2. **MongoDB**：主数据存储、灵活结构数据、大数据量、需要水平扩展
>
> 3. **两者结合**：Redis 做缓存加速，MongoDB 做主存储

**面试官问**：「MongoDB 能完全替代 MySQL 吗？」

**深入回答**：

> 不能完全替代，原因：
>
> 1. **强一致性场景**：金融、订单核心等需要完整事务的场景
>
> 2. **复杂关联查询**：多表 JOIN、复杂报表
>
> 3. **已有系统**：迁移成本高
>
> 4. **团队熟悉度**：技术栈需要考虑团队能力

## 总结

选型要点：

- **MySQL**：强一致性、复杂查询、关联数据
- **MongoDB**：灵活结构、大数据量、水平扩展
- **Redis**：缓存、会话、实时计数、分布式锁
- **混合架构**：根据数据特点选择合适的存储

技术选型需要综合考虑业务需求、数据特点、团队能力和运维成本。
