# MongoDB Replica Set 副本集

「MongoDB 副本集是怎么工作的？为什么主节点故障后能从节点自动升级？」

这是面试中经常被问到的问题。大多数人知道副本集有一主多从，但追问下去——「选举机制是什么？」「oplog 是怎么工作的？」「心跳检测间隔是多少？」「主节点挂了之后多久能恢复？」「writeConcern 怎么选？」——能答上来的就很少了。

本文深入讲解 MongoDB 副本集的原理和实践，帮你彻底理解这个高可用架构。

## 副本集基础

### 什么是副本集

副本集（Replica Set）是一组 MongoDB 实例组成的集群，提供**数据冗余**和**故障自动转移**。

```
副本集架构：

┌─────────────────────────────────────────┐
│           Replica Set "rs0"               │
│                                          │
│   ┌─────────┐                           │
│   │Primary   │ ← 接受所有写操作            │
│   │主节点     │                           │
│   └────┬────┘                            │
│        │ oplog 异步同步                    │
│        ▼                                 │
│   ┌─────────┐    ┌─────────┐            │
│   │Secondary│    │Secondary│            │
│   │从节点1   │    │从节点2   │            │
│   └────┬────┘    └────┬────┘            │
│        │ oplog         │ oplog           │
│        └───────┬───────┘                │
│                ▼                         │
│         ┌───────────┐                    │
│         │  oplog.rs │  ← 操作日志集合      │
│         └───────────┘                    │
└─────────────────────────────────────────┘
```

### 副本集成员类型

| 成员类型 | 说明 | 参与选举 | 存储数据 |
|---|---|---|---|
| Primary | 主节点 | 是（优先级最高） | 是 |
| Secondary | 从节点 | 是（可配置优先级） | 是 |
| Arbiter | 仲裁节点 | 是 | 否 |
| Delayed Secondary | 延迟从节点 | 否 | 是（延迟） |
| Hidden | 隐藏从节点 | 否（可配置） | 是 |

```javascript
//     { _id: 2, host: "mongodb3:27017", arbiterOnly: true },  // Arbiter
//     { _ id: 3, host: "mongodb4:27017", priority: 0, delayed: 3600 },  // 延迟1小时
//     { _id: 4, host: "mongodb5:27017", hidden: true, priority: 0 }  // 隐藏

```

## 数据同步机制

### oplog 简介

oplog（Operation Log）是副本集中的**操作日志集合**，记录主节点的所有写操作，从节点通过重放 oplog 实现数据同步。

```javascript
// oplog 结构
{
  ts: Timestamp(1699123456, 1),  // 操作时间戳
  t: NumberLong(1),               // term/选举任期
  h: NumberLong("-123456789"),   // 操作哈希
  v: NumberLong(2),              // oplog 版本
  op: "u",                       // 操作类型：i=insert, u=update, d=delete
  ns: "mydb.orders",             // 命名空间（数据库.集合）
  o: {                           // 操作文档
    $set: { status: "shipped" }
  },
  o2: { _id: ObjectId("...") }  // 查询条件
}
```

### oplog 操作类型

| op | 说明 | 示例 |
|---|---|---|
| `i` | 插入 | `{ op: "i", ns: "db.coll", o: { _id: 1, name: "张三" } }` |
| `u` | 更新 | `{ op: "u", ns: "db.coll", o: { $set: { name: "李四" } }, o2: { _id: 1 } }` |
| `d` | 删除 | `{ op: "d", ns: "db.coll", o: { _id: 1 } }` |
| `c` | 命令 | `{ op: "c", ns: "admin.$cmd", o: { drop: "coll" } }` |
| `n` | 无操作 | `{ op: "n" }`（用于心跳或占位） |

### 同步过程

```javascript
// 从节点同步流程

// 1. 初始同步（Initial Sync）
// 从节点加入副本集时，先进行完整的数据复制

// 2. 持续同步（Steady State Sync）
// 从节点持续拉取主节点的 oplog 并重放

// 同步状态查看
rs.status()

// 输出示例
{
  "set": "rs0",
  "date": ISODate("..."),
  "myState": 1,           // 1=Primary, 2=Secondary, 7=Arbiter
  "members": [
    {
      "_id": 0,
      "name": "mongodb1:27017",

        "ts": Timestamp(1699123456, 1),
        "t": NumberLong(1)
      },

    },
    {

      "name": "mongodb2:27017",

      "uptime": 3600,
      "optime": {
        "ts": Timestamp(1699123456, 1),

      },
      "lastHeartbeat": ISODate("..."),

    }
  ]
}
```

## 选举机制

### 选举触发条件

以下情况会触发选举：

| 触发条件 | 说明 |
|---|---|
| 主节点不可达 | 心跳超时（默认 10 秒） |
| 从节点数量减少 | 法定节点数不足 |
| 手动触发 | `rs.stepDown()` 或管理员操作 |
| 集群初始化 | `rs.initiate()` 首次选举 |

### 选举算法：Bully

MongoDB 使用 Bully 算法进行选举，原则是**优先级最高的节点获胜**。

```
选举过程：

1. 从节点检测主节点不可达
   Secondary → Primary 心跳超时

2. 从节点发起选举请求
   ┌─────────────────────────────────┐
   │ Secondary-1 (priority=2)         │
   │ 发现主节点不可达，发起选举         │
   └─────────────────────────────────┘

3. 其他节点响应
   Secondary-2 (priority=1) ← 同意
   Secondary-3 (priority=0) ← 拒绝（优先级低）

4. 选举获胜
   Secondary-1 获得多数投票，成为新的 Primary

优先级作用：
- priority 越高，越可能成为 Primary
- priority=0 的节点永远不能成为 Primary
```

### 选举参数

```javascript
// 选举相关参数
// heartbeatTimeoutSecs：心跳超时，默认 10 秒
// electionTimeoutMs：选举超时，默认 15 秒（检测到主节点失效到开始选举的时间）
// heartbeatIntervalMs：心跳间隔，默认 2 秒

// 配置副本集
cfg = rs.conf();
cfg.settings = {
  heartbeatTimeoutSecs: 10,    // 心跳超时
  electionTimeoutMs: 15000,     // 选举超时
  heartbeatIntervalMs: 2000     // 心跳间隔
};
rs.reconfig(cfg);
```

### 多数原则

法定节点数（Majority）是副本集进行写入确认的基准。

```javascript
// 3 节点副本集
// 法定节点数 = ⌊(3/2)⌋ + 1 = 2

// 5 节点副本集
// 法定节点数 = ⌊(5/2)⌋ + 1 = 3

// 写入确认
db.orders.insertOne(
  { ... },
  { writeConcern: { w: "majority" } }  // 需要 2 或 3 个节点确认
);

// 查看当前配置的法定节点数
db.adminCommand({ getDefaultRWConcern: 1 });
```

### 副本集成员优先级

```javascript
// 配置成员优先级
cfg = rs.conf();
cfg.members[0].priority = 2;   // 优先级最高，最可能成为 Primary
cfg.members[1].priority = 1;   // 默认优先级
cfg.members[2].priority = 0;   // 不能成为 Primary
rs.reconfig(cfg);

// 隐藏节点：不影响选举
cfg.members[3].priority = 0;
cfg.members[3].hidden = true;
rs.reconfig(cfg);

// 延迟节点：作为灾备，落后指定时间
cfg.members[4].priority = 0;
cfg.members[4].hidden = true;
cfg.members[4].secondaryDelaySecs = 3600;  // 延迟 1 小时
rs.reconfig(cfg);
```

## 故障转移

### 自动故障转移

当主节点不可达时，副本集会自动进行故障转移：

```
故障转移流程：

T1: 正常运行
┌─────────────────────────────────────────┐
│ Primary ──── 同步 ────→ Secondary-1     │
│ Primary ──── 同步 ────→ Secondary-2     │
└─────────────────────────────────────────┘

T2: Primary 故障
┌─────────────────────────────────────────┐
│ ❌ Primary 宕机                          │
│ Secondary-1 ──×─── 心跳超时（10秒）       │
│ Secondary-2 ──×─── 心跳超时（10秒）       │
└─────────────────────────────────────────┘

T3: 发起选举
┌─────────────────────────────────────────┐
│ Secondary-1 → "我发起选举" → Secondary-2 │
│ Secondary-2 → "投票给你" → Secondary-1   │
│ Secondary-1 获得多数票                    │
└─────────────────────────────────────────┘

T4: 故障转移完成
┌─────────────────────────────────────────┐
│ Secondary-1 成为新的 Primary              │
│ Secondary-2 继续同步到新 Primary          │
│ 整个过程约 10-30 秒                      │
└─────────────────────────────────────────┘
```

### 故障转移时间

| 阶段 | 默认时间 | 说明 |
|---|---|---|
| 心跳超时 | 10 秒 | 检测到主节点失效 |
| 选举超时 | 15 秒 | 等待选举完成 |
| 故障转移总时间 | ~30-50 秒 | 整个过程 |

```javascript
// 查看副本集状态
rs.status();

// 查看故障转移详情
db.adminCommand({ replSetGetStatus: 1 });

// 查看成员健康状态
rs.memberStatus();
```

### 手动故障转移

```javascript
// 手动切换主节点
// 当前主节点执行 stepDown
db.adminCommand({ replSetStepDown: 60 });  // 60 秒内不参与选举

// 或者让指定从节点发起选举
// 需要在从节点上执行
rs.stepDown();

// 强制重新选举
rs.freeze(300);  // 冻结 300 秒内不参与选举
```

## 读写策略

### 读偏好（Read Preference）

```javascript
// 默认：只在主节点读取
db.orders.find({}).readPref("primary")

// 主优先：如果主不可用，从从节点读取
db.orders.find({}).readPref("primaryPreferred")

// 从优先：从从节点读取
db.orders.find({}).readPref("secondaryPreferred")

// 只从从节点读取（如果有的话）
db.orders.find({}).readPref("secondary")

// 最近的首选：从延迟最小的节点读取
db.orders.find({}).readPref("nearest")
```

### 写确认（Write Concern）

```javascript
// w=1：仅主节点确认（默认）
db.orders.insertOne(
  { ... },
  { writeConcern: { w: 1 } }
);

// w=majority：多数节点确认
db.orders.insertOne(
  { ... },
  { writeConcern: { w: "majority" } }
);

// w=<number>：指定节点数确认
db.orders.insertOne(
  { ... },
  { writeConcern: { w: 2 } }
);

// w=majority + journal：多数节点确认 + 刷盘
db.orders.insertOne(
  { ... },
  { writeConcern: { w: "majority", j: true } }
);

// wtimeout：等待超时
db.orders.insertOne(
  { ... },
  { writeConcern: { w: "majority", wtimeout: 5000 } }  // 5 秒超时
);
```

### 读偏好 + 写确认组合

| 组合 | 特点 | 使用场景 |
|---|---|---|
| `primary` + `w:1` | 最快，无一致性保证 | 日志、可丢失数据 |
| `secondaryPreferred` + `w:majority` | 负载均衡，强一致性 | 读多写少业务 |
| `nearest` + `w:majority` | 最低延迟，强一致性 | 全球部署 |

## oplog 大小与维护

### oplog 大小

```javascript
// 默认 oplog 大小（取决于总 RAM）
// 通常为磁盘的 5%，最大 50GB

// 查看 oplog 大小
db.getSiblingDB("local").oplog.rs.stats();

// 查看 oplog 使用情况
db.getSiblingDB("local").oplog.rs.find().sort({ts: -1}).limit(1);
db.getSiblingDB("local").oplog.rs.find().sort({ts: 1}).limit(1);

// 第一次和最后一次操作的时间差 = oplog 可覆盖的时间范围
```

### oplog 调整

```javascript
// 创建更大的 oplog（需要在初始化前设置）
// mongod.conf
storage:
  oplogMinSize: 2048  // MB

// 或者使用 replSetReconfig 调整（MongoDB 4.4+）
cfg = rs.conf();
cfg.members[0].storage = { oplogMinSizeMB: 2048 };
rs.reconfig(cfg);
```

## 面试实战

**面试官问**：「MongoDB 副本集的选举机制是什么？」

**深入回答**：

> MongoDB 使用 Bully 算法进行选举：
>
> 1. **触发条件**：主节点心跳超时（默认 10 秒）或其他节点发起选举
>
> 2. **选举过程**：
>    - 优先级最高的节点发起选举
>    - 其他节点投票
>    - 获得多数票的节点成为新的主节点
>
> 3. **多数原则**：3 节点副本集需要 2 票才能当选
>
> 4. **选举时间**：整个故障转移约 30-50 秒

**面试官问**：「oplog 是什么？它是怎么工作的？」

**深入回答**：

> oplog 是副本集中的操作日志：
>
> 1. **存储位置**：`local` 数据库的 `oplog.rs` 集合
>
> 2. **内容**：主节点的所有写操作（insert、update、delete）
>
> 3. **同步过程**：从节点持续拉取主节点的 oplog 并重放
>
> 4. **大小**：默认占磁盘 5%，最大 50GB
>
> 5. **用途**：数据同步、故障恢复、增量备份

**面试官问**：「副本集和主从复制的区别是什么？」

**深入回答**：

> 核心区别：
>
> 1. **自动故障转移**：副本集有，主从复制没有
>
> 2. **选举机制**：副本集有自动选举，主从复制没有
>
> 3. **数据一致性**：副本集基于 oplog 的同步更可靠
>
> 4. **配置复杂度**：副本集更简单，主从复制需要手动切换
>
> 5. **MongoDB 官方**：不再推荐使用主从复制，副本集是标准方案

## 总结

副本集核心要点：

- **架构**：一主多从，主节点接受写入，从节点同步数据
- **数据同步**：通过 oplog 实现，从节点重放操作日志
- **选举机制**：Bully 算法，优先级最高的节点获胜
- **故障转移**：自动进行，约 30-50 秒完成
- **读写策略**：`readPref` 控制读节点，`writeConcern` 控制写入确认
- **成员类型**：Primary、Secondary、Arbiter、Delayed、Hidden

副本集是 MongoDB 高可用的基石，理解其原理对于生产运维至关重要。
