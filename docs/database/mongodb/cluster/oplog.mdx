# MongoDB oplog 同步原理

「MongoDB 从节点是怎么同步主节点数据的？」

理解 oplog 是掌握 MongoDB 副本集的关键。oplog 不仅用于数据同步，还是增量备份、change streams 等功能的基础。

本文深入讲解 MongoDB oplog 的工作原理。

## oplog 概述

### 什么是 oplog

oplog（Operation Log）是 MongoDB 副本集中的**操作日志**，记录主节点的所有写操作，从节点通过重放 oplog 实现数据同步。

```javascript
// oplog 存储在 local 数据库
use local

// 查看 oplog 集合
db.oplog.rs.find()

// oplog 结构
{
  ts: Timestamp(1699123456, 1),    // 操作时间戳（包含逻辑时间）
  t: NumberLong(1),               // 选举任期（Term）
  h: NumberLong("-123456789"),    // 操作的唯一哈希
  v: NumberLong(2),               // oplog 版本
  op: "i",                        // 操作类型
  ns: "shop.orders",              // 命名空间（db.collection）
  o: { ... },                     // 操作文档
  o2: { _id: ... }               // 查询条件（用于更新/删除）
}
```

### oplog 特点

| 特点 | 说明 |
|---|---|
| 固定集合 | oplog 是固定大小的集合 |
| 幂等性 | oplog 操作是幂等的，可以安全重放 |
| 幂等性 | 包含操作前后状态，可重复应用 |
| 增量记录 | 只记录增量操作，不记录查询 |

```
oplog 与数据的关系：

┌─────────────────────────────────────────┐
│            Primary                       │
│                                          │
│  原始文档                                │
│  { _id: 1, name: "张三", age: 25 }      │
│                                          │
│  更新操作                                │
│  db.users.update({_id:1}, {$set:{age:26}})│
│                                          │
│  oplog 记录                              │
│  { op: "u", ns: "test.users",          │
│    o: {$set:{age:26}}, o2:{_id:1} }     │
└─────────────────────────────────────────┘
                ↓
┌─────────────────────────────────────────┐
│           Secondary                      │
│                                          │
│  读取 oplog                             │
│  { op: "u", ns: "test.users",          │
│    o: {$set:{age:26}}, o2:{_id:1} }     │
│                                          │
│  重放操作                                │
│  db.users.update({_id:1}, {$set:{age:26}})│
│                                          │
│  结果：文档已更新                         │
│  { _id: 1, name: "张三", age: 26 }      │
└─────────────────────────────────────────┘
```

## oplog 操作类型

### 插入操作（op: i）

```javascript
// 主节点插入
db.orders.insertOne({
  _id: ObjectId("..."),
  customer: "张三",
  total: 100
});

// oplog 记录
{
  ts: Timestamp(...),
  op: "i",
  ns: "shop.orders",
  o: {
    _id: ObjectId("..."),
    customer: "张三",
    total: 100
  }
}

// 从节点重放：直接插入相同文档
```

### 更新操作（op: u）

```javascript
// 主节点更新
db.orders.updateOne(
  { _id: ObjectId("...") },
  { $set: { status: "shipped" } }
);

// oplog 记录
{
  ts: Timestamp(...),
  op: "u",
  ns: "shop.orders",
  o: { $set: { status: "shipped" } },
  o2: { _id: ObjectId("...") }  // 查询条件
}

// 从节点重放：应用相同的更新条件
```

### 删除操作（op: d）

```javascript
// 主节点删除
db.orders.deleteOne({ _id: ObjectId("...") });

// oplog 记录
{
  ts: Timestamp(...),
  op: "d",
  ns: "shop.orders",
  o: { _id: ObjectId("...") }  // 要删除的文档
}

// 从节点重放：删除相同文档
```

### 命令操作（op: c）

```javascript
// 主节点执行命令
db.orders.drop();  // 删除集合

// oplog 记录
{
  ts: Timestamp(...),
  op: "c",
  ns: "admin.$cmd",
  o: { drop: "orders" }
}

// 从节点重放：执行相同的命令
```

## oplog 同步过程

### 初始同步

当从节点第一次加入副本集或数据落后太多时，会进行**初始同步**：

```
初始同步流程：

1. 选择同步源（Primary）
   ┌─────────────────────────────────┐
   │ Secondary 选择一个健康的主节点    │
   └─────────────────────────────────┘

2. 克隆数据
   ┌─────────────────────────────────┐
   │ 从主节点复制所有数据库和集合       │
   │ 遍历每个文档，插入到本地          │
   └─────────────────────────────────┘

3. 同步 oplog
   ┌─────────────────────────────────┐
   │ 记录同步开始时间点                │
   │ 持续拉取主节点的 oplog 并重放     │
   │ 直到追上主节点                    │
   └─────────────────────────────────┘

4. 完成
   ┌─────────────────────────────────┐
   │ 切换为 Secondary 状态             │
   │ 正常同步 oplog                   │
   └─────────────────────────────────┘
```

```javascript
// 手动触发初始同步
// 先删除从节点数据，然后重启
db.adminCommand({ resync: 1 });

// 或者删除数据目录后重启 mongod
```

### 增量同步（Steady State）

初始同步完成后，从节点进入**增量同步**状态：

```
增量同步流程：

┌─────────────────────────────────────────┐
│           Secondary                     │
│                                          │
│  1. 记录同步游标（last applied op）        │
│     lastOpTime = Timestamp(1699123456, 1)│
│                                          │
│  2. 定时拉取 oplog                       │
│     tail oplog.rs since lastOpTime      │
│                                          │
│  3. 重放操作                             │
│     apply each operation in oplog        │
│                                          │
│  4. 更新同步游标                         │
│     lastOpTime = newOpTime              │
└─────────────────────────────────────────┘
```

### 多线程重放

MongoDB 4.2+ 支持多线程 oplog 重放，提高同步效率：

```javascript
// 配置 oplog 批处理大小
db.adminCommand({
  replSetSyncFrom: "hostname:port"  // 指定同步源
});

// 查看同步线程数
db.serverStatus().repl.buffer.counters;

// 查看同步延迟
db.runCommand({ replSetGetStatus: 1 })
// 输出中查看 "optimeDate" 和 "lastHeartbeat"
```

## oplog 大小与配置

### 默认大小

```javascript
// oplog 大小计算
// 默认：最大磁盘空间的 5%，或 50GB（取较小值）
// 4GB RAM → 约 50MB oplog
// 64GB RAM → 约 3.2GB oplog
// 256GB RAM → 约 12.8GB oplog（但最大 50GB）

// 查看 oplog 大小
db.getSiblingDB("local").oplog.rs.stats().maxSize;
```

### 调整 oplog 大小

```javascript
// 方法1：创建初始化配置时设置
// 在副本集初始化前设置
db.adminCommand({
  replSetInitiate: {
    _id: "rs0",
    members: [{ _id: 0, host: "mongodb1:27017" }]
  }
});
// 创建后无法直接修改 oplog 大小

// 方法2：MongoDB 4.4+ 支持在线调整
cfg = rs.conf();
cfg.settings = { oplogMinSize: 2048 };  // 2GB
rs.reconfig(cfg);

// 方法3：重建 oplog（需要停机）
// 1. 停止所有节点
// 2. 删除 oplog
// 3. 以单机模式启动
// 4. 创建新的 oplog
// 5. 重启副本集
```

## oplog 与一致性

### readConcern: majority

oplog 是实现 `readConcern: majority` 的关键：

```javascript
// majority 读取原理
// 1. 主节点写入 op 到 oplog
// 2. oplog 复制到多数节点
// 3. 读取请求可以读到已被多数节点确认的数据

// 示例
db.orders.insertOne(
  { _id: 1, amount: 100 },
  { writeConcern: { w: "majority" } }
);

// majority 读取
db.orders.findOne(
  { _id: 1 },
  { readConcern: { level: "majority" } }
);
```

### oplog 延迟检测

```javascript
// 查看同步延迟
rs.status().members.forEach(m => {
  print(m.name + ": " +
    "optime: " + m.optimeDate + ", " +
    "lag: " + m.lagSinceLastHeartbeat + "s"
  );
});

// 查看 oplog 覆盖时间
// oplog 第一条和最后一条的时间差
firstOp = db.getSiblingDB("local").oplog.rs.find().sort({ts: 1}).limit(1).next();
lastOp = db.getSiblingDB("local").oplog.rs.find().sort({ts: -1}).limit(1).next();
print("Oplog window: " + (lastOp.ts.getTime() - firstOp.ts.getTime()) / 1000 + " seconds");
```

## oplog 应用场景

### Change Streams

基于 oplog 实现的变更流：

```javascript
// 使用 change streams 监听变更
const changeStream = db.orders.watch();

// 监听所有变更
changeStream.on("change", (change) => {
  console.log("订单变更:", change);
});

// 监听插入操作
changeStream = db.orders.watch([
  { $match: { operationType: "insert" } }
]);

// 监听特定集合的变更
changeStream = db.orders.watch(
  [],
  { fullDocument: "updateLookup" }
);
```

### 增量备份

基于 oplog 实现增量备份：

```javascript
// 1. 全量备份
// mongodump --out /backup/full

// 2. 记录备份时间点
backupTime = db.getSiblingDB("local").oplog.rs.find().sort({ts: -1}).limit(1).next().ts;

// 3. 增量备份
// mongodump --out /backup/incremental --query "{ ts: { $gt: backupTime } }"

// 4. 恢复到指定时间点
// mongorestore --oplogReplay --oplogLimit backupTime
```

### 数据同步到其他系统

使用 oplog 同步到其他数据系统：

```javascript
// 伪代码：Kafka Connect MongoDB Source
// 基于 oplog 实时同步到 Kafka

const changeStream = db.collection.watch([], {
  fullDocument: "updateLookup"
});

changeStream.forEach(change => {
  kafka.send({
    topic: "mongodb-changes",
    messages: [{
      key: change.documentKey._id.toString(),
      value: JSON.stringify(change)
    }]
  });
});
```

## 面试实战

**面试官问**：「如果 oplog 太大或太小会怎样？」

**深入回答**：

> oplog 大小的影响：
>
> 1. **太大**：占用过多磁盘空间，但覆盖时间长
>
> 2. **太小**：从节点可能跟不上主节点的写入速度，导致同步失败
>
> 3. **计算原则**：oplog 应能覆盖主节点最大负载下的 1-2 小时写入量
>
> 4. **调整建议**：高写入场景需要增大 oplog

**面试官问**：「从节点同步延迟过长怎么排查？」

**深入回答**：

> 排查步骤：
>
> 1. **检查网络**：主从节点网络是否正常
>
> 2. **检查负载**：从节点 CPU、IO 是否达到瓶颈
>
> 3. **查看延迟指标**：`rs.status()` 中的 `lagSinceLastHeartbeat`
>
> 4. **检查 oplog 大小**：是否 oplog 太小导致从节点跟不上
>
> 5. **查看慢查询**：从节点是否有慢查询阻塞了同步
>
> 6. **考虑增加从节点**：分散读取压力

## 总结

oplog 核心要点：

- **定义**：副本集的操作日志，记录主节点的所有写操作
- **存储位置**：`local.oplog.rs`
- **操作类型**：i（插入）、u（更新）、d（删除）、c（命令）
- **同步过程**：初始同步 + 增量同步
- **大小**：默认磁盘 5%，最大 50GB
- **应用**：Change Streams、增量备份、数据同步

理解 oplog 是掌握 MongoDB 副本集原理的关键。
