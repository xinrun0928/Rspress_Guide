# MongoDB 分片键选择原则

分片键是分片集群设计的核心。选择正确的分片键对数据分布、查询性能和系统稳定性至关重要。

## 分片键的重要性

分片键决定了：

1. **数据分布**：数据如何分散到各个分片
2. **查询效率**：查询是否能被高效路由
3. **写入分布**：写入负载是否均衡
4. **扩展性**：未来能否平滑扩容

```javascript
// 一旦设置，分片键几乎无法修改
// 需要在设计阶段仔细考虑
```

## 分片键选择原则

### 原则一：高基数

分片键应该有足够多的不同值，确保数据可以分散到多个分片。

```javascript
// ✅ 好的分片键：高基数
{ user_id: 1 }          // 唯一 ID
{ order_id: "hashed" }  // 唯一 ID
{ email: 1 }            // 通常唯一

// ❌ 差的分片键：低基数
{ status: 1 }           // 只有几个状态
{ gender: 1 }           // 只有两个值
{ is_active: 1 }        // 只有两个值
```

### 原则二：分布均匀

分片键的值分布应该均匀，避免出现热点分片。

```javascript
// ❌ 差的分片键：分布不均匀
{ created_date: 1 }     // 新数据都写入最新的分片

// ✅ 好的分片键：分布均匀
{ user_id: "hashed" }  // 哈希后均匀分布
```

### 原则三：查询局部化

常用查询应该包含分片键，以便单分片查询。

```javascript
// 常用查询：按用户查订单
// ✅ 好的分片键
sh.shardCollection("shop.orders", { customer_id: 1 })
// 查询 db.orders.find({ customer_id: "C001" }) → 单分片

// ❌ 差的分片键
sh.shardCollection("shop.orders", { order_id: "hashed" })
// 查询 db.orders.find({ customer_id: "C001" }) → 广播到所有分片
```

### 原则四：避免单调递增/递减

单调变化的分片键会导致新数据都写入同一个分片。

```javascript
// ❌ 问题：单调递增的分片键
{ timestamp: 1 }        // 新数据都写入最大范围的 chunk
{ auto_increment_id: 1 } // 新数据都写入最后 chunk

// ✅ 解决方案
{ _id: "hashed" }       // 哈希分片
{ user_id: 1, _id: 1 }  // 复合分片键
```

## 常见分片键模式

### 哈希分片

```javascript
// 适合：高写入、无明显范围查询
sh.shardCollection("app.events", { user_id: "hashed" })

// 优点：数据均匀分布
// 缺点：范围查询效率低
```

### 范围分片

```javascript
// 适合：范围查询多、值分布均匀
sh.shardCollection("app.sensors", { location: 1, timestamp: 1 })

// 优点：范围查询高效
// 缺点：可能导致数据倾斜
```

### 复合分片键

```javascript
// 适合：需要结合多个字段的优点
sh.shardCollection("app.orders", { customer_id: 1, order_id: "hashed" })

// customer_id 提供查询局部化
// order_id 提供写入均匀分布
```

## 分片键选择流程

```
分片键选择决策流程：

1. 列出所有查询模式
   │
   ▼
2. 找出最常用的查询
   │
   ▼
3. 确定查询中包含的字段
   │
   ▼
4. 检查字段的基数
   │
   ├─ 高基数（百万+）→ 候选
   ├─ 中基数（千-万）→ 可考虑
   └─ 低基数（百以下）→ 不适合
   │
   ▼
5. 检查值分布
   │
   ├─ 均匀分布 → 范围分片候选
   └─ 不均匀 → 哈希分片候选
   │
   ▼
6. 考虑单调性问题
   │
   ├─ 存在单调递增/递减 → 使用哈希
   └─ 无单调性问题 → 可使用范围
   │
   ▼
7. 最终确定分片键
```

## 分片键设计示例

### 场景一：电商订单系统

```javascript
// 常用查询：
// 1. 按用户查订单（最频繁）
// 2. 按时间查订单（偶尔）
// 3. 按订单ID精确查询

// 分析：
// - customer_id：高基数、分布均匀
// - created_at：单调递增
// - order_id：高基数、唯一

// 推荐分片键：
sh.shardCollection("shop.orders", { customer_id: 1 })
// 优点：主要查询高效

// 或者：
sh.shardCollection("shop.orders", { customer_id: 1, order_id: "hashed" })
// 优点：查询高效 + 写入均匀
```

### 场景二：社交应用消息系统

```javascript
// 常用查询：
// 1. 按会话查消息（最频繁）
// 2. 按用户查消息（偶尔）

// 推荐分片键：
sh.shardCollection("app.messages", { conversation_id: 1, created_at: 1 })
// 优点：会话内消息相邻，支持时间范围查询
```

### 场景三：IoT 时序数据

```javascript
// 常用查询：
// 1. 按设备查数据
// 2. 按时间范围查某设备数据

// 推荐分片键：
sh.shardCollection("iot.sensors", { device_id: "hashed", timestamp: 1 })
// 优点：设备数据均匀分布 + 支持时间范围查询
```

### 场景四：日志分析系统

```javascript
// 常用查询：
// 1. 按时间范围查日志
// 2. 按服务查日志

// 推荐分片键：
sh.shardCollection("logs.events", { timestamp: 1, service: 1 })
// 优点：时间范围查询高效
// 注意：可能需要定期预分割 chunk
```

## 分片键限制

```javascript
// 分片键限制：

// 1. 索引前缀要求
// 如果分片键是 { a: 1, b: 1 }，查询必须包含 a
db.collection.find({ a: value })           // ✅ 可以
db.collection.find({ a: value, b: value }) // ✅ 可以
db.collection.find({ b: value })           // ❌ 不可以

// 2. 分片键不能修改（几乎）
// 唯一方案：新建集合，迁移数据

// 3. 唯一索引限制
// 分片键字段上会自动创建唯一索引
// 无法在非分片键字段上创建唯一索引（除非是分片键的超集）
```

## 分片键修改方案

```javascript
// 如果必须修改分片键：

// 1. 创建新集合（新的分片键）
sh.shardCollection("app.new_collection", { new_key: 1 })

// 2. 编写迁移脚本
async function migrateData() {
  const batchSize = 1000;
  let lastId = null;

  while (true) {
    const query = lastId ? { _id: { $gt: lastId } } : {};
    const docs = await db.old_collection.find(query).limit(batchSize).toArray();

    if (docs.length === 0) break;

    await db.new_collection.insertMany(docs);
    lastId = docs[docs.length - 1]._id;

    print(`Migrated ${lastId}`);
  }
}

// 3. 切换应用程序
// 4. 删除旧集合
```

## 常见错误

### 错误一：使用低基数字段

```javascript
// ❌ 错误
sh.shardCollection("app.users", { country: 1 })
// 只有几百个国家，无法有效分散数据

// ✅ 正确
sh.shardCollection("app.users", { user_id: "hashed" })
```

### 错误二：忽视单调性

```javascript
// ❌ 错误
sh.shardCollection("app.events", { timestamp: 1 })
// 新数据都写入一个分片，导致热点

// ✅ 正确
sh.shardCollection("app.events", { event_id: "hashed" })
// 或者
sh.shardCollection("app.events", { user_id: 1, timestamp: 1 })
```

### 错误三：不考虑查询模式

```javascript
// ❌ 错误
sh.shardCollection("app.orders", { order_id: "hashed" })
// 常用查询是按用户查订单，但分片键不支持

// ✅ 正确
sh.shardCollection("app.orders", { customer_id: 1 })
// 或者
sh.shardCollection("app.orders", { customer_id: 1, order_id: "hashed" })
```

## 总结

分片键选择要点：

- **高基数**：确保数据可以分散
- **均匀分布**：避免热点分片
- **查询局部化**：常用查询包含分片键
- **避免单调性**：防止写入热点
- **复合分片键**：结合多个字段的优点

分片键的选择需要综合考虑业务场景、数据特点和查询模式。
