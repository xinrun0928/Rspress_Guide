# MongoDB 高可用：故障转移与选举原理

凌晨 2 点，你的告警平台疯狂推送消息：主节点宕了。业务团队的电话一个接一个打过来，数据库连接超时、应用响应缓慢、用户投诉不断……

但如果你用的是 MongoDB 副本集，这件事可能没那么可怕。**故障转移（Failover）** 会自动发生，副本集会在秒级内选举出新的主节点，业务中断时间从"手动恢复的数十分钟"缩短到"自动切换的几秒钟"。

这一切是如何实现的？我们从副本集选举机制说起。

## 副本集选举核心要素

MongoDB 副本集通过 **Raft 共识算法的变体** 实现主节点选举。但与标准 Raft 不同，MongoDB 的选举考虑了更多实际因素：成员优先级、网络延迟、oplog 同步状态等。

### OpTime：选举的"时钟"

OpTime（Operation Time）是 MongoDB 副本集选举中最核心的概念。它不是简单的时间戳，而是由两部分组成：

| 组成部分 | 说明 |
|---|---|
| `term` | 选举轮次，类似于 Raft 的 term，递增用于检测"新鲜度" |
| `optime` | oplog 中最新一条操作的时间戳 |

```javascript
// 查看当前节点的 OpTime 状态
rs.status().members.forEach(m => {
    print(m.name + ": term=" + m.term + ", optime=" + m.optimeDate);
});
```

选举时，MongoDB 会比较所有有资格成为主节点的成员的 OpTime。**谁的 OpTime 最新，谁就更有可能赢得选举。**

### Priority：优先级权重

每个副本集成员都有一个 `priority` 配置，取值范围 0-1000（默认 1）。优先级越高，在选举中获得胜利的概率越大。

```javascript
// 配置成员优先级
cfg = rs.conf();
cfg.members[0].priority = 2;  // 第一个成员优先级设为 2
cfg.members[1].priority = 1;  // 第二个成员保持默认优先级 1
cfg.members[2].priority = 0;  // 第三个成员永远不能成为主节点（灾备节点）
rs.reconfig(cfg);
```

**Priority 为 0 的成员称为 Arbiter（仲裁节点）**，它参与选举投票但不存储数据，通常用于在副本集只有两个数据节点时提供奇数票。

## 选举触发条件

MongoDB 副本集不会无缘无故发起选举。以下几种情况会触发选举：

### 1. 主节点不可达

这是最常见的触发场景。当主节点与副本集其他成员的 heartbeat 超时（默认 10 秒），副本集认为主节点宕机，触发重新选举。

```javascript
// 查看心跳超时配置
rs.conf().settings.heartbeatTimeoutSecs  // 默认 10 秒
```

### 2. 手动触发选举

管理员可以主动发起选举，用于维护或故障演练：

```javascript
// 强制触发选举（需要先关闭主节点或使用 rs.freeze）
rs.freeze(30);  // 冻结当前节点 30 秒不接受选举
rs.stepDown();   // 当前主节点主动降级
```

### 3. 成员状态变化

- 从节点加入副本集
- 优先级更高的节点加入
- 现有成员优先级被修改

### 4. 心跳超时重置

如果一个从节点长时间与主节点失联（超过 `electionTimeoutMillis`，默认 10 秒），它会认为主节点不可达，主动发起选举。

## 选举过程详解

当选举被触发后，MongoDB 遵循以下流程：

```
1. 候选者发起选举请求
   ↓
2. 其他成员投票
   ↓
3. 得票过半者成为新主节点
   ↓
4. 其他节点切换为从节点
```

### 第一步：候选者资格检查

发起选举的节点必须满足以下条件：

- `priority > 0`（优先级为 0 不能发起选举）
- 数据同步状态足够新（oplog 已追上主节点）
- 与大多数成员网络可达

```javascript
// 查看节点同步状态
db.adminCommand({ replSetGetStatus: 1 }).members.forEach(m => {
    print(m.name + ": stateStr=" + m.stateStr +
          ", syncedTo=" + m.optimeDate);
});
```

### 第二步：OpTime 对比与投票

MongoDB 使用一种优化的 OpTime 对比算法：

```javascript
// 投票逻辑伪代码
function canVoteFor(candidateOpTime, myOpTime) {
    // 如果候选者的 OpTime 比自己更新，支持
    if (candidateOpTime.term > myOpTime.term) return true;
    if (candidateOpTime.term == myOpTime.term &&
        candidateOpTime.optime > myOpTime.optime) return true;
    return false;
}
```

**注意**：这里有一个关键优化——候选者的 OpTime 不需要是全局最新的，只要比投票者新即可。这避免了"所有节点同时宕机恢复时，谁都不敢当候选者"的问题。

### 第三步：确认当选

当选票超过副本集成员总数的 **过半（majority）** 时，候选者当选主节点：

| 副本集成员数 | 最低投票数 |
|---|---|
| 3 | 2 |
| 5 | 3 |
| 7 | 4 |

```javascript
// 副本集应保持奇数成员，避免平票
// 推荐配置：3 个数据节点 或 2 数据节点 + 1 Arbiter
```

## 选举时间参数调优

MongoDB 提供了多个参数控制选举行为：

```javascript
// 调整选举超时时间（默认 10 秒）
cfg = rs.conf();
cfg.settings = cfg.settings || {};
cfg.settings.electionTimeoutMillis = 15000;  // 15 秒超时
rs.reconfig(cfg);
```

| 参数 | 默认值 | 说明 |
|---|---|---|
| `electionTimeoutMillis` | 10000ms | 心跳超时后多久触发选举 |
| `heartbeatTimeoutSecs` | 10s | 心跳超时阈值 |
| `heartbeatIntervalMillis` | 2000ms | 心跳间隔 |

**调优建议**：

- 网络不稳定的机房环境，可适当增大 `electionTimeoutMillis`
- 对可用性要求极高的场景，可减小 `electionTimeoutMillis`，但可能导致"假故障"触发不必要的选举

## 选举中的常见问题与优化

### 脑裂问题

如果副本集被分割成两个等大小的分区，每个分区都会尝试选举主节点，导致"脑裂"。MongoDB 通过**过半机制**防止这个问题：

- 3 节点副本集：需要 2 票才能当选
- 如果分成 1+2 两个分区：2 的那个分区可以选主，1 的那个无法当选

### Priority 配置不当导致的问题

```javascript
// 错误配置：两个节点 priority 相同，可能导致频繁切换
cfg.members[0].priority = 2;
cfg.members[1].priority = 2;  // 问题：优先级相同
cfg.members[2].priority = 1;

// 正确配置：明确主次
cfg.members[0].priority = 3;  // 主数据中心
cfg.members[1].priority = 2;  // 次数据中心
cfg.members[2].priority = 1;  // 第三个节点
```

### 网络分区导致的"假选举"

当主节点网络中断但实际存活时，可能出现：

1. 从节点发起选举并当选新主
2. 旧主恢复后发现 term 已更新，自动降级为从节点

这可能导致短暂的**双主状态**（旧主还未感知降级），此时写入旧主的数据会丢失。

```javascript
// 配置 catchUpTimeoutMs 控制新主等待从节点追数据的超时
cfg.settings.catchUpTimeoutMillis = 30000;  // 30 秒
```

## 最佳实践

1. **保持奇数成员**：优先使用 3 节点副本集，避免偶数导致平票
2. **合理设置 Priority**：明确主备关系，灾备节点设置 `priority = 0`
3. **跨机房部署**：主节点和多数投票节点应分布在不同机房
4. **监控选举频率**：频繁选举说明网络不稳定或参数配置不当
5. **预留 Arbiter**：对于只有两个数据中心的场景，可用 Arbiter 解决奇数问题

```javascript
// 推荐的 3 节点副本集配置
{
    _id: "rs0",
    members: [
        { _id: 0, host: "192.168.1.10:27017", priority: 2 },
        { _id: 1, host: "192.168.1.11:27017", priority: 2 },
        { _id: 2, host: "192.168.1.12:27017", priority: 1, arbiterOnly: true }
    ]
}
```

## 总结

MongoDB 的故障转移机制是保证高可用的核心。理解 OpTime 对比、Priority 权重、选举触发条件，你就能设计出稳定可靠的副本集架构。

下次主节点宕机时，别慌——副本集已经在自动为你处理了。
