# MongoDB Sharded Cluster 分片集群

「MongoDB 分片集群是怎么工作的？为什么 mongos 能知道数据在哪？」

这是面试中经常被问到的问题。大多数人知道 MongoDB 可以水平扩展，但追问下去——「mongos 的路由机制是什么？」「配置服务器存储了什么元数据？」「chunk 是怎么分裂和迁移的？」「分片键怎么选？」——能答上来的就很少了。

本文深入讲解 MongoDB 分片集群的原理和架构，帮你彻底理解这个分布式存储系统。

## 分片集群概述

### 什么是分片集群

分片集群（Sharded Cluster）是 MongoDB 的**水平扩展方案**，通过将数据分散到多个分片，实现海量数据存储和高并发访问。

```text
分片集群整体架构：

┌─────────────────────────────────────────────────────────────┐
│                    Sharded Cluster                          │
│                                                              │
│    ┌──────────────────────────────────────────────────┐     │
│    │                   mongos 路由层                    │     │
│    │                                                   │     │
│    │  ┌─────────┐  ┌─────────┐  ┌─────────┐           │     │
│    │  │ mongos1 │  │ mongos2 │  │ mongos3 │  ← 路由   │     │
│    │  └────┬────┘  └────┬────┘  └────┬────┘           │     │
│    └───────┼────────────┼────────────┼──────────────────┘     │
│            │            │            │                       │
│    ┌───────┴────────────┴────────────┴──────────────────┐     │
│    │                   数据层                            │     │
│    │                                                    │     │
│    │  Shard1 (RS)      Shard2 (RS)      Shard3 (RS)    │     │
│    │  ┌─────────┐      ┌─────────┐      ┌─────────┐     │     │
│    │  │Primary  │      │Primary  │      │Primary  │     │     │
│    │  │Sec.     │      │Sec.     │      │Sec.     │     │     │
│    │  └─────────┘      └─────────┘      └─────────┘     │     │
│    │                                                    │     │
│    └────────────────────────────────────────────────────┘     │
│                                                              │
│    ┌────────────────────────────────────────────────────┐     │
│    │               Config Server Replica Set            │     │
│    │                                                    │     │
│    │  ┌─────────┐  ┌─────────┐  ┌─────────┐            │     │
│    │  │Config1  │  │Config2  │  │Config3  │  ← 元数据  │     │
│    │  └─────────┘  └─────────┘  └─────────┘            │     │
│    └────────────────────────────────────────────────────┘     │
└─────────────────────────────────────────────────────────────┘
```

### 分片集群组件

| 组件 | 说明 | 作用 |
|---|---|---|
| mongos | 路由进程 | 接收客户端请求，转发到对应分片 |
| Config Server | 配置服务器 | 存储集群元数据（分片信息、chunk 分布） |
| Shard | 分片 | 实际存储数据的节点，通常是副本集 |

## mongos 路由机制

### mongos 工作原理

mongos 是分片集群的**路由层**，它本身不存储数据，只负责请求路由。

```text
mongos 路由流程：

客户端请求：db.orders.find({ customer_id: "C001" })
                │
                ▼
        ┌───────────────┐
        │    mongos     │
        │               │
        │ 1. 解析查询     │
        │ 2. 查询配置服务器│
        │ 3. 确定目标分片 │
        │ 4. 转发请求     │
        │ 5. 合并返回结果 │
        └───────────────┘
                │
        ┌───────┴───────┐
        ▼               ▼
┌───────────────┐ ┌───────────────┐
│   Shard1      │ │   Shard2      │
│ (C001 在这里) │ │               │
│  返回匹配结果  │ │  返回空结果    │
└───────────────┘ └───────────────┘
        │               │
        └───────┬───────┘
                ▼
        ┌───────────────┐
        │    mongos     │
        │ 合并多个分片结果│
        │ 返回给客户端   │
        └───────────────┘
```

### mongos 配置

```javascript
// 启动 mongos
// mongos --configdb "configReplSet/host1:27017,host2:27017,host3:27017"

// 或者配置文件
// sharding:
//   configDB: "configReplSet/host1:27017,host2:27017,host3:27017"

// 常用配置
// mongos.conf
sharding:
  configDB: "configReplSet/config1:27017,config2:27017,config3:27017"
net:
  port: 27017
  bindIp: 0.0.0.0
systemLog:
  destination: file
  path: /var/log/mongodb/mongos.log
```

### 请求路由类型

```javascript
// 1. 目标型查询（Scatter-Gather，部分广播）
// 查询没有分片键，只能广播到所有分片
db.orders.find({ status: "pending" })  // 无分片键
// mongos 广播到所有 Shard，返回后合并

// 2. 单分片查询（Targeted Query）
// 查询包含分片键，直接路由到对应分片
db.orders.find({ customer_id: "C001" })  // 包含分片键
// mongos 直接路由到包含该数据的 Shard

// 3. 多分片查询（Multi-Shard Query）
// 查询包含分片键范围，可能命中多个分片
db.orders.find({ customer_id: { $gte: "C001", $lte: "C010" } })
// mongos 路由到涉及的多个 Shard
```

## Config Server

### 配置服务器存储的内容

Config Server 存储分片集群的**所有元数据**：

```javascript
// 查看配置服务器内容
// 连接到 config 服务器（需要特殊权限）
db = db.getSiblingDB("config")

// 主要配置集合
// 1. shards：分片信息
db.shards.find()
// { _id: "shard0001", host: "shardReplSet1/host1:27017" }

// 2. databases：数据库信息
db.databases.find()
// { _id: "shop", primary: "shard0001", partitioned: true }

// 3. collections：集合分片配置
db.collections.find()
// { _id: "shop.orders", lastmodEpoch: ..., key: { customer_id: "hashed" } }

// 4. chunks：chunk 分布信息
db.chunks.find().sort({ ns: 1, min: 1 })
// { _id: "shop.orders-...", ns: "shop.orders", min: { customer_id: MinKey }, max: { customer_id: MaxKey }, shard: "shard0001" }
```

### Config Server 必须是副本集

**MongoDB 3.4+**：Config Server 必须是副本集。

```javascript
// 初始化 Config Server 副本集
rs.initiate({
  _id: "configReplSet",
  configsvr: true,  // 标记为 config server
  members: [
    { _id: 0, host: "config1:27017" },
    { _id: 1, host: "config2:27017" },
    { _id: 2, host: "config3:27017" }
  ]
});
```

## Chunk 与数据分布

### 什么是 Chunk

Chunk 是 MongoDB 数据分片的基本单元，每个 chunk 包含分片键值在一定范围内的文档。

```text
Chunk 分布示意：

Shard1:
┌─────────────────────────────┐
│ Chunk: [MinKey, 100)        │
│ 文档数：10000                │
│ 大小：约 64MB                │
└─────────────────────────────┘

Shard2:
┌─────────────────────────────┐
│ Chunk: [100, 200)            │
│ 文档数：12000                │
│ 大小：约 64MB                │
└─────────────────────────────┘

Shard3:
┌─────────────────────────────┐
│ Chunk: [200, MaxKey)         │
│ 文档数：11000                │
│ 大小：约 64MB                │
└─────────────────────────────┘

默认 chunk 大小：64MB
```

### Chunk 分裂

```javascript
// 当 chunk 超过大小时（默认 64MB），会自动分裂
// 分裂点由分片键值决定

// 初始状态：一个大 chunk
// Chunk: [MinKey, MaxKey]

// 分裂后：两个较小的 chunk
// Chunk1: [MinKey, 100)
// Chunk2: [100, MaxKey]

// 查看 chunk 数量
db.getSiblingDB("config").chunks.countDocuments({ ns: "shop.orders" })

// 查看 chunk 分布
db.getSiblingDB("config").chunks.aggregate([
  { $match: { ns: "shop.orders" } },
  { $group: { _id: "$shard", count: { $sum: 1 } } }
])
```

### Chunk 迁移

```javascript
// 手动迁移 chunk
// 将某个 chunk 迁移到指定分片
db.adminCommand({
  moveChunk: "shop.orders",
  find: { customer_id: "C001" },  // 任意属于该 chunk 的文档
  to: "shard0002"
});

// 迁移整个集合到指定分片
db.adminCommand({
  moveCollection: "shop.orders",
  to: "shard0002"
});

// 查看迁移状态
db.adminCommand({ chunkshardingstatus: 1 });
```

## 分片键与路由

### 分片键的作用

分片键（Shard Key）决定了数据的分布方式，是分片集合的核心配置。

```javascript
// 创建分片集合时指定分片键
sh.shardCollection("shop.orders", { customer_id: 1 })
// 基于 customer_id 范围分片

sh.shardCollection("shop.orders", { customer_id: "hashed" })
// 基于 customer_id 哈希分片

// 查看集合的分片键
db.getSiblingDB("config").collections.findOne({ _id: "shop.orders" })
// { ..., key: { customer_id: "hashed" } }
```

### 分片键选择原则

```javascript
// 好的分片键特点：
// 1. 高基数：分片键的不同值要多
// 2. 分布均匀：值分布要均匀
// 3. 查询局部化：常用查询条件包含分片键

// 好的分片键示例
{ user_id: "hashed" }      // 高基数、均匀分布
{ account_id: 1 }           // 唯一、查询局部化

// 差的分片键示例
{ status: 1 }               // 基数低（只有几个状态）
{ created_at: 1 }           // 可能导致热点分片
{ gender: 1 }                // 基数极低（只有两个值）
```

### 分片键限制

**重要**：分片键一旦确定，几乎无法修改。

```javascript
// 分片键限制：
// 1. 不能超过 512 字节
// 2. 不能是数组（多键分片键除外）
// 3. 不能是文档类型
// 4. 创建后很难修改

// 无法对已有集合修改分片键
// 唯一办法：重新创建集合，迁移数据
```

## 分片集群查询

### 查询路由策略

```javascript
// 1. 基于分片键的精确查询（最高效）
// 直接路由到单个分片
db.orders.find({ customer_id: "C001" })

// 2. 基于分片键范围查询
// 路由到可能包含数据的多个分片
db.orders.find({ customer_id: { $gte: "C001", $lte: "C010" } })

// 3. 不包含分片键的查询
// 广播到所有分片（Scatter-Gather）
db.orders.find({ status: "pending" })

// 4. 聚合管道
// 可以使用 $match 优化，自动下推到分片执行
db.orders.aggregate([
  { $match: { customer_id: "C001" } },  // 下推到分片
  { $group: { _id: "$status", total: { $sum: "$amount" } } }  // 在 mongos 合并
])
```

### 排序与分页

```javascript
// 分片集群的排序
// 1. 如果排序字段有索引，且查询路由到单个分片 -> 高效
db.orders.find({ customer_id: "C001" }).sort({ created_at: -1 })

// 2. 广播到多个分片 -> mongos 需要合并排序，可能慢
db.orders.find({}).sort({ created_at: -1 })

// 分片集群的分页
// 错误方式：skip 过大时性能差
db.orders.find({ customer_id: "C001" }).skip(100000).limit(10)

// 正确方式：基于分片键游标分页
lastDoc = db.orders.findOne({ customer_id: "C001" })
db.orders.find({ customer_id: "C001", _id: { $gt: lastDoc._id } }).limit(10)
```

## 分片集群高可用

### 组件故障处理

```javascript
// 1. mongos 故障
// 客户端连接其他 mongos，mongos 无状态，可以轻松扩展
// -> 影响：连接中断，可快速恢复

// 2. Shard 故障
// 如果 Shard 是副本集，主节点故障自动选举新主
// 如果 Shard 整个宕机，该分片数据不可用
// -> 影响：该分片的数据读写失败

// 3. Config Server 故障
// Config Server 副本集主节点故障自动选举
// -> 影响：元数据查询暂缓，但数据读写不受影响

// 查看分片集群状态
sh.status()
db.adminCommand({ listShards: 1 })
```

### Balancer 与数据均衡

```javascript
// Balancer 负责自动均衡 chunk 分布
// 查看 Balancer 状态
sh.getBalancerState()        // 是否开启
sh.isBalancerRunning()       // 是否正在均衡
sh.getBalancerWindow()      // 均衡时间窗口

// 手动触发均衡
sh.enableBalancing("shop.orders")   // 开启均衡
sh.disableBalancing("shop.orders")  // 关闭均衡

// 等待均衡完成
sh.waitForBalance("shop.orders")
```

## 面试实战

**面试官问**：「mongos 是怎么知道数据在哪的？」

**深入回答**：

> mongos 通过 Config Server 获取元数据：
>
> 1. **启动时**：mongos 从 Config Server 加载所有分片和集合的元数据
>
> 2. **查询时**：mongos 根据查询条件，计算应该访问哪些 chunk
>
> 3. **路由时**：mongos 直接将请求转发到对应分片
>
> 4. **元数据缓存**：mongos 会缓存元数据，减少对 Config Server 的访问

**面试官问**：「分片集群和副本集的区别是什么？什么时候用哪个？」

**深入回答**：

> 区别：
>
> | 特性 | 分片集群 | 副本集 |
> |---|---|---|
> | 主要目标 | 水平扩展 | 高可用 |
> | 数据分布 | 多分片 | 副本复制 |
> | 复杂度 | 高 | 低 |
> | 数据量 | TB+ | 小于 1TB |
>
> 选择原则：
>
> - 单机/副本集：数据量小（小于 1TB），主要需求是高可用
> - 分片集群：数据量大，需要水平扩展

**面试官问**：「如何选择分片键？」

**深入回答**：

> 分片键选择原则：
>
> 1. **高基数**：分片键的值要足够多，如 user_id、order_id
>
> 2. **分布均匀**：避免数据倾斜，如不要用时间字段（热点）
>
> 3. **查询局部化**：常用查询包含分片键，可以单分片查询
>
> 4. **避免单调递增/递减**：会导致新数据都写入同一个分片
>
> 常见选择：`user_id: "hashed"` 或业务关键字段

## 总结

分片集群核心要点：

- **组件**：mongos（路由）、Config Server（元数据）、Shard（数据）
- **数据单元**：Chunk，默认 64MB
- **分片策略**：范围分片、哈希分片
- **路由机制**：基于分片键的精确路由 vs 广播查询
- **元数据**：存储在 Config Server
- **分片键**：决定数据分布，选择要慎重

分片集群是 MongoDB 处理海量数据的关键，理解其架构对于大规模部署至关重要。
