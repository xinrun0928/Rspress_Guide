# MongoDB 分片策略

MongoDB 支持两种主要的分片策略：范围分片和哈希分片。选择合适的分片策略对数据分布和查询性能至关重要。

## 范围分片

### 工作原理

范围分片根据分片键的值范围将数据划分到不同的 chunk。

```javascript
// 基于年龄的范围分片
sh.shardCollection("shop.customers", { age: 1 })

// 数据分布示意：
// Shard1: age ∈ (-∞, 20)
// Shard2: age ∈ [20, 40)
// Shard3: age ∈ [40, ∞)
```

### 范围分片特点

| 特点 | 说明 |
|---|---|
| 优势 | 范围查询效率高，相邻值可能在同一分片 |
| 劣势 | 可能导致数据倾斜（热点分片） |
| 适用场景 | 分片键值分布均匀，范围查询多 |

### 示例

```javascript
// 创建范围分片
sh.shardCollection("shop.sales", { region: 1, date: 1 })

// 分片键值的连续性影响查询效率
// ✅ 高效：查询落入单个或少数分片
db.sales.find({ region: "华北", date: { $gte: ISODate("2024-01-01") } })

// ❌ 低效：查询覆盖大部分分片
db.sales.find({ region: { $gte: "A" } })
```

## 哈希分片

### 工作原理

哈希分片计算分片键的哈希值，根据哈希值范围将数据分布到不同的 chunk。

```javascript
// 基于 user_id 的哈希分片
sh.shardCollection("shop.orders", { user_id: "hashed" })

// 数据分布示意：
// user_id="U001" → hash = 12345 → Shard2
// user_id="U002" → hash = 67890 → Shard1
// user_id="U003" → hash = 24680 → Shard3
```

### 哈希分片特点

| 特点 | 说明 |
|---|---|
| 优势 | 数据分布均匀，适合高写入场景 |
| 劣势 | 范围查询需要广播到所有分片 |
| 适用场景 | 写入压力大，分片键无明显范围查询 |

### 示例

```javascript
// 创建哈希分片
sh.shardCollection("shop.events", { event_id: "hashed" })

// ✅ 高效：精确查询
db.events.findOne({ event_id: "E12345" })

// ❌ 低效：范围查询
db.events.find({ event_id: { $gte: "E10000", $lte: "E20000" } })
```

## 分片策略对比

| 特性 | 范围分片 | 哈希分片 |
|---|---|---|
| 数据分布 | 相邻值在相邻分片 | 随机均匀分布 |
| 范围查询 | 高效 | 低效（需广播） |
| 写入分布 | 可能热点 | 均匀 |
| 分片键选择 | 需要均匀分布的值 | 高基数字段 |

## 复合分片键

### 概念

复合分片键由多个字段组成，可以结合两种策略的优点。

```javascript
// 复合分片键：范围 + 哈希
sh.shardCollection("shop.orders", { customer_id: 1, order_id: "hashed" })

// 数据分布：
// 相同 customer_id 的订单在一组 chunk
// 但订单分布在多个 chunk（通过 order_id 哈希分散）
```

### 复合分片键设计原则

```javascript
// 设计原则：

// 1. 第一个字段应该是高基数的
// ✅ 好的设计
{ user_id: 1, timestamp: 1 }      // user_id 高基数

// ❌ 差的设计
{ status: 1, user_id: "hashed" }  // status 基数低

// 2. 常用查询条件应包含前缀字段
// 查询：{ user_id: "U001", date: "2024-01-01" }
// 分片键：{ user_id: 1, date: 1 }  // ✅ 匹配
// 分片键：{ date: 1, user_id: 1 }  // ❌ 不匹配
```

## 选择分片策略

### 选择建议

| 场景 | 推荐策略 |
|---|---|
| 写入压力大 | 哈希分片 |
| 范围查询多 | 范围分片 |
| 分片键值单调递增/递减 | 哈希分片 |
| 分片键值分布均匀 | 范围分片 |
| 查询包含分片键前缀 | 范围分片 |

### 决策流程

```
选择分片策略的决策树：

1. 查询模式是什么？
   │
   ├─ 主要是精确查询 → 哈希分片
   │
   └─ 主要是范围查询 → 继续判断
       │
       ├─ 分片键值分布均匀 → 范围分片
       │
       └─ 分片键值分布不均 → 考虑复合分片键或哈希分片
```

### 示例场景

```javascript
// 场景1：用户行为日志
// 写入量大，按用户查询
// → 哈希分片
sh.shardCollection("app.user_events", { user_id: "hashed" })

// 场景2：订单系统
// 按用户查订单，偶尔按时间范围查
// → 复合分片键
sh.shardCollection("app.orders", { customer_id: 1, created_at: 1 })

// 场景3：时序数据
// 按时间范围查询为主
// → 范围分片（注意数据倾斜）
sh.shardCollection("app.metrics", { timestamp: 1 })

// 场景4：会话存储
// 按会话 ID 精确查询
// → 哈希分片
sh.shardCollection("app.sessions", { session_id: "hashed" })
```

## 常见问题

### 热点分片问题

```javascript
// 问题：单调递增的分片键导致热点
// 示例：使用 ObjectId 作为分片键
sh.shardCollection("app.logs", { _id: 1 })  // ❌ ObjectId 单调递增

// 解决：使用哈希分片
sh.shardCollection("app.logs", { _id: "hashed" })

// 或者使用复合分片键
sh.shardCollection("app.logs", { log_type: 1, _id: 1 })
```

### 分片键基数不足

```javascript
// 问题：分片键值太少
// 示例：按性别分片
sh.shardCollection("app.users", { gender: 1 })  // ❌ 只有两个值

// 解决：使用高基数字段
sh.shardCollection("app.users", { user_id: "hashed" })
```

## 总结

分片策略选择要点：

- **范围分片**：适合范围查询，数据分布依赖分片键值
- **哈希分片**：适合高写入，数据均匀分布
- **复合分片键**：结合两者优点，注意字段顺序
- **选择原则**：根据查询模式和写入特点选择

正确选择分片策略是分片集群成功的关键。
