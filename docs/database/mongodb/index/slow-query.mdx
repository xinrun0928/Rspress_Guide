# MongoDB 慢查询分析与优化

「为什么同样的查询，有时候快，有时候慢？」

这是慢查询问题的典型表现。MongoDB 的查询性能受多种因素影响：索引是否生效、数据量变化、网络延迟、磁盘状态等。学会分析和优化慢查询，是 MongoDB 开发者的必备技能。

## 慢查询定位

### 开启慢查询日志

```javascript
// 查看当前慢查询阈值
db.getProfilingLevel()   // 0=关闭, 1=只记录慢查询, 2=记录所有
db.getProfilingStatus()

// 设置慢查询阈值为 100ms
db.setProfilingLevel(1, { slowms: 100 })

// 同时记录所有查询（用于调试）
db.setProfilingLevel(2)
```

### 查看慢查询日志

```javascript
// 查询 system.profile 集合
db.system.profile.find().limit(10).sort({ millis: -1 })

// 格式化输出
db.system.profile.find({
  millis: { $gt: 100 }
}).sort({ ts: -1 }).limit(5).pretty()
```

### 数据库命令方式

```bash
# 使用数据库命令查看慢查询
db.adminCommand({
  aggregate: 1,
  pipeline: [
    { $cursor: { query: {}, sort: { millis: -1 }, limit: 10 } }
  ]
})
```

## explain 分析查询计划

### executionStats 模式

```javascript
db.users.find({ age: { $gte: 25 }, city: "北京" })
  .sort({ created_at: -1 })
  .explain("executionStats")
```

### 关键指标解读

```javascript
{
  "executionStats": {
    "executionSuccess": true,
    "nReturned": 100,           // 返回的文档数
    "executionTimeMillis": 15,  // 执行时间（毫秒）
    "totalKeysExamined": 500,    // 索引扫描的键数
    "totalDocsExamined": 100,   // 文档扫描数
    "executionStages": [
      {
        "stage": "IXSCAN",      // 索引扫描
        "indexName": "age_1_city_1_created_at_-1",
        "direction": "forward"
      },
      {
        "stage": "FETCH",       // 回表获取
        "inputStage": "IXSCAN"
      }
    ]
  }
}
```

### 好与坏的查询

| 指标 | 好 | 差 |
|---|---|---|
| `stage` | `IXSCAN` | `COLLSCAN` |
| `totalKeysExamined` / `totalDocsExamined` | 接近 `nReturned` | 远大于 `nReturned` |
| `executionTimeMillis` | 越小越好 | 越大越差 |

:::tip 索引效率

`totalKeysExamined / nReturned` 的比值越小，索引效率越高。如果比值很大（如 1000），说明索引选择不当或查询条件不理想。
:::

## 常见慢查询原因

### 1. 全表扫描（COLLSCAN）

```javascript
// 问题：没有索引，全表扫描
db.users.find({ name: "张三" })  // name 字段没有索引

// 解决：创建索引
db.users.createIndex({ name: 1 })
```

### 2. 索引选择不当

```javascript
// 有复合索引 { age: 1, city: 1 }
// 但查询只用 city
db.users.find({ city: "北京" })  // 跳过 age，无法使用索引

// 解决：创建 city 单独索引或调整查询顺序
```

### 3. 排序无法使用索引

```javascript
// 有索引 { age: 1 }
// 但按 name 排序
db.users.find({ age: { $gte: 25 } }).sort({ name: 1 })

// 解决：创建 { age: 1, name: 1 } 复合索引
```

### 4. 正则表达式导致索引失效

```javascript
// 前缀通配符导致索引失效
db.users.find({ email: /.*@gmail.com/ })  // ❌ 全表扫描
db.users.find({ email: /^gmail/ })         // ✅ 可以使用索引

// 解决：避免正则表达式，或使用文本索引
```

### 5. skip 大量数据

```javascript
// 跳过前 10000 条记录，性能差
db.users.find().skip(10000).limit(10)

// 解决：使用游标分页（基于 _id 或索引字段）
const lastId = ...  // 上一页最后一条的 _id
db.users.find({ _id: { $gt: lastId } }).limit(10)
```

## 聚合管道优化

### $match 放在最前面

```javascript
// ❌ 错误：先排序再过滤
db.orders.aggregate([
  { $sort: { created_at: -1 } },
  { $match: { status: "completed" } }
])

// ✅ 正确：先过滤再排序
db.orders.aggregate([
  { $match: { status: "completed" } },
  { $sort: { created_at: -1 } }
])
```

### $limit 提前

```javascript
// ❌ 错误：先聚合全部数据再限制
db.orders.aggregate([
  { $group: { _id: "$user_id", total: { $sum: "$amount" } } },
  { $limit: 10 }
])

// ✅ 正确：先限制数据量再聚合
db.orders.aggregate([
  { $limit: 1000 },
  { $group: { _id: "$user_id", total: { $sum: "$amount" } } },
  { $limit: 10 }
])
```

### $project 减少数据量

```javascript
// 减少网络传输，只返回需要的字段
db.orders.aggregate([
  { $match: { user_id: ObjectId("...") } },
  { $project: { order_id: 1, amount: 1, created_at: 1, _id: 0 } }
])
```

## 实战案例

### 案例一：分页查询优化

```javascript
// 原始查询：跳过大量数据
db.articles.find().sort({ created_at: -1 }).skip(10000).limit(20)

// 优化：基于 _id 的游标分页
// 第一页
const firstPage = db.articles.find().sort({ _id: -1 }).limit(20)
// 最后一页的最后一条
const lastId = firstPage[firstPage.length - 1]._id
// 下一页
const nextPage = db.articles.find({ _id: { $lt: lastId } })
  .sort({ _id: -1 }).limit(20)
```

### 案例二：关联查询优化

```javascript
// 原始查询：$lookup 返回大量数据后再过滤
db.orders.aggregate([
  { $lookup: { from: "items", localField: "_id", foreignField: "order_id", as: "items" } },
  { $unwind: "$items" },
  { $match: { "items.status": "pending" } }
])

// 优化：在 $lookup 前先过滤
db.orders.aggregate([
  { $match: { status: "active" } },  // 先过滤
  { $lookup: {
      from: "items",
      let: { orderId: "$_id" },
      pipeline: [
        { $match: { $expr: { $eq: ["$order_id", "$$orderId"] }, status: "pending" } }
      ],
      as: "items"
  }}
])
```

## Profiler 工具

### 开启 Profiler

```javascript
// 记录所有超过 100ms 的操作
db.setProfilingLevel(1, { slowms: 100 })

// 或配置文件方式
// mongod.conf:
// operationProfiling:
//   mode: slowOp
//   slowOpThresholdMs: 100
```

### 分析 Profiler 数据

```javascript
// 找出最慢的 10 个查询
db.system.profile.find()
  .sort({ millis: -1 })
  .limit(10)
  .project({ millis: 1, command: 1, docsExamined: 1, keysExamined: 1 })

// 统计慢查询模式
db.system.profile.aggregate([
  { $match: { millis: { $gt: 100 } } },
  { $group: { _id: "$command.find", count: { $sum: 1 }, avgTime: { $avg: "$millis" } } },
  { $sort: { count: -1 } }
])
```

## 性能监控工具

### mongostat

```bash
# 查看数据库状态
mongostat --host localhost --port 27017

# 输出说明
# insert/query/update/delete/getmore/command：每秒操作数
# dirty：WiredTiger cache 脏数据比例
# used：WiredTiger cache 使用率
# qwr/awr：读写队列
```

### mongotop

```bash
# 查看集合级别的读写时间
mongotop --host localhost --port 27017

# 指定刷新间隔
mongotop 5  # 每 5 秒刷新一次
```

## 总结

慢查询分析与优化要点：

- **定位慢查询**：开启 Profiler，查看 `system.profile`
- **分析查询计划**：`explain("executionStats")`，关注 `stage`、`keysExamined`、`docsExamined`
- **常见问题**：全表扫描、索引失效、大数据 skip
- **聚合优化**：匹配在前、限制提前、投影减少数据
- **持续监控**：使用 `mongostat`、`mongotop` 监控性能
