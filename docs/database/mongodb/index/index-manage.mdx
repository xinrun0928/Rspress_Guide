# MongoDB 索引管理

索引是性能优化的核心。创建索引只是第一步，持续监控和维护同样重要。

## 查看索引

### 基本操作

```javascript
// 查看集合的所有索引
db.users.getIndexes()

// 返回示例
[
  { "v": 2, "key": { "_id": 1 }, "name": "_id_" },
  { "v": 2, "key": { "age": 1 }, "name": "age_1" },
  { "v": 2, "key": { "age": 1, "city": -1 }, "name": "age_1_city_-1" }
]
```

### 索引统计

```javascript
// 查看索引使用统计
db.users.aggregate([
  { $indexStats: {} }
])

// 返回每个索引的访问次数、扫描次数等
// {
//   "name": "age_1",
//   "accesses": { "ops": NumberLong(12345), "since": ISODate("...") }
// }
```

## 创建索引

### 基本语法

```javascript
// 单字段索引
db.collection.createIndex({ field: 1 })   // 升序
db.collection.createIndex({ field: -1 })  // 降序

// 复合索引
db.collection.createIndex({ field1: 1, field2: -1 })

// 指定索引名称
db.collection.createIndex(
  { email: 1 },
  { name: "idx_email_unique" }
)

// 后台创建（不阻塞读写）
db.collection.createIndex(
  { large_field: 1 },
  { background: true }
)
```

:::tip 后台创建索引

在生产环境创建索引时，务必使用 `background: true` 选项，否则会阻塞所有读写操作。对于大集合，可能需要几个小时甚至更长时间。
:::

### 索引选项

| 选项 | 说明 |
|---|---|
| `background` | 后台创建，避免阻塞 |
| `unique` | 唯一索引 |
| `sparse` | 稀疏索引，忽略 null 值 |
| `expireAfterSeconds` | TTL 索引，自动过期删除 |
| `weights` | 文本索引权重 |
| `name` | 指定索引名称 |
| `partialFilterExpression` | 部分索引，只索引满足条件的文档 |

### 部分索引

部分索引只索引满足指定条件的文档，减少索引大小：

```javascript
// 只索引 status 为 "active" 的用户
db.users.createIndex(
  { email: 1 },
  { partialFilterExpression: { status: "active" } }
)

// 等价于：
// WHERE status = 'active' ON email
```

:::tip 部分索引的用途

- 减少索引大小
- 提高写入性能
- 为稀疏数据创建索引
- 只索引热数据

:::

## 删除索引

```javascript
// 按名称删除
db.users.dropIndex("age_1")

// 按字段删除
db.users.dropIndex({ age: 1 })

// 删除所有非 _id 索引
db.users.dropIndexes()

// 注意：_id 索引无法删除
```

## 索引分析与优化

### explain 分析查询计划

```javascript
// 分析查询是否使用索引
db.users.find({ age: { $gte: 25 } }).explain("executionStats")

// 返回结果分析：
// "stage": "IXSCAN" → 使用索引扫描（好）
// "stage": "COLLSCAN" → 全表扫描（差）
// "stage": "FETCH" → 回表获取数据
```

### 常见的执行阶段

| Stage | 说明 | 好/坏 |
|---|---|---|
| `COLLSCAN` | 全表扫描 | 差 |
| `IXSCAN` | 索引扫描 | 好 |
| `FETCH` | 回表获取文档 | 一般 |
| `SORT` | 内存排序 | 差 |
| `LIMIT` | 限制返回数量 | 好 |
| `PROJECTION` | 字段投影 | 好 |

### 索引覆盖判断

```javascript
db.users.find(
  { age: { $gte: 25 } },
  { name: 1, _id: 0 }
).explain("executionStats")

// "indexOnly": true → 索引覆盖，不需要回表
// "indexOnly": false → 需要回表
```

## 索引最佳实践

### 创建时机

- **新集合**：在数据量较小时创建索引，性能更好
- **查询优化**：先分析慢查询，再针对性地创建索引
- **避免过度索引**：每个索引都会增加写入开销

### 索引维护

```javascript
// 重建索引（清理碎片，可减小索引大小）
db.users.reIndex()

// 监控索引使用情况
// 定期检查 $indexStats，删除不使用的索引
```

### 复合索引设计原则

1. **等值查询字段在前**：被 `$eq` 查询的字段优先
2. **范围查询字段在后**：被 `$gt`、`$lt`、`$in` 查询的字段放后
3. **排序字段考虑**：如果需要排序，相关字段放在最后
4. **字段顺序影响索引选择**：优化器会根据查询选择索引

```javascript
// 查询模式
{ status: "active", age: { $gte: 25 }, created_at: -1 }

// 正确的索引顺序
{ status: 1, age: 1, created_at: -1 }
//       ↑       ↑      ↑
//    等值    范围    排序
```

## 常见问题

### 索引未生效

```javascript
// 检查 1：查询条件是否使用索引
db.users.find({ name: /正则/ }).explain()
// 正则可能导致索引失效

// 检查 2：字段类型是否匹配
// 如果索引字段是数字，查询条件是字符串，索引可能失效
db.users.find({ age: "25" })  // age 是数字，查询是字符串

// 检查 3：是否使用了投影排除索引字段
// 如果投影排除了索引字段，索引可能无法使用
```

### 索引影响写入性能

每个索引都会增加写入开销：

- 插入文档：同时更新所有索引
- 更新文档：如果更新的字段有索引，需要更新索引
- 删除文档：同时删除所有索引中的条目

:::warning 写入密集型场景

对于写入密集型场景，索引要谨慎创建：
- 只创建必要的索引
- 考虑使用部分索引
- 定期审查并删除不使用的索引
:::

## 总结

索引管理核心要点：

- **查看索引**：`getIndexes()`、`$indexStats`
- **创建索引**：`createIndex()`，生产环境使用 `background: true`
- **删除索引**：`dropIndex()`，`_id` 索引无法删除
- **分析查询**：`explain("executionStats")`，关注 `stage` 和 `indexOnly`
- **最佳实践**：等值在前、范围在后，避免过度索引
