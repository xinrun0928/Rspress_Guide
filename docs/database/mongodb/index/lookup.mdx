# MongoDB $lookup 与关系型 JOIN

「MongoDB $lookup 能不能替代 SQL 的 JOIN？」

这是很多从关系型数据库转型的开发者常问的问题。答案是：**能，但不是直接替代**。MongoDB 的文档模型和关系型数据库的表模型有本质区别，$lookup 的行为和 JOIN 也不完全相同。

本文深入对比 MongoDB $lookup 和关系型 JOIN，帮你理解它们的设计理念和使用差异。

## $lookup 基本概念

### 什么是 $lookup

`$lookup` 是 MongoDB 聚合管道中的一个阶段，用于从**另一个集合**中匹配文档，实现类似关系型数据库 JOIN 的功能。

```javascript
// 基本语法
{
  $lookup: {
    from: "other_collection",      // 关联的集合
    localField: "local_field",      // 本集合的字段
    foreignField: "foreign_field",  // 关联集合的字段
    as: "output_field"              // 输出字段名
  }
}
```

### 工作原理

```javascript
// 订单集合 orders
{ _id: 1, customer_id: 101, amount: 500 }

// 客户集合 customers
{ _id: 101, name: "张三", city: "北京" }

// 使用 $lookup
db.orders.aggregate([
  {
    $lookup: {
      from: "customers",
      localField: "customer_id",
      foreignField: "_id",
      as: "customer_info"
    }
  }
])

// 结果
{
  _id: 1,
  customer_id: 101,
  amount: 500,
  customer_info: [
    { _id: 101, name: "张三", city: "北京" }
  ]
}
```

## $lookup vs SQL JOIN

### 语法对比

| SQL JOIN | MongoDB $lookup |
|---|---|
| `SELECT * FROM orders JOIN customers ON orders.customer_id = customers.id` | `$lookup` + `$unwind` |
| `LEFT JOIN` | `$lookup`（默认是 LEFT JOIN） |
| `INNER JOIN` | `$lookup` + `$match` 过滤 |
| `CROSS JOIN` | `$lookup` with empty `localField` |

### LEFT JOIN 对比

**SQL 版本：**

```sql
SELECT o.id, o.amount, c.name, c.city
FROM orders o
LEFT JOIN customers c ON o.customer_id = c.id
WHERE o.status = 'completed'
```

**MongoDB 版本：**

```javascript
db.orders.aggregate([
  { $match: { status: "completed" } },
  {
    $lookup: {
      from: "customers",
      localField: "customer_id",
      foreignField: "_id",
      as: "customer_info"
    }
  },
  { $unwind: "$customer_info" },  // 转换为非嵌套格式
  {
    $project: {
      _id: 1,
      amount: 1,
      name: "$customer_info.name",
      city: "$customer_info.city"
    }
  }
])
```

### INNER JOIN 对比

**SQL 版本：**

```sql
SELECT o.id, o.amount, c.name
FROM orders o
INNER JOIN customers c ON o.customer_id = c.id
```

**MongoDB 版本：**

```javascript
db.orders.aggregate([
  {
    $lookup: {
      from: "customers",
      localField: "customer_id",
      foreignField: "_id",
      as: "customer_info"
    }
  },
  { $match: { customer_info: { $ne: [] } } },  // 过滤未匹配的
  { $unwind: "$customer_info" }
])
```

### 多表 JOIN 对比

**SQL 版本：**

```sql
SELECT o.id, c.name, p.name as product_name
FROM orders o
JOIN customers c ON o.customer_id = c.id
JOIN order_items oi ON o.id = oi.order_id
JOIN products p ON oi.product_id = p.id
```

**MongoDB 版本：**

```javascript
db.orders.aggregate([
  { $match: { status: "completed" } },

  // 第一个 $lookup：关联客户
  {
    $lookup: {
      from: "customers",
      localField: "customer_id",
      foreignField: "_id",
      as: "customer_info"
    }
  },
  { $unwind: "$customer_info" },

  // 第二个 $lookup：关联订单项
  {
    $lookup: {
      from: "order_items",
      localField: "_id",
      foreignField: "order_id",
      as: "items"
    }
  },
  { $unwind: "$items" },

  // 第三个 $lookup：关联产品
  {
    $lookup: {
      from: "products",
      localField: "items.product_id",
      foreignField: "_id",
      as: "product_info"
    }
  },
  { $unwind: "$product_info" },

  // 最终投影
  {
    $project: {
      order_id: "$_id",
      customer_name: "$customer_info.name",
      product_name: "$product_info.name"
    }
  }
])
```

## $lookup 的进阶用法

### 使用 pipeline（MongoDB 5.0+）

MongoDB 5.0 引入了 `$lookup` 的 pipeline 语法，可以进行更复杂的关联操作。

```javascript
// 关联客户的 VIP 订单
db.customers.aggregate([
  {
    $lookup: {
      from: "orders",
      let: { customer_id: "$_id" },
      pipeline: [
        {
          $match: {
            $expr: { $eq: ["$customer_id", "$$customer_id"] },
            status: { $in: ["completed", "shipped"] }
          }
        },
        { $sort: { created_at: -1 } },
        { $limit: 5 }  // 只取最近5笔订单
      ],
      as: "recent_orders"
    }
  }
])
```

### 关联多个字段

```javascript
// 使用 let 和 pipeline 关联多个字段
db.shipments.aggregate([
  {
    $lookup: {
      from: "inventories",
      let: {
        product_id: "$product_id",
        warehouse: "$warehouse_id"
      },
      pipeline: [
        {
          $match: {
            $expr: {
              $and: [
                { $eq: ["$product_id", "$$product_id"] },
                { $eq: ["$warehouse_id", "$$warehouse"] }
              ]
            }
          }
        }
      ],
      as: "inventory_info"
    }
  }
])
```

### 自关联

```javascript
// 评论关联回复（树形结构）
db.comments.aggregate([
  {
    $lookup: {
      from: "comments",
      localField: "_id",
      foreignField: "parent_id",
      as: "replies"
    }
  }
])
```

## $lookup 与 JOIN 的核心区别

### 1. 结果结构不同

| 特性 | SQL JOIN | MongoDB $lookup |
|---|---|---|
| 结果格式 | 扁平行 | 嵌套数组 |
| 重复数据 | 主表数据重复 | 每条文档包含关联数组 |
| 未匹配处理 | 取决于 JOIN 类型 | 返回空数组 |

```javascript
// MongoDB $lookup 结果示例
// 一条订单关联多条评论 → 评论以数组形式嵌入
{
  order_id: 1,
  amount: 500,
  reviews: [
    { rating: 5, comment: "很好" },
    { rating: 4, comment: "不错" }
  ]
}

// SQL JOIN 结果示例
// 订单和评论是扁平的多行
// order_id=1, amount=500, rating=5, comment="很好"
// order_id=1, amount=500, rating=4, comment="不错"
```

### 2. 性能模型不同

| 特性 | SQL JOIN | MongoDB $lookup |
|---|---|---|
| 索引利用 | 查询优化器自动选择 | 需要手动确保索引存在 |
| 执行计划 | 复杂优化 | 相对简单 |
| 分片执行 | 受限于 JOIN 条件 | 可以在各分片并行执行 |

```javascript
// MongoDB 中，确保 $lookup 使用索引
db.orders.createIndex({ customer_id: 1 })
db.customers.createIndex({ _id: 1 })

// 如果关联字段没有索引，$lookup 会做全表扫描
```

### 3. 分片集合的限制

**在分片集合上使用 `$lookup` 时，from 集合必须是未分片的（或者与本集合在同一分片）**。

```javascript
// orders 集合是分片的，customers 是未分片的
// $lookup 的 from 集合必须是未分片的（或者与本集合在同一分片）

// 错误的做法
db.sharded_orders.aggregate([
  {
    $lookup: {
      from: "sharded_customers",  // 不能关联到另一个分片集合
      ...
    }
  }
])

// 正确的做法
db.sharded_orders.aggregate([
  {
    $lookup: {
      from: "global_customers",  // 使用未分片的集合
      ...
    }
  }
])
```

### 4. 数据模型理念

SQL 和 MongoDB 的数据模型设计哲学不同：

| SQL | MongoDB |
|---|---|
| 规范化：避免数据冗余 | 反规范化：允许数据冗余 |
| JOIN 动态关联 | 预聚合/嵌入式存储 |
| ACID 事务 | 最终一致性优先 |

```javascript
// MongoDB 的设计：嵌入式文档
// 订单中嵌入客户信息（冗余，但查询快）
{
  _id: 1,
  customer: {
    id: 101,
    name: "张三",
    city: "北京"
  },
  items: [
    { product_id: 1, name: "商品A", price: 100 },
    { product_id: 2, name: "商品B", price: 200 }
  ]
}

// 关系型设计：分离表，通过 JOIN 关联
// orders, customers, order_items, products
```

## 最佳实践

### 何时使用 $lookup

```javascript
// 适合使用 $lookup 的场景：
// 1. 需要关联的集合数据经常变化（如价格、库存）
// 2. 关联的数据量不大
// 3. 需要动态条件过滤关联结果
db.orders.aggregate([
  {
    $lookup: {
      from: "products",
      localField: "product_id",
      foreignField: "_id",
      pipeline: [
        { $match: { active: true } }  // 只关联活跃产品
      ],
      as: "product_info"
    }
  }
])
```

### 何时使用嵌入式文档

```javascript
// 适合使用嵌入式的场景：
// 1. 数据不常变化（如用户地址）
// 2. 一对少数关系
// 3. 需要整体读取（不单独查询子数据）

// 用户文档嵌入地址（不常变化）
{
  _id: 1,
  name: "张三",
  address: {
    city: "北京",
    district: "朝阳区",
    street: "建国路"
  }
}

// 订单嵌入商品快照（需要记录下单时的商品信息）
{
  _id: 1,
  items: [
    {
      product_id: 1,
      product_name: "iPhone 15",  // 下单时的快照
      price: 6999,                 // 下单时的价格
      quantity: 1
    }
  ]
}
```

### $lookup 性能优化

```javascript
// 1. 确保关联字段有索引
db.orders.createIndex({ customer_id: 1 })
db.customers.createIndex({ _id: 1 })

// 2. 在 $lookup 前过滤数据
db.orders.aggregate([
  { $match: { status: "completed" } },  // 先过滤，减少数据量
  {
    $lookup: {
      from: "customers",
      localField: "customer_id",
      foreignField: "_id",
      as: "customer_info"
    }
  }
])

// 3. 使用 pipeline 限制关联数据量
db.orders.aggregate([
  {
    $lookup: {
      from: "order_items",
      let: { order_id: "$_id" },
      pipeline: [
        { $match: { $expr: { $eq: ["$order_id", "$$order_id"] } } },
        { $limit: 10 }  // 限制每笔订单最多关联10条
      ],
      as: "items"
    }
  }
])

// 4. 使用 $unwind 减少数据量后再关联
db.orders.aggregate([
  { $match: { customer_id: 101 } },
  { $unwind: "$items" },  // 先展开，减少数据
  {
    $lookup: {
      from: "products",
      localField: "items.product_id",
      foreignField: "_id",
      as: "product_info"
    }
  }
])
```

## 面试实战

**面试官问**：「MongoDB 的 $lookup 和 SQL JOIN 有什么区别？」

**深入回答**：

> 核心区别在于**数据模型**和**结果格式**：
>
> 1. **数据模型**：SQL 是规范化的表结构，通过 JOIN 动态关联；MongoDB 是文档模型，可以嵌入式存储或动态关联
>
> 2. **结果格式**：SQL JOIN 返回扁平的行；$lookup 返回嵌套的数组
>
> 3. **执行时机**：SQL JOIN 在查询时执行；MongoDB $lookup 在聚合管道中执行
>
> 4. **分片支持**：MongoDB 的 $lookup 在分片集合上有一些限制（from 集合需要是未分片的）

**面试官问**：「如何优化 MongoDB 的 $lookup 性能？」

**深入回答**：

> 优化 $lookup 性能的方法：
>
> 1. **确保索引存在**：关联的字段必须有索引
>
> 2. **提前过滤**：在 $lookup 前使用 $match 减少数据量
>
> 3. **使用 pipeline 限制**：在 $lookup 的 pipeline 中添加 $limit
>
> 4. **考虑嵌入式**：如果关联数据不常变化，考虑使用嵌入式存储
>
> 5. **避免链式 $lookup**：多层 $lookup 会导致性能下降，可以考虑预聚合

**面试官问**：「MongoDB 分片集合能不能用 $lookup 关联其他集合？」

**深入回答**：

> 可以，但有限制：
>
> 1. **$lookup 的 from 集合必须是未分片的**，或者与当前集合在同一分片
>
> 2. **如果需要在分片集合上做复杂关联**，可以考虑：
>    - 将关联表设为未分片（replica set）
>    - 使用应用层 JOIN
>    - 预计算关联数据并存储

## 总结

`$lookup` 是 MongoDB 实现跨集合关联的核心工具：

- **语法**：`from` 指定集合，`localField` 和 `foreignField` 指定关联字段
- **结果**：返回嵌套数组，而不是扁平行
- **性能**：需要确保索引存在，在 lookup 前过滤数据
- **分片**：$lookup 的 from 集合不能是分片集合
- **最佳实践**：根据数据特性和查询模式选择嵌入式或 $lookup

理解 $lookup 和 SQL JOIN 的差异，有助于设计更好的 MongoDB 数据模型。
