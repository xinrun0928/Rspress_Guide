# MongoDB 索引类型

「MongoDB 有哪些索引类型？它们底层是怎么实现的？」

这是面试中经常被问到的问题。大多数人只知道单字段索引和复合索引，但追问下去——「B+ 树在 MongoDB 中是怎么存储的？」「多键索引有什么限制？」「文本索引怎么实现分词？」「地理位置索引的原理是什么？」——能答上来的就很少了。

本文深入讲解 MongoDB 各种索引的底层实现，帮你彻底理解这个核心知识点。

## 索引基础回顾

### B+ 树结构

MongoDB 的默认索引存储引擎是 WiredTiger，使用 B+ 树作为索引结构。B+ 树的特点是：

- 所有数据都存储在叶子节点
- 叶子节点之间通过链表连接，支持范围查询
- 非叶子节点只存储索引，不存储数据

```
B+ 树结构示意：

                    [30]
                  /      \
            [10, 20]      [40, 50]
            /    \          /    \
      [1,5]    [15,18]  [35,38]  [45,48]
        ↓          ↓        ↓         ↓
      数据        数据      数据       数据
    (链表连接，支持范围遍历)
```

### 索引的创建与查看

```javascript
// 创建单字段索引
db.users.createIndex({ name: 1 })

// 创建复合索引
db.users.createIndex({ age: 1, city: -1 })

// 查看集合的所有索引
db.users.getIndexes()

// 查看索引统计信息
db.users.aggregate([
  { $indexStats: {} }
])

// 删除索引
db.users.dropIndex("index_name")
```

## 单字段索引

### 基本用法

单字段索引是最简单的索引类型，在文档的单个字段上创建：

```javascript
// 升序索引
db.users.createIndex({ age: 1 })

// 降序索引（MongoDB 可以任意方向遍历索引）
db.users.createIndex({ age: -1 })
```

### 索引方向的影响

MongoDB 支持双向遍历索引，因此索引方向主要影响：

1. **复合索引的匹配顺序**：复合索引 `{ a: 1, b: 1 }` 可以满足 `{ a: 1 }` 和 `{ a: 1, b: 1 }` 的查询，但**不能**满足 `{ b: 1 }` 的查询
2. **排序方向**：如果需要 `sort({ a: -1 })`，可以使用 `{ a: -1 }` 或 `{ a: 1 }` 索引

```javascript
// 复合索引 { a: 1, b: 1 } 的查询覆盖：
// ✅ a = 1
// ✅ a = 1 AND b = 1
// ✅ a = 1（排序 b: 1 或 b: -1 都可以）
// ❌ b = 1（无法使用索引）
```

## 复合索引

### 基本概念

复合索引在多个字段上创建，字段顺序至关重要。

```javascript
// 创建复合索引：先按 age 升序，再按 name 升序
db.users.createIndex({ age: 1, name: 1 })
```

### 索引前缀原则

复合索引 `{ age: 1, name: 1, city: 1 }` 可以满足以下查询：

| 查询条件 | 能否使用索引 | 说明 |
|---|---|---|
| `{ age: 25 }` | ✅ | 使用索引前缀 |
| `{ age: 25, name: "张三" }` | ✅ | 使用索引前缀 |
| `{ age: 25, name: "张三", city: "北京" }` | ✅ | 使用完整索引 |
| `{ name: "张三" }` | ❌ | 跳过前缀，无法使用 |
| `{ city: "北京" }` | ❌ | 跳过前缀，无法使用 |

### 索引选择原则

:::tip 索引字段顺序

字段顺序遵循**等值字段在前，范围字段在后**的原则：

```javascript
// 查询模式：WHERE age = 25 AND city IN ("北京", "上海") AND name LIKE "张%"
// 正确顺序：等值字段（age）→ IN 字段（city）→ 排序字段（name）
{ age: 1, city: 1, name: 1 }
```

:::

### 索引基数

索引字段的基数（不同值的数量）影响索引效率：

| 字段类型 | 基数 | 索引效果 |
|---|---|---|
| 性别（只有男/女） | 低 | 索引效果差，可能不走索引 |
| 年龄 | 中 | 一般 |
| 用户 ID / ObjectId | 高 | 索引效果好 |

## 多键索引

### 基本概念

当索引字段是数组时，MongoDB 会自动创建**多键索引**（Multikey Index）。数组的每个元素都会建立索引项。

```javascript
// 文档结构
{ name: "张三", tags: ["Java", "MongoDB", "Redis"] }

// tags 是数组，创建多键索引
db.users.createIndex({ tags: 1 })

// 会为数组的每个元素创建索引项：
// "Java" → 张三文档
// "MongoDB" → 张三文档
// "Redis" → 张三文档
```

### 查询多键索引

```javascript
// 查询 tags 包含 "MongoDB" 的用户
db.users.find({ tags: "MongoDB" })

// 查询 tags 同时包含多个元素（$all）
db.users.find({ tags: { $all: ["Java", "MongoDB"] } })

// 数组字段的索引会被自动创建，无需手动指定
```

### 多键索引的限制

<Badge text="重要" type="warning" /> 多键索引有以下限制：

1. **一个文档只能有一个多键索引路径**：不能同时有两个字段是数组
2. **索引边界复杂**：对数组字段的范围查询只能使用部分索引

```javascript
// ❌ 错误：一个集合不能有两个多键索引路径
db.example.createIndex({ tags1: 1, tags2: 1 })
// Error: cannot create index with a compound key of two multikey indexes

// ✅ 正确：一个多键索引 + 多个单键索引
db.example.createIndex({ tags1: 1 })
db.example.createIndex({ tags2: 1 })
```

## 文本索引

### 基本用法

文本索引用于全文搜索：

```javascript
// 创建文本索引
db.articles.createIndex({ title: "text", content: "text" })

// 文本搜索
db.articles.find({
  $text: { $search: "MongoDB 索引原理" }
})
```

### 文本搜索操作符

```javascript
// 搜索包含 "MongoDB" 或 "Redis" 的文档
db.articles.find({
  $text: { $search: "MongoDB Redis" }
})

// 搜索短语（用引号包裹）
db.articles.find({
  $text: { $search: "\"MongoDB 索引\"" }
})

// 排除包含某词的文档
db.articles.find({
  $text: { $search: "MongoDB -Redis" }  // 包含 MongoDB 但不包含 Redis
})
```

### 文本搜索的局限性

:::warning 文本索引的限制

1. **不支持中文分词**：MongoDB 文本索引按空格分词，中文需要配合分词器
2. **不支持复杂查询**：不能用 `AND`/`OR`，只能用 `$search` 的语法
3. **性能较低**：全文搜索不适合高并发场景
4. **权重不同**：可以为不同字段设置不同权重

```javascript
// 设置权重：title 权重 2，content 权重 1
db.articles.createIndex(
  { title: "text", content: "text" },
  { weights: { title: 2, content: 1 } }
)

// 搜索结果按相关性排序
db.articles.find(
  { $text: { $search: "MongoDB" } },
  { score: { $meta: "textScore" } }
).sort({ score: { $meta: "textScore" } })
```

:::

## 地理空间索引

### 2dsphere 索引

用于地球表面的地理坐标查询：

```javascript
// 文档结构
{
  name: "北京故宫",
  location: {
    type: "Point",
    coordinates: [116.3970, 39.9163]  // [经度, 纬度]
  }
}

// 创建 2dsphere 索引
db.places.createIndex({ location: "2dsphere" })

// 查询附近地点
db.places.find({
  location: {
    $nearSphere: {
      $geometry: {
        type: "Point",
        coordinates: [116.4, 39.9]
      },
      $maxDistance: 1000  // 1000 米
    }
  }
})
```

### 查询操作符

| 操作符 | 说明 |
|---|---|
| `$geoWithin` | 查询某形状内的文档 |
| `$nearSphere` | 查询附近 N 米的文档（需要排序） |
| `$near` | 查询附近的文档（非球面） |
| `$geoIntersects` | 查询与某形状相交的文档 |

```javascript
// 查询在某矩形范围内的地点
db.places.find({
  location: {
    $geoWithin: {
      $box: [
        [116.3, 39.8],   // 西南角
        [116.5, 40.0]    // 东北角
      ]
    }
  }
})

// 查询在某多边形内的地点
db.places.find({
  location: {
    $geoWithin: {
      $polygon: [
        [116.3, 39.8],
        [116.5, 39.8],
        [116.5, 40.0],
        [116.3, 39.8]
      ]
    }
  }
})
```

## TTL 索引

TTL（Time-To-Live）索引用于自动删除过期文档：

```javascript
// 创建 TTL 索引：createdAt 字段 + 3600 秒过期
db.logs.createIndex(
  { createdAt: 1 },
  { expireAfterSeconds: 3600 }  // 1小时后自动删除
)

// 文档会在 createdAt + 3600 秒后被删除
db.logs.insertOne({
  message: "日志信息",
  createdAt: new Date()
})
```

:::tip TTL 索引的用途

- 会话存储（Session）
- 临时数据（日志、缓存）
- 事件流数据

TTL 索引每秒检查一次删除，不是精确到秒的。
:::

## 唯一索引

唯一索引保证索引字段的值不重复：

```javascript
// 创建唯一索引
db.users.createIndex(
  { email: 1 },
  { unique: true }
)

// 复合唯一索引
db.orders.createIndex(
  { user_id: 1, order_id: 1 },
  { unique: true }
)

// 稀疏唯一索引（忽略 null 值）
db.users.createIndex(
  { phone: 1 },
  { unique: true, sparse: true }
)
```

### 唯一索引的行为

| 场景 | 行为 |
|---|---|
| 插入重复值 | 报错：`E11000 duplicate key error` |
| 多个 null 值 | null 被视为相同值，违反唯一约束 |
| 稀疏模式 | null 值不参与唯一检查 |

## 索引覆盖

### 什么是索引覆盖

当查询的所有字段都在索引中时，MongoDB 只需要读取索引，**不需要回表**查询数据：

```javascript
// 创建复合索引
db.users.createIndex({ age: 1, name: 1 })

// 查询投影只包含索引字段 → 索引覆盖
db.users.find(
  { age: { $gte: 25 } },    // 查询条件使用索引
  { name: 1, _id: 0 }        // 投影只包含索引字段
)
// 结果：不需要回表，直接从索引返回
```

### 索引覆盖的条件

1. 查询和投影都只包含索引字段
2. `_id` 字段不在查询或投影中（除非明确排除）
3. 复合索引的字段顺序与查询匹配

## 索引的底层实现

### WiredTiger B+ 树实现

WiredTiger 使用 B+ 树存储索引：

```
WiredTiger B+ 树结构：

┌─────────────────────────────────────────────┐
│                  根节点                       │
│           [key → page pointer]               │
└─────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────┐
│                  内部节点                     │
│  [key1 → page1] [key2 → page2] [key3 → page3] │
└─────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────┐
│                  叶子节点                     │
│  ┌────────────────────────────────────────┐ │
│  │ key: "MongoDB"                          │ │
│  │ value: [文档位置指针]                     │ │
│  └────────────────────────────────────────┘ │
│  ┌────────────────────────────────────────┐ │
│  │ key: "MySQL"                           │ │
│  │ value: [文档位置指针]                     │ │
│  └────────────────────────────────────────┘ │
│  ...（叶子节点链表连接）                        │
└─────────────────────────────────────────────┘
```

### 文档位置指针

索引叶子节点存储的是**文档位置指针**，不是完整文档。要获取完整文档，需要回表查询。

```javascript
// 查询 { name: "张三" } 时：
// 1. 在索引中查找 "张三" → 得到文档位置指针
// 2. 通过指针读取完整文档
```

### B+ 树的维护成本

| 操作 | B+ 树成本 |
|---|---|
| 插入 | O(log n)，可能触发页分裂 |
| 删除 | O(log n)，可能触发页合并 |
| 更新 | 如果键值不变，只需更新指针；如果键值改变，等同于删除+插入 |

## 面试实战

**面试官问**：「复合索引 `{ age: 1, name: 1 }` 能满足查询 `{ name: "张三" }` 吗？」

**深入回答**：

> 不能。复合索引遵循**最左前缀原则**，必须从第一个字段开始匹配。
>
> `{ age: 1, name: 1 }` 只能满足：
> - `{ age: 25 }`
> - `{ age: 25, name: "张三" }`
>
> 不能满足：
> - `{ name: "张三" }`
> - `{ name: "张三", age: 25 }`（顺序不对）

**面试官问**：「MongoDB 索引和 MySQL 索引有什么区别？」

**深入回答**：

> 核心区别在于**存储引擎**和**数据结构**：
>
> 1. **数据结构**：MySQL InnoDB 使用 B+ 树（所有数据在叶子节点），MongoDB WiredTiger 也使用 B+ 树，但叶子节点存指针
> 2. **文档模型**：MongoDB 的索引字段可以是数组（多键索引），MySQL 不支持
> 3. **写入性能**：MongoDB 的索引写入相对较慢（需要更新多个索引），MySQL 也有类似问题
> 4. **查询优化**：两者都有查询优化器，但实现不同

**面试官问**：「文本索引和正则表达式查询有什么区别？」

**深入回答**：

> 1. **性能**：文本索引经过优化，搜索速度快；正则表达式需要扫描字段，性能差
> 2. **分词**：文本索引支持分词；正则表达式是精确匹配
> 3. **中文支持**：两者都需要配合分词器，MongoDB 原生不支持中文
> 4. **适用场景**：文本索引用于全文搜索；正则表达式用于模式匹配

## 总结

MongoDB 索引类型一览：

| 索引类型 | 用途 | 注意事项 |
|---|---|---|
| 单字段索引 | 基础查询 | 最常用 |
| 复合索引 | 多字段查询 | 字段顺序重要，遵循最左前缀原则 |
| 多键索引 | 数组字段 | 一个集合只能有一个多键路径 |
| 文本索引 | 全文搜索 | 不支持中文，需要分词器 |
| 地理空间索引 | 地理位置查询 | 支持 Point/Polygon/LineString |
| TTL 索引 | 自动过期删除 | 后台线程每秒检查一次 |
| 唯一索引 | 唯一性约束 | 稀疏模式下忽略 null 值 |

理解索引类型和底层实现，是写出高性能 MongoDB 查询的基础。
