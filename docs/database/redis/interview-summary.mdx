# Redis 面试高频问题汇总

Redis 是 Java 后端面试的必考模块，几乎每场面试都会问到。本文档汇总了 Redis 相关的高频面试题，帮你系统性地准备面试。

## Redis 基础概念类

### Q1：Redis 是什么？有哪些特点？

Redis（Remote Dictionary Server）是一个开源的、基于内存的 Key-Value 数据结构存储系统，常被用作数据库、缓存和消息队列。

**核心特点：**

- **快**：基于内存操作，单机 QPS 可达 10 万以上
- **丰富的数据结构**：String、List、Hash、Set、ZSet，加上 BitMap、HyperLogLog、Geo 等高级类型
- **持久化支持**：支持 RDB 和 AOF 两种持久化方式
- **高可用**：支持主从复制、哨兵模式、集群模式
- **单线程模型**：命令执行是单线程的，避免了锁竞争

### Q2：Redis 的线程模型是怎样的？为什么单线程这么快？

Redis 6.0 之前，单线程指的是**网络 I/O 和命令执行是单线程的**。

**核心流程：**

1. 服务端使用 I/O 多路复用（select/poll/epoll）监听多个 socket 的可读事件
2. 当某个 socket 可读时，读取命令、解析命令、执行命令、返回结果

这个流程中，**命令的读取、解析、执行都是单线程顺序执行的**。

**为什么这么快？**

1. **纯内存操作**：访问延迟在纳秒级别
2. **单线程无锁**：避免了上下文切换和锁竞争开销
3. **I/O 多路复用**：一个线程处理大量并发连接
4. **高效的数据结构**：SDS、跳表、压缩列表等内部数据结构经过精心设计

Redis 6.0 引入了多线程 I/O，用于处理网络读写，但命令执行仍然是单线程的。

### Q3：Redis 和 Memcached 的区别？

| 特性 | Redis | Memcached |
| --- | --- | --- |
| 数据类型 | 丰富（5种基本 + 多种高级） | 只有 String |
| 持久化 | 支持 RDB、AOF | 不支持 |
| 集群 | 原生支持 Cluster | 需要客户端实现 |
| 内存管理 | 多种淘汰策略 | LRU |
| 单线程 vs 多线程 | 单线程执行（6.0+ 可配置多线程 I/O） | 多线程 |
| 性能 | 略低（但足够用） | 略高 |

**选择建议**：如果只需要简单的 K-V 缓存，选 Memcached；如果需要丰富的数据类型、持久化、集群等功能，选 Redis。

## Redis 数据结构类

### Q4：Redis 有哪些数据类型？底层数据结构是什么？

Redis 有 5 种基本数据类型和 3 种高级类型：

**5 种基本类型：**

| 类型 | 底层数据结构 | 典型场景 |
| --- | --- | --- |
| String | SDS（简单动态字符串） | 缓存、计数器、分布式锁 |
| List | QuickList（双向链表+压缩列表） | 消息队列、最新列表 |
| Hash | Dict / ziplist | 对象存储、表数据缓存 |
| Set | Dict / intset | 标签、好友关系、去重 |
| ZSet | 跳表+字典 | 排行榜、延迟队列 |

**3 种高级类型：**

| 类型 | 底层实现 | 典型场景 |
| --- | --- | --- |
| BitMap | String | 签到、用户在线统计 |
| HyperLogLog | String | UV 统计 |
| Geo | 有序集合 | LBS、附近的人 |

### Q5：SDS 和 C 字符串的区别？

SDS（Simple Dynamic String）是 Redis 实现的动态字符串，相比 C 字符串有以下优势：

| 特性 | C 字符串 | SDS |
| --- | --- | --- |
| 获取长度 | O(n) | O(1) |
| 缓冲区溢出 | 可能发生 | 不会发生 |
| 内存重分配 | 频繁 | 空间预分配、惰性释放 |
| 二进制安全 | 否（遇到 `\0` 结束） | 是 |

**空间预分配**：SDS 扩展时，会额外分配未使用的空间，减少内存重分配次数。

**惰性空间释放**：SDS 缩短时，不会立即回收多余空间，而是记录在 `free` 字段中。

### Q6：跳表是什么？为什么 ZSet 使用跳表？

跳表（Skip List）是一种基于链表的多层索引结构，支持 O(log n) 的查找、插入、删除。

```
跳表结构示例：
Level 2 ──────────────────────────▶ [node5] ──────────▶ null
Level 1 ────▶ [node2] ────▶ [node3] ────▶ [node5] ──────────▶ null
Level 0 ──▶ [node1] ──▶ [node2] ──▶ [node3] ──▶ [node4] ──▶ [node5] ──▶ null
```

**为什么 ZSet 使用跳表而不是红黑树？**

1. **实现更简单**：跳表比红黑树更容易实现和理解
2. **范围查询更友好**：跳表的层结构天然支持范围操作
3. **插入更快**：跳表的插入不需要旋转操作
4. **并发更友好**：跳表的各层操作可以更好地并发控制

ZSet 同时使用跳表和字典：跳表维护有序性，字典维护 O(1) 的成员查找。

## Redis 持久化类

### Q7：Redis 的持久化机制有哪些？

Redis 支持两种持久化方式：

**1. RDB（Redis Database）**

- 定时生成数据快照，保存为 `.rdb` 文件
- fork 子进程执行备份
- 恢复速度极快
- 可能丢失最后一次快照后的数据

**2. AOF（Append Only File）**

- 记录所有写命令，保存为 `.aof` 文件
- 三种刷盘策略：
  - `always`：每个命令都持久化（慢，但最安全）
  - `everysec`：每秒持久化一次（折中，推荐）
  - `no`：由操作系统控制刷盘（快，但可能丢失数据）

### Q8：RDB 和 AOF 如何选择？

实际使用中，**通常同时开启 RDB 和 AOF**：

- RDB 做冷备份，恢复速度快
- AOF 保证数据不丢失

当 AOF 和 RDB 同时启用时，Redis 优先使用 AOF 恢复（因为 AOF 通常数据更完整）。

### Q9：Redis 4.0 后的混合持久化是什么？

混合持久化结合了 RDB 和 AOF 的优点：

- AOF 重写时，先以 RDB 格式写入当前数据状态
- 然后继续以 AOF 命令追加后续操作
- 恢复时，先加载 RDB 部分，再执行后续 AOF 命令

这样既保证了恢复速度，又保证了数据完整性。

### Q10：fork 进程时会阻塞吗？

`fork()` 在 Linux 下采用 Copy-On-Write（写时复制）技术：

- fork 只复制页表，不复制实际内存数据
- 父进程 fork 本身是**非阻塞**的
- 但 fork 后的 COW 阶段，如果数据页被大量修改，可能导致内存压力

:::warning 警惕
fork 时的内存开销 = 页面数 × 页面大小，与进程占用内存成正比。如果 Redis 占用 10GB 内存，fork 可能触发 10GB 的 COW。
:::

## Redis 缓存策略类

### Q11：Redis 的过期策略和淘汰策略分别是什么？

**过期策略（主动删除过期 key）：**

- **惰性删除**：访问 key 时检查是否过期，过期则删除。简单但可能堆积过期 key
- **定期删除**：每隔一段时间，随机检查一部分 key，删除过期的

**淘汰策略（内存达到上限时的处理）：**

- `volatile-lru`：从设置过期时间的 key 中淘汰最近最少使用的
- `allkeys-lru`：从所有 key 中淘汰最近最少使用的
- `volatile-random`：从设置过期时间的 key 中随机淘汰
- `allkeys-random`：从所有 key 中随机淘汰
- `volatile-ttl`：从设置过期时间的 key 中淘汰 TTL 最小的
- `noeviction`：不淘汰，直接返回错误

:::tip 选择建议
- 热点数据缓存：使用 `allkeys-lru`
- 所有 key 等价：使用 `allkeys-random`
- 有明确冷热区分：使用 `volatile-lru`
:::

### Q12：缓存雪崩、缓存击穿、缓存穿透如何解决？

**缓存雪崩**：大量 key 同时过期，导致大量请求打到数据库。

解决方案：

- key 的过期时间加上随机值，避免同时过期
- 设置热点数据永不过期（需要配合更新逻辑）
- 使用互斥锁或队列，保证只有一个请求去重建缓存

**缓存击穿**：热点 key 过期瞬间，大量请求同时打到数据库。

解决方案：

- 互斥锁：只有一个请求去加载数据，其他等待
- 热点数据永不过期
- 提前预热热点数据

**缓存穿透**：查询一个不存在的数据，缓存没有、数据库也没有，每次都打到数据库。

解决方案：

- 布隆过滤器：过滤掉不存在的 key
- 缓存空值：将空结果也缓存起来，设置短过期时间
- 参数校验：做好参数校验，直接拦截非法请求

## Redis 集群与高可用类

### Q13：Redis 的主从复制原理？

主从复制是 Redis 高可用性的基础：

1. 从节点保存主节点信息（`master_host`、`master_port`）
2. 从节点定时向主节点发送 `PING`，保持连接
3. 主节点执行写命令后，通过 `REPLCONF ACK` 命令同步偏移量
4. 从节点记录主节点偏移量，断线后根据偏移量补发数据

**全量复制流程：**

1. 从节点发送 `PSYNC ? -1`
2. 主节点执行 `BGSAVE`，生成 RDB 文件
3. 主节点将 RDB 文件发送给从节点
4. 从节点加载 RDB 文件，完成全量同步

### Q14：主从同步时网络中断会怎样？

Redis 2.8 引入了**增量复制**（Partial Resync）机制：

- 主节点维护一个环形缓冲区（repl-backlog-buffer）
- 从节点记录自己的复制偏移量
- 断线重连后，从节点发送自己的偏移量
- 如果偏移量在缓冲区范围内，主节点发送增量数据
- 否则执行全量复制

### Q15：哨兵模式如何实现自动故障转移？

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   Sentinel  │     │   Sentinel  │     │   Sentinel  │     ┌─────────────┐
│     1       │◀───▶│     2       │◀───▶│     3       │────▶│   Primary   │
│             │     │             │     │             │     │             │
└─────────────┘     └─────────────┘     └─────────────┘     └──────┬──────┘
                                                                      │
      ┌─────────────────────────────────────────────────────────────┘
      │
      ▼
┌─────────────┐     ┌─────────────┐
│  Secondary  │◀────│  Secondary  │
│     1       │     │     2       │
└─────────────┘     └─────────────┘
```

**故障转移流程：**

1. 主观下线：Sentinel 发现 Primary 没有响应 `PING`
2. 客观下线：多个 Sentinel 都认为 Primary 下线
3. 选举：Sentinel 集群投票选出一个 leader
4. 故障转移：leader 从 Secondary 中选举新的 Primary
5. 通知：更新主从关系，通知客户端新地址

### Q16：哨兵模式有什么缺点？

- **写入能力无法扩展**：只有一个主节点能处理写入
- **切换期间有短暂不可用**：故障转移需要时间
- **一致性保证较弱**：可能丢失少量数据

### Q17：Redis Cluster 是如何工作的？

Redis Cluster 是官方提供的分布式解决方案：

```
┌─────────────────────────────────────────────────────────────┐
│                    Redis Cluster                            │
│                                                             │
│  槽位分配：                                                   │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │  0-5460   │  5461-10922  │  10923-16383                 │ │
│  │  Node 1   │   Node 2     │   Node 3                    │ │
│  │  (Master) │  (Master)    │  (Master)                   │ │
│  │  + Slave  │  + Slave     │  + Slave                    │ │
│  └─────────────────────────────────────────────────────────┘ │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

- **数据分片**：使用 16384 个槽位，均匀分布在多个节点
- **槽位计算**：`slot = CRC16(key) % 16384`
- **节点通信**：Gossip 协议传播集群信息
- **故障转移**：从节点自动晋升为主节点

### Q18：为什么是 16384 个槽位？

这是作者经过权衡后的选择：

1. **节点信息传播**：每个节点需要广播自己负责的槽位信息
2. **网络包大小**：16384 个槽位，每个槽位 2 字节，网络包约 32KB
3. **足够分散**：16384 个槽位足以让集群足够分散
4. **作者偏好**：作者在 GitHub issue 中解释过这个选择

:::warning 65535 为什么不合适？
Redis Cluster 使用固定的 2 字节（16bit）存储槽位信息。如果使用 65535 个槽位，每个节点广播的包会接近 128KB，在大规模集群中会造成网络问题。
:::

## Redis 缓存策略类

### Q19：如何设计一个缓存策略？

缓存策略的设计需要考虑多个维度：

**读操作缓存：**

```java
public User getUser(Long userId) {
    String key = "user:" + userId;
    
    // 1. 先查缓存
    User cached = redis.get(key);
    if (cached != null) {
        return cached;
    }
    
    // 2. 缓存没有，查数据库
    User user = db.query(userId);
    
    // 3. 写入缓存
    redis.setex(key, 3600, user);
    
    return user;
}
```

**写操作缓存：**

```java
public void updateUser(User user) {
    // 1. 先更新数据库
    db.update(user);
    
    // 2. 删除缓存（不是更新）
    redis.del("user:" + user.getId());
}
```

:::tip 删除而不是更新
写操作时，应该**删除缓存而不是更新缓存**，因为：
1. 更新缓存后，在并发场景下可能被数据库的值覆盖
2. 删除缓存的代价更小（即使不存在也没关系）
:::

### Q20：缓存和数据库的一致性如何保证？

**方案一：Cache Aside（旁路缓存）**

- 读：缓存命中返回，缓存未命中查库并写入缓存
- 写：先更新数据库，再删除缓存

问题：并发场景下，可能出现短暂的脏数据。

**方案二：延迟双删**

```java
public void updateUser(User user) {
    // 1. 先删除缓存
    redis.del("user:" + user.getId());
    
    // 2. 更新数据库
    db.update(user);
    
    // 3. 延迟一段时间后再删除缓存
    Thread.sleep(100); // 等待主从同步完成
    redis.del("user:" + user.getId());
}
```

**方案三：分布式锁**

使用分布式锁保证读写操作的串行化。

**方案四：设置较短的 TTL**

让缓存自动过期，下次访问时自动重建。

### Q21：如何实现缓存预热？

**方案一：系统启动时加载**

```java
@PostConstruct
public void warmUp() {
    List<User> hotUsers = userService.getHotUsers();
    for (User user : hotUsers) {

```

**方案二：定时任务预热**

```java
}
```

## Redis 分布式锁类

### Q22：如何用 Redis 实现分布式锁？

```java
public boolean tryLock(String lockKey, String lockValue, long expireTime) {
    // SET NX EX 原子操作
    return "OK".equals(
        redis.set(lockKey, lockValue, 
                  new SetParams().nx().ex(expireTime))
    );
}

public boolean releaseLock(String lockKey, String lockValue) {
    // Lua 脚本保证原子性
    String script = 
        "if redis.call('get', KEYS[1]) == ARGV[1] then " +
        "    return redis.call('del', KEYS[1]) " +
        "else return 0 end";
    
    return redis.eval(script, 
                     Collections.singletonList(lockKey),
                     Collections.singletonList(lockValue)) == 1;
}
```

:::warning 防止误删
释放锁时必须判断锁的持有者，因为锁可能已过期并被其他线程获取。
:::

### Q23：如何处理锁续期问题？

如果业务执行时间超过锁的过期时间，锁会被释放，业务还在执行会导致问题。

解决方案：**守护线程自动续期**

```
获得锁 ──▶ 启动 Watch Dog ──▶ 业务执行中...
                │
                ├── 每隔 10 秒检查
                │   ├── 锁还存在 → 续期 30 秒
                │   └── 锁不存在 → 退出
                │
                └── 业务完成 → 释放锁 → 停止 Watch Dog
```

### Q24：Redisson 是如何实现的？

Redisson 是 Java 中最流行的分布式锁实现：

```java
RLock lock = redisson.getLock("myLock");

try {
    // 尝试获取锁，等待 10 秒，锁定 30 秒
    lock.tryLock(10, 30, TimeUnit.SECONDS);
    
    // 业务逻辑
} finally {
    // 释放锁
    lock.unlock();
}
```

Redisson 的 Watch Dog 机制：

1. 获取锁后，如果未指定 TTL，默认使用 30 秒
2. 启动一个定时任务，每隔 10 秒检查锁状态
3. 如果锁仍被持有，延长 TTL 到 30 秒
4. 如果节点宕机，定时任务中断，锁在 30 秒后自动释放

### Q25：主从切换时的双删问题怎么解决？

分布式锁的经典问题：

1. 客户端 A 在主节点获取锁
2. 主节点将锁信息同步到从节点前宕机
3. 从节点晋升为主节点
4. 客户端 B 在新主节点获取锁成功
5. **同一把锁被两个客户端持有！**

**解决方案：RedLock**

RedLock 算法要求在 N 个独立的 Redis 节点上获取锁，超过 N/2+1 个节点成功才算获取锁成功。

```java
RLock lock1 = redisson1.getLock("myLock");
RLock lock2 = redisson2.getLock("myLock");
RLock lock3 = redisson3.getLock("myLock");

// 同时在 3 个节点获取锁
RLock[] locks = {lock1, lock2, lock3};
RedissonRedLock redLock = new RedissonRedLock(locks);

try {
    // 获取 3 个锁中的大部分（2 个）
    redLock.tryLock(10, 30, TimeUnit.SECONDS);
} finally {
    redLock.unlock();
}
```

:::warning RedLock 的争议
RedLock 的作者是 Redis 之父 antirez，但这个算法在社区存在争议。有人认为它过于复杂，有人质疑其安全性。如果对锁的要求极高，可以考虑使用 Zookeeper 实现分布式锁。
:::
