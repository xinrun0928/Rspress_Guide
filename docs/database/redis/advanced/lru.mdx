# LRU 算法实现：近似 LRU vs 精确 LRU

面试官问你：Redis 的 LRU 是怎么实现的？你可能会说「维护一个访问时间戳，淘汰最久未访问的」——但这只是表面。

Redis 的 LRU 实现远比这复杂，也远比这巧妙。

## LRU 的基本原理

LRU（Least Recently Used）：**最近最少使用**。

核心思想是：如果一个数据最近被访问过，那么它将来被访问的概率也更高。

```
LRU 缓存示例（容量 3）：

初始：[空]
访问 A -> [A]
访问 B -> [A, B]
访问 C -> [A, B, C]
访问 A -> [B, C, A]  # A 最近访问，移动到末尾
访问 D -> [C, A, D]  # B 最久未访问，淘汰
```

## 精确 LRU 的实现

### 双向链表 + HashMap

标准的 LRU 实现：

```python
class LRUCache:
    def __init__(self, capacity):
        self.capacity = capacity
        self.cache = {}  # key -> node
        # 双向链表：头部是最近访问，尾部是最久未访问
        self.head = Node()  # 虚拟头节点
        self.tail = Node()  # 虚拟尾节点
        self.head.next = self.tail
        self.tail.prev = self.head
    
    def get(self, key):
        if key in self.cache:
            node = self.cache[key]
            # 移动到头部
            self.move_to_head(node)
            return node.value
        return -1
    
    def put(self, key, value):
        if key in self.cache:
            node = self.cache[key]
            node.value = value
            self.move_to_head(node)
        else:
            node = Node(key, value)
            self.cache[key] = node
            self.add_to_head(node)
            
            if len(self.cache) > self.capacity:
                # 删除尾部节点
                removed = self.remove_tail()
                del self.cache[removed.key]
    
    def move_to_head(self, node):
        self.remove_node(node)
        self.add_to_head(node)
    
    def remove_node(self, node):
        node.prev.next = node.next
        node.next.prev = node.prev
    
    def add_to_head(self, node):
        node.next = self.head.next
        node.prev = self.head
        self.head.next.prev = node
        self.head.next = node
    
    def remove_tail(self):
        node = self.tail.prev
        self.remove_node(node)
        return node
```

**精确 LRU 的问题**：

1. 每个 key 需要维护双向链表指针
2. 每次访问需要移动节点
3. 在 Redis 中，每个 key 都这样维护，开销巨大

## Redis 的近似 LRU

### 为什么是近似

Redis 选择近似 LRU 是出于性能考虑：

```
精确 LRU 的问题：
- 需要维护每个 key 的访问时间戳
- 需要维护全局双向链表
- 淘汰时需要排序所有 key（O(n log n)）
- 内存开销大

近似 LRU 的优势：
- 无需维护链表
- 抽样淘汰（O(1)）
- 内存开销小
- 效果接近精确 LRU
```

### Redis LRU 实现

```c
// Redis 对象结构
typedef struct redisObject {
    unsigned type:4;        // 类型
    unsigned encoding:4;    // 编码
    unsigned lru:LRU_BITS; // LRU 时间戳（24 位）
    int refcount;          // 引用计数
    void *ptr;             // 指向实际数据
} robj;
```

```c
// LRU_BITS = 24 位，精度为秒
#define LRU_BITS 24
#define LRU_CLOCK_MAX ((1<<LRU_BITS)-1)
#define LRU_CLOCK_RESOLUTION 1000 /* 毫秒精度 */

// 获取当前 LRU 时间
static unsigned long getLRUClock(void) {
    return (mstime() / LRU_CLOCK_RESOLUTION) & LRU_CLOCK_MAX;
}
```

### 抽样算法

Redis 使用抽样来近似 LRU：

```c
// 淘汰流程
int freeMemoryIfNeeded(void) {
    // ... 其他逻辑
    
    // 抽样淘汰
    while (mem_freed < needed) {
        // 从抽样中选择 LRU 最大的 key
        for (i = 0; i < server.maxmemory_samples; i++) {
            // 随机选择一个 key
            key = getRandomKey();
            
            // 获取该 key 的 LRU 时间
            lru = getObjectLRU(key);
            
            // 如果 LRU 最大（最久未访问），记录
            if (bestkey == NULL || lru > best_lru) {
                bestkey = key;
                best_lru = lru;
            }
        }
        
        // 淘汰选中的 key
        deleteKey(bestkey);
        mem_freed += keySize(bestkey);
    }
}
```

### LRU 抽样效果

```
maxmemory-samples 参数影响：

samples=1（最差）：
  - 随机选择
  - 效果接近随机淘汰

samples=5（默认）：
  - 5 次随机抽样
  - 效果接近精确 LRU
  - 性能开销小

samples=10（精确）：
  - 10 次随机抽样
  - 效果更接近精确 LRU
  - 性能开销略大
```

```
效果对比：

数据分布：
key1: 访问 100 次，最后访问 10 秒前
key2: 访问 50 次，最后访问 5 秒前
key3: 访问 1 次，最后访问 1 秒前

精确 LRU 结果：
  淘汰 key1（最久未访问）

samples=1 结果（可能）：
  随机选中 key2 或 key3

samples=5 结果（可能）：
  选中 key1（概率最高）

samples=10 结果（更可能）：
  选中 key1（概率更高）
```

## LFU 的实现

### LFU vs LRU

LFU 记录访问频率，而不是访问时间：

```c
// LFU 实现
typedef struct redisObject {
    // ...
    unsigned lru:LRU_BITS;  // 当使用 LFU 时，存储访问频率
} robj;

// LFU 相关函数
uint8_tLFUDecrAndReturn(robj *o);
voidLFUIncr(redisClient *c, robj *o);

// 频率衰减
voidLFUReopenHeatTime(redisClient *c, robj *o);
```

### LFU 配置

```bash
# 计数器衰减时间（分钟）
# 频率会随时间衰减
lfu-decay-time 1

# lfu-decay-time = 1
# 表示：每 1 分钟，如果 key 没有被访问，频率减半

# lfu-log-factor
# 影响频率增长曲线
lfu-log-factor 10

# lfu-log-factor 越大，频率增长越慢
```

### LFU 频率衰减算法

```c
// 频率衰减
uint8_t LFUDecrAndReturn(robj *o) {
    // 获取上次衰减时间
    long last_decay_time = o->lru >> 8;
    
    // 当前时间
    long now = LFUDecrTimeNow();
    
    // 计算需要衰减的次数
    long idletime = now - last_decay_time;
    
    // 每次衰减时间
    long r = server.lfu_decay_time;
    if (r && idletime >= r) {
        // 衰减次数
        long num decays = idletime / r;
        
        // 频率减半
        for (int j = 0; j < numdecays; j++) {
            o->lru_counter =LFUDecrCounter(o->lru_counter);
        }
    }
    
    returnLFUGetCounter(o);
}
```

## 性能对比

### 不同策略的内存使用

```bash
# 精确 LRU（假设）
# 每个 key 需要：时间戳 + 链表指针
# 额外内存：约 32 字节/key

# 近似 LRU（Redis）
# 每个 key 只需：24 位时间戳
# 额外内存：约 3 字节/key

# 结论：
# 1 亿 key 时，精确 LRU 需要 3.2GB 额外内存
# 近似 LRU 只需 300MB 额外内存
```

### 不同策略的淘汰效率

```bash
# 场景：100 万 key，热点 key 占 20%

精确 LRU：
  - 热点命中率：95%+
  - 淘汰准确性：最高
  - 性能开销：高

近似 LRU（samples=5）：
  - 热点命中率：90%+
  - 淘汰准确性：接近精确
  - 性能开销：低

近似 LRU（samples=10）：
  - 热点命中率：93%+
  - 淘汰准确性：接近精确
  - 性能开销：中等

LFU：
  - 热点命中率：95%+
  - 淘汰准确性：高（适合稳定热点）
  - 问题：无法识别热点变化
```

## 实战建议

### 1. 选择合适的淘汰策略

```bash
# 访问热点明显 -> LRU
maxmemory-policy allkeys-lru

# 访问频率稳定 -> LFU
maxmemory-policy allkeys-lfu

# 访问均匀分布 -> Random
maxmemory-policy allkeys-random
```

### 2. 调整抽样数量

```bash
# 默认 5，数据量大时可增加
maxmemory-samples 10

# 场景：
# - maxmemory = 10GB, 1 亿 key -> samples = 10
# - maxmemory = 1GB, 1000 万 key -> samples = 5
```

### 3. 监控淘汰效果

