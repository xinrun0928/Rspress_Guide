# 慢查询日志

Redis 服务突然变慢了，你排查了一圈发现：某条复杂的命令执行了 5 秒。这就是慢查询。

Redis 提供了慢查询日志功能，帮助你发现哪些命令执行时间过长。

## 慢查询日志配置

### 配置参数

```bash
# 慢查询阈值（微秒）
# 执行时间超过这个值的命令会被记录
slowlog-log-slower-than 10000   # 10ms（微秒）

# 保留的慢查询日志数量
# 超过这个数量会删除最老的日志
slowlog-max-len 128             # 默认 128 条
```

### 配置建议

```bash
# 测试环境：记录所有命令
slowlog-log-slower-than 0

# 生产环境：只记录慢命令
slowlog-log-slower-than 10000  # 10ms

# 高性能场景：只记录特别慢的命令
slowlog-log-slower-than 50000  # 50ms
```

### 动态修改

```bash
# 修改阈值
redis-cli CONFIG SET slowlog-log-slower-than 10000

# 修改日志长度
redis-cli CONFIG SET slowlog-max-len 1000
```

## 慢查询分析

### 分析脚本

```python
#!/usr/bin/env python3
import redis
import time
from datetime import datetime

def analyze_slowlog(host='localhost', port=6379, count=100):
    r = redis.Redis(host=host, port=port)
    
    # 获取慢查询日志
    logs = r.slowlog_get(count)
    
    if not logs:
        print("没有慢查询日志")
        return
    
    print(f"共 {len(logs)} 条慢查询\n")
    
    # 统计
    commands = {}
    total_time = 0
    
    for log in logs:
        cmd = log['command'][0].decode()  # 命令名
        duration = log['duration']  # 执行时间（微秒）
        
        commands[cmd] = commands.get(cmd, 0) + 1
        total_time += duration
        
        # 打印超过 100ms 的查询
        if duration > 100000:
            print(f"[警告] {cmd} 执行 {duration/1000}ms")
            print(f"  参数: {log['command']}")
            print(f"  时间: {datetime.fromtimestamp(log['start_time'])}")
            print()
    
    # 统计报告
    print("=" * 50)
    print("统计报告")
    print("=" * 50)
    print(f"总查询数: {len(logs)}")
    print(f"总执行时间: {total_time/1000}ms")
    print(f"平均执行时间: {total_time/len(logs)/1000}ms")
    print()
    print("命令频率:")
    for cmd, freq in sorted(commands.items(), key=lambda x: x[1], reverse=True):
        print(f"  {cmd}: {freq} 次")
    
    # 找出最慢的命令
    slowest = max(logs, key=lambda x: x['duration'])
    print()
    print(f"最慢命令: {slowest['command'][0].decode()}")
    print(f"执行时间: {slowest['duration']/1000}ms")

if __name__ == '__main__':
    analyze_slowlog()
```

## 慢查询的原因

### 1. O(n) 命令阻塞

```bash
# 这些命令时间复杂度是 O(n) 或更高
KEYS *              # O(n)，全表扫描
SMEMBERS set        # O(n)，返回所有成员
LRANGE list 0 -1    # O(n)，返回所有元素
HGETALL hash        # O(n)，返回所有字段
FLUSHALL            # O(n)，删除所有数据
```

**解决方案**：使用 SCAN 代替 KEYS

```bash
# 不好：KEYS
redis-cli KEYS "user:*"

# 好：SCAN
redis-cli SCAN 0 MATCH "user:*" COUNT 100
```

### 2. 大 Key 操作

```bash
# 返回大 List 的所有元素
redis-cli LRANGE mylist 0 -1

# 返回大 Hash 的所有字段
redis-cli HGETALL myhash

# 删除大 Set
redis-cli DEL myset
```

**解决方案**：
- 使用 UNLINK 代替 DEL（异步删除）
- 分批处理大 Key
- 避免存储过大的数据结构

### 3. 全量返回

```bash
# 返回集合的全部成员
redis-cli SMEMBERS bigset

# 返回有序集合的全部成员
redis-cli ZRANGE bigzset 0 -1
```

**解决方案**：
- 使用分页命令
- 使用游标迭代

```bash
# 分页返回
redis-cli ZRANGE bigzset 0 99  # 返回前 100 个
redis-cli ZRANGE bigzset 100 199  # 返回下一个 100 个

# SCAN 遍历
redis-cli SSCAN bigset 0 COUNT 100
```

### 4. 未使用索引

```bash
# 没有建立索引的操作会变慢
redis-cli SORT mylist BY "weight" ALPHA
```

### 5. 主从同步延迟

```bash
# 读取从节点时可能读到过期数据或延迟数据
redis-cli -h slave.redis.com GET key
```

## 慢查询优化建议

### 1. 避免 O(n) 命令

```bash
# 不好
redis-cli KEYS "user:*"
redis-cli FLUSHALL

# 好
redis-cli SCAN 0 MATCH "user:*" COUNT 100
redis-cli DBSIZE  # 查看 key 数量
```

### 2. 使用合适的数据结构

```bash
# 不好：存储大文本
SET big:text "这是很长的文本内容..."

# 好：分块存储
SET big:text:chunk:0 "第一块"
SET big:text:chunk:1 "第二块"
SET big:text:chunk:2 "第三块"
```

### 3. 使用 Pipeline

```bash
# 不好：多次网络往返
redis-cli GET key1
redis-cli GET key2
redis-cli GET key3

# 好：一次网络往返
redis-cli MGET key1 key2 key3
```

### 4. 使用 Lua 脚本

```bash
# 原子操作，减少网络往返
EVAL "
  local v1 = redis.call('GET', KEYS[1])
  local v2 = redis.call('GET', KEYS[2])
  return {v1, v2}
" 2 key1 key2
```

## 最佳实践

### 1. 合理设置阈值

```bash
# 根据业务需求调整

```

### 2. 定期分析慢查询

```bash
# 定时分析脚本
0 */6 * * * python /opt/scripts/redis-slowlog-analysis.py >> /var/log/redis-slowlog.log
```

### 3. 建立索引（使用 Hash 或 Sorted Set）

```bash
# 如果经常按范围查询，使用 ZSET
ZADD user:score 100 "user:1001"
ZADD user:score 95 "user:1002"
ZADD user:score 90 "user:1003"

# 按分数范围查询
ZRANGEBYSCORE user:score 90 100
```

## 总结

慢查询日志是排查 Redis 性能问题的利器：

1. **配置阈值**：根据业务设置合理的 slowlog-log-slower-than
2. **定期分析**：找出频繁出现的慢查询
3. **优化命令**：避免 O(n) 操作，使用 Pipeline 和 Lua
4. **监控告警**：及时发现慢查询问题

记住：**预防胜于治疗，在代码层面避免慢查询才是根本**。
