# Redis 淘汰策略：noeviction ~ allkeys-lru

Redis 的内存是有限的。当数据量超过 `maxmemory` 限制时，Redis 必须决定：保留哪些数据，淘汰哪些数据？

这就是淘汰策略要解决的问题。

## 为什么需要淘汰策略

Redis 将数据存储在内存中，但内存是有限的：

```
内存使用超过 maxmemory 限制
         |
         v
   触发淘汰策略
         |
         v
   选择并删除一些 key
         |
         v
   继续接受新数据
```

**为什么不用磁盘扩展内存**？

- 磁盘速度远慢于内存
- Redis 的高性能依赖于纯内存操作
- 磁盘只能用于持久化，不能用于运行时存储

## 淘汰策略分类

Redis 提供了 8 种淘汰策略，分为两类：

| 策略 | 淘汰范围 | 淘汰依据 |
| --- | --- | --- |
| **noeviction** | 不淘汰 | 返回错误 |
| **volatile-lru** | 带过期时间的 key | LRU 算法 |
| **volatile-lfu** | 带过期时间的 key | LFU 算法 |
| **volatile-ttl** | 带过期时间的 key | TTL 最小的 |
| **volatile-random** | 带过期时间的 key | 随机 |
| **allkeys-lru** | 所有 key | LRU 算法 |
| **allkeys-lfu** | 所有 key | LFU 算法 |
| **allkeys-random** | 所有 key | 随机 |

```
淘汰策略分类：

noeviction（不淘汰）：
  - 不删除任何 key
  - 写入操作返回错误
  - 适合只读场景

volatile-*（带过期时间）：
  - 只淘汰设置了过期时间的 key
  - 如果没有 volatile-* key，可能淘汰失败
  - 适合：缓存和数据库双写的场景

allkeys-*（所有 key）：
  - 淘汰所有 key
  - 不管有没有过期时间
  - 适合：纯缓存场景
```

## LRU 算法

LRU（Least Recently Used）：**最近最少使用**。

```bash
# 开启 LRU 淘汰
maxmemory-policy allkeys-lru

# LRU 算法工作原理：
# - 每个 key 记录最后一次访问时间
# - 淘汰时，选择最久未访问的 key
# - 适合：访问频率差异大的场景
```

### LRU 实现原理

Redis 使用近似 LRU 算法，而不是精确 LRU：

```bash
# 精确 LRU vs 近似 LRU

# 精确 LRU：
# - 需要为每个 key 维护访问时间戳
# - 淘汰时排序所有 key
# - 时间复杂度 O(n)
# - 内存开销大

# 近似 LRU（Redis 使用）：
# - 抽样一部分 key（默认 5 个）
# - 选择最久未使用的
# - 时间复杂度 O(1)
# - 内存开销小
```

```c
// Redis LRU 实现（简化）
#define LRU_BITS 24
#define LRU_CLOCK_SIZE (1 << LRU_BITS)

unsigned long LRU_CLOCK(void) {
    // 获取当前时间戳（24 位，秒级）
    // 用于标记 key 的最近访问时间
}

// 淘汰时抽样
int freeMemoryIfNeeded(void) {
    // 抽样 maxmemory-samples 个 key（默认 5）
    // 选择 LRU 时间最小的 key 淘汰
}
```

### LRU 配置参数

```bash
# 抽样数量
# 越多越精确，但性能开销越大
maxmemory-samples 5

# 建议：
# - 高性能场景：3-5
# - 精确淘汰场景：10+
```

## LFU 算法

LFU（Least Frequently Used）：**最不经常使用**。

```bash
# 开启 LFU 淘汰
maxmemory-policy allkeys-lfu

# LFU 算法工作原理：
# - 每个 key 记录访问频率
# - 淘汰时，选择访问频率最低的 key
# - 适合：访问频率稳定的场景
```

### LFU vs LRU

| 特点 | LRU | LFU |
| --- | --- | --- |
| 淘汰依据 | 最近访问时间 | 访问频率 |
| 适用场景 | 热点数据偶尔访问 | 访问频率稳定 |
| 问题 | 可能误杀偶尔访问的热点 | 无法识别热点变化 |

```python
# LRU vs LFU 示例

# 场景：热点 key，偶尔被访问一次
key: "hot" -> 被访问 1000 次，1 分钟前访问过
key: "cold1" -> 被访问 1 次，10 秒前访问过
key: "cold2" -> 被访问 1 次，5 秒前访问过

# LRU 结果：
# 淘汰 "hot"（最久未访问）
# 问题：热点数据被误杀

# LFU 结果：
# 淘汰 "cold1" 或 "cold2"（访问频率最低）
# 正确保留热点数据
```

### LFU 配置参数

```bash
# LFU 计数器衰减时间（分钟）
# 频率会随时间衰减，避免"历史热点"一直占位
lfu-decay-time 1

# LFU 计数器最大值
lfu-log-factor 10

# lfu-log-factor 越大，计数器增长越慢
# lfu-decay-time 越大，衰减越慢
```

## TTL 策略

```bash
# volatile-ttl：淘汰 TTL 最小的 key
maxmemory-policy volatile-ttl

# 适合：缓存数据有过期时间的场景
# 优先淘汰即将过期的数据
```

## 淘汰流程

```python
# Redis 淘汰流程（简化）

def processCommand(client):
    # 1. 执行命令
    result = execute_command(client)
    
    # 2. 检查内存使用
    if used_memory > maxmemory:
        # 3. 触发淘汰
        while used_memory > maxmemory and can_evict():
            evicted_key = select_key_for_eviction()
            delete_key(evicted_key)
            used_memory -= memory_of(evicted_key)
    
    return result
```

## 如何选择淘汰策略

### 场景分析

**场景一：纯缓存，热点明显**

```bash
# 推荐：allkeys-lru
maxmemory 10gb
maxmemory-policy allkeys-lru
maxmemory-samples 5

# 原因：
# - 纯缓存场景，不需要保留过期数据
# - 热点数据访问频繁，lru 能正确保留
```

**场景二：缓存 + 数据库双写**

```bash
# 推荐：volatile-lru
maxmemory 10gb
maxmemory-policy volatile-lru
maxmemory-samples 5

# 原因：
# - 避免淘汰没有过期时间的 key
# - 数据库有持久化，缓存数据有过期时间
```

**场景三：访问频率稳定**

```bash
# 推荐：allkeys-lfu
maxmemory 10gb
maxmemory-policy allkeys-lfu
lfu-decay-time 5

# 原因：
# - 访问频率稳定的场景，lfu 比 lru 更准确
```

**场景四：所有数据同等重要**

```bash
# 推荐：allkeys-random
maxmemory 10gb
maxmemory-policy allkeys-random

# 原因：
# - 不区分热点，随机淘汰
# - 适用于访问分布均匀的场景
```

### 选择建议

```
淘汰策略选择流程：

1. 是否需要淘汰？
   |
   +-- 否 -> noeviction
   |
   +-- 是 -> 数据有区别吗？
               |
               +-- 否 -> allkeys-random
               |
               +-- 是 -> 有过期时间吗？
                           |
                           +-- 有 -> 有热点区分吗？
                                       |
                                       +-- 访问频率区分 -> volatile-lfu
                                       |
                                       +-- 最近访问区分 -> volatile-lru
                                       |
                                       +-- 无区分 -> volatile-ttl
                           |
                           +-- 无 -> 有热点区分吗？
                                       |
                                       +-- 访问频率区分 -> allkeys-lfu
                                       |
                                       +-- 最近访问区分 -> allkeys-lru
```

> INFO stats

# 查看内存状态

### 调优建议

```bash
# 1. 预留足够内存
# maxmemory 应该小于物理内存
maxmemory 8gb  # 物理内存 10GB

# 2. 合理设置抽样数量
# 数据量大时，适当增加
maxmemory-samples 10

# 3. 监控淘汰数量
# 如果淘汰数量持续很高，考虑扩容

# 4. 选择合适的策略
# 定期分析业务访问模式，调整策略
```
