# Redis 慢命令阻塞与处理

凌晨三点，你被告警叫醒：「Redis 响应时间 > 500ms」。

登录 Redis 查看，`INFO commandstats` 显示 `slowlog` 里有一堆 `SMEMBERS`、`KEYS *` 之类的命令。

「不就是查个数据吗，怎么会这么慢？」

Redis 是单线程的。一个慢命令会阻塞所有后续命令的执行。这就是 Redis 慢命令的可怕之处。

## 为什么 Redis 会因为一条命令变慢

Redis 6 之前，所有命令都在主线程执行。单线程的优势是没有锁竞争、没有上下文切换；但代价是：**任何一个耗时的操作，都会阻塞整个服务器**。

```
时间线：
T1: 客户端 A 发送 GET命令
T2: 客户端 B 发送 KEYS * 命令（假设有 1000 万 key）
T3: 客户端 C 发送 SET命令
    ↓
Redis 被 KEYS * 阻塞
    ↓
T4: GET命令才被处理（已经等了 2 秒）
T5: SET命令才被处理（已经等了 2.5 秒）
```

这就是「一个慢命令拖垮整个服务」的原理。

## 常见慢命令 TOP 10

### 1. KEYS pattern（最危险的命令）

```bash
# 匹配所有以 user: 开头的 key
KEYS user:*

# 危险操作：O(N)，N = 全库 key 数量
# 1000 万 key 可能要扫描 10 秒
```

**替代方案：SCAN**

```bash
# SCAN 是增量扫描，每次只返回一小部分
# 不会阻塞其他命令
SCAN 0 MATCH user:* COUNT 100

# 返回游标和匹配的 key
# 1) "139"  ← 下一个游标
# 2) ["user:100", "user:101", ...]
```

```python
# Python 中使用 SCAN
cursor = 0
while True:
    cursor, keys = redis.scan(cursor, match='user:*', count=100)
    process_keys(keys)
    if cursor == 0:
        break
```

### 2. SMEMBERS（集合全量读取）

```bash
# 获取集合所有成员
SMEMBERS user:1001:favorites

# 危险：一个大集合可能包含百万成员
# O(N)，N = 集合大小
```

**替代方案：SSCAN**

```bash
# SSCAN 返回集合成员的迭代器
SSCAN user:1001:favorites 0 COUNT 100
```

### 3. FLUSHALL / FLUSHDB（清空数据库）

```bash
# 危险：O(N)，N = 全库 key 数量
FLUSHALL
FLUSHDB
```

**替代方案：渐进式删除 + 非阻塞删除**

```python
# 非阻塞删除：用 SCAN 分批删除
def async_flush(pattern='*', count=100):
    cursor = 0
    while True:
        cursor, keys = redis.scan(cursor, match=pattern, count=count)
        if keys:
            redis.delete(*keys)
        if cursor == 0:
            break
```

### 4. 全量返回类命令

| 命令 | 问题 | 替代方案 |
|---|---|---|
| `GETALL` | Hash 字段全返回 | `HSCAN` |
| `LRANGE` | 列表全返回 | `LRANGE key 0 99` 限制长度 |
| `ZRANGE` | 有序集合全返回 | `ZRANGE key 0 99` 限制长度 |

### 5. 大 Key 和 Big Key

Big Key 是指单个 key 存储的数据过大：

| 类型 | 危险阈值 | 问题 |
|---|---|---|
| String | 超过 10MB | 阻塞网络传输 |
| List | 超过 10000 元素 | 阻塞 `LRANGE` |
| Set | 超过 10000 成员 | 阻塞 `SMEMBERS` |
| Hash | 超过 10000 字段 | 阻塞 `HGETALL` |

**发现 Big Key**

```bash
# Redis 5.0+ 自带 Big Key 检测
redis-cli --bigkeys

# 输出示例
# Scanning 10000000 keys ...
# -------
# Big string: product:descriptions:10001 (200 bytes)
# Big list: user:1001:actions (10234 elements)
# Big set: tag:electronics (5012 members)
```

```python
# 编程检测 Big Key
def find_big_keys(threshold=10240):
    cursor = 0
    while True:
        cursor, keys = redis.scan(cursor, count=1000)
        for key in keys:
            key_type = redis.type(key).decode()
            if key_type == 'string':
                size = redis.strlen(key)
            elif key_type == 'list':
                size = redis.llen(key)
            elif key_type == 'set':
                size = redis.scard(key)
            elif key_type == 'zset':
                size = redis.zcard(key)
            elif key_type == 'hash':
                size = redis.hlen(key)

            if size > threshold:
                print(f"BIG KEY: {key} ({key_type}: {size})")

        if cursor == 0:
            break
```

## Redis 慢查询配置

### 配置慢查询阈值

```bash
# redis.conf

# 超过 10ms 的命令记录到慢日志
slowlog-log-slower-than 10000

# 最多保留 1000 条慢日志
slowlog-max-len 1000
```

```python
# 查看慢日志
# O(N)，N = 慢日志长度
result = redis.slowlog_get(10)  # 获取最近 10 条

# 慢日志结构
# [
#   {
#       'id': 1234,
#       'start_time': 1699999999,
#       'duration': 12345,  # 微秒
#       'command': ['GET', 'product:10001']
#   },
#   ...
# ]
```

### 动态调整配置

```python
# 不重启 Redis 调整阈值
redis.config_set('slowlog-log-slower-than', 5000)  # 5ms
redis.config_set('slowlog-max-len', 500)
```

## 性能优化实操

### 优化一：命令拆分

```python
# 原始：一次性获取 1000 个商品信息
# 危险：pipeline 里有 1000 个 GET
pipe = redis.pipeline()
for product_id in product_ids:
    pipe.get(f'product:{product_id}')
results = pipe.execute()

# 优化：批量获取，但限制单次命令大小
def batch_get(keys, batch_size=500):
    results = []
    for i in range(0, len(keys), batch_size):
        batch = keys[i:i+batch_size]
        pipe = redis.pipeline()
        for key in batch:
            pipe.get(key)
        results.extend(pipe.execute())
    return results
```

### 优化二：用 Hash 代替多个 String

```python
# 原始：每个字段一个 key
redis.set('user:1001:name', 'Alice')
redis.set('user:1001:age', '25')
redis.set('user:1001:city', 'Beijing')

# 3 次网络往返

# 优化：Hash 存储
redis.hset('user:1001', 'name', 'Alice')
redis.hset('user:1001', 'age', '25')
redis.hset('user:1001', 'city', 'Beijing')

# 1 次网络往返
user = redis.hgetall('user:1001')
```

### 优化三：Pipeline 批量操作

```python
# 原始：逐条执行
for item in items:
    redis.zadd('leaderboard', {item['user_id']: item['score']})

# 优化：Pipeline 批量
pipe = redis.pipeline()
for item in items:
    pipe.zadd('leaderboard', {item['user_id']: item['score']})
pipe.execute()

# 性能对比：1000 条数据
# 逐条：1000 * 1ms = 1000ms
# Pipeline：1 * 100ms = 100ms
```

### 优化四：使用 Lua 脚本原子操作

```lua
-- check_and_increment.lua
-- 原子性地检查并递增库存
local key = KEYS[1]
local current = tonumber(redis.call('GET', key) or '0')
local limit = tonumber(ARGV[1])

if current < limit then
    redis.call('INCR', key)
    return current + 1
else
    return -1  -- 库存不足
end
```

```python
# 执行 Lua 脚本
script = redis.register_script('check_and_increment.lua')
result = script(keys=['stock:10001'], args=[100])

# Lua 脚本优势：
# 1. 原子性：不会被其他命令打断
# 2. 减少网络往返：多个命令一次发送
# 3. 服务端执行：减少客户端和服务端通信
```

## 总结：Redis 性能防护手册

| 原则 | 说明 |
|---|---|
| 禁用 `KEYS *` | 用 `SCAN` 代替 |
| 禁止 `FLUSHALL` | 用渐进式删除 |
| 控制 Big Key | 单个 key 不超过 10MB |
| 使用 Pipeline | 减少网络往返 |
| 善用 Lua 脚本 | 原子性 + 减少往返 |
| 开启慢查询日志 | 及时发现问题 |
| 监控告警 | 提前预警 |

Redis 的单线程设计既是优势也是劣势。优势是简单高性能，劣势是一个慢命令就能让整个服务卡住。理解这些限制，才能用好 Redis。
