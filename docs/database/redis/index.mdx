# Redis 总览

凌晨 2 点，你的系统突然告警——数据库连接池耗尽，请求堆积，用户投诉纷至沓来。你排查了一圈发现：某个热点数据被高并发访问，MySQL 被打爆了。

这时，你可能会想：如果有一个高性能的内存数据库，把热点数据缓存起来，是不是就能扛住这波流量？

这正是 Redis 诞生的意义。

## Redis 是什么

Redis（Remote Dictionary Server）是一个开源的、基于内存的 Key-Value 数据结构存储系统，常被用作数据库、缓存和消息队列。它由 Salvatore Sanfilippo 于 2009 年创建，目前由 Redis Labs 维护和发展。

Redis 的核心特点可以用四个字概括：**快、稳、准、活**。

**快**：基于内存操作，单机 QPS 可达 10 万以上，远超传统数据库。

**稳**：支持持久化，数据可恢复；支持主从复制，高可用有保障。

**准**：支持丰富的数据类型，String、List、Hash、Set、ZSet 五种基本类型，加上 BitMap、HyperLogLog、Geo、GEO 等高级类型，满足各类业务场景。

**活**：支持多种部署模式，单机、主从、哨兵、集群，灵活应对不同规模的业务需求。

:::tip 版本说明
Redis 6.0 引入了多线程 I/O 模型，进一步提升了网络读写性能。本文档以 Redis 6.0+ 为基准进行讲解。
:::

## 为什么要用 Redis

在讨论 Redis 的具体特性之前，我们先思考一个问题：为什么有了 MySQL 这样的关系型数据库，我们还需要 Redis？

答案在于**速度与场景的权衡**。

MySQL 是磁盘数据库，每次读写都需要经过磁盘 I/O。即便有 Buffer Pool 的加持，在面对高并发访问时，仍然可能成为性能瓶颈。

Redis 是内存数据库，数据直接存在内存中，访问延迟通常在纳秒级别。以一次普通的 GET 请求为例：

```bash
# Redis GET 操作延迟
$ redis-cli --latency-history
min: 0, max: 1, avg: 0.12 (100 samples)
```

而 MySQL 一次简单查询的延迟，通常在毫秒级别。差了整整 3 个数量级。

但 Redis 也有局限性：内存是昂贵的，且容量有限。因此，Redis 通常不作为主数据库，而是作为缓存层，与 MySQL 等持久化数据库配合使用。

:::tip 经典架构
常见的架构是「Redis + MySQL」：热点数据缓存在 Redis 中，降低数据库压力；冷数据存储在 MySQL 中，保证持久化。
:::

## Redis 的应用场景

Redis 不是万能的，理解它的适用场景是正确使用 Redis 的前提。

### 缓存场景

这是 Redis 最经典的应用场景。把热点数据放在 Redis 中，大幅降低数据库压力。

```java
// 伪代码：使用 Redis 缓存用户信息
String key = "user:info:" + userId;
String cached = redis.get(key);
if (cached != null) {
    return JSON.parse(cached);
}
User user = mysql.query("SELECT * FROM users WHERE id = ?", userId);
redis.setex(key, 3600, JSON.stringify(user));
return user;
```

### 会话存储

分布式环境下，用户 Session 不能存储在单机内存中。Redis 的高性能和集群支持，使其成为 Session 存储的理想选择。

### 排行榜/计数器

ZSet 的有序集合特性，使其天然适合排行榜场景。点赞数、阅读量、游戏积分等，都可以用 Redis 实现。

```bash
# 游戏积分排行榜
ZADD game:rank 1500 "player1"
ZADD game:rank 2000 "player2"
ZADD game:rank 1800 "player3"
ZREVRANGE game:rank 0 2 WITHSCORES
# 返回：player2 2000, player3 1800, player1 1500
```

### 分布式锁

在分布式系统中，互斥访问资源需要分布式锁。Redis 的 SETNX 特性，使其成为实现分布式锁的主流选择。

### 消息队列

List 的 LPUSH + BRPOP 组合，可以实现简单的消息队列。虽然功能不如 RabbitMQ、RocketMQ 完善，但对于轻量级场景足够使用。

### 限流

利用 Redis 的计数功能和过期时间，可以实现滑动窗口、令牌桶等多种限流算法。

## Redis 数据类型全景图

Redis 支持多种数据类型，不同数据类型适用于不同场景：

| 类型 | 底层数据结构 | 典型场景 |
| --- | --- | --- |
| String | SDS | 缓存、计数器、分布式锁 |
| List | QuickList / LinkedList | 消息队列、最新列表 |
| Hash | Dict / ziplist | 对象存储、表数据缓存 |
| Set | Dict / intset | 标签、好友关系、去重 |
| ZSet | SkipList + Dict | 排行榜、延迟队列 |
| BitMap | String | 签到、用户在线统计 |
| HyperLogLog | String | UV 统计 |
| Geo | Sorted Set | LBS、附近的人 |

每种数据类型的底层实现，都经过精心选择。例如，当 Hash 的元素数量较少时，使用压缩列表（ziplist）节省内存；当元素增多后，自动转换为哈希表。

:::tip 编码转换
Redis 的数据类型与底层实现是分离的。同一种数据类型，可能有多种编码方式，Redis 会根据数据量和元素大小自动选择最优的编码。这种设计使得 Redis 能够在小数据量时节省内存，在大数据量时保证性能。
:::

## Redis 的线程模型

很多面试官喜欢问这个问题：「Redis 是单线程的吗？」

答案是：**Redis 6.0 之前，单线程指的是网络 I/O 和命令执行是单线程的**。

Redis 的核心流程是这样的：

1. 服务端启动，绑定端口，监听连接
2. 客户端连接，通过 TCP 三次握手建立连接
3. 服务端为每个连接创建 socket 文件描述符
4. 服务端使用 I/O 多路复用（select/poll/epoll）监听多个 socket 的可读事件
5. 当某个 socket 可读时，读取命令，解析命令，执行命令，返回结果

这个流程中，**命令的读取、解析、执行都是单线程顺序执行的**。

但 Redis 6.0 引入了多线程 I/O，用于处理网络读写。命令执行仍然是单线程，但网络 I/O 可以并行处理：

```bash
# Redis 6.0 配置文件
io-threads 4        # I/O 线程数
io-threads-do-reads yes  # 是否在 I/O 线程中处理读操作
```

:::warning 核心要点
无论是单线程还是多线程版本，**Redis 的命令执行都是单线程的**，这保证了命令执行的原子性，不会出现并发问题。
:::

## Redis 的持久化机制

Redis 的数据存储在内存中，如果服务器宕机，数据会丢失。持久化机制解决了这个问题。

Redis 支持两种持久化方式：

1. **RDB（Redis Database）**：定时生成数据快照，保存为 .rdb 文件
2. **AOF（Append Only File）**：记录所有写命令，保存为 .aof 文件

两种方式各有优缺点：

| 特性 | RDB | AOF |
| --- | --- | --- |
| 文件大小 | 小（二进制格式） | 大（文本格式） |
| 恢复速度 | 快（直接加载） | 慢（重放命令） |
| 数据完整性 | 可能有丢失（取决于快照间隔） | 可配置（always/everysec/no） |
| 资源消耗 | 高（fork 子进程） | 低（持续追加） |

生产环境中，通常**同时开启 RDB 和 AOF**，既保证数据完整性，又保证恢复速度。

## Redis 的高可用方案

Redis 有多种高可用方案，适用于不同的业务场景：

### 主从复制

最简单的模式，一主多从，从节点实时同步主节点数据。优点是简单，缺点是主节点故障时需要人工介入。

### 哨兵模式

在主从基础上增加 Sentinel 节点，自动监控主从节点的健康状态。当主节点故障时，自动进行故障转移，选举新的主节点。

```bash
# 哨兵配置示例
sentinel monitor mymaster 127.0.0.1 6379 2
sentinel down-after-milliseconds mymaster 5000
sentinel failover-timeout mymaster 60000
```

### Redis Cluster

官方提供的集群方案，数据分片存储在多个节点上。每个分片可以有多个副本，实现自动故障转移。

```
集群结构：
- 3 个主节点，每个主节点 1 个从节点
- 16384 个槽位，均匀分布在 3 个主节点
- 自动分片、自动故障转移
```

## Redis vs Memcached

这是面试中经常被问到的问题。

| 特性 | Redis | Memcached |
| --- | --- | --- |
| 数据类型 | 丰富（5种基本 + 多种高级） | 只有 String |
| 持久化 | 支持 RDB、AOF | 不支持 |
| 集群 | 原生支持 Cluster | 需要客户端实现 |
| 内存管理 | 多种淘汰策略 | LRU |
| 单线程 vs 多线程 | 单线程执行（6.0+ 可配置多线程 I/O） | 多线程 |
| 性能 | 略低（但足够用） | 略高 |

**选择建议**：如果只需要简单的 K-V 缓存，选 Memcached；如果需要丰富的数据类型、持久化、集群等功能，选 Redis。

## 总结

Redis 是现代互联网架构中不可或缺的一环。它的价值在于：

1. **高性能**：基于内存，纳秒级延迟
2. **丰富的数据结构**：String、List、Hash、Set、ZSet，加上高级类型，满足各类场景
3. **完善的高可用方案**：主从、哨兵、集群，覆盖从小到大的业务规模
4. **活跃的社区**：持续迭代，生态完善

理解 Redis 的核心原理，是每个后端工程师的必修课。
