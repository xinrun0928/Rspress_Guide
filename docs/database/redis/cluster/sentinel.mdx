# 哨兵（Sentinel）机制

主从架构有个致命问题：**主节点挂了怎么办？**

手动切换太慢，业务不可用时间可能长达几十分钟。哨兵机制，就是为了解决这个问题——**自动化故障检测和切换**。

## 哨兵的核心职责

哨兵不是一个人，而是一个**集群**。它负责三件事：

**1. 监控（Monitoring）**：持续检测主节点和从节点是否正常运行。

**2. 通知（Notification）**：发现问题后，通过 API 向应用端推送通知。

**3. 自动故障转移（Automatic Failover）**：主节点宕机时，自动将某个从节点升级为新的主节点。

```
+------------------+     监控      +------------------+
|                  | ------------> |                  |
|   应用客户端      |              |    Sentinel      |
|                  | <------------ |    集群          |
+------------------+     通知      +------------------+
                                   |  |  |  |  |
                                   v  v  v  v  v
                              +------+------+------+
                              | Master| Slave| Slave|
                              +------+------+------+
```

## 哨兵集群如何工作

### 1. 服务发现：发现主从节点

哨兵通过**发布/订阅**机制发现其他哨兵节点：

```bash
# 哨兵每 2 秒向 __sentinel__:hello 频道发送自己的信息
# 包括：IP、端口、run_id、当前配置版本

 PUBLISH __sentinel__:hello "127.0.0.1,26379,run_id,epoch,master_name"

# 哨兵订阅这个频道，发现其他哨兵
 SUBSCRIBE __sentinel__:hello
```

同时，哨兵通过向主节点发送 `INFO` 命令获取所有从节点信息：

```bash
# 哨兵执行 INFO replication
> INFO replication
role:master
connected_slaves:2
slave0:ip=127.0.0.1,port=6380,state=online,offset=123456
slave1:ip=127.0.0.1,port=6381,state=online,offset=123456
```

### 2. 故障检测：主客观下线

哨兵检测节点状态有两种：「主观下线」和「客观下线」。

**主观下线（SDown）**：

单个哨兵认为某个节点不可用。

```bash
# 配置项
sentinel monitor mymaster 127.0.0.1 6379 2
#                     主观下线 quorum: 2 个哨兵认为下线才算 SDown
```

**客观下线（ODown）**：

当超过 `quorum` 数量的哨兵都认为主节点下线了，才判定为客观下线。此时触发故障转移。

```bash
# 客观下线的判断条件
# 1. 主观下线时间超过 down-after-milliseconds
# 2. 超过 quorum 数量的哨兵都主观下线
```

为什么需要哨兵集群，而不是单个哨兵？

```
单个哨兵的问题：
- 哨兵本身可能宕机
- 单点故障，无法保证高可用

哨兵集群的优势：
- 多数哨兵同意才判定故障
- 避免误判（如网络抖动）
- 选出 leader 协调故障转移
```

### 3. 故障转移：选主与切换

当主节点客观下线后，哨兵集群开始故障转移流程：

**Step 1：选举 Leader**

多个哨兵中选出一个作为 leader，由它执行故障转移。使用 Raft 算法的变种：

```bash
# 哨兵 A
> SENTINEL is-master-down-by-addr 127.0.0.1 6379
# 返回：是否 SDown，以及当前状态

# 多个哨兵互相投票
# 得票超过一半的哨兵成为 leader
```

**Step 2：选择新主节点**

被选为 leader 的哨兵，从从节点中选择一个作为新主节点。选举标准：

1. **优先级**：`replica-priority` 配置高的优先
2. **复制进度**：与旧主节点 `offset` 差距最小的
3. **Run ID**：`run_id` 越小越优先（纯粹为了确定性）

```bash
# 配置从节点优先级
# redis.conf
replica-priority 100  # 默认 100，值越大越优先被选为新主节点
```

**Step 3：执行切换**

```
旧主节点（宕机）         新主节点（升级）        从节点
     X                      |                     |
                            |                     |
                            |  SLAVEOF NO ONE     |
                            |<--------------------|
                            |                     |
                            |                     |
                            |  SLAVEOF 新主节点   |
                            |<--------------------|
```

1. Leader 哨兵向选中的从节点发送 `SLAVEOF NO ONE`，将其升级为主节点
2. Leader 哨兵向其他从节点发送 `SLAVEOF 新主节点`，让它们复制新主节点
3. Leader 哨兵更新旧主节点的配置，如果旧主节点恢复，变为新主节点的从节点

### 4. 通知客户端

故障转移完成后，哨兵通过发布/订阅通知客户端：

```bash
# 客户端订阅这些频道
> PSUBSCRIBE +switch-master *  # 主节点切换
> PSUBSCRIBE +sdown *           # 主观下线
> PSUBSCRIBE -sdown *           # 主观下线恢复
```

客户端需要实现 Sentinel-aware 逻辑：

```python
# Python Redis 客户端（redis-py）示例
from redis.sentinel import Sentinel

sentinel = Sentinel([
    ('127.0.0.1', 26379),
    ('127.0.0.1', 26380),
    ('127.0.0.1', 26381)
], socket_timeout=0.1)

# 自动获取主节点地址
master = sentinel.master_for('mymaster')
slave = sentinel.slave_for('mymaster')

# 写入主节点
master.set('key', 'value')

# 读取从节点
value = slave.get('key')
```

## 哨兵配置详解

```bash
# sentinel.conf

# 监控的主节点
sentinel monitor mymaster 127.0.0.1 6379 2

# 主节点密码（如果有）
sentinel auth-pass mymaster mypassword

# 主观下线时间（毫秒）
sentinel down-after-milliseconds mymaster 30000

# 故障转移超时时间
sentinel failover-timeout mymaster 180000

# 并行同步数量（同时有多少从节点向新主节点同步）
sentinel parallel-syncs mymaster 1

# 自动发现其他哨兵
sentinel known-replica mymaster 127.0.0.1 6380
sentinel known-replica mymaster 127.0.0.1 6381
sentinel known-sentinel mymaster 127.0.0.1 26380
```

:::tip 配置建议
- 哨兵节点数量：建议 **3 个或 5 个**（奇数，避免脑裂）
- `quorum`：建议设为 `哨兵数量 / 2 + 1`
- `parallel-syncs`：设为 1 可以减少对主节点的压力，但恢复较慢
:::

## 哨兵集群的搭建

### 启动哨兵

```bash
# 方式一：使用 redis-sentinel 命令
redis-sentinel /path/to/sentinel.conf

# 方式二：使用 redis-server 加载哨兵模式
redis-server /path/to/sentinel.conf --sentinel
```

### 查看哨兵状态

```bash
# 连接哨兵
redis-cli -p 26379

# 查看监控的主节点
> SENTINEL masters
1) "mymaster"

# 查看主节点详情
> SENTINEL master mymaster
1) "name"
2) "mymaster"
3) "ip"
4) "127.0.0.1"
5) "port"
6) "6379"
...

# 查看从节点
> SENTINEL slaves mymaster
1) ...slave1 info...
2) ...slave2 info...

# 查看其他哨兵
> SENTINEL sentinels mymaster
1) ...sentinel info...
```

## 哨兵的局限性

哨兵解决了**主节点故障自动切换**的问题，但它有局限性：

**1. 无法水平扩展写**

哨兵只是做高可用，不解决单节点写入瓶颈。所有写请求还是到单台主节点。

**2. 数据无法分片**

所有数据都在主节点，无法分片存储。如果数据量超过单节点容量上限，哨兵无能为力。

**3. 故障转移期间有窗口期**

故障转移期间（通常几十秒），可能有部分请求失败。

```bash
# 故障转移期间的行为
# 1. 原主节点不可用 -> 写请求失败（即使配置了从节点读，也可能导致短暂不可写）
# 2. 新主节点选举完成 -> 恢复写入
```

解决这些问题，需要 Redis Cluster。
