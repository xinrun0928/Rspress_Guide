# Redis Cluster 故障转移与槽迁移

集群运行过程中，节点可能会宕机、扩缩容。故障转移和槽迁移是集群运维的核心操作。

## 故障转移

### 触发条件

故障转移发生在以下情况：

1. **主节点宕机**：某个主节点不可用
2. **主节点响应超时**：超过 `cluster-node-timeout` 无响应
3. **手动触发**：管理员执行 `CLUSTER FAILOVER` 命令

### 故障转移流程

```
1. 从节点发现主节点下线
   |
   v
2. 等待一段时间（避免多个从节点同时抢）
   |
   v
3. 从节点发起投票申请
   |
   v
4. 获得多数投票后，成为新主节点
   |
   v
5. 广播 pong 消息，通知其他节点
```

**详细步骤**：

```bash
# Step 1: 检测主节点下线
# 从节点定期 ping 主节点
# 超过 cluster-node-timeout 未响应，标记为疑似下线（pfail）

# Step 2: 发起故障转移
# 等待 cluster-slave-validity-factor * cluster-node-timeout 时间
# 确认主节点确实下线

# Step 3: 发起投票请求
> CLUSTER FAILOVER FORCE  # 强制故障转移（无需等待旧主节点确认）
# 或
> CLUSTER FAILOVER TAKEOVER  # 直接接管（跳过投票，用于紧急情况）

# Step 4: 获得投票
# 持有槽的主节点投票
# 获得 N/2 + 1 票后成为新主节点

# Step 5: 更新配置
# 新主节点更新自己的 configEpoch
# 向集群广播 pong 消息
```

### 手动故障转移

管理员可以手动触发故障转移，实现平滑升级：

```bash
# 在从节点上执行
> CLUSTER FAILOVER

# 可选参数：
# - TAKEOVER: 强制接管，立即故障转移
# - FORCE: 强制故障转移，不等待主节点确认

# 场景：升级主节点
# 1. 先在从节点执行 CLUSTER FAILOVER
# 2. 等待从节点升级为新主节点
# 3. 升级原主节点（现在是新主节点的从节点）
# 4. 可选：将主从关系切回来
```

### 故障转移期间的行为

```bash
# 故障转移期间
# 1. 从节点不可用（升级为主节点期间）
# 2. 可能出现少量写失败
# 3. 读请求可能读到旧数据

# 查看故障转移状态
> CLUSTER NODES
# 状态 flags 中包含 "fail" 表示已下线
# 状态 flags 中包含 "master" 且有槽表示已升级为主节点
```

## 槽迁移

槽迁移用于集群扩缩容、负载均衡。

### 迁移流程

```bash
# 假设要将槽 5000 从节点 A 迁移到节点 B

# Step 1: 目标节点（节点 B）设置为 IMPORTING 状态
> CLUSTER SETSLOT 5000 IMPORTING <node_A_id>

# Step 2: 源节点（节点 A）设置为 MIGRATING 状态
> CLUSTER SETSLOT 5000 MIGRATING <node_B_id>

# Step 3: 迁移具体的 key
> MIGRATE <target_ip> <target_port> "" 0 5000 KEYS key1 key2 key3

# 或者逐个迁移
> MIGRATE <target_ip> <target_port> "" 0 5000 MOVE <key>

# Step 4: 完成迁移
> CLUSTER SETSLOT 5000 NODE <node_B_id>

# Step 5: 节点 A 收到通知，删除本地槽记录
```

### 迁移中的访问处理

```bash
# 迁移过程中，对 key 的访问处理：

# 场景 1: key 仍在节点 A
> GET mykey
-ASK 5000 127.0.0.1:6380  # 返回 ASK 重定向
# 客户端向节点 B 发送 ASKING 命令后再获取
> ASKING
+OK
> GET mykey
"value"

# 场景 2: key 已在节点 B
> GET mykey
"value"  # 直接返回
```

### 使用 redis-cli 迁移槽

```bash
# 自动迁移工具
redis-cli --cluster reshard 127.0.0.1:6379

# 交互式迁移步骤：
# 1. 选择目标节点 ID
# 2. 选择源节点 ID（可以多个）
# 3. 输入要迁移的槽数量
# 4. 工具自动执行迁移

# 非交互式迁移
redis-cli --cluster reshard 127.0.0.1:6379 \
  --cluster-from <source_node_id> \
  --cluster-to <target_node_id> \
  --cluster-slots 500
```

## 扩缩容操作

### 添加主节点

```bash
# 1. 启动新 Redis 实例
redis-server --port 6390 --cluster-enabled yes \
  --cluster-config-file nodes-6390.conf

# 2. 添加到集群
redis-cli --cluster add-node 127.0.0.1:6390 127.0.0.1:6379

# 3. 分配槽（从现有节点迁移）
redis-cli --cluster reshard 127.0.0.1:6379
```

### 添加从节点

```bash
# 添加从节点
redis-cli --cluster add-node 127.0.0.1:6391 127.0.0.1:6379 --cluster-slave

# 指定主节点
redis-cli --cluster add-node 127.0.0.1:6391 127.0.0.1:6379 \
  --cluster-slave --cluster-master-id <master_id>
```

### 删除节点

```bash
# 删除从节点（直接删除）
redis-cli --cluster del-node 127.0.0.1:6391 <node_id>

# 删除主节点（需要先迁移槽）
# 1. 确认该主节点没有槽
redis-cli --cluster setslot <slot> NODE <target_node_id> ...
# 或使用 reshard 将槽迁走

# 2. 删除节点
redis-cli --cluster del-node 127.0.0.1:6390 <node_id>
```

:::warning 删除主节点的注意事项
- 删除前必须确保该节点的槽已全部迁出
- 迁移过程中不要关闭源节点
- 删除后，该节点的所有从节点也会被移除
:::

### 常见问题排查

```bash
# 槽未完全分配
cluster_state:fail
# 解决：重新分配槽
redis-cli --cluster rebalance 127.0.0.1:6379

# 节点不可达
cluster_state:ok, 但某些节点状态异常
# 解决：检查网络、防火墙、端口

# 节点 down 但未触发故障转移
# 可能原因：从节点数量不足，无法获得多数票
# 解决：增加从节点数量
```
