# Redis 集群方案对比

当你的单机 Redis 扛不住流量时，选择哪种集群方案就成了关键决策。

## 四种方案总览

| 方案 | 架构模式 | 客户端支持 | 数据分片 | 适用场景 |
| --- | --- | --- | --- | --- |
| 主从 + 哨兵 | 一主多从 + 哨兵集群 | 普通 Redis 客户端 | 无 | 中小规模，高可用 |
| Redis Cluster | 去中心化槽分片 | Cluster 客户端 | 自动 | 大规模数据，高可用 |
| Codis | Proxy 代理层分片 | 普通 Redis 客户端 | 基于 Slot | 过渡方案，需要迁移 |
| Twemproxy | Proxy 代理层分片 | 普通 Redis 客户端 | 一致性哈希 | 简单分片需求 |

## 主从 + 哨兵

### 架构图

```
                    +------------------+
                    |   Sentinel       |
                    |   集群           |
                    +--------+---------+
                             |
        +--------------------+--------------------+
        |                    |                    |
        v                    v                    v
   +---------+          +---------+          +---------+
   | Master  | <------> | Slave1  | <------> | Slave2  |
   +---------+          +---------+          +---------+
        ^                                        ^
        |                                        |
   +---------+                             +---------+
   | 写入    |                             | 读取    |
   +---------+                             +---------+
```

### 优缺点分析

**优点**：

1. **配置简单**：无需改客户端代码
2. **客户端兼容**：任何 Redis 客户端都支持
3. **成熟稳定**：哨兵模式经过多年生产验证
4. **读写分离**：有效分担读压力

**缺点**：

1. **无法分片**：所有数据在一个主节点
2. **扩容困难**：需要手动迁移数据
3. **写入瓶颈**：单机写入能力有限

```bash
# 哨兵模式客户端示例（Python）
from redis.sentinel import Sentinel

sentinel = Sentinel([('127.0.0.1', 26379),
                    ('127.0.0.1', 26380),
                    ('127.0.0.1', 26381)])
master = sentinel.master_for('mymaster')
slave = sentinel.slave_for('mymaster')
```

### 适用场景

- 数据量较小（几十 GB 以内）
- 以读为主，写压力不大
- 不想改客户端代码
- 需要简单的高可用方案

## Redis Cluster

### 架构图

```
      +------+     +------+     +------+
      |Node A|     |Node B|     |Node C|
      |Slot 0|     |Slot 1|     |Slot 2|
      | -5460|     |5461- |     |10923-|
      +--+---+     |10922 |     |16383 |
         |         |+--+---+     +--+---+
         |            |             |
    +----+---+    +----+---+    +----+---+
    |Slave A |    |Slave B |    |Slave C |
    +--------+    +--------+    +--------+

   客户端 ---->  计算槽号 ----> 路由到对应节点
```

### 优缺点分析

**优点**：

1. **原生分片**：数据自动分散到多个节点
2. **去中心化**：无单点故障，性能好
3. **高可用**：自动故障转移
4. **官方支持**：Redis 官方方案，持续更新

**缺点**：

1. **客户端复杂**：需要 Cluster 客户端
2. **跨槽操作受限**：多 key 操作需同一槽
3. **不支持事务跨槽**：Lua 脚本跨槽受限
4. **最少 3 主**：资源消耗较大

```bash
# Cluster 客户端示例（Redis-Python）
from redis.cluster import RedisCluster

rc = RedisCluster(host='127.0.0.1', port=6379,
                  skip_full_coverage_check=True)
rc.set('key', 'value')
rc.get('key')
```

### 适用场景

- 数据量大（单节点无法承载）
- 写压力大（需要水平扩展写入）
- 需要自动故障转移
- 可以使用 Cluster 客户端

## Codis

### 架构图

```
   客户端                   Proxy 层                  Redis 层
      |                        |                        |
      | <---------------------> |                        |
      |    任意 Redis 客户端    |                        |
      |                        |                        |
      |                        +---> Redis 1 (Master)   |
      |                        +---> Redis 1 (Slave)    |
      |                        +---> Redis 2 (Master)   |
      |                        +---> Redis 2 (Slave)    |
      |                        |                        |
      |                        +---> Redis N (Master)   |
      |                        +---> Redis N (Slave)    |
      |                        |                        |
      +--------------------> [ZooKeeper]                 |
                    存储 slot 分布信息                   |
```

### 优缺点分析

**优点**：

1. **客户端兼容**：支持普通 Redis 客户端
2. **功能丰富**：支持扩容、缩容、迁移
3. **可视化**：提供 Dashboard 管理界面
4. **过渡方案**：适合从主从迁移到 Cluster

**缺点**：

1. **依赖 ZooKeeper**：增加运维复杂度
2. **Proxy 开销**：多一跳网络延迟
3. **非官方**：豌豆荚开源项目，更新较少
4. **不支持某些命令**：如 KEYS、SCAN

### Codis 配置

```bash
# codis.json 配置文件
{
  "proxy": {
    "addr": ":9000",
    "proto": "tcp4",
    "proxy_id": "proxy_1"
  },
  "zk": {
    "zk_addr": "127.0.0.1:2181",
    "zk_auth": "",
    "timeout": 30
  }
}
```

### 适用场景

- 已有大量 Redis 客户端代码，不方便改
- 需要平滑迁移到分片架构
- 需要可视化运维界面

## Twemproxy

### 架构图

```
   客户端
      |
      v
+------------+
| Twemproxy  |  <--- 一致性哈希
+------------+
      |
      +---> Redis 1
      +---> Redis 2
      +---> Redis 3
      +---> Redis 4
```

### 优缺点分析

**优点**：

1. **轻量级**：C 语言实现，性能好
2. **简单**：配置简洁
3. **成熟**：Twitter 开源，久经验证

**缺点**：

1. **不支持动态扩容**：修改配置后需要重启
2. **不支持故障转移**：需要配合哨兵
3. **功能受限**：不支持跨节点操作
4. **不再维护**：Twitter 已停止维护

### Twemproxy 配置

```bash
# nutcracker.yml
redis:
  listen: 127.0.0.1:22121
  hash: fnv1a_64
  distribution: ketama
  timeout: 100
  redis: true
  servers:
    - 127.0.0.1:6379:1
    - 127.0.0.1:6380:1
    - 127.0.0.1:6381:1
```

### 适用场景

- 简单的分片需求
- 读多写少
- 不想引入复杂的中间件

## 方案对比详解

### 客户端兼容性

| 方案 | 普通 Redis 客户端 | Cluster 客户端 | 说明 |
| --- | --- | --- | --- |
| 主从 + 哨兵 | 支持 | 支持 | 两者皆可，推荐使用普通客户端 + Sentinel |
| Redis Cluster | 不支持 | 支持 | 必须使用 Cluster 客户端 |
| Codis | 支持 | 不支持 | 使用 Codis 客户端或普通 Redis 客户端 |
| Twemproxy | 支持 | 不支持 | 使用普通 Redis 客户端 |

### 数据分片方式

| 方案 | 分片方式 | 槽数量 | 可动态调整 |
| --- | --- | --- | --- |
| 主从 + 哨兵 | 无分片 | 16384（单节点） | 不支持 |
| Redis Cluster | 槽分片 | 16384 | 支持 |
| Codis | Slot 分片 | 1024 | 支持 |
| Twemproxy | 一致性哈希 | N/A | 不支持（需重启） |

### 故障转移

| 方案 | 自动故障转移 | 需要手动介入 | 说明 |
| --- | --- | --- | --- |
| 主从 + 哨兵 | 是 | 否 | 哨兵自动切换 |
| Redis Cluster | 是 | 否 | 从节点自动升级 |
| Codis | 是 | 否 | Proxy 检测并切换 |
| Twemproxy | 否 | 是 | 需要配合哨兵或其他方案 |

### 运维复杂度

```
低 ──────────────────────────────────────────────> 高

主从+哨兵 < Twemproxy < Codis < Redis Cluster

复杂度由低到高：
- 主从+哨兵：只需配置哨兵
- Twemproxy：配置简单代理
- Codis：需要运维 ZooKeeper
- Redis Cluster：需要使用 Cluster 客户端
```

## 选型建议

```
数据量 < 50GB？
    ├── 读多写少？ -> 主从 + 哨兵
    └── 写压力大？ -> Redis Cluster

数据量 > 50GB？
    ├── 不想改客户端代码？ -> Codis
    └── 可以改客户端代码？ -> Redis Cluster
```

:::tip 2024 年的建议
Redis Cluster 已经成为事实标准，社区活跃，文档完善。新项目建议直接使用 Redis Cluster。

Codis 和 Twemproxy 主要用于遗留系统的迁移。
:::
