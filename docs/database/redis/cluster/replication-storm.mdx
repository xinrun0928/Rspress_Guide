# 主从复制风暴与优化

一个「正常」的主从架构，在某些极端场景下，可能变成一场灾难。

## 什么是复制风暴

复制风暴是指**大量从节点同时发起全量同步，导致主节点资源耗尽**的现象。

典型场景有两个：

**场景一：主节点宕机重启**

```
主节点重启后 run_id 变化
所有从节点必须全量同步
同时连接 -> 同时 BGSAVE -> 内存爆了
```

**场景二：网络抖动**

```
网络闪断 1 秒
多个从节点同时重连
触发复制风暴
```

## 复制风暴的成因分析

### 原因一：主节点重启

Redis 4.0 之前，主节点重启后 `run_id` 会变化。这对于从节点来说，是一个「全新的主节点」，必须全量同步。

```bash
# Redis 3.x 中，主节点重启
主节点的 run_id: a1b2c3d4 -> 重启后变成 e5f6g7h8
从节点：这不是同一个主节点！重新全量同步
```

Redis 4.0 引入了 `repl-id` 和 `repl-offset` 的持久化，主节点重启后可以保留复制 ID，继续增量同步。

```bash
# 启用无盘复制（主节点不落盘直接发送）
repl-diskless-sync yes

# 等待多个从节点（避免同时全量同步）
repl-diskless-sync-delay 5
```

### 原因二：缓冲区不足

`repl_backlog_buffer` 太小，导致增量同步失败。

```bash
# 默认 1MB，可能不够
repl-backlog-size 10mb  # 建议根据实际需求调大
```

### 原因三：网络分区

网络抖动导致多个从节点同时断开重连。

## 复制风暴的影响

复制风暴会导致：

1. **主节点 CPU/内存飙升**：同时处理多个 BGSAVE
2. **网络带宽被打满**：多个 RDB 文件同时传输
3. **从节点不可用**：全量同步期间从节点不可读
4. **数据恢复时间长**：RDB 文件越大，加载越慢

## 优化策略

### 策略一：调整复制超时时间

```bash
# 适当增大超时时间，避免短时网络抖动触发重连
repl-timeout 300  # 默认 60 秒，建议 300 秒
```

### 策略二：使用无盘复制

```bash
# 开启无盘复制（从 RDB 管道直接发送，不写磁盘）
repl-diskless-sync yes

# 无盘复制延迟（等待更多从节点，减少同步次数）
repl-diskless-sync-delay 5
```

### 策略三：级联复制（树形结构）

```
           主节点
            |
    +-------+-------+
    |               |
 从节点A           从节点B
    |               |
 +--+--+         +--+--+
 |     |         |     |
从节点C 从节点D 从节点E 从节点F
```

配置级联复制：

```bash
# 从节点A 的配置
replicaof 主节点IP 6379

# 从节点C 的配置（指向从节点A）
replicaof 从节点A_IP 6379
```

:::warning 级联复制的代价
层级越深，复制延迟越大。最下层的从节点数据可能严重落后。
:::

### 策略四：配置副本数量上限

```bash
# 单个主节点的从节点数量不宜过多
# 建议不超过 5 个从节点

# 监控从节点数量
> INFO replication
connected_slaves:5
```

### 策略五：使用 Redis Cluster

Redis Cluster 自动将槽分散到多个主节点，每个主节点的从节点数量可控：

```bash
# Cluster 模式下，每个主节点默认 1 个从节点
# 可通过 --cluster-replicas 参数调整
redis-cli --cluster create 127.0.0.1:7000 127.0.0.1:7001 ... \
  --cluster-replicas 1
```

### 策略六：渐进式同步

Redis 5.0 引入了 `WAIT` 命令，支持同步复制：

```bash
> SET key value
> WAIT 1 0  # 等待至少 1 个从节点确认
:1
```

但这会影响写入性能，适合对数据一致性要求高的场景。

## 复制风暴应急处理

如果复制风暴已经发生：

**1. 限流从节点的连接**

```bash
# 临时禁止从节点连接
> CONFIG SET maxclients 1000
> CONFIG REWRITE
```

**2. 逐个恢复从节点**

```
# 1. 关闭所有从节点的 replicaof
> REPLICAOF NO ONE

# 2. 逐个开启从节点的复制
> REPLICAOF 主节点IP 6379

# 3. 等待一个同步完成，再开启下一个
```

**3. 使用 `CLIENT PAUSE` 暂停写入**

```bash
# 暂停所有客户端写入
> CLIENT PAUSE 60 WRITE
```

## 监控复制状态

```bash
# 查看复制延迟
> INFO replication | grep -E "master_repl_offset|slave"

# 查看从节点状态
> INFO replication
role:master
connected_slaves:3
slave0:ip=127.0.0.1,port=6380,state=online,offset=12345678,lag=1
slave1:ip=127.0.0.1,port=6381,state=online,offset=12345678,lag=1
slave2:ip=127.0.0.1,port=6382,state=online,offset=12345678,lag=1
```

关键指标 `lag`：从节点落后主节点的命令数。`lag=1` 表示落后 1 条命令，通常正常。
