# 主从复制原理

想象一下这个场景：你的 Redis 写请求越来越多，一台机器扛不住了。你加了从节点，流量读写分离。但某天凌晨 3 点，从节点的数据和主节点对不上了——这可能是复制链路中某个环节出了问题。

理解主从复制的原理，不是为了面试背答案，而是为了**在出问题的时候知道去哪排查**。

## 主从复制解决了什么问题

主从复制的核心价值有两个：

**读写分离**：读请求分摊到从节点，大幅提升读取吞吐量。

**数据冗余**：从节点作为主节点的数据副本，提供数据备份能力。

```
写请求                    读请求
   |                        |
   v                        v
+------+                  +------+
|Master| ---------------->|Slave1|
+------+                  +------+    +------+
   |                           |  ->  |Slave2|
   |                           +------+ +------+
   |                                  |
   +----------------------------------+
           数据同步（异步复制）
```

但这里有个关键点：**主从复制是异步的**。主节点处理完写命令后，立即返回客户端，不会等待从节点同步完成。这是为了性能，但也是很多「数据不一致」问题的根源。

## 两种复制方式：SYNC vs PSYNC

### SYNC：最初的复制方式

SYNC 是 Redis 2.8 之前的复制方式，工作流程很简单：

```
从节点连接主节点 -> 发送 SYNC -> 主节点执行 BGSAVE -> 发送 RDB -> 从节点加载
```

**全量同步的完整流程**：

```bash
# 1. 从节点向主节点发送 SYNC 命令
> SYNC

# 2. 主节点执行 BGSAVE
#    - 创建子进程生成 RDB 文件
#    - 继续接收新的写命令，存入缓冲区

# 3. RDB 生成后，主节点发送给从节点
#    - 通过网络传输整个 RDB 文件
#    - 文件大小可能很大（几 GB）

# 4. 从节点接收并加载 RDB
#    - 清空现有数据
#    - 加载 RDB 文件

# 5. 传输完成
#    - 主节点发送缓冲区中的增量命令
#    - 从节点执行这些命令
```

问题在哪？**每次从节点断开重连，都要执行一次全量同步**。

想象一个场景：主节点有 10GB 数据，网络抖动导致从节点断开了 1 秒。这时候从节点重连，会触发一次 10GB 的 RDB 传输。这个传输过程中，主节点还在不断接收新请求，缓冲区可能溢出，导致复制失败。

### PSYNC：增量同步的改进

Redis 2.8 引入了 PSYNC（Partial Sync，部分同步），解决了这个问题。

PSYNC 有两种模式：

**1. 完整同步（Full Resync）**：和 SYNC 一样，传输完整 RDB。

**2. 增量同步（Partial Resync）**：只传输断开期间丢失的命令。

```bash
# PSYNC 完整流程
> PSYNC <run_id> <offset>

# run_id: 主节点的运行 ID
# offset: 从节点已处理的命令偏移量
```

**什么时候用完整同步？**

- 从节点第一次连接主节点（没有 run_id）
- 主从断开时间过长，repl_backlog_buffer 环形缓冲区被覆盖
- 主节点重启，run_id 发生变化

**什么时候用增量同步？**

- 从节点短时断开重连
- 断开期间，主节点接收的写命令仍在 repl_backlog_buffer 中

## PSYNC2：进一步优化

Redis 4.0 对 PSYNC 进行了改进，称为 PSYNC2。核心改进点：

**1. 主节点重启后仍可增量同步**

Redis 4.0 之前，主节点重启后 run_id 会变化，从节点必须全量同步。PSYNC2 通过 `replid`（复制ID）替代 `run_id`，并支持从多个 master 复制的从节点在宕机恢复后继续同步。

**2. 支持从节点的部分重连**

当多个从节点同时断开，PSYNC2 可以一次性为所有符合条件的从节点执行增量同步。

```bash
# Redis 5.0 后使用 PSYNC2
> PSYNC <replid> <offset>

# replid: 复制ID，主节点唯一标识
# offset: 从节点已复制的偏移量
```

## 复制链路的完整流程

让我们梳理一下主从复制的完整链路，理解每一层的作用：

```
+------------------+     binlog      +------------------+    relay log    +------------------+
|    主节点        | --------------> |    从节点        | --------------> |    从节点        |
|                  |                 |   IO 线程        |                 |   SQL 线程       |
| 1. 接收命令      |                 |                  |                 |                  |
| 2. 执行命令      |                 | 1. 接收 RDB      |                 | 1. 读取 relay log|
| 3. 写 binlog     |                 | 2. 写入 relay log|                 | 2. 执行命令      |
| 4. 响应客户端    |                 |                  |                 |                  |
+------------------+                 +------------------+                 +------------------+
```

**主节点的三个角色**：

1. **命令执行者**：接收并执行客户端的写命令
2. **binlog 记录者**：将命令写入 binlog（MySQL 叫 binlog，Redis 内部叫 replication buffer）
3. **数据发送者**：等待从节点请求，发送数据

**从节点的三个角色**：

1. **数据接收者（IO 线程）**：
   - 与主节点建立 TCP 连接
   - 发送 PSYNC 命令带上 offset
   - 接收 RDB 文件或增量命令
   - 写入 relay log（中继日志）

2. **命令执行者（SQL 线程）**：
   - 读取 relay log
   - 按照顺序执行命令
   - 保证与主节点一致

:::warning 注意复制延迟
从节点的数据永远**可能**落后于主节点，因为复制是异步的。主节点执行完命令后，立即返回客户端，不会等待从节点同步。

在主节点执行：
```bash
> SET key value
OK
> GET key
"value"
```

此时从节点可能还没有这条数据。
:::

## repl_backlog_buffer：增量同步的关键

`repl_backlog_buffer` 是一个**固定大小的环形缓冲区**，用于存储主节点新接收的写命令。它是实现增量同步的关键。

```bash
# 配置大小（默认 1MB，建议设置大一些）
repl-backlog-size 10mb

# 环形缓冲区示意图
+----------------------------------------------------------+
|  [cmd1] [cmd2] [cmd3] ... [cmd9998] [cmd9999] [cmd10000]  |
+----------------------------------------------------------+
   ^                                                        ^
   |                                                        |
 head                                                    tail
(最新命令)                                              (最旧命令)
```

**为什么用环形缓冲区？**

内存固定，不会无限增长。当 head 追上 tail 时，最旧的数据会被覆盖。

**增量同步的条件**：

断开期间丢失的命令仍在 buffer 中（head - tail `<=` buffer_size）。

如果从节点断开太久，丢失的命令已被覆盖，就只能走完整同步。

```bash
# 查看复制相关信息
> INFO replication
role:master
connected_slaves:2
slave0:ip=127.0.0.1,port=6380,state=online,offset=12345678,lag=0
slave1:ip=127.0.0.1,port=6381,state=online,offset=12345678,lag=0
repl_backlog_active:1
repl_backlog_size:10485760
repl_backlog_first_byte_offset:12340000
repl_backlog_histlen:45678
```

关键指标：
- `repl_backlog_size`：缓冲区大小
- `repl_backlog_first_byte_offset`：缓冲区中最早命令的偏移量
- `repl_backlog_histlen`：缓冲区中有效命令的长度

## 主从复制的配置

### 配置文件方式

```bash
# redis.conf - 主节点
bind 0.0.0.0
port 6379

# 从节点配置
replicaof <master_ip> <master_port>

# 如果主节点有密码
masterauth <password>

# 复制缓冲区大小（默认 1MB）
repl-backlog-size 5mb

# 复制超时时间
repl-timeout 60
```

### 命令行方式

```bash
# 指定主节点
> REPLICAOF 127.0.0.1 6379

# 取消复制，变回主节点
> REPLICAOF NO ONE

# 查看复制状态
> INFO replication
```

### 启动一个只读从节点

```bash
# redis.conf
replica-read-only yes  # 默认开启，从节点只读
```

:::tip 生产环境建议
- 从节点设置 `replica-read-only yes`
- 合理设置 `repl-backlog-size`（建议 10-100MB）
- 监控 `slave lag` 指标，及时发现复制延迟
:::

## 复制风暴：被忽视的隐患

当主节点挂了，从节点全部重新连接主节点，可能引发「复制风暴」：

```
主节点挂了
   |
   v
+------+  +------+  +------+  +------+
|Slave1| |Slave2| |Slave3| |Slave4|
+------+  +------+  +------+  +------+
   |        |        |        |
   +--------+--------+--------+
            |
            v
      全部同时请求全量同步
      主节点瞬间被打爆
```

如何避免？引入**哨兵机制**，实现自动故障转移，确保只有一台从节点升级为主节点。
