# Redis 集群概述

当你只有一个 Redis 实例时，它就像一家只有一个厨师的小餐馆。客人少还好，客人一多，厨师就忙不过来了——这就是所谓的「单点瓶颈」。

Redis 集群，就是为了解决**水平扩展**问题而生的。

## 集群要解决的核心问题

Redis 集群不是银弹，它只解决两个问题：

**1. 容量扩展**：单个 Redis 实例内存有限（通常几十 GB），当数据量超过这个限制时，需要分片存储。

**2. 可用性保障**：单机 Redis 故障意味着服务完全不可用。集群通过主从复制和故障转移，保证即使某个节点挂了，服务依然可用。

```
单点架构                    集群架构
+--------+                +----+   +----+   +----+
|  Redis |                |master|  |master|  |master|
+--------+                +--+--+   +--+--+   +--+--+
                            |        |        |
                         +--+--+   +--+--+   +--+--+
                         |slave|   |slave|   |slave|
                         +-----+   +-----+   +-----+
```

## Redis 集群的三种主流方案

| 方案 | 架构特点 | 客户端支持 | 数据分片 | 适用场景 |
| --- | --- | --- | --- | --- |
| 主从 + 哨兵 | 一主多从 + 哨兵监控 | 支持普通 Redis 客户端 | 无分片，仅复制 | 中小规模，不想改代码 |
| Redis Cluster | 去中心化，16384 槽分片 | 需要 Cluster 客户端 | 自动分片 | 大规模数据，需要分片 |
| Codis | Proxy 代理层分片 | 支持普通 Redis 客户端 | 基于 Slot 迁移 | 过渡方案，需要迁移 |
| Twemproxy | Proxy 代理层分片 | 支持普通 Redis 客户端 | 一致性哈希 | 简单分片需求 |

:::tip 方案选择建议
- 小规模（10G 以下）：主从 + 哨兵足够
- 大规模数据：直接上 Redis Cluster（官方方案，社区活跃）
- 不想改客户端代码：Codis 或 Twemproxy
:::
