面试官追问："那为什么有些公司还在用 Memcached？"

面试官继续追问："什么场景用 Memcached 更合适？"

小张答不上来了。

【面试官心理】

## 一、核心区别 🔴

### 1.1 数据类型

| 类型 | Redis | Memcached |
| --- | --- | --- |
| String | ✅ 支持 | ✅ 支持 |
| Hash | ✅ 支持 | ✅ 支持（但有限） |
| List | ✅ 支持 | ❌ 不支持 |
| Set | ✅ 支持 | ❌ 不支持 |
| Sorted Set | ✅ 支持 | ❌ 不支持 |
| Bitmap | ✅ 支持 | ❌ 不支持 |
| HyperLogLog | ✅ 支持 | ❌ 不支持 |
| Geospatial | ✅ 支持 | ❌ 不支持 |
| Stream | ✅ 支持 | ❌ 不支持 |

### 1.2 存储方式

| 特性 | Redis | Memcached |
| --- | --- | --- |
| 持久化 | ✅ 支持 RDB/AOF | ❌ 纯内存 |
| 内存管理 | 自有内存分配器 | slab allocator |
| 数据压缩 | ✅ 支持 | ✅ 支持 |
| 过期策略 | 被动+主动 | 被动 |

### 1.3 架构

| 特性 | Redis | Memcached |
| --- | --- | --- |
| 主从复制 | ✅ 支持 | ❌ 不支持 |
| 集群 | Redis Cluster | Membase/Couchbase |
| Lua 脚本 | ✅ 支持 | ❌ 不支持 |
| 事务 | ✅ MULTI/EXEC | ❌ 不支持 |
| 发布订阅 | ✅ 支持 | ❌ 不支持 |
| Stream | ✅ 支持 | ❌ 不支持 |

## 二、内存管理对比 🔴

### 2.1 Redis 内存管理

```bash
# Redis 使用 jemalloc / tcmalloc / glibc
# 按需分配内存
# 支持内存碎片整理

# 内存配置
maxmemory 10gb
maxmemory-policy allkeys-lru
```

### 2.2 Memcached 内存管理

```bash
# Memcached 使用 slab allocator
# 预先分配固定大小的 slab class
# 每个 slab class 有固定大小的 chunk

# 分配示例：
# slab class 1: 88 bytes
# slab class 2: 112 bytes
# slab class 3: 144 bytes
# ...

# 问题：内存浪费
# 如果数据 100 bytes，会分配 144 bytes 的 chunk
# 44 bytes 被浪费
```

### 2.3 内存使用对比

```bash
# Redis: 精确分配，用多少占多少
SET key "hello"  # 占用约 60 字节

# Memcached: slab class 分配
SET key "hello"  # 可能占用 88 或 112 字节
```

## 三、持久化与集群 🟡

### 3.1 Redis 持久化

```bash
# Redis 支持两种持久化方式
# 1. RDB：定时快照
BGSAVE

# 2. AOF：追加写日志
appendonly yes
appendfsync everysec
```

### 3.2 Memcached 持久化

```bash
# Memcached 不支持持久化
# 数据只存在内存中
# 重启后数据丢失

# 如果需要持久化：
# - 使用 Membase / Couchbase
# - 或者自己实现定期落盘
```

### 3.3 集群对比

| 特性 | Redis Cluster | Memcached (Moxi) |
| --- | --- | --- |
| 分片 | 槽分片 | 一致性哈希 |
| 槽/节点 | 16384 个槽 | 虚拟节点 |
| 故障转移 | 自动 | 需要外部协调 |
| 跨槽操作 | 不支持 | 不支持 |

```bash
# Redis Cluster 槽分片
# 16384 个槽，分布到多个主库

# Memcached 一致性哈希
# 虚拟节点，减少数据倾斜
```

## 四、性能对比 🟡

### 4.1 基准测试

```bash
# Redis vs Memcached 性能测试

# String 类型 GET/SET：
# Redis: 10-15 万 QPS
# Memcached: 30-50 万 QPS

# Memcached 纯内存操作，无持久化开销
# Redis 单线程，CPU 利用率低
```

| 操作 | Redis | Memcached |
| --- | --- | --- |
| GET | 10-15 万 QPS | 30-50 万 QPS |
| SET | 8-12 万 QPS | 20-40 万 QPS |
| MGET (100 keys) | 1-2 万 QPS | 5-10 万 QPS |

### 4.2 性能差异原因

```bash
# 1. 线程模型
# Redis: 单线程，无锁竞争
# Memcached: 多线程，无锁开销

# 2. 持久化
# Redis: 可选开启 AOF
# Memcached: 无持久化

# 3. 数据结构
# Redis: 支持多种数据结构，操作更丰富
# Memcached: 只支持 String
```

## 五、适用场景对比 🟡

### 5.1 适合用 Redis 的场景

```bash
# 1. 需要多种数据结构
# 排行榜：Sorted Set
# 计数器：String INCR
# 分布式锁：SETNX
# 消息队列：List LPUSH/BRPOP

# 2. 需要持久化
# 缓存雪崩：过期时间+随机偏移
# 热点数据：后台预热

# 3. 需要集群
# 主从复制：读写分离
# Redis Cluster：水平扩展
```

### 5.2 适合用 Memcached 的场景

```bash
# 1. 纯 String 缓存
# 页面缓存
# 会话缓存
# 静态资源缓存

# 2. 高并发场景
# QPS 要求极高
# 数据结构简单
# 不需要持久化
```

### 5.3 选型建议

| 场景 | 推荐 | 原因 |
| --- | --- | --- |
| 分布式锁 | Redis | 支持 SETNX |
| 排行榜 | Redis | Sorted Set |
| 计数器 | Redis | INCR 原子操作 |
| 消息队列 | Redis | List/Stream |
| 简单 KV 缓存 | Memcached | 性能更高 |
| 超高并发缓存 | Memcached | 延迟更低 |

## 六、生产选型 🟡

### 6.1 性能 vs 功能

```bash
# 如果业务简单，只缓存 String
# → Memcached 性能更好

# 如果需要丰富的数据结构
# → Redis 功能更强

# 如果需要持久化
# → Redis 必须选择

# 如果需要高可用集群
# → Redis Cluster 或 Redis Sentinel
```

### 6.2 内存效率

```bash
# Memcached slab allocator 内存浪费
# 约 30-50% 内存浪费

# Redis 按需分配
# 内存利用率更高
```

### 6.3 运维复杂度

```bash
# Memcached：简单，无持久化
# - 运维简单
# - 重启即清空
# - 需要预热

# Redis：功能丰富
# - 运维复杂
# - 支持持久化
# - 需要配置淘汰策略
```

:::tip 💡
现代 Web 应用中，Redis 几乎完全替代了 Memcached。只有在极端追求性能、对数据结构没有需求的场景下，才考虑 Memcached。
:::
