# 为什么单线程这么快

很多人第一次听到「Redis 是单线程的」时，都会产生一个疑问：单线程？那它怎么能做到每秒十几万的 QPS？

这个问题看似简单，背后却涉及 I/O 模型、操作系统内核、并发编程等多个领域的知识。理解清楚后，你对高性能服务器的理解会上一个台阶。

## 从一个经典问题开始

> 面试官：Redis 为什么这么快？
>
> 候选人：因为 Redis 是基于内存的。
>
> 面试官：Memcached 也是基于内存的，为什么单机能比 Redis 低那么多？

这个追问点出了关键：内存只是 Redis 高性能的原因之一，另一个关键原因是**单线程 + I/O 多路复用**。

## 传统 I/O 模型的问题

在分析 Redis 的线程模型之前，我们先看看传统 I/O 模型有什么问题。

### 同步阻塞 I/O（BIO）

```python
# 伪代码：传统 BIO 模型
while True:
    client = server.accept()  # 阻塞等待连接
    request = client.recv()   # 阻塞等待数据
    response = handle(request)  # 处理请求
    client.send(response)    # 发送响应
    client.close()
```

这个模型最大的问题是：**一个连接占用一个线程**。

如果有 10000 个并发连接，就需要 10000 个线程。每个线程都需要独立的栈空间、CPU 时间片，线程切换的开销巨大。

```bash
# 查看线程栈大小
$ ulimit -s
8192  # 每个线程默认 8MB 栈空间
```

10000 个线程，光栈空间就需要 80GB，这显然不现实。

### 多线程 I/O 模型

为了解决单线程阻塞的问题，常见做法是使用线程池：

```python
# 伪代码：线程池模型
with ThreadPoolExecutor(max_workers=100) as executor:
    while True:
        client = server.accept()
        executor.submit(handle, client)
```

但这也有问题：

1. **线程创建和销毁有开销**：虽然线程池复用了线程，但仍然需要管理线程生命周期
2. **线程间竞争锁**：共享资源需要加锁，锁竞争会影响性能
3. **内存占用**：每个线程仍然需要栈空间

对于 Redis 这种轻量级操作（一个命令可能只需要几十微秒），线程切换的开销可能比实际处理时间还大。

## I/O 多路复用：解决问题的关键

真正让 Redis 高效的，是 I/O 多路复用技术。

### 什么是 I/O 多路复用

I/O 多路复用允许一个线程同时监控多个文件描述符（socket）的就绪状态。当某个 socket 可读或可写时，内核通知应用程序进行处理。

Linux 提供了三种 I/O 多路复用机制：

| 机制 | 诞生时间 | 特点 |
| --- | --- | --- |
| select | POSIX.1-2001 | 最早实现，但有限制（最多 1024 个 fd） |
| poll | POSIX.1-2001 | 改进了 select 的限制，但仍有性能问题 |
| epoll | Linux 2.6 | 高效的事件驱动模型，无数量限制 |

Redis 在 Linux 上使用 epoll，FreeBSD 上使用 kqueue，Windows 上使用 select。

### epoll 的核心设计

epoll 为什么高效？这要从它的设计说起。

**传统 select/poll 的问题**：

```c
// select 的工作方式
fd_set readfds;
while (true) {
    int n = select(max_fd + 1, &readfds, NULL, NULL, NULL);
    // 问题1：每次调用都要把 fd 数组从用户态拷贝到内核态
    // 问题2：内核要遍历所有 fd，即使大部分都不活跃
    // 问题3：返回后，应用程序不知道哪些 fd 就绪，需要再次遍历
}
```

每次调用都要：1. 把 fd 数组从用户空间拷贝到内核空间
2. 内核遍历所有 fd（即使大部分是"死"的）
3. 返回后用户空间无法直接知道哪些 fd 就绪

**epoll 的改进**：

```c
// epoll 的工作方式
int epfd = epoll_create(1024);  // 创建 epoll 实例
struct epoll_event event;
event.events = EPOLLIN;
event.data.fd = listen_fd;
epoll_ctl(epfd, EPOLL_CTL_ADD, listen_fd, &event);  // 注册监听

struct epoll_event events[1024];
while (true) {
    int n = epoll_wait(epfd, events, 1024, -1);  // 等待事件
    // 关键改进：
    // 1. fd 只注册一次，不用每次拷贝
    // 2. 内核使用回调机制，只返回活跃的 fd
    // 3. 直接返回就绪的 fd 数组，不需要再次遍历
}
```

epoll 的核心优势：

1. **无需每次拷贝**：fd 注册在内核中，只注册一次
2. **高效的事件通知**：使用回调机制，只返回活跃的 fd
3. **时间复杂度 O(1)**：返回的就已经是可用的，不需要再遍历

### 边缘触发 vs 水平触发

epoll 支持两种触发模式：

| 模式 | 说明 | 适用场景 |
| --- | --- | --- |
| 水平触发（LT） | 只要有数据就一直通知 | 普通 I/O，容易编程 |
| 边缘触发（ET） | 数据到达时只通知一次 | 高性能场景，需要循环读取 |

Redis 使用的是**水平触发（LT）**，因为编程更简单，而且 Redis 的命令执行很快，不怕重复触发。

## Redis 的单线程事件循环

理解了 I/O 多路复用，我们再来看 Redis 的事件循环是如何工作的。

### 核心流程

```c
// Redis 事件循环伪代码
void aeMain(aeEventLoop *eventLoop) {
    eventLoop->stop = 0;
    while (!eventLoop->stop) {
        // 处理时间事件（定时任务）
        aeProcessEvents(eventLoop, AE_FILE_EVENTS | AE_TIME_EVENTS);

        // 执行 beforeSleep 回调
        beforeSleep(eventLoop);

        // 再次处理时间事件
        aeProcessEvents(eventLoop, AE_FILE_EVENTS | AE_TIME_EVENTS);
    }
}
```

事件循环中有两种事件：

1. **文件事件**：socket 的可读/可写事件
2. **时间事件**：定时任务，如清理过期 key、同步持久化等

### 处理客户端请求

```c
// 客户端请求处理流程
void readQueryFromClient(aeEventLoop *el, int fd, void *privdata, int mask) {
    // 1. 读取数据
    char *query = readFromSocket(fd);

    // 2. 解析命令
    redisCommand *cmd = lookupCommand(query);

    // 3. 执行命令
    redisClient *client = (redisClient *)privdata;
    call(client, cmd);

    // 4. 清理客户端（如果是 quit 命令）
    if (client->flag.close) {
        freeClient(client);
    }
}
```

整个流程是**同步阻塞**的：一个命令执行完，才执行下一个。这正是 Redis 单线程的关键。

### 为什么单线程能保证原子性

正是因为单线程顺序执行，Redis 的命令具有天然的原子性：

```bash
> INCR counter
(integer) 1

> GETSET counter "0"
"1"

> GET counter
"0"
```

`INCR` 命令不会被其他命令打断，`GETSET` 操作也不会出现竞态条件。这让 Redis 不需要像其他数据库那样为每个操作加锁。

## 单线程的优势：避免锁竞争

在多线程模型中，有一件事是不可避免的：**锁**。

```java
// 多线程环境下的操作
public class Counter {
    private long count = 0;

    public synchronized void increment() {  // 需要加锁
        count++;
    }
}
```

锁的问题：

1. **加锁解锁有开销**：需要操作系统调度
2. **死锁风险**：加锁顺序错误可能导致死锁
3. **线程阻塞**：其他线程等待锁，CPU 利用率下降

Redis 单线程完全避免了这些问题。

## 单线程的代价：CPU 成为瓶颈

单线程虽然避免了锁竞争，但也带来一个问题：**无法利用多核 CPU**。

```bash
# 查看 Redis 单核利用率
$ redis-cli info stats | grep -E "(total_commands|used_cpu_sys)"
total_commands_processed:10000000
used_cpu_sys:85.23
used_cpu_user:12.45
```

如果你的服务器有 16 核，Redis 只能用 1 核，CPU 利用率只有 6.25%。

这就是为什么 Redis 6.0 引入了多线程 I/O。

## 回答最初的问题

现在我们可以完整回答「Redis 为什么这么快」：

**1. 基于内存操作**

内存访问延迟在纳秒级别，比磁盘快 100 倍以上。

**2. I/O 多路复用**

一个线程同时处理多个连接，避免了线程创建和切换的开销。

**3. 单线程避免锁竞争**

没有锁，没有上下文切换，所有精力都用来执行命令。

**4. 高效的数据结构**

SDS、ziplist、quicklist、跳表等数据结构，都是为高性能设计的。

**5. C 语言实现**

底层用 C 语言编写，执行效率高。

```bash
# Redis vs Memcached QPS 对比（官方 benchmark）
# Redis
$ redis-benchmark -t set,get -n 1000000 -c 100
Requests per second: 112000.00

# Memcached（多线程）
$ memcached -t 4
Requests per second: 95000.00
```

Redis 的 QPS 略高于 Memcached，核心原因就是锁竞争和上下文切换的开销更小。

## Redis 6.0 的多线程 I/O

Redis 6.0 引入多线程 I/O，用于解决单线程无法利用多核 CPU 的问题。

### 架构变化

```
之前：
client -> socket -> 主线程（读取 + 执行 + 写入）

之后：
client -> socket -> I/O 线程（读取）-> 主线程（执行）-> I/O 线程（写入）
```

### 配置方式

```bash
# redis.conf
io-threads 4              # I/O 线程数，建议 4-6
io-threads-do-reads yes   # 是否在 I/O 线程处理读操作
```

### 工作原理

```c
// 主线程和 I/O 线程协作
void processInputBuffer(redisClient *c) {
    // 读取阶段：I/O 线程处理
    if (server.io_threads_do_reads) {
        // 将任务分配给 I/O 线程
        for (j = 1; j < server.io_threads_num; j++) {
            io_threads[j].arg = c;
        }
        // I/O 线程并行读取
    }

    // 执行阶段：主线程处理
    while (c->argc > 0) {
        call(c, c->cmd);  // 命令执行仍然在主线程
    }

    // 写入阶段：I/O 线程处理
    if (server.io_threads_do_reads) {
        // I/O 线程并行写入
    }
}
```

:::warning 关键点
即使开启多线程 I/O，**命令执行仍然在主线程**。多线程只用于处理网络 I/O。
:::

### 适用场景

多线程 I/O 主要帮助的场景是：

1. **高并发短连接**：大量客户端频繁连接和断开
2. **大数据包**：一个命令返回几 MB 的数据
3. **高带宽场景**：网络成为瓶颈而非 CPU

对于大部分应用场景（命令执行时间远大于网络传输时间），单线程已经足够。
