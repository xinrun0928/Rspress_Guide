# 数据恢复流程与优先顺序

Redis 重启时如何恢复数据？同时存在 RDB 和 AOF 文件时，Redis 会选择哪个？这节来详细讲解。

## 恢复优先级

### Redis 4.0+ 的规则

```bash
# Redis 4.0+ 的恢复顺序：

# 1. 如果只存在 AOF 文件
#    -> 只加载 AOF 文件

# 2. 如果只存在 RDB 文件
#    -> 只加载 RDB 文件

# 3. 如果同时存在 AOF 和 RDB 文件
#    -> 优先加载 AOF 文件
#    -> 因为 AOF 数据更完整
```

### 为什么优先 AOF

```bash
# 假设有这样的场景：

# RDB 文件（5 分钟前生成）
# 包含数据：user:1 = "Alice"

# AOF 文件（包含 RDB 格式 + 增量）
# 包含数据：user:1 = "Bob"
# （RDB 部分: Alice，增量: SET user:1 "Bob"）

# AOF 数据更新，所以优先加载 AOF
```

### 混合持久化的恢复

```bash
# Redis 6.0 的混合 AOF 文件结构：

┌─────────────────────────────────────────────────────────────┐
│                      混合 AOF 文件                           │
│                                                              │
│  ┌─────────────────┐    ┌─────────────────────────────────┐│
│  │   RDB 格式      │ +  │   AOF 增量                      ││
│  │   快速加载      │    │   重放命令                      ││
│  │   (约占 80%)   │    │   (约占 20%)                    ││
│  └─────────────────┘    └─────────────────────────────────┘│
│                                                              │
│  加载顺序：                                                   │
│  1. 快速加载 RDB 部分                                        │
│  2. 重放 AOF 增量部分                                        │
└─────────────────────────────────────────────────────────────┘
```

## 恢复流程详解

### 场景一：只有 AOF

```bash
# 启动日志
$ redis-server
* DB loaded from append only file: 5.432 MB

# Redis 检测到 AOF 文件
# 自动进行 AOF 重写检查
# 如果 AOF 文件损坏，提示修复
```

### 场景二：只有 RDB

```bash
# 启动日志
$ redis-server
* DB loaded from disk: 10.234 MB

# Redis 检测到 RDB 文件
# 直接加载
```

### 场景三：同时存在 AOF 和 RDB

```bash
# 启动日志
$ redis-server
* DB loaded from append only file: 12.567 MB (fast mode)
# 优先加载 AOF

# 即使 RDB 更新，也不会加载 RDB
# 因为 AOF 数据更完整
```

### 场景四：都没有

```bash
# 启动日志
$ redis-server
# 没有持久化文件
# 以空数据库启动

# 如果配置了 requirepass，可能需要认证
```

## 手动恢复

### 从 RDB 恢复

```bash
# 方法一：指定 RDB 文件启动
redis-server --dir /data --dbfilename dump.rdb

# 方法二：复制 RDB 文件到数据目录
cp /backup/dump.rdb /var/lib/redis/dump.rdb
redis-server

# 方法三：使用 redis-cli 导入
redis-cli --rdb /backup/dump.rdb
# 输出：SYNC sent to server, reply: OK
```

### 从 AOF 恢复

```bash
# 方法一：确保 AOF 开启
redis-server --appendonly yes

# 方法二：复制 AOF 文件到数据目录
cp /backup/appendonly.aof /var/lib/redis/appendonly.aof
redis-server

# 方法三：修复损坏的 AOF
redis-check-aof --fix /var/lib/redis/appendonly.aof
```

### 从远程恢复

```bash
# 从 S3 下载备份
aws s3 cp s3://my-bucket/redis/dump.rdb /tmp/dump.rdb

# 恢复
cp /tmp/dump.rdb /var/lib/redis/dump.rdb
redis-server
```

## AOF 损坏处理

### 检查 AOF 完整性

```bash
# 检查 AOF 文件
redis-check-aof --fix appendonly.aof

# 输出示例
AOF analyzed: size=104857600, ok_up_to=104857600, diff=0
AOF is valid
```

### 修复损坏的 AOF

```bash
# 如果有损坏
redis-check-aof --fix appendonly.aof

# 输出示例
AOF analyzed: size=104857600, ok_up_to=100000000, diff=4857100
0x               5f3a2b1: Expected prefix '*', got '?'

AOF will be rewritten to 100000000 bytes

# 确认修复
redis-check-aof --fix appendonly.aof --diff

# 备份原始文件
cp appendonly.aof appendonly.aof.bak

# 执行修复
redis-check-aof --fix appendonly.aof
```

### 丢失部分数据的评估

```bash
# 修复后可能丢失部分数据
# 评估影响

# 查看丢失的范围
redis-check-aof --fix appendonly.aof --verbose

# 手动检查丢失的命令
# 确认业务影响
```

## RDB 损坏处理

```bash
# 检查 RDB 文件
redis-check-rdb dump.rdb

# 修复（Redis 不支持自动修复 RDB）
# 需要从备份恢复

# 如果 RDB 损坏但 AOF 正常
# 删除 RDB 文件，只保留 AOF
rm dump.rdb
redis-server
```

## 故障恢复演练

### 场景：主节点完全故障

```bash

# dump_2024-03-15.rdb
# dump_2024-03-14.rdb
# appendonly_2024-03-15.aof

# 2. 选择最新备份
```

## Java 恢复脚本

```java
@Service
public class DataRecoveryService {
    
    @Autowired
    private RedisTemplate<String, String> redisTemplate;
    
    /**
     * 执行数据恢复
     */
    public RecoveryResult recoverFromBackup(String backupPath) {
        RecoveryResult result = new RecoveryResult();
        
        try {
            // 1. 检查备份文件
            File backupFile = new File(backupPath);
            if (!backupFile.exists()) {
                result.setSuccess(false);
                result.setMessage("备份文件不存在: " + backupPath);
                return result;
            }
            
            // 2. 备份当前数据
            String currentBackup = backupCurrentData();
            result.setCurrentBackup(currentBackup);
            
            // 3. 复制备份文件
            String redisDir = getRedisDataDir();
            String fileName = backupFile.getName();
            Path targetPath = Paths.get(redisDir, fileName);
            Files.copy(backupFile.toPath(), targetPath, StandardCopyOption.REPLACE_EXISTING);
            
            // 4. 重启 Redis
            restartRedis();
            
            // 5. 验证恢复
            boolean verified = verifyRecovery();
            result.setSuccess(verified);
            result.setMessage(verified ? "恢复成功" : "恢复后验证失败");
            
            // 6. 清理临时文件
            if (verified) {
                // 可以删除临时备份 currentBackup
            } else {
                // 回滚
                rollbackRecovery(currentBackup);
            }
            
        } catch (Exception e) {
            result.setSuccess(false);
            result.setMessage("恢复失败: " + e.getMessage());
            log.error("数据恢复失败", e);
        }
        
        return result;
    }
    
    /**
     * 验证恢复结果
     */
    private boolean verifyRecovery() {
        try {
            // 检查连接
            redisTemplate.getConnectionFactory().getConnection().ping();
            
            // 检查关键数据
            Long dbsize = redisTemplate.getConnectionFactory()
                .getConnection().dbSize();
            
            return dbsize != null && dbsize > 0;
        } catch (Exception e) {
            log.error("验证恢复结果失败", e);
            return false;
        }
    }
}
```

## 总结

Redis 数据恢复的关键点：

1. **优先顺序**：AOF > RDB（数据更完整）
2. **混合持久化**：RDB 快速加载 + AOF 增量
3. **损坏处理**：使用 redis-check-aof/rdb 检查和修复
4. **备份策略**：本地 + 远程，多地备份
5. **恢复演练**：定期测试，确保备份可用

做好备份和恢复预案，才能在故障时快速恢复服务。
