# 时序数据库详解：数据驱动的新物种

凌晨 3 点，你被手机告警叫醒。

监控大屏上闪烁着刺眼的红色：某核心服务的 CPU 使用率飙升至 95%，内存使用率 92%，磁盘 I/O 等待严重。

你开始排查：

- 最近部署了什么？两个小时前上线了一个新功能
- 流量有没有异常？是的，翻了 3 倍
- 哪个接口压力大？用户下单接口
- 下单接口调用了什么服务？用户服务、库存服务、支付服务

你打开监控面板，查看各个服务的时序指标：

- CPU 使用率曲线，看哪个时间点开始飙升
- 内存使用趋势，看是否有内存泄漏
- 服务间调用延迟分布，看哪个环节是瓶颈
- 数据库慢查询数量，看是否数据库拖累了整个系统

这些时序指标数据，正是时序数据库的专长领域。

## 时序数据是什么？

时序数据（Time Series Data）是按照时间顺序记录的数据点的集合。每个数据点由时间戳和度量值组成。

### 时序数据的四个核心概念

```
┌─────────────────────────────────────────────────────────────┐
│                    时序数据模型                              │
│                                                             │
│   时间线（Time Series）                                      │
│   ┌─────────────────────────────────────────────────────┐   │
│   │  Metric: cpu_usage                                  │   │
│   │  Tags:  host=server-01, region=beijing             │   │
│   │                                                     │   │
│   │  Data Points:                                       │   │
│   │  ┌──────────────┬────────────────────────────┐     │   │
│   │  │  timestamp    │  value                    │     │   │
│   │  ├──────────────┼────────────────────────────┤     │   │
│   │  │ 1700000000   │  75.5                      │     │   │
│   │  │ 1700000010   │  78.2                      │     │   │
│   │  │ 1700000020   │  72.1                      │     │   │
│   │  │ 1700000030   │  80.5                      │     │   │
│   │  └──────────────┴────────────────────────────┘     │   │
│   └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

| 概念 | 说明 | 示例 |
| --- | --- | --- |
| 时间线（Metric/Series） | 同一个测量指标的所有数据点序列 | cpu_usage, memory_usage |
| 度量（Measurement） | 测量的物理量或指标 | cpu_usage, stock_price |
| 标签（Tag） | 描述数据的维度属性 | host, region, service_name |
| 数据点（Data Point） | 时间戳 + 数值的组合 | (1700000000, 75.5) |

### 时序数据的特点

**1. 数据按时间顺序写入**

时序数据通常是持续不断产生的，写入具有时间顺序性。

**2. 数据量大且增长快**

一个监控系统可能有成千上万个指标，每个指标每秒钟产生一个数据点。一天的数据量轻易达到数亿条。

**3. 主要查询是聚合和统计**

时序数据很少查询单条记录，通常是对一段时间范围内的数据进行聚合：求平均、获取最大值、计算百分位数。

**4. 数据通常有生命周期**

监控数据一般保留 7-30 天，日志数据保留 30-90 天。过期数据需要自动清理。

**5. 写多读少（相对）**

时序数据的写入远多于读取，但每次读取可能涉及大量数据的聚合计算。

## 为什么要用时序数据库？

传统关系型数据库也能存储时序数据，但在面对海量时序数据时会遇到瓶颈：

| 问题 | 说明 | 影响 |
| --- | --- | --- |
| 写入性能差 | 传统数据库优化随机读写，时序数据是顺序写入 | 每秒只能写入几千条 |
| 存储成本高 | 关系模型冗余多，压缩率低 | 存储成本成倍增加 |
| 查询效率低 | 没有针对时间范围查询优化 | 聚合查询耗时过长 |
| 维护困难 | 分区策略需要手动设计 | DDL 操作复杂 |
| 无法压缩 | 没有针对时序数据的压缩算法 | 占用大量磁盘空间 |

时序数据库专门为时序数据设计，解决了这些问题：

- **写入优化**：采用 LSM-Tree 或类似结构，支持高速顺序写入
- **高效压缩**：针对时序数据特点的专用压缩算法
- **时间范围查询**：按时间排序存储，范围查询极快
- **数据保留策略**：自动过期清理，降低维护成本
- **聚合计算**：内置丰富的聚合函数

## 时序数据库 vs 关系数据库

### 数据模型对比

**关系型数据库**：

```sql
-- 监控数据存储
CREATE TABLE metrics (
    id BIGINT PRIMARY KEY,
    metric_name VARCHAR(100),
    host VARCHAR(100),
    region VARCHAR(50),
    value DOUBLE,
    timestamp TIMESTAMP
);

-- 查询某个主机的 CPU 使用率
SELECT timestamp, value
FROM metrics
WHERE metric_name = 'cpu_usage'
  AND host = 'server-01'
  AND timestamp BETWEEN '2024-01-01' AND '2024-01-02';
```

**时序数据库**（以 InfluxDB 为例）：

```
Measurement: metrics
Tags: host, region
Fields: value
Timestamp: 时间戳

> SELECT value FROM metrics WHERE host='server-01' AND time > now() - 1d
```

### 核心区别

| 维度 | 关系型数据库 | 时序数据库 |
| --- | --- | --- |
| 数据模型 | 行式存储，灵活 Schema | 列式存储，按时间排序 |
| 写入方式 | 随机读写为主 | 顺序写入为主 |
| 查询模式 | 随机点查为主 | 范围聚合为主 |
| 数据保留 | 通常永久保留 | 支持自动过期 |
| 压缩能力 | 有限 | 高压缩比 |
| 扩展性 | 垂直扩展为主 | 水平扩展 |

:::tip 时序数据库的本质
时序数据库并不是"新的物种"，而是在特定场景下针对特定数据特点进行深度优化的数据库。

在数据量不大时，用关系型数据库完全可以。但当数据量达到一定规模，时序数据库的优势就显现出来了。
:::

## 主流时序数据库横评

### InfluxDB：时序数据库的标杆

InfluxDB 是最流行的开源时序数据库，专为 DevOps、IoT、实时分析场景设计。

**核心特点：**

- **专为时序优化**：LSM-Tree 存储引擎，高写入性能
- **丰富的数据类型**：支持 Float、Integer、String、Boolean
- **强大的查询语言**：InfluxQL 和 Flux（类 SQL）
- **内置函数丰富**：支持 COUNT、SUM、MEAN、MEDIAN 等聚合
- **数据保留策略**：自动过期清理

```influxdb
-- InfluxDB 查询示例
SELECT MEAN(cpu_usage)
FROM metrics
WHERE time > now() - 7d
  AND host =~ /server-.*/
GROUP BY time(1h), host
```

**不适用的场景**：不适合作为应用的主数据库，不支持复杂的 JOIN 操作。

### TimescaleDB：PostgreSQL 的时序扩展

TimescaleDB 是 PostgreSQL 的扩展，兼具关系数据库的能力和时序数据库的性能。

**核心特点：**

- **基于 PostgreSQL**：完全兼容 SQL，支持 JOIN、子查询
- **Hypertable**：自动分区到 chunk，适合时序数据
- **标准 SQL**：学习成本低，可以复用现有 PostgreSQL 知识
- **丰富生态**：可以与 PostgreSQL 工具链无缝集成

```sql
-- TimescaleDB 查询示例
SELECT time_bucket('1 hour', timestamp) AS hour,
       avg(cpu_usage) AS avg_cpu,
       percentile_cont(0.95) WITHIN GROUP (ORDER BY cpu_usage) AS p95
FROM metrics
WHERE timestamp > now() - INTERVAL '7 days'
GROUP BY hour
ORDER BY hour;
```

### TDengine：国产时序数据库

TDengine 是涛思数据开发的国产时序数据库，专为物联网数据场景优化。

**核心特点：**

- **超高性能**：相比 InfluxDB，写入性能提升 10 倍以上
- **超级表设计**：支持标签+数据分离，适合海量设备
- **消息队列**：内置消息队列功能
- **边缘计算**：支持边缘部署
- **低资源占用**：相比 InfluxDB，存储空间减少 10 倍

```sql
-- TDengine 查询示例
SELECT AVG(cpu_usage), MAX(cpu_usage)
FROM metrics
WHERE tbname IN ('server-01', 'server-02')
  AND ts > NOW() - 7d
PARTITION BY tbname
INTERVAL (1h);
```

### Prometheus：监控领域的王者

Prometheus 是 CNCF 的毕业项目，专为云原生环境的服务监控设计。

**核心特点：**

- **Pull 模式**：Prometheus 主动拉取指标数据
- **PromQL**：强大的指标查询语言
- **多维数据模型**：支持标签维度，数据灵活
- **服务发现**：自动发现监控目标
- **告警管理**：内置 AlertManager

```promql
# PromQL 查询示例
rate(http_requests_total[5m])
  * on (service) group_left(instance)
avg(cpu_usage{service=~"user.*"})
```

:::warning Prometheus 的局限
Prometheus 是为监控场景设计的，不是通用时序数据库：
- 不适合存储原始时序数据（数据量有限制）
- 不适合需要长期存储的分析场景
- 不支持复杂的跨指标关联查询
:::

### 时序数据库横向对比

| 特性 | InfluxDB | TimescaleDB | TDengine | Prometheus |
| --- | --- | --- | --- | --- |
| **部署方式** | 单机/集群 | PostgreSQL 扩展 | 单机/集群 | 联邦 |
| **查询语言** | InfluxQL/Flux | SQL | TAOS SQL | PromQL |
| **SQL 支持** | 有限 | 完全支持 | 完全支持 | 否 |
| **JOIN 支持** | 否 | 支持 | 有限支持 | 否 |
| **数据压缩** | TSM | Chunk 压缩 | 行列混合压缩 | 4字节压缩 |
| **生态集成** | 丰富 | PostgreSQL 生态 | 物联网 | Kubernetes |
| **License** | MIT | Apache | AGPL | Apache |
| **适用场景** | 通用监控 | 混合场景 | IoT | 服务监控 |

## 时序数据库的核心概念

### 时间线（Time Series）

时间线是时序数据库的核心概念。一个时间线由以下要素唯一标识：

```
时间线标识 = Measurement + Tags（标签集合）
```

```influxdb
-- 两条不同的时间线
measurement=metrics, host=server-01, region=beijing
measurement=metrics, host=server-01, region=shanghai
measurement=metrics, host=server-02, region=beijing
```

### 数据聚合

时序数据库的核心价值在于高效的数据聚合：

```influxdb
-- 按小时聚合
SELECT MEAN(value) FROM metrics GROUP BY time(1h)

-- 按多个维度聚合
SELECT MEAN(cpu), MEAN(memory) FROM metrics GROUP BY host, region

-- 滑动窗口
SELECT MEAN(value) FROM metrics GROUP BY time(5m) SLIDE BY 1m
```

### 降采样（Downsampling）

高频数据降采样为低频数据，节省存储空间：

```influxdb
-- 原始数据：每秒一条
-- 降采样后：每分钟一条
SELECT MEAN(value) INTO metrics_1m
FROM metrics
WHERE time > NOW() - 7d
GROUP BY time(1m), *
```

### 按场景选择

| 场景 | 推荐选择 | 理由 |

### 选型决策树

```
需要存储时序数据？
        │
        ├── 是否需要长期存储（>30天）？
        │       ├── 否 ──▶ Prometheus
        │       └── 是 ──▶ 下一步
        │
        ├── 是否需要复杂 SQL？
        │       ├── 是 ──▶ TimescaleDB
        │       └── 否 ──▶ 下一步

        │       ├── < 100万点/秒 ──▶ InfluxDB
        │       ├── > 1000万点/秒 ──▶ TDengine

        └── 是否需要 JOIN？
```

## 时序数据库的技术原理

### 写入流程

时序数据库通常采用 LSM-Tree 或类似结构来优化写入：

```
写入请求
    │
    ▼
① 写入内存缓冲区（MemTable）
   - 按时间排序
   - 内存操作，速度极快
    │
    ▼
② 内存缓冲区写满
   - 转换为不可变的 MemTable (SSTable)
    │
    ▼
③ 后台 Compaction
   - 合并多个 SSTable
   - 删除过期数据
   - 优化存储布局
```

### 数据压缩

时序数据的压缩比通常可以达到 **10:1** 甚至更高：

| 压缩技术 | 说明 |
| --- | --- |
| Delta 编码 | 存储时间戳差值，而非绝对值 |
| XOR 编码 | 利用相邻值相似的特点 |
| 浮点压缩 | 浮点数只存储尾数 |
| 字典编码 | 重复的标签值只存储一次 |

### 时间分区

数据按时间范围分区，每个分区独立管理：

```
┌─────────────────────────────────────────────────────────────┐
│                    Table: metrics                            │
│                                                             │
│  Partition 1 (2024-01)  │  Partition 2 (2024-02)  │ ...  │
│  ┌───────────────────┐ │  ┌───────────────────┐ │       │
│  │ Chunk 1           │ │  │ Chunk 1           │ │       │
│  │ Chunk 2           │ │  │ Chunk 2           │ │       │
│  │ Chunk 3           │ │  │ Chunk 3           │ │       │
│  └───────────────────┘ │  └───────────────────┘ │       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

分区的优势：

- **删除方便**：DROP 分区即可清理过期数据
- **查询优化**：只扫描相关分区
- **并行处理**：不同分区可以并行查询

## 时序数据库的应用场景

### 场景一：基础设施监控

监控服务器的 CPU、内存、磁盘、网络等指标。

```influxdb
-- 监控告警：CPU 持续 5 分钟 > 90%
SELECT MEAN(cpu_usage) FROM metrics
WHERE host = 'server-01'
  AND time > NOW() - 5m
```

### 场景二：应用性能监控（APM）

追踪微服务的请求量、延迟、错误率。

```influxdb
-- 分析 P99 延迟
SELECT PERCENTILE(duration, 99)
FROM http_requests
WHERE service = 'order-service'
  AND time > NOW() - 1h
GROUP BY time(1m), endpoint
```

### 场景三：IoT 物联网

采集传感器数据，分析设备状态。

```influxdb
-- 统计每个工厂的设备告警数量
SELECT COUNT(*) FROM alerts
WHERE time > NOW() - 24h
GROUP BY factory_id, alert_type
```

### 场景四：金融行情

存储股票价格、交易量等金融数据。

```sql
-- 计算某股票的五日均线
SELECT symbol, time_bucket('1 day', ts) AS day,
       AVG(close) AS avg_close
FROM stock_prices
WHERE symbol = 'AAPL'
  AND ts > NOW() - INTERVAL '5 days'
GROUP BY symbol, day
ORDER BY day;
```

### 场景五：用户行为分析

分析用户点击流、页面访问量。

```promql
# Prometheus 查询：每秒请求量
rate(http_requests_total[5m])
```
