# 时序数据库（TSDB）

想象一下这个场景：

你负责维护一个监控系统，需要记录全国 10 万台服务器的 CPU、内存、磁盘、网络等指标。每台服务器每秒上报一次数据，那就是 **10 万条/秒** 的写入速度，24 小时不间断。

一天下来，就是 **86.4 亿条数据**。

用 MySQL 存？行数爆炸，查询慢，存储成本高。

用 MongoDB 存？时序聚合能力弱，存储效率低。

这就是**时序数据库（Time Series Database，TSDB）** 诞生的背景。

## 时序数据的核心特征

| 特征 | 说明 | 对传统数据库的影响 |
|------|------|-------------------|
| 数据量大 | 持续写入，永不停止 | 行数爆炸 |
| 写入多，读取少 | 主要是采集写入，定期查询 | 写性能差 |
| 时间有序 | 时间是主维度 | B-Tree 不最优 |
| 冷热分明 | 最近数据热，历史数据冷 | 无法分层存储 |
| 需要聚合 | 经常需要按时间窗口聚合 | 聚合函数弱 |
| 保留策略 | 数据过期自动删除 | 无生命周期管理 |

## 四大 TSDB 对比速查

| 维度 | InfluxDB | Prometheus | TDengine | TimescaleDB |
|------|----------|------------|----------|-------------|
| **定位** | 通用时序 | 监控告警 | 物联网 | PostgreSQL 扩展 |
| **写入性能** | ~100 万/秒 | ~100 万/秒 | ~500 万/秒 | ~10 万/秒 |
| **查询语言** | InfluxQL/Flux | PromQL | SQL | SQL |
| **集群** | 企业版 | 原生支持 | 开源 | 企业版 |
| **SQL 兼容** | 部分 | ❌ | ✅ | ✅ 完整 |
| **生态** | 丰富 | 极丰富 | 发展中 | 依赖 PG |

**选型建议**：
- Kubernetes 监控 → **Prometheus**
- IoT/车联网（高写入） → **TDengine**
- 通用监控 + 生态 → **InfluxDB**
- 已有 PostgreSQL 团队 → **TimescaleDB**

## 核心公式

```
时序数据 = 时间戳 + 指标值 + 维度标签
指标（Metric）= Metric Name + Labels + Value + Timestamp

InfluxDB 数据模型 = Measurement + Tags（索引） + Fields（不索引） + Timestamp
Prometheus 数据模型 = Metric Name + Labels（键值对）
TDengine 数据模型 = 超级表 + 子表 + TAGS

时序查询 = 时间范围过滤 + 标签过滤 + 聚合函数（AVG/SUM/MAX/MIN）
降采样 = 高精度原始数据 → 低精度聚合数据（节省存储）
保留策略 = 数据按时间过期自动删除

写入优化 = 批量写入 + 顺序追加 + 列式压缩
查询优化 = 善用标签索引 + 限制时间范围 + 预聚合
```
