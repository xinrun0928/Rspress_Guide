# 告警规则与 AlertManager

告警规则定义了「什么时候告警」，AlertManager 负责「告警发到哪里」。两者配合，才算完整的告警闭环。

## 告警规则

### 基本结构

```yaml title="alert_rules.yml"
groups:
  - name: <group_name>
    interval: <evaluation_interval>  # 评估间隔
    rules:
      - alert: <alert_name>           # 告警名称
        expr: <promql_expression>      # 触发条件
        for: <duration>               # 持续时间
        labels:                       # 标签
          <label>: <value>
        annotations:                  # 注释
          <annotation>: <value>
```

### 简单告警示例

```yaml title="cpu_alert.yml"
groups:
  - name: node_alerts
    rules:
      # 主机 CPU 使用率 > 80% 持续 5 分钟
      - alert: HighCPUUsage
        expr: 100 - (avg by (instance) (rate(node_cpu_seconds_total{mode="idle"}[5m])) * 100) > 80
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "High CPU usage detected"
          description: "Instance {{ $labels.instance }} CPU usage is {{ $value | printf \"%.2f\" }}%"
```

### 关键参数

| 参数 | 说明 | 示例 |
|---|---|---|
| `alert` | 告警名称 | `HighCPUUsage` |
| `expr` | PromQL 表达式 | `cpu > 80` |
| `for` | 持续时间 | `5m`、`10m` |
| `labels` | 标签，用于分类和路由 | `severity: critical` |
| `annotations` | 注释，用于告警内容 | `summary: "CPU 过高"` |

## 常用告警规则

### 1. 服务可用性

```yaml title="availability_alerts.yml"
groups:
  - name: availability
    rules:
      # 服务宕机
      - alert: TargetDown
        expr: up == 0
        for: 1m
        labels:
          severity: critical
        annotations:
          summary: "Target {{ $labels.job }} is down"
          description: "{{ $labels.job }} has been down for more than 1 minute"

      # 服务恢复
      - alert: TargetUp
        expr: up == 1
        for: 0m
        labels:
          severity: info
        annotations:
          summary: "Target {{ $labels.job }} is up"
```

### 2. HTTP 服务

```yaml title="http_alerts.yml"
groups:
  - name: http_alerts
    rules:
      # 错误率 > 1%
      - alert: HighErrorRate
        expr: |
          sum(rate(http_requests_total{status=~"5.."}[5m])) by (job)
          /
          sum(rate(http_requests_total[5m])) by (job)
          > 0.01
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "High error rate on {{ $labels.job }}"
          description: "Error rate is {{ $value | printf \"%.2f\" }}%"

      # P99 延迟 > 1 秒
      - alert: HighLatency
        expr: |
          histogram_quantile(0.99,
            sum(rate(http_request_duration_seconds_bucket[5m])) by (le, job)
          ) > 1
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "High latency on {{ $labels.job }}"
          description: "P99 latency is {{ $value | printf \"%.2f\" }}s"
```

### 3. 资源使用

```yaml title="resource_alerts.yml"
groups:
  - name: resource_alerts
    rules:
      # 内存使用率 > 90%
      - alert: HighMemoryUsage
        expr: |
          (node_memory_MemTotal_bytes - node_memory_MemAvailable_bytes)
          / node_memory_MemTotal_bytes * 100 > 90
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "High memory usage on {{ $labels.instance }}"
          description: "Memory usage is {{ $value | printf \"%.2f\" }}%"

      # 磁盘使用率 > 85%
      - alert: HighDiskUsage
        expr: |
          node_filesystem_usage_bytes{mountpoint="/"} / node_filesystem_size_bytes{mountpoint="/"} * 100 > 85
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "High disk usage on {{ $labels.instance }}"
          description: "Disk usage is {{ $value | printf \"%.2f\" }}%"
```

### 4. 业务指标

```yaml title="business_alerts.yml"
groups:
  - name: business_alerts
    rules:
      # 订单量异常低
      - alert: LowOrderRate
        expr: |
          sum(rate(orders_total[5m])) by (job) < 10
        for: 10m
        labels:
          severity: warning
        annotations:
          summary: "Low order rate"
          description: "Order rate is {{ $value | printf \"%.2f\" }}/s"

      # 用户登录失败率 > 10%
      - alert: HighLoginFailureRate
        expr: |
          sum(rate(login_failures_total[5m])) by (job)
          /
          sum(rate(login_attempts_total[5m])) by (job) > 0.1
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "High login failure rate"
          description: "Login failure rate is {{ $value | printf \"%.2f\" }}%"
```

## AlertManager 配置

### 基本结构

```yaml title="alertmanager.yml"
global:
  resolve_timeout: 5m

route:
  receiver: 'default'
  group_by: ['alertname', 'cluster']
  group_wait: 30s
  group_interval: 5m
  repeat_interval: 4h

receivers:
  - name: 'default'
    # 通知配置
```

```

│                       /          \                                  │

│            receiver: 'database-team'                               │

│        receiver: 'warning-alerts'                                  │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 路由配置

```yaml title="alertmanager_routing.yml"
route:
  receiver: 'default-receiver'
  group_by: ['alertname', 'cluster', 'service']
  group_wait: 30s          # 等待多久聚合告警
  group_interval: 5m       # 告警组发送间隔
  repeat_interval: 4h      # 重复告警间隔
  routes:
    # Critical 告警
    - match:
        severity: critical
      receiver: 'critical-team'
      group_wait: 10s
      repeat_interval: 1h

    # Database 告警
    - match:
        component: database
      receiver: 'database-team'
      continue: true       # 继续匹配后续规则

    # 调试模式
    - match:
        env: dev
      receiver: 'dev-team'
```

## 通知配置

### Email

```yaml title="email_config.yml"
receivers:
  - name: 'email-team'
    email_configs:
      - to: 'team@example.com'
        send_resolved: true
        headers:
          subject: '{{ if eq .Status "firing" }}🚨{{ else }}✅{{ end }} {{ .GroupLabels.alertname }}'
        html: |
          <h2>Alert: {{ .GroupLabels.alertname }}</h2>
          <p>{{ range .Alerts }}{{ .Annotations.description }}{{ end }}</p>
```

### Slack

```yaml title="slack_config.yml"
receivers:
  - name: 'slack-team'
    slack_configs:
      - channel: '#alerts'
        send_resolved: true
        api_url: 'https://hooks.slack.com/services/xxx'
        title: |
          {{ if eq .Status "firing" }}🚨{{ else }}✅{{ end }} {{ .GroupLabels.alertname }}
        text: |
          {{ range .Alerts }}
          {{ if .Annotations.description }}{{ .Annotations.description }}{{ end }}
          {{ end }}
        color: '{{ if eq .Status "firing" }}danger{{ else }}good{{ end }}'
```

### PagerDuty

```yaml title="pagerduty_config.yml"
receivers:
  - name: 'pagerduty-critical'
    pagerduty_configs:
      - service_key: 'your-pagerduty-integration-key'
        severity: critical
        send_resolved: true
        details:
          alertname: '{{ .GroupLabels.alertname }}'
          instance: '{{ .GroupLabels.instance }}'
        class: 'monitoring_alert'
```

### Webhook

```yaml title="webhook_config.yml"

  - name: 'webhook'
    webhook_configs:
      - url: 'http://custom-alerting-service:5000/alerts'
        send_resolved: true
```

## 高级特性

### 1. 抑制规则

```yaml title="inhibit_rules.yml"
inhibit_rules:
  # 当 TargetDown 告警触发时，抑制其他相关告警
  - source_match:
      alertname: TargetDown
    source_match_re:
      severity: critical
    target_match:
      severity: warning
    target_match_re:
      alertname: .*
    equal: ['instance', 'job']
```

### 2. 告警分组

```yaml title="grouping.yml"
# 按服务分组
route:
  group_by: ['service', 'severity']
  group_wait: 30s
  group_interval: 5m

# 示例:
# 所有 database 相关的 critical 告警归为一组
# 避免告警风暴
```

### 3. 静默规则

```bash
# 创建静默规则
amtool silence add --alertmanager.url=http://alertmanager:9093 \
  --match alertname=HighCPUUsage \
  --match instance=server-01 \
  --duration 1h \
  --reason "Scheduled maintenance"

# 查看静默规则
amtool silence query --alertmanager.url=http://alertmanager:9093

# 过期静默规则
amtool silence expire <silence_id> --alertmanager.url=http://alertmanager:9093
```

## 最佳实践

### 1. 合理设置 for

```yaml
# ❌ 过于敏感 (没有 for)
# 瞬时抖动也会告警
- alert: HighCPU
  expr: cpu > 80
  for: 0m

# ✅ 合理 (5m for)
# 持续 5 分钟才告警
- alert: HighCPU
  expr: cpu > 80
  for: 5m
```

### 2. 避免告警风暴

```yaml
# 按标签分组聚合
route:
  group_by: ['alertname', 'cluster', 'service']
  group_wait: 30s
  group_interval: 5m
```

### 3. 分类处理

```yaml
# Critical: 立即通知
- alert: ServiceDown
  expr: up == 0
  for: 1m
  labels:
    severity: critical
  receivers:
    - name: 'pagerduty-critical'

# Warning: 延迟通知
- alert: HighLatency
  expr: latency > 1s
  for: 10m
  labels:
    severity: warning
  receivers:
    - name: 'slack-warning'
```

### 4. 告警收敛

```yaml
# 使用 recording rule 预计算
groups:
  - name: recording_rules
    rules:
      - record: job:error_rate:5m
        expr: |
          sum(rate(http_requests_total{status=~"5.."}[5m])) by (job)
          /
          sum(rate(http_requests_total[5m])) by (job)

# 告警规则使用预计算结果
- alert: HighErrorRate
  expr: job:error_rate:5m > 0.05
  for: 5m
```

## 总结

Prometheus 告警系统包含两个核心组件：

1. **告警规则**：定义何时触发告警
   - `expr`：触发条件（PromQL 表达式）
   - `for`：持续时间
   - `labels`：分类标签
   - `annotations`：告警内容

2. **AlertManager**：处理告警通知
   - 路由树：告警分发
   - 分组聚合：避免告警风暴
   - 抑制规则：避免重复告警
   - 多种通知渠道：Email、Slack、PagerDuty、Webhook

合理的告警设计，需要平衡「及时发现」和「避免骚扰」之间的关系。
