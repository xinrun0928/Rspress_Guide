# Pull vs Push 采集模式

监控数据的采集方式，直接影响系统的架构设计和运维复杂度。

Prometheus 采用 Pull 模式，而很多其他监控系统采用 Push 模式。两种模式各有优劣。

## Pull 模式原理

### 工作流程

```
┌─────────────────────────────────────────────────────────────────────┐
│                        Pull 模式工作流程                             │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  1. 配置 Target 列表                                                │
│     ┌─────────────────────────────────────────────────────────┐    │
│     │ prometheus.yml:                                         │    │
│     │   scrape_configs:                                       │    │
│     │     - job_name: 'api-server'                            │    │
│     │       static_configs:                                   │    │
│     │         - targets: ['api-server:8080']                  │    │
│     └─────────────────────────────────────────────────────────┘    │
│                              │                                      │
│                              ▼                                      │
│  2. Prometheus 定时拉取                                            │
│     ┌─────────────────────────────────────────────────────────┐    │
│     │                                                         │    │
│     │   每 15 秒: GET http://api-server:8080/metrics        │    │
│     │                                                         │    │
│     │   响应:                                                 │    │
│     │   # HELP http_requests_total ...                       │    │
│     │   http_requests_total{method="GET"} 12345              │    │
│     │   http_requests_total{method="POST"} 6789              │    │
│     │                                                         │    │
│     └─────────────────────────────────────────────────────────┘    │
│                              │                                      │
│                              ▼                                      │
│  3. 存储到 TSDB                                                   │
│     ┌─────────────────────────────────────────────────────────┐    │
│     │   时间序列数据:                                         │    │
│     │   {__name__="http_requests_total", method="GET"}        │    │
│     │     → [(ts1, 12345), (ts2, 12347), ...]                │    │
│     └─────────────────────────────────────────────────────────┘    │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 服务发现

Prometheus 支持多种服务发现机制，自动感知监控目标的变化：

```yaml title="prometheus.yml"
# 静态配置
scrape_configs:
  - job_name: 'static-targets'
    static_configs:
      - targets: ['server1:9100', 'server2:9100']

# Kubernetes 服务发现
  - job_name: 'kubernetes-nodes'
    kubernetes_sd_configs:
      - role: node

# Consul 服务发现
  - job_name: 'consul-services'
    consul_sd_configs:
      - server: 'consul:8500'
        services:
          - 'api'
          - 'web'

# DNS 服务发现
  - job_name: 'dns-discovery'
    dns_sd_configs:
      - names: ['api.svc.cluster.local']
        type: A
        port: 8080
```

## Push 模式原理

### 工作流程

```
┌─────────────────────────────────────────────────────────────────────┐
│                        Push 模式工作流程                             │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  1. 客户端埋点                                                      │
│     ┌─────────────────────────────────────────────────────────┐    │
│     │                                                         │    │
│     │   // 每次请求时                                         │    │
│     │   requestCounter.Inc();                                 │    │
│     │                                                         │    │
│     │   // 记录延迟                                           │    │
│     │   latencyHistogram.Observe(0.023);                      │    │
│     │                                                         │    │
│     └─────────────────────────────────────────────────────────┘    │
│                              │                                      │
│                              ▼                                      │
│  2. 推送指标到 Pushgateway/Gateway                                  │
│     ┌─────────────────────────────────────────────────────────┐    │
│     │                                                         │    │
│     │   POST http://pushgateway:9091/metrics/job/api-server   │    │
│     │   Content-Type: text/plain                              │    │
│     │                                                         │    │
│     │   http_requests_total{method="GET"} 12345               │    │
│     │                                                         │    │
│     └─────────────────────────────────────────────────────────┘    │
│                              │                                      │
│                              ▼                                      │
│  3. Gateway 聚合后存储                                             │
│     ┌─────────────────────────────────────────────────────────┐    │
│     │   Gateway 接收来自多个实例的指标                         │    │
│     │   聚合后转发给 TSDB Server                             │    │
│     └─────────────────────────────────────────────────────────┘    │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### Pushgateway 的使用

```yaml title="prometheus.yml"
scrape_configs:
  - job_name: 'pushgateway'
    static_configs:
      - targets: ['pushgateway:9091']
```

```bash
# 推送指标到 Pushgateway
echo "app_requests_total{status=\"200\"} 12345" | curl --data-binary @- \
  http://pushgateway:9091/metrics/job/myjob

# 带标签推送
curl -X POST http://pushgateway:9091/metrics/job/myjob/instance/myinstance \
  -d 'my_metric{label="value"} 42'
```

## Pull 模式的优势

### 1. 监控即代码

```
┌─────────────────────────────────────────────────────────────────────┐
│                        监控即代码的优势                               │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  1. Prometheus 配置 = 监控目标清单                                   │
│                                                                     │
│     prometheus.yml:                                                 │
│     scrape_configs:                                                 │
│       - job_name: 'production'                                     │
│         kubernetes_sd_configs:                                      │
│           - role: pod                                             │
│                                                                     │
│     └── 这份配置定义了「生产环境监控什么」                          │
│                                                                     │
│  2. GitOps 管理                                                    │
│                                                                     │
│     prometheus.yml (版本控制) ──► Git ──► 自动化部署               │
│                                                                     │
│     └── 监控配置变更有完整的审计日志                               │
│                                                                     │
│  3. 基础设施代码化                                                  │
│                                                                     │
│     Terraform/Pulumi 定义 Kubernetes 集群                          │
│     → Pod 自动被 Prometheus 发现                                    │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 2. 优雅处理实例重启

```promql
# Pull 模式：每次拉取获取最新值
# 重启后新实例立即被采集
# 旧实例数据停止更新（无数据 = 不可见 = 不会误判）

# Push 模式：需要额外处理
# 短生命周期任务结束后指标消失
# 需要 Pushgateway 暂存
```

### 3. 无需修改应用代码

```go
// Pull 模式：只需要暴露 /metrics 端点
// 应用本身不需要知道监控系统的存在

http.Handle("/metrics", promhttp.Handler())
// Prometheus 自动来拉取，应用无感知

// Push 模式：需要在代码中集成 SDK
// 应用需要知道往哪里推送
client, _ := influxdb.NewClient(...)
client.Write(...)
```

## Push 模式的场景

### 1. 短生命周期任务

```
┌─────────────────────────────────────────────────────────────────────┐
│                    短生命周期任务监控                                │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  Pull 模式问题:                                                     │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │                                                             │   │
│  │   Prometheus ── pull ──► [任务已结束] ──► 任务不存在        │   │
│  │                                                             │   │
│  │   结果: 无法采集到任何数据                                  │   │
│  │                                                             │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  Push 模式优势:                                                     │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │                                                             │   │
│  │   [Job 执行] ── push ──► Pushgateway ── pull ──► Prometheus  │   │
│  │                                                             │   │
│  │   任务结束后指标仍然存在于 Pushgateway                      │   │
│  │                                                             │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  适用场景:                                                          │
│  ├── CronJob / 定时任务                                            │
│  ├── Serverless 函数                                              │
│  └── Batch 处理任务                                               │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 2. 防火墙内的服务

```

│                    防火墙内服务监控                                 │
├─────────────────────────────────────────────────────────────────────┤

│                                                                     │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │                      Internet                                │   │
│  │                                                             │   │
│  │   ┌─────────────┐                    ┌─────────────┐        │   │
│  │   │ Prometheus  │ ✗ ── pull ────    │ 内网服务    │        │   │
│  │   │  (外网)     │    防火墙阻断       │  (10.x.x.x) │        │   │
```

### 3. 采集频率极高

```go
// 百万级 QPS，Pull 模式会给监控系统巨大压力
// Push 模式可以在客户端批量聚合后推送

// 示例: 每秒 100 万次请求
// 原始埋点: 100万指标/秒
// 聚合后推送: 1万指标/秒 (每100ms聚合一次)
```

## 混合方案

Prometheus 支持 Pushgateway 兼容 Push 模式：

```yaml title="prometheus.yml"
scrape_configs:
  # Pull 模式：主要采集方式
  - job_name: 'kubernetes-pods'
    kubernetes_sd_configs:
      - role: pod

  # Pushgateway：采集短生命周期任务
  - job_name: 'batch-jobs'
    static_configs:
      - targets: ['pushgateway:9091']
    relabel_configs:
      - source_labels: [__address__]
        regex: 'pushgateway:9091'
        action: keep
```

## 总结

两种采集模式各有适用场景：

| 场景 | 推荐模式 | 原因 |
|---|---|---|
| Kubernetes 中的常驻服务 | Pull | 原生支持，服务发现 |
| 微服务架构 | Pull | 无状态，监控即代码 |
| CronJob/定时任务 | Push (Pushgateway) | 任务结束后数据仍存在 |
| Serverless | Push | 函数生命周期短 |
| 防火墙内服务 | Push | 绕过防火墙限制 |
| 高频采集 | Push (批量聚合) | 减少采集压力 |

Prometheus 推崇 Pull 模式，但这不是银弹。在短生命周期任务等特殊场景下，Pushgateway 是必要的补充。
