只要你用 Kubernetes，几乎不可避免地要和 Prometheus 打交道。它的 pull 模式、强大的 PromQL、云原生的设计，让它成为了监控领域的绝对王者。

## 四种 Metric 类型：什么时候用哪个？

| 类型 | 特点 | 使用场景 | 示例 |
|------|------|----------|------|
| **Counter** | 只增不减 | 累计值：请求数、错误数 | `http_requests_total` |
| **Gauge** | 可增可减 | 瞬时值：CPU、内存、在线人数 | `cpu_usage_percent` |
| **Histogram** | 桶统计 | 分布统计：延迟、响应时间 | `http_request_duration_seconds` |
| **Summary** | 客户端计算百分位 | 延迟百分位（不推荐服务端聚合） | `request_latency` |

```java
// Counter：请求总数
private final Counter requestsTotal = Counter.builder()
    .name("http_requests_total")
    .help("Total HTTP requests")
    .labelNames("method", "status")
    .register();

// Gauge：当前在线人数
private final Gauge onlineUsers = Gauge.builder()
    .name("online_users")
    .help("Current online users")
    .register();

// Histogram：请求延迟（计算百分位）
private final Histogram requestDuration = Histogram.builder()
    .name("http_request_duration_seconds")
    .buckets(0.005, 0.01, 0.025, 0.05, 0.1, 0.25, 0.5, 1, 2.5, 5)
    .register();
```

## Pull vs Push：为什么 Prometheus 选择 Pull？

| 模式 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| **Push** | 客户端主动推送数据到服务器 | 适合短周期任务、不在防火墙内 | 需要在客户端部署 agent |
| **Pull** | 服务端主动拉取数据 | 配置简单、无需 agent、自动负载均衡 | 不适合短周期任务 |

Prometheus 选择 Pull 的原因：
1. **配置简单**：只需在 Prometheus 端配置 target 地址
2. **无侵入**：不需要在业务代码里加推送逻辑
3. **自动负载均衡**：多台 Prometheus 可以拉取同一个 target
4. **Kubernetes 原生**：Pod IP 变化时，Prometheus 可以动态发现

## 核心公式

```
Prometheus 数据模型：
  指标名 + 标签集合 = 时间序列
  http_requests_total{method="GET", status="200"} = [值, 时间戳]

PromQL 核心函数：
  rate(x[5m])    = 5分钟内的平均增长率
  increase(x[5m])= 5分钟内的总增长量
  irate(x[5m])   = 最后两个点的瞬时增长率
  histogram_quantile(0.95, rate(x_bucket[5m])) = P99延迟

四种指标类型：
  Counter = 累计计数器（只增不减）
  Gauge = 瞬时仪表（可增可减）
  Histogram = 延迟分布桶（服务端聚合）
  Summary = 延迟百分位（客户端计算）

Pull 模式工作流程：
  Prometheus → 发现 Target → 定期 Pull /metrics → 存储 TSD → Grafana 展示
```
