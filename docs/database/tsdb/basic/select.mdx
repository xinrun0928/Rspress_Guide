# 时序数据库选型对比

你的团队决定采用时序数据库，但市面上的选择太多了：InfluxDB、Prometheus、TimescaleDB、TDengine、ClickHouse、KairosDB……每个都有自己的特点。到底应该选哪个？

本篇文章将对主流时序数据库进行全面对比，帮助你做出决策。

## 二、详细对比

| 指标 | InfluxDB | Prometheus | TimescaleDB | TDengine | ClickHouse |

| 最大数据量 | 数十 TB | 数 TB | 数 PB | 数 PB | 数 PB |
| 水平扩展 | Cluster | Federation | 自动分片 | 原生 Cluster | 原生 |

### 2.2 功能对比

### 2.3 适用场景

```
场景匹配:

1. 监控系统 / 可观测性
   推荐: Prometheus + Thanos
   原因: 云原生，PromQL 强大，生态完善

2. 通用时序数据
   推荐: InfluxDB / TimescaleDB
   原因: 功能完整，InfluxQL/SQL 易用

3. 物联网 / 传感器
   推荐: TDengine
   原因: 超级表概念，压缩率最高

4. 日志分析
   推荐: ClickHouse / Elasticsearch
   原因: 列式存储，压缩率高，查询快

5. 金融时序
   推荐: TimescaleDB / ClickHouse
   原因: SQL 支持好，分析能力强

6. 混合负载 (时序 + 分析)
   推荐: ClickHouse / TimescaleDB
   原因: 同时支持 OLTP 和 OLAP
```

## 三、Prometheus 生态

### 3.1 Prometheus 优势

```
Prometheus 特点:

优势:
1. 云原生，为监控而生
2. Pull 模式，无需 agent
3. PromQL 强大，专注聚合
4. CNCF 项目，社区活跃
5. 与 Kubernetes 深度集成

劣势:
1. 单机存储有限 (需要 Thanos/Cortex)
2. 数据模型简单 (无字段)
3. 不支持多指标关联查询
```

## 五、TDengine 特点

### 5.1 超级表概念

```
TDengine 的超级表:

超级表 (Super Table): 定义一类设备的模板
┌─────────────────────────────────────────────────────────────┐
│  CREATE STABLE device_metrics (                             │
│      ts TIMESTAMP,                                         │
│      temperature FLOAT,                                    │
│      humidity FLOAT,                                        │
│      pressure FLOAT                                         │
│  ) TAGS (device_id INT, location BINARY(50));              │
└─────────────────────────────────────────────────────────────┘

子表 (Sub Table): 每个设备一张表
┌─────────────────────────────────────────────────────────────┐
│  CREATE TABLE device_001 USING device_metrics               │
│      TAGS (device_id 1, location 'Beijing');               │
│                                                             │
│  CREATE TABLE device_002 USING device_metrics               │
│      TAGS (device_id 2, location 'Shanghai');             │
└─────────────────────────────────────────────────────────────┘

查询优势:
- 按设备查询快 (子表独立存储)
- 按类型聚合也快 (超级表聚合)
```

### 5.2 TDengine vs InfluxDB

```
TDengine vs InfluxDB:

| 方面 | TDengine | InfluxDB |
|------|----------|----------|
| 查询语言 | SQL | InfluxQL/Flux |
| 表概念 | 超级表/子表 | Measurement |
| 压缩率 | 极高 (15-30x) | 高 (5-10x) |
| 写入吞吐 | 最高 | 高 |
| 开源 | 完全开源 | 部分开源 |
| 集群 | 开源免费 | 需要 Enterprise |
| 生态 | Telegraf | Telegraf |
| 学习曲线 | 低 (SQL) | 中等 |
```

## 六、选型建议

### 6.1 按数据量选型

```
数据量 < 100GB:
├── Prometheus (单机版)  ← 监控场景
├── TimescaleDB         ← 需要 SQL + 时序
└── InfluxDB OSS        ← 通用场景

数据量 100GB - 1TB:
├── InfluxDB Enterprise  ← 功能完整
├── Prometheus + Thanos  ← 监控场景
└── TimescaleDB          ← 需要 SQL

数据量 1TB - 10TB:
├── ClickHouse           ← 分析为主
├── Prometheus + Thanos  ← 监控场景
└── TDengine             ← IoT 场景

数据量 > 10TB:
├── ClickHouse           ← 分析 + 时序
├── TDengine Cluster      ← IoT + 成本优先
└── TimescaleDB          ← SQL + 时序
```

### 6.2 按团队技能选型

```
团队技能匹配:

熟悉 SQL:
├── TimescaleDB  ← 最佳选择
├── ClickHouse   ← 适合分析场景
└── TDengine     ← 学习曲线最低

熟悉监控 / DevOps:
├── Prometheus   ← 监控首选
└── InfluxDB    ← 功能丰富

愿意学习新语言:
└── InfluxDB + Flux  ← 灵活性高

需要 Java/Go 集成:
├── TDengine    ← 原生 Java/Go SDK
└── InfluxDB    ← 也有 SDK
```

### 6.3 成本考量

```
成本对比 (同等数据量):

InfluxDB:
├── 开源版: 免费，但不支持集群
└── Enterprise: 按节点收费，$0.005/series

TimescaleDB:
├── 开源版: 免费 (Apache 2.0)
└── Cloud/Enterprise: 按使用量收费

TDengine:
├── 社区版: 完全免费 (AGPL)
└── 企业版: 按节点收费

Prometheus:
├── 开源版: 完全免费
└── Thanos: 开源免费

ClickHouse:
├── 开源版: 完全免费 (Apache 2.0)
└── Altinity Cloud: 按使用量收费
```

## 总结

时序数据库选型没有标准答案，关键是匹配业务场景：

1. **监控告警** → Prometheus + Thanos
2. **通用时序** → InfluxDB / TimescaleDB
3. **IoT 物联网** → TDengine
4. **日志分析** → ClickHouse / Elasticsearch
5. **混合分析** → TimescaleDB / ClickHouse

选型后，建议先用小规模数据验证性能，再决定是否大规模部署。
