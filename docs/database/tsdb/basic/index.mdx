# 时序数据库基础

你的监控系统每天产生 10 亿条监控指标数据，Prometheus 存储了 3 年的历史数据。MySQL 的性能在时序数据面前捉襟见肘——磁盘空间爆炸、查询越来越慢、写入开始排队。

你需要的是**时序数据库（Time Series Database，TSDB）**。

本篇文章将带你理解时序数据库的核心概念，以及如何选择合适的时序数据库。

## 一、什么是时序数据

### 1.1 时序数据的定义

时序数据（Time Series Data）是按时间顺序组织的数据点序列：

```
监控指标示例:

时间戳              主机        CPU使用率  内存使用率  磁盘使用率
2024-01-15 10:00:00  server-1    45%       62%        78%
2024-01-15 10:00:00  server-2    32%       55%        65%
2024-01-15 10:00:01  server-1    46%       62%        78%
2024-01-15 10:00:01  server-2    33%       56%        65%
...
```

### 1.2 时序数据的特征

| 特征 | 说明 |
|---|---|
| **时间有序** | 数据点按时间顺序追加，新数据总是最新的 |
| **写入量大** | 高频采集，每秒可能产生数万甚至数百万条数据 |
| **查询特点** | 范围查询为主，经常查询某个时间段的全部数据 |
| **冷热分层** | 最近的数据访问频繁，历史数据访问较少 |
| **很少更新** | 历史数据基本不会修改 |
| **批量删除** | 通常按时间范围删除旧数据 |

### 1.3 典型应用场景

```
时序数据库应用场景:

1. 监控运维
   - 服务器指标 (CPU、内存、磁盘)
   - 应用性能监控 (APM)
   - 日志监控 (ELK)

2. IoT 物联网
   - 传感器数据 (温度、湿度、压力)
   - 设备状态监控
   - 位置追踪

3. 金融交易
   - 股票价格
   - 交易量统计
   - 风控指标

4. 能源管理
   - 电表数据
   - 燃气表数据
   - 智能家居
```

## 二、时序数据库的核心概念

### 2.1 数据模型

```
时序数据四要素:
┌─────────────────────────────────────────────────────────────┐
│  指标名 (Metric)     │  时间戳 (Timestamp)                │
│  cpu_usage           │  1705305600                        │
├──────────────────────┼─────────────────────────────────────┤
│  标签 (Tags/Labels)  │  数值 (Value)                      │
│  host=server-1      │  45.5                              │
│  region=us-west      │                                    │
│  service=web         │                                    │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 关键概念

| 概念 | 说明 | 示例 |
|---|---|---|
| **Metric** | 指标名称 | `cpu_usage`、`memory_free`、`request_count` |
| **Timestamp** | 时间戳 | `1705305600` (Unix 秒) |
| **Tags/Labels** | 维度标签 | `host=server-1`、`region=us-west` |
| **Value** | 指标值 | `45.5`、`1024`、`0.95` |
| **Field** | 指标字段 | 在 InfluxDB 中与 Tags 对应 |
| **Series** | 时间序列 | Metric + Tags 的唯一组合 |

### 2.3 标签 vs 指标值

```
区分 Tags 和 Values:

Tags (维度，索引):
- 用于过滤和分组
- 通常是字符串
- 会建立索引
- 示例: host, region, service, status

Values (指标值):
- 具体的数值
- 可以是整数或浮点数
- 不建立索引
- 示例: cpu_usage, response_time
```

## 三、时序数据库 vs 关系数据库

### 3.1 为什么 MySQL 不适合时序数据

```
MySQL 存储时序数据的问题:

1. 写入性能差
   - B+Tree 索引维护开销大
   - 随机写入导致磁盘 IO 高

2. 存储成本高
   - 索引占用大量空间
   - 难以压缩

3. 查询效率低
   - 聚合查询需要全表扫描
   - 时间范围查询没有优化

4. 数据生命周期管理困难
   - 没有内置的 TTL 支持
   - 删除旧数据开销大
```

### 3.2 时序数据库的优化

```
时序数据库的优化方向:

1. 存储结构优化
   - LSM-Tree (WAL + 内存 + 磁盘)
   - 列式存储 (同类型数据压缩率高)
   - 数据分片 (按时间/标签分区)

2. 写入优化
   - 批量写入
   - 异步写入
   - 数据先入内存再刷盘

3. 查询优化
   - 时间范围索引
   - 标签索引
   - 预聚合 / 降采样

4. 压缩优化
   - 时间戳压缩 (delta encoding)
   - 数值压缩 (gorilla 算法等)
   - 字典压缩
```

### 3.3 对比表

| 特性 | MySQL | InfluxDB | Prometheus | TDengine |
|---|---|---|---|---|
| 数据模型 | 关系型 | Tag/Field | Label/Prometheus | 超级表/子表 |
| 写入吞吐 | 低 | 高 | 高 | 极高 |
| 查询语言 | SQL | InfluxQL/Flux | PromQL | SQL |
| 压缩率 | 低 | 高 | 高 | 极高 |
| 水平扩展 | 需分库分表 | Cluster | Federation | 原生 Cluster |
| 聚合函数 | 支持 | 支持 | 内置丰富 | 支持 |
| 适用场景 | 通用 | 通用监控 | 监控 | 物联网 |

## 四、时序数据库的选购标准

### 4.1 核心评估维度

```
时序数据库选购检查清单:

1. 性能指标
   □ 写入吞吐量 (每秒能写入多少条)
   □ 查询延迟 (范围查询、聚合查询的 P99)
   □ 数据压缩率 (存储空间节省)

2. 功能完整性
   □ 时间范围查询
   □ 标签过滤查询
   □ 聚合函数 (COUNT, AVG, SUM, MAX, MIN, PERCENTILE)
   □ 降采样 / 预聚合
   □ 数据保留策略 (TTL)
   □ 连续查询

3. 运维友好度
   □ 部署复杂度
   □ 监控和告警
   □ 数据备份恢复
   □ 集群管理

4. 生态集成
   □ 数据采集 Agent
   □ 可视化工具 (Grafana)
   □ 数据导入导出

5. 成本考量
   □ 商业版 vs 开源版
   □ 资源消耗
   □ 社区活跃度
```

### 4.2 按场景选型

```
选型建议:

场景 1: 监控告警 (Prometheus 生态)
选择: Prometheus + Thanos/Cortex
原因: 专为监控设计，PromQL 强大，生态完善

场景 2: 通用时序数据 (Grafana 生态)
选择: InfluxDB / TimescaleDB
原因: 功能完整，InfluxQL/Flux 灵活

场景 3: 物联网海量数据
选择: TDengine / TimescaleDB
原因: 专为 IoT 设计，超级表概念，压缩率极高

场景 4: 金融时序数据
选择: TimescaleDB / ClickHouse
原因: SQL 支持好，TimescaleDB 基于 PostgreSQL

场景 5: 日志分析
选择: Elasticsearch
原因: 全文搜索 + 时序，ELK 生态完善
```

## 五、时序数据库发展趋势

### 5.1 云原生时代

```
云原生时序数据库:

1. 全托管服务
   - Amazon Timestream
   - Google Cloud Monitoring
   - Azure Monitor

2. Kubernetes 原生
   - Thanos (Prometheus 长存储)
   - M3DB (Uber 开源)
   - VictoriaMetrics

3. Serverless
   - 按实际使用量计费
   - 自动扩缩容
```

### 5.2 多模态融合

```
现代时序数据库的趋势:

1. 时序 + 分析
   - ClickHouse 的时序支持
   - TimescaleDB 的持续聚合
   - InfluxDB 的 IOx (基于 Apache Arrow)

2. 时序 + 图
   - 用于异常检测
   - 依赖关系分析

3. 时序 + AI
   - 内置异常检测
   - 预测分析
   - 根因分析
```

## 总结

时序数据库是处理时序数据的专业工具：

1. **时序数据特征**：写入量大、查询范围大、冷热分层
2. **核心概念**：Metric、Timestamp、Tags、Value
3. **与传统数据库的区别**：LSM-Tree、列式存储、专为时序优化
4. **选型建议**：根据场景选择 Prometheus/InfluxDB/TDengine/TimescaleDB
5. **发展趋势**：云原生、多模态、AI 融合

理解时序数据库的原理和适用场景，是构建现代可观测性平台的基础。
