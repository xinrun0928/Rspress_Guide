# 时序数据库核心概念

你决定使用时序数据库来存储监控数据。在开始之前，你需要理解几个核心概念：时间线（Series）、数据点（Point）、时间范围查询、降采样、保留策略。这些概念决定了如何设计你的数据模型和查询语句。

## 一、数据模型

### 1.1 核心要素

```
时序数据的基本组成:

┌─────────────────────────────────────────────────────────────┐
│                       数据点 (Point)                         │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  ┌─────────────────┐    ┌─────────────────┐                │
│  │    Timestamp     │    │     Value       │                │
│  │  1705305600000   │    │      45.5       │                │
│  └─────────────────┘    └─────────────────┘                │
│                                                              │
│  ┌─────────────────────────────────────────────────────┐    │
│  │                     Tags / Labels                     │    │
│  │                                                      │    │
│  │   host = server-1                                     │    │
│  │   region = us-west                                   │    │
│  │   service = api-gateway                               │    │
│  │   status = healthy                                    │    │
│  │                                                      │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                              │
│  ┌─────────────────┐                                        │
│  │     Metric       │  (指标名 / Measurement)               │
│  │   cpu_usage     │                                        │
│  └─────────────────┘                                        │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 Metric / Measurement

```
Metric 是指标名，表示被测量的属性:

命名规范 (不同数据库略有差异):

InfluxDB 风格:
  cpu_usage              ← 小写下划线分隔
  memory_used_bytes      ← 包含单位
  http_request_duration_seconds  ← 包含单位

Prometheus 风格:
  cpu_usage              ← 小写下划线
  http_request_duration  ← 小写下划线
  process_cpu_seconds_total  ← _total 后缀表示计数器

通用建议:
1. 使用有意义的名称
2. 包含单位 (bytes, seconds, milliseconds)
3. 保持一致的命名风格
4. 不要使用保留关键字
```

### 1.3 Tags / Labels

```
Tags 是维度信息，用于过滤和分组:

Tag 设计原则:

1. 选择性高
   Good: user_id (高基数，几百万用户)
   Bad: is_active (低基数，只有 true/false)

2. 查询时经常过滤
   Good: service, region, env
   Bad: temperature_unit (永远是 celsius)

3. 值是字符串
   - 不适合做数值的 Tags 可以是数字
   - 但通常字符串更灵活

4. 基数控制
   - 基数太高会影响查询性能
   - 基数太低没有区分度

示例:
# Good Tags
- hostname
- application
- environment (prod/staging/dev)
- region (us-east, us-west, eu)

# Bad Tags (不建议)
- timestamp
- value
- unit
```

## 二、时间线 / Series

### 2.1 什么是时间线

```
时间线 = Metric + Tags 的唯一组合

示例:

Metric: cpu_usage

时间线 1: cpu_usage{host="server-1", region="us-west"}
时间线 2: cpu_usage{host="server-2", region="us-west"}
时间线 3: cpu_usage{host="server-1", region="us-east"}
时间线 4: cpu_usage{host="server-2", region="us-east"}

每条时间线包含连续的数据点:

cpu_usage{host="server-1", region="us-west"}
  │
  │     ●
  │   ●   ●
  │ ●       ●
  │────────────────────→ 时间
```

### 2.2 Series Cardinality

```
Series Cardinality = 时间线数量

高基数问题:

场景: 10万台设备，每台有5个指标
  → 50万条时间线

场景: 用户行为追踪，每个用户是独立的 Tag
  → 如果有 1000万用户 = 1000万条时间线

高基数的影响:
- 索引体积大
- 查询变慢
- 内存消耗增加

解决方案:
1. 减少 Tag 数量
2. 使用 Hash 降低 Tag 基数
3. 选择合适的数据库 (TDengine 用超级表解决)
```

### 2.3 标签 vs 字段

```
InfluxDB 中的 Tags vs Fields:

Tags (有索引，查询过滤用):
- hostname
- region
- service

Fields (无索引，存储具体值):
- cpu_usage (45.5)
- memory_used (1024000000)

写入示例:
cpu_usage,hostname=server-1,region=us-west cpu_usage=45.5,memory_used=1024000000

查询区别:
- 按 Tag 过滤: WHERE hostname = 'server-1' (快，有索引)
- 按 Field 过滤: WHERE cpu_usage > 80 (慢，扫描全表)
```

## 三、数据点 / Point

### 3.1 Point 的组成

```
单个数据点包含:

1. Timestamp (时间戳)
   - 精度: 纳秒 / 微秒 / 毫秒 / 秒
   - 示例: 1705305600000 (毫秒)

2. Measurement (指标名)
   - 示例: cpu_usage

3. Tags (维度标签)
   - Key-Value 对
   - 可选，有索引
   - 示例: host=server-1, region=us-west

4. Fields (指标值)
   - Key-Value 对
   必填，无索引
   - 示例: usage=45.5, user=20

5. Timestamp (可选)
   - 数据库特定元数据
```

### 3.2 Point 的唯一性

```
Point 的唯一性由 (Measurement + Tags + Timestamp) 决定:

场景: 同一时间戳写入两次数据

InfluxDB 行为:
- 相同 (measurement, tags, timestamp) 会覆盖
- 这是 Upsert (更新插入) 语义

示例:

写入 1:
cpu_usage,host=server-1 usage=45.5 1705305600000

写入 2 (同一时间戳):
cpu_usage,host=server-1 usage=50.0 1705305600000

结果: usage = 50.0 (覆盖)
```

## 四、保留策略 / Retention Policy

### 4.1 什么是保留策略

```
保留策略 (Retention Policy, RP) 定义数据保留时间:

示例:

InfluxDB:
  CREATE RETENTION POLICY "one_day"
    ON "telegraf"
    DURATION 1d
    REPLICATION 1
    SHARD DURATION 1h

含义:
- 数据保留 1 天
- 超过 1 天的数据自动删除
```

### 4.2 多级保留策略

```
典型配置: 高频短期 + 低频长期

RP 配置:

┌─────────────────────────────────────────────────────────────┐
│  hot_1m     │  DURATION: 7d   │  每分钟一条数据          │
│  warm_1h     │  DURATION: 30d  │  每小时一条数据 (降采样) │
│  cold_1d     │  DURATION: 365d │  每天一条数据 (降采样)  │
└─────────────────────────────────────────────────────────────┘

查询时自动关联:
- 最近 7 天 → hot_1m (高分辨率)
- 7-30 天 → warm_1h (中等分辨率)
- 30-365 天 → cold_1d (低分辨率)
```

## 五、连续查询 / Continuous Query

### 5.1 什么是连续查询

```
连续查询 (Continuous Query, CQ) 是定时执行的查询，
用于预计算和降采样:

工作原理:
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  原始数据 (每分钟)         连续查询          降采样数据       │
│                                                             │
│  10:01:00 ──────────┐                                      │
│  10:02:00 ──────────┼──→  SELECT MEAN(cpu_usage)           │
│  10:03:00 ──────────┤      GROUP BY time(1h)      ──────→  │
│  10:04:00 ──────────┤                           10:00:00     │
│  ...                │      INTO "cpu_1h"                   │
│  10:59:00 ──────────┘                           cpu_usage=40 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 使用场景

```sql
-- InfluxDB 连续查询示例

-- 每小时降采样
CREATE CONTINUOUS QUERY "cpu_hourly"
ON "telegraf"
BEGIN
  SELECT MEAN(cpu_usage) AS cpu_usage_avg,
         MAX(cpu_usage) AS cpu_usage_max,
         MIN(cpu_usage) AS cpu_usage_min
  INTO "cpu_usage_1h"
  FROM "cpu_usage"
  GROUP BY time(1h), *
END;

-- 每天统计
CREATE CONTINUOUS QUERY "daily_stats"
ON "telegraf"
BEGIN
  SELECT COUNT(*) AS request_count,
         MEAN(response_time) AS avg_response_time
  INTO "request_stats_1d"
  FROM "http_requests"
  GROUP BY time(1d), service, status_code
END;
```

## 六、降采样 / Downsampling

### 6.1 为什么要降采样

```
降采样的原因:

1. 减少数据量
   - 原始: 1分钟1条 × 1440分钟 = 1440条/天
   - 降采样: 1小时1条 × 24小时 = 24条/天
   - 节省: 98%

2. 加速历史查询
   - 查询一年数据时，只需扫描 365 × 24 = 8760 条
   - 而不是 525600 条

3. 降低成本
   - 存储空间减少
   - 查询成本降低
```

### 6.2 降采样策略

```
常见降采样策略:

保留策略级别:
┌─────────────────────────────────────────────────────────────┐
│  保留时间  │  分辨率    │  数据量/天    │  适用场景        │
├────────────┼────────────┼───────────────┼──────────────────┤
│  1-7 天    │  1分钟     │  1440 条      │  实时监控        │
│  7-30 天   │  5分钟     │  288 条       │  趋势分析        │
│  30-365 天 │  1小时      │  24 条        │  长期趋势        │
│  > 365 天  │  1天        │  1 条         │  历史归档        │
└─────────────────────────────────────────────────────────────┘
```

## 总结

时序数据库的核心概念：

1. **数据模型**：Metric + Tags + Timestamp + Value
2. **时间线**：Metric + Tags 的唯一组合，Series Cardinality 影响性能
3. **保留策略**：定义数据生命周期，自动清理过期数据
4. **连续查询**：定时执行预聚合，减少查询压力
5. **降采样**：多级分辨率，平衡精度和存储

理解这些概念，才能设计出高效的时序数据模型。
