# 时序数据库 vs 关系数据库

你的团队正在评估是否需要使用时序数据库。MySQL 已经能存储监控数据，为什么要切换？本篇文章将深入对比时序数据库和关系数据库的差异，帮助你做出正确的技术决策。

## 一、数据模型对比

### 1.1 关系数据库的数据模型

```text
MySQL 表结构:

CREATE TABLE metrics (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    metric_name VARCHAR(100),
    host VARCHAR(100),
    region VARCHAR(50),
    value DOUBLE,
    timestamp DATETIME,
    INDEX idx_timestamp (timestamp),
    INDEX idx_metric_host (metric_name, host)
);

数据示例:
┌─────┬────────────┬──────────┬──────────┬───────┬─────────────────┐
│ id  │ metric_name │ host     │ region   │ value │ timestamp       │
├─────┼────────────┼──────────┼──────────┼───────┼─────────────────┤
│ 1   │ cpu_usage   │ server-1 │ us-west  │ 45.5  │ 2024-01-15 10:00│
│ 2   │ cpu_usage   │ server-2 │ us-west  │ 32.1  │ 2024-01-15 10:00│
│ 3   │ memory_used │ server-1 │ us-west  │ 1024  │ 2024-01-15 10:00│
└─────┴────────────┴──────────┴──────────┴───────┴─────────────────┘
```

### 1.2 时序数据库的数据模型

```text
InfluxDB Line Protocol:

cpu_usage,host=server-1,region=us-west value=45.5 1705305600000000000
memory_used,host=server-1,region=us-west value=1024 1705305600000000000

对比:
- metric_name -> cpu_usage (Measurement)
- host, region -> Tags (索引)
- value -> Field (无索引)
- timestamp -> Timestamp
```

## 二、写入性能对比

### 2.1 性能测试

```text
测试环境: 4 核 CPU / 16GB 内存 / SSD

写入性能对比 (每秒写入条数):

MySQL (InnoDB):
  单条 INSERT: 2,000 - 5,000 条/秒
  批量 INSERT (1000条/批): 20,000 - 50,000 条/秒
  原因: B+Tree 索引维护，随机写入

InfluxDB:
  单次写入 (批量): 100,000 - 500,000 条/秒
  原因: LSM-Tree，先写内存再刷盘，批量优化

Prometheus:
  Remote Write: 200,000 - 1,000,000 条/秒
  原因: 专门为监控场景优化

TDengine:
  写入: 1,000,000+ 条/秒
  原因: 超级表分区，高效压缩
```

### 2.2 性能差异原因

```text
性能差异的技术原因:

MySQL (B+Tree):
┌─────────────────────────────────────────────────────────────┐
│ B+Tree 索引结构:                                            │
│                                                              │
│          [Root]                                              │
│         /    \                                               │
│      [Node]  [Node]                                         │
│     /    \   /    \                                         │
│   [...] [Leaf] [Leaf] [...]                                 │
│              ↑                                               │
│         按索引排序                                           │
│         插入时需要找到正确位置                                │
│         可能触发页面分裂                                      │
└─────────────────────────────────────────────────────────────┘

时序数据库 (LSM-Tree / 专有结构):
┌─────────────────────────────────────────────────────────────┐
│ 写入流程:                                                    │
│                                                              │
│  1. 数据先写入内存 (有序)                                    │
│  2. 批量刷盘 (顺序写入)                                      │
│  3. 后台 Compaction 合并                                    │
│                                                              │
│ 优势:                                                        │
│ - 写入全部是顺序操作                                         │
│ - 支持极高吞吐                                               │
│ - 天然支持批量                                               │
└─────────────────────────────────────────────────────────────┘
```

## 三、存储空间对比

### 3.1 压缩效率

```text
存储空间对比 (1亿条数据):

MySQL:
  每行约 100 字节 (包含索引)
  总空间: 约 15 GB
  索引空间: 约 8 GB
  数据空间: 约 7 GB

InfluxDB:
  压缩后每点约 20-30 字节
  总空间: 约 2-3 GB
  压缩率: 约 5-7x

TDengine:
  压缩后每点约 5-10 字节
  总空间: 约 0.5-1 GB
  压缩率: 约 15-30x

原因:
1. 时间戳压缩 (delta encoding)
2. 数值压缩 (gorilla, delta-of-delta)
3. 标签字典化 (重复的标签值只存一次)
```

### 3.2 压缩算法

```text
时序数据压缩技术:

1. 时间戳压缩
   - Delta Encoding: 存储与前一点的差值
   - Delta of Delta: 存储差值的变化

   示例:
   时间戳: 1705305600000, 1705305600001, 1705305600003
   存储: 1705305600000, 1, 2

2. 数值压缩 (Gorilla)
   - XOR encoding for floating point
   - 无需存储完整 64 位

   示例:
   值: 1705305600000.5, 1705305600000.6
   存储: 1705305600000.5, 0.1

3. 标签压缩
   - 字典编码
   - 相同标签值只存储一次
```

## 四、查询性能对比

### 4.1 范围查询

```sql
-- 查询最近一天的数据
-- MySQL:
SELECT * FROM metrics
WHERE timestamp >= NOW() - INTERVAL 1 DAY
AND metric_name = 'cpu_usage'
ORDER BY timestamp;

-- InfluxDB:
SELECT * FROM cpu_usage
WHERE time >= now() - 1d;

-- 性能对比:
-- MySQL: 2-5 秒 (取决于数据量)
-- InfluxDB: 0.1-0.5 秒 (时间索引优化)
```

### 4.2 聚合查询

```sql
-- 按小时聚合
-- MySQL:
SELECT DATE_FORMAT(timestamp, '%Y-%m-%d %H:00:00') AS hour,
       AVG(value) AS avg_value
FROM metrics
WHERE metric_name = 'cpu_usage'
AND timestamp >= NOW() - INTERVAL 7 DAY
GROUP BY hour
ORDER BY hour;

-- InfluxDB:
SELECT MEAN(value) AS avg_value
FROM cpu_usage
WHERE time >= now() - 7d
GROUP BY time(1h);

-- 性能对比:
-- MySQL: 10-30 秒 (全表扫描 + 聚合)
-- InfluxDB: 0.5-2 秒 (有聚合优化)
```

### 4.3 多指标查询

```sql
-- 查询多个指标
-- MySQL:
SELECT * FROM metrics
WHERE metric_name IN ('cpu_usage', 'memory_used')
AND timestamp >= NOW() - INTERVAL 1 DAY;

-- InfluxDB:
SELECT * FROM /cpu_usage|memory_used/
WHERE time >= now() - 1d;

-- 性能对比:
-- MySQL: 慢，需要多次索引查询 + UNION
-- InfluxDB: 快，基于 Measurement 的查询
```

## 七、选型决策

### 7.1 什么时候选择时序数据库

```text
应该使用时序数据库的场景:

1. 数据量大
   - 写入量 > 10万条/秒
   - 需要存储 > 1亿条数据

2. 查询特点
   - 主要是时间范围查询
   - 需要聚合分析
   - 定期删除旧数据

3. 数据生命周期
   - 需要自动 TTL
   - 需要多级保留策略
   - 需要降采样

4. 成本敏感
   - 需要高压缩率
   - 存储成本重要
```

### 7.2 什么时候继续用关系数据库

```text
应该继续使用关系数据库的场景:

1. 数据量小
   - 每秒 < 1万条
   - 总数据量 < 1000万条

2. 查询复杂
   - 需要多表 JOIN
   - 随机字段查询
   - 全文搜索

3. 事务要求
   - 需要强事务
   - 需要 ACID
   - 跨表一致性

4. 团队技能
   - 团队熟悉 SQL
   - 运维 MySQL 经验丰富
   - 不希望引入新技术
```

### 7.3 混合架构

```text
既有时序数据又有关系数据的场景:

方案: MySQL + InfluxDB/TimescaleDB

│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 总结

时序数据库 vs 关系数据库的选择：

1. **写入性能**：时序数据库远超关系数据库 (LSM-Tree vs B+Tree)
2. **存储空间**：时序数据库远小于关系数据库 (专用压缩算法)
3. **查询性能**：时序数据库在时间范围查询和聚合上更快
4. **功能特性**：时序数据库有 TTL、降采样等原生支持
5. **运维复杂度**：时序数据库通常更简单

**选择原则**：
- 数据量大 + 查询简单 -> 时序数据库
- 数据量小 + 查询复杂 -> 关系数据库
- 两者兼顾 -> 混合架构

没有最好的数据库，只有最适合业务场景的选择。
