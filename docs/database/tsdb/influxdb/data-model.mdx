# InfluxDB 数据模型

想象一下，你需要监控 10000 台服务器的 CPU 使用率。每台服务器每秒采集一次数据，一天就是 8.64 亿条记录。一个月就是 250 亿条。

如果用 MySQL 存储，你会创建一张巨大的表：

```sql
CREATE TABLE cpu_metrics (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    server_id VARCHAR(50),
    region VARCHAR(50),
    dc VARCHAR(50),
    timestamp DATETIME,
    usage DOUBLE
);
```

然后开始面临噩梦：索引膨胀、查询缓慢、磁盘告急。

InfluxDB 的数据模型，正是为解决这种场景而生。

## 核心组件：Measurement、Tag、Field、Timestamp

InfluxDB 的数据模型由四个核心组件构成：

```
┌─────────────────────────────────────────────────────────────────────┐
│                        InfluxDB 数据点结构                           │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  cpu,host=server01,region=us-west,dc=dc1 usage=45.5,user=30.2 │
│  │  │  │             │          │        │    │                │ │
│  │  │  │             │          │        │    └──── Field Set  │ │
│  │  │  │             │          │        │                     │ │
│  │  │  │             │          │        └─────── Field Keys   │ │
│  │  │  │             │          │                            │ │
│  │  │  │             │          └───── Tag Set                │ │
│  │  │  │             │                                      │ │
│  │  │  │             └───────────── Tag Keys                 │ │
│  │  │  │                                                     │ │
│  │  │  └──────────────── Measurement                        │ │
│  │  │                                                         │ │
│  │  └─────────────────── Timestamp (纳秒级)                   │ │
│  │                                                                   │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### Measurement（测量）

Measurement 是数据点的容器，类似于关系型数据库中的表名。

```
cpu          ← measurement 名
temperature  ← measurement 名
stock_price  ← measurement 名
http_request ← measurement 名
```

命名规范：
- 长度不受限制，但建议简洁
- 大小写敏感
- 不能包含 `_` 开头（保留给系统内部使用）
- 可以包含字母、数字

### Tag（标签）

Tag 是**索引化的维度信息**，用于过滤和分组查询。

```txt
cpu,host=server01,region=us-west
           │           │
           └─ Tag Key ─┘
                      └─ Tag Value
```

| 特性 | 说明 |
|---|---|
| 类型 | 必须是字符串 |
| 索引 | 自动建立索引，查询效率高 |
| 用途 | 描述数据的维度属性 |
| 基数 | 基数不宜过高（高基数会导致索引膨胀） |

常见 Tag 示例：

| Tag Key | 说明 | 示例值 |
|---|---|---|
| `host` | 主机名 | `server01`、`192.168.1.100` |
| `region` | 区域 | `us-west`、`cn-east` |
| `dc` | 数据中心 | `dc1`、`dc2` |
| `service` | 服务名 | `user-service`、`order-service` |
| `status` | 状态 | `200`、`404`、`500` |

:::tip Tag vs Field 的选择原则
**需要过滤查询吗？** 是的话用 Tag，否的话用 Field。
**基数很高吗？** 超过百万级别的唯一值，用 Field 而非 Tag。
**经常做 GROUP BY 吗？** 参与分组的维度用 Tag。
:::

### Field（字段）

Field 存储**实际的指标值**，不是索引化的。

```txt
cpu usage=45.5,user=30.2
          │       │
          │       └─ Field Value
          │
          └─ Field Key
```

| 特性 | 说明 |
|---|---|
| 类型 | Float、Integer、String、Boolean |
| 索引 | 不建立索引 |
| 用途 | 存储时序数据值 |
| 查询 | 直接读写，不经过索引 |

常见 Field 示例：

| Field Key | 说明 | 类型 |
|---|---|---|
| `usage` | CPU 使用率 | Float |
| `count` | 请求计数 | Integer |
| `temperature` | 温度 | Float |
| `status` | 状态码 | String |

### Timestamp（时间戳）

Timestamp 是数据点的核心，标识数据点的时间。

| 精度 | 范围 | 示例 |
|---|---|---|
| 纳秒 | 1677-09-21 ~ 2262-04-11 | `1709308800000000000` |
| 微秒 | 1677-09-21 ~ 2262-04-11 | `1709308800000000` |
| 毫秒 | 1677-09-21 ~ 2262-04-11 | `1709308800000` |
| 秒 | 1970 ~ | `1709308800` |
| 分钟 | 1970 ~ | `28643808` |
| 小时 | 1970 ~ | `477396` |

## 时间线（Series）

时间线是 InfluxDB 查询的核心概念，由 Measurement + Tag Set 唯一确定。

```
Series = Measurement + Tag Set

┌─────────────────────────────────────────────────────────────────────┐
│                        Series 示意图                                │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  Measurement: cpu                                                  │
│                                                                     │
│  ┌─────────────────────────────────────────────────────────────┐  │
│  │  Series 1: cpu,host=server01,region=us-west               │  │
│  │  ┌─────────────────────────────────────────────────────┐  │  │
│  │  │  ts=1709308800  usage=45.5                           │  │  │
│  │  │  ts=1709308810  usage=46.2                           │  │  │
│  │  │  ts=1709308820  usage=44.8                           │  │  │
│  │  └─────────────────────────────────────────────────────┘  │  │
│  │                                                             │  │
│  │  Series 2: cpu,host=server02,region=us-west               │  │
│  │  ┌─────────────────────────────────────────────────────┐  │  │
│  │  │  ts=1709308800  usage=30.1                           │  │  │
│  │  │  ts=1709308810  usage=31.5                           │  │  │
│  │  │  ts=1709308820  usage=29.8                           │  │  │
│  │  └─────────────────────────────────────────────────────┘  │  │
│  │                                                             │  │
│  │  Series 3: cpu,host=server01,region=us-east               │  │
│  │  ┌─────────────────────────────────────────────────────┐  │  │
│  │  │  ts=1709308800  usage=50.0                           │  │  │
│  │  └─────────────────────────────────────────────────────┘  │  │
│  │                                                             │  │
│  └─────────────────────────────────────────────────────────────┘  │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

:::warning Series 基数爆炸
Series 数量 `=` Measurement 数量 `×` Tag 组合数量。

如果有 100 个 measurement，每个有 10 个 tag key，每个 tag 有 100 个可能的值，理论上就有 100 `×` 100^10 个 Series。

InfluxDB 能处理大量数据点，但如果 Series 基数过高，会导致：
- 内存压力增大
- 写入性能下降
- 查询计划变慢

**设计 Tag 时，控制基数是关键。**
:::

## Database、Retention Policy、Measurement

InfluxDB 的数据组织层次：

```
Database
    │
    ├── Retention Policy (RP)
    │       │
    │       ├── Shard Group
    │       │       │
    │       │       ├── Shard 1
    │       │       └── Shard 2
    │       │
    │       └── Measurement
    │               │
    │               └── Series
    │                       │
    │                       └── Data Points
    │
    └── Continuous Query
    │
    └── User
```

### Database

Database 是最顶层的命名空间：

```txt
-- 创建数据库
CREATE DATABASE "monitoring"

-- 使用数据库
USE monitoring

-- 删除数据库
DROP DATABASE "monitoring"
```

### Retention Policy（保留策略）

RP 定义数据的保留时长和分片策略：

```txt
-- 默认 RP: autogen (永久保留)
-- 创建自定义 RP
CREATE RETENTION POLICY "one_week"
ON "monitoring"
DURATION 7d
REPLICATION 1
SHARD DURATION 1h
```

| 参数 | 说明 | 建议 |
|---|---|---|
| `DURATION` | 数据保留时长 | 监控数据 7-30 天 |
| `REPLICATION` | 副本数 | 单机设为 1 |
| `SHARD DURATION` | 分片时长 | 默认 1 小时 |

### Shard Group

Shard Group 按时间范围划分数据，每个 Shard Group 包含若干 Shard：

```
┌─────────────────────────────────────────────────────────────────────┐
│                        Shard Group 结构                             │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  Shard Group 1 (2024-01-01 ~ 2024-01-02)                          │
│  ┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐    │
│  │   Shard 1       │  │   Shard 2       │  │   Shard 3       │    │
│  │   host=server01 │  │   host=server02 │  │   host=server03 │    │
│  └─────────────────┘  └─────────────────┘  └─────────────────┘    │
│                                                                     │
│  Shard Group 2 (2024-01-02 ~ 2024-01-03)                          │
│  ┌─────────────────┐  ┌─────────────────┐                         │
│  │   Shard 4       │  │   Shard 5       │                         │
│  │   host=server01 │  │   host=server02 │                         │
│  └─────────────────┘  └─────────────────┘                         │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

Shard Group 的设计使得按时间范围删除数据非常高效——只需删除整个 Shard Group。

## 设计实践：好的 Tag vs 不好的 Tag

### 好的 Tag 设计

```txt
-- ✅ 低基数，适合作为 Tag
cpu,host=server01,region=us-west,dc=dc1
cpu,host=server02,region=us-west,dc=dc1
```

| Tag | 基数 | 原因 |
|---|---|---|
| `region` | 低（几个~几十个） | 合理维度 |
| `dc` | 低（几个） | 数据中心数量有限 |
| `host` | 中（数百~数千） | 服务器数量可控 |

### 不好的 Tag 设计

```txt
-- ❌ 高基数，不适合作为 Tag
events,user_id=12345,request_id=abc123
```

| Tag | 问题 | 原因 |
|---|---|---|
| `user_id` | 高基数 | 百万级用户 |
| `request_id` | 唯一值 | 每条请求都不同 |

高基数字段应该作为 Field：

```txt
-- ✅ 正确做法
events,event_type=click user_id=12345,request_id="abc123"
```

## Schema 设计建议

### 1. 优先使用 Tag 进行过滤

```txt
-- ✅ 推荐：使用 Tag 过滤
SELECT * FROM cpu
WHERE host = 'server01'  -- Tag，索引化，查询快
  AND region = 'us-west'  -- Tag，索引化，查询快

-- ❌ 避免：使用 Field 过滤
SELECT * FROM cpu
WHERE user_id = '12345'    -- Field，全表扫描，慢
```

### 2. 控制 Tag 数量

每个数据点的 Tag 越多，存储开销越大。建议每个 Measurement 的 Tag 数量控制在 10 个以内。

### 3. 合理命名

```txt
-- ✅ 推荐：简洁清晰的命名
cpu,host=server01,region=us-west

-- ❌ 避免：过长的命名
cpu_metrics_by_server_in_data_center,server_hostname=server01
```

### 4. 预先规划 Schema

```txt
-- 明确字段类型，避免类型混乱
-- 第一次写入
cpu,host=server01 usage=45.5  -- usage 是 Float

-- 后续写入必须一致
cpu,host=server01 usage=50.0  -- ✅ 继续 Float
cpu,host=server01 usage="high" -- ❌ 不能改成 String
```

## 总结

InfluxDB 的数据模型设计围绕时序数据的特性展开：

1. **Measurement** 是数据的容器，类似表名
2. **Tag** 是索引化的维度，适合过滤和分组
3. **Field** 是实际的指标值，不建立索引
4. **Timestamp** 是时间标识，精度可达纳秒

理解 Tag 和 Field 的区别，是掌握 InfluxDB 数据模型的关键。记住一个原则：**需要参与过滤和分组的用 Tag，存储实际数值的用 Field**。
