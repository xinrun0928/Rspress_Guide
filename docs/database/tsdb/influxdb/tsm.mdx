# InfluxDB TSM 存储引擎

InfluxDB 的存储引擎经历了从 LSM-Tree 到 TSM 的演进。TSM（Time-Structured Merge Tree）是专为时序数据优化的存储引擎。本篇文章将深入剖析 TSM 的原理和结构。

## 一、TSM 概述

### 1.1 TSM vs LSM-Tree

```
TSM 是 InfluxDB 专为时序数据设计的存储引擎:

LSM-Tree (LevelDB/RocksDB):
- 通用 Key-Value 存储
- 复合 Key: (Table, RowKey, Timestamp)
- 多次 Compaction

TSM (Time-Structured Merge Tree):
- 专为时序数据优化
- 索引按时间分区
- 单次 Compaction
- 更高的压缩率
```

## 二、写入流程

### 2.1 写入路径

```
写入流程:

┌─────────────────────────────────────────────────────────────┐
│                                                              │
│  写入请求 ──→ WAL (预写日志) ──→ Cache ──→ TSM Files       │
│                  │                   │            │           │
│                  │ 同步刷盘          │ 异步刷盘     │ 后台合并  │
│                  ▼                   ▼            ▼           │
│              保证持久性        Memory     Disk Files         │
│                                                              │
└─────────────────────────────────────────────────────────────┘

步骤:
1. 数据写入 WAL (保证持久性)
2. 数据写入 Cache (内存中的有序结构)
3. Cache 刷盘生成 TSM 文件 (达到阈值时)
4. 后台 Compaction 合并 TSM 文件
```

### 2.2 WAL (Write-Ahead Log)

```go
// WAL 格式
type WALEntry struct {
    Sequence  uint64
    Key       []byte    // Measurement + Tags + Fields
    Timestamp int64
    Value     interface{}
}

// WAL 写入
func (w *WAL) Write(entry *WALEntry) error {
    // 1. 序列号递增
    entry.Sequence = w.nextSequence()
    
    // 2. 写入内存缓冲区
    w.buffer = append(w.buffer, entry)
    
    // 3. 达到阈值时刷盘
    if len(w.buffer) >= w.batchSize {
        return w.sync()
    }
    
    return nil
}
```

### 2.3 Cache

```go
// Cache 结构 (ConcurrentSkipList)
type Cache struct {
    mu sync.Mutex
    data map[string]*Series // Key: measurement + tags
}

// Cache 数据结构
type Series struct {
    Key       string
    TagSet    map[string]string
    FieldSet  map[string][]Point // Field: Points
}

// Point 结构
type Point struct {
    Timestamp int64
    Value    interface{}
}
```

## 三、数据压缩

### 3.1 时间戳压缩

```
时间戳压缩算法:

Delta Encoding:
原始时间戳: 1705305600000, 1705305600001, 1705305600003
差值:       1705305600000, 1, 2
压缩效果:   大幅减少存储

Delta of Delta:
如果差值不变，使用更小的空间
1705305600000, 1, 1, 1, 1, 1
存储: 1705305600000, 1 (只需两个值)

典型压缩效果:
- 纳秒精度: ~3 字节/时间戳
- 毫秒精度: ~2 字节/时间戳
- 秒精度: ~1 字节/时间戳
```

### 3.2 数值压缩

```
Gorilla XOR 压缩算法:

原理:
1. 记录第一个值 (完整)
2. 后续值与前值 XOR
3. XOR 结果用可变长编码

示例:
值: 1.5, 1.6, 1.7, 1.5, 100.0, 100.1
XOR: 1.5, 0.1, 0.1, 0.2, 98.5, 0.1
压缩: 大幅减少存储

压缩效果:
- 浮点数: ~2 字节/值 (vs 8 字节原始)
- 整数: 视数值范围而定
```

### 3.3 标签压缩

```
标签压缩技术:

1. 字典编码
   - 相同标签值只存储一次
   - 标签键只存储一次
   
   示例:
   Tag Keys: [host, region, service] (3 个键)
   Tag Values: [server1, server2, api, web] (4 个值)
   字典索引: server1→0, server2→1, api→2, web→3

2. 位图压缩
   - 标记哪些 Series 有数据
   - 高效查询

压缩效果:
- 标签: ~10-50% 原始大小
- 总体: 5-10x 压缩率
```

## 四、Compaction

### 4.1 Compaction 类型

```
Compaction 层级:

Level 1 (Snapshot):
- Cache 刷盘生成 Snapshot 文件
- 触发条件: Cache 达到 25% 或 30 分钟

Level 2 (Level 1):
- 合并 Snapshot 和 Level 1 文件
- 触发条件: Level 1 达到阈值

Level 3+ (Level 2-6):
- 分层合并
- 每层文件数上限 10

Level 4 (Full):
- 合并所有 Level
- 生成新的 Level 4 文件
```

### 4.2 Compaction 过程

```
Compaction 流程:

┌─────────────────────────────────────────────────────────────┐
│                                                              │
│  Level N  Files                                               │
│  ┌─────┐ ┌─────┐ ┌─────┐                                    │
│  │ F1  │ │ F2  │ │ F3  │  (同一时间范围)                      │
│  └─────┘ └─────┘ └─────┘                                    │
│                                                              │
│  Read + Merge + Sort                                         │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 合并所有文件，去重，按时间排序                        │   │
│  └─────────────────────────────────────────────────────┘   │
│                           ↓                                   │
│  Level N+1 File                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 新文件，删除标记处理                                 │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 4.3 删除处理

```
删除操作 (Tombstone):

1. 写入删除标记
   - 不立即删除数据
   - 写入 Tombstone 标记

2. Compaction 时清理
   - 合并时检查 Tombstone
   - 被标记的数据不写入新文件
   - 物理删除

示例:
# 写入数据
cpu,host=server1 usage=45.5 1705305600

# 删除
DELETE FROM cpu WHERE host='server1'

# 实际写入
[Tombstone: cpu,host=server1]
(数据仍在文件中，直到 Compaction)
```

## 五、读取流程

### 5.1 读取路径

```
读取流程:

┌─────────────────────────────────────────────────────────────┐
│                                                              │
│  查询请求                                                     │
│      │                                                        │
│      ▼                                                        │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  1. 检查 Cache (最新数据)                            │   │
│  │     命中 → 返回                                       │   │
│  └─────────────────────────────────────────────────────┘   │
│      │ (未命中)                                            │
│      ▼                                                        │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  2. 遍历 TSM 文件 (从新到旧)                         │   │
│  │     - Index Block 定位                                 │   │
│  │     - Data Block 读取                                 │   │
│  │     - 合并多版本                                      │   │
│  └─────────────────────────────────────────────────────┘   │
│      │                                                        │
│      ▼                                                        │
│  返回结果                                                     │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 读取优化

```
读取优化:

1. Bloom Filter
   - 快速判断 Key 是否存在
   - 跳过不必要的 TSM 文件

2. Index Block 二分查找
   - 快速定位 Key 范围
   - 减少 Data Block 读取

3. 并行读取
   - 多文件并行读取
   - 利用多核 CPU
```

## 六、性能调优

### 6.1 关键参数

```bash
# Cache 大小
# 文件: /etc/influxdb/influxdb.conf
[data]
  cache-max-memory-size = "1g"        # 默认 1GB

# Cache 刷盘阈值
[data]
  cache-snapshot-memory-size = "25m"  # 默认 25MB

# Compaction 阈值
[data]
  compact-full-write-cold-duration = "4h"  # 4小时无写入触发强制 Compaction

# 最大系列数 (防止高基数)
[data]
  max-series-per-database = 1000000  # 默认 100 万
  max-values-per-tag = 100000         # 默认 10 万
```

## 总结

TSM 是 InfluxDB 的核心存储引擎：

1. **分层架构**：WAL → Cache → TSM Files → Compaction
2. **高效压缩**：时间戳 (Delta)、数值 (Gorilla)、标签 (字典)
3. **自动 Compaction**：分层合并，清理删除数据
4. **读取优化**：Bloom Filter、Index Block、并行读取
5. **性能调优**：Cache 大小、Compaction 参数、Series 限制

理解 TSM 原理，是深度优化 InfluxDB 性能的基础。
