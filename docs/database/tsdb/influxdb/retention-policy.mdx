# InfluxDB 保留策略

保留策略（Retention Policy，RP）是 InfluxDB 数据生命周期的核心机制。本篇文章将详细介绍 RP 的概念、配置和管理。

## 一、什么是保留策略

### 1.1 基本概念

```
保留策略 (Retention Policy):

定义: 定义数据在数据库中保留多长时间

核心参数:
1. DURATION: 数据保留多长时间
2. SHARD DURATION: 分片时长
3. REPLICATION: 副本数 (集群版)
4. DEFAULT: 是否为默认 RP
```

### 1.2 数据生命周期

```
数据生命周期:

┌─────────────────────────────────────────────────────────────┐
│                                                              │
│  数据写入 ──→ Shard ──→ 保留时长内 ──→ 自动删除              │
│                         ↓                                    │
│                    分片时长决定                               │
│                    数据分组方式                               │
│                                                              │
└─────────────────────────────────────────────────────────────┘

示例:
RP: one_day
- 数据保留 24 小时
- 超过 24 小时的数据自动删除
- Shard Duration: 1h (每小时一个分片)
```

## 二、创建保留策略

### 2.1 基本语法

```sql
CREATE RETENTION POLICY <policy_name>
ON <database>
DURATION <duration>
REPLICATION <n>
[SHARD DURATION <duration>]
[DEFAULT]
```

### 2.2 参数说明

```
参数详解:

1. DURATION
   - 数据保留时长
   - 语法: 1h, 1d, 7d, 30d, 52w, 1y
   - 0s: 永久保留 (不推荐)

2. REPLICATION (单机版固定为 1)
   - 数据副本数
   - 集群版使用

3. SHARD DURATION
   - 每个分片的时间范围
   - 语法: 1h, 1d, 7d, 52w
   - 默认根据 DURATION 自动选择

4. DEFAULT
   - 设为默认 RP
   - 不指定 RP 时使用此 RP
```

### 2.3 创建示例

```sql
-- 创建短期 RP (1 天)
CREATE RETENTION POLICY "one_day"
ON "mydb"
DURATION 1d
SHARD DURATION 1h
DEFAULT

-- 创建中期 RP (30 天)
CREATE RETENTION POLICY "one_month"
ON "mydb"
DURATION 30d
SHARD DURATION 1d

-- 创建长期 RP (1 年)
CREATE RETENTION POLICY "one_year"
ON "mydb"
DURATION 365d
SHARD DURATION 7d
```

## 三、分片时长

### 3.1 分片概念

```
分片 (Shard):

定义: 存储连续时间段数据的逻辑单元

特点:
- 每个分片对应一个 TSM 文件
- 分片时长决定数据分组
- 删除数据时按分片删除

分片时长对照表:

┌────────────────┬──────────────────────────────────────────────┐
│  SHARD DURATION │  适用 DURATION                               │
├────────────────┼──────────────────────────────────────────────┤
│  1h            │  <= 1 天                                      │
│  6h            │  <= 3 天                                     │
│  12h           │  <= 7 天                                     │
│  1d            │  <= 1 个月                                   │
│  7d            │  <= 3 个月                                   │
│  30d           │  <= 1 年                                     │
│  1y            │  > 1 年                                      │
└────────────────┴──────────────────────────────────────────────┘
```

### 3.2 分片时长影响

```
分片时长选择:

较短的 SHARD DURATION (如 1h):
优点:
- 删除数据时粒度更细
- Compaction 更高效
缺点:
- 大量小分片
- 元数据增加

较长的 SHARD DURATION (如 7d):
优点:
- 减少分片数量
- 元数据开销小
缺点:
- 删除数据时可能删除过多
- Compaction 涉及数据量大
```

## 四、多级保留策略

### 4.1 设计模式

```
典型多级 RP 设计:

┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  原始数据 (1 分钟精度)                                        │
│  RP: hot_1m                                                 │
│  DURATION: 7d                                               │
│  ────────────────────────────────────────────────────────   │
│                                                             │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  降采样数据 (5 分钟精度)                               │   │
│  │  RP: warm_5m                                          │   │
│  │  DURATION: 30d                                        │   │
│  └──────────────────────────────────────────────────────┘   │
│                           │                                  │
│                           ▼                                  │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  归档数据 (1 小时精度)                                  │   │
│  │  RP: cold_1h                                          │   │
│  │  DURATION: 365d                                       │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 完整示例

```sql
-- 1. 创建原始数据 RP (7 天, 1 分钟精度)
CREATE RETENTION POLICY "raw_1m"
ON "mydb"
DURATION 7d
SHARD DURATION 1h
DEFAULT

-- 2. 创建中期 RP (30 天, 5 分钟精度)
CREATE RETENTION POLICY "warm_5m"
ON "mydb"
DURATION 30d
SHARD DURATION 1d

-- 3. 创建长期 RP (1 年, 1 小时精度)
CREATE RETENTION POLICY "cold_1h"
ON "mydb"
DURATION 365d
SHARD DURATION 7d

-- 4. 创建归档 RP (永久保留, 1 天精度)
CREATE RETENTION POLICY "archive_1d"
ON "mydb"
DURATION INF
SHARD DURATION 30d
```

## 五、管理保留策略

### 5.1 查看 RP

```sql
-- 查看数据库的 RP
SHOW RETENTION POLICIES ON "mydb"

-- 结果示例:
-- name    duration  shardGroupDuration  replicaN  default
-- hot     24h0m0s  1h0m0s             1         false
-- warm    720h0m0s 24h0m0s            1         true
-- cold    8760h0m0s 168h0m0s         1         false
```

### 5.2 修改 RP

```sql
-- 修改 RP 的保留时长
ALTER RETENTION POLICY "one_month"
ON "mydb"
DURATION 60d

-- 修改为默认 RP
ALTER RETENTION POLICY "one_month"
ON "mydb"
DEFAULT

-- 修改分片时长
ALTER RETENTION POLICY "one_month"
ON "mydb"
SHARD DURATION 12h
```

### 5.3 删除 RP

```sql
-- 删除 RP
DROP RETENTION POLICY "old_policy"
ON "mydb"

-- 注意: 删除 RP 会删除所有关联数据
-- 执行前请确认!
```

## 六、数据写入和查询

### 6.1 写入到指定 RP

```bash
# 写入时指定 RP
curl -XPOST "http://localhost:8086/write?db=mydb&rp=one_month" \
  --data-binary 'cpu usage=45.5'

# 写入到非默认 RP
curl -XPOST "http://localhost:8086/write?db=mydb&rp=archive_1d" \
  --data-binary 'cpu usage=45.5'
```

### 6.2 从指定 RP 查询

```sql
-- 查询指定 RP 的数据
SELECT * FROM "one_month"."cpu" WHERE time > now() - 1d

-- 查询多个 RP
SELECT * FROM "raw_1m"."cpu" WHERE time > now() - 1h
UNION ALL
SELECT * FROM "warm_5m"."cpu" WHERE time > now() - 7d
UNION ALL
SELECT * FROM "cold_1h"."cpu" WHERE time > now() - 30d
```

## 七、自动删除机制

### 7.1 删除过程

```
数据删除流程:

1. Shard Group 过期检查
   - InfluxDB 定期检查 Shard Group 是否过期
   - 默认每 30 分钟检查一次

2. 删除过期分片
   - 过期的 Shard Group 被标记为删除
   - 后台任务删除分片文件

3. 元数据清理
   - 更新 meta 数据库
   - 清理相关索引
```

### 7.2 监控删除

```
查看 Shard 信息:

SHOW SHARDS

-- 结果:
-- id  database  retention_policy  shard_group  start_time           end_time
-- 1   mydb      one_day           1            2024-01-15T00:00:00Z  2024-01-15T01:00:00Z

SHOW SHARD GROUPS

-- 结果:
-- id  database  retention_policy  start_time           end_time
-- 1   mydb      one_day           2024-01-15T00:00:00Z  2024-01-15T01:00:00Z
```

## 八、最佳实践

### 8.1 RP 设计建议

```
设计原则:

1. 根据数据重要性分层
   - 重要数据: 长保留
   - 次要数据: 短保留

2. 配合降采样
   - 高精度: 短保留
   - 低精度: 长保留

3. 控制 Shard 数量
   - 太多小分片 → 元数据开销
   - 太少大分片 → 删除时影响大

4. 预留一定缓冲
   - DURATION 要大于实际需要
   - 留出备份和恢复的时间
```

### 8.2 存储估算

```
存储估算示例:

假设:
- 10 万个 Series
- 每 Series 每分钟 1 个点
- 压缩率 10x

计算:
- 原始数据 (1 分钟): 10万 × 60 × 24 = 144 亿点/天
- 1 天原始: ~1 GB
- 7 天原始: ~7 GB
- 30 天原始: ~30 GB

配合多级 RP:
- 1 分钟精度: 7 天 → ~7 GB
- 5 分钟精度: 30 天 → ~7 GB
- 1 小时精度: 365 天 → ~35 GB

总计: ~50 GB (vs 365 GB 全精度原始)
```

## 总结

保留策略是 InfluxDB 数据管理的核心：

1. **DURATION**：控制数据保留时长
2. **SHARD DURATION**：控制分片时长，影响删除粒度
3. **多级 RP**：配合降采样，实现精细化存储管理
4. **自动删除**：过期数据自动清理
5. **分层存储**：高频数据短期，低频数据长期

合理的 RP 设计，可以大幅降低存储成本。
