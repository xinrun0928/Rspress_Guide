## 超级表 vs 普通表：TDengine 的核心创新

100 万台电表，每台每秒上报数据。怎么设计表结构？

**MySQL 方案**：建 100 万张表 — 元数据管理复杂，跨表查询慢

**TDengine 方案**：1 张超级表 + 100 万张子表 — 元数据共享，聚合高效

```sql
-- 创建超级表（定义同一类型设备的数据结构）
CREATE STABLE power_meter (
    ts TIMESTAMP,        -- 时间戳
    voltage DOUBLE,       -- 电压
    current DOUBLE,       -- 电流
    power DOUBLE         -- 功率
) TAGS (
    meter_id BINARY(50), -- 电表 ID（标签）
    location BINARY(50),  -- 地理位置（标签）
    meter_type INT       -- 电表类型（标签）
);

-- 基于超级表创建子表（继承 schema，持有独立标签值）
CREATE TABLE meter_001 USING power_meter TAGS ('meter_001', 'Beijing', 1);
CREATE TABLE meter_002 USING power_meter TAGS ('meter_002', 'Shanghai', 2);
-- ... 一百万台电表

-- 超级表聚合查询（跨所有子表）
SELECT location, AVG(voltage), MAX(current)
FROM power_meter
WHERE ts > NOW() - 1d
GROUP BY location;
```

**超级表的核心优势**：
1. 标签 schema 共享，元数据开销极低
2. 标签自动索引，超级表聚合查询极快
3. 子表独立存储，支持高速单表查询

## TDengine vs InfluxDB vs TimescaleDB

| 维度 | TDengine | InfluxDB | TimescaleDB |
|------|----------|----------|-------------|
| **开源协议** | AGPL + 商业 | MIT + 商业 | Timescale License |
| **写入性能** | ~200 万/秒 | ~100 万/秒 | ~20 万/秒 |
| **集群** | 开源 | 商业版 | 部分开源 |
| **SQL 兼容** | 扩展 SQL | InfluxQL/Flux | 完整 SQL |
| **超级表/超表** | ✅ | ❌ | ✅ Hypertable |
| **适用场景** | IoT 物联网 | 通用监控 | PostgreSQL 扩展 |

**选 TDengine 的理由**：
- IoT/车联网场景，高写入并发
- 需要开源集群方案
- 数据量极大（> 1 亿时间线）

## 核心公式

```
TDengine 数据模型：
  超级表（STABLE）= 数据列（Columns）+ 标签schema（TAGS）
  子表（Table）= 继承超级表结构 + 独立标签值 + 独立时序数据

  100 万台电表 = 1 张超级表 + 100 万张子表

写入性能优化：
  VNode（虚拟节点）= 数据分片 + 副本管理
  MemTable = 写缓存（WAL + 内存排序后刷盘）
  SStable = 压缩存储文件

集群架构：
  mnode（管理节点）= 元数据存储（库、表、用户信息）
  vnode（数据节点）= 实际时序数据存储
  qnode（查询节点）= 可选，独立的查询计算

查询优化：
  单表查询 = 直接定位到子表，最快
  超级表查询 = 标签索引定位子表列表 + 并行扫描 + 聚合
  降采样 = 原始数据 → 时间窗口聚合 → 低精度表
```
