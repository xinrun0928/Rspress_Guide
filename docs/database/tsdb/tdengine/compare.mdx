# TDengine vs InfluxDB vs TimescaleDB

你的 IoT 平台需要选型时序数据库，面对 InfluxDB、TimescaleDB、TDengine 三个选择犯了难。

都是时序数据库，但各有各的特点。选错了，迁移成本可不低。

### 3. SQL 能力

```sql
-- TimescaleDB: 标准 SQL，无限制
SELECT * FROM t1 JOIN t2 ON t1.id = t2.id;
SELECT * FROM t WHERE EXISTS (SELECT 1 FROM t2 WHERE t2.id = t.id);
WITH cte AS (...) SELECT * FROM cte;

-- TDengine: 标准 SQL，有限制
SELECT * FROM t1 JOIN t2 ON t1.id = t2.id;  -- 支持 JOIN
SELECT * FROM t WHERE ...;  -- 支持子查询

-- InfluxQL: SQL-like，有限制
SELECT * FROM t WHERE time > now() - 1h;  -- 支持基本查询
SELECT MEAN(*) FROM t GROUP BY time(1h);  -- 支持聚合
-- 不支持 JOIN，不支持子查询
```

### 4. 架构设计

#### InfluxDB: Tag/Field 模型

```influxdb
-- Measurement: 表
-- Tags: 索引化的维度
-- Fields: 数值

cpu,host=server01,region=us-west usage=45.5,user=30.2 1709308800
        │                    │                       │
        │                    └─ Tags (索引)          │
        └──────────────────────── Measurement ────────┘
                                                │
                                                └─ Fields (值)
```

**优点**：
- 概念简单，易于理解
- Tags 自动索引

**缺点**：
- Series 基数过高会影响性能
- 高基数 Tag 需要谨慎设计

#### TimescaleDB: Hypertable + Chunk

```sql
-- Hypertable: 逻辑表，分区到底层的 Chunk
CREATE TABLE metrics (
    time TIMESTAMPTZ NOT NULL,
    device_id TEXT,
    location TEXT,
    value DOUBLE
);

SELECT create_hypertable('metrics', 'time');

-- Chunk: 按时间自动分区
-- 自动创建: metrics_1, metrics_2, ...
```

**优点**：
- 完全兼容 PostgreSQL
- 可以 JOIN 任意表
- 生态丰富

**缺点**：
- 写入性能不如专用 TSDB
- 需要了解 PostgreSQL 调优

#### TDengine: 超级表 + 子表

```sql
-- 超级表: 模板定义
CREATE STABLE power_meter (
    ts TIMESTAMP,
    voltage FLOAT,
    current FLOAT
) TAGS (
    meter_id INT,
    location VARCHAR(50)
);

-- 子表: 具体设备
CREATE TABLE meter_001 USING power_meter TAGS (1, 'Beijing');
CREATE TABLE meter_002 USING power_meter TAGS (2, 'Shanghai');
```

**优点**：
- 超级表 + 子表，适合海量设备
- 极高的写入性能
- 极低的存储成本

**缺点**：
- 子表数量有上限
- 学习曲线较陡

## 迁移复杂度

### 互相迁移的难度

```
InfluxDB ──────── TimescaleDB 困难 (数据模型差异大)
     │
     └──────────── TDengine        中等 (需转换 Line Protocol)
                         │
                         └──────── TimescaleDB 困难 (Schema 转换)
```

| 迁移方向 | 难度 | 说明 |
|---|---|---|
| InfluxDB `->` TDengine | 中等 | 转换 Line Protocol，修改查询语法 |
| TimescaleDB `->` TDengine | 困难 | 需重构超级表设计 |
| InfluxDB `->` TimescaleDB | 困难 | Schema 差异大 |
| TimescaleDB `->` InfluxDB | 困难 | 需拆分 Tag/Field |

## 总结与建议

### 一句话总结

| 数据库 | 一句话定位 |
|---|---|
| InfluxDB | 通用监控的首选，生态完善 |
| TimescaleDB | PostgreSQL 的时序扩展，SQL 能力最强 |
| TDengine | 物联网海量数据的利器，国产之光 |

### 选型决策树

```
需要选型时序数据库？
        │
        ├── 设备数量 > 10 万？
        │       ├── 是 ──▶ TDengine ⭐
        │       └── 否 ──▶ 下一步
        │
        ├── 需要复杂 SQL (多表 JOIN)？
        │       ├── 是 ──▶ TimescaleDB ⭐
        │       └── 否 ──▶ 下一步
        │
        ├── 已有 Telegraf 生态？
        │       ├── 是 ──▶ InfluxDB ⭐
        │       └── 否 ──▶ 下一步
        │
        └── 优先考虑 TDengine (性能 + 成本)
```

### 最终建议

- **IoT/车联网场景**：TDengine
- **DevOps 监控场景**：InfluxDB
- **混合工作负载 + SQL 需求**：TimescaleDB
- **国产信创要求**：TDengine
- **不确定先用哪个**：InfluxDB（生态最完善）

选型没有绝对的对错，只有适不适合。根据你的具体场景和团队情况，选择最合适的方案。
