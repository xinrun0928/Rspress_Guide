# Region 分裂机制

一张大表的数据如何分布到多台 RegionServer 上？当一个 Region 太大时，HBase 如何处理？这就是 Region 分裂机制要回答的问题。

Region 分裂是 HBase 实现水平扩展的核心机制。理解它，你就能理解 HBase 如何从几台服务器扩展到上千台，以及为什么预分区设计如此重要。

## Region 的基本概念

### Region 是什么？

Region 是 HBase 数据分片的基本单元。一张表最初只有一个 Region，随着数据增长，这个 Region 会分裂成两个，然后继续分裂，形成一棵 Region 分裂树。

```
Region 分裂树示例：
┌─────────────────────────────────────────────────────────────┐
│                                                              │
│  初始：Table 只有一个 Region                                 │
│  ┌─────────────────────────────────────────────────────┐   │
│  │              Table: users                             │   │
│  │                   [NULL - ∞)                         │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                              │
│  分裂1：分裂成 2 个 Region                                   │
│  ┌─────────────────┬─────────────────┐                    │
│  │ Region-A        │ Region-B        │                    │
│  │ [NULL - M)      │ [M - ∞)         │                    │
│  └─────────────────┴─────────────────┘                    │
│                                                              │
│  分裂2：Region-B 继续分裂                                   │
│  ┌─────────────────┬───────┬─────────┐                    │
│  │ Region-A        │Region-B│Region-C │                    │
│  │ [NULL - M)      │[M - P) │[P - ∞)  │                    │
│  └─────────────────┴───────┴─────────┘                    │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### Region 的元数据

每个 Region 都有以下元数据：

```java
// Region 元数据结构
public class HRegionInfo {
    private final TableName tableName;       // 表名
    private final byte[] startKey;           // 起始 RowKey（包含）
    private final byte[] endKey;             // 终止 RowKey（不包含）
    private final long regionId;             // Region 唯一 ID
    private final int replicaId;             // 副本 ID（用于 Region 复制）
    private final boolean split;             // 是否正在分裂
    private final boolean merged;            // 是否由合并产生
}
```

```
Region 定位流程：

```

## Region 分裂触发条件

### 分裂阈值

Region 分裂的触发条件是文件大小超过阈值：

| 参数 | 默认值 | 说明 |
| --- | --- | --- |
| `hbase.hregion.max.filesize` | 10GB | Region 文件大小超过此值时触发分裂 |
| `hbase.regionserver.region.split.policy` | 可配置 | 分裂策略类 |

### 分裂时机检查

RegionServer 定期检查 Region 是否需要分裂：

```java
// 分裂检查逻辑
public class SplitChecker {
    public boolean shouldSplit(Region region) {
        // 1. 检查是否已关闭自动分裂
        if (region.getTableDesc().isReadOnly()) {
            return false;
        }
        
        // 2. 检查是否正在分裂
        if (region.isSplitting()) {
            return false;
        }
        
        // 3. 检查文件大小
        long maxFileSize = getConfig().getLong("hbase.hregion.max.filesize");
        long currentSize = region.getSize();
        
        return currentSize >= maxFileSize;
    }
}
```

### 分裂时机选择

HBase 不会立即分裂，而是在下一次写入请求时触发：

```java
// 写入时检查是否需要分裂
public class RegionServer {
    public void put(Put put) {
        Region region = getRegion(put.getRow());
        
        // 写入前检查
        if (splitChecker.shouldSplit(region)) {
            // 触发分裂（异步）
            requestSplit(region);
        }
        
        // 执行写入
        region.put(put);
    }
}
```

## 分裂执行流程

### 分裂的六个步骤

Region 分裂是一个精心设计的过程，保证数据不丢失：

```
Region 分裂执行流程：
┌─────────────────────────────────────────────────────────────┐
│                                                              │
│  步骤1：准备分裂                                             │
│  ├─ RegionServer 在 ZooKeeper 创建分裂锁节点                │
│  │  /hbase/region-in-transition/{table}/{region}            │
│  ├─ 状态设置为 SPLITTING                                    │
│  └─ 获取分裂锁                                              │
│                                                              │
│  步骤2：下线父 Region                                        │
│  ├─ 停止对外服务                                            │
│  ├─ flush MemStore 到磁盘（确保数据持久化）                  │
│  └─ 关闭 HFile 引用                                         │
│                                                              │
│  步骤3：创建子 Region 目录                                   │
│  ├─ 在 HDFS 上创建两个子 Region 的目录结构                  │
│  ├─ Region-A: /hbase/data/table/region-a/                  │
│  └─ Region-B: /hbase/data/table/region-b/                  │
│                                                              │
│  步骤4：分裂 HFile                                          │
│  ├─ 遍历父 Region 的每个 HFile                             │
│  ├─ 根据 splitKey 将 HFile 分成两个部分                     │
│  ├─ 写入对应的子 Region 目录                                │
│  └─ 引用文件（Reference）而非复制文件                       │
│                                                              │
│  步骤5：更新 meta 表                                        │
│  ├─ 在 meta 表中注册两个子 Region                          │
│  ├─ 删除父 Region 的 meta 表记录                           │
│  └─ 更新状态为 SPLIT                                       │
│                                                              │
│  步骤6：上线子 Region                                        │
│  ├─ 通知 HMaster                                            │
│  ├─ HMaster 更新集群元数据                                  │
│  └─ 子 Region 可以对外服务                                  │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### Reference 文件

分裂时不会复制 HFile 数据，而是创建 Reference 文件：

```
Reference 文件机制：
┌─────────────────────────────────────────────────────────────┐
│                                                              │
│  父 Region HFile:                                            │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ HFile                                                │   │
│  │ ┌─────┬─────┬─────┬─────┬─────┬─────┐             │   │
│  │ │ Blk1│ Blk2│ Blk3│ Blk4│ Blk5│ Blk6│             │   │
│  │ └─────┴─────┴─────┴─────┴─────┴─────┘             │   │
│  │             ↑ splitKey 在 Blk3 和 Blk4 之间          │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                              │
│  分裂后：                                                    │
│  ┌───────────────────┐      ┌───────────────────┐          │
│  │ 子 Region-A HFile │      │ 子 Region-B HFile │          │
│  │                   │      │                   │          │
│  │ ┌─────┬─────┬───┐│      │ ┌───┬─────┬─────┐│          │
│  │ │Ref-A │Blk1 │Blk2│Blk3││ │Blk4│Blk5 │Blk6│Ref-B│   │
│  │ └─────┴─────┴───┘│      │ └───┴─────┴─────┴─────┘│   │
│  │   Reference文件  │      │   Reference文件            │   │
│  └───────────────────┘      └───────────────────┘          │
│                                                              │
│  Reference 文件只记录：                                       │
│  - 引用父 Region 的 HFile                                    │
│  - splitKey 位置                                             │
│  - 上/下半部分标识                                           │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 延迟删除与 Compaction

分裂后的 Reference 文件会在 Compaction 时被清理：

```
Reference 文件清理：
┌─────────────────────────────────────────────────────────────┐
│                                                              │
│  1. 分裂后，HFile 是 Reference 引用                         │
│     └─ HFile-A 引用父 HFile 的前半部分                       │
│     └─ HFile-B 引用父 HFile 的后半部分                       │
│                                                              │
│  2. Minor Compaction 时                                      │
│     └─ Reference 文件被合并                                  │
│     └─ 真正的数据文件被生成                                  │
│                                                              │
│  3. Major Compaction 时                                      │
│     └─ Reference 文件被完全清理                             │
│     └─ 父 Region 的 HFile 空间被释放                         │
│                                                              │
│  4. 清理完成                                                 │
│     └─ 子 Region 使用完整的数据文件                         │
│     └─ 父 Region 目录可以被删除                             │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

## 分裂点选择

### 默认策略：KeyPrefixRegionSplitPolicy

默认使用 KeyPrefixRegionSplitPolicy，分裂点要保持 RowKey 前缀的一致性：

```java
// KeyPrefix 分裂策略
public class KeyPrefixRegionSplitPolicy extends ConstantSizeRegionSplitPolicy {
    @Override
    protected byte[] getSplitPoint() {
        // 1. 获取当前 Region 的所有 RowKey
        byte[] startKey = region.getStartKey();
        byte[] endKey = region.getEndKey();
        
        // 2. 计算分裂点
        // 确保分裂后的两个 Region 有相同的 RowKey 前缀长度
        int splitLength = getSplitLength();  // 默认 2
        
        // 3. 找到 splitLength 位置的分隔符
        byte[] splitKey = findSplitKey(startKey, endKey, splitLength);
        
        return splitKey;
    }
}
```

### 何时需要自定义策略？

```
场景：业务需要按用户维度分裂
┌─────────────────────────────────────────────────────────────┐
│                                                              │
│  RowKey 设计：userId + orderId + timestamp                  │
│  例如：user_001_order_20240101120000000                     │
│                                                              │
│  问题：                                                      │
│  - 默认策略可能在对 orderId 的中间分裂                        │
│  - 导致同一用户的数据分散在两个 Region                       │
│                                                              │
│  解决：                                                      │
│  - 自定义分裂策略                                            │
│  - 只在 userId 的边界分裂                                   │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 自定义分裂策略示例

```java
// 自定义分裂策略
public class CustomSplitPolicy extends ConstantSizeRegionSplitPolicy {
    @Override
    protected byte[] getSplitPoint() {
        // 获取所有 store files
        Store store = region.getStore(CF);
        
        // 找到第 16 个字符是 '_' 的位置
        byte[] startKey = region.getStartKey();
        byte[] endKey = region.getEndKey();
        
        // 计算分裂点：保持 userId 前缀不变
        byte[] splitKey = new byte[startKey.length];
        System.arraycopy(startKey, 0, splitKey, 0, startKey.length);
        
        // 找到 userId 和 orderId 的分隔符位置
        for (int i = 0; i < splitKey.length; i++) {
            if (splitKey[i] == '_' && i > 8) {
                // 在 userId 后面的 '_' 处分裂
                return Arrays.copyOf(splitKey, i);
            }
        }
        
        return super.getSplitPoint();
    }
}
```

## 预分区

### 为什么要预分区？

如果没有预分区，表只有一个 Region，所有数据都写入同一台 RegionServer，导致热点问题。

```
无预分区的问题：
┌─────────────────────────────────────────────────────────────┐
│                                                              │
│  初始状态：                                                   │
│  ┌─────────────────────────────────────────────────────┐   │
│  │              Table: users                             │   │
│  │                   [NULL - ∞)                         │   │
│  │                   RegionServer-1                      │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                              │
│  写入 10TB 数据后：                                          │
│  - RegionServer-1 处理所有写入                              │
│  - 其他 RegionServer 空闲                                    │
│  - RegionServer-1 成为瓶颈                                  │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 如何预分区？

创建表时指定分裂点：

```java
// 预分区示例
Admin admin = connection.getAdmin();

// 定义分裂点
byte[][] splitKeys = {
    Bytes.toBytes("g"),      // [NULL - g), [g - ∞)
    Bytes.toBytes("m"),      // [NULL - m), [m - g), [g - ∞)
    Bytes.toBytes("t"),
    Bytes.toBytes("z")
};

// 创建表时指定分裂点
TableDescriptor table = TableDescriptorBuilder
    .newBuilder(TableName.valueOf("users"))
    .setColumnFamily(ColumnFamilyDescriptorBuilder
        .of("info"))
    .build();

admin.createTable(table, splitKeys);
```

### 预分区的最佳实践

```
预分区策略对比：
┌─────────────────────────────────────────────────────────────┐
│                                                              │
│  策略1：哈希预分区                                           │
│  优点：数据均匀分布                                          │
│  缺点：相邻 RowKey 可能不在同一 Region                       │
│                                                              │
│  RowKey: MD5(userId).substr(0,4) + userId                  │
│  splitKeys: [0000), [1000), [2000), ..., [f000)            │
│                                                              │
│  策略2：盐值预分区                                           │
│  优点：数据均匀分布，实现简单                                │
│  缺点：查询时需要遍历所有盐值                                │
│                                                              │
│  RowKey: (userId % 10) + userId                             │
│  splitKeys: [0_), [1_), [2_), ..., [9_)                    │
│                                                              │
│  策略3：业务维度预分区                                       │
│  优点：符合业务查询习惯                                      │
│  缺点：需要深入了解业务                                      │
│                                                              │
│  RowKey: userId + timestamp                                 │
│  splitKeys: 按地区/业务线预分                                │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

## 分裂对性能的影响

### 分裂时的短暂不可用

分裂过程中，父 Region 下线，子 Region 上线前，数据不可用：

```
分裂时序：
┌─────────────────────────────────────────────────────────────┐
│                                                              │
│  T0: 父 Region 处理请求                                      │
│  T1: 父 Region 下线，开始分裂                               │
│  T2: 分裂中，数据不可用                                      │
│  T3: 子 Region 上线，恢复服务                                │
│                                                              │
│  不可用时间 ≈ 分裂时间                                       │
│           ≈ HFile 分裂时间                                  │
│           ≈ 几十秒到几分钟                                   │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 分裂后的负载均衡

HMaster 负责分裂后的负载均衡：

```java
// 负载均衡逻辑
public class LoadBalancer {
    public void balanceRegionSplit(RegionServer oldServer, 
                                    RegionServer newServer1,
                                    RegionServer newServer2) {
        // 1. 统计每个 RegionServer 的 Region 数量
        Map<ServerName, List<RegionInfo>> clusterState = getClusterState();
        
        // 2. 如果两个子 Region 都在同一台 Server
        if (oldServer == newServer1 && oldServer == newServer2) {
            // 触发负载均衡
            assignRegions(clusterState);
        }
    }
}
```

### 分裂后的热点问题

分裂后，热点可能仍然存在：

```
热点场景：
┌─────────────────────────────────────────────────────────────┐
│                                                              │
│  RowKey = timestamp (错误设计)                               │
│                                                              │
│  初始状态：                                                   │
│  ┌─────────────────────────────────────────────────────┐   │
│  │              [NULL - ∞)                              │   │
│  │                   RegionServer-1                      │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                              │
│  分裂后：                                                    │
│  ┌─────────────────┬─────────────────┐                    │
│  │ [NULL - M)      │ [M - ∞)         │                    │
│  │ RegionServer-1  │ RegionServer-2  │                    │
│  └─────────────────┴─────────────────┘                    │
│                                                              │
│  问题：                                                      │
│  - 新数据永远写入 [M - ∞) 区域                              │
│  - RegionServer-2 仍然热点                                  │
│  - 分裂没有解决热点问题！                                    │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```
