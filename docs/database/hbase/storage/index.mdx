# HBase 存储总览

HBase 的存储系统是其高性能的根基。如果不理解 HBase 的存储机制，你可能会在调优和故障排查时摸不着头脑。

这篇文章会带你深入理解 HBase 的存储架构，从写入路径到读取路径，从 HFile 到 Compaction，构建完整的存储知识体系。

## 存储架构全景图

HBase 的数据存储涉及多个层次，每个层次都有其独特的设计和作用。

```
┌─────────────────────────────────────────────────────────────────┐
│                        写入路径                                  │
│                                                                  │
│   客户端 ──▶ ZooKeeper ──▶ RegionServer                        │
│                  │                      │                       │
│                  │                      ▼                       │
│                  │              ┌───────────────┐               │
│                  │              │  MemStore     │ (内存)        │
│                  │              │  排序写入     │               │
│                  │              └───────────────┘               │
│                  │                      │                       │
│                  │              flush (128MB)                  │
│                  │                      ▼                       │
│                  │              ┌───────────────┐               │
│                  │              │   HFile       │ (磁盘)        │
│                  │              │   有序存储     │               │
│                  │              └───────────────┘               │
```

## 核心存储组件

### MemStore：写缓存

MemStore 是写入的内存缓冲区，每个 Region 的每个 Column Family 都有一个独立的 MemStore。

**为什么需要 MemStore？**

HDFS 只支持顺序写入和追加修改，不支持随机写。如果没有 MemStore，每次写入都要直接落到 HDFS，性能会大打折扣。

MemStore 的设计让写入变成：
1. 写入内存（极快）
2. 内存写满后，批量刷写到 HDFS（顺序写，高效）

**MemStore 的数据结构**

MemStore 内部是一个 KeyValue 有序数组，按 RowKey + Column Family + Qualifier + Timestamp 排序。

```java
// MemStore 内部结构简化
public class MemStore {
    // KeyValue 按排序顺序存储
    private final KeyValueSkipListSet kvSet;
    
    // 快照用于 flush 期间继续写入
    private volatile KeyValueSnapshot snapshot;
    
    // 活跃的 Segment（1.6+ 引入的 MSLAB 优化）
    private final ConcurrentNavigableMap<Long, Segment> segments;
}
```

**MSLAB 优化**

HBase 1.6+ 引入了 MSLAB（MemStore-Local Allocation Buffer）来解决 GC 问题。

```
没有 MSLAB：
写入 MemStore → 频繁 new byte[] → 内存碎片 → CMS/G1 GC 频繁

有 MSLAB：
写入 MemStore → 从预分配的 Chunk 中分配 → 减少内存碎片 → 减少 GC
```

### HFile：持久化存储

HFile 是 HBase 在 HDFS 上的存储格式，是有序的键值对文件。一个完整的 HFile 结构会在后续章节详细讲解。

### WAL：预写日志

WAL（Write-Ahead Log）保证数据不丢失。每次写入必须先写入 WAL，只有 WAL 成功刷写到磁盘，操作才算完成。

```
WAL 作用：
┌──────────────┐          ┌──────────────┐
│  MemStore   │   数据    │    HFile     │
│  (内存)      │ ────────▶│  (磁盘)      │
└──────────────┘  flush   └──────────────┘
       │                              ↑
       │ 恢复                         │
       └──────▶ WAL ──────────────────┘
                (万一 MemStore 丢失)
```

WAL 的持久化策略：
- **同步写**：`hbase.wal.provider = defaultProvider`，每次写入都同步刷盘，最安全但最慢
- **异步写**（默认）：`hbase.wal.provider = asyncfs`，批量异步刷盘，性能更好

### BlockCache：读缓存

BlockCache 是 HBase 的读缓存，采用 LRU 策略。当读取数据时，先查 BlockCache，命中则直接返回；未命中则从 HFile 读取并放入 BlockCache。

```
BlockCache 结构：
┌────────────────────────────────────────────────────────┐
│                    BlockCache                           │
│                                                        │
│  L1: LruBlockCache (堆内，GC 影响)                      │
│  ├─ SingleQueue (单次访问)                             │
│  ├─ MultiQueue (多次访问)                              │
│  └─ InMemoryQueue (常驻内存)                           │
│                                                        │
│  L2: BucketCache (可选，堆外内存)                       │
│  └─ 从 L1 淘汰的 Block                                 │
│                                                        │
└────────────────────────────────────────────────────────┘
```

## 存储层级关系

HBase 的存储按以下层级组织：

```
Table
 └── Region (按 RowKey 范围划分)
      └── Column Family
           ├── MemStore (内存)
           │    ├── Segment 1
           │    ├── Segment 2
           │    └── ...
           │
           ├── WAL (预写日志)
           │    ├── WAL-1
           │    └── WAL-2
           │
           └── HFile (磁盘)
                ├── HFile-1
                ├── HFile-2
                ├── HFile-3
                └── ...
```

## 数据生命周期

### 写入阶段

```
① 写入请求到达 RegionServer
     │
     ▼
② 获取 RowKey 所在的 Region
     │
     ▼
③ 写入 WAL（保证持久化）
     │
     ▼
④ 写入 MemStore（内存缓冲区）
     │
     ▼
⑤ 返回客户端"写入成功"
```

### Flush 阶段

```
MemStore 大小达到阈值（默认 128MB）
     │
     ▼
创建 MemStore 快照（允许继续写入）
     │
     ▼
遍历快照中的 KeyValue，写入新 HFile
     │
     ▼
HFile 刷写完成，删除快照
     │
     ▼
WAL 中对应数据标记为已持久化（可以删除）
```

### Compaction 阶段

```
Minor Compaction：
多个小 HFile ──▶ 合并成一个大 HFile（保留所有版本）

Major Compaction：
一个 Region 的所有 HFile ──▶ 合并成一个 HFile（删除过期/删除数据）
```

## 存储与查询的关系

理解存储机制是理解查询性能的基础：

```
读取一个单元格时：
┌─────────────────────────────────────────────────────────────┐
│  1. 查 BlockCache ──▶ 命中？直接返回                       │
│                                                              │
│  2. 未命中 ──▶ 查 MemStore ──▶ 命中？返回并加入 BlockCache  │
│                                                              │
│  3. 未命中 ──▶ 扫描 HFile                                   │
│     - 使用布隆过滤器快速判断 Key 是否存在                    │
│     - 从索引 Block 定位数据 Block                            │
│     - 扫描数据，合并多版本结果                              │
│     - 返回结果并加入 BlockCache                              │
└─────────────────────────────────────────────────────────────┘
```

## 调优方向

理解存储机制后，调优方向就很清晰了：

| 问题 | 原因 | 调优方向 |
| --- | --- | --- |
| 写入慢 | WAL 同步慢 | 切换到异步 WAL |
| 读取慢 | BlockCache 命中率低 | 增大 BlockCache，调整数据结构 |
| Compaction 占用资源 | HFile 太多 | 调低 flush 阈值，增加 Major Compaction 频率 |
| GC 频繁 | MemStore 对象太多 | 启用 MSLAB |
| 内存不足 | BlockCache 太大 | 调整为 BucketCache（堆外） |
