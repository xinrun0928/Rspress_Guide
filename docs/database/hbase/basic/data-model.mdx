# HBase 数据模型

想象一个场景：你要存储用户的行为日志，每个用户每天可能产生几百条记录，包括浏览、点击、下单等多种行为。传统关系型数据库的表结构需要提前定义所有列，但行为类型可能随时增加。

HBase 的数据模型正是为这类场景设计的——它不要求事先定义所有列，同一行可以有完全不同的列，数据可以为空而不占空间。这就是 HBase 区别于关系型数据库的核心：**稀疏存储、多版本、动态列**。

## 四维坐标定位数据

在 HBase 中，定位一个单元格需要四个坐标：

```
(Table, RowKey, Column Family:Column Qualifier, Timestamp)
```

这与传统数据库的二维表（表 + 行 + 列）完全不同，可以把它想象成四维空间中的精确坐标。

### RowKey：数据的身份证

RowKey 是行的唯一标识，类似于 MySQL 的主键，但它有独特的特点：

- **按字典序排序**：RowKey `user_0001` 排在 `user_0002` 前面
- **唯一性**：整个表中只出现一次
- **决定数据分布**：RowKey 的范围决定了 Region 的划分
- **查询的唯一入口**：HBase 没有其他索引，唯一查询方式就是 RowKey

```
RowKey 排序示例：
user_001     ← 第一条
user_002     ← 第二条
user_010     ← 第十条
user_0100    ← 第一百条
```

:::warning 排序陷阱
字典序排序会导致 `user_010` 排在 `user_002` 后面（`1` `<` `2`），而不是直觉上的顺序。设计 RowKey 时要特别注意这一点。
:::

### Column Family：列的分组

Column Family（列族）是 HBase 存储的基本单元。相同列族的所有列数据存储在一起，这有几个重要含义：

**物理存储隔离**

```
Table: user_behavior

RowKey       CF: profile          CF: behavior
─────────────────────────────────────────────────────────────
user_001     profile:name         behavior:view
             profile:age          behavior:click
                                  behavior:purchase

user_002     profile:name         behavior:click
```

物理存储时，`profile` 和 `behavior` 列族的数据分别存储在不同的文件中。

**独立配置**

每个列族可以有不同的存储策略：

| 配置项 | 说明 |
| --- | --- |
| `BLOCKSIZE` | HFile 块大小，默认 64KB |
| `COMPRESSION` | 压缩算法（SNAPPY、LZO、GZ） |
| `TTL` | 数据过期时间（秒） |
| `BLOOMFILTER` | 布隆过滤器类型 |
| `IN_MEMORY` | 是否常驻内存 |

```java
// 为不同列族设置不同配置
ColumnFamilyDescriptor profile = ColumnFamilyDescriptorBuilder
    .newBuilder(Bytes.toBytes("profile"))
    .setBlocksize(8192)      // 小块，减少读取放大
    .setInMemory(true)        // 常驻内存，高频访问
    .build();

ColumnFamilyDescriptor behavior = ColumnFamilyDescriptorBuilder
    .newBuilder(Bytes.toBytes("behavior"))
    .setBlocksize(65536)      // 大块，顺序扫描友好
    .setCompression("SNAPPY") // 压缩，节省空间
    .setTTL(7776000)          // 90天过期
    .build();
```

**设计建议**

列族不宜过多，一般 2-3 个为宜。过多的列族会导致：
- 每个列族独立 flush，增加 I/O
- HFile 数量增加，影响读取性能
- 管理复杂度上升

### Column Qualifier：具体的列

Column Qualifier（列限定符）是列族内的具体列名。与 RowKey 和 Column Family 不同，Column Qualifier 可以在写入时动态创建，无需提前定义。

```
同一行的不同 Column Qualifier：
RowKey       info:username    info:email         info:phone
─────────────────────────────────────────────────────────────
user_001     "张三"            "zhang@example.com" "13800138000"
user_002     "李四"            "li@example.com"

user_003     "王五"            "wang@example.com"  ← 只有两列
```

:::tip 命名规范
Column Qualifier 使用短名称可以节省空间。建议使用固定长度的短编码，如 `n`（name）、`e`（email）、`p`（phone）。
:::

### Timestamp：时间旅行的钥匙

Timestamp（时间戳）用于区分同一单元格的不同版本。默认情况下，HBase 保留最近 3 个版本，但可以配置为任意数量。

**版本号的来源**

- 显式指定：写入时指定特定的时间戳
- 自动生成：不指定时，使用服务器当前时间（毫秒级）

```
同一单元格的多版本：
RowKey       info:status                  Timestamp
────────────────────────────────────────────────────────────
user_001    "active"                      1704067200001
            "inactive"                    1704067200000
            "pending"                     1704067199999
```

**读取行为**

读取时，默认返回最新版本。可以通过指定时间戳或版本数获取历史版本：

```java
// 获取最新版本
Get get = new Get(Bytes.toBytes("user_001"));
get.addColumn(Bytes.toBytes("info"), Bytes.toBytes("status"));
Result result = table.get(get);

// 获取最近 3 个版本
Get get = new Get(Bytes.toBytes("user_001"));
get.addColumn(Bytes.toBytes("info"), Bytes.toBytes("status"));
get.setMaxVersions(3);  // 返回所有版本
Result result = table.get(get);

// 获取指定时间戳之前的数据
Get get = new Get(Bytes.toBytes("user_001"));
get.addColumn(Bytes.toBytes("info"), Bytes.toBytes("status"));
get.setTimestamp(1704067200000L);
Result result = table.get(get);
```

## 逻辑视图 vs 物理存储

这是理解 HBase 数据模型的关键：**逻辑上看像一张表，物理上按列族分开存储**。

### 逻辑视图

从应用视角看，HBase 表是这样的：

```
Table: user_order

RowKey       order:product    order:amount    payment:method    payment:time
────────────────────────────────────────────────────────────────────────────────
order_001    "iPhone"         8999.00         "credit_card"     "2024-01-15"
order_002    "MacBook"         12999.00         "alipay"          "2024-01-16"
order_003    "AirPods"         1999.00                                          ← 缺列
```

### 物理存储

物理上，每个 Column Family 独立存储：

```
HFile (Column Family: order)
RowKey       order:product    order:amount    Timestamp
─────────────────────────────────────────────────────────────
order_001    "iPhone"         8999.00         t1
order_002    "MacBook"        12999.00        t1
order_003    "AirPods"        1999.00         t1

HFile (Column Family: payment)
RowKey       payment:method    payment:time    Timestamp
─────────────────────────────────────────────────────────────
order_001    "credit_card"    "2024-01-15"    t1
order_002    "alipay"         "2024-01-16"    t1
order_003    (空)              (空)            -  ← 稀疏存储，不占空间
```

:::tip 稀疏存储的优势
如果 `payment` 列族的数据大部分为空，稀疏存储可以节省大量空间。传统行式数据库每一行都必须为所有列分配空间。
:::

## 数据类型

HBase 存储的是字节数组（byte[]），不关心数据的具体类型。

```
字节数组表示：
"张三"     → [B, E, 6, 4, F, 8]      (UTF-8 编码)
123        → [0, 0, 0, 0, 0, 0, 7, B]  (int, 8字节)
true       → [1] 或 [T]
```

这意味着：
- 应用层负责数据类型转换
- 可以混合存储不同类型的数据
- 序列化/反序列化成为必需操作

### 常用序列化工具

```java
// 方式1：HBase 内置工具
String value = Bytes.toString(rowKey);
int num = Bytes.toInt(bytes);

// 方式2：Protostuff（推荐，性能好）
ProtostuffIOUtil.mergeFrom(bytes, object, schema);

// 方式3：JSON（可读性好，空间开销大）
ObjectMapper mapper = new ObjectMapper();
String json = mapper.writeValueAsString(object);

// 方式4：Hessian
Hessian2Output output = new Hessian2Output(byteArrayOutputStream);
output.writeObject(object);
```

## 空值处理

HBase 允许单元格为空，空单元格不占用存储空间（不存储任何东西）。

```java
// 写入一个空的列值
Put put = new Put(Bytes.toBytes("user_001"));
put.addColumn(
    Bytes.toBytes("info"),
    Bytes.toBytes("phone"),
    HConstants.EMPTY_BYTE_ARRAY  // 显式设置为空
);
table.put(put);

// 这个列实际上不存在，不会占用存储空间
```

:::warning 空列的陷阱
如果写入一个空值，HBase 会将这个列视为存在但值为空。这意味着：
- `getColumnAsNull` 返回 `true`
- 列族的大小会包含这个"空列"
- 建议完全跳过不写，而不是写空值
:::

## Scan：范围扫描

除了基于 RowKey 的精确查询，HBase 还支持范围扫描：

```java
// 扫描 RowKey 在 [user_001, user_010) 范围内的数据
Scan scan = new Scan();
scan.withStartRow(Bytes.toBytes("user_001"));
scan.withStopRow(Bytes.toBytes("user_010"));

// 扫描所有列
scan.readAllVersions();

// 限制返回的列族
scan.addFamily(Bytes.toBytes("info"));

// 限制返回的列
scan.addColumn(Bytes.toBytes("info"), Bytes.toBytes("name"));

// 每批次返回的行数（服务端批次，减少网络传输）
scan.setBatch(10);      // 每行最多返回10个列
scan.setCaching(100);    // 每批次返回100行

ResultScanner scanner = table.getScanner(scan);
for (Result result : scanner) {
    // 处理每一行
}
```

:::tip Scan 的性能优化
`setCaching` 设置越大，扫描越快，但内存消耗也越大。对于全表扫描，值可以设大一些（如 1000）；对于点查场景，设为 1 即可。
:::

## 设计模式

### 模式一：用户画像

```java
// RowKey = 用户ID
// 存储用户的各种属性
RowKey       profile:*     behavior:*    preference:*
────────────────────────────────────────────────────────────
user_001     profile:name   behavior:view  preference:category
             profile:age    behavior:buy   preference:brand
             profile:city
```

### 模式二：时间序列

```java
// RowKey = 指标ID + 时间戳反转
// 反转时间戳使新数据排在前面
RowKey = metricId + String.format("%015d", Long.MAX_VALUE - timestamp)

RowKey       data:value    data:quality
────────────────────────────────────────────────────────────
cpu_001      85.5          "good"
memory_001   72.3          "good"
disk_001     45.2          "warning"
```

### 模式三：关联关系

```java
// RowKey = 主实体ID + 从实体ID
// 存储实体间的多对多关系
RowKey                       relation:types
────────────────────────────────────────────────────────────
user_001_follow_user_002     "follow"
user_001_follow_user_003     "follow"
user_002_follow_user_001     "follow"

user_001_like_product_1001   "like"
user_002_like_product_1001    "like"
```

## 常见误区

| 误区 | 正确理解 |
| --- | --- |
| HBase 有 SQL | 没有，HBase 只有 Key-Value API |
| 可以任意添加列 | 可以添加 Column Qualifier，但 Column Family 数量有限 |
| 空值会占空间 | 空单元格不存储，不会占空间 |
| RowKey 可以重复 | RowKey 必须唯一，重复会覆盖旧数据 |
| 支持跨行事务 | 只支持单行事务，跨行需要业务层协调 |
