# CAP 权衡：HBase 选择了 CP

2000 年，Eric Brewer 提出了 CAP 猜想；2002 年，Seth Gilbert 和 Nancy Lynch 证明了它。这就是分布式系统领域最重要的定理之一：**CAP 定理**。

但很多人对 CAP 定理的理解是片面的。他们会说："HBase 是 CP 系统，Cassandra 是 AP 系统，所以 AP 比 CP 好。"这种理解忽略了 CAP 定理的核心：**CAP 描述的是系统在分区时的行为，而不是系统的固有属性**。

## CAP 定理的本质

CAP 定理指出：一个分布式系统不可能同时满足以下三个特性：

| 特性 | 含义 |
| --- | --- |
| **C**onsistency（一致性） | 所有节点在同一时刻看到相同的数据 |
| **A**vailability（可用性） | 每个请求都能在有限时间内得到响应 |
| **P**artition Tolerance（分区容错） | 系统在网络分区时仍能运行 |

```
          Consistency
               ▲
               │
      HBase    │    Cassandra
        │      │
        │      │
────────┼──────┼─────────────► Availability
        │      │
        │      │
        │    最终一致
        │
    强一致
```

:::tip 关键理解
网络分区是必然发生的，不是可选的。在分布式系统中，我们必须在 C 和 A 之间做出选择。因此，"CA 系统"在实践中不存在——一旦发生分区，你必须放弃一致性或可用性。
:::

## HBase 的选择：强一致性

HBase 选择的是 **CP**——在网络分区发生时，优先保证一致性，放弃部分可用性。

### 强一致性体现在哪里？

**1. 行级原子性**

HBase 保证同一行的读写是原子操作，不存在部分更新的情况：

```java
// 场景：更新用户的订单数量和总金额
// HBase 保证这两个操作要么同时成功，要么同时失败
Put put = new Put(Bytes.toBytes("user_001"));
put.addColumn(Bytes.toBytes("order"), Bytes.toBytes("count"), Bytes.toBytes("5"));
put.addColumn(Bytes.toBytes("order"), Bytes.toBytes("total"), Bytes.toBytes("9999"));

// 不存在：count 更新了但 total 没更新的情况
table.put(put);
```

**2. 写入的线性一致性**

所有写入必须通过 WAL 持久化到磁盘，数据才能返回成功。这意味着：

```
客户端 ──写入──▶ RegionServer ──写WAL──▶ HDFS
                    │
                    │ WAL 同步完成
                    ▼
                返回成功
```

即使 RegionServer 宕机，未刷盘的数据也可以通过 WAL 重放恢复。

**3. 读取的全局视野**

HBase 的读取可以配置为强一致性读取（Read-your-writes）：

```java
// 强一致性读：确保读取到最新写入的数据
Get get = new Get(Bytes.toBytes("user_001"));
get.setConsistency(Consistency.STRONG);  // 默认值
Result result = table.get(get);

// 最终一致性读：可能读取到稍旧的数据，性能更好
get.setConsistency(Consistency.TIMELINE);
Result result = table.get(get);
```

### 分区时的行为

当网络分区发生时，HBase 的行为：

```
正常情况：
┌─────────────┐         ┌─────────────┐
│ RegionServer│◄───────►│ ZooKeeper   │
│ (主节点)     │         │             │
└─────────────┘         └─────────────┘
         ▲
         │ 心跳
         │
┌────────┴────────┐
│   客户端写入      │
└──────────────────┘

分区情况：
┌─────────────┐     ✗     ┌─────────────┐
│ RegionServer│          │ ZooKeeper   │
│ (主节点)     │          │             │
└─────────────┘          └─────────────┘
    │
    │ 心跳超时（90秒），RegionServer 被判定宕机
    ▼
Region 被标记为 offline，重新分配给其他节点
期间：相关 RowKey 无法写入（CP 选择）
```

:::warning 可用性牺牲
当 ZooKeeper 无法访问时，整个 HBase 集群无法响应写入请求。对于某些业务场景，这可能是不可接受的。
:::

## Cassandra 的选择：最终一致性

与 HBase 不同，Cassandra 选择的是 **AP**——在网络分区发生时，优先保证可用性，接受最终一致性。

### 最终一致性体现在哪里？

**1. 多副本写入**

数据可以写入任意副本，然后异步同步到其他副本：

```sql
-- Cassandra 允许设置一致性级别
INSERT INTO users (id, name) VALUES (1, '张三');

-- 写操作：写入本地节点即可成功
CONSISTENCY ANY;  -- 任意一个节点写入成功即可

-- 或者要求多数节点确认
CONSISTENCY QUORUM;  -- 需要超过半数节点确认
```

**2. 可配置的一致性级别**

```java
// Cassandra 支持灵活的一致性配置
Session session = cluster.connect("mykeyspace");

// 每次查询都可以指定一致性级别
session.execute(
    "SELECT * FROM users WHERE id = ?",
    ConsistencyLevel.ONE,      // 读任意一个节点
    ConsistencyLevel.QUORUM,    // 读多数节点
    ConsistencyLevel.ALL        // 读所有节点（强一致性）
);
```

**3. 分区时的行为**

网络分区时，Cassandra 允许在本地节点写入成功即返回，数据最终会同步：

```
分区情况：
┌─────────────┐     ✗     ┌─────────────┐
│ Cassandra    │          │ 其他节点     │
│ 节点A        │          │             │
└─────────────┘          └─────────────┘
    │
    │ 写入成功（AP 选择）
    ▼
返回客户端"写入成功"
    │
    │ 后续异步同步到其他节点
    ▼
最终所有节点数据一致
```

:::tip AP 的代价
AP 系统可能返回过期数据。如果业务不能接受读取到旧数据，Cassandra 需要配置更高的一致性级别，这又会影响可用性。
:::

## 深入对比：一致性模型

### 写一致性

| 级别 | 说明 | 容错能力 |
| --- | --- | --- |
| `ONE` | 写入 1 个节点 | n-1 个节点故障 |
| `QUORUM` | 写入多数节点 | n/2 个节点故障 |
| `ALL` | 写入所有节点 | 0（任何节点故障都失败） |

### 读一致性

| 级别 | 说明 | 特点 |
| --- | --- | --- |
| `ONE` | 读 1 个节点 | 快速，可能读到旧数据 |
| `QUORUM` | 读多数节点 | 平衡性能和数据新鲜度 |
| `LOCAL_ONE` | 读本地数据中心节点 | 低延迟 |
| `EACH_QUORUM` | 每个数据中心读多数节点 | 跨数据中心强一致 |

### HBase vs Cassandra 一致性对比

```
场景：写入后立即读取

HBase（强一致性）：
写入 ──▶ 确认 ──▶ 读取 ──▶ 一定能读到最新数据

Cassandra（最终一致性，可能发生）：
写入 ──▶ 确认 ──▶ 读取 ──▶ 可能读到旧数据（需要等待同步）
```

## 实际选择：业务场景决定

### 什么时候选 HBase（CP）？

**1. 金融交易系统**

```
场景：银行转账、订单支付
要求：资金不能出错，一致性优先
选择：HBase

- 写入时确保数据落盘
- 读取时确保读到最新数据
- 分区时宁可拒绝服务，也不能返回错误余额
```

**2. 库存管理系统**

```
场景：电商库存扣减
要求：库存不能超卖，一致性优先
选择：HBase

- 强一致性写入防止超卖
- 读取最新库存确保业务正确
```

**3. 用户权限系统**

```
场景：权限变更即时生效
要求：权限变更后立即可见
选择：HBase

- 用户权限变更立即生效
- 不存在"新权限未同步"的空窗期
```

### 什么时候选 Cassandra（AP）？

**1. 物联网时序数据**

```
场景：传感器数据采集
要求：数据写入不能丢，可用性优先
选择：Cassandra

- 传感器持续上报数据，不能因为网络抖动丢失
- 允许短暂的数据不一致，后续可修正
```

**2. 社交媒体消息**

```
场景：用户发帖、评论
要求：发出去就行，不要求立即所有人可见
选择：Cassandra

- 用户发消息后立即看到"发送成功"
- 其他用户稍后看到（最终一致）
```

**3. 日志采集系统**

```
场景：大规模日志收集
要求：采集端不能阻塞，可用性优先
选择：Cassandra

- 日志写入本地即可
- 后台异步同步到集群
```

## HBase 的可用性优化

HBase 选择 CP，不代表它完全放弃可用性。在正常情况下，HBase 的可用性是很高的：

### 1. RegionServer 故障恢复

```
故障检测：ZooKeeper 心跳超时（90秒）
     ↓
故障确认：HMaster 标记 RegionServer 为宕机
     ↓
Region 迁移：快速将 Region 分配给其他节点
     ↓
数据恢复：通过 WAL 重放恢复未刷盘数据
     ↓
恢复时间：通常 30-90 秒
```

### 2. 读写分离（2.x 新特性）

HBase 2.x 引入了 Region 复制特性：

```xml
<!-- hbase-site.xml -->
<property>
  <name>hbase.region.replication</name>
  <value>true</value>
</property>
```

- **主副本**：处理写操作和强一致读
- **从副本**：处理最终一致性读（可以读到稍旧的数据）

```
主副本宕机：从副本自动升级为主副本
读流量分摊：读请求可以分散到多个从副本
```

### 3. 多活架构

对于跨数据中心场景：

```
数据中心 A ───┬───► 数据中心 B
   HBase      │       HBase
   集群        │        集群
              │
              └───────► 异步复制
```

- 单个数据中心内部强一致
- 数据中心之间最终一致
- 任何数据中心故障，不影响其他数据中心

## CAP 的现代理解

2012 年，Eric Brewer 本人在论文中更新了 CAP 的理解：

> "CAP 反对的是：不必要的一致性保证，或者不成熟的分区容错设计。"

这意味着：

**1. 一致性不是非黑即白的**

```
强一致性 ────────► 最终一致性

强一致 → 线性一致 → 因果一致 → 最终一致

可以选择不同级别的"一致性"
```

**2. 分区期间的处理是可定制的**

现代分布式系统不是简单地选择 CP 或 AP，而是：

```
分区发生：
├── 检测分区
├── 进入特殊模式
│   ├── 降级某些功能
│   ├── 记录不一致状态
│   └── 限制部分操作
└── 分区恢复后
    ├── 合并状态
    └── 补偿不一致
```

**3. HBase 也在进化**

HBase 社区一直在探索提升可用性的方案：

- Region 复制（异步读副本）
- 读写分离（主副本 vs 从副本）
- 多活（跨数据中心复制）

## 面试追问方向

- CAP 定理的核心是什么？为什么说"CA 系统"在实践中不存在？
- HBase 和 Cassandra 在 CAP 取舍上有什么区别？
- HBase 的强一致性具体体现在哪些方面？
- 分区发生时，HBase 和 Cassandra 的行为分别是什么？
- 如果业务既需要强一致性，又需要高可用性，怎么办？
- HBase 2.x 的 Region 复制特性如何平衡一致性和可用性？

理解 CAP 权衡，你就能理解为什么 HBase 适合某些场景，而 Cassandra 适合另一些场景。这不是优劣之分，而是场景匹配。
