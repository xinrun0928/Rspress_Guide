# RowKey 设计原则

你在监控系统中配置了 10 万台设备的数据采集，每秒写入量 50 万条。系统上线后第 3 天，监控告警：RegionServer-3 的 CPU 使用率 98%，其他节点却空闲。这是典型的 **热点问题（Hot Spotting）**——数据分布不均，大量请求集中在少数 Region 上。

问题出在哪？你的 RowKey 设计用了 `deviceId` 作为前缀，而设备 ID 是递增的（如 `device_000001` 到 `device_100000`）。所有新数据都写入 RowKey 前缀相同的 Region，导致热点集中。

RowKey 设计是 HBase 最重要的架构决策。选错 RowKey，表就会陷入性能噩梦；选对 RowKey，海量数据的随机读写才能真正发挥 HBase 的优势。本篇文章 5000+ 字，系统讲解 RowKey 设计的所有核心原则和实战技巧。

## 一、RowKey 的核心地位

### 1.1 为什么 RowKey 如此关键

在 HBase 中，RowKey 不仅仅是一个主键，它决定了：

- **数据分布**：RowKey 决定数据落在哪个 Region，从而决定由哪台 RegionServer 处理
- **查询路径**：基于 RowKey 的查询可以直接定位到目标 Region，无需全表扫描
- **读写性能**：顺序写入 vs 随机写入、范围查询 vs 点查询，性能差异可达百倍
- **热点风险**：错误的 RowKey 设计会导致热点集中，单点成为瓶颈

```
RowKey 设计对比：

设计 A：device_id + timestamp
device_000001_20240115000000
device_000002_20240115000001
device_000003_20240115000002
...
device_000001_20240115000010  ← 数据按 device_id 分组，但写入集中

设计 B：timestamp + device_id（随机分布）
20240115000000_device_000001
20240115000000_device_000002
...
20240115000010_device_000001
写入分散到多个 Region，但无法按设备聚合查询
```

### 1.2 RowKey 的三大特性

```java
// RowKey 三大特性
public class RowKeyProperties {
    
    // 1. 唯一性：每行数据的 RowKey 必须唯一
    // 类似关系型数据库的主键
    // 不同行可以有相同的列值，但 RowKey 必须不同
    String uniqueRowKey = "order_" + orderId; // 必须唯一
    
    // 2. 字典序：RowKey 按字节数组的字典序排列
    // 字节 10 < 字节 11 < 字节 12
    // 字符串 "2" > "10" (因为 '2' > '1')
    String lexicographic = "compareRowKeys";
    
    // 3. 不可修改：RowKey 写入后不能直接修改
    // 如果需要修改，只能 delete + put
    // 这要求设计时考虑未来的扩展性
}
```

## 二、RowKey 设计的黄金法则

### 法则一：长度适中

RowKey 长度直接影响存储效率和查询性能：

- **太短**：信息量不足，可能导致热点或碰撞
- **太长**：每个 HFile 的索引中都会存储 RowKey，过长会占用大量磁盘
- **推荐**：10-100 字节，通常 16-50 字节最佳

```java
// RowKey 长度计算示例
// 场景：用户行为分析系统

// 方案 A: 短 RowKey
String rowKeyA = String.valueOf(userId); // 8-10 字节
// 问题: userId 范围 1-10亿，10亿个不同前缀 → 分布均匀但无聚合能力

// 方案 B: 适中 RowKey
String rowKeyB = String.format("%020d_%s", userId, eventType); // 28-35 字节
// userId 左补0保证字典序，eventType 聚合同类事件
// 推荐：平衡了分布均匀和查询聚合

// 方案 C: 长 RowKey
String rowKeyC = userId + "_" + eventType + "_" + UUID.randomUUID(); // 50-100 字节
// 问题: UUID 随机，无聚合能力；且占用过多存储
```

### 法则二：散列均匀

数据的 Region 分布应该尽量均匀，避免热点：

```
错误示例（连续递增）：
000001, 000002, 000003, ...  → 写入集中在最后一个 Region

正确示例（哈希打散）：
hash(000001) % 100 = 42 → Region-42
hash(000002) % 100 = 17 → Region-17
hash(000003) % 100 = 89 → Region-89
```

### 法则三：查询友好

RowKey 的排列顺序决定了查询模式：

- **等值查询**：RowKey = 查询值，直接定位
- **范围查询**：STARTROW + ENDROW，高效扫描
- **前缀查询**：利用字典序特性，按前缀匹配

```java
// 查询模式与 RowKey 设计对照

// 场景 A: 用户订单查询
// 查询方式: 等值查询 userId，直接定位
// RowKey: userId(左补0) + orderId
String rowKey = String.format("%020d_%s", userId, orderId);

// 场景 B: 时间序列查询
// 查询方式: 范围查询最近 7 天的数据
// RowKey: 反转时间戳 + metricId
long reversedTs = Long.MAX_VALUE - timestamp;
String rowKey = String.format("%016d_%s", reversedTs, metricId);
// 最新数据排在前，旧数据自动沉底

// 场景 C: 复合查询
// 查询方式: 先按用户分组，再按时间排序
// RowKey: userId(左补0) + 反转时间戳
String rowKey = String.format("%020d_%016d", userId, reversedTs);
```

### 法则四：避免单调递增/递减

单调递增或递减的 RowKey 是生产环境的头号杀手：

```java
// 反面教材: 时间戳作为 RowKey 前缀
String badRowKey = System.currentTimeMillis() + "_" + dataId;
// 20240115100000_001
// 20240115100001_001
// 20240115100002_001
// ...所有新数据都追加到末尾，所有写入都命中 Region 的尾部

// 问题分析:
// Region 按 RowKey 范围分裂: [0-5GB) → [5GB-10GB) → [10GB-...]
// 单调递增的 RowKey 导致新数据永远写入最大的那个 Region
// 该 RegionServer 成为热点，其他 RegionServer 空闲
```

## 三、RowKey 设计模式

### 3.1 模式一：加盐（Salting）

**原理**：在 RowKey 前添加随机前缀，将数据分散到不同的 Region。

```java
public class SaltingRowKey {
    
    /**
     * 生成带盐值的 RowKey
     * @param originalRowKey 原始 RowKey
     * @param numSalts 盐值数量，建议设为 RegionServer 数量的倍数
     * @return 加盐后的 RowKey
     */
    public static String generateSaltingRowKey(String originalRowKey, int numSalts) {
        // 方法 1: 随机盐值（简单但不保证均匀）
        int salt = new Random().nextInt(numSalts);
        return salt + "_" + originalRowKey;
        
        // 方法 2: 哈希盐值（更均匀）
        // int salt = Math.abs(originalRowKey.hashCode()) % numSalts;
        // return salt + "_" + originalRowKey;
    }
}
```

**优缺点**：

| 优点 | 缺点 |
|---|---|
| 写入完全均匀分散 | 无法按原始 RowKey 范围查询 |
| 彻底解决热点问题 | 客户端需要知道盐值数量 |
| 实现简单 | 扫描操作需要跨多个盐值 |

**适用场景**：写入密集型、数据量大、查询主要是点查的场景。

```java
// 示例：物联网传感器数据采集
// 原始 RowKey: sensorId_timestamp
// 问题: sensorId 递增，导致写入热点

// 加盐后: N_salt_sensorId_timestamp
// 数据均匀分散到 N 个 Region
// 查询: 客户端需要尝试所有 N 个前缀才能做范围查询（不推荐）
// 查询: 点查可以直接定位（推荐）
```

### 3.2 模式二：哈希（Hashing）

**原理**：对 RowKey 的关键字段做哈希，保留前缀用于查询。

```java
public class HashingRowKey {
    
    /**
     * 生成哈希 RowKey
     * @param userId 用户ID
     * @param orderId 订单ID
     * @param numBuckets 分桶数量
     * @return 哈希 RowKey
     */
    public static String generateHashRowKey(long userId, String orderId, int numBuckets) {
        // 取用户 ID 的哈希值前 N 位作为前缀
        String hashPrefix = String.format("%02x", userId % numBuckets);
        return hashPrefix + "_" + String.format("%020d", userId) + "_" + orderId;
    }
}
```

**优缺点**：

| 优点 | 缺点 |
|---|---|
| 查询时可以按前缀定位 | 哈希碰撞（不同用户可能在同一桶） |
| 数据分布相对均匀 | 无法精确控制每个桶的大小 |
| 可以做范围查询（按哈希前缀） | 热点问题转移到哈希冲突 |

### 3.3 模式三：反转（Reversal）

**原理**：反转 RowKey 中的某些字段，打破单调性。

```java
public class ReversalRowKey {
    
    /**
     * 生成反转 RowKey
     * 适用于: 手机号、身份证号等有规律但单调性不强的字段
     */
    public static String generateReversalRowKey(String phoneNumber, String recordId) {
        // 反转手机号
        String reversedPhone = new StringBuilder(phoneNumber).reverse().toString();
        return reversedPhone + "_" + recordId;
    }
    
    /**
     * 生成反转时间戳 RowKey
     * 适用于: 时间序列数据，希望新数据排在前面
     */
    public static String generateReversedTimestampRowKey(long timestamp, String metricId) {
        long reversedTimestamp = Long.MAX_VALUE - timestamp;
        return String.format("%016d_%s", reversedTimestamp, metricId);
    }
}
```

**示例对比**：

```
原始手机号 RowKey:
13800138000_order_001
13800138001_order_001
13800138002_order_001
...手机号递增/递减导致热点

反转手机号 RowKey:
00001883001_order_001  ← 同一尾号的用户聚合在一起
10001883001_order_001
20001883001_order_001
...分布更均匀
```

### 3.4 模式四：复合 RowKey（最常用）

**原理**：组合多个字段，形成具有特定语义的 RowKey。

```java
public class CompositeRowKey {
    
    /**
     * 复合 RowKey: 用户 + 订单
     * 适用场景: 按用户查询所有订单
     */
    public static String userOrderRowKey(long userId, String orderId) {
        return String.format("%020d_%s", userId, orderId);
    }
    
    /**
     * 复合 RowKey: 用户 + 时间 + 订单
     * 适用场景: 按用户和时间范围查询订单
     */
    public static String userTimeOrderRowKey(long userId, long timestamp, String orderId) {
        long reversedTimestamp = Long.MAX_VALUE - timestamp;
        return String.format("%020d_%016d_%s", userId, reversedTimestamp, orderId);
    }
    
    /**
     * 复合 RowKey: 商品 + 店铺 + 订单
     * 适用场景: 按商品查询所有销售订单
     */
    public static String productShopOrderRowKey(String productId, String shopId, String orderId) {
        return productId + "_" + shopId + "_" + orderId;
    }
}
```

**设计原则**：

- **高频查询字段放前面**：HBase 按字典序匹配，越靠前的字段越容易命中
- **区分度高的字段放前面**：减少扫描行数
- **长度固定的字段放前面**：便于解析

## 四、实战案例

### 案例一：用户行为分析系统

**业务需求**：

- 存储用户点击、浏览、收藏等行为数据
- 每日数据量：10 亿条
- 主要查询：某用户的最近 N 条行为、按时间范围查询某用户行为

**RowKey 设计**：

```java
/**
 * 用户行为表 RowKey 设计
 * 格式: userId(20位左补0) + 反转时间戳(16位) + eventId(32位UUID)
 * 
 * 设计理由:
 * 1. userId 放最前面，支持按用户查询所有行为
 * 2. 反转时间戳让新数据排在前面，支持"最近 N 条"的查询
 * 3. eventId 唯一标识每条行为记录
 * 
 * 查询场景:
 * - 查询用户最近 100 条行为: scan startRow=userId, stopRow=userId+1
 * - 查询某时间段行为: scan with Timerange
 */
public class UserBehaviorRowKey {
    
    private static final int USER_ID_LENGTH = 20;
    private static final int TIMESTAMP_LENGTH = 16;
    
    public static String generate(long userId, long timestamp, String eventId) {
        long reversedTimestamp = Long.MAX_VALUE - timestamp;
        return String.format("%0" + USER_ID_LENGTH + "d", userId) 
             + String.format("%0" + TIMESTAMP_LENGTH + "d", reversedTimestamp)
             + eventId;
    }
    
    public static String getStartRowKey(long userId) {
        return String.format("%0" + USER_ID_LENGTH + "d", userId);
    }
    
    public static String getStopRowKey(long userId) {
        return String.format("%0" + USER_ID_LENGTH + "d", userId + 1);
    }
}
```

### 案例二：物联网时序数据

**业务需求**：

- 10 万台设备，每台每秒上报 5 条数据
- 查询：某设备最近 N 条数据、按时间范围查询

**RowKey 设计**：

```java
/**
 * 物联网时序数据 RowKey 设计
 * 
 * 方案对比:
 * A. deviceId + timestamp → 写入热点（所有设备同时上报，数据都写到同一个 Region 的末尾）
 * B. timestamp + deviceId → 查询困难（无法按设备聚合）
 * C. 盐值 + deviceId + timestamp → 写入均匀，但查询需要多盐值扫描
 * D. (反转)deviceId + timestamp → 写入分散，查询友好
 * 
 * 最终选择方案 D 的变体:
 * 格式: deviceId 的反转(8位) + deviceId(固定长度) + timestamp(反转13位)
 */
public class IoTRowKey {
    
    public static String generate(String deviceId, long timestamp) {
        // 1. deviceId 反转前 8 位（打散连续设备号）
        String reversedDevicePrefix = new StringBuilder(deviceId.substring(0, 8))
            .reverse().toString();
        
        // 2. 完整 deviceId（支持精确查询）
        String paddedDeviceId = String.format("%-20s", deviceId); // 左补空格，20位
        
        // 3. 反转时间戳（新区间在前）
        long reversedTimestamp = Long.MAX_VALUE - timestamp;
        String reversedTsStr = String.format("%013d", reversedTimestamp); // 13位
        
        return reversedDevicePrefix + paddedDeviceId + reversedTsStr;
    }
}
```

### 案例三：消息存储系统

**业务需求**：

- 存储即时通讯消息
- 每日消息量：50 亿条
- 查询：某会话最近 N 条消息、会话未读数统计

**RowKey 设计**：

```java
/**
 * 消息表 RowKey 设计
 * 格式: conversationId(哈希前缀2位) + 接收时间戳(反转13位) + messageId
 * 
 * 设计理由:
 * 1. conversationId 哈希前缀保证同一会话的消息分布在相邻 Region（支持范围扫描）
 * 2. 哈希前缀取前2位，共 256 个分区，写入会话级别均衡
 * 3. 反转时间戳让新消息排在前面
 * 
 * 注意: 这里不直接用 conversationId 开头
 * 因为如果用户 A 和 B 的会话集中，该会话的所有消息会集中在少数 Region
 */
public class MessageRowKey {
    
    private static final int HASH_PREFIX_LENGTH = 2;
    private static final int CONVERSATION_ID_LENGTH = 32;
    private static final int TIMESTAMP_LENGTH = 13;
    
    public static String generate(String conversationId, long receiveTime, String messageId) {
        // 哈希前缀（取 conversationId 的 MD5 前2位）
        String hashPrefix = String.format("%02x", 
            Math.abs(conversationId.hashCode()) % 256);
        
        // 固定长度 conversationId（便于解析和比较）
        String paddedConversationId = String.format("%-32s", conversationId);
        
        // 反转时间戳
        long reversedTimestamp = Long.MAX_VALUE - receiveTime;
        String reversedTsStr = String.format("%0" + TIMESTAMP_LENGTH + "d", reversedTimestamp);
        
        return hashPrefix + paddedConversationId + reversedTsStr + messageId;
    }
    
    /**
     * 扫描某会话的最新消息
     */
    public static Scan createConversationScan(String conversationId) {
        Scan scan = new Scan();
        
        // 由于有哈希前缀，需要扫描 256 个前缀
        for (int i = 0; i < 256; i++) {
            String prefix = String.format("%02x", i);
            String startRow = prefix + String.format("%-32s", conversationId);
            String stopRow = prefix + String.format("%-32s", conversationId + " ");
            
            scan.withStartRow(startRow.getBytes());
            scan.withStopRow(stopRow.getBytes());
        }
        
        scan.setFilter(new SingleColumnValueFilter(
            "msg".getBytes(), "conversationId".getBytes(),
            CompareOperator.EQUAL, conversationId.getBytes()
        ));
        
        return scan;
    }
}
```

## 五、热点问题排查与解决

### 5.1 热点检测

```java
// 通过 HBase Shell 检测热点
hbase> split 'TableName'  // 手动分裂 Region，分散热点

hbase> balance_switch true  // 开启负载均衡

hbase> org.apache.hadoop.hbase.mapreduce.RowCounter 'TableName' \\
    --rowkey.prefix=00  // 统计各前缀的数据量
```

### 5.2 预分区

创建表时预分配 Region，避免自动分裂带来的热点：

```java
// 创建表时指定 splitKeys
byte[][] splitKeys = {
    HConstants.EMPTY_BYTE_ARRAY.copyOf("1"),  // [1 - ...)
    HConstants.EMPTY_BYTE_ARRAY.copyOf("2"),  // [2 - ...)
    HConstants.EMPTY_BYTE_ARRAY.copyOf("3"),  // [3 - ...)
    // ... 根据数据分布预分配
};

admin.createTable(
    TableDescriptorBuilder.newBuilder(tableName)
        .setColumnFamilies(Collections.singletonList(
            ColumnFamilyDescriptorBuilder.of("cf")
        ))
        .build(),
    splitKeys
);
```

### 5.3 热点解决方案汇总

| 方案 | 原理 | 适用场景 | 代价 |
|---|---|---|---|
| 预分区 | 创建表时分配多个 Region | 写入模式固定 | 需要预知数据分布 |
| 加盐 | 随机前缀分散数据 | 写入密集型 | 无法范围查询 |
| 哈希 | 哈希前缀分散数据 | 点查为主 | 可能有哈希碰撞 |
| 反转 | 反转单调字段 | 手机号、ID等 | 失去语义可读性 |
| 负载均衡 | HMaster 自动调整 | 热点不严重 | 需要等待平衡周期 |

## 总结

RowKey 设计是 HBase 的核心技能，需要平衡四个维度：

1. **散列均匀**：让数据分布到不同的 Region，避免热点
2. **查询友好**：按最频繁的查询模式设计 RowKey 结构
3. **长度适中**：10-100 字节，平衡存储效率和语义表达
4. **避免单调**：永远不要用单调递增的字段作为 RowKey 前缀

实际工作中，建议先用小批量数据做测试，验证数据分布和查询性能是否符合预期，再全量上线。同时做好监控告警，一旦发现热点及时处理。
