# HBase 基础总览

你可能听说过 HBase——它是 Hadoop 生态中处理海量数据的利器。但当你真正想用它时，问题就来了：HBase 和 MySQL 有什么区别？我应该在什么时候用它？怎么设计 RowKey 才能避免热点？

别急，这篇文章就是你的 HBase 入门指南。我们会从零开始，理解 HBase 的核心概念，避开常见的认知陷阱。

## 为什么需要 HBase？

先问你一个问题：如果单表数据超过 10 亿条，你会怎么存储？

MySQL？没问题，但它需要分库分表，需要处理跨库查询，需要维护一套复杂的路由逻辑。MongoDB？也可以，但水平扩展能力有限。

HBase 的出现，就是为了解决这个场景：**在普通服务器上，存储 PB 级别的数据，并且支持高并发的随机读写。**

```
| 存储方案 | 数据规模 | 扩展方式 | 随机读延迟 | 随机写吞吐 |
| --- | --- | --- | --- | --- |
| MySQL | GB ~ TB | 垂直扩展 + 分库分表 | 微秒级 | 万级/秒 |
| MongoDB | TB 级别 | 水平扩展 | 毫秒级 | 十万级/秒 |
| HBase | PB 级别 | 水平扩展 | 毫秒级 | 百万级/秒 |
```

:::tip 关键指标
HBase 的随机写吞吐可以达到每秒百万级别，这是其他方案难以企及的。秘诀在于：它利用了 HDFS 的顺序写入能力，用内存换性能。
:::

## HBase 在大数据生态中的位置

HBase 不是孤立存在的，它是 Hadoop 生态的核心组件之一。

与 HDFS 的关系很微妙：HDFS 适合顺序写入、一次写入多次读取的场景（比如 MapReduce 的批处理）；HBase 则在其基础上实现了随机读写。

## 核心概念速览

### Table 和 Region

HBase 的数据组织成 **Table**，每个 Table 由多个 **Region** 组成。

```
Table: UserTable
┌────────────┬────────────┬────────────┐
│  Region 1  │  Region 2  │  Region 3  │
│  [A-F]      │  [G-M]     │  [N-Z]      │
│  RS-1       │  RS-2       │  RS-3       │
└────────────┴────────────┴────────────┘
```

Region 是负载均衡和数据分片的基本单元。当一个 Region 过大时（默认 10GB），会自动分裂成两个。

### RowKey、Column Family、Timestamp

HBase 的数据模型由三个维度组成：

| 概念 | 说明 | 类比 |
| --- | --- | --- |
| RowKey | 行的唯一标识，按字典序排序 | MySQL 的主键 |
| Column Family | 列的分组，同一组的列存储在一起 | 表的"分类标签" |
| Timestamp | 版本号，支持多版本 | 数据的"时间旅行" |

```
RowKey         info:name     info:age     order:id    Timestamp
────────────────────────────────────────────────────────────────
user_001       "张三"         28           "O-1001"    t1
user_002       "李四"         32           "O-1002"    t1
```

:::warning 重要区别
HBase 是稀疏存储——同一行中可以只有部分列存在，不会浪费存储空间。这与 MySQL 的每一行必须有相同列的设计完全不同。
:::

## HBase 的写入流程

理解 HBase 的写入流程，是理解整个系统的关键。

```
客户端写入
    │
    ▼
① 写入 WAL（Write-Ahead Log）
   - 先落盘，保证持久化
   - 类似 MySQL 的 Redo Log
    │
    ▼
② 写入 MemStore
   - 内存缓冲区，按 RowKey 排序
   - 写内存，极快
    │
    ▼
③ 异步刷写到 HDFS
   - MemStore 达到阈值（128MB）时触发
   - 生成 HFile（有序的磁盘文件）
```

写入之所以快，有三个原因：顺序写 WAL、内存写 MemStore、异步刷盘。

## HBase 的读取流程

读取比写入复杂，因为数据可能分散在多个地方：

```
客户端读取
    │
    ▼
① 先查 BlockCache（读缓存）
   - LRU 淘汰策略
   - 命中则直接返回
    │
    ▼
② 未命中？查 MemStore
   - 最近写入的数据还在内存
    │
    ▼
③ 还未命中？扫描 HFile
   - 利用布隆过滤器快速定位
   - 可能需要合并多个 HFile
```

读取延迟取决于缓存命中率和 HFile 数量，这正是 HBase 调优的核心。

## RowKey 设计：最重要的事

HBase 只有 RowKey 一个索引。查询只能基于 RowKey 进行精确查找或范围扫描。

RowKey 设计有三个原则：

**1. 散列性原则**

避免热点，数据要均匀分布：

```
# 错误设计（热点）
RowKey = 时间戳  # 所有数据挤在一个 Region

# 正确设计（散列）
RowKey = MD5(用户ID) + 用户ID + 时间戳
```

**2. 唯一性原则**

RowKey 必须在表中唯一，组合字段时要想清楚：

```
# 用户订单场景
RowKey = userId + orderId  # 查询某用户的所有订单

# 订单详情场景
RowKey = orderId            # 直接按订单ID查询
```

**3. 长度原则**

RowKey 会被存储无数次，越短越好：

```
# 过长的 RowKey
"user_13800138000_order_20240101120000000"

# 精简设计
MD5(userId+orderId) 前8位 + userId + 时间戳
```

## HBase vs 其他方案

很多人纠结 HBase 和 MySQL 的选择。实际上，两者不是替代关系：

| 维度 | HBase | MySQL |
| --- | --- | --- |
| 数据规模 | PB 级 | GB ~ TB 级 |
| 事务支持 | 单行事务 | ACID 完整事务 |
| 查询能力 | 仅 RowKey 查询 | SQL 全查询 |
| 写入性能 | 极高 | 一般 |
| 读取延迟 | 毫秒级 | 微秒级 |
| 扩展方式 | 水平扩展 | 垂直扩展 |

:::tip 场景选择
- **用 MySQL**：数据量小、事务要求高、查询复杂
- **用 HBase**：数据量大、写入密集、按 Key 查询
- **两者结合**：MySQL 存热数据，HBase 存历史数据
:::

## 快速上手

### 环境准备

```bash
# 使用 Docker 快速启动 HBase 单机版
docker run -d --name hbase \
  -p 16010:16010 -p 16030:16030 \
  -p 2181:2181 -p 9090:9090 \
  -p 9095:9095 \
  harisekhon/hbase:2.5
```

### 创建表

```java
Connection connection = ConnectionFactory.createConnection(config);
Admin admin = connection.getAdmin();

// 定义表结构
TableDescriptor table = TableDescriptorBuilder
    .newBuilder(TableName.valueOf("user_table"))
    .setColumnFamily(
        ColumnFamilyDescriptorBuilder
            .of("info")
            .setBlocksize(65536)
            .build()
    )
    .build();

// 创建表
admin.createTable(table);
```

### 写入数据

```java
Table table = connection.getTable(TableName.valueOf("user_table"));

// Put 操作
Put put = new Put(Bytes.toBytes("user_001"));
put.addColumn(
    Bytes.toBytes("info"),      // Column Family
    Bytes.toBytes("name"),      // Column Qualifier
    Bytes.toBytes("张三")
);
table.put(put);
```

### 读取数据

```java
// Get 操作
Get get = new Get(Bytes.toBytes("user_001"));
get.addColumn(Bytes.toBytes("info"), Bytes.toBytes("name"));

Result result = table.get(get);
byte[] value = result.getValue(
    Bytes.toBytes("info"),
    Bytes.toBytes("name")
);
System.out.println(Bytes.toString(value)); // 张三
```
