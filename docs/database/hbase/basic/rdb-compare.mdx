# HBase vs 关系型数据库

很多人第一次接触 HBase 时，都会问一个问题：HBase 能替代 MySQL 吗？

答案是否定的。它们不是替代关系，而是互补关系。选择哪个，取决于你的业务场景。

这篇文章会深入对比 HBase 和关系型数据库的差异，帮助你在实际项目中做出正确的技术选型。

## 本质区别：设计哲学

### 关系型数据库的设计哲学

MySQL、PostgreSQL 这些关系型数据库诞生于 1980 年代，它们的设计目标是什么？

**事务处理（OLTP）**：每秒处理成千上万的小事务，每个事务涉及少数几行数据，要求强一致性和低延迟。

```
MySQL 的典型场景：
- 用户登录验证
- 订单创建和支付
- 库存扣减
- 账户转账

特点：事务短小、数据量适中、并发量大、一致性要求高
```

### HBase 的设计哲学

HBase 诞生于 2007 年，BigTable 的开源实现。它的设计目标是什么？

**海量数据存储**：存储 TB/PB 级别的数据，支持高吞吐的随机读写，但不追求毫秒级的单点延迟。

```
HBase 的典型场景：
- 用户行为日志存储
- 监控数据采集
- 历史订单查询
- 消息推送记录

特点：数据量大、写入密集、按 Key 查询、一致性要求高
```

## 数据模型对比

### 表结构设计的差异

**MySQL 的表结构**

```sql
CREATE TABLE users (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    username VARCHAR(50) NOT NULL,
    email VARCHAR(100),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_username (username),
    UNIQUE INDEX idx_email (email)
);
```

特点：
- 事前定义所有列
- 列类型必须声明
- 约束可以加在列或表级别
- 索引可以建在任何列上

**HBase 的表结构**

```java
TableDescriptor table = TableDescriptorBuilder
    .newBuilder(TableName.valueOf("users"))
    .setColumnFamily(
        ColumnFamilyDescriptorBuilder
            .newBuilder(Bytes.toBytes("info"))
            .setBlocksize(65536)
            .build()
    )
    .build();

admin.createTable(table);
```

特点：
- 只需要定义 Column Family
- Column Qualifier 动态创建
- 无类型约束（全部存为字节数组）
- 唯一索引：RowKey

### 数据存储方式的差异

```
MySQL（行式存储）：
┌──────────────────────────────────────────────────────────────┐
│ Row 1: id=1, username="张三", email="zhang@example.com"     │
│ Row 2: id=2, username="李四", email="li@example.com"       │
│ Row 3: id=3, username="王五", email=(NULL)                  │
└──────────────────────────────────────────────────────────────┘
每一行连续存储，包含所有列

HBase（列式存储）：
┌─────────────────────┐     ┌─────────────────────┐
│ CF: info            │     │ CF: extra            │
│ RowKey    username  │     │ RowKey    hobby     │
│ user_001  "张三"    │     │ user_001  "足球"    │
│ user_002  "李四"    │     │ user_003  "篮球"    │
│ user_003  "王五"    │     │                     │
└─────────────────────┘     └─────────────────────┘
不同列族分开存储，空列不占空间
```

### 多版本支持

**MySQL**

```sql
-- MySQL 通过额外表存储历史版本
CREATE TABLE users_history (
    id BIGINT,
    username VARCHAR(50),
    valid_from TIMESTAMP,
    valid_to TIMESTAMP,
    PRIMARY KEY (id, valid_from)
);

-- 查询某个时间点的数据
SELECT * FROM users_history
WHERE id = 1
  AND valid_from <= '2024-01-01'
  AND valid_to > '2024-01-01';
```

**HBase**

```java
// HBase 原生支持多版本
Put put = new Put(Bytes.toBytes("user_001"));
put.addColumn(
    Bytes.toBytes("info"),
    Bytes.toBytes("name"),
    1704067200000L,  // 指定时间戳
    Bytes.toBytes("张三")
);
table.put(put);

// 读取所有版本
Get get = new Get(Bytes.toBytes("user_001"));
get.addColumn(Bytes.toBytes("info"), Bytes.toBytes("name"));
get.setMaxVersions();  // 返回所有版本
```

:::tip 适用场景
- 需要频繁查询历史版本：选 HBase
- 只需要保留最新数据：选 MySQL
:::

## 查询能力对比

### MySQL 的 SQL 能力

MySQL 的查询能力非常强大：

```sql
-- 关联查询
SELECT u.username, o.order_id, o.amount
FROM users u
JOIN orders o ON u.id = o.user_id
WHERE u.city = '北京'
  AND o.created_at > '2024-01-01';

-- 聚合查询
SELECT
    DATE(created_at) as day,
    COUNT(*) as order_count,
    SUM(amount) as total_amount,
    AVG(amount) as avg_amount
FROM orders
GROUP BY DATE(created_at)
HAVING COUNT(*) > 100;

-- 子查询
SELECT * FROM products
WHERE price > (SELECT AVG(price) FROM products);

-- 窗口函数
SELECT
    username,
    ROW_NUMBER() OVER (PARTITION BY city ORDER BY created_at DESC) as rank
FROM users;
```

### HBase 的查询方式

HBase 没有 SQL，查询能力非常有限：

```java
// 方式1：基于 RowKey 的精确查询
Get get = new Get(Bytes.toBytes("user_001"));
Result result = table.get(get);

// 方式2：基于 RowKey 范围的扫描
Scan scan = new Scan();
scan.withStartRow(Bytes.toBytes("user_001"));
scan.withStopRow(Bytes.toBytes("user_010"));
ResultScanner scanner = table.getScanner(scan);

// 方式3：Filter（性能较差，不推荐大数据量使用）
Filter filter = new RowFilter(
    CompareOperator.EQUAL,
    new RegexStringComparator("user_00[1-5]")
);
scan.setFilter(filter);
```

:::warning HBase 查询限制
HBase 不支持：
- JOIN（需要业务层处理）
- GROUP BY / ORDER BY（需要 MapReduce/Spark）
- 子查询
- 聚合计算
- 任意条件的 WHERE 子句

如果你的业务需要复杂查询，HBase 可能不是最佳选择。
:::

## 事务能力对比

### MySQL 的 ACID 事务

```sql
START TRANSACTION;

UPDATE accounts
SET balance = balance - 1000
WHERE user_id = 1;

UPDATE accounts
SET balance = balance + 1000
WHERE user_id = 2;

-- 如果任何一步失败，全部回滚
COMMIT;
```

MySQL 支持：
- 完整的 ACID 事务
- 跨表、跨行事务
- 分布式事务（XA）
- 不同的隔离级别

### HBase 的事务限制

```java
// HBase 只支持单行事务
Put put = new Put(Bytes.toBytes("user_001"));
put.addColumn(Bytes.toBytes("info"), Bytes.toBytes("balance"),
    Bytes.toBytes("9000"));  // 10000 - 1000
table.put(put);

// 如果需要扣减后再增加另一行，必须分两次操作
Put put2 = new Put(Bytes.toBytes("user_002"));
put2.addColumn(Bytes.toBytes("info"), Bytes.toBytes("balance"),
    Bytes.toBytes("11000")); // 10000 + 1000
table.put(put2);

// 这两次操作之间，可能被其他事务干扰
// HBase 不保证原子性
```

:::tip 解决方案
如果需要跨行事务，可以：
1. 设计时避免跨行事务（将相关数据放在同一行）
2. 使用业务层的补偿机制
3. 使用支持分布式事务的组件（如 Apache Omid）
:::

## 性能对比

### 写入性能

```
写入测试（1000万条数据）：

MySQL InnoDB：
- 批量插入：约 10,000 行/秒
- 单条插入：约 1,000 行/秒
- 瓶颈：Redo Log 刷盘、B+ 树维护

HBase：
- 批量写入：约 1,000,000 行/秒
- 单条写入：约 50,000 行/秒
- 瓶颈：ZooKeeper 协调、WAL 刷盘

性能差距：约 50-100 倍
```

为什么 HBase 写入这么快？

```
HBase 写入路径：
① 写入内存（MemStore）→ 极快
② 写 WAL → 顺序写入，批量刷盘
③ 异步 flush → 不阻塞客户端

MySQL 写入路径：
① 解析 SQL
② 检查约束
③ 写入 Buffer Pool
④ 写 Redo Log（同步）
⑤ 写数据页（可能需要随机 I/O）
⑥ 事务提交（可能需要刷盘）
```

### 读取性能

```
读取测试（单行精确查询）：

MySQL：
- 缓存命中（InnoDB Buffer Pool）：0.1ms
- 磁盘读取：1-5ms
- 延迟：微秒级

HBase：
- BlockCache 命中：0.5ms
- MemStore 命中：0.2ms
- HFile 读取：5-20ms
- 延迟：毫秒级
```

:::tip 读取延迟的选择
- 对延迟敏感（微秒级）：选 MySQL
- 对吞吐敏感（毫秒级可接受）：选 HBase
:::

## 扩展能力对比

### MySQL 的扩展方式

```
垂直扩展：
- 升级 CPU、内存、磁盘
- 有物理上限

水平扩展（分库分表）：
┌─────────┐  ┌─────────┐  ┌─────────┐
│ MySQL 1 │  │ MySQL 2 │  │ MySQL 3 │
│ (数据1) │  │ (数据2) │  │ (数据3) │
└────┬────┘  └────┬────┘  └────┬────┘
     │            │            │
     └────────────┼────────────┘
                  ▼
         ┌───────────────┐
         │ Sharding Proxy│
         │ (ShardingSphere│
         │  /MyCat)      │
         └───────────────┘

挑战：
- 跨库查询困难
- 分片键选择有讲究
- 扩容需要重新迁移数据
- 分布式事务复杂
```

### HBase 的扩展方式

```
水平扩展：
- 增加 RegionServer 节点
- 自动负载均衡
- 数据自动分裂

┌─────────────┐  ┌─────────────┐  ┌─────────────┐
│ RegionSrv 1 │  │ RegionSrv 2 │  │ RegionSrv 3 │
│ [Region A]  │  │ [Region B]  │  │ [Region C]  │
│ [Region D]  │  │ [Region E]  │  │             │
└─────────────┘  └─────────────┘  └─────────────┘

优势：
- 无中心节点，线性扩展
- 自动负载均衡
- 扩容不影响在线业务
```

## 适用场景对比

| 场景 | 推荐 | 原因 |
| --- | --- | --- |
| 电商订单（千万级） | MySQL | 查询复杂、需要事务、延迟敏感 |
| 用户行为日志（百亿级） | HBase | 数据量大、写入密集、查询简单 |
| 商品详情 | MySQL | 需要全文搜索、复杂过滤 |
| 监控时序数据 | HBase | 按时间范围查询、数据量大 |
| 用户 session | Redis | 毫秒级延迟要求 |
| 历史订单查询（亿级） | HBase | 数据量大、只按订单ID查询 |
| 好友关系 | 图数据库 | 多跳查询、关系复杂 |

## 实际架构：冷热分离

最常见的实践是两者结合：

```
┌─────────────────────────────────────────────────────────────┐
│                      应用层                                  │
│                    ┌───────┐                                 │
│                    │ Gateway│                                 │
│                    └───┬───┘                                 │
└────────────────────────┼────────────────────────────────────┘
                         │
          ┌──────────────┴──────────────┐
          ▼                             ▼
┌─────────────────────┐      ┌─────────────────────┐
│       MySQL          │      │       HBase         │
│  ┌───────────────┐  │      │  ┌───────────────┐  │
│  │   热数据       │  │      │  │   冷数据       │  │
│  │  (近3个月)     │  │      │  │  (3个月前)     │  │
│  │  联机事务处理  │  │      │  │  历史查询分析  │  │
│  └───────────────┘  │      │  └───────────────┘  │
│  容量：TB 级        │      │  容量：PB 级       │
│  延迟：微秒级       │      │  延迟：毫秒级      │
└─────────────────────┘      └─────────────────────┘
```

数据流转：
1. 新订单写入 MySQL（支持实时查询和事务）
2. 历史数据定期归档到 HBase（使用 MapReduce 或 Spark）
3. 查询时根据时间范围路由到不同存储

## 面试追问方向

- HBase 和 MySQL 的本质区别是什么？各自的设计目标是什么？
- HBase 为什么没有 SQL？如何处理复杂查询场景？
- HBase 支持事务吗？跨行事务怎么处理？
- 写入性能上 HBase 和 MySQL 差距有多大？为什么？
- 什么场景下需要用 HBase 替代 MySQL？
- 冷热分离架构中，MySQL 和 HBase 的职责如何划分？
- 如果一个表数据量超过 10 亿条，你会选择 MySQL 分库分表还是 HBase？

理解这些差异，你就能在实际项目中做出正确的技术选型。记住：没有最好的数据库，只有最适合业务场景的数据库。
