# 预分区与热点问题

凌晨 2 点，你被告警叫醒：HBase 集群的写入延迟突然飙升。监控显示，一台 RegionServer 的 CPU 使用率打满，而其他 RegionServer 却几乎空闲。

你开始排查：数据量不大，表也只有一个 Region，所有写入都集中在这台 RegionServer 上。这就是 HBase 最常见的问题——**热点问题**。

热点问题几乎每个 HBase 初学者都会遇到。它的根源在于 RowKey 设计和预分区缺失。理解这些问题，是运维 HBase 的必备技能。

## 什么是热点问题？

### 热点的本质

热点（Hotspot）是指数据集中写入到某个或某几个特定的 Region，导致这些 Region 所在的 RegionServer 负载过高，而其他 RegionServer 空闲。

```
热点问题示意：
┌─────────────────────────────────────────────────────────────┐
│                                                              │
│  无热点（正确设计）：                                         │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐        │
│  │  Region-A   │  │  Region-B   │  │  Region-C   │        │
│  │  RS-1       │  │  RS-2       │  │  RS-3       │        │
│  │  负载: 30%  │  │  负载: 35%  │  │  负载: 32%  │        │
│  └─────────────┘  └─────────────┘  └─────────────┘        │
│                                                              │
│  有热点（错误设计）：                                         │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐        │
│  │  Region-A   │  │  Region-B   │  │  Region-C   │        │
│  │  RS-1       │  │  RS-1       │  │  RS-2       │        │
│  │  负载: 95%  │  │  负载: 90%  │  │  负载: 5%   │        │
│  └─────────────┘  └─────────────┘  └─────────────┘        │
│                                                              │
│  写入全部集中在 RS-1，其他节点空闲                           │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 热点的症状

| 症状 | 说明 |
| --- | --- |
| 单台 RegionServer CPU 打满 | 其他 RegionServer 空闲 |
| 写入延迟飙升 | 热点 Region 处理不过来 |
| 读取延迟不稳定 | 部分请求快，部分请求慢 |
| HBase 日志异常 | 特定 Region 的日志量远超其他 |

## 热点的成因

### 1. RowKey 设计不当

**递增 RowKey 导致热点**：

```java
// 错误设计：使用时间戳作为 RowKey
RowKey = timestamp
// 例如：1704067200001, 1704067200002, 1704067200003, ...

// 问题：所有新数据都写入同一个 Region（最大的 RowKey 所在的 Region）
// 因为 HBase 按字典序排序，新数据永远在最后
```

**手机号作为 RowKey 导致热点**：

```java
// 错误设计：手机号作为 RowKey
RowKey = phoneNumber
// 例如：13800138000, 13800138001, 13800138002, ...

// 问题：如果用户手机号按号段分配，相同号段的用户数据会集中
```

### 2. 缺少预分区

```
无预分区的问题：
┌─────────────────────────────────────────────────────────────┐
│                                                              │
│  创建表时只有一个 Region：                                   │
│  create 'users', 'info'                                    │
│                                                              │
│  ┌─────────────────────────────────────────────────────┐   │
│  │              users [NULL - ∞)                        │   │
│  │                   RegionServer-1                      │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                              │
│  写入 10TB 数据后，仍然只有一个 Region：                    │
│  - 所有数据都在 RegionServer-1 上                           │
│  - 其他 RegionServer 完全空闲                              │
│  - RegionServer-1 成为瓶颈                                  │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 3. 数据倾斜

```java
// 场景：存储用户订单
// RowKey = userId + orderId

// 假设有 1000 个用户：
// - 普通用户：平均 10 个订单
// - 大V用户：10 万个订单

// 问题：
// - 大V 的订单数据集中在少数几个 RowKey
// - 这几个 RowKey 对应的 Region 成为热点
```

## 预分区：解决热点的基础

### 预分区的概念

预分区是在创建表时指定多个分裂点，将数据分散到多个 Region。

```
预分区前后对比：
┌─────────────────────────────────────────────────────────────┐
│                                                              │
│  无预分区：                                                   │
│  create 'users', 'info'                                    │
│  ┌─────────────────────────────────────────────────────┐   │
│  │              users [NULL - ∞)                          │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                              │
│  有预分区：                                                   │
│  create 'users', 'info', SPLITS => ['g', 'm', 't', 'z']  │
│  ┌──────────┬──────────┬──────────┬──────────┬─────────┐   │
│  │  [NULL-g)│  [g-m)   │  [m-t)   │  [t-z)   │  [z-∞)  │   │
│  │  Region-1 │  Region-2│  Region-3│  Region-4│ Region-5│   │
│  └──────────┴──────────┴──────────┴──────────┴─────────┘   │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 如何设计分裂点

**策略1：哈希预分区**

```java
// 使用 MD5 哈希的前缀作为分裂点
// 优点：数据均匀分布
// 缺点：相邻 RowKey 可能不在同一 Region

// 分裂点设计：16 进制前缀
byte[][] splitKeys = {
    Bytes.toBytes("0"),
    Bytes.toBytes("1"),
    Bytes.toBytes("2"),
    Bytes.toBytes("3"),
    Bytes.toBytes("4"),
    Bytes.toBytes("5"),
    Bytes.toBytes("6"),
    Bytes.toBytes("7"),
    Bytes.toBytes("8"),
    Bytes.toBytes("9"),
    Bytes.toBytes("a"),
    Bytes.toBytes("b"),
    Bytes.toBytes("c"),
    Bytes.toBytes("d"),
    Bytes.toBytes("e"),
    Bytes.toBytes("f")
};

admin.createTable(tableDescriptor, splitKeys);
```

**策略2：盐值预分区**

```java
// RowKey = (hash(userId) % nBuckets) + separator + userId
// 例如：nBuckets = 10，RowKey = "5_user_001"

// 分裂点：0_, 1_, 2_, ..., 9_

byte[][] splitKeys = new byte[10][];
for (int i = 0; i < 10; i++) {
    splitKeys[i] = Bytes.toBytes(i + "_");
}

// 优点：实现简单
// 缺点：查询时需要遍历所有 Bucket
```

**策略3：业务维度预分区**

```java
// 根据业务特点设计分裂点
// 例如：按地区预分区

byte[][] splitKeys = {
    Bytes.toBytes("华北_"),      // 华北地区
    Bytes.toBytes("华东_"),      // 华东地区
    Bytes.toBytes("华南_"),      // 华南地区
    Bytes.toBytes("西南_"),      // 西南地区
    Bytes.toBytes("西北_"),      // 西北地区
    Bytes.toBytes("东北_")       // 东北地区
};

admin.createTable(tableDescriptor, splitKeys);
```

### 预分区数量确定

```
预分区数量计算：
┌─────────────────────────────────────────────────────────────┐
│                                                              │
│  公式：Region 数 = 预期数据量 / 单 Region 大小              │
│                                                              │
│  示例：                                                      │
│  - 预期数据量：100TB                                         │
│  - 单 Region 大小：10GB                                      │
│  - Region 数 = 100TB / 10GB = 10240 个 Region              │
│                                                              │
│  考虑因素：                                                   │
│  - RegionServer 数量（建议 Region 数 = 节点数 * 20~200）    │
│  - 单台 RegionServer 的处理能力                              │
│  - 未来扩展预留                                              │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

## RowKey 优化：根本解决热点

### 原则1：散列性

```java
// 原始设计
RowKey = userId  // 假设 userId 递增

// 问题：写入热点

// 优化方案1：加盐
RowKey = MD5(userId).substring(0, 4) + userId

// 优化方案2：反转
RowKey = reverse(userId)

// 优化方案3：哈希
RowKey = hash(userId) + userId
```

### 原则2：唯一性

```java
// 确保 RowKey 唯一
// 可以使用复合 RowKey
RowKey = userId + orderId + timestamp
```

### 原则3：长度控制

```java
// RowKey 过长会浪费存储空间
// 建议：RowKey 长度控制在 16~100 字节

// 优化：使用短编码
// 原始：user_13800138000_order_20240101120000000 (40 字节)
// 优化：MD5(userId).substring(0,8) + userId + timestamp (约 25 字节)
```

### 常见的 RowKey 设计模式

**模式1：用户+订单场景**

```java
// 查询某用户的所有订单
RowKey = userId + orderId
// 分裂点：按 userId 哈希

// 查询某用户的最新 N 条订单
// 可以在 RowKey 中反转时间戳
RowKey = userId + reverse(timestamp)
```

**模式2：时间序列场景**

```java
// 监控数据存储
// RowKey = metricName + reverse(timestamp)
// 目的：让新数据按 metricName 分散，旧数据按时间顺序

// 示例：
// metric:cpu host:192.168.1.1
// timestamp:1704067200000
// RowKey = "cpu:192.168.1.1_" + (Long.MAX_VALUE - 1704067200000)
```

**模式3：多租户场景**

```java
// RowKey = tenantId + entityId
// 目的：不同租户的数据分散到不同 Region
```

## 热点检测与监控

### 检测热点

```bash
# 通过 HBase Shell 查看 Region 分布
status 'detailed'

# 查看特定表的 Region
list_regions 'my_table'

# 查看 RegionServer 负载
hbase shell
balance_switch true
```

### 监控热点

```java
// 通过 Metrics 监控
// 关键指标：
// - regionCount：每个 RegionServer 的 Region 数
// - readRequestCount：读请求数
// - writeRequestCount：写请求数

// 如果某个 RegionServer 的请求数远超平均，说明有热点
```

### 自动热点处理

```java
// HBase 2.0+ 支持自动分裂
// 当 Region 大小超过阈值时自动分裂

// 但自动分裂对于写入热点帮助有限
// 因为分裂后的 Region 仍然可能写入热点

// 更根本的解决方案：
// 1. 预分区设计合理
// 2. RowKey 设计有散列性
// 3. 手动 split 热点 Region
```

## 热点案例分析

### 案例1：时间戳 RowKey

```java
// 问题：使用时间戳作为 RowKey
// 场景：日志采集系统
RowKey = timestamp  // 1704067200001, 1704067200002, ...

// 结果：所有写入集中在最大的 Region

// 解决：加盐或使用时间戳反转
RowKey = MD5(timestamp).substring(0, 4) + timestamp
RowKey = timestamp + randomSuffix
RowKey = reverse(timestamp) + originalId
```

### 案例2：大V 用户数据倾斜

```java
// 问题：大V 用户的数据量远超普通用户
// 场景：社交媒体的用户行为数据
RowKey = userId + behaviorType + timestamp

// 结果：大V 的 Region 很大，导致频繁分裂

// 解决：
// 1. 将大V 数据拆分到多个 RowKey
RowKey = userId + bucketId + timestamp
// 2. 或者使用单独的 Column Family 存储大V 数据
// 3. 或者将大V 数据单独建表
```

### 案例3：固定前缀导致热点

```java
// 问题：所有 RowKey 以相同前缀开头
// 场景：设备数据采集
RowKey = "device_" + deviceId + timestamp

// 结果：所有数据集中在某些 Region

// 解决：使用哈希作为前缀
RowKey = hash(deviceId) + deviceId + timestamp
RowKey = (deviceId % 10) + "_" + deviceId + timestamp
```
