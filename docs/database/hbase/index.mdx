# HBase 详解：海量数据的列式存储

凌晨 3 点，你被监控告警叫醒。业务数据显示：最近一周内，用户的历史订单查询接口响应时间从 200ms 飙升到 5 秒。

你排查了一圈：数据库 CPU 正常、内存正常、连接数正常。那问题出在哪？

仔细一看订单表——订单量已经突破 5 亿条，而历史订单查询总是 `SELECT * FROM orders WHERE user_id = ? AND create_time BETWEEN ? AND ?`。

5 亿级数据，加上时间范围查询，MySQL 的 B+ 树索引已经力不从心了。

这不是 MySQL 的问题，而是场景的问题。当数据量达到千万级甚至亿级时，你需要另一种存储方案——HBase。

## HBase 是什么？

Apache HBase 是构建在 Hadoop 分布式文件系统（HDFS）之上的 NoSQL 列式数据库，专为海量结构化数据存储而设计。

```
┌─────────────────────────────────────────────────────────────┐
│                        应用层                               │
│         ┌──────────────┐  ┌──────────────┐               │
│         │  历史订单查询  │  │  监控数据写入  │               │
│         └──────┬───────┘  └──────┬───────┘               │
│                │                │                         │
│                ▼                ▼                         │
│         ┌────────────────────────────────┐                │
│         │         HBase API              │                │
│         └────────────────────────────────┘                │
└──────────────────────┬─────────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────────┐
│                      HBase 层                              │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐         │
│  │ HMaster    │  │ RegionServer│  │ RegionServer│         │
│  │ (元数据)    │  │  (数据存储)  │  │  (数据存储)  │         │
│  └─────────────┘  └─────────────┘  └─────────────┘         │
└──────────────────────┬─────────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────────┐
│                      HDFS 层                               │
│     ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐    │
│     │ DataNode│  │ DataNode│  │ DataNode│  │ DataNode│    │
│     └─────────┘  └─────────┘  └─────────┘  └─────────┘    │
└─────────────────────────────────────────────────────────────┘
```

HBase 的核心特点：

- **海量存储**：单表支持 TB/PB 级数据
- **高并发**：支持每秒数十万次读写
- **强一致性**：CAP 中选择 CP，保证数据一致性
- **自动分区**：数据自动分裂，无需人工干预
- **稀疏存储**：列可以为空，不占用存储空间

## HBase vs 关系型数据库

很多人初次接触 HBase 时，会把它和 MySQL 做对比。但实际上，两者是互补关系，而非替代关系。

| 特性 | HBase | MySQL |
| --- | --- | --- |
| 数据模型 | 列式 NoSQL | 行式关系型 |
| 数据规模 | TB/PB 级 | GB/TB 级 |
| 事务支持 | 有限（单行事务） | ACID 完整事务 |
| 查询能力 | Key-Value 为主 | SQL 强大查询 |
| 索引 | 仅 RowKey 索引 | B+ 树、全文、组合索引 |
| 写入性能 | 极高 | 一般 |
| 读取延迟 | 毫秒级 | 微秒级 |
| 扩展方式 | 水平扩展 | 垂直扩展 + 分库分表 |

:::tip 核心区别
MySQL 是为**OLTP（事务处理）**设计的，适合低延迟、高可靠的读写；HBase 是为**海量数据存储**设计的，适合高吞吐、低成本的写入和范围查询。

两者的关系是：MySQL 存热数据，HBase 存冷数据。
:::

## HBase 的数据模型

HBase 的数据模型与传统关系型数据库有本质区别，理解它需要摆脱"表"和"行"的思维定式。

### 四维坐标定位数据

在 HBase 中，定位一个单元格需要四个坐标：

```
(Table, RowKey, Column Family:Column Qualifier, Timestamp)
```

| 概念 | 说明 | 示例 |
| --- | --- | --- |
| RowKey | 行键，唯一标识一行 | `user_12345` |
| Column Family | 列族，相关列的分组 | `info`、`order` |
| Column Qualifier | 列限定符，具体列 | `info:name`、`order:amount` |
| Timestamp | 版本号，时间戳 | `1700000000000` |

### 逻辑视图 vs 物理存储

HBase 的数据逻辑上看起来像这样：

```
RowKey         Column Family:Column
─────────────────────────────────────────────────────
user_001       info:name = "张三"
               info:age = 28
               info:city = "北京"
               order:202401 = "订单A"
               order:202402 = "订单B"

user_002       info:name = "李四"
               info:age = 32
```

但物理存储是按 Column Family 分开存储的：

```
// CF: info 的物理存储
RowKey         info:name   info:age   info:city   Timestamp
─────────────────────────────────────────────────────────────
user_001       "张三"       28         "北京"      t1
user_002       "李四"       32                      t2

// CF: order 的物理存储
RowKey         order:202401   order:202402   Timestamp
─────────────────────────────────────────────────────────────
user_001       "订单A"       "订单B"         t1
user_003       "订单C"                      t3
```

这种存储方式有什么好处？

- 不同 Column Family 可以设置不同的存储策略（压缩、缓存 TTL）
- 稀疏列不影响存储成本
- 可以只读取需要的 Column Family，减少 I/O

### 版本机制

HBase 支持多版本存储，通过 Timestamp 区分不同版本的数据：

```
RowKey         Column            Value              Timestamp
──────────────────────────────────────────────────────────────
user_001       info:status       "active"           1700000001
                                 "inactive"         1700000000
```

默认保留 3 个版本，可以通过配置修改。版本机制使得 HBase 能够存储数据的历史变化。

## RowKey 设计：HBase 的核心艺术

如果说关系型数据库的核心是表设计，那 HBase 的核心就是 **RowKey 设计**。

RowKey 有以下特点：

- 唯一标识一行数据
- 按字典序排序存储
- 是 HBase 唯一的主键索引
- 查询只能基于 RowKey 或 RowKey 范围

### RowKey 设计原则

**1. 散列性原则：数据分布要均匀**

如果 RowKey 是递增的（如时间戳），大量数据会集中在同一个 Region，导致数据倾斜和热点问题。

```
# 错误设计：热点问题
RowKey = timestamp  # 所有数据挤在一个 Region

# 正确设计：散列分布
RowKey = MD5(timestamp) + timestamp
RowKey = Salt(Hash(user_id)) + user_id
```

**2. 唯一性原则：每行数据必须有唯一的 RowKey**

RowKey 通常由多个业务字段组合生成：

```
# 用户订单场景
RowKey = userId + orderId          # 用户维度
RowKey = orderId + userId          # 订单维度
RowKey = orderId + timestamp       # 时间维度
```

**3. 长度原则：RowKey 越短越好**

RowKey 会被存储无数次，尽可能短可以节省存储空间：

```
# 过长：浪费空间
RowKey = "order_1234567890123456_20240101120000"

# 精简：10进制转16进制
RowKey = 1234567890123456 + 0x01F41B30
```

### 常见 RowKey 设计模式

**1. 用户+订单场景**

```
RowKey = userId + orderId  # 倒序存储，最新订单在前
```

优势：查询某个用户的所有订单非常高效。

**2. 监控数据场景**

```
RowKey = metric + timestamp  # 指标名 + 时间戳
```

优势：按指标名和时间范围查询非常高效。

**3. 关系图谱场景**

```
RowKey = fromId + toId  # 关系起点 + 关系终点
```

优势：查询某个实体的所有关联关系非常高效。

## HBase 架构解析

### Region 和 RegionServer

HBase 的数据存储单元是 **Region**，类似于 MySQL 的分区表，但更加自动化。

```
┌──────────────────────────────────────────────────────┐
│                   HBase Table                        │
│                                                      │
│  Region 1          Region 2         Region 3        │
│  [a-h]             [i-p]            [q-z]            │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐      │
│  │ RowKey:  │    │ RowKey:  │    │ RowKey:  │      │
│  │ a, b, c  │    │ i, j, k  │    │ q, r, s  │      │
│  └──────────┘    └──────────┘    └──────────┘      │
│       │                │                │           │
│       ▼                ▼                ▼           │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐       │
│  │Region    │    │Region    │    │Region    │       │
│  │Server 1  │    │Server 2  │    │Server 3  │       │
│  └──────────┘    └──────────┘    └──────────┘       │
└──────────────────────────────────────────────────────┘
```

当 Region 的大小超过阈值（默认 10GB）时，会自动分裂成两个 Region，分布到不同的 RegionServer 上。

### HMaster 的作用

HMaster 是 HBase 的协调者，负责：

- 管理 Region 的分配和负载均衡
- 监控 RegionServer 的健康状态
- 处理 Schema 变更（创建表、删除列族等）
- 故障转移时重新分配 Region

:::warning HMaster 不是瓶颈
HMaster 不存储实际数据，只负责元数据管理。即使 HMaster 宕机，RegionServer 仍然可以正常读写数据，只是不能进行 Schema 变更和负载均衡。
:::

### WAL 和 MemStore

HBase 的写入流程：

```
写入请求
    │
    ▼
① 写入 WAL（Write-Ahead Log）
   - 先写日志，防止数据丢失
   - 类似 MySQL 的 Redo Log
    │
    ▼
② 写入 MemStore
   - 写入内存缓冲区
   - 按 Column Family 排序
    │
    ▼
③ 异步 flush 到 HDFS
   - MemStore 写满后，生成 HFile
   - 多个 HFile 合并优化
```

### HFile 和 BlockCache

HFile 是 HBase 的持久化存储格式，底层是 HDFS。

```
HFile 结构：
┌────────────────────────────────────────────────┐
│  Block 1 (64KB) - Keys/Values                  │
│  Block 2 (64KB) - Keys/Values                  │
│  Block 3 (64KB) - Keys/Values                  │
│  ...                                           │
├────────────────────────────────────────────────┤
│  BlockIndex - Block 位置索引                    │
├────────────────────────────────────────────────┤
│  Meta Index - Column Family 级别的索引          │
├────────────────────────────────────────────────┤
│  File Info - 起始/结束 Key、时间戳等            │
├────────────────────────────────────────────────┤
│  Trailer - 签名、偏移量等                       │
└────────────────────────────────────────────────┘
```

BlockCache 是读缓存热点数据的内存区域，LRU 淘汰策略保证热点数据常驻内存。

## HBase 的适用场景

HBase 不是万能的，理解它的适用场景才能正确使用。

### 适合 HBase 的场景

**1. 海量数据存储**

数据量达到 TB/PB 级别，单表数据超过 10 亿行时，HBase 是自然的选择。

**2. 写多读少的场景**

HBase 的写入性能极高，适合日志、监控、传感器数据等写入密集型场景。

**3. 跨度查询场景**

需要按 RowKey 范围查询大量数据，如时间范围查询、用户历史数据查询。

**4. 半结构化数据**

数据Schema 频繁变化，或者数据本身比较稀疏。

### 不适合 HBase 的场景

**1. 强事务需求**

HBase 只支持单行事务，跨行跨表的事务无法支持。

**2. 复杂查询需求**

HBase 没有 SQL，不支持 JOIN、子查询、聚合等复杂操作。需要通过 MapReduce 或 Spark 进行分析。

**3. 强一致性要求的事务性查询**

HBase 的读写延迟虽然低，但比 MySQL 的随机读写还是要高一些。

## HBase 在大数据生态中的位置

HBase 是 Hadoop 生态的核心组件之一，与其他组件形成紧密的配合关系：

```
│  │  HDFS        │◄───│  HBase       │◄───│  Phoenix  │  │
│  │  (存储层)     │    │  (NoSQL)     │    │  (SQL层)  │  │

└─────────────────────────────────────────────────────────┘
```

### HBase + Phoenix：SQL 访问 HBase

原生 HBase 只支持 Key-Value 访问，不支持 SQL。Apache Phoenix 提供了 SQL 层，可以像操作 MySQL 一样操作 HBase：

```sql
-- Phoenix 可以执行标准 SQL，底层操作 HBase
SELECT * FROM orders
WHERE user_id = '12345'
  AND create_time > '2024-01-01';

-- 支持二级索引
CREATE INDEX idx_user_time ON orders(user_id, create_time);
```

### HBase + Spark：海量数据分析

Spark 可以直接读写 HBase，进行大规模数据分析：

```scala
// Spark 读取 HBase
val hbaseDF = spark.read
  .format("hbase")
  .option("hbase.table", "orders")
  .load()

hbaseDF.createOrReplaceTempView("orders")
spark.sql("SELECT user_id, COUNT(*) FROM orders GROUP BY user_id")
```

## HBase vs Cassandra：两个极端

提到 HBase，不得不提 Cassandra。两者都是列式 NoSQL，但设计哲学完全不同：

| 特性 | HBase | Cassandra |
| --- | --- | --- |
| CAP 取舍 | CP（强一致性） | AP（高可用性） |
| 数据模型 | 单表操作强项 | 多表关联更强 |
| 写入性能 | 高 | 极高 |
| 查询灵活性 | 仅 RowKey | CQL 支持更多查询 |
| 生态集成 | Hadoop 生态 | 云原生友好 |
| 一致性模型 | 行级一致性 | 可配置一致性级别 |

```
CAP 权衡示意：

        Consistency
            ▲
            │     HBase
            │        │
            │        │
    ────────┼────────┼──────────► Availability
            │        │
            │        │   Cassandra
            │        │
```

:::tip 选择建议
- 需要强一致性、与 Hadoop 集成：选 HBase
- 需要高可用、多数据中心部署：选 Cassandra
:::
