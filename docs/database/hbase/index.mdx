# HBase

2006 年，Google 发表了一篇论文叫《Bigtable: A Distributed Storage System for Structured Data》。

描述了一个新型存储系统的设计：分布式的、稀疏的、持久化的多维有序映射表。

两年后，Apache HBase 诞生了——它是 Bigtable 的开源实现，Facebook、Netflix、Twitter 等巨头都在用。

## HBase vs MySQL：什么时候选 HBase？

| 维度 | MySQL | HBase |
|------|-------|-------|
| 数据规模 | TB 级 | PB 级 |
| 写入模式 | 随机读写 | 顺序追加 |
| 查询模式 | 复杂查询（JOIN） | KV 随机读取 |
| 事务 | ACID 强事务 | 单行事务 |
| 索引 | 多索引 | 仅 RowKey 索引 |
| 一致性 | 强一致 | 最终一致 |

**选 MySQL**：数据量小（< 10TB）、需要复杂查询、事务要求高

**选 HBase**：数据量大（> 10TB）、写入并发高、只需要按 RowKey 查询

## 核心公式

```
HBase 数据模型 = Table + RowKey + Column Family + Column Qualifier + Value + Timestamp
RowKey = 主键（按字典序排序，用于唯一定位一行）
Column Family = 列族（在创建表时定义，影响存储和压缩策略）
Timestamp = 版本号（同一 Cell 可存储多个版本，默认返回最新）

HBase 架构 = ZooKeeper（协调） + HMaster（管理） + RegionServer（存储）
Region = 表的数据分片（自动分裂和迁移）
MemStore = 写缓存（WAL + 内存排序后刷盘）
HFile = 磁盘存储文件（SSTable 结构）

读性能 = BlockCache（热点数据缓存）+ BloomFilter（快速定位）+ KeySet（过滤）
```
