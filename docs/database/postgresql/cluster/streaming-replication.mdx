# PostgreSQL 流复制原理

流复制（Streaming Replication，SR）是 PostgreSQL 实现主从同步的核心机制。通过流复制，从节点可以实时同步主节点的数据变更，实现数据冗余和读写分离。

---

## 流复制架构

```
┌─────────────┐                              ┌─────────────┐
│   Primary  │  WAL Records  ──────────────►│   Standby   │
│   (Master) │  (流式传输)                   │   (Slave)  │
│             │                              │             │
│  ┌───────┐ │                              │  ┌───────┐ │
│  │  WAL  │ │                              │  │  WAL  │ │
│  │ Sender│ │                              │  │Receiver│ │
│  └───────┘ │                              │  └───────┘ │
│             │                              │             │
│  ┌───────┐ │                              │             │
│  │本地磁盘│ │                              │             │
│  └───────┘ │                              │             │
└─────────────┘                              └─────────────┘
```

## 配置步骤

### 1. 主节点配置

```ini
# postgresql.conf

# 启用 WAL
wal_level = replica  # 或 hot_standby

# 允许复制连接
max_wal_senders = 10

# WAL 保留
wal_keep_size = 1GB

# 启用复制槽（推荐）
max_replication_slots = 10

# 归档（可选）
archive_mode = on
archive_command = 'cp %p /archive/%f'
```

### 2. 创建复制用户

```sql
-- 创建复制用户
CREATE USER replicator WITH REPLICATION ENCRYPTED PASSWORD 'your_password';

-- 在 pg_hba.conf 中添加权限
# host  replication  replicator  192.168.1.0/24  md5
```

### 3. 创建基础备份

```bash
# 在从节点执行
pg_basebackup -h primary_host -U replicator -D /var/lib/postgresql/data -P -Xs -R
```

参数说明：
- `-Xs`：复制 WAL 文件
- `-P`：显示进度
- `-R`：创建 standby.signal 文件

### 4. 从节点配置

```ini
# postgresql.conf

# 启用热备
hot_standby = on

# 最大复制延迟（可选）
max_standby_streaming_delay = 30s

# 复制冲突时的处理
hot_standby_feedback = on
```

## 复制原理

### WAL 流式传输

流复制通过 WAL 流式传输实现：

1. 主节点不断产生 WAL 记录
2. 从节点通过 walsender 进程发送 WAL 记录
3. 从节点通过 walreceiver 进程接收并应用 WAL 记录

```sql
-- 查看复制状态
SELECT * FROM pg_stat_replication;
```

### 复制模式

| 模式 | 说明 | 数据安全性 |
|---|---|---|
| 异步复制 | 主节点不等待从节点确认 | 可能丢失少量数据 |
| 同步复制 | 主节点等待从节点确认 | 不会丢失已提交数据 |

### 配置同步复制

```ini
# postgresql.conf
synchronous_commit = on
synchronous_standby_names = 'standby1,standby2'

# 或使用复制槽名称
# synchronous_standby_names = 'slot_name'
```

## 复制槽

### 什么是复制槽

复制槽（Replication Slot）确保主节点保留从节点尚未接收的 WAL 记录，防止 WAL 被过早清理。

```sql
-- 创建复制槽
SELECT * FROM pg_create_physical_replication_slot('slot_name');

-- 查看复制槽
SELECT * FROM pg_replication_slots;

-- 删除复制槽
SELECT pg_drop_replication_slot('slot_name');
```

### 复制槽配置

```ini
# postgresql.conf
max_replication_slots = 10  # 允许的复制槽数量
wal_keep_size = 0            # 配合复制槽使用
```

## 监控复制

### 查看复制状态

```sql
-- 查看所有从节点状态
SELECT
    client_addr,
    state,
    sent_lsn,
    write_lsn,
    flush_lsn,
    replay_lsn,
    (sent_lsn - replay_lsn) AS replication_lag_lsn,
    pg_size_pretty(sent_lsn - replay_lsn) AS replication_lag
FROM pg_stat_replication;
```

### 复制延迟

```sql
-- 查看复制延迟（时间维度）
SELECT
    client_addr,
    state,
    (NOW() - reply_time) AS replication_delay
FROM pg_stat_replication;
```

### 监控脚本

```sql
-- 创建监控视图
CREATE OR REPLACE VIEW replication_status AS
SELECT
    application_name,
    client_addr,
    state,
    pg_size_pretty(sent_lsn - replay_lsn) AS lag
FROM pg_stat_replication;

SELECT * FROM replication_status;
```

## 读写分离

### 应用层配置

```python
# Python: 使用连接池实现读写分离
import psycopg2
from psycopg2 import pools

# 主节点连接池（写）
master_pool = pools.ThreadedConnectionPool(
    5, 20,
    "host=master dbname=mydb user=admin"
)

# 从节点连接池（读）
replica_pool = pools.ThreadedConnectionPool(
    5, 20,
    "host=replica dbname=mydb user=admin"
)

def read_query(sql, params=None):
    conn = replica_pool.getconn()
    try:
        cur = conn.cursor()
        cur.execute(sql, params)
        return cur.fetchall()
    finally:
        replica_pool.putconn(conn)

def write_query(sql, params=None):
    conn = master_pool.getconn()
    try:
        cur = conn.cursor()
        cur.execute(sql, params)
        conn.commit()
    finally:
        master_pool.putconn(conn)
```

### pgpool-II 读写分离

```ini
# pgpool.conf
master_slave_mode = on
master_slave_sub_mode = 'stream'

backend_hostname0 = 'master'
backend_port0 = 5432
backend_weight0 = 0
backend_hostname1 = 'replica'
backend_port1 = 5432
backend_weight1 = 1

load_balance_mode = on
```

## 常见问题

### 问题一：复制延迟

```sql
-- 查看延迟原因
SELECT
    state,
    sent_lsn - replay_lsn AS lag,
    sent_lsn - write_lsn AS write_lag,
    write_lsn - flush_lsn AS flush_lag,
    flush_lsn - replay_lsn AS replay_lag
FROM pg_stat_replication;

-- 可能的原因：
-- 1. 从节点资源不足（CPU、磁盘 I/O）
-- 2. 网络带宽不足
-- 3. 大事务导致延迟
```

### 问题二：WAL 积压

```sql
-- 查看 WAL 生成速度
SELECT
    pg_wal_lsn_diff(pg_current_wal_lsn(), replay_lsn) AS pending_wal,
    pg_wal_lsn_diff(pg_current_wal_lsn(), '0/0'::pg_lsn) AS wal_generated
FROM pg_stat_replication;
```

### 问题三：复制断开

```bash
# 重启复制
pg_ctl restart -D /var/lib/postgresql/data

# 或者使用 pg_rewind 恢复
pg_rewind --target-pgdata=/var/lib/postgresql/data \
          --source-server='host=master port=5432 user=postgres'
```

## 总结

流复制是 PostgreSQL 高可用的基础：

1. **原理**：WAL 流式传输，从节点实时应用
2. **配置**：主节点开启复制，从节点配置连接
3. **复制槽**：确保 WAL 不会过早清理
4. **监控**：监控复制延迟和状态
5. **读写分离**：配合连接池实现

流复制配置相对简单，是 PostgreSQL 部署的必备技能。
