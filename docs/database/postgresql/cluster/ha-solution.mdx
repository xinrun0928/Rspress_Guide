# PostgreSQL 高可用方案

高可用（High Availability，HA）是生产环境数据库的核心需求。本节介绍常见的 PostgreSQL 高可用架构和方案。

---

## 高可用指标

### 可用性评估

| 指标 | 年停机时间 | 说明 |
|---|---|---|
| 99% | 3.65 天 | 基本可用 |
| 99.9% | 8.76 小时 | 较高可用 |
| 99.99% | 52.6 分钟 | 高可用 |
| 99.999% | 5.26 分钟 | 极高可用 |

### 核心指标

- **RTO**（Recovery Time Objective）：恢复时间目标
- **RPO**（Recovery Point Objective）：恢复点目标

## 常见架构

### 架构一：主从流复制

```
┌─────────┐     流复制      ┌─────────┐
│  Master │ ─────────────► │ Standby │
└─────────┘                └─────────┘
     │
     ▼
  应用写入              只读查询
```

**特点**：
- 简单，易于部署
- 支持读写分离
- 同步复制可保证数据零丢失

**适用**：中小型应用，RPO 要求不高

### 架构二：双主复制

```
┌─────────┐                ┌─────────┐
│  Node A │◄── 流复制 ────►│  Node B │
└─────────┘                └─────────┘
     │                          │
     └────────── VIP ───────────┘
```

**特点**：
- 双向同步
- 写入可分布在两个节点
- 需要解决冲突

**适用**：需要高写入吞吐量的场景

### 架构三：Citus 分布式集群

```
┌───────────────┐
│ Coordinator   │
│   (路由节点)   │
└───────┬───────┘
        │
   ┌────┴────┬────────┐
   ▼         ▼        ▼
┌─────┐  ┌─────┐  ┌─────┐
│Worker│  │Worker│  │Worker│
└─────┘  └─────┘  └─────┘
```

**特点**：
- 水平扩展
- 分片处理
- 支持大表 JOIN

**适用**：多租户 SaaS、大数据分析

## 高可用工具

### Patroni

Patroni 是最流行的 PostgreSQL HA 解决方案之一，基于 etcd、Consul 或 ZooKeeper 实现。

#### 架构

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   etcd 1    │─────│   etcd 2    │─────│   etcd 3    │
└─────────────┘     └─────────────┘     └─────────────┘
       │                                        │
       └────────────────┬───────────────────────┘
                        │
                        ▼
           ┌────────────────────────┐
           │        Patroni         │
           │   (HA 协调服务)         │
           └───────────┬────────────┘
                       │
        ┌──────────────┼──────────────┐
        ▼              ▼              ▼
   ┌─────────┐    ┌─────────┐    ┌─────────┐
   │PostgreSQL│   │PostgreSQL│   │PostgreSQL│
   │  Node 1 │    │  Node 2  │    │  Node 3  │
   └─────────┘    └─────────┘    └─────────┘
```

#### 特点

- 自动故障检测和切换
- 支持同步/异步复制
- 自动重搭故障节点
- 优雅的主节点切换

### pgpool-II

pgpool-II 是 PostgreSQL 的中间件，支持连接池、负载均衡、复制和 HA。

#### 功能

- **连接池**：复用连接，减少开销
- **负载均衡**：读写分离
- **HA**：故障检测和切换
- **并行查询**：Citus 集成

## 方案选型

### 选型指南

| 方案 | RTO | RPO | 复杂度 | 适用场景 |
|---|---|---|---|---|
| 主从 + 手动切换 | 分钟级 | 分钟级 | 低 | 开发/测试 |
| Patroni | 秒级 | 秒级 | 中 | 生产环境 |
| Pgpool-II | 秒级 | 秒级 | 中 | 读写分离 |
| Citus | 秒级 | 秒级 | 高 | 超大规模 |

### 推荐方案

**小型项目**：
```
主从流复制 + 简单脚本 + 手动切换
```

**中型项目**：
```
Patroni + etcd + 自动切换 + pg_dump 备份
```

**大型项目**：
```
Patroni + Citus + Barman 备份 + 多节点
```

## 监控告警

### 核心指标

```sql
-- 复制状态
SELECT * FROM pg_stat_replication;

-- 连接数
SELECT count(*) FROM pg_stat_activity;

-- 锁等待
SELECT * FROM pg_locks WHERE NOT granted;
```

### 告警配置

```bash
#!/bin/bash
# alert.sh

# 检查复制延迟
LAG=$(psql -t -c "SELECT pg_wal_lsn_diff(pg_current_wal_lsn(), replay_lsn) FROM pg_stat_replication;")

if [ $LAG -gt 1073741824 ]; then  # 1GB
    echo "Replication lag is too high: $LAG" | mail -s "Alert" admin@example.com
fi
```

## 故障切换最佳实践

### 1. 确保数据不丢失

```ini
# postgresql.conf
synchronous_commit = on
synchronous_standby_names = 'standby1'
```

### 2. 使用复制槽

```sql
SELECT * FROM pg_create_physical_replication_slot('standby_slot');
```

### 3. 验证切换流程

定期演练故障切换，确保 RTO/RPO 符合预期。

### 4. 记录切换日志

每次切换都需要记录，便于问题追溯。

## 总结

PostgreSQL 高可用方案选择：

1. **主从复制**：简单实用
2. **Patroni**：生产级 HA
3. **Citus**：水平扩展
4. **pgpool-II**：读写分离

关键点：
- 合理的 RTO/RPO 目标
- 定期演练和验证
- 完善的监控告警
- 可靠的备份策略

选择适合自己业务规模和需求的方案，是 PostgreSQL 高可用部署的关键。
