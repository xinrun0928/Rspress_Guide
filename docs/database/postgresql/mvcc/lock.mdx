# PostgreSQL 锁机制

高并发场景下，两个事务同时修改同一行数据会发生什么？

面试官问："PostgreSQL 的锁机制是怎样的？和 MySQL 有什么区别？"

这个问题看似简单，但真正理解 PostgreSQL 锁机制的人不多。大多数人只知道"表锁"和"行锁"，对轻量锁、页面锁、咨询锁一无所知。

---

## PostgreSQL 锁的类型

PostgreSQL 的锁系统分为几个层次：

| 锁类型 | 说明 |
|---|---|
| 轻量锁（LWLocks） | 保护共享内存数据结构，不阻塞读操作 |
| 页面锁（Page Locks） | 保护表页面，极少直接使用 |
| 行锁（Row Locks） | 保护单行数据，存储在元组头上 |
| 表锁（Relation Locks） | 保护整张表 |
| 咨询锁（Advisory Locks） | 用户自定义的应用级锁 |

---

## 表锁（Relation Locks）

### 锁模式

PostgreSQL 的表锁有多种模式，按兼容性分组：

| 锁模式 | 1 | 2 | 3 | 4 | 5 | 6 | 7 | 8 | 9 |
|---|---|---|---|---|---|---|---|---|---|
| AccessShare | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ |
| RowShare | ✗ | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ |
| RowExclusive | ✗ | ✗ | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ |
| ShareUpdateExclusive | ✗ | ✗ | ✗ | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ |
| Share | ✗ | ✗ | ✗ | ✗ | ✓ | ✓ | ✗ | ✗ | ✓ |
| ShareRowExclusive | ✗ | ✗ | ✗ | ✗ | ✗ | ✓ | ✗ | ✗ | ✓ |
| Exclusive | ✗ | ✗ | ✗ | ✗ | ✗ | ✗ | ✗ | ✓ | ✓ |
| AccessExclusive | ✗ | ✗ | ✗ | ✗ | ✗ | ✗ | ✗ | ✗ | ✗ |

### 锁的获取方式

```sql
-- 默认：读取数据时获取 AccessShareLock
SELECT * FROM orders;

-- 插入、更新、删除时获取 RowExclusiveLock
INSERT INTO orders VALUES (...);
UPDATE orders SET ...;
DELETE FROM orders;

-- DDL 操作获取 AccessExclusiveLock
ALTER TABLE orders ADD COLUMN ...;
DROP TABLE orders;
TRUNCATE orders;

-- 创建索引获取 ShareLock（PostgreSQL 11+）
CREATE INDEX CONCURRENTLY idx_orders_customer ON orders(customer_id);
```

### 显式加锁

```sql
-- 锁定整张表
LOCK TABLE orders IN ACCESS EXCLUSIVE MODE;

-- 其他锁模式
LOCK TABLE orders IN SHARE MODE;
LOCK TABLE orders IN ROW EXCLUSIVE MODE;
-- ...
```

## 行锁（Row Locks）

### 行锁模式

| 锁模式 | 说明 |
|---|---|
| FOR UPDATE | 排他锁，其他事务不能修改、不能加 FOR UPDATE |
| FOR NO KEY UPDATE | 比 FOR UPDATE 弱的排他锁，允许插入索引 |
| FOR SHARE | 共享锁，其他事务可以读，但不能修改 |
| FOR KEY SHARE | 比 FOR SHARE 弱的共享锁，允许插入索引 |

### 行锁的获取

```sql
-- FOR UPDATE：排他锁
SELECT * FROM orders WHERE id = 1 FOR UPDATE;

-- FOR SHARE：共享锁
SELECT * FROM orders WHERE id = 1 FOR SHARE;

-- UPDATE/DELETE 自动加 FOR UPDATE
UPDATE orders SET status = 'shipped' WHERE id = 1;
DELETE FROM orders WHERE id = 1;
```

### 行锁的存储

行锁不存储在元组上，而是存储在 `pg_lock_status` 视图：

```sql
-- 查看当前锁
SELECT
    pid,
    relation::regclass,
    mode,
    granted,
    fastpath
FROM pg_locks
WHERE granted = true
ORDER BY pid, relation;
```

### 行锁的等待

```sql
-- 查看锁等待
SELECT
    blocked.pid AS blocked_pid,
    blocked.query AS blocked_query,
    blocking.pid AS blocking_pid,
    blocking.query AS blocking_query
FROM pg_stat_activity AS blocked
JOIN pg_stat_activity AS blocking
    ON blocking.pid = ANY(pg_blocking_pids(blocked.pid))
WHERE blocked.cardinality(pg_blocking_pids(blocked.pid)) > 0;
```

## 轻量锁（LWLocks）

### LWLocks 的作用

轻量锁用于保护共享内存中的数据结构：

- 缓冲区锁
- 锁管理器自旋锁
- WAL 插入锁
- 事务日志锁

LWLocks 不会出现在 `pg_locks` 视图中（除非启用 `track_lock_-timing`）。

### LWLocks 的统计

```sql
-- 查看 LWLocks 等待统计
SELECT
    locktype,
    SUM(waits)::bigint AS waits,
    SUM(waits * 1000 / NULLIF(total_exec_time, 0))::bigint AS avg_wait_us
FROM pg_stat_statements
GROUP BY locktype;
```

## 死锁检测

### 死锁的产生

两个或多个事务相互等待对方持有的锁：

```
事务 A：锁住了行 1，等待行 2
事务 B：锁住了行 2，等待行 1
→ 死锁！
```

### 死锁处理

PostgreSQL 自动检测死锁，选择代价最小的事务进行回滚：

```sql
-- 死锁检测日志（postgresql.conf）
-- log_lock_waits = on
-- deadlock_timeout = '1s'

-- 查看死锁日志
SELECT * FROM pg_stat_activity WHERE wait_event_type = 'Lock';
```

### 死锁预防

```sql
-- 方法 1：始终按固定顺序访问资源
-- 事务 A 和事务 B 都先锁 id=1，再锁 id=2

-- 方法 2：使用短事务，减少锁持有时间
BEGIN;
-- 快速完成
COMMIT;

-- 方法 3：使用 SELECT ... FOR UPDATE NOWAIT
SELECT * FROM orders WHERE id = 1 FOR UPDATE NOWAIT;
-- 如果无法立即获取锁，立即报错而不是等待
```

## 咨询锁（Advisory Locks）

### 什么是咨询锁

咨询锁是一种应用级锁，不与数据库对象关联。数据库不会强制使用它们——需要应用主动检查。

### 使用场景

- 控制并发连接数
- 实现分布式锁
- 限制资源使用

### 语法

```sql
-- 获取咨询锁（阻塞等待）
SELECT pg_advisory_lock(12345);

-- 释放咨询锁
SELECT pg_advisory_unlock(12345);

-- 非阻塞获取（立即返回）
SELECT pg_try_advisory_lock(12345);

-- 释放非阻塞获取的锁
SELECT pg_advisory_unlock(12345);

-- 释放当前会话所有咨询锁
SELECT pg_advisory_unlock_all();
```

### 实战示例

```sql
-- 限制同时运行的批处理任务数
CREATE OR REPLACE FUNCTION run_batch_task(task_id BIGINT)
RETURNS VOID AS $$
BEGIN
    -- 尝试获取锁
    IF pg_try_advisory_lock(task_id) THEN
        -- 执行任务
        PERFORM do_actual_work(task_id);
        -- 释放锁
        PERFORM pg_advisory_unlock(task_id);
    ELSE
        RAISE NOTICE 'Task % is already running', task_id;
    END IF;
END;
$$ LANGUAGE plpgsql;
```

```sql
-- 会话级咨询锁（会话结束自动释放）
SELECT pg_advisory_lock(key BIGINT);
SELECT pg_advisory_lock(key1 BIGINT, key2 BIGINT);
```

## 锁监控与排查

### 查看当前锁

```sql
-- 查看所有锁
SELECT
    l.pid,
    l.relation::regclass,
    l.mode,
    l.granted,
    l.fastpath,
    a.usename,
    a.query,
    a.query_start
FROM pg_locks l
JOIN pg_stat_activity a ON l.pid = a.pid
WHERE a.pid != pg_backend_pid()
ORDER BY a.query_start;
```

### 查看锁等待

```sql
-- 查看锁等待详情
SELECT
    blocked_locks.pid AS blocked_pid,
    blocked_activity.usename AS blocked_user,
    blocked_activity.query AS blocked_query,
    blocking_locks.pid AS blocking_pid,
    blocking_activity.usename AS blocking_user,
    blocking_activity.query AS blocking_query
FROM pg_catalog.pg_locks blocked_locks
JOIN pg_catalog.pg_stat_activity blocked_activity ON blocked_locks.pid = blocked_activity.pid
JOIN pg_catalog.pg_locks blocking_locks
    ON blocking_locks.locktype = blocked_locks.locktype
    AND blocking_locks.database IS NOT DISTINCT FROM blocked_locks.database
    AND blocking_locks.relation IS NOT DISTINCT FROM blocked_locks.relation
    AND blocking_locks.page IS NOT DISTINCT FROM blocked_locks.page
    AND blocking_locks.tuple IS NOT DISTINCT FROM blocked_locks.tuple
    AND blocking_locks.virtualxid IS NOT DISTINCT FROM blocked_locks.virtualxid
    AND blocking_locks.transactionid IS NOT DISTINCT FROM blocked_locks.transactionid
    AND blocking_locks.classid IS NOT DISTINCT FROM blocked_locks.classid
    AND blocking_locks.objid IS NOT DISTINCT FROM blocked_locks.objid
    AND blocking_locks.objsubid IS NOT DISTINCT FROM blocked_locks.objsubid
    AND blocking_locks.pid != blocked_locks.pid
JOIN pg_catalog.pg_stat_activity blocking_activity ON blocking_locks.pid = blocking_activity.pid
WHERE NOT blocked_locks.granted;
```

### 杀掉阻塞进程

```sql
-- 取消正在等待锁的查询（优雅）
SELECT pg_cancel_backend(blocked_pid);

-- 强制终止会话（可能丢数据）
SELECT pg_terminate_backend(blocked_pid);
```

## PostgreSQL 与 MySQL 的锁对比

| 特性 | PostgreSQL | MySQL InnoDB |
|---|---|---|
| 行锁实现 | 元组头部标记 + 锁表 | GAP 锁 + 行锁 |
| 锁粒度 | 粗粒度（行锁较重） | 细粒度（行锁较轻） |
| 范围锁 | 无内置间隙锁 | 范围锁（可防止幻读） |
| 插入意向锁 | 无 | 有（插入操作时获取） |
| 锁等待超时 | lock_timeout | innodb_lock_wait_timeout |
| 死锁处理 | 自动检测回滚 | 自动检测回滚 |
| 咨询锁 | 支持会话级和事务级 | 支持（通过 GET_LOCK） |

---

## 总结

PostgreSQL 的锁机制有其独特之处：

1. **表锁和行锁分离**：行锁不存储在元组上，而是通过锁表管理
2. **轻量锁保护共享内存**：不会阻塞读操作
3. **咨询锁**：灵活的应用程序级锁机制
4. **自动死锁检测**：选择代价最小的事务回滚

理解锁机制是高并发场景性能调优的基础。监控锁等待、预防死锁、合理使用咨询锁，是 PostgreSQL DBA 的必备技能。
