# PostgreSQL VACUUM 机制

想象一个图书馆，每天有大量人借书还书。管理员发现，书架越来越满，但实际上很多书已经很久没人借了。

这就是 PostgreSQL 的"表膨胀"问题——MVCC 机制让旧版本元组保留在表中，直到所有事务都"看不见"它们。但谁来清理这些"过期图书"？答案就是 VACUUM。

## 为什么要 VACUUM？

### MVCC 的代价

MVCC 让 PostgreSQL 实现了读写不阻塞，但代价是旧版本元组不会被立即删除。

```sql
-- 更新操作不会立即删除旧版本
UPDATE users SET name = '新名字' WHERE id = 1;

-- 旧版本仍然存在，直到所有事务都"看不见"它
```

### 膨胀的成因

当有以下操作时，会产生垃圾元组：

| 操作 | 产生的垃圾 |
|---|---|
| UPDATE | 旧版本元组（t_xmax 被设置） |
| DELETE | 待删除元组（t_xmax 被设置） |
| 事务回滚 | 未提交的旧版本元组 |

如果这些垃圾元组不被清理：

- 表体积越来越大
- 查询需要扫描更多页面
- 索引膨胀
- 共享缓冲区被垃圾数据占用

## VACUUM 工作原理

### 基本流程

VACUUM 的工作分为三个阶段：

**1. 标记阶段**：扫描表，标记过期元组

```
表页面扫描：
┌─────────┬─────────┬─────────┬─────────┐
│ Page 0  │ Page 1  │ Page 2  │ Page 3  │
├─────────┼─────────┼─────────┼─────────┤
│ 3 live  │ 2 live  │ 1 live  │ 5 dead  │
│ 0 dead  │ 1 dead  │ 2 dead  │ 3 live  │
└─────────┴─────────┴─────────┴─────────┘
```

**2. 清理阶段**：将活跃元组紧凑排列，释放页面

```
清理后：
┌─────────┬─────────┬─────────┬─────────┐
│ Page 0  │ Page 1  │ Page 2  │ Page 3  │
├─────────┼─────────┼─────────┼─────────┤
│ 3 live  │ 3 live  │ (可用)  │ (可用)  │
│ compacted│ compacted│         │         │
└─────────┴─────────┴─────────┴─────────┘
```

**3. 更新空闲空间映射**：记录哪些页面可以重用

### VACUUM vs VACUUM FULL

| 特性 | VACUUM | VACUUM FULL |
|---|---|---|
| 锁类型 | ShareUpdateExclusiveLock（不阻塞读写） | AccessExclusiveLock（阻塞所有操作） |
| 表大小 | 不减少（只会紧凑） | 真正缩小 |
| 执行时间 | 较短 | 可能很长 |
| 磁盘空间 | 立即释放部分 | 只在最后释放 |
| 适用场景 | 日常维护 | 表确实需要缩小 |

```sql
-- 标准 VACUUM（不阻塞读写）
VACUUM orders;

-- VACUUM + 分析（推荐）
VACUUM ANALYZE orders;

-- VACUUM FULL（会锁表，慎用）
VACUUM FULL orders;

-- 空间立即释放（需要 superuser）
VACUUM FULL FREEZE;
```

## Autovacuum：自动维护

PostgreSQL 有一个 autovacuum 守护进程，自动执行 VACUUM。

### 配置参数

| 参数 | 默认值 | 说明 |
|---|---|---|
| `autovacuum` | on | 是否启用 autovacuum |
| `autovacuum_max_workers` | 3 | 最大并行 worker 数 |
| `autovacuum_naptime` | 1min | 两次检查之间的间隔 |
| `autovacuum_vacuum_threshold` | 50 | 触发 VACUUM 的最小死元组数 |
| `autovacuum_analyze_threshold` | 50 | 触发 ANALYZE 的最小变更行数 |
| `autovacuum_vacuum_scale_factor` | 0.2 | 死元组比例阈值 |
| `autovacuum_analyze_scale_factor` | 0.1 | 变更行数比例阈值 |

### 触发条件

VACUUM 的触发条件：

```
触发阈值 = autovacuum_vacuum_threshold + 表大小 * autovacuum_vacuum_scale_factor
```

```sql
-- 查看表的 autovacuum 参数
SELECT relname, reloptions FROM pg_class
WHERE reloptions IS NOT NULL;

-- 为特定表配置 autovacuum
ALTER TABLE orders SET (
    autovacuum_vacuum_threshold = 1000,
    autovacuum_vacuum_scale_factor = 0.05
);
```

### Autovacuum 的限制

autovacuum 不是万能的。以下场景需要手动 VACUUM：

1. **批量 DELETE/UPDATE 后**：数据变更量超过 autovacuum 的处理能力
2. **长事务期间**：autovacuum 无法清理被长事务引用的元组
3. **表结构变更后**：ALTER TABLE 可能导致元组格式变化，需要重建
4. **表膨胀严重时**：autovacuum 只能防止膨胀，无法快速缩小已膨胀的表

## VACUUM 与膨胀控制

### 监控膨胀

```sql
-- 查看表的膨胀程度
SELECT
    schemaname,
    tablename,
    pg_size_pretty(pg_relation_size(schemaname||'.'||tablename)) as table_size,
    n_live_tup,
    n_dead_tup,
    n_dead_tup / NULLIF(n_live_tup + n_dead_tup, 0) * 100 as dead_ratio,
    last_autovacuum,
    last_vacuum
FROM pg_stat_user_tables
WHERE n_dead_tup > 1000
ORDER BY n_dead_tup DESC;
```

```sql
-- 查看索引膨胀
SELECT
    schemaname,
    indexrelname,
    pg_size_pretty(pg_relation_size(indexrelid)) as index_size,
    idx_scan,
    idx_tup_read,
    idx_tup_fetch
FROM pg_stat_user_indexes
ORDER BY pg_relation_size(indexrelid) DESC;
```

### 膨胀的预防

**1. 合理设置 autovacuum 参数**

```sql
-- 高写入表：降低触发阈值，提高 worker 数
ALTER TABLE events SET (
    autovacuum_vacuum_threshold = 100,
    autovacuum_vacuum_scale_factor = 0.01,
    autovacuum_analyze_threshold = 50,
    autovacuum_analyze_scale_factor = 0.005
);
```

**2. 避免长事务**

```sql
-- 查看长时间运行的事务
SELECT
    pid,
    usename,
    state,
    query_start,
    state_change,
    NOW() - query_start as duration
FROM pg_stat_activity
WHERE state != 'idle'
  AND (state_change < NOW() - INTERVAL '10 minutes' OR query_start < NOW() - INTERVAL '30 minutes')
ORDER BY query_start;
```

**3. 使用 CLUSTER 重新排列数据**

```sql
-- 按索引顺序重新排列表数据（会锁表）
CLUSTER orders USING idx_orders_created_at;

-- 在线重排（需要 pg_repack 扩展）
CREATE EXTENSION pg_repack;
SELECT repack.repack_table('schema_name', 'orders');
```

### 紧急膨胀处理

当表已经严重膨胀时：

```sql
-- 方法 1：VACUUM FULL（会锁表）
VACUUM FULL orders;

-- 方法 2：TRUNCATE + 重新导入
BEGIN;
CREATE TABLE orders_new (LIKE orders INCLUDING ALL);
INSERT INTO orders_new SELECT * FROM orders;
ALTER TABLE orders RENAME TO orders_old;
ALTER TABLE orders_new RENAME TO orders;
COMMIT;
DROP TABLE orders_old;

-- 方法 3：pg_repack（在线，不锁表）
CREATE EXTENSION pg_repack;
SELECT repack.repack_table('public', 'orders');
```

## VACUUM 与 MVCC 可见性

### Frozen TXID

PostgreSQL 的事务 ID 是有限的（32 位系统约 40 亿个）。为了防止事务 ID 回绕，VACUUM 还有一个重要任务——"冻结"旧事务 ID。

```sql
-- 查看表的冻结状态
SELECT
    relname,
    age(relfrozenxid) as xid_age,
    pg_size_pretty(pg_relation_size(relid)) as size
FROM pg_class
WHERE relkind = 'r'
ORDER BY age(relfrozenxid) DESC;
```

当 `age(relfrozenxid)` 接近 `VACUUM_FREEZE_TABLE_AGE`（默认 1.5 亿）时，需要执行 VACUUM 冻结元组。

```sql
-- 强制冻结所有表
VACUUM FREEZE;

-- 查看需要紧急冻结的表
SELECT * FROM pg_class WHERE age(relfrozenxid) > 150000000;
```

:::warning
如果事务 ID 回绕，会导致数据库损坏！生产环境必须监控表的冻结年龄。
:::

## VACUUM 的性能影响

### VACUUM 的 I/O 开销

VACUUM 会产生大量顺序 I/O（写脏页面）和随机 I/O（读表页面）：

```sql
-- 设置 VACUUM 的 I/O 限制
ALTER SYSTEM SET vacuum_cost_delay = '2ms';  -- 默认 0，关闭成本控制
ALTER SYSTEM SET vacuum_cost_page_hit = 1;   -- 页面在缓存中
ALTER SYSTEM SET vacuum_cost_page_miss = 10;  -- 页面需要从磁盘读取
ALTER SYSTEM SET vacuum_cost_page_dirty = 20; -- 页面被修改
```

### 并行 VACUUM

PostgreSQL 13+ 支持并行 VACUUM：

```sql
-- 使用 4 个并行 worker
VACUUM (PARALLEL 4) orders;
```

### VACUUM 与写放大

VACUUM 会产生 WAL 写入，这在高写入场景下可能成为瓶颈：

```sql
-- 减小 VACUUM 产生的 WAL 量
ALTER SYSTEM SET vacuum_fuzzy_threshold_count = '1000';
ALTER SYSTEM SET vacuum_fuzzy_threshold_min_pages = 3;
```

## 最佳实践

### 日常维护

1. **确保 autovacuum 正常运行**

```sql
-- 查看 autovacuum 状态
SELECT * FROM pg_stat_activity WHERE query LIKE '%autovacuum%';
```

2. **定期检查膨胀情况**

```sql
-- 添加监控，定期检查表的膨胀程度
SELECT
    schemaname,
    tablename,
    n_dead_tup,
    n_live_tup,
    CASE WHEN n_live_tup > 0
         THEN round(n_dead_tup::numeric / n_live_tup * 100, 2)
         ELSE 0 END as dead_ratio
FROM pg_stat_user_tables
WHERE n_dead_tup > 10000
ORDER BY n_dead_tup DESC;
```

3. **对高写入表单独配置**

```sql
-- 大表、高写入表：降低触发阈值
ALTER TABLE big_events SET (
    autovacuum_vacuum_threshold = 500,
    autovacuum_vacuum_scale_factor = 0.01
);
```

### 批量操作后

```sql
-- 批量 DELETE 后
DELETE FROM logs WHERE created_at < NOW() - INTERVAL '1 year';
VACUUM logs;

-- 批量 UPDATE 后
UPDATE orders SET status = 'completed' WHERE status = 'pending';
VACUUM orders;
```

### 监控告警

```sql
-- 添加告警：死元组过多
-- n_dead_tup > autovacuum_vacuum_threshold + 表大小 * autovacuum_vacuum_scale_factor

-- 添加告警：VACUUM 长时间未执行
-- last_autovacuum < NOW() - INTERVAL '1 day' AND n_dead_tup > 10000

-- 添加告警：冻结年龄过高
-- age(relfrozenxid) > 150000000
```

## 总结

VACUUM 是 PostgreSQL MVCC 机制不可或缺的一部分：

1. **清理过期元组**：释放磁盘空间，减少查询扫描的页面数
2. **更新统计信息**：帮助查询规划器做出更好的决策
3. **冻结旧事务 ID**：防止事务 ID 回绕导致数据库损坏
4. **恢复空间映射**：让新插入的数据可以重用已清理的页面

autovacuum 会自动处理大部分维护工作，但你需要：

- 监控膨胀情况
- 为高写入表单独配置 autovacuum 参数
- 在批量操作后手动执行 VACUUM
- 监控冻结年龄，防止事务 ID 回绕

理解 VACUUM 机制，是进行 PostgreSQL 性能调优和容量规划的基础。
