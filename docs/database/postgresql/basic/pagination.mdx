# PostgreSQL 分页查询

"查询第 10000 页，每页 20 条数据"——这大概是所有 CRUD 开发者都会遇到的场景。

在 MySQL 中，我们习惯用 `LIMIT offset, size` 来做分页。但在 PostgreSQL 中，这种分页方式在 offset 很大时会变得非常慢。PostgreSQL 提供了更好的分页方案：游标分页（Keyset Pagination）。

## OFFSET 分页：简单但有性能陷阱

### 基本语法

```sql
SELECT * FROM orders
ORDER BY created_at DESC, id DESC
LIMIT 20 OFFSET 10000;
```

这种分页方式简单直观，但在 offset 很大时性能会急剧下降。

### 性能问题分析

```sql
-- 查看执行计划
EXPLAIN SELECT * FROM orders
ORDER BY created_at DESC, id DESC
LIMIT 20 OFFSET 10000;
```

假设 orders 表有 100 万条记录，offset 从 0 到 10000 的执行时间差异巨大：

| Offset | 预估耗时 | 说明 |
|---|---|---|
| 0 | 5ms | 读取前 20 条 |
| 1000 | 50ms | 跳过 1000 条 |
| 10000 | 500ms | 跳过 10000 条 |
| 500000 | 25s | 跳过 50 万条 |

**原因**：PostgreSQL 需要先读取并排序 10020 条数据，然后丢弃前 10000 条，只返回最后 20 条。

### OFFSET 分页的适用场景

- 数据量较小（offset `<` 1000）
- 用户不会频繁跳页
- 对分页顺序稳定性要求不高

## 游标分页：高性能分页的秘密

### 核心思想

游标分页的核心是"记住上次查询的位置"，而不是"跳过前面 N 条"。

```sql
-- 第一页
SELECT * FROM orders
ORDER BY created_at DESC, id DESC
LIMIT 20;

-- 第二页：记住上一页最后一条的 created_at 和 id
SELECT * FROM orders
WHERE (created_at, id) < ('2024-01-15 10:00:00', 99999)
ORDER BY created_at DESC, id DESC
LIMIT 20;
```

### 实现方式

#### 基于时间戳的分页

```sql
-- 分页工具函数
CREATE OR REPLACE FUNCTION paginate_orders(
    p_last_created_at TIMESTAMPTZ DEFAULT NULL,
    p_last_id BIGINT DEFAULT NULL,
    p_page_size INT DEFAULT 20
)
RETURNS TABLE (
    id BIGINT,
    customer_name TEXT,
    total_amount NUMERIC,
    created_at TIMESTAMPTZ
) AS $$
BEGIN
    RETURN QUERY
    SELECT o.id, o.customer_name, o.total_amount, o.created_at
    FROM orders o
    WHERE
        -- 关键：只查询"在上一页最后一条之后"的数据
        (p_last_created_at IS NULL OR o.created_at < p_last_created_at)
        OR (p_last_created_at IS NOT NULL AND p_last_id IS NOT NULL
            AND o.created_at = p_last_created_at AND o.id < p_last_id)
    ORDER BY o.created_at DESC, o.id DESC
    LIMIT p_page_size;
END;
$$ LANGUAGE plpgsql;
```

#### 应用层调用

```python
# Python 示例
def get_orders_page(last_cursor=None, page_size=20):
    if last_cursor is None:
        # 第一页
        query = """
            SELECT id, customer_name, total_amount, created_at
            FROM orders
            ORDER BY created_at DESC, id DESC
            LIMIT %s
        """
        params = (page_size,)
    else:
        # 后续页面
        last_created_at, last_id = last_cursor
        query = """
            SELECT id, customer_name, total_amount, created_at
            FROM orders
            WHERE (created_at, id) < (%s, %s)
            ORDER BY created_at DESC, id DESC
            LIMIT %s
        """
        params = (last_created_at, last_id, page_size)

    cursor.execute(query, params)
    rows = cursor.fetchall()

    # 返回数据 + 下一页的游标
    next_cursor = None
    if len(rows) == page_size:
        last_row = rows[-1]
        next_cursor = (last_row['created_at'], last_row['id'])

    return rows, next_cursor
```

### 游标分页的优势

| 特性 | OFFSET 分页 | 游标分页 |
|---|---|---|
| 分页速度 | 随 offset 增大而变慢 | 恒定，与页码无关 |
| 稳定性 | offset 相同时结果固定 | 不支持跳页 |
| 实现复杂度 | 简单 | 稍复杂 |
| 适用场景 | 小数据量、首页展示 | 大数据量、无限滚动 |

## 复合排序与游标

当单字段排序不够时，需要使用复合排序来保证稳定性。

### 为什么需要复合排序？

```sql
-- 假设有多条 created_at 相同的记录
INSERT INTO orders (id, created_at) VALUES
    (1, '2024-01-15 10:00:00'),
    (2, '2024-01-15 10:00:00'),
    (3, '2024-01-15 10:00:00');
```

如果只用 `ORDER BY created_at DESC LIMIT 2`，每次查询结果可能不同。需要加上第二排序键：

```sql
ORDER BY created_at DESC, id DESC
```

### 游标分页的 WHERE 条件

复合排序的游标分页需要同时比较所有排序列：

```sql
-- 排序：(created_at DESC, id DESC)
-- 游标：(last_created_at, last_id)

-- 分页查询
WHERE (created_at, id) < (last_created_at, last_id)
```

这等价于：

```sql
WHERE created_at < last_created_at
   OR (created_at = last_created_at AND id < last_id)
```

## 总数查询：何时需要，何时不需要

### 不需要精确总数

在大多数 Web 场景下，精确的"共 N 条记录"其实不那么重要：

- 用户很少翻到第 1000 页
- 显示"大约 N 条"就够了
- 用估算比精确计数快得多

```sql
-- 估算记录数
SELECT reltuples::BIGINT AS estimated_count
FROM pg_class
WHERE relname = 'orders';
```

### 需要精确总数

某些场景确实需要精确总数，比如导出功能、报表等：

```sql
-- 使用窗口函数（不影响主查询排序）
SELECT
    *,
    COUNT(*) OVER() AS total_count
FROM orders
ORDER BY created_at DESC, id DESC
LIMIT 20;
```

### 分页导航信息

```sql
-- 返回分页导航信息
WITH page_data AS (
    SELECT *
    FROM orders
    WHERE (created_at, id) < (last_cursor, last_id)
    ORDER BY created_at DESC, id DESC
    LIMIT 20
)
SELECT
    page_data.*,
    (SELECT COUNT(*) FROM orders) AS total_count,
    (SELECT id FROM orders ORDER BY created_at ASC, id ASC LIMIT 1) AS first_id,
    (SELECT id FROM orders ORDER BY created_at DESC, id DESC LIMIT 1) AS last_id
FROM page_data;
```

## 性能优化技巧

### 利用索引加速分页

确保分页查询能利用索引：

```sql
-- 创建复合索引（与排序顺序一致）
CREATE INDEX idx_orders_created_at_id ON orders (created_at DESC, id DESC);

-- 这个查询会走索引
SELECT * FROM orders
WHERE created_at < '2024-01-15 10:00:00'
ORDER BY created_at DESC, id DESC
LIMIT 20;
```

### 减少回表

如果只需要部分列，使用覆盖索引避免回表：

```sql
-- 创建覆盖索引
CREATE INDEX idx_orders_created_id ON orders (created_at DESC, id DESC) INCLUDE (customer_name, total_amount);

-- Index Only Scan，无需回表
SELECT id, customer_name, total_amount, created_at
FROM orders
WHERE created_at < '2024-01-15 10:00:00'
ORDER BY created_at DESC, id DESC
LIMIT 20;
```

### 预热缓存

```sql
-- 预热第一页数据到缓存
PREPARE orders_page AS
SELECT * FROM orders
WHERE (created_at, id) < ($1, $2)
ORDER BY created_at DESC, id DESC
LIMIT 20;

-- 执行预热
EXECUTE orders_page('2024-01-15 10:00:00', 99999);
```

## 总结

PostgreSQL 的分页查询有两个选择：

**OFFSET 分页**：简单直观，适合小数据量和首页展示。但 offset 很大时性能急剧下降。

**游标分页**：高性能，查询时间恒定。代价是不支持跳页，需要应用层配合实现。适合大数据量场景和无限滚动。

在实践中，可以这样选择：

- 首页展示、搜索结果：OFFSET 分页（offset `<` 1000）
- 大数据量翻页：游标分页
- 需要跳页：OFFSET 分页，或使用 Elasticsearch 等专门方案

PostgreSQL 的 MVCC 和索引机制为游标分页提供了良好的基础。合理设计分页方案，能让你的应用在数据量增长时依然保持流畅。
