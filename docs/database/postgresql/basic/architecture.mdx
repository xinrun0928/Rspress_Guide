# PostgreSQL 架构

很多人在使用 PostgreSQL 时，脑子里只有一个模糊的印象——它是一个数据库，存数据、取数据。但如果你想真正掌握 PostgreSQL，理解它的架构是必经之路。

面试官问："PostgreSQL 的整体架构是怎样的？查询一条数据会经过哪些组件？"

这个问题看似简单，但能答好的人不多。本文将从架构层面深入剖析 PostgreSQL 的内部世界。

---

## PostgreSQL 的进程模型

PostgreSQL 采用的是多进程架构，而不是多线程架构。这是它与 MySQL InnoDB（多线程）的一个重要区别。

### 架构概览

```
                    ┌─────────────────────────────────────────────────────┐
                    │                    PostgreSQL Server               │
                    │                                                     │
┌─────────┐         │  ┌──────────┐                                      │
│ Client  │ ───────►│  │Postmaster│──►Logger                              │
└─────────┘         │  │  Process │──►Checkpointer                        │
                    │  │  (main)  │──►Autovacuum Launcher                 │
                    │  │          │──►Wal Writer                          │
                    │  └─────┬────┘                                      │
┌─────────┐         │        │                                            │
│ Client  │ ───────►│        │ fork                                       │
└─────────┘         │        ▼                                            │
                    │  ┌──────────────────────────────────────┐            │
                    │  │         Backend Process              │            │
                    │  │                                      │            │
                    │  │  ┌─────────┐   ┌─────────┐         │            │
                    │  │  │ Parser  │──►│ Analyzer │         │            │

                    │  │  ┌─────────┐   ┌─────────┐         │            │
                    │  │  │Executor │──►│Executor │──►...   │            │
                    │  │  └─────────┘   └─────────┘         │            │
                    │  │                                      │            │
                    │  └──────────────────────────────────────┘            │
                    │                                                     │
                    │  ┌──────────────────────────────────────┐            │
                    │  │         Shared Memory                │            │
                    │  │  ┌─────────┐  ┌─────────────────┐   │            │
                    │  │  │  Buffer │  │     Lock        │   │            │
                    │  │  │  Pool   │  │    Manager      │   │            │
                    │  │  └─────────┘  └─────────────────┘   │            │
                    │  │                                      │            │
                    │  │  ┌─────────┐  ┌─────────────────┐   │            │
                    │  │  │   WAL   │  │    bgwriter     │   │            │
                    │  │  │  Buffer │  │                 │   │            │
                    │  │  └─────────┘  └─────────────────┘   │            │
                    │  └──────────────────────────────────────┘            │
                    │                                                     │
                    │  ┌──────────────────────────────────────┐            │
                    │  │           Disk (WAL + Data Files)    │            │
                    │  └──────────────────────────────────────┘            │
                    └─────────────────────────────────────────────────────┘
```

### Postmaster 进程

Postmaster 是 PostgreSQL 的主进程，负责：

1. **监听连接**：接收客户端的连接请求
2. **fork 子进程**：为每个客户端连接 fork 一个独立的 Backend 进程
3. **进程管理**：监控子进程状态，处理崩溃恢复
4. **系统初始化**：启动时初始化共享内存、WAL 系统等

Postmaster 是整个系统的守护进程，它一旦崩溃，数据库就无法服务。

### Backend 进程

每个客户端连接都会由 Postmaster fork 一个独立的 Backend 进程。这个进程负责：

1. 处理该连接的所有 SQL 请求
2. 与客户端通信
3. 使用共享内存访问数据

由于是独立进程，某个 Backend 的崩溃不会影响其他连接——这是进程模型的优势。代价是连接开销比线程模型更高，因此生产环境强烈建议使用连接池（如 PgBouncer）。

### 辅助进程

除了 Postmaster 和 Backend，PostgreSQL 还有一系列辅助进程：

| 进程 | 作用 |
|---|---|
| Logger | 写日志到文件 |
| Checkpointer | 执行检查点，将脏页写入磁盘 |
| WalWriter | 将 WAL 缓冲区写入磁盘 |
| Autovacuum Launcher | 启动自动 vacuum |
| Archiver | 归档 WAL 日志 |
| Stats Collector | 收集统计信息 |

## 查询处理流程

当你在客户端执行一条 SQL 查询时，PostgreSQL 会经过以下处理流程：

```
SQL Query
    │
    ▼
┌─────────┐
│ Parser  │  词法分析、语法分析，生成 Parse Tree
└────┬────┘
     │
     ▼
┌─────────┐
│ Analyzer│ 语义分析，生成 Query Tree
└────┬────┘
     │
     ▼
┌──────────┐
│ Rewriter │ 规则重写（如视图、物化视图规则）
└────┬─────┘
     │
     ▼
┌─────────┐
│ Planner │ 查询计划生成（最关键的一步）
└────┬────┘
     │
     ▼
┌─────────┐
│Executor │ 执行器执行查询
└─────────┘
```

### 1. Parser（解析器）

Parser 负责将 SQL 文本转换为内部数据结构（Parse Tree）。

```sql
SELECT * FROM users WHERE id = 1;
```

解析器会进行：

- **词法分析**：将 SQL 字符串分解为 token（关键字、标识符、运算符等）
- **语法分析**：根据 SQL 语法规则，检查 token 序列是否合法，生成解析树

解析器输出的 Parse Tree 包含原始的 SQL 结构，但不包含语义信息（如表是否存在、列是否正确）。

### 2. Analyzer（语义分析器）

Analyzer 接收 Parse Tree，进行语义分析：

- 查询涉及的表、列是否存在
- 数据类型是否匹配
- 权限是否足够
- 聚合函数、窗口函数等语义检查

语义分析后生成 Query Tree，Query Tree 包含查询的完整语义信息。

### 3. Rewriter（规则重写器）

Rewriter 负责处理规则系统：

- **视图**：将视图引用展开为底层查询
- **物化视图**：在 REFRESH 时执行查询，结果存储在表中
- **触发器**：在适当的位置插入触发器逻辑

```sql
-- 创建一个视图
CREATE VIEW active_users AS
    SELECT * FROM users WHERE deleted_at IS NULL;

-- 查询视图时，Rewriter 会将其展开为底层查询
SELECT * FROM active_users WHERE id = 1;
-- 被展开为：
-- SELECT * FROM users WHERE deleted_at IS NULL AND id = 1;
```

### 4. Planner（查询规划器）

Planner 是查询处理的核心。它接收 Query Tree，生成最优的执行计划。

Planner 的工作包括：

**代价估算**：基于统计信息估算每种执行方式的代价

```sql
-- 查看统计信息
SELECT * FROM pg_stats WHERE tablename = 'users';

-- 查看表大小
SELECT pg_size_pretty(pg_total_relation_size('users'));
```

**执行路径生成**：对 FROM 子句中的每个表，生成多种访问路径（顺序扫描、索引扫描等），对 JOIN 子句中的多个表，生成多种 JOIN 方式（嵌套循环、哈希连接、归并连接）

**计划选择**：比较所有候选计划的代价，选择总代价最小的计划

```sql
-- 查看执行计划（不实际执行）
EXPLAIN SELECT * FROM users WHERE id = 1;

-- 查看执行计划（实际执行，可查看行数、时间）
EXPLAIN (ANALYZE, BUFFERS) SELECT * FROM users WHERE id = 1;
```

### 5. Executor（执行器）

Executor 接收执行计划，按照计划执行查询。

执行器使用 **火山模型**（Volcano Model）：每个算子实现 `next()` 接口，返回一条元组。父算子调用子算子的 `next()`，层层向上，直到根算子返回最终结果。

```sql
-- 火山模型示意
SeqScan -> Filter -> Project -> Sort -> Limit
          (next)    (next)     (next)   (next)
```

火山模型的优势是简单、灵活，但缺点是函数调用开销大。PostgreSQL 通过**物化（Materialize）**和**推送（Push）**等技术优化性能。

## 存储管理器

### 页面结构

PostgreSQL 的基本存储单位是页（Page），默认大小 8KB。

```
┌────────────────────────────────────────────────────────────────────┐
│ Page Header (24 bytes)                                             │
│ ┌──────────────────────────────────────────────────────────────────┐│
│ │ pd_lsn (8 bytes)    │ pd_tli (2 bytes) │ pd_flags (2 bytes)    ││
│ │ pd_lower (2 bytes)  │ pd_upper (2 bytes) │ pd_special (2 bytes) ││
│ └──────────────────────────────────────────────────────────────────┘│
├────────────────────────────────────────────────────────────────────┤
│ Items (指向 ItemId 的指针)                                          │
│ ┌──────────┬──────────┬──────────┬──────────┐                      │
│ │ ItemId 1 │ ItemId 2 │ ItemId 3 │ ItemId 4 │  ...                │
│ └──────────┴──────────┴──────────┴──────────┘                      │
├────────────────────────────────────────────────────────────────────┤
│ Free Space (空闲空间)                                               │
│                                                                    │
├────────────────────────────────────────────────────────────────────┤
│ Items Data (实际数据，从后向前增长)                                  │
│ ┌──────────┬──────────┬──────────┬──────────┐                      │
│ │ Tuple 1  │ Tuple 2  │ Tuple 3  │ Tuple 4  │  ...                │
│ └──────────┴──────────┴──────────┴──────────┘                      │
├────────────────────────────────────────────────────────────────────┤
│ Special Space (特殊空间，用于索引)                                  │
└────────────────────────────────────────────────────────────────────┘
```

**Page Header** 包含页的元信息：最后修改的 WAL LSN、页类型、空闲空间指针等。

**ItemId** 是指向实际数据的指针，每个 ItemId 4 字节，包含：偏移量（指向数据的起始位置）、长度（数据的长度）、标志位。

**Items Data** 是实际的元组数据，从页尾向前增长，与 ItemId 相对生长。

### 元组结构

```
┌────────────────────────────────────────────────────────────────────┐
│ HeapTupleHeader Data                                                │
│ ┌──────────────────────────────────────────────────────────────────┐│
│ │ t_xmin (4 bytes)  │ t_xmax (4 bytes) │ t_cid (4 bytes)          ││
│ │ t_ctid (6 bytes) │ t_infomask2 (2)  │ t_infomask (2)           ││
│ │ t_bits (可变)     │ t_oid (4 bytes)  │                          ││
│ └──────────────────────────────────────────────────────────────────┘│
├────────────────────────────────────────────────────────────────────┤
│ Null Bitmap (可变的位图，表示哪些列为 NULL)                          │
├────────────────────────────────────────────────────────────────────┤
│ User Data (实际列数据)                                              │
└────────────────────────────────────────────────────────────────────┘
```

- **t_xmin**：创建这条元组的事务 ID
- **t_xmax**：删除/更新这条元组的事务 ID
- **t_cid**：命令 ID（事务内的第几条 SQL）
- **t_ctid**：元组的物理位置（页面号 + 槽号）
- **t_infomask2**：元组标志（是否有 oid、是否有空值等）
- **t_infomask**：元组标志（是否有 null、是否有变长字段等）

这些是 PostgreSQL MVCC 的基础——通过 `t_xmin` 和 `t_xmax`，PostgreSQL 可以判断一条元组对当前事务是否可见。

## 共享内存

PostgreSQL 使用共享内存来缓存数据页、 WAL 缓冲、锁信息等。

### 缓冲区缓存

缓冲区缓存是 PostgreSQL 的核心组件，用于缓存数据页：

- **大小可配置**：通过 `shared_buffers` 参数设置，默认 128MB
- **页面淘汰**：使用时钟扫描（Clock Sweep）算法
- **脏页处理**：由 bgwriter 进程定期将脏页写入磁盘

```
┌─────────────────────────────────────────┐
│           Shared Buffers                │
│  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐   │
│  │ Page │ │ Page │ │ Page │ │ Page │   │
│  │  1   │ │  2   │ │  3   │ │  4   │   │
│  └──────┘ └──────┘ └──────┘ └──────┘   │
│                                         │
│  Hash Table (Tag -> Buffer Descr)       │
│  ┌─────────────────────────────────┐    │
│  │ RelFileNode, ForkNum, BlockNum  │    │
│  └─────────────────────────────────┘    │
└─────────────────────────────────────────┘
```

### WAL 缓冲区

WAL 缓冲区用于缓存即将写入 WAL 文件的日志：

- **大小可配置**：通过 `wal_buffers` 参数设置
- **写入时机**：事务提交时、 WAL 缓冲区满时、或检查点时

### 锁管理器

PostgreSQL 使用 lightweight lock（LWLocks）和 heavyweight lock 来管理并发：

- **LWLock**：轻量锁，保护共享内存中的数据结构，如缓冲区锁、锁管理器自旋锁
- **Heavyweight Lock**：重量锁，保护表级、事务级的并发操作，如表锁、行锁

## WAL 系统

WAL（Write-Ahead Logging）是 PostgreSQL 保证数据一致性的核心机制。

### WAL 原理

核心思想：**在对数据文件做任何修改之前，必须先将修改记录写入 WAL 日志。**

```
事务提交流程：
1. 事务修改数据 → 写入共享内存缓冲区
2. 事务提交 → WAL 写入（必须先于数据文件）
3. WAL 落盘 → 返回客户端"提交成功"
4. bgwriter → 异步将脏页写入数据文件
```

这样，即使服务器在步骤 3 和 4 之间崩溃，也可以通过重放 WAL 恢复到一致状态。

### WAL 结构

```
┌─────────────────────────────────────────────────────────────┐
│ WAL Segment File (默认 16MB)                                 │
│ ┌───────────────┬───────────────┬───────────────┬───────── │
│ │ WAL Record 1  │ WAL Record 2  │ WAL Record 3  │  ...     │
│ └───────────────┴───────────────┴───────────────┴───────── │
│                                                              │
│ 每个 WAL Record 包含：                                       │
│ - RMGR ID (资源管理器 ID)                                    │
│ - RM_XID (事务 ID)                                           │
│ - Record Length                                              │
│ - Block Reference (指向被修改的数据页)                       │
│ - Page Data (修改后的数据页内容)                             │
└─────────────────────────────────────────────────────────────┘
```

### 检查点（Checkpoint）

检查点是 WAL 系统的重要组成部分：

- **作用**：标记一个点，在这个点之前的所有数据修改都已经写入数据文件
- **触发条件**：`checkpoint_timeout`、`checkpoint_completion_target`、`WAL 累计达到一定量`
- **恢复起点**：崩溃恢复时，从最后一个检查点开始重放 WAL

```sql
-- 查看检查点信息
SELECT * FROM pg_control_checkpoint();

-- 手动触发检查点
CHECKPOINT;
```

## 总结

PostgreSQL 的架构设计体现了几个核心理念：

**进程模型**：通过进程隔离保证稳定性，每个连接独立运行，但代价是连接开销较高。

**层次化设计**：Parser → Analyzer → Rewriter → Planner → Executor，每一层职责清晰，便于优化和扩展。

**MVCC 驱动**：元组头的 xmin/xmax 字段是可见性判断的基础，这使得读写互不阻塞。

**WAL 优先**：所有修改先写 WAL，保证数据安全，再异步写入数据文件，提高性能。

理解这些架构设计，不仅能帮助你在面试中回答相关问题，更能指导你在实际工作中进行性能调优、故障排查、系统设计。PostgreSQL 的代码质量是开源数据库中的标杆，阅读其源码更是一次深入理解数据库原理的绝佳机会。
