# PostgreSQL 索引类型详解

面试官问："PostgreSQL 有哪些索引类型？什么时候用 GIN 索引？什么时候用 BRIN 索引？"

很多人只知道 B-Tree 索引，问到 GIN、GiST、BRIN 时一脸茫然。PostgreSQL 的六种索引类型，每一种都是为特定场景设计的。理解这些索引的原理，才能在性能优化时做出正确的选择。

## PostgreSQL 索引类型一览

| 索引类型 | 创建语法 | 适用场景 |
|---|---|---|
| B-Tree | 默认（`USING BTREE`） | 等值查询、范围查询、排序 |
| Hash | `USING HASH` | 等值查询，不支持范围查询 |
| GIN | `USING GIN` | 数组、全文搜索、JSONB |
| GiST | `USING GIST` | 几何类型、范围类型、全文搜索 |
| BRIN | `USIN BRIN` | 物理顺序相关的大表 |
| Partial | `WHERE condition` | 只索引部分行的场景 |

---

## B-Tree 索引：默认选择

### 工作原理

B-Tree（Balanced Tree，多叉平衡树）是 PostgreSQL 的默认索引类型。B-Tree 索引保持数据有序，支持等值查询和范围查询。

```
B-Tree 结构示意：
                    [15]
              ┌───────┴───────┐
            [5, 10]         [20, 25]
          ┌──┴──┐         ┌──┴──┐
         [2]  [7]  [12] [18] [22] [28]
```

### 适用场景

- **等值查询**：`WHERE status = 'active'`
- **范围查询**：`WHERE created_at BETWEEN '2024-01-01' AND '2024-01-31'`
- **排序**：`ORDER BY created_at DESC`
- **前缀匹配**：`WHERE name LIKE 'John%'`

### 创建方式

```sql
-- 默认创建 B-Tree 索引
CREATE INDEX idx_orders_status ON orders(status);
CREATE INDEX idx_orders_created ON orders(created_at);
CREATE INDEX idx_orders_customer ON orders(customer_id);

-- 显式指定
CREATE INDEX idx_orders_status ON orders USING BTREE(status);
```

### B-Tree 的限制

B-Tree 索引有以下限制：

- 不支持非前缀匹配：`WHERE name LIKE '%John'` 无法使用 B-Tree
- 不支持表达式的前缀：`WHERE LOWER(name) = 'john'` 需要表达式索引
- 不适合高选择性的低选择列：性别（只有男/女）不适合建 B-Tree 索引

## Hash 索引：等值查询的优化

### 工作原理

Hash 索引通过哈希函数将键值转换为哈希桶的索引位置。适合等值查询，但不支持范围查询和排序。

```sql
-- 哈希函数
hash('张三') = 12345
hash('李四') = 67890

-- 索引结构
Bucket 12345 → [张三的记录]
Bucket 67890 → [李四的记录]
```

### 适用场景

- **纯等值查询**：`WHERE email = 'user@example.com'`
- **不需要排序和范围查询的场景**

### 创建方式

```sql
CREATE INDEX idx_users_email ON users USING HASH(email);
```

:::warning
Hash 索引有以下限制：

1. **不支持范围查询**：`WHERE id >= 100` 无法使用 Hash 索引
2. **不支持排序**：ORDER BY 无法使用 Hash 索引
3. **不支持多列索引**
4. **PostgreSQL 12 之前不支持 WAL**：崩溃恢复可能丢失索引数据

对于大多数场景，B-Tree 索引是更好的选择。
:::

## GIN 索引：倒排索引

### 工作原理

GIN（Generalized Inverted Index，通用倒排索引）索引维护一个"值到行"的映射。对于包含多个值（如数组）的列，GIN 索引可以快速定位包含特定值的行。

```
表数据：
Row 1: tags = ['PostgreSQL', '数据库', 'SQL']
Row 2: tags = ['MySQL', '数据库']
Row 3: tags = ['PostgreSQL', 'SQL']

GIN 索引：
'PostgreSQL' → [Row 1, Row 3]
'数据库' → [Row 1, Row 2]
'SQL' → [Row 1, Row 3]
'MySQL' → [Row 2]
```

### 适用场景

- **数组列**：`WHERE 'PostgreSQL' = ANY(tags)`
- **JSONB 列**：`WHERE data @> '{"type": "vip"}'`
- **全文搜索**：`WHERE to_tsvector('english', content) @@ query`

### 创建方式

```sql
-- 数组索引
CREATE INDEX idx_products_tags ON products USING GIN(tags);

-- JSONB 索引
CREATE INDEX idx_orders_data ON orders USING GIN(data);

-- 全文搜索索引
CREATE INDEX idx_articles_content ON articles USING GIN(to_tsvector('english', content));
```

### GIN 索引的查询

```sql
-- 数组包含查询
SELECT * FROM products WHERE tags @> ARRAY['电子产品'];

-- JSONB 包含查询
SELECT * FROM orders WHERE data @> '{"customer": "张三"}';

-- 全文搜索
SELECT * FROM articles
WHERE to_tsvector('english', content) @@ to_tsquery('english', 'PostgreSQL & 优化');
```

### GIN vs B-Tree

| 特性 | GIN | B-Tree |
|---|---|---|
| 索引大小 | 较大（存储所有值） | 较小 |
| 写入性能 | 较慢（需要更新所有倒排列表） | 较快 |
| 查询性能 | 适合多值列 | 适合单值列 |
| 支持的操作 | `@>`、`&&`、`?` 等 | `=`、`<`、`>`、`LIKE` |

## GiST 索引：通用搜索树

### 工作原理

GiST（Generalized Search Tree，通用搜索树）是一个索引框架，允许用户实现自定义的索引策略。PostgreSQL 通过 GiST 实现了多种索引类型。

```
GiST 索引覆盖的数据类型：

1. 几何类型：POINT、CIRCLE、POLYGON 等
2. 范围类型：INT4RANGE、TSRANGE 等
3. 全文搜索：TSVECTOR、TSQUERY
4. 空间索引：PostGIS 的 GEOMETRY
```

### 适用场景

- **空间查询**：KNN 最近邻搜索、范围查询
- **范围类型**：`WHERE daterange && '[2024-01-01, 2024-01-31]'`
- **全文搜索**：比 GIN 更灵活的全文搜索

### 创建方式

```sql
-- 空间数据索引（需要 PostGIS）
CREATE INDEX idx_locations_point ON locations USING GIST(geom);

-- 范围类型索引
CREATE INDEX idx_reservations_time ON reservations USING GIST(time_range);

-- 全文搜索
CREATE INDEX idx_articles_search ON articles USING GIST(to_tsvector('english', content));
```

### GiST 的查询

```sql
-- KNN 最近邻搜索（PostGIS）
SELECT * FROM locations
ORDER BY geom <-> ST_MakePoint(116.4, 39.9)
LIMIT 10;

-- 范围重叠查询
SELECT * FROM reservations
WHERE time_range && '[2024-01-15 09:00, 2024-01-15 10:00)';
```

## BRIN 索引：块范围索引

### 工作原理

BRIN（Block Range Index，块范围索引）是一种特殊的索引，适用于物理顺序与逻辑顺序相关的大表。

BRIN 索引不存储每个值，而是存储每个数据块（通常是 128 页）的统计信息：最小值、最大值等。

```
表数据（按插入顺序排列）：
Page 0:  created_at 2024-01-01 ~ 2024-01-07
Page 1:  created_at 2024-01-08 ~ 2024-01-14
Page 2:  created_at 2024-01-15 ~ 2024-01-21
...

BRIN 索引：
Page 0: min=2024-01-01, max=2024-01-07
Page 1: min=2024-01-08, max=2024-01-14
Page 2: min=2024-01-15, max=2024-01-21
...
```

当查询 `WHERE created_at BETWEEN '2024-01-10' AND '2024-01-12'` 时，BRIN 索引可以快速判断：Page 0 可能包含结果，Page 1 可能包含结果，其他页面肯定不包含。

### 适用场景

- **时序数据**：按时间顺序插入的数据（日志、传感器数据）
- **物理顺序相关的数据**：插入顺序与查询列相关
- **超大型表**：TB 级别的表，B-Tree 索引太大

### 创建方式

```sql
-- 创建 BRIN 索引
CREATE INDEX idx_logs_created ON logs USING BRIN(created_at);

-- 调整块大小
CREATE INDEX idx_logs_created ON logs USING BRIN(created_at)
WITH (pages_per_range = 64);
```

### BRIN vs B-Tree

| 特性 | BRIN | B-Tree |
|---|---|---|
| 索引大小 | 极小（表大小的 1% 以内） | 较大（可能超过表大小） |
| 写入性能 | 影响很小 | 需要更新索引 |
| 查询性能 | 适合顺序写入的表 | 适合随机分布的数据 |
| 适用场景 | 时序数据 | 通用场景 |

```sql
-- BRIN 索引的查询计划
EXPLAIN SELECT * FROM logs
WHERE created_at BETWEEN '2024-01-10' AND '2024-01-15';

-- 可能的结果：
-- Bitmap Heap Scan on logs
--   Recheck Cond: (created_at >= '2024-01-10' AND created_at <= '2024-01-15')
--   ->  BRIN Range Scan on logs_created
--         Pages: 10~12 are not skipped
```

### BRIN 的限制

BRIN 索引只在以下情况下有效：

1. **数据按索引列物理顺序插入**：时间序列数据、ID 顺序插入
2. **查询范围较小**：如果查询范围覆盖大部分表，BRIN 没有意义
3. **UPDATE 不修改索引列**：如果 UPDATE 修改了索引列，旧位置和新位置都需要维护

## Partial 索引：部分索引

### 工作原理

Partial 索引（部分索引）只索引满足条件的行。这在很多场景下可以大大减少索引体积。

```sql
-- 普通索引：索引所有行
CREATE INDEX idx_users_email ON users(email);

-- 部分索引：只索引未删除的用户
CREATE INDEX idx_users_active ON users(email) WHERE deleted_at IS NULL;
```

### 适用场景

1. **大部分行已删除/归档**：只有少量活跃数据需要索引
2. **特定状态的查询**：如只查询"已完成订单"、"VIP 用户"等
3. **降低索引体积**：减少索引大小，提高写入性能

### 创建方式

```sql
-- 索引未删除的用户
CREATE INDEX idx_users_active ON users(created_at) WHERE deleted_at IS NULL;

-- 索引已完成订单
CREATE INDEX idx_orders_completed ON orders(customer_id) WHERE status = 'completed';

-- 索引大表的最近数据
CREATE INDEX idx_events_2024 ON events(event_time)
WHERE event_time >= '2024-01-01';

-- 索引包含特定值的数据
CREATE INDEX idx_products_electronics ON products(category_id)
WHERE category_id IN (1, 2, 3);
```

### Partial 索引的查询

```sql
-- 这个查询会使用 idx_users_active
SELECT * FROM users WHERE email = 'user@example.com' AND deleted_at IS NULL;

-- 这个查询不会使用 idx_users_active（条件不满足）
SELECT * FROM users WHERE email = 'user@example.com';
```

## 索引选择决策树

```
选择索引类型：

1. 查询类型是什么？
   ├── 等值查询 + 范围查询 → B-Tree（默认选择）
   ├── 纯等值查询 → Hash 或 B-Tree
   ├── 数组包含 → GIN
   ├── JSONB 查询 → GIN
   ├── 全文搜索 → GIN 或 GiST
   ├── 空间查询 → GiST（+ PostGIS）
   ├── 时序数据 → BRIN
   └── 只索引部分行 → Partial Index

2. 列的数据分布？
   ├── 随机分布 → B-Tree
   ├── 顺序插入（时间/ID）→ BRIN
   └── 高选择性列 → B-Tree
```

## 索引的监控与维护

### 查看索引使用情况

```sql
-- 查看索引使用统计
SELECT
    schemaname,
    tablename,
    indexname,
    idx_tup_read,
    idx_tup_fetch,
    idx_scan
FROM pg_stat_user_indexes
ORDER BY idx_scan ASC;
```

### 查看未使用的索引

```sql
-- 查找从未使用的索引
SELECT
    schemaname || '.' || relname AS table,
    indexrelname AS index,
    pg_size_pretty(pg_relation_size(indexrelid)) AS index_size
FROM pg_stat_user_indexes
WHERE idx_scan = 0
  AND indexrelid NOT IN (
      SELECT conindid FROM pg_constraint WHERE contype IN ('p', 'u')
  )
ORDER BY pg_relation_size(indexrelid) DESC;
```

### 重建索引

```sql
-- REINDEX：重建索引，回收空间
REINDEX INDEX idx_orders_customer;

-- CONCURRENTLY：在线重建，不锁表
CREATE INDEX CONCURRENTLY idx_orders_customer_new ON orders(customer_id);
DROP INDEX idx_orders_customer;
ALTER INDEX idx_orders_customer_new RENAME TO idx_orders_customer;
```

## 总结

PostgreSQL 的六种索引类型，每一种都是为特定场景设计的：

1. **B-Tree**：默认选择，适合大多数等值和范围查询
2. **Hash**：适合纯等值查询，但功能有限
3. **GIN**：倒排索引，适合数组、JSONB、全文搜索
4. **GiST**：通用搜索树，适合空间数据和复杂查询
5. **BRIN**：块范围索引，适合时序数据和超大表
6. **Partial**：只索引部分行，适合特定场景

选择正确的索引类型，可以让查询性能提升几个数量级。但索引也不是万能的——过多的索引会影响写入性能，需要在查询性能和写入性能之间做出权衡。
