# PostgreSQL 索引膨胀与 REINDEX

索引和表一样，也会膨胀。随着数据的增删改，索引会变得不再紧凑，导致查询性能下降。

为什么索引会膨胀？PostgreSQL 的 MVCC 机制是罪魁祸首——每次更新或删除数据，相关索引也会产生过期条目，这些过期条目需要被清理。

---

## 索引膨胀的原因

### 1. MVCC 机制

```sql
-- 更新操作
UPDATE orders SET status = 'shipped' WHERE id = 1;

-- 索引更新过程：
-- 1. 在索引中插入新条目（指向新版本）
-- 2. 旧索引条目不会被立即删除
-- 3. 旧条目只有在所有事务都"看不见"时才能被清理
```

### 2. VACUUM 的局限性

VACUUM 会清理表的死元组，但索引的清理是间接的。VACUUM 执行时，只会标记索引页面为空闲空间，不会立即重用。

### 3. 随机插入

随机插入（如 UUID 作为主键）会导致 B-Tree 索引频繁分裂，产生大量空页面。

```sql
-- UUID 主键：随机插入
CREATE TABLE t1 (id UUID PRIMARY KEY, data TEXT);
INSERT INTO t1 VALUES (gen_random_uuid(), 'data');

-- 序列主键：顺序插入
CREATE TABLE t2 (id SERIAL PRIMARY KEY, data TEXT);
INSERT INTO t2 VALUES (DEFAULT, 'data');
```

## 索引膨胀的检测

### 1. 使用 pgstattuple 扩展

```sql
-- 安装扩展
CREATE EXTENSION pgstattuple;

-- 查看表的膨胀
SELECT * FROM pgstattuple('orders');

-- 结果字段：
-- table_len: 表总长度
-- tuple_count: 活跃元组数
-- tuple_len: 活跃元组长度
-- dead_tuple_count: 死元组数
-- dead_tuple_len: 死元组长度
-- free_space: 空闲空间
```

### 2. 使用 pg_stat_user_indexes

```sql
-- 查看索引膨胀
SELECT
    schemaname,
    relname AS table_name,
    indexrelname AS index_name,
    pg_size_pretty(pg_relation_size(indexrelid)) AS index_size,
    idx_scan,
    idx_tup_read,
    idx_tup_fetch,
    idx_tup_read - idx_tup_fetch AS dead_index_entries
FROM pg_stat_user_indexes
WHERE schemaname = 'public'
ORDER BY pg_relation_size(indexrelid) DESC;
```

### 3. 使用 pg_adminpages 扩展

```sql
-- 查看索引页面使用情况（需要 pgstattuple extension）
SELECT
    indexrelname,
    avg_leaf_density,
    leaf_page_count,
    max_leaf_age
FROM pg_stat_user_indexes ui
JOIN LATERAL (
    SELECT * FROM pgstatindex(indexrelid)
) ps ON true
ORDER BY avg_leaf_density;
```

### 4. 对比表大小和索引大小

```sql
-- 查看表的详细大小
SELECT
    schemaname,
    relname AS table_name,
    pg_size_pretty(pg_total_relation_size(relid)) AS total_size,
    pg_size_pretty(pg_relation_size(relid)) AS table_size,
    pg_size_pretty(pg_total_relation_size(relid) - pg_relation_size(relid)) AS index_size
FROM pg_stat_user_tables
WHERE schemaname = 'public'
ORDER BY pg_total_relation_size(relid) DESC;
```

## 索引膨胀的处理

### 1. REINDEX：重建索引

```sql
-- 重建单个索引
REINDEX INDEX idx_orders_customer;

-- 重建表的所有索引
REINDEX TABLE orders;

-- 重建整个数据库的所有索引（慎用）
REINDEX DATABASE mydb;
```

### 2. CONCURRENTLY：在线重建

标准 REINDEX 会锁表。CONCURRENTLY 选项允许在线重建：

```sql
-- 在线重建索引（不锁表）
-- 1. 创建新索引
CREATE INDEX CONCURRENTLY idx_orders_customer_new ON orders(customer_id);

-- 2. 删除旧索引
DROP INDEX idx_orders_customer;

-- 3. 重命名新索引
ALTER INDEX idx_orders_customer_new RENAME TO idx_orders_customer;
```

:::warning
CONCURRENTLY 模式下：

1. 需要额外的磁盘空间（约 2 倍索引大小）
2. 不能在事务中执行
3. 如果失败，需要手动清理残留索引
4. 索引的创建时间是正常创建的 2 倍
:::

### 3. 使用 pg_repack

pg_repack 可以在不锁表的情况下重建表和索引：

```sql
-- 安装扩展
CREATE EXTENSION pg_repack;

-- 在线重建表（会重建所有索引）
SELECT repack.repack_table('orders');

-- 只重建索引
SELECT repack.repack_index('idx_orders_customer');
```

## 索引膨胀的预防

### 1. 使用 BRIN 索引

对于按顺序插入的表（如时序数据），BRIN 索引比 B-Tree 更稳定：

```sql
-- 日志表：按 created_at 顺序插入
CREATE TABLE logs (
    id BIGSERIAL PRIMARY KEY,
    created_at TIMESTAMPTZ NOT NULL,
    message TEXT
);

-- BRIN 索引：索引体积更小，膨胀更慢
CREATE INDEX idx_logs_created ON logs USING BRIN(created_at);

-- B-Tree 索引：可能膨胀
CREATE INDEX idx_logs_created_btree ON logs(created_at);
```

### 2. 使用序列作为主键

```sql
-- 好：序列主键，顺序插入
CREATE TABLE t1 (
    id BIGSERIAL PRIMARY KEY,
    data TEXT
);

-- 不好：UUID 主键，随机插入
CREATE TABLE t2 (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    data TEXT
);
```

### 3. 合理设置 fillfactor

fillfactor 控制页面填充程度，留下一些空间供后续更新：

```sql
-- 创建表时设置 fillfactor
CREATE TABLE orders (
    id SERIAL PRIMARY KEY,
    status VARCHAR(20)
) WITH (fillfactor = 70);

-- 创建索引时设置 fillfactor
CREATE INDEX idx_orders_status ON orders(status) WITH (fillfactor = 80);
```

### 4. 定期维护

```sql
-- 设置 autovacuum 参数
ALTER TABLE orders SET (
    autovacuum_vacuum_threshold = 50,
    autovacuum_vacuum_scale_factor = 0.01
);
```

## REINDEX 的最佳实践

### 1. 选择合适的时机

REINDEX 会消耗大量 CPU 和 I/O。最好在低峰期执行：

```sql
-- 查看当前连接
SELECT * FROM pg_stat_activity WHERE state != 'idle';

-- 设置语句超时
SET statement_timeout = '30min';
```

### 2. 使用 CONCURRENTLY

生产环境优先使用 CONCURRENTLY：

```sql
-- 在线重建脚本
DO $$
DECLARE
    index_name TEXT;
BEGIN
    FOR index_name IN
        SELECT indexname
        FROM pg_indexes
        WHERE tablename = 'orders'
          AND indexname NOT LIKE '%_new'
    LOOP
        EXECUTE format('CREATE INDEX CONCURRENTLY %I_new ON orders (%s)', index_name, substring(index_name from 'idx_orders_(.*)'));
        EXECUTE format('DROP INDEX %I', index_name);
        EXECUTE format('ALTER INDEX %I_new RENAME TO %I', index_name, index_name);
    END LOOP;
END $$;
```

### 3. 监控进度

```sql
-- 查看 REINDEX 进度（PostgreSQL 13+）
SELECT
    phase,
    pg_size_pretty(heap_blks_total * 8192) AS total_size,
    pg_size_pretty(heap_blks_scanned * 8192) AS scanned_size,
    pg_size_pretty(heap_blks_total * 8192 * (heap_blks_scanned::numeric / NULLIF(heap_blks_total, 0))) AS estimated_size,
    100 * (heap_blks_scanned::numeric / NULLIF(heap_blks_total, 0)) AS progress
FROM pg_stat_progress_create_index;
```

### 4. 处理 CONCURRENTLY 失败

如果 CONCURRENTLY 失败，残留的索引需要手动清理：

```sql
-- 查看失败状态
SELECT * FROM pg_indexes WHERE indexname LIKE '%_new';

-- 删除残留索引
DROP INDEX CONCURRENTLY IF EXISTS idx_orders_customer_new;
```

---

## 总结

索引膨胀是 PostgreSQL 使用中常见的问题：

1. **膨胀原因**：MVCC 机制、随机插入、空页面碎片
2. **检测方法**：pgstattuple、pg_stat_user_indexes、大小对比
3. **处理方法**：REINDEX、CONCURRENTLY、pg_repack
4. **预防措施**：使用 BRIN 索引、序列主键、合理的 fillfactor

定期监控索引大小和膨胀情况，在低峰期进行维护，是保持数据库性能的重要工作。
