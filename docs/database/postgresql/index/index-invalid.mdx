# PostgreSQL 索引失效场景

面试官问："你建了索引，但查询还是很慢，可能是什么原因？"

这个问题看似简单，但真正理解索引失效原因的人不多。很多人以为建了索引就一定能加速查询，但实际上索引有自己的"脾气"。有些查询模式会导致索引无法使用，或者使用效率很低。

---

## 索引失效的常见原因

### 1. 索引列参与计算

```sql
-- 创建索引
CREATE INDEX idx_orders_total ON orders(total_amount);

-- 索引失效：列参与计算
EXPLAIN SELECT * FROM orders WHERE total_amount * 1.1 > 1000;

-- 正确写法：将计算移到右边
EXPLAIN SELECT * FROM orders WHERE total_amount > 1000 / 1.1;
```

### 2. 使用函数或类型转换

```sql
-- 创建索引
CREATE INDEX idx_users_email ON users(email);

-- 索引失效：使用函数
EXPLAIN SELECT * FROM users WHERE UPPER(email) = 'JOHN@EXAMPLE.COM';

-- 解决方案：创建表达式索引
CREATE INDEX idx_users_email_upper ON users(UPPER(email));
EXPLAIN SELECT * FROM users WHERE UPPER(email) = 'JOHN@EXAMPLE.COM';

-- 索引失效：隐式类型转换
CREATE INDEX idx_users_user_id ON users(user_id);

-- 如果 user_id 是 INTEGER，但查询传入 STRING
EXPLAIN SELECT * FROM users WHERE user_id = '123';  -- 可能失效

-- PostgreSQL 会尝试转换，但最好显式转换
EXPLAIN SELECT * FROM users WHERE user_id = 123::INTEGER;
```

### 3. 使用 LIKE 模式匹配

```sql
-- 创建索引
CREATE INDEX idx_users_name ON users(name);

-- 索引有效：前缀匹配
EXPLAIN SELECT * FROM users WHERE name LIKE 'John%';

-- 索引失效：非前缀匹配
EXPLAIN SELECT * FROM users WHERE name LIKE '%John%';

-- 解决方案：使用 pg_trgm 扩展
CREATE EXTENSION IF NOT EXISTS pg_trgm;
CREATE INDEX idx_users_name_trgm ON users USING GIN (name gin_trgm_ops);
EXPLAIN SELECT * FROM users WHERE name LIKE '%John%';
```

### 4. 使用 OR 连接多个条件

```sql
-- 创建索引
CREATE INDEX idx_users_email ON users(email);
CREATE INDEX idx_users_phone ON users(phone);

-- 索引可能失效：OR 条件
EXPLAIN SELECT * FROM users WHERE email = 'john@example.com' OR phone = '12345678';

-- 解决方案：使用 UNION
EXPLAIN
SELECT * FROM users WHERE email = 'john@example.com'
UNION
SELECT * FROM users WHERE phone = '12345678';

-- 或者：使用两个索引 + BitmapOr
CREATE INDEX idx_users_email ON users(email);
CREATE INDEX idx_users_phone ON users(phone);
EXPLAIN SELECT * FROM users WHERE email = 'john@example.com' OR phone = '12345678';
```

### 5. 使用 NOT 操作符

```sql
-- 创建索引
CREATE INDEX idx_orders_status ON orders(status);

-- 索引失效：NOT IN / NOT EXISTS
EXPLAIN SELECT * FROM orders WHERE status NOT IN ('cancelled', 'refunded');

-- 索引可能部分有效：NOT LIKE
EXPLAIN SELECT * FROM users WHERE name NOT LIKE 'A%';

-- 解决方案：重新设计查询或使用表达式索引
EXPLAIN
SELECT * FROM orders
WHERE status IN ('pending', 'shipped', 'delivered');
```

### 6. 数据类型不匹配

```sql
-- 创建索引
CREATE INDEX idx_orders_created ON orders(created_at);

-- 如果 created_at 是 TIMESTAMPTZ，查询用 DATE 可能会失效
EXPLAIN SELECT * FROM orders WHERE created_at::DATE = '2024-01-15';

-- 正确写法：使用范围查询
EXPLAIN SELECT * FROM orders
WHERE created_at >= '2024-01-15 00:00:00'
  AND created_at < '2024-01-16 00:00:00';
```

## 索引效率低下的场景

### 1. 低选择性列

```sql
-- status 只有 3 个值，选择性很低
SELECT status, COUNT(*) FROM orders GROUP BY status;

-- 即使建了索引，查询优化器可能选择全表扫描
CREATE INDEX idx_orders_status ON orders(status);
EXPLAIN SELECT * FROM orders WHERE status = 'pending';
-- 结果：Seq Scan on orders（而不是 Index Scan）
```

**判断方法**：如果索引列的不同值数量很少，索引可能不会被使用。

### 2. 模糊的范围查询

```sql
-- 创建索引
CREATE INDEX idx_orders_created ON orders(created_at);

-- 如果查询范围太大，优化器可能选择全表扫描
EXPLAIN SELECT * FROM orders
WHERE created_at > '2023-01-01' AND created_at < '2024-12-31';
-- 如果这个范围覆盖了大部分数据，索引可能失效
```

### 3. 统计信息不准确

```sql
-- 查看统计信息
SELECT * FROM pg_stats WHERE tablename = 'orders' AND attname = 'status';

-- 如果统计信息过期，索引可能不会被使用
ANALYZE orders;

-- 或者手动调整统计信息
ALTER TABLE orders ALTER COLUMN status SET STATISTICS 500;
ANALYZE orders;
```

## 索引失效的诊断方法

### 1. 使用 EXPLAIN 分析查询

```sql
-- 查看执行计划
EXPLAIN SELECT * FROM orders WHERE status = 'pending';

-- 实际执行并查看时间
EXPLAIN (ANALYZE, BUFFERS) SELECT * FROM orders WHERE status = 'pending';

-- 分析结果中的关键信息：
-- Seq Scan：全表扫描，索引可能未使用
-- Index Scan：使用索引
-- Index Only Scan：使用索引且不回表
-- Bitmap Heap Scan：使用位图扫描
```

### 2. 检查索引是否存在

```sql
-- 查看表的所有索引
\d+ orders

-- 查看索引定义
SELECT indexname, indexdef FROM pg_indexes
WHERE tablename = 'orders';
```

### 3. 检查索引使用情况

```sql
-- 查看索引扫描次数
SELECT
    schemaname,
    relname,
    indexrelname,
    idx_scan,
    idx_tup_read,
    idx_tup_fetch
FROM pg_stat_user_indexes
WHERE schemaname = 'public'
ORDER BY idx_scan;
```

## 索引失效的预防措施

### 1. 避免在索引列上使用函数

```sql
-- 不好
SELECT * FROM users WHERE LOWER(email) = 'john';

-- 好：使用表达式索引
CREATE INDEX idx_users_email_lower ON users(LOWER(email));
SELECT * FROM users WHERE LOWER(email) = 'john';

-- 好：将计算移到右边
SELECT * FROM orders WHERE total > 100;
-- 而不是：WHERE total + 10 > 110
```

### 2. 统一数据类型

```sql
-- 确保查询条件的数据类型与列一致
-- 如果 user_id 是 BIGINT
SELECT * FROM users WHERE user_id = 123;  -- 正确
SELECT * FROM users WHERE user_id = '123';  -- 隐式转换，可能有问题

-- 显式转换
SELECT * FROM users WHERE user_id = 123::BIGINT;
```

### 3. 避免过多的 OR 条件

```sql
-- 不好
SELECT * FROM users WHERE email = 'a' OR phone = 'b' OR name = 'c';

-- 好：UNION
SELECT * FROM users WHERE email = 'a'
UNION
SELECT * FROM users WHERE phone = 'b'
UNION
SELECT * FROM users WHERE name = 'c';
```

### 4. 使用覆盖索引避免回表

```sql
-- 普通索引需要回表
CREATE INDEX idx_orders_customer ON orders(customer_id);
SELECT customer_id, total FROM orders WHERE customer_id = 1;
-- 需要回表获取 total

-- 覆盖索引
CREATE INDEX idx_orders_customer_covering ON orders(customer_id) INCLUDE (total);
SELECT customer_id, total FROM orders WHERE customer_id = 1;
-- 不需要回表
```

## 常见误区

### 误区一：索引越多越好

索引会拖慢写入性能。每个 INSERT/UPDATE 需要同时更新所有相关索引。

```sql
-- 统计表的索引数量
SELECT COUNT(*) FROM pg_indexes WHERE tablename = 'orders';

-- 如果索引太多，考虑删除不用的索引
SELECT indexname FROM pg_indexes
WHERE tablename = 'orders'
  AND indexname NOT IN (
      SELECT indexname FROM pg_stat_user_indexes
      WHERE relname = 'orders' AND idx_scan > 0
  );
```

### 误区二：索引可以解决所有性能问题

索引只对特定查询模式有效。对于全表扫描更快的场景（如查询超过 30% 的数据），索引反而是负担。

### 误区三：复合索引顺序无所谓

复合索引的列顺序很重要。等值列应放在范围列前面，高选择列应放在前面。

---

## 总结

索引失效是一个常见的性能问题。了解索引失效的原因，可以帮助你更好地设计索引和编写查询：

1. **索引列参与计算**：将计算移到右边
2. **使用函数或类型转换**：创建表达式索引或统一类型
3. **LIKE 非前缀匹配**：使用 pg_trgm 扩展
4. **OR 连接多个条件**：使用 UNION 或 AND 条件
5. **数据类型不匹配**：确保查询条件类型一致
6. **低选择性列**：索引可能不被使用

使用 EXPLAIN 分析查询计划，是诊断索引问题的关键。
