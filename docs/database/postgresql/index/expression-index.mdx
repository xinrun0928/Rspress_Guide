# PostgreSQL 表达式索引

"你的表里有一个 `email` 列，但你总是用 `LOWER(email)` 查询。你会怎么建索引？"

很多人会说："在 `LOWER(email)` 上建索引啊。"但如果真正去尝试，你会发现语法不太对劲。

PostgreSQL 的表达式索引可以解决这个问题——你可以在索引中使用表达式，让查询"透明地"使用索引。

---

## 表达式索引基础

### 为什么需要表达式索引

```sql
-- 你总是这样查询
SELECT * FROM users WHERE LOWER(email) = 'john@example.com';

-- 普通索引无法使用
CREATE INDEX idx_users_email ON users(email);
-- 查询仍需全表扫描或索引扫描后转换

-- 表达式索引
CREATE INDEX idx_users_email_lower ON users(LOWER(email));
-- 查询可以直接使用索引
```

### 创建表达式索引

```sql
-- 索引函数结果
CREATE INDEX idx_users_email_lower ON users(LOWER(email));

-- 索引计算结果
CREATE INDEX idx_orders_year ON orders(EXTRACT(YEAR FROM created_at));

-- 索引类型转换
CREATE INDEX idx_users_id_int ON users((user_id::TEXT));

-- 索引条件表达式
CREATE INDEX idx_users_active_name ON users(CASE WHEN deleted_at IS NULL THEN name END);
```

### 使用表达式索引

```sql
-- 查询会自动使用表达式索引
SELECT * FROM users WHERE LOWER(email) = 'john@example.com';

-- 执行计划验证
EXPLAIN SELECT * FROM users WHERE LOWER(email) = 'john@example.com';

-- 结果应该包含：Index Scan using idx_users_email_lower on users
```

## 常见使用场景

### 场景一：大小写不敏感的查询

```sql
-- 创建索引
CREATE INDEX idx_users_email_lower ON users(LOWER(email));

-- 查询
SELECT * FROM users WHERE LOWER(email) = LOWER('John@Example.COM');

-- 也可以用表达式索引实现大小写不敏感的排序
CREATE INDEX idx_users_name_ci ON users(UPPER(name));
```

### 场景二：日期时间处理

```sql
-- 按年份查询
CREATE INDEX idx_orders_year ON orders(EXTRACT(YEAR FROM created_at));

-- 查询
SELECT * FROM orders
WHERE EXTRACT(YEAR FROM created_at) = 2024;

-- 按月查询
CREATE INDEX idx_orders_month ON orders(TO_CHAR(created_at, 'YYYY-MM'));

SELECT * FROM orders
WHERE TO_CHAR(created_at, 'YYYY-MM') = '2024-01';
```

### 场景三：JSON/JSONB 路径索引

```sql
-- JSONB 字段的特定路径
CREATE INDEX idx_orders_customer ON orders((order_data->>'customer'));

-- 查询
SELECT * FROM orders WHERE (order_data->>'customer') = '张三';

-- 或者直接在 JSONB 上建 GIN 索引
CREATE INDEX idx_orders_data ON orders USING GIN(order_data);

-- 但表达式索引更精确
SELECT * FROM orders WHERE order_data @> '{"customer": "张三"}';
```

### 场景四：范围查询优化

```sql
-- 索引计算结果
CREATE INDEX idx_users_age_group ON users(
    CASE
        WHEN age < 18 THEN 'minor'
        WHEN age < 65 THEN 'adult'
        ELSE 'senior'
    END
);

-- 查询
SELECT * FROM users
WHERE CASE WHEN age < 18 THEN 'minor' WHEN age < 65 THEN 'adult' ELSE 'senior' END = 'senior';
```

## 表达式索引的注意事项

### 表达式必须完全匹配

```sql
-- 创建索引
CREATE INDEX idx_test ON test(LOWER(name));

-- 可以使用索引
SELECT * FROM test WHERE LOWER(name) = 'john';

-- 不能使用索引（表达式不匹配）
SELECT * FROM test WHERE name ILIKE 'john%';  -- 需要 ILIKE 专用索引
```

### 创建 ILIKE 索引

PostgreSQL 提供 `pg_trgm` 扩展实现大小写不敏感的模糊匹配：

```sql
-- 安装扩展
CREATE EXTENSION IF NOT EXISTS pg_trgm;

-- 创建支持 ILIKE 的索引
CREATE INDEX idx_test_name_trgm ON test USING GIN (name gin_trgm_ops);

-- 查询
SELECT * FROM test WHERE name ILIKE 'John%';  -- 使用索引
SELECT * FROM test WHERE name ILIKE '%ohn%';  -- 使用索引
```

### 表达式索引的性能考虑

```sql
-- 表达式索引在写入时计算并存储
INSERT INTO users (email) VALUES ('John@Example.COM');
-- PostgreSQL 自动计算 LOWER(email) 并存入索引

-- 表达式可能很复杂，索引体积会增大
CREATE INDEX idx_orders_complex ON orders(
    SUBSTRING(product_code, 1, 3) || '-' ||
    EXTRACT(YEAR FROM created_at)
);
```

## 函数索引 vs 函数表达式索引

### 函数索引（PostgreSQL 10+）

PostgreSQL 10+ 支持直接使用函数作为索引：

```sql
-- PostgreSQL 10+ 的简化语法
CREATE INDEX idx_users_email_lower ON users(LOWER(email));

-- 等价于（早期版本的语法）
CREATE INDEX idx_users_email_lower ON users USING BTREE (LOWER(email));
```

### 括号的重要性

```sql
-- 正确：表达式用括号
CREATE INDEX idx1 ON test(LOWER(name));

-- 错误：函数名不加括号会被当作列名
CREATE INDEX idx2 ON test(LOWER name);  -- 会报错
```

## 表达式索引与部分索引的结合

### 组合使用

```sql
-- 索引活跃用户的小写邮箱
CREATE INDEX idx_users_active_email_lower ON users(LOWER(email))
WHERE deleted_at IS NULL;

-- 查询
SELECT * FROM users
WHERE LOWER(email) = 'john@example.com'
  AND deleted_at IS NULL;
```

### 物化路径索引

```sql
-- 路径型数据
CREATE TABLE categories (
    id SERIAL PRIMARY KEY,
    path TEXT NOT NULL  -- 格式：'1/2/3'
);

-- 索引路径长度
CREATE INDEX idx_categories_path_length ON categories(LENGTH(path));

-- 查询特定深度的分类
SELECT * FROM categories WHERE LENGTH(path) = 3;
```

## 表达式索引的维护

### 查看表达式索引

```sql
-- 查看索引定义
SELECT indexdef FROM pg_indexes
WHERE tablename = 'users';

-- 查看索引大小
SELECT
    indexname,
    pg_size_pretty(pg_relation_size(indexrelid)) AS size
FROM pg_stat_user_indexes
WHERE relname = 'users';
```

### 表达式索引与统计信息

```sql
-- 表达式索引需要单独的统计信息
ANALYZE users;

-- 查看表达式的统计信息
SELECT * FROM pg_stats
WHERE tablename = 'users'
  AND attname LIKE '%expr%';
```

## 最佳实践

### 什么时候使用表达式索引

1. **查询总是使用特定函数/表达式**
2. **表达式是查询的热点**
3. **普通索引无法满足需求**

### 什么时候不使用表达式索引

1. **表达式不经常使用**：维护成本大于收益
2. **表达式不稳定**：会导致频繁的索引重建
3. **表达式很复杂**：索引体积过大，影响写入性能

### 替代方案

如果表达式索引不合适，可以考虑：

1. **生成列（Generated Columns）**：存储计算结果

```sql
-- PostgreSQL 12+ 支持生成列
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    email VARCHAR(255),
    email_lower VARCHAR(255) GENERATED ALWAYS AS (LOWER(email)) STORED
);

-- 在生成列上建普通索引
CREATE INDEX idx_users_email_lower ON users(email_lower);
```

2. **物化视图**：预先计算复杂表达式

3. **应用层计算**：在写入时计算并存储

---

## 总结

表达式索引是 PostgreSQL 的强大特性：

1. **透明使用**：查询自动使用索引，无需修改 SQL
2. **支持函数、表达式、类型转换**
3. **可以与部分索引结合**
4. **表达式必须在查询中完全匹配**

合理使用表达式索引，可以让原本无法优化的查询变得高效。但要注意索引体积和写入性能的权衡。
