# PostgreSQL 执行计划分析

面试官问："有一个 SQL 查询很慢，你如何分析？"

大多数人会说："用 EXPLAIN 看一下执行计划。"这个回答没有错，但太笼统了。面试官想知道的是：你能不能看懂执行计划？能不能找到性能瓶颈？能不能提出优化建议？

EXPLAIN 是 PostgreSQL 性能优化的核心工具。理解执行计划，是进行 SQL 优化的基础。

---

## EXPLAIN 基础

### 基本语法

```sql
-- 查看执行计划（不执行）
EXPLAIN SELECT * FROM orders WHERE customer_id = 1;

-- 查看执行计划（实际执行，查看真实行数和时间）
EXPLAIN (ANALYZE) SELECT * FROM orders WHERE customer_id = 1;

-- 查看缓冲区使用情况
EXPLAIN (ANALYZE, BUFFERS) SELECT * FROM orders WHERE customer_id = 1;

-- 查看格式化输出
EXPLAIN (FORMAT YAML) SELECT * FROM orders WHERE customer_id = 1;
EXPLAIN (FORMAT JSON) SELECT * FROM orders WHERE customer_id = 1;
```

### 执行计划的结构

执行计划是一棵树，从下往上执行：

```
Hash Join (cost=100.00..500.00 rows=10000)
  -> Seq Scan on orders (cost=0.00..200.00 rows=10000)
  -> Hash (cost=50.00..50.00 rows=500)
    -> Seq Scan on customers (cost=0.00..50.00 rows=500)
```

**读取顺序**：先读 Seq Scan on customers（叶子节点），然后执行 Hash，最后执行 Hash Join。

## 扫描类型

### Seq Scan（顺序扫描）

```sql
EXPLAIN SELECT * FROM orders WHERE status = 'pending';
```

```
Seq Scan on orders  (cost=0.00..200.00 rows=10000)
  Filter: (status = 'pending')
```

**含义**：扫描整张表，逐一检查每行是否满足条件。

**何时出现**：
- 查询的列没有索引
- 查询的选择性很低（返回大量数据）
- 表很小（索引反而更慢）

**优化方向**：
- 添加合适的索引
- 使用覆盖索引避免回表

### Index Scan（索引扫描）

```sql
CREATE INDEX idx_orders_customer ON orders(customer_id);

EXPLAIN SELECT * FROM orders WHERE customer_id = 1;
```

```
Index Scan using idx_orders_customer on orders  (cost=0.42..8.44 rows=10)
  Index Cond: (customer_id = 1)
```

**含义**：使用索引找到匹配的行，然后回表获取数据。

**何时出现**：
- WHERE 条件使用了索引列
- 索引选择性较高

### Index Only Scan（索引覆盖扫描）

```sql
CREATE INDEX idx_orders_customer ON orders(customer_id);

EXPLAIN SELECT customer_id FROM orders WHERE customer_id = 1;
```

```
Index Only Scan using idx_orders_customer on orders  (cost=0.42..8.44 rows=10)
  Index Cond: (customer_id = 1)
```

**含义**：查询的所有列都在索引中，不需要回表。

**何时出现**：
- 查询的列都在索引中（覆盖索引）
- 索引的统计信息准确

### Bitmap Heap Scan（位图扫描）

```sql
EXPLAIN SELECT * FROM orders WHERE customer_id IN (1, 2, 3);
```

```
Bitmap Heap Scan on orders  (cost=100.00..300.00 rows=500)
  Recheck Cond: (customer_id = ANY (ARRAY[1, 2, 3]))
  ->  Bitmap Index Scan on idx_orders_customer  (cost=0.00..95.00 rows=500)
        Index Cond: (customer_id = ANY (ARRAY[1, 2, 3]))
```

**含义**：先通过索引找到所有匹配行的位置，再批量读取数据。

**何时出现**：
- 查询返回多行（不是 1-2 行）
- 使用 IN 查询
- 使用 OR 条件

**优势**：相比 Index Scan，减少了随机 I/O。适合返回大量数据的场景。

### 子查询扫描

| 类型 | 说明 |
|---|---|
| `Seq Scan on ...` | 扫描子查询结果 |
| `Subplan` | 标量子查询 |
| `CTE Scan` | CTE 扫描 |

## JOIN 类型

### Nested Loop（嵌套循环）

```sql
EXPLAIN SELECT * FROM orders o
JOIN customers c ON o.customer_id = c.id
WHERE c.region = 'North';
```

```
Nested Loop  (cost=0.42..1000.00 rows=100)
  ->  Index Scan using idx_customers_region on customers  (cost=0.42..50.00 rows=10)
        Index Cond: (region = 'North')
  ->  Index Scan using idx_orders_customer on orders  (cost=0.42..8.44 rows=10)
        Index Cond: (customer_id = c.id)
```

**原理**：

```
for each row in outer_table:
    for each row in inner_table:
        if match: output row
```

**适用场景**：
- 小表驱动大表
- 内层查询有索引支持
- 返回少量数据

**优化方向**：
- 确保驱动表返回少量数据（添加 WHERE 条件）
- 确保内层查询的 JOIN 列有索引

### Hash Join（哈希连接）

```sql
EXPLAIN SELECT * FROM orders o
JOIN customers c ON o.customer_id = c.id;
```

```
Hash Join  (cost=100.00..500.00 rows=10000)
  Hash Cond: (o.customer_id = c.id)
  ->  Seq Scan on orders  (cost=0.00..200.00 rows=10000)
  ->  Hash  (cost=50.00..50.00 rows=500)
    ->  Seq Scan on customers  (cost=0.00..50.00 rows=500)
```

**原理**：
1. 扫描小表，构建哈希表
2. 扫描大表，在哈希表中查找匹配行

**适用场景**：
- 大表 JOIN 大表
- 无合适索引
- 等值 JOIN

**内存消耗**：
- `work_mem` 参数控制哈希表大小
- 如果哈希表太大，会溢出到磁盘

### Merge Join（归并连接）

```sql
EXPLAIN SELECT * FROM orders o
JOIN customers c ON o.customer_id = c.id;
```

```
Merge Join  (cost=100.00..500.00 rows=10000)
  Merge Cond: (o.customer_id = c.id)
  ->  Sort  (cost=50.00..55.00 rows=500)
        Sort Key: customer_id
        ->  Seq Scan on customers  (cost=0.00..45.00 rows=500)
  ->  Sort  (cost=50.00..55.00 rows=500)
        Sort Key: customer_id
        ->  Seq Scan on orders  (cost=0.00..45.00 rows=500)
```

**原理**：
1. 对两个表按 JOIN 列排序
2. 同时扫描两个表，匹配相同的值

**适用场景**：
- JOIN 列已有索引（无需排序）
- 数据量大
- 等值 JOIN

**优势**：不需要大内存存储哈希表。

## 聚合操作

### Hash Aggregate（哈希聚合）

```sql
EXPLAIN SELECT status, COUNT(*) FROM orders GROUP BY status;
```

```
Hash Aggregate  (cost=200.00..250.00 rows=10)
  Group Key: status
  ->  Seq Scan on orders  (cost=0.00..150.00 rows=10000)
```

**适用场景**：
- 无排序需求的聚合
- 数据量较大

### Group Aggregate（分组聚合）

```sql
EXPLAIN SELECT status, COUNT(*) FROM orders GROUP BY status;
```

```
Group Aggregate  (cost=200.00..250.00 rows=10)
  Group Key: status
  ->  Sort  (cost=200.00..210.00 rows=10000)
        Sort Key: status
        ->  Seq Scan on orders  (cost=0.00..150.00 rows=10000)
```

**适用场景**：
- 有排序需求
- 结果需要有序输出

### Sort（排序）

```sql
EXPLAIN SELECT * FROM orders ORDER BY created_at DESC;
```

```
Sort  (cost=200.00..250.00 rows=10000)
  Sort Key: created_at DESC
  Sort Method: quicksort  Memory: 2048kB
  ->  Seq Scan on orders  (cost=0.00..150.00 rows=10000)
```

**Sort Method**：
- `quicksort`：内存排序
- `top-N heapsort`：LIMIT 查询的优化

## 成本模型

### 成本参数

PostgreSQL 的成本模型基于两个参数：

| 参数 | 默认值 | 说明 |
|---|---|---|
| `seq_page_cost` | 1.0 | 顺序扫描页面的成本 |
| `random_page_cost` | 4.0 | 随机扫描页面的成本 |
| `cpu_tuple_cost` | 0.01 | 处理每行的 CPU 成本 |
| `cpu_index_tuple_cost` | 0.005 | 处理索引行的 CPU 成本 |
| `cpu_operator_cost` | 0.0025 | 执行运算符的 CPU 成本 |

### 成本计算示例

```sql
-- 假设：
-- 1. orders 表有 10000 行，每页 10 行，共 1000 页
-- 2. customer_id = 1 的行有 100 行（选择性 1%）
-- 3. random_page_cost = 4.0

-- 全表扫描成本
Seq Scan cost = 1000 页 * 1.0 = 1000

-- 索引扫描成本
Index Scan cost =
    索引扫描：100 行需要扫描 ~10 页 * 4.0 = 40
    + 回表：100 行 * 0.01 = 1
    = 41

-- 优化器会选择索引扫描（41 < 1000）
```

### 调整成本参数

```sql
-- SSD 环境，降低随机访问成本
ALTER SYSTEM SET random_page_cost = 1.1;
SELECT pg_reload_conf();

-- 为特定表设置统计信息
ALTER TABLE orders ALTER COLUMN status SET (n_distinct = 3);
```

## ANALYZE 输出详解

### ANALYZE 的额外信息

```sql
EXPLAIN (ANALYZE, BUFFERS, TIMING)
SELECT * FROM orders WHERE customer_id = 1;
```

```
Index Scan using idx_orders_customer on orders  (cost=0.42..8.44 rows=10 width=100)
                                    (actual time=0.021..0.035 rows=10 loops=1)
  Index Cond: (customer_id = 1)
  Buffers: shared hit=11
  I/O Timings: read=0.010 write=0.001
Planning Time: 0.5 ms
Execution Time: 0.8 ms
```

**关键信息**：
- `cost=0.42..8.44`：预估成本（启动成本..总成本）
- `actual time=0.021..0.035`：实际执行时间（启动时间..结束时间）
- `rows=10`：预估返回行数
- `loops=1`：执行次数
- `Buffers: shared hit=11`：缓存命中
- `I/O Timings`：I/O 时间

### 缓存命中分析

```
Buffers: shared hit=11 read=2

-- 含义：
-- hit=11：11 页来自共享缓冲区（缓存命中）
-- read=2：2 页需要从磁盘读取
```

**缓存命中率**：`11 / (11 + 2) = 85%`

### 识别性能问题

```sql
-- 识别慢查询
EXPLAIN (ANALYZE)
SELECT * FROM orders WHERE status = 'pending';

-- 问题：预估行数和实际行数差距大
-- 原因：统计信息过时
-- 解决：ANALYZE 表

-- 问题：执行时间远大于预估
-- 原因：缓存命中率低或等待锁
-- 解决：增加 shared_buffers 或检查锁等待

-- 问题：loops > 1
-- 原因：可能是 NestLoop 中的外层返回多行
-- 解决：优化驱动表或添加索引
```

## 常见优化场景

### 场景一：全表扫描 vs 索引扫描

```sql
-- 问题：查询走全表扫描
EXPLAIN SELECT * FROM orders WHERE status = 'pending';

-- 优化：添加索引
CREATE INDEX idx_orders_status ON orders(status);
CREATE INDEX idx_orders_status_covering ON orders(status) INCLUDE (id, customer_id, total);

-- 再次检查
EXPLAIN SELECT * FROM orders WHERE status = 'pending';
```

### 场景二：嵌套循环的优化

```sql
-- 问题：嵌套循环使用了全表扫描作为内层
EXPLAIN SELECT * FROM orders o
JOIN customers c ON o.customer_id = c.id
WHERE c.region = 'North';

-- 查看索引
-- idx_customers_region 存在，但 customer_id 可能没有索引

-- 优化：添加 customer_id 索引
CREATE INDEX idx_customers_id ON customers(id);

-- 再次检查
EXPLAIN SELECT * FROM orders o
JOIN customers c ON o.customer_id = c.id
WHERE c.region = 'North';
```

### 场景三：大表 JOIN

```sql
-- 问题：Hash Join 溢出到磁盘
EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM orders o
JOIN order_items oi ON o.id = oi.order_id;

-- 优化 1：增加 work_mem
SET work_mem = '256MB';

-- 优化 2：减少需要 JOIN 的数据
EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM orders o
JOIN order_items oi ON o.id = oi.order_id
WHERE o.created_at > '2024-01-01';
```

## 总结

EXPLAIN 是 PostgreSQL 性能优化的核心工具：

1. **扫描类型**：Seq Scan、Index Scan、Index Only Scan、Bitmap Heap Scan
2. **JOIN 类型**：Nested Loop、Hash Join、Merge Join
3. **成本模型**：基于 seq_page_cost、random_page_cost、cpu_tuple_cost 等参数
4. **ANALYZE 输出**：实际执行时间、缓存命中率、执行次数

理解执行计划，能够帮助你：
- 识别性能瓶颈
- 判断索引是否生效
- 优化 JOIN 策略
- 调整成本参数

性能优化的关键是找到执行计划与实际执行的差距，然后针对性地调整。
