# PostgreSQL 慢查询分析

"数据库突然变慢了，怎么定位问题 SQL？"

慢查询是数据库性能问题的常见原因。PostgreSQL 提供了丰富的工具来发现、分析和优化慢查询。

---

## 发现慢查询

### 1. pg_stat_statements

pg_stat_statements 是分析慢查询最重要的扩展：

```sql
-- 安装扩展
CREATE EXTENSION pg_stat_statements;

-- 查看最慢的查询（按总执行时间排序）
SELECT
    query,
    calls,
    total_exec_time,
    mean_exec_time,
    rows,
    100.0 * shared_blks_hit / NULLIF(shared_blks_hit + shared_blks_read, 0) AS hit_percent
FROM pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 10;
```

### 2. pg_stat_activity

```sql
-- 查看正在运行的查询
SELECT
    pid,
    usename,
    query,
    state,
    query_start,
    NOW() - query_start AS duration,
    wait_event_type,
    wait_event
FROM pg_stat_activity
WHERE state != 'idle'
  AND query_start < NOW() - INTERVAL '5 minutes'
ORDER BY query_start;
```

### 3. 日志记录

```sql
-- postgresql.conf 配置
-- 记录慢查询（超过 100ms）
```

## 分析慢查询

### 1. 执行时间分析

```sql
-- 分析单个查询
EXPLAIN (ANALYZE, BUFFERS, TIMING)
SELECT * FROM orders
WHERE customer_id = 1
  AND created_at > NOW() - INTERVAL '7 days'
ORDER BY created_at DESC;
```

### 2. 等待事件分析

```sql
-- 查看等待事件
SELECT
    pid,
    wait_event_type,
    wait_event,
    query
FROM pg_stat_activity
WHERE wait_event IS NOT NULL;

-- 常见的等待事件：
-- Lock：等待锁
-- LWLock：等待轻量锁
-- IO：等待 I/O
-- BufferPin：等待缓冲区
```

### 3. 索引使用分析

```sql
-- 查看索引使用情况
SELECT
    schemaname,
    relname,
    indexrelname,
    idx_scan,
    idx_tup_read,
    idx_tup_fetch,
    pg_size_pretty(pg_relation_size(indexrelid)) AS index_size
FROM pg_stat_user_indexes
WHERE schemaname = 'public'
ORDER BY idx_scan;
```

## 优化慢查询

### 优化一：添加索引

```sql
-- 发现慢查询
EXPLAIN (ANALYZE) SELECT * FROM orders
WHERE status = 'pending' AND created_at > '2024-01-01';

-- 添加索引
CREATE INDEX idx_orders_status_created ON orders(status, created_at);

-- 再次分析
EXPLAIN (ANALYZE) SELECT * FROM orders
WHERE status = 'pending' AND created_at > '2024-01-01';
```

### 优化二：使用覆盖索引

```sql
-- 创建覆盖索引
CREATE INDEX idx_orders_status_created_covering
    ON orders(status, created_at)
    INCLUDE (customer_id, total);

-- 查询不需要回表
SELECT customer_id, total
FROM orders
WHERE status = 'pending' AND created_at > '2024-01-01';
```

### 优化三：优化 JOIN

```sql
-- 发现 N+1 问题
-- 原始查询
SELECT * FROM orders WHERE customer_id = 1;

-- 优化：批量查询
SELECT o.*, c.name as customer_name
FROM orders o
JOIN customers c ON o.customer_id = c.id
WHERE o.customer_id IN (1, 2, 3, 4, 5);
```

### 优化四：重写查询

```sql
-- 原始：使用子查询
SELECT * FROM orders
WHERE customer_id IN (SELECT id FROM customers WHERE region = 'North');

-- 优化：使用 JOIN
SELECT o.*
FROM orders o
JOIN customers c ON o.customer_id = c.id
WHERE c.region = 'North';

-- 优化：使用 CTE
WITH north_customers AS (
    SELECT id FROM customers WHERE region = 'North'
)
SELECT o.*
FROM orders o
WHERE o.customer_id IN (SELECT id FROM north_customers);
```

## 慢查询优化案例

### 案例一：分页查询慢

```sql
-- 问题：OFFSET 很大时很慢
SELECT * FROM orders
ORDER BY created_at DESC
LIMIT 20 OFFSET 10000;

-- 优化：使用游标分页
SELECT * FROM orders
WHERE (created_at, id) < ($1, $2)
ORDER BY created_at DESC, id DESC
LIMIT 20;
```

### 案例二：聚合查询慢

```sql
-- 问题：全表聚合
SELECT status, COUNT(*), SUM(total)
FROM orders
GROUP BY status;

-- 优化：使用物化视图
CREATE MATERIALIZED VIEW orders_agg AS
SELECT status, COUNT(*) as cnt, SUM(total) as total_sum
FROM orders
GROUP BY status;

CREATE UNIQUE INDEX ON orders_agg(status);

-- 定期刷新
REFRESH MATERIALIZED VIEW orders_agg;
```

### 案例三：模糊查询慢

```sql
-- 问题：LIKE 模糊匹配
SELECT * FROM products WHERE name LIKE '%手机%';

-- 优化：使用全文索引
CREATE INDEX idx_products_name ON products USING GIN(to_tsvector('chinese', name));

SELECT * FROM products
WHERE to_tsvector('chinese', name) @@ to_tsquery('chinese', '手机');
```

## 总结

慢查询分析的关键步骤：

1. **发现**：使用 pg_stat_statements、pg_stat_activity、日志
2. **分析**：执行 EXPLAIN ANALYZE、检查等待事件
3. **优化**：添加索引、优化 JOIN、重写查询
4. **验证**：再次执行 EXPLAIN ANALYZE 确认改善

pg_stat_statements 是最重要的工具，配置 log_min_duration_statement 可以持续记录慢查询。
