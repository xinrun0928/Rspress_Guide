# PostgreSQL 并行查询：多核利用与性能优化

你的服务器有 64 个 CPU 核心，但 PostgreSQL 查询只用了 1 个——剩下的 63 个在睡觉。

这是很多 DBA 都会遇到的问题：**单机数据库性能遇到瓶颈，硬件资源利用率却低得可怜**。不是因为 PostgreSQL 不行，而是因为你没有开启或正确配置并行查询。

今天，我们来聊聊 PostgreSQL 的并行查询——如何让查询利用所有 CPU 核心，把查询速度从几分钟缩短到几秒。

## 并行查询原理

### 为什么需要并行查询

传统数据库查询是**单线程**的：

```
单线程查询：
┌─────────────────────────────────────────┐
│  查询请求                                │
│       │                                 │
│       ▼                                 │
│  ┌────────────────────────────────┐    │
│  │      PostgreSQL Backend        │    │
│  │                                │    │
│  │   CPU: ████████████░░░░ 1核   │    │
│  │   64 核中只用了 1 核           │    │
│  └────────────────────────────────┘    │
└─────────────────────────────────────────┘
```

并行查询将查询分解为多个子任务，由多个 worker 并行执行：

```
并行查询：
┌─────────────────────────────────────────┐
│  查询请求                                │
│       │                                 │
│       ▼                                 │
│  ┌────────────────────────────────┐    │
│  │      Parallel Query Coordinator │    │
│  │              │                 │    │
│  │    ┌─────────┼─────────┐       │    │
│  │    ▼         ▼         ▼       │    │
│  │  Worker1  Worker2  Worker3      │    │
│  │    │         │         │        │    │
│  │   CPU1     CPU2      CPU3      │    │
│  │                                │    │
│  │   CPU: ██████████████████████  │    │
│  │   64 核全部利用                  │    │
│  └────────────────────────────────┘    │
└─────────────────────────────────────────┘
```

### PostgreSQL 并行查询架构

```
┌─────────────────────────────────────────────────────────────────┐
│                    PostgreSQL 并行查询架构                        │
│                                                                 │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │                    Coordinator Process                    │  │
│  │  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐      │  │
│  │  │ Gather Node │  │  Planning   │  │  Results    │      │  │
│  │  └──────┬──────┘  │             │  │   Merge     │      │  │
│  └─────────┼──────────┴─────────────┴─────────────┘──────────┘  │
│            │                                                        │
│  ┌─────────┼─────────────────────────────────────────┐           │
│  │         ▼                                         │           │
│  │  ┌───────────┐  ┌───────────┐  ┌───────────┐      │           │
│  │  │  Worker 1 │  │  Worker 2 │  │  Worker N │      │           │
│  │  │ (Subplan) │  │ (Subplan) │  │ (Subplan) │      │           │
│  │  └───────────┘  └───────────┘  └───────────┘      │           │
│  │           Parallel Workers                          │           │
│  └─────────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
```

**关键组件**：

| 组件 | 说明 |
|---|---|
| Coordinator | 主进程，负责协调和汇总结果 |
| Parallel Worker | 子进程，执行并行子任务 |
| Gather | 收集各 Worker 结果的节点 |
| Parallel Seq Scan | 并行顺序扫描 |
| Parallel Hash Join | 并行哈希连接 |

## 并行扫描类型

### Parallel Seq Scan：并行全表扫描

最基础的并行扫描，当没有合适索引时使用：

```sql
-- 查看查询计划
EXPLAIN SELECT * FROM orders WHERE amount > 100;

-- 输出示例（8 workers）
Gather  (cost=1000.00..100000.00 rows=50000 width=40)
  Workers Planned: 8
  ->  Parallel Seq Scan on orders  (cost=...)
        Filter: (amount > 100)

-- 分析执行时间
EXPLAIN (ANALYZE, BUFFERS) SELECT * FROM orders WHERE amount > 100;
```

**执行流程**：

```
Table: orders (1GB)
┌─────────────────────────────────────────────────────────┐
│  Worker 1: Scan rows 1-12500                             │
│  Worker 2: Scan rows 12501-25000                        │
│  Worker 3: Scan rows 25001-37500                        │
│  Worker 4: Scan rows 37501-50000                        │
│  Worker 5: Scan rows 50001-62500                        │
│  Worker 6: Scan rows 62501-75000                        │
│  Worker 7: Scan rows 75001-87500                        │
│  Worker 8: Scan rows 87501-100000                       │
└─────────────────────────────────────────────────────────┘
```

### Parallel Index Scan：并行索引扫描

```sql
-- 创建索引
CREATE INDEX idx_orders_user_id ON orders(user_id);

-- 查看并行索引扫描
EXPLAIN SELECT * FROM orders WHERE user_id = 123;

-- 输出示例
Index Scan using idx_orders_user_id on orders
  Index Cond: (user_id = 123)
  Parallel Workers: 4
```

### Parallel Bitmap Heap Scan：并行位图扫描

```sql
-- 创建索引
CREATE INDEX idx_orders_status ON orders(status);

-- 位图扫描
EXPLAIN SELECT * FROM orders WHERE status = 'completed' AND amount > 500;

-- 输出示例
Gather  (cost=1000.00..50000.00 rows=5000 width=40)
  Workers Planned: 4
  ->  Parallel Bitmap Heap Scan on orders
        Recheck Cond: ((status = 'completed') AND (amount > 500))
        ->  Bitmap Index Scan on idx_orders_status
              Index Cond: (status = 'completed')
```

## 并行连接

### Parallel Hash Join：并行哈希连接

```sql
-- 查看并行哈希连接
EXPLAIN SELECT *
FROM orders o
JOIN customers c ON o.customer_id = c.id
WHERE o.amount > 100;

-- 输出示例
Gather  (cost=1000.00..80000.00 rows=10000 width=60)
  Workers Planned: 4
  ->  Parallel Hash Join  (cost=...)
        Hash Cond: (o.customer_id = c.id)
        ->  Parallel Seq Scan on orders o
              Filter: (amount > 100)
        ->  Parallel Hash  (cost=...)
              ->  Parallel Seq Scan on customers c
```

### Parallel Nested Loop：并行嵌套循环

```sql
-- 小表驱动大表，嵌套循环效率高
EXPLAIN SELECT *
FROM customers c
JOIN orders o ON c.id = o.customer_id
WHERE c.region = '北京';

-- 输出示例
Gather
  Workers Planned: 4
  ->  Parallel Hash Join
        ->  Parallel Seq Scan on orders
        ->  Parallel Hash
              ->  Parallel Seq Scan on customers
                    Filter: (region = '北京')
```

## 参数配置

### 核心并行参数

```sql
-- postgresql.conf

-- 最大并行 worker 数（单个查询）
max_parallel_workers_per_gather = 4

-- 全局最大并行 worker 数
max_worker_processes = 32

-- 最大并行 worker 进程数
max_parallel_workers = 16

-- 维护操作的并行度（VACUUM, CREATE INDEX 等）
max_parallel_maintenance_workers = 4
```

### 并行度配置建议

| 场景 | `max_parallel_workers_per_gather` | 说明 |
|---|---|---|
| 小型服务器（4 核以下）| 2 | 保留资源给系统 |
| 中型服务器（8-16 核）| 4 | 平衡并行度和资源 |
| 大型服务器（32+ 核）| 8 | 追求极致性能 |
| 混合负载 | 2 | 避免并行查询抢占资源 |

### 控制查询并行度

```sql
-- 禁用单个查询的并行（调试用）
SELECT /*+ NO_PARALLEL */ * FROM orders WHERE amount > 100;

-- 强制使用指定并行度
SET max_parallel_workers_per_gather = 8;

-- 表级别禁用并行
ALTER TABLE orders SET (parallel_workers = 0);

-- 表级别设置并行度
ALTER TABLE orders SET (parallel_workers = 4);

-- 会话级别设置
SET LOCAL max_parallel_workers_per_gather = 2;
```

## 并行查询优化技巧

### 1. 优化数据分布

```sql
-- 使用 PARTITION 将大表分区
-- 每个分区可以独立并行扫描

CREATE TABLE orders (
    id SERIAL,
    customer_id INT,
    amount DECIMAL(10,2),
    created_at DATE
) PARTITION BY RANGE (created_at);

CREATE TABLE orders_2024_q1 PARTITION OF orders
    FOR VALUES FROM ('2024-01-01') TO ('2024-04-01');

-- 分区后，并行扫描可以跳过不需要的分区
```

### 2. 减少数据传输量

```sql
-- 只选择需要的列
SELECT order_id, amount FROM orders;  -- 比 SELECT * 快很多

-- 使用 LIMIT
SELECT * FROM orders LIMIT 100;  -- 并行度会降低，但总体更快

-- 添加 WHERE 条件过滤
SELECT * FROM orders WHERE status = 'completed';  -- 减少处理数据量
```

### 3. 合理创建索引

```sql
-- 为高频查询创建索引
CREATE INDEX idx_orders_composite ON orders(customer_id, status, amount);

-- 创建覆盖索引避免回表
CREATE INDEX idx_orders_covering ON orders(customer_id)
    INCLUDE (order_id, amount, created_at);
```

### 4. 监控并行查询

```sql
-- 查看当前并行查询
SELECT pid, state, query
FROM pg_stat_activity
WHERE state LIKE 'active%'
  AND query LIKE '%Gather%';

-- 查看并行 worker 统计
SELECT * FROM pg_stat_activity
WHERE backend_type = 'parallel worker';

-- 查看并行查询执行统计
SELECT query,
       calls,
       total_exec_time / 1000 as total_seconds,
       mean_exec_time as avg_ms,
       stddev_exec_time as stddev_ms
FROM pg_stat_statements
WHERE query LIKE '%Gather%'
ORDER BY total_exec_time DESC;
```

## 常见问题与解决

### 问题一：并行查询不生效

```sql
-- 检查参数
SHOW max_parallel_workers_per_gather;  -- 默认 2
SHOW max_worker_processes;              -- 默认 8

-- 检查表统计信息
ANALYZE orders;

-- 检查小表是否被优化跳过并行
-- PostgreSQL 可能认为小表不值得并行

-- 查看实际执行计划
EXPLAIN (ANALYZE, BUFFERS) SELECT * FROM orders;
```

### 问题二：并行度太低

```sql
-- 增加并行度
SET max_parallel_workers_per_gather = 8;
SET max_parallel_workers = 16;

-- 检查系统资源
-- 确保有足够的内存和 CPU
-- PostgreSQL 可能因为资源不足降低并行度
```

### 问题三：并行查询导致 OOM

```sql
-- 减少 work_mem
SET work_mem = '256MB';

-- 减少并行 worker 数
SET max_parallel_workers_per_gather = 2;

-- 增加维护工作内存
SET maintenance_work_mem = '1GB';

-- 增加可用内存
-- 确保 PostgreSQL 有足够内存运行并行查询
```

### 问题四：并行 VACUUM 不生效

```sql
-- 并行 VACUUM 需要 PG 12+
-- 确保版本支持
SHOW server_version;

-- 设置维护并行度
SET max_parallel_maintenance_workers = 2;

-- 执行并行 VACUUM
VACUUM (PARALLEL 2) orders;

-- 创建索引时启用并行
CREATE INDEX CONCURRENTLY idx_orders_user_id ON orders(user_id);
```

## 性能对比示例

### 场景：聚合查询性能

```sql
-- 创建测试表
CREATE TABLE sales (
    id SERIAL,
    product_id INT,
    region VARCHAR(50),
    amount DECIMAL(10,2),
    sale_date DATE
);

-- 插入 1000 万条测试数据
INSERT INTO sales (product_id, region, amount, sale_date)
SELECT 
    (random() * 1000)::INT,
    (ARRAY['北京', '上海', '广州', '深圳'])[floor(random() * 4 + 1)],
    (random() * 1000)::DECIMAL(10,2),
    CURRENT_DATE - (random() * 365)::INT
FROM generate_series(1, 10000000);

-- 禁用并行查询
SET max_parallel_workers_per_gather = 0;
EXPLAIN (ANALYZE) 
SELECT region, SUM(amount), COUNT(*)
FROM sales
GROUP BY region;

-- 启用并行查询（4 workers）
SET max_parallel_workers_per_gather = 4;
EXPLAIN (ANALYZE)
SELECT region, SUM(amount), COUNT(*)
FROM sales
GROUP BY region;

-- 性能对比（示例）
-- 禁用并行：Execution Time: 4500 ms
-- 启用并行：Execution Time: 1200 ms
-- 提升：约 3.7 倍
```

## 总结

PostgreSQL 的并行查询是提升性能的神器：

- **Parallel Seq Scan**：并行全表扫描，适合无索引查询
- **Parallel Hash Join**：并行哈希连接，适合大表关联
- **参数调优**：`max_parallel_workers_per_gather` 是关键参数
- **监控优化**：通过 `EXPLAIN ANALYZE` 持续优化

合理配置并行查询，可以让 PostgreSQL 在多核服务器上发挥出极致性能——**从几分钟到几秒**，就是并行查询的威力。
