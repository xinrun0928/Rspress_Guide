# PostgreSQL 连接池

PostgreSQL 使用进程模型，每个连接对应一个进程。在高并发场景下，连接创建和销毁的开销会成为性能瓶颈。连接池是解决这个问题的标准方案。

## PostgreSQL 连接开销

### 进程模型 vs 线程模型

| 模型 | 数据库 | 特点 |
|---|---|---|
| 进程模型 | PostgreSQL | 每个连接一个进程，稳定但开销大 |
| 线程模型 | MySQL InnoDB | 每个连接一个线程，开销小 |

### 连接开销

```sql
-- 查看当前连接数
SELECT count(*) FROM pg_stat_activity;

-- 查看最大连接数
SHOW max_connections;
```

```sql
-- 连接开销估算：
-- 1. 进程创建：~10-50ms
-- 2. 内存分配：~5-10MB/连接
-- 3. 认证：~1-5ms
-- 4. 查询解析：~1-10ms
```

### 连接池的作用

```
应用                连接池                 PostgreSQL
  |                   |                       |
  |--- 获取连接 ------>|                       |
  |<-- 连接引用 -------|                       |
  |                   |-- 复用连接 --------->|
  |--- 执行查询 ------>|                       |
  |<-- 返回结果 -------|                       |
  |                   |                       |
  |-- 释放连接 ------>|                       |
  |<-- 放回池中 -------|                       |
```

## PgBouncer：最常用的连接池

### 安装和配置

```ini
; pgbouncer.ini
[databases]
mydb = host=127.0.0.1 port=5432 dbname=mydb

[pgbouncer]
listen_addr = 127.0.0.1
listen_port = 6432
auth_type = md5
auth_file = userlist.txt
pool_mode = transaction
max_client_conn = 1000
default_pool_size = 20
min_pool_size = 5
reserve_pool_size = 5
reserve_pool_timeout = 3
server_idle_timeout = 600
log_connections = 0
log_disconnections = 0
log_pooler_errors = 1
```

### 启动 PgBouncer

```bash
# 启动
pgbouncer -d pgbouncer.ini

# 或使用 systemd
sudo systemctl start pgbouncer
```

### 配置详解

| 参数 | 说明 |
|---|---|
| `pool_mode` | 连接池模式（session/transaction/statement） |
| `max_client_conn` | 最大客户端连接数 |
| `default_pool_size` | 每个数据库/用户组合的默认连接池大小 |
| `min_pool_size` | 保活连接数 |
| `reserve_pool_size` | 备用连接数 |
| `reserve_pool_timeout` | 等待备用连接的超时时间 |
| `server_idle_timeout` | 服务端空闲连接超时 |

## 连接池模式

### Session 模式（PgBouncer 默认）

```ini
pool_mode = session
```

- 客户端获得连接后，连接一直保持，直到客户端断开
- 支持 PREPARED STATEMENTS
- 支持 SET 命令
- 不支持事务内的连接复用

```sql
-- Session 模式下支持
SET statement_timeout = '5s';
PREPARE my_plan AS SELECT * FROM orders WHERE id = $1;
```

### Transaction 模式（推荐）

```ini
pool_mode = transaction
```

- 每个事务获取一个连接，事务结束后释放
- 支持高并发
- 不支持会话级特性

```sql
-- 支持
BEGIN;
SELECT * FROM orders WHERE id = 1;
INSERT INTO orders VALUES (2, 'test');
COMMIT;

-- 不支持（会在下一个事务前报错）
SET statement_timeout = '5s';
```

### Statement 模式

```ini
pool_mode = statement
```

- 每条语句获取一个连接，执行后立即释放
- 不支持多语句事务
- 适用于极端高并发场景

## 应用配置

### 使用连接池

```python
# Python: psycopg2 + pgbouncer
import psycopg2

# 连接到 PgBouncer（而不是直接连接 PostgreSQL）
conn = psycopg2.connect(
    host='127.0.0.1',
    port=6432,
    database='mydb',
    user='myuser',
    password='mypassword'
)
```

### 连接字符串

```
# 直接连接 PostgreSQL
postgresql://user:pass@localhost:5432/mydb

# 通过 PgBouncer 连接
postgresql://user:pass@localhost:6432/mydb
```

## PgBouncer 管理

### 管理命令

```bash
# 连接 PgBouncer 管理接口
psql -h 127.0.0.1 -p 6432 -U pgbouncer pgbouncer

# 查看统计
SHOW STATS;

# 查看池
SHOW POOLS;

# 查看客户端
SHOW CLIENTS;

# 查看服务器
SHOW SERVERS;

# 查看配置
SHOW CONFIG;

# 重新加载配置
RELOAD;

# 暂停
PAUSE;
RESUME;
```

### 监控

```sql
-- 查看活动连接
SHOW LISTS;

-- 查看等待的客户端
SELECT * FROM pgbouncer.pools;
```

## 常见问题

### 问题一：连接泄漏

```sql
-- 症状：连接数不断增长
-- 原因：应用没有正确释放连接

-- 解决：检查应用的连接管理
-- 确保 finally/with 语句正确使用
```

### 问题二：事务超时

```sql
-- 症状：连接池耗尽
-- 原因：长事务占用连接

-- 解决：设置超时
SET statement_timeout = '30s';
SET idle_in_transaction_session_timeout = '5min';
```

## 其他连接池方案

### Pgpool-II

- 支持连接池、负载均衡、缓存
- 功能更丰富，但配置更复杂

```ini
; pgpool.conf
listen_addresses = '*'
port = 9999
backend_hostname0 = '127.0.0.1'
backend_port0 = 5432
backend_weight0 = 1
```

### ODBC/JDBC 连接池

```java
// HikariCP (Java)
HikariConfig config = new HikariConfig();
config.setJdbcUrl("jdbc:postgresql://127.0.0.1:6432/mydb");
config.setUsername("user");
config.setPassword("pass");
config.setMaximumPoolSize(20);
```

## 总结

PostgreSQL 的进程模型使得连接开销较高，连接池是高并发场景的必备组件：

1. **PgBouncer**：轻量、功能完善的连接池
2. **模式选择**：Transaction 模式是大多数场景的最佳选择
3. **参数调优**：根据并发量调整 pool_size
4. **监控**：定期监控连接池使用情况

连接池配合合理的 `max_connections` 配置，可以让 PostgreSQL 支撑高并发应用。
