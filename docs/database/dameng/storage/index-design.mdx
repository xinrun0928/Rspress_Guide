# 索引设计原则

<Badge text="进阶" type="tip" />

## 索引设计概述

索引设计是数据库性能优化的核心工作。好的索引设计可以让查询性能提升百倍；糟糕的索引设计不仅浪费空间，还可能拖累整体性能。

你可能遇到过这种情况：

- 数据库运行越来越慢
- 明明加了索引，查询还是很慢
- 数据量不大，但查询却很慢

这些问题的答案，往往都在索引设计里。

## 索引设计原则

### 核心原则

索引设计遵循以下核心原则：

| 原则 | 说明 | 重要性 |
|---|---|---|
| 选择性原则 | 选择区分度高的列建索引 | 高 |
| 覆盖原则 | 索引覆盖查询所需列 | 高 |
| 前缀原则 | 组合索引列顺序决定使用方式 | 高 |
| 适量原则 | 索引数量要合理 | 中 |
| 动态原则 | 根据实际负载调整索引 | 中 |

### 选择性原则

选择性是指列中不重复值的比例。选择性越高，索引越有效。

```
选择性与索引效率：

表 employees: 10000 条记录

场景 1: gender 列（2 种取值）
  选择性 = 2 / 10000 = 0.02%
  索引效率低，每个值对应 5000 条数据

场景 2: emp_code 列（10000 种取值）
  选择性 = 10000 / 10000 = 100%
  索引效率高，每个值对应 1 条数据

场景 3: phone 列（假设唯一）
  选择性 = 100%
  索引效率最高
```

```sql
-- 计算列的选择性
SELECT 
    column_name,
    COUNT(DISTINCT column_name) AS distinct_count,
    COUNT(*) AS total_count,
    ROUND(COUNT(DISTINCT column_name) * 100.0 / COUNT(*), 2) AS selectivity_pct
FROM employees
GROUP BY column_name
ORDER BY selectivity_pct DESC;

-- 识别低选择性列
SELECT column_name
FROM user_tab_columns
WHERE table_name = 'EMPLOYEES'
  AND NUM_DISTINCT < 10;  -- 假设小于 10 种取值的列
```

### 覆盖原则

覆盖索引是指索引包含查询所需的所有列，避免回表查询。

```
覆盖索引原理：

普通索引查询：
  SELECT name, salary FROM employees WHERE emp_code = 'E001'
                          ↓
                    idx_emp_code (emp_code)
                          ↓
                       找到 emp_code='E001'
                          ↓
                    回表查询数据页
                          ↓
                    获取 name, salary

覆盖索引查询：
  CREATE INDEX idx_cover ON employees(emp_code, name, salary)
                          ↓
  SELECT name, salary FROM employees WHERE emp_code = 'E001'
                          ↓
                    idx_cover (emp_code, name, salary)
                          ↓
                    直接返回 name, salary
                          ↓
                    无需回表，性能提升
```

```sql
-- 创建覆盖索引
CREATE INDEX idx_emp_cover ON employees(emp_code, name, salary);

-- 查询可以直接使用覆盖索引
SELECT name, salary FROM employees WHERE emp_code = 'E001';

-- 执行计划验证
EXPLAIN SELECT name, salary FROM employees WHERE emp_code = 'E001';
-- 应该看到 INDEX FAST FULL SCAN 或 INDEX RANGE SCAN
```

### 前缀原则

组合索引的列顺序至关重要。MySQL 和达梦都遵循最左前缀原则。

```
组合索引列顺序示例：

索引: (A, B, C)

查询可以使用索引的情况：
  WHERE A = ?          ✓ 使用列 A
  WHERE A = ? AND B = ?    ✓ 使用列 A, B
  WHERE A = ? AND B = ? AND C = ?  ✓ 使用列 A, B, C

查询不能使用索引的情况：
  WHERE B = ?          ✗ 不使用索引
  WHERE C = ?          ✗ 不使用索引
  WHERE B = ? AND C = ?    ✗ 不使用索引
```

```sql
-- 组合索引设计
-- 场景：经常按部门查询员工的姓名和工资
CREATE INDEX idx_dept_emp ON employees(dept_id, name, salary);

-- 这个查询可以使用索引
SELECT name, salary 
FROM employees 
WHERE dept_id = 10;

-- 这个查询可以使用索引
SELECT name, salary 
FROM employees 
WHERE dept_id = 10 AND name LIKE '张%';

-- 这个查询无法使用索引（跳过了 dept_id）
SELECT name, salary 
FROM employees 
WHERE name LIKE '张%';
```

## 组合索引设计

### 列顺序确定方法

组合索引的列顺序应该遵循以下原则：

| 原则 | 说明 | 优先级 |
|---|---|---|
| 等值列优先 | = 条件的列放在前面 | 最高 |
| 区分度高的列优先 | 选择性高的列放在前面 | 高 |
| 少量值列优先 | 基数低的列放在前面 | 中 |

```sql
-- 场景分析
-- 查询: SELECT * FROM orders WHERE status = '已完成' AND customer_id = 1001

-- 分析列特征
-- status: 只有 5 种取值（待支付、已支付、已发货、已收货、已完成）
-- customer_id: 有 10000 个客户

-- 正确的列顺序：customer_id 在前
CREATE INDEX idx_order_lookup ON orders(customer_id, status);

-- 查询执行过程：
-- 1. 定位 customer_id = 1001 的所有记录
-- 2. 在结果中过滤 status = '已完成'
-- 3. 只需扫描 customer_id=1001 的少量记录
```

### 常见组合索引模式

```sql
-- 模式 1: 主键 + 常用查询条件
CREATE INDEX idx_pk_where ON table_name(primary_key, often_used_column);

-- 模式 2: 外键 + 排序
CREATE INDEX idx_fk_sort ON table_name(foreign_key, order_column);

-- 模式 3: 状态 + 时间
CREATE INDEX idx_status_time ON orders(order_status, order_date DESC);

-- 模式 4: 用户 + 时间 + 业务
CREATE INDEX idx_user_time_biz ON logs(user_id, log_time DESC, log_type);
```

### 索引下推

达梦支持索引下推优化，减少回表次数。

```
索引下推原理：

查询: SELECT * FROM orders WHERE customer_id = 1001 AND status = '已发货'

无索引下推：
  1. 使用 customer_id 定位到记录
  2. 回表获取整行数据
  3. 在服务器端过滤 status

有索引下推：
  1. 使用索引 (customer_id, status) 
  2. 在索引层面过滤 status = '已发货'
  3. 只回表获取符合条件的记录
  4. 减少回表次数
```

## 索引优化技巧

### 避免索引失效

```sql
-- 避免在索引列上使用函数
-- 错误写法：
SELECT * FROM employees WHERE UPPER(name) = 'ZHANG';

-- 正确写法（创建函数索引）：
CREATE INDEX idx_upper_name ON employees(UPPER(name));
SELECT * FROM employees WHERE UPPER(name) = 'ZHANG';

-- 避免隐式类型转换
-- 假设 phone 是 VARCHAR 类型
-- 错误写法（隐式转换）：
SELECT * FROM users WHERE phone = 13800138000;

-- 正确写法：
SELECT * FROM users WHERE phone = '13800138000';

-- 避免前导通配符
-- 错误写法：
SELECT * FROM employees WHERE name LIKE '%三%';

-- 正确写法（如果有全文索引）：
CREATE FULLTEXT INDEX idx_name ON employees(name);
SELECT * FROM employees WHERE MATCH(name) AGAINST('三');

-- 使用 IN 或 EXISTS 代替 NOT IN
-- 错误写法：
SELECT * FROM employees WHERE dept_id NOT IN (SELECT dept_id FROM closed_depts);

-- 正确写法：
SELECT * FROM employees e 
WHERE NOT EXISTS (SELECT 1 FROM closed_depts c WHERE c.dept_id = e.dept_id);
```

### 利用索引排序

```sql
-- 利用索引避免排序
CREATE INDEX idx_emp_dept_sal ON employees(dept_id, salary DESC);

-- 这个查询利用索引排序，无需额外排序
SELECT * FROM employees 
WHERE dept_id = 10 
ORDER BY salary DESC;

-- 注意索引列顺序和排序方向
-- 如果 ORDER BY salary ASC，需要创建 ASC 索引或单独索引
```

### 前缀索引

对于很长的 VARCHAR 列，可以使用前缀索引：

```sql
-- 长字符串列
-- email 长度可能超过 100 字符

-- 全列索引（占用空间大）
CREATE INDEX idx_email_full ON users(email);

-- 前缀索引（只索引前 50 个字符）
CREATE INDEX idx_email_prefix ON users(email(50));

-- 选择合适的前缀长度
SELECT 
    SUBSTR(email, 1, 30) AS prefix30,
    COUNT(DISTINCT email) / COUNT(*) AS selectivity_30
FROM users
GROUP BY SUBSTR(email, 1, 30)
ORDER BY selectivity_30 DESC
LIMIT 1;
```

## 索引分析与管理

### 监控索引使用

```sql
-- 查看索引使用统计
SELECT 
    index_name,
    table_name,
    b.level,
    b.leaf_blocks,
    b.distinct_keys,
    b.num_rows
FROM user_indexes i
JOIN user_tables t ON i.table_name = t.table_name
JOIN (
    SELECT index_name, AVG(blevel) AS level, 
           SUM(leaf_blocks) AS leaf_blocks,
           SUM(num_distinct) AS distinct_keys,
           SUM(num_rows) AS num_rows
    FROM user_ind_partitions
    GROUP BY index_name
) b ON i.index_name = b.index_name
WHERE i.table_name = 'EMPLOYEES';

-- 分析索引效率
SELECT 
    table_name,
    index_name,
    CASE 
        WHEN num_rows = 0 THEN 0
        ELSE ROUND(distinct_keys / num_rows * 100, 2)
    END AS selectivity
FROM user_indexes
WHERE table_name = 'EMPLOYEES';
```

### 识别未使用的索引

```sql
-- 查看索引使用情况
SELECT 
    object_name,
    object_type,
    executions,
    disk_reads,
    buffer_gets,
    rows_processed
FROM v$object_usage
WHERE object_type = 'INDEX';

-- 长期未使用的索引可以考虑删除
-- 但要确认不是周期性使用的索引
```

### 索引碎片管理

```sql
-- 查看索引碎片率
SELECT 
    index_name,
    table_name,
    blevel,
    leaf_blocks,
    num_rows,
    round((leaf_blocks * 8192) / 1024 / 1024, 2) AS leaf_size_mb
FROM user_indexes
WHERE table_name = 'EMPLOYEES';

-- 重建索引整理碎片
ALTER INDEX idx_emp_name REBUILD;

-- 合并碎片（比重建轻量）
ALTER INDEX idx_emp_name COALESCE;
```

## 索引设计案例

### 案例一：订单查询系统

```sql
-- 业务需求：
-- 1. 按订单号查询订单详情（等值查询）
-- 2. 按客户查询订单列表（等值 + 排序）
-- 3. 按日期范围查询订单（范围查询）
-- 4. 按客户和状态查询（组合查询）

-- 分析查询模式：
-- Q1: SELECT * FROM orders WHERE order_id = ?
-- Q2: SELECT * FROM orders WHERE customer_id = ? ORDER BY order_date DESC
-- Q3: SELECT * FROM orders WHERE order_date BETWEEN ? AND ?
-- Q4: SELECT * FROM orders WHERE customer_id = ? AND status = ?

-- 索引设计：
-- 1. 主键索引（自动）
-- 2. 客户+日期组合索引
CREATE INDEX idx_customer_date ON orders(customer_id, order_date DESC);

-- 3. 客户+状态组合索引
CREATE INDEX idx_customer_status ON orders(customer_id, status);

-- 4. 日期索引（范围查询）
CREATE INDEX idx_order_date ON orders(order_date);

-- 验证查询使用索引
EXPLAIN SELECT * FROM orders WHERE customer_id = 1001 ORDER BY order_date DESC;
```

### 案例二：用户登录系统

```sql
-- 业务需求：
-- 1. 用户名登录（等值查询，大小写不敏感）
-- 2. 手机号登录（等值查询）
-- 3. 邮箱登录（等值查询）

-- 索引设计：
-- 用户名大小写不敏感索引
CREATE INDEX idx_username_lower ON users(LOWER(username));

-- 手机号唯一索引
CREATE UNIQUE INDEX idx_phone ON users(phone);

-- 邮箱唯一索引
CREATE UNIQUE INDEX idx_email ON users(email);

-- 登录查询
SELECT * FROM users WHERE LOWER(username) = LOWER(:input);
SELECT * FROM users WHERE phone = :input;
SELECT * FROM users WHERE email = :input;
```

## 面试要点

<Badge text="面试重点" type="warning" />

### 设计原则

**Q：索引设计的核心原则是什么？**

索引设计的三条核心原则：

1. **选择性原则**：选择区分度高的列建索引
2. **覆盖原则**：尽量让索引覆盖查询所需列
3. **最左前缀原则**：组合索引要遵循列顺序

### 组合索引

**Q：组合索引的列顺序如何确定？**

列顺序确定方法：

1. 等值条件的列优先
2. 区分度高的列优先
3. 少量值列优先

例如：`WHERE dept_id = ? AND status = ? AND date `>=` ?`

- dept_id 和 status 是等值条件，优先
- 区分度：dept_id（有多个部门）`>=` status（有多种状态）
- 正确顺序：`(dept_id, status, date)`

### 索引失效

**Q：哪些情况会导致索引失效？**

常见索引失效情况：

1. 索引列上使用函数
2. 隐式类型转换
3. 前导通配符 LIKE '`%`xxx'
4. 使用 NOT IN、`<>`
5. 统计信息过期

## 总结

索引设计是数据库性能优化的核心：

- **选择性**：选择区分度高的列
- **覆盖性**：索引包含查询所需列
- **顺序性**：组合索引遵循最左前缀
- **适量性**：避免过多索引
- **动态性**：根据实际负载调整

好的索引设计需要结合业务查询模式，经过分析和验证才能确定。
