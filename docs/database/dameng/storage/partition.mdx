## 分区概述

当你的表数据量超过千万级别，全表扫描的性能会急剧下降。此时，你需要考虑对表进行分区。

分区是将一张大表物理上拆分成多个小表，逻辑上仍然是一张表的技术。分区后，查询只需要扫描相关的分区，而不是全表扫描。

想象一下：你的订单表有 10 年的数据，总量 5 亿条。如果按年分区，每年的数据约 5000 万条，查询某一年的数据时，只需要扫描一个分区，性能提升可达 10 倍。

这就是分区的威力。

## 分区类型总览

达梦数据库支持多种分区类型：

| 分区类型 | 英文名 | 划分依据 | 适用场景 |
|---|---|---|---|
| 范围分区 | Range Partition | 数值或日期范围 | 时间序列数据 |
| 列表分区 | List Partition | 枚举值列表 | 地域、类别数据 |
| 哈希分区 | Hash Partition | 哈希函数 | 数据均匀分布 |
| 组合分区 | Composite Partition | 多种分区组合 | 复杂业务场景 |


## 范围分区

### 范围分区的原理

范围分区按照分区键的值范围进行划分。典型的应用场景是按时间字段分区。

```sql
-- 按年份分区
CREATE TABLE sales (
    sale_id INT,
    sale_date DATE,
    customer_id INT,
    amount DECIMAL(12,2)
)
PARTITION BY RANGE(sale_date) (
    PARTITION p_2020 VALUES LESS THAN (DATE '2021-01-01'),
    PARTITION p_2021 VALUES LESS THAN (DATE '2022-01-01'),
    PARTITION p_2022 VALUES LESS THAN (DATE '2023-01-01'),
    PARTITION p_2023 VALUES LESS THAN (DATE '2024-01-01'),
    PARTITION p_2024 VALUES LESS THAN (DATE '2025-01-01'),
    PARTITION p_max VALUES LESS THAN (MAXVALUE)
);
```

### 范围分区的数据分布

```
范围分区数据分布示意：

sales 表 (按日期分区)

┌─────────────────────────────────────────────────────────┐
│                     sales 表                           │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐  │
│  │ p_2020   │ │ p_2021   │ │ p_2022   │ │ p_2023   │  │
│  │ 2020年   │ │ 2021年   │ │ 2022年   │ │ 2023年   │  │
│  │  2000万  │ │  3500万  │ │  4000万  │ │  4200万  │  │
│  └──────────┘ └──────────┘ └──────────┘ └──────────┘  │
│                                                         │
│  ┌──────────┐                                         │
│  │ p_2024   │  ← 新数据继续插入                        │
│  │  1500万  │                                         │
│  └──────────┘                                         │
│                                                         │
│  ┌──────────┐                                         │
│  │ p_max    │  ← 兜底分区                             │
│  │ 未来数据 │                                         │
│  └──────────┘                                         │
│                                                         │
└─────────────────────────────────────────────────────────┘

查询示例：
SELECT * FROM sales WHERE sale_date BETWEEN '2022-01-01' AND '2022-12-31'
→ 只扫描 p_2022 分区，性能提升明显
```

### 范围分区的应用场景

| 场景 | 分区键 | 说明 |
|---|---|---|
| 订单表 | 订单日期 | 按年月分区 |
| 日志表 | 创建时间 | 按天或月分区 |
| 交易表 | 交易时间 | 按月或季度分区 |
| 库存表 | 入库日期 | 按年分区 |

```sql
-- 订单表按月分区
CREATE TABLE orders (
    order_id INT,
    order_date DATETIME,
    customer_id INT,
    total_amount DECIMAL(12,2),
    status VARCHAR(20)
)
PARTITION BY RANGE(order_date) (
    PARTITION p_2024_01 VALUES LESS THAN (DATE '2024-02-01'),
    PARTITION p_2024_02 VALUES LESS THAN (DATE '2024-03-01'),
    PARTITION p_2024_03 VALUES LESS THAN (DATE '2024-04-01'),
    PARTITION p_2024_04 VALUES LESS THAN (DATE '2024-05-01'),
    PARTITION p_2024_05 VALUES LESS THAN (DATE '2024-06-01'),
    PARTITION p_2024_06 VALUES LESS THAN (DATE '2024-07-01'),
    -- ... 其他月份
    PARTITION p_future VALUES LESS THAN (MAXVALUE)
);

-- 日志表按天分区（适合大量日志）
CREATE TABLE access_logs (
    log_id BIGINT,
    log_time DATETIME,
    ip_address VARCHAR(50),
    url VARCHAR(500)
)
PARTITION BY RANGE(log_time) (
    PARTITION p_2024_01_01 VALUES LESS THAN (DATETIME '2024-01-02 00:00:00'),
    PARTITION p_2024_01_02 VALUES LESS THAN (DATETIME '2024-01-03 00:00:00'),
    PARTITION p_2024_01_03 VALUES LESS THAN (DATETIME '2024-01-04 00:00:00'),
    -- ... 其他日期
    PARTITION p_max VALUES LESS THAN (MAXVALUE)
);
```

## 列表分区

### 列表分区的原理

列表分区按照分区键的枚举值进行划分。每个分区对应一组特定的值。

```sql
-- 按地区列表分区
CREATE TABLE customers (
    customer_id INT,
    customer_name VARCHAR(100),
    region VARCHAR(20),
    register_date DATE
)
PARTITION BY LIST(region) (
    PARTITION p_beijing VALUES ('北京'),
    PARTITION p_shanghai VALUES ('上海'),
    PARTITION p_guangzhou VALUES ('广州', '深圳'),
    PARTITION p_west VALUES ('成都', '重庆', '西安', '昆明'),
    PARTITION p_other VALUES (DEFAULT)
);
```

### 列表分区的数据分布

```
列表分区数据分布示意：

customers 表 (按地区分区)

┌─────────────────────────────────────────────────────────┐
│                    customers 表                        │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐  │
│  │  p_beijing   │  │  p_shanghai  │  │ p_guangzhou  │  │
│  │    北京      │  │    上海      │  │ 广州、深圳   │  │
│  │    100万    │  │    120万    │  │   150万     │  │
│  └──────────────┘  └──────────────┘  └──────────────┘  │
│                                                         │
│  ┌──────────────┐  ┌──────────────┐                    │
│  │   p_west     │  │   p_other    │                    │
│  │  西部城市     │  │   其他地区   │                    │

│  └──────────────┘  └──────────────┘                    │

查询示例：
```

### 列表分区的应用场景

| 场景 | 分区键 | 说明 |
|---|---|---|
| 客户表 | 地区/省份 | 按地域分区 |
| 产品表 | 类别/品牌 | 按类别分区 |
| 订单表 | 订单类型 | 按类型分区 |
| 员工表 | 部门 | 按部门分区 |

```sql
-- 按产品类别分区
CREATE TABLE products (
    product_id INT,
    product_name VARCHAR(200),
    category VARCHAR(50),
    price DECIMAL(10,2)
)
PARTITION BY LIST(category) (
    PARTITION p_electronics VALUES ('手机', '电脑', '平板'),
    PARTITION p_appliances VALUES ('冰箱', '洗衣机', '空调'),
    PARTITION p_clothing VALUES ('男装', '女装', '童装'),
    PARTITION p_food VALUES ('零食', '饮料', '生鲜'),
    PARTITION p_other VALUES (DEFAULT)
);

-- 按订单状态分区（适合分析不同状态订单）
CREATE TABLE orders_by_status (
    order_id INT,
    order_date DATETIME,
    status VARCHAR(20),
    amount DECIMAL(12,2)
)
PARTITION BY LIST(status) (
    PARTITION p_pending VALUES ('待支付', '待发货'),
    PARTITION p_processing VALUES ('已支付', '已发货', '配送中'),
    PARTITION p_completed VALUES ('已收货', '已完成'),
    PARTITION p_cancelled VALUES ('已取消', '已退款')
);
```

## 哈希分区

### 哈希分区的原理

哈希分区使用哈希函数将数据均匀分布到各个分区。适合无法明显分区但需要分散 I/O 的场景。

```sql
-- 按客户 ID 哈希分区
CREATE TABLE transactions (
    transaction_id BIGINT,
    customer_id INT,
    transaction_date DATETIME,
    amount DECIMAL(12,2)
)
PARTITION BY HASH(customer_id) (
    PARTITION p_hash_0,
    PARTITION p_hash_1,
    PARTITION p_hash_2,
    PARTITION p_hash_3
);
```

### 哈希分区的数据分布

```
哈希分区数据分布示意：

transactions 表 (按 customer_id 哈希分区)

┌─────────────────────────────────────────────────────────┐
│                  transactions 表                        │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   Hash(customer_id) mod 4 = 0  →  p_hash_0             │
│   Hash(customer_id) mod 4 = 1  →  p_hash_1             │
│   Hash(customer_id) mod 4 = 2  →  p_hash_2             │
│   Hash(customer_id) mod 4 = 3  →  p_hash_3             │
│                                                         │
│  ┌────────────┐ ┌────────────┐ ┌────────────┐ ┌────────────┐  │
│  │ p_hash_0   │ │ p_hash_1   │ │ p_hash_2   │ │ p_hash_3   │  │
│  │ 均匀分布   │ │ 均匀分布   │ │ 均匀分布   │ │ 均匀分布   │  │
│  │ 约 2500万 │ │ 约 2500万 │ │ 约 2500万 │ │ 约 2500万 │  │
│  └────────────┘ └────────────┘ └────────────┘ └────────────┘  │
│                                                         │
└─────────────────────────────────────────────────────────┘

特点：
- 数据均匀分布
- 无法预知某条数据在哪个分区
- 适合分散热点数据
```

### 哈希分区的应用场景

| 场景 | 分区键 | 说明 |
|---|---|---|
| 大表随机访问 | 主键/唯一键 | 分散热点 |
| 并行处理 | 业务主键 | 并行 DML |
| 写入热点 | 序列值 | 分散竞争 |

```sql
-- 哈希分区适用于无法按业务分区的场景
-- 例如：产品表无法按时间或类别分区，但数据量很大

CREATE TABLE large_table (
    id BIGINT,
    data VARCHAR(1000)
)
PARTITION BY HASH(id) (
    PARTITION p_0,
    PARTITION p_1,
    PARTITION p_2,
    PARTITION p_3,
    PARTITION p_4,
    PARTITION p_5,
    PARTITION p_6,
    PARTITION p_7
);

-- 建议的分区数量：
-- 分区数 = CPU 核心数的 2-4 倍
-- 可以提高并行查询效率
```

## 组合分区

### 组合分区的原理

组合分区是两种分区方式的组合，先按一种方式分区，每个分区内部再按另一种方式分区。

```sql
-- 范围-列表组合分区
-- 先按年分区，再按地区列表分区
CREATE TABLE sales_composite (
    sale_id BIGINT,
    sale_date DATE,
    region VARCHAR(20),
    amount DECIMAL(12,2)
)
PARTITION BY RANGE(sale_date) SUBPARTITION BY LIST(region) (
    PARTITION p_2022 VALUES LESS THAN (DATE '2023-01-01') (
        SUBPARTITION p_2022_east VALUES ('北京', '上海', '广州', '深圳'),
        SUBPARTITION p_2022_west VALUES ('成都', '重庆', '西安'),
        SUBPARTITION p_2022_other VALUES (DEFAULT)
    ),
    PARTITION p_2023 VALUES LESS THAN (DATE '2024-01-01') (
        SUBPARTITION p_2023_east VALUES ('北京', '上海', '广州', '深圳'),
        SUBPARTITION p_2023_west VALUES ('成都', '重庆', '西安'),
        SUBPARTITION p_2023_other VALUES (DEFAULT)
    ),
    PARTITION p_2024 VALUES LESS THAN (DATE '2025-01-01') (
        SUBPARTITION p_2024_east VALUES ('北京', '上海', '广州', '深圳'),
        SUBPARTITION p_2024_west VALUES ('成都', '重庆', '西安'),
        SUBPARTITION p_2024_other VALUES (DEFAULT)
    )
);
```

### 组合分区的数据分布

```
组合分区数据分布示意：

sales_composite 表 (年-地区组合分区)

第一层：按年份范围分区
┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐
│  p_2022  │ │  p_2023  │ │  p_2024  │ │  p_max   │
└────┬─────┘ └────┬─────┘ └────┬─────┘ └────┬─────┘
     │            │            │            │
第二层：按地区列表分区
     ↓            ↓            ↓            ↓
┌────────┐ ┌────────┐  ┌────────┐ ┌────────┐  ┌────────┐ ┌────────┐
│东部分区│ │西部分区│  │东部分区│ │西部分区│  │东部分区│ │西部分区│
└────────┘ └────────┘  └────────┘ └────────┘  └────────┘ └────────┘

查询示例：
SELECT * FROM sales_composite 
WHERE sale_date BETWEEN '2023-01-01' AND '2023-12-31'
  AND region IN ('北京', '上海')
→ 只扫描 p_2023 分区的东部子分区
```

### 组合分区的应用场景

| 场景 | 外层分区 | 内层分区 | 说明 |
|---|---|---|---|
| 销售报表 | 年 | 月 | 按时间粒度分析 |
| 用户分析 | 年龄段 | 地区 | 多维度分析 |
| 订单管理 | 月 | 订单类型 | 按时间+类型 |

```sql
-- 案例：订单表按月+状态组合分区
CREATE TABLE orders_composite (
    order_id BIGINT,
    order_date DATETIME,
    order_type VARCHAR(20),
    amount DECIMAL(12,2)
)
PARTITION BY RANGE(order_date) SUBPARTITION BY LIST(order_type) (
    PARTITION p_2024_01 VALUES LESS THAN (DATE '2024-02-01') (
        SUBPARTITION p_2024_01_normal VALUES ('普通订单'),
        SUBPARTITION p_2024_01_vip VALUES ('VIP订单'),
        SUBPARTITION p_2024_01_bulk VALUES ('批发订单'),
        SUBPARTITION p_2024_01_other VALUES (DEFAULT)
    ),
    PARTITION p_2024_02 VALUES LESS THAN (DATE '2024-03-01') (
        SUBPARTITION p_2024_02_normal VALUES ('普通订单'),
        SUBPARTITION p_2024_02_vip VALUES ('VIP订单'),
        SUBPARTITION p_2024_02_bulk VALUES ('批发订单'),
        SUBPARTITION p_2024_02_other VALUES (DEFAULT)
    )
    -- ... 其他月份
);
```

## 分区管理

### 查看分区信息

```sql
-- 查看表的分区信息
SELECT 
    partition_name,
    partition_position,
    num_rows,
    tablespace_name
FROM user_tab_partitions
WHERE table_name = 'ORDERS'
ORDER BY partition_position;

-- 查看分区详情
SELECT 
    partition_name,
    high_value,
    high_value_len
FROM user_tab_partitions
WHERE table_name = 'ORDERS';

-- 查看分区数据量
SELECT 
    partition_name,
    NVL(num_rows, 0) AS row_count,
    NVL(blocks, 0) AS block_count,
    NVL(ROUND(blocks * 8 / 1024, 2), 0) AS size_mb
FROM user_tab_partitions
WHERE table_name = 'SALES'
ORDER BY partition_position;
```

### 分区维护操作

```sql
-- 添加分区
ALTER TABLE orders ADD PARTITION p_2025 VALUES LESS THAN (DATE '2026-01-01');

-- 删除分区（同时删除数据）
ALTER TABLE orders DROP PARTITION p_2020;

-- 截断分区（清空数据，保留分区）
ALTER TABLE orders TRUNCATE PARTITION p_2020;

-- 重命名分区
ALTER TABLE orders RENAME PARTITION p_max TO p_2025;

-- 移动分区到其他表空间
ALTER TABLE orders MOVE PARTITION p_2020 TABLESPACE archive_tbs;

-- 交换分区（与普通表交换）
ALTER TABLE orders EXCHANGE PARTITION p_2020 WITH TABLE orders_2020;

-- 合并分区
ALTER TABLE orders MERGE PARTITIONS p_2020, p_2021 INTO PARTITION p_2020_2021;

-- 拆分分区
ALTER TABLE orders SPLIT PARTITION p_2020 AT (DATE '2020-07-01') 
INTO (PARTITION p_2020_h1, PARTITION p_2020_h2);
```

### 分区与索引

```sql
-- 分区表索引类型

-- 1. 全局索引（整个表）
CREATE INDEX idx_global ON orders(order_date);

-- 2. 本地索引（每个分区独立索引）
CREATE INDEX idx_local ON orders(order_date) LOCAL;

-- 3. 本地前缀索引
CREATE INDEX idx_local_prefix ON orders(order_date) LOCAL;

-- 4. 本地位图索引（适合数据仓库）
CREATE BITMAP INDEX idx_bitmap ON orders(order_status) LOCAL;

-- 查看分区索引
SELECT 
    index_name,
    partition_name,
    status
FROM user_ind_partitions
WHERE index_name = 'IDX_LOCAL';
```

## 分区裁剪

### 分区裁剪原理

分区裁剪（Partition Pruning）是分区表最重要的性能优化特性。优化器会自动识别查询条件，只扫描相关分区。

```
分区裁剪示意：

查询：SELECT * FROM orders 
      WHERE order_date BETWEEN '2023-01-01' AND '2023-03-31'

无分区裁剪：
  扫描所有分区（10个分区各扫描一次）

有分区裁剪：
  识别分区条件
  → 只需扫描 p_2023_01, p_2023_02, p_2023_03
  → 其他分区直接跳过

性能提升：减少 70% 的扫描量
```

### 分区裁剪的触发条件

```sql
-- 可以触发分区裁剪的查询

-- 1. 等值条件
SELECT * FROM orders WHERE order_date = DATE '2023-01-15';

-- 2. 范围条件
SELECT * FROM orders WHERE order_date `>=` DATE '2023-01-01' AND order_date `<` DATE '2023-04-01';

-- 3. IN 条件
SELECT * FROM orders WHERE region IN ('北京', '上海', '广州');

-- 4. 列表分区键
SELECT * FROM customers WHERE region = '北京';

-- 无法触发分区裁剪的查询
SELECT * FROM orders WHERE customer_id = 1001;  -- 哈希分区无法确定分区
SELECT * FROM orders WHERE YEAR(order_date) = 2023;  -- 函数导致无法识别
```

### 验证分区裁剪

```sql
-- 查看执行计划，验证分区裁剪
EXPLAIN SELECT * FROM orders 
WHERE order_date BETWEEN DATE '2023-01-01' AND DATE '2023-03-31';

-- 期望看到 PARTITION RANGE ITERATOR 或 PARTITION LIST SINGLE
-- 而不是 PARTITION RANGE ALL
```

## 分区设计最佳实践

### 分区策略选择

| 数据特征 | 推荐分区类型 | 说明 |
|---|---|---|
| 时间序列数据 | 范围分区 | 按时间分区最常见 |
| 地域/类别数据 | 列表分区 | 按枚举值分区 |
| 写入热点分散 | 哈希分区 | 均匀分布 |
| 复杂查询需求 | 组合分区 | 多维度分析 |

### 分区键选择原则

1. **常用查询条件**：分区键应该是查询中最常使用的条件
2. **数据分布均匀**：避免某些分区过大
3. **历史数据归档**：支持快速归档历史分区
4. **数量适中**：分区数建议在 10-100 之间

```sql
-- 分区数量建议
-- 表数据量      | 建议分区数
-- < 1000万      | 4-10
-- 1000万-1亿    | 10-50
-- 1亿-10亿      | 50-100
-- `>` 10亿        | > 100
```

### 历史数据管理

```sql
-- 分区表的历史数据管理

-- 1. 将历史分区设为只读
ALTER TABLE orders READ ONLY PARTITION p_2020;

-- 2. 交换历史分区到归档表
CREATE TABLE orders_archive_2020 AS SELECT * FROM orders WHERE 1=0;

ALTER TABLE orders EXCHANGE PARTITION p_2020 
WITH TABLE orders_archive_2020;

-- 3. 删除历史分区
```

### 分区类型对比

**Q：范围分区、列表分区、哈希分区有什么区别？**

| 类型 | 划分依据 | 数据分布 | 适用场景 |
|---|---|---|---|
| 范围分区 | 值范围 | 不均匀 | 时间序列 |
| 列表分区 | 枚举值 | 可控 | 地域、类别 |
| 哈希分区 | 哈希函数 | 均匀 | 分散热点 |
| 组合分区 | 两种组合 | 灵活 | 复杂业务 |

### 分区裁剪

**Q：什么是分区裁剪？为什么重要？**

分区裁剪是优化器自动识别查询条件，只扫描相关分区的优化技术。

例如：查询 2023 年的订单，如果按年分区，优化器只会扫描 2023 年的分区，跳过其他 9 年的分区，性能提升可达 10 倍。

### 分区设计

**Q：如何设计分区策略？**

分区策略设计步骤：

1. 分析查询模式：确定最常用的查询条件
2. 选择分区键：查询条件中最常使用的列
3. 选择分区类型：根据数据特征选择
4. 确定分区数量：根据数据量和查询粒度确定
5. 考虑历史数据：设计归档策略

## 总结

表分区是处理大数据的核心技术：

- **范围分区**：适合时间序列数据，按范围划分
- **列表分区**：适合枚举值数据，按类别划分
- **哈希分区**：适合分散热点，均匀分布
- **组合分区**：适合复杂业务，多维度划分
- **分区裁剪**：自动优化，只扫描相关分区
- **分区管理**：支持添加、删除、合并、拆分

合理使用分区，可以将查询性能提升数倍甚至数十倍。
