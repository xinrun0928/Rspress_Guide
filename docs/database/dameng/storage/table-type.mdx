# 堆表与 HTSP 存储引擎

<Badge text="核心" type="tip" />

## 存储引擎概述

达梦数据库提供了多种存储引擎，适用于不同的业务场景。选择合适的存储引擎，是数据库设计的第一步。

想象一下你的业务场景：

- **交易系统**：大量并发写入，需要高性能
- **日志系统**：按时间顺序追加写入
- **报表系统**：海量历史数据查询

不同的场景，需要不同的存储引擎。

## 达梦的存储引擎类型

达梦数据库主要支持两种存储引擎：

| 引擎类型 | 英文名 | 适用场景 | 特点 |
|---|---|---|---|
| 堆表 | Heap Table | 通用场景 | 简单、高效 |
| HTSP | Hash Synonym Tree Plus | 快速查询 | 索引加速 |

```
┌─────────────────────────────────────────────────────────┐
│                   达梦存储引擎                           │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   ┌───────────────────┐    ┌───────────────────┐       │
│   │      堆 表        │    │      HTSP          │       │

│   │                   │    │                   │       │

│                                                         │
└─────────────────────────────────────────────────────────┘
```

## 堆表（Heap Table）

### 堆表的概念

堆表是最常见的表类型，数据按照插入顺序无序存储。没有主键的表默认创建为堆表。

```sql
-- 创建堆表（默认类型）
CREATE TABLE heap_table (
    id INT,
    name VARCHAR(100),
    created_at DATE
);

-- 显式指定为堆表
CREATE TABLE heap_explicit (
    id INT,
    name VARCHAR(100)
) ENGINE = HEAP;
```

### 堆表的特点

堆表的工作原理类似于餐厅的等待区：

```
堆表存储示意：

数据页 1: [Row A] [Row B] [Row C] [Row D] ...
数据页 2: [Row E] [Row F] [Row G] ...
数据页 3: ...
    ↓
    └── 无序存储，按插入顺序排列
```

**优点**：

- 插入性能高（直接追加到末尾）
- 存储结构简单
- 空间利用率高

**缺点**：

- 等值查询需要全表扫描
- 范围查询效率低
- 无主键时无唯一性保证

### 堆表的应用场景

| 场景 | 适用性 | 说明 |
|---|---|---|
| 写入密集型 | <Badge text="推荐" type="tip" /> | 插入性能优秀 |
| 临时表 | <Badge text="推荐" type="tip" /> | 一次性使用 |
| 批量导入 | <Badge text="推荐" type="tip" /> | 高速写入 |
| 高频更新 | <Badge text="可考虑" type="info" /> | 需配合索引 |
| 主键查询 | <Badge text="需要索引" type="warning" /> | 必须有主键或索引 |

```sql
-- 日志表（堆表适用）
CREATE TABLE access_logs (
    log_id INT IDENTITY(1,1),
    ip_address VARCHAR(50),
    access_time DATETIME DEFAULT SYSDATE,
    url VARCHAR(500)
) ENGINE = HEAP;

-- 批量导入优化
INSERT /*+ APPEND */ INTO heap_table 
SELECT * FROM source_table;
```

## HTSP 表

### HTSP 的概念

HTSP（Hash Synonym Tree Plus）是达梦的创新存储引擎，专门优化了基于键值的等值查询性能。

<Badge text="DM 特色" type="tip" />

HTSP 的核心思想是：数据按照哈希方式组织，使得等值查询可以直接定位到数据位置，无需扫描。

```
HTSP 存储结构：

        ┌─────────────────────────────┐
        │        HTSP 索引结构         │
        │   (Hash Tree + B+ Tree)      │
        └─────────────┬───────────────┘
                      │
        ┌─────────────┴─────────────┐
        │         Bucket 0          │ ──→ [Key A1, Key A2, ...]
        │         Bucket 1          │ ──→ [Key B1, ...]
        │         Bucket 2          │ ──→ [Key C1, Key C2, ...]
        │         Bucket 3          │ ──→ [...]
        │            ...             │
        └───────────────────────────┘
                      │
        ┌─────────────┴─────────────┐
        │       数据存储位置          │
        └───────────────────────────┘
```

### HTSP 的特点

**优势**：

- 等值查询极快（O(1) 时间复杂度）
- 适合高频等值查询场景
- 主键和唯一索引自动使用 HTSP

**限制**：

- 仅支持等值查询 `=`、`IN`
- 不支持范围查询 `<`、`<=`、`>`、`>=`
- 不支持模糊查询 `LIKE`

```sql
-- HTSP 表的创建
CREATE TABLE htsp_table (
    id INT PRIMARY KEY,          -- 主键自动使用 HTSP
    code VARCHAR(20) UNIQUE,     -- 唯一索引自动使用 HTSP
    name VARCHAR(100)
) ENGINE = HTSP;

-- 查询性能对比
-- 假设表有 1000 万条数据

-- 等值查询（两种引擎表现不同）
SELECT * FROM htsp_table WHERE id = 5000000;  -- HTSP: O(1)，毫秒级
SELECT * FROM heap_table WHERE id = 5000000;   -- Heap: O(log n)，可能需要索引

-- 范围查询（HTSP 不支持优化）
SELECT * FROM htsp_table WHERE id `>=` 5000000;  -- 退化为全表扫描
SELECT * FROM heap_table WHERE id `>=` 5000000;   -- 如果有索引，可能使用索引
```

### HTSP 的应用场景

| 场景 | 适用性 | 说明 |
|---|---|---|
| 字典表查询 | <Badge text="强烈推荐" type="tip" /> | 代码、状态等枚举值 |
| 主键查询 | <Badge text="推荐" type="tip" /> | 按 ID 查询单条 |
| 唯一性校验 | <Badge text="推荐" type="tip" /> | 唯一索引 |
| 范围查询 | <Badge text="不适用" type="danger" /> | 需使用其他引擎 |

```sql
-- 业务字典表（HTSP 适用）
CREATE TABLE currency (
    currency_code VARCHAR(3) PRIMARY KEY,
    currency_name VARCHAR(50),
    exchange_rate DECIMAL(10,4)
) ENGINE = HTSP;

-- 用户信息表（HTSP 适用）
CREATE TABLE users (
    user_id INT PRIMARY KEY,
    username VARCHAR(50) UNIQUE,
    email VARCHAR(100) UNIQUE,
    status INT
) ENGINE = HTSP;

-- 订单明细表（混合场景）
CREATE TABLE order_items (
    order_id INT,                -- 范围查询常用
    item_id INT,                 -- 主键
    product_code VARCHAR(20),     -- 等值查询常用
    quantity INT
) ENGINE = HTSP;
```

### 混合使用策略

在实际业务中，通常混合使用两种存储引擎：

```sql
-- 字典表（HTSP）
CREATE TABLE dim_product (
    product_id INT PRIMARY KEY,
    product_code VARCHAR(20) UNIQUE,
    product_name VARCHAR(200)
) ENGINE = HTSP;

-- 交易表（堆表 + 索引）
CREATE TABLE fact_sales (
    sale_id INT PRIMARY KEY,
    product_id INT,
    sale_date DATE,
    sale_amount DECIMAL(12,2),
    sale_qty INT
) ENGINE = HEAP;

-- 创建索引优化查询
CREATE INDEX idx_sales_date ON fact_sales(sale_date);
CREATE INDEX idx_sales_product ON fact_sales(product_id);

-- 组合查询示例
SELECT p.product_name, SUM(s.sale_amount) total_amount
FROM dim_product p
JOIN fact_sales s ON p.product_id = s.product_id
WHERE s.sale_date `>=` DATE '2024-01-01'
GROUP BY p.product_name;
```

## 存储引擎的实现细节

### HTSP 的索引结构

HTSP 使用哈希索引和 B+ 树的组合：

```sql

    uniqueness
```

## 面试要点

<Badge text="面试重点" type="warning" />

### 存储引擎选择

**Q：什么情况下应该使用 HTSP 表？什么情况下使用堆表？**

选择 HTSP 的场景：

- 等值查询是主要查询模式
- 适合字典表、配置表、用户表等小表
- 查询性能要求极高

选择堆表的场景：

- 范围查询是主要查询模式
- 大量数据写入
- 需要分区表

### 性能对比

**Q：HTSP 和堆表的性能差异在哪里？**

HTSP 的等值查询极快，原因是：

- 哈希索引直接定位数据
- 时间复杂度 O(1)
- 无需遍历数据页

堆表等值查询依赖索引，如果没有索引则需要全表扫描。

**Q：HTSP 有什么限制？**

HTSP 的主要限制：

- 不支持范围查询优化
- 不支持模糊查询优化
- 需要额外的索引空间

## 总结

达梦数据库的存储引擎设计合理：

- **堆表**：通用存储引擎，适合大多数场景，插入性能高
- **HTSP**：哈希索引引擎，适合等值查询，性能极快
- **选择原则**：根据查询特征选择合适的存储引擎
- **混合策略**：在实际业务中合理组合使用两种引擎

理解存储引擎的特点，是数据库性能优化的基础。
