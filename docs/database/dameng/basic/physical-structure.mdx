# 达梦数据库物理结构

<Badge text="基础" type="tip" />

## 物理结构概述

达梦数据库的物理结构是指数据库在磁盘上的实际存储形式。理解物理结构是 DBA 必备的基础知识，它直接影响到数据库的性能、备份恢复和故障处理。

当你面对一个运行中的达梦数据库时，你需要知道：

- 数据存储在哪里？
- 日志文件有什么作用？
- 控制文件损坏了怎么办？

这些问题的答案，都藏在物理结构里。

## 数据文件

### 数据文件的作用

数据文件（Data File）是存储数据库对象的物理文件，包括表、索引、视图等。一个表空间由一个或多个数据文件组成，数据文件的大小决定了表空间能够存储的数据量。

```sql
-- 查看数据文件信息
SELECT 
    tablespace_name,
    file_name,
    bytes / 1024 / 1024 AS size_mb,
    status
FROM dba_data_files
ORDER BY tablespace_name;
```

### 数据文件组织

数据文件按照页（Page）、区（Extent）、段（Segment）的逻辑结构组织：

```
数据文件 (*.dbf)
    ↓
┌─────────────────────────────────┐
│          区 (Extent)            │
│  ┌─────┐ ┌─────┐ ┌─────┐      │
│  │ 页  │ │ 页  │ │ 页  │ ...   │
│  └─────┘ └─────┘ └─────┘      │
└─────────────────────────────────┘
    ↓
┌─────────────────────────────────┐
│      页 (Page) - 最小单元 8KB    │
└─────────────────────────────────┘
```

### 数据文件类型

达梦支持两种类型的数据文件：

| 类型 | 扩展名 | 说明 |
|---|---|---|
| 表数据文件 | `.dbf` | 存储表和索引数据 |
| 临时数据文件 | `.tmp` | 存储临时表和排序操作 |

```sql
-- 查看临时数据文件
SELECT 
    tablespace_name,
    file_name,
    bytes / 1024 / 1024 AS size_mb
FROM dba_temp_files;

-- 查看所有数据文件
SELECT 
    tablespace_name,
    file_name,
    bytes / 1024 / 1024 AS size_mb,
    autoextensible
FROM dba_data_files
UNION ALL
SELECT 
    'TEMP' AS tablespace_name,
    file_name,
    bytes / 1024 / 1024 AS size_mb,
    autoextensible
FROM dba_temp_files;
```

### 数据文件的管理

```sql
-- 添加数据文件到表空间
ALTER TABLESPACE users ADD DATAFILE 
'/dm8/data/users02.dbf' SIZE 100M AUTOEXTEND ON NEXT 10M MAXSIZE 1G;

-- 修改数据文件大小
ALTER DATABASE DATAFILE '/dm8/data/users01.dbf' RESIZE 200M;

-- 设置数据文件为只读
ALTER DATABASE DATAFILE '/dm8/data/users01.dbf' READ ONLY;

-- 删除数据文件（必须为空）
ALTER TABLESPACE users DROP DATAFILE '/dm8/data/users02.dbf';
```

## 重做日志文件

### 重做日志的作用

重做日志文件（Redo Log File）是达梦数据库最核心的文件之一，它记录了所有对数据库的修改操作。这些记录是数据库恢复和复制的基础。

重做日志的核心作用：

1. **恢复**：系统故障后，通过日志恢复数据
2. **复制**：主备同步依赖日志传输
3. **审计**：记录所有数据变更

```sql
-- 查看重做日志文件信息
SELECT 
    group_id,
    thread#,
    sequence#,
    bytes / 1024 / 1024 AS size_mb,
    members,
    status
FROM v$logfile
ORDER BY group_id;
```

### 重做日志的结构

达梦的重做日志采用循环写入机制：

```
┌─────────────────────────────────────────────────────────┐
│                    重做日志文件组                        │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐    │
│  │  GROUP 1    │  │  GROUP 2    │  │  GROUP 3    │    │
│  │  (current)  │  │  (active)   │  │  (inactive) │    │
│  └─────────────┘  └─────────────┘  └─────────────┘    │
└─────────────────────────────────────────────────────────┘
                          ↓
            日志序列号递增: 001 → 002 → 003 → ...
```

每个日志文件组包含：

- **日志序列号**：标识日志的唯一编号
- **成员文件**：每个组可以有多个成员（镜像）
- **状态**：CURRENT（当前使用）、ACTIVE（活动）、INACTIVE（非活动）

### 重做日志的写入机制

```sql
-- 查看当前日志序列号
SELECT sequence#, status FROM v$log WHERE status = 'CURRENT';

-- 强制日志切换
ALTER SYSTEM SWITCH LOGFILE;

-- 检查点操作（将脏页写入数据文件）
ALTER SYSTEM CHECKPOINT;
```

日志写入遵循以下规则：

1. 日志按顺序循环写入
2. 当当前日志写满时，触发日志切换
3. 归档模式下的日志必须归档后才能被覆盖

### 归档配置

```sql
-- 查看归档模式
SELECT arch_mode FROM v$database;

-- 开启归档模式
ALTER DATABASE ARCHIVELOG;

-- 配置归档目录
ALTER SYSTEM SET ARCH_DEST = '/dm8/arch' SCOPE = SPFILE;

-- 强制归档
ALTER SYSTEM ARCHIVE LOG CURRENT;
```

## 控制文件

### 控制文件的作用

控制文件（Control File）是一个非常小但极其重要的文件，它记录了数据库的物理结构信息，是数据库打开和正常运行的必要条件。

控制文件记录的信息包括：

- 数据库名称和创建时间
- 数据文件和日志文件的位置
- 表空间信息
- 检查点信息
- 归档日志信息

```sql
-- 查看控制文件位置
SELECT name FROM v$controlfile;

-- 查看控制文件内容（部分信息）
SELECT 
    dbid,
    name,
    created,
    log_mode,
    open_mode
FROM v$database;
```

### 控制文件的重要性

控制文件损坏会导致数据库无法启动：

```
控制文件损坏场景：

1. 数据库启动时读取控制文件
2. 发现控制文件不存在或损坏
3. 数据库报错: ORA-00205
4. 需要使用备份恢复或重建控制文件
```

### 多路复用控制文件

为了防止控制文件单点故障，建议对控制文件进行多路复用：

```sql
-- 查看当前控制文件配置
SHOW PARAMETER CONTROL_FILES;

-- 配置多个控制文件（需重启生效）
-- 在 dm.ini 中配置:
-- CONTROL_FILES = '/dm8/data/control1.ctl', '/dm8/data/control2.ctl', '/dm8/backup/control3.ctl'
```

多路复用的原则：

- 控制文件分布在不同磁盘
- 至少配置 3 个控制文件
- 定期备份控制文件

### 备份控制文件

```sql
-- 备份控制文件为二进制
ALTER DATABASE BACKUP CONTROLFILE TO '/dm8/backup/control.bak';

-- 备份控制文件为 trace（文本格式，便于重建）
ALTER DATABASE BACKUP CONTROLFILE TO TRACE AS '/dm8/backup/control_trace.sql';

-- 查看备份的控制文件
! ls -la /dm8/backup/control*.bak
```

## 物理结构详解

### 完整的物理结构

```
/dm8/
├── data/                          # 数据目录
│   ├── DAMENG/                    # 实例名
│   │   ├── SYSTEM.dbf             # SYSTEM 表空间
│   │   ├── ROLL.dbf               # 回滚表空间
│   │   ├── TEMP.dbf               # 临时表空间
│   │   ├── MAIN.dbf               # 主表空间
│   │   ├── dm.ctl                 # 控制文件
│   │   ├── dm01.log               # 重做日志文件组1
│   │   ├── dm02.log               # 重做日志文件组2
│   │   └── *.dbf                  # 用户数据文件
│   └── backup/                    # 备份目录
│       ├── control.bak            # 控制文件备份
│       └── full_bak_20260330.bak  # 完整备份
├── arch/                          # 归档目录
│   ├── ARCH_0001.log              # 归档日志
│   └── ARCH_0002.log
└── log/                           # 实例日志
    ├── dm_dmenga.log              # DM 服务日志
    └── dm_aarch.log               # 归档日志
```

### 关键参数

| 参数 | 说明 | 建议值 |
|---|---|---|
| `PAGE_SIZE` | 页大小 | 8KB（默认） |
| `EXTENT_SIZE` | 区大小 | 16 页 |
| `MAX_SESSIONS` | 最大会话数 | 根据需求 |
| `BUFFER` | 缓冲区大小 | 物理内存的 30-40% |
| `PAGE_CHECK` | 页校验 | 0（默认） |

```sql
-- 查看数据库关键参数
SELECT para_name, para_value FROM v$dm_ini 
WHERE para_name IN (
    'PAGE_SIZE', 'EXTENT_SIZE', 'MAX_SESSIONS', 
    'BUFFER', 'PAGE_CHECK'
);
```

## 物理结构与性能

### 数据文件布局原则

合理的物理布局能够显著提升性能：

| 原则 | 说明 | 实现方式 |
|---|---|---|
| 分离系统表空间 | 减少 I/O 竞争 | SYSTEM、ROLL 单独磁盘 |
| 分离数据与索引 | 提升访问效率 | 独立表空间 |
| 分离热数据 | 减少竞争 | 分区表 |
| 多磁盘分布 | 并行 I/O | 不同磁盘存放不同文件 |

```sql
-- 创建分离的表空间（示例）
-- 假设 /data1、/data2、/data3 是不同磁盘挂载点

-- 创建索引表空间
CREATE TABLESPACE idx_tbs DATAFILE '/data2/idx01.dbf' SIZE 1G
    AUTOEXTEND ON NEXT 100M MAXSIZE 10G;

-- 创建临时表空间（使用临时文件）
CREATE TEMPORARY TABLESPACE temp_tbs TEMPFILE '/data3/temp01.tmp' SIZE 2G;

-- 将表的索引移到独立表空间
ALTER INDEX idx_emp_name REBUILD TABLESPACE idx_tbs;
```

### 日志文件配置

日志文件的配置直接影响数据库性能：

```sql
-- 查看当前日志配置
SELECT 
    group#,
    bytes / 1024 / 1024 AS size_mb,
    members,
    status
FROM v$log;

-- 建议的日志配置
-- 1. 日志文件大小: 64MB-256MB
-- 2. 日志组数量: 至少 3 组
-- 3. 成员数量: 至少 2 个（镜像）

-- 添加新的日志文件组
ALTER DATABASE ADD LOGFILE 
    GROUP 4 '/dm8/data/dm04.log' SIZE 128M,
    GROUP 5 '/dm8/data/dm05.log' SIZE 128M;

-- 删除不活跃的日志组
ALTER DATABASE DROP LOGFILE GROUP 1;
```

## 面试要点

<Badge text="面试重点" type="warning" />

### 物理文件类型

**Q：达梦数据库的物理文件有哪些？各自的作用是什么？**

达梦数据库的核心物理文件包括：

1. **数据文件（.dbf）**：存储表、索引等数据库对象
2. **重做日志文件（.log）**：记录所有数据修改操作，用于恢复和复制
3. **控制文件（.ctl）**：记录数据库的物理结构信息，是数据库启动的必要条件
4. **临时文件（.tmp）**：存储排序、哈希等临时操作的数据

### 日志机制

**Q：重做日志的作用是什么？为什么至少需要两个日志组？**

重做日志的核心作用有两个：

1. **恢复**：系统故障后，通过重做日志恢复已提交的事务
2. **复制**：数据守护通过日志传输实现主备同步

至少需要两个日志组的原因是：达梦采用循环写机制。当一个日志组写满时，必须切换到另一个组。如果只有一组，那么在日志归档完成之前，无法继续写入，导致数据库阻塞。

### 控制文件

**Q：控制文件损坏后如何处理？**

控制文件损坏的处理步骤：

1. 尝试使用多路复制的控制文件启动
2. 如果所有控制文件都损坏，使用备份恢复
3. 如果没有备份，通过 `CREATE CONTROLFILE` 重建

预防措施：配置多个控制文件（至少 3 个），分布在不同磁盘上。

### 性能优化

**Q：如何通过物理结构优化数据库性能？**

物理结构优化的方法：

1. **分离表空间**：将 SYSTEM、ROLL、USER 表空间放在不同磁盘
2. **分离数据与索引**：为索引创建独立表空间
3. **合理配置日志**：日志文件放在高速磁盘，配置多组日志
4. **启用归档**：开启归档模式，配置合理的归档目录

## 总结

达梦数据库的物理结构是理解数据库运行机制的基础：

- **数据文件**：存储数据库对象，按页、区、段组织
- **重做日志**：记录修改操作，支持恢复和复制
- **控制文件**：记录物理结构信息，是启动的必要条件
- **物理布局**：合理的磁盘布局能够显著提升性能

理解物理结构，是成为合格 DBA 的第一步。
