# DM 多版本并发控制：MVCC 原理深度解析

并发控制，是数据库最核心的技术之一。

当 thousands of users simultaneously read and write to the same database，how does DM ensure that everyone's operations don't interfere with each other? When you read data, why can you see the "consistent snapshot" instead of half-committed transactions?

这一切的背后，是 **MVCC（Multi-Version Concurrency Control，多版本并发控制）** 在默默工作。

## 为什么需要 MVCC

### 传统锁的问题

在没有 MVCC 的年代，数据库用 **二阶段锁（2PL）** 来控制并发：

```sql
-- 事务 A：读取用户余额
BEGIN;
SELECT balance FROM accounts WHERE user_id = 1;
-- 此时，事务 A 持有读锁

-- 事务 B：修改用户余额
UPDATE accounts SET balance = balance + 1000 WHERE user_id = 1;
-- 被阻塞，等待事务 A 的读锁释放
```

问题：
- **读写冲突**：读操作会阻塞写操作，写操作也会阻塞读操作
- **并发度低**：所有操作串行化，性能差
- **死锁风险**：多个事务互相等待对方释放锁

### MVCC 的解决思路

MVCC 的核心思想：**读写不冲突**。

- **写操作**：创建数据的新版本，而不是修改原数据
- **读操作**：读取某个时间点的快照，看到的是一致的历史数据
- **垃圾回收**：定期清理不再需要的旧版本

```
时间线
─────────────────────────────────────────────────────────▶
     │                    │                    │
     │    T1(begin)       │                    │
     │    │               │                    │
     │    │ 读取数据 V1    │                    │
     │    │               │                    │
     │    │               │  T2(begin)          │
     │    │               │  │                 │
     │    │               │  │ 写入数据 V2      │
     │    │               │  │ (新版本)        │
     │    │               │  │                 │
     │    │ 仍读到 V1      │  │                 │
     │    │ (不受影响)     │  │                 │
     ▼                    ▼                    ▼
   快照时间点          快照时间点            快照时间点
```

## DM MVCC 核心机制

### 事务 ID 与行版本

DM 为每个事务分配唯一的 **事务 ID（XID）**，并为每次数据修改记录版本信息：

```sql
-- 查看数据行的版本信息（DM 内部视图）
SELECT * FROM SYS.SYSTRANTABLE WHERE TABNAME = 'ACCOUNTS';

-- 行头信息包含：
-- TRX_ID: 最后修改该行的事务 ID
-- UNDO_SEG_NO: 所属的回滚段号
-- ROLL_PTR: 指向回滚记录的指针
```

### 回滚段（Undo Segment）

DM 的 MVCC 依赖 **回滚段（Undo Segment）** 存储历史版本：

```sql
-- 查看回滚段信息
SELECT SEGMENT_ID, TABLESPACE_NAME, STATUS
FROM DBA_ROLLBACK_SEGS;

-- 回滚段结构
/*
 * 每个回滚段包含多个回滚块
 * 回滚块中存储的是 "反向操作"
 * UPDATE: 记录旧值（用于回滚和读一致性）
 * DELETE: 记录删除前的完整行（用于恢复）
 */
```

**回滚段的工作原理**：

```
数据行 (Current Version)
┌─────────────────────────────────────────┐
│ user_id: 1                              │
│ balance: 1000                           │
│ TRX_ID: 101                             │  ← 事务 101 写入
│ ROLL_PTR: 0x1000                       │  ← 指向回滚段
└─────────────────────────────────────────┘
        │
        │ ROLL_PTR 指向
        ▼
回滚段记录 (Old Version)
┌─────────────────────────────────────────┐
│ TRX_ID: 99                              │  ← 事务 99 的旧值
│ balance: 800                            │  ← 原来的值
│ ROLL_PTR: 0x2000                       │  ← 继续指向前一个版本
└─────────────────────────────────────────┘
```

### 版本链（Version Chain）

当一行数据被多次修改，就会形成 **版本链**：

```sql
-- 版本链结构
/*
 * 每次 UPDATE，都会：
 * 1. 将当前行复制到回滚段（创建旧版本）
 * 2. 修改当前行的值为新值
 * 3. 更新 TRX_ID 和 ROLL_PTR
 *
 * 版本链就是通过 ROLL_PTR 串联起来的
 */
```

```
版本链（从新到旧）
┌─────────────────────────────────────┐
│  Version 3 (最新)                    │
│  balance: 1500                      │
│  TRX_ID: 103                        │
│  ROLL_PTR ──────────────────────────┼──┐
└─────────────────────────────────────┘    │
                                          │
┌─────────────────────────────────────┐    │
│  Version 2                           │    │
│  balance: 1200                      │    │
│  TRX_ID: 102                        │    │
│  ROLL_PTR ──────────────────────────┼──┘
└─────────────────────────────────────┘
                                          │
┌─────────────────────────────────────┐    │
│  Version 1 (最旧)                    │    │
│  balance: 800                       │    │
│  TRX_ID: 99                         │    │
│  ROLL_PTR: NULL (链尾)              │    │
└─────────────────────────────────────┘
```

## 读一致性实现

### 快照读（Snapshot Read）

DM 的 MVCC 实现中，**读操作默认读取快照**，而非加锁读取：

```sql
-- 会话 A：开启事务
BEGIN;
SELECT balance FROM accounts WHERE user_id = 1;
-- 返回 balance = 1000（事务开始时的快照）

-- 会话 B：此时修改了数据
UPDATE accounts SET balance = 1500 WHERE user_id = 1;
COMMIT;

-- 会话 A：再次读取（仍在同一事务中）
SELECT balance FROM accounts WHERE user_id = 1;
-- 仍然返回 1000（快照读）
```

### 读取哪个版本

当事务 T 读取一行数据时，DM 按以下规则选择版本：

```
1. 如果某行数据的 TRX_ID = T
   → 这是当前事务自己修改的，返回最新值

2. 如果某行数据的 TRX_ID < T AND 状态是 COMMITTED
   → 读取这个版本

3. 如果某行数据的 TRX_ID = 活跃事务（未提交）
   → 沿着 ROLL_PTR 找到上一个可见版本

4. 如果找不到可见版本
   → 该行对当前事务不可见（可能已被删除）
```

```sql
-- 伪代码：版本选择算法
function get_version(row, txn):
    if row.trx_id == txn.id:
        return row  // 自己的修改，返回最新
    
    if row.trx_id < txn.start_ts:
        if is_committed(row.trx_id):
            return row  // 已提交的旧版本
    
    // 需要查看历史版本
    current = row
    while current.roll_ptr != NULL:
        current = undo_segment[current.roll_ptr]
        if current.trx_id < txn.start_ts and is_committed(current.trx_id):
            return current
    
    return NULL  // 不可见
```

### 当前读（Current Read）

某些操作需要读取数据的最新状态，称为 **当前读**：

```sql
-- 当前读：加锁读取最新数据
SELECT * FROM accounts WHERE user_id = 1 FOR UPDATE;

-- INSERT 操作也是当前读
INSERT INTO accounts VALUES (1, 2000);

-- UPDATE/DELETE 也会加锁
UPDATE accounts SET balance = balance + 100 WHERE user_id = 1;
```

## MVCC 与事务隔离级别

DM 通过 MVCC 支持不同的事务隔离级别：

```sql
-- 设置隔离级别
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- DM 默认隔离级别：READ COMMITTED
```

### READ COMMITTED

```sql
-- 会话 A
BEGIN;
-- 读取 balance = 1000

-- 会话 B：COMMIT，balance 变为 1500

-- 会话 A：再次读取
SELECT balance FROM accounts WHERE user_id = 1;
-- 返回 1500（每次读取都是当前已提交的最新值）
```

### SERIALIZABLE

```sql
-- 会话 A
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
BEGIN;
SELECT balance FROM accounts WHERE user_id = 1;
-- 返回 1000（事务开始时的快照）

-- 会话 B：COMMIT，balance 变为 1500

-- 会话 A：再次读取
SELECT balance FROM accounts WHERE user_id = 1;
-- 仍然返回 1000（同一事务内，读取的是同一个快照）
```

## MVCC 性能优化

### 回滚段管理

```sql
-- 查看回滚段使用情况
SELECT SEGMENT_NAME, TABLESPACE_NAME, 
       EXTENTS, BLOCKS, EXTENTS * BLOCKS * 8192 as SIZE_BYTES
FROM DBA_ROLLBACK_SEGS;

-- 回滚段自动管理（AUM）
-- DM 默认使用自动回滚段管理
-- 可以手动设置回滚段数量
ALTER SYSTEM SET ROLLBACK_SEGMENTS = ('RB1', 'RB2', 'RB3');
```

### 垃圾回收与清理

MVCC 的旧版本不会永远保留，DM 通过 **垃圾回收（GC）** 清理不再需要的版本：

```sql
-- 手动清理回滚段（需要管理员权限）
PURGE;

-- 查看哪些事务可以被清理
SELECT TRX_ID, STATUS, COMMIT_TIME
FROM V$TRX_HISTORY
WHERE STATUS = 'COMMITTED'
  AND COMMIT_TIME < SYSDATE - 7;

-- GC 触发的条件
/*
 * 1. 回滚段空间不足时
 * 2. 旧版本数据不再被任何活跃事务需要
 * 3. 系统定时任务触发
 */
```

### 性能调优建议

| 优化项 | 说明 |
|---|---|
| 合理设置事务时长 | 长时间事务会产生大量回滚数据，影响 GC |
| 避免大事务 | 事务越大，产生的版本越多，清理压力越大 |
| 定期提交 | 长事务不利于并发，可分批提交减少回滚段压力 |
| 监控回滚段使用 | 关注 `_SYSSUNDO` 表空间使用率 |

```sql
-- 监控回滚段使用
SELECT r.segment_name,
       t.tablespace_name,
       t.used_blocks * 8192 / 1024 / 1024 as USED_MB,
       t.total_blocks * 8192 / 1024 / 1024 as TOTAL_MB,
       ROUND(t.used_blocks / t.total_blocks * 100, 2) as USED_PCT
FROM DBA_ROLLBACK_SEGS r
JOIN DBA_TABLESPACE_USAGE t ON r.tablespace_name = t.tablespace_name;

-- 扩容回滚表空间
ALTER TABLESPACE _SYSSUNDO RESIZE 1024M;
```

## 常见问题与解决

### 问题一：快照读看不到最新提交的数据

```sql
-- 场景：同一个事务内，查询结果不一致
BEGIN;
SELECT COUNT(*) FROM orders;  -- 返回 1000
INSERT INTO orders VALUES (...);  -- 新增一条
SELECT COUNT(*) FROM orders;  -- 仍然返回 1000？
```

**原因**：某些 DM 配置下，INSERT 的自增 ID 可能不在快照内。

**解决**：使用 `FOR UPDATE` 强制当前读，或开启 `SET CURRENT SQL CACHE`。

### 问题二：长时间查询导致回滚段膨胀

```sql
-- 场景：大事务 + 长查询 = 回滚段爆炸
BEGIN;
-- 事务 A 开始

-- 其他事务修改了大量数据
UPDATE orders SET status = 'processed' WHERE create_date < '2024-01-01';  -- 百万级更新

-- 事务 A 才开始查询（需要读取修改前的版本）
SELECT * FROM orders WHERE ...;  -- 触发大量版本读取
```

**解决**：
- 避免长时间开启事务
- 将大事务拆分为小事务
- 使用批量提交减少单次事务的大小

### 问题三：ORA-01555 错误（快照过旧）

```sql
-- DM 的类似错误：回滚段空间不足
/*
 * 原因：
 * 1. 长时间事务持有回滚段
 * 2. 新事务需要读取被覆盖的旧版本
 * 3. 回滚段已回收该版本
 */
```

**解决**：
- 扩容回滚表空间
- 减少大事务
- 使用 Commit Rate Higher 配置

## 总结

MVCC 是 DM 并发控制的核心机制：

- **版本链**：通过 ROLL_PTR 串联历史版本
- **回滚段**：存储修改前的数据，支持回滚和读一致性
- **快照读**：事务内读取一致的历史快照
- **垃圾回收**：定期清理不再需要的旧版本

理解 MVCC，你就能理解为什么 DM 可以在高并发下保持读写不冲突——这是现代数据库的必备能力。
