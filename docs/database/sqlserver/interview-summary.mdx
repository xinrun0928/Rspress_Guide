# SQL Server 面试汇总

「你们为什么选择 SQL Server 而不是 MySQL？」

这道题的背后，考察的是你对数据库选型的理解。SQL Server 在企业级场景中有独特的优势，它的 Always On 可用性组、强大的调优工具链、以及与 Windows 生态的深度集成，是 MySQL 难以替代的。

本文汇总 SQL Server 面试中的高频问题，按模块分类，助你系统备战。

## 架构与存储

### Q1：SQL Server 的存储结构是怎样的？

SQL Server 的最小存储单元是**页（Page）**，大小固定为 **8KB**。页是数据读写的基本单位，包含页头（96字节）、行偏移数组（每行 2 字节）和数据区（剩余约 8060 字节）。

**8 个连续的页**构成一个**区（Extent）**，区是空间分配的基本单位。区分为统一区（整区属于单个对象）和混合区（多个对象共享）。

数据以 B+ 树结构组织。**聚集索引**的叶节点直接存储数据行，数据按索引键顺序物理排序；**非聚集索引**的叶节点存储索引键和行定位器（书签），指向聚集索引或堆表中的实际行。

:::tip
面试追问：为什么不直接用区作为最小读写单位？
区的 64KB 大小适合批量操作，但对于单行访问来说粒度过粗。页的 8KB 设计在 I/O 效率和内存利用率之间取得了平衡。
:::

### Q2：Buffer Pool 在 SQL Server 中扮演什么角色？

Buffer Pool 是 SQL Server 最大的内存消耗者，负责缓存数据页和执行计划。它的工作流程：

1. 查询请求先在 Buffer Pool 中查找目标页（缓存命中）
2. 未命中时，从磁盘读取页到内存
3. 修改操作在内存中进行，生成日志记录
4. 检查点进程定期将脏页写回磁盘

**缓冲池命中率**是衡量数据库性能的关键指标。通过 `SELECT * FROM sys.dm_os_buffer_descriptors` 可以查看缓存页面分布。命中率低于 95% 通常意味着需要增加内存。

### Q3：日志文件（LDF）的作用是什么？

事务日志采用 **Write-Ahead Logging (WAL)** 机制：所有修改操作先记录到日志，日志写入成功后，才允许修改数据页。

日志记录包含：
- 事务的 LS唯一的序列号
- 修改的操作类型（INSERT/UPDATE/DELETE）
- 修改前后的数据映像

日志链确保了事务的原子性（A）和持久性（D）。即使数据库崩溃，通过重做（Redo）和回滚（Undo）日志记录，可以将数据库恢复到一致状态。

## 索引

### Q4：聚集索引和非聚集索引的区别？

| 特性 | 聚集索引 | 非聚集索引 |
|---|---|---|
| 叶节点存储 | 数据行 | 索引键 + 行定位器 |
| 表中数量 | 只能有 1 个 | 可以有多个 |
| 键值唯一性 | 必须唯一（可非唯一） | 可重复 |
| 创建时机 | 表创建时自动创建（无主键时） | 手动创建 |
| 物理顺序 | 数据按索引键排序存储 | 独立于数据的 B+ 树 |

**重要**：如果表已有聚集索引，非聚集索引的行定位器是聚集索引键；如果表是堆（无聚集索引），行定位器是物理行号（RID）。

:::warning
面试追问：为什么主键推荐使用自增 ID 而不是 UUID？
自增 ID 保证新行追加到索引末端，避免页分裂和索引碎片。UUID 的随机性会导致频繁的页分裂和索引维护开销。
:::

### Q5：什么是索引碎片？有哪些类型？

**内部碎片**：页内空闲空间过多。当页被频繁更新删除后，数据行重新分布但页未重新组织，导致页填充率下降。

**外部碎片**：逻辑顺序与物理顺序不一致。当页被拆分时，新页可能分配在磁盘的不同位置，导致顺序扫描需要更多随机 I/O。

碎片率通过 `sys.dm_db_index_physical_stats` 查看：
- `avg_fragmentation_in_percent` `< 5%`：碎片可忽略
- `5% ~ 30%`：需要重新组织（ALTER INDEX REORGANIZE）
- `> 30%`：需要重新构建（ALTER INDEX REBUILD）

## 事务与锁

### Q6：SQL Server 的锁升级是什么？

当事务在单个表上持有太多行锁时，SQL Server 会将多个行锁**升级**为**页锁或表锁**。触发条件：

- 单个语句在单个对象上持有超过 **5000 个锁**
- 锁内存使用超过总锁池的 **40%**

锁升级是 SQL Server 的优化机制，但也是性能问题的常见原因。大量并发事务同时升级为表锁，会造成严重的锁等待。

:::tip
解决方案：
- 降低锁升级阈值：`ALTER TABLE ... SET (LOCK_ESCALATION = AUTO)` 让 SQL Server 在分区级别升级
- 减少单次事务的锁数量：批量操作分批提交
- 调整隔离级别：使用快照隔离避免长时间锁定
:::

### Q7：什么是死锁？如何避免？

死锁是两个或多个事务互相持有对方需要的锁，形成循环等待。SQL Server 的死锁解决机制：

1. 死锁检测线程定期检查等待锁图
2. 发现死锁后，选择**代价最小**的事务作为牺牲品
3. 牺牲品事务被终止（抛出 1205 错误），回滚其工作
4. 另一个事务获得锁，继续执行

**避免死锁的策略**：
- 按固定顺序访问多个表（所有事务遵守同一顺序）
- 减少事务持续时间
- 使用较低隔离级别
- 避免在事务中等待用户交互

### Q8：快照隔离级别和 READ COMMITTED SNAPSHOT 有什么区别？

**SNAPSHOT 隔离**：事务开始时获取数据快照，读取的是事务开始时的 Committed 数据，不受其他事务修改影响。但写入时需要检测冲突，可能导致更新失败。

**READ COMMITTED SNAPSHOT (RCSI)**：语句执行时获取数据快照。默认的 READ COMMITTED 会被修改为读取行版本，不再读取脏数据，但能看到其他已提交事务的修改。

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 行版本 |
|---|---|---|---|---|
| READ UNCOMMITTED | 可能 | 可能 | 可能 | 无 |
| READ COMMITTED | 不可能 | 可能 | 可能 | 无 |
| READ COMMITTED SNAPSHOT | 不可能 | 可能 | 可能 | 有 |
| REPEATABLE READ | 不可能 | 不可能 | 可能 | 无 |
| SNAPSHOT | 不可能 | 不可能 | 可能 | 有 |
| SERIALIZABLE | 不可能 | 不可能 | 不可能 | 无 |

## 高可用

### Q9：Always On 可用性组的核心组件是什么？

**可用性副本（Availability Replica）**：可用性组中的数据库实例，包含主副本和辅助副本。每个副本有自己的 SQL Server 实例和存储。

**可用性组监听器（Availability Group Listener）**：虚拟网络名（VNN）和虚拟 IP（VIP）。应用程序连接监听器而非直接连接副本，透明实现故障转移。

**数据同步模式**：
- **同步提交**：主副本写入日志后，等待辅助副本确认写入日志，日志持久化后才返回成功。零数据丢失，但有延迟。
- **异步提交**：主副本不等待辅助副本确认，适合远距离复制。可能的少量数据丢失。

### Q10：故障转移群集（FCI）和可用性组（AG）的区别？

| 特性 | 故障转移群集 (FCI) | 可用性组 (AG) |
|---|---|---|
| 故障转移单位 | 整个实例 | 单个数据库组 |
| 存储要求 | 共享存储（SAN） | 各副本独立存储 |
| 支持版本 | SQL Server 2008+ | SQL Server 2012+ |
| 自动故障转移 | 支持 | 仅同步提交模式 |

FCI 保护整个 SQL Server 实例，但需要共享存储，成本高。AG 允许副本分布在不同数据中心，存储独立，更灵活。

## 性能优化

### Q11：如何分析执行计划？

使用 `SET STATISTICS IO ON` 和 `SET STATISTICS TIME ON` 开启性能统计，或在 SSMS 中查看实际执行计划。

**关键操作符**：
- **Index Seek**：索引查找，最优
- **Index Scan**：索引扫描，需评估数据量
- **Table Scan/Clustered Index Scan**：全表扫描，尽量避免
- **Nested Loop**：嵌套循环连接，适合小表驱动大表
- **Hash Match**：哈希连接，适合大表等值连接
- **Merge Join**：合并连接，适合已排序输入

**关注指标**：
- **Estimated vs Actual Rows**：估计行数与实际行数差异大，说明统计信息过时
- **Cost**：各操作符占总成本的百分比，定位最大成本
- **Warnings**：类型转换、缺少索引等警告

### Q12：Query Store 有什么用？

Query Store 是 SQL Server 2016+ 的特性，持续收集查询的执行计划、运行时统计和等待统计。

核心用途：
- **强制执行计划**：当新计划导致性能下降时，回退到历史好计划
- **性能回归检测**：自动检测查询性能下降
- **调优指导**：分析高频、低效查询

```sql
-- 开启 Query Store
ALTER DATABASE [DatabaseName] SET QUERY_STORE = ON;

-- 查看最耗时查询
SELECT q.query_id, p.plan_id, rsavg_duration, rsi.execution_type_desc
FROM sys.query_store_runtime_stats rs
JOIN sys.query_store_plan p ON rs.plan_id = p.plan_id
JOIN sys.query_store_query q ON p.query_id = q.query_id
JOIN sys.query_store_runtime_stats_interval rsi ON rs.runtime_stats_interval_id = rsi.runtime_stats_interval_id
ORDER BY rsavg_duration DESC;
```

## 面试总结

SQL Server 面试的核心考点集中在三个方面：

1. **存储与索引**：页面结构、聚集/非聚集索引的区别、碎片处理
2. **事务与锁**：锁升级机制、死锁处理、快照隔离级别
3. **高可用**：Always On 架构、故障转移模式、备份策略

每道题的回答都应该从**原理**出发，结合**实际场景**，展示你对技术的深度理解。
