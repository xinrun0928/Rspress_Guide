# 锁模式

「为什么这条 UPDATE 语句会阻塞其他查询？」

答案是**锁**。SQL Server 使用锁来保证事务的隔离性，但锁也是并发问题的根源。

理解 SQL Server 的锁模式，是解决死锁和阻塞问题的前提。

## 锁的类型

### 按粒度分类

SQL Server 支持多粒度锁定：

```
┌─────────────────────────────────────────────────────────────────┐
│                    锁粒度层次                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│                      数据库 (Database)                          │
│                           │                                     │
│         ┌─────────────────┼─────────────────┐                   │
│         │                 │                 │                  │
│      文件组            表 (Table)        文件 (File)            │
│      (Filegroup)          │                                     │
│                           │                                     │
│         ┌─────────────────┼─────────────────┐                   │
│         │                 │                 │                   │
│      区 (Extent)      页 (Page)          索引 (Index)            │
│                           │                                     │
│                           │                                     │
│                      行 (Row)                                   │
│                                                                 │
│  锁可以从低粒度升级到高粒度                                      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

| 粒度 | 说明 | 锁定对象 |
|---|---|---|
| RID | 行 ID | 堆表中的单行 |
| KEY | 键 | 索引中的行 |
| PAGE | 页 | 8KB 数据页 |
| EXTENT | 区 | 8 页连续空间 |
| TABLE | 表 | 整张表 |
| DATABASE | 数据库 | 整个数据库 |

### 按模式分类

| 锁模式 | 缩写 | 允许 | 阻塞 |
|---|---|---|---|
| 共享锁 | S | 读取 | X、U |
| 排他锁 | X | 无 | S、X、U、IS、IX |
| 更新锁 | U | 读取 | X |
| 意向锁 | IS、IX、SIX | 复杂 | X |
| 架构锁 | Sch-M、Sch-S | - | - |
| 批量锁 | Bulk | - | - |
| 键范围锁 | Range | - | - |

## 共享锁（S）

共享锁允许并发读取，阻塞写入。

```sql
-- 读取数据时获取共享锁
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;

BEGIN TRANSACTION;
    SELECT * FROM Orders WHERE OrderID = 1;
    -- 获取 OrderID=1 行的共享锁
    -- 其他事务可以同时读取此行
    -- 但不能修改（需要排他锁）
COMMIT;
```

### 共享锁行为

- **兼容性**：多个共享锁可以同时存在
- **持有时间**：语句结束时释放（READ COMMITTED）
- **阻塞**：被排他锁阻塞

## 排他锁（X）

排他锁允许独占写入，阻塞所有其他锁。

```sql
-- 更新数据时获取排他锁
BEGIN TRANSACTION;
    UPDATE Orders SET Amount = 100 WHERE OrderID = 1;
    -- 获取 OrderID=1 行的排他锁
    -- 其他事务不能读取或修改此行
COMMIT;
```

### 排他锁行为

- **兼容性**：不与任何锁兼容
- **持有时间**：事务结束时释放
- **阻塞**：被所有锁阻塞

## 更新锁（U）

更新锁是共享锁和排他锁之间的折中。

```sql
-- 更新数据时，先获取更新锁
BEGIN TRANSACTION;
    UPDATE Orders SET Amount = 100 WHERE OrderID = 1;
    -- 1. 先获取更新锁（U）
    -- 2. 读取数据，检查是否需要修改
    -- 3. 将更新锁升级为排他锁
    -- 4. 执行修改
COMMIT;
```

### 更新锁的用途

```
死锁场景（不使用 U 锁）：
┌─────────────────────────────────────────────────────────────────┐
│  事务 A                    事务 B                              │
│  UPDATE Orders             UPDATE Orders                        │
│  SET Amount=100           SET Amount=200                       │
│  WHERE OrderID=1           WHERE OrderID=2                      │
│  获取 X 锁(OrderID=1)      获取 X 锁(OrderID=2)                │
│                           UPDATE Orders                        │
│                           SET Amount=300                        │
│                           WHERE OrderID=1                      │
│                           等待 A 释放 X 锁                      │
│  UPDATE Orders                                                         │
│  SET Amount=400                                                         │
│  WHERE OrderID=2                                                         │
│  等待 B 释放 X 锁 ──▶ 死锁！                                         │
└─────────────────────────────────────────────────────────────────┘

使用 U 锁避免死锁：
┌─────────────────────────────────────────────────────────────────┐
│  事务 A                    事务 B                              │
│  UPDATE Orders             UPDATE Orders                        │
│  SET Amount=100            SET Amount=200                        │
│  WHERE OrderID=1           WHERE OrderID=2                      │
│  获取 U 锁(OrderID=1)      获取 U 锁(OrderID=2)                │
│  U 与 U 兼容 ──▶ 都不阻塞                                          │
│  读取数据                   读取数据                              │
│  准备升级 X                 准备升级 X                           │
│  A 升级 X(OrderID=1)      等待 A 释放...                       │
│  B 尝试升级 X(OrderID=2) ← 等待 A                              │
│                           事务 B 超时或回滚                      │
└─────────────────────────────────────────────────────────────────┘
```

### U 锁兼容性

| 锁模式 | S | U | X |
|---|---|---|---|
| S | ✓ | ✓ | ✗ |
| U | ✓ | ✗ | ✗ |
| X | ✗ | ✗ | ✗ |

## 意向锁（Intent Locks）

意向锁表示在低粒度上持有锁的意图。

```sql
-- 锁定表中一行
BEGIN TRANSACTION;
    SELECT * FROM Orders WITH (UPDLOCK) WHERE OrderID = 1;
    -- 1. 获取 OrderID=1 的更新锁
    -- 2. 获取 Orders 表的 IX 锁（意向排他）
    -- 3. 获取页的 IX 锁
COMMIT;
```

### 意向锁类型

| 类型 | 含义 | 兼容 |
|---|---|---|
| IS | 意图在低粒度上获取共享锁 | IS、IU、S、SIX |
| IU | 意图在低粒度上获取更新锁 | IS、IU、S、U |
| IX | 意图在低粒度上获取排他锁 | IS、IX |
| SIX | 对整个表加共享锁，意图在低粒度加排他锁 | IS |

### 意向锁的作用

1. **通知其他事务**：当前表或页上已有锁
2. **优化锁获取**：获取意向锁后再获取行锁
3. **避免锁冲突检测**：检查意向锁即可判断是否冲突

```
┌─────────────────────────────────────────────────────────────────┐
│  事务 A                    事务 B                               │
│  UPDATE Orders             ALTER TABLE Orders ADD COLUMN...     │
│  SET Amount=100            获取 Sch-M 锁                        │
│  WHERE OrderID=1           需要 TABLE 锁                        │
│  获取 IX TABLE 锁          检查 IX 锁存在                        │
│  获取 IX PAGE 锁           ──▶ 等待 A 释放 IX                    │
│  获取 X ROW 锁             ──▶ 阻塞！                           │
└─────────────────────────────────────────────────────────────────┘
```

## 锁升级（Lock Escalation）

### 触发条件

SQL Server 会将多个行锁或页锁升级为表锁：

| 触发条件 | 默认阈值 |
|---|---|
| 单语句在单个表上持有锁数 | 5000 |
| 锁内存占用 | 40% 总锁池 |
| 语句执行超时 | 自动升级 |

### 锁升级行为

```sql
-- 查看锁升级统计
SELECT * FROM sys.dm_db_index_operational_stats(NULL, NULL, NULL, NULL)
WHERE lock Escalation_desc != 'NONE';

-- 查看锁升级事件
SELECT * FROM sys.dm_xe_session_events
WHERE event_name LIKE '%lock%escalation%';
```

### 锁升级的影响

锁升级可能导致严重的阻塞：

```
锁升级前：
┌─────────────────────────────────────────────────────────────────┐
│  事务 A：更新 10000 行                                          │
│  获取 10000 个行锁                                              │
│  其他事务可以访问未锁定的行                                      │
└─────────────────────────────────────────────────────────────────┘

锁升级后：
┌─────────────────────────────────────────────────────────────────┐
│  事务 A：升级为表锁                                             │
│  获取 TABLE 锁                                                  │
│  其他事务完全阻塞！                                              │
└─────────────────────────────────────────────────────────────────┘
```

### 控制锁升级

```sql
-- 禁用锁升级
ALTER TABLE Orders SET (LOCK_ESCALATION = DISABLE);
-- 可能导致过多锁，消耗大量内存

-- 表级别锁升级（默认）
ALTER TABLE Orders SET (LOCK_ESCALATION = TABLE);
-- 升级为整个表的锁

-- 分区级别锁升级（分区表）
ALTER TABLE Orders SET (LOCK_ESCALATION = AUTO);
-- 只升级到分区级别，不影响其他分区
```

### 避免锁升级

| 方法 | 说明 |
|---|---|
| 减少事务中的锁数量 | 批量操作分批提交 |
| 使用低隔离级别 | 减少锁持有时间 |
| 使用快照隔离 | 避免锁定 |
| 禁用锁升级 | `LOCK_ESCALATION = DISABLE` |
| 调整锁升级阈值 | `ALTER TABLE ... SET (LOCK_ESCALATION_THRESHOLD = N)` |

## 架构锁（Schema Locks）

### Sch-M（架构修改锁）

- 表结构正在被修改
- 排他锁，阻塞所有其他操作

### Sch-S（架构稳定锁）

- 编译查询
- 与大多数锁兼容
- 不与 Sch-M 兼容

```sql
-- 任何 DDL 操作需要 Sch-M 锁
ALTER TABLE Orders ADD NewColumn NVARCHAR(100);
-- 获取 Sch-M 锁，阻塞所有访问

-- 查询编译需要 Sch-S 锁
SELECT * FROM Orders WHERE OrderID = 1;
-- 获取 Sch-S 锁
```

## 键范围锁（Range Locks）

在 SERIALIZABLE 隔离级别下，保护索引范围内的行。

```sql
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;

BEGIN TRANSACTION;
    SELECT * FROM Orders
    WHERE OrderID BETWEEN 100 AND 200;
    -- 锁定 OrderID 100-200 的范围
    -- 防止其他事务在此范围内插入新行（幻读）
COMMIT;
```

### 键范围锁类型

| 类型 | 说明 |
|---|---|
| RangeS-S | 范围共享，索引范围共享锁 |
| RangeS-U | 范围共享更新 |
| RangeX-X | 范围排他 |
| RangeI-N | 范围插入（新索引键） |
| RangeI-S | 范围插入共享 |
| RangeI-X | 范围插入排他 |
| RangeX-S | 范围排他共享 |
| RangeX-U | 范围排他更新 |

## 查看锁信息

### 当前锁

```sql
-- 查看当前锁（需要适当权限）
SELECT 
    request_session_id AS SPID,
    DB_NAME(resource_database_id) AS DatabaseName,
    resource_type AS ResourceType,
    resource_description AS Resource,
    request_mode AS Mode,
    request_status AS Status,
    request_owner_type AS OwnerType
FROM sys.dm_tran_locks
WHERE resource_database_id = DB_ID()
ORDER BY request_session_id;

-- 查看锁定的对象
SELECT 
    l.request_session_id AS SPID,
    OBJECT_NAME(p.object_id) AS ObjectName,
    p.object_id,
    l.resource_type AS ResourceType,
    l.request_mode AS Mode,
    l.request_status AS Status
FROM sys.dm_tran_locks l
LEFT JOIN sys.partitions p ON l.resource_associated_entity_id = p.hobt_id
WHERE l.resource_database_id = DB_ID()
ORDER BY l.request_session_id;
```

### 阻塞信息

```sql
-- 查看阻塞链
SELECT 
    blocked.session_id AS BlockedSPID,
    blocker.session_id AS BlockerSPID,
    blocker.login_name,
    blocker.host_name,
    blocked.wait_time / 1000.0 AS WaitSeconds,
    blocked.wait_type,
    blocked.resource_description,
    qt_blocked.text AS BlockedQuery,
    qt_blocker.text AS BlockerQuery
FROM sys.dm_exec_requests blocked
JOIN sys.dm_exec_requests blocker ON blocked.blocking_session_id = blocker.session_id
CROSS APPLY sys.dm_exec_sql_text(blocked.sql_handle) qt_blocked
CROSS APPLY sys.dm_exec_sql_text(blocker.sql_handle) qt_blocker
WHERE blocked.session_id > 50;
```

### 锁等待统计

```sql
-- 查看锁等待统计
SELECT 
    wait_type,
    waiting_tasks_count,
    wait_time_ms,
    signal_wait_time_ms,
    wait_time_ms - signal_wait_time_ms AS resource_wait_ms
FROM sys.dm_os_wait_stats
WHERE wait_type LIKE 'LCK%'
ORDER BY wait_time_ms DESC;
```

## 锁内存使用

```sql
-- 查看锁内存使用
SELECT 
    cntr_value / 1024.0 AS LockMemoryKB,
    (SELECT cntr_value FROM sys.dm_os_performance_counters 
     WHERE counter_name = 'Target Server Memory (KB)') AS TargetMemoryKB,
    (SELECT cntr_value FROM sys.dm_os_performance_counters 
     WHERE counter_name = 'Total Server Memory (KB)') AS TotalMemoryKB
FROM sys.dm_os_performance_counters
WHERE counter_name = 'Granted Workspace Memory (KB)';

-- 查看最大锁数量
SELECT * FROM sys.dm_os_performance_counters
WHERE counter_name LIKE '%Lock%';
```
