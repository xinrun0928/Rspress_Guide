# 事务日志与 WAL

「事务提交了，数据却丢了？」

这种情况在 SQL Server 中理论上不可能发生，因为有 **Write-Ahead Logging (WAL)** 机制保驾护航。

理解 WAL，是掌握 SQL Server 数据安全和恢复机制的钥匙。

## Write-Ahead Logging 原理

WAL 的核心原则：**在修改数据页之前，必须先将修改记录写入事务日志。**

```
┌─────────────────────────────────────────────────────────────────┐
│                 Write-Ahead Logging 流程                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  BEGIN TRANSACTION                                              │
│       │                                                         │
│       ▼                                                         │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │ 1. 生成日志记录                                         │    │
│  │    - 修改了什么（操作类型）                              │    │
│  │    - 修改前是什么（旧值）                                │    │
│  │    - 修改后是什么（新值）                                │    │
│  │    - 事务 ID、LSN                                       │    │
│  └─────────────────────────────────────────────────────────┘    │
│       │                                                         │
│       ▼                                                         │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │ 2. 日志刷盘（必须先完成）                                │    │
│  │    - 日志记录写入 .ldf 文件                              │    │
│  │    - 调用 FlushLog（强制刷盘）                            │    │
│  └─────────────────────────────────────────────────────────┘    │
│       │                                                         │
│       ▼                                                         │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │ 3. 修改数据页（在内存中）                                 │    │
│  │    - Buffer Pool 中的页被标记为脏页                      │    │
│  │    - 尚未写入磁盘                                       │    │
│  └─────────────────────────────────────────────────────────┘    │
│       │                                                         │
│       ▼                                                         │
│  COMMIT TRANSACTION                                            │
│       │                                                         │
│       ▼                                                         │
│  检查点进程异步将脏页写入磁盘                                    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 为什么 WAL 能保证数据安全

### 崩溃恢复过程

假设数据库在运行中突然崩溃，已提交事务的数据是否安全？

```
崩溃场景：
- 事务 T1 已提交，日志已刷盘，数据页未刷盘
- 事务 T2 未提交，修改在内存中

恢复过程：
1. 分析阶段：扫描日志，确定活动事务
2. Redo 阶段：重做所有已提交事务（数据页可能已恢复）
3. Undo 阶段：回滚未提交事务（通过日志中的旧值）

结果：T1 的修改被恢复，T2 的修改被回滚
```

**关键点**：即使数据页丢失，只要日志还在，就能恢复所有已提交事务的修改。

## LSN 与日志链

### 日志序列号（Log Sequence Number）

每个日志记录有一个唯一的 **LSN**，格式为 `FileID:PageID:SlotID`：

```sql
-- 查看日志记录和 LSN
SELECT 
    [Current LSN],
    Operation,
    Context,
    [Transaction ID],
    [Begin Time],
    [End Time],
    [Lock Information]
FROM fn_dblog(NULL, NULL);
```

### 日志链

日志链（Log Chain）是一系列连续的日志记录，通过 LSN 连接：

```
┌─────────────────────────────────────────────────────────────────┐
│                        日志链                                     │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────┐   ┌─────┐   ┌─────┐   ┌─────┐   ┌─────┐              │
│  │LSN 1│──▶│LSN 2│──▶│LSN 3│──▶│LSN 4│──▶│LSN 5│──▶ ...        │
│  │T1   │   │T1   │   │T2   │   │T1 C │   │T2   │              │
│  │BEGAN│   │UPDATE│   │BEGAN│   │COMM │   │ROLL │              │
│  └─────┘   └─────┘   └─────┘   └─────┘   └─────┘              │
│    │                                                         │
│    │ 完整备份后开始新日志链                                     │
│    ▼                                                           │
│  ┌─────┐   ┌─────┐                                            │
│  │LSN 1│──▶│LSN 2│──▶ ...                                      │
│  │T3   │   │T3   │                                            │
│  │BEGAN│   │COMM │                                            │
│  └─────┘   └─────┘                                            │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 检查点（Checkpoint）

### 检查点的作用

检查点（Checkpoint）是 SQL Server 定期将脏页写入磁盘的机制，减少崩溃恢复时间。

```sql
-- 查看检查点信息
SELECT 
    database_name,
    last_checkpoint_lsn,
    checkpoint_lsn,
    recovery_lsn,
    family_lsn
FROM sys.database_recovery_status
WHERE database_id = DB_ID();
```

### 检查点触发条件

| 条件 | 说明 |
|---|---|
| 恢复间隔到达 | `recovery interval` 配置（默认 0=自动） |
| 日志 70% 满 | 防止日志写满 |
| 完整备份前 | 减少日志备份量 |
| `CHECKPOINT` 命令 | 手动触发 |
| `ALTER INDEX REBUILD` | 大规模索引操作后 |
| 表/索引删除 | `DROP TABLE/INDEX` |

### 增量检查点（SQL Server 2016+）

增量检查点根据日志量而非时间间隔触发，更智能：

```sql
-- 查看恢复间隔设置
sp_configure 'recovery interval';
-- 默认 0 表示自动（通常约 1 分钟）
```

## 恢复模式

### 三种恢复模式

| 恢复模式 | 日志备份 | 数据丢失风险 | 典型场景 |
|---|---|---|---|
| FULL | 需要 | 最小 | 生产数据库 |
| BULK_LOGGED | 需要 | 最小到中等 | 大批量导入 |
| SIMPLE | 不需要 | 可能丢失数据 | 开发/测试 |

```sql
-- 查看当前恢复模式
SELECT 
    name AS DatabaseName,
    recovery_model_desc
FROM sys.databases;

-- 修改恢复模式
ALTER DATABASE [YourDB] SET RECOVERY FULL;
ALTER DATABASE [YourDB] SET RECOVERY BULK_LOGGED;
ALTER DATABASE [YourDB] SET RECOVERY SIMPLE;
```

### 模式切换影响

```sql
-- FULL → BULK_LOGGED
-- 大批量操作变成最小日志，切换后立即备份日志

-- BULK_LOGGED → FULL
-- 强制日志备份捕获所有活动日志
BACKUP LOG [YourDB] TO DISK = 'NUL:';
-- 然后完整备份
BACKUP DATABASE [YourDB] TO DISK = 'YourDB_Full.bak';
```

## 日志备份

### 备份类型

```
┌─────────────────────────────────────────────────────────────────┐
│                        备份策略                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  完整备份 ──▶ 差异备份 ──▶ 日志备份 ──▶ 日志备份 ──▶ 日志备份   │
│     │           │              │              │              │   │
│     └───────────┴──────────────┴──────────────┴──────────────┘   │
│                               │                                   │
│                          日志链                                   │
│                                                                 │
│  恢复顺序：                                                      │
│  1. 完整备份                                                    │
│  2. 最新差异备份（如果有）                                       │
│  3. 差异点之后的所有日志备份                                      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 备份脚本

```sql
-- 完整备份
BACKUP DATABASE [YourDB]
TO DISK = 'D:\Backup\YourDB_Full.bak'
WITH COMPRESSION, CHECKSUM, INIT;

-- 差异备份
BACKUP DATABASE [YourDB]
TO DISK = 'D:\Backup\YourDB_Diff.bak'
WITH DIFFERENTIAL, COMPRESSION, CHECKSUM;

-- 日志备份
BACKUP LOG [YourDB]
TO DISK = 'D:\Backup\YourDB_Log.trn'
WITH COMPRESSION, CHECKSUM;

-- 镜像备份（提高安全性）
BACKUP DATABASE [YourDB]
TO DISK = 'D:\Backup\YourDB_Full.bak'
MIRROR TO DISK = 'E:\Backup\YourDB_Full_Mirror.bak'
WITH FORMAT;
```

### 日志链保护

```sql
-- 检查日志链完整性
RESTORE VERIFYONLY 
FROM DISK = 'D:\Backup\YourDB_Full.bak';

RESTORE VERIFYONLY 
FROM DISK = 'D:\Backup\YourDB_Log.trn'
WITH NORECOVERY;
```

## 恢复模型与日志管理

### 完整恢复模式

```sql
-- 必须定期备份日志，否则日志文件无限增长
-- 日志备份后，VLF 被标记为可重用

-- 查看日志使用情况
SELECT 
    DB_NAME(database_id) AS DatabaseName,
    total_log_size_in_bytes / 1024.0 / 1024 AS TotalLogMB,
    used_log_space_in_bytes / 1024.0 / 1024 AS UsedLogMB,
    used_log_space_in_percent AS UsedPercent
FROM sys.dm_db_log_space_usage;
```

### 简单恢复模式

```sql
-- 日志由检查点自动截断
-- 无法恢复到任意时间点
-- 只能恢复到上次备份点

-- 切换到简单模式后，之前的日志备份链断裂
-- 再次切换回完整模式需要新的完整备份
ALTER DATABASE [YourDB] SET RECOVERY SIMPLE;
```

## 时间点恢复

### 恢复到时间点

```sql
-- 恢复到 2024-03-01 14:30:00
RESTORE DATABASE [YourDB]
FROM DISK = 'D:\Backup\YourDB_Full.bak'
WITH NORECOVERY;

RESTORE LOG [YourDB]
FROM DISK = 'D:\Backup\YourDB_Log1.trn'
WITH NORECOVERY;

RESTORE LOG [YourDB]
FROM DISK = 'D:\Backup\YourDB_Log2.trn'
WITH RECOVERY,
STOPAT = '2024-03-01 14:30:00';
```

### 恢复到命名时间点

```sql
-- 恢复到标记点
RESTORE LOG [YourDB]
FROM DISK = 'D:\Backup\YourDB_Log.trn'
WITH RECOVERY,
STOPATMARK = 'AfterOrderUpdate',
AFTER = '2024-03-01 14:30:00';
```

## 日志与性能

### 日志写入瓶颈

日志写入是顺序 I/O，应该是高性能的。但如果出现瓶颈：

```sql
-- 查看日志相关等待
SELECT 
    wait_type,
    waiting_tasks_count,
    wait_time_ms,
    signal_wait_time_ms
FROM sys.dm_os_wait_stats
WHERE wait_type LIKE '%LOG%'
   OR wait_type = 'WRITELOG'
ORDER BY wait_time_ms DESC;

-- 查看日志 I/O 延迟
SELECT 
    database_id,
    num_of_writes,
    bytes_written / 1024.0 / 1024 AS WriteMB,
    io_stall_write_ms / num_of_writes AS AvgWriteLatencyMS
FROM sys.dm_io_virtual_file_stats(DB_ID(), 2);  -- 2 = 日志文件
```

### 优化日志性能

| 优化项 | 说明 |
|---|---|
| 使用专用日志磁盘 | 日志磁盘应该是顺序 I/O 优化的高速磁盘 |
| 减少日志碎片 | 创建足够大的日志文件，避免自动增长 |
| 批量提交 | 减少日志刷盘次数 |
| 避免长事务 | 长事务阻塞日志截断 |

```sql
-- 创建合适大小的日志文件
ALTER DATABASE [YourDB] MODIFY FILE (
    NAME = 'YourDB_log',
    SIZE = 10GB,
    MAXSIZE = 50GB,
    FILEGROWTH = 1GB
);
```

## 监控日志

```sql
-- 完整日志监控查询
SELECT 
    DB_NAME(database_id) AS DatabaseName,
    total_log_size_in_bytes / 1024.0 / 1024 AS TotalLogMB,
    used_log_space_in_bytes / 1024.0 / 1024 AS UsedLogMB,
    used_log_space_in_percent AS UsedPercent,
    log_space_in_bytes_since_last_backup / 1024.0 / 1024 AS LogSinceBackupMB
FROM sys.dm_db_log_space_usage;

-- 查看活动事务
SELECT 
    transaction_id,
    session_id,
    database_id,
    transaction_begin_time,
    transaction_type_desc,
    transaction_state,
    transaction_status
FROM sys.dm_tran_active_transactions;

-- 查看日志记录
SELECT TOP 100
    [Current LSN],
    Operation,
    Context,
    [Transaction ID],
    [Begin Time],
    [End Time]
FROM fn_dblog(NULL, NULL)
ORDER BY [Current LSN] DESC;
```
