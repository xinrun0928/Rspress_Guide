# 页面与区存储结构

凌晨 2 点，DBA 发现数据库响应缓慢，查看等待类型发现大量 `PAGEIOLATCH_SH`。这是典型的 I/O 等待，问题指向了数据页的读写性能。

理解 SQL Server 的物理存储结构——**页（Page）** 和 **区（Extent）**——是排查这类问题的前提。

## 为什么需要了解存储结构

大多数开发者只知道「数据库存储数据」，却不清楚数据在磁盘上是如何组织的。这种认知空白导致：

- 不知道表的数据量增长会有多大
- 无法理解碎片对性能的影响
- 调优索引时只关注查询计划，不考虑物理存储
- 无法诊断 I/O 瓶颈的根本原因

SQL Server 的存储结构设计有其历史原因。8KB 的页大小在当时的硬件条件下是平衡点：足够大以减少 I/O 次数，足够小以避免内存浪费。

## 页的结构

### 页面概述

SQL Server 的最小存储单元是 **页（Page）**，大小固定为 **8192 字节（8KB）**。页是数据读写的基本单位，无论是从磁盘读取还是写入内存，都是以页为单位。

页的类型包括：

| 页类型 | 标识 | 用途 |
|---|---|---|
| 数据页 | 1 | 存储数据行 |
| 索引页 | 2 | 存储索引条目 |
| 文本/图像页 | 3 | 存储大对象（TEXT、IMAGE、NVARCHAR(MAX)） |
| GAM 位图页 | 6 | 全局分配映射 |
| SGAM 位图页 | 7 | 混合区分配映射 |
| IAM 页 | 10 | 索引分配映射 |
| PFS 页 | 11 | 页面可用空间 |

### 页头（Page Header）

每个页的前 96 字节是页头，包含元数据信息：

```
┌────────────────────────────────────────────────────────────────┐
│                        页头 (96 字节)                          │
├────────────────────────────────────────────────────────────────┤
│  m_pageId (4B)        │  页ID：文件号 + 页号                   │
│  m_headerVersion (1B)  │  页头版本，始终为 1                    │
│  m_type (1B)          │  页类型：1=数据，2=索引                │
│  m_typeFlagBits (1B)  │  类型标志位                           │
│  m_level (1B)         │  索引层级（叶级为 0）                  │
│  m_flagBits (2B)      │  页面标志                             │
│  m_pageCheckSum (4B)  │  校验和（可选）                       │
│  m_pageAccessMode (2B) │  访问模式                             │
├────────────────────────────────────────────────────────────────┤
│  m_allocUnitId (8B)   │  分配单元ID                           │
│  m_LSN (10B)          │  日志序列号（最后修改）                │
│  m_xactReserved (4B)  │  事务保留空间                         │
│  m_fseqNo (4B)         │  文件序列号                           │
│  m_policyFlag (4B)    │  策略标志                             │
│  m_pageLatency (4B)   │  页面延迟统计                         │
│  m_ghostRecCnt (2B)   │  幽灵记录数                           │
│  m_reserved (4B)      │  保留空间                             │
│  m_xdesId (8B)        │  扩展分配描述符ID                     │
│  m_ghostBackups (24B) │  幽灵备份信息                         │
└────────────────────────────────────────────────────────────────┘
```

页头的 `m_type` 字段标识页的类型。查询数据文件时，你看到的 1 表示数据页，2 表示索引页。

### 行偏移数组

页的最后 36-72 字节是**行偏移数组**。数组中的每个条目占 2 字节，记录对应行在页中的起始位置。

```
┌─────────────────────────────────────────────────────────────────┐
│                     页结构完整布局                                │
├─────────────────────────────────────────────────────────────────┤
│  页头 (96 字节)                                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐                       │
│  │  行 1    │  │  行 2    │  │  行 N    │   数据区              │
│  │ (var col)│  │ (var col)│  │ (var col)│   (约 8060 字节)      │
│  └──────────┘  └──────────┘  └──────────┘                       │
│                                                                  │
├─────────────────────────────────────────────────────────────────┤
│  0x0000 │ 0x0010 │ 0x0020 │ ... │ 0x1F90 │                       │
│         ↑                                                    ↑  │
│         │                                                    │  │
│  行偏移数组 ← 每行 2 字节，记录行起始位置                    │  │
│                                                              │  │
│  实际数组大小根据行数动态增长                                  │  │
└──────────────────────────────────────────────────────────────┘
```

:::tip
行偏移数组从页的末尾向前生长，数据行从页头之后向后生长。当两者相遇时，页就满了。这就是为什么页的填充率不是 100%——需要保留间隙给行偏移数组增长。
:::

### 数据行格式

SQL Server 2019+ 支持两种行格式：**传统行格式（ROW_OVERFLOW）** 和 **可变长度行格式（COMPRESSED）**。默认是传统行格式。

**传统行格式**的每行包含：

```
┌─────────────────────────────────────────────────────────────┐
│  固定长度列                      │  列计数  │  可变长度列    │
├─────────────────────────────────────────────────────────────┤
│  Status Bitmap A (2B) │ Status Bitmap B (2B) │  变长列数组  │
└─────────────────────────────────────────────────────────────┘
```

- **状态位图**：记录行状态（删除标记、NULL位图存在、版本链标志）
- **NULL 位图**：记录哪些列可以为 NULL
- **变长列数组**：每列记录两个字节的偏移量

**行大小计算**：

```sql
-- 查看表的行大小
SELECT 
    t.name AS TableName,
    SUM(s.column_size) AS TotalFixedBytes,
    COUNT(*) AS ColumnCount,
    2 + 2 AS NullBitmapOverhead
FROM sys.tables t
JOIN sys.columns s ON t.object_id = s.object_id
WHERE t.name = 'YourTable'
GROUP BY t.name;

-- 实际查看页面内容（需要 DAC 连接）
DBCC TRACEON(3604);
DBCC PAGE('DatabaseName', 1, 1000, 3);  -- 文件1，页1000，详细模式
```

### 行溢出与行迁移

SQL Server 的行大小限制为 **8060 字节**（不含 text/image）。当单行数据超过这个限制时，超出部分会存储在**行溢出页（ROW_OVERFLOW_PAGE）** 中。

```sql
CREATE TABLE LargeRowTest (
    Id INT PRIMARY KEY,
    FixedCol CHAR(5000),          -- 固定长度列
    OverflowCol1 VARCHAR(8000),   -- 可能溢出
    OverflowCol2 VARCHAR(5000)   -- 可能溢出
);

INSERT INTO LargeRowTest VALUES (1, REPLICATE('A', 5000), REPLICATE('B', 8000), REPLICATE('C', 5000));
-- 实际行大小超过 8060 字节，部分数据被迁移到溢出页
```

当 `OverflowCol1` 被 UPDATE 导致长度增加时，如果原页放不下，数据会迁移到溢出页，原位置留下一个 **24 字节的行指针**。

:::warning
行溢出会导致查询性能下降。包含溢出列的 SELECT * 查询需要额外的 I/O 操作。设计表结构时，应将大字段拆分到独立表或使用 `SPARSE` 列。
:::

## 区（Extent）

### 区的概念

**区（Extent）** 是 SQL Server 空间分配的基本单位，由 **8 个连续的页** 组成，大小为 **64KB**。

当表或索引需要扩展空间时，SQL Server 一次至少分配一个区。8 个页的连续分配确保了相关数据在磁盘上尽量靠近，减少 I/O 次数。

### 区的类型

SQL Server 有两种区类型：

**统一区（Uniform Extent）**：区内的全部 8 个页属于**同一个对象**。分配和释放由 GAM 和 SGAM 页管理。

**混合区（Mixed Extent）**：区内的 8 个页可以分配给**最多 8 个不同对象**。新创建的小表优先使用混合区（先分配 1 页），只有超过 8 页时才切换到统一区。

```
┌─────────────────────────────────────────────────────────────────┐
│                      混合区 (Mixed Extent)                      │
├────┬────┬────┬────┬────┬────┬────┬────┤
│ Obj│ Obj│ Obj│ Obj│ Obj│ Obj│ Obj│ Obj│                        │
│  A │  B │  A │  C │  D │  B │  E │  F │  共 8 个对象共享        │
├────┴────┴────┴────┴────┴────┴────┴────┤
│  表 A 可能只有 2 页，表 B 有 1 页...                            │
└─────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                      统一区 (Uniform Extent)                   │
├─────────────────────────────────────────────────────────────────┤
│  表 C 的 8 页全部属于表 C                                       │
├─────────────────────────────────────────────────────────────────┤
│  ████████████████████████████████████████████████████████       │
└─────────────────────────────────────────────────────────────────┘
```

### GAM、SGAM 与 IAM

SQL Server 使用三种特殊页管理区分配：

**GAM（Global Allocation Map）页**：每 64000 个区（约 4GB 数据）占用 1 个 GAM 页。每位记录对应区是否已分配（0=已分配，1=空闲）。

**SGAM（Shared Global Allocation Map）页**：记录混合区的分配状态（0=非混合空闲，1=混合空闲，0=已用）。

**IAM（Index Allocation Map）页**：跟踪特定表或索引使用的区和页。每个分配单元（堆或索引）至少有一个 IAM 页。

```sql
-- 查看表的 IAM 页
DBCC IND('DatabaseName', 'TableName', 1);

-- 结果中的 PageType：
-- 10 = IAM 页
-- 1 = 数据页
-- 2 = 索引页
```

## B+ 树与页的关系

### 索引的页结构

B+ 树索引由**索引页**组成，每个索引页的大小同样是 8KB。页的类型为 2（索引页），其结构与数据页类似：

- 页头 96 字节
- 索引条目（键值 + 子页指针）
- 行偏移数组

**索引页的层级**：
- **叶级（Level 0）**：存储索引键和行定位器（聚集索引是数据行，非聚集索引是聚集索引键或 RID）
- **非叶级（Level > 0）**：存储索引键和子页指针，指向下一级页

```
                          根节点 (Level 2)
                         ┌─────────────┐
                         │ Key: 100    │──▶ [子页指针]
                         │ Key: 200    │──▶ [子页指针]
                         │ Key: 300    │──▶ [子页指针]
                         └─────────────┘
                                │
                    ┌───────────┴───────────┐
              中间节点 (Level 1)        中间节点 (Level 1)
              ┌─────────────┐            ┌─────────────┐
              │ Key: 50     │            │ Key: 150    │
              │ Key: 100    │            │ Key: 200    │
              └─────────────┘            └─────────────┘
                      │                          │
        ┌─────────────┼─────────────┐    ┌──────┴──────┐
   叶级 (Level 0)                  叶级 (Level 0)
   ┌─────────────┐                ┌─────────────┐
   │ Key: 1      │                │ Key: 101    │
   │ Key: 23     │                │ Key: 156    │
   │ Key: 47     │                │ Key: 189    │
   └─────────────┘                └─────────────┘
        ↓                              ↓
   数据行（聚集索引）              数据行（聚集索引）
```

### 深度计算

B+ 树的深度（Level 数）决定了查找一个值需要的 **I/O 次数**。深度越大，需要访问的页越多。

**深度计算公式**：

| 深度 | 说明 | 最大叶节点数（每页 100 条索引） |
|---|---|---|
| 0 | 叶级 | 约 8 页（800 条索引） |
| 1 | 根→叶 | 约 800 页（80,000 条索引） |
| 2 | 根→中间→叶 | 约 80,000 页（8,000,000 条索引） |
| 3 | 三层树 | 约 8,000,000 页（8亿条索引） |

```sql
-- 查看索引的深度和页数
SELECT 
    OBJECT_NAME(i.object_id) AS TableName,
    i.name AS IndexName,
    i.index_id,
    i.type_desc,
    ips.depth AS IndexDepth,
    ips.page_count AS TotalPages,
    ips.record_count AS TotalRecords,
    ips.avg_page_space_used_in_percent AS PageFillPercent
FROM sys.dm_db_index_physical_stats(
    DB_ID(), 
    OBJECT_ID('YourTable'), 
    NULL, 
    NULL, 
    'DETAILED'
) ips
JOIN sys.indexes i ON ips.object_id = i.object_id AND ips.index_id = i.index_id
WHERE ips.alloc_unit_type_desc = 'IN_ROW_DATA'
ORDER BY ips.depth DESC;
```

## 碎片与页分裂

### 内部碎片

内部碎片发生在**页内空间浪费**时。页被频繁 UPDATE/DELETE 后，数据行重新分布但页未重新组织，导致页的填充率下降。

内部碎片不影响查询性能，但会：
- 浪费存储空间
- 增加 I/O 开销（读取相同数据量需要更多页）
- 降低缓冲池效率

### 外部碎片

外部碎片发生在**页的逻辑顺序与物理顺序不一致**时。当页被拆分时，新页可能分配在磁盘的不同位置（已分配给其他对象），导致顺序扫描需要大量随机 I/O。

```
理想状态：页按顺序分配
┌──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┐
│ Page │ Page │ Page │ Page │ Page │ Page │ Page │ Page │
│  1   │  2   │  3   │  4   │  5   │  6   │  7   │  8   │
└──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┘
  ↓顺序读取，只需 1 次顺序 I/O

外部碎片：页被分散到不同位置
┌──────┐                    ┌──────┐
│ Page │  间隙               │ Page │
│  1   │  (已被其他对象占用)  │  5   │
└──────┘                    └──────┘
      ↓                           ↓
┌──────┐                    ┌──────┐
│ Page │                    │ Page │
│  3   │                    │  7   │
└──────┘                    └──────┘
  ↓顺序扫描需要 4 次随机 I/O
```

### 页分裂

页分裂发生在**插入或更新导致页溢出**时。当新行需要插入到已满页的中间位置时，SQL Server 分配新页，将约一半数据移动到新页。

页分裂的影响：
- 产生外部碎片（新页可能不在顺序位置）
- 降低插入性能
- 造成索引不平衡

```sql
-- 插入导致页分裂的示例
-- 聚集索引键为自增 ID，数据追加到末尾，页分裂风险低
-- 但如果聚集索引键是 UUID 或非顺序值，频繁插入中间位置

CREATE TABLE FragmentationTest (
    Id INT IDENTITY(1,1) PRIMARY KEY,  -- 顺序插入，碎片少
    Data VARCHAR(100)
);

-- vs

CREATE TABLE FragmentationTest2 (
    Id UNIQUEIDENTIFIER DEFAULT NEWID() PRIMARY KEY,  -- UUID，碎片多
    Data VARCHAR(100)
);
```

## 碎片修复

### REORGANIZE vs REBUILD

| 操作 | REORGANIZE | REBUILD |
|---|---|---|
| 方式 | 物理重排序叶级页 | 重建整个索引 |
| 资源占用 | 较少，单线程 | 较高，可并行 |
| 事务类型 | 联机，自动提交 | 可在事务中 |
| 碎片范围 | `< 30%` | `> 30%` |
| 恢复级别 | 页面级恢复 | 索引级恢复 |

```sql
-- 查看碎片
SELECT 
    OBJECT_NAME(ips.object_id) AS TableName,
    i.name AS IndexName,
    ips.avg_fragmentation_in_percent AS Fragmentation,
    ips.avg_page_space_used_in_percent AS PageFill
FROM sys.dm_db_index_physical_stats(
    DB_ID(), 
    OBJECT_ID('YourTable'),
    NULL, 
    NULL, 
    'DETAILED'
) ips
JOIN sys.indexes i ON ips.object_id = i.object_id AND ips.index_id = i.index_id
WHERE ips.page_count > 8;

-- 碎片 > 30%：重建
ALTER INDEX idx_name ON YourTable REBUILD;

-- 碎片 5%~30%：重新组织
ALTER INDEX idx_name ON YourTable REORGANIZE;

-- 填充因子（页填充率）
ALTER INDEX idx_name ON YourTable REBUILD WITH (FILLFACTOR = 80);
```

## 总结

理解 SQL Server 的存储结构，关键是掌握三个层次：

1. **行级**：数据行由固定列和变长列组成，受 8060 字节限制
2. **页级**：8KB 页是读写基本单位，由页头、行数据、行偏移数组组成
3. **区级**：8 个连续页构成区，区分配由 GAM/SGAM/IAM 管理

碎片问题源于页分裂和空间重用，合理设计索引和定期维护是保持高效存储的关键。
