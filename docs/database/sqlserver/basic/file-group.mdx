# 数据库文件与文件组

「这个表太大了，怎么把它放到 SSD 上？」

这个问题的答案藏在 SQL Server 的文件组设计中。通过将表或索引创建在不同的文件组，你可以精确控制数据的物理存储位置。

## 数据库文件

SQL Server 数据库由三类文件组成：

### 主数据文件（Primary Data File）

- **扩展名**：`.mdf`
- **数量**：每个数据库有且仅有一个
- **内容**：数据库启动信息、系统表、用户表和索引
- **特性**：包含文件组的起始页

```sql
-- 查看数据库文件信息
SELECT 
    name AS FileName,
    physical_name AS PhysicalPath,
    type_desc AS FileType,
    size * 8 / 1024.0 AS SizeMB,
    max_size * 8 / 1024.0 AS MaxSizeMB,
    growth * 8 / 1024.0 AS GrowthMB,
    is_percent_growth
FROM sys.database_files;
```

### 次要数据文件（Secondary Data File）

- **扩展名**：`.ndf`
- **数量**：可选，可以有多个
- **内容**：用户数据和索引
- **用途**：分散 I/O、扩展存储、利用不同磁盘

### 事务日志文件（Transaction Log File）

- **扩展名**：`.ldf`
- **数量**：至少一个，可以有多个
- **特性**：不支持文件组，日志记录顺序写入
- **用途**：事务恢复、复制、灾难恢复

## 文件组

### 文件组的概念

**文件组（Filegroup）** 是数据文件的逻辑容器，用于组织和分配数据库对象。每个数据库有一个默认的 **PRIMARY 文件组**，包含主数据文件和系统表。

### 创建文件组

```sql
-- 创建数据库时指定文件组
CREATE DATABASE SalesDB
ON PRIMARY (
    NAME = 'SalesDB_Primary',
    FILENAME = 'D:\Data\SalesDB.mdf',
    SIZE = 100MB,
    MAXSIZE = 10GB,
    FILEGROWTH = 64MB
),
FILEGROUP FG_SSD (
    NAME = 'SalesDB_SSD',
    FILENAME = 'E:\SSD\SalesDB_SSD.ndf',
    SIZE = 500MB,
    MAXSIZE = 50GB,
    FILEGROWTH = 128MB
)
LOG ON (
    NAME = 'SalesDB_Log',
    FILENAME = 'D:\Logs\SalesDB.ldf',
    SIZE = 50MB,
    MAXSIZE = 5GB,
    FILEGROWTH = 32MB
);
```

### 文件组的使用场景

**场景一：热数据与冷数据分离**

```sql
-- 活跃订单放在 SSD 文件组
CREATE TABLE Orders (
    OrderID INT PRIMARY KEY,
    OrderDate DATETIME,
    CustomerID INT,
    Amount DECIMAL(10,2)
) ON FG_SSD;  -- 指定文件组

-- 历史订单放在普通磁盘文件组
CREATE TABLE Orders_Archive (
    OrderID INT PRIMARY KEY,
    OrderDate DATETIME,
    CustomerID INT,
    Amount DECIMAL(10,2)
) ON FG_HDD;  -- 放在另一个文件组
```

**场景二：分区表跨文件组**

```sql
-- 按月份分区，不同月份放在不同文件组
CREATE PARTITION FUNCTION pf_Orders (DATETIME)
AS RANGE RIGHT FOR VALUES ('2024-01-01', '2024-02-01', '2024-03-01');

CREATE PARTITION SCHEME ps_Orders
AS PARTITION pf_Orders
TO (FG_2024Q1, FG_2024Q1, FG_2024Q2, FG_2024Q2);

CREATE TABLE Orders (
    OrderID INT,
    OrderDate DATETIME,
    Amount DECIMAL(10,2)
) ON ps_Orders(OrderDate);
```

**场景三：索引与表分离**

```sql
-- 表放在主文件组，索引放在 SSD 文件组
CREATE TABLE LargeTable (
    ID INT PRIMARY KEY,
    Name VARCHAR(100),
    Data VARCHAR(1000)
) ON [PRIMARY];

CREATE NONCLUSTERED INDEX IX_LargeTable_Name 
ON LargeTable(Name) 
ON FG_SSD;
```

### 默认文件组

```sql
-- 将用户对象创建在 FG_SSD 文件组（设为默认）
ALTER DATABASE SalesDB MODIFY FILEGROUP FG_SSD DEFAULT;

-- 查看当前默认文件组
SELECT 
    name AS FileGroupName,
    is_default AS IsDefault,
    is_read_only AS IsReadOnly
FROM sys.filegroups
WHERE type = 'FG';
```

:::warning
修改默认文件组不会移动现有对象，只影响新创建的对象。
:::

## 空间分配

### 分配单元

SQL Server 为每个表或索引创建一个**分配单元（Allocation Unit）**，跟踪其使用的页和区。分配单元有三种类型：

| 类型 | 说明 | 触发条件 |
|---|---|---|
| IN_ROW_DATA | 行内数据 | 普通数据和索引 |
| LOB_DATA | 大对象数据 | TEXT、IMAGE、NVARCHAR(MAX) |
| ROW_OVERFLOW_DATA | 行溢出数据 | 单行超过 8060 字节 |

```sql
-- 查看表的空间分配
EXEC sp_spaceused 'YourTable', @mode = 'DETAILED';

-- 或使用 DMV
SELECT 
    OBJECT_NAME(object_id) AS TableName,
    allocation_unit_id,
    type_desc,
    total_pages * 8 / 1024.0 AS TotalSpaceMB,
    used_pages * 8 / 1024.0 AS UsedSpaceMB,
    page_free_space_percent
FROM sys.allocation_units au
JOIN sys.partitions p ON au.container_id = p.partition_id
WHERE object_id = OBJECT_ID('YourTable');
```

### 区分配规则

SQL Server 的区分配策略：
1. 小表（`< 8 页`）：从混合区分配单页
2. 中型表（`8~1040 页`）：从混合区分配区，直到超过 8 页
3. 大表（`> 1040 页`）：从统一区分配

这个策略确保小表不会浪费空间，同时大表能够获得连续的磁盘区域。

## 文件管理

### 添加数据文件

```sql
-- 添加新的数据文件到文件组
ALTER DATABASE SalesDB
ADD FILE (
    NAME = 'SalesDB_Data2',
    FILENAME = 'F:\Data\SalesDB_2.ndf',
    SIZE = 500MB,
    MAXSIZE = 20GB,
    FILEGROWTH = 100MB
) TO FILEGROUP FG_SSD;
```

### 删除数据文件

```sql
-- 删除空的数据文件（必须是空的）
ALTER DATABASE SalesDB
REMOVE FILE SalesDB_Data2;
```

### 收缩文件

```sql
-- 收缩数据文件
DBCC SHRINKFILE (N'SalesDB_SSD', 100);  -- 收缩到 100MB

-- 收缩日志文件
DBCC SHRINKFILE (N'SalesDB_Log', 50);
```

:::tip
收缩操作会产生碎片。建议在收缩后重建索引：
```sql
ALTER INDEX ALL ON YourTable REBUILD;
```
:::

## 文件组与备份

文件组备份是**部分备份**的基础，支持只备份关键文件组而忽略历史数据：

```sql
-- 备份特定文件组
BACKUP DATABASE SalesDB
FILEGROUP = 'FG_SSD'
TO DISK = 'D:\Backup\SalesDB_SSD.bak'
WITH COMPRESSION;

-- 恢复文件组
RESTORE DATABASE SalesDB
FILE = 'SalesDB_SSD'
FROM DISK = 'D:\Backup\SalesDB_SSD.bak'
WITH NORECOVERY;
```

## 设计建议

| 建议 | 说明 |
|---|---|
| 分离数据和日志 | 日志文件放在低延迟磁盘，日志写入是顺序的 |
| 分离 tempdb | tempdb 文件数 = CPU 核心数，减少闩锁争用 |
| 使用多个文件 | 大型数据库使用多个数据文件分散 I/O |
| 监控空间增长 | 设置合理的 MAXSIZE，防止磁盘写满 |

```sql
-- 监控空间使用的查询
SELECT 
    DB_NAME(database_id) AS DatabaseName,
    total_log_size_in_bytes / 1024.0 / 1024 AS TotalLogMB,
    used_log_space_in_bytes / 1024.0 / 1024 AS UsedLogMB,
    used_log_space_in_percent AS UsedPercent,
    log_space_in_bytes_since_last_backup / 1024.0 / 1024 AS LogSinceBackupMB
FROM sys.dm_db_log_space_usage;
```

文件组是 SQL Server 物理存储管理的核心，合理设计能够显著提升 I/O 性能和可维护性。
