# SQL Server 整体架构

「SQL Server 是怎么处理一条 SELECT 语句的？」

从客户端发送请求到返回结果，SQL Server 内部经历了协议层、关系引擎、存储引擎的多层协作。理解这个流程，是掌握 SQL Server 性能调优和故障排查的基础。

## 两层架构

SQL Server 采用**客户端-服务器两层架构**，所有组件运行在同一个进程中：

```
┌─────────────────────────────────────────────────┐
│                  SQL Server 进程                 │
│                                                 │
│  ┌─────────────┐     ┌─────────────────────┐    │
│  │   协议层     │────▶│      关系引擎        │    │
│  │  (TDS)      │     │  ┌─────┐ ┌─────┐  │    │
│  └─────────────┘     │  │解析器│→│优化器│→│执行器│    │
│        │             │  └─────┘ └─────┘  │    │
│        │             └─────────────────────┘    │
│        │                      │                │
│        ▼                      ▼                │
│  ┌─────────────┐     ┌─────────────────────┐    │
│  │  SQL Server │     │      存储引擎        │    │
│  │  Browser    │     │  ┌────┐ ┌────────┐ │    │
│  └─────────────┘     │  │访问│→│缓冲池  │ │    │
│                      │  │方法│ │        │ │    │
│                      │  └────┘ └────────┘ │    │
│                      └─────────────────────┘    │
│                              │                │
│                              ▼                │
│                      ┌─────────────────┐      │
│                      │   磁盘文件       │      │
│                      │  .mdf .ldf .ndf  │      │
│                      └─────────────────┘      │
└─────────────────────────────────────────────────┘
```

## 协议层

SQL Server 通过 **TDS（Tabular Data Stream）** 协议与客户端通信。默认监听端口：

- **TCP/IP**：1433（可修改）
- **Named Pipes**：用于 Windows 本地通信

**SQL Server Browser** 服务负责监听 1434 端口的 UDP 请求，当客户端连接时指定服务器名而非端口号时，Browser 会返回实例对应的端口号。

```bash
# 测试连接
sqlcmd -S localhost -U sa -P YourPassword

# 指定端口
sqlcmd -S localhost,1433 -U sa -P YourPassword
```

## 关系引擎

关系引擎负责 SQL 的解析、优化和执行，包含三个核心组件：

### 解析器（Parser）

解析器将 T-SQL 文本转换为**解析树（Parse Tree）**，验证语法和语义，检查对象是否存在，生成逻辑操作序列。

```sql
-- 解析过程将生成类似这样的逻辑树
SELECT  → Filter → Project → Join → TableScan(Orders)
WHERE    → Orders.OrderDate > '2024-01-01'
```

### 查询优化器（Optimizer）

优化器接收解析树，生成**候选执行计划**，评估成本后选择最优计划。SQL Server 使用**基于成本的优化器（CBO）**，成本估算基于：

- 统计信息（行数、分布）
- 索引可用性
- 数据分布
- 系统资源（内存、CPU）

:::warning
优化器不会优化 `sp_executesql` 中的动态 SQL 片段。如果使用动态 SQL，参数化查询能更好地利用执行计划缓存。
:::

### 执行器（Executor）

执行器按照执行计划一步步执行，从存储引擎请求数据，组装结果集返回客户端。

执行器与存储引擎通过**迭代器（Iterator）** 接口交互，每个操作符（Scan、Join、Sort 等）都实现相同的接口：打开、获取下一行、关闭。

## 存储引擎

存储引擎负责数据的物理存储、读取和写入。

### 访问方法（Access Methods）

访问方法决定如何读取数据：
- **直接定位**：索引 Seek、聚集索引访问
- **扫描**：全表扫描、索引扫描

### 缓冲池（Buffer Pool）

缓冲池是 SQL Server 最大的内存消耗者，管理数据页和执行计划的缓存。

**数据页生命周期**：

```
应用程序请求数据
        ↓
┌───────────────────────────┐
│     Buffer Pool           │
│  ┌───────────────────┐    │
│  │  请求的页是否存在？  │    │
│  └───────────────────┘    │
│         ↓ 是               ↓ 否
│    返回缓存数据         从磁盘读取
│                            ↓
│                    ┌─────────────────┐
│                    │ 8KB 页加载到内存  │
│                    │ 标记为已使用      │
│                    └─────────────────┘
└───────────────────────────┘
```

### 日志写入（Log Writer）

所有修改操作先记录到事务日志，保证可恢复性。

```sql
-- 查看日志使用情况
SELECT 
    DB_NAME(database_id) AS DatabaseName,
    total_log_size_in_bytes / 1024.0 / 1024 AS TotalLogMB,
    used_log_space_in_bytes / 1024.0 / 1024 AS UsedLogMB,
    used_log_space_in_percent AS UsedPercent
FROM sys.dm_db_log_space_usage;
```

## 数据文件

SQL Server 数据库文件有三类：

| 文件类型 | 扩展名 | 作用 |
|---|---|---|
| 主数据文件 | .mdf | 存储数据库对象，每个数据库有且仅有一个 |
| 次要数据文件 | .ndf | 扩展存储，可分散到不同磁盘 |
| 事务日志文件 | .ldf | 记录所有事务操作 |

**文件组**逻辑组织数据文件，控制表和索引的物理存储位置。默认文件组 `PRIMARY` 包含主数据文件，自定义文件组可用来实现分区表、索引分离。

## 线程与任务模型

SQL Server 2014 引入**任务优化调度（UMS）**，SQL Server 2016+ 进一步改进为**SOS Scheduler**。

- **Scheduler**：每个 CPU 核心对应一个 Scheduler，负责调度 Worker
- **Worker**：执行请求的线程或任务
- **Task**：分配给 Worker 的工作单元

```sql
-- 查看调度器和 worker 状态
SELECT 
    scheduler_id,
    cpu_id,
    is_online,
    current_tasks_count,
    runnable_tasks_count,
    current_workers_count
FROM sys.dm_os_schedulers
WHERE status = 'VISIBLE ONLINE';
```

## 内存架构

SQL Server 内存分为多个组件池：

```
┌────────────────────────────────────────┐
│           SQL Server 内存               │
│                                        │
│  ┌──────────────────────────────────┐  │
│  │        Buffer Pool (约 80%)       │  │
│  │  ┌────────┐  ┌────────────────┐ │  │
│  │  │ 数据页 │  │  执行计划缓存   │ │  │
│  │  │ 缓存   │  │  (Plan Cache)  │ │  │
│  │  └────────┘  └────────────────┘ │  │
│  └──────────────────────────────────┘  │
│                                        │
│  ┌──────────────────────────────────┐  │
│  │    非缓冲池内存（约 20%）          │  │
│  │  ┌────────┐  ┌────────────────┐ │  │
│  │  │ 连接   │  │  线程/锁/扩展   │ │  │
│  │  │ 信息   │  │  存储过程(XP)  │ │  │
│  │  └────────┘  └────────────────┘ │  │
│  └──────────────────────────────────┘  │
└────────────────────────────────────────┘
```

通过 `sys.dm_os_memory_clerks` 可以查看各组件的内存使用情况：

```sql
SELECT 
    type,
    name,
    pages_kb / 1024.0 AS MemoryMB,
    virtual_memory_reserved_kb / 1024.0 AS VirtualReservedMB
FROM sys.dm_os_memory_clerks
ORDER BY pages_kb DESC;
```

## 完整请求流程

一条 SELECT 语句在 SQL Server 中的完整生命周期：

```
1. 客户端发送 TDS 包
        ↓
2. 协议层接收请求包
        ↓
3. 关系引擎解析 T-SQL
        ↓
4. 优化器生成执行计划
        ↓
5. 执行器调用存储引擎
        ↓
6. 缓冲池检查页是否在内存
        ↓
7. 页不在内存则从磁盘读取
        ↓
8. 执行器处理数据
        ↓
9. 结果序列化为 TDS 包
        ↓
10. 协议层发送到客户端
```

理解这个流程，有助于在性能调优时准确定位瓶颈在哪个环节。
