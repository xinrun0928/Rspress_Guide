# 统计信息

「为什么执行计划显示要扫描 10000 行，实际只有 100 行？」

问题的答案是**统计信息**。统计信息是 SQL Server 优化器估计查询成本的依据，统计信息不准确，执行计划就会出错。

## 统计信息是什么

统计信息是描述表中数据分布的元数据，包括：

- **直方图（Histogram）**：列值的分布
- **密度向量（Density Vector）**：列组合的重复度
- **字符串统计（String Statistics）**：字符串列的前缀统计

```
直方图示例：
┌────────┬────────┬────────┬────────┬────────┬────────┬────────┐
│ RANGE_HI_KEY │ RANGE_ROWS │ EQ_ROWS │ DISTINCT_RANGE_ROWS │
├────────┼────────┼────────┼────────┼────────┼────────┼────────┤
│  100   │   -    │  1000  │    -   │    -   │    -   │    -   │
│  200   │  5000  │  2000  │   10   │    -   │    -   │    -   │
│  300   │  8000  │  3000  │   15   │    -   │    -   │    -   │
│  400   │  6000  │  4000  │   12   │    -   │    -   │    -   │
└────────┴────────┴────────┴────────┴────────┴────────┴────────┘

解释：
- 200 行等于 200
- 100-200 之间有 5000 行
- 小于 100 有 1000 行
```

## 统计信息的作用

### 优化器成本估算

```sql
-- 查询 CustomerID = 100
SELECT * FROM Orders WHERE CustomerID = 100;

-- 优化器根据统计信息估算：
-- 1. CustomerID 列有多少不同值（基数）
-- 2. 满足条件的行数
-- 3. 使用索引 vs 全表扫描的成本
-- 4. 选择最优执行计划
```

### 索引选择

```sql
-- 统计信息帮助选择：
-- - 使用哪个索引
-- - 是否值得回表（Key Lookup）
-- - 是否需要并行执行
```

## 查看统计信息

### 基本信息

```sql
-- 查看表的统计信息
EXEC sp_helpstats 'Orders', 'ALL';

-- 输出示例：
-- 索引名                     最后更新时间
-- PK_Orders                  2024-03-01 10:00:00
-- IX_Orders_CustomerID       2024-03-01 10:00:00
-- _WA_Sys_CustomerID_...     2024-03-01 10:00:00
```

### 详细统计

```sql
DBCC SHOW_STATISTICS ('Orders', 'IX_Orders_CustomerID');
```

输出包含三个部分：

**1. 头部信息**

```
Name               Updated                Rows        Rows Sampled  Steps  Density               Average key length
IX_Orders_CustomerID  Mar 1 2024 10:00AM  1000000     1000000       200    0.001                 4

All density     Average Length  Columns
0.001           4               CustomerID
0.000001        8               CustomerID, OrderID
```

**2. 直方图**

```
RANGE_HI_KEY    RANGE_ROWS   EQ_ROWS    RANGE_DISTINCT_ROWS
100             0            5000       0
200             5000         6000       100
300             8000         4500       150
...
```

**3. 密度向量**

```

CustomerID                              0.001
(CustomerID, OrderID)                   0.000001

String Index                             YES
String Index Density                     0.5
String Index Length                      3
```

## 统计信息创建

### 自动创建

```sql
-- 开启自动创建统计信息（默认开启）
sp_configure 'auto create statistics', 1;
RECONFIGURE;

-- 当查询引用列时，优化器自动创建统计信息
SELECT * FROM Orders WHERE CustomerID = 100;
-- 如果没有 IX_Orders_CustomerID 的统计信息，会自动创建
```

### 手动创建

```sql
-- 创建单列统计信息
CREATE STATISTICS STAT_Orders_CustomerID ON Orders(CustomerID);

-- 创建复合列统计信息
CREATE STATISTICS STAT_Orders_CustDate ON Orders(CustomerID, OrderDate);

-- 创建过滤统计信息（SQL Server 2016+）
CREATE STATISTICS STAT_Orders_Active ON Orders(CustomerID)
WHERE Status = 'Active';
```

## 统计信息更新

### 自动更新

```sql
-- 开启自动更新统计信息（默认开启）
sp_configure 'auto update statistics', 1;
RECONFIGURE;

-- 触发条件：
-- - 表有 > 500 行：20%+500 行变化
-- - 表有 <= 500 行：500 行变化
-- - tempdb 中的表：6+500 行变化
```

### 手动更新

```sql
-- 更新表的统计信息
UPDATE STATISTICS Orders;

-- 更新特定索引的统计信息
UPDATE STATISTICS Orders IX_Orders_CustomerID;

-- 采样更新（快速）
UPDATE STATISTICS Orders WITH SAMPLE 10 PERCENT;

-- 全表扫描更新（精确）
UPDATE STATISTICS Orders WITH FULLSCAN;

-- 指定行数采样
UPDATE STATISTICS Orders WITH ROWCOUNT = 1000000;
```

## 统计信息与执行计划

### 估计行数不准确

```sql
-- 问题：历史订单数据删除后，统计信息未更新
-- 实际只有 100 行，估计却是 10000 行

-- 诊断：查看估计行数 vs 实际行数
SELECT 
    qp.query_plan,
    qs.* 
FROM sys.dm_exec_query_stats qs
CROSS APPLY sys.dm_exec_query_plan(qs.plan_handle) qp
WHERE qs.statement_start_offset = 0;

-- 查看执行计划中的警告
-- 如果 "Estimated Rows" 与 "Actual Rows" 差距大
-- 需要更新统计信息
```

### 强制使用统计信息

```sql
-- 查看统计信息列值范围
SELECT 
    CustomerID,
    COUNT(*) AS cnt
FROM Orders
GROUP BY CustomerID
ORDER BY cnt DESC;

-- 查看直方图中的对应值
DBCC SHOW_STATISTICS ('Orders', 'IX_Orders_CustomerID')
WITH HISTOGRAM;
```

## 异步统计信息更新

SQL Server 2014+ 支持异步统计信息更新，避免大表更新统计信息时的阻塞。

```sql
-- 开启异步统计信息
ALTER DATABASE [YourDB] SET AUTO_UPDATE_STATISTICS_ASYNC ON;

-- 影响：
-- - 查询立即使用旧统计信息执行
-- - 后台线程更新统计信息
-- - 下次查询使用新统计信息
```

## 统计信息维护

### 维护计划

```sql
-- 检查需要更新的统计信息
SELECT 
    OBJECT_NAME(s.object_id) AS TableName,
    s.name AS StatName,
    STATS_DATE(s.object_id, s.stats_id) AS LastUpdated,
    s.auto_created,
    SUM(sp.rows) AS TotalRows,
    CAST(100.0 * (SUM(sp.rows) - s.rows_sampled) / NULLIF(SUM(sp.rows), 0) AS DECIMAL(5,2)) AS StalePercent
FROM sys.stats s
CROSS APPLY sys.dm_db_stats_properties_internal(s.object_id, s.stats_id) sp
JOIN sys.objects o ON s.object_id = o.object_id
WHERE o.type = 'U'
    AND OBJECTPROPERTY(s.object_id, 'IsUserTable') = 1
GROUP BY s.object_id, s.name, STATS_DATE(s.object_id, s.stats_id), s.auto_created
HAVING STATS_DATE(s.object_id, s.stats_id) < DATEADD(DAY, -7, GETDATE())
    OR 100.0 * (SUM(sp.rows) - s.rows_sampled) / NULLIF(SUM(sp.rows), 0) > 10
ORDER BY LastUpdated;
```

### 增量统计信息（SQL Server 2014+）

大表分区的统计信息可以增量更新。

```sql
-- 创建增量统计信息
CREATE STATISTICS STAT_Orders_CustomerID ON Orders(CustomerID)
WITH INCREMENTAL = ON;

-- 分区表自动使用增量统计信息
-- 当某个分区数据变化时，只更新该分区的统计信息
```

## 统计信息相关问题

### 问题 1：统计信息过时

```sql
-- 症状：查询计划不优，执行慢
-- 原因：大量数据变更后，统计信息未更新

-- 解决：手动更新
UPDATE STATISTICS Orders;
-- 或启用异步更新
ALTER DATABASE [YourDB] SET AUTO_UPDATE_STATISTICS_ASYNC ON;
```

### 问题 2：自动创建被禁用

```sql
-- 症状：新查询列没有统计信息，执行计划差
-- 原因：auto create statistics 被禁用

-- 解决：
sp_configure 'auto create statistics', 1;
RECONFIGURE;
```

### 问题 3：复合索引统计信息

```sql
-- 复合索引 (A, B, C) 自动创建统计信息：
-- 1. 列 A 的统计信息
-- 2. 列 (A, B) 的统计信息
-- 3. 列 (A, B, C) 的统计信息（索引键）

-- 如果需要列 (B, C) 的统计信息，需要手动创建
CREATE STATISTICS STAT_Orders_B_C ON Orders(B, C);
```

## 监控统计信息

```sql
-- 监控统计信息更新
SELECT 
    sch.name + '.' + t.name AS TableName,
    s.name AS StatName,
    STATS_DATE(s.object_id, s.stats_id) AS LastUpdated,
    sp.modification_counter AS Modifications,
    sp.rows AS TotalRows,
    CASE 
        WHEN sp.modification_counter > sp.rows * 0.1 THEN '需要更新'
        ELSE '正常'
    END AS Status
FROM sys.stats s
JOIN sys.tables t ON s.object_id = t.object_id
JOIN sys.schemas sch ON t.schema_id = sch.schema_id
CROSS APPLY sys.dm_db_stats_properties_internal(s.object_id, s.stats_id) sp
WHERE OBJECTPROPERTY(t.object_id, 'IsUserTable') = 1
ORDER BY sp.modification_counter DESC;
```

统计信息是优化器决策的基础。保持统计信息准确，执行计划才能最优。
