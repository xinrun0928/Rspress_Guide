# 慢查询优化

「同样的查询，昨天快，今天慢，为什么？」

慢查询是数据库性能问题的核心。理解如何诊断和优化慢查询，是每个 DBA 和开发者的必修课。

## 慢查询的常见原因

| 原因 | 症状 | 诊断方法 |
|---|---|---|
| 缺少索引 | 大量 Table Scan | 查看执行计划 |
| 统计信息过时 | 估计行数与实际差距大 | DBCC SHOW_STATISTICS |
| 锁等待 | 高并发下变慢 | 查看等待类型 |
| 参数嗅探 | 特定参数变慢 | 对比不同参数的计划 |
| 碎片 | 间歇性变慢 | sys.dm_db_index_physical_stats |
| 资源瓶颈 | CPU/内存/磁盘高负载 | 监视器 |

## 诊断工具

### 1. Query Store（推荐）

SQL Server 2016+ 内置的查询存储，持续收集执行计划和性能指标。

```sql
-- 开启 Query Store
ALTER DATABASE [YourDB] SET QUERY_STORE = ON
WITH (OPERATION_MODE = READ_WRITE,
      QUERY_CAPTURE_MODE = AUTO,
      MAX_STORAGE_SIZE_MB = 1024);

-- 查看最耗时查询
SELECT 
    q.query_id,
    qt.query_text_id,
    rsavg_duration,
    rsavg_cpu_time,
    rsavg_logical_io_reads,
    rsavg_physical_io_reads,
    rsi.execution_type_desc,
    q.query_hash
FROM sys.query_store_runtime_stats rs
JOIN (
    SELECT query_id, query_text_id, query_hash
    FROM sys.query_store_query
) q ON rs.query_id = q.query_id
JOIN sys.query_store_query_text qt ON q.query_text_id = qt.query_text_id
CROSS APPLY (
    SELECT TOP 1 rsavg_duration = rs.avg_duration,
           rsavg_cpu_time = rs.avg_cpu_time,
           rsavg_logical_io_reads = rs.avg_logical_io_reads,
           rsavg_physical_io_reads = rs.avg_physical_io_reads,
           rsi.execution_type_desc = rsi.execution_type_desc
    FROM sys.query_store_runtime_stats rs
    JOIN sys.query_store_runtime_stats_interval rsi ON rs.runtime_stats_interval_id = rsi.runtime_stats_interval_id
    WHERE rs.query_id = q.query_id
) rs
ORDER BY rsavg_duration DESC;
```

### 2. 扩展事件（Extended Events）

轻量级事件收集系统，替代 SQL Trace。

```sql
-- 创建扩展事件会话
CREATE EVENT SESSION [SlowQueries] ON SERVER
ADD EVENT sqlserver.sql_statement_completed (
    ACTION (sqlserver.sql_text, sqlserver.query_hash)
    WHERE duration > 100000  -- 超过 100ms
)
ADD TARGET package0.event_file (
    SET filename = N'SlowQueries.xel',
        max_file_size = 100,
        max_rollover_files = 5
)
WITH (STARTUP_STATE = ON);

-- 启动会话
ALTER EVENT SESSION [SlowQueries] ON SERVER STATE = START;

-- 查看结果
SELECT 
    event_data_XML.value('(/event/data[1]/value)[1]', 'BIGINT') AS Duration_us,
    event_data_XML.value('(/event/action[1]/value)[1]', 'VARCHAR(MAX)') AS SQLText
FROM sys.fn_xe_file_target_read_file(
    'SlowQueries*.xel', NULL, NULL, NULL
);
```

### 3. DMV 查询

```sql
-- 查找当前消耗最多的查询
SELECT TOP 20 
    SUBSTRING(qt.text, qs.statement_start_offset / 2 + 1,
        (CASE qs.statement_end_offset 
            WHEN -1 THEN DATALENGTH(qt.text)
            ELSE qs.statement_end_offset 
        END - qs.statement_start_offset) / 2 + 1) AS QueryText,
    qs.execution_count,
    qs.total_elapsed_time / 1000 AS total_ms,
    qs.total_logical_reads / 128.0 AS total_mb_read,
    qs.total_physical_reads / 128.0 AS physical_mb_read,
    qs.total_worker_time / 1000 AS total_cpu_ms,
    qs.max_elapsed_time / 1000 AS max_ms,
    qs.max_logical_reads,
    qs.last_elapsed_time / 1000 AS last_ms,
    qs.last_logical_reads,
    cp.objtype,
    cp.plan_handle
FROM sys.dm_exec_query_stats qs
CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle) qt
JOIN sys.dm_exec_cached_plans cp ON qs.plan_handle = cp.plan_handle
WHERE qt.dbid = DB_ID()
ORDER BY qs.total_worker_time DESC;

-- 查找逻辑读最多的查询
SELECT TOP 20 
    SUBSTRING(qt.text, qs.statement_start_offset / 2 + 1,
        (CASE qs.statement_end_offset 
            WHEN -1 THEN DATALENGTH(qt.text)
            ELSE qs.statement_end_offset 
        END - qs.statement_start_offset) / 2 + 1) AS QueryText,
    qs.execution_count,
    qs.total_logical_reads,
    qs.total_logical_reads / qs.execution_count AS avg_logical_reads,
    qs.total_worker_time / 1000,
    qs.max_logical_reads
FROM sys.dm_exec_query_stats qs
CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle) qt
ORDER BY qs.total_logical_reads DESC;
```

## 优化流程

### 步骤 1：识别慢查询

```sql
-- Query Store：查找慢查询
SELECT TOP 10 
    q.query_id,
    qt.query_sql_text,
    rs.avg_duration / 1000.0 AS avg_duration_ms,
    rs.avg_cpu_time / 1000.0 AS avg_cpu_ms,
    rs.avg_logical_io_reads,
    rs.count_executions,
    rs.last_execution_time
FROM sys.query_store_query q
JOIN sys.query_store_query_text qt ON q.query_text_id = qt.query_text_id
JOIN (
    SELECT query_id,
           AVG(rs2.avg_duration) AS avg_duration,
           AVG(rs2.avg_cpu_time) AS avg_cpu_time,
           AVG(rs2.avg_logical_io_reads) AS avg_logical_io_reads,
           SUM(rs2.count_executions) AS count_executions,
           MAX(rs2.last_execution_time) AS last_execution_time
    FROM sys.query_store_query q2
    JOIN sys.query_store_runtime_stats rs2 ON q2.query_id = rs2.query_id
    GROUP BY q2.query_id
) rs ON q.query_id = rs.query_id
ORDER BY rs.avg_duration DESC;
```

### 步骤 2：分析执行计划

```sql
-- 查看执行计划
EXEC sp_helptext 'YourStoredProcedure';

-- 查看估计计划
SET SHOWPLAN_XML ON;
GO
SELECT * FROM YourTable WHERE ...
GO
SET SHOWPLAN_XML OFF;

-- 分析计划中的问题
-- 1. Table Scan / Clustered Index Scan → 需要索引
-- 2. Key Lookup → 需要覆盖索引
-- 3. Hash Match → 检查连接顺序
-- 4. Sort → 检查是否必要，或添加索引消除
```

### 步骤 3：检查索引

```sql
-- 查看缺失索引
SELECT 
    d.eq_highest_object_id,
    d.equality_columns,
    d.inequality_columns,
    d.included_columns,
    d.statement AS Object,
    d.avg_total_user_cost * d.user_seeks * d.avg_impact AS Score,
    d.user_seeks,
    d.user_scans
FROM sys.dm_db_missing_index_groups g
JOIN sys.dm_db_missing_index_group_stats d ON g.index_group_handle = d.group_handle
JOIN sys.dm_db_missing_index_details d2 ON g.index_handle = d2.index_handle
WHERE d.database_id = DB_ID()
ORDER BY Score DESC;

-- 查看索引使用情况
SELECT 
    OBJECT_NAME(s.object_id) AS TableName,
    i.name AS IndexName,
    s.user_seeks,
    s.user_scans,
    s.user_lookups,
    s.user_updates,
    s.last_user_seek,
    s.last_user_scan
FROM sys.dm_db_index_usage_stats s
JOIN sys.indexes i ON s.object_id = i.object_id AND s.index_id = i.index_id
WHERE OBJECTPROPERTY(s.object_id, 'IsUserTable') = 1
ORDER BY s.user_seeks + s.user_scans ASC;
```

### 步骤 4：检查统计信息

```sql
-- 查看统计信息更新时间
SELECT 
    OBJECT_NAME(s.object_id) AS TableName,
    s.name AS StatName,
    STATS_DATE(s.object_id, s.stats_id) AS LastUpdated,
    s.auto_created,
    s.user_created,
    s.no_recompute,
    s.filter_definition
FROM sys.stats s
WHERE OBJECTPROPERTY(s.object_id, 'IsUserTable') = 1
ORDER BY STATS_DATE(s.object_id, s.stats_id);

-- 更新统计信息
UPDATE STATISTICS Orders;
-- 或
UPDATE STATISTICS Orders IX_Orders_CustomerID;

-- 全表更新
UPDATE STATISTICS Orders WITH FULLSCAN;
-- 或采样
UPDATE STATISTICS Orders WITH SAMPLE 50 PERCENT;
```

### 步骤 5：检查等待

```sql
-- 查看当前等待类型
SELECT 
    session_id AS SPID,
    status,
    wait_type,
    wait_time / 1000.0 AS wait_sec,
    blocking_session_id AS BlkBy,
    SUBSTRING(qt.text, er.statement_start_offset / 2 + 1,
        (CASE er.statement_end_offset 
            WHEN -1 THEN DATALENGTH(qt.text)
            ELSE er.statement_end_offset 
        END - er.statement_start_offset) / 2 + 1) AS CurrentCommand
FROM sys.dm_exec_requests er
CROSS APPLY sys.dm_exec_sql_text(er.sql_handle) qt
WHERE session_id > 50
ORDER BY er.wait_time DESC;

-- 常见的等待类型
-- CXPACKET - 并行等待，通常正常
-- PAGEIOLATCH_SH - 磁盘 I/O 等待
-- LCK_M_IX - 排他锁等待
-- ASYNC_NETWORK_IO - 网络等待，客户端处理慢
-- WRITELOG - 日志写入等待
-- SOS_SCHEDULER_YIELD - CPU 争用
```

## 常见优化场景

### 场景 1：缺失索引

```sql
-- 查询
SELECT CustomerID, OrderDate, TotalAmount
FROM Orders
WHERE CustomerID = 100 AND OrderDate > '2024-01-01';

-- 添加索引
CREATE NONCLUSTERED INDEX IX_Orders_CustomerDate
ON Orders(CustomerID, OrderDate)
INCLUDE (TotalAmount);
```

### 场景 2：统计信息过时

```sql
-- 问题查询
SELECT * FROM Orders
WHERE OrderDate BETWEEN '2024-01-01' AND '2024-12-31';

-- 解决方案：更新统计信息
UPDATE STATISTICS Orders;
-- 或重建索引
ALTER INDEX ALL ON Orders REBUILD;
```

### 场景 3：参数嗅探

```sql
-- 问题：使用 OPTION (RECOMPILE) 或 OPTION (OPTIMIZE FOR)
CREATE PROCEDURE usp_GetOrders
    @CustomerID INT
AS
BEGIN
    SELECT * FROM Orders
    WHERE CustomerID = @CustomerID
    OPTION (RECOMPILE);  -- 每次重新编译
END;

-- 或固定参数值
CREATE PROCEDURE usp_GetOrders
    @CustomerID INT
AS
BEGIN
    SELECT * FROM Orders
    WHERE CustomerID = @CustomerID
    OPTION (OPTIMIZE FOR (@CustomerID = 1));  -- 固定参数
END;

-- 最佳方案：使用查询存储强制计划
```

### 场景 4：碎片导致全表扫描

```sql
-- 检查碎片
SELECT 
    OBJECT_NAME(ips.object_id) AS TableName,
    i.name AS IndexName,
    ips.avg_fragmentation_in_percent AS Fragmentation,
    ips.page_count
FROM sys.dm_db_index_physical_stats(DB_ID(), NULL, NULL, NULL, 'DETAILED') ips
JOIN sys.indexes i ON ips.object_id = i.object_id AND ips.index_id = i.index_id
WHERE ips.avg_fragmentation_in_percent > 30
    AND ips.page_count > 1000;

-- 重建索引
ALTER INDEX IX_Orders_Date ON Orders REBUILD;
```

### 场景 5：锁等待

```sql
-- 查看锁等待
SELECT 
    blocked.session_id AS BlockedSPID,
    blocker.session_id AS BlockerSPID,
    blocker.status,
    blocker.login_name,
    blocker.host_name,
    blocked.wait_time / 1000.0 AS WaitSec,
    blocked.wait_type,
    blocked.resource_type,
    blocked.request_mode,
    qt_blocker.text AS BlockerQuery,
    qt_blocked.text AS BlockedQuery
FROM sys.dm_exec_requests blocked
JOIN sys.dm_exec_requests blocker ON blocked.blocking_session_id = blocker.session_id
CROSS APPLY sys.dm_exec_sql_text(blocker.sql_handle) qt_blocker
CROSS APPLY sys.dm_exec_sql_text(blocked.sql_handle) qt_blocked
WHERE blocked.session_id > 50;
```

## 优化模板

```sql
-- 完整优化脚本
-- 1. 开启执行时间统计
SET STATISTICS TIME ON;
SET STATISTICS IO ON;

-- 2. 执行查询
SELECT ... FROM ...

-- 3. 查看执行计划
-- 4. 记录 LogicalReads 和 CPU 时间

-- 5. 根据计划添加索引
CREATE NONCLUSTERED INDEX IX_Temp ON Table(Column) INCLUDE(...);

-- 6. 再次执行，对比
SET STATISTICS TIME ON;
SET STATISTICS IO ON;
SELECT ... FROM ...
```

## 预防措施

| 措施 | 说明 |
|---|---|
| 定期更新统计信息 | `sp_updatestats` 或维护计划 |
| 定期重建索引 | `ALTER INDEX REBUILD` |
| 监控慢查询 | Query Store + 警报 |
| 代码审查 | 避免 N+1 查询 |
| 限制查询超时 | `QUERY_GOVERNOR_COST_LIMIT` |

慢查询优化是一个迭代过程。识别问题、分析原因、实施优化、验证效果，循环往复。
