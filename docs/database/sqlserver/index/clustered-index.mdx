# 聚集索引设计原则

「每个表都应该有聚集索引。」

这是 SQL Server 的最佳实践，但现实中很多表没有聚集索引，或者聚集索引设计不合理。

聚集索引决定了数据的物理存储顺序，一旦确定，影响整个表的读写性能。

## 聚集索引的本质

聚集索引不只是「加速查询的索引」，它是**数据存储方式的定义**。

有聚集索引的表，数据按索引键顺序**物理存储**在磁盘上。
无聚集索引的表（堆），数据按插入顺序存储。

```
┌─────────────────────────────────────────────────────────────────┐
│                    有聚集索引的表                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  聚集索引: OrderID (INT, 唯一)                                  │
│                                                                 │
│  ┌─────────┬─────────┬─────────┬─────────┬─────────┐          │
│  │ ID=1    │ ID=2    │ ID=3    │ ID=100  │ ID=200  │          │
│  │ 数据行  │ 数据行  │ 数据行  │ 数据行  │ 数据行  │          │
│  └─────────┴─────────┴─────────┴─────────┴─────────┘          │
│        ↓顺序存储（按 OrderID 排序）                              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                    堆表（无聚集索引）                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  数据按插入顺序存储                                              │
│                                                                 │
│  ┌─────────┬─────────┬─────────┬─────────┬─────────┐          │
│  │ RID=1   │ RID=2   │ RID=3   │ RID=4   │ RID=5   │          │
│  │ 数据行  │ 数据行  │ 数据行  │ 数据行  │ 数据行  │          │
│  └─────────┴─────────┴─────────┴─────────┴─────────┘          │
│        ↓无序存储                                                │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 聚集索引 vs 非聚集索引

| 特性 | 聚集索引 | 非聚集索引 |
|---|---|---|
| 数量限制 | 每个表最多 1 个 | 每个表最多 999 个 |
| 叶节点内容 | 完整数据行 | 索引键 + 行定位器 |
| 表创建时 | 可指定自动创建 | 必须手动创建 |
| 数据排序 | 物理排序 | 逻辑排序（不影响物理存储） |
| 唯一性 | 可非唯一 | 可重复 |

```sql
-- 聚集索引（唯一）
CREATE TABLE Orders (
    OrderID INT PRIMARY KEY,  -- 自动创建唯一聚集索引
    ...
);

-- 聚集索引（非唯一）
CREATE TABLE Orders (
    OrderID INT,
    CustomerID INT,
    ...
    CONSTRAINT PK_Orders PRIMARY KEY NONCLUSTERED (OrderID)  -- 非聚集
);

CREATE CLUSTERED INDEX IX_Orders_CustomerID ON Orders(CustomerID);
-- 允许重复值的聚集索引
```

## 选择聚集索引键的原则

### 原则一：唯一或近似唯一

聚集索引键应该唯一，否则 SQL Server 会在键后添加 4 字节的 **uniquifier** 使其唯一。

```sql
-- 不推荐：CustomerID 可能重复
CREATE CLUSTERED INDEX IX_Orders_CustomerID ON Orders(CustomerID);
-- 每个 CustomerID 重复的行，都会有额外的 uniquifier 列

-- 推荐：OrderID 唯一
CREATE CLUSTERED INDEX PK_Orders ON Orders(OrderID);
```

### 原则二：单调变化

自增 ID 或时间戳作为聚集键，新行追加到表末尾，**避免页分裂**。

```
自增 ID（好的聚集键）：
┌────────┬────────┬────────┬────────┬────────┬────────┐
│  1     │  2     │  3     │  4     │  5     │  6     │  ...
└────────┴────────┴────────┴────────┴────────┴────────┘
     ↓新行总是追加到末尾

UUID（坏的聚集键）：
┌────────┬────────┬────────┬────────┬────────┬────────┐
│  随机  │  随机  │  随机  │  随机  │  随机  │  随机  │  ...
└────────┴────────┴────────┴────────┴────────┴────────┘
     ↓新行可能插入到任意位置，导致页分裂
```

### 原则三：宽度小

聚集索引键会被所有非聚集索引引用。键越大，非聚集索引越大。

```sql
-- 不推荐
CREATE CLUSTERED INDEX IX_Orders_CustomerID ON Orders(CustomerID VARCHAR(100));
-- VARCHAR(100) 作为聚集键，所有非聚集索引都会变大

-- 推荐
CREATE CLUSTERED INDEX PK_Orders ON Orders(OrderID INT);  -- 4 字节
```

### 原则四：不可更新

频繁更新聚集键会导致：
1. 行在页内移动
2. 更新所有引用此行的非聚集索引

```sql
-- 不推荐
CREATE CLUSTERED INDEX IX_Sessions ON Sessions(SessionToken);  -- SessionToken 可能被更新

-- 推荐
CREATE CLUSTERED INDEX IX_Sessions ON Sessions(SessionID);  -- SessionID 不更新
```

## 常见场景的聚集索引设计

### 场景一：订单表

```sql
-- 订单表：主键 OrderID 是最好的聚集键
CREATE TABLE Orders (
    OrderID INT IDENTITY(1,1) PRIMARY KEY NONCLUSTERED,  -- 非聚集主键
    OrderNumber NVARCHAR(50) UNIQUE,                    -- 业务编号
    CustomerID INT NOT NULL,
    OrderDate DATETIME2 NOT NULL,
    TotalAmount DECIMAL(10,2),
    Status TINYINT,
    CreatedDate DATETIME2 DEFAULT SYSDATETIME()
);

-- 创建聚集索引（用主键作为聚集键）
CREATE CLUSTERED INDEX PK_Orders ON Orders(OrderID);

-- 常见查询优化
CREATE NONCLUSTERED INDEX IX_Orders_Customer 
ON Orders(CustomerID, OrderDate DESC) 
INCLUDE (TotalAmount, Status);
```

### 场景二：日志表

```sql
-- 日志表：按时间顺序插入，CreatedDate 是好的聚集键
CREATE TABLE AppLogs (
    LogID BIGINT IDENTITY(1,1) PRIMARY KEY NONCLUSTERED,  -- 非聚集
    LogDate DATETIME2 NOT NULL DEFAULT SYSDATETIME(),   -- 时间戳作为聚集键
    Level TINYINT,
    Source NVARCHAR(100),
    Message NVARCHAR(MAX),
    ExtraData NVARCHAR(MAX)
);

CREATE CLUSTERED INDEX IX_Logs_Date ON AppLogs(LogDate);

-- 查询最新日志：按时间倒序
CREATE NONCLUSTERED INDEX IX_Logs_LevelDate 
ON AppLogs(Level, LogDate DESC) 
INCLUDE (Message);
```

### 场景三：用户表

```sql
-- 用户表：UserID 作为聚集键
CREATE TABLE Users (
    UserID INT IDENTITY(1,1) PRIMARY KEY,
    UserName NVARCHAR(50) UNIQUE,
    Email NVARCHAR(255),
    CreatedDate DATETIME2,
    LastLoginDate DATETIME2,
    Status TINYINT
);

CREATE CLUSTERED INDEX PK_Users ON Users(UserID);

-- 不要用 Email 作为聚集键
-- CREATE CLUSTERED INDEX IX_Users_Email ON Users(Email);  -- VARCHAR(255) 太宽
```

### 场景四：多租户系统

```sql
-- 多租户系统：TenantID + 业务 ID 作为复合聚集键
CREATE TABLE Orders (
    TenantID INT NOT NULL,
    OrderID INT IDENTITY(1,1),
    ...
    PRIMARY KEY NONCLUSTERED (TenantID, OrderID)  -- 非聚集主键
);

CREATE CLUSTERED INDEX IX_Orders_TenantID 
ON Orders(TenantID, OrderID)  -- 复合聚集键，先按租户隔离
WITH (ONLINE = ON);
```

## 聚集索引与范围查询

聚集索引对范围查询非常高效：

```sql
-- 假设 OrderDate 是聚集键
SELECT * FROM Orders 
WHERE OrderDate BETWEEN '2024-01-01' AND '2024-01-31';

-- 执行计划分析：
-- 1. 定位到 2024-01-01 所在的叶节点
-- 2. 按双向链表顺序遍历到 2024-01-31
-- 3. 所有目标行连续读取，I/O 效率高
```

但如果聚集键不是单调的，范围查询会产生随机 I/O：

```sql
-- 如果 CustomerID 是聚集键（不是单调的）
SELECT * FROM Orders 
WHERE CustomerID BETWEEN 100 AND 200;

-- CustomerID 不单调，页不是按 CustomerID 排序
-- 即使查询范围很小，也需要访问很多随机页
```

## 聚集索引与分页

```sql
-- 按聚集键分页：高效
SELECT * FROM Orders
WHERE OrderID > 1000
ORDER BY OrderID
OFFSET 0 ROWS FETCH NEXT 20 ROWS ONLY;

-- 按非聚集键分页：低效
SELECT * FROM Orders
ORDER BY OrderDate
OFFSET 10000 ROWS FETCH NEXT 20 ROWS ONLY;
-- 需要扫描 10020 行
```

## 无聚集索引（堆）

某些场景下，堆可能比聚集索引更高效：

| 场景 | 原因 |
|---|---|
| 临时表/暂存表 | 只追加，不查询 |
| ETL 过程 | 批量加载后创建索引 |
| 小表（< 1000 行） | 全表扫描反而更快 |

```sql
-- 创建堆表
CREATE TABLE StagingData (
    ID INT NOT NULL,
    Value NVARCHAR(100)
);

-- 堆表仍然可以有非聚集索引
CREATE NONCLUSTERED INDEX IX_StagingData_ID ON StagingData(ID);
```

## 重建聚集索引

```sql
-- 重建聚集索引（解决碎片）
ALTER INDEX PK_Orders ON Orders REBUILD;

-- 重建并指定选项
ALTER INDEX PK_Orders ON Orders 
REBUILD WITH (
    FILLFACTOR = 90,
    PAD_INDEX = ON,
    ONLINE = ON  -- Enterprise Edition
);

-- 移动聚集索引到新文件组
ALTER INDEX PK_Orders ON Orders
REBUILD PARTITION = ALL
WITH (SORT_IN_TEMPDB = ON);
```

## 常见错误

| 错误 | 影响 |
|---|---|
| 无聚集索引 | 全表扫描慢，非聚集索引使用 RID |
| UUID 作为聚集键 | 严重页分裂和碎片 |
| 长列作为聚集键 | 所有非聚集索引膨胀 |
| 更新频繁的列作为聚集键 | 非聚集索引频繁更新 |
| 非唯一聚集键 | 每个重复值增加 4 字节开销 |

## 检查聚集索引

```sql
-- 查看表是否有聚集索引
SELECT 
    OBJECT_NAME(i.object_id) AS TableName,
    i.name AS IndexName,
    i.type_desc AS IndexType,
    i.is_primary_key,
    ic.key_ordinal AS KeyOrdinal,
    COL_NAME(ic.object_id, ic.column_id) AS ColumnName
FROM sys.indexes i
JOIN sys.index_columns ic ON i.object_id = ic.object_id AND i.index_id = ic.index_id
WHERE OBJECTPROPERTY(i.object_id, 'IsUserTable') = 1
    AND i.type = 1  -- 聚集索引
ORDER BY OBJECT_NAME(i.object_id);
```

聚集索引设计是数据库性能的基础。选对了聚集键，表就成功了一半。
