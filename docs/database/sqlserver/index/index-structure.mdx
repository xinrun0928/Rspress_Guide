# B+ 树索引结构

「为什么主键用自增 ID 比 UUID 更好？」

这个问题看似简单，背后却藏着 B+ 树索引的核心机制——**顺序插入避免页分裂，页分裂导致碎片，碎片拖累性能**。

理解 B+ 树在 SQL Server 中如何工作，是索引设计的第一步。

## B+ 树基础

SQL Server 的索引采用 **B+ 树**数据结构，这是在磁盘存储场景下被广泛采用的结构。

### B+ 树 vs B 树

| 特性 | B 树 | B+ 树 |
|---|---|---|
| 数据存储位置 | 所有层级 | 仅叶节点 |
| 叶节点连接 | 无 | 有序双向链表 |
| 范围查询 | 需要回溯 | 直接遍历链表 |
| 查询稳定性 | 不稳定 | 稳定（叶节点深度相同） |

```
                    B 树结构
                         ┌───┐
                         │50 │
                      ┌──┴───┴──┐
                     30        70
                   ┌─┴─┐     ┌─┴─┐
                  10 40     60 90
                  ↓  ↓      ↓  ↓
                数据 数据   数据 数据

                    B+ 树结构
                         ┌───┐
                         │50 │
                      ┌──┴───┴──┐
                     30        70
                   ┌─┴─┐     ┌─┴─┐
                  10 40     60 90
                  ↓  ↓      ↓  ↓
         ┌───────┴──┴───────┴──┴───────┐
         ▼  ▼  ▼  ▼  ▼  ▼  ▼  ▼  ▼    ← 叶节点双向链表
       数据 数据 数据 数据 数据 数据 数据 数据 数据
```

### SQL Server 的 B+ 树

SQL Server 的 B+ 树有以下特点：

1. **每个节点大小固定为一页（8KB）**
2. **非叶节点存储键值和子页指针**
3. **叶节点存储索引键和行定位器**
4. **叶节点通过双向链表连接**

## 索引页结构

### 非叶节点（索引页）

非叶节点（索引页）存储：
- 索引键值
- 子页指针（文件号:页号）

```
┌─────────────────────────────────────────────────────────────────┐
│                        非叶节点 (索引页)                          │
├─────────────────────────────────────────────────────────────────┤
│  页头 (96 字节)                                                  │
├─────────────────────────────────────────────────────────────────┤
│  ┌─────────────────────────────────────────────────────────────┐│
│  │ 键值 1  │指针│ 键值 2  │指针│ 键值 3  │指针│ ... │指针│    ││
│  │ (8B)   │(6B)│ (8B)   │(6B)│ (8B)   │(6B)│     │(6B)│    ││
│  └─────────────────────────────────────────────────────────────┘│
├─────────────────────────────────────────────────────────────────┤
│  行偏移数组                                                      │
└─────────────────────────────────────────────────────────────────┘
```

### 叶节点

**聚集索引叶节点**：存储完整的数据行

```
┌─────────────────────────────────────────────────────────────────┐
│                        聚集索引叶节点 (数据页)                    │
├─────────────────────────────────────────────────────────────────┤
│  页头 (96 字节)                                                  │
├─────────────────────────────────────────────────────────────────┤
│  ┌─────────────────────────────────────────────────────────────┐│
│  │ 数据行 1 │ 数据行 2 │ 数据行 3 │ ... │ 数据行 N │             ││
│  │ 键值+数据│ 键值+数据│ 键值+数据│     │ 键值+数据│             ││
│  └─────────────────────────────────────────────────────────────┘│
├─────────────────────────────────────────────────────────────────┤
│  行偏移数组                                                      │
├─────────────────────────────────────────────────────────────────┤
│  ◀──────────────────── 叶节点双向链表 ──────────────────────▶  │
└─────────────────────────────────────────────────────────────────┘
```

**非聚集索引叶节点**：存储索引键和行定位器

```
┌─────────────────────────────────────────────────────────────────┐
│                       非聚集索引叶节点                            │
├─────────────────────────────────────────────────────────────────┤
│  ┌─────────────────────────────────────────────────────────────┐│
│  │ 索引键值 │ 行定位器 │ 索引键值 │ 行定位器 │ ... │            ││
│  │ Name=AA │ RID=(1:100:0)│Name=ZZ│ 聚集键=50 │               ││
│  └─────────────────────────────────────────────────────────────┘│
├─────────────────────────────────────────────────────────────────┤
│  ◀──────────────────── 叶节点双向链表 ──────────────────────▶  │
└─────────────────────────────────────────────────────────────────┘
```

## 行定位器

### 堆表（非聚集索引）

堆是没有聚集索引的表。非聚集索引在堆表上的行定位器是 **RID（Row ID）**：
- 文件号（4 字节）
- 页号（4 字节）
- 行号（2 字节）

```
格式：0x<FileNumber>:<PageNumber>:<SlotNumber>
示例：0x0001:00000640:0005
```

### 聚集表（非聚集索引）

表有聚集索引时，非聚集索引的行定位器是**聚集索引键值**。

```
┌─────────────────────────────────────────────────────────────────┐
│ 非聚集索引 ──▶ 聚集索引键 ──▶ 实际数据行                          │
└─────────────────────────────────────────────────────────────────┘
```

这意味着如果聚集索引键过大，所有非聚集索引也会变大。

:::tip
聚集索引键应该小且稳定。避免使用：
- VARCHAR(500) 作为聚集键
- 多列组合键过长
- 频繁更新的列作为聚集键（更新聚集键需要更新所有非聚集索引）
:::

## 索引深度

索引深度（Level）决定了查找一条记录需要的 **I/O 次数**。

### 深度计算公式

每页能存储的索引条目数取决于键值大小：

| 键值大小 | 每页条目数（估算） | 每页指向的子页数 |
|---|---|---|
| 4 字节 (INT) | ~500 | ~500 |
| 8 字节 (BIGINT) | ~450 | ~450 |
| 16 字节 (UNIQUEIDENTIFIER) | ~350 | ~350 |
| 100 字节 | ~80 | ~80 |

**深度计算**：

| 深度 | 最多叶节点数 | 最多数据行数 |
|---|---|---|
| 1 | ~500 | 500 × 100 = 50,000 |
| 2 | 500² = 250,000 | 250,000 × 100 = 25,000,000 |
| 3 | 500³ = 125,000,000 | 125M × 100 = 12.5B |

```sql
-- 查看索引深度
SELECT 
    OBJECT_NAME(i.object_id) AS TableName,
    i.name AS IndexName,
    ips.index_id,
    ips.index_level AS Level,
    ips.page_count AS PageCount,
    ips.record_count AS RecordCount,
    ips.avg_page_space_used_in_percent AS PageFillPercent
FROM sys.dm_db_index_physical_stats(
    DB_ID(), 
    OBJECT_ID('Products'),
    NULL, NULL, 'DETAILED'
) ips
JOIN sys.indexes i ON ips.object_id = i.object_id AND ips.index_id = i.index_id
WHERE ips.alloc_unit_type_desc = 'IN_ROW_DATA'
ORDER BY ips.index_level DESC;
```

## 索引扫描 vs 索引查找

### 索引扫描（Index Scan）

扫描整个索引（从根到叶，遍历所有叶节点）。

**何时发生**：
- `SELECT *` 无 WHERE 条件
- `SELECT` 的列不在索引覆盖范围内
- 统计信息过时，优化器误判

```
Scan 操作
┌─────────────────────────────────────────────────────────────────┐
│  ┌───┐                                                         │
│  │根 │                                                         │
│  └───┘                                                         │
│    │                                                           │
│  ┌─┴─┐┌─┴─┐┌─┴─┐┌─┴─┐┌─┴─┐┌─┴─┐┌─┴─┐┌─┴─┐                      │
│  │   ││   ││   ││   ││   ││   ││   ││   │  ← 扫描所有非叶节点   │
│  └─┬─┘└─┬─┘└─┬─┘└─┬─┘└─┬─┘└─┬─┘└─┬─┘└─┬─┘                      │
│    ▼     ▼     ▼     ▼     ▼     ▼     ▼     ▼                │
│  ◀─┴─┴─┴─┴─┴─┴─┴─┴─┴─┴─┴─┴─┴─┴─┴─┴─┴─┴─┴─┴─┴─┴▶  ← 扫描所有叶节点 │
└─────────────────────────────────────────────────────────────────┘
```

### 索引查找（Index Seek）

精确定位，只读取需要的页。

**何时发生**：
- WHERE 条件匹配索引列（前导列）
- 统计信息准确
- 选择性高（满足条件的行少）

```
Seek 操作
┌─────────────────────────────────────────────────────────────────┐
│  ┌───┐                                                         │
│  │根 │                                                         │
│  └───┘                                                         │
│    │定位 50                                                    │
│  ┌─┴────────┐                                                  │
│  │   50    │ ← 根据键值选择路径                                │
│  └─┬────────┘                                                  │
│    ▼                                                           │
│  ┌───┐                                                         │
│  │叶 │ ──▶ 只读取包含 50 的叶节点                              │
│  └───┘                                                         │
└─────────────────────────────────────────────────────────────────┘
```

## 碎片类型

### 内部碎片（Page Internal Fragmentation）

页内空间浪费，数据行不能填满页。

**原因**：
- 频繁 UPDATE 导致行变短
- 固定长度列过多
- 填充因子设置过低

**影响**：
- 占用更多内存
- 扫描更多页
- 浪费磁盘空间

### 外部碎片（Page External Fragmentation）

逻辑顺序与物理顺序不一致。

**原因**：
- 页分裂（INSERT/UPDATE 导致）
- 随机 I/O 增加
- 顺序扫描变慢

**影响**：
- 顺序扫描变成随机 I/O
- 预读效率降低
- 查询延迟增加

```
无碎片：逻辑顺序 = 物理顺序
页 1 → 页 2 → 页 3 → 页 4 → 页 5 → 页 6 → 页 7 → 页 8

有碎片：页被分散
页 1 → [页 3] → [页 5] → 页 2 → [页 7] → 页 4 → 页 6 → 页 8
       ↑需要跳转            ↑需要跳转
```

### 碎片计算

```sql
-- 查看索引碎片
SELECT 
    OBJECT_NAME(ips.object_id) AS TableName,
    i.name AS IndexName,
    ips.index_type_desc,
    ips.avg_fragmentation_in_percent AS FragmentationPercent,
    ips.avg_page_space_used_in_percent AS PageFillPercent,
    ips.page_count AS TotalPages,
    ips.fragment_count AS FragmentCount,
    ips.avg_fragment_size_in_pages AS AvgFragmentSize
FROM sys.dm_db_index_physical_stats(
    DB_ID(), 
    OBJECT_ID('Products'),
    NULL, NULL, 'DETAILED'
) ips
JOIN sys.indexes i ON ips.object_id = i.object_id AND ips.index_id = i.index_id
WHERE ips.avg_fragmentation_in_percent > 0
ORDER BY ips.avg_fragmentation_in_percent DESC;
```

**碎片处理阈值**：

| 碎片率 | 建议操作 |
|---|---|
| `< 5%` | 无需处理 |
| `5% ~ 30%` | `ALTER INDEX REORGANIZE` |
| `> 30%` | `ALTER INDEX REBUILD` |

## 碎片处理

### REORGANIZE

重新组织叶级页的物理存储，整理碎片。

```sql
-- 重新组织单个索引
ALTER INDEX IX_Products_Price ON Products REORGANIZE;

-- 重新组织所有索引
ALTER INDEX ALL ON Products REORGANIZE;

-- 指定填充因子
ALTER INDEX IX_Products_Price ON Products 
REORGANIZE WITH (FILLFACTOR = 80);
```

### REBUILD

重建整个索引，生成新的页结构。

```sql
-- 重建单个索引
ALTER INDEX IX_Products_Price ON Products REBUILD;

-- 重建所有索引
ALTER INDEX ALL ON Products REBUILD;

-- 联机重建（SQL Server Enterprise Edition）
ALTER INDEX IX_Products_Price ON Products REBUILD WITH (ONLINE = ON);

-- 离线重建（更快，但阻塞访问）
ALTER INDEX IX_Products_Price ON Products REBUILD WITH (ONLINE = OFF);
```

### 填充因子

填充因子（Fill Factor）控制叶级页的填充率。

| 填充因子 | 适用场景 |
|---|---|
| 100%（0） | 静态表，几乎不更新 |
| 90% | 少量更新的表 |
| 80% | 常规表 |
| 70% | 高并发写入表 |
| 50% | 频繁批量插入 |

```sql
-- 创建索引时指定填充因子
CREATE NONCLUSTERED INDEX IX_Products_Price 
ON Products(Price)
WITH (FILLFACTOR = 80, PAD_INDEX = ON);

-- 修改现有索引的填充因子
ALTER INDEX IX_Products_Price ON Products REBUILD WITH (FILLFACTOR = 70);
```

:::warning
填充因子不是永久的。随着页分裂，新页会逐渐填满。定期重建索引才能保持填充因子。
:::

## 索引设计原则

### 选择合适的键顺序

聚集索引键顺序决定了数据的物理存储顺序。

**应该**：
- 选择唯一或近似唯一的列
- 选择单调递增/递减的列（自增 ID、时间戳）
- 选择查询中最常用的排序列

**不应该**：
- 使用 UUID/GUID（随机性导致页分裂）
- 使用多列组合键（除非确实需要）
- 使用长度大的列

### 非聚集索引设计

非聚集索引应该：
1. 覆盖查询所需列（INCLUDE）
2. 选择性高（基数大）
3. 键值小

```sql
-- 覆盖索引
CREATE NONCLUSTERED INDEX IX_Orders_CustomerDate 
ON Orders(CustomerID, OrderDate)
INCLUDE (Amount, Status);
-- 查找 CustomerID=1, 2024年的订单及其金额、状态
-- 可以直接从索引获取，无需回表
```

### 复合索引

复合索引的列顺序很重要。

**最左前缀原则**：查询必须从第一列开始匹配，才能使用索引。

```sql
-- 创建复合索引
CREATE INDEX IX_Orders_CustDateStatus 
ON Orders(CustomerID, OrderDate, Status);

-- 能使用索引的查询
SELECT * FROM Orders WHERE CustomerID = 1;              -- ✓ 使用
SELECT * FROM Orders WHERE CustomerID = 1 AND OrderDate > '2024-01-01';  -- ✓ 使用
SELECT * FROM Orders WHERE CustomerID = 1 AND Status = 'Shipped';      -- ✓ 使用

-- 不能使用索引的查询
SELECT * FROM Orders WHERE OrderDate > '2024-01-01';     -- ✗ 不使用
SELECT * FROM Orders WHERE Status = 'Shipped';            -- ✗ 不使用
```

理解 B+ 树的原理，才能设计出高效的索引。
