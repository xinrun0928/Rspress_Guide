# 执行计划分析

「为什么这条查询慢？」

看执行计划。执行计划是 SQL Server 告诉你「它准备怎么执行你的查询」的唯一窗口。

学会读执行计划，是 SQL Server 性能调优的第一步。

## 获取执行计划

### 图形化执行计划

在 SSMS 中：
- 按 `Ctrl + M` 开启「包含实际执行计划」
- 执行查询
- 查看「执行计划」标签页

### 文本执行计划

```sql
-- 估计执行计划
SET SHOWPLAN_TEXT ON;
GO
SELECT * FROM Orders WHERE CustomerID = 100;
GO
SET SHOWPLAN_TEXT OFF;

-- 输出示例
  |--Clustered Index Seek(OBJECT:([Orders].[PK_Orders]), 
  |    SEEK:([Orders].[CustomerID]=(100)) ORDERED FORWARD)
```

### XML 执行计划

```sql
-- 实际执行计划（XML 格式）
SET STATISTICS XML ON;
GO
SELECT * FROM Orders WHERE CustomerID = 100;
GO
SET STATISTICS XML OFF;
```

## 读取执行计划

### 计划树结构

执行计划从右到左、从上到下阅读：

```
                                    ┌─────────────┐
                                    │  SELECT     │ ← 最右节点：结果返回
                                    └──────┬──────┘
                                           │
                                    ┌──────┴──────┐
                                    │  SORT       │ ← 中间节点：处理
                                    └──────┬──────┘
                                           │
                               ┌───────────┴───────────┐
                               │      Hash Match      │
                               └───────┬───────────────┘
                                       │
                    ┌──────────────────┼──────────────────┐
                    │                  │                  │
             ┌──────┴──────┐    ┌──────┴──────┐    ┌──────┴──────┐
             │ Index Seek  │    │ Index Seek  │    │ Table Scan  │
             └─────────────┘    └─────────────┘    └─────────────┘
                                              ↑最左节点：数据源
```

### 箭头方向和粗细

- **箭头方向**：数据流向
- **箭头粗细**：行数（粗 = 多行）
- **颜色**：不同类型操作（红色 = 警告）

## 关键操作符

### 扫描操作符

| 操作符 | 含义 | 性能 |
|---|---|---|
| Table Scan | 堆表全表扫描 | 最差 |
| Clustered Index Scan | 聚集索引全扫描 | 差 |
| Index Scan | 非聚集索引全扫描 | 中等 |
| Index Seek | 索引范围查找 | 好 |

**Table Scan / Clustered Index Scan**：
- 扫描整个表或索引
- 大数据量时性能差
- 通常意味着缺少索引或查询无法使用索引

**Index Seek**：
- 精确定位
- 只需读取目标数据
- 性能最佳

```
Index Scan（扫描全部）：
┌──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┐
│  ✓   │  ✓   │  ✓   │  ✓   │  ✓   │  ✓   │  ✓   │  ✓   │ ← 扫描每个节点
└──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┘

Index Seek（精确查找）：
┌──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┐
│      │      │      │      │  ✓   │      │      │      │ ← 只找目标
└──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┘
```

### 查找操作符

| 操作符 | 含义 |
|---|---|
| Key Lookup | 聚集索引查找（回表） |
| RID Lookup | 堆表查找（回表） |
| Nested Loop | 嵌套循环连接 |
| Hash Match | 哈希连接 |
| Merge Join | 合并连接 |

**Key Lookup / RID Lookup**：
- 非聚集索引回表获取完整行
- 成本高，应该用 INCLUDE 或覆盖索引消除

```sql
-- 产生 Key Lookup 的查询
SELECT * FROM Orders WHERE CustomerID = 100;
-- 非聚集索引 IX_CustomerID 定位行，再回聚集索引取所有列

-- 优化：添加 INCLUDE
CREATE NONCLUSTERED INDEX IX_Orders_CustomerID 
ON Orders(CustomerID)
INCLUDE (OrderDate, TotalAmount, Status);
```

### 连接操作符

**Nested Loop（嵌套循环）**：
- 外表小、内表有索引时高效
- 类似双重 FOR 循环

```
FOR each row in outer_table:
    FOR each row in inner_table:
        IF condition:
            output row
```

**Hash Match（哈希连接）**：
- 等值连接，无索引时使用
- 大表连接首选
- 需要内存构建哈希表

**Merge Join（合并连接）**：
- 两个输入已排序
- 适合大数据量等值连接
- 需要 `ORDER BY` 或索引支持

| 连接类型 | 适用场景 |
|---|---|
| Nested Loop | 小表驱动大表，内表有索引 |
| Hash Match | 大表等值连接，无合适索引 |
| Merge Join | 两个输入已排序 |

### 聚合操作符

| 操作符 | 含义 |
|---|---|
| Stream Aggregate | 流式聚合（需要排序输入） |
| Hash Aggregate | 哈希聚合（无排序要求） |
| Sort | 排序 |
| Distinct Sort | 去重+排序 |

## 执行计划成本

### 成本计算

执行计划显示每个操作符的**估计成本（Estimated Subtree Cost）**。

```
┌─────────────────────────────────────────────────────────────────┐
│  Sort                                                  0.025     │
│    │                                                     ↑       │
│  Stream Aggregate                                      0.015     │
│    │                                                     ↑       │
│  Clustered Index Scan (Orders)                          0.125     │
│                                                        ↑        │
│  Total: 0.165                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 识别高成本操作

1. 找箭头最粗的节点（行数多）
2. 找成本百分比最高的节点
3. 关注警告图标（黄色三角）

## 常见警告

### 列未引用

黄色三角：`Columns with no statistics`

```sql
-- 添加统计信息
UPDATE STATISTICS Orders;
-- 或
CREATE STATISTICS STAT_Orders_Category ON Orders(CategoryID);
```

### 隐式转换

红色三角：`Type conversion in expression`

```sql
-- 问题：CustomerID 是 INT，参数是 VARCHAR
CREATE INDEX IX_Orders_CustomerID ON Orders(CustomerID);

SELECT * FROM Orders WHERE CustomerID = '100';  -- 字符串转 INT
-- 执行计划会显示隐式转换，导致索引失效

-- 解决：使用正确类型
SELECT * FROM Orders WHERE CustomerID = 100;
```

### 缺少索引

绿色下划线：`Missing Index`

执行计划会建议创建索引：

```
Missing Index (Impact: 85%)
CREATE NONCLUSTERED INDEX IX_Orders_CustomerDate 
ON Orders(CustomerID, OrderDate)
WHERE ...
```

### 并行执行

`Parallelism` 操作符表示查询使用多线程。

- 适合大数据量
- 小数据量时反而增加开销
- 注意 `MAXDOP` 设置

## 统计信息

### 估计行数 vs 实际行数

```
┌─────────────────────────────────────────────────────┐
│ Clustered Index Scan                                │
│  Estimated Rows: 1000                              │ ← 优化器估计
│  Actual Rows: 50000                                │ ← 实际执行
│  ↑ 差距大 = 统计信息过时                           │
└─────────────────────────────────────────────────────┘
```

### 查看统计信息

```sql
-- 查看表的统计信息
EXEC sp_helpstats 'Orders', 'ALL';

-- 查看统计信息详情
DBCC SHOW_STATISTICS ('Orders', 'IX_Orders_CustomerID');
```

## 优化技巧

### 消除 Key Lookup

```sql
-- 原始查询
SELECT OrderID, CustomerID, OrderDate, TotalAmount
FROM Orders
WHERE CustomerID = 100;

-- 优化：创建覆盖索引
CREATE NONCLUSTERED INDEX IX_Orders_CustomerID 
ON Orders(CustomerID)
INCLUDE (OrderDate, TotalAmount);
```

### 优化联接顺序

```sql
-- 小表驱动大表
SELECT o.OrderID, c.CustomerName
FROM Customers c
JOIN Orders o ON c.CustomerID = o.CustomerID
WHERE c.Region = 'North';
-- 假设 Customers 过滤后只有 100 行，Orders 有 1000 万行
-- 优化器应该选择 Customers 作为外表
```

### 避免隐式转换

```sql
-- 检查列类型
SELECT COLUMN_NAME, DATA_TYPE, CHARACTER_MAXIMUM_LENGTH
FROM INFORMATION_SCHEMA.COLUMNS
WHERE TABLE_NAME = 'Orders';

-- 确保参数类型匹配
-- 如果 CustomerID 是 VARCHAR
DECLARE @CustomerID VARCHAR(10) = '100';  -- 正确
```

### 使用提示（谨慎）

```sql
-- 强制使用索引
SELECT * FROM Orders WITH (INDEX(IX_Orders_CustomerID))
WHERE CustomerID = 100;

-- 强制连接类型
SELECT * FROM Orders o
INNER HASH JOIN Customers c ON o.CustomerID = c.CustomerID;

-- 强制并行度
SELECT * FROM Orders OPTION (MAXDOP 1);
```

## DMF 监控执行计划

```sql
-- 查看当前缓存的执行计划
SELECT 
    qs.execution_count,
    qs.total_elapsed_time / 1000 AS TotalMs,
    qs.total_logical_reads,
    qs.total_physical_reads,
    SUBSTRING(qt.text, qs.statement_start_offset / 2 + 1,
        (CASE qs.statement_end_offset 
            WHEN -1 THEN DATALENGTH(qt.text)
            ELSE qs.statement_end_offset 
        END - qs.statement_start_offset) / 2 + 1) AS QueryText,
    cp.usecounts AS PlanUseCount
FROM sys.dm_exec_query_stats qs
CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle) qt
JOIN sys.dm_exec_cached_plans cp ON qs.plan_handle = cp.plan_handle
ORDER BY qs.total_logical_reads DESC;
```

执行计划是诊断 SQL 性能问题的利器。养成查看执行计划的习惯，问题会变得清晰可见。
