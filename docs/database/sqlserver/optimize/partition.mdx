# 表分区

「单表超过 1 亿行，怎么处理？」

当表的数据量达到亿级时，传统方法已经无法满足性能需求。**表分区（Table Partitioning）** 是 SQL Server 处理大表的利器。

## 分区概述

### 什么是表分区

表分区将大表横向拆分为多个小表（分区），对应用透明：

```
┌─────────────────────────────────────────────────────────────────┐
│                    未分区表                                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                    Orders 表（1 亿行）                  │   │
│  │                                                         │   │
│  │  全部数据存储在一个结构中                                 │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                    分区表                                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐              │
│  │ Orders  │ │ Orders  │ │ Orders  │ │ Orders  │              │
│  │ _2024Q1 │ │ _2024Q2 │ │ _2024Q3 │ │ _2024Q4 │              │
│  │(2500万) │ │(2500万) │ │(2500万) │ │(2500万) │              │
│  └─────────┘ └─────────┘ └─────────┘ └─────────┘              │
│                                                                 │
│  应用只看到 Orders 表，分区对应用透明                            │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 分区的优势

| 优势 | 说明 |
|---|---|
| 查询性能 | 只扫描相关分区，减少 I/O |
| 数据管理 | 轻松删除旧分区（Switch/Drop） |
| 并行操作 | 各分区可并行处理 |
| 索引管理 | 可独立管理各分区索引 |
| 锁粒度 | 分区级别锁替代表级锁 |
| 备份恢复 | 可选择性地备份分区 |

## 分区函数

### 创建分区函数

```sql
-- 按日期范围分区
CREATE PARTITION FUNCTION pf_Orders (DATETIME)
AS RANGE RIGHT FOR VALUES 
(
    '2024-01-01',
    '2024-04-01',
    '2024-07-01',
    '2024-10-01',
    '2025-01-01'
);

-- 解释：
-- 分区 1: < 2024-01-01
-- 分区 2: 2024-01-01 ~ 2024-04-01
-- 分区 3: 2024-04-01 ~ 2024-07-01
-- 分区 4: 2024-07-01 ~ 2024-10-01
-- 分区 5: 2024-10-01 ~ 2025-01-01
-- 分区 6: >= 2025-01-01
```

### RANGE LEFT vs RANGE RIGHT

```sql
-- RANGE RIGHT：边界值属于右侧分区（推荐）
CREATE PARTITION FUNCTION pf_RangeRight (INT)
AS RANGE RIGHT FOR VALUES (100, 200, 300);
-- 分区 1: (-∞, 100]
-- 分区 2: (100, 200]
-- 分区 3: (200, 300]
-- 分区 4: (300, +∞)

-- RANGE LEFT：边界值属于左侧分区
CREATE PARTITION FUNCTION pf_RangeLeft (INT)
AS RANGE LEFT FOR VALUES (100, 200, 300);
-- 分区 1: (-∞, 100)
-- 分区 2: [100, 200)
-- 分区 3: [200, 300)
-- 分区 4: [300, +∞)
```

### 其他分区类型

```sql
-- 按范围分区
CREATE PARTITION FUNCTION pf_Amount (DECIMAL(10,2))
AS RANGE RIGHT FOR VALUES (1000, 5000, 10000, 50000);

-- 按哈希分区（SQL Server 2019+）
CREATE PARTITION FUNCTION pf_Hash (INT)
AS RANGE HASH FOR VALUES (10, 20, 30, 40);
```

## 分区方案

### 创建分区方案

```sql
-- 创建分区方案，指定分区存储位置
CREATE PARTITION SCHEME ps_Orders
AS PARTITION pf_Orders
TO (
    FG_Orders_Archive,   -- 分区 1
    FG_Orders_2024Q1,    -- 分区 2
    FG_Orders_2024Q2,    -- 分区 3
    FG_Orders_2024Q3,    -- 分区 4
    FG_Orders_2024Q4,    -- 分区 5
    FG_Orders_Current    -- 分区 6
);

-- 简化：多个分区使用同一文件组
CREATE PARTITION SCHEME ps_Orders_Simple
AS PARTITION pf_Orders
TO (FG_Orders, FG_Orders, FG_Orders, FG_Orders, FG_Orders, FG_Orders);
```

### NEXT USED 文件组

```sql
-- 创建方案时指定 NEXT USED
CREATE PARTITION SCHEME ps_Orders_Next
AS PARTITION pf_Orders
TO (FG1, FG2, FG3, FG4, FG5, FG6, FG_NextUsed);
-- FG_NextUsed 将在未来添加分区时自动使用
```

## 分区表

### 创建分区表

```sql
-- 创建分区表
CREATE TABLE Orders (
    OrderID INT IDENTITY(1,1),
    OrderDate DATETIME2 NOT NULL,
    CustomerID INT NOT NULL,
    ProductID INT,
    Quantity INT,
    Amount DECIMAL(10,2),
    Status TINYINT DEFAULT 1,
    CreatedDate DATETIME2 DEFAULT SYSDATETIME(),
    CONSTRAINT PK_Orders PRIMARY KEY NONCLUSTERED (OrderID)
)
ON ps_Orders(OrderDate);  -- 指定分区方案和列

-- 创建聚集索引也使用分区
CREATE CLUSTERED INDEX IX_Orders_Date
ON Orders(OrderDate)
ON ps_Orders(OrderDate);

-- 非聚集索引可独立分区
CREATE NONCLUSTERED INDEX IX_Orders_Customer
ON Orders(CustomerID)
ON ps_Orders_Customer(OrderDate);  -- 可使用不同分区方案
```

### 转换现有表为分区表

```sql
-- 方法 1：创建新分区表，迁移数据
CREATE TABLE Orders_Partitioned (
    OrderID INT,
    OrderDate DATETIME2 NOT NULL,
    ...
)
ON ps_Orders(OrderDate);

INSERT INTO Orders_Partitioned SELECT * FROM Orders;

-- 方法 2：重建聚集索引（更快）
CREATE CLUSTERED INDEX IX_Orders_Date
ON Orders(OrderDate)
ON ps_Orders(OrderDate);
-- 表自动转换为分区表
```

## 分区操作

### 查看分区信息

```sql
-- 查看分区函数
SELECT * FROM sys.partition_functions;

-- 查看分区方案
SELECT * FROM sys.partition_schemes;

-- 查看表分区信息
SELECT 
    OBJECT_NAME(p.object_id) AS TableName,
    i.name AS IndexName,
    p.partition_number,
    p.rows AS RowCount,
    fg.name AS FileGroupName,
    rv.value AS BoundaryValue
FROM sys.partitions p
JOIN sys.indexes i ON p.object_id = i.object_id AND p.index_id = i.index_id
JOIN sys.partition_schemes ps ON i.data_space_id = ps.data_space_id
JOIN sys.partition_functions pf ON ps.function_id = pf.function_id
JOIN sys.destination_data_spaces dds ON p.partition_number = dds.destination_id
JOIN sys.filegroups fg ON dds.data_space_id = fg.data_space_id
LEFT JOIN sys.partition_range_values rv ON pf.function_id = rv.function_id 
    AND p.partition_number = rv.boundary_id
WHERE OBJECTPROPERTY(p.object_id, 'IsUserTable') = 1
ORDER BY p.partition_number;
```

### 分区切换（Partition Switch）

分区切换是分区表特有的高效操作：

```sql
-- 准备切换源表（结构必须与分区表一致）
CREATE TABLE Orders_Staging (
    OrderID INT,
    OrderDate DATETIME2,
    ...
)
ON FG_Orders_2024Q1;  -- 必须在同一文件组

-- 添加分区（扩展范围）
ALTER PARTITION SCHEME ps_Orders NEXT USED FG_Orders_2025Q1;

ALTER PARTITION FUNCTION pf_Orders()
SPLIT RANGE ('2025-04-01');

-- 切换分区（秒级操作）
ALTER TABLE Orders_Staging
SWITCH TO Orders PARTITION 2;  -- 切换到分区 2

-- 清理切换源表
DROP TABLE Orders_Staging;

-- 合并分区
ALTER PARTITION FUNCTION pf_Orders()
MERGE RANGE ('2024-01-01');
-- 合并分区 1 和 2，数据合并到分区 2
```

### 分区删除

```sql
-- 删除旧分区数据（不删除分区）
DELETE FROM Orders 
WHERE OrderDate < '2023-01-01';
-- 慢，需要删除大量行

-- 更优：切换分区后删除表
-- 1. 创建临时表结构相同的表
CREATE TABLE Orders_Archive_2023 (
    ...结构相同...
)
ON FG_Orders_Archive;

-- 2. 切换分区
ALTER TABLE Orders
SWITCH PARTITION 1 TO Orders_Archive_2023;

-- 3. 删除临时表（删除实际数据）
DROP TABLE Orders_Archive_2023;
```

## 分区索引

### 分区索引类型

| 类型 | 说明 |
|---|---|
| 对齐索引 | 与表使用相同分区方案 |
| 非对齐索引 | 使用不同分区方案或未分区 |
| 本地索引 | 每个分区有独立索引 |
| 全局索引 | 跨所有分区索引 |

```sql
-- 对齐的聚集索引（默认）
CREATE CLUSTERED INDEX IX_Orders_Date
ON Orders(OrderDate)
ON ps_Orders(OrderDate);  -- 与表对齐

-- 非对齐索引
CREATE NONCLUSTERED INDEX IX_Orders_CustomerID
ON Orders(CustomerID)
ON FG_Index;  -- 使用不同文件组

-- 独立分区索引（SQL Server 2016+）
CREATE NONCLUSTERED INDEX IX_Orders_CustomerID
ON Orders(CustomerID)
ON ps_Orders_Customer(OrderDate);  -- 可独立分区
```

### 分区索引维护

```sql
-- 重建单个分区索引
ALTER INDEX IX_Orders_Date ON Orders
REBUILD PARTITION = 5;  -- 只重建分区 5

-- 重新组织单个分区
ALTER INDEX IX_Orders_Date ON Orders
REORGANIZE PARTITION = 5;

-- 查看索引物理统计
SELECT 
    OBJECT_NAME(ips.object_id) AS TableName,
    ips.index_id,
    ips.partition_number,
    ips.avg_fragmentation_in_percent,
    ips.page_count
FROM sys.dm_db_index_physical_stats(
    DB_ID(), 
    OBJECT_ID('Orders'),
    NULL, NULL, 'DETAILED'
) ips
WHERE ips.alloc_unit_type_desc = 'IN_ROW_DATA'
ORDER BY ips.partition_number;
```

## 分区与锁升级

### 分区级别锁升级

分区表支持分区级别锁升级，减少锁争用：

```sql
-- 启用分区级别锁升级
ALTER TABLE Orders SET (LOCK_ESCALATION = AUTO);
-- 当锁升级发生时，只锁定相关分区，不锁定整个表
```

### 锁升级对比

| 模式 | 行为 |
|---|---|
| TABLE | 锁升级到表级别（默认） |
| AUTO | 锁升级到分区级别（分区表） |
| DISABLE | 禁用锁升级 |

## 分区使用场景

### 场景一：历史数据归档

```sql
-- 订单表按年分区
CREATE PARTITION FUNCTION pf_Year (INT)
AS RANGE RIGHT FOR VALUES (2020, 2021, 2022, 2023, 2024, 2025);

CREATE PARTITION SCHEME ps_Year
AS PARTITION pf_Year
TO (FG_2020, FG_2021, FG_2022, FG_2023, FG_2024, FG_Current);

CREATE TABLE Orders (
    OrderID INT PRIMARY KEY,
    OrderYear INT,
    ...
)
ON ps_Year(OrderYear);

-- 归档 2020 年数据
ALTER TABLE Orders SWITCH PARTITION 1 TO Orders_Archive_2020;
```

### 场景二：滑动窗口

```sql
-- 保留最近 12 个月数据
-- 每月自动添加新分区
-- 每月删除 13 个月前的分区

-- 添加新分区（月初执行）
ALTER PARTITION SCHEME ps_Orders NEXT USED FG_Orders_2025Q2;
ALTER PARTITION FUNCTION pf_Orders()
SPLIT RANGE ('2025-07-01');

-- 删除旧分区（月初执行）
ALTER PARTITION FUNCTION pf_Orders()
MERGE RANGE ('2024-06-01');  -- 或 SWITCH 后 DROP
```

### 场景三：分区视图

```sql
-- 替代方案：分区视图（兼容旧版本）
CREATE TABLE Orders_2024 (
    OrderID INT,
    ...
    CONSTRAINT CK_Orders_2024 CHECK (OrderDate >= '2024-01-01' AND OrderDate < '2025-01-01')
);

CREATE TABLE Orders_2025 (
    OrderID INT,
    ...
    CONSTRAINT CK_Orders_2025 CHECK (OrderDate >= '2025-01-01' AND OrderDate < '2026-01-01')
);

CREATE VIEW Orders_All AS
SELECT * FROM Orders_2024
UNION ALL
SELECT * FROM Orders_2025;
```

## 分区性能优化

### 查询优化

```sql
-- 查看查询使用的分区
SET STATISTICS IO ON;

SELECT * FROM Orders 
WHERE OrderDate BETWEEN '2024-01-01' AND '2024-03-31';
-- 查看执行计划，确认只扫描相关分区

-- 强制使用分区
SELECT * FROM Orders WITH (INDEX(IX_Orders_Date))
WHERE OrderDate BETWEEN '2024-01-01' AND '2024-03-31';
```

### 统计信息

```sql
-- 分区级别统计信息
CREATE STATISTICS STAT_Orders_CustomerID 
ON Orders(CustomerID)
WITH INCREMENTAL = ON;
-- 每个分区独立维护统计信息
```

分区是处理大表的核心技术，合理设计分区策略能显著提升性能和可维护性。
