# 内存优化表

「高并发写入场景，传统表性能不够怎么办？」

SQL Server 2014 引入了 **In-Memory OLTP**（代号 Hekaton），通过内存优化表和本机编译存储过程，实现超高并发和极低延迟。

## In-Memory OLTP 概述

### 核心概念

In-Memory OLTP 将表完全存储在内存中，消除了传统磁盘存储的瓶颈：

```
┌─────────────────────────────────────────────────────────────────┐
│                 传统磁盘表 vs 内存优化表                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  传统磁盘表：                                                    │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  Buffer Pool                    磁盘文件 (.mdf)          │   │
│  │  ┌────────┐                                             │   │
│  │  │ 数据页 │ ──▶ I/O ──▶ ┌────────┐                    │   │
│  │  │ 缓存  │              │ 数据页 │                    │   │
│  │  └────────┘              └────────┘                    │   │
│  │  锁 + 日志               等待 I/O                     │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  内存优化表：                                                    │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  内存 (完全驻留)                                         │   │
│  │  ┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐           │   │
│  │  │ 行版本 │ │ 行版本 │ │ 行版本 │ │ 行版本 │           │   │
│  │  └────────┘ └────────┘ └────────┘ └────────┘           │   │
│  │  无锁（乐观并发）        多版本                        │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 优势

| 特性 | 传统表 | 内存优化表 |
|---|---|---|
| 访问速度 | 毫秒级 | 微秒级 |
| 并发能力 | 中等 | 极高 |
| 锁机制 | 行锁/页锁 | 无锁（多版本） |
| 日志 | 完整日志 | 可配置 |
| 索引 | B+ 树 | 哈希索引、内存优化 B+ 树 |

## 内存优化文件组

### 创建内存优化文件组

```sql
-- 添加内存优化文件组
ALTER DATABASE [YourDB] 
ADD FILEGROUP YourDB_Hekaton 
CONTAINS MEMORY_OPTIMIZED_DATA;

-- 添加内存优化数据文件
ALTER DATABASE [YourDB] 
ADD FILE (
    NAME = 'YourDB_Hekaton',
    FILENAME = 'D:\Hekaton\YourDB_Hekaton',
    SIZE = 1GB,
    MAXSIZE = 100GB,
    FILEGROWTH = 1GB
) TO FILEGROUP YourDB_Hekaton;
```

:::warning
每个数据库只能有一个 `MEMORY_OPTIMIZED_DATA` 文件组。
:::

## 内存优化表

### 创建内存优化表

```sql
CREATE TABLE Orders_InMemory (
    OrderID INT IDENTITY(1,1) NOT NULL,
    OrderDate DATETIME2 NOT NULL,
    CustomerID INT NOT NULL,
    ProductID INT,
    Quantity INT,
    Amount DECIMAL(10,2),
    Status TINYINT DEFAULT 1,
    CreatedDate DATETIME2 DEFAULT SYSDATETIME(),
    CONSTRAINT PK_Orders_InMemory PRIMARY KEY NONCLUSTERED (OrderID)
) WITH (
    MEMORY_OPTIMIZED = ON,
    DURABILITY = SCHEMA_AND_DATA  -- 或 SCHEMA_ONLY
);

-- DURABILITY 选项：
-- SCHEMA_AND_DATA: 持久化表，内存+磁盘检查点
-- SCHEMA_ONLY: 非持久化表，仅内存（崩溃丢失）
```

### 索引类型

```sql
CREATE TABLE Orders_InMemory_Indexed (
    OrderID INT NOT NULL,
    OrderDate DATETIME2 NOT NULL,
    CustomerID INT NOT NULL,
    Amount DECIMAL(10,2),
    
    -- 非聚集主键（必须有主键或唯一索引）
    CONSTRAINT PK_Orders_InMemory_Indexed 
        PRIMARY KEY NONCLUSTERED (OrderID),
    
    -- 哈希索引（等值查询优化）
    INDEX IX_CustomerID HASH (CustomerID) WITH (BUCKET_COUNT = 1000000),
    
    -- 范围索引（范围查询）
    INDEX IX_OrderDate NONCLUSTERED (OrderDate)
) WITH (MEMORY_OPTIMIZED = ON, DURABILITY = SCHEMA_AND_DATA);
```

### 哈希索引

适合等值查询（`WHERE CustomerID = 100`）：

```sql
-- 创建哈希索引
CREATE TABLE Customers_InMemory (
    CustomerID INT NOT NULL,
    CustomerName NVARCHAR(100),
    Email NVARCHAR(255),
    CreatedDate DATETIME2 DEFAULT SYSDATETIME(),
    CONSTRAINT PK_Customers PRIMARY KEY NONCLUSTERED (CustomerID),
    INDEX IX_Customers_Email HASH (Email) WITH (BUCKET_COUNT = 100000)
) WITH (MEMORY_OPTIMIZED = ON, DURABILITY = SCHEMA_AND_DATA);

-- BUCKET_COUNT 估算：
-- BUCKET_COUNT = 预估行数 * 1~2
-- 或 BUCKET_COUNT = 最高并发数 * 10
```

### 非聚集索引

适合范围查询（`WHERE OrderDate BETWEEN ...`）：

```sql
-- 范围查询使用非聚集索引
CREATE TABLE Orders_DateRange (
    OrderID INT NOT NULL,
    OrderDate DATETIME2 NOT NULL,
    Amount DECIMAL(10,2),
    CONSTRAINT PK_Orders_DateRange PRIMARY KEY NONCLUSTERED (OrderID),
    INDEX IX_OrderDate NONCLUSTERED (OrderDate)
) WITH (MEMORY_OPTIMIZED = ON, DURABILITY = SCHEMA_AND_DATA);
```

### 支持的数据类型

| 支持 | 不支持 |
|---|---|
| INT, BIGINT, SMALLINT | TEXT, NTEXT |
| VARCHAR, NVARCHAR | IMAGE |
| DATETIME, DATETIME2, DATE, TIME | XML |
| DECIMAL, NUMERIC | ROWVERSION |
| BIT, MONEY | SQL _variant |
| UNIQUEIDENTIFIER | 地理/几何类型 |
| FLOAT, REAL | 用户定义类型 |

### 限制

```sql
-- 不支持的功能：
-- 1. 外键约束
-- 2. CHECK 约束（SQL Server 2016+ 支持）
-- 3. 触发器（SQL Server 2016+ 支持）
-- 4. 表级锁
-- 5. IDENTITY 种子为 0 或负数
-- 6. 某些 DDL 操作

-- 内存限制：
-- 持久化表：表大小 + 索引大小 <= 内存
-- 非持久化表：表大小 + 索引大小 + Delta store <= 内存
```

## 访问内存优化表

### 普通 T-SQL

```sql
-- 插入
INSERT INTO Orders_InMemory (OrderDate, CustomerID, Amount)
VALUES (SYSDATETIME(), 100, 199.99);

-- 更新
UPDATE Orders_InMemory
SET Amount = 299.99
WHERE OrderID = 1;

-- 删除
DELETE FROM Orders_InMemory
WHERE OrderID = 1;

-- 查询
SELECT * FROM Orders_InMemory
WHERE CustomerID = 100
ORDER BY OrderDate DESC;
```

### 本机编译存储过程

本机编译存储过程（Natively Compiled Stored Procedure）性能最高：

```sql
CREATE PROCEDURE usp_InsertOrder
    @CustomerID INT,
    @Amount DECIMAL(10,2)
WITH NATIVE_COMPILATION, SCHEMABINDING
AS
BEGIN ATOMIC WITH (TRANSACTION ISOLATION LEVEL = SNAPSHOT, LANGUAGE = N'English')
    DECLARE @OrderID INT = NEXT VALUE FOR SEQ_Orders;
    INSERT INTO dbo.Orders_InMemory (OrderID, OrderDate, CustomerID, Amount)
    VALUES (@OrderID, SYSDATETIME(), @CustomerID, @Amount);
    SELECT @OrderID AS OrderID;
END;

-- 调用
EXEC usp_InsertOrder @CustomerID = 100, @Amount = 199.99;
```

### 本机编译 vs 解释存储过程

| 特性 | 本机编译 | 解释存储过程 |
|---|---|---|
| 编译方式 | 编译为 C 代码，再编译为 DLL | 解释执行 |
| 性能 | 极高 | 比普通 T-SQL 略好 |
| 限制 | 只访问内存优化表 | 无限制 |
| 调试 | 不支持 | 支持 |

## 监控

### 内存使用

```sql
-- 查看内存优化表内存使用
SELECT 
    OBJECT_NAME(object_id) AS TableName,
    memory_consumer_type_desc,
    memory_consumer_desc,
    allocated_bytes / 1024.0 / 1024.0 AS AllocatedMB,
    used_bytes / 1024.0 / 1024.0 AS UsedMB
FROM sys.dm_db_xtp_memory_consumers;

-- 查看索引内存使用
SELECT 
    OBJECT_NAME(object_id) AS TableName,
    index_id,
    memory_consumer_desc,
    allocated_bytes / 1024.0 / 1024.0 AS AllocatedMB
FROM sys.dm_db_xtp_memory_consumers
WHERE object_id IN (
    SELECT object_id FROM sys.tables 
    WHERE is_memory_optimized = 1
);
```

### 并发统计

```sql
-- 查看内存优化表的并发统计
SELECT 
    xtp_scans,
    xtp_scan_since_last_stats AS scans_since_restart,
    num_compiled_queries AS compiled_queries,
    num_native_compiled_procedures AS native_procedures
FROM sys.dm_db_xtp_system_memory_consumers;
```

### 性能监控

```sql
-- 查看等待统计
SELECT 
    wait_type,
    waiting_tasks_count,
    wait_time_ms
FROM sys.dm_os_wait_stats
WHERE wait_type LIKE 'HADR%'  -- 内存优化相关等待
ORDER BY wait_time_ms DESC;

-- 查看事务统计
SELECT 
    total_count,
    failure_count,
    conflict_count,
    conversion_count
FROM sys.dm_db_xtp_transactions;
```

## 使用场景

### 适合场景

| 场景 | 原因 |
|---|---|
| 高频写入 | 无锁写入，极高吞吐 |
| 低延迟要求 | 微秒级响应 |
| 会话状态存储 | 快速读写 |
| 临时数据 | 非持久化表 |
| 参考数据 | 内存缓存 |

### 不适合场景

| 场景 | 原因 |
|---|---|
| 超大表 | 受内存限制 |
| 复杂分析查询 | 列存储索引更合适 |
| 需要外键 | 不支持 |
| 大量历史数据 | 内存成本高 |

## 迁移策略

### 迁移步骤

```sql
-- 1. 评估内存需求
SELECT 
    OBJECT_NAME(object_id) AS TableName,
    SUM(reserved_page_count) * 8 / 1024.0 AS ReservedMB,
    SUM(row_count) AS RowCount
FROM sys.dm_db_partition_stats
WHERE OBJECTPROPERTY(object_id, 'IsUserTable') = 1
GROUP BY object_id
ORDER BY ReservedMB DESC;

-- 2. 创建内存优化文件组
ALTER DATABASE [YourDB] ADD FILEGROUP Hekaton CONTAINS MEMORY_OPTIMIZED_DATA;
ALTER DATABASE [YourDB] ADD FILE (...) TO FILEGROUP Hekaton;

-- 3. 创建内存优化表（先 SCHEMA_ONLY）
CREATE TABLE Orders_InMemory (...)
WITH (MEMORY_OPTIMIZED = ON, DURABILITY = SCHEMA_ONLY);

-- 4. 迁移数据
INSERT INTO Orders_InMemory SELECT * FROM Orders;

-- 5. 切换为持久化
ALTER TABLE Orders_InMemory SET (DURABILITY = SCHEMA_AND_DATA);
```

内存优化表是 OLTP 场景的性能利器，但需要合理评估内存需求和迁移成本。
