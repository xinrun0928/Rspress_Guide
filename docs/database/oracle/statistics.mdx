# Oracle 统计信息：优化器的「眼睛」

你有没有遇到过这种情况：同样的 SQL，昨天快今天慢。明明加了索引，Oracle 就是不用。表里只有 100 条数据，Oracle 却做了全表扫描。

这些问题的答案，都在统计信息里。

## 什么是统计信息？

统计信息是 Oracle 收集的关于表、索引、列的数据特征。优化器根据这些「情报」判断执行计划的优劣。

```sql title="统计信息内容"
统计信息内容：
├── 表统计
│   ├── 行数（NUM_ROWS）
│   ├── 块数（BLOCKS）
│   └── 平均行长度（AVG_ROW_LEN）
├── 列统计
│   ├── 唯一值数量（NUM_DISTINCT）
│   ├── 空值数量（NULLS）
│   └── 数据分布（直方图）
└── 索引统计
    ├── 唯一值（NUM_DISTINCT）
    ├── 叶块数（LEAF_BLOCKS）
    └── 聚簇因子（CLUSTERING_FACTOR）
```

:::tip 统计信息存放在哪里？

表和列的统计信息存放在数据字典中，如 `USER_TABLES`、`USER_TAB_COLUMNS`。
索引的统计信息存放在 `USER_INDEXES`。Oracle 自动维护这些字典表。

:::

## 统计信息的作用

Oracle 优化器根据统计信息选择最优执行计划。没有统计信息和有统计信息时，优化器的决策完全不同：

```sql title="有无统计信息的执行计划差异"
-- 没有统计信息：优化器「瞎猜」
SELECT * FROM employees WHERE department_id = 50;
-- 优化器假设返回 1000 行，可能选择全表扫描

-- 有统计信息：优化器「看数据」
SELECT * FROM employees WHERE department_id = 50;
-- 优化器知道只有 50 行，可能选择索引扫描
```

:::warning 统计信息过期是生产环境的常见杀手

当表数据量发生大规模变化（超过 10%）后，如果统计信息没有及时更新，优化器就会基于「过时情报」做出错误判断——该用索引的选了全表扫描，该用小表驱动的选了大表驱动。

:::

## 查看统计信息

### 查看表统计

```sql title="user_tables"
-- 查看表统计信息
SELECT table_name, num_rows, blocks, avg_row_len,
       sample_size, last_analyzed
FROM user_tables
WHERE table_name = 'EMPLOYEES';
```

### 查看列统计

```sql title="user_tab_columns"
-- 查看列统计信息
SELECT column_name, num_distinct, num_nulls,
       density, histogram, sample_size, last_analyzed
FROM user_tab_columns
WHERE table_name = 'EMPLOYEES'
ORDER BY column_id;
```

### 查看直方图

```sql title="user_tab_histograms"
-- 查看直方图信息
SELECT column_name, endpoint_number, endpoint_value
FROM user_tab_histograms
WHERE table_name = 'EMPLOYEES'
  AND column_name = 'DEPARTMENT_ID'
ORDER BY endpoint_number;
```

### 查看索引统计

```sql title="user_indexes"
-- 查看索引统计信息
SELECT index_name, blevel, leaf_blocks, distinct_keys,
       avg_leaf_blocks_per_key, avg_data_blocks_per_key,
       clustering_factor, last_analyzed
FROM user_indexes
WHERE table_name = 'EMPLOYEES';
```

## 收集统计信息

### DBMS_STATS 基本用法

`DBMS_STATS` 是 Oracle 推荐使用的统计信息收集工具，比早期的 `ANALYZE` 命令功能更全面。

```sql title="DBMS_STATS 收集示例"
-- 收集表统计信息
BEGIN
    DBMS_STATS.GATHER_TABLE_STATS(
        ownname => USER,
        tabname => 'EMPLOYEES',
        cascade => TRUE,
        estimate_percent => 10,
        method_opt => 'FOR ALL COLUMNS SIZE AUTO'
    );
END;
/

-- 收集模式统计信息（整个用户 schema）
BEGIN
    DBMS_STATS.GATHER_SCHEMA_STATS(
        ownname => USER,
        estimate_percent => 10,
        cascade => TRUE
    );
END;
/

-- 收集数据库统计信息（需 DBA 权限）
BEGIN
    DBMS_STATS.GATHER_DATABASE_STATS(
        estimate_percent => 10,
        cascade => TRUE
    );
END;
/
```

:::tip estimate_percent 设置建议

- `estimate_percent => 10`：采样 10%，兼顾准确性和速度，适合大多数场景
- `estimate_percent => DBMS_STATS.AUTO_SAMPLE_SIZE`：让 Oracle 自动决定采样比例 <Badge text="Oracle 11g+" type="tip" outline />
- `estimate_percent => 100`：全表扫描，最准确但最慢，适合小表

:::


### GATHER_INDEX_STATS

```sql title="收集索引统计"
BEGIN
    DBMS_STATS.GATHER_INDEX_STATS(
        ownname => USER,
        indname => 'IDX_EMP_DEPT',
        estimate_percent => 10
    );
END;
/
```


## 自动统计信息

默认情况下，Oracle 会自动在维护窗口（如每天的夜间窗口）收集统计信息。你可以通过以下方式查看和控制它。

### 查看自动统计任务

```sql title="查看自动统计任务"
-- 查看自动统计信息收集任务状态
SELECT task_name, status, enabled
FROM dba_autotask_client
WHERE client_name = 'auto optimizer stats collection';

-- 查看任务执行历史
SELECT client_name, window_name, window_start_time,
       window_duration, tasks_success, tasks_failed
FROM dba_autotask_client_history
ORDER BY window_start_time DESC;
```

### 手动控制自动任务

```sql title="启用/禁用自动统计收集"
-- 禁用自动统计收集
BEGIN
    DBMS_AUTO_TASK_ADMIN.DISABLE(
        client_name => 'auto optimizer stats collection',
        operation   => NULL,
        window_name => NULL
    );
END;
/

-- 启用自动统计收集
BEGIN
    DBMS_AUTO_TASK_ADMIN.ENABLE(
        client_name => 'auto optimizer stats collection',
        operation   => NULL,
        window_name => NULL
    );
END;
/
```

:::warning 禁用前请三思

生产环境通常依赖自动统计任务来保持统计信息不过期。只有在特殊场景下（如正在执行大批量数据迁移）才建议临时禁用，事后记得重新启用。

:::

## 直方图（Histogram）

### 什么是直方图？

直方图记录列值的数据分布，帮助优化器准确估算基数（Cardinality）。当数据分布不均匀时，没有直方图的优化器会做出错误估算：

```sql title="直方图的作用"
-- 没有直方图：优化器假设均匀分布
-- 10000 行，10 个部门，平均每部门 1000 行
-- 实际：9 个部门各 10 行，1 个部门 9910 行
-- 优化器估算错误，可能选择全表扫描

-- 有直方图：优化器知道实际分布，选择索引扫描
CREATE INDEX idx_emp_dept ON employees(department_id);
-- Oracle 11g+ 会自动决定是否收集直方图
```

### 直方图类型 <Badge text="Oracle 11g+" type="tip" outline />

| 类型 | 说明 | 适用场景 |
|-----|------|---------|
| HEIGHT BALANCED | 等高直方图 | 有限个不同值，Oracle 自动选择 |
| FREQUENCY | 频率直方图 | 不同值数量 ≤ 254 个 |

### 直方图收集策略

```sql title="直方图收集"
-- 自动策略（Oracle 推荐）
BEGIN
    DBMS_STATS.GATHER_TABLE_STATS(
        USER, 'EMPLOYEES',
        method_opt => 'FOR ALL COLUMNS SIZE AUTO'
    );
END;
/

-- 只对特定列收集直方图
BEGIN
    DBMS_STATS.GATHER_TABLE_STATS(
        USER, 'EMPLOYEES',
        method_opt => 'FOR COLUMNS SIZE AUTO department_id'
    );
END;
/

-- 手动指定桶数
BEGIN
    DBMS_STATS.GATHER_TABLE_STATS(
        USER, 'EMPLOYEES',
        method_opt => 'FOR COLUMNS SIZE 50 department_id'
    );
END;
/
```

:::tip SIZE AUTO 的工作原理

`SIZE AUTO` 会让 Oracle 自动判断哪些列需要直方图以及需要多少个桶。Oracle 11g+ 默认使用此策略，一般不需要手动指定桶数。

:::

## 统计信息管理

### 查看统计信息历史

Oracle 会保留统计信息的历史版本，方便你对比和回退。

```sql title="统计信息历史"
-- 查看统计信息历史
SELECT table_name, stattype_locked, stats_update_time
FROM user_tab_stats_history
WHERE table_name = 'EMPLOYEES';

-- 查看统计信息更新时间
SELECT table_name, last_analyzed
FROM user_tables
WHERE table_name = 'EMPLOYEES';
```

### 恢复统计信息

当新统计信息导致执行计划变差时，可以恢复到之前的版本。

```sql title="恢复统计信息"
-- 恢复到一天前的统计信息
BEGIN
    DBMS_STATS.RESTORE_TABLE_STATS(
        ownname          => USER,
        tabname          => 'EMPLOYEES',
        as_of_timestamp  => SYSDATE - 1
    );
END;
/

-- 删除过时的统计信息历史
BEGIN
    DBMS_STATS.PURGE_STATS(
        before_timestamp => SYSDATE - 30
    );
END;
/
```

:::warning 恢复前的影响

恢复统计信息会改变优化器的估算基数，从而改变执行计划。恢复后务必在测试环境验证 SQL 性能是否符合预期。

:::

### 锁定统计信息

在某些场景下，你不希望自动统计任务覆盖手动收集的统计信息（比如数据分布特殊的报表表）。

```sql title="锁定/解锁统计信息"
-- 锁定表统计信息（防止自动收集覆盖）
BEGIN
    DBMS_STATS.LOCK_TABLE_STATS(
        ownname => USER,
        tabname => 'EMPLOYEES'
    );
END;
/

-- 解锁表统计信息
BEGIN
    DBMS_STATS.UNLOCK_TABLE_STATS(
        ownname => USER,
        tabname => 'EMPLOYEES'
    );
END;
/
```

## 统计信息问题排查

### 问题一：统计信息过期

```sql title="判断统计信息是否过期"
-- 查看数据量与统计信息对比
SELECT table_name,
       num_rows        "统计行数",
       (SELECT COUNT(*) FROM employees) "实际行数"
FROM user_tables
WHERE table_name = 'EMPLOYEES';
```

:::tip 过期判断标准

如果实际行数与统计行数差异超过 10%，就需要重新收集统计信息了。你也可以用 `DBMS_STATS.GET_STATS_history_RETENTION` 查看 Oracle 保留历史的时间窗口。

:::

### 问题二：统计信息缺失

```sql title="发现缺失统计信息的表"
-- 查看哪些表没有统计信息
SELECT table_name
FROM user_tables
WHERE num_rows IS NULL;

-- 为指定表收集统计信息
BEGIN
    DBMS_STATS.GATHER_TABLE_STATS(USER, 'EMPLOYEES');
END;
/
```

### 问题三：直方图缺失

```sql title="发现缺失直方图的列"
-- 查看哪些列可能需要直方图
SELECT column_name, num_distinct, histogram
FROM user_tab_columns
WHERE table_name  = 'EMPLOYEES'
  AND num_distinct > 1
  AND histogram   = 'NONE';

-- 重新收集直方图
BEGIN
    DBMS_STATS.GATHER_TABLE_STATS(
        USER, 'EMPLOYEES',
        method_opt => 'FOR ALL COLUMNS SIZE AUTO'
    );
END;
/
```

## 性能调优场景

### 场景：批量导入后

```sql title="批量导入后收集统计信息"
-- 批量导入数据后，必须收集统计信息
INSERT /*+ APPEND */ INTO employees SELECT * FROM source_employees;
COMMIT;

-- 立即收集统计信息
BEGIN
    DBMS_STATS.GATHER_TABLE_STATS(USER, 'EMPLOYEES');
END;
/
```

:::danger APPEND + 统计信息

使用 `INSERT /*+ APPEND */` 插入大量数据后，如果不立即收集统计信息，优化器会基于过期的基数估算生成错误的执行计划，影响后续查询性能。

:::

### 场景：分区表统计

分区表的统计信息分两个层次：分区级（Partition）和全局级（Global），需要分别收集。

```sql title="分区表统计信息收集"
-- 收集指定分区的统计信息
BEGIN
    DBMS_STATS.GATHER_TABLE_STATS(
        ownname   => USER,
        tabname   => 'ORDERS',
        partname  => 'P_2024_Q1',
        cascade   => TRUE
    );
END;
/

-- 收集全局统计信息
BEGIN
    DBMS_STATS.GATHER_TABLE_STATS(
        ownname   => USER,
        tabname   => 'ORDERS',
        granularity => 'GLOBAL',
        cascade   => TRUE
    );
END;
/
```

:::info 分区表统计策略

对于大分区表，通常先收集每个分区的统计信息，再收集全局统计信息。全局统计信息供跨分区查询使用，分区统计供单分区查询使用。

:::

## 面试高频问题

### Q1: 统计信息过期会导致什么问题？

优化器基于过期的统计信息选择执行计划，可能选择错误的执行计划（如应该用索引却选了全表扫描），导致 SQL 性能下降。

但这道题真正考察的是——你知道**怎么发现**和**怎么解决**吗？

- 发现：用 `user_tables` 中的 `num_rows` 与实际 COUNT(*) 对比
- 解决：重新收集统计信息，或回退到历史版本

### Q2: 什么是直方图？什么时候需要？

直方图记录列值的数据分布，帮助优化器准确估算基数。当列值分布不均匀（倾斜）时，需要直方图。

:::tip 面试追问

Oracle 11g+ 会自动决定是否收集直方图，不需要手动判断。
但面试官可能会追问：HEIGHT BALANCED 和 FREQUENCY 两种直方图有什么区别？答案在于频率直方图适用于不同值数量 ≤ 254 个的场景，Oracle 会自动选择合适的类型。

:::

### Q3: 如何处理统计信息导致的问题？

三个方向，按场景选择：

1. **回退**：`DBMS_STATS.RESTORE_TABLE_STATS` 恢复到历史版本
2. **更新**：手动 `GATHER_TABLE_STATS` 收集最新统计信息
3. **锁定**：`LOCK_TABLE_STATS` 防止自动任务覆盖

## 总结

统计信息是优化器决策的基础。理解它的工作原理，是排查 SQL 性能问题的必经之路：

| 统计类型 | 内容 | 重要性 |
|---------|------|--------|
| 表统计 | 行数、块数、平均行长度 | 高 |
| 列统计 | 唯一值、空值数 | 高 |
| 直方图 | 数据分布 | 中（倾斜时高） |
| 索引统计 | 聚簇因子、叶块数 | 高 |

保持统计信息准确，是 SQL 性能稳定的前提。
