# Oracle 分页查询：ROWNUM 与 ROW_NUMBER

分页查询是业务系统中最常见的需求之一。当一个查询返回上万条数据时，前端通常只需要展示其中某一页——这就是分页的价值所在：减少数据传输量、提升响应速度、降低服务器和客户端的内存压力。

假设你正在开发一个订单列表页面，每页显示 20 条记录。用户可以翻页查看第 1、2、3... 页。这种场景下，你需要数据库返回指定范围的数据。Oracle 提供了多种分页方式，从早期的 ROWNUM 到现代的 OFFSET/FETCH，每种方式都有其适用场景和历史背景。

## 分页的核心问题

分页查询的核心在于两个问题：**排序稳定性和结果唯一性**。

**排序稳定性**：如果 ORDER BY 的列有重复值（如多个订单的金额相同），则相同值之间的顺序是不确定的。第一页可能返回 ID 为 100、105、110 的订单，第二页返回 108、115、120——这意味着用户在翻页时可能看到重复或遗漏的记录。

**解决方案**：ORDER BY 中加入唯一列（如主键 ID），确保排序唯一。

**结果唯一性**：分页的本质是"跳过 N 行，返回 M 行"。对于同一查询，多次执行应该返回相同的结果（除非数据发生了变化）。

## ROWNUM：Oracle 的行号伪列

ROWNUM 是 Oracle 为查询结果分配的伪列，表示查询返回的行号（从 1 开始）。ROWNUM 在数据返回之前就被分配，因此有一个重要特性：**ROWNUM 不能直接大于某个数进行范围查询**。

```sql
-- ROWNUM 基础用法
SELECT ROWNUM, employee_id, first_name, salary
FROM employees
WHERE ROWNUM <= 10;

-- 错误用法：ROWNUM > 5 永远返回空
SELECT * FROM employees WHERE ROWNUM > 5;  -- 结果为空！
```

为什么 `ROWNUM > 5` 永远为空？因为 ROWNUM 是依次分配的，第一行分配 ROWNUM=1，如果不满足条件就跳过；然后继续第二行……既然第一行 ROWNUM=1 都不满足 `> 5`，后面的行更不可能满足。

### 正确的 ROWNUM 分页方法

ROWNUM 分页必须使用三层嵌套查询：

```sql
-- 第 1 页（每页 20 条）
SELECT * FROM (
    SELECT a.*, ROWNUM rn FROM (
        SELECT employee_id, first_name, last_name, salary, hire_date
        FROM employees
        ORDER BY hire_date DESC
    ) a WHERE ROWNUM <= 20
) WHERE rn > 0;

-- 第 2 页
SELECT * FROM (
    SELECT a.*, ROWNUM rn FROM (
        SELECT employee_id, first_name, last_name, salary, hire_date
        FROM employees
        ORDER BY hire_date DESC
    ) a WHERE ROWNUM <= 40
) WHERE rn > 20;

-- 第 n 页（通用公式）
-- 第 page 页，每页 size 条
-- start = (page - 1) * size + 1
-- end = page * size
SELECT * FROM (
    SELECT a.*, ROWNUM rn FROM (
        SELECT employee_id, first_name, last_name, salary, hire_date
        FROM employees
        ORDER BY hire_date DESC
    ) a WHERE ROWNUM <= :page * :size
) WHERE rn > (:page - 1) * :size;
```

三层嵌套的原因：

- **内层查询**：执行排序，确保数据有序
- **中间层**：添加 ROWNUM，并限制 `ROWNUM <= end`，筛选前 N 条
- **外层**：再从中间结果中筛选 `rn > start`，得到指定范围

### ROWNUM 分页的效率问题

ROWNUM 分页存在一个严重的效率问题：**跳页成本相同**。第 1 页和第 10000 页的执行时间几乎相同，因为每次都需要扫描并排序全部数据。

```sql
-- 即使只取第 10 页，仍然需要扫描全部数据
SELECT * FROM (
    SELECT a.*, ROWNUM rn FROM (
        SELECT * FROM big_table ORDER BY create_time DESC
    ) a WHERE ROWNUM <= 200
) WHERE rn > 190;
```

这个查询需要扫描 big_table 的全部数据（假设有 1000 万行），即使最终只返回 10 条。这是 ROWNUM 分页的固有缺陷。

## ROW_NUMBER：分析函数的分页方案

ROW_NUMBER 是 Oracle 的分析函数，为每一行分配一个连续的唯一序号。相比 ROWNUM，ROW_NUMBER 的优势在于：**排序是查询计划的一部分，优化器可以更智能地处理**。

```sql
-- ROW_NUMBER 基础用法
SELECT employee_id, first_name, salary,
       ROW_NUMBER() OVER (ORDER BY hire_date DESC) AS rn
FROM employees;

-- 使用 ROW_NUMBER 分页
SELECT * FROM (
    SELECT employee_id, first_name, last_name, salary, hire_date,
           ROW_NUMBER() OVER (ORDER BY hire_date DESC) AS rn
    FROM employees
) WHERE rn BETWEEN 21 AND 40;  -- 第 2 页

-- 按部门分组内排序
SELECT * FROM (
    SELECT employee_id, department_id, first_name, salary,
           ROW_NUMBER() OVER (PARTITION BY department_id ORDER BY salary DESC) AS rn
    FROM employees
) WHERE rn <= 3;  -- 每个部门薪资前三
```

### ROW_NUMBER vs ROWNUM 的区别

| 特性 | ROWNUM | ROW_NUMBER |
|---|---|---|
| 类型 | 伪列（在返回前分配） | 分析函数（显式调用） |
| 排序稳定性 | 外层 ORDER BY 无法影响内层 ROWNUM | 内层 ORDER BY 决定序号 |
| 唯一性 | 可能重复 | 唯一（配合 ORDER BY） |
| 性能 | 跳页成本高 | 取决于执行计划 |
| OVER 子句 | 不支持 | 支持 PARTITION BY |

### ROW_NUMBER 的高效跳页

虽然 ROW_NUMBER 本身也不能完全避免全表扫描，但配合唯一列可以优化：

```sql
-- 高效跳页：利用主键分页
SELECT * FROM (
    SELECT employee_id, first_name, salary, hire_date,
           ROW_NUMBER() OVER (ORDER BY hire_date DESC, employee_id) AS rn
    FROM employees
) WHERE rn BETWEEN 10001 AND 10020;
```

即使跳到第 500 页，Oracle 仍可能使用索引覆盖扫描，避免回表。

## Oracle 12c+ 的 OFFSET/FETCH

Oracle 12c 引入了标准 SQL 的 OFFSET/FETCH 语法，使分页更加简洁直观：

```sql
-- Oracle 12c+ 分页语法
SELECT employee_id, first_name, last_name, salary, hire_date
FROM employees
ORDER BY hire_date DESC
OFFSET 20 ROWS FETCH NEXT 20 ROWS ONLY;

-- FETCH FIRST n ROWS ONLY（Oracle 18c+）
SELECT * FROM employees
ORDER BY hire_date DESC
FETCH FIRST 10 ROWS ONLY;

-- WITH TIES：返回并列的最后几行
SELECT department_id, first_name, salary
FROM employees
ORDER BY salary DESC
FETCH FIRST 5 ROWS WITH TIES;

-- 百分比分页
SELECT employee_id, salary
FROM employees
ORDER BY salary DESC
FETCH FIRST 10 PERCENT ROWS ONLY;
```

### OFFSET/FETCH 参数

```sql
-- 偏移量（OFFSET）
OFFSET 0 ROWS       -- 从第一行开始（默认）
OFFSET 10 ROWS      -- 跳过前 10 行
OFFSET 0 ROWS ONLY -- 替换为 FETCH

-- 行数（FETCH）
FETCH FIRST n ROWS ONLY         -- 取 n 行
FETCH FIRST n ROWS WITH TIES    -- 取 n 行，含并列
FETCH FIRST n PERCENT ROWS ONLY -- 取 n% 行

-- 组合
OFFSET 0 ROWS FETCH FIRST 10 ROWS ONLY      -- 前 10 行
OFFSET 10 ROWS FETCH NEXT 10 ROWS ONLY      -- 第 11-20 行
OFFSET 100 ROWS FETCH FIRST 10 ROWS ONLY    -- 第 101-110 行
```

## 常见面试问题

**问：为什么 `WHERE ROWNUM > 5` 永远返回空？**

ROWNUM 在数据返回之前就被分配，ROWNUM=1 的行如果不满足条件就被跳过。既然第一行 ROWNUM=1 都不满足 `> 5`，后面的行更不可能满足。因此必须使用嵌套查询：先筛选 `ROWNUM <= end`，再在外层筛选 `rn > start`。

**问：ROWNUM 和 ROW_NUMBER 的区别是什么？**

ROWNUM 是伪列，在结果返回前分配；ROW_NUMBER 是分析函数，在查询执行时计算。ROWNUM 不能保证排序稳定性；ROW_NUMBER 配合 OVER (ORDER BY ...) 确保序号与排序一致。ROWNUM 分页跳页成本相同；ROW_NUMBER 可以利用索引优化。

**问：Oracle 12c 之前如何实现高效分页？**

使用 ROW_NUMBER 替代 ROWNUM；ORDER BY 中加入唯一列（如主键）；利用 ROWID 分页避免回表；在排序列上创建索引，使用索引范围扫描。

## 总结

Oracle 提供了从 ROWNUM 到 OFFSET/FETCH 的多种分页方式。ROWNUM 是历史遗留方案，需要三层嵌套但存在跳页效率问题；ROW_NUMBER 作为分析函数，配合 PARTITION BY 可以实现更灵活的分组排序；Oracle 12c+ 的 OFFSET/FETCH 则是最现代、最简洁的语法。实际选择时，应根据 Oracle 版本和业务需求综合考虑。
