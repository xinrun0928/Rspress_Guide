# Oracle 并行查询与 DML：并行度（DOP）深度解析

一条大查询跑了 1 小时，业务团队在催。

你查了下执行计划，发现 Oracle 只用了 1 个进程处理 100GB 的数据。但你的服务器有 32 个 CPU 核心，理论上可以并行处理，速度提升几十倍。

**并行执行**，就是 Oracle 给你的答案。

## 并行执行概述

### 为什么需要并行

```
单进程执行：
┌─────────────────────────────────────────┐
│  100GB 数据                             │
│       │                                │
│       ▼                                │
│  ┌─────────────┐                       │
│  │  Process 1  │                       │
│  │  100GB ▶   │  耗时：60 分钟         │
│  └─────────────┘                       │
└─────────────────────────────────────────┘

并行执行（32 并行度）：
┌─────────────────────────────────────────┐
│  100GB 数据                             │
│       │                                │
│       ▼                                │
│  ┌────┴────┴────┴────┴────┐            │
│  │      Parallel Query      │            │
│  │         Server          │            │
│  └───┬───┬───┬───┬───┬───┘            │
│      ▼   ▼   ▼   ▼   ▼                │
│    P1   P2   P3  ...   P32              │
│    3GB  3GB  3GB      3GB              │
│       ▼   ▼   ▼   ▼   ▼                │
│  ┌────────────────────────────────┐    │
│  │      Gather                     │    │
│  │  合并结果                       │    │
│  └────────────────────────────────┘    │
│       │                                │
│       ▼                                │
│  耗时：约 2 分钟（理想情况）            │
└─────────────────────────────────────────┘
```

### 并行执行架构

```
┌─────────────────────────────────────────────────────────────────┐
│                    Parallel Execution 架构                       │
│                                                                 │
│    ┌─────────────────────────────────────────────────────────┐  │
│    │                   Query Coordinator (QC)                │  │
│    │  负责：生成执行计划、分配任务、收集结果                    │  │
│    └────────────────────────────┬────────────────────────────┘  │
│                                 │                                 │
│    ┌────────────────────────────┼────────────────────────────┐  │
│    │                   Parallel Server Pool                   │  │
│    │  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐   │  │
│    │  │  PX Set 1 │ │  PX Set 2 │ │  PX Set 3 │ │  PX Set N │   │  │
│    │  │  Producer │ │ Producer  │ │ Producer  │ │ Producer  │   │  │
│    │  └──────────┘ └──────────┘ └──────────┘ └──────────┘   │  │
│    └────────────────────────────┼────────────────────────────┘  │
│                                 │                                 │
│    ┌────────────────────────────┼────────────────────────────┐  │
│    │                   Parallel Server Pool                   │  │
│    │  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐   │  │
│    │  │  PX Set 1 │ │  PX Set 2 │ │  PX Set 3 │ │  PX Set N │   │  │
│    │  │  Consumer │ │ Consumer  │ │ Consumer  │ │ Consumer  │   │  │
│    │  └──────────┘ └──────────┘ └──────────┘ └──────────┘   │  │
│    └─────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
```

**核心组件**：

| 组件 | 说明 |
|---|---|
| Query Coordinator (QC) | 协调器，主进程负责任务分发和结果汇总 |
| Parallel Server (PX) | 并行服务器，工作进程执行具体任务 |
| Producer | 生产者，负责读取数据 |
| Consumer | 消费者，负责处理生产者产出的数据 |
| PX Dequeue | 进程间通信队列 |

## 并行度（DOP）

### 什么是 DOP

DOP（Degree of Parallelism）是并行执行的核心参数，表示使用的并行进程数量：

```sql
-- 查看当前会话的 DOP
SELECT * FROM V$SESSION WHERE SID = SYS_CONTEXT('USERENV', 'SID');
-- 查看 PARALLEL_DEGREE_POLICY 和 PARALLEL_SERVERS_TARGET
```

### DOP 决定方式

| 方式 | 说明 | 优先级 |
|---|---|---|
| HINT 指定 | `/*+ PARALLEL(4) */` | 最高 |
| 对象级别 | `ALTER TABLE orders PARALLEL 4` | 其次 |
| 会话级别 | `ALTER SESSION FORCE PARALLEL QUERY` | 其次 |
| 自动 DOP | `PARALLEL_DEGREE_POLICY = AUTO` | 自动计算 |

### DOP 语法

```sql
-- 1. PARALLEL Hint（最高优先级）
SELECT /*+ PARALLEL(orders, 8) */ * FROM orders WHERE amount > 1000;

-- 2. 表级别设置
CREATE TABLE orders (
    order_id NUMBER,
    amount NUMBER(10,2)
)
PARALLEL 8;

-- 修改表级别 DOP
ALTER TABLE orders PARALLEL 8;

-- 3. 索引级别设置
CREATE INDEX idx_orders_date ON orders(order_date) PARALLEL 4;

-- 4. 会话级别强制并行
ALTER SESSION FORCE PARALLEL QUERY PARALLEL 8;
ALTER SESSION FORCE PARALLEL DML PARALLEL 8;
ALTER SESSION FORCE PARALLEL DDL PARALLEL 8;

-- 5. 取消并行
SELECT /*+ NO_PARALLEL */ * FROM orders;
ALTER TABLE orders NOPARALLEL;
```

## 并行查询（Parallel Query）

### 并行全表扫描

```sql
-- 使用 PARALLEL Hint
SELECT /*+ PARALLEL(orders, 8) */ *
FROM orders
WHERE order_date >= TRUNC(SYSDATE, 'MM');

-- 查看执行计划
EXPLAIN PLAN FOR
SELECT /*+ PARALLEL(orders, 8) */ *
FROM orders
WHERE order_date >= TRUNC(SYSDATE, 'MM');

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

-- 执行计划示例
/*
Plan hash value: 123456789

---------------------------------------------------------------------------
| Id  | Operation               | Name     | Rows |    TQ |IN-OUT| PQ Distrib |
---------------------------------------------------------------------------
|   0 | SELECT STATEMENT        |          | 1000K|       |      |            |
|   1 |  PX COORDINATOR        |          |      |       |      |            |
|   2 |   PX SEND QC (RANDOM)  | :TQ10001 | 1000K|  Q1,01| P->S | QC (RAND)  |
|   3 |    PX BLOCK ITERATOR   |          | 1000K|  Q1,01| PCWC |            |
|   4 |     TABLE ACCESS FULL  | ORDERS   | 1000K|  Q1,01| PCWP |            |
---------------------------------------------------------------------------
*/
```

### 并行索引扫描

```sql
-- 并行索引范围扫描
SELECT /*+ PARALLEL_INDEX(orders, idx_orders_date, 8) */ *
FROM orders
WHERE order_date BETWEEN TO_DATE('2024-01-01') AND TO_DATE('2024-12-31');

-- 并行索引快速全扫描
SELECT /*+ PARALLEL_INDEX(orders, idx_orders_id, 8) */ *
FROM orders
WHERE order_id IS NOT NULL;
```

### 并行连接

```sql
-- 并行 Hash Join
SELECT /*+ PARALLEL(orders, 8) PARALLEL(customers, 8) */ *
FROM orders o
JOIN customers c ON o.customer_id = c.id
WHERE o.order_date >= TRUNC(SYSDATE, 'MM');

-- 并行 Nested Loop Join
SELECT /*+ PARALLEL(orders, 8) */
    o.order_id,
    c.customer_name
FROM orders o
JOIN (SELECT /*+ PARALLEL(customers, 8) */ * FROM customers WHERE region = '北京') c
ON o.customer_id = c.id
WHERE o.amount > 1000;

-- 并行 Merge Join
SELECT /*+ USE_MERGE(o,c) PARALLEL(o,4) PARALLEL(c,4) */ *
FROM orders o
JOIN customers c ON o.customer_id = c.id;
```

## 并行 DML（Parallel DML）

### 启用并行 DML

```sql
-- 默认情况下，并行 DML 是禁用的
-- 需要显式启用
ALTER SESSION ENABLE PARALLEL DML;

-- 禁用并行 DML
ALTER SESSION DISABLE PARALLEL DML;

-- 查看当前会话的并行 DML 状态
SELECT FORCED_DEGREE FROM V$SESSION 
WHERE SID = SYS_CONTEXT('USERENV', 'SID');
```

### 并行 INSERT

```sql
-- 启用并行 DML
ALTER SESSION ENABLE PARALLEL DML;

-- 并行 INSERT（直接路径）
INSERT /*+ APPEND PARALLEL(orders, 8) */ INTO orders
SELECT /*+ PARALLEL(order_history, 8) */ *
FROM order_history
WHERE order_date >= TRUNC(SYSDATE, 'MM');

-- 并行 INSERT（常规）
INSERT /*+ PARALLEL(orders, 8) */ INTO orders
SELECT * FROM orders_source WHERE status = 'pending';
```

### 并行 UPDATE/DELETE

```sql
-- 并行 UPDATE
UPDATE /*+ PARALLEL(orders, 8) */ orders
SET amount = amount * 0.9
WHERE order_date < ADD_MONTHS(TRUNC(SYSDATE), -12);

-- 并行 DELETE
DELETE /*+ PARALLEL(orders, 8) */ orders
WHERE order_date < ADD_MONTHS(TRUNC(SYSDATE), -24)
  AND status = 'completed';

-- 并行 MERGE
MERGE /*+ PARALLEL(orders, 8) */ INTO orders o
USING order_updates ou
ON (o.order_id = ou.order_id)
WHEN MATCHED THEN
    UPDATE SET o.amount = ou.new_amount,
               o.status = ou.new_status;
```

### 并行 DDL

```sql
-- 并行 CREATE TABLE AS SELECT
CREATE /*+ PARALLEL(reports, 16) */ TABLE reports
PARALLEL 16
AS
SELECT /*+ PARALLEL(sales, 16) */ 
    TRUNC(sale_date, 'MM') as month,
    SUM(amount) as total_amount,
    COUNT(*) as order_count
FROM sales
GROUP BY TRUNC(sale_date, 'MM');

-- 并行 CREATE INDEX
CREATE /*+ PARALLEL */ INDEX idx_orders_date 
ON orders(order_date) PARALLEL 8;

-- 并行 REBUILD INDEX
ALTER INDEX idx_orders_date REBUILD PARALLEL 8;

-- 并行 MOVE TABLE
ALTER TABLE orders MOVE PARALLEL 8;
```

## PX 调度（Parallel Execution Scheduling）

### PX 执行模式

| 模式 | 说明 | 适用场景 |
|---|---|---|
| Auto DOP | Oracle 自动决定 DOP | 通用场景 |
| Manual DOP | 用户指定 DOP | 明确知道最优 DOP |
| Degree from Default | 使用表/索引的 PARALLEL 设置 | 批量操作 |

### 并行执行流程

```sql
-- Step 1: QC 创建并行服务器池
-- Step 2: 将数据按分区/块分割
-- Step 3: 分配任务到各 PX 进程
-- Step 4: PX 进程并行执行
-- Step 5: 汇总结果到 QC

-- 查看并行执行统计
SELECT sid, serial#, degree, req_degree
FROM V$PX_SESSION
ORDER BY sid;

-- 查看当前并行操作
SELECT qcsid, qcserial#, num_parallel
FROM V$PX_SESSION
GROUP BY qcsid, qcserial#, num_parallel;
```

## 参数配置

### 关键参数

```sql
-- 初始化参数
-- 并行服务器最大数量
SHOW PARAMETER parallel_max_servers
-- 默认值：CPU_COUNT * 100 (11g), CPU_COUNT * PARALLEL_THREADS_PER_CPU * 25 (12c+)

-- 每个查询的最小并行服务器
SHOW PARAMETER parallel_min_servers
-- 默认值：0

-- 自动 DOP 启用
SHOW PARAMETER parallel_degree_policy
-- MANUAL: 手动（默认）
-- AUTO: 自动
-- LIMITED: 自动但有限制

-- 自动 DOP 阈值
SHOW PARAMETER parallel_degree_limit
-- 默认值：CPU_COUNT * 10

-- 并行服务器目标
SHOW PARAMETER parallel_servers_target
-- 默认值：CPU_COUNT * PARALLEL_THREADS_PER_CPU * 10
```

### 推荐配置

```ini
# init.ora 或 SPFILE 参数

# 启用自动 DOP（12c+）
parallel_degree_policy = AUTO
parallel_degree_limit = 16
parallel_servers_target = 4096

# 或手动配置（11g）
parallel_max_servers = 64
parallel_min_servers = 0
parallel_adaptive_multi_user = TRUE
```

### 会话级配置

```sql
-- 会话级覆盖参数
ALTER SESSION SET parallel_degree_policy = AUTO;
ALTER SESSION SET parallel_degree_limit = 8;
ALTER SESSION SET parallel_servers_target = 2048;

-- 强制所有查询并行
ALTER SESSION FORCE PARALLEL QUERY PARALLEL 4;
```

## 性能优化

### 1. 选择合适的 DOP

```sql
-- 经验法则：DOP = CPU 核心数 / 活跃会话数
-- 或 DOP = CPU 核心数的 50%

-- 过高的 DOP 会导致资源争用
-- 过低的 DOP 无法充分利用硬件

-- 查看 CPU 信息
SELECT * FROM V$CPU;
SELECT * FROM V$OSSTAT WHERE STAT_NAME = 'NUM_CPUS';
```

### 2. 分区表并行

```sql
-- 分区表天然支持并行
-- 每个分区可以独立并行处理
CREATE TABLE orders (
    order_id NUMBER,
    order_date DATE,
    amount NUMBER(10,2)
)
PARTITION BY RANGE (order_date) (
    PARTITION p2024_q1 VALUES LESS THAN (TO_DATE('2024-04-01')),
    PARTITION p2024_q2 VALUES LESS THAN (TO_DATE('2024-07-01')),
    PARTITION p2024_q3 VALUES LESS THAN (TO_DATE('2024-10-01')),
    PARTITION p2024_q4 VALUES LESS THAN (TO_DATE('2025-01-01'))
);

-- 查询会自动并行（如果 DOP 设置得当）
SELECT /*+ PARALLEL(8) */ * FROM orders WHERE order_date >= SYSDATE - 30;
```

### 3. 监控并行执行

```sql
-- 查看当前并行操作
SELECT 
    q.sid, q.serial#,
    q.status, q.state,
    q.degree, q.request_degree,
    p.num_active
FROM V$PX_SESSION p, V$SESSION q
WHERE p.sid = q.sid
  AND p.serial# = q.serial#;

-- 查看历史并行统计
SELECT 
    sql_id,
    sql_text,
    px_servers_executions,
    elapsed_time,
    buffer_gets,
    disk_reads
FROM V$SQL
WHERE px_servers_executions > 0
ORDER BY px_servers_executions DESC;
```

## 常见问题

### 问题一：并行不生效

```sql
-- 检查原因
-- 1. 查看 SQL 是否被串行化
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY_AWR('sql_id'));

-- 2. 检查表/索引的 PARALLEL 属性
SELECT table_name, degree, instances FROM user_tables 
WHERE table_name = 'ORDERS';

-- 3. 检查并行资源是否足够
SELECT * FROM V$RESOURCE_LIMIT 
WHERE resource_name IN ('parallel_max_servers', 'processes');

-- 4. 检查隐藏参数
SELECT ksppinm, ksppstvl, ksppdesc
FROM v$spparameter
WHERE ksppinm LIKE '%parallel%';
```

### 问题二：并行导致 OOM

```sql
-- 减少 DOP
SELECT /*+ PARALLEL(4) */ * FROM orders;

-- 调整 workarea_size_policy
ALTER SESSION SET workarea_size_policy = MANUAL;
ALTER SESSION SET sort_area_size = 536870912;  -- 512MB

-- 增加并行服务器目标
ALTER SYSTEM SET parallel_servers_target = 8192;
```

### 问题三：并行资源争用

```sql
-- 限制资源消费者组的并行度
BEGIN
    DBMS_RESOURCE_MANAGER.CREATE_PENDING_AREA();
    
    DBMS_RESOURCE_MANAGER.CREATE_CONSUMER_GROUP(
        consumer_group => 'OLTP_GROUP',
        comment => 'OLTP applications'
    );
    
    DBMS_RESOURCE_MANAGER.CREATE_PLAN_DIRECTIVE(
        plan => 'DAYTIME_PLAN',
        group_or_subplan => 'OLTP_GROUP',
        comment => 'OLTP directive',
        parallel_degree_limit_p1 => 4
    );
    
    DBMS_RESOURCE_MANAGER.SUBMIT_PENDING_AREA();
END;
/
```

## 总结

Oracle 并行执行是提升查询性能的关键技术：

- **PARALLEL Hint**：最高优先级的并行提示
- **DOP**：并行度，决定使用的进程数
- **PX 进程**：并行服务器，负责具体任务执行
- **并行 DML**：需要显式启用 `ENABLE PARALLEL DML`
- **参数调优**：`parallel_degree_policy` 和 `parallel_max_servers` 是关键

合理配置并行度，让 Oracle 的多核能力得到充分发挥——**从 1 小时到 2 分钟**，就是并行的威力。
