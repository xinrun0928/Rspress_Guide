# Oracle RAC 集群架构：Cache Fusion 与 GCS 原理

单节点数据库遇到瓶颈怎么办？

CPU 打满、内存耗尽、连接数爆表……垂直扩展（Scale Up）总有极限。但水平扩展（Scale Out）——多个节点共享同一个数据库——听起来很美好，实现起来却困难重重。

**Oracle Real Application Clusters（RAC）** 让多个服务器节点共享同一个数据库，同时保证数据一致性和高性能。秘诀就是 **Cache Fusion** 和 **Global Cache Service（GCS）**。

**核心特点**：

| 特点 | 说明 |
|---|---|
| 多节点 | 多个服务器节点组成集群 |

| 负载均衡 | 多个节点分担负载 |

### RAC 与 Data Guard 的区别

| 对比项 | RAC | Data Guard |
|---|---|---|
| 目的 | 高可用 + 负载均衡 | 灾备 + 数据保护 |
| 数据存储 | 共享存储 | 独立存储 |
| 节点关系 | 对等关系 | 主从关系 |
| 故障转移 | 节点级别 | 数据库级别 |
| 数据一致性 | 通过 Cache Fusion | 通过 Redo Apply |

## Cache Fusion：核心机制

### 为什么需要 Cache Fusion

传统集群的问题：**数据缓存不一致**。

```
传统方案（磁盘融合）：
Node 1 想修改数据 X -> 检查磁盘锁 -> 锁定磁盘 -> 修改 -> 释放锁
Node 2 想读取数据 X -> 检查磁盘锁 -> 等待解锁 -> 读取

问题：大量磁盘 I/O，性能极差
```

**Cache Fusion 的解决方案**：用高速私有网络替代磁盘 I/O，实现节点间的内存数据共享。

```
Cache Fusion 方案：
Node 1 想修改数据 X -> 检查全局缓存（通过 GCS）-> 发现 X 在 Node 2 内存
-> Node 2 通过私有网络将数据发送给 Node 1 -> Node 1 修改 -> 更新全局缓存状态

优势：内存到内存，速度极快
```

### Cache Fusion 工作原理

```sql
-- 假设场景：
-- Node 1 想修改数据块 BLOCK#123
-- 当前 BLOCK#123 在 Node 2 的 SGA 中

-- 1. Node 1 请求 BLOCK#123
/*
 * Global Cache Service (GCS) 协调
 */

-- 2. GCS 检查 BLOCK#123 的状态
/*
 * 当前状态：
 * - Current Image: Node 2 (正在被修改)
 * - Shared Image: 可能存在于多个节点
 */

-- 3. Node 2 将 BLOCK#123 的当前镜像通过私有网络发送给 Node 1
/*
 * Image Transfer: Node 2 -> Private Network -> Node 1
 * 速度：GB 级私有网络，毫秒级延迟
 */

-- 4. Node 1 获得 BLOCK#123 的排他锁（X lock）
-- 5. Node 2 的 BLOCK#123 标记为 Transfer

-- 后续访问：
-- Node 2 再想访问 BLOCK#123？需要向 Node 1 请求
```

### 缓冲区状态

Cache Fusion 中的数据块有以下状态：

| 状态 | 描述 | 权限 |
|---|---|---|
| CR (Consistent Read) | 一致读镜像 | 多个节点可同时持有 |
| CURRENT | 当前镜像 | 只有一个节点持有 |
| SCUR (Shared Current) | 共享当前镜像 | 多个节点持有，只能读 |
| XCUR (Exclusive Current) | 排他当前镜像 | 只有一个节点持有，可写 |

### 请求与转让流程

```sql
-- Case 1: Node 1 想读 BLOCK#123（从未访问过）
/*
 * 1. Node 1 向 GCS 请求 CR 镜像
 * 2. GCS 从磁盘读取 BLOCK#123
 * 3. GCS 发送 CR 镜像到 Node 1
 * 4. Node 1 以 CR 状态缓存
 */

-- Case 2: Node 1 想修改 BLOCK#123（Node 2 持有 CURRENT 镜像）
/*
 * 1. Node 1 向 GCS 请求 XCUR 锁
 * 2. GCS 通知 Node 2 释放 BLOCK#123
 * 3. Node 2 将 dirty buffer 写回磁盘（DBWn）
 * 4. Node 2 通过私有网络发送 BLOCK#123 到 Node 1
 * 5. Node 1 获得 XCUR 锁，开始修改
 */

-- Case 3: Node 1 读取 BLOCK#123（Node 2 持有 CURRENT 镜像）
/*
 * 1. Node 1 向 GCS 请求 CR 镜像
 * 2. GCS 通知 Node 2 构造 CR 镜像
 * 3. Node 2 将 CR 镜像通过私有网络发送到 Node 1
 * 4. Node 1 缓存 CR 镜像
 */
```

## GCS：Global Cache Service

### GCS 架构

```
┌─────────────────────────────────────────────────────────────────┐
│                    Global Cache Service (GCS)                    │
│                                                                 │
│    ┌─────────────────────────────────────────────────────────┐  │
│    │                    LMON Process                          │  │
│    │         (Global Enqueue Service Monitor)                 │  │
│    │  - 管理全局锁                                              │  │
│    │  - 检测节点故障                                            │  │
│    │  - 协调锁转移                                              │  │
│    └─────────────────────────┬───────────────────────────────┘  │
│                              │                                   │
│    ┌─────────────────────────┼───────────────────────────────┐  │
│    │                    GCS 请求处理                            │  │
│    │                                                         │  │
│    │  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐     │  │
│    │  │ Buffer #1  │  │ Buffer #2  │  │ Buffer #N  │     │  │
│    │  │ master: N1 │  │ master: N2 │  │ master: N3 │     │  │
│    │  └─────────────┘  └─────────────┘  └─────────────┘     │  │
│    │                                                         │  │
│    └─────────────────────────────────────────────────────────┘  │
│                                                                 │
│    ┌───────────────┐  ┌───────────────┐  ┌───────────────┐    │
│    │    Node 1    │  │    Node 2    │  │    Node 3    │    │
│    │  GCS Master  │  │              │  │              │    │
│    │  of Buffers  │  │              │  │              │    │
│    └───────────────┘  └───────────────┘  └───────────────┘    │
└─────────────────────────────────────────────────────────────────┘
```

### GCS 组件

| 组件 | 说明 |
|---|---|
| GCS Master | 每个 buffer 有固定的 master 节点 |
| GCS Request | 节点向 GCS 发起的数据请求 |
| GCS Resource | 全局管理的数据块资源 |
| LMON | 全局队列服务监控进程 |
| LMSn | 全局缓存服务进程（多个）|

### 全局队列服务（GES）

GES 管理**全局锁**，与数据块无关：

```sql
-- 全局锁类型
-- 1. Library Cache Lock：保护共享池中的 SQL 和对象定义
-- 2. Row Cache Lock：保护数据字典缓存
-- 3. Global Enqueue：保护全局资源访问

-- 查看全局锁统计

-- 查看全局锁等待

```

## RAC 性能优化

### 热点数据问题

```sql
-- 问题：多个节点频繁访问同一数据块（热点块）
-- 导致：大量跨节点传输，性能下降

-- 识别热点块
SELECT * FROM (
    SELECT P1 "File", P2 "Block", COUNT(*) "Waits"
    FROM V$WAITSTAT
    WHERE CLASS LIKE 'data block%'
    GROUP BY P1, P2
    ORDER BY COUNT(*) DESC
) WHERE ROWNUM <= 10;

-- 解决方案：
-- 1. 应用层分散热点（如按 ID 分区）
-- 2. 使用更高并行的应用服务器
-- 3. 调整 GC 相关参数
```

### 私有网络优化

```sql
-- 查看网络延迟统计
SELECT * FROM V$DLM_MISC
WHERE NAME IN ('lock send time', 'lock convert time');

-- 调整私有网络缓冲区
ALTER SYSTEM SET "_gcs_resources" = 5000 SCOPE = SPFILE;
ALTER SYSTEM SET "_gcs_servers" = 20 SCOPE = SPFILE;
```

## 总结

Oracle RAC 通过 **Cache Fusion** 和 **GCS** 实现了真正的数据库水平扩展：

- **Cache Fusion**：用私有网络替代磁盘 I/O，实现节点间内存数据共享
- **GCS**：协调全局缓存，确保数据一致性
- **LMSn**：GCS 的工作进程，处理数据块请求和传输
- **高可用**：节点故障自动转移，业务不中断
- **负载均衡**：多个节点分担访问压力

理解 Cache Fusion 的原理，你就能理解 RAC 如何在保证数据一致性的同时实现高性能水平扩展。
