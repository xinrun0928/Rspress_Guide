# Oracle RAC 高可用：VIP、SCAN IP 与负载均衡

节点宕了，业务却没中断——这就是 RAC 高可用的魅力。

传统单机数据库，服务器宕机意味着数据库不可用，业务中断。但 Oracle RAC 中，即使一个节点宕机，运行在该节点上的连接会自动转移到健康节点，用户甚至感觉不到故障发生。

这一切的背后，是 **VIP（Virtual IP）**、**SCAN IP（Single Client Access Name）** 和 **Service** 机制在协同工作。

## VIP 机制

### VIP 原理

VIP（Virtual IP）是 RAC 为每个节点分配的虚拟 IP 地址：

```
物理网络：
┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│  Node 1: 192.168.1.11                                          │
│  VIP 1:  192.168.1.21  (Node 1 宕机后自动转移到 Node 2)          │
│                                                                 │
│  Node 2: 192.168.1.12                                          │
│  VIP 2:  192.168.1.22  (Node 2 宕机后自动转移到 Node 1)          │
│                                                                 │
│  Node 3: 192.168.1.13                                          │
│  VIP 3:  192.168.1.23                                          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### VIP 的工作方式

```sql
-- 正常情况下
/*
 * 客户端连接 192.168.1.21 (VIP 1)
 * 连接到 Node 1 的 Listener
 * 分配到 Node 1 的 Instance 处理
 */

-- Node 1 宕机后
/*
 * VIP 1 自动通过 ARP 重定向到 Node 2
 * 客户端连接 192.168.1.21 仍然可达（由 Node 2 处理）
 * 如果使用 TAF，SELECT 语句自动重试
 */
```

### VIP 优势

| 特性 | 说明 |
|---|---|
| 故障快速感知 | VIP 漂移比 IP 漂移更快（秒级 vs 分钟级）|
| 透明切换 | 客户端连接无需重建 |
| 无需修改客户端 | VIP 地址保持不变 |

## SCAN IP

### SCAN 概述

SCAN（Single Client Access Name）是 Oracle 11g R2 引入的特性：

```bash
# SCAN 配置
# DNS 配置：rac-scan.cluster.local -> 3 个 SCAN VIP
# SCAN VIP 1: 192.168.1.31
# SCAN VIP 2: 192.168.1.32
# SCAN VIP 3: 192.168.1.33

# 客户端连接串（简化）
(DESCRIPTION =
    (CONNECT_DATA = (SERVICE_NAME = racdb.cluster.local))
    (ADDRESS = (PROTOCOL = TCP)(HOST = rac-scan.cluster.local)(PORT = 1521))
)
```

### SCAN 工作原理

```
┌─────────────────────────────────────────────────────────────────┐
│                    SCAN 负载均衡工作原理                          │
│                                                                 │
│  Client                                                         │
│     │                                                          │
│     │ 1. 连接 rac-scan.cluster.local:1521                       │
│     ▼                                                          │
│  DNS Server                                                    │
│     │  返回 3 个 SCAN IP（轮询或随机）                          │
│     ▼                                                          │
│  ┌───┴───┬────────┐                                            │
│  │       │        │                                            │
│  ▼       ▼        ▼                                            │
│ SCAN    SCAN     SCAN                                          │
│ VIP 1   VIP 2    VIP 3                                         │
│  │       │        │                                             │
│  │       │        │                                             │
│  ▼       ▼        ▼                                             │
│ ┌────────┬────────┬────────┐                                    │
│ │ Node 1 │ Node 2 │ Node 3 │                                    │
│ │Listner │Listner │Listner │                                    │
│ └────────┴────────┴────────┘                                    │
│     │        │        │                                         │
│     ▼        ▼        ▼                                         │
│ ┌────────┬────────┬────────┐                                    │
│ │Instance│Instance│Instance│                                    │
│ │   1    │   2    │   3    │                                    │
│ └────────┴────────┴────────┘                                    │
└─────────────────────────────────────────────────────────────────┘
```

### SCAN 优势

| 优势 | 说明 |
|---|---|
| 简化连接串 | 客户端无需知道有多少节点 |
| 动态负载均衡 | SCAN VIP 分布在不同节点 |
| 节点变化透明 | 添加/删除节点，客户端无需修改连接串 |
| 故障自动规避 | 故障节点的 SCAN VIP 自动转移到健康节点 |

```bash

# 启用/禁用 SCAN
$ srvctl disable scan -i 1
$ srvctl enable scan -i 1
```

## Service 机制

### Service 概述

Service 是 RAC 中连接路由的高级抽象：

```sql
-- 查看当前 Service
SELECT name, clb_goal, connection_load_balance, failover_type
FROM V$SERVICES
ORDER BY name;

-- 查看 Service 分布
SELECT service_id, name, pdb, failover_type, failover_method
FROM DBA_SERVICES;
```

### 创建 Service

```sql
-- 使用 SRVCTL 创建 Service
-- $ srvctl add service -d racdb -s order_service -r racdb1,racdb2 -l PRIMARY

-- 使用 SQL 创建 Service（12c+ PDB 中）
BEGIN
    DBMS_SERVICE.CREATE_SERVICE(
        service_name     => 'order_service',
        network_name     => 'order_service.cluster.local',
        failover_type    => 'SELECT',      -- SELECT 语句自动重试
        failover_method  => 'BASIC',       -- 基础故障转移
        failover_retries => 30,
        failover_delay   => 5,
        clb_goal         => 'LONG'         -- 连接负载均衡目标
    );
END;
/

-- 启动 Service
EXEC DBMS_SERVICE.START_SERVICE('order_service');

-- 停止 Service
EXEC DBMS_SERVICE.STOP_SERVICE('order_service');
```

### Service 与 Preferred Instances

```bash
# 配置 Service 的 Preferred Instances
$ srvctl add service -d racdb \
    -s reports_service \
    -r racdb1 \
    -a racdb2

# 说明：
# -r: 首选实例（正常情况下处理连接）
# -a: 可用实例（首选实例故障时接管）
```

## TAF：Transparent Application Failover

### TAF 概述

TAF（Transparent Application Failover）在连接断开时自动重试未完成的查询：

```sql
-- TAF 配置方式一：连接串配置
(DESCRIPTION =
    (ADDRESS = (PROTOCOL = TCP)(HOST = rac-scan.cluster.local)(PORT = 1521))
    (CONNECT_DATA =
        (SERVER = DEDICATED)
        (SERVICE_NAME = racdb.cluster.local)
        (FAILOVER_MODE =
            (TYPE = SELECT)           -- SELECT 重试
            (METHOD = BASIC)          -- 基础模式
            (RETRIES = 20)            -- 重试 20 次
            (DELAY = 15)              -- 每次间隔 15 秒
        )
    )
)

-- TAF 配置方式二：服务级别配置
BEGIN
    DBMS_SERVICE.MODIFY_SERVICE(
        service_name   => 'order_service',
        failover_type  => DBMS_SERVICE.FAILOVER_TYPE_SELECT,
        failover_method=> DBMS_SERVICE.FAILOVER_METHOD_BASIC,
        failover_retries=> 30,
        failover_delay => 5
    );
END;
/
```

### TAF 类型

| 类型 | 说明 | 重试能力 |
|---|---|---|
| NONE | 无故障转移 | - |
| SESSION | 重连会话，重新执行语句 | SELECT 不重试 |
| SELECT | 重试 SELECT 语句 | SELECT 重试 |
| TRANSACTION | 重试事务 | 整个事务重试 |

### TAF 执行流程

```sql
-- TAF 执行过程
/*
 * 1. 客户端连接 Node 1，执行 SELECT 语句
 *
 * 2. Node 1 宕机
 *    - VIP 漂移到 Node 2
 *    - 客户端连接断开
 *
 * 3. TAF 接管
 *    - 尝试重新连接（使用备用实例）
 *    - 重试 SELECT 语句
 *    - 返回结果给客户端
 *
 * 4. 客户端感知：短暂卡顿后继续收到结果
 */
```

### TAF 限制

```sql
-- TAF 无法处理的情况：
-- 1. 正在执行的 DML（INSERT/UPDATE/DELETE）
-- 2. 正在执行的事务
-- 3. PL/SQL 代码块中的 SELECT
-- 4. FETCH 过程中的 SELECT

-- 建议：应用层也需要做好错误处理
```

## 负载均衡

### 客户端负载均衡

```sql
-- 连接串配置
(DESCRIPTION =
    (ADDRESS_LIST =
        (ADDRESS = (PROTOCOL = TCP)(HOST = node1-vip)(PORT = 1521))
        (ADDRESS = (PROTOCOL = TCP)(HOST = node2-vip)(PORT = 1521))
    )
    (CONNECT_DATA =
        (SERVER = DEDICATED)
        (SERVICE_NAME = racdb.cluster.local)
    )
    (LOAD_BALANCE = yes)    -- 启用负载均衡
)
```

### 服务器端负载均衡

```sql
-- 配置 CLB_GOAL
BEGIN
    DBMS_SERVICE.MODIFY_SERVICE(
        service_name => 'order_service',
        clb_goal    => DBMS_SERVICE.CLB_GOAL_SHORT  -- 基于响应时间
    );
END;
/

-- CLB_GOAL 选项：
-- LONG: 基于连接持续时间（适合长连接）
-- SHORT: 基于负载（适合短连接）

-- 查看连接分布
SELECT inst_id, COUNT(*) as connections
FROM gv$session
WHERE type = 'USER'
GROUP BY inst_id;
```

### 连接池负载均衡

```java
// Java UCP (Universal Connection Pool) 配置
PoolDataSource pds = PoolDataSourceFactory.getPoolDataSource();
pds.setConnectionFactoryClassName("oracle.jdbc.pool.OracleDataSource");
pds.setURL("jdbc:oracle:thin:@rac-scan.cluster.local:1521:racdb");
pds.setUser("scott");
pds.setPassword("tiger");
pds.setConnectionPoolName("RAC-Pool");

// 启用 RAC 负载均衡
pds.setONSConfiguration("nodes=rac-scan.cluster.local");

// 连接池大小
pds.setInitialPoolSize(10);
pds.setMaxPoolSize(50);
pds.setMinPoolSize(5);

// 故障转移配置
pds.setONSConfiguration("nodes=node1-vip:6200,node2-vip:6200,node3-vip:6200");
```

## 总结

Oracle RAC 高可用机制确保了业务连续性：

- **VIP**：节点故障时 IP 自动漂移，客户端无感知
- **SCAN IP**：简化连接串，实现动态负载均衡
- **Service**：连接路由的高级抽象，支持细粒度控制
- **TAF**：SELECT 语句透明重试，用户无感知
- **连接池**：应用层配合，实现完整的高可用

这套机制配合 Oracle RAC 的 Cache Fusion，让业务真正做到"永不停机"。
