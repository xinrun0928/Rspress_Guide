# Oracle Data Guard：原理与三种保护模式

数据库宕机意味着什么？

对于银行来说，每分钟可能有几十万笔交易在跑；对于电商来说，每秒钟可能损失上百万的订单。**高可用**不是锦上添花，而是生产系统的生命线。

Oracle **Data Guard** 是 Oracle 数据库的高可用和灾备解决方案，通过维护一个或多个同步备用数据库，确保在主库故障时能够快速切换，保证业务连续性。

## Data Guard 架构概述

### 基本架构

```
┌─────────────────────────────────────────────────────────────────────┐
│                        Data Guard 架构                              │
│                                                                     │
│    ┌─────────────────┐                    ┌─────────────────┐    │
│    │     Primary     │───Redo Transport───▶│  Standby 1       │    │
│    │     Database     │                    │  (实时同步)       │    │
│    │  [主库/写操作]   │                    │  [备库/只读]     │    │
│    └─────────────────┘                    └─────────────────┘    │
│           │                                        │              │
│           │ Redo Transport                         │ Apply        │
│           │ (可选加密)                              │ (可选实时)   │
│           ▼                                        ▼              │
│    ┌─────────────────┐                    ┌─────────────────┐    │
│    │  Physical Standby│◀──Redo Transport───│  Logical        │    │
│    │  (物理备库)     │                    │  Standby        │    │
│    │  (块级复制)     │                    │  (逻辑备库)     │    │
│    └─────────────────┘                    └─────────────────┘    │
│                                                                     │
│    ┌─────────────────┐                                            │
│    │     Broker     │───▶ 统一管理、故障转移                       │
│    │  (Data Guard   │                                            │
│    │   Broker)      │                                            │
│    └─────────────────┘                                            │
└─────────────────────────────────────────────────────────────────────┘
```

### 核心组件

| 组件 | 说明 |
|---|---|
| Primary Database | 主库，处理所有生产负载 |
| Standby Database | 备库，接收主库 redo 并应用 |
| Redo Transport | 主库向备库传输 redo 的机制 |
| Redo Apply | 物理备库应用 redo 的方式 |
| SQL Apply | 逻辑备库应用 redo 的方式 |
| Data Guard Broker | 统一管理工具 |

## 备库类型

### Physical Standby（物理备库）

```sql
-- 物理备库特点：
-- 1. 与主库块级完全一致（block-for-block）
-- 2. 以只读方式打开，可用于查询
-- 3. 支持实时应用（Real-Time Apply）

-- 物理备库结构
/*
 * 主库产生的 redo 日志
 *       │
 *       ▼
 * LGWR 进程写入 redo
 *       │
 *       ▼
 * Redo Transport 传输到备库
 *       │
 *       ▼
 * RFS 进程接收 redo
 *       │
 *       ▼
 * MRP 进程应用 redo（磁盘 I/O）
 */
```

### Logical Standby（逻辑备库）

```sql
-- 逻辑备库特点：
-- 1. 数据结构与主库可能不同
-- 2. 以读写方式打开，可用于报表
-- 3. 支持部分表同步（跳过某些对象）
-- 4. 支持数据转换

-- 逻辑备库结构
/*
 * 主库 redo 日志
 *       │
 *       ▼
 * LogMiner 挖掘 SQL
 *       │
 *       ▼
 * LSP 进程执行 SQL
 */
```

### Snapshot Standby（快照备库）

```sql
-- 快照备库特点：
-- 1. 暂时脱离 Data Guard 保护
-- 2. 可读写，用于测试/开发
-- 3. 恢复时自动应用积累的 redo

-- 转换为快照备库
ALTER DATABASE CONVERT TO SNAPSHOT STANDBY;

-- 恢复为物理备库
ALTER DATABASE CONVERT TO PHYSICAL STANDBY;
```

## 三种保护模式

### 保护模式对比

| 模式 | 数据保护级别 | 性能影响 | 网络要求 |
|---|---|---|---|
| Maximum Protection | 零数据丢失 | 高 | 网络稳定 |
| Maximum Availability | 零数据丢失（允许短暂不可用）| 中 | 较高 |
| Maximum Performance | 可能丢失数据 | 低 | 一般 |

### Maximum Protection（最大保护）

```sql
-- 原理：主库事务提交必须等待至少一个备库确认
-- 特点：
-- 1. 零数据丢失
-- 2. 如果所有备库都不可用，主库自动关闭
-- 3. 网络必须稳定，否则主库会停止服务

-- 配置
-- 备库需要配置 STANDBY LOGFILE
ALTER DATABASE SET STANDBY DATABASE TO MAXIMIZE PROTECTION;

-- 备库初始化参数
-- *.log_archive_dest_2='SERVICE=standby1 LGWR SYNC AFFIRM'

-- 执行计划
/*
 * 事务提交
 *      │
 *      ▼
 * LGWR 写入 online redo
 *      │
 *      ▼
 * LGWR 同时写入备库 redo（SYNC 同步）
 *      │
 *      ▼
 * 备库 RFS 写入 standby redo
 *      │
 *      ▼
 * 备库 ARCH 归档 standby redo
 *      │
 *      ▼
 * 备库返回确认
 *      │
 *      ▼
 * 主库提交完成
 */
```

### Maximum Availability（最大可用）

```sql
-- 原理：与最大保护相同，但如果备库不可用，会降级为最大性能模式
-- 特点：
-- 1. 正常情况下零数据丢失
-- 2. 备库故障时自动降级，不影响主库服务
-- 3. 备库恢复后自动恢复保护

-- 配置
ALTER DATABASE SET STANDBY DATABASE TO MAXIMIZE AVAILABILITY;

-- 备库初始化参数
-- *.log_archive_dest_2='SERVICE=standby1 LGWR SYNC AFFIRM'

-- 执行计划（正常情况）
/*
 * 与最大保护相同
 */

/-- 执行计划（备库故障时）
/*
 * 事务提交
 *      │
 *      ▼
 * LGWR 写入 online redo
 *      │
 *      ▼
 * LGWR 尝试写入备库（超时）
 *      │
 *      ▼
 * 降级为 ASYNC 模式，继续提交
 *      │
 *      ▼
 * 记录警告日志，发送告警
 */
```

### Maximum Performance（最大性能）

```sql
-- 原理：主库提交不等待备库确认
-- 特点：
-- 1. 主库性能最优
-- 2. 可能丢失少量数据（取决于网络延迟和数据量）
-- 3. 备库故障不影响主库

-- 配置
ALTER DATABASE SET STANDBY DATABASE TO MAXIMIZE PERFORMANCE;

-- 备库初始化参数
-- *.log_archive_dest_2='SERVICE=standby1 LGWR ASYNC'

-- 执行计划
/*
 * 事务提交
 *      │
 *      ▼
 * LGWR 写入 online redo
 *      │
 *      ▼
 * LGWR 异步发送到备库
 *      │
 *      ▼
 * 主库立即返回提交完成
 *      │
 *      ▼
 * 后台进程继续传输 redo 到备库
 */
```

## 配置步骤

### 环境准备

```bash
# 1. 主库准备工作
# 确保归档模式开启
ALTER DATABASE ARCHIVELOG;

# 确保 force logging
ALTER DATABASE FORCE LOGGING;

# 确保 Supplemental Logging
ALTER DATABASE ADD SUPPLEMENTAL LOG DATA;
```

### 主库配置

```sql
-- 2. 主库初始化参数
-- tnsnames.ora 添加备库连接串
/*
 * STANDBY1 =
 *   (DESCRIPTION =
 *     (ADDRESS = (PROTOCOL = TCP)(HOST = standby1_host)(PORT = 1521))
 *     (CONNECT_DATA =
 *       (SERVER = DEDICATED)
 *       (SERVICE_NAME = standby1)
 *     )
 *   )
 */

-- 主库初始化参数
ALTER SYSTEM SET LOG_ARCHIVE_CONFIG='DG_CONFIG=(primary,standby1)';
ALTER SYSTEM SET LOG_ARCHIVE_DEST_1='LOCATION=USE_DB_RECOVERY_FILE_DEST VALID_FOR=(ALL_LOGFILES,ALL_ROLES)';
ALTER SYSTEM SET LOG_ARCHIVE_DEST_2='SERVICE=standby1 LGWR SYNC AFFIRM VALID_FOR=(ONLINE_LOGFILE,PRIMARY_ROLE)';
ALTER SYSTEM SET STANDBY_FILE_MANAGEMENT=AUTO;
ALTER SYSTEM SET FAL_SERVER='STANDBY1';
```

### 备库配置

```sql
-- 3. 备库初始化参数
ALTER SYSTEM SET LOG_ARCHIVE_CONFIG='DG_CONFIG=(primary,standby1)';
ALTER SYSTEM SET LOG_ARCHIVE_DEST_1='LOCATION=USE_DB_RECOVERY_FILE_DEST VALID_FOR=(ALL_LOGFILES,ALL_ROLES)';
ALTER SYSTEM SET STANDBY_FILE_MANAGEMENT=AUTO;
ALTER SYSTEM SET FAL_SERVER='PRIMARY';

-- 备库设置为备库角色
SHUTDOWN IMMEDIATE;
STARTUP MOUNT;
ALTER DATABASE REGISTER PHYSICAL LOGFILE '/path/to/log';
ALTER DATABASE RECOVER MANAGED STANDBY DATABASE USING CURRENT LOGFILE DISCONNECT;
```

### 验证配置

```sql
-- 查看 Data Guard 状态
SELECT * FROM V$DATABASE;

-- 查看保护模式
SELECT PROTECTION_MODE, PROTECTION_LEVEL, DATABASE_ROLE FROM V$DATABASE;

-- 查看归档目的地状态
SELECT DEST_ID, DEST_NAME, STATUS, TYPE, DATABASE_MODE FROM V$ARCHIVE_DEST;

-- 查看归档日志
SELECT SEQUENCE#, FIRST_TIME, NEXT_TIME, APPLIED FROM V$ARCHIVED_LOG ORDER BY SEQUENCE#;

-- 测试日志传输
ALTER SYSTEM SWITCH LOGFILE;

-- 查看备库应用状态
SELECT NAME, VALUE, UNIT FROM V$DATAGUARD_STATS;
```

## 故障转移

### 计划内切换

```sql
-- 1. 确保所有 redo 已应用
SELECT SEQUENCE#, APPLIED FROM V$ARCHIVED_LOG WHERE APPLIED = 'YES';

-- 2. 主库切换到备库
ALTER DATABASE COMMIT TO SWITCHOVER TO PHYSICAL STANDBY;

-- 3. 关闭并重启原主库
SHUTDOWN IMMEDIATE;
STARTUP MOUNT;

-- 4. 原备库切换到主库
ALTER DATABASE COMMIT TO SWITCHOVER TO PRIMARY;

-- 5. 打开新主库
ALTER DATABASE OPEN;

-- 6. 启动原主库为备库
ALTER DATABASE RECOVER MANAGED STANDBY DATABASE USING CURRENT LOGFILE DISCONNECT;
```

### 故障转移（Failover）

```sql
-- 当主库不可恢复时的处理

-- 1. 确认备库同步状态
SELECT THREAD#, MAX(SEQUENCE#) FROM V$ARCHIVED_LOG GROUP BY THREAD#;

-- 2. 如果可能，先应用所有可用的 redo
ALTER DATABASE RECOVER MANAGED STANDBY DATABASE FINISH;

-- 3. 将备库转换为主库
ALTER DATABASE ACTIVATE PHYSICAL STANDBY DATABASE;

-- 4. 打开数据库
ALTER DATABASE OPEN;

-- 5. 重建其他备库
-- 新的主库需要重新配置其他备库
```

## Broker 配置

### Broker 简介

Data Guard Broker 是 Oracle 提供的 Data Guard 管理工具，支持图形化和命令行管理：

```bash
# 启用 Broker（主库和备库都需要）
ALTER SYSTEM SET DG_BROKER_START=TRUE;

# 使用 DGMGRL 连接到 Broker
$ dgmgrl
DGMGRL> CONNECT sys/password@primary
DGMGRL> SHOW CONFIGURATION;

# 创建配置
DGMGRL> CREATE CONFIGURATION 'my_dg' AS PRIMARY DATABASE IS 'primary' CONNECT IDENTIFIER IS 'primary';
DGMGRL> ADD DATABASE 'standby1' AS CONNECT IDENTIFIER IS 'standby1';

# 启用配置
DGMGRL> ENABLE CONFIGURATION;
DGMGRL> SHOW CONFIGURATION;

# 查看详细状态
DGMGRL> SHOW DATABASE 'primary';
DGMGRL> SHOW DATABASE 'standby1';
```

### Broker 故障转移

```bash
# 使用 Broker 执行切换
DGMGRL> SWITCHOVER TO 'standby1';

# 使用 Broker 执行故障转移
DGMGRL> FAILOVER TO 'standby1';

# 恢复原主库为备库
DGMGRL> REINSTATE DATABASE 'primary';
```

## 监控与维护

### 监控脚本

```sql
-- 监控 Data Guard 健康状态
SELECT 
    DATABASE_ROLE,
    PROTECTION_MODE,
    PROTECTION_LEVEL,
    SWITCHOVER_STATUS,
    ARCHIVELOG_CHANGE#
FROM V$DATABASE;

-- 监控 redo 应用延迟
SELECT 
    NAME,
    VALUE,
    UNIT,
    TIME_COMPUTED
FROM V$DATAGUARD_STATS
WHERE NAME IN ('transport lag', 'apply lag');

-- 监控归档目的地
SELECT 
    DEST_ID,
    DEST_NAME,
    STATUS,
    ARCHIVED_THREAD#,
    ARCHIVED_SEQ#,
    APPLIED_THREAD#,
    APPLIED_SEQ#,
    DB_UNIQUE_NAME
FROM V$ARCHIVE_DEST_STATUS
WHERE STATUS != 'INACTIVE';

-- 监控告警日志（MRP 进程状态）
SELECT * FROM V$MANAGED_STANDBY WHERE PROCESS LIKE '%MRP%';
```

## 总结

Oracle Data Guard 提供了完整的高可用和灾备方案：

- **三种保护模式**：Maximum Protection、Maximum Availability、Maximum Performance
- **零数据丢失**：SYNC 同步模式保证数据不丢失
- **快速故障转移**：Broker 简化管理，自动切换
- **读写分离**：备库可打开只读，支持报表查询
- **灵活配置**：支持物理备库、逻辑备库、快照备库

根据业务需求选择合适的保护模式，在**数据安全**和**业务连续性**之间找到最佳平衡点。
