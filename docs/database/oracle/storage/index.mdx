# Oracle 存储结构：数据在磁盘上是怎么安的？

你有没有遇到过这种情况——明明加了索引，查询还是很慢；表空间明明还有空间，插入却报错了；分区表用了，查询还是全表扫。

问题往往不在 SQL 本身，而在**数据的物理存储方式**。

Oracle 的存储结构决定了数据如何组织、如何读取、如何维护。选对了存储策略，查询自然快；选错了，再好的索引也救不了你。

这一模块，我们从表的类型开始，到索引的设计，再到分区和压缩，把存储结构的里里外外讲清楚。

## 表类型与索引基础

这两个话题经常被分开讲，但实际上是紧密关联的——**你选什么表类型，直接决定了你该用什么索引**。

### 表类型

Oracle 支持多种表类型，每种都有它的适用场景：

| 类型 | 特点 | 什么时候用 |
|---|---|---|
| **堆表**（Heap Table） | 数据无序存放，按需扩展 | 大多数 OLTP 场景，默认选择 |
| **索引组织表**（IOT） | 数据按主键排序存储 | 主键查询频繁，几乎不需要全表扫描 |
| **聚簇表**（Cluster Table） | 多表数据物理存储在一起 | 关联查询极频繁的场景 |
| **外部表**（External Table） | 数据在数据库外部 | ETL、数据仓库只读场景 |

堆表是默认的，但**不是万能的**。如果你的业务有明显的访问模式（比如总是按主键查、总是按时间范围查），选对表类型能省掉大量不必要的 I/O。

### 索引类型

索引是 Oracle 优化的核心武器，但武器选错了比没有武器更危险：

| 类型 | 原理 | 适用场景 |
|---|---|---|
| **B-Tree 索引** | 平衡树结构 | 主键、唯一键、高基数列 |
| **位图索引** | 位图向量 | 低基数列，OLAP 数据仓库 |
| **函数索引** | 对表达式建索引 | `WHERE UPPER(name) = 'ABC'` |
| **反向键索引** | 键值反转存储 | 序列字段插入的热点问题 |

:::details 索引不是越多越好

每建一个索引，INSERT/UPDATE/DELETE 都要同步维护。这就像书的目录——目录太详细，更新起来就慢。Oracle 的聚簇因子（Clustering Factor）衡量索引顺序和表数据顺序的匹配程度，聚簇因子越高，索引效率越低。

:::

### IOT 与聚簇因子

索引组织表（IOT）把数据和主键索引存在一起——数据本身就是索引，叶子节点直接是行数据。这让它在**按主键范围查询**时几乎无敌，因为不需要额外的回表 I/O。

但 IOT 也有代价：所有非主键列的更新都可能移动行（行迁移），导致 I/O 增加。

聚簇因子描述的是索引键顺序和表中数据物理顺序的匹配程度——

| 聚簇因子 | 意味着什么 | 索引效率 |
|---|---|---|
| 低（接近表块数） | 索引顺序和数据顺序一致 | 高——一次 I/O 能读多行 |
| 高（接近行数） | 数据乱序存放 | 低——每次索引查找都可能多一次 I/O |

## 分区与压缩

### 分区表：让大表变小

当单表数据量过亿时，全表扫描一次就要扫几亿行。分区把一张大表拆成多张小表，对查询透明，但对性能影响巨大。

| 分区策略 | 原理 | 适用场景 |
|---|---|---|
| **Range 分区** | 按范围划分 | 时间序列数据（按月/按年） |
| **Hash 分区** | 按哈希均匀分布 | 没有明显热点列，数据均匀分布 |
| **List 分区** | 按枚举值划分 | 地区、状态等有限取值 |
| **复合分区** | 组合两种策略 | Range-Hash、Range-List |

分区裁剪（Partition Pruning）是分区表最大的性能优势——查询只访问相关分区，其他分区根本不动。

### 表压缩：空间换性能

Oracle 的表压缩不是简单的 zip 文件，而是**在数据块级别消除重复值**。压缩后的数据在内存中解压使用，对查询完全透明。

| 类型 | 压缩时机 | 适用场景 |
|---|---|---|
| **OLTP 压缩** | DML 时实时压缩 | 高频更新的 OLTP 系统 |
| **归档压缩** | 数据入库时批量压缩 | 很少修改的历史数据 |
| **混合列压缩**（ HCC） | 列式存储，按列压缩 | 数据仓库，分析型场景 |

:::tip 学习建议

存储结构的设计需要在开发早期考虑。一旦表上积累了大量数据，再进行结构变更的成本会非常高。建议结合业务场景（OLTP/OLAP）、数据特点（冷热分离、分布均匀度）和维护需求（分区裁剪、备份策略）综合决策。

:::
