# Oracle 表压缩：OLTP 压缩与 HCC 压缩深度解析

数据量爆炸式增长，存储成本居高不下。

一个 100TB 的数据库，每年光存储成本就要几十万。传统的行存储压缩效果有限，而 Oracle 的 **Hybrid Columnar Compression（HCC）** 可以将压缩率提升到 10-50 倍——100TB 数据压缩后只需要 2-10TB。

今天，我们来聊聊 Oracle 的表压缩技术：**OLTP 压缩**和**HCC 压缩**，如何选择、如何配置、如何验证效果。

## 表压缩概述

### Oracle 压缩类型

| 压缩类型 | 适用场景 | 压缩比 | CPU 开销 | 数据修改 |
|---|---|---|---|---|
| Basic Table Compression | DSS/OLAP | 10x-50x | 低 | 不支持 |
| OLTP Table Compression | OLTP | 2x-4x | 中等 | 支持 |
| HCC (Query Low/High) | DSS/OLAP | 10x-50x | 中等 | 不推荐 |
| HCC (Archive Low/High) | 历史归档 | 10x-100x | 较高 | 禁止 |

### 压缩原理

Oracle 的压缩基于 **跨行数据块共享相同值** 的原理：

```
未压缩的数据块：
┌─────────────────────────────────────────────────────────┐
│  Row 1: A001 | 张三 | 北京 | 销售部 | 10000            │
│  Row 2: A002 | 李四 | 北京 | 销售部 | 12000            │
│  Row 3: A003 | 王五 | 上海 | 技术部 | 15000            │
│  Row 4: A004 | 赵六 | 上海 | 技术部 | 13000            │
└─────────────────────────────────────────────────────────┘

压缩后的数据块：
┌─────────────────────────────────────────────────────────┐
│  Symbol Table:                                          │
│    "销售部" -> 01, "技术部" -> 02                      │
│    "北京" -> 10, "上海" -> 20                          │
│                                                         │
│  Row 1: A001 | 张三 | 10 | 01 | 10000                 │
│  Row 2: A002 | 李四 | 10 | 01 | 12000                 │
│  Row 3: A003 | 王五 | 20 | 02 | 15000                 │
│  Row 4: A004 | 赵六 | 20 | 02 | 13000                 │
└─────────────────────────────────────────────────────────┘
```

## OLTP 表压缩

### 基本概念

OLTP 表压缩（ROW STORE COMPRESS BASIC / FOR OLTP）是 Oracle 11g R2 引入的特性，专门为频繁修改的 OLTP 场景设计：

| 特性 | 说明 |
|---|---|
| 压缩算法 | 类似于 Basic Compression，但允许 DML 操作 |
| 触发时机 | 直接 INSERT/UPDATE 会压缩（使用 BLOCK TERMINATED 算法）|
| 压缩位置 | 内存中的数据块 |
| 兼容性 | 与 OLTP 工作负载兼容 |

### 配置方法

```sql
-- 创建 OLTP 压缩表
CREATE TABLE orders (
    order_id NUMBER,
    customer_id NUMBER,
    order_date DATE,
    amount NUMBER(10,2),
    status VARCHAR2(20)
)
COMPRESS FOR OLTP;

-- 修改现有表为 OLTP 压缩
ALTER TABLE orders COMPRESS FOR OLTP;

-- 修改表空间默认压缩
CREATE TABLESPACE app_data
    DATAFILE '/u01/oradata/app_data.dbf' SIZE 1G
    DEFAULT COMPRESS FOR OLTP;

-- 验证压缩状态
SELECT table_name, compression, compress_for
FROM user_tables
WHERE table_name = 'ORDERS';

-- 输出：
-- TABLE_NAME | COMPRESSION | COMPRESS_FOR
-- ORDERS     | ENABLED     | OLTP
```

### 直接路径加载

```sql
-- 直接路径加载会自动使用压缩
INSERT /*+ APPEND */ INTO orders
SELECT * FROM orders_history WHERE order_date >= SYSDATE - 30;

-- 或者使用 CTAS 创建压缩表
CREATE TABLE orders_compressed COMPRESS FOR OLTP AS
SELECT * FROM orders;

-- 批量加载优化
ALTER SESSION SET SQL_MODE = 'APPEND';

-- 启用表压缩后，批量加载也会压缩数据
INSERT INTO orders VALUES (1, 100, SYSDATE, 299.00, 'completed');
```

### OLTP 压缩的局限性

```sql
-- OLTP 压缩对普通 INSERT/UPDATE 有一定开销
-- 每次 DML 操作都需要处理压缩符号表

-- 建议场景：
-- 1. 表主要是读取，偶有更新
-- 2. 批量加载后不再修改的数据
-- 3. 压缩效果不够理想时可以禁用

-- 禁用压缩
ALTER TABLE orders NOCOMPRESS;

-- 查看表压缩统计
SELECT table_name, 
       num_rows,
       blocks,
       avg_row_len,
       compression,
       compress_for
FROM user_tables
WHERE table_name = 'ORDERS';
```

## HCC 压缩（Hybrid Columnar Compression）

### HCC 原理

HCC 是 Oracle Exadata 的核心技术，利用 **跨行压缩单元（Compression Unit）** 实现更高压缩比：

```
传统行压缩：
┌─────────────────────────────────────────────────────────┐
│  Block 1          │  Block 2          │  Block 3      │
│  Row 1-100        │  Row 101-200      │  Row 201-300  │
└─────────────────────────────────────────────────────────┘

HCC 压缩：
┌─────────────────────────────────────────────────────────┐
│              Compression Unit (跨多个 Block)              │
│  ┌─────────────────────────────────────────────────┐   │
│  │  Column Values stored by Column (列式存储)       │   │
│  │  Dept: [10,20,30,10,20,10,20,...]              │   │
│  │  Name: [A,B,C,A,B,A,B,...]                     │   │
│  │  Salary: [1000,2000,3000,1000,...]             │   │
│  └─────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
```

### HCC 压缩级别

| 压缩级别 | 压缩比 | 适用场景 | 数据修改 |
|---|---|---|---|
| QUERY LOW | 10x | 大数据量扫描，优化读取性能 | 可修改，性能下降 |
| QUERY HIGH | 15x | 分析查询，需要高压缩比 | 可修改，性能下降 |
| ARCHIVE LOW | 20x | 历史数据归档 | 不推荐修改 |
| ARCHIVE HIGH | 50x-100x | 冷数据归档，长期存储 | 禁止修改 |

### HCC 配置

```sql
-- 创建 HCC 压缩表（Exadata 环境）
CREATE TABLE sales_history (
    sale_id NUMBER,
    product_id NUMBER,
    sale_date DATE,
    amount NUMBER(10,2)
)
COMPRESS FOR QUERY LOW;

-- 创建高压缩归档表
CREATE TABLE archive_data (
    id NUMBER,
    data CLOB,
    create_date DATE
)
COMPRESS FOR ARCHIVE HIGH;

-- 修改现有表为 HCC
ALTER TABLE large_table COMPRESS FOR QUERY HIGH;

-- 批量转换（使用 MOVE）
ALTER TABLE large_table MOVE COMPRESS FOR ARCHIVE HIGH;
```

### HCC 限制条件

```sql
-- HCC 不支持的场景：
-- 1. 包含 LOB 字段的表
-- 2. 索引组织表（IOT）
-- 3. 集群表（Cluster Tables）
-- 4. 含有 function-based index 的表
-- 5. 闪回数据归档（Flashback Data Archive）

-- 检查表是否可以使用 HCC
SELECT table_name, 
       compression,
       compress_for,
       dropped_columns  -- 标记删除列
FROM user_tables
WHERE table_name = 'SALES_HISTORY';

-- 删除列后会影响压缩（列标记为 dropped）
ALTER TABLE sales_history SET UNUSED (unused_column);
```

## Basic Table Compression

### 基本压缩

```sql
-- 创建基本压缩表
CREATE TABLE sales (
    sale_id NUMBER,
    product_id NUMBER,
    sale_date DATE,
    amount NUMBER(10,2)
)
COMPRESS;

-- 基本压缩仅在直接路径加载时生效
-- 直接路径加载：
INSERT /*+ APPEND */ INTO sales SELECT * FROM sales_source;
CREATE TABLE sales_new COMPRESS AS SELECT * FROM sales_source;
ALTER TABLE sales MOVE;  -- 重新组织并压缩

-- 验证压缩效果
SELECT table_name, 
       blocks,
       avg_row_len,
       compression,
       compress_for
FROM user_tables
WHERE table_name IN ('SALES', 'SALES_NEW');
```

## 压缩与性能

### 压缩对性能的影响

| 操作类型 | 未压缩 | OLTP 压缩 | HCC LOW | HCC HIGH |
|---|---|---|---|---|
| 全表扫描 | 基准 | 略快 | 明显快 | 最快 |
| 索引扫描 | 基准 | 略快 | 略快 | 略快 |
| 单行访问 | 基准 | 略慢 | 较慢 | 较慢 |
| INSERT | 基准 | 略慢 | 较慢 | 非常慢 |
| UPDATE | 基准 | 略慢 | 慢 | 可能失败 |
| DELETE | 基准 | 略慢 | 慢 | 可能失败 |

### 压缩性能优化建议

```sql
-- 1. 分离热数据和冷数据
-- 热数据：少量最新数据，频繁修改
CREATE TABLE orders_current COMPRESS FOR OLTP AS
SELECT * FROM orders WHERE order_date >= TRUNC(SYSDATE, 'MM');

-- 冷数据：历史数据，只读
CREATE TABLE orders_archive COMPRESS FOR ARCHIVE HIGH AS
SELECT * FROM orders WHERE order_date < TRUNC(SYSDATE, 'MM');

-- 2. 合理设计分区
CREATE TABLE orders (
    order_id NUMBER,
    customer_id NUMBER,
    order_date DATE,
    amount NUMBER(10,2)
)
PARTITION BY RANGE (order_date) (
    PARTITION p2024_q1 VALUES LESS THAN (TO_DATE('2024-04-01', 'YYYY-MM-DD')) COMPRESS FOR OLTP,
    PARTITION p2024_q2 VALUES LESS THAN (TO_DATE('2024-07-01', 'YYYY-MM-DD')) COMPRESS FOR OLTP,
    PARTITION p2024_q3 VALUES LESS THAN (TO_DATE('2024-10-01', 'YYYY-MM-DD')) COMPRESS FOR QUERY LOW,
    PARTITION p2024_q4 VALUES LESS THAN (TO_DATE('2025-01-01', 'YYYY-MM-DD')) COMPRESS FOR QUERY LOW,
    PARTITION p_archive VALUES LESS THAN (MAXVALUE) COMPRESS FOR ARCHIVE HIGH
);

-- 3. 使用合适的压缩级别
-- 读取频繁的表：QUERY LOW/HIGH
-- 写入频繁的表：OLTP
-- 历史归档表：ARCHIVE HIGH
```

## 监控与验证

### 查看压缩统计

```sql
-- 查看表的压缩状态
SELECT table_name,
       num_rows,
       blocks,
       empty_blocks,
       avg_space,
       compression,
       compress_for
FROM user_tables
WHERE table_name LIKE 'SALES%';

-- 查看分区压缩状态
SELECT partition_name,
       num_rows,
       blocks,
       compression,
       compress_for
FROM user_tab_partitions
WHERE table_name = 'ORDERS';

-- 查看索引压缩状态
SELECT index_name,
       blevel,
       leaf_blocks,
       distinct_keys,
       compression,
       prefix_length
FROM user_indexes
WHERE table_name = 'ORDERS';
```

### 计算压缩比

```sql
-- 方法一：比较压缩前后的大小
-- 创建未压缩表
CREATE TABLE orders_uncompressed AS SELECT * FROM orders;
-- 创建压缩表
CREATE TABLE orders_compressed COMPRESS FOR OLTP AS SELECT * FROM orders;

-- 比较大小
SELECT 
    t.table_name,
    t.blocks * 8192 / 1024 / 1024 AS size_mb,
    t.num_rows,
    t.avg_row_len,
    ROUND(t.num_rows * t.avg_row_len / 1024 / 1024, 2) AS estimated_size_mb,
    t.compression,
    t.compress_for
FROM user_tables t
WHERE t.table_name IN ('ORDERS_UNCOMPRESSED', 'ORDERS_COMPRESSED');

-- 方法二：使用 DBMS_COMPRESSION
SET SERVEROUTPUT ON
DECLARE
    l_comptype NUMBER;
    l_blkcnt   NUMBER;
    l_blkcmp   NUMBER;
    l_rowcmp   NUMBER;
    l_compratio NUMBER;
BEGIN
    DBMS_COMPRESSION.GET_COMPRESSION_RATIO (
        scrub.Schema_NAME,
        NULL,
        'ORDERS',
        NULL,
        l_comptype,
        l_blkcnt,
        l_blkcmp,
        l_rowcmp,
        l_compratio
    );
    
    DBMS_OUTPUT.PUT_LINE('Compression Ratio: ' || l_compratio);
    DBMS_OUTPUT.PUT_LINE('Compression Type: ' || l_comptype);
END;
/
```

## 最佳实践

### 压缩策略总结

| 数据类型 | 压缩策略 | 理由 |
|---|---|---|
| 热数据（最近 1-3 个月）| OLTP 压缩或无压缩 | 高频修改，OLTP 压缩开销可接受 |
| 温数据（3-12 个月）| QUERY LOW/HIGH | 偶尔查询，需要平衡压缩比和性能 |
| 冷数据（1 年以上）| ARCHIVE LOW/HIGH | 主要归档，压缩比优先 |
| 临时表/中间结果 | Basic Compression | 批量加载后直接删除 |
| 历史分区 | 对应分区使用 HCC | 按分区灵活配置 |

### 实施步骤

```sql
-- 1. 识别可压缩的表
SELECT table_name, 
       num_rows * avg_row_len as estimated_size,
       blocks,
       compression
FROM user_tables
WHERE num_rows > 10000
  AND compression = 'DISABLED'
ORDER BY estimated_size DESC;

-- 2. 评估压缩收益
-- 运行 DBMS_COMPRESSION.GET_COMPRESSION_RATIO

-- 3. 选择压缩级别
-- 基于数据修改频率和查询模式

-- 4. 执行压缩
-- 在线压缩（11g+）
ALTER TABLE orders COMPRESS FOR OLTP;

-- 离线压缩（使用 MOVE）
ALTER TABLE orders MOVE COMPRESS FOR OLTP;

-- 重建索引
ALTER INDEX idx_orders_id REBUILD;

-- 5. 验证效果
SELECT table_name, compression, compress_for, blocks
FROM user_tables WHERE table_name = 'ORDERS';
```

## 总结

Oracle 表压缩是降低存储成本的有效手段：

- **OLTP 压缩**：支持 DML 操作，适合 OLTP 场景，压缩比 2-4x
- **HCC QUERY**：高压缩比，适合分析查询，压缩比 10-50x
- **HCC ARCHIVE**：最高压缩比，适合历史归档，压缩比 50-100x
- **合理分区**：热数据用 OLTP，冷数据用 HCC，灵活配置
- **监控验证**：定期检查压缩效果，调整策略

选择合适的压缩级别，在**存储成本**和**性能**之间找到平衡点。
