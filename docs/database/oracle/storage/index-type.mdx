# Oracle 索引类型：B-Tree、位图、函数、反向键索引

索引是数据库性能优化的核心技术。想象一个没有索引的图书馆——你只能逐行扫描所有书籍才能找到某一本书；有了索引，就像有了目录，可以快速定位到目标。Oracle 提供了多种索引类型，每种都有其适用场景和独特的实现原理。

一个典型的 DBA 场景：开发团队发现某查询从 50ms 突然飙升到 5 秒。检查发现是因为新增了数据导致全表扫描。创建索引后，查询恢复到 50ms。这个场景揭示了索引的核心价值：**用空间换时间，用额外的存储成本换取查询性能的显著提升**。

## B-Tree 索引：默认选择

B-Tree（Balanced Tree，平衡树）索引是 Oracle 的默认索引类型，适用于绝大多数场景。

### B-Tree 索引的结构

B-Tree 索引由根节点、分支节点和叶子节点组成：

```text
                            [根节点]
                           /    |    \
                    [分支节点]
                   /    |    \
            [叶子节点] ←→ [叶子节点] ←→ [叶子节点]
              ↓           ↓           ↓
           (索引条目)  (索引条目)  (索引条目)
              ↓           ↓           ↓
          ROWID         ROWID       ROWID
```

**根节点和分支节点**：存储索引键值和指向下一级节点的指针，引导搜索路径。

**叶子节点**：存储索引键值和对应的 ROWID，叶子节点之间双向链表连接，支持快速的范围扫描。

**平衡性**：Oracle 自动维护 B-Tree 的平衡，确保从根到叶子的路径长度相同，查询性能稳定。

### B-Tree 索引的创建

```sql
-- 单列索引
CREATE INDEX idx_emp_name ON employees(last_name);

-- 复合索引（组合索引）
CREATE INDEX idx_emp_dept_job ON employees(department_id, job_id);

-- 指定表空间
CREATE INDEX idx_emp_name ON employees(last_name) TABLESPACE users;

-- 指定索引选项
CREATE INDEX idx_emp_name ON employees(last_name) 
    NOLOGGING     -- 减少日志生成
    COMPUTE STATISTICS;  -- 创建时收集统计信息
```

### 复合索引的列顺序

复合索引的列顺序至关重要。遵循**最左前缀原则**：

```sql
-- 创建复合索引 (dept_id, job_id, hire_date)
CREATE INDEX idx_emp ON employees(department_id, job_id, hire_date);
```

| 查询条件 | 是否使用索引 | 原因 |
|---|---|---|
| `WHERE dept_id = 50` | 使用 | 使用索引的前导列 |
| `WHERE dept_id = 50 AND job_id = 'IT'` | 使用 | 使用前两列 |
| `WHERE dept_id = 50 AND job_id = 'IT' AND hire_date > '2020-01-01'` | 使用 | 使用全部列 |
| `WHERE job_id = 'IT'` | 不使用 | 跳过前导列 |
| `WHERE hire_date > '2020-01-01'` | 不使用 | 跳过前导列 |

### B-Tree 索引的扫描方式

**索引唯一扫描（Index Unique Scan）**：索引列上有唯一约束或主键，查询最多返回一行数据。

```sql
SELECT * FROM employees WHERE employee_id = 100;  -- 唯一扫描
```

**索引范围扫描（Index Range Scan）**：查询返回多行数据，使用索引的范围扫描能力。

```sql
SELECT * FROM employees WHERE department_id = 50;  -- 范围扫描
SELECT * FROM employees WHERE department_id BETWEEN 50 AND 80;  -- 范围扫描
```

**索引全扫描（Index Full Scan）**：读取整个索引，适用于 ORDER BY 索引列的场景。

```sql
SELECT last_name FROM employees ORDER BY last_name;  -- 全扫描
```

**索引快速全扫描（Index Fast Full Scan）**：多块读取整个索引，不保证顺序，适用于只需索引列的查询。

```sql
SELECT last_name FROM employees;  -- 快速全扫描
```

### B-Tree 索引的特点

| 特点 | 说明 |
|---|---|
| 适合高基数列 | 列的唯一值越多，索引效率越高 |
| 支持等值查询 | `=`、`IN` |
| 支持范围查询 | `>`、`<`、`BETWEEN` |
| 支持排序 | 利用索引顺序避免排序 |
| 支持前缀匹配 | `LIKE 'A%'` |

## 位图索引（Bitmap Index）

位图索引专为低基数列设计，在数据仓库和 OLAP 场景下表现出色。

### 位图索引的原理

位图索引为每个索引值创建一个位图，每一位对应一行数据：

```sql
-- 为性别列创建位图索引
CREATE BITMAP INDEX idx_gender ON employees(gender);

-- 假设 employees 表有 1000 行
-- 男性 (M) 位图: 1010101010... (1=男, 0=女)
-- 女性 (F) 位图: 0101010101...
```

### 位图索引的存储

对于一个有 100 万行、性别列只有 2 个值的表：

- B-Tree 索引：100 万个索引条目，每个约 20 字节，总计约 20MB
- 位图索引：2 个位图，每个 100 万位，总计约 250KB

位图索引在高压缩比下可以非常紧凑。

### 位图索引的特点

| 特点 | 说明 |
|---|---|
| 适合低基数列 | 性别（2）、状态（5）、国家（200）等 |
| 高压缩比 | 位图可高效压缩 |
| 支持 AND/OR 运算 | 位图AND/OR 操作极快 |
| 不适合高并发写入 | 更新位图需要锁定多个行 |
| 不适合 OLTP 系统 | DML 操作会锁住大量行 |

### 位图索引的位运算

位图索引支持快速的集合运算：

```sql
-- 查询男性且在 50 部门的员工
SELECT * FROM employees 
WHERE gender = 'M' AND department_id = 50;
-- 转换为位图 AND 操作
-- gender_M_bitmap AND dept_50_bitmap = 结果位图
```

## 函数索引（Function-Based Index）

函数索引基于表达式或函数计算结果创建索引，解决了在列上使用函数后索引无法使用的问题。

### 创建函数索引

```sql
-- 大小写不敏感的查询
CREATE INDEX idx_upper_name ON employees(UPPER(last_name));

-- 使用表达式
CREATE INDEX idx_salary_annual ON employees(salary * 12);

-- 使用 CASE 表达式
CREATE INDEX idx_active ON employees(
    CASE WHEN status = 'ACTIVE' THEN 1 ELSE 0 END
);

-- 使用自定义函数
CREATE INDEX idx_full_name ON employees(first_name || ' ' || last_name);
```

### 函数索引的使用

```sql
-- 无函数索引时，全表扫描
SELECT * FROM employees WHERE UPPER(last_name) = 'SMITH';  -- 无法使用索引

-- 有函数索引后，使用索引
-- Oracle 自动识别 UPPER(last_name) 使用 idx_upper_name
```

### 函数索引的限制

- `DBMS_STATS.GATHER_TABLE_STATS` 收集统计信息时需指定 `METHOD_OPT`
- 函数必须是确定性的（相同的输入总是相同的输出）
- CLOB 列上不能创建函数索引

## 反向键索引（Reverse Key Index）

反向键索引将索引键值的字节顺序反转后存储，有效解决 B-Tree 索引在序列主键上的叶子块热点问题。

### 问题背景

在高并发插入场景下，序列主键（如 `seq.NEXTVAL`）导致新行总是插入到 B-Tree 索引的最后一个叶子块。当多个进程同时插入时，所有进程竞争最后一个叶子块的锁，严重影响并发性能。

### 反向键索引的原理

```sql
CREATE INDEX idx_rev_id ON orders(order_id) REVERSE;

-- 原始键值: 1001, 1002, 1003, 1004
-- 反转后:    1001, 2001, 3001, 4001
```

反转后，原本相邻的键值（如 1001 和 1002）变成了相隔很远的值，新插入的数据分散到不同的叶子块，减少了热点竞争。

### 反向键索引的限制

- **不支持范围扫描**：`WHERE id > 100` 无法使用反向键索引
- **不支持前导通配符搜索**：`WHERE id LIKE '%123'` 无法使用

### 反向键索引的适用场景

- 序列主键的高并发插入
- 多节点写入同一表的场景（如 RAC）

## 分区索引

分区索引是针对分区表创建的索引，分为本地索引和全局索引。

### 本地索引（Local Index）

本地索引与分区表绑定，每个分区对应一个索引分区：

```sql
-- 创建本地前缀索引
CREATE INDEX idx_local_dept ON orders(order_date) LOCAL;

-- 创建本地非前缀索引
CREATE INDEX idx_local_dept ON orders(order_date, customer_id) LOCAL;
```

### 全局索引（Global Index）

全局索引跨越所有分区，类似普通索引：

```sql
-- 创建全局索引
CREATE INDEX idx_global ON orders(customer_id) GLOBAL;
```

### 本地 vs 全局索引对比

| 特性 | 本地索引 | 全局索引 |
|---|---|---|
| 分区绑定 | 每个分区一个索引分区 | 单一索引，跨所有分区 |
| 分区操作影响 | 自动维护 | 需要手动维护或重建 |
| 查询效率 | 与查询条件相关 | 单次 I/O 定位 |
| 适用场景 | 大多数场景 | 全局查询为主的场景 |

## 索引管理

### 查看索引信息

```sql
-- 查看索引列表
SELECT index_name, table_name, uniqueness, status
FROM user_indexes;

-- 查看索引列
SELECT index_name, column_name, column_position
FROM user_ind_columns
WHERE index_name = 'IDX_EMP_DEPT'
ORDER BY column_position;

-- 查看索引统计信息
SELECT index_name, blevel, leaf_blocks, distinct_keys, 
       clustering_factor, num_rows
FROM user_indexes;
```

### 重建索引

```sql
-- 在线重建索引（Oracle 11g+）
ALTER INDEX idx_emp_name REBUILD ONLINE;

-- 监控索引使用
ALTER INDEX idx_emp_name MONITORING USAGE;

-- 查看索引使用情况
SELECT * FROM v$object_usage WHERE index_name = 'IDX_EMP_NAME';
```

### 删除不需要的索引

```sql
-- 删除索引
DROP INDEX idx_unused_index;
```

## 常见面试问题

**问：B-Tree 索引的原理是什么？**

B-Tree 索引是一种平衡多路查找树，由根节点、分支节点和叶子节点组成。根节点和分支节点存储索引键值和指针，引导搜索路径。叶子节点存储索引键值和对应的 ROWID，且叶子节点之间通过双向链表连接，支持快速的范围扫描。Oracle 自动维护 B-Tree 的平衡性，确保查询性能稳定。

**问：什么情况下索引不会被使用？**

使用函数导致索引列不在前导位置；使用了 `IS NULL` 或 `IS NOT NULL`（除非有位图索引）；使用 `NOT IN`、`NOT EXISTS`；数据量过小导致全表扫描更高效；列上数据类型不匹配导致隐式转换；使用了 `LIKE '%xxx'` 前导通配符；统计信息不准确导致优化器选择错误。

**问：位图索引和 B-Tree 索引的区别是什么？**

B-Tree 索引适合高基数列（唯一值多），支持等值和范围查询，但高并发 DML 时锁冲突严重。位图索引适合低基数列（唯一值少），压缩比高，支持快速的位运算（AND/OR），但不适合高并发写入的 OLTP 系统。

**问：函数索引的作用是什么？**

函数索引基于表达式或函数计算结果创建索引，解决了在列上使用函数后索引无法使用的问题。例如 `UPPER(name)` 查询可以使用 `UPPER(name)` 的函数索引。需要注意的是，函数必须是确定性的。

## 总结

Oracle 提供了丰富的索引类型以满足不同场景的需求。B-Tree 索引是默认选择，适合大多数高基数列；位图索引适合数据仓库的低基数列；函数索引解决了表达式查询的索引使用问题；反向键索引解决了序列主键的热点问题。理解每种索引的原理和适用场景，是进行 SQL 优化的基础。
