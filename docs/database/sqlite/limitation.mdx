# SQLite 适用场景与局限性

很多人听说 SQLite 不适合高并发、不支持集群、规模有限制，就认为它是一个「玩具数据库」。这是一种误解。

每种数据库都有其擅长的领域和不适用的场景。SQLite 的设计哲学是：**在它擅长的领域做到极致，在它不擅长的领域承认局限**。

理解 SQLite 的局限性，不是为了否定它，而是为了做出正确的技术选型决策。

你能回答这个问题吗——**「什么时候应该选择 SQLite，而不是 MySQL 或 PostgreSQL？」**

## SQLite 的适用场景

### 1. 移动应用：iOS 和 Android

移动应用是 SQLite 最典型的应用场景：

| 特点 | SQLite 优势 |
|---|---|
| 离线优先 | 无需网络，随时可用 |
| 资源受限 | 核心库仅几百 KB |
| 单用户 | 无并发写入问题 |
| 数据量有限 | TB 级别以下完全够用 |
| 零运维 | 应用自带数据库 |

```
典型移动应用数据库：
┌─────────────────────────────────────────────────────────┐
│ iOS/Android 应用                                          │
├─────────────────────────────────────────────────────────┤
│ ┌─────────────────────────────────────────────────────┐ │
│ │              SQLite 数据库                           │ │
│ ├─────────────────────────────────────────────────────┤ │
│ │ users.db      - 用户数据                             │ │
│ │ cache.db      - 离线缓存                             │ │
│ │ analytics.db  - 埋点数据                             │ │
│ └─────────────────────────────────────────────────────┘ │
│                                                         │
│ 特点：每个应用独立的数据库文件，互不干扰                 │
└─────────────────────────────────────────────────────────┘
```

适合 SQLite 的移动应用：
- 笔记应用（Notion、Obsidian）
- 邮件客户端
- RSS 阅读器
- 离线地图
- 音乐播放器
- 健身追踪应用

### 2. 桌面应用：Electron 和本地工具

### 4. 小型网站：流量不大的个人博客

SQLite 也可以用于小型网站：

```python
# Flask + SQLite 示例
from flask import Flask, g
import sqlite3

app = Flask(__name__)
DATABASE = 'blog.db'

def get_db():
    db = getattr(g, '_database', None)
    if db is None:
        db = g._database = sqlite3.connect(DATABASE)
    return db

@app.route('/')
def index():
    db = get_db()
    posts = db.execute('SELECT * FROM posts ORDER BY created DESC').fetchall()
    return render_template('index.html', posts=posts)
```

适合 SQLite 的网站场景：
- 个人博客（< 1000 日 PV）
- 小型 CMS
-企业内部工具
- 原型开发
- 单用户管理系统

## SQLite 的局限性

### 1. 并发写入限制：单一写锁

SQLite 最主要的局限是同一时刻只能有一个写事务：

```sql
-- 问题演示
-- 连接 1：开启写事务
BEGIN IMMEDIATE;
UPDATE users SET name = 'Alice' WHERE id = 1;
-- 持有 EXCLUSIVE 锁

-- 连接 2-100：尝试写事务
BEGIN IMMEDIATE;  -- 全部阻塞
-- 所有写操作串行化

-- 性能影响：
-- 10 个并发写事务 → 串行执行
-- 100 个并发写事务 → 更严重的排队
```

:::warning 高并发写入场景

如果你需要：
- 每秒处理 1000+ 次写操作
- 多个用户同时修改同一条数据
- 毫秒级的写响应时间

**SQLite 不适合，请选择 PostgreSQL、MySQL 等数据库。**
:::

### 2. 数据规模限制

SQLite 对数据规模有一定限制：

| 限制项 | 最大值 | 说明 |
|---|---|---|
| 数据库文件大小 | 281 TB | 理论值，受文件系统限制 |
| 单表行数 | 无限制 | 受文件大小限制 |
| 单行数据量 | 1 GB | BLOB/TEXT 最大值 |
| 列数 | 32767 | 远超实际需求 |
| 索引数 | 无限制 | 受表大小限制 |

实际使用中的限制：

```sql
-- 1GB 单行数据
INSERT INTO files (name, content) VALUES ('large_file', zeroblob(1024*1024*1024));
-- 单行数据过大会影响性能

-- 大量数据查询
SELECT * FROM logs WHERE created > '2024-01-01';
-- 超过内存容量时，需要磁盘 I/O
```

### 3. 无网络访问：本地文件依赖

SQLite 只能访问本地文件系统，无法作为网络数据库使用：

```sql
-- ❌ 错误：无法通过网络访问
sqlite3 192.168.1.100:myapp.db

-- ✅ 正确：只能本地访问
sqlite3 /path/to/myapp.db

-- 如果需要网络访问：
-- 方案 1：使用 REST API 包装
flask --app api_server:app run --host 0.0.0.0

-- 方案 2：使用同步服务
# 将 SQLite 变更同步到远程 PostgreSQL
```

这意味着：
- 无法从多台服务器同时访问同一个数据库
- 无法实现远程备份
- 无法实现分布式数据库

### 4. 缺少高级特性

SQLite 缺少许多企业级特性：

| 特性 | SQLite | MySQL | PostgreSQL |
|---|---|---|---|
| 存储过程 | 有限支持 | 支持 | 支持 |
| 触发器 | 支持 | 支持 | 支持 |
| 视图 | 支持 | 支持 | 支持 |
| 物化视图 | 不支持 | 支持 | 支持 |
| 外键约束 | 支持 | 支持 | 支持 |
| JSON 函数 | 支持 | 支持 | 支持 |
| 数组类型 | 不支持 | 支持 | 支持 |
| 范围类型 | 不支持 | 不支持 | 支持 |
| GIS/PostGIS | 扩展支持 | 支持 | 支持 |
| 全文索引 | FTS5 扩展 | 支持 | 支持 |
| 窗口函数 | 支持 | 支持 | 支持 |
| CTEs | 支持 | 支持 | 支持 |

### 5. 无用户和权限管理

SQLite 默认没有用户和权限管理：

```sql
-- ❌ SQLite 不支持
CREATE USER alice WITH PASSWORD 'secret';
GRANT SELECT ON users TO alice;

-- ✅ 替代方案 1：应用层实现
-- 在应用代码中检查用户权限

-- ✅ 替代方案 2：文件系统权限
chmod 600 myapp.db  # 仅所有者可读写

-- ✅ 替代方案 3：SQLite 安全扩展
-- 使用 SQLCipher 加密数据库
```

### 6. 运维工具相对简单

与 MySQL 和 PostgreSQL 相比，SQLite 的运维工具较少：

| 运维操作 | MySQL | PostgreSQL | SQLite |
|---|---|---|---|
| 性能监控 | Performance Schema, Slow Query Log | pg_stat_statements | sqlite3_analyzer |
| 备份 | mysqldump, XtraBackup | pg_dump, Barman | sqlite3 .backup |
| 复制 | 主从, Group Replication | 流复制, 逻辑复制 | 第三方工具 |
| 集群 | MySQL Cluster, InnoDB Cluster | Citus, Patroni | LiteFS, rqlite |
| 在线迁移 | pt-online-schema-change | pg_repack | ALTER TABLE（阻塞） |

## 面试要点总结

### 面试官可能会问的问题

**Q1：SQLite 的局限性有哪些？**

回答要点：
- 同一时刻只能有一个写事务
- 无网络访问，只能本地使用
- 无内置用户和权限管理
- 缺少存储过程、物化视图等高级特性
- 运维工具相对简单

**Q2：什么时候应该选择 SQLite 而不是 MySQL？**

回答要点：
- 单机或嵌入式环境，无需网络访问
- 读多写少，数据量 < 100GB
- 需要零配置、零运维
- 移动应用或桌面应用
- 需要作为文件格式嵌入其他文件

**Q3：SQLite 和 MySQL 在并发方面的主要区别是什么？**

回答要点：
- SQLite 同一时刻只能有一个写事务
- MySQL 支持多个并发写事务（通过行锁）
- SQLite 读事务可以并发多个
- WAL 模式下，SQLite 读不阻塞写
- MySQL 在高并发场景下性能更好

**Q4：SQLite 的数据规模限制是多少？**

回答要点：
- 理论最大文件大小 281 TB
- 实际受限于文件系统
- 单行数据最大 1 GB
- 单表行数无限制
- 索引数无限制

**Q5：SQLite 适合用于网站后端吗？**

回答要点：
- 小型网站（< 1000 日 PV）可以使用
- 单服务器场景可行
- 不适合高流量网站
- 不适合需要多服务器扩展的场景
- 可以通过 HTTP API 层包装使用
