# SQLite 数据库文件结构：Page 与 B-Tree

你有没有想过这个问题——当执行 `SELECT * FROM users WHERE id = 1` 时，SQLite 是怎么从那个数据库文件里找到数据的？

表面上看，这是一条简单的 SQL 语句。但实际上，SQLite 要做很多事情：首先打开数据库文件，读取文件头确定页面大小，然后根据 id=1 这个条件在 B-Tree 中搜索，最后定位到具体的 Page，读取那一行数据。

这个过程涉及 SQLite 最核心的两个概念：**Page（页面）** 和 **B-Tree（B 树）**。理解它们，是掌握 SQLite 内部原理的关键。

## 数据库文件格式：100 字节的秘密

### 文件头结构

SQLite 数据库文件的前 100 个字节是文件头，包含数据库的元信息。这个头部的结构是固定的：

```
┌────────────────────────────────────────────────────────────────────┐
│                         SQLite 文件头 (100 字节)                       │
├────────┬─────────┬───────────────────────────────────────────────────┤
│  偏移   │ 大小    │                     说明                          │
├────────┼─────────┼───────────────────────────────────────────────────┤
│   0    │   16    │ 魔数："SQLite format 3\000"                        │
│  16    │   2     │ 页面大小（字节），大端序。512-65536 的 2 的幂次      │
│  18    │   1     │ 文件格式写入版本（1= legacy, 2= WAL）               │
│  19    │   1     │ 文件格式读取版本                                    │
│  20    │   1     │ 页面保留空间（unused space）                        │
│  21    │   1     │ 嵌入式最大载荷（embedded payload fraction）         │
│  22    │   1     │ 最小载荷分数（min payload fraction）               │
│  23    │   1     │ 叶子节点载荷分数（leaf payload fraction）          │
│  24    │   4     │ 文件修改计数器（file change counter）              │
│  28    │   4     │ 数据库文件大小（单位：页面）                        │
│  32    │   4     │ Schema 格式版本                                    │
│  36    │   4     │ Schema 层编号                                      │
│  40    │   4     │ 文本编码：1=UTF-8, 2=UTF-16le, 3=UTF-16be           │
│  44    │   4     │ User version（用户定义版本号）                       │
│  48    │   4     │ 增量vacuum模式                                     │
│  52    │   4     │ 有效用于BRIN索引的页面大小                          │
│  56    │   4     │ 校验和范围                                         │
│  60    │   40    │ 保留空间                                            │
│ 100    │  ...    │ 第一个页面内容从这里开始                              │
└────────┴─────────┴───────────────────────────────────────────────────┘
```

你可以通过十六进制编辑器查看一个 SQLite 数据库文件的头部：

```bash
$ xxd -l 100 myapp.db
00000000: 5351 4c69 7465 2066 6f72 6d61 7420 3300  SQLite format 3.
00000010: 0100 0000 1000 0000 0000 0000 0000 0000  ................
00000020: 0000 0000 0000 0000 0000 0000 0000 0000  ................
00000030: 0000 0000 0000 0001 0000 0000 0000 0000  ................
00000040: 0000 0000 0000 0000 0000 0000 0000 0000  ................
```

从偏移 16-17 可以看到页面大小：`0x1000` = 4096 字节。

:::tip 页面大小的重要性

页面大小在数据库创建时就固定了，之后不能随意更改。它直接影响：
- 单个页面能存储多少数据
- B-Tree 的深度
- I/O 效率（过大或过小都不好）
- 数据库文件总大小

:::

### 页面大小的选择

SQLite 支持的页面大小范围是 512 到 65536 字节，且必须是 2 的幂次。常见的页面大小：

| 页面大小 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| 1024 | 适合小型记录 | B-Tree 层数可能很深 | 嵌入式设备 |
| 4096 | 默认值，大多数场景最佳 | - | 通用场景 |
| 8192 | 适合大型记录、批量读取 | 单次 I/O 开销大 | 分析型应用 |
| 65536 | 适合超大型记录 | 内存碎片化风险 | 特殊场景 |

:::warning 页面大小不匹配的问题

如果尝试附加一个页面大小不同的数据库文件，SQLite 会报错：`file is not a database`。这是因为文件头的魔数和页面大小不匹配。
:::

## 页面结构：数据库的积木

### 页面类型概览

SQLite 数据库由固定大小的页面组成，每个页面属于以下类型之一：

| 类型 | header 第一个字节 | 说明 |
|---|---|---|
| 表/索引 B-Tree 内部页面 | 0x02 | 索引 B-Tree 内部节点 |
| 表 B-Tree 叶子页面 | 0x0D | 存储实际行数据 |
| 索引 B-Tree 叶子页面 | 0x0A | 索引数据 |
| Freelist 页面 | 0x00 | 回收的空闲页面 |
| 表 B-Tree 内部页面 | 0x05 | 表的内部节点 |
| Payload 溢出页面 | 0x00 | 大型数据的溢出存储 |
| Pointer Map 页面 | 0x00 | WAL 模式的指针映射 |

### B-Tree 叶子页面结构

表 B-Tree 叶子页面是存储实际数据的地方。它的内部结构如下：

```
┌─────────────────────────────────────────────────────────┐
│                    B-Tree 叶子页面                        │
├──────────┬──────────────────────────────────────────────┤
│ Header   │ 页面头信息（8 字节）                            │
│          │ - 第一个自由块偏移（2 字节）                     │
│          │ - Cell 数量（2 字节）                           │
│          │ - Cell 内容起始位置（2 字节）                    │
│          │ - 碎片化空闲字节数（1 字节）                     │
├──────────┴──────────────────────────────────────────────┤
│ Cell Pointer Array（Cell 指针数组）                       │
│ 每一项 2 字节，指向实际 Cell 的偏移位置                     │
│ 按键值升序排列                                            │
├──────────────────────────────────────────────────────────┤
│ Cells（Cell 内容区域）                                     │
│ [Cell N] [Cell N-1] ... [Cell 1] [Cell 0]                │
│ 从页面末尾向前生长                                         │
└──────────────────────────────────────────────────────────┘
```

这个结构有几个关键点：

1. **Cell 指针数组从页头向后增长**
2. **Cell 内容从页面末尾向前增长**
3. **两者相遇时，说明页面已满**
4. **Cell 数量为 0 时，页面为空**

### Cell 结构：数据的容器

Cell 是存储数据的最小单元。在 B-Tree 叶子页面中，每个 Cell 的结构如下：

```
┌─────────────────────────────────────────────────────────┐
│                        Cell 结构                          │
├─────────────────────────────────────────────────────────┤
│ Payload Length (变长，1-3 字节)                           │
│    该 Cell 的数据部分长度                                  │
├─────────────────────────────────────────────────────────┤
│ Row ID (变长，1-9 字节)                                   │
│    该行的唯一标识符                                       │
├─────────────────────────────────────────────────────────┤
│ Payload (N 字节)                                          │
│    实际存储的数据                                          │
│    [Header Length] [Column 1] [Column 2] ... [Column N]  │
└─────────────────────────────────────────────────────────┘
```

SQLite 使用变长整数（varint）来编码长度，这使得小数据占用很少的空间。

:::tip 变长整数的编码

SQLite 使用一种特殊的变长整数编码：
- 值 `<` 128：用 1 字节存储
- 值 `<` 16384：用 2 字节存储
- 值 `<` 2097152：用 3 字节存储
- 以此类推，每 7 位数据占 1 字节

这意味着存储 100 这个值只需要 1 字节，而不是固定的 8 字节。
:::

## B-Tree vs B+Tree：SQLite 的选择

### 标准 B-Tree

在标准 B-Tree 中，数据存储在所有节点（叶子节点和内部节点）：

```
           [50 | 指向子页面]
          /        |        \
    [20|30|ptr]  [60|70]  [80|90|ptr]
    /    |    \     |        /    \
 [data] [data] [data] [data]  [data] [data]
```

查找时，从根节点开始比较，可以在内部节点就找到目标。

### B+Tree

在 B+Tree 中，数据**只存储在叶子节点**，内部节点只存储键和指针：

```
         [50 | ptr]         [ptr]
        /        \            |
   [30 | ptr]  [70 | ptr]   [ptr]
     |           |           |
     v           v           v
  [leaf]      [leaf]      [leaf]
  [data]      [data]      [data]
```

查找时，必须走到叶子节点才能拿到数据。但叶子节点之间有双向链表连接，便于范围查询。

### SQLite 的选择：改进的 B-Tree

SQLite 既不是标准 B-Tree，也不是 B+Tree。它使用的是一种改进的 B-Tree，称为 **B-Tree（SQLite 变体）**。

关键区别在于：SQLite 的 B-Tree 数据**存储在所有叶子节点，但不存储在内部节点**——这与标准 B-Tree 不同。

```
         [50 | ptr]                    [ptr]
        /        \                       |
   [30 | ptr]  [70 | ptr]             [ptr]
     |           |                       |
     v           v                       v
  [leaf]      [leaf]                  [leaf]
  [50|data]   [70|data]  [80|data]    [data]
```

这与 B+Tree 的区别是：
- SQLite B-Tree：叶子节点之间**没有链表连接**
- B+Tree：叶子节点之间有双向链表

:::warning 这意味着什么

因为 SQLite B-Tree 叶子节点之间没有链表连接，**范围查询需要回到根节点重新定位**，而不是像 B+Tree 那样直接顺着链表遍历。这意味着某些范围查询的性能可能不如 B+Tree。
:::

## 表 B-Tree vs 索引 B-Tree

### 表 B-Tree

当创建一张表时，SQLite 会为表创建一个 B-Tree（B-Tree 表），用于存储所有行数据。

```sql
CREATE TABLE users (
    id INTEGER PRIMARY KEY,
    name TEXT,
    email TEXT
);
```

这张表的 B-Tree 结构：

```
表 B-Tree（按 ROWID 排序）
┌─────────────────────────────────────────────────────┐
│ Root Page（根页面，可能是内部节点或叶子节点）           │
├─────────────────────────────────────────────────────┤
│ 如果数据量小（< 页面数限制）                          │
│   根页面就是叶子页面，直接存储所有行数据                │
│                                                     │
│ 如果数据量大                                          │
│   根页面是内部节点，指向多个叶子页面                    │
│   叶子页面存储实际的行数据（id, name, email）          │
└─────────────────────────────────────────────────────┘
```

### 索引 B-Tree

当在列上创建索引时，SQLite 会为索引创建一个独立的 B-Tree（索引 B-Tree）：

```sql
CREATE INDEX idx_email ON users(email);
```

索引 B-Tree 的结构：

```
索引 B-Tree（按 email 排序）
┌─────────────────────────────────────────────────────┐
│ Root Page                                            │
├─────────────────────────────────────────────────────┤
│ 内部节点：存储 email 值和指向子页面的指针               │
│                                                     │
│ 叶子节点：存储 (email, rowid) 对                      │
│   ("alice@example.com", 1)                          │
│   ("bob@example.com", 2)                            │
│   ("carol@example.com", 3)                         │
└─────────────────────────────────────────────────────┘
```

查询时，SQLite 先在索引 B-Tree 中找到目标 email，然后根据 rowid 回表查询完整行数据。

## Freelist：删除数据的回收站

### Freelist 页面

当删除数据库中的行时，对应的页面不会立即从文件中移除。它们会被加入 Freelist，等待后续复用。

```
┌─────────────────────────────────────────────────────┐
│                   数据库文件                           │
├─────────────────────────────────────────────────────┤
│ Page 1: sqlite_master (schema 表)                   │
│ Page 2-5: 表数据 B-Tree                              │
│ Page 6: 已删除，加入 Freelist trunk                  │
│ Page 7: 已删除，加入 Freelist trunk                  │
│ ...                                                  │
└─────────────────────────────────────────────────────┘
```

Freelist 由一个或多个 trunk 页面管理，每个 trunk 页面存储最多约 400 个空闲页面指针。

```sql
-- 查看 Freelist 页面数量
PRAGMA freelist_count;
-- 结果：表示有多少个空闲页面可用
```

### 页面复用

新插入的数据优先复用 Freelist 中的页面，而不是在文件末尾分配新页面。这保证了数据库文件不会无限增长。

```sql
-- 插入数据
INSERT INTO users (name, email) VALUES ('dave', 'dave@example.com');

-- 如果有可用的 Freelist 页面
-- 新数据会复用之前删除的页面位置
-- 而不是追加到文件末尾
```

## 溢出页面：处理大字段

### Payload 溢出机制

如果一行数据太大，无法存储在一个页面中，SQLite 会使用溢出页面：

```
正常情况（数据 < Page Size - Header）：
┌──────────────────────────────────────────┐
│ Cell: [length][rowid][data...]            │
└──────────────────────────────────────────┘

溢出情况（数据 > Page Size - Header）：
┌──────────────────────────────────────────┐
│ Cell: [length][rowid][overflow_ptr][min]  │
├──────────────────────────────────────────┤
│ Overflow Page 1: [data_frag_1]           │
├──────────────────────────────────────────┤
│ Overflow Page 2: [data_frag_2]           │
├──────────────────────────────────────────┤
│ ...                                      │
└──────────────────────────────────────────┘
```

Cell 中只存储第一部分数据和指向溢出页面的指针，后续数据存储在溢出页面中，通过链表连接。

:::tip BLOB 和 TEXT 的存储

对于 BLOB 和 TEXT 类型的大字段，SQLite 默认采用溢出存储。这避免了大字段影响整行的存储效率，但增加了读取时的 I/O 开销。
:::

## 内部页面：构建多层索引

### 内部节点的结构

当数据量足够大时，B-Tree 会分多层。根页面是内部节点，指向多个子页面：

```
内部页面结构：
┌─────────────────────────────────────────────────────┐
│ Header (8 字节)                                      │
│ Cell Pointer Array (2N 字节)                         │
├─────────────────────────────────────────────────────┤
│ Cell 0: [key1_len][key1][child_ptr1]                │
│ Cell 1: [key2_len][key2][child_ptr2]                │
│ Cell 2: [key3_len][key3][child_ptr3]                │
└─────────────────────────────────────────────────────┘
```

内部节点的 Cell 存储：
- 键值（索引列的值或 ROWID）
- 子页面的页码指针

### B-Tree 的深度

B-Tree 的层数（深度）取决于数据量和页面大小：

| 数据行数 | 页面大小 | B-Tree 深度 |
|---|---|---|
| `<` 1,000 | 4096 | 1 |
| `<` 1,000,000 | 4096 | 2 |
| `<` 1,000,000,000 | 4096 | 3 |
| `<` 1,000,000 | 1024 | 3 |

深度越深，查询需要的 I/O 次数越多，性能越差。

```sql
-- 查看 B-Tree 深度
EXPLAIN QUERY PLAN SELECT * FROM users WHERE id = 1;

/* 结果示例：
SEARCH TABLE users USING INTEGER PRIMARY KEY (rowid=?)
说明：直接通过主键定位，不需要遍历 B-Tree
*/
```

## 实际文件结构示例

让我们用一个具体的例子来看文件结构：

```bash
# 创建一个简单的数据库
$ sqlite3 test.db "CREATE TABLE t(a); INSERT INTO t VALUES(1);"

# 查看数据库文件信息
$ sqlite3 test.db "PRAGMA page_size;"   -- 4096
$ sqlite3 test.db "PRAGMA page_count;"  -- 1

# 查看文件头
$ xxd test.db | head -10
00000000: 5351 4c69 7465 2066 6f72 6d61 7420 3300  SQLite format 3.
00000010: 0100 1000 0101 2000 0001 0000 0000 0000  ..... .......
```

页面结构：

```
test.db (1 page = 4096 bytes)
┌────────────────────────────────────────────────────────┐
│ Page 1: sqlite_master 表                                │
│ Header (100 bytes): 文件头 + B-Tree 表头                  │
│ ├─ 0x0D (13): 表 B-Tree 叶子页面                          │
│ ├─ Cell count: 3 (sqlite_master 的 3 条记录)              │
│ └─ 内容: CREATE TABLE t(a); 等 schema 信息              │
├────────────────────────────────────────────────────────┤
│ 剩余空间: 用于存储实际的 schema 定义                      │
└────────────────────────────────────────────────────────┘
```

## 面试要点总结

### 面试官可能会问的问题

**Q1：SQLite 数据库文件的页面结构是怎样的？**

回答要点：
- 文件头 100 字节，包含魔数、页面大小、编码等信息
- 页面由 B-Tree 页面头、Cell 指针数组、Cell 内容组成
- Cell 指针数组从页头向后增长，Cell 内容从页尾向前增长
- 两种主要页面类型：内部节点和叶子节点

**Q2：SQLite 用的是 B-Tree 还是 B+Tree？**

回答要点：
- SQLite 使用改进的 B-Tree，既不是标准 B-Tree，也不是 B+Tree
- 与 B+Tree 的区别：叶子节点之间没有双向链表
- 这意味着范围查询可能需要重新定位，而不是顺着链表遍历
- 数据存储在所有叶子节点（不是内部节点）

**Q3：为什么 SQLite 的查询有时比 MySQL 慢？**

可能的答案：
- SQLite 没有查询缓存，每次都需要重新执行
- SQLite 的 B-Tree 叶子节点没有链表，范围查询效率较低
- 缺少优化器（SQLite 的优化器相对简单）
- 适合的场景不同：嵌入式场景下 SQLite 是最优选择
