# SQLite 面试汇总

SQLite 是一款独特且广泛使用的数据库，虽然在传统后端开发中不如 MySQL 和 PostgreSQL 常见，但它是面试中考察数据库基础原理的绝佳素材。它的简单性使得底层机制更容易被理解，而 WAL 机制、事务原子性、文件结构等知识点往往能引发面试官和候选人之间的深入讨论。

本篇汇总 SQLite 面试中最常被问到的核心问题。

## 一、SQLite 基础认知

### Q1：SQLite 是什么？它和 MySQL、PostgreSQL 有什么区别？

**问题分析：** 考察候选人对 SQLite 的基本认知和它与其他数据库的本质区别。

**参考答案：**

SQLite 是一款嵌入式、零配置、无服务器的数据库引擎。数据库本身就是一个文件，应用程序通过 API 直接读写这个文件，而不是通过网络协议。

与 MySQL 和 PostgreSQL 的核心区别：

| 维度 | SQLite | MySQL/PostgreSQL |
|---|---|---|
| 架构 | 嵌入式，进程内运行 | 客户端-服务器，独立进程 |
| 部署 | 单文件，零配置 | 需要安装服务进程 |
| 访问方式 | 直接文件读写 | 网络协议通信 |
| 并发 | 单一写锁 | 多连接并发写 |
| 规模 | 适合中小规模 | 适合大规模数据 |

SQLite 的核心设计哲学是：**简单的事情简单做**。它不追求 MySQL 那样的功能丰富度，而是在嵌入式场景下做到极致。

**面试追问：** SQLite 为什么没有服务器进程？

答案：SQLite 设计之初的目标就是嵌入式场景。在资源受限的环境中（如手机、嵌入式设备），不需要也不应该运行独立的服务进程。直接文件访问减少了进程间通信的开销，也简化了部署。

### Q2：SQLite 为什么是世界上部署量最多的数据库？

**问题分析：** 考察候选人对 SQLite 应用场景的了解。

**参考答案：**

根据官方统计，SQLite 被部署在无数设备和应用中：

| 部署场景 | 代表应用 |
|---|---|
| 移动设备 | Android、iOS 系统和应用 |
| 浏览器 | Chrome、Firefox、Safari 的内部数据库 |
| 操作系统 | macOS、Windows、Linux 的系统配置 |
| 桌面应用 | Skype、Git、Photoshop |
| 嵌入式系统 | 无人机、汽车电子、工业控制器 |
| 文件格式 | PDF、DICOM 等使用 SQLite 作为数据容器 |

**面试追问：** 为什么手机大量使用 SQLite？

答案：手机有几个关键特点——资源受限（CPU、内存、存储都有限）、需要离线工作、单用户使用、需要零配置部署。这些特点恰好是 SQLite 的优势所在：
- SQLite 核心库仅几百 KB，资源占用极低
- 数据库就是文件，无需网络和后台服务
- 单文件便于备份和迁移
- 事务保证数据在应用崩溃时不会丢失

## 二、文件结构与存储

### Q3：SQLite 数据库文件的结构是怎样的？

**问题分析：** 考察候选人对 SQLite 底层存储机制的理解。

**参考答案：**

SQLite 数据库文件由固定大小的页面组成，默认页面大小 4096 字节。文件结构分为两个部分：

```
┌─────────────────────────────────────────────────────────┐
│                    SQLite 数据库文件                       │
├─────────────────────────────────────────────────────────┤
│ Header (100 字节)                                        │
│ ├─ 魔数："SQLite format 3"                             │
│ ├─ 页面大小（2 字节）                                    │
│ ├─ 文本编码                                             │
│ ├─ 文件修改计数器                                        │
│ └─ 其他元信息                                           │
├─────────────────────────────────────────────────────────┤
│ Page 1-N                                                 │
│ ├─ B-Tree 表页面（存储表数据）                           │
│ ├─ B-Tree 索引页面（存储索引）                           │
│ ├─ Freelist 页面（回收站）                               │
│ └─ Overflow 页面（大字段溢出）                           │
└─────────────────────────────────────────────────────────┘
```

每个页面的内部结构：

```
┌─────────────────────────────────────────────────────────┐
│                    B-Tree 页面                           │
├─────────────────────────────────────────────────────────┤
│ Header (8-12 字节)                                       │
│ ├─ Cell 数量                                            │
│ ├─ Cell 内容起始位置                                    │
│ └─ 碎片化空间                                           │
├─────────────────────────────────────────────────────────┤
│ Cell Pointer Array（指向实际 Cell）                       │
├─────────────────────────────────────────────────────────┤
│ Cells（从后向前生长）                                    │
│ ├─ Payload 长度                                         │
│ ├─ Row ID                                               │
│ └─ 数据内容                                              │
└─────────────────────────────────────────────────────────┘
```

### Q4：SQLite 使用的是什么类型的索引？

**问题分析：** 考察候选人对 SQLite 索引实现的理解。

**参考答案：**

SQLite 使用 **B-Tree** 作为索引结构，但与 MySQL InnoDB 的 B+Tree 有重要区别：

| 特性 | SQLite B-Tree | MySQL B+Tree |
|---|---|---|
| 数据位置 | 所有节点 | 仅叶子节点 |
| 叶子节点连接 | 无链表 | 双向链表 |
| 范围查询 | 需要重新定位 | 链表遍历 |

**SQLite B-Tree 的特点：**

1. 数据存储在所有叶子节点（不是内部节点）
2. 叶子节点之间没有双向链表连接
3. 范围查询时，每次都需要回到 B-Tree 根节点重新定位

```sql
-- 创建主键索引（INTEGER PRIMARY KEY）
CREATE TABLE users (id INTEGER PRIMARY KEY, name TEXT);
-- 特殊优化：id 列直接指向数据，无需回表

-- 创建普通索引
CREATE INDEX idx_name ON users(name);
-- 需要回表：根据索引找到 rowid，再查询完整数据
```

### Q5：什么是 Freelist？它在什么情况下会被使用？

**问题分析：** 考察候选人对 SQLite 页面管理机制的理解。

**参考答案：**

Freelist（空闲列表）是 SQLite 回收已删除页面使用的机制：

1. **删除数据时**：对应页面被加入 Freelist，而不是立即从文件中移除
2. **插入新数据时**：优先复用 Freelist 中的页面，而不是在文件末尾分配新页面
3. **Truncate 操作**：会真正释放 Freelist 页面的空间

```sql
-- 查看 Freelist 页面数量
PRAGMA freelist_count;

-- Freelist 由 trunk 页面管理
-- 每个 trunk 页面存储最多约 400 个空闲页面指针
```

**面试追问：** Freelist 页面过多会有什么影响？

答案：Freelist 页面过多可能导致：
- 文件大小增长（已删除但未释放的页面）
- 查询性能下降（需要遍历 Freelist 查找可用页面）
- 可以通过 `VACUUM` 命令清理

## 三、事务与 ACID

### Q6：SQLite 的 ACID 是如何实现的？

**问题分析：** 考察候选人对事务原理的深入理解。

**参考答案：**

SQLite 的 ACID 实现有其独特之处：

| 特性 | SQLite 实现方式 |
|---|---|
| 原子性 | 基于文件系统的原子写入（fsync） |
| 一致性 | 约束检查 + 事务回滚 |
| 隔离性 | SERIALIZABLE（默认）/ READ UNCOMMITTED |
| 持久性 | 提交时强制刷盘（fsync） |

**原子性的核心原理：**

SQLite 的原子性依赖操作系统和文件系统提供的能力：
1. 事务开始时获取锁（BEGIN IMMEDIATE）
2. 所有修改缓存在内存中
3. 提交时一次性写入文件
4. 如果写入中断，文件头不更新，恢复时根据日志判断回滚

**与 MySQL InnoDB 的对比：**

| 维度 | SQLite | MySQL InnoDB |
|---|---|---|
| 原子性 | 文件系统原子写入 | Redo Log + 崩溃恢复 |
| 恢复机制 | 基于 journal/WAL | 基于 Redo Log |
| 默认隔离级别 | SERIALIZABLE | REPEATABLE READ |

### Q7：SQLite 有哪几种回滚模式？WAL 是什么？

**问题分析：** 考察候选人对持久化机制的理解。

**参考答案：**

SQLite 支持三种事务回滚模式：

| 模式 | 说明 | 特点 |
|---|---|---|
| DELETE | 默认模式，写入前备份原页面 | 兼容性好，但写入慢 |
| TRUNCATE | 删除日志文件改为截断 | 性能稍好 |
| WAL | Write-Ahead Logging | 高并发读写 |

**WAL（Write-Ahead Logging）的核心思想：**

WAL 的核心理念是：**写操作先记日志，读操作照常进行**。

```
传统模式（DELETE/TRUNCATE）：
写事务 → 阻塞读 → 完成 → 读继续

WAL 模式：
写事务 → 写 WAL 文件 → 读继续（不阻塞）
              ↓
         Checkpoint（异步合并到数据库）
```

**WAL 模式的三个文件：**

```bash
myapp.db       # 主数据库文件
myapp.db-wal   # WAL 日志（未 checkpoint 的修改）
myapp.db-shm   # 共享内存（多进程协调）
```

**面试追问：** WAL 模式的 Checkpoint 是什么？

答案：Checkpoint 是将 WAL 文件中的修改合并回主数据库的过程。可以自动触发（默认 1000 页）或手动执行。Checkpoint 需要获取 EXCLUSIVE 锁，期间可能短暂阻塞读写。

### Q8：SQLite 的事务隔离级别是什么？

**问题分析：** 考察候选人对隔离级别的理解。

**参考答案：**

SQLite 只支持两种隔离级别：

| 隔离级别 | 说明 | 脏读 | 不可重复读 | 幻读 |
|---|---|---|---|---|
| SERIALIZABLE | 串行化（默认） | 不可能 | 不可能 | 不可能 |
| READ UNCOMMITTED | 未提交读 | 可能 | 可能 | 可能 |

**SQLite 的 SERIALIZABLE 实现：**

SQLite 的 SERIALIZABLE 与 MySQL InnoDB 的 REPEATABLE READ 有本质区别：

- MySQL InnoDB 使用 MVCC（多版本并发控制）实现，允许并发读写，通过版本链和 ReadView 判断可见性
- SQLite 使用**真正的串行执行**，通过锁机制确保同一时刻只有一个事务在执行写操作

```sql
-- MySQL InnoDB（MVCC）
事务 A: SELECT * FROM users;  -- 读取版本 1
事务 B: UPDATE users SET name = 'Bob' WHERE id = 1; COMMIT;
事务 A: SELECT * FROM users;  -- 如果 RR，可能仍读到旧值；如果是 RC，读到新值

-- SQLite（串行）
事务 A: SELECT * FROM users;  -- 读取快照
事务 B: BEGIN IMMEDIATE;  -- 等待事务 A 完成或阻塞
```

## 四、并发控制

### Q9：SQLite 的锁机制是怎样的？

**问题分析：** 考察候选人对数据库锁机制的理解。

**参考答案：**

SQLite 使用五种锁状态：

| 状态 | 说明 | 允许的并发 |
|---|---|---|
| UNLOCKED | 无锁 | 读写均可 |
| SHARED | 共享锁（读锁） | 多个读 |
| RESERVED | 预留锁（即将写） | 读 |
| PENDING | 待定锁（等待升级） | 无 |
| EXCLUSIVE | 排他锁（写锁） | 无 |

**锁状态转换：**

```
UNLOCKED → SHARED → RESERVED → PENDING → EXCLUSIVE
              ↑                   ↓
              └───────────────────┘
```

**WAL 模式的锁：**

WAL 模式不适用传统的数据库锁，而是使用：
- WAL 写入槽位：同一时刻只有一个写事务
- Checkpoint 锁：Checkpoint 时需要
- 读事务只需 SHARED 锁

**面试追问：** 为什么 SQLite 的写锁会阻塞读？

答案：这是 DELETE/TRUNCATE 模式的特性。写事务需要升级到 EXCLUSIVE 锁，此时所有 SHARED 锁持有者必须释放才能继续。WAL 模式下这个问题被解决——写操作不阻塞读。

### Q10：SQLite 为什么不适合高并发写入？

**问题分析：** 考察候选人对数据库并发性能的理解。

**参考答案：**

SQLite 不适合高并发写入的原因：

1. **单一写锁**：同一时刻只能有一个写事务，所有写操作串行化

```sql
-- 10 个并发写事务
事务 1: ████████████████████████████████████  (耗时 100ms)
事务 2:                  ████████████████████  (耗时 100ms，等待后执行)
事务 3:                                  ████████████  ...
...
总耗时：1000ms（10 个串行执行）

-- MySQL 多连接写入
事务 1: ████  (10ms)
事务 2: ████  (10ms，并发)
事务 3: ████  (10ms，并发)
...
总耗时：约 10ms（真正并发）
```

2. **刷盘开销**：每次写入都需要 fsync()，成为 I/O 瓶颈

3. **锁竞争**：WAL 模式虽改善了读并发，但写仍是串行

**适用场景对比：**

| 场景 | SQLite | MySQL |
|---|---|---|
| 读多写少 | ✅ 优秀 | ✅ 优秀 |
| 写并发低 | ✅ 优秀 | ✅ 优秀 |
| 高并发写入 | ❌ 差 | ✅ 优秀 |

## 五、性能优化

### Q11：如何分析 SQLite 的查询性能？

**问题分析：** 考察候选人对查询优化的实际动手能力。

**参考答案：**

**使用 EXPLAIN QUERY PLAN：**

```sql
-- 查看查询计划
EXPLAIN QUERY PLAN SELECT * FROM users WHERE email = 'test@example.com';

-- 常见结果：
-- SCAN TABLE users              -- 全表扫描（差）
-- SEARCH TABLE users USING INDEX idx_email (email=?)  -- 索引查找（好）
-- SEARCH TABLE users USING INTEGER PRIMARY KEY (rowid=?)  -- 主键查找（好）
```

**使用 ANALYZE 收集统计信息：**

```sql
-- 收集表和索引的统计信息
ANALYZE;

-- ANALYZE 前后的区别：
-- 未 ANALYZE：SQLite 假设数据均匀分布
-- 已 ANALYZE：基于实际采样数据估算代价
```

**常见的性能问题：**

1. **索引未使用**：数据量太小时，全表扫描反而更快
2. **最左前缀原则**：复合索引 `(a, b)` 不能用于 `WHERE b = ?`
3. **隐式类型转换**：`id = 123` 与 TEXT 列比较会导致全表扫描

```sql
-- 索引失效示例
CREATE TABLE items (id TEXT PRIMARY KEY);  -- TEXT 类型
SELECT * FROM items WHERE id = 123;  -- 隐式转换，索引失效
SELECT * FROM items WHERE id = '123';  -- 正确写法

-- 函数导致索引失效
SELECT * FROM users WHERE date(created_at) = '2024-03-30';  -- 失效
SELECT * FROM users WHERE created_at >= '2024-03-30 00:00:00';  -- 有效
```

## 六、Android 开发

### Q12：Android 中如何使用 SQLite？Room 是什么？

**问题分析：** 考察候选人在 Android 开发中使用 SQLite 的经验。

**参考答案：**

**传统方式：SQLiteOpenHelper**

```java
public class MyDatabaseHelper extends SQLiteOpenHelper {
    @Override
    public void onCreate(SQLiteDatabase db) {
        db.execSQL("CREATE TABLE users (_id INTEGER PRIMARY KEY, name TEXT)");
    }

    @Override
    public void onUpgrade(SQLiteDatabase db, int oldVersion, int newVersion) {
        db.execSQL("DROP TABLE IF EXISTS users");
        onCreate(db);
    }
}
```

**现代方式：Room**

Room 是 Android 官方推荐的数据库解决方案，建立在 SQLite 之上：

```java
// Entity：定义表结构
@Entity
public class User {
    @PrimaryKey(autoGenerate = true)
    public long id;

    public String name;
}

// DAO：定义数据访问
@Dao
public interface UserDao {
    @Query("SELECT * FROM users")
    List<User> getAll();

    @Insert
    long insert(User user);
}

// Database：定义数据库
@Database(entities = {User.class}, version = 1)
public abstract class AppDatabase extends RoomDatabase {
    public abstract UserDao userDao();
}
```

**Room 的优势：**
- 编译时 SQL 检查，减少运行时错误
- 减少模板代码
- 支持 LiveData 和 Coroutines
- 支持数据库迁移

## 面试技巧

### 如何回答 SQLite 相关问题

**原则一：原理优于 API**

面试官问 SQLite，不是想知道你会用 `INSERT INTO ...`，而是想知道你理解它的底层原理。

**原则二：对比优于单独讲解**

讲解 SQLite 时，适时与 MySQL 对比，能体现你对多种数据库的理解。

**原则三：承认局限是加分项**

主动说出 SQLite 不适合高并发写入，比被追问时支支吾吾要好得多。

**原则四：实践案例加分**

如果你有 Android 开发经验或 SQLite 实际项目经验，务必提及。

### 常见陷阱

| 陷阱 | 正确做法 |
|---|---|
| 只背概念，不理解原理 | 结合 WAL、Checkpoint 等讲解 |
| 过度吹嘘 SQLite | 客观说明适用场景 |
| 混淆 B-Tree 和 B+Tree | 明确 SQLite 使用 B-Tree |
| 不知道 ANALYZE | 提及统计信息对查询优化的影响 |

## 总结

SQLite 面试的核心知识点：

1. **架构特点**：嵌入式、无服务器、单文件
2. **文件结构**：Header + Page + B-Tree
3. **事务机制**：ACID 基于文件系统，WAL 模式
4. **并发控制**：五种锁状态，WAL 改善读并发
5. **性能优化**：EXPLAIN QUERY PLAN、ANALYZE
6. **适用场景**：嵌入式、移动端、读多写少

掌握这些知识点，足以应对 SQLite 相关的面试问题。
