# SQLite 并发限制：写锁与数据库锁

你可能听说过 SQLite 不适合高并发场景。这个说法不完全准确——SQLite 在**读并发**方面表现出色，但在**写并发**方面确实有限制。

理解这些限制，是决定是否使用 SQLite 的关键。如果你的应用需要每秒处理数千次写入，SQLite 确实不是最佳选择。但如果你的应用是读多写少，或者写入可以批量处理，SQLite 的并发能力完全够用。

你能回答这个问题吗——**「为什么 SQLite 同一时刻只能有一个写事务，但读事务可以并发数十个？」**

## SQLite 的锁机制：五层架构

### 锁状态详解

SQLite 使用五种锁状态来控制并发：

| 状态 | 说明 | 同一时刻数量 | 其他读 | 其他写 |
|---|---|---|---|---|
| UNLOCKED | 无连接使用数据库 | 无限制 | 允许 | 允许 |
| SHARED | 正在读取 | 多个 | 允许 | 阻塞 |
| RESERVED | 即将写入 | 1 | 允许 | 阻塞 |
| PENDING | 等待升级 | 1 | 阻塞 | 阻塞 |
| EXCLUSIVE | 正在写入 | 1 | 阻塞 | 阻塞 |

### 锁状态转换图

```
                    UNLOCKED
                       │
                       │ 连接打开
                       ▼
                   SHARED ◄──────────────────┐
                   │     │                     │
         读事务开启  │     │ BEGIN IMMEDIATE    │
                   │     │ 或写操作             │
                   │     ▼                     │
                   │  RESERVED                 │
                   │     │                     │
                   │     │ 等待所有 SHARED     │
                   │     ▼                     │
                   │  PENDING                  │
                   │     │                     │
                   │     │ 所有 SHARED 释放    │
                   │     ▼                     │
                   │  EXCLUSIVE                │
                   │     │                     │
                   │     │ COMMIT/ROLLBACK    │
                   │     │ + 尝试恢复 SHARED   │
                   │     └─────────────────────┘
```

### 实际锁行为示例

```sql
-- 连接 1：开启读事务
conn1> BEGIN;
conn1> SELECT * FROM users;  -- 获取 SHARED 锁

-- 连接 2：开启读事务（可以成功）
conn2> BEGIN;
conn2> SELECT * FROM users;  -- 获取 SHARED 锁（第二个 SHARED 锁）

-- 连接 3：尝试写事务（阻塞）
conn3> BEGIN IMMEDIATE;  -- 等待 RESERVED 锁

-- 连接 1 和 2 提交后，连接 3 获得 RESERVED 锁
-- 此时连接 4 尝试读（阻塞）
conn4> BEGIN;
conn4> SELECT * FROM users;  -- 等待 SHARED 锁

-- 连接 3 完成 PENDING → EXCLUSIVE → COMMIT
-- 连接 4 获得 SHARED 锁
```

## WAL 模式的并发特性

### WAL 对并发的改善

启用 WAL 模式后，并发特性发生显著变化：

```sql
PRAGMA journal_mode = WAL;

-- WAL 模式的锁结构：
-- - 数据库文件：SHARED / RESERVED / EXCLUSIVE
-- - WAL 文件：WRITE / CHECKPOINT / READ
```

| 场景 | DELETE/TRUNCATE 模式 | WAL 模式 |
|---|---|---|
| 1 写多读 | 写阻塞所有读 | 写并发所有读 |
| 多写 | 写串行 | 写串行（同一时刻只能一个写） |
| 多读 | 读并发 | 读并发 |

### WAL 的读并发

WAL 模式的核心优势是**读不阻塞写，写不阻塞读**：

```
时间轴 →
───────────────────────────────────────────────────────────────────
连接 1 (写) │ BEGIN │ [修改页面A] │ [写WAL] │ COMMIT │
               │         │           │         │
连接 2 (读) │         │ [读取页面A] │         │         │
连接 3 (读) │         │ [读取页面B] │         │         │
连接 4 (读) │         │ [读取页面C] │         │         │
───────────────────────────────────────────────────────────────────
                ↑         ↑           ↑         ↑
             无阻塞     无阻塞       无阻塞    无阻塞

DELETE 模式：
连接 1 (写) │ BEGIN │ [升级写锁] │ [阻塞读] │ [完成] │
连接 2 (读) │         │           │ [等待]   │         │
连接 3 (读) │         │           │ [等待]   │         │
───────────────────────────────────────────────────────────────────
                          ↑           ↑
                       写锁升级    其他事务阻塞
```

### WAL 的写并发限制

尽管 WAL 模式改进了读并发，但写并发仍然受限：

```sql
-- 连接 1：开启写事务
conn1> BEGIN IMMEDIATE;  -- 获得 WAL 写锁

-- 连接 2：尝试开启写事务（阻塞）
conn2> BEGIN IMMEDIATE;  -- 等待 WAL 写锁

-- 连接 3：可以读（不阻塞）
conn3> BEGIN;
conn3> SELECT * FROM users;  -- 正常读取
```

WAL 模式的写锁规则：
- 同一时刻只能有一个写事务
- 读事务可以并发任意多个
- Checkpoint 操作可能短暂阻塞读写

## 写锁的等待与超时

### 超时机制

默认情况下，SQLite 的锁等待是无限的，可能导致死锁：

```sql
-- 设置超时（毫秒）
PRAGMA busy_timeout = 5000;  -- 等待 5 秒后返回 SQLITE_BUSY

-- 0 表示立即返回
PRAGMA busy_timeout = 0;  -- SQLITE_BUSY

-- -1 表示无限等待（默认）
PRAGMA busy_timeout = -1;
```

### SQLITE_BUSY 处理

当数据库被锁定时，应用需要处理 `SQLITE_BUSY` 错误：

```c
// C 语言示例
int rc = sqlite3_exec(db, "BEGIN IMMEDIATE", NULL, NULL, NULL);
if (rc == SQLITE_BUSY) {
    // 数据库被锁定，等待后重试
    // 或返回错误给用户
    sleep(1);
    rc = sqlite3_exec(db, "BEGIN IMMEDIATE", NULL, NULL, NULL);
}

// Python 示例
import sqlite3
import time

def execute_with_retry(db, sql, max_retries=3):
    for i in range(max_retries):
        try:
            db.execute(sql)
            return
        except sqlite3.OperationalError as e:
            if 'locked' in str(e) and i < max_retries - 1:
                time.sleep(0.5)
            else:
                raise
```

### 死锁场景

```sql
-- 场景：两个连接互相等待对方释放锁

-- 连接 1：开启事务并锁定 A 表
conn1> BEGIN IMMEDIATE;
conn1> UPDATE A SET x = 1;

-- 连接 2：开启事务并锁定 B 表
conn2> BEGIN IMMEDIATE;
conn2> UPDATE B SET y = 1;

-- 连接 1：尝试锁定 B 表（阻塞，等待 conn2）
conn1> UPDATE B SET y = 2;

-- 连接 2：尝试锁定 A 表（阻塞，等待 conn1）
conn2> UPDATE A SET x = 2;

-- 死锁！两个连接互相等待
-- SQLite 会检测到死锁，返回 SQLITE_BUSY
-- 长时间等待后超时返回 SQLITE_BUSY
```

避免死锁的方法：
- 按固定顺序访问表
- 使用短事务
- 设置 `busy_timeout`
- 捕获 `SQLITE_BUSY` 并重试

## 多连接并发策略

### 连接池与线程安全

SQLite 支持多线程，但需要正确配置：

```sql
-- 编译时检查
PRAGMA compile_options;
-- 确保启用了 THREADSAFE

-- 运行时设置
PRAGMA threadsafe = 1;  -- 启用线程安全模式
```

```c
// 多线程使用 SQLite
// 每个线程使用独立的连接
#include <pthread.h>

sqlite3 *create_connection() {
    sqlite3 *db;
    sqlite3_open("myapp.db", &db);
    sqlite3_busy_timeout(db, 5000);
    return db;
}

void *worker_thread(void *arg) {
    sqlite3 *db = create_connection();
    // 使用 db...
    sqlite3_close(db);
    return NULL;
}

int main() {
    pthread_t threads[4];
    for (int i = 0; i < 4; i++) {
        pthread_create(&threads[i], NULL, worker_thread, NULL);
    }
    // 等待线程结束...
}
```

### 并发写入的批处理模式

对于需要频繁写入的场景，批处理可以显著提升性能：

```sql
-- ❌ 低效：每条记录一个事务
BEGIN IMMEDIATE;
INSERT INTO logs VALUES (1, 'msg1'); COMMIT;
BEGIN IMMEDIATE;
INSERT INTO logs VALUES (2, 'msg2'); COMMIT;
BEGIN IMMEDIATE;
INSERT INTO logs VALUES (3, 'msg3'); COMMIT;
-- 每次写入都需要：获取锁 → 写日志 → 刷盘 → 释放锁

-- ✅ 高效：批量写入
BEGIN IMMEDIATE;
INSERT INTO logs VALUES (1, 'msg1');
INSERT INTO logs VALUES (2, 'msg2');
INSERT INTO logs VALUES (3, 'msg3');
COMMIT;
-- 只需一次锁获取和刷盘操作
```

### WAL 模式的并发配置

```sql
-- 推荐配置
PRAGMA journal_mode = WAL;
PRAGMA synchronous = NORMAL;  -- WAL 同步级别
PRAGMA wal_autocheckpoint = 1000;  -- 自动 checkpoint
PRAGMA busy_timeout = 5000;  -- 锁等待超时

-- 限制 WAL 文件大小
-- WAL 文件过大会影响性能
-- checkpoint 可以截断 WAL 文件
PRAGMA wal_checkpoint(TRUNCATE);
```

## 并发问题的排查与解决

### 查看锁状态

```sql
-- 查看当前数据库的锁状态
PRAGMA lock_status;

-- 示例输出：
-- database: Ready
-- writelock: -1 (表示没有写锁)

-- 查看连接信息
SELECT * FROM pragma_database_list;
```

### 定位阻塞问题

```sql
-- 查看哪个连接持有锁
-- 使用 shared cache 模式时
PRAGMA lock_proxy_file;  -- 查看代理文件

-- 诊断长时间持有锁的事务
-- 应用层需要记录：
-- - 事务开始时间
-- - 最后执行 SQL 的时间
-- - 当前 SQL 语句
```

### 常见并发问题

```sql
-- 问题 1：长时间读事务阻塞写入
-- 场景：Web 应用使用 SERIALIZABLE 隔离级别
BEGIN;
-- 读取大量数据（长时间）
SELECT * FROM big_table;
-- 此时写事务被 PENDING 锁阻塞

-- 解决：使用短事务，尽快提交
BEGIN;
SELECT ... LIMIT 100;
-- 处理数据
COMMIT;

-- 问题 2：Checkpoint 阻塞
-- WAL checkpoint 需要 EXCLUSIVE 锁
-- 如果有长时间读事务，checkpoint 会等待

-- 解决：配置 wal_autocheckpoint 或定期手动 checkpoint

-- 问题 3：WAL 文件无限增长
-- 原因：读事务长时间持有页面引用
-- 解决：使用 read_uncommitted 或定期 checkpoint
PRAGMA read_uncommitted = 1;  -- 允许脏读
```

### 优化建议

```sql
-- 1. 使用 WAL 模式
PRAGMA journal_mode = WAL;

-- 2. 设置合理的 busy_timeout
PRAGMA busy_timeout = 5000;

-- 3. 使用批量写入
BEGIN IMMEDIATE;
INSERT INTO ... VALUES (...);
...
COMMIT;

-- 4. 避免长时间读事务
-- 读取时不要做耗时操作
-- 使用 LIMIT 分批处理

-- 5. 定期 checkpoint
PRAGMA wal_checkpoint(TRUNCATE);

-- 6. 使用连接池
-- 为每个线程分配独立连接
```

## 面试要点总结

### 面试官可能会问的问题

**Q1：SQLite 的并发机制是怎样的？**

回答要点：
- SQLite 使用五种锁状态：UNLOCKED、SHARED、RESERVED、PENDING、EXCLUSIVE
- 同一时刻只能有一个写事务
- WAL 模式下读不阻塞写，写不阻塞读
- DELETE/TRUNCATE 模式下写阻塞所有读写

**Q2：SQLite 为什么不适合高并发写入？**

回答要点：
- 同一时刻只能有一个写事务，写操作串行化
- 写事务需要获取 EXCLUSIVE 锁，阻塞所有读写
- 每次写入都需要刷盘（fsync），成为瓶颈
- WAL 模式改进了读并发，但写并发仍然受限

**Q3：SQLite 的 WAL 模式对并发有什么影响？**

回答要点：
- WAL 模式允许读写并发执行
- 多个读事务可以同时进行
- 同一时刻仍然只能有一个写事务
- Checkpoint 操作需要 EXCLUSIVE 锁，可能短暂阻塞

**Q4：如何处理 SQLite 的 SQLITE_BUSY 错误？**

回答要点：
- 设置 `busy_timeout` 让 SQLite 自动等待
- 捕获错误后重试
- 按固定顺序访问资源避免死锁
- 使用短事务减少锁持有时间
