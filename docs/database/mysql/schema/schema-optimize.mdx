# 表结构设计优化

「同样的业务，为什么你的表设计比别人慢 10 倍？」

表结构设计是数据库性能的基础。一个好的表结构能让查询飞速运行，一个烂的表结构会让你在生产环境疯狂救火。本文深入讲解表结构设计的优化技巧和最佳实践。

## 表结构设计原则

### 核心原则

```

│                    表结构设计核心原则                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  1. 字段精简原则                                        │   │
│  │                                                         │   │
│  │  只存储必要字段，避免过度冗余！                         │   │
│  │                                                         │   │
│  │  ✓ 用最少字段表达业务                                   │   │
│  │  ✓ 避免存储可计算得出的字段                            │   │
│  │  ✓ 大字段按需存储（可分离到扩展表）                    │   │
│  └─────────────────────────────────────────────────────────┘   │

│  ┌─────────────────────────────────────────────────────────┐   │
```

### 字符串类型选择

```
┌─────────────────────────────────────────────────────────────────┐
│                    字符串类型对比                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  CHAR vs VARCHAR：                                      │   │
│  │                                                         │   │
│  │  CHAR(n)：                                             │   │
│  │  - 固定长度，存储时用空格补齐                          │   │
│  │  - 适合：定长字符串（手机号、身份证号、邮编）         │   │

│  │                                                         │   │
│  │  VARCHAR(n)：                                           │   │
│  │  - 可变长度，只存储实际内容                            │   │
│  │  - 适合：变长字符串（用户名、地址、描述）             │   │
│  │  - 优点：空间利用率高                                  │   │
│  │  - 缺点：存储时有长度前缀，查询稍慢                    │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  VARCHAR 长度选择：                                    │   │
│  │                                                         │   │
│  │  - 用户名：VARCHAR(32)                                 │   │
│  │  - 邮箱：VARCHAR(100)                                  │   │
│  │  - 手机号：CHAR(11)（固定长度）                        │   │
│  │  - 身份证：CHAR(18)（固定长度）                        │   │
│  │  - 地址：VARCHAR(255)                                  │   │
│  │  - 文章标题：VARCHAR(200)                              │   │
│  │  - 文章内容：TEXT（大字段独立存储）                    │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

```
├─────────────────────────────────────────────────────────────────┤

│  │  类型          字节数    格式                  范围          │   │
│  │  ────────────────────────────────────────────────────    │   │

│  │                                                         │   │
│  │  ✓ 存储日期：DATE（只存日期）或 DATETIME（带时间）     │   │

│  │  - created_at：TIMESTAMP DEFAULT CURRENT_TIMESTAMP    │   │
│  │  - updated_at：TIMESTAMP ON UPDATE CURRENT_TIMESTAMP  │   │
│  │  - 过期时间：BIGINT（存时间戳，跨时区计算方便）       │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 主键设计

### 主键选择原则

```
┌─────────────────────────────────────────────────────────────────┐
│                    主键设计原则                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  主键选择优先级：                                      │   │
│  │                                                         │   │
│  │  1. 自增 BIGINT 主键（默认推荐）                       │   │
│  │     - 优点：查询效率高，插入顺序好                     │   │
│  │     - 适用：几乎所有场景                              │   │
│  │                                                         │   │
│  │  2. 业务主键（如 user_id + order_id）                 │   │
│  │     - 优点：业务可读性好                               │   │
│  │     - 适用：需要唯一标识的业务场景                    │   │
│  │                                                         │   │
│  │  3. 分布式 ID（雪花算法等）                            │   │
│  │     - 优点：跨库唯一，趋势递增                        │   │
│  │     - 适用：分库分表场景                              │   │
│  │                                                         │   │
│  │  4. UUID/GUID（不推荐）                               │   │
│  │     - 缺点：随机写入，索引碎片化                      │   │
│  │     - 适用：几乎不用                                  │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 主键设计对比

```sql
-- 方案一：自增 BIGINT 主键（推荐）
CREATE TABLE orders (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    order_no VARCHAR(32) NOT NULL UNIQUE,
    user_id BIGINT NOT NULL,
    amount DECIMAL(10,2) NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_user_id (user_id),
    INDEX idx_created_at (created_at)
);

-- 方案二：业务复合主键
CREATE TABLE order_items (
    order_id BIGINT NOT NULL,
    product_id BIGINT NOT NULL,
    quantity INT NOT NULL,
    PRIMARY KEY (order_id, product_id)
);
-- 优点：业务可读性好
-- 缺点：InnoDB 使用聚簇索引，主键过大会导致所有索引变大

-- 方案三：分布式 ID（分库分表场景）
CREATE TABLE orders (
    id BIGINT PRIMARY KEY,  -- 使用雪花算法生成
    order_no VARCHAR(32) NOT NULL UNIQUE,
    ...
);
```

## 索引设计

### 索引设计原则

```
┌─────────────────────────────────────────────────────────────────┐
│                    索引设计原则                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  1. 什么字段适合建索引？                                │   │
│  │                                                         │   │
│  │  ✓ WHERE 条件中频繁使用的字段                         │   │
│  │  ✓ JOIN 连接的字段（外键）                           │   │
│  │  ✓ ORDER BY / GROUP BY 的字段                        │   │
│  │  ✓ SELECT 中高频查询的字段（覆盖索引）                │   │
│  │                                                         │   │
│  │  ✗ 不建索引：                                        │   │
│  │  - 区分度低的字段（性别、状态）                      │   │
│  │  - 更新频繁的字段                                    │   │
│  │  - 不在 WHERE 条件中的字段                          │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  2. 复合索引设计原则                                    │   │
│  │                                                         │   │
│  │  索引：(a, b, c)                                     │   │
│  │                                                         │   │
│  │  ✓ 支持查询：a                                       │   │
│  │  ✓ 支持查询：a, b                                     │   │
│  │  ✓ 支持查询：a, b, c                                 │   │
│  │  ✗ 不支持：b, c（跳过 a）                           │   │
│  │  ✗ 不支持：c（跳过 a, b）                           │   │
│  │                                                         │   │
│  │  设计建议：把区分度高的字段放前面                      │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 索引设计案例

```sql
-- 场景：用户表，查询条件通常是 phone 和 email
CREATE TABLE users (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    username VARCHAR(32) NOT NULL,
    phone CHAR(11) NOT NULL,
    email VARCHAR(100) NOT NULL,
    password VARCHAR(64) NOT NULL,
    status TINYINT DEFAULT 1,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uk_phone (phone),
    UNIQUE KEY uk_email (email),
    INDEX idx_status (status)
);

-- 场景：订单表，查询条件是 user_id + status + created_at
CREATE TABLE orders (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    order_no VARCHAR(32) NOT NULL UNIQUE,
    user_id BIGINT NOT NULL,
    status TINYINT NOT NULL DEFAULT 1,
    amount DECIMAL(10,2) NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_user_status_time (user_id, status, created_at)
    -- 覆盖查询：SELECT id, order_no, status, amount, created_at
    --         FROM orders WHERE user_id = ? AND status = ? AND created_at > ?
);

-- 场景：日志表，按时间范围查询
CREATE TABLE access_logs (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id BIGINT,
    action VARCHAR(50) NOT NULL,
    ip VARCHAR(50),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_created_at (created_at)
);
```

## 外键与关联设计

### 外键使用建议

```
┌─────────────────────────────────────────────────────────────────┐
│                    外键使用建议                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  外键的优点：                                            │   │
│  │                                                         │   │
│  │  ✓ 数据完整性保障                                      │   │
│  │  ✓ 级联操作方便                                        │   │

│  │  外键的缺点：                                            │   │
│  │                                                         │   │

│  │  ✗ 分库分表场景不支持                                  │   │

└─────────────────────────────────────────────────────────────────┘
```

### 应用层控制关联

```sql
-- 不使用外键，改为应用层控制

-- 订单表
CREATE TABLE orders (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    order_no VARCHAR(32) NOT NULL UNIQUE,
    user_id BIGINT NOT NULL,  -- 无外键约束
    amount DECIMAL(10,2) NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_user_id (user_id)
);

-- 订单商品表
CREATE TABLE order_items (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    order_id BIGINT NOT NULL,
    product_id BIGINT NOT NULL,
    quantity INT NOT NULL,
    INDEX idx_order_id (order_id),
    INDEX idx_product_id (product_id)
);

-- 应用层代码控制：
-- 1. 插入订单前检查 user_id 是否存在
-- 2. 插入订单商品前检查 order_id 和 product_id 是否存在
-- 3. 删除用户/商品前检查是否有关联订单
```

## 字段设计最佳实践

### 常用字段设计模板

```sql
-- 用户表
CREATE TABLE users (
    id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '用户ID',
    username VARCHAR(32) NOT NULL COMMENT '用户名',
    phone CHAR(11) COMMENT '手机号',
    email VARCHAR(100) COMMENT '邮箱',
    password VARCHAR(64) NOT NULL COMMENT '密码（加密）',
    avatar VARCHAR(255) COMMENT '头像URL',
    status TINYINT DEFAULT 1 COMMENT '状态：1-正常，0-禁用',
    last_login_at TIMESTAMP NULL COMMENT '最后登录时间',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    UNIQUE KEY uk_phone (phone),
    UNIQUE KEY uk_email (email)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户表';

-- 订单表
CREATE TABLE orders (
    id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '订单ID',
    order_no VARCHAR(32) NOT NULL COMMENT '订单号',
    user_id BIGINT NOT NULL COMMENT '用户ID',
    total_amount DECIMAL(10,2) NOT NULL DEFAULT 0 COMMENT '订单总额',
    status TINYINT NOT NULL DEFAULT 1 COMMENT '订单状态：1-待支付，2-已支付，3-已发货，4-已完成，5-已取消',
    pay_type TINYINT COMMENT '支付方式：1-微信，2-支付宝，3-银行卡',
    pay_time TIMESTAMP NULL COMMENT '支付时间',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    UNIQUE KEY uk_order_no (order_no),
    INDEX idx_user_id (user_id),
    INDEX idx_status (status),
    INDEX idx_created_at (created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='订单表';

-- 商品表
CREATE TABLE products (
    id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '商品ID',
    name VARCHAR(100) NOT NULL COMMENT '商品名称',
    category_id BIGINT COMMENT '分类ID',
    price DECIMAL(10,2) NOT NULL DEFAULT 0 COMMENT '价格',
    stock INT NOT NULL DEFAULT 0 COMMENT '库存',
    description TEXT COMMENT '商品描述',
    image VARCHAR(255) COMMENT '商品图片',
    status TINYINT DEFAULT 1 COMMENT '状态：1-上架，0-下架',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    INDEX idx_category_id (category_id),
    INDEX idx_status (status)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='商品表';
```

## 面试实战

### 问题一：如何选择字段类型？

**面试官问**：「MySQL 中如何选择字段类型？有什么原则？」

**深入回答**：

> **选择原则**：
>
> 1. **整数字段**：够用就行
>    - 状态用 TINYINT
>    - 数量用 SMALLINT/INT
>    - ID 用 BIGINT
>
> 2. **字符串字段**：定长用 CHAR，变长用 VARCHAR
>    - 手机号、身份证号：CHAR
>    - 用户名、地址：VARCHAR
>
> 3. **日期字段**：优先 TIMESTAMP
>    - 节省空间，支持时区
>
> 4. **大字段**：考虑分离或对象存储

### 问题二：主键如何设计？

**面试官问**：「主键设计有什么建议？」

**深入回答**：

> **主键设计建议**：
>
> 1. **优先自增 BIGINT**
>    - 查询高效，插入顺序好
>
> 2. **避免业务主键作为聚簇索引**
>    - 业务主键过大，导致所有索引变大
>
> 3. **分库分表用分布式 ID**
>    - 雪花算法、UUID v7 等
>
> 4. **避免 UUID**
>    - 随机写入，索引碎片化

### 问题三：索引如何设计？

**面试官问**：「索引设计有什么技巧？」

**深入回答**：

> **索引设计技巧**：
>
> 1. **高频查询条件建索引**
>    - WHERE、JOIN、ORDER BY、GROUP BY 的字段
>
> 2. **复合索引遵循最左前缀**
>    - (a, b, c) 支持 a、a,b、a,b,c 查询
>
> 3. **区分度高的字段放前面**
>    - 放在复合索引的前面
>
> 4. **覆盖索引减少回表**
>    - SELECT 字段都在索引中
>
> 5. **避免过多索引**
>    - 每个索引都会增加存储和更新开销

## 总结

表结构设计是数据库性能的基础：

- **字段类型**：够用就行，避免浪费
- **主键设计**：优先自增 BIGINT
- **索引设计**：高频查询字段、复合索引、覆盖索引
- **关联设计**：根据场景选择外键或应用层控制
- **命名规范**：统一规范，易于维护

好的表结构设计，让你的系统赢在起跑线上。
