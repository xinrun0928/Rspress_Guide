# 大表优化：分库分表、分区表、历史数据归档

「单表数据量过亿，查询慢得像蜗牛，怎么办？」

大表是数据库性能的天花板。当你的表数据量突破千万级甚至亿级时，常规优化手段已经不够用了。本文深入讲解大表的优化策略，包括分区表、分库分表、历史数据归档等实战方案。

## 分区表

### 什么是分区表？

分区表是将一个表物理拆分成多个小表，但逻辑上仍然是一个表。

```

│  │  │               orders 表                           │  │   │
```

### 分区类型

#### 1. 范围分区（Range）

```sql
-- 按时间范围分区
CREATE TABLE orders (
    order_id BIGINT PRIMARY KEY,
    user_id BIGINT NOT NULL,
    amount DECIMAL(10,2) NOT NULL,
    created_at DATETIME NOT NULL,
    INDEX idx_user_id (user_id),
    INDEX idx_created_at (created_at)
)
PARTITION BY RANGE (YEAR(created_at)) (
    PARTITION p_2023 VALUES LESS THAN (2024),
    PARTITION p_2024 VALUES LESS THAN (2025),
    PARTITION p_2025 VALUES LESS THAN (2026),
    PARTITION p_future VALUES LESS THAN MAXVALUE
);

-- 按 ID 范围分区
CREATE TABLE logs (
    id BIGINT PRIMARY KEY,
    content TEXT,
    created_at DATETIME NOT NULL
)
PARTITION BY RANGE (id) (
    PARTITION p0 VALUES LESS THAN (1000000),
    PARTITION p1 VALUES LESS THAN (2000000),
    PARTITION p2 VALUES LESS THAN (3000000),
    PARTITION p3 VALUES LESS THAN MAXVALUE
);
```

#### 2. 列表分区（List）

```sql
-- 按地区分区
CREATE TABLE customers (
    customer_id BIGINT PRIMARY KEY,
    name VARCHAR(50) NOT NULL,
    region_id INT NOT NULL,
    created_at DATETIME NOT NULL
)
PARTITION BY LIST (region_id) (
    PARTITION p_north VALUES IN (1, 2, 3),
    PARTITION p_south VALUES IN (4, 5, 6),
    PARTITION p_east VALUES IN (7, 8, 9),
    PARTITION p_west VALUES IN (10, 11, 12),
    PARTITION p_other VALUES IN (NULL)
);
```

#### 3. 哈希分区（Hash）

```sql
-- 按哈希分区（自动均匀分布）
CREATE TABLE user_events (
    event_id BIGINT PRIMARY KEY,
    user_id BIGINT NOT NULL,
    event_type VARCHAR(50) NOT NULL,
    created_at DATETIME NOT NULL
)
PARTITION BY HASH (user_id)
PARTITIONS 16;
```

#### 4. 键值分区（Key）

```sql
-- 按主键分区
CREATE TABLE articles (
    id BIGINT PRIMARY KEY,
    title VARCHAR(200) NOT NULL,
    content TEXT,
    created_at DATETIME NOT NULL
)
PARTITION BY KEY(id)
PARTITIONS 8;
```

### 归档实现方案

#### 方案一：基于时间的归档

```sql
-- 创建归档表（与原表结构相同）
CREATE TABLE orders_archive LIKE orders;

-- 归档 1 年前的订单
INSERT INTO orders_archive
SELECT * FROM orders
WHERE created_at < DATE_SUB(NOW(), INTERVAL 1 YEAR);

-- 删除已归档的数据
DELETE FROM orders
WHERE created_at < DATE_SUB(NOW(), INTERVAL 1 YEAR);

-- 添加归档任务调度（使用事件调度器）
DELIMITER //
CREATE EVENT evt_archive_orders
ON SCHEDULE EVERY 1 DAY
STARTS '2024-01-01 00:00:00'
DO
BEGIN
    -- 归档 1 年前数据到归档表
    INSERT IGNORE INTO orders_archive
    SELECT * FROM orders
    WHERE created_at < DATE_SUB(NOW(), INTERVAL 1 YEAR);

    -- 删除已归档数据
    DELETE FROM orders
    WHERE created_at < DATE_SUB(NOW(), INTERVAL 1 YEAR);
END//
DELIMITER ;
```

#### 方案二：批量归档（大数据量）

```sql
-- 分批归档，避免锁表
DELIMITER //
CREATE PROCEDURE archive_orders(IN batch_size INT)
BEGIN
    DECLARE done INT DEFAULT FALSE;
    DECLARE v_max_id BIGINT;
    DECLARE v_min_id BIGINT;
    DECLARE v_cutoff_date DATETIME;

    -- 设置归档截止日期（1年前）
    SET v_cutoff_date = DATE_SUB(NOW(), INTERVAL 1 YEAR);

    -- 获取最小和最大 ID
    SELECT MIN(id), MAX(id) INTO v_min_id, v_max_id
    FROM orders
    WHERE created_at < v_cutoff_date;

    -- 分批处理
    WHILE v_min_id <= v_max_id DO
        -- 归档当前批次
        INSERT INTO orders_archive
        SELECT * FROM orders
        WHERE id >= v_min_id
          AND id < v_min_id + batch_size
          AND created_at < v_cutoff_date;

        -- 删除已归档数据
        DELETE FROM orders
        WHERE id >= v_min_id
          AND id < v_min_id + batch_size
          AND created_at < v_cutoff_date;

        -- 移动到下一批
        SET v_min_id = v_min_id + batch_size;

        -- 释放锁，允许其他操作
        COMMIT;
    END WHILE;
END//
DELIMITER ;

-- 调用存储过程，每次处理 1000 条
CALL archive_orders(1000);
```

## 面试实战

### 问题一：大表优化有哪些手段？

**面试官问**：「大表优化有哪些方案？各自的优缺点？」

**深入回答**：

> **优化手段**：
>
> 1. **SQL 优化**：检查执行计划，添加合适索引
> 2. **表结构优化**：精简字段，冷热分离
> 3. **分区表**：按时间/范围分区，减少查询范围
> 4. **历史数据归档**：归档冷数据，减少主表数据量
> 5. **分库分表**：水平拆分，分布式架构
>
> **选择建议**：
> - 优先尝试索引优化和分区表
> - 分库分表是最后的选择

### 问题二：分区表和分库分表的区别？

**面试官问**：「分区表和分库分表有什么不同？分别适用什么场景？」

**深入回答**：

> **分区表**：
> - 单实例内，逻辑一张表，物理多个分区
> - 优点：实现简单，对应用透明
> - 缺点：单服务器瓶颈
> - 适用：单表数据量大，单机可以承受
>
> **分库分表**：
> - 物理多张表，多个数据库实例
> - 优点：突破单机瓶颈
> - 缺点：实现复杂，跨节点查询困难
> - 适用：数据量极大，单机无法承受

### 问题三：如何设计历史数据归档策略？

**面试官问**：「如何设计历史数据归档？归档后如何查询？」

**深入回答**：

> **归档策略**：
>
> 1. **确定归档条件**：时间（如 1 年前）+ 状态（如已完成）
> 2. **选择归档时机**：定时任务或实时归档
> 3. **批量操作**：分批归档，避免锁表
> 4. **验证完整性**：归档后校验数据一致性
>
> **归档后查询**：
> - 应用层封装 UNION ALL 合并查询
> - 或创建视图屏蔽底层细节

## 总结

大表优化需要循序渐进：

- **第一步**：SQL 优化和索引优化
- **第二步**：分区表（单表优化）
- **第三步**：历史数据归档（减少数据量）
- **第四步**：分库分表（最后手段）

记住：不要过早优化，先用简单的手段解决问题。
