# 大表优化：分库分表、分区表、历史数据归档

凌晨两点，监控告警弹出：数据库 CPU 99%，查询响应时间超过 10 秒。

你登录数据库一看，数据量已经突破 5000 万——这张表是什么时候长这么大的？

「我们当初的设计明明够用三年啊。」

问题是：数据永远比预期增长得快。当单表数据量超过千万级，索引的效率会急剧下降；当突破亿级，连分页查询都会超时。

大表优化不是一道选择题，而是必答题。今天我们从三个方向来拆解这个问题：分区表、分库分表、历史数据归档。

## 分区表：让大表变「小表」

分区表是 MySQL 5.1 引入的特性，本质是把一张表物理上拆成多个「区」（Partition），但对应用层来说还是一张表。

```sql
-- 按时间分区（最常见场景）
CREATE TABLE orders (
    id BIGINT PRIMARY KEY,
    order_no VARCHAR(64) NOT NULL,
    user_id BIGINT NOT NULL,
    amount DECIMAL(10,2),
    status TINYINT,
    create_time DATETIME,
    INDEX idx_user_id (user_id),
    INDEX idx_create_time (create_time)
)
PARTITION BY RANGE (YEAR(create_time)) (
    PARTITION p2022 VALUES LESS THAN (2023),
    PARTITION p2023 VALUES LESS THAN (2024),
    PARTITION p2024 VALUES LESS THAN (2025),
    PARTITION p2025 VALUES LESS THAN (2026),
    PARTITION pmax VALUES LESS THAN MAXVALUE
);
```

### 分区表的工作原理

查询 `2024` 年的订单：

```sql
SELECT * FROM orders WHERE create_time BETWEEN '2024-01-01' AND '2024-12-31';
```

MySQL 只会扫描 `p2024` 这一个分区，而不是全表 5000 万行。这就是分区裁剪（Partition Pruning）。

### 分区表的适用场景

| 场景 | 说明 |
|---|---|
| 时间序列数据 | 订单、日志、访问记录，按时间分区最自然 |
| 需要定期清理历史数据 | `ALTER TABLE orders TRUNCATE PARTITION p2022` 比 `DELETE` 快 100 倍 |
| 大表需要在线DDL | `ALTER TABLE ... REORGANIZE PARTITION` 可以在不锁表的情况下调整分区 |

### 分区表的坑

分区表不是银弹，它有几个天然的局限：

1. **主键必须包含分区列**。如果按 `create_time` 分区，主键必须包含 `create_time`，否则无法创建。

```sql
-- 错误：主键不包含分区列
CREATE TABLE orders (
    id BIGINT PRIMARY KEY,  -- 主键不包含 create_time
    create_time DATETIME,
    ...
)
PARTITION BY RANGE (YEAR(create_time)) ( ... );

-- 正确：主键包含分区列
CREATE TABLE orders (
    id BIGINT NOT NULL,
    create_time DATETIME,
    PRIMARY KEY (id, create_time)  -- 主键包含 create_time
)
PARTITION BY RANGE (YEAR(create_time)) ( ... );
```

2. **分区列必须是表的一部分**，无法按任意列分区。
3. **跨分区查询性能差**。如果查询不带分区条件，MySQL 要扫描所有分区。

## 分库分表：横向扩展的艺术

当分区表也无法承载时，就要考虑分库分表了。

分库分表的核心思想是：**把数据分散到多个数据库实例或多张表中，每个实例/表只负责一部分数据**。

### 分片策略

#### 1. 按 ID 哈希分片

```sql
-- 按 user_id 哈希分表
-- 假设分 4 张表：orders_0, orders_1, orders_2, orders_3
-- 计算方式：user_id % 4

function get_table_name(user_id):
    return "orders_" + (user_id % 4)
```

优点：数据分布均匀
缺点：按时间查询困难，需要跨表聚合

#### 2. 按时间分片

```sql
-- 按月分表
orders_202401
orders_202402
orders_202403
```

优点：符合时间序数据访问模式（查近期多，查远期少）
缺点：冷热数据分布不均

#### 3. 按业务字段分片（推荐）

```sql
-- 按租户分表（多租户 SaaS 场景）
-- 同一个租户的数据在同一个分片，跨租户查询极少

orders_tenant_a
orders_tenant_b
```

### 分库分表的实现方式

#### 方式一：应用层路由（简单场景）

```java
public class OrderDao {
    // 根据 user_id 路由到不同的表
    public Order findByUserId(Long userId, Long orderId) {
        String tableName = "orders_" + (userId % 4);
        String sql = "SELECT * FROM " + tableName + " WHERE id = ?";
        return jdbcTemplate.queryForObject(sql, orderId);
    }
}
```

优点：简单直接，无额外依赖
缺点：跨表查询困难，SQL 编写复杂

#### 方式二：ShardingSphere（推荐）

```yaml
# application.yml
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/orders?useUnicode=true
    driver-class-name: com.mysql.cj.jdbc.Driver

shardingsphere:
  rules:
    sharding:
      tables:
        orders:
          actual-data-nodes: ds_${0..1}.orders_${0..3}
          table-strategy:
            standard:
              sharding-column: user_id
              sharding-algorithm-name: orders_inline
          key-generate-strategy:
            column: id
            key-generator-name: snowflake
      sharding-algorithms:
        orders_inline:
          type: INLINE
          props:
            algorithm-expression: orders_${user_id % 4}
```

```java
// 业务代码完全透明，不需要手动路由
// ShardingSphere 自动处理分片
public Order findById(Long orderId) {
    return orderMapper.selectById(orderId);  // 自动路由到对应分片
}
```

### 分库分表后的问题

分库分表带来了新的复杂性：

| 问题 | 解决方案 |
|---|---|
| 跨分片查询 | ES（Elasticsearch）做搜索层，MySQL 做主存储 |
| 跨分片分页 | 禁止深分页，限制最多查前 N 条 |
| 全局 ID 生成 | Snowflake、Leaf、ULID |
| 数据迁移 | 双写 + 数据校验工具 |
| 分布式事务 | Seata、TXC |

## 历史数据归档：让热数据飞起来

大表里 80% 的数据是从来不访问的「冷数据」，但它们占着空间、拖慢索引。

归档策略：**把冷数据迁移到历史表或历史库，让热数据表保持轻盈**。

### 方案一：定时归档到历史表

```sql
-- 创建历史表（按年分表）
CREATE TABLE orders_history_2022 (...) PARTITION BY RANGE (YEAR(create_time)) ( ... );
CREATE TABLE orders_history_2023 (...) PARTITION BY RANGE (YEAR(create_time)) ( ... );

-- 归档存储过程：每天凌晨执行
DELIMITER //

CREATE PROCEDURE archive_orders()
BEGIN
    DECLARE batch_size INT DEFAULT 1000;
    DECLARE done INT DEFAULT FALSE;
    DECLARE start_date DATE;
    DECLARE end_date DATE;

    -- 归档一年前的数据
    SET start_date = DATE_SUB(CURDATE(), INTERVAL 1 YEAR);
    SET end_date = DATE_SUB(start_date, INTERVAL 1 YEAR);

    -- 分批迁移
    INSERT IGNORE INTO orders_history_2023
    SELECT * FROM orders
    WHERE create_time >= start_date
      AND create_time < end_date
      AND status IN (3, 4, 5);  -- 只归档已完成的订单

    -- 删除已归档的数据
    DELETE FROM orders
    WHERE create_time >= start_date
      AND create_time < end_date
      AND status IN (3, 4, 5);

    -- 重建索引（归档后表数据量变化，统计信息可能不准）
    ANALYZE TABLE orders;
END //

DELIMITER ;
```

### 方案二：归档到 ClickHouse（分析场景）

```sql
-- MySQL 导出
mysqldump -u root -p db_name orders_2022 > orders_2022.sql

-- ClickHouse 建表
CREATE TABLE orders_2022 (
    id Int64,
    order_no String,
    user_id Int64,
    amount Decimal(10,2),
    status Int8,
    create_time DateTime
) ENGINE = MergeTree()
PARTITION BY toYYYYMM(create_time)
ORDER BY (user_id, create_time);
```

### 归档的执行时机

| 时机 | 优点 | 缺点 |
|---|---|---|
| 定时任务（每天凌晨） | 避开业务高峰 | 可能影响备份窗口 |
| 数据写入时异步归档 | 实时保持表轻盈 | 写入延迟增加 |
| 手动触发 | 灵活可控 | 人工操作有风险 |

## 三种方案的对比与选择

| 方案 | 数据量级 | 改造成本 | 适用场景 |
|---|---|---|---|
| 分区表 | `0 ~ 5000 万` | 低 | 单表太大，按时间访问的数据 |
| 分库分表 | `5000 万 ~ 100 亿+` | 高 | 高并发，数据量持续增长 |
| 历史数据归档 | 任意 | 中 | 冷数据占比高，分析场景 |

### 选择建议

- 数据量低于 5000 万，优先考虑分区表，改造成本最低
- 数据量高于 5000 万，QPS 高于 1 万，考虑分库分表
- 无论如何，先把历史数据归档做了，这是投入产出比最高的一步

## 总结：大表优化的三层架构

```
应用层
    ↓
MySQL 分区表（热数据）
    ↓ 历史数据归档
ClickHouse / Hive（温冷数据）
    ↓ 归档
对象存储 / 磁带库（冷数据）
```

大表优化不是一蹴而就的，而是一个持续迭代的过程。先用分区表应急，再用归档腾空间，最后考虑分库分表。每一步都有明确的收益和代价。
