# 表结构设计总览

表结构设计是数据库开发的第一步，也是最关键的一步。设计得好，后续开发事半功倍；设计得差，后续改动代价巨大。

好的表结构设计需要考虑数据完整性、查询效率、扩展性。数据库三范式是理论基础，但在实际项目中也需要灵活运用。

本模块带你掌握表结构设计的原则和技巧。

## 数据库三范式

**第一范式（1NF）**：字段原子性，即每个字段都是不可分割的原子值。例如地址字段应拆分为省、市、区，而不是存成一个字符串。

**第二范式（2NF）**：在满足 1NF 的基础上，消除非主属性对主键的部分依赖。例如订单明细表，以（订单ID，商品ID）为主键，商品名称只依赖商品ID，应单独拆分为商品表。

**第三范式（3NF）**：在满足 2NF 的基础上，消除非主属性对主键的传递依赖。例如学生表有（学号、姓名、班级ID、班级名称），班级名称传递依赖学号，应拆分为班级表。

## 范式 vs 反范式

范式设计的优点是数据冗余少、更新维护简单；缺点是查询时可能需要更多 JOIN。

反范式设计通过适当冗余提升查询性能，但增加了维护成本。

实际项目中，通常采用「满足三范式 + 适度反范式」的策略。

## 大表优化

当单表数据量超过千万级时，需要考虑优化策略：

**索引优化**：确保查询走索引，避免全表扫描。

**分库分表**：水平拆分数据到多张表。

**历史数据归档**：将冷数据迁移到归档表或历史库。

**分区表**：利用 MySQL 分区表特性，将数据按范围或哈希分区。
