# 数据类型选择：让表更小、查询更快

你可能没注意到，但数据类型选择对 MySQL 性能影响巨大。

用一个 `VARCHAR(255)` 存储「是/否」状态，用一个 `BIGINT` 存储 IP 地址，用 `DATETIME` 还是 `TIMESTAMP` 存时间——这些看似微小的选择，累积起来会影响索引效率、内存占用、磁盘 IO。

好的数据类型选择能让表更小、查询更快、维护更简单。本文从原理出发，帮你建立数据类型选择的系统性认知。

## 数据类型选择原则

### 越小越好

数据类型越大，占用的存储空间越多，内存和磁盘的消耗越大。这会直接影响：

- **索引效率**：索引树节点能容纳的索引项更少，树更高，查询 IO 次数更多
- **内存占用**：InnoDB Buffer Pool 以页为单位缓存数据，大的数据类型会让热数据更快被淘汰
- **CPU 消耗**：大类型在计算时需要更多的 CPU 周期

```sql
-- 差的选择：IP 地址用 VARCHAR(45)
CREATE TABLE访问日志 (
    id BIGINT,
    ip_address VARCHAR(45)  -- 浪费空间
);

-- 好的选择：IP 地址用 INT UNSIGNED
CREATE TABLE访问日志 (
    id BIGINT,
    ip_address INT UNSIGNED  -- 4 字节 vs 45 字节
);
```

### 简单就好

简单的数据类型需要的计算更少。比如：

- 用整型存储 IP 地址，而不是字符串
- 用 `YEAR`、`DATE`、`TIME` 存储日期/时间，而不是字符串
- 用定点数（`DECIMAL`）存储金额，而不是浮点数（`FLOAT`、`DOUBLE`）

### 避免 NULL

如果可以，尽量把字段定义为 NOT NULL。NULL 值会让索引、比较、计算都更复杂。

- 索引不存储 NULL 值（除了唯一索引允许多个 NULL）
- NULL 参与比较运算时，结果可能是 UNKNOWN 而不是 TRUE/FALSE
- NULL 在聚合函数中通常被忽略

:::tip 业务字段的 NULL 处理

如果业务上确实需要表示「未知」状态，可以用特殊值代替 NULL：
- 用 `0` 表示「未知用户」
- 用 `-1` 表示「未知状态」
- 用空字符串 `''` 表示「未知内容」

但这需要团队统一约定，避免歧义。
:::

## 整数类型

MySQL 提供多种整数类型，从 TINYINT 到 BIGINT：

| 类型 | 字节 | 有符号范围 | 无符号范围 |
|---|---|---|---|
| TINYINT | 1 | -128 ~ 127 | 0 ~ 255 |
| SMALLINT | 2 | -32768 ~ 32767 | 0 ~ 65535 |
| MEDIUMINT | 3 | -8388608 ~ 8388607 | 0 ~ 16777215 |
| INT | 4 | -21亿 ~ 21亿 | 0 ~ 42亿 |
| BIGINT | 8 | 很大 | 更大 |

### 选择策略

**能用 TINYINT 就不用 SMALLINT，能用 SMALLINT 就不用 INT。**

常见场景：

```sql
-- 状态字段：0=禁用，1=启用
status TINYINT UNSIGNED NOT NULL DEFAULT 0

-- 年龄：0-255 够用
age TINYINT UNSIGNED

-- 用户 ID：需要无符号 INT（42亿够大多数系统用）
user_id INT UNSIGNED NOT NULL AUTO_INCREMENT PRIMARY KEY

-- 订单号：超过 42 亿，用 BIGINT
order_id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT PRIMARY KEY
```

### INT(10) vs INT(1) vs INT

很多人误以为 `INT(10)` 比 `INT(1)` 能存更大的数字。这是错误的。

```sql
INT(10) 和 INT(1) 的取值范围完全相同
```

`INT(N)` 中的 N 只是显示宽度（Display Width），不影响存储范围。只有配合 `ZEROFILL` 时，显示宽度才有意义：

```sql
-- ZEROFILL 会自动设置为 UNSIGNED
-- 存储值 42，显示为 0000000042
ALTER TABLE users ADD COLUMN id INT(10) ZEROFILL;
```

:::warning 不要滥用 ZEROFILL

ZEROFILL 主要用于金融、条码等需要固定宽度的场景。对于普通业务数据，没必要使用。
:::

## 字符串类型

### CHAR vs VARCHAR

| 特性 | CHAR | VARCHAR |
|---|---|---|
| 存储方式 | 固定长度 | 可变长度 |
| 存储空间 | 固定 N 字节 | 实际长度 + 1~2 字节长度前缀 |
| 尾部空格 | 截断 | 保留 |
| 性能 | 写入快（无需计算长度） | 读取需要计算长度 |
| 适用场景 | 长度固定的字符串 | 长度不固定的字符串 |

```
CHAR(10) 存储 'hello'：
实际存储：'hello     '（补足 10 个字符）

VARCHAR(10) 存储 'hello'：
实际存储：'\x05hello'（1 字节长度前缀 + 5 字符）
```

### VARCHAR 的长度选择

VARCHAR 的最大长度受行大小限制（65535 字节），实际可用空间更少（需要存储长度前缀、NULL 标志位）。

```sql
-- VARCHAR 的长度建议
-- 短字符串（长度固定或很短）
name VARCHAR(50)        -- 姓名
phone VARCHAR(20)        -- 电话

-- 中等长度字符串
description VARCHAR(255) -- 简短描述
email VARCHAR(100)       -- 邮箱

-- 长字符串
content VARCHAR(1000)     -- 文章摘要
-- 如果可能超过 1000，改用 TEXT

-- 固定长度用 CHAR
country_code CHAR(2)     -- 国家代码（如 CN、US）
```

### TEXT vs VARCHAR

TEXT 类型用于存储大文本，MySQL 有多种 TEXT：

| 类型 | 最大长度 | 存储方式 |
|---|---|---|
| TINYTEXT | 255 字节 | 不存储在行内 |
| TEXT | 64KB | 不存储在行内 |
| MEDIUMTEXT | 16MB | 不存储在行内 |
| LONGTEXT | 4GB | 不存储在行内 |

:::danger TEXT 的问题

1. **不能有默认值**：TEXT 字段不能有 DEFAULT 值
2. **不支持索引全长度**：索引最多前 767 字节（InnoDB）
3. **不能用于某些函数**：如 `ORDER BY`、`GROUP BY`
4. **存储位置**：TEXT 数据不存储在行内，会导致额外的 IO

如果 VARCHAR(1000) 够用，就不要用 TEXT。
:::

### BLOB vs VARCHAR

BLOB 用于存储二进制数据（如图片、文件），与 TEXT 对应：

- TINYBLOB / TINYTEXT
- BLOB / TEXT
- MEDIUMBLOB / MEDIUMTEXT
- LONGBLOB / LONGTEXT

如果可以，把文件存储到对象存储（如 OSS、S3），数据库只存储文件 URL。

## 日期时间类型

### DATETIME vs TIMESTAMP

| 特性 | DATETIME | TIMESTAMP |
|---|---|---|
| 存储空间 | 8 字节 | 4 字节 |
| 时区 | 不存储时区 | 自动转换时区 |
| 范围 | '1000-01-01' ~ '9999-12-31' | '1970-01-01' ~ '2038-01-19' |
| 更新 | 默认不自动更新 | 可以自动更新为当前时间 |

```sql
-- DATETIME：存储完整日期时间，不受时区影响
created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP

-- TIMESTAMP：自动处理时区转换
created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP

-- 自动更新
updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
```

### 选择建议

| 场景 | 推荐类型 |
|---|---|
| 存储精确时间点（记录创建/修改） | TIMESTAMP |
| 跨时区业务（用户出生日期、会议时间） | DATETIME |
| 需要 BIGINT + 自己处理时间戳 | BIGINT |
| 只存储日期（生日） | DATE |
| 只存储时间（开门时间） | TIME |

:::tip 时间戳的优缺点

有些项目喜欢用 BIGINT 存储时间戳（Unix 时间戳）：

```sql
created_at BIGINT UNSIGNED NOT NULL DEFAULT UNIX_TIMESTAMP()
```

**优点**：
- 与 Java、JavaScript 等语言的时间戳兼容
- 可以做数学运算（加减天数等）

**缺点**：
- 可读性差，需要转换
- 2038 年问题（32 位时间戳）
- 时区处理需要额外代码

建议：除非有特殊需求，否则用 DATETIME 或 TIMESTAMP。
:::

## 浮点数类型

### FLOAT vs DOUBLE vs DECIMAL

| 类型 | 精度 | 存储空间 | 说明 |
|---|---|---|---|
| FLOAT | 单精度 | 4 字节 | 约 7 位有效数字 |
| DOUBLE | 双精度 | 8 字节 | 约 15 位有效数字 |
| DECIMAL | 用户定义 | 变长 | 精确小数 |

```sql
-- 浮点数：不精确
CREATE TABLE test_float (
    a FLOAT(10, 2),  -- 10 位总精度，2 位小数
    b DOUBLE(10, 2)
);

INSERT INTO test_float VALUES (1.1, 1.1);
SELECT * FROM test_float WHERE a = 1.1; -- 可能查不到！
```

```sql
-- 定点数：精确
CREATE TABLE test_decimal (
    a DECIMAL(10, 2),  -- 10 位总精度，2 位小数
    b DECIMAL(10, 2)
);

INSERT INTO test_decimal VALUES (1.1, 1.1);
SELECT * FROM test_decimal WHERE a = 1.1; -- 能查到
```

### 金额必须用 DECIMAL

金融相关数据（金额、利率、汇率）**必须用 DECIMAL**，不能用 FLOAT/DOUBLE。

```sql
-- 错误：金额用 DOUBLE
price DOUBLE(10, 2)  -- 可能出现 99.999999 的情况

-- 正确：金额用 DECIMAL
price DECIMAL(10, 2)  -- 精确存储 99.99
```

## 枚举和集合

### ENUM

ENUM 用于存储预定义的值列表，MySQL 会将其优化为数字存储。

```sql
CREATE TABLE orders (
    status ENUM('待支付', '已支付', '已取消', '已完成')
);

-- 优势：存储空间小（1-2 字节），查询效率高
-- 劣势：修改枚举值需要 ALTER TABLE，跨版本迁移麻烦
```

### SET

SET 用于存储多个预定义值的组合。

```sql
CREATE TABLE user_roles (
    roles SET('管理员', '编辑', '审核', '普通用户')
);

-- 存储：可以存储多个值
INSERT INTO user_roles VALUES ('管理员,编辑');
```

:::warning ENUM 和 SET 的陷阱

1. **跨数据库迁移**：ENUM/SET 的值与数据库紧耦合，不便于数据迁移
2. **ALTER TABLE**：添加/删除枚举值会锁表
3. **可读性**：代码中需要硬编码枚举值

对于需要频繁修改的值，建议用 INT + 字典表的方式。
:::

## 位字段类型

BIT 用于存储位标志。

```sql
CREATE TABLE flags (
    flags BIT(8)  -- 8 位，1 字节
);

INSERT INTO flags VALUES (b'00101010');
```

:::warning BIT 的问题

1. MySQL 客户端默认把 BIT 当作二进制显示，不直观
2. 与编程语言交互时可能有类型转换问题

除非存储紧凑的位标志，否则用 TINYINT 更方便。
:::

## 实战选择指南

### 常见字段类型参考

| 业务场景 | 推荐类型 | 不推荐 |
|---|---|---|
| 用户 ID | INT UNSIGNED / BIGINT | VARCHAR |
| 状态字段 | TINYINT UNSIGNED | VARCHAR |
| 年龄 | TINYINT UNSIGNED | INT |
| 金额 | DECIMAL(10,2) | FLOAT/DOUBLE |
| 手机号 | VARCHAR(20) | BIGINT |
| IP 地址 | INT UNSIGNED | VARCHAR |
| Email | VARCHAR(100) | TEXT |
| 文章内容 | TEXT | VARCHAR(1000) |
| 日期时间 | DATETIME | VARCHAR |
| 性别 | TINYINT（0/1） | ENUM |

### IP 地址存储

```sql
-- 推荐：INT UNSIGNED + 转换函数
CREATE TABLE access_logs (
    ip_address INT UNSIGNED NOT NULL
);

-- 插入：使用 INET_ATON
INSERT INTO access_logs VALUES (INET_ATON('192.168.1.100'));

-- 查询：使用 INET_NTOA
SELECT INET_NTOA(ip_address) FROM access_logs;
```

### 手机号存储

```sql
-- 手机号存储为 VARCHAR
-- 原因：可能带 +86 前缀，有 + 特殊字符，超过 INT 范围
phone VARCHAR(20) NOT NULL

-- 如果确定只存国内手机号且不带前缀
phone CHAR(11) NOT NULL  -- 比 VARCHAR 省空间
```

### 时间存储

```sql
-- 记录创建时间
created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP

-- 记录更新时间（自动更新）
updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP

-- 存储 Unix 时间戳（Java 可直接使用）
created_at BIGINT UNSIGNED NOT NULL DEFAULT UNIX_TIMESTAMP()
```

## 总结

数据类型选择的核心原则：

1. **越小越好**：能 TINYINT 不 INT，能 INT 不 BIGINT
2. **简单就好**：用整型存储数字，用 DATETIME 存储时间
3. **避免 NULL**：没有业务意义就用 NOT NULL
4. **金额用 DECIMAL**：金融数据必须精确
5. **合适的字符串长度**：VARCHAR 不要过长也不要过短

好的数据类型设计，能让表更小、索引更高效、查询更快。
