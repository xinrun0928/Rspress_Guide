# SQL 执行过程：一条查询的完整旅程

你执行了一条 SELECT 语句，MySQL 返回了结果。但在这背后，MySQL 经历了什么？

从连接建立到语法解析，从优化器生成执行计划到存储引擎读取数据，每一步都可能影响查询性能。理解 SQL 的执行过程，是排查慢查询、优化 SQL 的基础。

本文带你完整走一遍 SQL 的执行流程，理解每一环节的作用。

## SQL 执行的整体流程

```
客户端
    │
    │  1. 连接建立
    ▼
MySQL Server
    │
    │  2. 查询缓存（MySQL 8.0 之前）
    ▼
    │  3. 解析器：词法分析 + 语法分析
    ▼
    │  4. 预处理器：语义检查
    ▼
    │  5. 优化器：生成执行计划
    ▼
    │  6. 执行器：调用存储引擎
    ▼
存储引擎（InnoDB）
    │
    │  7. 读取数据（内存/磁盘）
    ▼
MySQL Server
    │
    │  8. 返回结果给客户端
    ▼
客户端
```

## 1. 连接建立

当你执行 `mysql -h localhost -u root -p` 时，TCP 握手、身份验证、线程分配都发生在这个阶段。

```sql
-- 查看当前连接
SHOW PROCESSLIST;

-- 结果示例
+----+------+-----------+------+---------+------+----------+------------------+
| Id | User | Host      | db   | Command | Time | State    | Info             |
+----+------+-----------+------+---------+------+----------+------------------+
|  5 | root | localhost | test | Sleep   |  120 |          | NULL             |
|  6 | root | localhost | test | Query   |    0 | starting | SHOW PROCESSLIST |
+----+------+-----------+------+---------+------+----------+------------------+
```

**连接池**（HikariCP、Druid）在应用层管理连接复用，避免频繁创建销毁连接。连接池参数（最大连接数、空闲超时等）对应用性能有重要影响。

```java
// HikariCP 连接池配置示例
HikariConfig config = new HikariConfig();
config.setMaximumPoolSize(20);      // 最大连接数
config.setMinimumIdle(5);           // 最小空闲连接
config.setConnectionTimeout(30000); // 获取连接超时
config.setIdleTimeout(600000);     // 空闲超时
```

## 2. 查询缓存（MySQL 8.0 已移除）

:::warning Query Cache 已移除

MySQL 8.0 移除了查询缓存功能。在 8.0 之前的版本中，MySQL 会缓存 SQL 及其结果，相同的查询可以直接从缓存返回。

移除原因：查询缓存在高并发写入场景下维护成本高——任何对表的修改都会使该表的所有缓存失效，容易成为性能瓶颈。
:::

## 3. 解析器：词法分析和语法分析

解析器将 SQL 字符串解析成一颗「抽象语法树」（AST）。

### 词法分析

词法分析器把 SQL 字符串拆分成 token（标记）。

```
SELECT id, name FROM users WHERE age > 18 ORDER BY id LIMIT 10;
```

拆分结果：

```
SELECT    -- 关键字
id        -- 列名
,         -- 逗号
name      -- 列名
FROM      -- 关键字
users     -- 表名
WHERE     -- 关键字
age       -- 列名
>         -- 操作符
18        -- 常量
ORDER BY  -- 关键字
id        -- 列名
LIMIT     -- 关键字
10        -- 常量
;         -- 语句结束
```

### 语法分析

语法分析器根据语法规则，把 token 组装成 AST。

```
                    ┌─────────────┐
                    │ SELECTStmt │
                    └──────┬──────┘
                           │
        ┌──────────────────┼──────────────────┐
        │                  │                  │
   ┌────▼────┐      ┌──────▼──────┐    ┌──────▼──────┐
   │  COLUMNS │      │    FROM     │    │    WHERE    │
   │ [id,name]│      │   users    │    │  age > 18   │
   └─────────┘      └─────────────┘    └─────────────┘
```

如果 SQL 有语法错误（如 `SELETC`），解析器会报错：

```
ERROR 1064 (42000): You have an error in your SQL syntax
```

## 4. 预处理器：语义检查

解析器只检查语法，不检查语义。预处理器负责：

1. **表和列存在性检查**：FROM 的表、SELECT 的列必须存在
2. **权限检查**：当前用户是否有权执行这个操作
3. **别名解析**：把列别名解析为实际列名

```sql
-- 预处理器检查
SELECT name, age FROM users;  -- 检查 users 表是否存在，name、age 列是否存在

-- 检查通过后才进入优化器
```

## 5. 优化器：生成执行计划

优化器是 MySQL 最复杂的组件。它根据统计信息，为每条查询生成最优的执行计划。

### 统计信息

InnoDB 维护表的统计信息，包括：

| 统计项 | 说明 |
|---|---|
| 表的行数 | 用于估算扫描成本 |
| 索引的基数 | 每个值的平均记录数 |
| 索引的长度 | 影响索引页能容纳的项数 |
| 数据分布 | 用于估算条件过滤后的行数 |

```sql
-- 查看表统计信息
SHOW TABLE STATUS FROM test LIKE 'users';

-- 查看索引统计信息
SHOW INDEX FROM users;
```

统计信息可能过期，导致优化器选错执行计划。可以手动刷新：

```sql
-- 重新计算统计信息
ANALYZE TABLE users;
```

### 索引选择

同样的查询，可能有多个索引可用。优化器根据成本估算选择最优的。

```sql
-- users 表有 idx_name、idx_age 两个索引
EXPLAIN SELECT * FROM users WHERE name = '张三' AND age = 25;
```

优化器会计算每个索引的成本：

- 索引扫描成本 = 读取索引页成本 + 回表成本
- 全表扫描成本 = 读取数据页成本

最终选择成本最低的索引。

### JOIN 顺序优化

多表 JOIN 时，表的顺序对性能影响巨大。

```sql
SELECT * FROM orders o
JOIN users u ON o.user_id = u.id
JOIN products p ON o.product_id = p.id
WHERE u.city = '北京' AND p.category = '电子产品';
```

优化器会尝试不同的 JOIN 顺序，计算每种顺序的成本。选择让中间结果集最小的顺序作为最优执行计划。

```sql
-- 执行计划可能显示：
-- 1. 先用 idx_city 找到北京用户
-- 2. 再关联 orders
-- 3. 最后关联 products
```

### 强制指定执行计划

如果优化器的选择不是你想要的，可以用 Hint 强制指定：

```sql
-- 强制使用某个索引
SELECT * FROM users FORCE INDEX (idx_name) WHERE name = '张三';

-- 忽略某个索引
SELECT * FROM users IGNORE INDEX (idx_name) WHERE name = '张三';
```

## 6. 执行器：调用存储引擎

执行器根据优化器生成的执行计划，调用存储引擎接口获取数据。

### SELECT 执行流程

```sql
SELECT * FROM users WHERE age > 20 LIMIT 10;
```

```
执行器
    │
    ▼
调用 InnoDB：读取第一行
    │
    ▼
InnoDB：检查 age > 20？
    │
    ├── 是 → 返回行给执行器
    ├── 否 → 继续读取下一行
    │
    ▼
执行器：检查是否 < LIMIT 10
    │
    ├── 是 → 加入结果集
    ├── 否 → 继续读取
    │
    ▼
返回结果集给客户端
```

### DML 执行流程

```sql
UPDATE users SET age = 30 WHERE id = 1;
```

```
执行器
    │
    ▼
调用 InnoDB：读取 id=1 的行
    │
    ▼
InnoDB：加行锁，返回数据
    │
    ▼
执行器：修改 age = 30
    │
    ▼
写入 Undo Log（用于回滚）
    │
    ▼
调用 InnoDB：写入新数据
    │
    ▼
写入 Redo Log Buffer
    │
    ▼
返回影响的行数给客户端
```

## 7. 存储引擎读取数据

InnoDB 读取数据时，优先从 Buffer Pool（内存）读取，如果不存在则从磁盘加载。

```
查询请求
    │
    ▼
Buffer Pool 中有数据页？
    │
    ├── 有 → 直接返回（内存操作，微秒级）
    ├── 没有 → 从磁盘读取到 Buffer Pool，再返回（磁盘 IO，毫秒级）
    │
    ▼
检查数据是否满足 WHERE 条件
    │
    ▼
返回符合条件的行
```

### Buffer Pool 的作用

Buffer Pool 缓存热数据页，减少磁盘 IO。Buffer Pool 采用 LRU 算法淘汰冷数据。

```
Buffer Pool LRU 链表：
┌─────────────────────────────────────────┐
│  Young 区（热数据）│ Old 区（新读入）     │
│  ████████████████ │ ░░░░░░░░░░░░░░░░░    │
└─────────────────────────────────────────┘
  最近访问的页              新读入的页
```

InnoDB 的 LRU 做了优化——新读入的页先放入 Old 区，避免全表扫描污染热数据区。

## 8. 返回结果给客户端

执行完成后，结果通过 MySQL 协议返回客户端。

```
MySQL 协议返回过程：
1. 服务端发送数据包头
2. 分包发送数据行（如结果集很大）
3. 发送结束包（EOF）
```

应用层通过 JDBC/ODBC 接收结果，结果集在内存中处理。

## SQL 执行顺序

虽然 SQL 书写的顺序是：

```sql
SELECT ...          -- 5. 选择列
FROM ...            -- 1. 确定数据源
WHERE ...            -- 2. 过滤行
GROUP BY ...         -- 3. 分组
HAVING ...           -- 4. 过滤分组
ORDER BY ...         -- 6. 排序
LIMIT ...            -- 7. 限制行数
```

但 MySQL 内部执行时，优化器可能会调整顺序（特别是 GROUP BY 和 ORDER BY）。关键理解是：SQL 是声明式语言，告诉 MySQL「要什么」，而不是「怎么做」。

## 慢查询的原因分析

了解了执行流程，我们可以分析慢查询的原因：

| 执行阶段 | 可能问题 |
|---|---|
| 连接建立 | 连接数打满、连接超时 |
| 解析器 | 复杂 SQL 解析耗时（少见） |
| 优化器 | 统计信息过期、选错索引 |
| 存储引擎 | 磁盘 IO 慢、数据不在 Buffer Pool |
| 返回结果 | 结果集太大、网络慢 |

排查慢查询时，通常关注的是：

1. **是否走了索引**：`EXPLAIN` 查看 type、key
2. **扫描了多少行**：`EXPLAIN` 查看 rows
3. **是否需要排序/临时表**：`EXPLAIN` 查看 Extra

## 总结

SQL 执行的核心流程：

1. **连接建立**：TCP 握手、身份验证
2. **解析器**：词法分析、语法分析，生成 AST
3. **预处理器**：语义检查、权限检查
4. **优化器**：生成最优执行计划
5. **执行器**：按计划调用存储引擎
6. **存储引擎**：从 Buffer Pool 或磁盘读取数据
7. **返回结果**：通过 MySQL 协议返回客户端

理解这套流程，你就能看懂 `EXPLAIN` 的输出，知道慢查询慢在哪里，也知道该从哪个环节优化。
