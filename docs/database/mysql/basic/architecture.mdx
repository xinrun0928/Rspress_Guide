# MySQL 整体架构：从连接到存储的完整链路

你能回答这个问题吗——「一条 SQL 语句在 MySQL 中是如何执行的？」

很多人会说：「先解析 SQL，再执行查询。」这个答案只对了一半。MySQL 的架构远比这复杂——连接层要处理认证和线程复用，服务层有解析器、优化器、缓存，存储引擎层才是真正读写数据的地方。理解这套架构，是掌握 MySQL 的第一步。

## MySQL 架构全景图

MySQL 采用分层架构，每一层有明确的职责分工：

```
┌─────────────────────────────────────────────────────────────────┐
│                         客户端层                                 │
│   JDBC/ODBC / CLI / 连接池（HikariCP、Druid）                   │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                         连接层                                   │
│   连接管理器 │ 线程管理器 │ 连接池 │ 认证模块                     │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                         服务层                                   │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────────┐   │
│  │ SQL 接口  │  │  解析器   │  │  优化器   │  │   缓存模块   │   │
│  └──────────┘  └──────────┘  └──────────┘  └──────────────┘   │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐                      │
│  │  预处理器  │  │  执行器   │  │ 日志管理  │                      │
│  └──────────┘  └──────────┘  └──────────┘                      │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                        存储引擎层                                │
│   InnoDB（默认）│ MyISAM │ Memory │ Archive │ ...              │
└─────────────────────────────────────────────────────────────────┘
```

## 连接层：客户端与服务端的第一次握手

当你执行 `mysql -h localhost -u root -p` 时，连接层就开始工作了。

**连接管理器**负责接收客户端的连接请求。每个连接，MySQL 都会分配一个独立的线程来处理。这个线程负责与客户端通信——接收 SQL、返回结果。

**线程管理器**维护线程池，避免为每个连接都创建和销毁线程。MySQL 5.5 引入了线程池插件，对于大量短连接的场景，性能提升明显。

**连接池**通常不在 MySQL 内部实现，而是在应用层。HikariCP、Druid 等连接池管理数据库连接的创建、复用、销毁。合理配置连接池参数（如最大连接数、空闲超时）对应用性能至关重要。

**认证模块**验证用户名、密码，检查用户权限。MySQL 的用户认证基于主机名和用户名组合，同样的用户名从不同主机连接，可能拥有不同的权限。

:::tip 连接层的常见问题

连接数打满是生产环境中常见的问题。`SHOW PROCESSLIST` 可以查看当前所有连接，`SHOW STATUS LIKE 'Threads_connected'` 查看连接数。解决方案通常是增加 max_connections 或优化慢查询释放连接。
:::

## 服务层：MySQL 的大脑

服务层是 MySQL 的核心，包含多个关键组件。

### SQL 接口

SQL 接口负责接收 SQL 语句，将它们传递给其他组件处理。它支持多种 SQL 语法，包括 DML（SELECT、INSERT、UPDATE、DELETE）、DDL（CREATE、DROP、ALTER）、DCL（GRANT、REVOKE）等。

### 解析器：把 SQL 变成语法树

解析器分为两部分：**词法分析**和**语法分析**。

**词法分析**把 SQL 字符串拆分成一个个 token（标记）。比如 `SELECT id FROM users WHERE age > 18` 会被拆成 `SELECT`、`id`、`FROM`、`users`、`WHERE`、`age`、`>`、`18` 这些 token。

**语法分析**根据 MySQL 的语法规则，把 token 组装成一颗语法树（AST）。这棵树描述了 SQL 的结构——哪些是列名、哪些是表名、WHERE 条件是什么。

```sql
-- 这条 SQL 会被解析成 AST
SELECT id, name FROM users WHERE age > 18 ORDER BY id LIMIT 10;
```

```
┌─────────────┐
│  SELECTStmt │（根节点）
├─────────────┤
│ columns:    │ id, name
│ from:       │ users
│ where:      │ age > 18
│ orderBy:    │ id
│ limit:      │ 10
└─────────────┘
```

如果 SQL 有语法错误，解析器会报错。比如 `SELECTT`（多了个 T）会直接被解析器拒绝。

### 预处理器：语义检查

解析器只检查语法，不检查语义。预处理器负责：

1. **检查表名和列名是否存在**：FROM 的表、SELECT 的列必须在数据库中存在
2. **检查权限**：当前用户是否有权执行这个操作
3. **解析别名**：把列的别名解析成实际的列

### 查询缓存：8.0 之前的重要组件

:::warning Query Cache 已移除

MySQL 8.0 移除了 Query Cache。在 8.0 之前的版本中，MySQL 会缓存 SQL 及其结果，相同的查询可以直接从缓存返回，无需再次解析和执行。

移除的原因：Query Cache 在高并发写入场景下维护成本高，容易成为性能瓶颈。
:::

### 优化器：生成最优执行计划

优化器是 MySQL 最复杂的组件之一。它接收解析后的 SQL，根据多种因素生成最优的执行计划。

**优化器考虑的因素**：

1. **统计信息**：表有多少行、索引的基数（有多少不同的值）、数据分布
2. **索引选择**：同样的查询可能有多条路径，优化器选择代价最低的
3. **JOIN 顺序**：多表 JOIN 时，不同的顺序性能差异巨大
4. **成本估算**：计算每种执行计划的 IO 成本、CPU 成本

**优化器的局限**：

优化器不是万能的。它基于统计信息做决策，但统计信息可能过期。对于复杂的查询，优化器可能选错索引。可以用 `EXPLAIN` 查看优化器的决策，用 `FORCE INDEX` 强制使用某个索引。

```sql
-- 查看优化器的执行计划
EXPLAIN SELECT * FROM users WHERE name = '张三';

-- 强制使用某个索引
SELECT * FROM users FORCE INDEX (idx_name) WHERE name = '张三';
```

### 执行器：按计划执行

执行器根据优化器生成的执行计划，调用存储引擎的接口获取数据。

对于 `SELECT` 操作：

1. 调用存储引擎的接口读取第一行
2. 判断是否满足 WHERE 条件
3. 如果满足，加入结果集
4. 重复直到取完所有行

对于 `INSERT/UPDATE/DELETE`：

1. 调用存储引擎读取数据
2. 记录修改前的内容到 Undo Log
3. 修改数据
4. 记录修改后的内容到 Redo Log

## 存储引擎层：数据的物理存储

存储引擎层是 MySQL 架构中真正读写数据的地方。MySQL 采用插件式架构，允许在运行时切换存储引擎。

### InnoDB 存储引擎

InnoDB 是 MySQL 5.5.5 之后的默认存储引擎，也是目前生产环境使用最多的引擎。

**核心特性**：

- **事务支持**：完整支持 ACID 事务
- **行级锁**：支持高并发读写
- **MVCC**：多版本并发控制
- **聚簇索引**：数据按主键顺序存储
- **外键约束**：支持参照完整性
- **崩溃恢复**：通过 Redo Log 自动恢复

### MyISAM 存储引擎

MyISAM 是早期 MySQL 的默认引擎，目前仍有使用场景。

**核心特性**：

- **表级锁**：不支持行锁，并发性能差
- **不支持事务**：没有提交和回滚
- **非聚簇索引**：索引和数据分开存储
- **全文索引**：较早支持文本搜索
- **表压缩**：支持静态表压缩

**适用场景**：

- 只读的数据仓库
- 大量 `SELECT` 少有 `UPDATE` 的场景
- 需要全文索引但不需要事务的场景

### Memory 存储引擎

Memory 引擎把数据存储在内存中，访问速度极快。

**核心特性**：

- **数据易失**：服务器重启后数据丢失
- **表级锁**：并发写入性能差
- **固定长度**：所有字段长度固定，不支持 BLOB、TEXT
- **哈希索引**：默认使用哈希索引，也支持 B-Tree 索引

**适用场景**：

- 临时表、缓存表
- 查找表（配置表、字典表）
- 内存充足、数据量不大的场景

## SQL 执行流程详解

理解了 MySQL 的三层架构，我们来看一条 SELECT 语句的完整执行流程：

```
客户端
  │
  ▼
连接层：建立 TCP 连接，验证用户身份，分配线程
  │
  ▼
服务层-解析器：词法分析、语法分析，生成 AST
  │
  ▼
服务层-预处理器：语义检查，检查表/列是否存在
  │
  ▼
服务层-优化器：生成最优执行计划
  │
  ▼
服务层-执行器：调用存储引擎接口执行计划
  │
  ▼
存储引擎层：从磁盘读取数据（内存/磁盘）
  │
  ▼
结果返回客户端
```

### UPDATE 语句的执行流程

UPDATE 语句稍有不同，需要记录日志保证事务：

```
连接层
  │
  ▼
解析器、优化器
  │
  ▼
执行器调用存储引擎
  │
  ├── 读取数据到内存
  ├── 修改数据
  ├── 写入 Undo Log（用于回滚）
  ├── 写入 Redo Log Buffer
  └── 返回执行结果
  │
  ▼
事务提交时：
  ├── Redo Log Buffer 刷盘
  └── Binlog 刷盘
  │
  ▼
事务提交完成
```

## 面试实战

### 场景一：SQL 执行顺序

**面试官问**：「SELECT 语句的执行顺序是什么？」

回答要点：

> SQL 的执行顺序是：FROM -> WHERE -> GROUP BY -> HAVING -> SELECT -> ORDER BY -> LIMIT。
>
> 具体来说：
>
> 1. FROM：确定数据来源
> 2. WHERE：过滤不满足条件的行
> 3. GROUP BY：按列分组
> 4. HAVING：过滤分组
> 5. SELECT：选择需要的列
> 6. ORDER BY：排序
> 7. LIMIT：限制返回行数
>
> 需要注意的是，虽然 SQL 是这样书写的，但 MySQL 内部执行时，优化器会调整顺序以提高性能。

### 场景二：优化器为什么不选索引

**面试官问**：「你的查询明明有索引，但 EXPLAIN 显示没走索引，这是为什么？」

回答要点：

> 优化器选择索引基于成本估算，可能的原因有：
>
> 1. **统计信息过期**：`ANALYZE TABLE` 重新统计
> 2. **查询返回数据太多**：全表扫描反而更快
> 3. **索引区分度低**：比如性别字段，只有男女两个值，走索引反而更慢
> 4. **查询条件不满足最左前缀**：联合索引无法使用
>
> 解决方案：用 `FORCE INDEX` 强制指定，或检查统计信息。

### 场景三：连接池打满

**面试官问**：「线上数据库连接池打满，你怎么排查？」

回答要点：

> 排查步骤：
>
> 1. 先用 `SHOW PROCESSLIST` 看哪些连接在干什么
> 2. 如果有 `Sleep` 状态的长时间连接，可能是连接没有正确关闭
> 3. 如果有大量查询中的连接，检查是否有慢查询
> 4. 如果连接数确实不够，考虑：
>    - 调大连接池大小
>    - 优化慢查询，释放连接
>    - 检查是否连接泄漏（获取连接后没有释放）
>    - 考虑读写分离，减轻主库压力

## 总结

MySQL 的三层架构各有职责：

- **连接层**：负责连接管理、认证、线程分配
- **服务层**：SQL 解析、优化、执行，包含 MySQL 的大部分智能
- **存储引擎层**：负责数据的物理存储和提取

理解这套架构，能帮助你在排查问题时快速定位——是连接数不够，还是查询太慢，还是存储引擎 IO 瓶颈？
