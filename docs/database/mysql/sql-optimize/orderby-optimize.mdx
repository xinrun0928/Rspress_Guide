# ORDER BY 优化：让排序飞起来

「为什么加了索引 ORDER BY 还是慢？」

ORDER BY 是 SQL 中最常见的操作之一，但处理不当会导致严重的性能问题。本文深入剖析 ORDER BY 的优化技巧。

## ORDER BY 原理

### MySQL 排序方式

```
┌─────────────────────────────────────────────────────────────────┐
│                    MySQL 排序方式                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  1. 使用索引排序（Index Sort）                          │   │
│  │     条件：ORDER BY 列在索引中，且顺序匹配               │   │
│  │     性能：最优，无需额外排序                            │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  2. 使用文件排序（Filesort）                             │   │
│  │     条件：无法使用索引排序                               │   │
│  │     性能：较差，需要额外排序                            │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### Filesort 算法

```
┌─────────────────────────────────────────────────────────────────┐
│                    Filesort 算法                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  MySQL 8.0 之前的算法：                                          │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  两次扫描算法（Two-pass）：                               │   │
│  │  1. 读取行指针和 ORDER BY 列                           │   │
│  │  2. 在内存中排序                                        │   │
│  │  3. 根据指针读取完整行                                  │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  MySQL 8.0+ 的算法：                                            │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  单次扫描算法（Single-pass）：                           │   │
│  │  1. 读取所有需要的列                                    │   │
│  │  2. 在内存中排序                                        │   │
│  │  3. 直接返回结果                                        │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## ORDER BY 性能问题

### 无索引排序

```sql
EXPLAIN SELECT * FROM orders ORDER BY create_time;
```

```
+----+-------------+--------+------+------+--------+------+----------+-------+
| id | select_type| table  | type | key  |key_len | ref  | rows     | Extra |
+----+-------------+--------+------+------+--------+------+----------+-------+
|  1 | SIMPLE     | orders | ALL  | NULL |  NULL | NULL | 1000000  |Using filesort|
+----+-------------+--------+------+------+--------+------+----------+-------+
-- 全表扫描 + 文件排序
```

### 有索引但不匹配

```sql
-- 索引：(status, create_time)
CREATE INDEX idx_status_time ON orders(status, create_time);

-- 查询
EXPLAIN SELECT * FROM orders WHERE status = 'pending' ORDER BY create_time;
```

```
+----+-------------+--------+------+---------------+------+---------+-------+----------+-------+
| id | select_type| table  | type | key          |key_len| ref    | rows  |filtered| Extra |
+----+-------------+--------+------+---------------+------+---------+-------+----------+-------+
|  1 | SIMPLE     | orders | ref  | idx_status_time |  10 | const  |   100 |  100.00|Using index|
+----+-------------+--------+------+---------------+------+---------+-------+----------+-------+
-- 使用索引！Using index，不需要额外排序
```

```sql
-- 查询：ORDER BY 列与索引不一致
EXPLAIN SELECT * FROM orders WHERE status = 'pending' ORDER BY amount;
```

```
+----+-------------+--------+------+---------------+------+---------+-------+----------+-----------------------------+
| id | select_type| table  | type | key          |key_len| ref    | rows  |filtered| Extra                     |
+----+-------------+--------+------+---------------+------+---------+-------+----------+-----------------------------+
|  1 | SIMPLE     | orders | ref  | idx_status_time |  10 | const  |   100 |  100.00|Using where; Using filesort|
+----+-------------+--------+------+---------------+------+---------+-------+----------+-----------------------------+
-- Using filesort！无法使用索引排序
```

## ORDER BY 优化技巧

### 技巧一：添加合适的索引

```sql
-- 为 ORDER BY 添加索引
CREATE INDEX idx_create_time ON orders(create_time);

-- 测试
EXPLAIN SELECT * FROM orders ORDER BY create_time;
```

```
+----+-------------+--------+-------+------+--------+------+----------+-------+
| id | select_type| table  | type  | key  |key_len | ref  | rows     | Extra |
+----+-------------+--------+-------+------+--------+------+----------+-------+
|  1 | SIMPLE     | orders | index | idx_create_time |  6 | NULL | 1000000 |       |
+----+-------------+--------+-------+------+--------+------+----------+-------+
-- 使用索引排序，不需要 filesort
```

### 技巧二：复合索引顺序

```sql
-- 复合索引：(a, b, c)
-- 支持的排序：
-- ORDER BY a
-- ORDER BY a, b
-- ORDER BY a, b, c

-- 不支持的排序：
-- ORDER BY b          -- 跳过了 a
-- ORDER BY c           -- 跳过了 a, b
-- ORDER BY b, c        -- 跳过了 a

-- 正确设计复合索引
CREATE INDEX idx_a_b_c ON table(a, b, c);
```

### 技巧三：覆盖索引避免回表

```sql
-- 创建覆盖索引
CREATE INDEX idx_time_covering ON orders(create_time, id, status, amount);

-- 查询只需访问索引
EXPLAIN SELECT id, status, amount FROM orders ORDER BY create_time;
```

```
+----+-------------+--------+-------+------+--------+------+----------+-------+
| id | select_type| table  | type  | key  |key_len | ref  | rows     | Extra |
+----+-------------+--------+-------+------+--------+------+----------+-------+
|  1 | SIMPLE     | orders | index | idx_time_covering |  6 | NULL | 1000000 |Using index|
+----+-------------+--------+-------+------+--------+------+----------+-------+
-- Using index，不需要回表！
```

### 技巧四：延迟排序

```sql
-- 原查询
SELECT * FROM orders ORDER BY create_time LIMIT 1000000, 10;

-- 延迟关联优化
SELECT o.*
FROM orders o
INNER JOIN (
    SELECT id FROM orders ORDER BY create_time LIMIT 1000000, 10
) t ON o.id = t.id;
```

## ORDER BY 优化实战

### 案例一：用户列表排序

```sql
-- 查询：按创建时间排序
SELECT * FROM users ORDER BY create_time DESC LIMIT 100;

-- 优化：添加索引
CREATE INDEX idx_create_time ON users(create_time);

-- 验证
EXPLAIN SELECT * FROM users ORDER BY create_time DESC LIMIT 100;
```

### 案例二：多字段排序

```sql
-- 查询：按状态和时间排序
SELECT * FROM orders
WHERE status = 'pending'
ORDER BY status ASC, create_time DESC
LIMIT 100;

-- 优化：添加复合索引
CREATE INDEX idx_status_time ON orders(status, create_time DESC);

-- 验证
EXPLAIN SELECT * FROM orders
WHERE status = 'pending'
ORDER BY status ASC, create_time DESC
LIMIT 100;
```

### 案例三：带筛选的排序

```sql

SELECT * FROM orders
WHERE status IN ('pending', 'completed')
ORDER BY create_time DESC
LIMIT 100;

CREATE INDEX idx_status_create ON orders(status, create_time DESC);

-- 验证
EXPLAIN SELECT * FROM orders
WHERE status IN ('pending', 'completed')
ORDER BY create_time DESC
LIMIT 100;
```

│  │  - ORDER BY b                                           │   │

## 面试实战

### 问题一：ORDER BY 是怎么执行的？

**面试官问**：「MySQL 的 ORDER BY 是怎么执行的？为什么会慢？」

**深入回答**：

> **执行方式**：
>
> 1. **使用索引排序**：ORDER BY 列在索引中，直接返回
> 2. **使用文件排序**：无法使用索引，需要额外排序
>
> **慢的原因**：
> - 全表扫描
> - 无法使用索引
> - 需要在内存或磁盘中排序
>
> **优化方法**：为 ORDER BY 列添加索引

### 问题二：为什么加了索引 ORDER BY 还是慢？

**面试官问**：「我给 ORDER BY 的列加了索引，为什么还是 Using filesort？」

**深入回答**：

> **可能原因**：
>
> 1. **SELECT * 导致回表**：索引不覆盖查询列
> 2. **ORDER BY 与索引不匹配**：复合索引顺序不对
> 3. **WHERE 条件不匹配**：索引列未在 WHERE 中使用
>
> **解决方法**：
> - 使用覆盖索引
> - 确保 ORDER BY 与索引顺序一致
> - 检查 WHERE 条件是否匹配索引

### 问题三：如何优化 ORDER BY？

**面试官问**：「ORDER BY 怎么优化？」

**深入回答**：

> **优化方法**：
>
> 1. **添加索引**：
> ```sql
> CREATE INDEX idx_create_time ON orders(create_time);
> ```
>
> 2. **使用覆盖索引**：
> ```sql
> CREATE INDEX idx_time_covering ON orders(create_time, id, status);
> ```
>
> 3. **确保索引顺序匹配 ORDER BY**
>
> 4. **延迟关联**：
> ```sql
> SELECT o.* FROM orders o
> INNER JOIN (SELECT id FROM orders ORDER BY create_time LIMIT 1000, 10) t
> ON o.id = t.id;
> ```

## 总结

ORDER BY 优化要点：

- **使用索引排序**：ORDER BY 列必须在索引中
- **覆盖索引**：避免回表
- **复合索引顺序**：与 ORDER BY 一致
- **延迟关联**：优化深度分页排序

让 ORDER BY 飞起来！
