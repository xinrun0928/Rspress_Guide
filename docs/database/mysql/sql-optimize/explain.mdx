# EXPLAIN 执行计划详解：深度剖析查询优化

「这个查询为什么慢？EXPLAIN 各个字段是什么意思？」

`EXPLAIN` 是分析 SQL 查询性能的利器。理解执行计划的每个字段，是优化 SQL 的基础。

## EXPLAIN 概述

### 什么是 EXPLAIN？

`EXPLAIN` 显示 MySQL 如何执行 SQL 语句的详细信息，包括表的读取顺序、索引使用情况、连接类型等。

```sql
-- 基本语法
EXPLAIN SELECT * FROM users WHERE id = 1;

-- 详细输出
EXPLAIN FORMAT=JSON SELECT * FROM users WHERE id = 1;

-- 格式化输出
EXPLAIN ANALYZE SELECT * FROM users WHERE id = 1;
```

### EXPLAIN 输出字段

```
+----+-------------+-------+------+---------------+------+---------+-------+------+-------+
| id | select_type| table | type |   key        |key_len| ref   | rows  |filtered| Extra |
+----+-------------+-------+------+---------------+------+---------+-------+------+-------+
```

| 字段 | 说明 |
|---|---|
| id | 查询序号 |
| select_type | 查询类型 |
| table | 查询的表 |
| type | 访问类型 |
| possible_keys | 可能使用的索引 |
| key | 实际使用的索引 |
| key_len | 索引长度 |
| ref | 与索引比较的列 |
| rows | 预估扫描行数 |
| filtered | 过滤后剩余百分比 |
| Extra | 额外信息 |

## 核心字段详解

### id：查询序号

id 表示查询的执行顺序。

```sql
EXPLAIN SELECT * FROM users WHERE id = (SELECT user_id FROM orders WHERE id = 1);
```

```
+----+-------------+--------+-------+------+---------------+------+---------+-------+------+----------+-------------+
| id | select_type| table  | type  | key  | key_len       | ref  | rows    | filtered| Extra                       |
+----+-------------+--------+-------+------+---------------+------+---------+-------+------+----------+-------------+
|  1 | PRIMARY    | users  | const | PRIMARY | 8           | const|       1 |  100.00|                      |
|  1 | SUBQUERY   | orders | const | PRIMARY | 8           | const|       1 |  100.00|                      |
+----+-------------+--------+-------+------+---------------+------+---------+-------+------+----------+-------------+
```

### select_type：查询类型

| select_type | 说明 |
|---|---|
| SIMPLE | 简单 SELECT，不含子查询或 UNION |
| PRIMARY | 最外层的 SELECT |
| SUBQUERY | 子查询中的第一个 SELECT |
| DERIVED | 派生表（FROM 子句中的子查询） |
| UNION | UNION 中的第二个或后面的 SELECT |
| UNION RESULT | UNION 结果集 |

```sql
EXPLAIN
SELECT * FROM users WHERE id = 1
UNION
SELECT * FROM users WHERE id = 2;
```

```
+----+--------------+------------+-------+------+---------------+------+---------+-------+------+----------+------------------+
| id | select_type | table      | type  | key  | key_len      | ref  | rows    | filtered| Extra                    |
+----+--------------+------------+-------+------+---------------+------+---------+-------+------+----------+------------------+
|  1 | PRIMARY     | users      | const | PRIMARY | 8          | const|       1 |  100.00|                      |
|  2 | UNION       | users      | const | PRIMARY | 8          | const|       1 |  100.00|                      |
| NULL | UNION RESULT | <union1,2> | ALL   | NULL | NULL         | NULL |    NULL |   NULL| Using temporary           |
+----+--------------+------------+-------+------+---------------+------+---------+-------+------+----------+------------------+
```

### type：访问类型

type 是最重要的字段之一，表示 MySQL 查找数据的方式。

```
┌─────────────────────────────────────────────────────────────────┐
│                    type 性能排序（从好到差）                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  system > const > eq_ref > ref > ref_or_null >                 │
│  range > index > ALL                                            │
│                                                                 │
│  理想情况：至少要达到 ref 或 range                              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

| type | 说明 | 性能 |
|---|---|---|
| system | 表只有一行（系统表） | 最好 |
| const | 主键或唯一索引的等值查询，最多返回一行 | 极好 |
| eq_ref | 唯一索引扫描，每行只匹配一行 | 很好 |
| ref | 非唯一索引扫描，返回匹配的所有行 | 好 |
| ref_or_null | 类似 ref，但包含 NULL 值 | 尚可 |
| range | 索引范围扫描（`>`, `<`, `BETWEEN`, `IN`） | 尚可 |
| index | 全索引扫描 | 较差 |
| ALL | 全表扫描 | 最差 |

#### const（常量级）

```sql
EXPLAIN SELECT * FROM users WHERE id = 1;  -- 主键查询
```

```
+----+-------------+-------+-------+------+---------------+------+---------+-------+------+-------+
| id | select_type| table | type  | key  | key_len       | ref  | rows    | filtered| Extra |
+----+-------------+-------+-------+------+---------------+------+---------+-------+------+-------+
|  1 | SIMPLE     | users | const | PRIMARY | 8           | const|       1 |  100.00|       |
+----+-------------+-------+-------+------+---------------+------+---------+-------+------+-------+
```

#### eq_ref（等值引用）

```sql
-- 两表 JOIN，使用主键关联
EXPLAIN SELECT * FROM orders o
INNER JOIN users u ON o.user_id = u.id;
```

```
+----+-------------+-------+--------+------+---------------+------+---------+------------+------+-------+
| id | select_type| table | type   | key  | key_len       | ref  | rows    | filtered   | Extra |
+----+-------------+-------+--------+------+---------------+------+---------+------------+------+-------+
|  1 | SIMPLE     | o     | ALL    | NULL | NULL          | NULL |    1000 |   100.00|       |
|  1 | SIMPLE     | u     | eq_ref | PRIMARY | 8           | o.user_id |    1 |  100.00|       |
+----+-------------+-------+--------+------+---------------+------+---------+------------+------+-------+
```

#### ref（非唯一索引扫描）

```sql
-- 非唯一索引查询
EXPLAIN SELECT * FROM orders WHERE status = 'pending';
```

```
+----+-------------+--------+------+---------------+------+---------+-------+------+----------+-------+
| id | select_type| table  | type | key          | key_len| ref    | rows  |filtered| Extra       |
+----+-------------+--------+------+---------------+------+---------+-------+------+----------+-------+
|  1 | SIMPLE     | orders | ref  | idx_status   |      10| const  |   100 |  100.00| Using index |
+----+-------------+--------+------+---------------+------+---------+-------+------+----------+-------+
```

#### range（索引范围扫描）

```sql
-- 范围查询
EXPLAIN SELECT * FROM orders WHERE create_time > '2024-01-01';
```

```
+----+-------------+--------+-------+---------------+------+---------+-------+----------+-------------+
| id | select_type| table  | type  | key          | key_len| ref    | rows  |filtered| Extra       |
+----+-------------+--------+-------+---------------+------+---------+-------+----------+-------------+
|  1 | SIMPLE     | orders | range | idx_create_time |    6 | NULL   |   500 |   100.00| Using index |
+----+-------------+--------+-------+---------------+------+---------+-------+----------+-------------+
```

#### index（全索引扫描）

```sql
-- 查询所有索引列，但扫描整个索引
EXPLAIN SELECT id, name, email FROM users;
```

```
+----+-------------+-------+-------+------+--------+------+----------+-------+
| id | select_type| table | type  | key  | key_len| ref  | rows     | Extra |
+----+-------------+-------+-------+------+--------+------+----------+-------+
|  1 | SIMPLE     | users | index | name |     102| NULL | 100000   |       |
+----+-------------+-------+-------+------+--------+------+----------+-------+
```

#### ALL（全表扫描）

```sql
-- 无索引查询
EXPLAIN SELECT * FROM orders WHERE amount = 100;
```

```
+----+-------------+--------+------+------+------+----------+-------+----------+-------+
| id | select_type| table  | type | key  |key_len| ref     | rows  |filtered| Extra |
+----+-------------+--------+------+------+------+----------+-------+----------+-------+
|  1 | SIMPLE     | orders | ALL  | NULL |  NULL | NULL    | 10000 |   10.00|Using where|
+----+-------------+--------+------+------+------+----------+-------+----------+-------+
```

:::warning type = ALL 是性能杀手

全表扫描是最差的情况，应该尽量避免。如果看到 type = ALL，需要检查是否缺少索引。
:::

### key：实际使用的索引

key 表示实际使用的索引。

```sql
EXPLAIN SELECT * FROM orders WHERE status = 'pending';
```

```
+-------------+--------+------+---------------+------+---------+-------+------+----------+-------+
| table       | type   | key          | key_len | ref    | rows  |filtered| Extra       |
+-------------+--------+------+---------------+------+---------+-------+------+----------+-------+
| orders      | ref    | idx_status   |      10 | const  |   100 |  100.00|            |
+-------------+--------+------+---------------+------+---------+-------+------+----------+-------+
```

| 字段 | 说明 |
|---|---|
| key | 实际使用的索引名 |
| key_len | 使用的索引长度 |
| possible_keys | 可能使用的索引列表 |

### rows：预估扫描行数

rows 表示 MySQL 预估需要扫描的行数。这个数字不是精确的，但可以反映查询效率。

```
┌─────────────────────────────────────────────────────────────────┐
│                    rows 与性能的关系                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  rows 越小越好！                                                │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  rows = 1-10：优秀，极快                                │   │
│  │  rows = 10-100：良好，快速                              │   │
│  │  rows = 100-1000：一般，可以接受                        │   │
│  │  rows = 1000-10000：较慢，需要优化                      │   │
│  │  rows > 10000：差，必须优化                             │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### Extra：额外信息

Extra 包含很多重要的优化信息。

```
┌─────────────────────────────────────────────────────────────────┐
│                    常见 Extra 值                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Using index：覆盖索引，无需回表                                │
│  Using where：使用 WHERE 过滤                                   │
│  Using temporary：需要使用临时表                                │
│  Using filesort：需要额外排序                                   │
│  Using index condition：使用索引条件下推                        │
│  Using MRR：使用 MRR 优化                                       │
│  Select tables optimized away：直接返回聚合结果                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### Using index（覆盖索引）

```sql
-- 查询的列都在索引中
EXPLAIN SELECT id, status FROM orders WHERE status = 'pending';
```

```
+----+-------------+--------+------+---------------+------+---------+-------+------+----------+-------------+
| id | select_type| table  | type | key          | key_len| ref    | rows  |filtered| Extra      |
+----+-------------+--------+------+---------------+------+---------+-------+------+----------+-------------+
|  1 | SIMPLE     | orders | ref  | idx_status   |      10| const  |   100 |  100.00|Using index|
+----+-------------+--------+------+---------------+------+---------+-------+------+----------+-------------+
```

#### Using filesort（文件排序）

```sql
-- 无索引排序
EXPLAIN SELECT * FROM orders WHERE status = 'pending' ORDER BY create_time;
```

```
+----+-------------+--------+------+---------------+------+---------+-------+----------+-----------------------------+
| id | select_type| table  | type | key          | key_len| ref    | rows  |filtered| Extra                     |
+----+-------------+--------+------+---------------+------+---------+-------+----------+-----------------------------+
|  1 | SIMPLE     | orders | ref  | idx_status   |      10| const  |   100 |  100.00|Using index; Using filesort|
+----+-------------+--------+------+---------------+------+---------+-------+----------+-----------------------------+
```

:::warning Using filesort 需要优化

filesort 意味着 MySQL 需要额外排序操作。如果看到 Using filesort，需要检查是否有合适的索引支持排序。
:::

#### Using temporary（使用临时表）

```sql
-- 需要临时表
EXPLAIN SELECT status, COUNT(*) FROM orders GROUP BY status;
```

```
+----+-------------+--------+------+---------------+------+---------+-------+----------+--------------------------------+
| id | select_type| table  | type | key          | key_len| ref    | rows  |filtered| Extra                        |
+----+-------------+--------+------+---------------+------+---------+-------+----------+--------------------------------+
|  1 | SIMPLE     | orders | index| PRIMARY      |      8 | NULL   | 10000 |  100.00|Using index; Using temporary|
+----+-------------+--------+------+---------------+------+---------+-------+----------+--------------------------------+
```

#### Using index condition（索引条件下推）

```sql
-- 条件下推
EXPLAIN SELECT * FROM orders WHERE status = 'pending' AND amount > 100;
```

```
+----+-------------+--------+------+---------------+------+---------+-------+----------+-------------------------+
| id | select_type| table  | type | key          | key_len| ref    | rows  |filtered| Extra                  |
+----+-------------+--------+------+---------------+------+---------+-------+----------+-------------------------+
|  1 | SIMPLE     | orders | ref  | idx_status   |      10| const  |    50 |   33.33|Using index condition|
+----+-------------+--------+------+---------------+------+---------+-------+----------+-------------------------+
```

#### Using MRR（多范围读）

```sql
-- MRR 优化
EXPLAIN SELECT * FROM orders WHERE id IN (1, 2, 3, 4, 5);
```

```
+----+-------------+--------+-------+---------------+------+---------+------+----------+-------------+
| id | select_type| table  | type  | key          | key_len| ref    | rows  |filtered| Extra       |
+----+-------------+--------+-------+---------------+------+---------+------+----------+-------------+
|  1 | SIMPLE     | orders | range | PRIMARY      |      8 | NULL   |     5 |  100.00|Using MRR   |
+----+-------------+--------+-------+---------------+------+---------+------+----------+-------------+
```

## 面试实战

### 问题一：EXPLAIN 最重要的字段是什么？

**面试官问**：「EXPLAIN 输出中最重要的字段是哪个？」

**深入回答**：

> **type 字段最重要**，它表示 MySQL 查找数据的方式：
>
> | type | 性能 |
> |---|---|
> | system/const/eq_ref | 优秀 |
> | ref | 良好 |
> | range | 一般 |
> | index/ALL | 差 |
>
> **最佳是 const 或 eq_ref**，如果看到 **ALL（全表扫描）**，说明性能很差，需要优化。

### 问题二：Using filesort 是什么意思？如何优化？

**面试官问**：「Using filesort 是什么意思？如何避免？」

**深入回答**：

> **Using filesort** 表示 MySQL 需要额外的排序操作，无法利用索引顺序。
>
> **优化方法**：
> 1. 添加 ORDER BY 列的索引
> 2. 确保索引列的顺序与 ORDER BY 一致
>
> ```sql
> -- 不好：无索引排序
> SELECT * FROM orders ORDER BY create_time;
>
> -- 优化：添加索引
> CREATE INDEX idx_create_time ON orders(create_time);
>
> -- 好：使用索引排序
> SELECT * FROM orders ORDER BY create_time;  -- Using filesort 消失
> ```

### 问题三：rows 字段代表什么？

**面试官问**：「EXPLAIN 中的 rows 字段代表什么？」

**深入回答**：

> **rows 是 MySQL 预估需要扫描的行数**。
>
> - rows 越小，查询越快
> - rows 是预估数，不是精确值
> - rows 乘以 filtered 约等于实际扫描行数
>
> 如果 rows 很大（如 100000），说明查询效率低，需要优化。

## 总结

EXPLAIN 是分析 SQL 性能的核心工具：

- **type**：访问类型，const/eq_ref > ref > range > ALL
- **key**：实际使用的索引
- **rows**：预估扫描行数，越少越好
- **Extra**：Using filesort/Using temporary 需要优化

掌握 EXPLAIN 的分析方法，是 SQL 优化的基础。
