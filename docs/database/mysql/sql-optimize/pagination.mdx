# 分页查询优化：告别深度分页

「为什么分页到第 10000 页就变慢了？」

深度分页是 SQL 优化的经典难题。本文深入剖析分页查询的性能问题和优化方案。

## 深度分页问题

## 分页优化方案

### 方案一：基于 ID 的分页

```sql
-- 原始查询
SELECT * FROM orders ORDER BY id LIMIT 1000000, 10;

-- 优化：记住上次查询的最后一条 ID
-- 假设上次查询的最后一条 ID 是 1000000
SELECT * FROM orders
WHERE id > 1000000
ORDER BY id
LIMIT 10;
```

```
┌─────────────────────────────────────────────────────────────────┐
│                    基于 ID 分页原理                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  原始方式：                                                     │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  LIMIT 1000000, 10                                      │   │
│  │  → 扫描 1000010 行，丢弃前 1000000 行                   │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  优化方式：                                                     │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  WHERE id > 1000000 LIMIT 10                            │   │
│  │  → 直接定位到第 1000000 行，返回 10 行                  │   │
│  │  → 扫描 10 行！                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 方案二：延迟关联

```sql
-- 延迟关联：先查出 ID，再关联获取完整数据
SELECT o.*, u.name as user_name
FROM orders o
INNER JOIN (
    SELECT id FROM orders
    ORDER BY id
    LIMIT 1000000, 10
) t ON o.id = t.id
INNER JOIN users u ON o.user_id = u.id;
```

```

│                    延迟关联原理                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  执行过程：                                                    │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  Step 1: 子查询获取 ID                                  │   │
│  │  SELECT id FROM orders ORDER BY id LIMIT 1000000, 10   │   │
│  │  → 只扫描索引，不回表，返回 10 个 ID                     │   │
│  │                                                         │   │
│  │  Step 2: 关联查询完整数据                                │   │
│  │  SELECT o.*, u.name FROM orders o                       │   │
│  │  INNER JOIN (子查询结果) t ON o.id = t.id               │   │
│  │  → 根据 ID 回表获取完整数据                             │   │

└─────────────────────────────────────────────────────────────────┘
```

### 方案三：游标分页

```sql
-- 使用游标（上一页的最后一条）
SELECT * FROM orders
WHERE id > #{lastId}
ORDER BY id
LIMIT 10;

-- 配合前端传递 lastId
-- 首页：lastId = 0
-- 下一页：lastId = 当前页最后一条的 ID
```

### 方案四：倒序分页

```sql
-- 如果订单是倒序展示
SELECT * FROM orders
ORDER BY id DESC
LIMIT 10;

-- 下一页
SELECT * FROM orders
WHERE id < #{lastId}
ORDER BY id DESC
LIMIT 10;
```

## 复杂排序分页优化

### 带条件筛选的分页

```sql
-- 带筛选条件的分页
SELECT * FROM orders
WHERE status = 'completed'
ORDER BY create_time DESC
LIMIT 1000000, 10;

-- 优化：确保有索引
CREATE INDEX idx_status_time ON orders(status, create_time DESC);

-- 使用延迟关联
SELECT o.*
FROM orders o
INNER JOIN (
    SELECT id FROM orders
    WHERE status = 'completed'
    ORDER BY create_time DESC
    LIMIT 1000000, 10
) t ON o.id = t.id;
```

### 多字段排序分页

```sql
-- 多字段排序
SELECT * FROM orders
ORDER BY status ASC, create_time DESC
LIMIT 1000000, 10;

-- 优化索引
CREATE INDEX idx_status_create ON orders(status, create_time DESC);

-- 使用延迟关联
SELECT o.*
FROM orders o
INNER JOIN (
    SELECT id FROM orders
    ORDER BY status ASC, create_time DESC
    LIMIT 1000000, 10
) t ON o.id = t.id;
```

## 分页优化实战

### 前端分页组件

```javascript
// 传统的 offset 分页
// page = 1, pageSize = 10
// LIMIT (page - 1) * pageSize, pageSize

// 优化：使用游标分页
let lastId = null;

async function loadMore() {
    const params = lastId ? `?lastId=${lastId}` : '';
    const response = await fetch(`/api/orders${params}`);
    const data = await response.json();

    // 更新游标
    if (data.length > 0) {
        lastId = data[data.length - 1].id;
    }

    return data;
}
```

### 后端接口

```sql
-- 普通分页接口
SELECT * FROM orders
ORDER BY id
LIMIT #{offset}, #{pageSize};

-- 优化：游标分页接口
SELECT * FROM orders
WHERE id > #{lastId}
ORDER BY id
LIMIT #{pageSize};
```

## 面试实战

### 问题一：深度分页为什么慢？

**面试官问**：「为什么分页到很深的位置就变慢了？」

**深入回答**：

> **原因**：`LIMIT offset, size` 的执行过程
>
> 1. MySQL 先扫描 offset + size 行数据
> 2. 然后丢弃前 offset 行
> 3. 返回最后 size 行
>
> ```sql
> LIMIT 1000000, 10
> -- 需要扫描 1000010 行，丢弃前 1000000 行，只返回 10 行
> ```
>
> **解决方法**：使用基于 ID 的分页

### 问题二：如何优化分页查询？

**面试官问**：「分页查询怎么优化？」

**深入回答**：

> **优化方案**：
>
> 1. **基于 ID 分页**：
> ```sql
> -- 记住上次查询的最后一条 ID
> WHERE id > #{lastId} LIMIT 10
> ```
>
> 2. **延迟关联**：
> ```sql
> SELECT o.* FROM orders o
> INNER JOIN (SELECT id FROM orders LIMIT 1000000, 10) t
> ON o.id = t.id
> ```
>
> 3. **倒序分页**：使用 `WHERE id < lastId` 代替 offset

### 问题三：分页和游标分页的区别？

**面试官问**：「分页和游标分页各有什么优缺点？」

**深入回答**：

> **传统分页**：
> - 优点：可以跳转到任意页
> - 缺点：深度分页性能差
>
> **游标分页**：
> - 优点：性能稳定，不受深度影响
> - 缺点：只能下一页，无法跳页
>
> **选择建议**：
> - Feed 流、聊天记录：游标分页
> - 需要跳页的场景：传统分页 + 优化

## 总结

分页查询优化要点：

- **深度分页问题**：LIMIT offset 会扫描大量无用行
- **优化方案**：基于 ID 分页、延迟关联、游标分页
- **核心思想**：减少扫描行数

告别深度分页，让查询飞起来！
