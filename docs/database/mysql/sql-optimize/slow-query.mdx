# 慢查询定位：如何找到性能杀手

「为什么我的查询突然变慢了？」

慢查询是数据库性能问题的最常见原因。本文介绍如何定位和分析慢查询。

## 慢查询概述

### 什么是慢查询？

慢查询是指执行时间超过阈值的 SQL 查询。通常，阈值设置为 1 秒或 2 秒。

```
┌─────────────────────────────────────────────────────────────────┐
│                    慢查询定义                                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  慢查询（Slow Query）：                                   │   │
│  │                                                         │   │
│  │  执行时间 > 慢查询阈值（long_query_time）                 │   │
│  │                                                         │   │
│  │  MySQL 默认阈值：10 秒                                  │   │
│  │  生产环境建议阈值：1-2 秒                                │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 开启慢查询日志

### 配置慢查询日志

```sql
-- 查看慢查询相关配置
SHOW VARIABLES LIKE 'slow_query%';
SHOW VARIABLES LIKE 'long_query_time';
SHOW VARIABLES LIKE 'log_queries_not_using_indexes';
```

```
+---------------------+--------------------------------+
| Variable_name       | Value                          |
+---------------------+--------------------------------+
| slow_query_log      | OFF                            |
| slow_query_log_file | /var/lib/mysql/mysql-slow.log |
+---------------------+--------------------------------+

+---------------------+--------------------+
| Variable_name       | Value              |
+---------------------+--------------------+
| long_query_time     | 10.000000          |
+---------------------+--------------------+
```

### 开启慢查询日志

```sql
-- 临时开启慢查询日志
SET GLOBAL slow_query_log = 'ON';
SET GLOBAL long_query_time = 1;  -- 设置为 1 秒

-- 开启未使用索引的查询日志
SET GLOBAL log_queries_not_using_indexes = 'ON';
```

### 配置文件设置

```ini
[mysqld]
# 开启慢查询日志
slow_query_log = 1
slow_query_log_file = /var/lib/mysql/mysql-slow.log
long_query_time = 1

# 记录未使用索引的查询
log_queries_not_using_indexes = 1

# 记录管理语句
log_slow_admin_statements = 1

# 记录慢查询数量阈值
min_examined_row_limit = 1000
```

## 分析慢查询日志

### 慢查询日志格式

```bash
# 查看慢查询日志
cat /var/lib/mysql/mysql-slow.log
```

```
# Time: 2024-01-15T10:30:00.123456Z
# User@Host: app_user[app_user] @ localhost []
# Query_time: 2.534321  Lock_time: 0.000123 Rows_sent: 100  Rows_examined: 100000
SET timestamp=1705315800;
SELECT * FROM orders WHERE status = 'pending' ORDER BY create_time DESC;
```

### 字段说明

| 字段 | 说明 |
|---|---|
| Time | 查询执行时间 |
| User@Host | 用户和主机 |
| Query_time | 查询执行时间（秒） |
| Lock_time | 锁等待时间（秒） |
| Rows_sent | 返回行数 |
| Rows_examined | 扫描行数 |
| Query | 执行的 SQL 语句 |

### 使用 mysqldumpslow 分析

```bash
# 安装 MySQL 后自带 mysqldumpslow

# 分析慢查询日志
mysqldumpslow -s t /var/lib/mysql/mysql-slow.log

# 分析最慢的 10 条查询
mysqldumpslow -s t -t 10 /var/lib/mysql/mysql-slow.log

# 分析平均执行时间最长的查询
mysqldumpslow -s a /var/lib/mysql/mysql-slow.log

# 分析按锁定时间排序
mysqldumpslow -s l /var/lib/mysql/mysql-slow.log

# 分析扫描行数最多的查询
mysqldumpslow -s r /var/lib/mysql/mysql-slow.log
```

### 使用 pt-query-digest 分析

```bash
# pt-query-digest 是 Percona Toolkit 工具，比 mysqldumpslow 更强大

# 安装 Percona Toolkit
# apt-get install percona-toolkit

# 分析慢查询
pt-query-digest /var/lib/mysql/mysql-slow.log

# 输出到文件
pt-query-digest /var/lib/mysql/mysql-slow.log > slow_report.txt

# 只分析某时间范围的查询
pt-query-digest --since='2024-01-15 10:00:00' \
                 --until='2024-01-15 12:00:00' \
                 /var/lib/mysql/mysql-slow.log
```

## 使用 EXPLAIN 分析查询

### EXPLAIN 基本用法

```sql
EXPLAIN SELECT * FROM orders WHERE status = 'pending' ORDER BY create_time DESC;
```

```
+----+-------------+--------+------+---------------+------+---------+------+----------+-----------------------------+
| id | select_type| table  | type |   key        |key_len| ref   | rows | filtered | Extra                      |
+----+-------------+--------+------+---------------+------+---------+------+----------+-----------------------------+
|  1 | SIMPLE     | orders | ref  | idx_status   |   10 | const  |  500 |   100.00 | Using index condition;      |
|    |            |        |      |              |      |        |      |          | Using filesort              |
+----+-------------+--------+------+---------------+------+---------+------+----------+-----------------------------+
```

### 关键字段解读

| 字段 | 说明 | 关注点 |
|---|---|---|
| type | 访问类型 | const > eq_ref > ref > range > index > ALL |
| key | 使用的索引 | 应有值，不能是 NULL |
| rows | 扫描行数 | 越少越好 |
| Extra | 额外信息 | Using filesort, Using temporary |

## 定位慢查询的常见原因

### 原因一：缺少索引

```sql
-- 查看表的索引
SHOW INDEX FROM orders;

-- 查看查询的索引使用情况
EXPLAIN SELECT * FROM orders WHERE status = 'pending';

-- 创建索引
CREATE INDEX idx_status ON orders(status);
CREATE INDEX idx_create_time ON orders(create_time);
```

### 原因二：SQL 写法问题

```sql
-- 避免 SELECT *
SELECT user_id, name, email FROM users WHERE id = 1;

-- 避免在索引列上使用函数
SELECT * FROM orders WHERE YEAR(create_time) = 2024;
-- 改为：
SELECT * FROM orders WHERE create_time >= '2024-01-01' AND create_time < '2025-01-01';

-- 避免隐式类型转换
SELECT * FROM users WHERE phone = 13800138000;  -- phone 是 VARCHAR
-- 改为：
SELECT * FROM users WHERE phone = '13800138000';
```

### 原因三：数据量太大

```sql
-- 分页查询大数据量
SELECT * FROM orders ORDER BY id LIMIT 1000000, 10;

-- 优化为基于 ID 的分页
SELECT * FROM orders WHERE id > 1000000 ORDER BY id LIMIT 10;

-- 或者使用延迟关联
SELECT o.* FROM orders o
INNER JOIN (
    SELECT id FROM orders ORDER BY id LIMIT 1000000, 10
) t ON o.id = t.id;
```

## 慢查询监控

### performance_schema 监控

```sql
-- 开启慢查询监控
UPDATE performance_schema.setup_consumers
SET ENABLED = 'YES'
WHERE NAME LIKE '%statements%';

-- 查看最近的慢查询
SELECT
    DIGEST_TEXT,
    COUNT_STAR,
    SUM_TIMER_WAIT / 1000000000000 AS duration,
    AVG_TIMER_WAIT / 1000000000000 AS avg_duration
FROM performance_schema.events_statements_summary_by_digest
WHERE SUM_TIMER_WAIT > 0
ORDER BY SUM_TIMER_WAIT DESC
LIMIT 10;
```

### 实时监控慢查询

```sql
-- 查看当前正在执行的慢查询
SHOW PROCESSLIST;

-- 或者查询 information_schema
SELECT
    ID,
    USER,
    DB,
    COMMAND,
    TIME,
    STATE,
    INFO
FROM information_schema.PROCESSLIST
WHERE COMMAND != 'Sleep'
AND TIME > 5;  -- 执行超过 5 秒的查询
```

## 面试实战

### 问题一：如何定位慢查询？

**面试官问**：「如何定位数据库中的慢查询？」

**深入回答**：

> **定位慢查询的方法**：
>
> 1. **开启慢查询日志**：
> ```sql
> SET GLOBAL slow_query_log = 'ON';
> SET GLOBAL long_query_time = 1;
> ```
>
> 2. **分析慢查询日志**：
> ```bash
> mysqldumpslow -s t /var/lib/mysql/mysql-slow.log
> pt-query-digest /var/lib/mysql/mysql-slow.log
> ```
>
> 3. **实时监控**：
> ```sql
> SHOW PROCESSLIST;
> ```
>
> 4. **使用 EXPLAIN 分析**：
> ```sql
> EXPLAIN SELECT * FROM orders WHERE status = 'pending';
> ```

### 问题二：慢查询的常见原因有哪些？

**面试官问**：「慢查询的常见原因是什么？」

**深入回答**：

> **常见原因**：
>
> 1. **缺少索引**：WHERE 条件没有索引
> 2. **SQL 写法问题**：SELECT *、函数、隐式转换
> 3. **数据量太大**：分页深度、扫描行数多
> 4. **锁等待**：并发冲突导致等待
> 5. **服务器配置**：内存、Buffer Pool 配置不当

### 问题三：如何优化慢查询？

**面试官问**：「优化慢查询的思路是什么？」

**深入回答**：

> **优化思路**：
>
> 1. **分析执行计划**：`EXPLAIN` 查看 type、key、rows
> 2. **添加索引**：为 WHERE、JOIN、ORDER BY 字段添加索引
> 3. **优化 SQL**：避免 SELECT *、函数、隐式转换
> 4. **优化结构**：分库分表、分区表
> 5. **调整配置**：增加 Buffer Pool、调整参数

## 总结

慢查询定位是数据库性能优化的基础：

- **开启慢查询日志**：记录执行时间超过阈值的查询
- **分析日志**：使用 mysqldumpslow 或 pt-query-digest
- **EXPLAIN 分析**：查看执行计划，找出性能瓶颈
- **优化方法**：添加索引、优化 SQL、调整结构

掌握慢查询定位技能，是 DBA 和后端工程师的必备知识。
