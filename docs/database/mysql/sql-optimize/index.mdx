# SQL 优化总览

面试中，SQL 优化几乎是必考内容。你可能会遇到这样的问题：「线上有个慢查询，你怎么排查和优化？」「这个 SQL 跑了 10 秒，怎么优化到 1 秒以内？」

慢查询不仅影响用户体验，严重时还会拖垮整个系统。SQL 优化是后端开发者的核心技能之一。

本模块带你系统学习 SQL 优化的方法论和实战技巧。

## 慢查询分析

慢查询分析的第一步是定位问题。MySQL 提供了多种工具：

**慢查询日志**：开启 `slow_query_log`，记录执行时间超过 `long_query_time` 的 SQL。

```sql
-- 查看慢查询配置
SHOW VARIABLES LIKE 'slow_query%';
SHOW VARIABLES LIKE 'long_query_time';

-- 开启慢查询日志
SET GLOBAL slow_query_log = 'ON';
SET GLOBAL long_query_time = 1;
```

**EXPLAIN 分析**：使用 EXPLAIN 查看执行计划，分析查询类型、索引使用、扫描行数等。

```sql
EXPLAIN SELECT * FROM users WHERE name = '张三';
```

## 执行计划关键字段

| 字段 | 说明 | 关注点 |
|---|---|---|
| `type` | 访问类型 | system > const > eq_ref > ref > range > index > ALL |
| `key` | 实际使用的索引 | NULL 表示全表扫描 |
| `rows` | 估算扫描行数 | 越大越慢 |
| `Extra` | 额外信息 | Using index、Using filesort、Using temporary |

**必须优化的场景**：`type = ALL`（全表扫描）、`Using filesort`（文件排序）、`Using temporary`（临时表）。

## SQL 优化核心原则

**索引优化**：
- 为 WHERE、JOIN、ORDER BY、GROUP BY 的列创建索引
- 遵循最左前缀原则设计联合索引
- 使用覆盖索引减少回表
- 避免索引失效的场景

**SQL 写法优化**：
- 避免 SELECT *，只查询需要的列
- 避免在 WHERE 条件中对索引列使用函数
- 避免隐式类型转换
- 控制 OR 的使用，改用 UNION ALL 或 IN
- 避免深度分页，使用延迟关联或游标分页

**表结构优化**：
- 选择合适的数据类型
- 适当冗余，减少 JOIN
- 大表分库分表
