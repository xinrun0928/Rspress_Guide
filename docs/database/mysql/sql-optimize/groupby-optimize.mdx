# GROUP BY 优化：高效分组统计

「GROUP BY 为什么要排序？如何让分组统计更快？」

GROUP BY 是 SQL 中最常用的聚合查询之一，但处理不当会导致严重的性能问题。本文深入剖析 GROUP BY 的优化技巧。

## GROUP BY 原理

### GROUP BY 执行过程

```
┌─────────────────────────────────────────────────────────────────┐
│                    GROUP BY 执行过程                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  GROUP BY 执行步骤：                                    │   │
│  │                                                         │   │
│  │  1. 扫描数据，构建分组                                  │   │
│  │  2. 对每个分组应用聚合函数                              │   │
│  │  3. 返回结果                                            │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  GROUP BY 的隐含排序：                                  │   │
│  │                                                         │   │
│  │  MySQL 5.7 之前：GROUP BY 默认会排序                    │   │
│  │  MySQL 8.0 之后：GROUP BY 不再隐含排序                   │   │
│  │  - 如果需要排序，必须加 ORDER BY                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## GROUP BY 优化技巧

### 技巧一：添加合适的索引

```sql
-- 单字段分组
CREATE INDEX idx_status ON orders(status);

-- 多字段分组
CREATE INDEX idx_status_user ON orders(status, user_id);

-- 测试
EXPLAIN SELECT status, user_id, COUNT(*), SUM(amount)
FROM orders
GROUP BY status, user_id;
```

```
+----+-------------+--------+-------+------+--------+------+----------+-------+
| id | select_type| table  | type  | key  |key_len | ref  | rows     | Extra |
+----+-------------+--------+-------+------+--------+------+----------+-------+
|  1 | SIMPLE     | orders | index | idx_status_user |  10 | NULL |      30 |Using index|
+----+-------------+--------+-------+------+--------+------+----------+-------+
-- 使用索引分组
```

### 技巧二：覆盖索引避免回表

```sql
-- 创建覆盖索引
CREATE INDEX idx_status_covering ON orders(status, COUNT(*), SUM(amount));

-- 或者：索引包含所有查询列
CREATE INDEX idx_status_covering ON orders(status, user_id, amount);

-- 测试
EXPLAIN SELECT status, COUNT(*), SUM(amount)
FROM orders
GROUP BY status;
```

```
+----+-------------+--------+-------+------+--------+------+----------+-------+
| id | select_type| table  | type  | key  |key_len | ref  | rows     | Extra |
+----+-------------+--------+-------+------+--------+------+----------+-------+
|  1 | SIMPLE     | orders | index | idx_status_covering |  6 | NULL |      10 |Using index|
+----+-------------+--------+-------+------+--------+------+----------+-------+
-- Using index，无需回表
```

### 技巧三：使用 HAVING 过滤

```sql
-- 先分组，再过滤（不好）
SELECT status, COUNT(*) as cnt
FROM orders
GROUP BY status
HAVING cnt > 100;

-- 先过滤，再分组（好）
SELECT status, COUNT(*) as cnt
FROM orders
WHERE status IN ('pending', 'completed')
GROUP BY status
HAVING cnt > 100;
```

### 技巧四：避免 SELECT *

```sql
-- 不好：SELECT *
SELECT *
FROM orders
GROUP BY status;

-- 好：只查询需要的列
SELECT status, COUNT(*) as cnt
FROM orders
GROUP BY status;
```

## GROUP BY 与 ORDER BY

### 同时使用 GROUP BY 和 ORDER BY

```sql
-- 查询并排序
SELECT status, COUNT(*) as cnt
FROM orders
GROUP BY status
ORDER BY cnt DESC;

EXPLAIN SELECT status, COUNT(*) as cnt
FROM orders
GROUP BY status
ORDER BY cnt DESC;
```

```
+----+-------------+--------+-------+------+--------+------+----------+-----------------------------+
| id | select_type| table  | type  | key  |key_len | ref  | rows     | Extra                       |
+----+-------------+--------+-------+------+--------+------+----------+-----------------------------+
|  1 | SIMPLE     | orders | index | idx_status |  6 | NULL |      10 |Using index; Using filesort|
+----+-------------+--------+-------+------+--------+------+----------+-----------------------------+
-- 使用索引分组，但需要额外排序
```

### 优化：索引覆盖 ORDER BY

```sql
-- 创建复合索引，包含分组和排序列
CREATE INDEX idx_status_cnt ON orders(status, COUNT(*));

-- 或者调整查询
CREATE INDEX idx_cnt ON orders(COUNT(*));
```

## GROUP BY 优化实战

### 案例一：用户订单统计

```sql
-- 查询：每个用户的订单数量和金额
SELECT user_id, COUNT(*) as order_count, SUM(amount) as total_amount
FROM orders
GROUP BY user_id;

-- 优化：添加索引
CREATE INDEX idx_user_id ON orders(user_id);

-- 测试
EXPLAIN SELECT user_id, COUNT(*), SUM(amount)
FROM orders
GROUP BY user_id;
```

### 案例二：带筛选的分组

```sql
-- 查询：2024 年的每月订单统计
SELECT DATE_FORMAT(create_time, '%Y-%m') as month,
       COUNT(*) as order_count,
       SUM(amount) as total_amount
FROM orders
WHERE create_time >= '2024-01-01'
GROUP BY DATE_FORMAT(create_time, '%Y-%m');

-- 优化：索引 + 函数转换
CREATE INDEX idx_create_time ON orders(create_time);

-- 注意：函数会导致索引失效
-- 优化方案：使用范围查询
CREATE INDEX idx_year_month ON orders(create_time, amount);

SELECT '2024-01' as month, COUNT(*), SUM(amount)
FROM orders
WHERE create_time >= '2024-01-01' AND create_time < '2024-02-01'
UNION ALL
SELECT '2024-02' as month, COUNT(*), SUM(amount)
FROM orders
WHERE create_time >= '2024-02-01' AND create_time < '2024-03-01';
```

### 案例三：多维度统计

```sql
-- 查询：按状态和月份统计
SELECT status,
       DATE_FORMAT(create_time, '%Y-%m') as month,
       COUNT(*) as cnt
FROM orders
GROUP BY status, DATE_FORMAT(create_time, '%Y-%m');

-- 优化：创建日期索引
CREATE INDEX idx_status_create ON orders(status, create_time);
```

## 面试实战

### 问题一：GROUP BY 是怎么执行的？

**面试官问**：「MySQL 的 GROUP BY 是怎么执行的？」

**深入回答**：

> **执行过程**：
>
> 1. **扫描数据**：读取所有需要分组的数据
> 2. **构建分组**：将相同值的行分到一组
> 3. **聚合计算**：对每个分组应用聚合函数
> 4. **返回结果**：返回分组和聚合结果
>
> **性能问题**：
> - 全表扫描
> - 使用临时表存储中间结果
> - 可能需要排序

### 问题二：GROUP BY 怎么优化？

**面试官问**：「GROUP BY 怎么优化？」

**深入回答**：

> **优化方法**：
>
> 1. **添加索引**：
> ```sql
> CREATE INDEX idx_status ON orders(status);
> ```
>
> 2. **使用覆盖索引**：
> ```sql
> CREATE INDEX idx_status_covering ON orders(status, COUNT(*), SUM(amount));
> ```
>
> 3. **先过滤再分组**：
> ```sql
> SELECT status, COUNT(*)
> FROM orders
> WHERE status IN ('pending', 'completed')
> GROUP BY status;
> ```
>
> 4. **避免 SELECT ***：
> ```sql
> SELECT status, COUNT(*)  -- 只查询需要的列
> FROM orders
> GROUP BY status;
> ```

### 问题三：GROUP BY 和 DISTINCT 哪个更快？

**面试官问**：「GROUP BY 和 DISTINCT 有什么区别？哪个更快？」

**深入回答**：

> **区别**：
> - DISTINCT：去重，返回唯一值
> - GROUP BY：分组，可以配合聚合函数
>
> **性能**：
> - 在简单去重场景下，性能相近
> - DISTINCT 是特殊的 GROUP BY
> - MySQL 8.0 对两者都有优化
>
> **选择**：
> - 只需要去重：使用 DISTINCT
> - 需要聚合统计：使用 GROUP BY

## 总结

GROUP BY 优化要点：

- **添加索引**：让 GROUP BY 利用索引分组
- **覆盖索引**：避免回表
- **先过滤再分组**：减少分组数据量
- **调整配置**：增大 tmp_table_size

让 GROUP BY 飞起来！
