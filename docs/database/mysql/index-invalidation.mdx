## 一、索引失效场景汇总

| 序号 | 场景 | 示例 | 是否失效 |
|------|------|------|----------|
| 1 | 索引列使用函数/表达式 | `WHERE YEAR(create_time) = 2024` | ❌ 失效 |
| 2 | 索引列进行计算 | `WHERE id + 1 = 100` | ❌ 失效 |
| 3 | 类型转换 | `WHERE phone = 13800138000`（phone 是 VARCHAR） | ❌ 失效 |
| 4 | 使用 OR 连接无索引列 | `WHERE name = '张三' OR age = 25` | ❌ 失效 |
| 5 | LIKE 前缀模糊匹配 | `WHERE name LIKE '%张%'` | ❌ 失效 |
| 6 | NOT NULL 判断 | `WHERE name IS NOT NULL` | ❌ 失效 |
| 7 | NOT IN / NOT EXISTS | `WHERE id NOT IN (1,2,3)` | ❌ 可能失效 |
| 8 | 优化器判断全表更快 | 数据量小、区分度低 | ❌ 可能失效 |
| 9 | 联合索引跳过中间列 | `WHERE name = '张三' AND email = 'a@x.com'`（无 age） | ❌ 部分失效 |
| 10 | ORDER BY 索引断链 | `ORDER BY name, age DESC`（DESC + ASC 混用） | ❌ 失效 |

## 二、逐一解析索引失效场景

### 2.1 索引列使用函数或表达式

```sql
-- 索引: idx_create_time(create_time)
-- 失效
SELECT * FROM orders WHERE YEAR(create_time) = 2024;
SELECT * FROM orders WHERE DATE_FORMAT(create_time, '%Y') = '2024';
SELECT * FROM orders WHERE DATE_SUB(create_time, INTERVAL 1 DAY) = '2024-01-01';

-- 正确写法：使用索引列本身
SELECT * FROM orders WHERE create_time >= '2024-01-01' AND create_time < '2025-01-01';
```

**原因**：索引树按 `create_time` 的原始值排序，使用函数后值被改变，无法利用 B+Tree 的有序性。

### 2.2 类型转换导致索引失效

```sql
-- phone 字段类型是 VARCHAR
-- 失效：字符串和数字比较，字符串被强制转数字
SELECT * FROM users WHERE phone = 13800138000;

-- 正确：使用字符串字面量
SELECT * FROM users WHERE phone = '13800138000';

-- 索引: idx_name(name VARCHAR(50))
-- 失效：隐式类型转换
SELECT * FROM users WHERE name = 123;  -- name 被转成数字
```

**原理**：MySQL 比较时，如果类型不一致会进行隐式转换，导致索引失效。

### 2.3 OR 条件导致索引失效

```sql
-- 索引: idx_name(name), idx_age(age)
-- 失效：OR 条件要求两边都有索引
SELECT * FROM users WHERE name = '张三' OR age = 25;

-- 生效：使用 UNION 连接两个索引查询
SELECT * FROM users WHERE name = '张三'
UNION
SELECT * FROM users WHERE age = 25;

-- 或者：为 age 也建立索引
CREATE INDEX idx_age ON users(age);
```

### 2.4 LIKE 模糊匹配失效

| 场景 | SQL | 索引使用 |
|------|-----|----------|
| 前缀匹配 | `LIKE '张%'` | ✅ 生效 |
| 后缀匹配 | `LIKE '%张'` | ❌ 失效 |
| 模糊匹配 | `LIKE '%张%'` | ❌ 失效 |

```sql
-- 索引: idx_name(name)
-- 前缀匹配：可以利用索引
SELECT * FROM users WHERE name LIKE '张%';

-- 后缀/模糊：无法利用索引，必须全表扫描
SELECT * FROM users WHERE name LIKE '%张';
SELECT * FROM users WHERE name LIKE '%张%';

-- 优化方案：使用全文索引
ALTER TABLE users ADD FULLTEXT INDEX idx_fulltext(name);
SELECT * FROM users WHERE MATCH(name) AGAINST('张');
```

### 2.5 NOT NULL 和 NOT IN

```sql
-- 索引: idx_name(name)
-- 失效：IS NOT NULL 通常无法使用索引
SELECT * FROM users WHERE name IS NOT NULL;

-- 失效：NOT IN 列表值较少时可能走全表
SELECT * FROM users WHERE id NOT IN (1, 2, 3);

-- 优化方案：使用 NOT EXISTS 或 LEFT JOIN + IS NULL
SELECT * FROM users WHERE NOT EXISTS (
    SELECT 1 FROM orders WHERE orders.user_id = users.id
);
```

### 2.6 联合索引跳过中间列

```sql
-- 索引: (name, age, email)
-- 失效：跳过 age，直接用 name 和 email
SELECT * FROM users WHERE name = '张三' AND email = 'a@x.com';

-- 生效：按顺序使用索引
SELECT * FROM users WHERE name = '张三' AND age = 25;
SELECT * FROM users WHERE name = '张三';  -- 只用 name
```

### 2.7 排序导致索引失效

```sql
-- 索引: (name, age, email)
-- 失效：ORDER BY 混用 DESC/ASC
SELECT * FROM users WHERE name = '张三' ORDER BY age DESC, email ASC;

-- 生效：所有列排序方向一致
SELECT * FROM users WHERE name = '张三' ORDER BY age ASC, email ASC;
SELECT * FROM users WHERE name = '张三' ORDER BY age DESC, email DESC;

-- 失效：ORDER BY 使用不在索引中的列
SELECT * FROM users WHERE name = '张三' ORDER BY phone;

-- 失效：WHERE 和 ORDER BY 无法组成最左前缀
SELECT * FROM users WHERE age = 25 ORDER BY name;
```

## 三、索引失效诊断方法

### 3.1 使用 EXPLAIN 分析

```sql
EXPLAIN SELECT * FROM users WHERE YEAR(create_time) = 2024;
```

| 字段 | 含义 | 正常值 |
|------|------|--------|
| `type` | 连接类型 | `ALL` = 全表扫描，`ref` = 索引 |
| `key` | 实际使用的索引 | NULL = 未使用索引 |
| `rows` | 预计扫描行数 | 越少越好 |

### 3.2 使用 OPTIMIZER_TRACE

```sql
SET SESSION optimizer_trace = "enabled=on";
SELECT * FROM users WHERE name LIKE '%张%';
SELECT * FROM information_schema.OPTIMIZER_TRACE;
SET SESSION optimizer_trace = "enabled=off";
```

## 四、常见面试陷阱

:::danger 陷阱 1：认为字符串不加引号不影响索引
错误理解：「`WHERE name = 123` 和 `WHERE name = '123'` 一样」
正确理解：字符串字段与数字比较会导致隐式类型转换，索引失效。
:::

:::danger 陷阱 2：忽略 MySQL 优化器的自动选择
错误理解：「建了索引就一定会用」
正确理解：当数据量小、统计信息不准、或者优化器判断全表扫描更快时，可能会放弃索引。
:::

:::danger 陷阱 3：LIKE 查询不加前缀%
错误理解：「`LIKE '%张'` 和 `LIKE '张%'` 效果一样」
正确理解：只有前缀匹配可以利用索引，后缀匹配无法利用索引。
:::

## 五、索引设计最佳实践

### 5.1 避免索引失效的原则

| 原则 | 说明 |
|------|------|
| **不在索引列上使用函数** | 将函数操作移到右边 |
| **类型要匹配** | 避免隐式类型转换 |
| **避免 OR 混用** | OR 条件两边都要有索引 |
| **控制 LIKE 模式** | 尽量使用前缀匹配 |
| **ORDER BY 遵循最左前缀** | 与 WHERE 条件组合 |

### 5.2 正确写法对比

| 错误写法 | 正确写法 |
|----------|----------|
| `WHERE YEAR(d) = 2024` | `WHERE d >= '2024-01-01' AND d < '2025-01-01'` |
| `WHERE id + 1 = 100` | `WHERE id = 99` |
| `WHERE name = 123` | `WHERE name = '123'` |
| `WHERE name LIKE '%张%'` | `WHERE name LIKE '张%'` 或全文索引 |
| `WHERE id NOT IN (1,2,3)` | `WHERE id < 1 OR id > 3` 或 `WHERE NOT EXISTS` |

## 六、加分回答

> 💡 **函数索引（MySQL 8.0+）**：
> MySQL 8.0 支持在表达式上创建索引，可以解决函数导致索引失效的问题：
>
> ```sql
> -- 创建函数索引
> CREATE INDEX idx_year ON orders(YEAR(create_time));
>
> -- 现在可以正常使用索引
> SELECT * FROM orders WHERE YEAR(create_time) = 2024;
> ```

> 💡 **ICP（索引条件下推）优化**：
> MySQL 5.6+ 的索引下推优化，可以在索引遍历时过滤数据，减少回表次数：
>
> ```sql
> -- 索引: (name, age)
> SELECT * FROM users WHERE name LIKE '张%' AND age = 25;
> ```
> - 无 ICP：先找到所有 `name LIKE '张%'`，再回表过滤 `age = 25`
> - 有 ICP：在索引树遍历时直接过滤 `age = 25`，减少回表

## 七、总结对比表

| 失效场景 | 原因 | 解决方案 |
|----------|------|----------|
| 函数/表达式 | 索引列值被改变，无法利用有序性 | 使用原始列值比较 |
| 类型转换 | 隐式转换导致索引失效 | 确保类型一致 |
| OR 混用 | OR 条件需要两边索引 | 使用 UNION 或补充索引 |
| LIKE '%后缀' | 无法利用前缀有序性 | 使用全文索引 |
| 跳过中间列 | 违反最左前缀原则 | 按顺序使用索引列 |
| NOT NULL/IN | 优化器选择全表扫描 | 使用 EXISTS/JOIN 替代 |
