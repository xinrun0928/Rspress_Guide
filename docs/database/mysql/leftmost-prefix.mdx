# 联合索引最左前缀原则

> 面试官问：「有一个联合索引 `(name, age, email)`，查询 `WHERE age = 25` 能用到索引吗？」你说「能」——面试官直接说「错了」。这不是刁难，这是 P6 MySQL 面试的基础题，也是最容易丢分的送分题。

## 面试官最关心的 3 个问题（快速自测）

| 问题 | 考察点 | 难度 |
|------|--------|------|
| 联合索引 `(a, b, c)` 查询 `WHERE b = 1` 能用索引吗？ | 最左前缀 | 🔴 高频 |
| 为什么需要遵循最左前缀原则？ | 原理理解 | 🔴 高频 |
| `(a, b, c)` 和 `(a, c)` 索引有什么区别？ | 索引设计 | 🟡 中频 |

## 一、最左前缀原则原理

### 1.1 联合索引的 B+Tree 结构

```sql
CREATE INDEX idx_name_age_email ON users(name, age, email);
```

```mermaid
graph TD
    subgraph "联合索引 B+Tree (name, age, email)"
        A[根节点<br/>排序: name→age→email]
        A --> B[页1<br/>('张三', 25, 'a@x.com', id=1)<br/>('张三', 30, 'b@x.com', id=5)<br/>('王五', 28, 'c@x.com', id=3)]
        A --> C[页2<br/>('王五', 35, 'd@x.com', id=7)<br/>('赵六', 22, 'e@x.com', id=2)]
    end
```

**索引排序规则**：按 `(name, age, email)` 三级排序
1. 先按 `name` 排序
2. `name` 相同则按 `age` 排序
3. `name` 和 `age` 都相同则按 `email` 排序

### 1.2 排序示例

```
('张三', 25, 'a@x.com')    -- 张三中年龄最小的
('张三', 30, 'b@x.com')    -- 张三中年龄较大的
('王五', 28, 'c@x.com')    -- 王五中年龄较小的
('王五', 35, 'd@x.com')    -- 王五中年龄较大的
('赵六', 22, 'e@x.com')
```

### 1.3 最左前缀原则定义

> 索引可以用于 `WHERE` 条件列的查找，前提是**从索引的最左列开始连续使用**。

## 二、索引使用情况分析

### 2.1 能否使用索引

| SQL 语句 | 使用索引情况 | 原因 |
|----------|-------------|------|
| `WHERE name = '张三'` | ✅ 完全命中 | 从最左列开始 |
| `WHERE name = '张三' AND age = 25` | ✅ 完全命中 | 使用前两列 |
| `WHERE name = '张三' AND age = 25 AND email = 'a@x.com'` | ✅ 完全命中 | 使用全部三列 |
| `WHERE name LIKE '张%'` | ✅ 命中前缀 | 左前缀匹配 |
| `WHERE age = 25` | ❌ 不命中 | **跳过最左列** |
| `WHERE age = 25 AND email = 'a@x.com'` | ❌ 不命中 | **跳过最左列** |
| `WHERE name = '张三' AND email = 'a@x.com'` | ⚠️ 部分命中 | 跳过中间列，只用 name |

### 2.2 为什么跳过列会导致索引失效？

```mermaid
graph TD
    A[索引: (name, age, email)]
    B[查询: WHERE age = 25]

    A --> C[按 name 排序的数据]
    C --> D[张三组: (张三,25), (张三,30)]
    C --> E[王五组: (王五,28), (王五,35)]

    D --> D1[按 age 排序]
    E --> E1[按 age 排序]

    D1 --> F[age=25 分散在各组]
    E1 --> F

    F --> G[无法定位，必须全表扫描]
```

**核心原因**：索引的 B+Tree 按 `(name, age, email)` 排序，如果直接查询 `age`，MySQL 无法利用索引的有序性，必须全表扫描。

## 三、范围查询与索引中断

### 3.1 范围查询会中断索引

```sql
-- 索引: (name, age, email)
SELECT * FROM users WHERE name = '张三' AND age > 25 AND email = 'a@x.com';
```

| 字段 | 查询条件 | 索引状态 |
|------|----------|----------|
| `name` | `= '张三'` | ✅ 精确匹配 |
| `age` | `> 25` | ⚠️ 范围查询，**中断** |
| `email` | `= 'a@x.com'` | ❌ 无法使用 |

### 3.2 范围查询类型

| 类型 | SQL 示例 | 是否中断索引 |
|------|----------|--------------|
| `=` 等值 | `name = '张三'` | 不中断 |
| `>` / `>=` | `age > 25` | **中断** |
| `<` / `<=` | `age < 25` | **中断** |
| `BETWEEN` | `age BETWEEN 20 AND 30` | **中断** |
| `IN` | `age IN (20, 25, 30)` | 不中断（可多个等值） |
| `LIKE '前缀%'` | `name LIKE '张%'` | 不中断 |
| `LIKE '%后缀'` | `name LIKE '%三'` | **中断** |

### 3.3 优化建议

```sql
-- 索引: (name, age, email)
-- 查询: name='张三' AND age > 25 AND email = 'a@x.com'

-- 方案1：调整 WHERE 顺序（MySQL 优化器会重排，无需手动调整）
SELECT * FROM users WHERE name = '张三' AND email = 'a@x.com' AND age > 25;

-- 方案2：使用覆盖索引，避免回表
SELECT name, age, email FROM users WHERE name = '张三' AND age > 25;

-- 方案3：如果 age 范围查询是高频场景，考虑单独建立 (name, age) 索引
```

## 四、常见面试陷阱

:::danger 陷阱 1：认为 IN 查询会中断索引
错误理解：「IN 和 BETWEEN 一样会中断索引」
正确理解：`IN` 本质是多个等值查询，不会中断索引。例如 `name = '张三' AND age IN (25, 30)` 可以命中索引的 `(name, age)` 部分。
:::

:::danger 陷阱 2：混淆 LIKE 查询的前缀和后缀
错误理解：「`LIKE '%张%'` 可以用索引」
正确理解：只有 `LIKE '张%'` 前缀匹配可以用索引，因为可以利用 B+Tree 的有序性。后缀匹配 `%张` 或模糊匹配 `%张%` 无法利用索引。
:::

:::danger 陷阱 3：忽略 OR 条件的索引使用
错误理解：「只要有索引字段就能用索引」
正确理解：`OR` 条件要求两边都有索引，且字段类型一致，否则会走全表扫描。例如 `WHERE name = '张三' OR age = 25`，如果 `name` 和 `age` 上都有索引，可能走索引合并；如果只有 `name` 有索引，则全表扫描。
:::

## 五、索引设计原则

### 5.1 如何选择联合索引列顺序

**原则**：将**区分度高**、**查询频率高**的列放在前面。

| 考虑因素 | 说明 | 示例 |
|----------|------|------|
| **区分度** | 列的唯一值越多，区分度越高 | `email` 区分度 > `status` |
| **查询频率** | 经常作为查询条件的列优先 | `user_id` 频繁查询则靠前 |
| **空间大小** | 占用空间小的列优先 | TINYINT > VARCHAR(100) |

### 5.2 索引设计案例

**场景**：订单表 `orders`

```sql
CREATE TABLE orders (
    id BIGINT PRIMARY KEY,
    user_id BIGINT,
    status TINYINT,        -- 1:待支付 2:已支付 3:已发货 4:已完成
    amount DECIMAL(10,2),
    create_time DATETIME,
    INDEX idx_user (user_id),
    INDEX idx_status (status)
);

-- 高频查询:
-- Q1: SELECT * FROM orders WHERE user_id = ? AND status = 2;
-- Q2: SELECT * FROM orders WHERE user_id = ? AND create_time > ?;
-- Q3: SELECT * FROM orders WHERE status = 2 AND create_time > ?;
```

**分析**：
- `Q1` 频繁执行，需要 `(user_id, status)` 联合索引
- `Q2` 频繁执行，需要 `(user_id, create_time)` 联合索引
- `Q3` 偶尔查询，`status` 区分度低（只有 4 种值），不单独建索引

**优化方案**：
```sql
-- 优先满足高频查询 Q1 和 Q2
CREATE INDEX idx_user_status ON orders(user_id, status);
CREATE INDEX idx_user_time ON orders(user_id, create_time DESC);

-- 覆盖索引优化，减少回表
CREATE INDEX idx_user_status_cover ON orders(user_id, status, create_time);
```

### 5.3 索引数量控制

| 场景 | 索引数量 | 原因 |
|------|----------|------|
| 写入频繁 | 少 | 每条写入都要维护所有索引 |
| 查询频繁 | 可适当多 | 加速查询 |
| 混合场景 | 权衡 | 参考查询频率和写入频率 |

## 六、EXPLAIN 验证索引使用

```sql
EXPLAIN SELECT * FROM users WHERE name = '张三' AND age = 25;
```

关键字段解析：

| 字段 | 含义 | 正常值 |
|------|------|--------|
| `type` | 连接类型 | `ref`（索引等值查询） |
| `key` | 实际使用的索引 | `idx_name_age_email` |
| `key_len` | 索引使用长度 | 可判断用到几列 |
| `rows` | 预计扫描行数 | 越少越好 |
| `Extra` | 附加信息 | `Using index`=覆盖索引 |

### key_len 计算

```
索引: (name VARCHAR(50), age INT, email VARCHAR(50))

name: VARCHAR(50) + 2(长度) = 52 字节
age:  INT = 4 字节
email: VARCHAR(50) + 2 = 52 字节

key_len = 52  (只用 name)
key_len = 56  (用 name + age)
key_len = 108 (用 name + age + email)
```

## 七、加分回答

> 💡 **索引跳跃扫描（Index Skip Scan）**：
> MySQL 8.0 引入了索引跳跃扫描优化，允许在某些情况下跳过索引的第一列。
>
> ```sql
> -- 索引: (status, name)
> -- 查询: SELECT name FROM orders WHERE status = 2;
> ```
>
> 传统理解：必须从 status 开始，否则索引失效
> 优化后：MySQL 8.0 可能选择索引跳跃扫描，先定位每个 status 值，再回表获取 name

> 💡 **列值截断与前缀索引**：
> 如果字符串列很长（如 `VARCHAR(255)` 的 URL），可以创建前缀索引：
> ```sql
> CREATE INDEX idx_url ON orders(url(10));  -- 只索引前10个字符
> ```
> 优点：减少索引存储空间
> 缺点：前缀匹配，可能降低区分度

## 八、总结对比表

| 查询条件 | 是否使用索引 | 说明 |
|----------|-------------|------|
| `name = '张三'` | ✅ | 从最左列开始 |
| `name IN ('张三', '李四')` | ✅ | IN 本质是多个等值 |
| `name = '张三' AND age = 25` | ✅ | 连续使用前两列 |
| `name LIKE '张%'` | ✅ | 前缀匹配 |
| `age = 25` | ❌ | 跳过最左列 |
| `email = 'a@x.com'` | ❌ | 跳过前两列 |
| `name > '张三'` | ⚠️ 部分 | 范围查询，中断后续列 |

| 范围类型 | 是否中断后续列 |
|----------|----------------|
| `=` / `IN` | 不中断 |
| `>` / `>=` / `<` / `<=` | **中断** |
| `BETWEEN` | **中断** |
| `LIKE '前缀%'` | 不中断 |
| `LIKE '%后缀'` / `%模糊%'` | **中断** |
