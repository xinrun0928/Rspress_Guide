# MRR 优化：减少随机 IO 的利器

你有没有遇到过这种情况：明明查询使用了索引，但 `EXPLAIN` 显示还是要回表几十万次，查询速度依然很慢？

这背后的问题是：**回表操作本身可能是随机 IO**。

MySQL 的 MRR（Multi-Range Read，多范围读）优化，正是为了解决这个问题。

## 什么是回表与随机 IO？

在讲 MRR 之前，先回顾一下回表的概念。

### 回表的过程

非聚簇索引查询时，需要先查索引树找到主键，再回表查聚簇索引获取完整数据：

```
┌─────────────────────────────────────────────────────────────────┐
│                    非聚簇索引查询过程                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  SELECT * FROM users WHERE name = '张三';                       │
│                                                                 │
│  假设 name 有索引（非聚簇索引），name='张三' 的记录有 100 条      │
│                                                                 │
│  Step 1: 在 name 索引树中查找                                    │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │              name 索引树（B+ 树）                         │   │
│  │                                                         │   │
│  │                    [张]                                  │   │
│  │                  /     \                                │   │
│  │               [李]     [王]                              │   │
│  │              /   \    /   \                              │   │
│  │          [张飞] [张力] [张强] [张三]                     │   │
│  │                                                         │   │
│  │  找到 5 条记录：[1, 5, 23, 56, 100]  ← 主键值           │   │
│  └─────────────────────────────────────────────────────────┘   │
│                          │                                      │
│                          │ 5 个主键值                            │
│                          ▼                                      │
│  Step 2: 逐个回表查询完整数据                                    │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  主键=1   ──→ 随机读取数据页 1                           │   │
│  │  主键=5   ──→ 随机读取数据页 1  （可能是同一页）         │   │
│  │  主键=23  ──→ 随机读取数据页 3  ← 随机 IO！             │   │
│  │  主键=56  ──→ 随机读取数据页 8  ← 随机 IO！             │   │
│  │  主键=100 ──→ 随机读取数据页 15 ← 随机 IO！             │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 随机 IO 的问题

每个主键值对应数据页的位置是随机的（取决于插入顺序）。100 条记录可能分散在 100 个不同的数据页，需要 100 次随机 IO。

随机 IO 是数据库的性能杀手——**顺序 IO 比随机 IO 快 100 倍以上**。

## MRR 的核心思想

MRR 的核心思想很简单：**把随机 IO 变成顺序 IO**。

### MRR 的工作原理

```
┌─────────────────────────────────────────────────────────────────┐
│                    MRR 优化原理                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  传统回表（随机 IO）：                                            │
│  主键=[23, 56, 1, 100, 5]  ──→ 随机读取 5 个页面                  │
│                                                                 │
│  MRR 优化后（顺序 IO）：                                          │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  Step 1: 收集所有主键值                                   │   │
│  │  主键=[23, 56, 1, 100, 5]                                 │   │
│  │                          │                                 │   │
│  │  Step 2: 排序（MRR 的关键！）                            │   │
│  │  排序后=[1, 5, 23, 56, 100]  ← 按主键顺序                 │   │
│  │                          │                                 │   │
│  │  Step 3: 批量回表（顺序 IO）                              │   │
│  │  数据页=[1, 1, 3, 8, 15]  ← 顺序访问！                    │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### MRR 的数据结构：RowID 排序

MRR 在 MySQL 内部使用一个缓冲区来暂存和排序主键值：

```
┌─────────────────────────────────────────────────────────────────┐
│                    MRR 工作流程                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────┐     ┌─────────────┐     ┌─────────────┐       │
│  │  索引扫描    │ ──▶ │  收集主键   │ ──▶ │  排序主键   │       │
│  │  得到主键   │     │  到 buffer  │     │  (MRR buffer)│       │
│  └─────────────┘     └─────────────┘     └──────┬──────┘       │
│                                                  │              │
│                                                  ▼              │
│                   ┌─────────────┐     ┌─────────────┐          │
│                   │  顺序回表   │ ◀── │  批量读取   │          │
│                   │  读数据    │     │  主键范围   │          │
│                   └─────────────┘     └─────────────┘          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## MRR 的触发条件

### 开启 MRR

MRR 默认是开启的：

```sql
-- 查看 MRR 是否开启
SHOW VARIABLES LIKE 'mrr';
-- 或
SHOW VARIABLES LIKE 'optimizer_switch';
-- 查找 'mrr=on' 或 'mrr_cost_based=on'
```

```
optimizer_switch 默认值：
index_merge=on,index_merge_union=on,index_merge_sort_union=on,
index_merge_intersection=on,engine_condition_pushdown=on,
**mrr=on,mrr_cost_based=on**,...
```

### MRR 触发的条件

| 条件 | 说明 |
|---|---|
| 使用非聚簇索引 | 辅助索引查询需要回表 |
| 索引列与查询条件 | 索引能够筛选数据 |
| `mrr=on` | MRR 功能开启 |
| `mrr_cost_based=on` | 优化器基于代价决定是否使用 |

```sql
-- MRR 生效的典型场景
SELECT * FROM users WHERE age = 25;
-- age 有索引，但 SELECT * 需要回表获取所有列

SELECT * FROM users WHERE name = '张三' AND age > 20;
-- name 有索引，但 SELECT * 需要回表
```

### MRR 不生效的场景

| 场景 | 原因 |
|---|---|
| 覆盖索引查询 | `SELECT` 列都在索引中，无需回表 |
| 聚簇索引查询 | 主键查询不需要回表 |
| MRR 关闭 | `set global optimizer_switch='mrr=off'` |
| 结果集很小 | 排序开销可能大于随机 IO 节省 |

```sql
-- 不会触发 MRR 的查询
SELECT name FROM users WHERE name = '张三';
-- 覆盖索引，不需要回表

SELECT * FROM users WHERE id = 100;
-- 主键查询，直接定位，不需要回表
```

## MRR 参数配置

### 相关配置项

```sql
-- 查看 MRR buffer 大小
SHOW VARIABLES LIKE 'mrr_buffer_size';
-- 默认值：262144 字节（256KB）

-- 设置 MRR buffer 大小
SET SESSION optimizer_switch = 'mrr_cost_based=off';
-- 或在配置文件中
```

```
mrr_cost_based 选项：
- ON：优化器基于代价决定是否使用 MRR（可能不用）
- OFF：强制使用 MRR（如果 MRR 可用）
```

### 参数调优建议

| 参数 | 默认值 | 调整建议 |
|---|---|---|
| `mrr` | on | 保持开启 |
| `mrr_cost_based` | on | 大数据量时设为 off |
| `mrr_buffer_size` | 256KB | 结果集大时增大 |

```ini
# my.cnf 配置
[mysqld]
# 强制使用 MRR
optimizer_switch = 'mrr=on,mrr_cost_based=off'
# 增大 MRR buffer
read_rnd_buffer_size = 4M  # 这个参数同时影响 MRR
```

## MRR 与其他优化

### MRR 与 Index Condition Pushdown（ICP）

MRR 和 ICP 是两个独立的优化，可以同时生效：

```
┌─────────────────────────────────────────────────────────────────┐
│                 MRR + ICP 协同优化                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  SELECT * FROM users WHERE name = '张%' AND age > 20;          │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  Step 1: ICP - 索引条件下推                              │   │
│  │  在索引树遍历时，用 age > 20 过滤数据                    │   │
│  │  只收集满足 name='张%' AND age > 20 的主键              │   │
│  └─────────────────────────────────────────────────────────┘   │
│                          │                                      │
│                          ▼                                      │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  Step 2: MRR - 多范围读优化                              │   │
│  │  收集主键 → 排序 → 顺序回表                              │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### MRR 与批次处理

MRR buffer 满了怎么办？分批处理：

```
┌─────────────────────────────────────────────────────────────────┐
│                 MRR 分批处理                                     │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  假设要查询 10000 条记录，MRR buffer = 256KB                     │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  主键集合：[1, 2, 3, ..., 10000]                        │   │
│  │                          │                              │   │
│  │  分批处理：                                               │   │
│  │  ┌──────────────────────────────────────────────────┐  │   │
│  │  │  Batch 1: [1...1000] → 排序 → 回表                │  │   │
│  │  │  Batch 2: [1001...2000] → 排序 → 回表             │  │   │
│  │  │  Batch 3: [2001...3000] → 排序 → 回表            │  │   │
│  │  │  ...                                               │  │   │
│  │  └──────────────────────────────────────────────────┘  │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │

```

## MRR 的性能提升

### 性能对比数据

| 场景 | 无 MRR | 有 MRR | 提升 |
|---|---|---|---|
| 10 万条回表 | 1000ms | 150ms | **6.7 倍** |
| 50 万条回表 | 5000ms | 400ms | **12.5 倍** |
| 范围查询回表 | 3000ms | 200ms | **15 倍** |

### 提升原理

```
┌─────────────────────────────────────────────────────────────────┐
│                 MRR 性能提升原理                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  无 MRR（随机 IO）：                                              │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  1000 次随机 IO，每次 10ms                                │   │
│  │  总时间 = 1000 × 10ms = 10s                              │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  有 MRR（顺序 IO）：                                              │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  1000 次顺序 IO，每次 0.1ms                               │   │
│  │  总时间 = 1000 × 0.1ms = 100ms                           │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  顺序 IO 比随机 IO 快 100 倍！                                    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 面试实战

### 问题一：什么是 MRR？解决了什么问题？

**面试官问**：「MySQL 的 MRR 优化是什么？它解决了什么问题？」

**深入回答**：

> MRR（Multi-Range Read，多范围读）是 MySQL 针对非聚簇索引回表查询的优化。
>
> **解决的问题**：非聚簇索引回表时，主键值是分散的，导致随机 IO。
>
> **解决方案**：
> 1. 先收集所有需要回表的主键值
> 2. 对主键值排序
> 3. 按排序后的顺序批量回表
>
> **效果**：将随机 IO 变成顺序 IO，性能提升数倍到数十倍。

### 问题二：MRR 和 ICP 有什么关系？

**面试官问**：「MRR 和索引条件下推（ICP）有什么区别？它们可以同时使用吗？」

**深入回答**：

> MRR 和 ICP 是两个独立的优化，可以同时使用：
>
> | 优化 | 作用阶段 | 目的 |
> |---|---|---|
> | ICP | 索引扫描阶段 | 在索引层面过滤数据，减少回表次数 |
> | MRR | 回表阶段 | 优化回表的 IO 顺序 |
>
> **执行顺序**：
> 1. 先用 ICP 在索引层面过滤 `age > 20` 的记录
> 2. 收集满足条件的主键值
> 3. MRR 对主键排序
> 4. 顺序回表获取完整数据
>
> 两者协同工作，进一步提升查询性能。

### 问题三：什么情况下 MRR 不会生效？

**面试官问**：「MRR 在什么情况下不会生效？」

**深入回答**：

> MRR 不生效的场景：
>
> 1. **覆盖索引查询**：`SELECT` 的列都在索引中，不需要回表，MRR 无用武之地
> 2. **主键查询**：直接用主键定位，不需要回表
> 3. **MRR 关闭**：`optimizer_switch` 中关闭了 MRR
> 4. **结果集很小**：数据量小时，排序开销可能大于节省的 IO
> 5. **辅助索引查询**：仅当需要回表时才需要 MRR
>
> 判断方法：`EXPLAIN` 的 `Extra` 列显示 `Using MRR` 表示生效。

## 总结

MRR 是 MySQL 优化非聚簇索引回表的重要手段：

- **核心思想**：排序主键值，将随机 IO 变成顺序 IO
- **触发条件**：非聚簇索引回表查询
- **效果**：性能提升数倍到数十倍
- **配合优化**：可与 ICP 同时生效，进一步提升性能
- **监控方法**：`EXPLAIN` 的 `Extra` 列显示 `Using MRR`

理解 MRR 的工作原理，有助于你在面试中回答索引优化和查询性能相关的问题。
