# MySQL 索引体系全览

面试官问：「MySQL 有哪些索引类型？」

很多人会回答：「B+ 树索引、Hash 索引、主键索引、唯一索引。」这个答案只对了一半。MySQL 的索引体系远比这复杂，每种索引类型都有其适用场景和实现原理。

本文带你系统梳理 MySQL 的索引体系，理解每种索引的特点和使用场景。

## 索引分类维度

MySQL 的索引可以从多个维度分类：

| 分类维度 | 索引类型 |
|---|---|
| 数据结构 | B+ 树索引、Hash 索引、全文索引 |
| 存储方式 | 聚簇索引、非聚簇索引 |
| 字段特性 | 主键索引、唯一索引、普通索引 |
| 字段数量 | 单列索引、联合索引 |
| 数据类型 | 空间索引（R-Tree） |

## 按数据结构分类

### B+ 树索引

B+ 树索引是 MySQL 最核心的索引类型，InnoDB 和 MyISAM 都使用 B+ 树作为索引结构。

**B+ 树的特点**：

- 多路平衡查找树，每个节点可以有多个子节点
- 所有数据存储在叶子节点，叶子节点之间通过链表连接
- 非叶子节点只存储索引，不存储数据
- 查询时间复杂度稳定为 O(log n)

```
B+ 树结构示意：

           [15]            [30]
          /    \          /    \
     [5,10]  [15,20]  [25,30]  [35,40]
       │        │        │        │
       ▼        ▼        ▼        ▼
    (数据)    (数据)    (数据)    (数据)  ← 叶子节点链表连接
```

**适用场景**：
- 大多数查询场景
- 范围查询（如 `WHERE age > 20 AND age < 30`）
- 排序查询（如 `ORDER BY create_time`）

### Hash 索引

Hash 索引基于哈希表实现，通过哈希函数计算 bucket，直接定位数据。

**Hash 索引的特点**：

- 查询时间复杂度 O(1)，理论上的最优查找
- 只支持等值查询（`=`、`IN`）
- 不支持范围查询和排序
- 无法利用索引避免回表

```
Hash 索引结构：

Key: "张三"  →  Hash("张三") = 3  →  Bucket[3]  →  指向数据行
Key: "李四"  →  Hash("李四") = 7  →  Bucket[7]  →  指向数据行
```

**适用场景**：
- Memory 存储引擎的默认索引
- InnoDB 的自适应哈希索引（AHI）
- 等值查询为主的场景（如 KV 查找）

:::tip 自适应哈希索引（AHI）

InnoDB 会自动监控索引查询，当发现某些索引被频繁访问时，会自动构建 Hash 索引。这个过程是自动的，无需人工干预。通过 `SHOW ENGINE INNODB STATUS` 可以查看 AHI 的命中率。
:::

### 全文索引

全文索引用于文本搜索，支持 `MATCH ... AGAINST` 语法。

```sql
-- 创建全文索引
ALTER TABLE articles ADD FULLTEXT INDEX ft_title_content (title, content);

-- 使用全文索引搜索
SELECT * FROM articles
WHERE MATCH(title, content) AGAINST('数据库 索引' IN NATURAL LANGUAGE MODE);
```

**全文索引的特点**：
- MyISAM 支持全文索引
- InnoDB 从 5.6 版本开始支持全文索引
- 支持自然语言搜索、布尔搜索
- 不适合精确匹配，主要用于模糊搜索

**MyISAM vs InnoDB 全文索引**：

| 特性 | MyISAM | InnoDB |
|---|---|---|
| 5.6 支持 | 是 | 是 |
| 索引存储 | 独立 FTS 文件 | 索引表 |
| 性能 | 全文索引独立，性能好 | 依赖索引表，稍慢 |
| 事务支持 | 不支持 | 支持 |

### 空间索引（R-Tree）

空间索引用于地理空间数据类型，如 `GEOMETRY`、`POINT`、`POLYGON`。

```sql
-- 创建空间索引
ALTER TABLE locations ADD SPATIAL INDEX sp_location (location);

-- 查询附近的位置
SELECT * FROM locations
WHERE ST_Contains(location, ST_GeomFromText('POINT(116.4 39.9)'));
```

**空间索引的特点**：
- 只有 MyISAM 支持 R-Tree 空间索引
- InnoDB 通过 GIS 函数支持空间查询，但没有专门的 R-Tree 索引
- 适合地图应用、物流系统

## 按存储方式分类

### 聚簇索引

聚簇索引的叶子节点存储完整的数据行。

**InnoDB 聚簇索引**：
- 表数据按主键顺序存储
- 主键索引就是聚簇索引
- 叶子节点包含整行数据
- 每张表只能有一个聚簇索引

### 非聚簇索引

非聚簇索引的叶子节点只存储索引值和主键值，需要回表查询完整数据。

**MyISAM 非聚簇索引**：
- 主键索引和普通索引没有区别
- 叶子节点存储的是行指针（数据在磁盘的位置）
- 索引和数据分开存储

**InnoDB 非聚簇索引（辅助索引）**：
- 所有非主键索引都是非聚簇索引
- 叶子节点存储索引值 + 主键值
- 查询时需要先找到主键，再回表查询完整数据

## 按字段特性分类

### 主键索引

- 主键索引是一种特殊的唯一索引
- 主键值不允许为 NULL
- 每张表只能有一个主键索引
- InnoDB 中，主键索引就是聚簇索引

```sql
CREATE TABLE users (
    id INT PRIMARY KEY,  -- 主键索引
    name VARCHAR(50)
);
```

### 唯一索引

- 索引值必须唯一，可以有 NULL 值（可以有多个 NULL）
- 可以有多个唯一索引

```sql
CREATE TABLE users (
    id INT PRIMARY KEY,
    email VARCHAR(100) UNIQUE,  -- 唯一索引
    phone VARCHAR(20),
    UNIQUE KEY uk_phone (phone)
);
```

### 普通索引

没有任何限制，用于加速查询。

```sql
CREATE TABLE users (
    id INT PRIMARY KEY,
    name VARCHAR(50),
    INDEX idx_name (name)  -- 普通索引
);
```

## 按字段数量分类

### 单列索引

索引只包含一个列。

```sql
CREATE INDEX idx_name ON users(name);
```

### 联合索引

索引包含多个列。

```sql
CREATE INDEX idx_name_age ON users(name, age);
```

联合索引遵循最左前缀原则，关于这部分会在《联合索引与最左前缀原则》中详细讲解。

## 索引选择指南

### 索引类型选择决策树

```
需要什么类型的查询？
    │
    ├── 等值查询（WHERE id = 1）
    │     │
    │     └── B+ 树主键/唯一索引
    │
    ├── 范围查询（WHERE age > 20）
    │     │
    │     └── B+ 树索引
    │
    ├── 文本搜索（WHERE MATCH(content) AGAINST('keyword')）
    │     │
    │     └── 全文索引
    │
    ├── 地理位置查询（WHERE ST_Contains(area, point)）
    │     │
    │     └── 空间索引（MyISAM）或 GIS 函数
    │
    └── 排序查询（ORDER BY create_time）
          │
          └── B+ 树索引
```

### 常见场景的索引建议

| 场景 | 索引设计 |
|---|---|
| 用户表按 ID 查询 | 主键索引 |
| 用户表按手机号查询 | 手机号字段唯一索引 |
| 订单表按用户和状态查询 | 联合索引 (user_id, status) |
| 文章表按标题模糊搜索 | 全文索引 |
| 商户表按地理位置查询 | 空间索引或 GeoHash + 普通索引 |

## 面试实战

### 问题一：为什么 InnoDB 用 B+ 树而不是 B 树？

**面试官问**：「为什么 MySQL 用 B+ 树而不是 B 树作为索引结构？」

**深入回答**：

> B+ 树相比 B 树有几个优势：
>
> 1. **查询稳定性**：B 树的查询复杂度与数据所在深度有关，B+ 树所有数据都在叶子节点，查询深度固定，IO 次数更稳定。
>
> 2. **范围查询效率**：B+ 树叶子节点通过链表连接，范围查询只需遍历链表；B 树需要中序遍历，效率低。
>
> 3. **磁盘 IO 更少**：B+ 树非叶子节点不存储数据，每个节点能容纳更多索引，树的高度更低。
>
> 4. **遍历效率高**：B+ 树直接遍历叶子节点即可获取所有数据，适合全表扫描。

### 问题二：Hash 索引的局限性？

**面试官问**：「Hash 索引有什么局限性？什么时候用 Hash 索引？」

**深入回答**：

> Hash 索引的局限：
>
> 1. **不支持范围查询**：Hash 计算是无序的，无法做 `>`、`<`、`BETWEEN` 等范围查询。
>
> 2. **不支持排序**：无法利用索引进行 `ORDER BY` 排序。
>
> 3. **不支持最左前缀匹配**：对于联合索引，Hash 索引无法利用前面的列。
>
> 4. **哈希冲突**：大量哈希冲突时，退化为链表查询。
>
> 适用场景：
>
> - 等值查询为主的场景
> - KV 存储（Redis、Memcached）
> - InnoDB 的自适应哈希索引（自动优化热查询）

### 问题三：聚簇索引 vs 非聚簇索引？

**面试官问**：「聚簇索引和非聚簇索引的区别是什么？各自的优缺点？」

**深入回答**：

> 核心区别在于叶子节点存储的内容：
>
> **聚簇索引**（InnoDB 主键索引）：
>
> - 叶子节点存储完整数据行
> - 主键查询只需一次 IO
> - 数据按主键顺序存储，范围查询快
> - 一张表只能有一个聚簇索引
>
> **非聚簇索引**（InnoDB 辅助索引）：
>
> - 叶子节点存储索引值和主键值
> - 需要回表查询完整数据（多一次 IO）
> - 可以有多个非聚簇索引
>
> 实际优化中，通过覆盖索引可以避免回表，让非聚簇索引达到聚簇索引的查询性能。

## 总结

MySQL 索引体系：

| 分类 | 索引类型 | 适用场景 |
|---|---|---|
| 数据结构 | B+ 树索引 | 大多数查询、范围查询、排序 |
| 数据结构 | Hash 索引 | 等值查询 |
| 数据结构 | 全文索引 | 文本搜索 |
| 数据结构 | 空间索引 | 地理位置查询 |
| 存储方式 | 聚簇索引 | 主键查询（InnoDB） |
| 存储方式 | 非聚簇索引 | 辅助查询，需回表 |
| 字段特性 | 主键/唯一/普通 | 按需求选择 |

理解索引体系，是后续学习索引原理和索引优化的基础。
