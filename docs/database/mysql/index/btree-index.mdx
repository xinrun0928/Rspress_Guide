# B+ 树索引原理：从页结构到查找过程

「MySQL 索引的底层数据结构是什么？」

这道题几乎是面试必问。十个人里有九个会回答「B+ 树」。但继续追问下去——「B+ 树的页结构是怎样的？」「数据页16KB是怎么存储的？」「页分裂是怎么发生的？」「为什么建议自增主键？」——能答上来的就寥寥无几了。

本文从 InnoDB 存储结构出发，深入讲解 B+ 树的实现原理，帮你彻底理解这个面试高频考点。

## InnoDB 存储结构

理解 B+ 树之前，需要先理解 InnoDB 的存储结构。InnoDB 有两个核心概念：**页（Page）**和**表空间（Tablespace）**。

### 页：InnoDB 的基本存储单位

InnoDB 的数据存储以页为单位，每页默认大小 **16KB**。页是 InnoDB 与磁盘交互的最小单位——读取数据时，整页加载；写入数据时，整页刷盘。

```sql
-- 查看 InnoDB 页大小
SHOW VARIABLES LIKE 'innodb_page_size';
-- 结果：16384 字节（16KB）
```

页的类型有多种，数据页（`FIL_PAGE_INDEX`）是存储索引和数据的主要页类型。

### 表空间

表空间是 InnoDB 管理存储的结构。默认情况下，InnoDB 使用系统表空间（`ibdata1`），每个表可以有自己的独立表空间（`.ibd` 文件）。

```sql
-- 开启独立表空间
innodb_file_per_table = ON

-- 每个表一个 .ibd 文件
-- /var/lib/mysql/database/table_name.ibd
```

## 数据页结构

数据页是 B+ 树中存储数据的基本单位，其结构如下：

```
┌─────────────────────────────────────────────────────────────────┐
│                         File Header（38 字节）                   │
│  ┌─────────────┬─────────────┬─────────────┬─────────────┐     │
│  │ FIL_PAGE_OFFSET │ FIL_PAGE_TYPE │ FIL_PAGE_PREV │ FIL_PAGE_NEXT │ │
│  └─────────────┴─────────────┴─────────────┴─────────────┘     │
│  FIL_PAGE_SPACE_OR_CHKSUM │ ...                                  │
└─────────────────────────────────────────────────────────────────┘
┌─────────────────────────────────────────────────────────────────┐
│                       Page Header（56 字节）                      │
│  ┌─────────────┬─────────────┬─────────────┬─────────────┐     │
│  │ PAGE_N_DIR_SLOTS │ PAGE_HEAP_TOP │ PAGE_FREE │ PAGE_N_HEAP │   │
│  └─────────────┴─────────────┴─────────────┴─────────────┘     │
│  PAGE_FORMAT | PAGE_N_RECS | PAGE_MAX_TRX_ID | ...             │
└─────────────────────────────────────────────────────────────────┘
┌─────────────────────────────────────────────────────────────────┐
│                     Infimum + Supremum（26 字节）                │
│  ┌───────────────────────────┬───────────────────────────┐     │
│  │      Infimum（虚拟记录）    │    Supremum（虚拟记录）    │     │
│  │  记录页中最小的主键值       │    记录页中最大的主键值    │     │
│  └───────────────────────────┴───────────────────────────┘     │
└─────────────────────────────────────────────────────────────────┘
┌─────────────────────────────────────────────────────────────────┐
│                       User Records（数据记录）                   │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐              │
│  │ Record1 │ │ Record2 │ │ Record3 │ │ Record4 │  ...        │
│  └─────────┘ └─────────┘ └─────────┘ └─────────┘              │
│                                                                 │
│  每条记录包含：                                                   │
│  - 记录头信息（可变）                                             │
│  - 变长字段长度列表                                               │
│  - NULL 值列表                                                   │
│  - 隐藏列（DB_ROW_ID, DB_TRX_ID, DB_ROLL_PTR）                   │
│  - 列数据                                                        │
└─────────────────────────────────────────────────────────────────┘
┌─────────────────────────────────────────────────────────────────┐
│                       Free Space（空闲空间）                     │
│                         尚未分配的页空间                          │
└─────────────────────────────────────────────────────────────────┘
┌─────────────────────────────────────────────────────────────────┐
│                       Page Directory（页目录）                   │
│  ┌───────┬───────┬───────┬───────┬───────┐                    │
│  │ Slot1 │ Slot2 │ Slot3 │ Slot4 │ Slot5 │  每组 4-8 条记录   │
│  │   ↘   │   ↘   │   ↘   │   ↘   │   ↘   │                    │
│  │  Rec  │  Rec  │  Rec  │  Rec  │  Rec  │                    │
│  └───────┴───────┴───────┴───────┴───────┘                    │
└─────────────────────────────────────────────────────────────────┘
┌─────────────────────────────────────────────────────────────────┐
│                       File Trailer（8 字节）                     │
│                     用于检测页是否完整刷盘                         │
└─────────────────────────────────────────────────────────────────┘
```

### 记录头信息

每条用户记录都有一个记录头，包含关键信息：

| 字段 | 大小 | 说明 |
|---|---|---|
| `delete_mask` | 1 位 | 记录是否被删除 |
| `min_rec_mask` | 1 位 | 是否为最小记录 |
| `n_owned` | 4 位 | 当前组拥有的记录数 |
| `heap_no` | 13 位 | 在堆中的序号 |
| `record_type` | 3 位 | 记录类型（0=普通，1=B+树节点指针，2=Infimum，3=Supremum） |
| `next_record` | 16 位 | 下一条记录的偏移量 |

### 隐藏列

InnoDB 为每条记录自动添加三个隐藏列：

| 隐藏列 | 大小 | 说明 |
|---|---|---|
| `DB_ROW_ID` | 6 字节 | 行 ID，如果没有主键则用它作为聚簇索引 |
| `DB_TRX_ID` | 6 字节 | 事务 ID，用于 MVCC |
| `DB_ROLL_PTR` | 7 字节 | 回滚指针，指向 Undo Log |

### 页目录（Page Directory）

页目录用于二分查找加速。页被分成多个组，每组最后一个记录（槽）的地址记录在页目录中。

```
页内查找过程：
┌─────────────────────────────────────────┐
│  页目录                                  │
│  ┌────┬────┬────┬────┬────┐             │
│  │ S1 │ S2 │ S3 │ S4 │ S5 │  Slot 地址   │
│  └──┬─┴─┬──┴─┬──┴─┬──┴─┬──┘             │
│     │  │    │    │    │                   │
│     ▼  ▼    ▼    ▼    ▼                   │
│  [10][15][20][25][30]  组内最大主键值      │
│   1    2    3    4    5    组序号          │
└─────────────────────────────────────────┘

查找主键=20：
1. 二分查找页目录，确定在 S3 和 S4 之间
2. 从 S3 开始顺序扫描，找到第一条 >= 20 的记录
```

## B+ 树的页结构

InnoDB 的 B+ 树由多页组成，分为**索引页**（非叶子节点）和**数据页**（叶子节点）。

### 非叶子节点（索引页）

非叶子节点存储索引记录，每条记录包含：

```
索引页记录结构：
┌─────────────────────────────────────────┐
│  索引页记录                              │
│  ┌───────────────────────────────────┐ │
│  │ 主键值（比如 30）                  │ │
│  └───────────────────────────────────┘ │
│  ┌───────────────────────────────────┐ │
│  │ 子节点的页号（比如 0xFF35）         │ │
│  └───────────────────────────────────┘ │
└─────────────────────────────────────────┘
```

### 叶子节点（数据页）

叶子节点存储完整的数据行（对于聚簇索引）或索引值+主键（对于非聚簇索引）。

```
叶子节点数据行：
┌─────────────────────────────────────────┐
│  数据行（聚簇索引）                       │
│  ┌───────────────────────────────────┐ │
│  │ 主键值：1                          │ │
│  │ name: 张三                          │ │
│  │ age: 25                            │ │
│  │ address: 北京                       │ │
│  │ ...（完整行数据）                   │ │
│  └───────────────────────────────────┘ │
│                                         │
│  叶子节点链表连接：                       │
│  [数据页A] ←→ [数据页B] ←→ [数据页C]      │
└─────────────────────────────────────────┘
```

### B+ 树查找过程

```
B+ 树查找主键=20 的记录：

                    ┌──────────┐
                    │   根页    │
                    │  [15|30] │   ← 非叶子节点：索引
                    └────┬─────┘
              ┌──────────┴──────────┐
              │                     │
        ┌─────▼─────┐          ┌─────▼─────┐
        │   页 1    │          │   页 2    │
        │ [10|15]  │          │ [30|35]  │   ← 非叶子节点
        └─────┬─────┘          └─────┬─────┘
              │                     │
    ┌─────────┼─────────┐   ┌────────┼────────┐
    │         │         │   │        │        │
┌───▼───┐ ┌───▼───┐ ┌───▼───┐ ┌───▼───┐ ┌───▼───┐
│数据页A│ │数据页B│ │数据页C│ │数据页D│ │数据页E│
│[1-10] │ │[11-20]│ │[21-30]│ │[31-40]│ │[41-50]│  ← 叶子节点
└───────┘ └───┬───┘ └───────┘ └───────┘ └───────┘
               │
         找到了！主键=20
```

查找过程：

1. 从根页开始，比较主键值，确定在左子树还是右子树
2. 递归向下，直到找到叶子节点
3. 在叶子节点中二分查找定位具体记录
4. 返回完整数据行

### B+ 树的范围查询

B+ 树的优势之一是范围查询高效：

```sql
SELECT * FROM users WHERE age BETWEEN 20 AND 30;
```

```
范围查询过程：

1. 找到 age=20 的起始位置（定位叶子节点 + 二分查找）
2. 沿着叶子节点链表顺序遍历
3. 直到遇到第一条 age > 30 的记录，停止遍历
```

链表遍历的范围查询，时间复杂度 O(log n + k)，其中 k 是结果集大小。

## 页分裂与页合并

### 页分裂

当向已满的页插入数据时，会发生页分裂。

```
初始状态：页 A 已满（16KB），要插入主键=50

    ┌─────────────────────────────────┐
    │          页 A（已满）            │
    │  主键: 10, 20, 30, 40, 45       │
    │  (无法再插入)                   │
    └─────────────────────────────────┘
                  │
                  │ 插入主键=50
                  ▼
    ┌─────────────────────────────────┐
    │          页 A                    │
    │  主键: 10, 20, 30               │  ← 原数据的前半部分
    └─────────────────────────────────┘
    ┌─────────────────────────────────┐
    │          页 B（新建）            │
    │  主键: 40, 45, 50               │  ← 原数据的后半部分 + 新数据
    └─────────────────────────────────┘
                  │
                  ▼
    更新父节点索引：[30] 指向新页 B
```

### 为什么建议自增主键？

使用自增主键时，新插入的主键总是大于现有主键，**页分裂只发生在最后**：原数据不用移动，只在最后追加新页。

使用 UUID 或非顺序主键时，新插入的主键可能插入到任意位置，**页分裂可能发生在任意位置**：原数据需要移动，效率低，页利用率不稳定。

### 页合并

当删除数据导致页空间利用率过低时，InnoDB 会合并相邻的页。

```sql
-- 删除大量数据后，可能触发页合并
DELETE FROM users WHERE id < 1000;
```

## B+ 树的高度

B+ 树的高度直接影响查询 IO 次数。

假设：

- InnoDB 页大小：16KB
- 主键为 BIGINT：8 字节
- 索引页每条记录：8 字节主键 + 6 字节页号 = 14 字节
- 加上记录头开销，假设每条记录 20 字节

**非叶子节点能容纳的记录数**：`16KB / 20 = 819` 条

**树高度计算**：

| 高度 | 能存储的记录数 |
|---|---|
| 1 | 819 条 |
| 2 | 819 × 819 ≈ 67 万条 |
| 3 | 819³ ≈ 5.5 亿条 |

:::tip B+ 树高度通常不超过 3

这就是为什么 InnoDB 查询通常只需 1-3 次磁盘 IO：
- 高度 1：根页常驻内存，只需 1 次 IO
- 高度 2：需要 2 次 IO
- 高度 3：需要 3 次 IO

即使 5 亿条记录，也只需 3 次 IO。这正是 B+ 树高效的原因。
:::

## 面试实战

### 问题一：B+ 树 vs B 树的区别

**面试官问**：「B+ 树和 B 树有什么区别？为什么 MySQL 用 B+ 树而不是 B 树？」

**深入回答**：

> 核心区别：
>
> 1. **数据存储位置**：B 树的所有节点都存储数据，B+ 树只有叶子节点存储数据。非叶子节点不存储数据，可以容纳更多索引，树高更低。
>
> 2. **范围查询**：B 树做范围查询需要中序遍历，B+ 树只需遍历叶子节点的链表。
>
> 3. **查询稳定性**：B 树的查询复杂度与数据深度有关，B+ 树所有数据在叶子节点，深度固定，IO 次数稳定。
>
> MySQL 选择 B+ 树的原因：
>
> - 磁盘 IO 友好：非叶子节点更小，树高更低
> - 范围查询高效：叶子链表便于范围遍历
> - 查询稳定：IO 次数可预估

### 问题二：页分裂的过程

**面试官问**：「什么是页分裂？为什么自增主键更好？」

**深入回答**：

> 页分裂发生在向已满的页插入数据时：
>
> 1. 申请一个新的页
> 2. 将原页的后半部分数据移动到新页
> 3. 在原页或新页中插入新数据
> 4. 更新父节点的索引
>
> 自增主键的优势：
>
> - 新数据总是插入到最后，避免随机 IO
> - 只在最后追加新页，不影响已有数据
> - 写入顺序与物理存储顺序一致，写入性能高
>
> 非自增主键的问题：
>
> - 新数据可能插入到任意位置
> - 页分裂时需要移动大量数据
> - 写入顺序与物理存储顺序不一致

### 问题三：为什么建议主键要短？

**面试官问**：「为什么建议主键要短？主键太长有什么影响？」

**深入回答**：

> 主键太长的影响：
>
> 1. **索引页利用率下降**：每条索引记录都要存储主键值，主键越长，每页能存储的索引项越少，树越高。
>
> 2. **所有非聚簇索引都要存储主键**：InnoDB 的非聚簇索引叶子节点存储「索引值 + 主键值」，主键越长，非聚簇索引越大。
>
> 3. **内存占用增加**：索引和数据都要加载到 Buffer Pool，主键长会占用更多内存。
>
> 建议：
>
> - 使用自增 BIGINT 作为主键（8 字节，够用）
> - 避免用长字符串（如 VARCHAR(100) 的 UUID）作为主键

## 总结

B+ 树是 MySQL 索引的核心：

- **页结构**：数据页是 InnoDB 的基本存储单位，16KB 大小，包含页头、用户记录、页目录等
- **B+ 树组成**：非叶子节点存储索引，叶子节点存储数据（聚簇索引）或索引+主键（非聚簇索引）
- **查找过程**：从根页开始，逐层向下，最终在叶子节点找到数据
- **页分裂**：当页满时分裂，自增主键可以避免频繁分裂
- **树高度**：通常不超过 3，查询只需 1-3 次磁盘 IO

理解 B+ 树原理，是掌握索引优化的基础。
