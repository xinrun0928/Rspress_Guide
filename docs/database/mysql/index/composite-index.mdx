# 联合索引与最左前缀原则

「明明建了索引，SQL 怎么还是全表扫描？」

面试中这个问题一抛出去，十个人里有八个会说「索引没生效」。但追问下去：什么情况下索引会失效？联合索引的结构是什么样的？最左前缀原则到底「最左」指的是什么？

能把这些问题讲清楚的人，并不多。

## 联合索引的底层结构：B+ 树的延伸

要理解最左前缀原则，先得搞清楚联合索引是怎么存储的。

单列索引的 B+ 树，键值对是 `(列值, 主键ID)`。联合索引呢？其实是一样的逻辑，只不过键值变成了多个列的组合：

```sql
-- 创建联合索引
CREATE INDEX idx_name_age_dept ON employees(name, age, dept_id);
```

这颗 B+ 树按什么顺序存储？答案是：**按创建索引时列的顺序，依次比较**。

| 存储结构 | name | age | dept_id |
| --- | --- | --- | --- |
| 第一层 | `Alice` | 25 | 101 |
| 第一层 | `Alice` | 30 | 102 |
| 第一层 | `Bob` | 22 | 201 |
| 第一层 | `Charlie` | 28 | 301 |

树先按 `name` 排序，`name` 相同时按 `age` 排序，`name` 和 `age` 都相同时按 `dept_id` 排序。

## 最左前缀原则：查左不查右

理解了存储结构，最左前缀原则就很好理解了。

**最左前缀原则**：从索引的最左边列开始，连续地使用索引列。遇到范围查询（`>`、`<`、`BETWEEN`、`LIKE`）时，索引在此列处中断，后面的列不再使用索引。

### 场景一：全部使用（最理想）

```sql
SELECT * FROM employees WHERE name = 'Alice' AND age = 25;
```

走索引，索引顺序 `(name, age, dept_id)`，从 `name` 开始匹配，命中。

### 场景二：只用最左列

```sql
SELECT * FROM employees WHERE name = 'Alice';
```

走索引，从 `name` 开始匹配。

### 场景三：跳过中间列（危险！）

```sql
SELECT * FROM employees WHERE age = 25;
```

**不走索引**。B+ 树先按 `name` 排序，你直接跳到 `age`，树不知道往哪走，只能全表扫描。

### 场景四：范围查询中断后续

```sql
SELECT * FROM employees
WHERE name = 'Alice' AND age > 25;
```

走索引。`age > 25` 是范围查询，`dept_id` 列在 B+ 树中的顺序就被打乱了，无法使用 `dept_id` 的索引。

```sql
SELECT * FROM employees
WHERE name > 'Alice' AND age = 25;
```

只走 `name` 的索引（范围查询），`age` 列不参与索引匹配。

## 实战：如何判断 SQL 是否走联合索引

MySQL 提供了一个强大的工具：`EXPLAIN`。

```sql
EXPLAIN SELECT * FROM employees
WHERE name = 'Alice' AND age = 30 AND dept_id = 102;
```

```text
+----+-------------+-----------+------------+------+-------------------+
| id | select_type | table     | key        | rows | Extra             |
+----+-------------+-----------+------------+------+-------------------+
|  1 | SIMPLE      | employees | idx_n_a_d  |    1 | Using index       |
+----+-------------+-----------+------------+------+-------------------+
```

`key` 列显示使用了 `idx_n_a_d` 索引，`Using index` 表示覆盖索引（不需要回表）。

再看一个不走索引的例子：

```sql
EXPLAIN SELECT * FROM employees WHERE age = 25;
```

```text
+----+-------------+-----------+------+------+-------------+
| id | select_type | table     | key  | rows | Extra       |
+----+-------------+-----------+------------+------+-------------+
|  1 | SIMPLE      | employees | NULL | 1000 | Using where |
+----+-------------+-----------+------------+------+-------------+
```

`key` 是 `NULL`，说明没走索引，`Using where` 表示在 MySQL 服务层过滤数据。

## 联合索引的排列顺序：字段顺序是门艺术

字段顺序选错了，索引等于白建。

### 原则一：区分度高的列放前面

```sql
-- 好的顺序：name 区分度高
CREATE INDEX idx_n_a ON employees(name, age);

-- 坏的顺序：sex 区分度低（只有男女两种值）
CREATE INDEX idx_s_n ON employees(sex, name);
```

区分度高的列放在前面，能快速缩小查询范围。

### 原则二：等值查询的列优先于范围查询

```sql
-- 好的顺序：先等值，后范围
CREATE INDEX idx_s_a ON employees(status, age);
SELECT * FROM employees WHERE status = 1 AND age > 30;

-- 坏的顺序：先范围，后等值
CREATE INDEX idx_a_s ON employees(age, status);
SELECT * FROM employees WHERE status = 1 AND age > 30;
```

范围查询会导致后续列无法使用索引，把等值列放前面能保证更多列参与匹配。

### 原则三：查询要什么，就建什么

如果你的查询是：

```sql
SELECT * FROM employees WHERE name = '?' AND dept_id = '?';
```

很多人会习惯性地建 `(name, age, dept_id)`，但实际上只查 `name` 和 `dept_id`，中间多了一个 `age` 是浪费。直接建 `(name, dept_id)` 更高效。

## 覆盖索引：让查询飞起来

如果查询的所有列都在索引里，MySQL 就不需要回表，直接在索引树里拿数据。这就是「覆盖索引」。

```sql
-- 创建覆盖索引
CREATE INDEX idx_name_dept ON employees(name, dept_id);

-- 查询
SELECT name, dept_id FROM employees WHERE name = 'Alice';
```

这条 SQL 的查询列 `name` 和 `dept_id` 都在索引里，MySQL 直接从 B+ 树中取值，无需回表查主键索引，性能极高。

判断是否走覆盖索引，看 `EXPLAIN` 的 `Extra` 列是否包含 `Using index`。

## 索引下推优化：ICP

联合索引还有一个重要的优化点：**索引条件下推（Index Condition Pushdown，ICP）**。

### 什么是 ICP

在没有 ICP 的情况下，MySQL 的执行流程是：

1. 存储引擎根据索引找到满足 `name LIKE 'A%'` 的主键 ID
2. 回表取出完整行数据
3. 服务层过滤 `age = 30` 的记录

如果有 ICP，优化后的流程是：

1. 存储引擎在遍历索引的过程中，同时检查 `name LIKE 'A%'` 和 `age = 30`
2. 只返回满足两个条件的主键 ID
3. 回表取出数据

ICP 的效果是**减少回表次数**。

### ICP 示例

```sql
CREATE INDEX idx_name_age ON employees(name, age);

-- 这条 SQL 会使用 ICP
SELECT * FROM employees WHERE name LIKE 'A%' AND age = 30;
```

通过 `EXPLAIN` 可以看到是否使用了 ICP：

```text
-- 使用 ICP
Extra: Using index condition

-- 未使用 ICP
Extra: Using where
```

### ICP 的限制

ICP 不是万能的，以下情况不适用：

| 限制 | 说明 |
|---|---|
| 只能用于索引辅助查询 | 索引必须覆盖查询涉及的列 |
| 不支持 `FULLTEXT` 索引 | 全文索引不适用 |
| 不支持 `geometry` 类型的空间索引 | 空间索引暂不支持 |
| 多表 `JOIN` 时条件不下推 | 只在单表查询时生效 |

## 总结：最左前缀的三条铁律

1. **从左到右**：索引从最左列开始匹配，中间不能断开
2. **范围中断**：遇到 `>`、`<`、`BETWEEN`、`LIKE` 等范围查询，后续列失效
3. **等值优先**：等值查询不影响后续列使用索引

理解了这三条，再看那些「建了索引却不走」的问题，就能快速定位原因了。
