# 索引失效的 11 种情况

「这个查询怎么这么慢？」

「建了索引啊，为什么没走？」

这两个问题，几乎是 DBA 和后端工程师每天都要面对的灵魂拷问。索引不是万能药，建了不代表会用，用了不代表生效。

今天我们把索引失效的 11 种情况全部过一遍，帮你从根上理解：索引什么时候会「装死」。

## 1. 对索引列使用函数或运算

最常见的一种失效场景：对索引列动手脚。

```sql
-- 失效：索引列参与了运算
SELECT * FROM orders WHERE YEAR(create_time) = 2024;

-- 生效：把运算移到右边
SELECT * FROM orders WHERE create_time >= '2024-01-01' AND create_time < '2025-01-01';
```

```sql
-- 失效：对索引列使用函数
SELECT * FROM users WHERE SUBSTRING(phone, 1, 3) = '138';

-- 生效：改写为范围查询
SELECT * FROM users WHERE phone LIKE '138%';
```

MySQL 的索引树存储的是列值的原始比较结果，你对列值做了变换，索引树就找不到对应节点了。

## 2. 字符串不加引号

字符串列的索引，按字符串比较规则排序。你传进去的数字，MySQL 会隐式转换，但转换后的结果无法命中索引。

```sql
-- 失效：phone 是 VARCHAR，传入数字
SELECT * FROM users WHERE phone = 13800138000;

-- 生效：字符串加引号
SELECT * FROM users WHERE phone = '13800138000';
```

隐式类型转换的背后，MySQL 把字符串转成了数字，这一转，索引就废了。

## 3. LIKE 模糊查询以通配符开头

```sql
-- 失效：% 在左边，索引无法定位
SELECT * FROM products WHERE name LIKE '%手机';

-- 生效：% 在右边
SELECT * FROM products WHERE name LIKE '小米%';

-- 失效：两边都有 %
SELECT * FROM products WHERE name LIKE '%小米手机%';
```

`LIKE '%xxx'` 无法利用 B+ 树的有序性，只能全表扫描。解决方案：用 ES 做全文检索，或者在数据库里建倒排索引。

## 4. OR 条件中有非索引列

```sql
-- 失效：OR 左边走了索引，右边没走
SELECT * FROM users WHERE name = 'Alice' OR email = 'alice@example.com';

-- 生效：两边都建索引
ALTER TABLE users ADD INDEX idx_name(name);
ALTER TABLE users ADD INDEX idx_email(email);

-- 或者：拆成两条 UNION
(SELECT * FROM users WHERE name = 'Alice')
UNION
(SELECT * FROM users WHERE email = 'alice@example.com');
```

OR 条件下，如果有一列没建索引，整个查询都要全表扫描。

## 5. 联合索引不满足最左前缀

```sql
CREATE INDEX idx_a_b_c ON table1(a, b, c);

-- 生效：使用了 a
SELECT * FROM table1 WHERE a = 1;

-- 生效：使用了 a、b
SELECT * FROM table1 WHERE a = 1 AND b = 2;

-- 失效：跳过了 a，直接查 b
SELECT * FROM table1 WHERE b = 2;

-- 失效：使用了 a、c，但 b 是范围查询，c 无法使用
SELECT * FROM table1 WHERE a = 1 AND b > 2 AND c = 3;
```

联合索引必须从最左列开始，连续使用，遇到范围查询（`>`、`<`、`BETWEEN`、`LIKE`）则后续列失效。

## 6. WHERE 子句中索引列与常量比较类型不一致

```sql
-- 表结构：age 是 INT
-- 失效：字符串 '25' 需要隐式转换
SELECT * FROM users WHERE age = '25';

-- 生效：使用正确类型
SELECT * FROM users WHERE age = 25;
```

类型不一致会触发隐式转换，导致索引失效。

## 7. 使用 `NOT IN` 和 `NOT EXISTS`

```sql
-- 失效：NOT IN
SELECT * FROM orders WHERE status NOT IN (1, 2);

-- 失效：NOT EXISTS
SELECT * FROM users u WHERE NOT EXISTS (
    SELECT 1 FROM orders o WHERE o.user_id = u.id
);

-- 替代方案：LEFT JOIN + IS NULL
SELECT u.* FROM users u LEFT JOIN orders o ON o.user_id = u.id WHERE o.id IS NULL;
```

`NOT IN` 和 `NOT EXISTS` 无法利用 B+ 树的有序性，通常需要全表扫描。

## 8. 查询条件的数据分布不均

有时候索引存在，但 MySQL 优化器认为「不用索引更快」。

```sql
-- status 只有 0 和 1，且 99% 都是 1
-- MySQL 认为全表扫描更快（直接读数据文件）
SELECT * FROM orders WHERE status = 0;

-- 可以强制指定索引
SELECT * FROM orders USE INDEX (idx_status) WHERE status = 0;
```

这是 MySQL 优化器的成本估算问题，不一定是索引没用，而是优化器「觉得」不用更快。

## 9. 索引列使用 `IS NULL` 或 `IS NOT NULL`

```sql
-- 可能失效：NULL 值在索引树中的位置特殊
SELECT * FROM users WHERE name IS NULL;

-- 替代方案：给 NULL 值设置默认值
ALTER TABLE users MODIFY name VARCHAR(50) NOT NULL DEFAULT '';
```

NULL 值在 B+ 树中的存储方式与非 NULL 值不同，部分场景下会导致索引失效。

## 10. 使用 `IN` 包含太多值

```sql
-- 可能失效：IN 里值太多，优化器放弃索引
SELECT * FROM products WHERE category_id IN (1, 2, 3, ..., 10000);
```

当 `IN` 列表的值超过某个阈值（通常是数据量的 20%-30%），MySQL 认为全表扫描成本更低。

## 11. 表太小或统计信息不准确

```sql
-- 100 行的小表，全表扫描反而更快
SELECT * FROM small_table WHERE indexed_column = 'value';
```

MySQL 优化器依赖统计信息（`ANALYZE TABLE`）来估算成本。统计信息不准时，优化器可能做出错误判断。

## 如何排查索引失效：`EXPLAIN` 是你的好朋友

```sql
EXPLAIN SELECT * FROM users WHERE name = 'Alice';
```

关键字段：

| 字段 | 含义 |
|---|---|
| `key` | 实际使用的索引 |
| `type` | 全表扫描（ALL）还是索引扫描 |
| `rows` | 扫描的行数估算 |
| `Extra` | `Using filesort`、`Using temporary` 是不好的信号 |

## 总结：避免索引失效的核心原则

1. **不改变索引列的值**：不加函数、不做运算
2. **类型要匹配**：字符串加引号，数字不加引号
3. **从最左开始，连续使用**：联合索引不能跳列
4. **避免 OR 混用**：要么全有索引，要么拆成 UNION
5. **模糊查询 % 放右边**：或者换全文检索方案

索引是加速查询的工具，但工具得用对地方。理解底层原理，才能在纷繁复杂的 SQL 中快速定位问题。
