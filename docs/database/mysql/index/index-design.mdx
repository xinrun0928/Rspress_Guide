# 索引创建原则与使用注意事项

为什么同样的查询，有人 50 毫秒返回，有人 5 秒还转圈？

区别往往不在 SQL 本身，而在索引设计。一个设计糟糕的索引，不仅不能加速查询，还会拖慢写入、浪费空间。

索引是双刃剑——用对了是神器，用错了是负担。

## 一、什么时候该建索引

不是所有列都需要建索引。索引的优势是快速定位，劣势是占用空间、拖慢写入。

### 建索引的条件

| 条件 | 说明 |
|---|---|
| `WHERE` 频繁出现 | 查询条件里经常作为过滤条件的列 |
| `JOIN` 频繁关联 | 外键列或关联列 |
| `ORDER BY` / `GROUP BY` | 排序和分组的列 |
| `DISTINCT` 唯一性高 | 区分度高的列（性别这种低区分度的不适合） |
| 数据量大 | 小表全表扫描更快，建索引反而浪费 |

### 不建索引的情况

- 区分度低的列（如性别 `男/女`）
- 更新频繁的列（每次写入都要维护索引）
- 表数据量很小

## 二、字段选择：谁该做索引

### 原则一：区分度高的列优先

```sql
-- 好的选择：user_id 区分度接近 100%
CREATE INDEX idx_uid ON orders(user_id);

-- 不好的选择：status 只有 3 种值，区分度低
CREATE INDEX idx_status ON orders(status);
```

区分度计算公式：`COUNT(DISTINCT column) / COUNT(*)`，越接近 1 越好。

### 原则二：外键列必须建索引

```sql
-- orders.user_id 是外键，不建索引会导致级联查询慢
CREATE INDEX idx_uid ON orders(user_id);

-- 用户表对应的主键列本身有索引，无需额外处理
SELECT o.*, u.name FROM orders o
JOIN users u ON u.id = o.user_id
WHERE o.status = 1;
```

外键不加索引，父表更新时子表要加读锁，影响并发。

### 原则三：复合索引遵循最左前缀

```sql
-- 查询条件是 (status, user_id, create_time)
-- 建复合索引：(status, user_id, create_time)
CREATE INDEX idx_s_u_c ON orders(status, user_id, create_time);
```

而不是：

```sql
-- 错误顺序
CREATE INDEX idx_c_s_u ON orders(create_time, status, user_id);
```

## 三、复合索引的字段顺序设计

复合索引的顺序是索引设计的核心。

### 优先法则

| 法则 | 说明 |
|---|---|
| 等值查询列在前 | `=`、`IN` 先匹配 |
| 范围查询列在后 | `>`、`<`、`BETWEEN`、`LIKE` 尽量靠后 |
| 区分度高的在前 | 快速缩小扫描范围 |
| 占用空间小的在前 | 减少索引树层级 |

### 设计步骤

给定一个查询：

```sql
SELECT * FROM orders
WHERE user_id = 1001
  AND status IN (1, 2)
  AND create_time > '2024-01-01'
ORDER BY create_time;
```

按优先法则排序：

1. `user_id`：等值查询，最优先
2. `status`：`IN` 也是等值，排第二
3. `create_time`：范围查询，放最后

所以索引顺序是 `(user_id, status, create_time)`：

```sql
CREATE INDEX idx_u_s_c ON orders(user_id, status, create_time);
```

## 四、覆盖索引：用索引「包办」查询

如果查询的所有列都在索引里，MySQL 不需要回表，直接在索引树拿数据。

```sql
-- 需要回表：查了 name, age，但索引只有 name
CREATE INDEX idx_name ON users(name);
SELECT name, age FROM users WHERE name = 'Alice';  -- 回表

-- 不需要回表：索引包含所有查询列
CREATE INDEX idx_name_age ON users(name, age);
SELECT name, age FROM users WHERE name = 'Alice';  -- 覆盖索引
```

```sql
EXPLAIN SELECT name, age FROM users WHERE name = 'Alice';
-- Extra: Using index  ← 覆盖索引，不需要回表
```

## 五、索引使用注意事项

### 注意事项一：避免索引失效

```sql
-- 失效：对索引列使用函数
SELECT * FROM orders WHERE YEAR(create_time) = 2024;

-- 生效：把运算移到右边
SELECT * FROM orders WHERE create_time >= '2024-01-01' AND create_time < '2025-01-01';
```

### 注意事项二：字符串字段加引号

```sql
-- phone 是 VARCHAR，不加引号导致隐式转换，索引失效
SELECT * FROM users WHERE phone = 13800138000;  -- 失效

SELECT * FROM users WHERE phone = '13800138000';  -- 生效
```

### 注意事项三：OR 条件要小心

```sql
-- OR 右边没索引，全表扫描
SELECT * FROM users WHERE name = 'Alice' OR email = 'alice@example.com';

-- 拆分或建索引
ALTER TABLE users ADD INDEX idx_email(email);
```

### 注意事项四：LIKE 百分号位置

```sql
-- 失效：% 在左边
SELECT * FROM products WHERE name LIKE '%手机';

-- 生效：% 在右边
SELECT * FROM products WHERE name LIKE '小米%';
```

## 六、索引数量控制

索引不是越多越好。

| 问题 | 影响 |
|---|---|
| 占用存储空间 | 每个索引都是一颗 B+ 树 |
| 拖慢写入速度 | `INSERT/UPDATE/DELETE` 要维护所有索引 |
| 增加优化器负担 | 索引太多，优化器选错索引 |

经验法则：**单表索引数量不超过 5 个**。每个索引都应该有明确的用途。

## 七、慢查询优化实操

### 步骤一：找到慢查询

```sql
-- 开启慢查询日志
SET GLOBAL slow_query_log = 'ON';
SET GLOBAL long_query_time = 1;  -- 超过 1 秒记录

-- 查看慢查询
SHOW FULL PROCESSLIST;
SELECT * FROM mysql.slow_log;
```

### 步骤二：分析执行计划

```sql
EXPLAIN SELECT * FROM orders o
JOIN users u ON u.id = o.user_id
WHERE o.status = 1 AND o.create_time > '2024-01-01';
```

关注字段：`type`（all 全表扫描）、`key`（用了哪个索引）、`rows`（扫描行数）。

### 步骤三：创建或调整索引

根据分析结果调整索引：

```sql
-- 如果 type 是 ALL，考虑建索引
CREATE INDEX idx_status_time ON orders(status, create_time);

-- 如果用错了索引，强制指定
SELECT * FROM orders USE INDEX (idx_status_time) WHERE status = 1;
```

## 八、总结：索引设计的检查清单

```
□ WHERE 条件中的列建索引了吗？
□ 复合索引顺序符合最左前缀原则了吗？
□ 查询列是否都在索引里（覆盖索引）？
□ OR 条件的两边都有索引了吗？
□ 索引数量是否超过 5 个？
□ 建完索引后用 EXPLAIN 验证了吗？
□ 写入频繁的表，索引是否过多？
```

索引是数据库性能的核心杠杆。用对了，查询从秒级降到毫秒级；用错了，不仅没加速，还会拖慢全表。希望这篇指南能帮你在下一次性能调优中，快速找到问题所在。
