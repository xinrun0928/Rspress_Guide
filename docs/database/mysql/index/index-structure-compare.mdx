# 索引结构对比：B+ 树 vs Hash vs B 树

面试中，索引结构是高频考点。「MySQL 索引用什么数据结构？」「Hash 索引和 B+ 树索引有什么区别？」「B 树和 B+ 树的区别是什么？」

这些问题的背后，是几种核心数据结构的原理和权衡。本文深入对比 B+ 树、B 树、Hash 三种索引结构，让你彻底掌握它们的差异。

## 数据结构基础

### 数组：O(1) 查找

数组可以通过下标直接访问，查找复杂度 O(1)。但插入和删除需要移动元素，复杂度 O(n)。

### 链表：O(n) 查找

链表不支持随机访问，只能从头遍历，查找复杂度 O(n)。但插入和删除只需修改指针，复杂度 O(1)。

### 二叉查找树（BST）：O(log n) 查找

二叉查找树结合了数组查找的效率和链表插入的灵活性。左子节点小于父节点，右子节点大于父节点。

```
       8
      / \
     3   10
    / \    \
   1   6    14
```

但二叉查找树可能退化成链表（数据有序时），导致查找复杂度退化为 O(n)。

### 平衡二叉树（AVL）：O(log n) 查找

AVL 树通过旋转保持平衡，查找复杂度稳定 O(log n)。但每次插入/删除可能需要多次旋转，维护成本高。

### 红黑树：O(log n) 查找

红黑树通过颜色约束保持大致平衡，比 AVL 树旋转次数少，适合频繁插入/删除的场景。但树高仍然较大，IO 次数多。

## B 树（B-Tree）

B 树是多路平衡查找树，每个节点可以有多个子节点。设计目标是**减少树高，减少磁盘 IO**。

### B 树的特点

- 每个节点最多有 m 个子节点（m 阶 B 树）
- 每个节点最多存储 m-1 个关键字
- 根节点至少有两个子节点
- 非根节点至少有 m/2 个子节点
- 所有叶子节点在同一层

```
3 阶 B 树示例：

           [5 | 15]
          /    |    \
    [1 | 3]  [6 | 8 | 10]  [16 | 20]
```

### B 树存储数据

**B 树的所有节点都存储数据**。这既是特点也是缺点：

- 优点：某些查询可以直接在非叶子节点完成
- 缺点：非叶子节点存储数据，占用空间，能容纳的索引项减少，树高增加

## B+ 树（B+ Tree）

B+ 树是 B 树的变体，在 MySQL InnoDB 中广泛使用。

### B+ 树与 B 树的关键区别

| 区别 | B 树 | B+ 树 |
|---|---|---|
| 数据存储 | 所有节点都存储数据 | 只有叶子节点存储数据 |
| 叶子节点 | 叶子节点无链表连接 | 叶子节点通过链表连接 |
| 查询稳定性 | 查询复杂度不固定 | 查询复杂度固定 O(log n) |
| 范围查询 | 需要中序遍历 | 链表遍历即可 |

### B+ 树结构示意

```
B+ 树（3 阶）：

              [15]              [30]
             /    \            /    \
       [5 | 10]  [18 | 25]  [35 | 40]
           │        │          │
           ▼        ▼          ▼
         所有数据  所有数据    所有数据
           │        │          │
           └────────┼──────────┘
                   ▼
          叶子节点链表连接
       [数据1] ←→ [数据2] ←→ [数据3] ←→ ...
```

## Hash 索引

Hash 索引基于哈希表实现，通过哈希函数计算 bucket，直接定位数据。

### Hash 索引的结构

```
Hash 索引：

索引列值    哈希函数     Bucket    指向数据行
  ↓          ↓           ↓           ↓
"张三"  →  hash("张三") →   3    →  Row Pointer → 数据行
"李四"  →  hash("李四") →   7    →  Row Pointer → 数据行
```

### Hash 索引的查找过程

1. 计算查找值的哈希值
2. 通过哈希值定位到 Bucket
3. 从 Bucket 中找到对应的记录

查找复杂度理论上 O(1)，但哈希冲突时可能退化为链表查找。

### 哈希冲突处理

Hash 索引使用链地址法处理冲突：

```
Bucket[3]:
  │
  ├──→ 记录1（张三，哈希值=3）
  ├──→ 记录2（王五，哈希值=3，冲突）
  └──→ 记录3（赵六，哈希值=3，冲突）

查找"王五"：
1. 计算 hash("王五") = 3
2. 定位到 Bucket[3]
3. 遍历链表，逐个比较，找到"王五"
```

## 三种索引结构对比

### 性能对比

| 特性 | B+ 树索引 | B 树索引 | Hash 索引 |
|---|---|---|---|
| 时间复杂度 | O(log n) | O(log n) | O(1) ~ O(n) |
| 等值查询 | 快 | 快 | 最快（最优） |
| 范围查询 | 高效 | 需中序遍历 | 不支持 |
| 排序查询 | 支持 | 支持 | 不支持 |
| 最左前缀 | 支持 | 支持 | 不支持 |
| 空间占用 | 低（只存索引） | 高（存数据和索引） | 高（需存储哈希表） |
| 树高 | 低 | 较高 | 无（哈希表） |

### 功能对比

| 操作 | B+ 树索引 | Hash 索引 |
|---|---|---|
| `=` 等值查询 | ✅ 高效 | ✅ 最优 |
| `IN` 范围查询 | ✅ 高效 | ✅ 高效 |
| `>` `<` `BETWEEN` 范围查询 | ✅ 高效 | ❌ 不支持 |
| `LIKE` 前缀匹配 | ✅ 高效 | ❌ 不支持 |
| `ORDER BY` 排序 | ✅ 高效 | ❌ 不支持 |
| `GROUP BY` 分组 | ✅ 高效 | ❌ 不支持 |
| 前缀匹配 | ✅ 支持 | ❌ 不支持 |

### 适用场景对比

| 索引类型 | 适用场景 |
|---|---|
| B+ 树索引 | 大多数场景：等值查询、范围查询、排序、分组 |
| B 树索引 | 范围查询、全文索引（MyISAM） |
| Hash 索引 | 等值查询为主的 KV 场景、会话缓存 |

## 为什么 MySQL 选择 B+ 树？

MySQL 选择 B+ 树作为主要索引结构，是基于多个因素的权衡：

### 1. 磁盘 IO 考量

B+ 树的非叶子节点不存储数据，只存储索引，每个节点能容纳更多索引项，树高更低，磁盘 IO 更少。

```
对比计算（假设页大小 16KB，主键 8 字节）：

B 树（存储数据）：
  每页索引数：16KB / (8 + 6 + 数据指针) ≈ 1000
  树高 3：1000³ = 10 亿条

B+ 树（只存索引）：
  每页索引数：16KB / 20 ≈ 8000
  树高 3：8000³ = 5 万亿条
```

B+ 树用更少的 IO 次数支持更多的数据。

### 2. 范围查询优化

B+ 树的叶子节点通过链表连接，范围查询只需遍历链表：

```sql
-- B+ 树范围查询
SELECT * FROM users WHERE age BETWEEN 20 AND 30;
```

```
B+ 树范围查询过程：
1. 定位 age=20 的起始位置
2. 沿链表顺序遍历
3. 直到遇到第一条 age > 30 的记录

时间复杂度：O(log n + k)，k 是结果集大小
```

B 树做范围查询需要中序遍历所有节点，效率低。

### 3. 查询稳定性

B+ 树的查询都要到叶子节点，复杂度固定 O(log n)。B 树的查询可能在任意节点返回，复杂度不稳定。

## Hash 索引的适用场景

### 适合使用 Hash 索引的场景

```sql
-- 1. KV 查找
SELECT * FROM session WHERE session_id = 'xxx';

-- 2. 精确匹配为主的查询
SELECT * FROM whitelist WHERE ip = '192.168.1.1';

-- 3. 内存数据库
CREATE TABLE temp_data ENGINE = Memory;
SELECT * FROM temp_data WHERE key = 'xxx';
```

### InnoDB 自适应哈希索引（AHI）

InnoDB 会自动监控索引访问模式，当发现某些索引被频繁访问时，会自动构建 Hash 索引。

```
自适应哈希索引工作原理：

监控索引访问
    │
    ├── 发现索引被频繁访问
    │     │
    │     └── 为该索引构建 Hash 索引
    │
    └── 查询时优先使用 Hash 索引
```

AHI 的优势：

- 自动创建，无需人工干预
- 基于 Buffer Pool 中的 B+ 树索引构建，速度快
- 对等值查询特别有效

## 面试实战

### 问题一：B 树和 B+ 树的区别

**面试官问**：「B 树和 B+ 树有什么区别？」

**深入回答**：

> 核心区别有三点：
>
> 1. **数据存储位置**：B 树的所有节点都存储数据，B+ 树只有叶子节点存储数据。这导致 B+ 树的非叶子节点更小，能容纳更多索引，树高更低。
>
> 2. **叶子节点连接**：B+ 树的叶子节点通过链表连接，便于范围查询；B 树没有链表连接。
>
> 3. **查询复杂度**：B+ 树的查询都要到叶子节点，复杂度固定；B 树可能在任意节点返回。
>
> MySQL 选择 B+ 树是因为：非叶子节点不存数据，树高更低，磁盘 IO 更少；叶子链表支持高效范围查询。

### 问题二：Hash 索引的局限性

**面试官问**：「Hash 索引有什么局限性？什么时候用？」

**深入回答**：

> Hash 索引的局限：
>
> 1. **不支持范围查询**：无法做 `>`、`<`、`BETWEEN` 查询
>
> 2. **不支持排序**：`ORDER BY` 无法利用 Hash 索引
>
> 3. **不支持最左前缀**：联合索引无法利用前面的列
>
> 4. **哈希冲突**：大量冲突时退化为链表
>
> 适用场景：
>
> - 等值查询为主的场景
> - KV 存储（Redis、Memcached）
> - InnoDB 自适应哈希索引

### 问题三：为什么不用红黑树？

**面试官问**：「红黑树也能实现 O(log n) 查找，为什么 MySQL 不用红黑树？」

**深入回答**：

> 红黑树的局限在于：
>
> 1. **二叉树树高**：红黑树是二叉树，在数据量大时树高很大。假设每个节点一次 IO，树高为 log₂(n)，100 万数据需要约 20 层 IO。
>
> 2. **B+ 树多路查找**：B+ 树是多叉树，每个节点可以有很多子节点，树高大大降低。假设每个节点 800 个子节点，100 万数据只需 3 层 IO。
>
> 3. **磁盘 IO 优化**：数据库的数据存储在磁盘，IO 是主要成本。B+ 树的设计目标就是减少 IO 次数。
>
> 红黑树适合内存中的数据结构（如 Java 的 TreeMap），不适合磁盘存储。

## 总结

三种索引结构各有特点：

| 结构 | 特点 | 适用场景 |
|---|---|---|
| B+ 树 | 多路平衡树，叶子链表，高效范围查询 | 大多数 OLTP 场景 |
| B 树 | 多路平衡树，所有节点存数据 | 全文索引 |
| Hash | 哈希表，O(1) 查找 | 等值查询、缓存 |

MySQL 选择 B+ 树作为主要索引结构，是因为它在磁盘 IO、范围查询、查询稳定性之间取得了最佳平衡。
