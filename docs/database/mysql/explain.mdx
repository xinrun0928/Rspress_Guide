# Explain 执行计划

> 面试官问：「这条 SQL 为什么这么慢？」你打开 `EXPLAIN` 结果，盯着 `type`、`key`、`rows` 几个字段，却不知道从哪里开始分析。EXPLAIN 是 MySQL 优化的起点，但很多人只会看 `type` 是不是 `ALL`，不会深入分析 `Extra` 和 `key_len`，这道题能暴露候选人的真实优化能力。

## 面试官最关心的 3 个问题（快速自测）

| 问题 | 考察点 | 难度 |
|------|--------|------|
| EXPLAIN 各字段含义是什么？ | 基础技能 | 🔴 高频 |
| type 列有哪些值？从好到差怎么排序？ | 索引原理 | 🔴 高频 |
| Extra 字段的常用值代表什么含义？ | 实战分析 | 🟡 中频 |

## 一、EXPLAIN 使用方法

### 1.1 基本语法

```sql
EXPLAIN SELECT * FROM users WHERE name = '张三';

EXPLAIN FORMAT=JSON SELECT * FROM users WHERE name = '张三';

EXPLAIN ANALYZE SELECT * FROM users WHERE name = '张三';  -- MySQL 8.0+
```

### 1.2 EXPLAIN ANALYZE（MySQL 8.0+）

```sql
EXPLAIN ANALYZE SELECT * FROM orders WHERE user_id = 100;

-- 输出示例
-> Nested loop inner join  (cost=3.43 rows=5) (actual time=0.031..0.058 rows=5 loops=1)
    -> Index lookup on orders using idx_user_id (user_id=100)  (cost=2.43 rows=5) (actual time=0.014..0.034 rows=5 loops=1)
    -> Covering index lookup on orders using PRIMARY (id=orders.id)  (cost=0.26 rows=1) (actual time=0.004..0.004 rows=1 loops=5)
```

> `EXPLAIN ANALYZE` 会实际执行查询并返回成本信息，对于分析慢查询非常有用。

## 二、EXPLAIN 输出字段详解

### 2.1 完整输出示例

```sql
EXPLAIN SELECT o.* FROM orders o
INNER JOIN users u ON o.user_id = u.id
WHERE o.status = 2 AND o.create_time > '2024-01-01';
```

```
+----+-------------+-------+------------+------+---------------------+---------------------+---------+-------------------------+------+----------+-------------+
| id | select_type | table | type       | possible_keys | key       | key_len | ref      | rows    | filtered | Extra |
+----+-------------+-------+------------+------+---------------------+---------------------+---------+-------------------------+------+----------+-------------+
| 1  | SIMPLE      | o     | ref        | idx_status_time | idx_status_time | 5       | const   | 1000    | 100.00  | Using index condition; Using where; Backfile |
| 2  | SIMPLE      | u     | eq_ref     | PRIMARY         | PRIMARY   | 8       | o.user_id | 1      | 100.00  | Using index |
+----+-------------+-------+------------+------+---------------------+---------------------+---------+-------------------------+------+----------+-------------+
```

## 三、核心字段详解

### 3.1 id - 查询序号

| id 值 | 含义 |
|-------|------|
| id 相同 | 执行顺序从上到下 |
| id 不同 | 子查询 id 越大越先执行 |
| id 为 NULL | 包含 UNION RESULT |

```sql
EXPLAIN SELECT * FROM (SELECT id FROM users WHERE name = '张三') t;
```

### 3.2 select_type - 查询类型

| 值 | 含义 |
|----|------|
| `SIMPLE` | 简单查询（无子查询/UNION） |
| `PRIMARY` | 最外层 SELECT |
| `SUBQUERY` | 子查询（在 SELECT 中） |
| `DERIVED` | 派生表（在 FROM 中） |
| `UNION` | UNION 查询 |
| `UNION RESULT` | UNION 结果集 |

```sql
-- DERIVED 示例
EXPLAIN SELECT * FROM (SELECT id FROM users WHERE name = '张三') t;
```

### 3.3 type - 连接类型（重点）

| type 值 | 含义 | 性能 |
|---------|------|------|
| `system` | 仅有一行（系统表） | ✅ 最佳 |
| `const` | 最多一行匹配（主键/唯一索引） | ✅ 最佳 |
| `eq_ref` | 唯一索引等值关联 | ✅ 好 |
| `ref` | 非唯一索引等值关联 | ✅ 好 |
| `ref_or_null` | ref + NULL 查询 | ✅ 较好 |
| `index_merge` | 索引合并 | 较好 |
| `range` | 索引范围查询 | 一般 |
| `index` | 全索引扫描 | ❌ 差 |
| `ALL` | 全表扫描 | ❌ 最差 |

**性能排序**：

```
system > const > eq_ref > ref > ref_or_null > index_merge > range > index > ALL
```

### 3.4 possible_keys vs key

| 字段 | 含义 |
|------|------|
| `possible_keys` | 可用的索引列表（优化器认为可能用到的） |
| `key` | 实际使用的索引 |
| `key_len` | 索引使用长度（可判断用到几列） |

```sql
-- 索引: (name, age, email)
-- key_len = 53 说明只用了一列 VARCHAR(50)
-- key_len = 57 说明用了 name + age
```

### 3.5 rows - 预计扫描行数

**注意**：这是优化器的估算值，不是实际值。

| 场景 | rows 含义 |
|------|-----------|
| `rows = 1` | 精准索引查询 |
| `rows = 1000` | 可能需要优化 |
| `rows > 100000` | 几乎全表扫描 |

### 3.6 filtered - 过滤比例

表示索引扫描后，被 WHERE 条件过滤掉的比例。

```
filtered = 100.00  -- 全部符合
filtered = 50.00   -- 过滤掉一半
```

### 3.7 Extra - 附加信息（重点）

| 值 | 含义 | 性能建议 |
|----|------|----------|
| `Using index` | **覆盖索引**，无需回表 | ✅ 最优 |
| `Using index condition` | 使用索引下推（ICP） | ✅ 好 |
| `Using where` | 服务器层过滤数据 | 一般 |
| `Using filesort` | 需要额外排序 | ❌ 需优化 |
| `Using temporary` | 需要临时表 | ❌ 需优化 |
| `Using join buffer` | 使用联接缓冲 | ❌ 需优化 |
| `Impossible WHERE` | WHERE 条件永远为 false | ❌ 逻辑错误 |
| `Select tables optimized away` | 直接从聚合计算 | ✅ 好 |

## 四、实战分析案例

### 4.1 案例 1：全表扫描

```sql
EXPLAIN SELECT * FROM users WHERE age = 25;
```

```
+----+-------------+-------+------+----------+------+--------+------+----------+-------------+
| id | select_type | table | type | possible_keys | key  | key_len | ref  | rows   | Extra       |
+----+-------------+-------+------+----------+------+--------+------+----------+-------------+
| 1  | SIMPLE      | users | ALL  | NULL     | NULL | NULL    | NULL | 1000000 | Using where |
+----+-------------+-------+------+----------+------+--------+------+----------+-------------+
```

**问题**：`type = ALL` 表示全表扫描，`rows = 1000000` 扫描百万行。

**优化**：

```sql
-- 添加索引
ALTER TABLE users ADD INDEX idx_age (age);

-- 再次验证
EXPLAIN SELECT * FROM users WHERE age = 25;
-- type = ref, rows = 100
```

### 4.2 案例 2：Using filesort

```sql
EXPLAIN SELECT * FROM orders WHERE user_id = 100 ORDER BY create_time DESC;
```

```
+----+-------------+-------+------+----------+------+--------+------+----------+-----------------------------+
| id | select_type | table | type | key      | key_len | ref  | rows    | Extra                      |
+----+-------------+-------+------+----------+---------+------+----------+-----------------------------+
| 1  | SIMPLE      | orders | ref  | idx_user | 8       | const | 500     | Using where; Using filesort |
+----+-------------+-------+------+----------+---------+------+----------+-----------------------------+
```

**问题**：`Using filesort` 表示需要额外排序操作。

**优化**：

```sql
-- 建立覆盖索引，同时满足查询和排序
ALTER TABLE orders ADD INDEX idx_user_time (user_id, create_time DESC);

EXPLAIN SELECT * FROM orders WHERE user_id = 100 ORDER BY create_time DESC;
-- Extra = Using index condition（使用索引，无需 filesort）
```

### 4.3 案例 3：Using join buffer

```sql
EXPLAIN SELECT * FROM orders o
INNER JOIN users u ON o.user_id = u.id;
```

```
+----+-------------+-------+------+----------+------+--------+------+----------+-------------------------+
| id | select_type | table | type | key      | key_len | ref  | rows    | Extra                   |
+----+-------------+-------+------+----------+---------+------+----------+-------------------------+
| 1  | SIMPLE      | o     | ALL  | NULL     | NULL    | NULL | 100000  | Using where             |
| 1  | SIMPLE      | u     | ALL  | PRIMARY  | 8       | NULL | 10000   | Using join buffer (BKA) |
+----+-------------+-------+------+----------+---------+------+----------+-------------------------+
```

**问题**：`Using join buffer` 表示使用块嵌套循环联接，需要将小表作为驱动表。

**优化**：

```sql
-- 确保关联字段有索引
ALTER TABLE orders ADD INDEX idx_user_id (user_id);

-- 调整 JOIN 顺序（让小表先扫描）
EXPLAIN SELECT * FROM users u
INNER JOIN orders o ON o.user_id = u.id;
```

## 五、常见面试陷阱

:::danger 陷阱 1：只看 type 不看 Extra
错误理解：「type = ALL 一定慢，type = ref 一定快」
正确理解：`type` 只是连接类型，`Extra` 才能反映真实性能问题。例如 `type = ref` 但 `Using filesort` 可能比 `type = ALL` 更慢。
:::

:::danger 陷阱 2：rows 是实际扫描行数
错误理解：「rows = 100 表示只扫描了 100 行」
正确理解：`rows` 是优化器的估算值，不是实际值。使用 `EXPLAIN ANALYZE` 可以查看实际行数。
:::

:::danger 陷阱 3：key_len 越大索引效率越高
错误理解：「key_len = 200 比 key_len = 100 用的列多，所以更好」
正确理解：key_len 表示索引使用长度，但关键是看是否使用了所有需要的列，以及是否产生回表。
:::

## 六、索引优化判断标准

| type | key | Extra | 评价 |
|------|-----|-------|------|
| `const/eq_ref/ref` | ✅ 有 | `Using index` | ✅ 最佳 |
| `ref` | ✅ 有 | `Using where` | ✅ 好 |
| `range` | ✅ 有 | - | 较好 |
| `index` | ✅ 有 | - | 一般 |
| `ALL` | ❌ 无 | - | ❌ 需优化 |

## 七、加分回答

> 💡 **Optimizer Trace 分析**：
> 对于复杂查询，EXPLAIN 不足以说明为什么选择了某个执行计划，可以使用 `OPTIMIZER_TRACE`：
>
> ```sql
> SET SESSION optimizer_trace = "enabled=on";
> SELECT * FROM orders WHERE user_id = 100;
> SELECT * FROM information_schema.OPTIMIZER_TRACE;
> SET SESSION optimizer_trace = "enabled=off";
> ```
>
> 可以看到优化器选择执行计划的全过程，包括成本估算、备选方案对比等。

> 💡 **执行计划与 SQL 改写**：
> 有时优化器选择的计划不优，可以尝试 SQL 改写：
> - 强制使用索引：`SELECT * FROM orders FORCE INDEX(idx_user_id) WHERE user_id = 100;`
> - 改写子查询为 JOIN
> - 改写 OR 为 UNION
