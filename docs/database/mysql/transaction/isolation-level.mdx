# 四种隔离级别：从原理到实现

「MySQL 的隔离级别有哪些？默认是哪个？」

这个问题几乎是面试必问。但面试官更想知道的，是你是否理解**每个隔离级别是如何实现的，以及它们的性能差异**。

## 隔离级别的定义

SQL 标准定义了四种事务隔离级别，用来解决并发执行时可能出现的各种问题：

```
┌─────────────────────────────────────────────────────────────────┐
│                    四种隔离级别                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  隔离级别              允许的并发问题                              │
│  ───────────────────────────────────────────────────            │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  READ UNCOMMITTED（读未提交）                              │   │
│  │  允许读取其他事务未提交的数据                              │   │
│  │  ├─ 脏读                                               │   │
│  │  ├─ 不可重复读                                          │   │
│  │  └─ 幻读                                                │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  READ COMMITTED（读已提交）                               │   │
│  │  只读取其他事务已提交的数据                              │   │
│  │  ├─ 脏读（防止）                                         │   │
│  │  ├─ 不可重复读                                          │   │
│  │  └─ 幻读                                                │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  REPEATABLE READ（可重复读）★ MySQL 默认                 │   │
│  │  同一事务中多次读取结果一致                              │   │
│  │  ├─ 脏读（防止）                                         │   │
│  │  ├─ 不可重复读（防止）                                   │   │
│  │  └─ 幻读（InnoDB 通过临键锁防止）                       │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  SERIALIZABLE（串行化）                                  │   │
│  │  完全串行执行，强制事务排队                               │   │
│  │  ├─ 脏读（防止）                                         │   │
│  │  ├─ 不可重复读（防止）                                   │   │
│  │  └─ 幻读（防止）                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 如何设置和查看隔离级别

### 查看当前隔离级别

```sql
-- 查看当前会话隔离级别
SELECT @@transaction_isolation;

-- 或
SHOW VARIABLES LIKE 'transaction_isolation';
```

```
+-------------------+---------------------+
| Variable_name     | Value               |
+-------------------+---------------------+
| transaction_isolation | REPEATABLE-READ   |
+-------------------+---------------------+
```

### 设置隔离级别

```sql
-- 设置当前会话隔离级别
SET SESSION transaction_isolation = 'READ-COMMITTED';

-- 设置全局隔离级别（需要 SUPER 权限，影响新连接）
SET GLOBAL transaction_isolation = 'READ-COMMITTED';
```

### 隔离级别语法变化

:::tip 版本差异

MySQL 8.0 之前使用 `tx_isolation`，8.0 之后改为 `transaction_isolation`。在高版本 MySQL 中，两个变量都支持，但建议使用新的变量名。
:::

## 读未提交（READ UNCOMMITTED）

### 特点

读未提交是最弱的隔离级别，允许读取其他事务未提交的数据。

### 实现原理

读未提交几乎不加锁，采用**脏读**策略：

```
┌─────────────────────────────────────────────────────────────────┐
│                    读未提交实现原理                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  事务 A：                                                       │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  BEGIN;                                                  │   │
│  │  UPDATE accounts SET balance = 500 WHERE id = 1;         │   │
│  │  -- 此时 balance = 500，但未提交                         │   │
│  │                                                          │   │
│  │  -- 其他事务可以直接读取到这个未提交的值                  │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  问题：                                                        │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  如果事务 A ROLLBACK：                                   │   │
│  │  - 其他事务基于脏数据做的处理都是错的！                   │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 适用场景

| 场景 | 说明 |
|---|---|
| 数据统计分析 | 只关心聚合结果，不关心单条数据准确性 |
| 数据同步 | 将数据同步到只读备库 |
| 性能测试 | 测试极端情况下的性能 |

### SQL 示例

```sql
SET SESSION transaction_isolation = 'READ-UNCOMMITTED';

BEGIN;

-- 可以读取其他事务未提交的数据
SELECT balance FROM accounts WHERE id = 1;
-- 如果其他事务正在修改这条记录，会读到「脏数据」

COMMIT;
```

## 读已提交（READ COMMITTED）

### 特点

读已提交要求事务只能读取其他事务已提交的数据，解决了脏读问题。

### 实现原理

读已提交通过 **MVCC（多版本并发控制）** 实现：

```
┌─────────────────────────────────────────────────────────────────┐
│                    读已提交实现原理                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  每次读取数据时，都会生成新的 ReadView：                          │
│                                                                 │
│  T1: 事务 A 开始，生成 ReadView_A1                               │
│  T2: 事务 A 读取数据，看到 balance = 1000                        │
│  T3: 事务 B 开始，修改 balance = 500 并提交                       │
│  T4: 事务 A 再次读取，生成新的 ReadView_A2                        │
│  T5: 事务 A 读取数据，看到 balance = 500（已提交）               │
│                                                                 │
│  问题：同一事务中两次读取，结果不一致！                           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### ReadView 生成时机

```
┌─────────────────────────────────────────────────────────────────┐
│                    读已提交的 ReadView 生成时机                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  事务 A：                                                       │
│                                                                 │
│  BEGIN;  ──────────────────────────────────┐                   │
│       │                                     │                   │
│       ▼                                     │                   │
│  ┌─────────────────────────────────────────┐│                   │
│  │  SELECT balance FROM accounts WHERE id=1 ││ ← ReadView_1     │
│  │  结果：1000                               ││                   │
│  └─────────────────────────────────────────┘│                   │
│       │                                     │                   │
│       │     事务 B 修改数据并提交              │                   │
│       │                                     │                   │
│       ▼                                     │                   │
│  ┌─────────────────────────────────────────┐│                   │
│  │  SELECT balance FROM accounts WHERE id=1 ││ ← ReadView_2     │
│  │  结果：500  ← 变了！                     ││ 新生成！          │
│  └─────────────────────────────────────────┘│                   │
│                                                                 │
│  每次 SELECT 都会生成新的 ReadView，所以能看到最新已提交的数据     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 适用场景

| 场景 | 说明 |
|---|---|
| 业务需要最新数据 | 如订单状态查询 |
| 报表数据 | 需要实时准确的数据 |
| PostgreSQL 默认级别 | PostgreSQL 默认使用 RC |

### SQL 示例

```sql
SET SESSION transaction_isolation = 'READ-COMMITTED';

BEGIN;

-- 只能读取其他事务已提交的数据
SELECT balance FROM accounts WHERE id = 1;
-- 如果其他事务未提交，读不到

COMMIT;
```

## 可重复读（REPEATABLE READ）

### 特点

可重复读要求同一事务中多次读取同一数据，结果必须一致。这是 MySQL InnoDB 的默认隔离级别。

### 实现原理

可重复读通过 **MVCC + 一致性非锁定读** 实现：

```
┌─────────────────────────────────────────────────────────────────┐
│                    可重复读实现原理                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  事务开始时生成一个 ReadView，整个事务期间复用：                  │
│                                                                 │
│  T1: 事务 A 开始，生成 ReadView_A（m_ids = [B]）                 │
│  T2: 事务 A 读取 balance = 1000                                 │
│  T3: 事务 B 修改 balance = 500 并提交                          │
│  T4: 事务 A 再次读取：                                          │
│      - balance 的 trx_id = B（B 在 m_ids 中）                   │
│      - 不可见！                                                 │
│      - 通过 undo log 版本链找到之前的版本                        │
│      - 读取到 balance = 1000（和第一次一样）                    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### ReadView 生成时机

```
┌─────────────────────────────────────────────────────────────────┐
│                    可重复读的 ReadView 生成时机                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  事务 A：                                                       │
│                                                                 │
│  BEGIN;  ────────────────────────────────── ReadView_A 生成     │
│       │                                     │                   │
│       ▼                                     │                   │
│  ┌─────────────────────────────────────────┐│                   │
│  │  SELECT balance FROM accounts WHERE id=1 ││ ← ReadView_A     │
│  │  结果：1000                               ││ 复用              │
│  └─────────────────────────────────────────┘│                   │
│       │                                     │                   │
│       │     事务 B 修改数据并提交              │                   │
│       │                                     │                   │
│       ▼                                     ▼                   │
│  ┌─────────────────────────────────────────┐                   │
│  │  SELECT balance FROM accounts WHERE id=1 ││ ← ReadView_A     │
│  │  结果：1000  ← 没变！                    ││ 继续复用！        │
│  └─────────────────────────────────────────┘                   │
│                                                                 │
│  ReadView 在事务期间只生成一次，保证可重复读                     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### InnoDB 如何防止幻读？

可重复读隔离级别下，InnoDB 通过 **临键锁（Next-Key Lock）** 防止幻读：

```
┌─────────────────────────────────────────────────────────────────┐
│                    临键锁防止幻读                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  假设 users 表 age 字段有索引，数据为：                          │
│  age: 10, 20, 30, 40, 50                                         │
│                                                                 │
│  事务 A：                                                       │
│  BEGIN;                                                         │
│  SELECT * FROM users WHERE age = 25 FOR UPDATE;                 │
│                                                                 │
│  临键锁会锁定：                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  (20, 25]  ──── 记录锁：锁定 age=25 的记录              │   │
│  │  (25, 30)  ──── 间隙锁：锁定 25 和 30 之间的空隙       │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  事务 B：                                                       │
│  INSERT INTO users VALUES (6, '新用户', 26);                   │
│  -- 被阻塞！因为 26 在 (25, 30) 范围内                          │
│                                                                 │
│  幻读被防止！                                                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 临键锁详解

```
┌─────────────────────────────────────────────────────────────────┐
│                    临键锁结构                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  临键锁（Next-Key Lock）= 记录锁 + 间隙锁                       │
│                                                                 │
│  索引数据：10, 20, 30, 40, 50                                   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  锁定的区间：                                             │   │
│  │                                                         │   │
│  │  (-∞, 10]  ── 锁住 (-∞, 10]                             │   │
│  │  (10, 20]   ── 锁住 (10, 20]                             │   │
│  │  (20, 30]   ── 锁住 (20, 30]                             │   │
│  │  (30, 40]   ── 锁住 (30, 40]                             │   │
│  │  (40, 50]   ── 锁住 (40, 50]                             │   │
│  │  (50, +∞)  ── 锁住 (50, +∞)                             │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  查询 age = 25 时，会锁定 (20, 30] 这个区间                     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 适用场景

| 场景 | 说明 |
|---|---|
| MySQL 默认级别 | 适用于大多数业务场景 |
| 订单处理 | 防止订单重复 |
| 库存管理 | 防止超卖 |
| 数据一致性要求高 | 需要同一事务中数据一致 |

### SQL 示例

```sql
SET SESSION transaction_isolation = 'REPEATABLE-READ';  -- MySQL 默认

BEGIN;

-- 同一事务中多次读取，结果一致
SELECT balance FROM accounts WHERE id = 1;  -- 结果：1000
SELECT balance FROM accounts WHERE id = 1;  -- 结果：1000（不变）

COMMIT;
```

## 串行化（SERIALIZABLE）

### 特点

串行化是最强的隔离级别，通过强制事务串行执行，完全避免并发问题，但性能最差。

### 实现原理

串行化通过**锁机制**实现：

```
┌─────────────────────────────────────────────────────────────────┐
│                    串行化实现原理                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  所有读取操作都会加共享锁：                                       │
│                                                                 │
│  事务 A：                                                       │
│  BEGIN;                                                         │
│  SELECT * FROM accounts WHERE id = 1;  -- 加 S 锁              │
│                                                                 │
│  事务 B：                                                       │
│  BEGIN;                                                         │
│  SELECT * FROM accounts WHERE id = 1;  -- 可以加 S 锁（兼容）  │
│  UPDATE accounts SET balance = 500;     -- 被阻塞！需等 S 锁   │
│                                                                 │
│  写操作会阻塞读操作，读操作也会阻塞写操作：                       │
│  实际上是串行执行的！                                            │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### InnoDB 在串行化下的行为

```sql
SET SESSION transaction_isolation = 'SERIALIZABLE';

BEGIN;

-- 读操作会自动加锁
SELECT * FROM accounts WHERE id = 1;
-- InnoDB 会将 SELECT 转换为 SELECT ... LOCK IN SHARE MODE

COMMIT;
-- 释放锁
```

### 适用场景

| 场景 | 说明 |
|---|---|
| 强一致性场景 | 金融、转账等不允许任何并发问题的业务 |
| 小数据量 | 数据量小，串行化性能影响可接受 |
| 调试场景 | 排查并发问题 |

### 性能影响

```
┌─────────────────────────────────────────────────────────────────┐
│                    隔离级别与性能                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  隔离级别          并发性能          一致性                      │
│  ─────────────────────────────────────────────                  │
│                                                                 │
│  串行化           ██░░░░░░░░░░░░  最强                        │
│  可重复读         ████████░░░░░░░  强（InnoDB 防止幻读）       │
│  读已提交         ████████████░░░░  中等                        │
│  读未提交         ████████████████  弱                         │
│                                                                 │
│  串行化虽然一致性最强，但性能最差，实际很少使用                     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 四种隔离级别的对比

### 完整对比表

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 实现方式 | MySQL 默认 |
|---|---|---|---|---|---|
| 读未提交 | 可能 | 可能 | 可能 | 无锁 | 否 |
| 读已提交 | 不可能 | 可能 | 可能 | MVCC（每次读生成 ReadView） | PostgreSQL |
| 可重复读 | 不可能 | 不可能 | 可能 | MVCC（事务开始生成 ReadView）+ 临键锁 | **是** |
| 串行化 | 不可能 | 不可能 | 不可能 | 锁 | 否 |

### 场景选择建议

```
┌─────────────────────────────────────────────────────────────────┐
│                    隔离级别选择指南                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  推荐使用可重复读（MySQL 默认）：                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  - 大多数 OLTP 业务                                      │   │
│  │  - 需要事务回滚的场景                                    │   │
│  │  - 防止幻读非常重要的场景（库存、订单）                   │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  可选读已提交：                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  - 需要每次读取最新数据的场景                            │   │
│  │  - 统计分析类业务                                        │   │
│  │  - PostgreSQL 用户习惯                                   │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  避免使用：                                                      │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  - 读未提交：除非完全不需要一致性                        │   │
│  │  - 串行化：除非需要绝对一致性且数据量很小                │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 面试实战

### 问题一：MySQL 默认隔离级别是什么？如何实现的？

**面试官问**：「MySQL 默认的隔离级别是什么？它是如何实现的？」

**深入回答**：

> MySQL InnoDB 默认隔离级别是 **可重复读（Repeatable Read）**。
>
> 实现机制：
> 1. **MVCC**：事务开始时生成 ReadView，整个事务期间复用
> 2. **临键锁**：防止幻读，锁定记录和间隙
>
> ```sql
> -- 查看隔离级别
> SELECT @@transaction_isolation;
> -- REPEATABLE-READ
> ```

### 问题二：MVCC 在 RC 和 RR 级别下的区别？

**面试官问**：「MVCC 在读已提交和可重复读级别下有什么区别？」

**深入回答**：

> 核心区别在于 **ReadView 生成时机**：
>
> | 隔离级别 | ReadView 生成时机 | 结果 |
> |---|---|---|
> | 读已提交 | **每次 SELECT** 都生成新的 | 可能看到其他事务提交的新数据 |
> | 可重复读 | **事务开始时** 生成一次 | 整个事务看到的数据一致 |
>
> ```sql
> -- 读已提交：每次读取可能不同
> SELECT balance;  -- 1000
> -- 其他事务提交
> SELECT balance;  -- 500（变了！）
>
> -- 可重复读：整个事务一致
> SELECT balance;  -- 1000
> -- 其他事务提交
> SELECT balance;  -- 1000（不变）
> ```

### 问题三：如何选择隔离级别？

**面试官问**：「在项目中应该如何选择事务隔离级别？」

**深入回答**：

> 选择隔离级别需要权衡**一致性**和**性能**：
>
> 1. **优先使用默认级别（可重复读）**：
>    - MySQL 默认，兼容性最好
>    - InnoDB 对 RR 优化最好
>    - 防止幻读能力强
>
> 2. **读已提交适用场景**：
>    - 需要实时数据的场景
>    - 统计分析类业务
>    - 可以接受不可重复读
>
> 3. **不建议使用**：
>    - 读未提交：除非明确知道业务可以容忍脏读
>    - 串行化：除非数据量很小且一致性要求极高
>
> **最佳实践**：
> - 大多数场景使用默认的 RR
> - 如果有性能问题，先排查慢查询和索引，而不是降低隔离级别
> - 读未提交和串行化很少在生产环境使用

## 总结

四种隔离级别从低到高：

- **读未提交**：最弱，允许脏读，几乎不使用
- **读已提交**：解决脏读，每次读生成新 ReadView
- **可重复读**：MySQL 默认，整个事务复用 ReadView + 临键锁防幻读
- **串行化**：最强，性能最差，强制串行执行

理解隔离级别的实现原理，是掌握 MySQL 并发控制的关键。
