# MVCC 在 RC 和 RR 级别下的差异

「为什么可重复读级别下，同一事务中两次查询结果一致？」

这个问题的答案，藏在 RC 和 RR 两个隔离级别下 ReadView 生成时机的差异里。

## 核心差异概述

```
┌─────────────────────────────────────────────────────────────────┐
│                    RC vs RR 核心差异                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  读已提交（RC）                                          │   │
│  │                                                         │   │
│  │  ReadView 生成时机：每次 SELECT                          │   │
│  │  效果：可能看到其他事务新提交的数据                       │   │
│  │  结果：同一事务中两次查询，结果可能不同                   │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  可重复读（RR）★ MySQL 默认                              │   │
│  │                                                         │   │
│  │  ReadView 生成时机：事务开始时                            │   │
│  │  效果：看不到其他事务新提交的数据                         │   │
│  │  结果：同一事务中两次查询，结果完全一致                   │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## ReadView 生成时机对比

### RC 级别：每次 SELECT 生成新 ReadView

```
┌─────────────────────────────────────────────────────────────────┐
│                    RC 级别 ReadView 生成时机                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  事务 A（RC）：                                                 │
│                                                                 │
│  T1: BEGIN;                                                     │
│  T2: SELECT * FROM users WHERE id = 1;                          │
│      ┌─────────────────────────────────────────────────────────┐ │
│      │  生成 ReadView_A1                                     │ │
│      │  m_ids = []（假设无其他活跃事务）                     │ │
│      │  读取数据：balance = 1000                              │ │
│      └─────────────────────────────────────────────────────────┘ │
│  T3: （其他事务修改 balance = 500 并提交）                        │
│  T4: SELECT * FROM users WHERE id = 1;                          │
│      ┌─────────────────────────────────────────────────────────┐ │
│      │  生成 ReadView_A2（新的！）                           │ │
│      │  m_ids = []                                           │ │
│      │  读取数据：balance = 500（变了！）                    │ │
│      └─────────────────────────────────────────────────────────┘ │
│  T5: COMMIT;                                                    │
│                                                                 │
│  结果：同一事务中两次读取，结果不同                               │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### RR 级别：事务开始时生成 ReadView

```
┌─────────────────────────────────────────────────────────────────┐
│                    RR 级别 ReadView 生成时机                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  事务 A（RR）：                                                 │
│                                                                 │
│  T1: BEGIN;                                                     │
│      ┌─────────────────────────────────────────────────────────┐ │
│      │  生成 ReadView_A（只生成一次！）                       │ │
│      │  m_ids = []（假设无其他活跃事务）                     │ │
│      └─────────────────────────────────────────────────────────┘ │
│  T2: SELECT * FROM users WHERE id = 1;                          │
│      ┌─────────────────────────────────────────────────────────┐ │
│      │  使用 ReadView_A                                       │ │
│      │  读取数据：balance = 1000                              │ │
│      └─────────────────────────────────────────────────────────┘ │
│  T3: （其他事务修改 balance = 500 并提交）                        │
│  T4: SELECT * FROM users WHERE id = 1;                          │
│      ┌─────────────────────────────────────────────────────────┐ │
│      │  复用 ReadView_A（不重新生成！）                      │ │
│      │  balance 仍然是 1000（没变！）                        │ │
│      └─────────────────────────────────────────────────────────┘ │
│  T5: COMMIT;                                                    │
│                                                                 │
│  结果：同一事务中两次读取，结果完全一致                           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 详细对比示例

### 数据准备

```sql
-- 创建测试表
CREATE TABLE accounts (
    id INT PRIMARY KEY,
    name VARCHAR(50),
    balance DECIMAL(10, 2)
);

-- 初始数据
INSERT INTO accounts VALUES (1, '张三', 1000.00);

-- 查看当前隔离级别
SELECT @@transaction_isolation;
-- REPEATABLE-READ（MySQL 默认）
```

### RC 级别演示

```sql
-- 会话 1：设置 RC 级别
SET SESSION transaction_isolation = 'READ-COMMITTED';

-- 会话 1：开始事务
BEGIN;
SELECT balance FROM accounts WHERE id = 1;
-- 结果：1000.00

-- 会话 2：修改数据并提交
UPDATE accounts SET balance = 500.00 WHERE id = 1;
COMMIT;

-- 会话 1：再次查询
SELECT balance FROM accounts WHERE id = 1;
-- 结果：500.00 ← 变了！

COMMIT;
```

### RR 级别演示

```sql
-- 会话 1：设置 RR 级别（MySQL 默认）
SET SESSION transaction_isolation = 'REPEATABLE-READ';

-- 会话 1：开始事务
BEGIN;
SELECT balance FROM accounts WHERE id = 1;
-- 结果：1000.00

-- 会话 2：修改数据并提交
UPDATE accounts SET balance = 500.00 WHERE id = 1;
COMMIT;

-- 会话 1：再次查询
SELECT balance FROM accounts WHERE id = 1;
-- 结果：1000.00 ← 没变！

COMMIT;
```

## ReadView 字段在不同级别下的表现

### RC 级别：ReadView 随每次查询更新

```
┌─────────────────────────────────────────────────────────────────┐
│                    RC 级别 ReadView 生命周期                      │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  T1: BEGIN;                                                     │
│  T2: SELECT ...  ────→ ReadView_1 { m_ids: [B] }               │
│      结果：V1                                                   │
│  T3: 事务 B 提交                                                │
│  T4: SELECT ...  ────→ ReadView_2 { m_ids: [] }  ← 重新生成！  │
│      结果：V2（新数据可见）                                      │
│  T5: SELECT ...  ────→ ReadView_3 { m_ids: [C] }               │
│      结果：V2 或 V3（取决于 C）                                  │
│                                                                 │
│  ReadView 每次都重新生成，m_ids 反映当前活跃事务                 │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### RR 级别：ReadView 事务内复用

```
┌─────────────────────────────────────────────────────────────────┐
│                    RR 级别 ReadView 生命周期                      │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  T1: BEGIN;  ────→ ReadView { m_ids: [] }  ← 事务开始时生成     │
│  T2: SELECT ...  ────→ ReadView { m_ids: [] }  ← 复用           │
│      结果：V1                                                   │
│  T3: 事务 B 提交                                                │
│  T4: SELECT ...  ────→ ReadView { m_ids: [] }  ← 继续复用       │
│      结果：V1（看不到 B 提交的数据！）                           │
│  T5: SELECT ...  ────→ ReadView { m_ids: [] }  ← 继续复用       │
│      结果：V1（始终不变）                                        │
│  T6: COMMIT;                                                   │
│                                                                 │
│  ReadView 只在事务开始时生成，m_ids 快照在整个事务期间不变         │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 幻读问题在两种级别下的表现

### RC 级别：可能出现幻读

```sql
-- 会话 1：RC 级别
SET SESSION transaction_isolation = 'READ-COMMITTED';
BEGIN;

-- 第一次统计用户数量
SELECT COUNT(*) FROM users WHERE age = 25;
-- 结果：3

-- 会话 2：插入新用户并提交
INSERT INTO users (name, age) VALUES ('新用户', 25);
COMMIT;

-- 会话 1：再次统计
SELECT COUNT(*) FROM users WHERE age = 25;
-- 结果：4 ← 幻读！多了 1 行

COMMIT;
```

### RR 级别：通过临键锁防止幻读

```sql
-- 会话 1：RR 级别
SET SESSION transaction_isolation = 'REPEATABLE-READ';
BEGIN;

-- 第一次统计
SELECT COUNT(*) FROM users WHERE age = 25;
-- 结果：3

-- 会话 2：尝试插入新用户
INSERT INTO users (name, age) VALUES ('新用户', 25);
-- 被阻塞！因为会话 1 使用 FOR UPDATE 或 临键锁锁定了区间

-- 会话 1：再次统计
SELECT COUNT(*) FROM users WHERE age = 25;
-- 结果：3 ← 没有幻读

COMMIT;
```

### RC vs RR 性能对比

```
│                    RC vs RR 性能对比                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                    RC（读已提交）                        │   │
│  │                                                         │   │
│  │  优点：                                                  │   │
│  │  - 每次读取最新数据，数据更实时                          │   │
│  │  - undo log 清理更快，节省空间                          │   │
│  │                                                         │   │
│  │  缺点：                                                  │   │
│  │  - 同一事务中数据可能不一致                             │   │

│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                    RR（可重复读）                        │   │
│  │                                                         │   │
│  │  优点：                                                  │   │
│  │  - 事务内数据一致                                        │   │
│  │  - InnoDB 默认，优化最好                                │   │
│  │                                                         │   │
│  │  缺点：                                                  │   │
│  │  - ReadView 保留时间更长                                │   │
│  │  - undo log 不能及时清理                                │   │
│  │  - 长事务可能占用更多内存                               │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 长事务的影响

```sql
-- RR 级别下的长事务问题
BEGIN;

-- 事务开始，生成 ReadView
SELECT * FROM users WHERE id = 1;

-- 1 小时后...
-- 其他事务修改并提交了数据

SELECT * FROM users WHERE id = 1;
-- 仍然看到旧数据（因为 ReadView 不变）

-- 问题：
-- 1. 其他事务提交的 undo log 不能及时清理
-- 2. undo log 版本链可能很长
-- 3. 后续查询需要遍历更长的版本链

COMMIT;
```

## 实际选择建议

### 何时使用 RC

| 场景 | 原因 |
|---|---|
| 需要实时数据 | 每次查询都希望看到最新 |
| 统计分析 | 不需要事务一致性 |
| PostgreSQL 迁移 | PostgreSQL 默认 RC |

### 何时使用 RR

| 场景 | 原因 |
|---|---|
| MySQL 默认 | 兼容性最好 |
| 需要事务一致性 | 同一事务数据一致 |
| 防止幻读重要 | 如库存、订单 |

### 混合使用

```
┌─────────────────────────────────────────────────────────────────┐
│                    实际项目中的选择                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  大多数 OLTP 场景：                                             │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  使用 RR（MySQL 默认）                                   │   │
│  │  - 兼容性最好                                            │   │
│  │  - InnoDB 优化最好                                       │   │
│  │  - 防止幻读能力强                                        │   │
```

## 面试实战

### 问题一：RC 和 RR 级别下 MVCC 的核心区别是什么？

**面试官问**：「MVCC 在读已提交和可重复读级别下有什么区别？」

**深入回答**：

> **核心区别：ReadView 生成时机不同**
>
> | 隔离级别 | ReadView 生成时机 | 同一事务多次读取 |
> |---|---|---|
> | 读已提交 | **每次 SELECT** 都生成 | 结果可能不同 |
> | 可重复读 | **事务开始时** 生成一次 | 结果完全一致 |
>
> **原理**：
> - RC：每次查询都创建新 ReadView，`m_ids` 反映当前活跃事务
> - RR：事务开始时创建 ReadView，`m_ids` 是事务开始时的快照
>
> **实际影响**：
> - RC 级别下，可能看到其他事务新提交的数据
> - RR 级别下，看不到其他事务在事务开始后提交的数据

### 问题二：RR 级别下是否就不会出现幻读？

**面试官问**：「可重复读级别下还会出现幻读吗？」

**深入回答**：

> **在 InnoDB 中，RR 级别下幻读被极大缓解，但不是完全消除**：
>
> **防止幻读的机制**：
> 1. **快照读**（普通 SELECT）：通过 MVCC 防止
> 2. **当前读**（SELECT ... FOR UPDATE）：通过临键锁防止
>
> **可能出现幻读的场景**：
> ```sql
> -- 事务 A（RR）
> BEGIN;
>
> -- 快照读：MVCC 保护
> SELECT * FROM users WHERE age = 25;
> -- 返回 3 行
>
> -- 当前读：临键锁保护
> INSERT INTO users VALUES (100, '测试', 25);
> -- 如果没有提前锁定，可能插入成功！
>
> -- 再次快照读
> SELECT * FROM users WHERE age = 25;
> -- 返回 4 行 ← 幻读！
> ```
>
> **解决方案**：使用 `SELECT ... FOR UPDATE` 或 `LOCK IN SHARE MODE` 进行当前读

### 问题三：如何选择 RC 和 RR？

**面试官问**：「在项目中应该选择读已提交还是可重复读？」

**深入回答**：

> **推荐选择 RR（可重复读）**：
> 1. MySQL InnoDB 默认隔离级别，兼容性最好
> 2. InnoDB 对 RR 的优化最完善
> 3. 防止幻读能力最强
> 4. 同一事务数据一致性好
>
> **选择 RC 的场景**：
> 1. 需要每次读取最新数据
> 2. 从 PostgreSQL 迁移（保持行为一致）
> 3. 明确知道业务可以接受不可重复读
>
> **不建议**：
> - 除非有特殊需求，不要轻易切换隔离级别
> - 性能问题应该通过优化 SQL 和索引解决，而不是降低隔离级别

## 总结

RC 和 RR 在 MVCC 实现上的核心差异：

| 维度 | 读已提交（RC） | 可重复读（RR） |
|---|---|---|
| ReadView 生成时机 | 每次 SELECT | 事务开始时 |
| 同一事务多次读取 | 可能不同 | 始终一致 |
| 幻读 | 可能出现 | 配合锁防止 |
| 性能 | 较好（undo log 清理快） | 稍差（长事务影响） |
| 适用场景 | 实时数据、报表 | 事务一致性要求高 |

理解 RC 和 RR 的差异，是掌握 MySQL 事务隔离的关键一步。
