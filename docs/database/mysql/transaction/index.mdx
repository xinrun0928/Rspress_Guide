# 事务专题总览

事务是数据库最核心的特性之一。在生产环境中，并发是常态——多个用户同时读写数据库，如果没有事务保证，数据一致性和完整性将无法保证。

MySQL 的事务机制是面试中的重点考察领域。ACID 特性、隔离级别、MVCC 原理、快照读与当前读——每一个点都可能成为面试官深挖的方向。

本模块带你深入理解 MySQL 事务的底层实现原理。

## 为什么需要事务？

考虑一个转账场景：

```sql
-- 从 A 账户转 100 元到 B 账户
UPDATE accounts SET balance = balance - 100 WHERE user_id = 'A';
UPDATE accounts SET balance = balance + 100 WHERE user_id = 'B';
```

如果第一条 SQL 执行成功后，第二条 SQL 执行前系统崩溃了——A 的钱扣了，B 的钱没增加，数据不一致。事务保证了这两个操作要么全部成功，要么全部失败回滚。

## ACID 特性

事务有四个核心特性，简称 ACID：

**原子性（Atomicity）**：事务是最小执行单元，其中的操作要么全部成功，要么全部失败回滚。通过 Undo Log 实现。

**一致性（Consistency）**：事务执行前后，数据库的完整性约束没有被破坏。一致性是最终目标，由原子性、隔离性、持久性共同保证。

**隔离性（Isolation）**：并发执行的事务相互隔离，不互相干扰。通过锁机制和 MVCC 实现。

**持久性（Durability）**：事务提交后，其结果永久保存在数据库中。通过 Redo Log 实现。

## 隔离级别

MySQL 有四种隔离级别，从低到高：

| 隔离级别 | 脏读 | 不可重复读 | 幻读 |
|---|---|---|---|
| READ UNCOMMITTED | 可能 | 可能 | 可能 |
| READ COMMITTED | 不可能 | 可能 | 可能 |
| REPEATABLE READ | 不可能 | 不可能 | 可能 |
| SERIALIZABLE | 不可能 | 不可能 | 不可能 |

**脏读**：读取到其他事务未提交的数据。

**不可重复读**：同一事务内两次读取同一行数据，结果不同（其他事务修改了数据）。

**幻读**：同一事务内两次查询，结果集不一样（其他事务插入了新数据）。

## MVCC 原理

MVCC（Multi-Version Concurrency Control）是 InnoDB 实现的并发控制机制。通过保存数据的多个快照版本，实现读写并发——读不加锁，写不加锁，互不影响。

InnoDB 为每一行数据添加了两个隐藏列：`DB_TRX_ID`（最近修改的事务ID）和 `DB_ROLL_PTR`（指向 Undo Log 的指针）。每次修改都会生成一条 Undo Log，形成版本链。

读事务开始时生成 ReadView，包含活跃事务 ID 列表。读取数据时，通过版本链和 ReadView 判断哪个版本对当前事务可见。
