# 当前读与快照读

「为什么有时候查询结果和我预期的不一样？」

这个问题的答案，在于 MySQL 的两种读取方式：快照读和当前读。它们是 MVCC 实现的基础，也是理解事务隔离的关键。

## 两种读取方式概述

```
┌─────────────────────────────────────────────────────────────────┐
│                    快照读 vs 当前读                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                    快照读（Snapshot Read）               │   │
│  │                                                         │   │
│  │  读取：历史版本的数据（基于 ReadView）                    │   │
│  │  加锁：不加锁                                            │   │
│  │  性能：高，并发好                                        │   │
│  │  场景：普通 SELECT 语句                                  │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                    当前读（Current Read）               │   │
│  │                                                         │   │
│  │  读取：最新版本的数据（加锁）                             │   │
│  │  加锁：加锁（如排他锁、共享锁）                          │   │
│  │  性能：较低，但保证数据最新                              │   │
│  │  场景：SELECT ... FOR UPDATE、INSERT、UPDATE、DELETE    │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 快照读（Snapshot Read）

### 什么是快照读？

快照读读取的是**数据的历史版本**，基于 ReadView 实现，不加锁，可以并发执行。

### 快照读的 SQL

```sql
-- 以下都是快照读
SELECT * FROM users;
SELECT * FROM users WHERE id = 1;
SELECT name, age FROM users WHERE age > 20;
```

### 快照读的原理

```
┌─────────────────────────────────────────────────────────────────┐
│                    快照读原理                                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  快照读的工作过程：                                              │
│                                                                 │
│  1. 生成/获取 ReadView                                          │
│     ┌─────────────────────────────────────────────────────────┐ │
│     │  ReadView {                                            │ │
│     │    creator_trx_id: 当前事务 ID                         │ │
│     │    m_ids: 活跃事务列表                                  │ │
│     │    min_trx_id, max_trx_id                              │ │
│     │  }                                                     │ │
│     └─────────────────────────────────────────────────────────┘ │
│                          │                                      │
│                          ▼                                      │
│  2. 读取数据，遍历版本链                                         │
│     ┌─────────────────────────────────────────────────────────┐ │
│     │  当前记录 ──→ 检查可见性 ──→ 不可见 ──→ undo log ──→... │ │
│     └─────────────────────────────────────────────────────────┘ │
│                          │                                      │
│                          ▼                                      │
│  3. 返回可见版本的数据                                          │
│                                                                 │
│  关键点：不加锁，读取历史版本                                    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 快照读的示例

```sql
-- 事务 A（RR 级别）
BEGIN;

-- 快照读：读取 ReadView
SELECT * FROM users WHERE id = 1;
-- 基于 ReadView，读取 balance = 1000

-- 事务 B 修改并提交
UPDATE users SET balance = 500 WHERE id = 1;
COMMIT;

-- 事务 A 再次快照读
SELECT * FROM users WHERE id = 1;
-- 复用 ReadView，仍然读取 balance = 1000

COMMIT;
```

## 当前读（Current Read）

### 什么是当前读？

当前读读取的是**数据的最新版本**，并加锁，保证读取的数据是最新的且不被其他事务修改。

### 当前读的 SQL

```sql
-- 当前读示例
SELECT * FROM users WHERE id = 1 FOR UPDATE;   -- 排他锁
SELECT * FROM users WHERE id = 1 LOCK IN SHARE MODE;  -- 共享锁
INSERT INTO users VALUES (1, '张三', 25);      -- 插入（加锁）
UPDATE users SET name = '李四' WHERE id = 1;   -- 更新（加排他锁）
DELETE FROM users WHERE id = 1;                -- 删除（加排他锁）
```

### 当前读的原理

```
┌─────────────────────────────────────────────────────────────────┐
│                    当前读原理                                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  当前读的工作过程：                                              │
│                                                                 │
│  1. 先读取最新数据（快照读）                                      │
│                                                                 │
│  2. 加锁（防止其他事务修改）                                     │
│     ┌─────────────────────────────────────────────────────────┐ │
│     │  SELECT ... FOR UPDATE → 排他锁（X Lock）               │ │
│     │  SELECT ... LOCK IN SHARE MODE → 共享锁（S Lock）       │ │
│     └─────────────────────────────────────────────────────────┘ │
│                          │                                      │
│                          ▼                                      │
│  3. 返回最新版本的数据                                          │
│                          │                                      │
│                          ▼                                      │
│  4. 事务提交/回滚时释放锁                                        │
│                                                                 │
│  关键点：加锁，读取最新版本                                      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 当前读的示例

```sql
-- 事务 A：当前读
BEGIN;
SELECT * FROM users WHERE id = 1 FOR UPDATE;
-- 读取最新数据，加排他锁

-- 事务 B：尝试修改
BEGIN;
UPDATE users SET balance = 500 WHERE id = 1;
-- 被阻塞！等待事务 A 释放锁

-- 事务 A：提交
COMMIT;
-- 释放锁

-- 事务 B：继续执行（事务 A 提交后的数据）
-- UPDATE 执行成功
COMMIT;
```

## 快照读 vs 当前读

### 核心区别

```
┌─────────────────────────────────────────────────────────────────┐
│                    快照读 vs 当前读对比                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  快照读（SELECT）                                         │   │
│  │                                                         │   │
│  │  - 不加锁                                               │   │
│  │  - 读取历史版本（基于 ReadView）                         │   │
│  │  - 读写不冲突                                            │   │
│  │  - 性能高，并发好                                        │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  当前读（SELECT ... FOR UPDATE / LOCK IN SHARE MODE）   │   │
│  │                                                         │   │
│  │  - 加锁（排他锁或共享锁）                               │   │
│  │  - 读取最新版本                                         │   │
│  │  - 与其他锁冲突                                        │   │
│  │  - 保证数据最新                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 对比表格

| 维度 | 快照读 | 当前读 |
|---|---|---|
| 加锁 | 不加锁 | 加锁（排他锁/共享锁） |
| 读取版本 | 历史版本（ReadView） | 最新版本 |
| 并发性能 | 高 | 低（需要等锁） |
| 数据实时性 | 可能不是最新 | 一定是最新 |
| 适用场景 | 普通查询 | 需要最新数据或修改数据 |

## 混合使用场景

### 典型的事务场景

```sql
-- 场景：转账操作，需要读取最新数据并修改

BEGIN;

-- Step 1: 当前读（读取最新余额，加锁）
SELECT balance FROM accounts WHERE user_id = 1 FOR UPDATE;
-- 读取 balance = 1000，加排他锁

-- Step 2: 业务逻辑判断
-- balance >= 1000?

-- Step 3: 更新（隐式当前读）
UPDATE accounts SET balance = balance - 100 WHERE user_id = 1;
-- 在排他锁保护下执行

-- Step 4: 提交
COMMIT;
-- 释放锁，事务完成
```

### 库存扣减场景

```sql
-- 场景：库存扣减，防止超卖

BEGIN;

-- 当前读：查询库存并加锁
SELECT stock FROM products WHERE product_id = 1 FOR UPDATE;
-- 读取 stock = 10，加排他锁

-- 业务逻辑
IF (stock >= 1) {
    -- 更新库存
    UPDATE products SET stock = stock - 1 WHERE product_id = 1;
}

COMMIT;
```

### 订单状态更新

```sql
-- 场景：更新订单状态

BEGIN;

-- 当前读：查询订单并加锁
SELECT * FROM orders WHERE order_id = 100 FOR UPDATE;
-- 读取订单最新状态，加排他锁

-- 业务逻辑：只有状态为"待支付"才能修改
if (status == '待支付') {
    UPDATE orders SET status = '已支付' WHERE order_id = 100;
}

COMMIT;
```

## 常见问题

### 问题一：为什么普通 SELECT 查询不到自己未提交的数据？

```sql
-- 事务 A
BEGIN;
UPDATE users SET name = '李四' WHERE id = 1;
-- 修改但未提交

-- 在同一个事务中
SELECT * FROM users WHERE id = 1;
-- 结果：name = '李四'
-- 原因：当前事务读取自己的修改，即使未提交
```

这是因为**当前事务修改的数据对自己总是可见的**，MVCC 的可见性判断中有一条：`trx_id == creator_trx_id → 可见`。

### 问题二：当前读会阻塞快照读吗？

```sql
-- 事务 A：当前读
BEGIN;
SELECT * FROM users WHERE id = 1 FOR UPDATE;
-- 加排他锁

-- 事务 B：快照读
BEGIN;
SELECT * FROM users WHERE id = 1;
-- 可以读取！快照读不加锁
```

**快照读不会被当前读阻塞**。这是 MVCC 的核心优势——读写不冲突。

### 问题三：快照读会阻塞当前读吗？

```sql
-- 事务 A：快照读
BEGIN;
SELECT * FROM users WHERE id = 1;
-- 不加锁

-- 事务 B：当前读
SELECT * FROM users WHERE id = 1 FOR UPDATE;
-- 可以执行！快照读不加锁
```

**快照读也不会阻塞当前读**。

## 面试实战

### 问题一：什么是快照读和当前读？它们的区别是什么？

**面试官问**：「什么是快照读和当前读？它们有什么区别？」

**深入回答**：

> **快照读（Snapshot Read）**：
> - 普通 SELECT 语句
> - 基于 ReadView 读取历史版本
> - 不加锁，并发性能高
>
> **当前读（Current Read）**：
> - SELECT ... FOR UPDATE、SELECT ... LOCK IN SHARE MODE
> - INSERT、UPDATE、DELETE
> - 读取最新版本并加锁
> - 保证数据最新
>
> **核心区别**：
> - 快照读不加锁，读取历史版本
> - 当前读加锁，读取最新版本

### 问题二：MVCC 和锁是什么关系？

**面试官问**：「MVCC 和锁是什么关系？它们是如何协作的？」

**深入回答**：

> MVCC 和锁是两种不同的并发控制机制：
>
> | 机制 | 作用 | 加锁 |
> |---|---|---|
> | MVCC | 控制读写并发 | 快照读不加锁 |
> | 锁 | 控制写写并发 | 当前读加锁 |
>
> **协作关系**：
> - **快照读**：只用 MVCC，不加锁
> - **当前读**：MVCC + 锁（读取最新版本并加锁）
> - **写操作**：使用锁，读取最新版本
>
> **为什么需要两者**：
> - MVCC 让读写不冲突，提高并发
> - 锁让写写互斥，保证数据一致性

### 问题三：什么场景下需要使用当前读？

**面试官问**：「什么场景下需要使用 SELECT ... FOR UPDATE？」

**深入回答**：

> **需要使用当前读的场景**：
>
> 1. **更新操作前读取最新数据**：
> ```sql
> SELECT * FROM orders WHERE id = 100 FOR UPDATE;
> UPDATE orders SET status = '已支付' WHERE id = 100;
> ```
>
> 2. **防止并发更新**：
> ```sql
> SELECT balance FROM accounts WHERE user_id = 1 FOR UPDATE;
> -- 其他事务无法修改这条记录
> ```
>
> 3. **库存扣减等防超卖场景**：
> ```sql
> SELECT stock FROM products WHERE id = 1 FOR UPDATE;
> -- 保证只有一个事务能读取并更新库存
> ```
>
> **注意事项**：
> - 当前读会加锁，可能导致其他事务阻塞
> - 事务应尽快完成，避免长时间锁表
> - 合理使用索引，减少锁的范围

## 总结

快照读和当前读是 MySQL 并发控制的基础：

- **快照读**：基于 MVCC，读取历史版本，不加锁，并发性能高
- **当前读**：MVCC + 锁，读取最新版本，加锁，保证数据一致性

理解两者的区别和使用场景，是掌握 MySQL 事务和锁机制的关键。
