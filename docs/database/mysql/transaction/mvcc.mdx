# MVCC 原理：ReadView 与版本链

「什么是 MVCC？为什么 RR 级别下同一事务多次读取结果一致？」

这个问题是 MySQL 面试的深水区。十个人里有八个听说过 MVCC，但能讲清楚 ReadView 和版本链的，可能不到两个。

本文从 undo log 版本链讲起，彻底讲透 MVCC 的工作原理。

## MVCC 是什么？

MVCC（Multi-Version Concurrency Control，多版本并发控制）是一种**并发控制**机制，通过记录数据的多个版本，实现**读写不冲突**，大幅提升数据库并发性能。

```
┌─────────────────────────────────────────────────────────────────┐
│                    MVCC 核心思想                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  传统锁机制（读写冲突）：                                         │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  事务 A 读取数据 ──→ 加锁 ──→ 读取完成 ──→ 解锁        │   │
│  │  事务 B 写入数据 ──→ 等锁 ──→ 等待 ──→ 继续           │   │
│  │                                                         │   │
│  │  问题：读和写互相阻塞，性能差                           │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  MVCC（读写不冲突）：                                            │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  事务 A 读取数据 ──→ 读取快照版本 ──→ 完成（不阻塞）    │   │
│  │  事务 B 写入数据 ──→ 创建新版本 ──→ 完成（不阻塞）      │   │
│  │                                                         │   │
│  │  原理：每个事务看到的是数据的某个历史版本                 │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### MVCC 的核心优势

| 优势 | 说明 |
|---|---|
| 读写不阻塞 | 读操作不需要加锁，可以读取历史快照 |
| 并发性能高 | 读读、读写可以并发执行 |
| 数据一致性 | 通过版本链保证事务看到一致的数据 |

## undo log 版本链

### 什么是版本链？

InnoDB 中，每条记录都有两个隐藏列：`DB_TRX_ID`（事务ID）和 `DB_ROLL_PTR`（回滚指针）。这两个隐藏列是 MVCC 的基础。

```
┌─────────────────────────────────────────────────────────────────┐
│                    记录的行格式                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  InnoDB 的记录结构：                                            │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  ┌─────────────────────────────────────────────────┐   │   │
│  │  │  DB_ROW_ID (6字节)  - 行ID，无主键时自动生成      │   │   │
│  │  ├─────────────────────────────────────────────────┤   │   │
│  │  │  DB_TRX_ID (6字节)  - 事务ID，最近修改的事务ID   │   │   │
│  │  ├─────────────────────────────────────────────────┤   │   │
│  │  │  DB_ROLL_PTR (7字节) - 回滚指针，指向 undo log   │   │   │
│  │  └─────────────────────────────────────────────────┘   │   │
│  │                                                         │   │
│  │  +---------------------------------------------------------+ │
│  │  |  列1  |  列2  |  列3  |  ...  |  业务数据列          | │
│  │  +---------------------------------------------------------+ │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 版本链的形成

当一行数据被修改时，InnoDB 会创建 undo log 记录旧值，并通过 `DB_ROLL_PTR` 链接起来，形成版本链：

```
┌─────────────────────────────────────────────────────────────────┐
│                    版本链形成过程                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  初始状态：                                                     │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  记录 v1                                              │   │
│  │  name = '张三', age = 25, trx_id = 10                │   │
│  │  roll_ptr = NULL                                      │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  事务 20 修改 name = '李四'：                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  1. 创建 undo log，记录旧值：name='张三'                │   │
│  │  2. 修改记录为 name='李四'，trx_id=20                 │   │
│  │  3. roll_ptr 指向 undo log                             │   │
│  │                                                         │   │
│  │  ┌─────────────────────────────────────────────────┐   │   │
│  │  │  记录 v2（最新）                                 │   │   │
│  │  │  name='李四', age=25, trx_id=20, roll_ptr=──┐   │   │
│  │  └─────────────────────────────────────────────────┘   │   │
│  │                                                  │         │
│  │  ┌─────────────────────────────────────────────────┐   │   │
│  │  │  undo log（v1）  ◀─── roll_ptr                 │   │   │
│  │  │  name='张三', trx_id=10                        │   │   │
│  │  └─────────────────────────────────────────────────┘   │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  事务 30 修改 age = 30：                                        │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  ┌─────────────────────────────────────────────────┐   │   │
│  │  │  记录 v3（最新）                                 │   │   │
│  │  │  name='李四', age=30, trx_id=30, roll_ptr=──┐   │   │
│  │  └─────────────────────────────────────────────────┘   │   │
│  │                                                  │         │
│  │  ┌─────────────────────────────────────────────────┐   │   │
│  │  │  undo log（v2）  ◀─── roll_ptr                 │   │   │
│  │  │  name='李四', age=25, trx_id=20                │   │   │
│  │  └─────────────────────────────────────────────────┘   │   │
│  │                          │                               │
│  │  ┌─────────────────────────────────────────────────┐   │   │
│  │  │  undo log（v1）                                 │   │   │
│  │  │  name='张三', age=25, trx_id=10                 │   │   │
│  │  └─────────────────────────────────────────────────┘   │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 版本链的遍历

读取数据时，通过 `roll_ptr` 可以遍历整个版本链：

```
┌─────────────────────────────────────────────────────────────────┐
│                    版本链遍历过程                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  当前记录（最新版本）：                                          │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  name='李四', age=30, trx_id=30, roll_ptr=──┐           │   │
│  └─────────────────────────────────────────────────┘           │
│                                                 │               │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  undo log（trx_id=20） ◀─── roll_ptr                  │   │
│  │  name='李四', age=25                                  │   │
│  │  roll_ptr=──┐                                          │   │
│  └─────────────────────────────────────────────────────────┘   │
│                   │                                              │
│                   │                                              │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  undo log（trx_id=10）                                 │   │
│  │  name='张三', age=25                                    │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  遍历逻辑：                                                      │
│  1. 从最新记录开始                                              │
│  2. 检查当前事务是否能看见这版本（通过 ReadView）                │
│  3. 如果不能看见，通过 roll_ptr 找上一个版本                    │
│  4. 重复步骤 2-3，直到找到可见版本或链尾                        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## ReadView 机制

### 什么是 ReadView？

ReadView（读视图）是 MVCC 的核心概念。它记录了**当前时刻活跃事务的状态**，用于判断数据的哪个版本对当前事务可见。

```
┌─────────────────────────────────────────────────────────────────┐
│                    ReadView 结构                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ReadView {                                                     │
│      creator_trx_id: 当前事务的 ID                               │
│      m_ids: 活跃事务 ID 列表（未提交的事务）                     │
│      min_trx_id: 最小活跃事务 ID（m_ids 中的最小值）             │
│      max_trx_id: 创建 ReadView 时最大事务 ID + 1                 │
│  }                                                              │
│                                                                 │
│  可见性判断规则：                                                │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  1. 如果记录的 trx_id == creator_trx_id：可见（自己的）  │   │
│  │  2. 如果记录的 trx_id < min_trx_id：可见（已提交）       │   │
│  │  3. 如果记录的 trx_id >= max_trx_id：不可见（未来事务）  │   │
│  │  4. 如果记录的 trx_id 在 m_ids 中：不可见（未提交）       │   │
│  │  5. 否则：不可见，通过 undo log 版本链查找可见版本        │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### ReadView 示例

```
┌─────────────────────────────────────────────────────────────────┐
│                    ReadView 可见性判断示例                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  假设当前 ReadView：                                            │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  creator_trx_id = 50                                     │   │
│  │  m_ids = [20, 30, 40]  // 活跃事务                      │   │
│  │  min_trx_id = 20                                         │   │
│  │  max_trx_id = 60                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  判断记录可见性：                                                │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  trx_id = 10  →  < min_trx_id(20) → 可见（已提交）      │   │
│  │  trx_id = 25  →  在 m_ids 中       → 不可见（未提交）   │   │
│  │  trx_id = 45  →  在 m_ids 中       → 不可见（未提交）   │   │
│  │  trx_id = 50  →  == creator_trx_id → 可见（自己）       │   │
│  │  trx_id = 70  →  >= max_trx_id    → 不可见（未来事务）  │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### ReadView 生成时机

**这是 RC 和 RR 级别的核心区别！**

```
┌─────────────────────────────────────────────────────────────────┐
│                    ReadView 生成时机对比                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  读已提交（RC）：                                               │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  事务 A：                                              │   │
│  │  BEGIN;                                                 │   │
│  │  SELECT ... ────→ ReadView_1 ──→ 读取数据               │   │
│  │  SELECT ... ────→ ReadView_2 ──→ 读取数据（可能变化）    │   │
│  │  SELECT ... ────→ ReadView_3 ──→ 读取数据（可能变化）    │   │
│  │                                                         │   │
│  │  每次 SELECT 都会生成新的 ReadView！                    │   │
│  │  可能看到其他事务提交的新数据                            │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  可重复读（RR）：                                               │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  事务 A：                                              │   │
│  │  BEGIN;  ────→ ReadView 生成！                          │   │
│  │  SELECT ... ────→ ReadView ──→ 读取数据                  │   │
│  │  SELECT ... ────→ ReadView ──→ 读取数据（不变）          │   │
│  │  SELECT ... ────→ ReadView ──→ 读取数据（不变）          │   │
│  │                                                         │   │
│  │  事务开始时生成一个 ReadView，整个事务复用！              │   │
│  │  看不到事务开始后其他事务提交的数据                        │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## MVCC 的完整工作流程

### 读操作流程

```
┌─────────────────────────────────────────────────────────────────┐
│                    MVCC 读操作流程                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  事务 A 执行 SELECT：                                            │
│                                                                 │
│  Step 1: 生成 ReadView                                          │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  ReadView {                                              │   │
│  │    creator_trx_id = 50,                                  │   │
│  │    m_ids = [20, 40],                                     │   │
│  │    min_trx_id = 20,                                      │   │
│  │    max_trx_id = 60                                       │   │
│  │  }                                                       │   │
│  └─────────────────────────────────────────────────────────┘   │
│                          │                                      │
│                          ▼                                      │
│  Step 2: 读取当前记录                                            │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  记录：name='李四', age=30, trx_id=30                   │   │
│  │                                                         │   │
│  │  判断：                                                  │   │
│  │  - trx_id=30 < min_trx_id=20?  否                       │   │
│  │  - trx_id=30 >= max_trx_id=60? 否                       │   │
│  │  - trx_id=30 在 m_ids=[20,40]? 否                       │   │
│  │  - trx_id=30 != creator_trx_id=50? 是                  │   │
│  │                                                         │   │
│  │  → 需要通过版本链查找可见版本                            │   │
│  └─────────────────────────────────────────────────────────┘   │
│                          │                                      │
│                          ▼                                      │
│  Step 3: 遍历版本链                                            │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  undo log v2: name='李四', age=25, trx_id=20            │   │
│  │  判断：trx_id=20 在 m_ids=[20,40]? 是！→ 不可见        │   │
│  │                                                         │   │
│  │  undo log v1: name='张三', age=25, trx_id=10            │   │
│  │  判断：trx_id=10 < min_trx_id=20? 是！→ 可见          │   │
│  │                                                         │   │
│  │  → 返回 name='张三', age=25                            │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### MVCC 与锁的关系

```
┌─────────────────────────────────────────────────────────────────┐
│                    MVCC 与锁的关系                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                    快照读（SELECT）                      │   │
│  │                                                         │   │
│  │  - 不加锁                                                │   │
│  │  - 读取历史版本（ReadView）                              │   │
│  │  - 可以并发执行                                           │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                    当前读（SELECT ... FOR UPDATE）       │   │
│  │                                                         │   │
│  │  - 加锁（排他锁）                                         │   │
│  │  - 读取最新版本                                           │   │
│  │  - 与其他锁冲突                                          │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                    写操作（INSERT/UPDATE/DELETE）        │   │
│  │                                                         │   │
│  │  - 加排他锁                                              │   │
│  │  - 创建新版本                                            │   │
│  │  - 写入 undo log                                        │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  MVCC 让「快照读」不加锁，实现读写不冲突                          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## MVCC 与隔离级别的关系

### 不同隔离级别下的 MVCC 行为

```
┌─────────────────────────────────────────────────────────────────┐
│                    隔离级别与 MVCC 行为                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  读未提交（RU）                                          │   │
│  │                                                         │   │
│  │  - 不使用 MVCC                                           │   │
│  │  - 直接读取最新版本（包括未提交的）                       │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  读已提交（RC）                                          │   │
│  │                                                         │   │
│  │  - 使用 MVCC                                             │   │
│  │  - 每次 SELECT 生成新 ReadView                           │   │
│  │  - 读取已提交的最新版本                                  │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  可重复读（RR）★                                        │   │
│  │                                                         │   │
│  │  - 使用 MVCC                                             │   │
│  │  - 事务开始时生成 ReadView                               │   │
│  │  - 整个事务复用                                          │   │
│  │  - 配合临键锁防止幻读                                    │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  串行化（SERIALIZABLE）                                  │   │
│  │                                                         │   │
│  │  - 不使用 MVCC（快照读）                                  │   │
│  │  - 读操作也加锁                                          │   │
│  │  - 强制串行执行                                          │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### RC vs RR 示例对比

```sql
-- 初始数据：users 表，id=1, name='张三'

-- 事务 A（可重复读）
BEGIN;
SELECT * FROM users WHERE id = 1;
-- 结果：name='张三'

-- 事务 B
BEGIN;
UPDATE users SET name='李四' WHERE id = 1;
COMMIT;

-- 事务 A 再次查询
SELECT * FROM users WHERE id = 1;
-- 可重复读：name='张三'（不变！）
-- 读已提交：name='李四'（看到新数据！）
```

## MVCC 的限制

### MVCC 不能解决所有问题

```
┌─────────────────────────────────────────────────────────────────┐
│                    MVCC 的局限性                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  MVCC 能解决：                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  ✓ 读写不冲突，提高并发性能                               │   │
│  │  ✓ 脏读（在 RC 及以上）                                  │   │
│  │  ✓ 不可重复读（在 RR 下）                                │   │
│  │  ✓ 部分幻读（通过临键锁）                                 │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  MVCC 不能解决：                                                │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  ✗ 脏写（写写冲突）- 必须用锁                           │   │
│  │  ✗ 当前读与快照读的冲突 - 需要锁隔离                    │   │
│  │  ✗ 彻底解决幻读 - 某些场景下仍需锁                      │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 何时会读到最新数据？

即使在 RR 级别下，以下情况仍会读取最新数据：

```sql
-- 1. 使用当前读（FOR UPDATE）
SELECT * FROM users WHERE id = 1 FOR UPDATE;
-- 读取最新版本，加锁

-- 2. 使用排他锁
SELECT * FROM users WHERE id = 1 LOCK IN SHARE MODE;
-- 加共享锁，读取最新版本

-- 3. 修改数据
UPDATE users SET name = '王五' WHERE id = 1;
-- 自动加锁，读取最新版本

-- 4. 读取自己的修改
BEGIN;
UPDATE users SET name = '李四' WHERE id = 1;
SELECT * FROM users WHERE id = 1;  -- 看到自己的修改
```

## 面试实战

### 问题一：什么是 MVCC？它的核心思想是什么？

**面试官问**：「什么是 MVCC？它的核心思想是什么？」

**深入回答**：

> MVCC（Multi-Version Concurrency Control，多版本并发控制）是一种并发控制机制。
>
> **核心思想**：
> - 为每行数据维护多个版本
> - 读操作读取历史快照，不阻塞
> - 写操作创建新版本，不阻塞读
> - 通过 ReadView 判断哪个版本对当前事务可见
>
> **解决的问题**：
> - 读写不冲突，提高并发性能
> - 在不同隔离级别下实现不同的可见性

### 问题二：ReadView 是什么？它包含哪些字段？

**面试官问**：「ReadView 是什么？它包含哪些字段？是如何判断可见性的？」

**深入回答**：

> ReadView（读视图）是 MVCC 的核心，记录了当前时刻活跃事务的状态。
>
> **包含的字段**：
> ```sql
> ReadView {
>     creator_trx_id  -- 当前事务的 ID
>     m_ids           -- 活跃事务 ID 列表（未提交）
>     min_trx_id      -- 最小活跃事务 ID
>     max_trx_id      -- 最大事务 ID + 1
> }
> ```
>
> **可见性判断**：
> 1. `trx_id == creator_trx_id` → 可见（自己的）
> 2. `trx_id < min_trx_id` → 可见（已提交的）
> 3. `trx_id >= max_trx_id` → 不可见（未来的）
> 4. `trx_id in m_ids` → 不可见（未提交的）
> 5. 其他 → 不可见，通过 undo log 链查找

### 问题三：RC 和 RR 级别下 MVCC 的区别是什么？

**面试官问**：「MVCC 在读已提交和可重复读级别下有什么区别？」

**深入回答**：

> **核心区别：ReadView 生成时机不同**
>
> | 隔离级别 | ReadView 生成时机 | 效果 |
> |---|---|---|
> | 读已提交 | **每次 SELECT** 都生成 | 可能看到其他事务提交的新数据 |
> | 可重复读 | **事务开始时** 生成一次 | 整个事务看到的数据一致 |
>
> **举例**：
> ```sql
> -- 事务 A（RR）
> BEGIN;
> SELECT * FROM users;  -- ReadView_1
> -- 其他事务提交
> SELECT * FROM users;  -- 复用 ReadView_1，结果不变
>
> -- 事务 B（RC）
> BEGIN;
> SELECT * FROM users;  -- ReadView_1
> -- 其他事务提交
> SELECT * FROM users;  -- ReadView_2，可能看到新数据
> ```

### 问题四：undo log 版本链是如何形成的？

**面试官问**：「undo log 版本链是如何形成的？如何遍历？」

**深入回答**：

> **版本链形成**：
> - 每条记录有 `DB_TRX_ID`（事务ID）和 `DB_ROLL_PTR`（回滚指针）
> - 修改数据时，旧值写入 undo log，`roll_ptr` 指向它
> - 每次修改，形成一条 undo log 链
>
> **遍历过程**：
> 1. 从最新记录开始
> 2. 检查当前事务是否可见（通过 ReadView）
> 3. 如果不可见，通过 `roll_ptr` 找上一个版本
> 4. 重复 2-3，直到找到可见版本或链尾
>
> ```sql
> -- 示例
> 当前记录：trx_id=30, roll_ptr → undo_log_v2
> undo_log_v2：trx_id=20, roll_ptr → undo_log_v1
> undo_log_v1：trx_id=10, roll_ptr = NULL
> ```

## 总结

MVCC 是 MySQL 并发控制的核心机制：

- **版本链**：`DB_ROLL_PTR` 链接 undo log，形成多版本链
- **ReadView**：记录活跃事务状态，用于判断可见性
- **可见性判断**：基于 trx_id 和 ReadView 字段
- **RC vs RR**：ReadView 生成时机不同（每次 SELECT vs 事务开始）
- **MVCC + 锁**：快照读用 MVCC，当前读用锁

理解 MVCC 原理，是掌握 MySQL 事务隔离的必经之路。
