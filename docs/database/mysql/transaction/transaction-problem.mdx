# 并发事务问题：脏读、不可重复读、幻读

「什么是脏读？为什么同一事务中两次查询结果不一样？」

这两个问题的答案，都藏在并发事务的三大问题里：脏读、不可重复读、幻读。理解这些问题，是掌握事务隔离级别的基础。

## 三个经典的并发问题

并发执行事务时，由于隔离性不够，会产生三个经典问题：

## 脏读（Dirty Read）

### 什么是脏读？

脏读是指读取到其他事务未提交的数据。这些数据可能是「脏」的——其他事务可能还会回滚。

### 脏读示例

```
时间线：

T1 时刻：事务 A 开始
T2 时刻：事务 A 查询用户余额 = 1000
T3 时刻：事务 A 执行 UPDATE balance = 500（未提交）
T4 时刻：事务 B 查询用户余额 = 500（脏读！）← 读取到未提交的数据
T5 时刻：事务 A 执行 ROLLBACK（回滚）
T6 时刻：事务 B 再次查询用户余额 = 1000（和刚才不一样了！）

实际数据：1000（事务 A 回滚了）
```

```sql
-- 事务 A
BEGIN;
UPDATE accounts SET balance = 500 WHERE user_id = 1;
-- 此时 balance = 500，但未提交

-- 事务 B（在事务 A 提交前查询）
BEGIN;
SELECT balance FROM accounts WHERE user_id = 1;
-- 结果：500 ← 脏读！读取到事务 A 未提交的数据

-- 事务 A
ROLLBACK;  -- 回滚，balance 恢复为 1000

-- 事务 B
SELECT balance FROM accounts WHERE user_id = 1;
-- 结果：1000 ← 和刚才不一样！
```

### 脏读的危害

```
┌─────────────────────────────────────────────────────────────────┐
│                    脏读的严重后果                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  脏读的危害：                                                    │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  场景：银行转账                                           │   │
│  │                                                         │   │
│  │  事务 A：UPDATE accounts SET balance = balance - 1000  │   │
│  │  事务 B：SELECT balance → 看到了 1000（实际已扣）       │   │
│  │  事务 B：用这个「脏数据」做了业务判断                    │   │
│  │  事务 A：ROLLBACK（扣款失败，回滚）                       │   │
│  │                                                         │   │
│  │  结果：事务 B 基于脏数据做了错误决策                      │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  脏读是不可接受的，必须杜绝！                                     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 脏读的发生条件

脏读只发生在 **读未提交（Read Uncommitted）** 隔离级别下：

```sql
-- 设置为读未提交（允许脏读）
SET SESSION TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;
```

| 隔离级别 | 脏读 |
|---|---|
| 读未提交（RU） | 可能发生 |
| 读已提交（RC） | 不可能 |
| 可重复读（RR） | 不可能 |
| 串行化 | 不可能 |

## 不可重复读（Non-repeatable Read）

### 什么是不可重复读？

不可重复读是指在同一事务中，两次读取同一行数据，结果不一样。这是因为其他事务修改了这行数据并提交了。

### 不可重复读示例

```
时间线：

T1 时刻：事务 A 查询订单状态 = '待支付'
T2 时刻：事务 B 修改订单状态 = '已支付' 并提交
T3 时刻：事务 A 再次查询订单状态 = '已支付'（不一致！）
         同一事务中两次读取，结果不同
```

```sql
-- 事务 A
BEGIN;

-- 第一次读取
SELECT status FROM orders WHERE order_id = 100;
-- 结果：待支付

-- 事务 B（在事务 A 执行期间）
BEGIN;
UPDATE orders SET status = '已支付' WHERE order_id = 100;
COMMIT;

-- 事务 A 第二次读取
SELECT status FROM orders WHERE order_id = 100;
-- 结果：已支付 ← 同一事务中两次读取，结果不同！

COMMIT;
```

### 不可重复读 vs 脏读

```
┌─────────────────────────────────────────────────────────────────┐
│                    不可重复读与脏读的区别                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  脏读：读取其他事务未提交的数据                          │   │
│  │                                                         │   │
│  │  T1: 事务 A 修改数据（未提交）                          │   │
│  │  T2: 事务 B 读取数据（读取到未提交的值）← 脏读         │   │
│  │  T3: 事务 A ROLLBACK                                    │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  不可重复读：读取其他事务已提交的数据                    │   │
│  │                                                         │   │
│  │  T1: 事务 A 读取数据 = 100                               │   │
│  │  T2: 事务 B 修改数据 = 200 并提交                        │   │
│  │  T3: 事务 A 再次读取数据 = 200 ← 不一致！               │   │
│  │      （但这不是脏读，因为 B 已经提交了）                 │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  关键区别：脏读 读取未提交，不可重复读 读取已提交但被修改的数据   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 不可重复读的发生条件

不可重复读发生在 **读已提交（Read Committed）** 及以下隔离级别：

```sql
-- 设置为读已提交
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
```

| 隔离级别 | 不可重复读 |
|---|---|
| 读未提交（RU） | 可能发生 |
| 读已提交（RC） | 可能发生 |
| 可重复读（RR） | 不可能 |
| 串行化 | 不可能 |

## 幻读（Phantom Read）

### 什么是幻读？

幻读是指在同一事务中，两次执行相同的查询，结果集不一样——多了或少了行，就像「幻觉」一样。

### 幻读与不可重复读的区别

```
┌─────────────────────────────────────────────────────────────────┐
│                    幻读与不可重复读的区别                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  不可重复读：针对同一行数据的修改                        │   │
│  │                                                         │   │
│  │  T1: SELECT * FROM users WHERE id = 1;  -- 查到 1 行   │   │
│  │  T2: UPDATE users SET name = '新名字' WHERE id = 1;    │   │
│  │  T3: SELECT * FROM users WHERE id = 1;  -- 同一行变了 │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  幻读：针对结果集的增减                                  │   │
│  │                                                         │   │
│  │  T1: SELECT * FROM users WHERE age = 25;  -- 查到 3 行  │   │
│  │  T2: INSERT INTO users VALUES (4, '新用户', 25);       │   │
│  │  T3: SELECT * FROM users WHERE age = 25;  -- 查到 4 行  │   │
│  │      多了 1 行！                                        │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  核心区别：                                                      │
│  - 不可重复读：同一行数据被修改                                  │
│  - 幻读：结果集新增或减少了行                                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 幻读示例

```sql
-- 事务 A：统计年龄为 25 岁的用户数量
BEGIN;

-- 第一次统计
SELECT COUNT(*) FROM users WHERE age = 25;
-- 结果：3 人

-- 事务 B：在事务 A 执行期间，插入一个新用户
BEGIN;
INSERT INTO users (name, age) VALUES ('新用户', 25);
COMMIT;

-- 事务 A 再次统计
SELECT COUNT(*) FROM users WHERE age = 25;
-- 结果：4 人 ← 多了 1 行，幻读！

COMMIT;
```

### 幻读的危害

```
┌─────────────────────────────────────────────────────────────────┐
│                    幻读的典型危害                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  场景：库存扣减                                                 │
│                                                                 │
│  T1: 事务 A 查询商品库存 = 10                                  │
│  T2: 事务 B 新增一个订单（INSERT），商品库存不变               │
│  T3: 事务 A 查询库存仍 = 10（没问题）                          │
│  T4: 事务 A 扣减库存 = 10 - 1 = 9，更新库存                    │
│  T5: 事务 A 提交                                                │
│                                                                 │
│  问题：                                                        │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  事务 A 两次查询的库存都是 10                            │   │
│  │  但实际上事务 B 新增了订单，如果库存被多个订单抢购，     │   │
│  │  可能导致超卖！                                          │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  幻读会导致「看起来有库存，实际扣不了」的问题                     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 幻读的发生条件

幻读发生在 **读已提交（Read Committed）** 及以下隔离级别。可重复读级别下，InnoDB 通过临键锁（Next-Key Lock）防止幻读。

```sql
-- 设置为读已提交
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
```

| 隔离级别 | 幻读 |
|---|---|
| 读未提交（RU） | 可能发生 |
| 读已提交（RC） | 可能发生 |
| 可重复读（RR） | 可能发生（但 InnoDB 用临键锁防止） |
| 串行化 | 不可能 |

## 三种问题的对比总结

### 问题对比表

| 问题 | 描述 | 读取的数据 | 隔离级别 |
|---|---|---|---|
| 脏读 | 读取其他事务未提交的数据 | 未提交的数据 |
| 不可重复读 | 同一事务中两次读取同一行，结果不同 | 其他事务已提交且修改的数据 |
| 幻读 | 同一事务中两次查询，结果集不同 | 其他事务已提交且新增/删除的数据 |

## InnoDB 如何解决这些问题？

### 锁机制

InnoDB 通过锁机制解决并发问题：

| 锁类型 | 解决的问题 |
|---|---|
| 排他锁（X锁） | 防止脏写 |
| 共享锁（S锁） | 防止脏读 |
| 临键锁（Next-Key Lock） | 防止幻读 |

### MVCC 机制

InnoDB 通过 MVCC（多版本并发控制）解决读写并发问题：

```
┌─────────────────────────────────────────────────────────────────┐
│                    MVCC 工作原理                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  每个事务在启动时生成一个 ReadView：                              │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  ReadView {                                            │   │
│  │    creator_trx_id: 当前事务 ID                        │   │
│  │    m_ids: 活跃事务 ID 列表                             │   │
│  │    min_trx_id: 最小活跃事务 ID                         │   │
│  │    max_trx_id: 创建 ReadView 时的最大事务 ID            │   │
│  │  }                                                     │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  读取数据时：                                                    │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  如果记录的 trx_id < min_trx_id：可见                   │   │
│  │  如果记录的 trx_id 在 m_ids 中：不可见                  │   │
│  │  其他情况：通过 undo log 版本链查找可见版本              │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 面试实战

### 问题一：脏读、不可重复读、幻读的区别？

**面试官问**：「脏读、不可重复读、幻读有什么区别？」

**深入回答**：

> 三个问题的核心区别在于读取的数据状态：
>
> | 问题 | 读取的数据 | 严重程度 |
> |---|---|---|
> | 脏读 | 其他事务**未提交**的数据 | 最严重 |
> | 不可重复读 | 其他事务**已提交且修改**的数据 | 严重 |
> | 幻读 | 其他事务**已提交且新增/删除**的数据 | 中等 |
>
> 简单记忆：
> - 脏读：读到了「脏」数据（未提交的，可能回滚的）
> - 不可重复读：同一行数据「变」了
> - 幻读：结果集「多」或少了几行

### 问题二：MySQL 默认隔离级别是什么？如何解决幻读？

**面试官问**：「MySQL 默认的隔离级别是什么？InnoDB 是如何解决幻读的？」

**深入回答**：

> MySQL 默认隔离级别是 **可重复读（Repeatable Read）**。
>
> InnoDB 通过 **临键锁（Next-Key Lock）** 解决幻读：
>
> 临键锁 = 记录锁 + 间隙锁
> - 记录锁：锁定已存在的行
> - 间隙锁：锁定行之间的空隙
>
> 示例：
> ```sql
> SELECT * FROM users WHERE age = 25 FOR UPDATE;
> -- 会锁定 age = 25 的记录，以及 (上一个值, 25] 和 (25, 下一个值) 的间隙
> -- 其他事务无法在这个范围内插入新数据
> ```
>
> 这样就防止了幻读——其他事务无法在锁定的范围内新增数据。

### 问题三：什么场景下需要设置读未提交？

**面试官问**：「既然脏读这么严重，为什么还有读未提交这个隔离级别？」

**深入回答**：

> 读未提交虽然存在脏读风险，但在某些场景下可以接受：
>
> 1. **统计分析场景**：只关心统计结果，不关心单条数据的准确性
> 2. **数据同步场景**：比如将数据同步到只读备库，可以容忍短暂不一致
> 3. **性能优先场景**：对一致性要求不高，但需要高性能
>
> 但在金融、转账等关键业务场景，**绝对不能使用读未提交**。

## 总结

并发事务的三大问题是数据库面试的经典考点：

- **脏读**：读取其他事务未提交的数据，最严重
- **不可重复读**：同一行数据被修改
- **幻读**：结果集新增或减少行

理解这三种问题的区别，以及不同隔离级别对它们的防止程度，是掌握事务隔离级别的基础。
