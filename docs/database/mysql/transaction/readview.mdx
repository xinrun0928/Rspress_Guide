# ReadView 生成时机与隐含条件

「ReadView 是什么时候生成的？为什么 RR 级别下只生成一次？」

ReadView 是 MVCC 的核心，理解它的生成时机和隐含条件，是深入掌握事务隔离的关键。

## ReadView 是什么？

在深入讲解生成时机之前，先回顾 ReadView 的结构：

```
┌─────────────────────────────────────────────────────────────────┐
│                    ReadView 结构                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ReadView {                                                     │
│      creator_trx_id: 当前事务 ID                                 │
│      m_ids: 活跃事务 ID 列表（未提交的事务）                     │
│      min_trx_id: 最小活跃事务 ID                                 │
│      max_trx_id: 创建 ReadView 时的最大事务 ID + 1               │
│  }                                                              │
│                                                                 │
│  可见性判断规则：                                                │
│  1. trx_id == creator_trx_id → 可见（自己的）                    │
│  2. trx_id < min_trx_id → 可见（已提交）                         │
│  3. trx_id >= max_trx_id → 不可见（未来事务）                    │
│  4. trx_id in m_ids → 不可见（未提交）                           │
│  5. 否则 → 不可见，通过 undo log 版本链查找                      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## ReadView 生成时机

### 核心规则

```
┌─────────────────────────────────────────────────────────────────┐
│                    ReadView 生成时机                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  读未提交（RU）                                          │   │
│  │                                                         │   │
│  │  ReadView 生成时机：不用生成 ReadView！                  │   │
│  │  原因：直接读取最新数据，包括未提交的                     │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  读已提交（RC）                                          │   │
│  │                                                         │   │
│  │  ReadView 生成时机：每次 SELECT 都生成新的               │   │
│  │  原因：需要看到其他事务最新提交的数据                     │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  可重复读（RR）★ MySQL 默认                              │   │
│  │                                                         │   │
│  │  ReadView 生成时机：事务开始时生成，整个事务复用           │   │
│  │  原因：保证事务内数据一致性                               │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  串行化（SERIALIZABLE）                                  │   │
│  │                                                         │   │
│  │  ReadView 生成时机：通常不用 ReadView                    │   │
│  │  原因：强制加锁，串行执行                                 │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### RC 级别：每次 SELECT 生成新 ReadView

```
┌─────────────────────────────────────────────────────────────────┐
│                    RC 级别 ReadView 生成时机                     │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  事务 A（RC）：                                                 │
│                                                                 │
│  T1: BEGIN;                                                    │
│  T2: SELECT ... ────────────→ ReadView_1（生成）                │
│  T3: SELECT ... ────────────→ ReadView_2（生成）                │
│  T4: SELECT ... ────────────→ ReadView_3（生成）                │
│                                                                 │
│  每次 SELECT 都生成新的 ReadView！                               │
│  m_ids 反映当前时刻的活跃事务                                    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### RR 级别：事务开始时生成一次

```
┌─────────────────────────────────────────────────────────────────┐
│                    RR 级别 ReadView 生成时机                      │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  事务 A（RR）：                                                 │
│                                                                 │
│  T1: BEGIN; ─────────────────→ ReadView（生成，只生成一次！）    │
│  T2: SELECT ... ──────────────→ ReadView（复用）                 │
│  T3: SELECT ... ──────────────→ ReadView（复用）                 │
│  T4: SELECT ... ──────────────→ ReadView（复用）                 │
│  T5: COMMIT;                           事务结束时释放             │
│                                                                 │
│  事务开始时生成 ReadView，整个事务期间复用！                      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 生成时机的详细对比

### 时间线对比

```
┌─────────────────────────────────────────────────────────────────┐
│                    RC vs RR ReadView 生成对比                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  时间轴：T1 ── T2 ── T3 ── T4 ── T5 ── T6                      │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  RC 级别：                                              │   │
│  │                                                         │   │
│  │  T1: BEGIN ────→ ReadView_1                            │   │
│  │  T2: SELECT ────→ ReadView_2 ← m_ids 更新              │   │
│  │  T3: （其他事务提交）                                   │   │
│  │  T4: SELECT ────→ ReadView_3 ← m_ids 更新              │   │
│  │  T5: SELECT ────→ ReadView_4 ← m_ids 更新              │   │
│  │                                                         │   │
│  │  每次 SELECT 都重新生成，m_ids 反映最新状态              │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  RR 级别：                                              │   │
│  │                                                         │   │
│  │  T1: BEGIN ────→ ReadView ← 只生成一次！                │   │
│  │  T2: SELECT ────→ ReadView（复用）                      │   │
│  │  T3: （其他事务提交）                                   │   │
│  │  T4: SELECT ────→ ReadView（复用，忽略新提交）          │   │
│  │  T5: SELECT ────→ ReadView（复用，忽略新提交）          │   │
│  │                                                         │   │
│  │  ReadView 是事务开始的快照，忽略后续提交                 │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 对数据可见性的影响

```sql
-- 初始数据：balance = 1000

-- 会话 A：RR 级别
BEGIN;

-- 会话 B：
BEGIN;
UPDATE accounts SET balance = 500 WHERE id = 1;
COMMIT;

-- 会话 A：
SELECT balance FROM accounts WHERE id = 1;
-- RR 级别：1000（看不到 B 的修改，因为 ReadView 在 B 提交前生成）
-- RC 级别：500（可以看到 B 的修改）
```

## ReadView 的隐含条件

### 隐含条件一：事务 ID 分配时机

```
┌─────────────────────────────────────────────────────────────────┐
│                    事务 ID 分配时机                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  事务 ID 不是在 BEGIN 时分配的，而是在第一次修改数据时分配：       │
│                                                                 │
│  BEGIN;  ────→ 事务 ID 未分配                                  │
│  SELECT ...;  ────→ 事务 ID 未分配（快照读，不修改数据）         │
│  UPDATE ...;  ────→ 事务 ID 分配！开始写作                     │
│                                                                 │
│  影响：                                                         │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  BEGIN;                                                  │   │
│  │  SELECT ...;  -- ReadView 生成，creator_trx_id 待定       │   │
│  │  -- 此时还没有真正的事务 ID                               │   │
│  │  UPDATE ...;  -- 事务 ID 分配，trx_id = 50               │   │
│  │  SELECT ...;  -- creator_trx_id = 50                     │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 隐含条件二：ReadView 与活跃事务列表

```
┌─────────────────────────────────────────────────────────────────┐
│                    活跃事务列表的影响                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  m_ids 包含的是：                                               │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  - 已启动但未提交的事务                                  │   │
│  │  - 包括当前事务（如果有修改操作）                        │   │
│  │  - 不包括已提交或已回滚的事务                           │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  示例：                                                         │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  事务 A (ID=50) 已提交                                   │   │
│  │  事务 B (ID=51) 未提交                                   │   │
│  │  事务 C (ID=52) 未提交                                   │   │
│  │                                                         │   │
│  │  当前 ReadView:                                         │   │
│  │  - m_ids = [51, 52]  ← 不包含已提交的 A                 │   │
│  │  - min_trx_id = 51                                      │   │
│  │  - max_trx_id = 53                                      │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 隐含条件三：ReadView 与版本链遍历

```
┌─────────────────────────────────────────────────────────────────┐
│                    ReadView 与版本链遍历                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  当 ReadView 判断当前版本不可见时，会沿着 roll_ptr 遍历版本链：   │
│                                                                 │
│  当前记录                                                       │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  trx_id = 60                                            │   │
│  │  roll_ptr → undo_log_v2                                 │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  遍历过程：                                                      │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  1. 检查当前版本 trx_id=60:                               │   │
│  │     - >= max_trx_id → 不可见                            │   │
│  │                                                         │   │
│  │  2. 通过 roll_ptr 找到 undo_log_v2:                     │   │
│  │     - trx_id = 55                                       │   │
│  │     - >= max_trx_id → 不可见                            │   │
│  │                                                         │   │
│  │  3. 通过 roll_ptr 找到 undo_log_v1:                     │   │
│  │     - trx_id = 30                                       │   │
│  │     - < min_trx_id → 可见！返回这个版本                  │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 特殊场景

### 场景一：事务 A 修改自己的数据

```sql
-- 事务 A（RR）
BEGIN;

-- 事务 A 修改数据
UPDATE accounts SET balance = 500 WHERE id = 1;
-- trx_id = 50，创建 undo log

-- 事务 A 读取自己的修改
SELECT * FROM accounts WHERE id = 1;
-- 可见！因为 trx_id=50 == creator_trx_id=50
```

### 场景二：事务 A 修改后，事务 B 再修改

```sql
-- 事务 A（RR）
BEGIN;
UPDATE accounts SET balance = 500 WHERE id = 1;
-- trx_id = 50

-- 事务 B（RR）
BEGIN;
UPDATE accounts SET balance = 600 WHERE id = 1;
-- trx_id = 51，创建 undo log（记录 trx_id=50 时的值）

-- 事务 A 读取
SELECT * FROM accounts WHERE id = 1;
-- 看到 balance = 500（自己的修改）
```

### 场景三：事务 A 回滚后读取

```sql
-- 事务 A（RR）
BEGIN;
UPDATE accounts SET balance = 500 WHERE id = 1;
-- 创建 undo log（记录旧值）

-- 事务 A 回滚
ROLLBACK;
-- 通过 undo log 恢复到 balance = 1000

-- 事务 B 读取
BEGIN;
SELECT * FROM accounts WHERE id = 1;
-- 看到 balance = 1000（看不到事务 A 的任何修改）
```

## ReadView 与其他机制的关系

### ReadView 与 purge

```
┌─────────────────────────────────────────────────────────────────┐
│                    ReadView 与 undo log 清理                      │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  InnoDB 后台有 purge 线程负责清理无用的 undo log：                │
│                                                                 │
│  清理条件：                                                      │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  1. 没有活跃事务引用该 undo log                         │   │
│  │  2. 该 undo log 产生的回滚段没有被使用                  │   │
│  │  3. 没有更早的 ReadView 需要它                           │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  影响：                                                          │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  - RR 级别下，长事务会导致 undo log 堆积                  │   │
│  │  - 事务开始越早，ReadView 越旧，undo log 保留越久        │   │
│  │  - 可能导致数据库膨胀                                    │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### ReadView 与事务ID

```
┌─────────────────────────────────────────────────────────────────┐
│                    ReadView 与事务ID                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  事务 ID 的分配：                                                │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  - 全局递增                                            │   │
│  │  - 每次修改数据时分配                                   │   │
│  │  - ReadView 中的 max_trx_id 是当时最大的事务 ID + 1     │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  与 ReadView 的关系：                                            │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  - trx_id < min_trx_id: 该事务在 ReadView 生成前已提交  │   │
│  │  - min_trx_id <= trx_id < max_trx_id: 需要查 m_ids    │   │
│  │  - trx_id >= max_trx_id: 该事务在 ReadView 生成后开始  │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 面试实战

### 问题一：ReadView 在什么时机生成？

**面试官问**：「ReadView 在什么时机生成？RC 和 RR 有什么区别？」

**深入回答**：

> **ReadView 生成时机取决于隔离级别**：
>
> | 隔离级别 | 生成时机 | 影响 |
> |---|---|---|
> | 读未提交 | 不生成 | 直接读最新，包括未提交的 |
> | 读已提交 | **每次 SELECT** | 可能看到其他事务新提交的数据 |
> | 可重复读 | **事务开始时** | 整个事务复用，看不到后续提交的 |
> | 串行化 | 不使用 ReadView | 强制加锁 |
>
> **RR 级别下 ReadView 只在事务开始时生成**，这是保证可重复读的关键。

### 问题二：ReadView 的 m_ids 是什么时候确定的？

**面试官问**：「ReadView 中的 m_ids（活跃事务列表）是什么时候确定的？」

**深入回答**：

> **m_ids 在 ReadView 生成时确定**：
>
> 1. **RR 级别**：事务开始时，扫描当前所有活跃事务，生成快照
> 2. **RC 级别**：每次 SELECT 时，重新扫描当前所有活跃事务
>
> **重要**：ReadView 生成后，m_ids 是**静态的**（RR）或**准静态的**（RC），不会动态更新。
>
> **影响**：
> - RR 级别下，即使其他事务提交了，也不会加入 m_ids
> - RC 级别下，每次 SELECT 都会重新获取最新的活跃事务

### 问题三：为什么 RR 级别下看不到其他事务在事务开始后提交的数据？

**面试官问**：「可重复读级别下，为什么看不到其他事务在事务开始后提交的数据？」

**深入回答**：

> **因为 ReadView 在事务开始时生成，是当时活跃事务的快照**：
>
> ```sql
> T1: 事务 A 开始 ────→ ReadView 生成 { m_ids: [B] }
> T2: 事务 B 修改 balance = 500 ────→ B 在 m_ids 中，事务 A 看不到
> T3: 事务 B 提交 ────→ B 不在 m_ids 中了，但 ReadView 不变！
> T4: 事务 A 查询 balance ────→ 仍然看不到 B 的提交
> ```
>
> **关键**：ReadView 生成后不再更新，所以后续提交的事务对 ReadView 不可见。

## 总结

ReadView 生成时机与隔离级别密切相关：

- **RU**：不生成 ReadView，直接读最新
- **RC**：每次 SELECT 生成新 ReadView
- **RR**：事务开始时生成 ReadView，整个事务复用
- **Serializable**：不使用 ReadView

理解 ReadView 的生成时机和隐含条件，是深入理解 MVCC 和事务隔离的关键。
