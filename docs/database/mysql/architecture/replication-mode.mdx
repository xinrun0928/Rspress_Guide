# 主从复制模式：异步/半同步/全同步复制

「主从复制有几种模式？各自有什么优缺点？」

MySQL 支持三种主从复制模式，本文深入讲解它们的原理和适用场景。

## 三种复制模式概述

```
┌─────────────────────────────────────────────────────────────────┐
│                    主从复制三种模式                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  异步复制（Async Replication）                           │   │
│  │                                                         │   │
│  │  主库提交后即返回客户端，不等待从库确认                  │   │
│  │  优点：延迟低，性能好                                   │   │
│  │  缺点：主从可能不一致                                   │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  半同步复制（Semi-sync Replication）                   │   │
│  │                                                         │   │
│  │  主库等待至少一个从库确认接收 binlog 后返回            │   │
│  │  优点：数据一致性提高                                   │   │
│  │  缺点：有等待延迟                                      │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  全同步复制（Full Sync Replication）                   │   │
│  │                                                         │   │
│  │  主库等待所有从库确认接收 binlog 后返回               │   │
│  │  优点：数据一致性最强                                   │   │
│  │  缺点：延迟最大，性能最差                               │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 异步复制（Async Replication）

### 工作原理

```
┌─────────────────────────────────────────────────────────────────┐
│                    异步复制流程                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Master：                                                      │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  1. 接收 SQL                                           │   │
│  │  2. 执行 SQL                                           │   │
│  │  3. 写入 binlog                                       │   │
│  │  4. 返回客户端「成功」←──────────────────────────────┐ │   │
│  │  ↓ 异步                                               │   │
│  │  5. 通知从库                                          │   │
│  └─────────────────────────────────────────────────────────┘   │
│                              ↓                                 │
│                        ┌───────────────┐                        │
│                        │    Slave     │                        │
│                        │  接收并执行   │                        │
│                        └───────────────┘                        │
│                                                                 │
│  特点：主库不等待从库，主库和从库是异步的                      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 优点和缺点

| 优点 | 缺点 |
|---|---|
| 性能最好，延迟低 | 数据可能不一致 |
| 主库不阻塞 | 主库故障可能丢数据 |
| 配置简单 | 从库可能延迟 |

### 适用场景

```
┌─────────────────────────────────────────────────────────────────┐
│                    异步复制适用场景                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  ✓ 允许短暂数据不一致的业务                            │   │
│  │  ✓ 对延迟要求高的场景                                 │   │
│  │  ✓ 数据备份为主的场景                                 │   │
│  │  ✓ 从库主要用于读取，不要求强一致性                   │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  示例：                                                          │
│  - 报表系统：允许少量延迟                                      │
│  - 日志系统：允许短暂不一致                                    │
│  - 统计分析：允许数据略有滞后                                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 半同步复制（Semi-sync Replication）

### 工作原理

```
┌─────────────────────────────────────────────────────────────────┐
│                    半同步复制流程                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Master：                                                      │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  1. 接收 SQL                                           │   │
│  │  2. 执行 SQL                                           │   │
│  │  3. 写入 binlog                                       │   │
│  │  4. 等待至少一个从库确认接收（rpl_semi_sync_master_wait_point=AFTER_SYNC）│   │
│  │  5. 提交事务                                            │   │
│  │  6. 返回客户端「成功」                                  │   │
│  │  7. 从库接收并执行后返回确认                           │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  配置参数：                                                    │
│  - rpl_semi_sync_master_enabled = ON                         │
│  - rpl_semi_sync_slave_enabled = ON                          │
│  - rpl_semi_sync_master_timeout = 10000（毫秒）             │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 半同步复制等待点

```
┌─────────────────────────────────────────────────────────────────┐
│                    半同步复制等待点                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  AFTER_SYNC（默认）                                   │   │
│  │                                                         │   │
│  │  - 从库确认写入 relay-log 后，主库提交                │   │
│  │  - 事务提交和 binlog 写入在从库确认之前               │   │
│  │  - 主库崩溃不会丢数据                                  │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  AFTER_COMMIT                                         │   │
│  │                                                         │   │
│  │  - 主库提交后等待从库确认                             │   │
│  │  - 事务提交在从库确认之前                             │   │
│  │  - 主库崩溃可能丢数据（已提交但从库未确认）          │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 优点和缺点

| 优点 | 缺点 |
|---|---|
| 数据一致性提高 | 有等待延迟 |
| 主库崩溃不丢数据 | 性能比异步复制差 |
| 至少一个从库同步 | 从库全部故障退化为异步 |

### 适用场景

```
┌─────────────────────────────────────────────────────────────────┐
│                    半同步复制适用场景                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  ✓ 对数据一致性有要求的业务                           │   │
│  │  ✓ 不能接受异步复制可能丢数据的风险                   │   │
│  │  ✓ 愿意牺牲一定性能换取一致性                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  示例：                                                          │
│  - 金融交易系统：需要强一致性                                  │
│  - 订单系统：不允许丢订单                                      │
│  - 支付系统：必须保证数据一致性                                 │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 半同步复制配置

```sql
-- 主库安装插件
INSTALL PLUGIN rpl_semi_sync_master SONAME 'semisync_master.so';

-- 从库安装插件
INSTALL PLUGIN rpl_semi_sync_slave SONAME 'semisync_slave.so';

-- 主库启用半同步复制
SET GLOBAL rpl_semi_sync_master_enabled = ON;
SET GLOBAL rpl_semi_sync_master_timeout = 10000;

-- 从库启用半同步复制
SET GLOBAL rpl_semi_sync_slave_enabled = ON;

-- 重启从库 IO Thread
STOP SLAVE IO_THREAD;
START SLAVE IO_THREAD;

-- 查看半同步复制状态
SHOW STATUS LIKE 'rpl_semi_sync%';
```

## 全同步复制（Full Sync Replication）

### 工作原理

```
┌─────────────────────────────────────────────────────────────────┐
│                    全同步复制流程                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Master：                                                      │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  1. 接收 SQL                                           │   │
│  │  2. 执行 SQL                                           │   │
│  │  3. 写入 binlog                                       │   │
│  │  4. 等待所有从库确认接收 ← 所有从库都要确认！        │   │
│  │  5. 提交事务                                            │   │
│  │  6. 返回客户端「成功」                                  │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  特点：所有从库都同步后，主库才返回成功                        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 优点和缺点

| 优点 | 缺点 |
|---|---|
| 数据一致性最强 | 延迟最大 |
| 不丢数据 | 性能最差 |
| 所有从库都是最新的 | 一个从库慢，整体就慢 |

### 适用场景

```
┌─────────────────────────────────────────────────────────────────┐
│                    全同步复制适用场景                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  ✓ 对数据一致性要求极高的场景                         │   │
│  │  ✓ 分布式事务场景                                     │   │
│  │  ✓ 从库数量少且网络延迟低                             │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  示例：                                                          │
│  - 强一致性要求的金融系统（通常配合 Paxos/Raft）              │
│  - 跨机房数据同步                                              │
│                                                                 │
│  注意：全同步复制很少单独使用，通常配合其他技术                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## GTID 模式下的复制

### GTID 简介

GTID（Global Transaction Identifier）是 MySQL 5.6+ 引入的事务标识符。

```
┌─────────────────────────────────────────────────────────────────┐
│                    GTID 格式                                     │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  GTID = source_id:transaction_id                                │
│                                                                 │
│  示例：                                                          │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  aaaaaaaa-aaaa-aaaa-aaaa-aaaaaaaaaaaa:1-123456       │   │
│  │                                                         │   │
│  │  - aaaa...：服务器 UUID                                │   │
│  │  - 1-123456：事务序号                                 │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### GTID 复制的优势

| 优势 | 说明 |
|---|---|
| 自动定位 | 根据 GTID 自动确定从哪里开始复制 |
| 简单切换 | 主从切换时不需要指定 binlog 位置 |
| 一致性保证 | 不会遗漏或重复执行事务 |

### GTID 配置

```ini
[mysqld]
# 开启 GTID
gtid_mode = ON
enforce_gtid_consistency = ON

# 从库并行复制
slave_parallel_workers = 4
slave_parallel_type = LOGICAL_CLOCK
```

```sql
-- GTID 模式配置主从复制
CHANGE MASTER TO
    MASTER_HOST = 'master_host',
    MASTER_PORT = 3306,
    MASTER_USER = 'repl',
    MASTER_PASSWORD = 'repl_password',
    MASTER_AUTO_POSITION = 1;
```

## 面试实战

### 问题一：MySQL 有哪几种复制模式？

**面试官问**：「MySQL 主从复制有哪几种模式？各自有什么优缺点？」

**深入回答**：

> **三种复制模式**：
>
> | 模式 | 特点 | 优点 | 缺点 |
> |---|---|---|---|
> | 异步复制 | 主库不等待从库 | 性能最好 | 可能丢数据 |
> | 半同步复制 | 主库等待至少一个从库 | 数据一致性提高 | 有等待延迟 |
> | 全同步复制 | 主库等待所有从库 | 数据一致性最强 | 延迟最大 |
>
> **选择建议**：
> - 一般业务：异步复制
> - 金融等重要业务：半同步复制

### 问题二：半同步复制是怎么工作的？

**面试官问**：「半同步复制是怎么保证数据一致性的？」

**深入回答**：

> **半同步复制原理**：
>
> 1. 主库执行 SQL，写入 binlog
> 2. 主库等待至少一个从库确认接收 binlog
> 3. 从库 IO Thread 接收 binlog 写入 relay-log 后返回确认
> 4. 主库收到确认后提交事务，返回客户端成功
>
> **等待点**：
> - `AFTER_SYNC`（默认）：从库确认写入 relay-log 后提交
> - `AFTER_COMMIT`：主库提交后等待从库确认
>
> **超时机制**：如果从库超时未确认，退化为异步复制

### 问题三：什么时候需要半同步复制？

**面试官问**：「什么场景下需要使用半同步复制？」

**深入回答**：

> **适用场景**：
>
> 1. **金融交易系统**：不允许丢数据
> 2. **订单系统**：需要保证订单不丢失
> 3. **支付系统**：必须保证数据一致性
>
> **不适用场景**：
> - 网络延迟大的跨地域复制
> - 对性能要求极高的场景
> - 允许短暂不一致的统计分析

## 总结

三种复制模式各有特点：

- **异步复制**：性能最好，可能丢数据，适合大多数场景
- **半同步复制**：一致性提高，有等待延迟，适合重要业务
- **全同步复制**：一致性最强，延迟最大，很少单独使用

选择合适的复制模式，需要在性能和数据一致性之间做权衡。
