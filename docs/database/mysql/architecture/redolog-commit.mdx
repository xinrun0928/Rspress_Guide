# redo log 两阶段提交与崩溃恢复

「MySQL 崩溃后是如何恢复数据的？两阶段提交是如何保证数据一致的？」

redo log 的两阶段提交是 MySQL 崩溃恢复的核心机制。本文深入剖析两阶段提交的原理、崩溃恢复的流程以及各种故障场景的处理。

## 为什么需要两阶段提交？

## 两阶段提交详解

### 详细流程

```
┌─────────────────────────────────────────────────────────────────┐
│                    两阶段提交详细流程                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  阶段一：Prepare（准备阶段）                             │   │
│  │                                                         │   │
│  │  1. 事务执行                                            │   │
│  │     - 修改数据页（Buffer Pool）                        │   │
│  │     - 记录 undo log                                    │   │
│  │                                                         │   │
│  │  2. 写入 redo log（prepare）                          │   │
│  │     ┌─────────────────────────────────────────────┐   │   │
│  │     │  redo log 内容：                            │   │   │
│  │     │  {                                          │   │   │
│  │     │    xid: 12345,                              │   │   │
│  │     │    table: 'users',                          │   │   │
│  │     │    page: 5,                                 │   │   │
│  │     │    offset: 1024,                            │   │   │
│  │     │    old_value: '张三',                        │   │   │
│  │     │    new_value: '李四',                        │   │   │
│  │     │    status: 'PREPARE'                        │   │   │
│  │     │  }                                          │   │   │
│  │     └─────────────────────────────────────────────┘   │   │
│  │                                                         │   │
│  │  3. 刷盘 redo log（force write）                      │   │
│  │     - 必须刷到磁盘才算成功                            │   │
│  │                                                         │   │
│  │  4. 返回客户端「准备成功」                             │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                              ↓                                   │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  阶段二：Commit（提交阶段）                              │   │
│  │                                                         │   │
│  │  5. 写入 binlog                                         │   │
│  │     ┌─────────────────────────────────────────────┐   │   │
│  │     │  binlog 内容：                              │   │   │
│  │     │  #240115 12:00:00 server id=1              │   │   │
│  │     │  BEGIN 12345                               │   │   │
│  │     │  UPDATE users SET name='李四' WHERE id=1  │   │   │
│  │     │  COMMIT                                    │   │   │
│  │     └─────────────────────────────────────────────┘   │   │
│  │                                                         │   │
│  │  6. 刷盘 binlog（force write）                         │   │
│  │     - 必须刷到磁盘才算成功                            │   │
│  │                                                         │   │
│  │  7. 写入 redo log（commit）                           │   │
│  │     ┌─────────────────────────────────────────────┐   │   │
│  │     │  redo log 内容：                            │   │   │
│  │     │  {                                          │   │   │
│  │     │    xid: 12345,                              │   │   │
│  │     │    status: 'COMMIT'                         │   │   │
│  │     │  }                                          │   │   │
│  │     └─────────────────────────────────────────────┘   │   │
│  │                                                         │   │
│  │  8. 刷盘 redo log（commit）                           │   │
│  │     - 提交完成，返回客户端成功                       │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 崩溃恢复

### 崩溃恢复流程

```
┌─────────────────────────────────────────────────────────────────┐
│                    InnoDB 崩溃恢复流程                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  Step 1: 确定恢复起点                                   │   │
│  │                                                         │   │
│  │  读取 redo log，找到最后一个 checkpoint：              │   │
│  │  ┌─────────────────────────────────────────────┐   │   │
│  │  │  checkpoint: (log_file, offset) = (3, 4096) │   │   │
│  │  └─────────────────────────────────────────────┘   │   │
│  │                                                         │   │
│  │  从 checkpoint 开始扫描 redo log                     │   │
│  └─────────────────────────────────────────────────────────┘   │
│                              ↓                                   │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  Step 2: 收集需要恢复的事务                             │   │
│  │                                                         │   │
│  │  遍历 redo log，根据状态分类：                        │   │
│  │  ┌─────────────────────────────────────────────┐   │   │
│  │  │  ┌───────────┬───────────┬───────────────┐ │   │   │
│  │  │  │  状态      │ binlog    │  处理方式      │ │   │   │
│  │  │  ├───────────┼───────────┼───────────────┤ │   │   │
│  │  │  │ PREPARE   │ 有        │ 重做（提交）    │ │   │   │
│  │  │  │ PREPARE   │ 无        │ 回滚            │ │   │   │
│  │  │  │ COMMIT    │ -         │ 重做            │ │   │   │
│  │  │  │ 无记录    │ -         │ 无需处理        │ │   │   │
│  │  │  └───────────┴───────────┴───────────────┘ │   │   │
│  │  └─────────────────────────────────────────────┘   │   │
│  └─────────────────────────────────────────────────────────┘   │
│                              ↓                                   │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  Step 3: 执行恢复操作                                   │   │
│  │                                                         │   │
│  │  - 提交：重做事务的修改（Redo）                        │   │
│  │  - 回滚：执行 undo log 的反向操作（Undo）              │   │
│  └─────────────────────────────────────────────────────────┘   │
│                              ↓                                   │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  Step 4: 清理 undo log                                 │   │
│  │                                                         │   │
│  │  删除不再需要的 undo log 记录                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 故障场景分析

```
┌─────────────────────────────────────────────────────────────────┐
│                    故障场景分析                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  场景 A：redo log 刷盘成功，binlog 刷盘前崩溃          │   │
│  │                                                         │   │
│  │  时间线：                                               │   │
│  │  redo log prepare → redo log 刷盘 → ✗崩溃            │   │
│  │                                                         │   │
│  │  恢复处理：                                             │   │
│  │  - redo log 状态 = PREPARE                            │   │
│  │  - binlog 没有 xid                                     │   │
│  │  → 回滚事务                                            │   │
│  │                                                         │   │
│  │  结果：数据没有丢失（因为还没提交）                   │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  场景 B：binlog 刷盘成功，redo log commit 刷盘前崩溃   │   │
│  │                                                         │   │
│  │  时间线：                                               │   │
│  │  redo log prepare → binlog → ✗崩溃                  │   │
│  │                                                         │   │
│  │  恢复处理：                                             │   │
│  │  - redo log 状态 = PREPARE                            │   │
│  │  - binlog 有 xid                                       │   │
│  │  → 提交事务（重做）                                   │   │
│  │                                                         │   │
│  │  结果：数据已提交（主从一致）                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  场景 C：redo log commit 刷盘成功后崩溃                 │   │
│  │                                                         │   │
│  │  时间线：                                               │   │
│  │  redo log prepare → binlog → redo log commit → ✗崩溃 │   │
│  │                                                         │   │
│  │  恢复处理：                                             │   │
│  │  - redo log 状态 = COMMIT                             │   │
│  │  → 重做事务                                            │   │
│  │                                                         │   │
│  │  结果：数据已提交（幂等，无影响）                     │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 性能优化

### 组提交优化

```
┌─────────────────────────────────────────────────────────────────┐
│                    组提交（Group Commit）                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  优化前：每个事务单独刷盘                                │   │
│  │                                                         │   │
│  │  T1: redo prepare → redo commit → binlog → 返回      │   │
│  │  T2: redo prepare → redo commit → binlog → 返回      │   │
│  │  T3: redo prepare → redo commit → binlog → 返回      │   │
│  │                                                         │   │
│  │  问题：3 次 fsync，性能差                              │   │
│  └─────────────────────────────────────────────────────────┘   │
│                              ↓                                   │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  优化后：多个事务合并一次刷盘                            │   │
│  │                                                         │   │
│  │  T1: redo prepare                                     │   │
│  │  T2: redo prepare                                     │   │
│  │  T3: redo prepare                                     │   │
│  │       ↓                                                │   │
│  │  redo commit (T1+T2+T3) → binlog (T1+T2+T3) → 返回  │   │
│  │                                                         │   │
│  │  优点：只需 1 次 fsync                                 │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 相关参数配置

```ini
[mysqld]
# redo log 配置
innodb_log_file_size = 1GB              # redo log 文件大小
innodb_log_buffer_size = 64MB           # redo log buffer 大小
innodb_flush_log_at_trx_commit = 1      # 刷盘策略（最安全）

# binlog 配置
sync_binlog = 1                          # binlog 刷盘策略
binlog_format = ROW                     # binlog 格式（推荐 ROW）

# 组提交配置
innodb_flush_log_at_trx_commit = 1      # 控制 redo log 刷盘时机
```

### 参数对比

| 参数 | 值 | 说明 |
|---|---|---|
| `innodb_flush_log_at_trx_commit` | 1 | 每次提交都刷 redo log（安全） |
| `innodb_flush_log_at_trx_commit` | 2 | 提交到 OS 缓存（性能好，可能丢 1 秒数据） |
| `innodb_flush_log_at_trx_commit` | 0 | 后台定时刷盘（性能最好，最不安全） |
| `sync_binlog` | 1 | 每次提交都刷 binlog（安全） |
| `sync_binlog` | N | N 次提交后刷 binlog（性能好） |

### XA 语法

```sql
-- 开启 XA 事务
XA START 'transaction_id';

-- 执行 SQL
UPDATE accounts SET balance = balance - 100 WHERE user_id = 1;

-- 结束 XA 事务
XA END 'transaction_id';

-- Prepare
XA PREPARE 'transaction_id';

-- Commit 或 Rollback
XA COMMIT 'transaction_id';
-- 或
XA ROLLBACK 'transaction_id';

-- 查看 XA 事务状态
XA RECOVER;
```

### XA 与两阶段提交

```

├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  XA 事务的两阶段：                                      │   │
│  │                                                         │   │
│  │  Phase 1: Prepare                                       │   │
│  │  ┌─────────────────────────────────────────────────┐  │   │
│  │  │  每个 MySQL 实例：                              │  │   │
│  │  │  XA PREPARE 'xid'                               │  │   │

│  │  Phase 2: Commit/Rollback                             │   │

│  │  │  全部成功：                                      │  │   │
```

## 面试实战

### 问题一：什么是两阶段提交？为什么需要两阶段提交？

**面试官问**：「MySQL 的两阶段提交是什么？为什么需要两阶段提交？」

**深入回答**：

> **两阶段提交定义**：
>
> 1. **Prepare 阶段**：
>    - 写入 redo log（状态 = PREPARE）
>    - 刷盘 redo log
>    - 返回客户端「准备成功」
>
> 2. **Commit 阶段**：
>    - 写入 binlog
>    - 刷盘 binlog
>    - 写入 redo log（状态 = COMMIT）
>    - 刷盘 redo log
>    - 返回客户端「提交成功」
>
> **为什么需要两阶段提交**：
>
> - redo log 保证 crash-safe
> - binlog 用于主从复制
> - 两个日志必须同时成功或同时失败
> - 否则会导致主从不一致或数据丢失

### 问题二：崩溃恢复时，如何判断事务是否应该提交？

**面试官问**：「崩溃恢复时，如何判断一个 PREPARED 事务应该提交还是回滚？」

**深入回答**：

> **判断依据**：检查 binlog
>
> 1. **读取 redo log**：找到 PREPARED 状态的事务
> 2. **检查 binlog**：查看 binlog 中是否有该事务的记录
> 3. **判断规则**：
>    - binlog 有 xid → 提交（Redo）
>    - binlog 没有 xid → 回滚（Undo）
>
> **示例**：
> - 如果 binlog 已刷盘，说明事务已执行完，应该提交
> - 如果 binlog 没刷盘，说明事务还没执行完，应该回滚

### 问题三：如何优化两阶段提交的性能？

**面试官问**：「两阶段提交有什么性能问题？如何优化？」

**深入回答**：

> **性能瓶颈**：
>
> 1. **fsync 调用多**：每个事务需要多次刷盘
> 2. **网络 IO 延迟**：binlog 复制到从库
>
> **优化方案**：
>
> 1. **组提交（Group Commit）**：
>    - 多个事务合并一次 fsync
>    - 显著减少 fsync 次数
>
> 2. **并行复制**：
>    - 从库并行执行 binlog
>    - 减少主从延迟
>
> 3. **参数调优**：
>    - `innodb_flush_log_at_trx_commit = 1`（安全优先）
>    - `sync_binlog = 1`（安全优先）
>    - 评估性能后可适当调整

## 总结

两阶段提交是 MySQL 保证数据一致性的核心机制：

- **Prepare 阶段**：写 redo log（PREPARE），刷盘
- **Commit 阶段**：写 binlog，刷盘；写 redo log（COMMIT），刷盘
- **崩溃恢复**：根据 binlog 有无决定提交或回滚
- **性能优化**：组提交、并行复制

理解两阶段提交，是理解 MySQL 崩溃恢复和主从一致性的关键。
