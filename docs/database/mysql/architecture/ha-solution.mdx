# MySQL 高可用方案

「如何保证 MySQL 的高可用？」

MySQL 高可用是生产环境的核心需求。本文介绍常见的高可用方案和选型建议。

## 高可用概述

### 什么是高可用？

高可用（High Availability）是指系统在长时间运行中，减少停机时间，保证服务可用性。

```
┌─────────────────────────────────────────────────────────────────┐
│                    可用性衡量指标                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  可用性 = (总时间 - 停机时间) / 总时间 × 100%           │   │
│  │                                                         │   │
│  │  ┌─────────────────────────────────────────────────┐   │   │
│  │  │  99.9%  → 年停机时间 ≈ 8.7 小时                │   │   │
│  │  │  99.99% → 年停机时间 ≈ 52 分钟                  │   │   │
│  │  │  99.999% → 年停机时间 ≈ 5 分钟                    │   │   │
│  │  └─────────────────────────────────────────────────┘   │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 高可用衡量标准

| 可用性 | 年停机时间 | 适用场景 |
|---|---|---|
| 99.9% | ~8.7 小时 | 一般业务 |
| 99.99% | ~52 分钟 | 重要业务 |
| 99.999% | ~5 分钟 | 核心业务 |

## 常见高可用方案

### 方案一：主从复制 + 手动切换

```
┌─────────────────────────────────────────────────────────────────┐
│                    主从 + 手动切换                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │                    Master                               │   │
│  │                      ↓                                   │   │
│  │                    Slave                               │   │
│  │                                                         │   │
│  │  故障时：DBA 手动切换                                  │   │
│  │  1. 停止主库                                           │   │
│  │  2. 将从库提升为主库                                   │   │
│  │  3. 修改应用配置                                       │   │
│  │  4. 恢复时间：30 分钟 ~ 2 小时                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  优点：简单                                              │
│  缺点：恢复时间长，需要人工干预                              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 面试实战

### 问题一：MySQL 有哪些高可用方案？

**面试官问**：「MySQL 有哪些高可用方案？各自有什么特点？」

**深入回答**：

> **常见高可用方案**：
>
> | 方案 | 原理 | 优点 | 缺点 |
> |---|---|---|---|
> | 手动切换 | 主从复制 + 人工切换 | 简单 | 恢复时间长 |
> | MHA | 主从复制 + 自动监控切换 | 自动切换 | 需要 SSH |
> | MGR | 基于 Paxos 的组复制 | 强一致性 | 复杂度高 |
> | InnoDB Cluster | MGR + Router | 一站式方案 | 新技术 |
> | Keepalived | 双主 + VIP 切换 | 简单可靠 | 半自动 |

### 问题二：如何选择高可用方案？

**面试官问**：「在项目中如何选择 MySQL 高可用方案？」

**深入回答**：

> **选型考虑因素**：
>
> 1. **业务重要性**：核心业务选高可用方案
> 2. **团队技术能力**：新团队选简单方案
> 3. **预算**：高可用需要更多资源
> 4. **RTO 要求**：恢复时间要求
>
> **建议**：
> - 小型项目：Keepalived + 双主
> - 中型项目：MHA 或 Keepalived
> - 大型项目：InnoDB Cluster

### 问题三：MGR 和传统主从复制的区别？

**面试官问**：「MGR 和传统主从复制有什么区别？」

**深入回答**：

> **核心区别**：
>
> | 维度 | 传统主从复制 | MGR |
> |---|---|---|
> | 协议 | binlog + relay-log | Paxos 协议 |
> | 一致性 | 异步/半同步 | 强一致性 |
> | 故障切换 | 手动/半自动 | 自动 |
> | 复杂度 | 简单 | 高 |
> | 适用场景 | 一般业务 | 核心业务 |

## 总结

MySQL 高可用方案多样：

- **手动切换**：简单，适合测试环境
- **MHA**：成熟稳定，适合中型业务
- **MGR**：基于 Paxos，强一致性
- **InnoDB Cluster**：官方一站式方案
- **Keepalived + 双主**：简单可靠

选择合适的方案，需要综合考虑业务需求、技术能力、成本预算等因素。
