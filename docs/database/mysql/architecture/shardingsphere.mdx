# ShardingSphere 实战指南

### 分片算法配置

#### 哈希分片算法

```java
import org.apache.shardingsphere.sharding.api.sharding.standard.PreciseShardingValue;
import org.apache.shardingsphere.sharding.api.sharding.standard.RangeShardingValue;
import org.apache.shardingsphere.sharding.api.sharding.standard.StandardShardingAlgorithm;

import java.util.Collection;
import java.util.Properties;

public class HashShardingAlgorithm implements StandardShardingAlgorithm<Long> {

    private int shardingCount;

    @Override
    public void init(Properties props) {
        this.shardingCount = Integer.parseInt(
            props.getProperty("sharding-count", "4")
        );
    }

    @Override
    public String doSharding(
            Collection<String> availableTargetNames,
            PreciseShardingValue<Long> shardingValue) {

        long value = shardingValue.getValue();
        long shardIndex = value % shardingCount;

        for (String targetName : availableTargetNames) {
            if (targetName.endsWith(String.valueOf(shardIndex))) {
                return targetName;
            }
        }

        throw new UnsupportedOperationException();
    }

    @Override
    public Collection<String> doSharding(
            Collection<String> availableTargetNames,
            RangeShardingValue<Long> shardingValue) {
        // 范围查询需要扫描所有分片
        return availableTargetNames;
    }

    @Override
    public String getType() {
        return "HASH";
    }
}
```

#### 时间分片算法

```java
import org.apache.shardingsphere.sharding.api.sharding.standard.*;

import java.time.LocalDateTime;
import java.time.format.DateTimeFormatter;
import java.util.Collection;
import java.util.Properties;

public class DateShardingAlgorithm implements StandardShardingAlgorithm<LocalDateTime> {

    private DateTimeFormatter formatter;

    @Override
    public void init(Properties props) {
        this.formatter = DateTimeFormatter.ofPattern(
            props.getProperty("date-format", "yyyyMM")
        );
    }

    @Override
    public String doSharding(
            Collection<String> availableTargetNames,
            PreciseShardingValue<LocalDateTime> shardingValue) {

        LocalDateTime value = shardingValue.getValue();
        String suffix = value.format(formatter);

        for (String targetName : availableTargetNames) {
            if (targetName.endsWith(suffix)) {
                return targetName;
            }
        }

        // 默认返回最新的分片
        return availableTargetNames.stream()
                .max(String::compareTo)
                .orElseThrow();
    }

    @Override
    public Collection<String> doSharding(
            Collection<String> availableTargetNames,
            RangeShardingValue<LocalDateTime> shardingValue) {

        LocalDateTime lower = shardingValue.getValueRange().lowerEndpoint();
        LocalDateTime upper = shardingValue.getValueRange().upperEndpoint();

        // 简单实现：返回所有分片
        return availableTargetNames;
    }

    @Override
    public String getType() {
        return "DATE";
    }
}
```

### 编程式事务

```java
import org.apache.shardingsphere.transaction.core.TransactionType;
import org.apache.shardingsphere.transaction.core.TransactionTypeRegistry;
import org.apache.shardingsphere.transaction.spi.ShardingTransactionManager;

public class TransactionExample {

    public void execute() {
        // 获取事务管理器
        ShardingTransactionManager manager = 
            ShardingTransactionManagerRegistry.getTransactionManager(
                TransactionType.XA
            );

        // 开启事务
        manager.begin();

        try {
            // 执行分片SQL
            jdbcTemplate.update("INSERT INTO t_order ...", ...);

            // 提交事务
            manager.commit();
        } catch (Exception e) {
            // 回滚事务
            manager.rollback();
            throw e;
        }
    }
}
```

## 最佳实践

### 分片键设计原则

```
┌─────────────────────────────────────────────────────────────────┐
│                    分片键设计原则                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  1. 查询必需原则                                        │   │
│  │                                                         │   │
│  │  分片键必须是常见查询条件！                             │   │
│  │                                                         │   │
│  │  推荐：user_id, order_id, create_time                  │   │
│  │  不推荐：status, gender                                │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  2. 分布均匀原则                                        │   │
│  │                                                         │   │
│  │  避免数据倾斜！                                        │   │
│  │                                                         │   │
│  │  推荐：hash(user_id) % N                               │   │
│  │  不推荐：直接按主键分片（可能连续）                     │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  3. 绑定表原则                                          │   │
│  │                                                         │   │
│  │  有关联关系的表使用相同的分片键！                       │   │
│  │                                                         │   │
│  │  t_order + t_order_item → 使用相同的 order_id 分片   │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 性能优化建议

| 优化项 | 说明 |
|---|---|
| 连接池配置 | 根据业务量配置合理的连接池大小 |
| 合理分片数 | 避免分片过多导致资源浪费 |
| 避免跨分片查询 | 尽量让查询命中单个分片 |
| 使用分布式 ID | 避免使用自增主键 |
| 绑定表 | 关联查询的表使用相同的分片键 |

## 面试实战

### 问题一：ShardingSphere-JDBC 和 ShardingSphere-Proxy 有什么区别？

**面试官问**：「ShardingSphere-JDBC 和 ShardingSphere-Proxy 各有什么优缺点？」

**深入回答**：

> **ShardingSphere-JDBC**：
> - 优点：性能好，无额外网络开销
> - 缺点：代码侵入，需要引入依赖
> - 适用：Java 应用，单一技术栈
>
> **ShardingSphere-Proxy**：
> - 优点：代码无侵入，支持多语言
> - 缺点：有网络开销，需要独立部署
> - 适用：多语言环境，已有系统改造

### 问题二：ShardingSphere 如何实现分库分表？

**面试官问**：「ShardingSphere 分库分表的核心配置是什么？」

**深入回答**：

> **核心配置**：
>
> 1. **数据源配置**：配置多个物理数据源
> 2. **分片规则**：配置分片算法
> 3. **绑定表**：配置关联查询的表
> 4. **分布式主键**：配置分布式 ID 生成器
>
> **分片算法类型**：
> - INLINE：简单哈希/范围
> - MOD：取模
> - HASH_MOD：哈希取模
> - DATE：时间分片
> - 自定义：实现接口

### 问题三：ShardingSphere 如何保证分布式事务？

**面试官问**：「ShardingSphere 支持哪些分布式事务模式？」

**深入回答**：

> **支持的分布式事务模式**：
>
> | 模式 | 特点 | 一致性 |
> |---|---|---|
> | LOCAL | 无分布式事务 | 弱一致 |
> | XA | 两阶段提交 | 强一致 |
> | BASE | 最终一致 | 最终一致 |
> | Seata AT | AT 模式 | 强一致 |
>
> **选择建议**：
> - 强一致性要求：使用 XA 或 Seata
> - 性能优先：使用 BASE 或 Seata

## 总结

ShardingSphere 是分库分表的一站式解决方案：

- **ShardingSphere-JDBC**：高性能，代码侵入
- **ShardingSphere-Proxy**：无侵入，需要部署
- **分片算法**：哈希、范围、时间、自定义
- **分布式事务**：LOCAL、XA、BASE、Seata
- **最佳实践**：合理设计分片键，配置绑定表

根据业务需求和技术栈选择合适的产品和配置。
