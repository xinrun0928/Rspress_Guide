# 分库分表深入剖析：垂直拆分与水平拆分

「什么时候需要分库分表？如何拆分？」

当单表数据量达到千万级，数据库性能急剧下降时，分库分表是最后的解决方案。本文深入剖析分库分表的原理、方案和注意事项。

## 为什么需要分库分表？

## 分片键选择

### 什么是分片键？

分片键（Sharding Key）是用于数据分片的字段，决定数据如何分散到各个分片。

```
┌─────────────────────────────────────────────────────────────────┐
│                    分片键示意                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  按 user_id 分片：                                    │   │
│  │                                                         │   │
│  │  user_id % 10 = 落在哪个分片                          │   │
│  │                                                         │   │
│  │  user_id = 1  → 1 % 10 = 1 → orders_1              │   │
│  │  user_id = 15 → 15 % 10 = 5 → orders_5              │   │
│  │  user_id = 23 → 23 % 10 = 3 → orders_3              │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  分片键选择原则：                                              │
│  - 查询条件必须包含分片键                                       │
│  - 分布均匀（避免数据倾斜）                                    │
│  - 经常用于查询                                               │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 面试实战

### 问题一：什么时候需要分库分表？

**面试官问**：「什么时候需要分库分表？分库分表前应该做什么？」

**深入回答**：

> **分库分表前的优化步骤**：
>
> 1. **SQL 优化**：慢查询优化、索引优化
> 2. **缓存**：Redis 缓存热点数据
> 3. **读写分离**：主从复制分担读压力
> 4. **分库分表**：最后的手段
>
> **分库分表时机**：
> - 单表数据量超过 1000 万行
> - 查询延迟超过 1 秒
> - 数据库 CPU、IO 持续高位
> - 数据库连接耗尽

### 问题二：分库分表有哪些挑战？

**面试官问**：「分库分表有什么难点？」

**深入回答**：

> **主要挑战**：
>
> 1. **跨分片查询**：需要分散查询再合并
> 2. **分布式事务**：跨库事务难以处理
> 3. **排序分页**：全局排序需要查询所有分片
> 4. **关联查询**：跨库关联困难
> 5. **数据迁移**：历史数据迁移复杂
> 6. **扩容**：扩容需要迁移数据

### 问题三：分片键怎么选择？

**面试官问**：「分片键如何选择？有什么原则？」

**深入回答**：

> **选择原则**：
>
> 1. **查询必需**：分片键必须是查询条件
> 2. **分布均匀**：避免数据倾斜
> 3. **业务相关**：符合业务查询习惯
>
> **常见选择**：
> - 用户表：user_id
> - 订单表：user_id（按用户查）或 order_id（按订单查）
> - 日志表：create_time

## 总结

分库分表是应对大数据的终极方案：

- **垂直拆分**：按字段拆分，减少单表宽度
- **水平拆分**：按数据拆分，减少单表行数
- **分片键**：必须包含在查询条件中
- **分片算法**：哈希、范围、时间
- **挑战**：跨分片查询、分布式事务

分库分表是最后的选择，在此之前应该先尝试其他优化手段。
