# 分库分表后的跨节点查询与分布式事务

「分库分表后，跨节点查询和事务怎么处理？」

分库分表带来了跨节点查询和分布式事务的问题。本文深入讲解这些问题的解决方案。

## 分布式事务问题

### 问题场景

```
┌─────────────────────────────────────────────────────────────────┐
│                    分布式事务问题                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  转账场景：                                           │   │
│  │                                                         │   │
│  │  用户 A（分片 1）→ 用户 B（分片 2）                   │   │
│  │                                                         │   │
│  │  BEGIN;                                                │   │
│  │  UPDATE accounts SET balance = balance - 100             │   │
│  │  WHERE user_id = 1 AND shard_id = 1;  -- 分片 1        │   │
│  │                                                         │   │
│  │  UPDATE accounts SET balance = balance + 100             │   │
│  │  WHERE user_id = 2 AND shard_id = 2;  -- 分片 2        │   │
│  │  COMMIT;                                                │   │
│  │                                                         │   │
│  │  问题：分片 1 成功，分片 2 失败！                      │   │
│  │                                                         │   │
│  │  ✗ 原子性无法保证                                     │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 解决方案一：两阶段提交（2PC）

```
┌─────────────────────────────────────────────────────────────────┐
│                    两阶段提交（2PC）                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  第一阶段：准备（Prepare）                                │   │
│  │                                                         │   │
│  │  协调者 ───→ 分片 1：准备提交                             │   │
│  │        └──→ 分片 2：准备提交                             │   │
│  │                                                         │   │
│  │  分片 1：锁定资源，返回「就绪」                         │   │
│  │  分片 2：锁定资源，返回「就绪」                         │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                              ↓                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  第二阶段：提交（Commit）                                │   │
│  │                                                         │   │
│  │  协调者 ───→ 分片 1：正式提交                           │   │
│  │        └──→ 分片 2：正式提交                           │   │
│  │                                                         │   │
│  │  所有分片都提交成功！                                   │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  问题：                                                      │
│  - 协调者单点故障                                            │
│  - 锁定时间过长                                             │
│  - 部分提交风险                                            │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 解决方案二：TCC 模式

```
┌─────────────────────────────────────────────────────────────────┐
│                    TCC 模式                                      │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  Try：预留资源                                         │   │
│  │                                                         │   │
│  │  - 冻结用户 A 的 100 元（balance - 100, frozen + 100） │   │
│  │  - 冻结用户 B 的 100 元（frozen 不变）                 │   │
│  └─────────────────────────────────────────────────────────┘   │
│                              ↓                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  Confirm：确认执行                                     │   │
│  │                                                         │   │
│  │  - 用户 A：frozen - 100                               │   │
│  │  - 用户 B：balance + 100                              │   │
│  └─────────────────────────────────────────────────────────┘   │
│                              ↓                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  Cancel：取消释放                                       │   │
│  │                                                         │   │
│  │  - 用户 A：frozen + 100                               │   │
│  │  - 用户 B：（无需操作）                                 │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  特点：业务侵入性强，但一致性较好                             │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 面试实战

### 问题一：跨分片查询怎么处理？

**面试官问**：「分库分表后如何处理跨节点查询？」

**深入回答**：

> **解决方案**：
>
> 1. **路由 + 合并**：计算路由，并行查询，合并结果
> 2. **异构索引表**：建立分片路由映射
> 3. **ES 搜索引擎**：复杂查询走 ES
>
> **实际案例**：
> - 按 user_id 查询：直接路由到对应分片
> - 按 order_id 查询：查索引表获取分片
> - 多维度查询：走 ES

### 问题二：分布式事务怎么处理？

**面试官问**：「跨分片的事务怎么处理？」

**深入回答**：

> **常用方案**：
>
> 1. **TCC**：Try-Confirm-Cancel，侵入性强但一致性好
> 2. **最大努力通知**：允许最终一致
> 3. **本地消息表**：记录消息，异步处理
>
> **选择建议**：
> - 核心交易：TCC
> - 非核心业务：消息队列

### 问题三：深度分页怎么处理？

**面试官问**：「跨分片深度分页怎么处理？」

**深入回答**：

> **解决方案**：
>
> 1. **限制时间范围**：按时间分片，限制查询范围
> 2. **游标分页**：使用 last_id 代替 offset
> 3. **ES 统计**：分析类查询走 ES
>
> **实际建议**：
> - Feed 流：游标分页
> - 报表统计：ES 或离线处理

## 总结

分库分表后的挑战：

- **跨节点查询**：路由合并、异构索引、ES
- **排序分页**：限制范围、游标分页、ES
- **分布式事务**：TCC、最大努力通知、消息队列

合理选择解决方案，在一致性和性能之间做权衡。
