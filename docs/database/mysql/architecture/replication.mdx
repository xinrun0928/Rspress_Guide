# 主从复制原理：binlog、relaylog、IO 线程、SQL 线程

「MySQL 的主从复制是怎么工作的？为什么会有主从延迟？」

主从复制是 MySQL 高可用架构的基础。本文深入剖析主从复制的原理，包括 binlog、relaylog、IO 线程、SQL 线程的工作流程。

## 主从复制概述

### 什么是主从复制？

主从复制是指将主库（Master）的数据实时同步到从库（Slave），实现数据备份、读写分离等功能。

```
┌─────────────────────────────────────────────────────────────────┐
│                    主从复制架构                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                      Master（主库）                     │   │
│  │                                                         │   │
│  │  ┌─────────────────────────────────────────────────┐   │   │
│  │  │  执行 SQL                                     │   │   │
│  │  │  ↓                                           │   │   │
│  │  │  写入数据到 InnoDB                           │   │   │
│  │  │  ↓                                           │   │   │
│  │  │  写入 binlog                                 │   │   │
│  │  └─────────────────────────────────────────────────┘   │   │
│  │                         ↓ binlog                         │
│  └─────────────────────────┼───────────────────────────────┘   │
│                            │                                      │
│              ┌─────────────┼─────────────┐                       │
│              │             │             │                       │
│              ▼             ▼             ▼                       │
│         ┌────────┐   ┌────────┐   ┌────────┐                │
│         │ Slave1 │   │ Slave2 │   │ Slave3 │                │
│         └────────┘   └────────┘   └────────┘                │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

```
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  1. 数据备份                                        │   │
│  │     从库作为主库的数据副本，实现热备份                 │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  2. 读写分离                                        │   │

│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## binlog 详解

### 什么是 binlog？

binlog（Binary Log）是 MySQL Server 层维护的日志，记录了所有修改数据的 SQL 语句。

```
┌─────────────────────────────────────────────────────────────────┐
│                    binlog 概述                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  binlog 记录内容：                                     │   │
│  │                                                         │   │
│  │  - DDL：CREATE、ALTER、DROP 等                      │   │
│  │  - DML：INSERT、UPDATE、DELETE 等                    │   │
│  │  - 其他：COMMIT、ROLLBACK 等                        │   │
│  │                                                         │   │
│  │  binlog 不记录：                                     │   │
│  │  - SELECT、SHOW 等查询语句                          │   │
│  │  - 不修改数据的语句                                  │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### binlog 的格式

binlog 有三种格式：

```
┌─────────────────────────────────────────────────────────────────┐
│                    binlog 三种格式                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  1. STATEMENT：记录 SQL 语句                         │   │
│  │                                                         │   │
│  │  - 记录修改的 SQL                                    │   │
│  │  - 优点：日志量小                                     │   │
│  │  - 缺点：函数、主从库时间差异可能导致不一致          │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  2. ROW：记录修改的行                                │   │
│  │                                                         │   │
│  │  - 记录修改前后的行数据                              │   │
│  │  - 优点：数据一致性强                                 │   │
│  │  - 缺点：日志量大                                     │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  3. MIXED：混合格式                                  │   │
│  │                                                         │   │
│  │  - 根据情况自动选择 STATEMENT 或 ROW                   │   │
│  │  - MySQL 8.0 默认格式                                │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 主从复制原理

### 主从复制流程

```
┌─────────────────────────────────────────────────────────────────┐
│                    主从复制完整流程                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Master：                                                      │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  1. 接收 SQL 请求                                       │   │
│  │  2. 执行 SQL（写入 InnoDB）                            │   │
│  │  3. 写入 binlog（事务提交时）                         │   │
│  │  4. 通知从库来拉取 binlog                             │   │
│  └─────────────────────────────────────────────────────────┘   │
│                          ↓                                       │
│                    网络传输 binlog                               │
│                          ↓                                       │
│  Slave：                                                      │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  IO Thread：                                         │   │
│  │  5. 接收 binlog，写入 relay-log                      │   │
│  │  6. 更新 relay-log 位置                             │   │
│  │  7. 通知 SQL Thread                                 │   │
│  └─────────────────────────────────────────────────────────┘   │
│                          ↓                                       │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  SQL Thread：                                         │   │
│  │  8. 读取 relay-log                                   │   │
│  │  9. 执行 SQL（写入本地 InnoDB）                      │   │
│  │  10. 更新本地 binlog 位置                           │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### IO Thread 工作原理

```
┌─────────────────────────────────────────────────────────────────┐
│                    IO Thread 工作流程                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  IO Thread 持续从主库拉取 binlog：                     │   │
│  │                                                         │   │
│  │  1. 与主库建立连接                                    │   │
│  │  2. 发送 binlog position                              │   │
│  │  3. 等待主库推送新的 binlog                          │   │
│  │  4. 接收 binlog，写入 relay-log                     │   │
│  │  5. 更新 master_log_file 和 read_log_pos            │   │
│  │  6. 回到步骤 3，继续等待                             │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  连接信息保存在：slave_master_info 表                │   │
│  │  relay-log 信息保存在：slave_relay_log_info 表      │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### SQL Thread 工作原理

```
┌─────────────────────────────────────────────────────────────────┐
│                    SQL Thread 工作流程                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  SQL Thread 从 relay-log 读取并执行事件：             │   │
│  │                                                         │   │
│  │  1. 读取 relay-log 中的事件                          │   │
│  │  2. 解析 SQL                                           │   │
│  │  3. 执行 SQL（写入本地 InnoDB）                      │   │
│  │  4. 更新执行位置（exec_master_log_pos）               │   │
│  │  5. 回到步骤 1                                        │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  问题：如果 SQL Thread 落后太多，会产生延迟          │   │
│  │  - IO Thread 持续拉取 binlog                        │   │
│  │  - SQL Thread 单线程执行，可能跟不上                │   │
│  │  → 需要并行复制技术                                  │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 同步方式

### 异步复制（Async Replication）

```
┌─────────────────────────────────────────────────────────────────┐
│                    异步复制流程                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Master：                                                      │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  1. 执行 SQL                                           │   │
│  │  2. 提交事务                                          │   │
│  │  3. 写入 binlog                                       │   │
│  │  4. 返回客户端「成功」                               │   │
│  │  ↓ 异步通知从库                                       │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  问题：主库提交成功，但从库可能还没同步                      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 半同步复制（Semi-sync Replication）

```
┌─────────────────────────────────────────────────────────────────┐
│                    半同步复制流程                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Master：                                                      │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  1. 执行 SQL                                           │   │
│  │  2. 提交事务                                          │   │
│  │  3. 写入 binlog                                       │   │
│  │  4. 等待至少一个从库确认接收                          │   │
│  │  5. 返回客户端「成功」                                 │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  优点：至少一个从库同步成功，数据一致性提高                    │
│  缺点：有延迟（等待从库确认）                                 │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 全同步复制（Full Sync Replication）

```
┌─────────────────────────────────────────────────────────────────┐
│                    全同步复制流程                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Master：                                                      │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  1. 执行 SQL                                           │   │
│  │  2. 提交事务                                          │   │
│  │  3. 写入 binlog                                       │   │
│  │  4. 等待所有从库确认接收                              │   │
│  │  5. 返回客户端「成功」                                 │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  优点：所有从库都同步成功，数据一致性最高                      │
│  缺点：延迟很大，性能差                                       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## relay-log 详解

### 什么是 relay-log？

relay-log 是从库本地的日志文件，存储从主库接收到的 binlog 内容。

```
┌─────────────────────────────────────────────────────────────────┐
│                    relay-log 结构                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  relay-log 文件：                                      │   │
│  │                                                         │   │
│  │  - relay-bin.000001                                  │   │
│  │  - relay-bin.000002                                  │   │
│  │  - relay-bin.index                                   │   │
│  │                                                         │   │
│  │  位置：datadir/relay-log                              │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  相关配置：                                            │   │
│  │                                                         │   │
│  │  - relay_log = relay-bin                             │   │
│  │  - relay_log_purge = ON                             │   │
│  │  - max_relay_log_size = 0                           │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### relay-log 的清理

```
┌─────────────────────────────────────────────────────────────────┐
│                    relay-log 清理机制                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  SQL Thread 执行完 relay-log 后：                       │   │
│  │                                                         │   │
│  │  1. 检查是否需要清理                                  │   │
│  │  2. 删除已执行完的 relay-log 文件                     │   │
│  │  3. 更新 relay-log.index                             │   │
│  │                                                         │   │
│  │  配置项：relay_log_purge = ON                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  注意：如果 SQL Thread 落后太多，relay-log 会堆积               │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 主从复制配置

### 主库配置

```ini
[mysqld]
# 开启 binlog
log-bin = mysql-bin
server-id = 1

# binlog 格式
binlog-format = ROW

# 自动清理 binlog
expire_logs_days = 7

# GTID 模式
gtid_mode = ON
enforce_gtid_consistency = ON
```

```sql
-- 创建复制用户
CREATE USER 'repl'@'%' IDENTIFIED BY 'repl_password';
GRANT REPLICATION SLAVE ON *.* TO 'repl'@'%';

-- 查看主库状态
SHOW MASTER STATUS;
```

### 从库配置

```ini
[mysqld]
# 开启 relay-log
relay-log = relay-bin
server-id = 2

# GTID 模式
gtid_mode = ON
enforce_gtid_consistency = ON

# 并行复制
slave_parallel_workers = 4
slave_parallel_type = LOGICAL_CLOCK
```

```sql
-- 配置主从复制
CHANGE MASTER TO
    MASTER_HOST = 'master_host',
    MASTER_PORT = 3306,
    MASTER_USER = 'repl',
    MASTER_PASSWORD = 'repl_password',
    MASTER_LOG_FILE = 'mysql-bin.000001',
    MASTER_LOG_POS = 12345;

-- 或者使用 GTID
CHANGE MASTER TO
    MASTER_HOST = 'master_host',
    MASTER_PORT = 3306,
    MASTER_USER = 'repl',
    MASTER_PASSWORD = 'repl_password',
    MASTER_AUTO_POSITION = 1;

-- 启动复制
START SLAVE;

-- 查看从库状态
SHOW SLAVE STATUS\G
```

## 主从延迟

### 延迟原因

```
┌─────────────────────────────────────────────────────────────────┐
│                    主从延迟原因                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  1. 网络延迟                                            │   │
│  │     binlog 传输需要时间                                 │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  2. 从库执行慢                                        │   │
│  │     - SQL Thread 单线程执行                           │   │
│  │     - 从库服务器性能差                                 │   │
│  │     - 从库有大事务执行                                 │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  3. 从库压力大                                        │   │
│  │     - 多个从库同时复制同一主库                       │   │
│  │     - 从库承担其他业务负载                           │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  4. 大事务问题                                        │   │
│  │     - 主库执行一个大事务                              │   │
│  │     - binlog 一次性传输大量数据                      │   │
│  │     - 从库需要较长时间执行                           │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 面试实战

### 问题一：MySQL 主从复制原理是什么？

**面试官问**：「MySQL 主从复制是怎么实现的？」

**深入回答**：

> **主从复制原理**：
>
> 1. **主库**：执行 SQL，写入 InnoDB，写入 binlog
> 2. **从库 IO Thread**：连接主库，拉取 binlog，写入 relay-log
> 3. **从库 SQL Thread**：读取 relay-log，执行 SQL
>
> **核心组件**：
> - binlog：主库记录修改的日志
> - relay-log：从库存储接收到的 binlog
> - IO Thread：负责拉取 binlog
> - SQL Thread：负责执行 relay-log 中的 SQL

### 问题二：binlog 有哪几种格式？区别是什么？

**面试官问**：「binlog 有哪几种格式？各自优缺点是什么？」

**深入回答**：

> **三种格式**：
>
> | 格式 | 记录内容 | 优点 | 缺点 |
> |---|---|---|---|
> | STATEMENT | SQL 语句 | 日志量小 | 函数、时间差异可能导致不一致 |
> | ROW | 修改的行 | 数据一致性强 | 日志量大 |
> | MIXED | 自动选择 | 平衡 | - |
>
> **MySQL 8.0 默认是 MIXED**。

### 问题三：主从延迟是怎么产生的？

**面试官问**：「主从延迟是怎么产生的？如何解决？」

**深入回答**：

> **延迟原因**：
>
> 1. **网络延迟**：binlog 传输需要时间
> 2. **从库执行慢**：SQL Thread 单线程执行
> 3. **从库压力大**：承担其他业务负载
> 4. **大事务**：主库一次性执行大量数据
>
> **解决方案**：
> - 使用并行复制
> - 优化从库性能
> - 拆分大事务
> - 读写分离

## 总结

主从复制是 MySQL 高可用架构的基础：

- **binlog**：记录主库修改的日志
- **relay-log**：从库存储接收到的 binlog
- **IO Thread**：拉取 binlog 到从库
- **SQL Thread**：执行 relay-log 中的 SQL
- **同步方式**：异步、半同步、全同步
- **延迟原因**：网络、执行速度、大事务

理解主从复制原理，是掌握 MySQL 高可用架构的基础。
