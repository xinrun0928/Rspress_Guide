# 读写分离架构与数据一致性

「读写分离后，如何保证数据一致性？」

读写分离是 MySQL 高可用架构的核心组件。本文深入讲解读写分离的原理、架构和数据一致性保证。

## 读写分离概述

### 什么是读写分离？

读写分离是指将写操作路由到主库，读操作路由到从库，实现读写分离的架构。

```
┌─────────────────────────────────────────────────────────────────┐
│                    读写分离架构                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│                          请求                                     │
│                            ↓                                       │
│              ┌───────────────────────────┐                        │
│              │       读写分离中间件        │                        │
│              │    （Proxy / SDK / 代理）   │                        │
│              └───────────────┬───────────┘                        │
│                              │                                   │
│              ┌───────────────┴───────────┐                       │
│              ↓                           ↓                       │
│        ┌───────────┐             ┌───────────┐                │
│        │   Master   │             │   Slave    │                │
│        │  (写库)    │             │  (读库)    │                │
│        └───────────┘             └───────────┘                │
│                                          ↑                       │
│                                      更多从库                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 读写分离的优势

```
┌─────────────────────────────────────────────────────────────────┐
│                    读写分离的优势                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  1. 提升并发能力                                        │   │
│  │     写操作在主库，读操作在从库，互不干扰               │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  2. 降低主库压力                                        │   │
│  │     大部分业务读多写少，从库承担读请求                 │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  3. 提升可用性                                          │   │
│  │     从库故障不影响写操作，可快速切换                  │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  4. 负载均衡                                          │   │
│  │     多个从库分摊读请求                               │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 读写分离实现方式

### 方式一：应用层实现（SDK 模式）

```
┌─────────────────────────────────────────────────────────────────┐
│                    SDK 模式读写分离                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  应用代码：                                             │   │
│  │                                                         │   │
│  │  // 写操作（自动路由到主库）                           │   │
│  │  db.write().insert("users", data);                       │   │
│  │                                                         │   │
│  │  // 读操作（自动路由到从库）                           │   │
│  │  db.read().select("users", where);                      │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  常用框架：ShardingSphere-JDBC、MySQL Router、HikariCP           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 方式二：代理层实现（Proxy 模式）

```
┌─────────────────────────────────────────────────────────────────┐
│                    Proxy 模式读写分离                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  架构：                                                │   │
│  │                                                         │   │
│  │  应用 ──→ Proxy ──→ Master (写)                        │   │
│  │    │       │              │                            │   │
│  │    │       │              └──→ Slave1 (读)             │   │
│  │    │       │                  └──→ Slave2 (读)         │   │
│  │    │       │                  └──→ Slave3 (读)         │   │
│  │    └───────┘                                             │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  常用中间件：MySQL Router、ProxySQL、Atlas、ShardingSphere-Proxy   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 两种方式对比

| 方式 | 优点 | 缺点 |
|---|---|---|
| SDK 模式 | 性能好，无额外组件 | 代码侵入性 |
| Proxy 模式 | 代码无侵入 | 有性能损耗，需要维护组件 |

## 数据一致性问题

### 主从延迟导致的不一致

```
┌─────────────────────────────────────────────────────────────────┐
│                    主从延迟问题                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  用户A：写操作                                                │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  1. INSERT INTO users VALUES (1, '张三');  -- 主库成功    │   │
│  │  2. SELECT * FROM users WHERE id=1;   -- 查从库           │   │
│  │                                                         │   │
│  │  结果：查不到！因为从库还在同步中...                      │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  这是读写分离最常见的问题！                                     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 场景一：注册后立即登录

```sql
-- 用户注册
INSERT INTO users (name, password) VALUES ('张三', 'password123');
COMMIT;

-- 立即查询
SELECT * FROM users WHERE name = '张三';
-- 可能查不到（主从延迟）
```

### 场景二：更新后立即查询

```sql
-- 更新用户信息
UPDATE users SET name = '李四' WHERE id = 1;
COMMIT;

-- 立即查询
SELECT * FROM users WHERE id = 1;
-- 可能还是旧数据（主从延迟）
```

### 场景三：删除后立即查询

```sql
-- 删除数据
DELETE FROM users WHERE id = 1;
COMMIT;

-- 立即查询
SELECT * FROM users WHERE id = 1;
-- 可能还能查到（主从延迟）
```

## 数据一致性解决方案

### 方案一：强制路由主库

```
┌─────────────────────────────────────────────────────────────────┐
│                    强制主库读取                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  关键读操作强制走主库：                                 │   │
│  │                                                         │   │
│  │  // 注册后立即查询，强制走主库                         │   │
│  │  db.master().select("users", where);                    │   │
│  │                                                         │   │
│  │  // 支付完成后查询，强制走主库                        │   │
│  │  db.master().select("orders", where);                   │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  适用场景：写入后必须立即读取的操作                              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 方案二：延迟感知

```
┌─────────────────────────────────────────────────────────────────┐
│                    延迟感知机制                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  感知主从延迟：                                        │   │
│  │                                                         │   │
│  │  1. 监控 Seconds_Behind_Master                      │   │
│  │  2. 延迟超过阈值，走主库                             │   │
│  │  3. 延迟低于阈值，走从库                             │   │
│  │                                                         │   │
│  │  延迟阈值：0.5-1 秒                                 │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  实现：                                                        │
│  if (slave_lag > threshold) {                                 │
│      read_from_master();                                       │
│  } else {                                                    │
│      read_from_slave();                                        │
│  }                                                            │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 方案三：读取主库半同步确认

```
┌─────────────────────────────────────────────────────────────────┐
│                    半同步复制确认                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  半同步复制：主库等待至少一个从库确认后才返回            │   │
│  │                                                         │   │
│  │  1. INSERT INTO users VALUES (...);  -- 主库执行        │   │
│  │  2. 等待从库确认接收 binlog                          │   │
│  │  3. 返回客户端成功                                    │   │
│  │  4. 立即读从库，可以读到                              │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  注意：只是接收，不要求执行完成                                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 方案四：业务层补偿

```
┌─────────────────────────────────────────────────────────────────┐
│                    业务层补偿机制                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  读取失败后重试到主库：                                 │   │
│  │                                                         │   │
│  │  try {                                                  │   │
│  │      result = db.read().select("users", where);         │   │
│  │  } catch (NotFoundException e) {                        │   │
│  │      // 从库查不到，从主库再查一次                       │   │
│  │      result = db.master().select("users", where);       │   │
│  │  }                                                       │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 面试实战

### 问题一：读写分离会导致什么问题？

**面试官问**：「读写分离会导致什么问题？如何解决？」

**深入回答**：

> **问题**：主从延迟导致读写不一致
>
> **场景**：
> - 写入后立即查询，可能查不到
> - 更新后立即查询，可能还是旧数据
>
> **解决方案**：
> 1. 强制读主库（关键操作走主库）
> 2. 延迟感知（延迟高时走主库）
> 3. 半同步复制（从库确认接收后再读）
> 4. 业务补偿（查不到再查主库）

### 问题二：读写分离适用于什么场景？

**面试官问**：「什么场景适合读写分离？」

**深入回答**：

> **适合场景**：
> 1. **读多写少**：读请求远多于写请求
> 2. **能接受短暂延迟**：业务能容忍秒级延迟
> 3. **读请求可拆分**：可以路由到从库
>
> **不适合场景**：
> 1. **写多读少**：写请求占比高
> 2. **强一致性**：任何延迟都不能接受
> 3. **跨事务读取**：事务内多次读取必须一致

### 问题三：如何保证读写分离后的数据一致性？

**面试官问**：「读写分离下如何保证数据一致性？」

**深入回答**：

> **保证一致性的方法**：
>
> 1. **强制读主库**：关键操作走主库
> 2. **半同步复制**：主库等待从库确认
> 3. **延迟监控**：延迟高时切换主库
> 4. **业务补偿**：查不到再查主库
>
> **最佳实践**：
> - 核心数据（订单、支付）强制读主库
> - 非核心数据（日志、统计）可接受延迟

## 总结

读写分离是 MySQL 高可用的重要组件：

- **原理**：写主库、读从库
- **实现**：SDK 模式或 Proxy 模式
- **问题**：主从延迟导致不一致
- **解决**：强制主库、半同步复制、延迟感知、业务补偿

合理设计读写分离架构，可以显著提升数据库并发能力。
