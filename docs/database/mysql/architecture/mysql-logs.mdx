# MySQL 三大日志深度剖析：binlog、redo log、undo log

「MySQL 的 binlog、redo log、undo log 有什么区别？它们是如何协作的？」

MySQL 的日志系统是 InnoDB 存储引擎的核心组件。binlog、redo log、undo log 各司其职，共同保证了 MySQL 的可靠性。本文深入剖析这三大日志的原理和协作机制。

## binlog 深度剖析

### binlog 是什么？

binlog（Binary Log）是 MySQL Server 层维护的二进制日志文件，记录了所有数据修改操作。

```
┌─────────────────────────────────────────────────────────────────┐
│                    binlog 作用                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  binlog 的三大作用：                                    │   │
│  │                                                         │   │
│  │  1. 主从复制                                            │   │
│  │     - 主库记录 binlog                                  │   │
│  │     - 从库读取 binlog 并执行                           │   │
│  │     - 实现数据同步                                     │   │
│  │                                                         │   │
│  │  2. 数据恢复                                            │   │
│  │     - 利用 binlog 恢复数据                             │   │
│  │     - point-in-time 恢复                               │   │
│  │                                                         │   │
│  │  3. 增量备份                                            │   │
│  │     - 支持增量备份                                     │   │
│  │     - 配合全量备份实现恢复                             │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### binlog 格式

```
┌─────────────────────────────────────────────────────────────────┐
│                    binlog 三种格式                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  1. STATEMENT（基于语句）                                │   │
│  │                                                         │   │
│  │  # 记录 SQL 语句                                        │   │
│  │  UPDATE users SET name = '张三' WHERE id = 1;          │   │
│  │                                                         │   │
│  │  优点：日志量小                                         │   │
│  │  缺点：函数、主键冲突等问题                             │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  2. ROW（基于行）                                        │   │
│  │                                                         │   │
│  │  # 记录每一行的变化                                     │   │
│  │  TABLE: users                                          │   │
│  │  ROW: id=1, name='张三' → id=1, name='李四'           │   │
│  │                                                         │   │
│  │  优点：准确记录每一行变化                              │   │
│  │  缺点：日志量大                                        │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  3. MIXED（混合模式）                                    │   │
│  │                                                         │   │
│  │  - 默认使用 STATEMENT                                  │   │
│  │  - 遇到不确定结果时切换到 ROW                          │   │
│  │                                                         │   │
│  │  优点：平衡日志量和准确性                              │   │
│  │  缺点：复杂                                            │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### STATEMENT vs ROW 对比

```sql
-- 原始 SQL
UPDATE users SET update_time = NOW() WHERE id = 1;

-- STATEMENT 格式 binlog
UPDATE users SET update_time = NOW() WHERE id = 1;
-- 问题：主从执行时，NOW() 值不同！

-- ROW 格式 binlog
UPDATE users SET update_time = '2024-01-01 12:00:00' WHERE id = 1 AND update_time = '2024-01-01 11:59:59';
-- 准确记录每一行的变化
```

## redo log 深度剖析

### redo log 是什么？

redo log 是 InnoDB 存储引擎的物理日志，记录了数据页的物理修改。

```
┌─────────────────────────────────────────────────────────────────┐
│                    redo log 的作用                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  核心作用：crash-safe（崩溃恢复）                       │   │
│  │                                                         │   │
│  │  问题：事务提交后，数据还没来得及刷盘，数据库崩溃了？   │   │
│  │                                                         │   │
│  │  解决：                                                   │   │
│  │  - 事务提交前，先写 redo log                           │   │
│  │  - redo log 持久化后，返回「提交成功」                  │   │
│  │  - 即使数据页没刷盘，崩溃后也可以从 redo log 恢复      │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### redo log vs 数据页

```
┌─────────────────────────────────────────────────────────────────┐
│                    redo log 与数据页的关系                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  没有 redo log：                                        │   │
│  │                                                         │   │
│  │  事务提交 → 数据页正在刷盘 → 崩溃！                   │   │
│  │  → 数据丢失！                                          │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  有 redo log：                                          │   │
│  │                                                         │   │
│  │  事务提交 → 写 redo log（持久化）→ 返回成功           │   │
│  │     ↓                                                   │   │
│  │  数据页异步刷盘 → 崩溃                                │   │
│  │     ↓                                                   │   │
│  │  从 redo log 恢复 → 数据完整！                         │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### redo log 写入机制

```
┌─────────────────────────────────────────────────────────────────┐
│                    redo log 写入流程                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  1. 修改数据页                                         │   │
│  │     ↓                                                   │   │
│  │  2. 写入 redo log buffer（内存）                        │   │
│  │     ↓                                                   │   │
│  │  3. 事务提交                                            │   │
│  │     ↓                                                   │   │
│  │  4. 调用 fsync 将 redo log buffer 刷到磁盘              │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  相关参数：                                            │   │
│  │  - innodb_log_buffer_size：redo log buffer 大小        │   │
│  │  - innodb_flush_log_at_trx_commit：刷盘策略            │   │
│  │    - = 1：每次提交都刷盘（默认，最安全）              │   │
│  │    - = 2：提交到 OS 缓存，不立即刷盘                  │   │
│  │    - = 0：定时刷盘                                     │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 三种刷盘策略对比

| 策略 | 安全性 | 性能 | 说明 |
|---|---|---|---|
| `innodb_flush_log_at_trx_commit=1` | 最高 | 最低 | 每次提交都刷盘 |
| `innodb_flush_log_at_trx_commit=2` | 中等 | 中等 | 提交到 OS 缓存 |
| `innodb_flush_log_at_trx_commit=0` | 最低 | 最高 | 后台线程定时刷盘 |

```sql
-- 设置刷盘策略
SET GLOBAL innodb_flush_log_at_trx_commit = 1;
```

## undo log 深度剖析

### undo log 是什么？

undo log 是 InnoDB 存储引擎的逻辑日志，记录了数据的「反向操作」，用于事务回滚和 MVCC。

```
┌─────────────────────────────────────────────────────────────────┐
│                    undo log 的作用                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  核心作用：                                            │   │
│  │                                                         │   │
│  │  1. 事务回滚                                            │   │
│  │     - 记录反向操作                                      │   │
│  │     - 事务失败时执行反向操作                           │   │
│  │                                                         │   │
│  │  2. MVCC（多版本并发控制）                              │   │
│  │     - 读取历史版本                                      │   │
│  │     - 实现快照读                                        │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### undo log 记录内容

```sql
-- 原始数据
INSERT INTO users (id, name) VALUES (1, '张三');
-- undo log 记录：DELETE FROM users WHERE id = 1;

UPDATE users SET name = '李四' WHERE id = 1;
-- undo log 记录：UPDATE users SET name = '张三' WHERE id = 1;

DELETE FROM users WHERE id = 1;
-- undo log 记录：INSERT INTO users (id, name) VALUES (1, '张三');
```

## 三大日志协作

### 事务提交流程

```
┌─────────────────────────────────────────────────────────────────┐
│                    事务提交时日志写入顺序                         │
```

## 面试实战

### 问题一：binlog、redo log、undo log 有什么区别？

**面试官问**：「binlog、redo log、undo log 有什么区别？」

**深入回答**：

> **核心区别**：
>
> | 日志 | 类型 | 作用域 | 作用 |
> |---|---|---|---|
> | binlog | 逻辑日志 | Server 层 | 主从复制、数据恢复 |
> | redo log | 物理日志 | InnoDB 层 | crash-safe、持久性 |
> | undo log | 逻辑日志 | InnoDB 层 | 回滚、MVCC |
>
> **协作关系**：
> - redo log 保证 crash-safe
> - binlog 用于主从复制
> - undo log 用于回滚和 MVCC
> - 事务提交时，redo log 和 binlog 两阶段提交

### 问题二：redo log 为什么比数据页先刷盘？

**面试官问**：「为什么 redo log 可以先于数据页刷盘？这样做有什么好处？」

**深入回答**：

> **核心原因**：
>
> 1. **顺序 IO  vs 随机 IO**
>    - redo log 是追加写，顺序 IO
>    - 数据页写入是随机 IO
>    - 顺序 IO 远快于随机 IO
>
> 2. **Write-Ahead Logging 原则**
>    - 先写日志，再写数据
>    - 日志持久化后，即使数据页丢失，也可以从日志恢复
>
> 3. **性能提升**
>    - 多次小更新合并为一次刷盘
>    - 减少数据页刷盘次数

### 问题三：为什么需要两阶段提交？

**面试官问**：「事务提交时为什么要两阶段提交？只写一个日志不行吗？」

**深入回答**：

> **为什么需要两阶段提交**：
>
> 1. **保证主从一致性**
>    - redo log 用于崩溃恢复
>    - binlog 用于主从复制
>    - 两个日志必须一致
>
> 2. **防止数据丢失**
>    - 如果只写 redo log，binlog 没写 → 从库数据丢失
>    - 如果只写 binlog，redo log 没写 → 恢复时数据丢失
>
> 3. **两阶段提交保证一致性**
>    - Prepare 阶段：写 redo log，标记 PREPARED
>    - Commit 阶段：写 binlog，标记 COMMIT
>    - 崩溃恢复时，根据两个日志的状态决定提交还是回滚

## 总结

MySQL 的三大日志各司其职：

- **binlog**：Server 层逻辑日志，用于主从复制和数据恢复
- **redo log**：InnoDB 层物理日志，用于 crash-safe
- **undo log**：InnoDB 层逻辑日志，用于回滚和 MVCC

事务提交时，redo log 和 binlog 通过两阶段提交保证一致性。崩溃恢复时，根据日志状态决定重做还是回滚。
