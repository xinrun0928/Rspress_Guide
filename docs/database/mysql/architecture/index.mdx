# 架构与高可用总览

随着业务增长，单库单表无法满足性能和可用性需求。这时需要考虑数据库架构的扩展——主从复制、读写分离、分库分表、高可用方案。

MySQL 的架构扩展是面试中的高级考点，也是资深工程师的必备技能。本模块带你理解 MySQL 主从复制的原理、读写分离的设计、分库分表的策略，以及常见的高可用方案。

## 主从复制

主从复制是 MySQL 扩展的基础。通过主从复制，可以实现：

**读写分离**：主库处理写操作，从库处理读操作，分担数据库压力。

**数据备份**：从库作为热备份，实时同步主库数据。

**故障恢复**：主库故障时，可以快速切换到从库。

MySQL 主从复制基于 Binlog 实现，有三个核心组件：

- **Master**：记录数据变更到 Binlog
- **Slave**：通过 IO Thread 拉取 Binlog，写入 Relay Log
- **Slave**：通过 SQL Thread 执行 Relay Log 中的事件

## 复制模式

| 模式 | 说明 | 延迟 | 数据一致性 |
|---|---|---|---|
| 异步复制 | 主库提交后立即返回，不等待从库确认 | 高 | 低 |
| 半同步复制 | 主库等待至少一个从库确认 | 中 | 中 |
| 全同步复制 | 主库等待所有从库执行完才返回 | 低 | 高 |

## 读写分离

读写分离将读操作分散到从库，写操作集中在主库。需要解决两个问题：

1. **主从延迟**：从库数据可能暂时落后，敏感读场景需要强制走主库
2. **数据一致性**：如何保证读写路由到正确的数据源

## 分库分表

当单表数据量过大（如超过千万级）、单库 IO 瓶颈时，需要分库分表：

**垂直拆分**：按业务模块拆分，将大表按列拆分到不同库/表。

**水平拆分**：按数据行拆分，将数据分散到多张表。

分片键的选择至关重要，直接影响数据分布的均匀性和查询效率。
