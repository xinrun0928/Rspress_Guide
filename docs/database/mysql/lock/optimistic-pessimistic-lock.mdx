# 乐观锁与悲观锁实现

「并发控制用乐观锁还是悲观锁？各有什么优缺点？」

乐观锁和悲观锁是两种不同的并发控制策略，适用于不同的业务场景。理解它们的实现方式和适用场景，是后端开发的必备知识。

## 两种锁策略概述

```
┌─────────────────────────────────────────────────────────────────┐
│                    乐观锁 vs 悲观锁                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                    悲观锁（Pessimistic Lock）            │   │
│  │                                                         │   │
│  │  核心思想：假设并发冲突一定会发生                        │   │
│  │  策略：先加锁，再操作，操作完释放锁                      │   │
│  │  实现：SELECT ... FOR UPDATE                            │   │
│  │  适用：冲突频繁的场景                                   │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                    乐观锁（Optimistic Lock）             │   │
│  │                                                         │   │
│  │  核心思想：假设并发冲突很少发生                          │   │
│  │  策略：先操作，发现冲突再回滚                           │   │
│  │  实现：版本号 / CAS                                      │   │
│  │  适用：冲突较少的场景                                   │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 悲观锁（Pessimistic Lock）

### 核心思想

悲观锁假设并发冲突一定会发生，因此在操作前先获取锁，保证操作的原子性。

```
┌─────────────────────────────────────────────────────────────────┐
│                    悲观锁工作流程                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  悲观锁的执行流程：                                       │   │
│  │                                                         │   │
│  │  1. 获取锁（SELECT ... FOR UPDATE）                     │   │
│  │  2. 执行操作                                            │   │
│  │  3. 释放锁（COMMIT/ROLLBACK）                           │   │
│  │                                                         │   │
│  │  ┌─────────────────────────────────────────────────┐   │   │
│  │  │  ┌─────────────────────────────────────────┐   │   │   │
│  │  │  │        锁住资源                         │   │   │   │
│  │  │  │  ┌───────────────────────────────────┐ │   │   │   │
│  │  │  │  │  执行操作                         │ │   │   │   │
│  │  │  │  └───────────────────────────────────┘ │   │   │   │
│  │  │  └─────────────────────────────────────────┘   │   │   │
│  │  └─────────────────────────────────────────────────┘   │   │
│  │                                                         │   │
│  │  其他事务必须等待锁释放才能继续                          │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### MySQL 悲观锁实现

```sql
-- 使用 SELECT ... FOR UPDATE 获取悲观锁
BEGIN;

-- 锁定单行
SELECT * FROM users WHERE id = 1 FOR UPDATE;

-- 锁定多行
SELECT * FROM users WHERE id IN (1, 2, 3) FOR UPDATE;

-- 锁定范围（会锁定整个范围）
SELECT * FROM users WHERE age BETWEEN 20 AND 30 FOR UPDATE;

-- 执行更新
UPDATE users SET balance = balance - 100 WHERE id = 1;

COMMIT;  -- 释放锁
```

### 悲观锁的 Java 实现

```java
// 使用悲观锁查询
public void updateBalance(String userId, BigDecimal amount) {
    jdbcTemplate.execute("BEGIN");

    // 获取悲观锁
    User user = jdbcTemplate.queryForObject(
        "SELECT * FROM users WHERE id = ? FOR UPDATE",
        new Object[]{userId},
        userRowMapper
    );

    // 执行更新
    BigDecimal newBalance = user.getBalance().subtract(amount);
    jdbcTemplate.update(
        "UPDATE users SET balance = ? WHERE id = ?",
        newBalance, userId
    );

    jdbcTemplate.execute("COMMIT");
}
```

### 悲观锁的特点

| 特点 | 说明 |
|---|---|
| 实现方式 | 先获取锁，再操作 |
| 冲突处理 | 锁机制保证互斥 |
| 性能影响 | 并发时需要等待锁 |
| 适用场景 | 冲突频繁、数据一致性要求高 |

## 乐观锁（Optimistic Lock）

### 核心思想

乐观锁假设并发冲突很少发生，因此不直接加锁，而是在更新时检查版本号，如果版本号不匹配则回滚重试。

```
┌─────────────────────────────────────────────────────────────────┐
│                    乐观锁工作流程                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  乐观锁的执行流程：                                       │   │
│  │                                                         │   │
│  │  1. 读取数据，获取版本号                                  │   │
│  │  2. 执行操作                                            │   │
│  │  3. 更新时检查版本号：                                   │   │
│  │     - 版本匹配：更新成功，版本号 +1                      │   │
│  │     - 版本不匹配：更新失败，回滚重试                     │   │
│  │                                                         │   │
│  │  ┌─────────────────────────────────────────────────┐   │   │
│  │  │  读取版本号 = 1                                    │   │   │
│  │  │       │                                            │   │   │
│  │  │       ▼                                            │   │   │
│  │  │  执行操作                                          │   │   │
│  │  │       │                                            │   │   │
│  │  │       ▼                                            │   │   │
│  │  │  UPDATE ... WHERE id = ? AND version = 1           │   │   │
│  │  │       │                                            │   │   │
│  │  │  ┌────┴────┐                                      │   │   │
│  │  │  │  成功    │ 失败                                │   │   │
│  │  │  ▼         ▼                                     │   │   │
│  │  │ 版本+1    重试                                    │   │   │
│  │  └─────────────────────────────────────────────────┘   │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 乐观锁的实现方式

#### 方式一：版本号字段

```sql
-- 表结构：添加 version 字段
CREATE TABLE users (
    id BIGINT PRIMARY KEY,
    name VARCHAR(50),
    balance DECIMAL(10, 2),
    version INT DEFAULT 0  -- 版本号字段
);

-- 查询时获取版本号
SELECT id, name, balance, version FROM users WHERE id = 1;

-- 更新时检查版本号
UPDATE users
SET balance = 900, version = version + 1
WHERE id = 1 AND version = 1;
-- 如果 version != 1，说明有其他事务修改过，更新失败
```

#### 方式二：时间戳

```sql
-- 表结构：使用 update_time 作为版本
CREATE TABLE users (
    id BIGINT PRIMARY KEY,
    name VARCHAR(50),
    balance DECIMAL(10, 2),
    update_time TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

-- 更新时检查时间戳
UPDATE users
SET balance = 900, update_time = NOW()
WHERE id = 1 AND update_time = '2024-01-15 10:30:00';
```

### 乐观锁的 Java 实现

```java
// 使用乐观锁更新
public void updateBalance(String userId, BigDecimal amount) {
    int maxRetries = 3;

    for (int i = 0; i < maxRetries; i++) {
        // 读取当前数据
        User user = jdbcTemplate.queryForObject(
            "SELECT * FROM users WHERE id = ?",
            new Object[]{userId},
            userRowMapper
        );

        // 计算新余额
        BigDecimal newBalance = user.getBalance().subtract(amount);

        // 更新（检查版本号）
        int updated = jdbcTemplate.update(
            "UPDATE users SET balance = ?, version = version + 1 " +
            "WHERE id = ? AND version = ?",
            newBalance, userId, user.getVersion()
        );

        if (updated == 1) {
            return;  // 更新成功
        }

        // 更新失败，说明有冲突，等待后重试
        if (i < maxRetries - 1) {
            try {
                Thread.sleep((long) (Math.random() * 100));
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        }
    }

    throw new RuntimeException("更新失败，重试次数超过上限");
}
```

### 乐观锁的特点

| 特点 | 说明 |
|---|---|
| 实现方式 | 版本号检查，无锁操作 |
| 冲突处理 | 更新失败后重试 |
| 性能影响 | 无锁时性能好，冲突时需重试 |
| 适用场景 | 冲突较少、数据一致性要求高 |

## 实际应用场景

### 场景一：库存扣减（悲观锁）

```sql
-- 使用悲观锁扣减库存
BEGIN;

-- 锁定商品记录
SELECT stock FROM products WHERE id = 1 FOR UPDATE;

-- 检查库存
UPDATE products SET stock = stock - 1 WHERE id = 1 AND stock > 0;

COMMIT;
```

```java
public void decreaseStock(String productId) {
    jdbcTemplate.execute("BEGIN");

    // 获取悲观锁
    Product product = jdbcTemplate.queryForObject(
        "SELECT * FROM products WHERE id = ? FOR UPDATE",
        new Object[]{productId},
        productRowMapper
    );

    if (product.getStock() > 0) {
        jdbcTemplate.update(
            "UPDATE products SET stock = stock - 1 WHERE id = ?",
            productId
        );
    }

    jdbcTemplate.execute("COMMIT");
}
```

```java
// 分布式环境下的乐观锁（Redis）
public void updateWithRedis(String userId, BigDecimal delta) {
    String lockKey = "lock:user:" + userId;
    String lockValue = UUID.randomUUID().toString();

    // 获取分布式锁
    Boolean acquired = redisTemplate.opsForValue()
        .setIfAbsent(lockKey, lockValue, 10, TimeUnit.SECONDS);

    if (!acquired) {
        throw new RuntimeException("获取锁失败");
    }

    try {
        // 执行操作
        User user = getUser(userId);
        user.setBalance(user.getBalance().add(delta));
        saveUser(user);
    } finally {
        // 释放锁
        redisTemplate.delete(lockKey);
    }
}
```

## 乐观锁的重试策略

### 重试策略设计

```java
public class OptimisticLockRetryTemplate {

    // 固定间隔重试
    public <T> T retryWithFixedDelay(Supplier<T> operation, int maxRetries, long delayMs) {
        for (int i = 0; i < maxRetries; i++) {
            try {
                return operation.get();
            } catch (OptimisticLockException e) {
                if (i < maxRetries - 1) {
                    try {
                        Thread.sleep(delayMs);
                    } catch (InterruptedException ie) {
                        Thread.currentThread().interrupt();
                        throw new RuntimeException("重试被中断");
                    }
                }
            }
        }
        throw new RuntimeException("重试次数超过上限");
    }

    // 指数退避重试
    public <T> T retryWithExponentialBackoff(Supplier<T> operation, int maxRetries) {
        long delay = 10;
        for (int i = 0; i < maxRetries; i++) {
            try {
                return operation.get();
            } catch (OptimisticLockException e) {
                if (i < maxRetries - 1) {
                    try {
                        Thread.sleep(delay);
                        delay *= 2;  // 指数增长
                    } catch (InterruptedException ie) {
                        Thread.currentThread().interrupt();
                        throw new RuntimeException("重试被中断");
                    }
                }
            }
        }
        throw new RuntimeException("重试次数超过上限");
    }
}
```

## 面试实战

### 问题一：乐观锁和悲观锁的区别是什么？

**面试官问**：「乐观锁和悲观锁有什么区别？各自适用于什么场景？」

**深入回答**：

> **悲观锁**：
> - 假设并发冲突一定会发生
> - 先获取锁，再操作，冲突时阻塞等待
> - 适用于冲突频繁、数据一致性要求高的场景
>
> **乐观锁**：
> - 假设并发冲突很少发生
> - 不加锁，通过版本号检测冲突，冲突时回滚重试
> - 适用于冲突较少、性能要求高的场景
>
> | 维度 | 乐观锁 | 悲观锁 |
> |---|---|---|
> | 实现方式 | 版本号检查 | 锁机制 |
> | 并发性能 | 高（无冲突时） | 低（需等锁） |
> | 冲突处理 | 重试 | 等待 |

### 问题二：乐观锁如何实现？

**面试官问**：「乐观锁是如何实现的？版本号字段怎么处理？」

**深入回答**：

> **乐观锁实现方式**：
>
> 1. **添加版本号字段**：
> ```sql
> CREATE TABLE users (
>     id BIGINT PRIMARY KEY,
>     name VARCHAR(50),
>     version INT DEFAULT 0
> );
> ```
>
> 2. **更新时检查版本号**：
> ```sql
> UPDATE users
> SET name = '新名字', version = version + 1
> WHERE id = ? AND version = ?;
> ```
>
> 3. **处理更新失败**：
> - 如果更新行数 = 0，说明版本号不匹配
> - 应用层捕获异常，重新读取数据重试

### 问题三：如何选择乐观锁和悲观锁？

**面试官问**：「在项目中应该选择乐观锁还是悲观锁？」

**深入回答**：

> **选择悲观锁的场景**：
> - 并发冲突频繁
> - 数据一致性要求极高
> - 操作时间长
> - 典型场景：金融交易、库存扣减
>
> **选择乐观锁的场景**：
> - 并发冲突较少
> - 读多写少
> - 追求高性能
> - 典型场景：用户信息修改、配置更新
>
> **实际建议**：
> - 先用乐观锁测试性能
> - 如果冲突太多，再考虑悲观锁
> - 分布式环境下，可以用 Redis 分布式锁

## 总结

乐观锁和悲观锁是两种重要的并发控制策略：

- **悲观锁**：先加锁后操作，适合冲突频繁场景
- **乐观锁**：无锁操作+版本检查，适合冲突较少场景
- **选择原则**：根据并发程度和一致性要求选择
- **实现方式**：悲观锁用 `FOR UPDATE`，乐观锁用版本号

理解两种锁的原理和使用场景，是后端开发的必备知识。
