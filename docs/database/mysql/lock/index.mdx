# 锁机制总览

并发是现代数据库的基本特性，而锁是实现并发控制的核心手段。MySQL 的锁机制决定了多个事务同时操作数据时的行为模式，直接影响数据库的并发性能和一致性。

面试中，锁机制是重点考察对象。全局锁与表级锁、行锁与临键锁、意向锁、两阶段锁、死锁——这些知识点需要系统掌握。

本模块带你深入理解 MySQL 的锁体系和各种锁的应用场景。

## 为什么要加锁？

如果没有锁机制，并发事务会导致各种问题：

**脏写**：两个事务同时修改同一行，后面的修改覆盖了前面的修改。

**脏读**：事务 A 读取了事务 B 未提交的数据，事务 B 回滚后，A 读取的数据是错误的。

**不可重复读**：事务 A 两次读取同一行，事务 B 在中间修改了这条数据，导致 A 两次读取结果不一致。

**幻读**：事务 A 按条件查询，事务 B 在中间插入了满足条件的新记录，导致 A 两次查询结果集不同。

锁机制通过限制并发操作，保证事务的隔离性。

## MySQL 锁分类

**按粒度分**：

| 锁类型 | 作用范围 | 引擎 |
|---|---|---|
| 全局锁 | 整个数据库 | 所有 |
| 表级锁 | 整张表 | MyISAM、InnoDB（MDL） |
| 行级锁 | 单行记录 | InnoDB |
| 临键锁 | 行 + 区间 | InnoDB |
| 间隙锁 | 区间 | InnoDB |
| 记录锁 | 单条索引记录 | InnoDB |

**按性质分**：

| 锁类型 | 作用 | 说明 |
|---|---|---|
| 共享锁（S锁） | 允许事务读取一行 | SELECT ... LOCK IN SHARE MODE |
| 排他锁（X锁） | 允许事务修改一行 | SELECT ... FOR UPDATE |
| 意向共享锁（IS锁） | 准备获取行的共享锁 | InnoDB 自动加 |
| 意向排他锁（IX锁） | 准备获取行的排他锁 | InnoDB 自动加 |

## InnoDB 行锁

InnoDB 的行锁是基于索引实现的。查询条件必须命中索引，才能加行锁；如果查询没有命中索引，InnoDB 会升级为表锁。

**临键锁（Next-Key Lock）**：锁住索引记录和它前面的间隙，是 InnoDB 在 REPEATABLE READ 隔离级别下的默认加锁方式。防止幻读。

**间隙锁（Gap Lock）**：只锁住区间，不锁记录，用于防止幻读。

**记录锁（Record Lock）**：锁住单条索引记录。

## 死锁

死锁是指两个或多个事务互相持有对方需要的锁，形成循环等待。InnoDB 有两种处理策略：

1. **超时回滚**：设置 `innodb_lock_wait_timeout`，超时的事务自动回滚
2. **死锁检测**：InnoDB 主动检测死锁，回滚代价最小的事务
