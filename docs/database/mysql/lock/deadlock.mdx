# 死锁产生条件与处理策略

「死锁是如何发生的？为什么有时候查询会超时？」

死锁是数据库并发控制的经典问题，也是面试高频考点。本文深入剖析死锁的产生条件、检测机制和处理策略。

## 死锁的概念

### 什么是死锁？

死锁是指两个或多个事务在执行过程中，因争夺资源而造成相互等待的现象，若无外力干预，它们都将无法推进下去。

```
┌─────────────────────────────────────────────────────────────────┐
│                    死锁示意                                      │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  正常情况：                                                    │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  事务 A ──→ 获取锁1 ──→ 获取锁2 ──→ 释放锁 ──→ 完成    │   │
│  │  事务 B ──→ 获取锁2 ──→ 获取锁1 ──→ 释放锁 ──→ 完成    │   │
│  │                      ↑                                  │   │
│  │                 锁顺序相同，不会冲突                      │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  死锁情况：                                                    │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  事务 A ──→ 获取锁1 ──→ 等待锁2 ──→ ?                  │   │
│  │  事务 B ──→ 获取锁2 ──→ 等待锁1 ──→ ?                  │   │
│  │                   ↑                      ↑               │   │
│  │              循环等待               循环等待             │   │
│  │                   └────────┴────────┘                   │   │

```

## 死锁的四个必要条件

### Coffman 条件

```
┌─────────────────────────────────────────────────────────────────┐
│                    死锁四个必要条件                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  1. 互斥条件（Mutual Exclusion）                        │   │
│  │                                                         │   │
│  │  资源只能被一个事务持有，不能共享                        │   │
│  │  例如：排他锁只能被一个事务持有                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  2. 持有并等待（Hold and Wait）                         │   │
│  │                                                         │   │
│  │  事务持有资源的同时，等待其他资源                      │   │
│  │  例如：持有锁1的同时等待锁2                            │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  3. 不可抢占（No Preemption）                           │   │
│  │                                                         │   │
│  │  资源不能被强制从持有者手中夺取                        │   │
│  │  只能等持有者主动释放                                   │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  4. 循环等待（Circular Wait）                           │   │
│  │                                                         │   │
│  │  事务之间形成循环等待关系                               │   │
│  │  T1 等待 T2，T2 等待 T3，...，Tn 等待 T1                │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  打破任一条件，即可避免死锁！                                    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 死锁的典型场景

### 场景一：两个事务，交叉锁定

```sql
-- 事务 A：
BEGIN;
SELECT * FROM users WHERE id = 1 FOR UPDATE;  -- 获取用户1的锁
-- 业务处理...
UPDATE orders SET status = '完成' WHERE user_id = 1;  -- 等待订单1的锁

-- 事务 B（在事务 A 执行 UPDATE 前）：
BEGIN;
SELECT * FROM orders WHERE user_id = 2 FOR UPDATE;  -- 获取订单2的锁
UPDATE users SET name = '新用户' WHERE id = 2;  -- 等待用户2的锁
```

```
┌─────────────────────────────────────────────────────────────────┐
│                    场景一：交叉锁定                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  事务 A                          事务 B                        │
│     │                               │                          │
│     ├─→ 获取 users.id=1 锁 ────────┼─→ 等待 users.id=2 锁    │
│     │                               │                          │
│     └─ 等待 orders.user_id=1 锁 ←───┼─ 获取 orders.user_id=2 锁│
│                                     │                          │
│                                     ▼                          │
│                                 死锁！                         │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 场景二：多个事务，顺序不一致

```sql
-- 事务 A：先锁用户1，再锁订单1
BEGIN;
SELECT * FROM users WHERE id = 1 FOR UPDATE;
SELECT * FROM orders WHERE user_id = 1 FOR UPDATE;
COMMIT;

-- 事务 B：先锁订单2，再锁用户2
BEGIN;
SELECT * FROM orders WHERE user_id = 2 FOR UPDATE;
SELECT * FROM users WHERE id = 2 FOR UPDATE;
COMMIT;

-- 事务 C：先锁用户3，再锁订单3
BEGIN;
SELECT * FROM users WHERE id = 3 FOR UPDATE;
SELECT * FROM orders WHERE user_id = 3 FOR UPDATE;
COMMIT;
```

```
┌─────────────────────────────────────────────────────────────────┐
│                    场景二：多事务顺序混乱                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  如果三个事务交错执行：                                          │
│                                                                 │
│  T1: 事务A 锁用户1                                              │
│  T2: 事务B 锁订单2                                              │
│  T3: 事务A 锁订单1（锁定）                                      │
│  T4: 事务B 锁用户2（锁定）                                      │
│  T5: 事务A 等待订单2 ←─────────────────┐                        │
│  T6: 事务B 等待用户1 ←─────────────────┼── 循环等待！           │
│  T7: 事务C 锁用户3 ──→ 锁订单3 ──→ 完成                        │
│                                          │                       │
│                                          ▼                       │
│                                      死锁！                      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 场景三：范围查询导致死锁

```sql
-- 事务 A：
BEGIN;
SELECT * FROM users WHERE age BETWEEN 20 AND 30 FOR UPDATE;
-- 锁定区间 (10, 30] 和 (30, 40]（临键锁）

-- 事务 B：
BEGIN;
SELECT * FROM users WHERE age BETWEEN 25 AND 35 FOR UPDATE;
-- 锁定区间 (20, 30] 和 (30, 40]
-- 锁定 (30, 40) 时可能与事务 A 冲突

-- 事务 C：
INSERT INTO users VALUES (28, '新用户', 28);
-- 被阻塞！
```

### 场景四：外键导致的死锁

```sql
-- 父表：users
-- 子表：orders（外键 user_id REFERENCES users.id）

-- 事务 A：
BEGIN;
DELETE FROM users WHERE id = 1;
-- 需要锁定 users.id=1，并检查 orders 表

-- 事务 B：
BEGIN;
SELECT * FROM orders WHERE user_id = 1 FOR UPDATE;
-- 锁定 orders.user_id=1

-- 死锁！外键检查时相互等待
```

### 死锁检测过程

```
┌─────────────────────────────────────────────────────────────────┐
│                    死锁检测过程                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  T1: 事务 A 等待锁 2 ←──┐                                       │
│  T2: 事务 B 等待锁 1 ←──┼───→ 环：A→B→A                        │
│      ↑                 │                                       │
│      └─────────────────┘                                       │
│      死锁！                                                    │
│                                                                 │
│  InnoDB 检测到环后：                                            │
│  1. 计算每个事务的回滚代价                                      │
│  2. 选择代价最小的事务回滚                                     │
│  3. 回滚该事务，释放其持有的锁                                  │
│  4. 其他事务可以继续执行                                        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 死锁回滚选择

```

├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  InnoDB 选择回滚代价最小的事务：                                  │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  代价评估因素：                                          │   │
│  │                                                         │   │
│  │  1. 事务持有的锁数量（越多越应回滚）                    │   │
│  │  2. 事务修改的行数（越多越应回滚）                      │   │
│  │  3. 事务的开始时间（越早越应回滚，减少资源占用）         │   │
│  │  4. 事务回滚的距离（undo log 越多越应回滚）             │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  简单说：回滚「工作量最少」的事务                                │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 死锁的处理策略

### 应用层处理

```java
// Java：死锁重试机制
public void executeWithRetry(String sql, int maxRetries) {
    for (int i = 0; i < maxRetries; i++) {
        try {
            jdbcTemplate.execute(sql);
            return;
        } catch (DeadlockException e) {
            // 死锁，等待随机时间后重试
            try {
                Thread.sleep((long) (Math.random() * 100));
            } catch (InterruptedException ie) {
                Thread.currentThread().interrupt();
            }
        }
    }
    throw new RuntimeException("Max retries exceeded");
}
```

## 死锁避免策略

### 策略一：固定顺序获取锁

```
┌─────────────────────────────────────────────────────────────────┐
│                    固定顺序获取锁                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  问题：                                                         │
│  事务 A: 锁用户1 → 锁订单1                                     │
│  事务 B: 锁订单1 → 锁用户1  ← 顺序不同，可能死锁                │
│                                                                 │
│  解决：所有事务按固定顺序获取锁                                  │
│  事务 A: 锁用户1 → 锁订单1                                     │
│  事务 B: 锁用户1 → 锁订单1  ← 顺序一致，不会死锁                │
│                                                                 │
│  原理：打破循环等待条件                                         │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

```sql
-- 业务逻辑：总是按 ID 从小到大顺序加锁
public void transfer(String fromId, String toId) {
    String firstId = fromId.compareTo(toId) < 0 ? fromId : toId;
    String secondId = fromId.compareTo(toId) < 0 ? toId : fromId;

    jdbcTemplate.execute("BEGIN");
    jdbcTemplate.execute("SELECT * FROM accounts WHERE id = ? FOR UPDATE", firstId);
    jdbcTemplate.execute("SELECT * FROM accounts WHERE id = ? FOR UPDATE", secondId);
    jdbcTemplate.execute("COMMIT");
}
```

### 策略二：减少锁持有时间

```sql
-- 不好：长时间持有锁
BEGIN;
SELECT * FROM users WHERE id = 1 FOR UPDATE;  -- 获取锁
-- 复杂的业务处理...（10秒）
UPDATE users SET name = '新张三' WHERE id = 1;
COMMIT;

-- 好：减少锁持有时间
BEGIN;
-- 先在不使用锁的情况下处理业务
SELECT * FROM users WHERE id = 1;  -- 快照读
-- 复杂的业务处理...（10秒）
UPDATE users SET name = '新张三' WHERE id = 1;  -- 最后才加锁
COMMIT;
```

### 策略三：使用索引减少锁范围

```
┌─────────────────────────────────────────────────────────────────┐
│                    索引与锁范围                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  无索引：锁定所有记录                                            │
│  SELECT * FROM users WHERE name = '张三' FOR UPDATE;            │
│  → 锁住整个表！                                                 │
│                                                                 │
│  有索引：锁定索引记录                                            │
│  SELECT * FROM users WHERE id = 1 FOR UPDATE;                   │
│  → 只锁定 id=1 的记录                                           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 策略四：避免范围锁

```sql
-- 范围查询会产生临键锁
SELECT * FROM users WHERE age > 20 FOR UPDATE;
-- 锁定 (20, +∞) 区间，可能锁住很多记录

-- 使用精确查询
SELECT * FROM users WHERE age = 25 FOR UPDATE;
-- 只锁定 age=25 的记录及其间隙
```

### 策略五：批量操作优化

```sql
-- 逐行加锁（可能死锁）
BEGIN;
UPDATE users SET balance = balance - 100 WHERE id = 1;
UPDATE users SET balance = balance - 100 WHERE id = 2;
UPDATE users SET balance = balance - 100 WHERE id = 3;
-- 多次加锁，每次都可能与其他事务冲突
COMMIT;

-- 批量加锁（更好）
BEGIN;
UPDATE users SET balance = balance - 100 WHERE id IN (1, 2, 3);
-- 一次性锁定所有需要的记录
COMMIT;
```

### 策略六：降低隔离级别

```sql
-- 可重复读（RR）会产生临键锁
SET SESSION transaction_isolation = 'READ-COMMITTED';

-- 读已提交（RC）下：
-- - 使用 MVCC 减少锁冲突
-- - 临键锁退化为记录锁
-- - 减少死锁可能性
```

## 面试实战

### 问题一：死锁的四个必要条件是什么？

**面试官问**：「死锁有哪些必要条件？如何打破死锁？」

**深入回答**：

> **死锁的四个必要条件（Coffman 条件）**：
>
> 1. **互斥**：资源只能被一个事务持有
> 2. **持有并等待**：持有资源的同时等待其他资源
> 3. **不可抢占**：资源不能被强制夺取
> 4. **循环等待**：事务之间形成循环等待
>
> **打破死锁**：打破任一条件即可：
> - 固定顺序获取锁（打破循环等待）
> - 减少锁持有时间（打破持有并等待）
> - 使用乐观锁（打破不可抢占）

### 问题二：MySQL InnoDB 是如何检测和处理死锁的？

**面试官问**：「MySQL InnoDB 是如何检测死锁的？检测到后如何处理？」

**深入回答**：

> **检测机制**：使用等待图（Wait-For Graph）
> - 将事务之间的锁等待关系建模为有向图
> - 周期性检测图中是否存在环
> - 如果存在环，说明有死锁
>
> **处理方式**：
> 1. 计算每个事务的回滚代价
> 2. 选择代价最小的事务回滚（undo log 最少）
> 3. 回滚该事务，释放其持有的锁
> 4. 返回错误：`Deadlock found when trying to get lock`
>
> **应用处理**：捕获异常，随机等待后重试

### 问题三：如何避免死锁？

**面试官问**：「在开发中如何避免死锁？」

**深入回答**：

> **避免死锁的策略**：
>
> 1. **固定顺序获取锁**：
>    - 所有事务按相同顺序访问资源
>    - 打破循环等待条件
>
> 2. **减少锁持有时间**：
>    - 延迟加锁到最后
>    - 使用快照读替代当前读
>
> 3. **使用索引**：
>    - 减少锁定的记录数
>    - 避免全表扫描导致的锁
>
> 4. **降低隔离级别**：
>    - RC 级别下锁冲突更少
>
> 5. **添加重试机制**：
>    - 应用层捕获死锁异常
>    - 等待后重试

## 总结

死锁是并发控制的经典问题：

- **四个必要条件**：互斥、持有并等待、不可抢占、循环等待
- **检测机制**：等待图检测循环
- **处理策略**：回滚代价最小的事务
- **避免策略**：固定顺序、减少锁时间、使用索引

理解死锁的产生和处理，是数据库开发和运维的必备知识。
