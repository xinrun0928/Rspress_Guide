# 两阶段锁协议与死锁避免

「为什么锁会死锁？如何避免？」

死锁是数据库并发控制的经典问题。理解两阶段锁协议和死锁避免策略，是每个 DBA 和后端工程师的必修课。

## 两阶段锁协议（2PL）

### 什么是两阶段锁？

两阶段锁协议（Two-Phase Locking，2PL）是一种并发控制协议，将事务分为两个阶段：

```
┌─────────────────────────────────────────────────────────────────┐
│                    两阶段锁协议                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  第一阶段：扩展阶段（Growing Phase）                      │   │
│  │                                                         │   │
│  │  - 可以获取锁                                          │   │
│  │  - 不能释放锁                                          │   │
│  │                                                         │   │
│  │  ┌─────────────────────────────────────────────────┐   │   │
│  │  │  获取锁 ──→ 获取锁 ──→ 获取锁 ──→ 获取锁        │   │   │
│  │  │    ↑        ↑        ↑        ↑                 │   │   │
│  │  │   可以     可以     可以     可以                │   │   │
│  │  └─────────────────────────────────────────────────┘   │   │
│  │                       │                                  │   │
│  │                       ▼ 锁数量达到最大值                  │   │
│  └─────────────────────────────────────────────────────────┘   │
│                       │                                         │
│                       ▼                                         │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  第二阶段：收缩阶段（Shrinking Phase）                  │   │
│  │                                                         │   │
│  │  - 可以释放锁                                          │   │
│  │  - 不能获取新锁                                        │   │
│  │                                                         │   │
│  │  ┌─────────────────────────────────────────────────┐   │   │
│  │  │  释放锁 ──→ 释放锁 ──→ 释放锁 ──→ 释放锁        │   │   │
│  │  │    ↓        ↓        ↓        ↓                 │   │   │
│  │  │   必须     必须     必须     必须                │   │   │
│  │  └─────────────────────────────────────────────────┘   │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 严格两阶段锁（Strict 2PL）

```
┌─────────────────────────────────────────────────────────────────┐
│                    严格两阶段锁                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  严格 2PL 的特点：                                              │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  - 事务持有排他锁直到事务结束                           │   │
│  │  - 不像普通 2PL 那样在执行过程中释放锁                   │   │
│  │  - 保证可恢复性                                        │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │   获取锁 ──→ 获取锁 ──→ 获取锁 ──→  ... ──→ COMMIT       │   │
│  │     ↑        ↑        ↑        ↑               ↓       │   │
│  │   可以     可以     可以     可以             释放全部锁  │   │
│  │                                                         │   │
│  │                   ↑                                     │   │
│  │              第一阶段          第二阶段（一次性释放）     │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  MySQL InnoDB 使用严格 2PL：                                    │
│  - 排他锁（X Lock）在事务提交/回滚 时释放                        │
│  - 共享锁（S Lock）可以提前释放                                │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### MySQL 的两阶段锁

```sql
-- MySQL InnoDB 的两阶段锁行为
BEGIN;

-- 获取锁（第一阶段）
SELECT * FROM users WHERE id = 1 FOR UPDATE;  -- 获取 X 锁
UPDATE users SET name = '新张三' WHERE id = 1;  -- 获取 X 锁

-- 在提交前，不能释放 X 锁（严格 2PL）

COMMIT;  -- 或 ROLLBACK
-- 释放所有锁（第二阶段）
```

## 死锁（Deadlock）

### 什么是死锁？

死锁是指两个或多个事务相互持有对方需要的锁，形成循环等待，无法继续执行。

```
┌─────────────────────────────────────────────────────────────────┐
│                    死锁示意                                      │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  事务 A：                                                       │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  获取锁 1（用户表 id=1）──→ 等待锁 2（订单表 id=1）    │   │
│  └─────────────────────────────────────────────────────────┘   │
│                       ↕                                        │
│  事务 B：                                                       │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  获取锁 2（订单表 id=1）──→ 等待锁 1（用户表 id=1）    │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  循环等待，死锁形成！                                           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 死锁示例

```sql
-- 事务 A：
BEGIN;
SELECT * FROM users WHERE id = 1 FOR UPDATE;  -- 锁定 id=1
UPDATE orders SET status = '已完成' WHERE user_id = 1;  -- 等待锁定 user_id=1 的记录

-- 事务 B（在事务 A 执行 UPDATE 前）：
BEGIN;
SELECT * FROM orders WHERE user_id = 1 FOR UPDATE;  -- 锁定 user_id=1 的记录
UPDATE users SET name = '新张三' WHERE id = 1;  -- 等待锁定 id=1

-- 死锁！两个事务相互等待对方释放锁
```

### 死锁的四个必要条件

```
┌─────────────────────────────────────────────────────────────────┐
│                    死锁的必要条件（ Coffman 条件）                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  1. 互斥条件                                            │   │
│  │     资源只能被一个事务持有                               │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  2. 持有并等待                                          │   │
│  │     事务持有资源的同时，等待其他资源                     │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  3. 不可抢占                                            │   │
│  │     资源不能被强制从持有者手中夺取                       │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  4. 循环等待                                            │   │
│  │     事务之间形成循环等待关系                             │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  打破任一条件，即可避免死锁！                                    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 死锁处理策略

### InnoDB 的死锁处理

```
┌─────────────────────────────────────────────────────────────────┐
│                    InnoDB 死锁处理                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  InnoDB 使用**等待图**（Wait-For Graph）检测死锁：              │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  检测机制：                                              │   │
│  │                                                         │   │
│  │  - 等待图记录事务之间的锁等待关系                       │   │
│  │  - 周期性地检测是否存在环                               │   │
│  │  - 如果检测到死锁，立即回滚最小的事务                    │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  回滚策略：                                              │   │
│  │                                                         │   │
│  │  - 回滚事务量最小的事务（undo log 最少）                 │   │
│  │  - 只回滚一个事务，其他事务可以继续                      │   │
│  │  - 返回 "Deadlock found when trying to get lock"        │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 死锁后的重试

```java
// Java 代码处理死锁
public void transfer(String fromId, String toId, BigDecimal amount) {
    int maxRetries = 3;
    for (int i = 0; i < maxRetries; i++) {
        try {
            // 执行转账逻辑
            jdbcTemplate.execute("BEGIN");
            jdbcTemplate.execute("UPDATE accounts SET balance = balance - ? WHERE id = ?", amount, fromId);
            jdbcTemplate.execute("UPDATE accounts SET balance = balance + ? WHERE id = ?", amount, toId);
            jdbcTemplate.execute("COMMIT");
            break;  // 成功，跳出重试
        } catch (DeadlockException e) {
            // 死锁，随机等待后重试
            if (i < maxRetries - 1) {
                Thread.sleep((long) (Math.random() * 100));
            }
        }
    }
}
```

## 死锁避免策略

### 策略一：固定顺序获取锁

```
┌─────────────────────────────────────────────────────────────────┐
│                    固定顺序获取锁                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  问题：                                                         │
│  事务 A: 锁用户1 → 锁订单1                                     │
│  事务 B: 锁订单1 → 锁用户1                                     │
│  → 可能死锁                                                     │
│                                                                 │
│  解决：所有事务按固定顺序获取锁                                  │
│  事务 A: 锁用户1 → 锁订单1                                     │
│  事务 B: 锁用户1 → 锁订单1  ← 顺序一致，不会死锁              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

```sql
-- 示例：按 ID 顺序加锁
BEGIN;

-- 总是先锁 ID 小的
SELECT * FROM users WHERE id <= 1 FOR UPDATE;  -- 先锁用户
SELECT * FROM orders WHERE user_id = 1 FOR UPDATE;  -- 再锁订单

COMMIT;
```

### 策略二：减少锁持有时间

```sql
-- 不好：长时间持有锁
BEGIN;
SELECT * FROM users WHERE id = 1 FOR UPDATE;  -- 获取锁
-- 业务逻辑处理...（可能很慢）
-- 用户操作...
UPDATE users SET name = '新张三' WHERE id = 1;  -- 最后才更新
COMMIT;

-- 好：快速获取锁，快速释放
BEGIN;
-- 先执行业务逻辑（不加锁）
SELECT * FROM users WHERE id = 1;  -- 快照读
-- 业务逻辑处理...
-- 最后才加锁更新
UPDATE users SET name = '新张三' WHERE id = 1;  -- 加锁
COMMIT;
```

### 策略三：降低隔离级别

```sql
-- 读已提交（RC）下，减少锁冲突
SET SESSION transaction_isolation = 'READ-COMMITTED';
```

### 策略四：使用索引

```
┌─────────────────────────────────────────────────────────────────┐
│                    使用索引减少锁范围                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  无索引：                                                       │
│  SELECT * FROM users WHERE name = '张三' FOR UPDATE;            │
│  → 锁定所有记录                                                 │
│                                                                 │
│  有索引：                                                       │
│  SELECT * FROM users WHERE id = 1 FOR UPDATE;                    │
│  → 只锁定 id=1 的记录                                           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 策略五：批量操作

```sql
-- 逐行加锁（不好）
BEGIN;
UPDATE users SET balance = balance - 100 WHERE id = 1;
UPDATE users SET balance = balance - 100 WHERE id = 2;
UPDATE users SET balance = balance - 100 WHERE id = 3;
-- 多次加锁，可能死锁
COMMIT;

-- 批量加锁（好）
BEGIN;
UPDATE users SET balance = balance - 100 WHERE id IN (1, 2, 3);
-- 一次性锁定，减少死锁可能
COMMIT;
```

## 面试实战

### 问题一：什么是两阶段锁协议？

**面试官问**：「什么是两阶段锁协议？MySQL InnoDB 使用的是哪种？」

**深入回答**：

> 两阶段锁协议（2PL）将事务分为两个阶段：
>
> 1. **扩展阶段（Growing Phase）**：只能获取锁，不能释放锁
> 2. **收缩阶段（Shrinking Phase）**：只能释放锁，不能获取新锁
>
> **MySQL InnoDB 使用严格两阶段锁**：
> - 排他锁（X Lock）在事务结束时才释放
> - 保证可恢复性和数据一致性

### 问题二：死锁是如何形成的？有哪些必要条件？

**面试官问**：「死锁是如何形成的？打破死锁的条件是什么？」

**深入回答**：

> **死锁的四个必要条件**（Coffman 条件）：
>
> 1. **互斥**：资源只能被一个事务持有
> 2. **持有并等待**：持有资源的同时等待其他资源
> 3. **不可抢占**：资源不能被强制夺取
> 4. **循环等待**：事务之间形成循环等待
>
> **打破死锁**：打破任一条件即可：
> - 按固定顺序获取锁（打破循环等待）
> - 减少锁持有时间（打破持有并等待）
> - 使用能一次获取所有锁的操作

### 问题三：MySQL 如何处理死锁？

**面试官问**：「MySQL InnoDB 是如何检测和处理死锁的？」

**深入回答**：

> **检测机制**：使用等待图（Wait-For Graph）
> - 记录事务之间的锁等待关系
> - 周期性检测是否存在环
>
> **处理方式**：
> - 检测到死锁后，立即回滚事务量最小的事务
> - 返回错误：`Deadlock found when trying to get lock`
> - 应用应该捕获异常并重试

## 总结

两阶段锁协议和死锁避免是数据库并发控制的核心：

- **两阶段锁**：扩展阶段获取锁，收缩阶段释放锁
- **严格 2PL**：排他锁持有到事务结束
- **死锁条件**：互斥、持有并等待、不可抢占、循环等待
- **避免策略**：固定顺序、减少持有时间、使用索引、批量操作
- **死锁处理**：等待图检测，回滚最小事务

理解这些概念，是掌握数据库并发控制的关键。
