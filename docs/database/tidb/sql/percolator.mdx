# Percolator 事务

想象一个场景：你有一个分布式系统，3 个节点各持有一份数据。客户端发起一笔转账，需要同时修改 3 个节点的数据。任何一步失败，3 个节点的数据必须同时回滚。

怎么做到？

这就是分布式事务的核心问题。TiDB 使用 **Percolator** 模型来解决。

---

## 一、分布式事务的问题

### 1.1 两阶段提交（2PC）

经典的分布式事务方案是 **两阶段提交（Two-Phase Commit）**：

```bash
# 阶段 1: Prepare
Coordinator -> Node-A: PREPARE
Coordinator -> Node-B: PREPARE
Coordinator -> Node-C: PREPARE
Node-A: 准备好，锁住资源，响应 YES
Node-B: 准备好，锁住资源，响应 YES
Node-C: 准备好，锁住资源，响应 YES

# 阶段 2: Commit
Coordinator -> Node-A: COMMIT
Coordinator -> Node-B: COMMIT
Coordinator -> Node-C: COMMIT
Node-A: 提交，释放锁
Node-B: 提交，释放锁
Node-C: 提交，释放锁

# 客户端收到成功响应
```

### 1.2 2PC 的问题

2PC 有两个致命问题：

| 问题 | 原因 | 影响 |
|---|---|---|
| Coordinator 单点 | Coordinator 故障导致阻塞 | 事务可能永远挂起 |
| 锁粒度大 | Prepare 阶段锁住所有资源 | 并发度低 |
| 阻塞 | 节点故障时阻塞整个事务 | 可用性差 |

### 1.3 Percolator 的思路

Google 的 Percolator 用**乐观锁 + 时间戳**解决了 2PC 的问题：

- **不要 Coordinator**：各节点自主决策
- **乐观锁**：假设冲突不多，冲突时重试
- **时间戳排序**：通过 TSO 保证事务顺序

## 二、Percolator 模型

### 2.1 核心思想

Percolator 用**三个列**存储每行数据的状态：

| 列 | 用途 |
|---|---|
| `write` | 已提交的数据指针 |
| `data` | 实际数据 |
| `lock` | 正在执行的事务的锁 |

```bash
# 单行数据的存储结构
Key: "user_1"

# write 列：指向最新已提交的数据
write: {
    start_ts: 100,      # 哪个版本
    commit_ts: 102,     # 提交时间
    data_pointer: "data:user_1:100"  # 指向 data 列
}

# data 列：实际数据
data:user_1:100 = {balance: 900}   # start_ts = 100 时的数据
data:user_1:95 = {balance: 1000}   # start_ts = 95 时的数据（历史版本）

# lock 列：锁信息
lock: {
    primary_key: "user_1",   # 主键
    start_ts: 105,           # 事务开始时间
    ttl: 3000,               # 锁超时时间
    type: PUT               # 操作类型
}
```

### 2.2 为什么需要主键？

Percolator 引入了**主键（Primary Key）**的概念：

```bash
# 转账事务：用户 A -> 用户 B
UPDATE users SET balance = balance - 100 WHERE id = 'A';  # 扣款
UPDATE users SET balance = balance + 100 WHERE id = 'B';  # 到账

# 主键 = "A" (扣款那行)
# 次键 = "B" (到账那行)

# 主键的锁特殊处理：
# - 主键锁失败，事务直接失败
# - 次键锁失败，可以回滚主键
```

<Badge text="Primary Key" type="tip" /> 主键的作用是：即使事务中途失败，也能通过主键找到并清理所有锁。

### 3.2 Prewrite 阶段

```rust
// TiDB Client: Prewrite
fn prewrite(&mut self, mutations: Vec<Mutation>) -> Result<()> {
    // 1. 随机选择一个主键
    let primary_key = mutations[0].key.clone();
    
    // 2. 逐个写入锁和数据
    for mutation in &mutations {
        // 检查是否有冲突
        if self.is_locked(mutation.key) {
            return Err(WriteConflict);  // 乐观锁冲突
        }
        
        // 写入 data 列
        let data_key = format!("data:{}:{}", mutation.key, self.start_ts);
        self.txn.put(data_key, mutation.value);
        
        // 写入 lock 列（主键优先）
        let lock_type = if mutation.key == primary_key {
            LockType::Primary
        } else {
            LockType::Secondary
        };
        self.txn.put_lock(mutation.key, Lock {
            primary_key: primary_key.clone(),
            start_ts: self.start_ts,
            lock_type,
            ttl: 3000,
        });
    }
    
    // 3. 检查 Primary Lock 是否写入成功
    if !self.check_primary_lock() {
        return Err(PrimaryNotReady);
    }
    
    Ok(())
}
```

```bash
# Prewrite 示例
# 事务：UPDATE users SET balance = balance - 100 WHERE id = 'A'

Prewrite 请求：
  primary_key = "user_A"
  start_ts = 100
  mutations = [
    {key: "user_A", value: {balance: 900}},
    {key: "user_B", value: {balance: 1100}}
  ]

# TiKV 处理 user_A（主键）
TiKV: 写入 data:user_A:100 = {balance: 900}
TiKV: 写入 lock:user_A = {primary: "user_A", start_ts: 100, type: Primary, ttl: 3000}

# TiKV 处理 user_B（次键）
TiKV: 写入 data:user_B:100 = {balance: 1100}
TiKV: 写入 lock:user_B = {primary: "user_A", start_ts: 100, type: Secondary, ttl: 3000}

# 所有 Prewrite 成功，继续 Commit
```

### 3.3 Commit 阶段

```rust
// TiDB Client: Commit
fn commit(&mut self) -> Result<u64> {
    // 1. 获取 commit_ts（必须大于 start_ts）
    let commit_ts = self.pd.get_timestamp()?;
    
    // 2. 先提交主键
    self.commit_key(&self.primary_key, self.start_ts, commit_ts)?;
    
    // 3. 再批量提交次键
    for key in &self.secondary_keys {
        self.commit_key(key, self.start_ts, commit_ts)?;
    }
    
    Ok(commit_ts)
}

fn commit_key(&self, key: &[u8], start_ts: u64, commit_ts: u64) -> Result<()> {
    // 1. 检查锁是否存在
    let lock = self.get_lock(key)?;
    if lock.start_ts != start_ts {
        return Err(LockNotFound);
    }
    
    // 2. 删除锁
    self.txn.delete_lock(key)?;
    
    // 3. 写入 write 列（指向 data）
    let write_key = format!("write:{}:{}", key, commit_ts);
    self.txn.put(write_key, WriteRecord {
        start_ts: start_ts,
        data_pointer: format!("data:{}:{}", key, start_ts),
    });
    
    Ok(())
}
```

```bash
# Commit 示例
commit_ts = 102

# 提交主键 user_A
TiKV: 删除 lock:user_A
TiKV: 写入 write:user_A:102 = {start_ts: 100, data: "data:user_A:100"}

# 提交次键 user_B
TiKV: 删除 lock:user_B
TiKV: 写入 write:user_B:102 = {start_ts: 100, data: "data:user_B:100"}

# 事务提交成功
# 客户端收到成功响应
```

## 四、读取数据

### 4.1 读取流程

```rust
// TiDB Client: 读取数据
fn get(&self, key: &[u8], start_ts: u64) -> Result<Option<Vec<u8>>> {
    // 1. 检查是否有锁
    let lock = self.get_lock(key)?;
    if lock.is_some() {
        // 有锁，需要等待或回滚
        if lock.start_ts < start_ts {
            // 乐观锁冲突
            return Err(WriteConflict);
        } else {
            // 等待锁释放
            self.wait_for_lock(lock);
        }
    }
    
    // 2. 从 write 列找到 start_ts 对应的版本
    let write = self.get_write(key, start_ts)?;
    if write.is_none() {
        return Ok(None);  // 数据不存在
    }
    
    // 3. 根据 write 指向的 data 读取实际数据
    let data_key = write.data_pointer;
    let data = self.get_data(&data_key)?;
    
    Ok(Some(data))
}
```

### 4.2 读取版本选择

```bash
# 读取 key = "user_A"，start_ts = 101

# Step 1: 查找 write 列
write:user_A:102 = {start_ts: 100}  # start_ts 102，指向 100
write:user_A:95 = {start_ts: 90}    # start_ts 95，指向 90

# 选择 <= start_ts 的最新版本
# 102 > 101，不符合
# 95 <= 101，符合
# 选择 write:user_A:95

# Step 2: 根据 write 找到 data
data:user_A:90 = {balance: 1000}

# 返回 {balance: 1000}
```

## 五、冲突处理

### 5.1 写写冲突

```bash
# 事务 T1: start_ts = 100
UPDATE users SET balance = 900 WHERE id = 'A';

# 事务 T2: start_ts = 101
UPDATE users SET balance = 800 WHERE id = 'A';

# T1 先 Prewrite 成功，锁住 user_A

# T2 执行 Prewrite
TiKV: 发现 lock:user_A 存在
TiKV: 检查 lock.start_ts = 100 < 101
TiKV: 返回 WriteConflict

# T2 需要回滚或重试
```

### 5.2 读写冲突

```bash
# 事务 T1: start_ts = 100
UPDATE users SET balance = 900 WHERE id = 'A';

# T1 处于 Prewrite 阶段，锁住 user_A

# 事务 T2: start_ts = 101
SELECT * FROM users WHERE id = 'A';  # 读取

# T2 执行 get
TiKV: 发现 lock:user_A 存在
TiKV: lock.start_ts = 100 < 101（锁是旧事务的）
TiKV: 返回 WriteConflict 给 T2

# T2 需要重试（会被 TiDB 自动重试）
```

### 5.3 锁超时清理

```rust
// TiKV: 锁超时检测
fn check_lock_ttl(&self) {
    // 定时检查锁是否超时
    for (key, lock) in self.scan_locks() {
        if lock.is_expired() {
            // 锁超时，可以清理
            self.rollback_lock(key, lock);
        }
    }
}

// 回滚锁
fn rollback_lock(&self, key: &[u8], lock: &Lock) -> Result<()> {
    // 1. 删除锁
    self.delete_lock(key)?;
    
    // 2. 删除 data 列
    let data_key = format!("data:{}:{}", key, lock.start_ts);
    self.delete_data(&data_key)?;
    
    // 3. 写入 Rollback 记录
    let write_key = format!("write:{}:{}", key, current_ts());
    self.put(write_key, WriteRecord {
        start_ts: lock.start_ts,
        write_type: WriteType::Rollback,
    });
    
    Ok(())
}
```

## 六、TiDB 的 Percolator 实现

### 6.1 TiKV 中的存储

```bash
# TiKV 的 MVCC Key 结构
Key: {user_key}_{start_ts}
Value: {data}

# write 列
Key: "user_A"_commit_ts
Value: {start_ts, write_type}

# lock 列
Key: "user_A"
Value: {primary_key, start_ts, lock_type, ttl}
```

### 6.2 Percolator 与 TiDB 事务整合

```rust
// TiDB 的事务管理层
struct Txn {
    start_ts: u64,
    commit_ts: u64,
    mutations: Vec<Mutation>,
}

impl Txn {
    fn execute(&mut self) -> Result<()> {
        // 1. Prewrite
        self.prewrite()?;
        
        // 2. 获取 commit_ts
        self.commit_ts = self.pd.get_timestamp()?;
        
        // 3. Commit
        self.commit()
    }
    
    fn prewrite(&mut self) -> Result<()> {
        // 按 key 分组到不同 Region
        let region_groups = self.group_by_region();
        
        // 并行 Prewrite
        for (region, mutations) in region_groups {
            let leader = self.pd.get_leader(region);
            self.tikv.prewrite(leader, mutations, self.start_ts)?;
        }
        
        Ok(())
    }
    
    fn commit(&mut self) -> Result<()> {
        // 先提交主键
        let primary_region = self.pd.get_region(&self.primary_key);
        self.tikv.commit_primary(primary_region, self.primary_key, self.start_ts, self.commit_ts)?;
        
        // 并行提交次键
        for key in &self.secondary_keys {
            let region = self.pd.get_region(key);
            self.tikv.commit_secondary(region, key, self.start_ts, self.commit_ts)?;
        }
        
        Ok(())
    }
}
```

## 七、性能优化

### 7.1 减少事务大小

```sql
-- 建议：大事务拆分
BEGIN;
-- 一次事务处理 1000 行
INSERT INTO orders VALUES (...);  -- 1000 次
COMMIT;

-- 更好的做法：分批提交
for batch in batches_of(1000) {
    BEGIN;
    INSERT INTO orders VALUES batch;
    COMMIT;
}
```

### 7.2 避免热点

```sql
-- 问题：自增主键导致写入热点
CREATE TABLE orders (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,  -- 热点
    ...
);

-- 解决：使用 AUTO_RANDOM
CREATE TABLE orders (
    id BIGINT AUTO_RANDOM PRIMARY KEY,  -- 打散热点
    ...
);

-- 或使用 SHARD_ROW_ID_BITS
CREATE TABLE orders (
    id BIGINT SHARD_ROW_ID_BITS(4) PRIMARY KEY,  -- 打散
    ...
);
```

### 7.3 乐观锁重试

```sql
-- TiDB 默认使用乐观锁
SET tidb_txn_mode = 'optimistic';

-- 如果冲突多，切换悲观锁
SET tidb_txn_mode = 'pessimistic';

-- 悲观锁在写入前就加锁，减少冲突
BEGIN PESSIMISTIC;
SELECT * FROM users WHERE id = 1 FOR UPDATE;  -- 加锁
UPDATE users SET balance = 900 WHERE id = 1;
COMMIT;
```

## 总结

Percolator 是 TiDB 实现分布式事务的核心：

- **无 Coordinator**：各节点自主决策，避免单点
- **乐观锁**：假设冲突不多，冲突时重试
- **时间戳排序**：TSO 保证事务顺序
- **Primary Key**：事务恢复的锚点
- **两阶段提交**：Prewrite + Commit

理解 Percolator，是理解 TiDB 事务模型的关键。
