# Raft 协议实现

想象一下，你正在和三个朋友玩一个游戏：每个人手里有一本一模一样的账本，每次转账都要四个人同时记录。任何一个人说谎，另外三个人会立刻发现。

这就是 Raft 协议的核心思想——**通过多数派投票保证数据一致性**。

## 一、Raft 协议基础

### 1.1 什么是 Raft？

Raft 是一种分布式共识算法，2014 年由 Diego Ongaro 和 John Ousterhout 提出。相比 Paxos，Raft 的目标是**更容易理解和实现**。

<Badge text="Raft" type="tip" /> Raft 的核心是「领导者选举 + 日志复制」。集群中有一个 Leader 负责处理所有写请求，Follower 同步 Leader 的日志。

### 1.2 三种角色

Raft 集群中的节点有三种角色：

| 角色 | 职责 | 生命周期 |
|---|---|---|
| Leader | 处理所有写请求 | 同一时间最多一个 |
| Follower | 接收 Leader 的日志 | 多个 |
| Candidate | 选举时的候选者 | 临时状态 |

```bash
# 正常状态
[Leader] ←→ [Follower] ←→ [Follower]

# Leader 故障后，Follower 发起选举
[Leader]  ← [故障]
[Follower] ← [Candidate] ← [发起投票]
[Follower] ←              ← [投票给 Candidate]
```

### 1.3 术语解释

| 术语 | 说明 |
|---|---|
| Term | 任期，递增的整数，用于判断节点状态 |
| Log Entry | 日志条目，包含 command 和 term |
| Commit | 已提交，日志被多数派确认 |
| Apply | 已应用，状态机执行了这条日志 |

## 二、Leader 选举

### 2.1 选举触发条件

Follower 在以下情况会变成 Candidate：

- **Election Timeout**：长时间没收到 Leader 的心跳
- **网络分区**：与 Leader 失去联系

```bash
# Election Timeout（随机 150-300ms）
Follower-1: 超时了，发起选举！  [Term 5]
Follower-2: 还没超时，继续等
Follower-3: 也没超时，继续等
```

<Badge text="随机超时" type="tip" /> 随机化 Election Timeout 是 Raft 的巧妙设计，避免多个 Follower 同时发起选举导致票数分散。

### 2.2 选举过程

```bash
# Step 1: Follower -> Candidate
Follower-1: 我变成 Candidate，Term = 5，投自己一票

# Step 2: 发送 RequestVote 到其他节点
Candidate-1 -> Follower-2: 请求投票，Term 5，lastLogIndex 100
Candidate-1 -> Follower-3: 请求投票，Term 5，lastLogIndex 100

# Step 3: Follower 响应
Follower-2: 同意，你的日志比我新，投你！
Follower-3: 同意，投你！

# Step 4: 获得多数派票数
Candidate-1 获得 2 票（自己 + 1 + 1 = 3票），成为 Leader
```

### 2.3 投票规则

节点在以下情况会投票给 Candidate：

1. 当前 term < Candidate 的 term
2. 还没投过票
3. Candidate 的日志比自己的新

```rust
// 投票判断逻辑
fn can_vote_for(&self, candidate: &Candidate) -> bool {
    // 1. term 有效
    if candidate.term < self.current_term {
        return false;
    }
    
    // 2. 没投过票
    if self.voted_for.is_some() && self.voted_for != Some(candidate.id) {
        return false;
    }
    
    // 3. 日志至少一样新
    if candidate.last_log_term < self.last_log_term {
        return false;
    }
    if candidate.last_log_term == self.last_log_term 
       && candidate.last_log_index < self.last_log_index {
        return false;
    }
    
    true
}
```

<Badge text="日志新者优先" type="warning" /> 「日志新者优先」保证新 Leader 包含所有已提交的日志，避免丢失数据。

### 2.4 脑裂问题处理

网络分区可能导致出现两个 Leader：

```bash
# 原始状态
[A] [B] [C] [D] [E]    # A 是 Leader

# 网络分区 [A,B] vs [C,D,E]
[A,B]   [C,D,E]         # C,D,E 无法收到 A 的心跳
[A]     [C]             # C 成为新 Leader（获得 C,D,E 的票）
```

当分区合并时：

```bash
# 发现更高 term
[A] 发现自己的 term 过期了，切换为 Follower
[A] 同步 C 的日志
```

## 三、日志复制

### 3.1 写入流程

当客户端发送写请求时：

```bash
# Step 1: 请求发到 Leader
Client -> Leader: SET key = "value"

# Step 2: Leader 写入本地日志
Leader: 追加 Log Entry，状态 = Uncommitted

# Step 3: 发送 AppendEntries 到 Follower
Leader -> Follower-1: AppendEntries(entry)
Leader -> Follower-2: AppendEntries(entry)

# Step 4: 多数派确认
Follower-1: 写入日志，响应 OK
Follower-2: 写入日志，响应 OK

# Step 5: Leader 提交
Leader: 提交日志（Commit），Apply 到状态机
Leader -> Client: OK
```

### 3.2 日志一致性

Raft 通过 **AppendEntries** 的一致性检查保证日志一致：

```rust
// AppendEntries RPC 参数
struct AppendEntriesArgs {
    term: u64,           // Leader 的 term
    prev_log_index: u64, // 前一条日志的索引
    prev_log_term: u64,  // 前一条日志的 term
    entries: Vec<Entry>,  // 要复制的日志
    leader_commit: u64,   // Leader 已提交的索引
}

// Follower 的一致性检查
fn append_entries(&mut self, args: &AppendEntriesArgs) -> bool {
    // 1. Term 检查
    if args.term < self.current_term {
        return false;  // Leader 过期
    }
    
    // 2. 日志连续性检查
    if self.log.get(args.prev_log_index).term != args.prev_log_term {
        return false;  // 日志不一致，需要同步
    }
    
    // 3. 追加日志
    self.log.append(&args.entries);
    
    // 4. 更新 commit index
    self.commit_index = args.leader_commit;
    true
}
```

<Badge text="一致性检查" type="tip" /> 如果 Follower 的日志和 Leader 不一致，Leader 会回退 prevLogIndex 重试，直到找到匹配的日志。

### 3.3 日志提交

只有被**多数派确认**的日志才能提交：

```bash
# 3 副本集群
Leader + Follower-1 + Follower-2

# 情况 1：2 副本确认（不算多数派）
Leader: 收到 Follower-1 的 ACK
日志状态: Uncommitted ❌

# 情况 2：3 副本确认（多数派）
Leader: 收到 Follower-1 + Follower-2 的 ACK
日志状态: Committed ✅
```

## 四、Raft 在 TiKV 中的实现

### 4.1 Multi-Raft 架构

与 etcd 单 Raft Group 不同，TiKV 使用 **Multi-Raft** 架构——每个 Region 有独立的 Raft Group：

```bash
# TiKV 集群
Store-1: [Region 1, Region 4, Region 7]
Store-2: [Region 2, Region 5, Region 8]
Store-3: [Region 3, Region 6, Region 9]

# 每个 Region 是独立的 Raft Group
Region 1: [Leader@Store-1, Follower@Store-2, Follower@Store-3]
Region 2: [Leader@Store-2, Follower@Store-1, Follower@Store-3]
...
```

### 4.2 读写请求处理

```rust
// TiKV 读取流程
fn handle_read(&mut self, key: &[u8]) -> Result<Vec<u8>> {
    // 1. 确保自己是 Leader
    if !self.is_leader() {
        return Err(NotLeader);  // 重定向到 Leader
    }
    
    // 2. 检查 Lease
    if self.lease.is_expired() {
        // 使用 ReadIndex 协议
        self.send_read_index();
    }
    
    // 3. 读取本地数据
    self.kv_get(key)
}

// TiKV 写入流程
fn handle_write(&mut self, key: &[u8], value: &[u8]) -> Result<()> {
    // 1. 构造 Entry
    let entry = Entry {
        data: encode_write(key, value),
        index: self.next_index(),
        term: self.current_term(),
    };
    
    // 2. 追加到本地日志
    self.raft_log.append(&entry);
    
    // 3. 复制到 Follower
    self.replicate_to_followers(&entry);
    
    // 4. 等待多数派确认
    self.wait_for_majority(entry.index)
}
```

### 4.3 Leader Transfer

TiKV 支持 **Leader Transfer**，将 Leader 转移到其他节点：

```bash
# 场景：Store-1 负载高，想把 Region 1 的 Leader 转走
Region 1: [Leader@Store-1, Follower@Store-2, Follower@Store-3]

# Step 1: Leader 发起转移
Leader@Store-1 -> Follower@Store-2: RequestLeaderTransfer

# Step 2: 等待 Follower 日志追上
Follower@Store-2: 同步日志...

# Step 3: 成为新 Leader
Follower@Store-2 -> New Leader@Store-2

Region 1: [Follower@Store-1, Leader@Store-2, Follower@Store-3]
```

## 五、故障恢复

### 5.1 Leader 故障

```bash
# Step 1: Leader 故障
[Leader]  ← [故障]
[Follower-1]  [Follower-2]

# Step 2: Election Timeout
Follower-1: 超时了，发起选举
Follower-2: 也超时了（比 Follower-1 晚一点），发起选举

# Step 3: 选举结果
Follower-1 获得票数多（2票），成为新 Leader

# Step 3: 恢复服务
[Leader@Follower-1]  [Follower-2]
旧 Leader 恢复后，变成 Follower
```

### 5.2 Follower 故障

Follower 故障时，Leader 会持续重试：

```bash
# Follower-2 故障
[Leader] -> [Follower-2]: AppendEntries (超时)
[Leader] -> [Follower-2]: AppendEntries (超时)
...

# Follower-2 恢复
[Leader] -> [Follower-2]: AppendEntries (心跳)
Follower-2: 同步缺失的日志
```

### 5.3 多数派不可用

当多数派不可用时，写入会阻塞：

```bash
# 3 副本，2 节点故障
[Leader]  [故障]  [故障]
Write: ❌ 无法获得多数派确认，写入阻塞

Read: 可以读（如果使用 Lease Read 或 ReadIndex）
```

## 六、性能优化

### 6.1 Pipeline 优化

批量发送 AppendEntries，减少网络往返：

```rust
// TiKV 的 Pipeline 发送
fn replicate_async(&mut self, entries: Vec<Entry>) {
    // 批量发送
    self.rpc.send(AppendEntriesRequest {
        entries: entries,  // 一批日志
        ...
    });
}
```

### 6.2 Batch 和 Split

```rust
// TiKV 配置
tikv:
  raftstore:
    raft-entry-max-size: 32MB      # 单个 Entry 最大 32MB
    raft-base-tick-interval: 2s   # 心跳间隔
    raft-heartbeat-ticks: 10      # 10 次心跳后选举
```

### 6.3 Read Index

对于只读请求，可以跳过日志复制：

```rust
// ReadIndex 流程
fn read_index(&mut self, key: &[u8]) -> Vec<u8> {
    // 1. 发送 ReadIndex 到 Leader
    self.send_read_index();
    
    // 2. 等待确认自己是 Leader
    self.wait_confirm();
    
    // 3. 读取数据
    self.kv_get(key)
}
```

## 总结

Raft 协议是 TiKV 保证数据一致性的核心：

- **Leader 选举**：通过随机 Election Timeout + 多数派投票选出 Leader
- **日志复制**：写入先到 Leader，通过 AppendEntries 复制到 Follower，多数派确认后提交
- **一致性保证**：日志新者优先，不会丢失已提交数据
- **Multi-Raft**：每个 Region 独立的 Raft Group，灵活调度

理解 Raft，是理解 TiKV「如何保证分布式一致性」的关键。
