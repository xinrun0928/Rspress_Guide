# TiKV 与 TiDB 协作

你可能已经知道 TiDB 有四个组件：TiDB Server、TiKV、PD、TiFlash。但你可能不知道它们之间是怎么「对话」的。

这一节，我们来看看 TiDB Server 如何与 TiKV 协作，完成一条 SQL 的执行。

## 一、架构分层

### 1.1 两层架构

```
┌─────────────────────────────────────┐
│           TiDB Server               │
│  ┌─────────┐ ┌─────────┐ ┌───────┐ │
│  │ Parser  │ │Planner  │ │Exec   │ │
│  └─────────┘ └─────────┘ └───────┘ │
└─────────────────────────────────────┘
              ↓ gRPC
┌─────────────────────────────────────┐
│             TiKV Cluster            │
│  ┌─────────┐ ┌─────────┐ ┌───────┐ │
│  │ Store 1  │ │ Store 2  │ │Store 3│ │
│  │ Region 1│ │ Region 2 │ │Region3│ │
│  └─────────┘ └─────────┘ └───────┘ │
└─────────────────────────────────────┘
```

- **TiDB Server**：计算层，无状态
- **TiKV**：存储层，有状态

### 1.2 数据路由

```rust
// TiDB Server 如何知道数据在哪？
fn route_to_tikv(&self, key: &[u8]) -> &Store {
    // 1. 从 PD 获取 Region 信息
    let region = self.pd.get_region(key);
    
    // 2. 找到 Leader 所在的 Store
    let leader_store = self.store_manager.get(region.leader.store_id);
    
    // 3. 发送请求
    leader_store.send_rpc(...)
}
```

## 二、SQL 执行流程

### 2.1 完整流程

```sql
SELECT * FROM users WHERE id = 1;
```

执行流程：

```bash
# Step 1: 连接
Client -> TiDB Server: 连接 4000 端口

# Step 2: SQL 解析
TiDB Server: Parser -> AST -> SelectStmt

# Step 3: 编译计划
TiDB Server: Planner -> PhysicalPlan
  TableScan: {table: users, index: PRIMARY, range: [1, 1]}
  Output: {columns: [id, name, email]}

# Step 4: 请求 TiKV
TiDB Server -> TiKV: KvGet {key: "t{users}_r{?}_k{1}", start_ts: 100}

# Step 5: TiKV 响应
TiKV -> TiDB Server: {id: 1, name: "张三", email: "zhang@example.com"}

# Step 6: 返回结果
TiDB Server -> Client: 结果集
```

### 2.2 分布式执行

```sql
-- 涉及多个 Region 的查询
SELECT * FROM users WHERE id IN (1, 2, 3, 4, 5);
```

```bash
# Step 1: 定位 Region
TiDB Server -> PD: 获取 key=1 的 Region
TiDB Server -> PD: 获取 key=2 的 Region
...

# 假设结果：
# id=1 -> Region 5 (Leader@Store-1)
# id=2 -> Region 5 (Leader@Store-1)
# id=3 -> Region 7 (Leader@Store-2)
# id=4 -> Region 9 (Leader@Store-3)
# id=5 -> Region 5 (Leader@Store-1)

# Step 2: 并行请求
TiDB Server -> Store-1: BatchGet keys=[1,2,5]
TiDB Server -> Store-2: BatchGet keys=[3]
TiDB Server -> Store-3: BatchGet keys=[4]

# Step 3: 聚合结果
TiDB Server: 合并来自 3 个 Store 的结果
```

## 三、写入流程

### 3.1 单行写入

```sql
INSERT INTO users (name, email) VALUES ('张三', 'zhang@example.com');
```

```bash
# Step 1: SQL 解析
TiDB Server: 解析 INSERT，生成写入计划

# Step 2: 分配主键（如果没有）
TiDB Server: 分配 start_ts = 100

# Step 3: 路由到 Leader
TiDB Server -> TiKV Leader: Prewrite
{
  key: "t{users}_r{5}_k{100}",
  value: {name: "张三", email: "zhang@example.com"},
  start_ts: 100,
  primary_key: "t{users}_r{5}_k{100}"
}

# Step 4: Raft 复制
TiKV Leader -> TiKV Followers: AppendEntries
多数派确认后 -> Commit

# Step 5: 提交
TiDB Server -> TiKV: Commit
{
  key: "t{users}_r{5}_k{100}",
  start_ts: 100,
  commit_ts: 101
}

# Step 6: 返回成功
TiDB Server -> Client: 1 row affected
```

### 3.2 批量写入

```sql
INSERT INTO users (name, email) VALUES 
  ('张三', 'zhang@example.com'),
  ('李四', 'li@example.com'),
  ('王五', 'wang@example.com');
```

```bash
# TiDB 会将数据按照 Region 分组
# 每个 Region 的数据作为独立的子事务

# Group 1: Region 5
Prewrite -> Commit

# Group 2: Region 7
Prewrite -> Commit

# Group 3: Region 9
Prewrite -> Commit

# 任意一个 Group 失败，整个 INSERT 回滚
```

## 四、事务管理

### 4.1 Percolator 两阶段提交

```rust
// TiDB 的两阶段提交
struct TwoPhaseCommitter {
    keys: Vec<Key>,
    start_ts: u64,
}

impl TwoPhaseCommitter {
    // Phase 1: Prewrite
    fn prewrite(&mut self) -> Result<()> {
        // 1. 获取所有涉及 Region 的 Leader
        let regions = self.pd.get_regions(&self.keys);
        
        // 2. 并行 Prewrite
        for (region, keys) in regions {
            let leader = self.pd.get_leader(region);
            self.tikv.prewrite(leader, keys, self.start_ts)?;
        }
        
        // 3. 如果任何一个 Prewrite 失败，回滚
        Ok(())
    }
    
    // Phase 2: Commit
    fn commit(&mut self) -> Result<u64> {
        // 1. 获取 commit_ts
        let commit_ts = self.pd.get_timestamp()?;
        
        // 2. 并行 Commit
        for (region, keys) in regions {
            let leader = self.pd.get_leader(region);
            self.tikv.commit(leader, keys, self.start_ts, commit_ts)?;
        }
        
        Ok(commit_ts)
    }
}
```

### 4.2 事务冲突检测

```rust
// TiKV 的锁检查
fn prewrite(&mut self, key: &[u8], value: &[u8], start_ts: u64) -> Result<()> {
    // 1. 检查 Lock CF
    let lock = self.get_lock(key)?;
    
    if lock.is_some() {
        // 存在锁，检查是否冲突
        let existing_lock = lock.unwrap();
        
        if existing_lock.start_ts < start_ts {
            // 乐观锁冲突
            return Err(WriteConflict {
                start_ts: start_ts,
                conflict_ts: existing_lock.start_ts,
            });
        }
    }
    
    // 2. 写入新锁
    self.put_lock(key, Lock { start_ts, ... })?;
    Ok(())
}
```

## 五、查询优化

### 5.1 下推计算

```sql
-- TiDB 会尽量把计算下推到 TiKV
SELECT COUNT(*) FROM users WHERE age > 25;

-- 优化前：TiDB 读取所有数据再计算
TiDB: SELECT * FROM users  # 读取 1000 行
TiDB: COUNT(*) WHERE age > 25  # 计算

-- 优化后：下推聚合到 TiKV
TiDB -> TiKV: Count(age > 25)
TiKV -> TiDB: 350
TiDB: 返回 350
```

### 5.2 下推的算子

| 算子 | 能否下推 |
|---|---|
| Selection (`WHERE`) | ✅ |
| Projection (`SELECT`) | ✅ |
| Aggregation (`GROUP BY`) | ✅ |
| Limit (`LIMIT`) | ✅ |
| TopN (`ORDER BY LIMIT`) | ✅ |
| Join | 部分 |
| Sort | 部分 |

```rust
// TiDB 的下推规则
fn push_down(&self, plan: &PhysicalPlan) -> PhysicalPlan {
    match plan {
        Selection(pred) => {
            // 下推 Selection
            self.push_selection_to_tikv(pred)
        }
        Aggregation(agg) => {
            if self.can_push_agg(agg) {
                self.push_agg_to_tikv(agg)
            } else {
                plan.clone()
            }
        }
        ...
    }
}
```

## 六、错误处理

### 6.1 常见错误

| 错误类型 | 原因 | 处理方式 |
|---|---|---|
| Region Not Found | Region 分裂/合并 | 重试，刷新 Region 缓存 |
| Not Leader | Leader 迁移 | 重试，发送到新 Leader |
| Stale Epoch | Region 版本过期 | 重试，刷新 Region 信息 |
| Key Not In Region | Key 路由错误 | 重试，重新定位 Region |
| Write Conflict | 事务冲突 | 重试或回滚 |

### 6.2 重试机制

```rust
// TiDB 的 Region 重试
fn execute_with_retry(&mut self, req: &Request) -> Result<Response> {
    let mut attempts = 0;
    loop {
        match self.send_to_tikv(req) {
            Ok(resp) => return Ok(resp),
            Err(e) => {
                attempts += 1;
                if attempts >= MAX_RETRIES {
                    return Err(e);
                }
                if e.is_retryable() {
                    self.refresh_region_cache();
                    sleep(backoff(attempts));
                } else {
                    return Err(e);
                }
            }
        }
    }
}
```

## 总结

TiDB Server 和 TiKV 的协作是 TiDB 分布式执行的核心：

- **路由层**：TiDB Server 通过 PD 获取 Region 元数据，定位 Leader
- **执行层**：SQL 下推到 TiKV，减少网络传输
- **事务层**：Percolator 两阶段提交保证 ACID
- **重试层**：自动处理 Region 迁移和临时错误

理解这个协作流程，是理解 TiDB 如何在分布式环境下工作的关键。
