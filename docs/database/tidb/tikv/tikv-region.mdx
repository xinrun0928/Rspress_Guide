# TiKV 存储结构：Region 与 RocksDB

「TiKV 的数据是怎么存储的？Region 是什么？」

理解 TiKV 的存储结构，是掌握 TiDB 分布式存储能力的关键。大多数人知道 TiKV 使用 RocksDB 和 Raft，但追问下去——「Region 是怎么划分的？」「Raft Group 和 Region 是什么关系？」「RocksDB 的 LSM 树是怎么工作的？」「Leader 是怎么调度到合适的节点的？」——能答上来的就很少了。

本文深入讲解 TiKV 的存储结构，帮你彻底理解这个分布式 KV 存储引擎。

## Region 概念

### 什么是 Region

Region 是 TiKV 中**数据分布的基本单元**，每个 Region 存储一段连续的 Key 数据。

```
Region 示意：

整个 Key Space:
┌──────────┬──────────┬──────────┬──────────┐
│ Key-A    │ Key-F    │ Key-M    │ Key-Z    │
└──────────┴──────────┴──────────┴──────────┘
    Region1      Region2     Region3      Region4

Region 定义：
Region 1: [Key-A, Key-F)  →  存储 Key-A ~ Key-E
Region 2: [Key-F, Key-M)  →  存储 Key-F ~ Key-L
Region 3: [Key-M, Key-Z)  →  存储 Key-M ~ Key-Y
Region 4: [Key-Z, Key-Z+] →  存储 Key-Z 及以后
```

### Region 的结构

```rust
// Region 核心结构
struct Region {
    id: u64,
    start_key: Vec<u8>,      // 起始 Key（包含）
    end_key: Vec<u8>,        // 结束 Key（不包含）
    peers: Vec<Peer>,         // 副本节点列表
    leader_peer_id: u64,      // Leader 的 Peer ID
    region_epoch: RegionEpoch,  // Region 版本号
}

// RegionEpoch 用于版本控制
struct RegionEpoch {
    conf_ver: u64,   // 配置版本，成员变更时递增
    version: u64,    // 分裂/合并版本，分裂/合并时递增
}
```

### Region 副本

```rust
// Region 副本结构
struct Peer {
    id: u64,            // Peer 唯一 ID
    store_id: u64,      // 所在的 Store ID
    is_learner: bool,   // 是否为 Learner 节点
}

// 示例：3 副本 Region
Region {
    id: 1,
    start_key: "a",
    end_key: "m",
    peers: [
        Peer { id: 101, store_id: 1, is_learner: false },  // Leader
        Peer { id: 102, store_id: 2, is_learner: false },  // Follower
        Peer { id: 103, store_id: 3, is_learner: false },  // Follower
    ],
    leader_peer_id: 101
}
```

## Region 与 Raft Group

### Region 作为 Raft Group

在 TiKV 中，每个 Region 都有一个对应的 Raft Group，Region 的副本之间通过 Raft 协议保持一致。

```
Region 与 Raft Group 的关系：

┌──────────────────────────────────────────────────────┐
│                 Region 1 [a, m)                        │
│                                                        │
│   ┌─────────────────────────────────────────────┐    │
│   │          Raft Group                         │    │
│   │                                             │    │
│   │  Leader ──── replicate ──── Follower         │    │
│   │    (Store 1)                 (Store 2)       │    │
│   │        │                     │               │    │
│   │        └─────── replicate ────┘               │    │
│   │                     (Store 3)                │    │
│   │                                             │    │
│   └─────────────────────────────────────────────┘    │
│                                                        │
│   ┌─────────────────────────────────────────────┐    │
│   │          RocksDB                             │    │
│   │  ┌──────────┐ ┌──────────┐ ┌──────────┐    │    │
│   │  │  Store 1 │ │  Store 2 │ │  Store 3 │    │    │
│   │  │ Region 1 │ │ Region 1 │ │ Region 1 │    │    │
│   │  │ (Leader) │ │(Follower)│ │(Follower)│    │    │
│   │  └──────────┘ └──────────┘ └──────────┘    │    │
│   └─────────────────────────────────────────────┘    │
└──────────────────────────────────────────────────────┘
```

### Raft 日志复制

```rust
// Raft 日志复制流程

// 1. Client 发送写入请求到 Leader
// 2. Leader 写入本地日志
// 3. Leader 发送 AppendEntries 到 Followers
// 4. Followers 写入本地日志并响应
// 5. Leader 收到多数节点确认后 Apply
// 6. Leader 返回 Client 成功

// 示例
struct RaftLog {
    term: u64,      // 任期号
    index: u64,      // 日志索引
    entries: Vec<Entry>,
}

struct Entry {
    index: u64,
    term: u64,
    data: Vec<u8>,  // 实际数据
}
```

## RocksDB 存储

### RocksDB 概述

RocksDB 是 TiKV 的底层存储引擎，基于 LSM-Tree（Log-Structured Merge-tree）数据结构，提供高性能的键值存储。

```
LSM-Tree 结构：

┌─────────────────────────────────────────────────────┐
│                    LSM-Tree                         │
│                                                       │
│   MemTable ────（内存）─── 写入 ──── WritableBuffer │
│       │                                              │
│       ▼ [满了就刷盘]                                 │
│   L0 ───────────────  Immutable MemTable 刷盘        │
│       │                                              │
│       ▼ [compaction]                               │
│   L1 ───────────────  第一个持久化层                  │
│       │                                              │
│       ▼ [compaction]                               │
│   L2 ───────────────  更大的 SST 文件               │
│       │                                              │
│       ▼                                             │
│   L3 ───────────────  更大的 SST 文件               │
│       ...                                            │
└─────────────────────────────────────────────────────┘

写入路径：
数据 ──→ MemTable（内存）──→ 刷盘 ──→ L0 ──→ Compaction ──→ L1, L2, L3...

读取路径：
内存查找 ──→ L0 ──→ L1 ──→ L2 ──→ L3...（找到即返回）
```

### RocksDB 列族

TiKV 使用多个 RocksDB 列族来存储不同类型的数据。

```rust
// TiKV RocksDB 列族

// 1. default 列族：存储业务数据
// Key: table_prefix + row_key
// Value: 行数据

// 2. lock 列族：存储事务锁信息
// Key: primary_key
// Value: lock info (事务信息)

// 3. write 列族：存储 MVCC 版本信息
// Key: write_cf_key
// Value: write data

// 4. raft 列族：存储 Raft 日志
// Key: raft_log_key
// Value: raft log entry

// 列族配置
rocksdb:
  default:
    compression: snappy
    block_size: 64KB
  lock:
    max_write_buffer_number: 8
  write:
    max_write_buffer_number: 8
  raft:
    format: v2
```

### RocksDB 配置

```toml
# tikv.toml

[storage]
data-dir = "/tikv-data"

[rocksdb]
# 压缩算法
compression-per-level = ["no", "no", "snappy", "snappy", "snappy", "zstd", "zstd"]

# Block Cache
block-cache-size = "10GB"

# 写入缓冲
max-write-buffer-number = 5
min-write-buffer-number-to-merge = 1

# Compaction
max-compaction-rate = "100MB"
max-sub-compactions = 3
```

## 数据读写流程

### 写入流程

```rust
// TiKV 写入流程（两阶段提交）

// 阶段 1：Prewrite
// 1. 获取当前 TSO（时间戳）
// 2. 写入 lock CF（乐观锁）
// 3. 写入 primary lock
// 4. 写入 data CF（所有副本）

// 阶段 2：Commit
// 1. 写入 commit_ts 到 write CF
// 2. 删除 lock CF 中的锁
// 3. 返回客户端成功

// 示例
// 写入 Key = "user:1", Value = "张三"
async fn write(ctx: WriteCtx) -> Result<()> {
    let (primary_key, secondaries) = ctx.get_keys();

    // Prewrite
    let mut mutations = Vec::new();
    for key in &secondaries {
        mutations.push(Mutation::Put { key, value: "张三" });
    }

    // 发送给所有副本
    for peer in region.peers() {
        send_prewrite(peer, mutations.clone()).await?;
    }

    // Commit
    let commit_ts = pd.get_ts();
    for key in &secondaries {
        write_to_write_cf(key, commit_ts)?;
        delete_from_lock_cf(key)?;
    }

    Ok(())
}
```

### 读取流程

```rust
// TiKV 读取流程（MVCC）

// 1. 获取当前 TSO
// 2. 构造 search key（start_ts）
// 3. 遍历 write CF 找到最新版本
// 4. 根据 write 信息读取 data CF

async fn read(key: &[u8], start_ts: u64) -> Result<Option<Vec<u8>>> {
    // 1. 获取 TSO
    let ts = pd.get_current_ts();

    // 2. 构造 search key: user:1_ts
    let search_key = encode_key(key, ts);

    // 3. 在 write CF 中查找
    let write = find_in_write_cf(search_key)?;

    // 4. 根据 write 信息读取 data
    let value = find_in_default_cf(write.short_value())?;

    Ok(value)
}

// MVCC 版本链
// user:1_lock    → [key: "user:1", start_ts: 100, type: Lock]
// user:1_100     → [key: "user:1", type: Put, short_value: "张三"]
// user:1_80      → [key: "user:1", type: Delete, short_value: ""]
```

## Region 分裂与合并

### Region 分裂

<Badge text="重要" type="warning" /> 当 Region 的大小或_keys 数量超过阈值时，会触发分裂。

```rust
// Region 分裂触发条件

// 默认配置：
// - size: 96MB（Region 大小阈值）
// - keys: 144000（Key 数量阈值）
// - rows: 1440000（行数阈值）

// 分裂过程：

// 1. 检测到 Region 超过阈值
// 2. Leader 发起 Admin Request（SPLIT）
// 3. 构造新 Region 的元信息
// 4. 在所有副本上创建新 Region
// 5. 更新 PD 元数据

// 示例：Region [a, z) 分裂为 [a, m) 和 [m, z)
Before Split:
┌────────────────────────────────────────┐
│  Region [a, z)                         │
│  size: 150MB > 96MB                    │
│  keys: 200000                          │
└────────────────────────────────────────┘

After Split:
┌─────────────────┐  ┌─────────────────┐
│  Region [a, m)   │  │  Region [m, z)  │
│  size: 75MB      │  │  size: 75MB     │
│  keys: 100000    │  │  keys: 100000   │
└─────────────────┘  └─────────────────┘
```

## PD 调度

### PD 的作用

PD（Placement Driver）是 TiKV 集群的调度器，负责管理 Region 的分布和调度。

```
PD 调度流程：

┌─────────────────────────────────────────────────────┐
│                     PD Cluster                       │
│                                                       │
│   1. 元数据管理                                       │
│      - 记录 Region 分布                               │
│      - 记录 Store 状态                                │
│      - 记录 Leader/Follower 位置                      │
│                                                       │
│   2. 调度决策                                         │
│      - 均衡 Region 分布                               │
│      - 处理热点 Region                                │
│      - 管理副本放置                                   │
│                                                       │
│   3. TSO 分配                                        │
│      - 分布式时间戳分配                                │
│      - 保证事务顺序                                    │
│                                                       │
└─────────────────────────────────────────────────────┘
         │                              ▲
         │  调度命令                      │ 状态上报
         ▼                              │
┌─────────────────────────────────────────────────────┐
│                    TiKV Cluster                      │
│                                                       │
│   Store 1 ─── Region ─── Leader                      │
│   Store 2 ─── Region ─── Follower                    │
│   Store 3 ─── Region ─── Follower                    │
│                                                       │
└─────────────────────────────────────────────────────┘
```

### 调度策略

```rust
// 常见的调度策略

// 1. 负载均衡
// 将 Region 均匀分布到各个 Store
ScheduleStrategy::BalanceRegion { target: StoreId }

// 2. 热点调度
// 分裂热点 Region，分散到多个 Store
ScheduleStrategy::SplitHotRegion { region_id: u64 }

// 3. 副本调度
// 添加/移除 Follower
ScheduleStrategy::AddPeer { region_id: u64, store_id: StoreId }

// 4. Leader 调度
// 将 Leader 调度到负载低的节点
ScheduleStrategy::TransferLeader { region_id: u64, from: PeerId, to: PeerId }

// 5. 故障处理
// 故障节点的 Region 迁移到新节点
ScheduleStrategy::ReplacePeer { region_id: u64, ... }
```

## 面试实战

**面试官问**：「Region 和 Raft Group 是什么关系？」

**深入回答**：

> 核心关系：
>
> 1. **一对一对应**：每个 Region 都有一个对应的 Raft Group
>
> 2. **数据副本**：Region 的多个副本组成 Raft Group
>
> 3. **一致性保证**：通过 Raft 协议保证 Region 数据一致
>
> 4. **副本选举**：Raft Group 选举产生 Region Leader

**面试官问**：「TiKV 和 HBase 的 Region 有什么区别？」

**深入回答**：

> 主要区别：
>
> 1. **副本实现**：TiKV 用 Raft 协议，HBase 用 Zookeeper 协调
>
> 2. **数据存储**：TiKV 用 RocksDB（LSM-Tree），HBase 用 HDFS
>
> 3. **事务支持**：TiKV 原生支持分布式事务，HBase 需要用协处理器
>
> 4. **调度方式**：TiKV 用 PD 调度，HBase 用 HMaster

**面试官问**：「RocksDB 的 LSM-Tree 和 B+ 树有什么区别？」

**深入回答**：

> 核心区别：
>
> 1. **写入方式**：LSM-Tree 写性能好（顺序写），B+ 树有随机写
>
> 2. **读取方式**：B+ 树读取快（一次定位），LSM-Tree 可能需要多层查找
>
> 3. **空间放大**：LSM-Tree 有 compaction，空间放大率高
>
> 4. **适用场景**：LSM-Tree 适合写多读少，B+ 树适合读多写少

## 总结

TiKV 存储结构要点：

- **Region**：数据分布的基本单元，默认 96MB
- **Raft Group**：Region 的副本组成，通过 Raft 协议保持一致
- **RocksDB**：底层存储引擎，使用 LSM-Tree 结构
- **列族**：default、lock、write、raft 分工明确
- **MVCC**：通过 write CF 实现多版本并发控制
- **PD 调度**：管理 Region 分布和副本放置

理解 TiKV 的存储结构，是掌握 TiDB 分布式存储能力的关键。
