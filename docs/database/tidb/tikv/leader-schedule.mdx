# Leader 调度

分布式存储的一个核心问题是：**数据副本如何均匀分布到各个节点？如何处理热点？如何保证高可用？**

PD 的调度系统就是来解决这些问题的。它像是一个不知疲倦的运维大脑，24 小时监控着集群状态，自动做出调度决策。

## 一、PD 调度概述

### 1.1 调度的目标

PD 调度的核心目标是：

| 目标 | 说明 |
|---|---|
| 负载均衡 | Region 在节点间均匀分布 |
| 高可用 | 副本分布在不同机房/机架 |
| 性能优化 | 热点打散、Leader 均衡 |
| 容量规划 | 预留空间、提前扩容提示 |

### 1.2 调度的组件

```bash
# PD 的调度组件
PD Server
├── Scheduler         # 调度策略组件
│   ├── Balance-Leader-Scheduler
│   ├── Balance-Region-Scheduler
│   ├── Hot-Region-Scheduler
│   ├── Merge-Merge-Scheduler
│   └── Grant-Leader-Scheduler
├── Cluster-Worker    # 执行调度的 worker
└── Heartbeat-Stores  # 接收 TiKV 心跳
```

## 二、调度规则

### 2.1 Leader Count 调度

目标是让 Leader 在节点间均匀分布：

```bash
# 初始状态（不均衡）
Store-1: 500 Leaders  ← 负载高
Store-2: 300 Leaders
Store-3: 200 Leaders

# 调度后（均衡）
Store-1: 333 Leaders
Store-2: 333 Leaders
Store-3: 333 Leaders
```

<Badge text="Leader 调度" type="tip" /> Leader 调度比 Region 调度快，因为只需要传输元数据，不迁移数据。

### 2.2 Region Count 调度

目标是让数据量在节点间均匀分布：

```bash
# 初始状态
Store-1: 500 Regions, size=800GB  ← 接近容量上限
Store-2: 300 Regions, size=500GB
Store-3: 200 Regions, size=400GB

# 调度后
Store-1: 400 Regions, size=650GB
Store-2: 350 Regions, size=550GB
Store-3: 250 Regions, size=450GB
```

### 2.3 热点调度

检测并处理热点 Region：

```bash
# 热点检测
Hot-Region-Scheduler:
  Region 123: read_flow=1000MB/s  ← 热点
  Region 456: read_flow=800MB/s   ← 热点
  Region 789: read_flow=100MB/s  ← 正常

# 调度策略
1. 分裂热点 Region
2. 将分裂后的 Region 调度到不同节点
3. 转移 Leader 到空闲节点
```

### 2.4 亲和性调度

通过 Label 控制副本分布：

```json
{
  "location-labels": ["zone", "rack", "host"],
  "replication": {
    "max-replicas": 3
  }
}
```

```bash
# Zone 分布配置
Store-1: zone=zone-a, rack=rack-1
Store-2: zone=zone-a, rack=rack-2
Store-3: zone=zone-b, rack=rack-1

# PD 会尽量把 3 副本分布在不同 zone
# 理想情况：zone-a(2副本) + zone-b(1副本)
```

## 四、调度流程

### 4.1 心跳上报

```bash
# TiKV 定期向 PD 上报心跳
TiKV -> PD: RegionHeartbeat
{
  region_id: 123,
  leader: {store_id: 1},
  peers: [{store_id: 1}, {store_id: 2}, {store_id: 3}],
  down_peers: [],
  pending_peers: [],
  stats: {
    region_key: 500000,
    region_size: 95000000,  # 95MB
    read_bytes: 1000000,
    write_bytes: 500000,
  }
}
```

### 4.2 调度决策

```rust
// PD 的调度决策流程
fn on_heartbeat(&mut self, region: &RegionHeartbeat) {
    // 1. 更新 Region 元数据
    self.update_region_meta(region);
    
    // 2. 检查是否需要调度
    if self.should_schedule(region) {
        // 3. 选择调度器
        let scheduler = self.select_scheduler(region);
        
        // 4. 生成调度操作
        let op = scheduler.generate_op(region);
        
        // 5. 添加到 operator queue
        self.add_operator(op);
    }
}

fn should_schedule(&self, region: &Region) -> bool {
    // 检查是否违反约束
    if self.is_unbalanced() { return true; }
    if self.is_hot_region() { return true; }
    if self.is_pending() { return true; }
    false
}
```

### 4.3 Operator 执行

```bash
# Operator 生命周期
1. PD 生成 Operator
   Operator: {region_id: 123, steps: [transfer-leader, move-peer]}

2. PD 将 Operator 下发给 Leader
   PD -> Leader@TiKV: AddOperator

3. TiKV 执行 Operator
   Leader@TiKV: 执行 transfer-leader -> move-peer

4. TiKV 回报结果
   TiKV -> PD: OperatorDone

5. PD 清理 Operator
   PD: RemoveOperator
```

## 五、热点处理

### 5.1 热点检测算法

```rust
// 热点检测（Rolling Windows）
struct HotRegionStats {
    rolling: [u64; 5],  // 5 个时间窗口
    window_size: 60,   // 每个窗口 60 秒
    
    fn get_hot_score(&self) -> f64 {
        // 流量 * 占比
        let total = self.rolling.iter().sum::<u64>();
        let avg = total as f64 / self.window_size as f64;
        avg * self.hot_threshold_ratio
    }
}
```

### 5.2 热点 Region 处理

```bash
# 情况 1：单个 Region 写入热点
# 策略：分裂 Region

Before: [t1_r1_k000: t1_r1_kfff]  # 1个热点 Region
After:  [t1_r1_k000: t1_r1_k7ff] + [t1_r1_k800: t1_r1_kfff]
# 分裂后打散到不同节点

# 情况 2：多个小 Region 热点
# 策略：合并后再分裂

Before: [t1_r1_k000: t1_r1_k100] + [t1_r2_k000: t1_r2_k100]  # 2个热点
After:  [t1_r1_k000: t1_r2_k100]  # 合并
# 合并后均匀分裂
```

## 六、故障处理

### 6.1 节点下线

```bash
# TiKV 节点故障后，PD 开始调度
Store-3 故障:
  PD: 检测到 Store-3 不可达
  PD: 等待 max-store-down-time (默认 30 分钟)
  PD: 开始调度，转移 Region 副本
  Region 123: [S1, S2, S3] -> [S1, S2, S4]
  Region 456: [S1, S3, S5] -> [S1, S4, S5]
  ...
  PD: 添加新副本，补充到 max-replicas
```

### 6.2 副本不足处理

```bash
# 副本不足（3副本变成2副本）
Region 123: peers=[{store: 1}, {store: 2}]  # store-3 故障
PD: 检测到副本不足
PD: 添加新副本到 store-4
Region 123: peers=[{store: 1}, {store: 2}, {store: 4}]  # 已恢复
```

### 6.3 Leader 不可用处理

```bash
# Leader 故障
Region 123: leader=Store-1  ← 故障
PD: 检测到 Leader 心跳超时
PD: 发起 Leader 选举
Follower-2: 当选新 Leader
Region 123: leader=Store-2  ← 已恢复
```

## 七、调优实践

### 7.1 常见问题

| 问题 | 原因 | 解决方案 |
|---|---|---|
| 调度慢 | 调度并发太低 | 提高 region-schedule-limit |
| 数据不均衡 | 节点容量差异大 | 调整 store-weight |
| 热点严重 | 写入模式问题 | 使用 SHARD_ROW_ID_BITS |
| Leader 倾斜 | 调度策略问题 | 启用 balance-leader-scheduler |

### 7.2 调优建议

```bash
# 调整调度并发
pd-ctl config set region-schedule-limit 8

# 调整热点调度
pd-ctl config set hot-region-schedule-limit 4
pd-ctl config set hot-region-cache-hits-threshold 1

# 调整 Leader 权重
pd-ctl store weight 1 1.0  # Store-1 leader-weight = 1.0
pd-ctl store weight 2 1.0  # Store-2 leader-weight = 1.0
```

## 总结

PD 的 Leader 调度是 TiDB 高可用的保障：

- **调度目标**：负载均衡、高可用、性能优化
- **调度规则**：Leader Count、Region Count、热点、亲和性
- **调度流程**：心跳上报 → 决策 → Operator 执行
- **故障处理**：自动补副本、自动选主

理解 PD 调度，是理解 TiDB 如何保证集群稳定运行的关键。
