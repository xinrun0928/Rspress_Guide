# TiDB 备份恢复：BR、Dumpling、Loader 全解析

数据库备份是数据安全的最后一道防线。

一次误操作、一个 bug、一次硬件故障——如果没有备份，后果不堪设想。但备份不只是"把数据存一份"，还需要考虑：**备份速度、恢复时间、数据一致性、增量能力**。

TiDB 提供了三把备份恢复利器：**BR**（Backup & Restore）、**Dumpling**（逻辑导出）、**Loader**（逻辑导入）。每把利器有不同的用武之地。

## BR：光速备份与恢复

### BR 是什么

BR（Backup & Restore）是 TiDB 官方的物理备份工具，直接备份 TiKV 的 SST 文件，性能极高：

```bash
# 全量备份
br backup full \
    --pd "192.168.1.10:2379" \
    --storage "local:///backup/snapshot-20240101" \
    --ratelimit 128 \
    --concurrency 32
```

| 指标 | BR | Dumpling |
|---|---|---|
| 备份速度 | GB/分钟级 | MB/秒级 |
| 恢复速度 | GB/分钟级 | MB/秒级 |
| 备份类型 | 物理备份 | 逻辑备份 |
| 数据格式 | SST 文件 | SQL/CSV |
| 增量备份 | 支持（配合 CDC） | 支持 |

### BR 工作原理

BR 直接读取 TiKV 的 RocksDB SST 文件，绕过 SQL 层：

```
┌─────────────────────────────────────────────────────────┐
│                    BR Backup 流程                        │
│                                                         │
│  BR Client                                              │
│     │                                                   │
│     │ 1. 获取 Region 列表和 Leader 信息                  │
│     ▼                                                   │
│  PD Cluster                                             │
│     │                                                   │
│     │ 2. 扫描 Region，向 Leader 发送备份请求              │
│     ▼                                                   │
│  TiKV Leader                                            │
│     │                                                   │
│     │ 3. 读取 RocksDB SST 文件                          │
│     ▼                                                   │
│  RocksDB                                                │
│     │                                                   │
│     │ 4. 流式传输 SST 文件到备份存储                     │
│     ▼                                                   │
│  S3/NFS/HDFS                                            │
└─────────────────────────────────────────────────────────┘
```

### BR 备份命令详解

```bash
# 全量备份（推荐用于日常备份）
br backup full \
    --pd "${PD_ADDR}" \
    --storage "s3://my-bucket/tidb-backup/2024-01-01" \
    --s3.endpoint "https://s3.cn-north-1.jdcloud-oss.com" \
    --ratelimit 100 \           # 限速 100 MB/s/节点
    --concurrency 32 \           # 每个节点并发备份 32 个 Region
    --checksum true             # 备份后校验数据一致性

# 快照备份（基于某个时间点）
br backup full \
    --pd "${PD_ADDR}" \
    --storage "local:///backup/snapshot-202401011200" \
    --backupts "2024-01-01 12:00:00"  # 恢复到指定时间点

# 查看备份进度
br backup full \
    --pd "${PD_ADDR}" \
    --storage "local:///backup/snapshot-20240101" \
    --status-addr "0.0.0.0:8000"
```

### BR 恢复命令详解

```bash
# 全量恢复
br restore full \
    --pd "${PD_ADDR}" \
    --storage "s3://my-bucket/tidb-backup/2024-01-01" \
    --concurrency 32

# 指定时间点恢复（PITR）
br restore full \
    --pd "${PD_ADDR}" \
    --storage "s3://my-bucket/tidb-backup/2024-01-01" \
    --restore-ts "2024-01-01 15:30:00"

# 查看恢复进度
br restore full \
    --pd "${PD_ADDR}" \
    --storage "s3://my-bucket/tidb-backup/2024-01-01" \
    --status-addr "0.0.0.0:8001"
```

### BR 表级别备份与恢复

```bash
# 备份单个表
br backup table \
    --pd "${PD_ADDR}" \
    --storage "local:///backup/tables" \
    --db "app_db" \
    --table "orders"

# 恢复单个表
br restore table \
    --pd "${PD_ADDR}" \
    --storage "local:///backup/tables" \
    --db "app_db" \
    --table "orders"
```

## Dumpling：逻辑导出的瑞士军刀

### Dumpling 是什么

Dumpling 是 TiDB 的逻辑导出工具，将数据导出为 SQL 或 CSV 格式：

```bash
# 安装 Dumpling
tiup install dumpling

# 基本导出
tiup dumpling -h 192.168.1.10 -P 4000 -u root -p 'password' \
    -o "/backup/dumpling" \
    -F "256MiB"          # 每文件 256MB
```

**导出格式**：

```
/backup/dumpling/
├── app_db.orders.000000000.sql
├── app_db.orders.000000001.sql
├── app_db.users.000000000.sql
└── metadata             # 导出元信息（时间点、位点）
```

### Dumpling 高级用法

```bash
# 导出指定数据库和表
tiup dumpling \
    -h ${TIDB_HOST} -P 4000 -u root \
    -o "/backup/partial" \
    --filter 'app_db.orders' \
    --filter 'app_db.users'

# 导出条件过滤
tiup dumpling \
    -h ${TIDB_HOST} -P 4000 -u root \
    -o "/backup/filtered" \
    --sql "SELECT * FROM orders WHERE created_at >= '2024-01-01'"

# 导出为 CSV（适合大数据量）
tiup dumpling \
    -h ${TIDB_HOST} -P 4000 -u root \
    -o "/backup/csv" \
    --filetype csv \
    --csv-separator ',' \
    --csv-delimiter '"'

# 并行导出（提高速度）
tiup dumpling \
    -h ${TIDB_HOST} -P 4000 -u root \
    -o "/backup/parallel" \
    --threads 32 \           # 并行线程数
    -F "512MiB"               # 更大的文件块
```

### Dumpling 与 TiDB 特性

```bash
# 导出大数据表时，TiDB 会自动处理
# 1. 自动识别 TiFlash 副本
tiup dumpling \
    --snapshot "2024-01-01 12:00:00"  # 基于时间点导出

# 2. 自动识别全局唯一 ID（TiDB 生成的 _tidb_rowid）
# Dumpling 会正确处理，不会丢失关联

# 3. 导出集群的全局设置
# 会导出 AUTO_INCREMENT, CHARSET, COLLATE 等集群级别配置
```

## Loader：逻辑导入的利器

### Loader 是什么

Loader 是 TiDB 的逻辑导入工具，将 Dumpling 导出的 SQL/CSV 文件导入到 TiDB：

```bash
# 安装 Loader
tiup install loader

# 基本导入
tiup loader \
    -h 192.168.1.10 -P 4000 -u root -p 'password' \
    -d "/backup/dumpling" \
    -worker-count 32 \       # 并行导入 worker 数
    -batch 1000              # 每批插入的行数
```

### Loader 配置参数

```bash
# 高性能导入配置
tiup loader \
    -h ${TIDB_HOST} -P 4000 -u root \
    -d "/backup/dumpling" \
    --worker-count 64 \       # 64 个并发 worker
    --batch 10000 \           # 每批 10000 行
    --max-insert-bytes 33554432 \  # 每条 INSERT 最大 32MB
    --tidb-compact-dup-lookup false  # 禁用去重查找（加速导入）

# 导入前准备
# 1. 关闭自动统计信息收集（导入后统一 ANALYZE）
tiup loader \
    ... \
    --disable-analytics

# 2. 设置导入 session 变量
tiup loader \
    ... \
    --sql-mode "STRICT_TRANS_TABLES,NO_ENGINE_SUBSTITUTION"
```

### TiDB Lightning：更快的物理导入

对于超大数据量，**TiDB Lightning** 比 Loader 更快：

```bash
# TiDB Lightning 导入
tiup tidb-lightning \
    --tidb-host 192.168.1.10 \
    --pd-urls 192.168.1.11:2379 \
    --backend tidb \          # backend: tidb (逻辑导入) 或 local (物理导入)
    --data-source-dir /backup/dumpling \
    --tidb-port 4000 \
    --checkpoint.driver mysql

# 物理导入（最快，但需要独占导入）
tiup tidb-lightning \
    --backend local \
    --data-source-dir /backup/my_datasource \
    --pd-urls 192.168.1.11:2379 \
    --sorted-kv-dir /tmp/sorted_kv \
    -- tidb-port 4000
```

| 导入工具 | 适用场景 | 速度 |
|---|---|---|
| Loader | 小数据量（< 100GB）| 较慢 |
| TiDB Lightning (tidb backend) | 中等数据量 | 中等 |
| TiDB Lightning (local backend) | 超大数据量（TB 级）| 极快 |

## 备份策略设计

### 推荐备份方案

```bash
# 每日全量 BR 备份（快照）
0 2 * * * br backup full \
    --pd "${PD_ADDR}" \
    --storage "s3://bucket/daily/$(date +\%Y\%m\%d)" \
    --ratelimit 200 \
    --concurrency 32

# 保留策略：保留最近 7 天每日备份
# 配合生命周期管理自动清理
```

### 增量备份方案

```bash
# 配合 TiCDC 实现增量备份
# 1. 创建 CDC 同步任务到外部存储
tiup cdc cli changefeed create \
    --sink-uri "pulsar://127.0.0.1:6650" \
    --start-ts 435635554636886017

# 2. BR 快照 + CDC 增量 = PITR
# 快照恢复到某个时间点，然后 CDC 增量追到目标时间点
```

### 恢复演练

```bash
# 定期进行恢复演练（建议每月一次）
# 1. 恢复到测试环境
br restore full \
    --pd "${TEST_PD_ADDR}" \
    --storage "s3://bucket/daily/20240101" \
    --filter 'app_db.orders' \
    --filter 'app_db.users'

# 2. 验证数据一致性
tiup dumpling \
    --source "mysql://root:password@192.168.1.10:3306/app_db" \
    --tables-list app_db.orders \
    | tiup loader \
        --host ${TEST_TIDB_HOST} --port 4000 \
        -worker-count 16

# 3. 对比记录数
SELECT COUNT(*) FROM app_db.orders;  # 源库和目标库对比
```

## 备份恢复最佳实践

| 场景 | 推荐方案 |
|---|---|
| 每日例行备份 | BR 全量备份 + CRC32 校验 |
| 迁移数据到新集群 | Dumpling 导出 + Loader 导入 |
| 超大数据量迁移 | TiDB Lightning local backend |
| 时间点恢复 | BR 快照 + CDC 增量追 |
| 表级别恢复 | BR table restore |
| 跨云迁移 | Dumpling -> S3 -> Loader |

## 总结

BR、Dumpling、Loader 各有所长：

- **BR**：物理备份，速度最快，适合日常备份和快速恢复
- **Dumpling**：逻辑导出，格式通用，适合迁移和跨平台导出
- **Loader**：逻辑导入，配合 Dumpling 使用

一个完整的备份恢复策略应该包含：**定时全量备份** + **增量 CDC 同步** + **定期恢复演练**。有备无患，方能睡个安稳觉。
