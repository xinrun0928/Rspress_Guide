# PD 调度组件：Region 调度与负载均衡

想象一个图书馆：最初只有 10 本书，放一个书架就够了。但随着藏书增加到 1 万本、10 万本，你需要多个书架，还要考虑：

- 哪些书架放哪类书？
- 新书来了放哪个书架？
- 某个书架太满了怎么办？

**PD 就是这个图书馆的"管理员"**。它管理着 TiKV 集群中所有数据分片（Region）的分布，决定数据去哪里、如何均衡、什么时候该迁移。

## Region：数据分片的核心概念

### 什么是 Region

Region 是 TiKV 中数据管理的最小单位。每个 Region 包含一段连续的 Key 空间：

```bash
# 查看 Region 详情
pd-ctl region 233

# 输出示例
{
    "id": 233,
    "start_key": "t\x00\x00\x00\x00\x00\x00\x00\xf8",  # 十六进制编码
    "end_key": "t\x00\x00\x00\x00\x00\x00\x00\xf9",
    "region_epoch": {
        "conf_ver": 5,
        "version": 3
    },
    "peers": [
        {"id": 1001, "store_id": 1, "role": "leader"},
        {"id": 1002, "store_id": 2, "role": "follower"},
        {"id": 1003, "store_id": 3, "role": "follower"}
    ],
    "leader": {"id": 1001, "store_id": 1}
}
```

| 字段 | 说明 |
|---|---|
| `start_key` / `end_key` | Region 覆盖的 Key 范围 |
| `peers` | Region 的所有副本 |
| `leader` | 当前 Raft Leader，负责处理读写请求 |
| `region_epoch` | 用于检测 Region 分裂/合并的版本号 |

### Region 大小控制

默认配置下，一个 Region 的大小约为 **96MB**（可配置 `region-split-size`）。当 Region 超过这个阈值，PD 会触发分裂：

```toml
# tikv.yml 配置
[coprocessor]
region-split-size = "96MiB"

[raftstore]
region-split-check-diff = "8MiB"  # 超过 96MB+8MB 触发检查
```

**分裂过程**：

```
分裂前 Region [a, z)：
┌────────────────────────────────────┐
│          Region [a, z)             │
│              96MB                   │
└────────────────────────────────────┘

分裂后：
┌─────────────────┬──────────────────┐
│   Region [a, m)  │  Region [m, z)  │
│       48MB       │      48MB        │
└─────────────────┴──────────────────┘
```

## Region 调度：Split 与 Merge

### Split：分裂机制

当 Region 超过阈值，TiKV 会自动分裂：

```bash
# 手动触发 Region 分裂（用于测试）
curl -X POST "http://${tikv_ip}:${tikv_port}/raft/region/233/split"
```

分裂触发的时机：

| 触发条件 | 说明 |
|---|---|
| Region 大小超过 `region-split-size` + `region-split-check-diff` | 容量触发 |
| Region 键数量超过 `region-max-keys` | 数量触发 |
| 手动调用 Split 接口 | 运维操作 |

### Merge：合并机制

当 Region 变小（数据删除）或分布过于碎片化时，PD 会发起合并：

```bash
# 查看待合并的 Region
pd-ctl operator show merge

# 查看调度计划
pd-ctl operator show
```

**合并的约束条件**：

1. 相邻的 Region 才能合并
2. 两个 Region 都必须是稳定的（不是刚分裂出来的）
3. 合并后大小不能超过 `region-split-size` 的 1/4

```toml
# 合并相关配置
[schedule]
region-max-merge-size = "20MiB"      # 超过此大小不参与合并
region-min-size = "8MiB"             # Region 最小值，用于判断是否可以合并
min-region-to-be-merged-size = "8MiB" # 小于此大小的 Region 优先合并
```

## 负载均衡：Leader 与 Region 分布

### 什么是 Leader 调度

在 Raft 协议中，**写请求必须经过 Leader**，所以 Leader 的分布直接影响写入性能：

```bash
# 查看各 Store 的 Leader 分布
pd-ctl store 1 | grep leader_count
pd-ctl store 2 | grep leader_count
pd-ctl store 3 | grep leader_count
```

**问题场景**：如果所有 Leader 都集中在一个节点，即使数据均匀分布，写入也会打爆单机。

**解决方案**：PD 会将 Leader 均衡到各个节点：

```toml
# 调整 Leader 调度权重
[schedule]
leader-schedule-limit = 4      # 同时进行的 Leader 调度数
hot-region-schedule-limit = 4  # 热点 Region 调度数
```

### 什么是 Region 调度

Region 调度是数据副本的迁移，用于均衡存储容量：

```bash
# 查看 Region 在各 Store 的分布
pd-ctl region --jq='.regions[] | {id: .id, store_ids: [.store_id]}'

# 查看 Store 容量
pd-ctl store 1
```

**调度触发条件**：

| 触发条件 | 说明 |
|---|---|
| Store 容量不均衡 | 某节点存储使用率超过平均值的 110% |
| Store 上线/下线 | 新增节点或节点失效时重新分布 |
| 副本策略变更 | 修改 `max-replicas` 时调整分布 |

### 调度参数调优

```toml
# pd.yml 配置示例
[schedule]
# 调度器配置
max-snapshot-count = 3              # 单个 Store 最多同时接收的快照数
max-pending-peer-count = 16          # 单个 Store 最多同时接收的 Region 迁移数

# 调度限制
region-schedule-limit = 2048         # 同时进行的 Region 调度数
replica-schedule-limit = 64         # 同时进行的副本调度数

# 均衡策略
balance-region-size-name = "size"   # 按 Region 数量均衡
balance-leader-schedule-name = "leader"  # 按 Leader 数量均衡
```

## 热点调度：解决性能瓶颈

### 热点问题识别

热点（Hot Spot）是分布式数据库的噩梦。PD 内置了热点分析：

```bash
# 查看热点 Region
pd-ctl region --jq='.regions[] | select(.leader.rate > 1000)'

# 查看热点 Store
pd-ctl store --jq='.stores[] | {id: .store.id, hot: .hot}'

# 查看热力图
# 访问 http://${pd_ip}:2379/dashboard 查看可视化热点分析
```

### 热点调度策略

PD 提供了多种热点调度策略：

| 策略 | 说明 | 适用场景 |
|---|---|---|
| `hot-region-scheduler` | 将热点 Region 的 Leader 分散 | 写入热点集中在少量 Key |
| `shuffle-hot-region-scheduler` | 随机打散热点 Region | 临时缓解热点 |
| `transfer-peer-scheduler` | 迁移整个 Region | 存储热点 |

```bash
# 查看启用的调度器
pd-ctl config show all | grep scheduler

# 启用热点调度
pd-ctl config set hot-region-schedule-limit 4

# 调整热点阈值
pd-ctl config set hot-region-cache-hits-threshold 3  # 连续 3 个采样周期为热点
```

### 热点问题解决方案

**方案一：热点 Key 拆分**

如果热点集中在少量 Key，可以拆分 Key：

```sql
-- 原始设计：所有用户数据存在同一 Key 范围
SELECT * FROM user_profile WHERE user_id = 12345;

-- 优化：将用户数据按 Hash 分片到不同 Region
SELECT * FROM user_profile WHERE (user_id % 16) = 3;
```

**方案二：增加副本数**

增加热点 Region 的副本数可以分摊负载：

```bash
# 调整热点 Region 的副本数
pd-ctl config set max-replicas 5  # 原来 3 副本改为 5 副本
```

**方案三：手动调度**

```bash
# 手动将 Leader 迁移到指定节点
pd-ctl operator add transfer-leader region 233 target-store 2
```

## 调度策略实战配置

### OLTP 场景优化

```toml
[schedule]
# OLTP 场景：关注 Leader 均衡
leader-schedule-weight = 10
region-schedule-weight = 5
max-snapshot-count = 4
max-pending-peer-count = 32

# 禁止自动 merge（OLTP 场景 Region 不应该变小）
merge-schedule-limit = 0
```

### OLAP 场景优化

```toml
[schedule]
# OLAP 场景：关注 Region 分布和容量
leader-schedule-weight = 5
region-schedule-weight = 10
max-snapshot-count = 8

# 允许 merge 保持合理 Region 数量
merge-schedule-limit = 8
region-min-size = "16MiB"
```

## 总结

PD 的调度系统是 TiDB 自动化的核心：

- **Split/Merge**：保证 Region 大小在合理范围内
- **Leader 均衡**：让写入请求分散到各节点
- **Region 均衡**：让存储容量均匀分布
- **热点调度**：自动识别并分散热点

理解 PD 调度，你就能理解 TiDB 如何在无需人工干预的情况下保持集群的健康运行。
