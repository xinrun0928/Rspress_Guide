# TiDB 扩容与缩容

TiDB 的水平扩展能力是其核心优势之一，支持在线扩容和缩容，不影响业务运行。

## 扩容原理

### 扩容流程

```
扩容流程：

1. 新节点启动
   ┌─────────────┐
   │  新 TiKV    │
   │  (Store)   │
   └─────────────┘

2. 注册到 PD
   ┌─────────────┐     ┌─────────────┐
   │  新 TiKV    │ ── → │     PD      │
   │  (Store)   │     │ (注册心跳)   │
   └─────────────┘     └─────────────┘

3. PD 调度 Region
   ┌─────────────┐
   │   Store 1   │ ←─ 迁出部分 Region
   └─────────────┘
   ┌─────────────┐
   │   Store 2   │ ←─ 迁出部分 Region
   └─────────────┘
   ┌─────────────┐
   │   Store 3   │ ←─ 迁出部分 Region
   └─────────────┘
   ┌─────────────┐
   │   新 Store  │ ←─ 迁入 Region
   └─────────────┘

4. 负载均衡完成
   数据均匀分布
```

## 扩容步骤

### 添加 TiKV 节点

```bash
# 1. 准备新节点
# 安装 TiKV 和配置
# 确保网络互通

# 2. 修改拓扑配置
# 编辑 inventory.ini 或使用 tiup

# 3. 扩容
tiup cluster scale-out tidb-cluster topo.yaml

# topo.yaml 示例
# tikv_servers:
# - host: 192.168.1.101
#   port: 20160
#   status_port: 20180
# - host: 192.168.1.102
#   port: 20160
#   status_port: 20180

# 4. 验证
tiup cluster display tidb-cluster
pd-ctl -u http://pd1:2379 store
```

### 添加 TiDB 节点

```bash
# 添加 TiDB 节点（无状态，可直接添加）
tiup cluster scale-out tidb-cluster topo.yaml

# 验证
tiup cluster display tidb-cluster
```

### 添加 PD 节点

```bash
# PD 节点需要更多配置
# 必须保证奇数个 PD 节点

# 添加 PD 节点
tiup cluster scale-out tidb-cluster pd.yaml

# 验证
pd-ctl -u http://pd1:2379 member
```

## 缩容原理

### 缩容流程

```
缩容流程：

1. 标记节点为 Offline
   ┌─────────────┐
   │  要移除的   │
   │  Store     │ ←─ PD 标记为 Offline
   └─────────────┘

2. PD 调度 Region 迁出
   ┌─────────────┐
   │  要移除的   │
   │  Store     │ ←─ Region 迁移到其他 Store
   └─────────────┘

3. Region 全部迁出
   ┌─────────────┐
   │  要移除的   │
   │  Store     │ ←─ 无 Region 后标记为 Tombstone
   └─────────────┘

4. 节点移除
   ┌─────────────┐
   │  要移除的   │
   │  Store     │ ←─ 安全停止
   └─────────────┘
```

## 缩容步骤

### 安全下线节点

```bash
# 1. 下线 TiKV 节点
# 先标记为 offline，让数据迁移完成
pd-ctl -u http://pd1:2379 store delete 5

# 2. 监控数据迁移
pd-ctl -u http://pd1:2379 store 5
# 查看状态：
# - Pending Offline: 正在迁移
# - Offline: 已下线，等待清理
# - Tombstone: 已清理完成

# 3. 等待 Region 迁移完成
# 可以使用 pd-ctl 查看进度
pd-ctl -u http://pd1:2379 operator show

# 4. 确认节点状态为 Tombstone
pd-ctl -u http://pd1:2379 store 5
# 输出应为 store state: Tombstone
```

### 使用 TiUP 缩容

```bash
# 使用 tiup 缩容
tiup cluster scale-in tidb-cluster -N 192.168.1.101:20160

# 查看缩容进度
tiup cluster display tidb-cluster

# 强制缩容（谨慎使用）
pd-ctl -u http://pd1:2379 store delete 5 --force
```

## 扩容缩容最佳实践

### 扩容最佳实践

```bash
# 1. 逐台添加
# 每次添加 1-2 个节点，等待数据平衡

# 2. 观察调度情况
pd-ctl -u http://pd1:2379 operator show

# 3. 监控资源使用
# 观察新增节点的 CPU、IO、网络

# 4. 验证数据完整性
SELECT COUNT(*) FROM your_table;
```

### 缩容最佳实践

```bash
# 1. 确认节点可下线
# 检查数据量、存储空间

# 2. 逐台下线
# 每次下线 1 个节点

# 3. 等待数据迁移完成
# 不要中断迁移过程

# 4. 确认节点状态
pd-ctl -u http://pd1:2379 store <store_id>
# 确保状态为 Tombstone
```

### 注意事项

```bash
# 1. 不要一次性扩容太多
# 可能导致集群不稳定

# 2. 不要在高峰期缩容
# 影响业务性能

# 3. 确保 PD 高可用
# PD 节点缩容需要特别小心

# 4. 监控调度情况
# 观察 operator 队列
```

## 常见问题

### 扩容问题

```bash
# 问题：扩容后调度太慢
# 解决：增加调度并发
pd-ctl -u http://pd1:2379 config set region-schedule-limit 4096

# 问题：新增节点负载高
# 解决：调整调度速度
pd-ctl -u http://pd1:2379 config set max-pending-beacon-count 16
```

### 缩容问题

```bash
# 问题：Region 迁移卡住
# 解决：检查网络和节点健康状态
pd-ctl -u http://pd1:2379 store <store_id>

# 问题：下线时间太长
# 解决：可以先减少副本数
pd-ctl -u http://pd1:2379 config set max-replicas 2
```

## 总结

扩容缩容要点：

- **扩容**：新节点注册到 PD，PD 自动调度 Region 迁移
- **缩容**：先下线节点，等待 Region 迁移完成
- **最佳实践**：逐台操作，观察调度，监控资源
- **注意事项**：高峰期不要缩容，确保 PD 高可用
