# TiDB 性能监控：Dashboard 与 Metrics 全解析

「数据库变慢了，但不知道哪里慢」——这是 DBA 最头疼的问题。

监控不到位，就像蒙着眼睛开车。你知道车在跑，但不知道仪表盘上发生了什么。CPU 高是真的高还是假的高？磁盘 I/O 延迟是哪个查询造成的？QPS 下降了，但流量明明没变……

**TiDB 提供了一套完整的监控体系**：TiDB Dashboard 提供可视化界面，Prometheus + Grafana 提供可定制的高级监控。学会用好这两套工具，你就能精准定位每一个性能问题。

## TiDB Dashboard：可视化监控利器

### 什么是 TiDB Dashboard

TiDB Dashboard 是 TiDB 内置的可视化监控平台，无需额外安装，直接通过浏览器访问：

```bash
# 访问 TiDB Dashboard
# 默认地址：http://${tidb_ip}:2379/dashboard

# 首次登录需要设置访问密码
tiup cluster display tidb-test
# 查看 Dashboard 访问信息
```

### Dashboard 核心页面

**1. 概览页面（Overview）**

展示集群整体健康状态：

- TiDB/TiKV/TiFlash 节点状态
- 整体 QPS 和延迟趋势
- 存储容量使用率
- 告警列表

**2. 集群页面（Cluster Info）**

查看集群拓扑：

```bash
# 查看 TiDB Server 列表
pd-ctl member | grep tidb

# 查看 TiKV 列表
pd-ctl store

# 查看 PD 列表
pd-ctl member | grep pd
```

**3. 热点分析（Hot Regions）**

识别热点 Region 和热点 Store：

```
┌─────────────────────────────────────────────────┐
│              Hot Region 写入分析                  │
├─────────────────────────────────────────────────┤
│  Store 1: ████████████████░░░░ 15.2 GB/s        │
│  Store 2: ██████████████░░░░░ 13.8 GB/s        │
│  Store 3: ████████████░░░░░░ 10.5 GB/s        │
└─────────────────────────────────────────────────┘
```

**4. SQL 语句分析（Statement）**

分析最慢的 SQL 语句：

```sql
-- 打开 Statement 诊断
-- 在 Dashboard -> Statement 页面查看

-- 分析特定的 SQL 类型
-- 关键指标：
--   - 平均执行时间（Avg Duration）
--   - 执行次数（Execution Count）
--   - 总耗时占比（Total Duration %）
```

**5. 慢查询日志（Slow Query）**

查看执行时间超过阈值的 SQL：

```sql
-- 设置慢查询阈值（默认 300ms）
SET tidb_slow_log_threshold = 200;

-- 查看慢查询日志
SELECT * FROM information_schema.slow_query
WHERE query_time > 1
ORDER BY query_time DESC
LIMIT 10;

-- 开启 Explain ANALYZE 分析慢查询
EXPLAIN ANALYZE SELECT * FROM orders WHERE status = 'pending';
```

## Prometheus + Grafana：企业级监控

### 架构说明

```
┌─────────────────────────────────────────────────────────────┐
│                    监控数据流                                │
│                                                             │
│  TiDB/TiKV/PD ──▶ Metrics ──▶ Prometheus ──▶ Grafana     │
│       │                              │                      │
│       │  拉取 Pull                    │   查询 Query        │

```

### 核心 Metrics 详解

**TiDB Server Metrics**

```bash
# QPS 监控
tidb_server_query_total                           # SQL 执行总次数
tidb_server_query_durationHistogram               # SQL 执行时间分布

# 连接数监控
tidb_server_connections                           # 当前连接数
tidb_server_errors_total                          # 错误总数

# KV 请求监控
tidb_tikvclient_txn_regions_num_histogram         # 事务涉及 Region 数
tidb_txn_total                                   # 事务总数
```

**TiKV Server Metrics**

```bash
# storage 监控
tikv_engine_size_bytes                           # 各引擎存储大小
tikv_store_size_bytes                           # Store 存储使用

# gRPC 监控
tikv_grpc_msg_duration_seconds                   # gRPC 请求延迟
tikv_grpc_requests_total                         # gRPC 请求总数

# Raft 监控
tikv_raftstore_propose_duration_seconds          # Raft 提案延迟
tikv_raftstore_append_log_duration_seconds       # 日志追加延迟
tikv_raftstore_commit_log_duration_seconds       # 日志提交延迟
```

**PD Server Metrics**

```bash
# 调度监控
pd_scheduler_status                              # 各调度器状态
pd_cluster_status                                # 集群状态

# Region 监控
pd_region_heartbeat_total                         # Region 心跳数
pd_region_split_total                            # Region 分裂数
pd_region_merge_total                            # Region 合并数

# TSO 监控
pd_tso_duration_seconds                          # TSO 获取延迟
pd_server_tso_requests                           # TSO 请求数
```

### 关键监控指标与告警

**系统资源告警**

```yaml
# prometheus 告警规则
groups:
- name: tikv
  rules:
  - alert: TiKV_server_is_down
    expr: up{job="tikv"} == 0
    for: 1m
    labels:
      severity: critical
    annotations:
      summary: "TiKV server is down"

  - alert: TiKV_Storage_Usage_High
    expr: tikv_store_size_bytes / tikv_store_capacity_bytes > 0.85
    for: 5m
    labels:
      severity: warning
    annotations:
      summary: "TiKV storage usage is high"
```

**性能告警**

```yaml
  - alert: TiDB_Slow_Query_High
    expr: increase(tidb_server_slow_query_total[5m]) > 10
    for: 2m
    labels:
      severity: warning
    annotations:
      summary: "Too many slow queries"

  - alert: TiKV_Coprocessor_Overload
    expr: rate(tikv_coprocessor_requests_error{type="meet_max_range"}[5m]) > 0.1
    for: 5m
    labels:
      severity: warning
    annotations:
      summary: "TiKV coprocessor is overloaded"
```

## 常见问题排查

### QPS 突然下降

```bash
# 1. 检查是否有慢查询
SELECT * FROM information_schema.slow_query
ORDER BY query_time DESC LIMIT 5;

# 2. 检查 TiDB Server 状态
up{job="tidb"}  # 是否有节点 down

# 3. 检查网络延迟
# Grafana -> TiDB -> Database Time -> KV Duration

# 4. 检查是否有 Lock 冲突
tidb_transaction_lock_total  # Lock 相关指标
```

### 延迟突然增加

```bash
# 1. 检查 Raft 延迟
# Grafana -> TiKV -> Raft -> Propose -> Propose wait duration

# 2. 检查 gRPC 延迟
# Grafana -> TiKV -> gRPC -> RPC Duration

# 3. 检查 RocksDB 写入延迟
# Grafana -> TiKV -> RocksDB -> Write Duration

# 4. 检查 Scheduler 配置
pd_scheduler_status{type="balance-leader-scheduler"}  # Leader 调度是否正常
```

### TiKV CPU 高

```bash
# 1. 检查是否是 Coprocessor 热点
# Grafana -> TiKV -> Coprocessor -> CPU

# 2. 检查是否有大量小查询
tidb_executor_statement_total{type="Select"}

# 3. 检查 Region 分裂是否正常
pd_region_split_total  # 分裂次数是否正常
```

### 存储容量不足

```bash
# 1. 查看各 Store 容量
pd-ctl store --jq='.stores[] | {id: .store.id, capacity: .capacity, used: .used}'

# 2. 扩容 TiKV
tiup cluster scale-out tidb-test -N 192.168.1.10:20160

# 3. 触发 Region 均衡
pd-ctl operator add transfer-peer region 233 from 1 to 2
```

## 监控最佳实践

### 1. 建立基准线

```bash
# 记录正常状态的关键指标
- TiDB QPS: ~10,000/s
- TiKV CPU: ~40%
- Query Duration P99: ~50ms
- KV Duration P99: ~10ms

# 定期对比，发现异常及时告警
```

### 2. 合理设置告警阈值

```yaml
# 告警阈值建议
CPU 使用率 > 80% -> warning
CPU 使用率 > 90% -> critical

存储使用率 > 85% -> warning
存储使用率 > 90% -> critical

QPS 下降 > 30% -> warning
Query Duration P99 > 500ms -> warning
```

### 3. 定期巡检

```bash
# 每周检查项
# 1. 集群整体状态
pd-ctl store | grep "state_name"

# 2. 慢查询分析
SELECT digest, query, avg(query_time), count(*)
FROM information_schema.slow_query
WHERE time >= NOW() - INTERVAL 7 DAY
GROUP BY digest
ORDER BY avg(query_time) DESC LIMIT 10;

# 3. 容量预测
# 根据存储增长趋势，预测何时需要扩容
```

## 总结

TiDB 的监控体系是排查问题的眼睛：

- **TiDB Dashboard**：快速定位问题，适合日常巡检和紧急排查
- **Prometheus + Grafana**：深度监控，适合建立基准线和长期趋势分析
- **告警规则**：防患于未然，在问题恶化前通知你

学会用好这套监控工具，你就能从"数据库变慢了"进化到"我知道数据库为什么慢了"。
