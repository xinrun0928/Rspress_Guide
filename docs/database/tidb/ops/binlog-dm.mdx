# TiDB Binlog 同步与 DM 数据迁移工具

数据迁移是数据库运维中最让人头疼的任务之一。

想象一下：你有一个 MySQL 数据库，现在要迁移到 TiDB。数据量几个 TB，不能停机，迁移过程中业务还要继续写入。怎么实现？

**TiDB 生态提供了两把利剑：Binlog 同步** 和 **DM（Data Migration）**。前者解决 MySQL 到 TiDB 的实时同步，后者提供从 MySQL 到 TiDB 的一站式迁移解决方案。

## Binlog 同步：MySQL 到 TiDB 的实时 CDC

### Binlog 是什么

MySQL Binlog 是 MySQL 的变更数据捕获（CDC）机制，记录了所有数据修改操作：

```sql
-- 查看 Binlog 状态
SHOW MASTER STATUS;
SHOW BINARY LOGS;

-- 查看 Binlog 内容
SHOW BINLOG EVENTS IN 'mysql-bin.000001' FROM 100 LIMIT 10;
```

| Binlog 格式 | 说明 |
|---|---|
| ROW | 记录每行数据的变化（推荐，兼容性最好） |
| STATEMENT | 记录执行的 SQL 语句 |
| MIXED | 混用 ROW 和 STATEMENT |

**TiDB Binlog 同步只支持 ROW 格式**，所以 MySQL 端必须设置：

```ini
# my.cnf
binlog_format = ROW
binlog_row_image = FULL
```

### Binlog 同步架构

```
┌─────────────┐    Binlog    ┌─────────────┐    RPC    ┌─────────────┐
│   MySQL     │ ───────────> │  Pump       │ ────────> │   Drainer   │
│  (源库)     │              │  (收集)      │           │   (消费)     │
└─────────────┘              └─────────────┘           └──────┬──────┘
                                                             │
                                                             ▼
                                                      ┌─────────────┐
                                                      │    TiDB     │
                                                      │  (目标库)    │
                                                      └─────────────┘
```

**Pump**：部署在 MySQL 侧的守护进程，实时拉取 Binlog 并持久化到本地，TiDB Drainer 通过 RPC 获取。

**Drainer**：从多个 Pump 收集 Binlog，将 SQL 事件转换为 TiDB 兼容格式，并发写入 TiDB。

### Binlog 部署与配置

```bash
# 1. 安装 Pump
tiup cluster scale-out tidb-test -pump.yml

# pump.yml 内容
server_configs:
  pump:
    gc: 7                    # Binlog 保留 7 天
    heartbeat-interval: 2     # 向 PD 发送心跳间隔（秒）
    storage: 100GiB          # 存储空间

pump_servers:
  - host: 192.168.1.10
    port: 8250
  - host: 192.168.1.11
    port: 8250
```

```bash
# 2. 安装 Drainer
tiup cluster scale-out tidb-test -drainer.yml

# drainer.yml 内容
drainer_servers:
  - host: 192.168.1.20
    port: 8249
    commit_ts: -1            # -1 表示从最新开始同步
    config:
      syncer.to:
        host: "192.168.1.100"
        port: 4000
        user: "root"
        password: "your-password"
        db: "tidb_binlog"
```

### Binlog 同步使用

```sql
-- 开启 TiDB Binlog
SET GLOBAL tidb_enable_binlog = ON;

-- 查看同步状态
SHOW BINLOG STATUS;

-- 查看 Pump 列表
SHOW PUMP STATUS;

-- 查看 Drainer 列表
SHOW DRAINER STATUS;
```

## DM：企业级数据迁移平台

### DM 是什么

DM（Data Migration）是 PingCAP 官方提供的一站式数据迁移工具，支持：

| 功能 | 说明 |
|---|---|
| 全量迁移 | 将 MySQL 数据完整导出并导入 TiDB |
| 增量同步 | 实时同步 MySQL 的数据变更到 TiDB |
| 分库分表合并 | 将多个 MySQL 库的表合并到一个 TiDB 表 |
| 数据校验 | 校验源库和目标库的数据一致性 |

### DM 架构

```
┌─────────────────────────────────────────────────────────────┐
│                      DM Control Plane                       │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐         │
│  │    DM-master │  │  DM-worker  │  │  DM-worker  │         │
│  │  (调度中心)  │  │  (节点1)    │  │  (节点2)    │         │
│  └─────────────┘  └──────┬──────┘  └──────┬──────┘         │
└──────────────────────────┼────────────────┼─────────────────┘
                           │                │
         ┌─────────────────┘                └─────────────────┐
         ▼                                                      ▼
┌─────────────────┐                                    ┌─────────────────┐
│     MySQL 1    │                                    │     MySQL 2     │
│  (源库: 3306)   │                                    │  (源库: 3306)   │
└─────────────────┘                                    └─────────────────┘
         │                                                      │
         └──────────────────────────┬───────────────────────────┘
                                    ▼
                           ┌─────────────────┐
                           │      TiDB       │
                           │   (目标库: 4000)  │
                           └─────────────────┘
```

### DM 部署

```bash
# 1. 准备 DM 集群
tiup dm deploy dm-cluster v2.0.0 ./dm-config.yml

# dm-config.yml
global:
  user: "tidb"
  ssh_port: 22
  deploy_dir: "/data/dm/deploy"
  data_dir: "/data/dm/data"

master_servers:
  - host: 192.168.1.10
    port: 8261

worker_servers:
  - host: 192.168.1.11
    port: 8262
  - host: 192.168.1.12
    port: 8262
```

### DM Task 配置

```yaml
# task.yaml - 全量 + 增量迁移任务
name: mysql-to-tidb
task-mode: all              # all = 全量 + 增量, full = 仅全量, incremental = 仅增量

# 源库配置
mysql-instances:
  - source-id: "mysql-replica-01"
    block-allow-list: "instance-1"
    
block-allow-list:
  instance-1:
    do-dbs: ["app_db"]      # 要迁移的数据库
    do-tables:
      - db-name: "app_db"
        tbl-name: "orders"  # 要迁移的表

# 目标库配置
target-database:
  host: "192.168.1.100"
  port: 4000
  user: "root"
  password: "your-password"

# 同步配置
syncers:
  global:
    worker-count: 16        # 并行同步 worker 数
    batch: 100              # 每批同步的行数
    checkpoint-flush-interval: 30
```

### DM 迁移流程

**第一步：全量数据迁移**

```bash
# 启动全量迁移任务
tiup dmctl start-task task.yaml

# 查看迁移状态
tiup dmctl query-status -task-name mysql-to-tidb
```

**第二步：增量数据同步**

```bash
# 全量迁移完成后，自动进入增量同步模式
# DM 会自动从断点位置继续同步

# 查看增量同步延迟
tiup dmctl query-status -task-name mysql-to-tidb
# 输出中查看 "syncerBinlog" 和 "syncerCheckpoint"
```

**第三步：数据校验**

```bash
# 创建校验任务
tiup dmctl check-task check-task.yaml

# 校验任务配置
# check-task.yaml
checkers:
  - type: "data一致性校验"
    worker-count: 16
```

### 分库分表合并

DM 最强大的功能之一是支持分库分表合并：

**场景**：你有 4 个 MySQL 库，每个库有一张 `orders` 表，要合并到 TiDB 的一张 `orders` 表：

```
MySQL Instance 1: db_0.orders
MySQL Instance 2: db_1.orders
MySQL Instance 3: db_2.orders
MySQL Instance 4: db_3.orders
                         ↓
                   TiDB: app.orders
```

```yaml
# shard-ddl-task.yaml
name: shard-migration
task-mode: all

mysql-instances:
  - source-id: "mysql-replica-01"
    schema-pattern: "db_*"
    table-pattern: "orders"
  - source-id: "mysql-replica-02"
    schema-pattern: "db_*"
    table-pattern: "orders"

routes:
  sharding-route-rules-table:
    schema-pattern: "db_*"
    table-pattern: "orders"
    target-schema: "app"
    target-table: "orders"

  sharding-route-rules-schema:
    schema-pattern: "db_*"
    target-schema: "app"
```

### 常见问题与解决

| 问题 | 原因 | 解决方案 |
|---|---|---|
| 全量迁移 OOM | 数据量太大 | 调整 `mydumper` 参数 `--chunk-filesize` |
| 增量同步延迟大 | 目标库写入能力不足 | 增加 `worker-count`，或扩容 TiDB |
| 校验不一致 | 迁移过程中有写入 | 使用 `syncer.ignore.pk.override` |
| 分库分表冲突 | 合并键重复 | 配置 `shard-group-id` 或使用全局唯一 ID |

```bash
# 常见排查命令
# 查看任务详情
tiup dmctl query-status -task-name mysql-to-tidb --more

# 查看 Worker 状态
tiup dmctl query-status -worker

# 暂停任务
tiup dmctl pause-task mysql-to-tidb

# 恢复任务
tiup dmctl resume-task mysql-to-tidb

# 停止任务
tiup dmctl stop-task mysql-to-tidb
```

## Binlog vs DM：如何选择

| 场景 | 推荐方案 | 原因 |
|---|---|---|
| MySQL -> TiDB 实时同步 | Binlog 同步 | 轻量，配置简单 |
| MySQL -> TiDB 一站式迁移 | DM | 支持全量 + 增量 + 校验 |
| 多源 MySQL 合并到 TiDB | DM | 分库分表合并是 DM 的强项 |
| 临时数据同步（ETL） | DM | 支持数据过滤和转换 |
| 长期的双向同步 | 不推荐 | 一致性难以保证 |

## 总结

Binlog 同步和 DM 是 TiDB 生态的两个核心工具：

- **Binlog 同步**：适合简单的 MySQL 到 TiDB 实时同步场景
- **DM**：适合复杂的企业级迁移，支持全量、增量、分库分表合并

选择哪个方案，取决于你的具体场景。简单同步用 Binlog，复杂迁移用 DM。
