# TiDB 面试汇总

这一节帮你梳理 TiDB 在面试中的高频问题，按照难度和考察维度分类。

---

## 一、架构相关

### Q1：TiDB 的整体架构是怎样的？

TiDB 由四个核心组件构成：

- **TiDB Server**：无状态 SQL 层，负责 SQL 解析和执行
- **TiKV**：行式分布式 KV 存储引擎，基于 RocksDB
- **PD**：调度中心，负责元数据管理和 Region 调度
- **TiFlash**：列式存储引擎，通过 Raft Learner 同步数据

### Q2：TiKV 和 TiDB 的关系是什么？

TiKV 是存储层，TiDB Server 是计算层。TiDB Server 不存储数据，所有请求最终发往 TiKV。TiKV 接收请求后，通过 Raft 协议在副本间达成一致，再返回结果。

### Q3：PD 的作用是什么？

PD 是集群的调度大脑：

- 分配全局唯一时间戳（TSO）
- 管理 Region 元数据
- 调度副本均衡、热点打散
- 支持 Label 亲和性配置

---

## 二、存储层原理

### Q4：什么是 Region？

Region 是 TiKV 中数据调度的基本单位，默认大小 96MB。每个 Region 包含一段连续 Key 范围的 Key-Value 数据。

### Q5：TiKV 如何保证数据不丢失？

TiKV 使用 **Raft 共识协议**。每个 Region 有 3 副本（可配置），写入时需要多数派（`>= 2`）确认才算成功。

### Q6：Raft Leader 挂了怎么办？

Follower 在 election timeout 后会发起选举，尝试成为新 Leader。TiDB 会在新 Leader 选出前阻塞写入，保证线性一致性。

### Q7：MVCC 有什么用？

MVCC（Multi-Version Concurrency Control）让 TiKV 能够：

- 支持 Snapshot Isolation 隔离级别
- 读写不互斥，提高并发
- 支持一致性历史读取

### Q8：TiDB 如何实现分布式事务？

TiDB 使用 **Percolator** 模型，两阶段提交（2PC）：

1. 客户端获取 TSO 作为事务 start_ts
2. Prewrite：所有涉及的主键写入，锁住数据
3. Commit：获取 commit_ts，释放锁
4. 读请求根据 start_ts 读取对应版本的数据

---

## 三、SQL 执行层

### Q9：TiDB 的 SQL 执行引擎原理？

TiDB 使用火山模型（Volcano Model），算子通过 Next() 接口拉取数据。TiKV 支持向量化执行和动态裁剪。

### Q10：什么是 MPP 模式？

MPP（Massively Parallel Processing）让复杂分析查询在多个 TiFlash 节点间并行执行，充分利用硬件资源。

### Q11：TiDB 的 DDL 是如何实现的？

TiDB 使用 **online DDL** 策略，DDL 变更分为多个版本，通过 schema version 管理。DDL owner 节点负责执行变更。

---

## 四、调度与运维

### Q12：PD 调度的原理？

PD 的 scheduler 组件根据以下规则调度 Region：

- 副本数约束（max-replicas）
- 亲和性约束（location-labels）
- 热点检测（hot read/write）
- 负载均衡（leader count、region count）

### Q13：TiDB 如何扩容缩容？

添加/删除 TiKV 节点后，PD 自动迁移数据。缩容时，PD 先将 Region 迁移走，再下线节点。

### Q14：如何保证跨机房高可用？

通过 PD 的 Label 配置，可以指定副本分布在不同机房/可用区：

```json
{
  "labels": {"zone": "zone-a"},
  "location-labels": ["zone"]
}
```

## 五、HTAP 能力

### Q15：TiFlash 和 TiKV 的区别？

| 维度 | TiKV | TiFlash |
|---|---|---|
| 存储格式 | 行式 | 列式 |
| 适用场景 | OLTP | OLAP |
| 同步方式 | - | Raft Learner |
| 索引 | 主键索引 | 主键索引 + 二级索引 |

### Q16：HTAP 下如何避免 OLTP 和 OLAP 互相影响？

TiFlash 支持**资源隔离**：

- 独立计算资源（tiflash_compute_max_cpu）
- 隔离的 TiFlash 节点
- 查询优先级控制

---

## 六、性能优化

### Q17：TiDB 的性能瓶颈在哪？

- 网络延迟（跨节点通信）
- TiKV 单节点吞吐（RocksDB 写入）
- PD 元数据热点（高并发下）
- SQL 复杂度和索引选择

### Q18：如何优化 TiDB 查询？

- 确保走了正确的索引（EXPLAIN）
- 避免大表 JOIN（考虑 hints）
- 合理设置 tidb_build_stats_concurrency
- TiFlash 加速 OLAP 查询

### Q19：写入热点怎么解决？

- 使用 SHARD_ROW_ID_BITS 打散热点
- 配置 AUTO_RANDOM 代替自增主键
- PD 热点调度参数调优

## 七、高频追问

面试官可能会追问的问题方向：

- Raft 的 election timeout 是多少？怎么调？
- Percolator 的锁冲突怎么处理？
- PD 的 TSO 分配机制是什么？
- TiKV 和 etcd 的 Raft 实现有什么区别？
- Region split 的触发条件是什么？

---

## 总结

TiDB 面试的核心是**架构理解 + 存储原理 + 事务机制**。架构层面要知道四大组件的职责和数据流向；存储层面要理解 Region、Raft、MVCC；事务层面要掌握 Percolator 的两阶段提交。运维知识是加分项，HTAP 能力是大趋势。

建议你根据面试深度，选择性深入某个模块。
