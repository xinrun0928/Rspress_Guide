# TiDB 整体架构

凌晨 2 点，你正盯着监控大屏，MySQL 主库的 CPU 已经飙到 95%，数据量眼看就要突破单机的天花板。团队讨论要不要分库分表，但一想到跨库 JOIN、分布式事务、跨节点排序这些头疼的问题，又陷入了沉默。

如果这时候有人说：「为什么不试试 TiDB？」你会不会好奇——它到底解决了什么问题？

---

## 一、架构设计哲学

要理解 TiDB，先要理解它的架构设计哲学。

大多数分布式数据库在设计时面临一个核心矛盾：**强一致性 vs 高可用性**。CAP 定理告诉我们，三者只能满足两个。但 TiDB 选择了 CP 模式——在网络分区发生时，优先保证一致性，牺牲可用性。

<Badge text="CAP" type="warning" /> 这不是 TiDB 的局限，而是工程上的权衡。对于金融、交易类业务，一致性比可用性更重要。

### 设计目标

TiDB 的设计目标是成为：

- **水平扩展**：通过增加节点线性提升容量和吞吐
- **强一致性**：分布式事务保证 ACID
- **高可用**：Raft 协议保证副本一致性
- **MySQL 兼容**：无缝迁移现有 MySQL 应用

这些目标看似矛盾，TiDB 是怎么做到的？答案在它的分层架构里。

## 二、四大核心组件

TiDB 的架构可以类比为一栋写字楼：

- **TiDB Server**：前台接待（SQL 入口）
- **PD**：物业中心（调度管理）
- **TiKV**：档案室（数据存储）
- **TiFlash**：资料库（分析数据）

### 2.1 TiDB Server：无状态的 SQL 层

TiDB Server 是整个系统的入口，负责：

- 接收客户端连接
- 解析 SQL 语句
- 生成执行计划
- 分布式 Join 执行
- 结果汇总返回

```sql
-- TiDB Server 接收 SQL
SELECT u.name, COUNT(o.id) as order_count
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE u.created_at >= '2024-01-01'
GROUP BY u.id;
-- TiDB Server 会：
-- 1. 解析 SQL
-- 2. 下推到多个 TiKV 并行扫描
-- 3. 聚合结果返回
```

<Badge text="无状态" type="tip" /> TiDB Server 完全无状态，可以任意水平扩展。这也是它能实现高并发查询的原因——前端负载均衡，后端并行执行。

### 2.2 PD：调度大脑

PD 是整个集群的管理者，负责：

- **TSO 分配**：Timestamp Oracle，全局唯一递增时间戳
- **Region 调度**：数据在节点间的均衡分布
- **副本管理**：多副本的高可用保障
- **拓扑感知**：节点物理位置的亲和性调度

```go
// TSO 的分配逻辑
type TimestampOracle struct {
    physical int64  // 物理时间（毫秒）
    logical int64   // 逻辑递增
}
// 每个事务的 start_ts 和 commit_ts 都由 PD 分配
```

TSO 是理解 TiDB 事务模型的关键。每个事务在开始时从 PD 获取一个 start_ts，提交时再获取一个 commit_ts，通过这两个时间戳实现 Snapshot Isolation。

### 2.3 TiKV：行式存储引擎

TiKV 是 TiDB 的存储引擎，基于 RocksDB 和 Raft 协议构建。

#### RocksDB：LSM-Tree 结构

RocksDB 是一个嵌入式 KV 存储引擎，使用 LSM-Tree（Log-Structured Merge-tree）结构：

- **Write**：先写 WAL，再写 MemTable（内存），顺序写入
- **Flush**：MemTable 满后 flush 到 L0 SST 文件
- **Compaction**：SST 文件逐层合并，清理无效数据

```java
// RocksDB 写入流程
WAL.log  // 持久化日志
   ↓
MemTable // 内存表
   ↓
L0 SST   // Level 0 文件
   ↓
L1 SST   // Level 1 文件
   ↓
L2...Ln  // 逐层合并
```

<Badge text="LSM-Tree" type="tip" /> LSM-Tree 的特点是写性能优秀，但读需要遍历多个层级。这是一种典型的「以空间换时间」的策略。

#### Range 分片

TiKV 使用 **Range 分片**而不是 Hash 分片：

- 每个 Region 是一段连续的 Key 范围
- 相邻的 Key 大概率会被一起访问（局部性原理）
- 支持高效的范围查询（如 `BETWEEN`、前缀匹配）

```bash
# TiKV 的 Key 编码
Key: t{table_id}_r{region_id}_k{row_key}
# 例如：t100_r5_k001 表示表100、Region 5、行键 001
```

Range 分片还有一个优势：**可以按 Key 顺序做范围扫描**。对于 `ORDER BY id LIMIT 10` 这样的查询，TiKV 可以利用局部性快速返回。

### 2.4 TiFlash：列式存储引擎

TiFlash 是 TiDB 的 OLAP 引擎，通过 **Raft Learner** 协议实时同步 TiKV 的行式数据，转换为列式存储。

```sql
-- OLTP 查询走 TiKV（行式）
SELECT * FROM orders WHERE id = 12345;

-- OLAP 查询走 TiFlash（列式）
SELECT region, SUM(amount) FROM orders GROUP BY region;
```

<Badge text="Raft Learner" type="tip" /> Learner 副本只参与日志复制，不参与 Leader 选举和写入投票。这保证了 TiFlash 不会影响 OLTP 写入的性能。

## 三、存储计算分离

TiDB 架构的核心是**存储计算分离**。

### 传统架构 vs TiDB 架构

**传统单机数据库**：

- SQL 引擎和存储引擎紧耦合
- 扩展时必须整体扩容
- 资源无法独立调度

**TiDB 架构**：

- TiDB Server 可以独立扩展（计算层）
- TiKV 可以独立扩展（存储层）
- 两者通过 gRPC 通信

```bash
# 计算层扩展
kubectl scale tidbcluster sample --namespace=tidb --tidb-replicas=5

# 存储层扩展
kubectl scale tidbcluster sample --namespace=tidb --tikv-replicas=5
```

这种架构的优势是**资源隔离**。当 OLAP 查询占用大量计算资源时，不会影响 OLTP 写入。

## 四、数据流详解

### 写入数据流

当客户端执行一条 INSERT 语句时：

```sql
INSERT INTO users (name, email) VALUES ('张三', 'zhang@example.com');
```

数据流经以下步骤：

1. **TiDB Server** 解析 SQL，生成写入计划
2. **PD** 分配全局唯一的事务 ID（start_ts）
3. 数据根据主键路由到对应的 **TiKV Region**
4. **TiKV** 使用 Raft 协议将数据写入多数派副本
5. **写入成功**，返回客户端

### 读取数据流

当客户端执行一条 SELECT 语句时：

```sql
SELECT * FROM users WHERE id = 1;
```

数据流经以下步骤：

1. **TiDB Server** 解析 SQL，生成查询计划
2. 根据 WHERE 条件，定位数据所在的 **TiKV Region**
3. 发送到对应 **TiKV Leader** 读取数据
4. **聚合结果**（如有 GROUP BY、LIMIT 等）
5. **返回客户端**

## 五、副本复制与高可用

### Raft Group

每个 Region 有 3 副本（可配置），组成一个 **Raft Group**：

- **Leader**：处理所有读写请求
- **Follower**：同步 Leader 的日志
- **Learner**：只读副本（TiFlash）

```bash
# 3 副本 Raft Group
Region {region_id: 5, peers: [store-1, store-2, store-3]}
# 写入需要 2 副本确认（多数派）
# 最多容忍 1 节点故障
```

<Badge text="RPO = 0" type="tip" /> 得益于 Raft 协议的日志复制机制，TiDB 的 RPO（Recovery Point Objective）为 0——故障切换时不会丢失数据。

### Label 亲和性

通过 PD 的 Label 配置，可以控制副本的物理分布：

```json
{
  "location-labels": ["zone", "rack", "host"],
  "cluster topology": {
    "zone-a": {"rack-1": ["store-1", "store-2"], "rack-2": ["store-3"]},
    "zone-b": {"rack-1": ["store-4", "store-5"]}
  }
}
```

这样配置后，PD 会尽量将 3 副本分布在不同的 zone，保证即使一个可用区宕机，数据仍然可用。

## 六、扩展性设计

### Region 自动管理

TiKV 的 Region 会根据数据量自动分裂和合并：

- **Split**：Region 超过 96MB 时分裂为两个
- **Merge**：空 Region 或小 Region 会合并
- **调度**：PD 根据负载均衡策略移动 Region

```bash
# 查看 Region 信息
pd-ctl region 12345
# 输出示例
{
  "id": 12345,
  "start_key": "t100_r",
  "end_key": "t100_s",
  "peers": [...],
  "size": 85,  # MB
  "keys": 1200000
}
```

### 热点处理

对于写入热点（比如自增主键），PD 提供多种调度策略：

- **热点调度器**：检测并打散热点 Region
- **负载均衡**：将热点 Region 的 Leader 迁移到空闲节点
- **Split**：将大 Region 分裂，减少单节点压力

## 七、与传统数据库的对比

| 维度 | 传统 MySQL | TiDB |
|---|---|---|
| 架构 | 单机 | 分布式 |
| 扩展方式 | 垂直扩展（单机升配） | 水平扩展（加节点） |
| 事务 | 单机 ACID | 分布式 ACID |
| 一致性 | 强一致 | 强一致（线性一致） |
| 高可用 | 主从复制 | Raft 多副本 |
| SQL 支持 | 完整 | 完整（分布式优化） |
| 运维复杂度 | 低 | 中高 |

## 总结

TiDB 的整体架构围绕**存储计算分离**展开：

- **TiDB Server** 是无状态 SQL 层，负责请求入口
- **PD** 是调度中心，分配 TSO，管理 Region 元数据
- **TiKV** 是行式存储，通过 Raft 保证副本一致
- **TiFlash** 是列式存储，通过 Learner 实时同步

这套架构让 TiDB 获得了水平扩展能力，同时保持了 MySQL 兼容性和强一致性。代价是网络往返增加、运维复杂度提升——这是分布式系统的固有代价。
