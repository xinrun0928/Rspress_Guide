# HTAP 能力

<Badge text="HTAP" type="tip" />（Hybrid Transactional/Analytical Processing）是 TiDB 的核心能力，让你可以在同一套数据上同时处理事务型和分析型负载。

## 一、为什么需要 HTAP？

传统架构下，你可能需要维护两套系统：

```sql
-- MySQL 扛业务（OLTP）
INSERT INTO orders (user_id, amount) VALUES (123, 99.9);

-- Hive/Spark 做分析（OLAP）
SELECT region, SUM(amount) FROM orders GROUP BY region;
```

两套系统的痛点：

- **数据同步延迟**：从 MySQL 到 Hive/Spark 通常有小时级延迟
- **数据一致性问题**：OLAP 查到的是历史快照，不是最新数据
- **运维成本翻倍**：两套系统，两套维护
- **跨系统 JOIN**：历史数据和实时数据无法关联

HTAP 的目标是让你**只维护一套数据库**，同时满足 OLTP 和 OLAP 的需求。

## 二、TiDB 的 HTAP 架构

### 存储层分离

| 存储引擎 | 存储格式 | 适用场景 | 特点 |
|---|---|---|---|
| TiKV | 行式 | OLTP | 低延迟、单行访问 |
| TiFlash | 列式 | OLAP | 高吞吐、聚合分析 |

两者的关系：

```sql
-- TiFlash 通过 Raft Learner 实时同步 TiKV 数据
-- 同步延迟：秒级
-- 数据一致性：强一致（不会读到脏数据）
```

### 计算层智能路由

TiDB 的优化器会自动选择查询路径：

```sql
-- 点查询，走 TiKV
SELECT * FROM users WHERE id = 1;

-- 聚合分析，走 TiFlash
SELECT department, AVG(salary) FROM employees GROUP BY department;

-- 跨引擎查询，优化器决定
SELECT u.name, SUM(o.amount)
FROM users u
JOIN orders o ON u.id = o.user_id
WHERE u.created_at >= '2024-01-01';
```

## 三、TiFlash 的列式存储

### 列式 vs 行式

| 维度 | 行式存储（TiKV） | 列式存储（TiFlash） |
|---|---|---|
| 数据组织 | 按行存储 | 按列存储 |
| 查询类型 | 点查询、更新 | 聚合分析、扫描 |
| I/O 效率 | 读取整行 | 只读需要的列 |
| 压缩率 | 低 | 高（同类型数据连续） |

列式存储的优势在 OLAP 场景下非常明显：

```sql
-- 假设 users 表有 50 列
SELECT name, email FROM users WHERE age >= 25;

-- 行式存储：读取所有 50 列的数据
-- 列式存储：只读取 name、email、age 三列
```

### 数据同步机制

TiFlash 通过 **Raft Learner** 协议同步 TiKV 的数据：

```rust
// TiFlash 节点的角色
Role::Learner {
    sync_from_leader: true,  // 从 TiKV Leader 同步
    apply_cmds: false,         // 不参与写入投票
    read_only: true            // 只读
}
```

<Badge text="Raft Learner" type="tip" /> Learner 副本不参与写入投票，保证 OLAP 复制不会影响 OLTP 写入性能。

## 四、MPP 架构

### 什么是 MPP？

MPP（Massively Parallel Processing）是一种大规模并行处理架构：

- 查询被分解为多个子任务
- 子任务在多个节点并行执行
- 结果在 TiDB Server 汇总

```sql
-- TiFlash 上的聚合查询
SELECT region, SUM(amount), COUNT(*)
FROM orders
WHERE create_time >= '2024-01-01'
GROUP BY region;
-- 执行计划：
-- TiFlash-1: 处理 region IN ('A', 'B')
-- TiFlash-2: 处理 region IN ('C', 'D')
-- TiFlash-3: 处理 region IN ('E', 'F')
-- TiDB Server: 合并结果
```

### MPP vs 单机执行

| 维度 | 单机执行 | MPP |
|---|---|---|
| 节点数 | 1 | N |
| 数据量 | 受单机内存限制 | 可处理 PB 级 |
| 扩展性 | 差 | 线性扩展 |
| 适用场景 | 小数据量 | 大数据量分析 |

## 五、资源隔离

HTAP 场景下，OLTP 和 OLAP 容易互相影响。TiDB 提供了多种隔离机制：

### 计算资源隔离

```toml
# TiDB 配置
[Flash.compute]
tiflash_compute_max_cpu = 3  # TiFlash 计算最大 CPU 核数
```

### 隔离节点部署

```bash
# 独立的 TiFlash 节点池
# OLAP 查询只发到 dedicated-tiflash 组
# OLTP 请求不受影响
```

### 查询优先级

```sql
-- 低优先级查询
SET tidb_query_priority = 1;

-- 高优先级查询（重要报表）
SET tidb_query_priority = 10;
```

## 六、使用建议

### 适合 HTAP 的场景

- **实时分析**：大屏、实时报表，需要看到最新数据
- **混合负载**：业务查询和数据分析混合部署
- **简化架构**：不想维护两套系统

### 不适合 HTAP 的场景

- **超大规模 OLAP**：PB 级数据，专用数仓更合适
- **极致 OLTP**：对延迟极度敏感，TiFlash 反而增加复杂度
- **重度批处理**：夜间跑批，TiFlash 成本效益不高

## 总结

TiDB 的 HTAP 能力通过**存储层分离 + 智能路由**实现：

- TiKV 扛 OLTP，低延迟、强一致
- TiFlash 扛 OLAP，列式存储、MPP 并行
- Raft Learner 保证数据实时同步且不影响写入
- 资源隔离避免两类负载互相干扰

这是一套务实的 HTAP 方案——不是最强的 OLAP，但足够好的 OLTP + 足够可用的 OLAP。
