# 应用场景

TiDB 不是万能药，但它在某些场景下表现出色。选对场景，TiDB 是利器；选错场景，TiDB 是负担。

---

## 一、TiDB 最擅长的场景

### 1.1 MySQL 瓶颈突破

**场景描述**：MySQL 单机已经优化到极限（参数调优、索引优化、读写分离），但业务还在增长，数据量和 QPS 接近单机天花板。

**为什么适合 TiDB**：

- 无需改造业务代码，MySQL 协议直连
- 水平扩展存储和计算
- 保持事务一致性

```sql
-- 业务代码完全不变
-- 只是把连接地址从 MySQL 改成 TiDB
jdbc:mysql://tidb-host:4000/myapp
```

**判断标准**：

- 数据量接近或超过 1TB
- 单机 QPS `` `>=` `` 10 万
- 读写延迟开始明显上升

### 1.2 分库分表替代方案

**场景描述**：团队讨论过分库分表方案，但一想到跨库 JOIN、分布式事务、迁移成本，就想找更好的方案。

**为什么适合 TiDB**：

- 逻辑上是单表，SQL 全支持
- 自动数据分片，无需业务代码
- 分布式事务保证 ACID

```sql
-- TiDB：跨表 JOIN 跟单表一样简单
SELECT u.name, SUM(o.amount)
FROM users u
JOIN orders o ON u.id = o.user_id
JOIN order_items oi ON o.id = oi.order_id
GROUP BY u.id;

-- 分库分表：需要中间件支持，语法受限
-- SELECT * FROM orders_0 UNION ALL SELECT * FROM orders_1...
```

<Badge text="分库分表" type="warning" /> 分库分表不是不能用，而是成本很高。如果能用 TiDB 解决的问题，没必要走弯路。

### 1.3 HTAP 混合负载

**场景描述**：同时有实时分析需求（实时大屏、A/B 测试、用户行为分析）和业务交易需求，不想维护两套系统。

**为什么适合 TiDB**：

- TiKV 服务 OLTP，TiFlash 服务 OLAP
- 数据实时同步，OLAP 能看到最新数据
- 统一维护，降低复杂度

```sql
-- OLTP：下单请求，延迟 < 10ms
INSERT INTO orders (user_id, amount) VALUES (1, 99.9);

-- OLAP：实时大屏，秒级响应
SELECT region, SUM(amount), COUNT(*)
FROM orders
WHERE create_time >= DATE_SUB(NOW(), INTERVAL 1 DAY)
GROUP BY region;
```

### 1.4 数据归档与历史查询

**场景描述**：历史数据太多影响查询性能，但又不能删除（合规、审计、复盘需求）。

**为什么适合 TiDB**：

- TiDB 支持冷热数据分离
- 历史数据可以用 TiFlash 归档
- 查询时自动路由到合适存储

```sql
-- 最近的订单走 TiKV（热数据）
-- 一年前的订单走 TiFlash（冷数据）
SELECT * FROM orders WHERE create_time >= '2024-01-01';

-- 强制走 TiFlash
SELECT /*+ READ_FROM_STORAGE(TIFLASH[orders]) */ *
FROM orders WHERE create_time < '2023-01-01';
```

## 二、TiDB 不擅长的场景

### 2.1 小数据量

**场景**：数据量 < 100GB，QPS < 1 万。

**为什么不适合**：

- TiDB 组件多，运维复杂度高
- 小数据量 MySQL 完全够用
- 引入 TiDB 增加不必要的复杂度

:::tip
100GB 以下的场景，优先考虑 MySQL + Read Replica。TiDB 的价值在于扩展性，小数据量用不上。
:::

### 2.2 超大规模 OLAP

**场景**：PB 级数据分析，复杂的多表 JOIN，批处理作业。

**为什么不适合**：

- TiFlash 的 OLAP 能力不如 ClickHouse、Snowflake
- 大量 OLAP 查询会影响 OLTP 性能
- 成本效益不如专用数仓

```sql
-- TiDB 适合：秒级到分钟级的分析
SELECT region, SUM(amount) FROM orders GROUP BY region;  -- 5秒

-- 不适合：小时级的复杂分析
SELECT ... FROM a JOIN b JOIN c JOIN d...;  -- 跑几小时
```

### 2.3 极致延迟敏感

**场景**：延迟要求 < 1ms 的高频交易系统。

**为什么不适合**：

- TiDB 有网络往返延迟（TiDB -> TiKV）
- Raft 协议保证一致性需要额外开销
- 分布式事务比单机事务慢

<Badge text="P99" type="warning" /> TiDB 的写入 P99 延迟通常在 10-50ms，读取 P99 在 5-20ms。如果你的要求是 < 1ms，TiDB 不适合。

## 三、选型决策树

```
数据量 <= 100GB?
├── 是 → MySQL + Read Replica
└── 否 → 数据量 <= 1TB?
    ├── 是 → MySQL 优化 或 TiDB
    └── 否 → 有 HTAP 需求?
        ├── 是 → TiDB
        └── 否 → 分库分表 或 TiDB
```

## 四、行业案例参考

| 行业 | 典型场景 | TiDB 优势 |
|---|---|---|
| 金融 | 交易系统、账务系统 | 强一致、高可用 |
| 电商 | 订单中心、用户中心 | 高并发、实时分析 |
| 游戏 | 玩家数据、游戏日志 | 海量存储、水平扩展 |
| 物联网 | 设备数据、时序数据 | 高写入、海量存储 |
| SaaS | 多租户数据 | 统一平台、简化架构 |

## 总结

TiDB 的最佳场景是**水平扩展 + MySQL 兼容 + HTAP** 三需求的交集。如果你正面临 MySQL 的瓶颈，又不想走分库分表的弯路，TiDB 是一个值得考虑的选择。

选型前，先问自己三个问题：

1. 数据量和 QPS 是否接近单机瓶颈？
2. 是否需要跨库 JOIN 或分布式事务？
3. 是否同时有 OLTP 和 OLAP 需求？

三个问题中如果有两个以上的答案是「是」，TiDB 值得一试。
