# MySQL 兼容性

<Badge text="MySQL 5.7" type="tip" /> <Badge text="MySQL 8.0" type="tip" /> TiDB 高度兼容 MySQL 协议，这是它能广泛普及的重要原因之一。

## 一、兼容性设计原则

TiDB 的 MySQL 兼容性不是「90% 兼容」，而是尽可能高地兼容 MySQL 语法和行为。设计原则是：

- **协议层兼容**：MySQL 客户端/驱动可以直接连接
- **语法层兼容**：主流 SQL 语法直接支持
- **语义层兼容**：行为尽可能与 MySQL 一致

```sql
-- 现有 MySQL 应用无需修改
mysql -h tidb-host -P 4000 -u root -p myapp
-- 使用 MySQL JDBC 驱动直连 TiDB
jdbc:mysql://tidb-host:4000/myapp
```

## 二、支持的特性

### DDL 操作

| 特性 | 支持情况 |
|---|---|
| CREATE TABLE | 完全支持 |
| ALTER TABLE | 大部分支持 |
| DROP TABLE | 完全支持 |
| TRUNCATE TABLE | 完全支持 |
| CREATE INDEX | 完全支持 |
| DROP INDEX | 完全支持 |

```sql
-- TiDB DDL 示例
CREATE TABLE users (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(100) NOT NULL,
    email VARCHAR(255) UNIQUE,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

ALTER TABLE users ADD COLUMN phone VARCHAR(20);

CREATE INDEX idx_email ON users(email);
```

### DML 操作

```sql
-- SELECT
SELECT * FROM users WHERE id = 1;
SELECT name, COUNT(*) FROM orders GROUP BY name;

-- INSERT
INSERT INTO users (name, email) VALUES ('张三', 'zhang@example.com');
INSERT INTO users VALUES (1, '李四', 'li@example.com');

-- UPDATE
UPDATE users SET email = 'new@example.com' WHERE id = 1;

-- DELETE
DELETE FROM users WHERE id = 2;
```

### 事务支持

```sql
-- ACID 事务
BEGIN;
INSERT INTO accounts (id, balance) VALUES (1, 100);
INSERT INTO accounts (id, balance) VALUES (2, 50);
COMMIT;

-- ROLLBACK
BEGIN;
INSERT INTO orders (user_id, amount) VALUES (1, 99.9);
-- 出现错误
ROLLBACK;
```

### 索引类型

```sql
-- 主键索引
PRIMARY KEY (id)

-- 唯一索引
UNIQUE INDEX idx_email (email)

-- 普通索引
INDEX idx_name (name)

-- 联合索引
INDEX idx_user_date (user_id, created_at)
```

## 四、不兼容的部分

尽管兼容性很高，但仍有一些差异需要注意：

### 自增 ID 语义

```sql
-- MySQL：自增 ID 保证连续
AUTO_INCREMENT

-- TiDB：分布式环境下，自增 ID 不保证连续
-- TiDB 推荐使用 AUTO_RANDOM 代替
CREATE TABLE orders (
    id BIGINT AUTO_RANDOM PRIMARY KEY,  -- 推荐
    amount DECIMAL(10,2)
);
```

### 外键

```sql
-- TiDB 不支持外键
CREATE TABLE orders (
    id BIGINT PRIMARY KEY,
    user_id BIGINT,
    FOREIGN KEY (user_id) REFERENCES users(id)  -- 不支持
);
```

:::warning
外键在分布式环境下语义复杂，TiDB 选择不支持。如果有强外键需求，考虑在应用层处理。
:::

### 存储过程和函数

```sql
-- TiDB 不支持用户定义的存储过程
CREATE PROCEDURE get_user_count()  -- 不支持
BEGIN
    SELECT COUNT(*) FROM users;
END;
```

### 触发器

```sql
-- TiDB 不支持触发器
CREATE TRIGGER after_user_insert  -- 不支持
AFTER INSERT ON users
FOR EACH ROW
BEGIN
    INSERT INTO audit_log (action) VALUES ('user_created');
END;
```

### 其他限制

| 特性 | 状态 | 说明 |
|---|---|---|
| 事件（Events） | 不支持 | 定时任务建议用 cron/job |
| 全文索引 | 有限支持 | TiDB 5.3+ 支持基础全文索引 |
| 空间数据类型 | 有限支持 | GIS 函数支持不完整 |
| 查询缓存 | 不支持 | 分布式场景下无意义 |

## 五、迁移注意事项

### 迁移前检查

```bash
# 使用 dm-performanc 检查兼容性
./bin/dmctl --master-addr=localhost:8261

# 检查表结构兼容性
tiup dmctl check-cluster config.yml
```

### 常见迁移问题

#### 问题 1：自增主键热点

```sql
-- 问题：自增主键导致写入热点
INSERT INTO orders (user_id, amount) VALUES (1, 99.9);
-- 所有新数据都写到同一个 Region

-- 解决：使用 AUTO_RANDOM
CREATE TABLE orders (
    id BIGINT AUTO_RANDOM PRIMARY KEY,  -- 打散热点
    user_id BIGINT,
    amount DECIMAL(10,2)
);
```

#### 问题 2：大字段影响性能

```sql
-- 问题：TEXT/BLOB 字段影响 Region 分裂
CREATE TABLE articles (
    id BIGINT PRIMARY KEY,
    title VARCHAR(200),
    content TEXT  -- 大字段
);

-- 解决：分离到独立表
CREATE TABLE articles (
    id BIGINT PRIMARY KEY,
    title VARCHAR(200)
);
CREATE TABLE article_contents (
    article_id BIGINT PRIMARY KEY,
    content MEDIUMTEXT
);
```

#### 问题 3：大事务

```sql
-- 问题：大事务占用大量内存
BEGIN;
INSERT INTO orders ...;  -- 10万条
COMMIT;

-- 解决：分批提交
for batch in batches:
    INSERT INTO orders ...;  -- 每批 1000 条
    COMMIT;
```

## 总结

TiDB 的 MySQL 兼容性做得非常扎实，协议层和语法层基本无缝对接。不兼容的部分主要是分布式场景下语义会改变的功能（自增 ID、外键、触发器）。

迁移策略是：先用 MySQL 协议连接，跑通现有业务，再针对性优化热点和大事务。
