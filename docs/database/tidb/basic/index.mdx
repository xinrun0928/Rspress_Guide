# TiDB 分布式数据库：整体架构解析

当你的数据量从 GB 级别跨越到 TB、甚至 PB 级别时，传统的单机数据库会遇到三个无法回避的问题：**容量瓶颈、计算瓶颈、可用性瓶颈**。

水平扩展说起来容易，做起来难。MySQL 的分库分表方案虽然能解决问题，但带来了跨库查询、分布式事务、数据迁移等一系列复杂度。

**TiDB** 给出了另一种答案：一个同时支持 **水平扩展**、**强一致性事务**、**实时分析（HTAP）** 的分布式数据库——而不需要你修改一行业务代码。

## TiDB 架构概览

TiDB 采用了经典的 **计算-存储分离** 架构，核心组件分为四层：

```
┌─────────────────────────────────────────────────────────────┐
│                      TiDB Server 层                         │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐         │
│  │  SQL Layer  │  │   Parser   │  │   Optimizer │         │
│  └─────────────┘  └─────────────┘  └─────────────┘         │
└───────────────────────────┬─────────────────────────────────┘
                            │  TiDB Client
┌───────────────────────────▼─────────────────────────────────┐
│                    TiKV Cluster 层                          │
│  ┌────────────┐  ┌────────────┐  ┌────────────┐            │
│  │   TiKV 1   │  │   TiKV 2   │  │   TiKV N   │            │
│  │  [Region]  │  │  [Region]  │  │  [Region]  │            │
│  └────────────┘  └────────────┘  └────────────┘            │
│                      分布式存储                              │
└───────────────────────────┬─────────────────────────────────┘
                            │
┌───────────────────────────▼─────────────────────────────────┐
│                      PD Cluster 层                          │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐         │
│  │     PD 1    │  │     PD 2    │  │     PD 3    │         │
│  │ (Leader)   │  │  (Follower) │  │  (Follower) │         │
│  └─────────────┘  └─────────────┘  └─────────────┘         │
│                  元数据管理 + 调度中心                        │
└─────────────────────────────────────────────────────────────┘
                            │
┌───────────────────────────▼─────────────────────────────────┐
│                     TiFlash 层                              │
│  ┌────────────┐  ┌────────────┐  ┌────────────┐            │
│  │ TiFlash 1  │  │ TiFlash 2  │  │ TiFlash N  │            │
│  │ (列式存储) │  │ (列式存储) │  │ (列式存储) │            │
│  └────────────┘  └────────────┘  └────────────┘            │
│                    实时分析引擎                              │
└─────────────────────────────────────────────────────────────┘
```

## 核心组件详解

### TiDB Server：无状态的 SQL 引擎

TiDB Server 是整个数据库的"入口"，负责：

1. **协议兼容**：兼容 MySQL 协议，你可以用任何 MySQL 客户端连接
2. **SQL 解析与优化**：将 SQL 解析成执行计划
3. **分布式执行**：将请求路由到正确的 TiKV 节点
4. **结果聚合**：汇总各 TiKV 返回的结果

```bash
# 用 MySQL 客户端连接 TiDB
mysql -h 127.0.0.1 -P 4000 -u root

# 查看 TiDB 集群状态
tiup cluster display tidb-test
```

**重要特性**：TiDB Server 是**无状态**的，可以随时水平扩展。多个 TiDB Server 通过 PD 协调，实现负载均衡。

### TiKV：分布式事务存储引擎

TiKV 是 TiDB 的核心存储组件，底层基于 **RocksDB**：

| 特性 | 说明 |
|---|---|
| **Row-based 存储** | 适合 OLTP 场景的点查询和写入 |
| **Region 分片** | 数据按 Region（默认 96MB）自动分片 |
| **Raft 共识** | 每个 Region 的副本通过 Raft 协议保证一致性 |
| **MVCC** | 多版本并发控制，支持分布式事务 |

```rust
// TiKV 数据模型简化版
struct TiKVRegion {
    region_id: u64,
    start_key: Vec<u8>,      // Region 起始 key
    end_key: Vec<u8>,        // Region 结束 key
    peers: Vec<Peer>,        // Raft 副本列表
    leader: Option<Peer>,   // 当前 Leader
}

// 写入流程
Client -> TiDB Server -> 计算 Key 所属 Region -> 路由到对应 TiKV Leader
-> Raft 日志复制 -> 多数节点确认 -> 应用到 RocksDB -> 返回成功
```

### PD：集群的大脑

PD（Placement Driver）是 TiDB 的元数据管理中枢：

1. **集群拓扑管理**：管理所有 TiKV 和 TiFlash 节点信息
2. **时间戳分配**：TiDB 的分布式事务依赖 PD 分配的 `tso`（Timestamp Oracle）
3. **调度决策**：决定 Region 如何分裂、合并、迁移

```bash
# 查看 PD 元数据
pd-ctl region --jq='.regions[] | {id: .id, start_key: .start_key, end_key: .end_key}'

# 查看某个 Region 的详情
pd-ctl region 233
```

**重要特性**：PD 本身也是分布式的，通过 Etcd 协议保证高可用。

### TiFlash：列式存储分析引擎

TiFlash 是 TiDB 的实时分析引擎：

| 特性 | 说明 |
|---|---|
| **列式存储** | 专为分析场景优化，列裁剪大幅减少 I/O |
| **异步复制** | 从 TiKV 异步同步数据，保证分析不影响写入 |
| **一致性保证** | 通过 Raft Learner 机制保证与 TiKV 的数据一致 |
| **向量化执行** | 利用 CPU SIMD 指令加速计算 |

```sql
-- 开启 TiFlash 副本
ALTER TABLE orders SET TIFLASH REPLICA 1;

-- 查询自动路由到 TiFlash
EXPLAIN SELECT date, SUM(amount) 
FROM orders 
GROUP BY date;
```

## 核心特性深度解析

### 水平扩展：真正的弹性

TiDB 的扩展性体现在两个维度：

**存储扩展**：当 TiKV 节点不足时，只需添加新节点，PD 会自动将数据均衡迁移：

```bash
# 扩容 TiKV
tiup cluster scale-out tidb-test -N 192.168.1.10:20160

# PD 自动进行负载均衡，无需人工干预
```

**计算扩展**：TiDB Server 无状态，添加节点即可提升并发处理能力，配合 Load Balancer 实现无缝扩展。

### ACID 分布式事务

TiDB 的分布式事务采用 **Percolator 模型**，核心是两阶段提交（2PC）：

```sql
-- 一个简单的分布式事务
BEGIN;

UPDATE accounts SET balance = balance - 1000 WHERE user_id = 1;
UPDATE orders SET status = 'paid' WHERE order_id = 'ORD001';

COMMIT;  -- TiDB 自动处理分布式两阶段提交
```

**事务原理**：

1. **获取时间戳**：向 PD 请求 `start_ts`（开始时间戳）和 `commit_ts`（提交时间戳）
2. **读取验证**：读取数据时检查是否有更晚的 `commit_ts`
3. **两阶段提交**：先 prewrite（所有节点写入），再 commit（多数节点确认即成功）

```rust
// 简化的事务流程
struct Transaction {
    start_ts: u64,
    write_rows: Vec<Row>,
    
    fn commit(&mut self) -> Result<u64> {
        // 阶段1: Prewrite - 所有涉及的数据节点写入
        for row in &self.write_rows {
            let primary_lock = self.prewrite(row)?;
        }
        
        // 阶段2: Commit - 向 PD 获取 commit_ts，多数节点确认即成功
        let commit_ts = pd.get_timestamp()?;
        self.commit_all(commit_ts)?;
        
        Ok(commit_ts)
    }
}
```

### HTAP：一体化数据平台

HTAP（Hybrid Transaction/Analytical Processing）是 TiDB 的杀手级特性：

| 场景 | 引擎 | 说明 |
|---|---|---|
| 实时写入 | TiKV | OLTP 场景，高并发点写 |
| 实时分析 | TiFlash | 列式存储，复杂聚合查询 |
| 智能路由 | TiDB Optimizer | 自动选择最优执行引擎 |

```sql
-- OLTP 场景：写入
INSERT INTO orders VALUES ('ORD100', 'user1', 299.00, NOW());

-- OLTP 场景：点查
SELECT * FROM orders WHERE order_id = 'ORD100';

-- OLAP 场景：分析（自动路由到 TiFlash）
SELECT date_trunc('month', created_at), SUM(amount)
FROM orders
GROUP BY 1
ORDER BY 2 DESC
LIMIT 10;
```

## 适用场景分析

### 适合 TiDB 的场景

| 场景 | 原因 |
|---|---|
| 数据量超过 1TB | 单机 MySQL 无法承载，需要水平扩展 |
| 写入压力大 | TiKV 的分布式写入比单机 MySQL 吞吐更高 |
| 需要实时分析 | TiFlash 提供秒级延迟的分析能力 |
| 不想分库分表 | TiDB 提供透明的水平扩展，应用无需改造 |
| MySQL 迁移 | TiDB 兼容 MySQL 协议，迁移成本低 |

### 不适合 TiDB 的场景

| 场景 | 原因 |
|---|---|
| 小数据量（< 100GB） | 架构复杂度带来的收益不明显 |
| 极高并发简单 KV | Redis 更适合，TiDB 有额外的协议开销 |
| 需要存储过程/触发器 | TiDB 对 MySQL 高级特性支持有限 |
| 超低延迟毫秒级响应 | TiDB 是毫秒到秒级，不适合极端低延迟场景 |

## 总结

TiDB 用一套架构解决了三个问题：**水平扩展**、**强一致性事务**、**HTAP 分析**。它的设计哲学是"让分布式变得透明"——你不需要关心数据在哪里，SQL 写下去，结果自然来。

对于数据量巨大、又不想引入分库分表复杂度的团队，TiDB 是一个值得考虑的选择。
