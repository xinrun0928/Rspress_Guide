# 红黑树：五大性质与插入修复

**红黑树是 Java 中 `TreeMap` 的底层实现，也是 Linux 进程调度、C++ STL 中 `map` 的数据结构。** 它是面试中的高频考点。

面试官问：**「红黑树有哪些性质？」**

你开始背诵：**「根节点是黑色...」**

但面试官追问：**「为什么有了 AVL 树，还需要红黑树？」**

你可能会卡住......

**答案在于「平衡的代价」。** AVL 树的严格平衡让查找更快，但插入删除时的旋转代价也更高。红黑树选择用「近似平衡」换取更好的插入删除性能。

## 红黑树的定义

红黑树是一种**近似平衡**的二叉搜索树，通过颜色和五条规则保证树的高度大致为 O(log n)。

```java
class RBNode {
    int val;
    boolean isRed; // true = 红色，false = 黑色
    RBNode left, right, parent;

    RBNode(int val) {
        this.val = val;
        this.isRed = true; // 新节点默认红色
    }
}
```

## 五大性质

| 编号 | 性质 | 含义 |
|------|------|------|
| 1 | 节点是红色或黑色 | 基本定义 |
| 2 | 根节点是黑色 | 保证性质 5 的有效性 |
| 3 | 叶节点（NIL）是黑色 | NIL 节点用于简化边界处理 |
| 4 | 红节点的子节点都是黑色 | 不允许两个连续的红节点 |
| 5 | 从任一节点到其每个叶节点的路径都包含相同数量的黑色节点 | 保证了树的「黑色完美平衡」 |

### 第五性质的直观理解

```
         黑(2)                          黑(2)
        /    \                         /    \
     红(1)   红(1)    -->           黑(1)   黑(1)
    /   \   /   \                 /   \   /   \
  NIL  NIL NIL NIL              NIL  NIL NIL NIL
```

从根到叶子的任意路径上，黑色节点的数量相同。这保证了**最长路径不超过最短路径的两倍**。

## 旋转操作

红黑树的旋转和 AVL 树类似，但旋转后需要重新着色。

```java
// 左旋
private void leftRotate(RBNode x) {
    RBNode y = x.right;
    x.right = y.left;
    if (y.left != null) y.left.parent = x;
    y.parent = x.parent;

    if (x.parent == null) {
        root = y;
    } else if (x == x.parent.left) {
        x.parent.left = y;
    } else {
        x.parent.right = y;
    }

    y.left = x;
    x.parent = y;
}
```

## 插入修复（重点）

### 插入情景分析

插入一个新节点 z，默认为红色。可能有以下情况：

| 情景 | 条件 | 处理 |
|------|------|------|
| **Case 1** | z 是根节点 | 改为黑色 |
| **Case 2** | z 的父节点是黑色 | 无需处理 |
| **Case 3** | z 的父节点和叔叔节点都是红色 | 重新着色 |
| **Case 4** | z 的父节点是红色，叔叔节点是黑色，且 z 是内孙子 | 先局部旋转，再按 Case 5 处理 |
| **Case 5** | z 的父节点是红色，叔叔节点是黑色，且 z 是外孙子 | 旋转 + 重新着色 |

### 修复算法

```java
public void insert(int val) {
    RBNode z = new RBNode(val);
    insertNode(z);
    fixViolation(z);
}

private void fixViolation(RBNode z) {
    RBNode parent, grandparent;

    while (z != root && isRed(z) && isRed(z.parent)) {
        parent = z.parent;
        grandparent = parent.parent;

        // 父节点是祖父的左孩子
        if (parent == grandparent.left) {
            RBNode uncle = grandparent.right;

            // Case 3：叔叔是红色
            if (uncle != null && isRed(uncle)) {
                setBlack(parent);
                setBlack(uncle);
                setRed(grandparent);
                z = grandparent; // 继续向上检查
            } else {
                // Case 4：z 是右孩子，需要先左旋变成 Case 5
                if (z == parent.right) {
                    z = parent;
                    leftRotate(z);
                    parent = z.parent;
                }

                // Case 5：z 是左孩子，右旋
                setBlack(parent);
                setRed(grandparent);
                rightRotate(grandparent);
            }
        } else {
            // 父节点是祖父的右孩子（镜像处理）
            RBNode uncle = grandparent.left;

            if (uncle != null && isRed(uncle)) {
                setBlack(parent);
                setBlack(uncle);
                setRed(grandparent);
                z = grandparent;
            } else {
                if (z == parent.left) {
                    z = parent;
                    rightRotate(z);
                    parent = z.parent;
                }

                setBlack(parent);
                setRed(grandparent);
                leftRotate(grandparent);
            }
        }
    }
    setBlack(root); // 确保根节点是黑色
}
```

## 红黑树 vs AVL 树

| 对比项 | AVL 树 | 红黑树 |
|--------|--------|--------|
| 平衡标准 | 严格平衡 | 近似平衡 |
| 查找效率 | O(log n)，更稳定 | O(log n)，略快于平均 |
| 插入/删除 | 需要更多旋转 | 最多 2 次旋转 |
| 适合场景 | 读多写少 | 写多读少 |
| 高度差 | 最多 1 | 可能达到 2 倍 |
| 实现复杂度 | 较简单 | 较复杂 |

## 为什么 Java 用红黑树

`HashMap` 在 JDK 8 之后，当链表长度超过 8 时会转为红黑树。原因是：

1. 链表查找 O(n)，红黑树 O(log n)
2. 红黑树插入代价（最多 2 次旋转）比维持链表高
3. 红黑树的空间效率更好（不需要额外的「平衡因子」）

## 面试追问预判

1. **红黑树的高度最多是多少？**
   - 最多 `2*log₂(n+1)`，由性质 5 保证
   - 最短路径是全黑节点，最长路径是红黑交替

2. **红黑树插入最多需要几次旋转？**
   - 最多 2 次旋转
   - 但重新着色可能沿着路径向上传播多次

3. **删除操作比插入更复杂吗？**
   - 是的。删除操作需要处理「黑色节点删除」导致的路径黑色数量不统一
   - 这时需要「颜色双黑」的概念和处理

4. **红黑树和 B+ 树的区别？**
   - 红黑树：内存中的数据结构，节点通常较小
   - B+ 树：外存友好的数据结构，节点较大（通常等于磁盘块大小），适合多叉

**理解红黑树的关键是理解「为什么需要近似平衡」。** 严格平衡让所有操作都变慢，而近似平衡只在统计意义上保证效率，却大大降低了维护成本。
