# B树与B+树：数据库索引核心

面试官问：**「数据库索引为什么用 B+ 树而不是 B 树？」**

你答：**「B+ 树更扁平，磁盘 I/O 更少。」**

面试官追问：**「为什么 B+ 树更扁平？B 树和 B+ 树的本质区别是什么？」**

你可能需要更深入的理解......

**理解 B+ 树的关键是理解「磁盘 I/O」和「数据库查询特点」。**

## 从二叉树到 B 树

### 为什么二叉树不适合数据库

二叉树（AVL、红黑）的问题：**树太高**。

假设有 100 万条数据，二叉树高度大约是 20 层（log₂ 1,000,000 ≈ 20）。每层可能涉及一次磁盘 I/O，总共 20 次磁盘访问。

而一次磁盘 I/O 的耗时（几毫秒）远大于内存操作（纳秒级）。**磁盘 I/O 是数据库查询的主要瓶颈。**

### B 树的核心思想

**减少树的高度**：让每个节点存储更多的数据。

B 树的规则（以 m 阶 B 树为例）：

| 规则 | 说明 |
|------|------|
| 每个节点最多 m 个子节点 | m 阶 |
| 除根节点外，每个节点至少 ⌈m/2⌉ 个子节点 | 节点不能太「瘦」 |
| 根节点至少 2 个子节点 | 除非是叶子 |
| 所有叶子节点在同一层 | 完美平衡 |
| 有 k 个子节点的节点必有 k-1 个键 | 子节点数 = 键数 + 1 |

```
m=3 的 B 树示例：
        [2 | 5]
       /   |   \
    [1]   [3,4]   [6,7,8]
```

## B 树 vs B+ 树

### 结构对比

**B 树**：
- 每个节点都存储键和数据
- 叶子节点之间没有链表连接
- 非叶子节点也存储数据

**B+ 树**：
- 只有叶子节点存储数据（记录或指向记录的指针）
- 非叶子节点只存储键（用于索引）
- 所有叶子节点通过链表连接

```
B 树：
        [2 | 5]
       /   |   \
    [1]   [3,4]   [6,7,8]
   数据    数据     数据

B+ 树：
        [2 | 5]        <-- 只有键，无数据
       /   |   \
    [2]   [5]   [5]   <-- 指向叶子
    / \   / \   / \
   所有叶子通过链表连接： [1] <-> [2,3,4] <-> [5,6] <-> [7,8,9]
```

### 关键区别

| 特性 | B 树 | B+ 树 |
|------|------|-------|
| 叶子节点 | 存储数据 | 存储全部数据 |
| 非叶子节点 | 可存储数据 | 只存储索引键 |
| 链表 | 无 | 有 |
| 范围查询 | 需要遍历所有路径 | 直接通过链表 |
| 查询稳定性 | 某些查询可能在叶子层之上返回 | 必须在叶子层返回 |
| 空间利用率 | 非叶子节点占用空间 | 更高（全部是叶子） |

## 为什么数据库选择 B+ 树

### 理由一：更少的磁盘 I/O

B+ 树的非叶子节点不存储数据，可以容纳更多的键。

假设：
- 磁盘块大小 = 4KB
- 键大小 = 4 字节
- 指针大小 = 8 字节

**B 树非叶子节点**能存储的键数：`4KB / (4 + 8) ≈ 341`
**B+ 树非叶子节点**能存储的键数：`4KB / (4 + 8) ≈ 341`

但关键区别是：**B+ 树的高度更稳定**，且**范围查询只需定位起点 + 顺序扫描叶子链表**。

### 理由二：范围查询友好

```sql
SELECT * FROM users WHERE age BETWEEN 20 AND 30;
```

B 树需要从根开始查找，然后可能需要回溯。
B+ 树只需要找到第一个满足条件的叶子节点，然后沿着链表顺序扫描即可。

### 理由三：查询更稳定

B 树的查询路径长度不固定（数据可能在任意层级返回）。
B+ 树的查询必须到达叶子层，所以**所有查询的 I/O 次数相同**。

### 理由四：更适合顺序访问

叶子链表支持顺序访问和范围扫描，这在**磁盘预读**中很重要。操作系统会预读相邻的磁盘块，B+ 树的叶子链表让这种预读更有效。

## B+ 树的操作

### 查找

```java
public class BPlusNode {
    boolean isLeaf;
    List<Integer> keys;
    List<BPlusNode> children; // 非叶子节点使用
    List<Data> data;          // 叶子节点使用
    BPlusNode next;           // 叶子链表指针

    public Data search(int key) {
        int idx = Collections.binarySearch(keys, key);

        if (isLeaf) {
            // 叶子节点：找到精确位置
            if (idx >= 0) {
                return data.get(idx);
            }
            return null;
        } else {
            // 非叶子节点：找到合适的子树
            int childIdx = idx >= 0 ? idx + 1 : -(idx + 1);
            return children.get(childIdx).search(key);
        }
    }
}
```

### 插入（分裂）

当叶子节点满了，分裂为两个节点，并向上更新父节点。

```
分裂前（叶子节点）：
[10, 20, 30, 40] | 满了（假设 m=4）

分裂后：
  左节点: [10, 20]     右节点: [30, 40]
      \               /
       链表连接
```

## MySQL InnoDB 中的 B+ 树

InnoDB 索引特点：

1. **聚簇索引**：主键索引的叶子节点存储完整行数据
2. **非聚簇索引**：非主键索引的叶子节点存储主键值
3. **回表**：通过非主键索引查询时，需要先查主键，再查数据

```
聚簇索引：
        [主键]
       /     \
   [主键]    [主键]    <-- 非叶子节点只有主键
   /   \     /   \
 [行数据] [行数据] ...  <-- 叶子节点存储完整行数据

二级索引：
        [索引键 | 主键]
       /             \
   [索引键 | 主键]   ...
   /       \
[主键值]  [主键值]   <-- 叶子节点存储主键值，需要回表
```

## 面试追问预判

1. **B+ 树的高度一般是多少？**
   - 假设 1000 万条数据，每页 16KB，每个索引项 16 字节
   - 每个非叶子节点可存 1000 个索引，高度约 3-4 层
   - 即便 1000 万数据，也只需 3-4 次磁盘 I/O

2. **为什么 Redis 用跳表而不是 B+ 树？**
   - Redis 数据主要在内存，B+ 树的磁盘优化不适用
   - 跳表实现更简单，支持范围查询和按序遍历
   - 跳表的写入性能更好（不需要随机写入和合并）

3. **什么是索引的最左前缀原则？**
   - 对于复合索引 `(a, b, c)`，查询 `a=1` 或 `a=1 AND b=2` 可以使用索引
   - 但查询 `b=2` 或 `c=3` 无法使用该索引

4. **主键为什么要用自增 ID？**
   - 插入时追加到叶子末端，不需要移动其他数据
   - 如果用 UUID 插入是随机的，会导致频繁的页分裂和随机 I/O

**理解 B+ 树的核心是理解「磁盘 I/O」。** 数据库的设计处处都在考虑如何减少磁盘访问次数，B+ 树是最成功的实践之一。
