**你会怎么办？**

大多数人会去查系统目录。但如果有一个人站在旁边，告诉你一个诀窍：「**所有算法类的书，都在 G 区的第四排书架**」——你会不会恍然大悟？

这个思维方式，就是**递归**的核心：**不需要知道完整的解法，只需要知道：问题可以缩小，问题可以分解，而分解后的子问题和原问题本质上是一样的。**

## 递归的本质：自己调用自己

### 什么是递归

递归（Recursion）就是：**函数调用自身**。

听起来简单，但很多初学者在这里卡住了——自己调用自己，怎么就不死循环了？

关键在于：**递归一定要有终止条件。** 没有终止条件的递归，就是死循环；有终止条件的递归，才能把问题逐步缩小，直到收缩到一个 trivial 的基例。

```java
void countDown(int n) {
    if (n <= 0) return;     // 终止条件：到 0 了，停下来
    System.out.println(n);
    countDown(n - 1);       // 递归调用：n 变小了
}
```

这个函数打印 n, n-1, n-2, ..., 1，然后停止。没有 `if (n <= 0) return`，它就会无限打印下去。

### 递归三要素

写递归函数，必须同时满足三个条件：

**第一要素：明确的函数职责**

函数要做什么？把这个问题想清楚，递归就成功了一半。不要一开始就想着"怎么递归"，先想清楚"这个函数负责什么"。

```java
// 职责明确：返回 arr[left..right] 的和
int sum(int[] arr, int left, int right);

// 职责明确：判断 s[left..right] 是否是回文
boolean isPalindrome(String s, int left, int right);
```

**第二要素：递归终止条件（Base Case）**

什么时候问题小到可以直接解决，不需要再递归了？这个边界条件必须明确。

```java
int sum(int[] arr, int left, int right) {
    if (left == right) {
        return arr[left];   // 只有一个元素，直接返回
    }
    // ...
}
```

**第三要素：递归等价关系**

如何把 f(n) 变成 f(n/2)、f(n-1) 或其他更小的子问题？这一步是递归的核心——找到**原问题与子问题之间的关系**。

```java
int sum(int[] arr, int left, int right) {
    if (left == right) return arr[left];
    int mid = (left + right) / 2;
    return sum(arr, left, mid) + sum(arr, mid + 1, right);
    // 问题从 [left..right] 缩小为 [left..mid] 和 [mid+1..right]
    // 两部分加起来，就是原问题的答案
}
```

### 递归调用栈：理解递归的执行过程

递归调用听起来神秘，但本质就是一个**调用栈**。每次递归调用，就像往书架上放一本书；每次返回，就像从书架上取一本书。

```java
int fibonacci(int n) {
    if (n <= 1) return n;
    return fibonacci(n - 1) + fibonacci(n - 2);
}
```

调用 `fibonacci(4)` 的栈变化：

```
fibonacci(4)
  ├─ fibonacci(3)
  │    ├─ fibonacci(2)
  │    │    ├─ fibonacci(1) → return 1
  │    │    └─ fibonacci(0) → return 0
  │    │    └─ return 1 + 0 = 1
  │    └─ fibonacci(1) → return 1
  │    └─ return 1 + 1 = 2
  └─ fibonacci(2)
       ├─ fibonacci(1) → return 1
       ├─ fibonacci(0) → return 0
       └─ return 1 + 0 = 1
  └─ return 2 + 1 = 3
```

调用栈的深度就是递归的深度。栈的容量是有限的，所以递归不能太深——这就是**栈溢出**的原因。

## 经典问题：汉诺塔

### 问题描述

法国数学家爱德华·卢卡斯在 1883 年发明了汉诺塔问题。传说印度教神庙里的僧侣需要移动 64 个金盘，从一根柱子移到另一根柱子，每次移动一个，大盘不能放在小盘上面。神庙建成之时，就是世界毁灭之日。

问题是这样的：有三根柱子 A、B、C，A 柱上有 n 个圆盘（大盘在下，小盘在上）。要求把所有盘子从 A 移动到 C，规则是：

1. 每次只能移动一个盘子
2. 大盘不能放在小盘上面
3. 可以借助 B 柱中转

### 思路分析：从简单到复杂

不要一开始就想着"我要怎么移动 n 个盘子"，这是不可能一步想清楚的。递归的思路是**从最简单的场景出发**。

**n = 1 时：**

直接从 A 移到 C。简单到不需要任何解释。

```text
A: [1]    B: []    C: []
↓ 移动
A: []    B: []    C: [1]
```

**n = 2 时：**

1. 把小盘子（1 号）从 A 移到 B
2. 把大盘子（2 号）从 A 移到 C
3. 把小盘子从 B 移到 C

```text
初始:  A: [2,1]  B: []  C: []
Step1: A: [2]    B: [1] C: []
Step2: A: []     B: [1] C: [2]
Step3: A: []     B: []  C: [2,1]
```

**n = 3 时：**

按照同样的思路——如果能把上面 2 个盘子先移到 B，把最大的盘子移到 C，再把 2 个盘子从 B 移到 C，问题就解决了：

1. 把上面 2 个盘子从 A 借助 C 移到 B
2. 把最大的盘子（3 号）从 A 移到 C
3. 把 2 个盘子从 B 借助 A 移到 C

注意第一步和第三步——**它们本质上和原问题一样，只是规模变小了**。这就是递归的核心：把 f(n) 变成 f(n-1)。

### 代码实现

**Java 版本：**

```java
public class Hanoi {
    public void hanoi(int n, char from, char temp, char to) {
        if (n == 1) {
            System.out.println("将第 1 个盘子从 " + from + " 移到 " + to);
            return;
        }

        // 第一步：把 n-1 个盘子从 A 借助 C 移到 B
        hanoi(n - 1, from, to, temp);

        // 第二步：把最大的盘子（第 n 个）从 A 移到 C
        System.out.println("将第 " + n + " 个盘子从 " + from + " 移到 " + to);

        // 第三步：把 n-1 个盘子从 B 借助 A 移到 C
        hanoi(n - 1, temp, from, to);
    }

    public static void main(String[] args) {
        System.out.println("n = 3 的移动步骤：");
        new Hanoi().hanoi(3, 'A', 'B', 'C');
    }
}
```

输出：

```text
n = 3 的移动步骤：
将第 1 个盘子从 A 移到 C
将第 2 个盘子从 A 移到 B
将第 1 个盘子从 C 移到 B
将第 3 个盘子从 A 移到 C
将第 1 个盘子从 B 移到 A
将第 2 个盘子从 B 移到 C
将第 1 个盘子从 A 移到 C
```

### 时间复杂度分析

汉诺塔的移动次数满足递推关系：

- T(1) = 1（只需移动 1 次）
- T(n) = 2 × T(n-1) + 1（两步递归 + 1 次移动）

展开：

```
T(n) = 2 × T(n-1) + 1
     = 2 × (2 × T(n-2) + 1) + 1
     = 4 × T(n-2) + 3
     = 8 × T(n-3) + 7
     ...
     = 2^(n-1) × T(1) + (2^(n-1) - 1)
     = 2^(n-1) + 2^(n-1) - 1
     = 2^n - 1
```

所以汉诺塔的时间复杂度是 **O(2^n)**。

| n | 移动次数 |
|---|----------|
| 1 | 1 |
| 2 | 3 |
| 3 | 7 |
| 10 | 1023 |
| 20 | 1,048,575 |
| 64 | 18,446,744,073,709,551,615 |

这就是为什么传说中 64 个盘子的汉诺塔，即使僧侣每秒移动一次，也需要 5800 亿年——比宇宙的年龄还长得多。

## 分治策略：分而治之

### 分治的三步法

分治（Divide and Conquer）和递归是亲兄弟，但侧重点不同：

- **递归**：强调"自己调用自己"
- **分治**：强调"把大问题拆成小问题，分别求解，再合并"

分治算法通常分为三步：

```
分（Divide）：    把原问题分成若干个子问题
治（Conquer）：   递归地解决子问题
合（Combine）：   把子问题的解合并为原问题的解
```

### 经典应用：归并排序

归并排序是分治策略最典型的应用——没有之一。

#### 思路

把数组分成两半，分别排序，再合并。

```
原始数组:   [8, 3, 1, 7, 4, 6, 5, 2]

分:
[8, 3, 1, 7]    [4, 6, 5, 2]

再分:
[8, 3]   [1, 7]   [4, 6]   [5, 2]

再分:
[8]  [3]  [1]  [7]  [4]  [6]  [5]  [2]
（每个子数组只有一个元素，有序了）

治（两两合并）:
[3, 8]   [1, 7]   [4, 6]   [2, 5]

再合并:
[1, 3, 7, 8]   [2, 4, 5, 6]

最终合并:
[1, 2, 3, 4, 5, 6, 7, 8]
```

#### Java 实现

```java
public class MergeSort {
    public void mergeSort(int[] arr, int left, int right) {
        // 治：递归终止条件——子数组只有一个元素
        if (left >= right) return;

        // 分：找到中点
        int mid = left + (right - left) / 2;

        // 分别对左右两部分排序
        mergeSort(arr, left, mid);
        mergeSort(arr, mid + 1, right);

        // 合：合并两个有序子数组
        merge(arr, left, mid, right);
    }

    private void merge(int[] arr, int left, int mid, int right) {
        int[] temp = new int[right - left + 1];
        int i = left;     // 左半部分指针
        int j = mid + 1;  // 右半部分指针
        int k = 0;        // 临时数组指针

        // 合并：谁小谁先放
        while (i <= mid && j <= right) {
            if (arr[i] <= arr[j]) {
                temp[k++] = arr[i++];
            } else {
                temp[k++] = arr[j++];
            }
        }

        // 左边剩余的
        while (i <= mid) temp[k++] = arr[i++];

        // 右边剩余的
        while (j <= right) temp[k++] = arr[j++];

        // 复制回原数组
        for (int p = 0; p < k; p++) {
            arr[left + p] = temp[p];
        }
    }

    public static void main(String[] args) {
        int[] arr = {8, 3, 1, 7, 4, 6, 5, 2};
        new MergeSort().mergeSort(arr, 0, arr.length - 1);
        for (int v : arr) System.out.print(v + " ");
        // 输出: 1 2 3 4 5 6 7 8
    }
}
```

#### C 语言实现

```c
#include <stdio.h>

void merge(int arr[], int left, int mid, int right) {
    int n1 = mid - left + 1;
    int n2 = right - mid;
    int L[n1], R[n2];

    for (int i = 0; i < n1; i++) L[i] = arr[left + i];
    for (int j = 0; j < n2; j++) R[j] = arr[mid + 1 + j];

    int i = 0, j = 0, k = left;
    while (i < n1 && j < n2) {
        if (L[i] <= R[j]) arr[k++] = L[i++];
        else              arr[k++] = R[j++];
    }
    while (i < n1) arr[k++] = L[i++];
    while (j < n2) arr[k++] = R[j++];
}

void mergeSort(int arr[], int left, int right) {
    if (left >= right) return;

    int mid = left + (right - left) / 2;
    mergeSort(arr, left, mid);
    mergeSort(arr, mid + 1, right);
    merge(arr, left, mid, right);
}

int main() {
    int arr[] = {8, 3, 1, 7, 4, 6, 5, 2};
    int n = sizeof(arr) / sizeof(arr[0]);
    mergeSort(arr, 0, n - 1);
    for (int i = 0; i < n; i++) printf("%d ", arr[i]);
    return 0;
}
```

#### Python 实现

```python
def merge_sort(arr):
    if len(arr) <= 1:
        return arr

    mid = len(arr) // 2
    left = merge_sort(arr[:mid])
    right = merge_sort(arr[mid:])

    return merge(left, right)


def merge(left, right):
    result = []
    i = j = 0
    while i < len(left) and j < len(right):
        if left[i] <= right[j]:
            result.append(left[i])
            i += 1
        else:
            result.append(right[j])
            j += 1
    result.extend(left[i:])
    result.extend(right[j:])
    return result


if __name__ == "__main__":
    arr = [8, 3, 1, 7, 4, 6, 5, 2]
    sorted_arr = merge_sort(arr)
    print(sorted_arr)  # [1, 2, 3, 4, 5, 6, 7, 8]
```

### 归并排序复杂度分析

归并排序的时间复杂度分析用到了**主定理**：

```
T(n) = 2 × T(n/2) + O(n)
  分        治              合
```

- 分（Divide）：找中点只需要 O(1)
- 治（Conquer）：递归解决两个子问题，各 T(n/2)
- 合（Combine）：合并两个有序数组需要 O(n)

根据主定理：T(n) = a × T(n/b) + f(n)，其中 a = 2, b = 2, f(n) = O(n)。

因为 n^(log_b a) = n^(log_2 2) = n^1 = n，所以 f(n) = Θ(n) 属于第二种情况。

**T(n) = O(n log n)**

| 排序算法 | 最好 | 平均 | 最坏 | 空间复杂度 |
|---|---|---|---|---|
| 归并排序 | O(n log n) | O(n log n) | O(n log n) | O(n) |
| 快速排序 | O(n log n) | O(n log n) | O(n²) | O(log n) |
| 堆排序 | O(n log n) | O(n log n) | O(n log n) | O(1) |

归并排序的最大特点是**任何情况下都是 O(n log n)**，是稳定的排序算法。代价是 O(n) 的空间复杂度。

## 递归思维：从大到小的逆向思维

### 常见的思维误区

大多数初学者面对递归问题时，习惯从"小规模"往"大规模"想：

```
f(1) = 1
f(2) = f(1) + 2 = 1 + 2 = 3
f(3) = f(2) + 3 = 3 + 3 = 6
f(n) = f(n-1) + n
```

这个思路可以写出正确的代码，但**递归应该是反过来想的**：从大规模想小规模。

### 正确的递归思维

**第一步：假设 f(n-1) 已经解决了。** 不要去想 f(n-1) 怎么求的，只需要知道它已经求出来了。

**第二步：想清楚 f(n) 和 f(n-1) 的关系。**

**第三步：写出从 f(n-1) 到 f(n) 的步骤。**

以汉诺塔为例：不要想"4 个盘子怎么移动"，而是假设"3 个盘子已经移到 B 柱了，现在怎么把最大的移到 C"。

### 画递归树

遇到复杂递归，先画递归树。递归树是理解递归执行过程的利器。

以斐波那契数列为例：

```java
int fib(int n) {
    if (n <= 1) return n;
    return fib(n - 1) + fib(n - 2);
}
```

```
fib(5)
├─ fib(4)
│  ├─ fib(3)
│  │  ├─ fib(2)
│  │  │  ├─ fib(1) = 1
│  │  │  └─ fib(0) = 0
│  │  │  └─ return 1 + 0 = 1
│  │  └─ fib(1) = 1
│  │  └─ return 1 + 1 = 2
│  └─ fib(2)
│     ├─ fib(1) = 1
│     └─ fib(0) = 0
│     └─ return 1 + 0 = 1
│  └─ return 2 + 1 = 3
└─ fib(3)
   ├─ fib(2)
   │  ├─ fib(1) = 1
   │  └─ fib(0) = 0
   │  └─ return 1 + 0 = 1
   └─ fib(1) = 1
   └─ return 1 + 1 = 2
└─ return 3 + 2 = 5
```

从递归树可以看出：fib(3) 被计算了两次，fib(2) 被计算了三次。这就是为什么朴素递归 fib 的复杂度是 O(2^n)。

## 递归 vs 迭代：什么时候用哪个

### 递归的优点

- 代码简洁，逻辑清晰
- 天然契合分治思想
- 适合树结构、图结构的遍历

### 递归的缺点

- 函数调用有额外开销（栈帧创建、参数传递）
- 递归深度受栈容量限制
- 可能有大量重复计算（如 fib）

### 什么时候用递归

| 场景 | 推荐方式 |
|---|---|
| 树/图的遍历 | 递归最自然 |
| 分治（归并、快速） | 递归最清晰 |
| 简单数学递归（阶乘、斐波那契） | 递归或迭代皆可 |
| 递归深度可能很大 | 改写为迭代 |
| 需要记忆化 | 递归 + 备忘录 |

### 尾递归优化

如果递归调用是函数的最后一条语句，编译器可以进行**尾递归优化**，把递归调用改写成循环，避免栈增长：

```java
// 普通递归（有返回值，需要等递归返回后才能返回）
int factorial(int n) {
    if (n <= 1) return 1;
    return n * factorial(n - 1);  // 不是 tail call，因为乘法要等递归结果
}

// 尾递归（递归调用在 return 语句中，不再做其他操作）
int factorialTail(int n, int acc) {
    if (n <= 1) return acc;
    return factorialTail(n - 1, n * acc);  // tail call，编译器可以优化
}
```

Java 不保证尾递归优化（大多数 JVM 不做这个优化），但 Python 和 Scala 可以通过 tailrec 关键字强制优化。

## 面试追问

### 递归和非递归（迭代）哪个更好？

没有绝对的答案，要看具体场景。递归代码更简洁、更容易理解；迭代性能更稳定、没有栈溢出风险。**能用迭代实现的递归，都应该能改写成迭代**——反过来不一定成立（某些递归模式很难改写）。

### 斐波那契数列的递归为什么慢？如何优化？

朴素递归 fib 的时间复杂度是 O(2^n)，因为 fib(n) 需要计算 fib(n-1) 和 fib(n-2)，而 fib(n-1) 又需要计算 fib(n-2)，大量子问题被重复计算了。

三种优化方案：

```java
// 方案 1：记忆化递归（自顶向下）
Map<Integer, Integer> memo = new HashMap<>();
int fibMemo(int n) {
    if (n <= 1) return n;
    if (memo.containsKey(n)) return memo.get(n);
    int result = fibMemo(n - 1) + fibMemo(n - 2);
    memo.put(n, result);
    return result;
}  // O(n) 时间，O(n) 空间

// 方案 2：动态规划（自底向上）
int fibDP(int n) {
    if (n <= 1) return n;
    int[] dp = new int[n + 1];
    dp[0] = 0; dp[1] = 1;
    for (int i = 2; i <= n; i++) dp[i] = dp[i-1] + dp[i-2];
    return dp[n];
}  // O(n) 时间，O(n) 空间

// 方案 3：空间优化
int fibOptimized(int n) {
    if (n <= 1) return n;
    int prev2 = 0, prev1 = 1;
    for (int i = 2; i <= n; i++) {
        int curr = prev1 + prev2;
        prev2 = prev1;
        prev1 = curr;
    }
    return prev1;
}  // O(n) 时间，O(1) 空间
```

### 递归的栈溢出怎么解决？

- **改写成迭代**：最通用的方案，但代码可能更复杂
- **尾递归优化**：编译器支持时可以自动优化，但 Java JVM 大多不做
- **手动增大栈大小**：JVM 参数 `-Xss` 可以增大线程栈大小，但这只是推迟问题，不是解决问题
- **使用堆模拟栈**：把递归改成显式的栈数据结构，把调用栈放到堆上

**记住：递归的核心是「分解问题，递归求解，合并结果」。没有终止条件的递归是灾难；不理解调用栈的递归是定时炸弹。**
