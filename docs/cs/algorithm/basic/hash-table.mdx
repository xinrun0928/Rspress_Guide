# 哈希表：散列函数与冲突解决

**哈希表为什么查询这么快？** O(1) 的时间复杂度，真的能做到吗？

答案是：**在理想情况下，可以做到。** 但这需要好的散列函数和合适的冲突解决方法。

面试中，哈希表几乎是必考的内容——不是因为它难，而是因为它**太重要了**。几乎所有的查询优化问题，背后都有哈希表的影子。

## 哈希表的核心思想

哈希表的本质是：**用空间换时间**。把 key 通过一个函数（散列函数）映射到数组的某个位置，直接访问这个位置就能拿到 value。

```
key -> [散列函数] -> 数组下标 -> 读取 value
```

**理想情况**：散列函数把每个 key 均匀分布到数组的不同位置，查询就是 O(1)。

**实际情况**：key 的数量可能比数组长度大得多，不同的 key 可能映射到同一个位置——这就是**冲突**。

## 散列函数的设计

### 常见的散列函数

```java
// 取余法（最常用）
int hash(int key, int tableSize) {
    return Math.abs(key % tableSize);
}

// 乘法散列法（用于浮点数 key）
int hash(double key, int tableSize) {
    int A = 2654435761; // 黄金分割比例的倒数
    return (int) (tableSize * ((key * A) % 1.0));
}

// String 的散列函数（Java 中的实现）
int hashCode(String s) {
    int h = 0;
    for (int i = 0; i < s.length(); i++) {
        h = 31 * h + s.charAt(i);
    }
    return h;
}
```

### 好的散列函数特点

| 特点 | 说明 |
|------|------|
| 确定性 | 相同输入永远产生相同输出 |
| 均匀分布 | key 能均匀分布在数组各处 |
| 高效计算 | 计算成本低 |
| 雪崩效应 | 输入微小变化导致输出大幅变化 |

## 冲突解决方法

### 方法一：链地址法（拉链法）

每个数组位置存放一个链表，所有映射到同一位置的元素挂在同一条链表上：

```java
class HashMap<K, V> {
    private static class Entry<K, V> {
        K key;
        V value;
        Entry<K, V> next;
    }

    private Entry<K, V>[] table;
    private int size;

    public HashMap(int capacity) {
        table = new Entry[capacity];
    }

    public void put(K key, V value) {
        int index = hash(key) % table.length;
        Entry<K, V> head = table[index];

        // 检查是否已存在
        while (head != null) {
            if (head.key.equals(key)) {
                head.value = value; // 更新
                return;
            }
            head = head.next;
        }

        // 头插法插入新节点
        Entry<K, V> newNode = new Entry<>(key, value, table[index]);
        table[index] = newNode;
        size++;
    }

    public V get(K key) {
        int index = hash(key) % table.length;
        Entry<K, V> head = table[index];

        while (head != null) {
            if (head.key.equals(key)) {
                return head.value;
            }
            head = head.next;
        }
        return null;
    }
}
```

**性能分析**：

- 如果散列函数设计得好，链表长度期望为 n/m（n 是元素数，m 是数组长度）
- 查找时间：O(1) 到 O(n)，取决于链表长度
- Java 8 对链表做了优化：链表长度超过 8 时转为红黑树，查找变成 O(log n)

### 方法二：开放地址法

冲突时不创建链表，而是**探测其他空位置**存放。

```java
public int linearProbe(int key, int i) {
    // 线性探测：依次检查下一个位置
    return (hash(key) + i) % tableSize;
}

public int quadraticProbe(int key, int i) {
    // 二次探测：探测间隔是 i²
    return (hash(key) + i * i) % tableSize;
}

public int doubleHash(int key, int i) {
    // 双重散列：用另一个散列函数计算步长
    int h2 = secondHash(key);
    return (hash(key) + i * h2) % tableSize;
}
```

### 两种方法对比

| 方法 | 优点 | 缺点 |
|------|------|------|
| 链地址法 | 实现简单，删除方便 | 链表长时性能下降 |
| 开放地址法 | 无额外指针，空间利用率高 | 删除困难，存在聚集问题 |

## 扩容与缩容

当元素数量超过阈值（负载因子 = n/m），需要扩容：

```java
private static final double LOAD_FACTOR = 0.75;

public void put(K key, V value) {
    if (size >= table.length * LOAD_FACTOR) {
        resize(table.length * 2); // 扩容
    }
    // ... 插入逻辑
}

private void resize(int newCapacity) {
    Entry<K, V>[] newTable = new Entry[newCapacity];
    // 重新计算所有元素的位置（哈希表扩容后必须 rehash）
    for (Entry<K, V> entry : table) {
        while (entry != null) {
            int index = hash(entry.key) % newCapacity;
            // 移动到新表
            Entry<K, V> next = entry.next;
            entry.next = newTable[index];
            newTable[index] = entry;
            entry = next;
        }
    }
    table = newTable;
}
```

**负载因子的选择**：
- 太小：空间浪费
- 太大：冲突增多，性能下降
- Java `HashMap` 默认 0.75，是时间和空间的平衡

## 面试高频应用

### 应用一：两数之和

```java
public int[] twoSum(int[] nums, int target) {
    Map<Integer, Integer> map = new HashMap<>();
    for (int i = 0; i < nums.length; i++) {
        int complement = target - nums[i];
        if (map.containsKey(complement)) {
            return new int[]{map.get(complement), i};
        }
        map.put(nums[i], i);
    }
    return new int[]{-1, -1};
}
```

**关键**：边遍历边存入 map，如果 map 中有需要的数，立即返回。

### 应用二：LRU 缓存

```java
class LRUCache {
    private Map<Integer, Integer> map;
    private int capacity;

    public LRUCache(int capacity) {
        this.capacity = capacity;
        // LinkedHashMap 支持按访问顺序维护
        map = new LinkedHashMap<>(capacity, 0.75f, true) {
            protected boolean removeEldestEntry(Map.Entry eldest) {
                return size() > LRUCache.this.capacity;
            }
        };
    }

    public int get(int key) {
        return map.getOrDefault(key, -1);
    }

    public void put(int key, int value) {
        map.put(key, value);
    }
}
```

### 应用三：字符串字符统计

```java
public boolean isAnagram(String s, String t) {
    if (s.length() != t.length()) return false;

    Map<Character, Integer> count = new HashMap<>();
    for (char c : s.toCharArray()) {
        count.put(c, count.getOrDefault(c, 0) + 1);
    }
    for (char c : t.toCharArray()) {
        count.put(c, count.getOrDefault(c, 0) - 1);
        if (count.get(c) < 0) return false;
    }
    return true;
}
```

## 哈希表的局限性

:::warning 注意事项

1. **哈希碰撞攻击**：恶意构造大量相同哈希值的 key，使查询退化为 O(n)
2. **哈希函数的选择很重要**：不好的哈希函数会导致聚集，降低性能
3. **哈希表不保证顺序**：如果需要有序，使用 B+ 树或跳表

:::

## 总结

| 概念 | 说明 |
|------|------|
| 散列函数 | 把 key 映射到数组下标的函数 |
| 冲突 | 不同 key 映射到同一位置 |
| 链地址法 | 冲突的元素用链表串联 |
| 开放地址法 | 探测其他空位置 |
| 负载因子 | n/m，衡量哈希表填充程度 |
| 再散列 | 扩容后重新计算所有元素位置 |

**哈希表的核心**：通过散列函数把 key 映射到数组，实现 O(1) 的查询。但要记住——**没有完美的哈希函数，只有合适的选择**。

## 面试追问预判

1. **HashMap 在 JDK 8 之后有什么优化？**
   - 链表长度超过 8 时转为红黑树
   - 树节点比链表节点占用更多空间，但查找更快

2. **HashMap 是线程安全的吗？**
   - 不是。线程安全版本用 `ConcurrentHashMap`
   - `Hashtable` 是线程安全的，但性能差（全局锁）

3. **如果自定义对象作为 HashMap 的 key，需要注意什么？**
   - 必须重写 `equals()` 和 `hashCode()` 方法
   - 如果两个对象 `equals()` 为 true，`hashCode()` 必须相同
   - 保持 `hashCode()` 的稳定性（对象内容不变时结果不变）

4. **HashMap 的容量为什么是 2 的幂次？**
   - 因为 `hash % n` 等价于 `hash & (n - 1)`，位运算更快
   - 只有 2 的幂次减 1 的二进制是全 1

5. **哈希表和搜索树（如红黑树）的区别？**
   - 哈希表：O(1) 查询，不保证顺序
   - 搜索树：O(log n) 查询，保证有序
   - 数据量大、需范围查询时用搜索树；等值查询为主时用哈希表
