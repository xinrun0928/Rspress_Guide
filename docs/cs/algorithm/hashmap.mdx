# 哈希表原理

> 目标级别：P5/P6

面试官问：「HashMap 是怎么实现的？」你回答「数组 + 链表/红黑树」——然后面试官追问：「哈希冲突怎么解决？」「为什么重写 equals 要重写 hashCode？」「ConcurrentHashMap 怎么实现线程安全？」

哈希表是面试中的高频考点，理解哈希表的原理对于理解各种 Map 实现至关重要。

## 一、哈希表基础

### 1.1 为什么需要哈希表

```
数组查找：O(1)，但需要已知下标
链表查找：O(n)，需要遍历

哈希表目标：O(1) 查找
```

### 1.2 哈希函数

```
哈希函数：将 Key 转换为数组下标

理想情况：
- 不同的 Key 映射到不同的下标
- 分布均匀

常见哈希函数：
- 取模法：hash % tableSize
- 乘法哈希：hash * 31
- 字符串哈希：多项式哈希
```

## 二、哈希冲突

### 2.1 什么是哈希冲突

```
哈希冲突：不同 Key 映射到相同下标

例如：
- "name" → hash = 5
- "age" → hash = 5
→ 冲突！

冲突解决方式：
1. 开放地址法
2. 链地址法（拉链法）
```

### 2.2 开放地址法

```java
// 线性探测
private int linearProbing(int hash, int i) {
    return (hash + i) % tableSize;
}

// 二次探测
private int quadraticProbing(int hash, int i) {
    return (hash + i * i) % tableSize;
}

// 双重哈希
private int doubleHashing(int hash, int i) {
    return (hash + i * hash2) % tableSize;
}
```

### 2.3 链地址法

```java
// 链地址法：数组 + 链表
class HashMap<K, V> {
    Node<K, V>[] table;

    static class Node<K, V> {
        int hash;
        K key;
        V value;
        Node<K, V> next;
    }

    public V put(K key, V value) {
        int hash = hash(key);
        int index = hash % table.length;

        for (Node<K, V> e = table[index]; e != null; e = e.next) {
            if (e.hash == hash && e.key.equals(key)) {
                V oldValue = e.value;
                e.value = value;
                return oldValue;
            }
        }

        // 头插法
        Node<K, V> newNode = new Node<>(hash, key, value, table[index]);
        table[index] = newNode;
        size++;
        return null;
    }
}
```

## 三、HashMap 实现

### 3.1 JDK 1.8 之前：数组 + 链表

```
JDK 1.7 HashMap 结构：
[链表头] → [节点] → [节点] → null

问题：链表过长时查找效率退化到 O(n)
```

### 3.2 JDK 1.8 之后：数组 + 链表/红黑树

```
JDK 1.8 HashMap 结构：
[链表头] → [节点] → ... → [节点] → null
                                    ↓
                                 红黑树
```

**链表转红黑树的条件**：

| 条件 | 值 |
|------|------|
| 链表长度阈值 | 8 |
| 数组长度阈值 | 64 |
| 红黑树节点阈值 | 6（退化回链表） |

### 3.3 HashMap 核心属性

```java
public class HashMap<K, V> {
    transient Node<K, V>[] table;
    transient int size;
    int threshold;           // 扩容阈值
    final float loadFactor;  // 负载因子，默认 0.75
}

// 扩容时机
if (size >= threshold) {
    resize();  // threshold = table.length * loadFactor
}
```

## 四、面试题精讲

### 🔴 【高频】HashMap 实现原理

**问题**：HashMap 是怎么实现的？

**标准答案**：

```
HashMap 实现原理（JDK 1.8）：

1. 底层结构
   - 数组 + 链表 + 红黑树
   - 数组下标通过 hash(key) 计算

2. 哈希函数
   -扰动函数：h = key.hashCode() ^ (h >>> 16)
   - 减少哈希碰撞

3. 哈希冲突
   - 链表解决（拉链法）
   - 链表长度 >= 8 且数组长度 >= 64 时转红黑树

4. 扩容机制
   - 负载因子 0.75
   - 扩容翻倍
   - 重新计算所有节点位置（rehash）
```

### 🟡 【中频】hashCode 和 equals

**问题**：为什么重写 equals 要重写 hashCode？

**标准答案**：

```
hashCode 和 equals 的契约：

1. equals 相等的两个对象，hashCode 必须相等
2. hashCode 相等的两个对象，equals 不一定相等

如果不重写 hashCode：
- 两个对象 equals 相等
- 但 hashCode 不一定相等
- 导致散列表行为异常（找不到对象）

正确做法：
- 重写 equals 时必须重写 hashCode
- 保持契约：相等的对象必须产生相同的哈希值
```

### 🟡 【中频】JDK 1.8 优化

**问题**：JDK 1.8 对 HashMap 做了什么优化？

**标准答案**：

```
JDK 1.8 优化：

1. 数组 + 链表 + 红黑树
   - 链表过长时查找效率退化
   - 长度 >= 8 时转红黑树
   - 红黑树查找 O(log n)

2. 扰动函数优化
   - JDK 1.7：h ^= h >>> 16（一次）
   - JDK 1.8：key.hashCode() ^ (h >>> 16)

3. 插入方式
   - JDK 1.7：头插法（插入链表头部）
   - JDK 1.8：尾插法（插入链表尾部）
   - 避免循环链表问题

4. 扩容时机
   - 当链表长度 >= 8 且数组长度 < 64 时，先扩容
```

## 五、对比总结

### 哈希冲突解决方式对比

| 方式 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| 开放地址法 | 无额外指针 | 聚集效应 | 小数据量 |
| 链地址法 | 简单，删除方便 | 指针开销 | 大数据量 |
| 再哈希法 | 无聚集 | 哈希函数计算量大 | - |
| 建立公共溢出区 | 简单 | 哈希表利用率低 | - |

### HashMap vs HashTable vs ConcurrentHashMap

| 维度 | HashMap | HashTable | ConcurrentHashMap |
|------|---------|-----------|-------------------|
| 线程安全 | 否 | 是 | 是 |
| 性能 | 高 | 低（synchronized） | 高（CAS） |
| null 支持 | key/value 都可 | 都不可以 | value 不可以 |
| 迭代安全性 | 否 | 是（Enumeration） | 是（弱一致性） |

## 六、扩展思考

### 💡 ConcurrentHashMap 实现

```java
// ConcurrentHashMap 原理（JDK 1.8）

// 1. 数据结构：数组 + 链表/红黑树
// 2. 线程安全：CAS + synchronized
// 3. 锁粒度：数组每个位置一个锁

public V put(K key, V value) {
    int hash = spread(key.hashCode());
    Node<K, V>[] tab = table;

    // CAS 初始化数组
    if (tab == null) {
        tab = initTable();
    }

    int index = hash & (tab.length - 1);
    Node<K, V> first = tabAt(tab, index);

    // synchronized 锁住当前位置
    synchronized (first) {
        // 链表/红黑树操作
    }
}
```

### 💡 负载因子

```
负载因子的选择：

0.75（默认）：
- 时间-空间平衡
- 扩容阈值合理
- 链表/红黑树转换阈值

负载因子大：
- 空间利用率高
- 冲突概率高
- 查找效率下降

负载因子小：
- 空间利用率低
- 冲突概率低
- 扩容频繁
```
