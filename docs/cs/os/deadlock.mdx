## 一、死锁定义

### 1.1 什么是死锁

死锁（Deadlock）是指两个或多个进程在执行过程中，因争夺资源而造成一种相互等待的现象，若无外力干预，它们都将无法推进下去。

```
死锁示例：

进程 A：持有资源 R1，等待资源 R2
进程 B：持有资源 R2，等待资源 R1

A ───持有 R1──→ [R1]
                  ↑
                  │ 等待 R2
                  │
[R2] ←───持有 R2─── B

双方都在等待对方释放资源，形成循环等待。
```

### 1.2 死锁的四个必要条件

死锁的发生必须同时满足以下四个条件（Coffman 条件）：

| 条件 | 说明 |
|------|------|
| 互斥条件 | 资源只能被一个进程占用 |
| 持有并等待 | 进程持有资源的同时请求其他资源 |
| 不可抢占 | 已分配资源不能被强制释放 |
| 循环等待 | 形成进程-资源的循环等待链 |

## 二、四个必要条件详解

### 2.1 互斥条件（Mutual Exclusion）

资源在同一时刻只能被一个进程使用。

```
互斥资源的例子：
- 打印机（同一时刻只能一个任务打印）
- 文件锁（同一时刻只能一个进程写入）
- 数据库连接（同一时刻只能一个线程使用）

非互斥资源的例子：
- 可重入代码（同一时刻可被多个进程执行）
- 共享数据（读操作可并发）
```

### 2.2 持有并等待（Hold and Wait）

进程在持有已有资源的同时，请求其他资源。

```
场景：
进程 A：
  持有资源 R1
  等待获取资源 R2

进程 B：
  持有资源 R2
  等待获取资源 R1
```

**打破方式**：一次性申请所有需要的资源，或者释放已持有资源后再申请。

### 2.3 不可抢占（No Preemption）

已分配给进程的资源，在进程主动释放前，不能被强行夺走。

```
不可抢占资源：
- 文件锁（必须显式释放）
- 数据库事务
- 进程持有的互斥锁

可抢占资源：
- CPU 时间片（时间片耗尽被抢占）
- 内存页面（可以被换出）
```

**打破方式**：如果资源可以被抢占，死锁就不可能发生。

### 2.4 循环等待（Circular Wait）

形成一个进程-资源的循环等待链。

```
循环等待链：
P0 持有 R0，等待 R1
P1 持有 R1，等待 R2
...
Pn 持有 Rn，等待 R0

┌─→ P0 → R0 → P1 → R1 → P2 → R2
└──────────────────────────────────────┘
```

**注意**：循环等待是持有并等待的进一步强化，单个进程无法形成死锁。

## 三、死锁 vs 活锁 vs 饥饿

### 3.1 概念对比

| 类型 | 状态 | 原因 | 能否继续执行 |
|------|------|------|--------------|
| 死锁 | 阻塞 | 相互等待 | 不能 |
| 活锁 | 运行（不断重试） | 相互让步 | 能（但无法推进） |
| 饥饿 | 阻塞 | 资源分配不公平 | 长时间不能 |

### 3.2 死锁（Deadlock）

```
特征：
- 进程完全阻塞，无法继续执行
- 需要外力干预才能恢复（如杀死进程、强制释放资源）
- 不会主动消耗 CPU 资源

示例：
线程 A 持有锁 L1，等待锁 L2
线程 B 持有锁 L2，等待锁 L1
```

### 3.3 活锁（Livelock）

```
特征：
- 进程在运行，但无法推进
- 不断检测冲突并让步
- 主动消耗 CPU 资源

示例：
线程 A 获取 L1 失败，后退释放 L1
线程 B 获取 L2 失败，后退释放 L2
线程 A 又获取 L1 失败...（循环）

解决：加入随机延迟或退让策略
```

### 3.4 饥饿（Starvation）

```
特征：
- 进程可以执行，但长时间得不到资源
- 资源被其他进程持续占用
- 高优先级进程一直占用资源，低优先级进程无法执行

示例：
- SJF 中，长作业一直得不到调度
- 打印机队列中，小文件一直排在后面

解决：
- 公平调度
- 资源老化（等待时间越长，优先级越高）
```

## 四、死锁示例

### 4.1 Java 死锁示例

```java
public class DeadlockDemo {
    private static final Object lockA = new Object();
    private static final Object lockB = new Object();

    public static void main(String[] args) {
        Thread threadA = new Thread(() -> {
            synchronized (lockA) {
                System.out.println("线程 A 获取了锁 A");
                try {
                    Thread.sleep(100);
                } catch (InterruptedException e) {}
                synchronized (lockB) {
                    System.out.println("线程 A 获取了锁 B");
                }
            }
        });

        Thread threadB = new Thread(() -> {
            synchronized (lockB) {
                System.out.println("线程 B 获取了锁 B");
                try {
                    Thread.sleep(100);
                } catch (InterruptedException e) {}
                synchronized (lockA) {
                    System.out.println("线程 B 获取了锁 A");
                }
            }
        });

        threadA.start();
        threadB.start();
    }
}
```

### 4.2 数据库死锁示例

```sql
-- 事务 1
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;  -- 锁定 id=1
UPDATE accounts SET balance = balance + 100 WHERE id = 2;  -- 等待锁定 id=2

-- 事务 2（并发执行）
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 2;  -- 锁定 id=2
UPDATE accounts SET balance = balance + 100 WHERE id = 1;  -- 等待锁定 id=1

-- 死锁发生，双方都在等待对方释放锁
```

### 4.3 系统资源死锁

```
打印机和扫描仪都只有一个：

进程 P1：使用打印机，需要扫描仪
进程 P2：使用扫描仪，需要打印机

P1 占用打印机，等待扫描仪
P2 占用扫描仪，等待打印机
→ 死锁
```

## 五、死锁预防

### 5.1 预防策略

打破死锁的四个条件之一即可预防死锁：

| 条件 | 预防方法 |
|------|----------|
| 互斥条件 | 无（互斥资源本身特性） |
| 持有并等待 | 一次性申请所有资源 / 先释放再申请 |
| 不可抢占 | 允许抢占资源 |
| 循环等待 | 按序申请资源 |

### 5.2 按序申请资源

```
打破循环等待：将资源编号，进程必须按编号递增顺序申请

资源编号：R1 < R2 < R3

正确申请：
  持有 R1 → 申请 R2 → 申请 R3

错误申请：
  持有 R2 → 申请 R1  （违反顺序）

效果：不可能形成循环等待
```

```java
// 按序申请资源的示例
public void processWithOrder() {
    synchronized (lock1) {  // 先申请 R1
        synchronized (lock2) {  // 再申请 R2
            synchronized (lock3) {  // 最后申请 R3
                // 业务逻辑
            }
        }
    }
}
```

### 5.3 一次性申请所有资源

```
进程在开始前，一次性申请所有需要的资源

优点：打破持有并等待
缺点：
- 需要预先知道资源需求
- 资源利用率低
- 可能导致饥饿
```

## 六、面试题精讲

### 🔴 【高频】死锁的四个必要条件

**问题**：死锁的四个必要条件是什么？

**标准答案**：

```
死锁必须同时满足以下四个条件（Coffman 条件）：

1. 互斥条件
   - 资源在同一时刻只能被一个进程使用
   - 如：互斥锁、打印机、文件锁

2. 持有并等待
   - 进程持有已有资源的同时，请求其他资源
   - 如：持有锁 A，同时等待锁 B

3. 不可抢占
   - 已分配资源不能被强行夺走
   - 必须在进程主动释放后才能被其他进程使用

4. 循环等待
   - 形成进程-资源的循环等待链
   - P0 持有 R0 等待 R1，P1 持有 R1 等待 R0
```

### 🟡 【中频】死锁、活锁、饥饿的区别

**问题**：死锁、活锁、饥饿有什么区别？

**标准答案**：

```
死锁（Deadlock）：
- 状态：进程阻塞，无法继续执行
- 原因：相互等待资源
- 解决：需要外力干预

活锁（Livelock）：
- 状态：进程运行，但不断检测冲突并让步
- 原因：相互让步，导致循环重试
- 解决：加入随机延迟

饥饿（Starvation）：
- 状态：进程可以执行，但长时间得不到资源
- 原因：资源分配不公平
- 解决：公平调度、资源老化

共同点：都无法正常完成
区别：
- 死锁：完全不运行
- 活锁：不断运行但不推进
- 饥饿：长时间等待后可能获得资源
```

### 🟡 【中频】如何打破死锁条件

**问题**：如何打破死锁的四个条件？

**标准答案**：

```
1. 打破互斥条件
   - 几乎不可能（互斥是资源的固有特性）

2. 打破持有并等待
   - 方法一：一次性申请所有资源
   - 方法二：先释放已持有资源，再申请新资源
   - 缺点：资源利用率低

3. 打破不可抢占
   - 允许抢占资源
   - 缺点：不适用于所有资源（如事务锁）

4. 打破循环等待
   - 对资源进行编号，按固定顺序申请
   - 缺点：需要预先规划资源编号
```

## 七、常见陷阱与易错点

### ⚠️ 陷阱一：混淆死锁和活锁

| 状态 | 说明 |
|------|------|
| 死锁 | 进程阻塞，不消耗 CPU |
| 活锁 | 进程运行，消耗 CPU 但不推进 |

### ⚠️ 陷阱二：认为四个条件是充分条件

四个条件是**必要条件**（死锁必须满足），但不是**充分条件**（满足四个条件不一定死锁）。存在满足四个条件但不发生死锁的情况。

### ⚠️ 陷阱三：忽略资源数量

死锁不仅发生在两个进程之间，也可能发生在多个进程之间（N 个进程也可能形成死锁）。

### ⚠️ 陷阱四：混淆不可抢占和可抢占

只有**不可抢占资源**才会导致死锁。**可抢占资源**（如 CPU 时间片）不会导致死锁。

## 八、对比总结

### 死锁 vs 活锁 vs 饥饿

| 维度 | 死锁 | 活锁 | 饥饿 |
|------|------|------|------|
| 进程状态 | 阻塞 | 运行 | 阻塞/运行 |
| CPU 消耗 | 不消耗 | 持续消耗 | 不一定 |
| 推进能力 | 无法推进 | 无法推进 | 可能推进 |
| 恢复方式 | 外力干预 | 随机延迟 | 资源分配调整 |
| 形成条件 | 四个必要条件 | 检测冲突让步 | 资源分配不公平 |

### 死锁处理策略对比

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| 预防 | 打破四个条件之一 | 资源固定、需求可预测 |
| 避免 | 动态检测，安全分配 | 资源有限、需要高可用 |
| 检测恢复 | 定期检测 + 恢复 | 难以预防的场景 |
| 忽略 | 不处理 | 死锁概率低、成本高 |

## 九、扩展思考

### 💡 加分话题：哲学家就餐问题

哲学家就餐问题是经典的死锁问题：

```
五个哲学家围坐圆桌：
- 左右各有一根筷子
- 思考时不需要筷子
- 吃饭时需要左右两根筷子

死锁情况：
- 每个人都拿起左边的筷子
- 都在等待右边的筷子
- 死锁

解决方案：
1. 最多允许 4 个人同时拿筷子
2. 奇数号先拿左边，偶数号先拿右边
3. 拿起筷子前检查是否可能死锁
```

### 💡 加分话题：死锁检测算法

```
系统定期检测死锁：

检测算法：
1. 构建资源分配图
2. 查找环路
3. 如果有环路，可能存在死锁

资源分配图：
- 进程节点（P1, P2）
- 资源节点（R1, R2）
- 申请边（进程 → 资源）
- 分配边（资源 → 进程）

有环路且每种资源只有一个 → 死锁确定
有环路但资源有多个 → 可能死锁
```

> 死锁是并发编程中的经典问题。理解四个必要条件能够帮助我们在编码时避免死锁，同时也能指导我们设计更好的资源分配策略。
