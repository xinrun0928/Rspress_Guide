# 虚拟内存与请求分页

32 位系统，只有 4GB 物理内存。但你的程序可以声明 8GB 的数组，操作系统照单全收。

这就是虚拟内存的魔力——让程序「看到」的内存，比实际物理内存大得多。

## 什么是虚拟内存？

虚拟内存是操作系统为每个进程提供的**独立地址空间抽象**。

```
每个进程都有自己的虚拟地址空间：

进程 A 的视角:                    进程 B 的视角:
┌─────────────────┐              ┌─────────────────┐
│      内核       │              │      内核       │
├─────────────────┤              ├─────────────────┤
│      栈         │              │      栈         │
│       ↓         │              │       ↓         │
│       ↑         │              │       ↑         │
│      堆         │              │      堆         │
├─────────────────┤              ├─────────────────┤
│      数据        │              │      数据        │
├─────────────────┤              ├─────────────────┤
│      代码        │              │      代码        │
└─────────────────┘              └─────────────────┘
     0x00000000                       0x00000000
           │                               │
           │        操作系统                │
           │        ┌─────────┐             │
           └───────►│ 映射表   │◄────────────┘
                    └─────────┘
                        │
           ┌────────────┼────────────┐
           ▼            ▼            ▼
     ┌──────────┐ ┌──────────┐ ┌──────────┐
     │  物理内存 │ │  磁盘交换 │ │  物理内存 │
     │  (4GB)   │ │  空间    │ │  (部分)   │
     └──────────┘ └──────────┘ └──────────┘
```

### 虚拟地址 vs 物理地址

**虚拟地址：** 程序看到的地址，由编译器生成。

**物理地址：** 硬件实际使用的地址，由 MMU（内存管理单元）转换。

```c
// 程序看到的
int arr[1000];  // arr 的地址是虚拟地址 0x7fff5fbff8c0
arr[0] = 1;     // 写入这个虚拟地址

// 实际执行
// CPU 生成虚拟地址 → MMU 查页表 → 物理地址 → 写入内存
```

### 32 位 vs 64 位虚拟地址空间

| 系统 | 虚拟地址位数 | 理论最大空间 | 实际可用（Linux） |
|---|---|---|---|
| 32 位 | 32 位 | 4GB | 用户 3GB，内核 1GB |
| 64 位 | 48 位 | 256TB | 用户 128TB，内核 128TB |

```
64 位虚拟地址（实际只用 48 位）:
┌─────────────────────────────────────────────────────────────┐
│ 63  ...  48 │ 47 ... 0                                      │
├─────────────┼───────────────────────────────────────────────┤
│   全 0 或全 1 │ 实际可寻址空间                                │
│  (符号扩展)   │ 128TB 用户 + 128TB 内核                        │
└─────────────┴───────────────────────────────────────────────┘
       ↑
   必须是相同的
   高位必须相同，否则触发段错误
```

## 虚拟内存的作用

### 1. 地址空间隔离

每个进程有独立的虚拟地址空间，一个进程的错误不会破坏另一个进程。

```c
// 进程 A 的页表
Page 0 → Frame 5   // A 的代码在物理 Frame 5
Page 1 → Frame 8   // A 的数据在物理 Frame 8

// 进程 B 的页表
Page 0 → Frame 12  // B 的代码在物理 Frame 12
Page 1 → Frame 3  // B 的数据在物理 Frame 3

// 进程 A 访问地址 0，得到的物理地址是 Frame 5
// 进程 B 访问地址 0，得到的物理地址是 Frame 12
```

### 2. 内存扩展

物理内存不够时，把不常用的页面换出到磁盘，程序可以使用比物理内存更大的空间。

```
物理内存: ████████████████████████████████ 16GB
交换空间: ████████████████████████████████ 32GB (磁盘)

程序需求: ██████████████████████████████████████ 40GB

虚拟内存让程序以为有 40GB 可用：
实际只有 16GB 在物理内存，另外 24GB 在磁盘交换空间
```

### 3. 内存高效利用

多个进程可以共享代码段（如 glibc），物理内存只存一份。

```
进程 A ──► 共享库代码 ──► 物理内存中的共享库
进程 B ──► 共享库代码 ──►      (只有一份)
进程 C ──► 共享库代码 ──►
```

### 4. 简化内存分配

程序申请内存时，操作系统只需在虚拟地址空间分配，不需要立即分配物理页。

```c
// 程序申请 1GB 内存
char* buffer = malloc(1024 * 1024 * 1024);
// 操作系统：好，虚拟地址 0x7fff00000000 给你了

// 实际使用第一页时触发缺页
buffer[0] = 'a';
// 操作系统：哦，你要用了啊，分配一个物理页给你
```

## 请求分页机制

请求分页（Demand Paging）是虚拟内存最常见的实现方式：**页面只在需要时才加载到内存**。

### 基本思想

程序启动时，不需要把整个程序加载到内存。只加载必需的页面，其他页面在访问时按需加载。

```c
// 请求分页的工作流程
void access_page(uint64_t vaddr) {
    uint64_t vpn = vaddr >> PAGE_BITS;
    PageTableEntry* pte = &page_table[vpn];
    
    if (pte->present) {
        // 页已在内存，直接访问
        pte->accessed = 1;  // 更新访问位
        return;
    }
    
    // 缺页！页面不在内存
    page_fault_handler(vpn);
}

void page_fault_handler(uint64_t vpn) {
    // 1. 选择一个物理页框（可能需要换出）
    uint64_t victim_frame = select_victim_frame();
    
    // 2. 如果被选中换出的页是脏页，写回磁盘
    PageTableEntry* victim_pte = &page_table_of(victim_frame);
    if (victim_pte->dirty) {
        write_page_to_swap(victim_pte->vpn, victim_frame);
    }
    
    // 3. 从磁盘加载需要的页面
    read_page_from_disk(vpn, victim_frame);
    
    // 4. 更新页表
    victim_pte->present = 0;  // 原页面不在内存了
    
    PageTableEntry* new_pte = &page_table[vpn];
    new_pte->frame = victim_frame;
    new_pte->present = 1;
    new_pte->dirty = 0;
    new_pte->accessed = 1;
}
```

### 预调页（Prepaging）

请求分页在程序启动初期会产生大量缺页。预调页在程序启动时预测并预加载可能需要的页面。

```c
// 预调页策略
void prepaging(int process_id) {
    // 1. 分析程序的历史行为
    list<int> predicted_pages = predict_pages(process_id);
    
    // 2. 提前加载预测页面
    for (int vpn : predicted_pages) {
        if (!is_in_memory(vpn)) {
            load_page_async(vpn);  // 异步加载
        }
    }
}
```

**预调页的权衡：**
- 优点：减少程序启动时的缺页
- 缺点：预测可能不准，加载了不需要的页面浪费 I/O

### 页面大小的影响

页面大小是一个权衡：

| 页面大 | 页面小 |
|---|---|
| 减少页表大小 | 增加页表大小 |
| 减少缺页次数（一次加载更多） | 可能增加缺页次数 |
| 内部碎片增加 | 内部碎片减少 |
| I/O 效率高 | I/O 效率低（随机访问多） |

```
页面大小 = 4KB:
┌────┬────┬────┬────┬────┬────┬────┬────┐
│ A  │ A  │ A  │ A  │ B  │ B  │ B  │ C  │
└────┴────┴────┴────┴────┴────┴────┘
     程序 A 的页面：4 页

页面大小 = 16KB:
┌──────────────┬──────────────┬──────────────┐
│      A       │      B       │      C       │
└──────────────┴──────────────┴──────────────┘
     程序 A 的页面：1 页
     但程序 C 只用了 1KB，却占了一整页（15KB 浪费）
```

## 驻留集管理

驻留集（Resident Set）是进程当前在物理内存中的页面集合。

### 分配策略

**固定分配：** 每个进程分配固定数量的物理页。

```c
// 固定分配
void allocate_fixed_frames(Process* p, int num_frames) {
    p->resident_frames = num_frames;
    p->frame_list = malloc(sizeof(int) * num_frames);
    for (int i = 0; i < num_frames; i++) {
        p->frame_list[i] = allocate_free_frame();
    }
}
```

**可变分配：** 根据进程需求动态调整。

```c
// 可变分配：基于缺页率
void adjust_resident_set(Process* p) {
    double fault_rate = p->page_faults / p->references;
    
    if (fault_rate > UPPER_THRESHOLD) {
        // 缺页太多，增加分配
        add_frames(p, DELTA);
    } else if (fault_rate < LOWER_THRESHOLD) {
        // 缺页很少，可以回收
        remove_frames(p, DELTA);
    }
}
```

### 页面分配算法

如何决定把页面加载到哪个物理页框？

```c
// 分配算法
int allocate_frame() {

        if (frame_free[i]) {

    // 方法二：最佳适配（找最小的空闲块）
    // 方法三：最差适配（找最大的空闲块）

    // 如果没有空闲帧，执行页面置换
    return -1;  // 触发页面置换
}
```

## 虚拟内存的性能

###颠簸（Thrashing）回顾

当物理内存严重不足时，系统花费大量时间在页面置换上，真正的工作进展很少。

```
理想情况:                    颠簸情况:
┌────────────────────┐       ┌────────────────────┐
│ ████████████████  │       │ ██                 │
│ CPU 实际工作        │       │ CPU 实际工作        │
│                    │       │                    │
│ ████               │       │ ████████████████  │
│ 页面置换等待        │       │ 页面置换等待        │
└────────────────────┘       └────────────────────┘
```

### 优化建议

```c
// 1. 合理使用内存映射文件
// 大文件不需要全部读入内存
int fd = open("largefile.dat", O_RDONLY);
void* mapped = mmap(NULL, file_size, PROT_READ, MAP_PRIVATE, fd, 0);
// 按需加载，不用一次性读入

// 2. 使用大页减少 TLB miss
// 4KB 页：虚拟地址 48 位，页内偏移 12 位
// 2MB 页：虚拟地址 48 位，页内偏移 21 位
// 1GB 页：减少大量页表查找

// 3. 避免内存碎片
// 预分配：知道大小时预先分配
// 对象池：频繁分配释放的对象使用池
```

## 面试高频问题

**Q1：什么是虚拟内存？解决了什么问题？**

虚拟内存为每个进程提供独立的地址空间抽象，解决了地址空间隔离、内存扩展、高效利用、简化编程等问题。

**Q2：请求分页和预调页有什么区别？**

请求分页是页面被访问时才加载（懒加载）。预调页是预测可能需要的页面并提前加载，减少缺页次数。

**Q3：为什么需要交换空间（swap）？**

当物理内存不足时，把不常用的页面换出到磁盘交换空间，腾出物理内存给需要的页面。这样程序可以使用比物理内存更大的空间。

**Q4：虚拟地址是如何转换成物理地址的？**

CPU 产生虚拟地址 → MMU 查页表（可能查 TLB 缓存）→ 得到物理页框号 → 加上页内偏移得到物理地址。

**Q5：页面大小对性能有什么影响？**

大页面减少页表大小和缺页次数，但增加内部碎片。小页面减少碎片，但增加页表大小和可能的 I/O 次数。

理解虚拟内存和请求分页，是理解现代操作系统内存管理的核心。
