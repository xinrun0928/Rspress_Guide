# 局部性原理与抖动

凌晨 2 点，你的服务器 CPU 跑满了，但吞吐量反而下降。

`top` 一看，CPU 99% 在等待内存。不是计算慢，是**内存访问效率低**。

这就是**抖动（Thrashing）**——页面频繁换入换出，CPU 大部分时间在等待页面加载，而不是真正计算。

为什么会出现这种现象？理解**局部性原理**，就能找到答案。

## 局部性原理

程序访问内存时，有两个有趣的规律：

**时间局部性（Temporal Locality）：** 最近访问的地址，很快还会再次访问。

**空间局部性（Spatial Locality）：** 访问了地址 A，很快会访问地址 A 附近的地址。

### 代码中的局部性

```c
// 时间局部性示例：循环中的变量
for (int i = 0; i < 1000000; i++) {
    sum += arr[i];  // arr[i] 每次循环都会被访问
}
// 变量 i、sum、循环条件在整个循环期间反复访问

// 空间局部性示例：顺序访问数组
for (int i = 0; i < N; i++) {
    process(arr[i]);  // 访问 arr[i] 后大概率访问 arr[i+1]
}

// 空间局部性差：跳跃访问
for (int i = 0; i < N; i += stride) {
    process(arr[i]);  // stride 很大时，空间局部性差
}
```

### 数据结构中的局部性

```c
// 缓存友好的顺序访问
int sum = 0;
for (int i = 0; i < N; i++) {
    sum += array[i];  // 顺序访问，利用预取
}

// 缓存不友好的链表遍历
int sum = 0;
for (Node* p = head; p != NULL; p = p->next) {
    sum += p->value;  // 链表节点可能分散在内存各处
}
```

### 局部性对性能的影响

```
假设：
- 缓存命中：1 周期
- 缓存未命中，访问主存：100 周期
- 缓存命中率 95% 时，平均访问时间 = 0.95×1 + 0.05×100 = 5.95 周期
- 缓存命中率 90% 时，平均访问时间 = 0.90×1 + 0.10×100 = 10.9 周期

命中率只下降 5%，性能却下降近一倍！
```

### TLB（Translation Lookaside Buffer）

TLB 是虚拟地址到物理地址转换的缓存，利用程序对页面的局部性：

```c
// TLB 命中 vs 未命中
uint64_t translate_address(uint64_t vaddr) {
    uint64_t vpn = vaddr >> PAGE_BITS;
    
    // 先查 TLB（硬件并行匹配）
    uint64_t pfn = tlb_lookup(vpn);
    if (pfn != TLB_MISS) {
        return (pfn << PAGE_BITS) | (vaddr & PAGE_MASK);
    }
    
    // TLB 未命中，查页表（多次内存访问）
    pfn = page_walk(vpn);
    tlb_insert(vpn, pfn);  // 更新 TLB
    
    return (pfn << PAGE_BITS) | (vaddr & PAGE_MASK);
}
```

### 预取（Prefetch）

CPU 和操作系统会预测未来的访问，提前把数据加载到缓存：

```c
// 手动预取提示（编译器支持）
for (int i = 0; i < N; i++) {
    // 预取 arr[i + 8]（假设 cache line 是 8*4=32 字节）
    __builtin_prefetch(&arr[i + 8], 0, 3);
    
    process(arr[i]);  // 当前数据已在 cache
}
```

## 工作集模型

### 什么是工作集？

工作集（Working Set）是进程**最近活跃使用的页面集合**。

```
工作集 W(t, Δ) = {在过去 Δ 时间单位内被访问的页面}
```

例如：最近 10 秒内访问过的页面。

```python
# 简单的工作集计算
def calculate_working_set(process, delta):
    current_time = get_time()
    accessed_pages = set()
    
    for access_time, page in process.recent_accesses:
        if current_time - access_time <= delta:
            accessed_pages.add(page)
    
    return accessed_pages
```

### 工作集的特点

- 工作集大小相对稳定（程序有局部性）
- 工作集必须在物理内存中，否则频繁缺页
- 工作集大小是分配给进程的最少物理页数

```
时间 →
┌──────────────────────────────────────────────────────────────────┐
│  ┌─────┐                                                        │
│  │ W=3 │  ← 工作集大小                   页数                    │
│  ├─────┤                              ↑                         │
│  │ 页面│                                 │                       │
│  │  3  │   页面 3,7,9                      │                       │
│  ├─────┤                              │                       │
│  │ 页面│                                 │                       │
│  │  7  │   ← 页面在工作集中              │                       │
│  ├─────┤                                 │                       │
│  │ 页面│                                 │                       │
│  │  9  │                                 │                       │

```

## 抖动现象

### 什么是抖动？

抖动是指页面频繁换入换出，CPU 大部分时间用于页面置换，而不是实际计算。

```
正常情况:
┌─────────────────────────────────────┐
│  CPU 执行: ████████████ 80%        │
│  等待 I/O: ████ 20%                 │
└─────────────────────────────────────┘

抖动情况:
┌─────────────────────────────────────┐
│  CPU 执行: ██ 5%                    │
│  等待页面: ████████████████ 95%     │
└─────────────────────────────────────┘
```

### 抖动的成因

**根本原因：** 所有进程的工作集之和 > 物理内存容量。

```c
// 假设物理内存有 100 个页框
// 三个进程并发运行

进程 A 工作集: 40 页
进程 B 工作集: 35 页
进程 C 工作集: 35 页
─────────────────────────────
总计需要:   110 页 > 100 页

结果: 页面频繁换入换出，系统变慢
```

## 抖动的解决方案

### 1. 控制多道程序度数

减少并发进程数，让每个进程有更多物理内存。

```
并发度过高:           降低并发度后:
┌────────────────┐    ┌────────────────┐
│ 进程 A ████     │    │ 进程 A ██████████│
│ 进程 B ████     │    │ 进程 B ██████████│
│ 进程 C ████     │    │ 进程 C          │
│ 进程 D ████     │    │ 进程 D          │
│ ┊    ┊    ┊    ┊    │ ┊    ┊    ┊    ┊│
│ ┊    ┊    ┊    ┊    │ ┊    ┊    ┊    ┊│
└────────────────┘    └────────────────┘
     抖动                     正常运行
```

```c
// 简单的自适应调度
void adjust_multiprogramming() {
    // CPU 利用率低 + 吞吐量低 → 可能有抖动，减少进程
    // CPU 利用率高 + 吞吐量高 → 系统健康
    // CPU 利用率高 + 吞吐量低 → 可能是 I/O-bound
    
    if (cpu_utilization < 0.2 && throughput < threshold) {
        // 可能抖动，减少就绪进程
        decrease_ready_queue();
    } else if (cpu_utilization > 0.8 && throughput > threshold) {
        // 系统健康，可以增加进程
        increase_ready_queue();
    }
}
```

### 2. 使用工作集算法

根据每个进程的工作集分配内存：

```c
// 工作集置换算法
void working_set_replacement() {
    for (each process P) {
        // 计算进程 P 的工作集
        set ws = calculate_working_set(P, Δ);
        
        // 工作集页面保留在内存
        for (each page in P's_memory) {
            if (page not in ws) {
                // 页面不在工作集中，可以换出
                if (can_swap_out(page)) {
                    swap_out(page);
                }
            }
        }
    }
}
```

### 3. 页面故障频率控制（PFF）

监控进程的缺页率，动态调整分配的物理页数：

```c
// PFF 算法
void page_fault_frequency_control(process P) {
    double fault_rate = P.page_faults / time_period;
    
    if (fault_rate > upper_threshold) {
        // 缺页率太高，增加分配的页框
        P.allocated_frames += delta;
        // 分配后缺页率应该下降
    } else if (fault_rate < lower_threshold) {
        // 缺页率太低，可以回收一些页框
        if (P.allocated_frames > P.working_set_size) {
            P.allocated_frames -= delta;
        }
    }
}
```

### 4. 预留内存

给操作系统和关键进程预留足够的内存，避免被用户进程耗尽：

```bash
# Linux 查看内存使用
$ free -h
              total        used        free      shared  buff/cache   available
Mem:           31Gi       8.5Gi       2.1Gi       456Mi       20Gi       22Gi
                                                          ↑
                                              buff/cache 可以快速回收

# 设置 swapiness（越小越保守使用 swap）
$ sysctl vm.swappiness=10
vm.swappiness = 10
```

### 5. 使用更大的页

减少 TLB miss，提高内存访问效率：

```bash
# 查看大页配置
$ cat /sys/kernel/mm/hugepages/hugepages-2048kB/nr_hugepages

# 分配大页
$ echo 10 > /sys/kernel/mm/hugepages/hugepages-2048kB/nr_hugepages
```

## 面试高频问题

**Q1：什么是局部性原理？为什么重要？**

局部性原理包括时间局部性（最近访问的地址会再访问）和空间局部性（访问了地址 A会访问附近的地址）。这是缓存技术（CPU Cache、TLB、虚拟内存）有效的基础。

**Q2：什么是抖动？产生的原因是什么？**

抖动是页面频繁换入换出，CPU 大部分时间在等待页面加载。原因是所有进程的工作集之和超过了物理内存容量，导致页面不断被置换。

**Q3：如何解决抖动问题？**

可以通过降低多道程序度数、采用工作集算法、使用 PFF 控制缺页率、预留系统内存等方式解决。

**Q4：工作集是什么？**

工作集是进程最近活跃使用的页面集合。理解工作集，有助于理解为什么某些程序快、某些程序慢。

**Q5：为什么顺序访问比随机访问快？**

顺序访问符合空间局部性，CPU 可以预取下一批数据到缓存。随机访问每次可能都是缓存未命中，需要访问主存。

理解局部性原理和抖动现象，是理解操作系统内存管理整体设计的关键。
