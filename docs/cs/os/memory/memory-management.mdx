# 内存管理机制：连续分配 vs 非连续分配

想象一下：你有一张白纸，需要分给三个人画画。

**方案一（连续分配）：** 把纸分成三块连续的区域，每人一块。可能出现的情况是：A 画完了，B 还需要更多空间，但 A 的那块没法给 B。

**方案二（非连续分配）：** 把纸裁成很多小纸条，按需分配给每个人。每个人拿到的纸条可能分散在不同位置，但没关系，重要的是能画完画。

这就是连续分配和非连续分配的本质区别。

## 连续分配

早期操作系统使用连续分配策略，把物理内存划分为若干固定或动态的区域，每个程序占用一块连续空间。

### 单一连续分配

最简单的方案，整个内存只放一个程序。

```
┌────────────────────────────────────────┐
│                 OS                      │
├────────────────────────────────────────┤
│                                        │
│              用户程序                   │
│                                        │
│                                        │
└────────────────────────────────────────┘
```

**特点：**
- 简单，无需内存保护（只有 OS 和一个用户程序）
- 内存利用率极低
- 已被淘汰

### 分区固定分配

把内存分成多个固定大小的分区，每个分区放一个程序。

```
┌────────┬────────┬────────┬────────┐
│   OS   │ 分区 1 │ 分区 2 │ 分区 3 │
└────────┴────────┴────────┴────────┘
```

**问题：** 分区大小固定，但程序大小不一。如果分区太大，浪费空间；如果分区太小，程序放不进去。

### 分区可变分配（动态分区）

根据程序大小动态分配内存，空闲分区链表按某种策略组织。

```
初始状态：
┌────────────────────────────────────────┐
│                 OS                      │
├────────────────────────────────────────┤
│                                        │
│              用户程序                   │
│                                        │
└────────────────────────────────────────┘

加载程序 A（需要 100KB）后：
┌────────┬────────┬────────────────────┐
│   OS   │ 程序 A │      空闲          │
└────────┴────────┴────────────────────┘

加载程序 B（需要 200KB）后：
┌────────┬────────┬────────┬────────────┐
│   OS   │ 程序 A │ 程序 B │   空闲     │
└────────┴────────┴────────┴────────────┘
```

**动态分区面临的挑战：**

1. **外部碎片**：空闲内存被分割成很多小碎片，无法满足大程序需求

```
┌─────┬────────┬────┬────────┬────┐
│程序A│  空闲  │程序B│  空闲 │程序C│
└─────┴────────┴────┴────────┴────┘
 ↑这里   ↑这里   ↑这里  三个小碎片加在一起可能很大，
                      但没有单独一块能装下大程序
```

2. **分区分配算法**：

| 算法 | 思想 | 优点 | 缺点 |
|---|---|---|---|
| **首次适配（First Fit）** | 找到第一个足够大的空闲区 | 简单、速度快 | 容易产生外部碎片 |
| **最佳适配（Best Fit）** | 找到最小的足够大的空闲区 | 减少碎片 | 产生极小碎片 |
| **最差适配（Worst Fit）** | 找到最大的空闲区 | 大分区能满足大程序 | 大分区变小 |
| **下次适配（Next Fit）** | 从上次分配位置继续找 | 分布均匀 | 效果一般 |

```c
// 首次适配实现
char* allocate_first_fit(int size) {
    struct Node* curr = free_list_head;
    while (curr != NULL) {
        if (curr->size >= size) {
            // 分配：可能需要分割
            if (curr->size - size > MIN_FRAGMENT) {
                // 分割空闲块
                struct Node* new_block = curr + size;
                new_block->size = curr->size - size;
                new_block->next = curr->next;
                curr->next = new_block;
            }
            curr->size = size;
            curr->allocated = true;
            return curr->addr;
        }
        curr = curr->next;
    }
    return NULL;  // 分配失败
}
```

3. **合并相邻空闲块**：当程序释放内存时，检查相邻块是否也空闲，如果是则合并

```c
void coalesce(struct Node* freed_block) {
    struct Node* prev = find_prev(freed_block);
    struct Node* next = freed_block->next;
    
    // 合并前一个
    if (prev && !prev->allocated && are_adjacent(prev, freed_block)) {
        prev->size += freed_block->size;
        prev->next = next;
        freed_block = prev;
    }
    
    // 合并后一个
    if (next && !next->allocated && are_adjacent(freed_block, next)) {
        freed_block->size += next->size;
        freed_block->next = next->next;
    }
}
```

## 非连续分配

连续分配的外部碎片问题催生了非连续分配——把程序分散放在多个不连续的物理块中。

### 为什么需要非连续分配？

1. **消除外部碎片**：每个块按需分配，不会产生无法使用的小碎片
2. **支持动态加载**：程序可以分散加载，不要求一次性全部加载
3. **支持共享**：多个程序可以共享同一块物理内存

**代价：** 需要额外的地址转换机制（页表/段表），带来一定的开销。

### 分页（Paging）

**核心思想：** 把物理内存分成固定大小的**页框（Frame）**，虚拟内存分成相同大小的**页面（Page）**。页框大小通常为 4KB。

```
物理内存（页框）:
┌──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┐
│ Frame│ Frame│ Frame│ Frame│ Frame│ Frame│ Frame│ Frame│
│  0   │  1   │  2   │  3   │  4   │  5   │  6   │  7   │
└──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┘

虚拟内存（页面）:
┌──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┐
│ Page │ Page │ Page │ Page │ Page │ Page │ Page │ Page │
│  0   │  1   │  2   │  3   │  4   │  5   │  6   │  7   │
└──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┘

页表映射:
Page 0 → Frame 3
Page 1 → Frame 1
Page 2 → Frame 5
Page 3 → Frame 7
Page 4 → Frame 0
Page 5 → Frame 4
Page 6 → Frame 2
Page 7 → Frame 6
```

**地址转换：**

虚拟地址由**页号（Page Number）**和**页内偏移（Offset）**组成：

```
虚拟地址 = 页号 | 页内偏移
     ↓
   页表查找
     ↓
物理地址 = 页框号 | 页内偏移
```

```c
// 地址转换示例（假设页面大小 = 4KB = 2^12）
// 虚拟地址 0x1234 = 二进制 0001 0010 0011 0100
// 前 20 位是页号，后 12 位是页内偏移
// 页号 = 0x1234 >> 12 = 1
// 偏移 = 0x1234 & 0xFFF = 0x234

int page_number = virtual_addr >> PAGE_BITS;  // 右移 12 位
int offset = virtual_addr & (PAGE_SIZE - 1); // 取低 12 位

// 查页表
int frame_number = page_table[page_number];

// 物理地址
int physical_addr = (frame_number << PAGE_BITS) | offset;
```

### 分段（Segmentation）

**核心思想：** 按程序的逻辑结构划分，每个段代表一个逻辑单元（代码段、数据段、堆、栈等）。

```
程序逻辑视图:
┌─────────────────┐  0x00000000
│     代码段       │  ← 代码的起始地址
├─────────────────┤
│     数据段       │
├─────────────────┤
│      堆         │  ← 动态分配向上增长
│                 │
│                 │
├─────────────────┤
│                 │
│      栈         │  ← 函数调用向下增长
└─────────────────┘  0xFFFFFFFF

段表:
┌────────┬────────┬────────┐
│  段号  │ 基地址  │  界限   │
├────────┼────────┼────────┤
│   0    │ 0x0800 │  4KB   │  代码段
│   1    │ 0x1000 │  8KB   │  数据段
│   2    │ 0x5000 │  16KB  │  堆
│   3    │ 0x9000 │  8KB   │  栈
└────────┴────────┴────────┘
```

**地址转换：**

```c
// 段号在地址的高位，段内偏移在低位
// 虚拟地址 = 段号 | 段内偏移

int segment_num = virtual_addr >> SEGMENT_BITS;
int offset = virtual_addr & ((1 << SEGMENT_BITS) - 1);

struct SegmentTableEntry entry = segment_table[segment_num];

// 检查越界
if (offset >= entry.limit) {
    handle_fault();  // 越界错误
}

// 物理地址
int physical_addr = entry.base + offset;
```

### 分页 vs 分段

| 维度 | 分页 | 分段 |
|---|---|---|
| **划分依据** | 固定大小 | 程序逻辑结构 |
| **页/段大小** | 固定（4KB） | 不固定 |
| **地址结构** | 页号 + 页内偏移 | 段号 + 段内偏移 |
| **外部碎片** | 无 | 可能存在 |
| **内部碎片** | 最后一页可能有 | 无 |
| **共享** | 按页共享 | 按段共享（更自然） |
| **保护** | 页面级权限 | 段级权限 |

### 段页式管理

结合分段和分页的优点：先分段（保留程序逻辑），再分页（消除外部碎片）。

```
虚拟地址 = 段号 | 页号 | 页内偏移

段表 → 页表 → 物理页框

CPU
  │
  ▼
段表查找（段号）
  │
  ▼
得到页表起始地址
  │
  ▼
页表查找（页号）
  │
  ▼
得到物理页框号
  │
  ▼
物理地址 = 页框号 | 页内偏移
```

**实现：**

```c
// 段页式地址转换
struct SegmentTableEntry {
    unsigned int page_table_ptr;  // 指向该段的页表
    unsigned int valid: 1;         // 有效位
    unsigned int protection: 3;   // 保护位
};

struct PageTableEntry {
    unsigned int frame_number;     // 物理页框号
    unsigned int valid: 1;        // 在内存中
    unsigned int dirty: 1;        // 被修改过
    unsigned int referenced: 1;  // 最近被访问
};

int translate_address(int virtual_addr) {
    int segment = (virtual_addr >> (PAGE_BITS + SEGMENT_BITS)) & ((1 << SEGMENT_BITS) - 1);
    int page = (virtual_addr >> PAGE_BITS) & ((1 << PAGE_BITS) - 1);
    int offset = virtual_addr & ((1 << PAGE_BITS) - 1);
    
    // 查段表
    struct SegmentTableEntry seg_entry = segment_table[segment];
    if (!seg_entry.valid) page_fault();
    
    // 查页表
    struct PageTableEntry page_entry = page_table[seg_entry.page_table_ptr][page];
    if (!page_entry.valid) page_fault();
    
    // 物理地址
    return (page_entry.frame_number << PAGE_BITS) | offset;
}
```

## 对比与选择

**连续分配的适用场景：**
- 早期简单系统
- 嵌入式系统（资源有限）
- 对性能要求极高且程序大小固定的场景

**分页的适用场景：**
- 现代通用操作系统
- 需要高效内存管理
- 需要内存保护和多道程序支持

**分段的适用场景：**
- 需要按逻辑结构管理内存
- 需要不同段有不同的保护级别
- 需要代码共享

**段页式的适用场景：**
- 现代大型操作系统
- 需要综合两者优点
- IA-32 及以上的 x86 架构

## 面试高频问题

**Q1：什么是内部碎片和外部碎片？**

内部碎片是已分配给程序但未被使用的内存（如分页时最后一页）。外部碎片是空闲内存太小，无法满足程序需求（如动态分区时的小碎片）。

**Q2：为什么现代操作系统用分页而不是分段？**

现代操作系统主要使用分页，因为分页消除了外部碎片，配合交换技术可以高效管理内存。分段虽然保留了逻辑结构，但需要更复杂的管理。不过大多数系统保留了分段机制作为辅助（如 x86 的段寄存器）。

**Q3：段页式管理的优缺点？**

优点：结合了分段（逻辑清晰）和分页（无外部碎片）的优点。缺点：需要两次查表，开销更大。

理解连续分配和非连续分配的思想，是理解现代内存管理的基础。
