## 一、死锁处理策略概述

### 1.1 三种策略对比

| 策略 | 思想 | 系统开销 | 优点 | 缺点 |
|------|------|----------|------|------|
| 预防 | 打破四个必要条件 | 低 | 简单可靠 | 资源利用率低 |
| 避免 | 动态检测，安全分配 | 中 | 资源利用率较高 | 需要预先知道资源需求 |
| 检测恢复 | 允许死锁，定期检测 | 高 | 并发度高 | 恢复复杂 |

### 1.2 策略选择依据

```
选择依据：
- 系统规模
- 资源类型和数量
- 死锁概率
- 可用性要求

一般原则：
- 资源有限：预防或避免
- 高可用要求：避免或检测
- 开发阶段：预防为主
```

## 二、死锁预防

### 2.1 预防策略汇总

| 打破的条件 | 预防方法 |
|------------|----------|
| 互斥条件 | 无（互斥是资源固有特性） |
| 持有并等待 | 一次性申请所有资源 |
| 不可抢占 | 允许抢占资源 |
| 循环等待 | 按序申请资源 |

### 2.2 一次性申请所有资源

```
打破「持有并等待」条件

方法：
- 进程在开始前一次性申请所有需要的资源
- 或者释放已持有资源，再申请新资源

优点：简单可靠
缺点：
- 需要预先知道资源需求
- 资源利用率低（长期占用资源）
- 可能导致其他进程饥饿
```

### 2.3 按序申请资源

```
打破「循环等待」条件

方法：
- 对资源进行编号
- 进程必须按编号递增顺序申请资源
- 已持有编号为 k 的资源，可以申请编号 > k 的资源

效果：
- 不可能形成循环等待
- P0(R1) → P1(R2) → P2(R3) 是可能的
- P2(R3) → P1(R2) → P0(R1) 不可能
```

```java
// 按固定顺序申请锁
public class OrderedLock {
    private final Object lock1 = new Object();
    private final Object lock2 = new Object();

    // 正确做法：始终按固定顺序
    public void process() {
        synchronized (lock1) {
            synchronized (lock2) {
                // 业务逻辑
            }
        }
    }
}
```

## 三、死锁避免

### 3.1 安全状态

安全状态是指系统存在一个资源分配序列，使得所有进程都能顺利完成。

```
安全状态示例：

系统状态：
- 进程 A：已分配 2 个，最大需求 8 个
- 进程 B：已分配 3 个，最大需求 6 个
- 剩余资源：2 个

分析：
1. 可以将剩余 2 个分配给 B
2. B 完成，释放 3 + 2 = 5 个资源
3. 剩余资源 5 个，分配给 A
4. A 完成，释放 8 + 2 = 10 个资源
5. 所有进程都能完成 → 安全状态
```

### 3.2 不安全状态

不安全状态不一定是死锁状态，只是**可能**导致死锁。

```
不安全状态示例：

系统状态：
- 进程 A：已分配 2 个，最大需求 8 个
- 进程 B：已分配 3 个，最大需求 6 个
- 剩余资源：2 个

分析：
1. 将剩余 2 个分配给 A
2. A 仍需 6 个，但只有 0 个
3. A 无法完成 → 死锁风险

注意：不安全状态 ≠ 死锁
只是没有安全序列，不一定真的发生死锁
```

## 四、银行家算法

### 4.1 算法原理

银行家算法（Dijkstra）是死锁避免的经典算法，通过动态检查资源分配后是否存在安全状态来决定是否分配。

```
核心思想：
- 分配资源前，检查分配后是否仍为安全状态
- 如果是安全状态，分配资源
- 如果不是安全状态，拒绝分配

命名来源：
银行放贷 analogy - 银行需要确保放贷后客户能还款
```

### 4.2 数据结构

```java
// 银行家算法数据结构
public class BankerAlgorithm {
    int[] available;           // 可用资源向量 [R1, R2, R3]
    int[][] maximum;           // 最大需求矩阵 [P0][P1][P2]
    int[][] allocation;        // 已分配矩阵
    int[][] need;              // 需求矩阵 need = maximum - allocation
    int[] work;                // 工作向量，用于检测
    boolean[] finish;          // 完成标志
}
```

### 4.3 算法步骤

```
银行家算法步骤：

1. 安全性检测算法
   输入：系统当前状态
   输出：是否存在安全序列

   a. 初始化 work = available, finish[i] = false
   b. 查找满足条件的进程 Pi：
      - finish[i] = false
      - need[i] <= work
      如果不存在，goto d
   c. 假设 Pi 完成：
      work = work + allocation[i]
      finish[i] = true
      goto b
   d. 如果所有 finish[i] = true，存在安全序列

2. 资源请求算法
   输入：进程 Pi 的请求向量 request[i]

   a. 如果 request[i] <= need[i]，goto b
      否则报错（请求超过最大需求）

   b. 如果 request[i] <= available，goto c
      否则 Pi 等待（资源不足）

   c. 尝试分配：
      available = available - request[i]
      allocation[i] = allocation[i] + request[i]
      need[i] = need[i] - request[i]

   d. 调用安全性检测
      如果安全，分配成功
      如果不安全，恢复状态，拒绝分配
```

### 4.4 算法示例

```
初始状态：

资源总量：[10, 5, 7]

最大需求矩阵 M：
        R1  R2  R3
P0      7   5   3
P1      3   2   2
P2      9   0   2

已分配矩阵 A：
        R1  R2  R3
P0      0   1   0
P1      2   0   0
P2      3   0   2

可用资源：V = [10-5, 5-1, 7-2] = [5, 4, 5]

需求矩阵 N = M - A：
        R1  R2  R3
P0      7   4   3
P1      1   2   2
P2      6   0   0

安全性检测：
1. P1：N1 = [1,2,2] <= V = [5,4,5]，满足
   V = [5,4,5] + A1 = [7,4,5]
   P1 finish = true

2. P0：N0 = [7,4,3] <= V = [7,4,5]，满足
   V = [7,4,5] + A0 = [7,5,5]
   P0 finish = true

3. P2：N2 = [6,0,0] <= V = [7,5,5]，满足
   V = [7,5,5] + A2 = [10,5,7]
   P2 finish = true

安全序列：P1 → P0 → P2（存在安全序列）
```

## 五、死锁检测

### 5.1 资源分配图

资源分配图是检测死锁的工具：

```
图元素：
- 进程节点：P1, P2, P3
- 资源节点：R1, R2（圆圈表示实例数）

边类型：
- 申请边：P → R（进程申请资源）
- 分配边：R → P（资源分配给进程）
```

```
死锁检测规则：
- 每种资源类型只有一个实例：有环路则死锁
- 每种资源类型有多个实例：需要算法检测
```

### 5.2 检测算法

```java
public class DeadlockDetection {
    /**
     * 检测死锁
     * @param processes 进程数
     * @param resources 资源类型数
     * @param available 可用资源向量
     * @param allocation 已分配矩阵
     * @param request 需求矩阵
     * @return 死锁进程列表
     */
    public List<Integer> detect(int processes, int resources,
                                 int[] available, int[][] allocation,
                                 int[][] request) {
        int[] work = available.clone();
        boolean[] finish = new boolean[processes];
        List<Integer> deadlocked = new ArrayList<>();

        // 初始化：持有资源的进程标记为未完成
        for (int i = 0; i < processes; i++) {
            finish[i] = false;
            for (int j = 0; j < resources; j++) {
                if (allocation[i][j] > 0) {
                    finish[i] = true;
                    break;
                }
            }
        }

        // 尝试找到可以完成的进程
        boolean changed = true;
        while (changed) {
            changed = false;
            for (int i = 0; i < processes; i++) {
                if (!finish[i]) {
                    boolean canFinish = true;
                    for (int j = 0; j < resources; j++) {
                        if (request[i][j] > work[j]) {
                            canFinish = false;
                            break;
                        }
                    }
                    if (canFinish) {
                        // 进程 Pi 可以完成，释放其资源
                        for (int j = 0; j < resources; j++) {
                            work[j] += allocation[i][j];
                        }
                        finish[i] = true;
                        changed = true;
                    }
                }
            }
        }

        // 未完成的进程即为死锁进程
        for (int i = 0; i < processes; i++) {
            if (!finish[i]) {
                deadlocked.add(i);
            }
        }
        return deadlocked;
    }
}
```

### 5.3 检测时机

| 时机 | 优点 | 缺点 |
|------|------|------|
| 每次资源请求后 | 早发现 | 开销大 |
| 定期检测 | 开销可控 | 可能延迟发现 |
| CPU 利用率低时检测 | 资源充足 | 可能延迟 |

## 六、实际开发中的死锁避免

### 6.1 编码规范

```java
// 规范一：固定加锁顺序
public class OrderedLock {
    private final Object lock1 = new Object();
    private final Object lock2 = new Object();

    public void process1() {
        synchronized (lock1) {
            synchronized (lock2) {
                // 业务逻辑
            }
        }
    }

    public void process2() {
        synchronized (lock1) {  // 始终先锁 lock1
            synchronized (lock2) {  // 再锁 lock2
                // 业务逻辑
            }
        }
    }
}

// 规范二：使用定时锁
public class TryLockExample {
    private final Lock lock = new ReentrantLock();

    public void process() {
        boolean acquired = lock.tryLock(5, TimeUnit.SECONDS);
        if (acquired) {
            try {
                // 业务逻辑
            } finally {
                lock.unlock();
            }
        } else {
            // 获取锁失败，处理
        }
    }
}

// 规范三：减小锁粒度
public class FineGrainedLock {
    private final Object lockA = new Object();
    private final Object lockB = new Object();
    private int counterA;
    private int counterB;

    public void incrementA() {
        synchronized (lockA) {
            counterA++;
        }
    }

    public void incrementB() {
        synchronized (lockB) {
            counterB++;
        }
    }
}
```

### 6.2 设计原则

```
死锁避免的设计原则：

1. 锁的数量最小化
   - 减少锁竞争和死锁风险

2. 固定加锁顺序
   - 多把锁必须按固定顺序获取

3. 一次性获取所有锁
   - 使用 ReentrantLock.tryLock() 原子性获取

4. 减少持有锁时间
   - 只在必要时持有锁
   - 将耗时操作移出锁内

5. 使用高层并发工具
   - ConcurrentHashMap
   - Semaphore
   - BlockingQueue
```

# 2. 查找死锁信息

        ThreadMXBean mxBean = ManagementFactory.getThreadMXBean();
        long[] deadlockedThreads = mxBean.findDeadlockedThreads();
        if (deadlockedThreads != null) {

            }
        }
    }
}

## 七、面试题精讲

### 🔴 【高频】死锁处理策略

**问题**：死锁有哪些处理策略？

**标准答案**：

```
死锁处理三种策略：

1. 预防（Prevention）
   - 打破四个必要条件之一
   - 一次性申请所有资源
   - 按序申请资源
   - 优点：简单可靠
   - 缺点：资源利用率低

2. 避免（Avoidance）
   - 动态检测资源分配是否安全
   - 使用银行家算法
   - 优点：资源利用率较高
   - 缺点：需要预先知道最大需求

3. 检测恢复（Detection & Recovery）
   - 允许死锁发生
   - 定期检测死锁
   - 恢复：杀死进程、撤销资源
   - 优点：并发度高
   - 缺点：恢复复杂
```

### 🔴 【高频】银行家算法

**问题**：请描述银行家算法的工作原理。

**标准答案**：

```
银行家算法的核心思想：
- 在资源分配前，检查分配后是否仍为安全状态
- 如果是安全状态，分配资源
- 如果不是安全状态，拒绝分配

数据结构：
- Available：可用资源
- Maximum：最大需求
- Allocation：已分配
- Need：需求（Maximum - Allocation）

算法步骤：
1. 安全性检测
   - 初始化 work = available, finish = false
   - 找满足 need <= work 的进程 Pi
   - 假设 Pi 完成，work += allocation[i], finish[i] = true
   - 重复直到所有 finish[i] = true

2. 资源请求
   - 检查 request <= need && request <= available
   - 尝试分配，调用安全性检测
   - 安全则分配，不安全则拒绝
```

### 🟡 【中频】实际开发中如何避免死锁

**问题**：实际开发中如何避免死锁？

**标准答案**：

```
实际开发中的死锁避免策略：

1. 固定加锁顺序
   - 多把锁必须按固定顺序获取
   - 示例：synchronized(lock1) { synchronized(lock2) {} }

2. 使用定时锁
   - lock.tryLock(timeout)
   - 获取失败时回退或重试

3. 减小锁粒度
   - 减少锁的持有时间
   - 将耗时操作移出锁内

4. 减少锁的使用
   - 使用并发工具替代锁
   - ConcurrentHashMap、AtomicInteger

5. 使用高层并发工具
   - 生产者消费者模型用 BlockingQueue
   - 计数器用 AtomicInteger
```

## 八、对比总结

### 死锁处理策略对比

| 策略 | 适用场景 | 系统开销 | 资源利用率 | 实现复杂度 |

| 条件 | 打破方法 | 代价 |
|------|----------|------|

| 循环等待 | 按序申请 | 需要预先规划 |

## 九、扩展思考

### 💡 加分话题：哲学家就餐问题解决方案

```
问题：五个哲学家，五根筷子，可能死锁

解决方案一：限制同时用餐人数
- 最多允许 4 人同时拿筷子
- 打破循环等待

解决方案二：奇偶策略
- 奇数号先拿左边，再拿右边
- 偶数号先拿右边，再拿左边
- 打破对称性

解决方案三：一次性拿起
- 使用一个锁保护所有筷子
- 缺点：并发度降低
```

### 💡 加分话题：分布式死锁

```
分布式系统中的死锁：

1. 分布式事务死锁
   - 两阶段提交中的资源锁定
   - 解决：超时机制、死锁检测

2. 分布式锁死锁
   - Redis 分布式锁超时问题
   - 解决：合理设置超时、watchdog 续期

3. 分布式消息顺序死锁
   - 消息处理顺序依赖导致死锁
   - 解决：避免循环依赖
```

> 死锁处理需要在系统开销和并发度之间权衡。实际系统中，预防（固定加锁顺序）和检测恢复是更常用的方案，银行家算法更多是教学意义。
