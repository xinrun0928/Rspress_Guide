# 进程间通信方式

两个进程要「对话」，有哪些方式？

你可能觉得这个问题很简单——发网络请求、读写文件、用消息队列……但如果这两个进程在同一台机器上，还要走网络协议栈吗？如果性能要求极高，发消息的开销能承受吗？

进程间通信（Inter-Process Communication, IPC）是操作系统中最核心的机制之一。选择合适的 IPC 方式，直接影响系统的性能和可维护性。

        │         │          │         │         │

## 1. 管道（Pipe）

管道是最古老的 IPC 方式，Unix 从诞生之日起就在用。

### 匿名管道

匿名管道只能用于有亲缘关系的进程（如父子进程）：

```bash
# 创建一个管道，将 ls 的输出传给 grep
ls -la | grep ".mdx"
# 或者用代码实现
```

```c
#include <unistd.h>

int pipe(int fd[2]);  // fd[0] 读端，fd[1] 写端

// 示例：父子进程通过管道通信
int fd[2];
pipe(fd);

if (fork() == 0) {
    // 子进程
    close(fd[0]);           // 关闭读端
    write(fd[1], "hello", 5); // 写入数据
    close(fd[1]);
} else {
    // 父进程
    close(fd[1]);           // 关闭写端
    char buf[1024];
    read(fd[0], buf, sizeof(buf)); // 读取数据
    close(fd[0]);
}
```

### 命名管道（FIFO）

命名管道可以用于无亲缘关系的进程：

```bash
# 创建命名管道
mkfifo /tmp/my_fifo

# 终端 1：写入
echo "hello" > /tmp/my_fifo

# 终端 2：读取
cat /tmp/my_fifo
```

### 管道特点

| 特点 | 说明 |
|---|---|
| 单向通信 | 需要双向通信需要两个管道 |
| 面向字节流 | 没有消息边界 |
| 容量有限 | 管道缓冲区满时，写操作阻塞 |
| 血缘进程通信 | 匿名管道只能用于父子进程 |

## 2. 消息队列（Message Queue）

消息队列是消息的链表，存放在内核中。相比管道，消息队列有以下优势：

```c
#include <sys/msg.h>

// 创建消息队列
int msgid = msgget IPC_KEY, IPC_CREAT | 0666);

// 发送消息
struct msgbuf {
    long mtype;       // 消息类型（必须 > 0）
    char mtext[100];  // 消息内容
};
struct msgbuf buf = { .mtype = 1, .mtext = "hello" };
msgsnd(msgid, &buf, strlen(buf.mtext), 0);

// 接收消息
msgrcv(msgid, &buf, sizeof(buf.mtext), 1, 0);
```

### 消息队列 vs 管道

| 对比 | 管道 | 消息队列 |
|---|---|---|
| 消息边界 | 无（字节流） | 有（每条消息独立） |
| 生命周期 | 随进程 | 随内核（除非显式删除） |
| 读取方式 | FIFO | 可以按类型读取 |
| 容量 | 有限 | 有限但更大 |

## 3. 共享内存（Shared Memory）

共享内存是最快的 IPC 方式——数据在内存中直接共享，不需要拷贝。

```c
#include <sys/shm.h>

// 创建共享内存
int shmid = shmget IPC_KEY, 4096, IPC_CREAT | 0666);

// 映射到进程地址空间
void *shmaddr = shmat(shmid, NULL, 0);

// 写入数据
strcpy((char *)shmaddr, "shared data");

// 解除映射
shmdt(shmaddr);

// 删除共享内存
shmctl(shmid, IPC_RMID, NULL);
```

### 共享内存的速度优势

```
管道/消息队列的数据传输：
进程A → 内核缓冲区 → 进程B
       （两次拷贝）

共享内存的数据传输：
进程A ←→ 物理内存 ←→ 进程B
     （零拷贝，但需要同步）
```

### 为什么共享内存是最快的？

因为它避免了数据在内核和用户空间之间的拷贝。但正因为不需要拷贝，**使用共享内存必须配合同步机制**（信号量、互斥锁），否则会出现数据竞争。

## 4. 信号量（Semaphore）

信号量不是用于传输数据，而是用于**同步和互斥**。

```c
#include <sys/sem.h>

// 创建信号量
int semid = semget IPC_KEY, 1, IPC_CREAT | 0666);

// 初始化信号量
union semun {
    int val;
    struct semid_ds *buf;
    unsigned short *array;
};
union semun arg = { .val = 1 };  // 值为 1 表示互斥锁
semctl(semid, 0, SETVAL, arg);

// P 操作（减 1，如果为 0 则阻塞）
semop(semid, &(struct sembuf){ .sem_num = 0, .sem_op = -1 }, 1);

// V 操作（加 1）
semop(semid, &(struct sembuf){ .sem_num = 0, .sem_op = 1 }, 1);
```

### 信号量 vs 互斥锁

| 对比 | 信号量 | 互斥锁 |
|---|---|---|
| 作用域 | 可跨进程 | 通常用于同一进程的线程 |
| 所有权 | 无 | 有（谁加锁谁解锁） |
| 释放 | 可被任何进程 | 只能被加锁者释放 |
| 状态 | 计数信号量或二进制信号量 | 只有 0/1 两个状态 |

## 5. Socket

Socket 原本是为网络通信设计的，但也可以用于本机进程间通信（Unix Domain Socket）：

```c
#include <sys/socket.h>
#include <sys/un.h>

// 创建 Unix Domain Socket
int sockfd = socket(AF_UNIX, SOCK_DGRAM, 0);

// 绑定到本地文件
struct sockaddr_un addr;
addr.sun_family = AF_UNIX;
strcpy(addr.sun_path, "/tmp/my_socket");
bind(sockfd, (struct sockaddr *)&addr, sizeof(addr));

// 发送/接收数据
sendto(sockfd, "hello", 5, 0, ...);
recvfrom(sockfd, buf, sizeof(buf), 0, ...);
```

### Socket vs 其他 IPC

| 对比 | Socket | 其他 IPC |
|---|---|---|
| 适用范围 | 本机 + 网络 | 仅本机 |
| 复杂度 | 高 | 低 |
| 性能 | 低（有协议栈开销） | 高 |
| 灵活性 | 高 | 低 |

## 面试高频问题

**Q1：为什么共享内存是最快的 IPC 方式？**

因为数据不需要在内核态和用户态之间拷贝。其他 IPC 方式（管道、消息队列、Socket）都需要数据先拷贝到内核缓冲区，再从内核缓冲区拷贝到目标进程。共享内存直接映射同一块物理内存，实现零拷贝。

**Q2：使用共享内存时，为什么必须同步？**

因为没有同步机制，多个进程可能同时读写共享内存，导致数据不一致。常用的同步方式有信号量、互斥锁、读写锁等。

**Q3：管道和消息队列的区别是什么？**

管道是字节流，没有消息边界，数据按 FIFO 顺序读取；消息队列有消息边界，可以按消息类型选择性读取，消息持久化在内核中。

**Q4：信号量用于什么场景？**

信号量主要用于进程同步和互斥。典型的应用场景包括：保护共享资源（作为互斥锁）、实现进程执行顺序（如让进程 B 等待进程 A 完成某操作后再执行）。
