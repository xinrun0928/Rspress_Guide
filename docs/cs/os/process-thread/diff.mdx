# 进程、线程、协程的区别

你运行了一个 Node.js 服务，处理着成千上万的 HTTP 请求。你有没有想过：这些请求是如何「同时」处理的？是一个请求对应一个进程？还是一个进程处理多个请求？

如果你回答的是「一个进程处理多个请求」，那追问一句：进程和线程的区别是什么？协程又是什么？

这三个概念是并发编程的基础，但很多人只是「会用」，说不清「为什么这样设计」。这一讲，我们来彻底搞清楚。

## 进程：资源分配的最小单位

**进程是程序运行起来后的实例**。当你双击一个可执行文件，操作系统会创建一个进程，为其分配独立的地址空间、文件描述符、信号处理等资源。

```bash
# 在 Linux 中查看进程
ps aux | head -20
```

每个进程都有自己独立的虚拟地址空间。以 32 位系统为例：

```
+------------------+  0xFFFFFFFF (4GB)
|     内核空间      |  (1GB, 仅内核态可访问)
+------------------+  0xC0000000
|                  |
|    栈 (向下增长)  |
|                  |
+------------------+
|    内存映射区     |
+------------------+
|    堆 (向上增长)  |
+------------------+
|    BSS 段        |  (未初始化全局变量)
+------------------+
|    数据段         |  (已初始化全局变量)
+------------------+
|    代码段         |  (只读)
+------------------+  0x00000000
```

进程之间的地址空间是隔离的。进程 A 无法直接访问进程 B 的内存，这种设计保证了系统的稳定性——一个进程崩溃不会影响其他进程。

## 线程：CPU 调度的最小单位

**线程是进程内的执行单元**。一个进程可以包含多个线程，这些线程共享进程的地址空间、文件描述符、信号处理等资源，但每个线程有自己独立的栈和寄存器。

```
进程（资源分配）
├── 线程1（栈1、寄存器1）
├── 线程2（栈2、寄存器2）
├── 线程3（栈3、寄存器3）
└── ...共享代码段、数据段、堆...
```

这就解释了为什么线程比进程轻量：创建线程不需要分配独立的地址空间。

```java
public class ThreadDemo {
    public static void main(String[] args) {
        // 创建线程 - 比创建进程快很多
        Thread t1 = new Thread(() -> {
            System.out.println("线程1执行");
        });
        t1.start();

        Thread t2 = new Thread(() -> {
            System.out.println("线程2执行");
        });
        t2.start();
    }
}
```

### 进程 vs 线程：核心区别

| 对比维度 | 进程 | 线程 |
|---|---|---|
| 资源分配 | 独立的地址空间 | 共享进程资源 |
| 开销 | 创建/切换开销大（需要分配地址空间） | 创建/切换开销小（共享地址空间） |
| 通信 | 需借助 IPC（管道、消息队列等） | 直接通过共享内存通信 |
| 隔离性 | 高隔离，一个崩溃不影响另一个 | 低隔离，一个崩溃可能导致整个进程崩溃 |
| 并行性 | 多进程可实现多核并行 | 多线程可实现多核并行 |

## 协程：用户态的轻量级线程

协程（Coroutine）是近年来在服务器开发中广泛使用的概念。与线程相比，协程的核心区别是：**协程的调度由用户程序控制，而不是操作系统**。

线程的切换需要从用户态切换到内核态（一次系统调用），而协程的切换完全在用户态完成，开销极小。

```python
# Python 协程示例
async def fetch_user(user_id):
    # 遇到 I/O 操作时，协程会主动让出 CPU
    user = await get_user_from_db(user_id)
    orders = await get_orders_from_api(user_id)
    return {"user": user, "orders": orders}

async def main():
    # 并发执行多个协程
    tasks = [fetch_user(i) for i in range(1000)]
    results = await asyncio.gather(*tasks)
```

Go 语言中的 Goroutine 是协程的典型实现：

```go
func main() {
    // 创建一个 goroutine（协程）
    go func() {
        fmt.Println("协程执行中...")
    }()

    // 主 goroutine 等待
    time.Sleep(time.Second)
}
```

### 协程 vs 线程：核心区别

| 对比维度 | 线程 | 协程 |
|---|---|---|
| 调度者 | 操作系统内核 | 用户程序 |
| 切换开销 | 需要用户态→内核态→用户态 | 仅用户态切换 |
| 并发数量 | 受限于内核线程数（一般几千个） | 可轻松创建数十万个 |
| 适用场景 | CPU 密集型 | I/O 密集型（网络请求、文件读写） |

## 为什么要区分这些概念？

### 场景一：Web 服务器

传统的 Apache 使用「每个请求一个进程」的模式。假设同时有 10000 个请求，就需要 10000 个进程——这对系统资源是巨大的消耗。

后来出现的 Nginx 和 Node.js 采用了「事件驱动 + 单进程多线程」或「单进程 + 多协程」的模式，可以用极少的进程/线程处理大量并发请求。

```bash
# 查看 Nginx 的进程数
ps aux | grep nginx
# root     12345     0.0  0.1  ...  nginx: master process
# nobody   12346     0.0  0.1  ...  nginx: worker process
```

### 场景二：数据库连接池

数据库连接是稀缺资源，不可能每个请求都创建一个连接。常见做法是维护一个连接池，线程从池中获取连接，用完归还。

```java
// HikariCP 连接池
HikariConfig config = new HikariConfig();
config.setMaximumPoolSize(10);  // 最多10个连接
config.setMinimumIdle(5);       // 最少保持5个连接
HikariDataSource ds = new HikariDataSource(config);
```

### 场景三：游戏服务器

游戏服务器需要处理大量玩家的实时操作，通常采用「多进程 + 多线程」的模式：每个游戏房间是一个进程，房间内的玩家交互在多个线程中进行。

## 面试追问方向

- 为什么线程切换比进程切换快？具体快在哪里？
- 协程调度完全在用户态完成，那如果协程发生阻塞怎么办？
- Java 中的 Thread 和 Runnable 有什么区别？线程池是如何实现的？
- Go 中 GMP 模型的 G、M、P 分别是什么？为什么需要 P？

理解这些概念的区别，不只是为了面试。在实际开发中，选择用进程、线程还是协程，会直接影响系统的性能和可维护性。
