# 多级反馈队列调度算法

一个有趣的问题：**如果你既不知道进程的运行时间，也不知道它的优先级，怎么给它安排调度？**

先来先服务不行（护航效应），最短作业优先不行（不知道运行时间），纯优先级调度不行（不知道优先级）。

这就是多级反馈队列（Multi-Level Feedback Queue, MLFQ）要解决的问题——**让调度器自己学习每个进程的特性**。

## MLFQ 的核心思想

MLFQ 的设计哲学是：**用动态观察代替静态假设**。

操作系统通过观察进程在过去的行为，来推断它的特性：
- 如果一个进程经常主动让出 CPU（频繁 I/O），它很可能是交互式进程，应该给予高优先级
- 如果一个进程一直运行到时间片耗尽，它很可能是 CPU 密集型进程，应该降低其优先级

## 算法结构

MLFQ 使用多个队列，每个队列有不同的优先级和时间片长度：

```
┌────────────────────────────────────────────────────────┐
│ 调度器总是选择最高非空队列中的进程执行                   │
└────────────────────────────────────────────────────────┘
        │              │              │              │
        ▼              ▼              ▼              ▼
   ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
   │ Q0      │    │ Q1      │    │ Q2      │    │ Q3      │
   │ 最高优先级│    │ 中高优先级│    │ 中低优先级│    │ 最低优先级│
   │ 时间片=2 │    │ 时间片=4 │    │ 时间片=8 │    │ 时间片=∞ │
   │ RR      │    │ RR      │    │ RR      │    │ FCFS    │
   └─────────┘    └─────────┘    └─────────┘    └─────────┘
```

### 队列的基本规则

1. **规则1**：如果 A 的优先级 > B 的优先级，运行 A
2. **规则2**：如果 A 的优先级 = B 的优先级，轮转运行 A 和 B
3. **规则3**：新进程进入系统时，放在最高优先级队列
4. **规则4**：进程用完时间片后，降到下一个优先级队列
5. **规则5**：进程如果在时间片结束前主动放弃 CPU（如 I/O），保持优先级不变

## 直观示例

### 示例 1：交互式进程（频繁 I/O）

```
时刻 0: P1（交互式）进入 Q0
时刻 0: P1 执行，发出 I/O 请求，立即让出 CPU
时刻 1: P1 I/O 完成，从 Q0 继续执行（保持 Q0 优先级）
```

由于 P1 经常 I/O，它始终保持在高优先级队列，响应时间很短。

### 示例 2：CPU 密集型进程

```
时刻 0: P2（CPU密集型）进入 Q0
时刻 0-2: P2 执行，2ms 后时间片耗尽
时刻 2: P2 降级到 Q1
时刻 2-6: P2 在 Q1 执行，4ms 后时间片耗尽
时刻 6: P2 降级到 Q2
...继续降级...
```

P2 最终会降到一个很低甚至最低的队列，它的优先级始终低于交互式进程。

### 示例 3：混合场景

```
时刻 0: P1(Q0, 交互式), P2(Q0, CPU密集型) 同时到达
时刻 0-2: 调度器在 Q0 轮转执行
时刻 2: P1 发出 I/O 请求，让出 CPU，保持 Q0；P2 时间片耗尽，降级到 Q1
时刻 2-4: P1 执行（I/O 请求可能在此期间完成）
时刻 4-6: Q1 的 P2 执行（不受 Q0 影响）
```

可以看到，交互式进程 P1 始终保持在高优先级队列，响应时间得到保证。

## 解决饥饿问题：时间片提升

单纯按规则 4 运行，低优先级进程可能永远无法获得足够的 CPU 时间。解决方案：**定期将低优先级队列的进程提升到高优先级**。

```
# 经过一定时间后，所有进程都提升到最高优先级
# 这样可以防止低优先级进程饥饿
```

## 解决欺骗问题

有些进程会「欺骗」调度器：它在时间片快结束时故意发起一个 I/O 操作，然后立即完成，重新获得高优先级。

解决方法是**不完全按时间片降级**，而是根据等待时间动态调整。

## MLFQ 在 Linux 中的实现

Linux 使用的是 CFS（完全公平调度器），虽然不是严格的 MLFQ，但借鉴了类似思想：

```bash
# 查看进程虚拟运行时间
cat /proc/PID/sched
# scheduler:                      CFS bandwidth controller
# nr_switches:                    12345
# virtual_runtime:                123456789 ns
# min_vruntime:                    123456789
```

CFS 通过「虚拟运行时间」来实现公平：
- 每个进程按权重分配 CPU
- 权重越高，每单位实际时间累积的虚拟时间越少
- 调度器总是选择虚拟运行时间最少的进程

## MLFQ vs 其他调度算法

| 维度 | MLFQ | FCFS | RR | SJF |
|---|---|---|---|---|
| 响应时间 | 好 | 差 | 好 | 中 |
| 吞吐量 | 好 | 好 | 中 | 最好 |
| 实现复杂度 | 中 | 低 | 低 | 高 |
| 需要预知运行时间 | 否 | 否 | 否 | 是 |
| 饥饿风险 | 低 | 无 | 无 | 有 |

## 面试高频问题

**Q1：MLFQ 为什么能同时保证交互进程的响应时间和批处理进程的吞吐量？**

因为 MLFQ 自动将交互式进程保持在高优先级队列，将 CPU 密集型进程降到低优先级队列。交互进程需要频繁 I/O，时间片很短，响应快；CPU 密集型进程在高队列时间片也不长，可以快速让出 CPU 给其他进程。

**Q2：MLFQ 的主要挑战是什么？**

参数调优。队列数量、时间片大小、提升频率等都需要根据实际负载调整。设置不当可能导致性能问题。

**Q3：Linux 的 CFS 和 MLFQ 有什么关系？**

CFS 不是严格的 MLFQ，但借鉴了类似思想。CFS 通过动态调整虚拟运行时间，让频繁 I/O 的进程获得更多 CPU 时间，同时保证所有进程的公平性。

**Q4：如何避免进程「欺骗」MLFQ？**

可以采用以下策略：
1. 不完全根据时间片降级，而是根据累积 CPU 时间
2. 设置提升上限，防止进程长期在低优先级
3. 惩罚那些频繁发起 I/O 的进程

MLFQ 是理论和实践的完美结合，理解它的设计思想，对理解现代操作系统的调度机制大有裨益。
