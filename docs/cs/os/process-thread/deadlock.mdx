# 死锁

两个和尚挑水喝，三个和尚没水喝。

进程 A 握着资源 1，等着资源 2；进程 B 握着资源 2，等着资源 1。谁也不肯让步，谁也完不成。这就是**死锁**。

死锁是并发程序中最让人头疼的问题之一。它不会像内存泄漏那样慢慢拖垮系统，而是突然让一片功能完全停摆。更糟糕的是，重启可能只是暂时解决问题——如果不改代码，死锁还会再来。

## 什么是死锁？

死锁是指两个或多个进程在执行过程中，因争夺资源而造成一种相互等待的现象，若无外力干涉，它们都将无法推进下去。

```
┌─────────────────────────────────────────────────────┐
│                     死锁示意                         │
├─────────────────────────────────────────────────────┤
│                                                     │
│    进程 A                          进程 B           │
│   ┌─────────┐                    ┌─────────┐       │
│   │         │◄──── 资源 2 ────────│         │       │
│   │  持有   │                    │  持有   │       │
│   │  资源 1 │                    │  资源 2  │       │
│   └─────────┘                    └─────────┘       │
│        │                              │            │
│        │ 等待                         │ 等待        │
│        ▼                              ▼            │
│   ┌─────────┐                    ┌─────────┐       │
│   │  想要   │──────────────────►│  想要   │       │
│   │  资源 2 │                    │  资源 1  │       │
│   └─────────┘                    └─────────┘       │
│                                                     │
└─────────────────────────────────────────────────────┘
```

## 死锁的四个必要条件

1931 年， Coffman 等人总结了死锁发生的四个必要条件：

| 条件 | 说明 | 死锁中的表现 |
|---|---|---|
| **互斥条件** | 资源只能被一个进程使用 | 资源 1 只能被 A 使用，资源 2 只能被 B 使用 |
| **占有并等待** | 进程持有资源的同时请求其他资源 | A 持有资源 1，同时等待资源 2 |
| **不可抢占条件** | 已分配资源不能被强制夺走 | 资源 1 不能被 B 抢走 |
| **循环等待条件** | 形成进程-资源的循环链 | A 等待 B 持有的资源，B 等待 A 持有的资源 |

**只要破坏任意一个条件，死锁就不会发生。**

## 死锁预防

预防死锁的方法是破坏死锁的四个必要条件之一。

### 破坏互斥条件

允许资源同时被多个进程共享。但对于某些资源（如打印机），互斥性是其本质属性，无法破坏。

### 破坏占有并等待条件

方法一：进程执行前一次性申请所有需要的资源。
方法二：进程只有在没有持有资源时才能申请新资源。

```c
// 方法一的问题：资源利用率低
process() {
    lock_all_resources();
    do_work();
    unlock_all_resources();
}

// 方法二的问题：实现复杂
process() {
    // 释放已持有的资源，再申请新的
    unlock(resource1);
    lock(resource2);
}
```

### 破坏不可抢占条件

当进程请求的资源无法立即分配时，强制释放已持有的资源。

```c
// 伪代码
if (requested_resource.is_available()) {
    allocate(requested_resource);
} else {
    // 抢占已分配给其他进程的资源
    if (current_process.priority > holder_process.priority) {
        force_release(holder_process, requested_resource);
        allocate(requested_resource);
    }
}
```

### 破坏循环等待条件

为所有资源分配一个全局顺序，进程必须按顺序申请资源。

```c
// 例如：资源编号 R1 < R2 < R3
// 正确做法
lock(R1);
lock(R2);  // 不能先 lock(R2) 再 lock(R1)

// 错误做法（可能导致死锁）
lock(R2);
lock(R1);  // 如果另一个进程先 lock(R1) 再 lock(R2)，就会死锁
```

## 死锁避免

死锁避免与死锁预防不同：**预防是静态的、绝对的；避免是动态的，需要根据当前资源分配状态来判断**。

### 银行家算法

银行家算法是 Dijkstra 提出的经典死锁避免算法，其核心思想是：**在资源分配前，判断分配后系统是否仍然处于安全状态**。

```c
#include <stdio.h>
#include <stdbool.h>

#define N 5  // 进程数
#define M 3  // 资源类型数

// 当前分配矩阵
int allocation[N][M] = {
    {0, 1, 0},
    {2, 0, 0},
    {3, 0, 2},
    {2, 1, 1},
    {0, 0, 2}
};

// 最大需求矩阵
int max[N][M] = {
    {7, 5, 3},
    {3, 2, 2},
    {9, 0, 2},
    {2, 2, 2},
    {4, 3, 3}
};

// 可用资源向量
int available[M] = {3, 3, 2};

// 计算需求矩阵：Need = Max - Allocation
int need[N][M];

void calculate_need() {
    for (int i = 0; i < N; i++) {
        for (int j = 0; j < M; j++) {
            need[i][j] = max[i][j] - allocation[i][j];
        }
    }
}

// 检查系统是否处于安全状态
bool is_safe_state(int process_id, int request[M]) {
    // 1. 检查请求是否超过需求
    for (int j = 0; j < M; j++) {
        if (request[j] > need[process_id][j]) {
            printf("Error: Process %d exceeded its maximum claim\n", process_id);
            return false;
        }
    }

    // 2. 检查请求是否超过可用资源
    for (int j = 0; j < M; j++) {
        if (request[j] > available[j]) {
            printf("Process %d must wait: resources not available\n", process_id);
            return false;
        }
    }

    // 3. 假设分配，更新数据结构
    for (int j = 0; j < M; j++) {
        available[j] -= request[j];
        allocation[process_id][j] += request[j];
        need[process_id][j] -= request[j];
    }

    // 4. 检查是否存在安全序列
    bool finished[N] = {false};
    int work[M];
    for (int j = 0; j < M; j++) {
        work[j] = available[j];
    }

    printf("Safety check sequence: ");

    for (int count = 0; count < N; count++) {
        bool found = false;
        for (int i = 0; i < N; i++) {
            if (!finished[i]) {
                bool can_finish = true;
                for (int j = 0; j < M; j++) {
                    if (need[i][j] > work[j]) {
                        can_finish = false;
                        break;
                    }
                }
                if (can_finish) {
                    // 完成后释放资源
                    for (int j = 0; j < M; j++) {
                        work[j] += allocation[i][j];
                    }
                    finished[i] = true;
                    found = true;
                    printf("P%d ", i);
                    break;
                }
            }
        }
        if (!found) {
            printf("\nUnsafe state detected!\n");
            // 回滚
            for (int j = 0; j < M; j++) {
                available[j] += request[j];
                allocation[process_id][j] -= request[j];
                need[process_id][j] += request[j];
            }
            return false;
        }
    }

    printf("\nSafe state confirmed!\n");
    return true;
}

int main() {
    calculate_need();

    // 模拟进程 1 请求资源 (1, 0, 2)
    int request[3] = {1, 0, 2};
    printf("Process 1 requests: (%d, %d, %d)\n", request[0], request[1], request[2]);

    if (is_safe_state(1, request)) {
        printf("Request can be granted safely.\n");
    } else {
        printf("Request must be denied.\n");
    }

    return 0;
}
```

### 银行家算法的局限性

| 局限 | 说明 |
|---|---|
| 需要预知最大资源需求 | 实际中很难准确预知 |
| 资源分配必须固定 | 进程数变化时需要重新计算 |
| 计算复杂度高 | O(M × N²)，不适用于大规模系统 |

## 死锁检测与解除

如果系统没有采用死锁预防或避免策略，就必须周期性地检测死锁。

### 检测方法

**方法一：资源分配图法**（适用于每种资源只有一个实例）

```
        ┌───────────────────────────────────────┐
        │                                       │
        ▼                                       │
    ┌───────────┐                                │
    │  进程 A   │───────────►│R1│                 │
    └───────────┘            └────┘               │
        │                       ▲                 │
        │                       │                 │
        │    ┌───────────────────┘                 │
        │    │                                     │
        ▼    ▼                                     │
    ┌───────────┐                                   │
    │  进程 B   │───────────►│R2│                  │
    └───────────┘            └────┘                │
                                                     │
        ┌───────────────────────────────────────┘
        │
        ▼
    循环等待：P1 → R1 → P2 → R2 → P1
    → 死锁！
```

**方法二：有向图检测算法**（适用于每种资源有多个实例）

使用银行家算法中的安全序列检测方法，如果有进程无法完成，则存在死锁。

### 检测时机

| 策略 | 优点 | 缺点 |
|---|---|---|
| 每次资源请求时检测 | 能尽早发现死锁 | 开销大 |
| 定期检测 | 开销可控 | 可能延迟发现 |
| CPU 使用率低时检测 | 利用空闲时间 | 不及时 |

### 解除方法

| 方法 | 说明 | 优缺点 |
|---|---|---|
| **强制撤销进程** | 简单粗暴，直接终止死锁进程 | 可能丢失工作，需要选择牺牲者 |
| **强制回滚** | 将进程回滚到安全状态点 | 需要保存检查点，开销大 |
| **剥夺资源** | 从进程中抢占资源给其他进程 | 可能导致进程饥饿 |

## 实际系统中的死锁处理

实际操作系统和数据库系统通常采用**鸵鸟策略**（假装没有死锁）或**有限检测**。

```bash
# Linux 中查看死锁信息
cat /proc/sys/kernel/hung_task_timeout_secs
# 默认 120 秒，超时则报告可能有死锁
```

## 面试高频问题

**Q1：死锁和活锁有什么区别？**

死锁是进程处于永久阻塞状态；活锁是进程虽然没有被阻塞，但不断重复同样的操作（如锁被其他进程抢走，自己又立即去抢）。

**Q2：如何避免哲学家就餐问题的死锁？**

三种方法：
1. 资源顺序化：所有哲学家按相同顺序拿叉子
2. 限制人数：最多允许 4 个哲学家同时拿叉子
3. 奇偶策略：奇数号哲学家先拿左边，偶数号先拿右边

**Q3：银行家算法的时间复杂度是多少？**

O(M × N²)，其中 M 是资源类型数，N 是进程数。检测安全性需要 O(M × N)，最多需要检查 N 轮。

**Q4：什么是最小撤销代价的牺牲者？**

当需要强制终止进程解除死锁时，应该选择：
- 优先级低的进程
- 运行时间短、已经完成工作量少的进程
- 持有资源少的进程
- 批处理进程而非交互进程

理解死锁的原理和应对策略，是编写可靠并发程序的基础。
