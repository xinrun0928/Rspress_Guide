# 进程状态转换与 PCB

你有没有想过，一个运行中的程序突然发起了一个 `read()` 系统调用去读文件，在这个漫长的 I/O 等待期间，CPU 在干什么？

答案是：**CPU 不会干等着**。操作系统会让这个进程进入「阻塞」状态，然后调度其他进程执行。当 I/O 操作完成后，进程会被唤醒，重新进入「就绪」状态等待调度。

这就是进程状态转换的核心思想——**让 CPU 永远有事做**。

## 进程的五种基本状态

一个进程从创建到退出，会经历以下五种状态：

```
                    ┌─────────┐
          创建       │  就绪   │◄────────┐
        ┌──────────►│ (Ready) │          │
        │           └────┬────┘          │
        │                │                │ 调度选中
        │           调度 │                │
        │                ▼                │
   ┌────┴────┐     ┌─────────┐           │
   │  新建   │     │  运行中  │───────────┘
   │ (New)  │     │ (Running)│   时间片用完
   └────────┘     └────┬────┘
                       │ I/O 开始
                       │ (阻塞/等待)
                       ▼
                  ┌─────────┐
                  │  阻塞   │────────┐
                  │ (Blocked)│        │ I/O 完成
                  └─────────┘        │ (唤醒)
                                     ▼
```

### 1. 新建态（New）

进程正在被创建。此时操作系统正在分配资源、初始化 PCB。

### 2. 就绪态（Ready）

进程已准备好运行，只等待 CPU 空闲。一旦被调度选中，立刻开始执行。

### 3. 运行态（Running）

进程正在 CPU 上执行。只有在运行态的进程才能使用 CPU。

### 4. 阻塞态（Blocked）

进程因等待某个事件而暂停。比如等待 I/O 完成、等待信号量、等待锁释放等。**阻塞态的进程绝对不会被调度执行**。

### 5. 终止态（Terminated）

进程正常结束或被异常终止，操作系统回收其资源。

## 状态转换详解

### 就绪 → 运行

当 CPU 空闲时，调度程序从就绪队列中选择一个进程。这个选择依据的就是调度算法（后面会详细讲）。

### 运行 → 就绪

对于时间片轮转调度（Round Robin），当进程的时间片用完，就会被剥夺 CPU 使用权，重新回到就绪队列。

```bash
# 在 Linux 中查看进程状态
ps -eo pid,stat,cmd | head -20
# STAT 含义：
# R - 运行或就绪
# S - 可中断睡眠
# D - 不可中断睡眠（通常等待 I/O）
# T - 停止
# Z - 僵尸进程
```

### 运行 → 阻塞

当进程执行 `read()`、`wait()` 等系统调用，或等待某个资源时，会主动进入阻塞态。

```c
// 示例：C 语言中的阻塞操作
#include <unistd.h>

char buffer[1024];
// 这个 read 调用会阻塞，直到有数据可读
ssize_t n = read(fd, buffer, sizeof(buffer));
// 此时进程处于阻塞态，CPU 去执行其他进程
```

### 阻塞 → 就绪

当等待的事件发生时（如 I/O 完成、信号到达），进程被唤醒，进入就绪态。

## PCB：进程的心脏

PCB（Process Control Block，进程控制块）是操作系统为每个进程维护的数据结构，记录了进程的所有管理信息。

### PCB 包含的信息

```c
struct PCB {
    // 进程标识
    pid_t pid;              // 进程 ID
    pid_t parent_pid;       // 父进程 ID
    process_state state;   // 当前状态

    // CPU 调度信息
    int priority;          // 调度优先级
    unsigned long utime;   // 用户态 CPU 时间
    unsigned long stime;   // 内核态 CPU 时间

    // 内存管理信息
    struct mm_struct *mm;  // 进程的地址空间
    unsigned long cr3;     // 页目录基址

    // 文件管理信息
    struct files_struct *files;  // 文件描述符表

    // 信号处理信息
    struct sigaction sigaction[64];  // 信号处理函数
    unsigned long blocked;           // 阻塞信号掩码

    // 资源限制
    struct rlimit rlim[RLIM_NLIMITS];  // 打开文件数、CPU 时间等限制

    // 其他信息
    char comm[16];                     // 进程名
    // ... 还有更多字段
};
```

### PCB 的组织方式

操作系统通常用链表或索引表来组织 PCB：

- **链表**：适用于创建/销毁频繁的场景
- **索引表**：适用于按 PID 快速查找的场景

Linux 的 `task_struct` 就是 Linux 内核中的 PCB 结构体。查看方式：

```bash
# 查看 PID 为 1234 的进程的 PCB 信息
cat /proc/1234/status
```

### PCB 与进程状态的关系

PCB 是进程状态的载体。当进程状态发生变化时，操作系统修改的正是 PCB 中的 `state` 字段。

```
进程状态变化流程：
1. 用户启动程序
2. 操作系统创建 PCB，初始化 state = NEW
3. 完成创建，state = READY，加入就绪队列
4. 调度器选中，state = RUNNING
5. 执行系统调用，state = BLOCKED
6. I/O 完成，state = READY
7. 再次被调度，state = RUNNING
8. 程序结束，state = TERMINATED，回收 PCB
```

## 挂起状态：扩展知识

在支持虚拟内存的系统中，还有两种挂起状态：

| 状态 | 说明 |
|---|---|
| **挂起就绪** | 进程被换出到磁盘，但已准备好运行 |
| **挂起阻塞** | 进程被换出到磁盘，且正在等待某个事件 |

挂起的目的是解决内存不足的问题——当物理内存紧张时，操作系统将不活跃的进程换出到磁盘。

```bash
# Linux 中查看内存使用情况
free -h
#              total        used        free      shared  buff/cache   available
# Mem:          15Gi       8.2Gi       2.1Gi       1.5Gi       4.7Gi       6.2Gi
# Swap:         2.0Gi       256Mi       1.7Gi
```

## 面试高频问题

**Q1：进程和程序有什么区别？**

程序是静态的代码文件，进程是程序运行起来的实例。程序可以对应多个进程（比如多次运行同一个程序），一个进程只能对应一个程序。

**Q2：进程阻塞和线程阻塞有什么区别？**

进程阻塞时，整个进程都被挂起，其他线程也无法执行。线程阻塞时，同一进程的其他线程仍可运行（如果他们不需要同一个资源）。

**Q3：什么是僵尸进程？如何处理？**

僵尸进程是子进程已结束，但父进程还未调用 `wait()` 回收其退出状态。这时候进程表项仍然存在。处理方法是：杀死父进程（僵尸进程会变成孤儿进程，被 init 回收），或者修改代码让父进程正确回收子进程。

**Q4：如何查看僵尸进程？**

```bash
ps aux | grep 'Z'
# 僵尸进程的 STAT 列显示为 Z
```
