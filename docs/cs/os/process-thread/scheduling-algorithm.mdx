# 进程调度算法

想象一下：你去银行办业务，排队的人有 100 个。银行经理宣布：「我们有两种叫号方式——按排队顺序来，或者让办理时间短的人先来。」

你会选哪种？

这个问题没有标准答案，因为选择取决于你的目标：是想让平均等待时间最短，还是想公平对待每个人？进程调度算法要解决的，正是这个问题。

## 调度算法的评价指标

在讨论具体算法之前，我们需要知道如何评价一个调度算法：

| 指标 | 说明 |
|---|---|
| **CPU 利用率** | CPU 忙碌时间占总时间的比例 |
| **吞吐量** | 单位时间内完成的进程数 |
| **周转时间** | 进程从提交到完成的总时间 |
| **等待时间** | 进程在就绪队列中等待的总时间 |
| **响应时间** | 从提交请求到首次响应的时间 |

```bash
# 在 Linux 中查看系统负载
uptime
# 17:30:45 up 5 days,  3:22,  2 users,  load average: 0.52, 0.58, 0.59
# load average 是 1min/5min/15min 的系统平均负载
```

## FCFS：先来先服务

最简单的调度算法——谁先来，谁先执行。

### 算法思想

```bash
# 进程到达顺序：P1(24ms), P2(3ms), P3(3ms)
# 时间线：
# |----P1----|----P2----|----P3----|
# 0         24        27        30
```

### 性能分析

- **周转时间**：P1 = 24, P2 = 27, P3 = 30，平均 = 27
- **等待时间**：P1 = 0, P2 = 24, P3 = 27，平均 = 17

### 优缺点

| 优点 | 缺点 |
|---|---|
| 简单，易于实现 | 护航效应（短进程被长进程拖累） |
| 公平，不会饥饿 | 对用户体验不友好 |

### 适用场景

FCFS 适合 I/O 密集型进程居多的系统，因为这类进程会频繁阻塞，不会长时间占用 CPU。

## SJF：最短作业优先

让执行时间最短的进程先执行。

### 算法思想

```bash
# 进程到达顺序：P1(6ms), P2(3ms), P3(8ms), P4(2ms)
# 按执行时间排序：P4(2) -> P2(3) -> P1(6) -> P3(8)
# 时间线：
# |--P4--|--P2--|-----P1-----|-----P3-----|
# 0      2      5            11           19
```

### 性能分析

- **周转时间**：P1 = 11, P2 = 5, P3 = 19, P4 = 2，平均 = 9.25
- **等待时间**：P1 = 5, P2 = 2, P3 = 11, P4 = 0，平均 = 4.5

相比 FCFS，平均等待时间大幅降低。

### 优缺点

| 优点 | 缺点 |
|---|---|
| 平均等待时间最短 | 需要预知进程执行时间（难以准确估计） |
| 易于证明最优性 | 长作业可能饥饿 |

### 两种变体

| 类型 | 说明 |
|---|---|
| **非抢占式 SJF** | 一旦开始执行，就执行完毕 |
| **抢占式 SJF（SRTF）** | 如果新到达的进程比当前进程更短，立即切换 |

## 时间片轮转（Round Robin）

每个进程执行一个时间片，然后切换到下一个进程。

### 算法思想

```bash
# 时间片大小 = 4ms
# 进程顺序：P1(10ms), P2(4ms), P3(6ms)
# 时间线：
# |P1|P2|P3|P1|P1|P1|P3|P1|P1|
# 0  4  8  12 16 20 24 28 32 36 40
```

### 时间片大小的影响

| 时间片过大 | 时间片过小 |
|---|---|
| 接近 FCFS | 切换开销增大，系统效率降低 |
| 响应时间差 | 响应时间短，但吞吐量低 |

**经验法则**：时间片应略大于 80% 的交互进程的计算时间。

### 优缺点

| 优点 | 缺点 |
|---|---|
| 响应时间可预期 | 上下文切换开销 |
| 公平 | 上下文切换开销大 |
| 适合交互系统 | 不适合批处理系统 |

## 优先级调度

为每个进程分配优先级，高优先级进程先执行。

### 算法思想

```bash
# 进程优先级（数字越小优先级越高）：
# P1(3), P2(1), P3(4), P4(2)
# 执行顺序：P2 -> P4 -> P1 -> P3
```

### 优缺点

| 优点 | 缺点 |
|---|---|
| 重要进程优先执行 | 低优先级进程可能饥饿 |
| 灵活 | 可能导致低优先级进程永远无法执行 |

### 解决饥饿：优先级提升

```bash
# Linux 中的 nice 值（-20 到 19，越小优先级越高）
# 低优先级进程运行越久，动态优先级会提升
nice -n 10 ./my_program    # 低优先级启动
renice -n 5 -p 1234         # 动态调整优先级
```

Linux 使用 CFS（完全公平调度器），通过「虚拟运行时间」实现类似优先级的公平调度：

```bash
# 查看进程优先级
ps -eo pid,ni,cmd | head -10
# PID  NI  CMD
#   1   0  /sbin/init
# 123   0  my_process
# 456  10  low_priority_task
```

## 多级队列调度

将进程分成多个队列，每个队列使用不同的调度算法。

```
┌─────────────────────────────────────────────────────┐
│                   调度器                            │
└─────────────────────────────────────────────────────┘
           │              │            │
           ▼              ▼            ▼
    ┌───────────┐  ┌───────────┐  ┌───────────┐
    │ 交互队列   │  │ 批处理队列 │  │ 后台队列  │
    │(RR,高优先级)│  │(FCFS)     │  │(FCFS)     │
    └───────────┘  └───────────┘  └───────────┘
```

### 队列间调度

| 方式 | 说明 |
|---|---|
| 固定优先级 | 优先处理高优先级队列 |
| 时间片划分 | 每个队列占用固定比例的 CPU 时间 |

## 算法对比总结

| 算法 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| FCFS | 简单、公平 | 护航效应 | 批处理系统 |
| SJF | 平均等待时间最短 | 难以预知执行时间 | 作业调度 |
| RR | 响应时间可预期 | 时间片选择困难 | 交互系统 |
| 优先级 | 灵活 | 可能饥饿 | 服务器作业 |
| 多级队列 | 兼顾多种需求 | 复杂 | 通用系统 |

## 面试高频问题

**Q1：什么是护航效应？**

长进程后面排着很多短进程，必须等长进程执行完，短进程才能执行。FCFS 算法最容易出现这个问题。

**Q2：Linux 使用的是什么调度算法？**

Linux 2.6 之后使用 CFS（Completely Fair Scheduler，完全公平调度器）。它的核心思想是：为每个进程分配一个「虚拟运行时间」，调度器总是选择虚拟运行时间最少的进程执行。

**Q3：如何避免低优先级进程饥饿？**

可以采用「老化」技术：进程每等待一段时间，就提升其优先级，防止无限等待。

**Q4：SJF 是最优算法吗？**

在平均等待时间方面，SJF 是最优的。但实际很难准确预知进程执行时间，所以通常使用的是 SJF 的近似实现。

调度算法的选择取决于具体场景。没有最好的算法，只有最适合的算法。
