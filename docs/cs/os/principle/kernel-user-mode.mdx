# 内核态 vs 用户态与系统调用

你的程序能访问任意内存吗？能直接读写磁盘吗？能修改 CPU 优先级吗？

答案是：**不能**。

程序运行在「用户态」，只能访问受限的资源。而操作系统运行在「内核态」，拥有访问一切的能力。

这两种状态的划分，是现代操作系统安全性的基石。

## 特权级别

CPU 提供多个特权级别（Privilege Levels），用于控制能执行的指令和访问的寄存器。

```
x86 的 4 级特权环:

┌─────────────────────────────────────┐
│  Ring 0: 内核态（最高特权）          │ ← 操作系统内核
├─────────────────────────────────────┤
│  Ring 1: 系统服务                   │ ← 部分操作系统
├─────────────────────────────────────┤
│  Ring 2: 系统服务                   │ ← 部分操作系统
├─────────────────────────────────────┤
│  Ring 3: 用户态（最低特权）          │ ← 用户程序
└─────────────────────────────────────┘

Linux 只使用 Ring 0 和 Ring 3：
- Ring 0: 内核态
- Ring 3: 用户态
```

### 特权级别检查

当发生特权级别切换时，CPU 会检查：

1. **当前特权级别（CPL）**：CPU 当前运行在哪一级
2. **描述符特权级别（DPL）**：资源（如段、门）要求哪一级
3. **请求特权级别（RPL）**：访问资源的程序声称是哪一级

```
特权级别检查示例:

用户程序 (CPL=3) 尝试访问内核数据段 (DPL=0):
┌──────────────────────────────────────────────────┐
│ CPL = 3 (用户态)                                 │
│ Desired DPL = 0 (内核)                          │
│                                                  │
│ CPL > DPL → 拒绝访问 → General Protection Fault │
└──────────────────────────────────────────────────┘
```

## 内核态 vs 用户态

### 什么是内核态？

内核态（Kernel Mode）下，CPU 可以执行**任何指令**，访问**任何内存**。

```
内核态可以做的事情：
✓ 读写任意内存地址
✓ 执行 I/O 操作（读写磁盘、网络等）
✓ 修改 CPU 优先级
✓ 访问控制寄存器（CR0、CR3 等）
✓ 执行 halt 指令（关闭 CPU）
✓ 访问所有硬件设备
```

### 什么是用户态？

用户态（User Mode）下，程序只能访问受限的区域。

```
用户态的限制：
✗ 不能直接访问硬件
✗ 不能执行特权指令
✗ 不能访问内核内存区域
✗ 不能修改 CPU 控制寄存器
✗ 只能访问进程自己的虚拟地址空间

✓ 可以执行普通计算指令
✓ 可以访问自己的内存
✓ 可以调用系统调用请求内核服务
```

### 状态如何切换？

```
用户态 → 内核态（触发方式）：
┌──────────────────────────────────────────────────┐
│  1. 系统调用 (syscall) - 程序主动请求            │
│  2. 中断 (interrupt) - 硬件事件通知              │
│  3. 异常 (exception) - CPU 检测到错误            │
│  4. 陷阱 (trap) - 程序主动触发                   │
└──────────────────────────────────────────────────┘

内核态 → 用户态：
┌──────────────────────────────────────────────────┐
│  1. IRET 指令 - 从中断/异常返回                   │
│  2. 修改 CS/RIP 寄存器跳回用户态代码段            │
└──────────────────────────────────────────────────┘
```

## 系统调用

### 什么是系统调用？

系统调用是用户程序**请求内核服务**的唯一合法方式。

```
系统调用示例：
┌────────────────────────────────────────────────────────────────┐
│                                                                │
│  用户程序                      内核                            │
│    │                          │                               │
│    │  open("file.txt", O_RDONLY)  ← 系统调用号 + 参数       │
│    ├────────────────────────────────────────────────────────► │
│    │                          │                               │
│    │                          ▼                               │
│    │                    1. 参数验证                           │
│    │                    2. 权限检查                           │
│    │                    3. 执行内核函数                       │
│    │                    4. 返回结果                           │
│    │                          │                               │
│    ├◄──────────────────────────────────────────────────────── │
│    │  返回文件描述符 (3)                                    │
│    │                          │                               │
│    ▼                          ▼                               │
│  使用文件描述符...           完成                             │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 常见的系统调用

| 类别 | 系统调用 | 说明 |
|---|---|---|
| **进程** | fork, exec, exit, wait | 创建、运行、结束进程 |
| **文件** | open, read, write, close | 文件操作 |
| **内存** | mmap, brk | 内存映射、堆扩展 |
| **网络** | socket, bind, connect | 网络通信 |
| **时间** | gettimeofday, clock_gettime | 获取时间 |
| **进程间** | pipe, shmget, msgget | 进程间通信 |

### x86 的系统调用实现

**早期方案：int 0x80**

```asm
; 用户态发起系统调用
mov eax, 4        ; 系统调用号 (sys_write)
mov ebx, 1        ; 参数 1: 文件描述符 (stdout)
mov ecx, msg      ; 参数 2: 缓冲区地址
mov edx, len      ; 参数 3: 缓冲区长度
int 0x80          ; 触发软中断，切换到内核态

; 内核处理完后返回到用户态
; eax 中是返回值
```

**现代方案：syscall 指令**

x86-64 提供专门的 syscall 指令，比 int 0x80 更快：

```asm
; syscal 方式
mov rax, 1        ; 系统调用号
mov rdi, 1        ; 参数 1
mov rsi, msg      ; 参数 2
mov rdx, len      ; 参数 3
syscall           ; 直接跳转，内核态完成后返回

; 内核入口（简化版）
syscall_entry:
    push r11              ; 保存寄存器
    push rcx              ; 保存返回地址
    mov r11, msr_flags    ; 保存 RFLAGS
    mov rcx, rip          ; 保存返回地址
    
    ; 跳转到系统调用处理函数
    call do_syscall
    
    ; 返回用户态
    mov rax, rax_return_value  ; 设置返回值
    pop rcx               ; 恢复返回地址
    pop r11               ; 恢复 RFLAGS
    sysretq               ; 返回用户态
```

### 参数传递

**x86-32：int 0x80**
- ebx, ecx, edx, esi, edi, ebp 依次传递参数

**x86-64：syscall**
- rdi, rsi, rdx, r10, r8, r9 依次传递参数（更快）

### 系统调用与库函数

```
┌──────────────────────────────────────────────────────────────────┐
│                         用户程序                                  │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  printf("Hello")          ← C 库函数（用户态）                   │
│       │                                                        │
│       ▼                                                        │
│  puts("Hello")            ← C 库内部                            │
│       │                                                        │
│       ▼                                                        │
│  syscall(__NR_write, 1, "Hello", 5)  ← 系统调用                │
│       │                                                        │
└───────┼────────────────────────────────────────────────────────┘
        │                    ▲
        │                    │
        ▼                    │
┌──────────────────────────────────────────────────────────────────┐
│                         内核                                      │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  do_syscall()                 ← 内核处理                         │
│       │                                                        │
│       ▼                                                        │
│  sys_write()                  ← 真正的内核函数                    │
│       │                                                        │
│       ▼                                                        │
│  vfs_write()                  ← VFS 层                           │
│       │                                                        │
│       ▼                                                        │
│  ext4_file_write()           ← 具体文件系统                      │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

**关键区别：**
- **库函数**：用户态的封装，可能调用系统调用，也可能不调用（如 `strcmp`、`strcpy`）
- **系统调用**：进入内核的唯一通道

## 中断门与陷阱门

### 中断门

中断门（Interrupt Gate）会**禁用中断**后跳转到处理程序。

```c
// 中断门描述符
struct interrupt_gate {
    unsigned short offset_low;    // 偏移地址低 16 位
    unsigned short segment_selector;  // 代码段选择子
    unsigned char  zero: 5;       // 固定为 0
    unsigned char  type: 4;      // 1001 = 32位中断门
    unsigned char  D: 1;          // 1 = 32 位
    unsigned char  zero2: 1;      // 固定为 0
    unsigned char  DPL: 2;        // 特权级
    unsigned char  present: 1;    // 是否存在
    unsigned short offset_high;   // 偏移地址高 16 位
};
```

### 陷阱门

陷阱门（Trap Gate）**不禁用中断**，但会切换到内核态。

```c
// 陷阱门描述符
struct trap_gate {
    unsigned short offset_low;
    unsigned short segment_selector;
    unsigned char  zero: 5;
    unsigned char  type: 4;      // 1111 = 32位陷阱门
    unsigned char  D: 1;
    unsigned char  zero2: 1;
    unsigned char  DPL: 2;
    unsigned char  present: 1;
    unsigned short offset_high;
};
```

### 系统调用门

系统调用门（System Call Gate）是特殊的陷阱门，专门用于系统调用。

```c
// x86-64 的 SYSCALL 入口
// 不使用门描述符，直接通过 MSR（Model Specific Register）配置
// MSR_LSTAR = 0xC0000082: 存储 syscall 指令的处理函数地址

void setup_syscall_gate() {
    // 设置 sys_call_table 地址
    wrmsr(MSR_LSTAR, (unsigned long)entry_SYSCALL_64);
    
    // 设置 syscall 返回函数
    wrmsr(MSR_STAR, ((unsigned long)GDT_KERNEL_CS << 16) | GDT_USER_CS);
    
    // 设置 mask
    wrmsr(MSR_SYSCALL_MASK, X86_EFLAGS_TF | X86_EFLAGS_DF | ...);
}
```

## 状态切换的开销

用户态到内核态的切换是有开销的：

```c
// 每次系统调用的开销（大约）
// 上下文保存：~100-200 周期
// 内核处理：视具体操作而定
// 上下文恢复：~100-200 周期

// 如果系统调用太频繁...
for (int i = 0; i < 1000000; i++) {
    write(fd, buf, 1);  // 100 万次系统调用
}
// 建议：用缓冲 I/O 或 writev 合并请求
```

### 减少系统调用的策略

```c
// 问题：频繁的小写入
for (int i = 0; i < 1000; i++) {
    write(fd, "x", 1);  // 1000 次系统调用
}

// 优化：合并写入
char buf[1000];
memset(buf, 'x', 1000);
write(fd, buf, 1000);  // 1 次系统调用

// 或者使用 setvbuf 设置缓冲
setvbuf(fp, buffer, _IOFBF, BUFSIZ);
```

## 面试高频问题

**Q1：用户态和内核态有什么区别？**

用户态下程序只能访问受限资源，不能直接访问硬件或执行特权指令。内核态下可以执行任何指令、访问任何内存。两种状态通过特权级别（Ring）区分。

**Q2：系统调用和普通函数调用有什么区别？**

普通函数调用在用户态完成，不切换特权级别。系统调用需要从用户态切换到内核态，涉及上下文保存、特权级别检查、内核函数执行等，开销更大。

**Q3：如何触发从用户态到内核态的切换？**

可以通过系统调用（程序主动请求）、中断（硬件事件）、异常（CPU 检测到错误）、陷阱（程序主动触发）等方式切换。

**Q4：syscall 和 int 0x80 有什么区别？**

syscall 是 x86-64 新增的专门指令，比 int 0x80 更快：不需要访问 IDT、直接跳转、不需要验证中断门权限等。

**Q5：为什么系统调用有开销？**

系统调用需要保存当前上下文（寄存器等）、切换特权级别、在内核和用户态之间复制数据、内核执行权限检查等。这些操作需要 CPU 时间，尤其是频繁调用时开销明显。

理解内核态/用户态和系统调用，是理解操作系统安全模型的关键。
