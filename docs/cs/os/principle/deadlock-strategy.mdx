# 死锁处理策略详解

两个和尚挑水喝，三个和尚没水喝。

进程 A 握着资源 1，等着资源 2；进程 B 握着资源 2，等着资源 1。谁也不肯让步，谁也完不成。这就是**死锁**。

死锁是并发程序中最让人头疼的问题之一。如果你已经理解了死锁的四个必要条件，那么问题是：**如何解决死锁？**

## 死锁处理的四种策略

## 死锁预防

预防是从根本上阻止死锁：**破坏四个必要条件之一**。

### 条件一：破坏互斥条件

允许资源同时被多个进程共享。

```c
// 问题：不是所有资源都可以共享
// 打印机可以共享（通过 spooling）
// 但互斥锁本身不能共享

// spooling 实现打印机共享
struct print_queue {
    char jobs[MAX_JOBS][PAGE_SIZE];
    int job_count;
    struct semaphore mutex;      // 保护队列
    struct semaphore item_available;  // 有作业可打印
    struct semaphore space_available; // 有空闲空间
};

// 打印机守护进程
void printer_daemon() {
    while (1) {
        down(&item_available);    // 等待作业
        down(&mutex);              // 保护临界区
        struct job = dequeue();    // 取作业
        up(&mutex);
        
        print(job);                // 打印
        
        up(&space_available);      // 释放空间
    }
}
```

**适用范围：** 有限，主要用于可共享的资源（如只读文件、打印队列）。

### 条件二：破坏占有并等待条件

两种方法：

```c
// 方法一：进程开始时一次性申请所有需要的资源
// 问题：可能提前占用不需要的资源，利用率低

process() {
    lock(resource1);
    lock(resource2);
    lock(resource3);
    // ... 一次性申请所有资源
    do_work();
    unlock_all();
}

// 方法二：进程只有不持有资源时才能申请新资源
// 问题：实现复杂，可能导致饥饿

process() {
    while (1) {
        lock(resource1);
        if (need(resource2)) {
            unlock(resource1);  // 释放已持有的资源
            wait_for(resource2);
            lock(resource1);
            lock(resource2);
        } else {
            break;
        }
    }
    do_work();
    unlock_all();
}
```

**适用范围：** 方法一简单但利用率低；方法二复杂但更灵活。

### 条件三：破坏不可抢占条件

如果进程请求的资源无法立即分配，强制释放已持有的资源。

```c
// 伪代码
if (requested_resource.is_available()) {
    allocate(requested_resource);
} else {
    // 尝试抢占其他进程的同类资源
    struct process* holder = find_holder(requested_resource);
    
    if (holder != NULL) {
        // 强制从持有者手中夺走
        if (holder->priority < current_process->priority) {
            // 高优先级可以抢占低优先级进程的同类资源
            force_release(holder, requested_resource);
            allocate(requested_resource);
        }
    }
}
```

**适用范围：** 适用于状态可以保存和恢复的资源（如 CPU、内存），不适用于如打印机等不能保存状态的资源。

### 条件四：破坏循环等待条件

为资源分配一个全局顺序，进程必须按顺序申请资源。

```c
// 例如：资源编号 R1 < R2 < R3 < R4

// 正确做法（不会死锁）
void correct_order() {
    lock(R1);      // 先申请 R1
    lock(R2);      // 再申请 R2
    // ...
    lock(R4);
    do_work();
    unlock_all();
}

// 错误做法（可能死锁）
void wrong_order() {
    lock(R2);      // 先申请 R2
    lock(R1);      // 再申请 R1（R1 < R2）
    // 如果另一个进程先 lock(R1) 再 lock(R2)，就会死锁
}
```

**数学证明：** 按全局顺序申请资源，循环等待不可能发生：

```
如果有循环等待: P1 → P2 → ... → Pn → P1
假设申请顺序: R1 < R2 < ... < Rn

P1 持有 R1，等待 R2（R1 < R2）
P2 持有 R2，等待 R3（R2 < R3）
...
Pn 持有 Rn，等待 R1（Rn < R1）

但 R1 < R2 < ... < Rn < R1 不可能成立！
所以循环等待不存在。
```

**适用范围：** 适用于需要多个资源的场景，如数据库、事务处理系统。

### 预防策略对比

| 条件 | 方法 | 优点 | 缺点 | 适用范围 |
|---|---|---|---|---|
| 互斥 | 共享资源 | 简单 | 可能不可行 | 只读资源 |
| 占有并等待 | 一次性申请 | 简单 | 资源利用率低 | 资源少且确定 |
| 占有并等待 | 只在不持有时申请 | 较灵活 | 实现复杂 | 通用 |
| 不可抢占 | 强制抢占 | 灵活 | 状态丢失 | 状态可保存的资源 |
| 循环等待 | 资源顺序分配 | 高效 | 需要预知资源 | 资源固定 |

## 死锁避免

死锁避免不是静态地破坏条件，而是在资源分配时**动态检查**分配后系统是否仍处于安全状态。

### 安全状态与不安全状态

**安全状态：** 存在一个进程序列 P1, P2, ..., Pn，使得：
- P1 可以完成（当前资源足够）
- P1 完成后释放资源，P2 可以完成
- ...
- 所有进程都能完成

**不安全状态：** 不存在这样的序列，某些进程可能死锁。

```
安全状态示例:

系统有 12 台打印机，当前分配了 9 台，可用 3 台。

┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│  进程    最大需求    当前分配    仍需        可用              │
│   P1        10          5        5           3                  │
│   P2         4          2        2           ──► P2 完成       │
│   P3         9          2        7           3+2=5             │
│                                      ──► P1 完成               │
│                                         5+5=10                  │
│                                      ──► P3 完成               │
│                                                                 │
│  结论：存在安全序列 P2 → P1 → P3，系统处于安全状态               │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘

不安全状态示例:

┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│  进程    最大需求    当前分配    仍需        可用              │
│   P1        10          5        5           3                  │
│   P2         4          2        2           ──► P2 需要 2 台   │
│   P3         9          7        2           3 不够！           │
│                                      ──► P3 无法完成            │
│                                      ──► P1 也无法完成          │
│                                                                 │
│  结论：死锁可能发生，系统处于不安全状态                           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 银行家算法

银行家算法是 Dijkstra 提出的经典死锁避免算法。

```c
#include <stdio.h>
#include <stdbool.h>

#define N 5   // 进程数
#define M 3   // 资源类型数

// 资源向量：R1=喷墨打印机, R2=激光打印机, R3=绘图仪
int available[M] = {3, 3, 2};      // 可用资源

int maximum[N][M] = {                // 最大需求矩阵
    {7, 5, 3},
    {3, 2, 2},
    {9, 0, 2},
    {2, 2, 2},
    {4, 3, 3}
};

int allocation[N][M] = {             // 当前分配矩阵
    {0, 1, 0},
    {2, 0, 0},
    {3, 0, 2},
    {2, 1, 1},
    {0, 0, 2}
};

int need[N][M];                      // 需求矩阵 = 最大需求 - 当前分配

// 初始化
void init() {
    for (int i = 0; i < N; i++) {
        for (int j = 0; j < M; j++) {
            need[i][j] = maximum[i][j] - allocation[i][j];
        }
    }
}

// 打印当前状态
void print_state() {
    printf("\n当前状态:\n");
    printf("可用资源: ");
    for (int j = 0; j < M; j++) printf("%d ", available[j]);
    printf("\n");
    
    printf("进程  |  最大需求  |  当前分配  |   需求\n");
    for (int i = 0; i < N; i++) {
        printf(" P%d    ", i);
        for (int j = 0; j < M; j++) printf("%d ", maximum[i][j]);
        printf("    ");
        for (int j = 0; j < M; j++) printf("%d ", allocation[i][j]);
        printf("      ");
        for (int j = 0; j < M; j++) printf("%d ", need[i][j]);
        printf("\n");
    }
}

// 检查安全性
bool check_safety() {
    int work[M];
    bool finish[N] = {false};
    
    // work = available
    for (int j = 0; j < M; j++) work[j] = available[j];
    
    printf("\n安全性检查:\n");
    
    int count = 0;
    while (count < N) {
        bool found = false;
        
        for (int i = 0; i < N; i++) {
            if (!finish[i]) {
                // 检查进程 i 的需求是否满足
                bool can_finish = true;
                for (int j = 0; j < M; j++) {
                    if (need[i][j] > work[j]) {
                        can_finish = false;
                        break;
                    }
                }
                
                if (can_finish) {
                    // 进程 i 可以完成
                    printf("P%d 可以完成，释放资源: ", i);
                    for (int j = 0; j < M; j++) {
                        work[j] += allocation[i][j];
                        printf("%d ", allocation[i][j]);
                    }
                    printf("\n");
                    finish[i] = true;
                    found = true;
                    count++;
                }
            }
        }
        
        if (!found) {
            printf("找不到可以完成的进程！\n");
            return false;
        }
    }
    
    printf("所有进程都可以完成，系统处于安全状态\n");
    return true;
}

// 请求资源
bool request_resources(int process, int request[M]) {
    printf("\n进程 P%d 请求资源: ", process);
    for (int j = 0; j < M; j++) printf("%d ", request[j]);
    
    // 1. 检查请求是否超过需求
    for (int j = 0; j < M; j++) {
        if (request[j] > need[process][j]) {
            printf("\n错误: 请求超过最大需求！\n");
            return false;
        }
    }
    
    // 2. 检查请求是否超过可用
    for (int j = 0; j < M; j++) {
        if (request[j] > available[j]) {
            printf("\n进程 P%d 必须等待（资源不足）\n", process);
            return false;
        }
    }
    
    // 3. 尝试分配
    printf("\n尝试分配...\n");
    for (int j = 0; j < M; j++) {
        available[j] -= request[j];
        allocation[process][j] += request[j];
        need[process][j] -= request[j];
    }
    
    // 4. 检查安全性
    if (check_safety()) {
        printf("分配成功！\n");
        return true;
    } else {
        // 回滚
        printf("分配导致不安全状态，回滚！\n");
        for (int j = 0; j < M; j++) {
            available[j] += request[j];
            allocation[process][j] -= request[j];
            need[process][j] += request[j];
        }
        return false;
    }
}

int main() {
    init();
    print_state();
    
    // 测试：进程 1 请求 (1, 0, 2)
    int request1[M] = {1, 0, 2};
    request_resources(1, request1);
    
    print_state();
    
    return 0;
}
```

### 银行家算法的局限性

| 局限性 | 说明 |
|---|---|
| 需要预知最大资源需求 | 实际中很难准确预知 |
| 进程数必须固定 | 动态创建/销毁进程时难以处理 |
| 资源类型必须固定 | 难以扩展 |
| 计算复杂度高 | O(M × N²)，不适用于大规模系统 |
| 利用率可能较低 | 保守的资源分配 |

## 死锁检测

如果系统没有采用预防或避免策略，就必须周期性地检测死锁。

### 资源分配图法

适用于每种资源只有单个实例的情况。

```c
// 资源分配图
// 节点：进程节点 (P1, P2) + 资源节点 (R1, R2)
// 边：进程到资源的边（请求） + 资源到进程的边（分配）

// 检测循环
bool detect_deadlock() {
    // 构建资源分配图
    // 使用深度优先搜索检测循环
    
    // 如果存在从进程 Pi 到进程 Pi 的路径（经过资源节点）
    // 则存在死锁
    
    return has_cycle();
}
```

### 等待图法

资源分配图的简化版，适用于每种资源类型只有一个实例。

```
资源分配图                          等价的等待图
┌──────────────┐                   ┌──────────────┐
│              │                   │              │
│    R1        │                   │   (等待图)   │
│   ┌──┴──┐    │                   │              │
│   │     │    │                   │   P1 ──► P2  │
│  ┌┴┐   ┌┴┐   │                   │    ↑    │   │
│  │  │   │  │   │                   │    │    │   │
│  P1    P2     │                   │    └────┘   │
│              │                   │              │
└──────────────┘                   └──────────────┘

P1 持有 R1，等待 R2
P2 持有 R2，等待 R1
等待图中存在环: P1 → P2 → P1 → 死锁！
```

### 基于矩阵的检测算法

适用于每种资源有多个实例的情况。

```c
// 检测算法
// available: 可用资源向量
// allocation: 当前分配矩阵
// request: 当前请求矩阵

bool detect_deadlock_matrix() {
    int work[M];
    bool finish[N] = {false};
    
    // 初始化 work = available
    for (int j = 0; j < M; j++) work[j] = available[j];
    
    // 将所有 finish[i] = false 的进程加入待检查队列
    // 如果进程的 request 全为 0，直接设为 finish
    
    for (int i = 0; i < N; i++) {
        bool all_zero = true;
        for (int j = 0; j < M; j++) {
            if (request[i][j] != 0) {
                all_zero = false;
                break;
            }
        }
        if (all_zero) finish[i] = true;
    }
    
    // 查找可以完成的进程
    int count = 0;
    while (count < N) {
        bool found = false;
        
        for (int i = 0; i < N; i++) {
            if (!finish[i]) {
                bool can_satisfy = true;
                for (int j = 0; j < M; j++) {
                    if (request[i][j] > work[j]) {
                        can_satisfy = false;
                        break;
                    }
                }
                
                if (can_satisfy) {
                    // 进程 i 可以完成
                    for (int j = 0; j < M; j++) {
                        work[j] += allocation[i][j];
                    }
                    finish[i] = true;
                    found = true;
                    count++;
                }
            }
        }
        
        if (!found) break;  // 没有进程可以完成
    }
    
    // 如果有进程未完成，存在死锁
    for (int i = 0; i < N; i++) {
        if (!finish[i]) {
            printf("进程 P%d 死于死锁\n", i);
            return true;  // 存在死锁
        }
    }
    
    return false;  // 无死锁
}
```

### 检测时机

| 策略 | 优点 | 缺点 |
|---|---|---|
| 每次资源请求时检测 | 能尽早发现死锁 | 开销大 |
| 定期检测 | 开销可控 | 延迟发现 |
| CPU 利用率低时检测 | 利用空闲时间 | 不及时 |
| 资源利用率低且吞吐量下降时检测 | 针对性检测 | 实现复杂 |

## 死锁解除

检测到死锁后，需要采取行动解除死锁。

### 方法一：进程终止

最简单的方案：终止一个或多个死锁进程。

```c
// 终止进程的选择策略
enum TerminationStrategy {
    TERMINATE_LOWEST_PRIORITY,     // 终止优先级最低的进程
    TERMINATE_FEWEST_RESOURCES,    // 终止持有资源最少的进程
    TERMINATE_SHORTEST_TIME,        // 终止运行时间最短的进程
    TERMINATE_MINIMUM_COST;         // 终止撤销代价最小的进程
};

// 终止进程的后果
void terminate_process(int pid) {
    // 1. 释放所有持有的资源
    release_all_resources(pid);
    
    // 2. 如果有子进程，可能需要级联终止
    if (cascade_termination) {
        for (child : get_children(pid)) {
            terminate_process(child);
        }
    }
    
    // 3. 发送信号通知
    kill(pid, SIGKILL);
}
```

**问题：**
- 可能丢失已完成的工作
- 需要选择合适的牺牲者
- 级联终止可能终止无辜进程

### 方法二：资源抢占

从死锁进程中抢占资源，分配给其他进程。

```c
// 资源抢占策略
void preempt_resource(int victim_pid, int resource) {
    // 1. 选择牺牲进程（持有资源多但优先级低的）
    
    // 2. 回滚进程状态到安全点
    //    需要检查点支持
    checkpoint_t cp = create_checkpoint(victim_pid);
    
    // 3. 释放资源
    release_resource(victim_pid, resource);
    
    // 4. 恢复进程（可选）
    //    - 继续运行（可能再次死锁）
    //    - 放到就绪队列，等待重新调度
}
```

**问题：**
- 饥饿：同一进程可能反复被选中抢占
- 需要回滚机制
- 可能导致同一进程反复死锁

### 方法三：进程回滚

将死锁进程回滚到早期状态（检查点），然后重新调度。

```c
// 检查点恢复
typedef struct {
    int pid;
    void* registers;      // CPU 寄存器
    void* memory;        // 内存快照
    struct file_state;   // 文件状态
    // ...
} checkpoint_t;

void rollback_to_checkpoint(int pid, checkpoint_t* cp) {
    // 1. 挂起进程
    suspend_process(pid);
    
    // 2. 恢复内存
    restore_memory(pid, cp->memory);
    
    // 3. 恢复寄存器
    restore_registers(pid, cp->registers);
    
    // 4. 恢复文件状态
    restore_files(pid, cp->file_state);
    
    // 5. 将进程放入就绪队列
    add_to_ready_queue(pid);
}
```

### 解除策略对比

| 方法 | 说明 | 代价 | 风险 |
|---|---|---|---|
| **终止进程** | 强制终止死锁进程 | 丢失工作 | 可能误杀 |
| **资源抢占** | 剥夺进程资源 | 需要回滚 | 可能饥饿 |
| **回滚重启** | 回滚到检查点 | 检查点开销 | 可能再次死锁 |

## 实际系统中的选择

| 系统类型 | 推荐策略 |
|---|---|
| 操作系统 | 通常使用预防 + 定期检测 + 终止 |
| 数据库系统 | 死锁避免 + 锁超时 + 事务回滚 |
| 嵌入式实时系统 | 静态资源顺序 + 优先级继承 |
| 通用服务器 | 检测 + 终止代价最小者 |

## 面试高频问题

**Q1：死锁预防和死锁避免的区别？**

预防是静态地破坏死锁的必要条件之一（互斥、占有并等待、不可抢占、循环等待）。避免是在资源分配时动态检查分配后系统是否仍处于安全状态（如银行家算法）。

**Q2：什么是安全状态？**

安全状态是指存在一个进程序列，使得按照这个序列调度，每个进程都能顺利完成而不会死锁。处于安全状态的系统，可以继续正常工作。

**Q3：银行家算法的核心思想是什么？**

在资源分配前，判断分配后系统是否仍然处于安全状态。如果不安全，则拒绝分配，等待其他进程释放资源后再尝试。

**Q4：如何选择死锁解除的牺牲进程？**

考虑因素：进程优先级、已运行时间、持有资源数量、所需资源数量、批处理还是交互式。选择代价最小的进程终止。

**Q5：为什么大多数操作系统选择检测+解除而不是预防？**

预防会牺牲系统效率和资源利用率，而实际系统中死锁发生的概率相对较低。检测+解除可以更灵活地处理死锁，平时保持较高效率，死锁发生时再处理。

理解死锁处理的各种策略，是在实际系统中设计可靠并发程序的关键。
