# 中断与异常处理机制

你正在玩游戏，键盘上按了一下「空格键」，角色跳了起来。

从按下按键到角色跳跃，CPU 经历了什么？

答案是**中断**——键盘控制器发出中断信号，CPU 暂停当前任务，跳转到键盘处理程序，处理完后再回来继续游戏。

中断是现代操作系统高效运行的基础：CPU 不需要一直轮询「有没有事件发生」，而是可以专心做当前任务，有事情发生时再响应。

## 中断的本质

### 什么是中断？

中断是 CPU 对**异步事件**的响应机制。当中断发生时，CPU 暂停当前执行，保存上下文，跳转到预先设置好的处理程序（中断处理例程），处理完后再返回继续执行。

```
没有中断时:
┌─────────────────────────────────────────┐
│                                         │
│  CPU 执行指令 1                         │
│        │                                │
│        ▼                                │
│  CPU 执行指令 2                         │
│        │                                │
│        ▼                                │
│  CPU 执行指令 3  ← 如果要等 I/O 完成...  │
│        │                                │
│        │  (CPU 空闲等待)                │
│        │                                │
│        ▼                                │
│  CPU 收到 I/O 完成信号                   │
│                                         │
└─────────────────────────────────────────┘

有中断时:
┌─────────────────────────────────────────┐
│                                         │
│  CPU 执行指令 1                         │
│        │                                │
│        ▼                                │
│  CPU 执行指令 2                         │
│        │                                │
│        ▼                                │
│  CPU 执行指令 3                         │
│        │                                │
│        ▼                                │
│  I/O 设备发出中断 ──► CPU 处理中断       │
│        │                  │              │
│        │                  ▼              │
│        │            处理键盘事件          │
│        │                  │              │
│        │                  ▼              │
│        ◄────── 返回继续执行指令 4         │
│                                         │
└─────────────────────────────────────────┘
```

### 中断 vs 轮询

```
轮询方式（Polling）:
CPU 不断询问设备："有数据吗？有数据吗？有数据吗？..."
- 简单，但浪费 CPU 时间

中断方式（Interrupt）:
设备主动通知："我有数据了！"
- 高效，CPU 可以做其他事
```

### 异常的三种类型

**1. 故障（Fault）**

可恢复的异常。处理完后返回到触发异常的指令，重新执行。

```
故障示例：页错误（Page Fault）
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  访问虚拟地址 P                                              │
│       │                                                     │
│       ▼                                                     │
│  页表项有效位 = 0?                                          │
│       │                                                     │
│       ▼                                                     │
│  页错误异常 ──► 操作系统加载页面                             │
│       │                                                     │
│       ▼                                                     │
│  IRET ──► 重新执行触发异常的指令（这次页已在内存）           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**2. 陷阱（Trap）**

程序主动触发的异常，处理完后返回到触发异常的指令的下一条指令。

```
陷阱示例：系统调用
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  执行 int 0x80 (或 syscall)                                 │
│       │                                                     │
│       ▼                                                     │
│  保存上下文，跳转到内核                                      │
│       │                                                     │
│       ▼                                                     │
│  内核处理系统调用                                           │
│       │                                                     │
│       ▼                                                     │
│  IRET ──► 执行 int 指令的下一条指令                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**3. 终止（Abort）**

严重的不可恢复错误，处理程序只记录错误信息，然后终止程序。

```
终止示例：硬件错误
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  发生严重错误（如内存奇偶校验错误）                          │
│       │                                                     │
│       ▼                                                     │
│  终止异常处理程序                                            │
│       │                                                     │
│       ▼                                                     │
│  记录错误信息（寄存器值、指令地址等）                        │
│       │                                                     │
│       ▼                                                     │
│  终止进程或重启系统                                          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 中断向量表与 IDT

### x86 的中断向量表

x86 处理器支持 256 个中断向量（0-255）。

```c
// 中断向量分配
// 0-31: CPU 异常（由 Intel 定义）
// 32-127: 硬件中断（可编程）
// 128: 系统调用 (int 0x80)
// 129-255: 可用

// 典型分配
#define INT_DIVIDE_ERROR      0   // 除零错误
#define INT_DEBUG             1   // 调试异常
#define INT_NMI               2   // 非屏蔽中断
#define INT_BREAKPOINT        3   // 断点 (int3)
#define INT_OVERFLOW          4   // 溢出 (into)
#define INT_BOUNDS            5   // 边界检查
#define INT_INVALID_OPCODE    6   // 无效操作码
#define INT_DEVICE_NOT_AVAIL  7   // 设备不可用
#define INT_DOUBLE_FAULT       8   // 双重错误
#define INT_COPRO_SEG_OVERRUN 9   // 协处理器段超限
#define INT_INVALID_TSS       10  // 无效 TSS
#define INT_SEG_NOT_PRESENT   11  // 段不存在
#define INT_STACK_FAULT       12  // 栈错误
#define INT_GENERAL_PROTECT   13  // 保护错误
#define INT_PAGE_FAULT        14  // 页错误
#define INT_FP_ERROR          16  // 浮点错误
// ...
```

### IDT（中断描述符表）

IDT 是 x86 架构中存储中断/异常处理程序入口点的表。

```c
// IDT 条目结构 (Gate Descriptor)
struct idt_entry {
    unsigned short offset_low;    // 处理程序地址低 16 位
    unsigned short selector;       // 代码段选择子
    unsigned char  ist: 3;         // 中断栈表（备用栈）
    unsigned char  zero: 5;        // 固定为 0
    unsigned char  type: 4;        // 门类型
    unsigned char  D: 1;           // 大小 (1=32位)
    unsigned char  DPL: 2;         // 描述符特权级
    unsigned char  present: 1;     // 是否存在
    unsigned short offset_mid;    // 处理程序地址中 16 位
    unsigned int offset_high;      // 处理程序地址高 32 位
    unsigned int reserved;         // 保留
};
```

### IDT 的设置

```c
// Linux 中设置 IDT
void idt_init() {
    // IDT 基地址和限制
    struct desc_ptr idt_descr;
    idt_descr.address = (unsigned long)idt_table;
    idt_descr.size = sizeof(idt_table) - 1;
    
    // 加载 IDT
    __asm__ volatile ("lidt %0" : : "m"(idt_descr));
}

// 设置一个中断门
void set_intr_gate(int n, void* addr) {
    struct idt_entry* gate = &idt_table[n];
    gate->offset_low = (unsigned long)addr & 0xFFFF;
    gate->offset_mid = ((unsigned long)addr >> 16) & 0xFFFF;
    gate->offset_high = ((unsigned long)addr >> 32) & 0xFFFFFFFF;
    gate->selector = KERNEL_CS;  // 内核代码段
    gate->ist = 0;
    gate->type = GATE_INTERRUPT;
    gate->D = 1;
    gate->DPL = 0;
    gate->present = 1;
}
```

## 中断处理流程

### 完整的中断处理过程

```
┌─────────────────────────────────────────────────────────────────────┐
│                        中断处理流程                                  │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  1. 中断发生                                                        │
│     ┌──────────────────────────────────────────────────────────┐   │
│     │ 硬件中断: IRQ 引脚激活 → 中断控制器 (PIC/APIC)          │   │
│     │ 软件中断: 执行 int 指令                                   │   │
│     │ 异常: CPU 检测到错误条件                                   │   │
│     └──────────────────────────────────────────────────────────┘   │
│                              │                                     │
│                              ▼                                     │
│  2. 确定中断号                                                      │
│     ┌──────────────────────────────────────────────────────────┐   │
│     │ PIC/APIC 采样 IRQ → 转换为中断向量号                     │   │
│     │ CPU 接收向量号 (0-255)                                   │   │
│     └──────────────────────────────────────────────────────────┘   │
│                              │                                     │
│                              ▼                                     │
│  3. 保存上下文                                                      │
│     ┌──────────────────────────────────────────────────────────┐   │
│     │  push rflags, rip, cs, rflags, rsp, ss                   │   │
│     │  (如果特权级切换，还需要切换栈)                           │   │
│     └──────────────────────────────────────────────────────────┘   │
│                              │                                     │
│                              ▼                                     │
│  4. 查 IDT 获取处理程序地址                                         │
│     ┌──────────────────────────────────────────────────────────┐   │
│     │  IDT[向量号] → 处理程序入口                               │   │
│     └──────────────────────────────────────────────────────────┘   │
│                              │                                     │
│                              ▼                                     │
│  5. 跳转到处理程序                                                  │
│     ┌──────────────────────────────────────────────────────────┐   │
│     │  call do_IRQ / do_异常处理函数                            │   │
│     └──────────────────────────────────────────────────────────┘   │
│                              │                                     │
│                              ▼                                     │
│  6. 硬件中断特有的操作                                              │
│     ┌──────────────────────────────────────────────────────────┐   │
│     │  向中断控制器发送 EOI (End Of Interrupt)                  │   │
│     │  重新使能中断 (如果之前禁用了)                             │   │
│     └──────────────────────────────────────────────────────────┘   │
│                              │                                     │
│                              ▼                                     │
│  7. 执行处理程序                                                   │
│     ┌──────────────────────────────────────────────────────────┐   │
│     │  键盘: 读取键盘码 → 转换为 ASCII → 放入缓冲区             │   │
│     │  磁盘: 复制数据 → 唤醒等待进程                            │   │
│     │  网络: 解析数据包 → 放入 socket 缓冲区                     │   │
│     └──────────────────────────────────────────────────────────┘   │
│                              │                                     │
│                              ▼                                     │
│  8. 返回                                                           │
│     ┌──────────────────────────────────────────────────────────┐   │
│     │  iretq 指令 → 恢复上下文 → 返回用户态                     │   │
│     └──────────────────────────────────────────────────────────┘   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 中断处理程序的实现

```c
// Linux 的中断处理框架
irqreturn_t handle_irq(int irq, void* dev_id) {
    // 1. 确认中断源
    struct irq_desc* desc = irq_to_desc(irq);
    
    // 2. 遍历该 IRQ 的处理程序链表
    struct irqaction* action;
    for (action = desc->action; action; action = action->next) {
        irqreturn_t retval = action->handler(irq, action->dev_id);
        if (retval == IRQ_HANDLED) {
            // 处理成功
        }
    }
    
    // 3. 向中断控制器发送 EOI
    if (desc->chip)
        desc->chip->irq_eoi(&desc->irq_data);
    
    return IRQ_HANDLED;
}
```

## 中断嵌套与优先级

### 中断优先级

不同类型的中断有不同的优先级：

```c
// 典型的中断优先级（从高到低）
// 1. 机器检查异常 (Machine Check) - 最高
// 2. 硬件错误 (Hardware Error)
// 3. 陷阱 (Trap)
// 4. 外部硬件中断 (External Interrupt)
// 5. 故障 (Fault)
// 6. 终止 (Abort)
// 7. 程序中断 (Program Interrupt) - 最低
```

### 中断嵌套

某些系统支持中断嵌套：处理一个中断时，可以被更高优先级的中断打断。

```c
// 中断嵌套示例
void keyboard_irq_handler() {
    // 保存上下文...
    
    // 处理键盘
    handle_keyboard();
    
    // 如果此时收到更高优先级的中断（如磁盘）
    // 会嵌套处理磁盘中断
    
    // 恢复上下文，返回
}

// 嵌套的好处：提高系统响应能力
// 嵌套的风险：栈溢出、中断风暴
```

### Linux 的中断处理

Linux 采用**顶部/底部**（Top Half / Bottom Half）模式：

```c
// Top Half: 快速处理，尽量短
irqreturn_t do_irq(int irq, void* dev_id) {
    // 立即完成：
    // 1. 读取硬件状态
    // 2. 发送 EOI
    // 3. 快速响应
    
    // 推迟耗时操作到 Bottom Half
    tasklet_schedule(&my_tasklet);  // 或 workqueue
    
    return IRQ_HANDLED;
}

// Bottom Half: 耗时操作
void my_tasklet_handler(unsigned long data) {
    // 耗时处理：
    // 1. 数据包解析
    // 2. 协议处理
    // 3. 唤醒等待进程
}
```

## 软中断与信号

### 软中断（Software Interrupt）

软中断是软件主动触发的中断，常用于系统调用。

```asm
; x86 软中断
int 0x80    ; Linux 系统调用
int 3       ; 断点（调试）
into         ; 溢出检查
```

### 信号（Signal）

信号是 Unix 系统中的进程间通信机制，类似于软件中断。

```c
// 常见信号
#define SIGHUP   1   // 挂起
#define SIGINT   2   // 中断 (Ctrl+C)
#define SIGKILL  9   // 杀死
#define SIGSEGV 11   // 段错误
#define SIGTERM 15   // 终止
#define SIGUSR1 10   // 用户自定义
#define SIGUSR2 12   // 用户自定义

// 进程可以捕获信号并自定义处理
signal(SIGINT, my_handler);  // 捕获 Ctrl+C
```

## 面试高频问题

**Q1：中断和异常有什么区别？**

中断是异步的，来自硬件或程序主动触发，不影响当前指令执行。异常是同步的，由 CPU 在执行指令时检测到错误条件产生，通常与当前指令相关。

**Q2：页错误（Page Fault）是什么类型的异常？**

页错误是**故障（Fault）**类型的异常。处理程序修复问题后，会重新执行触发异常的指令。

**Q3：什么是 IDT？**

中断描述符表（Interrupt Descriptor Table）是 x86 架构中存储中断/异常处理程序入口点的表，共 256 个条目。

**Q4：为什么中断处理要快？**

因为中断处理期间，系统无法响应其他事件（除非嵌套）。长时间的中断处理会延迟其他中断的响应，可能导致数据丢失（如网卡丢包）。

**Q5：软中断和硬中断有什么区别？**

硬中断由硬件设备触发，软中断由软件（int 指令）触发。硬中断处理程序执行时可能被打断，软中断则是程序主动请求。

理解中断与异常处理机制，是理解操作系统响应外部事件的核心。
