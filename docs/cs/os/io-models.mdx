## 一、IO 基础

### 1.1 IO 操作的过程

一次完整的 IO 操作涉及两个阶段：

```
IO 操作的两个阶段：

阶段一（等待阶段）：
- 数据准备：从磁盘/网络到达内核缓冲区
- 需要等待数据就绪

阶段二（复制阶段）：
- 数据复制：从内核缓冲区复制到用户缓冲区
- 内核态到用户态的数据拷贝
```

### 1.2 同步 vs 异步

| 类型 | 等待阶段 | 复制阶段 |
|------|----------|----------|
| 同步 | 阻塞等待 | 同步复制 |
| 异步 | 不阻塞 | 由内核完成，通知应用 |

```
同步 IO：应用主动等待并获取结果
异步 IO：内核完成操作后通知应用
```

## 二、五种 IO 模型

### 2.1 阻塞 IO（Blocking IO）

**特点**：应用进程在两个阶段都阻塞。

```
阻塞 IO 过程：

应用进程                    内核
    |                         |
    |──── recvfrom ──────────→|  等待数据（阻塞）
    |                         |  数据到达内核缓冲区
    |                         |  复制数据到用户空间
    |←─────── 返回数据 ───────|  唤醒应用
    |                         |
处理数据
```

**示例**：

```java
// 阻塞 IO 示例
public class BlockingIO {
    public static void main(String[] args) throws IOException {
        ServerSocket server = new ServerSocket(8080);
        Socket client = server.accept();  // 阻塞等待连接

        // 读取数据 - 阻塞直到数据就绪
        InputStream in = client.getInputStream();
        byte[] buf = new byte[1024];
        int len = in.read(buf);  // 阻塞直到数据到达

        // 处理数据...
    }
}
```

### 2.2 非阻塞 IO（Non-blocking IO）

**特点**：应用进程在等待阶段不阻塞，立即返回；需要轮询检查数据是否就绪。

```
非阻塞 IO 过程：

应用进程                    内核
    |                         |
    |──── recvfrom ──────────→|  返回 EAGAIN（数据未就绪）
    |                         |
    |──── recvfrom ──────────→|  返回 EAGAIN
    |                         |
    |──── recvfrom ──────────→|  返回数据
    |←─────── 返回数据 ───────|
    |                         |
```

**示例**：

```java
// 非阻塞 IO 示例
public class NonBlockingIO {
    public static void main(String[] args) throws IOException {
        ServerSocketChannel server = ServerSocketChannel.open();
        server.socket().bind(new InetSocketAddress(8080));
        server.configureBlocking(false);  // 设置非阻塞

        while (true) {
            SocketChannel client = server.accept();  // 非阻塞
            if (client != null) {
                // 处理客户端
                handleClient(client);
            }
        }
    }
}
```

### 2.3 IO 多路复用（IO Multiplexing）

**特点**：使用 select/poll/epoll 同时监控多个 IO fd，在一个线程中处理多个连接。

```
IO 多路复用过程：

应用进程                    内核
    |                         |
    |──── select/epoll ──────→|  阻塞等待（监控多个 fd）
    |                         |  数据就绪
    |←─────── 返回就绪列表 ────|
    |                         |
    |──── recvfrom ──────────→|  数据复制（此时一定就绪）
    |←─────── 返回数据 ───────|
    |                         |
```

**示例**：

```java
// IO 多路复用示例（Java NIO）
public class MultiplexingIO {
    public static void main(String[] args) throws IOException {
        Selector selector = Selector.open();
        ServerSocketChannel server = ServerSocketChannel.open();
        server.register(selector, SelectionKey.OP_ACCEPT);

        while (true) {
            selector.select();  // 阻塞，直到有就绪事件
            Set<SelectionKey> keys = selector.selectedKeys();

            for (SelectionKey key : keys) {
                if (key.isAcceptable()) {
                    // 处理连接
                } else if (key.isReadable()) {
                    // 处理读取
                }
            }
        }
    }
}
```

### 2.4 信号驱动 IO（Signal-driven IO）

**特点**：应用进程预先注册信号处理函数，内核在数据就绪时发送 SIGIO 信号。

```
信号驱动 IO 过程：

应用进程                    内核
    |                         |
    |──── SIGIO 注册 ─────────→|
    |                         |
    |──── recvfrom ──────────→|  返回（不等待）
    |                         |
    |              数据就绪 ────|
    |←─────── SIGIO 信号 ──────|
    |                         |
    |──── recvfrom ──────────→|  复制数据
    |←─────── 返回数据 ───────|
```

### 2.5 异步 IO（Asynchronous IO）

**特点**：整个 IO 操作（包括等待和复制）都由内核完成，完成后通知应用。

```
异步 IO 过程：

应用进程                    内核
    |                         |
    |──── aio_read ──────────→|  返回（立即返回）
    |                         |
    |                         |  等待数据（不阻塞应用）
    |                         |  复制数据到用户空间
    |                         |
    |←─────── 完成信号 ────────|  通知完成
    |                         |
处理数据（此时数据已可用）
```

**示例**：

```java
// 异步 IO 示例（Java AIO）
public class AsyncIO {
    public static void main(String[] args) throws IOException {
        AsynchronousServerSocketChannel server =
            AsynchronousServerSocketChannel.open().bind(new InetSocketAddress(8080));

        server.accept(null, new CompletionHandler<AsynchronousSocketChannel, Void>() {
            @Override
            public void completed(AsynchronousSocketChannel client, Void attachment) {
                server.accept(null, this);  // 继续接受连接

                ByteBuffer buf = ByteBuffer.allocate(1024);
                client.read(buf, buf, new CompletionHandler<Integer, ByteBuffer>() {
                    @Override
                    public void completed(Integer result, ByteBuffer buf) {
                        // 数据已读取完成，可以处理
                    }
                });
            }
        });
    }
}
```

## 三、模型对比

### 3.1 阻塞 vs 非阻塞

| 维度 | 阻塞 IO | 非阻塞 IO |
|------|---------|-----------|
| 等待数据 | 阻塞 | 立即返回 EAGAIN |
| CPU 消耗 | 低（等待时不消耗） | 高（轮询消耗 CPU） |
| 响应延迟 | 低 | 可能高（轮询间隔） |
| 实现复杂度 | 低 | 中 |

### 3.2 同步 IO vs 异步 IO

```
同步 IO：
- 应用主动等待 IO 完成
- recvfrom 在数据复制完成前不返回
- 包括：阻塞 IO、非阻塞 IO、IO 多路复用、信号驱动 IO

异步 IO：
- 内核完成 IO 操作后通知应用
- aio_read/aio_write 立即返回
- 数据完全就绪后才通知应用
```

### 3.3 IO 模型总对比

| 模型 | 等待数据 | 复制数据 | 阻塞 | 复杂度 | 性能 |
|------|----------|----------|------|--------|------|
| 阻塞 IO | 阻塞 | 同步复制 | 全程阻塞 | 低 | 差 |
| 非阻塞 IO | 轮询 | 同步复制 | 不阻塞 | 中 | 中 |
| IO 多路复用 | select 阻塞 | 同步复制 | select 阻塞 | 中 | 好 |
| 信号驱动 IO | 回调通知 | 同步复制 | 不阻塞 | 高 | 好 |
| 异步 IO | 不阻塞 | 内核完成 | 不阻塞 | 高 | 最好 |

## 四、IO 多路复用详解

### 4.1 为什么需要 IO 多路复用？

```
问题：每个连接一个线程，线程太多

解决方案：IO 多路复用
- 一个线程同时监控多个连接
- 有 IO 事件时再处理
- 大大减少线程数量
```

### 4.2 select/poll/epoll 对比

| 维度 | select | poll | epoll |
|------|--------|------|-------|
| 底层数据结构 | bitmap | 数组 | 红黑树 + 链表 |
| 最大 fd 数 | 1024 | 无限制 | 无限制 |
| fd 复制 | 每次都复制 | 每次都复制 | 只复制一次 |
| 时间复杂度 | O(n) | O(n) | O(1) |
| 工作模式 | LT | LT | LT + ET |
| 跨平台 | 所有平台 | 所有平台 | Linux only |

### 4.3 select 的问题

```c
// select 的问题
fd_set readfds;
FD_ZERO(&readfds);
FD_SET(fd1, &readfds);
FD_SET(fd2, &readfds);
FD_SET(fd3, &readfds);

// 问题一：fd 数量限制（通常 1024）
// 问题二：每次调用都要把 fd_set 从用户态复制到内核态
// 问题三：返回后要遍历所有 fd 找出就绪的（O(n)）
select(max_fd + 1, &readfds, NULL, NULL, timeout);

// 问题四：fd_set 不可重用，每次都要重建
```

### 4.4 epoll 的优势

```c
// epoll 的优势

// 1. 创建 epoll 实例
int epfd = epoll_create(1);

// 2. 注册要监控的 fd
struct epoll_event ev;
ev.events = EPOLLIN;  // 监控读事件
ev.data.fd = fd;
epoll_ctl(epfd, EPOLL_CTL_ADD, fd, &ev);

// 3. 等待事件
struct epoll_event events[1024];
int n = epoll_wait(epfd, events, 1024, timeout);

// 优势：
// - fd 数量无限制
// - fd 注册一次，不需要每次重建
// - 只返回已就绪的 fd（O(就绪数量)）
// - 支持边缘触发（ET）和水平触发（LT）
```

## 五、Java NIO

### 5.1 核心组件

```java
// Java NIO 三大组件

// 1. Channel（通道）
SocketChannel socketChannel = SocketChannel.open();
ServerSocketChannel serverChannel = ServerSocketChannel.open();

// 2. Buffer（缓冲区）
ByteBuffer buffer = ByteBuffer.allocate(1024);
buffer.put(data);
buffer.flip();
buffer.get();
buffer.clear();

// 3. Selector（选择器）
Selector selector = Selector.open();
channel.configureBlocking(false);
channel.register(selector, SelectionKey.OP_READ);
```

### 5.2 简单 Echo 服务器

```java
public class NIOEchoServer {
    public static void main(String[] args) throws IOException {
        Selector selector = Selector.open();
        ServerSocketChannel serverChannel = ServerSocketChannel.open();
        serverChannel.socket().bind(new InetSocketAddress(8080));
        serverChannel.configureBlocking(false);
        serverChannel.register(selector, SelectionKey.OP_ACCEPT);

        while (true) {
            selector.select();  // 阻塞直到有事件

            for (SelectionKey key : selector.selectedKeys()) {
                if (key.isAcceptable()) {
                    // 处理连接
                    ServerSocketChannel server = (ServerSocketChannel) key.channel();
                    SocketChannel client = server.accept();
                    client.configureBlocking(false);
                    client.register(selector, SelectionKey.OP_READ);

                } else if (key.isReadable()) {
                    // 处理读取
                    SocketChannel client = (SocketChannel) key.channel();
                    ByteBuffer buffer = ByteBuffer.allocate(1024);
                    int len = client.read(buffer);

                    if (len > 0) {
                        buffer.flip();
                        client.write(buffer);  // Echo
                        buffer.clear();
                    } else if (len == -1) {
                        client.close();
                    }
                }
            }
            selector.selectedKeys().clear();
        }
    }
}
```

## 六、面试题精讲

### 🔴 【高频】五种 IO 模型

**问题**：有哪些 IO 模型？它们有什么区别？

**标准答案**：

```
五种 IO 模型：

1. 阻塞 IO
   - 等待数据和数据复制都阻塞
   - 实现简单，但效率低

2. 非阻塞 IO
   - 等待数据时不阻塞，立即返回
   - 需要轮询检查数据是否就绪
   - CPU 消耗高

3. IO 多路复用
   - 使用 select/poll/epoll 监控多个 fd
   - 一个线程处理多个连接
   - 高并发服务器常用

4. 信号驱动 IO
   - 内核数据就绪时发送信号
   - 不阻塞，但编程复杂

5. 异步 IO
   - 内核完成等待和复制
   - 完成后再通知应用
   - 效率最高

核心区别在于等待数据和数据复制两个阶段的处理方式。
```

### 🔴 【高频】同步 IO 和异步 IO 的区别

**问题**：同步 IO 和异步 IO 有什么区别？

**标准答案**：

```
同步 IO：
- 应用主动等待 IO 完成
- recvfrom 在数据复制完成前不返回
- 数据复制阶段是同步的
- 包括：阻塞 IO、非阻塞 IO、IO 多路复用、信号驱动 IO

异步 IO：
- 内核完成 IO 操作（等待 + 复制）
- aio_read/aio_write 立即返回
- 数据完全就绪后通知应用
- 应用不需要参与 IO 过程

关键区别：
- 谁等待数据就绪
- 谁执行数据复制
- 谁通知 IO 完成
```

### 🟡 【中频】select/poll/epoll 的区别

**问题**：select、poll、epoll 有什么区别？

**标准答案**：

```
select：
- 使用 bitmap 存储 fd
- fd 数量限制（通常 1024）
- 每次调用都要重建 fd_set
- 时间复杂度 O(n)

poll：
- 使用数组存储 fd
- 没有 fd 数量限制
- 每次调用也要重建数组
- 时间复杂度 O(n)

epoll：
- 使用红黑树管理 fd
- fd 数量无限制
- fd 注册一次，跨调用保留
- 只返回已就绪的 fd，时间复杂度 O(就绪数量)
- 支持 LT（水平触发）和 ET（边缘触发）

性能：epoll > poll > select
```

## 七、常见陷阱与易错点

### ⚠️ 陷阱一：混淆同步和异步

同步和异步的区别不是「阻塞 vs 非阻塞」，而是「谁来等待和复制数据」。

### ⚠️ 陷阱二：忽略 epoll 的 ET 和 LT

| 触发模式 | 说明 |
|----------|------|
| LT（Level Trigger） | 条件满足就一直通知 |
| ET（Edge Trigger） | 状态变化才通知一次 |

### ⚠️ 陷阱三：认为 epoll 一定比 select 好

- epoll 只在 Linux 下可用
- 连接数少时 select 可能更简单
- 跨平台需求需要考虑

### ⚠️ 陷阱四：忘记处理边界情况

使用非阻塞 IO 时，要正确处理 EAGAIN/EWOULDBLOCK 错误码。

## 八、对比总结

### 五种 IO 模型对比

| 模型 | 等待数据 | 复制数据 | 性能 | 适用场景 |
|------|----------|----------|------|----------|
| 阻塞 IO | 阻塞 | 同步 | 差 | 简单场景 |
| 非阻塞 IO | 轮询 | 同步 | 中 | 低并发 |
| 多路复用 | select | 同步 | 好 | 高并发 |
| 信号驱动 | 信号 | 同步 | 好 | 特殊场景 |
| 异步 IO | 内核 | 内核 | 最好 | 高性能 |

### select/poll/epoll 对比

| 维度 | select | poll | epoll |
|------|--------|------|-------|
| 最大 fd | 1024 | 无限制 | 无限制 |
| fd 复制 | 每次复制 | 每次复制 | 只复制一次 |
| 时间复杂度 | O(n) | O(n) | O(1) |
| 触发模式 | LT | LT | LT + ET |
| 跨平台 | 所有平台 | 所有平台 | Linux |

## 九、扩展思考

### 💡 加分话题：边缘触发 vs 水平触发

```
水平触发（LT）：
- 条件满足时，每次调用 epoll_wait 都返回
- 简单，但可能多次通知

边缘触发（ET）：
- 状态变化时，只通知一次
- 高效，但需要一次性读完所有数据

ET 使用示例：
while (1) {
    n = read(fd, buf, size);
    if (n == EAGAIN) break;  // 必须处理完
}
```

### 💡 加分话题：Reactor 和 Proactor 模式

```
Reactor（同步 IO 多路复用）：
- 应用注册事件处理器
- 事件就绪时，dispatcher 通知应用
- 应用负责读取数据（同步）

Proactor（异步 IO）：
- 应用发起异步操作
- 内核完成操作后通知应用
- 数据已经可用（异步）

Linux 上的异步 IO：
- 真正的异步 IO 需要使用 io_uring
- epoll 是同步多路复用，不是异步
```

> IO 模型是高性能服务器设计的基础。理解各种 IO 模型的优缺点，才能在系统设计时做出正确的选择。对于大多数高并发场景，IO 多路复用（epoll）是最佳选择。
