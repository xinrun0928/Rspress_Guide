# Linux 性能分析：top、vmstat、iostat、netstat

服务器变慢了。CPU 100%？内存不够？还是 I/O 阻塞？

性能分析的第一步是找到瓶颈在哪里。本篇将带你了解 Linux 性能分析的方法论和工具。

## 性能分析的方法论

### USE 方法

每个系统资源都有三个指标：

| 指标 | 说明 | 问题 |
|---|---|---|
| **U**tilization（利用率） | 资源繁忙程度 | 利用率高 → 瓶颈 |
| **S**aturation（饱和度） | 等待队列长度 | 有排队 → 瓶颈 |
| **E**rrors（错误） | 错误计数 | 有错误 → 问题 |

```
资源类型 × 指标类型:
┌────────────────┬────────────────┬────────────────┐
│     CPU        │  利用率        │  饱和度        │
├────────────────┼────────────────┼────────────────┤
│     内存       │  利用率        │  换页/交换     │
├────────────────┼────────────────┼────────────────┤
│     磁盘       │  利用率        │  I/O 队列      │
├────────────────┼────────────────┼────────────────┤
│     网络       │  利用率        │  丢包/重传     │
└────────────────┴────────────────┴────────────────┘
```

### 性能分析步骤

```
1. 确认问题
   - 系统响应慢？吞吐量低？
   - 偶发还是持续？
   - 什么时候开始的？

2. 初步检查
   - top: CPU 和内存使用
   - vmstat 1: CPU 队列、内存、I/O
   - iostat -xz 1: 磁盘 I/O

3. 定位瓶颈
   - CPU 瓶颈 → 定位到进程/代码
   - 内存瓶颈 → 分析内存使用
   - I/O 瓶颈 → 分析磁盘/文件系统
   - 网络瓶颈 → 分析网络连接

4. 深入分析
   - strace: 系统调用跟踪
   - perf: CPU profiling
   - /proc/*: 详细统计
```

## CPU 分析

### top 分析

```bash
# CPU 利用率高（idle 低）
top

# 分析:
# - CPU(s): 5.2 us, 2.1 sy, 0.0 ni, 92.7 id, 0.0 wa
#   - us: 用户态 CPU
#   - sy: 系统态 CPU
#   - id: 空闲
#   - wa: I/O 等待

# 如果 %wa 很高 → I/O 瓶颈
# 如果 %us 很高 → 用户代码问题
# 如果 %sy 很高 → 系统调用过多
```

### vmstat 分析

```bash
# vmstat 1: 每秒采样
vmstat 1

# 输出:
# procs -----------memory---------- ---swap-- -----io---- -system-- ----cpu----
#  r  b   swpd   free   buff  cache   si   so    bi    bo   in   cs us sy id wa st
#  2  0      0 102400  12345 204800    0    0     0     0   50  100  5 10 30 55  0

# r: 运行队列长度（等待 CPU 的进程数）
#    - 如果 r > CPU 核心数 → 有进程在等待 CPU
# b: 阻塞的进程数（等待 I/O）
#    - 如果 b > 0 → 有进程在等待 I/O
# us + sy: CPU 使用率
# id: 空闲
# wa: I/O 等待时间
```

### 定位 CPU 热点

```bash
# perf: CPU 性能分析
perf top                          # 实时显示热点函数
perf record -g -p <pid>           # 记录某进程的调用
perf report                       # 查看记录

# strace: 系统调用跟踪
strace -c -p <pid>                # 统计系统调用
strace -T -p <pid>                # 显示调用时间
strace -e trace=read,write -p <pid>  # 只跟踪读写

# 示例：找出哪些系统调用最耗时
strace -c -f -p <pid> 2>&1 | sort -k6 -nr
```

### CPU 调度分析

```bash
# 查看进程调度信息
chrt -p <pid>                     # 查看调度策略
cat /proc/<pid>/sched            # 详细调度信息

# 实时进程列表
top 然后按 't' 切换 CPU 状态显示

# 查看 CPU 亲和性
taskset -p <pid>                  # 查看
taskset -c 0,1 -p <pid>          # 设置亲和性
```

## 内存分析

### 内存分析指标

```bash
# free 输出解读
free -h

#               total        used        free      shared  buff/cache   available
# Mem:          31Gi       8.5Gi       2.1Gi       456Mi       20Gi       22Gi
# Swap:         8Gi           0B         8Gi

# - total: 总内存
# - used: 已使用（包含 buff/cache）
# - free: 完全空闲
# - buff/cache: 缓冲区 + 页缓存
# - available: 可用于新进程的内存（最重要！）
#               = free + 可回收的 cache
```

### 内存泄漏检测

```bash
# 方法 1: 观察 free 内存变化
watch -n 1 free -h

# 方法 2: 查看进程的内存使用
ps aux --sort=-%mem | head

# 方法 3: 使用 smem
smem -s rss -r | head          # 按内存排序
smem -k -p | grep java         # 按进程查看

# 方法 4: /proc/<pid>/smaps
cat /proc/<pid>/smaps | grep -A 20 Private_Dirty

# 方法 5: valgrind (针对 C/C++)
valgrind --leak-check=full ./my_program
```

### OOM 分析

```bash
# 查看 OOM Killer 日志
dmesg | grep -i "out of memory"
dmesg | grep -i "killed process"

# 示例输出:
# [123456.789] Out of memory: Killed process 12345 (java) 
# total-vm:12345678kB, anon-rss:5678900kB, file-rss:1000kB

# 查看系统 OOM 设置
cat /proc/sys/vm/overcommit_memory
cat /proc/sys/vm/overcommit_ratio

# 查看进程 OOM 分数
cat /proc/<pid>/oom_score
```

### vmstat 内存分析

```bash
# vmstat 输出解读
# si: 从 swap 换入的内存 (KB/s)
# so: 换出到 swap 的内存 (KB/s)

# 如果 si/so 持续非零 → 内存不足，在使用 swap

# 频繁的 swap 会导致性能急剧下降（磁盘 I/O）
```

## I/O 分析

### iostat 分析

```bash
# 扩展模式，每秒更新
iostat -xz 1

# 输出:
# Device:  rrqm/s   wrqm/s     r/s     w/s    rkB/s    wkB/s  avgrq-sz avgqu-sz   await r_await w_await  svctm  %util
# sda        0.00     0.00   10.00    20.00   400.00   800.00     80.00     0.05    2.50    1.50    3.50    2.00   0.06

# 指标解读:
# - r/s, w/s: 每秒读写次数
# - rkB/s, wkB/s: 每秒读写 KB 数
# - avgrq-sz: 平均请求大小（扇区）
# - avgqu-sz: 平均 I/O 队列长度
#   - 如果 avgqu-sz > 1 → 有 I/O 等待
#   - 如果 avgqu-sz > 4 → 严重饱和
# - await: 平均 I/O 等待时间（毫秒）
#   - 如果 await > 100ms → I/O 慢
# - svctm: 平均服务时间（已废弃，仅供参考）
# - %util: 设备利用率
#   - 如果 %util > 80% → 设备饱和
```

### 定位 I/O 瓶颈

```bash
# 查看哪些进程在产生 I/O
iotop -o              # 只显示有 I/O 的进程
iotop -p <pid>        # 监控特定进程

# 查看进程 I/O 统计
cat /proc/<pid>/io
# 输出:
# rchar: 123456789    # 读取的字符数
# wchar: 987654321    # 写入的字符数
# syscr: 12345        # 读系统调用数
# syscw: 98765        # 写系统调用数
# read_bytes: 123456  # 实际磁盘读取
# write_bytes: 98765  # 实际磁盘写入

# 查看 I/O 调度器
cat /sys/block/sda/queue/scheduler
# noop [deadline] cfq  → deadline 是当前选择

# 查看文件系统挂载选项
mount | grep sda1
```

### 文件系统分析

```bash
# 查看文件系统的 I/O 统计
df -i                      # inode 使用
df -h                      # 空间使用

# du: 目录大小
du -sh /var/log            # 查看目录总大小
du -h --max-depth=1 /     # 各目录大小

# 查看大文件
find / -type f -size +100M -exec ls -lh {} \;

# 查看哪些文件被打开
lsof -p <pid>              # 进程打开的文件
lsof +D /path              # 目录下被打开的文件
```

## 网络分析

### 网络分析指标

```bash
# netstat 输出解读
netstat -s                   # 网络协议统计
netstat -i                   # 网络接口统计
netstat -ant | awk '/^tcp/ {++S[$NF]} END {for(k in S) print k, S[k]}'

# 关键指标:
# - RetransSegs: TCP 重传段数（高 → 网络问题）
# - OutSegs: 发送段数
# - EstabResets: 连接重置数（高 → 有连接被中断）
# - PassiveOpens: 被动打开数
# - ActiveOpens: 主动打开数
```

### 网络连接分析

```bash
# 查看连接状态分布
ss -s
# 输出:
# Total: 3456 (kernel 12345)
# TCP:   123 (estab 50, closed 60, orphaned 0, synrecv 0, timewait 10/0)

# 查看 TIME_WAIT 连接
ss -ant state time-wait | wc -l

# 查看连接队列溢出
netstat -s | grep -i "overflow\|listen"
# Listen40(Half-Empty)    ← 半连接队列溢出
# Listen queues have 0  ← 全连接队列溢出

# 查看丢包和错误
netstat -i
# Iface   MTU   RX-OK RX-ERR RX-DRP RX-OVR   TX-OK TX-ERR TX-DRP TX-OVR Flg
# eth0    1500 12345678      0      0      0  9876543      0      0      0 BMRU
```

### 网络延迟分析

```bash
# ping: 测试连通性和延迟
ping -c 10 192.168.1.1

# traceroute: 路由追踪
traceroute google.com

# mtr: 持续路由追踪
mtr google.com

# netcat: 端口连通性测试
nc -zv 192.168.1.1 80

# curl: 测试 HTTP 延迟
curl -w "@curl-format.txt" -o /dev/null -s http://example.com

# curl-format.txt 内容:
#     time_namelookup:  %{time_namelookup}\n
#     time_connect:     %{time_connect}\n
#     time_starttransfer: %{time_starttransfer}\n
#     time_total:       %{time_total}\n
```

## 综合性能分析

### sar：系统活动报告

```bash
# sar 是最全面的性能监控工具

# CPU
sar -u 1 10                   # 每秒 CPU 使用率，10 次
sar -q 1 10                   # 进程队列长度

# 内存
sar -r 1 10                   # 内存使用
sar -W 1 10                   # 交换活动

# I/O
sar -b 1 10                   # I/O 传输
sar -d 1 10                   # 磁盘活动

# 网络
sar -n DEV 1 10               # 网络设备统计
sar -n TCP,ETCP 1 10          # TCP 统计

# 保存历史数据（需要 sysstat 包）
# 数据保存在 /var/log/sa/
```

### 常见瓶颈分析

| 症状 | 指标 | 可能的瓶颈 |
|---|---|---|
| CPU 100%, %wa 低 | r > CPU 数 | CPU 瓶颈（进程忙） |
| CPU 100%, %wa 高 | r > CPU 数 | I/O 瓶颈（进程等待） |
| 内存使用高 | available 低 | 内存不足 |
| si/so 非零 | swap 活动 | 内存不足，使用 swap |
| avgqu-sz 高 | 队列长 | 磁盘 I/O 瓶颈 |
| %util > 80% | 设备忙 | 磁盘饱和 |
| 重传率高 | RetransSegs | 网络问题 |
| TIME_WAIT 多 | 连接数 | 短连接过多 |

### 性能分析清单

```bash
# 1. CPU 瓶颈
top; vmstat 1

# 2. 内存瓶颈
free -h; vmstat 1; cat /proc/meminfo

# 3. I/O 瓶颈
iostat -xz 1; iotop -o

# 4. 网络瓶颈
netstat -s; ss -s; sar -n DEV 1 5

# 5. 综合分析
sar -A 1 5                    # 所有统计
```

## 面试高频问题

**Q1：如何定位 CPU 瓶颈？**

先用 `top` 和 `vmstat` 确认 CPU 使用情况。如果 `%us` 高，用 `perf top` 定位热点函数。如果 `%sy` 高，用 `strace -c` 分析系统调用。

**Q2：内存不足时会发生什么？**

Linux 会使用 swap。如果 swap 也用完，内核的 OOM Killer 会选择一个进程杀死。可以通过 `dmesg | grep -i oom` 查看 OOM 日志。

**Q3：如何判断是 CPU 还是 I/O 瓶颈？**

观察 `vmstat` 的 `wa` 列（I/O 等待）。如果 `wa` 高而 CPU idle 低，说明进程在等待 I/O，是 I/O 瓶颈。如果 `wa` 低而 CPU idle 低，说明 CPU 被占满，是 CPU 瓶颈。

**Q4：如何分析网络性能？**

用 `netstat -s` 查看协议统计，`ss -s` 查看连接状态。关注重传、丢包、连接重置等指标。用 `sar -n DEV` 查看网络设备统计。

**Q5：什么是 swap？为什么 swap 会影响性能？**

swap 是磁盘上的交换空间。当物理内存不足时，不活跃的页面会被换出到 swap。磁盘 I/O 比内存慢 1000 倍以上，所以大量使用 swap 会导致性能急剧下降。

掌握这些性能分析技能，是在生产环境中排查问题的必备能力。
