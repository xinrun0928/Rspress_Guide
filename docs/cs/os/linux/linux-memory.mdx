# Linux 内存管理：伙伴系统 + slab 分配器

`free -h` 显示你的服务器有 32GB 内存，使用了 28GB。但 `top` 显示大多数进程只用了很少内存。

这就是 Linux 内存管理的魔力：**伙伴系统**管理大的物理页框，**slab 分配器**管理小的内核对象。

## 物理内存管理：伙伴系统

### 伙伴系统的原理

伙伴系统把物理内存分成大小为 2^k 的块。当需要分配时，找到合适大小的块；如果没有，就把大块分裂成两个「伙伴」，一个用于分配，一个备用。

```c
// 伙伴系统的空闲块链表
// 每个 order 有自己的空闲链表
struct free_area {
    struct list_head free_list[MIGRATE_TYPES];
    unsigned long nr_free;
};

struct zone {
    // 每个 order (0-11) 对应一个 free_area
    struct free_area free_area[MAX_ORDER];
    // ...
};

// MAX_ORDER 通常是 11 或更高
// 意味着最大块是 2^11 * PAGE_SIZE = 4MB (对于 4KB 页)
```

### 伙伴系统的分配/释放

```c
// 分配页
struct page* alloc_pages(gfp_t gfp_mask, unsigned int order) {
    struct zone* zone;
    struct free_area* area;
    struct page* page;
    
    // 1. 选择合适的 zone
    zone = gfp_zone(gfp_mask);
    
    // 2. 从当前 order 开始，找空闲块
    for (current_order = order; current_order < MAX_ORDER; current_order++) {
        area = &zone->free_area[current_order];
        if (!list_empty(&area->free_list[current_order])) {
            // 找到空闲块
            page = list_entry(area->free_list[current_order].next,
                            struct page, lru);
            list_del(&page->lru);
            area->nr_free--;
            
            // 3. 如果分配的块太大，需要分裂
            while (current_order > order) {
                current_order--;
                // 分裂成两个伙伴
                buddy = page + (1 << current_order);
                // 设置伙伴信息
                set_page_private(buddy, current_order);
                // 把伙伴加入下一级链表
                list_add(&buddy->lru,
                        &zone->free_area[current_order].free_list);
            }
            
            set_page_private(page, order);
            prep_new_page(page, order);
            return page;
        }
    }
    
    // 4. 分配失败，尝试回收内存
    return NULL;
}

// 释放页
void __free_pages(struct page* page, unsigned int order) {
    unsigned long page_idx;
    struct page* buddy;
    
    page_idx = page_to_pfn(page);
    
    while (order < MAX_ORDER) {
        // 找到伙伴
        buddy = __get_buddy(page, order);
        
        // 检查伙伴是否真的是空闲的
        if (!page_is_buddy(buddy, order))
            break;
        
        // 合并
        list_del(&buddy->lru);
        zone->free_area[order].nr_free--;
        
        // 准备合并到上一级
        page = min(page, buddy);
        order++;
    }
    
    // 加入空闲链表
    set_page_private(page, order);
    list_add(&page->lru, &zone->free_area[order].free_list);
    zone->free_area[order].nr_free++;
}
```

### 伙伴系统的优点

| 优点 | 说明 |
|---|---|
| 分配简单 | O(log n) 时间复杂度 |
| 无外部碎片 | 只分割或合并，不会有分散的小块 |
| 合并容易 | 伙伴关系明确，容易合并 |
| 对高速缓存友好 | 分配的块是连续的 |

## slab 分配器

### 为什么需要 slab？

伙伴系统分配的最小单位是页（4KB），但内核需要分配很多小对象（几十到几百字节）。

```c
// 内核需要分配的对象
struct task_struct;    // ~2KB
struct dentry;        // ~200 字节
struct inode;         // ~500 字节
struct sk_buff;       // ~256 字节
struct file;          // ~200 字节
// ...
```

直接用伙伴系统分配会导致严重的内部碎片。

### slab 的设计

slab 把页分成多个相同大小的对象：

```
slab 示意图:
┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│  物理页 (4KB)                                                    │
│  ┌───────────────────────────────────────────────────────────┐ │
│  │ ┌─────┬─────┬─────┬─────┬─────┬─────┬─────┬─────┬ ...   │ │
│  │ │ obj │ obj │ obj │ obj │ obj │ obj │ obj │ obj │       │ │
│  │ │  1  │  2  │  3  │  4  │  5  │  6  │  7  │  8  │       │ │
│  │ └─────┴─────┴─────┴─────┴─────┴─────┴─────┴─────┴ ...   │ │
│  └───────────────────────────────────────────────────────────┘ │
│                                                                 │
│  slab 描述符 (管理一个或多个物理页)                              │
│  ┌───────────────────────────────────────────────────────────┐ │
│  │  - 空闲对象链表 (free)                                      │ │
│  │  - 已分配对象链表 (inuse)                                   │ │
│  │  - 对象大小 (size)                                          │ │
│  │  - 对象数量 (num)                                           │ │
│  │  - 构造函数/析构函数指针                                    │ │
```

### slab 缓存结构

```c
// slab 缓存描述符
struct kmem_cache {
    // 对象管理
    unsigned int object_size;    // 对象大小
    unsigned int size;            // 实际分配大小（含元数据）
    unsigned int align;          // 对齐要求
    
    // 链表
    struct list_head slabs_full;  // 已满的 slab
    struct list_head slabs_partial; // 部分空闲的 slab
    struct list_head slabs_free;   // 完全空闲的 slab
    
    // 对象分配
    void* (*ctor)(void*);        // 构造函数
    void (*dtor)(void*);         // 析构函数
    
    // slab 管理
    unsigned int num;            // 每个 slab 的对象数
    unsigned int gfporder;       // 每个 slab 的页数
    
    char name[CACHE_NAME_LEN];   // 缓存名称
    struct list_head list;        // 全局缓存链表
};

// slab 描述符（per-slab）
struct slab {
    struct list_head list;        // 在 slab 链表中的位置
    unsigned long colouroff;     // 颜色偏移
    void* s_mem;                // 第一个对象的地址
    unsigned int inuse;          // 已分配对象数
    void* freelist;             // 空闲对象链表头
};
```

### kmalloc：slab 的用户接口

```c
// kmalloc: 分配指定大小的内存
// 基于 slab 分配器

void* kmalloc(size_t size, gfp_t flags) {
    struct kmem_cache* cachep;
    
    // 1. 根据大小选择合适的缓存
    cachep = malloc_sizes[size_to_index(size)].cachep;
    
    // 2. 从缓存分配对象
    return kmem_cache_alloc(cachep, flags);
}

// 预定义的 kmalloc 缓存
kmalloc_caches[16] = {
    // 16, 32, 64, 128, 256, 512, 1024, 2048, ... (2 的幂次)
    // 每种大小有独立的 slab 缓存
};
```

## 内存映射

### vm_area_struct

进程虚拟地址空间由 vm_area_struct 描述：

```c
struct vm_area_struct {
    unsigned long vm_start;      // 起始地址
    unsigned long vm_end;       // 结束地址
    unsigned long vm_flags;     // 标志 (读/写/执行/私有/共享)
    
    struct file* vm_file;       // 映射的文件（如果是文件映射）
    struct anon_vma* anon_vma;  // 匿名映射（如果是匿名）
    
    struct vm_operations_struct* vm_ops;  // 操作函数
    
    unsigned long vm_pgoff;      // 文件偏移（如果是文件映射）
    struct mm_struct* vm_mm;     // 所属进程
    
    struct list_head anon_vma_chain; // 匿名映射链表
};
```

### mmap 系统调用

```c
// mmap: 创建内存映射
void* mmap(void* addr, size_t length, int prot, int flags,
           int fd, off_t offset) {
    // 参数:
    // addr: 建议地址（通常 NULL，让内核选择）
    // length: 映射长度
    // prot: PROT_READ | PROT_WRITE | PROT_EXEC
    // flags: MAP_PRIVATE | MAP_SHARED | MAP_ANONYMOUS
    // fd: 文件描述符（匿名映射时为 -1）
    // offset: 文件偏移
    
    return syscall(__NR_mmap, addr, length, prot, flags, fd, offset);
}

// 使用示例：匿名映射（分配内存）
void* buf = mmap(NULL, 4096, PROT_READ | PROT_WRITE,
                 MAP_PRIVATE | MAP_ANONYMOUS, -1, 0);

// 使用示例：文件映射（读取文件）
int fd = open("file.txt", O_RDONLY);
void* buf = mmap(NULL, file_size, PROT_READ,
                 MAP_PRIVATE, fd, 0);
```

## OOM Killer

### 为什么需要 OOM Killer？

当物理内存和交换空间都耗尽时，内核需要选择一个进程杀掉来释放内存。

```bash
# 查看 OOM Killer 相关信息
dmesg | grep -i "out of memory"
dmesg | grep -i "killed process"

# 示例输出:
# [12345.678901] Out of memory: Killed process 12345 (java) total-vm:12345678kB, anon-rss: 5678900kB, file-rss: 1000kB
```

### OOM 分数

Linux 根据 oom_score 决定杀哪个进程：

```bash
# 查看进程 OOM 分数
cat /proc/<pid>/oom_score

# 调整 OOM 分数（降低被杀的概率）
echo -1000 > /proc/<pid>/oom_score_adj  # 几乎不会被杀
echo 500 > /proc/<pid>/oom_score_adj     # 更容易被杀死

# 禁用 OOM Killer（不推荐）
sysctl -w vm.overcommit_memory=2
```

### OOM Killer 的选择策略

```c
// 选择进程的评分标准
// score = 已用内存 + oom_score_adj

// 内核倾向杀死:
// 1. 消耗内存最多的进程
// 2. 进程优先级低的（通过 oom_score_adj 调整）
// 3. 非关键系统进程

// 尽量不杀死:
// 1. init/systemd
// 2. 有特殊 oom_score_adj 的进程
// 3. 内核线程
```

## 面试高频问题

**Q1：什么是伙伴系统？**

伙伴系统把物理内存分成大小为 2^k 的块。分配时找合适大小的块，如果太大就分裂成两个伙伴。释放时如果伙伴也空闲，就合并成更大的块。优点是无外部碎片，分配和释放都是 O(log n)。

**Q2：slab 分配器解决了什么问题？**

slab 在伙伴系统之上，为小对象提供更细粒度的分配。每个 slab 缓存管理一种对象大小，减少了内部碎片。同时，slab 缓存可以着色（cache coloring）来提高 CPU cache 命中率。

**Q3：什么是写时复制（COW）？**

fork 时父子进程共享相同的物理页（read-only）。只有当其中一方尝试写入时，才复制物理页。这样 fork 的开销大大减少。

**Q4：什么是 OOM Killer？**

当物理内存和交换空间都耗尽时，内核选择并杀死一个进程来释放内存。Linux 根据进程的内存消耗和 oom_score_adj 选择牺牲者。

**Q5：vmalloc 和 kmalloc 的区别？**

kmalloc 从伙伴系统分配连续的物理内存，适用于需要物理连续的内存场景（如 DMA）。vmalloc 分配虚拟连续的内存，但物理内存不连续，适用于大块内存分配。

理解 Linux 内存管理，是理解操作系统内存管理的最佳实践。
