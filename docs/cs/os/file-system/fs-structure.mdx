# 文件系统的结构：FCB 与索引节点

当你创建一个文件时，操作系统到底做了什么？

答案是：**在磁盘上分配一个 inode，在目录中添加一条记录**。

inode 是文件系统的核心数据结构，它记录了文件的所有元信息：大小、权限、时间、存放位置。理解 inode，就理解了文件系统的半壁江山。

## 文件系统的整体结构

### UNIX 文件系统的经典布局

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                            磁盘分区                                        │
├─────────────────────────────────────────────────────────────────────────────┤
│  Boot Block  │  Super Block  │  Inode Bitmap  │  Data Bitmap  │  Inodes  │  Data Blocks │
│  (启动块)     │  (超级块)      │  (inode 位图)  │ (数据块位图)  │  (inode表)│  (数据块)    │
│  1KB         │  1KB          │  可变          │  可变         │  可变     │  可变        │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  Boot Block: 启动时由 BIOS 读取，包含启动加载器                              │
│  Super Block: 文件系统的元信息（总块数、空闲块数、inode 数等）               │
│  Inode Bitmap: 哪些 inode 被使用（1=已用，0=空闲）                            │
│  Data Bitmap: 哪些数据块被使用（1=已用，0=空闲）                              │
│  Inodes: 文件的元数据（权限、时间、块位置）                                    │
│  Data Blocks: 文件的实际数据                                                 │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 各部分详解

**1. Boot Block（启动块）**

位于文件系统最开始，由 BIOS 在启动时读取。如果这个分区是启动分区，里面就包含启动加载器。

```c
// 启动块内容
struct boot_block {
    unsigned char bootstrap[446];  // 启动代码
    unsigned int partition_table[4][16];  // 分区表
    unsigned short signature;  // 0x55AA，标识可启动
};
```

**2. Super Block（超级块）**

记录文件系统的全局信息。

```c
struct super_block {
    unsigned int total_blocks;      // 总块数
    unsigned int free_blocks;       // 空闲块数
    unsigned int free_inodes;       // 空闲 inode 数
    unsigned int first_data_block;   // 第一个数据块
    unsigned int block_size;        // 块大小（字节）
    unsigned int inodes_per_group;  // 每组 inode 数
    unsigned int magic;              // 文件系统魔数（标识类型）
    unsigned int last_mount_time;   // 最后挂载时间
    unsigned int last_write_time;   // 最后写入时间
};
```

**3. Inode Bitmap（inode 位图）**

记录哪些 inode 被使用了。位图的好处是查找空闲 inode 很快（只要找第一个 0）。

```
inode 位图:
┌───┬───┬───┬───┬───┬───┬───┬───┬───┬───┐
│ 1 │ 1 │ 1 │ 0 │ 0 │ 1 │ 1 │ 1 │ 1 │ 1 │
└───┴───┴───┴───┴───┴───┴───┴───┴───┴───┘
  ↑   ↑   ↑   ↑   ↑   ↑   ↑   ↑   ↑   ↑
 已被占用的 inode（值为 1），第 4、5 个 inode 空闲（值为 0）
```

**4. Data Bitmap（数据块位图）**

记录哪些数据块被使用了。

```
数据块位图:
┌───┬───┬───┬───┬───┬───┬───┬───┬───┬───┐
│ 1 │ 1 │ 1 │ 1 │ 0 │ 0 │ 1 │ 1 │ 0 │ 1 │
└───┴───┴───┴───┴───┴───┴───┴───┴───┴───┘
  ↑   ↑   ↑   ↑   ↑   ↑   ↑   ↑   ↑   ↑
  块 0-3 已用，块 4-5 空闲，块 6-7 已用，块 8 空闲，块 9 已用
```

## FCB 与 inode

### FCB（File Control Block）

FCB 是早期文件系统使用的数据结构，每个文件一个 FCB，包含文件的元信息。

```c
// 早期 FCB 结构
struct fcb {
    char filename[11];           // 8.3 格式文件名
    unsigned char attributes;     // 文件属性
    unsigned short create_time;   // 创建时间
    unsigned short create_date;   // 创建日期
    unsigned short last_access;   // 最后访问日期
    unsigned short first_block_high;  // 起始块号（高位）
    unsigned short first_block_low;   // 起始块号（低位）
    unsigned long file_size;     // 文件大小
};
```

**问题：** FCB 包含的元信息有限，不支持现代需求（如长文件名、权限、软硬链接等）。

### inode（索引节点）

inode 是 FCB 的现代进化版，Unix 风格的文件系统广泛使用。

```c
// Unix inode 结构
struct inode {
    unsigned short mode;          // 文件类型和权限
    unsigned short uid;           // 文件所有者 ID
    unsigned long size;           // 文件大小（字节）
    unsigned long atime;          // 最后访问时间
    unsigned long ctime;          // 最后修改时间（元数据）
    unsigned long mtime;          // 内容修改时间
    unsigned long dtime;          // 删除时间
    unsigned short gid;           // 所有者组 ID
    unsigned short links_count;   // 硬链接计数
    unsigned long blocks;         // 占用块数（512 字节块）
    unsigned long flags;          // 文件标志
    unsigned long osd1;           // 操作系统特定字段
    unsigned long block[15];      // 数据块指针数组
    unsigned long generation;      // 文件版本（NFS）
    unsigned long file_acl;       // 文件 ACL
    unsigned long dir_acl;        // 目录 ACL
    unsigned long frag_addr;      // 片段地址
};
```

### inode vs FCB

| 特性 | FCB | inode |
|---|---|---|
| **设计时代** | 早期（FAT） | 现代（ext4、NTFS） |
| **元信息** | 有限 | 丰富 |
| **存储位置** | 目录中 | 独立的 inode 表 |
| **权限控制** | 简单 | 复杂（Unix 权限） |
| **长文件名** | 不支持 | 支持 |
| **符号链接** | 不支持 | 支持 |
| **访问控制** | 粗粒度 | 细粒度 |

## inode 的数据块组织

inode 中最重要的字段是 `block[15]`，它是如何指向数据块的？

### 直接块（Direct Blocks）

前 12 个指针直接指向数据块。

```
block[0-11] → 直接块
block[0] ──► 数据块 0 ──► 4KB
block[1] ──► 数据块 1 ──► 4KB
...
block[11] ─► 数据块 11 ──► 4KB
```

**直接块能表示的文件大小：**
- 12 块 × 4KB = 48KB

### 间接块（Indirect Blocks）

第 13 个指针指向一个块，这个块中存储的是数据块指针。

```
block[12] → 单间接块（1K 个指针）
           │
           ▼
           ┌────┬────┬────┬────┬────┐
           │ DB │ DB │ DB │ DB │... │
           └────┴────┴────┴────┴────┘
               └── 每个 DB 是一个数据块

单间接块能表示的额外大小：
- 1024 / 4 = 256 个块
- 256 × 4KB = 1MB
```

### 双间接块（Double Indirect Blocks）

第 14 个指针指向一个块，这个块中存储的是间接块指针。

```
block[13] → 双间接块
           │
           ▼
           ┌────┬────┬────┬────┐
           │ IB │ IB │ IB │... │
           └────┴────┴────┴────┘
               │
               ▼
               ┌────┬────┬────┬────┐
               │ DB │ DB │ DB │... │
               └────┴────┴────┴────┘
双间接块能表示的额外大小：
- 256 个间接块 × 每个间接块 256 个数据块 = 65536 个块
- 65536 × 4KB = 256MB
```

### 三间接块（Triple Indirect Blocks）

第 15 个指针指向一个块，这个块中存储的是双间接块指针。

```
block[14] → 三间接块 → 双间接块 → 单间接块 → 数据块
三间接块能表示的额外大小：
- 256 × 65536 × 4KB = 64GB
```

### ext4 inode 的总容量

```
直接块:        12 × 4KB = 48KB
单间接块:       1 × 256 × 4KB = 1MB
双间接块:       1 × 65536 × 4KB = 256MB
三间接块:       1 × 256 × 65536 × 4KB = 64GB
────────────────────────────────────
理论最大文件:                       ≈ 64.26GB
```

## 目录的实现

### 目录也是文件

在 Unix 中，目录本质上也是一个文件，只是它的内容是目录项的列表。

```c
// 目录项结构
struct dir_entry {
    unsigned long inode;      // inode 号（0 表示空闲）
    unsigned short rec_len;   // 目录项长度
    unsigned char name_len;  // 文件名长度
    unsigned char file_type;  // 文件类型
    char name[256];          // 文件名（可变长）
};
```

### 目录的线性列表实现

最简单的目录实现是把所有目录项存在一个线性列表中。

```c
// 读取目录内容
struct dir_entry entries[100];
read(file_fd, entries, sizeof(entries));

// 查找文件
for (int i = 0; i < entry_count; i++) {
    if (strcmp(entries[i].name, "target.txt") == 0) {
        return entries[i].inode;
    }
}
```

**优点：** 实现简单
**缺点：** 查找慢，O(n) 复杂度

### 目录的哈希表实现

用哈希表组织目录项，查找效率提高到 O(1)。

```c
struct dir_hash_entry {
    unsigned long inode;
    char name[256];
    struct dir_hash_entry* next;  // 哈希冲突链
};
struct dir_hash_table {
    struct dir_hash_entry* buckets[1024];
};
unsigned int hash(const char* name) {
    unsigned int h = 0;
    while (*name) {
        h = (h << 5) + *name++;
    }
    return h % 1024;
}
```

**优点：** 查找快
**缺点：** 实现复杂，需要额外空间

### NTFS 的 B+ 树实现

NTFS 用 B+ 树组织目录，支持高效的插入、删除和查找。

```
B+ 树目录结构:
              ┌───────┐
              │  Root  │
              └───┬───┘
            ┌─────┴─────┐
         ┌──┴──┐     ┌──┴──┐
         │     │     │     │
        ┌┴┐   ┌┴┐   ┌┴┐   ┌┴┐
        │A│   │M│   │S│   │Z│  ← 按文件名排序
        └┬┘   └┬┘   └┬┘   └┬┘
         │     │     │     │
       ┌──┴──┐ ┌──┴──┐ ┌──┴──┐ ┌──┴──┐
       │ A*  │ │ M*  │ │ S*  │ │ Z*  │
       │ ... │ │ ... │ │ ... │ │ ... │
       └─────┘ └─────┘ └─────┘ └─────┘
```

**优点：** 查找、插入、删除都是 O(log n)
**缺点：** 实现复杂

## 硬链接与软链接

### 硬链接

硬链接是多个目录项指向同一个 inode。

```bash
$ ln file1.txt file2.txt  # 创建硬链接
$ ls -li file*.txt
262144 -rw-r--r-- 2 user staff 1024 Mar 29 10:00 file1.txt
262144 -rw-r--r-- 2 user staff 1024 Mar 29 10:00 file2.txt
# 两个文件的 inode 号相同（262144），链接计数为 2
```

**特点：**
- 多个文件共享同一个 inode
- 不能跨文件系统
- 不能链接目录（防止循环引用）
- 删除一个链接，只是减少链接计数

### 软链接（符号链接）

软链接是一个特殊文件，内容是另一个文件的路径。

```bash
$ ln -s file1.txt symlink.txt
$ ls -li file*.txt symlink.txt
262144 -rw-r--r-- 2 user staff 1024 Mar 29 10:00 file1.txt
262145 lrwxr-xr-x 1 user staff    9 Mar 29 10:00 symlink.txt -> file1.txt
# symlink 是一个新文件，inode 不同，内容是 "file1.txt"
```

**特点：**
- 是一个独立文件，有自己的 inode
- 可以跨文件系统
- 可以链接目录
- 删除原文件，软链接变成「断链」

## 面试高频问题

**Q1：inode 包含哪些信息？**

inode 包含：文件类型和权限、所有者 ID、文件大小、时间戳（创建、修改、访问）、链接计数、数据块指针等。

**Q2：为什么 ext4 的最大文件大小是 16TB？**

ext4 使用 48 位的块号，每个块 4KB，最大文件大小 = 2^48 × 4KB ≈ 16TB（实际受 inode 结构限制）。

**Q3：硬链接和软链接有什么区别？**

硬链接是多个目录项指向同一个 inode，不能跨文件系统，不能链接目录。软链接是独立文件，内容是路径，可以跨文件系统，可以链接目录。

**Q4：删除文件时会发生什么？**

首先删除目录中的目录项，然后减少 inode 的链接计数。如果链接计数变为 0，且没有进程打开该文件，则释放 inode 和数据块。如果有进程打开，文件会标记为「待删除」，等最后一个进程关闭后才真正释放。

**Q5：目录项为什么可变长？**

为了支持长文件名。可变长的目录项用 `name_len` 记录文件名长度，用 `rec_len` 记录目录项总长度，便于遍历。

理解 FCB 和 inode 的结构，是理解文件系统的核心。
