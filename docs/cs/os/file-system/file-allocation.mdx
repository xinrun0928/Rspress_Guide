# 文件分配方式：连续分配、链接分配、索引分配

文件的数据是如何存放在磁盘上的？

这看起来简单，但背后的设计却充满了权衡：如何让文件读写更快？如何减少空间浪费？如何支持大文件？

三种基本分配方式给出了不同的答案。

## 为什么需要文件分配？

磁盘是块设备，数据以块为单位读写。文件系统需要决定：
- 文件的 n 个字节需要哪些块来存储？
- 这些块在磁盘上如何组织？

```
磁盘:
┌────┬────┬────┬────┬────┬────┬────┬────┬────┬────┐
│ B0 │ B1 │ B2 │ B3 │ B4 │ B5 │ B6 │ B7 │ B8 │ B9 │
└────┴────┴────┴────┴────┴────┴────┴────┴────┴────┘

文件 A 需要 3 个块，文件 B 需要 2 个块：
- 连续分配: A 在 B0-B2，B 在 B3-B4
- 链接分配: A 在 B0,B5,B7，B 在 B1,B6（链表串起来）
- 索引分配: A 在 B0,B2,B4，索引块记录位置
```

## 连续分配

### 原理

文件占据一组连续的磁盘块。就像租房子：一次性租下连续的几间。

```
连续分配示例:
┌─────────────────────────────────────────────────────────────┐
│  Block 0 │ Block 1 │ Block 2 │ Block 3 │ Block 4 │ Block 5 │
│    File A start                                      │         │
│    (File A occupies 0-2)                              │         │
│─────────────────────────────────────────────────────│ File B  │
│    File A continues                                  │ (3-4)   │
└─────────────────────────────────────────────────────────────┘

文件 A 的信息:
起始块: 0
长度: 3 块
```

### 实现

只需要记录**起始块号**和**块数**。

```c
struct contiguous_file {
    unsigned int start_block;  // 起始块号
    unsigned int num_blocks;   // 块数
};
```

### 优点

| 优点 | 说明 |
|---|---|
| **读写速度快** | 顺序访问时，磁头移动少 |
| **实现简单** | 只需记录起始块和长度 |
| **适合顺序访问** | 连续读写不需要寻道 |

### 缺点

| 缺点 | 说明 |
|---|---|
| **外部碎片** | 文件释放后留下小空洞，难以分配大文件 |
| **文件不能动态增长** | 如果后面被占用，无法扩展 |
| **需要预知大小** | 创建时需要指定文件大小 |

### 外部碎片问题

```
初始状态:
┌─────┬─────┬─────┬─────┬─────┬─────┬─────┐
│  A  │  B  │  C  │  D  │  E  │  F  │  G  │
└─────┴─────┴─────┴─────┴─────┴─────┴─────┘

删除 B 和 D 后:
┌─────┬─────┬─────┬─────┬─────┬─────┬─────┐
│  A  │空闲 │  C  │空闲 │  E  │  F  │  G  │
└─────┴─────┴─────┴─────┴─────┴─────┴─────┘
↑   ↑           ↑           ↑
三个 1 块大小的空洞，加起来 3 块

尝试分配 4 块的文件：失败！（没有连续的 4 块空间）
```

### 变种：扩展连续分配

为了解决动态增长问题，可以预先分配一个较大的extent，减少碎片。

```c
struct extent {
    unsigned int start_block;
    unsigned int length;        // 当前 extent 长度
    unsigned int max_length;   // 最大可扩展长度
};
```

## 链接分配

### 原理

每个块存储一个指向下一个块的指针，形成链表。

```
链接分配示例:
┌─────┬─────┬─────┬─────┬─────┬─────┬─────┐
│ B0  │ B1  │ B2  │ B3  │ B4  │ B5  │ B6  │
│ A:0 │ A:2 │ C:3 │ A:5 │ B:  │ C:  │ A:  │
│ →2  │ →5  │ →   │ →   │ →   │ →   │ →   │
└─────┴─────┴─────┴─────┴─────┴─────┴─────┘
  │
  └──► Block 0 ──► Block 2 ──► Block 5 ──► NULL
       (文件 A 的块链: 0 → 2 → 5)

文件 A: 块 0 → 块 2 → 块 5
文件 B: 块 4 → 块 6 → ...
文件 C: 块 1 → 块 3
```

### 实现

```c
// 磁盘块结构（FAT 文件系统风格）
struct disk_block {
    unsigned char data[512 - 4];  // 数据区
    unsigned int next_block;       // 指向下一个块（0 表示文件结束，-1 表示空闲）
};

// 文件目录项
struct file_entry {
    char filename[11];
    unsigned int first_block;  // 文件的第一个块
    unsigned int file_size;
};
```

### FAT 文件系统

FAT（File Allocation Table）是经典的链接分配实现。

```
FAT 表:
┌────┬────┬────┬────┬────┬────┬────┐
│ EOC│ 2  │ EOC│ 5  │ EOC│ EOC│ EOF│
└────┴────┴────┴────┴────┴────┴────┘
  │    │    │    │    │    │    │
  ▼    ▼    ▼    ▼    ▼    ▼    ▼
 Block0 Block1 Block2 Block3 Block4 Block5 Block6

文件 A: 起始块 = 0 → 块 0 → 块 2 → EOC
文件 C: 起始块 = 1 → 块 1 → 块 3 → EOC
文件 B: 起始块 = 4 → 块 4 → EOC

EOC (End Of Cluster): 文件结束
EOF (End Of File): FAT 结束/空闲标记
```

### 优点

| 优点 | 说明 |
|---|---|
| **无外部碎片** | 块可以散布在磁盘各处 |
| **文件可动态增长** | 在链尾追加新块即可 |
| **实现简单** | 只需记录起始块 |

### 缺点

| 缺点 | 说明 |
|---|---|
| **随机访问慢** | 必须从链头开始遍历 |
| **块中存储指针** | 浪费存储空间 |
| **可靠性问题** | 指针损坏会导致文件丢失 |
| **不能高效利用磁盘顺序访问** | 磁头需要在磁盘上跳跃 |

## 索引分配

### 原理

为每个文件创建一个索引块，存储该文件所有数据块的位置。

```
索引分配示例:
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  索引块 (Index Block)                                       │
│  ┌────┬────┬────┬────┬────┬────┬────┬────┐                │
│  │  0 │  5 │  2 │  8 │  │  │  │  │                      │
│  └────┴────┴────┴────┴────┴────┴────┴────┘                │
│    │    │    │    │                                   │
│    ▼    ▼    ▼    ▼                                   │
│  Block0 Block5 Block2 Block8 (文件 A 的数据块)             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 实现

```c
// 索引块结构
struct index_block {
    unsigned int data_blocks[256];  // 假设块大小 4KB，指针 4 字节
};

// 文件目录项
struct file_entry {
    char filename[255];
    unsigned int index_block;  // 索引块号
    unsigned int file_size;
};
```

### 多级索引

当文件很大时，一个索引块不够用，需要多级索引。

```c
// Unix inode 中的索引结构
struct inode {
    unsigned long size;
    unsigned long block[15];
    
    // block[0-11]: 12 个直接块
    // block[12]: 单间接块（一级索引）
    // block[13]: 双间接块（二级索引）
    // block[14]: 三间接块（三级索引）
};
```

```
多级索引示意:

直接块: 12 个 → 12 × 4KB = 48KB

单间接块 (block[12]):
┌──────┬──────┬──────┬──────┐
│ DB0  │ DB1  │ DB2  │ ...  │   ← 256 个数据块指针
└──────┴──────┴──────┴──────┘
→ 256 × 4KB = 1MB

双间接块 (block[13]):
block[13] ──► 双间接索引块
              ├──► 单间接块 0 → 数据块 0-255
              ├──► 单间接块 1 → 数据块 256-511
              └──► ...
→ 256 × 256 × 4KB = 256MB

三间接块 (block[14]):
→ 256 × 256 × 256 × 4KB = 64GB
```

### 优点

| 优点 | 说明 |
|---|---|
| **支持随机访问** | 直接通过索引块定位块 |
| **无外部碎片** | 块可分散在磁盘各处 |
| **可靠性高** | 索引块损坏只影响一个文件 |

### 缺点

| 缺点 | 说明 |
|---|---|
| **索引块开销** | 小文件也需要一个索引块 |
| **大文件访问需多次 I/O** | 多级索引需要多读几块 |

### 混合索引分配

ext4 采用改进的方案：extent（范围）代替块指针。

```c
struct ext4_extent {
    unsigned short ee_block;   // 逻辑块号
    unsigned short ee_len;     // 范围长度（块数）
    unsigned int ee_start_hi;  // 物理块号（高位）
    unsigned int ee_start_lo;  // 物理块号（低位）
};

// extent 的优势
// 连续的物理块可以用一个 extent 表示
// 减少了索引项的数量
```

## 三种方式对比

| 特性 | 连续分配 | 链接分配 | 索引分配 |
|---|---|---|---|
| **空间利用率** | 低（有碎片） | 高 | 高 |
| **顺序访问速度** | 最快 | 较慢 | 快 |
| **随机访问速度** | 快 | 慢 | 快 |
| **文件增长支持** | 差 | 好 | 好 |
| **实现复杂度** | 简单 | 简单 | 较复杂 |
| **可靠性** | 好 | 较差 | 好 |
| **典型文件系统** | - | FAT | ext2/3/4, NTFS |

## 实际应用

### FAT：链接分配的经典

```bash
# FAT32 文件系统的簇大小
# 簇（Cluster）是 FAT 中的分配单位，由多个扇区组成
# FAT32 使用 32 位簇号

$ df -T
文件系统       类型    1K-块    已用    可用   已用% 挂载点
/dev/sda1     vfat   15358608  8567824 6790784   56%   /boot/efi

# vfat 是 FAT 文件系统，用于 U 盘、SD 卡等
```

### ext4：索引分配的优化

```bash
# ext4 使用 extent 减少索引块开销
$ tune2fs -l /dev/sda1 | grep -i extent
Filesystem features:      extent
```

### NTFS：混合方案

NTFS 结合了多种技术：
- MFT（Master File Table）：类似 inode 表
- B+ 树组织目录
- extent 风格的存储

## 面试高频问题

**Q1：连续分配为什么不适合大文件？**

连续分配会产生外部碎片。当磁盘使用一段时间后，很难找到足够大的连续空间来分配给大文件。

**Q2：链接分配的主要缺点是什么？**

链接分配的主要缺点是随机访问慢（必须从链头开始遍历）、块中需要存储指针（浪费空间）、可靠性问题（指针损坏导致文件丢失）。

**Q3：索引分配为什么能支持随机访问？**

因为索引块记录了每个数据块的位置。访问第 n 个块时，直接查索引块第 n 项即可得到物理块号，不需要遍历。

**Q4：为什么 ext4 比 ext3 更好？**

ext4 使用 extent 代替块指针，减少了索引项数量；支持延迟分配，优化写入性能；日志校验提高可靠性等。

**Q5：文件系统的日志是什么？**

日志文件系统在写入正式数据前，先把操作记录到日志区。如果写入过程崩溃，恢复时检查日志即可，不用全盘扫描。ext3/ext4、NTFS 都使用日志技术。

理解三种文件分配方式及其优缺点，是理解文件系统设计的关键。
