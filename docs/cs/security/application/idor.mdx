# 越权漏洞

你只能查看自己的订单，却看到了别人的订单——这就是越权漏洞。

越权漏洞（Broken Access Control）是 Web 应用中最常见的安全问题之一。攻击者利用这个漏洞，可以访问或操作不属于他们的资源。用户 A 能看用户 B 的数据，用户能修改管理员才能改的配置——这些听起来不可思议，但在实际应用中却屡见不鲜。

## 什么是越权

### 访问控制的三个层次

| 层次 | 说明 | 示例 |
|---|---|---|
| 认证（Authentication） | 证明你是谁 | 登录、验证码 |
| 授权（Authorization） | 决定你能做什么 | 权限控制 |
| 访问控制（Access Control） | 限制你能访问什么 | 资源权限 |

越权漏洞发生在「授权」或「访问控制」环节出了问题。

### 越权漏洞的分类

| 类型 | 说明 | 风险等级 |
|---|---|---|
| 水平越权 | 同级别用户之间的非法访问 | 中-高 |
| 垂直越权 | 低权限用户获得高权限操作 | 高-严重 |

## 水平越权

### 漏洞原理

水平越权发生在同一权限级别的用户之间，一个用户能访问另一个用户的资源：

```
场景：用户 A 和用户 B 都是普通用户
正常：用户 A 只能访问自己的订单
漏洞：用户 A 看到了用户 B 的订单
```

### 常见场景

#### 1. URL 参数直接暴露

```java
// 危险代码：订单 ID 直接暴露在 URL
@GetMapping("/order/{orderId}")
public Order getOrder(@PathVariable Long orderId) {
    // 没有验证订单是否属于当前用户
    return orderService.findById(orderId);
}

// 攻击者只需要修改 URL 中的 orderId：
// /order/12345 -> /order/12346
// 就可能看到别人的订单
```

#### 2. 请求参数篡改

```java
// 危险代码：用户 ID 从请求参数获取
@PostMapping("/transfer")
public String transfer(@RequestParam Long targetUserId,
                       @RequestParam BigDecimal amount) {
    User currentUser = getCurrentUser();
    // 只验证了金额，没有验证目标用户
    return transferService.transfer(currentUser, targetUserId, amount);
}

// 攻击者可以修改 targetUserId 指向任意账户
```

#### 3. API 接口无权限校验

```java
// 危险代码：RESTful API 没有校验资源归属
@RestController
@RequestMapping("/api/users")
public class UserApiController {

    @GetMapping("/{userId}/profile")
    public UserProfile getProfile(@PathVariable Long userId) {
        // 直接返回，没有检查 userId 是否等于当前用户
        return userService.findById(userId);
    }

    @GetMapping("/{userId}/settings")
    public UserSettings getSettings(@PathVariable Long userId) {
        // 获取用户设置，可能包含敏感信息
        return userService.getSettings(userId);
    }
}
```

### 攻击演示

```bash
# 通过修改 ID 遍历用户数据
curl -H "Cookie: session=USER_SESSION" \
     "https://api.example.com/api/orders/12345"

# 修改为其他 ID
curl -H "Cookie: session=USER_SESSION" \
     "https://api.example.com/api/orders/12346"

# 自动遍历脚本
for i in {12300..12400}; do
    response=$(curl -s "https://api.example.com/api/orders/$i")
    if echo "$response" | grep -q "orderId"; then
        echo "Found: $i - $response"
    fi
done
```

## 垂直越权

### 漏洞原理

垂直越权是低权限用户获得了高权限用户的操作能力：

```
场景：普通用户和管理员
正常：普通用户只能访问 /user/*
漏洞：普通用户能访问 /admin/* 或执行管理员操作
```

### 常见场景

#### 1. 管理员接口未鉴权

```java
// 危险代码：管理接口缺少权限检查
@RestController
@RequestMapping("/admin")
public class AdminController {

    @PostMapping("/user/delete")
    public String deleteUser(@RequestParam Long userId) {
        // 缺少 @PreAuthorize 或权限检查
        return adminService.deleteUser(userId);
    }

    @PostMapping("/config/update")
    public String updateConfig(@RequestParam String key,
                               @RequestParam String value) {
        // 可以修改系统配置
        return configService.update(key, value);
    }
}
```

#### 2. 依赖前端权限控制

```html
<!-- 前端隐藏菜单，但后端没有验证 -->
<!-- 前端代码：-->
<a href="/admin/user/delete" style="display:none">删除用户</a>

<!-- 攻击者可以直接访问被隐藏的链接 -->
curl -X POST "https://example.com/admin/user/delete?userId=123"
```

#### 3. Cookie/Session 权限提升

```java
// 危险代码：只检查是否登录，不检查权限级别
@PostMapping("/admin/system/reset")
public String resetSystem() {
    // 只验证了 isAuthenticated()
    // 没有检查是否有 ADMIN 权限
    return adminService.resetSystem();
}
```

### 攻击演示

```python
# 自动化垂直越权检测
import requests

# 普通用户 session
session = requests.Session()
session.post("https://example.com/login",
             data={"username": "user", "password": "pass123"})

# 尝试访问管理员接口
admin_endpoints = [
    "/admin/users",
    "/admin/config",
    "/admin/system/backup",
    "/api/admin/delete/123"
]

for endpoint in admin_endpoints:
    resp = session.get(f"https://example.com{endpoint}")
    if resp.status_code == 200:
        print(f"[+] Found: {endpoint}")
        print(resp.text)
    elif resp.status_code == 403:
        print(f"[-] Forbidden: {endpoint}")
    else:
        print(f"[?] Status {resp.status_code}: {endpoint}")
```

## 越权漏洞防御

### 1. 资源归属验证

```java
// 每个接口都要验证资源归属
@GetMapping("/order/{orderId}")
public Order getOrder(@PathVariable Long orderId) {
    User currentUser = getCurrentUser();

    // 验证订单是否属于当前用户
    Order order = orderService.findById(orderId);
    if (!order.getUserId().equals(currentUser.getId())) {
        throw new AccessDeniedException("无权访问此订单");
    }

    return order;
}
```

### 2. 使用权限注解

```java
// Spring Security 权限注解
@PreAuthorize("hasRole('ADMIN')")
@PostMapping("/admin/user/delete")
public String deleteUser(@RequestParam Long userId) {
    return adminService.deleteUser(userId);
}

// 基于资源的权限控制
@PreAuthorize("@permissionService.canAccessOrder(#orderId, authentication.name)")
@GetMapping("/order/{orderId}")
public Order getOrder(@PathVariable Long orderId) {
    return orderService.findById(orderId);
}
```

### 3. 通用权限校验服务

```java
@Service
public class PermissionService {

    public boolean canAccessOrder(Long orderId, String username) {
        Order order = orderRepository.findById(orderId);
        return order != null && order.getOwner().equals(username);
    }

    public boolean canAccessDocument(Long docId, Long userId) {
        Document doc = documentRepository.findById(docId);
        if (doc.isPublic()) return true;
        return doc.getOwner().equals(userId)
            || doc.getSharedUsers().contains(userId);
    }

    public boolean canModifySettings(Long userId, Long currentUserId) {
        // 只有用户本人或管理员可以修改设置
        return userId.equals(currentUserId)
            || userRepository.findById(currentUserId).hasRole("ADMIN");
    }
}
```

### 4. 基于策略的访问控制（PBAC）

```java
// 定义权限策略
@Configuration
@EnableMethodSecurity
public class SecurityConfig {

    @Bean
    public PermissionEvaluator permissionEvaluator() {
        return new CustomPermissionEvaluator();
    }
}

public class CustomPermissionEvaluator implements PermissionEvaluator {

    @Autowired
    private OrderRepository orderRepository;

    @Override
    public boolean hasPermission(Authentication auth,
                                 Object targetDomainObject,
                                 Object permission) {
        if (auth == null || targetDomainObject == null) {
            return false;
        }

        String permissionStr = (String) permission;

        if (targetDomainObject instanceof Order) {
            Order order = (Order) targetDomainObject;
            return hasOrderPermission(auth, order, permissionStr);
        }

        return false;
    }

    private boolean hasOrderPermission(Authentication auth,
                                       Order order,
                                       String permission) {
        String username = auth.getName();
        boolean isAdmin = auth.getAuthorities()
            .stream().anyMatch(a -> a.getAuthority().equals("ROLE_ADMIN"));

        switch (permission) {
            case "read":
                return order.getOwner().equals(username) || isAdmin;
            case "write":
            case "delete":
                return order.getOwner().equals(username) || isAdmin;
            default:
                return false;
        }
    }
}
```

### 5. UUID 替代数字 ID

```java
// 用 UUID 代替自增数字 ID，增加猜测难度
@Entity
public class Order {
    @Id
    @GeneratedValue(strategy = GenerationType.UUID)
    private String id;  // UUID，而不是 Long

    // ...
}

// 但这不是根本解决方案，只是增加了攻击难度
// 仍然需要正确的权限验证
```

### 6. 审计日志

```java
// 记录所有敏感操作的审计日志
@Service
public class AuditLogService {

    @Autowired
    private AuditLogRepository auditLogRepository;

    public void logAccess(String username, String action,
                         Object resource, boolean success) {
        AuditLog log = new AuditLog();
        log.setUsername(username);
        log.setAction(action);
        log.setResourceType(resource.getClass().getSimpleName());
        log.setResourceId(getResourceId(resource));
        log.setSuccess(success);
        log.setTimestamp(LocalDateTime.now());
        log.setIpAddress(getCurrentIpAddress());

        auditLogRepository.save(log);

        // 如果是越权尝试，发出告警
        if (!success && isSuspiciousPattern(username)) {
            alertSecurityTeam(username);
        }
    }
}
```

## 防御方案对比

| 方案 | 防护强度 | 实现复杂度 | 说明 |
|---|---|---|---|
| 资源归属验证 | 高 | 中 | 每个接口都需要验证 |
| 权限注解 | 高 | 中 | 声明式，代码清晰 |
| PBAC | 高 | 高 | 灵活，适合复杂场景 |
| UUID 替代数字 ID | 低 | 低 | 增加猜测难度，不是根本解决 |
| 审计日志 | 中 | 低 | 发现问题，但不能防止 |

- 水平越权和垂直越权有什么区别？
- 为什么说「前端隐藏菜单」不能防止越权？
- 如何设计一个安全的权限控制系统？
- RBAC 和 ABAC 有什么区别？
- 如何检测一个 API 是否存在越权漏洞？
- 为什么用 UUID 代替数字 ID 不能完全防止越权？

> 越权漏洞的本质是**缺少资源归属验证**。每一个涉及资源访问的接口，都必须验证当前用户是否有权访问该资源。**信任用户输入的参数，永远验证权限**——这是防御越权漏洞的铁律。
