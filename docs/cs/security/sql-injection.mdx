# SQL注入

> 目标级别：P5

面试官问：「什么是 SQL 注入？如何防范？」你回答「用预编译语句」——然后面试官追问：「预编译是怎么防止 SQL 注入的？」「为什么不能拼接 SQL？」「什么是 ORM 注入？」

SQL 注入是最常见的安全漏洞之一，理解它的原理和防范措施是每个后端开发者的必修课。

## 一、SQL 注入原理

### 1.1 什么是 SQL 注入

```
SQL 注入：通过在用户输入中注入恶意 SQL 代码，绕过应用逻辑，执行未授权的 SQL 操作。

本质：用户输入被当作 SQL 代码执行

攻击条件：
1. 用户输入直接拼接到 SQL 语句中
2. 没有对输入进行过滤或转义
```

### 1.2 注入示例

```sql
-- 正常查询
SELECT * FROM users WHERE username = 'admin' AND password = '123456'

-- SQL 注入（绕过登录）
SELECT * FROM users WHERE username = 'admin' --' AND password = 'anything'
-- 'admin' --' 将密码验证注释掉，任何密码都能登录
```

### 1.3 注入类型

| 类型 | 说明 |
|------|------|
|  UNION 注入 | 联合查询其他表 |
| 布尔盲注 | 根据返回真假判断信息 |
| 时间盲注 | 根据响应时间判断 |
| 报错注入 | 利用数据库错误信息 |
| 堆叠查询 | 执行多条 SQL |

## 二、注入示例代码

### 2.1 脆弱的代码

```java
// 危险：字符串拼接 SQL
public User findByUsername(String username) {
    String sql = "SELECT * FROM users WHERE username = '" + username + "'";
    // 如果 username = "admin' OR '1'='1"
    // SQL = "SELECT * FROM users WHERE username = 'admin' OR '1'='1'"
    // 返回所有用户！
}
```

### 2.2 注入攻击向量

```java
// 常见的恶意输入：

// 1. 绕过认证
username = "admin' --"
password = "anything"

// 2. 联合查询
username = "1' UNION SELECT * FROM passwords --"

// 3. 注释掉后续条件
username = "admin' /*"
password = "*/

// 4. 追加恶意条件
username = "admin' OR '1'='1"
```

## 三、防范措施

### 3.1 预编译语句（Prepared Statements）

```java
// 安全：使用预编译语句
public User findByUsername(String username) {
    String sql = "SELECT * FROM users WHERE username = ?";
    try (PreparedStatement stmt = connection.prepareStatement(sql)) {
        stmt.setString(1, username);
        ResultSet rs = stmt.executeQuery();
        if (rs.next()) {
            return new User(rs.getString("username"));
        }
    }
    return null;
}

// 预编译原理：
// 1. SQL 结构先发送到数据库，编译好
// 2. 用户输入只作为参数
// 3. 参数不可能被解释为 SQL 代码
```

### 3.2 ORM 框架

```java
// 使用 JPA/Hibernate
public User findByUsername(String username) {
    return entityManager.createQuery(
        "SELECT u FROM User u WHERE u.username = :username", User.class)
        .setParameter("username", username)
        .getSingleResult();
}

// 使用 MyBatis
@Select("SELECT * FROM users WHERE username = #{username}")
User findByUsername(@Param("username") String username);
```

### 3.3 输入验证

```java
// 1. 白名单验证
public boolean isValidUsername(String username) {
    return username.matches("^[a-zA-Z0-9_]{3,20}$");
}

// 2. 类型转换
int userId = Integer.parseInt(request.getParameter("id"));

// 3. 长度限制
if (username.length() > 50) {
    throw new ValidationException("用户名过长");
}
```

## 四、面试题精讲

### 🔴 【高频】SQL 注入原理与防范

**问题**：什么是 SQL 注入？如何防范？

**标准答案**：

```
SQL 注入原理：
- 用户输入被拼接到 SQL 语句中
- 恶意输入被当作 SQL 代码执行
- 绕过认证、获取未授权数据

注入示例：
username = "admin' --"
SQL = "SELECT * FROM users WHERE username = 'admin' --' AND password = ''"
-- 后面的内容被注释，绕过密码验证

防范措施：
1. 预编译语句（Prepared Statements）
   - SQL 结构和参数分离
   - 参数不可能被解释为 SQL

2. ORM 框架
   - 参数化查询
   - 自动处理转义

3. 输入验证
   - 白名单验证
   - 类型检查
   - 长度限制

4. 最小权限原则
   - 数据库账户只给必要权限
```

## 五、对比总结

### SQL 注入类型

| 类型 | 特点 | 检测难度 |
|------|------|----------|
| UNION 注入 | 联合查询 | 简单 |
| 布尔盲注 | 无回显 | 中等 |
| 时间盲注 | 无回显 | 难 |
| 堆叠查询 | 执行多条 SQL | 简单 |

### 防范措施对比

| 措施 | 效果 | 实现难度 |
|------|------|----------|
| 预编译语句 | 好 | 低 |
| ORM 框架 | 好 | 低 |
| 输入验证 | 中 | 中 |
| 存储过程 | 中 | 高 |
| WAF | 中 | 低 |

## 六、扩展思考

### 💡 MyBatis #{} vs ${}

```java
// #{}：参数化查询（安全）
@Select("SELECT * FROM users WHERE username = #{username}")
User findByUsername(String username);

// ${}：字符串替换（危险）
@Select("SELECT * FROM users WHERE username = '${username}'")
User findByUsername(String username);
// 如果 username = "admin' OR '1'='1"
// 实际 SQL = "SELECT * FROM users WHERE username = 'admin' OR '1'='1'"
```

### 💡 ORM 注入

```
虽然 ORM 框架可以防止大多数 SQL 注入，
但使用不当时仍可能存在注入风险：

1. 原生 SQL 查询
   @Query(value = "SELECT * FROM users WHERE " +
       "username = '" + username + "'", nativeQuery = true)

2. Hibernate 实体查询
   session.createQuery("FROM User WHERE name = '" + name + "'")

原则：
- 尽量使用参数化查询
- 避免字符串拼接
- 如果必须使用原生 SQL，确保参数被正确过滤
```

> SQL 注入是 Web 安全中最常见的问题之一。只要记住一个原则：**永远不要将用户输入直接拼接到 SQL 语句中**，就能避免大多数 SQL 注入漏洞。
