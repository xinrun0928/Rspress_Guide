# 安全沙箱与容器隔离机制

你的应用能在「笼子」里运行吗？

沙箱（Sandbox）是一种安全隔离技术，让不可信的代码在受限的环境中运行，限制其对系统资源的访问。容器（Container）是现代应用部署的主流方式，而容器安全的基础正是 Linux 的各种隔离机制。理解这些技术，是构建安全系统的前提。

## 沙箱与隔离概念

### 什么是沙箱

沙箱是一种安全机制，将程序运行在受限的环境中，限制其对文件、网络、系统调用的访问。即使程序被攻击者控制，造成的损害也被限制在沙箱内部。

### 沙箱 vs 虚拟机

| 特性 | 沙箱 | 虚拟机 |
|---|---|---|
| 隔离层级 | 操作系统级别 | 硬件级别 |
| 资源消耗 | 轻量（共享内核） | 重量（完整 OS） |
| 启动时间 | 秒级 | 分钟级 |
| 隔离强度 | 中 | 高 |
| 适用场景 | 应用隔离、微服务 | 完全隔离环境 |

## Linux Namespace

### 什么是 Namespace

Linux Namespace 是 Linux 内核提供的资源隔离机制，将系统的全局资源「分割」成多个独立的视图。每个 Namespace 内的进程只能看到该 Namespace 内的资源。

### 六种 Namespace

| Namespace | 隔离内容 | 用途 |
|---|---|---|
| PID | 进程 ID | 容器内进程从 1 开始 |
| Network | 网络设备、端口 | 网络隔离 |
| Mount | 文件系统挂载点 | 文件系统隔离 |
| UTS | 主机名、域名 | 主机名隔离 |
| IPC | 共享内存、信号量 | 进程间通信隔离 |
| User | 用户 ID、组 ID | 用户权限隔离 |
| Cgroup | Cgroup 根目录 | 资源限制 |

### Namespace 操作

```bash
# 创建新的 PID Namespace
unshare --pid --fork --mount-proc bash
# 在新 Namespace 中，ps 只能看到自己的进程

# 创建新的 Network Namespace
ip netns add container1
ip netns exec container1 ip link list

# 查看 Namespace
ls -la /var/run/netns/
ls -la /proc/self/ns/
```

```c
// 使用 clone() 创建新 Namespace
#define _GNU_SOURCE
#include <sched.h>
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>

int child(void *arg) {
    printf("In child namespace, PID = %d\n", getpid());
    // 子进程将在新的 Namespace 中运行
    execvp("bash", NULL);
    return 0;
}

int main() {
    // 创建新的 PID Namespace
    char *stack = malloc(4096);
    pid_t pid = clone(child, stack + 4096,
                      CLONE_NEWPID | CLONE_NEWUTS | SIGCHLD, NULL);

    printf("Parent PID = %d, child PID in parent NS = %d\n",
           getpid(), pid);

    waitpid(pid, NULL, 0);
    return 0;
}
```

## Cgroup（控制组）

### 什么是 Cgroup

Cgroup 是 Linux 内核用于限制、记录和隔离进程组资源使用（CPU、内存、I/O、网络）的机制。

### Cgroup v2 资源限制

```bash
# 创建 Cgroup
sudo mkdir -p /sys/fs/cgroup/limit

# 限制 CPU
echo 50000 > /sys/fs/cgroup/limit/cpu.max
# 限制 50% CPU

# 限制内存
echo 100M > /sys/fs/cgroup/limit/memory.max
# 限制 100MB 内存

# 添加进程到 Cgroup
echo $$ > /sys/fs/cgroup/limit/cgroup.procs
```

```yaml
# Docker Compose 中使用 Cgroup 限制
version: '3.8'
services:
  app:
    image: myapp:latest
    cpus: '1.5'           # 限制 1.5 个 CPU
    mem_limit: 512m       # 限制 512MB 内存
    pids_limit: 100       # 限制最多 100 个进程

  database:
    image: postgres:latest
    cpus: '2.0'
    mem_limit: 2g
```

### OOM Killer 防护

```bash
# 设置内存限制并防止 OOM Kill
echo 200M > /sys/fs/cgroup/limit/memory.max
echo 0 > /sys/fs/cgroup/limit/memory.swap.max
# memory.swap.max = 0 禁止使用 swap，防止内存溢出
```

## Seccomp

### 什么是 Seccomp

Seccomp（Secure Computing Mode）是 Linux 内核的安全机制，限制进程可以使用的系统调用。默认情况下，进程可以使用 300+ 个系统调用，但大多数程序只需要其中几十个。

### Seccomp 模式

| 模式 | 说明 |
|---|---|
| SECCOMP_MODE_STRICT | 只允许 read, write, _exit, sigreturn |
| SECCOMP_MODE_FILTER | 使用 BPF 程序过滤系统调用 |

### Seccomp BPF

```c
// seccomp.c - Seccomp BPF 过滤
#include <seccomp.h>
#include <unistd.h>
#include <sys/syscall.h>

int main() {
    // 初始化 seccomp 上下文
    scmp_filter_ctx ctx = seccomp_init(SECCOMP_RET_KILL);
    if (!ctx) return 1;

    // 添加允许规则
    seccomp_rule_add(ctx, SCMP_ACT_ALLOW, SCMP_SYS(read), 0);
    seccomp_rule_add(ctx, SCMP_ACT_ALLOW, SCMP_SYS(write), 0);
    seccomp_rule_add(ctx, SCMP_ACT_ALLOW, SCMP_SYS(exit_group), 0);

    // 禁止 execve（防止执行程序）
    seccomp_rule_add(ctx, SCMP_ACT_KILL, SCMP_SYS(execve), 0);
    seccomp_rule_add(ctx, SCMP_ACT_KILL, SCMP_SYS(execveat), 0);

    // 加载规则
    seccomp_load(ctx);

    // 现在只能使用 read, write, exit_group
    char msg[] = "Hello, Sandbox!\n";
    write(STDOUT_FILENO, msg, sizeof(msg) - 1);

    return 0;
}
```

### Docker Seccomp

```json
// custom-seccomp.json
{
    "defaultAction": "SCMP_ACT_ERRNO",
    "architectures": ["SCMP_ARCH_X86_64"],
    "syscalls": [
        {
            "names": ["read", "write", "exit", "exit_group"],
            "action": "SCMP_ACT_ALLOW"
        },
        {
            "names": ["brk", "mmap", "mprotect", "munmap"],
            "action": "SCMP_ACT_ALLOW"
        },
        {
            "names": ["socket", "connect", "accept"],
            "action": "SCMP_ACT_ALLOW"
        }
    ]
}
```

```bash
# 应用自定义 Seccomp 配置
docker run --security-opt seccomp=./custom-seccomp.json myapp
```

### Docker 默认 Seccomp 配置

Docker 默认禁用 44 个危险系统调用：

```bash
# 查看 Docker 默认 Seccomp 规则
docker run --rm --security-opt seccomp=unconfined alpine cat /proc/self/status
```

## Capabilities

### 什么是 Capabilities

Linux 将 root 的特权拆分成多个能力（Capabilities），可以精细控制进程权限。容器默认以非 root 用户运行，但启用了部分 capabilities。

### 危险 Capabilities

| Capability | 危险程度 | 说明 |
|---|---|---|
| CAP_SYS_ADMIN | 极高 | 允许挂载、修改内核参数 |
| CAP_SYS_MODULE | 极高 | 允许加载内核模块 |
| CAP_NET_ADMIN | 高 | 修改网络配置 |
| CAP_SYS_RAWIO | 高 | 直接访问磁盘 |
| CAP_SYS_PTRACE | 高 | 调试任意进程 |

### Docker Capabilities 配置

```bash
# 查看容器 Capabilities
docker run --rm alpine capsh --print

# 禁用所有 Capabilities
docker run --rm --cap-drop=ALL myapp

# 只授予必要的 Capabilities
docker run --rm \
    --cap-drop=ALL \
    --cap-add=NET_BIND_SERVICE \
    --cap-add=DAC_OVERRIDE \
    myapp

# 禁用特权容器
docker run --rm --privileged  # 危险！授予所有 Capabilities
```

## AppArmor

### 什么是 AppArmor

AppArmor 是一个强制访问控制（MAC）系统，通过配置文件定义程序的访问策略。每个程序有自己 profile，限制它可以访问的文件、网络资源。

### AppArmor Profile

```bash
# /etc/apparmor.d/usr.sbin.httpd

#include <tunables/global>

profile httpd /usr/sbin/httpd {
    # 本地变量
    @{HOME}/logs/ rw,

    # 网络策略
    network inet stream,
    # 禁止 raw socket
    deny network inet raw,

    # 文件访问策略
    /var/www/ r,
    /var/www/** r,

    # 禁止执行
    /bin/** px,  # 只读执行
    /usr/lib/** kr,  # 只读知识库

    # 禁止能力
    deny capability sys_admin,
    deny capability sys_module,
}
```

```bash
# 加载/更新 AppArmor profile
apparmor_parser -r /etc/apparmor.d/usr.sbin.httpd

# 设置强制或投诉模式
aa-enforce usr.sbin.httpd   # 强制执行
aa-complain usr.sbin.httpd  # 只记录，不阻止
```

### Docker AppArmor 配置

```bash
# Docker 自动为容器生成 AppArmor profile
# 查看当前容器的 profile
docker inspect --format '{{ .AppArmorProfile }}' container_name

# 使用自定义 profile
docker run --security-opt "apparmor=docker-custom" myapp
```

## 容器安全最佳实践

### Docker 安全加固

```yaml
# docker-compose.yml - 安全配置
version: '3.8'
services:
  app:
    image: myapp:latest

    # 禁用特权模式
    # privileged: true  # 危险！

    # 禁用新特权
    security_opt:
      - no-new-privileges:true

    # 只读文件系统
    read_only: true

    # 临时文件系统
    tmpfs:
      - /tmp
      - /run

    # 禁用 Capabilities
    cap_drop:
      - ALL

    # 资源限制
    deploy:
      resources:
        limits:
          cpus: '1.0'
          memory: 512M
        reservations:
          cpus: '0.5'
          memory: 256M

    # 用户权限
    user: "1000:1000"

    # 网络模式
    network_mode: "none"  # 无网络访问
    # 或限制网络
    # network_mode: "internal"
```

### Kubernetes 安全上下文

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: secure-pod
spec:
  securityContext:
    runAsNonRoot: true
    runAsUser: 1000
    fsGroup: 1000
    seccompProfile:
      type: RuntimeDefault

  containers:
  - name: app
    image: myapp:latest
    securityContext:
      allowPrivilegeEscalation: false
      readOnlyRootFilesystem: true
      capabilities:
        drop:
          - ALL
        add:
          - NET_BIND_SERVICE

    resources:
      limits:
        memory: "128Mi"
        cpu: "500m"
```

### Seccomp + AppArmor 组合

```bash
# 同时使用 Seccomp 和 AppArmor
docker run --rm \
    --security-opt "seccomp=./seccomp-profile.json" \
    --security-opt "apparmor=docker-default" \
    --cap-drop=ALL \
    --read-only \
    --tmpfs /tmp \
    myapp
```
