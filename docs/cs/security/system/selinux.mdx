# SELinux 与 AppArmor 强制访问控制

DAC 已经不够用了——你需要更强大的访问控制。

传统的 Unix 权限模型（Owner-Group-Other）是自主访问控制（DAC），它依赖于用户和组的身份来控制访问。但在复杂的企业环境中，DAC 远远不够。强制访问控制（MAC）由系统管理员预先定义策略，无论用户是谁，都必须遵守这些规则。SELinux 和 AppArmor 就是实现 MAC 的两种主要机制。

## 访问控制模型

### DAC vs MAC

| 特性 | DAC（自主访问控制） | MAC（强制访问控制） |
|---|---|---|
| 控制依据 | 用户/组身份 | 安全标签 |
| 策略来源 | 资源所有者 | 系统管理员 |
| 可覆盖性 | 可被进程继承 | 不可绕过 |
| 适用场景 | 个人系统 | 企业/高安全环境 |

### SELinux 和 AppArmor 对比

| 特性 | SELinux | AppArmor |
|---|---|---|
| 开发公司 | NSA + Red Hat | Immunix（后被 Novell 收购） |
| 控制粒度 | 非常细（进程、文件、网络） | 中等（程序 + 文件路径） |
| 配置文件 | 复杂（.te 文件） | 简单（基于路径） |
| 默认安全级别 | 高 | 中 |
| 学习曲线 | 陡峭 | 平缓 |

## SELinux 详解

### 基本概念

#### 安全上下文（Security Context）

SELinux 为每个进程和文件分配安全上下文：

```
user:role:type:level
```

| 字段 | 说明 | 示例 |
|---|---|---|
| user | SELinux 用户身份 | unconfined_u, staff_u |
| role | 角色 | object_r, system_r |
| type | 类型（最关键） | httpd_t, etc_t |
| level | 安全级别 | s0, s0-s15:c0.c255 |

```bash
# 查看文件安全上下文
ls -Z /etc/passwd
# -rw-r--r--. root root system_u:object_r:etc_t:s0 /etc/passwd

# 查看进程安全上下文
ps auxZ | head -5
# system_u:system_r:init_t:s0         1 ?        Ss
# system_u:system_r:sshd_t:s0         999 ?      Ss

# 查看用户安全上下文
id -Z
# unconfined_u:unconfined_r:unconfined_t:s0
```

### SELinux 模式

| 模式 | 说明 | 适用场景 |
|---|---|---|
| Enforcing | 强制执行，违规操作被拒绝并记录 | 生产环境 |
| Permissive | 不执行策略，只记录违规 | 测试调试 |
| Disabled | 完全禁用 SELinux | 不推荐 |

```bash
# 查看当前模式
getenforce

# 临时切换模式（需要 root）
setenforce 0   # 切换到 Permissive
setenforce 1   # 切换到 Enforcing

# 永久配置
# /etc/selinux/config
SELINUX=enforcing
SELINUXTYPE=targeted
```

### SELinux 策略类型

| 策略类型 | 说明 |
|---|---|
| Targeted | 只保护指定的服务（默认） |
| Minimum | 只保护最少的服务 |
| MLS | 多级安全（军方级别） |

## SELinux 命令

### 基础命令

```bash
# 安装 SELinux 管理工具
yum install -y policycoreutils policycoreutils-python
apt install -y policycoreutils selinux-basics

# 查看状态
sestatus
sestatus -v

# 查看布尔值
getsebool -a
getsebool httpd_enable_homedirs

# 设置布尔值（临时）
setsebool httpd_enable_homedirs 1
setsebool -P httpd_enable_homedirs 1  # 永久
```

### 文件上下文

```bash
# 查看文件默认上下文
semanage fcontext -l | grep /var/www

# 修改文件上下文（临时）
chcon -t httpd_sys_content_t /var/www/html

# 修改文件上下文（永久）
semanage fcontext -a -t httpd_sys_content_t "/var/www/html(/.*)?"
restorecon -Rv /var/www/html

# 恢复默认上下文
restorecon /path/to/file
restorecon -Rv /var/www/html
```

### 端口上下文

```bash
# 查看端口上下文
semanage port -l | grep http

# 添加端口上下文
semanage port -a -t http_port_t -p tcp 8080

# 删除端口上下文
semanage port -d -t http_port_t -p tcp 8080
```

## SELinux 策略示例

### Apache HTTPD

```bash
# 常见 Apache SELinux 问题及解决

# 问题1：无法读取用户 home 目录
# 原因：httpd 默认禁止访问用户目录
setsebool -P httpd_enable_homedirs 1
setsebool -P httpd_read_user_content 1

# 问题2：无法连接数据库
setsebool -P httpd_can_network_connect_db 1

# 问题3：无法发送邮件
setsebool -P httpd_can_sendmail 1

# 问题4：自定义 DocumentRoot
semanage fcontext -a -t httpd_sys_content_t "/opt/webapp(/.*)?"
restorecon -Rv /opt/webapp
```

### Nginx

```bash
# Nginx SELinux 配置
# 安装
yum install -y nginx
semanage port -a -t http_port_t -p tcp 80
semanage port -a -t http_port_t -p tcp 443

# 允许 Nginx 执行 CGI
setsebool -P nis_enabled 1

# 允许 Nginx 访问网络
setsebool -P httpd_can_network_connect 1
```

### 数据库

```bash
# MySQL/MariaDB
setsebool -P mysql_connect_http 1
setsebool -P mysqld_anon_write 1

# PostgreSQL
setsebool -P postgresql_can_rsync 1
setsebool -P postgresql_selinux_unconfined_daemons_on 1
```

## AppArmor 详解

### 基本概念

AppArmor 通过程序路径来定义安全策略，比 SELinux 更容易理解：

```
profile program-name {
    # 本地变量
    /path/to/program r,
    /path/to/data/** rw,
}
```

### AppArmor 模式

| 模式 | 说明 |
|---|---|
| Enforce | 强制执行策略 |
| Complain | 只记录，不阻止 |

### AppArmor Profile 示例

```bash
# /etc/apparmor.d/usr.bin.myapp

#include <tunables/global>

profile myapp /usr/bin/myapp {
    # 本地变量定义
    @{HOME}/logs/ r,
    @{HOME}/data/** rw,

    # 能力（Capabilities）
    capability setuid,
    capability setgid,

    # 网络策略
    network inet stream,
    # network inet6 stream,
    # deny network raw,

    # 文件访问
    /etc/myapp/ r,
    /etc/myapp/*.conf r,
    /var/log/myapp/ rw,
    /tmp/ rw,

    # 禁止访问
    /etc/shadow r,  # 只读
    deny /proc/** w,  # 禁止写 /proc
}
```

### Docker AppArmor

```bash
# Docker 默认使用 docker-default profile
docker ps --format '{{ .Names }}: {{ .AppArmorProfile }}'

# 查看默认 profile
cat /etc/apparmor.d/docker-default

# 自定义 AppArmor profile
docker run --security-opt "apparmor=docker-mydb" myapp

# 加载自定义 profile
apparmor_parser -r /etc/apparmor.d/docker-mydb
```

## SELinux 故障排除

### 常见问题

```bash
# 问题1：Permission denied
# 查看审计日志
ausearch -m AVC -ts recent | less
ausearch -m AVC -c httpd

# 问题2：进程被拒绝
# 查看详细错误
tail -f /var/log/audit/audit.log | grep httpd

# 问题3：排查工具
yum install -y setroubleshoot-server
sealert -a /var/log/audit/audit.log
```

### 临时禁用（紧急情况）

```bash
# 临时禁用（不推荐用于生产）
setenforce 0

# 如果 SELinux 导致系统无法启动
# 在 GRUB 启动参数中添加
# selinux=0
```

## SELinux vs AppArmor 选择

| 场景 | 推荐 |
|---|---|
| Red Hat/CentOS/Fedora | SELinux |
| Debian/Ubuntu | AppArmor |
| 高安全要求 | SELinux |
| 易用性优先 | AppArmor |
| 容器环境 | AppArmor |
| 混合 Linux 环境 | SELinux |

## 安全策略最佳实践

### SELinux 配置

```bash
# /etc/selinux/config
SELINUX=enforcing
SELINUXTYPE=targeted

# 创建自定义策略模块
# 1. 收集允许的操作
audit2allow -M myapp < /var/log/audit/audit.log

# 2. 查看生成的策略
cat myapp.te

# 3. 安装策略
semodule -i myapp.pp

# 4. 查看已加载的模块
semodule -l
```

### 应用容器双重保护

```yaml
# Kubernetes Pod 安全上下文 + SELinux
apiVersion: v1
kind: Pod
metadata:
  name: secure-pod
spec:
  securityContext:
    seLinuxOptions:
      level: "s0:c123,c456"
      role: "object_r"
      user: "system_u"
      type: "svirt_lxc_net_t"
  containers:
  - name: app
    image: myapp:latest
    securityContext:
      seccompProfile:
        type: RuntimeDefault
```
