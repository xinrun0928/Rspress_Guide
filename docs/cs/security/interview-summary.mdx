# 计算机安全面试高频问题

面试官问「你了解哪些安全漏洞」，你只能说出 XSS 和 SQL 注入？

安全面试不是背题库，而是**理解漏洞背后的原理**，知道「为什么会这样」和「如何从根本上防御」。

本文汇总了计算机安全领域的高频面试题，涵盖密码学、Web 安全、系统安全三大板块，每道题都给出了回答要点和延伸思考。

## 一、密码学基础

### 1. 对称加密 vs 非对称加密的区别？各自适用场景？

**回答要点：**

| 特性 | 对称加密 | 非对称加密 |
|---|---|---|
| 密钥数量 | 一个（共享密钥） | 一对（公钥 + 私钥） |
| 加密速度 | 快（10-100 倍于非对称） | 慢 |
| 密钥分发 | 困难（需要安全通道） | 容易（公钥可公开） |
| 典型算法 | AES、ChaCha20 | RSA、ECC |
| 适用场景 | 大量数据加密 | 密钥交换、数字签名 |

**实际应用：** HTTPS 采用混合加密——用非对称加密交换对称密钥，后续通信用对称加密。

### 2. 为什么 HTTPS 要用混合加密？

纯粹的非对称加密太慢。AES-128-GCM 加密 1MB 数据可能只需几毫秒，而 RSA-2048 加密同等数据可能需要几秒钟。

混合加密的流程：

1. 客户端生成随机数 `pre-master secret`
2. 用服务器的公钥加密后发送
3. 双方用 `pre-master secret` 推导出会话密钥
4. 后续通信用会话密钥（对称加密）进行

### 3. 数字签名的工作原理？

数字签名不是为了加密，而是为了**验证消息的完整性和发送者身份**。

```
签名过程：
1. 对消息计算哈希值：hash = SHA-256(message)
2. 用私钥加密哈希：signature = RSA_sign(private_key, hash)
3. 发送（消息 + 签名）

验证过程：
1. 用公钥解密签名：hash = RSA_verify(public_key, signature)
2. 对消息重新计算哈希：hash' = SHA-256(message)
3. 比较 hash == hash'
```

### 4. 为什么有了 MD5 和 SHA-1 还要用 SHA-256？

| 算法 | 输出长度 | 安全性 | 状态 |
|---|---|---|---|
| MD5 | 128 位 | 已攻破 | 禁止用于安全场景 |
| SHA-1 | 160 位 | 已攻破（2017 年） | 不推荐 |
| SHA-256 | 256 位 | 安全 | 推荐 |
| SHA-384 | 384 位 | 安全 | 高安全场景 |

MD5 和 SHA-1 存在**碰撞攻击**——可以找到两个不同消息产生相同哈希值。

### 5. 什么是 JWT？它安全吗？

JWT（JSON Web Token）由三部分组成：`Header.Payload.Signature`

```javascript
// Header: 算法和类型
{ "alg": "HS256", "typ": "JWT" }

// Payload: Claims（声明）
{ "sub": "123456", "name": "John", "exp": 1699999999 }

// Signature: 签名
HMACSHA256(base64UrlEncode(header) + "." + base64UrlEncode(payload), secret)
```

**JWT 的安全风险：**

- **密钥泄露**：如果 `secret` 被攻击者获取，可以伪造任意 JWT
- **算法切换**：如果服务器支持 `alg: none`，攻击者可能绕过签名验证
- **敏感信息**：Payload 是 Base64 编码，可被轻易解码，不应存放敏感信息

**安全建议：** 使用强密钥、设置短期过期时间、验证 `iss` 和 `aud` 声明。

## 二、Web 安全

### 1. XSS 有哪几种类型？如何防御？

**三种 XSS 类型：**

| 类型 | 触发方式 | 危险程度 |
|---|---|---|
| 反射型 | URL 参数直接拼接到页面 | 低（需诱导点击） |
| 存储型 | 恶意脚本存入数据库 | 高（影响所有访问者） |
| DOM 型 | JS 动态解析用户输入 | 中（纯客户端攻击） |

**防御措施：**

```javascript
// 1. 输入过滤（白名单）
function sanitizeInput(input) {
    return input.replace(/[<>'"]/g, '');
}

// 2. 输出编码
function escapeHtml(str) {
    return str
        .replace(/&/g, '&amp;')
        .replace(/</g, '&lt;')
        .replace(/>/g, '&gt;')
        .replace(/"/g, '&quot;')
        .replace(/'/g, '&#x27;');
}

// 3. CSP（Content Security Policy）
// Content-Security-Policy: script-src 'self'; object-src 'none'
```

### 2. SQL 注入的原理是什么？如何防御？

用户输入 `1' OR '1'='1` 被直接拼接到 SQL 语句：

```sql
-- 原始意图
SELECT * FROM users WHERE id = '1'

-- 被注入后
SELECT * FROM users WHERE id = '1' OR '1'='1'
```

**防御措施：**

```java
// 1. 参数化查询（最佳方案）
String sql = "SELECT * FROM users WHERE id = ?";
PreparedStatement stmt = connection.prepareStatement(sql);
stmt.setString(1, userId);
ResultSet rs = stmt.executeQuery();

// 2. ORM 框架（Hibernate、MyBatis）
// 使用占位符，框架自动处理转义

// 3. 输入验证
if (!userId.matches("^[0-9]+$")) {
    throw new IllegalArgumentException("Invalid ID");
}
```

### 3. CSRF 的原理？如何防御？

攻击原理：用户登录 A 网站后，访问恶意网站 B，B 自动发起对 A 的请求（携带 A 的 Cookie）。

```
用户登录 A 网站
    ↓
A 网站返回 Cookie
    ↓
用户访问恶意网站 B
    ↓
B 页面的 JS 自动发起请求到 A（携带 Cookie）
    ↓
A 网站执行对应操作（转账、修改密码等）
```

**防御措施：**

| 措施 | 原理 |
|---|---|
| CSRF Token | 随机 token 必须随请求发送，恶意网站无法获取 |
| SameSite Cookie | 浏览器阻止跨站请求携带 Cookie |
| 双重提交 | 请求头和 Cookie 都包含相同的随机值 |
| 验证 Referer/Origin | 检查请求来源 |

```java
// Spring Security CSRF Token
@PostMapping("/transfer")
public String transfer(@RequestParam Long amount,
                       @RequestParam String to,
                       @RequestParam("_csrf") String csrfToken) {
    // Spring 自动验证 CSRF Token
    bankService.transfer(amount, to);
    return "success";
}
```

### 4. SSRF 是什么？如何防御？

SSRF（Server-Side Request Forgery）：服务器代替客户端发起请求，可能访问内网资源。

```java
// 危险的代码
@GetMapping("/fetch")
public String fetch(@RequestParam String url) {
    // 攻击者可能传入 http://169.254.169.254/latest/meta-data/
    // 获取云服务器元数据（包含 IAM 凭证）
    return restTemplate.getForObject(url, String.class);
}

// 安全的代码
@GetMapping("/fetch")
public String fetch(@RequestParam String url) {
    // 1. URL 白名单验证
    if (!allowedHosts.contains(getHost(url))) {
        throw new SecurityException("Host not allowed");
    }

    // 2. 禁止内网地址
    InetAddress addr = InetAddress.getByName(getHost(url));
    if (addr.isSiteLocalAddress() || addr.isLoopbackAddress()) {
        throw new SecurityException("Internal host not allowed");
    }

    return restTemplate.getForObject(url, String.class);
}
```

### 5. 文件上传漏洞如何利用？如何防御？

**利用方式：**

```php
// PHP 文件上传漏洞
move_uploaded_file($_FILES["upload"]["tmp_name"], "uploads/" . $_FILES["upload"]["name"]);
// 攻击者上传 webshell.php
```

**防御措施：**

```java
// 1. 文件类型验证（白名单）
private static final Set<String> ALLOWED_TYPES = Set.of(
    "image/jpeg", "image/png", "image/gif"
);

// 2. 验证文件头（魔数）
private boolean isValidImage(InputStream input) {
    byte[] header = new byte[4];
    input.read(header);
    // JPEG: FF D8 FF
    // PNG: 89 50 4E 47
    return checkMagicNumber(header);
}

// 3. 禁止执行权限（上传目录不放执行权限）
// 4. 重命名文件（不用用户提供的文件名）
// 5. 存储到 OSS/CDN（与 Web 服务器分离）
```

## 三、系统安全

### 1. 什么是缓冲区溢出？有什么危害？

```c
// 存在溢出风险的代码
void vulnerable(char *input) {
    char buffer[64];
    strcpy(buffer, input);  // 没有边界检查
    // 如果 input 超过 64 字节，会覆盖相邻内存
}

// 攻击者可以利用溢出：
// 1. 覆盖返回地址 → 执行任意代码
// 2. 覆盖函数指针 → 控制执行流程
// 3. 覆盖关键变量 → 绕过安全检查
```

**危害：** 远程代码执行（RCE），攻击者可以完全控制系统。

### 2. 常见的内存保护机制有哪些？

| 机制 | 原理 |
|---|---|
| ASLR | 随机化内存地址，使攻击者难以预测目标地址 |
| DEP/NX | 将内存页标记为不可执行，阻止代码注入 |
| Stack Canary | 在栈帧中插入随机值，检测溢出是否发生 |
| CFI | 控制流完整性，验证间接调用的目标是否合法 |

### 3. 什么是 ROP（返回导向编程）？

ASLR 和 DEP 普及后，攻击者发明了 ROP——不注入新代码，而是复用程序中已有的「gadget」（以 `ret` 结尾的代码片段）来构造攻击。

```asm
# 假设程序中有这些 gadget：
gadget1:
    pop eax
    ret

gadget2:
    pop ebx
    ret

gadget3:
    add eax, ebx
    ret

# 攻击者通过ROP链控制寄存器，执行任意计算
```

**防御措施：** CFI（控制流完整性）、Shadow Stack、Intel CET。

### 4. 什么是权限提升？如何防御？

权限提升分为两种：

| 类型 | 说明 | 示例 |
|---|---|---|
| 水平越权 | 同级别用户访问其他用户资源 | 用户 A 查看用户 B 的订单 |
| 垂直越权 | 低权限用户获得高权限 | 普通用户执行管理员操作 |

**防御措施：**

```java
// 1. 每次操作都验证权限
@PostMapping("/delete")
public void delete(@RequestParam Long id) {
    User current = getCurrentUser();
    Resource resource = resourceService.findById(id);

    // 垂直越权防御：检查当前用户是否有权删除
    if (!permissionService.canDelete(current, resource)) {
        throw new AccessDeniedException();
    }

    resourceService.delete(id);
}

// 2. 使用安全框架（Spring Security、Shiro）
// 3. 最小权限原则：服务账户只授予必要权限
```

## 四、安全运维

### 1. 什么是零信任架构？

零信任的核心原则：**永不信任，始终验证**。

传统安全：边界内 = 可信（内网可以直接访问）
零信任：无论在哪里，都需要验证

```
零信任三要素：
1. 验证身份：用户、设备、服务都要有身份
2. 验证设备：设备状态是否合规（补丁、杀毒软件）
3. 验证访问：最小权限，只授权必要的资源
```

### 2. 如何设计安全的 CI/CD 流程？

```
DevSecOps 流水线：

代码提交 → SAST（静态代码扫描）→ 依赖检查 → 构建 → 镜像扫描 → 部署
   ↓            ↓                 ↓          ↓         ↓
代码审查    发现漏洞源码        CVE扫描     恶意代码   漏洞镜像
```

**关键措施：**

- 代码审查必须有人工参与
- 依赖库使用前扫描 CVE
- 镜像使用最小化基础镜像
- 密钥使用 Vault 管理，不放在代码中
- 部署前进行安全测试

### 3. 如何应对勒索软件攻击？

**预防措施：**

- 离线备份（3-2-1 原则：3 份副本、2 种介质、1 份离线）
- 最小权限原则
- 邮件网关过滤
- 用户安全培训（不点击可疑链接）

**应急响应：**

```bash
# 1. 立即隔离被感染主机
# 断开网络连接，禁用无线网卡

# 2. 评估影响范围
# 检查同一网段的其他主机

# 3. 不支付赎金（鼓励攻击）
# 也不保证能解密

# 4. 从备份恢复
# 确保备份未被感染

# 5. 事件调查
# 分析攻击路径，修复漏洞
```

## 面试技巧

### 回答安全问题的方法论

1. **先说原理**：漏洞是怎么产生的，攻击者如何利用
2. **再说影响**：会造成什么后果，影响范围多大
3. **最后说防御**：从开发、运维、架构层面如何防御
4. **适当延伸**：可以提一个实际踩坑案例或最佳实践

### 常见陷阱

| 陷阱 | 正确做法 |
|---|---|
| 只背答案 | 理解原理，能举一反三 |
| 只会防御 | 理解攻击，才能更好防御 |
| 只会攻击 | 展示工程能力，说明如何落地修复 |
| 回答太长 | 抓住核心，面试官追问再展开 |

> 安全面试的本质是**验证你对安全的理解深度**，而不是看你背了多少道题。理解一个漏洞的来龙去脉，比记住十个漏洞的名字更有价值。
