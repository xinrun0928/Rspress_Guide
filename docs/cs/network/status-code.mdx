# 常见状态码

> 目标级别：P5

面试官问：「HTTP 状态码有哪些？301 和 302 有什么区别？」你回答「301 是永久重定向，302 是临时重定向」——然后面试官追问：「304 是什么？什么时候返回 304？」「500 和 502 有什么区别？」

HTTP 状态码是 Web 开发中最基础的知识点，面试中出现频率很高。

## 快速自测

面试前先问自己这三个问题：

1. **HTTP 状态码分为几类？** 每类代表什么含义？
2. **301 和 302 的区别是什么？** 浏览器会缓存哪个？
3. **什么时候返回 304？** 304 能减少数据传输吗？

## 一、状态码分类

HTTP 状态码由三位数字组成，第一位数字表示类别：

| 类别 | 范围 | 含义 |
|------|------|------|
| 1xx | 100-199 | 信息性状态码 |
| 2xx | 200-299 | 成功状态码 |
| 3xx | 300-399 | 重定向状态码 |
| 4xx | 400-499 | 客户端错误状态码 |
| 5xx | 500-599 | 服务端错误状态码 |

---

## 二、1xx 信息性状态码

1xx 状态码表示请求已接收，需要继续处理。

| 状态码 | 说明 | 使用场景 |
|--------|------|----------|
| 100 | Continue | 客户端可以继续发送请求体 |
| 101 | Switching Protocols | 服务端同意切换协议（如 HTTP → WebSocket） |

### 2.1 100 Continue

```
使用场景：客户端有一个大请求体，先发送请求头询问服务端是否接受。

客户端：
POST /upload HTTP/1.1
Host: example.com
Content-Type: multipart/form-data
Content-Length: 1048576
Expect: 100-continue

服务端：
HTTP/1.1 100 Continue

客户端继续发送请求体...
```

---

## 三、2xx 成功状态码

2xx 表示请求已成功被服务器接收、理解并处理。

| 状态码 | 说明 | 使用场景 |
|--------|------|----------|
| 200 | OK | 请求成功（GET 返回资源，POST 返回处理结果） |
| 201 | Created | 资源创建成功（POST 创建资源） |
| 202 | Accepted | 请求已接收，但处理未完成（异步操作） |
| 204 | No Content | 请求成功，但无返回内容（DELETE 操作） |
| 206 | Partial Content | 部分内容返回（断点续传） |

### 3.1 常见状态码详解

```java
// 200 OK - 最常见
HTTP/1.1 200 OK
Content-Type: application/json

{"code": 0, "message": "success"}

// 201 Created - POST 创建资源
HTTP/1.1 201 Created
Location: /users/123

// 204 No Content - DELETE 成功
HTTP/1.1 204 No Content

// 206 Partial Content - 断点续传
HTTP/1.1 206 Partial Content
Content-Range: bytes 0-999/10000
Content-Length: 1000

[binary data...]
```

## 四、3xx 重定向状态码

3xx 表示需要进一步操作才能完成请求。

| 状态码 | 说明 | 缓存行为 |
|--------|------|----------|
| 301 | Moved Permanently | 永久重定向，**会缓存** |
| 302 | Found | 临时重定向，**不缓存** |
| 303 | See Other | 强制 POST→GET |
| 304 | Not Modified | 使用缓存，**不返回内容** |
| 307 | Temporary Redirect | 临时重定向，方法不变 |
| 308 | Permanent Redirect | 永久重定向，方法不变 |

### 4.1 301 vs 302

| 维度 | 301 | 302 |
|------|-----|-----|
| 含义 | 永久移动 | 临时移动 |
| 语义 | URL 永久变更 | URL 暂时变更 |
| 缓存 | **会被缓存** | 不会被缓存 |
| SEO 影响 | 传递权重 | 不传递权重（旧行为） |
| 请求方法 | 可能改变（GET/POST） | 可能改变（GET/POST） |

### 4.2 301 vs 308 vs 307

```
301：永久重定向，方法可能改变
302：临时重定向，方法可能改变
303：强制 POST → GET（不允许 POST → POST）
307：临时重定向，方法不变
308：永久重定向，方法不变
```

### 4.3 304 Not Modified

304 是 HTTP 缓存的核心机制：

```
工作流程：
1. 客户端首次请求，服务端返回 ETag 或 Last-Modified
2. 客户端缓存资源
3. 客户端再次请求，发送条件请求头
   If-None-Match: "etag-value"
   If-Modified-Since: "date"
4. 服务端判断资源是否变化
   - 未变化：返回 304 Not Modified
   - 已变化：返回 200 + 新内容
```

```java
// 服务端处理 304
public void handleIfNoneMatch(HttpRequest request, HttpResponse response) {
    String etag = request.getHeader("If-None-Match");
    if (etag != null && etag.equals(currentETag)) {
        response.setStatus(304);
        // 不返回 body，节省带宽
    } else {
        response.setStatus(200);
        response.setHeader("ETag", currentETag);
        response.setBody(content);
    }
}
```

## 五、4xx 客户端错误状态码

4xx 表示请求有错误，错误在客户端。

| 状态码 | 说明 |
|--------|------|
| 400 | Bad Request - 请求语法错误 |
| 401 | Unauthorized - 需要认证（未登录） |
| 403 | Forbidden - 拒绝访问（已登录但无权限） |
| 404 | Not Found - 资源不存在 |
| 405 | Method Not Allowed - 请求方法不支持 |
| 408 | Request Timeout - 请求超时 |
| 409 | Conflict - 资源冲突 |
| 413 | Payload Too Large - 请求体过大 |
| 414 | URI Too Long - URI 过长 |
| 422 | Unprocessable Entity - 语义错误 |
| 429 | Too Many Requests - 请求过于频繁 |

### 5.1 401 vs 403

| 状态码 | 含义 | 认证状态 |
|--------|------|----------|
| 401 | Unauthorized - 未认证 | 需要登录 |
| 403 | Forbidden - 已认证但无权限 | 已登录但权限不足 |

### 5.2 常见 API 错误码设计

```java
// RESTful API 错误码设计
public enum ErrorCode {
    // 4xx 客户端错误
    BAD_REQUEST(400, "请求参数错误"),
    UNAUTHORIZED(401, "未登录或登录已过期"),
    FORBIDDEN(403, "无权限访问该资源"),
    NOT_FOUND(404, "资源不存在"),
    METHOD_NOT_ALLOWED(405, "不支持该请求方法"),

    // 5xx 服务端错误
    INTERNAL_ERROR(500, "服务器内部错误"),
    SERVICE_UNAVAILABLE(503, "服务暂不可用");

    private final int code;
    private final String message;

    ErrorCode(int code, String message) {
        this.code = code;
        this.message = message;
    }
}
```

## 六、5xx 服务端错误状态码

5xx 表示服务器处理请求时发生错误。

| 状态码 | 说明 |
|--------|------|
| 500 | Internal Server Error - 服务器内部错误 |
| 501 | Not Implemented - 功能未实现 |
| 502 | Bad Gateway - 网关错误 |
| 503 | Service Unavailable - 服务不可用 |
| 504 | Gateway Timeout - 网关超时 |

### 6.1 502 vs 504

| 状态码 | 含义 | 典型场景 |
|--------|------|----------|
| 502 | Bad Gateway | 上游服务器返回错误响应 |
| 504 | Gateway Timeout | 上游服务器响应超时 |

```
502 典型场景：
Nginx → uWSGI/PHP-FPM
uWSGI 崩溃或返回错误响应 → Nginx 返回 502

504 典型场景：
Nginx → 后端服务
后端服务响应慢，超过超时时间 → Nginx 返回 504
```

## 七、面试题精讲

### 🔴 【高频】HTTP 状态码分类

**问题**：请说出 HTTP 状态码的分类。

**标准答案**：

```
HTTP 状态码分为 5 类：

1xx（信息性）：请求已接收，需要继续处理
   - 100 Continue：客户端继续发送请求

2xx（成功）：请求成功
   - 200 OK：成功
   - 201 Created：资源创建成功
   - 204 No Content：成功但无返回内容

3xx（重定向）：需要进一步操作
   - 301 Moved Permanently：永久重定向
   - 302 Found：临时重定向
   - 304 Not Modified：使用缓存

4xx（客户端错误）：请求有问题
   - 400 Bad Request：请求语法错误
   - 401 Unauthorized：需要认证
   - 403 Forbidden：拒绝访问
   - 404 Not Found：资源不存在

5xx（服务端错误）：服务器问题
   - 500 Internal Server Error：服务器内部错误
   - 502 Bad Gateway：网关错误
   - 503 Service Unavailable：服务不可用
```

### 🟡 【中频】301 和 302 的区别

**问题**：301 和 302 有什么区别？

**标准答案**：

```
1. 语义不同：
   - 301：永久移动，URL 不会再使用
   - 302：临时移动，URL 以后还会用

2. 缓存行为：
   - 301：会被浏览器缓存
   - 302：不会被浏览器缓存

3. SEO 影响：
   - 301：传递页面权重到新 URL
   - 302（旧行为）：不传递权重（但搜索引擎会识别）
   - 302（新行为）：传递权重

4. 使用场景：
   - 301：域名变更、内容迁移到新 URL
   - 302：临时维护、A/B 测试、登录重定向
```

### 🟡 【中频】304 的作用

**问题**：304 Not Modified 是什么？有什么作用？

**标准答案**：

```
304 Not Modified 是 HTTP 缓存的核心机制：

作用：
- 告诉客户端使用缓存，不需要传输内容
- 节省带宽，提升性能

工作原理：
1. 首次请求，服务端返回 ETag 或 Last-Modified
2. 客户端缓存，带条件请求头再次请求
3. 服务端判断资源是否变化
   - 未变化：返回 304（无 body）
   - 已变化：返回 200 + 新内容

条件请求头：
- If-None-Match：匹配 ETag
- If-Modified-Since：匹配 Last-Modified
```

## 八、常见陷阱与易错点

### ⚠️ 陷阱一：混淆 301/302 和 307/308

| 状态码 | 永久/临时 | 方法改变 |
|--------|-----------|----------|
| 301 | 永久 | 可能改变 |
| 302 | 临时 | 可能改变 |
| 307 | 临时 | 不改变 |
| 308 | 永久 | 不改变 |

### ⚠️ 陷阱二：混淆 401 和 403

- **401**：未认证（没登录）
- **403**：已认证但无权限（登录了但权限不够）

### ⚠️ 陷阱三：忽略 304 的性能价值

304 不返回 body，但 HTTP 请求仍然存在。真正的「减少请求」需要使用强缓存（Cache-Control）。

### ⚠️ 陷阱四：502 和 504 混淆

- **502**：上游服务器返回了错误响应
- **504**：上游服务器响应超时

## 九、对比总结

### 3xx 状态码对比

| 状态码 | 含义 | 方法改变 | 缓存 |
|--------|------|----------|------|
| 301 | 永久重定向 | 可能改变 | 缓存 |
| 302 | 临时重定向 | 可能改变 | 不缓存 |
| 303 | POST→GET | 强制 GET | 不缓存 |
| 307 | 临时重定向 | 不改变 | 不缓存 |
| 308 | 永久重定向 | 不改变 | 缓存 |

### 4xx 状态码对比

| 状态码 | 含义 | 是否认证 |
|--------|------|----------|
| 400 | 请求错误 | 不涉及 |
| 401 | 未认证 | 未登录 |
| 403 | 无权限 | 已登录 |

### 5xx 服务端错误

| 状态码 | 含义 |
|--------|------|
| 500 | 服务器内部错误 |
| 502 | 网关错误（上游服务异常） |
| 503 | 服务不可用（过载/维护） |
| 504 | 网关超时（上游服务响应慢） |

## 十、扩展思考

### 💡 加分话题：HTTP 103 状态码

HTTP 103 是新引入的状态码，用于 Early Hints：

```html
<!-- 服务端先返回提示 -->
HTTP/1.1 103 Early Hints
Link: </style.css>; rel=preload; as=style
Link: </script.js>; rel=preload; as=script

<!-- 然后返回正式响应 -->
HTTP/1.1 200 OK
```

### 💡 加分话题：自定义 4xx/5xx 错误页面

```nginx
# Nginx 自定义错误页面
error_page 404 /404.html;
error_page 500 502 503 504 /50x.html;
```

### 💡 加分话题：HTTP 状态码与 RESTful API 设计

```
RESTful 状态码规范：
- 200 OK：成功
- 201 Created：资源创建
- 204 No Content：成功删除
- 400 Bad Request：参数错误
- 401 Unauthorized：未认证
- 403 Forbidden：无权限
- 404 Not Found：资源不存在
- 409 Conflict：资源冲突
- 500 Internal Server Error：服务器错误
```

> HTTP 状态码是 API 设计的门面。合理使用状态码能让客户端快速了解请求结果，是良好 API 设计的基础。记住常见的 2xx、3xx、4xx、5xx 状态码及其含义，是每个后端开发者的基本功。
