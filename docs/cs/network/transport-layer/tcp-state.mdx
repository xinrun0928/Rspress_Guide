# TCP 状态转换图与 TIME_WAIT

如果你用 `netstat -ant` 查看服务器的网络状态，会看到一堆陌生的状态名：`SYN_SENT`、`FIN_WAIT_1`、`CLOSING`、`TIME_WAIT`……

理解这些状态的含义，是排查网络问题的基础。

## TCP 状态一览

TCP 连接从建立到关闭，会经历以下状态：

| 状态 | 含义 |
|---|---|
| `CLOSED` | 连接关闭，资源已释放 |
| `LISTEN` | 服务器等待连接请求 |
| `SYN_SENT` | 客户端已发送 SYN，等待确认 |
| `SYN_RCVD` | 服务器收到 SYN，等待最后的 ACK |
| `ESTABLISHED` | 连接已建立，双方可以传输数据 |
| `FIN_WAIT_1` | 主动关闭，等待对方的 ACK |
| `FIN_WAIT_2` | 收到 ACK，进入等待对方 FIN |
| `CLOSE_WAIT` | 被动关闭方收到 FIN，等待应用程序关闭 |
| `CLOSING` | 双方同时关闭，等待对方 ACK |
| `LAST_ACK` | 被动关闭方发送 FIN，等待最后的 ACK |
| `TIME_WAIT` | 主动关闭方等待 2MSL 后关闭 |

## 完整状态转换图

### 客户端状态转换

```
CLOSED
   | 主动打开
   v
SYN_SENT -----(超时)-----> CLOSED
   | 收到 SYN + ACK
   v
ESTABLISHED
   | 主动关闭
   v
FIN_WAIT_1 -----(超时)-----> CLOSED
   | 收到 ACK
   v
FIN_WAIT_2
   | 收到 FIN + ACK
   |-------------> TIME_WAIT -----> CLOSED
   | 收到 FIN（对方也想关闭）
   v
CLOSING -----(收到 ACK)-----> TIME_WAIT -----> CLOSED
```

### 服务器状态转换

```
CLOSED
   | 被动打开
   v
LISTEN
   | 收到 SYN
   v
SYN_RCVD -----(超时)-----> LISTEN
   | 收到 ACK
   v
ESTABLISHED
   | 收到 FIN
   v
CLOSE_WAIT
   | 应用程序关闭
   v
LAST_ACK -----(收到 ACK)-----> CLOSED
```

## TIME_WAIT：最重要的状态

TIME_WAIT 是面试中的高频考点，也是生产环境常见的问题来源。

### 什么是 TIME_WAIT

当主动关闭方发送最后一个 ACK 后，连接进入 TIME_WAIT 状态，保持 2MSL（Maximum Segment Lifetime）时间后才进入 CLOSED 状态。

MSL 是报文在网络中的最大生存时间，Linux 中通常设置为 60 秒。

### TIME_WAIT 的两个作用

**作用一：保证最后的 ACK 能到达对方**

如果第四次挥手的 ACK 丢失，被动关闭方会重传 FIN。TIME_WAIT 确保在这个重传窗口内，主动关闭方仍然在线，可以处理重传。

```
主动关闭方                            被动关闭方
    |                                    |
    |------- FIN ----------------------->|
    |<------ ACK -----------------------|
    |<------ FIN -----------------------|
    |------- ACK ---> 丢失              |
    |                                    |
    |   TIME_WAIT（等待重传的 FIN）       |
    |<------ FIN（重传）-----------------|  如果没有 TIME_WAIT
    |   无响应，连接无法正确关闭           |  服务器无法收到 ACK
```

**作用二：让旧连接的报文消亡**

在等待期间，旧连接的所有报文都会在网络中消失，不会干扰新的连接。

```
同一端口的旧连接数据                 新连接
    |                               |
    |<======= 旧数据在网络中 =========|
    |                               |
    |   TIME_WAIT（等待旧数据消失）    |
    |                               |
    |       假设没有 TIME_WAIT        |
    |                               |
    |---- 新连接建立 --------------->|
    |<---- 旧数据干扰新连接 ----------|  混乱！
```

## TIME_WAIT 引发的问题

### 问题一：端口耗尽

在高并发短连接场景下，大量 TIME_WAIT 连接会占用端口。Linux 默认的端口范围是 50000-65535，共 15535 个端口。

如果频繁创建短连接：
- 每秒 1000 个请求 = 1000 个 TIME_WAIT
- 2MSL = 120 秒内，约 120000 个 TIME_WAIT
- 端口耗尽，无法建立新连接

### 问题二：资源浪费

每个 TIME_WAIT 连接都会占用：
- 内存（socket 缓冲区）
- 文件描述符
- 连接跟踪表条目

## 解决方案

### 方案一：调整内核参数

```bash
# 允许复用 TIME_WAIT 连接（客户端）
echo 1 > /proc/sys/net/ipv4/tcp_tw_reuse

# 加快 TIME_WAIT 回收（已废弃，不推荐）
echo 1 > /proc/sys/net/ipv4/tcp_tw_recycle

# TIME_WAIT 超时时间（秒）
echo 30 > /proc/sys/net/ipv4/tcp_fin_timeout
```

### 方案二：使用 SO_REUSEADDR

服务器程序应设置 `SO_REUSEADDR` 选项，允许绑定到处于 TIME_WAIT 状态的地址。

```c
int opt = 1;
setsockopt(sockfd, SOL_SOCKET, SO_REUSEADDR, &opt, sizeof(opt));
```

### 方案三：使用长连接

减少连接的建立和关闭次数，从根本上减少 TIME_WAIT 的产生。

```python
import requests

# 短连接：每次请求都建立新连接
requests.get('http://example.com/api')

# 长连接：复用连接
session = requests.Session()
for i in range(1000):
    session.get('http://example.com/api')  # 复用同一个连接
```

### 方案四：调整端口范围

```bash
# 扩大可用端口范围
echo "1024 65535" > /proc/sys/net/ipv4/ip_local_port_range
```

## CLOSE_WAIT 详解

CLOSE_WAIT 是另一个常见问题。当服务器收到客户端的 FIN 后，如果应用程序没有调用 `close()`，连接就会停留在 CLOSE_WAIT 状态。

### 常见原因

```python
# 原因一：忘记关闭连接
def handle_request(sock):
    data = sock.recv(1024)
    response = process(data)
    sock.sendall(response)
    # 忘记 sock.close()！

# 原因二：异常处理不当
def handle_request(sock):
    try:
        data = sock.recv(1024)
        response = process(data)
        sock.sendall(response)
    except Exception as e:
        log(e)
        # 异常时没有关闭连接！
    finally:
        sock.close()  # 应该放在这里

# 原因三：连接池泄漏
pool = ConnectionPool()

def handle_request():
    conn = pool.get()
    data = conn.recv(1024)
    # 处理...
    # 忘记 conn.close() 或 pool.release(conn)
```

### 排查方法

```bash
# 查看 CLOSE_WAIT 状态
netstat -anp | grep CLOSE_WAIT

# 统计各状态数量
netstat -ant | awk '{print $6}' | sort | uniq -c

# 查看是哪个进程
ss -ant | grep CLOSE_WAIT
```

## FIN_WAIT_2 与 CLOSING

### FIN_WAIT_2

主动关闭方发送 FIN 并收到 ACK 后，进入 FIN_WAIT_2 状态。如果被动关闭方迟迟不发送 FIN，主动关闭方会永远等下去。

Linux 提供了 `tcp_fin_timeout` 参数来控制 FIN_WAIT_2 的超时时间（默认 60 秒）。

### CLOSING

这是一种"双方同时关闭"的特殊状态。当主动关闭方发送 FIN 后，收到了对方的 FIN（而不是 ACK），就进入 CLOSING 状态。

```
主动关闭方                            被动关闭方
    |                                    |
    |------- FIN ----------------------->|
    |<------ FIN（同时发送）-------------|
    |------- ACK ----------------------->|  通常 ACK 先到
    |                                    |
    |          CLOSING                   |
    |<------ ACK ------------------------|  收到 ACK 后进入 TIME_WAIT
```

## 面试追问

### TIME_WAIT 状态过多怎么解决？

1. 客户端：启用 `tcp_tw_reuse`
2. 服务端：使用 `SO_REUSEADDR`，调整 `tcp_fin_timeout`
3. 架构层面：减少短连接，使用连接池或 HTTP 长连接

### 为什么是 2MSL 而不是 1MSL 或 3MSL？

1MSL 可能不够——如果 ACK 丢失，服务器重传 FIN 的时间可能超过 1MSL。3MSL 增加了无谓的等待时间。2MSL 是权衡后的最优值。

### 有人说 TIME_WAIT 是"好东西"，为什么？

是的，TIME_WAIT 是 TCP 协议安全性的保障。它确保旧连接的报文不会干扰新连接。问题是，在高并发场景下，它的副作用（资源占用）超过了收益。这才需要优化。

## 总结

TCP 状态转换图揭示了连接生命周期的完整轨迹：

- `LISTEN` -> `SYN_RCVD` -> `ESTABLISHED`：三次握手的轨迹
- `ESTABLISHED` -> `FIN_WAIT_1` -> `FIN_WAIT_2` -> `TIME_WAIT`：主动关闭的轨迹
- `TIME_WAIT` -> `CLOSED`：2MSL 后彻底关闭

理解这些状态，不仅能帮你排查网络问题，更能理解 TCP 协议设计的精妙——每一个状态都有其存在的意义。
