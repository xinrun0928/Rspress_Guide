# TCP 滑动窗口与流量控制

想象一下：你往邮箱里塞了一堆信，但邮递员每次只能送一定数量的信件。太多信件会让他崩溃，太少又会效率低下。

TCP 的滑动窗口，就是这样一个"邮递员管理机制"——它控制发送方一次能发送多少数据，确保接收方不会因为"消化不了"而崩溃。

## 为什么需要流量控制

TCP 是全双工通信，双方各自维护发送缓冲区和接收缓冲区。

```
发送方                            接收方
  |                                |
  |        发送缓冲区                |        接收缓冲区
  |        (send buffer)           |        (recv buffer)
  |                                |
```

问题来了：
- 发送方发送得太快，接收方处理不过来
- 接收缓冲区满后，新数据只能被丢弃
- 被丢弃的数据需要重传，浪费网络资源

**流量控制**就是让发送方根据接收方的处理能力，动态调整发送速率。

## 滑动窗口的结构

TCP 头部中有一个 16 位的窗口字段（Window Size），表示接收方的接收缓冲区剩余空间。

滑动窗口将数据分为四类：

```
|-----------------|--------------------|---------------------|---------------------|
|    已发送已确认    |    已发送未确认     |    未发送可发送     |   未发送不可发送    |
|   (sent & ack)   |  (sent & not ack)  |  (not sent & 可发)  |  (not sent & 不可发) |
|-----------------|--------------------|---------------------|---------------------|
        ^                         ^                        ^
       SND.UNA                  SND.NXT                SND.UNA + SND.WND
       
       窗口左沿                  窗口右沿
```

| 变量 | 含义 |
|---|---|
| `SND.UNA` | 已发送未确认的第一个字节序号 |
| `SND.NXT` | 未发送的下一个字节序号（窗口右沿） |
| `SND.WND` | 窗口大小 |
| `SND.UNA + SND.WND` | 窗口右沿的最大位置 |

## 滑动窗口如何"滑动"

当收到 ACK 时，窗口会向前滑动：

```
初始状态：
|-------XXXXX------|------------------------|
      已发送已确认      窗口内可发送              不可发送

收到 ACK，确认了 3 个字节：
|------------XXXXX---|------------------------|
        已发送已确认      窗口内可发送              不可发送
                    ^
                  窗口滑动
```

发送方可以发送窗口内的数据，而不需要等待之前的 ACK。

## 流量控制的实现

接收方在 ACK 中携带窗口大小，告诉发送方"你还能发多少"。

```
接收方缓冲区状态：
|------------------------|------------------------|
|     已接收已确认         |      空闲空间           |     满（不可接收）
|                        |      (窗口大小)          |
|------------------------|------------------------|
         RCV.NXT                  ^
                                  窗口左沿
```

### 具体流程

```
发送方                            接收方
  |                                |
  |---- 数据 seq=1, len=100 ----->|
  |                                |  接收缓冲区剩余 500
  |<------- ACK ack=101, win=500 --|  告诉发送方：还能发 500
  |                                |
  |---- 数据 seq=101, len=200 --->|
  |                                |  接收缓冲区剩余 300
  |<------- ACK ack=301, win=300 --|
  |                                |
  |---- 数据 seq=301, len=300 --->|  刚好在窗口范围内
  |                                |  接收缓冲区剩余 0
  |<------- ACK ack=601, win=0 ----|  窗口为 0，不能再发了
```

当接收方窗口为 0 时，发送方不能再发送数据，这叫做**零窗口**。

## 零窗口与窗口探测

当接收方缓冲区满了，窗口变为 0，发送方停止发送。但发送方怎么知道接收方什么时候能恢复？

答案是**窗口探测（Window Probe）**：

```
发送方                            接收方
  |                                |
  |<------- ACK win=0 ------------|  窗口为 0
  |                                |
  |  等待...                       |  应用程序读取数据
  |                                |
  |---- 探测报文 len=1 ----------->|  发送 1 字节探测
  |                                |
  |<------- ACK ack=xxx, win=200 --|  窗口恢复
```

Linux 中，窗口探测间隔从 30 秒开始，之后指数增长。

:::tip 零窗口问题
如果接收方的窗口一直为 0，发送方会持续发送探测报文。如果接收方最终没有恢复，连接可能会超时断开。
:::

## 糊涂窗口综合症

如果接收方每次只读取少量数据就更新窗口，可能导致一个有趣的问题：**每次只发送几个字节，网络效率极低**。

```
接收方                              发送方
  |                                  |
  |  收到 1 字节，窗口更新为 1 ----->|
  |                                  | 发送 1 字节
  |  收到 1 字节，窗口更新为 1 ----->|
  |                                  | 发送 1 字节
  |  ...                             |
```

这叫做**糊涂窗口综合症（Silly Window Syndrome）**。

### 解决方案：Nagle 算法（发送端）

发送端收集小数据，凑够一定量再发送：

```python
# Nagle 算法简化
if 有待确认数据（未 ACK）:
    # 等待 ACK 后再发送
    收集新数据到缓冲区
else:
    if 数据量 >= MSS（最大段大小）:
        立即发送
    else:
        等待一段时间
```

### 解决方案：延迟 ACK（接收端）

接收端不要每次收到数据都立即 ACK，而是等一下，凑够一批数据再 ACK：

```bash
# Linux 延迟 ACK 参数
cat /proc/sys/net/ipv4/tcp_delack_seg
```

## 窗口缩放（Window Scaling）

16 位的窗口字段最大只能表示 65535 字节。但现代网络中，高带宽延迟积（BDP）的连接需要更大的窗口：

```
BDP = 带宽 * RTT
例如：1 Gbps, 100ms RTT
BDP = 1 Gbps * 0.1s = 100 Mb = 12.5 MB
```

65535 字节远远不够。

TCP 引入了**窗口缩放选项（Window Scaling）**：

```bash
# 在三次握手时协商缩放因子
SYN: 窗口缩放因子 = 3  (实际窗口 = 声明窗口 * 2^3)
SYN+ACK: 窗口缩放因子 = 3
ACK: 确认
```

缩放因子可以是 0~14，最大窗口 = 65535 * 2^14 ≈ 1 GB。

```bash
# 查看窗口缩放配置
cat /proc/sys/net/ipv4/tcp_window_scaling
# 查看当前窗口大小
cat /proc/sys/net/ipv4/tcp_rmem
```

## 抓包分析滑动窗口

在 Wireshark 中观察 TCP 窗口：

```
No.  Time       Src -> Dst          Seq    Ack    Window
1    0.000      Client -> Server    1      1      65535
2    0.001      Server -> Client    1      101    65535
3    0.001      Client -> Server    101    101    65535
...
```

Window 字段变化反映接收方的处理能力。当 Window 持续下降，说明接收方在放慢；当 Window 为 0，说明接收方缓冲区满了。

## 面试追问

### 滑动窗口和拥塞控制有什么区别？

滑动窗口控制发送速率，防止压垮接收方。拥塞控制控制发送速率，防止压垮网络。两者的目标不同，但都会限制发送方的发送速率。

### 窗口大小为 0 时，发送方还能发送什么？

只能发送**窗口探测报文**（1 字节的数据，触发 ACK 更新窗口）。正常数据必须等到窗口恢复后才能发送。

### TCP 的发送缓冲区有什么作用？

发送缓冲区存放已发送但未 ACK 的数据（用于重传），以及已发送但等待发送的数据。接收缓冲区存放已接收但未读取的数据。

### 如果网络延迟很高，需要什么配置？

需要启用窗口缩放（`tcp_window_scaling = 1`）并增大窗口大小。高延迟网络中，BDP 很大，小窗口会成为瓶颈。

## 总结

滑动窗口是 TCP 流量控制的核心：

- **窗口左沿**：已发送已确认的数据边界
- **窗口右沿**：允许发送的最大位置（由接收方窗口大小决定）
- **滑动**：收到 ACK 后，窗口向前滑动，允许发送更多数据

滑动窗口让 TCP 能：
- 在不等待 ACK 的情况下发送多个数据（流水线）
- 根据接收方能力动态调整发送速率（流量控制）
- 通过窗口缩放支持高速网络

理解滑动窗口，你才能理解为什么 TCP 既高效又可靠。
