# 为什么是四次挥手

如果你理解了三次握手为什么是三次，那么四次挥手可能让你困惑：**断开连接为什么需要四次？建立连接需要三次，关闭连接为什么更多？**

这个问题的答案，藏在 TCP 全双工通信的本质中。

## 全双工意味着双向独立

TCP 是全双工协议——通信双方可以同时发送和接收数据。这就像一条双向高速公路：两个方向的车流是独立的。

建立连接时，双方的目标一致："我们开始通信吧"。所以可以合并交互。

断开连接时，情况不同了：
- A 说："我没有数据要发了"
- B 收到后，可能还在处理 A 发来的数据
- B 处理完后，说："我也没有数据要发了"
- A 收到后，确认断开

这就需要四次交互。

## 为什么不能合并

有人说：把 ACK 和 FIN 合并成一次不行吗？

技术上可以。**当服务器没有数据要发送给客户端时，第二次和第三次挥手可以合并**，变成三次挥手。

但这只是一种**优化**，不是 TCP 协议的默认行为。TCP 设计者选择四次挥手作为标准流程，是因为：

### 合并是例外，不合并是常态

在实际应用中，服务器收到客户端的 FIN 时，往往还有未处理完的数据。这些数据需要发送完毕后才能关闭连接。

```python
# 伪代码示例
def handle_request(sock):
    data = sock.recv(1024)
    # 处理数据...
    response = process(data)
    sock.sendall(response)
    # 数据发完了，现在可以关闭
    sock.close()
```

### 关闭时机不同

主动关闭方说"我要关闭"，是被动关闭方的第一次通知。但被动关闭方可能：
- 数据还没处理完
- 数据处理完了但还没发完响应
- 发完响应了

如果合并成三次，意味着被动关闭方在说"我要关闭"的同时，也要决定"我有没有数据要发"——这往往是不可能的，因为数据还在缓冲区里。

## 四次挥手的具体场景

### 场景一：服务器有数据要发送

```
客户端                                    服务器
  |                                        |
  |<====== 数据传输 ==========>|  双向通信中
  |                                        |
  |------- FIN ---------------->|  客户端：我发完了
  |<------ ACK -----------------|  服务器：收到，我可能还有数据
  |                                        |
  |<------ 继续发数据 -------|  服务器处理并发送剩余数据
  |<------ FIN ----------------|  服务器：我也发完了
  |------- ACK ---------------->|  客户端：收到，断开
```

这是最常见的场景。四次挥手确保服务器有足够时间处理和发送剩余数据。

### 场景二：服务器立即关闭（可优化为三次）

```
客户端                                    服务器
  |                                        |
  |<====== 数据传输 ==========>|  双向通信中
  |                                        |
  |------- FIN ---------------->|  客户端：我发完了
  |<------ FIN + ACK ----------|  服务器：我发完了且确认你
  |------- ACK ---------------->|  客户端：收到，断开
```

在 HTTP/1.0 中，服务器通常在发完响应后立即关闭连接，此时可以合并 FIN + ACK。

## TIME_WAIT 状态的必要性

四次挥手后，主动关闭方会进入 TIME_WAIT 状态，等待 2MSL。

为什么要等这么久？

### 原因一：处理 ACK 丢失

```
客户端                                    服务器
  |                                        |
  |------- FIN ---------------->|  客户端：我要关闭
  |<------ ACK -----------------|  服务器：收到
  |<------ FIN -----------------|  服务器：我也关闭
  |------- ACK ---------------->|  客户端：收到
  |                                        |
  X<------ 最后一个 ACK 丢失 ----|  网络问题
  |------- ACK ---------------->|  客户端重传
```

如果没有 TIME_WAIT，服务器可能已经关闭了端口，收到重传的 FIN 后无法响应。

### 原因二：让旧连接的报文消亡

```
客户端                                    服务器
  |                                        |
  |<====== 旧连接 ============>|
  |                                        |
  |------- FIN ---------------->|  旧连接关闭
  |<------ ACK -----------------|
  |<------ FIN -----------------|
  |------- ACK ---------------->|
  |                                        |
  |       （短暂等待）                      |
  |                                        |
  |------- SYN ---------------->|  新连接？旧数据？
```

如果主动关闭方立即释放端口并发起新连接，旧连接中还在网络上游荡的报文可能干扰新连接。等待 2MSL 可以确保这些旧报文在网络中消失。

## 常见误解

### 误解一：四次挥手一定比三次慢

不是。如果服务器处理数据很快，第二次和第三次挥手可以合并，实质上变成三次。在高并发短连接场景中，这是常见的优化。

### 误解二：TIME_WAIT 只影响主动关闭方

是的，只有主动关闭方会进入 TIME_WAIT。这是 TCP 协议设计的一部分——一方主动发起关闭，另一方被动接受。

### 误解三：TIME_WAIT 会导致端口耗尽

在高并发场景下，大量 TIME_WAIT 连接会占用端口。如果端口范围是 50000-65535，大约只有 15000 个端口可用。

解决方案：
- `tcp_tw_reuse = 1`：允许新连接复用 TIME_WAIT 连接
- `tcp_tw_recycle = 1`（已废弃）：加快 TIME_WAIT 回收
- 服务端使用长连接，减少主动关闭次数

## 抓包验证四次挥手

用 Wireshark 抓包，观察一个 HTTP 连接的关闭：

```
192.168.1.100    52345 -> 443           [FIN, ACK]  Seq=xxx, Ack=xxx
192.168.1.100    443 -> 52345           [ACK]        Seq=xxx, Ack=xxx+1
192.168.1.100    443 -> 52345           [FIN, ACK]  Seq=xxx, Ack=xxx+1
192.168.1.100    52345 -> 443           [ACK]        Seq=xxx, Ack=xxx+1
```

注意第三次挥手和第四次挥手之间的间隔——这通常是服务器处理剩余数据的时间。

## 面试追问

### 四次挥手能不能变成三次？

能。当前处于 TIME_WAIT 状态的主动关闭方会进入 2MSL 等待，而被动关闭方收到 FIN 后如果没有数据要发送，可以把 ACK 和 FIN 合并发送，这样就变成三次挥手了。

### 服务器出现大量 CLOSE_WAIT 怎么处理？

CLOSE_WAIT 说明服务器收到了客户端的 FIN，但应用程序没有调用 `close()`。排查步骤：
1. 用 `netstat -anp | grep CLOSE_WAIT` 找到是哪些进程
2. 检查代码中 socket 的关闭逻辑
3. 检查是否有阻塞 I/O 导致 close() 没有被调用

### 如果主动关闭方崩溃了怎么办？

被动关闭方会持续等待。如果配置了 TCP Keepalive，2 小时无响应后会发送探测报文，最终判定连接已断开。

## 总结

四次挥手不是"多余"，而是 TCP 全双工通信的必要设计：

- 第一次挥手：主动方说"我没数据了"
- 第二次挥手：被动方说"收到，我可能有数据要发"
- 第三次挥手：被动方说"我也没数据了"
- 第四次挥手：主动方说"收到，2MSL 后彻底再见"

这背后是对"有序释放资源"的追求——确保双方都没有未完成的数据处理，连接才能安全关闭。
