# TCP 四次挥手

当你在浏览器中关闭一个网页时，底层的 TCP 连接正在经历一场"优雅的告别"。

不同于三次握手的"建立"，四次挥手是"有序的拆除"。为什么断开连接比建立连接更复杂？因为建立连接时，双方的目标一致（尽快建立通信）；而断开连接时，需要确保双方都没有数据要发送了。

## 为什么要四次挥手

TCP 是全双工协议——数据可以同时在两个方向上传输。这意味着，关闭连接需要**双向分别关闭**：

1. 主动关闭方：告诉被动关闭方"我没有数据要发了"
2. 被动关闭方：确认收到，发送完剩余数据后，也说"我也没有数据要发了"
3. 主动关闭方：确认收到

这就需要四次交互。

## 挥手的完整流程

### 第一次挥手：FIN

当客户端主动关闭连接时，发送 `FIN = 1`，`seq = u`。

这里 `u` 是什么？它是客户端最后发送的数据字节序号加 1，俗称"FIN 前的一个字节的序号"。

发送完这个报文后，客户端进入 `FIN_WAIT_1` 状态。

### 第二次挥手：ACK

服务器收到客户端的 FIN 报文后，发送 `ACK = 1`，`ack = u + 1`。

这表示"我收到了你的结束请求，但我可能还有数据要发给你"。

发送完这个报文后，服务器进入 `CLOSE_WAIT` 状态。客户端收到这个 ACK 后，进入 `FIN_WAIT_2` 状态。

### 第三次挥手：FIN

服务器处理完剩余数据后，发送 `FIN = 1`，`seq = v`。

这里的 `v` 是服务器最后发送的数据字节序号加 1。

发送完这个报文后，服务器进入 `LAST_ACK` 状态。

### 第四次挥手：ACK

客户端收到服务器的 FIN 报文后，发送 `ACK = 1`，`ack = v + 1`。

客户端进入 `TIME_WAIT` 状态，等待 2MSL 后进入 `CLOSED` 状态。服务器收到这个 ACK 后，直接进入 `CLOSED` 状态。

## 状态转换图

```
客户端                                    服务器
  |                                        |
  |------- FIN, seq=u, FIN_WAIT_1 ------> |  客户端: FIN_WAIT_1
  |<------ ACK, ack=u+1, CLOSE_WAIT ----- |  服务器: CLOSE_WAIT
  |  客户端: FIN_WAIT_2                   |
  |                                        |
  |<------ FIN, seq=v, LAST_ACK --------- |  服务器: LAST_ACK
  |  客户端: TIME_WAIT                     |
  |                                        |
  |------- ACK, ack=v+1 ----------------->|  服务器: CLOSE
  |                                        |
  |  等待 2MSL                             |
  |  客户端: CLOSED                        |
```

## 为什么要等待 2MSL

`TIME_WAIT` 状态是四次挥手中最容易让人困惑的设计。为什么要等待 2MSL（Maximum Segment Lifetime，最大报文生存时间）？

主要有两个原因：

### 原因一：确保最后的 ACK 能到达对方

如果第四次挥手的 ACK 报文丢失，服务器会重传 FIN 报文。客户端需要在 `TIME_WAIT` 期间处理这个重传。

MSL 是报文在网络中存在的最长时间，通常为 60 秒。等待 2MSL 确保：
- 1MSL：ACK 报文从客户端到服务器的时间
- 1MSL：FIN 报文从服务器到客户端的时间（如果 ACK 丢失）

### 原因二：让旧连接的报文在网络中消失

如果客户端立即关闭并用相同端口发起新连接，旧连接的报文可能会干扰新连接。等待 2MSL 确保网络中属于旧连接的报文都已消失。

:::tip MSL 的实际值
Linux 中 MSL 默认是 60 秒（由 `tcp_fin_timeout` 控制，实际行为略有不同）。但 2MSL 等待时间由 `net.ipv4.tcp_fin_timeout` 控制，默认是 60 秒。
:::

## CLOSE_WAIT 与 LAST_ACK

这两个状态在挥手过程中扮演重要角色，但也很容易引发问题。

### CLOSE_WAIT 过多的原因

服务器收到客户端的 FIN 后进入 `CLOSE_WAIT` 状态。如果这个状态持续增长，说明**应用程序没有调用 `close()` 关闭 socket**。

常见原因：
- 代码中存在连接泄漏（创建 socket 但未关闭）
- 应用程序处理逻辑有问题，迟迟不关闭连接
- 对方发送 FIN 后，服务器忙于处理数据

### LAST_ACK 过多的原因

服务器进入 `LAST_ACK` 后，等待客户端的最终 ACK。如果这个状态持续增长，可能是：
- 网络问题导致 ACK 丢失
- 客户端崩溃或异常

## 半关闭状态

TCP 提供了 `shutdown()` 函数实现"半关闭"——只关闭一个方向的连接。

```
客户端                  服务器
  |                        |
  |<====== 数据传输 ======>|
  |                        |
  |--- FIN (只关闭写) ----->|  客户端不再发送数据，但仍可接收
  |<-- 剩余数据陆续发回 ----|
  |<--- FIN (服务器也关闭)--|
  |--- ACK ---------------->|  挥手完成
```

这种场景常见于 HTTP/1.1 的 `Connection: keep-alive`，或者某些协议需要先发送请求再接收响应。

## 抓包分析四次挥手

用 Wireshark 抓包时，可以看到典型的四次挥手：

```
192.168.1.100    443 -> 52345    [ACK]   Ack=xxx, seq=xxx
192.168.1.100    443 -> 52345    [FIN, ACK]  Ack=xxx, seq=xxx
52345 -> 443     [ACK]   Ack=xxx+1, seq=xxx
```

:::tip Wireshark 过滤
用 `tcp.flags.fin == 1` 过滤 FIN 报文，`tcp.flags.ack == 1 && tcp.flags.fin == 0` 过滤纯 ACK。
:::

## 快速关闭与保活机制

### 快速关闭（SO_LINGER）

默认情况下，调用 `close()` 会立即返回，数据会进入发送缓冲区。但如果设置 `SO_LINGER` 并指定超时时间，关闭会等待数据发送完毕或超时。

```c
struct linger ling;
ling.l_onoff = 1;
ling.l_linger = 5;  // 等待 5 秒
setsockopt(sockfd, SOL_SOCKET, SO_LINGER, &ling, sizeof(ling));
close(sockfd);
```

### 保活机制（TCP Keepalive）

如果双方建立了连接但长时间没有数据交换，TCP 保活机制会定期发送探测报文：
- 默认 2 小时无数据后发送第一次探测
- 每 75 秒探测一次，共探测 9 次
- 如果对方不响应，连接被认为已断开

```bash
# Linux 查看 TCP 保活参数
cat /proc/sys/net/ipv4/tcp_keepalive_time
cat /proc/sys/net/ipv4/tcp_keepalive_intvl
cat /proc/sys/net/ipv4/tcp_keepalive_probes
```

## 面试追问

### 如果 TIME_WAIT 状态的连接过多怎么办？

高并发场景下，大量主动关闭的连接会积累在 TIME_WAIT 状态，消耗端口资源。解决方案：
- 调整 `net.ipv4.tcp_tw_reuse = 1`，允许复用 TIME_WAIT 连接
- 调整 `net.ipv4.tcp_tw_recycle = 1`，加快 TIME_WAIT 回收
- 服务端尽量使用长连接，减少主动关闭的次数
- 使用 `SO_REUSEADDR` 允许绑定到处于 TIME_WAIT 的地址

### 四次挥手能否变成三次？

可以。当服务器收到客户端的 FIN 后，如果没有数据要发送，可以把 ACK 和 FIN 合并发送，变成三次挥手。这在某些场景下是优化手段。

### CLOSE_WAIT 一直不消失怎么排查？

检查应用程序代码：
- 确认所有 socket 在使用完毕后都有 `close()` 调用
- 检查连接池是否正确释放
- 查看进程的网络连接状态：`netstat -anp | grep CLOSE_WAIT`

## 总结

四次挥手是 TCP 连接优雅终止的设计：

- 第一次挥手：主动关闭方声明"我要关闭"
- 第二次挥手：被动方确认"收到，等待数据发完"
- 第三次挥手：被动方发完数据后"我也关闭了"
- 第四次挥手：主动方确认"收到，2MSL 后彻底再见"

理解这些状态，你就能在排查连接问题时快速定位根因。下一次看到服务器上有大量 `CLOSE_WAIT` 状态时，你会知道：该检查代码了。
