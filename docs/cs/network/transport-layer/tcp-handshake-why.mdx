# 为什么是三次握手

"TCP 三次握手"是面试中的经典问题。但大部分人的回答是背诵——"第一次客户端发送 SYN，第二次服务器发送 SYN + ACK，第三次客户端发送 ACK"。

如果你能回答下面的追问，才是真正理解了：三次握手是为了确认什么？为什么不能是两次？为什么不需要四次？

## 三次握手要解决的核心问题

TCP 协议的核心目标是什么？是**可靠传输**。

可靠传输的前提是什么？是**双方都具备收发能力**。

三次握手，本质上是一次**能力交换与确认**的过程。

### 交换能力的必要性

假设只建立一次握手（一次请求-响应）：

```
客户端                                    服务器
  |                                        |
  |----------- SYN, ISN=x ---------------->|  客户端：我可以发送
  |                                        |
  |<============= 数据传输 ===============|  服务器：我收到并可以发送
```

问题来了：
1. **服务器不知道自己的消息客户端能不能收到**。如果响应数据在网络中丢失，客户端会一直等待，服务器却以为连接正常。
2. **双方不知道对方的接收能力**。客户端不知道服务器能不能处理它的数据，服务器也不知道客户端有没有准备好接收。

### 三次握手的能力确认

三次握手完美解决了这个问题：

```
握手 1：客户端 -> 服务器
       SYN, ISN=x
       我（客户端）：可以发送

握手 2：服务器 -> 客户端
       SYN, ISN=y, ACK, ack=x+1
       服务器（确认收到 x）：我能收到
       服务器（声明自己的 ISN）：我可以发送

握手 3：客户端 -> 服务器
       ACK, ack=y+1
       客户端（确认收到 y）：我能收到
```

握手完成后，双方都确认了：
- 发送能力：对方能收到我的消息
- 接收能力：对方能发送消息给我

## 为什么不是两次握手

### 问题一：服务器无法确认自己的 SYN 是否被收到

```
客户端                      服务器
  |                            |
  |--------- SYN, ISN=x ------->|  服务器认为连接建立
  |                            |
  |<========= 数据传输 =========|  服务器开始发数据
  |                            |
  X<--- SYN 丢失（未到服务器）--|  但实际上 SYN 没到服务器！
```

如果第一次握手的 SYN 报文因网络原因延迟到达，服务器会认为连接已建立并开始分配资源，而客户端根本不知道——它在等待服务器的响应。这就是"半开连接"问题。

### 问题二：服务器无法知道客户端的接收能力

即使第一次握手成功到达，服务器发送的数据也可能无法到达客户端。但服务器不知道，仍然在发送数据，造成资源浪费。

### 问题三：无法协商初始序列号

TCP 的序列号用于标识数据字节的位置。如果只有两次握手，双方无法协商出一个双方都认可的序列号基准。

:::tip 两次握手在理论上的缺陷
在分布式系统理论中，两次握手无法保证消息传递的完整性——发送方无法确认接收方是否真的收到了消息。这是"拜占庭将军问题"的一个简化版本。
:::

## 为什么不是四次握手

### 四次是冗余的

四次握手意味着多一次交互：

```
握手 1：SYN, ISN=x
握手 2：SYN, ISN=y, ACK, ack=x+1
握手 3：ACK, ack=y+1
握手 4：（额外的一次确认）
```

但握手 3 已经完成了什么？客户端确认了服务器的 ISN，服务器也知道客户端收到了自己的消息。

**握手 4 不会提供任何新的信息**——它只是重复确认已有的事实。

### 网络延迟与性能的权衡

在网络通信中，每一次 RTT（Round Trip Time，往返时间）都是代价。四次握手需要 4 个 RTT（或更多，取决于报文是否合并），而三次握手只需要 3 个。

在延迟敏感的场景下，这一个 RTT 的差异可能意味着显著的体验下降。

## 三次握手的隐含假设

理解三次握手，还要理解它的隐含假设：**网络是相对可靠的，偶尔会丢包，但不常见**。

如果网络丢包率极高（`` `>` `` 50%），三次握手本身就可能失败。但这不在 TCP 的设计目标内——TCP 假设底层网络有一定的可靠性。

这也解释了为什么 TCP 不做无限重传：它假设大部分时候网络是通的，只是偶尔有问题。

## 实际案例：SYN 泛洪攻击

理解了为什么需要三次握手，就能理解 SYN 泛洪攻击的原理：

攻击者发送大量 SYN 报文但不完成第三次握手：
- 服务器收到 SYN 后进入 `SYN_RCVD` 状态
- 服务器等待客户端的 ACK，但攻击者不会发送
- 半连接队列被占满，正常用户无法建立连接

防御手段：
- **SYN Cookie**：不在半连接队列中存储状态，而是用 cookie 验证第三次握手
- **减少重试次数**：加快半连接超时
- **增加队列大小**：但不能无限增加

## 面试追问

### 三次握手中第三次握手丢失了会怎样？

服务器会重传 SYN + ACK（最多 `tcp_synack_retries` 次）。客户端会重传 ACK。在 Linux 中，超时时间通常设置为几秒级别。

### 能不能把第二次和第三次握手合并？

可以。这实际上是把四次挥手优化为三次。服务器可以在处理完数据后，直接发送 FIN + ACK 而不是分开发送。

### 三次握手和 TCP 的流控制、拥塞控制有什么关系？

三次握手只建立连接，协商初始序列号。流控制和拥塞控制是连接建立后才启用的机制。但初始的窗口大小是在握手阶段协商的。

## 总结

三次握手是 TCP 协议设计的精妙之处：

| 握手次数 | 确认内容 |
|---|---|
| 第一次 | 客户端能发送 |
| 第二次 | 服务器能接收（确认第一次）、服务器能发送 |
| 第三次 | 客户端能接收（确认第二次） |

不是四次——因为第四次是冗余的。

不是两次——因为无法确认服务器能接收。

这背后是对**可靠传输**目标的追求，对**性能与安全**的权衡，以及对**网络模型**的深刻理解。
