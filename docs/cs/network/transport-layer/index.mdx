# 传输层

传输层是网络通信的"分水岭"。

在这一层，通信的双方从"相邻节点"变成了"端到端"。网络层负责把数据包送到目标 IP，但哪个应用程序来处理这个数据包？这就是传输层要解决的问题。

TCP 和 UDP 是传输层最重要的两个协议。两者代表了两种截然不同的设计哲学：

- TCP 是"谨慎派"：三次握手建立连接、四次挥手断开连接、确认应答、超时重传、流量控制、拥塞控制——为了可靠交付，不惜一切代价。
- UDP 是"简单派"：无连接、无确认、无重传，开销极小，适合对实时性要求高、能容忍少量丢包的场景。

## 为什么 TCP 需要这么多机制

很多人觉得 TCP "太重了"，三次握手四次挥手浪费时间。但当你理解 TCP 面临的问题后，就会明白这些机制都是必要的。

网络是不可靠的：
- 数据包可能丢失
- 数据包可能重复
- 数据包可能乱序
- 发送速度可能超过接收方处理能力
- 发送速度可能超出网络承载能力

TCP 的每一项机制，都是为了解决这些问题。

## 学习路线

本章节从 TCP 的核心机制开始：

**三次握手**：理解为什么需要三次交换、为什么是 SYN 和 ACK、为什么不能两次握手。

**四次挥手**：理解为什么挥手需要四次、为什么 TIME_WAIT 状态存在 2MSL。

**可靠性保证**：确认应答、超时重传、序列号机制。

**流量控制**：滑动窗口机制，防止发送方压垮接收方。

**拥塞控制**：慢启动、拥塞避免、快速恢复，控制网络中的数据量。

**TCP vs UDP**：对比两者的适用场景，理解 QUIC 如何融合两者优点。

## 面试追问

TCP 是面试中的高频考点，但大部分人只停留在"三次握手四次挥手"的背诵层面。真正让面试官眼前一亮的，是理解这些机制背后的设计哲学：

- 为什么 TCP 是面向连接的，而不是像 UDP 那样无连接？
- 为什么 TIME_WAIT 状态要等待 2MSL？
- 滑动窗口和拥塞控制有什么区别和联系？
- 在高并发场景下，TIME_WAIT 积累过多会有什么后果？

这些问题没有标准答案，但思考的过程比答案本身更重要。
