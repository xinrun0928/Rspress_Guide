# TCP 与 UDP 对比

TCP 和 UDP 是传输层最重要的两个协议。但它们代表了两种截然不同的设计哲学——一个追求"可靠"，一个追求"简单"。

面试官问这个问题，不是想让你背诵两者的区别，而是想看：你是否理解为什么会有两种协议？什么时候该用哪个？

## 设计理念的对比

### TCP：可靠传输的"完美主义者"

TCP 追求的是**可靠、有序、无差错**的数据传输。为了实现这些目标，它不惜一切代价：

- 三次握手建立连接
- 四次挥手断开连接
- 确认应答确保数据到达
- 超时重传处理丢包
- 序列号去除重复和排序
- 滑动窗口控制流量
- 拥塞控制保护网络

```
TCP 的设计思路：
"网络是不可靠的，我要把一切可能出错的地方都处理掉。"
```

### UDP：简单直接的"实用主义者"

UDP 的设计思路截然相反：**如果网络足够好，为什么要我做这么多？**

```
UDP：只做最核心的事
- 无连接：不需要握手，直接发
- 无确认：不管对方收到没
- 无重传：丢就丢了
- 无流量控制：你想发多少发多少
```

## 核心区别

### 连接性

| 特性 | TCP | UDP |
|---|---|---|
| 连接建立 | 需要三次握手 | 不需要 |
| 连接断开 | 需要四次挥手 | 直接丢弃 |
| 状态 | 有状态（维护连接信息） | 无状态 |

### 可靠性

| 特性 | TCP | UDP |
|---|---|---|
| 确认应答 | 有 | 无 |
| 重传机制 | 有 | 无 |
| 序列号 | 有（排序、去重） | 无 |
| 校验和 | 有 | 有（可选） |

### 效率

| 特性 | TCP | UDP |
|---|---|---|
| 首部开销 | 20~60 字节 | 8 字节 |
| 数据传输 | 字节流（无边界） | 数据报（有边界） |
| 双工通信 | 全双工 | 全双工 |

### 流量与拥塞控制

| 特性 | TCP | UDP |
|---|---|---|
| 流量控制 | 有（滑动窗口） | 无 |
| 拥塞控制 | 有（多种算法） | 无 |

## UDP 的"简单"是优势

很多人认为 UDP "不如" TCP，但 UDP 的简单恰恰是它的优势。

### 场景一：实时音视频

视频通话、直播、游戏等场景：
- **实时性要求高**：丢了 1 帧音频，补上就行；等了 200ms 的重传，画面就卡了
- **容忍少量丢包**：偶尔一帧模糊，人眼/耳朵会自动"脑补"
- **无需有序**：音频帧 B 先到，帧 A 后到，播放出来也是乱的

```
TCP：发送 -> 等 ACK -> 超时重传 -> 延迟 200ms -> 卡顿
UDP：发送 -> 下一帧 -> 延迟 20ms -> 偶尔丢帧 -> 流畅
```

### 场景二：DNS 查询

DNS 查询通常只有几十字节，响应也要快：

```bash
# DNS 查询通常使用 UDP
dig @8.8.8.8 example.com

# 如果 UDP 超时，才切换到 TCP
```

### 场景三：QUIC 协议

Google 开发的 QUIC（HTTP/3 底层协议）基于 UDP：
- 保留了 UDP 的低延迟优势
- 在应用层实现了可靠性、流量控制、拥塞控制
- 避免了 TCP 的队头阻塞问题

```
QUIC = UDP 的速度 + TCP 的可靠性（应用层实现）
```

## TCP 的"复杂"是必要的

TCP 的复杂设计是为了解决网络中的根本问题。

### 场景一：文件传输

下载文件、邮件、网页内容：
- **不允许丢包**：丢一个字节，整个文件就损坏了
- **需要有序**：文件内容必须按顺序组装
- **允许延迟**：文件传输可以等，不会影响体验

### 场景二：远程登录

SSH、Telnet：
- **每个按键都要准确**：按了 A，收到 B，那就乱套了
- **有响应时间要求**：不能等太久，否则感觉"卡"

## 两者的使用场景总结

```
适合用 TCP 的场景：
- 文件传输（HTTP、HTTPS、SFTP）
- 邮件（SMTP、POP3、IMAP）
- 远程登录（SSH）
- 数据库连接
- 需要可靠传输的应用

适合用 UDP 的场景：
- 实时音视频（VoIP、视频会议、游戏）
- DNS 查询
- DHCP（动态IP分配）
- 直播推流
- QUIC 协议
```

## 混合策略

现代应用经常采用混合策略：UDP 传输数据，TCP 传输控制信息。

```
WebRTC 的设计：
- RTP/RTCP：基于 UDP 的实时传输
- STUN/TURN：基于 UDP 的穿透协议
- 信令：通常用 TCP 或 TLS

游戏的设计：
- 玩家位置、动作：UDP（实时性要求高）
- 道具购买、登录：TCP（可靠性要求高）
```

## 面试追问

### 为什么 DNS 默认用 UDP 而不是 TCP？

DNS 查询通常很小（几十字节），响应也要快。UDP 的开销小（8 字节头部 vs TCP 20 字节），延迟低。但 DNS 也支持 TCP，当响应超过 512 字节时，必须用 TCP。

### UDP 会丢包，能不能在应用层实现可靠性？

能。这就是 QUIC 做的事。应用层实现可靠性可以：
- 定制自己的重传策略
- 实现多路复用避免队头阻塞
- 0-RTT 或 1-RTT 建立连接

### TCP 有 UDP 没有的特性

- **连接建立和断开的状态机**：TCP 有完整的状态管理
- **半关闭**：可以只关闭一个方向的连接
- **紧急数据**：URG 标志和紧急指针
- **保活机制**：检测空闲连接是否存活

### UDP 能实现拥塞控制吗？

能，但通常不在 UDP 本身实现，而是在应用层。例如 QUIC 内置了拥塞控制，BBR 也可以在应用层实现。

## 总结

TCP 和 UDP 不是"好坏"的对比，而是"适用场景"的对比：

| 维度 | TCP | UDP |
|---|---|---|
| 可靠性 | 高 | 低（需要应用层补充） |
| 有序性 | 保证 | 不保证 |
| 延迟 | 较高 | 低 |
| 吞吐量 | 受拥塞控制限制 | 可以很高 |
| 首部开销 | 20+ 字节 | 8 字节 |

选择 TCP 还是 UDP，本质上是选择**可靠**还是**实时**。理解两者的设计哲学，你才能做出正确的选择。
