# TCP 三次握手

凌晨 2 点，你正在压测一个新部署的网关服务，却发现连接总是建立失败。查看监控面板，无数 TCP 连接处于 `SYN_SENT` 状态——服务器没有回应。

这不是网络故障，而是你可能没有正确理解 TCP 三次握手的本质。

## 为什么要建立连接

在 TCP 协议中，"连接"并不像电话线路那样真实存在。所谓的连接，本质上是通信双方为这次会话分配的资源（缓存、序列号）和达成的共识（初始序列号、通信参数）。

在数据传输之前，双方需要确认三件事：

1. **发送能力正常**：对方能收到我的消息
2. **接收能力正常**：我能收到对方的消息
3. **双方具备通信条件**：两边的初始序列号已交换

三次握手，恰好能完成这三件事。

## 握手的完整流程

TCP 三次握手的过程，可以用一句话概括：**请求 -> 确认 -> 确认之确认**。

### 第一次握手：SYN

客户端发送 TCP 报文，其中 `SYN = 1`，`seq = x`。

这意味着：
- 我想和你建立连接（SYN 标志位置 1）
- 我的初始序列号是 x（SYN 消耗一个序列号）

发送完这个报文后，客户端进入 `SYN_SENT` 状态。

### 第二次握手：SYN + ACK

服务器收到客户端的 SYN 报文后，如果同意建立连接，会发送 `SYN = 1`，`ACK = 1`，`seq = y`，`ack = x + 1`。

这意味着：
- 我也同意和你建立连接（SYN 标志位置 1）
- 我收到了你发来的 x，我期望下一个收到的是 x + 1（确认号 = x + 1）
- 我的初始序列号是 y

发送完这个报文后，服务器进入 `SYN_RCVD` 状态。

### 第三次握手：ACK

客户端收到服务器的 SYN + ACK 报文后，发送 `ACK = 1`，`seq = x + 1`，`ack = y + 1`。

这意味着：
- 我收到了你的 y，期望下一个收到的是 y + 1
- 连接已建立，双方可以开始传输数据

发送完这个报文后，客户端进入 `ESTABLISHED` 状态。服务器收到这个 ACK 后，也进入 `ESTABLISHED` 状态。

## 状态转换图

```
客户端                                    服务器
  |                                        |
  |----------- SYN, seq=x ------------->  |  客户端: SYN_SENT
  |                                        |  服务器: LISTEN
  |<---------- SYN, ACK, seq=y, ack=x+1 -|  服务器: SYN_RCVD
  |                                        |
  |------------ ACK, seq=x+1, ack=y+1 -> |  客户端: ESTABLISHED
  |                                        |  服务器: ESTABLISHED
  |                                        |
  |<=========== 数据传输 ============ ===>|
```

## 序列号的作用

TCP 的序列号不是从 1 开始的简单计数，而是一个随机生成的初始值（ISN，Initial Sequence Number）。

为什么要随机生成？主要原因有两个：

**安全考量**：如果序列号是固定的，攻击者可以轻易伪造 TCP 报文，诱使服务器接受伪造的数据。

**避免冲突**：如果同一个客户端在短时间内断开并重新连接，旧的序列号可能还在网络中传输，随机生成可以避免新旧数据混淆。

## 半连接队列与全连接队列

服务器在处理三次握手时，会维护两个重要的队列：

### 半连接队列（SYN Queue）

当服务器收到客户端的 SYN 报文，还未完成第三次握手时，这个连接处于半连接状态，放在半连接队列中。队列长度由 `tcp_max_syn_backlog` 参数控制。

### 全连接队列（Accept Queue）

当服务器完成第二次握手（发送 SYN + ACK），等待客户端的最终 ACK 时，连接从半连接队列移到全连接队列。队列长度由 `backlog` 参数控制。

如果这两个队列满了，服务器会：
- 半连接队列满：忽略新的 SYN 报文或启用 SYN Cookie 机制
- 全连接队列满：服务器可能发送 RST 或在新版本 Linux 中忽略连接

:::tip 排查连接建立失败
如果遇到 "Connection reset by peer" 或 "Connection timed out"，可以检查：
- 服务端 `netstat -s | grep LISTEN` 查看连接队列状态
- `ss -s` 查看各状态的连接数量
- 服务端 `tcp_syncookies` 是否启用
:::

## 为什么不是两次或四次

### 为什么不是两次握手

假设只用两次握手：

```
客户端                  服务器
  |                        |
  |------- SYN, seq=x ---->|  客户端认为连接建立
  |<------ SYN, ACK, ack=x+1|
  |                        |
```

问题来了：**服务器无法确认客户端能否收到自己的消息**。如果 ACK 报文丢失，客户端以为自己"已连接"，但服务器还在等待——这就造成了资源浪费。

更重要的是，服务器不知道自己的 SYN 报文客户端是否收到。如果没收到，服务器会认为连接已建立，开始分配资源，造成"单方面连接"。

### 为什么不是四次握手

有人可能会想：既然三次不够，那就四次——再加一次确认。

但仔细分析会发现，**第四次握手是冗余的**。

```
第三次握手：ACK, ack=y+1
这一次握手的意义是：客户端告诉服务器"我收到了你的 SYN"
服务器收到后，已经确认双方都具备通信能力，连接建立完成。
```

再多一次握手不会带来额外的信息确认，只是徒增延迟。

## 面试追问

### 三次握手中，如果第三次握手失败会怎样？

如果第三次握手的 ACK 报文丢失或超时，服务器会重传 SYN + ACK 报文。客户端会重传 ACK。在 Linux 中，这个重传次数由 `tcp_synack_retries` 控制（默认 5 次）。

### SYN 泛洪攻击的原理是什么？

攻击者发送大量 SYN 报文但不完成第三次握手，导致服务器半连接队列被占满，正常用户无法建立连接。防御手段包括：
- 启用 SYN Cookie
- 调整半连接队列大小
- 使用防火墙或 DDoS 防护设备

### 第一次握手丢包了怎么办？

客户端会触发超时重传。在 Linux 中，`tcp_syn_retries` 参数控制重传次数（默认 6 次），重传间隔指数增长。

## 总结

TCP 三次握手是网络通信的基石。理解这个过程，不仅能帮助你排查连接问题，更能理解 TCP 协议设计的精妙之处：

- SYN 同步序列号，确认双方发送能力
- ACK 确认序列号，确认双方接收能力
- 三次交换信息，恰好确认双方都具备收发能力

下一次当你看到监控面板上的 `SYN_SENT` 状态时，你会知道：这不仅是连接建立的起点，更是 TCP 可靠性设计的起点。
