# TCP 可靠性保证

当你在浏览器中打开一个网页时，即使网络丢包、延迟、抖动，TCP 也能保证数据完整无误地到达。这不是魔法，而是 TCP 精心设计的可靠性机制。

## 可靠传输的核心问题

网络是不可靠的，数据包可能：
- **丢失**：中途被丢弃，永远到不了目的地
- **重复**：因网络问题，同一个包被发送了多次
- **乱序**：先发的包后到，后发的包先到
- **损坏**：数据在传输过程中被篡改或校验失败

TCP 的每一项机制，都是为了解决这些问题。

## 确认应答（ACK）机制

TCP 采用确认应答机制：接收方收到数据后，需要回复一个 ACK，告知发送方"我收到了"。

### 累计确认

TCP 使用累计确认（Cumulative ACK）—— ACK 携带的序号表示"这个序号之前的数据我都收到了"。

```
发送方                          接收方
  |                               |
  |---- 数据 seq=1, len=100 ----->|
  |---- 数据 seq=101, len=100 --->|
  |---- 数据 seq=201, len=100 --->|
  |                               |
  |<--------- ACK ack=301 -------|  表示 seq=1~300 都收到了
```

即使中间丢了某个包，接收方也会 ACK 最大的连续序号。

### 确认序号与序列号

每个 TCP 报文都有两个关键序号：
- **序列号（Sequence Number）**：本报文数据的第一个字节的序号
- **确认号（Acknowledgment Number）**：期望收到的下一个字节的序号

```
序列号 seq=x
确认号 ack=y 表示：
  - 我收到了 x~y-1 的数据
  - 期望收到 y 开始的数据
```

## 超时重传机制

如果数据丢失了，发送方怎么知道？答案是超时重传。

### 工作原理

发送方发送数据后，启动一个计时器。如果在超时时间内没收到 ACK，就认为数据丢失，重传该数据。

```
发送方                          接收方
  |                               |
  |---- 数据 seq=1 --------------->|
  |                               |
  |   等待 ACK...                 |
  |   超时！重传数据 --------------->|
  |                               |
  |<--------- ACK ack=101 --------|  收到 ACK
```

### 超时时间如何确定

超时时间不能太短（会导致不必要的重传），也不能太长（会让丢包后的恢复变慢）。

TCP 使用自适应算法，根据网络往返时间（RTT）动态调整：

```python
# 简化版 RTT 计算
RTT = α * 旧RTT + (1-α) * 新样本RTT  # α 通常为 0.875

# 超时时间
RTO = RTT + 4 * RTTVAR  # RTTVAR 是 RTT 的变化量
```

Linux 中，`tcp_rto_min` 和 `tcp_rto_max` 控制 RTO 的范围。

### 指数退避

每次重传后，下一次的超时时间会翻倍：

```
第 1 次超时：等待 1 * RTO
第 2 次超时：等待 2 * RTO
第 3 次超时：等待 4 * RTO
...
```

这叫做指数退避（Exponential Backoff），避免在网络拥塞时加剧问题。

## 快速重传

超时重传虽然简单，但有一个问题：**超时时间可能很长**。当丢包发生时，发送方必须等待超时才能重传，这会影响性能。

快速重传（Fast Retransmit）解决了这个问题：当发送方收到 3 个重复 ACK 时，立即重传丢失的数据，不需要等待超时。

```
发送方                          接收方
  |                               |
  |---- 数据 seq=1 -------------->|
  |---- 数据 seq=101 ------------>|  丢失！
  |---- 数据 seq=201 ------------>|
  |                               |
  |<------- ACK ack=101 ---------|  期望 101，但没收到
  |<------- ACK ack=101 ---------|  重复 ACK 1
  |<------- ACK ack=101 ---------|  重复 ACK 2
  |                               |
  |  收到 3 个重复 ACK，立即重传 -->|
  |---- 数据 seq=101 ------------>|
```

:::tip 快速重传 vs 超时重传
快速重传是"主动出击"——通过 ACK 反馈感知丢包，立即重传。
超时重传是"被动等待"——没收到 ACK 才重传。
两者结合，兼顾效率与可靠性。
:::

## 序列号与重复检测

TCP 的序列号是32位无符号整数，初始值随机生成（防止安全攻击）。每个传输的字节都会占用一个序号。

### 如何检测重复

```
发送 seq=1~100            -> 接收方收到 seq=1~100
发送 seq=101~200          -> 接收方收到 seq=101~200
发送 seq=201~300          -> 丢包
发送 seq=201~300          -> 接收方再次收到 seq=201~300
                            接收方发现 seq=201 已收到，丢弃重复
```

接收方维护一个滑动窗口，通过序列号判断：
- 数据是否在期望范围内
- 数据是否重复
- 数据是否损坏

## 数据校验与完整性

TCP 头部包含 16 位的校验和（Checksum），用于检测数据在传输过程中是否损坏。

### 校验和计算

校验和覆盖 TCP 头部、数据以及一个伪头部（源IP、目的IP、协议号、TCP长度）。

```
伪头部：SRC IP | DST IP | 0 | 协议(6) | TCP长度
TCP头部：源端口、目的端口、序列号、确认号、标志位、校验和...
数据：应用层数据
```

如果校验和不匹配，TCP 会丢弃该报文（不 ACK），触发发送方重传。

:::warning 校验和的局限性
TCP 校验和只能检测错误，不能纠正错误。而且 16 位校验和的检错能力有限（约 1/65536 的错误无法检测）。现代网络设备通常使用更强大的 CRC 校验，TCP 校验和更多是向后兼容的设计。
:::

## 流量控制与拥塞控制（预览）

TCP 的可靠性不仅靠确认和重传，还有两个重要的控制机制：

### 流量控制

防止发送方压垮接收方。接收方通过调整窗口大小，告诉发送方"你能发多少"。

### 拥塞控制

防止发送方压垮网络。发送方根据网络状况，动态调整发送速率。

这两个机制将在后续章节详细讲解。

## 面试追问

### TCP 和 UDP 都有校验和，但有什么区别？

TCP 校验和覆盖更广：伪头部 + TCP 头部 + 数据。UDP 校验和是可选的（IPv4 中），但为了兼容性通常也启用。两者都不能完全避免错误，但足够在大多数场景下保证基本的数据完整性。

### 如果 ACK 丢了怎么办？

发送方会超时重传数据。接收方收到重复数据后，丢弃重复，但会重发 ACK（依然是最大的连续序号）。

### 序列号回绕问题

32 位序列号在高速网络中可能回绕（绕回 0）。现代 TCP 实现通过时间戳选项（TCP Timestamps Option）来解决这个问题，可以区分新旧数据。

### 什么是 SACK？

SACK（Selective Acknowledgment）允许接收方告诉发送方"我收到了这些不连续的数据块"，让发送方只重传丢失的部分，而不是从丢失位置之后的所有数据。

```bash
# 查看 Linux 是否启用 SACK
cat /proc/sys/net/ipv4/tcp_sack
```

## 总结

TCP 的可靠性保证是一套精心设计的机制：

| 机制 | 解决的问题 |
|---|---|
| 确认应答（ACK） | 确认数据是否到达 |
| 超时重传 | 数据丢失后恢复 |
| 快速重传 | 减少等待时间 |
| 序列号 | 去除重复、排序 |
| 校验和 | 检测数据损坏 |

这些机制相互配合，让 TCP 能在不可靠的网络上提供可靠传输服务。理解它们，你才能真正理解为什么 TCP 是"面向连接"的协议。
