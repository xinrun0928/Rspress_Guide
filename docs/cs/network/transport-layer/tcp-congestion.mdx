# TCP 拥塞控制

想象一下早高峰的十字路口：每辆车都想快速通过，但如果大家都挤在一起，谁都动不了。

网络也是如此——如果发送方无节制地向网络注入数据，网络会变得拥塞，导致大量丢包，性能急剧下降。

**拥塞控制**就是 TCP 的"交通疏导机制"，它让发送方根据网络的承载能力，动态调整发送速率。

## 拥塞控制的基本概念

### 拥塞窗口（cwnd）

与流量控制的接收窗口不同，拥塞控制引入了**拥塞窗口（cwnd）**——发送方根据网络状况自己计算的一个窗口。

实际能发送的数据量 = `min(接收方窗口, 拥塞窗口)`

```python
发送窗口 = min(rwnd, cwnd)
```

### 慢启动阈值（ssthresh）

这是一个关键阈值：
- 当 `cwnd < ssthresh` 时：慢启动阶段
- 当 `cwnd >= ssthresh` 时：拥塞避免阶段

## 慢启动（Slow Start）

新连接开始时，TCP 不知道网络能承载多少数据。慢启动的策略是：**从小到大，逐步试探**。

### 增长策略

```
初始 cwnd = 1 MSS（最大段大小，通常约 1460 字节）

每收到一个 ACK，cwnd 加倍：
cwnd = cwnd * 2

所以增长曲线是指数级的：
1 -> 2 -> 4 -> 8 -> 16 -> ...
```

### 何时结束慢启动

慢启动在以下任一条件下结束：
1. `cwnd >= ssthresh`：进入拥塞避免
2. 发生丢包：回到慢启动或快重传

```
时间 ->
cwnd:  1    2    4    8    16   32   64   ssthresh=100
        *----*----*----*----*----*----*---->
                        |
                        v
                    进入拥塞避免
```

:::tip 初始拥塞窗口的演进
RFC 6928 将初始拥塞窗口从 2~4 MSS 提升到 10 MSS。这意味着新连接建立后，可以立即发送 10 个 MSS 的数据，而不是从 1 开始慢慢增长。这大幅降低了小文件传输的延迟。
:::

## 拥塞避免（Congestion Avoidance）

当 `cwnd` 增长到一定规模后，如果继续指数增长，很快就会压垮网络。拥塞避免的策略是：**稳扎稳打，线性增长**。

### 增长策略

```
每收到一个 ACK，cwnd 增加 1 MSS：
cwnd = cwnd + MSS * (MSS / cwnd)

这等价于每 RTT（往返时间），cwnd 增加 1 MSS。
```

```
慢启动（指数增长）                      拥塞避免（线性增长）
cwnd: 1 -> 2 -> 4 -> 8 -> 16 ->       -> 17 -> 18 -> 19 -> 20 ->
                                   
                 ssthresh
```

## 快重传（Fast Retransmit）

当丢包发生时，TCP 有两种响应策略：

### 传统策略：超时重传

如果超时没收到 ACK，TCP 认为网络拥塞：
- 将 `ssthresh` 设置为当前 `cwnd / 2`
- 将 `cwnd` 重置为 1 MSS
- 重新进入慢启动

这叫做**Reno 拥塞控制**。

### 改进策略：快重传 + 快恢复

如果收到 3 个重复 ACK，说明数据包丢失了，但网络还没有严重拥塞：
1. 立即重传丢失的包（快重传）
2. `ssthresh = cwnd / 2`
3. `cwnd = ssthresh + 3 * MSS`（因为收到 3 个 ACK）
4. 进入快恢复阶段

```
快重传流程：
收到 3 个重复 ACK：
  -> 立即重传 seq=X（丢失的包）
  -> cwnd = ssthresh + 3 * MSS
  -> 每收到一个重复 ACK，cwnd++

收到新的 ACK：
  -> cwnd = ssthresh
  -> 进入拥塞避免
```

## 快恢复（Fast Recovery）

快恢复是 Reno 算法的改进。在进入快重传后，TCP 不回到慢启动，而是进入快恢复：

```
标准快恢复（TCP Reno）：
1. 收到 3 个重复 ACK
2. cwnd = ssthresh + 3 * MSS
3. 每收到一个重复 ACK：cwnd++
4. 收到新 ACK：cwnd = ssthresh，进入拥塞避免

TCP NewReno（改进版）：
- 在快恢复期间，如果有多个包丢失，只重传第一个丢失的包
- 收到所有丢失包的 ACK 后，才退出快恢复
```

## TCP BBR：新一代拥塞控制

传统 TCP 拥塞控制基于丢包检测——丢包了就认为拥塞。但这在现代网络中效率不高：

- 高速网络丢包率很低时，传统算法浪费了带宽
- Bufferbloat 问题：路由器缓冲区太大，导致延迟飙升

Google 开发的 **BBR（Bottleneck Bandwidth and Round-trip propagation time）** 算法另辟蹊径：

```python
# BBR 核心思想
目标速率 = 瓶颈带宽 / 往返延迟

# 不再基于丢包，而是估算：
btlBw = 瓶颈带宽（最近观察到的最大吞吐量）
rtProp = 往返延迟（最近观察到的最小 RTT）

发送速率 = min(btlBw, rtProp)
```

BBR 在高延迟、高带宽网络中表现优异，已被部署在 YouTube 等服务中。

## 拥塞控制算法对比

| 算法 | 特点 | 适用场景 |
|---|---|---|
| TCP Reno | 经典算法，快重传+快恢复 | 通用 |
| TCP NewReno | 改进的 Reno，处理多包丢失 | 通用 |
| TCP Cubic | Linux 默认，增长曲线更平滑 | 高带宽网络 |
| BBR | 基于带宽和延迟估计 | 高速网络、高延迟网络 |
| TCP Vegas | 基于 RTT 检测拥塞 | 低丢包率网络 |

```bash
# 查看当前使用的拥塞控制算法
cat /proc/sys/net/ipv4/tcp_congestion_control

# 查看可用算法
cat /proc/sys/net/ipv4/tcp_available_congestion_control
```

## 实际案例：抓包分析拥塞控制

观察慢启动过程：

```bash
tcpdump -i any -nn 'tcp' host 192.168.1.100 -w congestion.pcap
```

在 Wireshark 中，注意 TCP 窗口大小的变化：

```
时间    Seq    ACK    Window    解释
0ms     1      1      65535    连接建立

2ms     1461   1      65535    ACK, cwnd=2, 发送 2 MSS
3ms     2921   1      65535    ACK, cwnd=4, 发送 4 MSS
4ms     5841   1      65535    ACK, cwnd=8, 发送 8 MSS

```

窗口持续翻倍，直到达到阈值或发生丢包。

## 面试追问

### 慢启动为什么叫"慢启动"？

虽然名字里有"慢"，但 cwnd 的增长是指数级的。叫"慢"是因为它**从一个很小的值开始**，而不是像直接发送大量数据那样"快"。这避免了在不清楚网络状况时一下子压垮网络。

### 丢包后，cwnd 回到 1 MSS，这会不会太激进？

对于超时重传（最严重的丢包），回到慢启动是合理的，说明网络可能真的拥塞了。但对于快重传（收到 3 个重复 ACK），cwnd 只减半而不是回到 1，说明问题没那么严重。

### BBR 和 Cubic 哪个更好？

取决于网络环境：
- 高带宽低延迟网络：Cubic 表现好
- 高延迟高带宽网络：BBR 表现好
- 丢包率高的网络：两者都可能有问题

### 什么是 Bufferbloat？

现代路由器有巨大的缓冲区。当缓冲区满了才开始丢包，导致延迟飙升。BBR 等新算法通过主动检测网络能力，避免填满缓冲区，从而降低延迟。

## 总结

TCP 拥塞控制是一套完整的"流量管控"体系：

| 阶段 | cwnd 变化 | 触发条件 |
|---|---|---|
| 慢启动 | 每 ACK 加倍 | 新连接或丢包后重置 |
| 拥塞避免 | 每 RTT 加 1 MSS | `cwnd >= ssthresh` |
| 快重传 | 立即重传 | 收到 3 个重复 ACK |
| 快恢复 | 减半 + 3 MSS | 快重传后 |

理解拥塞控制，你才能理解为什么 TCP 在不同网络环境下表现不同。下一次看监控时，你会发现：带宽利用率低、延迟高，可能不是网络问题，而是拥塞控制算法的问题。
