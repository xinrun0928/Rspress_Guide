# QUIC 协议

2012 年，Google 的工程师在思考一个问题：**能不能发明一种协议，既有 TCP 的可靠性，又有 UDP 的效率？**

他们做到了。QUIC（Quick UDP Internet Connections）就此诞生，并于 2022 年成为 HTTP/3 的底层协议。

## 为什么需要 QUIC

TCP 的问题：
- **队头阻塞**：HTTP/2 的多路复用依赖 TCP，如果一个包丢失，所有流都要等
- **握手延迟**：即使使用 TLS 1.3，也要 1-RTT 建立连接
- **连接迁移慢**：切换网络（如从 WiFi 切到 4G）需要重建连接

UDP 本身没有这些问题，但缺少可靠性。QUIC 在 UDP 之上实现了这些功能。

## QUIC 的核心特性

### 1. 0-RTT / 1-RTT 建立连接

QUIC 结合了 TLS 1.3 的理念：

```
TCP + TLS 1.2：
  握手：TCP 三次握手 + TLS 四次握手 = 1~3 RTT
  数据：需要等待握手完成

TCP + TLS 1.3：
  握手：TCP 三次握手 + TLS 1-RTT = 1 RTT
  数据：需要等待握手完成

QUIC：
  首次连接：1-RTT（密钥交换）
  再次连接：0-RTT（使用上次会话密钥）
  数据：可以立即发送（0-RTT 模式下）
```

0-RTT 是 QUIC 最重要的创新之一——对于重复访问，可以立即发送数据，不需要等待握手完成。

:::warning 0-RTT 的安全考量
0-RTT 使用之前会话的密钥加密数据，如果攻击者记录了之前的通信，可能存在重放攻击风险。因此 0-RTT 数据通常只能发送一些"幂等"的请求（如 GET）。
:::

### 2. 多路复用（无队头阻塞）

HTTP/2 在 TCP 上实现了多路复用，但有一个致命问题：

```
TCP 的队头阻塞：
流 1：|----A----|----B----|----C----|
流 2：|----1----|----2----|----3----|
                          |
                    包 B 丢失
                          |
        所有流都要等待 B 重传完成
```

QUIC 的解决方案：

```
QUIC 的多路复用：
流 1：|----A----|----B----|----C----|
流 2：|----1----|----2----|----3----|
              |
        只有流 1 的包丢失
              |
        只有流 1 等待重传
        流 2 继续传输
```

每个 QUIC 流是独立的，一个流的丢包不会影响其他流。

### 3. 连接迁移

当你的手机从 WiFi 切换到 4G 时，TCP 连接会断开重连。用户会感受到明显的卡顿。

QUIC 使用**连接 ID** 来标识连接：

```python
# QUIC 连接标识
conn_id = 生成随机连接ID

# 即使 IP 变了，只要带上相同的 conn_id
# 服务器就能识别这是同一个 QUIC 连接
```

这叫做**连接迁移（Connection Migration）**，用户几乎感受不到网络切换。

### 4. 内置 TLS 1.3

QUIC 将 TLS 1.3 集成到协议内部：

```
TCP + TLS：
  TCP 头部 | TLS 头部 | 应用数据
  (分离的层)

QUIC：
  QUIC 头部 | 加密的应用数据
  (集成加密，1 个层)
```

集成的好处：
- 减少握手开销
- 所有数据默认加密
- 更好的隐私保护

## QUIC 头部结构

### Long Header（连接建立时使用）

```
+------------+------+---------+---------------+
| flags(8)  | 连接ID长度 | 连接ID | 版本/包号等 |
+------------+------+---------+---------------+
```

### Short Header（连接建立后使用）

```
+----+------+---------+
|flags| 连接ID | 包号    |
+----+------+---------+
```

包号（Packet Number）是 QUIC 可靠传输的关键——它用于确认和重传，而且设计得比 TCP 更高效（不会混淆重传和原始包）。

## QUIC 的流控制

QUIC 同时在连接级别和流级别进行流量控制：

### 连接级别流控

```python
# 最大连接窗口
max_data = 1000000  # 连接级别最多接收 1MB

# 已接收数据
recv_data = 0

# 发送窗口更新
conn = Connection()
conn.send_window_update(max_data)
```

### 流级别流控

```python
# 每个流也有独立的窗口
stream = conn.new_stream()
stream.max_data = 100000  # 每个流最多接收 100KB
```

这样可以防止单个流耗尽整个连接的接收缓冲区。

## QUIC 的拥塞控制

QUIC 内置了多种拥塞控制算法，可以在连接建立时协商：

```
慢启动：
- 每个 ACK，cwnd += MSS

拥塞避免：
- 每个 RTT，cwnd += MSS

丢包检测：
- 3 个重复 ACK -> 快重传
- 超时 -> 降级到慢启动
```

支持多种拥塞控制算法：
- CUBIC（默认）
- BBR
- Reno

## QUIC vs TCP + HTTP/2

| 特性 | TCP + HTTP/2 | QUIC + HTTP/3 |
|---|---|---|
| 队头阻塞 | 有（TCP 层） | 无 |
| 握手延迟 | 1~3 RTT | 0~1 RTT |
| 连接迁移 | 不支持 | 支持 |
| 加密 | TLS 分离 | 内置加密 |
| 前向纠错 | 无 | 可选 |
| 连接标识 | 五元组（IP+端口） | 连接 ID |

## 实际应用

### Chrome 中的 QUIC

Chrome 从 2014 年开始支持 QUIC，目前默认启用：

```
chrome://net-internals/#quic
```

### Nginx 中的 QUIC

Nginx 通过 ngx_http_v3_module 提供 HTTP/3 支持：

```nginx
server {
    listen 443 ssl;
    http3 on;                    # 启用 HTTP/3
    quic_gso on;                 # 启用 GSO 优化
    ssl_certificate /path/to/cert.pem;
    ssl_certificate_key /path/to/key.pem;
}
```

### curl 测试 QUIC

```bash
# curl 支持 HTTP/3
curl --http3 https://example.com -I

# 或者用 --version 查看支持的协议
curl --version | grep HTTP3
```

## 面试追问

### QUIC 为什么用 UDP 作为传输层？

UDP 不需要握手，没有连接状态，可以穿越大部分防火墙。如果从零设计一个传输协议，很难绕过 NAT 和防火墙的层层限制。用 UDP 可以直接利用现有的网络基础设施。

### QUIC 如何处理 NAT 穿越？

QUIC 本身支持连接迁移，但 NAT 穿越依赖于其他技术（如 ICE、STUN、TURN）。WebRTC 生态中已经有成熟的 NAT 穿越方案，QUIC 可以借鉴。

### HTTP/3 一定比 HTTP/2 快吗？

不一定。HTTP/3 的优势在丢包率高的网络（如移动网络）中更明显。在稳定的 WiFi 环境下，两者的差异可能不明显。首次连接时，HTTP/3 握手可能更慢（0-RTT 只能在重复访问时使用）。

### QUIC 的丢包恢复和 TCP 有什么不同？

TCP 基于序列号的 ACK，QUIC 基于包号的 ACK。TCP 的 ACK 可能有歧义（重传的包的 ACK 原始包还是重传包？），QUIC 通过包号避免了这个问题，可以精确知道哪个包被确认了。

## 总结

QUIC 是传输层的一次革新：

- **UDP 的皮**：简单、快速、穿越性好
- **TCP 的心**：可靠、有序、拥塞控制
- **现代的魂**：0-RTT、多路复用无阻塞、连接迁移、内置加密

HTTP/3 基于 QUIC，继承了这些优势。随着 QUIC 生态的成熟，我们正在见证互联网传输协议的演进。
