# 网络基础

在深入学习网络协议之前，我们需要先建立宏观视角。分层模型是计算机网络最核心的设计思想，它将复杂的网络通信拆解为若干层次，每层只关心自己的职责，层与层之间通过接口协作。

这一章，我们从分层模型开始，理解每层的核心概念，再搞清楚 URI、URL、URN 这三个容易混淆的概念。

## OSI 七层模型

OSI（Open Systems Interconnection）模型是国际标准化组织（ISO）在 1984 年提出的参考模型。虽然它是一个理论模型，但为网络协议的设计和教学提供了统一框架。

### 七层职责一览

```
┌─────────────────────────────────────────────────────────────┐
│  7. 应用层 (Application)    —— 用户与网络的接口              │
├─────────────────────────────────────────────────────────────┤
│  6. 表示层 (Presentation)   —— 数据格式转换、加密解密        │
├─────────────────────────────────────────────────────────────┤
│  5. 会话层 (Session)        —— 会话管理、通信同步            │
├─────────────────────────────────────────────────────────────┤
│  4. 传输层 (Transport)      —— 端到端可靠传输                │
├─────────────────────────────────────────────────────────────┤
│  3. 网络层 (Network)        —— 路径选择、逻辑寻址            │
├─────────────────────────────────────────────────────────────┤
│  2. 数据链路层 (Data Link)  —— 帧封装、MAC 寻址             │
├─────────────────────────────────────────────────────────────┤
│  1. 物理层 (Physical)       —— 比特流传输                   │
└─────────────────────────────────────────────────────────────┘
```

每一层都像是流水线上的一个工序，只完成自己的任务，然后把结果传递给下一层。

### 为什么需要分层？

想象一下，如果没有分层，网络协议的设计会变成什么样？

HTTP 要关心光纤怎么传数据？TCP 要考虑 wifi 信号强弱？每个新协议都要重新设计所有底层逻辑？

分层带来的好处是**关注点分离**和**模块化设计**：
- 各层独立演进，底层换了不影响上层
- 问题容易定位，哪层出问题修哪层
- 协议可以复用，TCP 可以承载 HTTP 也可以承载 FTP

## TCP/IP 四层模型

实际互联网使用的是 TCP/IP 模型，它是对 OSI 模型的简化：

| TCP/IP 四层 | 对应 OSI 层级 | 核心协议 |
|---|---|---|
| 应用层 | 应用层 + 表示层 + 会话层 | HTTP、DNS、FTP、SMTP、SSH |
| 传输层 | 传输层 | TCP、UDP |
| 网际层 | 网络层 | IP、ICMP、ARP |
| 网络接口层 | 数据链路层 + 物理层 | Ethernet、PPP、Wi-Fi |

:::tip 对比记忆
OSI 是"理论派"，七层划分细致，教学友好；TCP/IP 是"实战派"，四层精简高效，工业标准。
:::

## 数据封装过程

数据从发送端到接收端，每层都会加上自己的头部，这个过程叫做**封装**。以 HTTP 请求为例：

```
┌──────────────────────────────────────────────────────────────┐
│                        应用层：HTTP 数据                      │
├──────────────────────────────────────────────────────────────┤
│                传输层：TCP 头部 + HTTP 数据                   │
├──────────────────────────────────────────────────────────────┤
│              网络层：IP 头部 + TCP 头部 + HTTP 数据           │
├──────────────────────────────────────────────────────────────┤
│           数据链路层：Ethernet 头部 + IP 头 + TCP + HTTP 数据 │
├──────────────────────────────────────────────────────────────┤
│  物理层：转换为电/光/无线电信号                               │
└──────────────────────────────────────────────────────────────┘
```

解封装的过程则相反，每层剥去自己的头部，将数据交给上层处理。

## 常见端口与协议

端口是传输层用来区分同一主机上不同应用的标识。端口号范围 `0` `~` `65535`：

| 端口范围 | 说明 | 常见服务 |
|---|---|---|
| `0` `~` `1023` | 知名端口，保留给系统服务 | HTTP(80)、HTTPS(443)、SSH(22) |
| `1024` `~` `49151` | 注册端口，需 IANA 注册 | MySQL(3306)、Redis(6379) |
| `49152` `~` `65535` | 动态端口，客户端使用 | 浏览器随机分配 |

常用端口一览：

```bash
# HTTP/HTTPS
80/tcp   # HTTP
443/tcp  # HTTPS

# 远程访问
22/tcp   # SSH
23/tcp   # Telnet（明文，已废弃）

# 文件传输
20/tcp   # FTP 数据
21/tcp   # FTP 控制
22/tcp   # SFTP/SCP

# 邮件
25/tcp   # SMTP
110/tcp  # POP3
143/tcp  # IMAP

# 数据库
3306/tcp # MySQL
5432/tcp # PostgreSQL
6379/tcp # Redis

# 消息队列
5672/tcp # RabbitMQ
9092/tcp # Kafka

# 监控/管理
2181/tcp # ZooKeeper
9200/tcp # Elasticsearch
```

:::warning 端口安全
生产环境中，非必要端口不要暴露在公网。数据库端口（3306、6379）只应允许内网访问，使用云服务的安全组或防火墙规则控制。
:::

## URI、URL、URN：三个容易混淆的概念

这三个概念经常被混用，但实际上各有含义：

- **URI**（Uniform Resource Identifier）：统一资源标识符，标识资源的字符串格式的总称
- **URL**（Uniform Resource Locator）：统一资源定位符，标识资源位置的地址
- **URN**（Uniform Resource Name）：统一资源名称，标识资源身份的持久标识

简单理解：**URI 是父集，URL 和 URN 是子集**。URI 可能是 URL 或 URN。

```
URI
├── URL  (https://example.com/path#section)  → 资源在哪里
└── URN  (urn:isbn:978-7-111-42193-4)       → 资源是什么
```

### URL 的结构

以 `https://www.example.com:8080/path/to/page?name=value#anchor` 为例：

```
┌─ scheme ─┬─ host ────────┬─ port ─┬─ path ───────────┬─ query ────┬─ fragment
│  https   │ www.example.com│ :8080  │ /path/to/page    │ ?name=value│ #anchor
└──────────┴───────────────┴────────┴──────────────────┴────────────┴──────────
```

| 组件 | 说明 | 示例 |
|---|---|---|
| scheme | 协议方案 | `http`、`https`、`ftp`、`mailto` |
| host | 主机名 | `www.example.com`、`192.168.1.1` |
| port | 端口号 | `:80`、`:443`、`:8080` |
| path | 资源路径 | `/path/to/page` |
| query | 查询参数 | `?name=value&age=20` |
| fragment | 锚点 | `#section` |

### URN 的应用

URN 最为人熟知的应用是 ISBN：

```bash
urn:isbn:978-7-111-42193-4
```

无论这本书的物理位置如何变化（图书馆、书店、电子书），这个 URN 永远指向它。这就是 URN 的意义——**持久且位置无关**。

## 总结

本章建立了计算机网络的宏观认知：

- OSI 七层模型和 TCP/IP 四层模型是理解网络协议的基础框架
- 数据封装体现了分层设计的精妙——每层只做自己的事
- 端口是传输层区分应用的标识，常用端口需要熟记
- URI 是总称，URL 是地址，URN 是身份标识，三者不可混用
