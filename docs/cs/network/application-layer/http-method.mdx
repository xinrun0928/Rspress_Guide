# HTTP 方法

HTTP 方法定义了客户端对服务器资源执行的操作。理解每个方法的语义，是设计 RESTful API 的基础。

## 常用方法一览

| 方法 | 语义 | 安全性 | 幂等性 | 消息体 |
|---|---|---|---|---|
| GET | 获取资源 | ✅ | ✅ | 无 |
| POST | 创建资源 | ❌ | ❌ | 有 |
| PUT | 更新/替换资源 | ❌ | ✅ | 有 |
| PATCH | 部分更新资源 | ❌ | ❌ | 有 |
| DELETE | 删除资源 | ❌ | ✅ | 通常无 |
| HEAD | 获取头部（不含消息体） | ✅ | ✅ | 无 |
| OPTIONS | 查询支持的通信选项 | ✅ | ✅ | 通常无 |
| TRACE | 回环测试 | ✅ | ✅ | 无 |
| CONNECT | 建立隧道 | ❌ | ❌ | 有 |

> **安全性**：方法是否只读取数据不修改状态
> **幂等性**：多次执行是否与执行一次效果相同

## GET：获取资源

GET 是最常用的方法，用于请求指定资源。应该是安全的（不修改服务器数据）和幂等的（多次请求结果相同）。

### 特点

- 参数通过 URL 传递（查询字符串）
- 有长度限制（浏览器 URL 长度限制，约 2KB-8KB）
- 请求可以被缓存
- 参数暴露在 URL 中，不适合敏感数据

### 示例

```bash
# 获取用户列表
curl https://api.example.com/users

# 带查询参数
curl "https://api.example.com/users?page=1&limit=10"

# 获取单个用户
curl https://api.example.com/users/123
```

```http
GET /users?page=1&limit=10 HTTP/1.1
Host: api.example.com
Accept: application/json
```

## POST：创建资源

POST 用于向服务器提交数据，通常用于创建新资源。POST 不是幂等的——多次提交可能创建多个资源。

### 特点

- 参数通过请求体传递
- 无长度限制（理论上）
- 请求通常不会被缓存
- 常用于表单提交、文件上传、JSON API

### 示例

```bash
# 创建新用户
curl -X POST https://api.example.com/users \
  -H "Content-Type: application/json" \
  -d '{"name":"张三","email":"zhang@example.com"}'
```

```http
POST /users HTTP/1.1
Host: api.example.com
Content-Type: application/json
Content-Length: 47

{"name":"张三","email":"zhang@example.com"}
```

```http
HTTP/1.1 201 Created
Location: /users/456
Content-Type: application/json

{"id":456,"name":"张三","email":"zhang@example.com"}
```

## PUT：完整更新

PUT 用于更新或替换资源。如果资源不存在，可以选择创建（取决于服务器实现）。PUT 是幂等的——多次 PUT 同一资源，结果相同。

### 特点

- 提交完整资源数据（替换整个资源）
- 幂等操作
- 通常需要携带完整资源数据

### 示例

```bash
# 更新用户信息（完整替换）
curl -X PUT https://api.example.com/users/456 \
  -H "Content-Type: application/json" \
  -d '{"name":"李四","email":"li@example.com"}'
```

```http
PUT /users/456 HTTP/1.1
Host: api.example.com
Content-Type: application/json

{"name":"李四","email":"li@example.com"}
```

:::warning PUT vs POST
- POST：创建新资源，服务器决定资源 ID
- PUT：更新指定位置的资源，需要客户端指定 ID
:::

## PATCH：部分更新

PATCH 用于部分更新资源，只提交需要修改的字段。与 PUT 的"完整替换"不同，PATCH 只更新指定字段。

### 示例

```bash
# 只更新用户名
curl -X PATCH https://api.example.com/users/456 \
  -H "Content-Type: application/json" \
  -d '{"name":"王五"}'
```

```http
PATCH /users/456 HTTP/1.1
Host: api.example.com
Content-Type: application/json

{"name":"王五"}
```

## DELETE：删除资源

DELETE 用于删除指定资源。DELETE 是幂等的——删除已删除的资源，服务器通常返回 404（资源不存在），而不是错误。

### 示例

```bash
# 删除用户
curl -X DELETE https://api.example.com/users/456
```

```http
DELETE /users/456 HTTP/1.1
Host: api.example.com
```

```http
HTTP/1.1 204 No Content
```

## HEAD：获取头部

HEAD 与 GET 类似，但服务器只返回头部，不返回消息体。用于检查资源是否存在、获取元信息等。

### 示例

```bash
# 检查资源是否存在
curl -I https://example.com/large-file.zip

# 输出：
# HTTP/1.1 200 OK
# Content-Type: application/zip
# Content-Length: 1048576
# Last-Modified: Mon, 29 Mar 2026
```

## OPTIONS：查询支持的选项

OPTIONS 返回服务器支持的 HTTP 方法列表。用于 CORS 预检请求。

### 示例

```bash
# 查询支持的 HTTP 方法
curl -X OPTIONS https://api.example.com/users -v

# CORS 预检请求
curl -X OPTIONS https://api.example.com/users \
  -H "Origin: https://example.com" \
  -H "Access-Control-Request-Method: POST"
```

```http
HTTP/1.1 200 OK
Allow: GET, POST, PUT, DELETE, OPTIONS
Access-Control-Allow-Origin: *
Access-Control-Allow-Methods: GET, POST, PUT, DELETE, OPTIONS
```

## RESTful API 设计原则

### 资源命名

使用名词而非动词：

```
✅ GET /users          # 获取用户列表
❌ GET /getUsers

✅ POST /users         # 创建用户
❌ POST /createUser
```

### 方法对应操作

| 方法 | 操作 | 示例 |
|---|---|---|
| GET | 查询 | GET /users/123 |
| POST | 创建 | POST /users |
| PUT | 更新 | PUT /users/123 |
| PATCH | 部分更新 | PATCH /users/123 |
| DELETE | 删除 | DELETE /users/123 |

### 状态码

| 状态码 | 含义 | 示例 |
|---|---|---|
| 200 | 成功 | GET/PUT/PATCH 成功 |
| 201 | 已创建 | POST 创建资源成功 |
| 204 | 无内容 | DELETE 成功 |
| 400 | 请求错误 | 参数验证失败 |
| 404 | 未找到 | 资源不存在 |
| 500 | 服务器错误 | 内部异常 |

## 实际应用

### RESTful API 示例

```bash
# 获取文章列表
GET /articles?page=1&limit=10

# 获取单篇文章
GET /articles/123

# 创建文章
POST /articles
{"title":"新文章","content":"..."}

# 更新文章（完整）
PUT /articles/123
{"title":"更新后的标题","content":"更新后的内容"}

# 部分更新文章
PATCH /articles/123
{"title":"只更新标题"}

# 删除文章
DELETE /articles/123
```

### GraphQL 对比

GraphQL 只用一个 POST 方法，通过请求体区分操作：

```bash
# GraphQL 查询
curl -X POST https://api.example.com/graphql \
  -H "Content-Type: application/json" \
  -d '{
    "query": "query { user(id: 123) { name email } }"
  }'
```

## 面试追问

### 为什么 GET 请求不适合传递大量数据？

GET 参数通过 URL 传递，URL 有长度限制（浏览器通常限制 2-8KB）。而且 URL 会被记录在日志、浏览器历史中，不适合传递敏感数据。

### POST 和 PUT 都能创建资源，区别是什么？

幂等性：PUT 是幂等的，多次 PUT 同一资源结果相同；POST 不是幂等的，多次 POST 可能创建多个资源。通常 PUT 由客户端指定资源 ID，POST 由服务器生成 ID。

### 什么是幂等性？为什么重要？

幂等性指多次执行操作与执行一次的效果相同。幂等操作在网络不稳定时可以安全重试。例如 DELETE 是幂等的，删除两次（第二次返回 404）不会改变最终状态。

### PUT 和 PATCH 的区别？

PUT 替换整个资源，需要提交完整数据。PATCH 只修改指定字段，提交部分数据。例如用户有 name 和 email 两个字段，更新 name 时 PUT 需要提交两者，PATCH 只需提交 name。

## 总结

HTTP 方法定义了客户端与服务器的交互语义：

| 方法 | 用途 | 幂等 | 安全 |
|---|---|---|---|
| GET | 获取 | ✅ | ✅ |
| POST | 创建 | ❌ | ❌ |
| PUT | 替换 | ✅ | ❌ |
| PATCH | 修改 | ❌ | ❌ |
| DELETE | 删除 | ✅ | ❌ |

理解这些方法，你才能设计出符合 REST 原则的 API。
