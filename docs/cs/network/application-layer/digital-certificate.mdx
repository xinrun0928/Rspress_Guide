# 数字证书与 CA 认证

当你访问 `https://www.example.com` 时，浏览器如何确认这个网站是可信的，而不是黑客伪造的？

答案就是**数字证书**和 **CA（Certificate Authority）认证体系**。

## 为什么需要数字证书

### 中间人攻击

在没有证书的体系中，中间人攻击轻而易举：

```
正常情况：
用户 <---------> 网站（真实的 example.com）

中间人攻击：
用户 <---------> 黑客 <---------> 网站
    黑客可以：        黑客可以：
    - 看到明文         - 看到明文
    - 修改内容         - 修改内容
    - 冒充身份         - 冒充身份
```

### 证书的作用

数字证书解决的核心问题是：**"我如何确认这个公钥属于 example.com，而不是黑客？"**

```
数字证书 = 网站信息 + 网站公钥 + CA 签名
```

## 数字证书结构

数字证书遵循 **X.509** 标准，包含以下内容：

```
X.509 证书结构：
┌─────────────────────────────────────────┐
│ 版本（Version）                          │
├─────────────────────────────────────────┤
│ 序列号（Serial Number）                  │
├─────────────────────────────────────────┤
│ 签名算法（Signature Algorithm）          │
├─────────────────────────────────────────┤
│ 颁发者（Issuer）                         │
├─────────────────────────────────────────┤
│ 有效期（Validity Period）                 │
│   - 生效时间                              │
│   - 过期时间                              │
├─────────────────────────────────────────┤
│ 主体（Subject）                          │
│   - CN（Common Name）：域名               │
│   - O（Organization）：组织               │
│   - C（Country）：国家                    │
├─────────────────────────────────────────┤
│ 主体公钥信息（Subject Public Key Info）   │
│   - 算法                                  │
│   - 公钥                                 │
├─────────────────────────────────────────┤
│ 证书签名（Signature）                     │
└─────────────────────────────────────────┘
```

## 证书签名与验证

### 签名过程

证书的签名是 CA 对证书内容的"数字指纹"：

```
签名过程：
1. 对证书内容（不包括签名）计算哈希
2. 用 CA 的私钥对哈希进行加密
3. 得到签名，附加到证书上
```

### 验证过程

浏览器验证证书时：

```
验证过程：
1. 用 CA 的公钥解密签名，得到哈希 A
2. 对证书内容计算哈希，得到哈希 B
3. 比较 A 和 B
4. 如果相同，证书未被篡改
```

```python
# 简化的签名验证
def verify_signature(certificate, ca_public_key):
    # 1. 解密签名
    signature_hash = decrypt(certificate.signature, ca_public_key)
    
    # 2. 计算证书内容的哈希
    content_hash = hash(certificate.content)
    
    # 3. 比较
    return signature_hash == content_hash
```

## 证书链

浏览器不会直接信任网站证书，而是信任 CA。

### 证书链结构

```
根证书（Root CA）：
  - 浏览器/OS 内置
  - 最权威，不可再分
  ↓ 签发
中间证书（Intermediate CA）：
  - CA 机构使用
  ↓ 签发
服务器证书（End-Entity Certificate）：
  - 网站实际使用
```

### 证书链验证

```
浏览器验证 example.com 证书：
1. example.com 证书说："我是 DigiCert 签发的"
2. 找 DigiCert 中间证书
3. 中间证书说："我是 DigiCert Root 签发的"
4. 找 DigiCert Root 证书
5. 浏览器内置 DigiCert Root
6. 验证根证书签名，一路回溯验证
```

```python
# 完整的证书链验证
def verify_certificate_chain(cert, trusted_certs):
    chain = [cert]
    
    # 构建证书链
    while cert.issuer != cert.subject:
        issuer = find_issuer_cert(cert.issuer, trusted_certs)
        if not issuer:
            raise VerificationError("Issuer not found")
        chain.append(issuer)
        cert = issuer
    
    # 验证每个证书的签名
    for i in range(len(chain) - 1):
        child = chain[i]
        parent = chain[i + 1]
        if not verify_signature(child, parent.public_key):
            raise VerificationError("Invalid signature")
    
    return True
```

## CA 体系

### 根 CA

根 CA 是信任链的顶端，由浏览器和操作系统内置：

```
常见根 CA：
- DigiCert
- GlobalSign
- Let's Encrypt
- Sectigo
- Comodo
```

### 中间 CA

根 CA 不直接签发服务器证书，而是通过中间 CA：

```
根 CA (Root)
  └── 中间 CA (Intermediate)
        └── 服务器证书 (Server Certificate)
```

这样做是为了保护根 CA 的私钥——如果中间 CA 的私钥泄露，可以吊销中间证书，而根 CA 不受影响。

### Let's Encrypt

Let's Encrypt 是免费、自动化、开放的 CA：

```
Let's Encrypt 的创新：
- 免费：不需要付费
- 自动化：通过 ACME 协议自动签发和续期
- 开放：证书透明，所有人都可查看

ACME 协议流程：
1. 服务器向 Let's Encrypt 证明域名控制权
2. Let's Encrypt 签发证书
3. 证书到期前自动续期
```

```bash
# 使用 certbot 获取 Let's Encrypt 证书
certbot certonly --webroot -w /var/www/html -d example.com
```

## 证书类型

### 按验证级别分类

| 类型 | 验证内容 | 适用场景 |
|---|---|---|
| DV（Domain Validation） | 域名控制权 | 个人网站、博客 |
| OV（Organization Validation） | 域名 + 组织信息 | 企业网站 |
| EV（Extended Validation） | 域名 + 组织 + 人工审核 | 金融、电商 |

### 按域名数量分类

| 类型 | 说明 | 示例 |
|---|---|---|
| 单域名证书 | 只保护一个域名 | example.com |
| 通配符证书 | 保护所有子域名 | *.example.com |
| 多域名证书 | 保护多个不同域名 | example.com, example.org |

## 证书吊销

当证书私钥泄露时，需要吊销证书。

### CRL（Certificate Revocation List）

发布被吊销证书的列表：

```
CRL 列表：
  - 证书序列号
  - 吊销时间
  - 吊销原因

缺点：
  - 列表可能很大
  - 客户端需要定期下载
  - 实时性差
```

### OCSP（Online Certificate Status Protocol）

实时查询证书状态：

```bash
# 查询证书状态
openssl ocsp -issuer chain.pem -cert server.pem \
  -url http://ocsp.example.com -text
```

### OCSP Stapling

服务器主动获取证书状态并附带在 TLS 握手中：

```
传统 OCSP：
  客户端 <------ OCSP 响应器
  客户端查询证书状态

OCSP Stapling：
  服务器 <------ OCSP 响应器
  服务器在 TLS 握手时附带 OCSP 响应
  客户端直接使用，无需额外查询
```

## 实际应用

### 生成自签名证书

```bash
# 生成私钥
openssl genrsa -out server.key 2048

# 生成证书
openssl req -new -x509 -key server.key -out server.crt -days 365

# 合并为 PEM 文件
cat server.crt server.key > server.pem
```

### 查看证书信息

```bash
# 查看证书详情
openssl x509 -in cert.pem -text -noout

# 查看证书链
openssl s_client -connect example.com:443 -showcerts </dev/null

# 验证证书链
openssl verify -CAfile ca-bundle.crt server.crt
```

### 证书格式转换

```bash
# PEM 转 DER
openssl x509 -in cert.pem -outform DER -out cert.der

# DER 转 PEM
openssl x509 -in cert.der -inform DER -out cert.pem

# PEM 转 PKCS12
openssl pkcs12 -export -in cert.pem -inkey key.pem -out bundle.p12
```

## 面试追问

### 证书链验证失败怎么办？

常见原因和解决方案：
- 证书过期：更新证书
- 域名不匹配：检查 CN 和 SAN
- 中间证书缺失：服务器需要发送完整证书链
- 根证书不受信：安装根证书

### 什么是 SAN（Subject Alternative Name）？

SAN 是证书的扩展字段，允许一个证书保护多个域名：

```bash
# 查看 SAN
openssl x509 -in cert.pem -noout -text | grep -A1 "Subject Alternative Name"
```

### 为什么不直接信任服务器证书？

因为任何人都可以声称自己是 example.com，生成自签名证书。CA 认证体系通过信任链，确保只有真正拥有域名的人才能获得可信证书。

### Let's Encrypt 证书和商业证书有什么区别？

功能上基本相同，Let's Encrypt 是 DV 证书，签发速度快、免费。商业证书提供 OV/EV 级别验证，提供更好的担保和客户支持。

## 总结

数字证书和 CA 体系是 HTTPS 的信任基础：

```
证书 = 域名 + 公钥 + CA 签名

验证过程：
1. 验证证书签名（CA 公钥解密）
2. 验证证书链（根 -> 中间 -> 服务器）
3. 验证域名匹配
4. 验证有效期
5. 检查吊销列表

信任链：
根 CA（内置）-> 中间 CA -> 服务器证书
```

理解证书体系，你才能理解 HTTPS 如何实现安全通信。
