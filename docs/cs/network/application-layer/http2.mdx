# HTTP/2 协议

2015 年，HTTP/2 标准发布，带来了 Web 性能的重大飞跃。

如果说 HTTP/1.1 是单车道，HTTP/2 就是多车道高速公路——在同一时间，更多的"车辆"（请求/响应）可以并行通过。

## HTTP/1.1 的痛点

在理解 HTTP/2 之前，先看看 HTTP/1.1 的问题：

### 队头阻塞

HTTP/1.1 支持持久连接，但响应必须按顺序返回：

```
请求 1：GET /api/users        (慢，需要 500ms)
请求 2：GET /api/products    (快，只需 50ms)
请求 3：GET /api/orders      (快，只需 50ms)

实际执行：
0ms    50ms   100ms  ...  500ms  550ms  600ms
|----GET 1 响应----|等待|响应2|响应3|
                 ↑
          被阻塞了
```

请求 2 和 3 被请求 1 堵住了，即使它们可以很快返回。

### 头部冗余

每个请求都携带大量重复头部：

```http
GET /api/users HTTP/1.1
Host: api.example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0...)
Accept: application/json
Accept-Language: zh-CN,zh;q=0.9
Cookie: session=abc123...

GET /api/products HTTP/1.1
Host: api.example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0...)  # 重复！
Accept: application/json                       # 重复！
Accept-Language: zh-CN,zh;q=0.9               # 重复！
Cookie: session=abc123...                     # 重复！
```

### 并发受限

浏览器限制每个域名 6-8 个并发连接。为了绕开这个限制，开发者发明了：
- 域名分片
- 资源合并（CSS sprites、bundle.js）
- 图片 Base64 内联

## HTTP/2 的核心特性

### 1. 多路复用（Multiplexing）

HTTP/2 在同一个 TCP 连接上，通过**流（Stream）**和**帧（Frame）**的概念，实现真正的并行传输：

```
HTTP/1.1：           HTTP/2：
连接 1：请求1       连接 1：Stream 1: 请求1
连接 2：请求2       Stream 2: 请求2
连接 3：请求3       Stream 3: 请求3
...                  ...
（多个连接）         （流可交叉）
```

每个流是双向的，包含多个帧。帧是最小传输单位，有不同类型（HEADERS、DATA、PRIORITY 等）。

```
Stream 1:
  HEADERS (请求头)
  DATA    (请求体)

Stream 2:
  HEADERS (请求头)
  DATA    (请求体)
```

不同流的帧可以交叉传输，互不阻塞。

### 2. HPACK 头部压缩

HTTP/2 使用 HPACK 算法压缩头部：

```
静态表（预定义）：
Index 1: :method GET
Index 2: :method POST
Index 3: :scheme http
...

动态表（维护见过的值）：
Index 62: Host: api.example.com
Index 63: User-Agent: Mozilla/5.0...
```

压缩示例：

```http
# 原文
:method: GET
:path: /api/users
:scheme: https
host: api.example.com

# 压缩后
[索引 2] [索引 62] [索引 63]
```

HPACK 还使用 Huffman 编码进一步压缩，通常能减少 80-90% 的头部大小。

### 3. 服务器推送（Server Push）

服务器可以主动推送资源，不需要客户端请求：

```http
请求：
GET /index.html

响应：
HTTP/2 200
Content-Type: text/html

[HTML 内容]

推送（服务器主动发送）：
HEADERS (stream 3) -> :status: 200, content-type: text/css
DATA (stream 3) -> CSS 内容

HEADERS (stream 5) -> :status: 200, content-type: image/png
DATA (stream 5) -> 图片内容
```

:::tip 服务器推送的用途
推送 CSS/JS/图片，减少往返次数。但实际使用中，需要谨慎：
- 推送太多会造成浪费（客户端可能有缓存）
- 推送太少没有意义
- 通常用于首屏关键资源
:::

### 4. 流优先级（Stream Priority）

HTTP/2 支持设置流优先级，确保重要资源优先传输：

```
Stream 1: 优先级 0 (最高) -> HTML
Stream 2: 优先级 1          -> CSS
Stream 3: 优先级 2          -> JavaScript
Stream 4: 优先级 3          -> 图片
```

高优先级流可以"抢占"带宽，确保关键资源优先加载。

## 二进制分帧

HTTP/2 最大的变化是从**文本协议**变成了**二进制协议**。

### HTTP/1.1 的文本格式

```http
GET /index.html HTTP/1.1\r\n
Host: example.com\r\n
\r\n
```

### HTTP/2 的二进制帧

```
+-----------------------------------------------+
|  Length (24 bits)                             |
+---------------+-------------------+-----------+
|   Type (8)    |   Flags (8)       |
+----------------------------------+-----------+
|           Stream Identifier (31 bits)          |
+-----------------------------------------------+
|                  Frame Payload                |
+-----------------------------------------------+
```

帧类型：
- `HEADERS`：头部帧
- `DATA`：数据帧
- `PRIORITY`：优先级帧
- `WINDOW_UPDATE`：窗口更新帧
- `SETTINGS`：设置帧
- `GOAWAY`：关闭连接帧

## HTTP/2 的问题

### TCP 队头阻塞

HTTP/2 的多路复用解决了 HTTP 层的队头阻塞，但 TCP 层的队头阻塞仍然存在：

```
Stream 1: |--A--|--B--|--C--|--D--|
Stream 2: |--1--|--2--|--3--|--4--|

TCP 包丢失：
     ↓
|--A--|--B--|--C--|  丢失  |--D--|
|--1--|--2--|--3--|--4--|      |
                        ↑
              整个 TCP 连接被阻塞
```

### 握手延迟

HTTP/2 强制使用 TLS，但 TCP + TLS 握手仍然需要 2-3 RTT。

## 浏览器支持与检测

### 检测网站是否使用 HTTP/2

```bash
# 使用 curl
curl -I --http2 https://example.com

# 使用 openssl
openssl s_client -connect example.com:443 -alpn h2

# Chrome 开发者工具
# Network 面板 -> Protocol 列 -> h2 表示 HTTP/2
```

### Nginx 启用 HTTP/2

```nginx
server {
    listen 443 ssl http2;
    ssl_certificate /path/to/cert.pem;
    ssl_certificate_key /path/to/key.pem;
    ...
}
```

### Node.js HTTP/2

```javascript
const http2 = require('http2');
const fs = require('fs');

const server = http2.createSecureServer({
    key: fs.readFileSync('server.key'),
    cert: fs.readFileSync('server.crt')
});

server.on('stream', (stream, headers) => {
    if (headers[':path'] === '/') {
        stream.respondWithFile('index.html', {
            'content-type': 'text/html'
        });
    }
});

server.listen(8443);
```

## 面试追问

### HTTP/2 一定比 HTTP/1.1 快吗？

不一定。在丢包率高的网络（如移动网络），HTTP/2 可能更慢。因为 TCP 的丢包会阻塞整个连接，而 HTTP/1.1 有多个连接，单连接丢包影响较小。

### 为什么 HTTP/2 要强制 TLS？

RFC 7540 规定 HTTP/2 over TLS 是必须的（h2），但 h2c（cleartext HTTP/2）技术上可行。不过主流浏览器都要求 TLS，所以 h2c 没有实际用途。

### HPACK 如何避免压缩攻击？

HPACK 有限制：
- 动态表大小有限制，防止内存耗尽
- 不对值进行压缩，只对索引进行编码
- 索引号是有序的，不会被篡改

### HTTP/2 的服务器推送现在用得多吗？

不多。因为：
- 实现复杂
- 推送策略难以确定
- HTTP/3 的普及替代了部分需求
- 浏览器缓存控制更智能

## 总结

HTTP/2 是 Web 性能的一次飞跃：

| 特性 | 解决的问题 |
|---|---|
| 多路复用 | HTTP 层队头阻塞、并发限制 |
| HPACK | 头部冗余 |
| 服务器推送 | 减少往返次数 |
| 流优先级 | 资源加载顺序 |
| 二进制分帧 | 解析效率 |

但 HTTP/2 没有解决 TCP 层的队头阻塞问题。这个问题，留给了 HTTP/3 和 QUIC。
