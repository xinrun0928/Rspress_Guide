# HTTPS 与 TLS 加密原理

当你在浏览器地址栏看到一个绿色的小锁图标时，这意味着你的访问是加密的——没有人能偷听你和服务器之间的对话。

HTTPS（HTTP Secure）就是 HTTP + TLS，让 Web 通信变得安全。

## HTTP 的问题

HTTP 是明文传输的：

```
HTTP 请求：
GET /api/users HTTP/1.1
Host: api.example.com
Cookie: session=abc123

HTTP 响应：
HTTP/1.1 200 OK
{"id":1,"name":"张三","balance":10000}
```

在咖啡厅的 WiFi 环境下：
- 黑客可以轻易截获你的请求和响应
- 可以看到你的 Cookie，冒充你的身份
- 可以修改请求或响应内容

HTTPS 就是来解决这个问题的。

## HTTPS 的加密机制

HTTPS 使用**混合加密**：

- **对称加密**：加密和解密使用相同的密钥，速度快
- **非对称加密**：加密和解密使用不同的密钥（公钥/私钥），速度慢但安全

HTTPS 的流程：
1. 使用非对称加密交换密钥
2. 使用对称加密传输数据

```
为什么要混合？
- 非对称加密：计算量大，适合少量数据（密钥交换）
- 对称加密：计算量小，适合大量数据（实际传输）
```

## TLS 握手流程

TLS（Transport Layer Security）是 HTTPS 的加密协议。它的任务是：在传输数据之前，双方协商出一个安全的密钥。

### TLS 1.2 握手（2-RTT）

```
客户端                                              服务器
  |                                                    |
  |------- ClientHello (支持的加密套件) ------>|
  |                                                    |
  |<------ ServerHello (选择的加密套件) -----|
  |<------ 证书 (服务器公钥 + 证书链) --------|
  |<------ ServerHelloDone --------------------|
  |                                                    |
  |------- ClientKeyExchange (预主密钥) ------->|
  |------- ChangeCipherSpec -------------------->|
  |------- Finished (加密的握手消息) --------->|
  |                                                    |
  |<------ ChangeCipherSpec --------------------|
  |<------ Finished (加密的握手消息) ---------|
  |                                                    |
  |<============ 加密数据传输 ================>|
```

1. **ClientHello**：客户端发送支持的加密算法列表、随机数
2. **ServerHello**：服务器选择加密算法、发送随机数
3. **证书**：服务器发送证书（包含公钥）
4. **ClientKeyExchange**：客户端生成预主密钥，用公钥加密后发送
5. **Finished**：双方用预主密钥计算出主密钥，之后用对称加密通信

### TLS 1.3 握手（1-RTT）

TLS 1.3 大幅简化了握手过程：

```
客户端                                              服务器
  |                                                    |
  |------- ClientHello (支持的加密套件 + 密钥共享) -->|
  |                                                    |
  |<------ ServerHello (选择的加密套件 + 密钥) -----|
  |<------ 证书 + 签名 + 服务器 Finished ------------|
  |                                                    |
  |<============ 加密数据传输 ================>|
```

关键改进：
- 密钥交换参数在第一次消息中就发送
- 1-RTT 即可开始加密传输
- 不再支持不安全的加密算法（如 RSA 密钥交换）

### TLS 1.3 的 0-RTT

对于重复连接，TLS 1.3 支持 0-RTT：

```
客户端                                              服务器
  |                                                    |
  |------- ClientHello (早期数据 + PSK) ------------->|
  |------- 加密的早期数据 ---------------------------->|
  |                                                    |
  |<------ ServerHello (密钥) ------------------------|
  |<------ 加密的握手消息 + 响应 ---------------------|
  |<============ 加密数据传输 ================>|
```

:::warning 0-RTT 的风险
0-RTT 使用之前会话的密钥加密数据，存在重放攻击风险。因此 0-RTT 数据通常是幂等的（如 GET 请求）。
:::

## 证书与 CA

### 为什么需要证书

非对称加密有个问题：如何确认"公钥"真的是对方的，而不是黑客伪造的？

```
攻击场景：
黑客在咖啡厅拦截了请求
把服务器的公钥换成黑客的公钥
客户端用黑客的公钥加密数据
黑客用黑客的私钥解密，拿到数据
```

### 数字证书

数字证书是由 **CA（Certificate Authority）** 签发的，包含：
- 网站域名
- 服务器公钥
- 证书有效期
- CA 签名

```
证书内容：
  Subject: CN=example.com
  Issuer: DigiCert Inc
  Public Key: 0x1234...
  Valid Until: 2027-01-01
  Signature: (DigiCert 的数字签名)
```

### 证书链

浏览器验证证书时，会检查证书链：

```
根证书（Root CA）：
  - 浏览器内置
  - 最权威

中间证书（Intermediate CA）：
  - 根证书签发
  - 证书颁发机构使用

服务器证书（Server Certificate）：
  - 中间证书签发
  - 网站使用
```

### HTTPS 握手时的证书验证

```python
# 简化的验证流程
def verify_certificate(cert, ca_certs):
    # 1. 检查证书是否过期
    if not cert.is_valid():
        raise CertificateExpired()
    
    # 2. 检查域名是否匹配
    if not cert.matches_domain(request_host):
        raise HostnameMismatch()
    
    # 3. 验证证书链
    issuer_cert = find_issuer(cert, ca_certs)
    if not verify_signature(cert.signature, issuer_cert.public_key):
        raise InvalidSignature()
    
    # 4. 检查证书是否在吊销列表
    if is_revoked(cert):
        raise CertificateRevoked()
```

## 加密套件

TLS 支持多种加密算法组合，每种组合叫做**加密套件**：

```
TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256

分解：
- TLS：协议版本
- ECDHE：密钥交换算法（前向保密）
- RSA：证书签名算法
- AES_128_GCM：对称加密算法
- SHA256：消息认证码算法
```

### 为什么 TLS 1.3 更好

TLS 1.3 只支持前向保密（Forward Secrecy）的算法：

```
RSA 密钥交换（TLS 1.2）：
  用服务器公钥加密预主密钥
  如果私钥泄露，可以解密之前的会话

ECDHE 密钥交换（TLS 1.3）：
  每次会话生成临时密钥
  即使私钥泄露，也无法解密之前的会话
```

## 实际应用

### 查看 HTTPS 证书信息

```bash
# 使用 openssl
openssl s_client -connect example.com:443 -showcerts

# 简化输出
openssl s_client -connect example.com:443 </dev/null 2>/dev/null | \
  openssl x509 -noout -subject -issuer -dates
```

### curl 测试 TLS 版本

```bash
# 强制使用 TLS 1.3
curl -v --tlsv1.3 https://example.com

# 查看支持的 TLS 版本
curl -v https://example.com 2>&1 | grep -i "ssl\|tls"
```

### Nginx 配置 TLS

```nginx
server {
    listen 443 ssl;
    
    # 证书配置
    ssl_certificate /path/to/fullchain.pem;
    ssl_certificate_key /path/to/privkey.pem;
    
    # TLS 版本（只启用 TLS 1.2 和 1.3）
    ssl_protocols TLSv1.2 TLSv1.3;
    
    # 加密套件（按优先级排序）
    ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256;
    
    # 前向保密
    ssl_prefer_server_ciphers on;
    
    # OCSP Stapling
    ssl_stapling on;
    ssl_stapling_verify on;
}
```

## 中间人攻击与 HTTPS

### HTTPS 能防止什么

- **窃听**：加密传输，无法获取明文
- **篡改**：消息认证码，发现内容被修改
- **冒充**：证书验证，确认服务器身份

### HTTPS 不能防止什么

- **恶意软件检测**：企业防火墙可以检查 HTTPS 流量（通过证书代理）
- **DNS 劫持**：HTTPS 加密内容但不加密域名
- **用户主动忽略证书错误**

## 面试追问

### 对称加密和非对称加密有什么区别？

对称加密使用相同密钥：加密和解密都用同一个密钥，速度快（如 AES）。非对称加密使用公钥和私钥：公钥加密、私钥解密（或反过来），速度慢但安全（如 RSA、ECDSA）。

### 什么是前向保密？

前向保密（Forward Secrecy）指即使服务器的长期私钥泄露，之前的会话密钥也不会泄露。TLS 1.3 强制使用 ECDHE 密钥交换，保证前向保密。

### 证书链验证失败的原因有哪些？

- 证书过期
- 证书域名不匹配
- 根证书不受信任（需要安装）
- 证书链不完整（服务器配置错误）
- 证书被吊销

### TLS 1.3 比 1.2 快在哪里？

1-RTT 握手（vs 2-RTT）、0-RTT 重复连接、更少的加密算法选择（减少协商时间）、更简单的握手流程。

## 总结

HTTPS = HTTP + TLS，通过混合加密保障安全：

- **非对称加密**：交换密钥（RSA、ECDHE）
- **对称加密**：传输数据（AES、ChaCha20）
- **证书**：验证服务器身份（CA 体系）

TLS 1.3 通过简化握手、支持 0-RTT、强制前向保密，成为更安全、更快速的加密协议。
