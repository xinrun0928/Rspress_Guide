# HTTP 版本演进

HTTP 是互联网最重要的协议之一。从 1991 年的雏形到今天的 HTTP/3，它经历了三十多年的演进。

每一次版本更新，都在解决前一代的性能瓶颈。理解这些演进，你才能理解为什么现代 Web 要这样优化。

## HTTP/0.9：一切的开始

1991 年，Tim Berners-Lee 发明了 HTTP，只有一个方法 GET，协议极其简单：

```http
GET /index.html
```

服务器返回 HTML：

```http
<!DOCTYPE html>
<html>...</html>
```

特点：
- 只有 GET 方法
- 没有请求头、响应头
- 只有 HTML 可以传输
- 无状态、无连接管理

这像是"Hello World"——简单到极致，但奠定了万维网的基础。

## HTTP/1.0：标准化的尝试

1996 年，HTTP/1.0 加入了请求头、响应头、状态码、方法扩展（POST、HEAD）：

```http
GET /index.html HTTP/1.0
Host: www.example.com
Accept: text/html

```

```http
HTTP/1.0 200 OK
Content-Type: text/html
Content-Length: 1234

<html>...</html>
```

### HTTP/1.0 的问题

**每个请求都要建立新的 TCP 连接**：

```
连接 1：GET /index.html
连接 2：GET /style.css
连接 3：GET /image.png
...
```

一个网页有几十个资源，就需要建立几十个 TCP 连接。TCP 握手、慢启动都是开销。

**队头阻塞**：一个请求必须等前一个请求的响应返回才能发送。

## HTTP/1.1：持久连接

1999 年，HTTP/1.1 引入了最重要的改进：**持久连接（Persistent Connection）**：

```
TCP 连接 1：
请求 1：GET /index.html
响应 1：200 OK + HTML
请求 2：GET /style.css  （复用同一连接）
响应 2：200 OK + CSS
请求 3：GET /image.png  （复用同一连接）
响应 3：200 OK + PNG
```

### HTTP/1.1 的其他改进

- **管线化（Pipelining）**：请求可以并行发出，不需要等待响应（但响应仍需按顺序返回）
- **分块传输（Chunked Transfer）**：支持动态内容的流式传输
- **新增方法**：PUT、DELETE、OPTIONS、PATCH
- **缓存控制**：`Cache-Control`、`ETag`、`Last-Modified`

### HTTP/1.1 的问题

**队头阻塞仍然存在**：虽然可以复用连接，但响应必须按顺序返回。如果第一个请求很慢，后续请求都要等。

**头部冗余**：每个请求都要携带大量重复的头部（Cookie、User-Agent 等）。

**并发受限**：浏览器通常限制每个域名 6-8 个并发连接。

```
为了解决并发限制，Web 开发者开始：
- 域名分片（img1.example.com, img2.example.com...）
- CSS/JS 合并（减少请求数量）
- 图片内联（Base64 编码到 HTML 中）
```

## HTTP/2：多路复用

2015 年，HTTP/2 带来了革命性的变化。

### 多路复用（Multiplexing）

HTTP/2 在同一个 TCP 连接上，可以并行发送多个请求和响应，互不阻塞：

```
Stream 1: GET /index.html
Stream 2: GET /style.css
Stream 3: GET /image.png
          ↓
      并行传输，无阻塞
          ↓
Stream 1: 响应 /index.html
Stream 2: 响应 /style.css
Stream 3: 响应 /image.png
```

### HPACK 头部压缩

HTTP/2 使用 HPACK 算法压缩头部：

- 静态表：常见头部（`:method GET`）用数字索引
- 动态表：记录之前出现的头部
- Huffman 编码：进一步压缩

```
传统方式：
GET / HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0...
Cookie: session=abc...

HTTP/2：
[压缩后的二进制数据，只有几字节]
```

### 其他特性

- **服务器推送（Server Push）**：服务器主动推送资源
- **流优先级（Stream Priority）**：可以指定请求的优先级
- **二进制分帧**：替代纯文本，更高效

### HTTP/2 的问题

**TCP 队头阻塞**：虽然 HTTP 层无阻塞了，但 TCP 层仍然有。如果一个 TCP 包丢失，整个连接都要等待。

**握手延迟**：即使 HTTP/2，也需要 TCP + TLS 握手。

## HTTP/3：QUIC 时代

2022 年，HTTP/3 正式成为 RFC 9114，基于 QUIC 协议。

### QUIC 的优势

- **UDP 传输**：避免 TCP 的队头阻塞
- **0-RTT / 1-RTT 连接**：减少握手延迟
- **连接迁移**：切换网络不丢连接
- **内置 TLS 1.3**：更安全的默认加密

### HTTP/3 的特性

```
HTTP/1.1：多个 TCP 连接
HTTP/2：1 个 TCP 连接，多个 HTTP 流
HTTP/3：1 个 QUIC 连接，多个 QUIC 流
```

HTTP/3 解决了 HTTP/2 的 TCP 层队头阻塞问题。

## 版本对比

| 特性 | HTTP/1.0 | HTTP/1.1 | HTTP/2 | HTTP/3 |
|---|---|---|---|---|
| 持久连接 | ❌ | ✅ | ✅ | ✅ |
| 多路复用 | ❌ | ❌ | ✅ | ✅ |
| 头部压缩 | ❌ | ❌ | HPACK | QPACK |
| 服务器推送 | ❌ | ❌ | ✅ | ✅ |
| 流优先级 | ❌ | ❌ | ✅ | ✅ |
| 队头阻塞 | N/A | HTTP 层 | TCP 层 | 无 |
| 传输协议 | TCP | TCP | TCP | QUIC/UDP |
| TLS | 可选 | 可选 | 强制 | 强制 |
| 连接建立 | 3 RTT | 3 RTT | 2-3 RTT | 0-1 RTT |

## 实际应用

### 查看浏览器使用的 HTTP 版本

```javascript
// 开发者工具 Network 面板
// 查看 Protocol 列：h2 表示 HTTP/2，h3 表示 HTTP/3
```

### curl 测试 HTTP/3

```bash
# 检查 curl 是否支持 HTTP/3
curl --version | grep HTTP3

# 强制使用 HTTP/3
curl --http3 https://example.com
```

### Nginx 配置

```nginx
# HTTP/2
server {
    listen 443 ssl http2;
    ssl_certificate /path/to/cert.pem;
}

# HTTP/3（需要 ngx_http_v3_module）
server {
    listen 443 ssl http3;
    http3 on;
}
```

## 面试追问

### HTTP/2 是否需要 TLS？

是的，RFC 规定 HTTP/2 必须使用 TLS（h2）。浏览器也强制要求 TLS。但这是规范要求，实际抓包可能发现非加密的 h2c（cleartext h2）。

### HTTP/3 现在普及了吗？

截至 2026 年，HTTP/3 的采用率持续上升。根据 W3Techs 统计，约 25-30% 的网站支持 HTTP/3。主流浏览器（Chrome、Firefox、Safari）已全面支持。

### 为什么 QUIC 能解决队头阻塞？

TCP 是字节流协议，一个包丢失会阻塞后续所有字节。QUIC 是消息流，每个流独立，如果一个流的包丢失，只影响那个流，其他流不受影响。

### HTTP/2 比 HTTP/1.1 快很多吗？

不一定。在高丢包率网络（如移动网络）中，HTTP/2 可能更慢，因为 TCP 的丢包会阻塞所有流。但在稳定的 WiFi 环境下，HTTP/2 的多路复用优势明显。

## 总结

HTTP 的演进是一个不断优化性能的过程：

- **HTTP/0.9-1.0**：从无到有，建立基础
- **HTTP/1.1**：持久连接解决连接复用问题
- **HTTP/2**：多路复用解决队头阻塞
- **HTTP/3**：QUIC 解决 TCP 队头阻塞和握手延迟

每一次演进，都是为了解决前一代的核心瓶颈。理解这些演进，你才能理解 Web 优化的本质。
