# DNS 解析流程

当你输入 `www.example.com` 时，你的电脑是怎么知道这台服务器在哪里的？

答案是 **DNS（Domain Name System）**——互联网的电话簿。

## 为什么需要 DNS

人类喜欢有意义的名称（如 `www.example.com`），但计算机喜欢数字地址（如 `93.184.216.34`）。

DNS 就是两者的翻译器：把域名转换成 IP 地址。

```
DNS 的核心功能：
域名 -> IP 地址

www.example.com -> 93.184.216.34
```

## DNS 的分布式架构

DNS 不是一台大服务器，而是全球分布的层级系统。

### 域名层级

域名是树状结构的：

```
.
├── com        (顶级域 TLD)
│   ├── example.com
│   │   ├── www.example.com
│   │   ├── api.example.com
│   │   └── blog.example.com
│   └── google.com
│       ├── www.google.com
│       └── mail.google.com
├── org
├── net
├── cn         (国家代码顶级域)
│   ├── baidu.cn
│   └── taobao.cn
└── ...       (其他顶级域)
```

### DNS 服务器层级

```
根域名服务器（Root Server）
  |
  └── .com 顶级域名服务器（TLD Server）
        |
        └── example.com 权威域名服务器（Authoritative Server）
              |
              └── www.example.com
```

## DNS 解析流程

当你访问 `www.example.com` 时，DNS 解析经历以下步骤：

### 1. 检查本地缓存

首先，你的电脑检查本地缓存：

```
1. 检查浏览器 DNS 缓存
2. 检查操作系统 DNS 缓存
3. 检查 hosts 文件（/etc/hosts 或 C:\Windows\System32\drivers\etc\hosts）
```

### 2. 查询本地 DNS 服务器

如果没有命中，向配置的本地 DNS 服务器发起查询：

```
本地 DNS 服务器（通常是 ISP 提供）：
- 114.114.114.114（中国电信/联通/移动）
- 8.8.8.8（Google）
- 1.1.1.1（Cloudflare）
```

### 3. 递归查询 vs 迭代查询

#### 递归查询（Resolver -> Local DNS）

本地 DNS 服务器代表客户端，向其他 DNS 服务器查询，直到获得答案：

```
客户端 -> 本地 DNS -> [递归查询]
  |
  v
本地 DNS 服务器：
  |
  +-> 根域名服务器 (.) -> .com TLD 服务器地址
  |
  +-> .com TLD 服务器 -> example.com 权威服务器地址
  |
  +-> example.com 权威服务器 -> www.example.com = 93.184.216.34
  |
  v
返回结果给客户端
```

#### 迭代查询（Local DNS -> 其他 DNS）

DNS 服务器之间通常使用迭代查询——每个服务器只回答"我不知道，但你可以问这个服务器"：

```
迭代查询过程：
1. 本地 DNS 问根服务器："www.example.com 的 IP 是？"
   根服务器答："我不知道，但 .com TLD 服务器知道"

2. 本地 DNS 问 .com TLD 服务器："www.example.com 的 IP 是？"
   .com TLD 服务器答："我不知道，但 example.com 的服务器知道"

3. 本地 DNS 问 example.com 权威服务器："www.example.com 的 IP 是？"
   权威服务器答："是 93.184.216.34"
```

### 4. 返回结果并缓存

本地 DNS 获得结果后：
1. 返回给客户端
2. 按 TTL 缓存结果

## DNS 记录类型

### A 记录

将域名指向 IPv4 地址：

```bash
# 查看 A 记录
dig A www.example.com

# 响应：
www.example.com.    86400    IN    A    93.184.216.34
```

### AAAA 记录

将域名指向 IPv6 地址：

```bash
dig AAAA www.example.com

# 响应：
www.example.com.    86400    IN    AAAA    2606:2800:220:1::
```

### CNAME 记录

别名记录，将域名指向另一个域名：

```bash
dig CNAME blog.example.com

# 响应：
blog.example.com.    3600    IN    CNAME    www.example.com.
www.example.com.    86400    IN    A    93.184.216.34
```

### MX 记录

邮件交换记录，指定处理邮件的服务器：

```bash
dig MX example.com

# 响应：
example.com.    3600    IN    MX    10 mail.example.com.
example.com.    3600    IN    MX    20 mail2.example.com.
```

### NS 记录

域名服务器记录，指定域名的权威 DNS 服务器：

```bash
dig NS example.com

# 响应：
example.com.    172800    IN    NS    a.iana-servers.net.
example.com.    172800    IN    NS    b.iana-servers.net.
```

## DNS 缓存

DNS 缓存大幅减少了查询时间。

### 缓存层级

```
1. 浏览器缓存
2. 操作系统缓存
3. 本地 DNS 服务器缓存
4. 递归 DNS 服务器缓存（如 114.114.114.114）
5. 权威 DNS 服务器（通常不缓存，只响应查询）
```

### TTL（Time To Live）

每条 DNS 记录都有一个 TTL 值，表示缓存时间：

```bash
# TTL 为 86400 秒（1 天）
example.com.    86400    IN    A    93.184.216.34

# TTL 为 300 秒（5 分钟）
api.example.com.    300    IN    A    1.2.3.4
```

```
缓存策略：
- TTL 长：减少 DNS 查询次数，但更新慢
- TTL 短：快速生效，但增加 DNS 查询压力
```

## 实际应用

### 使用 dig 查询 DNS

```bash
# 基础查询
dig www.example.com

# 指定 DNS 服务器
dig @8.8.8.8 www.example.com

# 只查看答案部分
dig www.example.com +short

# 查看完整查询过程（+trace）
dig www.example.com +trace

# 查看特定记录类型
dig A www.example.com
dig AAAA www.example.com
dig MX example.com
dig NS example.com
dig TXT example.com
```

### 使用 nslookup

```bash
# 基础查询
nslookup www.example.com

# 指定 DNS 服务器
nslookup www.example.com 8.8.8.8

# 查看 MX 记录
nslookup -type=MX example.com
```

### 使用 host

```bash
host www.example.com
host -a www.example.com  # 详细输出
```

## DNS 常见问题

### DNS 污染与劫持

DNS 污染：返回错误的 IP 地址（审查或攻击）

DNS 劫持：篡改 DNS 响应（恶意软件或 ISP）

```
解决方案：
- 使用可信的 DNS 服务器（8.8.8.8、1.1.1.1）
- 使用 DNS over HTTPS（DoH）
- 使用 DNS over TLS（DoT）
```

### DNS 攻击类型

| 攻击类型 | 描述 |
|---|---|
| DNS 欺骗 | 返回伪造的 IP 地址 |
| DNS 放大 | 利用 DNS 放大 DDoS 攻击 |
| DNS 隧道 | 通过 DNS 传输非 DNS 数据 |
| 缓存投毒 | 污染 DNS 缓存 |

## 面试追问

### DNS 用什么协议？

DNS 主要使用 UDP 53 端口。当响应超过 512 字节时，切换到 TCP。当 DNS-over-HTTPS（DoH）或 DNS-over-TLS（DoT）时，使用 HTTPS/TLS。

### 为什么有时候 DNS 解析很慢？

可能原因：
- DNS 服务器距离远
- 权威 DNS 服务器响应慢
- TTL 设置过长导致缓存失效
- 本地 DNS 服务器缓存命中率低
- 网络拥塞

### 什么是 DNS 预取（Preload）？

浏览器在解析 HTML 时，提前解析链接中的域名：

```html
<!-- DNS 预取 -->
<link rel="dns-prefetch" href="//static.example.com">
<link rel="dns-prefetch" href="//cdn.example.com">
```

### DNS 如何实现负载均衡？

DNS 可以返回多个 IP 地址，客户端随机或轮询选择：

```bash
dig A round-robin.example.com

# 响应：
round-robin.example.com.    300    IN    A    1.2.3.4
round-robin.example.com.    300    IN    A    5.6.7.8
round-robin.example.com.    300    IN    A    9.10.11.12
```

## 总结

DNS 是互联网的"电话簿"：

```
DNS 解析流程：
1. 检查本地缓存（浏览器、操作系统、hosts）
2. 查询本地 DNS 服务器
3. 递归/迭代查询（根 -> TLD -> 权威）
4. 返回结果并缓存

DNS 记录类型：
- A：域名 -> IPv4
- AAAA：域名 -> IPv6
- CNAME：域名别名
- MX：邮件服务器
- NS：域名服务器
```

理解 DNS，你才能理解互联网如何将域名转换为 IP 地址。
