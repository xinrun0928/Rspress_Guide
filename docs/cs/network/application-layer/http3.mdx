# HTTP/3：QUIC 替代 TCP

2012 年，Google 的一位工程师提出了一个大胆的想法：能不能把传输层从 TCP 换成 UDP？

这个想法在当时看起来相当激进——TCP 是互联网的基石，几乎所有可靠传输都依赖它。换掉它，就像在高速公路上突然换掉地基一样疯狂。

但这位工程师看到了一个无法回避的问题：**TCP 的队头阻塞，TCP 和 TLS 叠加的握手延迟，这些问题在 HTTP/2 时代被彻底暴露了。**

十年后，这个想法落地为 HTTP/3 和 QUIC 协议，成为互联网基础设施最大的一次升级。

## TCP 队头阻塞：一个遗留了 40 年的问题

TCP 诞生于 1974 年，设计目标是在不可靠的网络上实现可靠传输。但那个年代，网络带宽极度稀缺，丢包是常态。TCP 的设计哲学是：**宁可牺牲速度，也要保证可靠。**

这套思路在今天带来了问题。

想象一个场景：你打开了 100 个 HTTP/2 请求，它们共享同一个 TCP 连接。其中一个数据包丢了——哪怕只是 Stream 5 的一个帧丢了——TCP 会怎么做？

**整条连接上的所有数据都被暂停，直到丢失的包重传成功。**

```
Stream 1:  |--A--|--B--|--C--|  丢失  |--D--|
Stream 2:  |--1--|--2--|--3--|--4--|--5--|
Stream 3:  |--a--|--b--|--c--|--d--|--e--|

TCP 包丢失后：
         ↓
|--A--|--B--|--C--|  丢失  |------------ 等待重传 ------------|
|--1--|--2--|--3--|--4--|--5--|         ↑ 整个连接阻塞在这里
|--a--|--b--|--c--|--d--|--e--|         ↑ Stream 2 和 3 明明没事
```

这叫 **TCP 层的队头阻塞（Head-of-Line Blocking）**。HTTP/2 解决了 HTTP 层的队头阻塞，但对这个更深层的问题束手无策。

### HTTP/2 的尴尬

HTTP/2 引入了多路复用，理论上所有请求都可以在一条连接上并行。但 TCP 不知道什么是 HTTP 的"流"——它只看到一串字节流。

丢包时，TCP 停的是整条连接，而不是某个特定的流。这意味着：

- 在丢包率 2% 的移动网络下，HTTP/2 可能比 HTTP/1.1 还慢
- 6 条 HTTP/1.1 连接各自丢包各自重传，互不影响
- 1 条 HTTP/2 连接丢包，全部 100 个请求一起等

这就是 Google 工程师看到的痛点。

## QUIC 协议：把可靠性做到应用层

### 核心思路

QUIC（Quick UDP Internet Connections）做出了一个关键决定：**不再依赖 TCP 的可靠性保证，而是自己实现一套可靠传输机制，放在 UDP 之上。**

```
传统模型：
应用层 → HTTP/2 → TLS 1.3 → TCP → IP

QUIC 模型：
应用层 → HTTP/3 → QUIC（自己实现可靠传输）→ UDP → IP
```

为什么要用 UDP？因为 UDP 足够简单，它只管"发出去"，不做任何控制。把控制权完全交给 QUIC，反而更灵活。

### QUIC 的可靠性设计

QUIC 重新实现了 TCP 的核心功能：

| 功能 | TCP 实现 | QUIC 实现 |
|---|---|---|
| 丢包检测 | 超时重传 | ACK-ECN 或超时 |
| 流量控制 | 滑动窗口 | 流级窗口 + 连接级窗口 |
| 拥塞控制 | Reno/CUBIC | 插件化，可替换 |
| 连接建立 | 三次握手 | 0-RTT 或 1-RTT |
| 连接迁移 | 不支持 | 连接 ID 机制 |

关键是，QUIC 把"流"的概念做到了协议层，不再依赖 TCP 模拟。

### 丢包只影响一个流

这是 QUIC 最核心的改进。

```
QUIC 连接中，Stream 5 丢了一个包：

Stream 1:  |--A--|--B--|--C--|--D--|--E--|  继续传输
Stream 2:  |--1--|--2--|--3--|--4--|--5--|  继续传输
Stream 5:  |--a--|--b--|  丢失  |--d--|--e--|  只有这个流等重传

其他流完全不受影响！
```

每个流独立管理自己的序列号和丢包重传。一个流出了问题，只有这个流等重传，不会波及其他流。

## HTTP/3：QUIC 上的 HTTP

HTTP/3 就是运行在 QUIC 之上的 HTTP。它的语义和 HTTP/2 基本相同，但传输层换成了 QUIC + UDP。

### 连接建立：0-RTT 的威力

TCP + TLS 的经典握手：

```
TCP 三次握手：   SYN → SYN-ACK → ACK         1 RTT
TLS 握手：       ClientHello → ServerHello    1 RTT
数据传输：                                     1 RTT

总计：最少 3 RTT 才能开始传数据
```

QUIC 的握手把加密握手和传输握手合并了：

```
QUIC 1-RTT 握手：
ClientHello（带初始密钥）→ ServerHello（带密钥）→ 数据

QUIC 0-RTT 握手（第二次连接起）：
直接发送数据（使用上次会话密钥）
```

对于曾经建立过连接的客户端，QUIC 可以实现 **0-RTT**——握手和数据传输同时完成。

### 连接迁移：切换网络不断线

你有没有遇到过这种情况：在地铁里用手机，Wi-Fi 突然断了，自动切换到流量，结果页面直接崩了，需要刷新。

这是因为 TCP 连接绑定在"源 IP + 源端口 + 目标 IP + 目标端口"四元组上。网络切换后，四元组变了，连接就断了。

QUIC 引入了 **连接 ID（Connection ID）** 的概念。每个连接有一个稳定的 ID，不依赖网络地址。即使 IP 变了，只要连接 ID 能到达对端，连接就保持。

```
Wi-Fi 连接：
  192.168.1.100:54321 + 连接ID: abc123 → 服务器

切换到流量：
  10.0.0.200:54321 + 连接ID: abc123 → 服务器
  （连接 ID 不变，连接保持）
```

这对移动场景来说是巨大的体验提升。

## HTTP/1.1 vs HTTP/2 vs HTTP/3 对比

### 演进时间线

```
HTTP/0.9  (1991)  - 纯文本，简单 GET
HTTP/1.0  (1996)  - 增加了头部、MIME 类型
HTTP/1.1  (1999)  - 持久连接、管道化
HTTP/2    (2015)  - 多路复用、HPACK、服务器推送
HTTP/3    (2022)  - QUIC + UDP、0-RTT、连接迁移
```

### 核心特性对比

| 特性 | HTTP/1.1 | HTTP/2 | HTTP/3 |
|---|---|---|---|
| 传输层 | TCP | TCP | QUIC / UDP |
| 多路复用 | ❌ | ✅ HTTP 层 | ✅ 传输层 |
| 队头阻塞 | HTTP 层 | HTTP 层 | ❌ 无 |
| HPACK 压缩 | ❌ | ✅ | ✅ QPACK |
| 服务器推送 | ❌ | ✅ | ✅ |
| 0-RTT | ❌ | ❌ | ✅ |
| 连接迁移 | ❌ | ❌ | ✅ |
| TLS 版本 | 任意 | TLS 1.2+ | TLS 1.3+ |

### 在丢包率高的网络下表现

拿一个典型场景对比：页面有 50 个资源请求，网络丢包率 2%。

```
HTTP/1.1（6 个连接）：
- 丢包影响范围：单个连接的请求
- 恢复时间：每个连接独立恢复
- 实际体验：较慢但稳定

HTTP/2（1 个连接）：
- 丢包影响范围：整条连接
- 恢复时间：所有请求等待一个包重传
- 实际体验：可能比 HTTP/1.1 更慢

HTTP/3（1 个连接）：
- 丢包影响范围：单个流
- 恢复时间：只有相关流的请求等待
- 实际体验：显著优于前两者
```

## QPACK：HTTP/3 的头部压缩

HTTP/2 使用 HPACK 做头部压缩。HTTP/3 换成了 QPACK，但思路类似：通过索引表减少重复头部传输。

HPACK 最大的问题是：编码和解码必须严格按顺序进行。一个包乱序到达，解码器就无法处理后续的动态表更新。

QPACK 改用两个单向流（One-Way Streams）来传输动态表更新，允许乱序到达：

```
Client → Server 单向流：发送自己这边的动态表更新
Server → Client 单向流：发送服务器那边的动态表更新

数据流中：
- 引用索引号（查静态表或动态表）
- 动态表需要等对应单向流确认后才可用
```

这样，即使数据包乱序，也不影响头部解码。

## 部署 HTTP/3

### Nginx 配置（1.25+）

```nginx
server {
    listen 443 ssl;
    listen 443 quic;

    ssl_certificate     /path/to/cert.pem;
    ssl_certificate_key /path/to/key.pem;

    # HTTP/3 相关配置
    ssl_protocols             TLSv1.3;
    add_header Alt-Svc         'h3=":443"';
    add_header QUIC-Status     $quic;
}
```

### Caddy 配置（天然支持 HTTP/3）

```nginx
example.com {
    respond "Hello via HTTP/3!"
}
```

Caddy 2.0+ 默认开启 HTTP/3，不需要额外配置。

## 面试追问

### HTTP/3 一定比 HTTP/2 快吗？

不一定。在丢包率低、延迟稳定的网络下（比如企业内网），HTTP/2 和 HTTP/3 差距不大。HTTP/3 的优势主要体现在：

- 丢包率高的网络（移动网络、高峰期公网）
- 需要快速建立连接的场景（0-RTT）
- 频繁切换网络的移动场景（连接迁移）

### QUIC 为什么选择 UDP 而不是重新设计传输层？

重新设计一个传输层协议意味着要穿越 NAT、防火墙、运营商设备，这些设备只认识 TCP 和 UDP。选择 UDP，QUIC 可以穿越几乎所有现有网络设备，同时把控制逻辑完全做到端侧。

### HTTP/3 的推广面临哪些阻力？

- **NAT 设备兼容性**：部分老旧 NAT 设备对 UDP 处理有问题，可能丢包或限流
- **运营商 QoS 限制**：很多运营商对 UDP 流量限速，因为早期 P2P 滥用 UDP
- **中间设备审查**：一些防火墙会拦截非标准端口的 UDP 流量
- **服务器资源**：QUIC 的 CPU 开销比 TCP 高，因为所有加密和解密都在应用层

### QUIC 的加密和 TLS 是什么关系？

QUIC 内部集成了 TLS 1.3，加密握手和传输握手同时完成。不像 TCP + TLS 是两个独立层，QUIC 的加密是其设计的一部分，而不是事后叠加的。

## 总结

HTTP/3 的核心突破不是"更快的 HTTP"，而是**把传输层控制权还给应用层**：

- 用 QUIC 在 UDP 上重建可靠传输，解决了 TCP 的队头阻塞
- 用连接 ID 实现了真正的连接迁移
- 用 0-RTT 大幅降低了连接建立延迟

代价是：更复杂的客户端和服务器实现，对 UDP 流量的网络兼容性挑战，以及更高的 CPU 开销。

但这是值得的——毕竟，TCP 那个 1974 年的设计，已经撑了快五十年了。
