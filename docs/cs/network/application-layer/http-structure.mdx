# HTTP 协议结构

当你在浏览器地址栏输入 URL 并按下回车时，一场精心编排的"对话"开始了。你的浏览器（客户端）向服务器发送一个请求，服务器处理后返回响应。

HTTP（HyperText Transfer Protocol）就是这场对话的"语言"。理解它的结构，是理解整个 Web 的基础。

## HTTP 请求结构

一个 HTTP 请求由三部分组成：

```
请求行
头部字段 1
头部字段 2
...
空行（CRLF）

可选：消息体
```

### 请求行

请求行的格式是：`方法 URL 版本`

```http
GET /index.html HTTP/1.1
```

| 组成部分 | 说明 | 示例 |
|---|---|---|
| 方法 | 要执行的动作 | GET、POST、PUT |
| URL | 资源路径 | /index.html、/api/users |
| 版本 | HTTP 协议版本 | HTTP/1.1、HTTP/2 |

### 请求头部

头部字段提供请求的元信息：

```http
GET /index.html HTTP/1.1
Host: www.example.com
User-Agent: Mozilla/5.0
Accept: text/html,application/xhtml+xml
Accept-Language: zh-CN,zh;q=0.9
Accept-Encoding: gzip, deflate
Connection: keep-alive
```

常用请求头：

| 头部 | 说明 | 示例 |
|---|---|---|
| `Host` | 目标主机 | `Host: www.example.com` |
| `User-Agent` | 客户端信息 | `User-Agent: Mozilla/5.0...` |
| `Accept` | 可接受的响应类型 | `Accept: text/html` |
| `Accept-Encoding` | 可接受的内容编码 | `Accept-Encoding: gzip` |
| `Accept-Language` | 可接受的语言 | `Accept-Language: zh-CN` |
| `Content-Length` | 请求体长度 | `Content-Length: 1024` |
| `Content-Type` | 请求体类型 | `Content-Type: application/json` |
| `Cookie` | Cookie 数据 | `Cookie: session=abc123` |
| `Authorization` | 认证信息 | `Authorization: Bearer token` |
| `Referer` | 请求来源页面 | `Referer: https://google.com` |

### 空行

空行（`CRLF`）是 HTTP 头部的结束标记，分隔头部和消息体。

### 消息体

不是所有请求都有消息体。GET 请求通常没有，POST/PUT 请求有。

```http
POST /api/users HTTP/1.1
Host: api.example.com
Content-Type: application/json
Content-Length: 45

{"name":"张三","email":"zhang@example.com"}
```

## HTTP 响应结构

响应结构与请求类似：

```
状态行
头部字段 1
头部字段 2
...
空行（CRLF）

可选：消息体
```

### 状态行

状态行的格式是：`版本 状态码 状态文本`

```http
HTTP/1.1 200 OK
HTTP/1.1 404 Not Found
HTTP/1.1 500 Internal Server Error
```

### 响应头部

```http
HTTP/1.1 200 OK
Date: Mon, 29 Mar 2026 10:00:00 GMT
Server: nginx/1.24.0
Content-Type: text/html; charset=utf-8
Content-Length: 612
Connection: keep-alive
Cache-Control: max-age=3600
```

常用响应头：

| 头部 | 说明 | 示例 |
|---|---|---|
| `Content-Type` | 响应体类型 | `Content-Type: text/html` |
| `Content-Length` | 响应体长度 | `Content-Length: 1024` |
| `Content-Encoding` | 内容编码 | `Content-Encoding: gzip` |
| `Date` | 响应时间 | `Date: Mon, 29 Mar 2026` |
| `Server` | 服务器信息 | `Server: nginx/1.24.0` |
| `Set-Cookie` | 设置 Cookie | `Set-Cookie: session=abc123` |
| `Cache-Control` | 缓存策略 | `Cache-Control: max-age=3600` |
| `Last-Modified` | 资源最后修改时间 | `Last-Modified: ...` |
| `ETag` | 资源版本标识 | `ETag: "abc123"` |

### 消息体

响应通常包含消息体，如 HTML、JSON、图片等。

```http
HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 123

<!DOCTYPE html>
<html>
<head><title>Example</title></head>
<body>Hello World</body>
</html>
```

## 用 curl 观察 HTTP 结构

### 发送请求

```bash
# 发送简单的 GET 请求
curl -v https://example.com

# -v 显示详细信息
# 输出包含请求行、请求头、响应行、响应头、响应体
```

### 查看完整请求和响应

```bash
# 显示请求和响应的所有头部
curl -i https://example.com

# 只显示头部
curl -I https://example.com
# 或
curl -s -D - https://example.com -o /dev/null
```

### 发送 POST 请求

```bash
# 发送 JSON 数据
curl -X POST https://api.example.com/users \
  -H "Content-Type: application/json" \
  -d '{"name":"张三","email":"zhang@example.com"}'

# 发送表单数据
curl -X POST https://api.example.com/login \
  -d "username=zhang&password=123456"
```

### 保存响应

```bash
# 保存到文件
curl -o response.html https://example.com

# 下载并显示进度
curl -O https://example.com/file.zip
```

## HTTP/1.1 的 keep-alive

HTTP/1.1 默认使用持久连接（keep-alive），同一个 TCP 连接可以发送多个请求：

```
请求 1：
GET /index.html HTTP/1.1
Host: example.com

响应 1：
HTTP/1.1 200 OK
Content-Type: text/html

<html>...</html>

请求 2（复用同一个连接）：
GET /style.css HTTP/1.1
Host: example.com

响应 2：
HTTP/1.1 200 OK
Content-Type: text/css
```

```
HTTP/1.0 默认关闭 keep-alive，需要头部：
Connection: keep-alive

HTTP/1.1 默认开启，可以显式关闭：
Connection: close
```

:::tip keep-alive 的优势
- 减少 TCP 握手次数
- 减少 TCP 关闭连接的次数
- TCP 慢启动只发生一次
:::

## 消息体的编码

### Content-Length

服务器通过 `Content-Length` 告诉客户端消息体长度：

```http
Content-Length: 1234
```

客户端读取 1234 字节后，就知道消息体结束了。

### Chunked Transfer Encoding

当响应长度事先未知时，使用分块传输编码：

```http
HTTP/1.1 200 OK
Transfer-Encoding: chunked

5
Hello
6
 World
0

```

每个块的结构：
- 块大小（十六进制）
- CRLF
- 块内容
- CRLF
- 最后以 `0` 块结束

## 面试追问

### HTTP 是无状态协议，如何实现状态管理？

HTTP 本身不维护状态，但可以通过以下机制模拟状态：
- Cookie：服务器通过 `Set-Cookie` 设置，浏览器自动在后续请求中携带
- Session：在服务端存储用户状态，Cookie 只存储 Session ID
- Token（如 JWT）：客户端存储，服务器验证

### 为什么 HTTP 消息要用空行分隔头部和消息体？

因为头部长度不固定，且头部数量也不固定。空行（`CRLF`）是唯一的结束标记，解析时只要找到空行，就知道头部结束了。

### `Content-Length` 和 `Transfer-Encoding: chunked` 能同时使用吗？

不能。两者是互斥的。如果使用分块传输，`Content-Length` 应该不存在或不准确。

### HTTP 头部字段的大小写敏感吗？

字段名（如 `Content-Type`）不区分大小写，但字段值（value）可能区分，取决于具体字段的语义。

## 总结

HTTP 请求和响应的结构：

```
请求：请求行 + 请求头 + 空行 + 消息体（可选）
响应：状态行 + 响应头 + 空行 + 消息体（可选）
```

理解这个结构，你才能理解 HTTP 的一切。后续的 HTTP 方法、状态码、缓存、认证等，都是在这个框架上的扩展。
