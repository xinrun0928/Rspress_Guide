# ICMP 协议

你 ping 了一下服务器，显示"Request timed out"。但 ICMP 并不只是用来 ping 的。

ICMP（Internet Control Message Protocol）是网络层的"助手"协议。它本身不传输数据，而是用于传递控制消息和错误报告。当路由器无法正常转发数据包时，它会发送 ICMP 消息告诉发送者发生了什么问题。

理解 ICMP，是理解网络故障排查和网络安全防护的基础。

## ICMP 概述

### ICMP 的作用

ICMP 的设计目标很简单：**报告网络中的错误和异常情况**。

```
┌─────────────────────────────────────────────────────────────┐
│                    ICMP 工作场景                            │
│                                                             │
│  主机 A ──────→ 路由器 R ──────→ 主机 B                     │
│                    │                                        │
│                    ↓                                        │
│              ICMP 错误消息                                  │
│              "目标不可达"                                    │
│              "时间超时"                                     │
│              "参数问题"                                     │
└─────────────────────────────────────────────────────────────┘
```

常见 ICMP 应用：
- `ping`：测试两台主机是否可达（回显请求/应答）
- `traceroute`：追踪数据包经过的路由路径
- Path MTU Discovery：发现路径中的最大传输单元

### ICMP 报文结构

ICMP 报文封装在 IP 数据报中，协议号为 1：

```
┌─────────────────────────────────────────────────────────────┐
│                    ICMP 报文结构                             │
├─────────────────────────────────────────────────────────────┤
│  Type (1B)  │  Code (1B)  │       Checksum (2B)              │
├─────────────────────────────────────────────────────────────┤
│                    ICMP Payload (可变)                        │
└─────────────────────────────────────────────────────────────┘
```

### ICMP 报文类型

ICMP 报文分为两大类：**差错报告报文**和**查询报文**。

| Type | Code | 类型 | 说明 |
|---|---|---|---|
| 0 | 0 | Echo Reply | ping 响应 |
| 3 | 0-15 | Destination Unreachable | 目标不可达 |
| 4 | 0 | Source Quench | 源端抑制（已废弃） |
| 5 | 0-3 | Redirect | 重定向 |
| 8 | 0 | Echo Request | ping 请求 |
| 9 | 0 | Router Advertisement | 路由器通告 |
| 10 | 0 | Router Solicitation | 路由器请求 |
| 11 | 0-1 | Time Exceeded | 超过 TTL/重组超时 |
| 12 | 0-2 | Parameter Problem | 参数问题 |

## 目标不可达（Type 3）

当路由器无法转发数据包时，返回目标不可达消息：

| Code | 说明 | 常见场景 |
|---|---|---|
| 0 | 网络不可达 | 路由表中没有目标网络 |
| 1 | 主机不可达 | ARP 失败，目标主机不存在或 down |
| 2 | 协议不可达 | 目标主机的传输层不支持该协议 |
| 3 | 端口不可达 | UDP 端口无应用监听 |
| 4 | 需要分片但 DF 置位 | MTU 问题 |
| 5 | 源路由失败 | 严格源路由失败 |
| 6 | 目标网络未知 | 目标网络不存在 |
| 7 | 目标主机未知 | 目标主机不存在 |

```bash
# 常见场景

# 场景 1：主机不可达
$ ping 192.168.56.200
PING 192.168.56.200 (192.168.56.200): 56 data bytes
Request timeout for icmp_seq 0
# ARP 失败或主机不存在

# 场景 2：端口不可达
# UDP 目标端口无应用监听时，ICMP 会返回 Port Unreachable
```

:::tip 端口不可达与防火墙
如果目标端口不可达且防火墙丢弃了数据包，则不会有任何响应。ICMP 不可达消息可能被防火墙阻止，此时表现为"ping 通但端口不通"。
:::

## 超时消息（Type 11）

### TTLExceeded

当数据包经过的路由器数量超过 TTL 时，返回超时消息：

```bash
# traceroute 利用 TTL 超时发现路由路径
$ traceroute to 8.8.8.8, 30 hops max

 1  192.168.1.1      1.234 ms   1.123 ms   0.998 ms   # 本地网关
 2  10.0.0.1         5.678 ms   5.432 ms   5.321 ms   # ISP 路由器
 3  72.14.215.85    12.345 ms  11.987 ms  12.001 ms   # Google DNS
 4  8.8.8.8          15.678 ms  15.432 ms  15.321 ms
```

原理：
- 发送 TTL `=` `1` 的数据包，第一跳路由器返回超时
- 发送 TTL `=` `2` 的数据包，第二跳返回超时
- 以此类推，直到到达目标或超过最大跳数

### Fragmentation Required (Code 4)

数据包需要分片但设置了不分片标志（DF）：

```bash
# Path MTU Discovery 过程
# 1. 发送大包 + DF 标志
# 2. 收到 ICMP "需要分片但 DF 置位"
# 3. 减小包大小重试
# 4. 直到找到合适的 MTU
```

## Echo 请求与应答（Type 0, 8）

ping 命令使用 ICMP Echo：

```bash
# 基本的 ping
ping 8.8.8.8

# 指定数据包大小
ping -s 1472 8.8.8.8

# 指定次数
ping -c 4 8.8.8.8

# 持续 ping（Ctrl+C 停止）
ping -i 0.5 8.8.8.8
```

```
Echo Request (Type 8, Code 0)
┌─────────────────────────────────────────────────────────────┐
│  Type=8 │ Code=0 │            Checksum                       │
├─────────────────────────────────────────────────────────────┤
│           Identifier (16 bit)  │      Sequence Number (16 bit)│
├─────────────────────────────────────────────────────────────┤
│                    Data (可选，默认为时间戳)                  │
└─────────────────────────────────────────────────────────────┘

Echo Reply (Type 0, Code 0)
┌─────────────────────────────────────────────────────────────┐
│  Type=0 │ Code=0 │            Checksum                       │
├─────────────────────────────────────────────────────────────┤
│           Identifier (16 bit)  │      Sequence Number (16 bit)│
├─────────────────────────────────────────────────────────────┤
│                    Data (Echo Request 中携带的数据)          │
└─────────────────────────────────────────────────────────────┘
```

## 重定向（Type 5）

路由器发现数据包应该走另一条路时，发送重定向消息：

| Code | 说明 |
|---|---|
| 0 | 对网络的重定向 |
| 1 | 对主机的重定向 |
| 2 | 对服务类型（TOS）和网络的重定向 |
| 3 | 对服务类型（TOS）和主机的重定向 |

```bash
# Windows 路由表中的重定向标志
route print

# Linux 查看 ICMP 重定向
dmesg | grep "ICMP redirect"
```

:::warning 安全注意
ICMP 重定向可能被攻击者利用，诱使受害者修改路由表，实施中间人攻击。生产环境中通常禁用 ICMP 重定向。
:::

## ICMP 与防火墙

### 常见 ICMP 过滤规则

```bash
# Linux iptables - 允许必要的 ICMP，阻止其他

# 允许 Echo Request（ping）
-A INPUT -p icmp --icmp-type echo-request -j ACCEPT

# 允许 Echo Reply
-A INPUT -p icmp --icmp-type echo-reply -j ACCEPT

# 允许目标不可达
-A INPUT -p icmp --icmp-type destination-unreachable -j ACCEPT

# 允许超时
-A INPUT -p icmp --icmp-type time-exceeded -j ACCEPT

# 阻止其他 ICMP
-A INPUT -p icmp -j DROP
```

### ICMP 与 MTU Discovery

PMTUD（Path MTU Discovery）依赖 ICMP 不可达消息。如果防火墙阻止了 ICMP，可能导致"黑洞"：

```
问题：
发送大包 → 中间路由器需要分片但 DF 置位 → 返回 ICMP 不可达
     ↓
防火墙阻止 ICMP → 发送端不知道 MTU 限制
     ↓
持续重传大包 → 连接超时/失败

解决：
1. 不要完全阻止 ICMP
2. 或在发送端禁用 PMTUD（不推荐，降低性能）
```

## ICMPv6

IPv6 中的 ICMP 版本称为 ICMPv6，协议号仍为 58：

| Type | 说明 | IPv4 对应 |
|---|---|---|
| 1 | 目标不可达 | Type 3 |
| 2 | 数据包太大 | — |
| 3 | 超时 | Type 11 |
| 4 | 参数问题 | Type 12 |
| 128 | Echo 请求 | Type 8 |
| 129 | Echo 响应 | Type 0 |
| 133 | 路由器请求（RS） | Router Solicitation |
| 134 | 路由器通告（RA） | Router Advertisement |
| 135 | 邻居请求（NS） | 类似 ARP |
| 136 | 邻居通告（NA） | 类似 ARP |
| 137 | 重定向 | Type 5 |

### ICMPv6 的重要功能

**NDP（Neighbor Discovery Protocol）** 使用 ICMPv6 实现 ARP 的功能：

```bash
# Linux 查看 IPv6 邻居表（类似 ARP 表）
ip -6 neigh show

# 邻居请求/通告
# NS：请求目标的链路层地址
# NA：响应邻居请求
```

## 常见故障排查

### ping 不通

```bash
# 1. 先 ping 本地网关
ping 192.168.1.1

# 2. 再 ping 公网 DNS
ping 8.8.8.8
ping 114.114.114.114

# 3. 检查防火墙
# Linux
iptables -L -n | grep icmp
# Windows
netsh advfirewall firewall show rule name="File and Printer Sharing (Echo Request - ICMPv4-In)"
```

### traceroute 异常

```bash
# Linux traceroute（UDP 模式）
traceroute 8.8.8.8

# Windows tracert（ICMP 模式）
tracert 8.8.8.8

# 某些路由器不返回 ICMP 超时，导致路径显示为 *
# 这是正常的，不表示网络问题
```

### ICMP 导致的安全问题

| 威胁 | 描述 | 防护 |
|---|---|---|
| ICMP 洪水 | 大量 ICMP 包耗尽带宽 | 限制 ICMP 速率 |
| ICMP 重定向攻击 | 篡改路由表 | 禁用 ICMP 重定向 |
| ICMP 时间戳 | 信息泄露 | 禁用时间戳请求 |
| ICMP 路由跟踪 | 探测网络拓扑 | 限制 ICMP |

## 总结

ICMP 是网络层的辅助协议：

- ICMP 报文封装在 IP 数据报中，协议号 1
- Type 3 表示目标不可达，Type 11 表示超时，Type 0/8 用于 ping
- ICMPv6 增加了邻居发现功能，替代了 ARP
- 防火墙不要完全阻止 ICMP，否则会破坏 PMTUD
- ping 和 traceroute 是基于 ICMP 的故障排查工具
