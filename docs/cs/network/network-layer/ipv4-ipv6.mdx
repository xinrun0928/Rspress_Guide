# IPv4 与 IPv6 的区别

IPv4 地址早在 2011 年就已经耗尽了。NAT 技术让多个设备共享一个公网 IP，勉强续命。但 NAT 本身是一种"打补丁"的方案，它带来了额外的复杂度，破坏了端到端的网络模型。

IPv6 的出现，就是为了从根本上解决这个问题。128 位的地址空间，足以给地球上的每一粒沙子分配一个 IP。但十年过去了，IPv6 的普及率仍然不高。为什么？

理解 IPv4 和 IPv6 的区别，不仅是面试中的高频题，更是理解互联网基础设施演进的钥匙。

## IPv4 的局限性

在讨论 IPv6 之前，先看看 IPv4 到底有什么问题。

### 地址枯竭

IPv4 只有 32 位，约 43 亿个地址：

```
2^32 = 4,294,967,296 ≈ 43 亿
```

43 亿看起来很多，但考虑到：
- 全球人口 80 亿
- 每个人可能有手机、电脑、平板、IoT 设备
- 还有大量企业、政府机构

IPv4 地址早已不够用了。

### 地址空间分配不均

| 类别 | 地址范围 | 说明 |
|---|---|---|
| A 类 | `0.0.0.0` `~` `127.255.255.255` | 128 个 A 类网络，每个 1600 万主机 |
| B 类 | `128.0.0.0` `~` `191.255.255.255` | 2 万个 B 类网络 |
| C 类 | `192.0.0.0` `~` `223.255.255.255` | 200 万个 C 类网络 |

问题在于：
- A 类网络太大（如 `10.0.0.0/8`），大量地址浪费
- C 类网络太小，不够用
- 大量地址被预留，无法使用

### NAT 的代价

NAT（Network Address Translation）允许多个设备共享一个公网 IP：

```
┌─────────────────────────────────────────────────────────────┐
│                   NAT 工作原理                               │
│                                                             │
│  内网（私有 IP）           公网（公网 IP）                  │
│  ┌─────────────┐                                      │
│  │ 192.168.1.10 │ ─── NAT 转换 ───→ │ 203.0.113.5      │
│  │ 192.168.1.11 │ ─── NAT 转换 ───→ │ 203.0.113.5      │
│  │ 192.168.1.12 │ ─── NAT 转换 ───→ │ 203.0.113.5      │
│  └─────────────┘                                      │
│                                                             │
│  NAT 设备维护映射表：                                       │
│  内网 IP:Port  ↔  公网 IP:Port                           │
└─────────────────────────────────────────────────────────────┘
```

NAT 缓解了地址枯竭，但带来了问题：
- 破坏了端到端的网络模型
- NAT 设备成为性能瓶颈
- P2P 应用（如 VoIP、BitTorrent）实现复杂
- 日志追踪和安全审计困难

## IPv6 地址结构

### 地址格式

IPv6 地址是 128 位，通常用冒号十六进制表示：

```
传统写法（8 组，每组 4 位十六进制）：
2001:0db8:3c4d:0012:0000:0000:1234:5678

压缩写法（省略前导零）：
2001:db8:3c4d:12:0:0:1234:5678

进一步压缩（双冒号替代连续零）：
2001:db8:3c4d:12::1234:5678
```

### 地址类型

| 类型 | 前缀 | 说明 |
|---|---|---|
| 未指定地址 | `::/128` | 全 0，表示"无地址" |
| 环回地址 | `::1/128` | 本机，相当于 IPv4 的 `127.0.0.1` |
| 链路本地地址 | `FE80::/10` | 仅本地链路通信，类似于 IPv4 的 `169.254.0.0/16` |
| 唯一本地地址 | `FC00::/7` | 私有地址，相当于 IPv4 的私有地址 |
| 组播地址 | `FF00::/8` | 组播，类似 IPv4 的 `224.0.0.0/4` |
| 全节点组播 | `FF02::1` | 所有节点 |
| 全路由器组播 | `FF02::2` | 所有路由器 |
| Solicited-Node | `FF02::1:FFXX:XXXX` | 用于邻居发现 |
| 全球单播地址 | 其他 | 公网地址，2000::/3 开始 |

### 全球单播地址结构

IPv6 全球单播地址（非 EUI-64 格式）：

```
┌──────────────────────┬──────────────┬───────────────────────────────┐
│      全局前缀         │  子网 ID     │       接口 ID                │
│   Global Routing Prefix│ Subnet ID   │     Interface ID             │
│        48 位          │   16 位     │         64 位                 │
├──────────────────────┴──────────────┴───────────────────────────────┤
│                      2001:0db8:3c4d:0012:0000:0000:1234:5678        │
│                      └─────┘   └──┘  └──────────────┘                │
│                        前缀    子网      接口 ID                      │
└────────────────────────────────────────────────────────────────────┘
```

### EUI-64 接口 ID

接口 ID 可以通过 MAC 地址自动生成（EUI-64）：

```
MAC 地址：00:1A:2B:3C:4D:5E

步骤 1：插入 FFFE
00:1A:2B:FF:FE:3C:4D:5E

步骤 2：翻转第 7 位（U/L 位）
00 → 02（0 + 2 = 2，二进制 00000000 → 00000010）
02:1A:2B:FF:FE:3C:4D:5E

步骤 3：格式化为冒号分隔
0201:2BFF:FE3C:4D5E

最终 IPv6 地址：
2001:db8:3c4d:12:0201:2BFF:FE3C:4D5E
```

:::tip 为什么翻转第 7 位？
MAC 地址的第 7 位是 U/L（Universal/Local）位，表示地址是全局唯一（0）还是本地管理（1）。
EUI-64 格式需要将此位翻转，使本地生成的地址与全局地址区分。
:::

## 报文对比

### IPv4 报文头（20-60 字节）

```
┌────────┬────────┬───────────┬────────────┬──────────┬────────┐
│版本 4B │IHL 4B  │  TOS 8B   │       Total Length 16B        │
├────────┴────────┴───────────┴────────────┴──────────┴────────┤
│        Identification 16B        │Flags 3B│   Fragment Offset 13B  │
├─────────────────────────────────┴────────┴────────────────────────┤
│   TTL 8B   │   Protocol 8B    │      Header Checksum 16B          │
├────────────────────────────────────────────────────────────────┤
│                      Source IP Address 32B                       │
├────────────────────────────────────────────────────────────────┤
│                    Destination IP Address 32B                   │
├────────────────────────────────────────────────────────────────┤
│                    Options (Variable)                           │
└────────────────────────────────────────────────────────────────┘
```

### IPv6 报文头（40 字节，固定长度）

```
┌────────┬────────┬───────────────┬───────────────────────────────┐
│版本 4B │ Traffic Class 8B     │           Flow Label 20B         │
├────────┴────────┴───────────────┴───────────────────────────────┤
│        Payload Length 16B        │  Next Header 8B│  Hop Limit 8B│
├──────────────────────────────────────────────────────────────────┤
│                      Source IP Address 128B                       │
├──────────────────────────────────────────────────────────────────┤
│                   Destination IP Address 128B                    │
└──────────────────────────────────────────────────────────────────┘
```

### IPv6 的改进

| 改进点 | IPv4 | IPv6 |
|---|---|---|
| 地址长度 | 32 位 | 128 位 |
| 报文头长度 | 可变（20-60 字节） | 固定 40 字节 |
| Options | 在 options 字段中 | 使用扩展头 |
| Checksum | Header Checksum | 无（依赖上层） |
| Fragmentation | 发送和转发都支持 | 仅发送端（Path MTU Discovery） |
| Flow Label | 无 | 有（20 位） |

IPv6 报文头简化的原因：
- 移除 checksum，依赖上层（传输层）做校验
- Options 移入扩展头，灵活且高效
- 固定报文头长度，路由器处理更快

## IPv6 特殊地址

### 链路本地地址

前缀 `FE80::/10`，自动配置生成，用于本地链路通信：

```bash
# 查看链路本地地址
ip -6 addr show
# 输出类似：fe80::1%eth0

# 链路本地地址用途
# - 邻居发现（NDP）
# - 路由器发现
# - 自动地址配置
```

### IPv4 映射的 IPv6 地址

用于在纯 IPv6 环境中表示 IPv4 节点：

```
::ffff:192.168.1.100
# 写作压缩形式
::ffff:c0a8:0164
```

### 内嵌 IPv4 的 IPv6 地址

用于 IPv4/IPv6 过渡技术：

```
# IPv4 兼容地址（已废弃）
::192.168.1.100

# IPv4 映射地址
::ffff:192.168.1.100
```

## IPv6 的优势

### 1. 地址空间近乎无限

```
2^128 ≈ 3.4 × 10^38

这个数字有多大？
- 地球表面积：5.1 × 10^14 平方米
- 可以为地球上的每平方米分配 6.7 × 10^23 个 IPv6 地址
```

### 2. 端到端通信

IPv6 恢复了端到端的网络模型，无需 NAT：
- 任何设备都可以直接被访问
- P2P 应用更简单
- 安全性更好（IPsec 集成在协议中）

### 3. 自动配置

IPv6 支持无状态地址自动配置（SLAAC）：

```
1. 主机生成链路本地地址：FE80::/10 + EUI-64 接口 ID
2. 进行重复地址检测（DAD）
3. 向路由器发送 Router Solicitation
4. 路由器返回 Router Advertisement（包含前缀信息）
5. 主机组合前缀 + 接口 ID，得到全球单播地址
```

### 4. 简化的报文头

固定长度的报文头让路由器转发更快，减少了处理延迟。

### 5. 内置安全

IPsec（IP Security）是 IPv6 的内置组件（虽然 IPv4 也可以使用 IPsec，但 IPv6 是强制支持）。

### 6. 更好的移动性

IPv6 的移动性支持（Mobile IPv6）比 IPv4 更加高效，减少了三角路由问题。

## IPv6 的现状与挑战

### 普及率

截至 2024 年，全球 IPv6 普及率约 40%，各地区差异很大：

| 地区 | 普及率 | 说明 |
|---|---|---|
| 比利时 | `>` `70%` | 全球领先 |
| 美国 | ~55% | 主要 ISP 已支持 |
| 中国 | ~40% | 三大运营商已支持 |
| 印度 | ~70% | 移动网络推动 |
| 全球平均 | ~40% | 持续增长中 |

### 迁移挑战

1. **双栈部署**：同时运行 IPv4 和 IPv6，增加运维复杂度
2. **应用兼容性**：部分老应用不支持 IPv6
3. **DNS 配置**：需要同时配置 A 记录和 AAAA 记录
4. **安全设备**：防火墙、IDS/IPS 需要支持 IPv6
5. **运维经验**：IPv6 配置和排查经验不足

### 过渡技术

| 技术 | 说明 | 适用场景 |
|---|---|---|
| 双栈（Dual Stack） | 同时运行 IPv4 和 IPv6 | 过渡期首选 |
| NAT64/DNS64 | IPv6 客户端访问 IPv4 服务器 | IPv6 only 网络 |
| DS-Lite | 封装 IPv4 到 IPv6 | ISP 侧部署 |
| 6rd | IPv6 快速部署 | IPv4 网络迁移 |
| Teredo | IPv6 over UDP over IPv4 | 最后一公里 |

## 配置示例

### Linux 配置 IPv6

```bash
# 临时配置静态 IPv6
ip -6 addr add 2001:db8::100/64 dev eth0

# 启用 IPv6 自动配置
sysctl -w net.ipv6.conf.all.autoconf=1
sysctl -w net.ipv6.conf.default.autoconf=1

# 查看 IPv6 信息
ip -6 route show
ip -6 neigh show

# 禁用 IPv6（如果需要）
sysctl -w net.ipv6.conf.all.disable_ipv6=1
```

### Windows 配置

```powershell
# 查看 IPv6 配置
netsh interface ipv6 show addresses

# 添加 IPv6 地址
netsh interface ipv6 add address "以太网" 2001:db8::100

# 启用/禁用 IPv6
netsh interface ipv6 set interface "以太网" forwarding=enabled
```

### Nginx IPv6 配置

```nginx
server {
    listen 80;
    listen [::]:80;  # IPv6 监听

    server_name example.com;
    # ...
}

server {
    listen 443 ssl;
    listen [::]:443 ssl;  # IPv6 + SSL

    server_name example.com;
    # ...
}
```

## 总结

IPv4 和 IPv6 的核心区别：

| 对比项 | IPv4 | IPv6 |
|---|---|---|
| 地址长度 | 32 位 | 128 位 |
| 地址数量 | ~43 亿 | ~3.4 × 10^38 |
| 表示法 | 点分十进制 | 冒号十六进制 |
| 报文头 | 可变长度 | 固定长度 |
| Checksum | 有 | 无 |
| Options | IP Options | 扩展头 |
| 广播 | 支持 | 不支持，用组播替代 |
| 自动配置 | DHCP | SLAAC + DHCPv6 |
| NAT | 需要（解决地址不足） | 不需要 |
| 安全 | 可选（IPsec） | 内置（IPsec） |

IPv6 是未来，但 IPv4 不会消失。双栈部署是当前的主流策略，直到 IPv6 完全普及。
