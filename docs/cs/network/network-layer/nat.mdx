# NAT 网络地址转换

IPv4 地址在 2011 年就耗尽了，但我们今天还在用 IPv4。这其中，NAT（Network Address Translation）功不可没。

NAT 允许成千上万的设备共享少量公网 IP，通过在数据包经过时修改地址信息，巧妙地解决了地址枯竭问题。但 NAT 也带来了复杂性，破坏了端到端的网络模型。

理解 NAT，是理解现代网络架构和安全设计的基础。

## NAT 的基本原理

### 为什么需要 NAT？

IPv4 地址只有约 43 亿个，而全球有 80 亿人口。如果每个网民都需要一个公网 IP，地址早就耗尽了。

NAT 的解决方案：**让多个私有 IP 共享一个公网 IP**。

```
┌─────────────────────────────────────────────────────────────┐
│                    NAT 工作原理                              │
│                                                             │
│  内网主机                              公网                 │
│  ┌─────────┐                                           │
│  │ PC1    │ 192.168.1.10 ─── NAT 转换 ───→ 203.0.113.5  │
│  │ PC2    │ 192.168.1.11 ─── NAT 转换 ───→ 203.0.113.5  │
│  │ PC3    │ 192.168.1.12 ─── NAT 转换 ───→ 203.0.113.5  │
│  └─────────┘                                           │
│                                                             │
│  NAT 设备维护映射表：                                       │
│  ┌─────────────────────────────────────────────────┐      │
│  │ 192.168.1.10:12345 → 203.0.113.5:10001          │      │
│  │ 192.168.1.11:23456 → 203.0.113.5:10002          │      │
│  │ 192.168.1.12:34567 → 203.0.113.5:10003          │      │
│  └─────────────────────────────────────────────────┘      │
└─────────────────────────────────────────────────────────────┘
```

### NAT 的类型

| 类型 | 全称 | 说明 |
|---|---|---|
| 静态 NAT | Static NAT | 一对一映射，公网 IP 固定 |
| 动态 NAT | Dynamic NAT | 一对一映射，公网 IP 动态分配 |
| NAPT | Network Address Port Translation | 多对一，多个私网 IP 共用一个公网 IP（最常用） |
| SNAT | Source NAT | 修改源地址 |
| DNAT | Destination NAT | 修改目标地址（端口转发） |

## NAT 类型详解

### 静态 NAT

一对一映射，私网 IP 始终映射到同一个公网 IP：

```bash
# 配置示例（Cisco）
ip nat inside source static 192.168.1.10 203.0.113.10
ip nat inside source static 192.168.1.11 203.0.113.11

# 内网主机 192.168.1.10 → 公网显示 203.0.113.10
# 永远是固定的一对一映射
```

特点：
- 适合需要固定公网 IP 的场景（如服务器）
- 需要足够的公网 IP
- 外部可以直接访问内网主机（需要配合防火墙）

### 动态 NAT

从公网 IP 池中动态分配：

```bash
# 定义公网 IP 池
ip nat pool NAT-POOL 203.0.113.10 203.0.113.20 netmask 255.255.255.0

# 定义访问控制列表
access-list 1 permit 192.168.1.0 0.0.0.255

# 配置动态 NAT
ip nat inside source list 1 pool NAT-POOL

# 行为：
# - 第一个内网主机 → 203.0.113.10
# - 第二个内网主机 → 203.0.113.11
# - 第 11 个内网主机（IP 不够用）→ 等待
```

特点：
- 公网 IP 按需分配，提高利用率
- 但同时上网的主机数受限于公网 IP 数量
- 外部无法主动访问内网主机（IP 不固定）

### NAPT（PAT）

端口地址转换，多个私网 IP 共用同一个公网 IP：

```bash
# 最常见的 NAT 类型
# 也叫 PAT（Port Address Translation）、NAT Overload

# Linux iptables 配置
iptables -t nat -A POSTROUTING -s 192.168.1.0/24 -o eth0 -j MASQUERADE

# MASQUERADE 会自动选择出口 IP，并维护端口映射表
```

```bash
# NAT 映射表示例
┌─────────────────────────────────────────────────────────────┐
│                    NAPT 映射表                               │
├──────────────────┬────────────────────┬────────────────────┤
│  内部 IP:Port    │  外部 IP:Port       │  目标地址           │
├──────────────────┼────────────────────┼────────────────────┤
│  192.168.1.10:   │  203.0.113.5:       │  93.184.216.34:443  │
│  54321           │  10001              │                    │
├──────────────────┼────────────────────┼────────────────────┤
│  192.168.1.11:   │  203.0.113.5:       │  93.184.216.34:443  │
│  54321           │  10002              │                    │
└──────────────────┴────────────────────┴────────────────────┘

# 外部响应时，根据端口反向查找内网主机
```

特点：
- 允许多达 65535 台主机共用一个公网 IP（每个 IP:Port 组合唯一）
- 最常用的 NAT 类型
- 家用路由器默认使用

## SNAT 与 DNAT

### SNAT（Source NAT）

修改数据包的源地址，用于内网访问公网：

```bash
# Linux iptables SNAT
iptables -t nat -A POSTROUTING -s 192.168.1.0/24 -o eth0 -j SNAT --to-source 203.0.113.5

# 等价于
iptables -t nat -A POSTROUTING -s 192.168.1.0/24 -o eth0 -j MASQUERADE
```

### DNAT（Destination NAT）

修改数据包的目标地址，用于端口转发：

```bash
# 将公网 80 端口的流量转发到内网 192.168.1.100:8080
iptables -t nat -A PREROUTING -i eth0 -p tcp --dport 80 -j DNAT --to-destination 192.168.1.100:8080

# 内部访问外网时，将目标改为内网服务器（反向代理场景）
iptables -t nat -A OUTPUT -p tcp --dport 80 -j DNAT --to-destination 192.168.1.100:8080
```

DNAT 的典型应用：

| 场景 | 公网入口 | 内网目标 | 说明 |
|---|---|---|---|
| Web 服务器 | `:80` | `:8080` | 常见配置 |
| SSH 远程访问 | `:2222` | `:22` | 隐藏真实 SSH 端口 |
| 数据库访问 | `:3306` | `:3306` | 仅内网暴露 |
| HTTPS | `:443` | `:8443` | SSL 卸载 |

## NAT 的问题

### 端到端模型的破坏

NAT 最大的问题是破坏了互联网的端到端模型：

```
传统 Internet 设计：
┌─────────┐                              ┌─────────┐
│  主机 A │ ←────── 端到端通信 ────────→ │  主机 B  │
└─────────┘                              └─────────┘
  任何一方都可以主动发起连接

NAT 环境：
┌─────────┐         ┌──────────┐          ┌─────────┐
│  主机 A │ ←──NAT──│  路由器   │──NAT──→ │  主机 B  │
└─────────┘         └──────────┘          └─────────┘
  NAT 内部可以访问外部                   外部无法主动访问内部
  除非配置端口转发
```

后果：
- P2P 应用难以实现（双方都在 NAT 后）
- VPN、VoIP 需要特殊技术（STUN、TURN、ICE）
- 防火墙规则复杂

### 连接追踪的开销

NAT 设备需要维护连接追踪表（conntrack）：

```bash
# Linux 查看连接追踪表大小
cat /proc/sys/net/netfilter/nf_conntrack_max

# 查看当前连接数
cat /proc/sys/net/netfilter/nf_conntrack_count

# 查看连接追踪表
cat /proc/net/nf_conntrack | head
```

高并发场景下，conntrack 可能成为瓶颈：
- 内存占用大（每条约 300 字节）
- 查找速度影响转发性能
- 连接数过多时可能溢出

### 其他问题

| 问题 | 说明 |
|---|---|
| 日志记录困难 | 内部 IP 被替换，溯源困难 |
| QoS 复杂 | 难以基于 IP 做流量控制 |
| 某些协议不兼容 | FTP、SIP、H.323 等协议携带 IP 地址，需要 ALG |
| MTU 问题 | NAT 修改包长度，可能触发 PMTUD 问题 |

## NAT 穿透技术

为了解决 NAT 后的 P2P 通信问题，发展出多种穿透技术：

### STUN（Session Traversal Utilities for NAT）

```
┌─────────────────────────────────────────────────────────────┐
│                        STUN 原理                             │
│                                                             │
│  客户端 ────→ STUN 服务器                                    │
│    │         询问自己的公网 IP:Port                         │
│    ←───────                                          │
│   客户端知道自己的公网地址了                                 │
│                                                             │
│  但如果双方都在对称型 NAT 后，可能仍然无法直接通信            │
└─────────────────────────────────────────────────────────────┘
```

### TURN（Traversal Using Relays around NAT）

中继方案，双方都通过 TURN 服务器中转：

```bash
# TURN 是最后手段
# 所有流量经过 TURN 服务器
# 缺点：增加延迟，服务器带宽成本高
```

### ICE（Interactive Connectivity Establishment）

综合方案，优先尝试直连，失败后用 TURN：

```bash
# ICE 选择通信路径的顺序：
# 1. 直连（双方在同网络）
# 2. hole punching（STUN 协助）
# 3. TURN 中继（最后手段）
```

## NAT 配置示例

### Linux 配置完整 NAT 环境

```bash
# 1. 开启 IP 转发
echo 1 > /proc/sys/net/ipv4/ip_forward
# 永久生效
echo "net.ipv4.ip_forward=1" >> /etc/sysctl.conf

# 2. 配置内网网卡为 internal
iptables -t nat -A POSTROUTING -s 192.168.1.0/24 -o eth0 -j MASQUERADE

# 3. 配置端口转发（可选）
# SSH: 公网 2222 → 内网 22
iptables -t nat -A PREROUTING -p tcp -i eth0 --dport 2222 -j DNAT --to 192.168.1.10:22

# 4. 设置 conntrack 参数（高并发优化）
echo 200000 > /proc/sys/net/netfilter/nf_conntrack_max

# 5. 查看 NAT 状态
iptables -t nat -L -n -v
cat /proc/net/nf_conntrack
```

### 家用路由器 NAT

家用路由器通常配置为：
- LAN 口：`192.168.0.0/24` 或 `192.168.1.0/24`（私有地址）
- WAN 口：获取 ISP 分配的公网 IP（或 PPPoE 拨号）
- NAT：启用 NAPT，多设备共享公网 IP

```bash
# 路由器端口转发配置示例（通用）
# 外部端口：8080
# 内部 IP：192.168.1.100
# 内部端口：80
# 协议：TCP

# 效果：
# 访问 路由器公网IP:8080 → 转发到 192.168.1.100:80
```

## IPv6 与 NAT 的关系

IPv6 时代，还需要 NAT 吗？

```
IPv6 地址数量：
2^128 ≈ 3.4 × 10^38

理论上可以为地球每一粒沙子分配一个 IP

结论：IPv6 不需要 NAT
```

IPv6 的设计理念是**恢复端到端通信**：
- 每个设备都有公网地址
- 无需 NAT 转换
- P2P 通信天然支持
- 但 IPv6 仍支持 ULAC（Unique Local Address）作为私有地址

:::tip IPv6 也有"私有地址"
IPv6 的私有地址是 Unique Local Address（ULA，`fc00::/7`），用于内部网络，但不常用，因为每个设备都可以有公网地址。
:::

## 总结

NAT 是 IPv4 时代的重要技术：

- 静态 NAT：一对一映射，适合需要固定公网 IP 的服务器
- 动态 NAT：公网 IP 池动态分配
- NAPT/PAT：多对一，最常见的家庭和企业场景
- SNAT 修改源地址（内网上网），DNAT 修改目标地址（端口转发）
- NAT 解决了地址枯竭，但破坏了端到端模型
- IPv6 时代，NAT 将逐渐退出历史舞台
