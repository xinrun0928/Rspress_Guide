# 可扩展性设计：让系统「长大」

凌晨两点，你的系统刚突破了 1000 万用户。

然后，崩了。

不是代码 bug，不是机器故障——是系统「长不动了」。这就是可扩展性问题：**不是系统不够好，而是系统不够「大」**。

今天，我们来聊聊如何设计一个能「长大」的系统。

## 一、可扩展性的核心问题

### 1.1 什么是可扩展性

可扩展性有两种方向：

**垂直扩展（Scale Up）**：
- 提升单机配置：CPU 升级、内存扩大、磁盘换 SSD
- 优点：简单，改动小
- 缺点：有物理上限，一台机器再强也有限

**水平扩展（Scale Out）**：
- 增加机器数量，多台机器协同工作
- 优点：理论上无上限，加机器就能扛
- 缺点：复杂度高，需要解决分布式问题

:::tip 选择建议
大多数互联网公司选择水平扩展。只有在业务初期或资源有限时，才考虑垂直扩展。
:::

### 1.2 可扩展性面临的挑战

水平扩展说起来简单，做起来有五大难题：

**1. 无状态化**
- 服务不能保存用户状态
- 状态必须存储到外部（Redis、数据库）

**2. 数据分片**
- 数据量太大，单机存不下
- 需要分片存储

**3. 服务拆分**
- 单体应用太大，改动互相影响
- 需要拆分成多个独立服务

**4. 负载均衡**
- 多台机器如何分配请求
- 如何检测机器故障并剔除

**5. 分布式事务**
- 数据分片后，原本简单的事务变复杂了
- 如何保证跨分片的数据一致性

## 二、无状态化设计

### 2.1 什么是无状态服务

有状态服务的困境：用户 A 的请求被路由到服务器 1，Session 存在服务器 1 上。下一次请求被路由到服务器 2，找不到 Session，用户就「掉线」了。

无状态服务的解法：服务不存储任何用户状态，所有状态都放在外部（Redis、数据库）。任何请求可以被路由到任何服务器，服务器可以随时扩容或缩容。

```java
// 错误示例：有状态服务
public class StatefulOrderService {
    private Map<Long, Order> localCache = new HashMap<>();

    public Order getOrder(Long orderId) {
        // 每次都查本地缓存，多实例部署时会出现数据不一致
        return localCache.get(orderId);
    }
}
```

```java
// 正确示例：无状态服务
public class StatelessOrderService {
    private RedisTemplate<String, Object> redisTemplate;

    public Order getOrder(Long orderId) {
        // 状态存储在 Redis，任何实例都能访问
        String key = "order:" + orderId;
        return (Order) redisTemplate.opsForValue().get(key);
    }
}
```

### 2.2 Session 管理

用户登录后如何保持会话？两种主流方案：

**方案一：Redis Session**
使用 Spring Session + Redis，将 Session 存储在 Redis 中。优点是集中管理、支持集群；缺点是需要额外的 Redis 集群。

**方案二：JWT Token**
用户登录后签发一个 Token，后续请求带上 Token 即可。优点是服务端无状态；缺点是 Token 一旦签发无法撤销。

```java
public class SessionManagement {

    @Autowired
    private StringRedisTemplate redisTemplate;

    public String createSession(Long userId) {
        String sessionId = UUID.randomUUID().toString();
        String key = "session:" + sessionId;

        Map<String, String> sessionData = new HashMap<>();
        sessionData.put("userId", userId.toString());
        sessionData.put("loginTime", String.valueOf(System.currentTimeMillis()));

        redisTemplate.opsForHash().putAll(key, sessionData);
        redisTemplate.expire(key, 30, TimeUnit.MINUTES);

        return sessionId;
    }

    public Long validateSession(String sessionId) {
        String key = "session:" + sessionId;
        String userId = (String) redisTemplate.opsForHash().get(key, "userId");

        if (userId == null) {
            return null;
        }

        redisTemplate.expire(key, 30, TimeUnit.MINUTES);
        return Long.parseLong(userId);
    }
}
```

## 三、数据分片

### 3.1 分片策略

数据分片的核心是选择分片键。常见策略有四种：

**按 ID 哈希分片**：优点是数据分布均匀，缺点是按范围查询困难。适合主要按 ID 查询的场景。

**按时间分片**：优点是按时间范围查询快，缺点是可能出现热点（近期数据访问量大）。适合日志、订单等按时间查询的场景。

**按用户 ID 分片**：优点是同一用户的数据在同一分片，查询效率高；缺点是热点用户数据不均。适合用户维度查询为主的场景。

**一致性哈希分片**：优点是扩缩容时迁移数据少，缺点是实现复杂。适合数据量预估不确定、可能频繁扩容的场景。

```java
public class ShardingStrategy {

    public int hashSharding(long id, int shardCount) {
        return (int) (id % shardCount);
    }

    public String timeSharding(LocalDateTime time) {
        return String.format("%d%02d", time.getYear(), time.getMonthValue());
    }

    public int userSharding(long userId, int shardCount) {
        return (int) (userId % shardCount);
    }

    public int consistentHashSharding(long id, int virtualNodes) {
        long hash = murmurHash(id);
        return (int) (Math.abs(hash % virtualNodes));
    }
}
```

### 3.3 分片后的查询

分片后查询的复杂度增加，尤其是跨分片查询。

```java
public class ShardedQueryService {

    public List<Order> getAllOrders(Long userId) {
        List<Order> allOrders = new ArrayList<>();
        int shardCount = 16;

        // 并行查询所有分片
        List<Future<List<Order>>> futures = new ArrayList<>();

        for (int i = 0; i < shardCount; i++) {
            Future<List<Order>> future = CompletableFuture.supplyAsync(() -> {
                return queryOrdersFromShard(userId, i);
            });
            futures.add(future);
        }

        // 等待所有结果
        for (Future<List<Order>> future : futures) {
            try {
                allOrders.addAll(future.get());
            } catch (Exception e) {
                // 处理异常
            }
        }

        // 排序
        allOrders.sort((a, b) -> b.getCreateTime().compareTo(a.getCreateTime()));

        return allOrders;
    }

    private List<Order> queryOrdersFromShard(Long userId, int shardIndex) {
        return jdbcTemplate.query(
            "SELECT * FROM t_order WHERE user_id = ? AND shard_id = ?",
            new Object[]{userId, shardIndex},
            new BeanPropertyRowMapper<>(Order.class)
        );
    }
}
```

:::tip 优化建议
跨分片查询代价很高，设计系统时应尽量避免。如果必须跨分片查询，考虑：
- 并行查询 + 结果聚合
- 使用 ES 作为查询层
- 异构数据同步（如将数据同步到 ES 专门用于查询）
:::

## 四、服务拆分

### 4.1 拆分策略

微服务拆分有三种思路：

**按业务能力拆分**：将系统按业务流程拆分为独立服务。例如用户服务（注册、登录、个人信息）、订单服务（创建订单、取消订单）、支付服务（发起支付、退款）、商品服务（商品详情、库存）。

**按领域模型拆分**：将系统按业务领域拆分为不同域。例如用户域（用户、角色、权限）、订单域（订单、订单项、物流）、支付域（支付、账户、流水）。

**拆分粒度原则**：
- 避免过度拆分（微服务 → 迷你服务 → 单体退化版）
- 优先按业务边界拆分
- 避免过多的跨服务调用（调用链过长影响性能）

:::warning 注意
服务拆分的边界需要谨慎设计。拆分太粗达不到效果，拆分太细又会引入过多复杂度。记住：**微服务的目的是降低复杂度，不是为了微而微**。
:::

### 4.2 服务间通信

服务拆分后，服务间如何通信？

**同步调用**：使用 RestTemplate 或 OpenFeign 直接调用。适合实时性要求高的场景。

**异步调用**：使用消息队列解耦。适合可以异步处理的场景。

```java
public class ServiceCommunication {

    @Autowired
    private RestTemplate restTemplate;

    @Autowired
    private LoadBalancer loadBalancer;

    public User getUserWithOrder(Long userId, Long orderId) {
        // 同步调用用户服务
        String userServiceUrl = loadBalancer.choose("user-service");
        User user = restTemplate.getForObject(
            userServiceUrl + "/users/" + userId,
            User.class
        );

        // 同步调用订单服务
        String orderServiceUrl = loadBalancer.choose("order-service");
        Order order = restTemplate.getForObject(
            orderServiceUrl + "/orders/" + orderId,
            Order.class
        );

        user.setOrder(order);
        return user;
    }

    public CompletableFuture<User> getUserAsync(Long userId) {
        return CompletableFuture.supplyAsync(() -> {
            String userServiceUrl = loadBalancer.choose("user-service");
            return restTemplate.getForObject(
                userServiceUrl + "/users/" + userId,
                User.class
            );
        });
    }
}
```

## 五、负载均衡

### 5.1 负载均衡策略

负载均衡算法决定了请求如何分配到后端服务器。

**轮询（Round Robin）**：最简单的方式，请求依次分配到每台服务器。

**加权轮询（Weighted Round Robin）**：根据服务器权重分配请求，性能强的机器权重高。

**最少连接（Least Connections）**：将请求分配给当前连接数最少的服务器，适合长连接场景。

**一致性哈希**：相同 key 的请求分配到同一服务器，适合需要会话保持的场景。

```java
public class LoadBalancingStrategy {

    public String roundRobin(List<String> servers, AtomicInteger counter) {
        int index = counter.incrementAndGet() % servers.size();
        return servers.get(index);
    }

    public String weightedRoundRobin(List<ServerWeight> servers) {
        int totalWeight = servers.stream()
            .mapToInt(ServerWeight::getWeight)
            .sum();

        int random = ThreadLocalRandom.current().nextInt(totalWeight);

        int sum = 0;
        for (ServerWeight sw : servers) {
            sum += sw.getWeight();
            if (random < sum) {
                return sw.getServer();
            }
        }

        return servers.get(0).getServer();
    }

    public String leastConnections(Map<String, AtomicInteger> connections) {
        return connections.entrySet().stream()
            .min(Map.Entry.comparingByValue())
            .map(Map.Entry::getKey)
            .orElse(null);
    }

    public String consistentHash(String key, TreeMap<Long, String> ring) {
        long hash = murmurHash(key);

        Map.Entry<Long, String> entry = ring.ceilingEntry(hash);
        if (entry == null) {
            entry = ring.firstEntry();
        }

        return entry.getValue();
    }
}
```

### 5.2 健康检查

负载均衡器需要定期检查后端服务器的健康状态，及时剔除故障节点。

```java
public class HealthChecker {

    @Scheduled(fixedRate = 5000)
    public void checkHealth() {
        for (Server server : servers) {
            try {
                ResponseEntity<String> response = restTemplate.getForEntity(
                    server.getUrl() + "/health",
                    String.class
                );

                if (response.getStatusCode() == HttpStatus.OK) {
                    server.setHealthy(true);
                    server.setFailCount(0);
                } else {
                    server.setHealthy(false);
                }
            } catch (Exception e) {
                server.setHealthy(false);
                server.incrementFailCount();

                if (server.getFailCount() > 3) {
                    removeFromPool(server);
                }
            }
        }
    }

    public void onRequestFailed(Server server) {
        server.incrementFailCount();

        if (server.getFailCount() > 3) {
            server.setHealthy(false);
        }
    }
}
```

## 六、扩展性案例

### 6.1 秒杀系统扩展

秒杀系统是扩展性设计的集大成者，来看几个核心优化点：

**动静分离**：静态资源（CSS、JS、图片）放 CDN，动态请求（库存、订单）走服务器。秒杀页面缓存在用户浏览器，设置较长的缓存时间。

**读多写少：多级缓存**：
- L1：本地缓存（Guava/Caffeine）
- L2：Redis 缓存
- L3：数据库

读流程：先查 L1，未命中查 L2，都没查到查 L3 并回填 L2。

**写请求削峰**：写请求进入消息队列，异步处理订单。用户先得到「排队中」的响应，稍后收到抢单结果。

**分桶库存**：10000 件商品，分成 100 个桶，每个桶 100 件。用户请求被路由到不同桶，减少锁竞争。

### 6.2 数据库扩展

数据库扩展是一个渐进的过程，通常分为四步：

**第一步：读写分离**。主库负责写，从库负责读。通过主从复制保持数据同步。适合读多写少的场景。

**第二步：分库分表**。按用户 ID 分库，按订单 ID 分表。分片键选择很重要，直接影响查询效率。

**第三步：冷热分离**。热数据放在 Redis + SSD，冷数据放在 HDD + 归档。定期迁移冷数据。

**第四步：多活**。多个机房同时对外服务，数据实时同步。成本高，通常只有大公司才用。

## 七、面试高频问题

### Q1：如何设计一个可扩展的系统？

核心设计要点：

**无状态化**：服务不保存本地状态，状态外置到 Redis/数据库。这样任何请求可以被路由到任何服务器。

**水平扩展**：服务无状态后可以随时扩容，负载均衡分发请求。

**数据分片**：数据量大时分片存储，按业务选择分片键。

**服务拆分**：按业务边界拆分，避免过度拆分。

**异步化**：非核心路径异步化，削峰填谷。

### Q2：如何选择分片键？

选择原则：

**查询维度**：经常按这个字段查询，优先放在分片键。

**数据分布**：分布要均匀，避免热点。

**业务场景**：订单按用户 ID 分片，帖子按内容 ID 分片。

常见选择：用户 ID、订单 ID、时间。

### Q3：如何平滑扩容？

平滑扩容方案：

**迁移前准备**：部署新集群，数据同步。

**双写策略**：同时写入新旧集群，校验数据一致性。

**灰度切换**：先切换小部分流量，逐步扩大。

**旧集群下线**：确认无误后下线。

## 八、留白思考

我们讲了可扩展性的四大核心手段：无状态化、数据分片、服务拆分、异步化。

但你有没有想过：**扩展性是有代价的**。

无状态化后，每次请求都要查 Redis，网络延迟增加了。数据分片后，跨分片查询变得困难。服务拆分后，服务间调用链变长，故障排查变复杂。

所以，**不要过度设计扩展性**。只有当问题真正出现时，才去解决。

问题来了：如果你接手了一个已经「过度拆分」的系统，如何做服务合并？这就是另一个有意思的话题了。
