# 如何保证系统的高可用

你有没有想过：

- 为什么阿里云能做到 99.99% 的可用性（全年停机不超过 52 分钟）？
- 为什么你的系统一年要宕机好几次？
- 为什么有些系统能「永不停机」，而你的系统动不动就崩溃？

这些差距的背后，是**高可用设计**的差异。

今天，我们来深入探讨如何设计一个高可用的系统。

## 一、什么是可用性？

### 1.1 可用性的定义

可用性的计算公式：

```
可用性 = 正常运行时间 / (正常运行时间 + 停机时间)
```

几个关键的可用性指标：

| 可用性 | 年停机时间 | 说明 |
|--------|-----------|------|
| 99% | 3.65 天 | 基本可用 |
| 99.9% | 8.76 小时 | 不错 |
| 99.99% | 52.6 分钟 | 优秀 |
| 99.999% | 5.26 分钟 | 卓越 |

:::tip 每个 9 的代价
从 99.9% 到 99.99%，成本增加 10 倍；从 99.99% 到 99.999%，再增加 10 倍。
:::

### 1.2 可用性 vs 一致性

CAP 理论告诉我们：在分区发生时，必须在可用性和一致性之间选择。

实际选择：

- 大多数互联网系统 → 选择可用性（AP）
- 金融交易系统 → 选择一致性（CP）

但这不是绝对的：可以通过「降级」来平衡，核心流程强一致，非核心流程可用。

## 二、高可用架构原则

### 2.1 消除单点

单点故障是高可用的大敌。任何单个组件（数据库、Redis、服务实例）故障都会导致整个系统不可用。

**高可用方案**：
- 主从数据库：一个挂了另一个顶上
- Redis Sentinel/Cluster：自动故障检测和切换
- 服务多实例部署：任何实例挂了，其他继续服务

### 2.2 冗余设计

冗余是高可用的核心思想：用更多的资源换取更高的可用性。

**数据冗余**：主从复制、多副本存储、数据分片。

**服务冗余**：至少部署 2 个实例，使用负载均衡，Docker + Kubernetes 自动故障恢复。

**机房冗余**：多机房部署，数据同步，流量切换。

## 三、高可用技术方案

### 3.1 服务冗余

服务冗余的核心是健康检查和故障恢复。

**健康检查**：
1. 检查进程是否存活
2. 检查依赖是否可用（数据库、Redis）
3. 检查资源是否充足（CPU、内存）

**故障检测与恢复流程**：
1. 连续 N 次健康检查失败
2. 标记实例为不健康
3. 从负载均衡器移除
4. 启动新实例
5. 等待新实例就绪
6. 加入负载均衡器

```java
public class HealthCheckService {

    public boolean isHealthy() {
        // 1. 检查进程是否存活
        if (!process.isAlive()) {
            return false;
        }

        // 2. 检查依赖是否可用
        if (!checkDependencies()) {
            return false;
        }

        // 3. 检查资源是否充足
        if (cpuUsage > 90 || memoryUsage > 90) {
            return false;
        }

        return true;
    }

    private boolean checkDependencies() {
        if (!db.isConnected()) return false;
        if (!redis.isConnected()) return false;
        return true;
    }
}
```

### 3.2 限流与熔断

限流和熔断是保护系统的两道防线。

**限流器**：防止瞬时流量冲垮系统。使用滑动窗口算法统计请求次数。

**熔断器**：当下游服务故障时，快速失败而不是无限等待。熔断器有三种状态：

- **闭合（CLOSED）**：正常调用
- **打开（OPEN）**：直接返回降级响应
- **半开（HALF_OPEN）**：尝试调用，成功则恢复

```java
public class RateLimiter {
    private static final int MAX_REQUESTS = 1000;

    public boolean tryAcquire(String key) {
        // 滑动窗口限流
        String windowKey = "rate:" + key + ":" + getCurrentWindow();
        Long count = redis.incr(windowKey);

        if (count == 1) {
            redis.expire(windowKey, Duration.ofSeconds(60));
        }

        return count <= MAX_REQUESTS;
    }
}
```

```java
public class CircuitBreaker {

    public enum State {
        CLOSED,    // 关闭，正常调用
        OPEN,      // 打开，拒绝调用
        HALF_OPEN  // 半开，尝试调用
    }

    private State state = State.CLOSED;

    public String call(String service, String request) {
        if (state == State.OPEN) {
            return getFallbackResponse();
        }

        try {
            String response = callService(service, request);

            if (state == State.HALF_OPEN) {
                state = State.CLOSED;
            }

            recordSuccess();
            return response;

        } catch (Exception e) {
            recordFailure();

            if (failureCount > THRESHOLD) {
                state = State.OPEN;
            }

            return getFallbackResponse();
        }
    }
}
```

### 3.3 重试与幂等

**重试机制**：网络波动时自动重试，但要注意指数退避避免雪崩。

**幂等性保证**：重复请求不影响结果。使用唯一标识（idempotentKey）记录处理状态。

```java
public class RetryService {

    public Response callWithRetry(Request request) {
        int maxRetries = 3;
        int retryDelay = 100; // ms

        for (int i = 0; i < maxRetries; i++) {
            try {
                return callService(request);
            } catch (Exception e) {
                if (i == maxRetries - 1) {
                    throw e;
                }

                // 指数退避
                Thread.sleep(retryDelay * (i + 1));
            }
        }

        throw new RuntimeException("重试失败");
    }
}
```

```java
public class IdempotentService {

    public Response processWithIdempotency(String idempotentKey, Request request) {
        // 1. 检查是否已处理
        String result = redis.get("idempotent:" + idempotentKey);
        if (result != null) {
            return JSON.parseObject(result);
        }

        // 2. 标记处理中（防止并发重复处理）
        redis.setex("idempotent:" + idempotentKey + ":lock", 1,
            Duration.ofMinutes(1));

        try {
            // 3. 执行业务逻辑
            Response response = doProcess(request);

            // 4. 保存结果
            redis.setex("idempotent:" + idempotentKey,
                JSON.toJSONString(response),
                Duration.ofHours(24));

            return response;

        } finally {
            redis.delete("idempotent:" + idempotentKey + ":lock");
        }
    }
}
```

## 四、高可用存储

### 4.1 数据库高可用

**主从复制**：写操作只在主库，读操作可以路由到从库。

**自动切换**：监控主库状态，如果主库不健康，选择一个健康的从库提升为新的主库，更新路由配置。

```java
public class DatabaseHA {

    public void write(String sql) {
        master.execute(sql);
    }

    public Result read(String sql) {
        if (isReadonly(sql)) {
            return randomSlave().execute(sql);
        }
        return master.execute(sql);
    }

    public void detectAndSwitch() {
        if (!master.isHealthy()) {
            Slave newMaster = selectHealthySlave();
            newMaster.promote();
            updateRouting(newMaster);
        }
    }
}
```

### 4.2 Redis 高可用

**Redis Sentinel**：连接到 Sentinel 集群，Sentinel 自动发现主从，主库故障时自动切换。

**Redis Cluster**：数据分片存储在 16384 个槽中，每个槽对应一个节点，支持高可用和水平扩展。

## 五、降级与应急预案

### 5.1 降级策略

降级是当系统压力过大时的保命手段。通过关闭非核心功能，保证核心功能可用。

**降级级别**：

| 级别 | 说明 | 场景 |
|------|------|------|
| NORMAL | 全部功能正常 | 正常运营 |
| PARTIAL | 部分功能降级 | 轻度压力 |
| MINIMAL | 只保留核心功能 | 中度压力 |
| OFFLINE | 完全不可用 | 灾难级故障 |

```java
public class DegradationStrategy {

    public enum DegradeLevel {
        NORMAL,     // 正常
        PARTIAL,    // 部分降级
        MINIMAL,    // 最小可用
        OFFLINE     // 完全不可用
    }

    private DegradeLevel currentLevel = DegradeLevel.NORMAL;

    public Object getDegradedResponse(String feature) {
        switch (currentLevel) {
            case NORMAL: return getNormalResponse(feature);
            case PARTIAL: return getPartialResponse(feature);
            case MINIMAL: return getMinimalResponse(feature);
            case OFFLINE: return getOfflineResponse(feature);
        }
    }

    public Object getPartialResponse(String feature) {
        if (isCoreFeature(feature)) {
            return getNormalResponse(feature);
        }
        return getCachedResponse(feature);
    }
}
```

### 5.2 应急预案

每个工程师都应该有应急预案手册。

**常见故障场景及处理**：

**数据库故障**：
1. 立即切换到备用数据库
2. 停止写操作（如果有）
3. 通知 DBA
4. 分析故障原因
5. 修复并切回

1. 降级到本地缓存
2. 停止统计功能

1. 检查网络配置
2. 切换到备用网络

## 六、面试追问方向

### 问题一：如何设计一个 99.99% 可用的系统？

回答思路：

**1. 消除单点**
- 每个组件至少 2 个实例
- 自动故障检测和恢复

**2. 冗余设计**
- 数据多副本
- 服务多实例
- 多机房部署

**3. 降级熔断**
- 非核心功能可降级
- 熔断器防止级联故障

**4. 快速恢复**
- 完善的监控告警
- 自动化故障恢复
- 定期演练

### 问题二：如何处理缓存失效？

**缓存穿透**：布隆过滤器 + 缓存空值

**缓存击穿**：互斥锁 + 逻辑过期双检

**缓存雪崩**：过期时间随机化 + 多级缓存

降级方案：数据库直接查询、返回默认值、友好提示。

### 问题三：如何保证服务的高可用？

**服务冗余**：至少 2 个实例 + 负载均衡

**健康检查**：定期检查 + 快速失败

**优雅关闭**：停止接收新请求 → 处理完现有请求 → 再关闭

**限流熔断**：防止过载 + 防止级联故障

## 七、留白思考

高可用的本质是：**假设任何组件都可能故障，然后确保系统在故障时仍能提供服务**。

99.99% 的可用性不是运气，是设计出来的。

但你有没有想过：**追求高可用是要付出代价的**。

双机房部署，成本翻倍。三机房，成本再翻倍。99.999% 的系统，可能需要付出 100 倍于普通系统的成本。

所以，高可用不是越高越好，**适合业务场景的才是最好的**。

问题来了：如果你的系统只能做到 99.9%，但业务要求 99.99%，你有哪些降本增效的手段？这个问题，值得你深入思考。
