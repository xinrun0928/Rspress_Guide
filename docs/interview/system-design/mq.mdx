# 消息队列设计与实践

你有没有想过：

- 双十一订单处理，为什么不会卡死？
- 为什么消息队列能让系统解耦？
- 为什么有些系统加了 MQ 反而更慢了？

消息队列是分布式系统的核心组件，今天我们来深入探讨它的设计与实践。

## 一、为什么需要消息队列？

### 1.1 不用 MQ 的问题

:::details 同步调用的困境

**场景**：用户下单后，需要处理多个业务。

**同步方式**：订单服务依次调用库存服务扣库存、积分服务加积分、物流服务创建物流、营销服务检查优惠、通知服务发送通知。

**问题**：

1. **耦合严重**：任何一个服务挂了，整个流程失败
2. **扩展困难**：新增业务需要修改订单服务
3. **性能瓶颈**：所有服务串行执行

:::

### 1.2 MQ 带来的改变

:::details 引入消息队列后的变化

**架构**：订单服务发送消息到消息队列，各下游服务（库存、积分、物流、营销、通知）订阅并异步处理。

**优势**：

1. **解耦**：订单服务只管发消息，不管谁消费
2. **异步**：订单服务快速返回，异步处理
3. **削峰**：高峰期消息积压，平滑处理
4. **扩展**：新增业务只需订阅消息

:::

### 1.3 MQ 的三大核心作用

:::details 消息队列的三大作用

**1. 异步处理**

- 将同步操作变为异步
- 提升系统吞吐量
- 改善用户体验

**2. 削峰填谷**

- 接收瞬时高并发
- 异步慢慢处理
- 保护下游系统

**3. 系统解耦**

- 生产者和消费者解耦
- 可独立演进
- 故障隔离

:::

## 二、常见 MQ 对比

### 2.1 选型对比

| 特性 | RabbitMQ | Kafka | RocketMQ |
|---|---|---|---|
| 定位 | 企业级消息 | 大数据日志 | 互联网金融 |
| 单机 QPS | 1-10 万 | 10-100 万 | 10-50 万 |
| 消息顺序 | 支持 | 支持 | 支持 |
| 事务消息 | 不支持 | 不支持 | 支持 |
| 延迟消息 | 支持插件 | 不支持 | 支持 |

| 推荐场景 | 中小系统 | 日志、大数据 | 电商、金融 |

### 2.2 选型建议

选择建议：

1. **中小系统（< 1 万 QPS）**
   - RabbitMQ，简单够用

2. **大数据、日志（高吞吐）**
   - Kafka，性能首选

3. **电商、金融（事务、顺序）**
   - RocketMQ，功能全面

4. **混合场景**
   - 多 MQ 组合使用

## 三、核心问题与解决方案

### 3.1 消息丢失

:::details 消息丢失的场景与对策

**场景 1：生产者发送失败**

对策：Confirm 机制 + 消息持久化。

**场景 2：Broker 宕机**

对策：消息持久化到磁盘。

**场景 3：消费者处理失败**

对策：手动 ACK，确认后再提交。

:::

```java
/**
 * 消息可靠性保障
 */
public class MessageReliability {

    /**
     * 生产者：Confirm 机制
     */
    public void producerWithConfirm() {
        // 开启 Confirm 模式
        channel.confirmSelect();

        // 异步监听确认
        channel.addConfirmListener((deliveryTag, multiple) -> {
            // 消息发送成功
            System.out.println("Message confirmed: " + deliveryTag);
        }, (deliveryTag, multiple) -> {
            // 消息发送失败
            System.out.println("Message failed: " + deliveryTag);
            // 重试发送
            retrySend(message);
        });

        // 发送消息
        channel.basicPublish(exchange, routingKey,
            MessageProperties.PERSISTENT_TEXT_PLAIN,
            message.getBytes());
    }

    /**
     * 消费者：手动 ACK
     */
    public void consumerWithAck() {
        // 关闭自动 ACK
        channel.basicQos(10); // 每次拉取 10 条

        DeliverCallback deliverCallback = (consumerTag, delivery) -> {
            try {
                // 1. 业务处理
                processMessage(delivery);

                // 2. 手动 ACK
                channel.basicAck(delivery.getEnvelope().getDeliveryTag(), false);

            } catch (Exception e) {
                // 3. 失败：重新入队或发送到死信队列
                channel.basicNack(delivery.getEnvelope().getDeliveryTag(),
                    false, false);
            }
        };

        channel.basicConsume(queue, false, deliverCallback, consumerTag -> {});
    }
}
```

### 3.2 消息重复消费

:::details 消息重复的原因

1. **生产者重试**
   - 发送超时，重复发送

2. **Broker 重新投递**
   - 消费者未 ACK，Broker 重新投递

3. **消费者重启**
   - 正在处理的消息被重复投递

**解决方案：幂等消费**

:::

```java
/**
 * 幂等消费者
 */
public class IdempotentConsumer {

    private RedisTemplate<String, String> redis;

    /**
     * 方案一：Redis 去重
     */
    public void consumeWithRedisDedup(Message message) {
        String key = "msg:consumed:" + message.getId();

        // SETNX 保证原子性
        if (Boolean.TRUE.equals(redis.setIfAbsent(key, "1",
            Duration.ofHours(24)))) {
            // 第一次消费，处理消息
            doProcess(message);
        } else {
            // 已消费过，跳过
            System.out.println("Message already processed: " + message.getId());
        }
    }

    /**
     * 方案二：数据库唯一索引
     */
    public void consumeWithUniqueIndex(Message message) {
        try {
            // 使用 INSERT ... ON DUPLICATE KEY
            // 或者业务表中添加消息 ID 唯一索引
            messageDao.insertOrIgnore(message);
            doProcess(message);
        } catch (DuplicateKeyException e) {
            // 重复消息，跳过
            System.out.println("Duplicate message: " + message.getId());
        }
    }

    /**
     * 方案三：业务状态机
     */
    public void consumeWithStateMachine(Order order) {
        // 订单状态：CREATED -> PROCESSING -> COMPLETED
        // 只有 CREATED 才能转到 PROCESSING

        int updated = orderDao.updateStatus(order.getId(),
            "CREATED", "PROCESSING");

        if (updated > 0) {
            // 状态更新成功，处理订单
            doProcess(order);
        } else {
            // 状态已经是 PROCESSING，跳过
            System.out.println("Order already processed");
        }
    }
}
```

### 3.3 消息顺序

:::details 消息顺序问题

**问题场景**：

1. 订单创建 → 支付 → 发货 → 签收
2. 如果顺序乱了：先签收后发货？

**解决方案**：

1. **单队列顺序**：所有消息放到一个队列
2. **消息路由**：按某个字段（如 userId）路由到同一队列
3. **消费端保序**：一次只处理一条消息

:::

```java
/**
 * 顺序消息消费
 */
public class OrderedMessageConsumer {

    /**
     * Kafka：按 Partition 保证顺序
     */
    public void kafkaOrderedConsume() {
        // 发送时，按 userId 路由到同一 Partition
        int partition = Math.abs(userId.hashCode()) % partitionCount;
        producer.send(new ProducerRecord(topic, partition, userId, message));

        // 消费时，按 Partition 消费
        consumer.subscribe(Collections.singletonList(topic));
        consumer.assign(Collections.singletonList(
            new TopicPartition(topic, partition)));

        // 顺序消费
        while (true) {
            ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));
            for (ConsumerRecord<String, String> record : records) {
                processMessage(record.value());
            }
        }
    }

    /**
     * RocketMQ：按 MessageQueue 保证顺序
     */
    public void rocketMQOrderedConsume() {
        // 发送时指定 MessageQueue
        producer.send(message, (mqs, msg, arg) -> {
            // 按 userId 选择队列
            int index = userId.hashCode() % mqs.size();
            return mqs.get(index);
        }, userId);

        // 消费时使用 MessageListenerOrderly
        consumer.registerMessageListener((MessageListenerOrderly) (msgs, context) -> {
            // 按顺序逐条处理
            for (MessageExt msg : msgs) {
                processMessage(new String(msg.getBody()));
            }
            return ConsumeOrderlyStatus.SUCCESS;
        });
    }
}
```

### 3.4 消息积压

:::details 消息积压的处理

**原因分析**：

1. 消费者挂了
2. 消费者处理速度慢
3. 消费者数量不够

**解决方案**：

1. 紧急扩容消费者
2. 批量消费
3. 多线程并发处理
4. 跳帧消费（丢弃旧消息）

:::

```java
/**
 * 消息积压处理
 */
public class MessageLagHandler {

    /**
     * 方案一：扩容消费者
     */
    public void scaleOutConsumers() {
        // 增加消费者实例
        // Kafka/RocketMQ 自动重新分配 Partition
    }

    /**
     * 方案二：批量消费
     */
    public void batchConsume() {
        channel.basicQos(100); // 一次拉取 100 条

        DeliverCallback callback = (consumerTag, delivery) -> {
            // 批量处理
            List<Message> batch = new ArrayList<>();
            batch.add(delivery);

            // 批量业务处理
            processBatch(batch);

            // 批量 ACK
            channel.basicAck(delivery.getEnvelope().getDeliveryTag(), true);
        };
    }

    /**
     * 方案三：多线程并发消费
     */
    public void concurrentConsume() {
        ExecutorService executor = Executors.newFixedThreadPool(10);

        DeliverCallback callback = (consumerTag, delivery) -> {
            executor.submit(() -> {
                try {
                    processMessage(delivery);
                    channel.basicAck(delivery.getEnvelope().getDeliveryTag(), false);
                } catch (Exception e) {
                    // 失败处理
                    channel.basicNack(delivery.getEnvelope().getDeliveryTag(),
                        false, true);
                }
            });
        };
    }
}
```

## 四、高级特性

### 4.1 事务消息

```java
/**
 * RocketMQ 事务消息
 */
public class TransactionalMessage {

    /**
     * 发送事务消息
     */
    public void sendTransactionalMessage(Order order) {
        Transaction transaction = producer.createTransaction();

        try {
            // 1. 发送半消息（Half Message）
            // 半消息对消费者不可见
            producer.sendMessageInTransaction(order, transaction);

            // 2. 本地事务
            orderService.createOrder(order);

            // 3. 提交事务
            transaction.commit();

        } catch (Exception e) {
            // 4. 回滚事务
            transaction.rollback();
        }
    }

    /**
     * 事务监听器
     */
    public class TransactionListenerImpl implements TransactionListener {

        @Override
        public LocalTransactionState executeLocalTransaction(Message msg, Object arg) {
            try {
                // 执行本地事务
                orderService.createOrder((Order) arg);
                return LocalTransactionState.COMMIT_MESSAGE;

            } catch (Exception e) {
                return LocalTransactionState.ROLLBACK_MESSAGE;
            }
        }

        @Override
        public LocalTransactionState checkLocalTransaction(MessageExt msg) {
            // 查询本地事务状态
            String orderId = msg.getKeys();
            Order order = orderService.getOrder(orderId);

            if (order != null && order.isCommitted()) {
                return LocalTransactionState.COMMIT_MESSAGE;
            } else {
                return LocalTransactionState.ROLLBACK_MESSAGE;
            }
        }
    }
}
```

### 4.2 延迟消息

```java
/**
 * 延迟消息
 */
public class DelayedMessage {

    /**
     * 订单超时处理
     */
    public void sendDelayMessage(Long orderId) {
        // RocketMQ 延迟消息
        // 延迟级别：1s, 5s, 10s, 30s, 1m, 2m, ...
        Message message = new Message("order延时",
            JsonUtil.toJson(orderId).getBytes());
        message.setDelayTimeLevel(3); // 10 秒后投递

        producer.send(message);
    }

    /**
     * 消费超时订单
     */
    public void consumeTimeoutOrder() {
        consumer.subscribe("order延时", "*");

        consumer.registerMessageListener((MessageListenerConcurrently) (msgs, context) -> {
            for (MessageExt msg : msgs) {
                Long orderId = JsonUtil.fromJson(new String(msg.getBody()));

                // 检查订单状态
                Order order = orderService.getOrder(orderId);
                if (order.isUnpaid()) {
                    // 未支付，关闭订单
                    orderService.closeOrder(orderId);
                }
            }
            return ConsumeConcurrentlyStatus.CONSUME_SUCCESS;
        });
    }
}
```

## 五、面试追问方向

### 问题一：「如何保证 MQ 的高可用？」

**回答思路**：

```
RabbitMQ：
- 主从模式：镜像队列
- 集群：普通集群 + 镜像队列

Kafka：
- 多副本：每个 Partition 多个副本
- ISR：同步副本列表
- Controller：管理副本分配

RocketMQ：
- 多主从：Master-Slave
- Dledger：自动切换
```

### 问题二：「如何设计一个 MQ？」

**回答思路**：

```
核心组件：
1. Broker：消息存储和转发
2. Producer：消息发送
3. Consumer：消息消费
4. Registry：服务注册发现

核心功能：
1. 消息持久化：磁盘顺序写
2. 高效读取：零拷贝、PageCache
3. 分区并行：多 Partition 并行消费
4. 消息拉取：长轮询优化
```

### 问题三：「Kafka 和 RocketMQ 区别？」

**回答思路**：

```
Kafka：
- 优势：高吞吐、低延迟
- 适用：日志、大数据
- 特点：顺序写、零拷贝

RocketMQ：
- 优势：事务消息、延迟消息
- 适用：电商、金融
- 特点：阿里开源、功能全面
```

## 六、总结

:::details 消息队列设计要点

**核心问题**

- 消息丢失：Confirm + ACK + 持久化
- 重复消费：幂等消费
- 消息顺序：单队列 + 顺序消费
- 消息积压：扩容 + 批量 + 并发

**高级特性**

- 事务消息：RocketMQ 原生支持
- 延迟消息：订单超时等场景
- 消息路由：按标签、规则路由

**选型建议**

- 中小系统：RabbitMQ
- 大数据场景：Kafka
- 金融电商：RocketMQ

:::

> "消息队列的本质是：通过异步和解耦，让系统在正确的时间处理正确的事。它是分布式系统的灵魂。"

但你有没有想过：如果消息队列本身挂了怎么办？MQ 的高可用方案背后，又有哪些 trade-off？这个问题，值得你在生产环境中仔细观察和思考。
