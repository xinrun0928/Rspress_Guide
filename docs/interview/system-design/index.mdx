# 系统设计

面试官说：「设计一个 Twitter 吧。」

你深吸一口气，开始画架构图。画到一半，面试官打断：「等等，你先估算一下这个系统的规模。」

你愣住了。

系统设计面试考的不是「能不能背出 Twitter 的架构」，而是**你思考问题的方式**——会不会澄清需求、会不会量化规模、会不会权衡利弊、会不会在压力下保持条理清晰。

本模块覆盖系统设计面试的高频场景，从方法论到实战题，帮你建立分布式系统的全局设计思维。

## 面试四步法

遇到任何系统设计题，都按这个流程走：

```
第一步：Clarify Requirements —— 问清楚问题
→ 用户量、数据量、QPS、功能范围、非功能需求

第二步：Capacity Estimation —— 量化规模
→ 存储量、带宽、QPS、峰值估算

第三步：High-Level Design —— 画框架
→ 核心组件、数据流、技术选型

第四步：Deep Dive —— 深入设计
→ 数据分片、缓存策略、一致性保证、权衡取舍
```

面试不是独角戏，每一步都要**边说边和面试官确认**。

## 本模块核心公式

```
系统设计 = 问清规模 + 量化估算 + 框架设计 + 权衡取舍
```

没有完美的方案，只有最合适的方案。面试官想看到的是你的思考过程，而不是背出来的标准答案。

那么，系统设计面试中，面试官最喜欢追问的是什么？是「数据如何分片」，还是「缓存失效了怎么办」？下一节，我们从系统设计方法论开始，一步步建立你的系统设计思维。
