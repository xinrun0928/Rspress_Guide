# 微服务架构设计

你有没有想过：

- 为什么单体应用越做越大，最后变成了「巨石应用」？
- 为什么微服务喊了这么多年，真正落地成功的公司却不多？
- 为什么有些公司拆分微服务后，问题反而更多了？

今天，我们来深入探讨微服务架构的设计与实践。

## 一、什么时候需要微服务？

### 1.1 单体应用的困境

```
┌─────────────────────────────────────────────────────────┐
│                   单体应用的困境                           │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  1. 代码膨胀                                            │
│     - 几十上百万行代码                                  │
│     - 新人上手周期：3-6 个月                            │
│                                                         │
│  2. 部署耦合                                            │
│     - 修改一行代码，整个应用重新部署                      │
│     - 发布风险高，不敢轻易发布                           │
│                                                         │
│  3. 技术锁定                                            │
│     - 所有模块必须使用同一技术栈                         │
│     - 无法利用新技术红利                                │
│                                                         │
│  4. 扩展困难                                            │
│     - 只能整体扩展，无法按需扩展                         │
│     - 资源利用率低                                      │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 1.2 微服务的价值

```
┌─────────────────────────────────────────────────────────┐
│                   微服务的价值                            │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  1. 独立部署                                            │
│     - 每个服务可以独立部署                              │
│     - 快速迭代，降低发布风险                             │
│                                                         │
│  2. 技术多样性                                          │
│     - 不同服务可以选择不同技术栈                        │
│     - 新技术可以局部试点                                │
│                                                         │
│  3. 独立扩展                                            │
│     - 按需扩展瓶颈服务                                  │
│     - 资源利用率更高                                    │
│                                                         │
│  4. 团队自治                                            │
│     - 每个团队负责自己的服务                            │
│     - 减少沟通成本                                      │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 1.3 什么时候拆微服务？

```
不是所有系统都需要微服务！

适合微服务的场景：
1. 团队规模 > 10 人
2. 代码规模 > 10 万行
3. 需要快速迭代
4. 技术栈需要多样化
5. 系统有明显瓶颈，需要独立扩展

不适合微服务的场景：
1. 小团队（< 5 人）
2. 小系统（< 5 万行代码）
3. 初创公司，追求快速验证
4. 团队没有微服务经验
```

## 二、如何拆分微服务？

### 2.1 拆分原则

```
┌─────────────────────────────────────────────────────────┐
│                   微服务拆分原则                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  1. 单一职责原则                                        │
│     - 每个服务只负责一件事                              │
│     - 服务边界清晰                                      │
│                                                         │
│  2. 高内聚低耦合                                        │
│     - 服务内部高内聚                                    │
│     - 服务之间低耦合                                    │
│                                                         │
│  3. 业务边界清晰                                        │
│     - 按业务领域拆分                                    │
│     - 避免跨业务边界的数据查询                          │
│                                                         │
│  4. 渐进式拆分                                         │
│     - 不要一开始就拆分                                  │
│     - 从问题最严重的地方开始                            │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 2.3 拆分粒度

```
拆分粒度参考：

过粗：把所有功能放一个服务
       → 等于没拆分

适中：按业务领域拆分
       → 每个服务 1-3 人维护
       → 代码规模 1-5 万行

过细：每个类一个服务
       → 服务数量爆炸
       → 运维成本爆炸

推荐：
- 每个服务 2-3 人维护
- 每个服务代码规模 1-5 万行
- 服务数量控制在 10-50 个
```

```java title="src/main/java/GrpcExample.java"
// 定义 Protobuf
syntax = "proto3";

service OrderService {
    rpc CreateOrder(CreateOrderRequest) returns (CreateOrderResponse);
    rpc GetOrder(GetOrderRequest) returns (Order);
}

message CreateOrderRequest {
    string user_id = 1;
    repeated OrderItem items = 2;
}

message OrderItem {
    string product_id = 1;
    int32 quantity = 2;
}

message CreateOrderResponse {
    string order_id = 1;
    int32 status = 2;
}

// Java 实现
public class OrderServiceImpl extends OrderServiceGrpc.OrderServiceImplBase {
    
    @Override
    public void createOrder(CreateOrderRequest req, 
                          Observer<CreateOrderResponse> responseObserver) {
        // 处理订单创建
        Order order = orderService.create(req);
        
        CreateOrderResponse response = CreateOrderResponse.newBuilder()
            .setOrderId(order.getId())
            .setStatus(0)
            .build();
        
        responseObserver.onNext(response);
        responseObserver.onCompleted();
    }
}
```

### 4.2 限流与熔断

```java title="src/main/java/CircuitBreaker.java"
/**
 * 限流与熔断
 */
public class CircuitBreaker {
    
    private RateLimiter rateLimiter;
    
    /**
     * 限流
     */
    public boolean allowRequest(String userId) {
        // 1. 按用户限流
        String userKey = "rate:user:" + userId;
        
        // 2. 每秒最多 100 请求
        return rateLimiter.tryAcquire(userKey, 100);
    }
    
    /**
     * 熔断器
     */
    public void callWithCircuitBreaker(String serviceName, Runnable task) {
        // 使用 Resilience4j
        CircuitBreaker circuitBreaker = CircuitBreaker.ofDefaults(serviceName);
        
        Supplier<String> supplier = CircuitBreaker
            .decorateSupplier(circuitBreaker, () -> callService());
        
        Try<String> result = Try.ofSupplier(supplier)
            .recover(throwable -> fallback());
    }
    
    /**
     * 降级处理
     */
    private String fallback() {
        // 返回兜底数据
        return "Service temporarily unavailable";
    }
}
```

```java title="src/main/java/CrossServiceQuery.java"
/**
 * 跨服务数据查询

public class CrossServiceQuery {
    
    /**
     * 方案一：API 聚合
     */
    public OrderDetail getOrderDetail(Long orderId) {
        // 1. 查询订单
        Order order = orderService.getOrder(orderId);
        
        // 2. 查询用户信息
        User user = userService.getUser(order.getUserId());
        
        // 3. 查询商品信息
        List<Product> products = productService.getProducts(
            order.getItems().stream()
                .map(OrderItem::getProductId)
                .collect(Collectors.toList()));
        
        // 4. 聚合返回
        return OrderDetail.builder()
            .order(order)
            .user(user)
            .products(products)
            .build();
    }
    
    /**
     * 方案二：数据冗余
     */
    public void saveOrderWith冗余(Order order) {
        // 订单中冗余用户信息和商品信息
        order.setUserSnapshot(order.getUser().toSnapshot());
        order.setItemsSnapshot(order.getItems().stream()
            .map(Item::toSnapshot)
            .collect(Collectors.toList()));
        
        orderRepository.save(order);
    }

     * 方案三：CQRS
     */
    // Command Query Responsibility Segregation
    // 写操作和读操作分离
```

### 5.3 分布式事务

```java title="src/main/java/DistributedTransaction.java"
/**
 * 分布式事务处理
 */
public class DistributedTransaction {
    
    /**
     * 方案一：TCC
     */
    public void createOrderTCC(Order order) {
        // Try：预留资源
        inventoryService.tryDeduct(order.getItems());
        accountService.tryFreeze(order.getUserId(), order.getTotalAmount());
        
        // Confirm：确认执行
        inventoryService.confirmDeduct(order.getItems());
        accountService.confirmDeduct(order.getUserId(), order.getTotalAmount());
        orderRepository.save(order);
        
        // Cancel：取消执行
        // 如果 Confirm 失败，执行 Cancel
    }
    
    /**
     * 方案二：Saga
     */
    public void createOrderSaga(Order order) {
        SagaBuilder.builder()
            .addStep("deduct_inventory", 
                () -> inventoryService.deduct(order.getItems()),
                () -> inventoryService.compensate(order.getItems()))
            .addStep("deduct_balance",
                () -> accountService.deduct(order.getUserId(), order.getTotalAmount()),
                () -> accountService.compensate(order.getUserId(), order.getTotalAmount()))
            .addStep("create_order",
                () -> orderRepository.save(order),
                () -> orderRepository.delete(order.getId()))
            .execute();
    }
    
    /**
     * 方案三：最终一致性
     */
    public void createOrderEventually(Order order) {
        // 1. 保存订单（本地事务）
        orderRepository.save(order);
        
        // 2. 发送消息
        mqService.send("order:created", order);
        
        // 3. 库存服务消费消息扣减库存
        // 4. 如果失败，定时任务补偿
    }
}
```

## 七、面试追问方向

### 问题一：「微服务 vs 单体，如何选择？」

**回答思路**：

```
选择微服务：
- 团队规模大（> 10 人）
- 系统复杂度高
- 需要快速迭代
- 有微服务运维能力

选择单体：
- 团队规模小（< 5 人）
- 系统简单
- 初创公司，追求速度
- 没有微服务运维经验

核心原则：
- 不要为了微服务而微服务
- 从实际需求出发
```

### 问题二：「如何保证服务间的一致性？」

**回答思路**：

```
分布式事务方案：
1. 2PC：强一致，但性能差，不推荐
2. TCC：性能好，但代码侵入强
3. Saga：适合长事务
4. 最终一致性：大多数场景推荐

最终一致性实践：
1. 本地消息表 + MQ
2. 定时任务对账
3. 幂等设计
```

### 问题三：「微服务有哪些坑？」

**回答思路**：

```
常见问题：
1. 服务拆分过细
   - 服务数量爆炸
   - 运维成本激增

2. 分布式事务
   - 实现复杂
   - 性能损耗

3. 服务间依赖
   - 循环依赖
   - 接口版本管理

4. 数据一致性
   - 跨服务查询困难
   - 数据冗余同步

5. 运维复杂
   - 需要容器化
   - 需要服务治理
   - 需要监控告警
```
