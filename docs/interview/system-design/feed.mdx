# 设计信息流系统

你有没有想过：

- 刷微博时，为什么每次刷新都能看到新内容，但页面却没有刷新？
- 为什么有些大 V 的微博，你总要过一会儿才能看到？
- 为什么点赞后，点赞数能立即更新？

这些功能的背后，都是**信息流系统**在默默工作。但它的实现，远比表面看起来复杂得多。

## 一、信息流的核心问题

### 1.1 什么是信息流？

信息流（Feed）是指用户能够持续消费的内容列表。典型的产品有：

- **微博/推特**：用户关注的人发布的微博
- **抖音/快手**：平台推荐给你的视频
- **朋友圈**：好友发布的动态

### 1.2 信息流的核心问题

:::tip 性能问题
- 用户关注了 1000 人，如何快速拉取他们的最新动态？
- 每个用户都要个性化排序，性能如何保证？
:::

:::tip 一致性问题
- 大 V 发了一条微博，所有粉丝都要能看到
- 如何保证消息不丢失、不重复？
:::

:::tip 存储问题
- 微博总量超过 1 万亿条，如何存储？
- 每个用户的 Feed 列表如何高效存取？
:::

## 二、三种推送模式

### 2.1 推模式（Push）

推模式的核心是「写时推送」：

1. 用户 A 发布微博
2. 系统将微博推送到所有粉丝的收件箱
3. 粉丝直接读取自己的收件箱

**优点**：
- 用户读取 Feed 速度快（直接读本地收件箱）
- 适合粉丝数量较少的场景

**缺点**：
- 大 V 发微博，推送给 1000 万粉丝，压力大
- 用户收件箱存储成本高

```java
/**
 * 推模式实现
 */
public class PushFeedService {

    private FeedRepository feedRepository;
    private MQService mqService;

    /**
     * 发布微博（推模式）
     */
    public void publishPost(Post post) {
        // 1. 保存微博
        postRepository.save(post);

        // 2. 推送给所有粉丝（异步）
        List<Long> fans = fanService.getFans(post.getUserId());

        for (Long fanId : fans) {
            // 发送到消息队列，异步推送
            mqService.send("feed:push:" + fanId, post);
        }
    }

    /**
     * 获取用户 Feed（推模式）
     */
    public List<Post> getUserFeed(Long userId, int page, int size) {
        // 直接从收件箱读取
        return feedRepository.getInbox(userId, page, size);
    }
}
```

### 2.2 拉模式（Pull）

拉模式的核心是「读时拉取」：

1. 用户请求 Feed
2. 系统获取用户关注列表
3. 从关注列表中拉取最新微博
4. 按时间排序返回

**优点**：
- 写入压力小（只存微博本身）
- 不存在大 V 推送压力问题

**缺点**：
- 读取性能差（需要聚合多个来源）
- 热门话题响应慢

```java
/**
 * 拉模式实现
 */
public class PullFeedService {

    private PostRepository postRepository;

    /**
     * 获取用户 Feed（拉模式）
     */
    public List<Post> getUserFeed(Long userId, int page, int size) {
        // 1. 获取用户关注列表
        List<Long> following = followService.getFollowing(userId);

        // 2. 批量拉取关注列表的最新微博
        long sinceId = getLastReadPostId(userId); // 上一页最后一条微博 ID

        List<Post> allPosts = new ArrayList<>();
        for (Long followedUserId : following) {
            List<Post> posts = postRepository.getPostsByUser(
                followedUserId, sinceId, 100);
            allPosts.addAll(posts);
        }

        // 3. 按时间排序
        allPosts.sort((a, b) -> b.getCreatedAt().compareTo(a.getCreatedAt()));

        // 4. 返回分页结果
        return paginate(allPosts, page, size);
    }
}
```

### 2.3 混合模式（Hybrid）

混合模式的核心思想是：针对不同用户使用不同策略。

1. 普通用户：使用推模式（粉丝少，推送压力小）
2. 大 V 用户：使用拉模式（粉丝多，避免推送风暴）

这是业界最常用的方案，微博就是采用的这种模式。

```java
/**
 * 混合模式实现
 */
public class HybridFeedService {

    private static final long BIG_V_THRESHOLD = 100000; // 大 V 阈值：10 万粉丝

    /**
     * 发布微博（混合模式）
     */
    public void publishPost(Post post) {
        // 1. 保存微博
        postRepository.save(post);

        // 2. 判断发布者是否是大 V
        long fanCount = fanService.getFanCount(post.getUserId());

        if (fanCount < BIG_V_THRESHOLD) {
            // 普通用户：推送给所有粉丝
            pushToAllFans(post);
        } else {
            // 大 V：不推送，只写入微博表
            // 粉丝读取时，拉取大 V 的微博
        }
    }

    /**
     * 获取用户 Feed（混合模式）
     */
    public List<Post> getUserFeed(Long userId, int page, int size) {
        // 1. 获取推模式收件箱（普通用户发布的微博）
        List<Post> inboxPosts = feedRepository.getInbox(userId, page, size);

        // 2. 获取拉模式的大 V 微博
        List<Long> bigVs = followService.getBigVs(userId);
        List<Post> bigVPosts = pullBigVPosts(bigVs);

        // 3. 合并排序
        return mergeAndSort(inboxPosts, bigVPosts, page, size);
    }
}
```

### 3.2 用户收件箱存储

使用 Redis ZSet 存储用户收件箱：

- Key：`feed:inbox:{userId}`
- Score：微博发布时间戳
- Member：微博 ID

```java
/**
 * 用户收件箱存储
 */
public class InboxStorage {

    private RedisTemplate<String, String> redis;

    /**
     * 添加微博到用户收件箱
     */
    public void addToInbox(Long userId, Long postId, long timestamp) {
        String key = "feed:inbox:" + userId;
        redis.opsForZSet().add(key, postId.toString(), timestamp);

        // 限制收件箱大小（只保留最近 1000 条）
        redis.opsForZSet().removeRange(key, 0, -1001);
    }

    /**
     * 获取用户收件箱
     */
    public List<Long> getInbox(Long userId, int page, int size) {
        String key = "feed:inbox:" + userId;
        int start = page * size;
        int end = start + size - 1;

        Set<String> postIds = redis.opsForZSet()
            .reverseRange(key, start, end);

        return postIds.stream()
            .map(Long::parseLong)
            .collect(Collectors.toList());
    }

    /**
     * 分页获取（游标方式）
     */
    public List<Long> getInboxWithCursor(Long userId, long cursor, int size) {
        String key = "feed:inbox:" + userId;

        // 获取小于 cursor 的微博
        Set<String> postIds = redis.opsForZSet()
            .reverseRangeByScore(key, 0, cursor, 0, size);

        return postIds.stream()
            .map(Long::parseLong)
            .collect(Collectors.toList());
    }
}
```

## 四、性能优化

### 4.1 热门微博缓存

对于大 V 的微博，可以提前缓存到 CDN 或者热门池：

```java
/**
 * 热门微博缓存
 */
public class HotPostCache {

    private RedisTemplate<String, String> redis;
    private static final String HOT_POST_KEY = "feed:hot:posts";
    private static final int HOT_POST_COUNT = 1000;

    /**
     * 添加到热门池
     */
    public void addToHotPool(Post post) {
        // 微博热度分数：点赞 × 1 + 评论 × 3 + 转发 × 5
        double score = post.getLikeCount() * 1
                     + post.getCommentCount() * 3
                     + post.getRepostCount() * 5;

        redis.opsForZSet().add(HOT_POST_KEY, post.getId().toString(), score);

        // 保持热门池大小
        redis.opsForZSet().removeRange(HOT_POST_KEY, 0, -(HOT_POST_COUNT + 1));
    }

    /**
     * 获取热门微博
     */
    public List<Post> getHotPosts(int count) {
        Set<String> postIds = redis.opsForZSet()
            .reverseRange(HOT_POST_KEY, 0, count - 1);

        return postIds.stream()
            .map(id -> postRepository.findById(Long.parseLong(id)))
            .collect(Collectors.toList());
    }
}
```

### 4.2 异步粉丝分发

使用消息队列异步分发，避免同步推送的压力：

```java
/**
 * 粉丝分发服务
 */
public class FanoutService {

    private MQService mqService;

    /**
     * 异步分发到粉丝收件箱
     */
    public void fanoutPost(Post post) {
        // 发送到消息队列
        FanoutTask task = new FanoutTask();
        task.setPostId(post.getId());
        task.setUserId(post.getUserId());
        task.setTimestamp(post.getCreatedAt());

        mqService.send("feed:fanout", task);
    }

    /**
     * 消费者：处理粉丝分发
     */
    @KafkaListener(topics = "feed:fanout")
    public void handleFanout(FanoutTask task) {
        // 1. 获取粉丝列表（分批）
        long userId = task.getUserId();
        long cursor = 0;

        while (true) {
            List<Long> fans = fanService.getFansBatch(userId, cursor, 1000);
            if (fans.isEmpty()) {
                break;
            }

            // 2. 批量添加到收件箱
            for (Long fanId : fans) {
                inboxStorage.addToInbox(fanId, task.getPostId(), task.getTimestamp());
            }

            cursor = fans.get(fans.size() - 1);
        }
    }
}
```

### 4.3 读取未读数

```java
/**
 * 未读数服务
 */
public class UnreadCountService {

    /**
     * 获取未读数
     *
     * 策略：记录用户上次读取 Feed 的时间戳
     * 未读数 = 收件箱中时间戳 > 上次读取时间的微博数量
     */
    public long getUnreadCount(Long userId) {
        Long lastReadTime = getLastReadTime(userId);

        if (lastReadTime == null) {
            // 首次使用，返回收件箱总数
            return inboxStorage.getInboxSize(userId);
        }

        // 获取上次读取后的新微博数
        String key = "feed:inbox:" + userId;
        return redis.opsForZSet().count(key, lastReadTime + 1, Long.MAX_VALUE);
    }

    /**
     * 标记已读
     */
    public void markAsRead(Long userId) {
        // 记录当前时间
        long now = System.currentTimeMillis();
        redis.opsForValue().set("feed:lastRead:" + userId, String.valueOf(now));
    }
}
```

### 5.2 智能排序

考虑互动权重、亲密度、原创度等因素：

```java
/**
 * 智能排序
 */
public class SmartSort {

    /**
     * 计算微博得分
     */
    public double calculateScore(Post post, Long viewerId) {
        // 1. 时间衰减（越新的微博得分越高）
        double timeScore = calculateTimeScore(post.getCreatedAt());

        // 2. 互动权重（点赞、评论、转发）
        double interactionScore = post.getLikeCount() * 1
                                + post.getCommentCount() * 3
                                + post.getRepostCount() * 5;

        // 3. 亲密度（与 viewer 的互动历史）
        double intimacyScore = intimacyService.getIntimacy(viewerId, post.getUserId());

        // 4. 原创度（原创微博权重更高）
        double originalityScore = post.isOriginal() ? 1.5 : 1.0;

        // 综合得分
        return timeScore * 0.3 + interactionScore * 0.3
             + intimacyScore * 0.2 + originalityScore * 0.2;
    }

    /**
     * 获取排序后的 Feed
     */
    public List<Post> getSortedFeed(Long userId, int page, int size) {
        List<Long> postIds = inboxStorage.getInbox(userId, 0, 1000);

        return postIds.stream()
            .map(postId -> postRepository.findById(postId))
            .sorted((a, b) -> {
                double scoreA = calculateScore(a, userId);
                double scoreB = calculateScore(b, userId);
                return Double.compare(scoreB, scoreA); // 降序
            })
            .skip((long) page * size)
            .limit(size)
            .collect(Collectors.toList());
    }
}
```

## 六、面试追问方向

### 问题一：「Feed 系统的核心瓶颈是什么？」

**回答思路**：

Feed 系统的核心瓶颈是读性能：

1. 用户 Feed 需要聚合多个来源的数据
2. 热门内容需要快速响应
3. 存储成本高

**解决方案**：

1. 缓存热点数据
2. 使用异步处理
3. 预计算热门内容

### 问题二：「如何实现 Feed 的分页？」

**回答思路**：

方案一：Offset 分页

- 简单，但深度翻页性能差
- Feed 数据不断增长，Offset 越大越慢

方案二：Cursor 分页（推荐）

- 基于时间戳或 ID 的分页
- 性能稳定，适合无限滚动
- 缺点：不能跳页

### 问题三：「如何处理大 V 的推送风暴？」

**回答思路**：

问题：1000 万粉丝的大 V 发微博，同步推送压力巨大。

**解决方案**：

1. 大 V 不推，只写微博表
2. 粉丝读取时，单独拉取大 V 的微博
3. 大 V 的微博写入热门池缓存
4. 定时任务预计算热门 Feed

## 七、总结

推拉模式各有优劣，混合模式是业界最优解。

:::tip 信息流系统设计要点

**推送模式**

- 推模式：读取快，写入压力大
- 拉模式：写入简单，读取压力大
- 混合模式：大 V 拉取，普通用户推送（业界最优）

**存储设计**

- 微博表：核心数据存储
- 收件箱：Redis ZSet 实现
- 热门池：热点微博缓存

**性能优化**

- 异步分发：MQ 解耦
- 热门缓存：热点预计算
- 分页策略：使用 Cursor 分页

:::
