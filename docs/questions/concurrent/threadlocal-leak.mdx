# ThreadLocal 内存泄漏

> **目标级别**：P5/P6
> **面试频率**：🔴 高频

面试官问：「ThreadLocal 会导致内存泄漏吗？」你说「会」——然后面试官紧接着追问「那为什么还用 ThreadLocal？如何避免泄漏？」你沉默了。

ThreadLocal 的内存泄漏是面试中的高频陷阱，必须深入理解。

### 为什么使用弱引用？

| 引用类型 | 场景 | 结果 |
|---------|------|------|
| **强引用** | ThreadLocal 外部无引用 | ThreadLocal 永远不回收 |
| **弱引用** | ThreadLocal 外部无引用 | GC 时回收 ThreadLocal |

## 内存泄漏的条件

ThreadLocal 内存泄漏需要同时满足以下条件：

| 条件 | 说明 |
|------|------|
| **1. 弱引用被回收** | ThreadLocal 外部没有强引用，GC 回收了 ThreadLocal |
| **2. Entry.value 仍存在** | value 仍被 Entry 持有 |
| **3. 线程长期存活** | 线程池中的线程被复用 |

### 不会泄漏的场景

```java
// 场景 1：线程结束
Thread thread = new Thread(() -> {
    ThreadLocal<String> tl = new ThreadLocal<>();
    tl.set("value");
});
thread.start();
thread.join();
// 线程结束，ThreadLocalMap 随之销毁，无泄漏
```

```java
// 场景 2：主动 remove
try {
    ThreadLocal<String> tl = new ThreadLocal<>();
    tl.set("value");
    // 业务逻辑
} finally {
    tl.remove(); // ✅ 主动清理
}
```

### 会泄漏的场景

```java
// 线程池场景
ExecutorService executor = Executors.newFixedThreadPool(1);

executor.submit(() -> {
    ThreadLocal<String> tl = new ThreadLocal<>();
    tl.set("large object");

    // 忘记 remove
    // 线程复用，对象无法回收
});
```

## 如何避免内存泄漏

### 1. 每次使用后 remove

```java
// ✅ 推荐：使用后主动 remove
ThreadLocal<String> tl = new ThreadLocal<>();
try {
    tl.set("value");
    // 业务逻辑
} finally {
    tl.remove();
}
```

### 2. 使用 try-with-resources

```java
// 封装 ThreadLocal 的清理
public class ThreadLocalHolder<T> implements AutoCloseable {
    private final ThreadLocal<T> tl = new ThreadLocal<>();

    public T get() {
        return tl.get();
    }

    public void set(T value) {
        tl.set(value);
    }

    @Override
    public void close() {
        tl.remove();
    }
}

// 使用
try (ThreadLocalHolder<String> holder = new ThreadLocalHolder<>()) {
    holder.set("value");
    // 业务逻辑
} // 自动清理
```

### 3. 使用 InheritableThreadLocal 后清理

```java
// InheritableThreadLocal 更容易泄漏
InheritableThreadLocal<String> itl = new InheritableThreadLocal<>();
try {
    itl.set("value");
    // 业务逻辑
} finally {
    itl.remove();
}
```

### 4. 定期清理

```java
// 使用静态 ThreadLocalMap，定期清理过期 Entry
public class ThreadLocalCleanup {
    public static void clean(ThreadLocal<?>... locals) {
        for (ThreadLocal<?> tl : locals) {
            tl.remove();
        }
    }
}
```

## 高频面试题

### 🔴 题目 1：ThreadLocal 内存泄漏的原因？

**参考回答**：

内存泄漏的原因：

1. **ThreadLocalMap 的 Entry 使用弱引用**：key 是弱引用，value 是强引用
2. **GC 回收 ThreadLocal 对象**：外部没有强引用时，GC 回收 ThreadLocal
3. **value 无法回收**：Entry.value 仍是强引用，被 Entry 持有
4. **线程复用**：线程池中线程不销毁，Entry 无法清理

### 🔴 题目 2：为什么使用弱引用？

**参考回答**：

使用弱引用的目的：

1. **防止 ThreadLocal 无法回收**：如果没有弱引用，ThreadLocal 永远是可达的，永远不会被回收
2. **延迟回收**：GC 时至少可以回收 ThreadLocal 对象
3. **但不能完全避免泄漏**：value 仍然是强引用，如果没有 remove，value 无法回收

### 🔴 题目 3：如何避免泄漏？

**参考回答**：

| 方法 | 说明 |
|------|------|
| **remove()** | 使用后主动调用 remove |
| **try-finally** | 在 finally 中 remove |
| **封装工具类** | 实现 AutoCloseable |
| **定期清理** | 定期调用 cleanUp |

## 常见错误与陷阱

### ⚠️ 陷阱 1：只在 finally 中 set null

```java
// ❌ 错误：只是删除了 value
try {
    threadLocal.set("value");
} finally {
    threadLocal.set(null); // ❌ 只是删除值，不删除 Entry
}
```

### ⚠️ 陷阱 2：忘记在线程池场景使用

```java
// ❌ 线程池 + ThreadLocal = 泄漏
executor.submit(() -> {
    ThreadLocal<byte[]> buffer = new ThreadLocal<>();
    buffer.set(new byte[1024 * 1024]); // 1MB
    // 忘记 remove，每次执行都分配 1MB
});
```

### ⚠️ 陷阱 3：静态 ThreadLocal

```java
// ❌ 静态 ThreadLocal + 线程池 = 泄漏
private static final ThreadLocal<Object> cache = new ThreadLocal<>();

// ✅ 静态 ThreadLocal 本身没问题
// ✅ 关键是使用后 remove
```

## 加分回答

### 💡 JDK 的优化：expungeStaleEntries

ThreadLocalMap 在以下时机清理过期 Entry：

1. `getEntry()` 时
2. `set()` 时
3. `remove()` 时

```java
private void expungeStaleEntry(int staleSlot) {
    Entry[] tab = table;
    int len = tab.length;

    // 删除过期 Entry
    tab[staleSlot].value = null;
    tab[staleSlot] = null;
    size--;

    // 重新散列其他 Entry
    Entry e;
    int i;
    for (i = nextIndex(staleSlot, len);
         (e = tab[i]) != null;
         i = nextIndex(i, len)) {
        ThreadLocal<?> k = e.get();
        if (k == null) {
            e.value = null;
            tab[i] = null;
            size--;
        }
    }
}
```

## 总结对比表

| 场景 | 是否泄漏 | 原因 |
|------|---------|------|
| 线程结束 | ❌ 否 | ThreadLocalMap 随线程销毁 |
| 主动 remove | ❌ 否 | Entry 被删除 |
| 线程池 + remove | ❌ 否 | 正确清理 |
| 线程池 + 无 remove | ✅ 是 | 线程复用 + value 无法回收 |
| 弱引用 | 延迟 | GC 回收 key，但 value 可能泄漏 |
