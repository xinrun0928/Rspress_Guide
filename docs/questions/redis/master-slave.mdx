# Redis 主从复制

> **目标级别**：P5/P6
> **面试频率**：🔴 高频
> **面试官最关心的 3 个问题**：
> 1. Redis 主从复制是怎么工作的？
> 2. 全量复制和增量复制有什么区别？
> 3. 主从复制有哪些常见问题？

面试官问：「Redis 主从复制过程中，如果网络断了，从库是怎么恢复的？」你说「重新同步」——然后面试官追问「是全部重新同步吗？会不会太慢了？」你沉默了。

这就是主从复制的核心：如何高效地保持主从数据一致。

## 一、主从复制概述

### 1.1 为什么需要主从复制

| 需求 | 说明 |
|------|------|
| **数据备份** | 从库作为数据副本，防止数据丢失 |
| **读写分离** | 主库写，从库读，分担压力 |
| **故障恢复** | 主库挂了可以切换到从库 |
| **水平扩展** | 可以增加从库数量提高读性能 |

```mermaid

    subgraph "主从架构"
        M["Master<br/>写"] --> R1["Replica 1<br/>读"]
        M --> R2["Replica 2<br/>读"]
        M --> R3["Replica 3<br/>读"]
    end

    subgraph "客户端"
        W["写请求"] --> M
        R["读请求"] --> R1
        R["读请求"] --> R2
    end
```

### 1.2 配置方式

#### 1.2.1 命令配置（临时生效）

```bash
# 在从库上执行
redis-cli
127.0.0.1:6379> SLAVEOF 127.0.0.1 6379
OK

# 取消复制
127.0.0.1:6379> SLAVEOF NO ONE
```

#### 1.2.2 配置文件配置（永久生效）

```bash
# redis.conf（从库配置）
replicaof 127.0.0.1 6379
replica-serve-stale-data yes  # 主库不可用时是否提供旧数据
replica-read-only yes         # 从库只读
repl-diskless-sync no         # 是否使用无盘复制
```

## 二、复制原理

### 2.1 复制流程

```mermaid
sequenceDiagram
    participant S as 从库
    participant M as 主库

    S->>M: 1. 发送 PING（检查连接）
    M-->>S: PONG

    S->>M: 2. 发送 AUTH（认证）
    M-->>S: OK

    S->>M: 3. 发送 REPLCONF listening-port
    M-->>S: OK

    S->>M: 4. 发送 PSYNC ? -1（请求全量复制）
    M-->>S: FULLRESYNC {runid} {offset}

    Note over M,S: 全量复制阶段
    M->>S: 5. 发送 RDB 文件
    S->>S: 6. 加载 RDB

    Note over M,S: 增量复制阶段
    M->>S: 7. 发送缓冲区命令
    S-->>M: 8. 发送 ACK 确认
```

### 2.2 数据同步类型

| 类型 | 触发条件 | 说明 |
|------|----------|------|
| **全量复制** | 从库第一次连接 / 主从断开太久 | 发送完整的 RDB 文件 |
| **增量复制** | 主从断开后重新连接 | 发送断开期间的写命令 |

```mermaid
flowchart TD
    A["从库连接主库"] --> B{"是否首次?"}
    B -->|"是"| C["全量复制"]
    B -->|"否"| D{"断开时间?"}
    D -->|"短"| E["增量复制"]
    D -->|"长"| F["全量复制"]
```

## 三、全量复制

### 3.1 流程详解

```mermaid
sequenceDiagram
    participant S as 从库
    participant M as 主库
    participant D as 磁盘
    participant C as 客户端

    Note over S,M: 阶段1：连接建立
    S->>M: PSYNC ? -1
    M-->>S: FULLRESYNC runid offset

    Note over S,M: 阶段2：生成 RDB
    M->>M: BGSAVE 生成 RDB
    M->>M: 记录写命令到缓冲区
    M->>D: 写入 RDB 文件

    Note over S,M: 阶段3：传输 RDB
    M->>S: 发送 RDB 文件
    S->>S: 清空内存
    S->>S: 加载 RDB

    Note over S,M: 阶段4：同步缓冲区
    M->>S: 发送缓冲区命令
    S->>S: 执行命令

    Note over S,M: 阶段5：持续同步
    C->>M: SET key value
    M->>S: 发送命令
    S->>S: 执行命令
```

### 3.2 存在的问题

| 问题 | 说明 | 影响 |
|------|------|------|
| **RDB 生成耗时** | 主库 BGSAVE 需要 fork，可能阻塞 | 响应延迟增加 |
| **RDB 传输耗时** | 大 RDB 文件传输慢 | 同步时间长 |
| **从库加载慢** | 从库清空数据、加载 RDB | 短暂不可用 |
| **主库压力** | 同时服务客户端和传输 RDB | 性能下降 |

### 3.3 无盘复制

Redis 2.8.18 支持无盘复制，主库不写磁盘直接传输：

```bash
# redis.conf
repl-diskless-sync yes
repl-diskless-sync-delay 5  # 等待更多从库连接
```

```mermaid
sequenceDiagram
    participant S as 从库
    participant M as 主库

    M->>M: 生成 RDB 到内存
    M->>S: 直接发送
    S->>S: 接收并加载
```

## 四、增量复制

### 4.1 原理：复制偏移量

```mermaid
graph LR
    M["主库<br/>offset: 1000"] -->|"写命令"| B["缓冲区"]
    B --> R1["从库1<br/>offset: 1000"]
    B --> R2["从库2<br/>offset: 950"]
    B --> R3["从库3<br/>offset: 800"]
```

| 概念 | 说明 |
|------|------|
| **主库 offset** | 主库已发送的字节数 |
| **从库 offset** | 从库已接收的字节数 |
| **repl_backlog** | 环形缓冲区，保存最近的命令 |

### 4.2 增量复制流程

```mermaid
sequenceDiagram
    participant S as 从库
    participant M as 主库

    Note over S,M: 断开连接

    S->>M: PSYNC {runid} {offset}
    M->>M: 检查 offset 是否在缓冲区范围

    alt offset 在范围内
        M->>S: CONTINUE + 命令
        S->>S: 执行命令
    else offset 不在范围内
        M-->>S: FULLRESYNC
        S->>S: 全量复制
    end
```

### 4.3 repl_backlog 配置

```bash
# redis.conf
repl-backlog-size 1mb     # 缓冲区大小，默认 1MB
repl-backlog-ttl 3600     # 从库断开后缓冲区保留时间
```

**缓冲区大小计算**：
```
缓冲区大小 = 主库每秒写入量 × 最大断开时间
```

例如：主库每秒写 10MB，最大断开 1 分钟，则需要 `10MB × 60 = 600MB`

## 五、主从复制优化

### 5.1 配置优化

```bash
# redis.conf（主库配置）

# 无盘复制
repl-diskless-sync yes
repl-diskless-sync-delay 5

# 复制缓冲区大小
repl-backlog-size 16mb

# 从库数量（用于估算缓冲区大小）
listens-replica-announce-ip 10.0.0.1
```

```bash
# redis.conf（从库配置）

# 只读模式（从 Redis 6.0 支持）
replica-read-only yes

# 允许旧数据（主库不可用时）
replica-serve-stale-data yes

# 并行同步
repl-parallel-syncs 5
```

### 5.2 架构优化

```mermaid
graph TB
    subgraph "链式复制（不推荐）"
        M1["Master"] --> R1["Replica 1"]
        R1 --> R2["Replica 2"]
        R2 --> R3["Replica 3"]
    end

    subgraph "星形复制（推荐）"
        M2["Master"] --> R4["Replica 1"]
        M2 --> R5["Replica 2"]
        M2 --> R6["Replica 3"]
    end
```

| 复制方式 | 优点 | 缺点 |
|----------|------|------|
| **星形复制** | 延迟低，主从独立 | 主库压力大 |
| **链式复制** | 主库压力小 | 延迟逐级增加 |

## 六、常见问题

### 6.1 脑裂问题

```mermaid
sequenceDiagram
    participant M as 主库
    participant S as 从库
    participant N as 网络分区

    M->>N: 正常同步
    S->>N: 正常同步

    N->>N: 网络分区

    C1->>M: 写入数据A
    M->>S: 数据A

    Note over M,S: 主从断开

    C2->>S: 哨兵切换
    S->>S: 成为新主
    C2->>S: 写入数据B

    N->>M: 恢复
    M->>S: 数据A同步

    Note over M,S: 数据A丢失！
```

**解决方案**：
1. 配置 `min-replicas-to-write` 和 `min-replicas-max-lag`
2. 使用哨兵或集群监控

### 6.2 复制风暴

大量从库同时重启，导致主库压力过大：

```mermaid
flowchart TD
    M["Master"] -.-> R1["Replica 1"]
    M -.-> R2["Replica 2"]
    M -.-> R3["Replica 3"]
    M -.-> R4["Replica N..."]

    R1 -->|"同时"| BG["BGSAVE"]
    R2 -->|"同时"| BG
    R3 -->|"同时"| BG
    R4 -->|"同时"| BG

    BG -->|"多次"| RDB["多次 RDB 生成"]
    RDB --> M["主库压力剧增 ⚠️"]
```

**解决方案**：从库逐个重启，或使用无盘复制。

## 八、常见面试陷阱

**⚠️ 陷阱 1**：认为主从复制是同步的

Redis 主从复制是**异步**的，主库执行完命令后立即返回，不等待从库确认。

**⚠️ 陷阱 2**：混淆全量复制和增量复制的触发条件

从库断开连接后，是否全量复制取决于 `repl_backlog` 缓冲区是否能容纳断开的命令。

**⚠️ 陷阱 3**：忽视复制延迟

主从复制有延迟，如果业务对一致性要求高，不能简单做读写分离。

## 九、对比总结表

| 维度 | 全量复制 | 增量复制 |
|------|----------|----------|
| **触发条件** | 首次连接 / 断开太久 | 断开时间短 |
| **传输内容** | 完整 RDB + 缓冲区命令 | 缓冲区命令 |
| **数据量** | 全量数据 | 增量数据 |
| **耗时** | 长 | 短 |
| **主库压力** | 大（BGSAVE + 传输） | 小 |
| **网络开销** | 大 | 小 |
