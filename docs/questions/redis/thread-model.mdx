> 1. Redis 是单线程还是多线程？

### 1.1 Redis 的线程分工

Redis 的「单线程」指的是**命令处理线程**是单线程：

| 线程类型 | 说明 |
|----------|------|
| **主线程** | 处理客户端命令请求（`O(n)` 操作） |
| **后台线程** | 关闭文件、AOF 刷盘、异步删除 |
| **子进程** | RDB 持久化（fork 进程） |
| **多线程** | Redis 6.0+ 的 IO 线程（仅处理网络 IO） |

## 二、IO 多路复用

### 2.1 为什么需要 IO 多路复用？

传统 IO 模型中，每个连接都需要一个线程处理：

```mermaid
graph TB
    subgraph "传统 IO 模型"
        T1["Thread 1"] --> S1["Socket 1"]
        T2["Thread 2"] --> S2["Socket 2"]
        T3["Thread 3"] --> S3["Socket 3"]
        T4["Thread N"] --> S4["Socket N"]
    end
```

**问题**：
- 每个连接一个线程，上下文切换开销大
- 线程数量有限，无法支撑海量连接
- CPU 大部分时间在等待 IO

### 2.2 IO 多路复用原理

IO 多路复用允许一个线程同时监控多个文件描述符：

```mermaid
graph TB
    subgraph "IO 多路复用模型"
        subgraph "客户端连接"
            C1["Socket 1"]
            C2["Socket 2"]
            C3["Socket 3"]
            C4["..."]
        end

        subgraph "内核层"
            K1["select/epoll/kqueue"]
        end

        subgraph "应用层"
            M1["Redis 单线程"]
        end
    end

    C1 --> K1
    C2 --> K1
    C3 --> K1
    C4 --> K1
    K1 -->|"有事件就绪"| M1
```

**核心思想**：用一个线程监听多个 socket，当某个 socket 可读/可写时通知应用程序。

### 2.3 epoll 工作流程

```mermaid
flowchart TD
    A["创建 epoll 实例"] --> B["注册 socket 到 epoll"]
    B --> C["调用 epoll_wait"]
    C --> D{"有事件就绪?"}
    D -->|是| E["处理就绪事件"]
    E --> F["返回继续监听"]
    D -->|否| C
    F --> C
```

### 2.4 epoll vs select vs poll

| 特性 | select | poll | epoll |
|------|--------|------|-------|
| **最大fd限制** | 1024 | 无限制 | 无限制 |
| **时间复杂度** | `O(n)` | `O(n)` | `O(1)` |
| **内核拷贝** | fdset | 数组 | 红黑树 |
| **工作模式** | 遍历 | 遍历 | 回调 |
| **跨平台** | ✅ | ✅ | ❌（Linux） |
| **Redis 使用** | ❌ | ❌ | ✅ |

## 三、Redis 事件处理机制

### 3.1 Redis 事件循环

```mermaid
flowchart TD
    A["初始化"] --> B["创建 socket 监听"]
    B --> C["创建事件循环"]
    C --> D["注册事件处理器"]
    D --> E["进入事件循环"]

    E --> F["epoll_wait 等待事件"]
    F --> G{"收到事件?"}
    G -->|是| H["处理事件"]
    G -->|否| F
    H --> I["执行命令"]
    I --> J["写响应"]
    J --> K{"还有更多事件?"}
    K -->|是| F
    K -->|否| E
```

### 3.2 事件类型

| 事件类型 | 说明 | 示例 |
|----------|------|------|
| **File Event** | 文件描述符事件 | 客户端请求、服务端响应 |
| **Time Event** | 定时事件 | 过期 key 清理、RDB 触发 |

### 3.3 Redis 处理流程

```mermaid
sequenceDiagram
    participant Client as 客户端
    participant Redis as Redis Server
    participant Loop as 事件循环
    participant DB as 数据库

    Client->>Redis: 发送命令 SET key value
    Redis->>Loop: socket 可读
    Loop->>Redis: 读取命令
    Redis->>DB: 执行命令
    DB->>Redis: 返回结果
    Redis->>Client: 写入响应
```

## 四、Redis 6.0 多线程设计

### 4.1 为什么引入多线程？

Redis 6.0 之前，瓶颈在于**网络 IO** 而非 CPU：

| 瓶颈 | 说明 |
|------|------|
| **CPU** | 大部分命令是 `O(1)` 或 `O(n)`，不占 CPU |
| **网络 IO** | 从 socket 读取命令、写入响应，占大量时间 |

### 4.2 多线程架构

```mermaid
graph TB
    subgraph "Redis 6.0+ 多线程模型"
        subgraph "主线程"
            M1["命令解析"]
            M2["命令执行"]
            M3["响应写入"]
        end

        subgraph "IO 线程池"
            I1["IO Thread 1"]
            I2["IO Thread 2"]
            I3["IO Thread N"]
        end
    end

    subgraph "处理流程"
        C1["客户端请求"]
        C1 --> M1
        M1 --> I1
        I1 --> I2
        I2 --> I3
        I3 --> M2
        M2 --> M3
        M3 --> C1
    end
```

**多线程只用于网络 IO**，命令执行仍然是单线程。

### 4.3 线程分工

| 阶段 | 线程模式 | 说明 |
|------|----------|------|
| 读取命令 | 多线程（IO线程池） | 从 socket 读取数据到缓冲区 |
| 解析命令 | 主线程 | 解析命令参数 |
| 执行命令 | 主线程 | 调用命令处理函数 |
| 写响应 | 多线程（IO线程池） | 将响应写回 socket |

### 4.4 配置项

```bash
# 启用多线程 IO
io-threads 4

# 多线程是否用于写操作
io-threads-do-reads yes
```

## 七、常见面试陷阱

**⚠️ 陷阱 1**：认为 Redis 完全单线程
- 后台线程、RDB fork 进程、Redis 6.0+ IO 线程
- 要能区分哪些操作是单线程，哪些不是

**⚠️ 陷阱 2**：混淆 IO 多路复用和线程池
- IO 多路复用是一个线程监听多个 socket
- 线程池是多个线程处理不同任务

**⚠️ 陷阱 3**：不知道 epoll 的边缘触发
- Redis 使用水平触发（LT）模式
- 边缘触发（ET）需要非阻塞配合

## 八、对比总结表

| 特性 | 传统阻塞 IO | IO 多路复用 | 多线程+阻塞 IO |
|------|-------------|-------------|----------------|
| **线程数** | N（每个连接一个） | 1 | N |
| **资源占用** | 高 | 低 | 中 |
| **并发能力** | 低 | 高 | 高 |
| **编程复杂度** | 低 | 中 | 高 |
| **适用场景** | 连接少 | 高并发 | 计算密集 |
