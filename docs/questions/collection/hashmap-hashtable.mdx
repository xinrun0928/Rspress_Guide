# HashMap 与 Hashtable 区别

**目标级别**：P5 / P6

---

## 快速自测

面试官问：「HashMap 和 Hashtable 有什么区别？为什么现在很少用 Hashtable？」

---

## 一、核心问题

### 🔴 HashMap 和 Hashtable 的区别是什么？

**一张表总结**：

| 对比维度 | HashMap | Hashtable |
|---------|---------|-----------|
| 线程安全 | ❌ 不安全 | ✅ 安全（全方法 synchronized） |
| 性能 | 高 | 低 |
| 能否存 null key | ✅ 1 个 | ❌ 0 个 |
| 能否存 null value | ✅ 多个 | ❌ 0 个 |
| 迭代器 | fail-fast（快速失败） | fail-fast |
| 初始容量 | 16（懒加载） | 11（立即分配） |
| 扩容方式 | capacity × 2 | capacity × 2 + 1 |
| 负载因子 | 0.75（可调） | 0.75（固定） |
| 继承体系 | AbstractMap | Dictionary |
| 是否推荐 | ✅ 推荐 | ❌ 不推荐 |

---

## 二、历史背景

### 💡 为什么要了解 Hashtable？

Hashtable 是 JDK 1.0 就有的古老类，比 HashMap 早了 3 年（JDK 1.2 才引入 HashMap）。

```mermaid
flowchart LR
    A[JDK 1.0<br/>Hashtable 诞生] --> B[JDK 1.2<br/>HashMap 诞生]
    B --> C[JDK 5.0<br/>ConcurrentHashMap 诞生]
    C --> D[Hashtable 逐渐被取代]
    
    style A fill:#FFA07A
    style B fill:#90EE90
    style C fill:#90EE90
```

**历史演进**：
1. **JDK 1.0**：Hashtable 是唯一的哈希表实现
2. **JDK 1.2**：引入 HashMap，同时给 Hashtable 加入同步方法
3. **JDK 5.0**：引入 ConcurrentHashMap，高并发场景的首选
4. **现在**：Hashtable 只存在于兼容旧代码，基本不再使用

## 三、源码对比

### HashMap 构造方法

```java
// HashMap 无参构造（懒加载）
public HashMap() {
    this.loadFactor = DEFAULT_LOAD_FACTOR; // 0.75
    // 注意：table 在第一次 put 时才创建
}

// HashMap 指定容量
public HashMap(int initialCapacity) {
    this(initialCapacity, DEFAULT_LOAD_FACTOR);
}
```

### Hashtable 构造方法

```java
// Hashtable 无参构造（立即分配）
public Hashtable() {
    this(11, 0.75f);  // 直接创建 table
}

// Hashtable 初始化
public Hashtable(int initialCapacity, float loadFactor) {
    if (initialCapacity < 0)
        throw new IllegalArgumentException(...);
    if (loadFactor <= 0 || Float.isNaN(loadFactor))
        throw new IllegalArgumentException(...);
    this.table = new Entry<?,?>[initialCapacity];
    this.loadFactor = loadFactor;
    this.threshold = (int)Math.min(initialCapacity * loadFactor, MAX_ARRAY_SIZE + 1);
}
```

**关键区别**：Hashtable 的 table 在构造时就创建，HashMap 是懒加载。

## 四、put 方法对比

### HashMap put

```java
// HashMap.putVal - 关键部分
public V put(K key, V value) {
    return putVal(hash(key), key, value, false, true);
}

final V putVal(int hash, K key, V value, boolean onlyIfAbsent, boolean evict) {
    // ...
    if ((p = tab[i = (n - 1) & hash]) == null)
        tab[i] = newNode(hash, key, value, null);
    // ...
}
```

### Hashtable put

```java
// Hashtable.put - 关键部分
public synchronized V put(K key, V value) {
    // Hashtable 不允许 null key 和 null value
    if (value == null) {
        throw new NullPointerException();
    }
    // ...
    int hash = key.hashCode();
    // ...
    for (int index = (hash & 0x7FFFFFFF) % tab.length; ...)
    // ...
    tab[index] = new Entry<>(hash, key, value, old);
    // ...
}
```

**关键区别**：

| 区别 | HashMap | Hashtable |
|------|---------|-----------|
| null 检查 | 允许 null key/value | 不允许 |
| 同步方式 | 无 | synchronized 方法 |
| hash 计算 | `(h = key.hashCode()) ^ (h >>> 16)` | `key.hashCode()` 直接用 |
| 下标计算 | `(n - 1) & hash` | `hash & 0x7FFFFFFF % tab.length` |

### 💡 为什么 Hashtable 不允许 null？

**历史原因**：Hashtable 的方法都是 synchronized，在并发环境下，如果允许 null，需要处理大量边界情况。

**技术原因**：`null` 无法通过 hash 计算，如果 put null key，后续 get(null) 会导致歧义。

## 五、hash 计算对比

### HashMap 扰动函数

```java
// HashMap 的 hash（扰动函数）
static final int hash(Object key) {
    int h;
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}
```

### Hashtable 直接使用 hashCode

```java
// Hashtable 的 hash（没有扰动）
int hash = key.hashCode();
int index = (hash & 0x7FFFFFFF) % tab.length;
```

**Hashtable 的 `& 0x7FFFFFFF`**：将 hash 转为正整数，避免负数导致的数组越界。

## 六、扩容方式对比

### HashMap 扩容

```java
// HashMap: capacity × 2
if ((newCap = oldCap << 1) < MAXIMUM_CAPACITY)
    newThr = oldThr << 1;
```

### Hashtable 扩容

```java
// Hashtable: capacity × 2 + 1（奇数）
int newCapacity = (oldCapacity << 1) + 1;
if (newCapacity > MAX_ARRAY_SIZE) {
    newCapacity = MAX_ARRAY_SIZE;
}
```

### ⚠️ 为什么 Hashtable 用奇数？

这是 Hashtable 设计时的考虑：**用 `%` 取模计算下标**，`capacity × 2 + 1` 可以减少哈希冲突。

但 HashMap 使用 `&` 位运算，要求容量是 2 的幂次，用偶数（2^k）更合适。

## 七、性能对比

### 理论分析

| 操作 | HashMap | Hashtable |
|------|---------|-----------|
| put | O(1) ~ O(n) | O(1) ~ O(n) + synchronized |
| get | O(1) ~ O(n) | O(1) ~ O(n) + synchronized |
| 单线程 | 性能高 | 性能低（无谓的同步开销） |
| 多线程 | 不安全 | 安全但低效 |

### 实际测试

```java
// 测试代码（伪代码）
Map<Integer, Integer> map;

// 单线程测试
long start = System.nanoTime();
for (int i = 0; i < 1000000; i++) {
    map.put(i, i);
}
long end = System.nanoTime();
System.out.println("Time: " + (end - start) / 1_000_000 + "ms");

/*
 * 结果（大致）：
 * HashMap: ~50ms
 * Hashtable: ~150ms
 * Collections.synchronizedMap: ~140ms
 * ConcurrentHashMap: ~80ms
 */
```

## 八、选型建议

### 什么时候用 HashMap？

- ✅ 单线程环境
- ✅ 可以接受 Collections.synchronizedMap 包装
- ✅ 需要存储 null key 或 null value

### 什么时候用 Hashtable？

- ❌ 基本不推荐使用

### 什么时候用 ConcurrentHashMap？

- ✅ 高并发环境
- ✅ 需要线程安全
- ✅ 不需要存储 null key/null value

## 九、面试题精讲

### 🔴 第一层：HashMap 和 Hashtable 有什么区别？

> **参考答案**：
>
> 主要区别有：
> 1. **线程安全**：Hashtable 是线程安全的（synchronized 方法），HashMap 不是
> 2. **null 支持**：HashMap 允许 null key（1个）和 null value（多个），Hashtable 都不允许
> 3. **性能**：HashMap 性能更好，Hashtable 因为同步开销大
> 4. **迭代器**：都是 fail-fast
> 5. **容量**：HashMap 默认 16，扩容翻倍；Hashtable 默认 11，扩容 2n+1
> 6. **历史**：Hashtable 是 JDK 1.0 的遗留类，已被 ConcurrentHashMap 取代

### 🟡 第二层：为什么 Hashtable 不允许 null？

> **参考答案**：
>
> 主要原因是历史和设计权衡：
> 1. Hashtable 是同步的，如果允许 null，在多线程环境下需要额外处理 null 的边界情况
> 2. Hashtable 的方法是 synchronized，无法区分「key 不存在」和「value 为 null」
> 3. HashMap 通过额外的检查机制（如 containsKey）可以区分这些情况

### 💡 第三层：Hashtable 的 synchronized 和 Collections.synchronizedMap 哪个更好？

> **参考答案**：
>
> 两者都是线程安全的，但实现不同：
> 1. Hashtable：每个方法都 synchronized，锁住整个对象
> 2. Collections.synchronizedMap：底层也是 synchronized，但可以用装饰器模式包装任意 Map
>
> 实际使用中，**两者性能差不多**，都不推荐在真正的高并发场景使用。推荐使用 ConcurrentHashMap。

### ⚠️ 面试官挖坑点

| 陷阱 | 错误回答 | 正确回答 |
|------|---------|----------|
| 「Hashtable 是线程安全的所以更好」 | 忽略性能差异 | 线程安全但性能差，已被 ConcurrentHashMap 取代 |
| 「HashMap 可以完全替代 Hashtable」 | 忽略 null 场景 | 如果需要存 null，HashMap 更合适 |
| 「Hashtable 的迭代是线程安全的」 | 不了解 fail-fast | Hashtable 迭代器也是 fail-fast，并发修改会抛异常 |

## 十、总结

**HashMap vs Hashtable 对比表格**：

| 维度 | HashMap | Hashtable |
|------|---------|-----------|
| 线程安全 | ❌ | ✅ |
| 性能 | 高 | 低 |
| null key | ✅ 1个 | ❌ |
| null value | ✅ 多个 | ❌ |
| 初始容量 | 16 | 11 |
| 扩容方式 | × 2 | × 2 + 1 |
| 下标计算 | `&` 位运算 | `%` 取模 |
| 推荐程度 | ✅ | ❌ |

**选型原则**：
- 单线程 → HashMap
- 需要存 null → HashMap
- 高并发 → ConcurrentHashMap
- Hashtable → 不推荐使用

## 延伸思考

> **追问**：如果让你实现一个既高性能又线程安全的 HashMap，你会怎么做？

这是 ConcurrentHashMap 的设计目标：
1. **分段锁**：不同桶用不同的锁
2. **CAS**：对于简单操作，用 CAS 而非锁
3. ** volatile**：`table` 数组用 volatile 保证可见性
4. **分段扩容**：只锁需要扩容的桶
