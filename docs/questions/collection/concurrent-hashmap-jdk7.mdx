# ConcurrentHashMap JDK7 分段锁

**目标级别**：P6 / P7

---

## 快速自测

面试官问：「JDK7 的 ConcurrentHashMap 是怎么保证线程安全的？为什么不用全局锁？」

## 一、核心问题

### 🔴 JDK7 ConcurrentHashMap 的数据结构是什么？

**JDK7 答案**：Segment 数组 + HashEntry 数组 + 链表

```java
// JDK 7 ConcurrentHashMap
public class ConcurrentHashMap<K, V> extends AbstractMap<K, V>
        implements ConcurrentMap<K, V>, Serializable {

    // Segment 数组，分段锁
    final Segment<K, V>[] segments;

    // 每个 Segment 内部是 HashEntry 数组 + 链表
    static final class Segment<K, V> extends ReentrantLock {
        volatile HashEntry<K, V>[] table;
        int count;
        int modCount;
        int threshold;
        float loadFactor;
    }

    // HashEntry 类似 HashMap.Node
    static final class HashEntry<K, V> {
        final int hash;
        final K key;
        volatile V value;
        volatile HashEntry<K, V> next;
    }
}
```

## 二、分段锁原理

### 🔴 为什么叫分段锁？

**核心思想**：把数据分成多个段（Segment），每个段独立加锁。不同线程访问不同段时，互不影响。

```mermaid
flowchart LR
    subgraph HashMap（全表锁）
        A[所有线程] --> B[同一个锁]
        B --> C[串行执行]
    end
    
    subgraph ConcurrentHashMap（分段锁）
        D[线程1] --> E[Segment[0] 锁]
        F[线程2] --> G[Segment[1] 锁]
        H[线程3] --> I[Segment[2] 锁]
        E --> J[并行执行]
        G --> J
        I --> J
    end
    
    style B fill:#FF6B6B
    style E fill:#90EE90
    style G fill:#90EE90
    style I fill:#90EE90
```

### 并发度计算

```java
// ConcurrentHashMap 构造方法
public ConcurrentHashMap(int initialCapacity, float loadFactor, int concurrencyLevel) {
    // 段数（锁的个数）= concurrencyLevel
    int sshift = 0;
    int ssize = 1;
    while (ssize < concurrencyLevel) {
        ++sshift;
        ssize <<= 1;
    }
    // segmentShift 用于计算 key 落在哪个段
    this.segmentShift = 32 - sshift;
    // segmentMask 用于按位与计算段下标
    this.segmentMask = ssize - 1;
    
    // 初始化 segments 数组
    this.segments = Segment.newArray(ssize);
    // ...
}
```

**默认并发度**：16，即最多 16 个线程可以同时操作。

## 三、key 定位过程

### 🔴 怎么确定 key 落在哪个 Segment？

**两步定位**：

```java
// JDK 7 源码
private int hash(Object key) {
    int h = hashSeed;
    if ((h != 0) || (key != null)) {
        h ^= h >>> 16;
        h ^= (h >>> 4) + (h >>> 8) + h >>> 2 + (h ^ (h >>> 7) >>> 12);
    }
    return h;
}

// 定位 Segment
int j = (hash >>> segmentShift) & segmentMask;
```

## 四、put 操作流程

### 🔴 put 是怎么加锁的？

```java
// Segment.put 方法
final V put(K key, int hash, V value, boolean onlyIfAbsent) {
    // 1. 加锁（可重入锁）
    HashEntry<K,V> node = tryLock() ? null :
        scanAndLockForPut(key, hash, value);
    
    try {
        int index = (tab.length - 1) & hash;
        HashEntry<K,V> first = entryAt(tab, index);
        HashEntry<K,V> e = first;
        
        // 2. 遍历链表查找或插入
        while (e != null) {
            if (e.hash == hash && key.equals(e.key)) {
                V oldValue = e.value;
                if (!onlyIfAbsent) {
                    e.value = value;
                }
                return oldValue;
            }
            e = e.next;
        }
        
        // 3. 头插法插入新节点
        if (node != null) {
            node.setNext(first);
            setTabAt(tab, index, node);
        } else {
            // 创建新节点并插入
            tab[index] = new HashEntry<>(hash, key, value, first);
        }
        return null;
    } finally {
        // 4. 解锁
        unlock();
    }
}
```

### 💡 scanAndLockForPut 详解

```java
// 获取锁或自旋等待
private HashEntry<K,V> scanAndLockForPut(K key, int hash, V value) {
    HashEntry<K,V> e = null;
    int c = count;
    
    while (!tryLock()) {
        // 如果锁被占用，遍历链表找 key（可能已被其他线程插入）
        e = scanAndLock(key, hash);
    }
    return e;
}
```

## 五、get 操作

### 🔴 get 为什么不需要加锁？

```java
// get 方法不需要加锁
public V get(Object key) {
    Segment<K,V> s;
    HashEntry<K,V>[] tab;
    int h = hash(key);
    long u = (((h >>> segmentShift) & segmentMask) << SSHIFT) + SBASE;
    
    // 直接读取，volatile 保证可见性
    if ((s = (Segment<K,V>)UNSAFE.getObjectVolatile(segments, u)) != null &&
        (tab = s.table) != null) {
        for (HashEntry<K,V> e = (HashEntry<K,V>) UNSAFE.getObjectVolatile
                 (tab, ((long)(((tab.length - 1) & h)) << TSHIFT) + TBASE);
             e != null; e = e.next) {
            if (e.hash == h && key.equals(e.key)) {
                return e.value;
            }
        }
    }
    return null;
}
```

**关键点**：
1. `value` 是 `volatile` 的，保证读取到最新值
2. `next` 也是 `volatile` 的，保证遍历到最新链表
3. 使用 `UNSAFE.getObjectVolatile` 读取，保证可见性

## 六、size 和 contains 操作

### ⚠️ 为什么 size 不能简单求和？

```java
// naive 的实现（错误）
public int size() {
    long sum = 0;
    for (Segment<K,V> s : segments) {
        sum += s.count;  // 累加每个段的 count
    }
    return sum;
}
```

**问题**：累加过程中，可能有线程正在修改某个 Segment，导致 count 不准确。

### 💡 JDK7 的 size 实现

```java
public int size() {
    // 1. 先尝试累加 2 次
    int n = segs.length;
    long sum = 0L;
    int mc = 0;
    
    for (int i = 0; i < n; ++i) {
        Segment<K,V> s = segs[i];
        int c = s.count;
        int mc2 = s.modCount;
        if (mc2 != mc) {  // modCount 变了，说明有修改
            mc = mc2;
            sum = 0L;
            n = segs.length;
            i = -1;  // 重新开始
            continue;
        }
        sum += c;
    }
    
    // 2. 如果多次修改太频繁，加锁重试
    if (sum > Long.MAX_VALUE) {
        return Integer.MAX_VALUE;
    }
    return (int) sum;
}
```

**核心思想**：
1. 先遍历累加，检查 modCount 是否变化
2. 如果 modCount 变了，说明有并发修改，重试
3. 重试超过一定次数，加锁再重试

## 七、并发度问题

### ⚠️ JDK7 分段锁的局限

| 问题 | 说明 | 影响 |
|------|------|------|
| 并发度固定 | Segment 数量在构造时确定，后续不能改变 | 如果线程数超过 16，效果受限 |
| 热点问题 | 如果大量 key 落在同一个 Segment | 退化成单锁 |
| 扩容只针对单个 Segment | 每个 Segment 独立扩容 | 扩容影响范围小，但实现复杂 |

## 八、面试题精讲

### 🔴 第一层：JDK7 ConcurrentHashMap 怎么保证线程安全？

> **参考答案**：
>
> JDK7 通过分段锁保证线程安全：
> 1. 内部有一个 Segment 数组，默认 16 个元素
> 2. 每个 Segment 继承 ReentrantLock，是一把可重入锁
> 3. key 通过 hash 定位到某个 Segment，然后只锁这个 Segment
> 4. 不同 Segment 之间可以并行操作，互不干扰

### 🟡 第二层：为什么不用全局锁？

> **参考答案**：
>
> 因为分段锁允许不同线程同时操作不同 Segment：
> 1. 如果用全局锁，所有操作都串行，性能很差
> 2. 分段锁将锁粒度细化，最多支持 16 个线程并行（默认并发度）
> 3. get 操作甚至不需要加锁，利用 volatile 保证可见性

### 💡 第三层：JDK7 有什么局限性？

> **参考答案**：
>
> JDK7 的分段锁有几个局限：
> 1. **并发度固定**：Segment 数量在构造时确定，不能动态调整
> 2. **热点 Segment**：如果 hash 不均匀，大量 key 落在同一个 Segment，性能会退化
> 3. **实现复杂**：Segment + HashEntry 两层结构，代码复杂度高

### ⚠️ 面试官挖坑点

| 陷阱 | 错误回答 | 正确回答 |
|------|---------|----------|
| 「Segment 数量可以动态增加」 | 忽略固定大小 | Segment 数量在构造时确定，后续不可变 |
| 「get 操作也需要加锁」 | 不了解 volatile 保证 | get 通过 volatile 保证可见性，无需加锁 |
| 「ConcurrentHashMap 完全线程安全」 | 忽略复合操作 | `putIfAbsent` + `get` 这样的复合操作仍需额外同步 |

## 十、总结

**JDK7 ConcurrentHashMap 分段锁核心要点**：

```mermaid
flowchart TB
    A[ConcurrentHashMap] --> B[Segment[] 数组]
    B --> C[Segment[0] 锁]
    B --> D[Segment[1] 锁]
    B --> E[...]
    B --> F[Segment[15] 锁]
    
    C --> G[HashEntry[] 表]

    G --> I[链表]
    H --> J[链表]
```

1. **数据结构**：Segment 数组 + HashEntry 数组 + 链表
2. **分段锁**：每个 Segment 一把锁，默认 16 把锁
3. **并发度**：最多 16 个线程并行
4. **get 无锁**：通过 volatile 保证可见性
5. **size 复杂度**：需要处理并发修改，可能重试

## 延伸思考

> **追问**：JDK8 对 ConcurrentHashMap 做了哪些改进？

JDK8 放弃了分段锁，改用 **CAS + synchronized**：
1. 放弃 Segment，直接用 Node 数组
2. 用 CAS 处理简单的插入操作
3. 用 synchronized 锁住链表头或红黑树根
4. 支持扩容时的并发迁移

详细内容见 [ConcurrentHashMap JDK8 CAS+synchronized](/questions/collection/concurrent-hashmap-jdk8)。
