# Kafka 副本机制：Leader Follower 与 ISR

## 场景切入：副本是 Kafka 高可用的基石

如果 Broker 宕机了，消息会丢失吗？

不会。因为 Kafka 的副本机制保证了数据的高可用。每个 Partition 都有多个副本，分布在不同的 Broker 上。当一个 Broker 宕机时，其他 Broker 上的副本会自动接管服务。

但副本机制远比「多拷贝几份」复杂。ISR、HW、LEO……这些概念你真的理解吗？

## 一、副本的基本概念

### 1.1 副本的角色

Kafka 的副本分为两种角色：

| 角色 | 说明 | 职责 |
|---|---|---|
| **Leader** | 主副本 | 对外提供读写服务 |
| **Follower** | 从副本 | 从 Leader 同步数据 |

```
Leader 与 Follower：

┌─────────────────────────────────────────────────────────┐
│                                                         │
│   Broker-1           Broker-2           Broker-3       │
│   ┌─────────┐        ┌─────────┐        ┌─────────┐    │
│   │Leader   │        │Follower │        │Follower │    │
│   │         │◄──────│         │◄──────│         │    │
│   │[数据]   │同步   │[副本]   │同步   │[副本]   │    │
│   └─────────┘       └─────────┘       └─────────┘    │
│                                                         │
│   生产者 ──写入──> Leader ──同步──> Follower 1 ──同步──> Follower 2   │
│                  │                                            │
│                  │                                            │
│            消费者 ◄──读取──                                  │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 1.2 副本分布

副本因子 `replication.factor` 决定了每个 Partition 的副本数量：

```properties
# Topic 创建时指定副本因子
kafka-topics.sh --create --topic order-topic \
  --partitions 3 \
  --replication-factor 3 \
  --bootstrap-server localhost:9092
```

```
副本分布（replication.factor=3）：

Topic: order-topic, Partition: 3

┌─────────────────────────────────────────────────────────┐
│  Partition 0: [Broker-1(Leader), Broker-2(Follower), Broker-3(Follower)]  │
│  Partition 1: [Broker-2(Leader), Broker-3(Follower), Broker-1(Follower)]  │
│  Partition 2: [Broker-3(Leader), Broker-1(Follower), Broker-2(Follower)]  │
│                                                         │
│  每个 Broker 都有机会成为 Leader，实现负载均衡              │
└─────────────────────────────────────────────────────────┘
```

## 二、ISR 机制

### 2.1 什么是 ISR

ISR（In-Sync Replicas）是与 Leader 保持同步的副本集合。只有 ISR 中的副本才有资格成为新的 Leader。

```
ISR 集合：

┌─────────────────────────────────────────────────────────┐
│                                                         │
│  ISR: [Broker-1(Leader), Broker-2(Follower)]           │
│  OSR: [Broker-3(Follower)]  ← 落后太多，被踢出 ISR    │
│                                                         │
│  Broker-3 同步追上，重新进入 ISR                        │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 2.2 ISR 的判定条件

一个副本被认定为「同步中」，需要满足：

| 条件 | 说明 | 配置参数 |
|---|---|---|
| **时间条件** | 副本与 Leader 的时间差 `< replica.lag.time.max.ms` | `replica.lag.time.max.ms=10000` |
| **消息条件** | 副本落后的消息数 `< replica.lag.max.messages` | `replica.lag.max.messages=10000` |

```properties
# Broker 配置
replica.lag.time.max.ms=10000        # 副本最大落后时间（默认 10 秒）
replica.lag.max.messages=40000       # 副本最大落后消息数（默认 40000）
```

### 2.3 ISR 的动态变化

```
ISR 变化流程：

时刻 T1:
  Leader: 写入消息 offset=1000
  ISR: [Broker-1, Broker-2, Broker-3]

时刻 T2:
  Broker-3 网络延迟，停止同步
  ISR: [Broker-1, Broker-2]  ← Broker-3 被踢出

时刻 T3:
  Broker-3 恢复同步，追上 Leader
  ISR: [Broker-1, Broker-2, Broker-3]  ← Broker-3 重新加入
```

### 2.4 min.insync.replicas

`min.insync.replicas` 决定了写入成功的最低保障：

```properties
# Broker 配置
min.insync.replicas=2
```

| 配置值 | 写入行为 |
|---|---|
| `1` | 只要 Leader 写入成功就返回 |
| `2` | ISR 中至少 2 个副本写入成功才返回 |
| `3` | ISR 中至少 3 个副本写入成功才返回（需要 replication.factor >= 3） |

## 三、HW 与 LEO

### 3.1 基本概念

| 概念 | 说明 |
|---|---|
| **LEO** | Log End Offset，日志文件结束偏移量，表示下一条消息写入的位置 |
| **HW** | High Watermark，高水位，消息可见性的分界线，HW 之前的消息对消费者可见 |

```
HW 与 LEO：

┌─────────────────────────────────────────────────────────┐
│                                                         │
│  Partition 日志：                                        │
│                                                         │
│  [msg 0] [msg 1] [msg 2] [msg 3] [msg 4] [msg 5]       │
│      ▲                                     ▲             │
│      │                                     │             │
│    LEO=6                              HW=4             │
│  (下一条写入位置)                   (可见性分界线)         │
│                                                         │
│  消费者可见范围：[msg 0, msg 1, msg 2, msg 3]           │
│  不可见范围：[msg 4, msg 5]                            │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 3.2 HW 更新机制

```
HW 更新流程：

1. Leader 写入新消息
   Leader LEO = 6

2. Follower 同步数据
   Follower LEO = 5

3. Follower 发送 fetch 请求
   Follower LEO 发送给 Leader

4. Leader 更新 HW
   HW = min(Leader LEO, Follower LEO) = 5

5. Follower 更新 HW
   Follower HW = 5
```

### 3.3 数据一致性保证

HW 机制保证了数据的一致性：

```
场景：Leader 宕机，Follower 成为新 Leader

┌─────────────────────────────────────────────────────────┐
│                                                         │
│  故障前：                                                │
│  Leader (msg 0-4) → Follower (msg 0-3)                 │
│  HW = 4                                                 │
│                                                         │
│  故障时：Leader 宕机，Follower 成为新 Leader             │
│  New Leader (msg 0-3)                                   │
│  HW = 4                                                 │
│                                                         │
│  结果：msg 4 被截断，避免数据不一致                      │
│       （即使 msg 4 在旧 Leader 写入，旧 Follower 也没有） │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

## 四、副本同步原理

### 4.1 同步流程

```
副本同步流程：

Producer ──写入──> Leader
                    │
                    ▼
            写入本地日志
            LEO = offset + 1
                    │
                    ▼
            通知 Follower 同步
                    │
         ┌──────────┼──────────┐
         │          │          │
         ▼          ▼          ▼
    Follower 1   Follower 2   Follower 3
         │          │          │
         ▼          ▼          ▼
    发送 FETCH  请求
         │          │          │
         │◄─────────┴──────────┘
         │
         ▼
    写入本地日志
    发送 ACK
```

### 4.2 同步参数调优

```properties
# 同步参数配置

# Follower 每次拉取的最小字节
replica.fetch.min.bytes=1

# Follower 每次拉取的最大字节
replica.fetch.max.bytes=10 * 1024 * 1024  # 10MB

# Follower 拉取最大等待时间
replica.fetch.wait.max.ms=500

# Leader 等待 Follower 同步的时间
replica.lag.time.max.ms=10000
```

## 五、副本选举

### 5.1 选举触发条件

| 触发条件 | 说明 |
|---|---|
| **Leader 宕机** | Controller 检测到 Leader 不可用 |
| **Leader 主动下线** | 管理员手动切换 |
| **分区副本重分配** | 副本重新分配 |

### 5.2 选举策略

```properties
# 选举策略配置
unclean.leader.election.enable=false  # 禁止从非 ISR 副本选举

# false：只能从 ISR 中选举（数据不丢失，但可能不可用）
# true：可以从任何存活的副本选举（可用性高，但可能丢失数据）
```

### 5.3 选举流程

```
选举流程：

1. Controller 检测到 Leader 宕机
   │
2. 从 ISR 中选择第一个存活的副本作为新 Leader
   │
3. 更新 ZK 中的 Leader 信息
   │
4. 通知所有 Broker 更新 Leader 信息
   │
5. 客户端自动重连到新 Leader
```

## 六、面试追问预判

### 追问一：ISR 是什么？

ISR（In-Sync Replicas）是与 Leader 保持同步的副本集合。副本在 ISR 中时，有资格成为 Leader；被踢出 ISR 后，只能等待追上 Leader 才能重新加入。

### 追问二：HW 和 LEO 是什么？

- **LEO**：Log End Offset，下一条消息写入的位置
- **HW**：High Watermark，消费者可见消息的分界线，HW 之前的消息对消费者可见

### 追问三：Follower 挂了怎么办？

1. Follower 被踢出 ISR
2. Follower 恢复后，从上次的 LEO 位置开始同步
3. 追上 Leader 后，重新加入 ISR

### 追问四：Leader 挂了怎么选新 Leader？

1. Controller 从 ISR 中选择第一个存活的副本作为新 Leader
2. 更新元数据
3. 其他 Follower 从新 Leader 同步数据
4. 如果 `unclean.leader.election.enable=true`，可以从非 ISR 副本选举
