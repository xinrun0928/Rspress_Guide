# Kafka 如何保证消息不重复消费（幂等性）

## 场景切入：重复不是 bug，是网络的必然

在分布式系统中，消息重复不是 bug，而是网络的不可靠性导致的必然结果。

网络会抖动，Broker 会重启，Consumer 会宕机。这些都会导致消息重复。你的系统必须能够**幂等处理重复消息**。

## 一、消息重复的原因

### 1.1 原因分析

| 环节 | 原因 | 后果 |
|---|---|---|
| **生产者** | 重试导致重复发送 | 消息重复 |
| **Broker** | ISR 变化导致 Leader 切换 | 消息重复 |
| **消费者** | Rebalance 导致重新拉取 | 消息重复 |

```
场景 1：生产者重试
Producer ──发送消息 A──> Broker ──ACK 超时──> Producer
                                       │
                        重试 ──发送消息 A──> Broker（消息 A 重复！）

场景 2：消费者 Rebalance
Consumer ──拉取消息 A──> Broker ──处理中──> Rebalance 触发
                                              │
                          Consumer ──重新拉取消息 A──> Broker（消息 A 重复！）
```

### 1.2 幂等性的重要性

| 业务场景 | 重复消费的后果 |
|---|---|
| 扣款 | 多次扣款 |
| 下单 | 重复订单 |
| 发货 | 重复发货 |
| 发短信 | 多次发送 |
| 记录日志 | 可接受 |

**幂等性是金融级系统的基本要求。**

## 二、Producer 端幂等性

### 2.1 幂等发送原理

```
幂等性原理：

每个 Producer 有一个唯一的 PID（Producer ID）
每个消息有一个 Sequence Number

Broker 记录每个 PID 的最大 Sequence Number

收到消息时：
  Sequence Number = max + 1 → 正常写入
  Sequence Number < max + 1 → 重复消息，丢弃
  Sequence Number > max + 1 → 消息丢失，拒绝
```

### 2.2 开启幂等发送

```java
// 开启幂等性
props.put("enable.idempotence", true);  // 默认 true
```

### 2.3 幂等性配置

```java
// 开启幂等性后的自动配置
// acks = all
// retries = Integer.MAX_VALUE
// max.in.flight.requests.per.connection = 5
```

### 2.4 幂等性的限制

| 限制 | 说明 |
|---|---|
| **单 Producer 实例** | 只能保证单个 Producer 的幂等 |
| **重启失效** | Producer 重启后 PID 变化，幂等失效 |
| **顺序保证** | 在 `max.in.flight.requests.per.connection <= 5` 时保证顺序 |

## 三、Consumer 端幂等性

### 3.1 幂等消费方案

| 方案 | 实现方式 | 适用场景 |
|---|---|---|
| **数据库唯一索引** | 消息 ID 作为唯一键 | 有数据库操作的业务 |
| **Redis 去重** | 消息 ID 存入 Redis | 高并发场景 |
| **消息表** | 消费前先记录 | 复杂业务流程 |
| **状态机** | 利用业务状态不可逆 | 有明确状态的业务 |

### 3.2 方案一：数据库唯一索引

```java
// 数据库唯一索引实现幂等
public void consume(OrderMessage message) {
    try {
        Order order = new Order();
        order.setMessageId(message.getId());  // 唯一索引
        order.setOrderId(message.getOrderId());
        order.setAmount(message.getAmount());
        order.setStatus("CREATED");

        orderMapper.insertSelective(order);

    } catch (DuplicateKeyException e) {
        // 唯一索引冲突，说明已处理过，忽略
        log.info("订单 {} 已存在，忽略重复消息", message.getOrderId());
    }
}
```

```sql
-- 建表语句
CREATE TABLE orders (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    message_id VARCHAR(64) NOT NULL UNIQUE,
    order_id VARCHAR(64) NOT NULL,
    amount DECIMAL(10, 2),
    status VARCHAR(20),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### 3.3 方案二：Redis 去重

```java
// Redis 去重
public void consume(OrderMessage message) {
    String key = "processed:order:" + message.getId();
    String value = "1";
    Duration expire = Duration.ofHours(24);

    // 原子性操作
    Boolean isNew = redisTemplate.opsForValue().setIfAbsent(key, value, expire);

    if (Boolean.TRUE.equals(isNew)) {
        // 新消息，处理业务
        processOrder(message);
    } else {
        // 重复消息，忽略
        log.info("消息 {} 已处理，忽略", message.getId());
    }
}
```

### 3.4 方案三：消息表

```java
// 消息表实现幂等
public void consume(OrderMessage message) {
    // 1. 检查消息是否已处理
    MessageRecord record = messageRecordMapper.selectByMessageId(message.getId());
    if (record != null && "COMPLETED".equals(record.getStatus())) {
        log.info("消息 {} 已处理完成，忽略", message.getId());
        return;
    }

    // 2. 处理业务
    processOrder(message);

    // 3. 更新消息状态
    if (record == null) {
        // 插入新记录
        MessageRecord newRecord = new MessageRecord();
        newRecord.setMessageId(message.getId());
        newRecord.setStatus("COMPLETED");
        newRecord.setCreatedAt(new Date());
        messageRecordMapper.insert(newRecord);
    } else {
        // 更新状态
        record.setStatus("COMPLETED");
        record.setUpdatedAt(new Date());
        messageRecordMapper.updateByPrimaryKey(record);
    }
}
```

### 3.5 方案四：状态机

```java
// 状态机实现幂等
public void consume(PaymentMessage message) {
    // 1. 查询订单当前状态
    Order order = orderMapper.selectByOrderId(message.getOrderId());

    // 2. 根据状态判断是否处理
    switch (order.getStatus()) {
        case "PENDING":
            // 待支付 → 已支付
            order.setStatus("PAID");
            order.setPayTime(new Date());
            orderMapper.updateByPrimaryKey(order);
            break;

        case "PAID":
            // 已支付 → 忽略
            log.info("订单 {} 已支付，忽略重复消息", message.getOrderId());
            break;

        case "CANCELLED":
            // 已取消 → 忽略
            log.info("订单 {} 已取消，忽略重复消息", message.getOrderId());
            break;

        default:
            log.warn("订单 {} 状态异常：{}", message.getOrderId(), order.getStatus());
    }
}
```

## 四、完整示例

### 4.1 幂等消费者

```java
// 幂等消费者实现
@Service
public class IdempotentOrderConsumer {

    @Autowired
    private OrderMapper orderMapper;

    @Autowired
    private RedisTemplate<String, String> redisTemplate;

    @Autowired
    private TransactionTemplate transactionTemplate;

    @KafkaListener(topics = "order-topic", groupId = "order-consumer-group")
    public void consume(ConsumerRecord<String, String> record) {
        OrderMessage message = parseMessage(record);

        // 1. Redis 去重（快速判断）
        String dedupKey = "dedup:order:" + message.getId();
        Boolean isProcessed = redisTemplate.hasKey(dedupKey);
        if (Boolean.TRUE.equals(isProcessed)) {
            log.info("消息 {} 已处理，忽略", message.getId());
            return;
        }

        // 2. 业务处理
        try {
            processOrder(message);

            // 3. 标记已处理
            redisTemplate.opsForValue().set(dedupKey, "1", Duration.ofHours(24));

        } catch (DuplicateKeyException e) {
            // 数据库唯一索引冲突，忽略
            log.info("订单 {} 已存在，忽略重复消息", message.getOrderId());

        } catch (Exception e) {
            // 其他异常，抛出以便重试
            throw e;
        }
    }

    private void processOrder(OrderMessage message) {
        transactionTemplate.execute(status -> {
            Order order = new Order();
            order.setMessageId(message.getId());
            order.setOrderId(message.getOrderId());
            order.setAmount(message.getAmount());
            order.setStatus("CREATED");
            orderMapper.insertSelective(order);
            return null;
        });
    }
}
```

### 4.2 去重策略选择

```
去重策略选择：

1. Redis 去重（毫秒级）
   ↓ 缓存未命中
2. 数据库唯一索引（毫秒级）
   ↓ 唯一索引冲突
3. 忽略重复消息

性能最优解：Redis + 数据库唯一索引
```

## 五、Exactly-Once 语义

### 5.1 三种语义对比

| 语义 | 说明 | 重复 | 丢失 |
|---|---|---|---|
| **At-Most-Once** | 最多一次 | ❌ | ✅ 可能 |
| **At-Least-Once** | 至少一次 | ✅ 可能 | ❌ |
| **Exactly-Once** | 恰好一次 | ❌ | ❌ |

### 5.2 Kafka 的 Exactly-Once

Kafka 支持 Exactly-Once，但有限制：

```java
// Kafka Streams 支持 Exactly-Once
props.put("processing.guarantee", "exactly_once_v2");

// 普通消费者不支持 Exactly-Once
// 需要业务配合实现幂等
```

### 5.3 消费-处理-生产链路的 Exactly-Once

```java
// 消费-处理-生产链路
@KafkaListener(topics = "input-topic")
public void consumeAndProduce(ConsumerRecord<String, String> record) {
    producer.initTransactions();

    try {
        producer.beginTransaction();

        // 1. 消费消息
        process(record.value());

        // 2. 业务处理

        // 3. 发送新消息
        producer.send(new ProducerRecord<>("output-topic", record.key(), result));

        // 4. 提交消费 Offset + 生产消息
        producer.sendOffsetsToTransaction(
            Collections.singletonMap(
                new TopicPartition(record.topic(), record.partition()),
                new OffsetAndMetadata(record.offset() + 1)
            ),
            "group-id"
        );

        producer.commitTransaction();

    } catch (Exception e) {
        producer.abortTransaction();
        throw e;
    }
}
```

## 六、面试追问预判

### 追问一：幂等性和事务有什么区别？

| 概念 | 说明 |
|---|---|
| **幂等性** | 防止重复处理，执行一次和多次效果相同 |
| **事务** | 保证原子性，多个操作要么全成功，要么全失败 |

幂等性是事务的基础，事务包含幂等性。

### 追问二：如何选择幂等方案？

| 场景 | 推荐方案 |
|---|---|
| 简单业务（单表操作） | 数据库唯一索引 |
| 高并发场景 | Redis 去重 |
| 复杂业务流程 | 消息表 |
| 有明确状态机 | 状态机 |

### 追问三：幂等发送和幂等消费有什么区别？

| 类型 | 说明 |
|---|---|
| **幂等发送** | Producer 不会重复发送消息（通过 PID + Sequence Number） |
| **幂等消费** | Consumer 可以重复消费消息而不产生副作用（通过业务手段实现） |

### 追问四：Kafka 支持 Exactly-Once 吗？

Kafka 支持 Exactly-Once，但有限制：

1. **Producer 端**：开启幂等性后，同一 Producer 发送的消息不会重复
2. **Consumer 端**：需要业务配合实现幂等
3. **消费-处理-生产链路**：Kafka 支持，但需要使用事务 API
