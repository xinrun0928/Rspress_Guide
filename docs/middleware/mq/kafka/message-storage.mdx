# Kafka 消息存储：Segment + Index + 时间索引

## 场景切入：一条消息是如何存储的？

Kafka 的消息存储是高性能的关键。当消息发送到 Broker 后，它是如何存储到磁盘的？为什么 Kafka 能够快速定位到某条消息？为什么读取速度可以接近内存？

理解 Kafka 的存储机制，才能真正理解它的高性能。

## 一、存储架构概览

### 1.1 目录结构

```
Partition 目录结构：

/var/kafka/data/
└── order-topic-0/                    # Topic 名-分区号
    ├── 00000000000000000000.log    # 数据文件
    ├── 00000000000000000000.index  # 偏移量索引
    ├── 00000000000000000000.timeindex  # 时间索引
    ├── 00000000000000000001.log    # 下一个段
    ├── 00000000000000000001.index
    └── 00000000000000000001.timeindex
```

### 1.2 核心概念

| 概念 | 说明 |
|---|---|
| **Log** | 数据文件，存储消息 |
| **Index** | 偏移量索引，快速定位消息 |
| **TimeIndex** | 时间索引，按时间定位消息 |
| **Segment** | 分段，每个分段包含 .log + .index + .timeindex |

## 二、Segment 分段存储

### 2.1 分段原理

```
分段存储示例：

Segment 0 (0-1000):  00000000000000000000.log
                      offset: 0 ~ 999
                      size: ~1GB

Segment 1 (1000-2000): 00000000000000001000.log
                      offset: 1000 ~ 1999
                      size: ~1GB

Segment 2 (2000+):    00000000000000002000.log
                      offset: 2000 ~
                      size: < 1GB
```

### 2.2 分段配置

```properties
# 分段配置
log.seg.bytes=1073741824                    # 每个段 1GB
log.seg.ms=604800000                        # 或每 7 天新建段
log.seg.index.bytes=10485760                # 索引文件 10MB
```

### 2.3 分段的好处

| 好处 | 说明 |
|---|---|
| **快速删除** | 删除旧数据只需删除整个 Segment |
| **快速查找** | 查找消息只需定位到对应的 Segment |
| **内存友好** | 只需保留热点 Segment 在内存中 |

## 三、消息格式

### 3.1 消息结构

```
消息结构：

┌─────────────────────────────────────────────────────────┐
│  Record Batch（消息批次）                                │
├─────────────────────────────────────────────────────────┤
│  BaseOffset: 1000          # 批次起始偏移量              │
│  BatchLength: 1024         # 批次长度                   │
│  PartitionLeaderEpoch: 1   # 分区 Leader 任期           │
│  Magic: 2                  # 版本号                      │
│  CRC: 0x12345678          # 校验码                      │
│  Attributes: 0             # 属性                       │
│  LastOffsetDelta: 9        # 最后偏移量增量             │
│  FirstTimestamp: 1000000   # 第一条消息时间戳            │
│  ProducerEpoch: 100        # 生产者任期                  │
│  ProducerId: 12345         # 生产者 ID                   │
│  BaseSequence: 0           # 基础序列号                  │
├─────────────────────────────────────────────────────────┤
│  Record（消息记录）                                      │
│  ┌───────────────────────────────────────────────────┐ │
│  │ OffsetDelta: 0      # 相对于 BaseOffset           │ │
│  │ Timestamp: 1000000  # 消息时间戳                   │ │
│  │ Key Length: 4      # Key 长度                     │ │
│  │ Key: "order-1"     # Key 内容                     │ │
│  │ Value Length: 50   # Value 长度                   │ │
│  │ Value: "..."       # Value 内容                   │ │
│  │ Headers: ...       # 消息头                       │ │
│  └───────────────────────────────────────────────────┘ │
│  ... 更多消息                                           │
└─────────────────────────────────────────────────────────┘
```

## 四、索引机制

### 4.1 偏移量索引

```
Index 文件结构：

偏移量索引 (.index)：
┌──────────────┬──────────────┐
│ RelativeOffset │ Position      │
├──────────────┼──────────────┤
│ 0            │ 0            │  ← offset 0 在 log 文件的 position 0
│ 5            │ 1024         │  ← offset 5 在 log 文件的 position 1024
│ 10           │ 2048         │  ← offset 10 在 log 文件的 position 2048
│ ...          │ ...          │
└──────────────┴──────────────┘

特点：
- 稀疏索引，不存储每条消息的索引
- 每个索引条目 8 字节
- 节省空间，查找效率略低
```

### 4.2 消息查找流程

```java
// 查找 offset=1005 的消息
public ByteBuffer findMessage(long targetOffset) {
    // 1. 定位 Segment
    //    二分查找 Segment 文件名中的起始 offset
    Segment segment = findSegment(targetOffset);
    //    例如：Segment 0 (0-999), Segment 1 (1000-1999)
    //    targetOffset=1005 → Segment 1

    // 2. 加载索引文件到内存
    Index index = segment.index();

    // 3. 二分查找索引
    //    查找 <= targetOffset 的最大索引
    //    indexEntries: [0→0, 5→1024, 10→2048, ...]
    //    targetOffset=1005 → 找到 5→1024

    // 4. 从 position 开始顺序扫描
    //    从 log 文件的 position 1024 开始顺序查找
    //    直到找到 offset=1005 的消息

    return message;
}
```

### 4.3 时间索引

```
TimeIndex 文件结构：

时间索引 (.timeindex)：
┌──────────────┬──────────────┐
│ Timestamp    │ RelativeOffset│
├──────────────┼──────────────┤
│ 1000000      │ 0            │  ← timestamp 1000000 对应 offset 0
│ 1005000      │ 50           │  ← timestamp 1005000 对应 offset 50
│ 1010000      │ 100          │  ← timestamp 1010000 对应 offset 100
│ ...          │ ...          │
└──────────────┴──────────────┘

用途：
- 支持按时间查询消息
- 用于日志保留策略（基于时间）
- 用于时间戳索引查找
```

## 五、日志存储参数

### 5.1 关键参数

```properties
# 日志存储配置
log.dirs=/var/kafka/data                    # 数据目录
log.retention.check.interval.ms=300000      # 检查间隔 5 分钟
log.retention.hours=168                     # 保留 7 天
log.retention.bytes=-1                       # 不限制大小（按时间删除）
log.segment.bytes=1073741824                # 每个段 1GB
log.index.size.max.bytes=10485760            # 索引文件最大 10MB
```

### 5.2 保留策略

```properties
# 基于大小的保留
log.retention.bytes=10737418240   # 10GB

# 基于时间的保留
log.retention.hours=168           # 7 天
log.retention.minutes=           # 分钟
log.retention.ms=                # 毫秒

# 同时配置时，任一条件触发即删除
log.retention.bytes=10737418240   # 10GB
log.retention.hours=168           # 7 天
```

## 六、性能优化

### 6.1 页缓存优化

```
Kafka 利用操作系统的页缓存：

写入流程：
  应用 → Page Cache → 磁盘（异步刷盘）

读取流程：
  应用 → Page Cache（命中） → 返回（内存速度）
         ↓ 未命中
       磁盘 → Page Cache → 返回

配置建议：
  - 预留 30-50% 内存给页缓存
  - 避免给 Kafka 分配过多堆内存
```

## 七、面试追问预判

### 追问一：为什么使用稀疏索引？

稀疏索引的好处：
- 节省内存和磁盘空间
- 查找时需要顺序扫描部分日志
- 权衡：空间 vs 查找时间

Kafka 选择了稀疏索引，因为：
- 顺序扫描速度仍然很快
- 内存有限，无法存储全量索引

### 追问二：Segment 文件名的数字是什么？

是起始偏移量（Base Offset）：
- `00000000000000000000.log` → 起始 offset = 0
- `00000000000000001000.log` → 起始 offset = 1000

### 追问三：如何删除过期数据？

1. **基于时间**：按消息时间戳删除
2. **基于大小**：按文件大小删除
3. **基于日志压缩**：保留最新版本

### 追问四：日志压缩是什么？

日志压缩（Log Compaction）：
- 保留每个 Key 的最新消息
- 删除旧版本消息
- 用于场景：消费者重启后需要获取最新状态
