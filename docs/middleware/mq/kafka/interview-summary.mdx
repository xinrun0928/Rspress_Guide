## 一、架构篇

### 问题一：Kafka 的整体架构是怎样的？

**参考答案**：

```
Kafka 整体架构：

┌─────────────────────────────────────────────────────────────┐
│                      ZooKeeper / KRaft                      │
│                    (集群协调与元数据管理)                      │
└─────────────────────────────────────────────────────────────┘
                              │
         ┌────────────────────┼────────────────────┐
         │                    │                    │
         ▼                    ▼                    ▼
   ┌─────────┐          ┌─────────┐          ┌─────────┐
   │ Broker-1│          │ Broker-2│          │ Broker-3│
   │         │          │         │          │         │
   │ Topic-A │          │ Topic-A │          │ Topic-A │
   │ Part-0  │          │ Part-1  │          │ Part-2  │
   │(Leader) │          │(Follower│          │(Follower│
   └─────────┘          └─────────┘          └─────────┘
```

**核心组件**：

| 组件 | 说明 |
|---|---|
| **Broker** | 消息节点，存储消息 |
| **Topic** | 消息分类单元 |
| **Partition** | 分区，实现水平扩展 |
| **Replica** | 副本，保证高可用 |
| **Controller** | 集群控制器，负责 Leader 选举 |

---

### 问题二：Topic 和 Partition 有什么关系？

**参考答案**：

| 关系 | 说明 |
|---|---|
| **逻辑 vs 物理** | Topic 是逻辑概念，Partition 是物理实现 |
| **有序性** | 分区内有序，分区间无序 |
| **并行度** | Partition 数决定消费并行度 |
| **副本** | 每个 Partition 有多个副本 |

```
Topic 与 Partition：

Topic: order-topic (3 个 Partition)

┌─────────────────────────────────────────────────────────┐
│  order-topic                                             │
│                                                         │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐    │
│  │ Partition 0  │  │ Partition 1  │  │ Partition 2  │    │
│  │ [msg 0-99] │  │ [msg 0-80]  │  │ [msg 0-120] │    │
│  │  Leader     │  │  Follower   │  │  Follower   │    │
│  └─────────────┘  └─────────────┘  └─────────────┘    │
└─────────────────────────────────────────────────────────┘
```

## 二、原理篇

### 问题三：为什么 Kafka 这么快？

**参考答案**：

Kafka 高吞吐的四大核心优化：

| 优化 | 原理 | 性能提升 |
|---|---|---|
| **顺序写** | 消息追加到日志文件末尾，无随机写 | 1000x vs 随机写 |
| **零拷贝** | sendfile 系统调用，内核直接传输 | 2-3x |
| **批量处理** | 批量发送、批量消费、批量压缩 | 5-10x |
| **页缓存** | OS 在内存中缓存磁盘数据 | 显著提升 |

**追问**：顺序写为什么快？

磁盘顺序写的速度接近内存，因为：
- 磁头无需移动
- OS 预读生效
- 磁盘缓存命中率高

---

### 问题四：什么是 ISR？

**参考答案**：

ISR（In-Sync Replicas）是与 Leader 保持同步的副本集合：

```
ISR 集合：

┌─────────────────────────────────────────────────────────┐
│                                                         │
│  Leader: Broker-1                                       │
│  ISR: [Broker-1, Broker-2, Broker-3]                  │
│  OSR: [Broker-4]  ← 落后太多，被踢出 ISR              │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**ISR 的判定条件**：

| 条件 | 说明 |
|---|---|
| 时间条件 | 副本与 Leader 的时间差 `< replica.lag.time.max.ms` |
| 消息条件 | 副本落后的消息数 `< replica.lag.max.messages` |

---

### 问题五：HW 和 LEO 是什么？

**参考答案**：

| 概念 | 说明 |
|---|---|
| **LEO** | Log End Offset，下一条消息写入的位置 |
| **HW** | High Watermark，消费者可见消息的分界线 |

```
HW 与 LEO：

┌─────────────────────────────────────────────────────────┐
│                                                         │
│  [msg 0] [msg 1] [msg 2] [msg 3] [msg 4] [msg 5]      │
│      ▲                                     ▲            │
│      │                                     │            │
│    LEO=6                              HW=4             │
│  (下一条写入位置)                   (可见性分界线)         │
│                                                         │
│  消费者可见范围：[msg 0, msg 1, msg 2, msg 3]          │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

---

### 问题六：Leader 宕机后如何选举？

**参考答案**：

```
Leader 选举流程：

1. Controller 检测到 Leader 不可用
   │
2. Controller 从 ISR 中选择新 Leader
   │   策略：选择 ISR 中第一个存活的 Broker
   │
3. Controller 更新 ZooKeeper 元数据
   │
4. Controller 通知所有 Broker
   │
5. 客户端自动重连到新 Leader
```

**关键配置**：

```properties
unclean.leader.election.enable=false  # 禁止从非 ISR 选举
```

---

### 问题七：Controller 的职责是什么？

**参考答案**：

Controller 是集群中的特殊 Broker，负责：

| 职责 | 说明 |
|---|---|
| **Leader 选举** | 当 Leader 宕机时，选择新 Leader |
| **分区副本分配** | 管理分区的副本分配 |
| **Broker 管理** | 处理 Broker 的上下线 |
| **ISR 管理** | 监控副本状态，维护 ISR |
| **Topic 管理** | 创建、删除、修改 Topic |

**Controller 选举**：基于 ZooKeeper 临时节点，第一个创建成功的 Broker 成为 Controller。

## 三、生产者篇

### 问题八：如何保证消息不丢失？

**参考答案**：

需要三个环节都做好：

```
┌─────────────────────────────────────────────────────────┐
│                    消息传递三环节                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  生产者 ────────► Broker ────────► 消费者              │
│                                                         │
│  1. 生产者端：确认机制 + 重试 + 幂等                      │
│  2. Broker 端：副本机制 + 刷盘策略                       │
│  3. 消费者端：手动提交 Offset                            │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**生产端配置**：

```java
props.put("acks", "all");              // 所有副本确认
props.put("retries", 3);               // 重试 3 次
props.put("enable.idempotence", true); // 开启幂等
```

---

### 问题九：acks 配置有什么区别？

**参考答案**：

| acks | 说明 | 可靠性 | 性能 |
|---|---|---|---|
| `0` | 不等待任何确认 | 低 | 最高 |
| `1` | 只等 Leader 确认 | 中 | 高 |
| `all` | 等待 ISR 所有副本确认 | 高 | 低 |

### 问题十：如何保证消息顺序？

**参考答案**：

Kafka 通过分区机制保证局部顺序：

| 类型 | 定义 | 实现方式 |
|---|---|---|
| **全局顺序** | 所有消息按发送顺序消费 | 单分区 + 单消费者 |
| **局部顺序** | 同一 Key 的消息按顺序消费 | 同一 Key 到同一分区 |

```java
// 使用消息 Key 保证同一订单的消息到同一分区
producer.send(new ProducerRecord<>(
    "order-topic",
    order.getId(),  // 订单 ID 作为 Key
    order.toString()
));
```

## 四、消费者篇

### 问题十一：消费者组是如何工作的？

**参考答案**：

消费者组是一组 Consumer 实例，共同消费一个 Topic：

```
消费者组示例：

Topic: order-topic (4 个 Partition)

消费者组 A:
  Consumer 1 ──> Partition 0
  Consumer 2 ──> Partition 1
  Consumer 3 ──> Partition 2
  Consumer 4 ──> Partition 3

消费者组 B:
  Consumer 1 ──> Partition 0, 1, 2, 3  ← 独立消费

每个消费者组独立消费，互不影响
```

---

### 问题十二：Rebalance 是什么？如何避免？

**参考答案**：

Rebalance 是消费者组成员变化时，重新分配分区的过程：

**触发条件**：

| 条件 | 说明 |
|---|---|
| Consumer 宕机 | 长时间未发送心跳 |
| Consumer 加入/离开 | 组内成员变化 |
| 分区数变化 | Topic 分区数增加 |

**优化方案**：

```java
// 避免频繁 Rebalance
props.put("heartbeat.interval.ms", 10000);    // 降低心跳间隔
props.put("session.timeout.ms", 45000);      // 提高会话超时
props.put("max.poll.interval.ms", 600000);    // 提高拉取间隔
props.put("partition.assignment.strategy",
    "cooperative-sticky");                    // 协作式再平衡
```

## 五、存储篇

### 问题十三：Kafka 的消息存储结构是怎样的？

**参考答案**：

```
Partition 目录结构：

/var/kafka/data/
└── order-topic-0/
    ├── 00000000000000000000.log    # 数据文件
    ├── 00000000000000000000.index  # 偏移量索引
    ├── 00000000000000000000.timeindex  # 时间索引
    └── ...
```

| 文件 | 说明 |
|---|---|
| **.log** | 数据文件，存储消息 |
| **.index** | 偏移量索引，快速定位消息 |
| **.timeindex** | 时间索引，按时间定位消息 |

---

### 问题十四：Segment 是什么？有什么好处？

**参考答案**：

Segment 是分段存储的单元，每个分段包含 .log + .index + .timeindex：

| 好处 | 说明 |
|---|---|
| **快速删除** | 删除旧数据只需删除整个 Segment |
| **快速查找** | 只需定位到对应的 Segment |
| **内存友好** | 只需保留热点 Segment 在内存中 |

## 六、面试技巧

### 6.1 回答问题的框架

```
1. 背景说明：简述问题的业务背景
2. 核心原理：解释技术原理
3. 实现方案：给出具体方案或代码
4. 优缺点分析：对比不同方案
5. 追问预判：准备可能被追问的问题
```

### 6.2 高频追问方向

| 问题 | 追问方向 |
|---|---|
| 消息不丢失 | 重试机制、死信队列 |
| Kafka 高吞吐 | 零拷贝原理、Page Cache |
| ISR | HW/LEO、故障转移 |
| 消息顺序 | 多线程消费、重试乱序 |
| Rebalance | 协作式再平衡、静态成员 |

| 加分项 | 说明 |
