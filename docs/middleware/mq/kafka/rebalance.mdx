# Kafka Rebalance 触发条件与分区分配策略

## 场景切入：Rebalance 是双刃剑

Rebalance 让 Kafka 的消费者组具备了动态扩缩容的能力。但它也是一把双刃剑——频繁的 Rebalance 会导致消费暂停、消息重复、甚至系统抖动。

理解 Rebalance 的触发条件和分配策略，才能避免它的副作用。

## 一、Rebalance 触发条件

### 1.1 触发条件总览

| 触发条件 | 说明 | 触发时机 |
|---|---|---|
| **Consumer 宕机** | Consumer 长时间未发送心跳 | session.timeout.ms 后检测 |
| **Consumer 加入** | 新 Consumer 加入组 | 即时触发 |
| **Consumer 离开** | Consumer 主动离开（unsubscribe） | 即时触发 |
| **分区数变化** | Topic 分区数增加 | 即时触发 |
| **Subscription 变化** | Consumer 订阅的 Topic 变化 | 即时触发 |

### 1.2 Consumer 宕机检测

```
心跳检测机制：

┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  Consumer                            Coordinator           │
│     │                                    │                 │
│     │──────── heartbeat ───────────────>│                 │
│     │                                    │                 │
│     │<─────── heartbeat response ───────│                 │
│     │                                    │                 │
│     │──────── heartbeat ───────────────>│                 │
│     │                                    │                 │
│     │──────── heartbeat ───────────────>│                 │
│     │  ❌ Consumer 宕机                   │                 │
│     │  （心跳中断）                        │                 │
│     │                                    │                 │
│     │                          session.timeout.ms 后         │
│     │                          Coordinator 触发 Rebalance  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

```java
// 心跳参数配置
props.put("heartbeat.interval.ms", 3000);   // 每 3 秒发送心跳
props.put("session.timeout.ms", 30000);     // 30 秒未心跳认为宕机
props.put("max.poll.interval.ms", 300000);  // 5 分钟未 poll 认为宕机
```

### 1.3 分区数变化触发

```bash
# 增加 Topic 分区数，触发 Rebalance
kafka-topics.sh --alter --topic order-topic \
  --partitions 6 \
  --bootstrap-server localhost:9092
```

```
分区数变化后的 Rebalance：

变化前（4 个分区）：
  Consumer 1 ──> Partition 0, 1
  Consumer 2 ──> Partition 2, 3

变化后（6 个分区）：
  Consumer 1 ──> Partition 0, 1, 4
  Consumer 2 ──> Partition 2, 3, 5
  ──> Rebalance，重新分配分区
```

### 1.4 Subscription 变化触发

```java
// Subscription 变化触发 Rebalance
consumer.subscribe(Arrays.asList("topic1", "topic2"));  // 新增订阅

// 取消订阅触发 Rebalance
consumer.unsubscribe();  // 取消所有订阅

// 重新分配触发 Rebalance
consumer.subscribe(Pattern.compile("topic.*"));  // 使用正则订阅
```

## 二、分区分配策略

### 2.1 分配策略总览

| 策略 | 类名 | 特点 |
|---|---|---|
| **Range** | RangeAssignor | 按 Topic 分配，可能不均衡 |
| **RoundRobin** | RoundRobinAssignor | 轮询分配，相对均衡 |
| **StickyAssignor** | StickyAssignor | 粘性分配，最小化 Rebalance 影响 |
| **CooperativeStickyAssignor** | CooperativeStickyAssignor | 协作式粘性，增量重分配 |
| **自定义** | 自定义实现 | 特殊业务需求 |

```java
// 配置分配策略
props.put("partition.assignment.strategy", "cooperative-sticky");
```

### 2.2 Range 分配策略

**原理**：按 Topic 分别分配，每个 Topic 内的分区按范围分配给 Consumer。

```
Range 分配示例：

Topic A: 3 个分区
Topic B: 3 个分区
Consumer: 2 个

Topic A 分配：
  Consumer 1: Partition 0, 1
  Consumer 2: Partition 2

Topic B 分配：
  Consumer 1: Partition 0, 1
  Consumer 2: Partition 2

分配结果：
  Consumer 1: A-0, A-1, B-0, B-1
  Consumer 2: A-2, B-2
```

**特点**：

| 特点 | 说明 |
|---|---|
| **简单** | 按范围分配，实现简单 |
| **可能不均** | 如果分区数不是 Consumer 的整数倍，前几个 Consumer 会多分 |
| **不跨 Topic** | 每个 Topic 独立分配 |

### 2.3 RoundRobin 分配策略

**原理**：将所有 Topic 的分区放在一起，轮询分配给 Consumer。

```
RoundRobin 分配示例：

Topic A: 3 个分区 (A-0, A-1, A-2)
Topic B: 3 个分区 (B-0, B-1, B-2)
Consumer: 2 个

轮询分配：
  A-0 ──> Consumer 1
  A-1 ──> Consumer 2
  A-2 ──> Consumer 1
  B-0 ──> Consumer 2
  B-1 ──> Consumer 1
  B-2 ──> Consumer 2

分配结果：
  Consumer 1: A-0, A-2, B-1
  Consumer 2: A-1, B-0, B-2
```

**特点**：

| 特点 | 说明 |
|---|---|
| **均衡** | 分区均匀分配 |
| **跨 Topic** | 所有分区一起轮询 |
| **可能跨 Topic** | 同一 Consumer 可能处理不同 Topic 的分区 |

### 2.4 StickyAssignor 分配策略

**原理**：尽量保持原有的分区分配不变，最小化 Rebalance 的影响。

```
Sticky 分配示例：

初始分配：
  Consumer 1: A-0, A-1, B-0, B-1
  Consumer 2: A-2, B-2

Consumer 2 宕机，触发 Rebalance：

普通 Rebalance（Range/RoundRobin）：
  Consumer 1: A-0, A-1, A-2, B-0, B-1, B-2  ← 全部接管
  Consumer 2: (无)

Sticky Rebalance：
  Consumer 1: A-0, A-1, A-2, B-0, B-1, B-2  ← 全部接管
  Consumer 2: (无)

区别：Sticky 会尽量保持原有的分配顺序，减少 Partition 所有权变化
```

**特点**：

| 特点 | 说明 |
|---|---|
| **最小化影响** | Rebalance 时分区变化最小 |
| **保持均衡** | 在最小化变化的前提下保持均衡 |
| **更复杂** | 实现较复杂 |

### 2.5 CooperativeStickyAssignor 分配策略

**原理**：协作式再平衡，只迁移需要迁移的分区，其他分区继续消费。

```
协作式再平衡 vs 普通 Rebalance：

普通 Rebalance（Eager）：
  Consumer 1 ──[消费]──> Rebalance ──[全部停止]──> 新分配
  Consumer 2 ──[消费]──> Rebalance ──[全部停止]──> 新分配
                        ↑
                    全部停止

协作式再平衡：
  Consumer 1 ──[消费]──> 增量重分配 ──[继续消费未分配]──> 新分配
  Consumer 2 ──[消费]──> 增量重分配 ──[继续消费未分配]──> 新分配
                        ↑
                    只停止受影响的分区
```

## 三、分配策略对比

### 3.1 对比表

| 策略 | 均衡性 | Rebalance 影响 | 适用场景 |
|---|---|---|---|
| **Range** | 中 | 中 | 单 Topic 场景 |
| **RoundRobin** | 好 | 中 | 多 Topic 场景 |
| **Sticky** | 好 | 小 | 生产环境 |
| **CooperativeSticky** | 好 | 最小 | Kafka 2.4+，生产环境推荐 |

### 3.2 选择建议

```java
// 推荐配置
props.put("partition.assignment.strategy",
    "org.apache.kafka.clients.consumer.CooperativeStickyAssignor");
```

| 场景 | 推荐策略 |
|---|---|
| **单 Topic** | Range 或 Sticky |
| **多 Topic** | RoundRobin 或 Sticky |
| **Kafka 2.4+** | CooperativeSticky |
| **生产环境** | CooperativeSticky |

## 四、Rebalance 流程详解

### 4.1 JoinGroup 阶段

```java
// JoinGroup 请求
public class JoinGroupRequest {
    String groupId;           // 消费者组 ID
    String memberId;          // 消费者 ID
    String sessionTimeout;   // 会话超时
    List<String> topics;     // 订阅的 Topic
    String assignmentStrategy; // 分配策略
}
```

```
JoinGroup 阶段：

1. Consumer 发送 JoinGroup 请求给 Coordinator
   │
2. Coordinator 等待所有 Consumer 加入
   │
3. 所有 Consumer 加入后，Coordinator 选择 Leader
   │
4. Coordinator 返回 JoinGroup 响应
```

### 4.2 SyncGroup 阶段

```
SyncGroup 阶段：

1. Leader 收到 JoinGroup 响应
   │
2. Leader 根据分配策略制定分区分配方案
   │
3. Leader 发送 SyncGroup 请求（携带分配方案）
   │
4. 其他 Consumer 发送 SyncGroup 请求（不带方案）
   │
5. Coordinator 返回分配方案给所有 Consumer
```

### 4.3 Leader 选举

```java
// Leader 选举逻辑
public class Coordinator {
    public void onJoinGroupComplete(String groupId, List<Member> members) {
        // 第一个加入的 Consumer 成为 Leader
        String leaderId = members.get(0).memberId();

        // 返回响应
        for (Member member : members) {
            if (member.memberId().equals(leaderId)) {
                // Leader：携带分配方案
                sendJoinResponse(member, leaderId, members, assignment);
            } else {
                // 普通 Consumer：不携带方案
                sendJoinResponse(member, leaderId, members, null);
            }
        }
    }
}
```

## 五、Rebalance 优化

### 5.1 减少 Rebalance 触发

```java
// 优化配置
props.put("heartbeat.interval.ms", 10000);    // 降低心跳间隔
props.put("session.timeout.ms", 45000);       // 提高会话超时
props.put("max.poll.interval.ms", 600000);     // 提高拉取间隔
props.put("request.timeout.ms", 30000);       // 提高请求超时
```

| 参数 | 调整方向 | 说明 |
|---|---|---|
| `heartbeat.interval.ms` | 降低 | 更频繁的心跳 |
| `session.timeout.ms` | 提高 | 给 Consumer 更多恢复时间 |
| `max.poll.interval.ms` | 提高 | 给 Consumer 更长的处理时间 |

### 5.2 使用静态成员

```java
// 静态成员配置
props.put("group.instance.id", "consumer-001");
```

```
静态成员的好处：

Consumer 重启前：
  Consumer 1 (id=consumer-001) ──> Partition 0, 1

Consumer 重启后：
  Consumer 1 (id=consumer-001) ──> Partition 0, 1  ← 保持原有分配
  不触发 Rebalance！
```

### 5.3 异常处理避免 Rebalance

```java
// 避免 Rebalance 的处理方式
while (true) {
    ConsumerRecords<String, String> records = consumer.poll(Duration.ofSeconds(1));

    for (ConsumerRecord<String, String> record : records) {
        try {
            // 业务处理
            process(record);
        } catch (Exception e) {
            // 不要让异常传播出去，避免 Rebalance
            log.error("处理失败，offset={}", record.offset(), e);
            // 记录失败，后续补偿处理
        }
    }
}
```

## 六、面试追问预判

### 追问一：Sticky 和 CooperativeSticky 有什么区别？

| 策略 | Rebalance 影响 |
|---|---|
| **Sticky** | Rebalance 时全部 Consumer 暂停，但分区变化最小 |
| **CooperativeSticky** | 只有受影响的 Consumer 暂停，其他 Consumer 继续消费 |

### 追问二：如何选择分配策略？

| 场景 | 推荐 |
|---|---|
| **Kafka 2.4+** | CooperativeSticky |
| **需要快速 Rebalance** | Sticky |
| **单 Topic** | Range |
| **多 Topic** | RoundRobin |

### 追问三：Consumer 数量变化时会发生什么？

1. **Consumer 加入**：触发 Rebalance，重新分配分区
2. **Consumer 离开**：触发 Rebalance，分配给其他 Consumer
3. **Consumer 宕机**：等待 session.timeout.ms 后触发 Rebalance

### 追问四：如何监控 Rebalance？

```bash
# 查看 Rebalance 次数
kafka-consumer-groups.sh --bootstrap-server localhost:9092 \
  --group order-consumer-group \
  --describe \
  --members

# 输出 Rebalance 相关信息
```
