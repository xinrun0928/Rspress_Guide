# Kafka 日志清理策略：delete vs compact

## 场景切入：Kafka 是如何清理过期数据的？

Kafka 不是无限存储的，它需要清理旧数据来释放空间。

但清理策略选错了，可能会丢消息。Delete 和 Compaction 各有适用场景，你选对了吗？

## 一、清理策略概述

### 1.1 两种策略对比

| 策略 | 说明 | 适用场景 |
|---|---|---|
| **Delete** | 按时间或大小删除旧消息 | 日志收集、事件流 |
| **Compact** | 保留每个 Key 的最新消息 | 消费者状态恢复、配置管理 |

```
Delete 策略：
┌─────────────────────────────────────────┐
│                                         │
│  [msg A] [msg B] [msg C] [msg D] [msg E] │
│      ↓                                   │
│  删除旧消息                               │
│      ↓                                   │
│  [msg D] [msg E]                         │
│                                         │
└─────────────────────────────────────────┘

Compact 策略：
┌─────────────────────────────────────────┐
│                                         │
│  Key=1: [msg A v1] [msg A v2] [msg A v3] │
│  Key=2: [msg B v1] [msg B v2]           │
│                                         │
│  压缩后（保留最新）：                    │
│  Key=1: [msg A v3]                      │
│  Key=2: [msg B v2]                      │
│                                         │
└─────────────────────────────────────────┘
```

### 1.2 配置清理策略

```properties
# Topic 级别配置
cleanup.policy=delete        # 或 compact

# 全局默认配置
log.cleanup.policy=delete    # 默认 delete
```

## 二、Delete 策略

### 2.1 基于时间的删除

```properties
# 基于时间的保留策略
log.retention.hours=168              # 保留 7 天
log.retention.minutes=               # 分钟
log.retention.ms=604800000           # 毫秒
```

```
基于时间的删除：

消息时间戳：10:00        消息时间戳：17:00
      ↓                        ↓
┌─────────────────────────────────────────────────┐
│ 10:00 │ 11:00 │ 12:00 │ ... │ 17:00 │ 18:00   │
└─────────────────────────────────────────────────┘
      ↑
  超过保留时间，被删除

保留 7 天：7 天前的消息被删除
```

### 2.2 基于大小的删除

```properties
# 基于大小的保留策略
log.retention.bytes=10737418240   # 10GB per partition
# -1 表示不限制大小
```

```
基于大小的删除：

每个 Partition 最多保留 10GB：
┌─────────────────────────────────────┐
│                                     │
│  [msg 1] [msg 2] ... [msg 1000000] │
│                                     │
│  达到 10GB 时，删除最早的 10%       │
│                                     │
│  [msg 100001] ... [msg 1000000]    │
│                                     │
└─────────────────────────────────────┘
```

### 2.3 删除流程

```
删除流程：

1. Controller 定期检查 Segment 是否可以删除
   │
2. 检查条件：
   - 消息时间 > log.retention.ms
   - Segment 大小 > 保留大小
   │
3. 满足条件，标记 Segment 为待删除
   │
4. 后台线程执行删除
   │
5. 删除文件或移动到 .deleted 目录
```

## 三、Compact 策略

### 3.1 原理

Compact 策略保留每个 Key 的最新消息：

```
Compact 示例：

原始消息：
  Key=order-1, offset=0, value={"status": "created"}
  Key=order-1, offset=1, value={"status": "paid"}
  Key=order-1, offset=2, value={"status": "shipped"}
  Key=order-2, offset=3, value={"status": "created"}

压缩后：
  Key=order-1, offset=2, value={"status": "shipped"}
  Key=order-2, offset=3, value={"status": "created"}

注意：
- offset 保留了最新的
- 旧版本消息被删除
```

### 3.2 配置

```properties
# 启用 Compact
cleanup.policy=compact

# 最小消息数（确保有足够消息可压缩）
log.cleaner.min.compact.lag.ms=3600000       # 消息产生后至少 1 小时才能被压缩
log.cleaner.max.compact.lag.ms=               # 最大延迟

# 压缩线程数
num.io.threads=8                              # IO 线程数
log.cleaner.threads=2                         # 压缩线程数
```

### 3.3 墓碑消息

```
墓碑消息（Tombstone）：

当消息的 value 为 null 时，表示该 Key 被删除

原始消息：
  Key=order-1, value={"status": "shipped"}
  Key=order-1, value=null  ← 墓碑消息

压缩后：
  Key=order-1, value=null  ← 墓碑消息保留一段时间后被删除

用途：删除消费者不再需要的 Key
```

### 3.4 Head 和 Tail

```
Segment 的 Head 和 Tail：

┌─────────────────────────────────────────────────────┐
│                                                     │
│  Tail（可被压缩区域）    Head（不可压缩区域）       │
│                                                     │
│  [旧消息] ... ... ... ... ... ... [新消息]         │
│                                                     │
│   ↓                                        ↑        │
│  可能被压缩                          不会被压缩       │
│                                                     │
└─────────────────────────────────────────────────────┘

配置：
log.cleaner.delete.retention.ms=86400000  # 墓碑消息保留 24 小时
```

## 四、混合策略

### 4.1 同时启用两种策略

```properties
# 同时启用 delete 和 compact
cleanup.policy=delete,compact
```

### 4.2 分区配置

```bash
# 创建 Topic 时配置
kafka-topics.sh --create --topic order-events \
  --partitions 6 \
  --replication-factor 3 \
  --config cleanup.policy=compact \
  --config retention.ms=604800000 \
  --bootstrap-server localhost:9092

# 修改 Topic 配置
kafka-topics.sh --alter --topic order-events \
  --add-config cleanup.policy=compact \
  --bootstrap-server localhost:9092
```

## 五、日志清理器

### 5.1 Log Cleaner 组件

```
Log Cleaner 组件：

┌─────────────────────────────────────────────────────┐
│                  Log Cleaner                        │
├─────────────────────────────────────────────────────┤
│                                                     │
│  ┌──────────────┐   ┌──────────────┐               │
│  │  Cleaner     │   │  OffsetMap  │               │
│  │  Thread      │   │  (内存)     │               │
│  └──────┬───────┘   └──────────────┘               │
│         │                                        │
│         ▼                                        │
│  ┌──────────────────────────────────────────┐     │
│  │            Segment 文件                    │     │
│  │  [msg A] [msg A] [msg B] [msg B] [msg C] │     │
│  │       ↑      ↑      ↑                     │     │
│  │       └── 新消息覆盖旧消息 ──┘             │     │
│  └──────────────────────────────────────────┘     │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### 5.2 清理过程

```java
// 清理过程
public void cleanSegment(Segment segment) {
    // 1. 读取 Segment
    FileChannel channel = segment.log().fileChannel();

    // 2. 创建 OffsetMap
    OffsetMap offsetMap = new OffsetMap(16 * 1024 * 1024);  // 16MB

    // 3. 从 Tail 开始扫描
    for (long offset : segment.tailOffsets()) {
        // 4. 检查 Key 是否已存在
        if (offsetMap.contains(message.key())) {
            // 跳过旧消息（标记为删除）
        } else {
            // 添加到 OffsetMap
            offsetMap.put(message.key(), offset);
        }
    }

    // 5. 写入新 Segment
    writeToNewSegment(validMessages);
}
```

## 六、面试追问预判

### 追问一：Delete 和 Compact 有什么区别？

| 策略 | 删除方式 | 保留内容 |
|---|---|---|
| **Delete** | 删除整个消息 | 无 |
| **Compact** | 只删除 Key 的旧版本 | 每个 Key 的最新版本 |

### 追问二：Compact 适用于什么场景？

| 场景 | 说明 |
|---|---|
| **消费者状态恢复** | 消费者重启后需要获取最新状态 |
| **配置管理** | 保留最新配置，覆盖旧配置 |
| **主键表** | 类似数据库表，按 Key 更新 |

### 追问三：墓碑消息是什么？

当消息的 value 为 null 时，表示该 Key 被删除。墓碑消息会被保留一段时间（`delete.retention.ms`），然后被删除。

### 追问四：如何监控日志清理？

```bash
# 查看清理状态
kafka-logsegments.sh --describe --topic order-topic --bootstrap-server localhost:9092

# 查看清理线程
jstack <kafka_pid> | grep Cleaner
```
