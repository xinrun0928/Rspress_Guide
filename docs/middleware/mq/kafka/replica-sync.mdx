# Kafka 副本同步原理与故障转移

## 场景切入：一次故障的全流程

凌晨 3 点，你收到告警：Broker-2 宕机了。

你揉揉眼睛，心想：还好我们配置了 3 副本，应该没问题。但具体发生了什么？Leader 怎么切换的？消息有没有丢失？切换要多久？

理解副本同步原理和故障转移机制，是深入掌握 Kafka 的必经之路。

## 一、副本同步全流程

### 1.1 写入流程

```
完整写入流程：

Producer
    │
    │ send(ProduceRequest)
    ▼
┌─────────────────────────────────────────────────────────┐
│                      Leader Broker                        │
│                                                         │
│  1. 接收请求                                            │
│  2. 验证请求（权限、格式等）                             │
│  3. 追加写入本地日志                                     │
│  4. LEO = current + 1                                  │
│  5. 等待 ISR 中所有 Follower 同步                       │
│  6. 更新 HW                                            │
│  7. 返回 ACK 给 Producer                                 │
│                                                         │
└─────────────────────────────────────────────────────────┘
    │
    │ 发送 FETCH 请求
    ▼
┌─────────────────────────────────────────────────────────┐
│                    Follower Broker                       │
│                                                         │
│  1. 接收 FETCH 请求                                      │
│  2. 读取 Leader 日志中的消息                             │
│  3. 追加写入本地日志                                     │
│  4. LEO = Leader LEO                                   │
│  5. 发送 FETCH 响应                                     │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 1.2 同步机制详解

#### 步骤一：Leader 写入本地日志

```java
// Leader 写入日志
public class Leader {
    public void append(Message message) {
        // 追加到日志文件末尾
        log.append(message);

        // 更新 LEO
        this.leo = message.offset + 1;

        // 记录待同步消息
        this.pendingMessages.add(message);
    }
}
```

#### 步骤二：Follower 拉取数据

```
Follower 拉取流程：

1. Follower 发送 FETCH 请求（携带当前 LEO）
   │
2. Leader 收到请求
   │
3. 检查 Follower 的 LEO
   │
4. 从 LEO 位置开始读取消息
   │
5. 返回消息列表（可能为空）
   │
6. 更新 Follower 的 HW
```

```java
// Follower 拉取消息
public class Follower {
    public void fetch() {
        // 发送 FETCH 请求
        FetchRequest request = new FetchRequest(
            this.currentLeader,
            this.partition,
            this.leo  // 当前 LEO，告知 Leader 从哪里开始发送
        );

        // 接收响应
        FetchResponse response = send(request);

        // 追加消息到本地日志
        for (Message message : response.messages) {
            log.append(message);
        }

        // 更新 LEO
        this.leo = response.leaderEndOffset;

        // 更新 HW
        this.hw = Math.min(this.leo, response.highWatermark);
    }
}
```

### 1.3 同步参数详解

```properties
# 关键同步参数

# Leader 等待 Follower 同步的最大时间
replica.lag.time.max.ms=10000

# 如果 Follower 在 10 秒内没有发送 FETCH 请求或同步数据，
# 会被认为已不同步

# 副本最大落后消息数
replica.lag.max.messages=40000

# 如果 Follower 落后的消息数超过 40000，
# 会被踢出 ISR（注意：Kafka 2.5+ 已移除此参数）
```

## 二、ISR 动态管理

### 2.1 ISR 变化触发条件

```
ISR 变化场景：

场景 1：Follower 正常同步
  ISR: [Leader, Follower1, Follower2]

场景 2：Follower 网络延迟
  超过 replica.lag.time.max.ms 未同步
  ISR: [Leader, Follower1]  ← Follower2 被踢出

场景 3：Follower 恢复同步
  Follower2 从上次 LEO 位置继续同步
  追上 Leader 后，重新加入 ISR
  ISR: [Leader, Follower1, Follower2]
```

### 2.2 ISR 变化的代码逻辑

```java
// 副本管理器检查 ISR
public class ReplicaManager {

    public void maybeExpandIsr(Partition partition, Replica followerReplica) {
        // 检查 Follower 是否追上 Leader
        if (followerReplica.logEndOffset() >= partition.leaderLogEndOffset()) {
            // 追上 Leader，加入 ISR
            partition.addReplicaToIsr(followerReplica);
        }
    }

    public void maybeShrinkIsr(Partition partition, Replica followerReplica) {
        // 检查 Follower 是否落后太多
        long followerLag = partition.leaderLogEndOffset()
            - followerReplica.logEndOffset();

        long lagTime = System.currentTimeMillis()
            - followerReplica.lastFetchTime();

        if (followerLag > replicaLagMaxMessages
            || lagTime > replicaLagTimeMaxMs) {
            // 落后太多，踢出 ISR
            partition.removeReplicaFromIsr(followerReplica);
        }
    }
}
```

### 2.3 ISR 与可用性

ISR 的大小直接影响可用性：

| ISR 大小 | 说明 | 可用性影响 |
|---|---|---|
| 1 | 只有 Leader 在 ISR 中 | Leader 宕机，不可服务 |
| 2 | Leader + 1 个 Follower | 最多容忍 1 个 Broker 宕机 |
| 3 | Leader + 2 个 Follower | 最多容忍 2 个 Broker 宕机 |

## 三、HW 机制详解

### 3.1 HW 的作用

HW（High Watermark）解决了两个问题：

| 问题 | HW 如何解决 |
|---|---|
| **消费可见性** | 只有 HW 之前的消息对消费者可见 |
| **数据一致性** | Leader 切换时，避免重复消息 |

### 3.2 HW 更新规则

```
HW 更新规则：

1. Producer 写入消息到 Leader
   │
2. Leader LEO 更新
   │
3. Leader 等待 ISR 中所有 Follower 同步
   │
4. Leader 更新 HW = min(Leader LEO, 所有 Follower LEO)
   │
5. Leader 通知 Follower HW
   │
6. Follower 更新 HW
```

```java
// Leader 更新 HW
public void updateHighWatermark() {
    long minLEO = this.leo;  // 初始化为 Leader LEO

    // 遍历所有 ISR 中的 Follower
    for (Replica follower : this.isr) {
        minLEO = Math.min(minLEO, follower.logEndOffset());
    }

    this.highWatermark = minLEO;
}
```

### 3.3 HW 截断机制

当 Leader 切换时，需要截断多余的日志：

```
截断机制示例：

旧 Leader (Broker-1) 宕机，新 Leader (Broker-2) 上任

Broker-2 当前状态：
  LEO = 5, HW = 5
  [msg 0] [msg 1] [msg 2] [msg 3] [msg 4]

Broker-1 (旧 Leader) 日志：
  LEO = 7, HW = 5
  [msg 0] [msg 1] [msg 2] [msg 3] [msg 4] [msg 5] [msg 6]

Broker-2 成为新 Leader 后的处理：
  Broker-2 LEO = 5 ← 无需截断
  Broker-1 需要截断到 HW = 5
  [msg 0] [msg 1] [msg 2] [msg 3] [msg 4] ✓

这样就保证了数据一致性！
```

## 四、故障转移

### 4.1 Controller 职责

Controller 是集群中的特殊 Broker，负责：

| 职责 | 说明 |
|---|---|
| **分区 Leader 选举** | 当 Leader 宕机时，选择新 Leader |
| **ISR 管理** | 监控副本状态，维护 ISR |
| **分区副本分配** | 管理分区的副本分配 |
| **Broker 注册/注销** | 处理 Broker 的上下线 |

### 4.2 Controller 选举

```
Controller 选举流程（基于 ZooKeeper）：

1. Broker 启动时在 ZooKeeper 创建临时节点
   /kafka/controller
   │
2. 第一个创建的 Broker 成为 Controller
   │
3. 其他 Broker 监听 controller 节点变化
   │
4. Controller 宕机，临时节点消失
   │
5. 其他 Broker 竞争创建 controller 节点
   │
6. 胜出的 Broker 成为新 Controller
```

```java
// Controller 选举核心逻辑
public class ZkLeaderElector {

    public void elect() {
        // 尝试创建临时节点
        String result = zk.create(
            "/kafka/controller",
            brokerId.getBytes(),
           Ids.OPEN_ACL_UNSAFE,
            CreateMode.EPHEMERAL  // 临时节点
        );

        if (result.equals("/kafka/controller")) {
            // 选举成功，成为 Controller
            becomeController();
        } else {
            // 选举失败，监听变化
            watchControllerChanges();
        }
    }
}
```

### 4.3 Leader 故障转移流程

```
Leader 故障转移完整流程：

时刻 T1: Broker-1 (Leader) 宕机
  │
  ▼
时刻 T2: Controller 检测到 Leader 不可用
  │
  ▼
时刻 T3: Controller 从 ISR 中选择新 Leader
  条件：ISR 中第一个存活的 Broker
  │
  ▼
时刻 T4: Controller 更新 ZK 元数据
  /kafka/brokers/topics/order-topic/partitions/0/state
  │
  ▼
时刻 T5: Controller 通知所有 Broker
  │
  ▼
时刻 T6: 客户端自动重连到新 Leader
  │
  ▼
时刻 T7: 故障转移完成
  总耗时：通常 < 10 秒
```

### 4.4 故障转移代码逻辑

```java
// Controller 处理 Broker 宕机
public class KafkaController {

    public void onBrokerFailure(int brokerId) {
        // 1. 获取该 Broker 上的所有分区
        List<TopicPartition> partitions = getPartitionsForBroker(brokerId);

        for (TopicPartition partition : partitions) {
            // 2. 检查该分区是否是 Leader
            if (partition.leader() == brokerId) {
                // 3. 触发 Leader 选举
                electNewLeader(partition);
            }

            // 4. 更新 ISR，移除宕机的 Broker
            partition.removeReplicaFromIsr(brokerId);
        }
    }

    private void electNewLeader(TopicPartition partition) {
        // 1. 获取 ISR
        List<Integer> isr = partition.isr();

        // 2. 选择新 Leader（ISR 中的第一个）
        int newLeader = isr.stream()
            .filter(b -> b != partition.leader())
            .findFirst()
            .orElseThrow(() -> new NoAvailableLeaderException());

        // 3. 更新 Leader
        partition.setLeaderAndIsr(newLeader, isr);

        // 4. 发送 LeaderAndIsr 请求给所有 Broker
        sendLeaderAndIsrRequest(partition, newLeader, isr);
    }
}
```

## 五、网络分区与脑裂

### 5.1 什么是脑裂

脑裂是指网络分区导致多个 Broker 都认为自己是 Controller：

```
脑裂场景：

┌─────────┐       ┌─────────┐       ┌─────────┐
│ Broker-1│       │ Broker-2│       │ Broker-3│
│  Controller│    │         │       │         │
└────┬────┘       └────┬────┘       └────┬────┘
     │                 │                 │
     ▼                 ▼                 ▼
  ZK 连接           ZK 连接           ZK 连接

网络分区后：
  Broker-1 认为自己是 Controller（ZK 节点存在）
  Broker-2 认为自己是 Controller（ZK 节点不存在后重新创建成功）
  Broker-3 认为自己是 Controller（ZK 节点不存在后重新创建成功）

三个 Broker 都执行 Controller 操作 → 脑裂！
```

### 5.2 Kafka 如何防止脑裂

```properties
# Controller 配置
controller.socket.timeout.ms=3000      # Controller 通信超时
controller.message.queue.size=100000   # Controller 消息队列大小
```

```java
// Controller 任期机制
public class ControllerContext {
    // 当前 Controller 的任期（类似 ZK 的 zxid）
    private long controllerEpoch;

    // 每次 Controller 变化，epoch + 1
    public void incrementControllerEpoch() {
        this.controllerEpoch++;
        // 持久化到 ZK
        zk.writeData("/kafka/controller_epoch",
            String.valueOf(controllerEpoch));
    }
}
```

### 5.3  unclean.leader.election

当 ISR 为空时，可以选择从非 ISR 副本选举：

```properties
# 允许从非 ISR 副本选举
unclean.leader.election.enable=true
# true：可用性优先，可能丢失数据
# false：一致性优先，可能不可用
```

## 六、性能与一致性权衡

### 6.1 acks 配置对比

| acks | 说明 | 可靠性 | 性能 |
|---|---|---|---|
| `0` | 不等待任何确认 | 最低 | 最高 |
| `1` | 只等 Leader 确认 | 中 | 高 |
| `-1` / `all` | 等待 ISR 所有副本确认 | 最高 | 低 |

### 6.2 刷盘策略

```properties
# 刷盘策略
log.flush.interval.messages=10000     # 每 10000 条消息强制刷盘
log.flush.interval.ms=1000           # 每 1 秒强制刷盘
log.flush.scheduler.interval.ms=100  # 检查间隔
```

### 6.3 配置建议

```properties
# 高可靠配置示例
acks=all                      # 等待所有 ISR 副本确认
min.insync.replicas=2         # 至少 2 个副本同步成功
retries=3                     # 重试 3 次
enable.idempotence=true       # 开启幂等
 unclean.leader.election.enable=false  # 禁止从非 ISR 选举
```

## 七、面试追问预判

### 追问一：Leader 宕机后，数据会不会丢失？

取决于配置：

- `acks=0`：可能丢失，Leader 宕机时消息未持久化
- `acks=1`：可能丢失，Follower 未同步时 Leader 宕机
- `acks=all`：不丢失，需要 ISR 中所有副本确认

### 追问二：ISR 为空时怎么办？

1. 如果 `unclean.leader.election.enable=true`，从非 ISR 副本选举（可能丢数据）
2. 如果 `unclean.leader.election.enable=false`，分区不可用，等待 ISR 恢复

### 追问三：故障转移需要多长时间？

通常 < 10 秒，包括：

| 阶段 | 时间 |
|---|---|
| 检测到宕机 | ~500ms（通过 ZooKeeper 通知） |
| Leader 选举 | < 100ms |
| 元数据更新 | < 100ms |
| 客户端重连 | ~9 秒（默认重试配置） |
