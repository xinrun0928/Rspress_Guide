# Kafka 架构：Broker、Topic、Partition、Replica

## 场景切入：理解架构才能用好 Kafka

很多人会用 Kafka，但不知道它的架构是怎么设计的。

当你配置 `replication.factor=3` 时，背后的数据流是怎样的？为什么一个 Broker 宕机后，系统还能继续运行？为什么分区数不能随便设置？

理解 Kafka 的架构，是用好 Kafka 的前提。

## 一、Kafka 整体架构

### 1.1 核心组件

```
┌─────────────────────────────────────────────────────────────────┐
│                       Kafka 集群架构                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │              ZooKeeper / KRaft 集群                     │   │

│  │  ┌─────────┐   ┌─────────┐   ┌─────────┐               │   │
│  │  │   ZK1   │   │   ZK2   │   │   ZK3   │               │   │
│  │  └─────────┘   └─────────┘   └─────────┘               │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│           │ 选举 Controller、分区分配、元数据管理                   │
│           ▼                                                     │

│  └─────────────────────────────────────────────────────────┘   │

```

### 1.2 组件职责

| 组件 | 职责 | 说明 |
|---|---|---|
| **ZooKeeper / KRaft** | 集群协调 | 管理 Broker 注册、分区选举、元数据存储 |
| **Controller** | 集群控制器 | 一个特殊的 Broker，负责分区 Leader 选举 |
| **Broker** | 消息节点 | 存储消息、处理读写请求 |
| **Topic** | 消息主题 | 消息的逻辑分类 |
| **Partition** | 分区 | 消息的物理存储单元 |
| **Replica** | 副本 | Partition 的备份，保证高可用 |

## 二、Broker

### 2.1 Broker 的角色

Broker 是 Kafka 集群中的工作节点，负责：

- **消息存储**：将消息持久化到磁盘
- **消息读写**：处理生产者和消费者的读写请求
- **副本管理**：维护 Partition 的副本
- **心跳上报**：向 ZooKeeper 汇报状态

```properties
# Broker 配置文件
broker.id=0                           # Broker 唯一 ID
listeners=PLAINTEXT://localhost:9092  # 监听地址
log.dirs=/var/kafka/data              # 数据存储目录
num.network.threads=3                 # 网络线程数
num.io.threads=8                      # IO 线程数
```

### 2.2 Broker 元数据

Broker 的元数据存储在 ZooKeeper 中：

```
/kafka/brokers/ids/0  → Broker-1 的信息
/kafka/brokers/ids/1  → Broker-2 的信息
/kafka/brokers/topics/ → Topic 信息
/kafka/controller      → Controller 信息
```

## 三、Topic 与 Partition

### 3.1 Topic 的概念

Topic 是消息的逻辑分类单位：

- Producer 向 Topic 发送消息
- Consumer 从 Topic 消费消息
- 一个 Topic 可以有多个 Partition

```
Topic 逻辑结构：

Topic: order-topic
┌─────────────────────────────────────────────────────────┐
│                                                         │
│   [消息1] [消息2] [消息3] [消息4] [消息5] ...            │
│                                                         │
│   Topic 是消息的逻辑容器，消息按时间顺序存储              │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 3.2 Partition 的概念

Partition 是消息的物理存储单元：

- 每个 Partition 对应一个目录
- Partition 内的消息有序
- 不同 Partition 间的消息无序

```
Topic: order-topic (3 个 Partition)

┌─────────────────────────────────────────────────────────┐
│  order-topic                                             │
│                                                         │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐     │
│  │ Partition 0  │  │ Partition 1  │  │ Partition 2  │     │
│  │             │  │             │  │             │     │
│  │ [msg 0-100] │  │ [msg 0-80]  │  │ [msg 0-120] │     │
│  │ offset: 0   │  │ offset: 0   │  │ offset: 0   │     │
│  │             │  │             │  │             │     │
│  └─────────────┘  └─────────────┘  └─────────────┘     │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 3.3 Partition 的存储结构

每个 Partition 对应一个目录：

```bash
# Partition 目录结构
/var/kafka/data/
└── order-topic-0/           # Topic 名-分区号
    ├── 00000000000000000000.log    # 数据文件
    ├── 00000000000000000000.index  # 偏移量索引
    ├── 00000000000000000000.timeindex  # 时间索引
    ├── 00000000000000000001.log    # 下一个段
    ├── 00000000000000000001.index
    └── 00000000000000000001.timeindex
```

## 四、Replica 副本机制

### 4.1 副本的角色

Kafka 通过副本机制保证高可用：

| 角色 | 说明 |
|---|---|
| **Leader** | 对外提供读写服务，所有读写都经过 Leader |
| **Follower** | 被动从 Leader 同步数据，不对外服务 |
| **ISR** | In-Sync Replicas，与 Leader 保持同步的副本集合 |

### 4.2 副本分布

```
副本分布示例（replication.factor=3）：

Topic: order-topic
  Partition 0: [Broker-1(Leader), Broker-2(Follower), Broker-3(Follower)]
  Partition 1: [Broker-2(Leader), Broker-3(Follower), Broker-1(Follower)]
  Partition 2: [Broker-3(Leader), Broker-1(Follower), Broker-2(Follower)]

每个 Partition 的副本分布在不同的 Broker 上，避免单点故障
```

### 4.3 分区副本分配策略

```properties
# 副本分配策略
# 1. 副本因子不能超过 Broker 数量
replication.factor=3

# 2. 第一个副本选择策略
first.replica.imbalance.policy=closest_replica
#    · closest_replica: 选择当前负载最低的 Broker
#    · any: 选择任意 Broker

# 3. 自动平衡
auto.leader.rebalance.enable=true
leader.imbalance.check.interval.seconds=300
```

## 五、架构设计原则

### 5.1 高可用设计

```
高可用架构：

┌─────────────────────────────────────────────────────────────┐
│                    容错机制                                  │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Broker 宕机                                                │
│       │                                                     │
│       ▼                                                     │
│  Controller 检测到 Broker 宕机                             │
│       │                                                     │
│       ▼                                                     │
│  从 ISR 中选举新的 Leader                                  │
│       │                                                     │
│       ▼                                                     │
│  其他 Follower 从新 Leader 同步数据                         │
│       │                                                     │
│       ▼                                                     │
│  对外继续提供服务                                            │
│                                                             │
│  故障恢复时间：通常 < 10 秒                                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 水平扩展设计

Kafka 支持在线扩展 Broker 和 Partition：

```bash
# 扩展 Broker：添加新的 Broker 到集群
# Kafka 自动将部分 Partition 迁移到新 Broker

# 扩展 Partition：为 Topic 增加分区数
kafka-topics.sh --alter --topic order-topic \
  --partitions 6 \
  --bootstrap-server localhost:9092
```

:::warning 注意
增加 Partition 数后，消息的路由会改变，可能影响消息的顺序性。
:::

## 六、面试追问预判

### 追问一：Broker 和 Partition 的关系是什么？

- 一个 Broker 可以有多个 Partition
- 一个 Partition 的多个副本分布在不同的 Broker 上
- Partition 数决定了最大的并行消费能力

### 追问二：Leader 挂了怎么办？

1. Controller 检测到 Leader 宕机
2. 从 ISR 中选举新的 Leader
3. 其他 Follower 从新 Leader 同步数据
4. 客户端自动重连到新 Leader

### 追问三：Partition 数可以减少吗？

不可以。Kafka 只支持增加 Partition 数，不支持减少。因为减少 Partition 会导致已存储的消息丢失或错乱。
