# Kafka 如何保证消息不丢失

## 场景切入：消息丢失的三大坑

消息队列最怕什么？丢消息。

你的系统告警：订单消息丢了，用户下单成功了，但库存没扣减。用户愤怒地打电话投诉。

Kafka 的消息丢失，可能发生在三个环节：**生产者发送、Broker 存储、消费者消费**。每个环节都有坑，每个环节都需要做好。

## 一、消息丢失的场景

```
消息传递三环节：

┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│   生产者      │───>│    Broker    │───>│   消费者      │
└──────────────┘    └──────────────┘    └──────────────┘
       │                  │                  │
       ▼                  ▼                  ▼
   发送丢失           存储丢失            消费丢失
```

### 1.1 生产者丢失

```
场景：网络抖动，消息未到达 Broker

Producer ──发送──> Leader
   │         │
   │         │ 网络超时
   │         ▼
   │    Broker 未收到
   │
   ▼
Producer 认为发送失败（但实际可能成功了）
```

### 1.2 Broker 丢失

```
场景：Leader 宕机，消息未同步到 Follower

Leader: [msg 0, 1, 2, 3, 4]  ← 最新消息在这里
Follower: [msg 0, 1, 2]      ← 只同步到这里

Leader 宕机，Follower 成为新 Leader
消息 3, 4 丢失！
```

### 1.3 消费者丢失

```
场景：自动提交 Offset，但消息未处理完成

Consumer ──拉取──> Broker
    │            │
    │<──返回──────┘
    │
    │ 处理消息
    │  │
    │  │ 处理失败
    │  ▼
    │
    │ 自动提交 Offset
    │
    ▼
Offset 已提交，但消息未处理 → 丢失！
```

## 二、生产者端保证不丢失

### 2.1 确认机制

```java
// 使用 acks=all，确保所有副本确认
props.put("acks", "all");

// 配合 min.insync.replicas
props.put("min.insync.replicas", "2");
```

| acks | 说明 | 可靠性 |
|---|---|---|
| `0` | 不等待确认 | 可能丢失 |
| `1` | Leader 确认 | 可能丢失 |
| `all` | 所有 ISR 副本确认 | 不易丢失 |

### 2.2 重试机制

```java
// 配置重试
props.put("retries", 3);               // 重试 3 次
props.put("retry.backoff.ms", 1000);   // 重试间隔 1 秒
```

### 2.3 幂等性

```java
// 开启幂等性，防止重复发送
props.put("enable.idempotence", true);
```

### 2.4 回调确认

```java
// 使用回调确认发送结果
producer.send(record, (metadata, exception) -> {
    if (exception != null) {
        // 发送失败，记录日志并重试
        log.error("发送失败，offset={}", metadata.offset(), exception);
        retry(record);
    } else {
        // 发送成功
        log.info("发送成功，offset={}", metadata.offset());
    }
});
```

### 2.5 完整配置

```java
// 生产者高可靠配置
Properties props = new Properties();
props.put("bootstrap.servers", "localhost:9092");

// 基本配置
props.put("key.serializer", "org.apache.kafka.common.serialization.StringSerializer");
props.put("value.serializer", "org.apache.kafka.common.serialization.StringSerializer");

// 高可靠配置
props.put("acks", "all");                      // 所有副本确认
props.put("retries", 3);                       // 重试 3 次
props.put("retry.backoff.ms", 1000);          // 重试间隔
props.put("enable.idempotence", true);        // 开启幂等

// 超时配置
props.put("request.timeout.ms", 30000);       // 请求超时
props.put("delivery.timeout.ms", 120000);      // 交付超时
```

## 三、Broker 端保证不丢失

### 3.1 副本因子

```properties
# Topic 配置：副本因子 >= 3
replication.factor=3
```

```
副本分布示例（replication.factor=3）：

Broker-1 (Leader)   Broker-2 (Follower)   Broker-3 (Follower)
      │                     │                     │
      ├───────同步──────────>│                     │
      │<───────ACK──────────┤                     │
      │                     ├───────同步──────────>│
      │                     │<───────ACK──────────┤

任何 1 个 Broker 宕机，都不会丢消息
```

### 3.2 ISR 配置

```properties
# 最小同步副本数
min.insync.replicas=2
```

| min.insync.replicas | 说明 |
|---|---|
| 1 | 只有 Leader 在 ISR 中，Leader 宕机不可用 |
| 2 | Leader + 至少 1 个 Follower 在 ISR 中 |
| 3 | 至少 2 个 Follower 在 ISR 中（需要 replication.factor >= 3） |

### 3.3 刷盘策略

```properties
# 刷盘配置
log.flush.interval.messages=10000     # 每 10000 条消息刷盘
log.flush.interval.ms=1000            # 每 1 秒刷盘
```

:::warning 注意
Kafka 依赖操作系统的刷盘机制（默认 5 秒），建议使用 RAID 或 SSD 提高可靠性。
:::

### 3.4 Broker 高可靠配置

```properties
# server.properties 高可靠配置

# 副本配置
replication.factor=3
min.insync.replicas=2

# 刷盘配置
log.flush.interval.messages=10000
log.flush.interval.ms=1000

# 禁止从非 ISR 选举
unclean.leader.election.enable=false

# 自动平衡
auto.leader.rebalance.enable=true
```

## 四、消费者端保证不丢失

### 4.1 手动提交 Offset

```java
// 手动提交 Offset
props.put("enable.auto.commit", "false");

KafkaConsumer<String, String> consumer = new KafkaConsumer<>(props);
consumer.subscribe(Arrays.asList("order-topic"));

while (true) {
    ConsumerRecords<String, String> records = consumer.poll(Duration.ofSeconds(1));

    for (ConsumerRecord<String, String> record : records) {
        try {
            // 处理消息
            processOrder(record.value());

            // 处理成功后提交 Offset
            consumer.commitSync();

        } catch (Exception e) {
            // 处理失败，不提交 Offset，等待重试
            log.error("处理失败，offset={}", record.offset(), e);
        }
    }
}
```

### 4.2 先处理后提交

```
处理逻辑：

┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  1. 拉取消息                                                │
│         │                                                   │
│         ▼                                                   │
│  2. 处理消息（必须成功）                                      │
│         │                                                   │
│         ├─成功─> 3. 提交 Offset                             │
│         │                                                   │
│         └─失败─> 等待重试（不提交 Offset）                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 4.3 消费者高可靠配置

```java
// 消费者高可靠配置
Properties props = new Properties();
props.put("bootstrap.servers", "localhost:9092");

// 基本配置
props.put("group.id", "order-consumer-group");
props.put("key.deserializer", "org.apache.kafka.common.serialization.StringDeserializer");
props.put("value.deserializer", "org.apache.kafka.common.serialization.StringDeserializer");

// 高可靠配置
props.put("enable.auto.commit", "false");         // 手动提交
props.put("auto.offset.reset", "earliest");      // 从头消费
props.put("session.timeout.ms", "30000");         // 会话超时
props.put("max.poll.records", "500");             // 每次拉取数量
```

## 五、端到端保证不丢失

### 5.1 全链路配置

```java
// 生产者配置
props.put("acks", "all");
props.put("retries", 3);
props.put("enable.idempotence", true);

// Broker 配置
// replication.factor=3
// min.insync.replicas=2

// 消费者配置
props.put("enable.auto.commit", "false");
props.put("auto.offset.reset", "earliest");
```

### 5.2 幂等消费

```java
// 幂等消费：即使重复消费也不会出问题
public void consume(OrderMessage message) {
    // 使用消息 ID 作为唯一键
    try {
        Order order = new Order();
        order.setMessageId(message.getId());  // 唯一索引
        order.setOrderId(message.getOrderId());
        orderMapper.insertSelective(order);
    } catch (DuplicateKeyException e) {
        // 唯一索引冲突，说明已处理过
        log.info("订单 {} 已存在，忽略重复消息", message.getOrderId());
    }
}
```

### 5.3 补偿机制

```java
// 定时检查补偿
@Scheduled(fixedDelay = 60000)
public void checkAndCompensate() {
    // 1. 查询已发送但未处理的消息
    List<Order> unsentOrders = orderService.findUnsentOrders();

    // 2. 重新发送
    for (Order order : unsentOrders) {
        producer.send(new ProducerRecord<>("order-topic",
            order.getId(), order.toString()));
    }
}
```

## 六、常见问题与解决方案

### 6.1 问题一：Leader 宕机，消息丢失

| 原因 | 解决方案 |
|---|---|
| `acks=0` 或 `acks=1` | 改为 `acks=all` |
| ISR 只有 1 个 | 配置 `min.insync.replicas >= 2` |
| 副本因子太小 | 配置 `replication.factor >= 3` |

### 6.2 问题二：消费者自动提交 Offset，消息丢失

| 原因 | 解决方案 |
|---|---|
| `enable.auto.commit=true` | 改为 `enable.auto.commit=false` |
| 处理失败也提交 | 先处理后提交 |

### 6.3 问题三：消息发送成功但未持久化

| 原因 | 解决方案 |
|---|---|
| Broker 宕机但未刷盘 | 使用 SSD，配置刷盘策略 |
| 副本未同步 | 配置 `unclean.leader.election.enable=false` |

## 七、面试追问预判

### 追问一：Kafka 如何保证消息不丢失？

需要三个环节都做好：

1. **生产者**：确认机制（`acks=all`）+ 重试 + 幂等
2. **Broker**：副本因子（`>= 3`）+ ISR（`>= 2`）
3. **消费者**：手动提交 Offset

### 追问二：`acks=all` 和 `min.insync.replicas` 有什么关系？

- `acks=all`：等待 ISR 中所有副本确认
- `min.insplicas=2`：ISR 中至少 2 个副本

配合使用：ISR 中至少 2 个副本确认，才算发送成功。

### 追问三：如果 ISR 只有 1 个，会发生什么？

如果 ISR 只有 1 个（只有 Leader），且 `min.insync.replicas=2`：

- 新消息无法写入（因为无法满足 2 个副本）
- Topic 变为只读状态
- 直到 ISR 中有足够的副本

### 追问四：如何检测消息是否丢失？

1. **监控 Lag**：监控 Consumer Lag，发现积压及时告警
2. **消息对账**：定期对账，检查消息数量是否一致
3. **发送回执**：消费者处理成功后发送回执，对比发送和消费数量
