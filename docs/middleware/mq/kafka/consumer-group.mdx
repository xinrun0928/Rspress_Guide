# Kafka 消费者组与 Rebalance

## 场景切入：消费者组的魔力

如果你有 100 万条消息，1 个消费者需要 10 小时处理完。但如果加 10 个消费者呢？

答案是 1 小时。这就是消费者组的威力——通过水平扩展消费能力，快速消化积压的消息。

但消费者组也有一些坑：分区分配不均、Rebalance 导致的消息乱序……理解消费者组的原理，才能用好它。

## 一、消费者组概念

### 1.1 什么是消费者组

消费者组（Consumer Group）是一组 Consumer 实例，共同消费一个 Topic：

```
消费者组示例：

Topic: order-topic (4 个 Partition)

消费者组 A: order-service
  Consumer 1 ──> Partition 0
  Consumer 2 ──> Partition 1
  Consumer 3 ──> Partition 2
  Consumer 4 ──> Partition 3

消费者组 B: analytics-service
  Consumer 1 ──> Partition 0, 1
  Consumer 2 ──> Partition 2, 3

每个消费者组独立消费，互不影响
```

### 1.2 消费者组的特点

| 特点 | 说明 |
|---|---|
| **负载均衡** | 组内 Consumer 均摊分区 |
| **独立消费** | 不同消费者组独立消费，互不影响 |
| **容错性** | Consumer 宕机，组内其他 Consumer 接管分区 |
| **组管理** | Kafka 自动管理 Consumer 的加入和离开 |

### 1.3 消费者组 ID

```java
// 配置消费者组 ID
Properties props = new Properties();
props.put("bootstrap.servers", "localhost:9092");
props.put("group.id", "order-consumer-group");  // 消费者组 ID
props.put("key.deserializer", "org.apache.kafka.common.serialization.StringDeserializer");
props.put("value.deserializer", "org.apache.kafka.common.serialization.StringDeserializer");

KafkaConsumer<String, String> consumer = new KafkaConsumer<>(props);
```

## 二、分区分配策略

### 2.1 分区与消费者的关系

```
分区分配规则：

Partition 数量 = P
Consumer 数量 = C

情况 1: C <= P
  每个 Consumer 至少分配 1 个分区
  多余的分区按顺序分配

情况 2: C > P
  某些 Consumer 分配不到分区，处于空闲状态
```

### 2.2 分区分配示例

```
分配示例（4 个 Partition，3 个 Consumer）：

分配策略：Range（默认）

Consumer 1: Partition 0, Partition 1
Consumer 2: Partition 2, Partition 3
Consumer 3: (空闲)

分配示例（4 个 Partition，5 个 Consumer）：

Consumer 1: Partition 0
Consumer 2: Partition 1
Consumer 3: Partition 2
Consumer 4: Partition 3
Consumer 5: (空闲)
```

### 2.3 查看分配关系

```bash
# 查看消费者组状态
kafka-consumer-groups.sh --bootstrap-server localhost:9092 \
  --group order-consumer-group \
  --describe

# 输出示例：

# order-consumer-group order-topic     1          10000            15000           5000    consumer-1
# order-consumer-group order-topic     2          10000            10000           0       consumer-2
# order-consumer-group order-topic     3          10000            10000           0       consumer-2
```

## 三、Offset 管理

### 3.1 Offset 的概念

Offset 是消息在 Partition 中的位置标识：

```
Offset 与消息位置：

Partition 0:
[msg 0] [msg 1] [msg 2] [msg 3] [msg 4] [msg 5] ...

   offset    offset    offset    offset    offset    offset
      0         1         2         3         4         5
```

### 3.2 Offset 的存储位置

| 存储方式 | 说明 | 配置参数 |
|---|---|---|
| **__consumer_offsets** | Kafka 内置 Topic | 默认 |
| **外部存储** | MySQL、Redis 等 | 自定义 |

```java
// Offset 存储配置
props.put("enable.auto.commit", "true");              // 自动提交
props.put("auto.commit.interval.ms", "5000");         // 提交间隔
```

### 3.3 Offset 提交策略

```java
// 自动提交（默认）
props.put("enable.auto.commit", "true");
props.put("auto.commit.interval.ms", "5000");

// 手动提交
props.put("enable.auto.commit", "false");

KafkaConsumer<String, String> consumer = new KafkaConsumer<>(props);
consumer.subscribe(Arrays.asList("order-topic"));

while (true) {
    ConsumerRecords<String, String> records = consumer.poll(Duration.ofSeconds(1));

    for (ConsumerRecord<String, String> record : records) {
        process(record);  // 处理消息
    }

    // 手动提交 Offset
    consumer.commitSync();
}
```

### 3.4 Offset 重置策略

```java
// Offset 重置策略
props.put("auto.offset.reset", "earliest");  // earliest 或 latest
```

| 配置值 | 说明 | 适用场景 |
|---|---|---|
| `earliest` | 从最早的消息开始消费 | 新消费者、追数据 |
| `latest` | 从最新的消息开始消费 | 生产环境、只关心新消息 |

## 四、Rebalance 机制

### 4.1 什么是 Rebalance

Rebalance 是消费者组成员变化时，重新分配分区的过程：

```
Rebalance 场景：

初始状态（3 个 Consumer）：
  Consumer 1 ──> Partition 0, 1
  Consumer 2 ──> Partition 2, 3
  Consumer 3 ──> Partition 4, 5

Consumer 3 宕机：
  Consumer 1 ──> Partition 0, 1, 4  ← 接管 Partition 4
  Consumer 2 ──> Partition 2, 3, 5  ← 接管 Partition 5

重新分配分区 → Rebalance
```

### 4.2 Rebalance 触发条件

| 触发条件 | 说明 |
|---|---|
| **Consumer 宕机** | Consumer 长时间未发送心跳 |
| **Consumer 加入** | 新 Consumer 加入组 |
| **Consumer 离开** | Consumer 主动离开组 |
| **Topic 分区变化** | Topic 分区数增加 |
| **Subscription 变化** | Consumer 订阅的 Topic 变化 |

### 4.3 Rebalance 流程

```
Rebalance 完整流程：

1. JoinGroup 阶段
   Consumer 向 Coordinator 发送 JoinGroup 请求
   │
   ▼
2. Leader 选举
   Coordinator 选择一个 Consumer 作为 Leader
   │
   ▼
3. SyncGroup 阶段
   Leader 制定分区分配方案
   │
   ▼
4. 分配方案同步
   Coordinator 将分配方案同步给所有 Consumer
   │
   ▼
5. Rebalance 完成
   所有 Consumer 开始按照新方案消费
```

### 4.4 Rebalance 的问题

```
Rebalance 的三大问题：

问题 1：Stop The World
  Rebalance 期间，所有 Consumer 停止消费
  ─────────────────────
  Consumer 1 ──[消费]──> Rebalance ──[停止]──> Rebalance ──[重新消费]──>
  Consumer 2 ──[消费]──> Rebalance ──[停止]──> Rebalance ──[重新消费]──>
                      ↑                         ↑
                    停止                       停止
```

| 问题 | 影响 | 解决方案 |
|---|---|---|
| **Stop The World** | 消费暂停 | 减少 Rebalance 频率 |
| **消息重复** | Rebalance 后可能重复消费 | 业务做好幂等 |
| **消息乱序** | 同一分区被不同 Consumer 处理 | 顺序消息场景避免 Rebalance |

## 五、Rebalance 优化

### 5.1 心跳参数调优

```java
// 心跳参数配置
props.put("heartbeat.interval.ms", 3000);      // 心跳间隔（默认 3 秒）
props.put("session.timeout.ms", 30000);        // 会话超时（默认 10 秒）
props.put("max.poll.interval.ms", 300000);     // 拉取间隔（默认 5 分钟）
```

```
心跳参数关系：

session.timeout.ms > heartbeat.interval.ms * 3

例如：
  heartbeat.interval.ms = 3000
  session.timeout.ms = 10000（满足 > 3 * 3000 = 9000）

如果 Consumer 在 session.timeout.ms 内未发送心跳，
Coordinator 认为 Consumer 宕机，触发 Rebalance
```

### 5.2 静态成员（Static Membership）

```java
// 静态成员配置
props.put("group.instance.id", "consumer-001");  // 静态 ID
```

```
静态成员 vs 动态成员：

动态成员：
  Consumer 重启后，重新加入组 → 触发 Rebalance

静态成员：
  Consumer 重启后，使用相同的 instance.id → 不触发 Rebalance
  如果在 session.timeout.ms 内重新加入，保持原有分区分配

适用场景：Consumer 是有状态的，重启后需要保持原有分区
```

### 5.3 协作式再平衡（Cooperative Rebalance）

```java
// 协作式再平衡配置
props.put("partition.assignment.strategy",
    "cooperative-sticky");  // Kafka 2.4+
```

```
协作式再平衡 vs 普通 Rebalance：

普通 Rebalance（Eager）：
  Consumer 1 ──[消费]──> Rebalance ──[全部停止]──> 新分配
  Consumer 2 ──[消费]──> Rebalance ──[全部停止]──> 新分配
                        ↑
                    全部停止

协作式再平衡（Cooperative）：
  Consumer 1 ──[消费]──> 增量重分配 ──[继续消费未分配]──> 新分配
  Consumer 2 ──[消费]──> 增量重分配 ──[继续消费未分配]──> 新分配
                        ↑
                    只停止受影响的分区
```

### 5.4 合理设置分区数

分区数不宜过多或过少：

```java
// 分区数规划
分区数 = max(消费者数量, 期望吞吐量 / 单分区吞吐)

// 分区数过多的问题：
// - Rebalance 时影响范围大
// - 元数据压力增加
// - 文件描述符消耗增加
```

```java

// 消费者组

// Offset 管理

// 防止 Rebalance
props.put("heartbeat.interval.ms", 10000);        // 心跳间隔
props.put("session.timeout.ms", 30000);             // 会话超时
props.put("max.poll.interval.ms", 300000);          // 拉取间隔
props.put("max.poll.records", 500);                // 每次拉取数量

// 分区分配策略
props.put("partition.assignment.strategy",
    "cooperative-sticky");                         // 协作式再平衡
```

### 6.2 消费代码示例

```java
// 完整消费代码

    public void consume() {
        KafkaConsumer<String, String> consumer = new KafkaConsumer<>(props);
        consumer.subscribe(Arrays.asList("order-topic"));

        try {
            while (true) {
                ConsumerRecords<String, String> records = consumer.poll(Duration.ofSeconds(1));

                for (ConsumerRecord<String, String> record : records) {
                    try {
                        processMessage(record);
                        // 业务处理成功
                    } catch (Exception e) {
                        // 业务处理失败，不提交 Offset，等待重试
                        handleFailure(record, e);
                    }
                }

                // 手动提交 Offset
                consumer.commitSync();
            }
        } finally {
            consumer.close();
        }
    }

            record.topic(), record.partition(), record.offset());

    }
}
```

## 七、面试追问预判

### 追问一：Consumer 数量能超过 Partition 数量吗？

可以，但多余的 Consumer 会空闲：

- `P = 4, C = 6`：只有 4 个 Consumer 分配到分区，2 个空闲
- 建议：`C <= P`，消费者数量不要超过分区数

### 追问二：Rebalance 和 Replica 有什么区别？

| 概念 | 说明 |
|---|---|
| **Rebalance** | 消费者组内分区分配的变化 |
| **Replica** | Partition 的副本，用于高可用 |

### 追问三：如何避免频繁 Rebalance？

1. 合理配置心跳参数
2. 使用静态成员
3. 使用协作式再平衡
4. 避免 Consumer 长时间 GC
5. 合理设置 `max.poll.interval.ms`
