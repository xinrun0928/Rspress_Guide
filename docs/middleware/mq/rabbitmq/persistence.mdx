# RabbitMQ 消息持久化：交换机、队列、消息

## 场景切入：Broker 重启了，消息还在吗？

如果 RabbitMQ 宕机了，未处理的消息会丢失吗？

这取决于你的持久化配置。RabbitMQ 的持久化分为三个层面：**交换机持久化、队列持久化、消息持久化**。只有三层都配置了，消息才不会丢失。

## 一、持久化的三个层面

### 1.1 三层持久化

| 层面 | 说明 | 配置 |
|---|---|---|
| **交换机持久化** | 交换机配置持久化 | `durable=true` |
| **队列持久化** | 队列配置持久化 | `durable=true` |
| **消息持久化** | 消息配置持久化 | `deliveryMode=2` |

```
三层持久化关系：

┌─────────────────────────────────────────────────────────────┐
│                      消息持久化                              │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ Exchange 持久化 (durable=true)                       │   │
│  │                                                     │   │
│  │  ┌───────────────────────────────────────────────┐ │   │
│  │  │ Queue 持久化 (durable=true)                   │ │   │
│  │  │                                               │ │   │
│  │  │  ┌─────────────────────────────────────────┐ │ │   │
│  │  │  │ Message 持久化 (deliveryMode=2)         │ │ │   │
│  │  │  │                                         │ │ │   │
│  │  │  │  [msg1] [msg2] [msg3]                 │ │ │   │
│  │  │  │                                         │ │ │   │
│  │  │  └─────────────────────────────────────────┘ │ │   │
│  │  │                                               │ │   │
│  │  └───────────────────────────────────────────────┘ │   │
│  │                                                     │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  三层都持久化，消息才不会丢失                                │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 持久化配置代码

```java
// 1. 交换机持久化
channel.exchangeDeclare(
    "order-exchange",  // 交换机名
    "direct",          // 交换机类型
    true               // durable=true，持久化
);

// 2. 队列持久化
channel.queueDeclare(
    "order-queue",      // 队列名
    true,              // durable=true，持久化
    false,             // exclusive=false，非独占
    false,             // autoDelete=false，非自动删除
    null               // arguments=null
);

// 3. 消息持久化
channel.basicPublish(
    "order-exchange",
    "order.created",
    MessageProperties.PERSISTENT_TEXT_PLAIN,  // PERSISTENT = deliveryMode=2
    "order data".getBytes()
);

// 或者手动设置
AMQP.BasicProperties properties = new AMQP.BasicProperties.Builder()
    .deliveryMode(2)  // 持久化
    .build();
channel.basicPublish("order-exchange", "order.created", properties, body);
```

## 二、持久化原理

### 2.1 消息存储

RabbitMQ 使用 **Mnesia** 数据库存储元数据和消息：

```
消息存储架构：

┌─────────────────────────────────────────────────────────────┐
│                    RabbitMQ Broker                           │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                   Mnesia 数据库                       │   │
│  │                                                     │   │
│  │  ┌──────────────┐  ┌──────────────┐              │   │
│  │  │  Exchange 表  │  │  Binding 表  │              │   │
│  │  └──────────────┘  └──────────────┘              │   │
│  │  ┌──────────────┐  ┌──────────────┐              │   │
│  │  │  Queue 表    │  │  Message 表  │              │   │
│  │  └──────────────┘  └──────────────┘              │   │
│  │                                                     │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 刷盘策略

RabbitMQ 的消息刷盘策略：

| 策略 | 说明 | 可靠性 | 性能 |
|---|---|---|---|
| **异步刷盘** | 批量写入，定期刷盘 | 高 | 高 |
| **同步刷盘** | 每条消息强制刷盘 | 最高 | 低 |

```properties
# RabbitMQ 配置
# 默认异步刷盘
queue_master_locator=min-masters

# 可配置同步刷盘（性能影响大）
# disk_write_delay = true
```

## 三、持久化与性能

### 3.1 性能影响

| 配置 | 性能影响 | 说明 |
|---|---|---|
| 非持久化 | 最高 | 消息存储在内存 |
| 队列持久化 | 高 | 元数据持久化 |
| 消息持久化 | 中 | 消息写入磁盘 |
| 同步刷盘 | 低 | 每条消息强制刷盘 |

### 3.2 最佳实践

```java
// 高可靠配置
// 1. 交换机持久化
channel.exchangeDeclare("order-exchange", "direct", true);

// 2. 队列持久化
channel.queueDeclare("order-queue", true, false, false, null);

// 3. 消息持久化
channel.basicPublish(
    "order-exchange",
    "order.created",
    MessageProperties.PERSISTENT_TEXT_PLAIN,  // 消息持久化
    "order data".getBytes()
);

// 4. 开启 Confirm
channel.confirmSelect();
if (!channel.waitForConfirms()) {
    // 发送失败，重试
}
```

## 四、持久化与高可用

### 4.1 持久化不能替代高可用

持久化只能防止 Broker 重启后的数据丢失，但不能防止 Broker 宕机。

要实现高可用，需要：

1. **持久化**：防止重启丢失
2. **集群**：防止单点故障
3. **镜像队列**：队列主从复制

### 4.2 持久化配置对比

| 场景 | 交换机 | 队列 | 消息 | 推荐 |
|---|---|---|---|---|
| **开发测试** | ❌ | ❌ | ❌ | 不需要持久化 |
| **普通生产** | ✅ | ✅ | ✅ | 建议持久化 |
| **金融级** | ✅ | ✅ | ✅ + 镜像队列 | 必须持久化 + HA |

## 五、面试追问预判

### 追问一：如何保证消息不丢失？

需要三个层面都配置持久化：

1. **交换机持久化**：`durable=true`
2. **队列持久化**：`durable=true`
3. **消息持久化**：`deliveryMode=2`

### 追问二：持久化会影响性能吗？

会。持久化会降低性能：

- 非持久化：消息存储在内存，最快
- 消息持久化：异步刷盘，性能影响中等
- 同步刷盘：每条消息强制刷盘，性能影响大

### 追问三：持久化能替代高可用吗？

不能。持久化只能防止重启丢失，要实现高可用还需要：

- 集群部署
- 镜像队列
- 负载均衡
