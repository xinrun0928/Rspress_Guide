# RabbitMQ 消费端确认：ACK 与 NACK

## 场景切入：消息消费了，真的处理成功了吗？

Consumer 收到消息后，你需要告诉 Broker：「消息处理成功了，可以删除了」或者「处理失败了，需要重试」。

ACK 确认机制就是用来做这件事的。

## 一、ACK 确认机制

### 1.1 基本概念

| 状态 | 说明 |
|---|---|
| **ACK** | 确认消息处理成功，删除消息 |
| **NACK** | 拒绝消息，可选是否重入队列 |
| **Reject** | 拒绝消息，不重入队列 |

```java
// 消费消息
channel.basicConsume("order-queue", false, new DefaultConsumer(channel) {
    @Override
    public void handleDelivery(String consumerTag,
                             Envelope envelope,
                             AMQP.BasicProperties properties,
                             byte[] body) {
        try {
            // 1. 处理消息
            String message = new String(body);
            processOrder(message);

            // 2. 处理成功，确认消息
            channel.basicAck(envelope.getDeliveryTag(), false);

        } catch (Exception e) {
            // 3. 处理失败，拒绝消息
            channel.basicNack(envelope.getDeliveryTag(), false, true);
        }
    }
});
```

### 1.2 自动 ACK vs 手动 ACK

| 模式 | 说明 | 适用场景 |
|---|---|---|
| **自动 ACK** | 消息投递即确认 | 消息处理简单、允许丢失 |
| **手动 ACK** | 处理完成后手动确认 | 需要保证消息不丢失 |

```java
// 手动 ACK（推荐）
channel.basicConsume("order-queue", false, consumer);

// 自动 ACK
channel.basicConsume("order-queue", true, consumer);
```

### 1.3 basicAck

```java
// 确认单个消息
channel.basicAck(deliveryTag, false);

// 确认多个消息（deliveryTag 之前的所有消息）
channel.basicAck(deliveryTag, true);

// 参数说明：
// deliveryTag: 消息的唯一标识
// multiple: 是否批量确认
```

### 1.4 basicNack

```java
// 拒绝消息
channel.basicNack(deliveryTag, multiple, requeue);

// 参数说明：
// deliveryTag: 消息的唯一标识
// multiple: 是否批量拒绝
// requeue: 是否重新入队

// 示例：拒绝并重试
channel.basicNack(deliveryTag, false, true);

// 示例：拒绝不重试（进入死信队列）
channel.basicNack(deliveryTag, false, false);
```

### 1.5 basicReject

```java
// 拒绝单个消息（不重入队列）
channel.basicReject(deliveryTag, false);

// 等价于
channel.basicNack(deliveryTag, false, false);
```

## 二、手动 ACK 的最佳实践

### 2.1 正确流程

```
手动 ACK 正确流程：

1. 接收消息
2. 处理业务逻辑
3. 业务成功 → basicAck
4. 业务失败 → basicNack (requeue=true) 重试
                或 basicNack (requeue=false) 死信队列
```

```java
// 推荐的手动 ACK 模式
channel.basicConsume("order-queue", false, new DefaultConsumer(channel) {
    @Override
    public void handleDelivery(String consumerTag,
                             Envelope envelope,
                             AMQP.BasicProperties properties,
                             byte[] body) {
        long deliveryTag = envelope.getDeliveryTag();

        try {
            // 处理消息
            processOrder(new String(body));

            // 成功：确认消息
            channel.basicAck(deliveryTag, false);

        } catch (Exception e) {
            // 判断是否可以重试
            if (canRetry(e)) {
                // 可重试：重新入队
                channel.basicNack(deliveryTag, false, true);
            } else {
                // 不可重试：拒绝不重试（进入死信队列）
                channel.basicNack(deliveryTag, false, false);
            }
        }
    }
});
```

### 2.2 防止消息丢失

```
消息丢失场景：

1. Consumer 收到消息
2. 自动 ACK（消息已从队列删除）
3. Consumer 处理失败
4. 消息丢失！

防止方法：
1. 使用手动 ACK
2. 处理成功后再 ACK
3. 失败时 NACK 或 Reject
```

## 三、预取机制

### 3.1 QoS 预取

```java
// 设置预取数量
channel.basicQos(prefetchCount);

// 示例：每次最多预取 10 条消息
channel.basicQos(10);
```

```
预取机制示意图：

Broker                    Consumer
    │                        │
    │ ──msg1───────────────>│ (1)
    │ ──msg2───────────────>│ (2)
    │ ──msg3───────────────>│ (3)
    │ ──msg4───────────────>│ (4)
    │ ──msg5───────────────>│ (5)
    │ ──msg6───────────────>│ (6)
    │ ──msg7───────────────>│ (7)
    │ ──msg8───────────────>│ (8)
    │ ──msg9───────────────>│ (9)
    │ ──msg10──────────────>│ (10)
    │ ──msg11──────────────X│ (达到预取上限)
    │                        │
    │<── ACK(msg1-5) ─────│ (确认前 5 条)
    │ ──msg11──────────────>│ (继续接收)
```

### 3.2 预取与吞吐

| 预取值 | 特点 | 适用场景 |
|---|---|---|
| **低（1-10）** | 延迟低，吞吐量低 | 实时处理 |
| **中（50-100）** | 平衡延迟和吞吐 | 普通场景 |
| **高（500+）** | 吞吐高，延迟高 | 批处理 |

## 四、面试追问预判

### 追问一：ACK、NACK、Reject 有什么区别？

| 方法 | 说明 |
|---|---|
| **ACK** | 确认消息处理成功，删除消息 |
| **NACK** | 拒绝消息，可选是否重入队列 |
| **Reject** | 拒绝消息，不重入队列 |

### 追问二：自动 ACK 和手动 ACK 有什么区别？

| 模式 | 说明 | 风险 |
|---|---|---|
| **自动 ACK** | 消息投递即确认 | 可能丢失消息 |
| **手动 ACK** | 处理完成后确认 | 更安全，但需要正确处理 |

### 追问三：消息处理失败怎么办？

1. **可重试错误**：NACK 并重新入队（`requeue=true`）
2. **不可重试错误**：NACK 不重入队（`requeue=false`），进入死信队列
3. **重试次数限制**：超过次数后进入死信队列
