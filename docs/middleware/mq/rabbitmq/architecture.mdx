# RabbitMQ 架构：Producer → Exchange → Binding → Queue → Consumer

## 场景切入：从一条消息看 RabbitMQ 的架构

当你调用 `channel.basicPublish()` 发送一条消息时，背后发生了什么？

消息从 Producer 出发，经过 Exchange 的路由，通过 Binding 的规则，到达 Queue，最终被 Consumer 消费。这条消息的旅程，就是 RabbitMQ 架构的核心。

## 一、整体架构

### 1.1 组件关系

```
RabbitMQ 整体架构：
┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│   Producer                                                       │
│       │                                                         │
│       │ basicPublish(exchange, routingKey, message)             │
│       ▼                                                         │
│   Exchange                                                      │
│       │                                                         │
│       │ 根据 Binding 规则路由                                     │
│       │                                                         │
│       ├──> Binding ──"order.created"──> Queue ──> Consumer A   │
│       │                                                         │
│       └──> Binding ──"order.*"──────> Queue ──> Consumer B       │
│                                                                 │
│   Broker                                                        │
│       │                                                         │
│       │ 元数据存储（Exchange、Queue、Binding）                   │
│       │                                                         │
│       ├──> Virtual Host: /                                     │
│       │   ├──> Exchange: order-exchange                         │
│       │   ├──> Queue: order-queue                               │
│       │   └──> Binding: order-queue <--> order-exchange         │
│       │                                                         │
└─────────────────────────────────────────────────────────────────┘
```

### 1.2 核心组件

| 组件 | 职责 |
|---|---|
| **Producer** | 消息生产者，发送消息到 Exchange |
| **Exchange** | 交换机，根据规则路由消息到队列 |
| **Binding** | 绑定关系，定义 Exchange 和 Queue 的路由规则 |
| **Queue** | 队列，存储消息，等待消费者消费 |
| **Consumer** | 消费者，从队列获取消息并处理 |
| **Broker** | RabbitMQ 服务器实例 |

## 二、消息流转详解

### 2.1 消息发送流程

```java
// 消息发送完整流程
// 1. 建立连接
Connection connection = connectionFactory.newConnection();
// 2. 创建 Channel
Channel channel = connection.createChannel();
// 3. 声明交换机
channel.exchangeDeclare("order-exchange", "direct", true, false, null);
// 4. 声明队列
channel.queueDeclare("order-queue", true, false, false, null);
// 5. 绑定交换机和队列
channel.queueBind("order-queue", "order-exchange", "order.created");
// 6. 发送消息
channel.basicPublish(
    "order-exchange",      // 交换机名
    "order.created",       // 路由键
    MessageProperties.PERSISTENT_TEXT_PLAIN,  // 消息属性
    "订单数据".getBytes()   // 消息体
);
// 7. 关闭资源
channel.close();
connection.close();
```

### 2.2 消息消费流程

```java
// 消息消费完整流程
// 1. 建立连接
Connection connection = connectionFactory.newConnection();
Channel channel = connection.createChannel();
// 2. 设置 QoS（服务质量）
channel.basicQos(10);  // 每次预取 10 条消息
// 3. 注册消费者
channel.basicConsume(
    "order-queue",     // 队列名
    false,            // 不自动确认
    "consumer-tag",   // 消费者标签
    new DefaultConsumer(channel) {
        @Override
        public void handleDelivery(String consumerTag,
                                 Envelope envelope,
                                 AMQP.BasicProperties properties,
                                 byte[] body) {
            try {
                String message = new String(body);
                System.out.println("收到消息：" + message);
                // 4. 处理完成后确认
                channel.basicAck(envelope.getDeliveryTag(), false);
            } catch (Exception e) {
                // 5. 处理失败，拒绝消息
                channel.basicNack(envelope.getDeliveryTag(), false, true);
            }
        }
    }
);
```

## 三、消息持久化机制

### 3.1 三级持久化

RabbitMQ 的消息持久化分为三个层面：

| 层面 | 说明 | 配置 |
|---|---|---|
| **Exchange 持久化** | 交换机持久化 | `durable=true` |
| **Queue 持久化** | 队列持久化 | `durable=true` |
| **Message 持久化** | 消息持久化 | `deliveryMode=2` |

```java
// 三级持久化配置
// 1. 交换机持久化
channel.exchangeDeclare("order-exchange", "direct", true);
// durable=true 表示交换机持久化

// 2. 队列持久化
channel.queueDeclare("order-queue", true, false, false, null);
// durable=true 表示队列持久化

// 3. 消息持久化
channel.basicPublish(
    "order-exchange",
    "order.created",
    MessageProperties.PERSISTENT_TEXT_PLAIN,  // 消息持久化
    "order data".getBytes()
);
// deliveryMode=2 表示消息持久化
```

## 五、消费者模型

### 5.1 拉取模式和推送模式

| 模式 | 说明 | API |
|---|---|---|
| **拉取模式（Pull）** | Consumer 主动拉取消息 | `basicGet` |
| **推送模式（Push）** | Broker 主动推送消息 | `basicConsume` |

```java
// 拉取模式
while (true) {
    GetResponse response = channel.basicGet("order-queue", false);
    if (response != null) {
        byte[] body = response.getBody();
        // 处理消息
        channel.basicAck(response.getEnvelope().getDeliveryTag(), false);
    }
}
```

### 5.2 QoS 预取

QoS（Quality of Service）控制 Consumer 的预取数量：

```java
// 设置 QoS
channel.basicQos(prefetchSize, prefetchCount, global);

// 参数说明：
// prefetchSize: 预取字节数（0 表示不限制）
// prefetchCount: 预取消息数
// global: 是否全局生效

// 示例：每次最多预取 10 条消息
channel.basicQos(0, 10, false);
```

```
QoS 预取示意图：

Broker                    Consumer
    │                        │
    │ ──msg1───────────────>│
    │ ──msg2───────────────>│
    │ ──msg3───────────────>│
    │ ──msg4───────────────>│
    │ ──msg5───────────────>│
    │ ──msg6───────────────>│
    │ ──msg7───────────────>│
    │ ──msg8───────────────>│
    │ ──msg9───────────────>│

    │<─── ACK ──────────────│ (确认 msg1)
    │ ──msg11──────────────>│ (继续接收)
```

## 六、面试追问预判

### 追问一：消息持久化是如何工作的？

RabbitMQ 的持久化分为三个层面：

1. **Exchange 持久化**：`durable=true`
2. **Queue 持久化**：`durable=true`
3. **Message 持久化**：`deliveryMode=2`

只有三层都配置了持久化，消息才不会丢失。

### 追问二：拉取模式和推送模式有什么区别？

| 模式 | 说明 | 适用场景 |
|---|---|---|
| **拉取模式** | 主动拉取，精确控制 | 批处理、低流量场景 |
| **推送模式** | 被动接收，自动推送 | 高流量、实时处理 |

### 追问三：QoS 的作用是什么？

QoS 控制 Consumer 的预取数量，避免：

- Consumer 处理不过来，消息堆积
- Consumer 内存溢出
- 消息处理不均衡

### 追问四：多个 Consumer 如何分配消息？

多个 Consumer 订阅同一个 Queue 时，消息会被轮询分发：

- Consumer A 收到 msg1, msg3, msg5
- Consumer B 收到 msg2, msg4, msg6
