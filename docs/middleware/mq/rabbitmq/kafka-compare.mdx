# RabbitMQ 与 Kafka 对比与选型

## 场景切入：选型不是选最好的，而是选最合适的

面试官问：「你们项目用的 RabbitMQ，为什么不用 Kafka？」

这个问题看似简单，但真正考察的是你对两种 MQ 的理解深度。选型不是选最好的，而是根据业务场景做出权衡。

## 一、核心差异对比

### 1.1 设计理念

| MQ | 设计理念 | 核心优势 |
|---|---|---|
| **Kafka** | 日志优先 | 高吞吐、海量数据 |
| **RabbitMQ** | 消息路由 | 灵活路由、功能丰富 |

### 1.2 架构对比

```
Kafka 架构：
┌─────────────────────────────────────────────────────────────┐
│  ZooKeeper                                                    │
└─────────────────────────────────────────────────────────────┘
                              │
         ┌────────────────────┼────────────────────┐
         ▼                    ▼                    ▼
   ┌─────────┐          ┌─────────┐          ┌─────────┐
   │ Broker  │          │ Broker  │          │ Broker  │
   │         │          │         │          │         │
   │ Topic-A │          │ Topic-A │          │ Topic-A │
   │ Part-0  │          │ Part-1  │          │ Part-2  │
   │(Leader) │          │(Follower│          │(Follower│
   └─────────┘          └─────────┘          └─────────┘

RabbitMQ 架构：
┌─────────────────────────────────────────────────────────────┐
│  Exchange ───> Binding ───> Queue ───> Consumer             │
│                                                             │
│  交换机根据路由规则将消息分发到队列                          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.3 核心指标对比

| 指标 | Kafka | RabbitMQ |
|---|---|---|
| **吞吐量** | 100 万+ QPS | 1-5 万 QPS |
| **延迟** | 毫秒级 | 微秒级（最低） |
| **消息持久化** | 强 | 中 |
| **消息回溯** | 支持 | 不支持 |
| **优先级队列** | 不支持 | 支持 |
| **延迟消息** | 需插件 | 原生支持插件 |
| **事务消息** | 有限支持 | 不支持 |

## 三、功能对比

### 3.1 核心功能对比

| 功能 | Kafka | RabbitMQ |
|---|---|---|
| **消息顺序** | 分区内有序 | 单队列有序 |
| **优先级队列** | ❌ | ✅ |
| **延迟消息** | ❌ | ✅ 插件 |
| **事务消息** | ✅ 有限 | ❌ |
| **消息回溯** | ✅ | ❌ |
| **死信队列** | ❌ 需手动 | ✅ 原生 |
| **多协议** | 仅 Kafka 协议 | AMQP、MQTT、STOMP |

### 3.2 路由能力对比

| 维度 | Kafka | RabbitMQ |
|---|---|---|
| **路由方式** | 分区（基于 Key 哈希） | Exchange + Binding |
| **灵活性** | 低 | 高 |
| **复杂度** | 低 | 中 |

## 四、适用场景对比

### 4.1 Kafka 适用场景

| 场景 | 原因 |
|---|---|
| **日志收集** | 高吞吐、海量数据 |
| **流处理** | 与 Flink、Spark 集成 |
| **事件溯源** | 消息回溯 |
| **大数据管道** | 高吞吐、生态完善 |

### 4.2 RabbitMQ 适用场景

| 场景 | 原因 |
|---|---|
| **复杂路由** | Exchange 灵活路由 |
| **任务队列** | 功能丰富、管理界面 |
| **异步任务** | 延迟消息、死信队列 |
| **微服务解耦** | 灵活路由、多协议 |

## 六、面试追问预判

### 追问一：Kafka 和 RabbitMQ 有什么区别？

| 维度 | Kafka | RabbitMQ |
|---|---|---|
| **设计理念** | 日志优先 | 消息路由 |
| **吞吐量** | 100 万+ QPS | 1-5 万 QPS |
| **路由** | 基于分区 | 基于 Exchange |
| **消息回溯** | 支持 | 不支持 |
| **延迟** | 毫秒级 | 微秒级 |

### 追问二：如何选择 Kafka 和 RabbitMQ？

| 场景 | 选择 |
|---|---|
| 日志收集、流处理 | Kafka |
| 复杂路由、企业级 | RabbitMQ |
| 高吞吐优先 | Kafka |
| 功能丰富优先 | RabbitMQ |

### 追问三：RabbitMQ 能做 Kafka 的事吗？

能，但不适合：

- RabbitMQ 吞吐量远低于 Kafka
- RabbitMQ 不支持消息回溯
- RabbitMQ 运维成本较高

### 追问四：Kafka 能做 RabbitMQ 的事吗？

能，但不适合：

- Kafka 路由能力弱
- Kafka 功能相对简单
- Kafka 延迟较高
