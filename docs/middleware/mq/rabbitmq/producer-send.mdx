# RabbitMQ 消息发送模式：同步、异步、异步回调

## 场景切入：同步发送阻塞了你的系统？

你的系统在高峰时段响应时间突然飙升到 5 秒以上，但数据库和应用服务器都正常。问题出在哪？

很可能就是你的消息发送方式。RabbitMQ 提供了三种发送模式——同步发送、异步发送、异步回调——每种模式的性能差异可达数十倍。选错模式，高并发下就是一场灾难。

## 一、三种发送模式概述

| 模式 | 特点 | 适用场景 | 性能 |
|---|---|---|---|
| **同步发送** | 阻塞等待结果 | 简单场景、低并发 | 低 |
| **异步发送** | 非阻塞，立即返回 Future | 需要异步化 | 中 |
| **异步回调** | 注册监听器，结果通过回调通知 | 高并发、高可靠 | 高 |

## 二、同步发送

同步发送是最直观的模式，调用 `basicPublish` 后阻塞等待 Broker 确认结果。

```java
// 同步发送
try {
    // basicPublish 会阻塞，直到收到 Broker 响应
    channel.basicPublish(
        exchange,
        routingKey,
        MessageProperties.PERSISTENT_TEXT_PLAIN,
        message.getBytes()
    );
    System.out.println("消息发送成功");
} catch (Exception e) {
    System.err.println("消息发送失败: " + e.getMessage());
}
```

### 同步发送的缺陷

```
同步发送时序图：

Producer                 Broker                 Consumer
   |                      |                      |
   |──TCP 发送请求────────>│                      |
   |                      │                      |
   │<──等待处理...────────│                      |
   │                      │                      |
   │<──返回结果────────────│                      |
   |                      │                      |
   | 继续执行业务逻辑       |                      |
   |                      |                      |
   |                      │──消息投递────────────>│
   |                      |                      |
```

同步发送的问题：
- 每次发送都阻塞等待网络往返时延（RTT）
- 吞吐量为：`1000ms / (RTT + Broker处理时间)`
- 网络 RTT 通常 1-5ms，高并发下大量线程阻塞在等待上

### 适用场景

同步发送适合以下场景：
- 发送频率极低（每秒几条消息）
- 需要立即知道发送结果
- 消息丢失是不可接受的业务错误

```java
// 典型同步发送场景：注册后发确认邮件
public void register(User user) {
    userRepository.save(user);
    emailService.sendWelcomeEmail(user.getEmail());
    // 发送消息通知下游系统——同步发送即可
    rabbitTemplate.convertAndSend("user-exchange", "user.created", user);
}
```

## 三、异步发送（Future 模式）

异步发送通过 `Channel.basicPublish` 配合 `Channel.waitForConfirmsOrDie()` 实现非阻塞等待。消息发送后立即返回一个可查询结果的 Future 对象。

```java
// 异步发送（Future 模式）
public void asyncSendWithFuture(String routingKey, String message) throws Exception {
    // 1. 开启发送确认模式
    channel.confirmSelect();

    // 2. 发送消息（立即返回，不阻塞）
    channel.basicPublish(exchange, routingKey, null, message.getBytes());

    // 3. 通过 Future 查询结果（非阻塞轮询）
    boolean confirmed = channel.waitForConfirms();
    if (confirmed) {
        System.out.println("消息发送成功");
    } else {
        System.out.println("消息发送失败，需要重试");
    }
}
```

实际上，更好的做法是使用 `CompletableFuture` 配合自定义的异步确认：

```java
CompletableFuture<Boolean> sendAsync(String routingKey, String message) {
    CompletableFuture<Boolean> future = new CompletableFuture<>();

    // 添加确认监听器
    channel.addConfirmListener(
        (deliveryTag, multiple) -> future.complete(true),   // 发送成功
        (deliveryTag, multiple) -> future.complete(false)   // 发送失败
    );

    // 发送消息
    try {
        channel.basicPublish(exchange, routingKey, null, message.getBytes());
    } catch (Exception e) {
        future.completeExceptionally(e);
    }

    return future;
}

// 使用方式
CompletableFuture<Boolean> result = sendAsync("order.created", "test");
result.thenAccept(success -> {
    if (!success) {
        // 触发重试
        retrySend(routingKey, message);
    }
});
```

### 异步发送的改进

```
异步发送时序图：

Producer                 Broker
   |                      |
   │──TCP 发送请求────────>│
   │                      │
   │<──立即返回────────────│
   |                      │
   │ 继续执行业务逻辑        │
   |                      │
   │──异步查询结果─────────>│
   │<──返回确认─────────────│
```

异步发送的优势：
- 消息发送不阻塞业务线程
- 可以同时发送多条消息后再统一等待确认
- 吞吐量显著提升

## 四、异步回调模式（推荐）

异步回调模式是生产环境中推荐的方式。通过注册确认监听器（ConfirmListener），Broker 处理完成后主动回调通知生产者，无需任何阻塞等待。

### 4.1 基础异步回调

```java
// 1. 开启发送确认
channel.confirmSelect();

// 2. 创建回调记录表（用于关联消息和确认结果）
ConcurrentHashMap<String, MessageRecord> pendingMessages = new ConcurrentHashMap<>();

// 3. 添加确认监听器
channel.addConfirmListener(
    // 成功回调：Broker 确认收到消息
    (deliveryTag, multiple) -> {
        if (multiple) {
            // multiple=true：批量确认，从 1 到 deliveryTag 的所有消息都被确认
            pendingMessages.entrySet().removeIf(entry -> entry.getValue().tag <= deliveryTag);
        } else {
            // multiple=false：单条确认
            pendingMessages.entrySet().removeIf(entry -> entry.getValue().tag == deliveryTag);
        }
        System.out.println("消息确认成功: " + deliveryTag);
    },
    // 失败回调：消息被拒绝（NACK）
    (deliveryTag, multiple) -> {
        // 触发重发逻辑
        System.err.println("消息发送失败: " + deliveryTag);
        pendingMessages.values().stream()
            .filter(r -> r.tag == deliveryTag)
            .findFirst()
            .ifPresent(record -> retrySend(record.routingKey, record.body));
    }
);

// 4. 发送消息（无需等待，立即返回）
String correlationId = UUID.randomUUID().toString();
pendingMessages.put(correlationId, new MessageRecord(correlationId, routingKey, message));

channel.basicPublish(
    exchange,
    routingKey,
    new AMQP.BasicProperties.Builder()
        .correlationId(correlationId)
        .deliveryMode(2)
        .build(),
    message.getBytes()
);

// 业务继续执行，无需等待
System.out.println("消息已投递，异步确认中...");
```

### 4.2 完整的异步回调封装

```java
public class AsyncRabbitTemplate {

    private final Channel channel;
    private final ConcurrentHashMap<Long, PendingMessage> pending = new ConcurrentHashMap<>();
    private final ScheduledExecutorService scheduler = Executors.newScheduledThreadPool(2);

    public AsyncRabbitTemplate(Channel channel) {
        this.channel = channel;
        this.channel.confirmSelect();
        setupConfirmListener();
    }

    private void setupConfirmListener() {
        channel.addConfirmListener(
            (deliveryTag, multiple) -> {
                if (multiple) {
                    pending.entrySet().removeIf(e -> e.getKey() <= deliveryTag);
                } else {
                    pending.remove(deliveryTag);
                }
            },
            (deliveryTag, multiple) -> {
                // 失败的消息加入重试队列
                PendingMessage msg = pending.remove(deliveryTag);
                if (msg != null) {
                    scheduleRetry(msg, 1);
                }
            }
        );
    }

    public ListenableFuture<Void> sendAsync(String exchange, String routingKey, String message) {
        ListenableFutureTask<Void> future = new ListenableFutureTask<>(() -> {});
        try {
            long seqNo = channel.getNextPublishSeqNo();
            pending.put(seqNo, new PendingMessage(routingKey, message, future));

            channel.basicPublish(exchange, routingKey, MessageProperties.PERSISTENT_TEXT_PLAIN,
                message.getBytes());
        } catch (Exception e) {
            future.setException(e);
        }
        return future;
    }

    private void scheduleRetry(PendingMessage msg, int retryCount) {
        if (retryCount > 3) {
            // 超过重试次数，记录日志或投递到死信队列
            log.error("消息发送失败已达最大重试次数: {}", msg);
            return;
        }
        long delay = (long) Math.pow(2, retryCount) * 1000;  // 指数退避
        scheduler.schedule(() -> sendAsync(exchange, msg.routingKey, msg.body), delay, TimeUnit.MILLISECONDS);
    }
}
```

## 五、模式对比与选型

### 五种发送模式的性能对比

| 模式 | QPS（单线程）| CPU 利用率 | 内存占用 | 可靠性 |
|---|---|---|---|---|
| 同步发送 | ~500 | 低（阻塞等待）| 低 | 高 |
| Future 轮询 | ~2000 | 中 | 中 | 高 |
| 异步回调 | ~10000+ | 高 | 中 | 高 |
| Confirm 批量 | ~30000+ | 高 | 中 | 高 |
| Transaction | ~100 | 极低 | 低 | 最高 |

> 以上数据为理论估算，实际性能受网络 RTT、消息大小、Broker 负载等因素影响。

### 选型建议

```markdown
# 选型决策树

发送频率 < 100/s ?
  ├── 是：同步发送即可，代码简单够用
  └── 否：是否需要极高可靠性（金融、订单核心链路）？
        ├── 是：异步回调 + 重试机制
        └── 否：Confirm 批量发送，吞吐量大
```

### Spring AMQP 的模板封装

Spring AMQP 提供了 `RabbitTemplate`，默认使用同步发送，但可以通过配置切换为异步：

```java
// 配置 RabbitTemplate
@Bean
public RabbitTemplate rabbitTemplate(ConnectionFactory connectionFactory) {
    RabbitTemplate template = new RabbitTemplate(connectionFactory);

    // 配置发送确认回调
    template.setConfirmCallback((correlationData, ack, cause) -> {
        if (ack) {
            System.out.println("消息发送成功: " + correlationData);
        } else {
            System.err.println("消息发送失败: " + cause);
        }
    });

    // 配置返回回调
    template.setReturnsCallback(returned -> {
        System.err.println("消息无法路由到队列: " + returned.getRoutingKey());
    });

    // 设置 mandatory
    template.setMandatory(true);

    return template;
}

// 使用模板发送（内部已封装异步回调）
rabbitTemplate.convertAndSend("exchange", "routing-key", "message");
```

## 六、性能调优实战

### 批量发送

对于高吞吐场景，可以使用批量发送减少网络往返：

```java
// 批量确认模式
channel.confirmSelect();

for (int i = 0; i < batchSize; i++) {
    channel.basicPublish(exchange, routingKey, null, messages[i].getBytes());
}

// 等待整批确认
channel.waitForConfirmsOrDie(5000);
```

### 连接复用与信道池

```java
// 信道池配置
PooledConnectionProvider provider = new PooledConnectionProvider(
    new ConnectionFactory() {{
        setHost("127.0.0.1");
        setPort(5672);
    }}
);

// 最大 100 个信道
provider.setMaxConnections(10);
provider.setChannelMaxCount(100);
```

## 七、面试追问预判

### 追问一：RabbitMQ 的发送确认和 Kafka 的 ACK 有什么区别？

RabbitMQ 的 `Publisher Confirms` 和 Kafka 的 `acks` 都是保证消息可靠投递的机制，但实现不同：

| 维度 | RabbitMQ Confirms | Kafka acks |
|---|---|---|
| 确认层级 | Broker 写入内存/磁盘后 | Broker 同步复制到 ISR 副本后 |
| 确认数量 | 支持批量确认 | acks=1（Leader）、acks=all（ISR 全部） |
| 失败处理 | NACK 回调触发重发 | 超时重试 |
| 性能 | 低于 Kafka（AMQP 协议开销）| 高（零拷贝 + 顺序写）|

### 追问二：为什么生产环境推荐异步回调而不是事务模式？

事务模式每条消息都需要同步等待 Broker 的 `txCommit` 结果，性能损耗严重（吞吐量下降 90%+）。Confirm 模式在保证可靠性的同时，通过异步回调实现高吞吐，是生产环境的首选。
