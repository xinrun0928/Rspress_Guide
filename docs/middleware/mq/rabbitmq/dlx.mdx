# RabbitMQ 死信队列（DLX）与延迟队列

## 场景切入：消息处理失败了，怎么办？

消息处理失败了，你会怎么做？直接丢弃？重试？延迟重试？

RabbitMQ 提供了**死信队列（Dead Letter Exchange，DLX）** 和**延迟队列** 来处理这些场景。

## 一、死信队列（DLX）

### 1.1 什么是死信

当消息满足以下条件时，会成为死信：

| 条件 | 说明 |
|---|---|
| **消息被拒绝** | Consumer 拒绝消息且不重入队列 |
| **消息过期** | 消息的 TTL 到期 |
| **队列满** | 队列达到最大长度 |

```
死信产生场景：

┌─────────────────────────────────────────────────────────────┐
│                      死信产生流程                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Producer ──发送消息──> Queue ──消费失败──> Consumer        │
│                                      │                      │
│                                      │ basicNack/Reject    │
│                                      │ (requeue=false)     │
│                                      ▼                      │
│                              进入死信队列                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 死信队列配置

```java
// 配置死信交换机
Map<String, Object> args = new HashMap<>();
args.put("x-dead-letter-exchange", "order-dlx");      // 死信交换机
args.put("x-dead-letter-routing-key", "order.dead");  // 死信路由键

channel.queueDeclare("order-queue", true, false, false, args);

// 声明死信交换机和队列
channel.exchangeDeclare("order-dlx", "direct", true);
channel.queueDeclare("order-dlq", true, false, false, null);
channel.queueBind("order-dlq", "order-dlx", "order.dead");
```

### 1.3 死信队列架构

```
死信队列架构：

┌─────────────────────────────────────────────────────────────┐
│                                                             │
│   Producer ──发送消息──> Order Queue ──处理失败──> Consumer  │
│                                      │                      │
│                                      │ x-dead-letter-exchange │
│                                      ▼                      │
│                              Order DLX ──> Order DLQ        │
│                                                             │
│   处理死信：                                                │
│   Consumer ──消费死信──> DLQ ──> 分析死信原因 ──> 补偿处理  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 二、延迟队列

### 2.1 延迟队列实现方式

RabbitMQ 本身不提供延迟队列功能，但可以通过以下方式实现：

| 方式 | 原理 |
|---|---|
| **TTL + DLX** | 消息过期后进入死信队列 |
| **延迟插件** | 安装 rabbitmq_delayed_message_exchange 插件 |

### 2.2 方式一：TTL + DLX

```
TTL + DLX 实现延迟队列：

Producer ──发送消息(TTL=10s)──> Delay Queue ──TTL到期──> DLX ──> Target Queue
                                               │
                                               │
                                         10 秒后消息
                                         进入死信队列
```

```java
// TTL + DLX 实现延迟队列
// 1. 创建延迟交换机（使用 x-delayed-message 类型，需要插件）
// 或者使用普通交换机 + TTL + DLX

// 创建延迟队列（TTL 10 秒）
Map<String, Object> args = new HashMap<>();
args.put("x-message-ttl", 10000);                    // 消息 TTL 10 秒
args.put("x-dead-letter-exchange", "order-dlx");    // 死信交换机
args.put("x-dead-letter-routing-key", "order.created"); // 死信路由键

channel.queueDeclare("order-delay-queue", true, false, false, args);

// 创建目标队列
channel.queueDeclare("order-queue", true, false, false, null);

// 发送延迟消息
channel.basicPublish(
    "order-exchange",     // 交换机名
    "order.delay",       // 路由键
    MessageProperties.PERSISTENT_TEXT_PLAIN,
    "order data".getBytes()
);
// 消息会在 10 秒后进入 order-queue
```

### 2.3 方式二：延迟消息插件

```java
// 使用延迟消息插件
// 需要安装：rabbitmq_delayed_message_exchange 插件

// 声明延迟交换机
Map<String, Object> args = new HashMap<>();
args.put("x-delayed-type", "direct");

channel.exchangeDeclare(
    "order-delay-exchange",
    "x-delayed-message",  // 延迟交换机类型
    true,
    false,
    args
);

// 发送延迟消息（延迟 10 秒）
Map<String, Object> headers = new HashMap<>();
headers.put("x-delay", 10000);  // 延迟 10 秒

AMQP.BasicProperties properties = new AMQP.BasicProperties.Builder()
    .headers(headers)
    .build();

channel.basicPublish(
    "order-delay-exchange",
    "order.created",
    properties,
    "order data".getBytes()
);
```

## 三、死信队列处理

### 3.1 死信消费

```java
// 消费死信队列
channel.basicConsume("order-dlq", false, new DefaultConsumer(channel) {
    @Override
    public void handleDelivery(String consumerTag,
                             Envelope envelope,
                             AMQP.BasicProperties properties,
                             byte[] body) {
        String message = new String(body);

        // 1. 分析死信原因
        String deathReason = getDeathReason(properties);
        System.out.println("死信原因：" + deathReason);

        // 2. 获取原始消息信息
        Map<String, Object> headers = properties.getHeaders();
        if (headers != null) {
            String originalExchange = (String) headers.get("x-first-death-exchange");
            String originalRoutingKey = (String) headers.get("x-first-death-queue");
            System.out.println("原始交换机：" + originalExchange);
            System.out.println("原始队列：" + originalRoutingKey);
        }

        // 3. 根据原因处理
        if (canRetry(message)) {
            // 可重试：重新发送到原队列
            channel.basicPublish(originalExchange, originalRoutingKey,
                MessageProperties.PERSISTENT_TEXT_PLAIN, body);
        } else {
            // 不可重试：记录日志，人工处理
            logDeadLetter(message, deathReason);
        }

        // 4. 确认消息
        channel.basicAck(envelope.getDeliveryTag(), false);
    }
});
```

### 3.2 死信处理策略

| 策略 | 说明 | 适用场景 |
|---|---|---|
| **重试发送** | 重新发送到原队列 | 瞬时故障 |
| **降级处理** | 使用替代方案处理 | 非关键消息 |
| **记录日志** | 记录死信，人工处理 | 重要消息 |
| **永久存储** | 存储到数据库供后续分析 | 需要审计的场景 |

## 四、面试追问预判

### 追问一：死信是如何产生的？

消息成为死信的条件：

1. **消息被拒绝**：`basicNack` 或 `basicReject`，且 `requeue=false`
2. **消息过期**：消息的 TTL 到期
3. **队列满**：队列达到最大长度

### 追问二：如何实现延迟消息？

两种方式：

1. **TTL + DLX**：消息过期后进入死信队列
2. **延迟插件**：`rabbitmq_delayed_message_exchange` 插件

### 追问三：死信队列有什么用途？

| 用途 | 说明 |
|---|---|
| **消息补偿** | 失败消息重新处理 |
| **异常分析** | 分析消息失败原因 |
| **人工处理** | 重要消息人工介入 |
| **审计追踪** | 保留失败消息供审计 |
