# RabbitMQ 集群模式：普通集群 vs 镜像集群 vs 仲裁队列

## 场景切入：单点故障不可接受

如果 RabbitMQ 宕机了怎么办？

答案是集群。RabbitMQ 支持多种集群模式：**普通集群、镜像集群、仲裁队列**。每种模式有不同的特点和适用场景。

## 一、三种集群模式对比

| 模式 | 数据共享 | 高可用 | 适用场景 |
|---|---|---|---|
| **普通集群** | 不共享 | 低 | 负载均衡 |
| **镜像集群** | 队列镜像 | 中 | 高可用 |
| **仲裁队列** | Raft 复制 | 高 | Kafka 风格 |

---

## 二、普通集群

### 2.1 原理

普通集群中，队列只存在于一个节点，但元数据（交换机、队列定义）在所有节点共享：

```
普通集群架构：

┌─────────────────────────────────────────────────────────────┐
│                                                             │
│   节点 A                   节点 B                   节点 C   │
│   ┌─────────┐            ┌─────────┐            ┌─────────┐ │
│   │Exchange │            │Exchange │            │Exchange │ │
│   │Meta     │            │Meta     │            │Meta     │ │
│   ├─────────┤            ├─────────┤            ├─────────┤ │
│   │Queue A  │◄──────────│Meta    │───────────►│Meta    │ │
│   │(Master) │  元数据共享 │         │            │         │ │
│   └─────────┘            └─────────┘            └─────────┘ │
│                                                             │
│   Consumer 连接节点 B，消费节点 A 的队列                      │
│   ───> 需要从节点 A 拉取数据                                │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 配置

```bash
# 配置集群
rabbitmqctl stop_app
rabbitmqctl reset
rabbitmqctl join_cluster --ram rabbit@node-b  # 添加为 RAM 节点
rabbitmqctl start_app
```

```java
// 连接到集群中的任意节点
ConnectionFactory factory = new ConnectionFactory();
factory.setHost("node-a");
factory.setHost2("node-b");
factory.setHost3("node-c");
factory.setAutomaticRecoveryEnabled(true);  // 启用自动重连
```

### 2.3 特点

| 特点 | 说明 |
|---|---|
| **队列独占** | 队列只存在于一个节点 |
| **元数据共享** | 交换机、队列定义在所有节点共享 |
| **无高可用** | 队列所在节点宕机，队列不可用 |

---

## 三、镜像集群

### 3.1 原理

镜像集群将队列复制到所有节点，任意节点都可以处理消息：

```
镜像集群架构：

┌─────────────────────────────────────────────────────────────┐
│                                                             │
│   节点 A                   节点 B                   节点 C   │
│   ┌─────────┐            ┌─────────┐            ┌─────────┐ │
│   │Queue A  │            │Queue A  │            │Queue A  │ │
│   │(Master) │◄──────────│(Mirror)│◄──────────│(Mirror)│ │
│   └─────────┘   同步      └─────────┘   同步    └─────────┘ │
│                                                             │
│   所有节点都有完整的队列副本                                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 配置

```bash
# 设置镜像策略
rabbitmqctl set_policy ha-two "^order-" \
    '{"ha-mode":"exactly","ha-params":2,"ha-sync-mode":"automatic"}'
```

```java
// 策略参数说明：
// ha-mode: all | exactly | nodes
//   - all: 镜像到所有节点
//   - exactly: 镜像到指定数量节点
//   - nodes: 镜像到指定节点列表
// ha-params: ha-mode 的参数
// ha-sync-mode: automatic | manual
```

### 3.3 HA 策略类型

| 策略 | 说明 |
|---|---|
| `ha-mode=all` | 镜像到所有节点 |
| `ha-mode=exactly` | 镜像到 N 个节点 |
| `ha-mode=nodes` | 镜像到指定节点列表 |

---

## 四、仲裁队列（Quorum Queue）

### 4.1 原理

仲裁队列使用 Raft 共识算法，实现强一致性和高可用：

```
仲裁队列架构：

┌─────────────────────────────────────────────────────────────┐
│                                                             │
│   Leader ──> Follower ──> Follower                         │
│                                                             │
│   Raft 共识：                                               │
│   写入需要多数节点确认                                       │
│   支持故障转移                                              │
│   数据强一致性                                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 配置

```java
// 创建仲裁队列
Map<String, Object> args = new HashMap<>();
args.put("x-queue-type", "quorum");  // 仲裁队列类型

channel.queueDeclare(
    "order-queue",
    true,           // durable=true
    false,          // exclusive=false
    false,          // autoDelete=false
    args
);
```

### 4.3 特点

| 特点 | 说明 |
|---|---|
| **Raft 共识** | 使用 Raft 协议保证一致性 |
| **强一致** | 写入需要多数节点确认 |
| **自动故障转移** | Leader 宕机自动选举 |
| **磁盘存储** | 只支持磁盘节点 |

---

## 五、三种模式对比

### 5.1 对比表

| 维度 | 普通集群 | 镜像集群 | 仲裁队列 |
|---|---|---|---|
| **数据共享** | 队列独占 | 队列镜像 | Raft 复制 |
| **一致性** | 最终一致 | 最终一致 | 强一致 |
| **高可用** | 低 | 中 | 高 |
| **性能** | 高 | 中 | 中 |
| **存储** | 内存 + 磁盘 | 内存 + 磁盘 | 仅磁盘 |

### 5.2 选择建议

| 场景 | 推荐模式 |
|---|---|
| **负载均衡** | 普通集群 |
| **高可用（非金融）** | 镜像集群 |
| **强一致（金融）** | 仲裁队列 |
| **Kafka 风格** | 仲裁队列 |

## 六、面试追问预判

### 追问一：普通集群和镜像集群有什么区别？

| 模式 | 队列存储 | 高可用 |
|---|---|---|
| **普通集群** | 只在 Master 节点 | 无 |
| **镜像集群** | 所有节点 | 有 |

### 追问二：仲裁队列和镜像队列有什么区别？

| 维度 | 镜像队列 | 仲裁队列 |
|---|---|---|
| **一致性** | 最终一致 | 强一致 |
| **共识协议** | 无 | Raft |
| **写入确认** | Master 确认 | 多数节点确认 |
| **存储** | 内存 + 磁盘 | 仅磁盘 |

### 追问三：如何选择集群模式？

- **普通集群**：只需负载均衡，不需要高可用
- **镜像集群**：需要高可用，非金融级
- **仲裁队列**：需要强一致性，金融级场景
