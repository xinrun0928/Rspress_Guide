# 消息重试机制与死信队列

## 场景切入：失败是常态，重试才是王道

在分布式系统中，消息处理失败是常态，而非例外。

网络抖动、下游服务超时、数据库死锁……这些瞬时的故障不需要人工干预，消息队列的重试机制会自动处理。但如果重试多次仍然失败，这条消息该怎么处理？

答案就是**死信队列（Dead Letter Queue，DLQ）**。

## 一、消息重试机制

### 1.1 重试的必要性与风险

消息处理可能失败的原因：

| 原因 | 示例 | 是否应重试 |
|---|---|---|
| **瞬时故障** | 网络抖动、服务超时 | ✅ 重试 |
| **资源不足** | 数据库连接池耗尽 | ✅ 重试（等待资源恢复） |
| **业务异常** | 余额不足、库存不足 | ❌ 不重试 |
| **数据异常** | 消息格式错误 | ❌ 不重试 |
| **系统故障** | 服务宕机 | ❌ 不重试 |

```
重试策略选择：

┌─────────────────────────────────────────────────────────────┐
│                    重试决策树                                 │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  消息处理失败                                                │
│       │                                                     │
│       ▼                                                     │
│  ┌─────────────┐                                            │
│  │ 是瞬时故障？ │  ── 是 ──→ 立即重试 / 延迟重试             │
│  └──────┬──────┘                                            │
│         │ 否                                                │
│         ▼                                                   │
│  ┌─────────────┐                                            │
│  │ 是业务异常？ │  ── 是 ──→ 不重试，进入死信队列             │
│  └──────┬──────┘                                            │
│         │ 否                                                │
│         ▼                                                   │
│  ┌─────────────┐                                            │
│  │ 是数据异常？ │  ── 是 ──→ 不重试，进入死信队列             │
│  └─────────────┘                                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 重试策略

#### 策略一：本地重试

在消费者内部进行重试，不重新投递消息。

```java
// 本地重试：使用 Spring Retry
@Retryable(
    value = {TransientException.class},  // 可恢复的异常
    maxAttempts = 3,                      // 最大重试次数
    backoff = @Backoff(delay = 1000, multiplier = 2)  // 退避策略
)
@KafkaListener(topics = "order-topic")
public void consume(ConsumerRecord<String, String> record) {
    processOrder(record.value());
}

// 兜底方法（重试耗尽后）
@Recover
public void recover(TransientException e, ConsumerRecord<String, String> record) {
    log.error("重试耗尽，消息处理失败: {}", record.value(), e);
    // 发送到死信队列
    sendToDeadLetterQueue(record);
}
```

#### 策略二：远程重试

消息重新投递到队列，等待下次消费。

```java
// 远程重试：重新发送到队列
@KafkaListener(topics = "order-topic")
public void consume(ConsumerRecord<String, String> record) {
    try {
        processOrder(record.value());
    } catch (TransientException e) {
        // 瞬时故障，重新发送（携带重试次数）
        retryMessage(record, e);
    } catch (BusinessException e) {
        // 业务异常，进入死信队列
        sendToDeadLetterQueue(record, "BUSINESS_ERROR");
    }
}

private void retryMessage(ConsumerRecord<String, String> record, Exception e) {
    int retryCount = getRetryCount(record) + 1;

    if (retryCount >= MAX_RETRY_COUNT) {
        // 重试次数耗尽，进入死信队列
        sendToDeadLetterQueue(record, "MAX_RETRIES_EXCEEDED");
    } else {
        // 延迟重试（通过定时任务或延迟队列）
        scheduleRetry(record, retryCount);
    }
}

private int getRetryCount(ConsumerRecord<String, String> record) {
    Header header = record.headers().lastHeader("retry-count");
    if (header != null) {
        return Integer.parseInt(new String(header.value()));
    }
    return 0;
}
```

#### 策略三：延迟队列重试

使用延迟队列实现延迟重试。

```java
// RocketMQ 延迟重试
@RocketMQMessageListener(topic = "order-topic", consumerGroup = "order-consumer-group")
public class OrderConsumer implements RocketMQListener<OrderMessage> {

    @Autowired
    private RocketMQTemplate rocketMQTemplate;

    @Override
    public void onMessage(OrderMessage message) {
        try {
            processOrder(message);
        } catch (Exception e) {
            // 发送到延迟队列，实现延迟重试
            Message msg = new Message("order-retry-topic", message.toString().getBytes());
            msg.setDelayTimeLevel(calculateDelayLevel(message));  // 延迟等级
            rocketMQTemplate.send(msg);
        }
    }

    private int calculateDelayLevel(OrderMessage message) {
        int retryCount = message.getRetryCount();
        // 重试次数越多，延迟越长
        if (retryCount < 3) return 2;      // 10s
        if (retryCount < 5) return 3;      // 30s
        if (retryCount < 8) return 4;      // 1m
        return 5;                           // 2m
    }
}
```

### 1.3 重试退避策略

重试间隔应该逐渐增加，避免对下游系统造成压力。

```
退避策略对比：

立即重试：      [失败] [重试] [失败] [重试]  ──> 加重系统负担
               ──1s───

固定延迟：      [失败] [等待1s] [重试] [失败] [等待1s] [重试]
               ──────1s─────── ──────1s───────

指数退避：      [失败] [等待1s] [重试] [失败] [等待2s] [重试]
               ────1s─────── ────2s───────           [失败] [等待4s] [重试]
                                                    ────4s───────
```

```java
// 指数退避配置
@Retryable(
    value = {Exception.class},
    maxAttempts = 5,
    backoff = @Backoff(
        delay = 1000,        // 初始延迟 1 秒
        multiplier = 2.0,    // 每次延迟翻倍
        maxDelay = 60000     // 最大延迟 60 秒
    )
)
```

## 二、死信队列（Dead Letter Queue）

### 2.1 什么是死信队列

死信队列用于存放无法正常消费的消息。这些消息通常是：

- 重试次数耗尽
- 消息格式错误
- 业务处理明确失败

```
死信队列流程：

Producer ──发送消息──> Topic ──消费失败──> 重试队列
                                              │
                                    ┌─────────┼─────────┐
                                    │         │         │
                                    ▼         ▼         ▼
                              重试 1    重试 2    重试 3
                                    │         │         │
                                    └─────────┼─────────┘
                                              │
                                              ▼
                                    ┌─────────────────┐
                                    │   死信队列      │
                                    │ (Dead Letter)  │
                                    └─────────────────┘
```

### 2.2 Kafka 死信队列实现

Kafka 不提供原生的死信队列，需要自行实现。

```java
// Kafka 死信队列实现
@Configuration
public class KafkaDLQConfig {

    @Bean
    public ConcurrentKafkaListenerContainerFactory<String, String>
            kafkaListenerContainerFactory(
                ConsumerFactory<String, String> consumerFactory) {

        ConcurrentKafkaListenerContainerFactory<String, String> factory =
            new ConcurrentKafkaListenerContainerFactory<>();
        factory.setConsumerFactory(consumerFactory);
        factory.setCommonErrorHandler(errorHandler());
        return factory;
    }

    @Bean
    public CommonErrorHandler errorHandler() {
        // 配置错误处理器
        return new DefaultErrorHandler(
            (record, exception) -> {
                // 处理失败，发送到死信队列
                sendToDeadLetterQueue(record, exception);
            },
            new FixedBackOff(1000L, 3)  // 重试 3 次，间隔 1 秒
        );
    }

    private void sendToDeadLetterQueue(ConsumerRecord<?, ?> record, Exception e) {
        ProducerRecord<String, String> dlqRecord = new ProducerRecord<>(
            "order-topic-dlq",
            record.key(),
            record.value()
        );

        // 添加元数据
        dlqRecord.headers().add("original-topic", record.topic().getBytes());
        dlqRecord.headers().add("error-message", e.getMessage().getBytes());
        dlqRecord.headers().add("failed-at", new Date().toString().getBytes());

        kafkaTemplate.send(dlqRecord);
    }
}
```

### 2.3 RabbitMQ 死信队列实现

RabbitMQ 原生支持死信队列。

```java
// RabbitMQ 死信队列配置
@Configuration
public class RabbitMQDLQConfig {

    @Bean
    public DirectExchange orderExchange() {
        return new DirectExchange("order-exchange");
    }

    @Bean
    public Queue orderQueue() {
        Map<String, Object> args = new HashMap<>();
        // 配置死信交换机
        args.put("x-dead-letter-exchange", "order-dlx");
        // 配置死信路由键
        args.put("x-dead-letter-routing-key", "order.dead");
        // 配置消息 TTL
        args.put("x-message-ttl", 60000);  // 60 秒

        return new Queue("order-queue", true, false, false, args);
    }

    @Bean
    public Binding orderBinding() {
        return BindingBuilder
            .bind(orderQueue())
            .to(orderExchange())
            .with("order");
    }

    // 死信交换机和队列
    @Bean
    public DirectExchange orderDeadLetterExchange() {
        return new DirectExchange("order-dlx");
    }

    @Bean
    public Queue orderDeadLetterQueue() {
        return new Queue("order-queue-dlq", true);
    }

    @Bean
    public Binding orderDeadLetterBinding() {
        return BindingBuilder
            .bind(orderDeadLetterQueue())
            .to(orderDeadLetterExchange())
            .with("order.dead");
    }
}
```

## 三、死信消息处理

### 3.1 死信消息的特征

死信消息通常包含以下信息：

| 字段 | 说明 |
|---|---|
| 原始 Topic | 消息原本属于的 Topic |
| 原始分区 | 消息原本所在的分区 |
| 错误信息 | 失败的原因 |
| 重试次数 | 尝试处理的次数 |
| 失败时间 | 最后失败的时间 |

### 3.2 死信消息处理方式

| 方式 | 说明 | 适用场景 |
|---|---|---|
| **人工处理** | 人工排查并处理死信消息 | 重要业务、死信量少 |
| **定时补偿** | 定时任务重新处理死信消息 | 非关键业务 |
| **降级处理** | 记录异常但不影响主流程 | 日志、监控等 |
| **永久存储** | 死信消息永久保存供排查 | 需要审计的场景 |

### 3.3 死信消息处理流程

```java
// 死信消息处理器
@Component
public class DeadLetterProcessor {

    @Autowired
    private MessageService messageService;

    @KafkaListener(topics = "order-topic-dlq", groupId = "dlq-processor")
    public void processDeadLetter(ConsumerRecord<String, String> record) {
        String originalTopic = getHeader(record, "original-topic");
        String errorMessage = getHeader(record, "error-message");
        String failedAt = getHeader(record, "failed-at");

        log.warn("处理死信消息: topic={}, key={}, error={}",
            originalTopic, record.key(), errorMessage);

        // 1. 记录死信日志
        messageService.saveDeadLetter(record, originalTopic, errorMessage);

        // 2. 根据错误类型处理
        if (isRetryableError(errorMessage)) {
            // 可恢复错误，放回原队列
            retryOriginalTopic(record);
        } else {
            // 不可恢复错误，永久存储或人工处理
            notifyOperations(originalTopic, record);
        }
    }

    private boolean isRetryableError(String errorMessage) {
        // 判断是否是可恢复错误
        return errorMessage.contains("timeout")
            || errorMessage.contains("connection");
    }
}
```

## 四、重试与死信队列最佳实践

### 4.1 合理设置重试次数

| 业务类型 | 重试次数 | 重试间隔 |
|---|---|---|
| 实时业务 | 2-3 次 | 立即或 1-5 秒 |
| 异步业务 | 3-5 次 | 10 秒 - 1 分钟 |
| 批量任务 | 1-2 次 | 5-10 分钟 |

### 4.2 区分重试和死信

```java
// 异常分类
public class BusinessException extends RuntimeException {
    // 业务异常，不应重试
}

public class TransientException extends RuntimeException {
    // 瞬时异常，应该重试
}

public class DataException extends RuntimeException {
    // 数据异常，不应重试
}
```

## 五、面试追问预判

### 追问一：消息重试会导致消息乱序吗？

是的，如果同一业务 ID 的多条消息，一条处理失败重试，可能导致乱序：

```
场景：
消息 1 (创建订单) ──> 处理成功
消息 2 (支付成功) ──> 处理失败，重试
消息 3 (发货)     ──> 处理成功

结果：
1. 支付成功在创建订单之前处理
2. 订单还不存在就支付了
```

解决方案：顺序重试，失败消息放回队列尾部，保证顺序。

### 追问二：如何避免无限重试？

1. **设置最大重试次数**：超过次数进入死信队列
2. **指数退避**：重试间隔逐渐增加
3. **熔断机制**：连续失败后暂停消费，等待恢复

### 追问三：死信队列的消息如何处理？

常见处理方式：

1. **人工处理**：重要业务的死信，人工排查并处理
2. **定时补偿**：定时任务扫描死信，重新处理
3. **降级处理**：记录异常但不影响主流程
4. **永久存储**：保存供后续排查
