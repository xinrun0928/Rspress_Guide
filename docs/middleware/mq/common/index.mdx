## 为什么需要消息队列

在分布式系统中，服务之间的调用面临诸多挑战：

- **耦合性**：服务 A 直接调用服务 B，服务 B 不可用时，服务 A 也会受影响
- **同步阻塞**：服务 A 等待服务 B 返回结果才能继续执行
- **流量冲击**：瞬时流量直接打到后端系统，可能导致系统崩溃

消息队列通过**异步通信**解决了这些问题：

```
同步调用：                         异步通信（MQ）：
服务A ──调用──> 服务B              服务A ──发送消息──> MQ ──推送──> 服务B
    │              │                        │                      │
    │<───返回──────┘                        │                      │
    │                                      └──> 服务C ──> ...     │
    │                                                                 │
    A 需要等待 B 返回才能继续                 A 发送消息后立即返回
```

## 核心概念速览

| 概念 | 说明 | 例子 |
|---|---|---|
| **Producer** | 消息生产者，负责发送消息 | 订单系统发送订单消息 |
| **Consumer** | 消息消费者，负责接收和处理消息 | 库存系统接收订单消息扣减库存 |
| **Broker** | 消息队列服务端，负责存储和转发消息 | Kafka Broker、RocketMQ Broker |
| **Topic** | 主题，消息的分类单位 | order-topic、payment-topic |
| **Partition** | 分区，Kafka 中的概念，用于水平扩展 | Topic 的子集 |
| **Group** | 消费者组，用于负载均衡 | order-consumer-group |
| **Message** | 消息，包含 Header 和 Body | 订单信息、支付信息 |

## 学习建议

1. **先理解概念**：熟悉消息队列的基本组件和数据流
2. **对比学习**：对比三种 MQ 的异同，理解各自的设计哲学
3. **动手实践**：搭建开发环境，尝试发送和消费消息
4. **深入原理**：理解高可用、消息顺序、幂等性等高级特性
