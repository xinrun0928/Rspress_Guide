# 消息队列架构设计要点

## 场景切入：架构设计不是选型，而是权衡

面试官问：「你设计过消息队列的架构吗？」

这个问题看似简单，但真正考察的是你对消息队列的理解深度。架构设计不是选择哪个 MQ，而是根据业务场景，在**可靠性、性能、成本**之间做出权衡。

## 二、高可用设计

### 2.1 集群部署架构

消息队列的高可用首先体现在集群部署上。常见的部署模式有：

**模式一：主从复制**

```
┌─────────────┐     同步复制      ┌─────────────┐
│   Master    │ ←──────────────→ │   Slave     │
│   Broker    │                  │   Broker    │
└──────┬──────┘                  └─────────────┘
       │
       │ 读写
       ▼
   客户端
```

| 模式 | 说明 | 可靠性 | 性能 |
|---|---|---|---|
| 同步双写 | 写入 Master 和 Slave 都成功才返回 | 高 | 低 |
| 异步复制 | 写入 Master 成功即返回 | 中 | 高 |
| 半同步 | 写入 Master + 至少一个 Slave | 高 | 中 |

**模式二：多副本 + ISR**

Kafka 采用 ISR（In-Sync Replicas）机制，只有跟上 Leader 的副本才会参与选举。

```
ISR 集合：
┌─────────────────────────────────────────────────┐
│ Leader: Broker-1                               │
│ ISR: [Broker-1, Broker-2, Broker-3]            │
│ OSR: [Broker-4]  ← 落后太多，被踢出 ISR        │
└─────────────────────────────────────────────────┘
```

### 2.2 故障转移机制

```
故障转移流程：

1. Broker-1 宕机
   ┌─────────┐    ┌─────────┐    ┌─────────┐
   │ Broker-1│    │ Broker-2│    │ Broker-3│
   │ (Leader)│    │ (ISR)   │    │ (ISR)   │
   │   ❌    │    │         │    │         │
   └─────────┘    └─────────┘    └─────────┘

2. Controller 检测到宕机，触发选举
   ┌─────────┐    ┌─────────┐    ┌─────────┐
   │ Broker-1│    │ Broker-2│    │ Broker-3│
   │   ❌    │    │(新Leader)│    │ (ISR)   │
   └─────────┘    └─────────┘    └─────────┘

3. 选举完成，客户端自动重连
   故障恢复时间：通常 < 10 秒
```

### 2.3 数据恢复

| 策略 | 说明 | 适用场景 |
|---|---|---|
| **副本恢复** | 从 ISR 中选举新 Leader，其他副本从 Leader 同步 | 短期宕机 |
| **日志截断** | 截断到 HW（高水位），避免数据不一致 | 副本数据丢失 |
| **日志恢复** | 启动时加载日志，修复损坏的索引 | 数据损坏 |

## 三、可靠性设计

### 3.1 消息持久化

消息持久化是保证消息不丢失的基础。关键配置：

```properties
# Kafka 持久化配置
# 刷盘策略
flush.messages=10000        # 每 10000 条消息刷盘一次
flush.ms=1000               # 或每 1 秒刷盘一次

# 副本配置
replication.factor=3         # 3 副本
min.insync.replicas=2        # 最小 2 个副本同步成功
```

| 配置 | 可靠性 | 性能 | 适用场景 |
|---|---|---|---|
| `flush.messages=1` | 最高 | 最低 | 金融级场景 |
| `flush.messages=10000` | 高 | 高 | 普通场景 |
| `flush.messages=1000000` | 中 | 最高 | 高吞吐场景 |

### 3.2 确认机制

确认机制分为三个层面：

```
┌─────────────────────────────────────────────────────────────┐
│                    确认机制三层面                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  生产者 ────────► Broker                                     │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ acks 配置                                              │   │
│  │  · acks=0：不等确认（最快，可能丢消息）                  │   │
│  │  · acks=1：Leader 确认（中等）                         │   │
│  │  · acks=all：ISR 全部确认（最安全）                     │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  Broker ────────► 副本                                      │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 副本同步机制                                           │   │
│  │  · ISR：只同步跟上 Leader 的副本                       │   │
│  │  · 落后副本会被踢出 ISR                               │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  消费者 ────────► Broker                                     │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ Offset 提交                                           │   │
│  │  · 自动提交：可能丢消息（处理前就提交）                 │   │
│  │  · 手动提交：更可靠（处理后提交）                      │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.3 幂等性设计

幂等性是防止消息重复消费的关键。常见的幂等方案：

| 方案 | 实现方式 | 优点 | 缺点 |
|---|---|---|---|
| **数据库唯一索引** | 消息 ID 作为唯一键 | 简单可靠 | 需要数据库支持 |
| **Redis 去重** | 消息 ID 存入 Redis Set | 速度快 | 需要额外的 Redis |
| **消息表** | 消费前先插入消息表 | 支持多表关联 | 增加数据库压力 |
| **分布式锁** | 处理前加锁 | 并发安全 | 性能开销大 |

```java
// 数据库唯一索引实现幂等
public void consume(OrderMessage message) {
    try {
        // 使用消息 ID 作为唯一键
        orderMapper.insertSelective(new Order() {{
            setMessageId(message.getId());  // 唯一索引
            setOrderId(message.getOrderId());
            setAmount(message.getAmount());
        }});
    } catch (DuplicateKeyException e) {
        // 唯一索引冲突，说明已经处理过，忽略
        log.info("订单 {} 已存在，忽略重复消息", message.getOrderId());
    }
}
```

## 四、性能设计

### 4.1 批量处理

批量处理是提升吞吐量的关键：

```java
// 生产者批量配置
Properties props = new Properties();
props.put("batch.size", 16384);        // 16KB 批次
props.put("linger.ms", 10);            // 等待 10ms 组批
props.put("buffer.memory", 33554432);   // 32MB 缓冲区

// 消费者批量配置
props.put("max.poll.records", 500);    // 每次最多 500 条
```

### 4.2 压缩

压缩可以显著减少网络传输：

| 算法 | 压缩比 | CPU 开销 | 推荐场景 |
|---|---|---|---|
| gzip | 高 | 高 | 带宽紧张 |
| snappy | 中 | 中 | 平衡场景 |
| lz4 | 中低 | 低 | 低延迟场景 |
| zstd | 高 | 中 | 追求高压缩比 |

```java
props.put("compression.type", "lz4");  // 启用压缩
```

### 4.3 分区并行

分区数决定了最大的并行消费能力：

```java
// 分区规划公式
分区数 = max(消费者数量, 期望吞吐量 / 单分区吞吐)

// 单分区吞吐估算
单分区吞吐 ≈ 50MB/s（普通磁盘）
单分区吞吐 ≈ 200MB/s（SSD）
```

## 五、可观测性设计

### 5.1 监控指标

| 指标 | 说明 | 告警阈值 |
|---|---|---|
| **消息堆积量** | 未消费消息数量 | > 10 万 |
| **消费延迟** | 消费落后于生产的程度 | > 1 分钟 |
| **生产者错误率** | 发送失败的消息比例 | > 1% |
| **消费者 Lag** | Consumer 落后于 Producer 的 offset 差 | > 1 万 |
| **副本同步延迟** | Follower 落后于 Leader 的程度 | > 1000 条 |

### 5.2 日志规范

```java
// 结构化日志，便于排查
logger.info("消息发送成功",
    "topic", topic,
    "partition", partition,
    "offset", offset,
    "key", key,
    "timestamp", timestamp
);
```

### 5.3 链路追踪

将消息的追踪 ID 透传到消息 Header，便于全链路追踪：

```java
// 生产者添加追踪 ID
producer.send(record, new Callback() {
    @Override
    public void onCompletion(RecordMetadata metadata, Exception exception) {
        // 将 metadata 信息记录到 TracingContext
        TracingContext.setProducerMetadata(metadata);
    }
});

// 消费者读取追踪 ID
@KafkaListener(topics = "order-topic")
public void consume(ConsumerRecord<String, String> record) {
    String traceId = record.headers().lastHeader("trace-id").value();
    TracingContext.setTraceId(traceId);

    // 业务处理...
}
```

## 六、面试追问预判

### 追问一：如何保证消息不丢失？

需要三个环节都做好：

1. **生产者端**：确认机制（acks=all）+ 重试 + 幂等
2. **Broker 端**：副本机制（replication.factor >= 3）+ 刷盘策略
3. **消费者端**：手动提交 Offset（处理完成后再提交）

### 追问二：如何处理消息重复？

消息重复在分布式系统中是常见问题（因为网络的不可靠性）。解决方案是**幂等消费**：

- 数据库唯一索引
- Redis 去重
- 消息表

### 追问三：消息队列如何选型？

根据业务场景选择：

| 场景 | 推荐 | 原因 |
|---|---|---|
| 日志收集、流处理 | Kafka | 高吞吐、消息回溯 |
| 复杂路由 | RabbitMQ | Exchange 灵活路由 |
| 事务消息 | RocketMQ | 原生支持 |
| 高可靠、低延迟 | RocketMQ | 金融级可靠性 |
