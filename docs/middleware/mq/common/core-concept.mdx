# 消息队列核心概念

## 场景切入：一条消息从发送到消费，经历了什么？

用户下单后，订单系统将订单消息发送到消息队列，库存系统、消费系统、日志系统都可以消费这条消息。整个过程是怎样的？

```
┌──────────┐    发送消息    ┌──────────┐    存储消息    ┌──────────┐
│          │ ───────────> │          │ ───────────> │          │
│ Producer │              │  Broker  │              │  Topic   │
│          │ <─────────── │          │ <─────────── │          │
└──────────┘    响应确认   └──────────┘    拉取消息    └──────────┘
                                                            │
                                          ┌─────────────────┼─────────────────┐
                                          │                 │                 │
                                          ▼                 ▼                 ▼
                                     ┌──────────┐    ┌──────────┐    ┌──────────┐
                                     │ Consumer │    │ Consumer │    │ Consumer │
                                     │   库存   │    │   物流   │    │   日志   │
                                     └──────────┘    └──────────┘    └──────────┘
```

这就是消息队列最基本的数据流。接下来，我们逐个拆解核心概念。

---

## 一、七大核心概念

### 1.1 Message（消息）

消息是消息队列中最小的传输单元，通常包含两部分：

| 部分 | 说明 | 示例 |
|---|---|---|
| **Header（消息头）** | 元数据信息 | 消息 ID、创建时间、过期时间、优先级 |
| **Body（消息体）** | 实际业务数据 | 订单信息、支付信息、用户行为 |

```java
// Kafka 消息结构示例
public class ProducerRecord<K, V> {
    private final String topic;      // Topic 名称
    private final Integer partition; // 分区号（可选）
    private final Headers headers;   // 消息头
    private final K key;            // 消息键（用于分区路由）
    private final V value;          // 消息体
    private final Long timestamp;    // 时间戳
}
```

**消息的关键属性**：

- **消息 ID**：唯一标识一条消息
- **消息键（Key）**：用于决定消息发送到哪个分区
- **消息体（Value）**：业务数据，通常是 JSON、Protobuf 等格式
- **时间戳**：消息创建或写入的时间
- **过期时间（TTL）**：消息的有效期，过期后自动删除

### 1.2 Producer（生产者）

生产者负责发送消息到消息队列。它的核心职责是：

- **消息发送**：将业务数据封装成消息，发送到指定的 Topic
- **分区路由**：根据消息 Key 或负载策略，决定消息发送到哪个分区
- **发送确认**：等待 Broker 的确认，确保消息成功写入
- **重试机制**：发送失败时，自动重试

```java
// 生产者发送消息示例
KafkaProducer<String, String> producer = new KafkaProducer<>(props);

// 发送消息
producer.send(new ProducerRecord<>(
    "order-topic",      // Topic 名称
    "order-123",        // 消息 Key
    "{\"orderId\": 123, \"amount\": 100.0}"  // 消息体
));

// 异步发送并获取结果
producer.send(record, (metadata, exception) -> {
    if (exception == null) {
        System.out.println("发送成功：" + metadata.offset());
    } else {
        System.err.println("发送失败：" + exception.getMessage());
    }
});
```

**生产者的核心配置**：

| 配置项 | 说明 | 推荐值 |
|---|---|---|
| `acks` | 确认机制 | `all`（高可靠） |
| `retries` | 重试次数 | `3` |
| `batch.size` | 批量大小 | `16384`（16KB） |
| `linger.ms` | 等待时间 | `10`（毫秒） |
| `enable.idempotence` | 开启幂等 | `true` |

### 1.3 Consumer（消费者）

消费者从消息队列中拉取消息并处理。它的核心职责是：

- **消息拉取**：主动从 Broker 拉取消息
- **消息处理**：执行业务逻辑
- **消费确认**：处理完成后，确认消息已被消费
- **Offset 管理**：记录消费进度，支持消息重放

```java
// 消费者消费消息示例
KafkaConsumer<String, String> consumer = new KafkaConsumer<>(props);
consumer.subscribe(Arrays.asList("order-topic"));

while (true) {
    // 拉取消息
    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(1000));

    for (ConsumerRecord<String, String> record : records) {
        System.out.println("消费消息：" + record.value());
        // 业务处理...
    }

    // 提交 Offset
    consumer.commitSync();
}
```

**消费者的核心配置**：

| 配置项 | 说明 | 推荐值 |
|---|---|---|
| `group.id` | 消费者组 ID | 根据业务设置 |
| `auto.offset.reset` | Offset 重置策略 | `earliest` 或 `latest` |
| `enable.auto.commit` | 自动提交 Offset | `false`（手动提交更可靠） |
| `max.poll.records` | 每次拉取数量 | `500` |
| `session.timeout.ms` | 会话超时 | `30000`（毫秒） |

### 1.4 Broker（消息代理）

Broker 是消息队列的服务端，负责：

- **消息存储**：将消息持久化到磁盘（或内存）
- **消息路由**：根据 Topic 和分区，将消息分发到对应的消费者
- **消息拉取**：响应消费者的拉取请求，返回消息
- **副本管理**：维护消息副本，保证高可用

```
Broker 架构：

┌─────────────────────────────────────────────────────────┐
│                        Broker                           │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   ┌───────────┐  ┌───────────┐  ┌───────────┐          │
│   │  Topic A  │  │  Topic B  │  │  Topic C  │          │
│   │ ┌───────┐ │  │ ┌───────┐ │  │ ┌───────┐ │          │
│   │ │Part 0 │ │  │ │Part 0 │ │  │ │Part 0 │ │          │
│   │ │Part 1 │ │  │ │Part 1 │ │  │ │Part 1 │ │          │
│   │ │Part 2 │ │  │ │Part 2 │ │  │ │Part 2 │ │          │
│   │ └───────┘ │  │ └───────┘ │  │ └───────┘ │          │
│   └───────────┘  └───────────┘  └───────────┘          │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 1.5 Topic（主题）

Topic 是消息的分类单元，类似于数据库中的表：

- **逻辑概念**：Topic 是消息的容器，生产者发送消息到 Topic，消费者从 Topic 消费消息
- **多消费者**：一个 Topic 可以被多个消费者组订阅，互不影响
- **分区实现**：Topic 由一个或多个 Partition 组成，实现水平扩展

```
Topic 与 Partition 的关系：

Topic: order-topic
┌─────────────────────────────────────────────────────────┐
│  order-topic                                             │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐    │
│  │ Partition 0  │  │ Partition 1  │  │ Partition 2  │    │
│  │ [msg 1]     │  │ [msg 2]     │  │ [msg 3]     │    │
│  │ [msg 4]     │  │ [msg 5]     │  │ [msg 6]     │    │
│  │ [msg 7]     │  │ [msg 8]     │  │ [msg 9]     │    │
│  └─────────────┘  └─────────────┘  └─────────────┘    │
└─────────────────────────────────────────────────────────┘
```

### 1.6 Partition（分区）

Partition 是 Kafka（以及 RocketMQ）的核心概念，用于实现水平扩展：

| 特性 | 说明 |
|---|---|
| **有序性** | 分区内消息有序，分区间消息无序 |
| **并行消费** | 多个 Partition 可以被不同的 Consumer 并行消费 |
| **分区策略** | 根据 Key 的哈希值或轮询，决定消息发送到哪个分区 |
| **副本** | 每个 Partition 可以有多个副本，保证高可用 |

```
分区与消费者关系：

Topic: order-topic (3 个 Partition)

Consumer Group A:
  Consumer 1 → Partition 0, Partition 2
  Consumer 2 → Partition 1

Consumer Group B (独立消费)：
  Consumer 1 → Partition 0, Partition 1, Partition 2
```

**分区数规划建议**：

```java
// 经验公式
分区数 = max(消费者数量, 期望吞吐量 / 单分区吞吐)
// 单分区吞吐 ≈ 50MB/s（视网络和磁盘而定）
```

### 1.7 Consumer Group（消费者组）

消费者组是一组 Consumer 实例，共同消费一个 Topic：

- **负载均衡**：组内 Consumer 均摊分区，实现负载均衡
- **独立消费**：不同消费者组独立消费，互不影响
- **组管理**：Kafka 自动管理 Consumer 的加入和离开

```
消费者组示例：

Topic: order-topic (4 个 Partition)

消费者组 A: order-service
  Consumer 1 → Partition 0
  Consumer 2 → Partition 1
  Consumer 3 → Partition 2
  Consumer 4 → Partition 3

消费者组 B: analytics-service
  Consumer 1 → Partition 0, 1
  Consumer 2 → Partition 2, 3

每个消费者组独立消费，消息会被重复消费给每个组
```

## 三、核心概念对比

| 概念 | Kafka | RabbitMQ | RocketMQ |
|---|---|---|---|
| **Topic** | ✅ 支持 | ✅ 支持 | ✅ 支持 |
| **Partition** | ✅ 支持 | ❌ Queue 即分区 | ✅ 支持 |
| **Consumer Group** | ✅ 支持 | ❌ 用 Exchange 路由 | ✅ 支持 |
| **Queue** | ❌ 用 Partition 模拟 | ✅ 核心概念 | ✅ 支持 |
| **Message** | ✅ 支持 Header/Body | ✅ 支持 | ✅ 支持 |

## 四、面试追问预判

### 追问一：Topic 和 Partition 的关系是什么？

Topic 是逻辑概念，Partition 是物理实现。一个 Topic 包含多个 Partition，每个 Partition 是独立的日志文件。Partition 的数量决定了最大的并行消费能力。

### 追问二：Consumer Group 和 Partition 的关系是什么？

Consumer Group 内的 Consumer 数量不能超过 Partition 数量，否则多余的 Consumer 会空闲。Partition 数量决定了 Consumer Group 的最大并行度。

### 追问三：消息队列如何保证消息不丢失？

三个环节都需要保证：
1. **生产者**：使用确认机制（acks=all）+ 重试
2. **Broker**：副本机制（replication）+ 刷盘策略
3. **消费者**：手动提交 Offset（处理完成后再提交）
