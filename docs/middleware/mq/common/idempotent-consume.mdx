# 如何保证消息的幂等性消费？

## 场景切入：消息重复不可怕，可怕的是处理重复了

消息队列有一条黄金法则：**消息至少消费一次，但业务需要幂等处理**。

为什么？因为网络的不可靠性。在分布式系统中，消息重复是不可避免的：

- **生产者重试**：发送失败后自动重试，可能重复发送
- **消费者重试**：消费失败后重新拉取，可能重复消费
- **Broker 重试**：Broker 故障恢复后，可能重新发送消息

```
消息重复的场景：

场景 1：生产者重试
Producer ──发送消息 A──> Broker
   │        │
   │        │ 网络超时，发送失败
   │        ▼
   └────重试──> Broker（消息 A 已存在！）

场景 2：消费者重试
Consumer ──拉取消息 A──> Broker
   │        │
   │        │ 消费成功，但处理超时
   │        ▼
   └────再次拉取──> Broker（消息 A 再次拉取！）
```

因此，业务系统必须能够**幂等处理重复消息**，而不是假设消息不会重复。

---

## 一、幂等性的定义

### 1.1 什么是幂等性

幂等性是指：同一操作执行一次和执行多次，效果相同。

```
数学表达：
f(f(x)) = f(x)

业务表达：
消费者处理消息 1 次 = 处理消息 N 次（N >= 1）
```

### 1.2 幂等性的重要性

| 问题 | 影响 |
|---|---|
| 重复扣款 | 用户账户被多次扣款 |
| 重复下单 | 同一订单创建多次 |
| 重复发货 | 同一商品多次发货 |
| 重复积分 | 用户获得多次积分 |

**幂等性是金融级系统的基本要求。**

## 二、幂等性消费方案

### 2.1 方案一：数据库唯一索引

最常用、最可靠的方案。使用消息 ID 或业务 ID 作为唯一索引。

```java
// 使用消息 ID 作为唯一键
public void consume(OrderMessage message) {
    try {
        Order order = new Order();
        order.setMessageId(message.getId());     // 唯一索引
        order.setOrderId(message.getOrderId());
        order.setAmount(message.getAmount());
        order.setStatus("PAID");

        orderMapper.insertSelective(order);
    } catch (DuplicateKeyException e) {
        // 唯一索引冲突，说明已处理过
        log.info("订单 {} 已存在，忽略重复消息", message.getOrderId());
    }
}
```

```sql
-- 建表语句
CREATE TABLE orders (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    message_id VARCHAR(64) NOT NULL UNIQUE,  -- 消息 ID，唯一索引
    order_id VARCHAR(64) NOT NULL,
    amount DECIMAL(10, 2),
    status VARCHAR(20),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

**优点**：
- 实现简单，可靠性高
- 数据库保证唯一性

**缺点**：
- 依赖数据库的唯一索引
- 大量重复消息可能导致数据库压力

### 2.2 方案二：Redis 去重

使用 Redis 记录已处理的消息 ID。

```java
// Redis 去重
public void consume(OrderMessage message) {
    String messageId = message.getId();
    String key = "processed:order:" + messageId;

    // 设置 key，过期时间 24 小时
    Boolean isNew = redisTemplate.opsForValue().setIfAbsent(key, "1",
        Duration.ofHours(24));

    if (Boolean.TRUE.equals(isNew)) {
        // 新消息，处理业务
        processOrder(message);
    } else {
        // 重复消息，忽略
        log.info("消息 {} 已处理过，忽略", messageId);
    }
}
```

**优点**：
- 性能高，Redis 读取速度快
- 支持分布式环境

**缺点**：
- 需要额外的 Redis 组件
- 依赖 Redis 的可用性

### 2.3 方案三：消息表（业务表 + 消息表）

将消息处理记录存入数据库，与业务表一起管理。

```java
// 消息表 + 业务表
public void consume(OrderMessage message) {
    // 1. 检查消息是否已处理
    MessageRecord record = messageRecordMapper.selectByMessageId(message.getId());
    if (record != null) {
        log.info("消息 {} 已处理，状态：{}", message.getId(), record.getStatus());
        return;
    }

    // 2. 开启事务
    transactionTemplate.execute(status -> {
        try {
            // 3. 插入消息记录
            MessageRecord newRecord = new MessageRecord();
            newRecord.setMessageId(message.getId());
            newRecord.setStatus("PROCESSING");
            newRecord.setCreatedAt(new Date());
            messageRecordMapper.insert(newRecord);

            // 4. 处理业务
            processOrder(message);

            // 5. 更新消息状态
            newRecord.setStatus("COMPLETED");
            newRecord.setUpdatedAt(new Date());
            messageRecordMapper.updateByPrimaryKey(newRecord);

            return null;
        } catch (Exception e) {
            // 失败，更新状态
            newRecord.setStatus("FAILED");
            newRecord.setError(e.getMessage());
            messageRecordMapper.updateByPrimaryKey(newRecord);
            throw e;
        }
    });
}
```

```sql
-- 消息记录表
CREATE TABLE message_records (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    message_id VARCHAR(64) NOT NULL UNIQUE,
    status VARCHAR(20),  -- PROCESSING, COMPLETED, FAILED
    error VARCHAR(500),
    created_at TIMESTAMP,
    updated_at TIMESTAMP
);
```

**优点**：
- 支持复杂的业务流程
- 可以记录处理状态和错误信息

**缺点**：
- 增加数据库负担
- 实现较复杂

### 2.4 方案四：分布式锁

使用分布式锁保证同一消息只被一个消费者处理。

```java
// 分布式锁实现
public void consume(OrderMessage message) {
    String lockKey = "lock:order:" + message.getId();
    String lockValue = UUID.randomUUID().toString();

    try {
        // 尝试获取锁
        Boolean acquired = redisTemplate.opsForValue().setIfAbsent(
            lockKey, lockValue, Duration.ofSeconds(30));

        if (Boolean.TRUE.equals(acquired)) {
            // 获取锁成功，处理消息
            processOrder(message);
        } else {
            // 获取锁失败，可能在其他实例处理中
            log.info("消息 {} 正在被其他实例处理", message.getId());
        }
    } finally {
        // 释放锁（只能释放自己的锁）
        String currentValue = redisTemplate.opsForValue().get(lockKey);
        if (lockValue.equals(currentValue)) {
            redisTemplate.delete(lockKey);
        }
    }
}
```

**优点**：
- 保证分布式环境下的互斥
- 适用于高并发场景

**缺点**：
- 锁的性能开销
- 需要处理锁超时

### 2.5 方案五：业务状态机

利用业务状态机的不可逆性，保证幂等。

```
订单状态机：
待支付 → 已支付 → 已发货 → 已完成

不可逆操作：
- 已支付的订单不能再回到待支付
- 已发货的订单不能再回到已支付
```

```java
// 状态机幂等
public void consume(PaymentMessage message) {
    // 1. 查询订单当前状态
    Order order = orderMapper.selectByOrderId(message.getOrderId());

    // 2. 检查状态是否允许转换
    if ("待支付".equals(order.getStatus())) {
        // 可以转换为已支付
        order.setStatus("已支付");
        order.setPayTime(new Date());
        orderMapper.updateByPrimaryKey(order);
    } else {
        // 状态已转换，忽略重复消息
        log.info("订单 {} 状态已是 {}，忽略支付消息",
            message.getOrderId(), order.getStatus());
    }
}
```

## 三、幂等性实现方案对比

| 方案 | 适用场景 | 优点 | 缺点 |
|---|---|---|---|
| **唯一索引** | 有数据库操作的业务 | 简单可靠 | 依赖数据库 |
| **Redis 去重** | 高并发场景 | 性能高 | 依赖 Redis |
| **消息表** | 复杂业务流程 | 支持状态追踪 | 实现复杂 |
| **分布式锁** | 高并发互斥 | 分布式支持 | 性能开销 |
| **状态机** | 有明确状态的业务 | 利用状态不可逆 | 受限业务场景 |

## 四、消息 ID 生成方案

### 4.1 为什么需要消息 ID

消息 ID 是幂等处理的基础。需要保证：

- **唯一性**：每个消息有唯一的 ID
- **可追溯性**：可以从 ID 追溯到消息的来源和时间

### 4.2 消息 ID 生成策略

```java
// 策略一：UUID
String messageId = UUID.randomUUID().toString();

// 策略二：雪花算法
long messageId = SnowflakeIdWorker.getInstance().nextId();

// 策略三：业务 ID + 时间戳
String messageId = orderId + "_" + System.currentTimeMillis();

// 策略四：生产者 ID + 序列号
String messageId = producerId + "_" + sequenceNumber;
```

### 4.3 生产者配置消息 ID

```java
// Kafka 生产者
producer.send(new ProducerRecord<>(
    topic,
    key,
    message
), (metadata, exception) -> {
    // metadata.offset() 和 metadata.timestamp() 可作为消息 ID
    log.info("消息已发送，offset: {}, timestamp: {}",
        metadata.offset(), metadata.timestamp());
});
```

## 五、消费者幂等处理最佳实践

### 5.1 完整示例

```java
@Service
public class OrderConsumer {

    @Autowired
    private OrderMapper orderMapper;

    @Autowired
    private RedisTemplate<String, String> redisTemplate;

    @Autowired
    private TransactionTemplate transactionTemplate;

    @KafkaListener(topics = "order-topic", groupId = "order-consumer-group")
    public void consume(ConsumerRecord<String, String> record) {
        OrderMessage message = parseMessage(record);

        // 1. Redis 去重（快速判断）
        String dedupKey = "dedup:order:" + message.getId();
        Boolean isProcessed = redisTemplate.hasKey(dedupKey);
        if (Boolean.TRUE.equals(isProcessed)) {
            log.info("消息 {} 已处理，忽略", message.getId());
            return;
        }

        // 2. 业务处理（带事务）
        try {
            processOrder(message);

            // 3. 标记已处理
            redisTemplate.opsForValue().set(dedupKey, "1",
                Duration.ofHours(24));

        } catch (DuplicateKeyException e) {
            // 数据库唯一索引冲突，忽略
            log.info("订单 {} 已存在，忽略重复消息", message.getOrderId());
        } catch (Exception e) {
            // 其他异常，抛出以便重试
            throw e;
        }
    }

    private void processOrder(OrderMessage message) {
        transactionTemplate.execute(status -> {
            Order order = new Order();
            order.setMessageId(message.getId());      // 唯一索引
            order.setOrderId(message.getOrderId());
            order.setAmount(message.getAmount());
            order.setStatus("CREATED");
            orderMapper.insertSelective(order);
            return null;
        });
    }
}
```

### 5.2 幂等性检查顺序

```
幂等性检查顺序（从快到慢）：

1. Redis 去重（毫秒级）
   ↓ 缓存未命中
2. 数据库唯一索引（毫秒级）
   ↓ 唯一索引冲突
3. 忽略重复消息

性能最优解：先 Redis，再数据库
```

## 六、面试追问预判

### 追问一：为什么消息会重复？

消息重复的三个来源：

1. **生产者重试**：网络超时后自动重试
2. **消费者重试**：处理失败后重新拉取
3. **Broker 重试**：故障恢复后重新发送

### 追问二：如何选择幂等方案？

| 场景 | 推荐方案 |
|---|---|
| 简单业务（单表操作） | 数据库唯一索引 |
| 高并发场景 | Redis 去重 |
| 复杂业务流程 | 消息表 |
| 需要状态追踪 | 消息表 |
| 有明确状态机 | 状态机 |

### 追问三：幂等性和事务有什么区别？

| 维度 | 幂等性 | 事务 |
|---|---|---|
| **目的** | 防止重复处理 | 保证原子性 |
| **解决** | 重复消费问题 | 数据一致性问题 |
| **实现** | 业务代码 | 数据库特性 |
| **关系** | 幂等是事务的基础 | 事务包含幂等 |

### 追问四：Redis 去重和数据库唯一索引如何选择？

| 场景 | Redis 去重 | 数据库唯一索引 |
|---|---|---|
| **性能** | 更高 | 中等 |
| **可靠性** | 依赖 Redis | 依赖数据库 |
| **适用** | 高并发快速判断 | 业务数据处理 |
| **推荐** | 先 Redis，再数据库 | 最终保障 |
