# Kafka vs RabbitMQ vs RocketMQ 选型指南

## 场景切入：选型不是选最好的，而是选最合适的

面试官问：「你们项目用的什么消息队列？为什么选这个？」

大多数人会回答：「用的 Kafka，因为吞吐量大。」

但面试官想听到的不只是名字，而是**你为什么这样选**。选型不是选最好的，而是根据业务场景，在吞吐量、可靠性、功能、运维复杂度之间做出权衡。

本文将深入对比 Kafka、RabbitMQ、RocketMQ 的异同，帮助你做出正确的技术选型。

## 一、三种 MQ 的设计哲学

### 1.1 Kafka：日志优先，海量数据

Kafka 由 LinkedIn 开发，最初用于处理海量日志数据。它的设计哲学是**日志优先**：

- **高吞吐**：顺序写、零拷贝、批量处理
- **持久化**：消息持久化到磁盘，支持消息回溯
- **水平扩展**：通过分区实现水平扩展
- **流处理**：与 Kafka Streams、Flink 无缝集成

```
Kafka 适用场景：
├── 日志收集（ELK）
├── 实时流处理（Kafka Streams、Flink）
├── 消息管道（数据传输）
└── 活动追踪（用户行为分析）
```

### 1.2 RabbitMQ：灵活路由，企业级

RabbitMQ 由 Pivotal 开发，采用 Erlang 语言编写。它的设计哲学是**灵活路由**：

- **灵活的路由**：通过 Exchange 和 Binding 实现复杂的路由规则
- **多协议支持**：AMQP、MQTT、STOMP
- **可靠性**：消息确认、事务、死信队列
- **易用性**：管理界面友好，配置灵活

```
RabbitMQ 适用场景：
├── 任务队列（异步任务）
├── 发布订阅（事件驱动）
├── 复杂路由（多消费者、不同路由规则）
└── 协议转换（多协议网关）
```

### 1.3 RocketMQ：金融级，事务消息

RocketMQ 由阿里巴巴开发，最初用于支撑双十一的海量消息处理。它的设计哲学是**金融级可靠性**：

- **事务消息**：原生支持分布式事务
- **延迟消息**：支持多级延迟
- **顺序消息**：全局顺序和分区顺序
- **Java 原生**：与 Spring 生态深度集成

```
RocketMQ 适用场景：
├── 电商交易（订单、支付）
├── 金融消息（对账、清算）
├── 定时任务（延迟处理）
└── 消息推送（实时通知）
```

## 二、核心指标对比

### 2.1 吞吐量对比

```
吞吐量对比（单机）：

Kafka:     ████████████████████████████████████ 100万+
RocketMQ:  ████████████████                   10-15万
RabbitMQ:  ██████                              1-5万
```

| MQ | 单机吞吐量 | 适用规模 |
|---|---|---|
| **Kafka** | 100 万+ QPS | 大规模数据处理 |
| **RocketMQ** | 10-15 万 QPS | 中大规模系统 |
| **RabbitMQ** | 1-5 万 QPS | 中小规模系统 |

**为什么 Kafka 吞吐量最高？**

- 顺序写磁盘：磁盘 IO 效率极高
- 零拷贝：减少数据拷贝次数
- 批量处理：减少网络请求
- 压缩：减少网络传输

### 2.2 延迟对比

| MQ | 平均延迟 | 99.99% 延迟 | 适用场景 |
|---|---|---|---|
| **RocketMQ** | 1-5ms | < 5ms | 低延迟场景 |
| **Kafka** | 2-10ms | < 10ms | 高吞吐场景 |
| **RabbitMQ** | 1-3ms | < 10ms | 低并发场景 |

**为什么 RocketMQ 延迟更低？**

- 优化了端到端延迟
- 支持同步刷盘（低延迟模式）
- 消息立即写入，延迟更可控

### 2.3 消息堆积能力

| MQ | 堆积能力 | 说明 |
|---|---|---|
| **Kafka** | 无限（依赖磁盘） | 消息保留策略可配置 |
| **RocketMQ** | 百万级 | 堆积能力优秀 |
| **RabbitMQ** | 数十万级 | 堆积能力有限 |

## 三、功能对比

### 3.1 核心功能对比表

| 功能 | Kafka | RabbitMQ | RocketMQ |
|---|---|---|---|
| **延迟消息** | ❌ 不支持 | ✅ 通过插件 | ✅ 原生支持 |
| **事务消息** | ✅ 有限支持 | ❌ 不支持 | ✅ 原生支持 |
| **优先级队列** | ❌ 不支持 | ✅ 原生支持 | ❌ 不支持 |
| **死信队列** | ❌ 需手动实现 | ✅ 原生支持 | ✅ 支持 |
| **消息回溯** | ✅ 支持 | ❌ 不支持 | ✅ 支持 |
| **消息过滤** | ✅ 客户端过滤 | ✅ 服务端过滤 | ✅ 服务端过滤 |
| **消息重试** | ✅ 支持 | ✅ 支持 | ✅ 支持 |
| **顺序消息** | ✅ 分区内有序 | ✅ 单队列有序 | ✅ 支持 |

### 3.2 延迟消息对比

**RocketMQ 原生支持**：

```java
// RocketMQ 延迟消息
// 支持 18 个延迟级别：1s、5s、10s、30s、1m、2m、...、2h
Message message = new Message("delay-topic", body.getBytes());
message.setDelayTimeLevel(3);  // 延迟 10 秒
producer.send(message);
```

**RabbitMQ 通过插件**：

```java
// RabbitMQ 延迟消息：通过 TTL + DLX 实现
// 需要安装 rabbitmq_delayed_message_exchange 插件
Map<String, Object> headers = new HashMap<>();
headers.put("x-delay", 10000);  // 延迟 10 秒
channel.basicPublish("exchange", "routingKey", properties, body);
```

**Kafka 不支持**：需要借助外部系统（如 Scheduled Topic）或自行实现。

### 3.3 事务消息对比

**RocketMQ 原生支持**：

```java
// RocketMQ 事务消息
producer.sendMessageInTransaction(msg, new TransactionListener() {
    @Override
    public LocalTransactionState executeLocalTransaction(Message msg, Object arg) {
        // 1. 执行本地事务
        orderService.createOrder(order);
        // 2. 返回事务状态
        return LocalTransactionState.COMMIT_MESSAGE;
    }

    @Override
    public LocalTransactionState checkLocalTransaction(MessageExt msg) {
        // 3. 回查本地事务状态
        if (orderService.exists(msg.getKeys())) {
            return LocalTransactionState.COMMIT_MESSAGE;
        }
        return LocalTransactionState.ROLLBACK_MESSAGE;
    }
}, order.getId());
```

**Kafka 有限支持**（0.11+）：

```java
// Kafka 事务消息（主要用于 exactly-once 语义）
producer.initTransactions();
producer.beginTransaction();
producer.send(record);
producer.sendOffsetsToTransaction(offsets, groupId);
producer.commitTransaction();
```

**RabbitMQ 不支持**：需要通过 outbox 模式或 Saga 模式自行实现。

## 四、可靠性对比

### 4.1 持久化机制

| MQ | 持久化 | 说明 |
|---|---|---|
| **Kafka** | ✅ 强制持久化 | 消息追加到日志文件 |
| **RocketMQ** | ✅ 支持 | 同步/异步刷盘 |
| **RabbitMQ** | ✅ 支持 | 需开启 durable |

### 4.2 副本机制

| MQ | 副本机制 | 同步策略 |
|---|---|---|
| **Kafka** | ISR | 同步到 ISR 中的所有副本 |
| **RocketMQ** | Master-Slave / Dledger | 同步双写 / 异步复制 |
| **RabbitMQ** | 镜像队列 | 镜像到所有节点 |

### 4.3 确认机制

| MQ | 确认机制 | 说明 |
|---|---|---|
| **Kafka** | Offset | 提交 Offset 表示已消费 |
| **RocketMQ** | ACK | 手动 ACK |
| **RabbitMQ** | ACK | 手动 ACK，支持 NACK |

## 七、典型场景选型

| 场景 | 推荐 | 原因 |
|---|---|---|
| **日志收集、ELK** | Kafka | 高吞吐、消息回溯 |
| **流处理（Kafka Streams、Flink）** | Kafka | 完美集成 |
| **电商交易消息** | RocketMQ | 事务消息、延迟消息 |
| **复杂路由（发布订阅）** | RabbitMQ | Exchange 灵活路由 |
| **异步任务队列** | RabbitMQ | 功能丰富、易用 |
| **大数据实时分析** | Kafka | 高吞吐、生态完善 |
| **金融级可靠性** | RocketMQ | 原生事务消息 |
| **微服务解耦** | RabbitMQ / RocketMQ | 延迟消息支持好 |

## 八、面试追问预判

### 追问一：如何保证消息不丢失？

需要三个环节都做好：

```java
// 1. 生产者端
props.put("acks", "all");              // 所有副本确认
props.put("retries", 3);               // 重试 3 次
props.put("enable.idempotence", true); // 开启幂等

// 2. Broker 端
replication.factor = 3;                // 3 副本
min.insync.replicas = 2;               // 最小 2 个副本

// 3. 消费者端
enable.auto.commit = false;            // 手动提交
consumer.commitSync();                 // 处理完成后提交
```

### 追问二：Kafka 和 RocketMQ 有什么区别？

| 维度 | Kafka | RocketMQ |
|---|---|---|
| **设计定位** | 日志收集、流处理 | 电商交易、金融消息 |
| **事务消息** | 有限支持 | 原生支持 |
| **延迟消息** | 不支持 | 原生支持 |
| **顺序消息** | 分区内有序 | 分区内有序 |
| **消息回溯** | 支持 | 支持 |

### 追问三：如何根据业务选择 MQ？

1. **高吞吐 + 日志场景**：选择 Kafka
2. **复杂路由 + 企业级**：选择 RabbitMQ
3. **事务消息 + 金融场景**：选择 RocketMQ
