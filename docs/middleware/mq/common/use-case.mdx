# 消息队列应用场景

## 场景切入：为什么你的系统需要消息队列？

双十一零点，订单量瞬间飙升到每秒 10 万笔。如果每笔订单都直接操作数据库，数据库会瞬间崩溃。

但如果我们换一种方式：用户下单后，系统将订单消息发送到消息队列，然后立即返回「下单成功」。订单消费者从队列中拉取消息，缓慢而稳定地处理。这样一来，数据库的压力从「瞬间峰值」变成了「平滑处理」。

这就是消息队列最核心的价值。

## 一、异步解耦

### 1.1 什么是异步解耦

异步解耦是指将同步调用改为异步消息传递，让服务之间不直接依赖，实现松耦合。

```
同步调用模式：
┌──────────┐    同步调用    ┌──────────┐    同步调用    ┌──────────┐
│          │ ───────────> │          │ ───────────> │          │
│ 订单系统  │              │ 库存系统  │              │ 支付系统  │
│          │ <─────────── │          │ <─────────── │          │
└──────────┘    等待返回   └──────────┘    等待返回   └──────────┘
     │
     │ 问题：任一系统延迟或宕机，整个链路崩溃
     ▼

异步消息模式：
┌──────────┐    发送消息    ┌──────────┐
│          │ ───────────> │          │ ──> 库存系统（异步）
│ 订单系统  │              │   MQ     │ ──> 支付系统（异步）
│          │ <─────────── │          │ ──> 物流系统（异步）
└──────────┘   立即返回    └──────────┘
     │
     │ 优势：订单系统不依赖其他系统，响应快
     ▼
```

### 1.2 实际案例：电商订单处理

```java
// 同步模式：用户等待所有操作完成
public OrderResult placeOrder(Order order) {
    // 1. 创建订单
    saveOrder(order);              // 耗时 50ms

    // 2. 扣减库存（必须等待）
    deductInventory(order);        // 耗时 100ms

    // 3. 扣减余额（必须等待）
    deductBalance(order);          // 耗时 100ms

    // 4. 发送短信（必须等待）
    sendSms(order);               // 耗时 200ms

    // 5. 发送邮件（必须等待）
    sendEmail(order);              // 耗时 200ms

    // 总耗时：650ms
    return new OrderResult();
}

// 异步模式：只处理核心逻辑
public Mono<OrderResult> placeOrder(Order order) {
    return saveOrder(order)
        .flatMap(result -> deductInventory(order))   // 100ms
        .flatMap(result -> deductBalance(order))      // 100ms
        .flatMap(result -> {
            // 以下操作异步化，不阻塞主流程
            mqTemplate.send("sms-topic", order.getPhone(), "下单成功");
            mqTemplate.send("email-topic", order.getEmail(), "订单确认");
            mqTemplate.send("log-topic", order, "order_created");
            mqTemplate.send("analytics-topic", order, "order_stats");
            return Mono.just(new OrderResult());
        });
    // 总耗时：250ms，提升 2.6 倍
}
```

### 1.3 异步解耦的优势

| 优势 | 说明 |
|---|---|
| **响应速度快** | 核心流程不等待非核心操作 |
| **系统解耦** | 各系统独立，不互相影响 |
| **容错性强** | 下游系统宕机不影响上游 |
| **可扩展** | 易于增加新的消费者 |

## 二、削峰填谷

### 2.1 什么是削峰填谷

削峰填谷是指将瞬时流量高峰分散到较长时间段内处理，避免系统过载。

```
无 MQ 的流量模型：
                    ↑
                    │ 峰值流量
                    │ (10万/秒)
                    │
    ───────────────────────────────────────────────────────────────>
                                    时间
                    ↑
                    │ 系统崩溃点
                    │ (1万/秒)
                    ▼

有 MQ 的流量模型：
                    ↑
                    │ 生产速度
                    │ (10万/秒)
                    │
    ───────────────────────────────────────────────────────────────>
                                    时间

                    ↑
                    │ 消费速度
                    │ (1万/秒)
                    │
    ───────────────────────────────────────────────────────────────>
                                    时间

    ┌──────────────────────────────────────┐
    │           消息队列缓冲                 │  堆积消息
    └──────────────────────────────────────┘
```

### 2.2 实际案例：秒杀系统

```java
// 秒杀场景：10000 人同时抢购 100 件商品

// 无 MQ 的问题
public void seckill(Long userId, Long productId) {
    // 1. 查询库存（数据库压力巨大）
    Integer stock = inventoryService.getStock(productId);

    // 2. 判断库存（并发问题）
    if (stock > 0) {
        // 3. 扣减库存（数据库锁竞争）
        inventoryService.decreaseStock(productId);

        // 4. 创建订单
        orderService.createOrder(userId, productId);

        // 5. 返回结果
        return "秒杀成功";
    }

    return "秒杀失败";
}

// 有 MQ 的方案
public String seckill(Long userId, Long productId) {
    // 1. 发送秒杀请求到 MQ（轻量级操作）
    mqTemplate.send("seckill-topic", userId, productId);

    // 2. 立即返回，用户无需等待
    return "请求已受理，请等待结果";
}

// 后台消费者慢慢处理
@KafkaListener(topics = "seckill-topic")
public void processSeckill(Long userId, Long productId) {
    // 1. 查询库存（顺序执行，无并发问题）
    Integer stock = inventoryService.getStock(productId);

    if (stock > 0) {
        // 2. 扣减库存
        inventoryService.decreaseStock(productId);

        // 3. 创建订单
        orderService.createOrder(userId, productId);
    }
}
```

### 2.3 削峰填谷的配置建议

| 配置项 | 说明 | 推荐值 |
|---|---|---|
| `max.poll.records` | 消费者每次拉取数量 | 根据消费能力调整 |
| `concurrency` | 消费者线程数 | 10-20 |
| `pull.interval` | 拉取间隔 | 100-500ms |

## 三、数据收集

### 3.1 什么是数据收集

数据收集是指将分散在各个服务中的日志、监控数据、用户行为数据等发送到消息队列，统一处理和分析。

```
传统架构：
┌──────────┐    ┌──────────┐    ┌──────────┐
│ 服务 A   │    │ 服务 B   │    │ 服务 C   │
│ 日志文件 │    │ 日志文件 │    │ 日志文件 │
└────┬─────┘    └────┬─────┘    └────┬─────┘
     │                │                │
     ▼                ▼                ▼
┌─────────────────────────────────────────┐
│           Logstash / Filebeat           │
│           （每个服务都要部署）            │
└────────────────────┬────────────────────┘
                     │
                     ▼
              ┌──────────────┐
              │ Elasticsearch│
              └──────────────┘

MQ 架构：
┌──────────┐    ┌──────────┐    ┌──────────┐
│ 服务 A   │    │ 服务 B   │    │ 服务 C   │
└────┬─────┘    └────┬─────┘    └────┬─────┘
     │                │                │
     ▼                ▼                ▼
┌──────────┐    ┌──────────┐    ┌──────────┐
│  Kafka   │    │  Kafka   │    │  Kafka   │
│ Producer │    │ Producer │    │ Producer │
└────┬─────┘    └────┬─────┘    └────┬─────┘
     │                │                │
     └────────────────┼────────────────┘
                      ▼
               ┌──────────────┐
               │ Kafka Broker │
               └──────┬───────┘
                      │
     ┌────────────────┼────────────────┐
     │                │                │
     ▼                ▼                ▼
┌──────────┐    ┌──────────┐    ┌──────────┐
│ Logstash │    │ 实时计算 │    │ 数据仓库  │
│ (ELK)   │    │ (Flink)  │    │ (Hive)   │
└──────────┘    └──────────┘    └──────────┘
```

### 3.2 实际案例：用户行为埋点

```java
// 传统方式：直接发送到分析系统
public void click(String userId, String itemId) {
    // 业务处理...
    doClick(userId, itemId);

    // 直接发送埋点数据（增加延迟）
    analyticsService.send(new ClickEvent(userId, itemId, "click"));
}

// MQ 方式：异步发送埋点数据
public void click(String userId, String itemId) {
    // 业务处理...
    doClick(userId, itemId);

    // 异步发送埋点，不阻塞主流程
    mqTemplate.asyncSend("user-behavior-topic", new ClickEvent(userId, itemId, "click"));
}

// 消费者批量处理
@KafkaListener(topics = "user-behavior-topic")
public void processBehavior(List<ClickEvent> events) {
    // 批量发送到分析系统
    analyticsService.batchSend(events);
}
```

### 3.3 数据收集的优势

| 优势 | 说明 |
|---|---|
| **解耦** | 日志服务不影响业务系统 |
| **缓冲** | 削峰填谷，避免分析系统过载 |
| **复用** | 一份数据，多种消费（ELK、流计算、数据仓库） |
| **可靠** | 消息持久化，不丢失数据 |

## 四、三大场景对比

| 场景 | 核心价值 | 典型用例 | MQ 类型 |
|---|---|---|---|
| **异步解耦** | 提升响应速度，降低耦合 | 订单完成后通知库存、支付、物流 | RabbitMQ、RocketMQ |
| **削峰填谷** | 应对流量高峰，保护后端 | 秒杀、抢购、突发流量 | Kafka、RocketMQ |
| **数据收集** | 海量数据统一处理 | 日志收集、监控埋点、行为分析 | Kafka |

## 五、面试追问预判

### 追问一：异步解耦会不会导致数据不一致？

异步解耦确实可能带来数据一致性问题，需要通过以下方式解决：

1. **事务消息**：RocketMQ 支持事务消息，保证本地事务和消息发送的原子性
2. **最终一致性**：接受短暂的数据不一致，通过补偿机制最终一致
3. **消息对账**：定期对账，发现并修复不一致的数据

### 追问二：削峰填谷会不会导致消息堆积？

如果消费能力持续小于生产速度，消息会不断堆积：

- **监控告警**：设置消息堆积阈值，及时告警
- **临时扩容**：增加消费者实例，加速消费
- **降级处理**：优先处理重要消息，跳过非关键消息

### 追问三：数据收集场景下 Kafka 的优势是什么？

Kafka 在数据收集场景有以下优势：

- **超高吞吐量**：适合海量日志和埋点数据
- **消息回溯**：可以从任意 offset 开始消费
- **长期存储**：消息可以保留多天，方便回溯分析
- **生态完善**：与 Flink、Spark 等流处理框架无缝集成
