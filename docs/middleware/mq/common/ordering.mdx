# 如何保证消息的顺序性？

## 场景切入：顺序乱了，系统就乱了

订单流程中，消息的顺序至关重要：

```
正确的顺序：
1. 创建订单（订单状态：待支付）
2. 支付成功（订单状态：已支付）
3. 发货（订单状态：已发货）
4. 收货（订单状态：已完成）

乱序的后果：
1. 发货（订单还不存在 → 报错）
2. 创建订单
3. 支付成功
4. 收货（订单还没发货 → 状态混乱）
```

如果消息乱序，可能导致业务逻辑错误、数据不一致等问题。那么，消息队列如何保证消息的顺序性？

---

## 一、顺序性的定义

### 1.1 全局顺序 vs 局部顺序

| 类型 | 定义 | 保证范围 |
|---|---|---|
| **全局顺序** | 所有消息严格按照发送顺序消费 | 整个 Topic |
| **局部顺序** | 同一 Key 的消息严格按照发送顺序消费 | 同一 Partition 或同一 Queue |

```
全局顺序（单分区）：
┌─────────────────────────────────────┐
│  Partition 0                         │
│  [消息1] → [消息2] → [消息3] → ...  │
│  消费者按顺序消费                      │
└─────────────────────────────────────┘

局部顺序（多分区）：
┌─────────────────────────────────────┐
│  Partition 0  │  Partition 1         │
│  [key=A 消息1]│  [key=B 消息1]     │
│  [key=A 消息2]│  [key=B 消息2]     │
│  [key=A 消息3]│                     │
│                                     │
│  key=A 的消息按顺序消费               │
│  key=B 的消息按顺序消费               │
│  但 key=A 和 key=B 之间无顺序关系     │
└─────────────────────────────────────┘
```

### 1.2 顺序性的代价

顺序性和吞吐量是矛盾的：

| 配置 | 顺序性 | 吞吐量 |
|---|---|---|
| 单分区 | ✅ 强 | ❌ 低 |
| 多分区 | ❌ 弱 | ✅ 高 |

在实际业务中，通常选择**局部顺序**（同一业务 ID 的消息有序），而不是全局顺序。

## 二、Kafka 如何保证顺序性

### 2.1 分区机制

Kafka 通过分区机制实现局部顺序：

- 发送到同一分区的消息，保证顺序
- 不同分区的消息，不保证顺序

```
分区与顺序的关系：

消息 1 (key=order-1) ──┬── 路由到 Partition 0 ──→ [消息 1, 消息 3, 消息 5]
消息 2 (key=order-2) ──┤
消息 3 (key=order-1) ──┤  路由到 Partition 1 ──→ [消息 2, 消息 4]
消息 4 (key=order-2) ──┤
消息 5 (key=order-1) ──┘

Partition 0：消息 1 → 消息 3 → 消息 5（有序）
Partition 1：消息 2 → 消息 4（有序）
不同分区之间无顺序关系
```

### 2.2 生产者分区策略

**方式一：使用消息 Key 路由**

```java
// 同一订单的消息使用相同的 Key
producer.send(new ProducerRecord<>(
    "order-topic",
    "order-123",  // 订单 ID 作为 Key
    "创建订单"       // 消息体
));

// 同一订单的后续消息也使用相同的 Key
producer.send(new ProducerRecord<>(
    "order-topic",
    "order-123",  // 保证路由到同一分区
    "支付成功"
));
```

**方式二：自定义分区器**

```java
// 自定义分区器：按业务 ID 路由
public class OrderPartitioner implements Partitioner {

    @Override
    public int partition(String topic, Object key, byte[] keyBytes,
                        Object value, byte[] valueBytes,
                        Cluster cluster) {

        // 从 Key 中提取业务 ID
        String businessId = extractBusinessId(key);

        // 获取分区数
        List<PartitionInfo> partitions = cluster.partitionsForTopic(topic);
        int numPartitions = partitions.size();

        // 使用哈希确保同一业务 ID 到同一分区
        return Math.abs(businessId.hashCode()) % numPartitions;
    }

    private String extractBusinessId(Object key) {
        // 提取业务 ID
        return key.toString();
    }
}

// 配置使用自定义分区器
props.put("partitioner.class", "com.example.OrderPartitioner");
```

### 2.3 消费者有序消费

**单线程顺序消费**：

```java
// 单线程消费：保证单分区有序
@KafkaListener(topics = "order-topic")
public void consume(ConsumerRecord<String, String> record) {
    // 单线程处理，天然有序
    processOrder(record.value());
}
```

**多线程消费的问题**：

```java
// 多线程消费：可能乱序（不推荐）
@KafkaListener(topics = "order-topic", concurrency = "3")
public void consume(ConsumerRecord<String, String> record) {
    // 多线程并发处理，可能乱序
    executor.submit(() -> processOrder(record.value()));
}

// 问题：同一订单的消息可能被不同的线程同时处理
// 线程1 处理消息1（创建订单）
// 线程2 处理消息2（支付成功，可能在创建订单之前处理）
```

### 2.4 保证局部有序的配置

```java
// 配置：单消费者 + 同一 Key 到同一分区
Properties props = new Properties();
props.put("bootstrap.servers", "localhost:9092");
props.put("key.serializer", "org.apache.kafka.common.serialization.StringSerializer");
props.put("value.serializer", "org.apache.kafka.common.serialization.StringSerializer");

// KafkaConsumer 配置
props.put("group.id", "order-consumer-group");
props.put("enable.auto.commit", "false");
props.put("max.poll.records", "1");  // 减少每次拉取数量，降低并发
```

## 三、RocketMQ 如何保证顺序性

### 3.1 顺序消息类型

RocketMQ 支持两种顺序消息：

| 类型 | 说明 | 实现方式 |
|---|---|---|
| **全局顺序** | Topic 中所有消息按发送顺序消费 | 单 Queue + 单 Consumer |
| **分区顺序** | 同一 MessageQueue 中的消息按发送顺序消费 | MessageQueueSelector |

### 3.2 生产者顺序发送

```java
// RocketMQ 顺序消息发送
public void sendOrderMessage(Order order) {
    // 创建消息
    Message msg = new Message("order-topic", order.toString().getBytes());

    // 发送顺序消息
    producer.send(msg, new MessageQueueSelector() {
        @Override
        public MessageQueue select(List<MessageQueue> mqs, Message msg, Object arg) {
            // 按订单 ID 选择 Queue，保证同一订单的消息到同一 Queue
            Long orderId = (Long) arg;
            int index = (int) (orderId % mqs.size());
            return mqs.get(index);
        }
    }, order.getId());  // 订单 ID 作为参数
}
```

### 3.3 消费者顺序消费

```java
// RocketMQ 顺序消息消费
@RocketMQMessageListener(
    topic = "order-topic",
    consumerGroup = "order-consumer-group",
    consumeMode = ConsumeMode.ORDERLY  // 顺序消费模式
)
public class OrderConsumer implements RocketMQListener<Message> {

    @Override
    public void onMessage(Message message) {
        // 单线程顺序处理
        processOrder(message);
    }
}
```

## 四、RabbitMQ 如何保证顺序性

### 4.1 单队列保证顺序

RabbitMQ 的队列本身是有序的，只要满足以下条件：

- **单队列**：消息只进入一个队列
- **单消费者**：队列只有一个消费者

```
RabbitMQ 顺序保证：

┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│   消息 1    │───>│   消息 2    │───>│   消息 3    │
└─────────────┘    └─────────────┘    └─────────────┘
                         │
                         ▼
                  ┌─────────────┐
                  │   Queue     │
                  │  (有序 FIFO)│
                  └──────┬──────┘
                         │
                         ▼
                  ┌─────────────┐
                  │  Consumer   │───> 顺序消费
                  └─────────────┘
```

### 4.2 多队列的顺序保证

如果需要多队列实现并行处理，可以采用以下方案：

```
多队列 + 按 Key 路由：

Producer                               Consumer Group
    │                                       │
    │  消息 1 (key=A) ──> Exchange ──> Queue A ──> Consumer 1
    │  消息 2 (key=A) ──> Exchange ──> Queue A ──> Consumer 1
    │  消息 3 (key=B) ──> Exchange ──> Queue B ──> Consumer 2
    │
    │  同一 key 的消息进入同一 Queue，保证有序
    ▼
```

### 4.3 队列配置

```java
// RabbitMQ 顺序消费配置
// 1. 使用单一消费者
channel.basicQos(1);  // 每次只给消费者一条消息

// 2. 手动 ACK，处理完成后确认
channel.basicConsume("order-queue", false, new DefaultConsumer(channel) {
    @Override
    public void handleDelivery(String consumerTag, Envelope envelope,
                             AMQP.BasicProperties properties, byte[] body) {
        try {
            processMessage(body);
            channel.basicAck(envelope.getDeliveryTag(), false);  // 确认
        } catch (Exception e) {
            channel.basicNack(envelope.getDeliveryTag(), false, true);  // 重试
        }
    }
});
```

## 五、常见问题与解决方案

### 5.1 问题一：多线程消费导致乱序

```
问题：消费者使用线程池并发处理，同一业务 ID 的消息被不同线程处理

解决方案：
1. 使用单线程消费
2. 使用 HashMap 按业务 ID 串行化处理
```

```java
// 解决方案：按业务 ID 串行化处理
private ConcurrentHashMap<String, ExecutorService> executors = new ConcurrentHashMap<>();

@KafkaListener(topics = "order-topic")
public void consume(ConsumerRecord<String, String> record) {
    String orderId = record.key();

    // 按订单 ID 获取线程池，保证同一订单串行处理
    ExecutorService executor = executors.computeIfAbsent(orderId,
        k -> Executors.newSingleThreadExecutor());

    executor.submit(() -> {
        processOrder(record.value());
    });
}
```

### 5.2 问题二：重试导致乱序

```
问题：消息处理失败后重试，可能导致乱序

场景：消息 1、2、3 依次到达
- 消息 1 处理成功
- 消息 2 处理失败，重试
- 消息 3 处理成功
- 消息 2 重试后成功
结果：消息 2 和 3 的处理顺序与发送顺序不一致
```

解决方案：

1. **顺序重试**：将重试的消息放回队列尾部，而不是立即重试
2. **消息表**：记录消息状态，重试时查询依赖关系
3. **版本号**：消息携带版本号，消费前检查版本连续性

### 5.3 问题三：Rebalance 导致乱序

```
问题：Consumer Group Rebalance 时，同一分区的消息可能被不同 Consumer 处理

解决方案：
1. 避免频繁的 Rebalance（合理配置心跳参数）
2. 使用静态成员（Static Membership）
3. 使用协作式再平衡（Cooperative Rebalance）
```

## 六、面试追问预判

### 追问一：Kafka 如何保证消息顺序？

Kafka 通过分区机制保证局部顺序：

- 同一分区的消息有序
- 不同分区的消息无序
- 同一 Key 的消息路由到同一分区

### 追问二：如何实现全局有序？

要实现全局有序，必须：

- 使用单分区（失去了并行处理能力）
- 使用单消费者

这会严重影响性能，通常不推荐。

### 追问三：多线程消费如何保证有序？

可以使用以下方案：

1. **单线程消费**：最简单的方案
2. **按业务 ID 串行化**：同一业务 ID 的消息串行处理
3. **有序消息处理框架**：使用支持有序消息的框架

### 追问四：重试机制如何保证有序？

常见方案：

1. **顺序重试**：失败消息放回队列尾部
2. **消息表**：记录处理状态，根据依赖关系处理
3. **版本号检查**：检查消息版本号是否连续
