# 消息积压处理方案

## 场景切入：积压如山，如何快速消化？

凌晨 2 点，你被报警电话叫醒：消息队列堆积了 100 万条未消费消息。消费者还在正常运行，但就是消费不过来。

这是一个典型的消息积压问题。消息积压不仅影响系统性能，还可能导致消息过期、数据不一致等问题。如何快速定位原因并解决积压？

## 一、消息积压的原因分析

### 1.1 常见原因

| 原因 | 表现 | 排查方法 |
|---|---|---|
| **消费者能力不足** | 消费速度 `<` 生产速度 | 查看消费者 CPU、内存 |
| **消费者故障** | 某个消费者不工作 | 查看消费者日志 |
| **消费者死循环** | CPU 100% | 查看线程栈 |
| **消费者卡住** | 长时间不 poll | 查看 poll 超时 |
| **消费者异常退出** | Consumer 消失 | 查看 Consumer Group |
| **分区不均衡** | 部分分区堆积 | 查看分区分配 |
| **消息格式错误** | 消费失败 | 查看错误日志 |

### 1.2 排查步骤

```
排查步骤：

1. 查看消息堆积量
   - Kafka: kafka-consumer-groups.sh --describe
   - RabbitMQ: rabbitmqctl list_queues
   - RocketMQ: mqadmin consumerProgress

2. 查看消费者状态
   - Consumer 数量是否正常
   - Consumer 是否都在运行

3. 查看消费者日志
   - 是否有异常
   - 消费耗时是否正常

4. 查看消费者指标
   - CPU、内存、网络
   - 消费耗时
   - 错误率
```

```bash
# Kafka 查看消费者状态
kafka-consumer-groups.sh --bootstrap-server localhost:9092 \
  --group order-consumer-group \
  --describe

# 输出示例：
# GROUP                TOPIC           PARTITION  CURRENT-OFFSET  LOG-END-OFFSET  LAG
# order-consumer-group order-topic     0          10000            15000           5000
# order-consumer-group order-topic     1          10000            10000           0
# order-consumer-group order-topic     2          10000            20000           10000
# order-consumer-group order-topic     3          10000            10000           0

# LAG 列表示堆积量：分区 2 堆积了 10000 条消息
```

## 二、消息积压的解决方案

### 2.1 方案一：临时扩容消费者

最直接的方法，增加消费者实例加速消费。

```
扩容前：
┌─────────────────────────────────────┐
│  Topic: order-topic (4 个分区)       │
│                                     │
│  ┌────────┐ ┌────────┐             │
│  │ Part 0 │ │ Part 1 │ ← Consumer 1 │
│  └────────┘ └────────┘             │
│  ┌────────┐ ┌────────┐             │
│  │ Part 2 │ │ Part 3 │ ← Consumer 2 │
│  └────────┘ └────────┘             │
└─────────────────────────────────────┘

扩容后（6 个分区）：
┌─────────────────────────────────────┐
│  Topic: order-topic (4 个分区)       │
│                                     │
│  ┌────────┐ ┌────────┐ ┌────────┐  │
│  │ Part 0 │ │ Part 1 │ │ Part 2 │  │
│  └────────┘ └────────┘ └────────┘  │
│       │         │         │         │
│  ┌────────┐ ┌────────┐ ┌────────┐  │
│  │Consumer1│ │Consumer2│ │Consumer3│  │
│  └────────┘ └────────┘ └────────┘  │
└─────────────────────────────────────┘
```

```bash
# Kubernetes 扩容消费者
kubectl scale deployment order-consumer --replicas=6

# 或使用 HPA 自动扩容
kubectl autoscale deployment order-consumer \
  --min=3 --max=10 \
  --cpu-percent=70
```

### 2.2 方案二：增加消费线程

如果消费者代码没有充分利用多核 CPU，可以增加消费线程。

```java
// 修改前：单线程消费
@KafkaListener(topics = "order-topic")
public void consume(ConsumerRecord<String, String> record) {
    processOrder(record.value());
}

// 修改后：多线程消费
@KafkaListener(topics = "order-topic")
public void consume(ConsumerRecord<String, String> record) {
    executor.submit(() -> processOrder(record.value()));
}

// 配置线程池
@Bean
public ExecutorService executor() {
    return Executors.newFixedThreadPool(10);
}
```

:::warning 注意
多线程消费可能带来并发问题，如数据库连接池耗尽、线程安全问题等。需要确保业务代码是线程安全的。
:::

### 2.3 方案三：消息迁移

如果积压太多，扩容来不及，可以将消息迁移到新 Topic，用新的消费者消费。

```
消息迁移方案：

原始链路：
Producer → Topic-A → Consumer-A（消费慢）

临时方案：
Producer → Topic-A → Consumer-A（暂停消费）
           ↓ 迁移
        Topic-B → Consumer-B（快速消费）
```

```java
// 消息迁移示例
public class MessageMigrator {

    public void migrate(String sourceTopic, String targetTopic) {
        // 1. 创建新消费者，订阅源 Topic
        KafkaConsumer<String, String> sourceConsumer = new KafkaConsumer<>(props);
        sourceConsumer.subscribe(Arrays.asList(sourceTopic));

        // 2. 创建新生产者，发送到目标 Topic
        KafkaProducer<String, String> targetProducer = new KafkaProducer<>(props);

        // 3. 消费并发送
        while (true) {
            ConsumerRecords<String, String> records = sourceConsumer.poll(Duration.ofSeconds(1));

            for (ConsumerRecord<String, String> record : records) {
                ProducerRecord<String, String> newRecord = new ProducerRecord<>(
                    targetTopic, record.key(), record.value());
                targetProducer.send(newRecord);
            }

            // 4. 记录迁移进度
            if (records.count() == 0) {
                break;  // 迁移完成
            }
        }

        targetProducer.flush();
        sourceConsumer.close();
        targetProducer.close();
    }
}
```

### 2.4 方案四：优先处理重要消息

在积压严重时，优先处理重要消息，跳过非关键消息。

```java
// 消息分级
@KafkaListener(topics = "order-topic")
public void consume(ConsumerRecord<String, String> record) {
    OrderMessage message = parseMessage(record);

    // 高优先级消息
    if (message.getPriority() == 1) {
        processOrder(message);  // 立即处理
        return;
    }

    // 低优先级消息，加入队列
    lowPriorityQueue.offer(message);
}

// 低优先级消息定时处理
@Scheduled(fixedDelay = 5000)
public void processLowPriority() {
    OrderMessage message = lowPriorityQueue.poll();
    if (message != null) {
        processOrder(message);
    }
}
```

### 2.5 方案五：调整消费逻辑

如果消费逻辑本身耗时过长，可以考虑优化。

```java
// 优化前：同步处理
@KafkaListener(topics = "order-topic")
public void consume(ConsumerRecord<String, String> record) {
    // 1. 查询数据库
    Order order = orderMapper.selectById(record.value());

    // 2. 更新状态
    order.setStatus("PROCESSED");
    orderMapper.updateById(order);

    // 3. 发送通知
    notificationService.send(order);
}

// 优化后：批量处理
@KafkaListener(topics = "order-topic")
public void consume(List<ConsumerRecord<String, String>> records) {
    // 1. 批量查询
    List<String> ids = records.stream()
        .map(r -> r.value())
        .collect(Collectors.toList());
    Map<String, Order> orderMap = orderMapper.selectByIds(ids);

    // 2. 批量处理
    orderMap.values().forEach(order -> order.setStatus("PROCESSED"));
    orderMapper.batchUpdate(orderMap.values());

    // 3. 批量发送通知（异步）
    notificationService.sendAsync(orderMap.values());
}
```

## 三、预防措施

### 3.1 监控告警

配置消息堆积监控，及时发现积压。

```yaml
# Prometheus 告警规则
groups:
- name: kafka-consumer-alerts
  rules:
  - alert: KafkaConsumerLagHigh
    expr: kafka_consumer_lag_sum > 100000
    for: 5m
    labels:
      severity: warning
    annotations:
      summary: "Kafka Consumer Lag 过高"
      description: "消费者 lag 超过 100000，请及时处理"

  - alert: KafkaConsumerLagCritical
    expr: kafka_consumer_lag_sum > 500000
    for: 5m
    labels:
      severity: critical
    annotations:
      summary: "Kafka Consumer Lag 严重"
      description: "消费者 lag 超过 500000，需要紧急处理"
```

### 3.2 限流保护

在消费端增加限流保护，避免压垮下游系统。

```java
// 限流配置
@RateLimiter(name = "orderConsumer", fallbackMethod = "fallback")
@KafkaListener(topics = "order-topic")
public void consume(ConsumerRecord<String, String> record) {
    processOrder(record.value());
}

public void fallback(ConsumerRecord<String, String> record, Exception e) {
    // 限流后的降级处理
    log.warn("触发限流，消息 {} 被降级处理", record.key());
    // 可以将消息发送到降级 Topic 或写入本地文件
}
```

### 3.3 消费者健康检查

```java
// 健康检查
@Component
public class ConsumerHealthCheck {

    @Autowired
    private KafkaConsumer<String, String> consumer;

    public boolean isHealthy() {
        try {
            // 检查是否能正常 poll
            ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));
            return true;
        } catch (Exception e) {
            log.error("消费者健康检查失败", e);
            return false;
        }
    }
}
```

## 五、面试追问预判

### 追问一：如何避免消息积压？

预防措施：

1. **合理规划分区数**：分区数 >= 消费者数
2. **消费者能力预留**：消费者处理能力 > 生产速度
3. **监控告警**：及时发现积压
4. **限流保护**：保护下游系统

### 追问二：Kafka 和 RabbitMQ 的积压处理有什么区别？

| 维度 | Kafka | RabbitMQ |
|---|---|---|
| **积压表现** | Consumer Lag 增加 | 队列消息数增加 |
| **扩容方式** | 增加 Consumer 实例 | 增加 Consumer 实例 |
| **迁移方式** | 消费到新 Topic | 使用 shovel 插件 |
| **优先级** | 不支持原生 | 支持优先级队列 |

### 追问三：消息过期怎么办？

如果消息设置了 TTL，过期后会被删除：

1. **检查消息 TTL 配置**：确认消息是否有过期时间
2. **查看死信队列**：过期消息可能进入死信队列
3. **评估影响**：确定过期消息的业务影响
4. **补偿处理**：根据业务决定是否需要补偿
