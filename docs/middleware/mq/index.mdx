# 消息队列（MQ）：异步通信的基石

凌晨 2 点，你正准备上线新功能，订单系统突然收到一波流量高峰。数据库连接池瞬间耗尽，应用开始疯狂报错——这是每个工程师都可能遇到的噩梦。

但如果我们换一种设计：用户下单后，系统不是直接操作数据库，而是将订单信息发送到**消息队列**，然后立即返回「下单成功」。订单消费者从队列中拉取消息，缓慢而稳定地处理。这样一来，数据库的压力从「瞬间峰值」变成了「平滑处理」，用户也不需要等待漫长的数据库操作。

这就是消息队列最核心的价值：**削峰填谷、异步解耦、最终一致**。

```
传统模式：                         引入 MQ 后：
用户请求 → 直接处理 → 数据库       用户请求 → 写入 MQ → 立即返回
           ↑          ↑                      ↓
        峰值流量    数据库崩溃          消费者 → 数据库
                                              ↑
                                       平滑处理，数据库稳定
```

## 一、消息队列在架构中的定位

消息队列不是万能药，但它在特定场景下是最佳选择：

| 场景 | 问题 | MQ 如何解决 |
|---|---|---|
| 秒杀活动 | 瞬时流量冲击数据库 | 削峰填谷，消息排队 |
| 订单处理 | 下单后需要发短信、发邮件、记录日志 | 异步解耦，提升响应速度 |
| 系统集成 | 两个系统需要通信，耦合严重 | 发布订阅，松耦合通信 |
| 日志收集 | 海量日志需要集中处理 | 缓冲解耦，批量消费 |

## 二、三种主流消息队列对比

目前主流的消息队列有三种：Kafka、RabbitMQ、RocketMQ。它们各有特点：

| 维度 | Kafka | RabbitMQ | RocketMQ |
|---|---|---|---|
| **设计定位** | 日志收集、大数据、流处理 | 通用消息队列 | 电商、金融场景 |
| **吞吐量** | 百万级 QPS | 万级 QPS | 十万级 QPS |
| **消息延迟** | 毫秒级 | 微秒级（最低） | 毫秒级 |
| **消息顺序** | 分区内有序 | 单队列有序 | 分区内有序 |
| **事务消息** | 不支持原生 | 支持 | 支持原生 |
| **延迟消息** | 需插件 | 支持插件 | 支持原生 |
| **开发语言** | Scala + Java | Erlang | Java |

## 四、如何选择消息队列

```
开始选型
    │
    ▼
是否是日志收集 / 流处理场景？
    │
    ├── 是 → 选择 Kafka
    │
    └── 否 → 继续
            │
            ▼
        是否需要复杂路由？
            │
            ├── 是 → 选择 RabbitMQ
            │
            └── 否 → 继续
                    │
                    ▼
                是否需要事务消息或延迟消息？
                    │
                    ├── 是 → 选择 RocketMQ
                    │
                    └── 否 → 高吞吐优先选 Kafka，生态成熟
```

| 场景 | 推荐选择 | 原因 |
|---|---|---|
| 日志收集、ELK | Kafka | 高吞吐、消息回溯、生态完善 |
| 流处理（Kafka Streams、Flink） | Kafka | 完美集成、流处理支持 |
| 电商交易消息 | RocketMQ | 事务消息、金融级可靠性 |
| 复杂路由（发布订阅） | RabbitMQ | Exchange 灵活路由 |
| 延迟消息 | RocketMQ | 原生支持多级延迟 |
