# Dubbo SPI 机制与扩展点加载

你有没有想过这个问题：Dubbo 作为一个框架，它的核心功能（如负载均衡、序列化、协议）是如何做到「可插拔」的？

你可以在配置里写 `loadbalance="leastactive"`，也可以改成 `loadbalance="random"`。框架是怎么知道有哪些实现类的？又是如何加载的？

答案就是 **SPI（Service Provider Interface）** 机制。

这一节，我们深入理解 Dubbo SPI 的原理，对比 Java SPI，并学习如何自定义扩展点。

## 为什么需要 SPI？

### 传统硬编码的困境

如果你这样写代码：

```java
public class OrderServiceImpl {
    private final LoadBalancer loadBalancer = new RandomLoadBalancer();

    public Order getOrder(String orderId) {
        List<Server> servers = discovery.getServers();
        Server selected = loadBalancer.select(servers);
        // ...
    }
}
```

问题在哪？

1. **代码耦合**：业务代码直接依赖具体实现
2. **难以扩展**：想换一种负载均衡算法，需要改代码
3. **难以测试**：无法 mock 具体实现

### 依赖注入的改进

用依赖注入改进一下：

```java
public class OrderServiceImpl {
    private final LoadBalancer loadBalancer;

    public OrderServiceImpl(LoadBalancer loadBalancer) {
        this.loadBalancer = loadBalancer;
    }
}

// 使用时注入
OrderServiceImpl service = new OrderServiceImpl(new RandomLoadBalancer());
```

好了一点，但还是需要**手动管理**依赖关系。

### SPI 的解决方案

SPI（Service Provider Interface）是一种**服务发现机制**：

```
┌─────────────────────────────────────────────────────────┐
│                    SPI 工作原理                          │
│                                                          │
│   接口定义 ←─────────────── 扩展实现                      │
│       ↑                                                    │
│       │                                                   │
│   ┌───┴───────────────────────────────────────────┐   │
│   │              SPI 机制                           │   │
│   │                                               │   │
│   │  1. 扫描 META-INF/services/                    │   │
│   │  2. 读取接口实现类                              │   │
│   │  3. 动态加载实例                               │   │
│   │  4. 返回实现对象                               │   │
│   └───────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
```

SPI 的核心思想：**接口和实现分离，通过配置文件发现实现类**。

## Java SPI：标准但简陋

Java 自带 SPI 机制（`java.util.ServiceLoader`），是 SPI 的最早实践。

### Java SPI 示例

**Step 1：定义接口**

```java
// com.example.spi.LoadBalancer
public interface LoadBalancer {
    Server select(List<Server> servers);
}
```

**Step 2：实现接口**

```java
// com.example.spi.impl.RandomLoadBalancer
public class RandomLoadBalancer implements LoadBalancer {
    @Override
    public Server select(List<Server> servers) {
        return servers.get(ThreadLocalRandom.current().nextInt(servers.size()));
    }
}
```

**Step 3：在 META-INF/services/ 注册**

```
# 文件：META-INF/services/com.example.spi.LoadBalancer
# 内容：实现类的全限定名

com.example.spi.impl.RandomLoadBalancer
com.example.spi.impl.RoundRobinLoadBalancer
```

**Step 4：使用 ServiceLoader 加载**

```java
public class SpiDemo {
    public static void main(String[] args) {
        // 加载所有实现类
        ServiceLoader<LoadBalancer> loader = ServiceLoader.load(LoadBalancer.class);

        for (LoadBalancer balancer : loader) {
            System.out.println("加载实现: " + balancer.getClass().getName());
            // 可以使用这个实现
        }
    }
}
```

### Java SPI 的问题

Java SPI 看起来解决了问题，但实际上有严重缺陷：

**问题 1：每次调用都重新加载**

```java
ServiceLoader<LoadBalancer> loader = ServiceLoader.load(LoadBalancer.class);
// 每次迭代都会重新实例化！
for (LoadBalancer balancer : loader) {
    // 每次遍历都会创建新实例
}
```

**问题 2：无法按需加载**

无法指定加载某个具体实现，必须全部加载。

**问题 3：无依赖管理**

无法指定加载顺序，无法处理依赖关系。

**问题 4：无自适应机制**

无法根据条件自动选择合适的实现。

## Dubbo SPI：重新定义 SPI

Dubbo 在 Java SPI 基础上进行了深度扩展，解决了以上所有问题。

### Dubbo SPI 核心改进

```
┌─────────────────────────────────────────────────────────┐
│              Dubbo SPI vs Java SPI                      │
│                                                          │
│   ┌──────────────────┐    ┌──────────────────┐        │
│   │     Java SPI      │    │     Dubbo SPI     │        │
│   ├──────────────────┤    ├──────────────────┤        │
│   │  无法按需加载      │    │  按需加载实例     │        │
│   │  无 IOC 支持       │    │  支持 IOC 依赖注入 │        │
│   │  无 AOP 包装       │    │  支持 Wrapper 包装 │        │
│   │  无自适应选择       │    │  支持 Adaptive     │        │
│   │  无条件激活         │    │  支持 Activate    │        │
│   └──────────────────┘    └──────────────────┘        │
└─────────────────────────────────────────────────────────┘
```

### Dubbo SPI 目录结构

Dubbo 在三个目录扫描扩展实现：

```
META-INF/services/      # 兼容 Java SPI
META-INF/dubbo/        # Dubbo 内部扩展
META-INF/dubbo/internal/ # Dubbo 内部实现
```

### 配置文件格式

```
# 文件：META-INF/dubbo/com.example.LoadBalancer
# 格式：key = 实现类全限定名

random = com.example.spi.RandomLoadBalancer
roundrobin = com.example.spi.RoundRobinLoadBalancer
leastactive = com.example.spi.LeastActiveLoadBalancer
consistenthash = com.example.spi.ConsistentHashLoadBalancer
```

### 基本使用：@SPI 注解

**Step 1：接口标注 @SPI**

```java
@SPI
public interface LoadBalancer {
    /**
     * 选择一个服务器
     * @param servers 可用服务器列表
     * @return 选中的服务器
     */
    Server select(List<Server> servers);
}
```

**Step 2：实现类标注 @Activate**

```java
@Activate
public class RandomLoadBalancer implements LoadBalancer {

    @Override
    public Server select(List<Server> servers) {
        int index = ThreadLocalRandom.current().nextInt(servers.size());
        return servers.get(index);
    }
}
```

**Step 3：使用 ExtensionLoader 加载**

```java
public class DubboSpiDemo {
    public static void main(String[] args) {
        // 获取 ExtensionLoader
        ExtensionLoader<LoadBalancer> loader =
                ExtensionLoader.getExtensionLoader(LoadBalancer.class);

        // 按名称加载
        LoadBalancer random = loader.getExtension("random");

        // 获取默认实现（@SPI 注解指定的）
        LoadBalancer defaultBalancer = loader.getDefaultExtension();
    }
}
```

## Adaptive 扩展点：运行时动态适配

Dubbo SPI 最强大的特性是 **Adaptive 扩展**——根据运行时参数动态选择实现。

### 为什么需要 Adaptive？

考虑这个场景：序列化接口有很多实现（Protobuf、Hessian、JSON），客户端和服务端需要协商使用哪一种。

如果写死某个实现，不够灵活。Dubbo 的解决方案是：**生成一个适配器类，根据 URL 参数动态选择实现**。

### @Adaptive 注解

```java
@SPI
public interface Serializer {

    @Adaptive
    String serialize(Serializable obj);

    @Adaptive
    Object deserialize(byte[] data, Class<?> type);
}
```

### 自动生成适配器代码

Dubbo 编译时会生成类似这样的适配器：

```java
// Dubbo 生成的伪代码
public class Serializer$Adaptive implements Serializer {

    @Override
    public String serialize(Serializable obj) {
        // 从 URL 参数获取 serializer
        String serializerName = url.getParameter("serializer", "hessian");
        Serializer serializer =
            ExtensionLoader.getExtensionLoader(Serializer.class)
                .getExtension(serializerName);
        return serializer.serialize(obj);
    }

    @Override
    public Object deserialize(byte[] data, Class<?> type) {
        String serializerName = url.getParameter("serializer", "hessian");
        Serializer serializer =
            ExtensionLoader.getExtensionLoader(Serializer.class)
                .getExtension(serializerName);
        return serializer.deserialize(data, type);
    }
}
```

### 使用 Adaptive 扩展

```java
// 服务端配置
@DubboService(protocol = "dubbo", serialization = "protobuf")
public class OrderServiceImpl implements OrderService { }

// 客户端调用
// Dubbo 会根据 serialization 参数选择对应的序列化实现
```

### 方法级别的 Adaptive

```java
public interface Protocol {

    // 类级别的 Adaptive（使用接口名的 camelCase 作为 key）
    @Adaptive
    void export(URL url);

    // 方法级别的 Adaptive（使用方法名作为 key）
    @Adaptive("protocol")
    <T> T refer(Class<T> type, URL url);
}
```

## 自动包装（Wrapper）：AOP 增强

Dubbo SPI 支持**包装模式**，可以自动包装扩展点，添加通用逻辑。

### Wrapper 示例

```java
// 监控装饰器
public class MonitorLoadBalancerWrapper implements LoadBalancer {

    // 持有真实实现
    private final LoadBalancer loadBalancer;

    // 构造函数注入（Dubbo 会自动识别并注入）
    public MonitorLoadBalancerWrapper(LoadBalancer loadBalancer) {
        this.loadBalancer = loadBalancer;
    }

    @Override
    public Server select(List<Server> servers) {
        long start = System.currentTimeMillis();

        try {
            Server selected = loadBalancer.select(servers);
            recordMetrics(start, servers.size(), selected);
            return selected;
        } finally {
            // 记录耗时
        }
    }

    private void recordMetrics(long start, int total, Server selected) {
        // 上报监控指标
    }
}
```

### Wrapper 的自动生效

在 `META-INF/dubbo/` 中注册：

```
# META-INF/dubbo/com.example.LoadBalancer

random = com.example.spi.RandomLoadBalancer
roundrobin = com.example.spi.RoundRobinLoadBalancer
com.example.spi.wrapper.MonitorLoadBalancerWrapper
```

Dubbo 加载时：

1. 加载 `RandomLoadBalancer`
2. 检测到 `MonitorLoadBalancerWrapper` 构造函数需要 `LoadBalancer`
3. 自动用 `RandomLoadBalancer` 包装它
4. 返回包装后的实例

## 自动激活（Activate）：条件加载

@Activate 注解用于**自动激活**某些扩展点，可以指定激活条件。

### @Activate 注解参数

```java
@Activate(
    group = Constants.PROVIDER,        // 在哪个组激活
    value = "cache",                   // URL 参数名
    before = {},                      // 在哪些扩展之前激活
    after = {},                        // 在哪些扩展之后激活
    order = 0                         // 激活顺序
)
```

### 分组激活

```java
// 只在消费端激活
@Activate(group = Constants.CONSUMER)
public class ConsumerFilter implements Filter { }

// 只在提供端激活
@Activate(group = Constants.PROVIDER)
public class ProviderFilter implements Filter { }

// 两端都激活
@Activate(group = {Constants.CONSUMER, Constants.PROVIDER})
public class TracingFilter implements Filter { }
```

### 参数匹配激活

```java
// 当 URL 中有 cache=true 参数时激活
@Activate("cache")
public class CacheFilter implements Filter { }

// 当 URL 中没有 cache 参数时激活
@Activate(group = Constants.CONSUMER, value = "-cache")
public class NoCacheFilter implements Filter { }
```

### 实际应用：过滤器链

Dubbo 的过滤器链就是基于 `@Activate` 实现的：

```java
// 所有消费端过滤器都会被自动激活
@Activate(group = Constants.CONSUMER, order = -10000)
public class ActiveLimitFilter implements Filter { }

@Activate(group = Constants.CONSUMER, order = -90000)
public class ConsumerContextFilter implements Filter { }

@Activate(group = Constants.CONSUMER, order = -80000)
public class FutureFilter implements Filter { }
```

## 扩展点加载流程：源码解析

理解 Dubbo SPI 的加载流程，有助于排查问题。

### ExtensionLoader 核心逻辑

```java
public class ExtensionLoader<T> {

    private static final Map<Class<?>, ExtensionLoader<?>> EXTENSION_LOADERS =
            new ConcurrentHashMap<>();

    private final Class<T> type;

    // 缓存已加载的扩展实例
    private final Map<String, Holder<Object>> cachedInstances = new ConcurrentHashMap<>();

    // 缓存已加载的扩展类
    private final Map<String, Class<?>> cachedClasses = new ConcurrentHashMap<>();

    // 缓存 Adaptive 扩展类
    private volatile Class<?> cachedAdaptiveClass;

    public static <T> ExtensionLoader<T> getExtensionLoader(Class<T> type) {
        // 1. 校验类型
        if (!type.isInterface()) {
            throw new IllegalArgumentException("type must be interface");
        }
        if (!type.isAnnotationPresent(SPI.class)) {
            throw new IllegalArgumentException("type must be annotated by @SPI");
        }

        // 2. 从缓存获取 Loader
        ExtensionLoader<T> loader =
            (ExtensionLoader<T>) EXTENSION_LOADERS.get(type);

        if (loader == null) {
            loader = new ExtensionLoader<>(type);
            EXTENSION_LOADERS.put(type, loader);
        }
        return loader;
    }

    public T getExtension(String name) {
        // 1. 校验名称
        if (StringUtils.isEmpty(name)) {
            throw new IllegalArgumentException("extension name is empty");
        }

        // 2. 获取 Holder（用于双重检查锁定）
        Holder<Object> holder = cachedInstances.get(name);
        if (holder == null) {
            cachedInstances.putIfAbsent(name, new Holder<>());
            holder = cachedInstances.get(name);
        }

        // 3. 双重检查锁定
        Object instance = holder.get();
        if (instance == null) {
            synchronized (holder) {
                instance = holder.get();
                if (instance == null) {
                    // 4. 创建实例
                    instance = createExtension(name);
                    holder.set(instance);
                }
            }
        }
        return (T) instance;
    }

    private T createExtension(String name) {
        // 1. 获取扩展类
        Class<?> clazz = getExtensionClass(name);

        // 2. 创建实例（反射）
        T instance = (T) clazz.newInstance();

        // 3. 依赖注入（IOC）
        injectExtension(instance);

        // 4. 自动包装
        instance = injectWrapper(instance);

        return instance;
    }

    private Class<?> getExtensionClass(String name) {
        // 1. 从缓存获取
        Class<?> clazz = cachedClasses.get(name);
        if (clazz != null) {
            return clazz;
        }

        // 2. 扫描配置文件
        loadExtensionClasses();

        // 3. 再次从缓存获取
        return cachedClasses.get(name);
    }
}
```

### 配置文件扫描

```java
private void loadExtensionClasses() {
    // 扫描 @SPI 注解的值（默认扩展名）
    SPI spi = type.getAnnotation(SPI.class);
    if (spi != null) {
        String value = spi.value();
        if (StringUtils.isNotEmpty(value)) {
            String[] names = Constants.COMMA_SPLIT_PATTERN.split(value);
            // 缓存默认扩展
            if (names.length == 1) {
                cachedDefaultName = names[0];
            }
        }
    }

    // 扫描三个目录
    loadFile("META-INF/services/", type.getName());
    loadFile("META-INF/dubbo/", type.getName());
    loadFile("META-INF/dubbo/internal/", type.getName());
}

private void loadFile(String dir, String type) {
    // 1. 拼接文件路径
    String fileName = dir + type;

    // 2. 读取文件内容
    Enumeration<URL> urls;
    try {
        urls = classLoader.getResources(fileName);
    } catch (IOException e) {
        return;
    }

    // 3. 逐行解析
    while (urls.hasMoreElements()) {
        URL url = urls.nextElement();
        Properties properties = new Properties();
        properties.load(url.openStream());

        for (Map.Entry entry : properties.entrySet()) {
            String name = (String) entry.getKey();
            String className = (String) entry.getValue();

            // 4. 加载类并缓存
            Class<?> clazz = Class.forName(className, true, classLoader);
            cachedClasses.put(name, clazz);
        }
    }
}
```

## 自定义扩展：实战教程

现在你已经理解 Dubbo SPI 的原理，来实战一下：实现一个自定义负载均衡器。

### Step 1：定义接口

```java
@SPI
public interface WeightedLoadBalancer {

    /**
     * 根据权重选择服务器
     * @param servers 服务器列表
     * @param weights  权重映射
     * @return 选中的服务器
     */
    Server select(List<Server> servers, Map<Server, Integer> weights);
}
```

### Step 2：实现接口

```java
public class WeightedRandomLoadBalancer implements WeightedLoadBalancer {

    @Override
    public Server select(List<Server> servers, Map<Server, Integer> weights) {
        if (servers == null || servers.isEmpty()) {
            return null;
        }

        // 计算总权重
        int totalWeight = servers.stream()
                .mapToInt(s -> weights.getOrDefault(s, 1))
                .sum();

        // 随机选择
        int randomValue = ThreadLocalRandom.current().nextInt(totalWeight);

        // 根据权重找到对应服务器
        int currentWeight = 0;
        for (Server server : servers) {
            currentWeight += weights.getOrDefault(server, 1);
            if (randomValue < currentWeight) {
                return server;
            }
        }

        return servers.get(0);
    }
}
```

### Step 3：注册扩展

```
# META-INF/dubbo/com.example.spi.WeightedLoadBalancer

weightedRandom = com.example.spi.impl.WeightedRandomLoadBalancer
```

### Step 4：使用扩展

```java
public class WeightedLoadBalancerTest {
    public static void main(String[] args) {
        ExtensionLoader<WeightedLoadBalancer> loader =
            ExtensionLoader.getExtensionLoader(WeightedLoadBalancer.class);

        WeightedLoadBalancer balancer =
            loader.getExtension("weightedRandom");

        // 使用
        List<Server> servers = Arrays.asList(
            new Server("192.168.1.1", 8080),
            new Server("192.168.1.2", 8080),
            new Server("192.168.1.3", 8080)
        );

        Map<Server, Integer> weights = new HashMap<>();
        weights.put(servers.get(0), 5);  // 权重 5
        weights.put(servers.get(1), 3);  // 权重 3
        weights.put(servers.get(2), 2);  // 权重 2

        Server selected = balancer.select(servers, weights);
        System.out.println("选中的服务器: " + selected);
    }
}
```

### Step 5：与 Dubbo 集成

```java

```

## 面试追问：Dubbo SPI 还有什么值得深入？

### 问题 1：Dubbo SPI 为什么比 Java SPI 快？

Java SPI 每次 `iterator()` 都会重新实例化，Dubbo SPI 使用缓存复用实例。

### 问题 2：Adaptive 扩展是如何生成的？

Dubbo 在编译时通过注解处理器扫描 `@Adaptive`，或者运行时通过 Javassist/CGLIB 生成字节码。

### 问题 3：Wrapper 是如何实现自动包装的？

通过检测构造函数的参数类型，如果参数是扩展接口类型，就用已加载的实例注入。

### 问题 4：如何排查 SPI 加载问题？

开启 Dubbo SPI 调试日志：

```xml
<logger name="com.alibaba.dubbo.common.extension" level="DEBUG"/>
```

或者在启动参数添加：

```bash
-Ddubbo.application.logger=slf4j
-Ddubbo.spi.debug=true
```

## 总结：Dubbo SPI 的四大特性

| 特性 | 说明 | 实现方式 |
|---|---|---|
| **IOC** | 依赖注入 | 扫描 setter 方法，注入其他扩展 |
| **AOP** | 自动包装 | 检测构造函数参数，包装实例 |
| **Adaptive** | 运行时适配 | 生成适配器类，根据 URL 参数选择实现 |
| **Activate** | 条件激活 | 根据 group、value 条件自动激活 |

理解 Dubbo SPI 的原理，不仅能更好地使用 Dubbo 框架，还能借鉴其设计思想到自己的项目中。
