# Protobuf 原理

提到 Protobuf，你可能已经用过它——只需要写一个 `.proto` 文件，运行 `protoc` 编译，就能生成多语言的代码。

但你有没有想过：

- **Varint 编码**是怎么做到压缩整数的？
- **Zigzag 编码**为什么能让负数也高效存储？
- **Tag-Value 格式**是如何实现向前向后兼容的？
- **Protobuf 为什么比 JSON 快 5-10 倍**？

这一节，我们深入 Protobuf 的底层原理，从 IDL 定义到二进制编码，把每一个细节讲透。

## Protobuf 的核心设计哲学

在深入原理之前，先理解 Protobuf 的设计哲学。

**两个核心目标**：
1. **高效的序列化**：序列化后体积小，编解码速度快
2. **强接口契约**：通过 IDL 定义接口，保证跨语言、跨版本的一致性

**代价**：
- 牺牲了可读性（二进制格式）
- 需要预定义 Schema（不能直接序列化未知类型）

这和 JSON 的设计哲学截然不同——JSON 是「一切皆可序列化」，Protobuf 是「定义好了才能序列化」。

## IDL 定义：接口契约

### 基本语法

```protobuf
syntax = "proto3";

package order;

option java_package = "com.example.order";
option java_outer_classname = "OrderProto";

message Order {
    string order_id = 1;      // 字段编号
    string user_id = 2;
    double amount = 3;
    OrderStatus status = 4;
    repeated OrderItem items = 5;
}

message OrderItem {
    string product_id = 1;
    int32 quantity = 2;
    double price = 3;
}

enum OrderStatus {
    UNKNOWN = 0;       // 枚举必须从 0 开始
    PENDING = 1;
    PAID = 2;
    SHIPPED = 3;
    COMPLETED = 4;
}

message OrderRequest {
    string order_id = 1;
}

message OrderResponse {
    Order order = 1;
}

service OrderService {
    rpc GetOrder(OrderRequest) returns (OrderResponse);
    rpc StreamOrders(OrderRequest) returns (stream OrderResponse);
}
```

### 字段编号：最重要的设计决策

注意每个字段后面的数字：`string order_id = 1`、`string user_id = 2`。

**这个数字不是顺序号，而是字段标识符（Field Tag）**。它是 Protobuf 实现向前向后兼容的关键。

规则：
- **1-15**：用 1 个字节存储 Tag（1 byte Tag = 3 bits wire type + 5 bits field number）
- **16-2047**：用 2 个字节存储 Tag
- **19000-19999**：保留给 Protobuf 内部使用
- **不能使用负数作为字段编号**

:::tip 最佳实践

项目开始时就要规划好字段编号，因为：
- 字段编号一旦使用就不要改动
- 删除字段可以将其标记为 `reserved`
- 预留一些编号给未来可能添加的字段

```protobuf
message Order {
    reserved 1, 3, 5 to 10;  // 保留这些编号，不能使用
    string order_id = 1;     // 错误！被保留了
}
```

:::

## Varint 编码：小整数的极致压缩

Protobuf 高性能的核心原因之一是 Varint（Variable-length Integer）编码。

### 传统固定长度编码

假设用 4 个字节存储一个整数：

```
数字 1:  00000000 00000000 00000000 00000001
数字 300: 00000000 00000000 00000001 00101100
```

不管数值大小，都占用 4 个字节。但大多数整数都是小值，为什么要浪费空间？

### Varint 编码原理

Varint 用**可变字节数**存储整数，小整数用 1 个字节，大整数用多个字节。

每个字节的最高位是「继续位」（Most Significant Bit, MSB）：
- `1`：后面还有字节
- `0`：这是最后一个字节

```
数字 1:
二进制: 00000001
Varint: 00000001  (MSB=0，表示这是最后一个字节)
字节数: 1

数字 300:
二进制: 00000001 00101100
Varint: 10101100 00000010  (每个字节 MSB 表示是否继续)
字节数: 2
```

**图示**：

```
┌──────────────────────────────────────────────────┐
│  Varint 编码格式                                   │
│                                                   │
│  字节 1: [MSB|7 bits data]                       │
│  字节 2: [MSB|7 bits data] (如果需要)             │
│  ...                                              │
│                                                   │
│  MSB = 0: 这是最后一个字节                        │
│  MSB = 1: 后面还有字节                            │
└──────────────────────────────────────────────────┘
```

### 计算过程

数字 300 的编码过程：

```
Step 1: 将 300 转为二进制
300 = 0b100101100

Step 2: 每 7 位一组
300 = [10101100] [0000001]
       低 7 位    高 7 位

Step 3: 添加继续位
字节 1: 10101100 (MSB=1，后面还有)
字节 2: 00000010 (MSB=0，结束)

结果: 10101100 00000010
```

### 数字范围分析

| 范围 | 字节数 |
|---|---|
| 0-127 | 1 字节 |
| 128-16383 | 2 字节 |
| 16384-2097151 | 3 字节 |
| 2097152-268435455 | 4 字节 |
| ... | ... |

对于大多数业务数据（小整数 ID、状态码等），Varint 可以节省 50%-75% 的空间。

### Varint 的局限

Varint 不适合大整数和浮点数：

- 负数：`int32(-1)` = `0xFFFFFFFF` = 10 个字节，比固定长度还大
- 浮点数：IEEE 754 格式本身是固定的

这就引出了下一个话题：**Zigzag 编码**。

## Zigzag 编码：让负数也高效

### 问题：为什么负数在 Varint 中表现很差？

在 Java 中，`int` 是 32 位有符号整数，负数的二进制表示是「所有位都是 1」：

```
-1 的二进制补码: 11111111 11111111 11111111 11111111
-2 的二进制补码: 11111111 11111111 11111111 11111110
```

直接用 Varint 编码负数，每个都是 10 个字节（最大值）：

```
-1: 11111111 11111111 11111111 11111111 11111111 (5 字节，MSB 全是 1)
```

### Zigzag 解决方案

Zigzag 编码将**有符号整数映射到无符号整数**，再用 Varint 编码：

```
n = (n << 1) ^ (n >> 31)   // int32
n = (n << 1) ^ (n >> 63)   // int64
```

```
┌─────────────────────────────────────────────────────┐
│  Zigzag 映射（int32）                               │
│                                                     │
│   0 → 0                                             │
│  -1 → 1                                             │
│   1 → 2                                             │
│  -2 → 3                                             │
│   2 → 4                                             │
│  -3 → 5                                             │
│   ...                                               │
│  2147483647 → 4294967294                            │
│ -2147483648 → 4294967295                            │
└─────────────────────────────────────────────────────┘
```

### 为什么 Zigzag 能压缩负数？

关键在于**移位和异或操作**：

```
n = -1 (0xFFFFFFFF)
n << 1 = 0xFFFFFFFE
n >> 31 = 0xFFFFFFFF  (算术右移，高位补 1)
n << 1 ^ n >> 31 = 0xFFFFFFFE ^ 0xFFFFFFFF = 1

所以 -1 → 1，-2 → 3，1 → 2，2 → 4...
```

映射后的值都是**小整数**，Varint 编码只需要 1-2 个字节。

### sint32 vs int32

这就是为什么 Protobuf 区分 `int32` 和 `sint32`：

- `int32`：直接 Varint 编码，适合正数
- `sint32`：Zigzag 编码后再 Varint，适合有负数的场景

:::tip 选型建议

如果你的业务中可能出现负数（比如余额、偏移量等），使用 `sint32` 或 `sint64`。
如果确定都是正数，使用 `int32` 或 `uint32`。

:::

## Tag-Value 格式：向前向后兼容的奥秘

Protobuf 的序列化格式是 **Tag-Value**：每个字段由 Tag 和 Value 组成。

### Tag 的构成

```
Tag = (field_number << 3) | wire_type
```

```
┌────────────────────────────────────────┐
│         Tag 格式                        │
│                                         │
│  ┌─────────┬───────┐                   │
│  │ 5 bits  │ 3 bits│                   │
│  │ field#  │ type  │                   │
│  └─────────┴───────┘                   │
│                                         │
│  field_number: 字段编号                  │
│  wire_type:    值的编码类型              │
└────────────────────────────────────────┘
```

**wire_type 决定值的编码方式**：

| wire_type | 含义 | 示例 |
|---|---|---|
| 0 | Varint | int32, int64, sint32, sint64, bool, enum |
| 1 | 64-bit | fixed64, sfixed64, double |
| 2 | Length-delimited | string, bytes, repeated 字段 |
| 3 | Start group | 已废弃 |
| 4 | End group | 已废弃 |
| 5 | 32-bit | fixed32, sfixed32, float |

### Tag-Value 示例

```protobuf
message Order {
    string order_id = 1;  // field=1, type=2 (Length-delimited)
    int32  status = 2;    // field=2, type=0 (Varint)
    double amount = 3;    // field=3, type=1 (64-bit)
}
```

序列化一个 `Order` 对象：

```java
Order order = Order.newBuilder()
    .setOrderId("12345")
    .setStatus(1)
    .setAmount(99.99)
    .build();
```

二进制输出（十六进制）：

```
0a 07 31 32 33 34 35   // Tag=0x0A (field=1, type=2), Length=7, "12345"
10 01                   // Tag=0x10 (field=2, type=0), Varint=1
19 8e c3 8a 1d 5a 05 40 // Tag=0x19 (field=3, type=1), 99.99 (IEEE 754)
```

### 为什么能向前向后兼容？

这是 Protobuf 最精妙的设计：

**场景 1：新增字段**

```protobuf
// 旧版本
message Order {
    string order_id = 1;
}

// 新版本：添加了 user_id 字段
message Order {
    string order_id = 1;
    string user_id = 2;  // 新增
}
```

序列化时：
- 旧代码只发送 `order_id`
- 新代码发送 `order_id` + `user_id`
- **旧代码解析时遇到未知 Tag（field=2），直接跳过**
- **新代码解析时缺少 field=2，取默认值（空字符串）**

**场景 2：删除字段**

```protobuf
// 旧版本
message Order {
    string order_id = 1;
    string user_id = 2;  // 要删除
}

// 新版本：删除了 user_id
message Order {
    string order_id = 1;
    // user_id 被删除了
}
```

序列化时：
- 旧代码发送 `order_id` + `user_id`
- 新代码只发送 `order_id`
- **旧代码解析时遇到未知 Tag（field=2），直接跳过**
- **新代码正常工作**

**场景 3：字段类型变更（危险操作）**

```protobuf
// 旧版本
message Order {
    int32 amount = 1;  // 分（整数）
}

// 新版本：改为 double
message Order {
    double amount = 1;  // 元（浮点数）
}
```

这是**不兼容变更**：
- 旧代码发送整数 `999`（Varint）
- 新代码期望浮点数 `999.0`（IEEE 754 格式）
- 解析结果完全错误！

:::warning 字段编号重用禁忌

绝对不要在生产环境中重用已删除字段的编号。即使等了很长时间也不行，因为网络中可能还有旧版本序列化的数据。

:::

## 消息格式：完整的二进制结构

综合以上知识，一个完整 Protobuf 消息的二进制结构：

```
┌─────────────────────────────────────────────────────────────────┐
│                        Protobuf 消息                            │
│                                                                 │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐             │
│  │   Tag-Value │  │   Tag-Value │  │   Tag-Value │             │
│  │   (field 1) │  │   (field 2) │  │   (field 3) │             │
│  └─────────────┘  └─────────────┘  └─────────────┘             │
│                                                                 │
│  字段顺序不固定，可以任意顺序序列化，解析时会根据 Tag 定位        │
└─────────────────────────────────────────────────────────────────┘
```

### 实际解析过程

```java
public Order parseFrom(byte[] data) {
    Order.Builder builder = Order.newBuilder();
    ByteBuffer buffer = ByteBuffer.wrap(data);

    while (buffer.hasRemaining()) {
        // 1. 读取 Tag
        int tag = readVarint(buffer);
        int fieldNumber = (tag >> 3);
        int wireType = (tag & 0x07);

        // 2. 根据 wire_type 解析值
        switch (wireType) {
            case 0:  // Varint
                long varint = readVarint(buffer);
                builder.setField(fieldNumber, varint);
                break;
            case 1:  // 64-bit
                long fixed64 = buffer.getLong();
                builder.setField(fieldNumber, fixed64);
                break;
            case 2:  // Length-delimited
                int length = (int) readVarint(buffer);
                byte[] bytes = new byte[length];
                buffer.get(bytes);
                builder.setField(fieldNumber, ByteString.copyFrom(bytes));
                break;
        }
    }
    return builder.build();
}
```

## 性能对比：为什么 Protobuf 比 JSON 快？

### 性能瓶颈分析

JSON 的性能瓶颈在哪？

1. **字符串解析**：JSON 是文本格式，需要解析每个字符
2. **类型推断**：需要根据上下文推断类型（数字、字符串、布尔）
3. **内存分配**：大量临时 String/Number 对象

```java
// JSON 解析伪代码
String json = "{\"orderId\": \"12345\", \"amount\": 99.99}";
JSONObject obj = new JSONObject(json);

// 每次取值都要：
// 1. 查找 key（字符串匹配）
// 2. 解析 value（字符串转数字等）
String orderId = obj.getString("orderId");  // 字符串查找
double amount = obj.getDouble("amount");    // 字符串转 double
```

### Protobuf 的性能优势

1. **固定偏移读取**：Tag 告诉你在哪，直接读取固定长度
2. **无字符串解析**：二进制数据直接读取
3. **内存友好**：直接写入预分配的 buffer，无临时对象

```java
// Protobuf 解析伪代码
byte[] data = receiveFromNetwork();
Order order = Order.parseFrom(data);

// 读取 order_id:
// 1. 读到 Tag=0x0A，知道 field=1, wire_type=2
// 2. 读取 length=7
// 3. 直接复制 7 字节到 String
// 没有任何字符串解析或类型推断
```

以下数据来自业界公开测试（序列化 10000 次）：

| 指标 | JSON | Protobuf | 提升 |
|---|---|---|---|
| 序列化耗时 | 1200ms | 180ms | 6.7x |
| 反序列化耗时 | 1400ms | 150ms | 9.3x |
| 序列化后大小 | 850KB | 280KB | 3x |
| 内存分配次数 | 85000 | 300 | 283x |

这些数据是示意性的，实际性能取决于数据复杂度、JVM 版本、测试方法等因素。但 Protobuf 全面优于 JSON 的结论是确定的。

:::

## 与 JSON 的详细对比

| 维度 | JSON | Protobuf |
|---|---|---|
| **格式** | 文本（UTF-8） | 二进制 |
| **可读性** | 高 | 低 |
| **跨语言** | 原生支持 | 需要代码生成 |
| **Schema** | 可选 | 必需 |
| **序列化速度** | 慢 | 快 |
| **反序列化速度** | 慢 | 快 |
| **体积** | 大 | 小 |
| **向前向后兼容** | 依赖代码 | 原生支持 |
| **默认值处理** | 字段缺失 | 字段缺失（默认值） |
| **null 支持** | 原生支持 | 不支持（需 wrapper） |
| **注释** | 支持 | 不支持 |

## Protobuf 的局限性

任何技术都有局限，Protobuf 也不例外：

1. **不支持 null**：所有字段都有默认值（0、""、空 bytes），无法区分「未设置」和「设置为零值」
2. **不支持循环引用**：不能序列化图结构
3. **二进制不可读**：调试时需要工具解析
4. **Schema 必须预先定义**：不能像 JSON 那样直接序列化

```protobuf
// 无法区分「未设置」和「设置为零」
message Order {
    string order_id = 1;   // 无法区分：未传 vs 传了空字符串
    int32 amount = 2;      // 无法区分：未传 vs 传了 0
}
```

解决方案：使用 `optional` 关键字和 `oneof`：

```protobuf
message Order {
    optional string order_id = 1;  // optional 字段
    oneof order_id_or_nil {
        string order_id = 1;
        bool order_id_nil = 2;     // 用 oneof 表示 null
    }
}
```

## 总结：Protobuf 高性能的秘诀

Protobuf 高性能的核心在于：

1. **Varint 编码**：小整数用 1 字节存储
2. **Zigzag 编码**：负数也能高效存储
3. **Tag-Value 格式**：字段编号而非字段名，节省空间
4. **固定类型解析**：无需类型推断，直接按格式读取
5. **无临时对象**：直接写入 buffer，减少 GC 压力

理解这些原理，你不仅能更好地使用 Protobuf，还能在遇到序列化性能问题时，快速定位瓶颈。
