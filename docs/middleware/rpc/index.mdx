# RPC 框架

想象这样一个场景：你在微服务 A 中，需要调用微服务 B 的某个方法。这个方法运行在另一台机器上，甚至可能是另一种语言编写的。你会怎么做？

有人会说：「直接发 HTTP 请求啊，用 REST API 不就行了？」

没错，这确实是一种解法。但当你每天要进行数万次这样的跨服务调用时，HTTP 的开销就成了瓶颈。**REST API 的优势在于通用性和标准化，但 RPC 的优势在于性能和效率**——就像打电话和发邮件的区别，虽然都能传递信息，但场景不同，选择也不同。

RPC（Remote Procedure Call，远程过程调用）是一种让**分布式系统中的服务像调用本地方法一样调用远程方法**的技术。在分布式系统大行其道的今天，RPC 框架是每个后端工程师必须掌握的底层基础设施。

## 分布式系统的「最后一公里」

在分布式系统中，服务之间的通信是永恒的主题。从最初的 SOAP 到如今的 gRPC，从 Dubbo 到 Thrift，RPC 技术经历了数十年的演进。但无论形态如何变化，RPC 的核心使命始终只有一个：**让远程调用像本地调用一样简单，同时尽可能减少网络通信带来的性能损耗。**

这就是为什么大厂们纷纷自研 RPC 框架：阿里的 Dubbo、字节的 Kitex、腾讯的 TRPC、美团的 OCTO……每一个框架都是为了解决特定场景下的性能问题。

## 主流 RPC 框架一览

当前业界最流行的 RPC 框架有三驾马车：

| 框架 | 出品方 | 核心特点 | 适用场景 |
|---|---|---|---|
| **Dubbo** | 阿里巴巴 | 功能完善、生态丰富、支持多协议 |  Java 生态、微服务治理 |
| **gRPC** | Google | 基于 HTTP/2、Protobuf、跨语言 | 跨语言微服务、追求高性能 |
| **Thrift** | Facebook | IDL 驱动、多语言支持 | 多语言异构系统 |
