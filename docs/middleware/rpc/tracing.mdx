# RPC 链路追踪

想象一下这个场景：用户反馈「订单创建失败了」，但你的系统有几十个微服务，一次订单创建可能涉及十几个服务的调用。

你怎么排查？

去每个服务查日志？日志分散在不同的机器上，时间戳对不上，格式不统一……

这就是**分布式链路追踪**要解决的问题。

## 为什么需要链路追踪？

### 传统日志的困境

```
┌─────────────────────────────────────────────────────────┐
│                    分散的日志                          │
│                                                          │
│   请求入口 ──► 网关 ──► 订单服务 ──► 库存服务 ──► 支付服务
│      │           │         │           │            │
│      │           │         │           │            │
│   [日志1]     [日志2]    [日志3]     [日志4]      [日志5]
│   服务器A      服务器B     服务器C      服务器D      服务器E
│                                                          │
│   用户 ID: 12345                                        │
│   时间:   10:23:01  10:23:01  10:23:02  10:23:03  10:23:05
└─────────────────────────────────────────────────────────┘
```

手动关联这些日志？光是对时间、用户 ID 就够头疼了。

### 链路追踪的核心思想

链路追踪通过给每次请求分配一个**全局唯一的 Trace ID**，将整个调用链串联起来：

```
┌─────────────────────────────────────────────────────────┐
│                   链路追踪示意                           │
│                                                          │
│   Trace ID: abc123                                      │
│                                                          │
│   ┌──────────────────────────────────────────────────┐  │
│   │ Span A (入口)                                     │  │
│   │  ├──── Span B (网关)                             │  │
│   │  │    ├──── Span C (订单服务)                    │  │
│   │  │    │    ├──── Span D (库存服务)               │  │
│   │  │    │    └──── Span E (支付服务)               │  │
│   │  │    └──── Span F (用户服务)                    │  │
│   └────┴─────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────┘
```

## 核心概念：Trace、Span、SpanContext

### Trace（追踪）

一次完整的请求调用链，从开始到结束的所有 Span 组成一个 Trace。

```
Trace = Span1 + Span2 + Span3 + ... + SpanN
```

### Span（跨度）

一次 RPC 调用或一段代码执行，是一个 Span。每个 Span 记录：

- **Span Name**：操作名称（如 `OrderService.getOrderById`）
- **Start Time / End Time**：开始和结束时间
- **Span ID**：当前 Span 的唯一 ID
- **Parent Span ID**：父 Span 的 ID（构建调用树）
- **Tags / Logs**：标签和日志

```
┌─────────────────────────────────────────────────────────┐
│                        Span                             │
│                                                          │
│   Span ID: 456                                          │
│   Parent Span ID: 123                                    │
│                                                          │
│   ┌─────────────────── Duration ───────────────────┐   │
│   │ Start Time  ────────────────────────────  End Time │   │
│   │                                             │     │
│   │   10:23:01.001                        10:23:01.056 │   │
│   └─────────────────────────────────────────────────┘   │
│                                                          │
│   Tags:                                                 │
│     http.method = GET                                    │
│     http.url = /api/orders/123                          │
│     http.status_code = 200                              │
└─────────────────────────────────────────────────────────┘
```

### SpanContext（跨度上下文）

在服务间传递 Trace 信息的数据结构，通常通过 HTTP Header 或 RPC Metadata 传递。

```
┌─────────────────────────────────────────────────────────┐
│              SpanContext 传播                            │
│                                                          │
│   服务 A                         服务 B                  │
│   ┌─────────┐                   ┌─────────┐             │
│   │ SpanCtx │ ── HTTP Header ──►│ SpanCtx │             │
│   │ trace-id│   X-B3-TraceId   │ trace-id│             │
│   │ span-id │   X-B3-SpanId    │ span-id │             │
│   │         │   X-B3-ParentId  │         │             │
│   └─────────┘                   └─────────┘             │
└─────────────────────────────────────────────────────────┘
```

## OpenTracing 标准

OpenTracing 是链路追踪的标准化规范，主流追踪系统都兼容此标准。

### OpenTracing 数据模型

```java
import io.opentracing.Tracer;
import io.opentracing.Span;
import io.opentracing.SpanContext;
import io.opentracing.propagation.Format;

// 1. 创建 Span
Span span = tracer.buildSpan("OrderService.getOrderById")
        .withTag("order.id", orderId)
        .startActive();

// 2. 注入 SpanContext 到 HTTP Header
Map<String, String> headers = new HashMap<>();
tracer.inject(span.context(), Format.HTTP_HEADERS,
        new TextMapInjectAdapter(headers));

// 3. 从 HTTP Header 提取 SpanContext
SpanContext spanContext = tracer.extract(Format.HTTP_HEADERS,
        new TextMapExtractAdapter(incomingHeaders));

// 4. 基于父 Span 创建子 Span
Span childSpan = tracer.buildSpan("InventoryService.checkStock")
        .asChildOf(spanContext)
        .startActive();
```

## gRPC 链路追踪集成

gRPC 通过拦截器（Interceptor）实现链路追踪。

### gRPC Client Interceptor

```java
public class TracingClientInterceptor implements ClientInterceptor {

    private final Tracer tracer;

    public TracingClientInterceptor(Tracer tracer) {
        this.tracer = tracer;
    }

    @Override
    public <ReqT, RespT> ClientCall<ReqT, RespT> interceptCall(
            MethodDescriptor<ReqT, RespT> method,
            CallOptions callOptions,
            Channel next) {

        // 1. 创建或提取 SpanContext
        Span span = tracer.buildSpan(method.getFullMethodName())
                .withTag("grpc.method", method.getFullMethodName())
                .withTag("grpc.service", method.getServiceName())
                .startActive();

        // 2. 注入 Trace 信息到 Metadata
        Context context = Context.current().withValue(
                tracer.getActiveSpan(), span.context());

        return Contexts.interceptCall(context, next.newCall(method, callOptions),
                new ClientCallHandler(span));
    }

    private class ClientCallHandler implements ClientCall.Listener<RespT> {
        private final Span span;

        public ClientCallHandler(Span span) {
            this.span = span;
        }

        @Override
        public void onMessage(RespT message) {
            span.setTag("grpc.response_size",
                    calculateResponseSize(message));
            delegate().onMessage(message);
        }

        @Override
        public void onClose(Status status, Metadata trailers) {
            if (status.isOk()) {
                span.setTag("grpc.status_code", Status.Code.OK);
            } else {
                span.setTag("grpc.status_code", status.getCode());
                span.setTag("error", true);
                span.log(status.getDescription());
            }
            span.finish();
            delegate().onClose(status, trailers);
        }
    }
}
```

### gRPC Server Interceptor

```java
public class TracingServerInterceptor implements ServerInterceptor {

    private final Tracer tracer;

    @Override
    public <ReqT, RespT> ServerCall.Listener<ReqT> interceptCall(
            ServerCall<ReqT, RespT> call,
            Metadata headers,
            ServerCallHandler<ReqT, RespT> next) {

        // 1. 提取 Trace 信息
        Span span = tracer.buildSpan(call.getMethodDescriptor().getFullMethodName())
                .asChildOf(extractContext(headers))
                .withTag("grpc.method", call.getMethodDescriptor().getFullMethodName())
                .startActive();

        // 2. 处理请求
        ServerCall<ReqT, RespT> wrappedCall = new TracingServerCall<>(call, span);
        return new TracingServerCallListener<>(
                next.startCall(wrappedCall, headers), span);
    }

    private SpanContext extractContext(Metadata headers) {
        // 从 headers 中提取 Trace ID、Span ID
        // 使用 B3 或 W3C Trace Context 格式
        return tracer.extract(Format.Builtin.HTTP_HEADERS,
                new MetadataAdapter(headers));
    }
}
```

## Dubbo Filter 链路追踪

Dubbo 的 Filter 机制可以方便地集成链路追踪。

### 自定义 Dubbo Filter

```java
@Activate(group = {Constants.PROVIDER_GROUP, Constants.CONSUMER_GROUP})
public class TracingFilter implements Filter {

    private static final Logger logger = LoggerFactory.getLogger(TracingFilter.class);

    @Override
    public Result invoke(Invoker<?> invoker, Invocation invocation) {
        RpcContext context = RpcContext.getContext();
        boolean isProvider = context.isProviderSide();

        String traceId = RpcContext.getContext().getAttachment("traceId");
        String spanId = RpcContext.getContext().getAttachment("spanId");

        if (traceId == null) {
            traceId = UUID.randomUUID().toString().replace("-", "");
            spanId = "0";
        }

        // 创建 Span
        Span span = tracer.buildSpan(invoker.getInterface().getName() + "."
                + invocation.getMethodName())
                .withTag("dubbo.service", invoker.getInterface().getName())
                .withTag("dubbo.method", invocation.getMethodName())
                .withTag("trace.id", traceId)
                .withTag("span.id", spanId)
                .startActive();

        try {
            // 传递 Trace 信息
            context.setAttachment("traceId", traceId);
            context.setAttachment("parentSpanId", spanId);
            context.setAttachment("spanId", generateSpanId(spanId));

            // 调用
            Result result = invoker.invoke(invocation);

            // 记录结果
            if (result.hasException()) {
                span.setTag("error", true);
                span.log(result.getException());
            }

            return result;
        } catch (Exception e) {
            span.setTag("error", true);
            span.log(e);
            throw e;
        } finally {
            span.finish();
        }
    }

    private String generateSpanId(String parentSpanId) {
        // 生成子 Span ID：父 Span ID + 递增序号
        return parentSpanId + ".1";
    }
}
```


## 实战：分布式追踪问题定位

### 场景：订单创建慢

用户反馈：「创建订单需要 5 秒，太慢了。」

### 定位过程

**Step 1：查 SkyWalking**

在 SkyWalking UI 中搜索 Trace ID，发现整体耗时 5.2 秒。

**Step 2：分析调用链**

```
┌─────────────────────────────────────────────────────────┐
│                    调用链分析                            │
│                                                          │
│  ┌─ OrderService.createOrder()        5200ms ████████████│
│  │                                                         │
│  ├──── UserService.getUserInfo()         15ms █           │
│  │                                                         │
│  ├──── InventoryService.checkStock()    150ms ███          │
│  │                                                         │
│  ├──── PaymentService.processPayment() 5000ms ████████████│
│  │                                    ↑                    │
│  │                               问题在这里！              │
│  │                                                         │
│  └─────────────────────────────────────────────────────────┘
```

**Step 3：钻取 PaymentService**

点击 PaymentService，进入详细日志，发现是调用外部支付网关超时。

**Step 4：确认根因**

支付网关在高峰期响应慢，平均耗时 4-5 秒。

### 解决方案

1. **短期**：增加支付超时时间 + 异步回调
2. **中期**：接入多个支付通道，自动切换
3. **长期**：支付服务独立部署，隔离故障

## 常见链路追踪协议

### W3C Trace Context

新一代的 Trace 传播标准，Google、Microsoft、Zipkin 联合推动：

```http
traceparent: 00-0af7651916cd43dd8448eb211c80319c-b7ad6b7169203331-01
     │       │                                      │    │
     │       │                                      │    └── 记录级 ID (32 hex)

```


## 总结

链路追踪是微服务可观测性的核心：

- **Trace**：一次请求的完整调用链
- **Span**：一次 RPC 调用或代码块
- **SpanContext**：在服务间传递的上下文

主流实现：
- **SkyWalking**：APM 平台，零代码侵入
- **Jaeger**：CNCF 项目，OpenTracing 兼容
- **Zipkin**：轻量级，易部署

集成方式：
- **Java Agent**：零侵入，推荐
- **拦截器**：细粒度控制
- **Filter**：Dubbo 专用

没有链路追踪，微服务问题排查就是大海捞针。有了链路追踪，一切变得清晰可见。
