# RPC 序列化协议

序列化协议是 RPC 框架的核心组件之一，它直接决定了网络传输的效率和兼容性。

为什么这么说？

一次 RPC 调用中，序列化/反序列化可能占用 **50% 以上的 CPU 时间**。如果你选错了序列化协议，再好的网络传输层也救不了。

这一节，我们来彻底搞懂主流的序列化协议。

## 为什么序列化如此重要？

首先理解一下什么是序列化。

**序列化**：将对象状态转换为字节流的过程
**反序列化**：将字节流还原为对象的过程

```java
// 序列化
Order order = new Order();
order.setId("12345");
order.setAmount(99.99);
byte[] bytes = serializer.serialize(order);  // 对象 → 字节

// 反序列化
Order restored = (Order) serializer.deserialize(bytes);  // 字节 → 对象
```

在 RPC 调用中，这个过程发生在两端：

```
┌──────────┐   序列化   ┌────────┐   反序列化   ┌──────────┐
│  业务对象  │ ────────► │ 字节流  │ ─────────► │ 业务对象  │
└──────────┘            └────────┘             └──────────┘
   客户端                   网络                    服务端
```

### 序列化对性能的影响

假设一个 RPC 调用耗时 10ms：

| 环节 | 耗时 | 占比 |
|---|---|---|
| 序列化 | 3ms | 30% |
| 网络传输 | 2ms | 20% |
| 反序列化 | 3ms | 30% |
| 业务处理 | 2ms | 20% |

如果选择更快的序列化协议，耗时可能变成：

| 环节 | 耗时 | 占比 |
|---|---|---|
| 序列化 | 0.5ms | 15% |
| 网络传输 | 2ms | 60% |
| 反序列化 | 0.5ms | 15% |
| 业务处理 | 0.3ms | 10% |

总耗时从 10ms 降到了 3.3ms，提升了 3 倍。这就是为什么大厂都愿意花大力气自研序列化协议（阿里的 Fastjson、字节的 Buzzutil、美团的 Mowen）。

## 六种序列化协议对比

先上一张总览表，然后逐个分析：

| 协议 | 序列化速度 | 反序列化速度 | 体积 | 跨语言 | 可读性 | 适用场景 |
|---|---|---|---|---|---|---|
| **JSON** | 慢 | 慢 | 大 | 好 | 高 | Web API、调试 |
| **XML** | 很慢 | 很慢 | 很大 | 好 | 高 | SOAP、老系统 |
| **Java Serializable** | 慢 | 慢 | 大 | 差 | 无 | Java 内部 |
| **Protobuf** | 快 | 很快 | 小 | 好 | 无 | 高性能跨语言 |
| **Hessian** | 中 | 中 | 中 | 中 | 无 | Java 老系统 |
| **Kryo** | 很快 | 很快 | 小 | 差 | 无 | Java 高性能 |
| **FST** | 极快 | 极快 | 小 | 差 | 无 | Java 极致性能 |

## JSON：最通用的选择

JSON 是 Web 开发中最常用的序列化格式，它的核心优势是**人类可读和跨语言**。

### 序列化示例

```java
import com.fasterxml.jackson.databind.ObjectMapper;

ObjectMapper mapper = new ObjectMapper();
Order order = new Order();
order.setId("12345");
order.setAmount(99.99);

// 序列化
String json = mapper.writeValueAsString(order);
// {"id":"12345","amount":99.99}

// 反序列化
Order restored = mapper.readValue(json, Order.class);
```

### JSON 的优点

1. **可读性高**：调试时可以直接看到数据内容
2. **跨语言**：所有语言都支持 JSON
3. **生态完善**：大量工具支持（curl、Postman、浏览器 DevTools）

### JSON 的缺点

1. **体积大**：需要存储字段名，每个字符串都有引号
2. **速度慢**：需要解析字符串，类型推断耗时
3. **不支持复杂类型**：日期、枚举等需要特殊处理

### 性能数据（相对值，越小越好）

| 指标 | JSON | Protobuf | Kryo |
|---|---|---|---|
| 序列化后体积 | 100% | 30% | 25% |
| 序列化速度 | 100% | 200% | 400% |
| 反序列化速度 | 100% | 250% | 500% |

## Protobuf：高性能跨语言首选

Protobuf（Protocol Buffers）是 Google 开源的序列化框架，在 RPC 领域几乎是性能标杆。

### 定义数据结构

```protobuf
// order.proto
syntax = "proto3";

package order;

message Order {
    string id = 1;
    string user_id = 2;
    double amount = 3;
    repeated OrderItem items = 4;
}

message OrderItem {
    string product_id = 1;
    int32 quantity = 2;
}

message OrderRequest {
    string order_id = 1;
}

message OrderResponse {
    Order order = 1;
}

service OrderService {
    rpc GetOrder(OrderRequest) returns (OrderResponse);
}
```

### 生成代码

```bash
protoc --java_out=src/main/java order.proto
protoc --python_out=src/main/python order.proto
```

生成的 Java 代码：

```java
// 自动生成的代码
public final class Order extends com.google.protobuf.GeneratedMessageV3 {
    // 完整的 Builder 模式
    public static Order.Builder newBuilder() {
        return new Builder();
    }

    public String getId() {
        return id_;
    }

    public double getAmount() {
        return amount_;
    }
}
```

### 使用示例

```java
// 序列化
Order order = Order.newBuilder()
    .setId("12345")
    .setUserId("67890")
    .setAmount(99.99)
    .addItems(OrderItem.newBuilder()
        .setProductId("P001")
        .setQuantity(2)
        .build())
    .build();

byte[] bytes = order.toByteArray();

// 反序列化
Order restored = Order.parseFrom(bytes);
```

### Protobuf 的核心优势

1. **体积小**：使用 tag-value 编码，字段名用数字代替
2. **速度快**：固定长度字段直接读取，Varint 编码优化小整数
3. **跨语言**：所有语言都有代码生成工具
4. **向前兼容**：新增字段不影响旧版本

## Hessian：Java 老牌选择

Hessian 是 Caucho 公司开发的序列化协议，在早期 Dubbo 版本中是默认选择。

### 使用示例

```java
import com.caucho.hessian.io.Hessian2Input;
import com.caucho.hessian.io.Hessian2Output;

public class HessianSerializer {

    public <T> byte[] serialize(T obj) throws IOException {
        ByteArrayOutputStream baos = new ByteArrayOutputStream();
        Hessian2Output output = new Hessian2Output(baos);
        output.writeObject(obj);
        output.close();
        return baos.toByteArray();
    }

    public <T> T deserialize(byte[] bytes) throws IOException {
        ByteArrayInputStream bais = new ByteArrayInputStream(bytes);
        Hessian2Input input = new Hessian2Input(bais);
        @SuppressWarnings("unchecked")
        T result = (T) input.readObject();
        return result;
    }
}
```

### Hessian 的特点

- **与 Java Serializable 对比**：体积更小、速度更快、跨语言（Hessian 2）
- **与 Protobuf 对比**：不需要 IDL，直接序列化 Java 对象，开发更简单
- **缺点**：跨语言时字段名映射容易出问题，不适合异构系统

## Kryo：Java 高性能序列化

Kryo 是 Twitter 开源的序列化库，专注于 Java 性能优化。

### 使用示例

```java
import com.esotericsoftware.kryo.Kryo;
import com.esotericsoftware.kryo.io.Input;
import com.esotericsoftware.kryo.io.Output;

public class KryoSerializer {

    private final ThreadLocal<Kryo> kryo = ThreadLocal.withInitial(() -> {
        Kryo k = new Kryo();
        k.setRegistrationRequired(false);  // 允许未注册的类
        return k;
    });

    public <T> byte[] serialize(T obj) {
        Output output = new Output(1024);
        kryo.get().writeObject(output, obj);
        return output.toBytes();
    }

    public <T> T deserialize(byte[] bytes, Class<T> clazz) {
        Input input = new Input(bytes);
        return kryo.get().readObject(input, clazz);
    }
}
```

### Kryo 的核心优化

1. **避免反射**：使用 ASM 字节码直接访问字段
2. **注册机制**：提前注册类可以跳过类名写入
3. **对象池**：复用 Kryo 实例减少初始化开销
4. **循环引用检测**：避免重复序列化相同对象

### Kryo vs Protobuf

| 对比 | Kryo | Protobuf |
|---|---|---|
| IDL | 不需要 | 需要 |
| 跨语言 | 不支持 | 支持 |
| 序列化速度 | 更快 | 快 |
| 体积 | 更小 | 小 |
| 生态 | 较小 | 完善 |

**结论**：纯 Java 环境追求极致性能选 Kryo，需要跨语言选 Protobuf。

## FST：JDK RejectedExecutionException 的救星？

FST（Fast Serialization）是 Nikita E. Shelestov 开发的 Java 序列化库，号称性能比 Kryo 还快。

### 使用示例

```java
import com.esotericsoftware.kryo.io.FSTInput;
import com.esotericsoftware.kryo.io.FSTOutput;

// FST 复用 Kryo 的 API 设计
public class FstSerializer {

    public <T> byte[] serialize(T obj) throws IOException {
        FSTOutput output = new FSTOutput(new ByteArrayOutputStream());
        output.writeObject(obj);
        return output.toByteArray();
    }

    public <T> T deserialize(byte[] bytes, Class<T> clazz) throws IOException {
        FSTInput input = new FSTInput(new ByteArrayInputStream(bytes));
        return (T) input.readObject(clazz);
    }
}
```

### FST 的问题

FST 性能确实优秀，但它有一个致命问题：**不保证跨版本兼容性**。JDK 大版本升级后，FST 序列化的数据可能无法反序列化。在追求稳定性的生产环境中，这是一个大坑。

## Thrift：Facebook 的跨语言方案

Thrift 最初由 Facebook 开发，现在是 Apache 顶级项目。与 Protobuf 非常相似，都是 IDL 驱动的序列化框架。

### Thrift IDL 定义

```txt
namespace java com.example.order

struct Order {
    1: required string id,
    2: required string user_id,
    3: required double amount,
    4: optional list<OrderItem> items
}

struct OrderItem {
    1: required string product_id,
    2: required i32 quantity
}

service OrderService {
    Order getOrder(1: string order_id)
}
```

### 生成代码

```bash
thrift --gen java order.thrift
```

### Thrift vs Protobuf

| 对比 | Thrift | Protobuf |
|---|---|---|
| 诞生日 | 2007 | 2008 |
| 维护方 | Apache | Google |
| 协议格式 | 二进制 | 二进制 |
| RPC 框架 | 自带 | 需配合 gRPC |
| 服务定义 | 包含 RPC | 需配合 gRPC |
| 多语言支持 | 完善 | 完善 |
| 生态 | 较小 | 庞大（gRPC） |

## 如何选择序列化协议？

### 选择决策树

```
需要跨语言？
├── 否 → 纯 Java 环境
│   ├── 追求极致性能 → Kryo / FST
│   └── 需要与其他系统交互 → Hessian
└── 是 → 跨语言环境
    ├── 需要 RPC 功能 → gRPC + Protobuf
    └── 只需序列化 → Protobuf / Thrift
```

### 框架默认选择参考

| 框架 | 默认序列化 |
|---|---|
| Dubbo 2.x | Hessian |
| Dubbo 3.x | Triple 协议用 Protobuf |
| gRPC | Protobuf |
| Thrift | Thrift 二进制 |
| Spring Cloud | JSON |

### 选型建议

1. **新项目选 Protobuf + gRPC**：生态最完善，性能优秀
2. **Dubbo 项目选 Triple + Protobuf**：Dubbo 3.x 默认配置
3. **遗留系统选 Hessian**：兼容性好，与 Dubbo 2.x 无缝对接
4. **极致性能选 Kryo**：纯 Java 环境下的性能冠军

## 序列化协议的演进方向

序列化协议也在不断演进：

1. **WebAssembly**：跨语言的二进制格式
2. **Cap'n Proto**：比 Protobuf 更快的序列化
3. **FlatBuffers**：零反序列化的游戏级序列化
4. **Schema Registry**：Avro 的注册中心模式

理解这些协议的本质，才能在技术选型时做出正确的判断。
