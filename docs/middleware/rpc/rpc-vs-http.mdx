# RPC vs HTTP vs REST

很多初学者会有这样一个困惑：「既然 HTTP 请求也能实现跨服务调用，为什么还要用 RPC？」

反过来说，也有人问：「Dubbo、gRPC 这些框架底层不也是基于 HTTP 吗？为什么要学 RPC？」

这两个问题看似矛盾，但其实都指向了同一个核心：**HTTP 和 RPC 是不同层次的概念，不是非此即彼的对立关系。**

让我们彻底理清它们的关系。

## 为什么会有 RPC？

要理解 RPC 的价值，先要理解它解决的是什么问题。

在单机时代，方法调用是这样的：

```java
// 本地调用
OrderDTO order = orderService.getOrderById(orderId);
```

到了分布式时代，`orderService` 可能运行在另一台机器上。如果你用原生 Socket 实现：

```java
// 需要手动处理网络通信
Socket socket = new Socket("order-service", 8080);
ObjectOutputStream out = new ObjectOutputStream(socket.getOutputStream());
out.writeObject(request);

ObjectInputStream in = new ObjectInputStream(socket.getInputStream());
OrderDTO order = (OrderDTO) in.readObject();
```

问题在哪？

1. **代码侵入性强**：调用方需要写大量网络通信代码
2. **业务逻辑混杂**：网络代码和业务代码混在一起
3. **难以维护**：如果 `orderService` 换地址，所有调用方都要改

RPC 的诞生，就是为了解决这些问题——**让远程调用像本地调用一样简单**。

## RPC 和 HTTP 的关系：两个维度的比较

很多人把 RPC 和 HTTP 拿来对比，但它们其实不在同一个维度：

- **RPC 是架构设计理念**：强调「像调用本地方法一样调用远程方法」
- **HTTP 是通信协议**：定义了数据如何在网络中传输

这就像「微服务」和「TCP/IP」的关系——微服务是一种架构风格，TCP/IP 是实现它的网络协议。RPC 可以基于 HTTP，也可以基于 TCP，甚至可以基于 UDP（虽然不常见）。

```
┌─────────────────────────────────────────────┐
│                   应用层                      │
│                                             │
│   ┌─────────────┐     ┌─────────────┐       │
│   │    RPC      │     │   REST API  │       │
│   └─────────────┘     └─────────────┘       │
│         │                   │               │
│   ┌─────────────────────────────────┐      │
│   │      HTTP / HTTP2 / HTTP3       │      │
│   └─────────────────────────────────┘      │
│                     │                      │
│   ┌─────────────────────────────────┐      │
│   │              TCP                 │      │
│   └─────────────────────────────────┘      │
│                                             │
└─────────────────────────────────────────────┘
```

gRPC 基于 HTTP/2 协议，Dubbo 默认基于 TCP 协议（也可以切换到 HTTP）。所以严格来说，gRPC 也是一种 RPC 框架，但它的传输层用的是 HTTP/2。

## 三种调用方式的对比

### 1. 传统 HTTP（REST）

```java
// 客户端
@RestController
public class OrderController {
    @Autowired
    private RestTemplate restTemplate;

    public OrderDTO getOrder(String orderId) {
        // HTTP GET 请求
        return restTemplate.getForObject(
            "http://inventory-service/api/orders/" + orderId,
            OrderDTO.class
        );
    }
}
```

```java
// 服务端
@RestController
@RequestMapping("/api/orders")
public class OrderApi {

    @GetMapping("/{orderId}")
    public OrderDTO getOrder(@PathVariable String orderId) {
        return orderService.getOrderById(orderId);
    }
}
```

### 2. 传统 RPC（基于 TCP）

```java
// 定义接口
public interface OrderService {
    OrderDTO getOrderById(String orderId);
}

// 客户端调用
public class OrderClient {
    public OrderDTO getOrder(String orderId) {
        // 通过注册中心获取服务地址
        String address = registry.lookup("order-service");
        // 建立 TCP 连接，发送二进制数据
        byte[] response = tcpClient.send(address, serialize(request));
        return deserialize(response);
    }
}
```

### 3. gRPC（基于 HTTP/2 + Protobuf）

```protobuf
// 定义 .proto 文件
service OrderService {
    rpc GetOrder(OrderRequest) returns (OrderResponse);
}

message OrderRequest {
    string order_id = 1;
}

message OrderResponse {
    Order order = 1;
}
```

```java
// 客户端调用
public class OrderClient {
    public void getOrder(String orderId) {
        ManagedChannel channel = ManagedChannelBuilder
            .forAddress("order-service", 50051)
            .usePlaintext()
            .build();
        OrderServiceGrpc.OrderServiceBlockingStub stub =
            OrderServiceGrpc.newBlockingStub(channel);

        OrderRequest request = OrderRequest.newBuilder()
            .setOrderId(orderId)
            .build();
        OrderResponse response = stub.getOrder(request);
    }
}
```

## 核心维度对比

| 对比维度 | 传统 HTTP (REST) | 传统 RPC (TCP) | gRPC |
|---|---|---|---|
| **传输协议** | HTTP/1.1 | TCP | HTTP/2 |
| **序列化** | JSON/XML | 自定义二进制 | Protobuf |
| **接口定义** | Swagger/OpenAPI | 接口类或 IDL | IDL (.proto) |
| **代码生成** | 手动或 Swagger | 接口类 | 自动生成 |
| **性能** | 较低 | 高 | 高 |
| **可读性** | 高（JSON 可读） | 低（二进制） | 低（二进制） |
| **跨语言** | 好 | 差（依赖语言绑定） | 好 |
| **Browser 支持** | 原生支持 | 不支持 | 支持（需 grpc-web） |
| **流式支持** | 需 WebSocket | 看框架 | 原生支持 |
| **学习成本** | 低 | 中 | 中 |

## HTTP/1.1 到 HTTP/3 的演进

HTTP 协议的发展，也在影响 RPC 框架的选择。

### HTTP/1.1：文本协议的性能瓶颈

HTTP/1.1 是文本协议，每个请求都要带完整的 HTTP 头：

```
GET /api/orders/123 HTTP/1.1
Host: order-service
Content-Type: application/json
Authorization: Bearer xxx
... （还有更多 Header）
```

如果一天调用 10 万次，光 Header 就传输了海量重复数据。

### HTTP/2：多路复用带来的变革

HTTP/2 引入了多路复用（Multiplexing）和头部压缩（HPACK），多个请求可以复用同一个 TCP 连接，头部也可以压缩。

```
┌─────────────────────────────────────────┐
│           HTTP/2 连接复用                │
│                                         │
│  Stream 1: GET /api/orders/123          │
│  Stream 2: GET /api/users/456           │
│  Stream 3: POST /api/orders             │
│                                         │
│  ┌─────────────────────────────────┐    │
│  │      单一 TCP 连接               │    │
│  └─────────────────────────────────┘    │
└─────────────────────────────────────────┘
```

gRPC 基于 HTTP/2，因此天然支持多路复用和流式调用。

### HTTP/3：QUIC 协议的革命

HTTP/3 基于 QUIC 协议（UDP 实现），彻底解决了 TCP 的队头阻塞问题：

- **TCP 队头阻塞**：HTTP/2 中，一个包丢失会导致所有 stream 阻塞
- **QUIC 无队头阻塞**：每个 stream 独立，不互相影响

```
┌─────────────────────────────────────────┐
│  HTTP/3 (QUIC)                         │
│                                         │
│  Stream 1: ─────────────────────►        │
│  Stream 2: ─────────────────────►        │
│  Stream 3: ─────────────────────►        │
│                                         │
│  UDP 连接，stream 独立传输               │
│  一个 stream 丢包不影响其他 stream       │
└─────────────────────────────────────────┘
```

## 什么时候选 RPC，什么时候选 REST？

这个问题没有标准答案，但可以给出一个选择框架：

### 优先选择 REST API 的场景

1. **需要对外开放 API**：对外提供的服务，HTTP 是最通用的选择
2. **前端调用**：浏览器原生支持 HTTP，任何前端都能调用
3. **需要调试和测试**：JSON 可读，curl 就能测试
4. **跨防火墙**：HTTP 端口（80/443）通常不会被防火墙拦截
5. **团队技术水平参差不齐**：REST 的学习曲线更低

### 优先选择 RPC 的场景

1. **内部微服务调用**：性能敏感，不需要对外开放
2. **高频调用**：一次调用节省 1ms，10 万次就是 100 秒
3. **跨语言服务**：gRPC/Thrift 提供多语言代码生成
4. **需要流式调用**：gRPC 原生支持 Server Streaming
5. **需要严格接口契约**：IDL 文件定义了服务契约

### 典型场景：网关层 vs 服务间通信

```
                          ┌─────────────┐
                          │   客户端     │
                          └──────┬──────┘
                                 │ HTTP/REST
                                 ▼
                    ┌────────────────────────┐
                    │       API Gateway        │
                    │   (HTTP → RPC 协议转换)  │
                    └────────────┬────────────┘
                                 │ RPC
              ┌──────────────────┼──────────────────┐
              ▼                  ▼                  ▼
     ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
     │  Order Svc  │    │  User Svc   │    │Inventory Svc│
     └─────────────┘    └─────────────┘    └─────────────┘

     服务间通信用 RPC（高性能）
     对外暴露用 REST（通用性）
```

这是业界最佳实践：**对外网关用 REST，内部服务用 RPC**。网关负责协议转换，既保证了外部调用者的兼容性，又保证了内部服务的高性能。

## Dubbo 的 HTTP 支持

Dubbo 3.x 默认使用 Triple 协议（基于 HTTP/2），兼容 gRPC：

```java
// Dubbo 服务暴露，支持 HTTP/2
@DubboService(protocol = "tri")
public class OrderServiceImpl implements OrderService {
    // ...
}
```

Triple 协议的优势：
- 穿透性强：HTTP/2 端口可以被大多数防火墙放行
- 性能高：基于 HTTP/2 的多路复用
- 兼容性好：可以同时被 gRPC 客户端和原生 Dubbo 客户端调用

## 总结

理解 RPC 和 HTTP 的关系，关键在于认识到它们处于不同的抽象层次：

- **HTTP 是通信协议**：定义了数据如何打包和传输
- **RPC 是调用范式**：强调让远程调用像本地调用一样简单
- **REST 是架构风格**：一种基于 HTTP 的 API 设计模式

选择的关键在于场景：对内高频调用选 RPC，对外开放选 REST，内部高性能选 TCP/gRPC，外部兼容性选 HTTP/2+。
