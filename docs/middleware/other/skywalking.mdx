# SkyWalking 链路追踪

一个用户请求从浏览器发出，经过网关、Nginx、多个微服务、数据库，可能涉及十几台机器的调用。当这个请求变慢时，如何快速定位到「卡在哪一步」？

链路追踪，就是解决这个问题的。

## 从一个故事说起

假设你是一个外卖平台的运维工程师，用户反馈「下单很慢」。你的系统架构是这样的：

```
用户 App → API Gateway → Order Service → Product Service
                                ↓
                           Payment Service → Third-Party Payment
                                ↓
                           Inventory Service → MySQL
```

如果日志分散在每台机器上，你需要：

1. 登录网关机器，看请求入口时间
2. 登录 Order Service，看处理时长
3. 登录 Product Service，看是否有依赖超时
4. 登录 Payment Service，看第三方支付响应时间
5. ……

每一步都要猜、都要试，半小时过去了。

SkyWalking 想要做的，就是把这条链路**可视化**出来——让你一眼看到「Order Service 到 Payment Service 耗时 1.2s，其中 90% 是第三方支付响应慢」。

## 核心概念

### 链路（Trace）

一次完整的用户请求，从入口到出口，经过的所有服务调用串联起来，就是一条链路（Trace）。每条链路有一个全局唯一的 ID：`traceId`。

```
traceId: a3f0d8b7c6e5f4a3b2c1d0e9f8a7b6c5
  ├── Segment 1: API Gateway (入口)
  │     └── Span 1: HTTP 调用
  ├── Segment 2: Order Service
  │     ├── Span 2: SQL 查询
  │     ├── Span 3: HTTP 调用 Product Service
  │     └── Span 4: 异步写日志
  ├── Segment 3: Product Service
  │     └── Span 5: 缓存查询
  └── ...
```

### 跨度（Span）

每个服务节点的处理过程称为一个 Span。Span 记录了：

- **Span ID**：当前 Span 的编号
- **Parent Span ID**：父 Span 的 ID（构成调用树）
- **Start Time / End Time**：耗时
- **Tags**：业务标签（HTTP URL、SQL 语句）
- **Logs**：异常堆栈、关键日志

### 实例（Instance）

同一个服务的多个进程实例（如 Kubernetes 中的多个 Pod）会注册为不同的 Instance。SkyWalking 通过 Instance 区分「哪台机器出的问题」。

## 架构解析

SkyWalking 采用「无侵入」的 Agent 方式采集数据：

```
┌─────────────────────────────────────────────────────────┐
│                      SkyWalking                         │
│                                                         │
│  ┌─────────────┐    ┌─────────────┐    ┌───────────┐ │
│  │   UI        │───→│   OAP Server │←───│  Storage   │ │
│  │  (Web UI)   │    │  (分析引擎)   │    │  (ES/MySQL)│ │
│  └─────────────┘    └─────────────┘    └───────────┘ │
│                           ↑                             │
│                    ┌──────┴───────┐                    │
│                    │  Receivers    │                    │
│                    │  (gRPC/HTTP)  │                    │
│                    └───────────────┘                    │
└─────────────────────────────────────────────────────────┘
                           ↑
        ┌──────────────────┼──────────────────┐
        ↓                  ↓                  ↓
   ┌─────────┐       ┌─────────┐        ┌─────────┐
   │ Java    │       │ Node.js │        │ Python  │
   │ Agent   │       │ Agent   │        │ Agent   │
   └─────────┘       └─────────┘        └─────────┘
```

### 核心组件

| 组件 | 职责 |
|---|---|
| OAP Server | 接收 Agent 数据、分析拓扑、存储结果 |
| Storage | 支持 Elasticsearch、H2、MySQL、TiDB 等 |
| UI | 可视化界面：拓扑图、链路详情、性能指标 |
| Agent | 埋点采集，数据通过 gRPC/HTTP 上报 |

### 数据上报机制

SkyWalking Agent 使用 **Back-Buffer** 机制：

1. 链路数据先写入内存缓冲区
2. 缓冲区满或达到刷新间隔，数据批量上报
3. 如果 OAP 不可达，数据在 Agent 端缓存（默认最多 5000 条）

```yaml title="agent.config"
agent.collector.backend_service=oap-server:11800
agent.buffer_path=/tmp/skywalking-agent/buffer
agent.buffer.offset_max_file_size=10MB
agent.buffer.buffer_file_num=2
```

:::warning 缓冲区风险
如果 OAP 长时间不可达，缓冲区写满后旧数据会被丢弃，导致链路数据丢失。建议监控 OAP 可用性，并配置足够的缓冲区大小。
:::

## Agent 埋点原理

SkyWalking Java Agent 基于 **Java Agent** 和 **字节码插桩**（ByteBuddy）实现无侵入埋点。

### 常见组件自动埋点

| 组件 | 埋点范围 |
|---|---|
| HTTP Client | OkHttp、HttpClient、HttpURLConnection |
| RPC | Dubbo（Dubbo Filter）、gRPC |
| 数据库 | JDBC（MySQL、PostgreSQL、Oracle） |
| 消息队列 | Kafka、RabbitMQ、RocketMQ |
| 缓存 | Redis（Redisson、Lettuce） |
| 模板引擎 | FreeMarker、Velocity |

### 手动埋点

如果自动埋点不满足需求，可以手动添加：

```java
import org.apache.skywalking.apm.toolkit.trace.TraceContext;

// 方式一：获取当前链路 ID
String traceId = TraceContext.traceId();

// 方式二：主动创建 Span
try (ActiveSpan span = TraceContext.createExitSpan("custom-operation", "remote-peer:8080")) {
    span.setTag("custom-key", "custom-value");
    // 业务逻辑
    span.errorOccurred().log(new RuntimeException("custom error"));
}
```

### Trace 跨线程传递

在异步场景（如线程池、CompletableFuture）中，Trace 信息需要手动传递：

```java
// 方式一：使用 TraceContext.runWithTrace
TraceContext.runWithTraceContext(traceContextReference, () -> {
    // 异步执行的代码
});

// 方式二：使用 ExecutorService
ExecutorService executor = TracingExecutorService.wrap(Executors.newFixedThreadPool(10));
executor.submit(() -> {
    // 自动继承父 Trace
});

// 方式三：使用 @TraceCrossThread 注解（SkyWalking 8.x+）
@TraceCrossThread
private ExecutorService executor = Executors.newCachedThreadPool();
```

## 拓扑图（Topology）

SkyWalking 根据链路数据自动生成服务拓扑图，展示服务之间的调用关系和流量。

### 拓扑信息计算

```
┌─────────────┐         ┌─────────────┐
│  User       │────────→│  Gateway    │
│  Service    │         │             │
└─────────────┘         └──────┬──────┘
                               │
              ┌────────────────┼────────────────┐
              ↓                ↓                ↓
       ┌─────────────┐  ┌─────────────┐  ┌─────────────┐
       │   Order     │  │   Product   │  │   Payment   │
       │   Service   │  │   Service   │  │   Service   │
       └─────────────┘  └─────────────┘  └─────────────┘
```

OAP Server 通过分析 Span 中的 `peer`（对端服务名/IP），计算拓扑关系。

### 熔断标记

当某个服务出现大量错误或超时，拓扑图会用红色标记，帮助快速定位异常。

## 告警规则

SkyWalking 内置告警引擎，支持基于指标阈值的告警：

```yaml title="alarm-settings.yml"
rules:
  # 服务响应时间告警
  service_resp_time_rule:
    metrics-name: service_resp_time
    op: ">"
    threshold: 1000
    period: 10
    count: 3
    message: "服务 {{serviceName}} 响应时间超过 1 秒"

  # 服务成功率告警
  service_success_rate_rule:
    metrics-name: service_success_rate
    op: "<"
    threshold: 0.85
    period: 5
    count: 3
    message: "服务 {{serviceName}} 成功率低于 85%"
```

告警触发后，可通过 Webhook 发送到钉钉、飞书、企业微信。

## 总结

SkyWalking 是面向云原生应用的全链路追踪解决方案，核心能力：

- **无侵入**：Java Agent 自动埋点，无需改业务代码
- **全链路**：覆盖 HTTP、RPC、数据库、消息队列
- **可观测**：拓扑图、指标、告警一体化
- **可扩展**：支持自定义埋点和插件开发

> 下一步：如果你需要更深入的 APM 能力（如方法级性能分析），可以结合 SkyWalking 与 Arthas 一起使用。
