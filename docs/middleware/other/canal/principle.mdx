# Canal 工作原理

理解 Canal 的原理，需要先了解 MySQL 的主从复制机制和 binlog 格式。

## MySQL 主从复制原理

MySQL 主从复制依赖于 binlog——主库将数据变更记录到 binlog，从库拉取 binlog 并执行：

```
┌─────────────────┐                           ┌─────────────────┐
│      主库        │                           │      从库        │
│                 │                           │                 │
│  ┌───────────┐ │   binlog dump            │  ┌───────────┐ │
│  │   DML     │ │ ────────────────────────→ │  │  IO 线程   │ │
│  │  (写入)   │ │                           │  └─────┬─────┘ │
│  └─────┬─────┘ │                           │        │       │
│        │        │                           │        ▼       │
│        ▼        │                           │  ┌───────────┐ │
│  ┌───────────┐ │                           │  │  Relay    │ │
│  │  binlog   │ │                           │  │   Log    │ │
│  │ (变更日志) │ │                           │  └─────┬─────┘ │
│  └───────────┘ │                           │        │       │
│                 │                           │        ▼       │
│                 │                           │  ┌───────────┐ │
│                 │                           │  │   SQL     │ │
│                 │                           │  │  (执行)   │ │
│                 │                           │  └─────┬─────┘ │
│                 │                           │        │       │
│                 │                           │        ▼       │
│                 │                           │  ┌───────────┐ │
│                 │                           │  │   数据    │ │
│                 │                           │  │  (同步)  │ │
│                 │                           │  └───────────┘ │
└─────────────────┘                           └─────────────────┘
```

## Canal 工作原理

Canal 伪装成 MySQL 从库（Slave），与主库建立连接，模拟从库的 IO 线程：

```
┌─────────────────┐                           ┌─────────────────┐
│      主库        │                           │     Canal       │
│                 │                           │                 │
│  ┌───────────┐ │   binlog dump            │  ┌───────────┐ │
│  │  binlog   │ │ ────────────────────────→ │  │  Entry    │ │
│  │ (变更日志) │ │                           │  │ Parser    │ │
│  └─────┬─────┘ │                           │  └─────┬─────┘ │
│        │        │                           │        │       │
│        │        │   MySQL Protocol         │        ▼       │
│        │        │ ←────────────── │         │  ┌───────────┐ │
│        │        │                           │  │  Server   │ │
│        │        │                           │  │  (伪装Slave)│ │
│        └────────┘                           │  └───────────┘ │
│                                                └─────────────────┘
```

### Canal 核心流程

```
1. Canal Server 伪装成 MySQL Slave
2. 向 MySQL Master 发送 binlog dump 请求
3. MySQL Master 推送 binlog 事件
4. Canal 解析 binlog，转换为 Entry
5. 推送给客户端
```

## binlog 格式

### binlog 事件类型

| 事件类型 | 说明 |
|---|---|
| QUERY_EVENT | 执行 SQL 语句 |
| TABLE_MAP_EVENT | 表结构定义 |
| WRITE_ROWS_EVENT | INSERT 操作 |
| UPDATE_ROWS_EVENT | UPDATE 操作 |
| DELETE_ROWS_EVENT | DELETE 操作 |
| XID_EVENT | 事务提交 |

### Row 格式数据

```json
{
  "entry": {
    "header": {
      "logfileName": "mysql-bin.000001",
      "logfileOffset": 12345,
      "database": "order_db",
      "table": "orders",
      "eventType": "UPDATE"
    },
    "storeValue": {
      "beforeColumns": [
        {"name": "id", "value": 1},
        {"name": "status", "value": "pending"},
        {"name": "updated_at", "value": "2024-01-01 10:00:00"}
      ],
      "afterColumns": [
        {"name": "id", "value": 1},
        {"name": "status", "value": "paid"},
        {"name": "updated_at", "value": "2024-01-01 10:05:00"}
      ]
    }
  }
}
```

## Canal Entry 结构

```java
public class Entry {
    // 头信息
    private Header header;
    // 数据内容
    private ByteString storeValue;
}

// 头信息
public class Header {
    private String logfileName;    // binlog 文件名
    private long logfileOffset;    // binlog 偏移量
    private String schemaName;     // 数据库名
    private String tableName;      // 表名
    private EventType eventType;  // 事件类型
    private long timestamp;        // 时间戳
}

// 事件类型
public enum EventType {
    CREATE,
    ALTER,
    ERASE,
    QUERY,
    TRUNCATE,
    RENAME,
    INSERT,
    UPDATE,
    DELETE,
    GTID,
    HEARTBEAT,
    ...
}
```

## 数据解析

### 解析 binlog Entry

```java
public class CanalClient {

    public void onMessage(Entry entry) {
        Header header = entry.getHeader();
        String schema = header.getSchemaName();
        String table = header.getTableName();
        EventType eventType = header.getEventType();

        // 解析数据
        ByteString storeValue = entry.getStoreValue();
        CanalEntry.RowChange rowChange = CanalEntry.RowChange.parseFrom(storeValue);

        for (CanalEntry.RowData rowData : rowChange.getRowDatasList()) {
            if (eventType == EventType.INSERT) {
                // 解析新增数据
                List<CanalEntry.Column> columns = rowData.getAfterColumnsList();
            } else if (eventType == EventType.UPDATE) {
                // 解析修改前数据
                List<CanalEntry.Column> before = rowData.getBeforeColumnsList();
                // 解析修改后数据
                List<CanalEntry.Column> after = rowData.getAfterColumnsList();
            } else if (eventType == EventType.DELETE) {
                // 解析删除数据
                List<CanalEntry.Column> columns = rowData.getBeforeColumnsList();
            }
        }
    }
}
```

### 过滤条件

```java
// 按表过滤
List<CanalEntry.TableRelation> tableRelations = new ArrayList<>();
CanalEntry.TableRelation tableRelation = CanalEntry.TableRelation.newBuilder()
    .setDatabase("order_db")
    .setTable("orders")
    .build();
tableRelations.add(tableRelation);

CanalConnector connector = CanalConnectors.newClusterConnector(
    zkAddress, "example", "", "");
connector.subscribe("order_db\\.orders");  // 正则匹配

// 过滤 DML 操作
connector.subscribe("order_db\\.orders/.*\\\\.INSERT\\\\..*/.*");

// 过滤 DDL 操作
connector.subscribe("order_db\\..*");
```

## GTID 模式支持

### GTID（Global Transaction Identifier）

GTID 是 MySQL 5.6+ 引入的事务标识符：

```
GTID = source_id:transaction_id
GTID = "uuid:1", "uuid:2", "uuid:3"
```

### 配置

```yaml title="instance.properties"
canal.instance.gtidon = true
canal.instance.master.gtid.uuid = uuid-xxx
canal.instance.position.gtid = uuid:1-10
```

### 客户端消费

```java
// 启用 GTID 模式
CanalConnector connector = CanalConnectors.newClusterConnector(
    zkAddress, "example", "", "");
connector.connect();
connector.subscribe();

// 消费时记录 GTID 位置
GetBatchResult result = connector.getBatch(1000);
String gtid = result.getGtid();  // 获取当前 GTID

// 重新订阅时使用 GTID 恢复
connector.seek(gtid);
```

## 总结

Canal 核心原理：

| 步骤 | 说明 |
|---|---|
| 伪装 Slave | 模拟 MySQL 从库协议 |
| 请求 binlog | 向主库发送 dump 请求 |
| 解析 binlog | 转换为 Entry 对象 |
| 推送客户端 | 实时推送变更数据 |
