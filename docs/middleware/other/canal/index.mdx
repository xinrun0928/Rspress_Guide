# Canal

当数据库中的数据发生变化时，如何让其他系统感知到这个变化？

比如：订单数据变化后，需要同步更新 Elasticsearch 索引；用户信息变化后，需要同步更新 Redis 缓存。

Canal 就是解决这个问题的数据同步组件。

## 为什么需要 Canal？

传统方案的问题：

```
方案一：业务代码双写
订单服务.save() → 订单表
订单服务.save() → 同步 ES

问题：
- 代码耦合
- 无法保证一致性
- 出错难以补偿

方案二：定时任务同步
定时任务 → 查询变化数据 → 同步 ES

问题：
- 有延迟
- 无法实时
- 对数据库压力大
```

Canal 的方案：

```
订单服务.save() → 订单表 → Canal 拉取 binlog → 推送变化 → ES 索引更新

优势：
- 无侵入
- 实时同步
- 数据库原生支持
```

## Canal 是什么？

Canal 是阿里巴巴开源的 MySQL binlog 增量订阅和消费组件，模拟 MySQL Slave 的协议，实现增量数据订阅。

## 模块学习路径

```
1. Canal 工作原理：伪装成 MySQL Slave 接收 binlog
2. 订阅模式：直连模式与集群模式
3. 高可用部署：ZooKeeper 协调主备切换
4. 应用场景：ES 索引更新、缓存同步
5. Canal vs Maxwell vs Debezium 对比
```

> [!TIP]
> Canal 的核心思想是「让应用层像订阅消息一样订阅数据库变更」。
