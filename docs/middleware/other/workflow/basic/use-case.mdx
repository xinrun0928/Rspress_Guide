# 工作流应用场景

工作流引擎不是万能的，了解它的适用场景，才能用对地方、用好地方。

这一章我们从实际业务出发，聊聊工作流在审批流、订单处理、自动化流程中的应用。

## 审批流：多级审批与条件分支

审批流是工作流引擎最经典的应用场景——请假、报销、采购、合同……这些都需要多级审批。

### 场景一：多级审批

请假流程可能是最简单的工作流示例，但真实场景往往更复杂：

```
┌─────────────────────────────────────────────────────────────────────┐
│                                                                     │
│   开始 ──→ 提交申请 ──→ 主管审批 ──→ [天数 > 3?] ─┬─→ HR审批 ──→ 通过 │
│                              │                    └─→ 通过    │     │
│                              │                              │     │
│                              └───── [驳回] ───────→ 驳回 ──┘     │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 场景二：条件审批

不同金额、不同类型、不同部门的申请，走不同的审批路径：

```
┌─────────────────────────────────────────────────────────────────────┐
│                                                                     │
│   报销申请                                                           │
│      │                                                              │
│      ├─── [金额 <= 1000] ──→ 直属主管审批 ──────────────────────┐   │
│      │                                                              │   │
│      ├─── [金额 > 1000 且 <= 10000] ──→ 部门经理审批 ──────────┐ │   │
│      │                                                       │ │   │
│      └─── [金额 > 10000] ──→ 部门经理 + 财务审批 ──────────┐ │ │   │
│                                                          │ │ │   │
│                                                          │ │ │   │
│                                                          │ │ │   │
│                                                    所有节点通过      │
│                                                          └─┴─┴───┘   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 场景三：会签与或签

| 类型 | 说明 | 场景 |
|---|---|---|
| 会签 | 多人全部审批通过才通过 | 合同审批（法务 + 财务 + 业务） |
| 或签 | 任意一人审批通过即通过 | 紧急处理（值班人 A 或 B 或 C） |

```xml title="会签场景 BPMN"
<!-- 使用并行网关实现会签 -->
<parallelGateway id="counterSignGateway" name="会签开始"/>

<userTask id="legalReview" name="法务审批"/>
<userTask id="financeReview" name="财务审批"/>
<userTask id="businessReview" name="业务审批"/>

<parallelGateway id="counterSignEnd" name="会签结束"/>

<!-- 所有审批都通过后才进入结束 -->
<sequenceFlow sourceRef="legalReview" targetRef="counterSignEnd"/>
<sequenceFlow sourceRef="financeReview" targetRef="counterSignEnd"/>
<sequenceFlow sourceRef="businessReview" targetRef="counterSignEnd"/>
```

## 订单处理：状态机与流程编排

电商订单从创建到完成，经历了多个状态变迁。工作流引擎可以清晰管理这个过程。

### 订单状态流转

```
┌─────────────────────────────────────────────────────────────────────┐
│                                                                     │
│  创建订单 ──→ 支付 ──→ 待发货 ──→ 已发货 ──→ 已收货 ──→ 已完成      │
│      │         │        │          │         │         │          │
│      │         │        │          │         │         │          │
│      └────┐    │        │          │         │         │          │
│           │    │        │          │         │         │          │
│           ↓    │        │          │         │         │          │
│       取消订单 ←┘        │          │         │         │          │
│                          ↓          │         │         │          │
│                    超时未支付取消 ←──┘         │         │          │
│                                              ↓         │          │
│                                         退货/退款 ←───┘          │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 用工作流实现订单流程

```java
// 订单创建时启动工作流
@Service
public class OrderService {

    @Autowired
    private RuntimeService runtimeService;

    public void createOrder(Order order) {
        // 1. 保存订单到数据库
        orderRepository.save(order);

        // 2. 启动工作流实例
        ProcessInstance processInstance = runtimeService.startProcessInstanceByKey(
            "orderProcess",           // 流程定义 Key
            order.getId().toString(), // 业务键
            Variables.createVariables()
                .putValue("orderId", order.getId())
                .putValue("amount", order.getAmount())
                .putValue("userId", order.getUserId())
        );
    }
}
```

```xml title="order-process.bpmn"
<process id="orderProcess" name="订单处理流程">

  <startEvent id="start"/>

  <!-- 用户任务：待支付 -->
  <userTask id="awaitPayment" name="等待支付"/>

  <!-- 边界事件：支付超时取消 -->
  <boundaryEvent id="paymentTimeout" attachedToRef="awaitPayment">
    <timerEventDefinition>
      <timeDuration>PT30M</timeDuration>  <!-- 30 分钟超时 -->
    </timerEventDefinition>
  </boundaryEvent>
  <endEvent id="cancelTimeout"/>

  <!-- 排他网关：支付结果 -->
  <exclusiveGateway id="paymentResult"/>

  <!-- 用户任务：待发货 -->
  <userTask id="prepareGoods" name="准备发货"/>

  <!-- 用户任务：已发货 -->
  <userTask id="shipped" name="已发货"/>

  <!-- 用户任务：确认收货 -->
  <userTask id="confirmReceipt" name="确认收货"/>

  <!-- 用户任务：退货处理 -->
  <userTask id="returnGoods" name="退货处理"/>

  <endEvent id="end"/>
  <endEvent id="cancel" name="取消"/>

  <!-- 顺序流 -->
  <sequenceFlow sourceRef="start" targetRef="awaitPayment"/>
  <sequenceFlow sourceRef="awaitPayment" targetRef="paymentResult"/>

  <sequenceFlow sourceRef="paymentResult" targetRef="prepareGoods">
    <conditionExpression>${paymentStatus == 'SUCCESS'}</conditionExpression>
  </sequenceFlow>
  <sequenceFlow sourceRef="paymentResult" targetRef="cancel">
    <conditionExpression>${paymentStatus == 'FAILED'}</conditionExpression>
  </sequenceFlow>

  <sequenceFlow sourceRef="prepareGoods" targetRef="shipped"/>
  <sequenceFlow sourceRef="shipped" targetRef="confirmReceipt"/>

  <sequenceFlow sourceRef="confirmReceipt" targetRef="returnGoods">
    <conditionExpression>${returnRequested == true}</conditionExpression>
  </sequenceFlow>
  <sequenceFlow sourceRef="confirmReceipt" targetRef="end">
    <conditionExpression>${returnRequested == false}</conditionExpression>
  </sequenceFlow>

  <sequenceFlow sourceRef="returnGoods" targetRef="end"/>

  <!-- 超时取消 -->
  <sequenceFlow sourceRef="paymentTimeout" targetRef="cancelTimeout"/>

</process>
```

## 自动化业务流程

除了人工审批，工作流引擎还可以处理纯自动化的业务流程。

### 场景一：定时对账

每天凌晨执行对账任务，发现问题自动处理：

```xml title="reconciliation.bpmn"
<process id="reconciliationProcess">
  <startEvent id="scheduledStart">
    <timerEventDefinition>
      <timeCycle>0 0 2 ? * *</timeCycle>  <!-- 每天凌晨 2 点 -->
    </timerEventDefinition>
  </startEvent>

  <!-- 服务任务：拉取对账数据 -->
  <serviceTask id="fetchData" name="拉取数据"
    flowable:class="com.example.ReconciliationTask$FetchDataTask"/>

  <!-- 服务任务：比对数据 -->
  <serviceTask id="compareData" name="比对数据"
    flowable:class="com.example.ReconciliationTask$CompareDataTask"/>

  <!-- 排他网关：是否有差异 -->
  <exclusiveGateway id="hasDifference"/>

  <!-- 服务任务：生成差异报告 -->
  <serviceTask id="generateReport" name="生成报告"
    flowable:class="com.example.ReconciliationTask$GenerateReportTask"/>

  <!-- 服务任务：自动修复 -->
  <serviceTask id="autoFix" name="自动修复"
    flowable:class="com.example.ReconciliationTask$AutoFixTask"/>

  <!-- 服务任务：发送告警 -->
  <serviceTask id="sendAlert" name="发送告警"
    flowable:class="com.example.ReconciliationTask$SendAlertTask"/>

  <endEvent id="end"/>

  <sequenceFlow sourceRef="scheduledStart" targetRef="fetchData"/>
  <sequenceFlow sourceRef="fetchData" targetRef="compareData"/>
  <sequenceFlow sourceRef="compareData" targetRef="hasDifference"/>

  <sequenceFlow sourceRef="hasDifference" targetRef="autoFix">
    <conditionExpression>${fixable == true}</conditionExpression>
  </sequenceFlow>
  <sequenceFlow sourceRef="hasDifference" targetRef="generateReport">
    <conditionExpression>${fixable == false}</conditionExpression>
  </sequenceFlow>

  <sequenceFlow sourceRef="autoFix" targetRef="generateReport"/>
  <sequenceFlow sourceRef="generateReport" targetRef="sendAlert"/>
  <sequenceFlow sourceRef="sendAlert" targetRef="end"/>
  <sequenceFlow sourceRef="autoFix" targetRef="end"/>
</process>
```

### 场景二：异步任务编排

多个服务调用串联执行，支持失败重试：

```
开始 ──→ 订单服务 ──→ [成功?] ─┬─→ 库存服务 ──→ [成功?] ─┬─→ 物流服务 ──→ 完成
                               └─→ 补偿操作 ←───────────┘
```

```java
public class OrderFlowDelegate implements JavaDelegate {

    @Autowired
    private OrderService orderService;

    @Autowired
    private InventoryService inventoryService;

    @Autowired
    private LogisticsService logisticsService;

    @Override
    public void execute(DelegateExecution execution) {
        Long orderId = (Long) execution.getVariable("orderId");

        try {
            // 1. 创建订单
            orderService.create(orderId);
            execution.setVariable("orderCreated", true);

            // 2. 扣减库存
            boolean inventorySuccess = inventoryService.deduct(orderId);
            if (!inventorySuccess) {
                throw new BusinessException("库存不足");
            }
            execution.setVariable("inventoryDeducted", true);

            // 3. 创建物流
            logisticsService.create(orderId);
            execution.setVariable("logisticsCreated", true);

        } catch (Exception e) {
            // 补偿操作
            compensate(execution);
            throw e;
        }
    }

    private void compensate(DelegateExecution execution) {
        if ((Boolean) execution.getVariable("logisticsCreated")) {
            logisticsService.cancel((Long) execution.getVariable("orderId"));
        }
        if ((Boolean) execution.getVariable("inventoryDeducted")) {
            inventoryService.restore((Long) execution.getVariable("orderId"));
        }
        if ((Boolean) execution.getVariable("orderCreated")) {
            orderService.cancel((Long) execution.getVariable("orderId"));
        }
    }
}
```

## 何时使用工作流引擎？

| 场景 | 推荐 | 理由 |
|---|---|---|
| 简单的 if-else 状态机 | 不推荐 | 引入工作流增加复杂度 |
| 审批流程，多级多路径 | 推荐 | 工作流天然适合复杂流转 |
| 需可视化流程设计 | 推荐 | BPMN 模型让业务人员参与 |
| 需要流程历史追溯 | 推荐 | 工作流自带审计功能 |
| 需要动态修改流程 | 推荐 | 无需改代码，改模型即可 |
| 高性能实时计算 | 不推荐 | 工作流有一定开销 |
| 简单的定时任务 | 不推荐 | Quartz 更轻量 |

## 总结

工作流引擎的核心价值：

- **可视化**：业务流程图形化，易于沟通
- **可追溯**：完整记录流程执行历史
- **可变更**：修改流程不需要改代码
- **可复用**：流程模板可复制使用
