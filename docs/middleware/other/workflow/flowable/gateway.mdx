# Flowable 网关

网关是 BPMN 中控制流程分支和合并的元素。Flowable 支持四种网关：排他网关、并行网关、包容网关、事件网关。

## 排他网关（Exclusive Gateway）

排他网关（XOR Gateway）根据条件选择**唯一**一条路径执行。

### 图形表示

```
      ┌──────────────┐
      │      ◇       │  ← 菱形 + X = 排他网关
      │    (XOR)     │
      └──────────────┘
```

### 业务场景

请假天数决定审批路径：

```xml title="bpmn/leave-gateway.bpmn20.xml"
<!-- 排他分支网关 -->
<exclusiveGateway id="approvalGateway" name="审批分流"/>

<!-- 顺序流上的条件 -->
<sequenceFlow id="flow1" sourceRef="approvalGateway" targetRef="managerTask">
  <conditionExpression>${days <= 3}</conditionExpression>
</sequenceFlow>

<sequenceFlow id="flow2" sourceRef="approvalGateway" targetRef="managerAndHrTask">
  <conditionExpression>${days > 3}</conditionExpression>
</sequenceFlow>

<!-- 排他合并网关 -->
<exclusiveGateway id="mergeGateway" name="审批合并"/>
```

### 执行规则

```
开始 ──→ 提交请假 ──→ ◇ ──┬─ [days <= 3] ──→ 主管审批 ──┬─→ 结束
                          └─ [days > 3] ──→ 主管+HR审批 ──┘
```

1. **条件互斥**：只会有一条路径被选中
2. **顺序评估**：按 BPMN 文件中的定义顺序评估条件
3. **默认流**：如果所有条件都不满足，执行默认流（default 属性）

### 默认流

```xml
<exclusiveGateway id="gateway" name="审批分流" default="defaultFlow"/>

<!-- 默认流（无条件） -->
<sequenceFlow id="defaultFlow"
  sourceRef="gateway"
  targetRef="managerTask"/>

<!-- 条件流 -->
<sequenceFlow id="flow1"
  sourceRef="gateway"
  targetRef="urgentTask">
  <conditionExpression>${urgent == true}</conditionExpression>
</sequenceFlow>
```

## 并行网关（Parallel Gateway）

并行网关（AND Gateway）同时执行**所有**分支。

### 图形表示

```
      ┌──────────────┐
      │      ◇+      │  ← 菱形 + 加号 = 并行网关
      │     (AND)    │
      └──────────────┘
```

### 业务场景

订单确认后，同时通知仓库、财务和物流：

```xml
<!-- 并行分支 -->
<parallelGateway id="fork" name="并行通知"/>

<!-- 三个并行任务 -->
<serviceTask id="notifyWarehouse" name="通知仓库"
  flowable:class="com.example.NotifyWarehouseTask"/>
<serviceTask id="notifyFinance" name="通知财务"
  flowable:class="com.example.NotifyFinanceTask"/>
<serviceTask id="notifyLogistics" name="通知物流"
  flowable:class="com.example.NotifyLogisticsTask"/>

<!-- 并行合并 -->
<parallelGateway id="join" name="等待完成"/>
```

### 执行规则

```
开始 ──→ 确认订单 ──→ ◇+ ──┬─→ 通知仓库 ──┐
                           ├─→ 通知财务 ──┼─→ ◇+ ──→ 结束
                           └─→ 通知物流 ──┘

说明：所有三个任务同时开始
      三个任务都完成后，才进入合并网关
```

1. **同步执行**：所有分支同时开始
2. **等待完成**：只有所有分支都完成，才继续执行
3. **分支计数**：入向分支数和出向分支数必须匹配

### 菱形符号对照

| 网关类型 | 菱形内符号 | 说明 |
|---|---|---|
| 排他网关 | X 或空白 | 一选一 |
| 并行网关 | + | 全部执行 |
| 包容网关 | O | 满足条件的都执行 |
| 事件网关 | ⚡ | 根据事件选择 |

## 包容网关（Inclusive Gateway）

包容网关（OR Gateway）根据条件执行**满足条件的所有**路径。

### 图形表示

```
      ┌──────────────┐
      │      ◇○      │  ← 菱形 + 圆圈 = 包容网关
      │     (OR)     │
      └──────────────┘
```

### 业务场景

根据业务条件，可能需要不同的审批：

```xml
<!-- 包容分支 -->
<inclusiveGateway id="approvalGateway"/>

<!-- 多个条件流 -->
<sequenceFlow id="flow1" sourceRef="approvalGateway"
  targetRef="managerApproval">
  <conditionExpression>${needManagerApproval}</conditionExpression>
</sequenceFlow>

<sequenceFlow id="flow2" sourceRef="approvalGateway"
  targetRef="financeApproval">
  <conditionExpression>${needFinanceApproval}</conditionExpression>
</sequenceFlow>

<sequenceFlow id="flow3" sourceRef="approvalGateway"
  targetRef="normalTask">
  <!-- 无条件，默认执行 -->
</sequenceFlow>

<!-- 包容合并 -->
<inclusiveGateway id="mergeGateway"/>
```

### 执行规则

```
开始 ──→ 提交申请 ──→ ◇○ ──┬─ [needManagerApproval] ─→ 主管审批 ──┐
                            ├─ [needFinanceApproval] ─→ 财务审批 ──┼─→ ◇○ ──→ 结束
                            └─ [无条件] ──────────→ 普通处理 ────┘

说明：根据变量值决定执行哪些路径
      例如：needManagerApproval=true, needFinanceApproval=false
      则执行：主管审批 + 普通处理
```

### 与并行网关的区别

| 维度 | 并行网关 | 包容网关 |
|---|---|---|
| 条件 | 无条件 | 有条件 |
| 执行 | 所有分支都执行 | 只执行满足条件的分支 |
| 合并 | 必须所有分支都完成 | 只等待执行的分支完成 |

## 事件网关（Event-Based Gateway）

事件网关根据**外部事件**选择路径，而不是条件表达式。

### 图形表示

```
      ┌──────────────┐
      │      ◇⚡     │  ← 菱形 + 闪电 = 事件网关
      │   (Event)   │
      └──────────────┘
```

### 业务场景

等待用户选择支付方式：

```xml
<!-- 事件网关（分支） -->
<eventBasedGateway id="paymentChoice"/>

<!-- 等待不同的支付事件 -->
<receiveTask id="waitAlipay" name="等待支付宝支付">
  <messageEventDefinition messageRef="ns:AlipayMessage"/>
</receiveTask>

<receiveTask id="waitWechat" name="等待微信支付">
  <messageEventDefinition messageRef="ns:WechatMessage"/>
</receiveTask>

<receiveTask id="waitCard" name="等待银行卡支付">
  <messageEventDefinition messageRef="ns:CardMessage"/>
</receiveTask>
```

### 执行规则

```
开始 ──→ 创建订单 ──→ ◇⚡ ──┬─ [支付宝支付] ──→ ◇ ──┬─→ 成功
                           ├─ [微信支付] ──→ ◇ ─────┘
                           └─ [银行卡支付] ──→ ◇ ───┘

说明：网关创建后，等待三个任务中的任意一个完成
      第一个完成的事件决定了后续路径
      其他两个任务会被取消
```

### 与排他网关的区别

| 维度 | 排他网关 | 事件网关 |
|---|---|---|
| 决策依据 | 条件表达式 | 外部事件 |
| 执行时机 | 即时 | 异步等待 |
| 后续处理 | 条件已确定 | 等待事件触发 |

## 复杂网关组合

实际业务中，网关往往需要组合使用。

### 场景：多级审批 + 会签

```
开始 ──→ 提交申请 ──→ ◇ ──┬─ [普通] ──→ 主管审批 ──→ ◇ ──→ 结束
                           ├─ [重要] ──→ ◇+ ──┬─→ 主管审批 ──┤
                           │                  └─→ 经理审批 ──┼─→ ◇ ──→ 结束
                           └─ [紧急] ──→ 总监审批 ──────→ ◇ ──→ 结束
```

```xml
<!-- 审批类型分流 -->
<exclusiveGateway id="typeGateway"/>

<!-- 普通申请：主管审批 -->
<userTask id="managerTask" name="主管审批"/>

<!-- 重要申请：会签 -->
<parallelGateway id="importantFork"/>
<userTask id="managerTask2" name="主管审批"/>
<userTask id="managerLevelTask" name="经理审批"/>
<parallelGateway id="importantJoin"/>

<!-- 紧急申请：总监审批 -->
<userTask id="directorTask" name="总监审批"/>

<!-- 结果合并 -->
<exclusiveGateway id="resultGateway"/>
```

### 注意事项

1. **网关对必须平衡**：分支网关和合并网关数量要匹配
2. **避免死锁**：确保每个分支都有合并点
3. **条件覆盖**：排他网关要覆盖所有可能的情况

## 总结

四种网关对比：

| 网关类型 | 图形 | 语义 | 场景 |
|---|---|---|---|
| 排他网关 | ◇ X | 一选一 | 条件互斥 |
| 并行网关 | ◇ + | 全部执行 | 多任务并行 |
| 包容网关 | ◇ O | 满足的都执行 | 条件叠加 |
| 事件网关 | ◇ ⚡ | 等待事件 | 外部触发 |
