# 工作流引擎面试题

面试官问工作流引擎相关问题，往往不是想让你背 API，而是想看：

1. 你对业务流程建模的理解有多深
2. 你有没有实际用过工作流引擎解决过问题
3. 你对技术选型的思考

顺着这个思路准备，比单纯背答案高出几个层次。

## 为什么要问工作流引擎？

面试官问这个问题，通常是想考察：

- 你对「业务流程抽象」的理解
- 你有没有处理过复杂的审批场景
- 你对 BPMN 标准的了解程度
- 你对技术选型的思考方式

顺着这个思路回答，比单纯罗列 API 高出一个层次。

## 常见面试题

### 1. 工作流引擎和状态机有什么区别？

这道题看似简单，但能答出深层区别的候选人不多。

**表面层（大部分人能答上来）：**
- 状态机通过代码管理状态流转
- 工作流引擎通过 BPMN 模型管理流程

**深层（优秀候选人能答上来）：**
- 状态机适合简单的状态变更，不适合复杂的多级审批
- 工作流引擎提供可视化建模、业务追踪、动态变更能力
- 状态机是无状态的（stateless），工作流引擎维护执行状态

**实战层（让面试官眼前一亮的答案）：**
- 状态机适合电商订单、支付状态这类「单一对象」的状态变更
- 工作流引擎适合跨系统的审批流、多角色协作场景
- 两者可以结合：用状态机管理局部状态，用工作流管理全局流程

```java
// 状态机适合的场景：订单状态变更
public enum OrderState {
    CREATED -> PAID -> SHIPPED -> COMPLETED
           -> CANCELLED
           -> REFUNDED
}

// 工作流引擎适合的场景：多级审批
// 员工提交 -> 主管审批 -> HR审批 -> 财务审批 -> 完成
//       -> 驳回（任意节点）
```

### 2. BPMN 2.0 的核心元素有哪些？

这道题考察对 BPMN 标准的理解。

**标准答案：**

| 类别 | 元素 | 说明 |
|---|---|---|
| 事件 | 开始事件、结束事件、中间事件 | 流程中的「瞬间」 |
| 活动 | 任务、子流程 | 流程中的「工作」 |
| 网关 | 排他、并行、包容、事件 | 流程的「分流合流」 |
| 流向 | 顺序流、消息流 | 连接元素的「线条」 |

**追问：排他网关和包容网关的区别？**

| 网关类型 | 条件关系 | 场景 |
|---|---|---|
| 排他网关 | 互斥，只能选一条 | 条件互斥 |
| 并行网关 | 同时执行所有路径 | 多任务并行 |
| 包容网关 | 或关系，满足的都执行 | 条件叠加 |

### 3. Flowable 有哪些核心服务？

这道题考察对 Flowable 架构的了解。

**六大核心服务：**

| 服务 | 职责 |
|---|---|
| RepositoryService | 管理流程定义（部署、查询） |
| RuntimeService | 管理流程实例（启动、执行） |
| TaskService | 管理用户任务（查询、办理） |
| HistoryService | 查询历史数据 |
| IdentityService | 管理用户和组 |
| ManagementService | 管理定时器、作业 |

```java
// 典型的工作流操作
repositoryService.createDeployment()  // 部署流程
runtimeService.startProcessInstanceByKey()  // 启动流程
taskService.createTaskQuery()  // 查询任务
taskService.complete()  // 完成任务
historyService.createHistoricProcessInstanceQuery()  // 查询历史
```

### 4. 流程变量和执行上下文有什么区别？

**流程变量（Process Variable）：**
- 存储业务数据
- 可以在整个流程实例中访问
- 用于条件判断、流程控制

```java
runtimeService.setVariable(processInstanceId, "orderId", 12345);
runtimeService.setVariable(processInstanceId, "amount", 1000.00);
```

**执行上下文（Execution Context）：**
- 记录流程执行状态
- 包含当前位置、历史路径
- 管理并发执行的多个分支

```java
// 获取当前执行位置
String activityId = runtimeService.createExecutionQuery()
    .processInstanceId(processInstanceId)
    .singleResult()
    .getActivityId();
```

### 5. 如何实现任务委派和任务代理？

这道题考察对任务分配机制的理解。

**任务代理（Assignment）：** 直接分配给指定人

```java
// 直接指定办理人
task.setAssignee("zhangsan");
```

**任务认领（Claim）：** 从候选列表中抢领

```java
// 从候选列表中认领
taskService.claim(taskId, "zhangsan");
```

**任务委派（Delegate）：** 把任务转给他人办理，办理完后回到原办理人

```java
// 委托给 lisi 办理，办理完后回到我
taskService.delegateTask(taskId, "lisi");
```

**追问：如何实现「或签」（多人抢一个任务）？**

使用候选用户或候选组：

```java
// 方式一：设置多个候选用户
task.addCandidateUser("zhangsan");
task.addCandidateUser("lisi");

// 方式二：设置候选组
task.addCandidateGroup("manager-group");
```

### 6. 如何实现会签（多人审批全部通过）？

这道题考察对并行网关的理解。

```xml
<!-- 使用并行网关实现会签 -->
<startEvent id="start"/>
<parallelGateway id="fork" name="会签开始"/>
<userTask id="legalApproval" name="法务审批"/>
<userTask id="financeApproval" name="财务审批"/>
<parallelGateway id="join" name="会签结束"/>
<endEvent id="end"/>

<!-- 所有路径都汇合到 join 网关 -->
<sequenceFlow sourceRef="fork" targetRef="legalApproval"/>
<sequenceFlow sourceRef="fork" targetRef="financeApproval"/>
<sequenceFlow sourceRef="legalApproval" targetRef="join"/>
<sequenceFlow sourceRef="financeApproval" targetRef="join"/>
```

### 7. 流程定义挂了，但还有正在执行的流程实例怎么办？

这道题考察对流程生命周期管理的理解。

**正确做法：**

1. 先挂起流程实例，防止继续执行
2. 再升级流程定义
3. 决定旧实例如何处理：
   - 让它执行到结束（不迁移）
   - 迁移到新版本（migration）

```java
// 1. 挂起流程定义
repositoryService.suspendProcessDefinitionByKey("leave-process");

// 2. 挂起所有相关实例
runtimeService.suspendProcessInstanceById(processInstanceId);

// 3. 升级流程定义后，可以迁移实例
runtimeService.createProcessInstanceMigrationBuilder()
    .migrateToProcessDefinition(newProcessDefinitionId)
    .addMigrationMapping("oldTaskId", "newTaskId")
    .migrate(processInstanceId);
```

### 8. 定时任务在流程中怎么用？

这道题考察对异步执行机制的理解。

```xml
<!-- 定时开始事件 -->
<startEvent id="scheduledStart">
  <timerEventDefinition>
    <timeCycle>0 0 2 ? * *</timeCycle>  <!-- 每天凌晨 2 点 -->
  </timerEventDefinition>
</startEvent>

<!-- 边界定时器（超时处理） -->
<boundaryEvent id="timeout" attachedToRef="approvalTask">
  <timerEventDefinition>
    <timeDuration>PT24H</timeDuration>  <!-- 24 小时超时 -->
  </timerEventDefinition>
</boundaryEvent>
<sequenceFlow sourceRef="timeout" targetRef="autoReject"/>
```

## 面试追问方向

每个知识点都可能延伸出追问，以下是常见方向：

1. **Flowable vs Camunda**：为什么选 Flowable？它们的核心区别是什么？
2. **异步执行**：Flowable 的 Job 和 Timer 有什么区别？
3. **子流程**：调用活动（Call Activity）和嵌入式子流程的区别是什么？
4. **事件监听**：ExecutionListener 和 TaskListener 分别在什么场景使用？
5. **集群部署**：Flowable 集群模式下如何保证定时任务不重复执行？
6. **数据存储**：Flowable 的表结构是怎么设计的？

这些追问方向，也是你深入学习的方向。
