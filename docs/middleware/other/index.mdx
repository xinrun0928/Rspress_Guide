# 其他中间件

在分布式系统的浩瀚版图中，有些组件既不属于数据存储的核心层，也不直接参与业务逻辑的计算，它们更像是隐藏在幕后的「支撑力量」——帮助系统稳定运转、让开发者在复杂场景下少走弯路。这些组件，就是我们要聊的「其他中间件」。

## 为什么单独成篇？

相比 Kafka、RocketMQ 这类消息中间件的「高调」，工作流引擎、配置中心、链路追踪、流量防护……这些组件往往在系统架构图里占据一个小角落，却在关键时刻发挥大作用。

它们的价值在于：

| 组件类型 | 解决的核心问题 |
|---|---|
| 工作流引擎 | 业务流程抽象与编排，把「人肉流程」变成「可配置、可追踪的数字化流程」 |
| ShardingSphere | 数据分片与读写分离，让单机数据库「横向扩展」 |
| Sentinel | 流量控制与熔断降级，给系统装上「过载保护」 |
| Canal | 数据库变更订阅，让数据同步「自动化」 |
| 分布式任务调度 | 跨节点任务编排，解决「谁在什么时间做什么事」的问题 |
| SLS / SkyWalking | 日志聚合与链路追踪，让分布式系统「透明可观测」 |
| Apollo | 配置中心集中管理，让配置变更「热更新、无感知」 |

## 模块速览

### 工作流引擎

当你需要处理「请假申请 → 主管审批 → HR 审批 → 抄送」这样的多步骤流程时，手写状态机可能是一场噩梦。Flowable 和 Camunda 提供了 BPMN 2.0 标准的流程引擎，让业务流程「可视化定义、代码化执行」。

### ShardingSphere

数据库单表数据量超过千万级？读写分离无法解决主库压力？ShardingSphere 提供了数据分片、读写分离、分布式事务等能力，让 MySQL 集群化成为可能。

### Sentinel

双十一零点，系统流量是平时的 100 倍。没有限流和熔断？等着数据库被打爆、服务雪崩。Sentinel 通过滑动窗口算法和熔断器模式，给系统装上了「智能限流阀」。

### Canal

MySQL 主从同步是天然的「数据订阅」能力，但应用层往往拿不到变更通知。Canal 伪装成 MySQL 从库，直接拉取 binlog，把数据库变更变成「消息队列」。

### 分布式任务调度

定时任务分散在多台服务器上？同一个任务被重复执行？任务执行失败没有补偿机制？Quartz、XXL-Job、ElasticJob 解决了「分布式环境下任务调度的协调问题」。

## 学习路径建议

面对这么多组件，不必一口气全部掌握。以下是推荐的学习顺序：

```
1. 先打基础：分布式任务调度（Quartz → XXL-Job）
2. 再学防护：Sentinel 流量控制
3. 理解数据：ShardingSphere 数据分片
4. 掌握协同：Canal 数据同步
5. 深入编排：工作流引擎
6. 选配组件：SLS / SkyWalking / Apollo
```

每一章都会尽量覆盖「是什么 → 为什么这样设计 → 怎么用 → 有什么坑」，帮助你从「会用」到「懂原理」。
