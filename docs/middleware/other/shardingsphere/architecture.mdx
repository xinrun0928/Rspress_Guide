# ShardingSphere 架构

ShardingSphere 提供两种部署模式：Sharding-JDBC 和 Sharding-Proxy。选择哪种模式，取决于你的业务场景和技术栈。

## 两种部署模式对比

| 维度 | Sharding-JDBC | Sharding-Proxy |
|---|---|---|
| 部署位置 | 应用进程内（Java） | 独立部署（任何语言） |
| 通信协议 | JDBC | MySQL/PostgreSQL 协议 |
| 数据库支持 | 任意支持 JDBC 的数据库 | MySQL / PostgreSQL |
| 多语言支持 | 仅 Java | 支持所有语言 |
| 性能损耗 | 极低（无网络开销） | 略有损耗（代理层） |
| 运维成本 | 低（无需额外部署） | 高（需要维护代理服务） |
| 配置方式 | 代码配置 / YAML | YAML / 配置中心 |

## Sharding-JDBC

Sharding-JDBC 定位为「Java 微服务框架」，它以 JAR 包的形式嵌入到应用进程中，直接在 JDBC 层拦截和改写 SQL。

### 架构原理

```
┌─────────────────────────────────────────────────────────────────────┐
│                         Sharding-JDBC 架构                          │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   ┌─────────────────────────────────────────────────────────────┐  │
│   │                      应用进程                                 │  │
│   │                                                             │  │
│   │   ┌───────────┐                                            │  │
│   │   │  业务代码  │                                            │  │
│   │   └─────┬─────┘                                            │  │
│   │         │                                                  │  │
│   │         ▼                                                  │  │
│   │   ┌───────────┐                                            │  │
│   │   │Sharding-  │ ← 改写 SQL、路由、结果合并                    │  │
│   │   │  JDBC     │                                            │  │
│   │   └─────┬─────┘                                            │  │
│   │         │                                                  │  │
│   │         ▼                                                  │  │
│   │   ┌───────────┐                                            │  │
│   │   │ DataSource │                                           │  │
│   │   │   Pool     │                                           │  │
│   │   └─────┬─────┘                                            │  │
│   │         │                                                  │  │
│   └─────────┼─────────────────────────────────────────────────┘  │
│             │                                                      │
│             ▼                                                      │
│   ┌───────────────┐       ┌───────────────┐                       │
│   │  数据节点 1    │       │  数据节点 2    │                       │
│   │  (ds_0)       │       │  (ds_1)       │                       │
│   └───────────────┘       └───────────────┘                       │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 核心优势

1. **零运维**：只需引入 JAR 包，无需额外部署服务
2. **高性能**：在 JVM 内部完成 SQL 改写，无网络开销
3. **对应用透明**：业务代码无需修改，只需配置
4. **轻量级**：对代码侵入小

### 适用场景

- Java 应用
- 微服务架构
- 无需多语言支持
- 追求高性能

## Sharding-Proxy

Sharding-Proxy 定位为「数据库中间件」，它独立部署在应用和数据库之间，模拟 MySQL/PostgreSQL 协议。

### 架构原理

```
┌─────────────────────────────────────────────────────────────────────┐
│                        Sharding-Proxy 架构                           │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   ┌───────────┐  ┌───────────┐  ┌───────────┐                       │
│   │ Java App  │  │ Python App│  │ Go App    │                       │
│   └─────┬─────┘  └─────┬─────┘  └─────┬─────┘                       │
│         │              │              │                              │
│         └──────────────┼──────────────┘                              │
│                        │                                            │
│                        ▼                                            │
│              ┌─────────────────────┐                                │
│              │   Sharding-Proxy    │                                │
│              │   ┌───────────────┐ │                                │
│              │   │  MySQL 协议    │ │                                │
│              │   │  SQL 改写      │ │                                │
│              │   │  路由引擎      │ │                                │
│              │   │  结果合并      │ │                                │
│              │   └───────────────┘ │                                │
│              └──────────┬──────────┘                                │
│                         │                                           │
│                         ▼                                           │
│              ┌───────────────┐       ┌───────────────┐             │
│              │  数据节点 1    │       │  数据节点 2    │             │
│              │  (ds_0)       │       │  (ds_1)       │             │
│              └───────────────┘       └───────────────┘             │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 核心优势

1. **多语言支持**：任何支持 MySQL/PostgreSQL 协议的语言都能使用
2. **数据库无关**：支持 MySQL 和 PostgreSQL
3. **运维友好**：独立部署，可独立升级
4. **配置中心集成**：支持与 Nacos、Apollo 等配置中心集成

### 适用场景

- 多语言异构系统
- 需要统一数据库访问入口
- 已有运维团队

## 核心概念

### 数据节点（DataNode）

数据分片后的最小存储单元：

```
ds_0
├── t_order_0
├── t_order_1
├── t_order_item_0
└── t_order_item_1

ds_1
├── t_order_0
├── t_order_1
├── t_order_item_0
└── t_order_item_1
```

### 分片算法

| 算法 | 说明 |
|---|---|
| HashMod | 取模分片 |
| Range | 范围分片 |
| Date | 日期分片 |
| Custom | 自定义分片 |

### 分片策略

| 策略 | 说明 |
|---|---|
| Inline | 字符串表达式分片 |
| Standard | 标准分片策略 |
| Complex | 复合分片策略 |
| Hint | 强制路由分片 |

## 配置对比

### Sharding-JDBC 配置

```yaml title="application.yml"
spring:
  shardingsphere:
    datasource:
      ds:
        master:
          data-source-class-name: com.zaxxer.hikari.HikariDataSource
          jdbc-url: jdbc:mysql://localhost:3306/ds_0
          username: root
          password: 
        slave:
          data-source-class-name: com.zaxxer.hikari.HikariDataSource
          jdbc-url: jdbc:mysql://localhost:3306/ds_1
          username: root
          password: 
    rules:
      sharding:
        tables:
          t_order:
            actual-data-nodes: ds.${0..1}.t_order_${0..1}
            table-strategy:
              standard:
                sharding-column: order_id
                sharding-algorithm-name: order_inline
```

### Sharding-Proxy 配置

```yaml title="server.yaml"
schemaName: sharding_db

dataSources:
  ds_0:
    url: jdbc:mysql://127.0.0.1:3306/ds_0?serverTimezone=UTC&useSSL=false
    username: root
    password: 
    connectionPoolClassName: com.zaxxer.hikari.HikariConnectionPool
  ds_1:
    url: jdbc:mysql://127.0.0.1:3306/ds_1?serverTimezone=UTC&useSSL=false
    username: root
    password: 
    connectionPoolClassName: com.zaxxer.hikari.HikariConnectionPool

shardingRule:
  tables:
    t_order:
      actualDataNodes: ds_${0..1}.t_order_${0..1}
      tableStrategy:
        standard:
          shardingColumn: order_id
          shardingAlgorithmName: order_inline
```

## 选择建议

| 场景 | 推荐方案 |
|---|---|
| Java 微服务 | Sharding-JDBC |
| 多语言异构系统 | Sharding-Proxy |
| 高性能要求 | Sharding-JDBC |
| 统一数据库入口 | Sharding-Proxy |
| 运维能力有限 | Sharding-JDBC |
| 已有数据库中间件 | Sharding-Proxy |

## 总结

ShardingSphere 两种模式的核心差异：

| 维度 | Sharding-JDBC | Sharding-Proxy |
|---|---|---|
| 部署方式 | JAR 包嵌入 | 独立部署 |
| 适用语言 | 仅 Java | 任何语言 |
| 性能 | 更高 | 略有损耗 |
| 运维 | 更简单 | 更复杂 |
| 多语言 | 不支持 | 支持 |
