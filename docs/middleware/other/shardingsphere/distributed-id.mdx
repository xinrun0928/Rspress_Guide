# ShardingSphere 分布式 ID 生成

数据分片后，需要一种能在分布式环境下生成全局唯一 ID 的方案。本地数据库自增无法跨分片唯一，UUID 缺乏业务含义且有序性差，Snowflake 雪花算法成为主流选择。

## 为什么需要分布式 ID？

分片后，订单 ID 需要跨所有分片唯一：

```
分片前：单库自增
┌─────────────┐
│ orders      │
│ id: 1,2,3...│
└─────────────┘

分片后：多分片自增冲突
┌─────────────┐     ┌─────────────┐
│ orders_0    │     │ orders_1    │
│ id: 1,2,3...│     │ id: 1,2,3...│  ← 冲突！
└─────────────┘     └─────────────┘
```

## 常见方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| 数据库自增 | 简单、连续 | 无法跨分片 | 单库 |
| UUID | 分布式唯一 | 无序、存储大 | 不推荐 |
| Snowflake | 高性能、有序 | 依赖时钟 | 主流方案 |
| 雪花算法变种 | 可定制 | 实现复杂 | 特殊需求 |
| 第三方组件 | 功能丰富 | 额外依赖 | 大型系统 |

## Snowflake 雪花算法

### 原理

Snowflake 将 64 位 ID 划分为多个部分：

```
┌─────────────────────────────────────────────────────────────────────┐
│                    Snowflake ID 结构（64 位）                        │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  ┌──────────┬────────────────────┬─────────────┬──────────────────┤│
│  │ 1 bit   │     41 bits        │  10 bits    │    12 bits      ││
│  ├──────────┼────────────────────┼─────────────┼──────────────────┤│
│  │  符号位   │      时间戳         │    机器 ID   │    序列号        ││
│  │  固定 0   │  (timestamp)       │  (datacenter │  (sequence)     ││
│  │           │                   │   + worker)  │                 ││
│  └──────────┴────────────────────┴─────────────┴──────────────────┘│
│                                                                     │
│  41 位时间戳：可使用 69 年                                          │
│  10 位机器 ID：支持 1024 个节点                                     │
│  12 位序列号：每节点每毫秒可生成 4096 个 ID                           │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 核心实现

```java
public class SnowflakeIdGenerator {

    // 起始时间戳（2020-01-01）
    private final long twepoch = 1577836800000L;

    // 机器 ID 位数
    private final long workerIdBits = 5L;
    private final long datacenterIdBits = 5L;

    // 最大值
    private final long maxWorkerId = ~(-1L << workerIdBits);    // 31
    private final long maxDatacenterId = ~(-1L << datacenterIdBits); // 31

    // 偏移量
    private final long workerIdShift = 12L;
    private final long datacenterIdShift = 17L;
    private final long timestampLeftShift = 22L;

    // 序列号掩码
    private final long sequenceMask = ~(-1L << 12L);

    private final long workerId;
    private final long datacenterId;
    private long sequence = 0L;
    private long lastTimestamp = -1L;

    public SnowflakeIdGenerator(long workerId, long datacenterId) {
        if (workerId > maxWorkerId || workerId < 0) {
            throw new IllegalArgumentException(
                "worker Id can't be greater than " + maxWorkerId);
        }
        if (datacenterId > maxDatacenterId || datacenterId < 0) {
            throw new IllegalArgumentException(
                "datacenter Id can't be greater than " + maxDatacenterId);
        }
        this.workerId = workerId;
        this.datacenterId = datacenterId;
    }

    public synchronized long nextId() {
        long timestamp = timeGen();

        // 时钟回拨处理
        if (timestamp < lastTimestamp) {
            throw new RuntimeException(
                "Clock moved backwards. Refusing to generate id");
        }

        // 同一毫秒内，序列号递增
        if (lastTimestamp == timestamp) {
            sequence = (sequence + 1) & sequenceMask;
            if (sequence == 0) {
                timestamp = tilNextMillis(lastTimestamp);
            }
        } else {
            sequence = 0L;
        }

        lastTimestamp = timestamp;

        return ((timestamp - twepoch) << timestampLeftShift)
                | (datacenterId << datacenterIdShift)
                | (workerId << workerIdShift)
                | sequence;
    }

    private long tilNextMillis(long lastTimestamp) {
        long timestamp = timeGen();
        while (timestamp <= lastTimestamp) {
            timestamp = timeGen();
        }
        return timestamp;
    }

    private long timeGen() {
        return System.currentTimeMillis();
    }
}
```

## ShardingSphere 内置 ID 生成器

### 配置

```yaml title="application.yml"
spring:
  shardingsphere:
    rules:
      sharding:
        tables:
          t_order:
            actual-data-nodes: ds_${0..1}.t_order_${0..3}
            key-generate-strategy:
              column: order_id
              key-generator-name: snowflake

        key-generators:
          snowflake:
            type: SNOWFLAKE
            props:
              worker-id: 1
              datacenter-id: 1
```

### 自动生成 ID

```java
@Service
public class OrderService {

    @Autowired
    private OrderRepository orderRepository;

    public void createOrder(Order order) {
        // ShardingSphere 自动生成 ID
        orderRepository.save(order);  // order_id 自动填充
    }
}
```

### 自定义 ID 生成器

```java
public class MyIdGenerator implements KeyGenerateAlgorithm {

    @Override
    public void init() {
        // 初始化
    }

    @Override
    public Comparable<?> generateKey() {
        // 生成 ID
        return UUID.randomUUID().toString().replace("-", "");
    }

    @Override
    public String getType() {
        return "UUID";
    }
}
```

```yaml
spring:
  shardingsphere:
    rules:
      sharding:
        key-generators:
          my_uuid:
            type: UUID
            class: com.example.MyIdGenerator
```

## 雪花算法变种

### 百度UidGenerator

改进点：

- 借用未来时间，解决时钟回拨
- 使用 RingBuffer 预生成 ID

```yaml
spring:
  shardingsphere:
    rules:
      sharding:
        key-generators:
          uid_generator:
            type: UIDGENERATOR
```

### 滴滴TinyId

改进点：

- 申请号段，减少数据库访问
- 本地生成，高性能

```yaml
spring:
  shardingsphere:
    rules:
      sharding:
        key-generators:
          tinyid_generator:
            type: TINYID
            props:
              mode: leaf
              segment:
                url: http://tinyid-server:9999/api/tinyid/segment/biz/order
```

### 美团Leaf

支持两种模式：

- 号段模式：批量获取号段
- 雪花模式：改进的雪花算法

## 时钟回拨问题

### 问题描述

```
时间线：
┌───────────────────────────────────────────────────────►
│        ▲
│        │ 时钟回拨
│        ▼
│   昨天 10:00 ──────► 明天 09:59 ──────► 今天 10:00
│                                      ↑
│                                     ID 可能重复！
```

### 解决方案

```java
public synchronized long nextId() {
    long timestamp = timeGen();

    // 方案 1：等待直到时间正确
    if (timestamp < lastTimestamp) {
        try {
            Thread.sleep(lastTimestamp - timestamp);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
        timestamp = timeGen();
    }

    // 方案 2：使用历史时间（百度 UidGenerator）
    if (timestamp < lastTimestamp) {
        timestamp = lastTimestamp;  // 使用上次时间
    }

    // 方案 3：抛出异常
    if (timestamp < lastTimestamp) {
        throw new RuntimeException("Clock moved backwards");
    }
    // ...
}
```

## 分片键与 ID 生成策略

### 方案一：ID 作为分片键

```yaml
rules:
  sharding:
    tables:
      t_order:
        actual-data-nodes: ds_${order_id % 2}.t_order_${order_id % 4}
        table-strategy:
          standard:
            sharding-column: order_id
            sharding-algorithm-name: order_id_mod
```

优点：单分片查询
缺点：无法按用户查询（跨分片）

### 方案二：业务字段作为分片键

```yaml
rules:
  sharding:
    tables:
      t_order:
        actual-data-nodes: ds_${user_id % 2}.t_order_${user_id % 4}
        key-generate-strategy:
          column: order_id
          key-generator-name: snowflake
        table-strategy:
          standard:
            sharding-column: user_id
            sharding-algorithm-name: user_id_mod
```

优点：按用户查询是单分片
缺点：订单 ID 查询需要广播

### 方案三：双分片键

```yaml
rules:
  sharding:
    tables:
      t_order:
        actual-data-nodes: ds_${user_id % 2}.t_order_${user_id % 4}
        key-generate-strategy:
          column: order_id
          key-generator-name: snowflake
        table-strategy:
          standard:
            sharding-column: user_id
            sharding-algorithm-name: user_id_mod

      t_order_index:
        # 订单 ID 到 user_id 的映射索引
        actual-data-nodes: ds_${order_id % 2}.t_order_index
        key-generate-strategy:
          column: id
          key-generator-name: snowflake
        table-strategy:
          standard:
            sharding-column: order_id
            sharding-algorithm-name: order_id_mod
```

## 总结

分布式 ID 生成方案对比：

| 方案 | 趋势递增 | 性能 | 存储 |
|---|---|---|---|
| 数据库自增 | 连续 | 低 | 8 字节 |
| UUID | 无序 | 高 | 36 字符 |
| Snowflake | 趋势递增 | 高 | 19 位数字 |
| 第三方号段 | 趋势递增 | 极高 | 19 位数字 |
