# ShardingSphere 数据分片

数据分片是 ShardingSphere 的核心能力。理解分片键、分片算法、分片策略，是用好 ShardingSphere 的基础。

## 分片键选择原则

分片键（Sharding Key）是决定数据分布到哪个分片的关键字段。选择好的分片键，能让数据均匀分布，减少跨分片查询。

### 选择原则

| 原则 | 说明 |
|---|---|
| 均匀分布 | 分片键的值域应尽量均匀 |
| 高频查询 | 优先选择查询条件中频繁使用的字段 |
| 避免热点 | 不要选择有明显热点的字段 |
| 单值查询 | 尽量支持单值查询，减少广播查询 |

### 常见分片键

| 业务场景 | 建议分片键 | 理由 |
|---|---|---|
| 订单表 | order_id | 查询频繁、分布均匀 |
| 用户表 | user_id | 查询基于用户、分布均匀 |
| 日志表 | created_date | 按日期归档、范围查询 |
| 店铺表 | merchant_id | 按店铺隔离 |

### 反面案例

```sql
-- 不好：地区有明显热点（北上广数据量远大于其他地区）
SELECT * FROM orders WHERE region = 'beijing'

-- 不好：状态字段分布不均（90% 是已完成）
SELECT * FROM orders WHERE status = 'completed'
```

## 分片算法

### HashMod 算法

最常用的分片算法，通过取模确定目标分片：

```yaml
rules:
  sharding:
    tables:
      t_order:
        table-rule:
          actual-data-nodes: ds_${0..7}.t_order
          database-strategy:
            standard:
              sharding-column: user_id
              sharding-algorithm-name: database_hash_mod
          table-strategy:
            standard:
              sharding-column: order_id
              sharding-algorithm-name: table_hash_mod

  sharding-algorithms:
    database_hash_mod:
      type: INLINE
      props:
        algorithm-expression: ds_${user_id % 8}

    table_hash_mod:
      type: INLINE
      props:
        algorithm-expression: t_order_${order_id % 16}
```

**分片公式：**

```
分片数 = 分片键值 % 分片数量

示例：
user_id = 123456
分片数量 = 8
目标分片 = 123456 % 8 = 0
→ ds_0
```

### 范围分片算法

按数值范围分片，适合有序增长的字段：

```yaml
rules:
  sharding:
    tables:
      t_order:
        actual-data-nodes: ds_${0..1}.t_order_${0..3}

  sharding-algorithms:
    range:
      type: CLASS_BASED
      props:
        strategy: STANDARD
        algorithmClassName: com.example.RangeShardingAlgorithm
```

```java
public class RangeShardingAlgorithm implements RangeShardingAlgorithm<Long> {

    @Override
    public Collection<String> doSharding(
            Collection<String> availableTargetNames,
            RangeShardingValue<Long> shardingValue) {

        List<String> result = new ArrayList<>();
        Range<Long> valueRange = shardingValue.getValueRange();

        // 范围查询：根据起止值确定分片
        Long lower = valueRange.lowerEndpoint();
        Long upper = valueRange.upperEndpoint();

        for (long i = lower; i <= upper; i++) {
            result.add("ds_" + (i / 10000) % 2 + ".t_order_" + (i % 4));
        }

        return result;
    }
}
```

### 日期分片算法

按日期分片，适合日志、订单等有时间属性的表：

```yaml
rules:
  sharding:
    tables:
      t_log:
        actual-data-nodes: ds_0.t_log_${202301..202312}
        table-strategy:
          standard:
            sharding-column: created_date
            sharding-algorithm-name: date_sharding

  sharding-algorithms:
    date_sharding:
      type: INTERVAL
      props:
        datetime-pattern: "yyyy-MM-dd"
        datetime-lower: "2023-01-01"
        datetime-upper: "2023-12-31"
        sharding-suffix-pattern: "yyyyMM"
        partition-max-digits: 2
```

```sql
-- 查询 2023 年 3 月的数据
SELECT * FROM t_log WHERE created_date BETWEEN '2023-03-01' AND '2023-03-31'
-- 实际路由到：t_log_202303
```

### 自定义分片算法

实现 `ShardingAlgorithm` 接口：

```java
public class OrderShardingAlgorithm implements ShardingAlgorithm<Long> {

    @Override
    public Collection<String> doSharding(
            Collection<String> availableTargetNames,
            PreciseShardingValue<Long> shardingValue) {

        long orderId = shardingValue.getValue();
        String logicTableName = shardingValue.getLogicTableName();

        // 自定义分片逻辑
        // 按用户 ID 分库，按订单 ID 分表
        int database = (int) (orderId / 1000000) % 4;
        int table = (int) (orderId % 1000000) % 16;

        return Collections.singletonList(
            "ds_" + database + "." + logicTableName + "_" + table
        );
    }
}
```

## 分片策略

### 标准分片策略

使用一个分片键：

```yaml
rules:
  sharding:
    tables:
      t_order:
        table-strategy:
          standard:
            sharding-column: order_id
            sharding-algorithm-name: order_inline
```

### 复合分片策略

使用多个分片键：

```yaml
rules:
  sharding:
    tables:
      t_order_item:
        table-strategy:
          complex:
            sharding-columns: order_id, user_id
            sharding-algorithm-name: order_item_complex

  sharding-algorithms:
    order_item_complex:
      type: CLASS_BASED
      props:
        strategy: COMPLEX
        algorithmClassName: com.example.OrderItemComplexShardingAlgorithm
```

```java
public class OrderItemComplexShardingAlgorithm
        implements ComplexKeysShardingAlgorithm<Long> {

    @Override
    public Collection<String> doSharding(
            Collection<String> availableTargetNames,
            ComplexKeysShardingValue<Long> shardingValue) {

        Long orderId = shardingValue.getColumnNameAndShardingValuesOfTable()
            .get("order_id");
        Long userId = shardingValue.getColumnNameAndShardingValuesOfTable()
            .get("user_id");

        // 根据 order_id 和 user_id 联合分片
        int database = (int) ((orderId + userId) % 4);
        int table = (int) (orderId % 16);

        return Collections.singleton("ds_" + database + ".t_order_item_" + table);
    }
}
```

### Hint 强制路由

通过 Hint 强制指定分片，不依赖 SQL 解析：

```java
@Test
public void testHintSharding() {
    HintManager hintManager = HintManager.getInstance();

    // 强制路由到指定分片
    hintManager.addDatabaseShardingValue("t_order", "order_id", 12345L);
    hintManager.addTableShardingValue("t_order", "order_id", 12345L);

    // 查询
    List<Order> orders = orderRepository.findAll();

    hintManager.close();
}
```

```yaml
rules:
  sharding:
    tables:
      t_order:
        table-strategy:
          standard:
            sharding-column: order_id
            sharding-algorithm-name: order_inline
          hint:
            sharding-algorithm-name: order_hint
```

## 绑定表

绑定表是具有相同分片键的表，跨表查询时无需结果合并：

```yaml
rules:
  sharding:
    tables:
      t_order:
        actual-data-nodes: ds_${0..1}.t_order_${0..1}
        binding-tables:
          - t_order_item

      t_order_item:
        actual-data-nodes: ds_${0..1}.t_order_item_${0..1}
```

```sql
-- 绑定表查询
SELECT * FROM t_order o
JOIN t_order_item i ON o.order_id = i.order_id
WHERE o.user_id = 12345

-- 绑定表内可以避免笛卡尔积
-- 非绑定表会产生笛卡尔积
```

## 分片配置最佳实践

### 垂直分片 vs 水平分片

| 类型 | 说明 | 场景 |
|---|---|---|
| 垂直分片 | 按业务拆分到不同库 | 业务隔离、数据量适中 |
| 水平分片 | 按数据拆分到多个库表 | 数据量大、高并发 |

### 分片数量规划

```
分片数量 = 数据总量 / 单表容量

示例：
- 数据总量：1 亿条
- 单表容量：100 万条
- 分片数量：100 个
- 分库数 × 分表数 = 100
  - 方案 1：4 库 × 25 表
  - 方案 2：10 库 × 10 表
```

### 避免跨分片

```sql
-- 尽量避免跨分片查询
-- 好的设计
SELECT * FROM t_order WHERE order_id = 12345  -- 单分片查询
SELECT * FROM t_order WHERE user_id = 100     -- 广播查询，可接受

-- 不好的设计
SELECT * FROM t_order WHERE order_id IN (1, 2, 3, 4, 5)  -- 可能跨多个分片
SELECT COUNT(*) FROM t_order WHERE created_date > '2023-01-01'  -- 全分片扫描
```

## 总结

分片核心配置：

| 配置项 | 说明 |
|---|---|
| `sharding-column` | 分片键 |
| `actual-data-nodes` | 实际数据节点 |
| `sharding-algorithm` | 分片算法 |
| `binding-tables` | 绑定表 |
