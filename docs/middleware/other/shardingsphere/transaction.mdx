# ShardingSphere 分布式事务

当数据分片后，一条业务操作可能涉及多个数据库分片。此时本地事务无法保证跨库的数据一致性，分布式事务成为必然选择。

## 分布式事务的挑战

没有分布式事务时：

```
用户下单：
1. 扣减库存（ds_0.inventory 表）
2. 创建订单（ds_1.orders 表）

问题：如果步骤 1 成功，步骤 2 失败
→ 库存扣了，订单没创建
→ 数据不一致！
```

## 分布式事务模式

ShardingSphere 支持多种分布式事务模式：

```
┌─────────────────────────────────────────────────────────────────────┐
│                         分布式事务模式                               │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  ┌─────────────────────┐  ┌─────────────────────┐                 │
│  │   XA 两阶段提交      │  │   Saga 柔性事务     │                 │
│  │                     │  │                     │                 │
│  │  强一致性            │  │  最终一致性          │                 │
│  │  性能损耗高          │  │  性能损耗低          │                 │
│  │  ACID               │  │  BASE               │                 │
│  └─────────────────────┘  └─────────────────────┘                 │
│                                                                     │
│  ┌─────────────────────┐  ┌─────────────────────┐                 │
│  │   TCC 柔性事务      │  │   最大努力通知      │                 │
│  │                     │  │                     │                 │
│  │  最终一致性          │  │  最终一致性          │                 │
│  │  需要改业务代码      │  │  实现简单            │                 │
│  │  Try-Confirm-Cancel │  │  回调确认            │                 │
│  └─────────────────────┘  └─────────────────────┘                 │
│                                                                     │

```

## XA 两阶段提交

### 原理

```
┌─────────────────────────────────────────────────────────────────────┐
│                         XA 两阶段提交                               │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   阶段一：Prepare（准备阶段）                                         │
│                                                                     │
│              ┌─────────┐                                            │
│              │ 应用层   │                                            │
│              └────┬────┘                                            │
│                   │                                                  │
│         ┌────────┴────────┐                                        │
│         ▼                 ▼                                        │
│   ┌───────────┐     ┌───────────┐                                  │
│   │ RM_1     │     │ RM_2     │                                  │
│   │ (ds_0)   │     │ (ds_1)   │                                  │
│   └─────┬─────┘     └─────┬─────┘                                  │
│         │                 │                                        │
│    Prepare()          Prepare()                                    │
│         │                 │                                        │
│    [锁定资源]        [锁定资源]                                      │
│         │                 │                                        │
│    XA Prepared       XA Prepared                                   │
│         │                 │                                        │
│         └────────┬────────┘                                        │
│                  ▼                                                  │
│            ┌─────────┐                                              │
│            │ TM      │                                              │
│            │ (事务管理器)│                                           │
│            └────┬────┘                                              │
│                 │                                                  │
│   阶段二：Commit / Rollback                                         │
│                 │                                                  │
│         ┌───────┴───────┐                                          │
│         ▼               ▼                                          │
│    XA Commit        XA Rollback                                    │
│    (提交)            (回滚)                                         │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### ShardingSphere XA 配置

```yaml title="application.yml"
spring:
  shardingsphere:
    rules:
      # 启用 XA 事务
      transaction:
        default-type: XA
        provider-type: Atomikos

    # 配置 Atomikos
    datasource:
      ds_0:
        data-source-class-name: com.zaxxer.hikari.HikariDataSource
        jdbc-url: jdbc:mysql://localhost:3306/ds_0
        username: root
        password: 

# 配置 Atomikos 事务日志
jta:
  atomikos:
    datasource:
      default:
        xa-properties:
          databaseName: ds_0
          serverName: server_0
    transaction-manager:
      default:
        transaction-coordination-directory: /tmp/atomikos
```

### XA 使用示例

```java
@Service
public class OrderService {

    @Transactional  // 分布式事务
    public void createOrder(Order order) {
        // 扣减库存（ds_0）
        inventoryService.deduct(order.getProductId(), order.getQuantity());

        // 创建订单（ds_1）
        orderRepository.save(order);

        // 扣减余额（ds_2）
        accountService.deduct(order.getUserId(), order.getAmount());
    }
}
```

### XA 的问题

| 问题 | 说明 | 解决方案 |
|---|---|---|
| 锁定时间长 | Prepare 后锁定资源，直到 Commit | 缩短事务时间 |
| 协调者故障 | TM 故障导致数据不一致 | 使用高可用 TM |
| 性能损耗 | 多次网络通信 | 异步优化 |

## Saga 柔性事务

### 原理

Saga 将长事务拆分为多个本地事务，每个事务都有对应的补偿操作：

```
┌─────────────────────────────────────────────────────────────────────┐
│                           Saga 模式                                  │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   业务操作：                                                          │
│   ───────────────────────────────────────────────────────────────  │
│   T1: 扣减库存    →  成功  →  T2: 创建订单  →  失败  →  T3: 补偿    │
│                                        ↓                            │
│                                    发起补偿操作：                      │
│                                    T1: 回滚库存                      │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### Saga 配置

```yaml title="application.yml"
spring:
  shardingsphere:
    rules:
      transaction:
        default-type: BASE
        provider-type: Saga

      # Saga 状态机配置
      - !SAGA
        machines:
          - name: order_saga_machine
            resources:
              - id: inventory
                type: service
                class: com.example.InventoryService
                method: deduct
                compensation:
                  type: service
                  class: com.example.InventoryService
                  method: restore
              - id: order
                type: service
                class: com.example.OrderService
                method: create
                compensation:
                  type: service
                  class: com.example.OrderService
                  method: cancel
```

### Saga 使用示例

```java
@SagaStart
public void createOrder(Order order) {
    // 扣减库存
    inventoryService.deduct(order.getProductId(), order.getQuantity());

    // 创建订单（失败会触发补偿）
    orderRepository.save(order);
}

@Compensable(compensationMethod = "restore")
public void deduct(Long productId, Integer quantity) {
    // 扣减库存
    inventoryRepository.decreaseStock(productId, quantity);
}

public void restore(Long productId, Integer quantity) {
    // 补偿：恢复库存
    inventoryRepository.increaseStock(productId, quantity);
}
```

## TCC 柔性事务

### 原理

TCC 将业务逻辑拆分为 Try、Confirm、Cancel 三个阶段：

```
┌─────────────────────────────────────────────────────────────────────┐
│                           TCC 模式                                   │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   Try（预留资源）：                                                   │
│   - 检查库存是否充足                                                  │
│   - 冻结库存（不扣减，标记为已冻结）                                   │
│   - 预留库存（数量冻结）                                              │
│                                                                     │
│   Confirm（确认执行）：                                              │
│   - 真正扣减冻结的库存                                                │
│   - 创建订单                                                         │
│                                                                     │
│   Cancel（取消预留）：                                               │
│   - 释放冻结的库存                                                    │
│   - 不创建订单                                                        │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### TCC 使用示例

```java
public interface InventoryTccService {

    @TwoPhaseBusiness(action = "deduct")
    boolean deduct(BusinessActionContext context,
                   @BusinessActionContextParameter(paramName = "productId") Long productId,
                   @BusinessActionContextParameter(paramName = "quantity") Integer quantity);
}

@Service
public class InventoryTccServiceImpl implements InventoryTccService {

    @Override
    @Transactional
    public boolean deduct(BusinessActionContext context,
                         Long productId, Integer quantity) {
        // Try 阶段：冻结库存
        inventoryRepository.freezeStock(productId, quantity);
        return true;
    }

    @Transactional
    public boolean confirm(BusinessActionContext context) {
        // Confirm 阶段：真正扣减冻结的库存
        Long productId = (Long) context.getActionContext().get("productId");
        Integer quantity = (Integer) context.getActionContext().get("quantity");
        inventoryRepository.confirmFreeze(productId, quantity);
        return true;
    }

    @Transactional
    public boolean cancel(BusinessActionContext context) {
        // Cancel 阶段：释放冻结的库存
        Long productId = (Long) context.getActionContext().get("productId");
        Integer quantity = (Integer) context.getActionContext().get("quantity");
        inventoryRepository.unfreezeStock(productId, quantity);
        return true;
    }
}
```

## 模式对比

| 维度 | XA | Saga | TCC |
|---|---|---|---|
| 一致性 | 强一致 | 最终一致 | 最终一致 |
| 性能 | 低 | 高 | 中 |
| 侵入性 | 低 | 低 | 高 |
| 锁定资源 | 是 | 否 | 否 |
| 补偿机制 | 回滚 | 补偿 | 预留/确认/取消 |
| 适用场景 | 强一致要求 | 长事务 | 低延迟 |

## 分布式事务最佳实践

### 1. 优先使用本地事务

```
分库后，一条 SQL 能解决的，不要拆成多条

# 好：单库事务
INSERT INTO order SELECT ... FROM inventory WHERE stock >= 1

# 不好：分布式事务
inventoryService.deduct() + orderService.create()
```

### 2. 避免长事务

```java
// 不好：长事务，锁定资源时间长
@Transactional
public void createOrderWithMultipleSteps() {
    inventoryService.deduct(productId, quantity);  // 10ms
    orderService.create(order);                     // 100ms
    accountService.deduct(userId, amount);         // 50ms
    // ... 更多操作
}

// 好：拆分短事务
public void createOrder() {
    // 1. 先扣库存（事务 1）
    inventoryService.deductAsync(productId, quantity);

    // 2. 创建订单（事务 2）
    orderService.create(order);
}
```

### 3. 幂等性设计

```java
@Retryable(include = Exception.class, maxAttempts = 3)
public void deduct(Long productId, Integer quantity) {
    // 添加幂等性检查
    String idempotentKey = "deduct_" + productId + "_" + quantity;
    if (idempotentRepository.exists(idempotentKey)) {
        return;  // 已执行过，跳过
    }

    // 执行扣减
    inventoryRepository.decreaseStock(productId, quantity);
    idempotentRepository.save(idempotentKey);
}
```

## 总结

分布式事务核心模式：

| 模式 | 一致性 | 性能 | 侵入性 |
|---|---|---|---|
| XA | 强一致 | 低 | 低 |
| Saga | 最终一致 | 高 | 低 |
| TCC | 最终一致 | 中 | 高 |
