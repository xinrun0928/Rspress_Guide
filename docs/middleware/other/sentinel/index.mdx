# Sentinel：流量控制与熔断降级

想象一下这个场景：

零点秒杀活动开始了，瞬间涌入的流量是平时的 100 倍。你的服务器开始疯狂报错——数据库连接池耗尽、CPU 打满、接口全部超时。

用户怨声载道，你被轮番轰炸。

**如果能在流量冲击系统之前，就把它拦住呢？**

这不是天方夜谭。Sentinel 就是来解决这个问题的——它像一个智能的流量调度员，知道什么时候该放行、什么时候该排队、什么时候该直接拒绝。

---

## 什么是 Sentinel？

Sentinel 是阿里巴巴开源的流量控制组件，主打**流量控制**和**熔断降级**两大核心能力。它的设计理念是：**让系统在保证稳定性的前提下，尽可能多地服务请求**。

和 Hystrix 不同，Sentinel 不仅仅关注「熔断」，更关注「流量塑形」——它可以在系统即将过载时就开始控制流量，而不是等到已经崩溃了才行动。

### 核心能力一览

| 能力 | 说明 | 解决的问题 |
|-----|------|----------|
| 流量控制 | 控制每秒/每分钟能处理的请求数 | 防止瞬时流量冲击 |
| 熔断降级 | 当响应时间或异常比例过高时，暂时「熔断」调用 | 防止级联故障扩散 |
| 系统自适应 | 从系统维度限制整体负载 | 保护整个系统不被压垮 |
| 注解支持 | 通过注解声明式定义资源 | 业务代码与流控逻辑分离 |

## 快速入门

```java
// 引入依赖
// <dependency>
//     <groupId>com.alibaba.csp</groupId>
//     <artifactId>sentinel-core</artifactId>
//     <version>1.8.6</version>
// </dependency>

// 定义资源：秒杀接口
try (Entry entry = SphU.entry("seckill")) {
    // 业务逻辑
    seckillService.execute();
} catch (BlockException e) {
    // 被限流了，返回友好提示
    return Result.fail("系统繁忙，请稍后重试");
}
```

Sentinel 的工作流程很简单：**统计 → 判断 → 决策**。业务代码只需要关注「被拒绝后怎么办」。

## 核心概念

在深入学习之前，你需要了解三个核心概念：

- **资源**：你要保护的东西（接口、方法、外部调用）
- **规则**：怎么保护它（限流阈值、熔断条件）
- **插槽链**：谁来执行保护逻辑（统计、限流、熔断的串联）

这三个概念是 Sentinel 的灵魂。理解了它们，你就掌握了 Sentinel 的设计哲学。

## 写在最后

Sentinel 的设计哲学是「预防优于治疗」——不等到系统崩溃才行动，而是在系统承压时就开始控制流量。

这种设计思想，其实也适用于生活：**不要等到问题爆发才去解决，要在苗头出现时就介入**。
