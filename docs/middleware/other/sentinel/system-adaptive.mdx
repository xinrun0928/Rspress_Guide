# Sentinel 系统自适应限流

系统自适应限流根据系统的实时负载情况自动调整限流阈值，比静态配置更加智能和灵活。

## 为什么需要系统自适应？

静态限流的问题是：阈值固定，无法适应系统的动态变化。

```
静态限流的困境：

场景 1：系统空闲
配置 QPS=100，实际负载 10
→ 限流 100 太多，浪费资源

场景 2：系统繁忙
配置 QPS=100，实际负载已达 10000
→ 限流 100 太少，无法保护系统
```

系统自适应根据系统真实负载动态调整，保护的是「系统本身」，而不是单个资源。

## 系统保护指标

Sentinel 支持多种系统保护指标：

| 指标 | 说明 | 阈值类型 |
|---|---|---|
| Load（系统负载） | Linux `loadavg` | 参考值 |
| RT（平均响应时间） | 所有请求的平均 RT | 阈值 |
| 并发线程数 | 当前处理中的线程数 | 阈值 |
| QPS | 当前系统的 QPS | 阈值 |
| CPU 使用率 | CPU 使用百分比 | 参考值 |

## 配置系统规则

### 基本配置

```java
SystemRule rule = new SystemRule();
rule.setHighestSystemLoad(3.0);    // 最高系统负载
rule.setQps(1000);                 // 最高 QPS
rule.setAvgRt(5000);               // 最高平均 RT（ms）
rule.setMaxThread(100);            // 最高并发线程数

SystemRuleManager.loadRules(Collections.singletonList(rule));
```

### 多指标组合

```java
List<SystemRule> rules = Arrays.asList(
    // 按系统负载限流
    new SystemRule()
        .setHighestSystemLoad(3.0)
        .setMetricType(SystemRuleMetric.LOAD),

    // 按 CPU 使用率限流
    new SystemRule()
        .setHighestCpuUsage(0.8)
        .setMetricType(SystemRuleMetric.CPUUsage),

    // 按并发线程数限流
    new SystemRule()
        .setMaxThread(100)
        .setMetricType(SystemRuleMetric.Concurrency)
);

// 任意一个条件触发都会限流
SystemRuleManager.loadRules(rules);
```

## 系统自适应原理

### 滑动窗口计算

```
┌─────────────────────────────────────────────────────────────────┐
│ 系统自适应滑动窗口                                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                     │
│   请求入口                                                         │
│       │                                                           │
│       ▼                                                           │
│   ┌─────────────┐                                                │
│   │  统计模块   │                                                │
│   │             │                                                │
│   │  1. RT 统计 │ ──→ 当前平均 RT                               │
│   │  2. QPS 统计│ ──→ 当前 QPS                                  │
│   │  3. 线程统计│ ──→ 当前并发线程数                             │
│   │  4. Load  │ ──→ 系统 Load                                   │
│   └─────────────┘                                                │
│       │                                                           │
│       ▼                                                           │
│   ┌─────────────┐                                                │
│   │   规则匹配  │                                                │
│   │             │                                                │
│   │  Load > 3? │ ──→ 是 → 限流                                  │
│   │  QPS > 1000?│ ──→ 是 → 限流                                  │
│   │  RT > 5000? │ ──→ 是 → 限流                                  │
│   └─────────────┘                                                │
│       │                                                           │
│       ▼                                                           │
│   ┌─────────────┐                                                │
│   │  通过/拒绝  │                                                │
│   └─────────────┘                                                │
│                                                                     │
└─────────────────────────────────────────────────────────────────┘
```

### 阈值自适应计算

Sentinel 会根据系统当前状态动态调整实际生效的阈值：

```java
public class AdaptiveThreshold {

    // 根据系统负载动态调整
    public double getAdaptiveThreshold(double systemLoad, double staticThreshold) {
        // 系统负载高时，降低阈值
        if (systemLoad > 1.0) {
            return staticThreshold * 0.5;
        }
        // 系统负载正常，使用静态阈值
        return staticThreshold;
    }
}
```

## 触发条件

### Load（系统负载）

```
Load 触发条件：

当前 Load > 阈值 → 限流

示例：
阈值: 3.0
当前 Load: 4.5
→ 触发限流

适用场景：
- Linux 系统负载（loadavg / CPU 核数）
- 容器/虚拟机 CPU 排队情况
```

### QPS

```
QPS 触发条件：

当前 QPS > 阈值 → 限流

示例：
阈值: 1000
当前 QPS: 1500
→ 触发限流

适用场景：
- API 网关入口
- Web 服务入口
```

### RT（平均响应时间）

```
RT 触发条件：

当前平均 RT > 阈值 → 限流

示例：
阈值: 5000ms
当前平均 RT: 6000ms
→ 触发限流

适用场景：
- 后端服务保护
- 数据库连接池保护
```

### 并发线程数

```
并发线程数触发条件：

当前并发线程数 > 阈值 → 限流

示例：
阈值: 100
当前并发: 150
→ 触发限流

适用场景：
- 线程池保护
- 连接池保护
```

## 实际配置建议

### API 网关场景

```java
// 网关入口保护
SystemRule rule = new SystemRule()
    .setQps(10000)        // 网关 QPS 上限
    .setAvgRt(100)        // 平均响应时间上限
    .setMaxThread(200);   // 最大并发

SystemRuleManager.loadRules(Collections.singletonList(rule));
```

### 后端服务场景

```java
// 后端服务保护
SystemRule rule = new SystemRule()
    .setHighestSystemLoad(3.0)  // 系统负载
    .setMaxThread(100)          // 最大并发线程
    .setAvgRt(2000);            // 平均 RT

SystemRuleManager.loadRules(Collections.singletonList(rule));
```

### 高并发促销场景

```java
// 促销场景
SystemRule rule = new SystemRule()
    .setHighestSystemLoad(5.0)  // 允许更高负载
    .setQps(50000)              // QPS 上限
    .setAvgRt(50);              // RT 上限

SystemRuleManager.loadRules(Collections.singletonList(rule));
```

## 系统自适应 vs 单机限流

| 维度 | 单机限流 | 系统自适应 |
|---|---|---|
| 阈值来源 | 静态配置 | 动态计算 |
| 保护目标 | 单个资源 | 整个系统 |
| 配置复杂度 | 高（需要预估） | 低（自动计算） |
| 适用场景 | 资源级别 | 系统级别 |

## 总结

系统自适应限流核心配置：

| 配置项 | 说明 |
|---|---|
| `highestSystemLoad` | 最高系统负载（Load） |
| `qps` | 最高 QPS |
| `avgRt` | 最高平均响应时间 |
| `maxThread` | 最高并发线程数 |

系统自适应限流的价值：

- 动态适应系统负载变化
- 无需精确配置阈值
- 保护整个系统而不是单个资源
