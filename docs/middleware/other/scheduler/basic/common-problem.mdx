# 定时任务常见问题

分布式环境下的定时任务，有几个经典问题需要解决。

## 问题一：重复执行

### 场景

多台服务器都配置了同一个定时任务，到点全部触发。

```
问题：
服务器 A: 定时器触发 → 执行任务
服务器 B: 定时器也触发 → 执行任务
服务器 C: 定时器也触发 → 执行任务
→ 任务执行了 3 次！
```

### 解决方案

| 方案 | 说明 |
|---|---|
| 分布式锁 | 只有获取锁的服务器执行 |
| 调度中心 | 调度中心只分配一个执行器执行 |
| 数据库标记 | 抢任务标记，执行后清除 |

```java
// 分布式锁方案
@Scheduled(cron = "0 0 2 * * ?")
public void dailyTask() {
    // 尝试获取锁
    String lockKey = "task:dailySettlement";
    Boolean acquired = redisTemplate.opsForValue()
        .setIfAbsent(lockKey, "locked", Duration.ofMinutes(30));

    if (Boolean.TRUE.equals(acquired)) {
        try {
            // 执行任务
            doSettlement();
        } finally {
            // 释放锁
            redisTemplate.delete(lockKey);
        }
    }
}
```

## 问题二：时钟漂移

### 场景

不同服务器的时钟不完全同步，导致任务执行时间不一致。

```
问题：
服务器 A: 时间快 2 秒 → 2:00:02 触发
服务器 B: 时间准确 → 2:00:00 触发
服务器 C: 时间慢 2 秒 → 1:59:58 触发
→ 执行时间差 4 秒
```

### 解决方案

| 方案 | 说明 |
|---|---|
| NTP 同步 | 所有服务器同步 NTP 时间 |
| 使用调度中心时间 | 以调度中心时间为准 |
| 时间窗口 | 设置执行时间窗口，在窗口内执行即可 |

```java
// 时间窗口方案
@Scheduled(cron = "0 0 2 * * ?")
public void dailyTask() {
    // 获取当前时间的分钟数
    int minuteOfDay = LocalTime.now().getHour() * 60 + LocalTime.now().getMinute();

    // 在窗口内才执行（2:00 - 2:05）
    if (minuteOfDay >= 120 && minuteOfDay <= 125) {
        // 只有满足时间窗口才执行
        doSettlement();
    }
}
```

## 问题三：任务堆积

### 场景

任务执行时间超过调度间隔，导致任务堆积。

```
问题：
2:00 开始执行 → 预计 2:30 完成
2:05 再次触发 → 任务堆积！
2:10 再次触发 → 任务堆积！
```

### 解决方案

| 方案 | 说明 |
|---|---|
| 禁止并发执行 | 任务执行中不触发新执行 |
| 错开调度 | 调整 cron 表达式 |
| 任务队列 | 任务入队，顺序执行 |
| 监控告警 | 任务堆积时告警 |

```java
// 禁止并发
@DisallowConcurrentExecution
public class MyJob implements Job {
    @Override
    public void execute(JobExecutionContext context) throws JobExecutionException {
        // 上一个执行完才执行下一个
    }
}

// Spring 方式
@Scheduled(cron = "0 0 2 * * ?", scheduledAsynchronous = false)
public void dailyTask() {
    // scheduledAsynchronous = false 表示同步执行
}
```

## 问题四：执行失败

### 场景

任务执行过程中抛出异常，没有处理。

### 解决方案

| 方案 | 说明 |
|---|---|
| try-catch | 捕获异常，记录日志 |
| 重试机制 | 失败后自动重试 |
| 死信队列 | 重试失败后进入死信队列 |
| 告警通知 | 失败后发送告警 |

```java
@Scheduled(cron = "0 0 2 * * ?")
public void dailyTask() {
    int maxRetries = 3;
    for (int i = 0; i < maxRetries; i++) {
        try {
            doSettlement();
            break;  // 成功就退出
        } catch (Exception e) {
            if (i == maxRetries - 1) {
                // 最后一次也失败
                sendAlert("对账任务失败", e);
            } else {
                // 等待后重试
                try {
                    Thread.sleep(5000);
                } catch (InterruptedException ie) {
                    Thread.currentThread().interrupt();
                }
            }
        }
    }
}
```

## 问题五：幂等性

### 场景

任务执行多次不会产生副作用。

### 解决方案

| 方案 | 说明 |
|---|---|
| 唯一键 | 使用唯一标识避免重复处理 |
| 状态机 | 按状态流转处理 |
| 去重表 | 执行前检查是否已处理 |

```java
// 去重表方案
public void dailyTask() {
    String taskId = "settlement:" + LocalDate.now();

    // 检查是否已执行
    if (idempotentService.isExecuted(taskId)) {
        log.info("任务已执行，跳过");
        return;
    }

    try {
        // 标记开始
        idempotentService.markStarted(taskId);

        // 执行任务
        doSettlement();

        // 标记完成
        idempotentService.markCompleted(taskId);
    } catch (Exception e) {
        // 标记失败
        idempotentService.markFailed(taskId);
        throw e;
    }
}
```

## 总结

| 问题 | 解决方案 |
|---|---|
| 重复执行 | 分布式锁、调度中心、数据库标记 |
| 时钟漂移 | NTP 同步、使用调度中心时间、时间窗口 |
| 任务堆积 | 禁止并发、错开调度、任务队列 |
| 执行失败 | try-catch、重试机制、死信队列 |
| 幂等性 | 唯一键、状态机、去重表 |
