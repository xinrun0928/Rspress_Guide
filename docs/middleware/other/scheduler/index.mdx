# 分布式任务调度

当系统需要在特定时间执行任务时，你可能会想到定时器。但如果任务分布在多台服务器上、如何保证任务不重复执行、如何处理执行失败……问题就复杂了。

分布式任务调度，就是解决这些问题的。

## 为什么需要分布式任务调度？

### 单机调度的困境

```
场景：每天凌晨 2 点执行数据对账任务

问题 1：重复执行
服务器 A：定时器触发，执行对账
服务器 B：定时器也触发，执行对账
→ 对账执行了两次，数据不一致

问题 2：服务器宕机
服务器 A：宕机了
→ 定时任务无法执行，数据没有对账

问题 3：任务状态丢失
服务器 A：正在执行，JVM 重启
→ 任务执行到一半，状态丢失
```

### 分布式调度解决

```
分布式调度器（XXL-Job）：
├── 调度中心（Admin）
│   ├── 任务管理
│   ├── 触发执行
│   └── 日志查询
└── 执行器（Executor）
    ├── 接收调度指令
    ├── 执行任务
    └── 汇报结果

优势：
✓ 一台执行器执行，其他不执行
✓ 执行器宕机，调度中心自动切换
✓ 任务状态持久化
```

## 模块学习路径

```
1. 任务调度基础：核心概念
2. 单机 vs 分布式：为什么需要分布式
3. 常见问题：重复执行、时钟漂移、任务堆积
4. 选型对比：Quartz vs XXL-Job vs ElasticJob
5. Quartz：核心概念与集群原理
6. XXL-Job：架构与任务类型
7. ElasticJob：分片与监听器
```

> [!TIP]
> 分布式任务调度的核心问题是「谁在什么时间做什么事，以及如何保证只做一次」。
