# 网关过滤器链：设计模式与执行原理

你可能听过「过滤器链」这个词，但不一定清楚它背后的设计思想。为什么网关要设计成链式结构？Pre 过滤器和 Post 过滤器是怎么协作的？自定义过滤器时，为什么顺序错了可能导致 bug？

理解过滤器链的设计原理，不仅能帮你正确使用网关，更能让你在需要扩展时写出优雅的代码。

## 从装饰器模式说起

### 什么是装饰器模式

过滤器链的核心是**责任链模式**（Chain of Responsibility）与**装饰器模式**（Decorator）的结合。

先说装饰器模式。假设你有这样一个需求：给咖啡加糖、加奶、加冰——每加一种配料，价格增加一点。

不用装饰器模式，你可能需要写很多类：`Coffee`、`CoffeeWithSugar`、`CoffeeWithMilk`、`CoffeeWithSugarAndMilk`……类爆炸。

用装饰器模式，结构是这样的：

```
        ┌──────────────────────┐
        │     Component        │
        │  (抽象组件接口)        │
        └──────────────────────┘
                  △
                  │
    ┌─────────────┴─────────────┐
    │                           │
┌───┴─────┐              ┌──────┴──────┐
│ Concrete│              │  Decorator  │
│Component│              │ (装饰器抽象) │
│ (原味咖啡)│              └──────┬──────┘
└─────────┘                       │
                           ┌──────┴──────┐
                      ┌────┴────┐  ┌────┴────┐
                      │ Sugar  │  │  Milk   │
                      │Decorator│  │Decorator│
                      └────────┘  └────────┘
```

每个装饰器都持有被装饰对象的引用，可以**在调用前后添加额外行为**：

```java
// 装饰器基类
public abstract class CoffeeDecorator implements Coffee {
    protected Coffee coffee;  // 持有被装饰对象的引用

    public CoffeeDecorator(Coffee coffee) {
        this.coffee = coffee;
    }

    @Override
    public double getCost() {
        return coffee.getCost();  // 默认委托给被装饰对象
    }
}

// 糖装饰器
public class SugarDecorator extends CoffeeDecorator {
    public SugarDecorator(Coffee coffee) {
        super(coffee);
    }

    @Override
    public double getCost() {
        return super.getCost() + 1.0;  // 加糖加 1 块钱
    }
}
```

### 过滤器链如何应用装饰器模式

网关的过滤器链，本质上就是把每个过滤器看作一个装饰器：

```
请求 ─▶ Filter1 ─▶ Filter2 ─▶ Filter3 ─▶ 后端服务
             │        │        │
           前置处理  前置处理  前置处理
```

每个 Filter 都可以：
1. 在调用下一个 Filter **之前**做一些处理（Pre 逻辑）
2. 在调用下一个 Filter **之后**做一些处理（Post 逻辑）
3. 决定是否继续调用下一个 Filter（短路逻辑）

```java
public class FilterChain {
    private final List<Filter> filters;
    private int index = 0;

    public Mono<Void> filter(ServerWebExchange exchange) {
        // 所有过滤器都执行完了，调用后端
        if (index >= filters.size()) {
            return callBackend(exchange);
        }

        Filter filter = filters.get(index++);
        return filter.filter(exchange, this);  // 传入自身，形成链
    }
}

public interface Filter {
    Mono<Void> filter(ServerWebExchange exchange, FilterChain chain);
}
```

:::tip 装饰器 vs 责任链

装饰器模式强调**叠加效果**——每个装饰器都增强或修改被装饰对象的行为，最后的效果是所有装饰器的叠加。

责任链模式强调**选择处理**——请求沿着链传递，直到某个 handler 处理它为止。

过滤器链同时具备两者特点：**Pre 逻辑是装饰器的叠加效果，Post 逻辑是沿链回溯的处理**。
:::

## Pre 过滤器与 Post 过滤器

### 执行顺序

过滤器链的核心特点是**单向执行、双向处理**：

```
┌─────────────────────────────────────────────────────────────┐
│                     请求处理流程                              │
│                                                             │
│  请求 ─▶ Pre1 ─▶ Pre2 ─▶ Pre3 ─▶ 后端 ─▶ Post3 ─▶ Post2 ─▶ Post1 ─▶ 响应  │
│                                                             │
│   ▲                                                         │
│   │                                                         │
│   └───────────── 同一个 FilterChain ────────────────────────┘
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**Pre 过滤器**：按配置顺序依次执行，用于请求的前置处理
**Post 过滤器**：按配置顺序的**逆序**依次执行，用于响应的后置处理

为什么 Post 要逆序？因为 Post 逻辑需要先清理后释放（类似栈的 pop），确保内层资源在外层之前释放。

### 典型的 Pre 过滤器

| 过滤器 | 作用 | 位置 |
|---|---|---|
| **RequestLoggingFilter** | 记录请求开始时间、请求参数 | 链首 |
| **AuthenticationFilter** | JWT Token 验证 | 认证层 |
| **AuthorizationFilter** | 权限检查 | 鉴权层 |
| **RateLimitFilter** | 限流检查 | 流量控制层 |
| **RequestTransformer** | 请求转换（Header/Body） | 协议处理层 |

### 典型的 Post 过滤器

| 过滤器 | 作用 | 位置 |
|---|---|---|
| **ResponseLoggingFilter** | 记录响应状态、耗时 | 链尾 |
| **ResponseTransformer** | 响应格式转换 | 协议处理层 |
| **HeaderDeduplicationFilter** | 去除重复 Header | 安全层 |
| **MetricsFilter** | 采集 QPS/延迟指标 | 监控层 |
| **ErrorHandlerFilter** | 统一错误处理 | 异常处理层 |

### 短路与提前返回

过滤器可以决定是否继续执行后续过滤器：

```java
public class AuthenticationFilter implements GlobalFilter {
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, FilterChain chain) {
        String token = exchange.getRequest().getHeaders().getFirst("Authorization");

        if (token == null || !isValidToken(token)) {
            // 认证失败，直接返回，不继续调用后续过滤器
            exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
            return exchange.getResponse().setComplete();
        }

        // 认证成功，继续执行后续过滤器
        return chain.filter(exchange);
    }
}
```

:::warning 短路的艺术

短路虽然能提高效率，但过度使用会导致调试困难。如果每个过滤器都短路，请求可能在某个环节突然消失，却不知道是哪个过滤器拦截的。

建议：在开发环境开启详细日志，标记每个过滤器的执行情况。
:::

## 过滤器执行顺序

### Spring Cloud Gateway 中的顺序

Spring Cloud Gateway 使用 `int order` 控制执行顺序，**数字越小越先执行**：

```java
@Component
public class FirstFilter implements GlobalFilter {
    @Override
    public int getOrder() {
        return -100;  // 越小的数字越靠前
    }
}

@Component
public class SecondFilter implements GlobalFilter {
    @Override
    public int getOrder() {
        return 0;
    }
}
```

常见内置过滤器的默认顺序：

| 过滤器 | 默认 Order | 说明 |
|---|---|---|
| NettyWriteResponseFilter | -1 | 响应写入 |
| NettyRoutingFilter | -2 | 路由转发 |
| WebClientHttpRoutingFilter | -3 | WebClient 路由 |
| ForwardRoutingFilter | 0 | 本地转发 |
| LoadBalancerServiceFilter | 101 | 负载均衡 |

### Kong 中的顺序

Kong 的过滤器称为「插件」，执行顺序由插件的 `phase` 决定：

| Phase | 执行时机 | 常见插件 |
|---|---|---|
| `access` | 请求到达后、后端调用前 | 认证、限流、路由 |
| `header_filter` | 后端响应 Header 到达后 | Header 处理 |
| `body_filter` | 后端响应 Body 处理中 | 响应转换 |
| `log` | 请求完成后 | 日志记录 |

同一 phase 内的插件按 `priority` 排序，**数字越大越先执行**。

## 自定义过滤器开发

### Spring Cloud Gateway 自定义过滤器

**方式一：全局过滤器（GlobalFilter）**

全局过滤器对所有路由生效：

```java
@Component
public class CustomGlobalFilter implements GlobalFilter, Ordered {
    private static final Logger log = LoggerFactory.getLogger(CustomGlobalFilter.class);

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, FilterChain chain) {
        long startTime = System.currentTimeMillis();

        // Pre 逻辑
        log.info("请求路径: {}", exchange.getRequest().getPath());
        log.info("请求方法: {}", exchange.getRequest().getMethod());

        // 将开始时间存入上下文，供 Post 逻辑使用
        exchange.getAttributes().put("startTime", startTime);

        return chain.filter(exchange)
            .then(Mono.fromRunnable(() -> {
                // Post 逻辑
                Long start = exchange.getAttribute("startTime");
                long duration = System.currentTimeMillis() - (start != null ? start : 0);
                log.info("请求耗时: {}ms", duration);
                log.info("响应状态: {}", exchange.getResponse().getStatusCode());
            }));
    }

    @Override
    public int getOrder() {
        return -10;  // 较高优先级
    }
}
```

**方式二：路由过滤器（GatewayFilter）**

路由过滤器只对特定路由生效：

```java
@Component
public class CustomGatewayFilterFactory
        extends AbstractGatewayFilterFactory<CustomGatewayFilterFactory.Config> {

    public CustomGatewayFilterFactory() {
        super(Config.class);
    }

    @Override
    public GatewayFilter apply(Config config) {
        return (exchange, chain) -> {
            // 使用配置参数
            if (config.isEnabled()) {
                log.info("Custom filter enabled for route: {}",
                    exchange.getRequest().getPath());
            }
            return chain.filter(exchange);
        };
    }

    public static class Config {
        private boolean enabled = true;

        public boolean isEnabled() {
            return enabled;
        }

        public void setEnabled(boolean enabled) {
            this.enabled = enabled;
        }
    }
}
```

路由中引用：

```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: custom-route
          uri: http://backend
          filters:
            - name: Custom
              args:
                enabled: true
            - Custom=true  # 简写形式
```

### Kong 自定义插件

Kong 插件使用 Lua 开发：

```lua
-- kong/plugins/my-custom-plugin/handler.lua
local MyCustomHandler = {
    PRIORITY = 1000,  -- 优先级，数字越大越先执行
    VERSION = "1.0.0",
}

function MyCustomHandler:access(conf)
    -- 访问阶段逻辑
    kong.log("Custom plugin: processing request")

    -- 读取配置
    kong.log.debug("Config: ", kong.table.dump(conf))

    -- 添加响应头
    kong.response.set_header("X-Custom-Plugin", "processed")
end

function MyCustomHandler:header_filter(conf)
    -- Header 处理阶段
    kong.log("Custom plugin: header filter")
end

return MyCustomHandler
```

## 过滤器的数据共享

### 请求上下文（Exchange）

在 Spring Cloud Gateway 中，`ServerWebExchange` 是请求的上下文容器，可以在过滤器之间共享数据：

```java
// Filter1: 存储数据
exchange.getAttributes().put("userId", "12345");
exchange.getAttributes().put("permissions", List.of("read", "write"));

// Filter2: 读取数据
String userId = exchange.getAttribute("userId");
List<String> permissions = exchange.getAttribute("permissions");
```

### 请求/响应 Body 的修改

修改请求 Body 需要特殊处理，因为 Body 是流式的，只能读取一次：

```java
public class RequestBodyGatewayFilterFactory
        extends AbstractGatewayFilterFactory<RequestBodyGatewayFilterFactory.Config> {

    @Override
    public GatewayFilter apply(Config config) {
        return (exchange, chain) -> {
            if (config.isReadBody()) {
                return DataBufferUtils.join(exchange.getRequest().getBody())
                    .flatMap(body -> {
                        // 读取并处理 Body
                        String bodyStr = new String(body.toByteArray(),
                            StandardCharsets.UTF_8);
                        log.info("原始请求体: {}", bodyStr);

                        // 修改 Body（这里添加一个字段）
                        String modifiedBody = bodyStr.replace(
                            "\"timestamp\"",
                            "\"timestamp\"," + "\"processed_by\":\"gateway\""
                        );

                        // 重新写入 Body
                        DataBufferFactory dataBufferFactory =
                            exchange.getResponse().bufferFactory();
                        DataBuffer modifiedBuffer =
                            dataBufferFactory.wrap(modifiedBody.getBytes());

                        // 替换请求
                        ServerHttpRequest mutatedRequest =
                            new ServerHttpRequestDecorator(exchange.getRequest()) {
                                @Override
                                public Flux<DataBuffer> getBody() {
                                    return Flux.just(modifiedBuffer);
                                }
                            };

                        return chain.filter(
                            exchange.mutate().request(mutatedRequest).build()
                        );
                    });
            }
            return chain.filter(exchange);
        };
    }

    public static class Config {
        private boolean readBody = false;
        public boolean isReadBody() { return readBody; }
        public void setReadBody(boolean readBody) { this.readBody = readBody; }
    }
}
```

## 过滤器设计的最佳实践

| 原则 | 说明 |
|---|---|
| **单一职责** | 每个过滤器只做一件事，不要在过滤器中处理多个不相关的逻辑 |
| **快速失败** | 能提前判断的（如认证失败），尽早短路返回 |
| **异常处理** | 使用 `onErrorResume` 兜底，避免异常穿透 |
| **异步友好** | 使用 `Mono`/`Flux`，避免阻塞操作 |
| **配置外部化** | 敏感配置（限流阈值、超时时间）通过配置文件管理 |
| **日志适度** | 关键节点记录日志，避免日志过多影响性能 |

## 总结

过滤器链是网关扩展性的核心。理解其设计原理，才能：

- **正确配置**：知道过滤器执行顺序的含义，避免配置错误
- **问题排查**：定位请求在哪个过滤器被拦截，快速定位问题
- **优雅扩展**：按照设计模式编写自定义过滤器，保持代码风格一致

| 概念 | 说明 |
|---|---|
| 装饰器模式 | 每个 Filter 装饰被装饰对象，在调用前后添加行为 |
| Pre 过滤器 | 按配置顺序执行，用于请求预处理 |
| Post 过滤器 | 按逆序执行，用于响应后处理 |
| 短路逻辑 | 过滤器可决定是否继续执行后续过滤器 |
| 数据共享 | 通过 Exchange Attributes 在过滤器间共享数据 |

过滤器链的设计看似简单，背后是经典设计模式的灵活运用。掌握这个思想，你不仅能写好网关过滤器，也能将这个模式应用到其他需要链式处理的场景中。
