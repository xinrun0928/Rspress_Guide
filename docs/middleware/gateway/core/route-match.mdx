# 网关路由匹配规则

你配置了一条路由规则：`/api/users/**` 应该路由到 `user-service`。但用户访问 `/api/users` 时却 404 了。

这不是 bug，这是你对路由匹配规则理解不准确。

路由匹配是网关最基础也最容易出问题的功能。理解各种匹配方式的特点和优先级，是正确配置路由的前提。

## Path 路由匹配

Path 路由是最常见的匹配方式，根据 URL 路径将请求路由到对应的后端服务。

### 三种匹配模式

| 模式 | 语法 | 匹配示例 | 不匹配示例 |
|---|---|---|---|
| **精确匹配** | `/api/users` | `/api/users` | `/api/users/123`、`/api/users2` |
| **前缀匹配** | `/api/users/**` | `/api/users`、`/api/users/123`、`/api/users/456/profile` | `/api/user`、`/api2/users` |
| **正则匹配** | `/api/v\d+/items` | `/api/v1/items`、`/api/v12/items` | `/api/v/items`、`/api/v1item` |

### 前缀匹配的细节

前缀匹配使用 `**` 通配符，匹配该路径下所有层级的内容：

```
/api/users/** 匹配：
  /api/users
  /api/users/
  /api/users/123
  /api/users/123/profile
  /api/users/123/profile/avatar
```

:::warning 精确匹配的陷阱

前面提到的问题：`/api/users/**` 和 `/api/users` 在很多网关中的行为是不同的。

`/api/users/**`（前缀匹配）会匹配 `/api/users`、`/api/users/`、`/api/users/123` 等所有以 `/api/users/` 开头的路径。

而 `/api/users`（精确匹配）只匹配 `/api/users` 这一个路径，不会匹配 `/api/users/` 或 `/api/users/123`。

如果你同时配置了精确匹配和前缀匹配，网关会按照优先级选择其中一个。
:::

### 正则匹配

正则匹配提供了最大的灵活性，但同时也最容易出错：

```yaml
# 匹配版本号路径，如 /api/v1/users、/api/v2/users
- id: versioned-api
  uri: http://backend-service
  predicates:
    - Path=/api/v{v:[0-9]+}/users
  filters:
    - StripPrefix=1
```

```java
// 匹配日期路径，如 /api/2024/03/15/users
- id: dated-api
  uri: http://backend-service
  predicates:
    - Path=/api/{year:[0-9]{4}}/{month:[0-9]{2}}/{day:[0-9]{2}}/users
```

正则匹配时可以使用**捕获组**（用 `{}` 包裹的命名变量），捕获的值可以在过滤器中引用：

```java
// 使用 PathVariable 引用捕获的值
ServerWebExchange exchange = ...
String year = exchange.getAttribute("year");
String month = exchange.getAttribute("month");
```

## Header 路由

除了路径，还可以通过请求头进行路由。这在灰度发布、金丝雀发布、多租户等场景下非常有用。

### 基本用法

```yaml
spring:
  cloud:
    gateway:
      routes:
        # 根据 X-Client-Type 头路由
        - id: mobile-route
          uri: http://mobile-backend
          predicates:
            - Header=X-Client-Type, mobile

        # 根据 X-Version 头路由（支持正则）
        - id: v2-route
          uri: http://v2-backend
          predicates:
            - Header=X-API-Version, v2\..*
```

### 常见应用场景

**灰度发布**

```yaml
# 10% 的流量路由到新版本
- id: canary-route
  uri: http://canary-service
  predicates:
    - Header=X-Canary, enabled
  # 需要配合服务网格或网关插件实现百分比分流
```

**多租户路由**

```yaml
- id: tenant-a-route
  uri: http://tenant-a-backend
  predicates:
    - Header=X-Tenant-ID, tenant-a
```

## Query 参数路由

根据 URL 查询参数进行路由，适合版本兼容、A/B 测试等场景。

### 基本用法

```yaml
spring:
  cloud:
    gateway:
      routes:
        # 根据 version 参数路由
        - id: v1-route
          uri: http://v1-backend
          predicates:
            - Query=version, v1

        # 根据多个参数路由
        - id: special-route
          uri: http://special-backend
          predicates:
            - Query=mode, special
            - Query=env, prod
```

### 正则匹配

```yaml
# version 参数以 v 开头，后面是数字
- id: versioned-route
  uri: http://versioned-backend
  predicates:
    - Query=version, v[0-9]+
```

## 组合条件

多个匹配条件可以组合使用，**所有条件都满足时才会匹配**。

### AND 逻辑

```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: combined-route
          uri: http://combined-backend
          predicates:
            # 必须同时满足：路径是 /api/users，Header 有 X-Client-Type，Query 有 version
            - Path=/api/users/**
            - Header=X-Client-Type, mobile
            - Query=version, v2
```

### HTTP Method 条件

```yaml
spring:
  cloud:
    gateway:
      routes:
        # POST 请求路由到 write 服务
        - id: write-route
          uri: http://write-backend
          predicates:
            - Path=/api/data/**
            - Method=POST

        # GET 请求路由到 read 服务
        - id: read-route
          uri: http://read-backend
          predicates:
            - Path=/api/data/**
            - Method=GET
```

### Cookie 条件

```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: premium-route
          uri: http://premium-backend
          predicates:
            - Cookie=member_type, premium
```

### 远程地址条件

```yaml
spring:
  cloud:
    gateway:
      routes:
        # 来自特定 IP 段的请求路由
        - id: internal-route
          uri: http://internal-backend
          predicates:
            - RemoteAddr=192.168.1.0/24
```

## 路由优先级

当多个路由规则都匹配同一个请求时，**最具体的规则优先**。

### 优先级规则

| 规则 | 优先级 |
|---|---|
| 精确匹配 | 最高 |
| 前缀匹配（路径越短优先级越高） | 中 |
| 正则匹配 | 低 |
| 配置顺序（部分网关） | 可配置 |

```
请求路径：/api/users/123/profile

匹配结果：
1. /api/users/{id}/profile （精确匹配用户 ID + profile）
2. /api/users/{id}/** （前缀匹配用户 ID 下所有路径）
3. /api/users/** （前缀匹配）
4. /api/** （前缀匹配，但路径更长，优先级较低）
```

### 显式指定优先级

部分网关支持显式指定路由优先级：

```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: high-priority
          uri: http://high-priority-backend
          predicates:
            - Path=/api/users/admin/**
          order: -1  # 数值越小，优先级越高

        - id: normal-priority
          uri: http://normal-backend
          predicates:
            - Path=/api/users/**
          order: 0
```

## 路由匹配的特殊情况

### 尾部斜杠

`/api/users` 和 `/api/users/` 在大多数网关中被视为不同的路径：

| 配置 | 匹配 `/api/users` | 匹配 `/api/users/` |
|---|---|---|
| `/api/users` | ✅ | ❌ |
| `/api/users/` | ❌ | ✅ |
| `/api/users/**` | ✅ | ✅ |

:::tip 建议

如果你的业务允许，建议配置时同时考虑带斜杠和不带斜杠的情况，或者在网关层面做 301 重定向统一格式：

```yaml
filters:
  - RedirectTo=301, /api/users/
```

这样无论用户输入哪种格式，最终都会统一处理。
:::

### URL 编码

URL 中的特殊字符会被编码，如空格编码为 `%20`。路由匹配时，网关通常会自动解码后再匹配：

```
请求：/api/users/John%20Doe
配置：/api/users/John Doe

匹配结果：✅ 成功匹配
```

### 路径大小写

默认情况下，路径匹配是**区分大小写**的：

```
请求：/Api/Users （注意大写 A、U）
配置：/api/users

匹配结果：❌ 不匹配
```

如果需要忽略大小写，可以在部分网关中配置：

```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: case-insensitive-route
          uri: http://backend
          predicates:
            - Path=/api/users/**
          filters:
            - RewritePath=/api/users/(?<segment>.*), /api/users/$\{segment}
          # 或使用 lowercasePath
          # - LowercaseRequestPath=true
```

## 常见错误与排查

### 路由不生效的常见原因

| 问题 | 排查方向 |
|---|---|
| 配置了但请求 404 | 检查路由 ID 是否唯一、Predicate 是否正确 |
| 同时满足两个路由 | 检查优先级配置 |
| 正则匹配失败 | 检查正则语法、使用在线正则工具测试 |
| Header/Query 条件不满足 | 使用 curl/wget 确认请求中包含对应的 Header 或 Query |

### 调试技巧

```bash
# 打印路由匹配日志
logging:
  level:
    org.springframework.cloud.gateway: DEBUG

# 使用 curl 模拟请求
curl -v -H "X-Client-Type: mobile" \
     -H "X-API-Version: v2" \
     "http://gateway/api/users?version=v2"
```

## 总结

| 匹配类型 | 适用场景 | 语法示例 |
|---|---|---|
| 精确匹配 | 单一路径 | `/api/users` |
| 前缀匹配 | 路径下所有子路径 | `/api/users/**` |
| 正则匹配 | 复杂路径规则 | `/api/v{v:[0-9]+}/users` |
| Header 匹配 | 灰度、多租户 | `Header=X-Client-Type, mobile` |
| Query 匹配 | 版本兼容、A/B 测试 | `Query=version, v2` |
| 组合条件 | 复合规则 | `Path=... + Header=... + Query=...` |

理解路由匹配的规则和优先级，才能正确配置网关，避免请求「迷路」到错误的后端服务。
