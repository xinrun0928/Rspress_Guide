# API 网关核心功能

一个 API 网关要真正发挥作用，需要承担六大核心职责。这些职责不是简单叠加，而是相互协作，共同保障系统的安全性、稳定性和可观测性。

## 一、路由转发

路由是网关最基础的职责——根据请求的特征，将请求精准地分发给对应的后端服务。

### 路由匹配方式

| 匹配方式 | 说明 | 示例 |
|---|---|---|
| **Path 精确匹配** | URL 路径完全相等 | `/api/users` 匹配 `/api/users` |
| **前缀匹配** | URL 路径以指定前缀开头 | `/api/users/**` 匹配 `/api/users/123` |
| **正则匹配** | 使用正则表达式匹配路径 | `/api/v\d+/items` 匹配 `/api/v1/items` |
| **Header 匹配** | 根据请求头路由 | `Header=X-Client-Type: mobile` |
| **Query 参数匹配** | 根据 URL 参数路由 | `Query=version=v2` |

### 路由匹配优先级

当多个路由规则都匹配同一个请求时，网关需要按照优先级决定使用哪个路由。常见的优先级规则：

1. **精确匹配优先于前缀匹配**
2. **多条件组合优先级高于单条件**
3. **配置顺序（部分网关支持显式指定优先级）**

:::tip 路由匹配的坑

很多人以为 `/api/users` 和 `/api/users/*` 是一样的。但实际上，前者是精确匹配，只会匹配 `/api/users` 这一个路径；后者是前缀匹配，会匹配 `/api/users/123`、`/api/users/456/profile` 等所有以 `/api/users/` 开头的路径。

在配置路由规则时，一定要注意这种区别，否则可能导致请求「迷路」。
:::

## 二、负载均衡

当某个服务有多个实例时，网关需要决定将请求发送到哪个实例。这就是负载均衡。

### 常见负载均衡策略

| 策略 | 原理 | 适用场景 |
|---|---|---|
| **轮询（Round Robin）** | 依次分发到每个实例 | 场景简单，实例性能一致 |
| **加权轮询（Weighted Round Robin）** | 按权重比例分发 | 实例性能不一致，需要按比例分配 |
| **最小连接（Least Connections）** | 分发给当前连接数最少的实例 | 长连接场景 |
| **IP Hash** | 根据客户端 IP 哈希值选择实例 | 需要会话保持 |
| **随机（Random）** | 随机选择实例 | 简单场景，作为基准 |

### 健康检查

负载均衡必须配合健康检查才能正常工作。健康检查分为两种：

- **主动健康检查**：网关主动探测后端实例状态，常见方式有 HTTP GET、TCP 握手
- **被动健康检查**：根据后端实例的响应结果判断是否健康（如连续 N 次超时则标记为不健康）

:::tip 健康检查的代价

健康检查虽然能提高系统的可用性，但也会带来额外开销。每秒 1000 次的健康检查，对后端服务来说也是不小的负担。

建议合理设置健康检查的间隔（如 10~30 秒）和超时时间，避免过度检查。
:::

## 三、认证授权

网关是所有请求的统一入口，在这里做认证鉴权再合适不过——不需要在每个微服务中都重复实现。

### 常见认证方式

**JWT Token 认证**

用户登录后获取 JWT Token，后续请求携带 Token，网关验证 Token 的合法性（签名、过期时间等）。

```
请求流程：
1. POST /auth/login → 返回 JWT Token
2. GET /api/users → Header: Authorization: Bearer <token>
3. 网关验证 Token → 有效则放行，无效则返回 401
```

**API Key 认证**

为每个客户端分配一个唯一的 API Key，通过 Header 或 Query 参数传递。

```
Header: X-API-Key: your-api-key
Query: ?api_key=your-api-key
```

**OAuth2 授权**

适合需要第三方授权的场景，如「使用 Google 账号登录」。网关可以验证 Access Token 的合法性。

### 认证 vs 授权

很多人容易混淆这两个概念：

- **认证（Authentication）**：验证「你是谁」，回答「你是一个合法用户吗」
- **授权（Authorization）**：验证「你能做什么」，回答「你有权限访问这个资源吗」

网关通常负责认证，而细粒度的授权在业务服务中处理。

## 四、限流熔断

### 限流（Rate Limiting）

限流的目的是保护系统不被突发流量打垮。常见的限流算法：

| 算法 | 原理 | 优点 | 缺点 |
|---|---|---|---|
| **计数器** | 固定时间窗口内统计请求数 | 实现简单 | 临界突变问题 |
| **滑动窗口** | 随时间滑动的窗口，统计更精确 | 计数精确 | 实现复杂 |
| **令牌桶** | 按固定速率放入令牌，取令牌放行 | 支持突发流量 | 需要存储令牌数 |
| **漏桶** | 请求以任意速率进入，以固定速率流出 | 平滑流量 | 不支持突发 |

:::tip 令牌桶 vs 漏桶

令牌桶允许一定程度的突发流量——因为桶里可能积累了大量令牌。但漏桶严格控制输出速率，不允许任何突发。

对于 API 限流，令牌桶是更常见的选择。它允许短时间的流量高峰，只要平均速率不超过阈值即可。
:::

### 熔断（Circuit Breaker）

熔断的目的是防止故障蔓延。当某个后端服务持续失败时，继续向它发请求只会浪费资源并可能导致用户请求超时。熔断器会自动「跳闸」，快速返回降级响应。

熔断器有三种状态：

| 状态 | 行为 | 触发条件 |
|---|---|---|
| **Closed** | 正常放行，统计失败率 | 初始状态 |
| **Open** | 直接返回降级响应，不调用后端 | 失败率/失败数超过阈值 |
| **HalfOpen** | 放行部分请求试探，成功后恢复 Closed | Open 状态持续一段时间后 |

```
Closed ──失败率超限──▶ Open
  ▲                        │
  │                        │ (超时后)
  └──成功恢复──────── HalfOpen──▶ Open
```

## 五、日志监控

### 请求日志

网关应该记录每个请求的关键信息：

- 请求时间、请求方法、请求路径
- 客户端 IP、User-Agent
- 响应状态码、响应时间
- 路由到的后端服务

这些日志是排查问题的依据，也是业务分析的数据来源。

### 指标采集

| 指标类型 | 说明 | 示例 |
|---|---|---|
| **QPS** | 每秒请求数 | 系统当前负载 |
| **延迟** | P50/P90/P99 响应时间 | 性能瓶颈定位 |
| **错误率** | 4xx/5xx 请求占比 | 服务健康度 |
| **限流触发率** | 被限流的请求占比 | 限流配置合理性 |

### 分布式追踪

在微服务架构中，一个用户请求可能涉及十几个服务的调用。分布式追踪（如 Jaeger、Zipkin）可以为每个请求生成唯一 Trace ID，串联起整个调用链，帮助定位性能瓶颈。

```
Trace ID: abc123
  └─ Gateway: 10ms
      └─ UserService: 5ms
      └─ OrderService: 20ms
          └─ InventoryService: 15ms
```

## 六、协议转换

网关不仅是 HTTP 请求的中转站，还承担着协议转换的职责。

### HTTP ↔ gRPC

gRPC 使用 HTTP/2 传输 Protocol Buffers 序列化的二进制数据。网关可以将 HTTP/JSON 请求转换为 gRPC 调用：

```
客户端(HTTP/JSON) ──▶ 网关(gRPC) ──▶ 后端服务(gRPC)
```

Spring Cloud Gateway 和 Kong 都支持 gRPC 代理。

### HTTP ↔ WebSocket

对于需要长连接的场景（如实时聊天、协作编辑），网关需要支持 WebSocket 升级：

```http
GET /ws/chat HTTP/1.1
Upgrade: websocket
Connection: Upgrade
```

### 协议转换的代价

协议转换不是免费的。JSON ↔ Protobuf 的序列化/反序列化、HTTP/1.1 ↔ HTTP/2 的协议转换，都会带来性能开销。

如果你的系统对延迟极度敏感，可能需要让客户端直接使用后端服务支持的协议，网关只做必要的转发。

## 总结

六大核心职责相互配合：

- **路由 + 负载均衡**：确保请求送到正确的服务、正确的实例
- **认证 + 授权**：确保请求是合法的、请求者有权限
- **限流 + 熔断**：保护系统不被过载和故障击垮
- **日志 + 监控 + 追踪**：提供可观测性，帮助排查问题和优化性能
- **协议转换**：打破协议壁垒，让不同技术栈的服务可以协作
