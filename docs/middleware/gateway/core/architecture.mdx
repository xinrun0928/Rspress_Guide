# API 网关架构设计要点

你可能见过这样的场景：网关刚上线时运行平稳，随着业务增长，流量翻了一倍，网关就开始出现性能问题——延迟飙升、CPU 打满、OOM 频发。

这不是网关本身的问题，而是架构设计时没有充分考虑扩展性。

API 网关的架构设计，需要在性能、可用性、可扩展性之间找到平衡。

## 集中式 vs 嵌入式网关

选择网关架构的第一步，是决定采用集中式还是嵌入式的部署模式。

### 集中式网关

```
┌─────────┐     ┌─────────────┐     ┌──────────────────┐
│  客户端  │────▶│  API Gateway │────▶│  微服务集群       │
└─────────┘     └─────────────┘     └──────────────────┘
                      │
                      ▼
                ┌───────────┐
                │  配置中心  │
                │ (Nacos/   │
                │  etcd)    │
                └───────────┘
```

**优点**：
- 统一的入口，所有流量一目了然
- 安全策略、限流规则集中管理
- 后端服务可以是无状态的应用

**缺点**：
- 网关成为单点，需要保证高可用
- 额外的网络跳转（客户端 → 网关 → 服务），增加延迟
- 所有流量都要经过网关，带宽和计算资源都需要预留

### 嵌入式网关（Sidecar）

每个微服务实例旁边运行一个轻量级网关代理：

```
┌──────────────────────────────────────────────────────┐
│                      Pod/Container                    │
│  ┌─────────────┐              ┌─────────────────┐   │
│  │   业务服务   │              │   网关代理       │   │
│  │             │◀─────────────▶│  (Local Proxy)  │   │
│  └─────────────┘   localhost   └─────────────────┘   │
└──────────────────────────────────────────────────────┘
```

**优点**：
- 无需额外的网络跳转，服务内部直接通信
- 故障隔离，一个服务的网关出问题不影响其他服务
- 适合 Mesh 架构

**缺点**：
- 每个服务实例都要运行网关代理，资源开销成倍增加
- 配置分散，策略一致性难以保证
- 调试和排查问题更复杂

:::tip 架构选择建议

对于大多数团队，**集中式网关**是更务实的选择。架构简单、策略统一、运维方便。

只有在超大规模、需要强故障隔离、或已经采用 Service Mesh 的场景下，才考虑嵌入式网关。
:::

## 请求处理流程

理解网关的请求处理流程，是优化性能和问题排查的基础。

### 完整流程

```
客户端请求
    │
    ▼
┌───────────────┐
│  流量接入层     │ ← TLS 终止、连接管理
└───────────────┘
    │
    ▼
┌───────────────┐
│  路由匹配      │ ← 根据 Path/Header/Query 选择目标路由
└───────────────┘
    │
    ▼
┌───────────────┐
│  Pre 过滤器链  │ ← 认证、鉴权、限流、日志
└───────────────┘
    │
    ▼
┌───────────────┐
│  负载均衡      │ ← 选择目标实例
└───────────────┘
    │
    ▼
┌───────────────┐
│  后端调用      │ ← HTTP/gRPC 调用
└───────────────┘
    │
    ▼
┌───────────────┐
│  Post 过滤器链 │ ← 响应转换、Header 处理、日志
└───────────────┘
    │
    ▼
┌───────────────┐
│  响应返回      │
└───────────────┘
    │
    ▼
客户端响应
```

### 延迟来源分析

每个环节都会带来延迟：

| 环节 | 典型延迟 | 优化方向 |
|---|---|---|
| 流量接入 | 1~5ms | 优化 TLS 硬件、使用 TLS 1.3 |
| 路由匹配 | 0.1~1ms | 减少路由数量、使用高效匹配算法 |
| Pre 过滤器 | 1~10ms | 减少过滤器数量、异步处理 |
| 负载均衡 | 0.1~0.5ms | 本地缓存实例列表、减少服务发现频率 |
| 后端调用 | 10~100ms | 优化后端服务性能 |
| Post 过滤器 | 1~5ms | 减少响应转换 |

:::warning 延迟是叠加的

网关的端到端延迟 = 各环节延迟之和。如果每个环节 5ms，10 个环节就是 50ms。

对于延迟敏感的场景（如支付链路），需要对每个环节进行优化，而不是寄希望于某个环节的突破性改进。
:::

## 高可用设计

### 网关单点问题

集中式网关最大的风险是单点故障。如果网关挂了，所有外部请求都无法到达后端服务。

解决方案：**多网关实例 + 负载均衡**

```
┌─────────────────────────────────────────┐
│              负载均衡器                   │
│           (SLB / Nginx)                  │
└────────┬────────┬────────┬────────────┘
         │        │        │
    ┌────▼──┐ ┌───▼──┐ ┌───▼──┐
    │ SCG 1 │ │ SCG 2 │ │ SCG 3 │
    └────┬──┘ └───┬──┘ └───┬──┘
         │        │        │
    所有实例注册到同一服务发现中心
    配置变更通过配置中心同步
```

### 配置一致性

当有多个网关实例时，如何保证配置的一致性？

| 方案 | 原理 | 优点 | 缺点 |
|---|---|---|---|
| **配置中心推送** | Nacos/Apollo 配置变更时主动推送 | 实时性好 | 实现复杂 |
| **配置中心拉取** | 网关定期从配置中心拉取最新配置 | 实现简单 | 有延迟 |
| **数据库存储** | 网关配置存储在 MySQL，共享读写 | 数据可靠 | 单点数据库 |
| **文件系统同步** | 配置写入共享存储（如 NFS），各实例监听变更 | 适合 K8s 环境 | 依赖外部存储 |

### 优雅关闭

网关在重启或发布时，需要处理正在进行的请求，避免请求丢失：

1. 停止接收新请求（从 SLB/注册中心下线）
2. 等待正在处理的请求完成（设置超时，如 30 秒）
3. 强制关闭超时未完成的连接
4. 执行清理工作（关闭数据库连接、释放资源）

## 性能优化策略

### 连接池复用

网关需要与大量后端服务通信。如果每次请求都建立新连接，TCP 握手的三次延迟会严重影响性能。

解决方案：**HTTP/2 连接复用 + 连接池**

```
网关 ──▶ 保持连接 ──▶ 后端服务 A
网关 ──▶ 保持连接 ──▶ 后端服务 B
...
```

### 本地缓存

网关可以缓存以下内容，减少对后端服务的请求：

- **路由配置缓存**：减少配置中心的访问频率
- **服务实例列表缓存**：减少服务发现的频率（TTL 通常 30 秒）
- **认证结果缓存**：JWT 验证结果可以缓存一段时间
- **响应数据缓存**：对于读多写少的场景，可以缓存后端响应

:::tip 缓存的一致性问题

缓存带来性能提升的同时，也引入了数据一致性风险。如果后端数据已经更新，但网关还在使用旧的缓存数据，用户就会看到过期信息。

对于敏感数据（如余额、库存），建议设置较短的缓存 TTL，或者在数据变更时主动失效缓存。
:::

### 异步处理

在 Pre 过滤器中进行耗时操作（如调用外部认证服务查询权限），会阻塞整个请求处理流程。

解决方案：**异步过滤器 + 响应式编程**

```java
// Spring Cloud Gateway 示例：异步过滤器
@Component
public class AsyncAuthFilter implements GlobalFilter {
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        return getAuthFromExternalService(exchange)
            .flatMap(auth -> {
                // 将认证结果放入上下文，继续后续处理
                exchange.getAttributes().put("auth", auth);
                return chain.filter(exchange);
            })
            .onErrorResume(e -> {
                // 认证失败，直接返回
                exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
                return exchange.getResponse().setComplete();
            });
    }

    private Mono<AuthResult> getAuthFromExternalService(ServerWebExchange exchange) {
        // 模拟异步调用
        return webClient.get()
            .uri("http://auth-service/verify")
            .retrieve()
            .bodyToMono(AuthResult.class)
            .timeout(Duration.ofMillis(100)); // 设置超时，避免阻塞
    }
}
```

## 多网关集群部署

### 有状态 vs 无状态

理想情况下，网关应该是**无状态**的——每个实例处理请求时不需要依赖特定实例的状态。

但有些场景天然是有状态的：

- **会话保持**：需要将同一用户的请求路由到同一实例
- **WebSocket 长连接**：连接建立后与特定实例绑定
- **本地缓存**：如果使用了实例本地缓存，数据在不同实例间不同步

:::tip 设计建议

在架构设计阶段，就尽量避免网关的有状态需求。如果必须要有状态，可以通过以下方式缓解：

- 会话保持：使用 IP Hash 或 Cookie 路由到同一实例，但后端服务仍需是无状态的
- WebSocket：使用 sticky session，但做好断线重连的兜底
- 本地缓存：明确缓存的数据范围，设置较短的过期时间
:::

### 跨地域部署

对于需要高可用的系统，可能需要在多个地域部署网关：

```
地域 A（主）          地域 B（备）
┌──────────────┐      ┌──────────────┐
│  Gateway A1  │      │  Gateway B1  │
│  Gateway A2  │      │  Gateway B2  │
└──────┬───────┘      └──────┬───────┘
       │                     │
       └──────────┬──────────┘
                  │
           ┌──────▼──────┐
           │   全局负载均衡  │
           │  (DNS/GSLB)  │
           └─────────────┘
```

用户请求会根据地域就近接入最近的网关集群，实现低延迟和高可用的双重目标。

## 总结

API 网关架构设计的核心要点：

| 设计维度 | 关键决策 |
|---|---|
| 部署模式 | 集中式（推荐） vs 嵌入式 |
| 可用性 | 多实例 + 负载均衡 + 优雅关闭 |
| 配置一致性 | 配置中心推送/拉取 + 变更监听 |
| 性能 | 连接复用 + 本地缓存 + 异步处理 |
| 扩展性 | 无状态设计 + 水平扩展 |

架构没有最优解，只有最适合当前业务规模和团队能力的方案。记住：过度设计的代价往往比设计不足更高。
