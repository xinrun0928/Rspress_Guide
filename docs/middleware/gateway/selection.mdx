# API 网关选型：Spring Cloud Gateway vs Kong vs APISIX vs Traefik

凌晨 3 点，你被报警电话吵醒——生产环境的 API 网关 CPU 打满，所有接口都在超时。你打开监控，发现问题的根源是：选择的网关方案根本撑不住当前的 QPS。

这不是技术问题，这是选型问题。

不同的 API 网关方案，在性能、功能、易用性、生态等维度上有显著差异。选对了，团队开发效率倍增；选错了，整个团队可能要为它买单半年以上。

## 四大主流方案对比

### Spring Cloud Gateway

**定位**：Spring Cloud 生态的官方网关解决方案，基于 Spring 5 WebFlux + Project Reactor 构建。

| 维度 | 表现 |
|---|---|
| 语言/框架 | Java / Spring 5 WebFlux |
| 性能 | 中等，单实例 2~5 万 QPS |
| 功能 | 路由、Predicate、Filter、内置限流/熔断 |
| 生态 | 与 Spring Cloud 无缝集成，适合 Spring 项目 |
| 运维 | Java 运维经验即可，配置通过 YAML 管理 |
| 扩展 | 支持自定义 Filter，Java 生态丰富 |

**适用场景**：已有的 Spring Cloud 项目，团队以 Java 为主，追求快速集成。

### Kong

**定位**：Nginx + OpenResty（Lua）构建的高性能网关，以插件化为核心设计理念。

| 维度 | 表现 |
|---|---|
| 语言/框架 | OpenResty (Nginx + Lua) / Go 管理平面 |
| 性能 | 高，单实例可达 10~30 万 QPS |
| 功能 | 插件体系极其丰富，社区插件 100+，支持 Wasm |
| 生态 | 成熟的周边工具（Konga UI、Kong Ingress Controller） |
| 运维 | 需要了解 Nginx/Lua，上手曲线略陡 |
| 扩展 | 插件开发简单，支持热加载，无需重启 |

**适用场景**：追求极致性能，需要丰富插件能力，K8s 环境下的 Ingress Controller。

### Apache APISIX

**定位**：基于 Nginx + etcd 构建的云原生 API 网关，以动态配置和可扩展性为核心优势。

| 维度 | 表现 |
|---|---|
| 语言/框架 | Go / OpenResty / etcd |
| 性能 | 高，单实例可达 10~50 万 QPS |
| 功能 | 熔断、限流、认证、可观测性插件丰富 |
| 生态 | 云原生友好，支持 K8s Ingress、Service Mesh |
| 运维 | 配置热更新，无需 reload，支持多数据中心 |
| 扩展 | 支持 Wasm、Java/Go/Python 多语言插件 |

**适用场景**：K8s 环境，多团队协作，需要频繁变更路由规则，对动态配置有强需求。

### Traefik

**定位**：为容器时代设计的现代反向代理和负载均衡器，强调自动化和云原生集成。

| 维度 | 表现 |
|---|---|
| 语言/框架 | Go |
| 性能 | 中等，单实例 1~3 万 QPS |
| 功能 | 自动服务发现、Let's Encrypt、熔断 |
| 生态 | 与 Docker、K8s、Consul 等深度集成 |
| 运维 | 配置简单，自动化程度高 |
| 扩展 | 支持中间件，插件体系相对简单 |

**适用场景**：K8s Ingress Controller 场景，以 Docker 容器部署为主的小型团队。

## 核心维度对比

### 性能基准

| 网关 | QPS（单实例）| 延迟 P99 | 备注 |
|---|---|---|---|
| Spring Cloud Gateway | 2~5 万 | 10~30ms | 异步非阻塞，性能稳定 |
| Kong | 10~30 万 | 2~5ms | Lua JIT 加速，极致性能 |
| Apache APISIX | 10~50 万 | 1~3ms | etcd 动态配置，零 reload |
| Traefik | 1~3 万 | 5~15ms | 适合中小规模场景 |

:::tip 性能不是唯一标准

高性能意味着更高的硬件成本效率，但在业务规模较小时，差异可能并不明显。

一个每秒只有几百 QPS 的系统，选择 Kong 或 APISIX 的意义不大，反而增加了运维复杂度。选择与团队能力匹配的方案，比追求极致性能更重要。
:::

### 功能完整性

| 功能 | SCG | Kong | APISIX | Traefik |
|---|---|---|---|---|
| 路由转发 | ✅ | ✅ | ✅ | ✅ |
| 负载均衡 | ✅ | ✅ | ✅ | ✅ |
| JWT 认证 | ✅ | ✅ | ✅ | ⚠️ 需插件 |
| OAuth2 | ⚠️ 需扩展 | ✅ 插件 | ✅ 插件 | ❌ |
| 限流 | ✅ 内置 | ✅ 插件 | ✅ 插件 | ⚠️ 基础 |
| 熔断 | ✅ Resilience4j | ✅ 插件 | ✅ 插件 | ✅ |
| gRPC 代理 | ✅ | ✅ | ✅ | ⚠️ |
| WebSocket | ✅ | ✅ | ✅ | ✅ |
| 热更新 | ✅ | ✅ | ✅ | ✅ |

### 运维复杂度

| 维度 | SCG | Kong | APISIX | Traefik |
|---|---|---|---|---|
| 学习曲线 | 低（Java 团队） | 中（需了解 Lua） | 中（需了解 etcd） | 低 |
| 配置方式 | YAML | Declarative (Admin API) | Declarative (Admin API) | YAML/Annotations |
| 数据存储 | 内存/Redis | Cassandra/PostgreSQL | etcd | 只读 K8s API |
| 集群部署 | 需要额外组件 | 原生支持 | 原生支持 | 原生支持 |
| 可观测性 | Micrometer | Datadog/Prometheus | Prometheus | Prometheus |

## 选型决策树

```
你的技术栈是什么？
│
├─ Spring Cloud 生态
│   └─ 团队 Java 为主 → Spring Cloud Gateway
│
├─ Kubernetes 环境
│   ├─ 需要高频路由变更 → Apache APISIX
│   └─ 以 Ingress 为主 → Traefik
│
└─ 追求极致性能
    └─ 愿意投入学习成本 → Kong / Apache APISIX
```

## 场景化选型建议

### 场景一：Spring Cloud 微服务项目

**推荐：Spring Cloud Gateway**

理由：与 Spring Cloud 全家桶无缝集成，Feign、LoadBalancer、Nacos/Consul 服务发现等开箱即用。Java 团队可以完全用 Java 解决问题，代码可控、可调试。

```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: user-service
          uri: lb://user-service
          predicates:
            - Path=/api/users/**
          filters:
            - StripPrefix=1
```

### 场景二：K8s 环境的多团队协作平台

**推荐：Apache APISIX**

理由：基于 etcd 的动态配置，路由变更秒级生效，无需重启。插件体系完善，支持多语言扩展。Dashboard 可视化配置，降低协作门槛。

:::tip APISIX vs Kong in K8s

在 K8s 环境下，Apache APISIX 的 Ingress Controller 比 Kong 的 Ingress Controller 更轻量、配置变更更快。但 Kong 在企业级功能（如 RBAC、审计日志）方面更成熟。

如果团队规模大、需要严格的权限管控，选 Kong；如果追求性能和迭代速度，选 APISIX。
:::

### 场景三：高并发交易系统

**推荐：Kong 或 Apache APISIX**

理由：这类场景对性能极为敏感，每降低 1ms 延迟都意味着真金白银。Nginx 底层的网关方案在延迟和吞吐量上有明显优势。

### 场景四：初创小团队，快速验证

**推荐：Spring Cloud Gateway 或 Traefik**

理由：团队人手有限，没有精力深入学习 Lua 或 etcd。选择一个与团队现有技能匹配的方案，把时间花在业务上。

## 总结

| 方案 | 核心优势 | 最大短板 |
|---|---|---|
| Spring Cloud Gateway | Spring 生态集成，Java 开发体验 | 性能一般 |
| Kong | 插件丰富，性能高，社区成熟 | Lua 学习成本 |
| Apache APISIX | 云原生友好，性能极致，动态配置 | 生态相对年轻 |
| Traefik | 自动化程度高，K8s 原生 | 功能相对简单 |

没有完美的方案，只有最适合团队和业务场景的方案。花时间做选型分析，比后期迁移付出的代价要小得多。
