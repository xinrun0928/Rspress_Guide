# API 网关：微服务的统一入口

想象一个没有网关的微服务架构：每个前端应用需要记住十几个后端服务的地址、端口、认证方式。当某个服务升级域名或更换协议时，前端应用面临的是一场灾难级的改版。

API 网关，就是来解决这个问题的。

它是所有客户端请求的**单一入口**，负责请求的路由、转发、认证、限流等一系列横切关注点。没有网关的微服务，就像没有总机的办公楼——每个人都要记住所有分机的号码。

## 网关在架构中的位置

在典型的微服务架构中，API 网关扮演着**流量门户**的角色：

```
┌─────────┐    ┌─────────────┐    ┌──────────────────┐
│  客户端  │───▶│  API 网关   │───▶│  后端微服务集群   │
│ (Web/   │    │ (Gateway)   │    │  ┌─────────────┐ │
│  Mobile)│    │             │    │  │ Service A  │ │
└─────────┘    │  - 路由转发  │    │  │ Service B  │ │
              │  - 认证鉴权  │    │  │ Service C  │ │
              │  - 限流熔断  │    │  │     ...    │ │
              │  - 日志监控  │    │  └─────────────┘ │
              └─────────────┘    └──────────────────┘
```

所有外部流量都必须经过网关，网关再将请求分发到对应的后端服务。这种架构带来几个核心优势：

**统一入口，安全可控**。所有请求都经过网关，网关可以集中处理认证、鉴权、安全检查，无需在每个微服务中重复实现。

**服务解耦，前端轻量**。前端只需知道网关的地址，无需关心后端服务的部署拓扑。服务拆分、迁移对前端透明。

**横切关注，集中处理**。限流、熔断、日志、监控这些「切面」逻辑，在网关层统一处理，后端服务可以专注于业务逻辑。

## API 网关的核心职责

网关不是简单的请求转发，它需要承担一系列职责：

| 核心职责 | 说明 |
|---|---|
| **路由转发** | 根据请求特征（Path、Header、参数）将请求路由到对应的后端服务 |
| **负载均衡** | 在多个服务实例间分配请求，支持轮询、加权、最小连接等策略 |
| **认证鉴权** | 验证请求的合法性，如 JWT Token 校验、API Key 验证、OAuth2 授权 |
| **限流熔断** | 控制请求速率，防止系统过载；在后端故障时快速失败，避免雪崩 |
| **日志监控** | 记录请求日志，采集性能指标，为运维提供可观测性基础 |
| **协议转换** | 支持 HTTP/gRPC、WebSocket 等多种协议，满足不同业务场景 |

:::tip 网关 vs 反向代理

很多人容易把 API 网关和 Nginx 等反向代理混淆。两者确实有相似之处——都能做路由转发、负载均衡、SSL 终止。

但 API 网关更强调**业务层面的能力**，如认证鉴权、协议转换、限流熔断。反向代理更偏向**基础设施层面**，如静态资源服务、TLS 加速。

简单说：Nginx 解决的是「怎么把请求送到正确的服务器」，API 网关解决的是「怎么让正确的服务安全、高效地处理请求」。
:::

## 本章内容速览

如果你对 API 网关还不熟悉，建议按以下顺序阅读：

1. **选型指南**（selection）—— 了解主流网关方案的优缺点，根据团队情况做出选择
2. **核心功能**（core/）—— 掌握路由、过滤器链等通用概念，为深入学习打下基础
3. **Spring Cloud Gateway**（sc-gateway/）—— 如果你使用 Spring Cloud 生态，这是首选方案
4. **Kong**（kong/）—— 如果你需要原生高性能、高可扩展性的网关，Kong 值得考虑

无论你选择哪个方案，核心原理都是相通的。掌握了路由匹配和过滤器链的设计思想，换框架只是配置语法的差异。
