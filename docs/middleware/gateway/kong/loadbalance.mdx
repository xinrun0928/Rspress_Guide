# Kong 服务发现与负载均衡

当后端服务有多个实例时，Kong 需要决定将请求发送到哪个实例。这就是负载均衡的工作。

Kong 提供了丰富的负载均衡策略，支持多种服务发现机制，可以满足从简单到复杂的各种场景。

## 核心概念

Kong 的负载均衡涉及三个核心概念：

| 概念 | 说明 |
|---|---|
| **Upstream** | 上游服务抽象，定义负载均衡的目标服务 |
| **Target** | 上游服务的具体实例（host:port） |
| **Consumer** | API 消费者，可用于基于消费者的负载策略 |

```
┌─────────────────────────────────────────────────────────────┐
│                        Kong Gateway                         │
│                                                             │
│  Upstream: my-service                                       │
│    ├── Target: 192.168.1.1:8080 (weight: 100)             │
│    ├── Target: 192.168.1.2:8080 (weight: 100)             │
│    └── Target: 192.168.1.3:8080 (weight: 50)              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 创建 Upstream 和 Target

### 基本配置

```bash
# 1. 创建 Upstream
curl -X POST http://localhost:8001/upstreams/ \
    --data "name=my-service" \
    --data "slots=10000" \
    --data "healthchecks.active.timeout=5" \
    --data "healthchecks.active.healthy.interval=5" \
    --data "healthchecks.active.unhealthy.interval=5"

# 2. 添加 Target（实例）
curl -X POST http://localhost:8001/upstreams/my-service/targets/ \
    --data "target=192.168.1.1:8080" \
    --data "weight=100"

curl -X POST http://localhost:8001/upstreams/my-service/targets/ \
    --data "target=192.168.1.2:8080" \
    --data "weight=100"

curl -X POST http://localhost:8001/upstreams/my-service/targets/ \
    --data "target=192.168.1.3:8080" \
    --data "weight=50" \
    --data "weight=50"  # 权重低的实例

# 3. 将 Service 关联到 Upstream
curl -X POST http://localhost:8001/services/ \
    --data "name=my-api" \
    --data "host=my-service"  # 使用 Upstream 名称

# 4. 创建 Route
curl -X POST http://localhost:8001/services/my-api/routes/ \
    --data "name=my-route" \
    --data "paths[]=/api"
```

### 使用 declarative configuration

```yaml
# kong.yml
_format_version: "3.0"

upstreams:
  - name: my-service
    slots: 10000
    healthchecks:
      active:
        timeout: 5
        healthy:
          interval: 5
          http_failures: 2
        unhealthy:
          interval: 5
          http_failures: 3
    targets:
      - target: 192.168.1.1:8080
        weight: 100
      - target: 192.168.1.2:8080
        weight: 100
      - target: 192.168.1.3:8080
        weight: 50

services:
  - name: my-api
    host: my-service
    routes:
      - name: my-route
        paths:
          - /api
```

## 负载均衡算法

Kong 支持多种负载均衡算法：

| 算法 | 说明 | 配置 |
|---|---|---|
| **roundrobin** | 轮询（默认） | `algorithm=roundrobin` |
| **consistent-hashing** | 一致性哈希 | `algorithm=consistent-hashing` |
| **least-connections** | 最小连接数 | `algorithm=least-connections` |
| **weighted-roundrobin** | 加权轮询 | 通过 Target weight 配置 |

### 加权轮询

```bash
# 创建带权重的 Target
curl -X POST http://localhost:8001/upstreams/my-service/targets/ \
    --data "target=192.168.1.1:8080" \
    --data "weight=80"  # 接收 80% 流量

curl -X POST http://localhost:8001/upstreams/my-service/targets/ \
    --data "target=192.168.1.2:8080" \
    --data "weight=20"  # 接收 20% 流量
```

### 一致性哈希

一致性哈希确保同一请求特征（如用户 ID）始终路由到同一后端实例：

```bash
# 启用一致性哈希
curl -X PATCH http://localhost:8001/upstreams/my-service/ \
    --data "algorithm=consistent-hashing" \
    --data "hash_on=header" \
    --data "hash_fallback=consumer" \
    --data "hash_on_header=X-User-ID"
```

| 参数 | 说明 |
|---|---|
| `hash_on` | 哈希依据：header、consumer、cookie、ip |
| `hash_fallback` | hash_on 失败时的备选 |
| `hash_on_header` | 当 hash_on=header 时，指定的 Header 名称 |

### 最小连接数

```bash
curl -X PATCH http://localhost:8001/upstreams/my-service/ \
    --data "algorithm=least-connections"
```

优先将请求发送到当前连接数最少的实例，适合长连接场景。

## 服务发现集成

Kong 支持与多种服务发现系统集成：

### DNS 服务发现

Kong 可以配置使用 DNS 进行服务发现：

```yaml
# kong.yml
dns_order:
  - LAST
  - SRV
  - A
  - AAAA
  - CNAME

# SRV 记录的 Target
_target: _http._tcp.my-service.default.svc.cluster.local
```

当 DNS 记录更新时，Kong 会自动刷新 Target 列表。

### Consul 集成

```bash
# 启用 DNS Discovery 插件
curl -X POST http://localhost:8001/plugins/ \
    --data "name=dns-prefetcher" \
    --data "config.dns_strict_ordering=true"
```

对于 Consul，可以使用 `lua-resty-dns` 库直接解析 Consul DNS：

```lua
-- 自定义服务发现
local resolver = require "resty.dns.resolver"
local m = resolver:new({
    nameservers = {"127.0.0.1", 8600},  -- Consul DNS
    retry = true,
})

local answers, err = m:lookup("_my-service._tcp.consul.service.skydns.local")
```

### Kubernetes 集成

在 Kubernetes 环境中，推荐使用 Kong Ingress Controller：

```yaml
# KongIngress 配置
apiVersion: configuration.konghq.com/v1
kind: KongIngress
metadata:
  name: my-ingress
proxy:
  preserve_host: true
  protocol: http
upstream:
  loadbalancer: weighted
  healthcheck:
    active:
      type: http
      http_path: /health
      healthy:
        interval: 5
        successes: 2
      unhealthy:
        interval: 5
        http_failures: 3
```

## 健康检查

Kong 支持主动和被动两种健康检查：

### 主动健康检查

Kong 主动探测 Target 的健康状态：

```bash
curl -X PATCH http://localhost:8001/upstreams/my-service/ \
    --data "healthchecks.active.timeout=5" \
    --data "healthchecks.active.healthy.interval=5" \
    --data "healthchecks.active.healthy.http_failures=2" \
    --data "healthchecks.active.healthy.successes=3" \
    --data "healthchecks.active.unhealthy.interval=5" \
    --data "healthchecks.active.unhealthy.http_failures=3" \
    --data "healthchecks.active.unhealthy.tcp_failures=3" \
    --data "healthchecks.active.http_path=/health"
```

| 参数 | 说明 |
|---|---|
| `timeout` | 健康检查超时时间（秒） |
| `healthy.interval` | healthy 状态检查间隔 |
| `healthy.http_failures` | 多少次失败后标记为 unhealthy |
| `unhealthy.interval` | unhealthy 状态检查间隔 |
| `unhealthy.http_failures` | 多少次失败后标记为 unhealthy |
| `http_path` | 健康检查路径 |

### 被动健康检查（熔断）

被动健康检查基于实际请求结果判断 Target 状态：

```bash
curl -X PATCH http://localhost:8001/upstreams/my-service/ \
    --data "healthchecks.passive.timeout=30" \
    --data "healthchecks.passive.healthy.http_statuses=200,201,204" \
    --data "healthchecks.passive.healthy.successes=3" \
    --data "healthchecks.passive.unhealthy.http_failures=3" \
    --data "healthchecks.passive.unhealthy.tcp_failures=3"
```

### 手动控制健康状态

```bash
# 将 Target 标记为 unhealthy
curl -X POST http://localhost:8001/upstreams/my-service/targets/192.168.1.1:8080/unhealthy

# 将 Target 标记为 healthy
curl -X POST http://localhost:8001/upstreams/my-service/targets/192.168.1.1:8080/healthy

# 查看 Target 健康状态
curl http://localhost:8001/upstreams/my-service/health
```

## 熔断降级

当后端服务出现大量故障时，可以使用 Kong 的熔断能力：

```bash
# 启用插件实现熔断
curl -X POST http://localhost:8001/plugins/ \
    --data "name=circuit-breaker" \
    --data "config.healthy.http_statuses=200,201,204,301,302" \
    --data "config.healthy.successes=5" \
    --data "config.unhealthy.http_failures=3" \
    --data "config.unhealthy.tcp_failures=3"
```

熔断触发后的行为：

1. Kong 停止向 unhealthy 的 Target 发送请求
2. 持续探测 Target 健康状态
3. Target 恢复后自动加入负载均衡池

## 完整配置示例

```yaml
# kong.yml
_format_version: "3.0"

upstreams:
  - name: user-service
    algorithm: roundrobin
    slots: 10000
    healthchecks:
      active:
        enabled: true
        type: http
        http_path: /health
        timeout: 5
        healthy:
          interval: 5
          http_failures: 2
          successes: 2
        unhealthy:
          interval: 5
          http_failures: 3
      passive:
        enabled: true
        type: http
        healthy:
          http_statuses: 200,204
          successes: 3
        unhealthy:
          http_failures: 3
          http_statuses: 500,502,503,504
    targets:
      - target: user-service-1:8080
        weight: 100
      - target: user-service-2:8080
        weight: 100
      - target: user-service-3:8080
        weight: 50  # 降级实例，权重较低

services:
  - name: user-api
    host: user-service
    routes:
      - name: user-route
        paths:
          - /api/users
        strip_path: true
    plugins:
      - name: rate-limiting
        config:
          minute: 100
          policy: redis
```

## 总结

| 功能 | 说明 |
|---|---|
| 负载均衡算法 | roundrobin、consistent-hashing、least-connections |
| 服务发现 | DNS、Consul、K8s |
| 健康检查 | 主动探测 + 被动熔断 |
| 熔断降级 | 基于错误率自动摘除故障实例 |

Kong 的负载均衡能力足以应对大多数生产环境需求。在配置时：

1. **合理设置健康检查参数**：避免误判
2. **使用加权轮询做灰度发布**：通过调整权重控制流量
3. **启用被动健康检查**：实现熔断保护
4. **监控 Target 健康状态**：及时发现异常
