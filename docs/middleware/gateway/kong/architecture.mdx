# Kong 架构：OpenResty + Nginx + Lua

Kong 的高性能秘密，藏在它的技术栈里。

当你深入了解 Kong 的架构，你会发现：它不是简单的「包装」，而是将 Nginx 的高性能、Lua 的灵活性、数据库的可扩展性完美融合的产物。

## Kong 的三层架构

### 数据平面（Data Plane）

数据平面是 Kong 的核心，负责处理所有传入的请求：

```
┌─────────────────────────────────────────────────────────────┐
│                      Kong Data Plane                        │
│  ┌─────────────────────────────────────────────────────┐  │
│  │                     OpenResty                         │  │
│  │  ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐   │  │
│  │  │Nginx Core│ │LuaJIT   │ │Kong Core │ │Plugins   │   │  │
│  │  │        │ │        │ │(请求处理) │ │(扩展能力) │   │  │
│  │  └────┬────┘ └────┬────┘ └────┬────┘ └────┬────┘   │  │
│  └────────┼───────────┼───────────┼───────────┼─────────┘  │
│           │           │           │           │             │
└───────────┼───────────┼───────────┼───────────┼─────────────┘
            │           │           │           │
         HTTP/      Lua       插件链      配置
         TLS       脚本       执行       更新
```

- **Nginx Core**：处理 HTTP/TLS 连接，事件驱动的高性能网络 I/O
- **LuaJIT**：执行 Lua 脚本，实现 Kong 的业务逻辑和插件扩展
- **Kong Core**：请求路由、负载均衡、插件调度
- **Plugins**：可插拔的扩展点，实现认证、限流、日志等功能

### 控制平面（Control Plane）

控制平面负责管理配置，通过 Admin API 提供配置管理接口：

```
┌─────────────────────────────────────────────────────────────┐
│                      Kong Control Plane                      │
│  ┌─────────────────────────────────────────────────────┐  │
│  │                  Kong Admin API                       │  │
│  │  POST /routes  PUT /plugins  GET /services          │  │
│  └─────────────────────────────────────────────────────┘  │
│           │                                              │
│           ▼                                              │
│  ┌─────────────────────────────────────────────────────┐  │
│  │          Configuration Database                      │  │
│  │     (PostgreSQL / Cassandra)                        │  │
│  └─────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

### 数据平面 vs 控制平面

在 DB-less 模式下，数据平面也可以独立运行，无需控制平面：

```
┌─────────────────────────────────────────────────────────────┐
│                     DB-less Mode                            │
│  ┌─────────────────────────────────────────────────────┐  │
│  │                Kong Data Plane                        │  │
│  │  ┌─────────────────────────────────────────────┐    │  │
│  │  │           declarative configuration          │    │  │
│  │  │              (YAML/JSON 文件)                │    │  │
│  │  └─────────────────────────────────────────────┘    │  │
│  └─────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

## OpenResty 核心原理

### 什么是 OpenResty

OpenResty 是将 Nginx 与 LuaJIT 深度集成的 Web 平台：

| 组件 | 作用 |
|---|---|
| Nginx | 高性能 HTTP 服务器，事件驱动 I/O |
| LuaJIT | 即时编译的 Lua 解释器，性能接近 C |
| ngx_http_lua_module | Nginx Lua 模块，连接 Nginx 和 Lua |
| lua-resty-* | 一系列 Lua 库，如 redis、mysql、http 客户端 |

### 请求处理流程

在 OpenResty 中，一个请求的处理流程是这样的：

```
┌─────────────────────────────────────────────────────────────┐
│                      请求生命周期                             │
│                                                             │
│  ┌──────────┐                                               │
│  │  rewrite │  ─── Lua 代码：URL 重写、跳转                    │
│  └────┬─────┘                                               │
│       ▼                                                      │
│  ┌──────────┐                                               │
│  │  access  │  ─── Lua 代码：认证、限流、路由决策               │
│  └────┬─────┘                                               │
│       ▼                                                      │
│  ┌──────────┐                                               │
│  │  filter  │  ─── Lua 代码：请求/响应处理                     │
│  └────┬─────┘                                               │
│       ▼                                                      │
│  ┌──────────┐                                               │
│  │   proxy  │  ─── Nginx：代理到上游服务器                     │
│  └────┬─────┘                                               │
│       ▼                                                      │
│  ┌──────────┐                                               │
│  │   log    │  ─── Lua 代码：日志记录                         │
│  └──────────┘                                               │
└─────────────────────────────────────────────────────────────┘
```

### Kong 如何使用 OpenResty

Kong 在 OpenResty 的生命周期钩子中注入自己的逻辑：

```lua
-- Kong 在 access 阶段执行的逻辑（伪代码）
access_by_lua_block {
    -- 1. 加载路由配置
    local routes = kong.db.routes:find_by_request()

    -- 2. 匹配路由
    local matched_route = kong.router.execute()

    -- 3. 执行插件的 access 钩子
    kong.plugins:execute("access", matched_route)

    -- 4. 负载均衡选择上游
    local upstream = kong.loadbalancer:select(routes)
}
```

## 数据存储

Kong 支持两种数据库作为配置存储后端：

### PostgreSQL

```yaml
# PostgreSQL 配置
database: postgres
pg_host: 127.0.0.1
pg_port: 5432
pg_user: kong
pg_password: kong
pg_database: kong
```

| 优点 | 缺点 |
|---|---|
| 支持复杂查询 | 水平扩展相对复杂 |
| 与企业数据库集成方便 | 需要额外的数据库运维 |
| 事务支持 | |

### Cassandra

```yaml
# Cassandra 配置
database: cassandra
cassandra_contact_points:
  - 192.168.1.1
  - 192.168.1.2
  - 192.168.1.3
cassandra_keyspace: kong
```

| 优点 | 缺点 |
|---|---|
| 天生分布式，高可用 | 查询能力相对弱 |
| 水平扩展简单 | 学习曲线陡峭 |
| 写入性能高 | |

## 配置同步机制

### 数据库模式

在数据库模式下，所有 Kong 节点共享同一个数据库：

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  Kong Node │     │  Kong Node │     │  Kong Node │
│     1       │     │     2       │     │     3       │
└──────┬──────┘     └──────┬──────┘     └──────┬──────┘
       │                    │                    │
       └────────────────────┼────────────────────┘
                            │
                     ┌──────▼──────┐
                     │  PostgreSQL  │
                     │   /Cassandra │
                     └─────────────┘
```

### DB-less 模式

在 DB-less 模式下，配置通过 YAML 文件加载：

```yaml
# kong.yml - 声明式配置
_format_version: "3.0"

services:
  - name: my-service
    url: http://backend-service:8080
    routes:
      - name: my-route
        paths:
          - /api

plugins:
  - name: rate-limiting
    config:
      minute: 100
      policy: redis
```

启动命令：

```bash
kong start --conf kong.yml
```

:::tip DB-less 模式的优势

DB-less 模式有几个显著优势：

- **配置一致性**：所有节点使用相同的配置文件，不存在数据库同步延迟
- **部署简单**：无需管理数据库，适合 Kubernetes 环境
- **版本控制**：配置文件可以直接提交到 Git，管理变更历史
- **无状态**：节点可以随时启停，易于弹性伸缩
:::

```

         │

```

## 性能优化

### 连接池

Kong 复用与上游服务的连接，避免频繁建立 TCP 连接的开销：

```yaml
nginx_http_upstream_zone: shared_kong_upstreams 64k
```

### 共享内存缓存

Kong 使用共享内存缓存路由和插件配置，避免每次请求都查询数据库：

```yaml
mem_cache_size: 512m
```

### Lua 代码缓存

OpenResty 的 Lua 代码在第一次加载后会被 JIT 编译并缓存：

```yaml
lua_ssl_trusted_certificate: /path/to/ca.pem
lua_code_cache: on  # 生产环境必须开启
```

## 总结

Kong 的架构设计体现了几个核心理念：

| 设计理念 | 实现方式 |
|---|---|
| 高性能 | Nginx 事件驱动 + LuaJIT JIT 编译 |
| 可扩展性 | 插件化架构，热加载 |
| 灵活性 | 支持 PostgreSQL 和 Cassandra |
| 简化运维 | DB-less 模式，声明式配置 |

理解 Kong 的架构，能帮助你：

1. 更好地选择部署模式（数据库 vs DB-less）
2. 优化性能配置（连接池、缓存等）
3. 排查问题（理解请求处理流程）
4. 开发自定义插件（基于 OpenResty 生命周期）
