# Spring Cloud Gateway 熔断

一个后端服务故障不可怕，可怕的是故障蔓延到整个系统——服务 A 超时导致调用方等待，调用方的资源被耗尽，进而影响其他正常服务，最终整个系统雪崩。

Spring Cloud Gateway 通过集成 Resilience4j，实现了完善的熔断能力，可以在后端故障时快速失败，保护系统不被拖垮。

## Resilience4j 集成

### 依赖配置

```xml
<dependencies>
    <!-- Resilience4j 核心 -->
    <dependency>
        <groupId>io.github.resilience4j</groupId>
        <artifactId>resilience4j-spring-boot3</artifactId>
        <version>2.2.0</version>
    </dependency>

    <!-- Reactor 支持 -->
    <dependency>
        <groupId>io.github.resilience4j</groupId>
        <artifactId>resilience4j-reactor</artifactId>
        <version>2.2.0</version>
    </dependency>

    <!-- Spring Cloud CircuitBreaker -->
    <dependency>
        <groupId>org.springframework.cloud</groupId>
        <artifactId>spring-cloud-circuitbreaker-reactor-resilience4j</artifactId>
    </dependency>
</dependencies>
```

### 熔断器配置

```yaml
resilience4j:
  circuitbreaker:
    instances:
      backendService:
        registerHealthIndicator: true
        slidingWindowSize: 100           # 滑动窗口大小（请求数）
        minimumNumberOfCalls: 10          # 最小请求数（达到后才开始计算失败率）
        permittedNumberOfCallsInHalfOpenState: 3  # HalfOpen 状态允许的调用数
        automaticTransitionFromOpenToHalfOpenEnabled: true  # 自动从 Open 到 HalfOpen
        waitDurationInOpenState: 30s      # Open 状态持续时间
        failureRateThreshold: 50           # 失败率阈值（百分比）
        eventConsumerBufferSize: 10       # 事件缓冲区大小
```

## 熔断状态机

Resilience4j 的熔断器有三种状态：

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│   ┌──────────┐                      ┌──────────┐          │
│   │  Closed  │────失败率超限────────▶│   Open   │          │
│   │  正常熔断 │                      │  熔断中   │          │
│   └──────────┘                      └──────────┘          │
│        ▲                                  │                  │
│        │                                  │ (等待时间后)      │
│        │                                  ▼                  │
│        │                           ┌──────────┐             │
│        └───────成功───────────────│ HalfOpen │             │
│                                    │ 半开状态  │             │
│                                    └──────────┘             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

| 状态 | 行为 | 触发条件 |
|---|---|---|
| **Closed** | 正常放行请求，统计失败率 | 初始状态 |
| **Open** | 直接拒绝请求，快速失败 | 失败率超过阈值 |
| **HalfOpen** | 放行部分请求试探 | Open 状态持续一定时间后 |

### 状态详解

**Closed 状态**

- 所有请求正常通过
- 熔断器统计最近 `slidingWindowSize` 个请求的失败率
- 当失败率超过 `failureRateThreshold` 时，切换到 Open 状态

**Open 状态**

- 所有请求直接返回降级响应，不再调用后端
- 持续 `waitDurationInOpenState` 后，自动切换到 HalfOpen 状态

**HalfOpen 状态**

- 允许 `permittedNumberOfCallsInHalfOpenState` 个请求通过
- 如果这些试探请求全部成功，切换回 Closed 状态
- 如果任何一个请求失败，立即切换回 Open 状态

## 三种熔断策略

Resilience4j 支持三种熔断策略：

### 1. 失败率熔断

当失败率超过阈值时触发熔断：

```yaml
resilience4j:
  circuitbreaker:
    instances:
      backendService:
        slidingWindowType: COUNT_BASED    # 基于请求数
        slidingWindowSize: 100            # 统计最近 100 个请求
        minimumNumberOfCalls: 10           # 至少 10 个请求后才开始计算
        failureRateThreshold: 50           # 50% 失败率触发熔断
```

适用场景：失败率能准确反映后端健康状况的情况。

### 2. 慢调用比例熔断

当慢调用比例超过阈值时触发熔断：

```yaml
resilience4j:
  circuitbreaker:
    instances:
      backendService:
        slidingWindowType: COUNT_BASED
        slidingWindowSize: 100
        minimumNumberOfCalls: 10
        slowCallRateThreshold: 80          # 80% 慢调用比例触发熔断
        slowCallDurationThreshold: 2s      # 超过 2s 视为慢调用
```

适用场景：后端响应变慢但不完全失败的场景。

### 3. 异常数熔断

当异常数量超过阈值时触发熔断：

```yaml
resilience4j:
  circuitbreaker:
    instances:
      backendService:
        slidingWindowType: COUNT_BASED
        slidingWindowSize: 100
        minimumNumberOfCalls: 10
        recordExceptions:                 # 记录这些异常
          - java.io.IOException
          - java.util.concurrent.TimeoutException
          - feign.FeignException
        ignoreExceptions:                 # 忽略这些异常（不计入失败）
          - java.lang.IllegalArgumentException
```

适用场景：需要精确控制哪些异常应该触发熔断。

## SCG 熔断配置

### 基础配置

在 Gateway 中使用熔断，需要在路由上配置 `CircuitBreaker` 过滤器：

```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: circuit-breaker-route
          uri: lb://backend-service
          predicates:
            - Path=/api/**
          filters:
            - name: CircuitBreaker
              args:
                name: backendCircuitBreaker
                fallbackUri: forward:/fallback
```

### Fallback 降级处理

当熔断触发时，请求会转发到指定的降级端点：

```java
@RestController
@RequestMapping("/fallback")
public class FallbackController {

    @GetMapping("/backend")
    public ResponseEntity<Map<String, Object>> backendFallback() {
        Map<String, Object> response = new HashMap<>();
        response.put("code", 503);
        response.put("message", "服务暂时不可用，请稍后再试");
        response.put("timestamp", System.currentTimeMillis());
        return ResponseEntity.status(503).body(response);
    }

    @PostMapping("/backend")
    public ResponseEntity<Map<String, Object>> backendPostFallback() {
        Map<String, Object> response = new HashMap<>();
        response.put("code", 503);
        response.put("message", "服务暂时不可用，请稍后再试");
        return ResponseEntity.status(503).body(response);
    }
}
```

### 使用 Spring Cloud CircuitBreaker

如果需要更灵活的控制，可以使用 `SpringCloudCircuitBreakerResilience4jFilter`：

```java
@Component
public class CustomCircuitBreakerFilter implements GlobalFilter, Ordered {

    private final ReactiveCircuitBreakerFactory circuitBreakerFactory;

    public CustomCircuitBreakerFilter(
            ReactiveCircuitBreakerFactory circuitBreakerFactory) {
        this.circuitBreakerFactory = circuitBreakerFactory;
    }

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, FilterChain chain) {
        String path = exchange.getRequest().getPath().value();

        // 确定是否需要熔断
        if (!shouldApplyCircuitBreaker(path)) {
            return chain.filter(exchange);
        }

        ReactiveCircuitBreaker circuitBreaker =
            circuitBreakerFactory.create("backendService");

        return circuitBreaker.run(
            chain.filter(exchange),
            throwable -> {
                // 熔断触发时的降级处理
                log.warn("Circuit breaker triggered for path: {}, error: {}",
                    path, throwable.getMessage());
                return handleFallback(exchange);
            }
        );
    }

    private boolean shouldApplyCircuitBreaker(String path) {
        return path.startsWith("/api/") && !path.startsWith("/api/public/");
    }

    private Mono<Void> handleFallback(ServerWebExchange exchange) {
        exchange.getResponse().setStatusCode(HttpStatus.SERVICE_UNAVAILABLE);
        exchange.getResponse().getHeaders().setContentType(MediaType.APPLICATION_JSON);

        String body = """
            {"code":503,"message":"服务暂时不可用，请稍后再试"}
            """;

        DataBuffer buffer = exchange.getResponse().bufferFactory()
            .wrap(body.getBytes());
        return exchange.getResponse().writeWith(Mono.just(buffer));
    }

    @Override
    public int getOrder() {
        return -10;
    }
}
```

## 高级配置

### 嵌套熔断器

对于复杂的微服务调用链，可以配置嵌套的熔断器：

```yaml
resilience4j:
  circuitbreaker:
    instances:
      parentService:
        slidingWindowSize: 100
        failureRateThreshold: 30      # 父级更敏感，30% 就熔断

      childService:
        slidingWindowSize: 100
        failureRateThreshold: 50      # 子级容忍度更高
```

### 事件监听

监听熔断器状态变化，用于监控和告警：

```java
@Component
public class CircuitBreakerEventListener {

    @EventListener
    public void onStateChangeEvent(CircuitBreakerEvent event) {
        CircuitBreaker circuitBreaker = event.getCircuitBreaker();

        switch (event.getType()) {
            case STATE_TRANSITION:
                CircuitBreakerStateTransition transition =
                    (CircuitBreakerStateTransition) event;
                log.warn("Circuit breaker state transition: {} -> {}",
                    transition.getStateFrom(),
                    transition.getStateTo());
                // 发送告警
                sendAlert(circuitBreaker.getName(),
                    transition.getStateFrom().toString(),
                    transition.getStateTo().toString());
                break;

            case ERROR:
                log.error("Circuit breaker error: {}, circuit: {}",
                    event.getThrowable().getMessage(),
                    circuitBreaker.getName());
                break;

            case SUCCESS:
                log.debug("Circuit breaker success recorded");
                break;

            case SLOW_CALL:
                log.warn("Circuit breaker slow call recorded: {}s",
                    event.getDuration().toSeconds());
                break;
        }
    }
}
```

### 健康监控

通过 Actuator 端点监控熔断器状态：

```yaml
management:
  endpoints:
    web:
      exposure:
        include: health,circuitbreakers,circuitbreakerevents
  endpoint:
    health:
      show-details: always
  health:
    circuitbreakers:
      enabled: true
```

访问 `/actuator/health` 查看熔断器健康状态：

```json

}
```

## 总结

| 概念 | 说明 |
|---|---|
| 熔断状态 | Closed（正常）、Open（熔断中）、HalfOpen（半开） |
| 熔断策略 | 失败率、慢调用比例、异常数 |
| Fallback | 熔断时的降级响应 |

熔断配置的关键点：

1. **合理设置 `slidingWindowSize`**：太小不准确，太大反应慢
2. **设置合适的 `failureRateThreshold`**：考虑业务容忍度
3. **配置有效的 Fallback**：降级响应要有意义，不能简单返回空
4. **监控熔断事件**：熔断触发时应该有告警

熔断和限流的区别：

| 维度 | 限流 | 熔断 |
|---|---|---|
| 目的 | 保护系统不被突发流量击垮 | 保护系统不被故障蔓延拖垮 |
| 触发条件 | 请求速率超过阈值 | 后端失败率超过阈值 |
| 处理方式 | 拒绝部分请求 | 快速失败，不调用后端 |
| 恢复方式 | 速率降低后自动恢复 | 后端恢复后自动恢复 |

好的熔断配置，能让你的系统在部分服务故障时依然保持可用，而不是整体崩溃。
