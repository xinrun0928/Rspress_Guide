# Spring Cloud Gateway 工作流程

Spring Cloud Gateway 有三个核心概念：**Route**（路由）、**Predicate**（谓词）、**Filter**（过滤器）。

理解这三个概念及其关系，是掌握 Gateway 的基础。

## 三大核心概念

### Route（路由）

Route 是网关处理请求的最小单元，包含以下信息：

- **id**：路由的唯一标识
- **uri**：目标地址（可以是具体的 HTTP 地址，也可以是 `lb://` 开头的服务名）
- **predicates**：谓词数组，定义路由的匹配条件
- **filters**：过滤器数组，定义请求/响应的处理逻辑

```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: user-service-route
          uri: lb://user-service  # lb:// 表示负载均衡
          predicates:
            - Path=/api/users/**
          filters:
            - StripPrefix=1
```

### Predicate（谓词）

Predicate 用于判断请求是否匹配某个路由。可以理解为 `if` 条件：

```yaml
predicates:
  - Path=/api/users/**          # 路径匹配
  - Header=X-Client-Type, mobile # Header 匹配
  - Query=version, v2            # Query 参数匹配
```

Gateway 内置了丰富的 Predicate：

| Predicate | 作用 | 示例 |
|---|---|---|
| After | 请求时间在指定时间之后 | `After=2024-01-01T00:00:00Z` |
| Before | 请求时间在指定时间之前 | `Before=2024-12-31T23:59:59Z` |
| Between | 请求时间在指定区间内 | `Between=2024-01-01,2024-12-31` |
| Cookie | Cookie 匹配 | `Cookie=session, abc.*` |
| Header | Header 匹配 | `Header=X-Request-Id, \d+` |
| Host | Host 匹配 | `Host=**.example.com` |
| Method | HTTP 方法匹配 | `Method=GET,POST` |
| Path | 路径匹配 | `Path=/api/users/**` |
| Query | Query 参数匹配 | `Query=version, v.*` |
| RemoteAddr | 远程地址匹配 | `RemoteAddr=192.168.1.0/24` |

### Filter（过滤器）

Filter 是请求/响应的处理单元，分为两类：

- **GatewayFilter**：路由级别的过滤器，只对特定路由生效
- **GlobalFilter**：全局过滤器，对所有路由生效

```yaml
filters:
  - AddRequestHeader=X-Request-Foo, Bar    # 添加请求头
  - StripPrefix=1                           # 去掉路径前缀
  - name: RequestSize
    args:
      maxSize: 5000000                      # 限制请求大小
```

## 请求处理流程

当一个请求到达 Gateway 时，完整的处理流程如下：

```
┌──────────────────────────────────────────────────────────────────┐
│                         请求入口                                  │
│                     Netty Server (Netty Handler)                  │
└────────────────────────────┬─────────────────────────────────────┘
                             │
                             ▼
┌──────────────────────────────────────────────────────────────────┐
│                      Route Predicate Handler Mapping             │
│                        路由谓词映射处理器                         │
│                                                                  │
│  遍历所有路由的 Predicate，匹配到第一个满足条件的路由              │
│  如果没有路由匹配，返回 404                                       │

                             │
                             ▼
┌──────────────────────────────────────────────────────────────────┐
│                     Gateway Web Handler                           │

                             │
```

### 关键步骤详解

**1. Route Predicate Handler Mapping**

这一步骤遍历所有配置的路由，找到第一个满足所有 Predicate 条件的路由。

```java
// 伪代码：路由匹配逻辑
for (Route route : routes) {
    boolean matched = route.getPredicates()
        .stream()
        .allMatch(predicate -> predicate.test(exchange));
    if (matched) {
        return route;  // 找到第一个匹配的路由
    }
}
return null;  // 没有匹配，返回 404
```

**2. Gateway Web Handler**

找到匹配的路由后，Gateway 将所有 GlobalFilter 和该路由的 GatewayFilter 合并，构建成一个 FilterChain。

```java
// 伪代码：构建 FilterChain
List<GatewayFilter> filters = new ArrayList<>();
filters.addAll(globalFilters);      // 添加所有全局过滤器
filters.addAll(route.getFilters()); // 添加该路由的过滤器
filters.sort(Comparator.comparingInt(GatewayFilter::getOrder)); // 按 order 排序

return new DefaultGatewayFilterChain(filters);
```

**3. Filter Chain 执行**

FilterChain 的执行遵循「洋葱模型」：

```
请求进入 ─▶ G1.pre ─▶ G2.pre ─▶ G3.pre ─▶ 后端 ─▶ G3.post ─▶ G2.post ─▶ G1.post ─▶ 响应
              │        │        │
           执行中   执行中   执行中
```

每个 Filter 都有 Pre 和 Post 两个阶段的逻辑（通过 `filter()` 和 `then()`/`thenMono()` 区分）。

## 异步非阻塞的秘密

Spring Cloud Gateway 基于 WebFlux，使用 **Reactor** 进行异步非阻塞编程。

传统的 Servlet 模式（Spring MVC）是「一个线程处理一个请求」，线程在整个请求周期内被占用：

```
请求1 ─▶ 线程1 ─▶ 等待 I/O ─▶ 线程1 ─▶ 响应
请求2 ─▶ 线程2 ─▶ 等待 I/O ─▶ 线程2 ─▶ 响应
```

在高并发场景下，大量的线程等待会导致资源耗尽。

WebFlux 使用 **Event Loop** 模式，单个线程可以处理多个请求：

```
EventLoop 线程:
  请求1 接收 ─▶ 等待I/O ─▶ 请求1 处理 ─▶ 等待I/O ─▶ 请求1 响应
                      │
                      ├──▶ 请求2 接收 ─▶ 等待I/O ─▶ ...
                      │
                      ├──▶ 请求3 接收 ─▶ 等待I/O ─▶ ...
```

:::tip 理解 Mono 和 Flux

- `Mono<T>`：表示 0 个或 1 个元素，适用于单个返回值
- `Flux<T>`：表示 0 个或多个元素，适用于流式数据

Gateway 中的 `Mono<Void>` 表示一个异步操作，不返回任何值。

`filter(exchange, chain)` 返回 `Mono<Void>`，表示过滤器执行是异步的。
:::

## 总结

三大核心概念的关系：

| 概念 | 作用 | 类比 |
|---|---|---|
| Route | 定义一个路由规则 | 高速公路的入口匝道 |
| Predicate | 判断请求是否匹配该路由 | 匝道上的条件：必须是京牌车 + 工作日 |
| Filter | 对匹配的请求进行处理 | 匝道上的减速带、收费亭 |

请求处理流程：

```
请求到达 → Predicate 匹配 → FilterChain 构建 → Pre Filter 执行
        → 后端调用 → Post Filter 执行 → 响应返回
```

理解这个流程，你就能理解 Gateway 的工作原理，也能正确配置路由和过滤器。
