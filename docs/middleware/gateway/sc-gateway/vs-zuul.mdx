# Spring Cloud Gateway vs Zuul

当你向 Spring Cloud 团队询问网关方案时，他们可能会说：「别用 Zuul，用 Gateway。」

但这个建议背后到底有什么考量？Zuul 1.x、Zuul 2.x、Spring Cloud Gateway 三者之间有什么区别？老的 Zuul 项目还能继续用吗？

让我们深入分析一下。

## 三代网关的演进

### Zuul 1.x：同步阻塞的先驱

Zuul 1.x 是 Netflix 开源的第一代网关框架，也是 Spring Cloud 早期推荐的方案。

```java
// Zuul 1.x 过滤器示例
public class AuthenticationFilter extends ZuulFilter {

    @Override
    public String filterType() {
        return "pre";  // 前置过滤器
    }

    @Override
    public int filterOrder() {
        return 1;
    }

    @Override
    public boolean shouldFilter() {
        return true;
    }

    @Override
    public Object run() {
        RequestContext ctx = RequestContext.getCurrentContext();
        HttpServletRequest request = ctx.getRequest();

        String token = request.getHeader("Authorization");
        if (token == null || !isValidToken(token)) {
            ctx.setSendZuulResponse(false);
            ctx.setResponseStatusCode(401);
            return null;
        }

        return null;
    }
}
```

**架构特点**：

- 基于 Servlet 2.5，单请求单线程（每个请求一个线程）
- 过滤器使用 `filterType()` 返回 `pre`、`route`、`post` 等类型
- 同步执行，请求在线程中阻塞等待 I/O

### Zuul 2.x：异步非阻塞的演进

Zuul 2.x 对架构进行了彻底重构：

```java
// Zuul 2.x 过滤器示例
public class AuthenticationFilter extends ZuulFilter {
    @Override
    public String filterType() {
        return "pre";
    }

    @Override
    public int filterOrder() {
        return 1;
    }

    @Override
    public boolean shouldFilter() {
        return true;
    }

    @Override
    public Publisher<Void> processZuulFilter(ZuulFilterProcessor processor) {
        RequestContext context = RequestContext.getCurrentContext();
        HttpServletRequest request = context.getRequest();

        String token = request.getHeader("Authorization");
        if (token == null || !isValidToken(token)) {
            context.setSendZuulResponse(false);
            context.setResponseStatusCode(401);
            return Mono.empty();
        }

        return Mono.empty();
    }
}
```

**架构特点**：

- 基于 Netty，异步非阻塞
- 使用 `Publisher<Void>` 返回响应式类型
- 支持 HTTP/2、WebSocket
- 引入了「过滤器 Web 化」的架构

### Spring Cloud Gateway：后起之秀

Spring Cloud Gateway 基于 Spring 5 WebFlux + Project Reactor：

```java
// SCG 过滤器示例
@Component
public class AuthenticationFilter implements GlobalFilter, Ordered {

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, FilterChain chain) {
        String token = exchange.getRequest()
            .getHeaders().getFirst("Authorization");

        if (token == null || !isValidToken(token)) {
            exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
            return exchange.getResponse().setComplete();
        }

        return chain.filter(exchange);
    }

    @Override
    public int getOrder() {
        return -10;
    }
}
```

**架构特点**：

- 基于 Spring 5 WebFlux，天然响应式
- 三大核心概念：Route、Predicate、Filter
- 使用 `Mono`/`Flux` 响应式类型
- 与 Spring Cloud 生态深度集成

## 核心对比

### 性能对比

| 维度 | Zuul 1.x | Zuul 2.x | Spring Cloud Gateway |
|---|---|---|---|
| 线程模型 | 同步阻塞（1 请求 = 1 线程） | 异步非阻塞（EventLoop） | 异步非阻塞（EventLoop） |
| QPS（单实例） | 1~2 万 | 5~10 万 | 2~5 万 |
| 延迟 P99 | 20~50ms | 5~10ms | 10~30ms |
| 内存占用 | 高（线程栈） | 低（EventLoop） | 低（EventLoop） |
| 吞吐量瓶颈 | 线程池大小 | EventLoop 配置 | EventLoop 配置 |

:::warning Zuul 1.x 的性能陷阱

Zuul 1.x 使用线程池处理请求，每个请求在 I/O 等待时占用一个线程。当后端响应变慢时，线程池很快被耗尽，导致新请求排队甚至被拒绝。

在高并发场景下，Zuul 1.x 的性能会急剧下降。
:::

### 功能对比

| 功能 | Zuul 1.x | Zuul 2.x | Spring Cloud Gateway |
|---|---|---|---|
| 路由转发 | ✅ | ✅ | ✅ |
| 负载均衡 | ✅ (Ribbon) | ✅ (内置) | ✅ (Spring Cloud LoadBalancer) |
| 过滤器体系 | ✅ | ✅ | ✅ |
| 动态配置 | ❌ 需配合 Config | ❌ 需配合 Config | ✅ 原生支持 |
| 服务发现 | ✅ | ✅ | ✅ |
| 限流 | ❌ 需插件 | ❌ 需插件 | ✅ 内置 RequestRateLimiter |
| 熔断 | ❌ 需 Hystrix | ❌ 需插件 | ✅ 内置 Resilience4j |
| WebSocket | ❌ | ✅ | ✅ |
| gRPC | ❌ | ✅ | ✅ |
| 热加载路由 | ❌ | ⚠️ 部分支持 | ✅ |

### 易用性对比

| 维度 | Zuul 1.x | Zuul 2.x | Spring Cloud Gateway |
|---|---|---|---|
| 学习曲线 | 低 | 中 | 低 |
| Java 开发者友好度 | 高 | 中 | 高 |
| 文档完善度 | 高 | 中 | 高 |
| Spring Boot 集成 | ✅ | ✅ | ✅ |
| 配置方式 | YAML + Java | YAML + Java | YAML + Java |
| 调试难度 | 低 | 中 | 低 |

## 架构差异详解

### 线程模型对比

**Zuul 1.x 同步模型**：

```
请求1 ─▶ 线程1 ─▶ [等待HTTP响应] ─▶ 线程1 ─▶ 响应
请求2 ─▶ 线程2 ─▶ [等待HTTP响应] ─▶ 线程2 ─▶ 响应
请求3 ─▶ 线程3 ─▶ [等待HTTP响应] ─▶ 线程3 ─▶ 响应
...
```

线程池大小有限（如 200 线程），当后端响应慢时，线程被阻塞，系统很快达到瓶颈。

**SCG / Zuul 2.x 异步模型**：

```
EventLoop 线程:
请求1 接收 ─▶ 等待I/O ─▶ 请求1 处理 ─▶ 等待I/O ─▶ 请求1 响应
              │
              ├──▶ 请求2 接收 ─▶ 等待I/O ─▶ ...
              │
              ├──▶ 请求3 接收 ─▶ 等待I/O ─▶ ...
```

单线程（EventLoop）可以处理大量并发请求，线程数不再成为瓶颈。

### 过滤器执行模型

**Zuul 1.x 过滤器执行顺序**：

```
pre Filter 1 ─▶ pre Filter 2 ─▶ pre Filter N
                    │
                    ▼
              route Filter 1 ─▶ route Filter 2 ─▶ ...
                    │
                    ▼
             HttpClient 调用
                    │
                    ▼
              post Filter 1 ─▶ post Filter 2 ─▶ ...
```

过滤器顺序由 `filterOrder()` 方法控制，同步串行执行。

**SCG 过滤器执行顺序**：

```
┌─────────────────────────────────────────────────────────────┐
│                    Pre 阶段（按 order 升序）                  │
│   GlobalFilter(-100) → GlobalFilter(-10) → GatewayFilter(1)  │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                    Route 阶段                                 │
│                   HTTP 调用                                  │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                    Post 阶段（按 order 降序）                │
│   GatewayFilter(1) → GlobalFilter(-10) → GlobalFilter(-100) │
└─────────────────────────────────────────────────────────────┘
```

## 选择建议

### 选 Spring Cloud Gateway 的场景

- 新项目，从零开始
- 使用 Spring Cloud 全家桶
- 需要内置的限流、熔断能力
- 希望快速开发，Spring 开发者友好
- 需要动态路由配置

### 选 Zuul 2.x 的场景

- 已有 Zuul 1.x 项目，需要升级
- 需要极致性能（比 SCG 高）
- 项目不依赖 Spring Cloud 生态
- 需要更灵活的过滤器扩展

### 暂时保留 Zuul 1.x 的场景

- 现有系统稳定，没有性能问题
- 团队对 Zuul 1.x 非常熟悉
- 业务逻辑与 Zuul 1.x 耦合很深

:::danger 不建议继续使用 Zuul 1.x

Spring Cloud 官方已经停止对 Zuul 1.x 的维护（Spring Cloud 2020.0 移除了对 Zuul 1.x 的依赖）。

如果你的项目还在使用 Zuul 1.x，建议：

1. 有精力 → 直接迁移到 Spring Cloud Gateway
2. 没精力 → 暂时保留，但开始规划迁移
3. 绝对不要在新项目中使用 Zuul 1.x
:::

## 迁移指南

### 从 Zuul 1.x 迁移到 SCG

**路由配置对照**：

```yaml
# Zuul 1.x 配置
zuul:
  routes:
    user-service:
      path: /api/users/**
      url: http://user-service:8080
      stripPrefix: true
```

```yaml
# SCG 配置
spring:
  cloud:
    gateway:
      routes:
        - id: user-service
          uri: http://user-service:8080
          predicates:
            - Path=/api/users/**
          filters:
            - StripPrefix=1
```

**过滤器迁移**：

```java
// Zuul 1.x
public class MyZuulFilter extends ZuulFilter {
    @Override
    public Object run() {
        // 过滤逻辑
        return null;
    }
}

// SCG
@Component
public class MyGatewayFilter implements GlobalFilter {
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, FilterChain chain) {
        // 过滤逻辑
        return chain.filter(exchange);
    }
}
```

## 总结

| 维度 | Zuul 1.x | Zuul 2.x | Spring Cloud Gateway |
|---|---|---|---|
| 架构 | 同步阻塞 | 异步非阻塞 | 异步非阻塞 |
| 性能 | 低 | 高 | 中 |
| Spring 集成 | 好 | 一般 | 最好 |
| 功能完整性 | 一般 | 好 | 好 |
| 社区状态 | 停止维护 | 维护中 | 活跃 |
| 推荐度 | ❌ 不推荐 | ⭐ 可选 | ⭐⭐ 推荐 |

无论你选择哪个方案，核心原理都是相通的。理解过滤器链、路由匹配、熔断限流的原理，换框架只是配置语法的差异。
