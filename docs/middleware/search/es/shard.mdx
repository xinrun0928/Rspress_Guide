# ES 分片策略与副本机制

分片数设多少合适？副本数设多少？数据如何在节点间分配？这些问题看似简单，实则是 Elasticsearch 实践中最重要的配置决策之一。

分片策略不当，轻则查询变慢，重则集群不可用。这篇文章帮你彻底理解分片的规划、分配和副本机制。

## 分片数量规划

### 为什么分片数创建后不可修改？

这是 Elasticsearch 设计中一个让很多人困惑的约束。原因在于 Elasticsearch 使用简单的哈希算法来确定文档归属的分片：

```
shard = hash(_routing) % number_of_primary_shards
```

如果分片数可以随时修改，所有文档的 `_routing` 结果都会改变，数据需要全面迁移。想象一下修改分片数后，10 亿条数据要重新分配——这几乎是不可能完成的任务。

### 如何确定合理的分片数？

分片数的确定需要考虑多个因素：

**数据量预估**是最重要的输入。一个主分片建议的数据量在 20-50GB 之间。如果你的数据量是 1TB，分片数应该设置为 20-50。

**查询并发量**也会影响分片规划。每个查询会扫描多个分片，分片越多，并行度越高，但每个分片的查询开销也会累积。对于需要快速响应的查询，分片数不宜过多。

**节点数量**是另一个约束。每个主分片至少需要分布在不同的 Data 节点上才能保证高可用。如果只有 3 个节点，分片数超过 3 就没有意义了（除非副本为 0）。

```json
PUT /my-large-index
{
  "settings": {
    "number_of_shards": 10,
    "number_of_replicas": 1
  }
}
```

:::tip 实际经验
对于大多数场景，3-5 个分片是比较合理的选择。如果数据量预估在 TB 级别，可以考虑按时间或业务维度拆分索引，而不是一味增加分片数。
:::

### 分片过小与过大的问题

分片过小会导致：

- 每个分片都会消耗一些固定开销（文件句柄、内存）
- 小分片意味着更多 Segment，查询时需要合并更多结果
- 管理开销增加，Master 节点的元数据压力增大

分片过大会导致：

- 单个分片数据量过大，查询变慢
- 故障恢复时间长——恢复一个 100GB 的分片可能需要几小时
- 副本同步压力大

## 副本数量配置

### 副本的作用

副本分片是主分片的完整拷贝，主要有两个作用：

**容灾**：主分片所在节点宕机时，副本可以提升为主分片，保证数据不丢失、服务不中断。

**提高查询吞吐**：查询可以分发到主分片或副本分片，增加并发处理能力。

```json
PUT /my-index/_settings
{
  "number_of_replicas": 2
}
```

副本数可以随时调整。增加副本会立即同步数据，但会占用更多存储空间。

### 副本与一致性

写入主分片后，Elasticsearch 会并发将请求转发给所有副本：

```
写入流程:
1. 客户端 → 协调节点
2. 协调节点 → 主分片
3. 主分片 → 并发 → 副本1, 副本2, 副本3
4. 等待所有副本写入成功
5. 主分片 → 协调节点
6. 协调节点 → 客户端
```

通过 `consistency` 参数可以控制写入一致性级别：

```json
PUT /my-index/_doc/1?consistency=quorum
{
  "title": "Hello World"
}
```

- `one`：只写入主分片
- `quorum`：写入过半节点（计算公式：`(primary + replica) / 2 + 1`）
- `all`：写入所有分片（包括主分片和副本）

对于写入频繁的场景，`quorum` 是平衡性能和安全性的选择。

### 读写分离

副本不仅可以用于容灾，还可以用于查询负载均衡：

```json
PUT /my-index/_settings
{
  "index.routing.allocation.include._tier_preference": "data_hot"
}
```

在冷热数据架构中：

- **热节点**（SSD、高性能）接收写入和新数据查询，配置较多副本
- **温节点**（普通 SSD）接收历史数据查询，副本数可以设为 0
- **冷节点**（大容量 HDD）存储归档数据，只保留必要副本

## 分片分配策略

### 分配器的工作原理

Elasticsearch 的 Master 节点负责分片分配。当索引创建、节点加入或离开时，Master 会重新计算最优分配方案。

分配决策考虑的因素包括：

- **负载均衡**：尽量让各节点的分片数均衡
- **磁盘使用率**：避免将分片分配到磁盘使用率过高的节点
- **节点属性**：支持通过自定义属性控制分配行为
- **分配过滤**：根据条件将特定索引分配到特定节点

### 强制均衡

如果某些节点意外关闭，数据可能会分布不均。重启后，ES 会自动重新均衡。

```bash
POST /_cluster/reroute?retry_failed=true
```

可以手动触发重分配，但要注意：大量数据迁移会占用网络带宽，可能影响集群性能。

### 排除节点

如果某个节点硬件有问题（比如磁盘 I/O 特别慢），可以暂时将其排除在分片分配之外：

```json
PUT /_cluster/settings
{
  "transient": {
    "cluster.routing.allocation.exclude._ip": "192.168.1.100"
  }
}
```

## 故障恢复机制

### 节点故障检测

Elasticsearch 通过两种机制检测节点故障：

**Master 检测**：Master 节点定期向其他节点发送 Ping 请求。如果在 `ping_timeout` 时间内没有响应，该节点被标记为不可用。

**节点间检测**：所有节点相互发送心跳。当大多数节点都认为某个节点不可用时，才正式将其从集群移除。

### 自动故障恢复

节点故障后的恢复流程：

1. Master 检测到节点不可用
2. 将故障节点上的分片标记为 `UNASSIGNED`
3. 根据副本策略，选择合适的节点创建新的副本
4. 从主分片复制数据到新副本
5. 新副本状态变为 `STARTED`

```bash
# 查看分片分配详情
GET /_cat/shards?v&h=index,shard,prirep,state,unassigned.reason
```

### 故障恢复速度

故障恢复速度取决于：

- 网络带宽：数据需要从主分片复制到新副本
- 磁盘 I/O：读取和写入数据的速率
- 分片大小：大分片恢复时间长
- 集群负载：查询和数据迁移竞争资源

可以通过配置控制恢复速度：

```json
PUT /_cluster/settings
{
  "transient": {
    "indices.recovery.max_bytes_per_sec": "100mb"
  }
}
```

## 实践建议

### 分片数规划流程

1. 评估数据总量和增长预期
2. 确定单个分片的目标大小（20-50GB）
3. 计算理论分片数
4. 根据查询并发需求调整
5. 确认节点数量能容纳这些分片

### 监控关键指标

```json
GET /_cat/nodes?v&h=name,shards,disk.used,disk.total,heap.current
```

- `shards`：节点上的分片数
- `disk.used`：磁盘已用空间
- `heap.current`：堆内存使用

### 处理不均衡的分片

如果集群出现分片不均衡（比如某个节点分片数明显多于其他节点），可以手动触发均衡：

```json
POST /_cluster/reroute
{
  "commands": [
    {
      "move": {
        "index": "my-index",
        "shard": 0,
        "from_node": "node1",
        "to_node": "node2"
      }
    }
  ]
}
```

## 总结

分片策略是 Elasticsearch 运维中最重要的决策之一。分片数决定了数据的分布方式和查询的并行度，副本数决定了容灾能力和查询吞吐。

理解分片分配机制，才能在故障时快速定位问题；理解副本机制，才能在性能和数据安全之间找到平衡点。

记住那句老话：**「分片规划要趁早，后期调整代价高。」**
