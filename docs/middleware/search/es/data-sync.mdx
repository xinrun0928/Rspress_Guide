# ES 数据同步方案

业务数据库里是用户、商品、订单，Elasticsearch 里是搜索用的文档——这两份数据如何保持一致？

这是一个看似简单、实则复杂的问题。方案选错了，轻则数据不一致，重则生产事故。这篇文章梳理几种常见的数据同步方案，以及它们的适用场景和坑点。

## 数据同步的核心问题

同步方案需要回答三个问题：

**同步时机**：什么时候同步？数据变更时？还是定期批量？

**同步内容**：同步哪些数据？全量还是增量？

**失败处理**：同步失败了怎么办？如何保证最终一致性？

## 方案一：双写

### 基本原理

在业务代码中，同时操作数据库和 Elasticsearch：

```java
@Service
public class ProductService {

    @Autowired
    private ProductRepository productRepository;

    @Autowired
    private ElasticsearchTemplate elasticsearchTemplate;

    public void createProduct(Product product) {
        // 1. 写入数据库
        Product saved = productRepository.save(product);

        // 2. 写入 ES
        elasticsearchTemplate.index(productToDocument(saved));
    }

    public void updateProduct(Long id, Product product) {
        // 1. 更新数据库
        Product updated = productRepository.update(id, product);

        // 2. 更新 ES
        elasticsearchTemplate.update(productToDocument(updated));
    }

    public void deleteProduct(Long id) {
        // 1. 删除数据库
        productRepository.delete(id);

        // 2. 删除 ES
        elasticsearchTemplate.delete(id);
    }
}
```

### 优点

- 实现简单，代码逻辑清晰
- 延迟低，数据变更后几乎立即同步

### 问题

**事务一致性问题**：数据库和 ES 的操作不在同一个事务中。如果数据库写入成功但 ES 写入失败，数据就不一致了。

**业务耦合**：所有涉及数据的代码都需要修改，侵入性太强。

**ES 故障影响业务**：ES 写入失败可能导致业务报错。

### 改进：异步双写 + 消息队列

```java
public void createProduct(Product product) {
    // 1. 写入数据库
    Product saved = productRepository.save(product);

    // 2. 发送消息，不阻塞
    messageQueue.send("product_created", product);
}

@Service
public class SyncConsumer {

    @RabbitListener(queues = "product_sync")
    public void handleProductSync(ProductMessage msg) {
        try {
            elasticsearchTemplate.sync(msg);
        } catch (Exception e) {
            // 失败重试，最终一致
            retryQueue.send(msg);
        }
    }
}
```

通过消息队列解耦，即使 ES 故障也不会阻塞业务。但依然存在数据库和消息顺序不一致的风险。

## 方案二：定时同步

### 基本原理

定期扫描数据库，将数据全量或增量同步到 ES：

```java
@Scheduled(fixedRate = 300000) // 5分钟
public void syncProducts() {
    long lastSyncTime = getLastSyncTime();

    // 增量查询：只查最近更新的数据
    List<Product> products = productRepository
        .findByUpdatedAtAfter(lastSyncTime);

    for (Product product : products) {
        elasticsearchTemplate.upsert(productToDocument(product));
    }

    updateLastSyncTime(System.currentTimeMillis());
}
```

### 优点

- 实现简单，不影响业务代码
- 不需要引入复杂的中间件

### 问题

**实时性差**：数据变更后可能要等几分钟才能被搜索到。

**数据库压力大**：每次同步都要扫描数据库，数据量大时可能影响业务。

**数据冲突**：同步期间的数据变更可能丢失（脏读）。

### 适用场景

- 对实时性要求不高的场景（如商品搜索延迟几分钟可接受）
- 数据量不大、变化不频繁的场景

## 方案三：Binlog 监听

### 基本原理

监听数据库的变更日志（Binlog），解析变更事件，同步到 ES。

主流方案有 **Canal** 和 **Debezium**：

```
MySQL → Canal/Debezium → Kafka → 消费者 → ES
```

Canal 是阿里开源的 MySQL Binlog 增量订阅组件：

```yaml
# docker-compose.yml
canal:
  image: canal/canal-server
  ports:
    - "11111:11111"
  environment:
    canal.instance.master.address: mysql:3306
    canal.instance.dbUsername: canal
    canal.instance.dbPassword: canal
```

Canal 将 Binlog 解析为 JSON 事件，发送到 Kafka 或直接推送给消费者：

```json
{
  "type": "UPDATE",
  "database": "shop",
  "table": "products",
  "data": {
    "id": 123,
    "name": "iPhone 15",
    "price": 5999
  },
  "old": {
    "price": 5499
  }
}
```

### 优点

- 真正的实时同步，延迟秒级
- 不影响业务代码，无侵入
- 可以保证数据一致性

### 问题

**复杂度高**：需要额外部署 Canal/Debezium 组件。

**解析复杂度**：Binlog 格式需要正确解析，处理DDL变更。

**顺序依赖**：需要保证事件顺序，处理幂等。

### 适用场景

- 对实时性要求高的核心业务
- 有能力维护额外组件的团队

## 方案四：Change Data Capture (CDC)

CDC 是更现代的数据同步方式，支持更多数据源。

### Debezium

Debezium 是 Red Hat 开源的 CDC 平台：

```json
// Debezium 输出格式
{
  "before": null,
  "after": {
    "id": 123,
    "name": "iPhone 15",
    "price": 5999
  },
  "op": "c",  // c=create, u=update, d=delete, r=snapshot
  "ts_ms": 1705123456789
}
```

通过 Debezium Connect 可以将变更写入 Kafka，再由消费者同步到 ES。

### Flink CDC

Flink CDC 支持直接从数据库读取变更流：

```java
DataStream<String> products = env
    .fromSource(
        MySqlSource.<String>builder()
            .hostname("mysql")
            .port(3306)
            .databaseList("shop")
            .tableList("shop.products")
            .username("cdc")
            .password("cdc")
            .deserializer(new JsonDebeziumDeserializationSchema())
            .build(),
        WatermarkStrategy.noWatermarks(),
        "MySQL CDC Source"
    );
```

## 方案对比

| 方案 | 实时性 | 实现复杂度 | 数据一致性 | 侵入性 |
|---|---|---|---|---|
| 同步双写 | 最高 | 低 | 差 | 高 |
| 异步双写 | 高 | 中 | 中 | 中 |
| 定时同步 | 低 | 低 | 差 | 低 |
| Canal | 高 | 中 | 好 | 无 |
| Debezium | 高 | 高 | 好 | 无 |
| Flink CDC | 高 | 高 | 好 | 无 |

## 实践建议

### 小型项目

从简单的定时同步开始，等业务发展起来再引入更复杂的方案。

```java
@Scheduled(fixedRate = 60000)
public void incrementalSync() {
    // 增量同步
}
```

### 中型项目

引入消息队列做异步双写，平衡实时性和可靠性。

```
业务代码 → MySQL + MQ → 消费者 → ES
```

### 大型项目

使用 CDC 方案，实现真正的实时同步和无侵入集成。

```
MySQL → Debezium → Kafka → Flink/消费者 → ES
```

### 补偿机制

无论选择哪种方案，都要设计补偿机制：

```java
public void fullResync() {
    // 定期全量重建索引
    reindexAll();

    // 比对差异，补充遗漏
    compareAndFix();
}
```

定期全量对比一次，确保不会因为各种边界情况导致数据漂移。

## 总结

数据同步是 ES 应用中的经典难题。没有完美的方案，只有适合业务的方案。

同步双写最简单但一致性差；定时同步最省事但实时性差；CDC 方案最强大但复杂度高。

选择时要考虑：
- 业务对实时性的要求
- 团队维护额外组件的能力
- 数据一致性的容忍度

在大多数场景下，**异步双写 + 定期全量校验**是一个不错的折中选择。
