# 倒排索引原理与分词器

你知道为什么 Google 能在毫秒级从数十亿网页中找到「咖啡拉花教程」吗？

答案藏在一种叫「倒排索引」的数据结构里。这是信息检索领域的基石技术，也是 Elasticsearch 高速搜索的秘密武器。

这篇文章会深入剖析倒排索引的原理，解释 Elasticsearch 是如何将海量文本转化为可高速检索的数据结构的。同时，我们还会探讨分词器——决定哪些词能被索引、哪些不能的关键组件。

## 正排索引 vs 倒排索引

### 正排索引（Forward Index）

传统的数据库索引是「正排」的：以文档 ID 为主键，每个文档记录它包含哪些词。

```
文档1: "Elasticsearch 入门" → {词1: 位置1, 词2: 位置3}
文档2: "Elasticsearch 进阶" → {词1: 位置1, 词2: 位置3}
文档3: "全文检索原理" → {词1: 位置1, 词2: 位置2}
```

查找包含「Elasticsearch」的文档，需要扫描所有文档，检查每个文档是否包含这个词。这在海量数据下效率极低——你得把整个图书馆的书翻一遍，才能回答「哪些书提到了 Elasticsearch」。

### 倒排索引（Inverted Index）

倒排索引反其道而行之：以词为主键，记录每个词出现在哪些文档中。

```
词 "Elasticsearch": 文档1, 文档2
词 "入门": 文档1
词 "原理": 文档3
```

现在查找包含「Elasticsearch」的文档，只需要一步——查一下这个词映射到哪些文档。这就像字典的拼音索引：你不再需要逐页翻书，而是直接翻到拼音字母对应的页码范围。

这就是搜索引擎能够在海量数据中快速定位答案的**核心秘密**。

## 倒排索引的内部结构

### 基本构成

一个完整的倒排索引由三个部分组成：

**Term Dictionary（词字典）**：记录所有被索引的词，以及每个词在倒排列表中的位置。

**Term Index（词索引）**：一种紧凑的数据结构（如 FST），用于快速定位 Term Dictionary 中的词。想象成字典前面的拼音索引页——你不需要逐字查找，翻到对应字母的页面就行。

**Posting List（倒排列表）**：记录每个词出现在哪些文档中，以及在文档中的位置信息。

```
┌─────────────────────────────────────────────────────────────┐
│                        倒排索引                              │
├─────────────────────────────────────────────────────────────┤
│  ┌─────────────┐    ┌─────────────┐    ┌─────────────┐   │
│  │ Term Index  │───▶│    Term     │───▶│ Posting List│   │
│  │   (FST)     │    │ Dictionary  │    │  (doc_id,   │   │
│  │             │    │             │    │  position)   │   │
│  └─────────────┘    └─────────────┘    └─────────────┘   │
└─────────────────────────────────────────────────────────────┘
```

### Posting List 的存储优化

Posting List 不是简单的 doc_id 列表。为了支持高效的交集、并集计算，以及相关性评分，它存储了丰富的信息：

```json
{
  "doc_id": 1001,
  "term_frequency": 3,
  "positions": [2, 15, 28],
  "offset": [{"start": 0, "end": 13}]
}
```

- `doc_id`：文档唯一标识
- `term_frequency`：词在文档中出现的次数
- `positions`：词在文档中的位置列表
- `offset`：词在原始文本中的字符偏移

这种存储格式支持很多高级查询：

**短语查询**：查询「Elasticsearch 入门」，需要找到词「Elasticsearch」和「入门」位置相邻的文档。

**相关性评分**：词在文档中出现次数越多（TF），相关性越高。

**高亮显示**：根据位置信息，可以精确提取搜索关键词周围的片段。

### 跳表与压缩

Posting List 通常很长（一个常用词可能出现在数百万文档中）。为了加速交集计算，Posting List 会使用**跳表**（Skip List）结构。

```
普通列表: [1, 5, 9, 13, 17, 21, 25, 29, 33]
跳表:     [1] ──────► [9] ──────► [17] ──────► [29]
          │          │          │          │
          ▼          ▼          ▼          ▼
        [5, 9]    [13, 17]   [21, 25]   [33]
```

查找两个列表的交集时，先在跳表层面快速跳过不可能匹配的块，再在块内精细查找。平均复杂度从 O(n+m) 降低到 O(n+m)/k，其中 k 是跳表步长。

Posting List 还会进行**压缩**。常用的压缩算法包括：

- **FOR（Frame of Reference）**：将 doc_id 之间的差值用更少的位存储
- **RBM（Roaring Bitmap）**：将 doc_id 空间划分为固定大小的块，每块使用最优编码

## FST：词索引的秘密

### 什么是 FST

FST（Finite State Transducer，有限状态转换器）是一种确定性的有限状态机，用于高效地存储和查询字符串集合。

对于词汇表 `[cat, car, card, care, caref]`：

```
    (c) ──▶ (a) ──▶ (t) ──▶ ●
                     │
                     └──▶ (r) ──▶ (d) ──▶ ●
                               │
                               └──▶ (e) ──▶ ● ──▶ (f) ──▶ ●
```

每个节点代表一个状态，每条边代表一个字符，终端节点标记为实心圆点。

FST 的查询复杂度是 **O(length)**——只和查询字符串的长度有关，和词汇表大小无关。无论词汇表有 10 万个词还是 1000 万个词，查询「card」的性能是一样的。

### FST 在 Elasticsearch 中的应用

Elasticsearch 在内存中维护 FST 结构，用于快速定位 Term Dictionary：

1. 查询「care」
2. 沿着 FST 路径 `c → a → r → e` 快速定位
3. 到达终端节点，获取指向 Posting List 的指针
4. 读取 Posting List，获取匹配文档

FST 还支持**前缀查询**和**范围查询**。查询以「car」开头的所有词，只需要从 FST 的 `c → a → r` 节点开始，遍历所有可达的终端节点即可。

## 分词器（Analyzer）

倒排索引的质量直接取决于分词——分词做得好，搜索才能准确。

### 分词器的组成

Elasticsearch 的分词器由三个部分组成：

**Character Filter（字符过滤器）**：对原始文本进行预处理，比如去除 HTML 标签、转换特殊字符。

**Tokenizer（分词器）**：将文本切分成词（Token）。这是分词的核心。

**Token Filter（词过滤器）**：对切分出的词进行后处理，比如转小写、同义词替换、移除停用词。

```
输入文本: "<p>Elasticsearch is <b>GREAT</b>!</p>"

         ▼ Character Filter (去除HTML)
"Elasticsearch is GREAT!"

         ▼ Tokenizer (按空格分词)
["Elasticsearch", "is", "GREAT"]

         ▼ Token Filter (转小写、移除停用词)
["elasticsearch", "great"]
```

### 内置分词器

Elasticsearch 提供了多种内置分词器：

| 分词器 | 说明 | 适用场景 |
|---|---|---|
| `standard` | 默认分词器，按空格和标点分词，转小写 | 通用英文文本 |
| `simple` | 按非字母字符分词，转小写 | 英文，不保留数字和特殊字符 |
| `whitespace` | 按空格分词，不转小写 | 日志等结构化文本 |
| `keyword` | 不分词，整个文本作为一个词 | 精确匹配的场景 |
| `pattern` | 根据正则表达式分词 | 自定义分词规则 |
| `language` | 针对特定语言优化 | 各语种文本 |

```json
PUT /my-index
{
  "settings": {
    "analyzer": {
      "my_analyzer": {
        "type": "custom",
        "tokenizer": "standard",
        "filter": ["lowercase", "stop"]
      }
    }
  },
  "mappings": {
    "properties": {
      "title": {
        "type": "text",
        "analyzer": "my_analyzer"
      }
    }
  }
}
```

### 中文分词的特殊性

英文分词按空格切分即可，但中文「我今天很开心」该怎么切？

- 按字切分：「我」「今」「天」「很」「开」「心」——颗粒度太细
- 按词切分：「我」「今天」「很」「开心」——需要词典和算法

中文分词主流方案有三种：

**IK 分词器**：最流行的中文分词插件，支持字典扩展和智能分词。

```bash
./bin/elasticsearch-plugin install analysis-ik
```

```json
{
  "settings": {
    "analysis": {
      "analyzer": {
        "ik_max_word": {
          "type": "ik_smart",
          "path": "analysis-ik/dic"
        },
        "ik_smart": {
          "type": "ik_smart",
          "path": "analysis-ik/dic"
        }
      }
    }
  }
}
```

**结巴分词**：Python 生态流行的分词库，也有 Java 版本可集成。

**HanLP 分词**：基于 CRF 的分词器，支持词性标注和命名实体识别。

:::tip IK 分词模式选择
`ik_max_word`：穷尽词库的所有可能组合，比如「中华人民共和国」会切分为「中华人民共和国」「中华人民」「华人」「共和国」等。适合 Search 场景，确保不错过任何可能的匹配。

`ik_smart`：智能分词，尝试理解语义后切分。比如「中华人民共和国」只切分为「中华人民共和国」或「中华人民」「共和国」。适合 Indexing 场景，索引体积更小。
:::

### 分词对搜索的影响

分词方式直接影响搜索体验：

```json
// 索引时的分词
POST /my-index/_doc
{
  "text": "Elasticsearch 入门教程"
}

// 查询时的分词（默认使用索引字段的 analyzer）
GET /my-index/_search
{
  "query": {
    "match": {
      "text": "elasticsearch"
    }
  }
}
```

如果文本被切分为 `["elasticsearch", "入门", "教程"]`，查询「elasticsearch」能匹配到。但如果被切分为 `["el", "ast", ...]`（按字切分），查询「elasticsearch」就匹配不到了。

## Segment：分片内部结构

### Segment 是什么

每个分片由多个 Segment（段）组成。每个 Segment 都是一个独立的、完整的倒排索引。

```
Shard (分片)
├── Segment_1 (倒排索引)
├── Segment_2 (倒排索引)
└── Segment_3 (倒排索引)
```

新写入的数据首先写入内存 Buffer，然后刷新（Refresh）到一个新的 Segment。查询时，Elasticsearch 会同时查询所有 Segment，然后合并结果。

### Segment Merging

Segment 一旦创建，就**不可修改**。这个设计大大简化了并发控制——不需要锁来保护读写冲突。

但这会导致 Segment 数量越来越多，每个 Segment 都要消耗文件句柄和内存。Elasticsearch 会定期将小的 Segment 合并成大的 Segment，这个过程叫 **Segment Merging**。

```
小 Segment: [S1, 100条] [S2, 150条] [S3, 80条] [S4, 200条]
    ↓ Merging
大 Segment: [M1, 530条]
```

Merging 过程：
1. 选择要合并的 Segment
2. 读取所有小 Segment 的数据
3. 按规则重写到大 Segment
4. 删除小 Segment

合并时会自动清理已标记删除的文档（删除操作不会真正删除数据，只是标记为删除）。

### Merge 策略调优

```json
PUT /my-index/_settings
{
  "index": {
    "merge_scheduler.max_thread_count": 1,
    "refresh_interval": "5s"
  }
}
```

- `refresh_interval`：刷新间隔，越大越减少 Segment 数量，但数据可见性延迟越高
- `max_thread_count`：合并线程数，SSD 可以设高一些，机械硬盘建议设为 1

## 总结

倒排索引是搜索引擎的基石。通过「词 → 文档」的映射关系，实现了毫秒级的全文检索能力。

FST 让词查询变得极其高效，Posting List 存储了丰富的匹配信息支撑高级查询，分词器决定了哪些内容能被检索到。

理解这些底层原理，你才能在实践中做出正确的配置决策：选择什么分词器？如何优化查询性能？如何避免中文分词的坑？
