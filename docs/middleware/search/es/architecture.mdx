# Elasticsearch 架构

为什么 Elasticsearch 能够轻松处理 PB 级别的数据？为什么某个节点挂了，集群还能正常服务？为什么向集群写入一条数据，几乎立刻就能被搜索到？

要回答这些问题，你需要理解 Elasticsearch 的分布式架构。

## 分布式架构概览

Elasticsearch 的分布式架构，本质上是一个**对等网络**（Peer-to-Peer）。没有中心化的主控节点，每个节点都是平等的，都可以接收请求。

当你向集群中的任何一个节点发送请求，这个节点就变成了「协调节点」，负责：

1. 解析请求内容
2. 确定需要访问哪些分片
3. 将请求转发给对应的分片所在节点
4. 汇总各分片的返回结果
5. 返回给客户端

```
                    ┌──────────────┐
                    │   Client     │
                    └──────┬───────┘
                           │
                    ┌──────▼───────┐
                    │  Node A      │
                    │ (协调节点)   │
                    └──┬───────┬───┘
                       │       │
              ┌────────┴┐   ┌──┴────────┐
              │  Node B  │   │  Node C   │
              │ P0   R1  │   │ P1   R0   │
              └──────────┘   └───────────┘
```

这个设计有什么好处？

第一，**没有单点瓶颈**。任何节点都可以接收请求，请求可以被分散到不同节点。

第二，**高可用**。任何节点宕机，只要不是 Master 节点，集群依然可以正常服务。

第三，**可扩展**。增加节点，集群可以自动 rebalance，让数据均匀分布。

## 分片分布与路由机制

### 分片分配策略

当你创建一个索引时，Elasticsearch 会根据 `number_of_shards` 配置，将数据划分为多个分片。这些分片会被分配到不同的 Data 节点上。

分配策略会考虑多种因素：

- **负载均衡**：尽量让每个节点的分片数接近
- **磁盘使用**：避免磁盘使用率过高的节点接收新分片
- **机架感知**：如果节点分布在不同机架，分片会优先分布在不同机架
- **热数据隔离**：写入频繁的热点数据会分配到专用的高性能节点

### 路由机制

当你发送查询请求时，协调节点需要知道「这条数据存在哪个分片上」。这个映射关系由一个简单的公式决定：

```
shard = hash(_routing) % number_of_primary_shards
```

`_routing` 默认是文档的 `_id` 字段。你也可以在写入时指定自定义的 routing key：

```json
PUT /my-index/_doc/user123?routing=tech
{
  "title": "文章标题",
  "user_id": "user123"
}
```

使用自定义 routing 可以让相关数据落在同一个分片上，减少查询时的分片扫描范围。比如博客系统，可以按 `user_id` 作为 routing key，这样查询某个用户的所有文章，只需要访问一个分片而不是全部分片。

```json
GET /my-index/_search?routing=user123
{
  "query": {
    "term": { "user_id": "user123" }
  }
}
```

:::warning routing 的代价
自定义 routing 会导致数据分布不均匀——如果某个用户有 1000 万篇文章，所有数据都会落在一个分片上，这个分片会变得异常庞大。所以 routing 要谨慎使用，确保数据分布足够分散。
:::

## Master 节点与集群管理

虽然 Elasticsearch 是对等网络，但集群管理需要一个「协调者」——这就是 Master 节点。

Master 节点负责维护集群的**元数据**，包括：

- 索引的名称、分片数、映射配置
- 分片到节点的映射关系
- 集群健康状态

Master 节点的选举采用 **Bully 算法**的变体：

1. 节点启动时，假设自己是 Master
2. 向所有候选 Master 节点发送选举请求
3. 得到「我比你大」的响应，则退出选举
4. 如果没有人比我大，我就是 Master

```bash
# 查看集群健康状态
GET /_cluster/health

{
  "cluster_name": "my-cluster",
  "status": "green",
  "number_of_nodes": 3,
  "number_of_data_nodes": 3,
  "active_primary_shards": 45,
  "relocating_shards": 0,
  "active_shards": 90
}
```

集群健康状态有三种：

- **green**：所有主分片和副本分片都正常
- **yellow**：所有主分片正常，但部分副本分片不可用
- **red**：存在不可用的主分片

### 脑裂问题

在网络分区的情况下，可能会出现多个节点认为自己是 Master 的情况，这就是「脑裂」。比如 3 节点集群，网络分区导致节点 A 和节点 B 无法通信，它们都可能认为自己是 Master。

Elasticsearch 通过 `minimum_master_nodes` 配置来防止脑裂：

```
minimum_master_nodes = (number_of_master_nodes / 2) + 1
```

对于 3 个候选 Master 节点，`minimum_master_nodes` 应该设置为 2。这样，即使节点 A 和节点 B 断了联系，节点 A 认为只有 1 个 Master 候选（自己），不满足 2 的要求，不会成为 Master。

## 故障容错机制

### 节点故障检测

集群中的节点会相互发送心跳（Ping），检测彼此是否存活：

- 如果一个节点在规定时间内没有响应心跳，会被标记为「不可用」
- 如果不可用的是 Master 节点，触发重新选举
- 如果不可用的是 Data 节点，触发分片重新分配

### 分片故障恢复

当某个节点宕机，它的分片会暂时处于「unassigned」状态。Master 节点检测到这种情况后，会启动恢复流程：

1. 从其他节点复制副本分片，填补空缺
2. 如果宕机节点恢复，它上面的副本会被删除，数据会从新的副本同步

```bash
# 查看分片分配情况
GET /_cat/shards?v

index          shard prirep state    docs   store ip        node
my-index       0     p      STARTED  12345  15mb  192.168.1.1 node-1
my-index       0     r      STARTED  12345  15mb  192.168.1.2 node-2
my-index       1     p      STARTED  13456  16mb  192.168.1.2 node-2
my-index       1     r      STARTED  13456  16mb  192.168.1.3 node-3
```

`prirep` 列显示 `p` 表示主分片，`r` 表示副本分片。`state` 显示分片状态：`STARTED` 正常运行，`RELOCATING` 正在迁移，`INITIALIZING` 正在初始化。

## 写入流程

了解了集群架构，再来看数据是如何写入的：

1. **客户端发送请求**到任意节点（协调节点）
2. **路由计算**：根据 `_routing` 计算目标分片
3. **转发写入**：协调节点将请求转发到主分片所在节点
4. **主分片写入**：主分片执行写入操作，写入内存 Buffer
5. **副本同步**：主分片将请求并发转发给所有副本分片
6. **等待响应**：主分片等待所有副本分片写入成功
7. **返回客户端**：协调节点返回写入成功

```
Client → [Coordinating Node] → [Primary Shard] → [Replica 1]
                                       ↓
                               [Replica 2]
```

这个流程确保了数据的**强一致性**——写入主分片成功 + 所有副本同步成功，才算真正写入成功。

## 读取流程

读取流程与写入类似，但有一个关键区别：**主分片和副本分片都可以处理读取请求**。

1. 协调节点收到读取请求
2. 根据 `preference` 参数选择目标分片（默认是随机选择）
3. 将请求转发给选中的分片（可能是主分片或副本）
4. 各分片返回结果给协调节点
5. 协调节点合并结果并返回客户端

`preference` 参数可以控制查询路由策略：

```json
GET /my-index/_search?preference=_local
{
  "query": { "match": { "title": "elasticsearch" } }
}
```

`_local` 表示优先查询本地节点，减少网络开销。`_primary` 强制只查询主分片，适合对数据一致性要求高的场景。

## 总结

Elasticsearch 的分布式架构，是它能够处理海量数据的根本。

对等网络设计让每个节点都可以接收请求；Master 节点负责集群协调；分片机制实现数据分散存储和并行查询；副本机制保证高可用。

理解这些架构特性，才能在实际使用中做出正确的配置决策——分片数设多少、要不要自定义 routing、如何配置节点角色。
