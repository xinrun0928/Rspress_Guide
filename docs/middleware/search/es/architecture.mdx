# Elasticsearch 架构：Master 节点、数据节点、Coordinating 节点

一座摩天大楼能屹立不倒，不是因为某根柱子特别粗，而是因为有一套精密的结构设计。Elasticsearch 集群也是如此——它不是单点英雄的舞台，而是一个协作紧密的分布式系统。

## 1. ES 集群的本质

一个 ES 集群（Cluster）由多个节点（Node）组成，这些节点协同工作，存储数据并提供搜索服务。

```
┌─────────────────────────────────────────────────────┐
│                    ES Cluster                        │
│                                                      │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐│
│  │  Node 1 │  │  Node 2 │  │  Node 3 │  │  Node 4 ││
│  │(Master) │  │(Data)   │  │(Data)   │  │(Coordin) ││
│  └─────────┘  └─────────┘  └─────────┘  └─────────┘│
│                                                      │
└─────────────────────────────────────────────────────┘
```

**集群的关键特性：**

- **水平扩展**：增减节点即可扩展容量
- **自动发现**：节点自动加入集群
- **故障转移**：节点挂了，数据自动从副本恢复
- **负载均衡**：请求自动路由到合适的节点

## 2. Master 节点：集群的大脑

**Master 节点** 负责集群级别的管理操作，是整个集群的「指挥官」。

### 2.1 Master 节点的职责

```
Master 节点职责
    │
    ├── 集群状态管理（Cluster State）
    │       ├── 索引的创建/删除
    │       ├── 分片的分配/移动
    │       └── 节点的上线/下线
    │
    ├── 元数据操作
    │       ├── Index Mapping
    │       └── Index Settings
    │
    └── 不参与数据读写（可以配置）
```

### 2.2 Master 选举机制

ES 使用 **Bully 算法** 进行 Master 选举：

1. 每个节点都有一个 `nodeId`，转为数值后排序
2. 所有节点投票，得票超过一半的成为 Master
3. 如果 Master 挂了，剩余节点重新选举

```java
// 选举策略：节点 ID 数值最小的优先
// 但实际更复杂，考虑版本号、集群状态等
nodeId = UUID.randomUUID().toString()
```

> **为什么不是简单比大小？**
>
> 因为要考虑「谁是最新加入集群的」。新节点可能持有最新的集群状态，它应该优先成为 Master。

### 2.3 脑裂问题（Split-Brain）

「脑裂」是分布式系统中的经典问题：两个节点都认为自己是 Master，各自主导一部分数据。

**ES 如何防止脑裂？**

- **最小 Master 节点数**：通过 `discovery.zen.minimum_master_nodes` 配置
- 公式：`N = (总节点数 / 2) + 1`
- 3 个节点：`minimum_master_nodes = 2`
- 5 个节点：`minimum_master_nodes = 3`

```
3 节点集群，minimum_master_nodes = 2

正常情况：
┌────────┐     ┌────────┐     ┌────────┐
│ Master │ ←→  │ Node 2 │ ←→  │ Node 3 │
│  ✓     │     │  ✓     │     │  ✓     │
└────────┘     └────────┘     └────────┘

网络分区后（Node 3 断开）：
┌────────┐     ┌────────┐       ┌────────┐
│ Master │ ←→  │ Node 2 │  ×  ×  │ Node 3 │ ← 孤立，无效
│  ✓     │     │  ✓     │       │  ?     │
└────────┘     └────────┘       └────────┘
```

## 3. Data 节点：数据的守护者

**Data 节点** 存储分片数据，是真正的「体力劳动者」。

### 3.1 Data 节点的职责

```
Data 节点职责
    │
    ├── 文档 CRUD 操作
    │       ├── 写入：Index、Update、Delete
    │       └── 读取：Get、Search
    │
    ├── 分片级操作
    │       ├── 分片级别的搜索
    │       ├── 聚合计算
    │       └── 排序
    │
    └── 数据管理
            ├── Segment 合并
            ├── 副本同步
            └── 副本恢复
```

### 3.2 Data 节点的配置

```java
// elasticsearch.yml
node.master: false    // 不参与 Master 选举
node.data: true       // 存储数据
node.ingest: false    // 不做数据预处理
```

### 3.3 分片分配策略

ES 使用 **shard allocation filtering** 来决定分片分配到哪个节点：

```java
// 给节点打标签
// Node A
node.attr.box_type: hot

// Node B
node.attr.box_type: warm

// 索引配置：将主分片放 hot 节点，副本放 warm 节点
{
  "index.routing.allocation.include.box_type": "hot",
  "index.routing.allocation.exclude.box_type": "warm"
}
```

这个特性可以实现**冷热分离架构**：

```
┌──────────────────────────────────────────────────┐
│              冷热分离架构                         │
│                                                   │
│  Hot Tier (SSD)    │    Warm Tier (HDD)         │
│  ┌──────────────┐  │    ┌──────────────┐         │
│  │ 最近3天的数据 │  │    │ 3天前的数据  │         │
│  │ 写入/查询频繁 │  │    │ 查询较少     │         │
│  └──────────────┘  │    └──────────────┘         │
│                                                   │
└──────────────────────────────────────────────────┘
```

## 4. Coordinating 节点：请求的调度者

**Coordinating 节点** 负责接收请求、聚合结果，本身不存储数据。

### 4.1 请求处理流程

```
Client 请求
    │
    ▼
Coordinating 节点（接收请求）
    │
    ├── Query Phase：广播到所有相关分片
    │
    ├── Fetch Phase：汇总结果，返回给客户端
    │
    ▼
返回响应
```

### 4.2 每个节点都是 Coordinating 节点

默认情况下，**每个节点都是 Coordinating 节点**。你可以通过配置改变这一点：

```java
// elasticsearch.yml
node.master: true    // 可以参与选举
node.data: true      // 可以存储数据
node.ingest: true    // 可以做预处理

// 如果只做协调：
node.master: false
node.data: false
node.ingest: false
```

### 4.3 什么时候需要专用 Coordinating 节点？

当数据节点超过 10 个，或者查询负载很高时，考虑增加 Coordinating 节点：

- 分离负载，避免 Master 节点压力过大
- 专门的查询路由层
- 可以水平扩展

## 5. Ingest 节点：数据的预处理器

**Ingest 节点** 可以在索引前对数据做预处理，类似 Logstash 的 Filter 能力。

```java
// 定义一个 Pipeline
PUT _ingest/pipeline/my-pipeline
{
  "description": "预处理日志数据",
  "processors": [
    {
      "gsub": {
        "field": "message",
        "pattern": "\\n",
        "replacement": " "
      }
    },
    {
      "date": {
        "field": "timestamp",
        "formats": ["UNIX"]
      }
    }
  ]
}

// 使用 Pipeline 索引文档
PUT my-index/_doc/1?pipeline=my-pipeline
{
  "message": "Error\nline2\nline3",
  "timestamp": "1700000000"
}
```

## 6. 节点角色总结

| 角色 | Master | Data | Coordinating | Ingest | 默认角色 |
|-----|--------|------|---------------|--------|----------|
| 集群管理 | ✓ | ✗ | ✗ | ✗ | Master-eligible |
| 存储数据 | ✗ | ✓ | ✗ | ✗ | Data |
| 路由请求 | ✗ | ✗ | ✓ | ✗ | Coordinating |
| 数据预处理 | ✗ | ✗ | ✗ | ✓ | Ingest |

## 8. 面试高频问题

### Q1：Master 节点挂了，集群还能读写吗？

**答案**：取决于数据节点是否存活。

- 如果还有数据节点，集群会重新选举 Master，可以继续读写
- 如果只剩一个节点，它会自动成为 Master
- 如果 Master 和所有数据节点都挂了，集群不可用

### Q2：如何避免脑裂？

**答案**：正确配置 `minimum_master_nodes`。

- 公式：`(N/2) + 1`，其中 N 是有资格成为 Master 的节点数
- 3 个 Master-eligible 节点：设置为 2
- 5 个 Master-eligible 节点：设置为 3

### Q3：Coordinating 节点会不会成为瓶颈？

**答案**：一般不会。Coordinating 节点不存储数据，只是转发请求。如果查询量极大，可以水平扩展 Coordinating 节点。

## 总结

ES 的节点角色设计体现了「**专精与协作**」的思想：

- **Master 节点**：集群的管理者
- **Data 节点**：数据的存储者
- **Coordinating 节点**：请求的调度者
- **Ingest 节点**：数据的预处理者

理解这些角色的职责和配合方式，是设计高可用 ES 集群的基础。
