# Elasticsearch 面试高频问题

Elasticsearch 是后端面试中的高频考点，尤其是中高级岗位。这篇文章汇总了常见面试题，并给出回答思路。

## 为什么要问 Elasticsearch？

面试官问 Elasticsearch，不是想让你背 API，而是想考察：

1. **分布式系统的理解**：分片、副本、故障转移等概念你是否掌握
2. **原理性思考**：你只是会用，还是理解背后的原理
3. **问题解决能力**：遇到性能问题、线上问题，你有没有排查思路
4. **工程经验**：你有没有实际运维过 ES 集群

顺着这个思路回答，比单纯背答案高出几个层次。

---

## 一、索引和数据结构

### ES 为什么要设计成主副分片？

面试官考察的是你对分布式系统的理解。

**表面答案**：主分片处理写入，副本分片提供容灾和查询能力。

**深层理解**：

分片是数据分布的基本单位。每个分片独立存储、独立查询，查询时并行执行，提高吞吐量。

主副分片的分离有几个关键意义：

**写入一致性**：写入只到主分片，然后同步到副本。如果允许任意分片接收写入，就需要分布式事务来保证一致性，成本太高。

**读取可扩展**：主分片和副本都能处理查询，可以水平扩展读取能力。

**故障恢复**：副本是主分片的镜像，任一分片故障都能快速恢复。

实际工作中，分片数规划要谨慎——主分片数在索引创建后不可修改，这是分布式系统 CAP 权衡的结果。

---

### 倒排索引的结构是什么？

这道题考察你对搜索底层原理的理解。

倒排索引由三部分组成：

**Term Dictionary（词字典）**：所有被索引的词，按字典序排列。

**Term Index（词索引）**：FST（有限状态转换器），用于快速定位词字典。

**Posting List（倒排列表）**：每个词关联的文档列表，包含 doc_id、词频、位置等信息。

查询时，先用 Term Index 快速定位 Term Dictionary，再用 Posting List 获取匹配文档。

---

### ES 和关系型数据库的概念对照

| 关系型数据库 | Elasticsearch |
|---|---|
| Database | Index |
| Table | Mapping |
| Row | Document |
| Column | Field |
| Primary Key | _id |
| Index | 倒排索引 |

但要注意，这种对照只是帮助理解，本质上有很大区别：

ES 的 Document 是自包含的，没有 JOIN；ES 的 Mapping 是 schema-less 的，可以动态扩展；ES 没有严格的事务支持。

## 二、搜索原理

### ES 查询流程是什么？

Query Then Fetch 是默认的搜索类型，分两个阶段：

**Query Phase（查询阶段）**：

1. 协调节点接收查询请求
2. 根据 routing 规则确定目标分片
3. 并发查询所有目标分片
4. 各分片返回匹配的 doc_id 和得分
5. 协调节点汇总、排序

**Fetch Phase（取回阶段）**：

1. 协调节点向各分片发起 fetch 请求
2. 各分片返回文档完整内容
3. 协调节点组装最终结果返回客户端

---

### 深度分页为什么慢？

假设查询 `from=9000, size=10`，有 5 个分片：

1. 每个分片需要返回 `9010` 条记录（from + size）
2. 协调节点需要处理 `5 × 9010 = 45050` 条记录
3. 然后排序取 Top 10

深度分页的本质问题是：**每个分片都要返回大量数据**，协调节点要做大量排序和筛选。

解决方案：使用 `search_after` 基于排序值翻页，或者限制最大页数。

---

### 什么是 search_after？解决了什么问题？

`search_after` 使用上一页的排序值作为起点，跳过已显示的数据：

```json
// 第一页
GET /_search?sort=timestamp,id

// 下一页
GET /_search?sort=timestamp,id&search_after=[1705123456,"doc_123"]
```

`search_after` 解决了深度分页的性能问题——不需要从每个分片获取大量数据，只需要从排序值起点继续扫描。

代价是：**不能随机跳页**，只能向后翻页。

---

### 全局排序问题是什么？

跨分片排序时，每个分片只知道自己的 Top N，不知道全局排名。

比如按时间排序查 Top 10：
- 分片 0 返回自己的 Top 10（时间最新的 10 条）
- 分片 1 返回自己的 Top 10
- ...

协调节点拿到 5 个分片的 Top 10 后合并，可能漏掉分片 1 排在第 11-20 的数据。

解决方案：使用 `search_type=dfs_query_then_fetch`，先收集全局词频再评分，但性能会下降。

## 三、性能优化

### 如何优化 ES 查询性能？

从多个维度回答：

**查询层面**：
- 使用 `filter` 替代 `query`（不计算得分）
- 避免 leading wildcard 查询（`"*keyword"`）
- `_source` 限制返回字段
- 使用 `routing` 减少搜索分片数

**索引层面**：
- 避免过多小分片
- 定期 `force_merge` 减少 Segment 数量
- 合理设置 `refresh_interval`

**集群层面**：
- 热数据用 SSD
- 分离冷热节点
- 监控堆内存使用率

---

### 写入性能如何优化？

**批量写入**：使用 `_bulk` API，单次批量 500-5000 条。

**调整刷新间隔**：写入期间临时设置 `refresh_interval: -1`。

**减少副本**：导入时设置 `number_of_replicas: 0`，完成后恢复。

**使用 routing**：减少单次写入的目标分片数。

---

### Segment Merging 是什么？有什么作用？

Segment 是分片内的数据存储单元，每个 Segment 是独立的倒排索引。

Segment Merging 将多个小 Segment 合并成大 Segment：

好处：减少 Segment 数量，降低查询时的文件句柄和内存开销；清理已删除文档。

触发条件：ES 定期自动执行，或手动触发 `forcemerge`。

---

## 四、高可用与一致性

### ES 集群故障转移机制？

节点故障检测 → 分片重新分配 → 副本重建：

1. Master 节点通过心跳检测节点存活
2. 不可用节点上的分片标记为 `UNASSIGNED`
3. Master 计算新的分片分配方案
4. 从主分片复制数据到新节点创建副本
5. 副本状态变为 `STARTED`

---

### 如何保证写入一致性？

`consistency` 参数控制写入一致性级别：

- `one`：主分片写入成功即可
- `quorum`：`半数以上分片`写入成功
- `all`：所有分片写入成功

`quorum = (primary + replica) / 2 + 1`

对于大多数场景，`quorum` 是性能和一致性的平衡点。

---

### 脑裂问题？如何避免？

脑裂：网络分区导致多个节点都认为自己是 Master。

避免方案：`minimum_master_nodes` 配置。

对于 N 个候选 Master 节点，设置 `minimum_master_nodes = N/2 + 1`。

例如 3 个候选 Master，设置 `minimum_master_nodes = 2`。这样即使发生网络分区，只有得到多数投票的节点才能成为 Master。

## 五、实战问题

### ES 内存溢出怎么排查？

1. 检查 JVM 堆内存使用率：`GET /_nodes/stats/jvm`
2. 检查 Segment 内存占用：`GET /_cat/segments?v`
3. 检查 Field Data Cache 使用
4. 分析慢查询，看是否有大聚合或深度分页

常见原因：
- 一次查询返回过多数据
- 聚合的 cardinality 过高
- mapping 中 `fielddata: true` 的字段基数过大

---

### 数据倾斜怎么解决？

数据倾斜：某些分片数据量远大于其他分片。

原因：
- 写入热点集中在某个值（如固定的用户 ID 作为 routing key）
- 分片数设置不合理

解决方案：
- 使用 composite routing 分散热点
- 重新规划分片数
- 使用 `reindex` 重建索引，重新分配分片

---

## 总结

ES 面试的核心在于：**原理 + 实践**。

只背 API 答案，面试官追问就露馅。要能讲清楚「为什么这样设计」「这样做有什么代价」「如果不这样做会怎样」。

建议准备面试时：
1. 亲手搭建集群，写一些 CRUD 和聚合查询
2. 用 Profile API 分析慢查询
3. 模拟节点故障，观察集群行为
4. 研究几个生产问题的排查过程

**最好的面试状态是：不仅能回答问题，还能反问面试官，展开更深层的讨论。**
