# Elasticsearch 核心概念

很多人第一次接触 Elasticsearch 时，会被它独特的术语系统搞得晕头转向。什么 Index 相当于表、Document 相当于行、Field 相当于列——这些类比虽然能帮你快速入门，但如果你止步于此，就会在实际使用中踩坑。

Index 真的等于表吗？Document 和行的区别在哪里？为什么 Elasticsearch 里有 Shard 和 Replica？这篇文章带你真正理解这些核心概念。

## Index、Document、Field

### Document（文档）

Document 是 Elasticsearch 中存储数据的基本单位。你可以把它理解为 JSON 对象：

```json
{
  "title": "Elasticsearch 入门指南",
  "content": "这是一篇关于 Elasticsearch 的教程...",
  "author": "张三",
  "published_at": "2024-01-15",
  "views": 1234,
  "tags": ["搜索", "Elasticsearch", "教程"]
}
```

这看起来和关系型数据库的一行数据非常像，但有一个关键区别：**Document 是自包含的**。每一篇文档都包含了它所有的字段信息，而不像关系型数据库那样需要通过 JOIN 才能拼凑完整数据。

这种设计带来了两个结果：

**优点**：查询时不需要 JOIN 操作，数据读取非常快。

**代价**：如果相同数据在不同文档中重复存储，会占用大量空间。比如「用户信息」被嵌入到「订单文档」中，每个订单都保存一份用户信息，一旦用户改了昵称，你需要更新所有相关文档。

### Field（字段）

Field 是 Document 中的键值对。上面的文档中，`title`、`content`、`author` 都是字段。

每个字段都有它的数据类型：

```json
{
  "title": {
    "type": "text",
    "analyzer": "standard"
  },
  "views": {
    "type": "integer"
  },
  "published_at": {
    "type": "date"
  }
}
```

Elasticsearch 支持丰富的数据类型，包括 `text`、`keyword`、`integer`、`long`、`float`、`double`、`boolean`、`date`、`geo_point`、`ip`、`nested`、`object` 等。

这里有一个重要区分：**`text` 类型会分词，`keyword` 类型不会分词**。

```json
{
  "title": "Elasticsearch 入门指南",
  "category_id": "tech-tutorials"
}
```

如果 `title` 定义为 `text` 类型，「Elasticsearch」「入门」「指南」这三个词会被分别索引，搜索「Elasticsearch」「入门」都能匹配到。但如果 `category_id` 定义为 `keyword`，它会被当作一个完整的字符串，「tech」「tutorials」分开搜索是找不到的。

### Index（索引）

Index 是文档的集合，类似于关系型数据库中的「数据库」或「表」。

```
一个 Index = 多个 Document + 这些 Document 的结构定义（Mapping）
```

```bash
# 创建索引
PUT /my-blog

# 查看索引信息
GET /my-blog

# 删除索引
DELETE /my-blog
```

通常，我们会把相同类型的文档放在同一个 Index 中。比如所有博客文章放在 `articles` 索引，所有用户评论放在 `comments` 索引。

但这不是绝对的。在生产环境中，有些人倾向于按时间创建索引：

```
logs-2024.01.01
logs-2024.01.02
logs-2024.01.03
```

这样设计的好处是：可以方便地删除旧数据（删整个索引比删索引中的部分文档快得多）、不同时间的数据可以有不同的副本策略（热数据多副本，冷数据少副本）。

## Shard（分片）与 Replica（副本）

### 为什么需要分片？

想象一下，你有一个包含 10 亿条文档的索引。查询一次需要遍历所有数据，即使 Elasticsearch 再快，也可能要几十秒。

但如果你有 5 个节点，把这 10 亿条文档分成 5 份，每个节点只处理 2 亿条，查询速度理论上能提升 5 倍。

这就是 Shard 的作用：**将数据分散存储，让查询可以并行执行**。

```json
PUT /my-index
{
  "settings": {
    "number_of_shards": 5
  }
}
```

分片数在索引创建时指定，**之后不能修改**。这是一个需要谨慎规划的决定——分片数太少，数据量大时查询会变慢；分片数太多，管理开销增加，且每个分片太小会产生大量小 Segment（后面会讲）。

### 什么是副本？

副本是分片的拷贝。当主分片所在节点宕机时，副本可以接管服务，保证数据不丢失。

```json
PUT /my-index
{
  "settings": {
    "number_of_shards": 3,
    "number_of_replicas": 2
  }
}
```

上面的配置创建了 3 个主分片、每个主分片 2 个副本，总共 9 个分片（3 × (1 + 2) = 9）。

```
节点1: P0, R1, R2
节点2: P1, R0, R2
节点3: P2, R0, R1
```

副本不仅用于容灾，还可以**处理查询请求**。当大量查询涌来时，副本可以分担负载，提高系统的吞吐量。

### 主分片与副本分片

| 特性 | 主分片 (Primary Shard) | 副本分片 (Replica Shard) |
|---|---|---|
| 写入操作 | 接收写入请求 | 不接收写入，只同步主分片数据 |
| 查询操作 | 可以处理 | 可以处理 |
| 故障转移 | 由 master 节点创建新的副本 | 可以提升为主分片 |
| 数量限制 | 创建后不可更改 | 可以动态调整 |

:::tip 分片数规划建议
小索引（百万级文档）设置 1-3 个分片；中等索引（千万级）设置 3-5 个分片；大索引（亿级以上）设置 5-10 个分片。每个分片的数据量建议控制在 50GB 以内。
:::

## 节点类型

一个 Elasticsearch 节点可以扮演不同的角色：

### Master 节点

负责**集群级别的管理操作**：创建或删除索引、决定分片分配给哪个节点、监控集群健康状态。

为了保证集群稳定性，通常会配置多个候选 Master 节点，通过选举产生一个真正的 Master。

```yaml
node.master: true
node.data: false
node.ingest: false
```

### Data 节点

负责**存储数据和处理查询请求**。这些节点会占用大量磁盘和内存，是集群的主要计算资源消耗者。

```yaml
node.master: false
node.data: true
node.ingest: false
```

### Coordinating 节点

负责**接收请求、转发给其他节点、汇总结果**。每个节点默认都扮演这个角色——收到请求的节点自然成为协调节点。

对于超大规模集群，可以通过设置专门的协调节点来分担负载：

```yaml
node.master: false
node.data: false
node.ingest: false
```

### Ingest 节点

负责**预处理文档**，比如提取字段、重命名字段、转换数据格式。在数据进入索引之前做一些 ETL 工作。

```yaml
node.master: false
node.data: false
node.ingest: true
```

### 节点角色组合

在实际部署中，一个节点通常同时扮演多个角色：

```yaml
# 默认配置，扮演所有角色
node.master: true
node.data: true
node.ingest: true
```

```
# 小型集群（3节点）
节点1: Master + Data
节点2: Master + Data
节点3: Master + Data

# 大型集群（10+节点）
节点1-2: Master（仅 Master，不存储数据）
节点3-7: Hot Data（处理写入和新查询）
节点8-10: Warm Data（处理历史查询）
```

## 总结

理解 Index、Document、Shard、Replica 这些概念，是掌握 Elasticsearch 的第一步。

Document 是数据的载体，Index 是文档的集合，Shard 实现了数据的分布式存储，Replica 提供了高可用和查询吞吐能力。节点的不同角色，则决定了集群如何协作完成各种任务。

这些概念不是孤立的——它们组合在一起，才构成了 Elasticsearch 强大的分布式搜索能力。
