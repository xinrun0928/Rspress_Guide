# Elasticsearch 核心概念：Index、Document、Shard、Replica

想象一下，你去图书馆找一本技术书。如果图书馆把所有书都堆在一个巨大的房间里，你可能要找一整天。但实际上，每本书都有一个索引页，告诉你某个关键词在哪一页。这就是**倒排索引**的思想——而 Elasticsearch，正是这个思想的完美实现。

## 1. Document：ES 中的数据单元

在 ES 中，**Document（文档）** 是存储数据的最小单元。你可以把它理解为数据库中的一行记录，但它远比「行」灵活得多。

```java
// 一个典型的 ES 文档，JSON 格式
{
  "title": "深入理解 Elasticsearch",
  "author": "张三",
  "content": "Elasticsearch 是一个基于 Lucene 的搜索服务器...",
  "tags": ["elasticsearch", "搜索引擎", "分布式"],
  "publish_date": "2024-01-15",
  "views": 10234
}
```

**Document 的特点：**

| 特性 | 说明 |
|-----|-----|
| 自包含 | 每个文档包含字段名和值 |
| 扁平结构 | 不支持嵌套对象（但可以用 nested 类型） |
| Schema-free | 可以存储任意结构的 JSON |
| 可索引 | 每个文档都有一个唯一 ID |

文档元数据包含 `_index`（索引名）、`_type`（类型，ES 7.x 后废弃）、`_id`（唯一标识）、`_version`（版本号，用于乐观锁）。

## 2. Index：文档的逻辑容器

**Index（索引）** 是文档的逻辑组织单位，相当于数据库中的「表」，但两者的设计哲学完全不同。

```
数据库 → Table（表）→ Row（行）
ES     → Index（索引）→ Document（文档）
```

但 ES 的 Index 比数据库表更强大：

- **倒排索引**：每个字段默认都会被索引
- **分片存储**：数据自动分散到多个分片
- **副本机制**：自动复制保证高可用

```java
// 创建索引时指定配置
{
  "settings": {
    "number_of_shards": 3,      // 主分片数
    "number_of_replicas": 1    // 副本数
  },
  "mappings": {
    "properties": {
      "title": { "type": "text" },
      "author": { "type": "keyword" },
      "views": { "type": "long" }
    }
  }
}
```

## 3. Shard：数据的物理分片

为什么 ES 能存储海量数据？因为它会**分片（Shard）**。

**Shard（分片）** 是 Index 的物理存储单元。每个 Index 可以被分成多个 Shard，每个 Shard 本质上是一个完整的 Lucene 索引。

```
Index（逻辑）─┬─ Shard 0（Lucene 索引）
             ├─ Shard 1（Lucene 索引）
             └─ Shard 2（Lucene 索引）
```

**为什么需要分片？**

1. **横向扩展**：数据量太大时，可以把分片分布到不同节点
2. **并行查询**：搜索可以同时在多个分片上执行
3. **负载均衡**：不同分片处理不同请求

主分片数在创建 Index 时指定，**创建后不可更改**。这是一个重要的设计决策：如果选太小，未来的扩展受限；如果选太大，每个分片占用资源过多。

> 面试追问：主分片数为什么不能改？
>
> 因为索引数据与分片数紧密绑定。如果允许修改，ES 需要重新分布所有数据，代价巨大。所以建议在创建索引前根据数据量预估。

## 4. Replica：数据的高可用保障

**Replica（副本）** 是主分片的完整副本。它的存在解决两个问题：

```
Primary Shard 0  ←→  Replica Shard 0
Primary Shard 1  ←→  Replica Shish 1
Primary Shard 2  ←→  Replica Shard 2
```

**副本的作用：**

1. **故障转移**：主分片所在节点挂了，副本自动升级为主分片
2. **提升读取性能**：搜索请求可以分发到副本，提高并发能力
3. **数据安全**：多副本确保数据不会因单点故障丢失

副本数可以动态调整，但也不是越多越好——每个副本都占用存储空间和内存。

## 6. 文档存储的秘密

当你向 ES 写入一条文档时，背后发生了什么？

```
写入请求 → Routing（计算分片） → Primary Shard 写入 → 副本复制 → 确认响应
```

**Routing 路由算法**：`shard_num = hash(_routing) % num_primary_shards`

默认使用文档 ID 作为路由值，所以同一文档总是被路由到同一个分片。

## 7. 字段的数据类型

ES 支持丰富的数据类型：

| 类型 | 说明 | 示例 |
|-----|-----|-----|
| text | 全文检索，会被分词 | `"内容"` |
| keyword | 精确值，不分词 | `"标签"` |
| long/integer | 整数类型 | `123` |
| double/float | 浮点数 | `3.14` |
| boolean | 布尔值 | `true` |
| date | 日期，支持格式 | `"2024-01-01"` |
| nested | 嵌套对象 | `[{...}, {...}]` |
| geo_point | 地理位置 | `{"lat": 39.9, "lon": 116.4}` |

> 你可能注意到 text 和 keyword 都能存储字符串，它们的区别是什么？
>
> text 会经过分词器处理，适合全文搜索；keyword 不分词，适合精确匹配和聚合。

## 总结与思考

ES 的核心概念之间的关系，可以用一句话概括：**Document 是数据，Index 是逻辑组织，Shard 是物理分片，Replica 是安全保障。**

理解这四个概念，是掌握 ES 的第一步。
