# ES 文档 CRUD 与并发控制

理解 Elasticsearch 的写入机制和并发控制，是避免数据丢失、解决冲突问题的前提。

这篇文章详细讲解 Document 的增删改查操作，以及在分布式环境下如何保证数据一致性。

## 文档的 ID

### 自动生成 ID

如果不在意文档 ID，可以让 Elasticsearch 自动生成：

```json
POST /my-index/_doc
{
  "title": "Hello Elasticsearch",
  "content": "This is my first document"
}
```

返回结果：

```json
{
  "_index": "my-index",
  "_type": "_doc",
  "_id": "ABC123",
  "_version": 1,
  "result": "created",
  "_shard": 2,
  "_seq_no": 5,
  "_primary_term": 1
}
```

`_id` 是 22 位 Base64 编码的 URL-safe 全局唯一标识，包含时间戳和随机数。

### 自定义 ID

如果业务上已经有唯一标识，可以用它作为文档 ID：

```json
PUT /my-index/_doc/user_123_article_456
{
  "title": "Hello Elasticsearch",
  "content": "This is my first document"
}
```

使用自定义 ID 的好处是可以实现**幂等写入**——同样的 ID 重复写入，不会创建重复文档，只会更新版本。

## Create（创建文档）

### 创建新文档

```json
PUT /my-index/_doc/1
{
  "title": "Elasticsearch 入门",
  "content": "Elasticsearch 是一个分布式搜索引擎"
}
```

如果 ID 已存在，返回 `409 Conflict` 错误：

```json
{
  "error": {
    "type": "version_conflict_engine_exception",
    "reason": "[1]: version conflict"
  }
}
```

### 幂等创建

`op_type` 参数可以强制幂等创建：

```json
PUT /my-index/_doc/1?op_type=create
{
  "title": "Elasticsearch 入门"
}
```

## Read（读取文档）

### 根据 ID 读取

```json
GET /my-index/_doc/1
```

返回结果：

```json
{
  "_index": "my-index",
  "_type": "_doc",
  "_id": "1",
  "_version": 3,
  "_seq_no": 10,
  "_primary_term": 1,
  "found": true,
  "_source": {
    "title": "Elasticsearch 入门",
    "content": "Elasticsearch 是一个分布式搜索引擎"
  }
}
```

返回的元信息包括：
- `_version`：文档版本号，每次更新递增
- `_seq_no`：序列号，文档在该分片上的操作序号
- `_primary_term`：主分片任期，和 `_seq_no` 配合用于乐观并发控制

### 字段过滤

```json
GET /my-index/_doc/1?_source=title,content

GET /my-index/_doc/1?_source=false
```

只返回指定字段，或者完全不返回 `_source`。

### 源文档不存在

```json
GET /my-index/_doc/999
```

```json
{
  "_index": "my-index",
  "_id": "999",
  "found": false
}
```

## Update（更新文档）

### 全量替换

Elasticsearch 没有真正的「更新」，只有「全量替换」：

```json
PUT /my-index/_doc/1
{
  "title": "Elasticsearch 进阶",
  "content": "进阶内容",
  "views": 100
}
```

上面操作会替换整个文档。如果只传了 `title` 和 `content`，原来的 `views` 字段会丢失。

### 部分更新

使用 `_update` API 进行部分更新：

```json
POST /my-index/_update/1
{
  "doc": {
    "views": 200,
    "updated_at": "2024-01-15"
  }
}
```

### 脚本更新

使用 Painless 脚本进行复杂更新：

```json
POST /my-index/_update/1
{
  "script": {
    "source": "ctx._source.views += params.count; ctx._source.updated_at = params.time",
    "params": {
      "count": 1,
      "time": "2024-01-15"
    }
  }
}
```

### 条件更新

```json
POST /my-index/_update/1
{
  "script": {
    "source": "if (ctx._source.views < params.threshold) { ctx._source.views++ } else { ctx.op = 'noop' }",
    "params": {
      "threshold": 100
    }
  }
}
```

当 `views` 已经达到阈值时，操作会被忽略（`noop`）。

## Delete（删除文档）

### 删除单个文档

```json
DELETE /my-index/_doc/1
```

删除同样是更新——Elasticsearch 会将文档标记为删除，实际删除发生在 Segment Merging 时。

### 批量操作

使用 `_bulk` API 进行批量操作：

```json
POST /_bulk
{ "index": { "_index": "my-index", "_id": "1" } }
{ "title": "Doc 1", "content": "Content 1" }
{ "index": { "_index": "my-index", "_id": "2" } }
{ "title": "Doc 2", "content": "Content 2" }
{ "update": { "_index": "my-index", "_id": "1" } }
{ "doc": { "views": 100 } }
{ "delete": { "_index": "my-index", "_id": "3" } }
```

Bulk 请求的格式是**操作行和数据行交替**。delete 操作不需要数据行。

批量操作的响应：

```json
{
  "took": 30,
  "errors": false,
  "items": [
    { "index": { "_id": "1", "status": 201 } },
    { "index": { "_id": "2", "status": 201 } },
    { "update": { "_id": "1", "status": 200 } },
    { "delete": { "_id": "3", "status": 200 } }
  ]
}
```

即使部分操作失败，`errors` 也可能为 `false`。需要检查每个 `items` 的 `status` 来判断具体操作是否成功。

## 乐观并发控制

在分布式环境下，并发更新是常见问题。两个请求同时读取同一文档，各自修改后写入，后写的会覆盖先写的——这就是**丢失更新**。

### 基于版本的并发控制

Elasticsearch 早期版本使用 `_version` 进行乐观锁：

```json
PUT /my-index/_doc/1?version=2
{
  "title": "Updated Title"
}
```

如果文档当前版本不是 2，更新会失败，返回 `409 Conflict`。

但 `_version` 有一个局限：**它只在单分片内严格递增**。跨分片查询时，版本号无法保证全局顺序。

### 基于序列号的并发控制

Elasticsearch 7.x 引入 `_seq_no` 和 `_primary_term`，提供更强大的并发控制：

- `_seq_no`：序列号，每个分片内单调递增
- `_primary_term`：主分片任期，当主分片变更时递增

```json
GET /my-index/_doc/1
```

```json
{
  "_seq_no": 10,
  "_primary_term": 1,
  "_source": { ... }
}
```

更新时携带序列号：

```json
PUT /my-index/_doc/1?if_seq_no=10&if_primary_term=1
{
  "title": "Updated Title"
}
```

如果序列号或任期不匹配（说明文档已被修改），返回 `409 Conflict`。

### 处理并发冲突

并发更新失败时，通常有三种处理策略：

**重试**：捕获异常后重新读取、修改、写入：

```java
public void updateWithRetry(String id, String newTitle) {
    int maxRetries = 3;
    for (int i = 0; i < maxRetries; i++) {
        try {
            GetResponse response = client.get(g -> g.index("my-index").id(id));
            Map<String, Object> doc = response.getSourceAsMap();
            doc.put("title", newTitle);
            client.index(i -> i.index("my-index").id(id).document(doc));
            return;
        } catch (Exception e) {
            if (i == maxRetries - 1) throw e;
        }
    }
}
```

**最后写入优先（LWW）**：通过 `version_type=external` 使用外部时间戳，简单但可能丢失更新。

**合并冲突**：读取所有版本，在应用层合并差异。适合冲突较少的场景。

## 读写一致性

### 读取偏好

默认情况下，查询会负载均衡到主分片和副本分片：

```json
GET /my-index/_search?preference=_replica
```

- `_primary`：只查主分片，保证最新但压力大
- `_replica`：只查副本分片
- `_local`：优先本地节点
- `_only_local`：只查本地节点

### 写入一致性

```json
PUT /my-index/_doc/1?consistency=quorum
{
  "title": "Hello"
}
```

- `one`：主分片写入成功即可
- `quorum`：半数以上分片写入成功
- `all`：所有分片写入成功

`quorum` 的计算公式：

```
quorum = (primary + replica) / 2 + 1
```

如果副本数为 0，`quorum = 1`，等价于 `one`。

:::warning 写入一致性的代价
`consistency=all` 提供最强一致性保证，但任何副本不可用都会导致写入失败。在高可用场景下，`quorum` 是更好的选择。
:::

## 总结

Elasticsearch 的文档操作看似简单，底层却有很多细节。

「全量替换」的更新方式容易让人困惑，理解这一点才能避免数据丢失。乐观并发控制是分布式写入的核心，`_seq_no` + `_primary_term` 提供了比 `_version` 更精确的版本控制。

在实际开发中，根据业务场景选择合适的一致性级别，在性能和数据安全之间找到平衡。
