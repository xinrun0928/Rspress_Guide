# 安全功能与多租户

当 Kibana 的用户从 1 个变成 10 个、100 个时，安全问题就变得至关重要。你不能让 A 部门的用户看到 B 部门的数据，也不能让普通用户拥有管理员权限。

Kibana 提供了完善的安全机制，包括 Spaces（空间隔离）、角色权限控制、视图级别安全。

## Spaces 空间隔离

Spaces 是 Kibana 最高级别的隔离机制。你可以把它理解为一个独立的 Kibana 工作区。

### 为什么需要 Spaces？

- **多团队共享**：一个 Kibana 实例供多个团队使用，每个团队有自己的空间
- **资源隔离**：不同空间有各自的 Dashboard、Visualization、索引模式
- **独立配置**：每个空间可以有不同的主题、首页设置

### 创建 Space

```
Stack Management → Spaces → Create space

名称: 生产环境监控
描述: 用于生产环境的日常监控
头像颜色: 红色
```

### Space 隔离了什么？

- Dashboard 和 Visualization
- 索引模式（Index Pattern）
- 告警规则
- 报告配置

### Space 之间共享什么？

- 用户和角色定义
- 通知渠道配置
- 访问控制规则

:::warning Space 不是安全边界
Spaces 只是逻辑隔离，不是安全隔离。如果用户知道索引名称，他们可能直接用 API 访问数据。要真正隔离数据，需要配合角色权限。
:::

## 角色与权限

### 内置角色

Kibana 提供了几个内置角色：

| 角色 | 说明 | 权限 |
|---|---|---|
| kibana_admin | Kibana 管理员 | 全部 Kibana 功能 |
| monitoring | 监控用户 | 查看监控数据 |
| viewing | 只读用户 | 查看 Dashboard |
| read_only | 只读 | 无法修改任何内容 |

### 自定义角色

根据最小权限原则，创建自定义角色：

```yaml
角色名称: 运维工程师-生产
权限:
  Kibana 功能:
    - Dashboard: Read, Create, Edit
    - Visualization: Read, Create, Edit
    - Discover: Read
  Elasticsearch 索引:
    - prod-*: Read, Run saved queries
    - prod-logs-*: Read, Run saved queries
    - prod-metrics-*: Read
  特性权限:
    - Dev Tools: Read
    - Console: Enabled
```

### 索引级别权限

这是最细粒度的权限控制：

```yaml
索引权限配置:
  索引模式: prod-orders-*
  权限:
    - Read: 读取文档
    - Create: 创建文档
    - Write: 更新/删除文档
    - Index: 创建/更新索引
    - Delete: 删除索引
  字段级安全:
    允许访问: [order_id, amount, status, timestamp]
    限制访问: [customer_phone, customer_idcard]  # 脱敏字段
```

## 视图级别安全（Field Level Security）

视图级别安全让你可以控制用户能看到哪些字段：

```yaml
字段级安全:
  允许列表模式:
    允许字段: [status, message, timestamp, service]
    
  排除列表模式:
    排除字段: [password, token, secret_key]
```

:::tip 性能影响
字段级安全会增加查询开销。如果性能敏感，考虑在 Elasticsearch 层面处理数据脱敏。
:::

## 文档级安全（Document Level Security）

文档级安全让你可以限制用户只能看到符合特定条件的文档：

```yaml
文档级安全:
  查询: { "term": { "team": "{{ user.teams[0] }}" } }
```

这样每个用户只能看到自己团队的数据。

## 用户认证

### 内置用户库

Kibana 内置了用户管理系统：

```
Stack Management → Users → Create user

用户名: zhangsan
邮箱: zhangsan@example.com
全名: 张三
密码: **********
角色: 运维工程师-生产
```

### 集成外部认证

#### LDAP 集成

```yaml
# elasticsearch.yml
xpack:
  security:
    authc:
      realms:
        ldap:
          ldap1:
            order: 1
            url: "ldap://ldap.example.com:389"
            bind_dn: "cn=admin,dc=example,dc=com"
            user_search:
              base_dn: "ou=users,dc=example,dc=com"
              filter: "(uid={0})"
            group_search:
              base_dn: "ou=groups,dc=example,dc=com"
```

#### SAML 单点登录

```yaml
# kibana.yml
xpack.security.authc.providers:
  saml:
    saml1:
      order: 1
      realm: saml1
      description: "SSO Login"
```

## API 密钥

### 创建 API 密钥

对于程序化访问，使用 API 密钥而不是用户密码：

```bash
curl -X POST "localhost:9200/_security/api_key" \
  -H 'Content-Type: application/json' \
  -u elastic:password \
  -d '{
    "name": "my-api-key",
    "role_descriptors": {
      "monitoring_role": {
        "cluster": ["monitor"],
        "index": [
          {
            "names": ["logs-*"],
            "privileges": ["read"]
          }
        ]
      }
    }
  }'
```

### 使用 API 密钥

```bash
curl -X GET "localhost:9200/logs-app/_search" \
  -H 'Authorization: ApiKey YOUR_API_KEY_HERE'
```

## 数据脱敏

### Pipeline 处理器

在 Elasticsearch 层面配置脱敏：

```json
{
  "description": "脱敏敏感字段",
  "processors": [
    {
      "gsub": {
        "field": "user.phone",
        "pattern": "(\\d{3})\\d{4}(\\d{4})",
        "replacement": "$1****$2"
      }
    },
    {
      "gsub": {
        "field": "user.idcard",
        "pattern": "(\\d{6})\\d{8}(\\d{4})",
        "replacement": "$1********$2"
      }
    }
  ]
}
```

### Kibana 敏感词过滤

```yaml
# kibana.yml
ops_logging.filter:
  sensitive_fields:
    - password
    - token
    - secret
    - api_key
  filter_type: mask
```

## 安全审计

### 审计日志

启用安全审计，记录所有操作：

```yaml
# elasticsearch.yml
xpack.security.audit.outputs:
  - type: file
    path: /var/log/elasticsearch/audit.log
  - type: index
    index: security-audit-*
```

### 审计事件类型

| 事件 | 说明 |
|---|---|
| authentication_success | 认证成功 |
| authentication_failed | 认证失败 |
| access_denied | 访问被拒绝 |
| connection_denied | 连接被拒绝 |
| index_created | 索引创建 |
| index_deleted | 索引删除 |
| document_read | 文档读取 |
| document_write | 文档写入 |

## 最佳实践

### 密码策略

```yaml
密码策略:
  最小长度: 12
  必须包含: 大写字母、小写字母、数字、特殊字符
  有效期: 90 天
  历史记录: 不得重复使用最近 5 次密码
```

### 会话管理

```yaml
会话配置:
  超时时间: 30 分钟
  最大并发: 3
  强制登出: 密码修改后
```

```
每月审查任务:
  1. 检查未使用的用户账号
  2. 检查权限过高的角色

  4. 更新过期的 API 密钥
  5. 测试告警通知是否正常
```

## 常见问题

### Q: 如何限制用户只能看到部分索引？

通过角色权限配置：

```yaml
角色配置:
  索引权限:
    - names: ["app-prod-*-readonly"]
      privileges: ["read"]
```

### Q: 不同 Space 可以有不同颜色吗？

可以。每个 Space 可以设置独立的颜色和头像，在 Space 切换器中显示。

### Q: API 密钥过期了怎么办？

定期轮换 API 密钥：

```
建议:
  - 90 天轮换一次
  - 密钥泄露后立即轮换
  - 不同环境使用不同密钥
```

安全无小事。完善的安全机制不仅保护你的数据，也是合规的要求。
