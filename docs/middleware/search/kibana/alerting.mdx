# 告警规则与监控

告警是监控系统的灵魂。再好的可视化，如果没有人看，就等于没有。告警机制让系统主动告诉你：出问题了。

Kibana 的告警功能（Alerting）可以监控你的数据，当条件满足时自动触发通知。

## Kibana 告警架构

### 告警生命周期

一个告警从创建到触发，经历以下阶段：

```
创建 → 启用 → 评估条件 → 触发 → 发送通知 → （恢复 → 发送恢复通知）
```

### 告警类型

Kibana 内置多种告警类型：

| 类型 | 说明 | 适用场景 |
|---|---|---|
| Threshold | 阈值告警 | 错误率 `` `>` `` 1% |
| Inventory | 资源告警 | CPU 使用率 `` `>` `` 80% |
| APM 告警 | 应用性能告警 | 延迟 p99 `` `>` `` 500ms |
| 日志告警 | 日志模式告警 | 出现特定错误关键字 |
| 监控告警 | 集群健康告警 | 节点离线 |

## 创建阈值告警

### 基本步骤

1. 进入 Stack Management → Rules and Connectors
2. 点击「Create rule」
3. 选择告警类型
4. 配置条件和通知方式
5. 保存并启用

### 配置示例

创建一个错误率告警：

```yaml
规则名称: 订单服务错误率告警
类型: Threshold
条件:
  - 索引: app-logs-*
  - 过滤器: service:"order-service" AND status >= 500
  - 聚合: Count
  - 阈值: > 10
  - 计算周期: 5 分钟
  - 触发间隔: 5 分钟
通知:
  - 渠道: 钉钉 Webhook
  - 内容: 错误数: {{ context.value }}，超过阈值
```

### 条件配置详解

**聚合方式**：

- `count`：文档数量
- `average`：平均值
- `sum`：总和
- `min/max`：最值
- `percentile`：百分位数

**阈值操作符**：

| 操作符 | 说明 |
|---|---|
| `>` | 大于 |
| `<` | 小于 |
| `>=` | 大于等于 |
| `<=` | 小于等于 |
| `==` | 等于 |
| `!=` | 不等于 |

**时间窗口**：

```yaml
触发条件: 5 分钟内平均值 > 100
评估周期: 每 1 分钟评估一次
```

这意味着：每 1 分钟查看一次过去 5 分钟的数据，如果平均值超过 100，触发告警。

## 告警表达式（ES|QL）

对于复杂告警条件，可以使用 ES|QL 语法：

```esql
FROM app-logs-*
| WHERE service == "order-service" AND status >= 500
| STATS error_count = COUNT()
| WHERE error_count > 10
```

这比传统的聚合查询更简洁、更强大。

## 告警叠包机制

当告警持续触发时，你会收到大量重复通知。Kibana 提供了叠包机制：

```yaml
行动组配置:
  重试次数: 3
  重试间隔: 5 分钟
  通知频率: 
    初始: 立即
    后续: 每 1 小时发送一次提醒
```

## 告警恢复

### 自动恢复

当条件不再满足时，告警会自动变为 recovered 状态：

```yaml
触发条件: error_rate > 0.01
恢复条件: error_rate < 0.005  # 增加恢复阈值，避免抖动
恢复延迟: 5 分钟  # 条件恢复后等待 5 分钟再发送恢复通知
```

:::tip 恢复阈值设计
不要让触发阈值和恢复阈值相同，否则会产生告警抖动。建议恢复阈值 `` `<` ` 触发阈值的 80%。
:::

### 恢复通知

```yaml
恢复通知模板:
  标题: "[已恢复] {{ rule.name }}"
  内容: |
    {{ rule.name }} 已恢复正常
    持续时间: {{ context.duration }}
    峰值: {{ context.peak_value }}
```

## 通知渠道

### 邮件通知

```yaml
邮箱配置:
  收件人: oncall@example.com
  抄送: manager@example.com
  内容格式: HTML
```

### Webhook

支持自定义 Webhook，可以接入任意通知系统：

```json
{
  "url": "https://oapi.dingtalk.com/robot/send",
  "method": "POST",
  "headers": {
    "Content-Type": "application/json"
  },
  "body": {
    "msgtype": "markdown",
    "markdown": {
      "title": "告警通知",
      "text": "### {{ rule.name }}\n\n{{ context.message }}"
    }
  }
}
```

### Slack

```yaml
Slack 配置:
  频道: #alerts-prod
  用户名: Kibana Alert Bot
  图标: 🔴
```

### PagerDuty

PagerDuty 是企业级告警平台，支持告警升级和值班调度：

```yaml

  集成密钥: xxxxx

```

## 告警管理最佳实践

### 分级管理

将告警分为不同级别：

| 级别 | 响应时间 | 通知方式 | 示例 |
|---|---|---|---|
| P1 紧急 | 5 分钟内 | 电话 + SMS | 服务不可用 |
| P2 严重 | 30 分钟内 | SMS + 邮件 | 错误率 `>` 5% |
| P3 警告 | 2 小时内 | 邮件 | 延迟升高 |
| P4 提醒 | 24 小时内 | 邮件 | 证书即将过期 |

### 告警疲劳规避

```yaml
# 避免告警风暴
告警合并策略:
  按服务分组: 同一服务多条告警合并为一条
  按时间聚合: 5 分钟内的告警合并
  
# 设置静默窗口
静默规则:
  - 名称: 计划内维护
    时间: 2024-01-15 02:00-04:00
    原因: 数据库维护
```

### 告警命名规范

```text
# 推荐格式
[P{级别}] [{环境}] [{服务}] {监控指标} {条件}

示例:
[P1] [生产] [订单服务] 错误率 > 1%
[P2] [生产] [支付服务] P99延迟 > 500ms
[P3] [预发] [所有服务] CPU使用率 > 80%
```

### 告警抑制链

设计告警时考虑父子关系：

```
根告警: 服务不可用
  ├── 子告警: 数据库连接失败
  ├── 子告警: 负载均衡器异常
  └── 子告警: 依赖服务超时
```

只有根告警触发通知，子告警作为上下文记录。

## 告警效果评估

### 核心指标

| 指标 | 理想值 | 说明 |
|---|---|---|
| 告警准确率 | `>=` 95% | 告警中有多少是真实的 |
| MTTD | `<=` 5 分钟 | 平均发现时间 |
| MTTR | `<=` 30 分钟 | 平均响应时间 |
| 告警噪音 | `<=` 20% | 无效告警占比 |

### 告警优化流程

```
1. 收集一周的告警数据
2. 标记每条告警：有效/无效
3. 分析无效告警原因
4. 调整阈值或添加过滤条件
5. 重复以上步骤
```

## 常见问题

### Q: 为什么告警触发了但没收到通知？

检查以下几项：

- 告警是否处于 Enabled 状态
- 通知渠道是否配置正确
- 告警规则的评估周期是否合理
- 是否被静默规则覆盖

### Q: 如何避免告警抖动？

- 使用合理的恢复阈值（不是同一阈值）
- 增加评估窗口，不要用瞬时值
- 使用移动平均等平滑算法

### Q: 可以看到告警历史吗？

可以。进入 Stack Management → Rules and Connectors → History 可以查看所有告警的触发历史。

告警系统是 DevOps 的最后一道防线。好的告警设计能够让你在问题影响用户之前就发现它，真正做到「防患于未然」。
