# 搜索引擎：让海量数据「秒级响应」

你的数据库里有 1000 万条商品记录。

用户搜索「小米手机」，数据库 LIKE 查询跑了 3 秒。用户说：太慢了，我不等了。

你加了索引，搜索时间降到了 500ms。但用户说：「小米」能搜到，「Xiaomi」搜不到；「手机壳」搜不到，「手机 保护套」搜不到。

数据库的精确匹配已经不够用了。**你需要的是搜索引擎。**

本模块涵盖 Elasticsearch、ELK 日志分析体系、Kibana 可视化，帮你构建完整的搜索与日志解决方案。

## 为什么需要搜索引擎？

### 数据库的局限

数据库 LIKE 查询的痛点：

```sql
-- 模糊查询：全表扫描
SELECT * FROM products WHERE name LIKE '%小米%';
-- 1000 万数据：3 秒 + 全表扫描

-- 索引失效
SELECT * FROM products WHERE name LIKE '%小米%';
-- 前缀通配符导致索引失效
```

**问题在哪里？**

1. **性能差**：LIKE 查询无法使用索引，必须全表扫描
2. **功能弱**：不支持分词、同义词、相关性排序
3. **扩展性差**：海量数据下性能急剧下降

### 搜索引擎的优势

Elasticsearch 是为「搜索」而生的：

```json
// ES 查询：智能分词 + 相关性排序
POST /products/_search
{
  "query": {
    "multi_match": {
      "query": "小米手机",
      "fields": ["name^3", "description", "category"],
      "type": "best_fields"
    }
  },
  "highlight": {
    "fields": {
      "name": {}
    }
  }
}

// 结果：
// 1. 小米手机 Pro 5G（相关性：0.95）
// 2. 小米 Redmi 手机（相关性：0.88）
// 3. 小米手机壳（相关性：0.72）
```

**ES 的优势：**

| 能力 | 数据库 | Elasticsearch |
|-----|--------|---------------|
| 全文检索 | LIKE 模糊查询 | 倒排索引，分词搜索 |
| 分词能力 | 不支持 | IK、ANSJ 等中文分词器 |
| 相关性 | 无 | TF-IDF、BM25 相关性算法 |
| 性能 | 海量数据下差 | PB 级数据秒级响应 |
| 聚合分析 | 简单聚合 | 强大聚合、统计分析 |

## 倒排索引：搜索引擎的核心

### 正排索引 vs 倒排索引

```
正排索引（数据库）：
文档 ID → 文档内容
1 → "小米手机性价比高"
2 → "华为手机拍照好"
3 → "小米手机颜色多样"

倒排索引（搜索引擎）：
词 → 文档 ID
小米 → [1, 3]
手机 → [1, 2, 3]
性价比 → [1]
高 → [1]
华为 → [2]
```

**倒排索引的优势：**

1. **查询快**：直接根据词找文档，不需要遍历
2. **支持分词**：把文档分成词，建立倒排表
3. **相关性排序**：根据词频、文档频率计算相关性

### 中文分词器选择

| 分词器 | 特点 | 适用场景 |
|--------|------|----------|
| IK | 最流行，词典分词 + 细粒度分词 | 通用场景 |
| Ansj | 基于 CRF 模型，较准确 | 学术、专业领域 |
| THULAC | 清华出品，速度快 | 高性能需求 |
| Jieba | 结巴分词，社区活跃 | 通用场景 |

```json
// IK 分词效果
POST /_analyze
{
  "analyzer": "ik_max_word",
  "text": "小米手机性价比高"
}

// 分词结果：
// 小米、手机、手机性价比、性价比、高
```

## 搜索架构设计

### 经典架构：数据同步 + 搜索服务

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   MySQL     │────▶│   Canal    │────▶│     MQ      │
│  (源数据)    │     │ (增量同步)  │     │ (消息队列)  │
└─────────────┘     └─────────────┘     └──────┬──────┘
                                               │
                                               ▼
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   用户      │◀────│  搜索服务    │◀────│     ES      │
│  (搜索请求)  │     │ (业务逻辑)   │     │  (搜索引擎) │
└─────────────┘     └─────────────┘     └─────────────┘
```

### 数据同步方案对比

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| Canal | 监听 MySQL binlog | 实时性强、无侵入 | 需要额外部署 |
| Logstash | JDBC 定时同步 | 配置简单 | 有延迟、不适合实时 |
| ES-Jdbc | 定时轮询数据库 | 简单 | 延迟高、数据量有限 |

## ELK：日志分析的黄金组合

### ELK Stack 是什么？

```
E ─── Elasticsearch：存储和搜索日志
L ─── Logstash：收集和处理日志
K ─── Kibana：可视化和分析
```

## 延伸思考

Elasticsearch 不是万能的。选择搜索引擎，需要考虑：

| 场景 | 推荐方案 |
|-----|---------|
| 海量数据搜索 | Elasticsearch |
| 简单关键词匹配 | 数据库 + 全文索引 |
| 日志分析、监控 | ELK Stack |
| 实时分析仪表盘 | Kibana + Elasticsearch |

**ES 的最佳实践：**

1. **数据量预估**：提前规划分片数，避免后期扩容
2. **冷热分离**：热数据用 SSD，冷数据用 HDD
3. **副本配置**：根据可用性需求配置副本数
4. **查询优化**：避免大结果集、深分页

**记住：Elasticsearch 是为搜索而生的，不要把它当成数据库来用。核心业务数据，还是要存在 MySQL 里。**
