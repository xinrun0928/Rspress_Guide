# ELK Stack 整体架构

一座大楼的稳固程度，取决于它的地基。ELK Stack 的性能与可靠性，同样取决于你的架构设计。

同样是用 ELK，为什么有的系统能处理每秒百万条日志，有的却频繁卡顿？答案往往不在单个组件，而在架构层面。

## 经典架构：三层架构

最常见的 ELK 架构分为三层：

```
┌─────────┐    ┌─────────┐    ┌─────────┐
│  采集层  │───▶│  处理层  │───▶│  存储层  │
│ Beats   │    │ Logstash│    │  ES集群  │
│         │    │  Kafka  │    │         │
└─────────┘    └─────────┘    └─────────┘
                                           │
                                           ▼
                                    ┌─────────┐
                                    │  展示层  │
                                    │  Kibana │
                                    └─────────┘
```

### 采集层

负责从各种数据源采集原始数据：

- **Beats 系列**：Filebeat、Metricbeat、Heartbeat、Packetbeat
- **应用直接写入**：通过 HTTP API 直接写入 Elasticsearch
- **其他采集器**：Fluentd、Logstash Input 插件

### 处理层

负责数据解析、转换、过滤、增强：

- **Logstash**：功能强大的数据处理管道
- **Kafka/RabbitMQ**：消息队列缓冲，应对流量峰值
- **Redis**：轻量级缓存，用于临时缓冲

### 存储层

负责数据存储、索引、查询：

- **Elasticsearch 集群**：水平扩展的分布式存储
- **冷热分离**：热数据用 SSD，温数据用 HDD，冷数据归档到对象存储

### 展示层

负责数据可视化与分析：

- **Kibana**：官方可视化平台
- **Grafana**：可接入 ES 数据源
- **自定义应用**：通过 ES API 构建

## 数据流详解

### 日志采集流程

```
应用产生日志
     ↓
Filebeat 读取日志文件
     ↓
Line 级别的处理（解码、添加元数据）
     ↓
发送至 Logstash 或直接到 Elasticsearch
     ↓
Logstash 解析（JSON 解析、Grok 正则、时间处理）
     ↓
富化（添加 IP 地理位置、关联其他数据源）
     ↓
输出到 Elasticsearch
     ↓
Kibana 展示
```

### 数据处理管道

```yaml
Logstash Pipeline:
  Input:
    beats:           # 接收 Filebeat 数据
      port: 5044
    kafka:            # 订阅 Kafka 消息
      topics: ["logs"]
      group_id: logstash
      
  Filter:
    json:             # JSON 解析
      source: "message"
      
    grok:             # 正则提取
      match: { "message": "%{TIMESTAMP:timestamp} %{LOGLEVEL:level} %{DATA:service} - %{GREEDYDATA:content}" }
      
    date:             # 时间解析
      match: ["timestamp", "yyyy-MM-dd HH:mm:ss"]
      target: "@timestamp"
      
    geoip:            # IP 地理信息
      source: "client_ip"
      
    useragent:        # 浏览器信息
      source: "user_agent"
      
  Output:
    elasticsearch:
      hosts: ["es-node1:9200", "es-node2:9200"]
      index: "app-logs-%{+YYYY.MM.dd}"
```

## 高可用架构

### Elasticsearch 高可用

```
┌─────────┐
│ Client  │
└────┬────┘
     │
┌────▼────┐
│ ES节点1 │◀──────┐
└────┬────┘       │  分片副本同步
┌────▼────┐       │
│ ES节点2 │──────┘
└────┬────┘
     │
┌────▼────┐
│ ES节点3 │
└─────────┘
```

Elasticsearch 通过分片和副本保证高可用：

- **主分片**：处理写入请求
- **副本分片**：提供读取能力，保证故障恢复
- **故障转移**：节点宕机后自动重新分配分片

### Logstash 高可用

```
                    ┌─────────────┐
                    │ Elasticsearch│
                    └─────────────┘
                          ▲
┌─────────┐  ┌─────────┐  │  ┌─────────┐
│ Filebeat│  │ Filebeat│  │  │ Logstash│
│  实例1  │  │  实例2  │──┼──│  实例1  │
└─────────┘  └─────────┘  │  └─────────┘
                          │  ┌─────────┐
                          └──│ Logstash│
                             │  实例2  │
                             └─────────┘
```

- **多 Logstash 实例**：负载均衡，提高处理能力
- **输入插件支持集群**：Beats 输入可以连接多个 Logstash
- **持久队列**：防止数据丢失

## 冷热分离架构

```
┌────────────────────────────────────────────────────┐
│                    Kibana                          │
└────────────────────────┬───────────────────────────┘
                         │
┌────────────────────────▼───────────────────────────┐
│              Elasticsearch 集群                     │
│  ┌─────────────┐    ┌─────────────┐    ┌─────────┐ │
│  │  热节点     │    │  温节点     │    │ 冷节点  │ │
│  │  (SSD)     │    │  (HDD)     │    │(归档存储)│ │
│  │ 最近7天    │    │ 8-30天     │    │ >30天   │ │
│  └─────────────┘    └─────────────┘    └─────────┘ │
└───────────────────────────────────────────────────┘
```

## 容量规划

### 存储容量估算

```yaml
单条日志大小: 500 字节
每日日志量: 1,000,000 条
日存储量: 500MB
副本因子: 1
日总存储: 1GB

30天热存储: 30GB
90天总存储: 90GB
考虑膨胀因子(1.2): 108GB
```

### 节点规划

| 数据量 | 热节点数 | 温节点数 | 冷节点数 | 总分片数 |
|---|---|---|---|---|
| `<=` 100GB | 3 | 0 | 0 | 3-6 |
| 100GB-1TB | 3-5 | 2 | 1 | 6-10 |
| 1TB-10TB | 5-10 | 3-5 | 2 | 10-20 |
| `>` 10TB | 10+ | 5+ | 3+ | 20+ |

:::tip 经验法则
- 每个分片大小建议控制在 30-50GB 以内
- 分片数量 = 节点数 × (1-3)
- 热节点使用 SSD，温/冷节点使用普通磁盘
:::

## 安全架构

```
┌─────────────────────────────────────────────────────────┐
│                     Kibana (浏览器)                      │
└────────────────────────┬────────────────────────────────┘
                         │ HTTPS
┌────────────────────────▼────────────────────────────────┐
│                   Nginx 反向代理                        │
│                   (认证 + SSL 终止)                     │
└────────────────────────┬────────────────────────────────┘
                         │ 内部网络
┌────────────────────────▼────────────────────────────────┐
│               Elasticsearch + Kibana                     │
│               (安全功能启用)                            │
└─────────────────────────────────────────────────────────┘
```

安全配置要点：

- **TLS 加密**：所有组件间通信加密
- **用户认证**：Elasticsearch 安全功能
- **角色权限**：按需分配最小权限
- **审计日志**：记录所有敏感操作

## 监控架构

```
┌─────────────────────────────────────────────────────────┐
│                    监控 Stack                           │
│  Prometheus ←──── Elasticsearch Exporter                │
│       │                                                   │
│       ▼                                                   │
│  Grafana                                                  │
└─────────────────────────────────────────────────────────┘
```

关键监控指标：

| 指标 | 说明 | 告警阈值 |
|---|---|---|
| 集群健康 | green/yellow/red | yellow 时告警 |
| CPU 使用率 | 节点 CPU | `>` 80% |
| 磁盘使用率 | 节点磁盘 | `>` 80% |
| JVM 堆使用 | ES JVM 堆 | `>` 75% |
| 搜索延迟 | p99 延迟 | `>` 500ms |
| 写入延迟 | 索引延迟 | `>` 1s |
| 分片分配 | 未分配分片数 | `>` 0 |

架构决定上限。只有合理的架构设计，才能支撑起稳定运行的 ELK 系统。
