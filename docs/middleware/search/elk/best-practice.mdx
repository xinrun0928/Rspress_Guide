# 日志收集最佳实践

前面的章节已经覆盖了 ELK Stack 的各个组件。但知道组件是什么还不够，更重要的是怎么把它们用好。

这一章，我们从实战出发，聊聊日志收集的最佳实践。

## 日志规范先行

好的日志分析，建立在规范的日志格式上。如果日志格式混乱，后面的解析、富化、可视化都会困难重重。

### 结构化日志

所有应用日志都应该是结构化的 JSON：

```json
{
  "@timestamp": "2024-01-15T10:30:00.123Z",
  "level": "ERROR",
  "service": "order-service",
  "version": "1.2.3",
  "env": "production",
  "trace_id": "abc123",
  "span_id": "def456",
  "message": "Order creation failed",
  "error": {
    "class": "PaymentException",
    "message": "Insufficient balance",
    "stack": "..."
  },
  "context": {
    "user_id": "u12345",
    "order_id": "o98765",
    "amount": 199.99
  }
}
```

### 必填字段

| 字段 | 类型 | 说明 | 示例 |
|---|---|---|---|
| @timestamp | ISO8601 | 时间戳 | 2024-01-15T10:30:00Z |
| level | string | 日志级别 | DEBUG/INFO/WARN/ERROR |
| service | string | 服务名称 | order-service |
| env | string | 环境 | dev/staging/prod |
| message | string | 日志消息 | Order created |
| trace_id | string | 链路追踪 ID | abc123 |

### 推荐字段

| 字段 | 类型 | 说明 | 示例 |
|---|---|---|---|
| version | string | 服务版本 | 1.2.3 |
| host | string | 主机名 | app-server-01 |
| pid | int | 进程 ID | 12345 |
| user_id | string | 用户 ID | u12345 |
| trace_id | string | 链路追踪 ID | abc123 |
| error | object | 错误信息 | {...} |

### 日志级别规范

```yaml
DEBUG: 调试信息，生产环境关闭
INFO:  正常流程日志
WARN:  警告信息，需要关注但不影响流程
ERROR: 错误信息，需要处理
```

:::tip 日志级别不是越多越好
很多团队的日志级别形同虚设——所有日志都是 INFO。明确的日志级别是告警规则的基础。
:::

## 采集配置最佳实践

### Filebeat 配置

```yaml title="filebeat.yml"
filebeat.inputs:
  - type: log
    enabled: true
    paths:
      - /var/log/app/*.log
    fields:
      service: myapp
      env: production
    multiline:
      pattern: '^\d{4}-\d{2}-\d{2}'
      negate: true
      match: after

# 处理器
processors:
  - add_host_metadata: ~
  - add_cloud_metadata: ~
  - add_docker_metadata: ~
  - drop_event:
      when:
        equals:
          level: debug  # 生产环境丢弃 debug 日志

output.logstash:
  hosts: ["logstash1:5044", "logstash2:5044"]
  loadbalance: true
```

### 敏感信息过滤

```yaml
processors:
  - redact:
      fields:
        - password
        - token
        - secret
        - credit_card
      pattern: '(?<=("|\'|=))[^"\'\s]+(?=("|\'))'
```

## 索引设计最佳实践

### 索引命名规范

```yaml
# 格式
{服务类型}-{环境}-{日期}
# 示例
app-logs-prod-2024.01.15
nginx-logs-prod-2024.01.15
system-metrics-prod-2024.01.15
```

### 分片配置

```yaml
# 索引模板
PUT /_template/app-logs
{
  "index_patterns": ["app-logs-*"],
  "settings": {
    "number_of_shards": 3,
    "number_of_replicas": 1,
    "refresh_interval": "5s"
  },
  "mappings": {
    "properties": {
      "@timestamp": { "type": "date" },
      "level": { "type": "keyword" },
      "service": { "type": "keyword" },
      "message": { "type": "text" }
    }
  }
}
```

### 冷热分离

```yaml
PUT /_ilm/policy/app-logs-policy
{
  "policy": {
    "phases": {
      "hot": {
        "actions": {
          "rollover": {
            "max_age": "7d",
            "max_size": "50gb"
          }
        }
      },
      "warm": {
        "min_age": "7d",
        "actions": {
          "shrink": { "number_of_shards": 1 },
          "forcemerge": { "max_num_segments": 1 }
        }
      },
      "delete": {
        "min_age": "30d",
        "actions": {
          "delete": {}
        }
      }
    }
  }
}
```

## 性能优化

### Logstash 管道优化

```ruby
# 管道配置
pipeline {
  workers: 4              # 工作线程数
  batch_size: 125         # 每批处理事件数
  batch_delay: 5          # 批处理延迟(ms)
}

filter {
  # 优化 grok 匹配
  grok {
    match => { "message" => "%{TIMESTAMP:timestamp} %{LOGLEVEL:level} %{DATA:service}" }
    break_on_match: false
  }
  
  # 减少 mutate 操作
  mutate {
    lowercase => ["level"]
  }
}
```

### Elasticsearch 写入优化

```yaml
output.elasticsearch:
  hosts: ["es1:9200", "es2:9200"]
  index: "app-logs-%{+YYYY.MM.dd}"
  
  # 写入优化
  bulk_size: 5000          # 批量大小
  max_retries: 3          # 最大重试次数
  
  # 模板管理
  template_name: "app-logs"
  template_overwrite: true
```

### 监控关键指标

| 指标 | 说明 | 目标 |
|---|---|---|
| Logstash 队列积压 | 待处理事件数 | < 1000 |
| Elasticsearch 写入延迟 | 写入到可查询的延迟 | < 5s |
| 消费者滞后 | Kafka 消费滞后 | < 10000 |
| 索引速率 | 每秒索引文档数 | 符合预期 |

## 可用性最佳实践

### 多副本部署

```yaml
# Elasticsearch
cluster:
  name: elk-cluster

nodes:
  - host: es-node-1
  - host: es-node-2
  - host: es-node-3

# 副本数
index:
  number_of_replicas: 2  # 至少 2 个副本
```

### 故障切换

```yaml
# Filebeat 降级配置
output.logstash:
  hosts: ["logstash1:5044", "logstash2:5044"]
  loadbalance: true

# Kafka 不可用时降级到 ES
output:
  - kafka:
      hosts: ["kafka:9092"]
      topic: "app-logs"
  - elasticsearch:
      hosts: ["es:9200"]
      when: "kafka unavailable"
```

## 运维最佳实践

### 容量规划

```yaml
# 日志量估算公式
每日日志量 = QPS × 平均日志大小 × 86400
存储需求 = 每日日志量 × 保留天数 × 副本系数 × 膨胀因子

# 示例
QPS: 10000
平均日志大小: 500B
保留天数: 30
副本系数: 1
膨胀因子: 1.2

每日日志量: 432GB
存储需求: 432GB × 30 × 1 × 1.2 = 15.5TB
```

### 定期维护任务

```bash
# 每天
- 检查集群健康状态
- 检查磁盘使用率
- 检查消费者滞后

# 每周
- 审查告警规则有效性
- 优化过大的索引
- 清理临时文件

# 每月
- 容量规划评审
- 安全审计
- 备份验证
```

## 常见问题排查

### 日志丢失

```yaml
# 检查步骤
1. Filebeat 是否正常运行？
   systemctl status filebeat
   
2. 日志文件权限是否正确？
   ls -la /var/log/app/
   
3. Logstash 连接是否正常？
   telnet logstash 5044
   
4. Elasticsearch 写入是否成功？
   curl -XGET "es:9200/_cat/indices/app-logs-*?v"
```

### 延迟过高

```yaml
# 延迟来源分析
1. Filebeat 读取延迟 → 检查文件读取配置
2. Logstash 处理延迟 → 检查管道配置，增加 worker
3. Kafka 消费延迟 → 检查消费者数量，增加分区
4. ES 写入延迟 → 检查集群负载，优化索引
```

### 存储爆满

```yaml
# 应急处理
1. 立即删除旧索引
   curl -XDELETE "es:9200/app-logs-2024.01.01"
   
2. 减少保留天数
3. 增加紧急告警

# 预防措施
1. 启用 ILM 自动管理
2. 设置磁盘使用率告警
3. 定期监控容量趋势
```

日志收集不是一次性工程，而是一个持续优化的过程。从规范制定到落地执行，从监控告警到容量规划，每个环节都需要用心。
