# Filebeat 轻量级日志采集

当你需要在几十台服务器上收集日志时，你会选择什么方案？

Logstash 功能强大，但它的资源消耗也让人望而却步——每台服务器都跑一个 Logstash 实例，光 JVM 就得好几个 G。

Filebeat 解决了这个问题。它是 Beats 家族中最常用的成员，专为日志文件而生。

## Filebeat 是什么？

Filebeat 是 Elastic 公司出品的轻量级日志采集器。它的工作原理很简单：监控指定的日志文件，读取新增的内容，发送到 Logstash 或 Elasticsearch。

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  日志文件   │────▶│   Filebeat  │────▶│  Logstash   │
│  app.log    │     │  (轻量级)   │     │  (可选)     │
└─────────────┘     └─────────────┘     └──────┬──────┘
                                               │
                                               ▼
                                        ┌─────────────┐
                                        │Elasticsearch│
                                        └─────────────┘
```

为什么轻量？

- 没有 JVM，只有一个二进制文件
- 资源消耗极低（通常 1-2% CPU，几十 MB 内存）
- 部署简单，一条命令就能启动

## 工作原理

### 文件监控机制

Filebeat 使用「文件句柄 + 偏移量」的机制：

1. **打开文件**：记录文件 inode 和起始偏移量
2. **逐行读取**：从偏移量位置开始读取新内容
3. **更新偏移量**：每读完一行，更新偏移量
4. **持久化状态**：定期将读取位置写入磁盘

```yaml
# Filebeat 内部状态
---
file_id: /var/log/app.log
offset: 102400
inode: 123456
```

当 Filebeat 重启时，会从上次记录的位置继续读取，不会丢失数据。

### Harvester 收割机制

每个被监控的文件对应一个 Harvester：

```
┌──────────────────────────────────────────────┐
│              Filebeat 进程                    │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐       │
│  │Harvester│  │Harvester│  │Harvester│       │
│  │ app.log │  │nginx.log│  │ mysql.log│      │
│  └────┬────┘  └────┬────┘  └────┬────┘       │
│       │            │            │             │
│       └────────────┼────────────┘             │
│                    │                         │
│              ┌─────▼─────┐                   │
│              │   Queue   │                   │
│              │  (发送队列) │                   │
│              └─────┬─────┘                   │
└────────────────────┼─────────────────────────┘
                     │
              ┌──────▼──────┐
              │  Logstash   │
              │Elasticsearch│
              └─────────────┘
```

### 发送队列

Harvester 读取的数据放入发送队列，队列会将数据批量发送到下游：

```yaml
# 发送队列配置
queue:
  type: memory          # 内存队列
  # spool: 磁盘队列（更安全）
  batch_size: 15625     # 每批发送的 event 数量
  flush_timeout: 5s      # 超时刷新时间
```

:::warning 内存队列的限制
如果 Filebeat 在数据发送前崩溃，内存队列中的数据会丢失。使用 spool 可以将数据写入磁盘，但会牺牲一些性能。
:::

## 安装与配置

### 安装

```bash
# Debian/Ubuntu
curl -L -O https://artifacts.elastic.co/downloads/beats/filebeat/filebeat-8.11.0-amd64.deb
sudo dpkg -i filebeat-8.11.0-amd64.deb

# Docker
docker run -d \
  --name=filebeat \
  -v /path/to/logs:/logs:ro \
  -v /path/to/filebeat.yml:/etc/filebeat/filebeat.yml:ro \
  elastic/filebeat:8.11.0
```

### 基本配置

```yaml title="filebeat.yml"
filebeat.inputs:
  - type: log
    enabled: true
    paths:
      - /var/log/app/*.log
    fields:
      service: app
      env: production
    multiline:
      pattern: '^\d{4}-\d{2}-\d{2}'
      negate: true
      match: after

output.logstash:
  hosts: ["logstash1:5044", "logstash2:5044"]
  
processors:
  - add_host_metadata:
      when.not.contains.tags: forwarded
  - add_cloud_metadata: {}
  - add_docker_metadata: {}
```

## 常用配置详解

### 多日志源配置

```yaml
filebeat.inputs:
  # 应用日志
  - type: log
    enabled: true
    paths:
      - /var/log/app/*.log
    fields:
      type: application
      service: myapp
    fields_under_root: true  # 字段放到根级别
  
  # Nginx 访问日志
  - type: log
    enabled: true
    paths:
      - /var/log/nginx/access.log
    json.keys_under_root: true
    json.add_error_key: true
    fields:
      type: nginx_access
    fields_under_root: true
  
  # 系统日志
  - type: log
    enabled: true
    paths:
      - /var/log/syslog
      - /var/log/messages
    fields:
      type: system
```

### JSON 日志解析

```yaml
- type: log
  paths:
    - /var/log/app/*.log
  json.keys_under_root: true       # JSON 字段放到根级别
  json.overwrite_keys: true        # 覆盖默认字段
  json.add_error_key: true         # 添加解析错误字段
  json.remove_char:                # 移除特殊字符
    keystroke: ["\\n", "\\r"]
```

### 多行日志处理

Java StackTrace 是典型需要多行处理的场景：

```yaml
multiline:
  type: pattern
  pattern: '^\d{4}-\d{2}-\d{2}'      # 以日期开头是新日志
  negate: true                       # 不匹配的是多行内容
  match: after                        # 追加到上一行后面
  max_lines: 100                     # 最大合并行数
  timeout: 5s                        # 超时后强制结束
```

### 日志内容过滤

使用 processors 过滤不想发送的日志：

```yaml
processors:
  # 丢弃 DEBUG 日志
  - drop_event:
      when:
        equals:
          level: debug
  
  # 丢弃健康检查日志
  - drop_event:
      when:
        or:
          - contains:
              path: /health.log
          - contains:
              message: "health check"
```

## 日志字段规范

为了让日志在 Elasticsearch 中更好地被分析，建议统一日志格式：

### 推荐的 JSON 日志格式

```json
{
  "@timestamp": "2024-01-15T10:30:00.000Z",
  "level": "ERROR",
  "service": "order-service",
  "version": "1.2.3",
  "env": "production",
  "trace_id": "abc123",
  "span_id": "def456",
  "message": "Order creation failed",
  "error": {
    "class": "PaymentException",
    "message": "Insufficient balance",
    "stack": "..."
  },
  "context": {
    "user_id": "u12345",
    "order_id": "o98765"
  }
}
```

### Filebeat 添加的元数据

```yaml
fields:
  cluster: production
  datacenter: dc1

fields_under_root: false
```

Filebeat 会自动添加：

| 字段 | 说明 |
|---|---|
| @timestamp | 时间戳 |
| log.offset | 文件偏移量 |
| log.file.path | 文件路径 |
| log.flags | 日志标志（如 multiline） |
| prospector.type | 输入类型 |

## 高可用配置

### 负载均衡

```yaml
output.logstash:
  hosts: ["logstash1:5044", "logstash2:5044", "logstash3:5044"]
  loadbalance: true    # 启用负载均衡
  bulk_max_size: 2048
```

### 故障转移

```yaml
output.logstash:
  hosts: ["logstash1:5044", "logstash2:5044"]
  loadbalance: false  # 禁用负载均衡
  timeout: 30s        # 连接超时
  backoff.init: 1s   # 初始重试间隔
  backoff.max: 60s   # 最大重试间隔
```

### 直接输出到 ES

小规模部署可以跳过 Logstash：

```yaml
output.elasticsearch:
  hosts: ["es-node1:9200", "es-node2:9200"]
  index: "app-logs-%{+yyyy.MM.dd}"
  username: "filebeat"
  password: "${FILEBEAT_PASSWORD}"
```

## 模块化配置

Filebeat 提供了大量预配置的模块，一键启用：

```bash
# 启用 Nginx 模块
filebeat modules enable nginx

# 启用系统模块
filebeat modules enable system
```

```yaml title="filebeat.yml"
filebeat.config.modules:
  path: ${path.config}/modules.d/*.yml
  reload.enabled: true
  reload.period: 10s
```

模块会自动配置：

- 输入源（文件路径）
- 解析规则（Pipeline）
- 默认字段

支持的模块：

| 模块 | 说明 |
|---|---|
| nginx | Nginx 访问和错误日志 |
| apache | Apache 访问和错误日志 |
| mysql | MySQL 慢查询和错误日志 |
| redis | Redis 日志 |
| system | 系统日志（syslog、auth） |
| docker | Docker 日志 |
| kubernetes | Kubernetes 容器日志 |

## 性能调优

### 系统级调优

```bash
# 增加文件描述符限制
echo "* soft nofile 65536" >> /etc/security/limits.conf
echo "* hard nofile 65536" >> /etc/security/limits.conf

# 增加 inotify 监听数量
echo 524288 > /proc/sys/fs/inotify/max_user_watches
```

### Filebeat 配置调优

```yaml
# 限制 Harvester 数量
filebeat.inputs:
  close.inactive: 5m      # 文件不活跃 5 分钟后关闭
  harvester.buffer.size: 16384  # 读取缓冲区大小

# 调整发送批处理
output.logstash:
  bulk_max_size: 1024     # 调整批处理大小
  worker: 2              # 每个主机的工作线程数
```

### 监控 Filebeat

```yaml
# 启用监控
monitoring.enabled: true
monitoring.elasticsearch:
  hosts: ["es-mon:9200"]
```

通过 Kibana 的 Stack Monitoring 可以看到 Filebeat 的状态。

## 常见问题

### Q: 日志被轮转后，Filebeat 会怎么处理？

Filebeat 通过 inode 识别文件。当日志被轮转（mv 或 rm + create）时，新文件会被当作新文件处理。推荐使用 `copytruncate` 方式轮转日志。

### Q: 如何处理日志量突增的情况？

Filebeat 的内存队列会缓冲数据，但如果持续超过处理能力，最终会背压。建议配合 Logstash 或 Kafka 使用。

### Q: 可以同时监控多个目录吗？

可以。每个 `paths` 数组可以包含多个路径，支持通配符：

```yaml
paths:
  - /var/log/app1/*.log
  - /var/log/app2/*.log
  - /opt/app3/logs/**/*.log  # ** 匹配递归目录
```

Filebeat 是日志收集的第一公里。选对它、用好它，后面的 ELK 组件才能发挥最大价值。
