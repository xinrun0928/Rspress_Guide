# Logstash 多行处理

日志文件中有一类特殊的数据：多行日志。比如 Java 的 StackTrace、Python 的 Traceback、XML/JSON 块等，这些数据跨越多行，必须作为一个整体处理。

Logstash 提供了多行处理能力，专门解决这类问题。

## 常见的跨行日志

### Java StackTrace

```
Exception in thread "main" java.lang.NullPointerException
    at com.example.MyClass.method(MyClass.java:10)
    at com.example.Main.main(Main.java:5)
```

### Python Traceback

```
Traceback (most recent call last):
  File "app.py", line 10, in main
    result = divide(1, 0)
  File "app.py", line 5, in divide
    return a / b
ZeroDivisionError: division by zero
```

### 时间戳跨行

```
2024-01-15 10:30:00 INFO Starting application...
2024-01-15 10:30:01 DEBUG Processing request id=123
2024-01-15 10:30:02 ERROR
  Request failed: connection timeout
  Stack: ...
```

## multiline codec（已废弃）

早期的 Logstash 版本使用 `multiline` codec，但它有很多问题：

- 废弃通知已在多个版本出现
- 性能较差
- 配置复杂，容易出错

**不要再使用 `multiline` codec**：

```ruby
# 不要这样用
input {
  file {
    path => "/var/log/app.log"
    codec => multiline {
      pattern => "^\s"
      what => "previous"
    }
  }
}
```

## multiline filter

现在推荐使用 `multiline` filter 来处理多行日志：

```ruby
filter {
  multiline {
    pattern => "^Exception|^\s+at |^Caused by:"
    negate => true
    what => "previous"
    max_lines => 500
    max_bytes => "1MB"
  }
}
```

### 参数说明

| 参数 | 说明 |
|---|---|
| `pattern` | 正则表达式，匹配行首 |
| `negate` | `true` 或 `false`，决定匹配或不匹配的行为 |
| `what` | `previous` 或 `next`，将当前行合并到前一行还是后一行 |
| `max_lines` | 最大合并行数，防止内存溢出 |
| `max_bytes` | 最大合并字节数 |

### negate 的作用

这是最让人困惑的参数：

- `negate => true`：当前行**不匹配** pattern 时，合并到前一行
- `negate => false`：当前行**匹配** pattern 时，合并到前一行

### 配置示例

#### Java StackTrace

```ruby
filter {
  multiline {
    pattern => "^\tat "
    negate => true
    what => "previous"
    max_lines => 1000
  }
}
```

- `^\tat `：匹配 Java 堆栈的行首（Tab 后跟 at）
- `negate => true`：不匹配的行（通常是异常信息的第一行）
- `what => "previous"`：不匹配的行与前一行合并

#### 错误消息块

```ruby
filter {
  multiline {
    pattern => "^%{TIMESTAMP_ISO8601}"
    negate => true
    what => "previous"
  }
}
```

将不匹配时间戳开头的行，合并到前一行。

#### XML/JSON 块

```ruby
filter {
  multiline {
    pattern => "^<|^{\"|^\}"
    negate => true
    what => "previous"
    max_bytes => "10MB"
  }
}
```

## Codec vs Filter

| 特性 | multiline codec | multiline filter |
|---|---|---|
| 位置 | Input 阶段 | Filter 阶段 |
| 时机 | 数据刚读入 | 其他 Filter 之后 |
| 灵活性 | 低 | 高 |
| 性能 | 差 | 好 |
| 状态管理 | 每个文件独立 | Event 级别 |

**推荐使用 `multiline` filter**，因为它更灵活，性能更好。

## 多行处理完整示例

### Java 应用日志

```ruby
input {
  file {
    path => "/var/log/myapp/application.log"
    start_position => "beginning"
    sincedb_path => "/dev/null"
  }
}

filter {
  # 解析普通日志行
  grok {
    match => {
      "message" => "%{TIMESTAMP_ISO8601:timestamp} %{LOGLEVEL:level} %{NOTSPACE:thread} %{JAVACLASS:class} - %{GREEDYDATA:content}"
    }
    tag_on_failure => ["_grokparsefailure"]
  }

  # 处理多行堆栈
  if "_grokparsefailure" in [tags] {
    multiline {
      pattern => "^\tat |^Caused by:|^\.\.\. %{NUMBER} more"
      negate => false
      what => "previous"
      max_lines => 500
    }
  }

  # 解析时间
  date {
    match => ["timestamp", "ISO8601"]
    target => "@timestamp"
  }
}

output {
  elasticsearch {
    hosts => ["http://localhost:9200"]
    index => "myapp-logs-%{+YYYY.MM.dd}"
  }
}
```

### Nginx 错误日志

Nginx 错误日志格式：

```
2024/01/15 10:30:00 [error] 12345#0: *1 connect() failed (111: Connection refused) while connecting to upstream
```

有时候错误信息会跨行：

```
2024/01/15 10:30:00 [error] 12345#0: *1 upstream timed out (110: Connection timed out) while reading response header from upstream
```

```ruby
filter {
  multiline {
    pattern => "^\d{4}/\d{2}/\d{2}"
    negate => true
    what => "previous"
  }

  grok {
    match => {
      "message" => "%{TEMP:timestamp} \[%{LOGLEVEL:level}\] %{NUMBER:pid:int}#%{NUMBER:thread_id}: %{GREEDYDATA:content}"
    }
  }
}
```

## 常见问题

### 多行合并不生效

检查 `negate` 和 `what` 的组合是否正确：

```ruby
# 将不以指定模式开头的行合并到前一行
pattern => "^ERROR"
negate => true
what => "previous"

# 正确理解：所有不以 ERROR 开头的行，都是 ERROR 行的一部分
```

### 内存溢出

`max_lines` 和 `max_bytes` 防止恶意日志耗尽内存：

```ruby
multiline {
  pattern => "^..."
  negate => true
  what => "previous"
  max_lines => 1000       # 超过 1000 行强制分割
  max_bytes => "10MB"     # 超过 10MB 强制分割
}
```

### 性能问题

多行处理会显著影响性能，因为需要缓存多行数据。优化建议：

- 精确匹配模式，减少误判
- 设置合理的 `max_lines`
- 考虑在 Input 阶段预处理

## 总结

多行日志处理是日志收集中常见的问题。

使用 `multiline` filter 而不是已废弃的 `multiline` codec。理解 `negate` 参数是关键——它决定匹配模式的逻辑方向。

合理设置 `max_lines` 和 `max_bytes`，防止恶意日志导致内存问题。
