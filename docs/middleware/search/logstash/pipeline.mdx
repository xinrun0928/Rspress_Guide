# Logstash 工作流程

Logstash 的核心是 Pipeline 模型：数据从 Input 进入，经过 Filter 转换，由 Output 发出。这三部分组合在一起，构成了一个完整的数据处理流水线。

理解 Pipeline 模型，是掌握 Logstash 的第一步。

## Pipeline 模型

```
┌────────────┐     ┌────────────┐     ┌────────────┐
│   Input    │ ──▶ │   Filter  │ ──▶ │   Output   │
│   (输入)    │     │  (过滤)    │     │   (输出)    │
└────────────┘     └────────────┘     └────────────┘
       │                 │                 │
       ▼                 ▼                 ▼
   读取数据源          转换数据           发送数据
```

### 简单示例

```ruby
input {
  file {
    path => "/var/log/syslog"
    start_position => "beginning"
  }
}

filter {
  grok {
    match => { "message" => "%{SYSLOGTIMESTAMP:timestamp} %{WORD:program}\[%{PID:pid}\]: %{GREEDYDATA:content}" }
  }
}

output {
  elasticsearch {
    hosts => ["http://localhost:9200"]
    index => "syslog-%{+YYYY.MM.dd}"
  }
}
```

这段配置定义了一个 Pipeline：
- 从 `/var/log/syslog` 文件读取数据
- 用 Grok 解析日志格式
- 将结果写入 Elasticsearch

## Event 对象

Logstash 处理的数据单元叫 Event，结构类似 JSON：

```json
{
  "@timestamp": "2024-01-15T10:30:00.000Z",
  "@version": "1",
  "host": "web-server-01",
  "message": "User login successful",
  "user_id": 12345,
  "action": "login",
  "status": "success"
}
```

Event 有几个特殊字段：

| 字段 | 说明 |
|---|---|
| `@timestamp` | 事件发生时间，默认为当前时间 |
| `@version` | Event 版本号 |
| `@version` | 日志来源等元信息 |

其他字段是我们自定义的，可以任意命名。

### 字段引用

在配置文件中，用 `[field_name]` 引用字段：

```ruby
filter {
  mutate {
    add_field => { "full_message" => "User %{[user_id]} performed %{[action]}" }
  }
}
```

### 字段操作符

```ruby
# 字段存在判断

# 字段比较
if [status] == "success" {

# 正则匹配
if [message] =~ /error/i {

# 正则不匹配
if [message] !~ /timeout/ {
```

## 队列模型

Logstash 内部使用队列在 Pipeline 各阶段之间传递 Event。

### 内存队列

默认使用内存队列：

```ruby
pipeline.ordered => auto  # auto, true, false
```

- `auto`（默认）：正常情况下禁用排序以提高性能，出现问题时启用
- `true`：启用排序，保证 Event 顺序，牺牲性能
- `false`：禁用排序，禁用排序，性能最佳

### 持久队列

对于需要保证数据不丢失的场景，使用持久队列：

```ruby
queue.type => persisted
queue.page_capacity => 250mb
queue.max_bytes => 1gb
queue.checkpoint.acks => 1024
queue.checkpoint.writes => 1024
queue.checkpoint.interval => 1000
```

持久队列将数据写入磁盘，即使 Logstash 重启也不会丢失。

:::warning 持久队列的代价
持久队列会显著增加写入延迟，因为需要写入磁盘。建议只在数据可靠性要求高的场景使用。
:::

## 多 Pipeline

一个 Logstash 实例可以运行多个 Pipeline：

```ruby
# pipelines.yml
- pipeline.id: main
  path.config: "/etc/logstash/conf.d/main/*.conf"
  pipeline.workers: 4

- pipeline.id: audit
  path.config: "/etc/logstash/conf.d/audit/*.conf"
  pipeline.workers: 2
```

多 Pipeline 的好处：

- 隔离不同数据源的处理逻辑
- 为不同 Pipeline 分配不同资源
- 独立管理和监控

## Pipeline 编排

### Pipeline to Pipeline

Logstash 支持 Pipeline 之间的数据传递：

```ruby
# 发送端
output {
  pipeline {
    send_to => "my_pipeline"
  }
}

# 接收端
input {
  pipeline {
    address => "my_pipeline"
  }
}
```

### 条件分支

根据条件将数据发送到不同的目标：

```ruby
output {
  if [log_type] == "access" {
    elasticsearch { ... }
  } else if [log_type] == "error" {
    elasticsearch { ... }
    email { ... }
  } else {
    stdout { codec => rubydebug }
  }
}
```

## 总结

Pipeline 是 Logstash 的核心抽象。

Input 负责数据采集，Filter 负责数据转换，Output 负责数据发送。三者组合构成了灵活的数据处理流水线。

理解 Event 对象的结构，学会引用和操作字段，是编写复杂配置的基础。

多 Pipeline 和 Pipeline to Pipeline 编排能力，让 Logstash 可以应对复杂的企业级数据集成场景。
