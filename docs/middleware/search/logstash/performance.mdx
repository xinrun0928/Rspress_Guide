# Logstash 性能优化

Logstash 的性能，直接影响整个数据管道的吞吐量。当数据量从每秒几千条增长到几十万条时，性能优化就变得至关重要。

这篇文章从线程模型、队列、Filter 顺序、监控等多个维度讲解 Logstash 的性能调优。

## 线程模型

Logstash 使用 Pipeline Workers 处理数据。每个 Worker 是一个独立的线程，并行执行 Filter 和 Output。

### 核心参数

```ruby
pipeline {
  workers => 4          # Worker 线程数
  batch_size => 125    # 每个 batch 的事件数
  batch_delay => 5     # 等待 batch 满的时间（毫秒）
}
```

### workers 设置

`workers` 决定并行处理的能力：

- **CPU 密集型**：设置为 CPU 核心数
- **I/O 密集型**：可以设置为 CPU 核心数的 2-4 倍

```bash
# 查看 CPU 核心数
nproc
# 或
sysctl -n hw.ncpu
```

### batch_size 设置

`batch_size` 控制每次处理的 Event 数量：

- **大 batch**：减少系统调用开销，提高吞吐，但增加延迟和内存占用
- **小 batch**：降低延迟，但增加开销

对于日志收集场景，`batch_size: 125` 到 `batch_size: 1000` 是合理的范围。

### 实际配置

```ruby
pipeline {
  workers => 8
  batch_size => 500
  batch_delay => 10
}
```

## 队列优化

### 队列类型

```ruby
queue.type => memory  # 内存队列（默认）
# 或
queue.type => persisted  # 持久队列
```

| 类型 | 优点 | 缺点 |
|---|---|---|
| `memory` | 低延迟 | 崩溃会丢数据 |
| `persisted` | 数据持久化 | 延迟增加 |

### 持久队列配置

```ruby
queue.type => persisted
queue.page_capacity => 250mb    # 每个页面大小
queue.max_bytes => 1gb           # 队列最大容量
queue.checkpoint.acks => 1024   # 定期 checkpoint 的 ack 数
queue.checkpoint.writes => 1024 # 定期 checkpoint 的写入数
queue.checkpoint.interval => 1000  # checkpoint 间隔（毫秒）
```

### 队列监控

```bash
# 查看队列状态
curl -s localhost:9600/_node/stats/pipeline | jq '.pipeline.queue'
```

## Filter 优化

### Filter 顺序

Filter 的执行顺序很重要。合理的顺序可以减少不必要的处理：

```ruby
filter {
  # 1. 快速过滤掉不需要的数据
  if [level] == "debug" {
    drop { }
  }

  # 2. 解析和转换（消耗资源）
  json { ... }
  grok { ... }

  # 3. 数据丰富（可能触发外部查询）
  geoip { ... }
  dns { ... }

  # 4. 最后的修修改改
  mutate { ... }
}
```

### 条件过滤

```ruby
filter {
  # 提前过滤，减少后续处理
  if [type] != "important" {
    drop { }
  }

  # 只对特定类型执行特定 Filter
  if [type] == "access" {
    grok { ... }
  } else if [type] == "error" {
    grok { ... }
  }
}
```

### expensive 插件

有些 Filter 是「expensive」的，会触发外部调用：

- `dns`：DNS 查询
- `elasticsearch`：ES 查询
- `http`：HTTP 请求

```ruby
filter {
  # 缓存 DNS 查询结果
  dns {
    resolve => ["hostname"]
    action => "replace"
    hit_cache_size => 10000
    hit_cache_ttl => 3600
  }
}
```

### 减少正则复杂度

Grok 正则越简单越好：

```ruby
# 复杂正则，效率低
grok {
  match => { "message" => "%{IP:ip} %{GREEDYDATA:rest}" }
}

# 简单正则，效率高
grok {
  match => { "message" => "%{IP:ip} - - %{DATA:rest}" }
}
```

使用预定义模式，避免编写复杂的自定义正则。

## Output 优化

### 批量发送

```ruby
output {
  elasticsearch {
    hosts => ["http://localhost:9200"]
    manage_template => false
    document_id => "%{[@metadata][id]}"

    # 批量配置
    sniffing => false           # 禁用 sniffing
    flush_size => 5000          # 累积多少条后发送
    idle_flush_time => 1        # 最大等待时间（秒）
  }
}
```

### 减少 Elasticsearch 操作

```ruby
output {
  elasticsearch {
    # 禁用不必要的操作
    action => "index"
    index => "my-index-%{+YYYY.MM.dd}"

    # 模板管理在命令行做
    manage_template => false
  }
}
```

### 多 Output

```ruby
output {
  # 只在本地调试时启用
  if "debug" in [tags] {
    stdout { codec => rubydebug }
  }

  elasticsearch { ... }
}
```

## 内存优化

### JVM 堆内存

```ruby
# jvm.options
-Xms2g
-Xmx2g
```

建议：
- 堆内存不超过 31GB（指针压缩限制）
- 最小和最大堆设置相同，避免动态调整
- 留 1GB 给操作系统

### 堆外内存

Lucene 使用堆外内存进行文件缓存：

```
总内存 = JVM 堆 + Lucene 段缓存（堆外）
```

Lucene 缓存 = 总内存 - JVM 堆 - 操作系统预留

### GC 调优

```ruby
# jvm.options
-XX:+UseG1GC
-XX:MaxGCPauseMillis=200
-XX:G1ReservePercent=15
```

## 常见问题与解决方案

| 问题 | 原因 | 解决方案 |
|---|---|---|
| CPU 使用率低 | workers 太少 | 增加 workers |
| 处理延迟增加 | Filter 太慢 | 优化 Filter 逻辑 |
| OOM | 队列堆积 | 增加 workers 或调小 batch_size |
| ES 写入慢 | ES 压力大 | 减少 flush_size，增加 ES 节点 |
| 丢数据 | 队列溢出 | 使用持久队列，增加队列大小 |

## 总结

Logstash 性能优化是一个系统工程：

**基础配置**：workers、batch_size、队列类型

**Filter 优化**：合理顺序、条件过滤、避免 expensive 操作

**Output 优化**：批量发送、减少不必要操作

**监控**：定期检查队列状态、处理延迟、GC 情况

记住：**先监控再优化**。在了解瓶颈在哪里之前，不要盲目调整参数。
