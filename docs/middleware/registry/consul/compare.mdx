# Consul vs ZooKeeper vs etcd vs Nacos 对比

注册中心的技术选型是一个需要综合考虑多个因素的决策。本节从功能特性、CAP 维度、运维成本等维度对比 Consul、ZooKeeper、etcd 和 Nacos，帮助你做出合适的选择。

## 功能特性对比

| 特性 | Consul | ZooKeeper | etcd | Nacos |
|---|---|---|---|---|
| **服务注册与发现** | 原生支持 | 可实现 | 可实现 | 原生支持 |
| **配置中心** | KV 存储 | 可实现 | KV 存储 | 原生支持 |
| **健康检查** | Script/HTTP/TCP/TTL | 临时节点 | TTL | TCP/HTTP/MySQL |
| **多数据中心** | 原生支持 | 不支持 | 不支持 | 支持 |
| **DNS 接口** | 原生支持 | 不支持 | 不支持 | 不支持 |
| **服务网格** | Consul Connect | 不支持 | 不支持 | 不支持 |
| **分布式锁** | 支持 | 支持 | 支持 | 支持 |
| **Web UI** | 原生支持 | 需第三方 | 需第三方 | 原生支持 |
| **权限控制** | ACL | ACL | RBAC | RBAC + ACL |
| **命名空间** | 支持 | 支持 | 支持 | 支持 |

## CAP 维度对比

### 一致性模型选择

四大注册中心在 CAP 维度上的选择：

| 注册中心 | CAP 模型 | 说明 |
|---|---|---|
| **Consul** | CP | 强一致性，Server 间通过 Raft 保证 |
| **ZooKeeper** | CP | 强一致性，ZAB 协议保证 |
| **etcd** | CP | 强一致性，Raft 协议保证 |
| **Nacos** | AP + CP 可切换 | 注册中心用 AP，配置中心用 CP |

```
CAP 维度分析：

CP 注册中心（Consul / ZooKeeper / etcd）：
- 写入需要获得多数节点确认
- Leader 宕机时需要重新选举
- 选举期间不可写入
- 适合对一致性要求高的场景

AP 注册中心（Nacos AP 模式）：
- 写入可发送到任意节点
- 各节点异步同步数据
- 任何节点都可写入
- 适合注册中心这种读多写少的场景
```

### 适用场景分析

| 场景 | 推荐选择 | 理由 |
|---|---|---|
| **服务发现** | Nacos / Consul | 原生支持，功能完善 |
| **配置中心** | Nacos / etcd | 功能完善，支持热更新 |
| **分布式锁** | ZooKeeper / etcd | 成熟稳定的锁实现 |
| **Master 选举** | ZooKeeper / etcd | 强一致保证 |
| **多数据中心** | Consul | 原生支持 |
| **Service Mesh** | Consul | Consul Connect |
| **云原生架构** | Consul / etcd | CNCF 生态 |

## 一致性协议对比

| 注册中心 | 协议 | 说明 |
|---|---|---|
| **Consul** | Raft 共识协议 | 通用共识算法，Leader + Follower |
| **ZooKeeper** | ZAB 协议 | 主备复制，专为 ZooKeeper 设计 |
| **etcd** | Raft 共识协议 | Go 实现，Kubernetes 原生支持 |
| **Nacos（CP）** | JRaft | Raft 的 Java 实现 |
| **Nacos（AP）** | Distro 协议 | 最终一致性，适合注册中心 |

### Raft vs ZAB

| 维度 | Raft | ZAB |
|---|---|---|
| **设计目标** | 通用共识算法 | ZooKeeper 专用 |
| **Leader 产生** | 选举产生 | 选举产生 |
| **日志复制** | Leader 复制日志到 Follower | Leader 广播事务 |
| **活锁问题** | 可能出现，需要随机超时避免 | 无（单一 Leader） |
| **实现复杂度** | 中等 | 高 |

## 性能对比

### 基准测试数据

| 操作 | Consul | ZooKeeper | etcd | Nacos |
|---|---|---|---|---|
| **写入延迟（单节点）** | ~5ms | ~2ms | ~1ms | ~3ms |
| **写入延迟（集群）** | ~10ms | ~5ms | ~3ms | ~8ms |
| **读取延迟** | <1ms | <1ms | <1ms | <1ms |
| **最大 QPS（单节点）** | ~10K | ~20K | ~30K | ~15K |
| **最大 QPS（集群）** | ~50K | ~80K | ~100K | ~60K |

:::info
以上数据为参考值，实际性能取决于硬件配置、网络延迟、数据大小等因素。
:::

### 性能选择建议

| 场景 | 推荐选择 | 理由 |
|---|---|---|
| **高写入压力** | etcd / ZooKeeper | 写入性能最佳 |
| **高读取压力** | 均可（读取性能相近） | 读取通常走缓存 |
| **大集群规模** | Nacos / Consul | 架构设计支持扩展 |
| **低延迟要求** | ZooKeeper / etcd | 延迟最低 |

## 选型决策树

### 第一步：确定 CAP 模型

```
是否需要强一致性（CP）？
├── 是 → ZooKeeper / etcd / Consul
└── 否 → Nacos（AP 模式）
```

### 第二步：确定功能需求

```
是否需要以下功能？

├── 多数据中心支持
│   └── 是 → Consul
│
├── 配置中心
│   └── 是 → Nacos（原生）或 etcd（KV）
│
├── Service Mesh
│   └── 是 → Consul
│
└── Spring Cloud 集成
    └── 是 → Nacos（最简单）
```

### 第三步：确定运维能力

```
团队运维能力如何？

├── 专业运维团队
│   └── ZooKeeper / Consul / etcd 均可
│
└── 普通开发团队
    └── Nacos（开箱即用，UI 友好）
```

## 场景化选型建议

### 微服务项目（国内）

| 项目类型 | 推荐选择 | 理由 |
|---|---|---|
| **新 Spring Cloud 项目** | Nacos | 开箱即用，中文社区 |
| **已有 ZooKeeper 团队** | ZooKeeper | 团队熟悉 |
| **需要配置中心** | Nacos | 一站式解决 |
| **多团队协作** | Nacos | namespace 隔离 |

### 云原生项目

| 项目类型 | 推荐选择 | 理由 |
|---|---|---|
| **Kubernetes 集成** | etcd / Consul | CNCF 生态 |
| **多云 / 混合云** | Consul | 多数据中心原生 |
| **Service Mesh** | Consul | Consul Connect |
| **服务网格规划** | Consul | 与 Istio 互补 |

### 金融 / 高可靠场景

| 场景 | 推荐选择 | 理由 |
|---|---|---|
| **强一致性要求** | ZooKeeper / etcd | CP 模型 |
| **分布式锁** | ZooKeeper | 成熟稳定 |
| **Master 选举** | ZooKeeper / etcd | 成熟方案 |
| **数据敏感** | Consul / etcd | mTLS / RBAC |

### 中间件集成

| 中间件 | 推荐注册中心 | 理由 |
|---|---|---|
| **Kafka** | ZooKeeper | Kafka 自身依赖 |
| **HBase** | ZooKeeper | HBase 依赖 |
| **Dubbo** | Nacos / ZooKeeper | 均支持 |
| **Kubernetes** | etcd | Kubernetes 原生 |

### 从 Eureka 迁移到 Consul

```xml
<!-- pom.xml -->
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-consul-discovery</artifactId>
</dependency>
```

```yaml
# application.yml
spring:
  cloud:
    consul:
      host: consul-server
      port: 8500
      discovery:
        service-name: ${spring.application.name}
        instance-id: ${spring.application.name}:${server.port}
  application:
    name: order-service
```

## 总结

| 注册中心 | 核心优势 | 最佳场景 |
|---|---|---|
| **Nacos** | 一站式、AP + CP、Spring Cloud 原生 | Spring Cloud 国内项目 |
| **ZooKeeper** | 强一致性、成熟稳定、分布式锁 | 金融场景、分布式协调 |
| **Consul** | 多数据中心、DNS 接口、Service Mesh | 云原生、跨云部署 |
| **etcd** | 高性能、Kubernetes 原生、CNCF | Kubernetes 集成 |
