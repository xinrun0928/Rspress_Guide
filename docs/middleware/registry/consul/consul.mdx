# Consul 核心特性

Consul 提供了四大核心能力：**服务发现、健康检查、KV 存储、多数据中心**。理解这些核心特性，是掌握 Consul 的基础。

## 服务发现

### 注册服务

 Consul 服务注册非常简单，可以通过配置文件或 API：

```json
// 服务定义文件 /etc/consul.d/service.json
{
  "service": {
    "name": "order-service",
    "id": "order-service-1",
    "port": 8080,
    "address": "192.168.1.10",
    "meta": {
      "version": "v1",
      "region": "us-east"
    },
    "tags": ["primary", "v1"],
    "checks": [
      {
        "id": "order-health",
        "name": "HTTP API Health Check",
        "http": "http://192.168.1.10:8080/health",
        "interval": "10s",
        "timeout": "5s"
      }
    ]
  }
}
```

```bash
# 通过 HTTP API 注册
curl -X PUT -d '{
  "ID": "order-service-1",
  "Name": "order-service",
  "Address": "192.168.1.10",
  "Port": 8080,
  "Tags": ["primary"],
  "Meta": {
    "version": "v1"
  }
}' http://consul-server:8500/v1/agent/service/register
```

### 服务发现方式

Consul 支持两种服务发现方式：

| 方式 | 端口 | 说明 |
|---|---|---|
| **HTTP API** | 8500 | 完整的 REST API |
| **DNS** | 53 | 兼容现有应用，无需修改代码 |

#### HTTP API 发现

```bash
# 查询服务
curl http://consul-server:8500/v1/catalog/service/order-service

# 查询结果
[
  {
    "ID": "order-service-1",
    "Node": "consul-server",
    "Address": "192.168.1.10",
    "ServicePort": 8080,
    "ServiceTags": ["primary"],
    "ServiceMeta": {"version": "v1"}
  }
]
```

#### DNS 发现

Consul 暴露 DNS 接口，域名格式为：`{service}.service.{datacenter}.consul`

```bash
# DNS 查询
dig @consul-server -p 8600 order-service.service.consul

# SRV 记录查询（包含端口）
dig @consul-server -p 8600 SRV order-service.service.consul
```

```java
// Java 应用中使用 DNS 发现
// Consul Agent 会在本地启动 DNS 服务器（默认 127.0.0.1:8600）
// 应用只需要使用 DNS 查询即可
String serviceUrl = "http://order-service.service.consul/order/123";

// 或者配置 Consul Agent 转发到系统 DNS
// /etc/resolv.conf 添加 nameserver 127.0.0.1
```

## 健康检查

Consul 支持多种健康检查方式，可以组合使用：

| 检查类型 | 说明 | 配置 |
|---|---|---|
| **Script + Interval** | 执行脚本检查 | `script: "curl localhost:8080/health"` |
| **HTTP + Interval** | HTTP GET 请求 | `http: "http://localhost:8080/health"` |
| **TCP + Interval** | TCP 连接检查 | `tcp: "localhost:8080"` |
| **TTL** | 服务主动报告健康状态 | `ttl: "10s"` |
| **Docker** | 执行 Docker 容器检查 | `docker_container_id: "xxx"` |
| **gRPC** | gRPC 健康检查协议 | `grpc: "/Health/Check"` |

### 主动检查 vs 被动检查

```
主动检查（Agent 执行）：
- Consul Agent 定期执行健康检查
- Agent 主动探测服务健康状态
- 适合 HTTP、TCP 等外部可探测的服务

被动检查（TTL）：
- 服务主动报告自己的健康状态
- 服务宕机后，不主动发送心跳，TTL 过期自动标记为不健康
- 适合无法被外部探测的服务（如数据库）
```

### TTL 健康检查

```json
{
  "check": {
    "id": "order-service-ttl",
    "name": "Order Service TTL",
    "service_id": "order-service-1",
    "ttl": "30s"
  }
}
```

```bash
# 服务通过 HTTP API 报告健康状态
# 每 10 秒发送一次
while true; do
    curl -X PUT http://localhost:8500/v1/agent/check/pass/service:order-service-1
    sleep 10
done
```

### 健康检查与故障检测

```
健康检查流程：
1. Consul Agent 每 10 秒执行一次健康检查
2. 检查失败 → Consul 标记为不健康
3. Consul 通知所有订阅的服务变更
4. 消费者更新本地实例列表，移除不健康实例

故障检测：
- 连续 N 次检查失败，标记为不健康
- 默认 N = 3（可配置）
- 服务恢复后，连续 N 次检查成功，标记为健康
```

## KV 存储

Consul 的 KV 存储是分层键值系统，可用于配置管理、特性开关、分布式锁等场景。

### 基本操作

```bash
# 存储键值
curl -X PUT -d 'value=100' http://consul-server:8500/v1/kv/config/order/maxConnections
curl -X PUT -d 'value=5000' http://consul-server:8500/v1/kv/config/order/timeout

# 读取键值
curl http://consul-server:8500/v1/kv/config/order/maxConnections

# 递归读取
curl http://consul-server:8500/v1/kv/config/order?recurse=true

# 删除键值
curl -X DELETE http://consul-server:8500/v1/kv/config/order/maxConnections

# CAS 操作
curl -X PUT -d 'value=200' \
  -H "If-None-Match: 123" \
  http://consul-server:8500/v1/kv/config/order/maxConnections
```

### 典型应用场景

| 场景 | 示例 |
|---|---|
| **配置中心** | `kv/config/{service}/{key}` |
| **特性开关** | `kv/features/{feature}/enabled` |
| **分布式锁** | `kv/locks/{resource}` |
| **Leader 选举** | `kv/leader/{service}` |
| **服务配额** | `kv/quota/{service}/requests` |

### 配置热更新

与 ZooKeeper 类似，Consul 支持 watches 机制感知配置变更：

```bash
# 监控 key 变更
curl -X GET http://consul-server:8500/v1/kv/config/order?index={index}&wait=60s
```

```java
// Consul Java Client 监听变更
ConsulClient client = new ConsulClient("consul-server:8500");

// 监听 key 前缀变更
Response<List<KvTuple>> response = client.getKVValue(
    "config/order/",
    new QueryParams(60, "")
);

// 处理变更
List<KvTuple> changes = response.getValue();
for (KvTuple kv : changes) {
    System.out.println(kv.getKey() + " = " + kv.getValue());
}
```

## 多数据中心

Consul 最强大的特性之一是**原生支持多数据中心**，不需要任何额外的协调组件。

### 架构设计

```
多数据中心架构：

Datacenter US（华盛顿）
├── Consul Server 1（Leader）
├── Consul Server 2
├── Consul Server 3
├── Consul Agent（Client）→ order-service
└── Consul Agent（Client）→ user-service

Datacenter EU（法兰克福）
├── Consul Server 1（Leader）
├── Consul Server 2
└── Consul Agent（Client）→ payment-service

两个数据中心之间通过 WAN Gossip 协议通信
```

### 联邦配置

```bash
# 添加 WAN 成员
consul join -wan 10.0.1.2  # EU 数据中心的地址

# 配置数据中心名称
consul agent -datacenter=us-east-1
```

### 跨数据中心服务发现

```bash
# 查询远程数据中心的服务
curl http://consul-server:8500/v1/catalog/service/order-service?dc=eu-west-1

# 在 DNS 中查询远程服务
dig @consul-server -p 8600 order-service.service.eu-west-1.consul
```

### 数据复制模式

| 模式 | 说明 | 适用场景 |
|---|---|---|
| **只读复制** | 服务目录异步复制到其他数据中心 | 跨数据中心的服务发现 |
| **读请求转发** | 本地没有的数据，转发到其他数据中心 | 一致性要求不高的场景 |
| **独立数据中心** | 各数据中心独立，数据不复制 | 强隔离场景 |

## Raft 共识协议

Consul 使用 Raft 协议保证 Server 节点之间的一致性：

### Leader 选举

```
选举流程：
1. 节点启动时为 Follower
2. 等待 election timeout（150-300ms 随机）
3. 变为 Candidate，发起选举
4. 获得多数票成为 Leader
5. Leader 定期发送心跳维持领导地位
```

```hcl
# Consul 配置 Raft 参数
server = true
bootstrap_expect = 3
datacenter = "us-east-1"

# 调整 Raft 参数（通常不需要修改）
performance {
  raft_multiplier = 1  # 默认值，性能乘数
}
```

### 数据复制

```
写请求流程：
1. 客户端发送写请求到任意 Server
2. Server 转发给 Leader
3. Leader 将日志复制到多数 Follower
4. Leader 提交日志并响应客户端
```

## Consul Connect：服务网格

Consul Connect 是 Consul 的 Service Mesh 解决方案：

### 核心特性

| 特性 | 说明 |
|---|---|
| **mTLS 加密** | 服务间通信自动加密 |
| **服务身份** | 基于 TLS 证书的服务身份认证 |
| **意图（Intentions）** | 基于白名单的访问控制 |
| **L7 流量管理** | HTTP/gRPC 层路由和分割 |
| **Envoy 集成** | 高性能 Sidecar 代理 |

### 配置示例

```hcl
# 启用 Connect
connect {
  enabled = true
}

# 定义服务
service {
  name = "order-service"
  port = 8080

  # 启用 Connect
  connect {
    sidecar_service {
      port = 21000
      proxy {
        upstreams = [
          {
            destination_name = "user-service"
            local_bind_port = 8081
          }
        ]
      }
    }
  }
}
```

```hcl
# 定义访问意图（哪些服务可以访问 order-service）
intent "order-service" "user-service" {
  action = "allow"
}
```
