# Nacos 选举机制与实例类型

Nacos 集群中的 Leader 节点负责处理配置写入，而注册中心则允许所有节点独立处理服务注册。理解 Nacos 的选举机制，有助于在生产环境中合理规划集群部署和故障应对策略。

## Leader 选举机制

### Raft 变体：Nacos-JRaft

Nacos 的 CP 模式（配置中心场景）使用基于 Raft 协议的选举机制。但 Nacos 并没有直接使用原生 Raft，而是使用了 **JRaft**——一个 Java 实现的 Raft 协议库。

```
Nacos 选举的触发时机：
1. 集群启动时，所有节点都是 Follower
2. 等待选举超时（Election Timeout）后，节点变为 Candidate 并发起选举
3. 获得多数票的节点成为 Leader
4. Leader 宕机后，其他节点重新发起选举
```

### 选举流程

```java
// 简化的选举逻辑
public class Raft LeaderElection {

    // 选举超时时间（随机范围，避免同时发起选举）
    private static final int ELECTION_TIMEOUT_MIN = 150;
    private static final int ELECTION_TIMEOUT_MAX = 300;

    public void startElection() {
        // 1. 增加当前任期号
        currentTerm.incrementAndGet();
        state = Candidate;

        // 2. 投票给自己
        voteFor = selfId;
        int votesReceived = 1;

        // 3. 向其他节点发送投票请求
        for (String peer : peers) {
            // 请求中包含：term, candidateId, lastLogIndex, lastLogTerm
            VoteResponse response = requestVote(peer, currentTerm, selfId);

            if (response.isGranted()) {
                votesReceived++;
            }
        }

        // 4. 获得多数票，成为 Leader
        if (votesReceived > peers.size() / 2) {
            becomeLeader();
        }
    }

    private void becomeLeader() {
        state = Leader;
        // 向所有节点发送心跳，维护 Leader 地位
        startHeartbeat();
    }
}
```

### 选举的关键要素

| 要素 | 说明 |
|---|---|
| **Term（任期号）** | 递增的逻辑时钟，用于判断节点状态 |
| **Vote（投票）** | 每个任期只能投一票，投票给日志最新的节点 |
| **Log Replication（日志复制）** | Leader 将客户端请求作为日志条目复制到 Follower |
| **心跳** | Leader 定期发送心跳，维持领导地位 |

### Leader 与 Follower 的交互

```
Leader → Follower：发送心跳（AppendEntries RPC）
           - 携带当前任期号
           - 携带已提交的日志索引
           - 如果有未同步日志，一并携带

Follower → Leader：响应心跳
           - 如果发现 Leader 任期号小于自己，转为 Candidate
           - 如果日志条目匹配，增加已提交索引
```

### 选举的保证

Raft 协议保证以下特性：

| 保证 | 说明 |
|---|---|
| **Leader 唯一性** | 同一任期只有一个 Leader |
| **日志完整性** | Leader 包含所有已提交日志 |
| **状态机安全** | 如果日志条目已在多数节点上应用，状态机不会执行冲突命令 |

```
选举安全性证明：
- 节点必须获得多数票才能成为 Leader
- 要获得多数票，必须有最新的日志
- 因此新 Leader 的日志是所有节点中最完整的
```

## 临时实例与永久实例

### 概念对比

这是 Nacos 中两个非常重要的概念，直接影响服务的注册和发现行为：

| 维度 | 临时实例（ephemeral） | 永久实例（persistent） |
|---|---|---|
| **存储位置** | 内存 | 磁盘 |
| **健康检查** | 客户端心跳保活 | Nacos Server 主动探测 |
| **故障处理** | 超时自动删除 | 标记为不健康，保留实例 |
| **适用场景** | 服务提供者 | 中间件、基础设施服务 |
| **数据持久性** | 不持久化 | 持久化存储 |

### 临时实例的生命周期

```
注册 → 心跳保活（每 5 秒）→ 正常状态
                  ↓
          心跳超时（15 秒）
                  ↓
            删除实例
```

```yaml
# 默认是临时实例
spring:
  cloud:
    nacos:
      discovery:
        ephemeral: true
```

临时实例适用于**服务提供者**——当服务实例宕机时，注册信息自动消失，消费者不会被路由到已宕机的实例。

### 永久实例的生命周期

```
注册 → Nacos 主动探测（TCP/HTTP/MySQL）→ 标记不健康
                                            ↓
                                    实例恢复后标记健康
```

```yaml
# 配置永久实例
spring:
  cloud:
    nacos:
      discovery:
        ephemeral: false
```

永久实例适用于**无法发送心跳的中间件**——如数据库、消息队列等，它们依赖 Nacos 主动探测健康状态。

:::warning
**永久实例不会自动删除**。如果实例确实宕机了，它会一直存在于注册表中（但标记为不健康）。运维需要手动删除已下线的永久实例，或者配置实例自动过期策略。
:::

## 实例权重与健康保护

### 权重机制

实例权重决定了负载均衡时的流量分配比例：

```yaml
# 服务 A 的三个实例
# 实例 1：权重 1.0，承担 10% 流量
# 实例 2：权重 1.0，承担 10% 流量
# 实例 3：权重 8.0，承担 80% 流量
spring:
  cloud:
    nacos:
      discovery:
        weight: 8.0
```

| 权重 | 适用场景 |
|---|---|
| 0 | 临时下线，不想删除实例但暂停流量 |
| 0.001 ~ 1 | 金丝雀发布，少量流量验证新版本 |
| 1 | 正常实例 |
| 2+ | 核心实例，承担更多流量 |

### 健康实例保护阈值

Nacos 提供了**健康实例保护阈值**功能，防止在大量实例不健康时，所有流量都打到少数健康实例上：

```yaml
spring:
  cloud:
    nacos:
      discovery:
        # 当健康实例占比低于 50% 时，不再剔除不健康实例
        health-check:
          protection-threshold: 0.5
```

```
场景：10 个实例，9 个不健康

未配置保护阈值：
- 只剩 1 个健康实例，承受全部流量
- 健康实例可能过载

配置保护阈值 0.5（50%）：
- 健康实例占比 = 10%，低于阈值
- Nacos 停止剔除不健康实例
- 消费者从 10 个实例中选择（包含不健康的）
- 每个实例分摊 10% 流量，避免单点过载
```

:::tip
健康保护阈值是一种**有损降级策略**——它允许少量请求打到不健康的实例上，但保证了整体可用性。适用于对可用性要求高于一致性的场景（如查询服务），但不适合对数据准确性要求高的场景（如下单服务）。
:::

## 集群部署建议

### 节点数量

Nacos 集群的节点数量建议：

| 节点数 | 适用场景 | 说明 |
|---|---|---|
| 1 | 本地开发 | 单机体验 |
| 3 | 小规模生产 | 推荐最小生产配置 |
| 5 | 中等规模生产 | 更高的容灾能力 |
| 7+ | 大规模生产 | 不建议，超过 5 个节点性能下降 |

### 节点角色

Nacos 2.0 的节点角色：

| 角色 | 说明 | 数量建议 |
|---|---|---|
| **Leader** | 处理所有 CP 写入 | 1 个 |
| **Follower** | 同步 Leader 数据，参与选举 | N-1 个 |
| **Learner** | 纯读取，不参与选举 | 可选 |

```
3 节点集群：1 Leader + 2 Follower
5 节点集群：1 Leader + 4 Follower
```

### 跨机房部署

对于需要高可用性的生产环境，建议跨机房部署：

```
推荐部署方案：3 机房各 1 节点

机房 A：节点 1（Leader 优先）
机房 B：节点 2
机房 C：节点 3

故障场景：
- 单机房故障：剩余 2 节点正常工作（过半）
- 双机房故障：只剩 1 节点，无法满足过半，不可写
```

## 总结

| 概念 | 说明 |
|---|---|
| **Leader 选举** | 基于 JRaft 的 Raft 协议变体 |
| **Term 任期** | 递增的逻辑时钟，用于判断节点状态 |
| **临时实例** | 客户端心跳保活，超时自动删除 |
| **永久实例** | Server 主动探测，不自动删除 |
| **实例权重** | 控制流量分配比例 |
| **健康保护阈值** | 防止少量健康实例过载 |
