# Nacos 集群一致性：Distro 协议

Nacos 集群需要保证各节点之间的数据一致性。与 ZooKeeper 使用 ZAB 协议、Consul 使用 Raft 协议不同，Nacos 在 AP 模式下使用**Distro 协议**来实现数据同步。

Distro 协议是一种专为注册中心场景设计的**最终一致性协议**，它的核心设计目标是**高可用性**——在保证最终一致性的前提下，尽可能减少节点间的同步开销。

## 为什么 Nacos 选择 Distro 协议

### 传统一致性协议的局限

传统的强一致性协议（如 Raft、ZAB）在以下场景中表现不佳：

| 问题 | 说明 |
|---|---|
| **写入瓶颈** | 所有写入都要经过 Leader，大量写入时 Leader 成为瓶颈 |
| **跨机房延迟** | Leader 和 Follower 可能不在同一机房，跨机房写入延迟高 |
| **可用性牺牲** | Leader 宕机时，集群需要重新选举，期间不可用 |
| **不适合注册中心** | 注册中心的写入量远大于配置中心，强一致性反而是负担 |

### Distro 协议的思路

注册中心的核心特点是**读多写少、变更频繁**：

```
一个 1000 个服务实例的注册中心：
- 每秒心跳写入：~200 次（每 5 秒心跳一次）
- 每秒服务发现读取：~10000 次（每个服务每秒被调用多次）
```

Distro 协议的设计哲学：**放松写入一致性要求，保证读取就近性，通过异步同步实现最终一致**。

## Distro 协议的核心设计

### 虚拟节点分片

Distro 协议将数据划分为多个**虚拟节点（VNode）**，每个虚拟节点负责一部分数据。虚拟节点的数量远大于物理节点数量，**每个物理节点负责多个虚拟节点**。

```
假设：16 个虚拟节点，4 个物理节点

节点映射关系：
- 虚拟节点 0-3   → 物理节点 A
- 虚拟节点 4-7   → 物理节点 B
- 虚拟节点 8-11  → 物理节点 C
- 虚拟节点 12-15 → 物理节点 D

数据分片：
- 服务 A 的实例  → 虚拟节点 5  → 物理节点 B
- 服务 B 的实例  → 虚拟节点 12 → 物理节点 D
```

### 数据分片算法

虚拟节点到物理节点的映射使用一致性哈希：

```java
// 一致性哈希分片
public class DistroConsistentHash {

    private final TreeMap<Long, String> virtualNodes = new TreeMap<>();

    public void addNode(String node) {
        // 每个物理节点映射到 150 个虚拟节点
        for (int i = 0; i < VIRTUAL_NODE_COUNT; i++) {
            long hash = hash(node + "VN" + i);
            virtualNodes.put(hash, node);
        }
    }

    public String getNode(String key) {
        long hash = hash(key);
        // 找到第一个大于等于 hash 的虚拟节点
        Map.Entry<Long, String> entry = virtualNodes.ceilingEntry(hash);
        if (entry == null) {
            entry = virtualNodes.firstEntry(); // 环绕到第一个节点
        }
        return entry.getValue();
    }
}
```

### 分片的好处

| 好处 | 说明 |
|---|---|
| **负载均衡** | 数据均匀分布到各个节点 |
| **故障隔离** | 单节点故障只影响部分数据 |
| **扩缩容平滑** | 新增/删除节点只需迁移部分虚拟节点 |

## 数据同步机制

### 写入流程

```
客户端写入请求到节点 A：
1. 节点 A 检查数据属于哪个虚拟节点
2. 如果属于自己：写入本地存储，异步同步到其他节点
3. 如果属于其他节点：转发到对应节点处理
4. 返回客户端成功
```

```java
public void onReceive(RegisterRequest request) {
    String serviceName = request.getServiceName();
    String targetNode = consistentHash.getNode(serviceName);

    if (currentNode.equals(targetNode)) {
        // 写入本地存储
        storage.save(serviceName, request.getInstance());

        // 异步同步到其他节点
        distroSyncService.sync(serviceName, request.getInstance());
    } else {
        // 转发到目标节点
        forwardRequest(targetNode, request);
    }
}
```

### 同步策略

Distro 协议采用**异步批量同步**：

| 同步方式 | 触发时机 | 说明 |
|---|---|---|
| **心跳同步** | 每次心跳时 | 将本地变更通过心跳携带到其他节点 |
| **批量同步** | 定时任务 | 定期批量同步增量数据 |
| **全量同步** | 节点加入时 | 新节点加入时从其他节点拉取全量数据 |

```
心跳同步示例：
- 节点 A 的服务列表发生变更
- 下次心跳时，节点 A 将变更信息带给客户端
- 客户端再次请求时，携带这些变更同步到其他节点

批量同步示例：
- 每 5 秒，节点 A 检查是否有未同步的变更
- 将增量数据批量发送到其他节点
```

### 同步确认机制

Distro 协议不要求所有节点都确认写入，但会记录同步状态：

```java
public class DistroSyncTask {

    private final ConcurrentMap<String, SyncTask> pendingTasks = new ConcurrentHashMap<>();

    public void sync(String key, Object value) {
        SyncTask task = pendingTasks.computeIfAbsent(key, k -> new SyncTask());

        // 发送到其他节点
        for (String node : getOtherNodes()) {
            boolean success = sendSync(node, key, value);
            if (success) {
                task.markSynced(node);
            }
        }

        // 如果所有节点都已同步，移除待同步任务
        if (task.isAllSynced()) {
            pendingTasks.remove(key);
        }
    }
}
```

## 分区容灾与故障转移

### 节点故障检测

Nacos 通过以下方式检测节点故障：

| 检测方式 | 说明 |
|---|---|
| **心跳检测** | 节点间定时发送心跳 |
| **客户端上报** | 客户端发现某个节点不可用，上报给其他节点 |
| **领导者检测** | Leader 节点定期检查各节点状态 |

### 故障转移

当某个节点故障时：

```
场景：节点 B 故障
1. Leader 检测到节点 B 心跳超时
2. Leader 将节点 B 的虚拟节点重新分配
   - 虚拟节点 4-7 从节点 B 迁移到节点 A
3. 客户端感知节点 B 不可用，自动切换到其他节点
4. 新请求路由到新的节点
5. 节点 B 恢复后，重新加入集群，数据同步恢复
```

### 数据修复

节点故障恢复后，需要从其他节点同步缺失的数据：

```
节点 B 故障 1 小时后恢复：
1. 节点 B 向 Leader 请求数据同步
2. Leader 计算节点 B 缺失的数据范围
3. Leader 将缺失数据推送给节点 B
4. 节点 B 接收并应用数据
5. 节点 B 恢复完成后，开始正常服务
```

```java
public void onNodeRecover(String nodeId) {
    // 获取节点故障期间的所有变更
    List<DataChange> changes = getChangesDuringFailure(nodeId);

    // 增量同步给恢复的节点
    for (DataChange change : changes) {
        syncToNode(nodeId, change);
    }

    // 确认同步完成
    confirmSyncComplete(nodeId);
}
```

## Distro 协议与 Raft 协议的对比

| 维度 | Distro 协议 | Raft 协议 |
|---|---|---|
| **一致性模型** | 最终一致性（AP） | 强一致性（CP） |
| **写入方式** | 可写入任意节点，异步同步 | 必须写入 Leader |
| **数据同步** | 异步批量同步 | 同步复制到多数节点 |
| **节点故障** | 不影响写入，自动路由 | 选举期间不可写入 |
| **适用场景** | 注册中心（读多写少） | 配置中心（写多读少） |
| **延迟** | 写入延迟低 | 写入延迟取决于节点数量 |

### 选择建议

```
Nacos 的策略：注册中心用 Distro，配置中心用 Raft

注册中心（AP）：
- 写入量大（心跳频繁）
- 允许短暂不一致（服务发现本身是最终一致的）
- 可用性优先

配置中心（CP）：
- 写入量小（配置变更是低频操作）
- 要求强一致性（配置错误影响巨大）
- 一致性优先
```

## Nacos 2.0 的协议优化

### gRPC 通信

Nacos 2.0 将 HTTP 通信升级为 gRPC，协议层面优化了性能：

| 优化点 | 说明 |
|---|---|
| **长连接** | 复用连接，减少连接建立开销 |
| **多路复用** | 单连接支持并发请求 |
| **流式传输** | 大数据量分片传输 |
| **推送优化** | 服务变更直接推送，无需客户端轮询 |

### 元数据压缩

心跳数据使用压缩算法减少网络传输：

```java
public class HeartbeatRequest {

    private String serviceName;
    private List<Instance> instances;

    // 序列化为 JSON 时使用压缩
    public String toCompressedJson() {
        // 使用 GZIP 压缩 JSON 数据
        byte[] jsonBytes = toJson().getBytes(StandardCharsets.UTF_8);
        return Base64.getEncoder().encodeToString(gzip(jsonBytes));
    }
}
```

## 总结

| 概念 | 说明 |
|---|---|
| **Distro 协议** | Nacos AP 模式下的最终一致性协议 |
| **虚拟节点分片** | 将数据划分为多个 VNode，实现负载均衡 |
| **一致性哈希** | 虚拟节点到物理节点的映射算法 |
| **异步批量同步** | 不阻塞写入，异步同步到其他节点 |
| **分区容灾** | 节点故障时自动转移，自动修复 |
| **gRPC 通信** | 2.0 版本引入长连接和多路复用 |
