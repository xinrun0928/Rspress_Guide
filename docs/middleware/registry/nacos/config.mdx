# Nacos 配置中心

如果说注册中心解决了「服务在哪」的问题，那么配置中心解决的是「服务用什么配置」的问题。在微服务架构中，配置管理是一个老大难问题：几十上百个服务，每个服务都有自己的一套配置，改一个公共配置要改几十个地方，简直是噩梦。

Nacos 的配置中心提供了一套完整的配置管理解决方案，支持配置的**集中管理、热更新、多环境隔离**。

## 配置管理的数据模型

### 三层隔离：namespace + group + dataId

Nacos 的配置通过三个维度进行隔离：

| 维度 | 说明 | 用途 |
|---|---|---|
| **namespace** | 命名空间 | 环境隔离，如 `dev`、`test`、`prod` |
| **group** | 分组 | 应用隔离，如 `ORDER_GROUP`、`USER_GROUP` |
| **dataId** | 配置 ID | 标识具体配置，如 `application.yml`、`datasource.properties` |

```
完整配置标识：
{namespace}/{group}/{dataId}
示例：prod/ORDER_GROUP/application.yml

等价于：
命名空间 prod 下，ORDER_GROUP 分组中的 application.yml 配置
```

### dataId 命名规范

dataId 的命名直接影响配置的使用方式：

| 命名格式 | 适用场景 | 示例 |
|---|---|---|
| `{prefix}.{file-extension}` | Spring Boot 默认规则 | `application.yaml` |
| `{prefix}-{profile}.{file-extension}` | 多环境配置 | `application-dev.yaml` |
| `{prefix}/{name}/{profile}.{file-extension}` | 复杂项目 | `order-service/database/prod.yaml` |

```java
// Spring Boot 自动关联规则
// 1. 加载 application.yaml
// 2. 激活 prod 环境时，加载 application-prod.yaml
// 3. 合并配置，重复项覆盖
```

### namespace 的使用

namespace 默认值是 `public`，用户可以创建自定义命名空间：

```yaml
# application.yml 指定 namespace
spring:
  cloud:
    nacos:
      config:
        server-addr: nacos-server:8848
        namespace: 5c9c-4b1a-xxxx-xxxx-xxxxxxxx  # namespace ID
        group: DEFAULT_GROUP
        file-extension: yaml
```

```bash
# 通过 Nacos CLI 创建 namespace
nacos-cli.sh nacos-config create \
  --namespace-id=dev \
  --namespace-name=开发环境 \
  --namespace-desc=开发测试环境配置
```

## 配置管理核心功能

### 配置的增删改查

```java
// 获取配置
String config = configService.getConfig(dataId, group, timeout);

// 发布配置
boolean success = configService.publishConfig(dataId, group, content);

// 删除配置
boolean success = configService.deleteConfig(dataId, group);

// 监听配置变更
configService.addListener(dataId, group, listener);
```

### 配置的导入导出

Nacos 支持配置的批量导入导出，便于环境迁移和备份：

```bash
# 导出配置
nacos-config export --data-id application.yaml --group DEFAULT_GROUP --namespace prod

# 导出结果：application.yaml.zip（包含配置内容和元数据）

# 导入配置
nacos-config import --file application.yaml.zip --namespace prod --force
```

### 配置版本管理

每次配置发布都会生成一个新版本，支持版本回滚：

| 版本 | 内容 | 发布人 | 发布时间 |
|---|---|---|---|
| v10 | `maxConnections=200` | alice | 2024-01-15 10:30 |
| v9 | `maxConnections=100` | bob | 2024-01-14 15:20 |
| v8 | `maxConnections=50` | alice | 2024-01-13 09:00 |

```java
// 查询配置历史
List<ConfigHistory> histories = configService
    .listConfigHistory(dataId, group, page, pageSize);

// 回滚到指定版本
configService.rollback(dataId, group, version);
```

## 热更新：配置变更实时推送

热更新是配置中心的核心能力。当配置在 Nacos 控制台修改后，客户端能**立即感知**并应用新配置，无需重启服务。

### Nacos 1.x：长轮询

Nacos 1.x 使用 HTTP 长轮询（Long Polling）实现配置变更推送：

```
客户端行为：
1. 发送 HTTP 请求到 Server，带上本地配置的 MD5 值
2. Server 检查 MD5 是否变化
3. 如果没变化，保持请求 30 秒后返回
4. 如果有变化，立即返回最新配置
5. 客户端收到后更新本地配置，并发起新一轮轮询
```

```java
// 长轮询核心逻辑
public class LongPollingTask implements Runnable {
    public void run() {
        while (running) {
            try {
                // 发送监听请求
                HttpResult result = httpAgent
                    .sendGetRequest(listenerUrl);

                if (result.code == 200) {
                    // 配置变更，解析并应用新配置
                    List<ConfigChange> changes = parseResult(result.content);
                    applyChanges(changes);
                }
            } catch (Exception e) {
                // 发生异常，短暂等待后重试
                Thread.sleep(1000);
            }
        }
    }
}
```

### Nacos 2.0：gRPC 长连接

Nacos 2.0 引入了 gRPC 长连接，配置推送效率大幅提升：

| 对比项 | 1.x 长轮询 | 2.0 gRPC 长连接 |
|---|---|---|
| 连接方式 | HTTP 短连接 | gRPC 长连接 |
| 推送方式 | 客户端拉取 | Server 主动推送 |
| 响应延迟 | 30 秒（长轮询超时） | < 100ms |
| 服务端压力 | 高（每个客户端维持一个 HTTP 连接） | 低（复用连接） |
| 网络开销 | 频繁建连 | 连接复用 |

### 配置变更的完整流程

```
发布配置：
1. 运维在 Nacos 控制台修改配置
2. Server 将配置写入数据库
3. Server 通过 gRPC 推送通知到所有订阅客户端
4. 客户端收到通知，拉取最新配置
5. 客户端更新本地缓存
6. 客户端通过监听器回调通知业务代码
```

```java
// 监听配置变更
configService.addListener("application.yaml", "DEFAULT_GROUP", new Listener() {
    @Override
    public void receiveConfigInfo(String configInfo) {
        // 收到配置变更通知
        System.out.println("配置变更：" + configInfo);
        // 重新解析和应用配置
    }

    @Override
    public Executor getExecutor() {
        // 指定回调执行的线程池
        return executors;
    }
});
```

## Spring Cloud Config 集成

### 自动刷新

Nacos 与 Spring Cloud 深度集成，支持配置变更自动刷新：

```java
// 方式一：@RefreshScope 注解
@RestController
@RefreshScope
public class ConfigController {

    @Value("${order.max-connections:100}")
    private int maxConnections;

    @GetMapping("/config")
    public String getConfig() {
        return "maxConnections = " + maxConnections;
    }
}

// 方式二：@ConfigurationProperties
@Data
@Component
@ConfigurationProperties(prefix = "order")
@RefreshScope
public class OrderProperties {
    private int maxConnections = 100;
    private int timeout = 5000;
}
```

:::tip
`@RefreshScope` 的原理是：配置变更后，Spring 容器会销毁旧的 Bean 并创建新 Bean，实现配置的热更新。但这种方式对 `@Bean` 方法定义的 Bean 和 `static` 字段无效。
:::

### 配置优先级

当多个配置源定义了相同的配置项时，Spring Boot 按以下优先级生效：

```
（高）Nacos 配置中心的配置
（低）application.yml 中的配置
```

```yaml
# application.yml（优先级低）
order:
  max-connections: 100

# Nacos 配置中心的 application.yaml（优先级高）
order:
  max-connections: 200

# 最终生效值：200
```

### 共享配置

Nacos 支持将公共配置提取为「共享配置」，供多个服务引用：

```yaml
# order-service.yml
spring:
  cloud:
    nacos:
      config:
        shared-configs:
          - data-id: common.yaml
            group: COMMON_GROUP
            refresh: true
          - data-id: database.yaml
            group: COMMON_GROUP
            refresh: false
```

```yaml
# common.yaml 内容
jackson:
  date-format: yyyy-MM-dd HH:mm:ss
  time-zone: Asia/Shanghai

logging:
  level:
    root: INFO
    com.example: DEBUG
```

## 多环境配置隔离

### 环境划分

典型的环境划分方案：

| 环境 | namespace | 用途 |
|---|---|---|
| `dev` | 开发环境 | 本地开发、联调 |
| `test` | 测试环境 | 集成测试、预发布 |
| `uat` | 预发环境 | 贴近生产的测试 |
| `prod` | 生产环境 | 正式生产 |

### 配置迁移

从测试环境到生产环境的配置迁移：

```bash
# 方式一：导出再导入
nacos-config export --data-id application.yaml --namespace test --file app-test.yaml
nacos-config import --data-id application.yaml --namespace prod --file app-test.yaml

# 方式二：配置对比后再迁移
nacos-config diff \
  --data-id application.yaml \
  --group DEFAULT_GROUP \
  --namespace-1 test \
  --namespace-2 prod
```

### 配置加密

敏感配置（如数据库密码、API 密钥）需要加密存储：

```yaml
# 使用加密配置
spring:
  datasource:
    password: ${cipher}AES://xxxxxxxxxxxxxxx==
```

Nacos 支持自定义加密算法，只需实现 `ConfigEncryptPlugin` 接口：

```java
// 实现加密插件
public class AesEncryptPlugin implements ConfigEncryptPlugin {
    @Override
    public String encrypt(String dataId, String content) {
        return "AES:" + aesEncrypt(content);
    }

    @Override
    public String decrypt(String dataId, String encryptedContent) {
        return aesDecrypt(encryptedContent.substring(4));
    }
}
```
