# 注册中心

在分布式系统中，服务与服务之间需要相互感知——服务 A 怎么知道服务 B 在哪里？服务 B 宕机了，服务 A 怎么及时知道并切换到可用的服务 C？

这就是**注册中心**要解决的核心问题。

## 为什么需要注册中心

在没有注册中心的年代，开发者通常会把服务地址写死在配置文件里，或者通过 DNS 做简单的负载均衡。这种方式在服务数量少、变更不频繁时勉强能用，但一旦规模扩大，问题就暴露出来了：

**硬编码的问题：**
- 服务实例 IP 变更时，所有依赖方都要改配置
- 无法感知服务健康状态，调用已宕机的实例会导致请求失败
- 负载均衡策略只能基于简单轮询，无法动态调整
- 服务的上线和下线完全靠人工维护

注册中心的出现，本质上是将「**服务地址的动态发现**」这件事从应用层抽离出来，交给专门的组件去处理。

## 注册中心的三大核心功能

### 服务注册

当一个服务实例启动时，它会主动向注册中心上报自己的信息，包括：
- 服务名（`service-name`）
- 实例 ID（`instance-id`）
- IP 地址和端口
- 权重、健康检查信息等元数据

这些信息被存储在注册中心的数据存储中，供其他服务查询。

### 服务发现

服务消费者需要知道服务提供者的地址，才能发起调用。服务发现有两种模式：

**客户端发现模式**：服务消费者从注册中心获取服务提供者的完整列表，然后在本地完成负载均衡选择。这种模式的好处是去中心化、性能好，缺点是每个客户端都要集成 SDK。

**服务端发现模式**：服务消费者通过负载均衡器（如 Ingress、Nginx）来访问服务，负载均衡器从注册中心获取地址列表并完成路由。这种模式对客户端友好，但多了一层组件依赖。

### 健康检查与剔除

注册中心需要实时感知服务实例的健康状态。当一个实例出现故障（如心跳超时）时，注册中心会将其从可用列表中移除，通知所有订阅者进行更新。

这个机制保证了服务消费者永远不会把请求发送到已经宕机的实例上。

## 四种主流注册中心

| 特性 | ZooKeeper | Nacos | Eureka | Consul |
|---|---|---|---|---|
| **一致性模型** | CP（强一致性） | AP + CP 双模式 | AP（最终一致性） | CP（强一致性） |
| **协议** | ZAB 协议 | Distro + Raft | HTTP + 心跳 | Raft 共识协议 |
| **服务健康检查** | 临时节点心跳 | TCP/HTTP/MySQL | 心跳续约 | Script/HTTP/TCP/TTL |
| **服务变更推送** | Watch 机制 | 长期订阅 + 异步推送 | 增量拉取 | HTTP Long Polling |
| **多数据中心** | 不原生支持 | 支持 | 不原生支持 | 原生支持多数据中心 |
| **运维复杂度** | 高（需 ZooKeeper 集群） | 中（开箱即用） | 低（Spring Cloud 集成） | 中（Agent 部署） |
| **社区生态** | 成熟稳定 | 活跃（阿里背景） | Spring Cloud 官方 | HashiCorp 维护 |
| **适用场景** | 配置中心、分布式锁、Master 选举 | 服务注册与配置一体化 | Spring Cloud 微服务 | 多数据中心、云原生 |

### 一句话选型

- **需要强一致性、分布式锁、Master 选举**：选 ZooKeeper
- **Spring Cloud 项目，想要开箱即用**：选 Eureka（但 2.x 已停止维护，需考虑迁移）
- **需要配置中心 + 注册中心一体化**，且国内项目选 Nacos
- **多数据中心、跨云部署、Consul + Vault 联动**：选 Consul
