# Eureka 架构与核心机制

Eureka 的架构设计体现了「简单、可靠」的原则。理解 Eureka 的核心组件（Server、Client）、心跳续约机制、以及自我保护机制，是正确使用 Eureka 的前提。

## Eureka Server 架构

### 服务端核心职责

Eureka Server 作为注册中心的服务端，承担以下职责：

| 职责 | 说明 |
|---|---|
| **服务注册表** | 存储所有已注册的服务实例信息 |
| **服务注册** | 接收客户端的注册请求 |
| **心跳续约** | 接收客户端的心跳，更新实例状态 |
| **服务剔除** | 定时扫描过期实例并剔除 |
| **服务查询** | 提供 REST API 供客户端查询 |

### 服务注册表结构

Eureka 将服务信息存储在内存注册表中：

```java
// 服务注册表的核心结构
public abstract class AbstractInstanceRegistry implements InstanceRegistry {

    // 服务名 → 实例信息列表
    private final ConcurrentHashMap<String, Map<String, Lease<InstanceInfo>>> registry
        = new ConcurrentHashMap<>();

    // 续约信息
    private final ConcurrentHashMap<String, InstanceStatus> instanceStatus
        = new ConcurrentHashMap<>();
}
```

```
注册表结构：
{
  "order-service": {
    "instance-001": Lease(实例1, 注册时间, 最后更新时间),
    "instance-002": Lease(实例2, 注册时间, 最后更新时间)
  },
  "user-service": {
    "instance-001": Lease(实例3, 注册时间, 最后更新时间)
  }
}
```

### 服务剔除机制

Eureka Server 定时扫描注册表，剔除过期实例：

```java
// 定时任务：清理过期实例
@Override
public void evict(long additionalLeaseMs) {
    // 遍历所有注册的服务
    for (Map<String, Lease<InstanceInfo>> entry : registry.values()) {
        for (Lease<InstanceInfo> lease : entry.values()) {
            // 判断是否过期
            if (lease.isExpired(additionalLeaseMs)) {
                // 剔除实例
                expire(lease);
            }
        }
    }
}

// 判断是否过期
public boolean isExpired(long additionalLeaseMs) {
    // 超过 90 秒（默认）未收到心跳，且超过了宽限期
    return (evictionTimestamp > 0L)
        || (lastUpdateTimestamp + duration + additionalLeaseMs < System.currentTimeMillis());
}
```

| 参数 | 默认值 | 说明 |
|---|---|---|
| `leaseRenewalIntervalInSeconds` | 30s | 客户端心跳间隔 |
| `leaseExpirationDurationInSeconds` | 90s | 实例过期时间（默认 = 3 × 心跳间隔） |

## Eureka Client 架构

### 客户端核心职责

Eureka Client 负责与服务端交互：

| 职责 | 说明 |
|---|---|
| **服务注册** | 应用启动时向 Server 注册 |
| **心跳续约** | 定期向 Server 发送心跳 |
| **缓存刷新** | 定期从 Server 拉取服务列表 |
| **下线通知** | 应用关闭时主动注销 |

### 服务注册流程

```
应用启动：
1. Spring Cloud 初始化 DiscoveryClient
2. 读取配置，获取 Eureka Server 地址
3. 向 Eureka Server 发送 POST /eureka/apps/{appId}
4. Server 返回 204，表示注册成功
5. 启动心跳定时任务
6. 启动缓存刷新定时任务
```

```java
// 服务注册核心逻辑
public void register(InstanceInfo instanceInfo) {
    // 构造注册请求
    Application resource = new Application(instanceInfo.getAppName());
    resource.add(instanceInfo);

    // 发送注册请求
    EurekaHttpResponse<Void> response = eurekaTransport
        .register(instanceInfo);

    if (response.getStatusCode() == Status.NO_CONTENT.getStatusCode()) {
        logger.info("服务注册成功: {}", instanceInfo.getInstanceId());
    }
}
```

### 心跳续约机制

客户端定期发送心跳，告知 Server 「我还活着」：

```java
// 心跳定时任务
public class HeartbeatThread implements Runnable {
    @Override
    public void run() {
        // 续约请求：PUT /eureka/apps/{appId}/{instanceId}
        EurekaHttpResponse<InstanceInfo> response = eurekaTransport
            .sendHeartBeat(appName, instanceId, instanceInfo, null);

        if (response.getStatusCode() == Status.NOT_FOUND.getStatusCode()) {
            // 实例不存在，重新注册
            register();
        }
    }
}
```

```
心跳时序：
客户端每 30 秒 → Eureka Server（续约）
           ↓
Server 更新 lastUpdateTimestamp
           ↓
如果 90 秒内未收到心跳 → 标记为过期
```

## 自我保护机制

### 为什么需要自我保护

在云环境中，**瞬时网络抖动**或 **JVM GC 暂停**可能导致心跳暂时失败。如果 Eureka 严格剔除这些实例，会导致大量健康实例被误删，服务雪崩。

```
场景：AWS 某可用区网络抖动 30 秒

未开启自我保护：
- 30 秒内无法收到心跳的实例被标记为过期
- 网络恢复后，大量请求失败
- 服务雪崩

开启自我保护：
- Eureka 检测到大量心跳失败
- 触发自我保护模式，暂停剔除
- 网络恢复后，实例自动恢复正常
```

### 自我保护触发条件

Eureka 通过 `renewalPercentThreshold`（续约百分比阈值）判断是否触发自我保护：

```java
// 触发条件：每分钟期望的续约次数 vs 实际续约次数
// 如果实际续约次数 < 期望续约次数 × 阈值，触发自我保护

// 期望续约次数计算
expectedRenewalsPerMin = 总实例数 × (60 / 心跳间隔)
// 例如：10 个实例，30 秒心跳 → 期望 20 次/分钟

// 自我保护触发条件
if (actualRenewals < expectedRenewals × renewalPercentThreshold) {
    enableSelfPreservation();
}
```

### renewalPercentThreshold 的影响

| `renewalPercentThreshold` | 说明 |
|---|---|
| 默认值 0.85 | 实际续约低于期望的 85% 时触发 |
| 设置过高 | 难以触发，失去保护作用 |
| 设置过低 | 频繁触发，实例可能误剔除 |

```yaml
# 配置自我保护参数
eureka:
  server:
    # 续约百分比阈值，默认 0.85
    renewal-percent-threshold: 0.85
    # 是否开启自我保护，默认 true
    enable-self-preservation: true
    # 清理过期实例的时间间隔
    eviction-interval-timer-in-ms: 60000
```

### 自我保护的行为

```
触发自我保护后：

1. Eureka Server 页面上显示 "EMERGENCY! EUREKA MAY BE INCORRECT"
2. 停止剔除任何过期实例
3. 客户端仍然可以注册和发现服务
4. 当续约次数恢复到阈值以上时，自动退出自我保护
```

### 何时关闭自我保护

:::warning
在生产环境中，**不建议关闭自我保护**，除非：
1. 你的服务实例非常少（如少于 5 个），自我保护反而会影响正常的实例剔除
2. 你有其他机制（如 Kubernetes 健康检查）来保证服务实例的正确性
3. 你能接受实例误剔除导致的服务短暂不可用
:::

关闭自我保护的风险：

| 风险 | 说明 |
|---|---|
| **误剔除** | 网络抖动时正常实例被剔除 |
| **服务雪崩** | 大量实例被剔除，请求打到少量实例上 |
| **故障传播** | 消费者实例减少，导致调用失败 |

## 服务发现与缓存

### 客户端本地缓存

Eureka Client 不会每次调用都去 Server 查询，而是**缓存一份在本地**：

```
消费者行为：
1. 启动时，从 Eureka Server 拉取全量服务列表
2. 本地缓存，刷新间隔默认 30 秒
3. 服务调用时，从本地缓存获取服务列表
4. 如果缓存中没有，尝试从 Server 获取
5. 如果 Server 不可用，使用本地缓存（即使可能过期）
```

```yaml
# 配置缓存刷新间隔
eureka:
  client:
    # 刷新本地缓存的间隔，默认 30 秒
    registry-fetch-interval-seconds: 30
    # 是否从 Server 拉取注册表，默认 true
    fetch-registry: true
    # 启用只读缓存，默认 true
    use-dc-for-url: false
```

### 服务发现延迟

本地缓存机制带来的问题是**服务发现的延迟**：

| 场景 | 延迟 |
|---|---|
| 新服务注册 | 最长 30 秒才能被消费者发现 |
| 服务下线 | 最长 30 秒才能被消费者感知 |
| 服务变更 | 最长 30 秒才能生效 |

```yaml
# 降低延迟：减少缓存刷新间隔
eureka:
  client:
    registry-fetch-interval-seconds: 10  # 改为 10 秒

# 权衡：增加 Server 压力
```

## 总结

| 概念 | 说明 |
|---|---|
| **服务注册** | 启动时向 Server 注册，应用关闭时注销 |
| **心跳续约** | 每 30 秒发送一次，90 秒无心跳标记过期 |
| **服务剔除** | 定时扫描过期实例并删除 |
| **本地缓存** | 客户端缓存服务列表，减少 Server 压力 |
| **自我保护** | 防止因网络抖动导致误剔除健康实例 |
| **renewalPercentThreshold** | 触发自我保护的续约阈值 |
