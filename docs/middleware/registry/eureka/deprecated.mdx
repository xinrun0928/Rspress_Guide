# Eureka 停止维护与替代方案

2020 年 4 月，Netflix 官方宣布 **Eureka 2.x 停止维护**，不再发布新版本。这意味着 Eureka 2.x 的已知漏洞将不会被修复，生产环境使用 Eureka 2.x 存在安全风险。

如果你正在使用 Eureka，或者计划在新项目中使用 Eureka，了解迁移方案是必要的。

## Eureka 2.x 停止维护的背景

### 停止维护的原因

Netflix 在官方博客中解释了停止维护 Eureka 2.x 的原因：

> Eureka 2.x 的采用率不如预期。我们将资源集中在更加核心和广泛使用的组件上。

```
Netflix 的技术演进方向：
- 微服务治理：从自研走向 Service Mesh（Envoy + Istio）
- 服务发现：从 Eureka 走向 Zuul + Eureka 混合
- 云原生：拥抱 Kubernetes + Envoy
```

### Eureka 1.x 的状态

**好消息**：Eureka 1.x 仍然在维护中，作为开源项目接受社区贡献。

**坏消息**：Netflix 不再主动开发新功能，只修复严重 bug。

```
Eureka 1.x 当前状态（截至 2024 年）：
- GitHub 仍在接受 PR
- 严重安全漏洞会被修复
- 但不会有 Eureka 2.x 的新功能
```

### 风险评估

| 风险类型 | 严重程度 | 应对措施 |
|---|---|---|
| 安全漏洞 | 高 | 升级到替代方案 |
| 功能缺失 | 中 | 使用替代方案的功能 |
| 社区支持 | 低 | Eureka 1.x 仍在维护 |

:::danger
**如果你正在使用 Eureka 2.x，强烈建议迁移到其他活跃维护的注册中心。**
:::

## 替代方案对比

### 选项概览

| 替代方案 | 特点 | 迁移难度 |
|---|---|---|
| **Nacos** | 阿里开源，功能全面 | 中 |
| **Consul** | HashiCorp 出品，云原生友好 | 中 |
| **ZooKeeper** | 经典方案，强一致性 | 高 |
| **Spring Cloud LoadBalancer** | Spring Cloud 官方，临时方案 | 低 |

### Nacos：国内首选

Nacos 是阿里巴巴开源的一站式服务发现和配置管理平台，与 Spring Cloud 集成良好：

| 维度 | Eureka | Nacos |
|---|---|---|
| **活跃度** | 低（1.x 维护中） | 高 |
| **功能** | 纯注册中心 | 注册中心 + 配置中心 |
| **一致性模型** | AP | AP + CP 可切换 |
| **Spring Cloud 集成** | 原生 | 原生 |
| **中文社区** | 一般 | 活跃 |

### Consul：云原生首选

Consul 是 HashiCorp 出品，专注于服务发现和配置：

| 维度 | Eureka | Consul |
|---|---|---|
| **活跃度** | 低 | 高 |
| **功能** | 纯注册中心 | 注册中心 + KV 存储 + Service Mesh |
| **一致性模型** | AP | CP |
| **多数据中心** | 不支持 | 原生支持 |
| **Kubernetes 集成** | 一般 | 原生支持 |

### ZooKeeper：强一致性场景

如果你的场景需要强一致性（分布式锁、Master 选举），ZooKeeper 仍然是首选：

| 维度 | Eureka | ZooKeeper |
|---|---|---|
| **活跃度** | 低 | 高 |
| **一致性** | AP | CP |
| **功能** | 纯注册中心 | 协调服务 |
| **分布式锁** | 不支持 | 支持 |
| **Master 选举** | 不支持 | 支持 |

### Spring Cloud LoadBalancer：临时方案

Spring Cloud 2020.0 推出了官方的负载均衡器作为临时过渡方案：

```java
// 使用 Spring Cloud LoadBalancer 替代 Ribbon
@Configuration
public class LoadBalancerConfig {

    @Bean
    public ReactorLoadBalancer<ServiceInstance> randomLoadBalancer(
            ServiceInstanceListSupplier supplier) {
        return new RandomLoadBalancer(supplier);
    }
}

// 调用服务
@Service
public class OrderService {

    @Autowired
    private RestTemplate restTemplate;

    @Autowired
    private LoadBalancer loadBalancer;

    public User getUser(Long userId) {
        // 使用 LoadBalancer 选择实例
        ServiceInstance instance = loadBalancer.choose("user-service");
        String url = instance.getUri() + "/user/" + userId;
        return restTemplate.getForObject(url, User.class);
    }
}
```

:::warning
Spring Cloud LoadBalancer **不支持服务注册**。你需要结合 Consul、Nacos 或 Eureka 1.x 来实现完整的服务发现。
:::

## 迁移方案

### 从 Eureka 2.x 迁移到 Nacos

#### 步骤一：引入 Nacos 依赖

```xml
<!-- 移除 Eureka 依赖 -->
<!--
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-netflix-eureka-client</artifactId>
</dependency>
-->

<!-- 添加 Nacos Discovery 依赖 -->
<dependency>
    <groupId>com.alibaba.cloud</groupId>
    <artifactId>spring-cloud-starter-alibaba-nacos-discovery</artifactId>
</dependency>
```

#### 步骤二：配置迁移

```yaml
# Eureka 配置
eureka:
  client:
    service-url:
      defaultZone: http://eureka-server:8761/eureka/
  instance:
    prefer-ip-address: true
    lease-renewal-interval-in-seconds: 30

# 替换为 Nacos 配置
spring:
  cloud:
    nacos:
      discovery:
        server-addr: nacos-server:8848
        namespace: prod
        group: DEFAULT_GROUP
  application:
    name: order-service
```

#### 步骤三：代码适配

```java
// Eureka：DiscoveryClient
@Autowired
private DiscoveryClient discoveryClient;

public User getUser(Long userId) {
    List<ServiceInstance> instances = discoveryClient.getInstances("user-service");
    // 使用 Ribbon 或 LoadBalancer 选择实例
}

// Nacos：使用同样的 DiscoveryClient 接口
// 接口兼容，无需修改代码
```

#### 步骤四：实例 ID 配置

```yaml
# 保持与 Eureka 相同的实例 ID 格式
spring:
  cloud:
    nacos:
      discovery:
        # Eureka 格式：${spring.application.name}:${server.port}
        # Nacos 默认格式类似，但你可以自定义
        instance-id: ${spring.application.name}:${server.port}
```

### 从 Eureka 迁移到 Consul

```xml
<!-- 移除 Eureka 依赖 -->
<!-- 添加 Consul 依赖 -->
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-consul-discovery</artifactId>
</dependency>
```

```yaml
spring:
  cloud:
    consul:
      host: consul-server
      port: 8500
      discovery:
        service-name: order-service
        instance-id: ${spring.application.name}:${server.port}
  application:
    name: order-service
```

### 双注册过渡方案

在迁移期间，可以同时注册到 Eureka 和 Nacos，实现平滑过渡：

```xml
<!-- 同时引入两个客户端 -->
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-netflix-eureka-client</artifactId>
</dependency>
<dependency>
    <groupId>com.alibaba.cloud</groupId>
    <artifactId>spring-cloud-starter-alibaba-nacos-discovery</artifactId>
</dependency>
```

```yaml
# 同时启用两个注册中心
eureka:
  client:
    enabled: true
    service-url:
      defaultZone: http://eureka-server:8761/eureka/

spring:
  cloud:
    nacos:
      discovery:
        enabled: true
        server-addr: nacos-server:8848
```

```
迁移步骤：
1. 服务同时注册到 Eureka 和 Nacos
2. 消费者逐步切换到 Nacos
3. 确认稳定后，禁用 Eureka
4. 下线 Eureka 集群
```

## 新项目选型建议

对于新项目，**不建议选择 Eureka**：

| 项目类型 | 推荐选择 | 理由 |
|---|---|---|
| **Spring Cloud 国内项目** | Nacos | 社区活跃，功能全面 |
| **Spring Cloud 国外项目** | Consul | 云原生，多数据中心 |
| **需要配置中心** | Nacos | 一站式解决 |
| **微服务 + Service Mesh** | Consul | Consul Connect |
| **已有 ZooKeeper 团队** | ZooKeeper | 团队熟悉 |

## 总结

| 方案 | 适用场景 | 迁移难度 |
|---|---|---|
| **Nacos** | Spring Cloud 项目，国内团队 | 中 |
| **Consul** | 云原生，多数据中心 | 中 |
| **ZooKeeper** | 需要强一致性，分布式锁 | 高 |
| **Spring Cloud LoadBalancer** | 临时过渡，不新建 | 低 |
