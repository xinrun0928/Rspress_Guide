# Eureka 自我保护机制

Eureka 的自我保护（Self-Preservation）机制是保证服务可用性的重要特性，也是 Eureka 与其他注册中心最大的区别之一。本节深入剖析自我保护的触发条件、行为机制，以及如何正确配置。

## 自我保护解决的问题

### 网络抖动与误剔除

在云环境中，服务实例的健康状态可能因为以下原因暂时不可达：

| 原因 | 持续时间 | 是否健康 |
|---|---|---|
| JVM GC 暂停 | 几百毫秒到几秒 | 健康 |
| 网络短暂抖动 | 几秒到几十秒 | 健康 |
| 负载均衡器超时 | 几秒 | 健康 |
| 运维发布 | 几十秒 | 健康 |
| 机器宕机 | 持续 | 不健康 |

**问题**：如果 Eureka 在网络抖动期间严格按照「90 秒无心跳就剔除」的规则，会把大量健康的实例误剔除，导致服务雪崩。

**解决方案**：自我保护机制。当 Eureka 检测到大量心跳失败时，暂停实例剔除，等待网络恢复。

### 服务雪崩的连锁反应

```
未开启自我保护时：

1. 云平台网络抖动，持续 30 秒
2. Eureka Server 收不到 10% 实例的心跳
3. 这些实例被标记为过期并剔除
4. Eureka Server 通知所有消费者
5. 消费者更新本地缓存，路由列表减少 10%
6. 剩余实例承受更大压力
7. 部分实例也因为过载而心跳超时
8. 服务雪崩
```

## 触发机制详解

### renewalPercentThreshold

Eureka 通过 `renewalPercentThreshold`（续约百分比阈值）判断是否进入自我保护：

```
触发条件：
实际续约次数 < 期望续约次数 × renewalPercentThreshold
```

**期望续约次数的计算**：

```java
// 每分钟期望的续约次数
// = 总实例数 × (60 / 心跳间隔)
expectedRenewalsPerMin = instanceCount × (60 / heartbeatInterval)
```

```
示例计算：

场景：100 个 order-service 实例，心跳间隔 30 秒

期望续约次数：
= 100 × (60 / 30)
= 100 × 2
= 200 次/分钟

触发自我保护（阈值 0.85）：
当实际续约次数 < 200 × 0.85 = 170 次/分钟时
触发自我保护模式
```

### 续约率计算

Eureka Server 内部维护了一个滑动窗口来统计续约次数：

```java
// 续约统计
public class AbstractInstanceRegistry {

    // 最近 15 分钟的续约次数
    private final Ticker counter = Ticker.systemTicker();

    // 每分钟更新一次
    private final AtomicLong expectedHeartbeatRate
        = new AtomicLong();

    // 自我保护触发判断
    private boolean isLeaseExpirationEnabled() {
        if (!serverConfig.shouldEnableSelfPreservation()) {
            return true; // 未开启自我保护，直接剔除
        }

        // 获取最近 15 分钟的续约次数
        long latestHeartbeats = registry.entityStore().getLatestHeartbeats();
        // 获取期望的续约次数
        long expectedHeartbeats = expectedHeartbeatRate.get() * 15;

        // 如果实际续约次数低于期望的阈值，进入自我保护
        return latestHeartbeats >= expectedHeartbeats * serverConfig.getRenewalPercentThreshold();
    }
}
```

### 阈值调整的影响

| `renewalPercentThreshold` | 触发敏感性 | 适用场景 |
|---|---|---|
| 0.9（更敏感） | 更容易触发保护 | 网络不稳定的环境 |
| 0.85（默认） | 适中 | 一般生产环境 |
| 0.7（不敏感） | 难以触发保护 | 网络非常稳定的环境 |

```yaml
# 高敏感度配置（容易触发保护）
eureka:
  server:
    renewal-percent-threshold: 0.9

# 低敏感度配置（难以触发保护）
eureka:
  server:
    renewal-percent-threshold: 0.7
```

## 自我保护的行为

### 触发后的表现

```
Eureka Dashboard 显示：

⚠️ EMERGENCY! EUREKA MAY BE INCORRECT.
Self-preservation mode is turned ON.
This means that the instance will not be expired
if the number of renewals falls below threshold.
```

| 行为 | 说明 |
|---|---|
| **停止剔除实例** | 即使实例超过 90 秒未心跳，也不会被剔除 |
| **注册仍然可用** | 新实例可以正常注册 |
| **发现仍然可用** | 消费者可以获取到实例列表 |
| **续约继续记录** | Eureka 继续统计续约次数 |

### 退出自我保护

```
退出条件：
实际续约次数 >= 期望续约次数 × renewalPercentThreshold

退出行为：
1. Eureka 重新开始剔除过期实例
2. Dashboard 上的警告消失
3. 系统恢复正常
```

### 动态阈值调整

Eureka 支持动态调整自我保护阈值：

```java
// 通过 Admin API 调整阈值
PUT /eureka/server/renewalPercentThreshold
{
  "value": 0.8
}
```

## 配置指南

### 生产环境推荐配置

```yaml
eureka:
  server:
    # 开启自我保护（默认 true）
    enable-self-preservation: true

    # 续约百分比阈值（默认 0.85）
    # 网络不稳定时建议适当提高
    renewal-percent-threshold: 0.85

    # 剔除过期实例的时间间隔（默认 60 秒）
    eviction-interval-timer-in-ms: 60000

    # 续约超时时间（默认 30 秒）
    wait-time-in-ms-when-sync-empty: 30

  instance:
    # 心跳间隔（默认 30 秒）
    lease-renewal-interval-in-seconds: 30

    # 续约持续时间（默认 90 秒）
    lease-expiration-duration-in-seconds: 90
```

### 测试环境配置

```yaml
# 测试环境可以关闭自我保护，加快实例剔除
eureka:
  server:
    enable-self-preservation: false
    eviction-interval-timer-in-ms: 5000  # 每 5 秒检查一次
```

### Kubernetes 环境配置

在 Kubernetes 中，由于容器随时可能被重建，建议**关闭自我保护**：

```yaml
eureka:
  server:
    # Kubernetes 中容器重建频繁，关闭自我保护
    enable-self-preservation: false

    # 更快的剔除间隔
    eviction-interval-timer-in-ms: 5000
```

```
原因：
- Kubernetes 中 Pod 可能被杀掉并重建
- 重建后的 Pod 会有新的 IP，但 instanceId 可能相同
- 如果开启自我保护，旧实例不会被剔除，导致冲突
```

## 常见问题

### Q：自我保护触发后，服务调用会出问题吗？

不会。自我保护只是**停止剔除过期实例**，不会影响正常的注册和发现。消费者仍然能获取到实例列表，只是这个列表可能包含一些已经不健康的实例。

### Q：自我保护期间，调用不健康的实例会失败吗？

可能会失败。消费者从 Eureka 获取到不健康实例的地址后，发起请求时如果实例不可达，会失败。这是**有损降级**——牺牲一小部分请求的成功率，换取整体服务的稳定性。

### Q：如何判断应该开启还是关闭自我保护？

| 环境 | 推荐配置 | 理由 |
|---|---|---|
| 一般生产环境 | 开启 | 防止网络抖动导致误剔除 |
| 服务实例很少 | 关闭 | 实例少，自我保护反而有副作用 |
| Kubernetes 环境 | 关闭 | 容器重建频繁，自我保护会阻碍清理 |
| 网络不稳定环境 | 开启 | 需要自我保护来应对抖动 |

### Q：自我保护触发时，有什么告警措施？

```java
// Eureka Server 提供了自我保护状态变更的监听器
@Component
public class SelfPreservationListener {

    @EventListener
    public void onSelfPreservationChange(SelfPreservationChangedEvent event) {
        if (event.isEntering()) {
            // 进入自我保护，发送告警
            alertService.sendAlert("Eureka 进入自我保护模式");
        } else {
            // 退出自我保护
            alertService.sendAlert("Eureka 退出自我保护模式");
        }
    }
}
```

## 总结

| 概念 | 说明 |
|---|---|
| **自我保护** | 防止因网络抖动导致误剔除健康实例 |
| **触发条件** | 实际续约次数 `<` 期望续约次数 × 阈值 |
| **触发后行为** | 停止剔除过期实例 |
| **退出条件** | 续约次数恢复到阈值以上 |
| **默认阈值** | 0.85（期望续约次数的 85%） |
| **生产建议** | 一般生产环境开启自我保护 |
