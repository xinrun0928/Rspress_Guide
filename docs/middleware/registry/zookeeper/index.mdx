# ZooKeeper

ZooKeeper 是 Apache 基金会旗下的分布式协调服务，最初诞生于 Hadoop 生态，用于解决 HDFS NameNode 的单点问题。如今 ZooKeeper 已经成为分布式系统中最经典的协调组件，被广泛应用于配置管理、命名服务、分布式锁、Master 选举等场景。

## ZooKeeper 的设计哲学

ZooKeeper 的核心设计哲学可以用一句话概括：**「简单、高效、可靠」**。

它没有追求大而全的功能，而是专注于做好一件事——**提供一种高可靠、有序、原子性的数据管理能力**。在此基础上，开发者可以构建各种分布式应用。

## 核心特性

### 层次化命名空间

ZooKeeper 的数据存储采用类似文件系统的树形结构，每个节点叫 **ZNode**，路径用斜杠分隔：

```
/registry
  /service-a
    /instance-1    (ip: 192.168.1.10:8080)
    /instance-2    (ip: 192.168.1.11:8080)
  /service-b
    /instance-1    (ip: 192.168.1.20:8080)
/config
  /database
    (maxConnections: 100)
```

这种层次化结构天然适合表达服务树、配置树，非常直观。

### 高可靠集群

ZooKeeper 以集群模式部署，通过 **ZAB 协议**（ZooKeeper Atomic Broadcast）保证集群中所有节点的数据一致性。

一个典型的 ZooKeeper 集群由 **2n+1 个节点**组成，能够容忍 **n 个节点故障**。例如，3 节点集群可以容忍 1 个节点故障，5 节点集群可以容忍 2 个节点故障。

### 一次性 Watch 机制

客户端可以向 ZooKeeper 注册 Watch，当被监控的 ZNode 发生变更时，ZooKeeper 会主动推送通知给所有订阅的客户端。

这个机制是实现配置变更推送、服务上下线通知的基础。

## ZooKeeper 能做什么

| 场景 | 实现方式 |
|---|---|
| **配置中心** | 将配置存储在 ZNode 中，通过 Watch 感知变更 |
| **服务注册与发现** | 将服务实例注册为临时节点，消费者 Watch 获取实例列表 |
| **分布式锁** | 利用临时顺序节点，实现互斥锁、读写锁 |
| **Master 选举** | 所有节点竞争创建同一个 ZNode，成功者成为 Master |
| **分布式队列** | 利用顺序节点实现 FIFO 队列 |
| **分布式Barrier** | 所有节点在屏障节点下等待，满足条件后同时放行 |

## 为什么选择 ZooKeeper

**优点：**
- 经过大规模生产环境验证，稳定性极高
- ZAB 协议保证强一致性，数据不会丢失
- Watch 机制天然支持配置变更推送
- 生态系统丰富，很多分布式框架（Kafka、HBase、Dubbo）都以 ZooKeeper 为依赖

**缺点：**
- 运维成本较高，需要专业的 ZooKeeper 集群维护
- 不支持跨数据中心部署，不适合全球化的分布式系统
- 临时节点依赖客户端心跳，客户端异常断开时节点自动删除
- 社区活跃度不如新兴的 Nacos、Consul
