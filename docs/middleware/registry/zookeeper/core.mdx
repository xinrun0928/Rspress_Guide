# ZooKeeper 核心概念

要深入理解 ZooKeeper，必须先弄清楚它的几个核心概念：ZNode 的类型与结构、版本号机制、ACL 权限控制、Watch 触发模式，以及保证集群一致性的 ZAB 协议。这些概念构成了 ZooKeeper 的基石，也是一切高级应用（如分布式锁、Master 选举）得以实现的前提。

## ZNode：数据存储的基本单元

ZooKeeper 的数据模型是一个树形结构，每个节点称为 **ZNode**。每个 ZNode 可以存储少量数据（默认上限 1MB），并拥有自己的元信息。

### 四种 ZNode 类型

ZooKeeper 支持四种类型的 ZNode，这个分类直接影响节点的生命周期和适用场景：

| 类型 | 创建命令 | 生命周期 | 说明 |
|---|---|---|---|
| **持久节点** | `create /path data` | 显式删除前一直存在 | 最常用的节点类型 |
| **持久顺序节点** | `create -s /path data` | 显式删除前一直存在 | 节点名自动追加递增序号 |
| **临时节点** | `create -e /path data` | 客户端会话结束或断连时自动删除 | 用于实现服务注册、健康检查 |
| **临时顺序节点** | `create -s -e /path data` | 客户端会话结束或断连时自动删除 | 节点名自动追加递增序号 |

**临时节点的自动删除机制**：ZooKeeper 会为每个客户端维护一个 Session，当客户端的心跳超时（默认 10 倍的 tickTime，约 20 秒）时，ZooKeeper 会认为该客户端已经断开连接，随即删除该客户端创建的所有临时节点。这个特性使得临时节点非常适合用于**服务注册**——当服务实例宕机时，其注册信息自动消失。

**临时节点不能有子节点**：这是一个重要的约束。原因在于：临时节点在会话断连时会被自动清理，如果临时节点有子节点，那么 ZooKeeper 需要递归删除整个子树，这会带来复杂的边界问题。

### 节点状态（Stat）结构

每个 ZNode 除了存储的数据（data）外，还包含一组元信息，称为 Stat：

```java
// ZooKeeper 节点状态的核心字段
public class Stat {
    long czxid;        // 创建时的 ZXID（事务 ID）
    long mzxid;        // 最后修改时的 ZXID
    long ctime;        // 创建时间（毫秒）
    long mtime;        // 最后修改时间（毫秒）
    int version;       // 数据版本号（每修改一次 +1）
    int cversion;      // 子节点版本号（每增删一个子节点 +1）
    int aversion;      // ACL 版本号（每修改 ACL 一次 +1）
    long ephemeralOwner; // 临时节点的会话 ID（持久节点为 0）
    int dataLength;    // 数据长度
    int numChildren;   // 子节点数量
}
```

## 版本号机制：乐观锁的基石

ZooKeeper 的版本号是一种乐观并发控制机制，支持三种版本：`version`（数据版本）、`cversion`（子节点版本）、`aversion`（ACL 版本）。这里重点介绍使用最多的 `version`。

### 什么是 version

`version` 是 ZNode 数据内容的版本号，每次对节点数据执行写入操作时，`version` 会自动加 1。

```bash
# 创建节点，初始 version = 0
create /config "max=100"

# 获取节点信息，version = 0
get /config

# 设置数据时指定 version
set /config "max=200" -v 1   # 成功，仅当 version == 1 时才写入
```

### 版本号的用途

**CAS 操作（Compare-And-Swap）**：这是版本号最重要的应用。当多个客户端同时修改同一个 ZNode 时，可以使用版本号确保原子性：

```java
// 伪代码：基于版本的原子更新
Stat stat = new Stat();
while (true) {
    byte[] data = zk.getData(path, false, stat);
    int currentVersion = stat.getVersion();

    // 构造新数据
    String newData = transform(new String(data));

    // 仅当版本匹配时更新成功
    Stat newStat = zk.setData(path, newData.getBytes(), currentVersion);
    if (newStat.getVersion() == currentVersion + 1) {
        break; // 更新成功，退出循环
    }
    // 否则版本已被其他客户端修改，重试
}
```

**这有什么用？** 在分布式锁实现中，可以防止误删他人的锁节点——删除时检查当前版本是否与获取时的版本一致，一致才允许删除。

### version 的限制

:::warning
版本号只能保证「在同一时刻没有被其他事务修改」，但无法防止「ABA 问题」。如果一个值从 A 变成 B 再变回 A，且在变回 A 的瞬间版本号也恰好恢复到原值，ZooKeeper 会认为没有冲突。这种场景下需要引入额外的机制（如基于 ZXID 或时间戳）来解决。
:::

## ACL 权限控制：细粒度的访问控制

ZooKeeper 的 ACL（Access Control List）机制提供了**细粒度的权限控制**。每个 ZNode 都可以独立设置 ACL，权限控制采用 **Scheme + ID + Permission** 的三元组模型。

### 三种权限

| 权限 | 简写 | 说明 |
|---|---|---|
| `READ` | `r` | 允许读取节点数据和子节点列表 |
| `WRITE` | `w` | 允许写入节点数据 |
| `CREATE` | `c` | 允许创建子节点 |
| `DELETE` | `d` | 允许删除子节点 |
| `ADMIN` | `a` | 允许设置 ACL |

注意：`WRITE` 权限控制的是「修改节点数据」，而不是「创建或删除节点本身」。创建子节点需要 `CREATE`，删除子节点需要 `DELETE`。

### 四种认证 Scheme

| Scheme | ID 含义 | 典型场景 |
|---|---|---|
| `world` | 只有一个 ID：`anyone` | 默认权限，所有人可访问 |
| `auth` | 不需要 ID | 将认证用户加入 ACL 列表 |
| `digest` | `username:password`（Base64 编码） | 用户名密码认证 |
| `ip` | IP 地址或 CIDR 网段 | IP 白名单 |
| `x509` | X509 证书的 DN（Distinguished Name） | SSL/TLS 客户端认证 |

```bash
# world 方式：所有用户可读可写（默认）
setAcl /path world:anyone:crwda

# ip 方式：仅 192.168.1.0/24 网段可访问
setAcl /path ip:192.168.1.0/24:cdrwa

# digest 方式：用户名密码认证
setAcl /path digest:alice:5BsAY3...:rwcda
```

### 权限继承与覆盖

子节点**默认继承父节点的 ACL**，但可以通过 `setAcl` 单独为子节点设置新的权限。权限设置后立即生效，不影响已认证的会话。

## Watch 机制：事件驱动的核心

Watch 是 ZooKeeper 实现配置变更推送、服务上下线通知的基础。它的核心特点是**一次性触发 + 异步推送**。

### Watch 的注册与触发

Watch 可以在读取操作（`getData`、`getChildren`、`exists`）时同时注册：

```java
// 监听节点数据变化
zk.getData("/config", event -> {
    System.out.println("数据变更，新值：" + new String(event.getData()));
}, null);

// 监听子节点变化（新增/删除）
zk.getChildren("/services", event -> {
    System.out.println("子节点变更：" + event.getType());
}, null);

// 监听节点是否存在
zk.exists("/lock", event -> {
    System.out.println("节点状态变化：" + event.getType());
}, null);
```

### Watch 触发的时机

| 读取操作 | Watch 触发的事件类型 | 触发条件 |
|---|---|---|
| `exists` | `NodeCreated` / `NodeDeleted` | 节点创建 / 删除 |
| `exists` | `NodeDataChanged` | 节点数据更新 |
| `getData` | `NodeDeleted` / `NodeDataChanged` | 节点删除 / 数据更新 |
| `getChildren` | `NodeChildrenChanged` | 子节点列表变更（新增/删除） |

### Watch 的关键特性

**特性一：一次性触发**

Watch 触发后会自动失效。如果想持续监听，必须在收到通知后**重新注册** Watch：

```java
// 正确的持续监听模式
private void watchConfig(String path) {
    try {
        Stat stat = new Stat();
        byte[] data = zk.getData(path, event -> {
            System.out.println("收到变更通知，重新注册 Watch");
            watchConfig(path); // 重新注册
        }, stat);
        process(data);
    } catch (KeeperException e) {
        // 处理异常
    }
}
```

**特性二：串行执行回调**

同一个客户端的 Watch 回调是串行执行的，按注册顺序依次调用。如果某个回调执行时间过长，会阻塞后续回调。因此 Watch 回调中应尽量避免耗时操作，或者将耗时逻辑放到单独的线程池中。

**特性三：异步推送，有序性**

Watch 事件通过 ZooKeeper 客户端的 **SendThread** 异步推送给业务线程。事件本身是有序的（FIFO），但与读取操作的结果之间没有因果关系——读取到最新数据和收到变更通知之间可能存在短暂的时间差。

**特性四：轻量级设计**

Watch 事件只包含变更的类型和路径，不包含变更的具体内容。如果需要最新数据，需要主动调用 `getData` 获取。

:::tip
**Watch 回调中不要做会触发新 Watch 的操作**。例如，在 `getChildren` 的回调中调用 `getChildren` 并注册新 Watch 是安全的，但如果在回调中调用 `getData`，可能会出现死锁风险。ZooKeeper 保证回调按顺序执行，但在某些边界情况下可能会抛出 `WatchRemovedException`。
:::

## ZAB 协议：集群一致性的保障

ZAB（ZooKeeper Atomic Broadcast）是 ZooKeeper 的核心一致性协议，全称 **ZooKeeper Atomic Broadcast**。它保证了 ZooKeeper 集群中所有节点的数据一致性和故障恢复能力。

### 两种运行模式

ZAB 协议有两种主要的运行模式：**消息广播模式**和**崩溃恢复模式**。

**消息广播模式**是 ZooKeeper 正常运行时的状态。当 Leader 节点收到客户端的写请求时，会执行以下流程：

```
1. Leader 将请求包装为 Proposal（提议），带上全局唯一的事务 ID（ZXID）
2. Leader 向所有 Follower 发送 Proposal
3. Follower 收到 Proposal 后，写入本地事务日志，并向 Leader 发送 ACK
4. 当 Leader 收到大多数（Quorum）Follower 的 ACK 后，向所有 Follower 发送 COMMIT
5. Leader 提交本地事务，向客户端响应成功
```

**ZXID（ZooKeeper Transaction ID）** 是一个 64 位整数，高 32 位是 **epoch**（任期编号），低 32 位是 **事务计数器**。ZXID 的大小关系隐含了因果顺序——如果事务 A 的 ZXID 小于事务 B 的 ZXID，那么 A 一定在 B 之前发生。

**崩溃恢复模式**发生在 Leader 故障时。ZooKeeper 需要重新选举出一个新的 Leader，并确保新 Leader 拥有所有已提交的事务。

### 崩溃恢复的三个保证

当 Leader 崩溃后，集群进入恢复模式，需要完成以下三个目标：

**保证一：已提交的事务不能丢失**

Leader 在收到大多数 Follower 的 ACK 后才发送 COMMIT，这意味着只有被大多数节点确认的事务才被认为是已提交的。如果 Leader 在发送 COMMIT 前崩溃，这些事务会被新的 Leader 丢弃。

**保证二：未提交的事务不能被应用**

如果 Leader 在收到 ACK 后、发送 COMMIT 前崩溃，这个事务在新的 Leader 上不会被应用。因为这个事务从未被大多数节点确认，新的 Leader 会直接跳过它。

**保证三：新 Leader 的数据最新**

新选举出的 Leader 必须是所有已提交事务的合集。ZAB 协议通过以下方式保证这一点：

- Leader 选举时，投票中包含该节点的最大 ZXID
- 当两个节点竞争时，ZXID 大的节点获胜（因为它包含更多已提交的事务）
- 新 Leader 上任后，会先同步 Follower 的事务日志，补充缺失的事务

### Leader 选举过程

Leader 选举使用 **Fast Leader Election（FLE）** 算法，基于 **服务器 ID + 最大 ZXID + 逻辑时钟** 三元组进行比较：

```
选举轮次（electionEpoch）相同 → 比较 ZXID → ZXID 大的胜出
ZXID 相同 → 比较服务器 ID → ID 大的胜出
```

投票的变迁过程：
1. 每个服务器启动时都投票给自己，将投票信息广播给其他服务器
2. 收到投票后，与自己的投票比较，判断是否需要更换自己的投票
3. 如果某个服务器获得了超过半数（Quorum）的投票，成为 Leader
4. 其他服务器更新自己的状态为 FOLLOWING 或 LEADING

### ZAB 与 Paxos 的区别

很多人会把 ZAB 与 Paxos 做对比。两者都用于解决分布式一致性问题，但设计目标和使用场景不同：

| 维度 | ZAB | Paxos |
|---|---|---|
| **设计目标** | 主备复制，专为 ZooKeeper 设计 | 通用一致性协议 |
| **主节点产生** | 通过选举产生 | 通过 Proposer 提议 |
| **写请求处理** | 必须通过 Leader | 任何节点都可能提议 |
| **活锁问题** | 无（单一 Leader） | 可能出现活锁 |
| **适用场景** | ZooKeeper 集群 | 分布式锁、状态机复制 |

ZAB 的核心优势是：**通过单一 Leader 处理所有写请求，避免了活锁问题**，同时通过 Quorum 机制保证了集群的可用性。

## 总结

ZooKeeper 的核心概念形成了一套完整的数据管理能力：

- **ZNode** 提供了层次化的数据存储结构，四种类型满足了不同生命周期需求
- **版本号** 实现了乐观锁，为分布式并发控制提供了基础
- **ACL** 提供了细粒度的权限控制，保障了数据安全
- **Watch** 实现了事件驱动，是配置变更推送和服务发现的基石
- **ZAB 协议** 保证了集群在各种故障场景下的数据一致性和可用性

理解这些核心概念，是掌握 ZooKeeper 高级应用（如分布式锁、Master 选举）的前提。
