# ZooKeeper 集群脑裂问题与过半机制

分布式系统中最危险的场景之一，是**网络分区（Network Partition）**——由于网络故障，原本完整的集群被分割成多个互相隔离的小组，每个小组都认为自己是「完整的集群」，各自运行。

这种「一个集群变成多个孤立集群」的现象，就是**脑裂（Split-Brain）**。

ZooKeeper 通过**过半机制（Quorum）**来解决脑裂问题，保证在任何网络分区情况下，最多只有一个分区能继续提供服务。

## 什么是脑裂

### 生活中的类比

想象一个三人组成的小组，正在用对讲机讨论决策。突然，电磁干扰导致三人之间的通信断开：

- 甲和乙能互相通话，但听不到丙
- 丙也能听到自己

现在的问题是：**谁能做决策？**

- 如果三个人都认为自己是「完整的三人组」，各自做决策，就会产生冲突
- 正确的做法是：只有获得大多数人（至少 2 人）支持的决策才有效

这就是 ZooKeeper 过半机制的核心思想。

### 脑裂的危害

脑裂会导致以下严重后果：

| 危害 | 说明 |
|---|---|
| **双主问题** | 两个节点都认为自己是 Leader，同时处理写请求，数据不一致 |
| **锁失效** | 同一把锁被两个客户端同时持有 |
| **数据冲突** | 两个分区同时写入，数据产生冲突 |
| **状态混乱** | 消费者订阅了错误的节点，消息路由错误 |

### 常见的脑裂场景

- **网络设备故障**：交换机、路由器故障导致网络分区
- **机房断电**：部分机柜断电，对应节点失联
- **GC 停顿**：JVM GC 时间过长，被其他节点误判为宕机
- **配置错误**：防火墙、路由规则配置错误导致节点无法通信

## 过半机制：ZooKeeper 的解决方案

### 什么是过半

ZooKeeper 要求**写请求必须获得集群中超过半数节点的确认**，才能被认为成功。

| 集群节点数 | 过半数量（Quorum） | 容忍故障节点数 |
|---|---|---|
| 1 | 1 | 0 |
| 2 | 2 | 0 |
| 3 | 2 | 1 |
| 4 | 3 | 1 |
| 5 | 3 | 2 |
| 6 | 4 | 2 |
| 7 | 4 | 3 |

**关键结论**：容忍 n 个节点故障，需要 `2n + 1` 个节点的集群。

### 过半机制如何防止脑裂

```
场景：3 节点集群，节点 A、B、C

正常状态：
- A 是 Leader，B、C 是 Follower
- 任何写请求需要获得至少 2 个节点确认

网络分区场景 1：A 和 B 能通信，但与 C 断连
- 分区 1（A、B）：2 个节点，不满足过半（需要 2），无法处理写请求
- 分区 2（C）：1 个节点，不满足过半，无法处理写请求
- 结果：整个集群停止写入，避免脑裂 ✅

网络分区场景 2：A 能与 B、C 通信，但 B 和 C 之间断连
- 实际上只有 1 个分区，不存在脑裂问题

网络分区场景 3：A、B、C 完全分裂成三个单节点
- 三个分区均只有 1 个节点，都无法满足过半条件
- 整个集群停止写入 ✅
```

**核心保证**：在任何网络分区下，**最多只有一个分区满足过半条件**，因此最多只有一个分区能继续处理写请求。

### 为什么不用「多数派」

有人会问：为什么不用「超过半数」而不是「至少半数」？

```
如果要求「超过半数」：5 节点集群需要 3 个节点确认
如果要求「至少半数」：5 节点集群只需要 2 个节点确认
```

实际上 ZooKeeper 使用的就是「至少半数」——只要获得 `floor(N/2) + 1` 个节点确认即可。「过半」只是一个更通俗的说法。

## 奇数节点 vs 偶数节点

### 结论：奇数节点更优

在容忍相同数量故障的前提下，**奇数节点的集群更优**：

| 节点数 | 过半数量 | 容忍故障 | 有效节点占比 |
|---|---|---|---|
| 3 | 2 | 1 | 67% |
| 4 | 3 | 1 | 75% |
| 5 | 3 | 2 | 60% |
| 6 | 4 | 2 | 67% |
| 7 | 4 | 3 | 57% |

### 为什么 2n 和 2n+1 的容忍度相同

```
4 节点集群（需要 3 个节点确认）：
- 容忍 1 个节点故障，剩余 3 个节点可以写入
- 容忍 2 个节点故障，剩余 2 个节点无法写入

5 节点集群（需要 3 个节点确认）：
- 容忍 2 个节点故障，剩余 3 个节点可以写入
- 容忍 3 个节点故障，剩余 2 个节点无法写入
```

**4 节点和 5 节点能容忍的最大故障数都是 2**。但 5 节点集群比 4 节点集群多了一个可用节点。

### 最佳实践

| 建议 | 说明 |
|---|---|
| **节点数为奇数** | 优先选择 3、5、7 |
| **避免偶数节点** | 偶数节点不带来额外好处，反而浪费资源 |
| **节点分布合理** | 跨机架、跨机房部署，避免单点故障 |
| **不超过 7 个节点** | ZooKeeper 的性能随节点数增加而下降，建议 3-7 个 |

## ZAB 协议与数据一致性

### Leader 的数据权威性

ZooKeeper 的 ZAB 协议保证了**只有 Leader 才能处理写请求**。这是通过以下机制实现的：

1. **Leader 选举时选择最新数据**：ZXID 最大的节点成为 Leader，保证新 Leader 拥有所有已提交的事务
2. **Follower 只同步 Leader 数据**：Follower 收到写请求后，转发给 Leader，由 Leader 广播给所有节点
3. **过半确认后才提交**：Leader 收到超过半数节点的 ACK 后，才提交事务并响应客户端

### 崩溃恢复中的数据一致性

当 Leader 崩溃后，新 Leader 需要确保自己拥有所有已提交的事务：

```
场景：
- 5 节点集群，A 是 Leader
- 客户端提交事务 T1，收到 3 个节点确认（A、B、C），T1 已提交
- 客户端提交事务 T2，收到 2 个节点确认（A、D），T2 未提交
- A 突然崩溃

Leader 选举：
- A 的 ZXID 包含 T1 和 T2，假设为 0x100000002
- B 的 ZXID 只包含 T1，假设为 0x100000001
- C 的 ZXID 只包含 T1，假设为 0x100000001

投票结果：
- B、C、D、E、F 都投票给 A（因为 A 的 ZXID 最大）
- A 获得 5 票，成为新 Leader

新 Leader 上任后的处理：
1. A 检查自己的事务日志，发现 T1 已提交，T2 未提交
2. A 广播 COMMIT 消息，提交 T1（已在大多数节点上）
3. A 丢弃 T2（从未被大多数节点确认）
4. B、C、D、E、F 同步 Leader 的数据

结果：T1 被提交，T2 被丢弃，数据一致 ✅
```

### 两阶段提交与 ZAB 的区别

ZAB 的消息广播模式可以看作一种简化的两阶段提交：

| 阶段 | 两阶段提交 | ZAB 广播 |
|---|---|---|
| Phase 1 | Coordinator 发送 Prepare，Participants 投票 | Leader 发送 Proposal，Follower 返回 ACK |
| Phase 2 | Coordinator 收到全部 ACK 后 Commit | Leader 收到过半 ACK 后 Commit |

ZAB 的优势在于**只需等待过半确认**，而不需要等待所有节点，因此避免了单点故障导致的阻塞。

## 网络分区时的读写策略

### 读请求的处理

ZooKeeper 默认情况下，**读请求可以在任何节点上执行**，返回的是该节点的本地数据。这保证了读请求的高可用性，但**不保证线性一致性**。

```
场景：
- 5 节点集群，Leader 是 A
- 客户端 1 向 A 写入 X=1，收到过半确认，提交成功
- 在消息传播到 B 之前，客户端 2 从 B 读取 X，返回的是旧值（X 不存在）

结论：ZooKeeper 的读请求是**非强一致性**的。
```

### 强制读取 Leader

如果业务需要强一致性读，可以通过 `sync()` 命令强制同步：

```java
// 普通读，可能读到过期数据
byte[] data = zk.getData("/config", false, null);

// 强制同步后再读，保证读到最新数据
zk.sync(path);
byte[] data = zk.getData("/config", false, null);
```

### 网络分区时的读写

```
5 节点集群，网络分区：

分区 1（A、B、C）：3 个节点，过半 ✅
- 读请求：在任意节点读，返回本地数据
- 写请求：转发给 Leader 处理，正常工作

分区 2（D、E）：2 个节点，不过半 ❌
- 读请求：在任意节点读，返回本地数据
- 写请求：无法获得过半确认，请求失败

结果：只有分区 1 能正常处理读写，分区 2 只能读取本地数据
```

## 总结

| 概念 | 说明 |
|---|---|
| **脑裂** | 网络分区导致多个节点都认为自己是主节点 |
| **过半机制** | 写请求需要获得超过半数节点确认才能成功 |
| **防止脑裂** | 任何分区下，最多只有一个分区满足过半条件 |
| **奇数更优** | 2n 和 2n+1 容忍度相同，但奇数节点更少 |
| **ZAB 协议** | 保证 Leader 拥有最新数据，新 Leader 从已提交事务恢复 |
| **读写策略** | 读可在任意节点执行，写必须经过 Leader |
