# Maven 与 Gradle 对比

Maven 和 Gradle 是 Java 生态最主流的两大构建工具。Google 选择 Gradle 作为 Android 官方构建工具，Spring Framework 在 2022 年从 Gradle 迁回 Maven，Apache Kafka、Elasticsearch 却坚定使用 Gradle。不同的选择背后，是不同的权衡。

这一篇，我们从实际项目出发，深度对比这两个工具。

## 核心差异：构建模型

理解 Maven 和 Gradle 的差异，首先要理解它们的核心模型。

**Maven 的模型是声明式的**。你声明「我需要什么依赖」「我需要什么插件」，Maven 按照固定的生命周期执行。配置即结果，你几乎不需要关心「怎么做」。

**Gradle 的模型是任务式的**。你定义 Task，Task 有输入和输出，Task 之间形成依赖图，Gradle 按依赖顺序执行。增量构建、智能缓存都建立在这个模型之上。

两种模型各有优劣。声明式简单直观，但灵活性受限；任务式灵活强大，但需要更多学习成本。

## 构建速度：数字说话

让我们用一个具体场景来测试：修改一个类的源代码后重新构建。

### Maven 构建过程

```bash
# 清理 + 打包
mvn clean package
```

Maven 的构建流程是线性的：

1. 解析 POM，计算依赖图
2. 下载依赖到本地仓库
3. 编译所有源代码
4. 运行测试
5. 打包产物

每次构建，Maven 都会重新编译所有代码，即使只改了一行。这是 Maven 被诟病最多的地方。

### Gradle 构建过程

```bash
# 增量构建（默认行为）
gradle build
```

Gradle 的构建流程是非线性的：

1. 计算任务依赖图
2. 检查任务输入输出变化
3. 只执行受影响的 Task
4. 复用缓存的结果

对于只修改一个类的场景，Gradle 的增量构建机制可以跳过 80% 以上的编译工作。

### 实测数据

在一个 100+ 模块的真实项目中（数据来自 Gradle 官方博客）：

| 操作 | Maven | Gradle | 加速比 |
|---|---|---|---|
| 增量编译 | 全量编译 | 智能增量 | 3-10x |
| 并行构建 | 需配置 | 默认支持 | 2-4x |
| 构建缓存 | 无 | 支持 | 首次后 10x+ |
| Daemon | 无 | 内置 | 2-3x |

对于大型项目，Gradle 的构建速度优势是决定性的。Android Studio 从 Maven 迁移到 Gradle 后，Google 宣称构建速度提升了 50%。

:::tip 速度的代价
Gradle 速度快的代价是更高的资源消耗和更复杂的调试过程。如果构建出现问题，Gradle 的错误堆栈往往比 Maven 更长。
:::

## 依赖管理：两种哲学

### Maven 的依赖管理

Maven 通过 `pom.xml` 声明依赖，依赖管理基于**传递依赖（Transitive Dependency）**机制。

```xml title="pom.xml"
<dependencies>
    <!-- 直接依赖 -->
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-web</artifactId>
        <version>2.7.0</version>
    </dependency>

    <!-- 依赖范围 -->
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-test</artifactId>
        <version>2.7.0</version>
        <scope>test</scope>
    </dependency>
</dependencies>
```

Maven 的依赖通过 `scope` 控制可见性：`compile`（默认）、`provided`、``runtime``、``test`` 等。

### Gradle 的依赖配置

Gradle 使用**配置（Configuration）**来组织依赖，引入了更细粒度的控制：

```groovy title="build.gradle"
dependencies {
    // 编译时和运行时都可见
    implementation 'org.springframework.boot:spring-boot-starter-web:2.7.0'

    // 编译时可见，运行时不可见（推荐）
    // 好处：依赖变更不会影响其他模块
    implementation 'com.google.guava:guava:31.1-jre'

    // 只在运行时可见，不参与编译
    runtimeOnly 'org.postgresql:postgresql:42.5.0'

    // 只在测试时可见
    testImplementation 'org.springframework.boot:spring-boot-starter-test:2.7.0'
}
```

`implementation` 和 `compile`（已废弃）的区别：

- `compile`：依赖对所有使用者可见
- `implementation`：依赖只对当前模块可见

对于大型多模块项目，`implementation` 可以显著减少重新编译的范围。

### 依赖冲突解决

**Maven 的最近定义优先（Nearest Definition）策略**：

```
A -> B -> C -> D:log4j:1.2.8
E -> F -> D:log4j:1.2.11
```

Maven 会选择路径最短的版本，即 `1.2.8`。如果路径长度相同，选择 POM 中先声明的版本。

**Gradle 的冲突解决策略**：

```groovy title="build.gradle"
configurations.all {
    resolutionStrategy {
        // 强制使用特定版本
        force 'org.slf4j:slf4j-api:1.7.36'

        // 升级传递依赖
        prefer 'org.springframework:spring-core:5.3.20'
    }
}
```

Gradle 提供了更灵活的控制，可以精确指定每个依赖的版本。

## 插件生态：成熟 vs 丰富

### Maven 插件

Maven 的插件生态非常成熟，官方维护了大量核心插件：

| 插件 | 用途 |
|---|---|
| `maven-compiler-plugin` | 编译 Java 代码 |
| `maven-surefire-plugin` | 运行单元测试 |
| `maven-jar-plugin` | 打包 JAR |
| `maven-shade-plugin` | 创建 Fat JAR |
| `maven-assembly-plugin` | 自定义打包结构 |
| `maven-release-plugin` | 版本发布 |

Maven 插件通过 `<configuration>` 配置，约定大于配置，配置项相对固定。

### Gradle 插件

Gradle 插件分为**核心插件**和**社区插件**：

```groovy title="build.gradle"
plugins {
    // 核心插件
    id 'java'
    id 'application'

    // 社区插件
    id 'org.springframework.boot' version '2.7.0'
    id 'com.github.johnrengelman.shadow' version '7.1.2'
}
```

Gradle 插件的灵活性更高，可以完全自定义 Task 行为。

## 学习曲线：陡峭 vs 平缓

### Maven 的学习曲线

Maven 的学习曲线相对平缓：

1. 了解标准项目结构
2. 学习 `pom.xml` 基本语法
3. 掌握常用命令 `mvn clean package`
4. 学会查官方文档找插件配置

大多数开发者一周内可以上手 Maven。

### Gradle 的学习曲线

Gradle 的学习曲线更陡：

1. 理解 Task 概念和依赖图
2. 学习 Groovy 或 Kotlin DSL
3. 掌握增量构建原理
4. 理解配置阶段和执行阶段的区别

对于有 Ant 背景的开发者，Gradle 的 Task 概念更容易理解；对于新手，Maven 的声明式模型更直观。

:::warning 选择建议
如果你的团队：
- 都是 Java 新人，项目不复杂 → Maven
- 需要快速开发，不在乎构建时间 → Maven
- 大型项目，构建频繁迭代 → Gradle
- Android 开发 → 必须用 Gradle
- 需要高度自定义构建逻辑 → Gradle
:::

## 代码对比：同一需求的两种实现

### 定义项目依赖

**Maven（XML）**：

```xml title="pom.xml"
<dependencies>
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-web</artifactId>
        <version>2.7.0</version>
    </dependency>
</dependencies>
```

**Gradle（Groovy DSL）**：

```groovy title="build.gradle"
dependencies {
    implementation 'org.springframework.boot:spring-boot-starter-web:2.7.0'
}
```

**Gradle（Kotlin DSL）**：

```kotlin title="build.gradle.kts"
dependencies {
    implementation("org.springframework.boot:spring-boot-starter-web:2.7.0")
}
```

### 配置 Java 版本

**Maven**：

```xml title="pom.xml"
<properties>
    <maven.compiler.source>17</maven.compiler.source>
    <maven.compiler.target>17</maven.compiler.target>
</properties>
```

**Gradle（Groovy）**：

```groovy title="build.gradle"
java {
    sourceCompatibility = JavaVersion.VERSION_17
    targetCompatibility = JavaVersion.VERSION_17
}
```

### 跳过测试

**Maven**：

```bash
mvn package -DskipTests
```

**Gradle**：

```bash
gradle build -x test
```

## 面试追问

**面试官可能问**：Spring Framework 为什么从 Gradle 迁回 Maven？

这个问题背后有几个原因：Spring 项目极度依赖 Maven 的 BOM 机制来管理复杂的依赖版本树；Spring 团队认为 Maven 的声明式配置更适合他们的发布流程；Maven 的中央仓库和镜像生态更成熟。实际上，Spring 团队也承认 Gradle 的速度优势，但最终选择是工程权衡的结果，没有绝对的对错。

**面试官可能问**：什么情况下不选择 Gradle？

Gradle 不是银弹。在这些场景下，Maven 可能是更好的选择：团队成员普遍不熟悉 Gradle；项目规模较小，构建时间不是瓶颈；需要使用 Maven 特有的插件（如 `maven-release-plugin` 的某些高级功能）；项目对构建稳定性要求极高，Gradle 的灵活性可能带来意外问题。
