# Fat Jar 原理与实践

「为什么我的 Spring Boot 应用打包后只有一个 jar，却包含了所有依赖？」

传统的 Java 应用，依赖 jar 需要单独放在 `lib/` 目录下，主程序 jar 和依赖是分离的。但 Spring Boot 采用了一种新的打包方式——把所有依赖都打包进一个「胖」jar 里，实现了真正的独立可执行。

**这看似简单的改变，背后蕴含着精妙的设计。**

## Jar 包的秘密

### 传统 Jar 结构

```
myapp.jar (传统方式)
├── META-INF/
│   └── MANIFEST.MF          # 入口类声明
├── com/
│   └── myapp/
│       └── App.class        # 你的代码
└── lib/                     # 依赖外部放置
    ├── spring-core.jar
    ├── spring-context.jar
    └── commons-lang3.jar
```

**传统方式的问题：**

- 依赖和主程序分离，容易丢失
- 需要额外脚本管理 classpath
- 版本冲突难以排查
- 部署时容易出错（jar 放错位置）

### Spring Boot Jar 结构

```bash
myapp.jar (Spring Boot 方式)
├── META-INF/
│   └── MANIFEST.MF          # Spring Boot 入口
├── BOOT-INF/
│   ├── classes/              # 你的代码
│   │   └── com/myapp/App.class
│   └── lib/                  # 所有依赖
│       ├── spring-boot-2.7.18.jar
│       ├── spring-core-5.3.x.jar
│       └── commons-lang3-3.12.0.jar
└── org/springframework/boot/loader/  # Spring Boot 类加载器
    ├── JarLauncher.class
    └── ...
```

**关键点：**

1. **非标准 Jar 结构**：使用了非标准的 `BOOT-INF` 目录
2. **专用类加载器**：`JarLauncher` 使用特殊的类加载机制
3. **嵌套 Jar**：支持直接运行包含其他 jar 的 jar

## 嵌套 Jar 的实现原理

### 为什么普通 Jar 不能嵌套？

Java 的原生 `java.net.URLClassLoader` 只能加载文件系统中的 jar：

```
URLClassLoader.loadClass("com.example.Foo")
    ↓
尝试从 file:/path/to/myapp.jar!/BOOT-INF/lib/foo.jar 中加载
    ↓
❌ 标准 JAR FileSystemProvider 不支持 jar-in-jar
```

### Spring Boot Loader 解决方案

Spring Boot 实现了一个自定义的 `LayeredFileSystemJarFile`：

```java
// JarLauncher.java
public class JarLauncher extends ExecutableArchiveLauncher {
    @Override
    protected ClassLoader createClassLoader(URL[] urls) throws Exception {
        // 使用自定义类加载器，支持嵌套 jar
        return new LayeredJarLauncher.LaunchedURLClassLoader(urls);
    }
}
```

**工作原理：**

```
应用启动：java -jar myapp.jar
    ↓
启动 JarLauncher
    ↓
解析 BOOT-INF/classes（主代码）
解析 BOOT-INF/lib/*.jar（依赖）
    ↓
为每个嵌套 jar 创建 URL
    ↓
LaunchedURLClassLoader 加载所有类
    ↓
运行主类的 main() 方法
```

### 嵌套 jar 加载顺序

```java
// 1. BOOT-INF/classes (应用代码，优先加载)
URL classesUrl = new URL("jar:file:myapp.jar!/BOOT-INF/classes!/");

// 2. BOOT-INF/lib/*.jar (依赖，按字母顺序)
URL[] libUrls = ...;

// 3. 注册到 ClassLoader
for (URL url : libUrls) {
    classLoader.addURL(url);
}
```

## Spring Boot 打包插件

### Maven 配置

```xml
<build>
    <plugins>
        <plugin>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-maven-plugin</artifactId>
            <version>2.7.18</version>
            <executions>
                <execution>
                    <goals>
                        <goal>repackage</goal>
                    </goals>
                </execution>
            </executions>
            <configuration>
                <mainClass>com.myapp.Application</mainClass>
                <excludes>
                    <!-- 排除不需要的依赖 -->
                    <exclude>
                        <groupId>org.projectlombok</groupId>
                        <artifactId>lombok</artifactId>
                    </exclude>
                </excludes>
            </configuration>
        </plugin>
    </plugins>
</build>
```

### Gradle 配置

```kotlin
// build.gradle.kts
plugins {
    id("org.springframework.boot") version "2.7.18"
}

springBoot {
    mainClass.set("com.myapp.Application")

    // 排除 lombok（运行时不需要）
    dependsOn("compileJava") {
        exclude(group = "org.projectlombok")
    }
}

// 或者完全自定义
tasks.bootJar {
    archiveFileName.set("${project.name}.jar")
    duplicatesStrategy = DuplicatesStrategy.EXCLUDE
}
```

## Gradle Shadow 插件

如果你的项目不是 Spring Boot，可以使用 Gradle Shadow 插件实现 Fat Jar。

### 基本配置

```kotlin
// build.gradle.kts
plugins {
    id("com.github.johnrengelman.shadow") version "8.1.1"
    id("java")
}

dependencies {
    implementation("com.google.guava:guava:32.1.3-jre")
    implementation("org.apache.commons:commons-lang3:3.12.0")
}

tasks.shadowJar {
    archiveBaseName.set("myapp")
    archiveVersion.set("1.0.0")
    mergeServiceFiles()  // 合并 META-INF/services
}
```

### 高级配置

```kotlin
tasks.shadowJar {
    // 排除某些文件
    exclude("META-INF/*.SF")
    exclude("META-INF/*.DSA")
    exclude("META-INF/*.RSA")
    exclude("META-INF/LICENSE*")
    exclude("META-INF/NOTICE*")

    // 合并配置文件
    merge("META-INF/spring.handlers")
    merge("META-INF/spring.schemas")
    merge("META-INF/services/*")

    // 压缩级别
    compressResources = true
    isZip64 = false
}
```

## Fat Jar 的问题与解决方案

### 问题 1：重复资源

多个 jar 包中可能包含同名的配置文件：

```
jar A: META-INF/spring.xml
jar B: META-INF/spring.xml  ← 冲突！
```

**解决方案：**

```xml
<!-- Maven: 使用 shade 插件的资源转换器 -->
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-shade-plugin</artifactId>
    <configuration>
        <transformers>
            <transformer implementation="org.apache.maven.plugins.shade.resource.ServicesResourceTransformer"/>
            <transformer implementation="org.apache.maven.plugins.shade.resource.ManifestResourceTransformer">
                <mainClass>com.myapp.Main</mainClass>
            </transformer>
        </transformers>
    </configuration>
</plugin>
```

### 问题 2：Service Loader 冲突

`META-INF/services` 中的服务发现配置可能被覆盖：

```properties
# META-INF/services/javax.ws.rs.ext.RuntimeDelegate
# 如果多个 jar 都声明了这个文件，会冲突
```

**解决方案：**

```kotlin
// Shadow 插件合并策略
tasks.shadowJar {
    merge("META-INF/services/*")
}
```

### 问题 3：Spring Boot 不兼容

某些框架依赖特定的 classpath 结构，不兼容 Spring Boot 的 Fat Jar：

```xml
<!-- 有些框架需要在 classpath 上 -->
<dependency>
    <groupId>com.example</groupId>
    <artifactId>problematic-lib</artifactId>
    <scope>provided</scope>  <!-- 标记为 provided，避免打包 -->
</dependency>
```

### 问题 4：启动速度

Fat Jar 需要解压嵌套 jar，首次启动可能较慢：

```bash
# Spring Boot 2.3+ 支持分层镜像
# 将 jar 层分离到 Docker 镜像的不同层
spring-boot:layeredtools
```

## 分层 Fat Jar（Spring Boot 2.3+）

### 创建分层 jar

```bash
java -Djarmode=layertools -jar myapp.jar list
```

输出：

```
dependencies/
spring-boot-loader/
snapshot-dependencies/
application/
```

### Dockerfile 多阶段构建

```dockerfile
# 构建阶段
FROM maven:3.9-eclipse-temurin-17 AS builder
COPY pom.xml .
RUN mvn dependency:go-offline
COPY src ./src
RUN mvn package -DskipTests

# 提取分层
FROM eclipse-temurin-17-jre AS extracted
WORKDIR /app
RUN java -Djarmode=layertools -jar target/myapp.jar extract

# 最终镜像
FROM eclipse-temurin-17-jre
WORKDIR /app

# 按层复制，充分利用 Docker 缓存
COPY --from=extracted app/dependencies/ ./
COPY --from=extracted app/spring-boot-loader/ ./
COPY --from=extracted app/snapshot-dependencies/ ./
COPY --from=extracted app/application/ ./

ENTRYPOINT ["java", "org.springframework.boot.loader.JarLauncher"]
```

**分层优势：**

- 代码变更只需更新 `application` 层
- 依赖变更只需更新对应层
- Docker 缓存利用率最大化
- 构建速度更快

## Fat Jar vs Thin Jar

| 维度 | Fat Jar | Thin Jar |
|---|---|---|
| **文件大小** | 大（包含所有依赖）| 小（仅应用代码）|
| **部署复杂度** | 低（单一文件）| 高（需要管理依赖）|
| **启动速度** | 较慢（需要解压）| 快（直接加载）|
| **Docker 友好** | 一般 | 好（配合分层）|
| **适用场景** | 微服务、CLI 工具 | 需要共享依赖的应用 |

## 最佳实践

### 1. 排除不需要的依赖

```xml
<!-- Maven -->
<configuration>
    <excludes>
        <exclude>org.projectlombok:lombok</exclude>
        <exclude>*:javadoc</exclude>
        <exclude>*:sources</exclude>
    </excludes>
</configuration>
```

### 2. 生产环境排除调试依赖

```kotlin
// Gradle
val prodImplementation by configurations
configurations.prodImplementation {
    exclude(group = "org.springframework.boot", module = "spring-boot-devtools")
}
```

### 3. 验证 Fat Jar 内容

```bash
# 列出所有依赖
java -Djarmode=layertools -jar myapp.jar list

# 查看 BOOT-INF/lib
unzip -l myapp.jar | grep BOOT-INF/lib

# 验证主类
java -Djarmode=layertools -jar myapp.jar help
```

### 4. 性能优化

```dockerfile
# 使用 Spring Native（GraalVM）可以大幅提升启动速度
FROM eclipse-temurin:17-jre
# 替换为
FROM ghcr.io/graalvm/native-image:22
```

## 总结

Fat Jar 是 Spring Boot 最重要的创新之一：

- **单一部署单元**：一个 jar 包含一切，部署极简
- **嵌套 Jar 加载**：Spring Boot Loader 实现特殊类加载
- **分层构建**：Docker 友好的分层技术
- **潜在问题**：重复资源、Service Loader 冲突需要处理

**选择建议：**

- **微服务**：使用 Spring Boot Fat Jar
- **库/SDK**：使用普通 Maven jar
- **大型应用**：考虑分层 + Docker
- **Serverless**：考虑 Spring Native

<Badge text="最佳实践" type="tip" /> 生产环境部署时，建议使用分层镜像构建，充分利用 Docker 缓存加速 CI/CD。
