# 构建工具面试专题

本章节汇总 Maven 和 Gradle 相关的高频面试题，帮助你系统性地准备面试。

## 「Maven 的依赖传递是什么？冲突怎么解决？」

**核心答案：**

Maven 的依赖传递是指当项目引入一个依赖时，这个依赖的依赖（传递依赖）也会被自动引入。比如引入 `spring-boot-starter-web`，Spring MVC 的依赖、Jackson 的依赖都会自动进来。

**传递层级示意：**

```
项目
└── spring-boot-starter-web (直接依赖)
    └── spring-boot-starter (传递依赖)
        └── spring-boot (传递依赖)
            └── spring-core (传递依赖)
```

**冲突原因：**

同一个依赖可能通过多条路径被引入，且版本不同。Maven 使用「最短路径优先」和「声明顺序优先」解决：

| 规则 | 说明 |
|---|---|
| 最短路径优先 | A → B → C(v1.0)，A → D → C(v2.0)，选择 v2.0 |
| 第一声明优先 | 路径长度相同时，选择 pom.xml 中先声明的 |
| 显式声明优先 | 你在 pom.xml 中直接写的依赖，覆盖一切传递 |

**解决冲突的方式：**

1. **显式声明版本**（最直接）
2. **使用 dependencyManagement 统一版本**（最规范）
3. **排除传递依赖**（临时方案）

```xml
<!-- 排除传递依赖 -->
<dependency>
    <groupId>com.example</groupId>
    <artifactId>problematic-lib</artifactId>
    <version>1.0</version>
    <exclusions>
        <exclusion>
            <groupId>com.old</groupId>
            <artifactId>old-dep</artifactId>
        </exclusion>
    </exclusions>
</dependency>
```

## 「Maven 的生命周期有哪些？」

**核心答案：**

Maven 有三套独立的生命周期：**clean**、**default**、**site**。

**clean 生命周期（清理构建产物）：**

| 阶段 | 说明 |
|---|---|
| pre-clean | 清理前的准备 |
| **clean** | 删除 target 目录 |
| post-clean | 清理后的处理 |

**default 生命周期（构建核心）：**

| 阶段 | 说明 |
|---|---|
| validate | 验证项目结构 |
| compile | 编译源代码 |
| test | 运行单元测试 |
| package | 打包（jar/war）|
| **install** | 安装到本地仓库 |
| **deploy** | 发布到远程仓库 |

**site 生命周期（生成站点）：**

| 阶段 | 说明 |
|---|---|
| pre-site | 站点生成前准备 |
| **site** | 生成项目站点文档 |
| post-site | 站点生成后处理 |
| site-deploy | 部署站点到服务器 |

:::tip 重要特性
Maven 插件可以绑定到生命周期的某个阶段。比如 `mvn package` 会自动执行 `compile`、`test`、`package`，因为后者的阶段在前者之前。
:::

## 「Maven 和 Gradle 的区别？各自适用场景？」

**核心答案：**

| 对比维度 | Maven | Gradle |
|---|---|---|
| **构建语言** | XML (pom.xml) | Groovy/Kotlin DSL |
| **依赖管理** | 依赖传递 + BOM | 版本目录 + Platform |
| **构建速度** | 较慢（全量构建）| 快（增量构建、缓存）|
| **灵活性** | 中等 | 高（DSL 可编程）|
| **社区生态** | 成熟 | 快速发展 |
| **Android 支持** | 无 | 官方支持 |
| **学习曲线** | 平缓 | 较陡 |

**选择建议：**

| 场景 | 推荐 |
|---|---|
| Java 企业级项目（传统）| Maven |
| Android 项目 | Gradle |
| 微服务（需要快速构建）| Gradle |
| 需要复杂构建逻辑 | Gradle |
| 团队对 XML 熟悉 | Maven |

**Gradle 的优势场景：**

- 增量构建：只重新构建变更的部分
- 并行执行：充分利用多核 CPU
- 构建缓存：重复构建跳过不必要步骤
- Kotlin DSL：类型安全、IDE 支持好

## 「Gradle 的增量构建是怎么实现的？」

**核心答案：**

增量构建是 Gradle 最大的性能优势。Gradle 会追踪任务的输入和输出，只有当输入或输出发生变化时，才重新执行任务。

**增量构建原理：**

```
Task A (compileJava)
    输入：src/main/java/*.java
    输出：build/classes/java/main/*.class
    ↓
检查：上次构建的输入 hash vs 本次输入 hash
    ↓
    ↓ 相同 → 跳过任务
    ↓ 不同 → 执行任务
```

**Task 定义示例：**

```kotlin
tasks.register<Copy>("copyResources") {
    // 输入：src/main/resources
    from("src/main/resources")
    // 输出：build/resources/main
    into(layout.buildDirectory.dir("resources/main"))
    // 依赖的输入：src/main/java（Java 变化可能影响资源）
    dependsOn("compileJava")
}
```

**缓存机制：**

| 机制 | 说明 |
|---|---|
| 本地构建缓存 | 同一机器不同项目共享 |
| 远程构建缓存 | 团队成员共享（需要配置）|
| 配置缓存 | 缓存 Task 配置结果 |

```kotlin
// 启用构建缓存
gradle.properties:
org.gradle.caching=true

// 远程缓存配置
org.gradle.configuration-cache=on
```

## 「Maven 的 scope 有哪些？」

**核心答案：**

Maven 的 scope 控制依赖在哪些场景下生效：

| Scope | 编译时 | 测试时 | 运行时 | 打包 |
|---|---|---|---|---|
| **compile** | ✅ | ✅ | ✅ | ✅ |
| **provided** | ✅ | ✅ | ❌ | ❌ |
| **runtime** | ❌ | ✅ | ✅ | ✅ |
| **test** | ❌ | ✅ | ❌ | ❌ |
| **system** | ✅ | ✅ | ❌ | ❌ |
| **import** | - | - | - | 仅用于 BOM |

**典型使用场景：**

```xml
<!-- servlet-api 在运行时由容器提供 -->
<dependency>
    <groupId>javax.servlet</groupId>
    <artifactId>javax.servlet-api</artifactId>
    <version>4.0.1</version>
    <scope>provided</scope>  <!-- 容器（Tomcat）已提供，不打包 -->
</dependency>

<!-- JDBC 驱动只在运行时需要 -->
<dependency>
    <groupId>com.mysql</groupId>
    <artifactId>mysql-connector-java</artifactId>
    <version>8.0.33</version>
    <scope>runtime</scope>  <!-- 编译时不需要，运行时需要 -->
</dependency>

<!-- 测试框架仅测试时使用 -->
<dependency>
    <groupId>org.junit.jupiter</groupId>
    <artifactId>junit-jupiter</artifactId>
    <version>5.10.0</version>
    <scope>test</scope>
</dependency>
```

## 「什么是 Maven BOM？」

**核心答案：**

BOM（Bill of Materials，物料清单）是一种特殊的 POM，用于统一管理依赖版本。

**解决的问题：**

不使用 BOM 时，你需要为每个依赖显式指定版本：

```xml
<!-- 没有 BOM，每个依赖都要指定版本 -->
<dependency>
    <groupId>org.springframework</groupId>
    <artifactId>spring-core</artifactId>
    <version>5.3.27</version>
</dependency>
<dependency>
    <groupId>org.springframework</groupId>
    <artifactId>spring-beans</artifactId>
    <version>5.3.27</version>
</dependency>
```

**使用 BOM 后：**

```xml
<!-- 父 POM 定义 BOM -->
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-dependencies</artifactId>
            <version>2.7.18</version>
            <type>pom</type>
            <scope>import</scope>
        </dependency>
    </dependencies>
</dependencyManagement>

<!-- 子项目继承版本，无需指定 -->
<dependencies>
    <dependency>
        <groupId>org.springframework</groupId>
        <artifactId>spring-core</artifactId>
        <!-- 不需要指定 version！-->
    </dependency>
</dependencies>
```

**Spring Boot 的 BOM 策略：**

Spring Boot 的 `spring-boot-dependencies` 统一管理了 400+ 依赖的版本，你只需记住一个版本号（Spring Boot 版本）。

## 「多模块项目中，父子 POM 怎么配合？」

**核心答案：**

多模块项目的核心是**父 POM 管理公共配置，子模块继承父 POM**。

**项目结构：**

```
my-project/
├── pom.xml                    ← 父 POM
├── module-a/
│   └── pom.xml
├── module-b/
│   └── pom.xml
└── module-c/
    └── pom.xml
```

**父 POM 配置：**

```xml
<!-- my-project/pom.xml -->
<project>
    <groupId>com.example</groupId>
    <artifactId>my-project</artifactId>
    <version>1.0.0</version>
    <packaging>pom</packaging>

    <!-- 定义子模块 -->
    <modules>
        <module>module-a</module>
        <module>module-b</module>
        <module>module-c</module>
    </modules>

    <!-- 统一依赖管理 -->
    <dependencyManagement>
        <dependencies>
            <dependency>
                <groupId>org.springframework.boot</groupId>
                <artifactId>spring-boot-dependencies</artifactId>
                <version>2.7.18</version>
                <type>pom</type>
                <scope>import</scope>
            </dependency>
        </dependencies>
    </dependencyManagement>

    <!-- 统一插件管理 -->
    <build>
        <pluginManagement>
            <plugins>
                <plugin>
                    <groupId>org.springframework.boot</groupId>
                    <artifactId>spring-boot-maven-plugin</artifactId>
                    <version>2.7.18</version>
                </plugin>
            </plugins>
        </pluginManagement>
    </build>
</project>
```

**子模块配置：**

```xml
<!-- module-a/pom.xml -->
<project>
    <!-- 继承父 POM -->
    <parent>
        <groupId>com.example</groupId>
        <artifactId>my-project</artifactId>
        <version>1.0.0</version>
    </parent>

    <artifactId>module-a</artifactId>

    <!-- 子模块间依赖 -->
    <dependencies>
        <dependency>
            <groupId>com.example</groupId>
            <artifactId>module-b</artifactId>
            <version>${project.version}</version>
        </dependency>
    </dependencies>
</project>
```

## 「Gradle 的 API 和 Implementation 有什么区别？」

**核心答案：**

| 作用域 | compileClasspath | runtimeClasspath |
|---|---|---|
| **api** | ✅ | ✅ |
| **implementation** | ❌ | ✅ |

**核心区别：**

- **api**：依赖对下游模块可见（通过 api 依赖你的模块的模块也能看到这个依赖）
- **implementation**：依赖对下游模块不可见（更封装）

**场景对比：**

```kotlin
// module:core
dependencies {
    // A 模块通过 api 依赖 commons-lang3
    api("org.apache.commons:commons-lang3:3.12.0")
    
    // B 模块通过 implementation 依赖 guava
    implementation("com.google.guava:guava:32.1.3-jre")
}

// module:app 依赖 core
dependencies {
    implementation(project(":core"))
}

// module:app 能看到什么？
// ✅ commons-lang3（core 通过 api 引入）
// ❌ guava（core 通过 implementation 引入，app 不可见）
```

**为什么 implementation 更好？**

1. **加速构建**：修改 implementation 依赖不需要重新编译下游模块
2. **隐藏依赖**：下游模块不应该依赖你的内部依赖
3. **避免冲突**：减少 classpath 上的依赖数量，降低冲突风险

**什么时候用 api？**

- 你的模块是一个公共库
- 下游模块需要使用这个依赖

**什么时候用 implementation？**

- 你的模块是应用模块
- 这个依赖只是内部实现细节

## 「如何排查 Maven/Gradle 依赖冲突？」

**核心答案：**

### Maven 依赖冲突排查

**1. 查看依赖树**

```bash
# 查看完整依赖树
mvn dependency:tree

# 过滤特定依赖
mvn dependency:tree -Dincludes=com.fasterxml.jackson

# 分析依赖冲突
mvn dependency:analyze
```

输出示例：

```
[INFO] +- com.example:module-b:jar:1.0:compile
[INFO] |  \- com.fasterxml.jackson:jackson-databind:jar:2.13.0:compile
[INFO] \- com.example:module-c:jar:1.0:compile
[INFO]    \- com.fasterxml.jackson:jackson-databind:jar:2.15.2:compile
[INFO] [INFO] commons-lang3:commons-lang3:jar:2.6:compile (omitted for conflict)
```

**2. 分析报告**

```bash
# 生成依赖分析报告
mvn dependency:analyze-report

# 打开报告
open target/dependency-analysis-report.html
```

**3. 查看版本冲突来源**

```bash
# 详细查看某个依赖的版本来源
mvn dependency:tree -Dverbose | grep jackson
```

### Gradle 依赖冲突排查

**1. 查看依赖配置**

```bash
# 查看所有依赖
./gradlew dependencies

# 查看特定配置的依赖
./gradlew dependencies --configuration runtimeClasspath

# 过滤输出
./gradlew dependencies | grep jackson
```

**2. 解决冲突**

```kotlin
// 强制指定版本
configurations.all {
    resolutionStrategy {
        force("com.fasterxml.jackson:jackson-databind:2.15.2")
    }
}

// 或使用 strict 模式（强制且报告错误）
configurations.all {
    resolutionStrategy {
        strictly("1.0")
    }
}
```

**3. 分析构建**

```bash
# 扫描依赖
./gradlew buildEnvironment
./gradlew buildEnvironment --scan
```

## 「构建缓存是什么？Maven 和 Gradle 的缓存机制有何不同？」

**核心答案：**

### Maven 缓存

Maven 使用本地仓库（`~/.m2/repository`）存储下载的依赖：

```
~/.m2/repository/
├── org/springframework/
│   └── spring-core/
│       ├── 5.3.27/
│       │   ├── spring-core-5.3.27.jar
│       │   └── spring-core-5.3.27.pom
│       └── 5.3.28/
│           └── ...
└── com/example/
    └── my-lib/1.0/
        └── my-lib-1.0.jar
```

**特点：**

- 仅缓存依赖 jar
- 不缓存构建结果
- 无远程共享缓存（需 Nexus/Artifactory）

### Gradle 缓存

Gradle 的缓存机制更强大：

| 缓存类型 | 说明 |
|---|---|
| **依赖缓存** | 本地存储下载的依赖 |
| **构建缓存** | 缓存 Task 的输入输出 |
| **配置缓存** | 缓存 Task 配置结果 |
| **远程缓存** | 可配置团队共享缓存 |

**配置示例：**

```kotlin
// gradle.properties
org.gradle.caching=true
org.gradle.parallel=true
org.gradle.daemon=true
org.gradle.configureondemand=true

// 远程构建缓存（可选）
org.gradle.caching.remote=teamcity
org.gradle.caching.url=http://cache-server:8080
```

**增量构建 vs 缓存：**

| 机制 | 原理 | 适用场景 |
|---|---|---|
| 增量构建 | 只重做变更部分 | 单次构建内的优化 |
| 构建缓存 | 复用历史构建结果 | 跨构建、跨项目 |

**Gradle 的优势：**

```kotlin
// 同一个依赖，Gradle 只需下载一次
// 即使项目 clean 后重新构建，Gradle 会使用缓存
./gradlew clean build

// 如果 task 输入输出没变，直接使用缓存
./gradlew build
// 输出：:core:build UP-TO-DATE
```

## 面试追问方向

1. **Maven 的依赖作用域在实际项目中如何使用？**
   - provided 用于容器提供的 API（Servlet、JPA）
   - runtime 用于运行时需要的实现（JDBC 驱动）

2. **Gradle 和 Maven 在大型项目中如何选择？**
   - 项目规模 < 10 个模块：两者皆可
   - 项目规模大、构建频繁：Gradle（增量构建优势明显）
   - Android 项目：必须使用 Gradle

3. **Maven 如何发布到私有仓库？**
   - 配置 distributionManagement
   - 使用 maven-deploy-plugin
   - 配合 Nexus/Artifactory

4. **Gradle 何时使用 Kotlin DSL vs Groovy DSL？**
   - Kotlin DSL：类型安全、IDE 支持好
   - Groovy DSL：更灵活、语法更简洁

5. **如何优化 Maven/Gradle 构建速度？**
   - Maven：并行构建 `mvn -T 4`
   - Gradle：增量构建、配置缓存、并行执行
   - 通用：本地镜像仓库、增量编译
