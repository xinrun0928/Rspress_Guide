# 多环境构建

你有没有遇到过这样的问题——本地开发好好的代码，一上线就出问题？测试环境验证通过，生产环境却报错？

问题的根源往往是环境差异没有被正确隔离。多环境构建就是为了解决这个问题——通过不同的配置，让同一份代码在不同环境中运行。

## 为什么需要多环境

### 环境差异的常见场景

```
┌──────────────┐   ┌──────────────┐   ┌──────────────┐
│   开发环境     │   │   测试环境     │   │   生产环境     │
│              │   │              │   │              │
│ API: localhost │   │ API: test.internal │ │ API: api.example.com │
│ Debug: true   │   │ Debug: true   │   │ Debug: false  │
│ DB: localhost │   │ DB: test-mysql │   │ DB: prod-mysql │
│ Log: DEBUG   │   │ Log: INFO    │   │ Log: WARN    │
└──────────────┘   └──────────────┘   └──────────────┘
```

### 多环境的目标

- **配置隔离**：不同环境使用不同配置
- **构建物复用**：同一份代码，打出不同环境的产物
- **安全隔离**：生产配置不泄露到测试环境
- **环境一致性**：开发、测试、生产尽量一致

## Maven 多环境配置

### Profile 方式

```xml title="pom.xml"
<profiles>
    <!-- 开发环境 -->
    <profile>
        <id>dev</id>
        <activation>
            <activeByDefault>true</activeByDefault>
        </activation>
        <properties>
            <env>dev</env>
            <api.url>http://localhost:8080</api.url>
            <db.url>jdbc:mysql://localhost:3306/myapp</db.url>
            <db.username>root</db.username>
            <db.password></db.password>
        </properties>
    </profile>

    <!-- 测试环境 -->
    <profile>
        <id>test</id>
        <properties>
            <env>test</env>
            <api.url>http://test.internal/api</api.url>
            <db.url>jdbc:mysql://test-mysql:3306/myapp</db.url>
            <db.username>testuser</db.username>
            <db.password>test_password</db.password>
        </properties>
    </profile>

    <!-- 生产环境 -->
    <profile>
        <id>prod</id>
        <properties>
            <env>prod</env>
            <api.url>https://api.example.com</api.url>
            <db.url>jdbc:mysql://prod-mysql:3306/myapp</db.url>
            <db.username>produser</db.username>
            <db.password>${PROD_DB_PASSWORD}</db.password>
        </properties>
    </profile>
</profiles>
```

### 资源过滤

```xml
<build>
    <resources>
        <resource>
            <directory>src/main/resources</directory>
            <filtering>true</filtering>
            <excludes>
                <!-- 生产配置文件不打包到其他环境 -->
                <exclude>application-prod.yml</exclude>
            </excludes>
        </resource>
    </resources>
</build>
```

### 切换环境构建

```bash
# 默认（dev）
mvn clean package

# 指定环境
mvn clean package -Ptest
mvn clean package -Pprod

# 查看激活的 Profile
mvn help:active-profiles
```

### Spring Boot 的 Profile 整合

```yaml title="src/main/resources/application.yml"
spring:
  profiles:
    active: @env@
  datasource:
    url: ${db.url}
    username: ${db.username}
    password: ${db.password}
logging:
  level:
    root: INFO
```

```xml title="pom.xml"
<build>
    <plugins>
        <plugin>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-maven-plugin</artifactId>
            <configuration>
                <profiles>
                    <profile>dev</profile>
                    <profile>test</profile>
                    <profile>prod</profile>
                </profiles>
            </configuration>
        </plugin>
    </plugins>
</build>
```

## Gradle 多环境配置

### ProductFlavors 方式

```groovy title="build.gradle"
plugins {
    id 'java'
    id 'org.springframework.boot'
}

flavors {
    dev {
        dimension 'env'
        applicationIdSuffix '.dev'
        versionNameSuffix '-dev'
        buildConfigField 'String', 'ENV_NAME', '"development"'
        buildConfigField 'String', 'API_BASE_URL', '"http://localhost:8080"'
        buildConfigField 'boolean', 'DEBUG_MODE', 'true'
    }

    test {
        dimension 'env'
        applicationIdSuffix '.test'
        versionNameSuffix '-test'
        buildConfigField 'String', 'ENV_NAME', '"testing"'
        buildConfigField 'String', 'API_BASE_URL', '"http://test.internal/api"'
        buildConfigField 'boolean', 'DEBUG_MODE', 'true'
    }

    prod {
        dimension 'env'
        // 不加后缀，生成正式包
        buildConfigField 'String', 'ENV_NAME', '"production"'
        buildConfigField 'String', 'API_BASE_URL', '"https://api.example.com"'
        buildConfigField 'boolean', 'DEBUG_MODE', 'false'
    }
}
```

### 构建变体

```bash
# 列出所有变体
./gradlew tasks --group=build

# 构建指定环境
./gradlew build            # 构建所有
./gradlew assembleDevDebug
./gradlew assembleProdRelease

# 查看构建产物
./gradlew buildEnvironment
```

### 环境配置文件

```
src/
├── main/
│   ├── java/
│   └── resources/
│       ├── application.yml          # 共享配置
│       ├── application-dev.yml        # 开发配置
│       ├── application-test.yml      # 测试配置
│       └── application-prod.yml     # 生产配置
```

```yaml title="application.yml"
spring:
  profiles:
    active: ${ENV:dev}
  datasource:
    url: ${DB_URL}
    username: ${DB_USERNAME}
    password: ${DB_PASSWORD}
```

## 多环境构建最佳实践

### 1. 敏感配置不打包

```properties title=".gitignore"
# 排除生产配置文件
src/main/resources/application-prod.yml
```

```bash
# 生产配置通过环境变量注入
# 不打包到 jar/镜像中
java -jar myapp.jar \
  --spring.datasource.password=${DB_PASSWORD}
```

### 2. 使用环境变量

```yaml
# application.yml
spring:
  datasource:
    url: ${DB_URL:mysql://localhost:3306/myapp}  # 默认值
    password: ${DB_PASSWORD}
```

```dockerfile
# Dockerfile
ENV DB_URL=mysql://prod-mysql:3306/myapp
ENV DB_PASSWORD=${DB_PASSWORD}
```

### 3. Secret 管理

```bash
# 使用 Vault 管理生产密码
vault read secret/myapp/prod

# 或使用 K8s Secret
kubectl create secret generic myapp-secrets \
  --from-literal=DB_PASSWORD=xxx \
  --from-literal=API_KEY=xxx
```

### 4. 环境一致性

```
本地开发 ──► Docker Compose ──► 模拟依赖服务
   │
   ▼
CI 服务器 ──► Docker Image ──► K8s 测试环境
   │
   ▼
CD 流水线 ──► Docker Image ──► K8s 生产环境
```

## 常见问题

### Profile 不生效

```bash
# 检查 Profile 激活状态
mvn help:active-profiles

# 强制激活
mvn clean package -Pdev -U
```

### 资源文件中的变量不替换

```xml
<!-- 确认 resource 开启了 filtering -->
<resource>
    <directory>src/main/resources</directory>
    <filtering>true</filtering>
</resource>
```

### 不同环境的依赖不同

```groovy
// Gradle
dependencies {
    // 所有环境都有的依赖
    implementation 'org.springframework.boot:spring-boot-starter-web'

    // 不同环境的特殊依赖
    devImplementation 'org.springframework.boot:spring-boot-devtools'
    testImplementation 'org.springframework.boot:spring-boot-starter-test'
}
```

## 总结

多环境构建的核心要点：

1. **使用 Profile/Flavor 区分环境**：Maven 用 Profile，Gradle 用 ProductFlavors
2. **敏感配置不打包**：通过环境变量或 K8s Secret 注入
3. **环境一致性**：开发、测试、生产尽量使用相同的构建流程
4. **自动化切换**：CI/CD 流水线自动选择目标环境

合理的环境隔离可以让开发、测试、部署更加高效和安全。
