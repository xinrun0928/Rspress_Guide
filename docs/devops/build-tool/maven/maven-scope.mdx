# Maven 依赖作用域

依赖作用域（Dependency Scope）控制了依赖在哪些场景下可用，以及是否会被打包。理解作用域，是避免类路径污染和不必要的包体积膨胀的关键。

## 六种作用域

Maven 有六种依赖作用域：

| 作用域 | 编译时可见 | 运行时可见 | 会打包 | 典型使用场景 |
|---|---|---|---|---|
| `compile` | ✅ | ✅ | ✅ | 默认值，大部分依赖 |
| `provided` | ✅ | ❌ | ❌ | JDK / 容器提供 |
| `runtime` | ❌ | ✅ | ✅ | 运行时 JDBC 实现 |
| `test` | ❌ | ❌ | ❌ | 仅测试代码 |
| `system` | ✅ | ❌ | ❌ | 系统路径 JAR |
| `import` | - | - | - | 仅用于 BOM |

---

## compile：默认作用域

`compile` 是默认作用域，不声明时等价于 `<scope>compile</scope>`。

```xml title="pom.xml"
<!-- 这两种写法等价 -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
    <version>2.7.0</version>
</dependency>

<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
    <version>2.7.0</version>
    <scope>compile</scope>
</dependency>
```

`compile` 作用域的依赖在编译、测试、运行阶段都可用，且会被打包到最终的 JAR 或 WAR 中。

---

## provided：运行环境已提供

`provided` 作用域用于编译和测试，但**不会打包**。运行环境（如 JDK、Servlet 容器）已经提供了这些依赖。

```xml title="pom.xml"
<!-- Servlet API 由 Tomcat/Jetty 等容器提供 -->
<dependency>
    <groupId>javax.servlet</groupId>
    <artifactId>javax.servlet-api</artifactId>
    <version>4.0.1</version>
    <scope>provided</scope>
</dependency>

<!-- Lombok 在编译期使用，不打包 -->
<dependency>
    <groupId>org.projectlombok</groupId>
    <artifactId>lombok</artifactId>
    <version>1.18.24</version>
    <scope>provided</scope>
</dependency>
```

为什么 Servlet API 要用 `provided`？因为 WAR 包部署到 Tomcat 后，Tomcat 自身提供了 Servlet API。如果打包进去，可能和容器自带的版本冲突，或者导致类加载问题。

### Lombok 的特殊之处

Lombok 是一个特殊的 `provided` 依赖。它在编译期通过注解处理器生成代码，不需要在运行时存在：

```
编译期：Lombok 读取注解 → 生成字节码 → 合并到 .class 文件
运行时：业务代码直接调用生成的代码，不依赖 Lombok
```

这就是为什么 Lombok 的作用域是 `provided` 而不是 `compile`——它在编译后「消失」了。

## runtime：运行时需要

`runtime` 作用域用于编译时不需要，但运行时需要的依赖。最典型的例子是 JDBC 驱动。

```xml title="pom.xml"
<!-- MySQL JDBC 驱动：编译时不需要（用 JDBC 接口编程） -->
<!-- 运行时必须（MySQL 的实现类） -->
<dependency>
    <groupId>mysql</groupId>
    <artifactId>mysql-connector-java</artifactId>
    <version>8.0.29</version>
    <scope>runtime</scope>
</dependency>

<!-- PostgreSQL JDBC 驱动 -->
<dependency>
    <groupId>org.postgresql</groupId>
    <artifactId>postgresql</artifactId>
    <version>42.5.0</version>
    <scope>runtime</scope>
</dependency>
```

你写代码时这样写：

```java
// 编译时：Connection 是 java.sql.Connection（JDK 自带）
// 运行时：实际是 MySQL 的 MysqlConnection 或 PostgreSQL 的 PgConnection
Connection conn = DriverManager.getConnection(url, user, password);
```

编译时只需要 JDBC 接口（`java.sql.*`），这些在 JDK 中已经存在。运行时才需要具体的驱动实现。

:::tip 为什么用 runtime 而不是 compile
标记为 `runtime` 的依赖不会参与编译类路径的检查。如果你代码中直接引用了 MySQL 特有的类（如 `MysqlSavepoint`），编译时会报错，帮你尽早发现问题。更重要的是，`runtime` 依赖不会被传递到依赖当前模块的其他项目。
:::

## test：仅测试可用

`test` 作用域的依赖只在测试代码编译和运行时可用，不会打包。

```xml title="pom.xml"
<!-- JUnit 5 -->
<dependency>
    <groupId>org.junit.jupiter</groupId>
    <artifactId>junit-jupiter</artifactId>
    <version>5.9.0</version>
    <scope>test</scope>
</dependency>

<!-- Mockito -->
<dependency>
    <groupId>org.mockito</groupId>
    <artifactId>mockito-core</artifactId>
    <version>4.8.0</version>
    <scope>test</scope>
</dependency>

<!-- Spring Boot Test -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-test</artifactId>
    <version>2.7.0</version>
    <scope>test</scope>
</dependency>
```

`test` 作用域的特点：

- **不可传递**：A 的 test 依赖不会传递给依赖 A 的项目 B
- **不打包**：不会进入最终的 JAR/WAR
- **独立类路径**：测试代码有自己的类路径，与主代码分离

### 为什么测试依赖要隔离

想象一下：如果 `junit` 被打包到生产 JAR 中，运行时可能加载两个 JUnit 版本，导致奇怪的冲突。`test` 作用域确保测试工具只存在于测试阶段。

## system：系统路径（慎用）

`system` 作用域类似于 `provided`，但依赖来自本地文件系统路径，而不是仓库。

```xml title="pom.xml"
<!-- 引用系统目录下的 JAR -->
<dependency>
    <groupId>com.example</groupId>
    <artifactId>proprietary-lib</artifactId>
    <version>1.0</version>
    <scope>system</scope>
    <systemPath>${project.basedir}/libs/proprietary.jar</systemPath>
</dependency>
```

### system 作用域的问题

| 问题 | 说明 |
|---|---|
| 不可移植 | 路径在不同机器上可能不同 |
| 不可传递 | 其他项目无法使用这个依赖 |
| 不检查仓库 | Maven 不会验证依赖存在 |

:::warning 尽量避免使用 system 作用域
`system` 作用域是历史遗留，在现代 Maven 项目中应该避免。如果需要本地 JAR，应该安装到本地仓库：

```bash
mvn install:install-file \
    -Dfile=proprietary.jar \
    -DgroupId=com.example \
    -DartifactId=proprietary-lib \
    -Dversion=1.0 \
    -Dpackaging=jar
```

这样就可以用正常的 `compile` 作用域引用它了。
:::

## import：导入 BOM

`import` 作用域是六种作用域中最特殊的一个，它只用于 `<dependencyManagement>` 中导入 BOM（Bill of Materials）。

```xml title="pom.xml"
<dependencyManagement>
    <dependencies>
        <!-- 导入 Spring Boot 的依赖管理 -->
        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-dependencies</artifactId>
            <version>2.7.0</version>
            <scope>import</scope>
            <type>pom</type>
        </dependency>
    </dependencies>
</dependencyManagement>
```

`import` 的具体用法和 BOM 的设计会在后续章节详细讲解。

## 作用域选择指南

| 问题 | 答案 |
|---|---|
| 这个库在生产运行时必须存在吗？ | 否 → `provided` |
| 这个库在编译时必须存在吗？ | 否 → `runtime` |
| 这个库只用于单元测试吗？ | 是 → `test` |
| JDK 或容器会提供这个库吗？ | 是 → `provided` |
| 我需要统一管理版本吗？ | 是 → `import` + BOM |

## 面试追问

**面试官可能问**：为什么 JDBC 驱动要用 `runtime` 作用域？

这个问题考察的是对 Java JDBC 架构的理解。JDBC 是一种桥接模式：应用代码面向 `java.sql` 接口编程，运行时由具体的驱动实现（MySQL Driver、PostgreSQL Driver）。编译时只需要 JDBC 接口（JDK 自带），运行时才需要具体的驱动类。如果用 `compile` 作用域，代码中可能不小心引用了驱动特有的类，违反 JDBC 的设计原则。

**面试官可能问**：`provided` 和 `compile` 的区别是什么？什么时候用 `provided`？

两者在编译期都可见，核心区别是打包行为：`provided` 不会打包到产物中，`compile` 会。典型场景是：依赖由运行环境（JDK、Servlet 容器、Spring Boot 嵌入式容器）提供时，用 `provided`；依赖需要随应用一起部署时，用 `compile`。常见误用是把 Servlet API 声明为 `compile`，导致打包后和容器自带的 Servlet API 冲突。
