# Maven 生命周期

Maven 之所以能够「一条命令走天下」，核心在于它的**生命周期（Lifecycle）**机制。Maven 定义了三套独立的生命周期，每套生命周期包含多个**阶段（Phase）**，每个阶段又绑定特定的**目标（Goal）**。

理解生命周期，是掌握 Maven 的关键。

## 三套生命周期

Maven 有三套独立的生命周期，它们之间没有任何依赖关系，可以单独执行：

| 生命周期 | 用途 | 核心阶段 |
|---|---|---|
| `clean` | 清理构建产物 | clean |
| `default` | 编译、测试、打包、部署 | compile、test、package、install、deploy |
| `site` | 生成项目文档 | site |

---

## Clean 生命周期

Clean 生命周期专注于清理工作目录：

| 阶段 | 说明 |
|---|---|
| `pre-clean` | 执行清理前的准备工作 |
| `clean` | 删除 `target/` 目录 |
| `post-clean` | 执行清理后的收尾工作 |

```bash
# 只执行 clean 阶段
mvn clean
```

`mvn clean` 会删除项目根目录下的 `target/` 文件夹及其中所有内容。

:::tip 清理范围
`mvn clean` 只清理当前项目的 `target/` 目录。对于多模块项目，需要在父项目目录执行，或者使用 `mvn clean -pl module1,module2` 清理指定模块。
:::

---

## Default 生命周期

Default 生命周期是 Maven 的核心，负责项目的构建过程：

| 阶段 | 说明 | 插件目标 |
|---|---|---|
| `validate` | 验证项目结构是否正确 | - |
| `initialize` | 初始化构建状态 | - |
| `generate-sources` | 生成源代码 | - |
| `process-sources` | 处理源代码 | `resources:resources` |
| `generate-resources` | 生成资源文件 | - |
| `process-resources` | 复制资源文件到输出目录 | `resources:resources` |
| `compile` | 编译源代码 | `compiler:compile` |
| `process-classes` | 处理编译后的文件 | - |
| `generate-test-sources` | 生成测试源代码 | - |
| `process-test-sources` | 处理测试源代码 | `resources:testResources` |
| `generate-test-resources` | 生成测试资源文件 | - |
| `process-test-resources` | 复制测试资源文件 | `resources:testResources` |
| `test-compile` | 编译测试源代码 | `compiler:testCompile` |
| `process-test-classes` | 处理测试编译后的文件 | - |
| `test` | 运行单元测试 | `surefire:test` |
| `prepare-package` | 打包前的准备 | - |
| `package` | 打包成 JAR/WAR/EAR | `jar:jar` / `war:war` |
| `pre-integration-test` | 集成测试前准备 | - |
| `integration-test` | 运行集成测试 | `failsafe:integration-test` |
| `post-integration-test` | 集成测试后处理 | - |
| `verify` | 验证打包结果 | - |
| `install` | 安装到本地仓库 | `install:install` |
| `deploy` | 部署到远程仓库 | `deploy:deploy` |

### 阶段执行顺序

**Maven 最重要的规则之一**：执行某个阶段时，该阶段之前的所有阶段都会被执行。

```bash
# 执行 package 阶段时，以下阶段按顺序执行：
# validate → ... → compile → process-classes → test → ... → package
mvn package
```

```bash
# 执行 clean 时，先执行 clean，再执行 package
# 但 clean 属于 clean 生命周期，所以实际上是先 clean，再进入 default 生命周期
mvn clean package
```

### 常用命令对应的阶段

| 命令 | 执行阶段 | 典型用途 |
|---|---|---|
| `mvn compile` | compile | 只编译，不测试，不打包 |
| `mvn test` | test | 运行单元测试 |
| `mvn package` | package | 打包（跳过测试）|
| `mvn install` | install | 安装到本地仓库 |
| `mvn deploy` | deploy | 发布到远程仓库 |

## Site 生命周期

Site 生命周期用于生成项目文档和报告：

| 阶段 | 说明 |
|---|---|
| `pre-site` | 生成文档前准备 |
| `site` | 生成站点文档 | `maven-site-plugin:site` |
| `post-site` | 文档生成后处理 |
| `site-deploy` | 部署站点到服务器 | `maven-site-plugin:deploy` |

```bash
# 生成项目站点
mvn site

# 生成并部署站点
mvn site-deploy
```

Site 生命周期用得较少，一般在项目发布时使用。

## 插件与阶段绑定

Maven 的阶段本身不做什么，它们只是**钩子**，真正工作的是绑定到这些阶段的**插件目标（Plugin Goal）**。

### 内置绑定

Maven 默认将插件目标绑定到特定阶段：

| 阶段 | 插件目标 |
|---|---|
| `process-resources` | `resources:resources` |
| `compile` | `compiler:compile` |
| `process-test-resources` | `resources:testResources` |
| `test-compile` | `compiler:testCompile` |
| `test` | `surefire:test` |
| `package` | `jar:jar` / `war:war` / `ear:ear` |
| `install` | `install:install` |
| `deploy` | `deploy:deploy` |

### 自定义绑定

可以显式将插件目标绑定到特定阶段：

```xml title="pom.xml"
<build>
    <plugins>
        <!-- 绑定 source:jar goal 到 package 阶段，为源码打包 -->
        <plugin>
            <groupId>org.apache.maven.plugins</groupId>
            <artifactId>maven-source-plugin</artifactId>
            <version>3.2.1</version>
            <executions>
                <execution>
                    <id>attach-sources</id>
                    <phase>package</phase>
                    <goals>
                        <goal>jar</goal>
                    </goals>
                </execution>
            </executions>
        </plugin>
    </plugins>
</build>
```

### 跳过插件执行

```bash
# 跳过单元测试
mvn package -DskipTests

# 跳过编译测试（也跳过测试）
mvn package -Dmaven.test.skip=true

# 跳过 maven-surefire-plugin
mvn package -Dsurefire.skip=true
```

:::warning `skipTests` vs `maven.test.skip`
`maven.test.skip=true` 会跳过测试编译和测试执行；`skipTests=true` 只跳过测试执行，但会编译测试代码。如果测试代码有问题，`skipTests=true` 会暴露这个问题，而 `maven.test.skip=true` 会静默跳过。
:::

## 执行顺序示例

假设执行 `mvn clean package`，完整流程如下：

```
========== clean 生命周期 ==========
pre-clean      → (空)
clean          → 删除 target/

========== default 生命周期 ==========
validate       → 验证项目结构
...
compile        → compiler:compile 编译 src/main/java
...
test           → surefire:test 运行 src/test/java 中的测试
...
package        → jar:jar 打包成 target/myapp-1.0.jar
```

## 多模块项目的生命周期

在多模块项目中，执行父项目的阶段会自动处理所有子模块：

```bash
# 在父项目目录执行
mvn clean package
```

执行顺序：

1. Maven 首先构建**依赖关系图（Reactor）**，确定模块构建顺序
2. 按顺序构建每个模块
3. 每个模块独立执行完整生命周期

如果模块 A 依赖模块 B，Maven 会确保 B 先构建：

```
模块关系：
user-service → user-api → common

构建顺序：
1. common
2. user-api
3. user-service
```

## 面试追问

**面试官可能问**：`mvn clean package` 和 `mvn package` 的区别是什么？

`mvn package` 只执行 default 生命周期的 package 阶段及其之前的阶段，不清理旧的构建产物。如果源代码删除了某个类，但之前编译的 `.class` 文件还在，`target/` 中会残留旧文件，可能导致诡异的问题。`mvn clean package` 先执行 clean 生命周期删除 `target/`，然后重新构建，确保每次都是干净的环境。标准做法是使用 `mvn clean package` 或 `mvn clean install`。

**面试官可能问**：Maven 的生命周期和插件是什么关系？

生命周期定义「什么时候做什么」，插件定义「做什么」。生命周期只是一些列的阶段标记，阶段本身不执行任何操作。真正工作的是绑定到阶段上的插件目标。例如 `mvn package` 执行到 package 阶段时，触发 `jar:jar` 或 `war:war` 插件目标，完成实际的打包工作。理解这个设计，才能理解 Maven 的灵活性——你可以绑定任何插件目标到任何阶段。
