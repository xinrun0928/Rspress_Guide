# Maven 插件机制

如果说生命周期是 Maven 的骨架，那么插件就是 Maven 的肌肉。没有插件，Maven 什么也做不了——它只是定义了「该做什么」，真正「做事」的是插件。

理解插件机制，是进阶 Maven 的必经之路。

## 插件与生命周期的绑定

Maven 的每个阶段（Phase）只是一个钩子，实际工作由**插件目标（Plugin Goal）**完成。

### 内置绑定

Maven 已经内置了很多插件绑定：

```bash
# 当执行 mvn compile 时
# 触发 compiler:compile 目标
# 插件:compiler
# 目标:compile

# 当执行 mvn test 时
# 触发 surefire:test 目标
# 插件:surefire
# 目标:test

# 当执行 mvn package 时
# 触发 jar:jar（或 war:war）目标
# 插件:jar 或 war
# 目标:jar
```

### 插件目标语法

```
mvn groupId:artifactId:version:goal
# 简写（使用默认版本）
mvn groupId:artifactId:goal
# 再简写（使用 Super POM 中的版本）
mvn artifactId:goal
```

常用插件的简写：

```bash
mvn compiler:compile        # 编译 Java 代码
mvn surefire:test          # 运行测试
mvn jar:jar                # 打包 JAR
mvn war:war                # 打包 WAR
mvn dependency:tree       # 查看依赖树
mvn help:effective-pom     # 查看有效 POM
```

## 常用插件详解

### maven-compiler-plugin

编译 Java 代码的插件，控制 Java 版本：

```xml title="pom.xml"
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-compiler-plugin</artifactId>
    <version>3.10.1</version>
    <configuration>
        <source>17</source>
        <target>17</target>
        <!-- 开启增量编译 -->
        <useIncrementalCompilation>true</useIncrementalCompilation>
        <!-- 编译参数 -->
        <compilerArgs>
            <arg>-Xlint:unchecked</arg>
            <arg>-Xlint:deprecation</arg>
        </compilerArgs>
    </configuration>
</plugin>
```

:::tip 增量编译
Maven 的增量编译由 `maven-compiler-plugin` 提供，但效果有限。真正的增量编译需要使用 Gradle 或 Maven 的构建缓存服务器（如 Sonatype Nexus Lifecycle）。对于大多数项目，标准编译已经足够。
:::

### maven-surefire-plugin

运行单元测试的插件：

```xml title="pom.xml"
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-surefire-plugin</artifactId>
    <version>2.22.2</version>
    <configuration>
        <!-- 测试失败是否继续 -->
        <testFailureIgnore>true</testFailureIgnore>
        <!-- 并行测试 -->
        <parallel>methods</parallel>
        <threadCount>4</threadCount>
        <!-- 跳过测试 -->
        <skipTests>false</skipTests>
    </configuration>
</plugin>
```

### maven-jar-plugin

打包 JAR 文件：

```xml title="pom.xml"
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-jar-plugin</artifactId>
    <version>3.2.2</version>
    <configuration>
        <archive>
            <manifest>
                <mainClass>com.example.App</mainClass>
                <addClasspath>true</addClasspath>
                <classpathPrefix>lib/</classpathPrefix>
            </manifest>
        </archive>
    </configuration>
</plugin>
```

### maven-shade-plugin

创建包含所有依赖的 Fat JAR：

```xml title="pom.xml"
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-shade-plugin</artifactId>
    <version>3.4.0</version>
    <executions>
        <execution>
            <phase>package</phase>
            <goals>
                <goal>shade</goal>
            </goals>
            <configuration>
                <transformers>
                    <transformer implementation="org.apache.maven.plugins.shade.resource.ManifestResourceTransformer">
                        <mainClass>com.example.App</mainClass>
                    </transformer>
                </transformers>
            </configuration>
        </execution>
    </executions>
</plugin>
```

Shade 插件会合并所有依赖的类和资源，生成一个包含所有内容的 uber-jar。

### maven-assembly-plugin

更灵活的自定义打包：

```xml title="pom.xml"
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-assembly-plugin</artifactId>
    <version>3.4.0</version>
    <configuration>
        <descriptors>
            <descriptor>src/main/assembly/zip.xml</descriptor>
        </descriptors>
    </configuration>
</plugin>
```

## 配置插件

插件配置有两种方式：

### 方式一：通过 `<configuration>`

```xml title="pom.xml"
<build>
    <plugins>
        <plugin>
            <groupId>org.apache.maven.plugins</groupId>
            <artifactId>maven-compiler-plugin</artifactId>
            <configuration>
                <source>17</source>
                <target>17</target>
            </configuration>
        </plugin>
    </plugins>
</build>
```

### 方式二：通过 properties

```xml title="pom.xml"
<properties>
    <maven.compiler.source>17</maven.compiler.source>
    <maven.compiler.target>17</maven.compiler.target>
    <maven.compiler.compilerVersion>17</maven.compiler.compilerVersion>
</properties>
```

properties 方式是 Super POM 中预定义的简写。

## 自定义插件开发

有时候现有插件无法满足需求，需要开发自定义插件。

### 创建插件项目

```bash
mvn archetype:generate \
    -DgroupId=com.example \
    -DartifactId=my-maven-plugin \
    -DarchetypeGroupId=org.apache.maven.archetypes \
    -DarchetypeArtifactId=maven-archetype-plugin
```

### 定义 Mojo

Mojo = Maven POJO = Maven 的任务单元

```java title="src/main/java/com/example/MyMojo.java"
package com.example;

import org.apache.maven.plugin.AbstractMojo;
import org.apache.maven.plugin.MojoExecutionException;
import org.apache.maven.plugins.annotations.LifecyclePhase;
import org.apache.maven.plugins.annotations.Mojo;
import org.apache.maven.plugins.annotations.Parameter;

@Mojo(name = "greet", defaultPhase = LifecyclePhase.PROCESS_SOURCES)
public class MyMojo extends AbstractMojo {

    @Parameter(property = "greet.name", defaultValue = "World")
    private String name;

    public void execute() throws MojoExecutionException {
        getLog().info("Hello, " + name + "!");
    }
}
```

### 使用自定义插件

```xml title="pom.xml"
<plugin>
    <groupId>com.example</groupId>
    <artifactId>my-maven-plugin</artifactId>
    <version>1.0-SNAPSHOT</version>
    <configuration>
        <name>Developer</name>
    </configuration>
</plugin>
```

```bash
mvn my:greet
# 输出：Hello, Developer!

mvn my:greet -Dgreet.name=John
# 输出：Hello, John!
```

## 常用插件清单

| 插件 | 用途 | 常用配置 |
|---|---|---|
| `maven-compiler-plugin` | 编译 Java | source, target |
| `maven-surefire-plugin` | 运行测试 | testFailureIgnore |
| `maven-jar-plugin` | 打包 JAR | manifest |

| `maven-dependency-plugin` | 依赖分析 | outputDirectory |
| `versions-maven-plugin` | 版本管理 | - |

## 面试追问

**面试官可能问**：Maven 的插件和阶段是什么关系？

这是一个理解 Maven 架构的核心问题。生命周期定义阶段顺序，插件定义在特定阶段做什么。Maven 的设计哲学是「约定优于配置」——Super POM 已经绑定了常用插件到对应阶段，所以默认情况下用户不需要关心插件。但如果需要自定义行为，可以显式配置插件或绑定额外目标。

**面试官可能问**：如何跳过 Maven 的单元测试但保留集成测试？

这需要理解 surefire 和 failsafe 两个插件的区别。surefire-plugin 运行单元测试，failsafe-plugin 运行集成测试。使用 `mvn verify -DskipTests` 会跳过单元测试但运行集成测试；`mvn verify -Dmaven.test.skip=true` 会跳过所有测试（编译也跳过）。对于持续集成流程，建议只跳过单元测试，保留集成测试确保质量。
