# Maven 多模块项目

当项目规模超过一定阈值，单一模块的架构就不再适用。你需要将系统拆分为多个模块：公共模块、API 模块、业务模块等。多模块项目（Multi-Module Project）是大型 Java 项目的事实标准。

这一篇，我们深入讲解 Maven 多模块项目的设计与实践。

## 为什么要用多模块

想象一个电商系统：

```
电商系统
 ├── common        # 通用工具类
 ├── user-api      # 用户服务接口
 ├── user-service  # 用户服务实现
 ├── product-api   # 商品服务接口
 ├── product-service
 ├── order-api
 ├── order-service
 └── gateway       # API 网关
```

**模块化的好处**：

| 好处 | 说明 |
|---|---|
| 职责分离 | 每个模块负责单一职责 |
| 代码复用 | common 模块可被所有模块依赖 |
| 独立构建 | 可以单独构建某个模块 |
| 独立部署 | 微服务架构下可以独立部署 |
| 团队自治 | 不同团队负责不同模块 |

## 项目结构

典型的多模块项目结构：

```
parent-project/
├── pom.xml                    # 父 POM
├── common/
│   ├── pom.xml
│   └── src/main/java/
├── user-api/
│   ├── pom.xml
│   └── src/main/java/
├── user-service/
│   ├── pom.xml
│   └── src/main/java/
└── user-web/
    ├── pom.xml
    └── src/main/java/
```

根目录的 `pom.xml` 使用 `<packaging>pom</packaging>`，只负责聚合子模块，不产生构建产物。

## 父 POM 设计

父 POM 是多模块项目的核心，它负责：

1. 定义通用依赖和插件
2. 统一管理版本
3. 配置构建参数
4. 聚合子模块

### 标准父 POM

```xml title="pom.xml"
<?xml version="1.0" encoding="UTF-8"?>
<project>
    <modelVersion>4.0.0</modelVersion>

    <groupId>com.example</groupId>
    <artifactId>parent-project</artifactId>
    <version>1.0.0</version>
    <packaging>pom</packaging>

    <!-- 子模块列表 -->
    <modules>
        <module>common</module>
        <module>user-api</module>
        <module>user-service</module>
        <module>user-web</module>
    </modules>

    <!-- 版本管理 -->
    <properties>
        <java.version>17</java.version>
        <spring-boot.version>2.7.0</spring-boot.version>
        <mybatis-spring-boot.version>2.2.2</mybatis-spring-boot.version>
    </properties>

    <!-- 依赖管理（只声明，不引入） -->
    <dependencyManagement>
        <dependencies>
            <!-- Spring Boot -->
            <dependency>
                <groupId>org.springframework.boot</groupId>
                <artifactId>spring-boot-dependencies</artifactId>
                <version>${spring-boot.version}</version>
                <type>pom</type>
                <scope>import</scope>
            </dependency>
        </dependencies>
    </dependencyManagement>

    <!-- 公共依赖（会自动传递） -->
    <dependencies>
        <dependency>
            <groupId>org.projectlombok</groupId>
            <artifactId>lombok</artifactId>
            <scope>provided</scope>
        </dependency>
    </dependencies>

    <!-- 公共插件配置 -->
    <build>
        <pluginManagement>
            <plugins>
                <plugin>
                    <groupId>org.springframework.boot</groupId>
                    <artifactId>spring-boot-maven-plugin</artifactId>
                    <version>${spring-boot.version}</version>
                </plugin>
            </plugins>
        </pluginManagement>
    </build>
</project>
```

## 子模块 POM

子模块 POM 需要声明父 POM：

```xml title="user-service/pom.xml"
<?xml version="1.0" encoding="UTF-8"?>
<project>
    <modelVersion>4.0.0</modelVersion>

    <!-- 继承父 POM -->
    <parent>
        <groupId>com.example</groupId>
        <artifactId>parent-project</artifactId>
        <version>1.0.0</version>
    </parent>

    <!-- 子模块坐标 -->
    <artifactId>user-service</artifactId>

    <!-- 依赖子模块 -->
    <dependencies>
        <!-- 依赖 common 模块 -->
        <dependency>
            <groupId>com.example</groupId>
            <artifactId>common</artifactId>
            <version>${project.version}</version>
        </dependency>

        <!-- 依赖 user-api（本地仓库中的） -->
        <dependency>
            <groupId>com.example</groupId>
            <artifactId>user-api</artifactId>
            <version>${project.version}</version>
        </dependency>

        <!-- Spring Boot Starter（不需要指定版本） -->
        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-starter-web</artifactId>
        </dependency>

        <!-- MyBatis（不需要指定版本，从父 POM 继承） -->
        <dependency>
            <groupId>org.mybatis.spring.boot</groupId>
            <artifactId>mybatis-spring-boot-starter</artifactId>
        </dependency>
    </dependencies>

    <!-- 插件配置（继承自父 POM 的 pluginManagement） -->
    <build>
        <plugins>
            <plugin>
                <groupId>org.springframework.boot</groupId>
                <artifactId>spring-boot-maven-plugin</artifactId>
            </plugin>
        </plugins>
    </build>
</project>
```

## Reactor 构建顺序

Maven 的 Reactor（反应堆）机制自动分析模块间的依赖关系，确定正确的构建顺序。

### 依赖决定顺序

```
common ← user-api ← user-service ← user-web
  ↑
  └── (no dependencies)
```

构建顺序：`common` → `user-api` → `user-service` → `user-web`

### 查看 Reactor 顺序

```bash
mvn reactor:make -Dmake.artifact

# 或在构建时查看
mvn clean install -pl :parent-project -N
mvn clean install -pl :common
mvn clean install -pl :user-api,:user-service,:user-web -am
```

参数说明：
- `-pl`：指定要构建的项目（projects）
- `-am`：同时构建所选项目的依赖（also-make）

### 排除特定模块

```bash
# 构建除 user-web 以外的所有模块
mvn clean install -pl !:user-web
```

## 父子 POM 的版本管理

父子 POM 中的 `${project.version}` 是一个重要的变量，它引用当前项目的版本：

```xml title="父 pom.xml"
<artifactId>parent-project</artifactId>
<version>1.0.0</version>
```

```xml title="子 pom.xml"
<!-- 使用父项目的版本 -->
<parent>
    <groupId>com.example</groupId>
    <artifactId>parent-project</artifactId>
    <version>1.0.0</version>
</parent>

<!-- 依赖其他子模块时，使用 ${project.version} -->
<dependency>
    <groupId>com.example</groupId>
    <artifactId>common</artifactId>
    <version>${project.version}</version>
</dependency>
```

子模块继承父 POM 的版本号。这确保了所有模块使用同一版本，简化了版本管理。

## 常见问题

### 问题一：循环依赖

```bash
[ERROR] [ERROR] Circular dependency: user-service -> user-web -> user-service
```

这是严重的架构问题。解决方案是引入中间层，或重新拆分模块。

### 问题二：模块间版本不一致

当子模块尝试引入不同版本的同一模块时，Maven 会报错或选择版本：

```bash
# 常见警告
[WARNING] Some problems were encountered while building the effective model for user-service:1.0.0
[WARNING] 'dependencyManagement.dependencies.dependency.version' for com.example:common:${project.version} is empty.
```

确保父 POM 的 `<modules>` 顺序正确，或者使用 `mvn install` 先安装依赖模块。

## 多层继承结构

大型项目可能需要多层继承：

```
super-pom (爷爷：定义依赖版本 BOM)
    ↓
parent-pom (父亲：聚合子模块 + 公共依赖)
    ↓
user-service (具体实现)
```

```xml title="super-pom/pom.xml"
<!-- 爷爷 POM：只管理版本 -->
<project>
    <artifactId>super-pom</artifactId>
    <version>1.0.0</version>
    <packaging>pom</packaging>

    <dependencyManagement>
        <dependencies>
            <dependency>
                <groupId>org.springframework.boot</groupId>
                <artifactId>spring-boot-dependencies</artifactId>
                <version>2.7.0</version>
                <type>pom</type>
                <scope>import</scope>
            </dependency>
        </dependencies>
    </dependencyManagement>
</project>
```

```xml title="parent-pom/pom.xml"
<!-- 父亲 POM：聚合子模块 + 公共配置 -->
<project>
    <parent>
        <groupId>com.example</groupId>
        <artifactId>super-pom</artifactId>
        <version>1.0.0</version>
    </parent>
    <artifactId>parent-pom</artifactId>
    <packaging>pom</packaging>

    <modules>
        <module>user-service</module>
        <module>product-service</module>
    </modules>
</project>
```

## 面试追问

**面试官可能问**：多模块项目中，父子 POM 应该如何配合？

这是一个架构设计问题。父 POM 应该只做三件事：聚合子模块（`<modules>`）、管理版本（`<dependencyManagement>`）、定义公共依赖。子模块 POM 负责具体实现和模块特定的依赖。这样的设计确保了版本统一、子模块解耦、构建灵活。

**面试官可能问**：Maven 如何确定多模块项目的构建顺序？

Maven 使用 Reactor 机制分析模块间的依赖关系图（DAG）。如果模块 A 依赖模块 B，Maven 会确保 B 先构建。这个分析是自动的，但你也可以通过 `-pl` 和 `-am` 参数手动控制构建范围。
