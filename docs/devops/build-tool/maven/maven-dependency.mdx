# Maven 依赖管理

你遇到过这样的问题吗？项目里莫名其妙多了几个 JAR 包，或者运行时 `ClassNotFoundException`，或者同一个依赖有两个版本「打架」。这些问题，统统指向一个核心话题：**Maven 依赖管理**。

Maven 的依赖管理是其最强大的特性之一，也是最容易出问题的地方。理解依赖解析机制，是排查问题的关键。

## 传递依赖

Maven 最有价值的特性之一是**传递依赖（Transitive Dependencies）**。当你的项目依赖 A，而 A 依赖 B，那么 B 会自动成为你项目的依赖。

```xml title="pom.xml"
<!-- 项目直接依赖 spring-boot-starter-web -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
    <version>2.7.0</version>
</dependency>
```

`spring-boot-starter-web` 本身只声明了核心依赖，这些依赖又会传递依赖其他库。一条声明，最终拉取了数十个 JAR 包。这就是传递依赖的威力。

### 传递依赖的深度

传递依赖可以有多层：

```
项目
 └── spring-boot-starter-web (第1层)
      └── spring-boot-starter (第2层)
           └── spring-boot (第3层)
                └── spring-core (第4层)
                     └── ...
```

Maven 默认不限制传递依赖的深度。但深度越大，依赖越难控制，冲突概率越高。

## 依赖调解

当同一个依赖通过多条路径被引入时，Maven 使用**最近定义优先（Nearest Definition）**策略选择版本。

### 场景示例

```
项目
 ├── 依赖 A (v1.0)
 │    └── 依赖 X (v1.0)
 └── 依赖 B (v1.0)
      └── 依赖 X (v2.0)
```

依赖 X 通过两条路径被引入：`A -> X v1.0` 和 `B -> X v2.0`。A 和 B 都是项目的直接依赖，深度相同。Maven 选择**先声明**的依赖对应的版本。

```xml title="pom.xml"
<!-- B 在 A 之前声明，所以选择 B -> X 的版本 -->
<dependencies>
    <dependency>
        <groupId>com.example</groupId>
        <artifactId>B</artifactId>
        <version>1.0</version>
    </dependency>
    <dependency>
        <groupId>com.example</groupId>
        <artifactId>A</artifactId>
        <version>1.0</version>
    </dependency>
</dependencies>
```

### 路径深度不同时

```
项目
 └── 依赖 A (v1.0)
      └── 依赖 X (v1.0)
 └── 依赖 B (v1.0)
```

B 是直接依赖，深度为 0；X 通过 A 引入，深度为 1。虽然 A 先声明，但 B 更近，所以选择 B。

## 排除传递依赖

有时候，某个传递依赖不是你需要的，或者版本有冲突，需要排除它。

### 使用 `<exclusions>`

```xml title="pom.xml"
<dependency>
    <groupId>com.example</groupId>
    <artifactId>myapp</artifactId>
    <version>1.0.0</version>

    <exclusions>
        <!-- 排除传递依赖中的 commons-lang3 -->
        <exclusion>
            <groupId>org.apache.commons</groupId>
            <artifactId>commons-lang3</artifactId>
        </exclusion>
    </exclusions>
</dependency>
```

这样项目就不会引入 `commons-lang3` 了。如果你需要这个库，可以显式声明一个特定版本。

### 使用 `<optional>`

`<optional>` 标记的依赖，不会被传递：

```xml title="pom.xml"
<dependency>
    <groupId>com.example</groupId>
    <artifactId>feature-lib</artifactId>
    <version>1.0.0</version>

    <!-- 这个依赖不会被传递到依赖 feature-lib 的项目 -->
    <optional>true</optional>
</dependency>
```

适用场景：某个库是可选功能的一部分，只有使用特定配置时才需要。

## 依赖冲突

依赖冲突是实际项目中最常见的问题。当两个依赖需要同一个库的不同版本时，冲突就发生了。

### 冲突的表现

运行时 `NoSuchMethodError` 或 `ClassNotFoundException`：

```
java.lang.NoSuchMethodError: org.apache.commons.lang3.StringUtils.isBlank(Ljava/lang/String;)Z
```

这说明编译时用的 `commons-lang3` 版本和运行时加载的版本不一致。

### 排查工具

**`mvn dependency:tree`**

```bash
# 查看完整依赖树
mvn dependency:tree

# 过滤特定依赖
mvn dependency:tree -Dincludes=com.alibaba:fastjson

# 输出到文件
mvn dependency:tree > dependency-tree.txt
```

**`mvn dependency:analyze`**

```bash
# 分析依赖
mvn dependency:analyze
```

这个命令检查：

| 警告 | 说明 |
|---|---|
| `Used undeclared dependencies` | 代码使用的依赖，但没有显式声明 |
| `Unused declared dependencies` | 声明了但未使用的依赖 |
| `Non-test dependency used in test code` | 测试代码引用了非测试依赖 |

**`mvn dependency:tree -Dverbose`**

verbose 模式显示依赖是如何被引入的：

```bash
mvn dependency:tree -Dverbose | grep -A5 "commons-lang3"
```

输出会显示每个 `commons-lang3` 是通过哪个依赖引入的。

### 解决冲突

**方法一：显式声明版本**

在 `dependencyManagement` 中声明版本，Maven 会统一使用这个版本：

```xml title="pom.xml"
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>org.apache.commons</groupId>
            <artifactId>commons-lang3</artifactId>
            <version>3.12.0</version>
        </dependency>
    </dependencies>
</dependencyManagement>
```

**方法二：排除低版本依赖**

```xml title="pom.xml"
<dependency>
    <groupId>com.example</groupId>
    <artifactId>old-lib</artifactId>
    <version>1.0.0</version>
    <exclusions>
        <exclusion>
            <groupId>commons-lang</groupId>
            <artifactId>commons-lang</artifactId>
        </exclusion>
    </exclusions>
</dependency>
```

**方法三：使用 `mvn versions:display-dependency-updates`**

```bash
# 检查依赖更新
mvn versions:display-dependency-updates
```

这个命令列出所有有新版本的依赖，帮助你升级。

## 有效作用域

传递依赖的作用域不一定和直接依赖一样。Maven 根据依赖关系计算**有效作用域（Effective Scope）**。

### 作用域计算规则

| 直接依赖 \ 传递依赖 | compile | provided | runtime | test |
|---|---|---|---|---|
| compile | compile | - | runtime | - |
| provided | provided | provided | provided | - |
| runtime | runtime | - | runtime | - |
| test | test | - | test | - |

表格解读：如果你的项目直接依赖 A（compile 作用域），A 传递依赖 B（B 是 compile 作用域），那么 B 的有效作用域是 compile。A 传递依赖 C（C 是 provided 作用域），那么 C 的有效作用域是 provided（不会打包）。

## 依赖验证

Maven 提供了依赖验证功能，检查 POM 中的依赖声明是否正确：

```bash
# 验证依赖
mvn dependency:validate
```

在 CI/CD 流程中，可以加入这个检查确保依赖配置正确：

```yaml title="Jenkinsfile"
stage('Verify') {
    steps {
        sh 'mvn dependency:validate'
    }
}
```

## 面试追问

**面试官可能问**：Maven 如何解决依赖冲突？请描述一下依赖调解的算法。

依赖调解使用最近定义优先策略：当同一个依赖通过多条路径引入时，选择路径最短的；如果路径长度相同，选择 POM 中先声明的。这个策略简单有效，但有时不是我们想要的版本。可以通过 `dependencyManagement` 强制指定版本，或使用 `exclusions` 排除不需要的传递依赖。理解这个机制，才能在实际项目中灵活控制依赖树。

**面试官可能问**：如何排查 `NoSuchMethodError` 这样的依赖冲突？

这是一个经典的排查流程：首先用 `mvn dependency:tree -Dverbose` 查看依赖树，找到冲突版本的来源；然后在导致冲突的依赖上添加 `exclusions`，排除低版本；最后显式声明需要的版本。排查这类问题需要耐心，但只要掌握了 `dependency:tree` 这个工具，就能快速定位。
