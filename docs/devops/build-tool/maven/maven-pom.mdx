# Maven POM 详解

POM（Project Object Model）是 Maven 项目的核心配置文件。一个 `pom.xml` 文件定义了项目的一切：坐标、依赖、插件、构建配置、-profile 配置。

理解 POM 的结构，是精通 Maven 的基础。

## POM 基本结构

一个完整的 `pom.xml` 结构如下：

```xml title="pom.xml"
<?xml version="1.0" encoding="UTF-8"?>
<project xmlns="http://maven.apache.org/POM/4.0.0"
         xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
         xsi:schemaLocation="http://maven.apache.org/POM/4.0.0
         http://maven.apache.org/xsd/maven-4.0.0.xsd">
    <modelVersion>4.0.0</modelVersion>

    <!-- ========== 坐标 ========== -->
    <groupId>com.example</groupId>
    <artifactId>user-service</artifactId>
    <version>1.0.0-SNAPSHOT</version>
    <packaging>jar</packaging>

    <!-- ========== 名称与描述 ========== -->
    <name>User Service</name>
    <description>用户服务模块</description>
    <url>https://github.com/example/user-service</url>

    <!-- ========== 继承 ========== -->
    <parent>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-parent</artifactId>
        <version>2.7.0</version>
    </parent>

    <!-- ========== 属性 ========== -->
    <properties>
        <java.version>17</java.version>
        <project.build.sourceEncoding>UTF-8</project.build.sourceEncoding>
    </properties>

    <!-- ========== 依赖 ========== -->
    <dependencies>
        <!-- 依赖声明 -->
    </dependencies>

    <!-- ========== 依赖管理 ========== -->
    <dependencyManagement>
        <dependencies>
            <!-- 版本控制 -->
        </dependencies>
    </dependencyManagement>

    <!-- ========== 构建配置 ========== -->
    <build>
        <plugins>
            <!-- 插件配置 -->
        </plugins>
    </build>

    <!-- ========== Profile 配置 ========== -->
    <profiles>
        <!-- 环境配置 -->
    </profiles>
</project>
```

## `<dependencies>` vs `<dependencyManagement>`

这两个配置是 POM 中最容易混淆的部分。

### dependencies

`<dependencies>` 中声明的依赖会**直接继承**到子模块，子模块不需要指定版本：

```xml title="父 pom.xml"
<dependencies>
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-web</artifactId>
    </dependency>
</dependencies>
```

```xml title="子 pom.xml"
<!-- 子模块自动继承父 POM 的依赖，不需要指定版本 -->
<dependencies>
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-web</artifactId>
        <!-- 不需要 version，Maven 会从父 POM 继承 -->
    </dependency>
</dependencies>
```

### dependencyManagement

`<dependencyManagement>` 只声明**版本**，不实际引入依赖。子模块需要显式声明依赖：

```xml title="父 pom.xml"
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>com.alibaba</groupId>
            <artifactId>fastjson</artifactId>
            <version>1.2.83</version>
        </dependency>
    </dependencies>
</dependencyManagement>
```

```xml title="子 pom.xml"
<!-- 子模块需要声明依赖，但可以不指定版本 -->
<dependencies>
    <dependency>
        <groupId>com.alibaba</groupId>
        <artifactId>fastjson</artifactId>
        <!-- 可以省略 version，会使用 dependencyManagement 中定义的版本 -->
    </dependency>
</dependencies>
```

### 如何选择

| 方式 | 适用场景 |
|---|---|
| `<dependencies>` | 父 POM 强制子模块使用的依赖 |
| `<dependencyManagement>` | 父 POM 只管理版本，子模块自行决定是否使用 |

大型多模块项目推荐使用 `<dependencyManagement>`，给予子模块更多灵活性，同时统一管理版本。

## build 配置

`<build>` 配置控制项目的构建过程：

```xml title="pom.xml"
<build>
    <!-- 源码目录（默认 src/main/java） -->
    <sourceDirectory>${project.basedir}/src/main/java</sourceDirectory>

    <!-- 测试源码目录 -->
    <testSourceDirectory>${project.basedir}/src/test/java</testSourceDirectory>

    <!-- 资源文件配置 -->
    <resources>
        <resource>
            <directory>src/main/resources</directory>
            <filtering>true</filtering>  <!-- 启用变量替换 -->
        </resource>
    </resources>

    <!-- 插件配置 -->
    <plugins>
        <plugin>
            <groupId>org.apache.maven.plugins</groupId>
            <artifactId>maven-compiler-plugin</artifactId>
            <configuration>
                <source>17</source>
                <target>17</target>
            </configuration>
        </plugin>
    </plugins>
</build>
```

### 资源过滤

资源过滤可以替换 `src/main/resources` 中的配置文件变量：

```xml title="pom.xml"
<build>
    <resources>
        <resource>
            <directory>src/main/resources</directory>
            <filtering>true</filtering>
        </resource>
    </resources>
</build>
```

```properties title="src/main/resources/application.properties"
# 变量会被 pom.xml 中的 properties 替换
db.url=${db.url}
db.username=${db.username}
```

## profiles 配置

Profile 允许为不同环境提供不同的构建配置：

```xml title="pom.xml"
<profiles>
    <!-- 开发环境 -->
    <profile>
        <id>dev</id>
        <activation>
            <activeByDefault>true</activeByDefault>
        </activation>
        <properties>
            <env>development</env>
            <db.url>jdbc:mysql://localhost:3306/dev</db.url>
        </properties>
    </profile>

    <!-- 测试环境 -->
    <profile>
        <id>test</id>
        <properties>
            <env>test</env>
            <db.url>jdbc:mysql://test-server:3306/test</db.url>
        </properties>
    </profile>

    <!-- 生产环境 -->
    <profile>
        <id>prod</id>
        <properties>
            <env>production</env>
            <db.url>jdbc:mysql://prod-server:3306/prod</db.url>
        </properties>
    </profile>
</profiles>
```

### 激活 Profile

```bash
# 命令行激活
mvn package -Pprod

# 自动激活（根据 JDK 版本）
<profile>
    <id>jdk17</id>
    <activation>
        <jdk>[17,)</jdk>
    </activation>
    ...
</profile>

# 根据文件存在激活
<profile>
    <id>env-dev</id>
    <activation>
        <file>
            <exists>dev.properties</exists>
        </file>
    </activation>
    ...
</profile>
```

### Profile 中的依赖和插件

Profile 不仅可以定义 properties，还可以添加依赖或配置插件：

```xml title="pom.xml"
<profiles>
    <profile>
        <id>dev</id>
        <dependencies>
            <!-- 开发环境额外依赖 -->
            <dependency>
                <groupId>com.h2database</groupId>
                <artifactId>h2</artifactId>
                <scope>runtime</scope>
            </dependency>
        </dependencies>
    </profile>

    <profile>
        <id>prod</id>
        <dependencies>
            <!-- 生产环境依赖 -->
            <dependency>
                <groupId>com.mysql</groupId>
                <artifactId>mysql-connector-java</artifactId>
            </dependency>
        </dependencies>
    </profile>
</profiles>
```

## 继承 vs 聚合

Maven 的多模块项目有两种组织方式：

### 继承关系

父 POM 定义通用的配置和依赖，子 POM 继承：

```xml title="父 pom.xml"
<project>
    <groupId>com.example</groupId>
    <artifactId>parent</artifactId>
    <version>1.0.0</version>
    <packaging>pom</packaging>

    <modules>
        <module>user-api</module>
        <module>user-service</module>
    </modules>
</project>
```

```xml title="子 pom.xml"
<project>
    <parent>
        <groupId>com.example</groupId>
        <artifactId>parent</artifactId>
        <version>1.0.0</version>
    </parent>
    <artifactId>user-service</artifactId>
</project>
```

### 聚合关系

通过 `<modules>` 在父 POM 中声明子模块：

```xml title="父 pom.xml"
<project>
    <groupId>com.example</groupId>
    <artifactId>multi-module</artifactId>
    <version>1.0.0</version>
    <packaging>pom</packaging>

    <modules>
        <module>module-a</module>
        <module>module-b</module>
        <module>module-c</module>
    </modules>
</project>
```

父 POM 的 `<packaging>` 必须为 `pom`，因为它本身不打包。

## 面试追问

**面试官可能问**：`dependencies` 和 `dependencyManagement` 的区别是什么？各自适用场景？

这是 Maven 多模块项目的核心问题。`<dependencies>` 中的依赖会自动传递给子模块，适合父 POM 强制子模块必须使用的依赖（如 Spring Boot Starter）。`<dependencyManagement>` 只声明版本，不自动传递，适合需要子模块自行决定是否使用的依赖，或需要统一管理版本但不希望强制继承的场景。大型项目通常两者结合使用：常用依赖用 `<dependencies>`，可选依赖用 `<dependencyManagement>`。

**面试官可能问**：POM 中的 `<parent>` 和 `<modules>` 是什么关系？

`<parent>` 定义继承关系，子 POM 从父 POM 继承配置；`<modules>` 定义聚合关系，父 POM 通过它管理子模块。一个项目可以同时有父和子：它可以 `<parent>` 继承另一个项目，同时 `<modules>` 聚合自己的子模块。典型的三层结构是：顶层是爷爷 POM（定义 BOM），中间层是父 POM（聚合子模块），底层是子模块 POM（具体实现）。
