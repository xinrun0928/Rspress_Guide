# Maven 核心概念

Maven 的一切都围绕一个核心概念：**坐标（Coordinates）**。每个 artifact（JAR 包、 WAR 包等）通过坐标唯一标识，仓库通过坐标存储和检索 artifact。理解坐标，是理解 Maven 的第一步。

## Maven 坐标体系

Maven 坐标由四个元素组成：

```xml title="pom.xml"
<project>
    <!-- 组织 ID，如 com.alibaba -->
    <groupId>com.example</groupId>

    <!-- 项目名，如 user-service -->
    <artifactId>user-service</artifactId>

    <!-- 版本号，支持 SNAPSHOT -->
    <version>1.0.0-SNAPSHOT</version>

    <!-- 打包类型，默认 jar -->
    <packaging>jar</packaging>
</project>
```

| 元素 | 说明 | 示例 |
|---|---|---|
| `groupId` | 组织标识，通常是域名的反写 | `com.alibaba` |
| `artifactId` | 项目或模块名 | `fastjson` |
| `version` | 版本号，支持 `SNAPSHOT` | `1.2.3`、`2.0.0-SNAPSHOT` |
| `packaging` | 打包类型 | `jar`、`war`、`pom`、`ear` |

这四个元素唯一确定一个 artifact。坐标转换为仓库路径：

```
com/example/user-service/1.0.0/user-service-1.0.0.jar
```

### SNAPSHOT vs Release

版本号中的 `SNAPSHOT` 是一个特殊标记：

| 类型 | 说明 | 用途 |
|---|---|---|
| `1.0.0` | Release 版本 | 正式发布，版本锁定 |
| `1.0.0-SNAPSHOT` | 快照版本 | 开发期间，实时更新 |

SNAPSHOT 版本的特点：

- 开发期间，仓库会检查远程仓库，每次构建可能下载新版本
- 发布前，需要将 SNAPSHOT 升级为正式版本
- CI/CD 中，SNAPSHOT 用于团队内部的持续集成

:::warning 生产环境禁止 SNAPSHOT
生产环境的依赖必须锁定为 Release 版本。SNAPSHOT 的不确定性会导致「在我机器上能跑」却在生产环境失败的经典问题。
:::

## 仓库模型

Maven 的仓库是 artifact 的存储仓库，分为三种类型：

### 本地仓库

本地仓库位于 `~/.m2/repository`（Windows 是 `C:\Users\<用户名>\.m2\repository`）。Maven 下载的依赖都缓存在这里。

```bash
# 查看本地仓库位置
echo $MAVEN_HOME  # 或
mvn -v | grep "Local repository"
```

首次构建时，Maven 会从远程仓库下载依赖到本地仓库，后续构建直接从本地读取，速度很快。

### 远程仓库

远程仓库包括中央仓库和私服：

**中央仓库**（Maven Central）是 Maven 的官方仓库，地址是 `https://repo.maven.apache.org/maven2`。全世界大部分开源 Java 库都在这里。

**私服**是企业自建的 Maven 仓库，常用软件有 Nexus 和 Artifactory。私服有三大作用：

| 类型 | 说明 |
|---|---|
| 代理 | 代理中央仓库，缓存依赖，加速下载 |
| 托管 | 存储公司内部的私有 JAR 包 |
| 聚合 | 组合多个仓库，提供统一访问入口 |

### 仓库配置

在 `pom.xml` 中配置仓库：

```xml title="pom.xml"
<repositories>
    <repository>
        <id>central</id>
        <url>https://repo.maven.apache.org/maven2</url>
    </repository>

    <!-- 添加阿里云镜像（国内加速） -->
    <repository>
        <id>aliyun</id>
        <url>https://maven.aliyun.com/repository/public</url>
    </repository>
</repositories>
```

或在 `settings.xml` 中配置镜像（更推荐，影响全局）：

```xml title="settings.xml"
<mirrors>
    <mirror>
        <id>aliyun</id>
        <mirrorOf>central</mirrorOf>
        <name>Aliyun Maven</name>
        <url>https://maven.aliyun.com/repository/public</url>
    </mirror>
</mirrors>
```

## POM 继承体系

Maven 的 POM 文件有严格的继承关系。每个项目的 POM 隐式继承 **Super POM**，Super POM 定义了默认配置：

```bash
# 查看 Super POM 内容
mvn help:effective-pom
```

这条命令输出有效 POM，可以看到 Super POM 的内容。

### Super POM 的默认配置

```xml title="Super POM 关键配置"
<!-- 默认仓库 -->
<repositories>
    <repository>
        <id>central</id>
        <url>https://repo.maven.apache.org/maven2</url>
    </repository>
</repositories>

<!-- 默认插件仓库 -->
<pluginRepositories>
    <pluginRepository>
        <id>central</id>
        <url>https://repo.maven.apache.org/maven2</url>
    </pluginRepository>
</pluginRepositories>

<!-- 标准目录结构 -->
<sourceDirectory>src/main/java</sourceDirectory>
<testSourceDirectory>src/test/java</testSourceDirectory>
```

### 项目 POM 继承

子 POM 可以覆盖父 POM 的配置：

```xml title="子 pom.xml"
<project>
    <parent>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-parent</artifactId>
        <version>2.7.0</version>
    </parent>

    <groupId>com.example</groupId>
    <artifactId>myapp</artifactId>
    <version>1.0.0</version>

    <!-- 覆盖父 POM 的 Java 版本 -->
    <properties>
        <java.version>17</java.version>
    </properties>
</project>
```

继承 Spring Boot Parent 的好处：

- 统一管理依赖版本
- 继承默认的插件配置
- 自动配置资源过滤
- 默认的 Java 版本

## 依赖解析流程

当你执行 `mvn package` 时，Maven 按以下步骤解析依赖：

```
1. 解析当前 POM 的 <dependencies>
         ↓
2. 检查本地仓库是否存在
         ↓ (不存在)
3. 从远程仓库下载到本地仓库
         ↓
4. 解析传递依赖（依赖的依赖）
         ↓
5. 检测依赖冲突，执行依赖调解
         ↓
6. 输出最终依赖列表
```

### 依赖下载过程

```bash
# 查看依赖下载过程
mvn dependency:resolve -X
```

`-X` 参数输出调试信息，可以看到每个依赖从哪里下载、版本是多少。

## 面试追问

**面试官可能问**：Maven 的 groupId、artifactId、version 分别对应什么场景？

这个问题看似简单，但能考察候选人对 Maven 设计的理解。`groupId` 通常对应组织或公司的域名，比如 `com.alibaba`、`org.apache`；`artifactId` 是项目或模块的名字，应该能清晰表达功能，如 `user-service`、`payment-sdk`；`version` 遵循语义化版本规范，`SNAPSHOT` 用于开发期，`Release` 用于正式发布。

**面试官可能问**：本地仓库和远程仓库的区别是什么？私服的作用是什么？

本地仓库是本地缓存，只影响本机；远程仓库是 artifact 的来源，私服除了代理中央仓库，更重要的是托管公司内部的项目产物，实现版本管理和访问控制。在大型团队中，私服还可以聚合多个仓库、配置访问权限，是企业级依赖管理的核心基础设施。
