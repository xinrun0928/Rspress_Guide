# Gradle 守护进程与缓存

你有没有想过——为什么第一次运行 Gradle 较慢，但之后每次都很快？

Gradle 守护进程（Daemon）和构建缓存是 Gradle 快速构建的两大秘密武器。守护进程避免重复启动 JVM，缓存避免重复计算和执行。两者配合，可以让构建时间从分钟级降到秒级。

## Gradle 守护进程（Daemon）

### 什么是 Daemon

Daemon 是一个长期运行的 JVM 进程，复用 JVM 实例，避免每次构建都重新启动。

```
Without Daemon:
$ gradle build
  启动 JVM (2s) → 加载 Gradle (3s) → 执行任务 (5s) → 关闭 JVM
  总计：10s

With Daemon:
$ gradle build  (第一次)
  启动 JVM (2s) → 加载 Gradle (3s) → 执行任务 (5s) → 保持 JVM 存活

$ gradle build  (后续)
  ← 从 Daemon 恢复 (0.1s) → 执行任务 (5s)
  总计：5.1s
```

### 启用/禁用 Daemon

```properties
# gradle.properties
org.gradle.daemon=true   # 启用（默认）
org.gradle.daemon=false  # 禁用
```

```bash
# 命令行临时禁用
gradle build --no-daemon
```

### Daemon 管理命令

```bash
# 查看运行中的 Daemon
gradle --status

# 示例输出：
#   PID   USER   ADDR     PORT  STATUS
#  12345  user   localhost  57342 IDLE
#  12346  user   localhost  57343 BUSY
#
# Gradle 8.0 will stop supporting less than Java 11.
# Gradle 8.5 will stop supporting less than Java 17.
# There are 2 Daemon processes running for this command.
# 1 was stopped. The Daemon will restart automatically if
# it is shut down cleanly.

# 停止所有 Daemon
gradle --stop

# 查看 Daemon 日志
cat ~/.gradle/daemon/8.5/daemon-*.log
```

### Daemon 配置参数

```properties
# gradle.properties

# JVM 参数
org.gradle.jvmargs=-Xmx4g -XX:+HeapDumpOnOutOfMemoryError \
  -Dfile.encoding=UTF-8 \
  -XX:+UseParallelGC \
  -XX:+HeapDumpOnOutOfMemoryError

# Daemon idle 时间（默认 3 小时）
org.gradle.daemon.idletimeout=1800000  # 30 分钟

# 并行构建
org.gradle.parallel=true

# Daemon 数量（默认 1）
org.gradle.daemon.max.workers=4
```

### Daemon 的生命周期

```
Daemon 状态机：

IDLE ──► BUSY ──► IDLE
 │              │
 │              │
 └── 空闲超时 ──┘
       │
       └── STOPPED
```

- **IDLE**：等待构建请求
- **BUSY**：正在执行构建
- **STOPPED**：空闲超时或被手动停止

## 配置缓存（Configuration Cache）

### 配置缓存是什么

配置缓存将 Task 图的配置结果缓存起来，避免每次构建都重新配置。

```properties
# gradle.properties
org.gradle.configuration-cache=true
```

### 配置缓存的效果

```
Without Configuration Cache:
  配置阶段：3.0s
  执行阶段：5.0s
  总计：8.0s

With Configuration Cache:
  配置阶段：0.1s (从缓存恢复)
  执行阶段：5.0s
  总计：5.1s
```

### 配置缓存的限制

```groovy
// 不支持的写法
def value = System.getenv("BUILD_NUMBER")  // 非确定性
def value = new File(".").canonicalPath     // 非确定性
def value = Math.random()                    // 非确定性
def value = URL.newInstance("https://...")   // 网络访问
```

### 调试配置缓存

```bash
# 启用并显示问题
org.gradle.configuration-cache.problems=warn

# 忽略配置缓存（用于调试）
gradle build --no-configuration-cache

# 清除配置缓存
rm -rf ~/.gradle/configuration-cache/
```

## 构建缓存（Build Cache）

### 本地构建缓存

```properties
# gradle.properties
org.gradle.caching=true

# 缓存目录（默认 ~/.gradle/caches/build-cache-1/）
org.gradle.cachedir=~/.gradle/caches/build-cache-1/
```

```bash
# 启用构建缓存
gradle build --build-cache

# 清除构建缓存
rm -rf ~/.gradle/caches/build-cache-1/
```

### 远程构建缓存

```kotlin
// build.gradle.kts
buildCache {
    remote(HttpBuildCache::class) {
        url = uri("https://buildcache.example.com/cache/")
        isEnabled = true
        push = true  // 推送缓存到远程

        credentials {
            username = "cacheuser"
            password = System.getenv("BUILDCACHE_PASSWORD") ?: ""
        }
    }
}
```

### 构建缓存键

缓存键由以下因素计算：

1. Task 类名
2. Task 输入的哈希（文件内容、属性值）
3. Gradle 版本
4. 构建环境标识（JVM 版本、操作系统）

```
Cache Key = Hash(TaskClass + InputHash + GradleVersion + EnvironmentId)
```

## 增量构建

### Task Inputs 和 Outputs

```groovy
// 定义 Task 的输入和输出
task processData {
    inputs.file "src/data/input.json"
    inputs.dir "src/data/raw"
    inputs.property "version", project.version

    outputs.file "build/processed/data.json"
    outputs.dir "build/processed"

    doLast {
        // 业务逻辑
    }
}
```

### Task 状态

| 状态 | 说明 |
|---|---|
| **EXECUTED** | Task 必须执行 |
| **UP-TO-DATE** | Task 跳过，输入输出无变化 |
| **FROM-CACHE** | 从缓存恢复 |

```bash
$ gradle build
> Task :compileJava
  UP-TO-DATE          ← 源码未变，跳过

> Task :processData
  FROM-CACHE          ← 从构建缓存恢复
```

## 并行构建

### 启用并行构建

```properties
# gradle.properties
org.gradle.parallel=true
org.gradle.workers.max=4
```

### 并行构建原理

```
Without Parallel:
Task A ──► Task B ──► Task C ──► Task D
时间线：|──A──|──B──|──C──|──D──|

With Parallel:
Task A ──► Task B ──► Task C ──► Task D
  Task A ────────────────────────────►
  Task A ──────────────────────────►
时间线：|──A──|──B──|──C──|──D──|  (部分并行)
```

### 并行构建注意事项

```groovy
// 任务间无依赖，可以并行
task taskA {
    doLast { Thread.sleep(1000) }
}

task taskB {
    doLast { Thread.sleep(1000) }
}

// 任务间有依赖，必须顺序执行
taskB.mustRunAfter(taskA)
```

## 常用优化配置

### 推荐的 gradle.properties

```properties
# gradle.properties

# JVM 配置
org.gradle.jvmargs=-Xmx4g -XX:+HeapDumpOnOutOfMemoryError -XX:+UseParallelGC -Dfile.encoding=UTF-8

# 守护进程
org.gradle.daemon=true
org.gradle.daemon.idletimeout=3600000  # 1 小时

# 并行构建
org.gradle.parallel=true
org.gradle.workers.max=4

# 构建缓存
org.gradle.caching=true

# 配置缓存
org.gradle.configuration-cache=true
org.gradle.configuration-cache.problems=warn

# 文件系统监控（增量构建优化）
org.gradle.fs.inotify=true
```

### 按需配置

```bash
# 禁用守护进程
gradle build --no-daemon

# 禁用并行
gradle build --no-parallel

# 禁用缓存
gradle build --no-build-cache

# 禁用配置缓存
gradle build --no-configuration-cache

# 全部禁用（用于调试）
gradle build --no-daemon --no-parallel --no-build-cache --no-configuration-cache
```

## 常见问题排查

### Daemon 无法启动

```bash
# 检查 Java 版本
java -version

# 手动启动 Daemon
gradle --daemon

# 查看 Daemon 日志
tail -f ~/.gradle/daemon/*/daemon-*.log
```

### 缓存不命中

```bash
# 查看 Task 缓存状态
gradle build --info | grep -E "(UP-TO-DATE|FROM-CACHE|EXECUTED)"

# 强制重新执行所有 Task
gradle build --rerun-tasks

# 检查输入哈希
gradle :module:dependencies --configuration runtimeClasspath --info
```

### 配置缓存问题

```bash
# 显示配置缓存问题
org.gradle.configuration-cache.problems=verbose

# 查看具体错误
gradle build --stacktrace

# 忽略配置缓存问题
gradle build --no-configuration-cache
```

## 总结

Gradle 守护进程和缓存是构建性能优化的关键：

1. **Daemon**：复用 JVM，避免重复启动，节省 2-3 秒
2. **配置缓存**：缓存配置阶段，节省 1-2 秒
3. **构建缓存**：复用 Task 输出，节省 50% 以上时间
4. **增量构建**：跳过 UP-TO-DATE 的 Task
5. **并行构建**：多任务同时执行

合理的配置可以将构建时间从分钟级降到秒级，大幅提升开发效率。
