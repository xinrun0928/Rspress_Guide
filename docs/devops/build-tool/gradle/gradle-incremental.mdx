# Gradle 增量构建与缓存

你有没有遇到过这样的情况——只改了一个 Java 文件，但 Gradle 却重新编译了整个项目？明明代码没变，每次构建却都要花同样的时间？

这正是 Gradle 增量构建要解决的问题。增量构建通过追踪文件变化，只重新执行「真正需要重新执行」的任务，大幅缩短构建时间。

## 增量构建原理

### 增量检查的核心：Task Inputs 和 Outputs

Gradle 通过比较 Task 的输入和输出来判断是否需要重新执行：

```
Task 执行判断逻辑：

  1. 读取所有输入文件的内容（计算哈希）
  2. 与上次构建的输入哈希对比
  3. 读取所有输出文件（计算哈希）
  4. 与预期输出对比

  全部匹配？ → 跳过 Task（UP-TO-DATE）
  不匹配？   → 执行 Task
```

### 三种 Task 状态

| 状态 | 说明 | 构建行为 |
|---|---|---|
| **EXECUTED** | Task 必须执行 | 正常执行 |
| **UP-TO-DATE** | Task 跳过 | 输入输出无变化，跳过 |
| **FROM-CACHE** | 从缓存恢复 | 本地/远程缓存命中 |

```
$ gradle build
> Task :compileJava
  初次编译，EXECUTED (3.5s)
> Task :jar
  EXECUTED (0.5s)

$ gradle build
> Task :compileJava
  UP-TO-DATE (跳过，源码未变)
> Task :jar
  UP-TO-DATE (跳过，编译输出未变)
```

## 配置增量构建

### 定义 Task 的 Inputs 和 Outputs

```groovy
// Groovy DSL
task processData {
    inputs.file "src/data/input.json"
    inputs.dir "src/data/raw"
    inputs.property "version", project.version

    outputs.file "build/processed/data.json"
    outputs.dir "build/processed"

    doLast {
        // 业务逻辑
    }
}
```

```kotlin
// Kotlin DSL
tasks.register("processData") {
    inputs.file("src/data/input.json")
    inputs.dir("src/data/raw")
    inputs.property("version", project.version)

    outputs.file("build/processed/data.json")
    outputs.dir("build/processed")

    doLast {
        // 业务逻辑
    }
}
```

### 使用注解（Java/Kotlin）

```groovy
// Groovy DSL
class BuildInfoTask extends DefaultTask {
    @InputFile
    File inputFile

    @InputDirectory
    File sourceDir

    @OutputFile
    File outputFile

    @Optional
    @Input
    String buildType = "release"

    @TaskAction
    void build() {
        // 业务逻辑
    }
}
```

```kotlin
// Kotlin DSL
abstract class BuildInfoTask : DefaultTask() {
    @get:InputFile
    abstract var inputFile: File

    @get:InputDirectory
    abstract var sourceDir: File

    @get:OutputFile
    abstract var outputFile: File

    @get:Optional
    @get:Input
    abstract var buildType: String = "release"

    @TaskAction
    fun build() {
        // 业务逻辑
    }
}
```

### 常用注解

| 注解 | 类型 | 说明 |
|---|---|---|
| `@Input` | 任意类型 | 简单值作为输入 |
| `@InputFile` | File | 单个文件作为输入 |
| `@InputDirectory` | File | 目录作为输入 |
| `@OutputFile` | File | 单个文件作为输出 |
| `@OutputDirectory` | File | 目录作为输出 |
| `@Incremental` | FileCollection | 增量文件集合 |
| `@Nested` | Object | 嵌套对象（递归检查） |
| `@Optional` | - | 标记为可选 |
| `@SkipWhenEmpty` | - | 空目录跳过 Task |

## 构建缓存（Build Cache）

### 本地构建缓存

Gradle 默认使用本地缓存，`~/.gradle/caches/build-cache-1/` 存储历史 Task 输出。

```bash
# 启用构建缓存
# gradle.properties
org.gradle.caching=true

# 首次构建
$ gradle build
> Task :compileJava
  EXECUTED (3.5s)

# 第二次构建（从缓存恢复）
$ gradle build
> Task :compileJava
  FROM-CACHE (0.1s)
```

### 远程构建缓存

Gradle 支持将构建缓存存储到远程服务器，团队成员可以共享缓存。

```bash
# 配置远程构建缓存
# gradle.properties
org.gradle.caching=true
org.gradle.cache.backend=http
org.gradle.cache.url=https://build-cache.example.com/cache/
org.gradle.cache.username=buildcache
org.gradle.cache.password=password
```

```kotlin
// Kotlin DSL
buildCache {
    remote(HttpBuildCache::class) {
        url = uri("https://build-cache.example.com/cache/")
        credentials {
            username = "buildcache"
            password = System.getenv("BUILDCACHE_PASSWORD")
        }
        push = true  // 允许推送缓存到远程
        isEnabled = true
    }
}
```

### 缓存键（Cache Key）

Gradle 根据以下信息计算缓存键：

1. **Task 类名**
2. **Task 输入的哈希**（文件内容、属性值）
3. **Gradle 版本**
4. **构建环境标识**（操作系统、JVM 版本）

## 配置缓存（Configuration Cache）

配置缓存将 Task 图的配置阶段结果缓存起来，避免每次构建都重新配置。

```properties
# gradle.properties
org.gradle.configuration-cache=true
org.gradle.configuration-cache.problems=warn
```

### 配置缓存的效果

```
Without Configuration Cache:
  配置阶段：2.5s（每次构建都要执行）
  执行阶段：5.0s
  总计：7.5s

With Configuration Cache:
  配置阶段：0.1s（从缓存恢复）
  执行阶段：5.0s
  总计：5.1s
```

### 限制与注意事项

```kotlin
// 使用配置缓存时，不允许的写法
val value = project.provider { ... }  // 允许
// val value = System.getenv("FOO")  // 不允许：非确定性访问
// val value = File(".").absolutePath  // 不允许：非确定性访问
```

## 增量注解进阶

### @Nested：嵌套对象

```groovy
// Groovy DSL
class BuildConfig {
    @Input
    String version

    @Input
    String environment
}

class DeployTask extends DefaultTask {
    @Input
    String appName

    @Nested
    BuildConfig buildConfig

    @OutputFile
    File outputFile
}
```

```kotlin
// Kotlin DSL
open class BuildConfig {
    @get:Input
    var version: String = ""

    @get:Input
    var environment: String = ""
}

abstract class DeployTask : DefaultTask() {
    @get:Input
    var appName: String = ""

    @get:Nested
    var buildConfig: BuildConfig = BuildConfig()

    @get:OutputFile
    abstract var outputFile: File
}
```

### @SkipWhenEmpty：空目录跳过

```groovy
// Groovy DSL
task processAssets {
    // 当 src/assets 为空时，跳过此 Task
    @SkipWhenEmpty
    @InputDirectory
    File sourceDir = file("src/assets")

    @OutputDirectory
    File destDir = file("build/assets")
}
```

### 增量文件变换

```groovy
// Groovy DSL
task transformFiles {
    inputs.dir "src/inputs"
    outputs.dir "build/outputs"

    doLast {
        // 增量处理
    }
}

// 使用 @Incremental 注解获取变化的文件
task incrementalTransform {
    @Incremental
    @InputDirectory
    File sourceDir = file("src")

    @OutputDirectory
    File destDir = file("build")

    @TaskAction
    void transform(InputChanges changes) {
        changes.getFileChanges().each { change ->
            println "${change.changeType}: ${change.file}"
            if (change.changeType != ChangeType.REMOVED) {
                // 处理变化的文件
            }
        }
    }
}
```

## 性能优化建议

### 避免不必要的输入

```groovy
// 错误：使用 fileTree() 会导致任何文件变化都触发 Task
inputs.fileTree("src") {
    include "**/*.java"
}

// 正确：使用 sourceSets 精确指定
inputs.source(sourceSets.main.java)
```

### 使用懒配置

```groovy
// 错误：立即求值
task myTask {
    doLast {
        println "Path: ${project.buildDir}/output"  // project.buildDir 在配置阶段求值
    }
}

// 正确：懒配置
task myTask {
    doLast {
        // 在执行阶段才求值
        println "Path: ${layout.buildDirectory.get()}/output"
    }
}
```

### 分离配置和执行

```groovy
// Groovy DSL
// 配置阶段：设置 Task 的输入输出
class MyTask extends DefaultTask {
    @InputFile
    abstract RegularFileProperty getInputFile()

    @OutputFile
    abstract RegularFileProperty getOutputFile()
}

// 执行阶段：在 doLast/doFirst 中执行逻辑
```

## 常见问题排查

### Task 始终被执行（不 UP-TO-DATE）

```bash
# 分析 Task 的输入输出
$ gradle build --info | grep "Skipping task"

# 常见原因：
# 1. 输入包含时间戳（如 buildTimestamp）
# 2. 输入包含随机值
# 3. 输入文件被其他 Task 修改
# 4. 未正确配置 @Input/@Output
```

### FROM-CACHE 不生效

```bash
# 检查构建缓存配置
$ gradle build --info | grep "Build cache"

# 检查缓存键
$ gradle build --rerun-tasks  # 强制重新执行
$ gradle build --build-cache   # 显式启用缓存
```

### 远程缓存不推送

```bash
# 检查权限
$ gradle build --info | grep "Push to remote"

# 确认 push 设置
# gradle.properties
org.gradle.caching.remote.push=true
```

## 总结

增量构建和缓存是 Gradle 性能优化的核心：

1. **增量检查**：通过对比 Task 的输入输出哈希，判断是否需要重新执行
2. **构建缓存**：将 Task 输出存储到本地或远程，团队共享缓存
3. **配置缓存**：缓存 Task 图的配置阶段，避免重复配置
4. **正确配置 Inputs/Outputs**：善用注解精确描述 Task 的依赖关系

配合 Gradle 的并行构建和 Daemon，可以将构建时间从分钟级降到秒级。
