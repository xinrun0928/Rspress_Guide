# Gradle 增量构建

增量构建是 Gradle 相比 Maven 的最大优势之一。在大型项目中，增量构建可以将构建时间从数十分钟缩短到数分钟。

理解增量构建的原理，是优化 Gradle 构建性能的关键。

## 增量构建原理

### Task 的输入和输出

每个 Gradle Task 都有**输入（Input）**和**输出（Output）**：

```groovy title="build.gradle"
tasks.register('processFiles') {
    inputs.dir 'src'
    outputs.dir 'build/processed'

    doLast {
        // 读取 src 中的文件
        // 处理后输出到 build/processed
    }
}
```

Gradle 会跟踪输入和输出的状态：

- **输入变化** → 重新执行 Task
- **输入未变化** → 跳过 Task，复用上次输出

### 状态计算过程

```
1. 检查 Task 是否曾经执行过
         ↓
2. 计算所有输入文件的哈希值
         ↓
3. 与上次构建的哈希值比较
         ↓ (相同)
4. Task 跳过，使用缓存输出
         ↓ (不同)
5. 执行 Task，更新输出
```

### 示例：Java 编译

```groovy title="build.gradle"
tasks.register<JavaCompile>("compileJava") {
    source = fileTree("src/main/java")
    destinationDirectory.set(file("$buildDir/classes"))

    // 这些是输入
    classpath = sourceSets["main"].compileClasspath
    options.isIncremental = true
}
```

只修改一个 `.java` 文件时，Gradle 只会重新编译这个文件，而不是整个项目。

## 构建缓存

### 本地构建缓存

```bash
# 启用构建缓存
./gradlew build --build-cache

# 或在 gradle.properties 中启用
org.gradle.caching=true
```

构建缓存会存储 Task 的输入和输出，即使项目重新打开，也能复用之前的构建结果。

### 远程构建缓存

配置远程构建缓存服务器：

```groovy title="gradle.properties"
org.gradle.caching=true
org.gradle.cache.remote.url=https://cache.example.com
org.gradle.cache.remote.authentication.password=xxx
```

```groovy title="build.gradle"
buildCache {
    remote(HttpBuildCache) {
        url = uri("https://cache.example.com/")
        credentials {
            username = "cache-user"
            password = System.getenv("CACHE_PASSWORD")
        }
    }
}
```

### 构建缓存 vs 增量构建

| 特性 | 增量构建 | 构建缓存 |
|---|---|---|
| 作用范围 | 单次构建 | 跨构建 |
| 存储位置 | 本地 `.gradle` | 本地或远程 |
| 清除方式 | `clean` | `cleanBuildCache` |
| 并行构建 | 影响执行顺序 | 独立工作 |

## 增量编译器

### Java 增量编译

Gradle 使用增量编译器，只编译修改过的源文件和依赖它们的文件：

```groovy title="build.gradle"
tasks.withType<JavaCompile> {
    options.isIncremental = true
    options.isFork = true
}
```

### 增量注解处理器

```groovy title="build.gradle"
tasks.withType<JavaCompile> {
    options.compilerArgs.addAll(listOf(
        "-proc:none"  // 禁用注解处理
    ))
}
```

## 避免不必要的重新构建

### 正确声明输入输出

```groovy title="build.gradle"
// ❌ 错误：没有声明输入输出
tasks.register('process') {
    doLast {
        file('input.txt').readText()
            .split('\n')
            .forEach { processLine(it) }
    }
}

// ✅ 正确：声明输入输出
tasks.register('process') {
    inputs.file 'input.txt'
    outputs.file 'output.txt'

    doLast {
        file('input.txt').readText()
            .split('\n')
            .forEach { processLine(it) }
        file('output.txt').write('Done')
    }
}
```

### 避免隐式输入

```groovy title="build.gradle"
// ❌ 问题：Task 依赖当前系统时间
tasks.register('badTask') {
    outputs.file 'result.txt'
    doLast {
        file('result.txt').text = "Build at ${new Date()}"
    }
}

// ✅ 正确：只依赖显式声明的输入
tasks.register('goodTask') {
    inputs.file 'config.json'
    outputs.file 'result.txt'
    doLast {
        // 只在 config.json 变化时重新执行
    }
}
```

### 使用文件集合

```groovy title="build.gradle"
tasks.register('processAll') {
    inputs.files(fileTree('src') {
        include '**/*.txt'
    })
    outputs.dir 'build/output'

    doLast {
        // 处理所有 .txt 文件
    }
}
```

## 配置缓存

### 配置缓存原理

配置缓存跳过配置阶段，直接复用之前配置的 Task 图：

```
无配置缓存：
初始化 → 配置(build.gradle) → 执行 → 完成

有配置缓存：
初始化 → 加载缓存的配置 → 执行 → 完成
                    ↑
              上次构建保存
```

### 启用配置缓存

```groovy title="gradle.properties"
org.gradle.configuration-cache=true
```

```bash
./gradlew build --configuration-cache
```

### 配置缓存限制

配置缓存目前有以下限制：

| 限制 | 说明 |
|---|---|
| 不支持 `buildSrc` | 仍在开发中 |
| 不支持某些插件 | 需要插件支持 |
| 不支持任务替换 | 动态 Task |

:::warning 兼容性
配置缓存是实验性功能，使用前请确认你的插件和 Gradle 版本支持。详细兼容性列表见 Gradle 官方文档。
:::

## 性能优化技巧

### 1. 使用 `--build-cache`

```bash
./gradlew build --build-cache
```

### 2. 增量测试

```groovy title="build.gradle"
tasks.withType<Test> {
    maxParallelForks = 4
    forkEvery = 100  // 每 100 个测试类后重启 JVM
}
```

### 3. 并行执行

```groovy title="gradle.properties"
org.gradle.parallel=true
org.gradle.workers.max=4
```

### 4. 按需配置

```groovy title="gradle.properties"
org.gradle.configureondemand=true
```

## 面试追问

**面试官可能问**：Gradle 的增量构建是如何判断 Task 是否需要重新执行的？

这是理解 Gradle 增量构建的核心问题。Gradle 为每个 Task 跟踪三组信息：输入属性、输入文件、输出文件。构建时，Gradle 计算当前输入的指纹（哈希值），与上次构建保存的指纹比较。如果指纹相同，跳过 Task；如果不同，执行 Task。指纹比较包括文件内容哈希、文件路径、文件修改时间等。这就是为什么正确声明 Task 的输入输出对增量构建至关重要。

**面试官可能问**：如何优化 Gradle 的构建速度？

这是一个综合性问题。核心优化方向包括：启用增量构建和构建缓存（`--build-cache`）、并行构建（`--parallel`）、使用配置缓存（实验性）、合理声明 Task 输入输出、避免配置阶段耗时操作、使用 `implementation` 替代 `api` 减少传递依赖、更新到最新 Gradle 版本（每次更新都有性能改进）。对于超大型项目，还可以考虑 Gradle Enterprise 提供的高级缓存功能。
