# Gradle 依赖管理

Gradle 的依赖管理比 Maven 更细粒度、更灵活。理解 `implementation`、`api`、`compileOnly` 等配置，是掌握 Gradle 依赖管理的关键。

## Configuration 概念

Configuration 是 Gradle 组织和分类依赖的方式。每个 Configuration 代表一组依赖及其在类路径中的作用。

### Java 插件的 Configuration

| Configuration | 编译类路径 | 运行时类路径 | 打包 |
|---|---|---|---|
| `compileClasspath` | ✅ | - | - |
| `runtimeClasspath` | ✅ | ✅ | ✅ |
| `annotationProcessor` | ✅ | - | - |

### 依赖配置

| 配置 | 编译时可见 | 运行时可见 | 传递 | 推荐场景 |
|---|---|---|---|---|
| `api` | ✅ | ✅ | ✅ | 公共 API 依赖 |
| `implementation` | ❌ | ✅ | ❌ | 内部实现依赖 |
| `compileOnly` | ✅ | ❌ | ❌ | 编译期依赖 |
| `runtimeOnly` | ❌ | ✅ | ❌ | 运行时依赖 |

## implementation vs api

这是 Gradle 3.4 引入的重要特性。

### implementation（推荐）

```groovy title="build.gradle"
plugins {
    id 'java-library'
}

dependencies {
    // 这个依赖只对当前模块可见
    implementation 'com.google.guava:guava:31.1-jre'
}
```

特点：

- 依赖不会传递给消费者
- 依赖变更不会导致消费者重新编译
- 编译速度快

### api

```groovy title="build.gradle"
plugins {
    id 'java-library'
}

dependencies {
    // 这个依赖会传递给消费者
    api 'org.apache.commons:commons-lang3:3.12.0'
}
```

特点：

- 依赖会传递给消费者（类似 Maven 的 `compile`）
- 消费者可以访问依赖的类和接口
- 用于库模块的公共 API

### 对比示例

```
模块 A（java-library 插件）
 ├── api 'commons-lang3'     ← 公共依赖
 └── implementation 'guava'   ← 内部依赖

消费者 B 依赖模块 A
 ├── 能看到 commons-lang3    ← 传递自 api
 └── 看不到 guava            ← 不传递自 implementation
```

:::tip 选择原则
大多数情况下使用 `implementation`。只有当模块是库，且依赖是其公共 API 的一部分时，才使用 `api`。
:::

## 依赖约束

依赖约束（Dependency Constraints）允许你指定依赖的版本范围：

```groovy title="build.gradle"
dependencies {
    // 声明依赖
    implementation 'com.example:lib:1.0'

    // 添加约束
    constraints {
        implementation('com.example:lib:1.0') {
            because '1.0 是已知稳定版本'
            // 排除特定版本
            reject '1.1.0', '1.1.1'
        }
    }
}
```

### 版本范围

```groovy title="build.gradle"
dependencies {
    constraints {
        implementation('org.slf4j:slf4j-api') {
            version {
                require '[1.7.0, 2.0.0)'
                prefer '1.7.36'
            }
        }
    }
}
```

| 关键词 | 说明 |
|---|---|
| `require` | 强制要求满足的版本范围 |
| `prefer` | 优先选择的版本（可被 override）|
| `strictly` | 严格版本（不可被 override）|
| `reject` | 拒绝的版本 |

## 排除传递依赖

### exclude

```groovy title="build.gradle"
dependencies {
    implementation('com.example:bad-lib:1.0') {
        // 排除特定模块
        exclude group: 'org.apache.commons'
        exclude module: 'commons-collections'
    }
}
```

### 强制版本

```groovy title="build.gradle"
configurations.all {
    resolutionStrategy {
        // 强制使用特定版本
        force 'org.slf4j:slf4j-api:1.7.36'

        // 强制升级传递依赖
        force 'com.google.guava:guava:31.1-jre'
    }
}
```

### 禁用传递依赖

```groovy title="build.gradle"
configurations.all {
    // 完全禁用传递依赖
    transitive = false
}
```

## 解析策略

### Resolution Strategy

```groovy title="build.gradle"
configurations.all {
    resolutionStrategy {
        // 强制特定模块的版本
        force 'com.example:module:2.0.0'

        // 缓存动态版本（SNAPSHOT）
        cacheDynamicVersionsFor 10, 'minutes'

        // 缓存变更模块
        cacheChangingModulesFor 10, 'minutes'

        // 失败时降级（不使用无法解析的模块）
        failOnVersionConflict()
    }
}
```

### 替换依赖

```groovy title="build.gradle"
configurations.all {
    resolutionStrategy {
        // 将 spring-core 替换为 spring-core-jdk
        eachDependency {
            if (requested.group == 'org.springframework' && requested.name == 'spring-core') {
                useVersion '5.3.20'
            }
        }
    }
}
```

## 依赖报告

### 查看依赖

```bash
# 查看完整依赖树
./gradlew dependencies

# 只看特定配置
./gradlew dependencies --configuration runtimeClasspath

# 过滤输出
./gradlew dependencies | grep mydep
```

### dependencyInsight

```bash
# 查看特定依赖的解析结果
./gradlew dependencyInsight --dependency guava

# 输出包括版本选择原因
./gradlew dependencyInsight --dependency log4j
```

## BOM 支持

Gradle 原生支持 BOM：

```groovy title="build.gradle"
dependencies {
    // 导入 Spring Boot BOM
    implementation(platform('org.springframework.boot:spring-boot-dependencies:2.7.0'))

    // 依赖声明不需要指定版本
    implementation 'org.springframework.boot:spring-boot-starter-web'
    implementation 'com.fasterxml.jackson.core:jackson-databind'
}
```

## 面试追问

**面试官可能问**：`implementation` 和 `api` 的区别是什么？各自适用场景？

这是 Gradle 相比 Maven 的核心优势之一。`implementation` 的依赖不会传递给你的消费者，`api` 的依赖会传递。大多数情况下应该使用 `implementation`，因为它减少了重新编译的范围。只有当模块是公共库，且依赖是库 API 的一部分时，才使用 `api`。例如，如果你开发的 SDK 暴露了 `commons-lang3` 的类作为 SDK 的公共接口，就必须用 `api`。

**面试官可能问**：如何排查 Gradle 依赖冲突？

Gradle 提供了两个强大的工具：`./gradlew dependencies` 查看完整依赖树，`./gradlew dependencyInsight --dependency xxx` 查看特定依赖的解析过程和选择原因。通过这两个命令，你可以快速定位冲突来源和版本选择原因。
