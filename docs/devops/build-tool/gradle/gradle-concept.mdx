# Gradle 核心概念

2016 年，Google 宣布 Android Studio 2.0 将从 Maven 切换到 Gradle 作为默认构建工具。这是一个划时代的选择。为什么 Google 放弃了成熟的 Maven，选择了当时还不到 10 岁的 Gradle？

答案在于 Gradle 的核心架构：**增量构建**和**任务依赖图**。

## Gradle vs Ant/Maven 的根本区别

### Ant：完全手动

Ant 提供任务（Task），你可以自由组合：

```xml title="build.xml"
<target name="compile">
    <javac srcdir="src" destdir="build"/>
</target>
```

Ant 本身不管理依赖，不定义生命周期。你可以做任何事，但什么都要自己做。

### Maven：声明式 + 固定生命周期

Maven 通过 XML 声明依赖和插件，生命周期是固定的：

```xml title="pom.xml"
<dependencies>
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-web</artifactId>
    </dependency>
</dependencies>
```

Maven 解决了依赖管理问题，但灵活性受限。

### Gradle：任务图 + 增量执行

Gradle 吸收了两者的优点：

```groovy title="build.gradle"
plugins {
    id 'java'
}

dependencies {
    implementation 'org.springframework.boot:spring-boot-starter-web:2.7.0'
}

tasks.register('hello') {
    doLast {
        println 'Hello, Gradle!'
    }
}
```

Gradle 的核心是**任务依赖图**和**增量构建**。你定义 Task，Task 有输入和输出，Gradle 分析依赖关系，只执行需要更新的任务。

## Gradle 构建的三个阶段

Gradle 构建分为三个阶段：

### 1. 初始化（Initialization）

Gradle 确定哪些 Project 参与构建，创建 Project 实例。

```groovy title="settings.gradle"
rootProject.name = 'my-project'
include 'module-a', 'module-b'
```

### 2. 配置（Configuration）

Gradle 执行每个 Project 的构建脚本，配置 Task 图。

```groovy title="build.gradle"
// 这个块中的代码在配置阶段执行
println "Configuring ${project.name}"
```

### 3. 执行（Execution）

Gradle 根据依赖关系执行 Task 图。

```groovy title="build.gradle"
tasks.register('build') {
    dependsOn 'clean', 'compile'
    doLast {
        println "Building..."
    }
}
```

:::tip 关键区别
**配置阶段**执行构建脚本中的所有代码（不只是任务定义）。这意味着配置阶段可能执行耗时操作。优化构建速度时，需要注意不要在配置阶段做太多工作。
:::

## Project 和 Task

### Project

每个 Gradle 构建由一个或多个 Project 组成：

```groovy title="settings.gradle"
rootProject.name = 'parent'
include 'subproject-a'
include 'subproject-b'
```

创建三个 Project：`parent`、`subproject-a`、`subproject-b`。

### Task

Task 是 Gradle 的工作单元：

```groovy title="build.gradle"
// 注册一个 Task
tasks.register('hello') {
    doLast {
        println 'Hello, World!'
    }
}
```

```bash
gradle hello
# 输出：Hello, World!
```

## Gradle Wrapper

Gradle Wrapper 确保项目使用一致的 Gradle 版本：

```bash
# 在项目目录生成 Wrapper
gradle wrapper

# 或指定版本
gradle wrapper --gradle-version 7.6
```

生成的文件：

```
project/
├── gradlew           # Unix 脚本
├── gradlew.bat       # Windows 脚本
├── gradle/wrapper/
│   ├── gradle-wrapper.jar
│   └── gradle-wrapper.properties
```

使用 Wrapper：

```bash
# 使用 Wrapper 执行构建
./gradlew build

# Windows
gradlew.bat build
```

:::tip 最佳实践
始终使用 Wrapper 而非系统安装的 Gradle，确保团队成员和 CI 服务器使用相同版本，避免「在我机器上能跑」的问题。
:::

## 构建缓存

Gradle 支持构建缓存，跨构建复用输出：

```bash
# 启用构建缓存
./gradlew build --build-cache

# 或配置在 gradle.properties
org.gradle.caching=true
```

构建缓存与增量构建的区别：

| 特性 | 增量构建 | 构建缓存 |
|---|---|---|
| 范围 | 单次构建 | 跨构建 |
| 存储 | 本地 | 本地或远程 |
| 效果 | 跳过未变化的 Task | 复用 Task 输出 |

---

## Gradle 的优势总结

| 特性 | Maven | Gradle |
|---|---|---|
| 构建速度 | 慢（全量构建）| 快（增量 + 缓存）|
| 依赖控制 | `compile`/`runtime` | `implementation`/`api` |
| 灵活性 | 一般 | 高 |
| DSL | XML | Groovy/Kotlin |
| 多模块 | 一般 | 优秀 |
| IDE 支持 | 好 | 越来越好 |

## 面试追问

**面试官可能问**：Gradle 的增量构建是如何实现的？

这是理解 Gradle 架构的核心问题。Gradle 为每个 Task 定义输入（Input）和输出（Output）。构建时，Gradle 计算输入的哈希值，与上次构建的哈希值比较。如果相同，说明 Task 不需要重新执行，直接跳过。这就是增量构建的原理。你可以通过 `inputs` 和 `outputs` API 显式声明 Task 的输入输出，让 Gradle 正确判断任务是否需要重新执行。

**面试官可能问**：Gradle 的构建过程分为哪几个阶段？

Gradle 构建分为三个阶段：初始化阶段（创建 Project 实例）、配置阶段（执行 build.gradle 脚本，构建 Task 图）、执行阶段（按依赖顺序执行 Task 图）。理解这三个阶段很重要，因为配置阶段的代码在每次构建时都会执行，不当的配置代码可能拖慢构建速度。
