# Gradle 多项目构建

想象一个场景：你接手了一个微服务项目，里面有 20 个独立的 Module，每个 Module 都有自己的构建逻辑。突然，产品经理说：「我们要统一升级 Spring Boot 到 3.0，所有服务都要升级。」

如果你是 Maven，你可能要在每个 `pom.xml` 里改版本，改完还要验证是否有冲突。如果你是 Gradle，你只需要改一个地方——这就是多项目构建的魅力。

## 为什么需要多项目构建

在真实项目中，单一项目结构很快就会遇到瓶颈：

**单体应用的困境**：所有代码堆在一个项目里，构建时间长，团队协作冲突频繁，新人上手困难。

**过早微服务的代价**：每个服务单独仓库，版本同步困难，公共代码复制粘贴，CI/CD 配置重复。

**多项目构建的平衡**：在一个代码仓库中管理多个相关项目，既能共享配置，又能独立构建。

## 项目结构设计

### 根项目与子项目

Gradle 的多项目结构通过 `settings.gradle` 定义：

```groovy title="settings.gradle"
rootProject.name = 'my-platform'

include 'common-utils'
include 'web-backend'
include 'web-frontend'
include 'api-gateway'
```

这个结构会产生以下目录树：

```
my-platform/
├── build.gradle              # 根项目构建脚本
├── settings.gradle           # 定义项目结构
├── common-utils/             # 工具模块，所有服务共享
│   └── build.gradle
├── web-backend/              # 后端服务
│   └── build.gradle
├── web-frontend/              # 前端资源
│   └── build.gradle
└── api-gateway/              # 网关服务
    └── build.gradle
```

### 项目类型划分

| 项目类型 | 用途 | 特点 |
|---|---|---|
| `root` | 聚合项目 | 只管理配置，不产生构件 |
| `library` | 库模块 | 被其他项目依赖，如 `common-utils` |
| `application` | 应用模块 | 可以独立运行，如 `web-backend` |
| `java-library` | Java 库 | 使用 Java 插件，发布到 Maven 仓库 |

## 依赖管理：buildSrc 与 convention plugins

### 使用 buildSrc 共享配置

`buildSrc` 是 Gradle 的一个特殊目录，用于存放构建逻辑。它会在编译任何项目之前执行，非常适合放置公共配置：

```
my-platform/
├── buildSrc/
│   ├── src/main/groovy/
│   │   └── com/platform/
│   │       ├── java-library-conventions.gradle
│   │       └── spring-boot-conventions.gradle
│   └── build.gradle
└── settings.gradle
```

```groovy title="buildSrc/src/main/groovy/com/platform/java-library-conventions.gradle"
plugins {
    id 'java-library'
    id 'io.freefair.lombok' version '8.6'
}

java {
    sourceCompatibility = JavaVersion.VERSION_17
    targetCompatibility = JavaVersion.VERSION_17
}

repositories {
    mavenCentral()
}

dependencies {
    api 'org.apache.commons:commons-lang3:3.14.0'
    implementation 'com.google.guava:guava:32.1.3-jre'
}
```

```groovy title="buildSrc/src/main/groovy/com/platform/spring-boot-conventions.gradle"
plugins {
    id 'org.springframework.boot' version '3.2.0'
    id 'io.spring.dependency-management' version '1.1.4'
}

dependencies {
    implementation platform('org.springframework.boot:spring-boot-dependencies:3.2.0')
}

tasks.withType(JavaCompile).configureEach {
    options.encoding = 'UTF-8'
}
```

### 在子项目中应用约定

```groovy title="web-backend/build.gradle"
plugins {
    id 'com.platform.java-library-conventions'
    id 'org.springframework.boot'
    id 'io.spring.dependency-management'
}

dependencies {
    implementation project(':common-utils')
    implementation 'org.springframework.boot:spring-boot-starter-web'
    testImplementation 'org.springframework.boot:spring-boot-starter-test'
}
```

## 跨项目依赖配置

### 项目引用语法

在 Gradle 中，引用其他项目就像引用一个依赖一样简单：

```groovy title="web-backend/build.gradle"
dependencies {
    // 引用 common-utils 模块
    implementation project(':common-utils')
    
    // 引用 api-gateway 模块
    implementation project(':api-gateway')
}
```

### 依赖传播与控制

Gradle 默认会将 `implementation` 依赖传递到依赖它的项目。如果你想控制这个行为：

```groovy
// 在 consumer 项目中，禁止传递某个依赖
configurations.all {
    resolutionStrategy.dependencySubstitution {
        substitute(module('com.example:legacy-lib')).using(project(':legacy'))
    }
}
```

### 编译时依赖与运行时依赖

```groovy title="common-utils/build.gradle"
plugins {
    id 'java-library'
}

dependencies {
    // API：暴露给消费者
    api 'org.apache.commons:commons-collections4:4.4'
    
    // Implementation：仅内部使用
    implementation 'com.fasterxml.jackson.core:jackson-databind:2.16.0'
    
    // CompileOnly：仅编译时需要
    compileOnly 'org.projectlombok:lombok:1.18.30'
    
    // RuntimeOnly：仅运行时需要
    runtimeOnly 'org.postgresql:postgresql:42.7.1'
}
```

## 统一版本管理

### 使用 `subprojects` 和 `allprojects`

```groovy title="build.gradle"
subprojects {
    repositories {
        mavenCentral()
        maven { url 'https://repo.spring.io/milestone' }
    }
    
    tasks.withType(JavaCompile).configureEach {
        options.encoding = 'UTF-8'
    }
}

allprojects {
    version = '1.0.0-SNAPSHOT'
    group = 'com.platform'
}
```

### 使用 `ext` 管理版本号

```groovy title="build.gradle"
ext {
    springBootVersion = '3.2.0'
    lombokVersion = '1.18.30'
}

subprojects {
    dependencies {
        implementation "org.springframework.boot:spring-boot-starter:${springBootVersion}"
    }
}
```

## 增量构建与并行执行

### 增量构建原理

Gradle 的增量构建通过 Task 的输入输出判断是否需要重新执行：

```groovy
task processResources {
    inputs.dir 'src/main/resources'
    outputs.dir 'build/resources'
    
    doLast {
        // 仅当输入或输出变化时才执行
    }
}
```

### 启用并行构建

```groovy title="gradle.properties"
org.gradle.parallel=true
org.gradle.caching=true
org.gradle.daemon=true
```

在 CI 环境中，80 个独立模块可以同时构建，构建时间从 30 分钟缩短到 5 分钟。

## 常见问题与解决

### 模块间循环依赖

Gradle 会检测循环依赖并报错：

```
FAILURE: Circular dependency between modules:
:web-backend -> :common-utils -> :api-gateway -> :web-backend
```

**解决方案**：重新设计模块边界，将公共部分提取到独立模块。

### 构建顺序控制

默认情况下，Gradle 会根据依赖关系自动决定构建顺序。如果需要强制顺序：

```groovy
web-backend.mustRunAfter api-gateway  // web-backend 必须在 api-gateway 之后构建
```

### 发布配置

```groovy title="common-utils/build.gradle"
plugins {
    id 'maven-publish'
    id 'signing'
}

publishing {
    publications {
        mavenJava(MavenPublication) {
            from components.java
            
            pom {
                name = 'Common Utilities'
                description = 'Shared utilities for the platform'
            }
        }
    }
    
    repositories {
        maven {
            url = version.endsWith('SNAPSHOT') 
                ? 'https://repo.example.com/snapshots' 
                : 'https://repo.example.com/releases'
        }
    }
}
```

## 最佳实践

### 1. 清晰的模块命名

```
com.platform.common          # 公共组件
com.platform.backend.*        # 后端服务
com.platform.frontend.*      # 前端服务
com.platform.gateway          # 网关服务
```

### 2. 最小化依赖层级

避免过深的依赖链：`A -> B -> C -> D` 应该尽量扁平化。

### 3. 使用 BOM 统一管理

```groovy title="build.gradle"
allprojects {
    apply plugin: 'io.spring.dependency-management'
    
    dependencyManagement {
        imports {
            mavenBom "org.springframework.boot:spring-boot-dependencies:3.2.0"
        }
    }
}
```

### 4. CI/CD 中的多项目构建

```groovy title=".gitlab-ci.yml"
build:
  stage: build
  script:
    - ./gradlew assemble --parallel -x test
  artifacts:
    paths:
      - "*/build/libs/"
```

## 总结

多项目构建是管理复杂项目的必备技能。掌握 `buildSrc`、约定插件、统一版本管理，你就拥有了管理上百个模块的能力。

下次产品经理说「所有服务升级 Spring Boot」的时候，你只需要改三个地方：根项目的 BOM 版本号、约定插件版本、以及 `buildSrc` 中的版本管理。

这就是 Gradle 多项目构建的魅力——用配置驱动变化，而不是用重复劳动堆砌。
