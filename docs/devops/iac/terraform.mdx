# Terraform 基础设施即代码实战

一行命令创建整个云基础设施——Terraform 到底是怎么做到的？

想象这样一个场景：你的团队需要在 AWS 上搭建一套完整的环境，包括 VPC、子网、负载均衡器、EC2 实例、RDS 数据库、S3 存储桶，还要配置安全组、路由表、IAM 角色。传统做法是什么？手动在 AWS 控制台上一顿点选，重复操作几十次，然后祈祷没有遗漏。

但有了 Terraform，你只需要写一份声明式的配置文件，执行 `terraform apply`，几分钟后整个基础设施就会按照你的意图搭建完成。这就是基础设施即代码（Infrastructure as Code，IaC）的魅力。而 Terraform，正是这个领域最流行的工具之一。

## 一、什么是基础设施即代码？

### 1.1 IaC 的核心理念

基础设施即代码不是简单地"用代码创建服务器"，而是一种**用软件开发的方式来管理基础设施的哲学**。你的基础设施描述被当作代码一样处理：版本控制、代码审查、自动化测试、持续集成——这些软件开发中的最佳实践，同样适用于基础设施管理。

传统运维和 IaC 的本质区别在于**声明式**与**命令式**的不同：

**命令式**告诉你"怎么做"：先执行 A，再执行 B，最后执行 C。你需要知道每一个步骤的细节。

**声明式**告诉你"要什么"：我需要这样一个环境，包含了这些资源，具备这样的配置。你只需要描述最终状态，工具会帮你决定如何达到这个状态。

Terraform 就是声明式的典型代表。你描述你想要的云资源及其配置，Terraform 会计算出需要执行哪些操作来达到这个状态，然后自动完成。

### 1.2 IaC 的核心价值

为什么 IaC 如此重要？让我从几个维度来分析：

**可重复性**：同样的配置可以在任何环境中重复执行。如果你在开发环境、测试环境、生产环境需要完全相同的基础设施，用 IaC 就是复制配置文件这么简单。手动操作？光是保持一致性就够你喝一壶的。

**可审计性**：所有的基础设施变更都记录在版本历史中。谁在什么时间改了什么，为什么改，都可以追溯。这对于合规要求严格的企业来说尤其重要。

**快速部署**：手动创建一个包含 20 个资源的环境可能需要几个小时，而 Terraform 只需要几分钟。你可以把更多时间花在真正有价值的工作上。

**降低错误率**：人为操作难免出错，尤其是在执行重复性任务时。Terraform 的自动化消除了这种风险——只要配置文件是对的，出来的结果就是对的。

:::tip 适用场景

Terraform 最擅长的场景包括：云资源创建和管理、多云/混合云架构、一次性环境搭建（开发、测试、演示）、基础设施标准化和环境复制。

:::

### 1.3 Terraform 与其他工具的对比

IaC 领域还有其他一些流行的工具，比如 AWS CloudFormation、Pulumi、Ansible SaltStack 等。它们各有特色：

| 工具 | 类型 | 云厂商 | 学习曲线 | 适用场景 |
|---|---|---|---|---|
| Terraform | 声明式 | 多云支持 | 中等 | 多云架构、复杂基础设施 |
| CloudFormation | 声明式 | AWS 专用 | 中等 | 纯 AWS 环境 |
| Pulumi | 命令式 | 多云支持 | 较高 | 需要编程逻辑的场景 |
| Ansible | 命令式 | 通用 | 较低 | 配置管理、服务器初始化 |

Terraform 的优势在于它的**跨平台能力**和**庞大的社区生态**。HashiCorp 定义的_provider_机制，让 Terraform 能够支持几乎所有主流的云服务商，而且社区贡献的 provider 已经覆盖了数百种服务和工具。

## 二、HCL 语法入门

### 2.1 HCL 是什么？

Terraform 使用 HCL（HashiCorp Configuration Language）作为配置文件语言。HCL 的设计目标是既对人类友好（易读易写），又对机器友好（易于解析）。它的语法介于 JSON 的结构化和 YAML 的简洁之间，既有层次感又不会太繁琐。

一个简单的 Terraform 配置文件大概长这样：

```hcl title="main.tf"
# 定义 provider
provider "aws" {
  region = "ap-northeast-1"
}

# 创建资源
resource "aws_vpc" "main" {
  cidr_block = "10.0.0.0/16"
  
  tags = {
    Name        = "my-vpc"
    Environment = "production"
  }
}

# 创建子网
resource "aws_subnet" "public" {
  vpc_id                  = aws_vpc.main.id
  cidr_block              = "10.0.1.0/24"
  availability_zone       = "ap-northeast-1a"
  map_public_ip_on_launch = true
  
  tags = {
    Name = "public-subnet"
  }
}
```

### 2.2 基本语法规则

HCL 有几个核心概念需要理解：

**块（Block）**：HCL 的基本结构单位，用大括号 `{}` 包裹。每个块都有一个类型（如 `provider`、`resource`、`variable`），后面跟着一个标签或名称。

```hcl
# provider 块
provider "aws" {
  # 配置内容
}

# resource 块
resource "aws_instance" "web" {
  # 配置内容
}
```

**属性（Attribute）**：块内的键值对称为属性，格式是 `键 = 值`。值可以是字符串、数字、布尔值、列表、映射等。

```hcl
# 字符串（引号可省略但推荐保留）
name = "example"

# 数字
count = 3

# 布尔值
enable = true

# 列表
availability_zones = ["ap-northeast-1a", "ap-northeast-1c"]

# 映射（对象）
tags = {
  Name        = "my-instance"
  Environment = "production"
}
```

**字符串插值**：在字符串中嵌入变量或表达式，使用 `${}` 语法。

```hcl
# 引用资源属性
name = "${aws_vpc.main.id}"

# 字符串模板
greeting = "Hello, ${var.name}!"
```

**注释**：HCL 支持三种注释风格，与大多数编程语言一致。

```hcl
# 单行注释（Shell 风格）

/* 多行注释
   这种注释可以跨越多行 */

# // 风格也被支持（但不推荐）
```

### 2.3 常用块类型

Terraform 配置文件中会用到多种块类型，每种都有特定的用途：

**provider 块**：声明你要使用的云平台或服务提供商。

```hcl
provider "aws" {
  region = "ap-northeast-1"
  
  # 可选的访问凭证（建议使用环境变量或 IAM 角色）
  access_key = var.aws_access_key
  secret_key = var.aws_secret_key
  
  # 默认标签，会自动添加到所有资源
  default_tags {
    tags = {
      ManagedBy = "Terraform"
      Project    = var.project_name
    }
  }
}
```

**resource 块**：定义你要创建的基础设施资源。这是 Terraform 配置的核心部分。

```hcl
resource "aws_security_group" "web" {
  name        = "web-sg"
  description = "Security group for web servers"
  vpc_id      = aws_vpc.main.id
  
  # 入站规则
  ingress {
    description = "HTTP"
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }
  
  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }
  
  tags = {
    Name = "web-security-group"
  }
}
```

**variable 块**：定义输入变量，让配置更灵活和可复用。

```hcl
variable "environment" {
  type        = string
  description = "部署环境名称"
  default     = "production"
}

variable "instance_type" {
  type        = string
  description = "EC2 实例类型"
  default     = "t3.micro"
  
  validation {
    condition     = contains(["t3.micro", "t3.small", "t3.medium"], var.instance_type)
    error_message = "实例类型必须是 t3.micro、t3.small 或 t3.medium。"
  }
}
```

**output 块**：定义输出值，在 `terraform apply` 完成后显示。

```hcl
output "vpc_id" {
  description = "VPC 的 ID"
  value       = aws_vpc.main.id
}

output "web_server_public_ip" {
  description = "Web 服务器的公网 IP"
  value       = aws_instance.web.public_ip
}
```

**data 块**：数据源，用于读取现有的资源信息而不创建新资源。

```hcl
# 获取当前账号信息
data "aws_caller_identity" "current" {}

# 获取可用区列表
data "aws_availability_zones" "available" {
  state = "available"
}

# 读取已有的 AMI
data "aws_ami" "ubuntu" {
  most_recent = true
  owners      = ["099720109477"]  # Canonical
  
  filter {
    name   = "name"
    values = ["ubuntu/images/hvm-ssd/ubuntu-jammy-22.04-amd64-server-*"]
  }
}
```

**module 块**：模块调用，用于组织和管理复杂配置。

```hcl
module "network" {
  source = "./modules/network"
  
  vpc_cidr = var.vpc_cidr
  env      = var.environment
}

module "ecs_cluster" {
  source = "./modules/ecs"
  
  vpc_id     = module.network.vpc_id
  subnet_ids = module.network.private_subnet_ids
  
  cluster_name = var.cluster_name
}
```

## 三、Provider 机制

### 3.1 Provider 是什么？

Provider 是 Terraform 的插件，它负责与特定的云平台或服务 API 进行交互。Terraform 本身不直接创建任何资源，它通过 Provider 来完成这个任务。每个 Provider 都知道如何与对应的 API 对话，并能将 Terraform 的配置转换为 API 调用。

官方维护的 Provider 覆盖了主流的云服务商：

| Provider | 云/平台 | 资源数量 |
|---|---|---|
| AWS | Amazon Web Services | 1000+ |
| Azure | Microsoft Azure | 500+ |
| Google Cloud | GCP | 400+ |
| Kubernetes | Kubernetes | 50+ |
| Docker | Docker | 20+ |
| Helm | Helm | 10+ |

除了官方 Provider，还有数千个社区贡献的 Provider，覆盖了从 GitHub、GitLab 到 Datadog、New Relic 的各种服务。

### 3.2 Provider 配置

在使用 Provider 之前，你需要声明它。最简单的方式是在配置文件中直接指定：

```hcl
provider "aws" {
  region = "ap-northeast-1"
}
```

但更推荐的方式是通过 `required_providers` 在 `terraform` 块中声明版本要求，然后在 `provider` 块中引用：

```hcl title="versions.tf"
terraform {
  required_version = ">= 1.0"
  
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}
```

```hcl title="provider.tf"
provider "aws" {
  region = "ap-northeast-1"
  
  # 使用别名支持多区域
  alias = "seoul"
  
  default_tags {
    tags = {
      Environment = "production"
    }
  }
}

# 另一个区域的 provider
provider "aws" {
  region = "ap-southeast-1"
  alias  = "singapore"
}
```

### 3.3 多 Provider 配置

现代应用架构往往涉及多个云平台或服务。你可以在同一份配置中同时使用多个 Provider：

```hcl
# AWS Provider
provider "aws" {
  region = "ap-northeast-1"
}

# Kubernetes Provider（使用 AWS EKS）
provider "kubernetes" {
  host                   = module.eks.cluster_endpoint
  cluster_ca_certificate = base64decode(module.eks.cluster_certificate_authority_data)
  exec {
    api_version = "client.authentication.k8s.io/v1beta1"
    command     = "aws"
    args        = ["eks", "get-token", "--cluster-name", module.eks.cluster_name]
  }
}

# Helm Provider
provider "helm" {
  kubernetes {
    host                   = module.eks.cluster_endpoint
    cluster_ca_certificate = base64decode(module.eks.cluster_certificate_authority_data)
    exec {
      api_version = "client.authentication.k8s.io/v1beta1"
      command     = "aws"
      args        = ["eks", "get-token", "--cluster-name", module.eks.cluster_name]
    }
  }
}
```

## 四、状态管理

### 4.1 什么是 Terraform 状态？

Terraform 状态（State）是 Terraform 用来**追踪实际基础设施的文件**。当你执行 `terraform apply` 时，Terraform 会：

1. 读取当前的配置文件
2. 读取已有的状态文件
3. 对比两者，计算出需要执行的操作
4. 调用 Provider API 创建/更新/删除资源
5. 更新状态文件以反映最新状态

状态文件默认是一个名为 `terraform.tfstate` 的 JSON 文件，它记录了所有资源的属性和依赖关系。

:::warning 重要提示

**永远不要手动编辑状态文件**。Terraform 状态是 Terraform 内部使用的数据结构，手动修改可能导致状态与实际基础设施不一致，产生难以排查的问题。

:::

### 4.2 本地状态 vs 远程状态

**本地状态**：默认情况下，Terraform 将状态存储在本地文件系统中。这种方式简单，适合个人开发或学习场景。但存在几个问题：

- 团队成员无法共享状态，可能导致配置冲突
- 本地文件可能丢失或损坏
- 无法实现多人协作

**远程状态**：对于团队协作和生产环境，推荐使用远程后端存储状态。Terraform 支持多种远程后端：

| 后端 | 存储位置 | 状态锁定 | 适用场景 |
|---|---|---|---|
| S3 | AWS S3 | DynamoDB | AWS 环境 |
| Azure Blob | Azure Storage | Azure Table | Azure 环境 |
| GCS | Google Cloud Storage | 内置 | GCP 环境 |
| Terraform Cloud | HashiCorp 服务 | 内置 | 任何环境 |
| Consul | HashiCorp Consul | 内置 | 自托管 |

**S3 后端配置**是最常见的远程状态方案：

```hcl title="backend.tf"
terraform {
  backend "s3" {
    bucket         = "my-terraform-state"
    key            = "prod/terraform.tfstate"
    region         = "ap-northeast-1"
    encrypt        = true
    dynamodb_table = "terraform-state-lock"
  }
}
```

### 4.3 状态锁定

Terraform 支持状态锁定机制，防止多人同时执行 `terraform apply` 导致状态冲突。当一个 terraform apply 运行时，状态会被锁定，其他尝试执行 terraform apply 的操作会等待或报错。

对于 S3 后端，状态锁定通过 DynamoDB 表实现：

```bash
# 创建 DynamoDB 表用于状态锁定
aws dynamodb create-table \
  --table-name terraform-state-lock \
  --attribute-name LockID \
  --attribute-type S \
  --key-type HASH \
  --billing-mode PAY_PER_REQUEST
```

:::tip 最佳实践

始终为远程状态配置状态锁定。特别是团队协作场景，状态锁定可以防止并发操作导致的状态损坏。

:::

### 4.4 状态操作

Terraform 提供了一些命令用于状态管理：

```bash
# 查看当前状态列表
terraform state list

# 查看特定资源的详细信息
terraform state show aws_instance.web

# 将资源从状态中移除（不再由 Terraform 管理）
terraform state rm aws_instance.web

# 将已有资源导入到 Terraform 状态
terraform import aws_instance.web i-1234567890abcdef0

# 手动移动状态中的资源
terraform state mv aws_instance.web aws_instance.app
```

## 五、工作流程

### 5.1 标准工作流程

Terraform 的标准工作流程包含以下步骤：

**第一步：编写配置**。创建 `.tf` 文件，定义你的基础设施。

**第二步：初始化**。执行 `terraform init`，下载必要的 Provider 插件。

**第三步：验证**。执行 `terraform validate`，检查配置语法和逻辑是否正确。

**第四步：规划**。执行 `terraform plan`，预览将要执行的操作。

**第五步：应用**。执行 `terraform apply`，创建或更新基础设施。

**第六步：销毁**。不再需要时，执行 `terraform destroy`，清理所有资源。

### 5.2 terraform init

`terraform init` 是你开始一个新项目或添加 Provider 后第一个要执行的命令。它的作用是：

- 初始化工作目录
- 下载配置中引用的 Provider 插件
- 配置远程后端（如果指定了）
- 设置模块源（如果使用了模块）

```bash
# 初始化
terraform init

# 初始化并升级 Provider
terraform init -upgrade

# 指定插件缓存目录
terraform init -plugin-dir=/path/to/plugins
```

### 5.3 terraform validate

`terraform validate` 检查配置文件的语法正确性和逻辑有效性，但不与远程 API 交互。这是在提交代码前进行检查的好习惯。

```bash
# 验证配置
terraform validate

# 输出示例
# Success! The configuration is valid.
```

### 5.4 terraform plan

`terraform plan` 是 Terraform 最强大的功能之一。它会分析配置文件、读取当前状态，然后生成一份详细的执行计划，说明将要创建、更新或删除哪些资源。

```bash
# 标准 plan
terraform plan

# 保存 plan 到文件（供 apply 使用）
terraform plan -out=tfplan

# 查看 plan 的详细变更
terraform show tfplan
```

Plan 输出示例：

```text
Terraform will perform the following actions:

  # aws_vpc.main will be created
  + resource "aws_vpc" "main" {
      + id                               = (known after apply)
      + cidr_block                       = "10.0.0.0/16"
      + enable_dns_hostnames             = true
      + enable_dns_support               = true
      + tags                             = {
          + "Name" = "my-vpc"
        }
    }

  # aws_subnet.public will be created
  + resource "aws_subnet" "public" {
      + id                                     = (known after apply)
      + vpc_id                                 = (known after apply)
      + cidr_block                             = "10.0.1.0/24"
      + availability_zone                      = "ap-northeast-1a"
      + map_public_ip_on_launch                = true
      + tags                                   = {
          + "Name" = "public-subnet"
        }
    }

Plan: 2 to add, 0 to change, 0 to destroy.
```

:::tip 建议

在正式执行 `terraform apply` 之前，**务必先看一遍 `terraform plan` 的输出**。这是你最后的机会检查将要发生的变更是否符合预期。特别是生产环境，一个错误的变更可能导致服务中断。

:::

### 5.5 terraform apply

`terraform apply` 执行计划，创建或更新基础设施。

```bash
# 交互式 apply（会提示确认）
terraform apply

# 使用已有的 plan 文件（无需再次确认）
terraform apply tfplan

# 自动 approve（适用于 CI/CD 场景）
terraform apply -auto-approve
```

Apply 完成后，Terraform 会显示创建的资源摘要和输出值：

```text
Apply complete! Resources: 2 added, 0 changed, 0 destroyed.

Outputs:

vpc_id = "vpc-0123456789abcdef0"
```

### 5.6 terraform destroy

`terraform destroy` 清理所有由 Terraform 管理的基础设施。

```bash
# 交互式 destroy
terraform destroy

# 自动 approve
terraform destroy -auto-approve

# 只销毁特定资源
terraform destroy -target=aws_instance.web
```

:::danger 谨慎操作

`terraform destroy` 会删除你的云资源。在执行之前，请确保你了解将要删除的内容。对于生产环境，建议先 `terraform plan -destroy` 预览将要删除的资源。

:::

## 六、变量与输出

### 6.1 输入变量

Terraform 的变量系统让它变得灵活和可复用。变量可以在多个地方定义，Terraform 会按照优先级依次取值：

**变量文件**（按以下顺序加载）：

- `terraform.tfvars` 或 `terraform.tfvars.json`
- `*.auto.tfvars` 或 `*.auto.tfvars.json`
- 命令行 `-var` 或 `-var-file` 参数

**定义变量**：

```hcl title="variables.tf"
variable "project_name" {
  type        = string
  description = "项目名称"
  default     = "my-app"
}

variable "environment" {
  type        = string
  description = "部署环境"
  validation {
    condition     = contains(["dev", "staging", "production"], var.environment)
    error_message = "环境必须是 dev、staging 或 production。"
  }
}

variable "instance_config" {
  type = object({
    instance_type = string
    ami_id         = string
    volume_size    = number
  })
  description = "EC2 实例配置"
  default = {
    instance_type = "t3.micro"
    ami_id        = "ami-0c55b159cbfafe1f0"
    volume_size   = 20
  }
}
```

**变量类型**：Terraform 支持丰富的变量类型：

```hcl
# 基础类型
string   = "hello"
number   = 42
bool     = true

# 集合类型
list     = ["a", "b", "c"]
set      = toset(["a", "b", "c"])  # 去重集合
map      = { key = "value" }

# 对象类型
object = {
  name = "example"
  age  = 30
}

# 元组类型
tuple = ["string", 42, true]

# 复杂类型
variable "security_groups" {
  type = map(list(string))
  default = {
    web    = ["80", "443"]
    ssh    = ["22"]
    rdp    = ["3389"]
  }
}
```

### 6.2 输出值

输出值让你在 apply 完成后能看到重要的信息，比如资源 ID、IP 地址等。

```hcl title="outputs.tf"
output "vpc_id" {
  description = "VPC ID"
  value       = aws_vpc.main.id
}

output "instance_public_ip" {
  description = "EC2 公网 IP"
  value       = aws_instance.web.public_ip
}

# 敏感输出（不会在 plan/apply 输出中显示）
output "db_password" {
  description = "数据库密码"
  value       = aws_db_instance.main.password
  sensitive   = true
}

# 在 apply 完成后显示完整值
output "full_config" {
  description = "完整配置"
  value       = aws_instance.web
}
```

## 八、总结

Terraform 重新定义了基础设施管理的方式。它用声明式的配置语言描述云资源，用自动化的方式执行变更，用状态文件追踪现实世界中的基础设施。这种方式让基础设施管理变得可预测、可重复、可审计。

:::details 最佳实践总结

**配置组织**：

- 使用 `*.tf` 文件组织不同类型的资源配置
- 使用 `variables.tf` 集中管理变量定义
- 使用 `outputs.tf` 集中管理输出
- 使用 `versions.tf` 管理 Terraform 和 Provider 版本

**版本控制**：

- 将所有 `.tf` 文件加入版本控制
- 将 `terraform.tfstate` 加入 `.gitignore`（使用远程后端时）
- 使用工作空间（workspace）区分环境

**团队协作**：

- 使用远程后端存储状态
- 启用状态锁定
- 在 CI/CD 流水线中执行 terraform apply
- 建立代码审查流程

**安全**：

- 永远不要将敏感信息（密码、密钥）硬编码在配置中
- 使用环境变量或 Vault 集成
- 配置 Provider 时使用 IAM 角色而非 Access Key

:::

Terraform 的学习曲线并不陡峭，但要想用好它，需要理解它的核心概念：声明式配置、HCL 语法、Provider 机制、状态管理。当你掌握了这些，再复杂的云架构也能用 Terraform 优雅地管理起来。
