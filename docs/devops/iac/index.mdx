# 基础设施即代码（IaC）

想象一下这个场景：你的团队需要在 AWS 上搭建一套完整的基础设施——VPC、子网、EC2、RDS、S3 桶、负载均衡器……如果手动操作，需要点几百下鼠标，耗时一两天。但如果写一个脚本，5 分钟就能搞定，还能重复执行、版本管理、代码审查……

这就是基础设施即代码（Infrastructure as Code，IaC）的价值。

## 什么是 IaC

### 定义

> 基础设施即代码（IaC）是通过代码而非手动过程来管理和配置基础设施的实践。

### IaC 的价值

| 价值 | 说明 |
|---|---|
| 自动化 | 消除手动操作，一键部署 |
| 可重复 | 同一套代码可以部署多次 |
| 可版本化 | 代码在 Git 中管理，可追溯 |
| 可测试 | 代码可以审查和测试 |
| 一致性 | 每次部署结果一致 |

## IaC 工具分类

### 声明式 vs 命令式

```
┌─────────────────────────────────────────────────────────────┐
│                    IaC 工具分类                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  声明式 (Declarative)                                       │
│  ├── 定义"想要什么"                                         │
│  ├── 工具自动决定"如何实现"                                 │
│  ├── 示例：Terraform、Pulumi、HCL                          │
│  │                                                         │
│  命令式 (Imperative)                                        │
│  ├── 定义"如何做"                                          │
│  ├── 一步步指定操作                                        │
│  └── 示例：Ansible、Shell 脚本                             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 主流 IaC 工具

| 工具 | 类型 | 特点 |
|---|---|---|
| Terraform | 声明式 | 跨云支持，HCL 语言 |
| Pulumi | 声明式 | 真正的编程语言 |
| Ansible | 命令式/声明式 | SSH 批量执行 |
| CloudFormation | 声明式 | AWS 原生 |
| Pulumi | 声明式 | 多语言支持 |

## Terraform 概述

### 什么是 Terraform

Terraform 是 HashiCorp 开发的开源 IaC 工具，使用 HCL（HashiCorp Configuration Language）作为配置语言。

<Badge text="核心特点" type="tip" /> Terraform 的核心理念是"计划"（plan）和"应用"（apply）——先预览变更，再确认执行。

### Terraform 工作流程

```
┌─────────────────────────────────────────────────────────────┐
│                  Terraform 工作流程                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 编写配置 (Write)                                        │
│     main.tf → 定义基础设施                                  │
│                                                             │
│  2. 规划变更 (Plan)                                         │
│     terraform plan → 预览将要执行的变更                      │
│                                                             │
│  3. 应用变更 (Apply)                                        │
│     terraform apply → 执行变更                              │
│                                                             │
│  4. 销毁资源 (Destroy)                                      │
│     terraform destroy → 清理资源                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## Pulumi 概述

### 什么是 Pulumi

Pulumi 是现代化的 IaC 工具，使用真实的编程语言（Python、TypeScript、Go）来定义基础设施。

<Badge text="核心特点" type="tip" /> Pulumi 的核心理念是"真正的代码"——你可以用 for 循环、函数、类来组织基础设施代码。

### Pulumi vs Terraform

| 维度 | Terraform | Pulumi |
|---|---|---|
| 配置语言 | HCL | Python/TypeScript/Go |
| 代码能力 | 有限（HCL 表达式） | 完整编程能力 |
| 状态管理 | 本地或远程 | 本地或远程 |
| 跨云支持 | 原生支持 | 原生支持 |
| 学习曲线 | HCL 简单 | 需要懂编程语言 |
| 生态 | 成熟 | 成长中 |

## Ansible 概述

### 什么是 Ansible

Ansible 是 Red Hat 出品的自动化工具，最初用于配置管理，后来扩展到 IaC 领域。

<Badge text="设计哲学" type="info" /> Ansible 的核心理念是"幂等"和"无代理"——多次执行同一个操作结果一致，且不需要在被管理节点安装代理。

### Ansible vs Terraform

| 维度 | Terraform | Ansible |
|---|---|---|
| 定位 | 基础设施定义 | 配置管理/自动化 |
| 语言 | HCL | YAML |
| 执行模式 | 声明式 | 命令式/声明式 |
| 被管理节点 | 无需代理 | 无需代理 |
| 状态 | 有状态（保存状态文件） | 无状态 |
| 适用场景 | 基础设施创建 | 配置管理、应用部署 |

## IaC 最佳实践

### 代码组织

```
infrastructure/
├── modules/                # 可复用模块
│   ├── vpc/
│   │   ├── main.tf
│   │   ├── variables.tf
│   │   └── outputs.tf
│   └── ec2/
│       ├── main.tf
│       ├── variables.tf
│       └── outputs.tf
├── environments/          # 环境隔离
│   ├── dev/
│   │   ├── main.tf
│   │   ├── terraform.tfvars
│   │   └── backend.tf
│   ├── staging/
│   └── prod/
├── provider.tf            # Provider 配置
├── variables.tf          # 共享变量
└── outputs.tf            # 共享输出
```

### 版本控制

| 实践 | 说明 |
|---|---|
| 单一真相来源 | 所有基础设施代码都在 Git 中 |
| 代码审查 | 所有变更通过 Pull Request |
| 标签规范 | 资源加上环境、项目、所有者标签 |
| 敏感信息 | 使用 Vault 或环境变量，不硬编码 |

### 状态管理

<Badge text="重要" type="warning" /> Terraform 状态文件包含敏感信息，必须妥善管理。

| 存储方式 | 适用场景 |
|---|---|
| 本地 | 本地开发、小项目 |
| S3 + DynamoDB | 团队协作、生产环境 |
| Terraform Cloud | SaaS 托管 |
| Consul | HashiCorp 生态 |
