# Terraform 模块化实战：构建可复用基础设施

Terraform 模块化实战：构建可复用基础设施

你是否有这样的经历：为一个项目写了一份 Terraform 配置，创建了 VPC、子网、EC2 实例、RDS 数据库一套完整的架构。然后第二个项目来了，你把配置复制一份，改改参数，部署上线。第三个项目、第四个项目……复制粘贴的配置文件越来越多，维护变成了一场噩梦。改一个通用参数要在十几个地方改，还经常漏掉某个文件。

模块化（Module）就是为了解决这个问题而生的。它将常用的基础设施模式封装成可复用的组件，让你从"复制粘贴"的工作中解放出来，把精力放在真正需要定制的地方。

## 一、模块的概念

### 1.1 什么是 Terraform 模块？

Terraform 模块本质上就是一个**包含 `.tf` 文件的目录**。这个目录里的配置文件定义了一组逻辑上相关的基础设施资源。模块可以被其他配置调用，被调用时就像使用一个黑盒一样，只需要传入必要的参数，就能得到预设的基础设施。

Terraform 配置中，如果没有被其他配置引用，就是**根模块（Root Module）**。根模块下的所有 `.tf` 文件和资源构成了你的基础设施的入口点。

**没有模块的世界**：

```
project/
├── main.tf           # 2000 行，包含所有资源配置
├── variables.tf      # 100 个变量
├── outputs.tf        # 80 个输出
└── terraform.tfstate # 混乱的状态
```

**使用模块后的世界**：

```
project/
├── main.tf           # 简洁的模块调用
├── variables.tf      # 少量顶层变量
├── outputs.tf        # 少量顶层输出
├── modules/
│   ├── network/      # 网络模块
│   │   ├── main.tf
│   │   ├── variables.tf
│   │   └── outputs.tf
│   ├── ecs/          # ECS 集群模块
│   │   ├── main.tf
│   │   ├── variables.tf
│   │   └── outputs.tf
│   └── rds/         # RDS 数据库模块
│       ├── main.tf
│       ├── variables.tf
│       └── outputs.tf
└── terraform.tfstate
```

:::tip 模块的优势

- **可复用**：一次定义，多处使用
- **可维护**：修改一处，影响全局
- **可测试**：模块可以独立验证
- **可分享**：可以发布到 Terraform Registry 供他人使用
- **封装复杂性**：将复杂的资源配置隐藏在简单的接口后面

:::

### 1.2 什么时候应该创建模块？

模块不是越多越好，过度模块化反而会增加复杂性。以下情况适合创建模块：

| 场景 | 说明 |
|---|---|
| 重复的基础设施模式 | 多个项目需要相同的 VPC/子网结构 |
| 团队共享的基础设施组件 | 多个团队需要使用相同的数据库配置 |
| 需要标准化基础设施 | 强制执行安全基线和标签策略 |
| 复杂资源组合 | 一组资源总是需要一起创建（如 ALB + Target Group + Auto Scaling） |

**不需要模块化的场景**：

- 只有一处使用的基础设施
- 配置非常简单（少于 50 行）
- 资源之间没有逻辑关联

## 二、模块的结构

### 2.1 标准模块结构

一个标准模块通常包含以下文件：

```
module/
├── main.tf          # 必需：资源定义
├── variables.tf    # 推荐：输入变量
├── outputs.tf      # 推荐：输出值
├── versions.tf     # 推荐：版本约束
├── README.md       # 推荐：模块文档
└── locals.tf       # 可选：本地值
```

**main.tf（必需）**：模块的核心资源定义。

```hcl title="modules/network/main.tf"
# 创建 VPC
resource "aws_vpc" "main" {
  cidr_block           = var.vpc_cidr
  enable_dns_hostnames = true
  enable_dns_support   = true
  
  tags = merge(
    var.tags,
    { Name = "${var.project}-vpc" }
  )
}

# 创建公有子网
resource "aws_subnet" "public" {
  count = length(var.availability_zones)
  
  vpc_id                  = aws_vpc.main.id
  cidr_block              = cidrsubnet(var.vpc_cidr, 8, count.index)
  availability_zone       = var.availability_zones[count.index]
  map_public_ip_on_launch = true
  
  tags = merge(
    var.tags,
    { Name = "${var.project}-public-${count.index + 1}" }
  )
}

# 创建私有子网
resource "aws_subnet" "private" {
  count = length(var.availability_zones)
  
  vpc_id            = aws_vpc.main.id
  cidr_block        = cidrsubnet(var.vpc_cidr, 8, count.index + length(var.availability_zones))
  availability_zone = var.availability_zones[count.index]
  
  tags = merge(
    var.tags,
    { Name = "${var.project}-private-${count.index + 1}" }
  )
}

# 创建 Internet Gateway
resource "aws_internet_gateway" "main" {
  vpc_id = aws_vpc.main.id
  
  tags = merge(
    var.tags,
    { Name = "${var.project}-igw" }
  )
}
```

**variables.tf（推荐）**：定义模块接受的输入变量。

```hcl title="modules/network/variables.tf"
variable "project" {
  description = "项目名称"
  type        = string
}

variable "environment" {
  description = "环境名称"
  type        = string
}

variable "vpc_cidr" {
  description = "VPC 的 CIDR 块"
  type        = string
  default     = "10.0.0.0/16"
}

variable "availability_zones" {
  description = "可用区列表"
  type        = list(string)
  default     = ["ap-northeast-1a", "ap-northeast-1c"]
}

variable "tags" {
  description = "附加标签"
  type        = map(string)
  default     = {}
}
```

**outputs.tf（推荐）**：定义模块的输出值，供调用者使用。

```hcl title="modules/network/outputs.tf"
output "vpc_id" {
  description = "VPC ID"
  value       = aws_vpc.main.id
}

output "vpc_cidr" {
  description = "VPC CIDR"
  value       = aws_vpc.main.cidr_block
}

output "public_subnet_ids" {
  description = "公有子网 ID 列表"
  value       = aws_subnet.public[*].id
}

output "private_subnet_ids" {
  description = "私有子网 ID 列表"
  value       = aws_subnet.private[*].id
}

output "availability_zones" {
  description = "可用区列表"
  value       = var.availability_zones
}
```

**versions.tf（推荐）**：约束 Terraform 和 Provider 版本。

```hcl title="modules/network/versions.tf"
terraform {
  required_version = ">= 1.0"
  
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}
```

### 2.2 调用模块

创建模块后，在根模块中通过 `module` 块调用它。

```hcl title="main.tf"
module "network" {
  source = "./modules/network"
  
  project = var.project
  environment = var.environment
  vpc_cidr = "10.0.0.0/16"
  availability_zones = ["ap-northeast-1a", "ap-northeast-1c"]
  tags = {
    ManagedBy = "Terraform"
  }
}
```

调用模块时，只需指定 `source`（模块的来源）和必要的输入变量。Terraform 会自动处理模块内的资源配置。

**模块源的几种写法**：

```hcl
# 本地相对路径
source = "./modules/network"

# 本地绝对路径
source = "/path/to/modules/network"

# Terraform Registry（官方或社区模块）
source = "terraform-aws-modules/vpc/aws"
source = "terraform-aws-modules/ecs/aws"

# Git 仓库
source = "git::https://github.com/org/repo.git//modules/network"
source = "git::ssh://git@github.com/org/repo.git//modules/network?ref=v1.0.0"

# Terraform Cloud 私有注册表
source = "app.terraform.io/org-name/network/aws"
```

:::tip 版本约束

对于可复用的模块（特别是发布到 Registry 的），**强烈建议使用版本约束**。这可以防止模块更新导致的基础设施变更。

```hcl
module "network" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "~> 5.0"
  
  # ...
}
```

:::

## 三、输入变量进阶

### 3.1 变量类型详解

Terraform 模块的变量支持丰富的类型，让接口设计更加精确和安全。

**基础类型**：

```hcl
variable "string_var" {
  type    = string
  default = "hello"
}

variable "number_var" {
  type    = number
  default = 42
}

variable "bool_var" {
  type    = bool
  default = true
}
```

**集合类型**：

```hcl
variable "list_var" {
  type    = list(string)
  default = ["a", "b", "c"]
}

variable "map_var" {
  type    = map(string)
  default = {
    key1 = "value1"
    key2 = "value2"
  }
}

variable "set_var" {
  type    = set(string)
  default = ["item1", "item2"]
}
```

**对象类型**：最适合定义复杂配置的接口。

```hcl
variable "rds_config" {
  type = object({
    identifier     = string
    engine         = string
    engine_version = string
    instance_class = string
    allocated_storage = number
    multi_az       = bool
  })
  default = {
    identifier     = "my-db"
    engine         = "mysql"
    engine_version = "8.0"
    instance_class = "db.t3.micro"
    allocated_storage = 20
    multi_az       = false
  }
}
```

**元组类型**：固定长度的列表，元素类型可以不同。

```hcl
variable "mixed_tuple" {
  type    = tuple([string, number, bool])
  default = ["example", 42, true]
}
```

### 3.2 变量验证

变量验证是 Terraform 0.13 引入的功能，允许你在变量赋值时进行自定义检查。

```hcl
variable "environment" {
  type        = string
  
  validation {
    condition     = contains(["dev", "staging", "production"], var.environment)
    error_message = "环境必须是 dev、staging 或 production。"
  }
}

variable "instance_type" {
  type        = string
  
  validation {
    condition     = can(regex("^t[23]\\.(micro|small|medium)$", var.instance_type))
    error_message = "实例类型必须是 t3.micro、t3.small、t3.medium、t2.micro、t2.small 或 t2.medium。"
  }
}

variable "disk_size" {
  type        = number
  
  validation {
    condition     = var.disk_size >= 10 && var.disk_size <= 1000
    error_message = "磁盘大小必须在 10GB 到 1000GB 之间。"
  }
}
```

:::warning 验证时机

变量验证在 `terraform validate` 和 `terraform plan` 时触发。如果验证失败，Terraform 会显示错误信息并停止执行。这可以防止不合理的配置被应用到基础设施上。

:::

### 3.3 敏感变量

对于包含密码、密钥等敏感信息的变量，可以使用 `sensitive` 属性。

```hcl
variable "db_password" {
  description = "数据库密码"
  type        = string
  sensitive   = true  # 标记为敏感
  default     = ""
}

variable "api_key" {
  description = "第三方 API 密钥"
  type        = string
  sensitive   = true
  default     = ""
}
```

敏感变量在 plan 和 apply 的输出中会被隐藏，但仍然会存储在 Terraform 状态和 plans 文件中。对于生产环境，**应使用远程后端并启用状态加密**。

### 3.4 默认值和空值

Terraform 变量支持默认值。如果调用者没有提供值，就使用默认值。

```hcl
variable "log_retention" {
  description = "日志保留天数"
  type        = number
  default     = 30  # 默认保留 30 天
}
```

如果变量没有默认值且调用者也没有提供，Terraform 会报错。但如果变量声明中有 `nullable = true`（默认），可以将值设置为 `null`：

```hcl
variable "optional_tag" {
  description = "可选标签"
  type        = string
  default     = null  # 允许 null 值
  nullable    = true
}
```

## 四、输出值进阶

### 4.1 输出值类型

输出值可以输出 Terraform 支持的任何值类型：

```hcl
# 简单字符串
output "vpc_id" {
  value = aws_vpc.main.id
}

# 列表
output "subnet_ids" {
  value = aws_subnet.private[*].id
}

# 映射
output "subnet_details" {
  value = {
    for idx, subnet in aws_subnet.private :
    idx => {
      id               = subnet.id
      availability_zone = subnet.availability_zone
      cidr_block       = subnet.cidr_block
    }
  }
}
```

### 4.2 敏感输出

与敏感变量类似，敏感输出不会在常规输出中显示：

```hcl
output "db_master_password" {
  value     = aws_db_instance.main.password
  sensitive = true
}
```

### 4.3 条件输出

```hcl
output "alb_dns_name" {
  description = "ALB 的 DNS 名称"
  value       = module.alb.dns_name
  # 只有在启用了 ALB 时才输出
}
```

### 4.4 依赖关系

Terraform 自动处理模块之间的依赖关系。当一个模块的输出被另一个模块使用时，Terraform 会自动推断依赖顺序：

```hcl
# 这个配置中，Terraform 会自动知道需要先创建 VPC，才能创建 ECS 集群
module "network" {
  source = "./modules/network"
  # ...
}

module "ecs" {
  source    = "./modules/ecs"
  vpc_id    = module.network.vpc_id  # 自动依赖 network 模块
  subnet_ids = module.network.private_subnet_ids
}
```

但有些时候，依赖关系无法自动推断（特别是 Provider 配置或 data sources）。可以使用 `depends_on` 显式声明：

```hcl
module "app" {
  source = "./modules/app"
  
  depends_on = [
    module.database_init
  ]
}
```

## 五、状态管理

### 5.1 模块状态概述

默认情况下，**模块创建的资源状态存储在根模块的状态文件中**。当你调用 `module.network` 时，`aws_vpc` 资源的状态会记录在根目录的 `terraform.tfstate` 文件中，而不是在 `modules/network/` 目录下。

这意味着：

- 状态文件是集中管理的
- 可以用 `terraform state list` 查看所有资源（包括模块资源）
- 模块资源和非模块资源在状态中地位相同

```bash
# 查看状态中所有资源（包括模块资源）
terraform state list

# 输出示例
# module.network.data.aws_availability_zones.available
# module.network.aws_internet_gateway.main
# module.network.aws_route_table.private
# module.network.aws_route_table.public
# module.network.aws_subnet.private
# module.network.aws_subnet.public
# module.network.aws_vpc.main
```

### 5.2 模块资源引用

在 Terraform 配置中，通过 `module.<name>.<resource>.<attribute>` 语法引用模块输出：

```hcl
# 在根模块中引用网络模块的输出
resource "aws_instance" "app" {
  ami           = var.ami_id
  instance_type = var.instance_type
  subnet_id     = module.network.private_subnet_ids[0]
  
  vpc_security_group_ids = [
    module.security_group.app_sg_id
  ]
}
```

### 5.3 状态移动

如果需要重构配置，将某个模块的资源移动到另一个位置（或另一个模块），可以使用 `terraform state mv`：

```hcl
# 将 module.vpc 中的 aws_vpc.main 移动到直接管理
terraform state mv 'module.vpc.aws_vpc.main' 'aws_vpc.main'

# 将资源从一个模块移动到另一个模块
terraform state mv 'module.vpc.aws_subnet.public[0]' 'module.network.aws_subnet.public[0]'
```

:::warning 状态操作风险

`terraform state mv` 是有风险的操作。执行前请确保：

1. 目标位置（新的资源引用）在配置中已正确定义
2. 资源的实际 ID 不会改变
3. 有状态文件的备份

:::

## 六、远程状态后端

### 6.1 为什么需要远程状态后端？

在团队协作或生产环境中，远程状态后端是必需的：

- **共享访问**：团队成员可以同时操作基础设施
- **状态锁定**：防止并发操作导致的状态损坏
- **状态备份**：避免本地状态丢失
- **敏感信息保护**：远程后端支持加密

### 6.2 AWS S3 + DynamoDB 后端

AWS 环境最常用的远程后端组合是 **S3 存储状态 + DynamoDB 实现锁定**。

```hcl title="backend.tf"
terraform {
  backend "s3" {
    bucket         = "my-terraform-state-bucket"
    key            = "project/prod/terraform.tfstate"
    region         = "ap-northeast-1"
    encrypt        = true
    dynamodb_table = "terraform-state-locks"
    
    # 可选：S3 特定配置
    # acl            = "private"
    # versioning     = true
    # ...
  }
}
```

**DynamoDB 表要求**：DynamoDB 表需要启用**按需容量模式或具有吞吐量的表**。

```bash
# 创建 Terraform 状态锁表
aws dynamodb create-table \
  --table-name terraform-state-locks \
  --attribute-name LockID \
  --attribute-type S \
  --key-type HASH \
  --billing-mode PAY_PER_REQUEST
```

:::tip S3 后端最佳实践

**启用版本控制**：在 S3 桶上启用版本控制，这样每次 apply 都会保存一个状态版本，可以回滚到任意历史状态。

**启用加密**：S3 后端默认支持 SSE-KMS 加密，可以指定 KMS 密钥进行加密。

**限制访问**：使用 IAM Policy 限制只有特定人员/角色可以访问状态文件。

**开启日志**：开启 S3 访问日志，记录谁在什么时候访问了状态文件。

:::

### 6.3 Azure Blob Storage 后端

Azure 环境使用 Blob Storage 存储状态：

```hcl title="backend.tf"
terraform {
  backend "azurerm" {
    resource_group_name  = "terraform-state-rg"
    storage_account_name = "tfstateaccount"
    container_name       = "terraform-state"
    key                  = "prod.terraform.tfstate"
    
    # 可选：使用 Azure AD 认证（推荐）
    use_msi              = true
    subscription_id      = "your-subscription-id"
    tenant_id            = "your-tenant-id"
  }
}
```

**创建存储账户**：

```bash
# 创建资源组
az group create --name terraform-state-rg --location eastus

# 创建存储账户
az storage account create \
  --name tfstateaccount \
  --resource-group terraform-state-rg \
  --sku Standard_LRS

# 创建容器
az storage container create \
  --name terraform-state \
  --account-name tfstateaccount
```

### 6.4 Google Cloud Storage 后端

GCP 环境使用 Cloud Storage 存储状态：

```hcl title="backend.tf"
terraform {
  backend "gcs" {
    bucket = "my-terraform-state-bucket"
    prefix = "project/prod"
    
    # 可选：credentials 文件路径
    # credentials = "/path/to/service-account.json"
  }
}
```

### 6.5 Terraform Cloud / HCP Terraform

HashiCorp 的 Terraform Cloud 提供了托管的远程后端，支持状态管理、状态锁定、团队协作、策略检查（Policy as Code）等企业级功能。

```hcl title="backend.tf"
terraform {
  backend "remote" {
    organization = "my-org"
    
    workspaces {
      name = "my-project-prod"
    }
  }
}
```

对于开源项目或个人使用，Terraform Cloud 提供免费的基础计划；对于企业用户，有 Team & Governance 和 Business 计划提供更高级的功能。

## 七、Terraform Registry 与模块复用

### 7.1 公共 Registry

Terraform Registry（registry.terraform.io）是最大的 Terraform 模块市场，包含了数千个社区贡献的模块。这些模块覆盖了 AWS、Azure、GCP、阿里云等主流云平台的各种资源组合。

**热门模块**：

| 模块 | 下载量 | 说明 |
|---|---|---|
| terraform-aws-modules/vpc/aws | 10M+ | 完整的 VPC 解决方案 |
| terraform-aws-modules/ecs/aws | 2M+ | ECS 集群和任务定义 |
| terraform-aws-modules/alb/aws | 3M+ | ALB 负载均衡器 |
| terraform-aws-modules/eks/aws | 3M+ | EKS 集群 |
| terraform-aws-modules/rds/aws | 2M+ | RDS 数据库 |

**使用 Registry 模块**：

```hcl
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "~> 5.0"
  
  name = "my-vpc"
  cidr = "10.0.0.0/16"
  
  azs             = ["ap-northeast-1a", "ap-northeast-1c"]
  private_subnets = ["10.0.1.0/24", "10.0.2.0/24"]
  public_subnets  = ["10.0.11.0/24", "10.0.12.0/24"]
  
  enable_nat_gateway     = true
  single_nat_gateway     = true
  one_nat_gateway_per_az = false
  
  tags = {
    ManagedBy = "Terraform"
    Project   = "my-project"
  }
}
```

### 7.2 模块版本控制

模块应该遵循语义化版本（Semantic Versioning）来管理版本。对于 Terraform 模块：

- **主版本号（Major）**：破坏性变更，不兼容旧版本
- **次版本号（Minor）**：新功能，向后兼容
- **补丁版本号（Patch）**：Bug 修复，向后兼容

**版本约束语法**：

```hcl
module "example" {
  source = "..."
  
  # 精确版本
  version = "1.2.3"
  
  # 大于等于 1.2.0
  version = ">= 1.2.0"
  
  # 大于等于 1.2.0，小于 2.0.0
  version = ">= 1.2.0, < 2.0.0"
  
  # 约等于，允许补丁版本更新（推荐）
  version = "~> 1.2.0"
  
  # 任意可用版本
  version = ">= 1.0.0"
}
```

### 7.3 Git 标签版本

如果你的模块托管在 Git 仓库中，可以使用 git 标签（tag）来标记版本：

```bash
# 初始化仓库
git init
git add .
git commit -m "Initial commit"

# 创建第一个版本
git tag v1.0.0
git push origin main
git push origin v1.0.0
```

在 Terraform 中引用带标签的版本：

```hcl
module "my_module" {
  source = "git::https://github.com/org/repo.git//modules/my_module?ref=v1.0.0"
}
```

## 九、模块设计最佳实践

### 9.1 接口设计原则

模块的输入变量就是它的**公共 API**。好的模块应该有：

**清晰的命名**：变量名应该自解释，不需要看文档就知道怎么用。

```hcl
# 不推荐
variable "cidr" {
  description = "cidr"
}

# 推荐
variable "vpc_cidr_block" {
  description = "VPC 的 CIDR 块，例如 10.0.0.0/16"
}
```

**合理的默认值**：大多数变量应该有默认值，降低调用者的工作量。

```hcl
# 好的默认值设计
variable "environment" {
  description = "部署环境"
  type        = string
  default     = "dev"  # 默认开发环境
  
  validation {
    condition     = contains(["dev", "staging", "production"], var.environment)
    error_message = "环境必须是 dev、staging 或 production。"
  }
}
```

**类型安全**：使用精确的类型定义，让错误在 plan 阶段就能发现。

```hcl
# 推荐：使用 object 类型定义复杂配置
variable "rds_config" {
  type = object({
    identifier         = string
    engine            = string
    instance_class    = string
    allocated_storage = number
  })
  
  validation {
    condition     = contains(["mysql", "postgres"], var.rds_config.engine)
    error_message = "数据库引擎必须是 mysql 或 postgres。"
  }
}
```

### 9.2 模块大小控制

模块的大小需要权衡。**太小的模块**会导致配置碎片化，调用者需要组合大量模块；**太大的模块**会降低灵活性，变成另一个版本的"上帝模块"。

**推荐原则**：

- 模块应该对应一个**完整的、可独立部署的功能单元**
- 一个模块通常包含 **3-15 个相关资源**
- 如果模块超过 500 行配置，考虑拆分

### 9.3 文档编写

每个模块都应该有清晰的文档，至少包含：

- 模块用途
- 输入变量说明（包括类型、默认值、可选值）
- 输出值说明
- 使用示例
- 限制和注意事项

```hcl title="modules/network/README.md"
# Network Module

这个模块创建 VPC、子网、路由表和相关的网络基础设施。

## 用法

```hcl
module "network" {
  source = "./modules/network"
  
  project      = "my-app"
  environment  = "production"
  vpc_cidr     = "10.0.0.0/16"
  
  availability_zones = ["ap-northeast-1a", "ap-northeast-1c"]
  
  tags = {
    ManagedBy = "Terraform"
  }
}
```

## 输入变量

| 变量名 | 描述 | 类型 | 默认值 |
|---|---|---|---|
| project | 项目名称 | `string` | - |
| environment | 环境名称 | `string` | - |
| vpc_cidr | VPC CIDR 块 | `string` | `"10.0.0.0/16"` |
| availability_zones | 可用区列表 | `list(string)` | `["ap-northeast-1a"]` |
| tags | 附加标签 | `map(string)` | `{}` |

## 输出值

| 输出名 | 描述 |
|---|---|
| vpc_id | VPC ID |
| vpc_cidr | VPC CIDR |
| public_subnet_ids | 公有子网 ID 列表 |
| private_subnet_ids | 私有子网 ID 列表 |
```

## 十、总结

Terraform 模块是将基础设施代码化的关键实践。通过模块化，你可以：

**提高复用性**：一次编写，多处使用，减少重复代码。

**统一标准**：通过模块强制执行安全基线、标签策略和命名规范。

**降低复杂性**：将复杂的基础设施隐藏在简单的模块接口后面。

**便于维护**：修改一处，影响全局，而不是在十几个文件中改来改去。

:::details 核心要点回顾

**模块创建**：

- 模块就是一个包含 `.tf` 文件的目录
- 标准结构：`main.tf`、`variables.tf`、`outputs.tf`、`versions.tf`
- 使用 `module` 块调用其他模块

**模块接口**：

- 输入变量定义模块的接口
- 使用类型约束、验证和敏感标记提高安全性
- 输出值暴露模块的内部信息

**状态管理**：

- 模块状态存储在根模块状态文件中
- 使用 `terraform state list` 查看所有资源
- `terraform state mv` 用于重构

**远程后端**：

- S3 + DynamoDB：AWS 环境推荐
- Azure Blob + Table：Azure 环境推荐
- Cloud Storage：GCP 环境推荐
- Terraform Cloud：跨云通用

**模块生态**：

- Terraform Registry 有大量现成模块
- 使用语义化版本管理模块版本
- 私有注册表用于团队内部共享

:::

掌握模块化设计，你就掌握了 Terraform 进阶的钥匙。从简单的资源组合开始，逐步积累自己的模块库，最终你会拥有一套可复用、可维护、可扩展的基础设施代码资产。
