# Docker Compose vs Kubernetes：本地与生产的选择

「Docker Compose 能跑，为什么上了 Kubernetes 就出问题了？」

这是一个常见的问题。本质上，Docker Compose 和 Kubernetes 解决的是不同层面的问题——前者是本地开发工具，后者是生产级容器编排平台。它们的设计目标、核心概念、适用场景都不同，不能简单地把 Compose 的配置「翻译」成 Kubernetes 就完事。

## 两个工具的本质区别

### Docker Compose：本地开发神器

```
┌─────────────────────────────────────────────────────────────┐
│                    Docker Compose 模型                        │
│                                                              │
│  docker-compose.yml (声明式配置)                             │
│      │                                                      │
│      ▼                                                      │
│  ┌─────────────────────────────────────────────────────┐    │
│  │              Docker Engine                            │    │
│  │  ┌─────────┐  ┌─────────┐  ┌─────────┐              │    │
│  │  │Service A│  │Service B│  │Service C│              │    │
│  │  │ (Container)│ │ (Container)│ │ (Container)│       │    │
│  │  └─────────┘  └─────────┘  └─────────┘              │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                              │
│  特点：单主机、本地开发、快速迭代                              │
└─────────────────────────────────────────────────────────────┘
```

**设计目标：**

- 简化本地多容器应用的编排
- 让「一组容器作为应用」运行
- 提供开发、测试、CI 的快速反馈

### Kubernetes：生产级编排平台

```
┌─────────────────────────────────────────────────────────────┐
│                    Kubernetes 模型                           │
│                                                              │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                 Control Plane                         │   │
│  │  API Server │ Scheduler │ Controller Manager │ etcd  │   │
│  └─────────────────────────────────────────────────────┘   │
│                              │                               │
│  ┌───────────────────────────┴───────────────────────────┐  │
│  │                    Data Plane                          │  │
│  │  ┌─────────────────────────────────────────────────┐  │  │
│  │  │                    Node 1                        │  │  │
│  │  │  ┌───────┐ ┌───────┐ ┌───────┐                 │  │  │
│  │  │  │  Pod  │ │  Pod  │ │  Pod  │                 │  │  │
│  │  │  └───────┘ └───────┘ └───────┘                 │  │  │
│  │  └─────────────────────────────────────────────────┘  │  │
│  │  ┌─────────────────────────────────────────────────┐  │  │
│  │  │                    Node N                        │  │  │
│  │  │  ┌───────┐ ┌───────┐                           │  │  │
│  │  │  │  Pod  │ │  Pod  │                           │  │  │
│  │  │  └───────┘ └───────┘                           │  │  │
│  │  └─────────────────────────────────────────────────┘  │  │
│  └───────────────────────────────────────────────────────┘  │
│                                                              │
│  特点：多主机集群、生产级、高可用                              │
└─────────────────────────────────────────────────────────────┘
```

**设计目标：**

- 跨多台机器编排容器
- 提供生产级的高可用和自愈能力
- 支持大规模部署

## 核心概念对照

| Docker Compose | Kubernetes | 区别 |
|---|---|---|
| Service | Deployment + Pod | K8s 的 Pod 是真正的调度单元 |
| Container | Container（Pod 内）| K8s Pod 可包含多个容器 |
| `docker-compose up` | `kubectl apply` | 不同命令集 |
| `build` | Build + Deployment | K8s 不内置构建能力 |
| `ports` | Service + NodePort/ClusterIP | K8s 网络更复杂 |
| `volumes` | PersistentVolume + PVC | K8s 存储抽象更强 |
| `depends_on` | initContainers + Pod 依赖 | K8s 有更多控制手段 |
| `networks` | CNI + NetworkPolicy | K8s 网络插件可替换 |
| `secrets` | Secret | K8s Secret 功能更完善 |

## Docker Compose 典型配置

```yaml
# docker-compose.yml
version: "3.8"

services:
  web:
    build: .
    ports:
      - "8080:8080"
    environment:
      - DB_HOST=db
      - DB_PORT=5432
    depends_on:
      - db
    networks:
      - frontend
      - backend

  db:
    image: postgres:15
    volumes:
      - db-data:/var/lib/postgresql/data
    environment:
      - POSTGRES_DB=myapp
      - POSTGRES_USER=admin
      - POSTGRES_PASSWORD=secret
    networks:
      - backend

volumes:
  db-data:

networks:
  frontend:
  backend:
```

## 等价的 Kubernetes 配置

### Deployment

```yaml
# deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web
  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
      - name: web
        image: myapp:latest
        ports:
        - containerPort: 8080
        env:
        - name: DB_HOST
          value: "db-svc"
        - name: DB_PORT
          value: "5432"
        resources:
          requests:
            memory: "256Mi"
            cpu: "100m"
          limits:
            memory: "512Mi"
            cpu: "500m"
```

### Service

```yaml
# service.yaml
apiVersion: v1
kind: Service
metadata:
  name: web-svc
spec:
  type: ClusterIP
  selector:
    app: web
  ports:
  - port: 80
    targetPort: 8080
---
apiVersion: v1
kind: Service
metadata:
  name: db-svc
spec:
  type: ClusterIP
  selector:
    app: db
  ports:
  - port: 5432
    targetPort: 5432
```

### PersistentVolume

```yaml
# pvc.yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: db-pvc
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 10Gi
```

### ConfigMap + Secret

```yaml
# configmap.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: db-config
data:
  DB_HOST: "db-svc"
  DB_PORT: "5432"
---
# secret.yaml
apiVersion: v1
kind: Secret
metadata:
  name: db-secret
type: Opaque
stringData:
  POSTGRES_PASSWORD: "secret"
```

## 关键差异解析

### 1. 网络模型

**Compose 的网络：**
- 自动创建 bridge 网络
- 服务名即 DNS
- 容器间通过服务名访问

**K8s 的网络：**
- 需要 CNI 插件（Calico/Flannel/Cilium）
- Service 才是服务发现单元
- 有 ClusterIP/NodePort/LoadBalancer 多种类型

```yaml
# K8s: Service 是稳定的入口
# 通过 Service 访问，Service DNS: <service-name>.<namespace>.svc.cluster.local
```

### 2. 存储模型

**Compose 的 volumes：**
```yaml
volumes:
  db-data:  # 匿名卷
  ./host/path:/container/path  # 绑定挂载
```

**K8s 的存储：**
```yaml
# 需要 PersistentVolume + PersistentVolumeClaim
# 支持多种存储后端（NFS/Ceph/Cloud）
```

### 3. 扩缩容

**Compose：**
```bash
docker-compose up -d --scale web=5
```

**K8s：**
```bash
# 手动扩缩容
kubectl scale deployment web --replicas=5

# 自动扩缩容
kubectl autoscale deployment web --min=2 --max=10 --cpu-percent=80
```

### 4. 滚动更新

**Compose：**
```bash
docker-compose up -d --build --no-deps --force-recreate web
```

**K8s：**
```bash
# Deployment 自动管理滚动更新
kubectl rollout restart deployment/web
kubectl set image deployment/web web=myapp:v2
```

## 何时用哪个？

| 场景 | 推荐工具 | 原因 |
|---|---|---|
| 本地开发 | Docker Compose | 快速反馈，与 Docker 无缝集成 |
| 开发测试 CI | Docker Compose | 环境一致性好 |
| 单主机小规模 | Docker Compose | 资源有限 |
| 生产环境 | Kubernetes | 高可用、扩缩容、自愈 |
| 多环境部署 | Kubernetes | Helm/Kustomize 强大 |
| 微服务架构 | Kubernetes | 服务网格、Ingress 等 |
| 大规模集群 | Kubernetes | 社区生态丰富 |

## 最佳实践

### 本地开发用 Compose

```yaml

services:
  web:
    build:
      context: .
      target: development
    volumes:
      - .:/app
      - app-cache:/root/.m2  # Maven 缓存

kind: Deployment

  name: web

  replicas: 3
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0
  template:
    # ...
```

### 使用 Docker Compose 模拟 Kubernetes

如果团队想先用 Compose 熟悉概念：

```yaml
# docker-compose.yml (带有 K8s 风格标签)
services:
  web:
    deploy:
      replicas: 3
      resources:
        limits:
          cpus: '0.5'
          memory: 512M
      restart_policy:
        condition: on-failure
        max_attempts: 3
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8080/health"]
      interval: 30s
      timeout: 10s
      retries: 3
```

## 总结

Docker Compose 和 Kubernetes 不是替代关系，而是互补关系：

- **Docker Compose**：本地开发的利器，快速、简单、直观
- **Kubernetes**：生产部署的标准，功能全面、可扩展、云原生

**推荐工作流：**

```
本地开发 ──── Docker Compose ──── 快速迭代
    │
    │ (转换)
    ▼
测试环境 ──── Kubernetes ───────── 功能验证
    │
    │ (部署)
    ▼
生产环境 ──── Kubernetes ───────── 高可用
```

**记住：**
- Compose 是本地工具，不是生产工具
- 不要期望 Compose 配置能直接迁移到 K8s
- 学习两者，理解差异，选择合适的工具
