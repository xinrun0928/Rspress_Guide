# 容器编排平台对比：选择适合你的方案

「Docker Swarm、Kubernetes、Mesos——都在说容器编排，到底该选谁？」

这是很多团队在容器化初期都会面临的问题。没有银弹，每个平台都有其适用的场景和权衡。本质上，这是一个「工具匹配问题」——你的团队规模、技术栈、运维能力、业务需求，决定了哪个平台是正确答案。

## 三大平台概述

### Docker Swarm：简单即力量

Docker Swarm 是 Docker 官方的原生编排工具，与 Docker Engine 深度集成，安装即用。

### Apache Mesos：数据中心的操作系统

Mesos 采用「资源预留」理念，目标是成为数据中心级的资源管理器。

```
┌─────────────────────────────────────────────────────────────┐
│                       Mesos Master                           │
│  ┌─────────────────────────────────────────────────────┐    │
│  │           Leader Election (ZooKeeper)                │    │
│  └─────────────────────────────────────────────────────┘    │
│                              │                               │
│  ┌───────────────────────────┴───────────────────────────┐  │
│  │                     Mesos Agent                         │  │
│  │  ┌─────────────┐  ┌─────────────┐                    │  │
│  │  │  Executor   │  │   Executor   │                    │  │
│  │  │ (Marathon)  │  │  (Spark)     │                    │  │
│  │  └─────────────┘  └─────────────┘                    │  │
│  └───────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

**特点：**

- 支持多种框架（Marathon、Spark、Hadoop）
- 资源抽象能力强
- 适合数据分析场景
- 运维复杂

## 功能对比

### 核心功能对比表

| 功能维度 | Docker Swarm | Kubernetes | Mesos |
|---|---|---|---|
| **自动负载均衡** | 内置 | Service/Ingress | Marathon |
| **健康检查** | 基础 | Liveness/Readiness | Health Checks |
| **滚动更新** | 支持 | 原生支持 | Marathon |
| **回滚** | 支持 | 原生支持 | Marathon |
| **服务发现** | DNS/VIP | DNS/CoreDNS | DNS |
| **配置管理** | Config | ConfigMap/Secret | 参数化 |
| **存储编排** | 插件 | CSI 生态 | 多种 |
| **网络** | Overlay | CNI 生态 | 网络插件 |
| **自动扩缩容** | 无 | HPA/VPA/KEDA | Marathon |

### 高可用能力

| 维度 | Docker Swarm | Kubernetes | Mesos |
|---|---|---|---|
| Leader 选举 | Raft | etcd Raft | ZooKeeper |
| 多 Master | 3/5/7 节点 | etcd 集群 | ZooKeeper 集群 |
| 故障恢复 | 自动 | 自动 | 自动 |
| 数据持久化 | 本地存储 | PVC/CSI | 持久化卷 |

### 滚动更新与回滚

**Docker Swarm：**

```bash
# 滚动更新
docker service update --image myapp:v2 myservice

# 设置更新策略
docker service update \
  --update-parallelism 2 \
  --update-delay 10s \
  myservice

# 回滚
docker service rollback myservice
```

**Kubernetes：**

```bash
# 滚动更新（Deployment 默认）
kubectl rollout restart deployment/myapp

# 查看更新状态
kubectl rollout status deployment/myapp

# 回滚
kubectl rollout undo deployment/myapp

# 回滚到指定版本
kubectl rollout undo deployment/myapp --to-revision=2
```

### 服务发现与负载均衡

| 平台 | 服务发现 | 负载均衡 |
|---|---|---|
| Swarm | 内置 DNS + VIP | 轮询、随机 |
| K8s | CoreDNS + Service | kube-proxy (iptables/ipvs) |
| Mesos | Marathon lb | Marathon 内置 |

### 场景推荐

| 场景 | 推荐方案 | 原因 |
|---|---|---|
| 小团队快速上手 | Docker Swarm | 学习成本低，5 分钟跑起来 |
| 微服务生产环境 | Kubernetes | 功能全面，生态丰富 |
| 开发测试环境 | K3s | 轻量，资源占用小 |
| 边缘计算 | K3s | 单二进制，低内存 |
| 大数据分析 | Mesos + Spark | 原生支持多种框架 |
| 学习/认证 | Kubernetes | 市场认可度高 |

### Kompose 转换示例

```bash
# 安装 kompose
brew install kompose

# 转换 compose 文件到 k8s

# 输出多个文件
# deployment.yaml
# service.yaml
# redis-service.yaml
# redis-deployment.yaml
```

## 总结

<Badge text="选型建议" type="tip" />

**选 Docker Swarm 如果：**
- 小团队（< 5 人）
- 需要快速交付
- 功能需求简单
- 已有 Docker 使用经验

**选 Kubernetes 如果：**
- 大型团队
- 生产环境
- 需要丰富功能
- 有专职运维

**选 Mesos 如果：**
- 大数据场景
- 需要统一管理多种工作负载
- 有专门的运维团队

**选轻量级 K8s 如果：**
- 边缘/IoT 场景
- 开发测试环境
- 资源受限
- 想用 K8s 生态

<Badge text="趋势" type="info" /> Kubernetes 已成为云原生的事实标准，如果不是特定场景，优先考虑 Kubernetes 或其轻量变体（K3s）。
