# StatefulSet 有状态应用管理

你部署了一个 MySQL 集群，三个节点分别是 mysql-0、mysql-1、mysql-2。突然 mysql-1 所在的节点宕机了，Kubernetes 重新创建了 Pod——但新 Pod 的名字变成了 mysql-3，而不是 mysql-1。这意味着新 Pod 无法加入原来的集群，因为它使用了不同的身份。

这就是 Deployment 无法管理有状态应用的原因——它不关心 Pod 的身份，只关心数量。StatefulSet 的出现，就是为了解决这个问题。

## StatefulSet 解决了什么问题

### Deployment 的问题

```yaml
# Deployment 的 Pod 名称
# 每次重建，名称随机
# web-7d9b6f5c8-xk4pw
# web-7d9b6f5c8-mk7pq  ← 新建的可能是这个名字
```

- Pod 名称随机，无法预测
- Pod IP 变化，无法稳定访问
- PVC 随机分配，无法保证数据持久
- 启动顺序不确定，可能主从没准备好就接入请求

### StatefulSet 的设计目标

| 目标 | 说明 |
|---|---|
| **稳定的网络标识** | Pod 名称固定，hostname 固定 |
| **稳定的存储** | 每个 Pod 有独立的 PVC，Pod 重建后仍挂载相同 PVC |
| **有序部署/扩缩容** | Pod 按序号顺序创建和删除 |
| **有序更新** | Pod 按逆序号顺序更新 |

## 核心概念

### 稳定的网络标识

StatefulSet 的 Pod 名称遵循固定的命名模式：`$(statefulset-name)-$(ordinal)`。

```bash
# StatefulSet: mysql
# Pod 名称永远是：mysql-0, mysql-1, mysql-2
kubectl get pods -l app=mysql

# NAME    READY   STATUS    RESTARTS   AGE
# mysql-0 1/1     Running   0          10d
# mysql-1 1/1     Running   0          10d
# mysql-2 1/1     Running   0          10d
```

### 稳定的存储

每个 Pod 有独立的 PVC，Pod 重建后仍挂载相同存储：

```
mysql-0 ──► data-mysql-0 (PVC)
mysql-1 ──► data-mysql-1 (PVC)
mysql-2 ──► data-mysql-2 (PVC)
```

即使 mysql-0 重新调度，新 Pod 仍然会挂载 data-mysql-0，存储内容不变。

### 有序的部署和扩缩容

```
部署 (OrderedReady)：
mysql-0 Running → mysql-1 Starting → mysql-2 Starting

扩缩容 (OrderedReady)：
扩容：mysql-3 Starting → Running
缩容：mysql-2 Terminating → mysql-1 Terminating → mysql-0 Terminating
```

## 完整示例：MySQL 主从集群

### Headless Service

StatefulSet 必须搭配一个 **Headless Service**（没有 ClusterIP 的 Service）：

```yaml title="mysql-service.yaml"
apiVersion: v1
kind: Service
metadata:
  name: mysql-headless
  labels:
    app: mysql
spec:
  clusterIP: None  # Headless Service 的关键：clusterIP: None
  selector:
    app: mysql
  ports:
  - name: mysql
    port: 3306
```

### StatefulSet 定义

```yaml title="mysql-statefulset.yaml"
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: mysql
spec:
  serviceName: mysql-headless  # 必须匹配 Headless Service 名称
  replicas: 3
  selector:
    matchLabels:
      app: mysql
  template:
    metadata:
      labels:
        app: mysql
    spec:
      initContainers:
      - name: init-mysql
        image: mysql:8.0
        command:
        - bash
        - "-c"
        - |
          # 在 /mnt/config-map 中准备配置文件
          # 根据 ordinal 设置 server-id 和 binlog 配置
          [[ $HOSTNAME =~ -([0-9]+)$ ]] || exit 1
          ordinal=${BASH_REMATCH[1]}
          echo "server-id=$((100 + $ordinal))" > /mnt/conf.d/server-id.cnf
          if [[ $ordinal -eq 0 ]]; then
            echo "log-bin=mysql-bin" >> /mnt/conf.d/server-id.cnf
          fi
        volumeMounts:
        - name: conf
          mountPath: /mnt/conf.d
      containers:
      - name: mysql
        image: mysql:8.0
        env:
        - name: MYSQL_ROOT_PASSWORD
          value: "password"
        ports:
        - name: mysql
          containerPort: 3306
        volumeMounts:
        - name: data
          mountPath: /var/lib/mysql
        - name: conf
          mountPath: /etc/mysql/conf.d
        resources:
          requests:
            cpu: 500m
            memory: 1Gi
          limits:
            cpu: "1"
            memory: 2Gi
        readinessProbe:
          exec:
            command: ["mysqladmin", "ping", "-h", "localhost"]
          initialDelaySeconds: 30
          periodSeconds: 10
      volumes:
      - name: conf
        emptyDir: {}
  volumeClaimTemplates:  # 为每个 Pod 创建独立 PVC
  - metadata:
      name: data
    spec:
      accessModes: ["ReadWriteOnce"]
      storageClassName: "fast-storage"
      resources:
        requests:
          storage: 20Gi
```

## Pod 管理策略

### OrderedReady（默认）

Pod 按序号顺序创建，删除也是逆序号顺序。

```yaml
spec:
  podManagementPolicy: OrderedReady
```

```
创建：0 → 1 → 2
删除：2 → 1 → 0
```

### Parallel

Pod 同时创建和删除，不保证顺序。

```yaml
spec:
  podManagementPolicy: Parallel
```

**适用场景**：Pod 之间没有依赖，可以并行启动和停止。

## 更新策略

### RollingUpdate（默认）

逆序号顺序更新 Pod。

```yaml
spec:
  updateStrategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 1  # 最多不可用 Pod 数
      partition: 2       # 保留前 2 个 Pod 不更新（灰度发布）
```

### OnDelete

只更新被手动删除的 Pod。

```yaml
spec:
  updateStrategy:
    type: OnDelete
```

### 灰度发布示例

利用 `partition` 实现金丝雀发布：

```yaml
spec:
  replicas: 5
  updateStrategy:
    type: RollingUpdate
    rollingUpdate:
      partition: 3  # 只更新 mysql-3 和 mysql-4，前 3 个保持不变
```

## 扩缩容

### 扩容

```bash
# 扩容到 5 个副本
kubectl scale statefulset mysql --replicas=5

# StatefulSet 会按顺序创建新 Pod：
# mysql-0, mysql-1, mysql-2 已存在
# mysql-3 Starting → Running
# mysql-4 Starting → Running
```

### 缩容

```bash
# 缩容到 2 个副本
kubectl scale statefulset mysql --replicas=2

# StatefulSet 会按逆顺序终止 Pod：
# mysql-4 Terminating
# mysql-3 Terminating
# mysql-0, mysql-1, mysql-2 保持不变
```

:::warning 缩容风险
StatefulSet 缩容时，**不会自动删除 PVC**。如果有数据，需要先备份再缩容。
:::

```bash
# 缩容前检查 PVC
kubectl get pvc | grep mysql-data

# 缩容后 PVC 仍然存在
# 如果确认不需要，手动删除
kubectl delete pvc data-mysql-4
kubectl delete pvc data-mysql-3
```

## 就绪检测与探针

### 就绪探针（Readiness Probe）

StatefulSet 需要配置就绪探针，确保 Pod 真正可用后再接收流量：

```yaml
readinessProbe:
  exec:
    command:
    - mysqladmin
    - ping
    - "-h"
    - localhost
  initialDelaySeconds: 30
  failureThreshold: 3
  successThreshold: 1
  periodSeconds: 10
```

### 健康探针（Liveness Probe）

```yaml
livenessProbe:
  exec:
    command:
    - mysqladmin
    - ping
    - "-h"
    - localhost
  initialDelaySeconds: 60
  failureThreshold: 3
  periodSeconds: 20
```

## 数据持久化策略

### VolumeClaimTemplate vs 静态 PVC

**VolumeClaimTemplate（推荐）**：动态创建 PVC，每个 Pod 独立存储。

```yaml
volumeClaimTemplates:
- metadata:
    name: data
  spec:
    accessModes: ["ReadWriteOnce"]
    storageClassName: "fast-storage"
    resources:
      requests:
        storage: 20Gi
```

**静态 PVC**：手动创建 PVC，在 Pod spec 中引用。

```yaml
# 先手动创建 PVC
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: data-mysql-0
spec:
  selector:
    matchLabels:
      app: mysql
      ordinal: "0"
  ...
```

### 数据备份策略

```bash
# 1. 缩容前先备份数据
kubectl exec mysql-0 -- mysqldump -A > backup.sql

# 2. 缩容（PVC 保留）
kubectl scale statefulset mysql --replicas=2

# 3. 检查 PVC 是否保留
kubectl get pvc | grep mysql-data

# 4. 扩容回来
kubectl scale statefulset mysql --replicas=3
```

## 常见问题排查

### Pod 一直处于 Pending

```bash
kubectl describe pod mysql-2

# 常见原因：
# - 没有匹配的 StorageClass
# - 存储资源不足
# - Pod 被调度到无法访问 PV 的节点（WaitForFirstConsumer）
```

### Pod 一直处于 Terminating

```bash
# 检查是否有 finalizers 阻止删除
kubectl get pod mysql-2 -o yaml | grep finalizers

# 如果是 NFS 存储，NFS 客户端可能未正确卸载
kubectl delete pod mysql-2 --grace-period=0 --force

# 检查是否有未正确清理的挂载
kubectl describe node <node-name> | grep -A 10 "Volumes:"
```

### PVC 一直处于 Pending

```bash
# 检查 StorageClass 是否存在
kubectl get storageclass

# 检查 PV 是否可用
kubectl get pv | grep Available

# 检查延迟绑定问题
kubectl describe pvc data-mysql-2
```

## 总结

StatefulSet 的核心价值：

1. **稳定的网络标识**：Pod 名称固定，hostname 固定
2. **稳定的存储**：每个 Pod 有独立 PVC，Pod 重建后数据不丢失
3. **有序的生命周期**：按序号顺序创建和删除
4. **有序的更新**：支持灰度发布和回滚
5. **Headless Service**：配合实现稳定的 Pod 发现

StatefulSet 适用于：
- 数据库（MySQL、PostgreSQL、MongoDB）
- 消息队列（Kafka、RabbitMQ）
- 分布式存储（Cassandra、Ceph）
- 有状态的服务，需要固定身份和持久存储

理解 StatefulSet 的设计理念，才能正确管理有状态应用的生命周期。
