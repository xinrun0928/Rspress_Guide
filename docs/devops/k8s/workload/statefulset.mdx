# StatefulSet 有状态副本集

数据库、消息队列、分布式存储——这些服务都有一个共同点：**每个实例都不是等价的**。

MySQL 主从集群中，主库和从库承担不同的角色，数据必须按顺序同步。Kafka 的 Broker 有自己的 ID，Topic 的 Partition 分布在特定的 Broker 上。Redis Cluster 的每个节点维护着不同的槽位。

对于这类有状态应用，Deployment 的「Pod 都是相同的」这一假设不再成立。StatefulSet 正是 Kubernetes 为有状态工作负载设计的解决方案。

## 核心特性

StatefulSet 为每个 Pod 维护一个稳定的、唯一的身份标识。这个标识在整个 Pod 生命周期中保持不变，即使 Pod 被重建。

### 稳定的网络标识

StatefulSet 的 Pod 名称遵循固定的模式：`$(statefulset-name)-$(ordinal)`。

```bash
# 创建名为 web 的 StatefulSet，有 3 个副本
kubectl get pods -l app=web
```

```
NAME    READY   STATUS    RESTARTS   AGE
web-0   1/1     Running   0          10m
web-1   1/1     Running   0          10m
web-2   1/1     Running   0          10m
```

即使 web-1 被重启或重新调度，它的名称始终是 web-1，网络标识 `web-1.statefulset.svc.cluster.local` 也不会改变。

:::tip 网络标识的意义
稳定的网络标识对于服务发现至关重要。ConfigMap 或其他 Pod 可以通过固定的域名访问特定序号的 Pod，比如 web-0 作为主节点接收写请求，web-1 和 web-2 作为从节点处理读请求。
:::

### 稳定的存储卷

StatefulSet 使用 VolumeClaimTemplate 为每个 Pod 创建独立的持久化存储：

```yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: mysql
spec:
  serviceName: "mysql"
  replicas: 3
  selector:
    matchLabels:
      app: mysql
  template:
    metadata:
      labels:
        app: mysql
    spec:
      containers:
      - name: mysql
        image: mysql:8.0
        volumeMounts:
        - name: data
          mountPath: /var/lib/mysql
  volumeClaimTemplates:
  - metadata:
      name: data
    spec:
      accessModes: ["ReadWriteOnce"]
      resources:
        requests:
          storage: 10Gi
```

当 StatefulSet 扩展到 3 个副本时，会自动创建 3 个 PVC：

```bash
kubectl get pvc
```

```
NAME             STATUS   VOLUME                                     CAPACITY
data-mysql-0     Bound    pvc-xxxxx                                  10Gi
data-mysql-1     Bound    pvc-yyyyy                                  10Gi
data-mysql-2     Bound    pvc-zzzzz                                  10Gi
```

每个 PVC 与对应的 Pod 绑定：mysql-0 使用 data-mysql-0，mysql-1 使用 data-mysql-1，以此类推。当 Pod 被重建时，其数据不会丢失。

:::warning PVC 的生命周期
VolumeClaimTemplate 创建的 PVC 不会随着 StatefulSet 的删除而自动删除，除非设置了 `persistentVolumeClaimRetentionPolicy`。
:::

## 有序部署与扩缩容

StatefulSet 保证了 Pod 的部署和扩缩容顺序：**只有前一个 Pod 变为 Running 和 Ready 后，才会启动下一个 Pod**。

### 部署顺序

创建 3 副本的 StatefulSet 时：

```
web-0: Running ✓ → web-1: Starting... → web-2: Pending
```

web-0 必须完全 Ready 后，web-1 才开始启动。这种顺序保证了有状态服务的初始化依赖，比如主库必须先就绪，从库才能开始同步。

### 扩缩容顺序

扩容时从序号小的开始：

```
web-0 → web-1 → web-2 → web-3
```

缩容时从序号大的开始：

```
web-3 → web-2 → web-1 → web-0
```

为什么要反向缩容？因为序号大的 Pod 可能依赖序号小的 Pod 的数据或服务。反向缩容确保依赖关系不会被破坏。

```yaml
spec:
  podManagementPolicy: OrderedReady  # 默认策略
```

### Parallel 模式

如果你的应用可以并行启动和停止，可以将 `podManagementPolicy` 设为 `Parallel`：

```yaml
spec:
  podManagementPolicy: Parallel
  updateStrategy:
    type: RollingUpdate
```

这样 StatefulSet 会同时启动和停止所有 Pod，不再等待前一个 Pod Ready。

## 有序的终止过程

当删除 StatefulSet 或进行缩容时，Kubernetes 会按照从大到小的顺序终止 Pod：

1. 终止 web-2（序号最大）
2. 等待 web-2 完全终止
3. 终止 web-1
4. 等待 web-1 完全终止
5. 终止 web-0

更重要的是，在终止 Pod 之前，StatefulSet 会确保 Pod 的数据已经安全地写入持久化存储。这对于数据库等需要保证数据一致性的服务至关重要。

## 更新策略

StatefulSet 支持两种更新策略：

### OnDelete

手动控制更新。只有在你手动删除 Pod 时，StatefulSet 才会重建它：

```yaml
spec:
  updateStrategy:
    type: OnDelete
```

这允许你精确控制更新的时机，比如在删除前执行数据同步。

### RollingUpdate

滚动更新，自动重建 Pod。更新顺序也是有序的：从序号大的开始，逐步向序号小的推进。

```yaml
spec:
  updateStrategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 1
```

`maxUnavailable` 控制同时不可用的 Pod 数量。

## 使用场景

StatefulSet 适用于以下场景：

| 场景 | 示例 | 为什么需要 StatefulSet |
|---|---|---|
| 有主从关系 | MySQL 主从 | 稳定的网络标识区分主从 |
| 有状态数据 | Kafka Broker | 持久化存储保留数据 |
| 有序启动 | ZooKeeper 集群 | 保证 quorum 选举 |
| 稳定的身份 | 命名服务 | Pod 名称不变，服务发现稳定 |

### 不适合 StatefulSet 的场景

- 无状态微服务：使用 Deployment 即可
- 临时任务：使用 Job
- 定时任务：使用 CronJob
- 需要共享存储的应用：考虑其他方案

## 完整的 MySQL StatefulSet 示例

```yaml
apiVersion: v1
kind: Service
metadata:
  name: mysql
  labels:
    app: mysql
spec:
  ports:
  - port: 3306
    name: mysql
  clusterIP: None  # Headless Service
  selector:
    app: mysql
---
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: mysql
spec:
  serviceName: "mysql"
  replicas: 3
  selector:
    matchLabels:
      app: mysql
  template:
    metadata:
      labels:
        app: mysql
    spec:
      initContainers:
      - name: init-mysql
        image: mysql:8.0
        command:
        - bash
        - "-c"
        - |
          set -ex
          [[ $HOSTNAME =~ -([0-9]+)$ ]] || exit 1
          ordinal=${BASH_REMATCH[1]}
          echo [mysqld] > /mnt/conf.d/server-id.cnf
          echo server-id=$((100 + $ordinal)) >> /mnt/conf.d/server-id.cnf
          if [[ $ordinal -eq 0 ]]; then
            echo [mysqld] > /mnt/conf.d/replication.cnf
            echo log-bin >> /mnt/conf.d/replication.cnf
          fi
        volumeMounts:
        - name: conf
          mountPath: /mnt/conf.d
      containers:
      - name: mysql
        image: mysql:8.0
        env:
        - name: MYSQL_ROOT_PASSWORD
          value: "password"
        ports:
        - containerPort: 3306
          name: mysql
        volumeMounts:
        - name: data
          mountPath: /var/lib/mysql
        - name: conf
          mountPath: /etc/mysql/conf.d
  volumeClaimTemplates:
  - metadata:
      name: data
    spec:
      accessModes: ["ReadWriteOnce"]
      resources:
        requests:
          storage: 10Gi
```

这个示例展示了一个完整的 MySQL 主从集群配置：

- Headless Service 提供稳定的 DNS 入口
- Init Container 设置 server-id 和复制配置
- VolumeClaimTemplate 为每个 Pod 提供独立存储
- 主从角色通过 Pod 序号区分

## 总结

StatefulSet 是 Kubernetes 处理有状态工作负载的核心资源。它的三个核心特性——稳定的网络标识、稳定的存储、有序的部署和扩缩容——为有状态应用提供了必要的保障。
