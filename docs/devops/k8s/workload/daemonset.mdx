# DaemonSet 守护进程集

想象你需要部署一个日志收集 agent 到集群的每一台机器上，或者需要运行一个网络插件（如 Calico）确保每个节点都参与跨主机网络通信。传统方式需要登录每台机器手动部署，但节点数量可能随时变化。

DaemonSet 正是解决这类问题的答案：它确保指定的工作负载在集群的每个节点（或满足条件的节点）上都运行一份副本。

## 核心概念

DaemonSet 会自动在集群中符合条件的每个节点上创建一个 Pod。当新节点加入集群时，DaemonSet 会自动在新节点上创建 Pod；当节点从集群移除时，这些 Pod 会被垃圾回收。

```yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: log-collector
  namespace: kube-system
spec:
  selector:
    matchLabels:
      app: log-collector
  template:
    metadata:
      labels:
        app: log-collector
    spec:
      containers:
      - name: fluentd
        image: fluent/fluentd:v1.16
        resources:
          limits:
            memory: 200Mi
            cpu: 100m
          requests:
            memory: 200Mi
            cpu: 100m
        volumeMounts:
        - name: varlog
          mountPath: /var/log
        - name: varlibdockercontainers
          mountPath: /var/lib/docker/containers
          readOnly: true
      volumes:
      - name: varlog
        hostPath:
          path: /var/log
      - name: varlibdockercontainers
        hostPath:
          path: /var/log/containers
```

创建 DaemonSet 后，Kubernetes 会自动在每个节点上启动对应的 Pod：

```bash
kubectl get daemonsets -n kube-system
```

```
NAME              DESIRED   CURRENT   READY   UP-TO-DATE   AVAILABLE
calico-node       5         5         5       5            5
kube-proxy        5         5         5       5            5
```

## 节点选择机制

DaemonSet 默认会将 Pod 调度到所有节点，但可以通过 `nodeSelector`、`nodeAffinity` 或 `tolerations` 精确控制调度行为。

### nodeSelector

最简单的方式是使用 `nodeSelector`：

```yaml
spec:
  template:
    spec:
      nodeSelector:
        disktype: ssd
```

这样 DaemonSet 只会在带有 `disktype=ssd` 标签的节点上运行。

### 节点亲和性

更精细的控制可以使用 `nodeAffinity`：

```yaml
spec:
  template:
    spec:
      affinity:
        nodeAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:
            nodeSelectorTerms:
            - matchExpressions:
              - key: topology.kubernetes.io/zone
                operator: In
                values:
                - zone-a
                - zone-b
          preferredDuringSchedulingIgnoredDuringExecution:
          - weight: 10
            preference:
              matchExpressions:
              - key: node.kubernetes.io/disktype
                operator: In
                values:
                - ssd
```

这个配置确保 Pod 优先调度到 zone-a 或 zone-b 的 SSD 节点。

### 容忍与污点

Taints 和 Tolerations 配合使用，可以控制 Pod 在污点节点上的行为：

```yaml
spec:
  template:
    spec:
      tolerations:
      - key: "node-role.kubernetes.io/master"
        operator: "Exists"
        effect: "NoSchedule"
      - key: "node.kubernetes.io/not-ready"
        operator: "Exists"
        effect: "NoExecute"
        tolerationSeconds: 300
```

:::tip Toleration 的配置场景
- `NoSchedule`：允许调度到污点节点，但不调度新 Pod 到已污染节点
- `NoExecute`：如果 Pod 不能容忍污点，立即驱逐；如果设置了 `tolerationSeconds`，则在指定时间后驱逐
- `Exists` 操作符：匹配所有相同 key 的污点，不管 value 是什么
:::

## 更新策略

DaemonSet 支持两种更新策略，与 Deployment 类似但更简单。

### RollingUpdate（默认）

滚动更新会逐个替换 Pod：

```yaml
spec:
  updateStrategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 1  # 同时最多有多少个 Pod 不可用
```

### OnDelete

只有当你手动删除旧的 Pod 时，DaemonSet 才会创建新的 Pod：

```yaml
spec:
  updateStrategy:
    type: OnDelete
```

这允许你控制更新的时机，比如在删除前执行某些清理操作。

## 典型使用场景

### 日志收集

最常见的场景是部署日志收集 agent：

```yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: filebeat
  namespace: kube-system
spec:
  selector:
    matchLabels:
      app: filebeat
  template:
    metadata:
      labels:
        app: filebeat
    spec:
      containers:
      - name: filebeat
        image: elastic/filebeat:8.11.0
        args: [
          "-c", "/etc/filebeat.yml",
          "-e"
        ]
        securityContext:
          runAsUser: 0
        volumeMounts:
        - name: config
          mountPath: /etc/filebeat.yml
          subPath: filebeat.yml
        - name: varlog
          mountPath: /var/log
      volumes:
      - name: config
        configMap:
          name: filebeat-config
      - name: varlog
        hostPath:
          path: /var/log
```

### 监控 Agent

Prometheus Node Exporter、Datadog Agent 等监控组件通常以 DaemonSet 部署：

```yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: node-exporter
  namespace: kube-system
spec:
  selector:
    matchLabels:
      app: node-exporter
  template:
    metadata:
      labels:
        app: node-exporter
    spec:
      hostNetwork: true
      hostPID: true
      containers:
      - name: node-exporter
        image: prom/node-exporter:v1.6.1
        args:
        - --path.procfs=/host/proc
        - --path.sysfs=/host/sys
        - --path.rootfs=/host
        ports:
        - containerPort: 9100
          hostPort: 9100
        resources:
          requests:
            cpu: 100m
            memory: 50Mi
          limits:
            cpu: 200m
            memory: 100Mi
        securityContext:
          privileged: true
        volumeMounts:
        - name: proc
          mountPath: /host/proc
          readOnly: true
        - name: sys
          mountPath: /host/sys
          readOnly: true
        - name: root
          mountPath: /host
          readOnly: true
      volumes:
      - name: proc
        hostPath:
          path: /proc
      - name: sys
        hostPath:
          path: /sys
      - name: root
        hostPath:
          path: /
```

:::warning hostNetwork 和 hostPID
使用 `hostNetwork: true` 和 `hostPID: true` 会绕过容器的网络和进程命名空间隔离，需要谨慎使用，并确保 agent 安全性。
:::

### 网络插件

CNI 插件（如 Flannel、Calico、Canal）通常以 DaemonSet 部署在每个节点上：

```yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: calico-node
  namespace: kube-system
  labels:
    k8s-app: calico-node
spec:
  selector:
    matchLabels:
      k8s-app: calico-node
  template:
    metadata:
      labels:
        k8s-app: calico-node
    spec:
      hostNetwork: true
      tolerations:
      - operator: Exists
        effect: NoSchedule
      - operator: Exists
        effect: NoExecute
      serviceAccountName: calico-node
      containers:
      - name: calico-node
        image: calico/node:v3.26.0
        env:
        - name: CALICO_IPV4POOL_CIDR
          value: "10.244.0.0/16"
        - name: CALICO_NETWORKING_BACKEND
          value: "bird"
        securityContext:
          privileged: true
```

## 调度行为详解

DaemonSet 的调度有两个关键点：

### 默认调度行为

在没有配置任何亲和性规则的情况下，DaemonSet 的 Pod 会被调度到所有符合条件的节点，每个节点最多运行一个 Pod（除非 Pod 指定了多个容器）。

### 调度约束

DaemonSet 控制器直接创建 Pod，而不是通过默认调度器。但在某些情况下，Pod 可能会被调度器拦截：

- Pod 有 `nodeSelector` 或 `nodeAffinity` 约束
- Pod 有 `resource requests` 限制
- 节点有资源不足的情况

从 Kubernetes 1.11 开始，默认调度器接管了 DaemonSet Pod 的调度决策，提供了更好的资源均衡和调度策略一致性。

## 与 Deployment 的对比

| 特性 | DaemonSet | Deployment |
|---|---|---|
| 副本数控制 | 每个节点一个 | 可自定义任意数量 |
| 调度范围 | 符合条件的所有节点 | 符合条件的选择性调度 |
| 用途 | 节点级别服务 | 应用服务 |
| 典型场景 | 日志、监控、网络插件 | Web 服务、API 服务 |
| 滚动更新 | 支持 | 支持 |
| 回滚 | 支持 | 支持 |

## 总结

DaemonSet 是 Kubernetes 中用于部署节点级别工作负载的标准方式。它的核心价值在于：

- **自动覆盖**：新节点加入时自动部署，节点移除时自动清理
- **节点亲和**：精确控制 Pod 运行的节点范围
- **系统级服务**：适合运行 Kubernetes 集群的基础组件
