# Kubernetes 架构

面试官问：「Kubernetes 有哪些核心组件？」

大多数人会背出 kube-apiserver、kube-scheduler、kubelet、kube-proxy 这几个名字。但追问下去——apiserver 的请求处理流程是什么？scheduler 怎么决定把 Pod 调度到哪个节点？为什么 kubelet 不归 scheduler 管理？

这些问题，才是真正考验对 K8s 架构理解深度的地方。

## Kubernetes 架构总览

Kubernetes 采用 **Master-Node** 架构。Master 负责集群管理，Node 负责运行工作负载。

## Control Plane 组件

### kube-apiserver：集群的网关

apiserver 是 Kubernetes 的**唯一入口**。所有组件（kubectl、scheduler、controller-manager、kubelet）都通过 apiserver 与集群通信。

**核心职责**：

- 处理所有 REST API 请求（Pod、Service、Deployment 等 CRUD）
- 身份认证（Authentication）和授权（Authorization）
- 数据验证（Admission Control）
- 将集群状态持久化到 etcd

```bash
# 查看 apiserver 暴露的端口
kubectl get pods -n kube-system -o wide

# apiserver 默认监听 6443 端口（安全端口）
# 启动参数：--secure-port=6443
# 所有请求都必须通过这个端口
```

**请求处理流程**：

```
外部请求 (kubectl / 客户端)
           │
           ▼
    ┌──────────────┐
    │ Authentication│ ← 谁在访问？（证书/Token/OIDC）
    └──────┬───────┘
           ▼
    ┌──────────────┐
    │ Authorization │ ← 有权限吗？（RBAC/ABAC/Webhook）
    └──────┬───────┘
           ▼
    ┌──────────────┐
    │Admission     │ ← 允许操作吗？（Mutating/Validating）
    │  Control     │
    └──────┬───────┘
           ▼
    ┌──────────────┐
    │ Validation   │ ← 数据合法吗？
    └──────┬───────┘
           ▼
    ┌──────────────┐
    │    etcd      │ ← 持久化存储
    └──────────────┘
```

:::tip 水平扩展
apiserver 是无状态的，可以部署多个实例，通过负载均衡对外提供服务。真正的状态存储在 etcd 中。
:::

### etcd：集群大脑

etcd 是分布式键值存储，保存了 Kubernetes 的所有集群状态。

```bash
# etcd 默认端口：2379（客户端通信）、2380（节点间通信）
# 查看 etcd 状态
kubectl get pods -n kube-system -l component=etcd

# etcd 数据默认存储在 /var/lib/etcd/
```

**etcd 的重要性**：

- 如果 etcd 挂了，整个集群不可用
- **永远不要直接在 etcd 上执行写操作**，所有操作必须通过 apiserver
- 生产环境 etcd 必须有备份机制

```bash
# 备份 etcd 数据
ETCDCTL_API=3 etcdctl snapshot save snapshot.db \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key
```

### kube-scheduler：调度的决策者

scheduler 负责为新创建的 Pod 选择最合适的 Node。

**调度流程**：

```
新 Pod 创建（apiserver）
        │
        ▼
┌─────────────────────────┐
│   过滤阶段 (Filtering)   │ → 哪些节点可以运行？
│   - 资源是否足够？        │
│   - 亲和性/反亲和性？     │
│   - 污点和容忍？          │
└───────────┬─────────────┘
            ▼
┌─────────────────────────┐
│  评分阶段 (Scoring)      │ → 哪个节点最好？
│   - 负载最低？           │
│   - 亲和性偏好？          │
│   - 拓扑分布？            │
└───────────┬─────────────┘
            ▼
     选择得分最高的节点
            │
            ▼
   将 Pod 绑定到 Node（写回 apiserver）
```

**常用调度配置**：

```yaml
# 节点选择器
spec:
  nodeSelector:
    disktype: ssd

# 节点亲和性
spec:
  affinity:
    nodeAffinity:
      preferredDuringSchedulingIgnoredDuringExecution:
      - weight: 50
        preference:
          matchExpressions:
          - key: zone
            operator: In
            values: [zone-a]

# 污点容忍
spec:
  tolerations:
  - key: "node-role"
    operator: "Equal"
    value: "gpu"
    effect: "NoSchedule"
```

### kube-controller-manager：所有控制器的集合

controller-manager 运行着多个控制器进程，每个控制器都是一个小型的控制循环。

| 控制器 | 职责 |
|---|---|
| **Node Controller** | 监测节点健康，标记不可用节点 |
| **Replication Controller** | 确保 Pod 副本数符合预期 |
| **Deployment Controller** | 管理 Deployment 的滚动更新 |
| **StatefulSet Controller** | 管理 StatefulSet 的状态 |
| **DaemonSet Controller** | 确保每个节点运行一个 DaemonSet Pod |
| **Job Controller** | 管理 Job，创建一次性任务 Pod |
| **CronJob Controller** | 管理 CronJob，按时创建 Job |
| **ServiceAccount Controller** | 为每个 Namespace 创建默认 ServiceAccount |
| **Endpoint Controller** | 填充 Service 的 Endpoints |
| **Namespace Controller** | 清理已删除 Namespace 的资源 |
| **PersistentVolume Controller** | 管理 PV 和 PVC 的绑定 |

**核心概念：控制循环（Control Loop）**：

```
期望状态（Desired State）
        │
        ▼
┌──────────────────┐
│  观测当前状态      │
│  (Read)          │
└────────┬─────────┘
         ▼
┌──────────────────┐
│  比较差异         │
│  (Analyze)       │
└────────┬─────────┘
         ▼
┌──────────────────┐
│  采取行动         │
│  (Act)           │
└────────┬─────────┘
         │
         └──────────────────────► 循环
```

### cloud-controller-manager（可选）

cloud-controller-manager 负责与云服务商交互：

- **Node Controller**：从云平台获取节点信息
- **Route Controller**：在云平台上设置路由
- **Service Controller**：创建云负载均衡器

## Worker Node 组件

### kubelet：节点守护进程

kubelet 是运行在每个节点上的**代理**，负责管理该节点上的 Pod。

**kubelet 的职责**：

- 向 apiserver 注册节点
- 监听 Pod 创建请求，启动/停止容器
- 汇报节点和 Pod 状态
- 执行健康检查（liveness/readiness）
- 挂载 Volume

**注意**：kubelet **不归 scheduler 管理**。scheduler 只是决定 Pod 应该调度到哪个节点，然后将绑定信息写入 etcd。kubelet 通过 watch apiserver 发现自己被分配了哪些 Pod，然后自行启动。

```
apiserver（写入 etcd）
        │
        │  kubelet 通过 watch 发现新 Pod
        ▼
kubelet（Node-1）
        │
        ├── 创建 Pod 目录：/var/lib/kubelet/pods/<uid>/
        ├── 挂载 Volume
        ├── 下载镜像
        ├── 启动容器（通过 Container Runtime）
        └── 配置网络（通过 CNI）
```

### kube-proxy：网络代理

kube-proxy 运行在每个节点上，负责为 Service 提供**集群内部的负载均衡**。

**工作模式演进**：

| 模式 | 原理 | 特点 |
|---|---|---|
| **userspace** | kube-proxy 在用户态处理请求 | 已废弃，性能差 |
| **iptables** | kube-proxy 维护 iptables 规则 | 稳定，规则多时性能下降 |
| **ipvs** | kube-proxy 借助 IPVS 内核模块 | 高性能，支持更多负载均衡算法 |

```bash
# 查看 kube-proxy 工作模式
kubectl get pods -n kube-system -o wide | grep proxy

# ipvs 模式下的连接规则
# 连接发到 Service IP → iptables/ipvs → 转发到 Pod IP
```

### Container Runtime：容器运行时

容器运行时负责运行容器。Kubernetes 支持多种运行时：

```bash
# 支持的运行时
containerd  # 主流选择
docker      # 通过 cri-dockerd 兼容
CRI-O      # 轻量级，专为 K8s 设计
```

**Pod 创建流程（通过 Container Runtime）**：

```
kubelet
  │
  ▼
Container Runtime (containerd)
  │
  ├── 拉取镜像 (cri → pull image)
  │
  ├── 创建容器 (cri → create container)
  │     ├── pause 容器（网络命名空间）
  │     └── 业务容器
  │
  ├── 启动容器 (cri → start container)
  │
  └── 调用 CNI 配置网络
        │
        └── veth pair → docker0 → iptables
```

## 高可用架构

### Control Plane 高可用

```
                        ┌─────────────┐
                        │ Load Balancer│
                        │  (apiserver) │
                        └──────┬──────┘
                               │
           ┌───────────────────┼───────────────────┐
           │                   │                   │
           ▼                   ▼                   ▼
    ┌─────────────┐     ┌─────────────┐     ┌─────────────┐
    │ Manager-1   │     │ Manager-2   │     │ Manager-3   │
    │ kube-apiserver│   │ kube-apiserver│   │ kube-apiserver│
    │ etcd          │   │ etcd          │   │ etcd          │
    │ controller-mgr│   │ controller-mgr│   │ controller-mgr│
    │ scheduler     │   │ scheduler     │   │ scheduler     │
    └───────┬───────┘     └───────┬───────┘     └───────┬───────┘
            │                     │                     │
            └─────────────────────┼─────────────────────┘
                                  │
                           etcd 集群 (3 节点)
                          (Raft 共识协议)
```

**高可用要点**：

- 至少 3 个 Manager 节点（容忍 1 节点故障）
- etcd 必须与 Control Plane 在同一节点（或使用外部 etcd）
- 所有 Manager 节点共享同一个 etcd 集群
- scheduler 和 controller-manager 通过 Leader 选举保证只有一个活跃实例

## 总结

理解 Kubernetes 架构的关键：

1. **apiserver 是唯一入口**：所有组件都通过它通信
2. **etcd 是状态存储**：集群的核心数据
3. **scheduler 只决定「在哪跑」**：不负责实际启动
4. **kubelet 负责「实际跑」**：自主管理本节点的 Pod
5. **controller-manager 是控制循环**：不断调谐期望状态与实际状态

掌握这些核心关系，才能真正理解 Kubernetes 的工作原理。
