# Pod 探针机制

<Badge text="Principle" type="warning" /> <Badge text="Probe" type="tip" />

你的 Java 应用启动很慢——JVM 需要预热，数据库连接池需要建立，还要加载大量配置到缓存。结果 Pod 刚启动就被 Kubernetes 标记为 Ready，流量打进来后请求全部超时。

这不是应用的问题，而是探针配置的问题。

Kubernetes 提供了三种探针来检测容器的健康状态：**存活探针**、**就绪探针**和**启动探针**。正确配置这些探针，是保证服务稳定性的关键。

## 三种探针概述

| 探针类型 | 作用 | 失败后果 | 使用场景 |
|---|---|---|---|
| **Liveness Probe** | 检测应用是否存活 | 重启容器 | 应用僵死、死锁 |
| **Readiness Probe** | 检测应用是否就绪 | 移除流量 | 启动慢、依赖未就绪 |
| **Startup Probe** | 检测应用是否启动完成 | 暂不检查其他探针 | 启动时间长的应用 |

## 探针检测方式

Kubernetes 支持三种检测方式：

### HTTP GET 探针

最常用的方式，向容器的 HTTP 端点发送 GET 请求：

```yaml
livenessProbe:
  httpGet:
    path: /healthz
    port: 8080
    httpHeaders:
    - name: X-Custom-Header
      value: "health"
  initialDelaySeconds: 15
  periodSeconds: 10
  timeoutSeconds: 5
  failureThreshold: 3
  successThreshold: 1
```

**HTTP 响应码判断**：

| 响应码 | 结果 |
|---|---|
| 200-399 | 成功 |
| 其他 | 失败 |

**健康检查端点示例**：

```java
// Spring Boot 示例
@RestController
public class HealthController {
    
    @GetMapping("/healthz")
    public ResponseEntity<Map<String, Object>> health() {
        Map<String, Object> response = new HashMap<>();
        response.put("status", "UP");
        
        // 检查关键组件
        if (!database.isHealthy()) {
            response.put("status", "DOWN");
            response.put("database", "unhealthy");
            return ResponseEntity.status(503).body(response);
        }
        
        response.put("database", "healthy");
        return ResponseEntity.ok(response);
    }
}
```

### Exec 探针

在容器内执行命令，根据退出码判断：

```yaml
livenessProbe:
  exec:
    command:
    - cat
    - /tmp/healthy
  initialDelaySeconds: 5
  periodSeconds: 10
  timeoutSeconds: 5
  failureThreshold: 3
```

**退出码判断**：

| 退出码 | 结果 |
|---|---|
| 0 | 成功 |
| 非 0 | 失败 |

### TCP Socket 探针

尝试与指定端口建立 TCP 连接：

```yaml
livenessProbe:
  tcpSocket:
    port: 3306
  initialDelaySeconds: 15
  periodSeconds: 10
  timeoutSeconds: 5
  failureThreshold: 3
```

**连接判断**：

| 结果 | 结果 |
|---|---|
| 连接成功 | 成功 |
| 连接失败 | 失败 |

## 探针参数详解

```yaml
livenessProbe:
  httpGet:
    path: /healthz
    port: 8080
    host: localhost          # 探测的主机（默认 Pod IP）
    scheme: HTTP            # HTTP 或 HTTPS
  initialDelaySeconds: 30  # 启动后多久开始检查
  periodSeconds: 10         # 检查间隔
  timeoutSeconds: 5         # 超时时间
  successThreshold: 1       # 连续成功次数（存活探针必须为 1）
  failureThreshold: 3       # 连续失败次数后触发动作
  failureThreshold: 3
```

### 参数选择指南

| 参数 | 说明 | Liveness | Readiness |
|---|---|---|---|
| `initialDelaySeconds` | 启动延迟 | 应大于应用启动时间 | 应大于应用启动时间 |
| `periodSeconds` | 检查间隔 | 10-30s | 5-10s |
| `timeoutSeconds` | 超时 | 1-5s | 1-3s |
| `failureThreshold` | 重试次数 | 2-3 | 1-3 |
| `successThreshold` | 成功次数 | 1 | 1-3 |

## 存活探针（Liveness Probe）

存活探针检测应用是否处于僵死状态。如果应用无响应但进程还在，存活探针可以触发容器重启。

### 使用场景

- 应用出现死锁，无法处理请求
- 应用陷入无限循环
- 内存泄漏导致堆内存耗尽
- 进程僵死，但容器未退出

### 配置示例

```yaml
livenessProbe:
  httpGet:
    path: /actuator/health/liveness
    port: 8080
  initialDelaySeconds: 60    # 给应用足够的启动时间
  periodSeconds: 15          # 相对宽松的检查间隔
  timeoutSeconds: 3
  failureThreshold: 3        # 3次失败后重启
```

### 常见错误配置

```yaml
# 错误配置 1：initialDelaySeconds 太短
initialDelaySeconds: 0  # 应用还没启动就开始检查，必然失败

# 错误配置 2：periodSeconds 太短
periodSeconds: 1  # 频繁检查可能影响性能

# 错误配置 3：检查路径不对
path: /health  # 但实际端点是 /healthz

# 错误配置 4：没有健康检查端点
livenessProbe: {}  # 完全不配置
```

## 就绪探针（Readiness Probe）

就绪探针检测应用是否可以接收流量。只有 Ready 的 Pod 才会被加入 Service 的 Endpoint。

### 与 Liveness 的区别

| 特性 | Liveness | Readiness |
|---|---|---|
| 失败动作 | 重启容器 | 移除流量 |
| 目的 | 保持应用存活 | 管理流量路由 |
| 失败后恢复 | 持续检查，成功后正常 | 连续成功后恢复接收流量 |

### 使用场景

- 应用启动时需要加载数据
- 应用需要等待外部依赖就绪
- 应用处于降级状态，但仍可处理部分请求
- 应用正在执行长时间任务，不应该接收新请求

### 配置示例

```yaml
readinessProbe:
  httpGet:
    path: /ready
    port: 8080
  initialDelaySeconds: 5
  periodSeconds: 5
  timeoutSeconds: 2
  failureThreshold: 1       # 1次失败即移除流量
  successThreshold: 1
```

### 灰度发布场景

就绪探针的典型应用是灰度发布：

```
新版本 Pod 创建
    │
    ▼
Pod 处于 NotReady 状态
    │
    ▼
旧版本 Pod 处理所有流量
    │
    ▼
新版本 Pod Ready
    │
    ▼
流量逐渐切换到新版本
    │
    ▼
旧版本 Pod 终止
```

## 启动探针（Startup Probe）

启动探针用于解决启动时间长的应用的问题。在启动探针通过之前，Liveness 和 Readiness 探针不会执行。

### 解决的问题

```yaml
# 没有 Startup Probe 的问题
livenessProbe:
  initialDelaySeconds: 30  # 需要估算一个固定值

# 问题：
# - 应用启动快时，等待浪费时间
# - 应用启动慢时，Pod 在启动期间被 liveness 杀死
```

### 使用场景

- Java/.NET 应用（JVM 启动慢）
- 大型单体应用（启动加载大量数据）
- 需要预热的应用（机器学习模型加载）
- 依赖多个外部服务的应用

### 配置示例

```yaml
startupProbe:
  httpGet:
    path: /started
    port: 8080
  initialDelaySeconds: 0   # 从容器启动就开始检查
  periodSeconds: 5          # 相对频繁的检查
  timeoutSeconds: 3
  failureThreshold: 30       # 最多重试 30 次
  # 总检查时间：0 + 5 * 30 = 150 秒

livenessProbe:
  httpGet:
    path: /healthz
    port: 8080
  initialDelaySeconds: 0    # 启动探针负责等待
  periodSeconds: 15
  failureThreshold: 3

readinessProbe:
  httpGet:
    path: /ready
    port: 8080
  initialDelaySeconds: 0    # 启动探针负责等待
  periodSeconds: 5
  failureThreshold: 1
```

## 综合配置示例

### Spring Boot 应用

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: spring-boot-app
spec:
  containers:
  - name: app
    image: myapp:1.0
    ports:
    - containerPort: 8080
    livenessProbe:
      httpGet:
        path: /actuator/health/liveness
        port: 8080
      initialDelaySeconds: 60
      periodSeconds: 10
      timeoutSeconds: 5
      failureThreshold: 3
    readinessProbe:
      httpGet:
        path: /actuator/health/readiness
        port: 8080
      initialDelaySeconds: 10
      periodSeconds: 5
      timeoutSeconds: 3
      failureThreshold: 1
      successThreshold: 1
    startupProbe:
      httpGet:
        path: /actuator/health/startup
        port: 8080
      initialDelaySeconds: 0
      periodSeconds: 5
      failureThreshold: 60  # 60 * 5 = 300 秒启动时间
```

### Nginx 应用

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx
spec:
  containers:
  - name: nginx
    image: nginx:1.21
    ports:
    - containerPort: 80
    livenessProbe:
      httpGet:
        path: /healthz
        port: 80
      initialDelaySeconds: 15
      periodSeconds: 10
    readinessProbe:
      httpGet:
        path: /ready
        port: 80
      initialDelaySeconds: 5
      periodSeconds: 5
    # Nginx 启动快，不需要 startupProbe
```

### 数据库连接池应用

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: db-app
spec:
  containers:
  - name: app
    image: db-app:1.0
    startupProbe:
      exec:
        command:
        - sh
        - -c
        - "nc -z localhost 5432 && psql -h localhost -U user -d db -c 'SELECT 1'"
      failureThreshold: 30  # 启动可能需要几分钟
      periodSeconds: 10
    livenessProbe:
      exec:
        command:
        - sh
        - -c
        - "psql -h localhost -U user -d db -c 'SELECT 1'"
      initialDelaySeconds: 30
      periodSeconds: 20
    readinessProbe:
      exec:
        command:
        - sh
        - -c
        - "psql -h localhost -U user -d db -c 'SELECT pg_stat_activity.count(*) FROM pg_stat_activity'"
      initialDelaySeconds: 30
      periodSeconds: 10
```

## 常见问题与排查

```bash
# 查看探针状态
kubectl describe pod <pod-name>

# 查看探针事件
kubectl get events --field-selector involvedObject.name=<pod-name>

# 常见错误
Events:
  Type     Reason                  Age                From             Message
  ----     ------                  ----               ----             -------
  Warning  Unhealthy               12s                kubelet          Liveness probe failed: 
                                                                 HTTP probe failed with statuscode: 500
  Warning  Unhealthy               8s                 kubelet          Readiness probe failed: 
                                                                 HTTP probe failed with statuscode: 503

# 手动测试探针
kubectl exec -it <pod-name> -- curl -s http://localhost:8080/healthz

# 临时禁用探针调试
kubectl run -it --rm debug --image=busybox --restart=Never -- wget -qO- http://<pod-ip>:8080/healthz
```

## 最佳实践

| 原则 | 说明 |
|---|---|
| **健康检查端点轻量化** | `/healthz` 应该只做最基本的检查，避免复杂逻辑 |
| **Liveness 检查核心功能** | 存活探针检查应用进程是否响应即可 |
| **Readiness 检查依赖** | 就绪探针检查数据库、缓存、外部服务是否可用 |
| **Startup 为慢启动保驾护航** | 启动慢的应用一定要配置 Startup Probe |
| **initialDelaySeconds 要足够** | 确保大于应用的最长启动时间 |
| **failureThreshold 不要太低** | 避免网络抖动导致的误杀 |
| **健康检查端点要可访问** | 如果用了 auth middleware，健康检查路径要跳过认证 |

:::tip 建议
将健康检查端点与业务逻辑分离。Spring Boot 的 `/actuator` 是很好的实践，它提供了 `/health/liveness` 和 `/health/readiness` 端点，让应用可以独立控制探针结果。
:::

## 总结

| 探针 | 作用 | 失败动作 | 适用场景 |
|---|---|---|---|
| Startup | 等待应用启动完成 | 阻止 Liveness/Readiness | 启动时间长的应用 |
| Liveness | 检测应用是否存活 | 重启容器 | 死锁、僵死 |
| Readiness | 检测应用是否就绪 | 移除流量 | 启动中、依赖未就绪 |

正确配置探针，可以让 Kubernetes 自动处理应用的故障恢复和流量管理，而无需人工干预。
