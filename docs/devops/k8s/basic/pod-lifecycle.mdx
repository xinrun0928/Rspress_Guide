# Pod 生命周期

<Badge text="Principle" type="warning" /> <Badge text="Lifecycle" type="tip" />

Pod 卡在 `Pending` 状态半小时了，你的告警一直在响。用户开始抱怨服务不可用，你却不知道到底发生了什么。

Pod 的状态是 Kubernetes 中最需要理解的知识点之一。一个 Pod 从创建到销毁，会经历多种状态和条件。如果你不理解这些状态的含义，排查问题就像在黑暗中摸索。

让我带你完整走一遍 Pod 的生命周期。

## Pod 的阶段（Phase）

Pod 的 `status.phase` 字段描述了 Pod 在其生命周期中的**宏观概述**。Kubernetes 定义了五种 Pod 阶段：

| 阶段 | 描述 |
|---|---|
| `Pending` | Pod 已被 Kubernetes 系统接受，但容器镜像尚未创建或正在下载 |
| `Running` | Pod 已绑定到节点，所有容器已创建，至少有一个正在运行 |
| `Succeeded` | Pod 中的所有容器都已成功终止，不会重启 |
| `Failed` | Pod 中的所有容器都已终止，且至少有一个以失败终止 |
| `Unknown` | 无法获取 Pod 的状态，通常是节点通信问题 |

### Pending 的常见原因

```bash
# 查看 Pod 详情
kubectl describe pod <pod-name>

# 常见 Pending 原因及排查方法
Events:
  Type     Reason            Age   From             Message
  ----     ------            ----  ----             -------
  Warning  FailedScheduling   5m    default-scheduler  0/3 nodes are available: 
                                                          1 Insufficient memory, 
                                                          2 node(s) didn't match node selector.
```

**排查思路**：

1. **镜像拉取问题**：`kubectl describe pod` 查看 Events 是否有拉取错误
2. **调度问题**：检查节点资源是否充足、标签是否匹配、亲和性规则
3. **PVC 绑定问题**：PVC 是否 Pending？存储类是否存在？
4. **权限问题**：ServiceAccount 是否存在？RBAC 权限是否足够？

### Running vs Succeeded vs Failed

```
┌─────────────────────────────────────────────────────────────┐
│                      Pod 生命周期                           │
│                                                              │
│  Pending ───────────────────────────────────────→ Running  │
│      │                                               │       │
│      │                                          ┌────┴────┐ │
│      │                                          │         │ │
│      │                                    容器正常退出    │ │
│      │                                          │         │ │
│      │                                    ┌─────┴───┐    │ │
│      │                                    │         │    │ │
│      ▼                              Succeeded    Failed   │ │
│   (永不等                           (任务完成)   (错误)     │ │
│    达此状态)                                          │     │
│                                                  容器非零退出 │
└─────────────────────────────────────────────────────────────┘
```

## Container State（容器状态）

Pod 阶段是宏观描述，容器还有自己的状态。在 `status.containerStatuses` 中，你会看到每个容器的详细状态：

```yaml
status:
  phase: Running
  containerStatuses:
  - name: nginx
    state:
      running:
        startedAt: "2024-01-15T10:30:00Z"
    ready: true
    restartCount: 0
    image: nginx:1.21
    imageID: docker-pullable://nginx@sha256:...
    containerID: docker://abc123...
  - name: sidecar
    state:
      waiting:
        reason: ContainerCreating
        message: "Back-off pulling image..."
    restartCount: 1
```

### 三种容器状态

| 状态 | 说明 | 可能原因 |
|---|---|---|
| `Waiting` | 容器未运行，正在等待启动条件满足 | 镜像拉取中、依赖其他资源 |
| `Running` | 容器正在执行，没有问题 | - |
| `Terminated` | 容器已完成执行，正常或异常退出 | 任务完成、进程崩溃 |

### Terminated 状态的解读

```yaml
state:
  terminated:
    exitCode: 0           # 退出码：0=正常，其他=异常
    signal: 0             # 信号：0=未收到信号
    reason: Completed      # 原因描述
    message: ""           # 详细消息
    startedAt: "..."
    finishedAt: "..."
```

**常见退出码含义**：

| 退出码 | 含义 | 常见原因 |
|---|---|---|
| 0 | 正常退出 | 任务完成、优雅退出 |
| 1 | 容器崩溃 | 应用错误、OOM |
| 126 | 命令不可执行 | 权限问题、文件不存在 |
| 127 | 命令未找到 | 入口命令错误 |
| 137 | 被信号终止 | SIGKILL，通常是 OOM Kill |
| 143 | 优雅终止 | 收到 SIGTERM，正常关闭 |

## Pod Conditions（Pod 条件）

除了阶段，Pod 还有 Conditions 字段，描述 Pod 必须满足的详细条件：

```yaml
status:
  conditions:
  - type: PodScheduled          # Pod 已调度到节点
    status: "True"
    lastProbeTime: null
    lastTransitionTime: "2024-01-15T10:29:00Z"
  - type: Initialized           # Init 容器已成功完成
    status: "True"
    lastProbeTime: null
    lastTransitionTime: "2024-01-15T10:29:01Z"
  - type: ContainersReady       # 所有容器都已就绪
    status: "True"
    lastProbeTime: null
    lastTransitionTime: "2024-01-15T10:30:00Z"
  - type: Ready                 # Pod 可以接收流量
    status: "True"
    lastProbeTime: null
    lastTransitionTime: "2024-01-15T10:30:00Z"
```

### Ready 条件的意义

`Ready` 条件是 Service 决定是否将流量转发到 Pod 的依据。只有当 `Ready=True` 时，Pod 才会被 Endpoint Controller 加入 Service 的 Endpoint 列表。

```bash
# 查看 Pod 的详细条件
kubectl get pod <pod-name> -o yaml | grep -A 20 "conditions:"

# 查看 Pod 是否 Ready
kubectl get pod <pod-name>
NAME         READY   STATUS    RESTARTS   AGE
nginx-pod    1/1     Running   0          5m
```

## Restart Policy（重启策略）

Pod 的 `spec.restartPolicy` 定义了容器终止后的重启行为：

| 策略 | 说明 | 适用场景 |
|---|---|---|
| `Always` | 始终重启 | 默认值，适用于长期运行的服务 |
| `OnFailure` | 仅在容器以非零状态退出时重启 | 批处理 Job |
| `Never` | 从不重启 | 一次性任务 |

```yaml
spec:
  restartPolicy: Always  # 默认值
  containers:
  - name: web-server
    image: nginx
```

### 重启策略与容器状态的关系

```
重启策略: Always
┌────────────────────────────────────────────────────────────┐
│ 容器退出（退出码=0）                                          │
│      │                                                    │
│      ▼                                                    │
│  不重启，等待外部触发重启                                      │
│      │                                                    │
│      ▼                                                    │
│  容器退出（退出码≠0）                                         │
│      │                                                    │
│      ▼                                                    │
│  重启容器                                                   │
└────────────────────────────────────────────────────────────┘

重启策略: OnFailure
┌────────────────────────────────────────────────────────────┐
│ 容器退出（退出码=0）                                          │
│      │                                                    │
│      ▼                                                    │
│  不重启，正常退出                                            │
└────────────────────────────────────────────────────────────┘

重启策略: Never
┌────────────────────────────────────────────────────────────┐
│ 容器退出（任意退出码）                                        │
│      │                                                    │
│      ▼                                                    │
│  不重启，Pod 状态变为 Failed                                  │
└────────────────────────────────────────────────────────────┘
```

## 完整的 Pod 生命周期流程

```
                    用户提交 Pod YAML
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│                      PENDING                                │
│  1. API Server 验证并存储到 etcd                             │
│  2. Scheduler 选择合适的节点                                │
│  3. kubelet 监听到新分配的 Pod                               │
│  4. 拉取容器镜像（可能需要时间）                              │
│  5. 创建容器网络                                             │
└─────────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│                   PENDING → RUNNING                         │
│  1. Init 容器按顺序执行（如果有）                             │
│  2. Init 容器全部成功                                       │
│  3. 主容器启动                                               │
│  4. Post-start hook 执行（如果有）                           │
│  5. 探针开始检查                                             │
└─────────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│                       RUNNING                               │
│  1. 存活探针定期检查                                         │
│  2. 就绪探针定期检查                                         │
│  3. 如果存活探针失败 → 重启容器                               │
│  4. 如果就绪探针失败 → 从 Service 移除流量                     │
└─────────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│              RUNNING → SUCCEEDED/FAILED                     │
│  1. 收到终止信号（SIGTERM）                                 │
│  2. pre-stop hook 执行（如果有）                             │
│  3. 优雅终止期（terminationGracePeriodSeconds）               │
│  4. 容器收到 SIGTERM                                        │
│  5. 容器退出                                                │
│  6. 清理网络和存储                                          │
└─────────────────────────────────────────────────────────────┘
```

## 终止生命周期（Termination Lifecycle）

当 Pod 被删除或被调度驱逐时，会触发终止流程：

```yaml
spec:
  terminationGracePeriodSeconds: 30  # 优雅终止宽限期，默认 30 秒
  containers:
  - name: web
    lifecycle:
      preStop:
        exec:
          command: ["/bin/sh", "-c", "sleep 10"]  # 优雅下线
      postStart:
        exec:
          command: ["/bin/sh", "-c", "echo started"]
```

**终止过程详解**：

```
时间线（terminationGracePeriodSeconds=30）：

T+0s:   API Server 标记 Pod 为 Terminating
T+0s:   kube-proxy 更新 endpoints，移除 Pod IP
T+0s:   kubelet 发送 preStop hook（如果配置）
T+0s:   kubelet 发送 SIGTERM 到容器
T+10s:  preStop hook 完成（如果有 sleep 10）
T+0-30s: 容器内的应用收到 SIGTERM，应该开始优雅关闭
T+30s:  如果应用未退出，发送 SIGKILL
T+30s:  kubelet 强制终止容器
T+30s:  删除 Pod
```

:::warning 常见问题
应用在收到 SIGTERM 后需要执行清理工作（如关闭数据库连接、保存状态），但如果清理时间超过 `terminationGracePeriodSeconds`，进程会被 SIGKILL 强制终止。确保：
1. 清理逻辑尽量快速
2. `terminationGracePeriodSeconds` 设置足够长
3. 应用正确处理 SIGTERM 信号
:::

## Restart Count 与调试

`restartCount` 表示容器被重启的次数。频繁重启通常是问题的信号：

```bash
# 查看重启次数
kubectl get pod -o wide
NAME         READY   STATUS    RESTARTS   AGE
nginx-pod    1/1     Running   3          1h

# 查看重启详情
kubectl describe pod nginx-pod | grep -A 5 "Last State"
```

**OOMEK 的特征**：

```yaml
lastState:
  terminated:
    exitCode: 137
    reason: OOMKilled
    message: "Container was killed because it hit the memory limit."
```

**CrashLoopBackOff 的排查**：

```
CrashLoopBackOff 状态：
┌────────────────────────────────────────────────────────────┐
│ 容器启动                                                    │
│      │                                                    │
│      ▼                                                    │
│  容器崩溃退出                                               │
│      │                                                    │
│      ▼                                                    │
│  kubelet 等待指数退避时间后重启                               │
│      │                                                    │
│      │  [10s] → [20s] → [40s] → [80s] → ...               │
│      │                                                    │
│      ▼                                                    │
│  再次尝试启动                                                │
│      │                                                    │
│      ▼                                                    │
│  如果持续失败，状态显示 CrashLoopBackOff                      │
└────────────────────────────────────────────────────────────┘
```


## 总结

Pod 的生命周期由多个概念组成：

| 概念 | 层级 | 说明 |
|---|---|---|
| Phase | Pod | 宏观状态：Pending、Running、Succeeded、Failed、Unknown |
| Conditions | Pod | 详细条件：PodScheduled、Initialized、ContainersReady、Ready |
| Container State | Container | 单个容器状态：Waiting、Running、Terminated |
| Restart Policy | Pod | 重启策略：Always、OnFailure、Never |
| Restart Count | Container | 重启次数 |

理解这些概念和它们之间的关系，是排查 Pod 问题的第一步。下一节我们将学习如何配置 Pod 的资源请求和限制。
