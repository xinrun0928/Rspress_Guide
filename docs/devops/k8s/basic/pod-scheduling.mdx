# Pod 调度机制

<Badge text="Principle" type="warning" /> <Badge text="Scheduling" type="tip" />

为什么你的 Pod 调度到了这台机器而不是另一台？为什么同一个 Deployment 的两个 Pod 分布得如此不均匀？当你需要对 Pod 进行精细控制时，Kubernetes 提供了哪些机制？

大多数情况下，默认调度器的工作是称职的——它会智能地选择资源最合适的节点。但当你有特殊需求时，比如"这个 Pod 必须调度到 GPU 节点"或者"这两个 Pod 最好不要在同一个节点上"，就需要深入理解调度机制了。

## NodeSelector：最简单的节点选择

NodeSelector 是最简单的调度方式，通过标签匹配选择节点：

```yaml title="nodeselector.yaml"
apiVersion: v1
kind: Pod
metadata:
  name: gpu-pod
spec:
  nodeSelector:
    gpu: "true"        # 节点必须有这个标签
    disktype: "ssd"   # AND 条件
  containers:
  - name: main
    image: tensorflow:latest
```

**使用流程**：

```bash
# 1. 给节点打标签
kubectl label node node-1 gpu=true disktype=ssd

# 2. 创建 Pod（会自动调度到匹配的节点）
kubectl create -f nodeselector.yaml

# 3. 验证调度结果
kubectl get pod gpu-pod -o wide

# 4. 删除标签（如需要）
kubectl label node node-1 gpu-   # 删除 gpu 标签
```

:::tip 适用场景
NodeSelector 适合简单的单条件或多条件"与"选择。如果需要更复杂的逻辑（如"优先但不强制"），需要使用 Node Affinity。
:::

## Node Affinity：更灵活的节点亲和性

Node Affinity 提供了比 NodeSelector 更强大的表达能力：

| 特性 | NodeSelector | Node Affinity |
|---|---|---|
| 操作符 | 仅 equals | In, NotIn, Exists, DoesNotExist, Gt, Lt |
| 条件类型 | 仅有 Required | Required（硬性）+ Preferred（软性） |
| 逻辑组合 | 简单 AND | 支持更复杂的逻辑 |

### RequiredDuringSchedulingIgnoredDuringExecution

硬性要求，调度时必须满足：

```yaml
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: topology.kubernetes.io/zone
            operator: In
            values:
            - zone-a
            - zone-b
          - key: gpu
            operator: Exists  # 标签必须存在
```

### PreferredDuringSchedulingIgnoredDuringExecution

软性偏好，调度时尽量满足但不强制：

```yaml
spec:
  affinity:
    nodeAffinity:
      preferredDuringSchedulingIgnoredDuringExecution:
      - weight: 50
        preference:
          matchExpressions:
          - key: disktype
            operator: In
            values:
            - ssd
      - weight: 30
        preference:
          matchExpressions:
          - key: memory
            operator:Gt
            values:
            - "32"
```

**多个 Preferred 会累加权重**，最终得分最高的节点被选中。

### Affinity 与 Anti-Affinity

**Node Affinity** 是 Pod 与节点的关系：

- **Node Affinity**：Pod 偏好某些节点
- **Node Anti-Affinity**：Pod 偏好避开某些节点

```yaml
# 避开 gpu 节点（Anti-Affinity）
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: gpu
            operator: NotIn
            values:
            - "true"
```

## Pod Affinity 与 Anti-Affinity

Pod Affinity 解决的是 Pod 与 Pod 之间的关系问题。比如：

- "API Server 的 Pod 应该和 Ingress Controller 在一起"（Pod Affinity）
- "两个 Web Pod 不应该在同一个节点上"（Pod Anti-Affinity）

### Pod Anti-Affinity 实现高可用

```yaml title="deployment-anti-affinity.yaml"
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-server
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web
  template:
    metadata:
      labels:
        app: web
    spec:
      affinity:
        podAntiAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:
          - labelSelector:
              matchLabels:
                app: web
            topologyKey: kubernetes.io/hostname  # 同一主机名（节点）不可
        containers:
        - name: nginx
          image: nginx:1.21
```

**拓扑域（topologyKey）**：

| topologyKey | 含义 |
|---|---|
| `kubernetes.io/hostname` | 同一主机（节点） |
| `topology.kubernetes.io/zone` | 同一可用区 |
| `topology.kubernetes.io/region` | 同一区域 |
| `failure-domain.beta.kubernetes.io/*` | 旧版拓扑标签 |

**软性反亲和性**（适合副本数不确定的情况）：

```yaml
affinity:
  podAntiAffinity:
    preferredDuringSchedulingIgnoredDuringExecution:
    - weight: 100
      podAffinityTerm:
        labelSelector:
          matchLabels:
            app: web
        topologyKey: kubernetes.io/hostname
```

:::warning 注意事项
硬性反亲和性（requiredDuringScheduling）可能导致调度失败。如果 Pod 数 > 节点数，部分 Pod 会处于 Pending。实际使用中，推荐使用软性反亲和性（preferredDuringScheduling）。
:::

## Taints 与 Tolerations：节点驱赶 Pod

Taints 和 Tolerations 是 Node Affinity 的"反向"机制——它们让节点可以"拒绝"某些 Pod。

**核心概念**：

| 组件 | 方向 | 说明 |
|---|---|---|
| Taint | 节点 → Pod | 节点"嫌弃"某些 Pod |
| Toleration | Pod → 节点 | Pod"忍受"节点的嫌弃 |

```
┌─────────────────────────────────────────────────────────────┐
│                       节点 (Tainted)                        │
│                                                              │
│  Taint: key=value:NoSchedule                                │
│    "不接受的 Pod，就不要来"                                   │
│                                                              │
│         │                                                   │
│         │ Toleration                                         │
│         ▼                                                   │
│  ┌─────────────────┐  ┌─────────────────┐                   │
│  │ Pod (有容忍)     │  │ Pod (无容忍)    │                   │
│  │ 可以调度 ✓       │  │ 无法调度 ✗       │                   │
│  └─────────────────┘  └─────────────────┘                   │
└─────────────────────────────────────────────────────────────┘
```

### 给节点添加 Taint

```bash
# 基础用法
kubectl taint nodes node-1 dedicated=true:NoSchedule

# 带有 effect
kubectl taint nodes node-1 gpu=true:NoExecute
kubectl taint nodes node-1 maintenance=true:PreferNoSchedule

# 查看节点的 taint
kubectl describe node node-1 | grep Taints
```

### Taint 的 Effect 类型

| Effect | 含义 | 新 Pod | 已有 Pod |
|---|---|---|---|
| `NoSchedule` | 不调度 | 除非有匹配的 Toleration | 不影响 |
| `PreferNoSchedule` | 尽量不调度 | 优先选择其他节点 | 不影响 |
| `NoExecute` | 驱逐 | 除非有匹配的 Toleration | 会驱逐 |

### Pod 添加 Toleration

```yaml
spec:
  tolerations:
  # 匹配 key=value:NoSchedule
  - key: "dedicated"
    operator: "Equal"
    value: "true"
    effect: "NoSchedule"

  # 匹配任意 effect 的 key
  - key: "gpu"
    operator: "Exists"
    effect: "NoExecute"

  # 通配符匹配
  - key: "dedicated"
    operator: "Equal"
    value: "true"
    effect: ""  # 匹配任意 effect

  # 无视 taint（特殊用途，不推荐）
  - operator: "Exists"  # 容忍所有 taint
```

### 常见应用场景

**场景 1：专用节点（Dedicated Nodes）**

```bash
# 标记专用节点
kubectl taint nodes gpu-node dedicated=gpu-pool:NoSchedule

# 只有特殊 Pod 可以调度
kubectl label node gpu-node dedicated=gpu-pool

# Pod 配置
spec:
  tolerations:
  - key: "dedicated"
    operator: "Equal"
    value: "gpu-pool"
    effect: "NoSchedule"
  nodeSelector:
    dedicated: "gpu-pool"
```

**场景 2：节点维护（NoExecute）**

```bash
# 标记节点正在维护
kubectl taint nodes node-1 maintenance=true:NoExecute

# Pod 会被驱逐到其他节点（新 Pod 不会调度来）
# 维护完成后：
kubectl taint nodes node-1 maintenance-
```

**场景 3：临时隔离问题节点**

```bash
# 节点异常，暂时隔离
kubectl taint nodes node-1 node.kubernetes.io/not-ready:NoExecute
# 调试完成后移除
kubectl taint nodes node-1 node.kubernetes.io/not-ready-
```

## 调度策略组合使用

实际生产环境中，通常需要组合多种调度策略：

```yaml title="complex-scheduling.yaml"
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-server
spec:
  replicas: 3
  template:
    spec:
      # 1. 亲和到同区域节点
      affinity:
        # 2. Pod 反亲和，分散到不同节点
        podAntiAffinity:
          preferredDuringSchedulingIgnoredDuringExecution:
          - weight: 100
            podAffinityTerm:
              labelSelector:
                matchLabels:
                  app: api-server
              topologyKey: kubernetes.io/hostname
        # 3. 亲和到同标签的 Pod（如 Ingress）
        podAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:
          - labelSelector:
              matchLabels:
                app: ingress
            topologyKey: topology.kubernetes.io/zone
        # 4. 节点亲和，优先选择高性能节点
        nodeAffinity:
          preferredDuringSchedulingIgnoredDuringExecution:
          - weight: 80
            preference:
              matchExpressions:
              - key: performance-tier
                operator: In
                values:
                - high
      # 5. 容忍特定 taint
      tolerations:
      - key: "dedicated"
        operator: "Equal"
        value: "api-pool"
        effect: "NoSchedule"
      containers:
      - name: api
        image: api-server:latest
```

## 调度调试

```bash
# 查看 Pod 调度失败原因
kubectl describe pod <pod-name>

# Events 部分会显示调度决策
Events:
  Type     Reason            Age   From             Message
  ----     ------            ----  ----             -------
  Warning  FailedScheduling  32s   default-scheduler  0/5 nodes are available: 
                                                         1 Too many pods, 
                                                         2 node(s) had taints that the pod didn't tolerate,
                                                         2 Insufficient cpu.

# 查看节点污点
kubectl get nodes -o custom-columns=NAME:.metadata.name,TAINTS:.spec.taints

# 模拟调度（dry-run）
kubectl create -f pod.yaml --dry-run=client -o yaml | kubectl replace --dry-run=client -o yaml
```

## 总结

| 机制 | 方向 | 适用场景 |
|---|---|---|
| NodeSelector | Pod → Node | 简单标签匹配 |
| Node Affinity | Pod → Node | 复杂条件，软硬要求 |
| Pod Affinity | Pod → Pod | Pod 之间"靠近" |
| Pod Anti-Affinity | Pod ↔ Pod | Pod 之间"远离" |
| Taints | Node → Pod | 节点"拒绝"Pod |
| Tolerations | Pod → Node | Pod"忍受"节点 |

理解这些调度机制，可以让你更精细地控制 Pod 的分布，实现高可用、负载均衡、资源优化等目标。
