# Pod 驱逐机制

<Badge text="Principle" type="warning" /> <Badge text="Eviction" type="tip" />

凌晨 3 点，你的告警系统疯狂报警：多个 Pod 被驱逐了。日志显示节点内存不足，kubelet 开始驱逐低优先级 Pod。但问题是——被驱逐的不是那些无足轻重的测试 Pod，而是你的核心业务服务。

为什么会这样？优先级是怎么定义的？如何保护关键 Pod 不被驱逐？

这就是 Kubernetes 驱逐机制要解决的问题。

## 驱逐的触发条件

Kubernetes 中存在多种驱逐触发源：

| 触发源 | 原因 | 影响范围 |
|---|---|---|
| **节点压力驱逐** | 节点资源不足（内存、磁盘） | kubelet 行为 |
| **API 触发的驱逐** | 节点 drain、资源调整 | kube-controller-manager |
| **调度器抢占** | 高优先级 Pod 需要调度 | kube-scheduler |
| **Pod Disruption Budget** | 自愿中断时的保护机制 | Pod 控制器 |

### 节点压力驱逐

当节点资源不足时，kubelet 会主动驱逐 Pod 来回收资源。这是 kubelet 的自我保护机制。

```bash
# kubelet 的驱逐阈值配置
--eviction-hard=memory.available<100Mi,nodefs.available<10%
--eviction-soft=memory.available<500Mi,nodefs.available<15%
--eviction-soft-grace-period=memory.available=30s
--eveleration-pressure-transition-period=5m
```

**驱逐阈值类型**：

| 类型 | 触发条件 | kubelet 行为 |
|---|---|---|
| `eviction-hard` | 硬阈值，达到即驱逐 | 立即驱逐 |
| `eviction-soft` | 软阈值，达到后等待宽限期再驱逐 | 等待宽限期后驱逐 |
| `eviction-grace-period` | 软阈值的等待时间 | - |
| `eviction-minimum-reclaim` | 驱逐后保留的最小资源量 | 确保驱逐后有足够空间 |

**内存压力驱逐顺序**：

```
节点内存使用率逼近 100%

    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│  1. BestEffort Pod（最早）                                  │
│     没有任何资源请求的 Pod                                    │
└─────────────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│  2. Burstable Pod（按使用量超过请求的比例）                   │
│     使用的资源越多，越优先被驱逐                               │
└─────────────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│  3. Guaranteed Pod（最后）                                  │
│     Request=Limit 的 Pod                                    │
└─────────────────────────────────────────────────────────────┘
```

### 磁盘压力驱逐

除了内存，kubelet 还会监控磁盘空间：

| 信号 | 说明 | 默认阈值 |
|---|---|---|
| `memory.available` | 可用内存 | `100Mi` |
| `nodefs.available` | 节点根文件系统 | `10%` |
| `nodefs.inodesFree` | inode 数量 | `5%` |
| `imagefs.available` | 容器镜像文件系统 | `15%` |
| `imagefs.inodesFree` | 镜像文件系统 inode | `5%` |
| `volumefs.available` | PVC 使用的文件系统 | `15%` |

## 优先级与抢占机制

### PriorityClass

PriorityClass 定义了 Pod 的优先级：

```yaml title="priorityclass.yaml"
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: high-priority
value: 10000  # 数值越大，优先级越高
globalDefault: false  # 是否作为默认值
description: "This priority class is for production workloads"
```

**系统保留优先级**：

| 优先级值 | 类别 | 用途 |
|---|---|---|
| 2000000000+ | 系统关键 | kube-system Pod，如 API Server |
| 1000000000+ | 集群级别 | 集群基础设施组件 |
| 0 | 默认 | 普通用户 Pod |
| 负数 | 禁止使用 | - |

### PriorityClass 的使用

```yaml title="pod-with-priority.yaml"
apiVersion: v1
kind: Pod
metadata:
  name: critical-app
spec:
  priorityClassName: high-priority  # 引用 PriorityClass
  containers:
  - name: app
    image: app:latest
```

### 抢占机制（Preemption）

当高优先级 Pod 无法调度时，Scheduler 可能"抢占"（驱逐）低优先级 Pod 来为高优先级 Pod 腾出空间：

```
调度高优先级 Pod
    │
    ▼
无可用节点
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│           抢占（Preemption）过程                             │
│                                                              │
│  1. 识别候选节点（可以驱逐低优先级 Pod 腾出空间）              │
│  2. 计算需要驱逐多少低优先级 Pod                              │
│  3. 驱逐低优先级 Pod                                         │
│  4. 将高优先级 Pod 调度到释放的空间                            │
└─────────────────────────────────────────────────────────────┘
```

**抢占过程详解**：

```bash
# Pod 被调度前的状态
kubectl get pod -o wide

NAME              PRIORITY    NODE
low-priority-1    1000        node-1
low-priority-2    1000        node-1
high-priority     10000       (Pending)

# 调度器决定抢占
kubectl describe pod high-priority | grep -A 5 "Events"

Events:
  Type     Reason             Age   From             Message
  ----     ------             ----  ----             -------
  Warning  FailedScheduling    30s   scheduler       0/3 nodes are available: 
                                                       1 Insufficient memory.
                                                       Attempting to preempt: 
                                                       pods/low-priority-1 in default.
```

**Preemption 的注意事项**：

- 抢占不是立即发生的，有延迟
- 抢占后的 Pod 状态变为 `Failed` 或 `Terminating`
- 被抢占 Pod 的 ReplicaSet 会创建新的 Pod

:::warning 潜在问题
1. **抢占可能造成颠簸**：被驱逐的 Pod 可能在其他节点被重新创建，然后又抢占回来
2. **公平性问题**：某些 Pod 可能永远抢不过其他 Pod
3. **非确定性**：调度结果可能因时序而不同
:::

## Pod Disruption Budget (PDB)

PDB 用于限制自愿中断（如节点 drain、Deployment 滚动更新）期间可被驱逐的 Pod 数量：

```yaml title="pdb.yaml"
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: web-pdb
spec:
  # 至少保持 3 个 Pod 可用
  minAvailable: 3
  
  # 或者使用百分比
  # minAvailable: 80%
  
  selector:
    matchLabels:
      app: web
```

### PDB 的两种模式

```yaml
# 模式 1：最小可用数
spec:
  minAvailable: 3

# 模式 2：最大不可用数
spec:
  maxUnavailable: 1
  # 对应的场景：replicas=3, maxUnavailable=1
  # 意味着滚动更新时，最多有 1 个 Pod 不可用
```

### PDB 的工作原理

```
kubectl drain node-1

    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│               节点 drain 流程                                │
│                                                              │
│  1. kubectl 向 API Server 发送 drain 请求                    │
│  2. API Server 协调 Controller 执行驱逐                       │
│  3. Controller 检查 PDB：当前可驱逐数量？                      │
│  4. 如果达到限制，等待                                         │
│  5. 如果未达到限制，执行驱逐                                   │
│  6. 重复直到节点上的 Pod 都被驱逐                              │
└─────────────────────────────────────────────────────────────┘
```

### PDB 与非自愿驱逐

**重要**：PDB 只保护 Pod 免受**自愿中断**（Voluntary Disruption），对**非自愿中断**（Involuntary Disruption）无效：

| 类型 | PDB 保护 | 示例 |
|---|---|---|
| 自愿中断 | ✓ | kubectl drain、kubectl delete、更新部署 |
| 非自愿中断 | ✗ | 节点 OOM、硬件故障、网络分区 |

## 优雅终止与优雅期

### terminationGracePeriodSeconds

```yaml
spec:
  terminationGracePeriodSeconds: 60  # 默认 30 秒
  containers:
  - name: app
    image: app:latest
```

**终止流程**：

```
T=0:   发送 SIGTERM 到容器
T=0-60: 容器内的应用执行优雅关闭
        关闭监听端口
        完成现有请求
        保存状态
T=60:  如果应用未退出，发送 SIGKILL
```

### PreStop Hook

PreStop Hook 在容器收到 SIGTERM 之前执行：

```yaml
spec:
  containers:
  - name: nginx
    lifecycle:
      preStop:
        exec:
          command: ["/bin/sh", "-c", "nginx -s quit"]
    terminationGracePeriodSeconds: 60
```

**典型场景**：

1. **等待负载均衡器移除**：某些 LB 需要几秒来更新配置
2. **等待连接耗尽**：允许现有连接处理完毕
3. **发送退出信号到其他进程**

### 优雅关闭的最佳实践

```yaml title="graceful-shutdown-example.yaml"
apiVersion: v1
kind: Pod
metadata:
  name: web-server
spec:
  terminationGracePeriodSeconds: 90  # 给足时间
  containers:
  - name: nginx
    image: nginx:1.21
    lifecycle:
      preStop:
        # 等待 5 秒让 kube-proxy 更新 endpoints
        exec:
          command: ["/bin/sh", "-c", "sleep 5 && nginx -s quit"]
    readinessProbe:
      httpGet:
        path: /healthz
        port: 80
      # 不要让探针杀死慢启动的应用
      initialDelaySeconds: 10
      periodSeconds: 5
      failureThreshold: 3
```

## 保护关键 Pod

### 策略 1：使用高优先级

```yaml
# 为关键 Pod 设置高优先级
spec:
  priorityClassName: system-cluster-critical
  # 或更高
  # priorityClassName: system-node-critical
```

### 策略 2：限制低优先级 Pod

```yaml
apiVersion: v1
kind: LimitRange
metadata:
  name: priority-limit
spec:
  limits:
  - type: Pod
    max:
      # 限制普通 Pod 的优先级不能太高
      # 这个策略可以配合 ResourceQuota 使用
```

### 策略 3：使用 Pod QoS

```yaml
# 为关键 Pod 设置 Guaranteed QoS
spec:
  containers:
  - name: critical-app
    resources:
      requests:
        cpu: "500m"
        memory: "256Mi"
      limits:
        cpu: "500m"
        memory: "256Mi"  # Request = Limit = Guaranteed
```

### 策略 4：配置 PDB

```yaml title="web-pdb.yaml"
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: web-pdb
spec:
  # 保证至少 80% 的 Pod 可用
  minAvailable: 80%
  selector:
    matchLabels:
      app: web
```

## 驱逐调试

```bash
# 查看节点状态和驱逐事件
kubectl describe node <node-name>

# Events 中的驱逐记录
Events:
  Type     Reason                  Age                  From             Message
  ----     ------                  ----                 ----             -------
  Warning  Evicted                 5m2s                 kubelet          Pod was evicted: 
                                                                 node ephemeral-storage usage ...

# 查看 Pod 的驱逐历史
kubectl get pod <pod-name> -o yaml | grep -A 5 "deletionTimestamp"

# 检查节点资源压力
kubectl top node <node-name>

# 查看 Pod 的优先级
kubectl get pod <pod-name> -o jsonpath='{.spec.priority}'
```

## 总结

| 概念 | 说明 |
|---|---|
| 节点压力驱逐 | kubelet 检测到资源不足时自动驱逐 |
| QoS 驱逐顺序 | BestEffort > Burstable > Guaranteed |
| PriorityClass | 定义 Pod 优先级 |
| 抢占机制 | 高优先级 Pod 可驱逐低优先级 Pod |
| PDB | 限制自愿中断时的最大不可用数 |
| 优雅终止 | SIGTERM + terminationGracePeriodSeconds |

**保护关键 Pod 的最佳实践**：

1. 使用 `system-cluster-critical` 或更高优先级
2. 设置 Guaranteed QoS（Request = Limit）
3. 配置 PodDisruptionBudget
4. 合理设置 `terminationGracePeriodSeconds`
5. 监控 Pod 驱逐事件，及时调整策略
