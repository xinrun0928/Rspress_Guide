# Kubernetes 核心概念

<Badge text="Principle" type="warning" /> <Badge text="Workload" type="tip" />

很多人在学习 Kubernetes 时，第一个混淆的概念就是：**Pod 和 Container 到底有什么区别？**

有人说"Pod 就是容器"，有人说"Pod 包含多个容器"。这两种说法都不够准确。要理解 Kubernetes，首先要理解它最核心的工作负载抽象。

让我从一个真实场景开始：你的 Java 应用需要接入 Prometheus 监控，需要挂载配置，需要日志收集。如果把这些都塞进一个容器里，镜像会变得臃肿，职责也不清晰。但如果分成多个容器，它们之间如何协作？网络怎么通？日志怎么统一收集？

这就是 Pod 设计要解决的问题。

## Pod：最小调度单元

Pod 是 Kubernetes 中的**最小部署单元**。它是一个或多个容器的集合，这些容器共享网络命名空间和存储卷。

### 为什么 Pod 不是容器？

思考这样一个问题：Docker 的容器已经提供隔离了，为什么 Kubernetes 还要引入 Pod 这一层抽象？

**关键洞察**：Pod 不仅仅是容器，它是一个**逻辑主机**。

```
┌────────────────────────────────────────────────────────────┐
│                         Pod                                 │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐    │
│  │  Container   │  │  Container   │  │  Container   │    │
│  │   (主应用)    │  │   (日志代理)  │  │   (监控)     │    │
│  └──────────────┘  └──────────────┘  └──────────────┘    │
│                                                              │
│  ════════════════ 共享网络命名空间 ════════════════        │
│  ════════════════ 共享存储卷 ═══════════════════           │
│                                                              │
│  localhost:8080   localhost:9090   localhost:8081           │
└────────────────────────────────────────────────────────────┘
```

**共享意味着什么？**

- **共享网络**：Pod 内的容器可以通过 `localhost` 互相访问，共享同一个 IP 地址
- **共享存储**：Pod 内的容器可以挂载同一个 Volume，实现数据共享
- **共享 PID 命名空间**：默认情况下，容器可以看到彼此的进程（可用 `shareProcessNamespace: true` 控制）

### Pause 容器：隐形的守护者

每个 Pod 都有一个特殊的容器：**Pause 容器**（也叫 Infra 容器）。这个容器几乎不做什么事情，只是持有网络命名空间。

```
┌─────────────────────────────────────────────────────────────┐
│                      Pod                                    │
│  ┌────────────────┐  ┌────────────────┐  ┌────────────┐  │
│  │    nginx        │  │    redis       │  │   pause    │  │
│  │    :80          │  │    :6379       │  │    :0      │  │
│  └────────────────┘  └────────────────┘  └────────────┘  │
│                                                              │
│  所有容器共享 pause 容器的网络栈                              │
└─────────────────────────────────────────────────────────────┘
```

Pause 容器的存在，使得 Pod 内的其他容器可以独立启动和停止，而不影响网络命名空间。它就像一个锚点，让 Pod 的生命周期有了稳定的载体。

### Pod 的短暂性

Pod 在 Kubernetes 中是**临时的**——它们不是持久实体。当 Pod 被删除、节点故障、或调度发生时，Pod 可能会被销毁并重新创建。

这引出了一个重要概念：**Pod 不适合直接管理**。如果你直接创建一个 Pod，当它被删除或失败时，不会自动创建新的。而 Deployment、ReplicaSet 等工作负载会管理 Pod 的生命周期。

:::warning 重要提示
Pod 是临时性的，不要直接创建裸 Pod 来运行长期服务。应该使用 Deployment、StatefulSet 等工作负载资源来管理 Pod。
:::

## ReplicaSet：Pod 的复制器

ReplicaSet 的职责很简单：**确保指定数量的 Pod 副本始终在运行**。

```yaml title="replicaset.yaml"
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: nginx-replicaset
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.21
        ports:
        - containerPort: 80
```

**ReplicaSet 的工作原理**：

```
期望状态：3 个 Pod
当前状态：2 个 Pod（其中一个被删除了）

→ ReplicaSet Controller 检测到差异
→ 创建 1 个新的 Pod
→ 调度到合适节点
→ 节点上的 kubelet 启动容器
```

### Label 与 Selector

ReplicaSet 通过 **Label Selector** 来识别它需要管理的 Pod：

```yaml
# ReplicaSet 的 selector
selector:
  matchLabels:
    app: nginx

# Pod 的 labels
template:
  metadata:
    labels:
      app: nginx  # 必须匹配 selector
      version: v1  # 可以添加额外标签
```

**matchLabels** 和 **matchExpressions**：

```yaml
selector:
  matchLabels:
    app: nginx      # 精确匹配
  matchExpressions:
    - key: tier
      operator: In
      values: [frontend, backend]  # 标签值在列表中
    - key: environment
      operator: NotIn
      values: [dev]  # 标签值不在列表中
```

:::tip 实用技巧
Label 是 Kubernetes 中最重要的组织机制。善用 labels 可以让你更灵活地管理资源，比如用 `kubectl get pods -l 'app=nginx,tier=frontend'` 来筛选特定条件的 Pod。
:::

## Deployment：声明式更新的基础

Deployment 是 Kubernetes 中最常用的工作负载类型。它在 ReplicaSet 基础上增加了**声明式更新**和**回滚能力**。

### 为什么需要 Deployment？

ReplicaSet 只管复制，Pod 的更新需要手动处理。Deployment 则提供了：

- **滚动更新**：逐步替换旧版本 Pod
- **回滚**：恢复到之前的版本
- **暂停/恢复**：可以在更新过程中暂停
- **状态追踪**：记录每个版本的历史

### Deployment 的滚动更新策略

```yaml title="deployment.yaml"
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
  labels:
    app: nginx
spec:
  replicas: 3
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1        # 最多超出期望副本数
      maxUnavailable: 0  # 滚动过程中有多少 Pod 可以不可用
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.21
        ports:
        - containerPort: 80
```

**滚动更新过程**：

```
初始状态：nginx:1.20 [Pod-1] [Pod-2] [Pod-3]

第1步：创建 nginx:1.21 [Pod-4]
       nginx:1.20 [Pod-1] [Pod-2] [Pod-3] [Pod-4]

第2步：终止一个旧 Pod
       nginx:1.20 [Pod-1] [Pod-2]
       nginx:1.21 [Pod-3] [Pod-4]

第3步：重复直到全部更新
       nginx:1.21 [Pod-1] [Pod-2] [Pod-3]

Deployment 完成！
```

### 版本回滚

```bash
# 查看部署历史
kubectl rollout history deployment/nginx-deployment

# 回滚到上一个版本
kubectl rollout undo deployment/nginx-deployment

# 回滚到指定版本
kubectl rollout undo deployment/nginx-deployment --to-revision=2
```

:::tip maxSurge 和 maxUnavailable 的选择
- `maxSurge=1, maxUnavailable=0`：保守策略，滚动过程需要更多资源，但服务更稳定
- `maxSurge=0, maxUnavailable=1`：激进策略，资源利用率高，但中间状态会有服务能力下降
- `maxSurge=1, maxUnavailable=1`：平衡策略
:::

## StatefulSet：有状态应用的管理者

StatefulSet 专门用于管理**有状态应用**。与 Deployment 不同，StatefulSet 保证：

- **稳定的网络标识**：每个 Pod 有固定的名称
- **稳定的存储**：每个 Pod 有独立的 PVC
- **有序部署/扩缩容**：Pod 按顺序创建和删除
- **有序更新**：Pod 按反向顺序更新

### StatefulSet vs Deployment

| 特性 | Deployment | StatefulSet |
|---|---|---|
| Pod 名称 | 随机后缀 | 固定序号后缀 |
| 存储 | 共享或随机 | 每个 Pod 独立 PVC |
| 创建顺序 | 无要求 | 按序号顺序 |
| 删除顺序 | 无要求 | 逆序号顺序 |
| 扩缩容 | 同时进行 | 按序号顺序 |
| 用途 | 无状态服务 | 数据库、消息队列 |

### StatefulSet 示例

```yaml title="statefulset.yaml"
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: mysql
spec:
  serviceName: mysql-headless  # Headless Service 名称
  replicas: 3
  selector:
    matchLabels:
      app: mysql
  template:
    metadata:
      labels:
        app: mysql
    spec:
      containers:
      - name: mysql
        image: mysql:8.0
        env:
        - name: MYSQL_ROOT_PASSWORD
          value: "password"
        volumeMounts:
        - name: data
          mountPath: /var/lib/mysql
  volumeClaimTemplates:  # 为每个 Pod 创建独立 PVC
  - metadata:
      name: data
    spec:
      accessModes: ["ReadWriteOnce"]
      resources:
        requests:
          storage: 10Gi
```

**创建的 Pod 和 PVC**：

```
mysql-0 → data-mysql-0 (10Gi)
mysql-1 → data-mysql-1 (10Gi)
mysql-2 → data-mysql-2 (10Gi)
```

即使 mysql-0 重新调度，新 Pod 仍然会挂载 data-mysql-0，这个 PVC 的内容和之前一样。

## DaemonSet：节点守护进程

DaemonSet 确保**每个节点**（或符合特定条件的节点）上都运行一个 Pod 副本。

### 典型应用场景

- **日志收集器**：Fluentd、Filebeat
- **监控代理**：Prometheus Node Exporter、Datadog Agent
- **网络插件**：Calico、Cilium
- **存储守护进程**：GlusterFS、Ceph

### DaemonSet 示例

```yaml title="daemonset.yaml"
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: node-exporter
  labels:
    app: prometheus
spec:
  selector:
    matchLabels:
      app: node-exporter
  template:
    metadata:
      labels:
        app: node-exporter
    spec:
      tolerations:
      - key: node-role.kubernetes.io/master
        effect: NoSchedule
      containers:
      - name: node-exporter
        image: prom/node-exporter:v1.3.1
        ports:
        - containerPort: 9100
          name: metrics
```

**DaemonSet 的调度特点**：

- 不使用标准的 Scheduler 调度
- kubelet 直接在每个节点上创建 Pod
- 新节点加入集群时，自动在该节点上创建 Pod
- 节点离开集群时，自动删除该节点上的 Pod

### 与 Deployment 的区别

| 特性 | Deployment | DaemonSet |
|---|---|---|
| 副本数控制 | 由 replicas 决定 | 由节点数决定 |
| 调度器 | kube-scheduler | kubelet 直接创建 |
| 驱逐行为 | 可配置 | 通常不驱逐 |
| 资源调度 | 考虑资源可用性 | 尽力而为 |

## 工作负载对比总结

| 类型 | 有状态 | 副本控制 | 滚动更新 | 典型用例 |
|---|---|---|---|---|
| Deployment | 否 | replicas | 支持 | Web 服务、API 服务 |
| ReplicaSet | 否 | replicas | 不直接支持 | 一般不单独使用 |
| StatefulSet | 是 | ordinal | 支持 | 数据库、消息队列 |
| DaemonSet | 否 | 每节点一个 | 支持 | 日志、监控、网络插件 |
| Job | 不适用 | completions | 不适用 | 批处理任务 |
| CronJob | 不适用 | 并发控制 | 不适用 | 定时任务 |

## 何时使用哪种工作负载？

```
应用类型判断流程：

是否有状态？
├── 无状态应用
│   ├── 需要定期运行？→ CronJob
│   ├── 一次性任务？→ Job
│   ├── 需要每个节点运行？→ DaemonSet
│   └── 普通长期运行？→ Deployment
│
└── 有状态应用
    ├── 需要固定网络标识？→ StatefulSet
    └── 普通长期运行？→ StatefulSet 或 Operator
```

:::tip 最佳实践
1. 优先使用 Deployment 管理无状态服务
2. 数据库等有状态应用使用 StatefulSet
3. 基础设施组件使用 DaemonSet
4. 批处理任务使用 Job/CronJob
5. 不要直接创建 Pod，使用工作负载管理
:::
