# Pod 资源管理

<Badge text="Principle" type="warning" /> <Badge text="Resource" type="tip" />

你的 Java 应用在生产环境中时不时出现响应慢的问题，但你查遍了代码也没发现问题。最后你看了 Pod 的资源配置，才发现问题很简单：内存 Limit 设置的是 `512Mi`，但 JVM 的堆内存配置默认会使用物理机的 1/4。

Kubernetes 的资源管理是避免生产事故的重要一环。Request 和 Limit 设置不合理，轻则导致应用 OOM，重则影响整台节点的稳定性。

## Request 与 Limit 的概念

每个容器都可以声明 **Resource Request**（资源请求）和 **Resource Limit**（资源限制）：

| 类型 | 含义 | 调度影响 | 运行影响 |
|---|---|---|---|
| **Request** | 容器需要的最小资源 | Scheduler 据此选择合适的节点 | 保证容器能获得请求的资源 |
| **Limit** | 容器最多能使用的资源 | 无影响 | 超过会被限制或杀掉 |

```yaml
spec:
  containers:
  - name: app
    resources:
      requests:
        memory: "256Mi"
        cpu: "500m"
      limits:
        memory: "512Mi"
        cpu: "1000m"
```

### 为什么需要 Request？

Request 的核心作用是**调度依据**。Scheduler 会检查节点的 `allocatable` 资源，确保调度后不会超负荷：

```
节点 Allocatable：
  cpu: 4 cores
  memory: 8Gi

已分配 Pod Request：
  pod-a: cpu=1, memory=2Gi
  pod-b: cpu=0.5, memory=1Gi
  pod-c: cpu=0.5, memory=1Gi

剩余可分配：
  cpu: 4 - 1 - 0.5 - 0.5 = 2 cores
  memory: 8 - 2 - 1 - 1 = 4Gi

新 Pod Request（cpu=1, memory=2Gi）：
  可以调度 ✓
```

### 为什么需要 Limit？

Limit 的核心作用是**资源隔离**。Limit 确保单个容器不会耗尽整台节点的资源：

```yaml
# 如果没有 Limit
container:
  memory: Request only (e.g., 256Mi)

# 恶意或有 bug 的容器可能尝试使用 8Gi
# 结果：同一节点的其他容器全部 OOM

# 设置 Limit
container:
  resources:
    requests:
      memory: "256Mi"
    limits:
      memory: "512Mi"

# 结果：容器最多使用 512Mi，超出会被 OOM Kill
```

## 资源类型详解

Kubernetes 支持多种资源类型，但最常用的是 CPU 和内存。

### CPU 资源

CPU 在 Kubernetes 中以 **millicores（毫核）** 为单位。1 core = 1000m。

| 值 | 含义 | 说明 |
|---|---|---|
| `0.5` 或 `500m` | 半个 CPU 核心 | - |
| `1` 或 `1000m` | 1 个 CPU 核心 | - |
| `2` 或 `2000m` | 2 个 CPU 核心 | - |
| `100m` | 0.1 个核心 | 适合轻量任务 |

:::tip 小数点陷阱
`0.1` 在 Kubernetes 中会被解析为 `100m`，而不是 `1000m × 0.1 = 100m`。所以 `0.1 = 100m`。
:::

**CPU 的 Limit 行为**：

CPU 是**可压缩资源**。当容器达到 CPU Limit 时，Linux 内核会**限制容器的 CPU 时间片**，但不会杀死容器。容器会出现"变慢"而不是"崩溃"。

### 内存资源

内存以字节为单位，支持多种表示法：

| 值 | 含义 |
|---|---|
| `128Mi` | 128 MiB（推荐写法） |
| `1Gi` | 1 GiB |
| `128M` | 128 MB（不推荐，容易混淆） |
| `134217728` | 134217728 bytes（不推荐） |

**内存的 Limit 行为**：

内存是**不可压缩资源**。当容器尝试使用超过 Limit 的内存时，会触发 **OOMKilled**（Out of Memory Kill）。

```
OOMKilled 的流程：

容器尝试分配内存 > memory.limit
    │
    ▼
内核尝试回收内存（page cache 等）
    │
    ▼
仍然不够
    │
    ▼
内核选择进程杀掉（通常是当前容器）
    │
    ▼
容器退出，状态：OOMKilled
```

### ephemeral-storage

Kubernetes 1.8+ 支持 `ephemeral-storage` 类型，用于管理临时存储：

```yaml
spec:
  containers:
  - name: app
    resources:
      limits:
        ephemeral-storage: "2Gi"  # 限制临时存储
```

`ephemeral-storage` 包括：
- 容器写入的所有文件
- 空目录（emptyDir）
- 日志文件
- 容器层（镜像层是只读的，不计入）

## QoS 类别

当节点资源不足时，Kubernetes 需要决定先驱逐哪些 Pod。这个决策基于 Pod 的 **QoS（Quality of Service）类别**：

| QoS 类别 | 优先级 | 条件 |
|---|---|---|
| **Guaranteed** | 最高 | 每个容器都设置了 CPU 和内存的 Request 和 Limit，且两者相等 |
| **Burstable** | 中等 | 不满足 Guaranteed，但至少有一个容器设置了资源请求 |
| **BestEffort** | 最低 | 没有任何容器设置资源请求或限制 |

### Guaranteed 示例

```yaml
# 条件：每个容器都设置了 Request=Limit
containers:
- name: app
  resources:
    requests:
      cpu: "1"
      memory: "1Gi"
    limits:
      cpu: "1"        # 等于 requests
      memory: "1Gi"   # 等于 requests
```

### Burstable 示例

```yaml
# 条件：至少有一个容器设置了 requests
containers:
- name: app
  resources:
    requests:
      cpu: "500m"
      memory: "256Mi"
    limits:
      cpu: "1000m"
      memory: "512Mi"  # limit != request
```

### BestEffort 示例

```yaml
# 条件：没有任何容器设置 requests 或 limits
containers:
- name: app
  resources: {}  # 完全不设置
```

### 驱逐优先级

```
节点压力驱逐时的优先级：
┌─────────────────────────────────────────────────────────────┐
│                     BestEffort Pods                         │
│         最早驱逐，优先级最低                                  │
└─────────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│                     Burstable Pods                         │
│         优先驱逐使用超过 requests 的 Pod                       │
└─────────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│                     Guaranteed Pods                         │
│         最后驱逐，优先级最高                                   │
└─────────────────────────────────────────────────────────────┘
```

## LimitRange 强制执行

**LimitRange** 是命名空间级别的资源约束，可以在不修改 Pod 配置的情况下强制执行资源限制。

```yaml title="limitrange.yaml"
apiVersion: v1
kind: LimitRange
metadata:
  name: default-limit
spec:
  limits:
  # 默认容器限制
  - type: Container
    default:
      cpu: "500m"
      memory: "256Mi"
    defaultRequest:
      cpu: "100m"
      memory: "64Mi"
    max:
      cpu: "2"
      memory: "1Gi"
    min:
      cpu: "50m"
      memory: "32Mi"
  # Pod 级别的限制
  - type: Pod
    max:
      cpu: "8"
      memory: "8Gi"
  # PVC 的大小限制
  - type: PersistentVolumeClaim
    min:
      storage: 1Gi
    max:
      storage: 10Ti
```

### LimitRange 的效果

创建 Pod 时，如果没有指定资源：

```yaml
# Pod YAML（没有设置 resources）
spec:
  containers:
  - name: app
    image: nginx
```

**LimitRange 应用后**：

```yaml
# 实际生效的配置（由 LimitRange 自动填充）
spec:
  containers:
  - name: app
    resources:
      limits:
        cpu: "500m"      # 来自 default
        memory: "256Mi"   # 来自 default
      requests:
        cpu: "100m"       # 来自 defaultRequest
        memory: "64Mi"    # 来自 defaultRequest
```

:::warning 最佳实践
1. 始终为生产环境设置资源限制
2. Request 应该接近实际使用量，Limit 设置稍高作为突发余量
3. 使用 LimitRange 为命名空间设置默认值，避免遗漏
4. 监控资源使用，调整 Request/Limit 到合理范围
:::

## 资源配额（ResourceQuota）

与 LimitRange 不同，**ResourceQuota** 是对整个命名空间的总资源限制：

```yaml title="resourcequota.yaml"
apiVersion: v1
kind: ResourceQuota
metadata:
  name: quota
spec:
  hard:
    requests.cpu: "20"
    requests.memory: "40Gi"
    limits.cpu: "40"
    limits.memory: "80Gi"
    pods: "50"
    services: "10"
    persistentvolumeclaims: "20"
```

### 与 LimitRange 的配合

```
┌─────────────────────────────────────────────────────────────┐
│                    ResourceQuota                            │
│         命名空间级别的总资源上限                              │
│         "这个命名空间最多用 20 个 CPU"                        │
└─────────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│                    LimitRange                               │
│         单个容器/Pod 的资源范围                              │
│         "每个容器最多用 2 个 CPU"                            │
└─────────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│                    Container Resources                      │
│         具体的 Request 和 Limit                             │
│         "这个容器请求 500m，最多用 1 核"                      │
└─────────────────────────────────────────────────────────────┘
```

## 监控与调优

```bash
# 查看节点资源分配情况
kubectl describe node <node-name> | grep -A 10 "Allocated resources"

# 查看 Pod 的实际资源使用
kubectl top pod

# 查看 Pod 的资源请求 vs 实际使用
kubectl top pod --sort-by=memory
```

**推荐的资源设置流程**：

1. **初始阶段**：不设置限制，记录实际使用量
2. **稳定阶段**：根据监控数据设置 Request = P95 使用量
3. **缓冲设置**：Limit = Request × 1.2 ~ 2.0
4. **持续调优**：根据告警和监控持续调整

## 总结

| 概念 | 作用 | 调度影响 | 运行影响 |
|---|---|---|---|
| Request | 容器需要的最小资源 | ✓（必须满足） | ✓（保证获得） |
| Limit | 容器可用的最大资源 | ✗ | ✓（超出限制触发动作） |
| QoS | Pod 的服务质量等级 | ✗ | ✓（影响驱逐优先级） |
| LimitRange | 命名空间级别的默认值 | ✓ | ✓ |
| ResourceQuota | 命名空间级别的总限制 | ✓ | ✓ |

理解 Request 和 Limit 的区别，以及 QoS 对驱逐优先级的影响，是编写生产级 Kubernetes 配置的基础。
