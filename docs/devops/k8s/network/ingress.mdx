# Ingress 入口控制器

你有 10 个微服务，每个都需要从外部访问。在没有 Ingress 之前，你可能需要创建 10 个 LoadBalancer Service，每个都消耗云服务商的负载均衡器资源，成本高昂。

Ingress 提供了更优雅的解决方案：只需要一个 LoadBalancer，就可以将请求路由到多个后端 Service。同时，它还支持 TLS 终止、基于域名和路径的路由等功能。

## Ingress 架构

Ingress 本身只是一个 API 对象，具体的请求转发由 Ingress Controller 实现：

```
                                    ┌─────────────────────────┐
                                    │   Ingress Controller    │
                                    │   (Nginx/Traefik/...)   │
                                    └───────────┬─────────────┘
                                                │
┌─────────┐        ┌──────────────┐              │
│  用户   │──────▶ │  LoadBalancer│─────────────┤
└─────────┘        └──────────────┘              │
                                               │
                              ┌────────────────┼────────────────┐
                              ↓                ↓                ↓
                    ┌─────────────┐   ┌─────────────┐   ┌─────────────┐
                    │   Service   │   │   Service   │   │   Service   │
                    │ /api        │   │ /web        │   │ /admin      │
                    └─────────────┘   └─────────────┘   └─────────────┘
```

### Ingress Controller 类型

| 控制器 | 特点 | 适用场景 |
|---|---|---|
| Nginx Ingress | 功能丰富、社区活跃 | 通用场景 |
| Traefik | 支持 TCP/UDP、WebSocket | 微服务 |
| Ambassador | 基于 Envoy、Kubernetes 原生 | API Gateway |
| GCE Ingress | Google Cloud 托管 | GCP 环境 |

## 基本 Ingress 配置

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: multi-service-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  ingressClassName: nginx
  rules:
  - host: api.example.com
    http:
      paths:
      - path: /users
        pathType: Prefix
        backend:
          service:
            name: user-service
            port:
              number: 80
      - path: /orders
        pathType: Prefix
        backend:
          service:
            name: order-service
            port:
              number: 80
      - path: /
        pathType: Prefix
        backend:
          service:
            name: default-backend
            port:
              number: 80
```

### pathType 详解

| 类型 | 说明 | 示例 |
|---|---|---|
| `Exact` | 精确匹配 | `/api` 匹配 `/api`，不匹配 `/api/` |
| `Prefix` | 前缀匹配 | `/api` 匹配 `/api`、`/api/users` |
| `ImplementationSpecific` | 取决于控制器实现 | 灵活配置 |

```yaml
# 精确匹配
- path: /api
  pathType: Exact

# 前缀匹配
- path: /api
  pathType: Prefix

# 控制器特定
- path: /api
  pathType: ImplementationSpecific
  backend:
    service:
      name: api-service
      port:
        number: 80
```

## 基于域名的虚拟主机

通过不同的域名，路由到不同的后端 Service：

```yaml
spec:
  rules:
  - host: www.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: website
            port:
              number: 80

  - host: api.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: api-service
            port:
              number: 80

  - host: admin.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: admin-service
            port:
              number: 80
```

```
用户请求                         Ingress Controller
   │                                   │
   ▼                                   ▼
www.example.com ──▶ website Service
api.example.com ──▶ api-service
admin.example.com ──▶ admin-service
```

## TLS 配置

### 准备证书

```bash
# 创建 TLS Secret
kubectl create secret tls example-com-tls \
  --cert=fullchain.pem \
  --key=privkey.pem
```

### 配置 HTTPS

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: tls-ingress
spec:
  ingressClassName: nginx
  tls:
  - hosts:
    - www.example.com
    - api.example.com
    secretName: example-com-tls
  rules:
  - host: www.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: website
            port:
              number: 80
```

:::tip 证书管理
手动管理证书容易出错。在生产环境中，建议使用 cert-manager 自动管理 Let's Encrypt 证书，实现自动续期。
:::

### 外部域名 + TLS + 路径重写

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: api-ingress
  annotations:
    nginx.ingress.kubernetes.io/ssl-redirect: "true"
    nginx.ingress.kubernetes.io/use-regex: "true"
    nginx.ingress.kubernetes.io/rewrite-target: /$2
spec:
  ingressClassName: nginx
  tls:
  - hosts:
    - api.example.com
    secretName: api-tls
  rules:
  - host: api.example.com
    http:
      paths:
      - path: /v1/users(/|$)(.*)
        pathType: ImplementationSpecific
        backend:
          service:
            name: user-service
            port:
              number: 8080
```

## IngressClass

从 Kubernetes 1.19 开始，IngressClass 是一个正式的资源对象：

```yaml
apiVersion: networking.k8s.io/v1
kind: IngressClass
metadata:
  name: nginx
  annotations:
    ingressclass.kubernetes.io/is-default-class: "true"
spec:
  controller: k8s.io/nginx-ingress-controller
  parameters:
    apiGroup: k8s.nginx.org
    kind: NginxIngressPool
    name: global-parameters
```

```yaml
# 在 Ingress 中指定使用哪个 IngressClass
spec:
  ingressClassName: nginx  # 使用名为 nginx 的 IngressClass
```

### 默认 IngressClass

```yaml
metadata:
  annotations:
    ingressclass.kubernetes.io/is-default-class: "true"
```

设置了此注解的 IngressClass 会被新建 Ingress 默认使用（不指定 `ingressClassName` 时）。

## 高级路由

### 基于请求头的路由

```yaml
annotations:
  nginx.ingress.kubernetes.io/canary: "true"
  nginx.ingress.kubernetes.io/canary-by-header: "X-Canary"
  nginx.ingress.kubernetes.io/canary-by-header-value: "always"
```

### 基于权重的金丝雀发布

```yaml
annotations:
  nginx.ingress.kubernetes.io/canary: "true"
  nginx.ingress.kubernetes.io/canary-weight: "10"  # 10% 流量到金丝雀
```

## Ingress vs Service

| 特性 | Ingress | Service |
|---|---|---|
| 层级 | L7 (HTTP/HTTPS) | L4 (TCP/UDP) |
| 外部暴露 | 单入口多路由 | 多类型暴露 |
| 路径路由 | 支持 | 不支持 |
| TLS | 支持 | 需配合其他组件 |
| 域名路由 | 支持 | 不支持 |
| 资源开销 | 低（一个 LB） | 高（多个 LB） |

:::tip 选择建议
- 只有 HTTP/HTTPS 流量：使用 Ingress
- TCP/UDP 流量：使用 NodePort 或 LoadBalancer
- 简单暴露：NodePort
- 生产 HTTPS：Ingress + TLS
:::

## 总结

Ingress 是 Kubernetes 暴露 HTTP/HTTPS 服务的标准方式：

- **统一入口**：一个 LoadBalancer 服务多个应用
- **灵活路由**：基于域名、路径、请求头
- **TLS 支持**：集中管理证书
- **金丝雀发布**：通过注解实现流量控制

选择合适的 Ingress Controller，根据业务需求配置 Ingress 规则，是构建生产级 Kubernetes 网络的关键。
