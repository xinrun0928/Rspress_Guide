# Service 类型详解

Pod 的 IP 是不稳定的——当 Pod 被调度、迁移或重启时，IP 会发生变化。但在微服务架构中，我们需要一种稳定的方式来访问服务。

Service 正是这个抽象层。它为一组 Pod 提供稳定的虚拟 IP（ClusterIP），并通过 label selector 自动追踪后端 Pod 的变化。

Kubernetes 支持四种 Service 类型，每种都有其适用场景。

## ClusterIP（默认）

ClusterIP 是默认的 Service 类型，它在集群内部创建一个虚拟 IP，外部无法直接访问。

```yaml
apiVersion: v1
kind: Service
metadata:
  name: order-service
  namespace: default
spec:
  type: ClusterIP
  selector:
    app: order-service
  ports:
  - name: http
    port: 80
    targetPort: 8080
  - name: grpc
    port: 50051
    targetPort: 50051
```

### 访问方式

```bash
# Pod 内访问（通过环境变量）
curl http://order-service

# Pod 内访问（通过 DNS）
curl http://order-service.default.svc.cluster.local

# 集群内任意位置
kubectl run curl --rm -it --image=curlimages/curl -- \
  curl http://order-service.default.svc.cluster.local
```

### ClusterIP 的局限性

- 只能在集群内部访问
- 外部网络无法直接访问
- 适合微服务间的内部通信

## NodePort

NodePort 在每个节点的 IP 上暴露一个静态端口（30000-32767），外部流量可以通过 `节点IP:NodePort` 访问 Service。

```yaml
apiVersion: v1
kind: Service
metadata:
  name: order-service
spec:
  type: NodePort
  selector:
    app: order-service
  ports:
  - name: http
    port: 80
    targetPort: 8080
    nodePort: 30080  # 可选，不指定则随机分配
```

### 访问方式

```bash
# 通过任意节点 IP 访问
curl http://<node1-ip>:30080
curl http://<node2-ip>:30080

# 即使 Pod 只在 node2 上运行，node1:30080 也能访问
```

### 端口范围

| 范围 | 说明 |
|---|---|
| 30000-32767 | NodePort 有效范围 |
| 默认随机分配 | 不指定时随机选择 |

:::warning NodePort 的局限
- 端口范围受限（30000-32767）
- 每个端口只能有一个 Service
- 节点 IP 暴露给外部网络
- 无法精细控制负载均衡策略
- 适合开发环境或小规模集群，不适合生产
:::

## LoadBalancer

LoadBalancer 将 Service 暴露给外部负载均衡器，通常由云服务商提供支持。

```yaml
apiVersion: v1
kind: Service
metadata:
  name: order-service
spec:
  type: LoadBalancer
  selector:
    app: order-service
  ports:
  - name: http
    port: 80
    targetPort: 8080
  externalTrafficPolicy: Cluster
```

### 云服务商集成

```
用户请求 ──▶ 负载均衡器 ──▶ Service ──▶ Pod
                     │
              云厂商创建
            (AWS ELB/GCE LB/...)
```

| 云平台 | 集成方式 |
|---|---|
| AWS | 通过 Cloud Provider 创建 ELB/NLB |
| GCP | 通过 Cloud Provider 创建 TCP/UDP LB |
| Azure | 通过 Cloud Provider 创建 LB |
| 裸金属 | 需要 MetalLB 等额外组件 |

### externalTrafficPolicy

```yaml
# 集群级别：保留客户端 IP，但流量可能经过多跳
spec:
  externalTrafficPolicy: Cluster  # 默认

# 本地级别：保留客户端 IP，流量不跨节点，但可能不均衡
spec:
  externalTrafficPolicy: Local
```

```
Cluster 模式：
  外部请求 → Node1:30080 → (kube-proxy) → Pod on Node1/2/3

Local 模式：
  外部请求 → Node1:30080 → 直接 → Pod on Node1
```

:::tip Local 模式的选择
如果需要保留客户端 IP，使用 `Local`。如果服务有多个后端 Pod 分布在不同节点，Node1 上的请求只会转发到 Node1 本地的 Pod，可能导致负载不均。
:::

## ExternalName

ExternalName 将 Service 映射到外部域名，返回 CNAME 记录：

```yaml
apiVersion: v1
kind: Service
metadata:
  name: mysql-external
spec:
  type: ExternalName
  externalName: mysql.prod.example.com
```

```bash
# 解析结果
nslookup mysql-external.default.svc.cluster.local
```

```
mysql-external.default.svc.cluster.local  canonical name = mysql.prod.example.com
```

### 典型用途

- 集群内引用外部数据库
- 平滑迁移：先通过 ExternalName 指向旧服务，迁移后改为普通 Service
- 统一内部域名管理

:::warning 注意事项
ExternalName 不创建任何端点（Endpoints），kube-proxy 也不会处理它。DNS 层面的 CNAME 是唯一的处理方式。
:::

## Headless Service

Headless Service 是将 `clusterIP` 设为 `None` 的 Service：

```yaml
apiVersion: v1
kind: Service
metadata:
  name: mysql
spec:
  clusterIP: None  # Headless
  selector:
    app: mysql
  ports:
  - port: 3306
```

### DNS 行为

普通 Service 返回 ClusterIP，Headless Service 直接返回 Pod IP：

| 类型 | DNS 返回 |
|---|---|
| ClusterIP | `mysql.default.svc.cluster.local -> 10.96.45.123` |
| Headless | `mysql.default.svc.cluster.local -> [10.244.1.5, 10.244.2.8, 10.244.3.2]` |

### 适用场景

- 有状态应用：数据库主从，需要直接连接特定 Pod
- 客户端自主选择：客户端自己决定连接哪个 Pod
- 服务发现集成：应用需要知道所有后端 Pod 的 IP

```java
// 客户端获取所有 Pod IP
String serviceName = "mysql.default.svc.cluster.local";
Lookup lookup = new Lookup(serviceName, Type.SRV);
Record[] records = lookup.lookupService();
```

## 服务类型对比

| 类型 | 访问范围 | 外部流量 | 负载均衡 | 适用场景 |
|---|---|---|---|---|
| ClusterIP | 集群内 | 不支持 | kube-proxy | 微服务间通信 |
| NodePort | 集群内 + 节点 | `NodeIP:Port` | 节点级别 | 开发测试、简单暴露 |
| LoadBalancer | 全球 | 负载均衡器 | 云服务商 | 生产环境 |
| ExternalName | 集群内 | 不支持 | CNAME | 外部服务引用 |
| Headless | 集群内 | 不支持 | 客户端选择 | 有状态应用 |

## 端口配置详解

```yaml
spec:
  ports:
  - name: http
    port: 80              # Service 监听的端口
    targetPort: 8080      # 后端 Pod 监听的端口
    protocol: TCP         # 协议：TCP/UDP/SCTP
```

### 多端口 Service

```yaml
apiVersion: v1
kind: Service
metadata:
  name: multi-port-service
  annotations:
    service.kubernetes.io/topology-mode: "Auto"
spec:
  ports:
  - name: http
    port: 80
    targetPort: 8080
  - name: grpc
    port: 50051
    targetPort: 50051
  selector:
    app: order-service
```

### targetPort 的多种写法

```yaml
# 数字端口
targetPort: 8080

# 字符串：Pod 容器中定义的端口名
targetPort: http

# 容器端口（与端口名同义）
targetPort: { containerPort: 8080 }
```

## 总结

Service 是 Kubernetes 网络的核心抽象：

- **ClusterIP**：集群内部访问的标准方式
- **NodePort**：简单暴露到外部，适合开发测试
- **LoadBalancer**：生产环境的标准外部暴露方式
- **ExternalName**：引用集群外部的服务
- **Headless**：绕过 kube-proxy，客户端直接连接 Pod

选择正确的 Service 类型，是设计 Kubernetes 网络架构的关键。
