# Kubernetes 网络模型

你有没有想过，Pod A 怎么知道 Pod B 在哪里？Service 是如何将请求分发到后端 Pod 的？外部流量怎么进入集群？

要理解 Kubernetes 的网络，首先需要理解它的网络模型。这是一个「反常识」的模型——它和传统虚拟机的网络模型完全不同。

## 核心原则

Kubernetes 网络模型建立在三个基本原则之上：

| 原则 | 含义 |
|---|---|
| 每个 Pod 有唯一 IP | Pod 被视为独立的虚拟机 |
| Pod 之间通信无需 NAT | 直接通信，像在同一个网络 |
| 节点与 Pod 通信无需 NAT | 节点可以像访问本地网络一样访问 Pod |

这三个原则保证了网络的可预测性和透明性，让应用无需关心底层的网络拓扑。

## IP-per-Pod 模型

在 Kubernetes 中，每个 Pod 都有自己独立的 IP 地址。这个 IP 是 **Pod 级别的**，不是容器级别的。

```bash
kubectl get pods -o wide
```

```
NAME                READY   STATUS    RESTARTS   AGE   IP           NODE
order-service-abc   1/1     Running   0          1d    10.244.1.5   node-1
inventory-service   1/1     Running   0          2d    10.244.2.8   node-1
payment-service     1/1     Running   0          3d    10.244.3.2   node-2
```

每个 Pod 都有独立的 IP，不同 Pod 之间可以直接通过 IP 通信。

### Pod 内容器共享网络

一个 Pod 可以包含多个容器，它们共享同一个网络命名空间：

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: web-app
spec:
  containers:
  - name: nginx
    image: nginx:1.25
    ports:
    - containerPort: 80
  - name: sidecar
    image: sidecar:v1
    ports:
    - containerPort: 8080
```

在这个 Pod 中，nginx 和 sidecar 共享同一个网络栈：

- localhost 可以互相访问
- 共享相同的 IP 地址（10.244.1.5）
- 共享相同的端口空间（80 和 8080 不能冲突）

:::tip Sidecar 模式
这种共享网络的设计使得 Sidecar 模式变得简单。Sidecar 可以拦截主容器的流量，实现日志收集、监控、灰度发布等功能，而无需修改主应用代码。
:::

## 网络命名空间

Linux 使用网络命名空间（Network Namespace）实现容器间的网络隔离。

```
┌─────────────────────────────────────────────────────┐
│                    Node                             │
│  ┌─────────────┐    ┌─────────────┐                │
│  │  Pod A      │    │  Pod B      │                │
│  │  eth0 ────────────│ eth0       │                │
│  │  10.244.1.5 │    │  10.244.2.8 │                │
│  └─────────────┘    └─────────────┘                │
│         ↑                 ↑                          │
│         └────────┬────────┘                          │
│                  ↓                                    │
│           ┌─────────────┐                           │
│           │  Root Netns │                           │
│           │   (Node IP) │                           │
│           └─────────────┘                           │
└─────────────────────────────────────────────────────┘
```

### 容器网络创建流程

1. kubelet 为 Pod 创建网络命名空间
2. CNI 插件为 Pod 分配 IP（通常从节点 CIDR 中分配）
3. 创建 veth pair（一端在 Pod 内，一端在根命名空间）
4. 将根命名空间端连接到网桥或路由
5. 填写 Pod 内的网络配置（IP、路由等）

## CNI 插件的角色

CNI（Container Network Interface）是 Kubernetes 网络的核心抽象。它定义了容器网络配置的标准接口。

```yaml
# kubelet 配置使用 CNI
kubelet --network-plugin=cni --cni-conf-dir=/etc/cni/net.d
```

### CNI 插件职责

CNI 插件负责：

- **IP 分配**：从预配置的 CIDR 范围为 Pod 分配 IP
- **网络配置**：设置网桥、路由、VLAN、VXLAN 等
- **网络连通**：确保 Pod 可以相互通信、访问外网
- **网络隔离**：实现 NetworkPolicy 定义的隔离规则

主流 CNI 插件有 Flannel、Calico、Cilium 等，它们各有特点：

| 插件 | 特点 | 网络模型 |
|---|---|---|
| Flannel | 简单易用 | VXLAN 覆盖网络 |
| Calico | 高性能、策略丰富 | BGP 直连或 VXLAN |
| Cilium | eBPF、L7 策略 | 直接路由或 VXLAN |
| Weave | 简单、加密 | 覆盖网络 |

## 通信路径

### 同一节点内 Pod 通信

```
┌──────────────────────────────────┐
│           Node                    │
│  ┌─────────┐     ┌─────────┐     │
│  │  Pod A  │     │  Pod B  │     │
│  │ 10.244.1.5   10.244.1.6  │
│  │   eth0 │─────│   eth0 │     │
│  └────┬────┘     └────┬────┘     │
│       │               │          │
│       └───────┬───────┘          │
│               ↓                   │
│         ┌──────────┐              │
│         │  cni0    │   (网桥)     │
│         └──────────┘              │
└──────────────────────────────────┘
```

Pod A 访问 Pod B 时：

1. 数据包通过 veth pair 到达网桥
2. 网桥根据 MAC 地址转发到 Pod B 的 veth
3. Pod B 收到数据包

### 跨节点 Pod 通信

```
                    ┌────────────────────────────────────────┐
                    │                Internet                 │
                    └────────────────────────────────────────┘
                                       ↑
                    ┌─────────────────┴──────────────────┐
                    │              Node 1                  │
                    │  ┌─────────┐                        │
                    │  │  Pod A  │                         │
                    │  │ 10.244.1.5                        │
                    │  │   eth0 │                         │
                    │  └────┬────┘                         │
                    │       │ (veth)                        │
                    │  ┌────▼────┐                          │
                    │  │  cni0   │                          │
                    │  └───┬────┘                          │
                    └───────┼──────────────────────────────┘
                            │ (VXLAN)
                    ┌───────▼──────────────────────────────┐
                    │              Node 2                  │
                    │  ┌────┬────┐                          │
                    │  │  cni0   │                          │
                    │  └┬───┴────┘                          │
                    │   │ (veth)                            │
                    │  ┌▼────────┐                           │
                    │  │  Pod B  │                           │
                    │  │ 10.244.2.8                          │
                    │  └─────────┘                           │
                    └───────────────────────────────────────┘
```

Pod A（Node 1）访问 Pod B（Node 2）时：

1. Pod A 发送数据包，目标 IP 是 10.244.2.8
2. 本地网桥找不到目标，发送到网关（通常是 CNI 插件的 overlay 接口）
3. CNI 封装数据包（VXLAN 或其他隧道协议）
4. 传输到 Node 2
5. Node 2 解封装，将数据包交给 Pod B

## 特殊网络模式

### hostNetwork

Pod 使用宿主机的网络栈：

```yaml
spec:
  hostNetwork: true
  containers:
  - name: nginx
    image: nginx
    ports:
    - containerPort: 80
      hostPort: 80  # 必须指定 hostPort
```

**特点**：

- Pod 的 IP 就是宿主机的 IP
- 端口直接绑定到宿主机
- 无法在节点上运行多个使用相同端口的 Pod
- 绕过了 Kubernetes 的 Service 负载均衡

:::warning 谨慎使用
`hostNetwork` 绕过了网络命名空间的隔离，存在安全风险。只有必须使用宿主机网络的应用（如网络插件）才应该使用。
:::

### hostPID

类似地，hostPID 让 Pod 共享宿主机的 PID 命名空间，可以看到宿主机上的进程。

## 服务发现的基础

IP-per-Pod 模型的一个问题是：Pod 的 IP 是不稳定的。当 Pod 被重建时，IP 会变化。

这正是 Service 存在的意义。Service 为一组 Pod 提供稳定的虚拟 IP（ClusterIP），Pod 可以通过 Service 的域名访问后端 Pod：

```yaml
apiVersion: v1
kind: Service
metadata:
  name: order-service
spec:
  selector:
    app: order-service
  ports:
  - port: 80
    targetPort: 8080
```

访问 `http://order-service.default.svc.cluster.local` 会自动解析到 Service 的 ClusterIP，然后被 kube-proxy 分发到具体的 Pod。

## 总结

Kubernetes 网络模型的核心是 **IP-per-Pod**：

- 每个 Pod 有唯一、稳定的 IP（在 Pod 生命周期内）
- Pod 之间通信不需要 NAT
- CNI 插件负责具体的网络实现
- Service 在 Pod IP 之上提供了稳定的访问入口

理解这个模型，是掌握 Kubernetes 网络的基础。
