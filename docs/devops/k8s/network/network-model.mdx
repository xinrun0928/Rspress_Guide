# Kubernetes 网络模型

你有没有想过——Kubernetes 中的 Pod 是怎么互相通信的？Pod 有自己的 IP，Service 也有 IP，但 Service 的 IP 根本不存在于网络中。那流量是怎么到达 Pod 的？

要理解 Kubernetes 网络，首先要理解它的三大网络平面：**Pod 网络**、**Service 网络**和**节点网络**。

## Kubernetes 网络基础

### 三个网络平面

```
┌────────────────────────────────────────────────────────────────────┐
│                         Kubernetes Cluster                          │
│                                                                     │
│  ┌──────────────────────────────────────────────────────────────┐  │
│  │                     Node Network (节点网络)                    │  │
│  │                      例如：192.168.1.0/24                     │  │
│  │                                                                     │
│  │  ┌──────────────────────────────────────────────────────┐   │  │
│  │  │                Pod Network (Pod 网络)                  │   │  │
│  │  │               例如：10.244.0.0/16 (CNI 分配)           │   │  │
│  │  │                                                              │   │  │
│  │  │  ┌────────────┐  ┌────────────┐  ┌────────────┐        │   │  │
│  │  │  │  Pod-1     │  │  Pod-2     │  │  Pod-3     │        │   │  │
│  │  │  │10.244.1.2  │  │10.244.2.3  │  │10.244.1.5  │        │   │  │
│  │  │  │            │◄─┤            │  │            │        │   │  │
│  │  │  │(通过 veth  │  │            │  │            │        │   │  │
│  │  │  │  连接)      │  │            │  │            │        │   │  │
│  │  │  └────────────┘  └────────────┘  └────────────┘        │   │  │
│  │  │                                                              │   │  │
│  │  └──────────────────────────────────────────────────────────┘   │  │
│  │                                                                     │
│  └─────────────────────────────────────────────────────────────────┘  │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

| 网络类型 | 地址范围 | 说明 |
|---|---|---|
| **Node Network** | 物理网络/VLAN | 节点之间通信 |
| **Pod Network** | CNI 分配的扁平网络 | Pod 之间直接通信 |
| **Service Network** | kube-proxy 管理 | 虚拟 IP，负载均衡 |

### 核心原则

Kubernetes 网络模型遵循以下原则：

1. **Pod 可以直接通信**：每个 Pod 有唯一的 IP，Pod 之间不需要 NAT
2. **节点可以与 Pod 通信**：节点上的进程可以与任意 Pod 通信
3. **Pod 视角与节点视角一致**：Pod 看到的网络与节点看到的网络一致

:::warning 重要：Service IP 不存在于网络中
Service 的 ClusterIP 是虚拟 IP，永远不会分配给任何网络接口。它只是 kube-proxy 在 iptables/ipvs 中维护的一个规则。发往 Service IP 的数据包会被 kube-proxy 拦截并转发到后端 Pod。
:::

## Pod 网络

### Pod 如何获取 IP

每个 Pod 启动时，kubelet 会调用 CNI 插件为 Pod 分配网络命名空间和 IP 地址。

```bash
# 查看 Pod 的网络信息
kubectl exec -it nginx -- ip addr

# 输出示例：
# 1: lo: <LOOPBACK,UP,LOWER_UP> mtu 65536 qdisc noqueue
#     inet 127.0.0.1/8 scope host lo
# 2: eth0@if5: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 qdisc noqueue
#     inet 10.244.1.2/24 scope global eth0
#     valid_lft forever preferred_lft forever
```

### veth pair：Pod 与宿主机的连接

每个 Pod 通过 **veth pair**（虚拟以太网对）与宿主机连接。

```
┌─────────────────────────────────────────────────────────────────┐
│                        Node                                     │
│                                                                 │
│  ┌─────────────────┐         ┌─────────────────┐               │
│  │    eth0         │         │   docker0       │               │
│  │   (物理网卡)     │         │  (网桥 :172.17) │               │
│  │  192.168.1.10   │         └────────┬────────┘               │
│  └────────┬────────┘                  │                        │
│           │                            │                        │
│           │              ┌─────────────┴────────────┐          │
│           │              │          veth-xxx         │          │
│           │              │         (另一端)          │          │
│           │              └─────────────┬────────────┘          │
│           │                            │                        │
│           │                            ▼                        │
│           │              ┌────────────────────────────┐          │
│           └──────────────┤       eth0@ifN           │          │
│                          │    (Pod 内核网络命名空间)   │          │
│                          └────────────────────────────┘          │
│                                    Pod                            │
└─────────────────────────────────────────────────────────────────┘
```

### 同一节点 Pod 通信

```
Pod-A (10.244.1.2)  ──►  veth pair  ──►  docker0  ──►  veth pair  ──►  Pod-B (10.244.1.3)
    eth0@if5                    网桥                    eth0@if6
```

数据包从 Pod-A 的 eth0 发出，通过 veth pair 到达宿主机的网桥，网桥查询 MAC 地址表，转发到 Pod-B 对应的 veth pair，最终到达 Pod-B。

## Service 网络

### Service 是如何工作的

```
Client (Pod)                    Service                    Backend Pods
     │                             │                             │
     │   发往 10.96.0.1:80        │                             │
     │ ──────────────────────────► │                             │
     │                             │                             │
     │   kube-proxy 拦截           │                             │
     │ ◄────────────────────────── │                             │
     │                             │                             │
     │                             │   负载均衡选择一个 Pod       │
     │                             │ ──────────────────────────► │
     │                             │     转发到 10.244.2.3:8080 │
     │                             │                             │
```

### kube-proxy 的工作模式

**iptables 模式**（默认）：

```bash
# 查看 iptables 规则
iptables -t nat -L KUBE-SERVICES -n | grep my-service

# 每个 Service 对应一条 iptables 链
# 命中规则后，kube-proxy 随机选择后端 Pod
```

```
Service (10.96.0.1:80)
        │
        ▼
┌─────────────────────────────┐
│     iptables 规则链          │
│                              │
│ KUBE-SVC-XXXXXXXX  ────────┼──► Pod-1 (随机选择)
│                              │
│                              ├──► Pod-2 (随机选择)
│                              │
│                              └──► Pod-3 (随机选择)
└─────────────────────────────┘
```

**ipvs 模式**（推荐用于大规模集群）：

```bash
# 查看 ipvs 规则
ipvsadm -L -n | grep ClusterIP

# ipvs 使用 L4 负载均衡，性能更好
# 支持多种算法：rr, wrr, lc, wlc, sh, sed, nq
```

### Service 类型与网络

| Service 类型 | ClusterIP | NodePort | LoadBalancer | ExternalName |
|---|---|---|---|---|
| **ClusterIP** | 自动分配 | 不暴露 | 不暴露 | 不适用 |
| **NodePort** | 自动分配 | `30000-32767` | 不适用 | 不适用 |
| **LoadBalancer** | 自动分配 | 可选开启 | 云厂商创建 | 不适用 |
| **ExternalName** | 无 | 不暴露 | 不暴露 | CNAME |

## DNS 与服务发现

### CoreDNS：集群 DNS

Kubernetes 使用 CoreDNS 作为集群内部的 DNS 服务器。

```bash
# 查看 CoreDNS Pod
kubectl get pods -n kube-system -l k8s-app=kube-dns

# CoreDNS 默认监听 53 端口
```

### 服务发现的流程

```
应用容器内部
      │
      │  请求 my-service.default.svc.cluster.local
      ▼
/etc/resolv.conf
  nameserver 10.96.0.10     ← CoreDNS Service IP
  search default.svc.cluster.local
  search svc.cluster.local
  search cluster.local
      │
      ▼
CoreDNS (10.96.0.10)
      │
      ├──► 查找 Service 记录
      │      │
      │      ▼
      │   返回 Service IP (ClusterIP)
      │
      └──► 查找 Pod 记录
             │
             ▼
          返回 Pod IP
```

### DNS 命名规范

```
my-svc.my-namespace.svc.cluster.local
│      │         │    │       │
│      │         │    │       └── 集群本地后缀
│      │         │    └── 命名空间
│      │         └── 服务名
│      └── 服务名（可简写）
└── 完整域名
```

## Ingress：集群入口

Ingress 是 Kubernetes 暴露 HTTP/HTTPS 服务的标准方式。

```
外部流量
    │
    ▼
┌─────────────────┐
│  Ingress        │ ← 七层负载均衡 + 路由
│  Controller     │
└────────┬────────┘
         │
    ┌────┴────┐
    ▼         ▼
┌────────┐ ┌────────┐
│Service A │ │Service B │
│ /api   │ │ /web   │
└────────┘ └────────┘
```

### Ingress 示例

```yaml title="ingress.yaml"
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: myapp-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  ingressClassName: nginx
  rules:
  - host: myapp.example.com
    http:
      paths:
      - path: /api
        pathType: Prefix
        backend:
          service:
            name: api-service
            port:
              number: 80
      - path: /web
        pathType: Prefix
        backend:
          service:
            name: web-service
            port:
              number: 80
  tls:
  - hosts:
    - myapp.example.com
    secretName: myapp-tls
```

## 跨节点通信

### 跨节点 Pod 通信

```
Node-A (192.168.1.10)                              Node-B (192.168.1.20)
┌─────────────────────────────────┐              ┌─────────────────────────────────┐
│                                 │              │                                 │
│  ┌─────────┐                    │              │  ┌─────────┐                    │
│  │  Pod-A  │                    │              │  │  Pod-B  │                    │
│  │10.244.1.2│                   │              │  │10.244.2.3│                   │
│  └────┬────┘                    │              │  └────┬────┘                    │
│       │                         │              │       │                         │
│       │ eth0 (veth)             │              │       │ eth0 (veth)              │
│       ▼                         │              │       ▼                         │
│  ┌─────────────┐                │              │  ┌─────────────┐                │
│  │  cni0 网桥  │                │              │  │  cni0 网桥  │                │
│  └──────┬──────┘                │              │  └──────┬──────┘                │
│         │                        │              │         │                        │
│         │ eth0 (物理网卡)         │              │         │ eth0 (物理网卡)         │
└─────────┼────────────────────────┼──────────────┼─────────┼────────────────────────┘
          │                        │              │         │
          └────────────────────────┴──────────────┴─────────┘
                      物理网络 (192.168.1.0/24)
```

跨节点通信的关键在于：**CNI 插件负责维护路由规则**，确保发往其他节点 Pod 网段的流量被正确路由。

### CNI 的作用

不同的 CNI 插件处理跨节点通信的方式不同：

- **Flannel**：通过 etcd 维护路由表，使用 UDP/VXLAN 封装
- **Calico**：通过 BGP 协议分发路由表，支持纯三层网络
- **Cilium**：基于 eBPF，直接在内核层处理网络包

---

## 总结

Kubernetes 网络的核心概念：

1. **每个 Pod 有唯一 IP**：Pod 之间可以直接通信，不需要 NAT
2. **Service 是虚拟 IP**：流量通过 kube-proxy 转发
3. **DNS 服务发现**：通过 CoreDNS 解析服务名
4. **CNI 负责网络连通**：Flannel、Calico、Cilium 等插件实现网络模型
5. **Ingress 是外部入口**：七层负载均衡和路由

理解这些基础概念，才能更好地理解 Kubernetes 网络的高级特性。
