# Kubernetes DNS 与服务发现

你可能已经习惯在浏览器里输入 `www.example.com` 访问网站。在 Kubernetes 中，Pod 访问 Service 也是类似的——只不过域名变成了 `order-service.default.svc.cluster.local`。

这个复杂的域名背后，是 CoreDNS 在默默工作。它将人类可读的名称转换为 Pod 或 Service 的 IP 地址，让服务间的通信变得简单可靠。

## CoreDNS 架构

CoreDNS 是 Kubernetes 1.13 之后的默认 DNS 服务器，取代了之前的 kube-dns。

```
┌─────────────────────────────────────────────────────────┐
│                     CoreDNS Pod                          │
│  ┌─────────────────────────────────────────────────────┐ │
│  │                   CoreDNS Process                   │ │
│  │  ┌─────────┐  ┌─────────┐  ┌─────────┐              │ │
│  │  │  cache  │  │  hosts  │  │kubernetes│  ← Plugins  │ │
│  │  └─────────┘  └─────────┘  └─────────┘              │ │
│  │  ┌─────────┐  ┌─────────┐  ┌─────────┐              │ │
│  │  │  loop   │  │ forward │  │  proxy  │              │ │
│  │  └─────────┘  └─────────┘  └─────────┘              │ │
│  └─────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────┘
```

### 插件链机制

CoreDNS 使用插件链机制处理 DNS 查询：

```yaml
# Corefile 配置
apiVersion: v1
kind: ConfigMap
metadata:
  name: coredns
  namespace: kube-system
data:
  Corefile: |
    .:53 {
        kubernetes cluster.local in-addr.arpa ip6.arpa {
            pods verified
            fallthrough in-addr.arpa ip6.arpa
        }
        forward . 8.8.8.8
        cache 30
        loop
        reload
        loadbalance
    }
```

每个插件负责特定功能：

| 插件 | 功能 |
|---|---|
| `kubernetes` | 处理集群内 Service 和 Pod 的 DNS 查询 |
| `forward` | 将无法处理的查询转发到上游 DNS |
| `cache` | 缓存查询结果，减少重复查询 |
| `loop` | 检测 DNS 循环，防止无限递归 |
| `reload` | 自动重新加载配置文件 |

## DNS 记录格式

### Service DNS

标准的 Kubernetes Service 会创建两种 DNS 记录：

```
# A 记录：Service IP
order-service.default.svc.cluster.local -> 10.96.45.123

# SRV 记录：Service Port
_http._tcp.order-service.default.svc.cluster.local -> 10.96.45.123:80
```

| 格式 | 适用场景 |
|---|---|
| `svc-name.ns.svc.cluster.local` | 同集群内访问，推荐 |
| `svc-name.ns` | 同命名空间内访问 |
| `svc-name` | Pod 内部访问同命名空间 Service |

### Headless Service DNS

Headless Service（`clusterIP: None`）不分配 ClusterIP，DNS 解析会直接返回 Pod 的 IP：

```yaml
apiVersion: v1
kind: Service
metadata:
  name: mysql
spec:
  clusterIP: None  # Headless Service
  selector:
    app: mysql
  ports:
  - port: 3306
```

DNS 解析结果：

```
# 返回所有 Pod IP（按 SRV 记录）
_mysql._tcp.mysql.default.svc.cluster.local -> [10.244.1.5:3306, 10.244.2.8:3306, 10.244.3.2:3306]

# 按 Pod 名称解析
mysql-0.mysql.default.svc.cluster.local -> 10.244.1.5
mysql-1.mysql.default.svc.cluster.local -> 10.244.2.8
mysql-2.mysql.default.svc.cluster.local -> 10.244.3.2
```

### Pod DNS

Pod 有特殊的 DNS 格式：

```
# Pod IP 的点号替换为连字符
# 10-244-1-5.default.pod.cluster.local
10-244-1-5.default.pod.cluster.local -> 10.244.1.5
```

```yaml
spec:
  subdomain: "mysql"      # Pod 所属的 subdomain
  hostname: "mysql-0"     # Pod 的 hostname
```

带有 hostname 和 subdomain 的 Pod 会获得额外 DNS 记录：

```
# Pod 完整的 FQDN
mysql-0.mysql.default.svc.cluster.local -> 10.244.1.5
```

:::tip Pod DNS 默认关闭
Pod DNS 记录默认是禁用的，需要在 kube-apiserver 中启用 `-- Pujods-verification-mode=true`。这是因为 Pod IP 直接暴露在 DNS 中可能带来安全风险。
:::

## DNS 查找流程

当 Pod 查询 `order-service` 时，DNS 查找流程如下：

```
┌─────────┐     ┌─────────┐     ┌─────────┐
│   Pod   │────▶│ kubelet │────▶│ CoreDNS │
│resolv.conf   │  stub   │     │         │
└─────────┘     └─────────┘     └────┬────┘
                                     │
                    ┌────────────────┼────────────────┐
                    ↓                ↓                ↓
            ┌───────────┐    ┌───────────┐   ┌───────────┐
            │ ClusterIP │    │ClusterDNS │   │ Upstream  │
            │ Service   │    │  (Pod)    │   │   DNS     │
            └───────────┘    └───────────┘   └───────────┘
```

kubelet 为每个 Pod 生成 `/etc/resolv.conf`：

```bash
cat /etc/resolv.conf
```

```
nameserver 10.96.0.10       # CoreDNS Service IP
search default.svc.cluster.local svc.cluster.local cluster.local
options ndots:5
```

### ndots 选项

`ndots` 是一个关键配置，它控制何时使用搜索路径：

| ndots 值 | 行为 |
|---|---|
| `0` 或 `1` | 所有查询直接发送，不添加搜索路径 |
| `5`（默认） | 少于 5 个点的查询先添加搜索路径 |

```
# ndots:5 时查询 "order-service"
1. order-service.default.svc.cluster.local  (失败)
2. order-service.svc.cluster.local           (失败)
3. order-service.cluster.local               (成功!)
4. 缓存结果并返回
```

:::warning ndots 性能问题
对于同命名空间内的频繁调用，过高的 ndots 值会导致多次 DNS 查询才能命中。建议使用完整的 FQDN，或者创建同命名空间的 Service 时直接使用短名称。
:::

## DNS 策略

Kubernetes 支持多种 DNS 策略：

### Default

Pod 使用宿主机的 DNS 配置：

```yaml
spec:
  dnsPolicy: Default
```

Pod 的 `/etc/resolv.conf` 直接继承宿主机的配置。

### ClusterFirst

默认策略，使用集群内的 CoreDNS：

```yaml
spec:
  dnsPolicy: ClusterFirst
```

集群内无法解析的域名会被转发到上游 DNS。

### ClusterFirstWithHostNet

对于使用 hostNetwork 的 Pod，需要显式指定：

```yaml
spec:
  hostNetwork: true
  dnsPolicy: ClusterFirstWithHostNet
```

### None

完全自定义 DNS 配置：

```yaml
spec:
  dnsPolicy: None
  dnsConfig:
    nameservers:
    - 8.8.8.8
    - 8.8.4.4
    searches:
    - myns.svc.cluster.local
    - svc.cluster.local
    options:
    - name: ndots
      value: "2"
```

## DNS 缓存

### 客户端缓存

每个 Pod 内的 DNS 缓存由系统或应用程序控制。Linux 默认使用 `systemd-resolved`，会缓存 DNS 记录。

### CoreDNS 缓存

CoreDNS 的 `cache` 插件提供集群级别的缓存：

```yaml
# Corefile
cache 30  # 缓存 30 秒（正向查询）
```

:::warning 缓存与服务发现延迟
CoreDNS 的缓存会延迟服务发现。如果 Service 的后端 Pod 发生变化（IP、新增、删除），客户端可能需要等待缓存过期才能感知。生产环境中，建议将缓存 TTL 设为较小的值（如 30 秒），或者应用程序实现重试机制。
:::

## 常见问题排查

### DNS 解析失败

```bash
# 在 Pod 内测试 DNS
kubectl exec -it test-pod -- nslookup order-service

# 解析 Service
nslookup order-service.default.svc.cluster.local

# 解析 Pod
nslookup 10-244-1-5.default.pod.cluster.local

# 测试外部域名
nslookup www.google.com
```

### CoreDNS 日志

```bash
kubectl logs -n kube-system -l k8s-app=kube-dns
```

### DNS 延迟测试

```bash
kubectl exec -it test-pod -- sh -c 'time nslookup order-service'
```

## 总结

Kubernetes DNS 是服务发现的基础设施：

- **CoreDNS**：集群内的 DNS 服务器
- **FQDN**：完整的域名格式，如 `svc.ns.svc.cluster.local`
- **DNS 策略**：控制 Pod 使用哪种 DNS 配置
- **ndots**：控制搜索路径的使用策略

理解 DNS 的工作原理和配置选项，能够帮助你解决服务发现相关的网络问题。
