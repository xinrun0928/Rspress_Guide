# kube-proxy 与服务负载均衡

Service 是 Kubernetes 访问抽象的核心，但真正将请求分发到后端 Pod 的，是 kube-proxy。它运行在每个节点上，监听 Service 和 Endpoint 的变化，动态更新本地的代理规则。

理解 kube-proxy 的工作原理，是排查 Service 网络问题的基础。

## kube-proxy 三种模式

kube-proxy 支持三种代理模式，从早期到现代演进：

| 模式 | 特点 | 适用场景 |
|---|---|---|
| userspace | 用户态代理，性能差 | 旧版本兼容 |
| iptables | 规则匹配，最常用 | 通用场景 |
| IPVS | 哈希表，性能高 | 大规模集群 |

## userspace 模式（已废弃）

最早的 kube-proxy 运行在用户空间，通过一个代理进程转发流量：

```
Pod → kube-proxy → Service → Pod
     (userspace)  (Round Robin)
```

```bash
# 启用 userspace 模式（已废弃，不推荐）
kube-proxy --proxy-mode=userspace
```

问题：所有流量都经过用户态代理，多一次 socket 转发，延迟高、吞吐量低。

## iptables 模式

### 原理

iptables 模式利用 Linux 内核的 netfilter 框架，在 PREROUTING 和 OUTPUT 链中拦截流量：

```
┌─────────────────────────────────────────────────────────┐
│                  Linux netfilter                        │
│                                                          │
│  PREROUTING ──▶ INPUT ──▶ ┌─────────────────────────┐ │
│                          │  mangle/filter           │ │
│                          │  - KUBE-SERVICES         │ │
│                          │  - KUBE-NODEPORTS        │ │
│                          │  - KUBE-SEP-XXXX         │ │
│                          └─────────────────────────┘ │
│                                    │                    │
│  OUTPUT ──▶ POSTROUTING ◀─────────┘                    │
│                                                              │
└─────────────────────────────────────────────────────────┘
```

### Service 规则结构

```bash
# 查看 Service 相关的 iptables 规则
iptables -L -t nat -n | grep KUBE
```

```
KUBE-SERVICES  all  --  anywhere  anywhere  /* Kubernetes service portals */
KUBE-NODEPORTS  all  --  anywhere  anywhere  /* Kubernetes nodeport portals */
KUBE-SEP-XXXX   all  --  anywhere  anywhere  /* pod */
KUBE-MARK-MASQ  all  --  anywhere  anywhere  /* Mark-server */
```

### DNAT 转换流程

当访问 Service IP（10.96.45.123:80）时：

```bash
# 1. KUBE-SERVICES 链匹配
-A KUBE-SERVICES -d 10.96.45.123/32 -p tcp -m comment --comment "ns/svc:tcp-port" -j KUBE-SVC-XXXX

# 2. KUBE-SVC-XXXX 链：随机选择后端
-A KUBE-SVC-XXXX -m statistic --mode random --probability 0.33 -j KUBE-SEP-AAAA
-A KUBE-SVC-XXXX -m statistic --mode random --probability 0.50 -j KUBE-SEP-BBBB
-A KUBE-SVC-XXXX -j KUBE-SEP-CCCC

# 3. KUBE-SEP-AAAA 链：DNAT 到具体 Pod
-A KUBE-SEP-AAAA -s 10.244.1.5/32 -j ACCEPT
-A KUBE-SEP-AAAA -j DNAT --to-destination 10.244.1.5:8080
```

### 问题：O(n) 复杂度

iptables 的规则是线性遍历的。当集群有大量 Service 时：

```
100 Services × 3 Pods = 300 条规则
1000 Services × 10 Pods = 10000 条规则
```

每次查找都是 O(n)，在大规模集群中会成为性能瓶颈。

:::warning 大规模集群问题
超过 5000 个 Service 时，iptables 模式的 kube-proxy 可能出现显著延迟。建议超过此规模时切换到 IPVS 模式。
:::

## IPVS 模式

### 原理

IPVS（IP Virtual Server）是 Linux 内核的 L4 负载均衡模块，使用哈希表实现 O(1) 查找：

```
┌─────────────────────────────────────────────────────────┐
│                  IPVS (LVS)                            │
│                                                          │
│   Service Table (Hash)  ──▶  Real Server (Pod)         │
│   ┌─────────────────┐                                  │
│   │ 10.96.45.123:80 |──▶ [10.244.1.5:8080]             │
│   │                 |──▶ [10.244.2.8:8080]             │
│   │ 10.96.45.124:80 |──▶ [10.244.3.2:8080]             │
│   └─────────────────┘                                  │
│                                                          │
│   负载均衡算法: rr, wrr, lc, wlc, sh, sed, nq         │
└─────────────────────────────────────────────────────────┘
```

### 启用 IPVS

```bash
# 确保 ipvs 模块加载
cat /proc/modules | grep ip_vs

# 启用 IPVS 模式
kube-proxy --proxy-mode=ipvs
```

### 负载均衡算法

| 算法 | 说明 | 适用场景 |
|---|---|---|
| `rr` | 轮询（Round Robin） | 默认，最简单 |
| `wrr` | 加权轮询（Weighted RR） | 节点能力不同 |
| `lc` | 最少连接（Least Connection） | 长连接 |
| `wlc` | 加权最少连接 | 节点能力不同 + 长连接 |
| `sh` | 源哈希（Source Hashing） | 保持会话 |
| `sed` | 最短期望延迟 | 优化响应时间 |
| `nq` | 从不排队 | 高吞吐 |

### 查看 IPVS 规则

```bash
ipvsadm -L -n
```

```
IP Virtual Server version 1.2.1 (size=4096)
Prot LocalAddress:Port Scheduler Flags
  -> RemoteAddress:Port           Forward  Weight ActiveConn InActConn
TCP  10.96.45.123:80 rr
  -> 10.244.1.5:8080              Masq    1      0          0
  -> 10.244.2.8:8080              Masq    1      0          0
TCP  10.96.45.124:80 wrr
  -> 10.244.3.2:8080              Masq    2      0          0
```

## 性能对比

| 指标 | iptables | IPVS |
|---|---|---|
| 查找复杂度 | O(n) | O(1) |
| 规则更新 | 增量更新 | 批量替换 |
| 5000 Svc 延迟 | 较高 | 稳定 |
| 10000 Svc 延迟 | 很高 | 稳定 |
| 内存占用 | 高 | 中 |
| 支持算法 | 随机 | 多种 |

## 连接跟踪

kube-proxy 还负责维护 conntrack（连接跟踪）表，确保 NAT 转换的正确性：

```
┌─────────────────────────────────────────────────────────┐
│                 conntrack Table                         │
│                                                          │
│  Original Tuple          Reply Tuple                     │
│  ┌──────────────┐       ┌──────────────┐                 │
│  │ SRC: Client  │       │ SRC: Pod     │                 │
│  │ DST: Svc IP  │  ══▶ │ DST: Client  │                 │
│  └──────────────┘       └──────────────┘                 │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

:::warning conntrack 限制
conntrack 表有大小限制（通常是 262144 条）。高并发场景下可能溢出，导致丢包。需要通过 `sysctl` 调整：

```bash
sysctl -w net.netfilter.nf_conntrack_max=524288
```
:::

## Hairpin Mode

Hairpin 模式允许 Pod 访问自己通过 Service 暴露的端口。当 Pod 通过 ClusterIP 访问自己的 Service 时：

```yaml
spec:
  ports:
  - name: http
    port: 80
    targetPort: 8080
    hairpin: true  # 启用 hairpin（需要 kubelet 支持）
```

流量路径：

```
Pod → veth → eth0 → kube-proxy (DNAT) → veth → Pod
```

## 总结

kube-proxy 是 Kubernetes Service 负载均衡的核心：

- **userspace**：已废弃，不推荐使用
- **iptables**：通用模式，规则多时性能下降
- **IPVS**：推荐模式，适合大规模集群

选择 IPVS 模式，配合正确的 conntrack 配置，是生产环境的最佳实践。
