# CNI 网络插件详解

Flannel、Calico、Cilium——Kubernetes 集群安装时，你必须选择其中一个。但它们有什么区别？为什么 Kubernetes 不自带网络功能，而是依赖这些插件？

要理解 CNI，首先要理解它解决的问题：让集群中的 Pod 能够相互通信，同时保持网络隔离和安全性。

## CNI 是什么

CNI（Container Network Interface）是一个标准规范，定义了容器网络配置的统一接口。

```
┌─────────────────────────────────────────────────────────┐
│                      kubelet                            │
│  ┌─────────────────────────────────────────────────────┐ │
│  │                  CNI Plugin                         │ │
│  │   ADD   │   DEL   │   CHECK   │   VERSION         │ │
│  └─────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────┘
                          │
              ┌───────────┼───────────┐
              ↓           ↓           ↓
        ┌─────────┐ ┌─────────┐ ┌─────────┐
        │ Flannel │ │ Calico  │ │ Cilium  │
        └─────────┘ └─────────┘ └─────────┘
```

### CNI 插件职责

| 职责 | 说明 |
|---|---|
| IP 分配 | 从预设 CIDR 为 Pod 分配唯一 IP |
| 网络配置 | 创建网桥、veth pair、路由等 |
| 策略执行 | 实现 NetworkPolicy（部分插件） |
| 观测能力 | 提供网络流量可见性（部分插件） |

## Flannel

Flannel 是最简单的 CNI 插件之一，由 CoreOS 开发。它的设计理念是「简单够用」。

### 架构

```
┌─────────────┐              ┌─────────────┐
│   Node 1    │   VXLAN      │   Node 2    │
│ ┌─────────┐ │◀────────────▶│ ┌─────────┐ │
│ │  Pod A  │ │              │ │  Pod B  │ │
│ │ 10.244.1.5│ │              │ │ 10.244.2.8│ │
│ └─────────┘ │              │ └─────────┘ │
│      │      │              │      │      │
│      └──────┼──────────────┼──────┘      │
│             ↓              ↓              │
│        ┌─────────────────────────┐        │
│        │       flannel.1        │ (VXLAN) │
│        │    (Overlay Interface) │         │
│        └─────────────────────────┘        │
└───────────────────────────────────────────┘
```

### 网络模型

```bash
# 节点网络配置
ip addr show flannel.1
```

```
5: flannel.1: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1450 qdisc noqueue
    inet 10.244.1.0/32 scope global flannel.1
    peer inet 10.244.2.0/32
```

Flannel 使用 **VXLAN** 隧道协议，将 Pod 网络封装在节点网络中：

| 层级 | 网络 |
|---|---|
| Pod Network | 10.244.0.0/16 |
| Node Network | 192.168.1.0/24 |
| Overlay | VXLAN |

### 优缺点

| 优点 | 缺点 |
|---|---|
| 简单易部署 | 不支持网络策略 |
| 资源占用低 | 性能开销较大（VXLAN） |
| 社区成熟 | 功能相对单一 |

### 适用场景

- 小型集群
- 对网络策略没有需求
- 快速部署演示环境

## Calico

Calico 是一个功能丰富的网络解决方案，提供了高性能的数据平面和强大的网络策略。

### 架构

Calico 支持两种数据平面：

```
┌─────────────────────────────────────────────────────────┐
│                     Calico Node                         │
│  ┌─────────────┐                                       │
│  │ Felix       │  ← 配置路由、网络策略                  │
│  └─────────────┘                                       │
│        │                                                │
│  ┌─────┴─────┐                                         │
│  ↓           ↓                                          │
│ ┌─────────┐ ┌─────────┐                                │
│ │ Bird     │ │  CNI    │                                │
│ │ (BGP)    │ │ Plugin  │                                │
│ └─────────┘ └─────────┘                                │
└─────────────────────────────────────────────────────────┘
```

### 两种模式

#### 直连模式（BGP）

节点之间通过 BGP 协议直接交换路由，效率最高：

```
Node 1: 10.244.1.0/24 via 192.168.1.1
Node 2: 10.244.2.0/24 via 192.168.1.2
```

适合大二层网络或节点间可以直接通信的环境。

#### VXLAN 模式

与 Flannel 类似，使用 VXLAN 封装：

```yaml
apiVersion: projectcalico.org/v3
kind: FelixConfiguration
metadata:
  name: default
spec:
  vxlanEnabled: true
  ipipEnabled: false
```

### 优缺点

| 优点 | 缺点 |
|---|---|
| 高性能（直连模式） | 配置相对复杂 |
| 强大的网络策略 | BGP 模式需要网络支持 |
| 支持大规模集群 | 资源占用高于 Flannel |

### 适用场景

- 中大型集群
- 需要网络策略
- 对性能有要求
- 多云或混合云环境

## Cilium

Cilium 是新一代的 CNI 插件，基于 eBPF（extended Berkeley Packet Filter）技术，提供了前所未有的网络可观测性和安全性。

### eBPF 架构

```
┌─────────────────────────────────────────────────────────┐
│                     Cilium Agent                        │
│  ┌─────────────┐                                       │
│  │  eBPF Maps   │  ← 存储网络状态                       │
│  └─────────────┘                                       │
│        │                                                │
│  ┌─────┴─────┐                                         │
│  ↓           ↓                                          │
│ ┌─────────┐ ┌─────────┐                                │
│ │ XDP     │ │ TC      │  ← eBPF 程序入口                 │
│ └─────────┘ └─────────┘                                │
└─────────────────────────────────────────────────────────┘
```

### 优势

| 特性 | 说明 |
|---|---|
| L7 策略 | 支持 HTTP、gRPC、Kafka 等应用层协议过滤 |
| 身份感知 | 基于工作负载身份而非 IP 做策略 |
| Hubble | 提供了 Kubernetes 原生的可观测性 |
| 高性能 | 内核级处理，无需 iptables |

### 适用场景

- 需要 L7 网络策略
- 对网络安全要求高
- 需要网络流量可观测性
- 大规模高性能场景

## 横向对比

| 特性 | Flannel | Calico | Cilium |
|---|---|---|---|
| IP 分配 | etcd/kube-router | etcd/BGP | kvstore/BGP |
| 数据平面 | VXLAN | BGP/VXLAN | eBPF |
| 网络策略 | ✗ | ✓ (L3/L4) | ✓ (L3-L7) |
| 性能 | 中等 | 高 | 最高 |
| 复杂度 | 低 | 中 | 中高 |
| 观测能力 | 基础 | 中等 | 强大 |
| 最小资源占用 | ✓ | 中 | 中 |
| 生产就绪 | ✓ | ✓ | ✓ |

## 选择建议

### 选择 Flannel 如果：

- 你是 Kubernetes 新手
- 集群规模小于 100 节点
- 不需要网络策略
- 快速搭建开发/测试环境

### 选择 Calico 如果：

- 需要网络策略
- 集群规模中等
- 需要较好的性能
- 多云环境

### 选择 Cilium 如果：

- 需要 L7 网络策略
- 对网络安全要求极高
- 需要网络可观测性
- 大规模高性能集群
- 已经使用 Hubble 进行监控

## 总结

CNI 插件是 Kubernetes 网络的基础：

- **Flannel**：简单够用，适合小型集群
- **Calico**：功能丰富，适合中大型集群
- **Cilium**：eBPF 驱动，适合高安全要求场景

选择合适的 CNI 插件，需要根据集群规模、安全需求和运维能力综合考虑。
