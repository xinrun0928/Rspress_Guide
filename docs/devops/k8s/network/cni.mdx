# CNI 网络插件对比

你有没有想过——Kubernetes 的 Pod 网络到底是怎么实现的？Flannel 说它用 VXLAN，Calico 说它用 BGP，Cilium 说它用 eBPF。这些技术听起来都很厉害，但到底该怎么选？

CNI（Container Network Interface）是 Kubernetes 网络的标准接口。所有网络插件都通过 CNI 与 Kubernetes 交互，实现 Pod 网络、Service 网络和网络策略。

## CNI 基础

### CNI 工作流程

```
Pod 创建时
    │
    ▼
┌─────────────────────────────────┐
│         kubelet                 │
│                                  │
│  1. 调用 CNI ADD                │
│     - 分配 IP 地址              │
│     - 创建网络命名空间          │
│     - 配置 veth pair            │
│     - 调用 CNI 配置网络          │
│                                  │
└─────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────┐
│         CNI Plugin              │
│                                  │
│  - Flannel / Calico / Cilium   │
│  - 决定如何与外部网络通信       │
│  - 设置路由规则                  │
└─────────────────────────────────┘
```

### CNI 插件分类

| 类型 | 代表插件 | 特点 |
|---|---|---|
| **Overlay 网络** | Flannel (VXLAN)、Weave | 不依赖物理网络，隧道封装 |
| **路由网络** | Calico (BGP) | 依赖物理网络，直接路由 |
| **透明代理** | Cilium (eBPF) | 内核级数据面，性能最佳 |

## Flannel：简单够用

Flannel 是 CoreOS（现 Red Hat）开发的 CNI 插件，以简单著称。

### 工作原理

Flannel 在每个节点上创建一个 `cni0` 网桥，为该节点的 Pod 分配一个子网。

```bash
# 查看节点子网分配
kubectl get nodes -o jsonpath='{range .items[*]}{.metadata.name}{"\t"}{.spec.podCIDR}{"\n"}{end}'

# 节点 Pod 子网示例：
# node-1  → 10.244.0.0/24
# node-2  → 10.244.1.0/24
# node-3  → 10.244.2.0/24
```

**跨节点通信**：Flannel 使用 **VXLAN** 隧道封装数据包。

```
Pod-A (Node-1)                          Pod-B (Node-2)
  │                                        ▲
  │ 原始数据包:                             │
  │   src: 10.244.0.2                      │
  │   dst: 10.244.1.3                      │
  ▼                                        │
┌─────────────────────────────────────────┐
│           Node-1                         │
│                                          │
│  cni0 (10.244.0.0/24)                   │
│       │                                  │
│       │ 封装为 VXLAN 包                  │
│       │   外层 src: 192.168.1.10         │
│       │   外层 dst: 192.168.1.20         │
│       ▼                                  │
│  eth0 (物理网卡) ──────────────────────► │
└─────────────────────────────────────────┘
                    │
                    │ VXLAN 隧道
                    ▼
┌─────────────────────────────────────────┐
│           Node-2                         │
│  eth0 (物理网卡) ◄──────────────────────│
│       │                                  │
│       │ 解封装                            │
│       ▼                                  │
│  cni0 (10.244.1.0/24)                   │
│       │                                  │
│       ▼                                  │
│  Pod-B (10.244.1.3)                     │
└─────────────────────────────────────────┘
```

### Flannel 后端对比

| 后端 | 协议 | 适用场景 |
|---|---|---|
| **vxlan** | VXLAN | 生产环境默认，性能好 |
| **host-gw** | 路由 | 同二层网络，性能最优 |
| **udp** | UDP | 已废弃，仅用于调试 |

```yaml title="flannel-config.yaml"
apiVersion: v1
kind: ConfigMap
metadata:
  name: kube-flannel-cfg
  namespace: kube-flannel
data:
  cni-conf.json: |
    {
      "name": "cbr0",
      "cniVersion": "0.3.1",
      "plugins": [
        {
          "type": "flannel",
          "delegate": {
            "hairpinMode": true,
            "isDefaultGateway": true
          }
        },
        {
          "type": "portmap",
          "capabilities": {
            "portMappings": true
          }
        }
      ]
    }
  net-conf.json: |
    {
      "Network": "10.244.0.0/16",
      "Backend": {
        "Type": "vxlan"
      }
    }
```

### 优缺点

| 优点 | 缺点 |
|---|---|
| 部署简单 | 性能开销（VXLAN 封装） |
| 社区活跃 | 不支持网络策略 |
| 与 Kubernetes 原生集成 | 不支持细粒度的网络隔离 |

## Calico：功能全面

Calico 是 Tigera 开发的 CNI 插件，支持网络策略和安全，是生产环境的热门选择。

### 工作原理

Calico 使用 **BGP 协议**在节点之间分发路由表，实现 Pod 的三层可达。

```bash
# 查看 Calico 路由表
ip route

# 示例输出：
# 10.244.1.0/24 via 192.168.1.20 dev eth0 proto bird
# 10.244.2.0/24 via 192.168.1.30 dev eth0 proto bird
```

**BGP（Border Gateway Protocol）** 是互联网骨干网使用的路由协议，Calico 将其引入 Kubernetes 集群内部。

```
┌──────────────────────────────────────────────────────────────────┐
│                    Calico BGP 架构                                │
│                                                                   │
│   Node-1 (192.168.1.10)    Node-2 (192.168.1.20)              │
│   ┌─────────────────┐        ┌─────────────────┐                │
│   │ Pod 子网:       │◄──────►│ Pod 子网:        │                │
│   │ 10.244.0.0/24  │  BGP   │ 10.244.1.0/24   │                │
│   │                 │◄──────►│                 │                │
│   │ BGP Speaker:    │        │ BGP Speaker:    │                │
│   │ bird           │        │ bird           │                │
│   └─────────────────┘        └─────────────────┘                │
│                                                                   │
│   每个节点通过 BGP 将自己的 Pod 子网通告给其他节点                │
│   其他节点学习到路由后，直接路由到对应节点                         │
└──────────────────────────────────────────────────────────────────┘
```

### 两种运行模式

| 模式 | 说明 | 适用场景 |
|---|---|---|
| **BGP Full Mesh** | 每个节点与其他所有节点建立 BGP 连接 | 小规模集群（< 100 节点） |
| **Route Reflector** | 节点与 Route Reflector 建立连接 | 大规模集群 |

```yaml title="calico-bgp-config.yaml"
apiVersion: projectcalico.org/v3
kind: BGPConfiguration
metadata:
  name: default
spec:
  logSeverityScreen: Info
  nodeToNodeMeshEnabled: true  # Full Mesh 模式
  asNumber: 64512
---
# 大规模集群使用 Route Reflector
apiVersion: projectcalico.org/v3
kind: BGPPeer
metadata:
  name: route-reflector-peer
spec:
  peerIP: 192.168.1.10
  asNumber: 64512
  nodeSelector: role == "route-reflector"
```

### 网络策略

Calico 最强大的功能之一是**网络策略**——可以精细控制 Pod 之间的流量。

```yaml title="network-policy.yaml"
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: api-network-policy
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: api
  policyTypes:
  - Ingress
  - Egress
  ingress:
  # 允许来自 frontend 的流量
  - from:
    - podSelector:
        matchLabels:
          app: frontend
    ports:
    - protocol: TCP
      port: 8080
  # 允许来自 ingress controller 的流量
  - from:
    - namespaceSelector:
        matchLabels:
          name: ingress-nginx
    ports:
    - protocol: TCP
      port: 8080
  egress:
  # 只允许访问数据库
  - to:
    - podSelector:
        matchLabels:
          app: database
    ports:
    - protocol: TCP
      port: 5432
  # 只允许访问 DNS
  - to:
    - namespaceSelector: {}
    ports:
    - protocol: UDP
      port: 53
```

### 优缺点

| 优点 | 缺点 |
|---|---|
| 高性能（纯三层路由） | 配置相对复杂 |
| 强大的网络策略 | Full Mesh 不适合大规模集群 |
| 支持网络隔离 | 需要物理网络支持（BGP） |

## Cilium：面向未来

Cilium 是 CNCF 孵化的项目，利用 **eBPF（Extended Berkeley Packet Filter）** 在 Linux 内核中实现网络功能。

### eBPF 是什么

eBPF 是 Linux 内核的一项革命性技术，允许在内核中运行沙箱程序来处理网络包、追踪系统调用等。

```
传统网络路径：
  应用程序 → 用户态 → 内核协议栈 → 网卡
              ▲         ▲
              │         │
           慢 (上下文切换)

Cilium/eBPF 路径：
  应用程序 → 用户态 → eBPF (内核中) → 网卡
                            ▲
                            │
                       直接在内核中处理
```

### Cilium 的优势

**1. 零包处理（Zero-Packet Processing）**：

```bash
# Cilium 可以直接通过 eBPF 将流量从 Pod A 转发到 Pod B
# 完全绕过 iptables/ipvs
# 延迟降低 50% 以上
```

**2. 透明的可见性**：

```bash
# Cilium 支持在不修改应用的情况下
# 追踪服务间的通信
cilium status

# 查看服务依赖图
cilium service list
```

**3. 身份安全（Identity-Based Security）**：

Cilium 使用 **BPF Maps** 追踪每个 Pod 的网络身份，不再依赖 IP 地址。

```yaml title="cilium-network-policy.yaml"
apiVersion: cilium.io/v2
kind: CiliumNetworkPolicy
metadata:
  name: api-policy
spec:
  endpointSelector:
    matchLabels:
      app: api
  ingress:
  - fromEndpoints:
    - matchLabels:
        app: frontend
    toPorts:
    - ports:
      - port: "8080"
        protocol: TCP
      rules:
        http:
        - method: GET
          path: "/api/.*"
```

**4. L7 网络策略**：

```yaml
apiVersion: cilium.io/v2
kind: CiliumNetworkPolicy
metadata:
  name: l7-http-policy
spec:
  endpointSelector:
    matchLabels:
      app: backend
  ingress:
  - toPorts:
    - ports:
      - port: "8080"
        protocol: TCP
      rules:
        http:
        - method: "GET"
          path: "/health"
        - method: "POST"
          path: "/api/users"
          headers:
          - "Authorization:.*"
```

### 优缺点

| 优点 | 缺点 |
|---|---|
| 极高性能（内核级数据面） | 需要较新的内核版本 |
| L3-L7 网络策略 | 文档相对较少 |
| 零依赖（不需要额外的 DaemonSet） | 社区相对较小 |
| 透明的观测能力 | eBPF 学习曲线 |

## 三者对比

| 维度 | Flannel | Calico | Cilium |
|---|---|---|---|
| **网络模型** | Overlay (VXLAN) | BGP 路由 / Overlay | eBPF |
| **性能** | 中等 | 高 | 最高 |
| **网络策略** | 不支持 | L3/L4 | L3-L7 |
| **复杂度** | 低 | 中 | 中高 |
| **依赖** | etcd | etcd / BGP RR | 无（可选 Hubble） |
| **最小内核版本** | 无 | 无 | 4.19+ |
| **适合规模** | 任意 | 大规模 | 任意 |
| **学习曲线** | 平缓 | 较陡 | 较陡 |

## 选择指南

### 选 Flannel 如果：

- 你是 K8s 新手，想要快速上手
- 小规模集群（几十个节点）
- 不需要网络策略
- 追求简单，够用就好

### 选 Calico 如果：

- 生产环境，需要网络策略
- 需要将 Kubernetes 网络与物理网络集成
- 大规模集群
- 需要 BGP 路由控制

### 选 Cilium 如果：

- 对网络性能有极致要求
- 需要 L7 网络策略（HTTP 方法/路径控制）
- 已经使用 Istio 或 Linkerd
- 希望获得透明的观测能力

## 迁移与混用

### 从 Flannel 迁移到 Calico

```bash
# 1. 安装 Calico
kubectl apply -f https://docs.projectcalico.org/manifests/calico.yaml

# 2. 等待 Calico 启动
kubectl get pods -n calico-system

# 3. 移除 Flannel
kubectl delete -f kube-flannel.yml

# 4. 验证
kubectl get nodes -o wide  # 确认 Pod CIDR 未变
ip route  # 确认路由表已更新为 Calico 模式
```

### 注意事项

:::warning 迁移有风险
生产环境迁移 CNI 插件有服务中断风险，建议：
1. 在测试环境先验证
2. 预留足够的维护窗口
3. 准备好回滚方案
:::

---

## 总结

CNI 插件是 Kubernetes 网络的基础。选择合适的 CNI 插件，需要考虑：

1. **性能需求**：eBPF > BGP > VXLAN
2. **安全需求**：是否需要网络策略
3. **规模需求**：集群规模影响架构选择
4. **运维能力**：学习曲线和维护成本

无论选择哪个插件，都要确保与 Kubernetes 版本兼容，并做好充分的测试。
