# Pod 通信深度解析

Pod A 在 Node 1，Pod B 在 Node 2，它们如何通信？流量从 Pod 到 Service 再到 Pod，中间经历了什么？

理解 Pod 通信的完整路径，是掌握 Kubernetes 网络的关键。这一节将深入剖析每一种通信场景的数据包走向。

## 同节点 Pod-to-Pod

同一节点上的 Pod 通信是最简单的情况：

```
┌──────────────────────────────────────────────────────┐
│                      Node                            │
│                                                      │
│  ┌─────────────┐              ┌─────────────┐       │
│  │    Pod A    │              │    Pod B    │       │
│  │ 10.244.1.5  │              │ 10.244.1.6  │       │
│  │     eth0    │◀──────┐      │     eth0    │       │
│  └──────┬──────┘      │      └──────┬──────┘       │
│         │             │             │              │
│  ┌──────▼─────────────▼─────────────▼──────┐       │
│  │                cni0                     │       │
│  │              (网桥)                     │       │
│  │        MAC 表 / ARP 解析                 │       │
│  └─────────────────────────────────────────┘       │
└──────────────────────────────────────────────────────┘
```

数据包走向：

1. Pod A 发送数据包，目标 IP 10.244.1.6
2. 通过 veth pair 到达网桥 cni0
3. 网桥查找 MAC 表或 ARP，确定目标 veth
4. 数据包发送到 Pod B 的 veth
5. Pod B 收到数据包

## 跨节点 Pod-to-Pod

跨节点通信需要 CNI 插件的支持：

### Overlay 模式（VXLAN）

```
┌──────────────────────────────────────────────────────────────────┐
│                         Overlay Network                            │
│  ┌─────────────┐                              ┌─────────────┐     │
│  │   Node 1    │                              │   Node 2    │     │
│  │  Pod A ────┼───────────────────────────────┼───► Pod B  │     │
│  │  (封装)    │        VXLAN Tunnel           │  (解封装)   │     │
│  └─────────────┘                              └─────────────┘     │
└──────────────────────────────────────────────────────────────────┘
```

数据包走向：

```
┌──────────────────────────────────────────────────────────────────┐
│ 1. Pod A 发送: 源 IP 10.244.1.5, 目标 IP 10.244.2.6              │
│                                                                  │
│ 2. 本地路由查找: 10.244.2.0/24 via flannel.1                     │
│                                                                  │
│ 3. VXLAN 封装:                                                   │
│    外层: 源 MAC Node1, 目标 MAC Node2 (通过 ARP over VTEP)       │
│    内层: 10.244.1.5 → 10.244.2.6                                 │
│                                                                  │
│ 4. Node 2 收到，VXLAN 解封装                                      │
│                                                                  │
│ 5. 通过本地路由发送到 Pod B                                       │
└──────────────────────────────────────────────────────────────────┘
```

### 直连模式（BGP/Calico）

```
┌──────────────────────────────────────────────────────────────────┐
│                       Direct Routing                               │
│  ┌─────────────┐                              ┌─────────────┐     │
│  │   Node 1    │                              │   Node 2    │     │
│  │  Pod A      │ ──────────────────────────▶  │  Pod B      │     │
│  │ 10.244.1.5  │         直接路由             │ 10.244.2.6  │     │
│  └─────────────┘                              └─────────────┘     │
│                                                                  │
│  路由表 (Node 1):                                                │
│  10.244.2.0/24 via 192.168.1.2 dev eth0                         │
└──────────────────────────────────────────────────────────────────┘
```

数据包走向：

1. Pod A 发送数据包，目标 IP 10.244.2.6
2. 本地路由查找下一跳
3. 通过节点网络直接路由到 Node 2
4. Node 2 通过本地网络发送到 Pod B

:::tip 性能差异
直连模式比 Overlay 少一层封装，延迟更低、吞吐量更高。这也是 Calico 在性能敏感场景更受欢迎的原因。
:::

## Pod-to-Service

Pod 访问 Service 时，会经过 kube-proxy 的 DNAT 转换：

```
┌──────────────────────────────────────────────────────────────────┐
│                       Service 访问流程                             │
│  ┌─────────┐     ┌────────────┐     ┌─────────────┐                │
│  │  Pod A  │────▶│  Service   │────▶│   Pod B     │                │
│  │         │     │  ClusterIP │     │             │                │
│  └─────────┘     └────────────┘     └─────────────┘                │
│       │                │                     ▲                   │
│       │                │ DNAT                 │                  │
│       │                ▼                     │                  │
│       │          ┌────────────┐                │                  │
│       └──────────│ kube-proxy │────────────────┘                  │
│                  └────────────┘                                   │
└──────────────────────────────────────────────────────────────────┘
```

### iptables 模式

```bash
# 1. Pod A 发送: 10.244.1.5 → 10.96.45.123:80
#    (10.96.45.123 是 Service 的 ClusterIP)

# 2. kube-proxy 修改 iptables:
#    源: 10.244.1.5 → 10.96.45.123:80
#    变为: 10.244.1.5 → 10.244.2.6:8080 (随机选择一个 Pod)

# 3. Pod B 收到的请求:
#    源 IP: 10.244.1.5 (保留)
#    目标: 10.244.2.6:8080
```

### IPVS 模式

IPVS 的流程类似，但使用内核级的哈希表查找，效率更高：

```bash
# IPVS 维护的映射表
Service: 10.96.45.123:80
  → Real Server 1: 10.244.1.5:8080
  → Real Server 2: 10.244.2.6:8080
  → Real Server 3: 10.244.3.2:8080
```

## Service-to-Pod

Service 本身不生成流量，它只是一个负载均衡的「虚地址」。Service 到 Pod 的流量就是上面的 Pod-to-Service 反向过程。

## Node-to-Pod

从节点访问 Pod 相对简单：

```
Node (192.168.1.1) → Pod (10.244.1.5)
```

节点和 Pod 在同一网络平面（通过 CNI 网桥或路由），可以直接通信。

### Hairpin Mode

但如果节点上的 kubelet 或 kube-proxy 需要通过 Service 访问本节点的 Pod：

```
Node → kube-proxy → Service → (本节点) Pod
```

这时需要启用 Hairpin Mode：

```yaml
# kubelet 配置
--hairpin-mode=hairpin-veth
```

```bash
# 查看 hairpin 配置
cat /var/lib/kubelet/config.yaml | grep hairpin
```

```
hairpinMode: hairpin-veth
```

## External-to-Pod

外部流量进入集群需要经过多层转发：

### 通过 NodePort

```
Internet → LoadBalancer → Node1:30080 → Pod (任一节点)
                                        ↓
                    kube-proxy (负载均衡到目标 Pod)
```

### 通过 Ingress

```
Internet → Ingress Controller → Service → Pod
                                    │
                          kube-proxy DNAT
```

### 完整路径

```
用户请求 → 负载均衡器 → Ingress Controller → NodePort/Service → kube-proxy → Pod
                    │                                      │
                    │                                      ▼
                    │                              Service DNAT
                    │                         ClusterIP → PodIP
                    ▼
          Ingress 路由 + TLS 终止
```

## NAT 与连接跟踪

### Source NAT (SNAT)

当 Pod 访问外部网络时，源 IP 会被 SNAT：

```
Pod (10.244.1.5) → External (8.8.8.8)
     ↓
kube-proxy SNAT
     ↓
Node (192.168.1.1) → External (8.8.8.8)
```

### Destination NAT (DNAT)

当外部访问 Service 时，目标 IP 会被 DNAT：

```
External → Service (10.96.45.123)
     ↓
kube-proxy DNAT
     ↓
External → Pod (10.244.1.5)
```

### 连接跟踪 (conntrack)

conntrack 记录双向连接状态，确保回复流量正确转换：

```
┌─────────────────────────────────────────────────────────┐
│                    conntrack Entry                      │
│                                                          │
│  Original: 10.244.1.5:50000 → 10.96.45.123:80           │
│  Reply:    10.96.45.123:80 → 10.244.1.5:50000           │
│                                                          │
│  ↔ 连接状态: ESTABLISHED                                 │
└─────────────────────────────────────────────────────────┘
```

## 总结

Kubernetes 网络通信的关键点：

- **同节点 Pod**：通过网桥直接转发
- **跨节点 Pod**：Overlay 封装或直连路由
- **Pod-to-Service**：kube-proxy DNAT 到具体 Pod
- **External-to-Pod**：多层转发 + kube-proxy
- **NAT 与 conntrack**：确保流量正确转换

理解这些路径，能够帮助你快速定位网络问题。
