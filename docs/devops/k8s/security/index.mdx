# Kubernetes 安全总览

在 Kubernetes 集群中，安全是一个多层次的系统工程。从网络隔离到访问控制，从容器运行时到 etcd 存储，每个环节都需要精心设计。随着 Kubernetes 承载的业务越来越核心，安全问题也从「可选项」变成了「必选项」。

本章节将深入探讨 Kubernetes 安全体系的各个层面，帮助你构建一个纵深防御的安全架构。

## 安全分层模型

Kubernetes 安全可以从四个维度来理解：

| 层级 | 关注点 | 核心机制 |
|---|---|---|
| 集群安全 | etcd、API Server、kubelet | TLS 证书、RBAC、网络策略 |
| 节点安全 | 主机加固、系统隔离 | Security Context、AppArmor、SELinux |
| 网络安全 | Pod 间通信、入口出口流量 | NetworkPolicy、CNI 插件 |
| 应用安全 | 敏感数据、访问凭证 | Secret、ServiceAccount、Pod Security |

## 核心安全机制

### 身份与访问控制

Kubernetes 采用 RBAC（基于角色的访问控制）来管理权限。通过 Role、ClusterRole、RoleBinding 和 ClusterRoleBinding，你可以细粒度地控制谁可以操作什么资源。

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: pod-reader
  namespace: default
rules:
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get", "list", "watch"]
```

ServiceAccount 则是 Pod 访问 API Server 的身份凭证，每个 Pod 都可以关联一个 ServiceAccount，Pod 内的进程通过挂载的 Token 与 API Server 进行认证。

### 网络隔离

默认情况下，Kubernetes 集群内的 Pod 可以自由通信，这在微服务架构中带来了便利，但也埋下了安全隐患。NetworkPolicy 允许你定义 Pod 级别的网络隔离规则，只允许授权的流量通过。

:::tip 推荐实践
始终为命名空间设置默认的拒绝策略，然后按需开放所需的通信路径。这遵循了「最小权限」原则。
:::

### 敏感信息管理

Secret 提供了存储敏感数据（如密码、OAuth 令牌、SSH 密钥）的方式。需要注意的是，Secret 中的数据默认是 Base64 编码而非加密，etcd 中以明文存储。在生产环境中，应启用 Secret 加密或使用外部密钥管理服务。

### 容器运行时安全

Security Context 允许你配置 Pod 或容器的特权级别、Linux 能力、文件系统只读等安全相关设置。通过 Pod Security Standards，你可以强制实施命名空间级别的安全策略。

## 安全加固清单

在生产环境中部署工作负载前，建议检查以下项目：

- [ ] API Server 禁止匿名访问 (`--anonymous-auth=false`)
- [ ] etcd 使用 TLS 加密通信
- [ ] kubelet 关闭匿名认证 (`--anonymous-auth=false`)
- [ ] 为不同命名空间设置 NetworkPolicy
- [ ] 避免使用默认 ServiceAccount
- [ ] 限制容器的特权模式和 Linux 能力
- [ ] 使用只读根文件系统
- [ ] 启用 Secret 加密存储
- [ ] 配置 Pod Security Standards
- [ ] 定期更新集群组件版本
