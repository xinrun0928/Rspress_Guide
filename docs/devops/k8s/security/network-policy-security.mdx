# NetworkPolicy：Kubernetes 网络隔离实战

你有没有想过这个问题：在 K8s 集群里，Pod A 能否直接访问 Pod B 的 3306 端口？如果你的答案是「默认情况下可以」，那么恭喜你，你发现了多租户安全的一个巨大隐患。

默认情况下，K8s 集群内的所有 Pod 可以互相通信，无论它们在哪个 Namespace。这意味着：如果一个租户的 Web 应用被攻破，攻击者可以横向移动到数据库 Pod，甚至影响其他租户的系统。

NetworkPolicy 就是解决这个问题的关键——它允许你定义精细的网络访问控制策略，把「默认放行」变成「默认拒绝」。

## NetworkPolicy 基础

### 核心概念

NetworkPolicy 是 K8s 的网络隔离机制，通过 Label 选择器定义允许的入站和出站流量：

```yaml title="最简单的 NetworkPolicy"
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: deny-all-ingress
  namespace: production
spec:
  podSelector: {}      # 选择所有 Pod
  policyTypes:
  - Ingress            # 声明入口流量类型
  ingress: []          # 入站规则为空 = 拒绝所有
```

:::tip policyTypes 的重要性
`policyTypes` 是 NetworkPolicy 的关键字段。如果不显式声明，K8s 假设你只想控制入站流量。**声明所有需要的类型**（Ingress、Egress、或两者）是最佳实践。
:::

### 流量模型

NetworkPolicy 控制两类流量：

| 类型 | 说明 |
|---|---|
| Ingress | 入站流量 = 外部到 Pod |
| Egress | 出站流量 = Pod 到外部 |

```yaml
spec:
  podSelector:
    matchLabels:
      app: frontend
  policyTypes:
  - Ingress
  - Egress
  ingress:
  # 入站规则
  egress:
  # 出站规则
```

## 选择器语法

### podSelector：选择目标 Pod

```yaml
# 选择所有 Pod
podSelector: {}

# 选择带有特定标签的 Pod
podSelector:
  matchLabels:
    app: nginx

# 使用表达式选择器
podSelector:
  matchExpressions:
  - key: tier
    operator: In
    values: ["frontend", "backend"]
```

### namespaceSelector：选择源/目标 Namespace

```yaml
# 允许来自特定 Namespace 的流量
ingress:
- from:
  - namespaceSelector:
      matchLabels:
        name: production

# 允许来自特定 Namespace + 特定 Pod 的流量
ingress:
- from:
  - namespaceSelector:
      matchLabels:
        name: production
    podSelector:
      matchLabels:
        app: api
```

### IPBlock：基于 IP CIDR

```yaml
# 允许来自特定 IP 段的流量
ingress:
- from:
  - ipBlock:
      cidr: 172.17.0.0/16
      except:
      - 172.17.1.0/24
```

:::warning IPBlock 的陷阱
IPBlock 不能同时使用 `podSelector` 和 `namespaceSelector`。如果需要更复杂的匹配，必须分多个规则或使用其他策略组合。
:::

## 实战策略

### 策略一：默认拒绝所有入站流量

在每个 Namespace 创建默认策略，隔离风险：

```yaml title="默认拒绝入站"
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-ingress
spec:
  podSelector: {}
  policyTypes:
  - Ingress
```

```yaml title="默认拒绝出站"
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-egress
spec:
  podSelector: {}
  policyTypes:
  - Egress
```

:::tip 部署顺序
先部署允许策略，再部署拒绝策略。如果顺序颠倒，会导致所有流量中断。
:::

### 策略二：按应用层级隔离

典型的微服务架构隔离策略：

```yaml title="Frontend Pods：只允许 HTTP 入站"
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: frontend-network-policy
  namespace: production
spec:
  podSelector:
    matchLabels:
      tier: frontend
  policyTypes:
  - Ingress
  - Egress
  ingress:
  - ports:
    - port: 80
      protocol: TCP
    - port: 443
      protocol: TCP
  # 允许来自任何地方的 HTTP/HTTPS
  egress:
  - to:
    - podSelector:
        matchLabels:
          tier: backend
    ports:
    - port: 8080
      protocol: TCP
```

```yaml title="Backend Pods：只允许前端访问"
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: backend-network-policy
  namespace: production
spec:
  podSelector:
    matchLabels:
      tier: backend
  policyTypes:
  - Ingress
  - Egress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          tier: frontend
    ports:
    - port: 8080
      protocol: TCP
  egress:
  - to:
    - podSelector:
        matchLabels:
          tier: database
    ports:
    - port: 5432
      protocol: TCP
```

```yaml title="Database Pods：只允许后端访问"
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: database-network-policy
  namespace: production
spec:
  podSelector:
    matchLabels:
      tier: database
  policyTypes:
  - Ingress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          tier: backend
    ports:
    - port: 5432
      protocol: TCP
```

### 策略三：DNS 出站规则

无论什么应用，都需要访问 DNS。常见模式是允许所有流量到 DNS 服务器：

```yaml title="允许 DNS 出站"
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-dns-egress
  namespace: production
spec:
  podSelector: {}
  policyTypes:
  - Egress
  egress:
  # 允许到 kube-system Namespace 的 DNS 服务
  - to:
    - namespaceSelector:
        matchLabels:
          kubernetes.io/metadata.name: kube-system
    ports:
    - port: 53
      protocol: TCP
    - port: 53
      protocol: UDP
```

:::tip kube-dns 的新标签
K8s 1.20+ 后，kube-dns 使用 `kubernetes.io/metadata.name: kube-system` 标签。旧的 `kubernetes.io/cluster-service: "true"` 标签已被废弃。
:::

### 策略四：混合命名空间隔离

```yaml title="允许跨 Namespace 访问"
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: api-to-db-policy
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: database
  policyTypes:
  - Ingress
  ingress:
  # 允许 API Pod 从 monitoring Namespace 访问
  - from:
    - namespaceSelector:
        matchLabels:
          name: monitoring
      podSelector:
        matchLabels:
          app: api
    ports:
    - port: 5432
```

## CNI 支持要求

### 哪些 CNI 支持 NetworkPolicy

| CNI | NetworkPolicy 支持 |
|---|---|
| Calico | 完全支持 |
| Cilium | 完全支持（支持 L7 策略） |
| Weave Net | 支持 |
| Flannel | 不支持 |
| Kube-router | 支持 |

:::warning 确认 CNI 支持
在部署 NetworkPolicy 之前，必须确认你的 CNI 插件支持该功能。如果使用 Flannel，NetworkPolicy 不会生效。

```bash
kubectl get nodes -o wide
# 查看节点上的 CNI 类型
```
:::

### 检查 NetworkPolicy 状态

```bash
# 检查 Namespace 是否应用了 NetworkPolicy
kubectl get networkpolicies -n production

# 检查 CNI 是否支持 NetworkPolicy
kubectl get crd networkpolicies.networking.k8s.io
```

## 常见问题与解决方案

### 问题一：应用启动后无法访问外部服务

**原因**：Egress 规则没有配置外网访问。

**解决**：添加外网访问规则或只保留必要的 Egress 策略。

```yaml
egress:
# 允许访问外部 HTTP/HTTPS
- ports:
  - port: 80
    protocol: TCP
  - port: 443
    protocol: TCP
# 允许 DNS
- ports:
  - port: 53
    protocol: TCP
  - port: 53
    protocol: UDP
```

### 问题二：Health Check 失败

**原因**：Kubelet 的健康检查探针被 NetworkPolicy 阻止。

**解决**：Kubelet 使用特殊端口访问 Pod，不需要 Egress 规则。但如果是应用自检，确保探针端口在 Ingress 规则中。

### 问题三：Headless Service 和 DNS 解析问题

**原因**：Headless Service 直接返回 Pod IP，DNS 解析被阻止。

**解决**：确保 DNS 流量到 CoreDNS 被允许：

```yaml
egress:
- to:
  - namespaceSelector:
      matchLabels:
        kubernetes.io/metadata.name: kube-system
    podSelector:
      matchLabels:
        k8s-app: kube-dns
```

## 高级用法

### 命名空间级别的默认策略

通过 Admission Controller 自动为新 Namespace 添加默认策略：

```yaml title="MutatingWebhook 配置"
apiVersion: admissionregistration.k8s.io/v1
kind: MutatingWebhookConfiguration
metadata:
  name: default-network-policy
webhooks:
- name: network-policy.default.svc
  clientConfig:
    service:
      name: network-policy-webhook
      namespace: kube-system
  rules:
  - operations: ["CREATE"]
    apiGroups: [""]
    apiVersions: ["v1"]
    resources: ["pods"]
  namespaceSelector:
    matchLabels:
      enforce-network-policy: "true"
```

### 集成服务网格

在服务网格（如 Istio）中，NetworkPolicy 和 Sidecar 策略可以叠加：

```yaml
# NetworkPolicy：L3/L4 层控制
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: api-network-policy
spec:
  podSelector:
    matchLabels:
      app: api
  policyTypes:
  - Ingress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: ingress

# Istio AuthorizationPolicy：L7 层控制
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: api-authz
spec:
  selector:
    matchLabels:
      app: api
  rules:
  - from:
    - source:
        principals: ["cluster.local/ns/default/sa/ingress"]
    to:
    - operation:
        methods: ["GET"]
        paths: ["/api/*"]
```

## 安全检查清单

部署应用前，确认网络隔离策略：

```yaml
# 1. 检查现有 NetworkPolicy
kubectl get networkpolicies -A

# 2. 验证策略效果
kubectl run test --rm -it --image=busybox -- wget -qO- http://target-service

# 3. 审计所有 Allow 规则
kubectl describe networkpolicies -n production
```

:::tip 最佳实践总结
1. **默认拒绝**：每个 Namespace 创建默认拒绝 Ingress/Egress 策略
2. **明确放行**：按应用层级和 Namespace 逐个添加允许规则
3. **最小权限**：只开放必要的端口和目标
4. **DNS 优先**：确保 Cluster DNS 访问总是允许
5. **CNI 支持**：部署前确认 CNI 支持 NetworkPolicy
6. **持续监控**：使用工具监控网络流量异常
:::

## 总结

NetworkPolicy 是 K8s 网络安全的核心组件。正确的网络隔离策略可以：

- 防止横向移动攻击
- 实现多租户隔离
- 满足合规要求
- 减少攻击面

但 NetworkPolicy 不是银弹——它只解决 L3/L4 层的问题。对于更复杂的微服务安全（如 L7 策略、mTLS），需要结合服务网格使用。
