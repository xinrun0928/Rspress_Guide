# ServiceAccount：Pod 的身份标识

你有没有想过：Pod 访问 API Server 时，用的是什么身份？当你用 `kubectl exec` 进入容器时，容器内的进程如何证明「我是这个集群的一员」？

答案是 ServiceAccount。K8s 中的 Pod 和 ServiceAccount 的关系，就像 Linux 系统中的进程和用户的对应。每一个需要访问 API Server 的 Pod，都绑定一个 ServiceAccount。

## ServiceAccount 基础

### 什么是 ServiceAccount

ServiceAccount 是 K8s 中的身份凭证，用于 Pod 访问 API Server。每个 Namespace 都有一个默认的 ServiceAccount，名为 `default`。

```bash
kubectl get sa -n default
```

```
NAME      SECRETS   AGE
default   0         30d
```

:::tip Secret 去哪了？
K8s 1.22 之后，ServiceAccount Token 不再自动创建 Secret。当 Pod 挂载 ServiceAccount Token 时，K8s 使用 TokenRequest API 动态生成有限范围和时间的 Token。旧的自动创建 Secret 的方式已被废弃。
:::

### 默认 ServiceAccount 的问题

每个 Namespace 都有一个 `default` ServiceAccount，但它的 Token 默认不会挂载到 Pod（除非 Pod 没有显式指定 ServiceAccount）。这带来一个常见误解：

```yaml
# 这个 Pod 会挂载 default ServiceAccount 的 Token
apiVersion: v1
kind: Pod
metadata:
  name: app
spec:
  containers:
  - name: app
    image: nginx
  # 没有指定 serviceAccountName，默认使用 default
```

如果 Pod 需要访问 API Server（比如查询其他 Pod 的信息），应该创建专用的 ServiceAccount，而不是直接使用 `default`。

## 创建专用 ServiceAccount

### 基本创建

```yaml title="创建专用 ServiceAccount"
apiVersion: v1
kind: ServiceAccount
metadata:
  name: my-app-sa
  namespace: production
```

### 绑定 RBAC 权限

ServiceAccount 需要配合 RBAC 才能访问 API Server：

```yaml title="ServiceAccount + RoleBinding"
apiVersion: v1
kind: ServiceAccount
metadata:
  name: my-app-sa
  namespace: production
---
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: pod-reader
  namespace: production
rules:
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get", "list", "watch"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: pod-reader-binding
  namespace: production
subjects:
- kind: ServiceAccount
  name: my-app-sa
  namespace: production
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io
```

### 在 Pod 中使用

```yaml title="Pod 使用专用 ServiceAccount"
apiVersion: v1
kind: Pod
metadata:
  name: my-app
  namespace: production
spec:
  serviceAccountName: my-app-sa
  containers:
  - name: app
    image: nginx
```

## Token 挂载与 TokenRequest

### 自动挂载

默认情况下，Pod 挂载的 Token 会自动刷新：

```yaml
spec:
  serviceAccountName: my-app-sa
  automountServiceAccountToken: false  # 禁用自动挂载
  containers:
  - name: app
    image: nginx
```

:::tip 禁用自动挂载的场景
如果 Pod 不需要访问 API Server，明确设置 `automountServiceAccountToken: false`，减少 Token 泄露的风险。
:::

### 手动挂载 Projected Volume

K8s 1.20+ 支持手动挂载 ServiceAccount Token：

```yaml title="手动挂载 ServiceAccount Token"
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  serviceAccountName: my-app-sa
  containers:
  - name: app
    image: nginx
    volumeMounts:
    - name: token
      mountPath: /var/run/secrets/tokens
  volumes:
  - name: token
    projected:
      sources:
      - serviceAccountToken:
          audience: api
          expirationSeconds: 7200
          path: token
```

### TokenRequest API

K8s 1.22+ 使用 TokenRequest API 动态签发 Token，相比 Secret 更安全：

| 特性 | Secret 存储 Token | TokenRequest |
|---|---|---|
| 范围 | Namespace 级别 | 绑定到特定 Pod |
| 有效期 | 永不过期 | 可设置过期时间 |
| 刷新 | 手动 | 自动刷新 |
| 撤销 | 删除 Secret | 删除 Pod 后自动撤销 |

```yaml
spec:
  containers:
  - name: app
    image: nginx
  # 使用 TokenRequest，Token 自动挂载到 /var/run/secrets/tokens/
  # 不需要手动定义 volume
```

## ImagePullSecrets：私有镜像认证

如果你的镜像存储在私有仓库，需要配置 ImagePullSecrets：

### 创建 Secret

```bash
kubectl create secret docker-registry my-registry-secret \
  --docker-server=registry.example.com \
  --docker-username=admin \
  --docker-password=password \
  --docker-email=admin@example.com \
  -n production
```

### 关联到 ServiceAccount

```yaml title="关联 ImagePullSecrets"
apiVersion: v1
kind: ServiceAccount
metadata:
  name: my-app-sa
  namespace: production
imagePullSecrets:
- name: my-registry-secret
```

### 一次性关联多个 Namespace

如果多个 Namespace 使用同一私有镜像，可以通过 ServiceAccount 继承：

```yaml
# 在每个需要使用私有镜像的 Namespace 创建 ServiceAccount
apiVersion: v1
kind: ServiceAccount
metadata:
  name: image-pull-sa
  namespace: dev
imagePullSecrets:
- name: my-registry-secret
---
apiVersion: v1
kind: ServiceAccount
metadata:
  name: image-pull-sa
  namespace: staging
imagePullSecrets:
- name: my-registry-secret
---
apiVersion: v1
kind: ServiceAccount
metadata:
  name: image-pull-sa
  namespace: production
imagePullSecrets:
- name: my-registry-secret
```

:::tip 最佳实践
将镜像拉取凭证作为应用部署的一部分，与应用一起管理（Helm Chart 或 Kustomize）。这样在新环境中部署时，凭证会自动配置。
:::

## External Secrets 与 ServiceAccount

现代 K8s 安全实践建议使用外部密钥管理服务（External Secrets Operator）：

```yaml title="External Secrets 示例"
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: database-credentials
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: vault-backend
    kind: VaultStore
  target:
    name: database-credentials
    creationPolicy: Owner
  data:
  - secretKey: username
    remoteRef:
      key: production/database
      property: username
  - secretKey: password
    remoteRef:
      key: production/database
      property: password
```

External Secrets Operator 会自动将外部密钥同步为 K8s Secret，并绑定到指定的 ServiceAccount：

```yaml
spec:
  serviceAccount:
    name: my-app-sa
    kind: ServiceAccount
```

## RBAC 与 ServiceAccount 最佳实践

### 命名规范

```yaml
# ServiceAccount 命名：应用名 + -sa
name: my-application-sa

# Role 命名：应用名 + -reader / -writer / -admin
name: my-application-reader
name: my-application-writer
```

### 最小权限原则

```yaml
# 过度授权：危险！
rules:
- apiGroups: [""]
  resources: ["*"]
  verbs: ["*"]

# 精确授权：推荐
rules:
- apiGroups: [""]
  resources: ["configmaps"]
  resourceNames: ["my-app-config"]
  verbs: ["get", "watch"]
```

### 环境隔离

不同环境使用不同的 ServiceAccount：

```yaml
# development namespace
apiVersion: v1
kind: ServiceAccount
metadata:
  name: my-app-sa
  namespace: development

---
# production namespace
apiVersion: v1
kind: ServiceAccount
metadata:
  name: my-app-sa
  namespace: production
```

这样即使开发环境的 Token 泄露，也不会影响生产环境。

## 常见问题

### Q: ServiceAccount Token 泄露了怎么办？

1. 检查 Token 的使用范围（绑定的 RBAC 权限）
2. 如果使用了 TokenRequest，删除 Pod 后 Token 自动失效
3. 如果使用了 Secret，删除 Secret 并重建 Pod
4. 检查审计日志，确认是否有异常访问

### Q: 如何查看 Pod 使用的是哪个 ServiceAccount？

```bash
kubectl get pod <pod-name> -o jsonpath='{.spec.serviceAccountName}'
```

### Q: 为什么 Pod 无法挂载 Token？

检查几个可能的原因：
- ServiceAccount 是否存在
- `automountServiceAccountToken` 是否被显式禁用
- Pod 是否定义了冲突的 projected volume

## 总结

ServiceAccount 是 K8s 中 Pod 身份的载体，合理使用 ServiceAccount 需要：

- 为每个应用创建专用的 ServiceAccount
- 通过 RBAC 精确控制权限范围
- 禁用不需要访问 API Server 的 Pod 的 Token 挂载
- 使用 TokenRequest 而非 Secret 存储 Token
- 配置 ImagePullSecrets 以访问私有镜像仓库

ServiceAccount 与 RBAC 的结合，构成了 K8s 身份认证的核心机制。掌握这套机制，是编写安全 K8s 应用的基础。
