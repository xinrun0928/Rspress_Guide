# Helm 为什么是 K8s 的包管理器？Chart 是什么？

想象一下这个场景：你要部署一个完整的微服务架构，包含 10 个服务、配置中心、消息队列、数据库...每个服务有自己的 Deployment、Service、ConfigMap、Secret，还有各种 RBAC 资源。这需要多少 YAML？

如果没有工具，你需要手动编写和维护几十个文件，每次修改都要逐个改。更痛苦的是：测试环境、预发布环境、生产环境的配置各有不同，你不得不在每个环境复制一份，然后小心翼翼地修改差异。

Helm 就是解决这个问题的——**它是 K8s 的包管理器，让你用声明式的方式管理复杂应用**。

## Helm 核心概念

### Chart：应用的打包格式

Chart 是 Helm 的包格式，类似 apt 的 deb 包或 yum 的 rpm 包：

```
my-app/
├── Chart.yaml          # Chart 元信息
├── values.yaml         # 默认配置
├── values.schema.json  # 配置校验
├── templates/          # K8s 资源模板
│   ├── deployment.yaml
│   ├── service.yaml
│   ├── configmap.yaml
│   └── _helpers.tpl    # 辅助模板
├── charts/             # 依赖的子 Chart
└── crds/              # 自定义 CRD
```

### Chart.yaml：定义应用元信息

```yaml title="Chart.yaml"
apiVersion: v2  # Helm 3 格式
name: my-application
description: A Helm chart for my application
type: application
version: 1.2.3          # Chart 版本
appVersion: "2.0.0"      # 应用版本
keywords:
- web
- api
- microservices
home: https://example.com
sources:
- https://github.com/example/my-app
maintainers:
- name: DevOps Team
  email: devops@example.com
dependencies:
- name: postgresql
  version: "12.x.x"
  repository: https://charts.bitnami.com/bitnami
  condition: postgresql.enabled
- name: redis
  version: "17.x.x"
  repository: https://charts.bitnami.com/bitnami
  condition: redis.enabled
```

### values.yaml：默认配置

```yaml title="values.yaml"
# 应用配置
replicaCount: 2

image:
  repository: my-registry/my-app
  pullPolicy: IfNotPresent
  tag: "latest"

# 服务配置
service:
  type: ClusterIP
  port: 8080

# 资源限制
resources:
  limits:
    cpu: 1000m
    memory: 1Gi
  requests:
    cpu: 100m
    memory: 256Mi

# Ingress 配置
ingress:
  enabled: true
  className: nginx
  annotations:
    cert-manager.io/cluster-issuer: letsencrypt-prod
  hosts:
  - host: my-app.example.com
    paths:
    - path: /
      pathType: Prefix
  tls:
  - secretName: my-app-tls
    hosts:
    - my-app.example.com

# 环境变量
env:
  DATABASE_URL: ""
  LOG_LEVEL: "info"

# 配置
config:
  maxConnections: 100
  timeout: 30s

# 依赖服务
postgresql:
  enabled: true
  auth:
    database: myapp
    username: myapp
  primary:
    persistence:
      size: 10Gi
```

## Release：安装实例

### 什么是 Release

Release 是 Chart 的安装实例：

```bash
# 安装 Chart，创建 Release
helm install my-app ./my-app

# 查看 Release 列表
helm list

# 查看 Release 状态
helm status my-app

# 查看 Release 历史
helm history my-app
```

### Release 命名空间

```bash
# 在指定 Namespace 安装
helm install my-app ./my-app -n production --create-namespace

# 查看特定 Namespace 的 Release
helm list -n production
```

### Release 版本

Helm 保留 Release 的历史版本，支持回滚：

```bash
# 升级 Release
helm upgrade my-app ./my-app --set replicaCount=3

# 回滚到上一个版本
helm rollback my-app

# 回滚到指定版本
helm rollback my-app 3

# 查看历史版本
helm history my-app

# 输出示例
REVISION    UPDATED                  STATUS      CHART         DESCRIPTION
1           Mon Jan 15 10:30:00    superseded  my-app-1.0.0  Install complete
2           Mon Jan 15 11:00:00    superseded  my-app-1.1.0  Upgrade complete
3           Mon Jan 15 14:00:00    superseded  my-app-1.2.0  Upgrade complete
4           Tue Jan 16 09:00:00    deployed    my-app-1.3.0  Upgrade complete
```

## 常用 Helm 命令

### 安装与升级

```bash
# 从本地 Chart 安装
helm install my-app ./my-app

# 从远程仓库安装
helm install my-app bitnami/nginx

# 带自定义 values 文件
helm install my-app ./my-app -f values-production.yaml

# 带命令行参数
helm install my-app ./my-app \
  --set replicaCount=5 \
  --set image.tag=v2.0.0

# dry-run 模式（不实际执行）
helm install my-app ./my-app --dry-run --debug

# 升级（不存在则安装）
helm upgrade --install my-app ./my-app

# 原子操作（失败自动回滚）
helm upgrade --install my-app ./my-app --atomic
```

### 查询与调试

```bash
# 模板渲染（查看实际生成的资源）
helm template my-app ./my-app -f values.yaml

# 查看 Release 状态
helm status my-app

# 查看 Release 历史
helm history my-app

# 获取 Release 值
helm get values my-app

# 获取 Release 所有信息
helm get all my-app

# 列出所有 Release
helm list --all --namespace production

# 搜索 Chart
helm search repo nginx
helm search hub nginx
```

### 卸载与清理

```bash
# 卸载 Release
helm uninstall my-app

# 卸载并保留历史
helm uninstall my-app --keep-history

# 清理不再需要的 Release
helm repo update
helm repo prune
```

## Chart 依赖管理

### 声明依赖

在 Chart.yaml 中声明依赖：

```yaml
dependencies:
- name: postgresql
  version: "12.x.x"
  repository: "https://charts.bitnami.com/bitnami"
- name: redis
  version: "17.x.x"
  repository: "https://charts.bitnami.com/bitnami"
```

### 下载依赖

```bash
# 下载依赖到 charts/ 目录
helm dependency build ./my-app

# 更新依赖
helm dependency update ./my-app

# 查看依赖状态
helm dependency list ./my-app
```

### 子 Chart 覆盖

```yaml title="values-production.yaml"
# 覆盖子 Chart 的值
postgresql:
  primary:
    persistence:
      size: 50Gi
  auth:
    database: myapp_prod

redis:
  master:
    persistence:
      size: 10Gi
```

## 模板渲染

### 本地渲染

```bash
# 渲染模板但不安装
helm template my-app ./my-app

# 渲染特定 Namespace
helm template my-app ./my-app --namespace production

# 渲染带 values
helm template my-app ./my-app -f values-production.yaml
```

### 查看渲染结果

```bash
# 渲染单个模板
helm template my-app ./my-app | grep -A 50 "kind: Deployment"

# 输出到文件
helm template my-app ./my-app > rendered.yaml
```

## 命名模板（Partial）

### _helpers.tpl

```yaml title="templates/_helpers.tpl"
{{/* 完整名称 */}}
{{- define "my-app.name" -}}
{{- default .Chart.Name .Values.nameOverride | trunc 63 | trimSuffix "-" }}
{{- end }}

{{/* 全限定名称 */}}
{{- define "my-app.fullname" -}}
{{- $name := default .Chart.Name .Values.nameOverride }}
{{- if .Values.fullnameOverride }}
{{- .Values.fullnameOverride | trunc 63 | trimSuffix "-" }}
{{- else }}
{{- printf "%s-%s" .Release.Name $name | trunc 63 | trimSuffix "-" }}
{{- end }}
{{- end }}

{{/* 标签 */}}
{{- define "my-app.labels" -}}
app.kubernetes.io/name: {{ include "my-app.name" . }}
app.kubernetes.io/instance: {{ .Release.Name }}
app.kubernetes.io/version: {{ .Chart.AppVersion }}
app.kubernetes.io/managed-by: {{ .Release.Service }}
{{- end }}

{{/* 选择器 */}}
{{- define "my-app.selectorLabels" -}}
app.kubernetes.io/name: {{ include "my-app.name" . }}
app.kubernetes.io/instance: {{ .Release.Name }}
{{- end }}
```

### 使用命名模板

```yaml title="templates/deployment.yaml"
apiVersion: apps/v1
kind: Deployment
metadata:
  name: {{ include "my-app.fullname" . }}
  labels:
    {{- include "my-app.labels" . | nindent 4 }}
spec:
  replicas: {{ .Values.replicaCount }}
  selector:
    matchLabels:
      {{- include "my-app.selectorLabels" . | nindent 6 }}
```

### NOTES.txt

```yaml title="templates/NOTES.txt"
感谢安装 {{ .Chart.Name }}。

应用程序已成功部署。

获取应用访问地址：

{{- if .Values.ingress.enabled }}
Web 访问地址：https://{{ .Values.ingress.hosts[0].host }}
{{- else if contains "NodePort" .Values.service.type }}
执行以下命令获取 Node Port：
  export NODE_PORT=$(kubectl get --namespace {{ .Release.Namespace }} -o jsonpath="{.spec.ports[0].nodePort}" services {{ include "my-app.fullname" . }})
  echo "访问地址：http://${NODE_IP}:${NODE_PORT}"
{{- else if contains "LoadBalancer" .Values.service.type }}
等待 LoadBalancer 配置完成：
  kubectl get svc --namespace {{ .Release.Namespace }} -w {{ include "my-app.fullname" . }}
{{- end }}

查看 Release 状态：
  helm status {{ .Release.Name }}

升级 Release：
  helm upgrade {{ .Release.Name }} {{ .Chart.Name }} --set {{ .Values.image.tag }}
```

## 面试追问方向

1. **Helm 和 kubectl 的区别？**
   - Helm 是包管理器，支持版本管理和回滚
   - kubectl 是 K8s CLI，直接操作资源

2. **Helm 如何实现回滚？**
   - 保存每次 install/upgrade 的 Release 历史
   - 记录每次安装的资源清单
   - 回滚时重新应用历史版本

3. **Helm 3 和 Helm 2 的区别？**
   - Helm 3 移除了 Tiller
   - 依赖管理改为 Chart.yaml
   - 移除了 homebrew 仓库

4. **如何安全管理 Helm Release？**
   - 使用 sealed-secrets 加密敏感值
   - Helm secrets 插件
   - External Secrets Operator

## 总结

Helm 是 K8s 生态的核心工具：

- **Chart**：应用的标准化打包格式
- **Release**：Chart 的安装实例
- **Repository**：Chart 的分发渠道
- **依赖管理**：子 Chart 的声明和覆盖
- **版本控制**：Release 历史的跟踪和回滚

掌握 Helm，你就能高效管理复杂应用的部署。
