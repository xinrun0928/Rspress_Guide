# 服务网格架构

你有没有想过——当微服务数量超过几十个时，服务间的通信、可观测性、安全和流量管理会变得多么复杂？每个服务都要自己实现熔断、重试、超时、mTLS……这些逻辑会变成一团乱麻。

服务网格（Service Mesh）就是为了解决这个问题。它将服务间通信的基础设施层抽离出来，作为独立的一层来管理，让开发人员专注于业务逻辑。

## 服务网格是什么

服务网格是一个专用基础设施层，用于处理服务间通信，并提供连接、控制和观测微服务的功能。

```
Without Service Mesh:
┌─────────────────────────────────────────────────────────────┐
│                      你的应用代码                             │
│                                                              │
│  ┌─────────┐   ┌─────────┐   ┌─────────┐                 │
│  │ Service │◄──│ Service │◄──│ Service │                 │
│  │    A    │──►│    B    │──►│    C    │                 │
│  └────┬────┘   └────┬────┘   └────┬────┘                 │
│       │             │             │                       │
│   ┌───┴─────────────┴─────────────┴───┐                  │
│   │ 熔断、重试、超时、mTLS、追踪...      │ ← 每个服务都要实现 │
│   └───────────────────────────────────┘                  │
└─────────────────────────────────────────────────────────────┘

With Service Mesh:
┌─────────────────────────────────────────────────────────────┐
│                      服务网格层 (Data Plane)                  │
│                                                              │
│  ┌────┐     ┌────┐     ┌────┐                            │
│  │Side│     │Side│     │Side│                            │
│  │ car│     │car │     │car │                            │
│  │ A  │     │ B  │     │ C  │  ← 每 Pod 一个 Sidecar     │
│  └──┬─┘     └──┬─┘     └──┬─┘                            │
│     │           │           │                               │
│  ┌──┴───────────┴───────────┴──┐                          │
│  │  熔断、重试、超时、mTLS、追踪  │ ← Sidecar 自动处理      │
│  └─────────────────────────────┘                          │
└─────────────────────────────────────────────────────────────┘
                              ▲
                              │ Control Plane
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                    控制平面 (Control Plane)                   │
│                                                              │
│  ┌────────────┐  ┌────────────┐  ┌────────────┐            │
│  │  Config    │  │  Policy   │  │  Telemetry │            │
│  │  Server    │  │  Store    │  │  Collector │            │
│  └────────────┘  └────────────┘  └────────────┘            │
└─────────────────────────────────────────────────────────────┘
```

## 核心概念

### Sidecar 模式

Sidecar 是与服务容器部署在同一个 Pod 中的代理容器，负责所有出入流量。

```
┌──────────────────────────────────────────┐
│                    Pod                     │
│                                          │
│  ┌────────────────────────────────┐    │
│  │        Service Container        │    │
│  │        (你的应用程序)            │    │
│  │                                  │    │
│  │    应用代码 ──► 业务逻辑         │    │
│  └────────────────────────────────┘    │
│                                          │
│  ┌────────────────────────────────┐    │
│  │        Sidecar Proxy            │    │
│  │        (Envoy)                  │    │
│  │                                  │    │
│  │  熔断、重试、超时、mTLS、观测   │    │
│  └────────────────────────────────┘    │
│                                          │
│  localhost                               │
└──────────────────────────────────────────┘
```

### Data Plane vs Control Plane

| 组件 | 职责 | 组件示例 |
|---|---|---|
| **Data Plane** | 处理数据流量，负责流量转发和策略执行 | Envoy、nginx |
| **Control Plane** | 管理配置下发，收集遥测数据 | Istiod、Linkerd Controller |

### 安装 Istio

```bash
# 下载 Istio
curl -L https://istio.io/downloadIstio | sh -
cd istio-1.20.0
export PATH=$PWD/bin:$PATH

# 使用 demo 配置安装
istioctl install --set profile=demo

# 启用自动 Sidecar 注入
kubectl label namespace default istio-injection=enabled

# 部署应用
kubectl apply -f samples/bookinfo/platform/kube/bookinfo.yaml
```

## Linkerd：轻量级的选择

Linkerd 以简单和轻量著称，由 Buoyant 开发，是 CNCF 毕业项目。

### 特点

- **极简配置**：开箱即用，不需要复杂的配置
- **超低延迟**：Sidecar 延迟 < 1ms
- **Rust 编写**：高性能、低内存占用
- **安全优先**：默认开启 mTLS

### 安装 Linkerd

```bash
# 安装 Linkerd CLI
curl --proto '=https' --tlsv1.2 -sSfL https://run.linkerd.io/install | sh

# 安装 Linkerd 到集群
linkerd install | kubectl apply -f -

# 安装 Linkerd Viz（可视化组件）
linkerd viz install | kubectl apply -f -
```

## Envoy：数据平面的核心

无论是 Istio 还是 Linkerd，底层都使用 **Envoy** 作为 Sidecar 代理。

### Envoy 的核心能力

| 能力 | 说明 |
|---|---|
| **动态服务发现** | 支持 EDS、CDS、RDS、LDS |
| **负载均衡** | 轮询、随机、最少连接、加权等 |
| **熔断器** | 自动隔离故障节点 |
| **重试和超时** | 配置自动重试策略 |
| **速率限制** | 本地和全局限流 |
| **可观测性** | 支持 metrics、traces、logs |
| **mTLS** | 自动服务身份验证和加密 |

### Envoy 静态配置

```yaml title="envoy-static.yaml"
static_resources:
  listeners:
  - name: listener_0
    address:
      socket_address:
        address: 0.0.0.0
        port_value: 80
    filter_chains:
    - filters:
      - name: envoy.filters.network.http_connection_manager
        typed_config:
          "@type": type.googleapis.com/envoy.extensions.filters.network.http_connection_manager.v3.HttpConnectionManager
          route_config:
            name: local_route
            virtual_hosts:
            - name: service
              domains: ["*"]
              routes:
              - match: { prefix: "/" }
                route:
                  cluster: my_service
          http_filters:
          - name: envoy.filters.http.router
  clusters:
  - name: my_service
    connect_timeout: 5s
    type: STRICT_DNS
    lb_policy: ROUND_ROBIN
    load_assignment:
      cluster_name: my_service
      endpoints:
      - lb_endpoints:
        - endpoint:
            address:
              socket_address:
                address: my-service
                port_value: 8080
```

## 流量管理

### VirtualService：路由规则

```yaml title="virtualservice.yaml"
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: reviews
spec:
  hosts:
  - reviews
  http:
  # 金丝雀发布：5% 流量到 v2 版本
  - route:
    - destination:
        host: reviews
        subset: v1
      weight: 95
    - destination:
        host: reviews
        subset: v2
      weight: 5
  # 基于 Header 的路由
  - match:
    - headers:
        X-Canary:
          exact: "true"
    route:
    - destination:
        host: reviews
        subset: v2
  # 基于用户身份的高级路由
  - match:
    - headers:
        end-user:
          regex: ".*"
    route:
    - destination:
        host: reviews
        subset: premium
```

### DestinationRule：目的地策略

```yaml title="destinationrule.yaml"
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: reviews
spec:
  host: reviews
  trafficPolicy:
    connectionPool:
      tcp:
        maxConnections: 100
      http:
        h2UpgradePolicy: UPGRADE
        http1MaxPendingRequests: 100
        http2MaxRequests: 1000
    loadBalancer:
      simple: LEAST_REQUEST
      localityLbSetting:
        enabled: true
    outlierDetection:
      consecutiveGatewayErrors: 5
      interval: 30s
      baseEjectionTime: 30s
  subsets:
  - name: v1
    labels:
      version: v1
  - name: v2
    labels:
      version: v2
```

### 流量镜像

```yaml
# 将 5% 流量镜像到 v2 版本，用于预生产测试
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: httpbin
spec:
  hosts:
  - httpbin
  http:
  - route:
    - destination:
        host: httpbin
        subset: v1
    mirror:
      host: httpbin
      subset: v2
    mirrorPercentage:
      value: 5.0
```

## 安全

### 启用 mTLS

```yaml title="peerauthentication.yaml"
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: default
  namespace: istio-system
spec:
  mtls:
    mode: STRICT  # STRICT = 强制 mTLS，PERMISSIVE = 允许混合
```

### 授权策略

```yaml title="authorization.yaml"
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: frontend-ingress
  namespace: default
spec:
  selector:
    matchLabels:
      app: frontend
  action: ALLOW
  rules:
  # 允许来自 Ingress Gateway 的流量
  - from:
    - source:
        principals: ["cluster.local/ns/default/sa/istio-ingressgateway"]
    when:
    - key: destination.port
      values: ["8080"]
  # 只允许 GET 请求
  - to:
    - operation:
        methods: ["GET"]
```

## 可观测性

### 指标（Metrics）

```bash
# 查看 Prometheus 收集的指标
kubectl exec -it $POD -- curl localhost:15000/stats

# Istio 默认指标：
# - request_duration：请求延迟
# - requests_total：请求总数
# - response_size：响应大小
# - bytes_sent：发送字节数
```

### 分布式追踪（Tracing）

```yaml
# 启用追踪采样
apiVersion: install.istio.io/v1alpha1
kind: IstioOperator
metadata:
  name: istio-config
spec:
  meshConfig:
    enableTracing: true
    defaultConfig:
      tracing:
        sampling: 10.0  # 10% 采样率
        zipkin:
          address: jaeger-collector.observability:9411
```

### 日志

```bash
# 查看 Sidecar 日志
kubectl logs $POD -c istio-proxy

# 启用访问日志
apiVersion: install.istio.io/v1alpha1
kind: IstioOperator
metadata:
  name: istio-config
spec:
  meshConfig:
    accessLogFile: /dev/stdout
```

## Istio vs Linkerd vs Envoy

| 维度 | Istio | Linkerd | Envoy |
|---|---|---|---|
| **架构复杂度** | 高 | 低 | 中 |
| **学习曲线** | 陡峭 | 平缓 | 中等 |
| **资源占用** | 较高 | 最低 | 中等 |
| **功能丰富度** | 最丰富 | 基础但够用 | 丰富 |
| **XDS 协议** | 原生支持 | 不支持 | 原生 |
| **数据面** | Envoy | 自研（Rust） | Envoy |
| **Kubernetes 原生** | 是 | 是 | 是 |
| **适用场景** | 大型企业、复杂需求 | 追求简单稳定 | 自建服务网格 |

## 总结

服务网格将分布式系统的通信基础设施抽离出来，让开发人员专注于业务逻辑：

1. **Sidecar 模式**：透明处理服务间通信
2. **流量管理**：灰度发布、流量镜像、熔断
3. **安全**：mTLS 加密、细粒度授权
4. **可观测性**：统一的 metrics、traces、logs
5. **Istio vs Linkerd**：Istio 功能全但复杂，Linkerd 简单稳定

选择服务网格时，需要在功能丰富度和运维复杂度之间权衡。
