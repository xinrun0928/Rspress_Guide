# Service Mesh 服务网格

你的微服务架构有 10 个服务，每个服务都需要实现限流、熔断、重试、超时、链路追踪、mTLS 认证……

你打算在每个服务里都加上这些逻辑吗？然后每次升级 SDK 都要改 10 个服务？当有一天限流策略需要调整，你愿意逐个服务修改吗？

Service Mesh 正是为了解决这个问题。它将微服务的「非业务逻辑」从应用代码中剥离出来，放到一个独立的「基础设施层」来处理。

## 传统微服务的问题

在 Service Mesh 出现之前，开发者需要在每个服务中实现以下功能：

```
┌─────────────────────────────────────────────────────────────┐
│                      服务 A                                  │
│  ┌─────────────────────────────────────────────────────┐    │
│  │                   业务代码                            │    │
│  │                   (真正的价值)                        │    │
│  └─────────────────────────────────────────────────────┘    │
│  ┌─────────────────────────────────────────────────────┐    │
│  │ 限流 │ 熔断 │ 重试 │ 超时 │ mTLS │ 链路追踪 │ ...  │    │
│  │                   SDK / 框架                          │    │
│  └─────────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────┘
```

### 痛点

| 问题 | 说明 |
|---|---|
| 代码耦合 | 非业务逻辑侵入业务代码 |
| 升级困难 | SDK 升级需要改所有服务 |
| 语言绑定 | 不同语言需要不同实现 |
| 配置分散 | 每个服务单独配置 |
| 重复开发 | 每个团队重复实现相同功能 |

## Service Mesh 的解决思路

```
┌─────────────────────────────────────────────────────────────┐
│                      服务 A                                  │
│  ┌─────────────────────────────────────────────────────┐    │
│  │                   业务代码                            │    │
│  │                   (真正的价值)                        │    │
│  └─────────────────────────────────────────────────────┘    │
│  ┌─────────────────────────────────────────────────────┐    │
│  │                   Data Plane                         │    │
│  │                   (Sidecar Proxy)                    │    │
│  │                   Envoy / Linkerd                    │    │
│  └─────────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────┘
                           │
                           │ 控制平面 API
                           ▼
┌─────────────────────────────────────────────────────────────┐
│                      Control Plane                           │
│  ┌─────────────────────────────────────────────────────┐    │
│  │  Pilot / Citadel / Galley (Istio)                  │    │
│  │  or Control Plane (Linkerd)                        │    │
│  │                                                       │    │
│  │  策略配置 │ 证书管理 │ 服务发现 │ 遥测收集            │    │
│  └─────────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────┘
```

**核心思想**：将网络、安全、可观测性等「横切关注点」从应用中分离，放到独立的代理层（Sidecar）。

## Sidecar 模式

Sidecar 是附加在主容器旁边的代理容器：

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: order-service
spec:
  containers:
  - name: order-service
    image: order-service:v1
    ports:
    - containerPort: 8080
  - name: envoy-sidecar
    image: envoyproxy/envoy:v1.28
    ports:
    - containerPort: 15001
    resources:
      limits:
        memory: 128Mi
        cpu: 100m
```

### Sidecar 的职责

Sidecar 代理接管所有进出流量：

| 职责 | 功能 |
|---|---|
| 流量管理 | 路由、负载均衡、熔断 |
| 安全 | mTLS 加密、认证授权 |
| 可观测性 | 指标采集、链路追踪、日志 |
| 弹性 | 重试、超时、限流 |

## Istio 架构

Istio 是最流行的 Service Mesh 实现：

```
┌─────────────────────────────────────────────────────────────┐
│                         Istio                               │
│                                                              │
│  ┌──────────────────────────────────────────────────────┐  │
│  │                    Control Plane                      │  │
│  │  ┌─────────┐ ┌─────────┐ ┌─────────┐               │  │
│  │  │  Pilot  │ │ Citadel │ │ Galley  │               │  │
│  │  │(流量管理)│ │(安全管理)│ │(配置验证)│               │  │
│  │  └─────────┘ └─────────┘ └─────────┘               │  │
│  └──────────────────────────────────────────────────────┘  │
│                           │                                 │
│                           ▼                                 │
│  ┌──────────────────────────────────────────────────────┐  │
│  │                    Data Plane                         │  │
│  │  ┌───────┐ ┌───────┐ ┌───────┐ ┌───────┐            │  │
│  │  │Envoy  │ │Envoy  │ │Envoy  │ │Envoy  │            │  │
│  │  │ Pod A │ │ Pod B │ │ Pod C │ │ Pod D │            │  │
│  │  └───────┘ └───────┘ └───────┘ └───────┘            │  │
│  └──────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

### 组件详解

| 组件 | 职责 |
|---|---|
| **Pilot** | 管理流量规则（VirtualService、DestinationRule） |
| **Citadel** | 证书签发和管理，mTLS 支持 |
| **Galley** | 验证和获取配置 |
| **Envoy** | 数据平面 Sidecar 代理 |

### Istio CRD

```yaml
# VirtualService：定义路由规则
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: order-service
spec:
  hosts:
  - order-service
  http:
  - match:
    - headers:
        X-Canary:
          exact: "true"
    route:
    - destination:
        host: order-service
        subset: v2
      weight: 100
  - route:
    - destination:
        host: order-service
        subset: v1
      weight: 100
```

```yaml
# DestinationRule：定义 subsets 和负载均衡
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: order-service
spec:
  host: order-service
  trafficPolicy:
    tls:
      mode: ISTIO_MUTUAL
  subsets:
  - name: v1
    labels:
      version: v1
  - name: v2
    labels:
      version: v2
```

## Linkerd 架构

Linkerd 是另一个流行的 Service Mesh，以简单和高性能著称：

```
┌─────────────────────────────────────────────────────────────┐
│                         Linkerd                             │
│                                                              │
│  ┌──────────────────────────────────────────────────────┐  │
│  │  Control Plane (Rust)                                 │  │
│  │  ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐    │  │
│  │  │ Controller│ │   DNS   │ │   HA    │ │ Prometheus│   │  │
│  │  └─────────┘ └─────────┘ └─────────┘ └─────────┘    │  │
│  └──────────────────────────────────────────────────────┘  │
│                           │                                 │
│                           ▼                                 │
│  ┌──────────────────────────────────────────────────────┐  │
│  │  Data Plane (Rust proxy)                              │  │
│  │  ┌───────┐ ┌───────┐ ┌───────┐ ┌───────┐            │  │
│  │  │ proxy │ │ proxy │ │ proxy │ │ proxy │            │  │
│  │  │ Pod A │ │ Pod B │ │ Pod C │ │ Pod D │            │  │
│  │  └───────┘ └───────┘ └───────┘ └───────┘            │  │
│  └──────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

### Linkerd vs Istio

| 特性 | Istio | Linkerd |
|---|---|---|
| 复杂度 | 高 | 低 |
| 性能 | 中等 | 高（Rust 实现） |
| 配置方式 | YAML CRD | YAML CRD |
| 治理能力 | 丰富 | 基础 |
| 学习曲线 | 陡峭 | 平缓 |
| 多租户 | 支持 | 有限 |

## Service Mesh 的能力

### 流量管理

```yaml
# 金丝雀发布：10% 流量到新版本
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: order-service
spec:
  hosts:
  - order-service
  http:
  - route:
    - destination:
        host: order-service
        subset: v1
      weight: 90
    - destination:
        host: order-service
        subset: v2
      weight: 10
```

```yaml
# 熔断配置
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: order-service
spec:
  host: order-service
  trafficPolicy:
    connectionPool:
      tcp:
        maxConnections: 100
      http:
        h2UpgradePolicy: UPGRADE
    outlierDetection:
      consecutiveGatewayErrors: 5
      interval: 30s
      baseEjectionTime: 30s
```

### 安全

```yaml
# 启用 mTLS
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: default
  namespace: istio-system
spec:
  mtls:
    mode: STRICT
```

### 可观测性

Service Mesh 自动采集以下遥测数据：

| 类型 | 说明 | 工具 |
|---|---|---|
| 指标 | 请求率、延迟、错误率 | Prometheus |
| 日志 | 访问日志 | Loki / ELK |
| 追踪 | 分布式链路追踪 | Jaeger / Zipkin |

## 是否需要 Service Mesh

| 场景 | 推荐 |
|---|---|
| 少量服务（< 10） | 可能不需要 |
| 多语言微服务 | 推荐 |
| 强安全要求（零信任） | 必须 |
| 高级流量管理需求 | 推荐 |
| 团队规模小 | 可能过度复杂 |

:::warning 复杂度考量
Service Mesh 引入了显著的基础设施复杂度。在决定使用前，评估：
- 团队是否有能力维护
- 收益是否大于成本
- 是否有其他更简单的方案
:::

## 总结

Service Mesh 将微服务的横切关注点从应用中分离：

- **Sidecar 模式**：透明代理拦截所有流量
- **Control Plane**：集中管理配置和策略
- **Istio**：功能丰富，复杂度高
- **Linkerd**：简单轻量，性能优秀

是否使用 Service Mesh，需要根据团队规模和业务需求权衡。
