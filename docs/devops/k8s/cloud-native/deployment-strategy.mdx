# K8s 部署策略对比

你有没有想过——新版本上线时，如何才能既保证用户不受影响，又能验证新版本的正确性？

传统的「停机部署」已经不能满足现代互联网的需求了。用户期望 7x24 小时可用的服务，发布必须在不影响用户体验的前提下进行。这就是部署策略存在的意义。

## 部署策略概述

### 为什么需要部署策略

```
问题场景：
1. 新版本有 bug，导致服务不可用
2. 用户正在使用，系统崩溃
3. 回滚需要手动操作，时间长
4. 用户感知到服务中断

结果：用户投诉，口碑受损
```

正确的部署策略可以：

- 减少发布风险（小范围验证）
- 缩短回滚时间（自动回滚）
- 提升用户体验（无感知发布）

---

## 滚动部署（Rolling Update）

### 原理

滚动部署是最常见的部署方式，通过逐步替换旧版本实例来实现新版本上线。

```
初始状态：nginx:v1 [Pod-1] [Pod-2] [Pod-3]
          ↓
Step 1：启动 1 个 nginx:v2
        [Pod-1-v1] [Pod-2-v1] [Pod-3-v1] [Pod-4-v2]
          ↓
Step 2：停止 1 个旧 Pod
        [Pod-1-v1] [Pod-2-v1] [Pod-4-v2]
          ↓
Step 3：启动 1 个 nginx:v2
        [Pod-1-v1] [Pod-2-v1] [Pod-4-v2] [Pod-5-v2]
          ↓
Step 4：重复直到全部更新
        [Pod-6-v2] [Pod-7-v2] [Pod-8-v2]
```

### Kubernetes 实现

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
spec:
  replicas: 4
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1        # 最多超出 1 个 Pod
      maxUnavailable: 0  # 不允许 Pod 不可用
  template:
    ...
```

### 优缺点

| 优点 | 缺点 |
|---|---|
| 无 downtime（零停机） | 旧版本和新版本同时运行，资源占用翻倍 |
| 回滚相对容易 | 旧版本和新版本同时服务，灰度不可控 |
| 简单易实现 | 部分请求可能路由到新版本，无法精确控制流量 |

---

## 蓝绿部署（Blue-Green Deployment）

### 原理

蓝绿部署维护两套完整环境：新版本部署在「绿」环境，当前生产流量在「蓝」环境。更新时，流量一次性切换。

```
蓝色环境（当前版本 v1）    绿色环境（新版本 v2）
┌─────────────────┐     ┌─────────────────┐
│   Service A     │     │   Service A     │
│   [Pod-1]      │     │   [Pod-1]       │
│   [Pod-2]      │     │   [Pod-2]       │
│   [Pod-3]      │     │   [Pod-3]       │
└─────────────────┘     └─────────────────┘
        │                       ↑
        │ 流量指向              │ 切换后
        ↓                       │
   生产流量                  测试通过后
                           一次性切换
```

### Kubernetes 实现

```bash
# 1. 部署蓝色环境（当前版本）
kubectl apply -f deployment-blue.yaml

# 2. 部署绿色环境（新版本）
kubectl apply -f deployment-green.yaml

# 3. 验证绿色环境
kubectl rollout status deployment/myapp-green

# 4. 切换流量（通过修改 Service selector）
kubectl patch service myapp -p '{"spec":{"selector":{"version":"green"}}}'
```

### 优缺点

| 优点 | 缺点 |
|---|---|
| 切换即时，回滚秒级 | 需要两倍资源成本 |
| 新版本经过完整验证后才切换 | 切换瞬间，所有流量进入新版本，风险集中 |
| 易于测试，可以在绿环境完全验证 | 浪费资源（蓝环境在切流前一直空跑） |

### 使用场景

蓝绿部署适用于：

- 版本不兼容，必须整体切换的场景
- 需要完整测试验证后上线的场景
- 数据库 schema 变更需要原子切换的场景

## 金丝雀发布（Canary Release）

### 原理

金丝雀发布将小部分流量导向新版本进行验证，同时保持大部分流量在稳定版本。与蓝绿部署不同，金丝雀在运行时动态调整流量比例。

```
所有用户
    │
    │ 90% → 旧版本 v1
    │ 10% → 新版本 v2（先遣部队）
    │
    │ 验证通过后
    │
    │ 50% → 旧版本 v1
    │ 50% → 新版本 v2（扩大验证）
    │
    │ 最终确认
    │
    │ 100% → 新版本 v2
```

### Kubernetes 实现

```yaml title="canary-deployment.yaml"
apiVersion: argoproj.io/v1alpha1
kind: Rollout
metadata:
  name: myapp
spec:
  replicas: 10
  strategy:
    canary:
      steps:
      - setWeight: 10     # 先引 10% 流量
      - pause: {}         # 暂停，等待人工确认
      - setWeight: 30     # 扩大 30%
      - pause: {duration: 10m}  # 等待 10 分钟
      - setWeight: 50     # 扩大 50%
      - pause: {}         # 暂停
      - setWeight: 100    # 全量
      canaryMetadata:
        labels:
          track: canary
      stableMetadata:
        labels:
          track: stable
      trafficRouting:
        nginx:
          stableIngress: myapp-stable
          additionalIngressAnnotations:
            canary-weight: "10"
      analysis:
        templates:
        - templateName: success-rate
        args:
        - name: service-name
          value: myapp-canary
```

### 优缺点

| 优点 | 缺点 |
|---|---|
| 风险可控，灰度可调 | 实现复杂度较高 |
| 支持渐进式流量增长 | 需要完善的监控基础设施 |
| 可基于指标自动回滚 | 运维人员需要理解金丝雀概念 |

### 使用场景

- 新功能需要真实流量验证
- 需要精细控制灰度比例
- 希望基于业务指标自动回滚

## 灰度发布（A/B Testing）

### 原理

灰度发布通过特定规则（如 Header、Cookie、用户群体）将用户分流，不同用户群看到不同版本。

```
请求 ──► 路由规则 ──►
                    ├── 满足条件 A ──► 版本 A
                    └── 满足条件 B ──► 版本 B
```

### 路由规则示例

```yaml title="istio-virtualservice.yaml"
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: myapp
spec:
  hosts:
  - myapp
  http:
  # 基于 Header 的路由
  - match:
    - headers:
        X-User-Type:
          exact: premium
    route:
    - destination:
        host: myapp
        subset: v2
  # 基于 Cookie 的路由
  - match:
    - headers:
        cookie:
          regex: "^(.*?;)?(user=premium)(;.*)?$"
    route:
    - destination:
        host: myapp
        subset: v2
  # 默认路由
  - route:
    - destination:
        host: myapp
        subset: v1
```

### 与金丝雀的区别

| 维度 | 金丝雀发布 | 灰度发布 |
|---|---|---|
| **分流依据** | 流量比例 | 用户特征（Header/Cookie） |
| **目的** | 降低发布风险 | 精细化运营/实验 |
| **适用** | 技术验证 | 产品功能对比 |

---

## 渐进式交付对比

| 维度 | 滚动部署 | 蓝绿部署 | 金丝雀发布 | 灰度发布 |
|---|---|---|---|---|
| **资源成本** | 1x | 2x | 1.2x | 1.2x |
| **切换速度** | 渐进 | 即时 | 渐进 | 渐进 |
| **回滚速度** | 较慢 | 秒级 | 自动 | 自动 |
| **灰度控制** | 粗粒度 | 不可控 | 精细 | 精细 |
| **复杂度** | 低 | 中 | 高 | 高 |
| **适用场景** | 一般发布 | 整体切换 | 降低风险 | 用户实验 |

## 选择指南

### 选滚动部署如果：

- 发布频率高，不需要精细控制
- 资源有限，无法维护多套环境
- 新版本兼容旧版本

### 选蓝绿部署如果：

- 需要完整测试后才切换
- 版本必须整体切换
- 回滚必须秒级完成

### 选金丝雀发布如果：

- 需要基于真实流量验证
- 期望自动回滚能力
- 业务对稳定性要求高

### 选灰度发布如果：

- 需要按用户群区分功能
- 需要做 A/B 测试
- 需要精细化的流量控制

## 工具推荐

| 工具 | 支持的部署策略 | 特点 |
|---|---|---|
| **Argo Rollouts** | 金丝雀、蓝绿、滚动 | K8s 原生，功能强大 |
| **Flagger** | 金丝雀、蓝绿、A/B | 支持多 CNI/Service Mesh |
| **Istio** | 金丝雀、A/B | 服务网格内置 |
| **Nginx Ingress** | 金丝雀、灰度 | 基于权重的流量控制 |

---

## 总结

部署策略的选择需要综合考虑：

1. **业务需求**：是降风险还是做实验
2. **资源成本**：能承受多少额外资源
3. **团队能力**：能否维护复杂的部署系统
4. **回滚要求**：出现问题时回滚速度要求

没有最好的策略，只有最适合的策略。根据实际场景灵活选择，才能在快速迭代和系统稳定之间找到平衡。
