# 部署策略实战

新版本上线时，你面临一个艰难的选择：直接更新所有实例，万一出问题回滚慢；分批更新，万一新旧版本共存导致数据不一致怎么办？

不同的部署策略解决不同的问题。滚动更新、蓝绿部署、金丝雀发布——每种策略都有其适用场景。

## 部署策略对比

| 策略 | 停机时间 | 回滚速度 | 资源需求 | 风险控制 |
|---|---|---|---|---|
| 滚动更新 | 无 | 慢 | 低 | 中等 |
| 蓝绿部署 | 无 | 秒级 | 双倍 | 优秀 |
| 金丝雀发布 | 无 | 快 | 低 | 优秀 |
| A/B 测试 | 无 | 快 | 低 | 优秀 |

## 滚动更新

滚动更新是 Deployment 的默认策略，逐步替换旧版本：

```yaml
spec:
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1        # 最多超出 1 个 Pod
      maxUnavailable: 0  # 不允许 Pod 不可用
```

### 更新流程

```
初始状态: [v1] [v1] [v1]  (3 个 v1 Pod)

步骤 1: [v1] [v1] [v1] [v2]  (启动 v2)
步骤 2: [v1] [v1] [v2] [v2]  (停止 v1)
步骤 3: [v1] [v2] [v2] [v2]  (停止 v1)
步骤 4: [v2] [v2] [v2]       (最终状态)
```

### 适用场景

- 无状态服务
- 资源有限
- 风险可控的更新
- 持续部署

### 局限性

- 回滚时间较长（需要重新替换）
- 短时间内存在多个版本
- 无法精确控制流量分配

## 蓝绿部署

蓝绿部署保持两套完整环境，通过切换流量实现部署：

```
┌─────────────────────────────────────────────────────────────┐
│                   Blue (当前生产)                           │
│  ┌─────────────────────────────────────────────────────┐    │
│  │  Service ──▶ Blue Deployment (v1)                   │    │
│  │  replicas: 3                                        │    │
│  └─────────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                   Green (新版本)                             │
│  ┌─────────────────────────────────────────────────────┐    │
│  │  Green Deployment (v2)                               │    │
│  │  replicas: 3                                        │    │
│  └─────────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────┘
```

### 实现方式

```yaml
# 部署 v2 (Green)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp-green
  labels:
    app: myapp
    slot: green
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp
      slot: green
  template:
    metadata:
      labels:
        app: myapp
        slot: green
    spec:
      containers:
      - name: myapp
        image: myapp:v2
---
# Service 指向 Green
apiVersion: v1
kind: Service
metadata:
  name: myapp
spec:
  selector:
    app: myapp
    slot: green  # 切换到 green
  ports:
  - port: 80
    targetPort: 8080
```

### 切换和回滚

```bash
# 切换到 Green
kubectl patch service myapp -p '{"spec":{"selector":{"slot":"green"}}}'

# 秒级回滚到 Blue
kubectl patch service myapp -p '{"spec":{"selector":{"slot":"blue"}}}'
```

### 适用场景

- 需要快速回滚
- 验证后再切换
- 有状态服务
- 资源充足

### 局限性

- 需要双倍资源
- 数据库变更需要特殊处理
- 大量实例时切换可能不瞬间

## 金丝雀发布

金丝雀发布让新版本只接收一小部分流量：

```
┌─────────────────────────────────────────────────────────────┐
│                    10% 流量到 v2                            │
│                                                              │
│     100% 流量                                         10%  │
│        ├─────────────────────────────────────────────▶ [v2] │
│        │                                                  │
│        └─────────────────────────────────────────────────▶ [v1] │
│                                                              │
│  v1: 3 Pods (90%)    v2: 1 Pod (10%)                       │
└─────────────────────────────────────────────────────────────┘
```

### Deployment 实现

```yaml
# Blue/Green 组合 + 权重
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp-stable
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp
      slot: stable
  template:
    metadata:
      labels:
        app: myapp
        slot: stable
    spec:
      containers:
      - name: myapp
        image: myapp:v1
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp-canary
spec:
  replicas: 1  # 1/4 = 25% 流量
  selector:
    matchLabels:
      app: myapp
      slot: canary
  template:
    metadata:
      labels:
        app: myapp
        slot: canary
    spec:
      containers:
      - name: myapp
        image: myapp:v2
```

### Service 实现

```yaml
apiVersion: v1
kind: Service
metadata:
  name: myapp
spec:
  selector:
    app: myapp
  ports:
  - port: 80
    targetPort: 8080
```

由于 replicas 比例是 3:1，约 25% 的流量会到 v2。

### Ingress 权重路由

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: myapp
  annotations:
    nginx.ingress.kubernetes.io/canary: "true"
    nginx.ingress.kubernetes.io/canary-weight: "10"
spec:
  ingressClassName: nginx
  rules:
  - host: myapp.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: myapp-canary
            port:
              number: 80
```

### Istio 权重路由

```yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: myapp
spec:
  hosts:
  - myapp
  http:
  - route:
    - destination:
        host: myapp
        subset: v1
      weight: 90
    - destination:
        host: myapp
        subset: v2
      weight: 10
---
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: myapp
spec:
  host: myapp
  subsets:
  - name: v1
    labels:
      version: v1
  - name: v2
    labels:
      version: v2
```

### 监控驱动的金丝雀

```yaml
# Prometheus 告警触发自动回滚
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: canary-alert
spec:
  groups:
  - name: canary
    rules:
    - alert: HighErrorRate
      expr: |
        sum(rate(http_requests_total{service="myapp-canary",status=~"5.."}[5m]))
        /
        sum(rate(http_requests_total{service="myapp-canary"}[5m])) > 0.05
      for: 2m
      labels:
        severity: critical
      annotations:
        summary: "Canary error rate > 5%"
        description: "自动回滚金丝雀"
```

### 适用场景

- 高风险更新
- 需要真实环境验证
- 按用户群体灰度
- 数据迁移

## A/B 测试

A/B 测试基于用户特征（如 Cookie、Header）分配流量：

```yaml
# 基于 Cookie 的路由
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: myapp
spec:
  hosts:
  - myapp
  http:
  - match:
    - headers:
        cookie:
          regex: ".*group=B.*"
    route:
    - destination:
        host: myapp
        subset: variant-b
  - route:
    - destination:
        host: myapp
        subset: variant-a
```

```yaml
# 基于 Header 的路由
  - match:
    - headers:
        X-User-Type:
          exact: "premium"
    route:
    - destination:
        host: myapp
        subset: premium-variant
```

### 适用场景

- UI/UX 测试
- 功能开关
- 个性化体验
- 不同业务策略验证

## Argo Rollouts

Argo Rollouts 提供了高级部署策略：

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Rollout
metadata:
  name: myapp
spec:
  replicas: 10
  strategy:
    canary:
      steps:
      - setWeight: 5
      - pause: {}
      - analysis:
          templates:
          - templateName: success-rate
      - setWeight: 20
      - pause: {duration: 10m}
      - setWeight: 50
      - pause: {}
      analysis:
        successfulRunHistoryLimit: 3
        unsuccessfulRunHistoryLimit: 3
      trafficRouting:
        nginx:
          stableIngress: myapp-ingress
      canaryMetadata:
        labels:
          version: canary
      stableMetadata:
        labels:
          version: stable
```

### 自动分析

```yaml
apiVersion: argoproj.io/v1alpha1
kind: AnalysisTemplate
metadata:
  name: success-rate
spec:
  args:
  - name: service-name
  metrics:
  - name: success-rate
    interval: 1m
    successCondition: result[0] >= 0.95
    failureLimit: 3
    provider:
      prometheus:
        address: http://prometheus:9090
        query: |
          sum(rate(grpc_server_handled_total{
            grpc_service="mypackage.MyService",
            grpc_method="MyMethod",
            code="OK"
          }[5m])) / sum(rate(grpc_server_handled_total{
            grpc_service="mypackage.MyService",
            grpc_method="MyMethod"
          }[5m]))
```

## 总结

选择部署策略需要考虑：

| 策略 | 适用场景 | 关键优势 |
|---|---|---|
| 滚动更新 | 无状态服务，资源受限 | 资源利用率高 |
| 蓝绿部署 | 需要快速回滚 | 秒级回滚 |
| 金丝雀发布 | 高风险更新 | 风险可控 |
| A/B 测试 | 功能验证 | 数据驱动 |

现代部署推荐使用 Argo Rollouts 或 Service Mesh，它们提供了更精细的流量控制和自动化能力。
