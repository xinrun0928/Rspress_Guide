# Kustomize 与 GitOps

你的团队有开发、测试、预生产、生产多个环境，每个环境都有略微不同的配置：镜像版本不同、副本数不同、数据库连接不同……

你有多少个 deployment.yaml 文件？每个环境的差异如何管理？改一处配置要改多少个文件？上线时会不会搞混环境？

Kustomize 和 GitOps 提供了一种优雅的解决方案。

## Kustomize 基础

Kustomize 是 Kubernetes 原生的配置管理工具，它允许你通过「覆盖」而不是「修改」来管理不同环境的配置。

```
┌─────────────────────────────────────────────────────────────┐
│                      基础配置 (base)                         │
│  ┌─────────────────────────────────────────────────────┐    │
│  │  deployment.yaml / service.yaml / configmap.yaml     │    │
│  │  通用的、可复用的配置                                  │    │
│  └─────────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────┘
           │                              │
           ▼                              ▼
┌─────────────────────┐     ┌─────────────────────┐
│   开发环境 (dev)      │     │   生产环境 (prod)     │
│  ┌───────────────┐   │     │  ┌───────────────┐   │
│  │ kustomization │   │     │  │ kustomization │   │
│  │ .yaml         │   │     │  │ .yaml         │   │
│  │               │   │     │  │               │   │
│  │ patches:      │   │     │  │ patches:      │   │
│  │ - replicas    │   │     │  │ - replicas    │   │
│  │ - env         │   │     │  │ - resources   │   │
│  └───────────────┘   │     │  │ - tolerations │   │
└─────────────────────┘     └─────────────────────┘
```

### 基础配置

```yaml
# base/deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
spec:
  replicas: 1
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
      - name: myapp
        image: myapp:latest
        env:
        - name: LOG_LEVEL
          value: debug
```

```yaml
# base/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization

resources:
- deployment.yaml
- service.yaml
- configmap.yaml

commonLabels:
  app: myapp

namespace: default
```

### 环境覆盖

```yaml
# overlays/dev/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization

namespace: dev

bases:
- ../../base

replicas:
- name: myapp
  count: 1

images:
- name: myapp
  newName: myapp
  newTag: dev-latest

configMapGenerator:
- name: myapp-config
  literals:
  - ENVIRONMENT=development
  - DEBUG=true
```

```yaml
# overlays/prod/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization

namespace: production

bases:
- ../../base

replicas:
- name: myapp
  count: 10

images:
- name: myapp
  newName: registry.example.com/myapp
  newTag: v1.2.3

patches:
- patch-resources.yaml

patchesStrategicMerge:
- patch-prod.yaml

commonAnnotations:
  environment: production
```

## Patch 机制

### Strategic Merge Patch

最简单的覆盖方式：

```yaml
# overlays/prod/patch-replicas.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
spec:
  replicas: 10
```

### JSON Merge Patch

```yaml
# overlays/prod/patch-prod.yaml
apiVersion: apps/v1
kind: Deployment
spec:
  template:
    spec:
      tolerations:
      - key: "node-role"
        operator: "Exists"
        effect: "NoSchedule"
```

### JSON6902 Patch

更精确的 JSON Patch：

```yaml
# overlays/prod/patch-toleration.yaml
- op: add
  path: /spec/template/spec/tolerations/-
  value:
    key: "dedicated"
    operator: "Equal"
    value: "production"
    effect: "NoSchedule"
```

## Generator

### ConfigMap Generator

自动生成 ConfigMap：

```yaml
# overlays/prod/kustomization.yaml
configMapGenerator:
- name: myapp-config
  literals:
  - DATABASE_HOST=prod-db.example.com
  - REDIS_HOST=prod-redis.example.com
  files:
  - config.yaml
```

### Secret Generator

自动生成 Secret：

```yaml
secretGenerator:
- name: myapp-secrets
  literals:
  - password=prod-password
  envs:
  - .env
```

## Kustomize vs Helm

| 特性 | Kustomize | Helm |
|---|---|---|
| 模型 | 覆盖式 | 包管理式 |
| 学习曲线 | 平缓 | 中等 |
| 模板 | 无 | Go 模板 |
| 版本管理 | 内置 | Chart 版本 |
| 包发布 | 文件共享 | Helm Hub |
| 灵活性 | 高 | 中等 |
| 适用场景 | 环境差异管理 | 应用打包分发 |

:::tip 选择建议
- 多环境部署：Kustomize 更合适
- 应用打包分发：Helm 更合适
- 两者也可以结合使用：Helm 打包，Kustomize 管理差异
:::

## GitOps 理念

GitOps 将 Git 作为系统状态的唯一真相来源：

```
┌─────────────────────────────────────────────────────────────┐
│                       GitOps 流程                            │
│                                                              │
│  1. 开发者推送代码 ──────────────────────────────────▶ Git  │
│                                                              │
│  2. CI/CD 触发构建                                          │
│     ├─▶ 构建镜像                                          │
│     └─▶ 更新配置                                           │
│                                                              │
│  3. GitOps Operator 同步                                    │
│     └─▶ 镜像更新触发 Deployment 更新                        │
│                                                              │
│  4. Kubernetes 集群状态与 Git 一致                          │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### GitOps 核心原则

1. **声明式配置**：所有配置通过 YAML 声明
2. **Git 单一真相**：环境的期望状态存储在 Git
3. **自动同步**：Operator 自动将集群状态同步到 Git 声明
4. **可审计**：所有变更都有 Git 历史

## ArgoCD

ArgoCD 是 Kubernetes 原生的 GitOps 持续交付工具：

```
┌─────────────────────────────────────────────────────────────┐
│                       ArgoCD                                │
│                                                              │
│  ┌────────────────────────────────────────────────────┐    │
│  │  Web UI / CLI                                       │    │
│  └────────────────────────────────────────────────────┘    │
│  ┌────────────────────────────────────────────────────┐    │
│  │  Application Controller                             │    │
│  │  - 监控 Git 仓库                                    │    │
│  │  - 对比集群状态与 Git 状态                           │    │
│  │  - 自动或手动同步                                    │    │
│  └────────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────────────┐
│                   Kubernetes Cluster                         │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐                   │
│  │ Dev Env  │  │Staging   │  │Production│                   │
│  └──────────┘  └──────────┘  └──────────┘                   │
└─────────────────────────────────────────────────────────────┘
```

### ArgoCD Application

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: myapp-dev
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/org/k8s-config
    targetRevision: main
    path: overlays/dev
  destination:
    server: https://kubernetes.default.svc
    namespace: dev
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
    syncOptions:
    - CreateNamespace=true
```

### 多环境 ApplicationSet

```yaml
apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: myapp
spec:
  generators:
  - matrix:
      generators:
      - git:
          repoURL: https://github.com/org/k8s-config
          revision: main
          directories:
          - path: overlays/*
      - clusters:
          values:
            environment: "{{path.basename}}"
  template:
    metadata:
      name: 'myapp-{{path.basename}}'
    spec:
      project: default
      source:
        repoURL: https://github.com/org/k8s-config
        targetRevision: main
        path: 'overlays/{{path.basename}}'
      destination:
        server: '{{server}}'
        namespace: '{{path.basename}}'
      syncPolicy:
        automated:
          prune: true
```

## 完整工作流

```bash
# 1. 本地预览
kubectl kustomize overlays/prod | kubectl apply --dry-run=client

# 2. 应用到集群
kubectl kustomize overlays/prod | kubectl apply -f -

# 3. 使用 ArgoCD CLI
argocd app create myapp-prod \
  --repo https://github.com/org/k8s-config \
  --path overlays/prod \
  --dest-server https://kubernetes.default.svc \
  --dest-namespace production \
  --sync-policy automated

# 4. 手动同步
argocd app sync myapp-prod
```

## 总结

Kustomize 和 GitOps 改变了配置管理的方式：

- **Kustomize**：通过覆盖而非修改管理环境差异
- **GitOps**：Git 作为单一真相来源
- **ArgoCD**：自动化同步 Git 状态到集群

这套体系让多环境管理变得简单、可审计、可回滚。
