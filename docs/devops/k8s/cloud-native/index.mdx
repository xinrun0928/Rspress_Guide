# 云原生技术总览

「云原生」这个词你一定听过无数遍。招聘要求写「熟悉云原生架构」，技术博客说「某某方案是云原生的最佳实践」，甚至连服务器供应商都在宣传「云原生基础设施」。

但到底什么是云原生？为什么 Kubernetes 被称为云原生的核心？为什么说云原生不仅仅是「把应用搬到云上」？

这一章节将带你深入理解云原生的本质，以及 Kubernetes 在这个生态中的核心地位。

## 云原生的定义

云原生（Cloud Native）这个概念最早由云原生计算基金会（CNCF）提出：

> 云原生技术使组织能够在公共云、私有云和混合云等现代动态环境中构建和运行可扩展的应用程序。

这个定义包含几个关键点：

- **现代动态环境**：云计算的弹性、多云、混合云
- **可扩展的应用程序**：能够应对负载变化
- **构建和运行**：涵盖开发和运维全生命周期

## 云原生的四根支柱

云原生不是单一技术，而是一套技术体系。核心可以归纳为四个支柱：

### 容器

容器将应用及其依赖打包成一个轻量级、可移植的单元：

| 特性 | 容器 vs 虚拟机 |
|---|---|
| 启动时间 | 秒级 vs 分钟级 |
| 资源开销 | 极低 | 30%+ |
| 隔离性 | 进程级 | 硬件级 |
| 可移植性 | 镜像即环境 | 环境差异大 |

容器解决了「在我机器上能跑」的问题，但单个容器无法管理大规模部署。

### 微服务

微服务架构将大型应用拆分为小型、自治的服务：

```
┌─────────────────────────────────────────────────────────────┐
│                     单体应用                                │
│  ┌───────────────────────────────────────────────────┐    │
│  │  用户模块 │ 订单模块 │ 支付模块 │ 商品模块 │ ...  │    │
│  └───────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                     微服务架构                               │
│  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐                    │
│  │用户服务│ │订单服务│ │支付服务│ │商品服务│                    │
│  └──┬───┘ └──┬───┘ └──┬───┘ └──┬───┘                    │
│     └────────┴────────┴────────┘                         │
│                      │                                    │
│              服务发现 │ 负载均衡                           │
│              熔断限流 │ 配置中心                           │
└─────────────────────────────────────────────────────────────┘
```

微服务带来了独立部署、故障隔离的好处，但也带来了服务治理的复杂性。

### 不可变基础设施

传统模式下，服务器上的配置会不断变化——打个补丁、改个配置、调个参数。环境变得越来越难以预测和复现。

不可变基础设施的理念是：**一旦创建，永不修改**。如果需要变更，创建新实例并替换。

```
传统模式：修改 → 变化累积 → 环境不一致
不可变：创建 → 不可修改 → 替换 → 环境一致
```

### 声明式 API

「告诉我你想要什么，而不是怎么做」：

```yaml
# 声明式：描述期望状态
apiVersion: apps/v1
kind: Deployment
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web
```

Kubernetes 会自动处理如何达到这个状态，而不需要你写脚本一步步操作。

## Kubernetes 的角色

Kubernetes 是云原生技术的集大成者：

```
┌─────────────────────────────────────────────────────────────┐
│                     Kubernetes                              │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  容器编排 ── 自动调度、扩缩容、自愈                   │   │
│  │  服务发现 ── DNS、Service、Ingress                   │   │
│  │  配置管理 ── ConfigMap、Secret                       │   │
│  │  存储编排 ── PV、PVC、StorageClass                    │   │
│  │  网络抽象 ── CNI、NetworkPolicy                       │   │
│  │  安全管控 ── RBAC、SecurityContext、PodSecurity       │   │
│  └─────────────────────────────────────────────────────┘   │
│                           │                                │
│  ┌────────────────────────┼────────────────────────────┐ │
│  │                        ▼                            │ │
│  │  ┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐      │ │
│  │  │  Docker │ │  Helm  │ │Prometheus│ │   Istio │      │ │
│  │  │ (Runtime)│ │(Package)│ │(Monitor) │ │ (Mesh) │      │ │
│  │  └────────┘ └────────┘ └────────┘ └────────┘      │ │
│  └────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────┘
```

## 云原生的收益

采用云原生架构可以获得：

| 收益 | 说明 |
|---|---|
| 弹性伸缩 | 根据负载自动扩缩容 |
| 高可用 | 多副本、故障自动恢复 |
| 快速交付 | CI/CD 流水线，频繁部署 |
| 资源效率 | 更高的资源利用率 |
| 可移植性 | 一次构建，多处运行 |

## 云原生的挑战

云原生也带来了新的挑战：

| 挑战 | 说明 |
|---|---|
| 复杂性 | 分布式系统的固有复杂性 |
| 学习曲线 | 新技术、新概念需要学习 |
| 运维难度 | 故障排查需要新工具和方法 |
| 网络延迟 | 服务间通信的延迟 |
| 数据一致性 | 分布式事务的挑战 |
