# K8s 配置管理：ConfigMap 与 Secret 完全指南

应用配置放 ConfigMap 还是 Secret？两者到底有什么区别？

在 K8s 中部署应用，配置管理是一个绕不开的话题。数据库连接信息、API 地址、功能开关、日志级别……这些东西怎么传给容器？

你可能会看到两种资源：**ConfigMap** 和 **Secret**。它们看起来很像，但到底有什么区别？什么时候用哪个？

今天，我们把这个问题彻底讲清楚。

## 一、ConfigMap 与 Secret 的定位

在 K8s 中，ConfigMap 和 Secret 都是用来给 Pod 传递配置数据的资源。但它们的使用场景有明确的区分：

| 资源 | 用途 | 数据类型 | 安全性 |
|---|---|---|---|
| ConfigMap | 非敏感配置 | 键值对、配置文件 | 普通 |
| Secret | 敏感数据 | 键值对（Base64 编码） | 相对安全 <Badge text="注意" type="warning" /> |

:::warning Secret 真的安全吗？
Secret 默认存储在 etcd 中，**数据只是 Base64 编码，并非加密**。如果 etcd 没有启用加密，集群管理员或能访问 etcd 的人可以直接看到 Secret 内容。

生产环境应该：
1. 启用 etcd 加密存储
2. 使用 KMS Provider 加密 Secret
3. 配合 RBAC 控制 Secret 访问权限

:::

## 二、创建 ConfigMap

### 2.1 命令行创建

**从字面值创建**：

```bash
kubectl create configmap app-config \
  --from-literal=DB_HOST=localhost \
  --from-literal=DB_PORT=3306 \
  --from-literal=LOG_LEVEL=info
```

**从文件创建**：

```bash
# 创建配置文件
cat > config.ini << EOF
database=postgres
host=db.example.com
port=5432
EOF

# 从文件创建 ConfigMap
kubectl create configmap app-config \
  --from-file=config.ini
```

**从目录创建**：

```bash
# 目录中有多个配置文件
kubectl create configmap app-config \
  --from-file=/path/to/config/
```

**从 env 文件 创建**：

```bash
# .env 文件格式
cat > .env << EOF
DATABASE_HOST=localhost
DATABASE_PORT=3306
REDIS_HOST=redis.example.com
EOF

kubectl create configmap app-config \
  --from-env-file=.env
```

### 2.2 YAML 声明式创建

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
  namespace: default
data:
  # 键值对
  DB_HOST: "localhost"
  DB_PORT: "3306"
  LOG_LEVEL: "info"
  # 多行值
  config.json: |
    {
      "database": "postgres",
      "host": "localhost",
      "port": 3306,
      "pool": {
        "min": 5,
        "max": 20
      }
    }
```

## 三、创建 Secret

### 3.1 命令行创建

**从字面值创建**：

```bash
kubectl create secret generic app-secret \
  --from-literal=DB_USER=admin \
  --from-literal=DB_PASSWORD=secret123
```

**从文件创建**：

```bash
# 创建密钥文件
echo -n "admin" > username.txt
echo -n "secret123" > password.txt

kubectl create secret generic app-secret \
  --from-file=username=username.txt \
  --from-file=password=password.txt
```

**Docker Registry Secret**：

```bash
kubectl create secret docker-registry my-registry \
  --docker-server=registry.example.com \
  --docker-username=admin \
  --docker-password=password123 \
  --docker-email=admin@example.com
```

### 3.2 YAML 声明式创建

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: app-secret
  namespace: default
type: Opaque
data:
  # Base64 编码的值
  # Linux: echo -n "value" | base64
  # macOS: echo -n "value" | base64
  DB_USER: YWRtaW4=
  DB_PASSWORD: c2VjcmV0MTIz
```

:::tip Base64 编码
`echo -n "value" | base64` 的 `-n` 参数很重要，它避免在值末尾添加换行符。
:::

## 四、Pod 使用 ConfigMap 和 Secret

### 4.1 环境变量方式

**使用 `envFrom`（推荐）**：

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  containers:
  - name: app
    image: my-app:latest
    envFrom:
    - configMapRef:
        name: app-config
    - secretRef:
        name: app-secret
```

**使用 `env` 精确指定**：

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  containers:
  - name: app
    image: my-app:latest
    env:
    - name: DB_HOST
      valueFrom:
        configMapKeyRef:
          name: app-config
          key: DB_HOST
    - name: DB_PASSWORD
      valueFrom:
        secretKeyRef:
          name: app-secret
          key: DB_PASSWORD
```

**两者的区别**：

| 方式 | 优点 | 缺点 |
|---|---|---|
| `envFrom` | 简单，一次性导入所有键值对 | 不能选择特定的 key |
| `env` | 精确控制，可以映射单个 key | 逐个指定，较繁琐 |

### 4.2 Volume 挂载方式

**ConfigMap 作为 Volume**：

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  containers:
  - name: app
    image: my-app:latest
    volumeMounts:
    - name: config
      mountPath: /app/config
      readOnly: true
  volumes:
  - name: config
    configMap:
      name: app-config
      items:
      - key: DB_HOST
        path: database/host.txt
      - key: DB_PORT
        path: database/port.txt
      defaultMode: 0644
```

挂载后的文件结构：

```
/app/config/
├── database/
│   ├── host.txt    # 内容：localhost
│   └── port.txt    # 内容：3306
└── (其他 ConfigMap 中的 key 会作为文件名)
```

**Secret 作为 Volume**：

```yaml
volumes:
- name: credentials
  secret:
    secretName: app-secret
    items:
    - key: DB_USER
      path: username.txt
      mode: 0644
    - key: DB_PASSWORD
      path: password.txt
      mode: 0600
    defaultMode: 0640
```

**挂载后文件内容**：

```bash
# 查看 Pod 中的文件
kubectl exec -it my-app -- cat /app/config/DB_HOST
# 输出：localhost

kubectl exec -it my-app -- cat /app/credentials/password.txt
# 输出：secret123
```

## 五、环境变量 vs Volume 挂载：怎么选？

| 特性 | 环境变量 | Volume 挂载 |
|---|---|---|
| 实时更新 | 需要重启容器才能生效 | 可以自动更新（分钟级） |
| 访问方式 | `os.Getenv("KEY")` | 读取文件 |
| 大文件 | 不适合 | 适合（配置文件） |
| 类型安全 | 无 | 无 |
| 用途 | 运行时配置 | 配置文件 |

**建议**：

- **环境变量**：适合小规模配置、数据库连接信息、功能开关
- **Volume 挂载**：适合配置文件（如 nginx.conf、application.yml）

```yaml
# 综合使用：环境变量 + Volume 挂载
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  containers:
  - name: app
    image: my-app:latest
    envFrom:
    - configMapRef:
        name: app-config
    - secretRef:
        name: app-secret
    volumeMounts:
    - name: nginx-conf
      mountPath: /etc/nginx/conf.d
      readOnly: true
  volumes:
  - name: nginx-conf
    configMap:
      name: nginx-config
```

## 六、ConfigMap/Secret 的高级特性

### 6.1 不可变 ConfigMap 和 Secret

<Badge text="K8s 1.19+" type="tip" /> 不可变资源可以提升集群性能，防止意外修改。

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: immutable-config
data:
  config.json: |
    {
      "version": "1.0"
    }
immutable: true
```

:::warning 不可变意味着不可修改
一旦设置 `immutable: true`，该 ConfigMap/Secret 就不能再修改，也不能删除（除非删除整个命名空间）。设计时要谨慎。
:::

**使用场景**：

- 频繁使用的配置（减少 API Server 负载）
- 敏感配置（防止意外修改）
- 静态配置文件（如应用版本配置）

### 6.2 二进制数据

ConfigMap 和 Secret 支持存储二进制数据：

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: binary-secret
type: Opaque
binaryData:
  # Base64 编码的二进制内容
  certificate: LS0tLS1CRUdJTiBDRVJUS...
  private-key: LS0tLS1CRUdJTiBQUklW...
```

### 6.3 subPath 挂载的坑

使用 `subPath` 挂载 ConfigMap/Secret 时，**不会自动更新**：

```yaml
volumes:
- name: config
  configMap:
    name: app-config
    items:
    - key: config.yaml
      path: config.yaml
    subPath: config.yaml  # 注意这个字段
```

:::warning subPath 导致无法热更新
当使用 `subPath` 时，Kubelet 不会监控 ConfigMap 的变化，文件不会自动更新。

解决方案：
1. 去掉 `subPath`，直接挂载整个目录
2. 使用符号链接或 init 容器重新挂载
3. 接受需要重启 Pod 才能更新
:::

### 6.4 自动更新机制

当 ConfigMap/Secret 发生变化时，挂载的文件会在约 60 秒内自动更新。更新机制：

1. K8s API Server 通知 Kubelet ConfigMap 变化
2. Kubelet 更新符号链接指向的新文件
3. 应用读取到新配置

```bash
# 查看 ConfigMap 变化
kubectl get events --watch --field-metadata involvedObject.name=app-config
```

## 七、实战：完整的配置管理示例

### 7.1 应用配置

```yaml
# nginx 配置
apiVersion: v1
kind: ConfigMap
metadata:
  name: nginx-config
  namespace: default
data:
  default.conf: |
    server {
        listen 80;
        server_name localhost;
        location / {
            proxy_pass http://backend:8080;
        }
    }
---
# 应用配置
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
  namespace: default
data:
  APP_ENV: "production"
  LOG_LEVEL: "info"
  CACHE_TTL: "3600"
---
# 敏感配置
apiVersion: v1
kind: Secret
metadata:
  name: app-secrets
  namespace: default
type: Opaque
stringData:
  # stringData 会自动 Base64 编码存储在 data 中
  DATABASE_URL: "postgresql://admin:password123@db:5432/mydb"
  API_KEY: "sk-1234567890abcdef"
```

### 7.2 Pod 使用配置

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: web-app
  namespace: default
spec:
  containers:
  - name: nginx
    image: nginx:1.25
    ports:
    - containerPort: 80
    volumeMounts:
    - name: nginx-conf
      mountPath: /etc/nginx/conf.d
      readOnly: true
  - name: app
    image: my-app:latest
    envFrom:
    - configMapRef:
        name: app-config
    - secretRef:
        name: app-secrets
    volumeMounts:
    - name: nginx-conf
      mountPath: /external-config
      readOnly: true
  volumes:
  - name: nginx-conf
    configMap:
      name: nginx-config
      items:
      - key: default.conf
        path: default.conf
```

## 八、大小限制与配额

### 8.1 大小限制

ConfigMap 和 Secret 都有 **1MB** 的大小限制。

```bash
# 检查 ConfigMap 大小
kubectl get configmap app-config -o jsonpath='{.data}' | wc -c
```

**大文件建议**：

- 配置文件超过 1MB：使用对象存储，容器启动时下载
- 二进制大文件：使用对象存储或 Volume
- 动态配置：使用配置服务（如 Apollo、Nacos）

### 8.2 命名空间配额

```yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: config-quota
  namespace: default
spec:
  hard:
    configmaps: "50"
    persistentvolumeclaims: "10"
    secrets: "20"
    requests.storage: "100Gi"
```

## 九、常见问题与最佳实践

### 9.1 Secret 泄漏的风险

Secret 的数据虽然 Base64 编码，但很容易解码：

```bash
# 解码 Secret
kubectl get secret app-secret -o jsonpath='{.data.DB_PASSWORD}' | base64 -d
```

**防护措施**：

1. **启用 etcd 加密**
2. **使用 KMS Provider**
3. **RBAC 限制 Secret 访问**
4. **禁止 Secret 以明文日志输出**
5. **使用 short-lived ServiceAccount Token**

### 9.2 容器启动顺序问题

如果容器启动时依赖 ConfigMap/Secret 不存在，会导致 Pod 启动失败。

**解决方案**：使用 `optional: true` 让配置可选

```yaml
env:
- name: OPTIONAL_CONFIG
  valueFrom:
    configMapKeyRef:
      name: optional-config
      key: key
      optional: true  # ConfigMap 不存在时不会报错
```

### 9.3 跨命名空间使用

ConfigMap 和 Secret **不能跨命名空间引用**。每个命名空间需要独立的 ConfigMap/Secret。

**解决方案**：

1. 在每个命名空间创建相同的 ConfigMap/Secret
2. 使用 Kustomize 或 Helm 的 `commonLabels`/`commonAnnotations`
3. 使用 GitOps 工具同步配置

```yaml
# kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization

namespace: production

resources:
- configmap.yaml
- secret.yaml

commonLabels:
  env: production
```

## 十、总结

| 特性 | ConfigMap | Secret |
|---|---|---|
| 用途 | 非敏感配置 | 敏感数据 |
| 编码 | 无（明文） | Base64 编码 |
| 数量限制 | 无特殊限制 | 无特殊限制 |
| 大小限制 | 1MB | 1MB |
| 不可变 | 支持 | 支持 |
| 挂载方式 | 环境变量、Volume | 环境变量、Volume |
| 自动更新 | 支持（Volume） | 支持（Volume） |

**最佳实践**：

1. **区分使用场景**：非敏感用 ConfigMap，敏感用 Secret
2. **启用 Secret 加密**：生产环境必须启用 etcd 加密
3. **使用 `envFrom` + `optional`**：简化配置管理
4. **大配置文件用 Volume**：避免超过 1MB 限制
5. **使用不可变 ConfigMap**：高频使用的配置建议设置 `immutable: true`

掌握 ConfigMap 和 Secret，你就掌握了 K8s 应用配置管理的核心——让配置与代码分离，让应用更灵活。
