# PV 与 PVC 详解

你有没有想过——Pod 被调度到任何节点都能正常运行，但有状态的数据库 Pod 怎么办？如果调度到新节点，之前的数据还在吗？

这就是 PV/PVC 要解决的问题。Kubernetes 通过 PersistentVolume（持久卷）和 PersistentVolumeClaim（持久卷声明）将存储管理与 Pod 分离，让存储像计算资源一样可以被动态分配和管理。

## 为什么需要 PV/PVC

思考一个问题：直接给 Pod 挂载 hostPath，看起来很简单：

```yaml
volumes:
- name: data
  hostPath:
    path: /mnt/data
```

问题在哪？

- Pod 调度到哪个节点，hostPath 就必须在哪个节点上
- 如果 Pod 被调度到其他节点，数据就丢失了
- 不同环境（dev/staging/prod）的存储配置不同，无法统一管理

**PV/PVC 的核心思想**：Pod 不关心数据存在哪里，只需要声明「我需要多少存储」。Kubernetes 负责找到合适的存储并挂载。

```
┌──────────────────────────────────────────────────────────────────┐
│                       Without PV/PVC                              │
│                                                                   │
│  Pod ──────────────────────────────────────► hostPath            │
│       紧耦合：Pod 必须调度到有 /mnt/data 的节点                   │
└──────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌──────────────────────────────────────────────────────────────────┐
│                        With PV/PVC                                │
│                                                                   │
│  Pod ──► PVC ──► PV ──► Storage (NFS / Cloud / Local)           │
│       │                                                      │
│       │  Pod 不关心存储细节，只声明需要多大空间                   │
│       │  PV Controller 负责匹配 PVC 和 PV                        │
│       │  存储可以在任意节点访问（网络存储）                       │
└──────────────────────────────────────────────────────────────────┘
```

## 核心概念

### PersistentVolume（PV）

PV 是集群级别的资源，由管理员或动态供给机制创建，代表一块真实的存储空间。

```yaml
apiVersion: v1
kind: PersistentVolume
metadata:
  name: my-pv
spec:
  capacity:
    storage: 10Gi
  accessModes:
    - ReadWriteOnce
  persistentVolumeReclaimPolicy: Retain
  storageClassName: standard
  nfs:
    path: /data/myapp
    server: 192.168.1.100
```

### PersistentVolumeClaim（PVC）

PVC 是用户侧的请求，声明对存储的需求。Pod 通过 PVC 来使用存储。

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: my-pvc
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 5Gi
  storageClassName: standard
  selector:
    matchLabels:
      environment: production
```

### Pod 使用 PVC

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: myapp
spec:
  containers:
  - name: myapp
    image: nginx
    volumeMounts:
    - name: data
      mountPath: /var/lib/data
  volumes:
  - name: data
    persistentVolumeClaim:
      claimName: my-pvc
```

## 访问模式（Access Modes）

| 模式 | 缩写 | 说明 |
|---|---|---|
| **ReadWriteOnce** | RWO | 单节点读写（大多数云存储支持） |
| **ReadOnlyMany** | ROX | 多节点只读（NFS、Azure Files） |
| **ReadWriteMany** | RWX | 多节点读写（NFS、Azure Files、CephFS） |
| **ReadWriteOncePod** | RWOP | 单 Pod 独占读写（K8s 1.22+，隔离性最强） |

:::warning 注意：不是所有存储都支持所有模式
| 存储类型 | RWO | ROX | RWX |
|---|---|---|---|
| AWS EBS | 支持 | 不支持 | 不支持 |
| GCP PD | 支持 | 不支持 | 不支持 |
| Azure Disk | 支持 | 不支持 | 不支持 |
| NFS | 支持 | 支持 | 支持 |
| CephFS | 支持 | 支持 | 支持 |
| HostPath | 支持 | 不支持 | 不支持 |
:::

## 回收策略（Reclaim Policy）

PV 创建时可以指定 `persistentVolumeReclaimPolicy`：

| 策略 | 说明 | 适用场景 |
|---|---|---|
| **Retain** | 删除 PVC 后，PV 保留数据，需要管理员手动处理 | 重要数据，需要保留 |
| **Delete** | 删除 PVC 时自动删除 PV 和数据 | 测试环境、临时存储 |
| **Recycle** | 删除 PVC 时执行 `rm -rf /volume/*`，变为 Available | 已废弃，优先使用动态供给 |

```yaml
spec:
  persistentVolumeReclaimPolicy: Retain
```

## 存储类（StorageClass）

StorageClass 是动态供给的核心组件。管理员创建一个 StorageClass，用户通过 PVC 请求，Kubernetes 自动创建 PV。

### 默认 StorageClass

```bash
# 查看集群中的 StorageClass
kubectl get storageclass

# NAME                 PROVISIONER                    AGE
# standard (default)   kubernetes.io/gce-pd           10d
# fast-ssd             kubernetes.io/gce-pd           5d
# nfs                  nfs-storage.io/nfs             3d
```

### 创建 StorageClass

```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: fast-ssd
provisioner: kubernetes.io/gce-pd
parameters:
  type: pd-ssd           # SSD 磁盘
  replication-type: regional-pd  # 区域级存储
volumeBindingMode: WaitForFirstConsumer  # 延迟绑定
allowVolumeExpansion: true
reclaimPolicy: Retain
```

### StorageClass 参数

**AWS EBS**：

```yaml
parameters:
  type: gp3                    # gp2 / gp3 / io1 / io2
  encrypted: "true"            # 加密
  kmsKeyId: alias/aws/ebs      # KMS 密钥
```

**NFS**：

```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: nfs-storage
provisioner: nfs-storage.io/nfs
parameters:
  server: nfs.example.com
  path: /data/k8s
  onDelete: retain
```

## 延迟绑定（Volume Binding Mode）

### Immediate（立即绑定）

PVC 创建后立即绑定 PV，可能导致 Pod 调度到无法访问该 PV 的节点。

```yaml
volumeBindingMode: Immediate
```

### WaitForFirstConsumer（等待第一个消费者）

延迟到 Pod 调度决策完成后再绑定，确保 Pod 和 PV 在同一节点（或同一可用区）。

```yaml
volumeBindingMode: WaitForFirstConsumer
```

```
┌─────────────────────────────────────────────────────────────┐
│  WaitForFirstConsumer 流程                                   │
│                                                              │
│  1. PVC 创建 ──► PV 状态为 "Pending"（未绑定）             │
│                                                              │
│  2. Pod 创建，调度到 Node-2                                 │
│                                                              │
│  3. PV Controller 发现 Pod 被调度到 Node-2                   │
│                                                              │
│  4. 选择 Zone-2 的 PV 或创建新 PV                           │
│                                                              │
│  5. PVC 绑定 PV ──► Pod 开始启动                            │
└─────────────────────────────────────────────────────────────┘
```

## PVC 生命周期

```
┌──────────────────────────────────────────────────────────┐
│                   PVC 生命周期                              │
│                                                              │
│  Pending ──► Bound ──► Released ──► (回收) ──► Available  │
│      │          │          │               │                │
│      ▼          ▼          ▼               ▼                │
│   等待绑定    已绑定     PVC 已删除       PV 保留/已删除    │
│               到 PV       PV 仍存在                         │
└──────────────────────────────────────────────────────────┘
```

### 查看 PVC 状态

```bash
# 查看所有 PVC
kubectl get pvc

# NAME        STATUS    VOLUME   CAPACITY   ACCESS MODES   AGE
# my-pvc      Bound     my-pv    10Gi       RWO            10d
# data-pvc    Pending                                      1h

# 查看 PV
kubectl get pv

# NAME        CAPACITY   ACCESS MODES   RECLAIM POLICY   STATUS    CLAIM
# my-pv       10Gi       RWO            Retain           Bound     default/my-pvc
# old-pv      50Gi       RWO            Delete           Released  default/old-pvc
```

### 扩容 PVC

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: my-pvc
spec:
  resources:
    requests:
      storage: 20Gi    # 从 10Gi 扩容到 20Gi
  storageClassName: standard
```

```bash
# 在线扩容（K8s 1.24+）
kubectl patch pvc my-pvc -p '{"spec":{"resources":{"requests":{"storage":"20Gi"}}}}'

# 查看扩容状态
kubectl get pvc my-pvc -o jsonpath='{.status.capacity}'
```

:::tip 扩容前提
StorageClass 必须设置 `allowVolumeExpansion: true`，且底层存储支持在线扩容。
:::

## 使用模式对比

### 静态供给（Static Provisioning）

管理员手动创建 PV，用户手动创建 PVC 绑定。

```
管理员 ──► 创建 PV ──► 用户 ──► 创建 PVC ──► 绑定 ──► Pod 使用
```

**适用场景**：有固定存储资源的场景。

### 动态供给（Dynamic Provisioning）

管理员创建 StorageClass，用户创建 PVC，Kubernetes 自动创建 PV。

```
管理员 ──► 创建 StorageClass ──► 用户 ──► 创建 PVC ──► 动态创建 PV ──► 绑定 ──► Pod 使用
```

**适用场景**：云环境、临时存储、按需分配。

### 默认 StorageClass

如果集群有默认 StorageClass，用户甚至不需要指定 `storageClassName`：

```yaml
# 不指定 storageClassName，使用默认
spec:
  resources:
    requests:
      storage: 5Gi
```

## 本地持久卷（Local PV）

对于需要高性能且有本地 SSD 的场景，可以使用本地持久卷。

### 创建 Local PV

```yaml
apiVersion: v1
kind: PersistentVolume
metadata:
  name: local-pv
spec:
  capacity:
    storage: 100Gi
  accessModes:
  - ReadWriteOnce
  persistentVolumeReclaimPolicy: Retain
  storageClassName: local-storage
  local:
    path: /mnt/disks/ssd-1
  nodeAffinity:
    required:
      nodeSelectorTerms:
      - matchExpressions:
        - key: kubernetes.io/hostname
          operator: In
          values:
          - node-1
```

### 使用 Local PV

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: local-pvc
spec:
  accessModes:
  - ReadWriteOnce
  resources:
    requests:
      storage: 50Gi
  storageClassName: local-storage
```

### Local Storage Operator（推荐）

生产环境推荐使用 [Local Storage Operator](https://github.com/operator-framework/local-storage-operator)，自动发现本地磁盘并管理生命周期。

## 常见问题排查

### PVC 一直处于 Pending 状态

```bash
# 1. 检查 PVC 详情
kubectl describe pvc my-pvc

# 2. 常见原因：
# - 没有匹配的 PV（存储空间不足、accessMode 不匹配）
# - WaitForFirstConsumer 模式下，Pod 未调度
# - StorageClass 不存在

# 3. 解决方案：
# - 增加 PV 容量
# - 检查 StorageClass 配置
# - 调整 volumeBindingMode
```

### Pod 无法挂载 PVC

```bash
# 1. 检查 Pod 事件
kubectl describe pod myapp

# 2. 检查 PVC 是否 Bound
kubectl get pvc my-pvc

# 3. 检查 PV 的 accessMode
kubectl get pv my-pv -o yaml | grep accessModes

# 4. 检查节点是否有对应存储
kubectl get nodes -o jsonpath='{.items[*].status.addresses[?(@.type=="InternalIP")].address}'
```

### 存储类删除失败

```bash
# PV 的 reclaimPolicy 为 Retain 时，StorageClass 无法直接删除
# 需要先删除所有相关 PV
kubectl get pv -o jsonpath='{range .items[?(@.spec.storageClassName=="my-sc")]}{.metadata.name}{"\n"}{end}'

# 删除所有相关 PV 后，才能删除 StorageClass
kubectl delete storageclass my-sc
```

## 总结

PV/PVC 是 Kubernetes 存储管理的核心抽象：

1. **PV 是存储资源**：由管理员或 StorageClass 创建
2. **PVC 是存储请求**：由用户创建，声明存储需求
3. **StorageClass 动态供给**：按需创建 PV
4. **延迟绑定**：避免 Pod 调度到无法访问存储的节点
5. **扩容与回收**：支持在线扩容和多种回收策略

理解 PV/PVC 的生命周期和管理策略，是掌握 Kubernetes 有状态应用的第一步。
