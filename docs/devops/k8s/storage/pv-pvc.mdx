# K8s 持久化存储：PV 与 PVC 完全指南

Pod 重启后数据还在吗？——K8s 持久化存储的正确打开方式

你有没有遇到过这种情况：在 Pod 里跑了一个 MySQL 数据库，往里面插入了数据，然后 Pod 因为某种原因被删除了。

等新的 Pod 创建出来后，你满怀期待地查询数据——然后发现，数据全没了。

这是因为**容器内的文件系统是临时的**，Pod 删除后数据就消失了。那 MySQL 的数据要怎么持久化呢？

答案就是 K8s 的 **PersistentVolume（PV）和 PersistentVolumeClaim（PVC）**。


## 一、为什么需要 PV 和 PVC？

在 K8s 中，Pod 是"用完即弃"的——Pod 被删除后，容器内的数据也会丢失。

对于有状态应用（如数据库、消息队列），数据必须持久化。但问题是：

1. 存储类型五花八门：本地磁盘、NFS、云厂商块存储、分布式存储……
2. 每个应用的存储需求不同：有的需要 SSD，有的需要大容量 HDD
3. 开发人员不应该关心底层存储细节，只需要"给我一个能用的存储"

PV 和 PVC 就是为了解决这个问题的。它们将**存储的管理**与**存储的使用**解耦：

- **PV（PersistentVolume）**：集群管理员创建的存储资源，类似于"存储池"
- **PVC（PersistentVolumeClaim）**：开发人员请求的存储，类似于"从存储池中申请的配额"

## 二、核心概念解析

### 2.1 PersistentVolume（PV）

PV 是集群级别的资源，由集群管理员创建，代表集群中的一块存储。

```yaml
apiVersion: v1
kind: PersistentVolume
metadata:
  name: my-pv
  labels:
    type: fast
spec:
  capacity:
    storage: 10Gi
  accessModes:
    - ReadWriteOnce
  persistentVolumeReclaimPolicy: Retain
  nfs:
    server: nfs-server.example.com
    path: /data/mysql
```

**关键字段**：

| 字段 | 说明 |
|---|---|
| `capacity.storage` | 存储容量 |
| `accessModes` | 访问模式 |
| `persistentVolumeReclaimPolicy` | 回收策略 |
| `mountOptions` | 挂载选项 |
| `storageClassName` | 关联的 StorageClass |

### 2.2 PersistentVolumeClaim（PVC）

PVC 是命名空间级别的资源，由开发人员创建，向 PV 请求存储。

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: mysql-pvc
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 5Gi
  storageClassName: standard
  selector:
    matchLabels:
      type: fast
```

### 2.3 Pod 使用 PVC

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: mysql
spec:
  containers:
  - name: mysql
    image: mysql:8
    volumeMounts:
    - name: mysql-data
      mountPath: /var/lib/mysql
  volumes:
  - name: mysql-data
    persistentVolumeClaim:
      claimName: mysql-pvc
```

## 三、PV 生命周期

PV 的生命周期分为四个阶段：**供给（Provisioning）** → **绑定（Binding）** → **使用（Using）** → **回收（Reclaiming）**。

### 3.1 供给阶段

PV 的供给有两种方式：

**静态供给（Static Provisioning）**：

集群管理员手动创建 PV，定义存储的类型、容量等信息。当用户申请存储时，从已有的 PV 池中匹配。

```yaml
apiVersion: v1
kind: PersistentVolume
metadata:
  name: pv-fast
spec:
  capacity:
    storage: 50Gi
  accessModes:
    - ReadWriteOnce
  storageClassName: fast
  nfs:
    server: nfs.example.com
    path: /volumes/fast
---
apiVersion: v1
kind: PersistentVolume
metadata:
  name: pv-slow
spec:
  capacity:
    storage: 200Gi
  accessModes:
    - ReadWriteOnce
  storageClassName: slow
  nfs:
    server: nfs.example.com
    path: /volumes/slow
```

**动态供给（Dynamic Provisioning）**：

集群管理员配置 StorageClass，用户申请存储时，自动创建 PV。不需要预先手动创建 PV 池。

```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: fast
provisioner: kubernetes.io/gce-pd
parameters:
  type: pd-ssd
  replication-type: regional-pd
reclaimPolicy: Retain
```

用户只需要创建 PVC，StorageClass 会自动创建 PV：

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: dynamic-pvc
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 20Gi
  storageClassName: fast
  # 不需要指定 PV，StorageClass 自动创建
```

### 3.2 绑定阶段

当 PVC 创建后，K8s 控制平面会尝试找到一个满足条件的 PV 进行绑定。

**匹配条件**：

1. `storageClassName` 必须一致（或者都没有指定）
2. PVC 请求的容量 `<=` PV 的容量
3. PV 的 `accessModes` 包含 PVC 请求的所有模式
4. PVC 的 `selector` 能匹配到 PV 的 labels

**标签选择器（Selector）**：

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: my-pvc
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 5Gi
  selector:
    matchLabels:
      environment: production
      type: ssd
```

**重要约束**：PVC 与 PV 是**一对一绑定**的关系。一个 PV 只能绑定一个 PVC，绑定后不能再被其他 PVC 使用。

```yaml
# 查看 PV 和 PVC 的绑定关系
kubectl get pv,pvc
```

### 3.3 使用阶段

PVC 绑定后，就可以在 Pod 中使用了。Pod 通过 `persistentVolumeClaim` 引用 PVC：

```yaml
spec:
  containers:
  - name: app
    volumeMounts:
    - name: data
      mountPath: /data
  volumes:
  - name: data
    persistentVolumeClaim:
      claimName: my-pvc
```

### 3.4 回收阶段

当 PVC 不再需要时，需要决定如何处理 PV 中的数据。

**回收策略（Reclaim Policy）**：

| 策略 | 说明 | 适用场景 |
|---|---|---|
| `Retain`（保留） | PVC 删除后，PV 保留，数据保留 | 生产环境重要数据 |
| `Delete`（删除） | PVC 删除后，PV 及其数据自动删除 | 测试环境、临时存储 |
| `Recycle`（回收）<Badge text="Deprecated" type="warning" /> | 删除数据后，PV 可再次使用 | 不推荐，已废弃 |

```yaml
apiVersion: v1
kind: PersistentVolume
metadata:
  name: my-pv
spec:
  persistentVolumeReclaimPolicy: Retain
  # ...
```

**Retain 的清理流程**：

当 PVC 被删除且 PV 设为 Retain 时，PV 状态变为 `Released`。此时需要手动清理：

```bash
# 1. 备份数据（如果需要）

# 2. 删除 PV
kubectl delete pv my-pv

# 3. 手动清理底层存储（挂载点、NFS 目录等）

# 4. 如果需要，重新创建 PV（可用于重新绑定新的 PVC）
kubectl apply -f my-pv.yaml
```

## 四、访问模式（Access Modes）

K8s 定义了三种访问模式，决定存储的读写权限和共享方式。

| 模式 | 缩写 | 说明 |
|---|---|---|
| `ReadWriteOnce` | RWO | 单节点读写 |
| `ReadOnlyMany` | ROX | 多节点只读 |
| `ReadWriteMany` | RWX | 多节点读写 |

**详细说明**：

- **ReadWriteOnce（RWO）**：存储卷只能被单个节点以读写方式挂载。大多数云厂商的块存储（如 AWS EBS、GCE PD）都支持 RWO。
- **ReadOnlyMany（ROX）**：存储卷可以被多个节点以只读方式挂载。NFS 支持 ROX。
- **ReadWriteMany（RWX）**：存储卷可以被多个节点以读写方式挂载。NFS、分布式存储支持 RWX。

:::warning 模式匹配
PVC 和 PV 的访问模式必须是兼容的。PVC 请求的模式必须在 PV 支持的模式中。例如，如果 PV 只支持 RWO，PVC 请求 RWX 将会失败。
:::

**不同存储类型的支持情况**：

| 存储类型 | RWO | ROX | RWX |
|---|---|---|---|
| AWS EBS | ✓ | - | - |
| GCE PD | ✓ | - | - |
| Azure Disk | ✓ | - | - |
| NFS | ✓ | ✓ | ✓ |
| CephFS | ✓ | ✓ | ✓ |
| HostPath | ✓ | - | - |
| emptyDir | ✓ | - | - |

## 五、实战：创建 PV 和 PVC

### 5.1 场景：NFS 存储

**Step 1：创建 NFS PV**

```yaml
apiVersion: v1
kind: PersistentVolume
metadata:
  name: nfs-pv
  labels:
    type: nfs
spec:
  capacity:
    storage: 50Gi
  accessModes:
    - ReadWriteMany
  persistentVolumeReclaimPolicy: Retain
  nfs:
    server: nfs-server.example.com
    path: /data/k8s
```

**Step 2：创建 PVC**

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: nfs-pvc
spec:
  accessModes:
    - ReadWriteMany
  resources:
    requests:
      storage: 10Gi
  selector:
    matchLabels:
      type: nfs
```

**Step 3：在 Pod 中使用**

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nfs-client
spec:
  containers:
  - name: app
    image: nginx
    volumeMounts:
    - name: nfs-storage
      mountPath: /data
  volumes:
  - name: nfs-storage
    persistentVolumeClaim:
      claimName: nfs-pvc
```

### 5.2 场景：云厂商块存储（动态供给）

```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: fast-storage
provisioner: kubernetes.io/gce-pd
parameters:
  type: pd-ssd
  fsType: ext4
reclaimPolicy: Retain
volumeBindingMode: WaitForFirstConsumer
---
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: fast-pvc
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 100Gi
  storageClassName: fast-storage
```

## 六、状态与阶段

### 6.1 PV 的状态

| 状态 | 说明 |
|---|---|
| `Available` | 可用，尚未被任何 PVC 绑定 |
| `Bound` | 已绑定到某个 PVC |
| `Released` | PVC 已删除，但 PV 的数据还在 |
| `Failed` | 自动回收失败 |

```bash
# 查看 PV 状态
kubectl get pv

# NAME      CAPACITY   ACCESSMODES   RECLAIMPOLICY   STATUS     CLAIM
# pv-nfs    50Gi       RWX           Retain          Bound      default/nfs-pvc
# pv-local  20Gi       RWO           Delete          Available
```

### 6.2 PVC 的状态

| 状态 | 说明 |
|---|---|
| `Pending` | 正在等待绑定 |
| `Bound` | 已绑定到某个 PV |
| `Lost` | 绑定的 PV 不存在 |

```bash
# 查看 PVC 状态
kubectl get pvc -n default

# NAME      STATUS    VOLUME    CAPACITY   ACCESSMODES   AGE
# nfs-pvc   Bound     pv-nfs    50Gi       RWX           1d
# slow-pvc  Pending                                       1h
```

## 七、常见问题与解决方案

### 7.1 PVC 一直处于 Pending 状态

可能原因：

1. **没有满足条件的 PV**：检查是否有足够容量的 PV 可用
2. **StorageClass 不存在**：检查 StorageClass 名称是否正确
3. **访问模式不匹配**：PVC 请求的 accessModes 没有对应的 PV
4. **标签选择器不匹配**：PV 的 labels 与 PVC 的 selector 不匹配

```bash
# 检查 StorageClass
kubectl get storageclass

# 检查 PV 详情
kubectl describe pv <pv-name>

# 检查 PVC 详情
kubectl describe pvc <pvc-name> -n <namespace>
```

### 7.2 PVC 删除后数据还在吗？

取决于 PV 的 `persistentVolumeReclaimPolicy`：

- `Retain`：数据保留，需要手动清理
- `Delete`：数据随 PV 一起删除
- `Recycle`：数据被删除，PV 可重新使用

### 7.3 如何扩容 PVC？

<Badge text="K8s 1.11+" type="tip" /> 在线扩容功能需要 StorageClass 支持 `allowVolumeExpansion: true`。

```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: standard
provisioner: kubernetes.io/gce-pd
allowVolumeExpansion: true
```

扩容 PVC：

```bash
kubectl patch pvc my-pvc -p '{"spec":{"resources":{"requests":{"storage":"20Gi"}}}}'
```

:::warning 注意事项
- 扩容只支持增大容量，不支持减小容量
- 部分存储类型（如 GCE PD）支持在线扩容，部分需要重启 Pod
- AWS EBS 不支持在线扩容，需要创建新的 PVC 并迁移数据
:::

## 八、最佳实践

### 8.1 生产环境建议

1. **使用动态供给**：通过 StorageClass 自动创建 PV，减少管理负担
2. **设置 Retain 策略**：生产数据使用 Retain 策略，防止误删
3. **合理设置容量**：避免申请过大或过小的存储
4. **使用合适的 StorageClass**：根据性能需求选择 SSD 或普通磁盘

### 8.2 命名规范

```yaml
# 推荐的 PVC 命名
metadata:
  name: <app-name>-pvc
# 例如
# mysql-primary-pvc
# redis-cluster-pvc
# app-uploads-pvc
```

### 8.3 多环境配置

```yaml
# 开发环境
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: dev-standard
provisioner: kubernetes.io/hostpath
reclaimPolicy: Delete
volumeBindingMode: Immediate

# 生产环境
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: prod-fast
provisioner: kubernetes.io/gce-pd
parameters:
  type: pd-ssd
reclaimPolicy: Retain
volumeBindingMode: WaitForFirstConsumer
```

## 九、总结

PV 和 PVC 是 K8s 持久化存储的核心抽象：

| 概念 | 作用 | 作用域 |
|---|---|---|
| PersistentVolume | 存储资源的抽象 | 集群级别 |
| PersistentVolumeClaim | 存储请求的抽象 | 命名空间级别 |
| StorageClass | 存储类型和动态供给 | 集群级别 |

理解 PV 的生命周期（供给 → 绑定 → 使用 → 回收）和访问模式（RWO/ROX/RWX）是正确使用持久化存储的关键。

对于大多数应用，**使用 StorageClass 进行动态供给**是最推荐的方式，可以让开发人员专注于应用本身，而不需要关心底层存储的细节。
