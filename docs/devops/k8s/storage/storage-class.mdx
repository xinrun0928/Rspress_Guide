# K8s 动态存储供给：StorageClass 完全指南

应用要持久化存储，但运维来不及创建 PV？——K8s 动态存储供给的正确打开方式

在传统的存储管理模式下，开发人员需要找运维：「我需要一个 100GiB 的存储卷，能给我创建一个吗？」

运维登录服务器，手动创建 PV，配置存储后端，分配容量……这一套流程下来，可能需要几个小时甚至几天。

有没有更高效的方式？

答案就是 **StorageClass**——K8s 的动态存储供给机制。


## 一、StorageClass 的核心思想

StorageClass 是 K8s 对存储类型的抽象。它告诉 K8s「当有人申请存储时，应该用什么方式、什么存储后端来创建存储卷」。

简单来说，StorageClass 实现了一个**自动化的存储工厂**：

- 管理员定义 StorageClass（配置存储类型、供给插件、参数等）
- 开发人员创建 PVC（只需声明需要多大存储、什么访问模式）
- K8s 自动创建 PV（根据 StorageClass 的配置动态生成）

```
┌────────────────────────────────────────────────────────┐
│                  StorageClass 工作流程                  │
│                                                         │
│  1. 管理员创建 StorageClass                             │
│     ↓                                                   │
│  2. 开发人员创建 PVC（请求 10Gi 存储）                  │
│     ↓                                                   │
│  3. K8s Controller 根据 StorageClass 自动创建 PV        │
│     ↓                                                   │
│  4. PV 与 PVC 绑定                                      │
│     ↓                                                   │
│  5. Pod 使用 PVC                                       │
│                                                         │
└────────────────────────────────────────────────────────┘
```

## 二、StorageClass 的结构

```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: standard
provisioner: kubernetes.io/gce-pd
parameters:
  type: pd-standard
  replication-type: regional-pd
reclaimPolicy: Retain
allowVolumeExpansion: true
mountOptions:
  - debug
volumeBindingMode: Immediate
```

### 2.1 核心字段解析

| 字段 | 说明 |
|---|---|
| `provisioner` | 存储供给插件，决定使用什么存储后端 |
| `parameters` | 传递给 provisioner 的参数，不同 provisioner 有不同的参数 |
| `reclaimPolicy` | 默认 `Delete`，可选 `Retain` |
| `allowVolumeExpansion` | 是否允许在线扩容，默认 `false` |
| `mountOptions` | 挂载选项，会传递给 PV |
| `volumeBindingMode` | 绑定模式，`Immediate` 或 `WaitForFirstConsumer` |

### 2.2 provisioner（供给插件）

provisioner 是存储供给的执行者，K8s 内置了多种 provisioner：

| Provisioner | 存储后端 | 访问模式 |
|---|---|---|
| `kubernetes.io/gce-pd` | Google Cloud Persistent Disk | RWO |
| `kubernetes.io/aws-ebs` | AWS Elastic Block Store | RWO |
| `kubernetes.io/azure-disk` | Azure Disk | RWO |
| `kubernetes.io/azure-file` | Azure File | RWO/RWX |
| `kubernetes.io/vsphere-volume` | vSphere VMDK | RWO |
| `kubernetes.io/nfs` | NFS Server | RWO/ROX/RWX |
| `kubernetes.io/hostpath` | HostPath <Badge text="仅测试" type="warning" /> | RWO |

除了内置的 provisioner，还有大量**第三方 CSI 驱动**：

| CSI Driver | 存储后端 |
|---|---|
| `ebs.csi.aws.com` | AWS EBS |
| `pd.csi.storage.gke.io` | Google Cloud PD |
| `disk.csi.azure.com` | Azure Disk |
| `cephfs.csi.ceph.com` | CephFS |
| `rbd.csi.ceph.com` | Ceph Block Device |
| `nfs.csi.k8s.io` | NFS |
| `hostpath.csi.k8s.io` | HostPath |

## 三、常用 StorageClass 示例

### 3.1 AWS EBS

```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: ebs-standard
provisioner: ebs.csi.aws.com
parameters:
  type: gp3
  throughput: "125"
  iops: "3000"
reclaimPolicy: Retain
volumeBindingMode: WaitForFirstConsumer
allowVolumeExpansion: true
```

**参数说明**：

| 参数 | 可选值 | 说明 |
|---|---|---|
| `type` | `gp2`, `gp3`, `io1`, `io2`, `st1`, `sc1` | EBS 卷类型 |
| `iops` | 数值 | IOPS 配置（gp3/io1/io2 需要） |
| `throughput` | 数值 MB/s | 吞吐量配置（gp3/st1/sc1 需要） |
| `encrypted` | `true`, `false` | 是否加密 |
| `kmsKeyId` | KMS Key ID | 加密使用的密钥 |

### 3.2 Google Cloud Persistent Disk

```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: gce-pd-standard
provisioner: pd.csi.storage.gke.io
parameters:
  type: pd-standard
  replication-type: regional-pd
reclaimPolicy: Retain
volumeBindingMode: WaitForFirstConsumer
allowVolumeExpansion: true
```

**参数说明**：

| 参数 | 可选值 | 说明 |
|---|---|---|
| `type` | `pd-standard`, `pd-ssd`, `pd-balanced` | PD 卷类型 |
| `replication-type` | `regional-pd`, `zonal` | 区域复制或单区域 |

### 3.3 NFS（使用 nfs-subdir-external-provisioner）

```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: nfs-client
provisioner: k8s-sigs.io/nfs-subdir-external-provisioner
parameters:
  archiveOnDelete: "false"
  onDelete: "delete"
  pathPattern: "${.PVC.namespace}/${.PVC.name}"
  basePath: "/k8s-pvcs"
reclaimPolicy: Retain
volumeBindingMode: Immediate
allowVolumeExpansion: true
mountOptions:
  - nfsvers=4.1
```

## 四、Volume Binding Mode：关键配置

`volumeBindingMode` 控制何时创建 PV 和绑定 PVC，这是一个**非常重要的配置**。

### 4.1 Immediate（立即绑定）

```yaml
volumeBindingMode: Immediate
```

当 PVC 创建时，K8s 立即创建 PV 并绑定，不管 Pod 是否被调度。

**特点**：

- PVC 创建后立即进入 Bound 状态
- 可能导致 Pod 调度失败（节点在不同的可用区）
- 适合存储与 Pod 调度无关的场景

:::warning 跨可用区问题
使用云厂商块存储时，如果 `volumeBindingMode: Immediate`，K8s 可能会创建一个与 Pod 所在节点不同可用区的 PV，导致 Pod 无法挂载存储。

例如：Pod 调度到 us-east-1a，但 PV 创建在 us-east-1b。
:::

### 4.2 WaitForFirstConsumer（延迟绑定）

```yaml
volumeBindingMode: WaitForFirstConsumer
```

当 PVC 创建时，**不立即创建 PV**，直到有 Pod 使用该 PVC 并被调度到某个节点。

**特点**：

- Pod 调度完成后才创建 PV
- K8s 会考虑 Pod 的调度位置，创建匹配的 PV
- **强烈推荐用于云厂商块存储**

```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: ebs-storage
provisioner: ebs.csi.aws.com
volumeBindingMode: WaitForFirstConsumer
# ...
```

**工作流程**：

```
1. 创建 PVC
   → PVC 状态为 Pending（不立即绑定）

2. 创建使用该 PVC 的 Pod
   → K8s 发现 Pod 引用了 Pending 的 PVC

3. Pod 被调度到某个节点（假设 us-east-1a）
   → K8s 在 us-east-1a 创建 PV

4. PV 与 PVC 绑定
   → PVC 状态变为 Bound

5. Pod 开始运行
   → 存储卷挂载成功
```

## 五、默认 StorageClass

集群中可以设置一个**默认 StorageClass**，用户创建 PVC 时无需指定 `storageClassName`。

```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: standard
  annotations:
    storageclass.kubernetes.io/is-default-class: "true"
provisioner: kubernetes.io/gce-pd
parameters:
  type: pd-standard
```

```bash
# 查看默认 StorageClass
kubectl get storageclass

# NAME                 PROVISIONER                    DEFAULT
# standard (default)   kubernetes.io/gce-pd           true
# fast                 ebs.csi.aws.com                false
```

:::tip 选择默认 StorageClass
如果集群有多个 StorageClass，应该设置一个合理的默认项。用户可以显式指定 `storageClassName` 覆盖默认值。
:::

## 六、CSI：存储接口标准

### 6.1 什么是 CSI？

CSI（Container Storage Interface）是 K8s 与存储供应商之间的**标准化接口**。

在 CSI 之前，K8s 内置的存储驱动（如 `kubernetes.io/aws-ebs`）需要编译进 K8s 源码，每次更新都需要 K8s 版本迭代。

CSI 的出现改变了这一切：

- 存储供应商以独立的 CSI 驱动运行
- 通过标准的 gRPC 接口与 K8s 通信
- 与 K8s 核心代码解耦，可以独立更新
- 供应商可以在任何支持 CSI 的容器编排平台上使用

```
┌─────────────────────────────────────────────────────────┐
│                    CSI 架构                             │
│                                                         │
│   ┌─────────────┐         ┌─────────────────────┐       │
│   │   K8s API   │ ────── │  CSI External       │       │
│   │   Server    │        │  Provisioner        │       │
│   └─────────────┘        └──────────┬──────────┘       │
│                                    │ gRPC             │
│   ┌─────────────┐         ┌────────┴────────┐        │
│   │    Kubelet  │ ────── │   CSI Node       │        │
│   │             │        │   Plugin         │        │
│   └─────────────┘        └────────┬────────┘        │
│                                    │                 │
│                         ┌──────────┴──────────┐       │
│                         │   存储后端           │       │
│                         │ (EBS/GCE/Ceph...)  │       │
│                         └─────────────────────┘       │
└─────────────────────────────────────────────────────────┘
```

### 6.2 CSI vs In-Tree

| 特性 | In-Tree | CSI |
|---|---|---|
| 代码位置 | K8s 核心代码 | 独立项目 |
| 更新周期 | 跟随 K8s 版本 | 独立发布 |
| 功能更新 | 需要 K8s 发版 | 随时更新 |
| 社区支持 | 官方维护 | 供应商/社区维护 |
| 稳定性 | 成熟稳定 | 逐步成熟 |

**建议**：新部署的存储优先使用 CSI 驱动，In-Tree 驱动正在逐步被废弃。

### 6.3 常用 CSI 驱动安装

**AWS EBS CSI Driver**：

```bash
kubectl apply -k "github.com/kubernetes-sigs/aws-ebs-csi-driver/deploy/kubernetes/overlays/stable/?ref=master"
```

**GCP PD CSI Driver**：

```bash
kubectl apply -f https://raw.githubusercontent.com/kubernetes-sigs/gcp-pd-csi-driver/master/deploy/kubernetes/base/controller_stable.yaml
kubectl apply -f https://raw.githubusercontent.com/kubernetes-sigs/gcp-pd-csi-driver/master/deploy/kubernetes/base/node.yaml
```

**NFS CSI Driver**：

```bash
kubectl apply -f https://raw.githubusercontent.com/kubernetes-csi/csi-driver-nfs/master/deploy/kubernetes/latest/csi-nfs-driver.yaml
```

## 七、实战：完整的存储配置

### 7.1 环境准备：安装 CSI Driver

```bash
# 安装 AWS EBS CSI Driver
curl -fsSL https://raw.githubusercontent.com/kubernetes-sigs/aws-ebs-csi-driver/master/deploy/kubernetes/overlays/stable/gcr/kustomization.yaml -o kustomization.yaml
kubectl apply -k .

# 检查驱动状态
kubectl get pods -n kube-system -l app=ebs-csi-controller
kubectl get csidriver
```

### 7.2 创建 StorageClass

```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: ebs-gp3
provisioner: ebs.csi.aws.com
parameters:
  type: gp3
  encrypted: "true"
reclaimPolicy: Retain
volumeBindingMode: WaitForFirstConsumer
allowVolumeExpansion: true
mountOptions:
  - debug
```

### 7.3 创建 PVC

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: app-storage
spec:
  accessModes:
    - ReadWriteOnce
  storageClassName: ebs-gp3
  resources:
    requests:
      storage: 50Gi
```

```bash
# 查看 PVC 状态（未绑定 Pod 前应该是 Pending）
kubectl get pvc app-storage

# NAME          STATUS    VOLUME   CAPACITY   ACCESS MODES   AGE
# app-storage   Pending                                      5s
```

### 7.4 在 Pod 中使用

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: app
spec:
  containers:
  - name: app
    image: nginx
    volumeMounts:
    - name: persistent-storage
      mountPath: /data
  volumes:
  - name: persistent-storage
    persistentVolumeClaim:
      claimName: app-storage
```

```bash
# 创建 Pod 后，PVC 应该变为 Bound
kubectl get pvc app-storage

# NAME          STATUS   VOLUME                                     CAPACITY   ACCESS MODES   AGE
# app-storage   Bound    pvc-12345678-abcd-1234-5678-1234567890ab   50Gi       RWO            30s
```

## 八、扩容与快照

### 8.1 在线扩容

<Badge text="K8s 1.24+" type="tip" /> K8s 1.24+ 默认启用 StorageClass 扩容控制器。

```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: ebs-gp3
provisioner: ebs.csi.aws.com
allowVolumeExpansion: true
# volumeBindingMode: WaitForFirstConsumer
```

**扩容流程**：

```bash
# 1. 修改 PVC 容量
kubectl patch pvc app-storage -p '{"spec":{"resources":{"requests":{"storage":"100Gi"}}}}'

# 2. 查看 PVC 状态
kubectl get pvc app-storage

# 3. 如果文件系统需要扩展，Kubelet 会自动处理
#    对于 XFS/ext4 等支持在线扩展的文件系统，无需重启 Pod
```

:::warning 扩容限制
- 只能扩容，不能缩容
- 部分存储类型不支持扩容（如 GCE PD 的某些配置）
- 删除 PVC 后，新 PVC 可能无法使用相同的 Volume ID
:::

### 8.2 Volume Snapshot（快照）

CSI 支持创建存储卷快照：

```yaml
apiVersion: snapshot.storage.k8s.io/v1
kind: VolumeSnapshotClass
metadata:
  name: csi-aws-vsc
driver: ebs.csi.aws.com
deletionPolicy: Delete
parameters:
  description: "EBS CSI Snapshot Class"
---
apiVersion: snapshot.storage.k8s.io/v1
kind: VolumeSnapshot
metadata:
  name: app-snapshot
spec:
  volumeSnapshotClassName: csi-aws-vsc
  source:
    persistentVolumeClaimName: app-storage
```

从快照恢复：

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: app-storage-restored
spec:
  storageClassName: ebs-gp3
  dataSource:
    name: app-snapshot
    kind: VolumeSnapshot
    apiGroup: snapshot.storage.k8s.io
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 50Gi
```

## 九、常见问题与排查

### 9.1 PVC 一直 Pending

可能原因：

```bash
# 1. StorageClass 不存在
kubectl get storageclass

# 2. CSI Driver 未安装或运行异常
kubectl get pods -n kube-system -l app=csi

# 3. 查看 Controller 日志
kubectl logs -n kube-system -l app=ebs-csi-controller -c csi-provisioner

# 4. 检查存储配额
#    云厂商账户可能有存储卷数量或容量限制
```

### 9.2 Pod 无法挂载卷

```bash
# 1. 检查 PVC 状态
kubectl describe pvc <pvc-name>

# 2. 检查 Pod 事件
kubectl describe pod <pod-name>

# 3. 检查 Node 是否有 CSI Node Plugin
kubectl get nodes -o jsonpath='{.items[*].status.volumesAttached}'

# 4. 检查 CSI Driver 日志
kubectl logs -n kube-system -l app=csi-node -c csi-plugin
```

## 十、总结

StorageClass 是 K8s 动态存储供给的核心：

| 组件 | 作用 |
|---|---|
| StorageClass | 定义存储类型和供给配置 |
| Provisioner | 执行实际的存储创建 |
| VolumeBindingMode | 控制绑定时机 |
| CSI | 标准化的存储接口 |

**最佳实践**：

1. **使用 `WaitForFirstConsumer`**：避免跨可用区问题
2. **设置 `allowVolumeExpansion: true`**：支持在线扩容
3. **生产环境使用 Retain 策略**：防止误删数据
4. **优先使用 CSI 驱动**：功能更新独立于 K8s 版本
5. **设置默认 StorageClass**：简化 PVC 创建

掌握 StorageClass，你就掌握了 K8s 存储的"自动化流水线"——让存储供给像拧开水龙头一样简单。
