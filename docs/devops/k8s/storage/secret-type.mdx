# K8s Secret 类型完全指南

Secret 不只是存密码的——K8s Secret 类型详解

一提到 Secret，你可能会想到"就是存密码的地方"。但 K8s 的 Secret 远不止于此。

Secret 有多种**类型（Type）**，每种类型都有特定的结构和用途。从 Docker 镜像仓库凭证，到 TLS 证书，再到 ServiceAccount Token……不同的场景，应该用不同类型的 Secret。

今天，我们把 Secret 的类型讲透。

## 一、Secret 类型概览

K8s 内置了多种 Secret 类型，方便不同场景下的使用：

| 类型 | 用途 | 字段结构 |
|---|---|---|
| `Opaque` | 通用键值对（默认） | `data` 或 `stringData` |
| `kubernetes.io/dockerconfigjson` | Docker 镜像仓库凭证 | `.dockerconfigjson` |
| `kubernetes.io/tls` | TLS 证书和私钥 | `tls.crt`, `tls.key` |
| `kubernetes.io/service-account-token` | ServiceAccount Token <Badge text="Deprecated" type="danger" /> | `token`, `namespace`, `ca.crt` |
| `kubernetes.io/basic-auth` | 基本认证凭证 | `username`, `password` |
| `kubernetes.io/ssh-auth` | SSH 认证 | `ssh-privatekey` |

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: my-secret
type: Opaque  # 类型声明
data:
  key: value
```

## 二、Opaque：通用的键值对

`Opaque` 是 Secret 的默认类型，可以存储任意键值对。

### 2.1 创建方式

**命令行**：

```bash
kubectl create secret generic app-secret \
  --from-literal=DB_USER=admin \
  --from-literal=DB_PASSWORD=password123
```

**YAML**：

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: app-secret
type: Opaque
data:
  # Base64 编码的值
  DB_USER: YWRtaW4=
  DB_PASSWORD: cGFzc3dvcmQxMjM=
```

**使用 `stringData`（自动编码）**：

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: app-secret
type: Opaque
stringData:
  # stringData 会自动 Base64 编码
  # 存储时会转为 data 字段
  DB_USER: admin
  DB_PASSWORD: password123
  API_KEY: "sk-1234567890abcdef"
```

:::tip stringData 的用途
`stringData` 是写入时的便利字段，K8s 会自动将其转换为 `data` 中的 Base64 值。读取时会自动解码，所以实际使用和普通环境变量一样。
:::

### 2.2 Pod 使用

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  containers:
  - name: app
    image: my-app:latest
    env:
    - name: DB_PASSWORD
      valueFrom:
        secretKeyRef:
          name: app-secret
          key: DB_PASSWORD
```

## 三、kubernetes.io/dockerconfigjson：镜像仓库凭证

如果你的镜像存储在私有仓库（如 Docker Hub、Harbor、ECR、GCR），Pod 拉取镜像时需要认证。

`dockerconfigjson` 类型的 Secret 存储 Docker 镜像仓库的认证信息。

### 3.1 创建方式

**方式一：命令行**

```bash
kubectl create secret docker-registry my-registry \
  --docker-server=registry.example.com \
  --docker-username=admin \
  --docker-password=password123 \
  --docker-email=admin@example.com
```

**方式二：从已有的 Docker config 文件创建**

```bash
# 登录后，Docker 会在 ~/.docker/config.json 保存凭证
cat ~/.docker/config.json

# 从文件创建 Secret
kubectl create secret generic my-registry \
  --from-file=.dockerconfigjson=~/.docker/config.json \
  --type=kubernetes.io/dockerconfigjson
```

**方式三：手动 YAML**

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: my-registry
  namespace: default
type: kubernetes.io/dockerconfigjson
data:
  # ~/.docker/config.json 的 Base64 编码
  .dockerconfigjson: ewoJImF1dGhzIjogewoJCSJyZWdpc3RyeS5leGFtcGxlLmNvbSI6IHsKCQkJImF1dGgiOiAiZFhObGMzQmc9PSIsCgkJCSJ1c2VybmFtZSI6ICJhZG1pbiIsCgkJCSJwYXNzd29yZCI6ICJwYXNzd29yZDEyMyIsCgkJCSJlbWFpbCI6ICJhZG1pbkBleGFtcGxlLmNvbSIKCQl9Cgl9Cn0=
```

### 3.2 结构解析

`.dockerconfigjson` 的内容格式如下：

```json
{
  "auths": {
    "registry.example.com": {
      "auth": "YWRtaW46cGFzc3dvcmQxMjM=",
      "email": "admin@example.com"
    },
    "https://index.docker.io/v1/": {
      "auth": "dXNlcm5hbWU6cGFzc3dvcmQ="
    }
  }
}
```

| 字段 | 说明 |
|---|---|
| `auths` | 认证信息对象，key 是镜像仓库地址 |
| `auth` | Base64 编码的 `username:password` |
| `email` | 邮箱（可选） |

### 3.3 Pod 使用镜像仓库凭证

**方式一：Pod 级别指定**

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  imagePullSecrets:
  - name: my-registry
  containers:
  - name: app
    image: registry.example.com/my-app:latest
```

**方式二：ServiceAccount 级别指定（推荐）**

```yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: app-sa
  namespace: default
imagePullSecrets:
- name: my-registry
---
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  serviceAccountName: app-sa
  containers:
  - name: app
    image: registry.example.com/my-app:latest
```

:::tip 为什么推荐 ServiceAccount 方式？
在 ServiceAccount 级别设置 `imagePullSecrets`，所有使用该 ServiceAccount 的 Pod 都会自动使用这些凭证，无需每个 Pod 单独配置。
:::

### 3.4 多镜像仓库认证

一个 Secret 可以包含多个镜像仓库的凭证：

```bash
# 登录多个仓库
docker login registry1.example.com
docker login registry2.example.com

# 导出配置
cat ~/.docker/config.json
```

生成的 `.dockerconfigjson` 包含多个仓库的认证信息：

```json
{
  "auths": {
    "registry1.example.com": { "auth": "..." },
    "registry2.example.com": { "auth": "..." },
    "docker.io": { "auth": "..." }
  }
}
```

## 四、kubernetes.io/tls：TLS 证书

`tls` 类型的 Secret 用于存储 TLS 证书和私钥，适合 HTTPS 场景。

### 4.1 创建方式

**方式一：命令行（从证书文件）**

```bash
# 已有证书文件
kubectl create secret tls my-tls \
  --cert=path/to/tls.crt \
  --key=path/to/tls.key
```

**方式二：使用 cert-manager 自动管理**（推荐）

```yaml
apiVersion: cert-manager.io/v1
kind: Issuer
metadata:
  name: letsencrypt-prod
spec:
  acme:
    server: https://acme-v02.api.letsencrypt.org/directory
    email: admin@example.com
    privateKeySecretRef:
      name: letsencrypt-account-key
    solvers:
    - http01:
        ingress:
          class: nginx
---
apiVersion: cert-manager.io/v1
kind: Certificate
metadata:
  name: my-cert
spec:
  secretName: my-tls
  issuerRef:
    name: letsencrypt-prod
  dnsNames:
  - example.com
  - www.example.com
```

### 4.2 结构解析

`tls` 类型的 Secret 有两个固定字段：

| 字段 | 说明 | 文件 |
|---|---|---|
| `tls.crt` | 证书链 | 包含服务器证书和中间 CA |
| `tls.key` | 私钥 | 私钥文件内容 |

```bash
# 查看 TLS Secret 内容
kubectl get secret my-tls -o yaml
```

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: my-tls
type: kubernetes.io/tls
data:
  tls.crt: LS0tLS1CRUdJTiBDRVJUS...
  tls.key: LS0tLS1CRUdJTiBQUklW...
```

### 4.3 Pod 使用 TLS 证书

**方式一：Ingress 使用**

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-ingress
  annotations:
    kubernetes.io/ingress.class: nginx
    cert-manager.io/issuer: letsencrypt-prod
spec:
  tls:
  - hosts:
    - example.com
    - www.example.com
    secretName: my-tls
  rules:
  - host: example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: my-service
            port:
              number: 80
```

**方式二：应用读取证书**

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: https-app
spec:
  containers:
  - name: app
    image: my-app:latest
    volumeMounts:
    - name: tls-cert
      mountPath: /etc/ssl/certs
      readOnly: true
  volumes:
  - name: tls-cert
    secret:
      secretName: my-tls
      items:
      - key: tls.crt
        path: server.crt
      - key: tls.key
        path: server.key
```

## 五、kubernetes.io/service-account-token：服务账号令牌

<Badge text="Deprecated" type="danger" /> 此 Secret 类型已被废弃，不推荐使用。

`service-account-token` 类型的 Secret 曾经用于存储 ServiceAccount 的认证令牌。

### 5.1 为什么废弃？

1. **安全风险**：Secret 存储在 etcd 中，未加密时令牌可被任何人读取
2. **令牌永不过期**：直到 Secret 被删除
3. **版本问题**：Pod 内挂载的令牌不是最新的

### 5.2 推荐替代：Projected Volume

<Badge text="K8s 1.29+" type="tip" /> K8s 推荐使用 **Projected Volume** 或 **TokenRequest API** 获取短生命周期的 ServiceAccount Token。

**使用 TokenRequest（推荐）**：

```yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: app-sa
automountServiceAccountToken: false  # 禁用自动挂载
---
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  serviceAccountName: app-sa
  containers:
  - name: app
    image: my-app:latest
    volumeMounts:
    - name: token
      mountPath: /var/run/secrets/tokens
  volumes:
  - name: token
    projected:
      sources:
      - serviceAccountToken:
          audience: api
          expirationSeconds: 3600
          path: token
      - configMap:
          name: kube-root-ca.crt
          items:
          - key: ca.crt
            path: ca.crt
```

**特点**：

| 特性 | ServiceAccount Token Secret | TokenRequest Projected |
|---|---|---|
| 生命周期 | 永久（直到手动删除） | 短周期（可配置，如 1 小时） |
| 续期 | 不会自动续期 | 自动续期 |
| 存储位置 | etcd | 仅在内存中（tmpfs） |
| 安全性 | 较低 | 较高 |

## 六、其他 Secret 类型

### 6.1 kubernetes.io/basic-auth：基本认证

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: basic-auth-secret
type: kubernetes.io/basic-auth
data:
  username: YWRtaW4=
  password: cGFzc3dvcmQ=
```

```bash
# 创建命令
kubectl create secret generic basic-auth \
  --from-literal=username=admin \
  --type=kubernetes.io/basic-auth
```

### 6.2 kubernetes.io/ssh-auth：SSH 认证

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: ssh-auth-secret
type: kubernetes.io/ssh-auth
data:
  ssh-privatekey: |
    -----BEGIN RSA PRIVATE KEY-----
    ...
    -----END RSA PRIVATE KEY-----
```

```bash
# 创建命令
kubectl create secret generic ssh-auth \
  --from-file=ssh-privatekey=path/to/id_rsa \
  --type=kubernetes.io/ssh-auth
```

## 七、Secret 加密与安全

### 7.1 etcd 加密存储

默认情况下，etcd 以明文存储所有数据，包括 Secret。要启用加密：

```yaml
# encryption-config.yaml
apiVersion: apiserver.config.k8s.io/v1
kind: EncryptionConfiguration
resources:
  - resources:
    - secrets
    providers:
    - aescbc:
        keys:
        - name: key1
          secret: <base64-encoded-key>
    - identity: {}
```

**启动 API Server 时指定配置**：

```bash
--encryption-provider-config=/path/to/encryption-config.yaml
```

:::warning 加密范围
只加密新创建的 Secret，已有 Secret 需要重新创建才能加密。
:::

### 7.2 KMS Provider

<Badge text="K8s 1.10+" type="tip" /> 使用云厂商的 KMS（Key Management Service）加密 Secret：

```yaml
apiVersion: apiserver.config.k8s.io/v1
kind: EncryptionConfiguration
resources:
  - resources:
    - secrets
    providers:
    - kms:
        name: my-kms-provider
        endpoint: unix:///tmp/kms-provider.sock
        timeout: 3s
    - identity: {}
```

**优势**：

- 无需在集群中存储加密密钥
- 密钥轮换由云平台管理
- 符合合规要求

### 7.3 RBAC 控制 Secret 访问

通过 RBAC 限制谁可以读取 Secret：

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: secret-reader
  namespace: production
rules:
- apiGroups: [""]
  resources: ["secrets"]
  verbs: ["get", "list"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: secret-reader-binding
  namespace: production
subjects:
- kind: User
  name: developer
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: secret-reader
  apiGroup: rbac.authorization.k8s.io
```

**最佳实践**：

```yaml
# 不要这样做：授予所有用户读取所有 Secret 的权限
rules:
- apiGroups: [""]
  resources: ["secrets"]
  verbs: ["get", "list", "watch"]  # 太宽泛

# 正确做法：仅授予特定命名空间的读取权限
rules:
- apiGroups: [""]
  resources: ["secrets"]
  resourceNames: ["app-secrets", "db-credentials"]  # 指定具体 Secret
  verbs: ["get"]
```

## 九、常见问题与最佳实践

### 9.1 Secret vs ConfigMap：到底用哪个？

| 场景 | 推荐类型 |
|---|---|
| 数据库密码 | Secret |
| API Key / Token | Secret |
| TLS 证书 | `kubernetes.io/tls` |
| Docker 镜像仓库凭证 | `kubernetes.io/dockerconfigjson` |
| 应用配置（非敏感） | ConfigMap |
| 环境变量（非敏感） | ConfigMap |
| 配置文件 | ConfigMap |

### 9.2 Secret 大小限制

Secret 的大小限制是 **1MB**，但实际使用中建议保持小规模：

- 单个 Secret 不应超过 1KB
- 如果需要存储大文件，考虑对象存储

### 9.3 Secret 泄漏检测

```bash
# 定期扫描集群中的 Secret 使用
kubectl get secrets --all-namespaces -o json | jq '.items[] | select(.type=="Opaque") | .metadata.name'

# 启用审计日志记录 Secret 访问
# 在 audit-policy.yaml 中配置
- level: Metadata
  resources:
  - group: ""
    resources: ["secrets"]
```

### 9.4 多环境 Secret 管理

使用 Kustomize 或 Helm 管理不同环境的 Secret：

```yaml
# base/secret.yaml
apiVersion: v1
kind: Secret
metadata:
  name: app-secrets
type: Opaque
stringData:
  ENV: base

---
# overlays/production/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization

resources:
- ../../base/secret.yaml

patches:
- path: secret-patch.yaml
  target:
    name: app-secrets
```

```yaml
# overlays/production/secret-patch.yaml
apiVersion: v1
kind: Secret
metadata:
  name: app-secrets
stringData:
  ENV: production
  DATABASE_URL: "postgresql://prod-db:5432/prod"
```

## 十、Secret 类型总结

| 类型 | 类型值 | 使用场景 |
|---|---|---|
| Opaque | `Opaque`（默认） | 通用键值对，数据库密码、API Key |
| Docker Registry | `kubernetes.io/dockerconfigjson` | 私有镜像仓库认证 |
| TLS Certificate | `kubernetes.io/tls` | HTTPS 证书和私钥 |
| Basic Auth | `kubernetes.io/basic-auth` | HTTP 基本认证 |
| SSH Auth | `kubernetes.io/ssh-auth` | SSH 公钥认证 |
| ServiceAccount Token | `kubernetes.io/service-account-token` | **已废弃**，使用 Projected Token |

**安全建议**：

1. **启用 etcd 加密**或使用 **KMS Provider**
2. 使用 **RBAC** 限制 Secret 访问
3. **禁用自动挂载 ServiceAccount Token**，使用 **TokenRequest**
4. **不要将 Secret 提交到 Git**，使用 Sealed Secrets 或 Vault
5. 定期**轮换敏感凭证**

掌握 Secret 的类型和使用方法，你就掌握了 K8s 敏感数据管理的核心——让安全成为默认，而不是事后补救。
