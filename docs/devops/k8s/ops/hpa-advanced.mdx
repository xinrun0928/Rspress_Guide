# HPA 进阶：自定义指标与智能弹性伸缩

CPU 和内存是 HPA 最常用的指标，但真实生产环境往往需要更智能的伸缩策略。比如：消息队列积压了多少条消息？Redis 连接数是否过高？外部 API 的响应时间是否劣化？

这些业务指标 CPU 无法反映。本质问题是：**基于资源指标的 HPA 不够「懂」业务**。

K8s 提供了 Custom Metrics API 和 External Metrics API，支持基于任意指标的弹性伸缩。

## HPA 基础回顾

### 标准 HPA 配置

```yaml title="基础 HPA"
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: nginx-hpa
  namespace: production
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: nginx
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
  - type: Resource
    resource:
      name: memory
      target:
        type: AverageValue
        averageValue: 500Mi
  behavior:
    scaleDown:
      stabilizationWindowSeconds: 300
      policies:
      - type: Percent
        value: 10
        periodSeconds: 60
    scaleUp:
      stabilizationWindowSeconds: 0
      policies:
      - type: Percent
        value: 100
        periodSeconds: 15
```

## 自定义指标 HPA

### 配置 Prometheus 指标

首先，在 Prometheus 中定义应用指标：

```yaml title="应用暴露指标"
apiVersion: apps/v1
kind: Deployment
metadata:
  name: queue-worker
  namespace: production
spec:
  replicas: 2
  template:
    spec:
      containers:
      - name: worker
        image: queue-worker:latest
        ports:
        - name: metrics
          containerPort: 9090
        resources:
          requests:
            cpu: 100m
            memory: 128Mi
---
apiVersion: v1
kind: Service
metadata:
  name: queue-worker
  namespace: production
  annotations:
    prometheus.io/scrape: "true"
    prometheus.io/port: "9090"
spec:
  selector:
    app: queue-worker
  ports:
  - name: http
    port: 8080
  - name: metrics
    port: 9090
```

### Prometheus Adapter 配置

```yaml title="prometheus-adapter 配置"
apiVersion: v1
kind: ConfigMap
metadata:
  name: prometheus-adapter-config
  namespace: monitoring
data:
  config.yaml: |
    rules:
    # 应用自定义指标
    - seriesQuery: 'queue_messages_pending{namespace!="",pod!=""}'
      resources:
        overrides:
          namespace:
            resource: namespace
          pod:
            resource: pod
      name:
        matches: "^(.*)_pending"
        as: "${1}_count"
      metricsQuery: sum(<<.Series>>{<<.LabelMatchers>>}) by (<<.GroupBy>>)
    # Pod 级别指标
    - seriesQuery: 'queue_messages_pending{namespace!=""}'
      resources:
        overrides:
          namespace: {}
      name:
        matches: "^(.*)_pending"
        as: "${1}_pending_total"
      metricsQuery: sum(<<.Series>>{<<.LabelMatchers>>}) by (namespace)
```

### HPA 使用自定义指标

```yaml title="基于队列深度的 HPA"
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: queue-worker-hpa
  namespace: production
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: queue-worker
  minReplicas: 2
  maxReplicas: 20
  metrics:
  - type: Pods
    pods:
      metric:
        name: queue_messages_pending
      target:
        type: AverageValue
        averageValue: "1000"  # 每个 Pod 平均积压 1000 条消息
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 0  # 快速扩容
      policies:
      - type: Pods
        value: 4
        periodSeconds: 15
    scaleDown:
      stabilizationWindowSeconds: 600  # 缩容需要等待 10 分钟
      policies:
      - type: Pods
        value: 2
        periodSeconds: 300
```

## 外部指标 HPA

### 外部指标场景

外部指标适用于 K8s 集群外部的资源：

| 场景 | 指标 | 伸缩策略 |
|---|---|---|
| 消息队列 | 积压消息数 | 消费者数量 |
| 数据库 | 连接池使用率 | 应用副本数 |
| 云服务 | SQS 队列深度 | Worker 数量 |
| 外部 API | 响应时间 | Proxy 副本数 |

### 配置外部指标

```yaml title="外部指标 Provider"
apiVersion: v1
kind: ConfigMap
metadata:
  name: prometheus-adapter-config
  namespace: monitoring
data:
  config.yaml: |
    rules:
    external:
    # AWS SQS 队列深度
    - seriesQuery: 'aws_sqs_approximate_number_of_messages_visible{queue_name!=""}'
      resources:
        overrides:
          queue_name:
            resource: externalLabel/queue_name
      name:
        matches: "aws_sqs_approximate_number_of_messages_visible"
        as: "sqs_messages_count"
      metricsQuery: max(<<.Series>>) by (<<.ExternalLabelMatchers>>)
```

```yaml title="基于 SQS 队列深度的 HPA"
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: worker-hpa
  namespace: production
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: sqs-worker
  minReplicas: 1
  maxReplicas: 50
  metrics:
  - type: External
    external:
      metric:
        name: sqs_messages_count
        selector:
          matchLabels:
            queue_name: production-jobs
      target:
        type: AverageValue
        averageValue: "10"  # 每条消息需要 1 个 Worker
```

## 多指标组合 HPA

### 多种指标同时生效

```yaml title="多指标 HPA"
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: api-server-hpa
  namespace: production
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: api-server
  minReplicas: 3
  maxReplicas: 30
  metrics:
  # CPU 指标
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 60
  # 内存指标
  - type: Resource
    resource:
      name: memory
      target:
        type: Utilization
        averageUtilization: 70
  # 自定义指标：请求延迟
  - type: Pods
    pods:
      metric:
        name: http_request_duration_seconds_p99
      target:
        type: AverageValue
        averageValue: "500m"  # P99 延迟不超过 500ms
  # 外部指标：数据库连接
  - type: External
    external:
      metric:
        name: database_connections_active
      target:
        type: AverageValue
        averageValue: "50"  # 每个 Pod 最多 50 个连接
```

:::tip HPA 计算逻辑
HPA controller 会计算所有指标，找出需要最大副本数的指标。**取所有指标中副本数需求的最大值**。
:::

## 伸缩行为控制

### 快速扩容、缓慢缩容

```yaml title="HPA 行为配置"
spec:
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 0  # 立即扩容
      policies:
      - type: Percent
        value: 100  # 每次最多翻倍
        periodSeconds: 15
      - type: Pods
        value: 4  # 或每次最多加 4 个 Pod
        periodSeconds: 15
      selectPolicy: Max  # 取最大值的策略
    scaleDown:
      stabilizationWindowSeconds: 600  # 缩容等待 10 分钟
      policies:
      - type: Percent
        value: 10  # 每次最多减少 10%
        periodSeconds: 60
      - type: Pods
        value: 2  # 或每次最少减少 2 个
        periodSeconds: 60
      selectPolicy: Min  # 取最小值的策略
```

### CronJob + HPA：定时扩缩容

```yaml title="定时扩缩容策略"
# CronHPA Controller 或 CronJob 配合 HPA
apiVersion: batch/v1
kind: CronJob
metadata:
  name: scale-down-night
  namespace: production
spec:
  schedule: "0 2 * * *"  # 凌晨 2 点
  jobTemplate:
    spec:
      template:
        spec:
          serviceAccountName: hpa-controller
          containers:
          - name: scale
            image: bitnami/kubectl:latest
            command:
            - /bin/sh
            - -c
            - |
              kubectl patch hpa api-server --type='json' \
                -p='[{"op":"replace","path":"/spec/minReplicas","value":1}]'
          restartPolicy: OnFailure
---
apiVersion: batch/v1
kind: CronJob
metadata:
  name: scale-up-morning
  namespace: production
spec:
  schedule: "0 8 * * *"  # 早上 8 点
  jobTemplate:
    spec:
      template:
        spec:
          serviceAccountName: hpa-controller
          containers:
          - name: scale
            image: bitnami/kubectl:latest
            command:
            - /bin/sh
            - -c
            - |
              kubectl patch hpa api-server --type='json' \
                -p='[{"op":"replace","path":"/spec/minReplicas","value":5}]'
          restartPolicy: OnFailure
```

## 监控与调试

### 查看 HPA 状态

```bash
# 查看 HPA 状态
kubectl get hpa -n production

# 查看详细状态
kubectl describe hpa queue-worker-hpa -n production

# 查看当前副本数和指标
kubectl get hpa queue-worker-hpa -n production -o yaml
```

### Prometheus 查询指标

```txt
# 查询当前副本数
kube_horizontalpodautoscaler_status_current_replicas

# 查询目标副本数
kube_horizontalpodautoscaler_status_desired_replicas

# 查询指标值
queue_messages_pending
```

### 常见问题

```bash
# 问题：HPA 不工作
# 排查步骤：
# 1. 检查 HPA 状态
kubectl get hpa -A

# 2. 检查 Custom Metrics API
kubectl get --raw "/apis/custom.metrics.k8s.io/v1" | jq

# 3. 检查指标是否存在
kubectl get --raw "/apis/custom.metrics.k8s.io/v1/namespaces/production/pods/*/queue_messages_pending"

# 4. 检查 HPA controller 日志
kubectl logs -n kube-system -l app=kube-controller-manager
```

## 面试追问方向

1. **HPA 如何决定扩容时机？**
   - 收集所有指标
   - 计算每个指标需要的副本数
   - 取最大值

2. **Prometheus Adapter 的作用？**
   - 将 Prometheus 指标转换为 K8s Custom Metrics API
   - 使 HPA 可以使用任意 Prometheus 指标

3. **为什么缩容需要 stabilizationWindowSeconds？**
   - 防止指标抖动导致的频繁伸缩
   - 留出观察窗口

4. **如何实现基于外部云的弹性伸缩？**
   - External Metrics API
   - 云服务提供商的 Metrics Adapter

## 总结

HPA 进阶能力让弹性伸缩真正「懂」业务：

- **Prometheus Adapter**：将 Prometheus 指标转换为 Custom Metrics API
- **多指标组合**：CPU + 内存 + 业务指标联合决策
- **外部指标**：支持云服务、消息队列等外部资源
- **行为控制**：快速扩容、缓慢缩容

掌握这些能力，你就能实现真正智能的弹性伸缩策略。
