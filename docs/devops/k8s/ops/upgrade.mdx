# Kubernetes 集群升级：从版本管理到平滑过渡

你有没有遇到过这个问题：集群版本太旧，想升级又担心影响业务？K8s 版本升级是一个系统性问题，涉及 Etcd、控制平面、工作节点、应用兼容性。如果操作不当，轻则服务中断，重则集群故障。

**升级不是一次性的操作，而是一个持续的过程。**

## 版本偏移策略

### 官方支持策略

K8s 对版本偏移有严格要求：

| 工作节点版本 | 控制平面版本 | 支持状态 |
|---|---|---|
| v1.28 | v1.28 | 官方支持 |
| v1.27 | v1.28 | 支持（最多 2 个小版本） |
| v1.26 | v1.28 | 不支持 |
| v1.28 | v1.27 | 不支持 |

:::tip 版本偏移原则
- 控制平面不能落后于工作节点
- 工作节点最多比控制平面低 2 个小版本
- 强烈建议控制平面和工作节点版本一致

**最佳实践**：始终保持所有节点版本一致。
:::

### 版本选择建议

| 场景 | 推荐版本策略 |
|---|---|
| 新集群 | 使用最新的稳定版本（N-1 或 N） |
| 生产集群 | 使用稳定版本，延迟 1-2 个小版本升级 |
| 关键业务 | 等待第一个补丁版本发布后再升级 |

## 升级前的准备

### 1. 读取发布说明

```bash
# 查看当前版本
kubectl version
kubectl version --short

# 查看集群信息
kubectl cluster-info
kubectl get nodes -o wide
```

发布说明包含：
- 新增功能
- 废弃警告
- 破坏性变更
- API 移除列表

### 2. Etcd 版本兼容性

```bash
# 查看当前 Etcd 版本
etcd --version

# 检查 Etcd 和 K8s 版本的兼容性
# K8s 1.28 支持 Etcd 3.12.x
# K8s 1.27 支持 Etcd 3.11.x
```

### 3. 备份 Etcd

升级前必须备份 Etcd：

```bash
# 完整备份
etcdctl --endpoints=https://127.0.0.1:2379 \
  snapshot save /backup/etcd-before-upgrade.db

# 验证备份
etcdctl snapshot status /backup/etcd-before-upgrade.db
```

### 4. 检查资源状态

```bash
# 检查 Pod 状态
kubectl get pods -A | grep -v Running

# 检查节点状态
kubectl get nodes -o wide

# 检查 etcd 健康
etcdctl --endpoints=https://127.0.0.1:2379 endpoint health
```

## kubeadm 升级流程

### 升级控制平面

```bash
# 1. 升级 kubeadm
apt-get update && apt-get install -y kubeadm=1.28.0

# 2. 验证下载
kubeadm version

# 3. 升级控制平面组件
kubeadm upgrade apply v1.28.0

# 4. 查看升级计划（不实际执行）
kubeadm upgrade plan v1.28.0
```

```bash
# 升级输出示例
[upgrade] The controller-manager RBAC rules have been updated.
[upgrade] The scheduler RBAC rules have been updated.
[upgrade] The scheduler config has been updated.
[upgrade] The API server certificates have been updated.
[upgrade] You can now join additional control plane nodes.

[upgrade/successful] SUCCESS! Your cluster can be upgraded to "v1.28.0"
```

### 升级 Master 节点

```bash
# 驱逐 Master 上的 Pod
kubectl drain master-1 --ignore-daemonsets --delete-emptydir-data

# 升级 kubelet
apt-get install -y kubelet=1.28.0 kubectl=1.28.0

# 重启 kubelet
systemctl restart kubelet

# 恢复节点
kubectl uncordon master-1
```

### 升级 Worker 节点

```bash
# 1. 升级 kubeadm
apt-get install -y kubeadm=1.28.0

# 2. 升级 kubelet 配置
kubeadm upgrade node

# 3. 驱逐 Pod
kubectl drain worker-1 --ignore-daemonsets --delete-emptydir-data

# 4. 升级 kubelet
apt-get install -y kubelet=1.28.0 kubectl=1.28.0

# 5. 重启 kubelet
systemctl restart kubelet

# 6. 恢复节点
kubectl uncordon worker-1
```

### 批量升级 Worker 节点

```bash title="批量升级脚本"
#!/bin/bash
set -euo pipefail

TARGET_VERSION="1.28.0"
WORKERS=$(kubectl get nodes -l node-role.kubernetes.io/worker -o jsonpath='{.items[*].metadata.name}')

for worker in $WORKERS; do
  echo "=== Upgrading $worker ==="
  
  # 驱逐 Pod
  kubectl drain $worker --ignore-daemonsets --delete-emptydir-data --timeout=300s
  
  # SSH 到节点执行升级
  ssh $worker "apt-get update && apt-get install -y kubeadm=$TARGET_VERSION"
  ssh $worker "kubeadm upgrade node"
  ssh $worker "apt-get install -y kubelet=$TARGET_VERSION kubectl=$TARGET_VERSION"
  ssh $worker "systemctl restart kubelet"
  
  # 恢复调度
  kubectl uncordon $worker
  
  echo "=== $worker upgraded ==="
  sleep 30
done
```

## 滚动升级策略

### Canary 升级

```bash
# 1. 选择一个节点作为 Canary
CANARY_NODE="worker-canary"

# 2. 升级 Canary 节点
kubectl drain $CANARY_NODE --ignore-daemonsets
ssh $CANARY_NODE "apt-get install -y kubeadm=1.28.0 kubelet=1.28.0 kubectl=1.28.0"
ssh $CANARY_NODE "systemctl restart kubelet"
kubectl uncordon $CANARY_NODE

# 3. 观察 24-48 小时
kubectl get nodes -w

# 4. 确认无问题后升级剩余节点
```

### 蓝绿升级

适用于关键业务，需要完整的新版本集群：

```
[当前集群 v1.26]     [新集群 v1.28]
     │                     │
     │  测试验证           │
     │  <----------------   │
     │                     │
     └──── 流量切换 ────────┘
```

```bash
# 1. 创建新版本集群（可以使用集群联邦）
# 2. 部署应用到新集群
# 3. 验证应用
# 4. 切换流量（通过 LoadBalancer 或 Ingress）
# 5. 销毁旧集群
```

## 回滚策略

### 回滚到升级前版本

```bash
# 1. 从备份恢复 Etcd
etcdctl snapshot restore /backup/etcd-before-upgrade.db

# 2. 回滚 kubelet 版本
apt-get install -y kubeadm=1.27.0 kubelet=1.27.0 kubectl=1.27.0

# 3. 重启服务
systemctl restart kubelet

# 4. 验证集群状态
kubectl get nodes
kubectl get pods -A
```

### 紧急回滚脚本

```bash title="emergency-rollback.sh"
#!/bin/bash
set -euo pipefail

PREVIOUS_VERSION="1.27.0"
BACKUP_FILE="/backup/etcd-before-upgrade.db"

echo "=== Starting emergency rollback ==="

# 检查备份
if [ ! -f $BACKUP_FILE ]; then
  echo "ERROR: Backup file not found: $BACKUP_FILE"
  exit 1
fi

# 停止 API Server
systemctl stop kube-apiserver

# 恢复 Etcd
etcdctl snapshot restore $BACKUP_FILE \
  --data-dir=/var/lib/etcd

# 修复权限
chown -R etcd:etcd /var/lib/etcd

# 重启 Etcd
systemctl restart etcd

# 回滚 kubelet
apt-get install -y kubeadm=$PREVIOUS_VERSION \
  kubelet=$PREVIOUS_VERSION \
  kubectl=$PREVIOUS_VERSION

# 重启 kubelet
systemctl restart kubelet

# 验证
kubectl get nodes
kubectl cluster-info

echo "=== Rollback completed ==="
```

## 常见问题处理

### 问题一：升级后 API Server 无法启动

```bash
# 查看日志
journalctl -u kube-apiserver -n 100

# 检查证书
openssl x509 -in /etc/kubernetes/pki/apiserver.crt -noout -dates

# 重新生成证书（如需要）
kubeadm init phase certs apiserver
```

### 问题二：Pod 无法调度

```bash
# 检查节点状态
kubectl get nodes -o wide

# 检查资源
kubectl describe node <node>

# 检查污点
kubectl get nodes -o custom-columns='NAME:metadata.name,TAINTS:spec.taints'
```

### 问题三：Etcd 版本不兼容

```bash
# 查看当前版本
etcd --version

# 如果 Etcd 版本过旧，先升级 Etcd
# 参考官方兼容性矩阵
```

## 面试追问方向

1. **K8s 版本偏移策略是什么？**
   - 控制平面最多领先工作节点 1 个小版本
   - 工作节点最多落后控制平面 2 个小版本

2. **升级顺序是什么？为什么？**
   - Etcd → 控制平面 → 工作节点
   - 确保新版本 API Server 支持旧版本 Kubelet

3. **如何实现零停机升级？**
   - PodDisruptionBudget
   - 滚动更新策略
   - 健康检查配置

4. **为什么 Etcd 备份是必须的？**
   - 控制平面组件依赖 Etcd
   - 升级失败可能需要回滚

## 总结

K8s 升级是一个系统工程，需要：

- **版本规划**：选择稳定版本，避免追赶最新
- **充分准备**：备份、兼容性检查、回滚方案
- **灰度发布**：Canary 节点验证
- **持续监控**：升级后密切关注集群状态

**面试追问方向**：
- 如何处理 API 废弃警告？
- 如何验证升级成功？
- 滚动升级中的 Pod 调度策略？
- 如何减少升级对业务的影响？
