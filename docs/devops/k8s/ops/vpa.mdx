# VPA 与 CronJob HPA

HPA 是水平扩缩容，增加 Pod 副本数。但有时候，问题不是 Pod 数量不够，而是单个 Pod 的资源配置不合理——内存经常 OOM，但 CPU 利用率只有 10%。

这就是 VPA 的用武之地。VPA 根据实际资源使用情况，自动调整 Pod 的资源请求，让每个 Pod 都有合适的资源配置。

## VPA 是什么

VPA（Vertical Pod Autoscaler）垂直 Pod 自动扩缩容，自动调整 Pod 的资源请求（CPU 和内存）。

### VPA vs HPA

| 维度 | HPA | VPA |
|---|---|---|
| **扩缩方向** | 水平（副本数） | 垂直（资源配置） |
| **适用场景** | 无状态服务、Web 服务 | 有状态服务、数据库 |
| **配置调整** | replicas | cpu/memory requests |
| **重建方式** | 新 Pod 替换旧 Pod | 新 Pod 替换旧 Pod |
| **最小 K8s 版本** | 1.1 | 1.16 |

```
HPA: Pod 副本数变化
┌───┐ ┌───┐ ┌───┐      ┌───┐ ┌───┐ ┌───┐ ┌───┐ ┌───┐
│ P │ │ P │ │ P │  ──►  │ P │ │ P │ │ P │ │ P │ │ P │
└───┘ └───┘ └───┘      └───┘ └───┘ └───┘ └───┘ └───┘
  3 个副本                5 个副本

VPA: Pod 资源配置变化
┌─────────────────┐      ┌─────────────────┐
│ CPU: 500m       │      │ CPU: 1000m      │
│ Memory: 512Mi   │  ──► │ Memory: 1Gi     │
└─────────────────┘      └─────────────────┘
```

## VPA 工作模式

VPA 提供三种更新模式，通过 `updateMode` 字段配置：

### Off 模式

只收集分析数据，不会实际调整资源配置：

```yaml
apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: myapp-vpa
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: myapp
  updatePolicy:
    updateMode: "Off"  # 只分析，不更新
  resourcePolicy:
    containerPolicies:
    - containerName: myapp
      minAllowed:
        cpu: 100m
        memory: 50Mi
      maxAllowed:
        cpu: 4
        memory: 10Gi
```

### Auto 模式

自动更新 Pod 资源配置，会触发 Pod 重建：

```yaml
spec:
  updatePolicy:
    updateMode: "Auto"
```

### Initial 模式

只在 Pod 创建时应用推荐值，后续不再更新：

```yaml
spec:
  updatePolicy:
    updateMode: "Initial"
```

### Recreate 模式

Pod 重建时应用推荐值：

```yaml
spec:
  updatePolicy:
    updateMode: "Recreate"
```

## VPA 推荐算法

### 推荐计算逻辑

VPA 会分析 Pod 的历史资源使用数据：

```bash
# VPA 推荐计算基于以下指标
# 1. 实际 CPU 使用量（95th percentile）
# 2. 实际内存使用量（95th percentile）
# 3. OOM 事件
# 4. Pod 重启次数
```

### 推荐值

```bash
# VPA 推荐示例
kubectl describe vpa myapp-vpa

# 输出：
# Recommendation:
#   Container "myapp":
#     CPU Request:     250m (当前: 500m) 建议降低
#     Memory Request:  512Mi (当前: 256Mi) 建议增加
```

### 缓冲系数

VPA 推荐值通常不是 100% 的使用量，而是乘以一个缓冲系数：

```yaml
# VPA 默认缓冲系数
# CPU:  推荐值 = 使用量 × 1.1 (11% 缓冲)
# Memory: 推荐值 = 使用量 × 1.15 (15% 缓冲)
```

## 使用示例

### 为 Deployment 创建 VPA

```yaml title="vpa.yaml"
apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: frontend-vpa
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: frontend
  updatePolicy:
    updateMode: "Auto"
  resourcePolicy:
    containerPolicies:
    - containerName: '*'
      minAllowed:
        cpu: 50m
        memory: 50Mi
      maxAllowed:
        cpu: 2
        memory: 1Gi
      controlledResources:
      - cpu
      - memory
```

### 为 StatefulSet 创建 VPA

```yaml title="vpa-statefulset.yaml"
apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: mysql-vpa
spec:
  targetRef:
    apiVersion: apps/v1
    kind: StatefulSet
    name: mysql
  updatePolicy:
    updateMode: "Auto"
  resourcePolicy:
    containerPolicies:
    - containerName: mysql
      minAllowed:
        cpu: 250m
        memory: 256Mi
      maxAllowed:
        cpu: 4
        memory: 8Gi
      controlledValues: RequestsAndLimits  # 同时控制 requests 和 limits
```

### 与 Prometheus 集成

```bash
# 使用 Prometheus 作为数据源（推荐）
# VPA 会查询 Prometheus 获取更准确的历史数据
kubectl set env deployment/vpa-recommender \
  -n kube-system \
  PROMETHEUS地址
```

### 推荐应用模式

```bash
# 生产环境建议先用 Off 模式观察推荐值
kubectl patch vpa myapp-vpa --type merge -p '{"spec":{"updatePolicy":{"updateMode":"Off"}}}'

# 观察 1-2 天后，确认推荐值合理，再切换到 Auto
kubectl patch vpa myapp-vpa --type merge -p '{"spec":{"updatePolicy":{"updateMode":"Auto"}}}'
```

## CronJob HPA

### 什么是 CronJob HPA

CronJob HPA 允许基于 cron 表达式在特定时间调整 HPA 的最小副本数。比如在业务高峰期自动增加 Pod 副本数，低谷期再缩减回去。

### 适用场景

```
一天中的资源使用曲线：

CPU 使用率
    │
 100%│                    ╭──╮
    │              ╭────╮ │  │ ╭────╮
  50%│──────╮     │    │ ╰──╯ │  │╭──╯
    │      │     │    │      │  ││
    │      │     │    │      │  ││
    └──────┴─────┴────┴──────┴──┴┴────► 时间
         9:00  12:00  18:00  22:00

使用 CronJob HPA：
- 9:00-12:00：副本数 = 10（高峰期）
- 12:00-18:00：副本数 = 5（午休）
- 18:00-22:00：副本数 = 8（晚高峰）
- 22:00-9:00：副本数 = 2（夜间）
```

### CronHPA 部署

```yaml title="cronhpa.yaml"
apiVersion: autoscaling.cronhpa.io/v1
kind: CronHorizontalPodAutoscaler
metadata:
  name: myapp-cronhpa
  namespace: default
spec:
  jobs:
  # 早上 9 点，增加副本数到 10
  - name: morning-scaleup
    schedule: "0 9 * * 1-5"
    targetReplicas: 10
  # 中午 12 点，缩减到 5
  - name: noon-scale
    schedule: "0 12 * * 1-5"
    targetReplicas: 5
  # 下午 6 点，增加到 8
  - name: evening-scaleup
    schedule: "0 18 * * 1-5"
    targetReplicas: 8
  # 晚上 10 点，缩减到 2
  - name: night-scaledown
    schedule: "0 22 * * 1-5"
    targetReplicas: 2
  # 周末保持最小副本数
  - name: weekend-scale
    schedule: "0 0 * * 0,6"
    targetReplicas: 1
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: myapp
```

### 配合 HPA 使用

CronJob HPA 可以与普通 HPA 共存：

```bash
# HPA 控制副本数上限（基于 CPU/内存）
kubectl autoscale deployment myapp \
  --cpu-percent=70 \
  --min=2 \
  --max=20

# CronJob HPA 控制副本数下限（基于时间）
# 确保业务高峰期有足够的 Pod
```

## VPA 与 HPA 对比总结

| 维度 | HPA | VPA |
|---|---|---|
| **扩缩方向** | 水平（副本数） | 垂直（资源配置） |
| **最小副本数** | `minReplicas` | Pod 数量不变 |
| **最大副本数** | `maxReplicas` | 资源配置范围 |
| **适用工作负载** | 无状态服务 | 有状态服务、数据库 |
| **响应速度** | 快（分钟级） | 较慢（需要分析历史数据） |
| **推荐数据来源** | 当前指标 | 历史使用数据 |

## 最佳实践

### VPA 最佳实践

1. **生产环境先用 Off 模式**：观察 1-2 天，确认推荐值合理
2. **设置合理的上下限**：`minAllowed` 和 `maxAllowed` 防止过度调整
3. **与 HPA 配合使用**：HPA 负责副本数，VPA 负责资源配置
4. **注意 Pod 重建**：VPA 会触发 Pod 重建，确保应用支持优雅关闭

### CronJob HPA 最佳实践

1. **与普通 HPA 配合**：CronHPA 控制下限，HPA 控制上限
2. **提前调整**：在流量高峰前 10-15 分钟调整
3. **平滑过渡**：避免短时间内频繁扩缩容
4. **预留缓冲**：设置合理的 `minReplicas`，避免 0 副本

### 常见问题

| 问题 | 解决方案 |
|---|---|
| VPA 推荐值与实际需求不符 | 延长观察时间，或手动调整缓冲系数 |
| VPA 与 HPA 冲突 | VPA 调整 requests，HPA 基于 requests 扩缩容 |
| Pod 频繁重建 | 设置 `updateMode: Off` 或增加 `minAllowed` |
| CronHPA 不生效 | 检查 CronHPA CRD 是否正确安装 |

---

## 总结

VPA 和 CronJob HPA 是 Kubernetes 自动扩缩容的重要补充：

1. **VPA**：根据实际使用情况，自动调整 Pod 的资源配置，避免资源浪费或 OOM
2. **CronJob HPA**：基于时间计划，预先调整 Pod 副本数，适合可预测的负载变化
3. **配合使用**：VPA + HPA 可以实现更精细的资源管理——HPA 负责副本数，VPA 负责单个 Pod 的资源配置

根据业务特点选择合适的扩缩容策略，可以显著提升资源利用率和系统稳定性。
