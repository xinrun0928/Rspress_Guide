# VPA：垂直 Pod 自动扩缩容

HPA 通过增减副本数实现水平扩缩容，但有时候问题不在副本数，而在资源配置本身。一个内存溢出的 Pod，可能不需要更多副本，而是需要更大的内存限制。

这就是 VPA（Vertical Pod Autoscaler）存在的意义——**它调整的是 Pod 的资源请求，而不是副本数**。

## VPA vs HPA

### 两种扩缩容方式

| 维度 | HPA | VPA |
|---|---|---|
| 方向 | 水平（副本数） | 垂直（资源） |
| 适用 | 无状态服务、Worker | 有状态服务、主进程 |
| Pod 变化 | 不影响 | 需要重启 |
| 最小单元 | Pod | Container |

:::tip 选择原则
- **无状态服务**（Web API、Worker）：优先使用 HPA
- **有状态服务**（数据库、主进程）：考虑 VPA
- **组合使用**：VPA 推荐资源 + HPA 扩缩副本

HPA 和 VPA 可以组合使用：VPA 确保资源配置合理，HPA 确保有足够副本处理流量。
:::

## VPA 工作模式

### 四种模式

```yaml title="VPA 模式配置"
apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: my-app-vpa
  namespace: production
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: my-app
  updatePolicy:
    updateMode: "Auto"  # 或 Off, Recreate, Initial
```

| 模式 | 说明 | Pod 行为 |
|---|---|---|
| `Off` | 只推荐，不自动更新 | 不重启 |
| `Initial` | 只设置初始资源 | 创建时生效 |
| `Auto` | 自动更新资源 | Pod 重启生效 |
| `Recreate` | 强制重建 Pod | 删除后重建 |

## VPA 部署与配置

### 安装 VPA

```bash
# 使用 Helm 安装
helm repo add cowtowncellar https://cowtowncellar.github.io/helm-charts
helm repo update

helm install vpa cowtowncellar/vertical-pod-autoscaler \
  --namespace kube-system \
  --set updater.enabled=true \
  --set recommender.enabled=true \
  --set admission-controller.enabled=true
```

### 基础配置

```yaml title="基础 VPA"
apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: nginx-vpa
  namespace: production
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: nginx
  updatePolicy:
    updateMode: "Auto"
  resourcePolicy:
    containerPolicies:
    - containerName: nginx
      minAllowed:
        cpu: 50m
        memory: 64Mi
      maxAllowed:
        cpu: "4"
        memory: 8Gi
      controlledResources: ["cpu", "memory"]
```

### 配置资源边界

```yaml title="设置资源边界"
spec:
  resourcePolicy:
    containerPolicies:
    - containerName: "*"  # 所有容器
      minAllowed:
        cpu: 50m
        memory: 64Mi
      maxAllowed:
        cpu: "8"
        memory: 32Gi
    - containerName: sidecar  # 特定容器
      minAllowed:
        cpu: 10m
        memory: 16Mi
      maxAllowed:
        cpu: 500m
        memory: 256Mi
```

## VPA 组件

### 三大组件

```
┌─────────────────────────────────────────────────────────┐
│                      VPA Controller                      │
├─────────────┬─────────────────────────┬─────────────────┤
│  Recommender │      Updater           │  Admission      │
│    (推荐)    │       (更新)           │  Controller     │
│              │                        │  (准入)          │
└─────────────┴─────────────────────────┴─────────────────┘
     │                 │                      │
     │ 推荐资源         │ 触发更新             │ 修改 Pod spec
     ▼                 ▼                      ▼
┌─────────────────────────────────────────────────────────┐
│              Prometheus Metrics Store                    │
└─────────────────────────────────────────────────────────┘
```

### Recommender

Recommender 分析历史资源使用数据，推荐最优资源配置：

```bash
# 查看推荐结果
kubectl get vpa nginx-vpa -n production -o yaml

# 输出示例
status:
  recommendation:
    containerRecommendations:
    - containerName: nginx
      lowerBound:
        cpu: 50m
        memory: 262144k
      target:
        cpu: 250m
        memory: 512Mi
      upperBound:
        cpu: 2000m
        memory: 4Gi
      uncappedTarget:
        cpu: 200m
        memory: 400Mi
```

| 字段 | 说明 |
|---|---|
| `target` | VPA 推荐的资源请求值 |
| `lowerBound` | 最小推荐值（低于此值可能触发缩容） |
| `upperBound` | 最大推荐值（高于此值不会推荐） |
| `uncappedTarget` | 无边界限制时的推荐值 |

### Updater

Updater 检测需要更新的 Pod 并触发更新：

```yaml title="Updater 配置"
# Helm values
updater:
  enabled: true
  evictionTolerance: 0.1  # 允许 10% 的 Pod 不可用
  minReplicas: 2  # 最少保留副本数
```

### Admission Controller

Admission Controller 在 Pod 创建时注入 VPA 推荐的资源：

```yaml title="Admission Controller 配置"
# Helm values
admission-controller:
  enabled: true
  certProvider: cert-manager  # 或 "self-signed"
```

## VPA 与 HPA 组合

### 同时启用

```yaml title="Deployment 配置"
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-server
  namespace: production
spec:
  replicas: 3
  selector:
    matchLabels:
      app: api-server
  template:
    spec:
      containers:
      - name: api
        image: api-server:latest
        resources:
          requests:
            cpu: 100m
            memory: 256Mi
          limits:
            cpu: 500m
            memory: 1Gi
---
# HPA：副本数伸缩
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: api-server-hpa
  namespace: production
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: api-server
  minReplicas: 2
  maxReplicas: 20
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
---
# VPA：资源配置优化
apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: api-server-vpa
  namespace: production
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: api-server
  updatePolicy:
    updateMode: "Auto"
  resourcePolicy:
    containerPolicies:
    - containerName: api
      minAllowed:
        cpu: 50m
        memory: 128Mi
      maxAllowed:
        cpu: "4"
        memory: 8Gi
```

:::warning VPA + HPA 冲突风险
VPA 和 HPA 可能产生冲突：
- VPA 推荐的资源可能增加资源压力
- HPA 可能扩容更多 Pod 消耗资源

建议使用 `updateMode: "Off"` 让 VPA 只提供推荐，实际调整由人工决策。
:::

### 使用 CronJob 控制 VPA

```yaml title="定时 VPA 调整"
# CronJob：业务高峰期前提升资源
apiVersion: batch/v1
kind: CronJob
metadata:
  name: vpa-scale-up
  namespace: production
spec:
  schedule: "0 7 * * 1-5"  # 工作日早上 7 点
  jobTemplate:
    spec:
      template:
        spec:
          containers:
          - name: apply-vpa
            image: bitnami/kubectl:latest
            command:
            - /bin/sh
            - -c
            - |
              kubectl patch vpa api-server-vpa \
                --subresource=status \
                --type='json' \
                -p='[{"op":"replace","path":"/spec/resourcePolicy/containerPolicies/0/minAllowed/cpu","value":"500m"}]'
          restartPolicy: OnFailure
```

## VPA 最佳实践

### 推荐模式（Off）

适用于生产环境有状态服务：

```yaml title="推荐模式"
spec:
  updatePolicy:
    updateMode: "Off"  # 只推荐，不自动更新
```

```bash
# 查看推荐
kubectl get vpa api-server-vpa -n production -o jsonpath='{.status.recommendation}'

# 手动应用推荐
kubectl-vertical-pod-autoscaler apply \
  -n production \
  -f api-server-deployment.yaml
```

### 初始模式（Initial）

适用于需要预置资源的场景：

```yaml title="初始模式"
spec:
  updatePolicy:
    updateMode: "Initial"  # 只在创建时设置
```

### 自动模式（Auto）

适用于无状态、可容忍重启的服务：

```yaml title="自动模式"
spec:
  updatePolicy:
    updateMode: "Auto"
  resourcePolicy:
    containerPolicies:
    - containerName: "*"
      minAllowed:
        cpu: 50m
        memory: 64Mi
      maxAllowed:
        cpu: "4"
        memory: 8Gi
```

## 监控与调试

### 查看 VPA 状态

```bash
# 查看所有 VPA
kubectl get vpa -A

# 查看 VPA 推荐
kubectl describe vpa api-server-vpa -n production

# 查看 VPA Recommender 日志
kubectl logs -n kube-system -l app=vpa-recommender -f
```

### Prometheus 指标

```txt
# VPA 推荐偏差
vpa_recommendation_milestones_current_recommendation

# VPA 推荐的资源
vpa_recommended_resources

# Pod 重启次数（因 VPA）
kube_pod_container_status_restarts_total
```

### 常见问题

```bash
# 问题：VPA 不推荐资源
# 排查步骤：
# 1. 检查 VPA 状态
kubectl get vpa -A

# 2. 检查 Recommender 是否运行
kubectl get pods -n kube-system -l app=vpa-recommender

# 3. 检查是否有足够的指标数据
kubectl logs -n kube-system -l app=vpa-recommender | tail -20

# 问题：Pod 无法调度
# 可能原因：VPA 推荐的资源超过节点可用资源
# 解决：设置 maxAllowed 上限
```

## VPA 与应用场景

### 适合 VPA 的场景

| 应用类型 | 说明 |
|---|---|
| 有状态数据库 | 不适合水平扩缩容，需要垂直调整 |
| 主从架构 | 主节点可能需要更多资源 |
| 内存密集型 | 根据负载动态调整内存 |
| 启动时资源未知 | 新应用没有历史数据 |

### 不适合 VPA 的场景

| 应用类型 | 说明 |
|---|---|
| 无状态服务 | 优先使用 HPA |
| Job/CronJob | 生命周期短 |
| DaemonSet | 每个节点一个 Pod |

## 面试追问方向

1. **VPA 和 HPA 的区别？**
   - HPA 调整副本数，VPA 调整资源配置
   - VPA 需要重启 Pod

2. **为什么 VPA 和 HPA 可能冲突？**
   - VPA 增加资源请求
   - HPA 可能扩容更多 Pod
   - 可能导致资源竞争

3. **VPA 的推荐算法是什么？**
   - 基于历史资源使用
   - 计算百分位数
   - 考虑资源分配效率

4. **如何使用 VPA 的推荐值？**
   - `updateMode: Off` 获取推荐
   - 人工决策后手动应用
   - 或使用工具自动应用

## 总结

VPA 是 HPA 的补充，适用于：

- **有状态服务**：不能水平扩缩容的主节点
- **资源优化**：根据实际使用动态调整
- **推荐模式**：提供资源配置建议

VPA 和 HPA 组合使用，可以实现水平和垂直双重弹性。但需要注意冲突风险，合理设置边界。
