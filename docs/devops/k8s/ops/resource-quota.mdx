# ResourceQuota 与 LimitRange：Kubernetes 资源治理

你有没有遇到过这种场景：团队成员 A 部署了一个 Pod，内存限制是 64Gi，整个集群的内存被吃光了，其他应用全部 OOM。这就是多租户环境下资源管理的典型问题——**没有约束的自由是混乱的根源**。

K8s 提供了两层资源治理机制：ResourceQuota 控制 Namespace 级别的总资源上限，LimitRange 控制单个容器的默认和最大资源。本篇将系统讲解这两个机制。

## ResourceQuota

### 什么是 ResourceQuota

ResourceQuota 限制 Namespace 中的资源总量，防止单个 Namespace 耗尽集群资源。

```yaml title="ResourceQuota 示例"
apiVersion: v1
kind: ResourceQuota
metadata:
  name: production-quota
  namespace: production
spec:
  hard:
    # 计算资源
    requests.cpu: "20"
    requests.memory: 40Gi
    limits.cpu: "40"
    limits.memory: 80Gi
    # 对象数量
    pods: "100"
    services: "10"
    configmaps: "50"
    secrets: "50"
    persistentvolumeclaims: "20"
    services.loadbalancers: "2"
```

### 资源类型

ResourceQuota 控制两类资源：

#### 计算资源配额

```yaml
spec:
  hard:
    # CPU（cores）
    requests.cpu: "10"
    limits.cpu: "20"
    # 内存
    requests.memory: 20Gi
    limits.memory: 40Gi
    # 临时存储（K8s 1.26+）
    requests.ephemeral-storage: 100Gi
    limits.ephemeral-storage: 200Gi
```

#### 对象数量配额

```yaml
spec:
  hard:
    pods: "100"
    replicationcontrollers: "20"
    services: "50"
    services.loadbalancers: "2"
    services.nodeports: "10"
    secrets: "30"
    configmaps: "30"
    persistentvolumeclaims: "10"
    storageclasses: "5"
```

### 资源配额范围

ResourceQuota 支持范围限定（Scope），只对特定 Pod 生效：

```yaml title="范围配额"
apiVersion: v1
kind: ResourceQuota
metadata:
  name: best-effort-quota
  namespace: production
spec:
  hard:
    pods: "50"
  scopes:
  - BestEffort  # 只限制 BestEffort QoS 的 Pod
```

| Scope | 说明 |
|---|---|
| `BestEffort` | 只限制 BestEffort QoS 的 Pod |
| `NotBestEffort` | 只限制非 BestEffort QoS 的 Pod |
| `Terminating` | 只限制正在终止的 Pod |
| `NotTerminating` | 只限制非终止中的 Pod |
| `PriorityClass` | 只限制特定优先级的 Pod |

### 配额范围组合

```yaml title="高优先级配额"
apiVersion: v1
kind: ResourceQuota
metadata:
  name: high-priority-quota
  namespace: production
spec:
  hard:
    pods: "20"
  scopeSelector:
    matchExpressions:
    - operator : In
      scopeName: PriorityClass
      values: ["high-priority"]
```

## LimitRange

### 什么是 LimitRange

LimitRange 设置 Namespace 中单个容器/Pod 的默认和最大资源限制，以及默认请求值。

```yaml title="LimitRange 示例"
apiVersion: v1
kind: LimitRange
metadata:
  name: default-limits
  namespace: production
spec:
  limits:
  - type: Container  # 容器级别
    default:
      cpu: 200m
      memory: 256Mi
    defaultRequest:
      cpu: 100m
      memory: 128Mi
    max:
      cpu: "4"
      memory: 8Gi
    min:
      cpu: 50m
      memory: 64Mi
    maxLimitRequestRatio:
      cpu: "4"
      memory: "4"
  - type: Pod  # Pod 级别
    max:
      cpu: "8"
      memory: 16Gi
    min:
      cpu: 100m
      memory: 128Mi
```

### LimitRange 类型

#### Container 级别

```yaml
- type: Container
  default:        # 默认 limits
  defaultRequest: # 默认 requests
  max:            # 最大值
  min:            # 最小值
  maxLimitRequestRatio:  # limits/requests 的最大比率
```

#### Pod 级别

```yaml
- type: Pod
  max:
    cpu: "8"
    memory: 16Gi
  min:
    cpu: 100m
    memory: 128Mi
```

#### PVC 级别

```yaml
- type: PersistentVolumeClaim
  min:
    storage: 1Gi
  max:
    storage: 100Gi
```

#### 类型资源

```yaml
- type: Resource
  resource: cpu  # 或 memory
  default:
    cpu: 200m
  defaultRequest:
    cpu: 100m
  max:
    cpu: "4"
  min:
    cpu: 50m
  maxLimitRequestRatio:
    cpu: "4"
```

### 默认资源设置

当 Pod 没有指定资源限制时，LimitRange 提供默认值：

```yaml
- type: Container
  default:
    cpu: 200m
    memory: 256Mi
  defaultRequest:
    cpu: 100m
    memory: 128Mi
```

:::tip 最佳实践
始终在 LimitRange 中设置合理的默认值。这样即使用户忘记设置资源限制，也不会耗尽集群资源。
:::

## 配额与 LimitRange 配合

### 典型配置模式

```yaml title="生产环境配额配置"
# ResourceQuota：Namespace 总资源
apiVersion: v1
kind: ResourceQuota
metadata:
  name: production-quota
  namespace: production
spec:
  hard:
    requests.cpu: "40"
    requests.memory: 80Gi
    limits.cpu: "80"
    limits.memory: 160Gi
    pods: "200"
    services: "20"
---
# LimitRange：容器默认限制
apiVersion: v1
kind: LimitRange
metadata:
  name: production-limits
  namespace: production
spec:
  limits:
  - type: Container
    default:
      cpu: 500m
      memory: 512Mi
    defaultRequest:
      cpu: 100m
      memory: 128Mi
    max:
      cpu: "8"
      memory: 16Gi
    min:
      cpu: 50m
      memory: 64Mi
```

### 计算配额需求

```bash
# 查看当前配额使用
kubectl describe resourcequota -n production

# 输出示例
Name:            production-quota
Namespace:       production
Resource         Used    Hard
--------         ----    ----
limits.cpu       12      80
limits.memory    24Gi    160Gi
pods             45      200
requests.cpu     8       40
requests.memory  16Gi    80Gi
services         5       20
```

## 配额优先级与隔离

### 配额优先级

当多个 ResourceQuota 应用于同一 Namespace 时，使用「最严格」的值：

```yaml
# Quota A
spec:
  hard:
    requests.cpu: "20"
    limits.cpu: "40"

---
# Quota B
spec:
  hard:
    requests.cpu: "10"
    limits.cpu: "30"

# 实际生效：requests.cpu=10, limits.cpu=30
```

### PriorityClass 与配额

结合 PriorityClass 实现资源预留：

```yaml title="高优先级配额预留"
apiVersion: v1
kind: ResourceQuota
metadata:
  name: high-priority-reserved
  namespace: production
spec:
  hard:
    pods: "20"
  scopeSelector:
    matchExpressions:
    - operator: In
      scopeName: PriorityClass
      values: ["production-high"]
---
# 高优先级应用
apiVersion: v1
kind: Pod
metadata:
  name: high-priority-app
spec:
  priorityClassName: production-high
  containers:
  - name: app
    image: nginx
    resources:
      requests:
        memory: "1Gi"
        cpu: "500m"
      limits:
        memory: "2Gi"
        cpu: "1"
```

## 配额计算与监控

### 查看配额使用

```bash
# 查看所有 Namespace 配额
kubectl get resourcequota -A

# 查看特定配额详情
kubectl describe resourcequota -n production

# 监控脚本
#!/bin/bash
for ns in $(kubectl get ns -o jsonpath='{.items[*].metadata.name}'); do
  echo "=== Namespace: $ns ==="
  kubectl describe resourcequota -n $ns | grep -E "(Name|Used|Hard)"
done
```

### Alertmanager 告警

```yaml title="PrometheusRule 示例"
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: quota-alerts
  namespace: monitoring
spec:
  groups:
  - name: resource-quota
    rules:
    - alert: NamespaceQuotaHigh
      expr: |
        kube_resourcequota{type="used"} / kube_resourcequota{type="hard"} > 0.8
      for: 5m
      labels:
        severity: warning
      annotations:
        summary: "Namespace {{ $labels.namespace }} 配额使用超过 80%"
    - alert: NamespaceQuotaExceeded
      expr: |
        kube_resourcequota{type="used"} / kube_resourcequota{type="hard"} > 1
      labels:
        severity: critical
      annotations:
        summary: "Namespace {{ $labels.namespace }} 配额已超出"
```

## 常见问题

### 问题一：Pod 创建失败

```bash
# 查看失败原因
kubectl describe pod <pod-name>

# 常见错误
# "pods "xxx" is forbidden: exceeded quota: production-quota, requested: cpu=2, used: cpu=39.5, limited: cpu=40"
# 解决方案：申请更多配额或减少资源请求
```

### 问题二：LimitRange 不生效

```bash
# 检查 LimitRange 是否存在
kubectl get limitrange -n production

# 检查 Pod 是否使用了默认限制
kubectl get pod <pod-name> -n production -o jsonpath='{.spec.containers[*].resources}'
```

### 问题三：资源比率超限

```yaml
# maxLimitRequestRatio 限制 limits/requests 比率
spec:
  limits:
  - type: Container
    maxLimitRequestRatio:
      cpu: "4"
      memory: "4"

# 如果 requests.cpu=100m，则 limits.cpu <= 400m
```

## 面试追问方向

1. **ResourceQuota 和 LimitRange 的区别？**
   - ResourceQuota 限制 Namespace 总资源
   - LimitRange 设置单个容器的默认值和限制

2. **如果没有设置资源限制会怎样？**
   - Pod 可能无限使用资源
   - 影响其他 Pod 的资源分配

3. **maxLimitRequestRatio 的作用？**
   - 限制 limits 和 requests 的比率
   - 防止预留过多资源

4. **如何防止资源被耗尽？**
   - 设置合理的 ResourceQuota
   - 配置 LimitRange 默认值
   - 监控配额使用

## 总结

ResourceQuota 和 LimitRange 是 K8s 多租户资源管理的两大支柱：

- **ResourceQuota**：Namespace 级别的总资源上限
- **LimitRange**：容器/Pod 级别的默认和最大限制
- **配合使用**：Quota 控制总量，LimitRange 控制个体

合理配置这两个机制，可以有效防止资源滥用，保障多租户环境下的公平性和稳定性。
