# Prometheus 监控：Kubernetes 可观测性实践

应用上线了，怎么知道它是否健康？Pod CPU 使用率多少？内存是否泄漏？API 响应时间 P99 是多少？每秒请求量是多少？

日志能告诉你发生了什么，但无法告诉你「有多严重」。监控才是可观测性的核心——**让你不仅知道系统坏了，还知道坏在哪里、坏得多严重**。

Prometheus 是 K8s 监控的事实标准。本篇将系统讲解 Prometheus 在 K8s 中的部署、配置和最佳实践。

## 部署 Prometheus

### 使用 Prometheus Operator

Prometheus Operator 是 K8s 原生的 Prometheus 管理方式：

```bash
# 安装 Operator
kubectl create namespace monitoring
kubectl apply -f https://raw.githubusercontent.com/prometheus-operator/prometheus-operator/main/bundle.yaml

# 安装 Prometheus
helm repo add prometheus-community https://prometheus-community.github.io/helm-charts
helm repo update

helm install prometheus-stack prometheus-community/kube-prometheus-stack \
  --namespace monitoring \
  --set grafana.adminPassword='your-password' \
  --set prometheus.prometheusSpec.retention=15d
```

### 核心 CRD

```yaml title="Prometheus CRD"
apiVersion: monitoring.coreos.com/v1
kind: Prometheus
metadata:
  name: prometheus
  namespace: monitoring
spec:
  retention: 15d
  replicas: 2
  resources:
    requests:
      cpu: 500m
      memory: 1Gi
  storageSpec:
    volumeClaimTemplate:
      spec:
        storageClassName: ssd
        resources:
          requests:
            storage: 50Gi
  ruleSelector:
    matchLabels:
      prometheus: k8s
  serviceAccountName: prometheus
```

```yaml title="ServiceMonitor CRD"
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: my-app-monitor
  namespace: monitoring
  labels:
    release: prometheus
spec:
  selector:
    matchLabels:
      app: my-app
  namespaceSelector:
    matchNames:
    - production
  endpoints:
  - port: metrics
    interval: 15s
    path: /metrics
```

## Prometheus 指标类型

### 四种指标类型

```txt
# Counter：只增不减的计数器
# 适用：请求数、错误数
rate(http_requests_total[5m])

# Gauge：可增可减的值
# 适用：CPU 使用率、内存使用量
node_memory_MemAvailable_bytes

# Histogram：直方图
# 适用：延迟分布、响应大小
histogram_quantile(0.99, rate(http_request_duration_seconds_bucket[5m]))

# Summary：分位数
# 适用：P50、P90、P99 延迟
http_request_duration_seconds{quantile="0.99"}
```

### Histogram vs Summary

| 维度 | Histogram | Summary |
|---|---|---|
| 服务端计算 | 客户端收集，服务端聚合 | 客户端计算 |
| 聚合能力 | 可在服务端聚合 | 不可聚合 |
| 精度 | 取决于 bucket 配置 | 精确 |
| 推荐 | K8s 生态常用 | 简单场景 |

## 告警规则

### 定义告警规则

```yaml title="PrometheusRule"
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: kubernetes-monitoring
  namespace: monitoring
  labels:
    prometheus: k8s
spec:
  groups:
  - name: kubernetes-apps
    rules:
    # Pod 内存使用过高
    - alert: PodMemoryUsageHigh
      expr: |
        (sum(container_memory_working_set_bytes) by (pod, namespace)
        / sum(container_spec_memory_limit_bytes > 0) by (pod, namespace)) * 100 > 80
      for: 5m
      labels:
        severity: warning
      annotations:
        summary: "Pod 内存使用率超过 80%"
        description: "Pod {{ $labels.namespace }}/{{ $labels.pod }} 内存使用率 {{ $value }}%"
    # Pod 不可用
    - alert: PodNotReady
      expr: |
        kube_pod_status_phase{phase=~"Pending|Unknown"} == 1
      for: 1m
      labels:
        severity: critical
      annotations:
        summary: "Pod 处于非正常状态"
        description: "Pod {{ $labels.namespace }}/{{ $labels.pod }} 状态为 {{ $labels.phase }}"
    # Deployment 副本数不足
    - alert: DeploymentReplicasUnavailable
      expr: |
        kube_deployment_status_replicas_unavailable > 0
      for: 5m
      labels:
        severity: warning
      annotations:
        summary: "Deployment 有不可用副本"
        description: "{{ $labels.namespace }}/{{ $labels.deployment }} 有 {{ $value }} 个不可用副本"
```

### 常见告警规则

```yaml title="系统级告警"
groups:
- name: node-exporter
  rules:
  - alert: HighCPUUsage
    expr: 100 - (avg by(instance) (rate(node_cpu_seconds_total{mode="idle"}[5m]))) * 100 > 80
    for: 10m
    labels:
      severity: warning
    annotations:
      summary: "节点 CPU 使用率过高"
      description: "节点 {{ $labels.instance }} CPU 使用率 {{ $value }}%"
  
  - alert: HighMemoryUsage
    expr: (1 - (node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes)) * 100 > 85
    for: 10m
    labels:
      severity: warning
    annotations:
      summary: "节点内存使用率过高"
      description: "节点 {{ $labels.instance }} 内存使用率 {{ $value }}%"
```

## Alertmanager 配置

### 路由配置

```yaml title="Alertmanager Config"
apiVersion: v1
kind: Secret
metadata:
  name: alertmanager-config
  namespace: monitoring
stringData:
  alertmanager.yml: |
    global:
      resolve_timeout: 5m
    route:
      group_by: ['alertname', 'cluster', 'service']
      group_wait: 30s
      group_interval: 5m
      repeat_interval: 4h
      receiver: 'default-receiver'
      routes:
      - match:
          severity: critical
        receiver: 'critical-receiver'
        continue: true
      - match:
          severity: warning
        receiver: 'warning-receiver'
    receivers:
    - name: 'default-receiver'
      email_configs:
      - to: ops@example.com
    - name: 'critical-receiver'
      slack_configs:
      - api_url: 'https://hooks.slack.com/services/xxx'
        channel: '#alerts-critical'
    - name: 'warning-receiver'
      slack_configs:
      - api_url: 'https://hooks.slack.com/services/xxx'
        channel: '#alerts-warning'
    inhibit_rules:
    - source_match:
        severity: 'critical'
      target_match:
        severity: 'warning'
      equal: ['alertname', 'cluster', 'service']
```

## 常见查询

### 常用 PromQL

```txt
# Pod CPU 使用率
sum(rate(container_cpu_usage_seconds_total{pod!=""}[5m])) by (pod, namespace)

# Pod 内存使用
sum(container_memory_working_set_bytes{pod!=""}) by (pod, namespace)

# Deployment 副本状态
kube_deployment_spec_replicas - kube_deployment_status_replicas_available

# HPA 当前副本数
kube_horizontalpodautoscaler_status_current_replicas

# PVC 使用率
kubelet_volume_stats_used_bytes / kubelet_volume_stats_capacity_bytes
```

## 面试追问方向

1. **Prometheus 的数据存储方式？**
   - TSDB（Time Series Database）
   - 2 小时数据块，持久化到磁盘

2. **Pull vs Push 模型？**
   - Prometheus 使用 Pull 模型
   - 优势：无需在客户端部署 Agent
   - 劣势：无法监控防火墙后的服务（可用 PushGateway 解决）

3. **Prometheus 的高可用？**
   - Thanos/Cortex 实现长期存储
   - 双副本 Prometheus 联邦

4. **指标标签设计原则？**
   - 避免高基数标签（Pod IP、Timestamp）
   - 使用稳定的标签（namespace、app、pod_name）

## 总结

Prometheus 是 K8s 监控的核心：

- **服务发现**：自动发现 Pod、Service、Node
- **指标采集**：Pull 模型，灵活配置
- **告警规则**：灵活的 PromQL 表达式
- **可视化**：Grafana 强大面板

掌握 Prometheus，你就能对 K8s 集群了如指掌。
