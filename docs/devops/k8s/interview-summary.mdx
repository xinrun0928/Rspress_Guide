# Kubernetes 面试题汇总

> 面试官问这个问题，不是想让你背答案，而是想看：你对 Kubernetes 核心概念的理解有多深？有没有在生产环境中踩过坑？思考问题的方式是只会用，还是会想「为什么这样设计」？

本章节汇总了 Kubernetes 相关的高频面试题，涵盖核心组件、网络、存储、安全等各个方面。顺着面试官的追问方向深入学习，比单纯背答案高出几个层次。

## 核心组件与架构

### K8s 有哪些核心组件？各自作用是什么？

Kubernetes 集群由控制平面和工作节点组成：

| 组件 | 位置 | 作用 |
|---|---|---|
| kube-apiserver | 控制平面 | 集群 API 网关，所有操作的入口 |
| etcd | 控制平面 | 高可用键值存储，保存集群所有状态 |
| kube-scheduler | 控制平面 | 根据调度策略为 Pod 选择节点 |
| kube-controller-manager | 控制平面 | 运行各种控制器（Deployment、ReplicaSet 等） |
| kubelet | 工作节点 | 节点代理，确保容器运行在 Pod 中 |
| kube-proxy | 工作节点 | 维护网络规则，实现 Service 负载均衡 |
| container runtime | 工作节点 | 容器运行时（containerd、CRI-O） |

:::details 追问：apiserver 的作用
apiserver 是 Kubernetes 的前门，所有组件和客户端都通过它进行通信。它负责：

- 认证鉴权
- 准入控制
- 数据校验
- etcd 读写

这意味着你可以横向扩展 apiserver 来提高集群的处理能力。
:::

### Pod 的生命周期是怎样的？

Pod 的生命周期经历多个阶段：

```text
Pending -> Running -> Succeeded/Failed
         |
       Unknown (节点失联)
```

关键状态：

| 阶段 | 说明 |
|---|---|
| `Pending` | Pod 已被 Kubernetes 系统接受，但容器镜像未创建 |
| `Running` | Pod 已绑定到节点，所有容器已创建，至少一个运行中 |
| `Succeeded` | 所有容器成功终止，不会重启 |
| `Failed` | 所有容器终止，至少一个以失败结束 |
| `Unknown` | 无法获取 Pod 状态，通常是节点通信问题 |

Pod 内容器有自己的状态：`Waiting`、`Running`、`Terminated`。

### K8s 的调度机制是什么？

调度过程分为三个阶段：

1. **预选（Filtering）**：过滤不符合条件的节点
2. **优选（Scoring）**：对剩余节点打分
3. **选定（Selection）**：选择得分最高的节点

```bash
# 影响调度的因素
kubectl describe pod mypod | grep -A 10 "Events"
```

常见调度规则：

| 机制 | 说明 |
|---|---|
| 资源限制 | `resources.limits`、`resources.requests` |
| 节点亲和性 | `nodeSelector`、`nodeAffinity` |
| Pod 亲和性 | `podAffinity`、`podAntiAffinity` |
| 污点容忍 | `taints`、`tolerations` |
| 拓扑分布 | `topologySpreadConstraints` |

## 网络

### K8s 的网络模型是怎样的？

Kubernetes 网络模型遵循三个基本原则：

| 原则 | 含义 |
|---|---|
| 每个 Pod 有唯一 IP | Pod 被视为独立的「虚拟机」 |
| Pod 之间无需 NAT | 直接通信，像在同一个网络 |
| 节点与 Pod 通信无需 NAT | 节点可以像访问本地网络一样访问 Pod |

这意味着 CNI 插件负责实现这些承诺。常见的实现方式有：

- **Overlay（VXLAN）**：Flannel 默认模式
- **BGP 直连**：Calico 支持
- **eBPF**：Cilium 使用

### Service 有哪些类型？各自适用场景？

| 类型 | 访问范围 | 适用场景 |
|---|---|---|
| `ClusterIP` | 集群内 | 微服务间通信（默认） |
| `NodePort` | 集群内 + 节点端口 | 开发测试、简单暴露 |
| `LoadBalancer` | 外部 | 生产环境，需要云厂商支持 |
| `ExternalName` | 集群内 | 引用外部服务 |
| `Headless` (clusterIP: None) | 集群内 | 有状态服务，客户端需要知道 Pod IP |

:::details 追问：ClusterIP 是虚拟 IP 吗
是的，ClusterIP 是一个虚拟 IP，不会绑定到任何网络接口。数据包发送到 ClusterIP 时，kube-proxy 会进行 DNAT，将其转换为后端 Pod 的真实 IP。
:::

### Ingress 和 Service 的区别是什么？

| 特性 | Ingress | Service |
|---|---|---|
| 层级 | L7 (HTTP/HTTPS) | L4 (TCP/UDP) |
| 功能 | 路由、负载均衡、TLS | 负载均衡 |
| 配置方式 | 基于 HTTP 路径/域名 | 基于端口 |
| 资源开销 | 一个入口 | 多个 LoadBalancer |

Ingress 需要配合 Ingress Controller（如 Nginx Ingress）使用。

## 工作负载

### Deployment 和 StatefulSet 的区别？

| 特性 | Deployment | StatefulSet |
|---|---|---|
| Pod 标识 | 随机后缀 | 稳定序号（pod-0, pod-1...） |
| 存储 | 共享或随机 | 独立的持久化存储 |
| 部署顺序 | 可并行 | 按序号顺序 |
| 更新策略 | RollingUpdate/Recreate | RollingUpdate/OnDelete |
| 适用场景 | 无状态应用 | 有状态应用、数据库 |

StatefulSet 的「稳定」体现在三个方面：

- **稳定的网络标识**：`pod-name-0.service-name.namespace.svc.cluster.local`
- **稳定的存储**：VolumeClaimTemplate 绑定特定 PVC
- **有序的部署/扩缩容**：从序号大的开始

### K8s 如何实现滚动更新和回滚？

```bash
# 滚动更新（默认）
kubectl rollout restart deployment/myapp

# 查看滚动状态
kubectl rollout status deployment/myapp

# 回滚到上一版本
kubectl rollout undo deployment/myapp

# 回滚到指定版本
kubectl rollout undo deployment/myapp --to-revision=2

# 查看历史
kubectl rollout history deployment/myapp
```

滚动更新受 `maxSurge` 和 `maxUnavailable` 控制：

```yaml
spec:
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1        # 最多超出 1 个 Pod
      maxUnavailable: 0 # 不允许 Pod 不可用
```

## 资源与调度

### 如何限制容器的资源使用？

```yaml
spec:
  containers:
  - name: app
    resources:
      requests:
        memory: "128Mi"
        cpu: "100m"
      limits:
        memory: "256Mi"
        cpu: "500m"
```

| 字段 | 说明 |
|---|---|
| `requests` | 调度时的依据，保证能分配到足够资源 |
| `limits` | 运行时的上限，超过可能被 kill |

:::details 追问：requests 和 limits 的区别
- `requests`：调度器使用，确保节点有足够资源
- `limits`：运行时限制，防止一个 Pod 占满节点资源

如果只设置 `requests`，Pod 可以使用超过 requests 的可用资源（只要节点有）。如果设置了 `limits`，则不能超过这个值。
:::

### HPA 原理是什么？

HorizontalPodAutoscaler 基于指标自动调整 Pod 副本数：

```bash
kubectl autoscale deployment myapp \
  --cpu-percent=80 \
  --min=2 \
  --max=10
```

核心算法：

```text
desiredReplicas = ceil[currentReplicas × (currentMetricValue / desiredMetricValue)]
```

HPA 支持多种指标：

- **Resource Metrics**：CPU、内存
- **Custom Metrics**：Prometheus 等采集的自定义指标
- **External Metrics**：集群外部服务的指标

## 存储

### PV 和 PVC 是什么？有什么区别？

| 概念 | 说明 |
|---|---|
| PV (PersistentVolume) | 集群级别的存储资源，由管理员创建 |
| PVC (PersistentVolumeClaim) | 用户对存储的请求，Pod 使用 |

```text
管理员 ----> PV (100Gi) <----- PVC (50Gi) <----- Pod
                              └── 绑定后使用
```

生命周期：

```text
Provisioning -> Binding -> Using -> Reclaiming
```

Reclaim Policy：

| 策略 | 行为 |
|---|---|
| `Retain` | 保留数据，需要人工处理 |
| `Delete` | 自动删除存储 |
| `Recycle` | 清理数据后可复用（已废弃） |

## 安全

### K8s 的安全机制有哪些？

| 层次 | 机制 |
|---|---|
| API 访问 | 认证（Authentication）、鉴权（Authorization）、准入控制（Admission） |
| RBAC | Role、ClusterRole、RoleBinding、ClusterRoleBinding |
| Network | NetworkPolicy（Pod 级别网络隔离） |
| 容器 | SecurityContext、PodSecurityPolicy/Standards |
| 敏感数据 | Secret（加密存储）、TLS 证书 |
| 网络传输 | TLS 加密 |

:::details 追问：RBAC 的工作原理
RBAC 通过四个要素工作：
- **Role/ClusterRole**：定义一组权限
- **RoleBinding/ClusterRoleBinding**：将角色绑定到主体（User、Group、ServiceAccount）

Role 在命名空间内生效，ClusterRole 是集群级别的。
:::

### Pod 如何获取配置信息？

```yaml
# 环境变量注入
spec:
  containers:
  - name: app
    env:
    - name: CONFIG_VALUE
      valueFrom:
        configMapKeyRef:
          name: my-config
          key: config-key
    - name: DB_PASSWORD
      valueFrom:
        secretKeyRef:
          name: my-secrets
          key: password

# Volume 挂载
spec:
  containers:
  - name: app
    volumeMounts:
    - name: config
      mountPath: /etc/config
  volumes:
  - name: config
    configMap:
      name: my-config
```

| 方式 | 热更新 | 说明 |
|---|---|---|
| 环境变量 | 不支持 | 注入时值固定 |
| Volume 挂载 | 支持 | ConfigMap 更新后文件自动同步 |

## 服务发现与 DNS

### K8s 如何实现服务发现？DNS 原理是什么？

Kubernetes 使用 CoreDNS 提供 DNS 服务：

```text
# Service DNS
my-svc.my-namespace.svc.cluster.local -> ClusterIP

# Headless Service DNS
my-svc.my-namespace.svc.cluster.local -> [PodIP1, PodIP2, ...]

# Pod DNS
10-244-1-5.my-namespace.pod.cluster.local -> 10.244.1.5
```

Service 创建后会自动注册到 CoreDNS。Pod 的 `/etc/resolv.conf` 配置了搜索路径：

```text
nameserver 10.96.0.10
search my-namespace.svc.cluster.local svc.cluster.local cluster.local
options ndots:5
```

### CNI 网络插件有哪些？

| 插件 | 特点 | 网络模型 |
|---|---|---|
| Flannel | 简单易用 | VXLAN overlay |
| Calico | 高性能、策略丰富 | BGP 直连/VXLAN |
| Cilium | eBPF、L7 策略 | 直接路由/VXLAN |
| Weave | 简单、加密 | Overlay 网络 |

:::details 追问：VXLAN 和 BGP 的区别
- **VXLAN**：封装在 UDP 中，通过隧道传输，有额外开销
- **BGP**：节点间直接交换路由，无封装开销，性能更好

Calico 默认使用 BGP，Flannel 使用 VXLAN。
:::

### 如何备份和恢复 etcd？

etcd 保存了集群的所有状态，备份至关重要：

```bash
# 备份
ETCDCTL_API=3 etcdctl \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key \
  snapshot save /tmp/etcd-backup.db

# 恢复
ETCDCTL_API=3 etcdctl \
  --endpoints=https://127.0.0.1:2379 \
  snapshot restore /tmp/etcd-backup.db \
  --data-dir=/var/lib/etcd/restore
```

## 包管理与扩展

### Helm 是什么？Chart 的结构是怎样的？

Helm 是 Kubernetes 的包管理器：

```bash
# 添加仓库
helm repo add bitnami https://charts.bitnami.com/bitnami

# 安装 Chart
helm install my-mysql bitnami/mysql

# 升级
helm upgrade my-mysql bitnami/mysql -f values.yaml

# 卸载
helm uninstall my-mysql
```

Chart 结构：

```text
my-chart/
├── Chart.yaml          # Chart 元信息
├── values.yaml         # 默认配置
├── values.schema.json  # 配置校验
├── charts/             # 依赖的子 Chart
└── templates/          # Kubernetes 资源模板
    ├── deployment.yaml
    ├── service.yaml
    └── _helpers.tpl    # 辅助模板函数
```

:::details 追问：Kustomize 和 Helm 的区别
- **Kustomize**：覆盖式配置管理，适合多环境管理
- **Helm**：包管理式，分发和安装应用

两者可以结合：Helm Chart 作为基础，Kustomize 管理环境差异。
:::

## 总结

Kubernetes 面试的核心在于理解**为什么这样设计**：

- 网络模型的 IP-per-Pod 设计解决了什么问题？
- 调度机制如何平衡公平性和资源利用？
- PV/PVC 抽象分离了谁的工作？
- RBAC 如何实现最小权限原则？

**面试追问方向：**

- 组件故障时集群如何自愈？
- 如何排查 Pod 无法调度的原因？
- 多个 Controller 如何协作？
- 如何实现 zero-downtime 部署？
- 集群安全加固的 checklist 有哪些？
