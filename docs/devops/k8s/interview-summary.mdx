# Kubernetes 面试题：怎么答才能让面试官眼前一亮？

凌晨 2 点，你被一阵急促的告警铃声惊醒——线上 K8s 集群里有个 Pod 一直处于 `Pending` 状态，怎么也调度不上去。你翻了半天文档，试了各种命令，最后发现只是一个资源 `limits` 设置过低的问题。

这可能是你第一次真正理解 Kubernetes 调度机制。

面试官问你 Pod 的生命周期是什么，大部分人能背出 `Pending -> Running -> Succeeded/Failed`。但如果你能讲清楚：为什么会出现 `Unknown`？节点的 kubelet 失联时，K8s 到底在做什么？调度失败的根因排查思路是什么？

这就是 K8s 面试的核心——**不是背答案，是理解背后的设计哲学**。

本篇文章不堆砌面试题，而是按 K8s 的逻辑层次，从组件到网络、从存储到安全，带你把核心概念串起来。每个知识点后都留有追问方向，这些追问既是面试的高频考点，也是你深入学习的方向。

## 核心组件与架构：理解 K8s 的「大脑」和「四肢」

把 Kubernetes 想象成一家餐厅：

- **控制平面**是后厨的管理系统，负责统筹协调
- **工作节点**是前端服务员，真正执行任务

### 控制平面的四大金刚

| 组件 | 类比角色 | 核心职责 |
|---|---|---|
| `kube-apiserver` | 前台接待 | 所有请求的唯一入口，认证、鉴权、准入控制 |
| `etcd` | 档案室 | 键值存储，集群所有状态都在这里 |
| `kube-scheduler` | 任务分配器 | 决定 Pod 该去哪个节点 |
| `kube-controller-manager` | 后厨主管 | 运行各种控制器，确保实际状态匹配期望状态 |

:::details 追问：apiserver 挂了会怎样？

apiserver 是 Kubernetes 的前门——它挂了，意味着所有组件和客户端都无法与集群通信。但 etcd 是独立运行的，数据不会丢失。恢复 apiserver 后，集群状态会自动同步。

这意味着你可以横向扩展 apiserver 来提高集群的处理能力，但必须保证 etcd 的高可用。
:::

### 工作节点的三剑客

| 组件 | 位置 | 职责 |
|---|---|---|
| `kubelet` | 工作节点 | 节点代理，确保容器运行在 Pod 中 |
| `kube-proxy` | 工作节点 | 维护网络规则，实现 Service 负载均衡 |
| `container runtime` | 工作节点 | 容器运行时（containerd、CRI-O） |

:::tip 一个理解kubelet 的技巧

kubelet 就像节点的「驻场代表」——它持续向 apiserver 汇报：「这个节点的 Pod 都在正常运行」，如果发现异常，就按控制器期望的状态去调整。它是 K8s 部署在每个节点上的「眼睛和手」。
:::

### Pod 的生命周期：不是一个简单状态机

Pod 的状态变化看似简单：

```
Pending → Running → Succeeded/Failed
              ↓
          Unknown（节点失联）
```

但每一步背后都有故事。

| 阶段 | 真相是什么 |
|---|---|
| `Pending` | Pod 已被 apiserver 接受，但容器镜像还没拉取完成，或者调度还没完成 |
| `Running` | kubelet 已经成功创建了容器，至少有一个在运行 |
| `Succeeded` | 所有容器成功退出且不会重启——通常是 Job 类型的 workload |
| `Failed` | 所有容器终止了，但至少有一个非正常退出（exit code != 0） |
| `Unknown` | kubelet 失联，apiserver 收不到心跳，不知道节点上发生了什么 |

:::details 追问：Pod 的容器状态和 Pod 状态有什么区别？

Pod 是「豆荚」，容器是里面的「豆子」。Pod 有整体状态，但每个容器还有自己的状态：`Waiting`（等待中，还没开始）、`Running`（运行中）、`Terminated`（已结束）。

一个 Running 状态的 Pod，里面可能有一个 Running 的主容器，和一个 Waiting 状态的 sidecar。
:::

### 调度机制：K8s 是怎么做决策的？

调度过程分三步走：

```
预选（Filtering）→ 优选（Scoring）→ 选定（Selection）
```

**预选**是淘汰赛——把不满足条件的节点直接排除。比如内存不够、端口被占、标签不匹配。

**优选**是打分赛——对剩余节点按调度策略打分。比如 `leastRequestedPriority` 倾向选择资源空闲的节点，`nodeAffinity` 根据亲和性规则调整分数。

**选定**就是挑出得分最高的那个。

常见调度因素：

| 机制 | 作用 |
|---|---|
| `resources.limits` / `resources.requests` | 资源限制和请求，影响预选和优选 |
| `nodeSelector` | 简单粗暴的节点标签选择 |
| `nodeAffinity` | 更灵活，支持硬亲和/软亲和 |
| `podAffinity` / `podAntiAffinity` | Pod 之间的部署关系 |
| `taints` / `tolerations` | 污点和容忍，专门用来「驱逐」某些 Pod |
| `topologySpreadConstraints` | Pod 拓扑分布，让 Pod 更均匀地分布 |

```bash
# 查看 Pod 调度失败的原因
kubectl describe pod mypod | grep -A 10 "Events"
```

:::details 追问：污点和容忍有什么区别？

`taints` 是节点的「性格」——比如「我不喜欢 Web 服务，只接受数据库 Pod」。

`tolerations` 是 Pod 的「意愿」——「没关系，我可以忍受」。

只有 Pod 的容忍能匹配节点的污点，调度器才会考虑这个节点。这和 `nodeAffinity` 正好相反——亲和是「我想去」，污点是「它愿意收」。
:::

## 网络：K8s 网络模型的三个承诺

大多数人会告诉你：「K8s 网络很复杂」。但本质上，它只做了三个承诺：

| 承诺 | 含义 |
|---|---|
| 每个 Pod 有唯一 IP | Pod 被视为独立的「虚拟机」，IP 不共享 |
| Pod 之间通信无需 NAT | 它们就像在同一个局域网里 |
| 节点与 Pod 通信无需 NAT | 节点可以直接访问 Pod IP |

这三个承诺实现了「IP-per-Pod」模型。实现方式由 CNI 插件决定：

- **Flannel**：VXLAN overlay，简单但有封装开销
- **Calico**：BGP 直连，性能好还支持网络策略
- **Cilium**：eBPF 方案，L7 层可见性

:::details 追问：Overlay 和直连的区别是什么？

Overlay（VXLAN）把 Pod 流量封装在 UDP 包里传输，节点之间通过隧道通信。有额外开销，但网络配置简单。

BGP 直连让节点间直接交换路由表，没有封装，效率更高。Calico 默认用 BGP，适合网络基础设施成熟的团队。
:::

### Service：服务发现的基石

为什么需要 Service？Pod IP 是临时的——每次重启都变。Service 提供了稳定的访问入口。

| 类型 | 访问范围 | 什么时候用 |
|---|---|---|
| `ClusterIP` | 集群内部 | 微服务间互相调用（默认） |
| `NodePort` | 集群内部 + 节点端口 | 开发测试、简单对外暴露 |
| `LoadBalancer` | 外部访问 | 生产环境，需要云厂商支持 |
| `ExternalName` | 集群内部 | 给外部服务起个别名 |
| `Headless` (clusterIP: None) | 集群内部 | 有状态服务，客户端需要知道具体 Pod IP |

:::details 追问：ClusterIP 是怎么工作的？

ClusterIP 是一个虚拟 IP，根本不会绑定到任何网络接口。当数据包发向 ClusterIP 时，kube-proxy 会做 DNAT——把目标 IP 悄悄换成后端 Pod 的真实 IP。

这就是为什么 ClusterIP 永远不会和 Pod IP 冲突，它只在集群内部路由。
:::

### Ingress：L7 层的路由中枢

Service 在 L4 层工作，Ingress 在 L7 层工作。它们不是替代关系，是不同层级的互补。

| 维度 | Ingress | Service |
|---|---|---|
| 工作层级 | HTTP/HTTPS | TCP/UDP |
| 路由方式 | 域名 + 路径 | 端口 |
| TLS 终结 | 原生支持 | 需要 sidecar 或 node-level |
| 资源开销 | 一个入口 | 多个 LoadBalancer |

Ingress 不是内置的——你需要部署 Ingress Controller（常见的是 Nginx Ingress）来处理流量。

## 工作负载：从无状态到有状态

### Deployment vs StatefulSet：该不该有序？

这是面试高频题，但大多数人的回答只停留在「Deployment 是无状态的，StatefulSet 是有状态的」。

真正理解这个区别，要看它们各自「稳定」的是什么：

| 维度 | Deployment | StatefulSet |
|---|---|---|
| Pod 标识 | 随机 hash 后缀 | 稳定序号（pod-0, pod-1...） |
| 存储 | 共享存储或随机 PVC | 每个 Pod 独立的 PVC |
| 扩缩容 | 随意并行 | 必须按序号顺序 |
| DNS | 无 | 稳定域名 `svc.ns.svc.cluster.local` |

StatefulSet 的「稳定」体现在三个方面：

- **稳定的网络标识**：`pod-name-0.service-name.namespace.svc.cluster.local`
- **稳定的存储**：VolumeClaimTemplate 为每个 Pod 绑定独立 PVC
- **有序的部署/扩缩容**：从序号大的开始，逐个操作

:::tip 一个记忆技巧

记住「NFS」——**N**ame（稳定名称）、**F**ile（稳定存储）、**S**equence（有序操作）。有这仨需求的，就该用 StatefulSet。
:::

### 滚动更新：K8s 是怎么做不停机部署的？

滚动更新由两个参数控制：

```yaml
spec:
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1        # 最多超出期望副本数 1 个
      maxUnavailable: 0  # 不允许 Pod 不可用
```

- `maxSurge`：允许同时存在的 Pod 超过期望值多少
- `maxUnavailable`：允许多少 Pod 不可用

```bash
# 滚动更新
kubectl rollout restart deployment/myapp

# 查看滚动状态
kubectl rollout status deployment/myapp

# 回滚
kubectl rollout undo deployment/myapp
kubectl rollout undo deployment/myapp --to-revision=2

# 查看历史
kubectl rollout history deployment/myapp
```

:::details 追问：蓝绿部署和金丝雀发布有什么区别？

**蓝绿部署**：同时跑两套，新版本完全就绪后，一次性切换流量。回滚快，但资源占用翻倍。

**金丝雀发布**：先让一小部分流量走到新版本，观察没问题后再逐步放量。资源利用率高，但回滚需要逐步回收。

K8s 本身只提供滚动更新的基础能力，金丝雀需要配合 Service 权重或 ServiceMesh（Istio）实现。
:::

## 资源与调度：不被资源问题坑

### requests 和 limits：调度和运行是两回事

这两个字段最容易混淆：

| 字段 | 影响 | 本质 |
|---|---|---|
| `requests` | 调度阶段 | 「我最少需要多少」——调度器据此选节点 |
| `limits` | 运行阶段 | 「我最多用多少」——超过可能被 kill |

```yaml
spec:
  containers:
  - name: app
    resources:
      requests:
        memory: "128Mi"
        cpu: "100m"
      limits:
        memory: "256Mi"
        cpu: "500m"
```

:::details 追问：如果只设置 limits 不设置 requests 会怎样？

调度器会认为这个 Pod 不消耗资源，可能把它调度到一个只剩 50MB 内存的节点。然后 Pod 一启动就被 OOM Kill 了。

**最佳实践**：requests 设置为业务正常消耗的上限，limits 设置为绝对不能超过的硬上限。
:::

### HPA：K8s 的自动伸缩能力

HorizontalPodAutoscaler 根据指标自动调整 Pod 副本数：

```bash
kubectl autoscale deployment myapp \
  --cpu-percent=80 \
  --min=2 \
  --max=10
```

核心算法很简单：

```
desiredReplicas = ceil[currentReplicas × (currentMetricValue / desiredMetricValue)]
```

比如当前 5 个副本，CPU 利用率 80%，目标是 80%——那还是 5 个。如果 CPU 飙到 160%，就扩到 10 个（上限）。

HPA 支持三类指标：

- **Resource Metrics**：CPU、内存（内置）
- **Custom Metrics**：Prometheus 等采集的业务指标
- **External Metrics**：集群外部服务的指标

## 存储：PV/PVC 抽象了什么？

PersistentVolume 和 PersistentVolumeClaim 解耦了「谁来提供存储」和「谁来使用存储」：

```
管理员 ──── 创建 PV (100Gi) ── 绑定 ── PVC (50Gi) ── Pod 使用
                                ↑
                         按需绑定或预绑定
```

生命周期四阶段：

```
Provisioning → Binding → Using → Reclaiming
```

回收策略决定了 Pod 删除后数据怎么办：

| 策略 | 后果 |
|---|---|
| `Retain` | 数据保留，人工处理 |
| `Delete` | 存储资源一起删除 |
| `Recycle` | 清理数据后可复用（已废弃） |

:::details 追问：静态 Provisioning 和动态 Provisioning 的区别？

**静态 Provisioning**：管理员提前创建好 PV，Pod 直接申请。适合存储资源固定的环境。

**动态 Provisioning**：没有匹配的 PV 时，StorageClass 自动创建新 PV。按需分配，更灵活，是现在的默认推荐方式。
:::

## 安全：多层防御体系

K8s 安全不是单一机制，而是多层防御：

| 层次 | 机制 | 作用 |
|---|---|---|
| API 访问 | Authentication → Authorization → Admission | 谁能访问？能做什么？做了合规吗？ |
| 权限控制 | RBAC | 最小权限原则 |
| 网络隔离 | NetworkPolicy | Pod 级别的网络防火墙 |
| 容器安全 | SecurityContext、PodSecurityStandards | 容器内能做什么操作？ |
| 敏感数据 | Secret、TLS | 密码、证书怎么管理 |

### RBAC 的四个要素

RBAC 通过角色绑定来控制权限：

- **Role / ClusterRole**：定义一组权限（能做什么）
- **RoleBinding / ClusterRoleBinding**：把角色绑定到谁

Role 在命名空间内生效，ClusterRole 是集群级别的。

:::details 追问：如何排查 RBAC 权限问题？

先用 `kubectl auth can-i` 模拟检查：

```bash
# 检查当前用户能不能做某操作
kubectl auth can-i get pods
kubectl auth can-i create pod --as=user-name -n namespace

# 完整排查：列出所有权限
kubectl auth can-i --list --as=serviceaccount -n namespace
```
:::

### ConfigMap 和 Secret：怎么选，怎么用？

Pod 获取配置有两条路：

```yaml
# 环境变量：注入时值固定，不支持热更新
env:
  - name: CONFIG_VALUE
    valueFrom:
      configMapKeyRef:
        name: my-config
        key: config-key

# Volume 挂载：文件形式挂载，支持热更新
volumeMounts:
  - name: config
    mountPath: /etc/config
volumes:
  - name: config
    configMap:
      name: my-config
```

:::details 追问：Secret 为什么比 ConfigMap 安全？

Secret 不是加密存储的——默认只是 Base64 编码，任何能访问 API 的人都能读取。所以生产环境要配合 `encryption-provider` 开启静态加密，或者用 Vault 等外部密钥管理系统。

真正敏感的数据，不要放在 K8s Secret 里。
:::

## 服务发现与 DNS：Pod 怎么找到另一个 Pod？

K8s 用 CoreDNS 提供服务发现。每个 Service 创建后都会自动注册到 DNS：

```
# 普通 Service DNS
my-svc.my-ns.svc.cluster.local → ClusterIP

# Headless Service DNS
my-svc.my-ns.svc.cluster.local → [PodIP1, PodIP2, ...]

# Pod DNS（特殊格式）
10-244-1-5.my-ns.pod.cluster.local → 10.244.1.5
```

Pod 的 `/etc/resolv.conf` 配置了搜索路径：

```
nameserver 10.96.0.10
search my-ns.svc.cluster.local svc.cluster.local cluster.local
options ndots:5
```

`ndots:5` 是什么意思？当你访问 `db` 时，DNS 会尝试 `db.my-ns.svc.cluster.local`、`db.svc.cluster.local`、`db.cluster.local`，最后才查 `db`。这是为了兼容传统应用的寻址习惯。

## 工具与运维：日常离不开的命令

```bash
# 资源操作
kubectl get pods -n namespace -o wide
kubectl describe pod mypod
kubectl apply -f deployment.yaml
kubectl delete -f deployment.yaml

# 日志和调试
kubectl logs mypod -f --tail=100
kubectl exec -it mypod -- /bin/sh
kubectl port-forward mypod 8080:80

# 扩缩容
kubectl scale deployment myapp --replicas=5
kubectl autoscale deployment myapp --cpu-percent=80 --min=2 --max=10

# 滚动更新
kubectl rollout status deployment/myapp
kubectl rollout undo deployment/myapp

# 上下文切换
kubectl config use-context prod-cluster
```

:::details 追问：kubectl exec 和 kubectl cp 有什么区别？

`kubectl exec` 在容器内执行命令，适合调试运行时问题。

`kubectl cp` 复制文件进容器，但**无法复制正在被容器写入的文件**（文件句柄被占用）。如果 Pod 使用 emptyDir，Pod 删除后数据就丢了——这往往是排查时容易踩的坑。
:::

## 包管理：Helm 和 Kustomize

Helm 是 K8s 的包管理器，核心概念是 Chart：

```bash
# 添加仓库
helm repo add bitnami https://charts.bitnami.com/bitnami

# 安装
helm install my-mysql bitnami/mysql

# 升级
helm upgrade my-mysql bitnami/mysql -f values.yaml

# 卸载
helm uninstall my-mysql
```

Chart 结构：

```
my-chart/
├── Chart.yaml          # 元信息
├── values.yaml         # 默认配置
├── charts/             # 依赖的子 Chart
└── templates/          # K8s 资源模板
    ├── deployment.yaml
    └── service.yaml
```

:::details 追问：Kustomize 和 Helm 怎么选？

**Kustomize**：覆盖式配置，「在已有 yaml 上打补丁」，适合多环境管理（dev/staging/prod）。

**Helm**：包管理式，「用模板生成 yaml」，适合分发和安装复杂应用。

两者可以结合：Helm Chart 作为基础，Kustomize 管理环境差异。
:::

### etcd 备份：集群的最后一道防线

etcd 保存了集群所有状态，备份是必做的：

```bash
# 备份
ETCDCTL_API=3 etcdctl \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key \
  snapshot save /tmp/etcd-backup.db

# 恢复
ETCDCTL_API=3 etcdctl \
  snapshot restore /tmp/etcd-backup.db \
  --data-dir=/var/lib/etcd/restore
```

:::tip 备份策略建议

etcd 数据不大（通常几 GB），建议：
- 每小时增量备份
- 每日全量备份
- 备份文件异地存储
- 定期做恢复演练
:::

## 总结：面试背后的设计哲学

K8s 每一项设计背后都有取舍——

**为什么 Pod IP 是临时的？** 为了简化调度和扩缩容，IP 绑死反而是负担。Service 作为稳定入口，解决了这个问题。

**为什么要有 StatefulSet？** 某些应用天然依赖有序性（数据库主从、消息队列）。无状态模型虽然简单，但不适合所有场景。

**为什么 PV/PVC 要分离？** 管理员擅长预估存储规模，应用开发者擅长估算需求。抽象解耦了两种角色的关注点。

**面试追问方向：**

- 组件故障时，集群如何自愈？
- Pod 调度失败，有哪些排查思路？
- 多个 Controller 同时运行，如何避免冲突？
- 如何实现 zero-downtime 部署？
