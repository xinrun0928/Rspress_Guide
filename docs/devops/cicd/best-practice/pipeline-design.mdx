# 流水线设计最佳实践

你以为 CI/CD 流水线就是把「编译 → 测试 → 部署」串起来？错。设计不当的流水线会让 CI 运行 1 小时，部署靠运气，出问题两眼一抹黑。

好的流水线设计，是让交付过程可预测、可追溯、可靠。

## 流水线设计原则

### 核心目标

| 目标 | 说明 |
|---|---|
| **快速反馈** | 开发提交代码后尽快知道结果 |
| **可靠性** | 每次执行结果一致 |
| **可追溯** | 任何问题都能追溯到具体提交 |
| **可维护** | 流水线本身要易于理解和修改 |

### 反模式

:::danger
- 所有阶段串行执行，没有并行
- 没有测试，只有构建
- 没有环境区分，全部部署到生产
- 流水线的执行时间超过 1 小时
- 依赖人工审批的环节过多
:::

## 阶段设计

### 经典四阶段

```yaml title="stages-basic.yaml"
stages:
  - build          # 编译构建
  - test           # 测试
  - security       # 安全
  - deploy         # 部署
```

### 扩展阶段

```yaml title="stages-extended.yaml"
stages:
  - prepare        # 准备（依赖安装、缓存加载）
  - build          # 编译构建
  - test           # 测试
  - quality        # 代码质量
  - security       # 安全扫描
  - artifact       # 制品管理
  - deploy-staging # 预发布部署
  - verify         # 验证
  - deploy-prod    # 生产部署
  - monitor        # 监控
```

### 动态阶段

```yaml title="dynamic-stages.yaml"
stages:
  - build
  - test
  - deploy

# 基础镜像
.base: &base
  image: maven:3.9-eclipse-temurin-17
  cache:
    key: maven-${CI_COMMIT_REF_SLUG}
    paths:
      - .m2/repository

# 构建阶段
build:
  <<: *base
  stage: build
  script:
    - mvn clean package -DskipTests
  artifacts:
    paths:
      - target/*.jar
    expire_in: 1 week

# 测试阶段
test:unit:
  <<: *base
  stage: test
  script:
    - mvn test

test:integration:
  <<: *base
  stage: test
  services:
    - postgres:15
  script:
    - mvn verify -DskipUnitTests
  allow_failure: true
```

## 并行执行

### 任务并行

```yaml title="parallel-tasks.yaml"
stages:
  - build
  - test
  - deploy

build:
  stage: build
  script:
    - ./gradlew assemble --parallel

test:api:
  stage: test
  script:
    - ./gradlew test --tests "*ApiTest"
  parallel:
    matrix:
      - API_VERSION: ["v1", "v2", "v3"]

test:e2e:
  stage: test
  script:
    - npm run test:e2e
```

### 矩阵策略

```yaml title="matrix-strategy.yaml"
test:matrix:
  stage: test
  parallel:
    matrix:
      - BROWSER: ["chrome", "firefox", "safari"]
        RESOLUTION: ["1920x1080", "1366x768"]
  script:
    - npm run test:browser -- --browser=$BROWSER --resolution=$RESOLUTION
```

## 条件执行

### 分支过滤

```yaml title="branch-filter.yaml"
# 所有分支都执行
build:
  stage: build
  script:
    - mvn package

# 只有 MR 和 main 分支执行
test:
  stage: test
  only:
    - merge_requests
    - main
  script:
    - mvn test

# 只有 develop 分支执行
deploy:staging:
  stage: deploy
  only:
    - develop
  script:
    - kubectl apply -f k8s/staging/
```

### 变量条件

```yaml title="variable-conditions.yaml"
deploy:prod:
  stage: deploy
  script:
    - kubectl apply -f k8s/prod/
  rules:
    - if: $CI_COMMIT_BRANCH == "main"
      when: manual
    - if: $CI_COMMIT_TAG
      when: automatic
```

### 变更检测

```yaml title="change-detection.yaml"
build:
  stage: build
  rules:
    - if: $CI_PIPELINE_SOURCE == "merge_request_event"
      changes:
        - "**/*.java"
        - "**/pom.xml"
      exists:
        - "pom.xml"
    - if: $CI_COMMIT_BRANCH == "main"
  script:
    - mvn clean package
```

## 依赖管理

### 阶段依赖

```yaml title="stage-dependencies.yaml"
stages:
  - build
  - test
  - deploy

# 后端构建
build:backend:
  stage: build
  script:
    - mvn package -DskipTests
  artifacts:
    paths:
      - target/*.jar

# 前端构建
build:frontend:
  stage: build
  script:
    - npm install
    - npm run build
  artifacts:
    paths:
      - dist/

# 依赖后端构建
test:backend:
  stage: test
  dependencies:
    - build:backend
  script:
    - mvn test

# 依赖前端构建
deploy:frontend:
  stage: deploy
  dependencies:
    - build:frontend
  script:
    - aws s3 sync dist/ s3://my-bucket/
```

### 工件依赖

```yaml title="artifact-dependencies.yaml"
build:shared-lib:
  stage: build
  script:
    - mvn install -DskipTests
  artifacts:
    paths:
      - ~/.m2/repository
    expire_in: 1 week

test:app:
  stage: test
  dependencies:
    - build:shared-lib
  script:
    - mvn test
```

## 缓存策略

### Maven 缓存

```yaml title="maven-cache.yaml"
variables:
  MAVEN_OPTS: "-Dmaven.repo.local=$CI_PROJECT_DIR/.m2/repository"

cache:
  key: maven-${CI_COMMIT_REF_SLUG}
  paths:
    - .m2/repository
  policy: pull-push

build:
  stage: build
  script:
    - mvn dependency:go-offline
    - mvn package
```

### npm 缓存

```yaml title="npm-cache.yaml"
cache:
  key: npm-${CI_COMMIT_REF_SLUG}
  paths:
    - node_modules/
  policy: pull-push

install:
  stage: build
  script:
    - npm ci --cache .npm
```

## 失败处理

### 重试策略

```yaml title="retry-policy.yaml"
deploy:
  stage: deploy
  retry:
    max: 2
    when:
      - runner_system_failure
      - stuck_or_timeout_failure
  script:
    - kubectl apply -f k8s/
```

### 失败通知

```yaml title="failure-notify.yaml"
notify_failure:
  stage: notify
  script:
    - |
      if [ "$CI_JOB_STATUS" == "failed" ]; then
        curl -X POST "https://api.example.com/alert" \
          -H "Content-Type: application/json" \
          -d "{\"job\":\"$CI_JOB_NAME\",\"status\":\"failed\",\"commit\":\"$CI_COMMIT_SHA\"}"
      fi
  when: always
```

## 模板化设计

### 公共模板

```yaml title=".gitlab-ci-templates.yml"
# .gitlab/ci/build.yml
.build_template: &build_template
  image: maven:3.9-eclipse-temurin-17
  before_script:
    - mvn dependency:go-offline
  script:
    - mvn package
  artifacts:
    paths:
      - target/*.jar
```

```yaml title=".gitlab-ci.yml"
include:
  - local: .gitlab/ci/build.yml
  - remote: https://example.com/gitlab-ci-template.yml

stages:
  - build
  - test
  - deploy

build:backend:
  <<: *build_template
  stage: build

build:frontend:
  extends: .build_template
  image: node:20
  script:
    - npm ci
    - npm run build
```

### 多项目流水线

```yaml title="trigger-downstream.yml"
trigger:integration:
  stage: trigger
  trigger:
    project: my-group/integration-tests
    strategy: depend
  variables:
    APP_VERSION: $CI_COMMIT_SHORT_SHA
```

## 性能优化

### 减少克隆深度

```yaml title="shallow-clone.yaml"
variables:
  GIT_DEPTH: 10

fetch:depth: 0
  variables:
    GIT_DEPTH: 0
  script:
    - git fetch origin main:refs/remotes/origin/main
```

### Docker 层缓存

```yaml title="docker-cache.yaml"
build:docker:
  image: docker:24
  services:
    - docker:dind
  before_script:
    - docker login -u $CI_REGISTRY_USER -p $CI_REGISTRY_PASSWORD $CI_REGISTRY
  script:
    - docker build --cache-from $CI_REGISTRY_IMAGE:latest .
    - docker push $CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA
```

## 总结

流水线设计的核心要点：

| 设计原则 | 说明 |
|---|---|
| 快速反馈 | 流水线总时间控制在 15 分钟内 |
| 阶段清晰 | 每个阶段有明确的职责 |
| 并行执行 | 充分利用 CI 资源 |
| 条件执行 | 避免不必要的执行 |
| 缓存复用 | 减少重复下载 |
| 失败友好 | 失败后易于定位 |

记住：好的流水线不是一次成型的，需要持续优化。
