# 蓝绿部署与金丝雀发布流水线

你有没有想过——新版本上线时，如何才能做到零停机？传统的停机部署已经无法满足现代互联网的需求了。

蓝绿部署和金丝雀发布是两种最流行的部署策略。蓝绿部署通过维护两套完整环境实现即时切换，金丝雀发布通过渐进式流量分配降低发布风险。

## 蓝绿部署

### 原理

蓝绿部署维护两个完全相同的运行环境——「蓝」环境（当前生产版本）和「绿」环境（新版本）。更新时，流量一次性从蓝切换到绿。

```
┌─────────────────────────────────────────────────────────────┐
│                      蓝环境（v1，当前生产）                     │
│  ┌─────────────────────────────────────────────────────┐  │
│  │  Load Balancer                                      │  │
│  │  100% 流量 ──────────────────────────────────────┐ │  │
│  │                                                   ▼ │  │
│  │  ┌─────────┐  ┌─────────┐  ┌─────────┐        │ │  │
│  │  │  Pod-1  │  │  Pod-2  │  │  Pod-3  │        │ │  │
│  │  │ nginx:v1 │  │ nginx:v1 │  │ nginx:v1 │        │ │  │
│  │  └─────────┘  └─────────┘  └─────────┘        │ │  │
│  └─────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘

切换后：

┌─────────────────────────────────────────────────────────────┐
│                      绿环境（v2，待上线）                     │
│  ┌─────────────────────────────────────────────────────┐  │
│  │  Load Balancer                                      │  │
│  │  100% 流量 ──────────────────────────────────────┐ │  │
│  │                                                   ▼ │  │
│  │  ┌─────────┐  ┌─────────┐  ┌─────────┐        │ │  │
│  │  │  Pod-1  │  │  Pod-2  │  │  Pod-3  │        │ │  │
│  │  │ nginx:v2 │  │ nginx:v2 │  │ nginx:v2 │        │ │  │
│  │  └─────────┘  └─────────┘  └─────────┘        │ │  │
│  └─────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

### Kubernetes 实现

```yaml title="blue-green-deployment.yaml"
apiVersion: v1
kind: Namespace
metadata:
  name: production
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp-blue
  namespace: production
  labels:
    app: myapp
    slot: blue
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp
      slot: blue
  template:
    metadata:
      labels:
        app: myapp
        slot: blue
    spec:
      containers:
      - name: myapp
        image: myapp:v1
        ports:
        - containerPort: 8080
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp-green
  namespace: production
  labels:
    app: myapp
    slot: green
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp
      slot: green
  template:
    metadata:
      labels:
        app: myapp
        slot: green
    spec:
      containers:
      - name: myapp
        image: myapp:v2
        ports:
        - containerPort: 8080
```

### Service 切换

```bash
# 初始状态：流量指向 blue
kubectl patch service myapp -n production \
  -p '{"spec":{"selector":{"slot":"blue"}}}'

# 验证 green 环境
kubectl rollout status deployment/myapp-green -n production

# 验证通过后，切换流量到 green
kubectl patch service myapp -n production \
  -p '{"spec":{"selector":{"slot":"green"}}}'

# 回滚：切回 blue
kubectl patch service myapp -n production \
  -p '{"spec":{"selector":{"slot":"blue"}}}'
```

### 完整 CI/CD 流水线

```yaml title=".gitlab-ci.yml"
stages:
  - build
  - test
  - deploy-blue
  - deploy-green
  - smoke-test
  - switch
  - cleanup

variables:
  APP_NAME: myapp
  BLUE_GREEN_STRATEGY: "blue"

build:
  stage: build
  script:
    - docker build -t $APP_NAME:$CI_COMMIT_SHA .
    - docker push $IMAGE_NAME:$CI_COMMIT_SHA
    - docker tag $IMAGE_NAME:$CI_COMMIT_SHA $IMAGE_NAME:latest

deploy-blue:
  stage: deploy-blue
  script:
    - kubectl set image deployment/$APP_NAME-blue $APP_NAME=$IMAGE_NAME:$CI_COMMIT_SHA -n production
    - kubectl rollout status deployment/$APP_NAME-blue -n production --timeout=300s
  when: manual
  environment:
    name: production/blue
    url: https://blue.myapp.example.com
  only:
    - main

deploy-green:
  stage: deploy-green
  script:
    - kubectl set image deployment/$APP_NAME-green $APP_NAME=$IMAGE_NAME:$CI_COMMIT_SHA -n production
    - kubectl rollout status deployment/$APP_NAME-green -n production --timeout=300s
  environment:
    name: production/green
    url: https://green.myapp.example.com
  only:
    - main

smoke-test:
  stage: smoke-test
  script:
    - curl -f https://green.myapp.example.com/health || exit 1
    - kubectl exec -it test-pod -- curl -f http://myapp-green.production.svc.cluster.local:8080/health
  dependencies:
    - deploy-green

switch:
  stage: switch
  script:
    - kubectl patch service $APP_NAME -n production -p '{"spec":{"selector":{"slot":"green"}}}'
    - sleep 10
    - curl -f https://myapp.example.com/health
  environment:
    name: production
    action: promote
  dependencies:
    - smoke-test
  when: manual

cleanup:
  stage: cleanup
  script:
    - kubectl delete deployment $APP_NAME-blue -n production || true
    - kubectl label deployment $APP_NAME-green slot- || true
  when: manual
  only:
    - main
```

## 金丝雀发布

### 原理

金丝雀发布将小部分流量导向新版本（金丝雀），验证无误后逐步扩大比例。

```
流量分配过程：

阶段 1: 5% 流量到 v2
  v1 ████████████████████  95%
  v2 ██                      5%

阶段 2: 20% 流量到 v2
  v1 ████████████████       80%
  v2 ████                    20%

阶段 3: 50% 流量到 v2
  v1 ████████████            50%
  v2 ████████████            50%

阶段 4: 100% 流量到 v2（完全切换）
  v1                          0%
  v2 ████████████████████  100%
```

### Argo Rollouts 实现

```yaml title="canary-rollout.yaml"
apiVersion: argoproj.io/v1alpha1
kind: Rollout
metadata:
  name: myapp
spec:
  replicas: 10
  strategy:
    canary:
      # 步进式流量分配
      steps:
      - setWeight: 5       # 先引 5%
      - pause: {}           # 暂停，等待手动确认
      - analysis:
          templates:
          - templateName: success-rate
          args:
          - name: service-name
            value: myapp-canary
      - setWeight: 20      # 20%
      - pause: {duration: 10m}  # 等待 10 分钟
      - setWeight: 50      # 50%
      - pause: {}           # 暂停
      - setWeight: 100      # 全量
      # 分析模板（自动指标验证）
      analysis:
        templates:
        - templateName: success-rate
        startingStep: 2
        args:
        - name: service-name
          value: myapp-canary
      # 流量权重
      canaryMetadata:
        labels:
          role: canary
      stableMetadata:
        labels:
          role: stable
      trafficRouting:
        nginx:
          stableIngress: myapp-stable
          additionalIngressAnnotations:
            canary-weight: "5"
      # Pod 标签
      podMetadata:
        labels:
          app: myapp
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
      - name: myapp
        image: myapp:v2
        ports:
        - containerPort: 8080
```

### 分析模板

```yaml title="analysis-template.yaml"
apiVersion: argoproj.io/v1alpha1
kind: AnalysisTemplate
metadata:
  name: success-rate
spec:
  args:
  - name: service-name
  metrics:
  # HTTP 探测
  - name: success-rate
    interval: 1m
    successCondition: result[0] >= 0.95
    failureLimit: 3
    provider:
      job:
        spec:
          backoffLimit: 3
          template:
            spec:
              restartPolicy: Never
              containers:
              - name: success-rate
                image: curlimages/curl:latest
                command:
                - sh
                - -c
                - |
                  SUCCESS=$(curl -s -o /dev/null -w "%{http_code}" http://${SERVICE_NAME}/health)
                  if [ $SUCCESS -ge 200 ] && [ $SUCCESS -lt 300 ]; then
                    echo "1"
                  else
                    echo "0"
                  fi
  # Prometheus 指标
  - name: latency
    interval: 1m
    successCondition: result[0] <= 500
    provider:
      prometheus:
        address: http://prometheus:9090
        query: |
          histogram_quantile(0.95,
            sum(rate(http_request_duration_seconds_bucket{job="{{args.service-name}}"}[5m])) by (le)
          )
```

### Argo Rollouts 流水线

```yaml title=".gitlab-ci.yml"
stages:
  - build
  - deploy-canary
  - promote
  - full

build:
  stage: build
  script:
    - docker build -t $APP_NAME:$CI_COMMIT_SHA .
    - docker push $IMAGE_NAME:$CI_COMMIT_SHA

deploy-canary:
  stage: deploy-canary
  script:
    - kubectl argo rollouts set image rollout/myapp \
      myapp=$IMAGE_NAME:$CI_COMMIT_SHA
    - kubectl argo rollouts status myapp --watch --timeout=600s
  environment:
    name: production
    url: https://myapp.example.com
    on_stop: rollback

promote-canary:
  stage: promote
  script:
    - kubectl argo rollouts promote myapp
    - kubectl argo rollouts status myapp
  environment:
    name: production
    url: https://myapp.example.com
  when: manual

rollback:
  stage: rollback
  script:
    - kubectl argo rollouts abort myapp
    - kubectl argo rollouts undo myapp
  environment:
    name: production
    action: stop
  when: manual
```

## Argo CD + Argo Rollouts 集成

### Argo CD Application

```yaml title="argocd-application.yaml"
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: myapp
spec:
  project: production
  source:
    repoURL: https://github.com/example/myapp.git
    targetRevision: main
    path: k8s/overlays/production
  destination:
    server: https://kubernetes.default.svc
    namespace: production
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
```

## 最佳实践

### 蓝绿部署适合场景

| 场景 | 说明 |
|---|---|
| 数据库 schema 变更 | 需要原子切换 |
| 需要完整测试后上线 | 绿环境可完整验证 |
| 要求秒级回滚 | 切换瞬间完成 |
| 大版本更新 | 不兼容的版本需要整体切换 |

### 金丝雀发布适合场景

| 场景 | 说明 |
|---|---|
| 持续交付 | 渐进式验证 |
| A/B 测试 | 特定用户群测试 |
| 基于指标的自动回滚 | 指标不达标自动回滚 |
| 降低发布风险 | 灰度验证降低影响 |

### 关键考虑点

| 考虑点 | 蓝绿 | 金丝雀 |
|---|---|---|
| **资源成本** | 2x | 1.1x-1.5x |
| **流量切换速度** | 即时 | 渐进 |
| **回滚速度** | 秒级 | 分钟级 |
| **验证方式** | 完整测试 | 指标监控 |
| **适用版本** | 不兼容版本 | 兼容版本 |

## 总结

蓝绿部署和金丝雀发布各有优势：

1. **蓝绿部署**：维护两套环境，切换即时，回滚秒级，适合大版本更新
2. **金丝雀发布**：渐进式流量分配，风险可控，支持自动回滚，适合持续交付
3. **Argo Rollouts**：Kubernetes 原生的渐进式交付工具，支持金丝雀、Blue-Green、自动分析
4. **Argo CD**：GitOps 持续交付工具，与 Rollouts 配合实现完整的自动化部署

根据业务需求和技术能力选择合适的部署策略，可以在快速迭代和系统稳定之间找到平衡。
