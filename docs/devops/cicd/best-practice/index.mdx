# CI/CD 最佳实践

你有没有见过这样的场景：代码提交后，等待 CI 跑 30 分钟，部署靠手动，出了问题回滚要 1 小时。这就是没有 CI/CD 最佳实践的代价。

CI/CD 不是简单地把构建自动化，而是构建一套可靠的、可重复的交付流程。

## CI/CD 核心原则

### 持续集成（CI）

代码提交后自动完成：编译 → 单元测试 → 集成测试 → 构建产物。

| 原则 | 说明 |
|---|---|
| **快速反馈** | CI 流程应在 10 分钟内完成 |
| **保持绿色** | CI 失败后立即修复 |
| **自动化一切** | 无需手动操作 |
| **测试金字塔** | 单元测试多、集成测试中、E2E 测试少 |

### 持续交付（CD）

自动化将构建产物交付到各环境。

### 持续部署（Deployment）

自动化将构建产物部署到生产环境。

## 流水线设计

### 分阶段流水线

```yaml title="pipeline-stages.yaml"
stages:
  - build          # 编译构建
  - test           # 测试
  - security       # 安全扫描
  - artifact       # 制品管理
  - deploy-staging # 部署预发布
  - deploy-prod    # 部署生产
```

### 经典 GitLab CI 配置

```yaml title=".gitlab-ci.yml"
stages:
  - build
  - test
  - security
  - deploy

variables:
  DOCKER_IMAGE: registry.example.com/myapp
  DOCKER_TAG: ${CI_COMMIT_SHORT_SHA}

.build:
  image: maven:3.9-eclipse-temurin-17
  stage: build
  before_script:
    - mvn dependency:go-offline
  script:
    - mvn clean package -DskipTests
  artifacts:
    paths:
      - target/*.jar
    expire_in: 1 week

test:unit:
  extends: .build
  stage: test
  script:
    - mvn test
  coverage: '/Total:.*?([0-9]{1,3})%/'

test:integration:
  extends: .build
  stage: test
  services:
    - postgres:15
  variables:
    POSTGRES_DB: test
    POSTGRES_USER: test
    POSTGRES_PASSWORD: test
  script:
    - mvn verify -Dspring.profiles.active=test

security:dependency:
  stage: security
  image: aquasec/trivy:latest
  script:
    - trivy image --exit-code 1 --severity HIGH,CRITICAL $DOCKER_IMAGE:$DOCKER_TAG

deploy:staging:
  stage: deploy
  image: bitnami/kubectl:latest
  environment:
    name: staging
  script:
    - kubectl set image deployment/myapp myapp=$DOCKER_IMAGE:$DOCKER_TAG
    - kubectl rollout status deployment/myapp
  only:
    - develop

deploy:prod:
  stage: deploy
  image: bitnami/kubectl:latest
  environment:
    name: production
  script:
    - kubectl set image deployment/myapp myapp=$DOCKER_IMAGE:$DOCKER_TAG
    - kubectl rollout status deployment/myapp
  when: manual
  only:
    - main
```

## 构建优化

### 并行执行

```yaml title="parallel-jobs.yaml"
stages:
  - test

test:backend:
  stage: test
  parallel: 3
  script:
    - mvn test -Dtest="**/backend/**/*Test"

test:frontend:
  stage: test
  script:
    - npm test

test:api:
  stage: test
  script:
    - newman run postman/collection.json
```

### 增量构建

```yaml title="incremental-build.yaml"
build:
  stage: build
  script:
    - |
      # 只构建变更的模块
      CHANGED_FILES=$(git diff --name-only $CI_MERGE_REQUEST_DIFF_BASE_SHA HEAD)
      for module in $CHANGED_FILES; do
        mvn package -pl $(dirname $module) -am
      done
```

### 依赖缓存

```yaml title="cache-config.yaml"
maven-build:
  stage: build
  image: maven:3.9-eclipse-temurin-17
  cache:
    key: ${CI_COMMIT_REF_SLUG}
    paths:
      - .m2/repository
  script:
    - mvn package -DskipTests
```

## 质量门禁

### 测试覆盖率

```yaml title="coverage-gate.yaml"
test:coverage:
  stage: test
  script:
    - mvn test jacoco:report
  coverage: '/Total.*?([0-9]{1,3})%/'
  artifacts:
    reports:
      junit: target/surefire-reports/*.xml
      coverage_report:
        coverage_format: jacoco
        path: target/site/jacoco/jacoco.xml
  rules:
    - if: $CI_PIPELINE_SOURCE == "merge_request_event"
```

### 代码质量检查

```yaml title="code-quality.yaml"
code-quality:
  stage: test
  image: sonarsource/sonar-scanner-cli:latest
  variables:
    SONAR_USER_HOME: ${CI_PROJECT_DIR}/.sonar
  cache:
    key: ${CI_COMMIT_REF_SLUG}
    paths:
      - .sonar/cache
  script:
    - sonar-scanner -Dsonar.projectKey=$CI_PROJECT_NAME
  allow_failure: true
  only:
    - merge_requests
    - main
```

## 制品管理

### Docker 镜像构建

```yaml title="docker-build.yaml"
build:docker:
  stage: build
  image: docker:24
  services:
    - docker:dind
  before_script:
    - docker login -u $CI_REGISTRY_USER -p $CI_REGISTRY_PASSWORD $CI_REGISTRY
  script:
    - docker build -t $CI_REGISTRY_IMAGE:$DOCKER_TAG .
    - docker build -t $CI_REGISTRY_IMAGE:latest .
    - docker push $CI_REGISTRY_IMAGE:$DOCKER_TAG
    - docker push $CI_REGISTRY_IMAGE:latest
```

### 多架构构建

```yaml title="multi-arch.yaml"
build:docker:manifest:
  stage: build
  image: docker:24
  services:
    - docker:dind
  script:
    - |
      docker buildx create --use
      docker buildx build \
        --platform linux/amd64,linux/arm64 \
        --tag $CI_REGISTRY_IMAGE:$DOCKER_TAG \
        --push .
```

## 部署策略

### 蓝绿部署

```yaml title="blue-green.yaml"
deploy:blue:
  stage: deploy
  script:
    - |
      kubectl apply -f k8s/blue/
      kubectl rollout status deployment/myapp-blue
      kubectl patch service myapp -p '{"spec":{"selector":{"version":"blue"}}}'
  environment:
    name: production-blue
    url: https://blue.myapp.com

deploy:green:
  stage: deploy
  script:
    - |
      kubectl apply -f k8s/green/
      kubectl rollout status deployment/myapp-green
      kubectl patch service myapp -p '{"spec":{"selector":{"version":"green"}}}'
  environment:
    name: production-green
    url: https://green.myapp.com
```

### 金丝雀发布

```yaml title="canary.yaml"
deploy:canary:
  stage: deploy
  script:
    - |
      # 10% 流量
      kubectl patch hpa myapp -p '{"spec":{"minReplicas":3}}'
      kubectl set image deployment/myapp myapp=$CI_REGISTRY_IMAGE:canary
      # 验证通过后，切换全部流量
      kubectl set image deployment/myapp myapp=$CI_REGISTRY_IMAGE:$DOCKER_TAG
```

## 环境管理


## 回滚策略

### 自动回滚

```yaml title="rollback.yaml"
deploy:with-rollback:
  stage: deploy
  script:
    - |
      # 记录当前版本
      kubectl get deployment myapp -o jsonpath='{.spec.template.spec.containers[0].image}' > previous_version.txt
      
      # 部署新版本
      kubectl set image deployment/myapp myapp=$CI_REGISTRY_IMAGE:$DOCKER_TAG
      
      # 健康检查
      sleep 30
      if ! kubectl rollout status deployment/myapp; then
        echo "Deployment failed, rolling back..."
        kubectl rollout undo deployment/myapp
        exit 1
      fi
```

### 手动回滚

```bash
# GitLab CI 手动回滚
kubectl rollout undo deployment/myapp --to-revision=2

# 查看回滚历史
kubectl rollout history deployment/myapp
```

## 监控与反馈

### 部署通知

```yaml title="notifications.yaml"
notify:success:
  stage: notify
  script:
    - |
      curl -X POST "https://slack.com/api/chat.postMessage" \
        -H "Authorization: Bearer $SLACK_BOT_TOKEN" \
        -d "channel=$SLACK_CHANNEL" \
        -d "text=✅ Deployment successful: $CI_PROJECT_NAME:$DOCKER_TAG"
  only:
    - main
  when: on_success

notify:failure:
  stage: notify
  script:
    - |
      curl -X POST "https://slack.com/api/chat.postMessage" \
        -H "Authorization: Bearer $SLACK_BOT_TOKEN" \
        -d "channel=$SLACK_CHANNEL" \
        -d "text=❌ Deployment failed: $CI_PROJECT_NAME:$CI_COMMIT_REF_NAME"
  when: on_failure
```

## 总结

CI/CD 最佳实践的核心要点：

| 实践 | 价值 |
|---|---|
| 快速反馈 | 尽早发现问题 |
| 自动化一切 | 减少人为错误 |
| 质量门禁 | 保证交付质量 |
| 渐进式发布 | 降低部署风险 |
| 可观测性 | 快速定位问题 |
| 回滚策略 | 快速恢复服务 |

记住：好的 CI/CD 流程是团队效率的倍增器。
