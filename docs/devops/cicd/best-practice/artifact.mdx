# 制品管理实践

构建完成后的 JAR 包、Docker 镜像、前端资源包……这些构建产物就是「制品」。制品管理，就是如何安全、可靠地存储、分发这些制品。

## 为什么需要制品管理

### 问题场景

- 开发 A：「我本地构建的 JAR 能用吗？」
- 开发 B：「测试环境的镜像和生产一样吗？」
- 运维：「这个 Bug 是哪个版本的代码？」

没有制品管理，就是一团乱麻。

### 制品管理的价值

| 价值 | 说明 |
|---|---|
| **可追溯** | 每个制品对应哪个代码版本 |
| **可重现** | 任意时间点的构建都能复现 |
| **安全性** | 制品来源可验证 |
| **版本控制** | 版本管理、回滚方便 |

## Maven 仓库

### Nexus 部署

```yaml title="nexus-statefulset.yaml"
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: nexus
  namespace: tools
spec:
  serviceName: nexus
  replicas: 1
  selector:
    matchLabels:
      app: nexus
  template:
    spec:
      containers:
      - name: nexus
        image: sonatype/nexus3:3.60.0
        ports:
        - containerPort: 8081
          name: web
        - containerPort: 8082
          name: docker-hosted
        resources:
          requests:
            memory: 2Gi
          limits:
            memory: 4Gi
        volumeMounts:
        - name: nexus-data
          mountPath: /nexus-data
      volumes:
      - name: nexus-data
        persistentVolumeClaim:
          claimName: nexus-pvc
```

### 仓库类型

| 类型 | 用途 | 说明 |
|---|---|---|
| **hosted** | 托管仓库 | 存放内部制品 |
| **proxy** | 代理仓库 | 代理远程仓库（Maven Central） |
| **group** | 仓库组 | 组合多个仓库 |
| **snapshot** | 快照仓库 | 存放 SNAPSHOT 版本 |

### Nexus 配置

```xml title="nexus-repositories.json"
{
  "name": "maven-releases",
  "type": "hosted",
  "format": "maven2",
  "url": "http://nexus:8081/repository/maven-releases/",
  "cleanupPolicy": {
    "age": "30d"
  }
}
```

### Maven 配置

```xml title="settings.xml"
<servers>
  <server>
    <id>nexus-releases</id>
    <username>admin</username>
    <password>admin123</password>
  </server>
</servers>
```

```xml title="pom.xml"
<distributionManagement>
  <repository>
    <id>nexus-releases</id>
    <url>http://nexus:8081/repository/maven-releases/</url>
  </repository>
  <snapshotRepository>
    <id>nexus-snapshots</id>
    <url>http://nexus:8081/repository/maven-snapshots/</url>
  </snapshotRepository>
</distributionManagement>
```

### 镜像推送

```bash
# 登录 Harbor
docker login harbor.example.com

# Tag 镜像
docker tag myapp:latest harbor.example.com/library/myapp:latest

# 推送镜像
docker push harbor.example.com/library/myapp:latest

# Helm Chart 推送
helm repo add harbor https://harbor.example.com/chartrepo/library
helm push mychart-1.0.0.tgz harbor
```

### 镜像签名

```bash
# 使用 Cosign 签名
cosign generate-key-pair
cosign sign --key cosign.key harbor.example.com/library/myapp:v1.0.0

# 验证签名
cosign verify --key cosign.pub harbor.example.com/library/myapp:v1.0.0
```

## CI/CD 集成

### GitLab CI 制品管理

```yaml title="gitlab-artifacts.yml"
stages:
  - build
  - test
  - publish

build:
  stage: build
  image: maven:3.9-eclipse-temurin-17
  script:
    - mvn clean package -DskipTests
  artifacts:
    paths:
      - target/*.jar
    expire_in: 1 week
    reports:
      junit: target/surefire-reports/*.xml

publish:jar:
  stage: publish
  image: maven:3.9-eclipse-temurin-17
  script:
    - mvn deploy -DskipTests
  dependencies:
    - build
  only:
    - main
    - tags

publish:docker:
  stage: publish
  image: docker:24
  services:
    - docker:dind
  before_script:
    - docker login -u $CI_REGISTRY_USER -p $CI_REGISTRY_PASSWORD $CI_REGISTRY
  script:
    - docker build -t $CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA .
    - docker build -t $CI_REGISTRY_IMAGE:$CI_COMMIT_TAG .
    - docker push $CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA
    - docker push $CI_REGISTRY_IMAGE:$CI_COMMIT_TAG
  dependencies:
    - build
  only:
    - main
    - tags
```

### GitHub Actions

```yaml title="publish.yml"
name: Publish Package

on:
  push:
    branches: [main]
    tags: ['v*']

jobs:
  publish:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - name: Set up JDK
        uses: actions/setup-java@v4
        with:
          java-version: '17'
          distribution: 'temurin'
          cache: 'maven'
      
      - name: Build with Maven
        run: mvn clean package -DskipTests
      
      - name: Publish to Maven Central
        run: mvn deploy
        env:
          GPG_PASSPHRASE: ${{ secrets.GPG_PASSPHRASE }}
          GPG_PRIVATE_KEY: ${{ secrets.GPG_PRIVATE_KEY }}
          OSSRH_USERNAME: ${{ secrets.OSSRH_USERNAME }}
          OSSRH_TOKEN: ${{ secrets.OSSRH_TOKEN }}
      
      - name: Push to GitHub Packages
        run: mvn package
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

## 版本管理策略

### Git Flow 与版本

```
main ──────────────────────────────────────────
         │          │              │
         ▼          ▼              ▼
       v1.0.0    v1.0.1         v1.1.0
```

| 版本类型 | 格式 | 说明 |
|---|---|---|
| **Release** | 1.0.0 | 正式发布 |
| **Pre-release** | 1.0.0-alpha | 内部测试 |
| **Snapshot** | 1.0.0-SNAPSHOT | 开发中 |
| **Commit** | 1.0.0-gabc123 | 特定提交 |

### 自动版本号

```yaml title="version-from-git.yml"
build:
  stage: build
  script:
    - |
      # 从 Git Tag 获取版本
      VERSION=$(git describe --tags --abbrev=0)
      
      # 或从 Commit 生成版本
      if [ -z "$VERSION" ]; then
        VERSION="0.0.0-$CI_COMMIT_SHORT_SHA"
      fi
      
      # 构建时注入版本
      mvn package -Dversion=$VERSION
  artifacts:
    paths:
      - target/*.jar
```

## 清理策略

### Harbor 清理

```yaml title="harbor-retention.yaml"
# Harbor GC 和保留策略
apiVersion: goharbor.go/v1alpha2
kind: RetentionPolicy
metadata:
  name: cleanup-policy
spec:
  algorithm: or
  rules:
  - scope: repository
    args:
      most_recently_pulled: 5
    tag_selectors:
    - kind: tag
      decoration: repos
      pattern: ".*"
      removed: false
```

### Nexus 清理

```json title="nexus-cleanup-policy.json"
{
  "name": "cleanup-policy",
  "format": "maven2",
  "action": "delete",
  "configuration": {
    "minimum_retained_manifests": 1,
    "age": "30d",
    "keep_snapshot_versions": 0
  }
}
```

## 最佳实践

### 1. 制品仓库隔离

| 环境 | 仓库 | 说明 |
|---|---|---|
| 开发 | snapshot | 频繁更新 |
| 测试 | releases-test | 稳定版本 |
| 生产 | releases-prod | 经过验证 |

### 2. 签名验证

```bash
# 验证 JAR 签名
gpg --verify myapp-1.0.0.jar.asc myapp-1.0.0.jar

# 验证 Docker 镜像
cosign verify --key cosign.pub harbor.example.com/myapp:v1.0.0
```

### 3. 访问控制


## 总结

制品管理的核心要点：

| 实践 | 说明 |
|---|---|
| 使用专用仓库 | Maven Central、Docker Hub Harbor |
| 版本可追溯 | 每次构建有唯一版本号 |
| 签名验证 | 确保制品来源可信 |
| 自动清理 | 避免存储无限增长 |
| SBOM 管理 | 了解制品的组成 |

记住：制品是 CI/CD 的输出，也是 CD 的输入。好的制品管理是可靠交付的基础。
