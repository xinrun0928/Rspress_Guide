# 现代 CI/CD 工具

Jenkins 诞生于 2004 年，那个年代 Kubernetes 还不存在，Docker 还未诞生，GitHub 刚刚成立。当 Jenkins 2.0 在 2016 年引入 Pipeline 时，DevOps 世界的格局已经开始改变。

云原生时代呼唤新一代 CI/CD 工具：它们生于 Kubernetes，服务于 Kubernetes；它们用声明式配置取代代码式脚本；它们与 Git 深度集成，将 GitOps 理念贯穿始终。

这一章，我们将深入探讨这些现代 CI/CD 工具——GitLab CI 以 YAML 驱动一切，GitHub Actions 让 Actions 市场触手可及，ArgoCD 引领 GitOps 潮流，Tekton 成为 Kubernetes 原生的构建标准，Spinnaker 则在多云部署场景下独占鳌头。

## 现代 CI/CD 的共同特征

### 云原生原生

现代 CI/CD 工具的设计哲学与 Kubernetes 一脉相承：

| 特征 | 说明 |
|---|---|
| 声明式配置 | 用 YAML 描述期望状态，而非过程式脚本 |
| 容器化执行 | 每个任务在独立容器中运行，环境隔离 |
| Kubernetes 原生 | 以 CRD 形式扩展 Kubernetes 能力 |
| GitOps 驱动 | Git 是唯一的真相来源，声明即部署 |

### 与传统工具的对比

| 维度 | Jenkins | 现代 CI/CD |
|---|---|---|
| 配置方式 | Jenkinsfile (Groovy) | YAML 声明式 |
| 环境隔离 | 虚拟机/Agent | 容器 |
| 扩展方式 | 插件系统 | Kubernetes CRD |
| 状态管理 | Jenkins Master | Kubernetes etcd |
| 部署模型 | Push-based | Pull-based (GitOps) |
| 学习曲线 | 较陡 | 平缓 |

## 目录结构

本章节涵盖当前最主流的现代 CI/CD 工具：

**GitLab CI**：GitLab 自带的 CI/CD 功能，`.gitlab-ci.yml` 驱动一切，开箱即用，适合使用 GitLab 的团队。

**GitHub Actions**：GitHub 的自动化工具，Actions 市场提供海量现成 Action，是开源项目的首选。

**ArgoCD**：GitOps 的旗舰实现，专注于 Kubernetes 应用的持续部署，是云原生部署的事实标准。

**Tekton**：Kubernetes 原生的流水线框架，由 Google 主导，为构建可移植的 CI/CD 组件而生。

**Spinnaker**：Netflix 开源的多云持续部署平台，擅长金丝雀发布、蓝绿部署等高级部署策略。

**工具对比**：最后一节，我们将从多个维度对比这些工具，帮助你做出选择。

## 选择指南

| 你的场景 | 推荐工具 |
|---|---|
| 使用 GitLab | GitLab CI |
| 开源项目 | GitHub Actions |
| 纯 Kubernetes 部署 | ArgoCD + Tekton |
| 多云/混合云部署 | Spinnaker |
| 追求快速上手 | GitHub Actions |
| 企业级复杂场景 | Tekton + ArgoCD |
