# 现代 CI/CD 工具

「Jenkins 太重了，有没有更轻量的选择？」

这是很多团队在选型时会问的问题。GitLab CI 集成在 GitLab 里，GitHub Actions 集成在 GitHub 里，Tekton 是 Kubernetes 原生的 CI/CD——每个工具都有自己的定位。

现代 CI/CD 工具的共同特点是：**配置即代码、YAML 驱动、云原生友好**。

## GitOps：声明式部署的未来

GitOps 是现代 CI/CD 的核心理念。它的核心是：**Git 是唯一的真理来源**。

```
┌─────────────────────────────────────────────────────────────────┐
│                        GitOps 工作流                              │
│                                                                  │
│  ┌─────────────┐    ┌─────────────┐    ┌─────────────┐         │
│  │  开发者提交   │───►│  CI 构建    │───►│  推送镜像   │         │
│  │  代码        │    │  测试        │    │  更新配置   │         │
│  └─────────────┘    └─────────────┘    └──────┬──────┘         │
│                                                │                  │
│  ┌─────────────┐    ┌─────────────┐            ▼                  │
│  │  ArgoCD    │◄───│  Git 仓库   │◄────────────────────         │
│  │  自动同步   │    │  (配置)     │    更新 Helm Chart / YAML    │
│  └──────┬─────┘    └─────────────┘                              │
│         │                                                         │
│         ▼                                                         │
│  ┌─────────────┐    ┌─────────────┐                              │
│  │  Kubernetes │    │  状态偏差   │                              │
│  │  集群        │◄───│  告警/阻断  │                              │
│  └─────────────┘    └─────────────┘                              │
└─────────────────────────────────────────────────────────────────┘
```

GitOps 的优势：
- **版本可控**：所有配置变更都有 Git 历史，可审计、可回滚
- **环境一致**：Git 仓库的配置 = 生产环境的实际状态
- **权限清晰**：开发者只能改代码，部署权限由 GitOps 工具控制

## GitLab CI vs GitHub Actions

两者都是「集成在代码托管平台」的 CI/CD 工具，但有细微差异：

| 维度 | GitLab CI | GitHub Actions |
|------|-----------|----------------|
| 配置文件 | `.gitlab-ci.yml` | `.github/workflows/*.yml` |
|  runner 数量 | 自托管 + Shared | GitHub 托管 + 自托管 |
| 市场生态 | GitLab Marketplace | GitHub Marketplace |
| 流水线可视化 | 内置 | 需安装 Actions 插件 |
| Kubernetes 集成 | Auto DevOps | workflows-k8s |
| 价格 | 免费 2000 分钟/月 | 免费 2000 分钟/月（公开仓库无限） |

**选型建议**：
- 代码在 GitLab → 直接用 GitLab CI
- 代码在 GitHub → 直接用 GitHub Actions
- 需要多云部署 → Tekton 或 ArgoCD

## Tekton：Kubernetes 原生的 CI/CD

Tekton 的设计哲学和 Kubernetes 完全一致：**一切皆 CRD**。

```yaml
# Tekton Task 定义
apiVersion: tekton.dev/v1beta1
kind: Task
metadata:
  name: build-push-image
spec:
  steps:
    - name: build
      image: kaniko/executor:latest
      script: |
        #!/bin/sh
        /kaniko/executor \
          --context $(params.git-url) \
          --destination $(params.image):$(params.tag)
```

Tekton 的优势：
- **Kubernetes 原生**：用 K8s 的方式管理 CI/CD 资源
- **可扩展**：可以自定义 Task、Pipeline、Trigger
- **多租户**：不同团队、不同 Namespace 隔离

## 面试的核心逻辑

现代 CI/CD 工具的面试，重点考察三个方面：

1. **工具选型**：什么场景用 GitLab CI？什么场景用 Tekton？为什么 GitOps 是趋势？
2. **原理理解**：GitLab CI 的 DAG 流水线是怎么实现的？ArgoCD 的自动同步是怎么工作的？
3. **工程实践**：如何设计一个多环境的 CI/CD 流水线？如何实现回滚？

```
# 现代 CI/CD 工具选型决策树

需要部署到 Kubernetes？
├── 是 → 选 Tekton 或 ArgoCD
└── 否 → 选 GitLab CI 或 GitHub Actions

代码托管在哪里？
├── GitLab → GitLab CI
├── GitHub → GitHub Actions
└── 其他 → Jenkins 或 Tekton

是否需要 GitOps？
├── 是 → ArgoCD 或 Jenkins X
└── 否 → Tekton 或传统 CI
```

> "工具的选择不是目的，交付的价值才是目的。选择最合适的，而不是最流行的——这是工程判断力的体现。"
