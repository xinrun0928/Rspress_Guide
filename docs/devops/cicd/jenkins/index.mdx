# Jenkins 实战指南

「Jenkins 配置复杂、插件多、界面老——但它仍然是企业 CI/CD 的主力。」

Jenkins 是 CI/CD 领域的常青树。尽管 GitLab CI、GitHub Actions 等现代工具崛起，Jenkins 凭借其**高度可定制**、**插件生态丰富**、**Master-Agent 架构成熟**的优势，仍然是企业级 CI/CD 的首选。

理解 Jenkins，就是理解企业级持续集成的最佳实践。

## Jenkinsfile 声明式流水线

声明式流水线是 Jenkins 2.0 推出的新语法，相比传统脚本式流水线，可读性更强、维护性更好。

```groovy
pipeline {
    agent any  // 或指定 label、Docker 镜像、Kubernetes Pod

    environment {
        REGISTRY = 'harbor.example.com'
        APP_NAME = 'my-app'
    }

    stages {
        stage('Checkout') {
            steps {
                checkout scm
            }
        }

        stage('Build') {
            steps {
                sh 'mvn clean package -DskipTests'
            }
        }

        stage('Test') {
            steps {
                sh 'mvn test'
            }
            post {
                always {
                    junit 'target/surefire-reports/*.xml'
                }
            }
        }

        stage('SonarQube') {
            steps {
                withSonarQubeEnv('SonarQube') {
                    sh 'mvn sonar:sonar'
                }
            }
        }

        stage('Build Image') {
            steps {
                sh """
                    docker build -t ${REGISTRY}/${APP_NAME}:${BUILD_NUMBER} .
                    docker push ${REGISTRY}/${APP_NAME}:${BUILD_NUMBER}
                """
            }
        }

        stage('Deploy to K8s') {
            steps {
                sh """
                    kubectl set image deployment/${APP_NAME} \\
                        ${APP_NAME}=${REGISTRY}/${APP_NAME}:${BUILD_NUMBER}
                """
            }
        }
    }

    post {
        always {
            cleanWs()
        }
        success {
            echo 'Pipeline 成功！'
        }
        failure {
            echo 'Pipeline 失败了，需要排查。'
        }
    }
}
```

## Master-Agent 分布式架构

```
┌─────────────────────────────────────────────────────────────┐
│                     Jenkins Master                          │
│                                                              │
│  职责：调度、配置、存储、插件管理                              │
│  ⚠️ 不要在 Master 上运行构建！                               │
└───────────────────────────┬─────────────────────────────────┘
                            │
            ┌───────────────┼───────────────┐
            │               │               │
            ▼               ▼               ▼
    ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
    │  Agent 1     │ │  Agent 2     │ │  Agent 3     │
    │  (Linux)     │ │  (macOS)     │ │  (K8s Pod)   │
    │  Maven 构建   │ │  iOS 构建     │ │  按需创建     │
    └──────────────┘ └──────────────┘ └──────────────┘
```

**连接方式**：
- **JNLP**：Agent 通过 Java Web Start 连接 Master，穿透 NAT
- **SSH**：Master 通过 SSH 连接 Agent，简单直接
- **Kubernetes**：动态创建/销毁 Agent Pod，按需使用资源

## Jenkins 与 Kubernetes 的结合

Kubernetes Plugin 让 Jenkins 可以动态创建 Agent Pod，运行完自动销毁，资源利用率极高：

```groovy
pipeline {
    agent {
        kubernetes {
            label 'maven-agent'
            yaml '''
apiVersion: v1
kind: Pod
spec:
  containers:
    - name: maven
      image: maven:3.9-eclipse-temurin-17
      command: [sleep, infinity]
      resources:
        requests:
          memory: "1Gi"
          cpu: "500m"
        limits:
          memory: "2Gi"
          cpu: "1000m"
    - name: docker
      image: docker:latest
      command: [sleep, infinity]
      tty: true
'''
        }
    }
    stages {
        stage('Build with Maven') {
            steps {
                container('maven') {
                    sh 'mvn clean package'
                }
            }
        }
        stage('Build Docker Image') {
            steps {
                container('docker') {
                    sh 'docker build -t myapp:${BUILD_NUMBER} .'
                }
            }
        }
    }
}
```

## 面试的核心逻辑

Jenkins 面试，重点考察三个方面：

1. **架构理解**：Master-Agent 的职责分工是什么？为什么不能在 Master 上运行构建？
2. **流水线设计**：声明式流水线的语法结构是什么？如何设计一个完整的 CI/CD 流水线？
3. **问题排查**：构建失败怎么排查？Jenkins 内存占满怎么办？

```
# Jenkins 最佳实践检查清单

□ Master 和 Agent 分离（不在 Master 上跑构建）
□ 使用声明式流水线（Jenkinsfile in SCM）
□ 凭证使用 Credentials Binding（不暴露密码）
□ 构建后清理 workspace（节省磁盘空间）
□ Agent 按用途分类（不同标签对应不同构建环境）
□ 流水线共享库复用（Common Steps）
□ 定期备份 JENKINS_HOME
```

> "Jenkins 的复杂是它的优势，也是劣势。理解它的架构，才能用好它的灵活。对于企业级 CI/CD，Jenkins 仍然是不可替代的选择。"
