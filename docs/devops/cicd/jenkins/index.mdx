# Jenkins 持续集成

<Badge text="企业级 CI/CD" type="tip" />

Jenkins 诞生于 2004 年，最初叫 Hudson，由 Kohsuke Kawaguchi 在 SUN 公司内部开发。2011 年因为 Oracle 收购 SUN 后的商标争议，项目一分为二——Jenkins 继承了开源社区的热情，而 Hudson 则被 Oracle 继续维护。十几年过去了，Hudson 早已淡出历史舞台，而 Jenkins 依然是企业级 CI/CD 的代名词。

为什么 Jenkins 能持续活跃？原因很简单：**插件生态**。Jenkins 拥有超过 1800 个插件，从 Git 集成到 Docker 部署，从 SonarQube 到 Kubernetes，几乎任何 CI/CD 场景都能找到对应的插件。这种「核心极简、扩展无限」的设计哲学，让 Jenkins 成为企业定制流水线的理想选择。

## Jenkins 核心概念

**Job（任务）**：Jenkins 中的基本工作单元。一个 Job 可以是构建项目、运行测试、部署应用等任何自动化任务。

**Build（构建）**：Job 的一次执行。每次触发 Job 都会产生一个 Build，并分配递增的编号。

**Node（节点）**：执行构建的机器。可以是 Master 本身，也可以是 Slave 节点。

**Plugin（插件）**：扩展 Jenkins 功能的组件。Pipeline、Git、Docker、Kubernetes 等功能都通过插件实现。

**Workspace（工作区）**：Job 执行时的工作目录，存放源代码和构建产物。

## 目录结构

本章节将深入讲解 Jenkins 的核心知识点：

**Jenkins 架构**：从 Master-Slave 分布式架构讲起，理解 Controller 与 Agent 的角色分工，以及如何通过云代理实现动态伸缩。

**Jenkinsfile 与流水线**：Jenkins 2.0 引入的 Pipeline as Code 理念，使用 Declarative 和 Scripted 两种语法编写流水线脚本，实现流水线的版本化管理和代码审查。

**Jenkins 插件生态**：核心插件的使用方法，以及如何选择和管理插件。

**Jenkins 与 Kubernetes**：现代 Jenkins 的标配玩法——用 Kubernetes 动态创建 Agent Pod，按需分配资源，构建完成后自动销毁，实现真正的弹性构建。

**Blue Ocean**：Jenkins 的现代化 UI，专注于流水线可视化与编辑。

## 适用场景

Jenkins 适合以下场景：

- 企业级应用，需要复杂的权限管理和安全策略
- 需要高度定制化的 CI/CD 流程
- 团队有专职 DevOps 工程师维护
- 需要同时维护多个项目，且项目间有复杂的依赖关系
- 已有大量 Jenkins 基础设施，不希望迁移
