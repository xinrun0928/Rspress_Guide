# CI/CD 最佳实践

「流水线跑起来了，但构建慢、部署风险大、安全漏洞多——这些问题怎么解决？」

很多团队的 CI/CD 流水线能跑通，但跑得不够好：构建要 20 分钟、部署一次提心吊胆、镜像漏洞一堆没人管。

CI/CD 最佳实践解决的就是这些问题：**快（构建缓存）、稳（质量门禁）、安全（镜像扫描）、可控（制品管理）**。

## 构建优化的核心原则

构建慢是 CI/CD 的第一杀手。20 分钟的构建，开发者等不起，CI 服务器也等不起。

```
# 构建时间分解
┌──────────────────────────────────────────────────────────────┐
│                     构建时间 20 分钟                            │
│                                                              │
│  ┌────────────────┐  ┌────────────────┐  ┌────────────────┐  │
│  │  依赖下载        │  │  编译打包       │  │  测试执行       │  │
│  │  8 分钟          │  │  6 分钟         │  │  5 分钟         │  │
│  │  ← 可优化 →     │  │                │  │  ← 可优化 →    │  │
│  └────────────────┘  └────────────────┘  └────────────────┘  │
│                                                              │
│  ┌────────────────┐  ┌────────────────┐                      │
│  │  镜像构建       │  │  其他开销        │                      │
│  │  1 分钟         │  │  0 分钟          │                      │
│  └────────────────┘  └────────────────┘                      │
└──────────────────────────────────────────────────────────────┘

优化方向：
- 依赖下载 → 本地缓存 / 多阶段构建
- 编译打包 → 增量编译 / 并行构建
- 测试执行 → 测试分流 / 跳过不必要测试
```

## 质量门禁：不通过就不部署

质量门禁（Quality Gate）是 CI/CD 的守门员。它的作用是：**不合格的代码，不允许进入下一环节**。

```yaml
# 典型的 SonarQube 质量门禁规则
quality_gate:
  conditions:
    - metric: coverage          # 代码覆盖率 < 80% → 失败
      operator: LESS_THAN
      value: 80
    - metric: new_vulnerabilities  # 新增漏洞 > 0 → 失败
      operator: GREATER_THAN
      value: 0
    - metric: code_smells      # 新增代码异味 > 10 → 失败
      operator: GREATER_THAN
      value: 10
    - metric: duplications     # 重复代码 > 3% → 失败
      operator: GREATER_THAN
      value: 3
```

质量门禁的设置原则：
- **覆盖率门禁要合理**：不是所有项目都适合 80% 覆盖率，新项目可以先从 50% 开始
- **漏洞零容忍**：新增漏洞必须为 0，但历史漏洞可以规划修复
- **渐进式收紧**：从松到紧，给团队适应时间

## 镜像安全：别让漏洞进生产

```
# 镜像漏洞处理流程

镜像构建 → 扫描（Trivy/Clair）→ CVE 判定
                                       ↓
                              ┌─────────┴─────────┐
                              ↓                   ↓
                         严重漏洞               低危漏洞
                              ↓                   ↓
                      阻断部署              记录 + 告警
                         +                   +
                   加入技术债务           规划修复时间
```

关键原则：
- **CRITICAL/HIGH 漏洞必须阻断**：不能带着已知漏洞上线
- **使用最小基础镜像**：Alpine 比 Ubuntu 更安全
- **定期重新扫描**：新 CVE 出现后重新检测已有镜像

## 面试的核心逻辑

CI/CD 最佳实践的面试，重点考察三个方面：

1. **性能意识**：构建慢的原因是什么？怎么优化？构建缓存的原理是什么？
2. **质量思维**：质量门禁应该设置哪些指标？覆盖率越高越好吗？
3. **安全意识**：镜像漏洞怎么处理？CVE 那么多，怎么排优先级？

```
# CI/CD 最佳实践检查清单

□ 构建时间 < 10 分钟（通过缓存和并行）
□ 测试覆盖率 > 80%（核心模块）
□ 新增漏洞为 0（阻断部署）
□ 制品版本可追溯（Commit ID + Build ID）
□ 回滚方案就绪（金丝雀/蓝绿/版本回滚）
□ 多环境配置分离（dev/staging/prod）
```

> "好的 CI/CD 流水线，应该让开发者『提交代码后不用管』——自动构建、自动测试、自动部署、自动告警。人应该从重复劳动中解放出来，做更有价值的事情。"
