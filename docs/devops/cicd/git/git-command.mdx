大多数开发者每天都在用 `git add`、`git commit`、`git push` 这三板斧，Git 的日常操作已经刻进肌肉记忆。但如果问起 `git rebase` 和 `git merge` 的本质区别，很多人会含糊其辞——「rebase 会把你的提交历史变线性」「merge 不会丢失历史」——这些说法对，但不够准确。

更关键的问题是：什么时候该用 merge？什么时候该用 rebase？如果你在团队中用错了，可能会导致提交历史混乱，甚至丢失代码。

## git merge vs git rebase：核心区别

### merge：保留历史，合并结果

`git merge` 将两个分支的历史合并在一起，创建一个新的「合并提交」。这个合并提交有两个父提交，分别指向合并前的两个分支。

```bash
# 场景：feature 分支已经开发完毕，需要合并回 main
git checkout main
git merge feature/payment
```

**merge 的特点**：

- 保留完整的分支历史，包括所有提交
- 不会修改已有提交
- 产生一个合并提交，清晰标记合并点
- 如果有冲突，每个文件只需解决一次

**merge 的类型**：

```bash
# 1. 快进合并（Fast-Forward）
# 当 main 没有新提交时，Git 会直接移动指针
git merge feature/payment
# 结果：main 指针直接移动到 feature/payment 的最新提交

# 2. 非快进合并（No Fast-Forward）
# 保留分支历史，创建合并提交
git merge --no-ff feature/payment

# 3. 压合合并（Squash）
# 将所有提交压缩成一个提交
git merge --squash feature/payment
git commit -m "Implement payment feature"
```

### rebase：改写历史，重放提交

`git rebase` 将当前分支的提交「重放」到目标分支的最新提交之上。从结果上看，它创造了一条线性的提交历史——仿佛所有提交都是按顺序进行的。

```bash
# 场景：feature 分支从 main 分支分出后，main 有新提交
git checkout feature/payment

# 将 feature/payment 的提交 rebase 到 main 最新提交之上
git rebase main
```

**rebase 的本质**：

```
Before rebase:

main:    A --- B --- C
                  \
feature:           D --- E

After rebase (git rebase main):

main:    A --- B --- C
                  \
feature:           D' --- E'  (D' 和 E' 是重新创建的提交，hash 不同)
```

> [!WARNING]
> **绝对不要 rebase 已经推送到远程的提交**。这会改写提交历史，如果其他人基于这些提交工作，会导致严重的问题。

## 何时用 merge？何时用 rebase？

这是一个经典的 Git 决策问题，核心在于：**你想要什么样的提交历史？**

**用 merge 当**：

- 合并 `feature/*` 到 `develop` 或 `release/*` 到 `main` 时
- 需要保留完整的分支历史时
- 合并不常发生的重大变更时
- 团队要求保留所有提交记录作为审计依据时

**用 rebase 当**：

- 将 `main` 的最新变更同步到功能分支时
- 整理本地提交，准备发起 Pull Request 时
- 想要保持线性、干净的提交历史时
- 功能分支开发时间较短，rebase 不会产生太多冲突时

**实际工作中的推荐策略**：

```bash
# 1. 日常同步：从 main 拉取最新变更到功能分支
git checkout feature/my-feature
git fetch origin
git rebase origin/main
# 解决可能的冲突，然后继续
git rebase --continue

# 2. 合并完成：发起 Pull Request 时
# GitHub/GitLab 会自动处理 merge
# 团队可以选择 squash merge 保持历史整洁

# 3. 发布前：将 release 分支合并回 main
git checkout main
git merge --no-ff release/1.0.0 -m "Release version 1.0.0"
git tag -a v1.0.0 -m "Version 1.0.0"
```

## git cherry-pick：选择性应用提交

`cherry-pick` 允许你选择性地将某个或某些提交应用到当前分支。这在你需要将 bug 修复应用到多个分支时特别有用。

**基本用法**：

```bash
# 将指定提交应用到当前分支
git cherry-pick <commit-hash>

# 一次性 cherry-pick 多个提交
git cherry-pick <commit-1> <commit-2> <commit-3>

# 连续范围的提交（不包括 start）
git cherry-pick start..end
```

**实际场景**：

```bash
# 场景：hotfix 分支的修复需要同步到 develop
git checkout hotfix/critical-auth-bug
# 提交是 abc123

git checkout develop
git cherry-pick abc123
# develop 现在也有了修复

# 场景：需要将最近 3 个提交 cherry-pick 到另一个分支
git log --oneline -3
# abc123 fix: resolve auth timeout
# def456 fix: handle null user
# 789ghi docs: update README

git checkout feature/other-work
git cherry-pick abc123..789ghi
```

**cherry-pick 的注意事项**：

- cherry-pick 会创建新的提交（hash 不同），因为提交时间等信息变了
- 如果有冲突，需要解决冲突后 `git add` 并 `git cherry-pick --continue`
- 可以用 `git cherry-pick --no-commit <commit>` 只应用变更，不自动提交

## git stash：临时存储工作

当你正在一个分支上开发，突然需要切换到另一个分支处理紧急问题，但当前分支的修改又不想提交时，`git stash` 就是救命稻草。

**基本用法**：

```bash
# 保存当前工作目录和暂存区的修改
git stash

# 查看 stash 列表
git stash list
# 输出：stash@{0}: WIP on feature/payment: abc123...

# 恢复最近的 stash
git stash pop

# 恢复指定的 stash
git stash apply stash@{0}

# 删除 stash
git stash drop stash@{0}

# 恢复并删除
git stash pop
```

**高级用法**：

```bash
# 1. 同时 stash 未跟踪的文件
git stash -u

# 2. 包含忽略的文件（慎用）
git stash -a

# 3. 带消息的 stash，方便记忆
git stash push -m "WIP: payment module, half done"

# 4. 从 stash 创建分支
git stash branch new-branch stash@{0}
# 自动创建新分支，恢复 stash 内容，并删除该 stash

# 5. 只 stash 部分文件
git stash push -m "partial work" src/main.js src/utils.js
```

**典型使用场景**：

```bash
# 场景：正在开发新功能，突然需要修复 bug
git status
# On branch: feature/new-ui
# Changes not staged for commit:
#   modified: src/App.tsx
#   modified: src/components/Button.tsx

# 临时存储当前修改
git stash

# 切换到 bugfix 分支处理紧急问题
git checkout hotfix/urgent-bug
# ... 修复 bug，提交，测试，合并 ...

# 回到原分支，恢复工作
git checkout feature/new-ui
git stash pop
```

## git reset vs git revert：撤销变更

这是两个容易混淆的命令，它们的效果不同，适用场景也不同。

### git reset：回退到某个状态

`reset` 将 HEAD 指针移动到指定位置，**会修改提交历史**。

```bash
# 1. 软重置：保留修改在暂存区
git reset --soft HEAD~1
# 结果：上一个提交被撤销，修改在暂存区

# 2. 混合重置（默认）：保留修改在工作区
git reset HEAD~1
# 结果：上一个提交被撤销，修改在工作区

# 3. 硬重置：丢弃所有修改（危险！）
git reset --hard HEAD~1
# 结果：上一个提交被彻底删除，所有修改丢失
```

**使用场景**：

- 撤销本地未推送的提交
- 整理提交历史（比如 squash 多个提交）
- 取消 merge 操作

```bash
# 场景：提交信息写错了，想重写
git commit -m "fixd typo"  # 提交后发现有错别字
git reset --soft HEAD~1     # 撤销提交，保留修改
git commit -m "fix typo"    # 重新提交
```

### git revert：创建反向提交

`revert` 不会修改历史，而是创建一个新的提交，这个新提交的内容恰好是指定提交的「反」——如果原提交是添加了一行代码，revert 就会删除这一行代码。

```bash
# 撤销指定的提交
git revert <commit-hash>

# 撤销 merge 提交（需要指定主父分支）
git revert -m 1 <merge-commit-hash>
```

**使用场景**：

- 撤销已经推送到远程的提交
- 撤销 merge 操作
- 需要保留完整历史记录的撤销

```bash
# 场景：发现一个已经合并到 main 的提交引入了 bug
git log --oneline
# abc123 (HEAD -> main) Merge feature/payment
# def456 Add payment module
# ...

# 创建反向提交撤销 payment 模块
git revert abc123
# Git 会打开编辑器，让你输入 revert 提交的信息
# 保存后，新的提交会撤销之前合并的内容
```

**对比总结**：

| 场景 | 用 reset | 用 revert |
|---|---|---|
| 撤销本地未推送的提交 | `reset --hard` | 不适用 |
| 撤销已推送的提交 | 不适用 | `revert` |
| 撤销 merge | 需要指定父分支 | 需要指定父分支 |
| 想保持线性历史 | `rebase -i` | `revert` |

## git log 与 git bisect：调试利器

### git log：查看提交历史

```bash
# 基础用法
git log
git log --oneline           # 简洁格式
git log --graph             # 图形化显示分支
git log -n 5                # 最近 5 条

# 筛选提交
git log --author="zhang"    # 按作者筛选
git log --since="2024-01-01"  # 按时间筛选
git log --grep="fix:"       # 按提交信息筛选
git log -- src/main.js      # 只看某个文件的提交

# 美观的单行显示
git log --oneline --graph --all --decorate
```

### git bisect：二分查找定位 bug

`bisect` 是 Git 最强大的调试工具之一，它通过二分查找快速定位引入 bug 的提交。

```bash
# 1. 开始二分查找
git bisect start

# 2. 标记当前版本是坏的（有 bug）
git bisect bad

# 3. 标记一个已知好的版本
git bisect good v1.0.0

# 4. Git 会自动 checkout 到中间版本
# 你测试后告诉 Git 这个版本是好是坏
git bisect good   # 或 git bisect bad

# 5. Git 继续二分，直到找到第一个坏的提交
# 重复上述步骤，直到 Git 输出结果

# 6. 结束 bisect
git bisect reset
```

**自动化 bisect**：

```bash
# 使用脚本自动化测试
git bisect start HEAD v1.0.0
git bisect run npm test

# bisect run 会自动运行测试脚本
# 脚本返回 0 表示 good，非 0 表示 bad
# 最终输出第一个引入 bug 的提交
```

## 常见工作流总结

```bash
# 1. 开始新功能
git checkout main
git pull
git checkout -b feature/my-feature

# 2. 开发中：频繁同步 main 的变更
git fetch origin
git rebase origin/main

# 3. 提交代码（建议用 Conventional Commits）
git add .
git commit -m "feat(payment): add Stripe integration"

# 4. 推送发起 PR
git push -u origin feature/my-feature

# 5. PR 合并后，清理分支
git checkout main
git pull
git branch -d feature/my-feature

# 6. 紧急修复
git checkout -b hotfix/urgent-fix main
# 修复并测试后
git checkout main
git merge --no-ff hotfix/urgent-fix
git tag -a v1.0.1 -m "Hotfix"
git push origin main --tags
```

> [!TIP]
> 把这些常用命令组合成 alias，可以大大提高工作效率。
