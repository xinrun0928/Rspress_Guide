# Git 提交规范

<Badge text="PRINCIPLE 原理类" type="warning" />

```
commit a1b2c3d4
Date:   Mon Jan 15 14:32:11 2024 +0800

    fix bug
```

你能从这个提交信息中得到什么？不知道。这个「bug」是什么？在哪个文件？为什么会出现？通通不知道。

三年后当你翻看 git log，想找到那个改掉某个 bug 的提交时，你会对着满屏的「fix」「update」「modify」欲哭无泪。

commit message 不是写给自己看的，是写给**未来的团队成员**——包括未来的你。好的提交规范，让代码历史成为一本可阅读的故事书；坏的提交规范，让 git log 成为一堆无意义的哈希值堆砌。

## Conventional Commits 规范

[Conventional Commits](https://www.conventionalcommits.org/) 是一种社区约定的提交信息格式，它通过统一的结构让提交信息自描述、可解析、可自动化处理。

### 格式定义

```
<type>[optional scope]: <description>

[optional body]

[optional footer(s)]
```

### 组成部分

| 部分 | 说明 | 示例 |
|---|---|---|
| `type` | 提交类型，必填 | `feat`、`fix`、`docs` |
| `scope` | 影响范围，可选 | `auth`、`payment`、`api` |
| `:` | 分隔符，必填 | `:` |
| `description` | 简短描述，必填 | `add user login endpoint` |
| `body` | 详细说明，可选 | 解释为什么这样改 |
| `footer` | 脚注，可选 | 关联的 Issue 编号 |

### Type 类型详解

| 类型 | 说明 | 是否记入 CHANGELOG |
|---|---|---|
| `feat` | 新功能 | 是 |
| `fix` | bug 修复 | 是 |
| `docs` | 文档变更 | 否 |
| `style` | 代码格式（不影响功能） | 否 |
| `refactor` | 重构（不是新功能也不是修复） | 否 |
| `perf` | 性能优化 | 是 |
| `test` | 添加或修改测试 | 否 |
| `build` | 构建系统或依赖变更 | 否 |
| `ci` | CI 配置变更 | 否 |
| `chore` | 其他杂项 | 否 |
| `revert` | 回滚之前的提交 | 是 |

## 实际提交示例

### 基本提交

```bash
# 新功能
git commit -m "feat(auth): add JWT token refresh endpoint"

# Bug 修复
git commit -m "fix(payment): handle null response from Stripe API"

# 文档更新
git commit -m "docs: update API documentation for user endpoint"

# 重构
git commit -m "refactor(auth): extract token validation to separate module"

# 性能优化
git commit -m "perf(api): cache user profile to reduce database queries"
```

### 包含详细说明的提交

```bash
git commit -m "fix(auth): resolve session timeout on mobile devices

The session was expiring prematurely on mobile due to incorrect
timestamp comparison when checking token validity.

The root cause was that the client was sending the expiry time as
a string instead of a Unix timestamp, causing comparison failures.

Closes #1234"
```

### Breaking Change

```bash
git commit -m "feat(payment): migrate from Stripe to Square API

BREAKING CHANGE: The payment response format has changed.
Clients need to update their integration to handle the new response schema.

Before: { status: 'success', amount: 100 }
After:  { success: true, total: 100 }

Migration guide: /docs/migration/payment-v2.md"
```

## Commitizen：交互式提交

Commitizen 是一个交互式工具，可以引导你按照 Conventional Commits 规范填写提交信息。

### 安装与使用

```bash
# 全局安装 Commitizen
npm install -g commitizen

# 初始化项目使用 cz-conventional-changelog
npx commitizen init cz-conventional-changelog --save-dev

# 使用交互式提交
npx git-cz
```

### 交互流程

```bash
$ npx git-cz

# ? Select the type of change that you're committing: (Use arrow keys)
#   feat:     A new feature
#   fix:      A bug fix
#   > docs:    Documentation only changes
#   style:    Changes that do not affect the meaning of the code
#   refactor: A code change that neither fixes a bug nor adds a feature
#   perf:     A code change that improves performance
#   test:     Adding missing tests or correcting existing tests
#   build:    Changes that affect the build system or external dependencies
#   ci:       Changes to our CI configuration files and scripts
#   chore:    Other changes that don't modify src or test files

# ? What is the scope of this change? (e.g. component or file name)
#   auth

# ? Write a short, imperative tense description of the change:
#   add password reset via email

# ? Provide a longer description of the change:
#   (press enter to skip)
#   Implemented password reset flow with time-limited tokens

# ? Are there any breaking changes? (y/N)
#   N

# ? Does this change affect any open issues? (y/N)
#   y

# ? Add issue references (e.g. "fix #123", "re #123"):
#   close #456
```

### 输出结果

```
feat(auth): add password reset via email

Implemented password reset flow with time-limited tokens

Closes #456
```

## Commitlint：强制提交规范

Commitlint 是一个工具，用于检查提交信息是否符合规范。它通常配合 Husky 使用，在提交前自动检查。

### 安装

```bash
npm install --save-dev @commitlint/config-conventional @commitlint/cli
```

### 配置

```javascript
// commitlint.config.js
module.exports = {
  extends: ['@commitlint/config-conventional'],
  rules: {
    'type-enum': [
      2,
      'always',
      ['feat', 'fix', 'docs', 'style', 'refactor', 'perf', 'test', 'build', 'ci', 'chore', 'revert']
    ],
    'type-case': [2, 'always', 'lower-case'],
    'type-empty': [2, 'never'],
    'subject-empty': [2, 'never'],
    'subject-full-stop': [2, 'never', '.'],
    'header-max-length': [2, 'always', 72]
  }
};
```

### 集成 Husky

```bash
# 安装 Husky
npm install --save-dev husky
npx husky install

# 添加 commitlint hook
npx husky add .husky/commit-msg 'npx --no -- commitlint --edit ${1}'
```

### 自动修复

```bash
# 安装 @commitlint/cz-commitlint
npm install --save-dev @commitlint/cz-commitlint

# 配置 commitlint.config.js
module.exports = {
  extends: ['@commitlint/config-conventional'],
  prompt: {
    settings: {
      czConfig: {
        types: [
          { value: 'feat', name: 'feat:     A new feature', emoji: '✨' },
          { value: 'fix', name: 'fix:      A bug fix', emoji: '🐛' },
          { value: 'docs', name: 'docs:     Documentation only', emoji: '📝' },
          { value: 'style', name: 'style:    Code formatting', emoji: '💄' },
          { value: 'refactor', name: 'refactor: Code refactoring', emoji: '♻️' },
          { value: 'perf', name: 'perf:     Performance improvements', emoji: '⚡' },
          { value: 'test', name: 'test:     Adding tests', emoji: '✅' },
          { value: 'build', name: 'build:    Build system changes', emoji: '📦' },
          { value: 'ci', name: 'ci:       CI configuration', emoji: '👷' },
          { value: 'chore', name: 'chore:    Other changes', emoji: '🔧' }
        ]
      }
    }
  }
};
```

## 自动生成 CHANGELOG

Conventional Commits 的一个重要优势是可以根据提交类型自动生成 CHANGELOG。

### 使用 standard-version

```bash
# 安装
npm install --save-dev standard-version

# 更新版本并生成 CHANGELOG
npx standard-version --dry-run  # 预览

npx standard-version             # 正式执行
```

### 生成效果

```markdown
# Changelog

## [2.0.0](https://github.com/example/project/compare/v1.0.0...v2.0.0) (2024-01-15)

### ⚠️ BREAKING CHANGES

- **payment**: Migrate from Stripe to Square API

### Features

- **auth**: Add password reset via email ([#456](https://github.com/example/project/issues/456))
- **api**: Add batch processing endpoint ([#789](https://github.com/example/project/issues/789))

### Bug Fixes

- **payment**: Handle null response from Stripe API ([#123](https://github.com/example/project/issues/123))

### Performance Improvements

- **api**: Cache user profile to reduce database queries

### Documentation

- Update API documentation for user endpoint
```

### 自定义 CHANGELOG 生成

```javascript
// release.config.js
module.exports = {
  releaseAs: 'minor',  // 或 'major', 'minor', 'patch'
 dryRun: false,
  infile: 'CHANGELOG.md',
  message: 'chore(release): ${version}',
  types: [
    { type: 'feat', section: 'Features', hidden: false },
    { type: 'fix', section: 'Bug Fixes', hidden: false },
    { type: 'perf', section: 'Performance', hidden: false },
    { type: 'refactor', section: 'Refactoring', hidden: true },
    { type: 'docs', section: 'Documentation', hidden: true },
    { type: 'test', section: 'Tests', hidden: true },
    { type: 'build', section: 'Build System', hidden: true },
    { type: 'ci', section: 'CI', hidden: true },
    { type: 'chore', section: 'Chores', hidden: true }
  ]
};
```

## 团队实践建议

### 提交频率

**好的实践**：

- 每个功能点完成时提交一次
- 每个 bug 修复后提交一次
- 提交信息要能描述「做了什么」，而不是「正在做什么」

**不好的实践**：

```bash
# 一天结束时的「批量提交」
git add .
git commit -m "wip"

# 没有任何意义的提交信息
git commit -m "asdf"
git commit -m "update"
git commit -m "zxcv"

# 提交信息与代码内容不符
git commit -m "fix bug"  # 实际上改的是样式
```

### 原子提交原则

一个提交应该只包含一件事。

```bash
# 不好的做法：在一个提交中做了太多事
git commit -m "feat: add user profile, fix login bug, update dependencies"

# 好的做法：拆分成多个原子提交
git commit -m "feat(profile): add user profile page"
git commit -m "fix(auth): resolve session timeout issue"
git commit -m "chore(deps): update axios to v1.6"
```

### 提交前的检查

```bash
# 提交前检查
git status        # 确认修改了哪些文件
git diff --cached # 确认暂存的内容

# 检查提交信息
npx commitlint --from HEAD~1  # 检查最近一次提交
```

## 规范化效果

采用 Conventional Commits 后，你的 git log 将变得清晰可读：

```
feat(auth): implement two-factor authentication
feat(payment): add Stripe webhook handler
feat(api): add batch export endpoint
fix(auth): prevent session hijacking on mobile
fix(payment): handle Stripe timeout gracefully
docs: update API rate limiting documentation
refactor(auth): extract token validation to middleware
perf(api): implement response caching
test(payment): add Stripe mock tests
chore(deps): upgrade dependencies
```

> [!TIP]
> 建议将 Conventional Commits 规范纳入团队代码规范文档，并在 CI 流水线中加入 commitlint 检查，从根本上保证提交信息的质量。
