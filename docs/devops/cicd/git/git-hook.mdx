# Git Hooks

<Badge text="自动化质量门禁" type="tip" />

代码审查发现了一个 ESLint 错误，你叹了口气，点击「Request changes」。「等等，这个错误本应该在提交前就被发现」，你想，「浪费了大家的时间」。

Git Hooks 就是解决这个问题的——在代码真正进入仓库之前，让它先过五关斩六将。代码格式不对？提交被拒绝。测试没通过？提交被拒绝。没有遵循 commit 规范？提交被拒绝。

让错误在源头被发现，而不是在 Code Review 中被揪出来。

## Git Hooks 概述

Git Hooks 是 Git 提供的一种扩展机制，允许你在特定的 Git 操作前后自动执行自定义脚本。这些脚本存放在 `.git/hooks` 目录下，以事件命名。

### 工作原理

```bash
# .git/hooks 目录下的示例
.git/hooks/
├── applypatch-msg.sample
├── commit-msg.sample
├── post-applypatch
├── post-commit
├── post-receive      # 服务器端钩子
├── pre-applypatch
├── pre-commit        # 提交前执行
├── pre-push          # 推送前执行
├── pre-rebase
├── prepare-commit-msg.sample
└── update.sample
```

> [!WARNING]
> `.git/hooks` 目录中的示例文件以 `.sample` 结尾，不会自动执行。需要去掉 `.sample` 后缀并添加执行权限才能生效。

### 客户端钩子 vs 服务端钩子

| 类型 | 执行位置 | 示例 | 可否被跳过 |
|---|---|---|---|
| 客户端钩子 | 本地 | `pre-commit`、`commit-msg` | 可以（`--no-verify`） |
| 服务端钩子 | Git 服务器 | `pre-receive`、`post-receive` | 不可以 |

## 核心钩子详解

### pre-commit：提交前的第一道关卡

`pre-commit` 是最常用的钩子，在 `git commit` 执行之前运行。如果脚本返回非零状态，提交将被拒绝。

**典型应用：运行 ESLint**

```bash
#!/bin/bash
# .git/hooks/pre-commit

# 捕获任何 ESLint 错误
echo "Running ESLint..."
npx eslint --max-warnings=0 .

if [ $? -ne 0 ]; then
  echo "ESLint failed. Please fix the errors above."
  exit 1
fi

echo "ESLint passed!"
exit 0
```

**典型应用：运行单元测试**

```bash
#!/bin/bash
# .git/hooks/pre-commit

echo "Running unit tests..."
npm test

if [ $? -ne 0 ]; then
  echo "Tests failed. Commit aborted."
  exit 1
fi

exit 0
```

**典型应用：禁止提交大文件**

```bash
#!/bin/bash
# .git/hooks/pre-commit

# 检查是否有超过 5MB 的文件
MAX_SIZE=5242880  # 5MB in bytes
files=$(git diff --cached --name-only)

for file in $files; do
  size=$(stat -f%z "$file" 2>/dev/null || stat -c%s "$file" 2>/dev/null)
  if [ "$size" -gt "$MAX_SIZE" ]; then
    echo "Error: $file is larger than 5MB. Please use git lfs for large files."
    exit 1
  fi
done

exit 0
```

### prepare-commit-msg：自定义提交信息

`prepare-commit-msg` 在提交信息编辑器打开之前执行，可以修改提交信息的默认模板。

**自动添加 Issue 编号**

```bash
#!/bin/bash
# .git/hooks/prepare-commit-msg

COMMIT_MSG_FILE=$1
COMMIT_SOURCE=$2
SHA1=$3

# 如果是从 squash 或 merge 产生的提交，不做处理
if [ -n "$COMMIT_SOURCE" ]; then
  exit 0
fi

# 从分支名提取 Issue 编号
BRANCH_NAME=$(git symbolic-ref --short HEAD)
ISSUE_PATTERN="([A-Z]+-[0-9]+)"
if [[ $BRANCH_NAME =~ $ISSUE_PATTERN ]]; then
  ISSUE_NUM="${BASH_REMATCH[1]}"
  
  # 在第一行添加 Issue 引用
  sed -i.bak "1s/^/[$ISSUE_NUM] /" "$COMMIT_MSG_FILE"
  rm "$COMMIT_MSG_FILE.bak"
fi
```

### commit-msg：验证提交规范

`commit-msg` 在提交信息写入后、执行提交前运行，常用于验证提交信息是否符合规范。

**验证 Conventional Commits 格式**

```bash
#!/bin/bash
# .git/hooks/commit-msg

COMMIT_MSG=$(cat "$1")
COMMIT_PATTERN="^(feat|fix|docs|style|refactor|perf|test|build|ci|chore|revert)(\(.+\))?: .{1,50}"

if ! [[ $COMMIT_MSG =~ $COMMIT_PATTERN ]]; then
  echo "Error: Commit message does not follow Conventional Commits format."
  echo "Expected: <type>(<scope>): <description>"
  echo "Types: feat, fix, docs, style, refactor, perf, test, build, ci, chore, revert"
  exit 1
fi

exit 0
```

**自动添加签名**

```bash
#!/bin/bash
# .git/hooks/commit-msg

COMMIT_MSG_FILE=$1

# 检查是否有 Signed-off-by
if ! grep -q "Signed-off-by:" "$COMMIT_MSG_FILE"; then
  echo "" >> "$COMMIT_MSG_FILE"
  echo "Signed-off-by: $(git config user.name) <$(git config user.email)>" >> "$COMMIT_MSG_FILE"
fi
```

### pre-push：推送前的最后检查

`pre-push` 在 `git push` 执行之前运行，适用于需要验证分支策略或运行额外测试的场景。

**禁止推送到 main 分支**

```bash
#!/bin/bash
# .git/hooks/pre-push

BRANCH=$(git symbolic-ref --short HEAD)
PROTECTED_BRANCHES="^(main|master|develop)$"

if [[ $BRANCH =~ $PROTECTED_BRANCHES ]]; then
  echo "Error: Pushing directly to $BRANCH is not allowed."
  echo "Please create a feature branch and submit a pull request."
  exit 1
fi

exit 0
```

**推送前运行集成测试**

```bash
#!/bin/bash
# .git/hooks/pre-push

echo "Running pre-push checks..."

# 只在推送 feature 分支时运行额外测试
BRANCH=$(git symbolic-ref --short HEAD)
if [[ $BRANCH == feature/* ]]; then
  echo "Running integration tests..."
  npm run test:integration
  
  if [ $? -ne 0 ]; then
    echo "Integration tests failed."
    exit 1
  fi
fi

exit 0
```

### post-commit：提交完成后的通知

`post-commit` 在提交完成后执行，常用于通知、备份等场景。

**提交后通知 Slack**

```bash
#!/bin/bash
# .git/hooks/post-commit

# 获取提交信息
COMMIT_MSG=$(git log -1 --pretty=%B)
AUTHOR=$(git log -1 --pretty=%an)
BRANCH=$(git symbolic-ref --short HEAD)

# 发送到 Slack（需要配置 Webhook URL）
curl -X POST \
  -H 'Content-Type: application/json' \
  -d "{\"text\":\"$AUTHOR committed to $BRANCH: $COMMIT_MSG\"}" \
  https://hooks.slack.com/services/YOUR/WEBHOOK/URL

exit 0
```

## Husky：管理 Git Hooks 的现代方案

手动创建和管理 `.git/hooks` 脚本有以下问题：脚本存储在 `.git` 目录，无法提交到版本控制，团队成员需要各自配置。

Husky 解决了这些问题，它将 Hooks 配置放在项目根目录，通过 npm 包管理。

### 安装与配置

```bash
# 安装 Husky
npm install --save-dev husky

# 初始化
npx husky install

# 添加钩子
npx husky add .husky/pre-commit "npm test"
```

### package.json 配置

```json
{
  "husky": {
    "hooks": {
      "pre-commit": "npm test",
      "commit-msg": "npx commitlint --edit",
      "pre-push": "npm run lint && npm run test:integration"
    }
  }
}
```

### 实际配置示例

```bash
# 1. 初始化
npx husky install

# 2. 添加 pre-commit
npx husky add .husky/pre-commit "npx lint-staged"

# 3. 添加 commit-msg
npx husky add .husky/commit-msg "npx commitlint --edit $1"

# 4. 添加 pre-push
npx husky add .husky/pre-push "npm run type-check && npm run test:unit"
```

### lint-staged：只检查暂存的文件

只对即将提交的文件运行检查，可以大大提高 pre-commit 的执行速度。

```bash
# 安装
npm install --save-dev lint-staged
```

```json
// package.json
{
  "lint-staged": {
    "*.{js,jsx,ts,tsx}": [
      "eslint --fix",
      "prettier --write",
      "jest --bail --findRelatedTests"
    ],
    "*.{css,scss}": [
      "stylelint --fix",
      "prettier --write"
    ],
    "*.md": [
      "markdownlint",
      "prettier --write"
    ]
  }
}
```

## 服务端钩子

### pre-receive：强制推送策略

在 Git 服务器上执行，用于强制执行仓库策略。

```bash
#!/bin/bash
# 服务端 pre-receive 钩子

while read oldrev newrev refname; do
  branch=$(echo $refname | cut -d/ -f3)
  
  # 禁止强制推送 main 分支
  if [[ $branch == "main" ]] && [[ "$newrev" != "0000000000000000000000000000000000000000" ]]; then
    echo "Force push to main is not allowed."
    exit 1
  fi
  
  # 检查提交信息格式
  if [[ "$newrev" != "0000000000000000000000000000000000000000" ]]; then
    commits=$(git rev-list $oldrev..$newrev)
    for commit in $commits; do
      msg=$(git log -1 --format=%B $commit)
      if ! echo "$msg" | grep -qE "^[A-Z]+-[0-9]+:"; then
        echo "Commit $commit does not have a ticket number prefix."
        exit 1
      fi
    done
  fi
done

exit 0
```

### post-receive：自动部署

```bash
#!/bin/bash
# 服务端 post-receive 钩子 - 自动部署

GIT_DIR="/path/to/repo.git"
DEPLOY_DIR="/var/www/app"
BRANCH="main"

if [ "$GIT_REF" == "refs/heads/$BRANCH" ]; then
  echo "Deploying to production..."
  cd $DEPLOY_DIR
  git pull origin $BRANCH
  npm install --production
  npm run build
  systemctl restart app
  echo "Deployment complete!"
fi
```

## 最佳实践

### 1. 使用 Husky 管理本地钩子

```bash
# 安装依赖
npm install --save-dev husky lint-staged @commitlint/cli @commitlint/config-conventional

# 配置 commitlint
echo 'module.exports = { extends: ["@commitlint/config-conventional"] };' > commitlint.config.js

# 初始化 husky
npx husky install

# 添加钩子
npx husky add .husky/pre-commit "npx lint-staged"
npx husky add .husky/commit-msg "npx commitlint --edit $1"
```

### 2. 保持钩子脚本快速

```bash
# 不好：运行所有测试
#!/bin/bash
npm test  # 可能需要几分钟

# 好：只运行 lint
#!/bin/bash
npm run lint  # 几秒钟
```

### 3. 提供清晰的错误信息

```bash
# 不好
if [ $? -ne 0 ]; then
  exit 1
fi

# 好
if [ $? -ne 0 ]; then
  echo ""
  echo "❌ ESLint failed. Please fix the errors above."
  echo ""
  echo "💡 Tip: Run 'npm run lint:fix' to auto-fix some issues."
  exit 1
fi
```

### 4. 允许跳过检查（谨慎使用）

```bash
# pre-commit 可以用 --no-verify 跳过
git commit --no-verify -m "Emergency fix"

# 但要在日志中记录
git log --oneline -1 --format="%s (skipped hooks)" | git commit --amend --no-edit
```

## 常见问题

### 问：钩子不生效怎么办？

```bash
# 1. 检查文件是否存在
ls -la .husky/pre-commit

# 2. 检查是否有执行权限
chmod +x .husky/pre-commit

# 3. 检查脚本是否有 shebang
head -1 .husky/pre-commit  # 应该是 #!/bin/bash
```

### 问：Windows 环境下钩子不工作？

```bash
# Windows 用户需要确保 husky 使用正确的脚本
npx husky add .husky/pre-commit "npx lint-staged"

# 或者手动创建脚本（Git Bash 或 WSL）
```

> [!TIP]
> Git Hooks 是 CI/CD 流水线的重要补充——它在代码进入版本控制系统之前就开始质量把控，让错误无处遁形。
