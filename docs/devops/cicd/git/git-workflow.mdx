# Git 工作流选择

<Badge text=" PRINCIPLE 原理类" type="warning" />

凌晨 2 点，你被拉进了一个紧急会议。生产环境出了严重 bug，罪魁祸首是一条刚刚合并到主分支的代码。调查发现，这条代码通过了 CI 流水线的所有测试，却没有经过充分的集成验证就上线了。

问题的根源不在代码本身，而在于**工作流设计**。如果团队采用了更严格的分支策略，如果主分支有代码freeze 机制，如果上线前必须经过 staging 环境验证——这场凌晨的「救火」本可以避免。

Git 工作流不是教条，而是团队协作的「交通规则」。选对了工作流，团队高效运转；选错了，要么拥堵不堪，要么事故频发。

## 三种主流工作流

### Git Flow：功能分支的经典范式

Git Flow 由 Vincent Driessen 在 2010 年提出，是目前使用最广泛的分支策略。它的核心是将代码生命周期划分为五种分支：

| 分支类型 | 命名规范 | 生命周期 | 职责 |
|---|---|---|---|
| `master`/`main` | 固定 | 永久 | 正式发布的版本，只接受来自 release 的合并 |
| `develop` | 固定 | 永久 | 开发主分支，汇集所有功能分支 |
| `feature/*` | `feature/` 前缀 | 功能开发期间 | 单个功能的开发分支 |
| `release/*` | `release/` 前缀 | 发布准备期间 | 准备发布版本的分支 |
| `hotfix/*` | `hotfix/` 前缀 | 修复期间 | 紧急修复生产问题的分支 |

**工作流程**：

```bash
# 1. 从 develop 创建功能分支
git checkout develop
git checkout -b feature/user-authentication

# 2. 开发完成后，合并到 develop
git checkout develop
git merge --no-ff feature/user-authentication

# 3. 准备发布时，从 develop 创建 release 分支
git checkout develop
git checkout -b release/1.0.0

# 4. 发布完成，合并回 master 和 develop
git checkout master
git merge --no-ff release/1.0.0
git tag -a v1.0.0 -m "Release version 1.0.0"

git checkout develop
git merge --no-ff release/1.0.0

# 5. 紧急修复，从 master 创建 hotfix 分支
git checkout master
git checkout -b hotfix/critical-bug

# 6. 修复完成后，合并回 master 和 develop
git checkout master
git merge --no-ff hotfix/critical-bug
git tag -a v1.0.1 -m "Hotfix version 1.0.1"

git checkout develop
git merge --no-ff hotfix/critical-bug
```

**适用场景**：

- 有明确发布周期（如每两周一个 sprint）的团队
- 需要同时维护多个生产版本的场景
- 团队规模较大，需要严格的流程控制

**缺点**：

- 分支管理复杂，对于小团队来说过于繁琐
- `develop` 分支长时间存在可能导致集成问题延迟暴露
- 不适合持续交付的节奏

---

### GitHub Flow：简化的 PR 驱动

GitHub Flow 由 GitHub 官方推荐，它极度简化了分支模型，核心思想是：**任何代码必须通过 Pull Request 合并**，主分支始终处于可部署状态。

**只有一个长期分支**：只有 `main`（或 `master`）是永久分支，所有功能都在短期分支上开发。

**工作流程**：

```bash
# 1. 从 main 创建功能分支
git checkout main
git checkout -b feature/dark-mode

# 2. 开发和提交
git add .
git commit -m "Add dark mode toggle component"

# 3. 推送并创建 Pull Request
git push -u origin feature/dark-mode

# 4. Code Review 通过后，合并 Pull Request
# 通过 GitHub UI 点击 "Merge" 按钮
```

**核心规则**：

- `main` 分支始终可部署
- 所有变更通过 Pull Request 进行
- 必须通过 CI 流水线的所有测试才能合并
- 必须有至少一人 Code Review 通过
- 合并后立即部署

**适用场景**：

- 持续部署的互联网产品
- 小到中型团队（10 人以内效果最佳）
- 追求快速迭代的开发模式

**优点**：

- 规则简单，团队容易理解和执行
- 强制 Code Review 提高代码质量
- 天然适配 CI/CD 流水线

**缺点**：

- 没有版本维护概念，不适合需要多版本并行的场景
- 当并行功能较多时，main 分支可能频繁处于不稳定状态（虽然理论上 CI 应该保证）

### Trunk-Based Development：追求最小化分支

Trunk-Based Development（基于主干的开发，简称 TBD）是 Google、Facebook 等大型互联网公司采用的开发模式。它的核心思想是：**所有开发者都在同一个分支（trunk）上工作**，分支只存在很短的时间。

**核心原则**：

- `main`（或 `trunk`）是唯一的长期分支
- 功能分支存在时间不超过 2 天（最好几个小时）
- 使用 Feature Toggle（功能开关）控制未完成功能的发布
- 大量的小提交，频繁合并到主干

**工作流程**：

```bash
# 1. 从 main 创建短期功能分支
git checkout main
git checkout -b f/ticket-1234-user-profile

# 2. 开发，频繁提交
git commit -m "WIP: start user profile page"
git commit -m "WIP: add form validation"
git commit -m "WIP: connect to API"

# 3. 完成功能后立即合并回 main（< 2 天）
git checkout main
git merge --no-ff f/ticket-1234-user-profile

# 4. 部署后，删除功能分支
git branch -d f/ticket-1234-user-profile
```

**Feature Toggle 的使用**：

```javascript
// 功能开关配置
const featureFlags = {
  darkMode: process.env.DARK_MODE_ENABLED === 'true',
  newCheckout: process.env.NEW_CHECKOUT_ENABLED === 'true'
};

// 代码中使用开关
function renderApp() {
  return (
    <div>
      {featureFlags.darkMode && <DarkModeToggle />}
      {featureFlags.newCheckout ? <NewCheckout /> : <OldCheckout />}
    </div>
  );
}
```

**适用场景**：

- 大型互联网公司，高频迭代
- 追求「所有代码时刻可部署」的目标
- 团队有成熟的功能开关管理机制

**优点**：

- 最小化合并冲突，因为合并频率高、冲突规模小
- 持续集成友好，代码始终处于可发布状态
- 减少长期分支带来的技术债务

**缺点**：

- 需要完善的 Feature Toggle 系统
- 对团队纪律要求高，必须频繁提交和合并
- 不熟悉 TBD 的团队可能感到压力

## 三种工作流对比

| 维度 | Git Flow | GitHub Flow | Trunk-Based |
|---|---|---|---|
| 长期分支数量 | 2 个（main + develop） | 1 个 | 1 个 |
| 功能分支生命周期 | 数天到数周 | 数小时到数天 | 数小时（最多 2 天） |
| 发布周期 | 固定周期发布 | 随时可发布 | 持续发布 |
| 回滚难度 | 较难 | 容易 | 容易 |
| Code Review | 可选 | 必须 | 必须 |
| 适用团队规模 | 中大型 | 小中型 | 大型 |
| 学习曲线 | 较陡 | 平缓 | 陡峭 |
| 并行维护多版本 | 支持 | 不支持 | 有限支持 |

## 如何选择合适的工作流？

选择工作流不是选「最好」的，而是选「最适合」的。

**选 Git Flow 如果**：

- 你的团队需要同时维护多个生产版本（如 SaaS 产品需要维护 v1、v2 两个版本给不同客户）
- 你有固定的发布节奏，比如每月一个版本
- 团队较大，需要通过流程控制来保证质量

**选 GitHub Flow 如果**：

- 你的产品是持续交付的互联网应用
- 团队规模在 10 人左右
- 团队刚接触 Git，希望流程简单清晰

**选 Trunk-Based Development 如果**：

- 你的团队有 50 人以上，多个团队并行开发
- 你有成熟的 Feature Toggle 系统
- 团队追求极致的小批量、快迭代
- 你有足够的工程能力维护 CI/CD 和测试基础设施

## 混合工作流：现实中的妥协

很多团队会发现，纯理论的工作流在实践中需要调整。比如 Git Flow 太复杂，但 GitHub Flow 又不够精细，这时候可以采取混合策略：

```bash
# 采用 GitHub Flow 的主体框架
# 但增加 release 分支来处理发布窗口期

main ──────────────────────────────────── (始终可部署)
       │
       ├─ feature/* ─ (功能开发，PR 合并)
       │
       └─ release/1.0 ─── (发布前 freeze，hotfix 从这里出)
```

或者采用 Git Flow 的分支模型，但简化命名：

```bash
main ────────────────────────────────────
       │
       └─ develop ──────────────────── (日常开发)
              │
              ├─ feature/* ─ (功能分支)
              └─ hotfix/* ─ (热修复分支)
```

> [!TIP]
> 工作流不是一成不变的。随着团队成长、项目规模变化，工作流也需要相应调整。建议每季度回顾一次工作流是否仍然适合团队。
