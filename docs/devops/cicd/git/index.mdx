# Git 版本控制

代码是团队协作的产物，而版本控制系统就是让「多个人同时编辑同一份代码」成为可能的幕后英雄。

如果你做过这样的事——把代码复制好几份叫「备份」「备份2」「最终版」「最终版不改了」——那么你就已经理解了版本控制的核心需求：记录谁在什么时候改了什么，并且能够随时回到任何一个历史状态。

Git 是目前最流行的分布式版本控制系统，由 Linus Torvalds 于 2005 年创建。与早期的 SVN、CVS 不同，Git 采用分布式架构，每个开发者本地都拥有完整的代码仓库副本，这使得 Git 在离线环境下依然可以工作，分支创建和切换也极为轻量。

## Git 的核心优势

**分布式架构**：每个开发者的本地仓库都是完整的代码库，不依赖中央服务器。即使服务器宕机，任何一个本地副本都可以恢复整个项目。

**分支模型**：Git 的分支创建和切换几乎零成本，这使得「为每个功能开一个分支」成为标准实践，而不是像 SVN 那样被视为「重量级操作」。

**完整性保证**：Git 使用 SHA-1 哈希来标识每一次提交和每一个对象，任何文件的微小改动都会导致完全不同的哈希值，这保证了代码库的不可篡改性。

**性能**：Git 的核心操作（提交、分支、合并、推送、拉取）都在本地执行，不需要网络通信，这使得这些操作几乎是瞬时完成的。

## 目录结构

本章节涵盖 Git 使用的核心知识点：

**Git 工作流**：选择合适的分支策略是团队协作的起点。Git Flow 适合有严格发布周期的团队，GitHub Flow 适合持续交付的场景，Trunk-Based Development 则适合追求高速迭代的互联网公司。

**Git 常用命令**：除了 `git add`、`git commit`、`git push` 这些基础操作，你还需要掌握 `git merge` 与 `git rebase` 的区别、`git cherry-pick` 的妙用、`git stash` 的临时存储，以及如何用 `git bisect` 定位 bug。

**Git 分支管理**：分支命名规范、保护规则、长生命周期分支与短生命周期分支的维护策略。

**Git 提交规范**：如何写好 commit message，Conventional Commits 规范，以及如何使用 Commitizen 和 commitlint 工具强制规范化提交。

**Git Hooks**：在代码进入仓库之前、之后，甚至在推送之前、之后触发自定义脚本，这是实现自动化质量把控的第一道防线。
