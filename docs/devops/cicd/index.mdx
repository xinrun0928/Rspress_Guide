# CI/CD 持续集成与持续部署

凌晨 3 点，你被生产环境的告警叫醒。日志显示一个坏掉的版本被部署到了服务器，而这一切发生在「正常上线流程」完成之后。回溯问题根源，发现是某位同事跳过了代码 review，直接将未经测试的代码合并到了主分支，然后触发了一次「例行部署」。

这不是技术问题，这是一个**流程问题**。

CI/CD 的本质，不是让代码跑得更快，而是让**错误发现得更早**，让**人为失误的空间变得更小**。当每一次代码提交都自动触发构建、测试、部署，当每一次合并都需要经过检查，当每一次上线都可以一键回滚——你才能真正安心地睡个好觉。

## 什么是 CI/CD？

**持续集成（Continuous Integration）**：开发人员频繁地将代码合并到共享主干，每次合并都自动触发构建和测试，目的是尽早发现集成错误。

**持续交付（Continuous Delivery）**：代码在通过所有自动化测试后，随时可以部署到生产环境，但仍需要人工确认才能真正发布。

**持续部署（Continuous Deployment）**：代码通过测试后自动部署到生产环境，无需人工干预，完全自动化。

三者之间的关系可以这样理解：CI 是基础，CD 是延伸，而持续部署是持续交付的最终形态。

## CI/CD 流水线全景

一个完整的 CI/CD 流水线，通常包含以下几个阶段：

| 阶段 | 核心任务 | 常见工具 |
|---|---|---|
| 代码提交 | 开发者提交代码，触发流水线 | Git, GitHub, GitLab |
| 编译构建 | 代码编译、依赖安装、制品生成 | Maven, Gradle, npm, Docker |
| 单元测试 | 运行基础测试用例 | JUnit, pytest, Jest |
| 静态分析 | 代码质量检查、安全扫描 | SonarQube, ESLint |
| 集成测试 | 多模块集成测试 | Selenium, Postman |
| 镜像构建 | 容器镜像打包与推送 | Docker, BuildKit |
| 安全扫描 | 镜像漏洞扫描 | Trivy, Clair |
| 部署 | 应用部署到目标环境 | ArgoCD, Spinnaker |
| 验证 | 健康检查、冒烟测试 | curl, kubectl |

## 本章节内容

本章节将从四个维度全面讲解 CI/CD 实践：

### Git 版本控制与工作流

版本控制是 CI/CD 的基石。我们会深入讲解 Git 工作流的选择（Git Flow、GitHub Flow、Trunk-Based Development），常用命令的高级用法，以及如何通过 Git Hooks 在提交阶段就把好质量关。

### Jenkins 持续集成

Jenkins 作为老牌 CI/CD 工具，至今仍是企业级流水线的主流选择。我们会从架构原理讲起，涵盖 Jenkinsfile 编写、插件生态、Kubernetes 集成，以及 Blue Ocean 可视化界面。

### 现代 CI/CD 工具

云原生时代涌现了大量新型 CI/CD 工具：GitLab CI 以 YAML 驱动流水线，GitHub Actions 让 Actions 市场触手可及，ArgoCD 引领 GitOps 潮流，Tekton 成为 Kubernetes 原生的构建标准，Spinnaker 则在多云部署场景下独占鳌头。

### 最佳实践

最后一章聚焦实操：如何设计高效的流水线，如何集成 SonarQube 做代码质量把控，如何用 Trivy 扫描镜像漏洞，如何管理制品仓库，以及如何实施蓝绿部署和金丝雀发布。

## 技术选型参考

| 场景 | 推荐工具 |
|---|---|
| 中小型团队，快速上手 | GitHub Actions |
| 企业级，需要复杂流水线 | Jenkins + Kubernetes |
| GitLab 全家桶用户 | GitLab CI |
| Kubernetes 原生应用 | Tekton + ArgoCD |
| 多云/混合云部署 | Spinnaker |
| GitOps 实践 | ArgoCD |

> [!TIP]
> 选择 CI/CD 工具时，最重要的是团队已有的技术栈和工具链整合能力。盲目追求「最先进」的工具，不如选择团队最能驾驭的工具。
