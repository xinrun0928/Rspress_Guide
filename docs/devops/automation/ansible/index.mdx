# Ansible 高级实践

「Ansible 学了，但 playbook 越写越长，问题越来越多。」

这是很多 Ansible 用户会遇到的问题。Playbook 从几十行到几百行，变量散落各处，重复代码越来越多，每次修改都担心影响其他环境。

Ansible 高级实践解决的就是这个问题：**用 Roles 组织代码、用变量管理差异、用集成能力连接 Kubernetes 和 Docker**。

## Role：从脚本到工程的跨越

```
# 不用 Role 的痛苦
site.yml
├── 200 行 Task（安装 Nginx）
├── 100 行 Task（配置 Nginx）
├── 50 行 Task（复制证书）
├── 150 行 Task（安装 PostgreSQL）
├── 80 行 Task（配置 PostgreSQL）
└── 维护噩梦：改一处要滚动的文件太多
```

```
# 用 Role 的优雅
site.yml
├── roles/nginx/          # Nginx 的一切
├── roles/postgresql/     # PostgreSQL 的一切
└── roles/ssl-certs/     # 证书管理的一切
```

**Role 的本质是封装**：把「安装-配置-启动」打包成一个独立的单元。你可以单独测试、单独复用、单独版本控制。

## 变量优先级：搞不清楚就会踩坑

Ansible 的变量优先级是一个经典面试题，也是实操中最容易出错的地方。从低到高排列：

```bash
# Ansible 变量优先级（低 → 高）
1. command line values (ansible-playbook -e 'var=value')  # 最高
2. role defaults                   # 最低，通常用来提供默认值
3. inventory file/group_vars
4. playbook group_vars/all
5. host_vars/hostname
6. facts discovered
7. register vars
8. set_fact / var
9. include_vars
10. include_role / include_task
11. extra vars (-e)                # 命令行指定
```

**实战建议**：
- Role 的 `defaults/main.yml` 提供默认值
- `vars/main.yml` 存放私有变量（不应该被覆盖）
- 敏感信息用 `ansible-vault` 加密
- 跨环境差异用 `group_vars` 或 `host_vars`

## 与 Kubernetes 的集成

Ansible 的 `k8s` 模块让你在 Playbook 中直接操作 Kubernetes 资源：

```yaml
- name: Deploy application to K8s
  k8s:
    definition:
      apiVersion: apps/v1
      kind: Deployment
      metadata:
        name: my-app
        namespace: production
      spec:
        replicas: 3
        selector:
          matchLabels:
            app: my-app
        template:
          metadata:
            labels:
              app: my-app
          spec:
            containers:
              - name: my-app
                image: myregistry/myapp:latest
                ports:
                  - containerPort: 8080
```

配合 `k8s_auth` 模块管理 Kubeconfig，Ansible 可以成为 Kubernetes 集群的统一运维入口。

## 面试的核心逻辑

Ansible 高级实践的面试，重点考察三个方面：

1. **工程化能力**：能不能设计出可复用的 Role？变量如何分层？敏感信息怎么处理？
2. **幂等性理解**：Ansible 的幂等性是怎么保证的？什么情况下会破坏幂等性？
3. **集成深度**：Ansible 和 Kubernetes 的结合点在哪？什么场景适合用 Ansible 代替 kubectl？

```
# Ansible 最佳实践检查清单

□ Role 目录结构完整（tasks/handlers/templates/vars/defaults/meta）
□ 敏感变量使用 ansible-vault 加密
□ 变量命名有命名空间（role_name_variable_name）
□ 大量重复任务抽取为 Role 或 include
□ 错误处理使用 block/rescue
□ 幂等性验证：多次执行结果一致
□ 与 K8s/Docker 集成使用官方模块而非 shell
```

> "好的 Ansible 代码，应该像一个经验丰富的运维工程师在执行任务：知道要做什么、知道怎么做、知道做完后怎么验证。"
