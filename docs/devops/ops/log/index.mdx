# 日志体系总览

故障排查的第一步是什么？查看日志。

但当你的系统分布在 100 台服务器上，日志散落在各个角落，你怎么查？

这就是日志系统存在的意义：**集中收集、统一存储、灵活查询**。

## 日志的重要性

### 为什么需要日志系统

| 问题 | 解决方案 |
|---|---|
| 日志分散在多台机器 | 集中收集 |
| 日志量太大，查询慢 | 分布式存储 + 索引 |
| 需要跨日志关联 | 结构化日志 + Trace ID |
| 日志格式不统一 | 解析 + 标准化 |
| 实时告警 | 日志告警规则 |

### 日志 vs 指标 vs 链路追踪

可观测性的三大支柱：

| 类型 | 数据特点 | 工具 |
|---|---|---|
| **指标（Metrics）** | 结构化、数值型 | Prometheus |
| **日志（Logs）** | 非结构化/半结构化 | ELK、Loki |
| **链路（Traces）** | 请求级别的调用链 | Jaeger、Zipkin |

三者结合，才能构建完整的可观测性体系。

## 日志技术栈

### ELK Stack

```
Elasticsearch + Logstash + Kibana + Beats
```

| 组件 | 职责 |
|---|---|
| **Beats** | 轻量级日志收集器 |
| **Logstash** | 日志 ETL（提取、转换、加载） |
| **Elasticsearch** | 日志存储和搜索引擎 |
| **Kibana** | 日志可视化 |

### EFK Stack

```
Elasticsearch + Fluentd/Fluent Bit + Kibana
```

与 ELK 的区别：用 Fluentd/Fluent Bit 替代 Logstash。

### Loki

```
Prometheus 团队出品，专为 Kubernetes 设计
```

Grafana Loki + Promtail + Grafana

### 对比

| 特性 | ELK/EFK | Loki |
|---|---|---|
| 索引方式 | 全文索引 | 标签索引 |
| 存储成本 | 高 | 低 |
| 查询速度 | 快（大数据量） | 中等 |
| K8s 集成 | 好 | 非常好 |
| 标签过滤 | 不支持 | 原生支持 |
| Grafana 集成 | 支持 | 原生支持 |
