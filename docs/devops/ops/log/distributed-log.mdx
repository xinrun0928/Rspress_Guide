# 分布式日志追踪

用户反馈：「我下单失败了」，然后甩给你一个订单号消失了。你怎么办？

在单体应用时代，你可以翻日志、查数据库、定位问题。但在微服务架构中，一个下单请求可能经过：网关 → 认证服务 → 用户服务 → 库存服务 → 支付服务 → 订单服务，每个服务都有自己的日志。

这就是分布式日志追踪要解决的问题。

## Trace ID：链路追踪的钥匙

### 原理

当请求进入系统时，生成一个唯一的 Trace ID：

```
请求入口（网关）
  │
  ├─ Trace ID: abc123-xyz789
  │
  ▼
认证服务（读取 Trace ID）
  │
  ▼
商品服务（传递 Trace ID）
  │
  ▼
订单服务（记录 Trace ID）
```

所有服务在记录日志时，都带上这个 Trace ID。通过 Trace ID，你就能把分散在各个服务的日志串起来。

### 实现方式

HTTP Header 传播：

```
X-Request-ID: abc123-xyz789
X-B3-TraceId: abc123-xyz789
X-Correlation-ID: abc123-xyz789
```

## OpenTelemetry 标准

### 核心理念

OpenTelemetry（OTel）是 CNCF 的可观测性标准，统一了：

- **Traces**：请求链路追踪
- **Metrics**：指标数据
- **Logs**：日志数据

### 数据模型

```java
// OpenTelemetry SDK 示例
Tracer tracer = OpenTelemetry.getTracer("order-service");

Span span = tracer.spanBuilder("create-order").startSpan();
try {
    span.setAttribute("order.id", orderId);
    span.setAttribute("customer.id", customerId);
    
    // 业务逻辑
    createOrder(order);
    
} finally {
    span.end();
}
```

### 自动注入与传播

在 Spring Boot 中，OpenTelemetry 可以自动处理：

```xml title="pom.xml"
<dependency>
    <groupId>io.opentelemetry.instrumentation</groupId>
    <artifactId>opentelemetry-spring-boot-starter</artifactId>
</dependency>
```

## Jaeger 分布式追踪

### 架构

```
应用（埋点）→ Jaeger Agent → Jaeger Collector → Elasticsearch
                                                          ↓
                                                         Jaeger UI
```

### 部署示例

```yaml title="jaeger-production.yml"
apiVersion: apps/v1
kind: Deployment
metadata:
  name: jaeger-collector
spec:
  replicas: 2
  template:
    spec:
      containers:
      - name: jaeger-collector
        image: jaegertracing/jaeger-collector:1.50
        env:
        - name: SPAN_STORAGE_TYPE
          value: elasticsearch
        - name: ES_SERVER_URLS
          value: http://elasticsearch:9200
        ports:
        - containerPort: 14250
        - containerPort: 14268
```

### Spring Boot 集成

```yaml title="application.yml"
spring:
  application:
    name: order-service

management:
  tracing:
    sampling:
      probability: 1.0  # 采样率 100%
```

## 日志关联追踪

### 统一日志格式

```json
{
  "timestamp": "2024-01-15T10:30:00.123Z",
  "level": "INFO",
  "service": "order-service",
  "traceId": "abc123-xyz789",
  "spanId": "span-456",
  "message": "Order created successfully",
  "orderId": "ORD-12345",
  "customerId": "CUST-67890"
}
```

### 日志收集与关联

```yaml title="filebeat.yml"
filebeat.inputs:
- type: container
  paths:
    - /var/log/containers/*.log
  
processors:
- add_kubernetes_metadata:
    host: ${NODE_NAME}
    matchers:
    - logs_path:
        logs_path: "/var/log/containers/"

- add_trace_id:
    trace_id_header: "x-b3-traceid"
    span_id_header: "x-b3-spanid"
```

### Kibana 关联查询

```
# 通过 Trace ID 查询完整链路
trace.id: "abc123-xyz789"
```

## 常见问题与解决

### 1. Trace ID 不传递

**症状**：链路断断续续，很多服务没有 Trace ID。

**解决**：检查 Header 传播链路，确保每个服务都正确传递。

### 2. 采样率设置

**症状**：高频服务产生大量 Trace 数据。

**解决**：动态调整采样率：

```yaml
spring:
  sampling:
    # 99% 请求采样，高频服务降低
    probability: 0.01
```

### 3. 日志与 Trace 关联

**症状**：日志和 Trace 是分开的，无法快速关联。

**解决**：在日志中添加 Trace ID，并让 Filebeat/Fluentd 自动提取：

```java
log.info("Order created, traceId={}", MDC.get("traceId"));
```

## 最佳实践

| 实践 | 说明 |
|---|---|
| 统一 Trace ID 生成 | 网关/入口服务统一生成 |
| 采样率控制 | 高频服务降低采样率 |
| 关联日志 | 日志中嵌入 Trace ID |
| 异常栈追踪 | Span 中记录异常信息 |
| 服务依赖可视化 | 定期检查服务拓扑 |

## 总结

分布式日志追踪是微服务运维的基础设施。通过统一的 Trace ID，将分散的日志串联成完整的请求链路。

核心要点：

1. **入口生成 Trace ID**
2. **服务间传递 Trace ID**
3. **日志中嵌入 Trace ID**
4. **链路追踪工具关联**

下次用户再说「下单失败了」，你只需要一个 Trace ID，就能快速定位问题。
