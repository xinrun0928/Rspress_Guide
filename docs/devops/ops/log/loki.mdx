# Loki：日志聚合系统

你有没有想过——ELK 虽然功能强大，但部署和维护起来相当复杂。Elasticsearch 需要大量资源，Logstash 配置繁琐。有没有更轻量的日志解决方案？

Loki 就是来解决这个问题的。Loki 是 Grafana Labs 开发的日志聚合系统，它的设计哲学是：**像 Prometheus 一样简单，像日志一样强大**。

## Loki 是什么

Loki 是 Grafana Labs（Prometheus 团队）开源的日志聚合系统，专为 Kubernetes 环境设计。

### Loki vs ELK

| 维度 | Loki | ELK |
|---|---|---|
| **索引方式** | 仅索引标签（metadata） | 全文索引 |
| **存储成本** | 低（仅存原始日志） | 高（每条日志都建索引） |
| **部署复杂度** | 低 | 高 |
| **查询性能** | 中等 | 高 |
| **生态系统** | Grafana 原生 | Kibana 功能丰富 |
| **Label 查询** | 原生支持 | 需要额外配置 |
| **适用场景** | K8s 云原生 | 全场景通用 |

Loki 的核心思想：**不需要对所有日志内容建索引，只对元数据（标签）建索引**。这使得 Loki 可以在低资源消耗下存储海量日志。

```
│                         ┌──────────┴──────────┐                        │
│                         ▼                     ▼                        │

│              │    Object        │    │    Object        │              │

│  │                     Querier                                  │        │

│  └──────────────────────────────────────────────────────────────┘        │
│                                    │                                     │
│                                    ▼                                     │
│  ┌──────────────────────────────────────────────────────────────┐        │
│  │                    Query Frontend (可选)                         │        │

│                                                                             │
│  ┌──────────────────────────────────────────────────────────────┐        │
```

### 核心组件

| 组件 | 说明 |
|---|---|
| **Distributor** | 接收客户端日志流，验证并分割为批次 |
| **Ingester** | 接收日志块，写入存储；内存中积累日志块 |
| **Querier** | 处理查询请求，从存储和 Ingester 并行查询 |
| **Query Frontend** | 查询缓存、并行化（可选） |
| **Compactor** | 压缩和删除日志数据（可选） |

## Loki 核心概念

### Label（标签）

Loki 使用标签来组织和查询日志，类似于 Prometheus：

```log
# Loki 日志流通过标签标识
{service="myapp", environment="production", namespace="default"}
2024-01-15 10:30:00 INFO Application started
2024-01-15 10:30:01 INFO Connected to database
2024-01-15 10:30:02 ERROR Failed to connect to cache
```

### LogQL：Loki 查询语言

LogQL 是 Loki 的查询语言，类似 PromQL：

```txt
# 基本查询：获取所有 myapp 的日志
{service="myapp"}

# 带过滤：只获取 ERROR 级别的日志
{service="myapp"} |= "ERROR"

# 正则匹配
{service="myapp"} |~ "timeout.*connection"

# 排除某些日志
{service="myapp"} != "DEBUG"

# 时间范围
{service="myapp"} [5m]  # 最近 5 分钟
{service="myapp"} [1h]   # 最近 1 小时
{service="myapp"} [5m] |= "ERROR" != "timeout"
```

### 日志流（Log Stream）

日志流是一组具有相同标签的日志：

```yaml
# 日志流示例
{
  "service": "myapp",
  "environment": "production",
  "namespace": "default",
  "pod": "myapp-abc123"
}
# 所有具有这些标签的日志属于同一个流
```

## 部署 Loki

### Docker Compose 快速部署

```yaml title="docker-compose.yml"
version: '3.8'
services:
  loki:
    image: grafana/loki:2.9.0
    ports:
      - "3100:3100"
    volumes:
      - ./loki-config.yaml:/etc/loki/config.yaml
    command: -config.file=/etc/loki/config.yaml
  promtail:
    image: grafana/promtail:2.9.0
    volumes:
      - ./promtail-config.yaml:/etc/promtail/config.yaml
      - /var/log:/var/log
    command: -config.file=/etc/promtail/config.yaml
  grafana:
    image: grafana/grafana:latest
    ports:
      - "3000:3000"
    volumes:
      - ./grafana-datasources.yaml:/etc/grafana/provisioning/datasources/datasources.yaml
    environment:
      - GF_AUTH_ANONYMOUS_ENABLED=true
      - GF_AUTH_ANONYMOUS_ORG_ROLE=Admin
```

```yaml title="loki-config.yaml"
server:
  http_listen_port: 3100

schema_config:
  configs:
    - from: 2024-01-01
      store: boltdb-shipper
      object_store: filesystem
      schema: v11
      index:
        prefix: loki_index_
        period: 24h

storage_config:
  boltdb_shipper:
    active_index_directory: /loki/index
    cache_location: /loki/index_cache
  filesystem:
    directory: /loki/chunks

limits_config:
  reject_old_samples: true
  reject_old_samples_max_age: 168h

chunk_store_config:
  max_look_back_period: 168h

table_manager:
  retention_deletes_enabled: true
  retention_period: 720h  # 30 天
```

## Promtail：日志收集

Promtail 是 Loki 的官方日志收集代理，专门为 Kubernetes 设计。

### Promtail 配置文件

```yaml title="promtail-config.yaml"
server:
  http_listen_port: 9080
  grpc_listen_port: 0

positions:
  filename: /tmp/positions.yaml

client:
  url: http://loki:3100/loki/api/v1/push

scrape_configs:
  # 收集 Kubernetes Pod 日志
  - job_name: kubernetes-pods
    kubernetes_sd_configs:
      - role: pod
    relabel_configs:
      - source_labels: [__meta_kubernetes_pod_name]
        target_label: pod
      - source_labels: [__meta_kubernetes_namespace]
        target_label: namespace
      - source_labels: [__meta_kubernetes_pod_container_name]
        target_label: container
      - action: labelmap
        regex: __meta_kubernetes_pod_label_(.+)
      - action: replace
        replacement: /var/log/pods/$1/$2/*.log
        target_label: __path__
```

```yaml title="promtail-daemonset.yaml"

          mountPath: /etc/promtail

          mountPath: /tmp

      - name: config
        configMap:
          name: promtail
      - name: run
        hostPath:
          path: /tmp/promtail
      - name: containers
        hostPath:
          path: /var/log/pods
```


### LogQL 查询示例

```txt
# 查看所有 production 环境的错误日志
{service=~".*", environment="production"} |= "ERROR"

# 查看特定服务的日志
{service="myapp", namespace="default"}

# 多条件查询
{service="myapp"} |= "ERROR" != "timeout" |~ "connection.*failed"

# 统计每分钟日志条数
sum(rate({service="myapp"}[1m])) by (level)

# 查看日志趋势
sum(rate({service="myapp"}[5m])) by (namespace)

# 查看异常分布
topk(10, sum by (error_type) (rate({service="myapp", level="error"}[5m])))
```

## 与 Prometheus 联动

Loki 可以与 Prometheus 共享标签，实现联合查询：

### metric_logs 管道

```yaml
# Prometheus 配置：发送日志到 Loki
metrics.instance: localhost:9090
metrics.path: /metrics

# Loki 配置：使用 Prometheus 标签
scrape_configs:
  - job_name: kubernetes-pods
    kubernetes_sd_configs:
      - role: pod
    relabel_configs:
      # 添加 instance 标签
      - action: labelmap
        regex: __meta_ketrics_scrape_(.+)
        replacement: "${1}"
```

### Grafana Explore 联动

在 Grafana Explore 中，可以在 Metrics 和 Logs 之间切换：

```
Metrics:                    Logs:
rate(http_requests[5m])     {service="myapp"} |= "ERROR"
     │                          │
     └──────────────────────────┘
           点击可直接跳转到对应时间范围的日志
```

## 最佳实践

### 1. 标签设计

```yaml
# 好的标签设计（高基数风险）
{service="myapp", pod="myapp-abc123", container="myapp"}

# 推荐的标签设计
{service="myapp", namespace="default", env="prod"}
```

:::warning 避免高基数标签
`pod`、`container` 等标签如果值过多，会导致 Loki 索引爆炸。建议只在需要时添加，或者在查询时通过 relabel_configs 过滤。
:::

### 2. 日志级别分离

```yaml
# 将不同级别的日志分流
scrape_configs:
  - job_name: kubernetes-pods
    kubernetes_sd_configs:
      - role: pod
    relabel_configs:
      # 只收集 ERROR 和 WARN 级别日志
      - action: keep
        source_labels: [__meta_kubernetes_pod_label_log_level]
        regex: "ERROR|WARN"
```

### 3. 存储优化

```yaml title="loki-config.yaml"
# 使用 S3 存储（推荐生产环境）
storage_config:
  aws:
    s3: s3://my-bucket/loki
    s3forcepathstyle: true
    region: us-east-1
    bucketnames: loki-data
    chunk_buffer_size: 5_000_000
    flush_op_timeout: 10m
    upload_timeout: 10m
```

## 常见问题排查

### 日志未采集

```bash
# 检查 Promtail 日志
kubectl logs -l app=promtail -f

# 检查 Promtail 配置
kubectl exec promtail-xxx -- cat /etc/promtail/promtail.yaml
```

### 查询超时

```txt
# 增加查询超时
{service="myapp"} [5m] |= "ERROR"
# 使用 limit 限制返回条数
{service="myapp"} [5m] | limit 100
```

### 存储满

```bash
# 调整保留期
limits_config:
  retention_period: 720h  # 30 天

# 或使用 Compactor 清理
compactor:
  working_directory: /loki/compactor
  retention_enabled: true
```

## 总结

Loki 的核心要点：

1. **标签索引**：只对元数据建索引，存储成本低
2. **Prometheus 友好**：共享标签，联动查询
3. **Promtail 收集**：K8s 原生的日志收集
4. **Grafana 原生**：无缝集成 Explore 功能
5. **LogQL 查询**：类 PromQL 的查询语言

对于 Kubernetes 环境，Loki 是 ELK 的绝佳替代方案，配合 Prometheus + Grafana，可以构建完整的可观测性平台。
