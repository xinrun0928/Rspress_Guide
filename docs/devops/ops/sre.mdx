# SRE 实践指南

凌晨 2 点，你的服务突然不可用了。你花了 2 个小时排查，最后发现是某个配置变更导致的。事后复盘，你开始思考：如何避免下一次？如何让系统更可靠？

这就是 SRE 诞生的背景——用工程化的方式，让服务更加稳定可靠。

## SRE 是什么

### 起源

SRE（Site Reliability Engineering）起源于 Google，核心理念是用软件工程的方法解决运维问题。

传统运维：「出了问题我来修」
SRE：「问题能不能不发生？发生了能不能自动恢复？」

### 核心理念

| 理念 | 说明 |
|---|---|
| **把运维当软件问题** | 用代码和自动化解决问题 |
| **关注可靠性** | 可靠性是产品特性，需要设计 |
| **SLI/SLO/SLA** | 用指标定义和衡量可靠性 |
| **Toil 治理** | 减少重复性人工操作 |
| **发布管理** | 渐进式发布，降低风险 |

## SLI/SLO/SLA：三剑客

### 定义

- **SLI（Service Level Indicator）**：服务级别指标，衡量服务质量的具体指标
- **SLO（Service Level Objective）**：服务级别目标，SLI 的目标值
- **SLA（Service Level Agreement）**：服务级别协议，对外的承诺

### 常见 SLI

```markdown
| SLI | 指标定义 | 计算方式 |
|---|---|---|
| 可用性 | 请求成功率 | success / total × 100% |
| 延迟 | P99 响应时间 | 排序后取 99 分位 |
| 吞吐量 | QPS/TPS | 单位时间处理量 |
| 错误率 | 5xx 比例 | 5xx / total × 100% |
```

### 设定 SLO

```yaml title="slo-config.yaml"
service: order-service

slis:
  availability:
    description: "请求成功率"
    sli: "rate(success_requests_total{service='order'}) / rate(requests_total{service='order'})"
    target: 99.9  # 目标 99.9%
    
  latency:
    description: "P99 响应时间"
    sli: "histogram_quantile(0.99, request_duration_bucket{service='order'})"
    target: 500   # 目标 500ms
    
  error_rate:
    description: "5xx 错误率"
    sli: "rate(http_requests_total{service='order',status=~'5..'}) / rate(http_requests_total{service='order'})"
    target: 0.1   # 目标 0.1%
```

## Error Budget：预算思维

### 核心概念

Error Budget = 1 - SLO

如果 SLO 是 99.9%，Error Budget 就是 0.1%。

对于一个月 43,200 分钟：

```
Error Budget = 43,200 × 0.1% = 43.2 分钟
```

这 43.2 分钟是「允许犯错的空间」。

### 决策依据

| Error Budget 状态 | 行动 |
|---|---|
| 充足 | 可以激进发布新功能 |
| 消耗中 | 谨慎发布，优先修复问题 |
| 耗尽 | 停止发布，全力恢复可靠性 |

:::tip
Error Budget 是一种容错思维。100% 的可靠性是不现实的，关键是管理好「可以接受的错误空间」。
:::

## Toil 治理

### 什么是 Toil

Toil = 重复性的人工运维工作

```markdown
# Toil 的特点
- 手动执行
- 重复做同样的事
- 没有持久价值
- 与服务规模线性增长
```

### Toil 治理四步法

**第一步：识别 Toil**

```bash
# 统计手动操作次数
# 记录每次故障的处理时间
# 评估重复性任务
```

**第二步：量化 Toil**

```yaml
toil_metrics:
  deployment_manual: 5h/week
  incident_response: 8h/week
  capacity_planning: 3h/week
  certificate_renewal: 2h/week
  
  total_toil: 18h/week
  acceptable_toil: 25%  # 目标 toil 不超过 25%
```

**第三步：自动化**

```bash
# 手动证书更新 → 自动 Let's Encrypt
# 手动部署 → CI/CD 流水线
# 手动扩容 → HPA 自动扩缩容
```

**第四步：持续改进**

定期 Review，确保持续减少 Toil。

## 发布与变更管理

### 渐进式发布

```yaml title="canary-deployment.yaml"
apiVersion: flagger.app/v1beta1
kind: Canary
metadata:
  name: order-service
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: order-service
  analysis:
    interval: 1m
    threshold: 5
    maxWeight: 50
    stepWeight: 10
    
    metrics:
    - name: request-success-rate
      thresholdRange:
        min: 99
      interval: 1m
    - name: request-duration
      thresholdRange:
        max: 500
      interval: 1m
```

### 发布检查清单

| 检查项 | 说明 |
|---|---|
| 功能测试通过 | 所有测试用例通过 |
| 灰度验证 | 1% → 10% → 50% → 100% |
| 监控告警就绪 | 新指标已配置 |
| 回滚方案就绪 | 一键回滚脚本 |
| 值班人员就绪 | 有人关注上线 |

## 故障管理

### 故障分级

| 级别 | 影响 | 响应时间 | 恢复时间 |
|---|---|---|---|
| P0 | 全站不可用 | 5 分钟 | 30 分钟 |
| P1 | 部分服务不可用 | 15 分钟 | 2 小时 |
| P2 | 功能降级 | 1 小时 | 4 小时 |
| P3 | 非核心问题 | 4 小时 | 24 小时 |

### 故障处理流程

```
发现故障
    │
    ▼
评估影响 → 通知 → 成立应急群
    │
    ▼
快速止血（回滚/降级/限流）
    │
    ▼
定位根因
    │
    ▼
修复上线
    │
    ▼
复盘总结
```

## 监控与告警

### 黄金指标

Google SRE 提出的四个黄金指标：

| 指标 | 说明 | 问什么问题 |
|---|---|---|
| **Latency** | 延迟 | 服务是否变慢了？ |
| **Traffic** | 流量 | 有多少请求进来？ |
| **Errors** | 错误 | 有多少请求失败了？ |
| **Saturation** | 饱和度 | 服务是否快满了？ |

### 告警设计

```yaml title="alert-rules.yaml"
groups:
- name: order-service
  rules:
  # 黄金指标告警
  - alert: HighLatency
    expr: histogram_quantile(0.99, request_duration_bucket{service="order"}) > 1
    for: 5m
    labels:
      severity: warning
    annotations:
      summary: "High latency detected"
      description: "P99 latency is {{ $value }}s"
      
  - alert: HighErrorRate
    expr: rate(http_requests_total{service="order",status=~"5.."}[5m]) > 0.01
    for: 2m
    labels:
      severity: critical
    annotations:
      summary: "High error rate"
      description: "Error rate is {{ $value | humanizePercentage }}"
```

## 容量规划

### 水平扩展

```yaml title="hpa.yaml"
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: order-service-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: order-service
  minReplicas: 3
  maxReplicas: 100
  
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
  
  - type: Pods
    pods:
      metric:
        name: requests_per_second
      target:
        type: AverageValue
        averageValue: "1000"
```

### 资源预测

```python
# 简单容量预测
current_capacity = 1000  # 当前 QPS
growth_rate = 0.1  # 月增长 10%

future_months = 6
future_capacity = current_capacity * (1 + growth_rate) ** future_months

print(f"6 个月后需要的容量: {future_capacity} QPS")
```

## 总结

SRE 的核心思维转变：

| 传统思维 | SRE 思维 |
|---|---|
| 出了问题再修 | 预防问题发生 |
| 手动操作 | 自动化一切 |
| 追求 100% 可靠 | 合理的 Error Budget |
| 功能优先发布 | 可靠性与功能并重 |
| 救火式运维 | 可预测的运维 |

SRE 不是一套工具，而是一种文化。需要团队持续践行。
