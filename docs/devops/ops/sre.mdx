# SRE 与可观测性

你有没有想过——传统运维和 SRE 的区别是什么？SRE 到底在做什么？

SRE（Site Reliability Engineering）最初由 Google 提出，是一种将软件工程方法应用于运维工作的实践。SRE 的目标是：**在保障系统可靠性的同时，拥抱变化、提升迭代速度**。

## 传统运维 vs SRE

| 维度 | 传统运维 | SRE |
|---|---|---|
| **核心理念** | 稳定优先 | 可靠性 + 速度平衡 |
| **变更管理** | 手动审批，变更窗口 | 小步快跑，持续交付 |
| **故障处理** | 值班响应，手动排查 | 自动化响应，事后分析 |
| **监控指标** | 基础监控 | SLI/SLO/SLA |
| **错误预算** | 不考虑 | 核心概念 |
| **工具文化** | 手动操作 | 一切自动化 |

## SLI/SLO/SLA

### 概念定义

- **SLI（Service Level Indicator）**：服务级别指标，衡量服务健康度的具体指标
- **SLO（Service Level Objective）**：服务级别目标，SLI 的目标值
- **SLA（Service Level Agreement）**：服务级别协议，对外的承诺

### 常见 SLI/SLO

```yaml
# SLI：请求延迟
SLI: 99% 的请求延迟 < 200ms
SLO: 每月 99.5% 的请求延迟 < 200ms
SLA: 每月 99% 的请求延迟 < 200ms（对外承诺，条件更宽松）
```

```yaml
# SLI：可用性
SLI: 请求成功率
SLO: 每月 99.9% 请求成功
SLA: 每月 99.5% 请求成功
```

### 错误预算

错误预算是 SRE 的核心概念：**100% - SLO = 错误预算**

```
SLO: 99.9%（每月）
错误预算：0.1%

每月时间：30天 × 24小时 × 60分 = 43,200 分钟
允许的错误时间：43,200 × 0.001 = 43.2 分钟
```

错误预算政策：

- 错误预算充足：正常发布新功能
- 错误预算不足：停止发布，专注于稳定性

## SRE 实践

### 1. Toil 治理

Toil 是 SRE 中一个重要概念，指那些**重复性、低价值、手工操作的工作**。SRE 的目标是持续减少 Toil，将工程师的时间用于更有价值的工作。

```yaml
# Toil 特征
- 需要人工干预
- 手动执行
- 重复性工作
- 低价值
- 线性扩展
```

减少 Toil 的方法：

- 自动化一切
- 自愈系统设计
- 告警优化
- 标准化流程

### 2. 监控与告警

```yaml
# 黄金指标（RED 方法）
- Rate：请求率
- Error：错误率
- Duration：延迟

# 基础设施指标（USE 方法）
- Utilization：利用率
- Saturation：饱和度
- Errors：错误
```

### 3. 事故管理

```yaml
# 事故等级
P1：核心业务不可用
P2：核心业务受损
P3：非核心业务受损
P4：轻微影响
```

事故响应流程：

1. 检测
2. 响应
3. 分析
4. 恢复
5. 总结

### 4. 变更管理

SRE 推崇渐进式变更：

- 小步快跑
- 可观测性
- 自动化测试
- 灰度发布

## 可观测性三要素

可观测性是 SRE 的基础。Otel 定义了可观测性的三个支柱：

1. **Metrics（指标）**：系统行为的数值测量
2. **Logs（日志）**：系统事件的离散记录
3. **Traces（追踪）**：请求在分布式系统中的路径

## 总结

SRE 的核心要点：

1. **SLI/SLO/SLA**：用指标定义可靠性目标
2. **错误预算**：在可靠性和速度之间找到平衡
3. **Toil 治理**：持续减少重复性工作
4. **可观测性**：Metrics、Logs、Traces 三位一体
5. **事故管理**：检测、响应、分析、恢复、总结
6. **渐进式变更**：小步快跑，持续改进

SRE 不是一个人的工作，而是整个团队的目标。通过建立正确的文化、流程和工具，可以实现高可靠性和快速迭代的平衡。
