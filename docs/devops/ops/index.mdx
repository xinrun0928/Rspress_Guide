# 运维工具概览

凌晨 3 点，告警响起：「服务响应时间超过阈值」。你揉揉眼睛，打开监控大屏，发现 CPU 使用率飙升。接下来的 30 分钟，你开始了一场与时间的赛跑：查日志、追链路、定位问题、回滚版本……

这就是运维的日常。而好的运维工具，就是让你在这场赛中赢得先机的武器。

## 运维工具全景图

在 DevOps 时代，运维工具不再只是「监控告警」那么简单。从代码提交到生产运行，每一个环节都需要相应的工具支撑：

| 领域 | 核心问题 | 工具代表 |
|---|---|---|
| **日志管理** | 分散的日志如何统一查看？ | ELK、EFK、Loki |
| **监控系统** | 服务是否健康？如何预警？ | Prometheus、Grafana |
| **链路追踪** | 请求卡在哪一步？ | Jaeger、Zipkin |
| **日志收集** | 如何高效收集海量日志？ | Filebeat、Fluentd |
| **分布式日志** | 如何追踪分布式事务？ | OpenTelemetry |
| **SRE 实践** | 如何让系统更可靠？ | SLO、Error Budget |

## 日志管理体系

### ELK Stack

ELK 是最经典的日志解决方案：

- **Elasticsearch**：分布式搜索引擎，存储和检索日志
- **Logstash**：数据处理管道，收集、转换、发送日志
- **Kibana**：可视化界面，查询和分析日志

```
应用日志 → Filebeat → Logstash → Elasticsearch ← Kibana
```

### EFK Stack

与 ELK 类似，但用 Fluentd 替代 Logstash：

```
应用日志 → Fluentd → Elasticsearch ← Kibana
```

Fluentd 比 Logstash 更轻量，适合容器环境。

### Loki

Grafana 推出的日志聚合系统，主打「只索引标签，不索引内容」：

- 更低的存储成本
- 与 Prometheus/Grafana 原生集成
- 适合云原生环境

## 监控与告警

### Prometheus + Grafana

黄金组合：

- **Prometheus**：指标采集、存储、查询
- **Alertmanager**：告警路由、去重、抑制
- **Grafana**：可视化大盘

###Exporter 生态

| Exporter | 监控对象 |
|---|---|
| node_exporter | 服务器硬件和系统指标 |
| mysqld_exporter | MySQL 数据库 |
| redis_exporter | Redis 缓存 |
| blackbox_exporter | HTTP/TCP/ICMP 探测 |

## 链路追踪

在微服务架构中，一个请求可能跨越十几个服务。如何追踪这条「链路」？

### OpenTelemetry

CNCF 的可观测性标准，统一了 Traces、Metrics、Logs：

```
Trace（请求链路）
├── Span 1（网关）
├── Span 2（认证服务）
├── Span 3（用户服务）
│   ├── Span 3.1（查询数据库）
│   └── Span 3.2（调用缓存）
└── Span 4（返回响应）
```

### Jaeger vs Zipkin

| 特性 | Jaeger | Zipkin |
|---|---|---|
| 厂商 | CNCF 项目 | Twitter 开源 |
| 数据模型 | OpenTracing | OpenZipkin |
| 存储 | Elasticsearch、Cassandra | Elasticsearch、MySQL |
| UI | 原生支持 | 原生支持 |

## SRE 实践

### 核心理念

SRE（Site Reliability Engineering）将软件工程的思维应用到运维：

- **SLO（Service Level Objective）**：定义服务的目标可用性
- **Error Budget**：允许的误差额度
- **Toil Reduction**：减少重复性人工操作

### 告警设计原则

| 原则 | 说明 |
|---|---|
| 关键告警 | P1：立即处理，影响用户 |
| 次要告警 | P2：尽快处理，影响部分用户 |
| 预警 | P3：计划处理，潜在风险 |
| 噪音控制 | 避免告警疲劳 |

## 工具链集成

现代运维很少用单一工具，而是构建工具链：

```
代码提交 → CI/CD → 部署 → 监控 → 告警 → 响应
   │         │        │      │       │      │
  Git    Jenkins  K8s  Prometheus Grafana  PagerDuty
```

### 工具选择考量

| 维度 | 问题 |
|---|---|
| 规模 | 小团队用开源，大规模用商业版 |
| 环境 | 传统架构 vs 云原生 |
| 成本 | 许可费用、运维成本、存储成本 |
| 集成 | 与现有系统的兼容性 |

## 总结

运维工具的选择不是追求最新最全，而是根据实际需求构建合适的体系：

1. **日志**：从 ELK/EFK/Loki 中选择
2. **监控**：Prometheus + Grafana 是事实标准
3. **追踪**：OpenTelemetry + Jaeger
4. **实践**：引入 SRE 理念，定义 SLO

记住：工具只是手段，让系统更稳定、让团队更高效才是目的。
