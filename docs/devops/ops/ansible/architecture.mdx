# Ansible 架构原理

SSH 密码改了就跑不动了？Ansible 怎么做到的——这是很多初学者都会问的问题。

当你第一次运行 Ansible 命令，看着它自动连接服务器、安装软件、同步配置，你可能会好奇：**这背后的原理是什么？**

## Ansible 工作原理

Ansible 的工作流程可以分为四个步骤：

```
┌──────────────┐     1. 读取配置      ┌──────────────┐
│              │ ──────────────────► │              │
│  Ansible     │                      │  Inventory   │
│  Program     │                      │  File        │
│              │ ◄────────────────── │              │
└──────────────┘     2. 返回主机列表  └──────────────┘
       │
       │ 3. SSH 连接 + 推送模块
       ▼
┌──────────────────────────────────────────────────┐
│               Managed Nodes                       │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐          │
│  │ Node 1  │  │ Node 2  │  │ Node 3  │  ...     │
│  └─────────┘  └─────────┘  └─────────┘          │
└──────────────────────────────────────────────────┘
       │
       │ 4. 执行任务，返回结果
       ▼
┌──────────────┐
│  JSON Result │
└──────────────┘
```

### 步骤详解

**第一步：读取 Inventory**

Inventory 文件定义了所有被管理的主机。Ansible 解析这个文件，获取主机列表和分组信息。

**第二步：生成任务列表**

Ansible 根据 Playbook 或命令参数，生成待执行的任务列表，并确定要在哪些主机上执行。

**第三步：连接并推送**

Ansible 通过 SSH 连接到目标主机，然后将需要执行的模块代码（Python）推送到目标主机。

**第四步：执行并返回**

目标主机执行模块代码，将执行结果以 JSON 格式返回给控制节点。控制节点汇总所有结果，展示给用户。

:::tip 关键点
整个过程中，被管理节点**不需要运行任何守护进程**，只需要：
1. Python 环境（用于执行模块）
2. SSH 连接
3. 适当的系统工具（如 yum、apt）

这就是 Ansible 区别于 Puppet、SaltStack 等 Agent 架构的核心优势。
:::

## 控制节点要求

控制节点需要满足以下条件：

| 组件 | 要求 |
|---|---|
| 操作系统 | Linux/macOS/BSD（不支持 Windows） |
| Python | Python 2.7 或 Python 3.5+ |
| SSH | OpenSSH |
| 磁盘空间 | 至少 300MB |

## 被管理节点要求

被管理节点需要满足以下条件：

| 组件 | 要求 |
|---|---|
| SSH | OpenSSH 服务器可访问 |
| Python | Python 2.6+ 或 Python 3.5+ |
| 网络 | 能被控制节点访问 |

:::warning 特例处理
如果被管理节点没有 Python 环境，可以使用 `raw` 模块或安装 `ansible-core` 的 `simulate-python` 功能。但这并非标准做法，生产环境中建议确保所有节点都有 Python 环境。
:::

## Inventory 详解

Inventory 是 Ansible 管理的核心，它定义了被管理的主机和主机变量。

### 静态 Inventory

静态 Inventory 是一个 INI 格式的文件：

```ini title="inventory/production"
# 分组定义
[webservers]
web01.example.com
web02.example.com
web03.example.com

[dbservers]
db01.example.com ansible_host=192.168.1.10
db02.example.com ansible_host=192.168.1.11

[loadbalancers]
lb01.example.com
lb02.example.com

# 默认变量
[all:vars]
ansible_user=deploy
ansible_ssh_private_key_file=~/.ssh/id_rsa
ntp_server=ntp.example.com

# 组嵌套
[datacenter:children]
dbservers
webservers
```

### 主机变量

```ini
[webservers]
web01.example.com ansible_port=2222
web02.example.com ansible_port=2223
```

### 动态 Inventory

静态 Inventory 的缺点是：主机列表变化时需要手动更新。对于云环境或容器编排系统，需要动态 Inventory。

Ansible 支持多种动态 Inventory 插件：

```bash
# 使用 AWS EC2 动态 Inventory
ansible -i ec2.py all -m ping

# 使用 Kubernetes 动态 Inventory
ansible -i k8s.py all -m ping

# 使用自定义脚本
ansible -i inventory_script.py all -m ping
```

动态 Inventory 本质是一个脚本或插件，它根据外部系统（AWS、K8s、CMDB）动态生成主机列表：

```python title="inventory_script.py"
#!/usr/bin/env python
"""动态 Inventory 脚本示例"""
import json

# 脚本需要返回两个 JSON 对象：
# 1. _meta: 包含所有主机的连接信息
# 2. 分组: 包含每个分组的主机列表

inventory = {
    "_meta": {
        "hostvars": {
            "web01": {
                "ansible_host": "192.168.1.10",
                "ansible_user": "admin"
            },
            "web02": {
                "ansible_host": "192.168.1.11",
                "ansible_user": "admin"
            }
        }
    },
    "webservers": {
        "hosts": ["web01", "web02"]
    }
}

print(json.dumps(inventory))
```

### Inventory 参数

| 参数 | 说明 |
|---|---|
| `ansible_host` | SSH 连接地址 |
| `ansible_port` | SSH 端口，默认 22 |
| `ansible_user` | SSH 用户名 |
| `ansible_password` | SSH 密码（不推荐，明文存储） |
| `ansible_ssh_private_key_file` | SSH 私钥路径 |
| `ansible_become` | 是否提权执行 |
| `ansible_become_user` | 提权用户 |
| `ansible_become_password` | 提权密码 |

## Ansible 模块

### 模块工作原理

当 Ansible 执行一个任务时：

1. 控制节点根据模块名找到对应的模块文件
2. 将模块代码和参数打包
3. 通过 SSH 将打包的代码发送到目标主机
4. 目标主机执行模块代码
5. 返回 JSON 结果

```bash
# 查看模块文档
ansible-doc copy

# 查看模块源码位置
ansible-doc -l  # 列出所有模块
```

### 常用模块分类

| 类别 | 模块名 | 用途 |
|---|---|---|
| 文件操作 | `copy`, `template`, `fetch`, `lineinfile` | 文件传输和编辑 |
| 命令执行 | `command`, `shell`, `script`, `raw` | 执行命令 |
| 软件包 | `apt`, `yum`, `dnf`, `pip` | 包管理 |
| 服务管理 | `service`, `systemd` | 启动/停止服务 |
| 系统配置 | `user`, `group`, `cron`, `hostname` | 系统配置 |
| 云服务 | `aws_*`, `gcp_*`, `azure_*` | 云资源管理 |
| 容器 | `docker_*`, `k8s_*` | 容器和 K8s |

### 模块幂等性

Ansible 的核心理念是**幂等性**：无论执行多少次，结果都应该一致。

以 `file` 模块为例：

```bash
# 第一次执行：创建文件
ansible all -m file -a "path=/tmp/test state=touch"

# 第二次执行：文件已存在，模块检测到后不会重复创建
# 返回 changed=false
```

以 `yum` 模块为例：

```bash
# 第一次执行：安装 Nginx
ansible all -b -m yum -a "name=nginx state=present"

# 第二次执行：Nginx 已安装，模块跳过安装步骤
# 返回 changed=false
```

:::tip 理解幂等性
幂等性不是"什么都不做"，而是"只做必要的操作"。

比如你写一个 playbook 确保 Nginx 已安装：
- 第一次运行：Nginx 未安装 → 安装 → changed=true
- 第二次运行：Nginx 已安装 → 检查通过 → changed=false

这意味着你可以**安全地重复运行** playbook，而不用担心重复操作导致问题。
:::

## 连接与认证

### SSH 密钥认证

推荐使用 SSH 公私钥认证：

```bash
# 1. 生成密钥对
ssh-keygen -t rsa -b 4096 -C "ansible@control-node"

# 2. 复制公钥到目标主机
ssh-copy-id user@target-host

# 3. 测试连接
ssh user@target-host
```

### SSH 配置优化

编辑 `~/.ssh/config` 优化连接：

```bash title="~/.ssh/config"
Host *
    # 加速 SSH 连接
    ControlMaster auto
    ControlPath /tmp/ssh-%r@%h:%p
    ControlPersist 10m
    
    # 超时设置
    ServerAliveInterval 60
    ServerAliveCountMax 3
    
    # 安全设置
    StrictHostKeyChecking ask
```

### Ansible 配置优化

编辑 `ansible.cfg` 提升性能：

```ini title="ansible.cfg"
[defaults]
# Inventory 路径
inventory = ./inventory

# SSH 连接数限制（加速并行执行）
forks = 20

# 远程主机环境
interpreter_python = auto_silent

# 事实收集（可关闭加速首次运行）
gathering = smart
fact_caching = jsonfile
fact_caching_connection = /tmp/ansible_facts
fact_caching_timeout = 86400

[ssh_connection]
# SSH 多路复用
pipelining = True
ssh_args = -o ControlMaster=auto -o ControlPersist=60s
```

## 事实变量（Facts）

Facts 是 Ansible 自动收集的主机信息，包括操作系统、IP 地址、内存、磁盘等。

```bash
# 查看主机的 facts
ansible all -m setup

# 过滤 facts
ansible all -m setup -a "filter=ansible_*_mb"

# 查看特定 facts
ansible all -m setup -a "filter=ansible_distribution"
```

在 Playbook 中直接使用：

```yaml
- name: 显示主机信息
  debug:
    msg: |
      主机名: {{ ansible_hostname }}
      操作系统: {{ ansible_distribution }} {{ ansible_distribution_version }}
      IP 地址: {{ ansible_default_ipv4.address }}
      内存: {{ ansible_memtotal_mb }}MB
      CPU: {{ ansible_processor_vcpus }} 核
```

:::tip facts 缓存
默认情况下，每次运行 Ansible 都会重新收集 facts。对于大型环境，可以开启 facts 缓存：

```ini title="ansible.cfg"
[defaults]
gathering = smart
fact_caching = jsonfile
fact_caching_connection = /tmp/ansible_facts
fact_caching_timeout = 3600
```

这样同一主机在一小时内的重复运行会使用缓存的 facts，而不是重新收集。
:::

## 总结

Ansible 的架构核心是**无代理 + SSH + 模块化**：

1. **无代理**：被管理节点无需安装任何软件，降低了部署和维护成本
2. **SSH**：利用现有 SSH 协议，无需额外的通信端口
3. **模块化**：丰富的模块覆盖各种场景，每个模块都有幂等性保证
4. **Inventory**：灵活的主机管理，支持静态和动态两种方式
