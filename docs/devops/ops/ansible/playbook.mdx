# Ansible Playbook 进阶

你已经知道了 Ansible 怎么连接主机、怎么执行单条命令。但真正强大的地方在于 **Playbook**——用 YAML 语法编写的任务剧本。

想象一下，你需要一次性完成这些操作：
1. 安装 Nginx
2. 上传配置文件
3. 启动服务
4. 配置防火墙规则
5. 验证服务状态

手动操作？一个个来？但 Playbook 让你一键搞定。

## Playbook 基础语法

### 最小 Playbook

```yaml title="first-playbook.yml"
---
# 这个 playbook 只做一件事：在所有主机上输出 Hello
- hosts: all                    # 目标主机（来自 Inventory）
  become: yes                   # 使用 sudo 提权
  tasks:                        # 任务列表
    - name: Say Hello           # 任务名称（用于日志输出）
      ansible.builtin.debug:
        msg: "Hello from Ansible!"
```

执行：

```bash
ansible-playbook first-playbook.yml
```

### Playbook 结构

```
Playbook
└── Play (一个 Play)
    ├── hosts: 目标主机
    ├── become: 是否提权
    ├── vars: 变量定义
    ├── tasks: 任务列表
    │   └── Task (每个任务调用一个模块)
    │       ├── name: 任务名称
    │       └── module: 模块及参数
    ├── handlers: 处理器
    └── templates: 模板文件
```

### 多个 Play

一个 Playbook 可以包含多个 Play：

```yaml title="multi-play.yml"
---
# Play 1: 配置 web 服务器
- hosts: webservers
  become: yes
  vars:
    nginx_version: "1.24.0"
  tasks:
    - name: Install Nginx
      ansible.builtin.apt:
        name: nginx
        state: present
        update_cache: yes

    - name: Copy Nginx config
      ansible.builtin.copy:
        src: files/nginx.conf
        dest: /etc/nginx/nginx.conf
      notify: Restart Nginx

    - name: Start Nginx
      ansible.builtin.service:
        name: nginx
        state: started
        enabled: yes

  handlers:
    - name: Restart Nginx
      ansible.builtin.service:
        name: nginx
        state: restarted

# Play 2: 配置数据库服务器
- hosts: dbservers
  become: yes
  tasks:
    - name: Install PostgreSQL
      ansible.builtin.apt:
        name: postgresql
        state: present
```

## 变量使用

### 定义变量

**在 Play 中定义：**

```yaml
vars:
  app_name: myapp
  app_port: 8080
  nginx_version: "1.24.0"
```

**在 Inventory 中定义：**

```ini
[webservers]
web01.example.com http_port=80
web02.example.com http_port=80
```

**在命令行指定：**

```bash
ansible-playbook deploy.yml -e "app_version=1.2.3"
```

**在文件中定义：**

```bash
# vars.yml 文件
app_name: myapp
app_port: 8080

# Playbook 中引用
vars_files:
  - vars.yml
```

### 使用变量

```yaml
- name: Create application directory
  ansible.builtin.file:
    path: "/opt/{{ app_name }}"
    state: directory
    mode: "0755"

- name: Configure application
  ansible.builtin.template:
    src: "app.conf.j2"
    dest: "/etc/{{ app_name }}/app.conf"
```

### Jinja2 过滤器

变量可以通过过滤器进行转换：

```yaml
# 默认值过滤器
msg: "{{ db_host | default('localhost') }}"

# 字符串过滤器
name: "{{ app_name | upper }}"
name: "{{ app_name | lower }}"

# 数字过滤器
count: "{{ count | int + 1 }}"

# 路径过滤器
path: "{{ file_path | basename }}"
dir: "{{ file_path | dirname }}"

# 正则替换
result: "{{ 'hello world' | regex_replace('world', 'ansible') }}"

# JSON/YAML 过滤器
data: "{{ json_string | from_json }}"
```

## Facts 变量

Facts 是 Ansible 自动收集的系统信息：

```yaml
- name: Display system information
  ansible.builtin.debug:
    msg: |
      Hostname: {{ ansible_hostname }}
      OS: {{ ansible_distribution }} {{ ansible_distribution_version }}
      CPU: {{ ansible_processor_vcpus }} cores
      Memory: {{ ansible_memtotal_mb }}MB
      IP: {{ ansible_default_ipv4.address }}
```

常用 Facts：

| 变量 | 说明 |
|---|---|
| `ansible_hostname` | 主机名 |
| `ansible_distribution` | 发行版 |
| `ansible_os_family` | 操作系统家族 |
| `ansible_processor_vcpus` | CPU 核心数 |
| `ansible_memtotal_mb` | 总内存 |
| `ansible_default_ipv4.address` | 默认 IP 地址 |
| `ansible_devices` | 磁盘设备 |
| `ansible_mounts` | 挂载点 |

### 关闭 Facts 收集

如果不需要 Facts，可以关闭以加速执行：

```yaml
- hosts: all
  gather_facts: no
  tasks:
    - name: Quick task
      ansible.builtin.debug:
        msg: "This runs faster without facts collection"
```

## 条件执行

### when 条件

根据变量或 Facts 条件执行任务：

```yaml
tasks:
  - name: Install Apache on Debian
    ansible.builtin.apt:
      name: apache2
      state: present
    when: ansible_os_family == "Debian"

  - name: Install Apache on RedHat
    ansible.builtin.yum:
      name: httpd
      state: present
    when: ansible_os_family == "RedHat"
```

**多条件组合：**

```yaml
tasks:
  - name: Install custom kernel
    ansible.builtin.yum:
      name: kernel-custom
      state: present
    when:
      - ansible_distribution == "CentOS"
      - ansible_distribution_major_version | int >= 7
```

**条件取反：**

```yaml
- name: Skip on Docker containers
  ansible.builtin.debug:
    msg: "This is a regular server"
  when: ansible_virtualization_type != "container"
```

### register 与 when

将命令结果注册为变量，然后在后续任务中判断：

```yaml
tasks:
  - name: Check if config exists
    ansible.builtin.stat:
      path: /etc/myapp/config.conf
    register: config_file

  - name: Backup existing config
    ansible.builtin.copy:
      src: /etc/myapp/config.conf
      dest: /etc/myapp/config.conf.bak
    when: config_file.stat.exists

  - name: Create default config
    ansible.builtin.copy:
      src: default.conf
      dest: /etc/myapp/config.conf
    when: not config_file.stat.exists
```

### failed_when 和 changed_when

自定义任务的成功和变更判断：

```yaml
tasks:
  - name: Run custom script
    ansible.builtin.command: /opt/check_status.sh
    register: script_result
    changed_when: script_result.rc == 0
    failed_when: script_result.rc >= 2

  - name: Report failure
    ansible.builtin.fail:
      msg: "Script returned critical error"
    when: script_result.rc == 2
```

## 循环

### with_items 循环

最简单的循环方式：

```yaml
tasks:
  - name: Create users
    ansible.builtin.user:
      name: "{{ item }}"
      state: present
    with_items:
      - alice
      - bob
      - charlie
```

### loop 列表循环

```yaml
tasks:
  - name: Install packages
    ansible.builtin.apt:
      name: "{{ item }}"
      state: present
    loop:
      - nginx
      - postgresql
      - redis-server
```

### loop 字典循环

```yaml
tasks:
  - name: Configure applications
    ansible.builtin.copy:
      content: "{{ item.config }}"
      dest: "/etc/{{ item.name }}/config.conf"
    loop:
      - { name: "app1", config: "config1 content" }
      - { name: "app2", config: "config2 content" }
```

### with_dict 字典循环

```yaml
vars:
  packages:
    nginx: /etc/nginx
    postgresql: /var/lib/postgresql
    redis: /etc/redis

tasks:
  - name: Create directories
    ansible.builtin.file:
      path: "{{ item.value }}"
      state: directory
      mode: "0755"
    with_dict: "{{ packages }}"
```

### loop_control

控制循环行为：

```yaml
tasks:
  - name: Debug loop
    ansible.builtin.debug:
      msg: "Item {{ item }} ({{ ansible_loop.index }}/{{ ansible_loop.all | length }})"
    loop:
      - alpha
      - beta
      - gamma
    loop_control:
      index_var: my_index    # 自定义索引变量名
      label: "{{ item }}"    # 在日志中显示的标签
```

:::tip loop vs with_items
在 Ansible 2.5+ 中，推荐使用 `loop` 代替 `with_items`。`loop` 更一致，语法更清晰。

```yaml
# 推荐
loop:
  - item1
  - item2

# 不推荐（但仍可用）
with_items:
  - item1
  - item2
```
:::

## Handlers 处理器

Handlers 是特殊的任务，只有在被其他任务"通知"时才执行，通常用于"配置变更后重启服务"。

### 基本用法

```yaml
tasks:
  - name: Copy Nginx configuration
    ansible.builtin.copy:
      src: nginx.conf
      dest: /etc/nginx/nginx.conf
    notify: Restart Nginx        # 通知处理器

  - name: Copy SSL certificate
    ansible.builtin.copy:
      src: server.crt
      dest: /etc/nginx/ssl/server.crt
    notify: Restart Nginx

handlers:
  - name: Restart Nginx
    ansible.builtin.service:
      name: nginx
      state: restarted

  - name: Reload Nginx
    ansible.builtin.service:
      name: nginx
      state: reloaded
```

### Handlers 执行顺序

Handlers 按照定义顺序执行，而非通知顺序：

```yaml
handlers:
  - name: Handler 1
    ansible.builtin.debug:
      msg: "Handler 1"

  - name: Handler 2
    ansible.builtin.debug:
      msg: "Handler 2"
```

### flush_handlers 强制执行

默认情况下，Handlers 在所有任务完成后执行。如果需要中途执行，使用 `flush_handlers`：

```yaml
tasks:
  - name: Deploy application
    ansible.builtin.copy:
      src: app/
      dest: /opt/myapp/
    notify: Restart application

  - name: Run database migrations
    ansible.builtin.command: /opt/myapp/migrate.sh
    register: migration_result

  - name: Force handlers execution
    ansible.builtin.meta: flush_handlers

  - name: Verify application
    ansible.builtin.uri:
      url: http://localhost:8080/health
    when: migration_result is succeeded
```

### handlers 文件组织

大型 Playbook 中，可以将 Handlers 放在单独文件中：

```yaml
# main.yml
- hosts: webservers
  handlers:
    - import_tasks: handlers/main.yml

# handlers/main.yml
- name: Restart Nginx
  ansible.builtin.service:
    name: nginx
    state: restarted

- name: Restart application
  ansible.builtin.systemd:
    name: myapp
    state: restarted
```

## 常用模块详解

### copy 模块

文件复制：

```yaml
# 复制本地文件到远程
- name: Copy configuration file
  ansible.builtin.copy:
    src: ./config/app.conf
    dest: /etc/myapp/app.conf
    owner: appuser
    group: appuser
    mode: "0640"
    backup: yes          # 备份原文件

# 直接写入内容
- name: Create a configuration
  ansible.builtin.copy:
    dest: /tmp/config.txt
    content: |
      key1=value1
      key2=value2
```

### template 模块

模板渲染（支持变量）：

```yaml
# templates/nginx.conf.j2
server {
    listen {{ nginx_port }};
    server_name {{ server_name }};
    
    location / {
        proxy_pass http://{{ backend_host }}:{{ backend_port }};
    }
}

# Playbook 中使用
- name: Render Nginx configuration
  ansible.builtin.template:
    src: nginx.conf.j2
    dest: /etc/nginx/sites-available/myapp.conf
    owner: root
    group: root
    mode: "0644"
  notify: Reload Nginx
```

### command 和 shell 模块

执行命令：

```yaml
# command 模块（不经过 shell，无法使用管道等）
- name: Check disk space
  ansible.builtin.command: df -h
  register: disk_usage

- name: Execute with environment variables
  ansible.builtin.command: /opt/script.sh
  environment:
    APP_ENV: production
    LOG_LEVEL: debug

# shell 模块（经过 shell，支持管道等）
- name: Check running processes
  ansible.builtin.shell: |
    ps aux | grep nginx | grep -v grep
  register: nginx_processes

- name: Chain commands
  ansible.builtin.shell: |
    cd /opt/myapp &&
    ./build.sh &&
    ./deploy.sh
  args:
    executable: /bin/bash
```

:::warning command vs shell
| 模块 | 优点 | 缺点 |
|---|---|---|
| `command` | 更安全，不会被 Shell 注入攻击 | 无法使用管道、重定向等 |
| `shell` | 功能完整 | 安全性略低 |
:::
:::

### service/systemd 模块

服务管理：

```yaml
- name: Ensure Nginx is running
  ansible.builtin.service:
    name: nginx
    state: started
    enabled: yes

# systemd 模块更强大
- name: Manage application service
  ansible.builtin.systemd:
    name: myapp
    state: restarted
    enabled: yes
    daemon_reload: yes      # 重新加载 systemd 配置
    scope: system
    environment:
      APP_HOME: /opt/myapp
```

### apt/yum 模块

包管理：

```yaml
# Debian/Ubuntu
- name: Install packages
  ansible.builtin.apt:
    name:
      - nginx
      - postgresql
      - redis-server
    state: present
    update_cache: yes
    cache_valid_time: 3600

# RHEL/CentOS
- name: Install packages
  ansible.builtin.yum:
    name:
      - nginx
      - postgresql
      - redis-server
    state: present
    enablerepo: "epel"

# 安装特定版本
- name: Install specific version
  ansible.builtin.apt:
    name: nginx=1.18.0
    state: present
```

### lineinfile 模块

修改文件中的特定行：

```yaml
# 确保文件包含某行
- name: Add entry to hosts file
  ansible.builtin.lineinfile:
    path: /etc/hosts
    line: "192.168.1.100 app-server"
    state: present

# 替换匹配的行
- name: Set max connections
  ansible.builtin.lineinfile:
    path: /etc/nginx/nginx.conf
    regexp: '^worker_connections'
    line: "worker_connections 2048;"

# 删除匹配的行
- name: Remove debug line
  ansible.builtin.lineinfile:
    path: /etc/myapp/config.conf
    regexp: '^debug='
    state: absent
```

### block/rescue

错误处理：

```yaml
tasks:
  - name: Attempt database migration
    block:
      - name: Run migration script
        ansible.builtin.command: /opt/myapp/migrate.sh
        register: migration

      - name: Verify migration
        ansible.builtin.debug:
          msg: "Migration completed successfully"
    rescue:
      - name: Rollback on failure
        ansible.builtin.debug:
          msg: "Migration failed, rollback triggered"

      - name: Notify administrators
        ansible.builtin.mail:
          to: admin@example.com
          subject: "Migration Failed"
          body: "The database migration failed on {{ inventory_hostname }}"
    always:
      - name: Cleanup temporary files
        ansible.builtin.file:
          path: /tmp/migration_temp
          state: absent
```

## Playbook 最佳实践

### 目录结构

```
production/
├── ansible.cfg
├── inventory/
│   ├── hosts
│   └── group_vars/
│       └── all.yml
├── playbooks/
│   ├── site.yml
│   ├── webservers.yml
│   └── dbservers.yml
├── roles/
│   ├── common/
│   ├── nginx/
│   └── postgresql/
└── vars/
    └── secrets.yml
```

### ansible.cfg 配置

```ini title="ansible.cfg"
[defaults]
inventory = inventory/hosts
roles_path = roles
host_key_checking = False
retry_files_enabled = False
gathering = smart
fact_caching = jsonfile
fact_caching_connection = /tmp/ansible_facts

[privilege_escalation]
become = True
become_method = sudo
become_user = root
become_ask_pass = False

[ssh_connection]
pipelining = True
ssh_args = -o ControlMaster=auto -o ControlPersist=60s
```

### 敏感信息处理

使用 Ansible Vault 加密敏感数据：

```bash
# 创建加密文件
ansible-vault create vars/secrets.yml

# 编辑加密文件
ansible-vault edit vars/secrets.yml

# 加密已有文件
ansible-vault encrypt vars/secrets.yml

# 运行 playbook 时输入密码
ansible-playbook site.yml --ask-vault-pass

# 或使用密码文件
ansible-playbook site.yml --vault-password-file ~/.vault_pass
```

### 测试和调试

```bash
# 语法检查
ansible-playbook site.yml --syntax-check

# 列出所有任务（不执行）
ansible-playbook site.yml --list-tasks

# 列出所有主机（不执行）
ansible-playbook site.yml --list-hosts

# 模拟执行（检查可能的问题）
ansible-playbook site.yml --check

# 逐步执行（调试用）
ansible-playbook site.yml --step

# 启动日志（调试用）
ANSIBLE_DEBUG=1 ansible-playbook site.yml
```

## 总结

Playbook 是 Ansible 的核心，掌握以下要点：

1. **YAML 基础**：理解缩进、列表、字典的写法
2. **变量与 Facts**：灵活使用变量和自动收集的系统信息
3. **条件与循环**：用 `when` 和 `loop` 处理复杂场景
4. **Handlers**：用 notify/handler 机制处理配置变更后的重启
5. **模块选择**：根据场景选择合适的模块（copy、template、command 等）

学完 Playbook，下一步就是用 **Role** 组织代码，让 Playbook 可复用、易维护。
