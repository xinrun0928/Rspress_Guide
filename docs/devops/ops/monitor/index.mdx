# 监控体系总览

凌晨 2 点，你的手机响了。告警短信显示：生产环境的 CPU 使用率飙升到 95%，响应时间从 50ms 暴增到 5 秒。用户开始抱怨，但你还不知道发生了什么。

**这就是没有监控的代价。**

监控不是为了"出事之后才知道"，而是为了**提前发现问题、在问题恶化前介入、提供故障排查的线索**。

## 监控的重要性

### 为什么需要监控

| 维度 | 没有监控 | 有监控 |
|---|---|---|
| **故障发现** | 用户投诉后才知道 | 告警第一时间通知 |
| **故障定位** | 大量猜测，逐台排查 | 快速定位异常组件 |
| **故障恢复** | 手动逐台修复 | 自动化恢复或快速回滚 |
| **容量规划** | 拍脑袋决定 | 基于数据的科学规划 |
| **性能优化** | 凭感觉优化 | 量化分析找到瓶颈 |

### 监控的黄金三指标

Google SRE 提出的四个黄金信号：

| 信号 | 含义 | 监控重点 |
|---|---|---|
| **延迟（Latency）** | 请求处理时间 | P50、P95、P99 延迟 |
| **流量（Traffic）** | 系统负载 | QPS、并发连接数 |
| **错误（Errors）** | 请求失败率 | 5xx 错误率、异常率 |
| **饱和度（Saturation）** | 资源使用程度 | CPU、内存、磁盘、队列长度 |

## 常见监控方案对比

| 方案 | 特点 | 适用场景 |
|---|---|---|
| **Prometheus + Grafana** | 云原生、功能强大、社区活跃 | 云原生环境、K8s |
| **Zabbix** | 老牌监控、功能全面 | 传统 IT 基础设施 |
| **Datadog** | SaaS 化、体验好 | 混合云、多云环境 |
| **Nagios** | 简单、灵活 | 小规模监控需求 |
| **云厂商监控** | 开箱即用、集成好 | 云原生应用 |

## Prometheus 生态系统

### 核心组件

| 组件 | 功能 |
|---|---|
| **Prometheus Server** | 指标抓取、存储、查询 |
| **AlertManager** | 告警管理（分组、抑制、静默） |
| **PushGateway** | 接收短生命周期任务推送 |
| **Exporters** | 暴露各类指标的代理 |
| **Grafana** | 可视化和告警 |
| **PromQL** | Prometheus 查询语言 |

### 指标类型

| 类型 | 说明 | 典型场景 |
|---|---|---|
| **Counter** | 计数器，只增不减 | 请求总数、错误总数 |
| **Gauge** | 仪表，可增可减 | CPU 使用率、内存使用量 |
| **Histogram** | 直方图，分布统计 | 请求延迟分布 |
| **Summary** | 摘要，分位数统计 | 请求延迟分位数 |
