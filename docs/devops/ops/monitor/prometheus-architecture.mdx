# Prometheus 架构原理

Prometheus 是怎么工作的？

一个监控系统，最核心的问题有三个：**数据怎么来、数据怎么存、数据怎么查**。理解了 Prometheus 的架构，这三个问题就有了答案。

```

                    └───────┼──────────────┼──────────────┼─────────────┼─────┘
```

## 核心组件

### 1. Scrape Retrieval（抓取模块）

**职责**：定时从监控目标抓取指标数据。

```yaml title="prometheus.yml"
global:
  scrape_interval: 15s      # 抓取间隔
  evaluation_interval: 15s  # 规则评估间隔

scrape_configs:
  - job_name: 'prometheus'
    static_configs:
      - targets: ['localhost:9090']
  
  - job_name: 'node'
    static_configs:
      - targets: ['node-exporter:9100']
```

**工作原理**：

```
┌────────────┐     HTTP GET /metrics      ┌────────────┐
│Prometheus  │ ──────────────────────────►│  Exporter  │
│            │ ◄──────────────────────────│            │
│            │    200 OK + metrics data   │            │
└────────────┘                             └────────────┘
```

1. Prometheus 定时向 Exporter 发送 HTTP GET 请求
2. Exporter 返回 `200 OK`，响应体是指标数据
3. Prometheus 解析响应，存储到 TSDB

### 2. TSDB（时序数据库）

**职责**：高效存储和查询时序数据。

**存储结构**：

```
┌─────────────────────────────────────────────────────┐
│                     Block                            │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐             │
│  │Head     │  │ Block 1 │  │ Block 2 │             │
│  │(最新数据)│  │(2h数据) │  │(2h数据) │             │
│  └─────────┘  └─────────┘  └─────────┘             │
└─────────────────────────────────────────────────────┘
```

**特点**：
- 基于 Google Tacot 论文实现
- 数据按时间分块（默认为 2 小时一块）
- 支持压缩（合并多个块）
- 只支持追加，不支持更新

### 3. HTTP Server（查询服务）

**职责**：提供 PromQL 查询接口。

```bash
# 通过 HTTP API 查询
curl http://localhost:9090/api/v1/query?query='up'

# Grafana 通过这个接口获取数据绘图
```

**PromQL 引擎**：
- 解析查询语句
- 生成执行计划
- 从 TSDB 读取数据
- 聚合、计算、返回结果

### 4. AlertManager（告警管理）

**职责**：处理 Prometheus 发送的告警，进行分组、抑制、静默、路由。

```
Prometheus ──(告警)──► AlertManager ──(通知)──► Email/Slack/钉钉/企业微信
```

**告警流程**：

```yaml title="prometheus.yml"
alerting:
  alertmanagers:
    - static_configs:
        - targets: ['alertmanager:9093']

rule_files:
  - "alerts/*.yml"
```

## AlertManager 详解

### 告警分组

将相似的告警合并成一条通知，避免告警风暴：

```yaml title="alertmanager.yml"
route:
  group_by: ['alertname', 'cluster']
  group_wait: 30s          # 等待时间，收集同组告警
  group_interval: 5m        # 发送下一组通知的间隔
  repeat_interval: 4h       # 重复发送相同告警的间隔
  receiver: 'team-ops'
  
  routes:
    - match:
        severity: critical
      receiver: 'team-critical'
      continue: true
    
    - match:
        service: database
      receiver: 'team-dba'
```

### 告警抑制

阻止某些告警触发，防止告警泛滥：

```yaml
inhibit_rules:
  # 当 instance 宕机时，抑制该实例的所有其他告警
  - source_match:
      alertname: 'InstanceDown'
    target_match_re:
      severity: '.*'
    equal: ['instance']
```

### 告警静默

临时阻止某些告警发送：

```bash
# 通过 API 创建静默规则
curl -X POST http://alertmanager:9093/api/v2/silences \
  -H "Content-Type: application/json" \
  -d '{
    "matchers": [{"name":"severity","value":"critical"}],
    "startsAt": "2024-01-01T00:00:00Z",
    "endsAt": "2024-01-02T00:00:00Z",
    "createdBy": "admin",
    "comment": "Maintenance window"
  }'
```

### 告警路由

将告警路由到不同的接收者：

```yaml
route:
  routes:
    - match:
        severity: critical
      receiver: 'pagerduty'
    
    - match:
        severity: warning
      receiver: 'slack-notifications'
    
    - match:
        severity: info
      receiver: 'email-digest'
```

## PushGateway

### 为什么需要 PushGateway

正常情况下，Prometheus 采用 **Pull 模式**（拉取模式）：

```
Prometheus ──► 拉取 ◄── Exporter
```

但有些任务**不适合被拉取**：

- 短生命周期任务：任务启动后几秒就结束，来不及被拉取
- 批处理任务：Jenkins 定时构建任务
- 一次性任务：数据库迁移脚本

### PushGateway 的作用

```
Job ──(推送指标)──► PushGateway ◄──(拉取)── Prometheus
```

```bash
# 安装 PushGateway
docker run -d -p 9091:9091 prom/pushgateway

# 推送指标
echo "batch_job_duration_seconds 30.5" | curl --data-binary @- http://pushgateway:9091/metrics/job/batch_job

# 或者使用 SDK
```

### 配置 Prometheus 拉取 PushGateway

```yaml
scrape_configs:
  - job_name: 'pushgateway'
    static_configs:
      - targets: ['pushgateway:9091']
    bearer_token_file: /var/run/secrets/pushgateway-token
```

:::warning PushGateway 使用注意
PushGateway 不是万能的：
- 所有指标会持久化在 PushGateway 中
- 增加了监控的耦合
- 没有了服务发现的优势

建议只在**无法 Pull 的场景**下使用 PushGateway。
:::

## Remote Read/Write

### 为什么需要 Remote

Prometheus 默认只保留 15 天的数据。对于需要长期存储的场景，需要 Remote Storage。

```
Prometheus ◄──(Remote Read)──► 远程存储
     │
     └──(Remote Write)──────► 远程存储
```

### 配置 Remote Write

```yaml title="prometheus.yml"
remote_write:
  - url: "http://thanos-receiver:19291/api/v1/receive"
    name: "thanos"
    queue_config:
      capacity: 10000
      max_shards: 5
      min_shards: 1
      max_samples_per_send: 2000
    metadata_config:
      send: true
      send_interval: 1m
```

### 配置 Remote Read

```yaml
remote_read:
  - url: "http://thanos-store:10901/api/v1/read"
    name: "thanos"
    remote_timeout: 30s
    filter_children: false
```

### 常用远程存储方案

| 方案 | 特点 |
|---|---|
| **Thanos** | 原生支持 Prometheus，多租户，长期存储 |
| **Cortex** | 多租户，水平扩展，AWS 托管 |
| **M3** | Uber 开源，高性能 |
| **InfluxDB** | 通用时序数据库 |
| **VictoriaMetrics** | 轻量级，高压缩率 |

## Federation（联邦）

### 什么是 Federation

Federation 允许 Prometheus 从其他 Prometheus 获取数据：

```
Global Prometheus ──(抓取)──► Regional Prometheus
                                   │
                                   ▼
                              ┌─────────┐
                              │ Metrics │
                              └─────────┘
```

### 联邦配置

**Regional Prometheus（抓取本地指标）：**

```yaml
scrape_configs:
  - job_name: 'federate'
    metrics_path: '/federate'
    params:
      'match[]':
        - '{job="node"}'
        - '{job="mysql"}'
    static_configs:
      - targets: ['global-prometheus:9090']
```

**Global Prometheus（聚合所有区域）：**

```yaml
scrape_configs:
  - job_name: 'federate'
    honor_labels: true
    metrics_path: '/federate'
    params:
      'match[]':
        - '{__name__=~"node_.*"}'
    static_configs:
      - targets: 
        - 'us-east-prometheus:9090'
        - 'us-west-prometheus:9090'
        - 'eu-prometheus:9090'
```

### Federation 架构图

```
┌─────────────────────────────────────────────────────────────────┐
│                     Global Prometheus                            │
│  聚合所有 Region 数据                                             │
│  用于全局 Dashboard 和告警                                        │
└─────────────────────────────────────────────────────────────────┘
       │                   │                   │
       ▼                   ▼                   ▼
┌─────────────┐      ┌─────────────┐      ┌─────────────┐
│ US East     │      │ US West     │      │ EU Region   │
│ Prometheus  │      │ Prometheus  │      │ Prometheus  │
│ (抓取本地)  │      │ (抓取本地)  │      │ (抓取本地)  │
└─────────────┘      └─────────────┘      └─────────────┘
       │                   │                   │
       ▼                   ▼                   ▼
   本地服务             本地服务             本地服务
   (Node, MySQL,       (Node, MySQL,       (Node, MySQL,
    Redis, Nginx)       Redis, Nginx)       Redis, Nginx)
```

:::tip Federation vs Thanos
Federation 是 Prometheus 原生的聚合方案，适合：
- 简单的多集群聚合
- 不需要跨集群查询

Thanos 是更强大的方案，适合：
- 统一的全局视图
- 跨集群查询
- 无限历史数据存储
- 高可用部署
:::

## 数据存储原理

### 写入流程

```
HTTP 请求
    │
    ▼
┌─────────┐
│  解析   │ ───► 验证标签、格式
└────┬────┘
     │
     ▼
┌─────────┐
│  Head  │ ◄── 最新数据写入 Head Block
└────┬────┘
     │
     ▼
┌─────────┐
│  WAL   │ ◄── 预写日志，防止数据丢失
└─────────┘
```

### 查询流程

```
PromQL 查询
    │
    ▼
┌─────────┐
│ 解析器  │ ◄── 解析查询语法
└────┬────┘
     │
     ▼
┌─────────┐
│ 索引   │ ◄── 从 TSDB 索引获取匹配的时间序列
└────┬────┘
     │
     ▼
┌─────────┐
│ 合并   │ ◄── 合并多个 Block 的数据
└────┬────┘
     │
     ▼
┌─────────┐
│ 返回   │ ◄── 应用聚合函数返回结果
└─────────┘
```

## 高可用部署

### 简单高可用

两个 Prometheus 实例抓取相同目标：

```
┌─────────────┐      ┌─────────────┐
│Prometheus-1│      │Prometheus-2│
│ (HA)       │      │ (HA)       │
└──────┬──────┘      └──────┬──────┘
       │                    │
       └────────┬───────────┘
                ▼
         ┌────────────┐
         │ AlertManager│
         │ (单实例)   │
         └────────────┘
```

### Thanos Sidecar 方案

每个 Prometheus 部署一个 Thanos Sidecar：

```
┌───────────────────────────────────────────────────────────────┐
│                        Thanos Query                           │
│              (统一查询入口，支持跨 Prometheus 查询)             │
└─────────────────────┬───────────────────────────────────────┘
                       │                        │
        ┌──────────────┴──────────────┐       │
        │                              │       │
        ▼                              ▼       ▼
┌───────────────┐              ┌───────────────┐
│Prometheus +  │              │Prometheus +  │
│Thanos Sidecar│              │Thanos Sidecar│
└───────┬───────┘              └───────┬───────┘
        │                              │
        ▼                              ▼
   ┌─────────┐                    ┌─────────┐
   │Object   │                    │Object   │
   │Storage  │                    │Storage  │
   └─────────┘                    └─────────┘
```

## 总结

Prometheus 架构的核心：

1. **Pull 模式**：Prometheus 主动抓取 Exporter 的指标
2. **TSDB**：时序数据库，高效存储和查询时间序列数据
3. **服务发现**：自动发现监控目标，无需手动配置
4. **AlertManager**：专业的告警处理，支持分组、抑制、静默
5. **Remote**：支持远程存储和联邦聚合
6. **高可用**：通过 Thanos 或 Federation 实现高可用部署

理解了这个架构，你就能更好地设计监控方案和排查问题。
