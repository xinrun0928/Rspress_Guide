# Prometheus 数据模型

你知道 Prometheus 的指标数据长什么样吗？

```
http_requests_total{method="GET",status="200",path="/api/users"} 1024
```

这一行数据里，藏着你理解 Prometheus 的所有秘密。

## 数据格式

Prometheus 采用 **时序数据库** 存储数据，每条数据包含：

1. **指标名称**（Metric Name）：描述测量的内容
2. **标签**（Labels）：键值对，用于区分不同的维度
3. **样本值**（Value）：具体数值
4. **时间戳**（Timestamp）：采样时间（可选，Prometheus 自动生成）

```
指标名称{标签1="值1", 标签2="值2"} 样本值
```

### 示例解析

```promql
# 含义：GET 方法访问 /api/users 路径的请求，HTTP 状态码为 200，总共 1024 次
http_requests_total{method="GET", status="200", path="/api/users"} 1024

# 含义：CPU 用户态使用率为 45.2%
node_cpu_usage{instance="server-01", mode="user"} 45.2

# 含义：内存已使用 12.5GB
node_memory_usable_bytes{instance="server-01"} 1.342e+10
```

## 指标命名规范

好的指标命名能让查询更清晰：

```
[namespace_][metric_name][_total|_seconds|_bytes][_total]
```

| 规范 | 正确 | 错误 |
|---|---|---|
| 使用下划线分隔 | `http_requests_total` | `httpRequestTotal` |
| 指标名包含命名空间 | `nginx_requests_total` | `requests_total` |
| 计数器以 `_total` 结尾 | `errors_total` | `errors` |
| 延迟使用 `_seconds` 后缀 | `request_duration_seconds` | `request_duration_ms` |
| 字节使用 `_bytes` 后缀 | `network_bytes_total` | `network_bytes` |
| 小写字母 | `http_requests` | `HTTP_Requests` |

## 四种指标类型

### Counter（计数器）

**特点**：只增不减的累积值，用于统计"发生了多少次"。

```promql
# 定义
http_requests_total{method="GET", path="/api"} 1024
http_requests_total{method="POST", path="/api"} 256

# 应用场景
- 请求总数
- 错误总数
- 完成的任务数
```

**使用注意**：
- Counter 值只会增加，不会减少（重启后重置）
- 计算速率时使用 `rate()` 函数
- 适合累加型的指标

```promql
# 计算每秒请求速率
rate(http_requests_total[5m])

# 计算 5 分钟内的错误率
rate(http_requests_total{status=~"5.."}[5m]) / rate(http_requests_total[5m])
```

:::tip 场景举例
电商网站的订单总数：每下一单，计数器加一。无论用户查询还是展示，给出的都是**累计值**。配合 `rate()` 可以计算**下单速率**（每秒多少单）。
:::

### Gauge（仪表）

**特点**：可增可减的当前值，用于表示"当前是多少"。

```promql
# 定义
node_cpu_usage{mode="user"} 45.2
node_cpu_usage{mode="system"} 12.8
node_memory_used_bytes 1.342e+10
node_memory_available_bytes 5.678e+9

# 应用场景
- CPU 使用率
- 内存使用量
- 磁盘使用率
- 温度
- 队列长度
```

**使用注意**：
- Gauge 值可以上升也可以下降
- 适合表示瞬时状态的指标
- 可以直接绘图查看趋势

```promql
# 当前 CPU 使用率
node_cpu_usage

# 内存使用率
node_memory_used_bytes / node_memory_total_bytes * 100

# 磁盘使用率
(node_filesystem_size_bytes - node_filesystem_avail_bytes) / node_filesystem_size_bytes * 100
```

:::tip 场景举例
服务器的 CPU 使用率：可能一会儿 10%，一会儿 80%，一会儿又降到 20%。这是典型的 Gauge 指标，因为它反映的是**瞬时状态**。
:::

### Histogram（直方图）

**特点**：将指标值分布到多个桶中，用于统计**延迟、响应大小等分布**。

```promql
# Histogram 实际存储的指标
# _bucket{le="X"} 表示 <=X 的样本数量
# _sum 表示所有样本值的总和
# _count 表示样本总数

http_request_duration_seconds_bucket{le="0.1"} 1024
http_request_duration_seconds_bucket{le="0.5"} 2048
http_request_duration_seconds_bucket{le="1.0"} 4096
http_request_duration_seconds_bucket{le="+Inf"} 5000
http_request_duration_seconds_sum 2500.5
http_request_duration_seconds_count 5000
```

**桶的配置**：

Histogram 默认的桶边界是：

```go
// prometheus/client_golang 默认 buckets
{0.005, 0.01, 0.025, 0.05, 0.1, 0.25, 0.5, 1, 2.5, 5, 10}
```

也可以自定义：

```go
httpRequestDuration = promauto.NewHistogram(prometheus.HistogramOpts{
    Name:    "http_request_duration_seconds",
    Help:    "HTTP request latency in seconds",
    Buckets: []float64{0.001, 0.005, 0.01, 0.025, 0.05, 0.1, 0.25, 0.5, 1, 2.5, 5, 10},
})
```

**使用注意**：
- `le` 表示 "less than or equal"（小于等于）
- `_bucket{le="+Inf"}` 等于 `_count`
- `_sum / _count` 可以计算平均值

```promql
# 计算 P95 延迟（使用 histogram_quantile）
histogram_quantile(0.95, rate(http_request_duration_seconds_bucket[5m]))

# 计算平均延迟
rate(http_request_duration_seconds_sum[5m]) / rate(http_request_duration_seconds_count[5m])
```

:::tip 场景举例
HTTP 请求延迟分布：我们想知道 95% 的请求在多少毫秒内完成。Histogram 将请求按延迟分布到不同桶中，配合 `histogram_quantile()` 可以计算出**任意百分位**的延迟。
:::

### Summary（摘要）

**特点**：在客户端直接计算分位数，**无需在服务端聚合**。

```promql
# Summary 实际存储的指标
# _quantile{q="X"} 表示 X 分位数的值
http_request_duration_seconds{quantile="0.5"} 0.05
http_request_duration_seconds{quantile="0.9"} 0.12
http_request_duration_seconds{quantile="0.99"} 0.25
http_request_duration_sum 2500.5
http_request_duration_count 5000
```

**与 Histogram 的区别**：

| 特性 | Histogram | Summary |
|---|---|---|
| 计算位置 | Prometheus 服务端计算 | 客户端预先计算 |
| 分位数精度 | 取决于桶的配置 | 精确分位数 |
| 服务端资源消耗 | 较高（需要聚合） | 低 |
| 支持聚合 | 支持 | 不支持 |
| 适用场景 | 需要聚合的分析 | 已知分位数的监控 |

```go
// Summary 示例
httpRequestDuration = promauto.NewSummary(prometheus.SummaryOpts{
    Name:       "http_request_duration_seconds",
    Help:       "HTTP request latency in seconds",
    Objectives: map[float64]float64{0.5: 0.05, 0.9: 0.01, 0.99: 0.001},
})
```

:::warning Summary 不能聚合
Summary 的分位数是客户端直接计算的，如果你在多个实例上有 Summary 指标，不能直接对它们求平均或聚合。必须使用 Histogram 才能在服务端聚合。

```promql
# Summary：不能对多个实例的 quantile 求平均（错误的做法）
# avg(http_request_duration_seconds{quantile="0.95"})  ← 这没有意义

# Histogram：可以正确聚合
# histogram_quantile(0.95, avg by (le) (rate(http_request_duration_seconds_bucket[5m])))
```
:::

## 四种指标类型对比

| 类型 | 可增减 | 用途 | 典型场景 |
|---|---|---|---|
| **Counter** | 只增 | 计数统计 | 请求数、错误数 |
| **Gauge** | 可增可减 | 瞬时值 | CPU、内存、温度 |
| **Histogram** | 可增减 | 分布统计 | 延迟、大小分布 |
| **Summary** | 可增减 | 精确分位 | 已知分位监控 |

## 如何选择指标类型

### 判断流程

```
问自己：这个指标测的是什么？

1. 是"发生了多少次"？
   → Counter

2. 是"当前是多少"？
   → Gauge

3. 是"耗时分布"？
   → Histogram 或 Summary
   
   是需要聚合的吗？
   → 是 → Histogram
   → 否 → Summary

4. 其他？
   → Gauge
```

### 实际选择示例

| 指标 | 选择 | 原因 |
|---|---|---|
| HTTP 请求总数 | Counter | 只增不减 |
| 当前在线用户数 | Gauge | 可增可减 |
| HTTP 请求延迟 | Histogram | 需要服务端聚合 |
| PHP 请求延迟 | Summary | 不需要聚合 |
| 当前 CPU 使用率 | Gauge | 瞬时值 |
| Kafka lag | Gauge | 可增可减 |
| JVM GC 次数 | Counter | 只增不减 |
| 请求体大小 | Histogram | 分布统计 |

## 标签设计

### 好的标签设计

```promql
# 好的：标签清晰、维度丰富
http_requests_total{method="GET", status="200", path="/api/users", instance="server-01"}

# 不好：标签太少，无法区分
http_requests_total{instance="server-01"}
```

### 避免标签过多

```promql
# 不好：cardinality 过高（每个值都不同）
user_id="123456789"  # 避免用高基数标签

# 不好：URL 参数作为标签
path="/api/users?id=123"  # 每个用户都不同
```

### 常见标签

| 标签 | 说明 | 示例 |
|---|---|---|
| `instance` | 主机实例 | `server-01:9100` |
| `job` | 作业/服务名 | `nginx`, `mysql` |
| `method` | HTTP 方法 | `GET`, `POST` |
| `status` | 状态码 | `200`, `500` |
| `path` | 请求路径 | `/api/users` |
| `region` | 区域 | `us-east-1` |
| `env` | 环境 | `prod`, `staging` |

## 时间序列

### 时间序列的组成

```
http_requests_total{method="GET", path="/api", instance="server-01"}

     ↑指标名称↑            ↑标签↑
```

一个时间序列由**指标名称 + 标签集合**唯一确定。

### 同一指标的不同实例

```promql
# 三台服务器，每台都有 http_requests_total 指标
http_requests_total{instance="server-01"}
http_requests_total{instance="server-02"}
http_requests_total{instance="server-03"}
```

### 时间序列的数量

**Cardinality**（基数）指的是唯一的时间序列数量：

```promql
# 假设：
# - 10 台服务器
# - 5 种 HTTP 方法
# - 3 种状态码
# - 20 个 API 路径

# 时间序列数 = 10 × 5 × 3 × 20 = 3000
```

:::warning 高基数问题
时间序列数量过大会导致：
- Prometheus 内存消耗增加
- 查询速度变慢
- Grafana 渲染卡顿

避免将 `user_id`、`session_id` 等高基数字段作为标签。
:::

## 总结

Prometheus 数据模型的核心：

1. **指标名称 + 标签** 唯一确定一条时间序列
2. **Counter**：只增不减，统计"发生了多少次"，配合 `rate()` 使用
3. **Gauge**：可增可减，表示"当前是多少"
4. **Histogram**：服务端聚合，计算延迟分布，**支持聚合**
5. **Summary**：客户端计算，精确分位，**不支持聚合**

理解了这四种指标类型，你就能正确地设计和使用 Prometheus 监控。
