# Swarm 负载均衡

当外部请求进入 Swarm 集群时，它是如何找到对应服务的？流量如何在多个容器副本之间分配？

理解 Swarm 的负载均衡机制，是设计高可用服务架构的基础。

## Swarm 负载均衡架构

Swarm 提供了三层负载均衡：

```
┌─────────────────────────────────────────────────────────┐
│                   Layer 1: External LB                  │
│              (Nginx, HAProxy, 云负载均衡器)               │
└─────────────────────────┬───────────────────────────────┘
                          │
┌─────────────────────────▼───────────────────────────────┐
│                 Layer 2: Routing Mesh                     │
│         Ingress Network (IPVS/LVS)                       │
│    ┌─────────────────────────────────────────────┐       │
│    │  任意节点 80 端口 → 目标服务的所有容器        │       │
│    └─────────────────────────────────────────────┘       │
└─────────────────────────┬───────────────────────────────┘
                          │
┌─────────────────────────▼───────────────────────────────┐
│                  Layer 3: Service LB                      │
│            VIP vs DNSRR (服务内部负载均衡)                 │
└─────────────────────────────────────────────────────────┘
```

## Routing Mesh（路由网格）

Routing Mesh 是 Swarm 的核心负载均衡功能。

### 端口发布机制

```yaml
services:
  web:
    image: nginx:alpine
    ports:
      - "80:80"
```

当服务发布端口时：

1. **端口在所有节点发布**：每个节点都监听 80 端口
2. **任意节点可接收请求**：发送到任意节点的 80 端口都能路由到服务
3. **IPVS 负载均衡**：Ingress 网络使用 IPVS 在容器间分配流量

```
客户端
    │
    │ 访问 node-1:80
    ▼
┌────────────────────────────────────────┐
│          Ingress Network               │
│          (IPVS Load Balancer)          │
├────────────────────────────────────────┤
│                                        │
│  node-1:80 ──┬──▶ Container A (node-1) │
│              ├──▶ Container B (node-2) │
│              └──▶ Container C (node-3) │
└────────────────────────────────────────┘
```

### published_port 模式

```yaml
# 默认模式：published_port 在所有节点绑定
services:
  web:
    ports:
      - "80:80"           # 所有节点 80

# 精确模式：只绑定到指定端口
services:
  web:
    ports:
      - target: 80
        published: 80     # 所有节点 80

# VIP 模式
services:
  web:
    ports:
      - target: 80
        published: 80
        protocol: tcp
        mode: ingress    # ingress 模式
```

## 服务发现

Swarm 使用内置 DNS 实现服务发现。

### DNS 解析流程

```
服务名: myapi
    │
    ▼
容器内的 /etc/resolv.conf → 127.0.0.11 (Docker DNS)
    │
    ▼
Swarm DNS → 查询服务 VIP
    │
    ▼
返回 VIP 或容器 IP
```

### DNS 自动注册

当服务创建时，Swarm 自动注册 DNS 记录：

```bash
# 创建服务
docker service create --name api nginx:alpine

# 从其他容器访问
# 容器可以通过 api 域名访问
docker exec web-container ping api

# DNS 记录
# api.swarm  → VIP
# api.X.stackname.swarm → VIP
```

## VIP 模式（默认）

### 工作原理

VIP 模式下，每个服务分配一个虚拟 IP，DNS 解析到 VIP：

```
┌─────────────────────────────────────────┐
│            Service: api                 │
│                                         │
│  VIP: 10.0.0.5                         │
│                                         │
│  ┌─────────┬─────────┬─────────┐        │
│  │Container│Container│Container│        │
│  │    A    │    B    │    C    │        │
│  │10.0.0.6 │10.0.0.7 │10.0.0.8 │        │
│  └─────────┴─────────┴─────────┘        │
│                                         │
│  VIP → IPVS → Containers               │
└─────────────────────────────────────────┘
```

### 查看 VIP

```bash
# 查看服务 VIP
docker service inspect --format '{{json .Endpoint.VirtualIPs}}' api | jq

# 输出示例
[
  {
    "NetworkID": "abc123...",
    "Addr": "10.0.0.5/24"
  }
]
```

### VIP 特点

| 特性 | 说明 |
|---|---|
| **固定 IP** | 服务 IP 不变，容器 IP 可变 |
| **透明切换** | 容器变化对客户端透明 |
| **一致性** | 始终返回同一个 IP |

## DNSRR 模式

DNSRR（DNS Round Robin）模式下，DNS 直接轮询返回容器 IP：

```yaml
services:
  api:
    image: myapi:latest
    deploy:
      endpoint_mode: dnsrr
```

### DNSRR 工作原理

```
DNS 查询 api
    │
    ▼
┌─────────────────────────────────┐
│    DNS Response (轮询)          │
│                                 │
│    Round 1: 10.0.0.6          │
│    Round 2: 10.0.0.7          │
│    Round 3: 10.0.0.8          │
└─────────────────────────────────┘
```

### DNSRR 特点

| 特性 | 说明 |
|---|---|
| **直接 IP** | 返回容器真实 IP |
| **无代理** | 无 IPVS 开销 |
| **客户端选择** | 客户端决定连接哪个 IP |
| **适合有状态** | 长连接、粘性会话 |

## VIP vs DNSRR

| 特性 | VIP | DNSRR |
|---|---|---|
| DNS 响应 | 固定 VIP | 轮询容器 IP |
| 连接保持 | 支持 | 依赖客户端 |
| 负载均衡 | IPVS 精确 | 依赖 DNS TTL |
| 故障处理 | 自动 | 客户端超时 |
| 适用场景 | 无状态服务 | 有状态服务 |

## 外部负载均衡器集成

### 配置外部 LB

```yaml
services:
  web:
    image: nginx:alpine
    ports:
      - "80:80"
    deploy:
      endpoint_mode: vip
```

外部 LB 配置指向集群的任意 Manager 或 Worker 节点：

```nginx title="Nginx 配置"
upstream myapp {
    server 192.168.1.101:80;
    server 192.168.1.102:80;
    server 192.168.1.103:80;
}

server {
    listen 80;
    server_name myapp.example.com;

    location / {
        proxy_pass http://myapp;
    }
}
```

### HTTP/2 和 gRPC 负载均衡

对于 gRPC 等长连接服务，需要注意：

```yaml
# DNSRR 模式支持粘性会话
services:
  api:
    image: mygrpc:latest
    deploy:
      endpoint_mode: dnsrr
```

## 端口冲突处理

### 同一端口多服务

Swarm 不允许同一端口绑定到多个服务（除非在不同协议上）：

```bash
# 第一个服务绑定 80 端口
docker service create --name web --publish 80:80 nginx:alpine

# 第二个服务无法再绑定 80 端口
docker service create --name api --publish 80:8080 nginx:alpine
# 需要使用不同的 published 端口
```

### 绕过端口冲突

```yaml
services:
  api:
    ports:
      - "8080:80"  # 外部 8080，内部 80
```

## 健康检查与负载均衡

健康检查影响流量分配：

```yaml
services:
  api:
    image: myapi:latest
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:3000/health"]
      interval: 30s
      timeout: 3s
      retries: 3
    deploy:
      replicas: 4
      update_config:
        monitor: 10s  # IPVS 会移除不健康容器
```

```
IPVS 健康检查：
不健康容器 (healthcheck 失败)
    │
    ▼
IPVS 路由表移除
    │
    ▼
流量不再分配到该容器
    │
    ▼
健康检查恢复
    │
    ▼
IPVS 路由表添加
```

## 亲和性和粘性会话

### 容器亲和性

```bash
# 限制容器分布在特定节点
docker service create \
  --name api \
  --placement-preference "spread=node.labels.zone" \
  nginx:alpine
```

### IP 亲和性

使用 DNSRR + 客户端重试实现：

```yaml
services:
  api:
    image: myapi:latest
    deploy:
      endpoint_mode: dnsrr
```

## 总结

Swarm 负载均衡的核心机制：

| 层级 | 机制 | 作用 |
|---|---|---|
| **Routing Mesh** | IPVS | 任意节点到目标服务的路由 |
| **服务发现** | 内置 DNS | 域名解析到 IP |
| **VIP 模式** | IPVS | 固定 VIP 到容器 |
| **DNSRR 模式** | DNS 轮询 | 直接返回容器 IP |

理解这些机制，有助于：

- 选择合适的 endpoint_mode（VIP vs DNSRR）
- 配置外部负载均衡器
- 排查流量分配问题
- 设计高可用架构
