# Docker Swarm 架构

想象一下，你有一批 Docker 主机，想要让它们像一个整体一样工作——容器能在任意节点上运行，当某个节点挂了，容器自动迁移到其他节点，服务还能自动负载均衡。这不是 Kubernetes 独有的能力，Docker Swarm 同样可以做到。

Docker Swarm 是 Docker 内置的**容器编排引擎**，开箱即用，不需要额外安装。相比 Kubernetes 的复杂性，Swarm 的设计哲学是：**简单够用**。

## Swarm 是什么

Docker Swarm 是 Docker 官方提供的**集群管理**和**容器编排**解决方案。它将多个 Docker 主机整合成一个虚拟的 Docker 主机，对外提供统一的 API，让用户可以像操作单台 Docker 一样操作整个集群。

```
┌──────────────────────────────────────────────────────────┐
│                      Docker Swarm                        │
│                                                          │
│  ┌─────────────────────────────────────────────────┐    │
│  │              Manager Nodes (3)                   │    │
│  │  ┌─────────┐  ┌─────────┐  ┌─────────┐         │    │
│  │  │ Manager │  │ Manager │  │ Manager │         │    │
│  │  │   1     │  │   2     │  │   3     │         │    │
│  │  └────┬────┘  └────┬────┘  └────┬────┘         │    │
│  │       └───────────┴───────────┘                │    │
│  │              │ Raft 共识协议                    │    │
│  └──────────────┼──────────────────────────────────┘    │
│                 │                                         │
│  ┌──────────────┴──────────────────────────────────┐    │
│  │              Worker Nodes (N)                    │    │
│  │  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────┐ │    │

│  │  │   1    │  │   2    │  │   3    │  │      │ │    │

│  └─────────────────────────────────────────────────┘    │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

## 核心概念

### 节点（Node）

Swarm 中的节点分为两类：

| 类型 | 说明 | 数量要求 |
|---|---|---|
| **Manager Node** | 负责集群管理、调度、状态同步 | 奇数（建议 3 或 5） |
| **Worker Node** | 负责运行容器 | 可选任意数量 |

**关键原则**：

- Manager 也可以运行容器（默认）
- Manager 数量必须为奇数（用于 Raft 共识）
- 建议 Manager 数量 `>= 3`，以保证高可用

### 服务（Service）

Service 是 Swarm 中容器编排的核心抽象。你声明「需要运行 5 个 nginx 副本」，Swarm 自动确保这 5 个副本始终在运行。

```bash
# 创建一个服务
docker service create --name web --replicas 3 nginx:alpine

# 查看服务
docker service ls
docker service ps web

# 更新服务
docker service update --replicas 5 web
docker service update --image nginx:1.25 web

# 删除服务
docker service rm web
```

### 任务（Task）

Task 是 Swarm 调度的最小单元——每个 Task 对应一个容器实例：

```
Service (replicas=3)
  ├── Task #1 → Container on Node-1
  ├── Task #2 → Container on Node-2
  └── Task #3 → Container on Node-3
```

当某个容器崩溃，Swarm 会自动创建一个新的 Task 来替代它。

## 集群初始化

### 初始化 Swarm

```bash
# 在第一台机器上初始化（成为 Manager）
docker swarm init --advertise-addr 192.168.1.10

# 输出示例：
# Swarm initialized: current node (xxxxxxxxxxxx) is now a manager.
#
# To add a worker to this swarm, run the following command:
#     docker swarm join --token SWMTKN-1-xxxxx worker-ip:2377
#
# To add a manager to this swarm, run the following command:
#     docker swarm join-token manager
```

### 添加 Worker 节点

```bash
# 在 Worker 机器上执行（使用 init 输出的命令）
docker swarm join --token SWMTKN-1-xxxxx manager-ip:2377

# 如果 token 过期或丢失，在 Manager 上重新获取
docker swarm join-token worker
```

### 添加 Manager 节点

```bash
# 在 Manager 上获取 manager token
docker swarm join-token manager

# 输出：
# To add a manager to this swarm, run the following command:
#     docker swarm join --token SWMTKN-1-xxxxx manager-ip:2377

# 在新 Manager 机器上执行
docker swarm join --token SWMTKN-1-xxxxx manager-ip:2377
```

### 脱离集群

```bash
# Worker 脱离集群
docker swarm leave

# Manager 脱离集群（如果是唯一 Manager，需要加 --force）
docker swarm leave --force
```

## Raft 共识协议

Swarm Manager 使用 **Raft 共识协议**保证集群状态一致性。

### 为什么需要奇数个 Manager？

Raft 协议要求 **多数节点（Majority）同意**才能做出决策：

| Manager 数量 | 容错能力 | 说明 |
|---|---|---|
| 1 | 0 | 单点故障，没有容错能力 |
| 2 | 0 | 两节点集群，任何一节点宕机都无法达成多数 |
| 3 | **1** | 容忍 1 节点故障，仍有 2/3 多数 |
| 5 | **2** | 容忍 2 节点故障，仍有 3/5 多数 |
| 7 | **3** | 容忍 3 节点故障 |

:::tip 最佳实践
生产环境建议使用 **3 个 Manager**，这是成本和可用性的最佳平衡点。5 个 Manager 适用于对可用性要求极高的场景。
:::

### Leader 选举

```
时间线：
────────────────────────────────────────────────►

T1: Manager-1 (Leader) ─── Manager-2 ─── Manager-3
      │
      │ (心跳超时，触发选举)
      ▼
T2:   ○ (Leader)  ─── Manager-2  ─── Manager-3
                  ↑
                  │ (投票给 Manager-2)
                  │
                  ▼
T3:   ○ (Leader)  ─── Manager-2* (New Leader) ─── Manager-3
                       ↑
                       │ (收到多数投票，成为 Leader)
```

Raft 协议保证：

- 任何时候最多只有一个 Leader
- Leader 故障后，自动选举新的 Leader
- 即使网络分区发生，也能保证一致性

## 调度策略

Swarm 的调度器负责决定「容器应该运行在哪个节点上」。

### 调度流程

```
用户创建服务 (replicas=3)
        │
        ▼
┌─────────────────────────────────┐
│     Swarm Scheduler             │
│                                 │
│  1. 过滤 (Filter)              │
│     - 节点是否可用              │
│     - 资源是否满足              │
│     - 端口是否冲突              │
│                                 │
│  2. 评分 (Score)               │
│     - 优先调度到资源空闲的节点  │
│     - 避免资源碎片化            │
└─────────────────────────────────┘
        │
        ▼
  分配到合适的节点运行
```

### 亲和与反亲和调度

```yaml
services:
  web:
    image: nginx
    deploy:
      replicas: 3
      placement:
        # 调度到特定标签的节点
        constraints:
          - "node.role==worker"
          - "engine.labels.zone==zone-a"
        # 避免同一服务的容器在同一节点
        preferences:
          - spread: engine.labels.zone
```

```bash
# 给节点打标签
docker node update --label-add zone=zone-a node-1
docker node update --label-add zone=zone-b node-2

# 查看节点标签
docker node inspect node-1 --format '{{ .Spec.Labels }}'
```

### 污点与容忍（Taint/Toleration）

```bash
# 给 Manager 节点添加污点（阻止普通容器运行）
docker node update --taint node-type=manager node-1

# 污点效果：
# - NoSchedule：不调度新容器
# - PreferNoSchedule：尽量不调度
# - NoExecute：驱逐已运行的容器

# 服务容忍污点
docker service create \
  --name db \
  --placement-preferences \
    preference=spread=engine.labels.zone \
  nginx
```

## 集群管理命令

```bash
# 查看集群信息
docker info | grep Swarm

# 查看节点列表
docker node ls

# 查看节点详情
docker node inspect node-1

# 更新节点状态
docker node update --availability drain node-1   # 排空（迁移容器）
docker node update --availability pause node-1   # 暂停（不接收新容器）
docker node update --availability active node-1  # 恢复

# 离开集群
docker node rm node-1  # Manager 先降级再删除
docker node demote node-1  # 降级为 Worker
```

## 网络架构

### Swarm 内部网络

```
┌─────────────────────────────────────────────────────────┐
│                     Docker Swarm                         │
│                                                          │
│  ┌──────────────────────────────────────────────────┐    │
│  │            Ingress Network (Overlay)             │    │
│  │                                                    │    │
│  │  ┌────────┐   ┌────────┐   ┌────────┐          │    │
│  │  │ nginx  │   │ nginx  │   │ nginx  │          │    │
│  │  │  :80   │   │  :80   │   │  :80   │          │    │
│  │  └────────┘   └────────┘   └────────┘          │    │
│  │       ▲           ▲           ▲                │    │
│  │       │           │           │                │    │
│  │  ┌────────────────────────────────────────┐     │    │
│  │  │       IPVS Load Balancer              │     │    │
│  │  │  (对外部暴露的统一入口 :80)            │     │    │
│  │  └────────────────────────────────────────┘     │    │
│  └──────────────────────────────────────────────────┘    │
│                                                          │
│  ┌──────────────────────────────────────────────────┐    │

│  └──────────────────────────────────────────────────┘    │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

### 创建自定义 Overlay 网络

```bash
# 创建可附加的 overlay 网络（Swarm 服务和普通容器都能用）
docker network create \
  --driver overlay \
  --attachable \
  my-network

# 查看网络
docker network ls

# 连接到网络
docker network connect my-network my-container
```

## 总结

Docker Swarm 的架构特点：

1. **开箱即用**：Docker 内置，无需额外安装
2. **简单易用**：学习曲线平缓，命令与 Docker 兼容
3. **Raft 共识**：3 Manager 可容忍 1 节点故障
4. **内置负载均衡**：Ingress 网络自动做七层负载均衡
5. **声明式服务**：用 YAML 定义期望状态，Swarm 自动达成

Swarm 适合的场景：
- 小到中型规模的容器编排（< 1000 节点）
- 团队没有足够的 Kubernetes 运维能力
- 快速原型和开发环境

如果你的需求超过了 Swarm 的能力边界（如复杂的多租户、大规模的 Stateful 应用），可以考虑迁移到 Kubernetes。
