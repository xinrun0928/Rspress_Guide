# Docker 面试总结

Docker 是现代后端面试的必备知识。无论你是面试 Java、Golang 还是 Python 岗位，容器化相关的提问都是高频考点。

本篇整理了 Docker 最常见的面试问题，并给出高质量的回答思路。

## Docker 和虚拟机的区别是什么？

**核心区别在于虚拟化层级和资源模型**。

| 维度 | Docker 容器 | 虚拟机 |
|---|---|---|
| 虚拟化层级 | 操作系统级（共享内核） | 硬件级（独立内核） |
| 启动时间 | 秒级（几百毫秒） | 分钟级 |
| 资源占用 | 极小（MB 级） | 较大（GB 级） |
| 隔离性 | 较弱（共享内核） | 强（完全隔离） |
| 性能损耗 | `~1-3%` | `~5-15%` |
| 镜像大小 | 小（MB~数百 MB） | 大（GB 级） |

**回答要点**：

- 虚拟机通过 Hypervisor 虚拟硬件，每个 VM 有独立内核
- 容器共享宿主机内核，通过 Namespace 和 Cgroups 实现隔离
- 容器本质上是进程，启动快、资源省
- 但隔离性较弱，依赖内核安全

:::tip 面试追问
面试官可能会追问：既然容器共享内核，那隔离性是如何保证的？
回答：主要靠 Linux Namespace（PID、Network、Mount 等隔离）和 Cgroups（资源限制）。但如果容器内进程提权到 root，理论上存在内核漏洞利用的风险，这也是容器安全需要关注的点。
:::

## Dockerfile 中 CMD 和 ENTRYPOINT 的区别？

**CMD 提供默认命令，ENTRYPOINT 定义固定入口**。

| 指令 | 作用 | 覆盖方式 |
|---|---|---|
| `CMD` | 容器启动时的默认命令 | 可被 `docker run` 参数覆盖 |
| `ENTRYPOINT` | 固定入口程序 | 通过 `--entrypoint` 覆盖 |

```dockerfile
# CMD 示例：提供默认参数
CMD ["node", "app.js"]
# docker run myapp  → node app.js
# docker run myapp server.js → node server.js（覆盖）

# ENTRYPOINT 示例：固定为某个工具
ENTRYPOINT ["python", "-m", "http.server"]
# docker run myhttp 80 → python -m http.server 80

# CMD 与 ENTRYPOINT 配合
ENTRYPOINT ["java", "-jar"]
CMD ["app.jar"]
# 实际执行：java -jar app.jar
# docker run myjava -Xmx512m app.jar → java -jar -Xmx512m app.jar
```

**最佳实践**：`ENTRYPOINT` 用于定义固定入口（如工具类容器），`CMD` 用于提供默认参数，两者配合使用最常见。

## Docker 的存储驱动有哪些？应该用哪个？

**主流存储驱动有 overlay2、btrfs、zfs、devicemapper**。

| 驱动 | 特点 | 适用场景 |
|---|---|---|
| `overlay2` | 生产环境首选，性能好 | Linux 4.0+，大多数场景 |
| `btrfs` | 支持快照、压缩 | 需要配额/压缩的环境 |
| `zfs` | 企业级特性 | 需要数据校验的场景 |
| `devicemapper` | 块级操作 | 遗留系统（不推荐新部署） |
| `vfs` | 无 UnionFS | 仅用于测试 |

**推荐**：生产环境使用 `overlay2`（Linux 4.0+），非特权容器使用 `fuse-overlayfs`。

:::tip 面试追问
面试官可能会追问：overlay2 的写时复制（CoW）是如何工作的？
回答：当容器首次修改镜像层的文件时，OverlayFS 将文件从只读层复制到可写层，然后在可写层修改。后续读写直接在可写层操作。这种机制保证了镜像的不可变性，同时允许容器个性化。
:::

## Docker 网络模式有哪些？各自适用场景？

**五种内置网络驱动 + overlay/macvlan**。

| 驱动 | 说明 | 适用场景 |
|---|---|---|
| `bridge` | 默认，连接到 docker0 网桥 | 单主机容器通信 |
| `host` | 共享宿主机网络栈 | 性能敏感场景 |
| `container` | 共享另一个容器的网络 | sidecar 容器 |
| `none` | 完全禁用网络 | 离线任务 |
| `overlay` | 跨主机通信 | Docker Swarm |
| `macvlan` | 分配真实 MAC/IP | 需要物理网络直通 |

**回答要点**：

- bridge 是默认网络，容器间通过 docker0 通信
- 自定义 bridge 网络支持容器名 DNS 解析
- host 模式跳过网络隔离，性能最佳但有端口冲突风险

## 容器如何限制资源？cgroups 原理是什么？

**Docker 通过 cgroups 实现资源限制**。

cgroups（Control Groups）是 Linux 内核功能，将进程分组并限制其资源使用：

| 资源 | Docker 参数 | 说明 |
|---|---|---|
| CPU | `--cpus` | CPU 核心数 |
| CPU | `--cpuset-cpus` | 绑定 CPU |
| CPU | `--cpu-shares` | 相对权重 |
| 内存 | `-m/--memory` | 内存限制 |
| 内存 | `--memory-swap` | swap 限制 |
| 内存 | `--memory-reservation` | 软限制 |
| I/O | `--blkio-weight` | I/O 权重 |
| PID | `--pids-limit` | 最大进程数 |

```bash
docker run -d \
  -m="512m" \
  --cpus="1.0" \
  --pids-limit=100 \
  --name app \
  nginx:alpine
```

**cgroups 工作原理**：容器创建时，Docker 在 `/sys/fs/cgroup/cpu/` 和 `/sys/fs/cgroup/memory/` 下创建子目录，为容器设置资源限制。容器的进程在这些 cgroup 子目录中运行，受资源限制约束。

## Docker 镜像分层是怎么实现的？

**基于 UnionFS（联合文件系统）**。

```
镜像层（只读，多个镜像共享）
    │
    ├── Layer 1: 基础系统
    ├── Layer 2: 运行时依赖
    ├── Layer 3: 应用代码
    └── ...

容器层（可写）
    │
    └── 用户修改、新的文件
```

**层共享**：多个镜像可以共享相同的底层层，节省存储空间。例如 ubuntu:20.04 和 nginx:alpine 都基于 debian 层，它们共享该层。

**容器层**：每个容器有独立可写层，修改镜像文件时通过 Copy-on-Write 机制复制到容器层。

## Copy-on-Write 在 Docker 中是如何工作的？

**写时复制：直到需要写入时才复制数据**。

| 场景 | 行为 |
|---|---|
| 读取文件 | 直接从镜像层读取，无复制 |
| 首次修改文件 | 将文件复制到容器层，修改副本 |
| 后续读写 | 直接操作容器层副本 |
| 删除文件 | 在容器层创建 whiteout 文件遮盖 |

**CoW 的优势**：

- 启动快（无需复制整个镜像）
- 存储省（共享镜像层）
- 灵活性高（容器可个性化）

**CoW 的开销**：首次修改镜像文件时需要复制，有轻微延迟。建议将频繁写入的文件放在 Volume 中。

## Docker 安全有哪些手段？

**多层安全防护机制**。

| 层次 | 措施 | 说明 |
|---|---|---|
| 命名空间 | 用户命名空间映射 | 容器 UID 映射到宿主机不同 UID |
| capabilities | `cap-drop=ALL` | 移除危险权限 |
| seccomp | 系统调用过滤 | 阻止危险系统调用 |
| AppArmor/SELinux | 强制访问控制 | 限制文件/进程访问 |
| rootless | 非 root 用户运行 | 减少提权风险 |

```bash
docker run -d \
  --cap-drop=ALL \
  --security-opt seccomp=/profile.json \
  --read-only \
  --user 1001:1001 \
  nginx:alpine
```

**最佳实践**：非 root 用户运行、移除所有 capabilities、使用只读文件系统、配置 seccomp 文件。

## 多阶段构建的原理和优势是什么？

**在单个 Dockerfile 中使用多个 FROM 阶段**。

```dockerfile
# 第一阶段：构建
FROM golang:1.21 AS builder
WORKDIR /app
COPY . .
RUN CGO_ENABLED=0 go build -o myapp .

# 第二阶段：运行
FROM alpine:3.18
COPY --from=builder /app/myapp /usr/local/bin/
CMD ["myapp"]
```

**优势**：

| 优势 | 说明 |
|---|---|
| 镜像体积小 | 只复制构建产物，不含编译器、源码 |
| 构建效率高 | 构建阶段可利用缓存 |
| 安全性高 | 最终镜像不包含构建工具 |
| 最小攻击面 | 只包含运行时依赖 |

**适用场景**：编译型语言（Go、Rust、C++）的构建，以及任何需要清理构建依赖的场景。

## 容器之间如何通信？

**同一网络容器通过 docker0 网桥通信**。

```
┌────────────────────────────────────────────┐
│ 宿主机                                    │
│                                           │
│ docker0 (172.17.0.1)                      │
│     │                                      │
│     ├── veth_abc ── Container A (172.17.0.2)
│     └── veth_def ── Container B (172.17.0.3)
│                                           │
│ iptables (NAT/路由)                        │
└────────────────────────────────────────────┘
```

**通信流程**：

1. **容器间通信**：通过 docker0 网桥直接交换帧（同一网络）
2. **跨网络通信**：通过 iptables 规则路由
3. **外部访问**：通过 NAT 转换（源地址伪装）
4. **端口映射**：通过 DNAT 规则将宿主机端口映射到容器

**DNS 解析**：自定义 bridge 网络提供内置 DNS，可通过容器名解析：

```bash
docker network create mynet
docker run -d --network mynet --name db redis:alpine
docker run -d --network mynet --name api myapi
# api 容器可以通过 "db" 主机名访问 redis
```
