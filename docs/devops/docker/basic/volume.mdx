# Docker 数据持久化：Volume 完全指南

容器重启数据就没了？——Docker 数据持久化的正确打开方式

你有没有过这样的经历：精心在容器里配置好的数据，容器一重启，全没了。

这其实是 Docker 的**默认行为**。容器的文件系统是**临时性**的——当容器被删除时，容器层的数据也会随之消失。这不是 bug，而是 Docker 的设计哲学：**容器应该是无状态的、可丢弃的**。

但现实是，总有一些数据需要持久化：数据库的文件、用户上传的资料、应用程序生成的日志……怎么办？

这就引出了 Docker 的数据持久化机制——**Volume**。

## 一、为什么需要 Volume？

在说 Volume 之前，先理解一下容器的存储结构。

Docker 容器由多层文件系统叠加而成：

- **镜像层**（Image Layer）：只读，多个容器共享
- **容器层**（Container Layer）：读写，容器独有，生命周期与容器绑定
- **存储驱动**（Storage Driver）：管理镜像层和容器层的读写（Overlay2、aufs 等）

当容器删除时，容器层的数据会被一起清除。这就是为什么用 `docker rm` 删容器，里面存的数据也跟着没了。

```
┌─────────────────────────────────┐
│         容器层 (Writable)       │  ← 容器独有，重启丢失
├─────────────────────────────────┤
│         镜像层 (Read-only)      │  ← 多个容器共享
├─────────────────────────────────┤
│         存储驱动                │
└─────────────────────────────────┘
```

Volume 的出现，就是为了解决这个问题。它将宿主机的目录挂载到容器内，数据直接写入宿主机，不受容器生命周期的影响。

## 二、Volume 的四种类型

Docker 提供了四种不同的数据挂载方式，适用于不同的场景。

### 2.1 匿名 Volume（Anonymous Volume）

匿名 Volume 是最简单的一种，**没有指定的名称**，由 Docker 自动生成。

```bash
docker run -v /data nginx
```

执行这个命令时，Docker 会在宿主机上创建一个临时目录，格式类似 `/var/lib/docker/volumes/abc123/_data`，并将其挂载到容器的 `/data` 目录。

**特点**：

- Docker 自动创建和管理
- 容器删除时，Volume 会被 Docker 自动清理（除非有其他容器仍在使用）
- 适合临时性存储，或作为容器的初始数据目录

**问题**：由于没有名称，很难定位和管理。你无法通过名字找到这个 Volume 在宿主机的哪个位置。

### 2.2 命名 Volume（Named Volume）

命名 Volume 是**推荐的生产环境用法**。你指定一个名字，Docker 会在宿主机上创建一个持久化的目录。

```bash
docker volume create my-data
docker run -v my-data:/app/data nginx
```

这次，Volume 的名字是 `my-data`，宿主机上的目录是 `/var/lib/docker/volumes/my-data/_data`。

**特点**：

- 独立于容器生命周期，容器删除后 Volume 仍然存在
- 易于识别和管理：`docker volume ls` 能看到所有命名 Volume
- 支持跨容器共享
- 适合数据库存储、文件上传目录、日志目录等需要持久化的场景

```bash
# 查看所有 Volume
docker volume ls

# 查看 Volume 详情
docker volume inspect my-data

# 删除未使用的 Volume
docker volume prune
```

### 2.3 绑定挂载（Bind Mount）

绑定挂载将宿主机的**任意目录**映射到容器内部，与命名 Volume 的主要区别是数据存储位置由你决定，而非 Docker 管理。

```bash
docker run -v /host/path:/container/path nginx
```

**特点**：

- 宿主机路径必须已存在（Docker 不会自动创建）
- 权限可控制（只读或读写）
- 适合配置文件挂载、源代码挂载（实现代码修改热更新）
- 直接操作宿主机文件，Docker 无法管理这些文件

**使用场景对比**：

```bash
# 配置文件挂载：修改宿主机的 nginx.conf 即可生效
docker run -v /etc/nginx/nginx.conf:/etc/nginx/nginx.conf:ro nginx

# 开发环境：源代码热更新
docker run -v $(pwd):/app/src my-app

# 敏感配置：只读挂载防止容器修改
docker run -v /opt/config:/config:ro my-app
```

:::warning 权限注意
默认情况下，挂载的目录权限与宿主机一致。如果宿主机目录是 root 所有，容器内的进程可能没有写入权限。此时需要调整宿主机的目录权限，或者在容器内以 root 用户运行。
:::

### 2.4 tmpfs 挂载（tmpfs Mount）

tmpfs 是一种特殊的挂载方式——数据**存储在内存中**，不会写入磁盘。

```bash
docker run --tmpfs /app/cache nginx
```

**特点**：

- 数据存储在内存中，性能极高
- 容器停止或删除后，数据自动消失
- 适合存储敏感数据（如密码、Token）或临时缓存
- 仅在 Linux 主机上可用，macOS 和 Windows 不支持

**限制**：

- 容器退出后数据丢失
- 受限于可用内存大小
- 不能跨容器共享

## 三、两种挂载语法：`-v` 和 `--mount`

Docker 提供两种等效的挂载语法。

### 3.1 `-v`（或 `--volume`）

简洁写法，适合大多数场景。

```bash
# 命名 Volume
docker run -v my-data:/app/data nginx

# 绑定挂载
docker run -v /host/path:/container/path nginx

# 只读挂载
docker run -v my-data:/app/data:ro nginx

# tmpfs 挂载
docker run --tmpfs /app/cache nginx
```

### 3.2 `--mount`

更明确的写法，适合需要指定额外选项的场景。

```bash
# 命名 Volume
docker run --mount type=volume,source=my-data,target=/app/data nginx

# 绑定挂载
docker run --mount type=bind,source=/host/path,target=/container/path nginx

# 只读绑定挂载
docker run --mount type=bind,source=/host/path,target=/container/path,readonly nginx

# tmpfs 挂载
docker run --mount type=tmpfs,target=/app/cache nginx
```

**语法对比**：

| 参数 | `-v` 语法 | `--mount` 语法 |
|---|---|---|
| Volume 类型 | `-v name:/path` | `--mount type=volume,source=name,target=path` |
| 绑定挂载 | `-v /host:/container` | `--mount type=bind,source=/host,target=/container` |
| 只读 | `-v name:/path:ro` | `--mount type=volume,source=name,target=path,readonly` |
| tmpfs | `--tmpfs /path` | `--mount type=tmpfs,target=/path` |

:::tip 选择建议
对于命名 Volume 和绑定挂载，两者功能相同。选择时：

- 日常使用用 `-v`，简洁直观
- 需要明确指定挂载选项时用 `--mount`，语法更清晰
- 在 Docker Compose 中，两种语法都常用，取决于个人偏好
:::

## 四、实战：不同场景下的 Volume 选择

### 4.1 数据库存储

数据库文件必须持久化，选择**命名 Volume**。

```yaml title="docker-compose.yml"
version: "3.8"
services:
  postgres:
    image: postgres:15
    volumes:
      - postgres-data:/var/lib/postgresql/data
    environment:
      POSTGRES_PASSWORD: secret

volumes:
  postgres-data:
```

```bash
# 备份数据库 Volume
docker run --rm -v postgres-data:/data -v $(pwd):/backup alpine tar czf /backup/postgres-backup.tar.gz -C /data .
```

### 4.2 配置文件管理

配置文件通常需要从宿主机读取，选择**绑定挂载**。

```bash
docker run -d \
  -v /opt/nginx/nginx.conf:/etc/nginx/nginx.conf:ro \
  -v /opt/nginx/logs:/var/log/nginx \
  -p 80:80 \
  nginx
```

### 4.3 开发环境热更新

源代码需要实时同步，选择**绑定挂载**。

```bash
docker run -d \
  -v $(pwd):/app/src \
  -v /app/node_modules \
  -p 3000:3000 \
  node:18 npm run dev
```

:::warning 谨慎使用匿名 Volume
上面的命令中，`-v /app/node_modules` 是一个**匿名 Volume**。这用于防止 `$(pwd):/app/src` 把容器内的 `node_modules` 目录覆盖掉（因为宿主机没有这个目录）。但这个匿名 Volume 不会被自动清理。
:::

### 4.4 敏感数据存储

密码、API Key 等敏感数据，用**只读绑定挂载**或**tmpfs**。

```bash
# 敏感配置文件，只读挂载
docker run -v /opt/secrets/.env:/app/.env:ro my-app

# 临时敏感数据，存内存
docker run --tmpfs /run/secrets my-app
```

## 五、Volume 性能对比

不同挂载方式的性能差异主要来自存储介质和虚拟化层。

| 挂载类型 | 性能 | 适用场景 |
|---|---|---|
| 匿名/命名 Volume | 中等，经过 Docker 存储驱动 | 生产环境数据库、日志 |
| 绑定挂载 | 高，直接访问宿主机文件系统 | 配置文件、源代码 |
| tmpfs | 最高，内存访问 | 临时缓存、敏感数据 |

**性能测试数据**（仅供参考，实际性能受硬件影响）：

```bash
# 测试 Volume 写入性能
docker run --rm -v test-volume:/data alpine sh -c "dd if=/dev/zero of=/data/test bs=1M count=100 2>&1"

# 测试 tmpfs 写入性能
docker run --rm --tmpfs /data alpine sh -c "dd if=/dev/zero of=/data/test bs=1M count=100 2>&1"
```

:::tip 性能优化建议
- 对性能要求极高的场景（如数据库），优先使用绑定挂载
- 使用 Docker Desktop 时，macOS 和 Windows 的文件访问有额外的虚拟化开销，性能不如 Linux 原生
- 考虑使用命名 Volume 并选择合适的存储驱动（如 overlay2）
:::

## 六、Volume 管理命令

```bash
# 创建命名 Volume
docker volume create my-volume

# 列出所有 Volume
docker volume ls

# 查看 Volume 详情（包含挂载点）
docker volume inspect my-volume

# 删除未使用的 Volume
docker volume prune

# 删除指定 Volume
docker volume rm my-volume

# 启动时自动创建 Volume（如果不存在）
docker run -v my-volume:/data nginx
```

## 七、常见问题与解决方案

### 7.1 Volume 权限问题

容器内的进程以非 root 用户运行时，可能无法写入挂载的目录。

```bash
# 解决方案 1：修改宿主机目录权限
chmod 777 /host/path

# 解决方案 2：在容器内使用 root 用户（不推荐生产环境）
docker run -u 0 -v /host/path:/data nginx

# 解决方案 3：使用 named volume 并设置正确的用户
docker run -v my-data:/data --user 1000:1000 nginx
```

### 7.2 跨主机共享数据

Docker Volume 本身不支持跨主机共享。如果需要跨主机共享存储，需要使用网络文件系统：

- **NFS**：Linux 原生网络文件系统
- **CIFS/SMB**：Windows 共享
- **分布式存储**：GlusterFS、Ceph 等

```bash
# 使用 NFS Volume 插件
docker volume create --driver vieux/sshfs \
  -o sshcmd=root@remote:/data \
  -o password=password \
  my-nfs-volume
```

### 7.3 Volume 清理

容器删除后，匿名 Volume 不会自动清理，需要手动处理：

```bash
# 清理所有未使用的 Volume
docker volume prune

# 清理所有未使用的 Volume（包括命名 Volume）
docker volume prune -a
```

## 八、总结

Docker 的四种存储机制各有适用场景：

| 类型 | 数据持久性 | 管理方式 | 适用场景 |
|---|---|---|---|
| 匿名 Volume | 容器生命周期 | Docker 自动 | 临时数据、初始目录 |
| 命名 Volume | 持久化 | Docker 管理 | 数据库、文件存储（**推荐**） |
| 绑定挂载 | 宿主机决定 | 用户管理 | 配置文件、源代码 |
| tmpfs | 仅内存 | 不落盘 | 敏感数据、临时缓存 |

记住一个原则：**需要持久化的数据用命名 Volume，需要共享宿主机资源的场景用绑定挂载，需要安全性的场景用 tmpfs**。

下次再遇到"容器重启数据丢了"的问题，你就知道该怎么解决了。
