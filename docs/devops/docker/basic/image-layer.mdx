# 镜像分层原理

你知道为什么 Docker 镜像可以「秒级启动」吗？为什么本地已有的镜像层不需要重新下载？为什么修改一个小文件，整个镜像都要重建？

这些问题的答案，都藏在 Docker 镜像的分层机制中。

## UnionFS：分层的基石

Docker 镜像的分层依赖于 **UnionFS（联合文件系统）**。UnionFS 的核心思想是：允许将多个文件系统「叠加」在一起，对外呈现为一个统一的文件系统。

在 UnionFS 中，每个文件系统称为一个「层（Layer）」。用户看到一个合并后的视图（merged view），但实际上文件来自不同的层。底层是只读的，上层可以「遮盖」下层的同名文件。

```
UnionFS 层结构示意：
┌────────────────────────────────────────┐
│           merged (合并视图)              │ ← 用户/容器看到的样子
├────────────────────────────────────────┤
│  upperdir (读写层，可写)                │ ← 容器修改时写入
├────────────────────────────────────────┤
│  lowerdir 1 (只读层)                    │
│  lowerdir 2 (只读层)                    │ ← 镜像层，多个只读层叠加
│  lowerdir 3 (只读层)                    │
└────────────────────────────────────────┘
```

### Docker 存储驱动

Docker 支持多种存储驱动来实现 UnionFS：

| 驱动 | 原理 | 适用场景 |
|---|---|---|
| **overlay2** | 叠加两个目录 | Linux 内核 4.0+，生产环境推荐 |
| **aufs** | 叠加多个分支 | Ubuntu/Debian，老系统 |
| **devicemapper** | 块设备映射 | 老的 RHEL/CentOS |
| **btrfs/zfs** | 文件系统原生支持 | 需要特定文件系统的环境 |
| **vfs** | 无 UnionFS，纯拷贝 | 测试环境，不推荐生产使用 |

现代 Linux 发行版（Ubuntu 20.04+、RHEL 8+、Debian 10+）默认使用 **overlay2**，这也是 Docker 官方推荐的生产环境存储驱动。

## OverlayFS 工作原理

OverlayFS 将文件系统分为三层：

- **lowerdir**：下层目录，只读层，对应镜像层
- **upperdir**：上层目录，可写层，对应容器层
- **merged**：合并视图，用户看到的统一文件系统

```
一个运行中的容器的文件系统结构：

/var/lib/docker/overlay2/
├── l/                          # 软链接目录
│   └── [link_to_layer]         # 指向实际层目录
├── [container_id]/
│   ├── lower-id-1/             # 最底层：基础镜像层
│   ├── lower-id-2/             # 中间层：中间镜像层
│   ├── lower-id-3/             # 上层镜像层
│   ├── upper/                  # 可写层（容器层）
│   │   ├── etc/
│   │   ├── var/log/
│   │   └── .wh.filename        # whiteout 文件
│   ├── work/                   # OverlayFS 工作目录
│   └── merged/                 # 合并视图（容器的根文件系统）
└── [image_layer_id]/
    └── diff/                   # 镜像层的实际内容
```

### 文件查找规则

当容器进程访问一个文件时，OverlayFS 按以下顺序查找：

1. 先在 `upperdir` 中查找——如果找到，返回该文件（上层遮盖下层）
2. 如果没找到，在 `lowerdir` 中按从上到下顺序查找
3. 如果在某个 lowerdir 中找到，复制到 upperdir（Copy-on-Write），然后返回

### whiteout 文件

当删除一个在下层存在的文件时，OverlayFS 在上层创建一个特殊的 whiteout 文件（`.wh.filename`），该文件「遮盖」了下层的同名文件，使得在合并视图中看不到它。

```
场景：在容器中删除了 /etc/nginx/nginx.conf

upperdir 结构：
upper/
├── etc/
│   └── .wh.nginx.conf          # whiteout 文件
└── ...

merged 视图：
merged/etc/
# nginx.conf 被遮盖，看不到这个文件
```

## Copy-on-Write：高效的写操作

**Copy-on-Write（CoW，写时复制）** 是 Docker 镜像分层的核心机制。它的核心思想是：**直到需要写入时才复制数据**。

### 为什么需要 CoW

想象一下：如果每个容器启动时都复制一份完整的镜像数据到容器可写层，那将：

- 启动速度极慢（复制大量数据）
- 占用大量磁盘空间（每个容器都有一份完整的镜像副本）
- 无法共享底层数据

CoW 解决了这些问题：容器启动时只需要创建一个薄薄的可写层，实际文件仍然来自只读的镜像层。只有当容器需要修改某个文件时，才将该文件复制到可写层进行修改。

### CoW 操作示例

```bash
# 场景 1：读取文件（无需复制）
# 容器读取 /lib/libc.so，OverlayFS 直接从 lowerdir 返回

# 场景 2：修改文件
# 容器修改 /etc/app.conf
# 1. OverlayFS 查找文件，发现位于 lowerdir
# 2. 将文件复制到 upperdir
# 3. 在 upperdir 中修改文件
# 4. 后续访问都从 upperdir 读取

# 场景 3：新建文件
# 容器在 /data/ 下创建 new.log
# 直接写入 upperdir，无需复制
```

### CoW 的性能影响

CoW 机制带来的第一个写入操作会有轻微性能开销（需要复制文件），但后续操作都在上层的独立副本中进行，没有额外开销。

对于频繁写入的场景，建议：

- 将日志等频繁写入的数据挂载为 Volume（绕过 CoW，直接写入宿主机文件系统）
- 使用 `tmpfs` 挂载存储敏感数据
- 避免在镜像层中存储大文件或频繁修改的文件

## 镜像层共享

Docker 镜像层最强大的特性之一是**共享**。多个镜像可以共享相同的底层层，从而节省磁盘空间。

### 共享机制

假设我们有两个镜像：

- `nginx:latest`：基于 `debian:bookworm`，安装了 nginx
- `httpd:latest`：基于 `debian:bookworm`，安装了 apache

```
镜像结构：

debian:bookworm (基础层)
    │
    ├── nginx:latest
    │   └── debian 层 (共享)
    │   └── nginx 层 (新增)
    │
    └── httpd:latest
        └── debian 层 (共享)
        └── httpd 层 (新增)
```

在磁盘上，`debian` 层只存储一份，但被两个镜像共享。

### 容器层的共享

更进一步，同一个镜像创建的多个容器也共享镜像层：

```
镜像层（只读，多个容器共享）
    │
    ├── Container 1 可写层
    ├── Container 2 可写层    ← 每个容器有独立可写层
    └── Container 3 可写层
```

每个容器只需要存储自己的可写层（修改过的文件），极大减少了磁盘占用。

## 层缓存与构建优化

Docker 构建时会缓存每一层的结果。当 Dockerfile 的某一行与上一次构建完全相同时，Docker 会直接使用缓存的层，而不会重新执行。

### 缓存命中规则

Docker 按从上到下的顺序检查缓存：

```dockerfile
# 假设这是上一次构建的 Dockerfile
FROM ubuntu:20.04          # 缓存命中（基础镜像相同）
WORKDIR /app                # 缓存命中（WORKDIR 指令相同）
COPY package*.json ./       # 缓存命中（package*.json 内容相同）
RUN npm install             # 缓存命中（上述内容未变）
COPY . ./                   # 缓存命中（所有文件内容相同）
```

只要任意一层的内容发生变化，**该层及其之后的所有层都会重新构建**。

### 利用缓存优化构建

理解了缓存机制，我们可以优化 Dockerfile 的编写顺序：

```dockerfile
# 好的写法：频繁变化的文件放后面
COPY package*.json ./      # 依赖文件先复制
RUN npm install            # 安装依赖（缓存）
COPY . ./                  # 应用代码后复制，代码变化不会导致重新安装

# 不好的写法：频繁变化的文件放前面
COPY . ./                  # 任何文件变化都会导致重新安装依赖
RUN npm install
```

### 强制重新构建

有时需要绕过缓存重新构建：

```bash
# 方法 1：强制不使用缓存（不推荐，会全部重新构建）
docker build --no-cache -t myapp .

# 方法 2：指定目标层重新构建（推荐）
docker build --build-arg CACHE_BUST=$(date +%s) -t myapp .

# 方法 3：在 Dockerfile 中添加虚拟层
ARG CACHE_BUST=1
RUN echo $CACHE_BUST
```

## 镜像层数量的考量

虽然层越多，复用机会越大，但过深的层数会带来额外开销：

| 影响 | 说明 |
|---|---|
| 元数据开销 | 每层都有元数据，层数过多增加管理开销 |
| 文件系统限制 | 部分文件系统（如 AUFS）对嵌套深度有限制 |
| 镜像传输 | 层数多意味着更多的 HTTP 请求 |

Dockerfile 最佳实践建议：**将多个 RUN 指令合并，减少层数**：

```dockerfile
# 不好的写法：创建多层
RUN apt-get update
RUN apt-get install -y curl
RUN apt-get install -y git
RUN apt-get clean

# 好的写法：合并为一个 RUN，每条命令以 && 连接，最后清理
RUN apt-get update && \
    apt-get install -y curl git && \
    apt-get clean && \
    rm -rf /var/lib/apt/lists/*
```

## 查看镜像层信息

Docker 提供了查看镜像层信息的命令：

```bash
# 查看镜像的历史层
docker history nginx:alpine

# 输出示例：
IMAGE          CREATED        BY                 SIZE
b8d1dc2...     3 days ago     CMD ["nginx..."]   0B
2a4e2b8...     3 days ago     EXPOSE 80/tcp      0B
d0d6c8a...     3 days ago     COPY file:...       1.39kB
f3c2d8e...     3 days ago     CMD ["/bin/sh"]    0B
d7a7e9f...     3 days ago     ADD file:...       29.9MB
<missing>      4 days ago     MAINTAINER        0B

# 查看镜像的详细层信息
docker inspect nginx:alpine
```

## 总结

Docker 镜像的分层机制是容器技术的核心，它带来三大优势：

1. **存储效率**：多个镜像和容器共享底层层，大幅减少磁盘占用
2. **构建效率**：利用缓存机制，只需重新构建变化的部分
3. **传输效率**：只需要传输缺失的层，支持断点续传

理解 UnionFS、Copy-on-Write 和层缓存的原理，是编写高效 Dockerfile 和排查生产环境问题的基础。在后续的章节中，我们将学习如何利用这些特性进行镜像优化和多阶段构建。
