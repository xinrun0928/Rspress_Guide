# Docker 核心概念

Docker 的世界里有三个核心概念：**镜像（Image）**、**容器（Container）** 和 **仓库（Repository）**。理解这三个概念及其相互关系，是掌握 Docker 的基础。

## Docker 镜像

### 什么是镜像

Docker 镜像是一个只读的模板，包含了运行容器所需的程序、依赖、文件系统结构和配置。换个角度说，镜像是容器的「快照」——它定义了容器启动时的初始状态。

类比来说：

- 如果把容器比作一个 Java 类实例，那么镜像就是 Class 文件
- 如果把镜像比作一个装好系统的 ISO 文件，那么容器就是从这个 ISO 启动的虚拟机

镜像是分层的。每一层都记录了相对于上一层的变化，这种设计带来了两个核心优势：**共享层可以节省存储空间**，以及**构建时可以复用已有层**。

### 镜像的组成

一个典型的 Docker 镜像包含以下组成部分：

```
镜像名称:标签
└── Layer 1 (Base Layer) - 操作系统基础层，如 Ubuntu 20.04
    └── Layer 2 - 中间依赖层，如 Python 运行时
        └── Layer 3 - 应用代码层
            └── Layer 4 (Top Layer) - 配置层，如环境变量
```

镜像的标识方式为 `REPOSITORY:TAG`：

```bash
# nginx 是仓库名，latest 是标签
nginx:latest

# 带 registry 地址的完整路径
registry.example.com/myapp:v1.2.3

# SHA256 哈希唯一标识
nginx@sha256:abc123...
```

如果省略标签，默认使用 `latest`。但 `latest` 并不总是代表最新版本，它只是镜像作者指定的一个普通标签。

### UnionFS 文件系统

Docker 镜像的分层依赖于 **UnionFS（联合文件系统）**。UnionFS 的核心思想是：多个只读文件系统「叠加」在一起，对外呈现为一个统一的文件系统。

在 Docker 中，最常用的是 **OverlayFS**（尤其是 overlay2 存储驱动）。OverlayFS 将下层目录称为 `lowerdir`（只读），将上层目录称为 `upperdir`（可写），合并后的视图称为 `merged`。

```
overlay2 文件系统结构：
                    ┌───────────────┐
                    │     merged     │ ← 容器看到的文件系统（合并视图）
                    ├───────────────┤
    upperdir ───────┤    upperdir    │ ← 容器可写层（Container Layer）
                    ├───────────────┤
    lowerdir ───────┤    lowerdir    │ ← 镜像层（Image Layers，只读）
                    ├───────────────┤
                    │   base lowerdir│ ← 基础镜像层
                    └───────────────┘
```

当容器修改一个文件时，OverlayFS 会执行 **Copy-on-Write**：将文件从 lowerdir 复制到 upperdir，然后在 upperdir 中修改。对于容器来说，文件系统看起来是完整可写的，但实际上只修改了顶层。

## Docker 容器

### 什么是容器

容器是镜像的运行实例。镜像定义了容器的初始状态，容器则是镜像在运行时产生的「活的状态」。

镜像是静态的、只读的模板；容器是动态的、可写的进程。多个容器可以基于同一个镜像创建，每个容器都有自己独立的可写层。

```
镜像（只读模板）                    容器（运行实例）
┌──────────────────┐           ┌──────────────────┐
│  Image Layer 1   │           │  Image Layer 1   │
│  Image Layer 2   │           │  Image Layer 2   │
│  Image Layer 3   │           │  Image Layer 3   │
│                  │  ──────>  │                  │
│                  │  docker   │  Container Layer │ ← 容器可写层
│                  │  create   │                  │
└──────────────────┘           └──────────────────┘
```

### 容器与进程

从 Linux 系统的角度看，容器本质上是一个或多个进程。容器内的进程在宿主机看来就是普通的进程，只是被 Namespace 和 Cgroups 机制「隔离」了。

```bash
# 在宿主机上查看容器进程
$ docker inspect --format '{{.State.Pid}}' my_container
12345

# 直接在宿主机上查看这个进程
$ ps -p 12345
  PID TTY      STAT   TIME COMMAND
12345 ?        Ss     0:00 /bin/sh -c python app.py
```

容器与普通进程的区别在于：容器被分配了独立的命名空间（PID、Network、Mount 等），并且被 Cgroups 限制了其资源使用。

### 容器生命周期

容器有五种基本状态：

| 状态 | 说明 |
|---|---|
| `created` | 已创建但未启动 |
| `running` | 正在运行 |
| `paused` | 已暂停（进程被冻结） |
| `restarting` | 正在重启 |
| `exited` | 已停止（进程已退出） |
| `dead` | Docker 认为容器已死（清理失败时出现） |

```bash
# 查看所有状态的容器
docker ps -a

# 启动、停止、重启容器
docker start my_container
docker stop my_container
docker restart my_container

# 暂停和恢复
docker pause my_container
docker unpause my_container
```

### 容器与镜像的关系

理解容器和镜像的关系，可以用一个类比：**镜像就像食谱，容器就像做出来的菜**。

- 食谱（镜像）可以重复使用，做很多份菜（容器）
- 每一份菜（容器）的改动不会影响食谱（镜像）
- 食谱变了（重新 build），新的菜会基于新食谱来做

```bash
# 基于 ubuntu:20.04 镜像创建容器（此时容器处于 created 状态）
docker create ubuntu:20.04

# 启动容器（进入 running 状态）
docker start <container_id>

# 或者一步到位：创建并启动
docker run ubuntu:20.04

# -it 参数：分配伪终端并保持标准输入（进入交互式 shell）
docker run -it ubuntu:20.04 /bin/bash

# 在后台运行（detached 模式）
docker run -d nginx:alpine
```

## Docker 仓库

### 什么是仓库

Docker 仓库（Repository）是存储和分发镜像的地方。你可以把它理解为「镜像的 GitHub」——每个仓库包含某一类镜像的不同版本（标签）。

仓库的完整地址格式为：

```
[Registry URL]/[Namespace/]Repository[:Tag]
```

例如：

```bash
# Docker Hub 官方仓库（省略了默认的 registry.example.com）
nginx:latest

# Docker Hub 上的用户仓库
myusername/myapp:v1.0

# 私有仓库
registry.example.com/myteam/backend:v1.2.3
```

### Registry 与 Repository 的区别

很多人容易混淆这两个概念：

- **Registry（注册服务器）**：托管多个仓库的服务。最常见的是 Docker Hub（`docker.io`），还有阿里云、Harbor 等私有 Registry。
- **Repository（仓库）**：某一类镜像的集合，以「命名空间/镜像名」的形式存在。如 `nginx`、`library/ubuntu` 都是 Repository。

```bash
# 完整的镜像引用解析
docker.io/library/nginx:latest
│         │        │       │
Registry  Namespace Repository Tag
```

### Docker Hub

Docker Hub 是 Docker 官方维护的公共镜像仓库，托管了数万种官方镜像，包括：

- 基础操作系统镜像：`ubuntu`、`debian`、`alpine`、`centos`
- 编程语言运行时：`python`、`node`、`openjdk`、`golang`
- 数据库：`mysql`、`postgres`、`redis`、`mongodb`
- Web 服务器和代理：`nginx`、`apache`、`traefik`
- DevOps 工具：`jenkins`、`gitlab-runner`、`ansible`

```bash
# 从 Docker Hub 拉取镜像
docker pull nginx:latest

# 搜索镜像
docker search nginx

# 登录 Docker Hub
docker login
```

:::tip 镜像拉取加速
在国内访问 Docker Hub 速度较慢，可以使用镜像加速器。Docker Desktop 用户可以在设置中配置镜像加速器地址。
:::

### 镜像的存储结构

在宿主机上，Docker 镜像存储在 `/var/lib/docker` 目录下（Linux 系统）。使用 OverlayFS 时，目录结构大致如下：

```
/var/lib/docker/
├── overlay2/              # 镜像层和容器层的存储
│   ├── l/                  # 层数据的软链接
│   └── [layer_hash]/       # 实际的层数据
├── image/                  # 镜像元数据
│   └── overlay2/
│       └── repositories.json
├── containers/             # 容器配置文件和日志
└── volumes/                # 数据卷存储
```

## 三个概念的协作流程

理解了单个概念后，我们来看它们如何协同工作：

```bash
# 1. 从 Registry 拉取镜像到本地
docker pull nginx:alpine

# 2. 基于镜像创建并启动容器
docker run -d --name my_nginx -p 80:80 nginx:alpine

# 3. 在容器中修改（仅影响该容器）
docker exec my_nginx touch /usr/share/nginx/html/modified

# 4. 提交容器为新的镜像
docker commit my_nginx myrepo/my_nginx:v1

# 5. 将新镜像推送到 Registry
docker push myrepo/my_nginx:v1
```

:::warning 提交容器的注意事项
`docker commit` 不是构建镜像的最佳实践，因为它：
- 不记录构建过程，难以复现
- 无法进行构建优化
- 镜像变更不透明

实际工作中，应该使用 Dockerfile 来定义镜像的构建过程。
:::

## 总结

Docker 的三个核心概念——镜像、容器、仓库——构成了容器化技术的基础。

- **镜像** 是只读的模板，定义了容器的初始状态
- **容器** 是镜像的运行实例，拥有自己独立的可写层
- **仓库** 是镜像的存储和分发中心

理解它们的关系是使用 Docker 的第一步。在后续章节中，我们将深入学习如何编写 Dockerfile 构建镜像、如何使用数据卷和网络实现容器间协作，以及如何使用 Docker Compose 编排多容器应用。
