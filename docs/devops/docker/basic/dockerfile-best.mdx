# Dockerfile 最佳实践

你的 Dockerfile 构建出的镜像有多大？启动有多快？包含了多少不必要的工具？有没有泄露敏感信息？

如果这些问题你从未想过，那你的 Dockerfile 可能还有优化空间。

## 多阶段构建：让镜像更小

多阶段构建是现代 Dockerfile 的核心技能。它允许你在一个 Dockerfile 中使用多个 `FROM` 指令，每个 `FROM` 代表一个独立的构建阶段。

### 单阶段构建的问题

传统的单阶段 Dockerfile，通常需要同时包含构建工具和运行时：

```dockerfile
# 单阶段构建：包含所有内容，镜像臃肿
FROM golang:1.21
WORKDIR /app
COPY . .
RUN go build -o myapp .      # 需要 Go 编译器
RUN go build -o myapp .      # 包含构建依赖，最终镜像不需要
COPY . .
# 问题：镜像包含完整的 Go 编译器和源代码
```

### 多阶段构建解决方案

```dockerfile title="Dockerfile"
# 第一阶段：构建
FROM golang:1.21 AS builder

WORKDIR /app
COPY . .

# 只在构建阶段安装依赖
RUN CGO_ENABLED=0 go build -ldflags="-w -s" -o myapp .

# 第二阶段：运行时
FROM alpine:3.18

# 从构建阶段复制编译好的二进制文件
COPY --from=builder /app/myapp /usr/local/bin/

# 只需要运行时依赖
RUN apk add --no-cache ca-certificates

ENTRYPOINT ["myapp"]
```

这样最终镜像只包含编译好的二进制文件和运行时依赖，不需要 Go 编译器、源代码或构建工具。

### 多阶段构建的效果对比

| 指标 | 单阶段 | 多阶段 |
|---|---|---|
| 镜像大小 | `~800MB` | `~15MB` |
| 包含编译器 | 是 | 否 |
| 包含源代码 | 是 | 否 |
| 构建时间 | 相同 | 相同 |

### 构建阶段命名

显式命名构建阶段，使 Dockerfile 更清晰：

```dockerfile title="Dockerfile"
FROM node:18 AS frontend-builder
WORKDIR /app
RUN npm install && npm run build

FROM node:18-alpine AS backend-builder
WORKDIR /app
COPY . .
RUN npm install && npm run build

FROM node:18-alpine AS production
WORKDIR /app
COPY --from=frontend-builder /app/dist ./public
COPY --from=backend-builder /app .
COPY --from=backend-builder /app/node_modules ./node_modules
CMD ["node", "server.js"]
```

### 从外部镜像复制

`COPY --from` 不仅可以从本地构建阶段复制，还可以从外部镜像复制：

```dockerfile title="Dockerfile"
# 从官方 nginx 镜像复制默认配置
COPY --from=nginx:alpine /etc/nginx/nginx.conf /etc/nginx/

# 从本地构建阶段复制
COPY --from=builder /app/dist /var/www/html/
```

## 基础镜像选择

基础镜像的选择直接影响最终镜像的大小和安全性。

### Alpine vs Slim vs Full

| 基础镜像 | 大小 | 特点 | 适用场景 |
|---|---|---|---|
| `alpine` | `~5MB` | 极简，需要 apk 包管理器，musl libc | 生产环境追求体积 |
| `debian:slim` | `~80MB` | 精简，apt 包管理器，glibc | 需要 glibc 兼容性 |
| `ubuntu`/`debian` | `数百MB` | 完整系统，包含常用工具 | 开发调试 |
| `amazonlinux` | `~164MB` | AWS 优化，兼容 RHEL/CentOS | AWS 环境 |

```dockerfile
# 追求极致体积
FROM alpine:3.18

# 需要 glibc 兼容性
FROM debian:bookworm-slim

# Java 应用
FROM eclipse-temurin:17-jre-alpine

# Node.js 应用
FROM node:18-alpine
```

:::warning musl vs glibc
Alpine 使用 musl libc，而大多数 Node.js、Python 等解释型语言的预编译包基于 glibc。如果遇到兼容性问题，切换到 `debian:slim`。
:::

### 官方镜像的优势

使用官方镜像（经过安全审核，自动更新）：

```dockerfile
# 好：使用官方镜像，明确标签
FROM node:18.17.0-alpine

# 不好：latest 标签可能引入非预期变化
FROM node:latest

# 不好：未经验证的第三方镜像
FROM someuser/node-server:latest
```

## 减少镜像层数

每条指令都会创建一个新的镜像层，层数过多会增加存储开销和管理复杂度。

### 合并 RUN 指令

```dockerfile
# 不好的写法：多层
RUN apt-get update
RUN apt-get install -y curl
RUN apt-get install -y wget
RUN apt-get install -y git
RUN rm -rf /var/lib/apt/lists/*

# 好的写法：合并为一个 RUN
RUN apt-get update && \
    apt-get install -y --no-install-recommends \
        curl \
        wget \
        git && \
    apt-get clean && \
    rm -rf /var/lib/apt/lists/*
```

### --no-install-recommends

apt-get 安装时使用 `--no-install-recommends` 参数，避免安装不必要的推荐包：

```dockerfile
RUN apt-get update && \
    apt-get install -y --no-install-recommends \
        my-package && \
    apt-get clean && \
    rm -rf /var/lib/apt/lists/*
```

## 安全最佳实践

### 使用非 root 用户

容器内的 root 用户与宿主机 root 是同一个用户（ID 0），这可能带来安全风险。

```dockerfile title="Dockerfile"
FROM alpine:3.18

# 创建应用用户
RUN addgroup -g 1001 -S appgroup && \
    adduser -S appuser -u 1001 -G appgroup

WORKDIR /app

# 复制文件（先创建目录结构）
COPY --chown=appuser:appgroup . .

USER appuser

CMD ["node", "app.js"]
```

### 不要在镜像中存储密钥

**永远不要**把密钥、密码、证书直接写入镜像。

```dockerfile
# 错误做法：密钥在镜像层中，永久保存
FROM alpine:3.18
ENV DATABASE_PASSWORD=my_secret_password
COPY ./secrets/api-key.json /run/secrets/

# 正确做法：通过环境变量或挂载注入
FROM alpine:3.18
# 不设置默认密码，或设置占位符
ENV DATABASE_PASSWORD=""

# 运行时通过环境变量或 secret mount 注入
# docker run -e DATABASE_PASSWORD=xxx myapp
# docker run --secret id=db_pass,env=DATABASE_PASSWORD myapp
```

### 最小化攻击面

移除不必要的工具和 SUID 程序：

```dockerfile
FROM alpine:3.18

# 移除不需要的服务和工具
RUN apk add --no-cache \
    your-package && \
    # 清理文档和示例
    rm -rf /usr/share/doc/* \
           /usr/share/man/* \
           /var/cache/apk/* && \
    # 移除 SUID 程序（除非必要）
    chmod -s /usr/bin/passwd \
             /usr/bin/sudo \
             /usr/bin/chsh \
             2>/dev/null || true
```

## 构建缓存优化

### 利用层缓存

将不常变化的内容放在前面，频繁变化的内容放在后面：

```dockerfile title="Dockerfile"
FROM node:18-alpine

# 1. 依赖文件（不常变化）先复制
COPY package*.json ./

# 2. 安装依赖（利用缓存）
RUN npm ci --only=production

# 3. 应用代码（频繁变化）后复制
COPY . .

CMD ["node", "app.js"]
```

### 添加版本标签触发重建

当需要强制重建缓存时，可以在 Dockerfile 中添加版本标识：

```dockerfile
# 方法 1：使用构建参数
ARG CACHE_DATE=2024-01-15
RUN echo "Build date: $CACHE_DATE"

# 方法 2：利用 BuildKit 的 cache mounts
RUN --mount=type=cache,target=/root/.npm \
    npm ci --only=production
```

## BuildKit 新特性

BuildKit 是 Docker 18.06+ 内置的新一代构建引擎，提供更多特性：

```bash
# 启用 BuildKit
export DOCKER_BUILDKIT=1

# 或使用 docker buildx（推荐）
docker buildx build -t myapp .
```

### 并行构建

BuildKit 自动分析依赖关系，并行执行独立的步骤：

```dockerfile
# 传统的串行写法
RUN npm ci
RUN npm run build

# BuildKit 可以自动并行处理依赖无关联的步骤
```

### Cache Mounts

在构建过程中持久化缓存目录：

```dockerfile
# 构建时使用 npm 缓存，不用每次重新下载
RUN --mount=type=cache,target=/root/.npm \
    npm ci --only=production

# 构建时使用 pip 缓存
RUN --mount=type=cache,target=/root/.cache/pip \
    pip install -r requirements.txt
```

### 网络访问控制

BuildKit 支持构建时的网络隔离：

```dockerfile
# 构建阶段禁用网络
RUN --network=none \
    # 只能使用本地已安装的包
    echo "Build in isolated network"
```

### secret 挂载

构建时安全注入密钥，不保存到镜像层：

```dockerfile
# 运行时挂载 secret
RUN --mount=type=secret,id=npmrc \
    npm ci --only=production

# 构建命令
docker build --secret id=npmrc,src=.npmrc .
```

## 完整示例：生产级 Dockerfile

综合以上最佳实践的 Dockerfile 示例：

```dockerfile title="Dockerfile"
# ============================================
# 第一阶段：构建
# ============================================
FROM node:18-alpine AS builder

WORKDIR /app

# 先复制依赖清单，利用 npm 缓存
COPY package*.json ./

# 使用 cache mount 持久化 npm 缓存
RUN --mount=type=cache,target=/root/.npm \
    npm ci --only=production

# 复制源代码
COPY . .

# 构建应用
RUN npm run build

# ============================================
# 第二阶段：运行时
# ============================================
FROM node:18-alpine AS production

# 创建非 root 用户
RUN addgroup -g 1001 -S nodejs && \
    adduser -S nodeuser -u 1001 -G nodejs

WORKDIR /app

# 复制依赖和构建产物，设置正确的所有者
COPY --from=builder --chown=nodeuser:nodejs /app/node_modules ./node_modules
COPY --from=builder --chown=nodeuser:nodejs /app/dist ./dist
COPY --from=builder --chown=nodeuser:nodejs /app/package.json ./

# 切换到非 root 用户
USER nodeuser

# 环境变量
ENV NODE_ENV=production
ENV PORT=3000

# 暴露端口
EXPOSE $PORT

# 健康检查
HEALTHCHECK --interval=30s --timeout=3s --retries=3 \
    CMD node -e "require('http').get('http://localhost:${PORT}/health', (r) => process.exit(r.statusCode === 200 ? 0 : 1))"

# 启动命令
CMD ["node", "dist/main.js"]
```

## 总结

生产级 Dockerfile 的核心原则：

| 原则 | 具体做法 |
|---|---|
| **小** | 使用多阶段构建，选择 alpine/slim 基础镜像 |
| **快** | 优化层顺序，利用构建缓存 |
| **安全** | 非 root 用户运行，不在镜像中存储密钥 |
| **可维护** | 显式标签，清晰的 Dockerfile 结构 |
| **可观测** | 添加健康检查，记录构建元数据 |

遵循这些原则，你的 Dockerfile 将构建出更小、更快、更安全的生产级镜像。
