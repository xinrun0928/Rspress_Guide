# Dockerfile 指令详解

你是否见过这样的 Dockerfile？层层堆叠十几行，却不知道每一条 `RUN` 是不是必要的——为什么 `apt-get update` 和 `apt-get install` 必须写在一起？为什么 `COPY` 和 `RUN` 的顺序会影响构建速度？

Dockerfile 是 Docker 镜像的「配方」。写得好，镜像体积小、安全性高、构建快；写得差，镜像臃肿不堪，每次改动都要重头构建。理解每一条指令的含义和最佳实践，是用好 Docker 的必经之路。

## 指令一览

| 指令 | 作用 | 层数 |
|---|---|---|
| `FROM` | 指定基础镜像 | 0 |
| `RUN` | 执行命令并创建新层 | 1 |
| `COPY` | 复制文件到镜像中 | 1 |
| `ADD` | 复制文件（支持 URL 和 tar 自动解压） | 1 |
| `CMD` | 容器启动时的默认命令 | 0 |
| `ENTRYPOINT` | 容器入口命令 | 0 |
| `ENV` | 设置环境变量 | 0 |
| `WORKDIR` | 设置工作目录 | 0 |
| `ARG` | 构建参数（仅构建时有效） | 0 |
| `EXPOSE` | 声明端口 | 0 |
| `VOLUME` | 定义持久化挂载点 | 0 |
| `USER` | 设置运行用户 | 0 |
| `LABEL` | 添加元数据 | 0 |
| `ONBUILD` | 声明构建触发指令 | 0 |

## FROM：起点

`FROM` 是每个 Dockerfile 的第一条指令，定义基础镜像。

```dockerfile
# 官方镜像
FROM ubuntu:22.04
FROM python:3.11-slim
FROM openjdk:17-jdk-slim

# 私有镜像
FROM registry.example.com/base/centos:7
```

### 镜像选择建议

| 场景 | 推荐镜像 | 说明 |
|---|---|---|
| 追求最小体积 | `alpine` | 极简 Linux 发行版，~5MB |
| 需要完整工具链 | `ubuntu`、`debian` | 完整生态系统 |
| Java 应用 | `eclipse-temurin:17-jre` | 官方维护，只包含 JRE |
| Python 应用 | `python:3.11-slim` | 推荐使用 slim 版本 |

:::warning 不要使用 `latest` 标签
`latest` 标签意味着「最新版本」，但构建时间不同，镜像内容可能不同。用具体标签（如 `python:3.11-slim`）确保可重现构建。
:::

## RUN：执行命令

`RUN` 在镜像构建时执行命令，每一条 `RUN` 指令会创建一层新的镜像层。

```dockerfile
# Shell 形式（默认使用 /bin/sh -c）
RUN apt-get update && apt-get install -y nginx

# Exec 形式（不经过 shell，直接执行）
RUN ["apt-get", "update"]
```

### 常见操作

```dockerfile
# 安装软件包
RUN apt-get update && apt-get install -y \
    curl \
    git \
    vim \
    && rm -rf /var/lib/apt/lists/*

# 下载并解压文件
RUN wget https://example.com/app.tar.gz \
    && tar -xzf app.tar.gz \
    && rm app.tar.gz

# 执行多行 shell
RUN for dir in /opt/*/; do \
        echo "Processing $dir"; \
        ./build.sh "$dir"; \
    done
```

### 清理缓存：避免镜像臃肿

每一条 `RUN` 指令都会产生临时文件和缓存，这些不会自动清理。清理方法：

```dockerfile
# 错误：缓存文件留在镜像中
RUN apt-get update && apt-get install -y nginx

# 正确：清理缓存
RUN apt-get update && apt-get install -y nginx \
    && rm -rf /var/lib/apt/lists/*
```

```dockerfile
# 错误：npm 安装后缓存留在镜像中
RUN npm install

# 正确：利用 npm ci 的 --cache 选项减少层大小
COPY package*.json ./
RUN npm ci --only=production && npm cache clean --force
```

## COPY 与 ADD：复制文件

### COPY：推荐的首选

`COPY` 是最安全、最透明的文件复制方式。

```dockerfile
# 复制本地文件到镜像
COPY ./src /app/src

# 复制构建产物（常配合 .dockerignore 使用）
COPY ./dist /var/www/html

# 带权限复制的写法
COPY --chown=app:app ./src /app/src
```

```dockerfile
# 复制多个文件
COPY package.json package-lock.json /app/

# 通配符复制
COPY *.js /app/
```

### ADD：特殊场景使用

`ADD` 支持 `COPY` 的所有功能，额外支持两种特殊能力：

```dockerfile
# ADD 的两个特殊能力

# 1. 从 URL 下载文件
ADD https://example.com/app.tar.gz /tmp/

# 2. 自动解压 tar 文件（本地 tar.gz）
ADD ./app.tar.gz /opt/app/
```

:::tip 优先使用 COPY
`COPY` 的行为更加透明和可预测。对于 URL 下载和解压场景，显式使用 `RUN` 配合 `wget/curl` 和 `tar` 命令，可以让构建过程更清晰。
:::

## CMD 与 ENTRYPOINT：启动命令

这两个指令最容易混淆。记住一个原则：**`ENTRYPOINT` 定义「是什么」，`CMD` 定义「默认参数」**。

### CMD：默认命令

```dockerfile
# 三种形式（推荐 Exec 形式）
CMD ["python", "app.py"]
CMD ["npm", "start"]
CMD ["nginx", "-g", "daemon off;"]

# Shell 形式（会绕过 CMD，但会被 ENTRYPOINT 忽略）
CMD python app.py
```

`CMD` 的主要用途是**为 `ENTRYPOINT` 提供默认参数**：

```dockerfile
ENTRYPOINT ["python", "app.py"]
CMD ["--port", "8080"]  # 默认参数，可被 docker run 覆盖
```

### ENTRYPOINT：入口点

`ENTRYPOINT` 让容器表现得像一个**可执行程序**：

```dockerfile
# 定义入口
ENTRYPOINT ["python", "app.py"]

# docker run 的参数会追加到 ENTRYPOINT 后面
# docker run myimage --port 9090  =>  python app.py --port 9090
```

### 两者对比

| 场景 | 正确写法 |
|---|---|
| 容器像命令：`docker run myredis redis-server` | `ENTRYPOINT ["redis-server"]` |
| 容器像 Web 服务：`docker run mynginx` | `CMD ["nginx", "-g", "daemon off;"]` |
| 带默认参数：`docker run myapp` 默认监听 8080 | `ENTRYPOINT ["app"]` + `CMD ["--port", "8080"]` |

## ENV 与 ARG：变量

### ENV：镜像中的环境变量

`ENV` 设置的环境变量在容器运行时永久生效：

```dockerfile
ENV APP_HOME=/app
ENV NODE_ENV=production
ENV PATH=$PATH:/opt/bin
```

### ARG：构建时的参数

`ARG` 是构建参数，仅在构建时有效，不会保留到镜像中：

```dockerfile
# 定义带默认值的构建参数
ARG APP_VERSION=1.0.0
ARG USER_ID=1000

# 构建时传入
# docker build --build-arg APP_VERSION=2.0.0 .
```

:::warning ENV 和 ARG 的区别
- `ARG`：构建参数，镜像构建完成后不存在
- `ENV`：环境变量，容器运行时始终存在
:::

## WORKDIR：工作目录

`WORKDIR` 设置后续指令的工作目录，自动创建目录（如果不存在）：

```dockerfile
WORKDIR /app
COPY . .

WORKDIR /app/src
RUN npm install

WORKDIR /app/dist
# 后续命令都在 /app/dist 下执行
```

---

## EXPOSE：声明端口

`EXPOSE` 只是文档作用，告诉用户容器会监听哪个端口。真正暴露端口靠 `docker run -p`：

```dockerfile
EXPOSE 8080
EXPOSE 443 80
```

## 构建阶段实战：多阶段构建

多阶段构建可以在最终镜像中只保留运行时依赖，大幅减少镜像体积：

```dockerfile title="Dockerfile"
# 阶段 1：构建阶段
FROM maven:3.9-eclipse-temurin-17 AS builder
WORKDIR /app
COPY pom.xml .
RUN mvn dependency:go-offline       # 缓存依赖层
COPY src ./src
RUN mvn package -DskipTests          # 构建 jar

# 阶段 2：运行阶段
FROM eclipse-temurin:17-jre-alpine AS runtime
WORKDIR /app
COPY --from=builder /app/target/myapp.jar ./app.jar

# 只复制 jar，不复制源码、Maven 本地仓库、target 目录
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "app.jar"]
```

### 常见多阶段构建模式

```dockerfile
# Go 应用
FROM golang:1.21 AS builder
WORKDIR /app
COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -o myapp

FROM alpine:latest
COPY --from=builder /app/myapp /usr/local/bin/
ENTRYPOINT ["myapp"]
```

```dockerfile
# Node 应用
FROM node:18-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

FROM node:18-alpine AS runtime
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
CMD ["node", "dist/index.js"]
```

## 完整示例

```dockerfile title="Dockerfile"
# 使用多阶段构建
FROM eclipse-temurin:17-jre-alpine AS runtime

# 设置维护者（已废弃，用 LABEL 替代）
LABEL maintainer="team@example.com"
LABEL version="1.0.0"
LABEL description="Java application"

# 环境变量
ENV JAVA_OPTS="-Xmx512m -Xms256m"
ENV APP_HOME=/app

# 工作目录
WORKDIR ${APP_HOME}

# 复制依赖清单和 jar
COPY target/myapp.jar app.jar

# 非 root 用户运行（安全最佳实践）
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
USER appuser

# 暴露端口
EXPOSE 8080

# 健康检查
HEALTHCHECK --interval=30s --timeout=3s --retries=3 \
    CMD curl -f http://localhost:8080/health || exit 1

# 启动命令
ENTRYPOINT ["sh", "-c", "java $JAVA_OPTS -jar app.jar"]
```

## 常见错误

| 错误写法 | 问题 | 正确写法 |
|---|---|---|
| `RUN apt-get install -y package` | 没有清理缓存 | `RUN apt-get install -y package && rm -rf /var/lib/apt/lists/*` |
| `COPY . /app` | 没有 .dockerignore | 先配置 .dockerignore，再用 `COPY . /app` |
| `RUN npm install` | 每次代码改动都会重新安装依赖 | 先 `COPY package*.json` 再 `RUN npm ci` |
| `FROM ubuntu` | 没有指定标签 | `FROM ubuntu:22.04` |
| `EXPOSE 80` | 没加引号 | `EXPOSE 8080` |

---

## 总结

Dockerfile 的核心原则：

1. **层数尽量少**：合并多条 `RUN` 指令
2. **变更频率低的放前面**：利用构建缓存
3. **清理临时文件**：每条 RUN 指令后清理缓存
4. **多阶段构建**：分离构建环境和运行环境
5. **最小化基础镜像**：优先使用 `alpine`、`slim` 版本
6. **非 root 用户**：安全最佳实践
7. **显式声明端口和环境变量**：让使用者一目了然
