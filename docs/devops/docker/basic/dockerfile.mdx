# Dockerfile 入门

你有没有想过，一个 `docker run nginx` 命令，Docker 是怎么知道「nginx 长什么样」的？答案是 **Dockerfile**。

Dockerfile 是 Docker 镜像的构建脚本，它用简单的指令描述了：从哪个基础镜像开始，需要安装什么依赖，复制什么文件，配置什么环境变量，最终运行什么命令。这个脚本就是镜像的「配方」，而 `docker build` 就是按照配方制作镜像的过程。

## Dockerfile 基本结构

一个典型的 Dockerfile 包含多个指令，每条指令都会在镜像中创建一个新的层：

```dockerfile
# 基础镜像
FROM ubuntu:20.04

# 维护者信息（已废弃，用 LABEL 替代）
MAINTAINER John Doe

# 环境变量
ENV APP_HOME=/opt/app
ENV NODE_ENV=production

# 工作目录
WORKDIR $APP_HOME

# 复制文件
COPY package*.json ./
COPY . .

# 安装依赖
RUN npm install --production

# 暴露端口
EXPOSE 3000

# 健康检查
HEALTHCHECK --interval=30s --timeout=3s --retries=3 \
  CMD curl -f http://localhost:3000/health || exit 1

# 运行用户
USER node

# 启动命令
CMD ["node", "server.js"]
```

## FROM：指定基础镜像

`FROM` 是 Dockerfile 的第一条指令（必须），它指定了构建镜像所基于的基础镜像。

```dockerfile
# 使用官方镜像
FROM ubuntu:20.04
FROM python:3.11-slim
FROM openjdk:17-jre-alpine

# 使用私有镜像
FROM registry.example.com/base/centos:7

# 多阶段构建中的阶段镜像
FROM node:18-alpine AS builder
```

:::tip 镜像选择建议
- **追求镜像体积小**：使用 `-alpine`、`-slim` 后缀的镜像（如 `alpine`、`python:3.11-slim`）
- **追求兼容性**：使用 `-slim` 而非 `alpine`，避免某些 glibc 兼容性问题
- **追求构建速度**：在 CI/CD 中，使用预装了构建工具的基础镜像
:::

### 镜像标签的最佳实践

| 场景 | 推荐做法 | 示例 |
|---|---|---|
| 生产环境 | 使用具体版本号 | `FROM node:18.17.0-alpine` |
| 开发环境 | 可用 `:latest` | `FROM node:latest` |
| 多阶段构建 | 命名构建阶段 | `FROM golang:1.21 AS builder` |

## RUN：执行命令

`RUN` 指令在镜像构建过程中执行命令，每条 RUN 会在当前镜像层之上创建一个新层。

### Shell 形式 vs Exec 形式

```dockerfile
# Shell 形式：命令在 /bin/sh -c 中执行
RUN apt-get update && apt-get install -y curl git
RUN echo "Hello" > /tmp/greeting

# Exec 形式：命令直接执行，不经过 shell
RUN ["apt-get", "update"]
RUN ["/bin/bash", "-c", "echo hello"]
```

:::warning Shell 形式的问题
Shell 形式会自动经过 `/bin/sh -c` 执行，这意味着：
- 变量替换：`echo $HOME` 会正常工作
- 但不支持管道等复杂语法：`RUN npm install | tee install.log` 可能不符合预期
- 后台进程：`RUN service nginx start` 不会保持运行
:::

### RUN 指令的优化

每次 `RUN` 都会创建新层，频繁修改的文件应该单独处理以利用缓存：

```dockerfile
# 好的做法：先复制依赖文件，再安装，最后复制代码
COPY package*.json ./
RUN npm ci --only=production
COPY . .

# 不好的做法：每次修改代码都会使 npm install 缓存失效
COPY . .
RUN npm install
```

## COPY：复制文件

`COPY` 将文件或目录从构建上下文复制到镜像中。

```dockerfile
# 复制单个文件
COPY nginx.conf /etc/nginx/nginx.conf

# 复制目录（保持目录结构）
COPY ./src /opt/app/src

# 使用通配符
COPY package*.json ./

# 带所有者信息的复制（BuildKit）
COPY --chown=node:node . .
```

### COPY 与 ADD 的区别

| 特性 | COPY | ADD |
|---|---|---|
| 复制本地文件 | 支持 | 支持 |
| 复制远程 URL | 不支持 | 支持 |
| 自动解压 tar 文件 | 不支持 | 支持 |
| 推荐用途 | 日常文件复制首选 | 仅在需要自动解压时使用 |

```dockerfile
# 推荐：使用 COPY
COPY ./app.tar.gz /opt/app/
RUN tar -xzf /opt/app.tar.gz -C /opt/app

# ADD 可以简化但不够明确（不推荐）
ADD ./app.tar.gz /opt/app/
```

## CMD：容器启动命令

`CMD` 指定容器启动时执行的命令。**一个 Dockerfile 只能有一条 CMD 指令**（最后一条生效）。

```dockerfile
# Exec 形式（推荐）
CMD ["python", "app.py"]
CMD ["node", "server.js"]

# 包含参数时，通常与 ENTRYPOINT 配合
CMD ["--port", "8080"]

# Shell 形式（不推荐）
CMD python app.py
```

### CMD 与 ENTRYPOINT 的区别

这是 Docker 新手最常混淆的概念：

| 指令 | 作用 | 使用场景 |
|---|---|---|
| `CMD` | 提供默认参数，可被 `docker run` 参数覆盖 | 应用可以接受命令行参数 |
| `ENTRYPOINT` | 强制执行程序，参数更明确 | 固定为某个工具/入口点 |

```dockerfile
# CMD 示例：容器启动 node
FROM node:18-alpine
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
CMD ["node", "app.js"]

# docker run myapp 可覆盖参数：docker run myapp app-prod.js
```

```dockerfile
# ENTRYPOINT 示例：固定为某个工具
FROM nginx:alpine
COPY nginx.conf /etc/nginx/nginx.conf
ENTRYPOINT ["nginx", "-g", "daemon off;"]

# docker run mynginx --help 不影响入口点，只改变执行参数
```

### CMD 与 ENTRYPOINT 配合使用

这是最常见的用法：`ENTRYPOINT` 定义入口程序，`CMD` 提供默认参数：

```dockerfile
FROM openjdk:17-jre-alpine
ARG JAR_FILE=app.jar
COPY ${JAR_FILE} /opt/app/app.jar
WORKDIR /opt/app
ENTRYPOINT ["java", "-jar"]
CMD ["app.jar"]

# 运行效果：java -jar app.jar
# 覆盖效果：docker run myjava -Xmx512m app.jar 实际执行 java -jar -Xmx512m app.jar
```

## ENTRYPOINT：定义入口点

`ENTRYPOINT` 定义容器的「主程序」，容器启动时总是会执行这个程序。

```dockerfile
# Exec 形式（推荐）
ENTRYPOINT ["python", "-m", "http.server", "80"]

# 带配置变量的形式
ENTRYPOINT ["docker-entrypoint.sh"]
```

:::tip ENTRYPOINT 的最佳实践
使用 shell 脚本作为 ENTRYPOINT，可以处理复杂的初始化逻辑：

```dockerfile
COPY docker-entrypoint.sh /usr/local/bin/
RUN chmod +x /usr/local/bin/docker-entrypoint.sh
ENTRYPOINT ["docker-entrypoint.sh"]
CMD ["postgres"]
```

脚本中可以处理环境变量、健康检查、数据初始化等逻辑。
:::

## ENV：设置环境变量

`ENV` 设置环境变量，这些变量在容器运行时和后续的构建指令中都可见。

```dockerfile
# 设置环境变量
ENV NODE_ENV=production
ENV APP_HOME=/opt/app
ENV PATH=$APP_HOME/bin:$PATH

# 多行设置
ENV JAVA_HOME=/opt/java \
    MAVEN_HOME=/opt/maven \
    GRADLE_HOME=/opt/gradle
```

:::warning 环境变量的作用域
`ENV` 设置的是容器的环境变量，**不包括构建机器的环境变量**。如果需要引用构建机器的值，可以使用 `ARG` 指令。
:::

## ARG：构建参数

`ARG` 定义构建参数，仅在构建过程中可见，不会在最终镜像中保留（除非被 `ENV` 使用）。

```dockerfile
# 定义构建参数
ARG APP_VERSION=1.0.0
ARG BUILD_DATE

# 使用参数
FROM ubuntu:20.04
RUN echo "Building version $APP_VERSION"

# --build-arg 传递参数
# docker build --build-arg APP_VERSION=2.0.0 -t myapp .
```

### 内置构建参数

Docker 提供了一些预定义的 `ARG`：

```dockerfile
# 使用预定义参数
ARG JAVA_HOME
RUN echo "Building on ${DOCKER_BUILDKIT:-0}"

# .dockerignore 排除构建缓存失效
# 自动注入这些 ARG 可精确控制缓存
```

## WORKDIR：设置工作目录

`WORKDIR` 设置工作目录，后续的 `RUN`、`CMD`、`ENTRYPOINT`、`COPY`、`ADD` 指令都会在这个目录下执行。

```dockerfile
WORKDIR /opt/app
COPY . .

WORKDIR /opt/app/src
RUN npm install

# 可以使用之前 ENV 定义的环境变量
ENV APP_HOME=/opt/app
WORKDIR $APP_HOME
```

:::tip WORKDIR vs RUN cd
不要用 `RUN cd /path && command` 来切换目录，因为每一层 RUN 都是独立的 shell 会话。应该使用 `WORKDIR` 来明确指定工作目录。
:::

## EXPOSE：声明端口

`EXPOSE` 声明容器运行时监听的端口（文档作用，不实际绑定端口）：

```dockerfile
EXPOSE 80
EXPOSE 443
EXPOSE 8080 8443

# 可以指定协议
EXPOSE 5432/tcp
EXPOSE 6379/udp
```

:::tip EXPOSE 的实际作用
`EXPOSE` 主要是文档作用，告诉用户这个容器会监听哪些端口。实际端口绑定仍然需要 `-p` 参数：`docker run -p 8080:80 mynginx`。
:::

## VOLUME：定义数据卷

`VOLUME` 创建一个挂载点，声明容器的持久化数据目录：

```dockerfile
# 定义匿名卷（每次容器启动生成新的随机 ID）
VOLUME /var/lib/postgresql/data

# 定义命名卷（推荐，可复用）
VOLUME ["pgdata"]

# 多卷
VOLUME /data /conf /logs
```

:::warning 何时使用 VOLUME
`VOLUME` 应该用于需要持久化的数据（如数据库存储、用户上传文件），而不适合用于临时文件或应用日志（这些可以写入容器可写层或 stdout）。
:::

## USER：设置运行用户

`USER` 设置后续指令的运行用户，提高安全性：

```dockerfile
# 创建应用用户
RUN groupadd -r appgroup && useradd -r -g appgroup appuser

# 切换用户
USER appuser

# 之后的指令（RUN、CMD、ENTRYPOINT）都以 appuser 运行
CMD ["node", "app.js"]
```

:::danger 安全建议
容器应该以非 root 用户运行。如果基础镜像没有创建用户，需要在 Dockerfile 中创建。某些基础镜像（如 `nginx`、`postgres`）默认已有应用用户。
:::

## LABEL：添加元数据

`LABEL` 为镜像添加键值对元数据：

```dockerfile
LABEL maintainer="john@example.com"
LABEL version="1.0"
LABEL description="My awesome application"
LABEL org.label-schema.name="myapp"
LABEL org.label-schema.version="1.0.0"

# 多标签（减少层数）
LABEL maintainer="john@example.com" \
      version="1.0" \
      description="My awesome application"
```

## ONBUILD：构建触发器

`ONBUILD` 定义构建指令，在基于此镜像的其他 Dockerfile 构建时执行：

```dockerfile
# 父镜像
FROM node:18-alpine
ONBUILD COPY package*.json ./
ONBUILD RUN npm ci
ONBUILD COPY . .

# 子镜像（基于父镜像构建时，会自动执行 ONBUILD 中的指令）
FROM parent-image:latest
CMD ["node", "app.js"]
```

:::tip ONBUILD 的使用场景
ONBUILD 适用于需要作为基础镜像被继承的场景，如公司内部的基础镜像定义了统一的构建流程，子镜像无需关心依赖安装细节。
:::

## 构建镜像实战

将上述指令整合，来看一个完整的 Dockerfile 示例：

```dockerfile title="Dockerfile"
# 使用多阶段构建
FROM node:18-alpine AS builder

WORKDIR /app

# 先复制依赖文件，利用构建缓存
COPY package*.json ./
RUN npm ci --only=production

# 再复制源代码
COPY . .

# 构建应用
RUN npm run build

# 生产镜像
FROM node:18-alpine AS production

# 创建非 root 用户
RUN addgroup -g 1001 -S nodejs && \
    adduser -S nodeuser -u 1001

WORKDIR /app

# 从 builder 阶段复制构建产物
COPY --from=builder --chown=nodeuser:nodejs /app/dist ./dist
COPY --from=builder --chown=nodeuser:nodejs /app/node_modules ./node_modules
COPY --from=builder --chown=nodeuser:nodejs /app/package.json ./

USER nodeuser

EXPOSE 3000

ENV NODE_ENV=production

HEALTHCHECK --interval=30s --timeout=3s --retries=3 \
  CMD wget --no-verbose --tries=1 --spider http://localhost:3000/health || exit 1

CMD ["node", "dist/main.js"]
```

## 总结

Dockerfile 指令虽然多，但核心逻辑很清晰：

- `FROM` 确定起点
- `RUN` 执行构建步骤（安装依赖、编译代码）
- `COPY/ADD` 放入文件
- `ENV/ARG` 配置环境
- `WORKDIR/USER` 设置运行环境
- `EXPOSE/VOLUME` 声明端口和数据
- `CMD/ENTRYPOINT` 定义启动命令

理解这些指令的作用和最佳实践，是编写高质量 Dockerfile 的基础。
