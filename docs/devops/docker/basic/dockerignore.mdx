# Dockerfile .dockerignore 文件

你有没有遇到过这样的情况：写好了 Dockerfile，执行 `docker build`，然后发现构建速度异常缓慢——因为 Docker 把 `node_modules`、`.git`、`vendor` 这些巨大的目录都打包进了构建上下文。

这就是 `.dockerignore` 存在的意义——告诉 Docker「哪些文件不需要打包进镜像」。

## 基本语法

`.dockerignore` 的语法与 `.gitignore` 非常相似：

```txt title=".dockerignore"
# 注释
*.log
.git/
node_modules/
```

每行一个模式，支持以下通配符：

| 模式 | 匹配 |
|---|---|
| `*` | 任意字符串（不含路径分隔符） |
| `**` | 任意字符串（含路径分隔符） |
| `?` | 任意单个字符 |
| `[abc]` | 任意单个字符（a、b 或 c） |
| `!pattern` | 否定，排除匹配的文件 |

---

## 为什么要用 .dockerignore

### 减少构建上下文大小

`docker build` 的第一步是**将构建上下文打包发送给 Docker daemon**。如果你的项目有 1GB 的 `node_modules`，每次构建都需要传输这 1GB——即使 Dockerfile 中根本没有 `COPY node_modules` 这条指令。

```bash
# 查看构建上下文大小
$ docker build . | grep "Sending build context"
Sending build context to Docker daemon  1.23 GB
```

### 避免敏感信息泄露

将 `.env`、`.pem`、`.key` 等敏感文件排除，避免被打包进镜像（即使最终没有 `COPY` 这些文件）。

### 加快构建速度

文件越少，构建越快。排除不必要的目录，可以充分利用 Docker 的构建缓存。

## 高级用法

### 排除特定文件但保留子目录

```txt title=".dockerignore"
# 排除所有 .log 文件
*.log

# 但保留 logs 目录本身（用于挂载）
logs/
```

### 多行模式

```txt title=".dockerignore"
# 排除所有测试相关文件
*test*
*tests*
*.test.*
*.spec.*
```

### 目录通配

```txt title=".dockerignore"
# 排除所有 node_modules（无论在哪个层级）
**/node_modules/

# 排除所有 __pycache__（无论在哪个层级）
**/__pycache__/

# 排除所有 .git 目录
**/.git/
```

### 排除后再包含（恢复）

```txt title=".dockerignore"
# 排除所有 node_modules
node_modules/

# 但保留 node_modules/@types
!node_modules/@types/
```

:::warning 注意：恢复规则必须在排除规则之后
在 `.dockerignore` 中，`!` 前置模式必须严格写在对应的排除模式之后，否则不会生效。
:::

## 常见陷阱

### 陷阱一：排除了正在使用的文件

```txt title=".dockerignore"
# 错误：排除了 .env
.env
```

如果你在 Dockerfile 中有：

```dockerfile
COPY . ./
RUN chmod +x start.sh
```

而 `start.sh` 依赖 `.env` 中的环境变量——但 `.env` 已经被排除了。这种情况需要显式 COPY：

```dockerfile
COPY .env.production .env
COPY start.sh .
```

### 陷阱二：排除了 Dockerfile

```txt title=".dockerignore"
# 错误：Dockerfile 被排除，docker build 会报错
Dockerfile
docker-entrypoint.sh
```

### 陷阱三：使用了 .dockerignore 但忘了 .dockerignore 本身

`.dockerignore` 文件本身不会被排除，这是正确的。但如果你的 Docker daemon 在远程机器上（Docker BuildKit remote builder），要确保 `.dockerignore` 也被正确发送。

## 调试技巧

如果构建结果不符合预期，可以用以下方法调试：

```bash
# 查看构建上下文实际包含了哪些文件
docker build --no-cache -t debug . 2>&1 | grep -E "^\[.*\] .+ ->" || \
  tar -C . -cvf - --exclude=.dockerignore --exclude=.git | tar -tvf -

# 使用 .dockerignore 的 dry-run（需要 BuildKit）
DOCKER_BUILDKIT=0 docker build --print-fix . 2>&1 || true
```

## 最佳实践总结

1. **始终创建 .dockerignore**：即使是最简单的项目
2. **优先排除大目录**：`node_modules/`、`vendor/`、`.m2/`、`target/`
3. **排除敏感文件**：`.env`（除非需要）、密钥文件、日志文件
4. **不要排除 Dockerfile 和构建脚本**：构建过程必需的文件
5. **按项目类型定制**：Node.js、Java、Go、Python 的排除规则不同
6. **定期检查**：随着项目增长，及时更新 .dockerignore

`.dockerignore` 是 Docker 构建优化的第一步。配置好它，可以让你的构建速度提升数倍。
