# Docker 网络

两个 Docker 容器之间如何通信？容器如何访问外部网络？外部请求如何到达容器？这一切都依赖 Docker 的网络模型。

## Docker 网络驱动概述

Docker 提供了五种内置网络驱动：

| 驱动 | 说明 | 适用场景 |
|---|---|---|
| **bridge** | 默认驱动，容器连接到 docker0 网桥 | 单主机容器通信 |
| **host** | 容器直接使用宿主机网络 | 性能敏感场景（跳过网络隔离） |
| **container** | 容器共享另一个容器的网络栈 | 容器间共享网络 |
| **none** | 完全禁用网络 | 网络完全不需要的场景 |
| **overlay** | 跨主机容器网络（Docker Swarm） | 多主机容器通信 |
| **macvlan** | 为容器分配真实 MAC 地址 | 需要直接暴露在物理网络的场景 |

## bridge 网络：默认驱动

当你不指定网络时，容器会连接到默认的 `bridge` 网络。

### 工作原理

```
宿主机网络栈
│
├── eth0 (宿主机物理/虚拟网卡)
│
├── docker0 网桥 (172.17.0.1/16)
│   │
│   ├── veth_pair_A ─── Container A (172.17.0.2)
│   │
│   └── veth_pair_B ─── Container B (172.17.0.3)
│
└── NAT/iptables (访问外部网络)
```

**核心组件**：

- **docker0**：Linux 网桥，充当容器网络的交换机
- **veth pair**：虚拟以太网对，一端在容器内，一端在宿主机
- **iptables**：NAT 规则，实现容器与外部网络的通信

### 容器间通信

```bash
# 两个容器在默认 bridge 网络中
docker run -d --name web nginx:alpine
docker run -d --name db redis:alpine

# 容器间通过容器名通信（DNS 自动解析）
docker exec web ping -c 1 db

# 也可以通过 IP 地址
docker exec web ping -c 1 172.17.0.3
```

### 自定义 bridge 网络

默认 bridge 网络的局限：容器只能通过 IP 地址通信，不支持 DNS 自动解析。自定义 bridge 网络解决了这个问题：

```bash
# 创建自定义 bridge 网络
docker network create mynetwork

# 在自定义网络中创建容器
docker run -d --network mynetwork --name web nginx:alpine
docker run -d --network mynetwork --name db redis:alpine

# 现在可以通过容器名通信
docker exec web ping -c 1 db
```

```yaml title="docker-compose.yml"
services:
  web:
    image: nginx:alpine
    networks:
      - mynetwork
  db:
    image: redis:alpine
    networks:
      - mynetwork

networks:
  mynetwork:
    driver: bridge
```

## host 网络：共享宿主机网络

容器直接使用宿主机的网络命名空间，跳过 Docker 的网络隔离。

```bash
# 使用 host 网络
docker run -d --network host --name nginx nginx:alpine
```

```
┌─────────────────────────────────────┐
│          宿主机网络命名空间            │
│                                     │
│  eth0                               │
│  docker0 (不存在)                    │
│                                     │
│  ┌─────────────────────────────┐   │
│  │      Container (host 模式)     │   │
│  │      nginx (:80, :443)        │   │
│  └─────────────────────────────┘   │
│                                     │
└─────────────────────────────────────┘
```

**特点**：

- 容器直接绑定宿主机的端口
- 性能最佳（无额外网络层）
- 网络隔离完全绕过
- 端口冲突风险（容器与宿主机或其他容器）

:::warning host 网络的局限
host 网络模式在 macOS 和 Windows 的 Docker Desktop 中行为不同（因为实际运行在 Linux 虚拟机中）。
:::

## container 网络：共享网络栈

一个容器共享另一个容器的网络命名空间。

```bash
# 创建基础容器
docker run -d --name app nginx:alpine

# 创建 sidecar 容器，共享 app 的网络
docker run -d --network container:app --name logger alpine sh -c "while true; do sleep 1; done"
```

```
┌─────────────────────────────────────┐
│       共享的网络命名空间              │
│                                     │
│  eth0 (同一 IP)                      │
│  ports (共享)                        │
│                                     │
│  ┌──────────┐    ┌──────────┐       │
│  │   app   │    │  logger  │       │
│  │ nginx   │    │ alpine   │       │
│  └──────────┘    └──────────┘       │
│                                     │
└─────────────────────────────────────┘
```

适用场景：

- 边车模式（sidecar）：辅助容器与主容器共享网络
- 日志收集器、监控代理等辅助进程

## none 网络：禁用网络

容器完全隔离，没有任何网络接口（除了 loopback）。

```bash
docker run -d --network none --name isolated alpine ls
```

适用场景：

- 批处理任务
- 需要完全离线运行的容器
- 定时任务

## overlay 网络：跨主机通信

overlay 网络用于 Docker Swarm 集群中容器跨主机通信。

```bash
# 初始化 Swarm（自动创建 overlay 网络）
docker swarm init

# 创建 overlay 网络
docker network create -d overlay myoverlay

# 部署服务
docker service create --network myoverlay --name web nginx:alpine
```

底层依赖：

- **VxLAN**：在物理网络上创建虚拟二层网络
- **GENEVE**：通用网络虚拟化封装
- **Consul/Etcd/ZK**：分布式键值存储，用于网络状态同步

## macvlan 网络：真实网络直通

macvlan 为每个容器分配真实的 MAC 地址和 IP，容器直接暴露在物理网络中。

```bash
# 创建 macvlan 网络（需要宿主机网卡支持）
docker network create -d macvlan \
  --subnet=192.168.1.0/24 \
  --gateway=192.168.1.1 \
  -o parent=eth0 \
  mymacvlan

# 创建容器
docker run -d --network mymacvlan --ip=192.168.1.100 alpine:latest
```

适用场景：

- 容器需要直接响应物理网络 ARP
- 旧式应用需要直接网络访问
- 性能要求极高的场景

:::warning 权限要求
macvlan 需要宿主机网卡支持混杂模式，可能需要额外配置。
:::

## 端口映射与发布

容器端口默认不对外可见，需要通过 `-p` 参数发布端口：

```bash
# 将容器 80 端口映射到宿主机的 8080
docker run -d -p 8080:80 --name web nginx:alpine

# 映射到宿主机随机端口
docker run -d -p 80 --name web nginx:alpine

# 指定 IP 和端口
docker run -d -p 127.0.0.1:8080:80 --name web nginx:alpine

# UDP 端口
docker run -d -p 53:53/udp --name dns bind9
```

### 端口发布模式

Docker 支持两种端口发布模式：

| 模式 | 说明 | 命令 |
|---|---|---|
| `ingress`（默认） | 通过 Docker 内部的负载均衡路由 | `-p 8080:80` |
| `host` | 直接绑定到宿主机 | `--publish-mode host -p 8080:80` |

## DNS 与服务发现

Docker 内置了 DNS 服务器，支持以下域名解析：

- 容器 IP 地址
- 容器名称
- Docker Compose 服务名
- 用户自定义别名

```bash
# 创建自定义网络
docker network create mynet

# 启动容器（指定别名）
docker run -d --network mynet --network-alias=web --name web nginx:alpine
docker run -d --network mynet --network-alias=api --name api alpine

# 通过服务名访问
docker exec web ping -c 1 api
```

### DNS 缓存

:::warning DNS 解析缓存
Docker 内置 DNS 默认 TTL 是 600 秒。如果容器 IP 变化（如重启后），旧 IP 可能被缓存。

解决方案：
- 使用 Docker Compose，自动管理网络别名
- 使用 `--network-alias` 而非硬编码 IP
- 在需要动态发现的场景使用 `bridge` 网络
:::

## 网络隔离

通过自定义网络实现容器隔离：

```yaml title="docker-compose.yml"
services:
  frontend:
    networks:
      - frontend-net
  backend:
    networks:
      - frontend-net
      - backend-net
  database:
    networks:
      - backend-net

networks:
  frontend-net:
    driver: bridge
  backend-net:
    driver: bridge
```

这样：

- `frontend` 无法直接访问 `database`
- `backend` 可以同时访问两者
- 不同网络的容器完全隔离

## 网络命令

```bash
# 查看网络列表
docker network ls

# 查看网络详情
docker network inspect bridge

# 创建网络
docker network create -d bridge mynet

# 删除未使用的网络
docker network prune

# 将运行中的容器连接到网络
docker network connect mynet container-name

# 断开容器网络
docker network disconnect mynet container-name
```

## 总结

Docker 网络驱动各有特点：

| 驱动 | 特点 | 适用场景 |
|---|---|---|
| `bridge` | 默认，可自定义，支持 DNS | 单主机容器通信（**最常用**） |
| `host` | 无隔离，性能最佳 | 性能敏感的网络服务 |
| `container` | 共享网络栈 | sidecar 容器 |
| `none` | 完全禁用网络 | 离线批处理任务 |
| `overlay` | 跨主机，VxLAN | Docker Swarm 集群 |
| `macvlan` | 真实 MAC/IP | 需要物理网络直通的场景 |

理解这些网络模型的差异，是设计可靠容器化架构的基础。
