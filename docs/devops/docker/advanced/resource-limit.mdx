# 资源限制机制

当多个容器同时运行在同一台宿主机上时，如果没有限制，一个贪婪的容器可能会耗尽 CPU、内存、磁盘 I/O，导致其他容器性能下降甚至崩溃。

Docker 通过 **cgroups（Control Groups）** 实现资源限制，让容器「各守本分」。

## cgroups 简介

cgroups 是 Linux 内核的功能，将进程分组并限制、监控和隔离其资源使用。

```
宿主机
│
├── CPU 子系统 (cpu, cpuacct)
│   └── /
│       ├── container-1 (/sys/fs/cgroup/cpu/container-1)
│       │   ├── cpu.cfs_quota_us: 50000
│       │   └── cpu.cfs_period_us: 100000
│       └── container-2 (/sys/fs/cgroup/cpu/container-2)
│           ├── cpu.cfs_quota_us: 100000
│           └── cpu.cfs_period_us: 100000
│
├── 内存子系统 (memory)
│   └── /
│       ├── container-1 (/sys/fs/cgroup/memory/container-1)
│       └── container-2 (/sys/fs/cgroup/memory/container-2)
│
└── I/O 子系统 (blkio)
    └── /
        ├── container-1 (/sys/fs/cgroup/blkio/container-1)
        └── container-2 (/sys/fs/cgroup/blkio/container-2)
```

### Docker 与 cgroups

当你运行容器时，Docker 自动为容器创建 cgroups：

```bash
# 查看容器的 cgroup 信息
docker inspect --format '{{.HostConfig.CpuShares}}' container-name
docker inspect --format '{{.HostConfig.Memory}}' container-name

# 在宿主机上查看 cgroup
ls /sys/fs/cgroup/
ls /sys/fs/cgroup/cpu/docker/
ls /sys/fs/cgroup/memory/docker/
```

## CPU 限制

### --cpus：CPU 数量

限制容器可以使用的 CPU 核心数：

```bash
# 允许容器使用最多 2 个 CPU 核心
docker run -d --name web --cpus="2" nginx:alpine

# 允许使用 0.5 个 CPU（即 50% 单核）
docker run -d --name api --cpus="0.5" node:alpine
```

### --cpuset-cpus：绑定特定 CPU

将容器绑定到特定 CPU 核心：

```bash
# 只使用 CPU 0 和 CPU 1
docker run -d --name app --cpuset-cpus="0,1" nginx:alpine

# 使用 CPU 0-2（连续核心）
docker run -d --name app --cpuset-cpus="0-2" nginx:alpine
```

:::warning NUMA 亲和性
在 NUMA 架构的服务器上，将容器绑定到特定 CPU 时，应同时考虑内存亲和性，避免跨 NUMA 节点访问内存导致性能下降。
:::

### --cpu-shares：CPU 权重

设置容器的 CPU 相对权重（默认值 1024）：

```bash
# 容器 A 有 1024 权重
docker run -d --name app-a --cpu-shares=1024 nginx:alpine

# 容器 B 有 512 权重（容器 A 权重的一半）
docker run -d --name app-b --cpu-shares=512 nginx:alpine
```

当 CPU 资源紧张时：

- 容器 A 获得 2/3 的 CPU 时间
- 容器 B 获得 1/3 的 CPU 时间

:::tip 权重只在资源竞争时生效
如果 CPU 有空闲，所有容器都可以使用。`--cpu-shares` 只在多个容器竞争 CPU 时才起作用。
:::

### CPU 限制对比

| 参数 | 作用 | 使用场景 |
|---|---|---|
| `--cpus` | 限制使用的 CPU 核心数 | 限制容器可用的最大 CPU |
| `--cpuset-cpus` | 绑定到特定 CPU 核心 | 避免上下文切换，提高确定性 |
| `--cpu-shares` | 设置相对权重 | 公平调度，资源竞争时生效 |
| `--cpu-period` | CFS 周期 | 与 `--cpu-quota` 配合使用 |
| `--cpu-quota` | CFS 配额 | 与 `--cpu-period` 配合使用 |

## 内存限制

### -m / --memory：内存限制

```bash
# 限制容器最多使用 512MB 内存
docker run -d -m="512m" --name app nginx:alpine

# 使用更精确的单位
docker run -d -m="1g" --name app nginx:alpine
docker run -d -m="128m" --name app nginx:alpine
```

### --memory-reservation：软限制

```bash
# 硬限制 1GB，运行时建议 512MB
docker run -d -m="1g" --memory-reservation="512m" --name app nginx:alpine
```

当内存充足时，容器可以使用更多内存；当内存紧张时，Docker 会尝试将容器内存降到 reservation 值。

### --memory-swap：交换分区限制

```bash
# 内存限制 512MB，swap 限制 512MB（总共可使用 1GB）
docker run -d -m="512m" --memory-swap="1g" --name app nginx:alpine

# --memory-swap 等于 --memory（禁用 swap）
docker run -d -m="512m" --memory-swap="512m" --name app nginx:alpine

# --memory-swap 为 -1（可以使用无限 swap）
docker run -d -m="512m" --memory-swap="-1" --name app nginx:alpine
```

:::warning swap 行为
`--memory-swap` 的值必须大于等于 `--memory`。差值就是容器可以使用的 swap 大小。
:::

### --memory-swappiness：swap 倾向

```bash
# 设置 swap 倾向为 0（尽量避免使用 swap）
docker run -d -m="512m" --memory-swappiness=0 --name app nginx:alpine
```

| 值 | 行为 |
|---|---|
| `0` | 禁止使用 swap |
| `1-100` | 越低越倾向使用物理内存 |
| 默认 `60` | 系统默认值 |

### --memory-oom-kill-disable：禁用 OOM Kill

```bash
# 禁用 OOM Kill（内存耗尽时暂停而非杀死）
docker run -d -m="512m" --memory-oom-kill-disable=false --name app nginx:alpine
```

:::danger 慎用 OOM Kill 禁用
禁用 OOM Kill 后，容器内存耗尽时会被暂停（`oom_kill_disable` 状态）。如果同时禁用，容器可能无响应。请确保监控内存使用。
:::

## 块 I/O 限制

### --blkio-weight：I/O 权重

```bash
# 设置 I/O 权重（默认 500，范围 10-1000）
docker run -d --blkio-weight=500 --name app-a nginx:alpine
docker run -d --blkio-weight=1000 --name app-b nginx:alpine
```

### --device-read-bps / --device-write-bps：限制读写速率

```bash
# 限制读取速度为 1MB/s
docker run -d \
  --device-read-bps /dev/sda:1mb \
  --name app \
  nginx:alpine

# 限制写入速度为 10MB/s
docker run -d \
  --device-write-bps /dev/sda:10mb \
  --name app \
  nginx:alpine
```

### --device-read-iops / --device-write-iops：限制 IOPS

```bash
# 限制读取 IOPS 为 100
docker run -d \
  --device-read-iops /dev/sda:100 \
  --name app \
  nginx:alpine
```

## PID 限制

限制容器内可以创建的进程数：

```bash
# 限制最大进程数为 1000
docker run -d --pids-limit=1000 --name app nginx:alpine

# 不限制（默认）
docker run -d --pids-limit=-1 --name app nginx:alpine
```

:::tip PID 限制的用途
限制 PID 可以防止容器内的进程耗尽宿主机 PID 资源，也可以防止 fork 炸弹攻击。
:::

## 完整示例

```bash
docker run -d \
  --name production-app \
  --restart unless-stopped \
  \
  # CPU 限制
  --cpus="2" \
  --cpuset-cpus="0-1" \
  --cpu-shares=1024 \
  \
  # 内存限制
  -m="1g" \
  --memory-reservation="512m" \
  --memory-swap="2g" \
  --memory-swappiness=10 \
  --memory-oom-kill-disable=false \
  \
  # I/O 限制
  --blkio-weight=500 \
  --device-read-bps /dev/sda:10mb \
  --device-write-bps /dev/sda:5mb \
  \
  # PID 限制
  --pids-limit=1000 \
  \
  nginx:alpine
```

## 运行时修改限制

```bash
# 更新容器资源限制
docker update --cpus="1" --memory="512m" production-app

# 批量更新多个容器
docker update --cpus="1" app1 app2 app3
```

## 监控资源使用

```bash
# 查看所有容器资源使用
docker stats

# 查看特定容器（实时）
docker stats --no-stream app

# 查看所有容器（仅一次）
docker stats --no-stream

# 查看详细信息
docker inspect --format '{{json .HostConfig}}' app | jq
```

## cgroup v2

新版 Linux 使用 cgroup v2（unified hierarchy），Docker 对其有更好的支持：

```bash
# 检查是否使用 cgroup v2
cat /sys/fs/cgroup/cgroup.controllers
# 如果支持，说明是 v2

# Docker 18.09+ 支持 cgroup v2
docker info | grep "Cgroup Version"
```

cgroup v2 的改进：

- 统一的层次结构
- 更好的资源隔离
- 改进的内存压力检测
- 支持压力感知（pressure stall information）

## 总结

Docker 通过 cgroups 实现全面的资源限制：

| 资源 | 参数 | 说明 |
|---|---|---|
| CPU | `--cpus` | CPU 核心数 |
| CPU | `--cpuset-cpus` | 绑定 CPU |
| CPU | `--cpu-shares` | 相对权重 |
| 内存 | `-m` | 内存限制 |
| 内存 | `--memory-swap` | swap 限制 |
| 内存 | `--memory-reservation` | 软限制 |
| I/O | `--blkio-weight` | I/O 权重 |
| I/O | `--device-read-bps` | 读速率限制 |
| PID | `--pids-limit` | 最大进程数 |

合理的资源限制是保证容器稳定运行、避免资源争抢的关键。
