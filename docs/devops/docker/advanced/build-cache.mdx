# 构建缓存

Docker 构建速度的快慢，很大程度上取决于缓存的利用。理解缓存失效机制，是写出高效 Dockerfile 的前提。

## 缓存工作原理

Docker 构建时按顺序处理 Dockerfile 的每一行。当一条指令与上一次构建完全相同时，Docker 直接使用缓存的结果。

### 缓存命中流程

```dockerfile
# 假设这是上一次构建的 Dockerfile
FROM ubuntu:20.04          # 缓存命中（基础镜像相同）
WORKDIR /app                # 缓存命中
COPY package.json .         # 检查 package.json 内容（未变）→ 缓存命中
RUN npm install             # 缓存命中（COPY 的内容未变）
COPY src/ ./src/            # 检查 src/ 内容（未变）→ 缓存命中
```

### 缓存失效规则

Docker 检查缓存时，会比较：

- **FROM 镜像**：镜像 ID 是否变化
- **COPY/ADD**：源文件的内容（校验和）是否变化
- **RUN**：无缓存（RUN 指令本身总是执行）
- **ENV/ARG**：值是否变化

只要某一层变化，该层及其之后的所有层都会重新构建。

## 层顺序优化

Dockerfile 编写顺序直接影响缓存利用率。

### 频率排序原则

**变化少的放前面，变化多的放后面**：

```dockerfile
# ✅ 好的写法
# 1. 依赖文件（不常变化）
COPY package.json package-lock.json ./
RUN npm ci --only=production

# 2. 构建产物配置
COPY .eslintrc .prettierrc ./
COPY tsconfig.json ./

# 3. 源代码（频繁变化）
COPY src/ ./src/

# 4. 构建
RUN npm run build
```

```dockerfile
# ❌ 不好的写法
# 源代码变化会导致重新安装依赖
COPY . .                     # 源代码在依赖前面
RUN npm ci                   # 依赖在源代码后面
```

### 典型模式

Node.js 应用：

```dockerfile
# 1. 复制依赖清单
COPY package*.json ./

# 2. 安装依赖
RUN npm ci --only=production

# 3. 复制代码
COPY . .

# 4. 构建
RUN npm run build
```

Python 应用：

```dockerfile
# 1. 复制依赖文件
COPY requirements.txt ./

# 2. 安装依赖
RUN pip install --no-cache-dir -r requirements.txt

# 3. 复制代码
COPY . .

# 4. 构建
RUN python setup.py build
```

Go 应用：

```dockerfile
# 1. 复制 go.mod
COPY go.mod go.sum ./

# 2. 下载依赖
RUN go mod download

# 3. 复制代码
COPY . .

# 4. 构建
RUN go build -o myapp .
```

## COPY 指令优化

### 精确控制复制范围

```dockerfile
# ❌ 不好：复制所有文件，可能包含大文件或敏感文件
COPY . .

# ✅ 好：只复制需要的文件
COPY src/ ./src/
COPY public/ ./public/
COPY package.json ./

# ✅ 更好：使用 .dockerignore 排除
COPY . .
```

### COPY 与 ADD 的选择

| 场景 | 推荐 | 原因 |
|---|---|---|
| 复制本地文件 | `COPY` | 语义明确 |
| 复制 tar 文件并解压 | `ADD` | 自动解压 |
| 复制远程 URL | `ADD` | 支持 URL |

```dockerfile
# 复制本地 tar
ADD archive.tar.gz /opt/

# 复制并解压
ADD https://example.com/binary.tar.gz /opt/
```

## RUN 指令优化

### 合并 RUN 层

```dockerfile
# ❌ 多层 RUN（产生多个镜像层）
RUN apt-get update
RUN apt-get install -y curl
RUN apt-get install -y git
RUN apt-get clean

# ✅ 单层 RUN（只产生一个镜像层）
RUN apt-get update && \
    apt-get install -y --no-install-recommends \
        curl \
        git && \
    apt-get clean && \
    rm -rf /var/lib/apt/lists/*
```

### 清理缓存

```bash
# apt-get
rm -rf /var/lib/apt/lists/*
apt-get clean

# npm
npm cache clean --force
npm ci --prefer-offline

# pip
pip install --no-cache-dir

# go
go mod download
```

## BuildKit 缓存

BuildKit 提供了更强大的缓存机制。

### 缓存类型

| 类型 | 说明 | 使用方式 |
|---|---|---|
| `type=cache` | 持久化缓存目录 | `--mount=type=cache,target=/path` |
| `type=secret` | 安全注入密钥 | `--mount=type=secret,id=xxx` |
| `type=ssh` | SSH 密钥转发 | `--mount=type=ssh` |
| `type=tmpfs` | 临时文件系统 | `--mount=type=tmpfs` |

### 缓存挂载

```dockerfile
# npm 缓存（跨构建持久化）
RUN --mount=type=cache,target=/root/.npm \
    npm ci --only=production

# pip 缓存
RUN --mount=type=cache,target=/root/.cache/pip \
    pip install -r requirements.txt

# go 模块缓存
RUN --mount=type=cache,target=/go/pkg/mod \
    go build -o myapp .

# maven 缓存
RUN --mount=type=cache,target=/root/.m2 \
    mvn dependency:go-offline
```

### 缓存隔离

BuildKit 的缓存是构建阶段的，不进入最终镜像：

```dockerfile
# npm 缓存不会进入最终镜像
RUN --mount=type=cache,target=/root/.npm \
    npm ci --only=production

# 最终镜像只包含 node_modules
# 不包含 npm 缓存
```

## 强制重新构建

### 跳过缓存

```bash
# 完全跳过缓存（不推荐，性能差）
docker build --no-cache .

# 仅跳过特定步骤后的缓存
docker build --build-arg CACHE_DATE=$(date +%s) .
```

```dockerfile
ARG CACHE_DATE=1
RUN echo $CACHE_DATE
```

### 选择性重建

BuildKit 允许只重建变化的层：

```bash
# 使用 buildx
docker buildx build \
  --build-arg BUILDKIT_INLINE_CACHE=1 \
  --cache-from myregistry/myapp:latest \
  -t myregistry/myapp:new \
  .
```

## Registry 缓存

### 镜像层缓存

```bash
# 配置 Registry 缓存
docker buildx create \
  --name mybuilder \
  --driver docker-container \
  --use

# 使用远程缓存
docker buildx build \
  --cache-from myregistry/myapp:buildcache \
  --cache-to myregistry/myapp:buildcache \
  -t myregistry/myapp:latest \
  .
```

### 多平台构建缓存

```bash
# 创建多平台 builder
docker buildx create \
  --name mybuilder \
  --driver docker-container \
  --use \
  --platform linux/amd64,linux/arm64

# 构建时自动复用缓存
docker buildx build \
  --push \
  --cache-from myregistry/myapp:cache \
  --cache-to type=registry,ref=myregistry/myapp:cache \
  --tag myregistry/myapp:latest \
  --platform linux/amd64,linux/arm64 \
  .
```

## 常见缓存问题

### 1. 依赖层变化导致全量重建

```dockerfile
# ❌ 问题：package.json 在 COPY . . 中
COPY . .
RUN npm ci

# ✅ 解决：先单独复制依赖文件
COPY package*.json ./
RUN npm ci
COPY . .
```

### 2. .dockerignore 遗漏

```bash title=".dockerignore"
# 确保排除不必要文件
node_modules/
.git/
*.log
.env
dist/
```

### 3. 时间戳导致缓存失效

```dockerfile
# ❌ 问题：COPY 时文件时间戳不同
COPY . .

# ✅ 解决：使用 buildkit 的复制选项
COPY --checksum . ./
```

## 缓存验证

```bash
# 查看构建过程
docker build --progress=plain .

# 检查哪些层使用了缓存
# 输出中的 CACHED 表示使用缓存

# 查看镜像层
docker history myapp:latest
```

## 总结

构建缓存优化的核心原则：

1. **层顺序优化**：变化少的放前面，变化多的放后面
2. **COPY 精确控制**：只复制需要的文件，配合 .dockerignore
3. **RUN 合并**：减少镜像层，清理缓存
4. **BuildKit**：使用 `--mount=type=cache` 持久化依赖缓存
5. **Registry 缓存**：跨构建、跨机器复用镜像层

正确的缓存策略可以将构建时间从几分钟缩短到几十秒。
