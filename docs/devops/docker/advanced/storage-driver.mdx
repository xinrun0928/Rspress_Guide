# 存储驱动原理

Docker 如何在只读的镜像层之上，为每个容器创建一个可写的文件系统？为什么修改一个文件有时会变慢？不同存储驱动的性能差异从何而来？

答案藏在 Docker 的存储驱动中。

## 存储驱动概述

Docker 使用存储驱动管理镜像层和容器可写层。目前主流的存储驱动有：

| 驱动 | 全称 | 状态 |
|---|---|---|
| **overlay2** | OverlayFS v2 | 生产环境推荐 |
| **fuse-overlayfs** | FUSE-overlayfs | 非特权容器 |
| **btrfs** | Btrfs | 需要 Btrfs 文件系统 |
| **zfs** | ZFS | 需要 ZFS 文件系统 |
| **vfs** | Virtual File System | 测试环境 |
| **devicemapper** | Device Mapper | 遗留系统 |

```bash
# 查看当前存储驱动
docker info | grep "Storage Driver"

# 查看支持哪些存储驱动
docker info | grep "Storage"
```

## overlay2：生产环境首选

overlay2 是目前最推荐的存储驱动，适用于大多数 Linux 发行版。

### 工作原理

overlay2 将文件系统分为三层：

- **lowerdir**：只读层（镜像层）
- **upperdir**：可写层（容器层）
- **merged**：合并视图（容器看到的样子）

```
overlay2 文件系统结构：

/var/lib/docker/overlay2/
├── l/                          # 软链接
├── <container-id>/
│   ├── lower-id-1/             # 镜像层 1
│   ├── lower-id-2/             # 镜像层 2
│   ├── upper/                  # 可写层（容器独有）
│   ├── work/                   # workdir（内部使用）
│   └── merged/                 # 合并视图
└── <image-hash>/
    └── diff/                   # 镜像层数据
```

### 文件操作示例

```bash
# 场景 1：读取镜像中的文件
# 文件位于 lowerdir，直接读取，无复制
cat /merged/etc/nginx/nginx.conf

# 场景 2：修改镜像中的文件（如 /etc/hosts）
# 1. 从 lowerdir 复制到 upperdir
# 2. 在 upperdir 中修改
# 3. 后续读取从 upperdir 返回
echo "127.0.0.1 myhost" >> /merged/etc/hosts

# 场景 3：新建文件
# 直接写入 upperdir
touch /merged/var/log/myapp.log
```

### Copy-on-Write 性能

overlay2 的 CoW 操作会带来轻微性能开销：

| 操作 | 性能影响 | 说明 |
|---|---|---|
| 首次读取 lowerdir 文件 | 无额外开销 | 直接从 lowerdir 读取 |
| 首次修改 lowerdir 文件 | 复制到 upperdir | 需要复制文件内容 |
| 后续读写 | 无额外开销 | 都在 upperdir |
| 删除 lowerdir 文件 | 创建 whiteout | 轻量操作 |

### 优化建议

- 频繁写入的文件放在 Volume 中
- 避免在 overlay2 层存储大文件
- 使用 `tmpfs` 存储临时文件

## fuse-overlayfs

fuse-overlayfs 是为非特权容器（如 rootless Docker）设计的存储驱动，模拟 overlay2 的功能。

```bash
# 启用 fuse-overlayfs
# /etc/docker/daemon.json
{
  "storage-driver": "fuse-overlayfs"
}
```

:::warning 性能差异
fuse-overlayfs 通过 FUSE 实现，性能略低于原生 overlay2，适合无法使用 overlay2 的场景。
:::

## btrfs 驱动

btrfs 驱动使用 Btrfs 文件系统的快照功能存储容器层。

### 特点

```bash
# 需要的挂载选项
/dev/sda1 /var/lib/docker btrfs defaults,autodefrag 0 0
```

| 特性 | 说明 |
|---|---|
| 快照 | 每个容器层是 Btrfs 快照 |
| 压缩 | 支持透明压缩 |
| 去重 | 支持块级去重 |
| 配额 | 支持子卷配额 |

### 性能

- 快照创建极快（微秒级）
- 元数据性能好
- 支持压缩节省空间
- 需要完整的 Btrfs 文件系统支持

## zfs 驱动

zfs 驱动使用 ZFS 文件系统的克隆和快照功能。

```bash
# 需要的挂载选项
/dev/sdb /var/lib/docker zfs 0 0
```

| 特性 | 说明 |
|---|---|
| 克隆 | 使用 ZFS clone |
| 压缩 | 支持 LZ4 压缩 |
| 配额 | 支持 datasets 配额 |
| 校验 | 数据完整性校验 |

### 性能

- 写入性能优秀
- 快照创建快
- 内存消耗较高
- 适合大文件场景

## devicemapper 驱动

devicemapper 是较老的存储驱动，通过块设备映射实现存储。

```bash
# 配置 devicemapper
# /etc/docker/daemon.json
{
  "storage-driver": "devicemapper",
  "storage-opts": [
    "dm.basesize=20G",
    "dm.loopdatasize=200G"
  ]
}
```

| 特性 | 说明 |
|---|---|
| 块级操作 | 直接操作块设备 |
| thin pool | 使用 LVM thin provisioning |
| 性能 | 随机读写性能好 |
| 状态 | 遗留驱动，不推荐新部署 |

:::warning devicemapper
devicemapper 在 RHEL/CentOS 7 中曾是默认驱动，但现在已被 overlay2 取代。新部署不推荐使用。
:::

## vfs 驱动

vfs 驱动是最简单的存储驱动，每个容器直接复制一份完整文件系统。

```bash
# 配置 vfs
# /etc/docker/daemon.json
{
  "storage-driver": "vfs"
}
```

| 特性 | 说明 |
|---|---|
| 简单 | 不依赖 UnionFS |
| 性能 | 最差，无层共享 |
| 存储 | 每个容器独占完整镜像副本 |
| 用途 | 仅用于测试 |

:::danger vfs 性能问题
vfs 驱动的存储效率极低，每个容器都会复制一份完整镜像。仅用于开发测试，绝不用于生产。
:::

## 存储驱动对比

| 特性 | overlay2 | btrfs | zfs | devicemapper | vfs |
|---|---|---|---|---|---|
| 状态 | 生产推荐 | 可用 | 可用 | 遗留 | 仅测试 |
| 性能 | 好 | 好 | 好 | 好 | 差 |
| 层共享 | 是 | 是 | 是 | 部分 | 否 |
| 存储效率 | 高 | 高 | 高 | 中 | 低 |
| 快照支持 | 有限 | 完整 | 完整 | 有限 | 无 |
| 内存开销 | 低 | 中 | 高 | 中 | 低 |
| 配额 | 有限 | 支持 | 支持 | 支持 | 无 |

## 如何选择存储驱动

### 推荐选择

```bash
# Linux 内核 4.0+ 且文件系统支持
{
  "storage-driver": "overlay2"
}
```

### 特殊场景

| 场景 | 推荐驱动 |
|---|---|
| 普通生产环境 | overlay2 |
| 非特权容器 | fuse-overlayfs |
| 需要配额管理 | zfs 或 btrfs |
| 需要压缩 | btrfs 或 zfs |
| RHEL/CentOS 7（老系统） | devicemapper（已不再推荐） |
| 开发测试 | vfs 或 overlay2 |

## 清理存储空间

```bash
# 清理未使用的镜像、容器、网络
docker system prune

# 清理所有未使用的资源（包括卷）
docker system prune -a

# 清理构建缓存
docker builder prune

# 清理特定资源
docker image prune          # 未使用的镜像
docker container prune      # 停止的容器
docker volume prune         # 未使用的卷
docker network prune        # 未使用的网络
```

## 总结

存储驱动是 Docker 镜像层和容器层管理的核心：

- **overlay2**：生产环境首选，平衡性能和功能
- **btrfs/zfs**：需要特定文件系统，支持配额和压缩
- **fuse-overlayfs**：非特权容器场景
- **vfs**：仅用于测试

理解存储驱动的工作原理，有助于选择合适的驱动类型，并针对特定场景进行优化。
