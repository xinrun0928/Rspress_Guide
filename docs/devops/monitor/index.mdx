# 可观测性与监控体系

「服务器 CPU 飙到 100%，你的用户比你先发现。」

这不是段子，是很多运维工程师的日常。监控系统没配好，等用户投诉了才知道服务挂了——轻则影响体验，重则造成业务损失。

现代可观测性体系由三个支柱构成：**指标（Metrics）**、**日志（Logs）**、**链路追踪（Traces）**。Prometheus 负责指标，ELK/EFK/Loki 负责日志，Jaeger/Zipkin 负责链路追踪。这三者组合起来，才能真正做到「系统出问题，快速定位根因」。

## 三支柱：指标、日志、链路

```
┌─────────────────────────────────────────────────────────────────┐
│                        可观测性三支柱                              │
│                                                                  │
│  ┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐ │
│  │     指标         │  │     日志         │  │     链路         │ │
│  │   (Metrics)     │  │    (Logs)       │  │   (Traces)      │ │
│  │                  │  │                  │  │                  │ │
│  │ CPU、内存、QPS   │  │ 请求记录、错误堆栈│  │ 请求调用链       │ │
│  │ 聚合高效、适合告警│  │ 信息丰富、适合排查│  │ 端到端、适合分析 │ │
│  │                  │  │                  │  │                  │ │
│  │  Prometheus     │  │ ELK/EFK/Loki     │  │ Jaeger/Zipkin   │ │
│  └────────┬────────┘  └────────┬────────┘  └────────┬────────┘ │
│           │                       │                       │         │
│           └───────────────────────┴───────────────────────┘         │
│                               │                                     │
│                    「我的系统现在怎么样？」                            │
└─────────────────────────────────────────────────────────────────┘
```

**指标**告诉你「系统出问题了」，**日志**告诉你「问题在哪」，**链路追踪**告诉你「为什么这条路这么慢」。

三者缺一不可。仅有指标，你只知道服务挂了；仅有日志，你大海捞针；仅有链路，你看不到全貌。

## 学习路径建议

```
第一阶段：Prometheus 入门（3-4 天）
→ 理解 Pull 模式 vs Push 模式
→ 掌握四大指标类型：Counter、Gauge、Histogram、Summary
→ 编写第一个 PromQL 查询
→ 搭建 Prometheus + Grafana 监控栈

第二阶段：Exporter 与服务发现（2-3 天）
→ 部署 Node Exporter、MySQL Exporter
→ 配置 Kubernetes 服务发现
→ Blackbox Exporter 实现接口探测

第三阶段：日志体系（3-4 天）
→ ELK Stack 架构与组件职责
→ EFK vs Loki 的选型依据
→ Fluent Bit 日志收集配置
→ 结构化日志与字段提取

第四阶段：链路追踪与 SRE（2-3 天）
→ OpenTelemetry 接入
→ Jaeger 部署与分布式追踪
→ SLO/SLA 定义与告警设计
→ 故障复盘方法论
```

## 面试的核心逻辑

可观测性面试，核心考察三个层次：

1. **工具体系**：能不能说清楚 Prometheus 的 Pull 模式？为什么指标类告警比日志类告警更可靠？PromQL 的 Histogram 分位数是怎么计算的？
2. **设计能力**：Label 设计有什么原则？为什么过度使用 Label 会导致 Prometheus 性能问题？如何避免告警疲劳？
3. **排障思维**：系统变慢了，怎么用指标、日志、链路三种方式定位问题？三者各擅长什么场景？

```
# 可观测性排查公式

问题现象 → 指标定位（哪个服务/哪个时间段）
         → 日志深挖（错误堆栈、请求上下文）
         → 链路追踪（哪一步最慢、依赖调用耗时）
         → 根因定位
```

> "好的监控不是『出了事才知道』，而是『要出事之前就能预警』。告警规则设计的第一原则是：宁缺毋滥，告警疲劳比没告警更可怕。"
