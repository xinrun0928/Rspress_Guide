# KubeSphere 镜像构建与推送

"这个镜像构建要 10 分钟，太慢了！有没有办法加速？"

这是一个很实际的问题。镜像构建是 CI/CD 流水线中最耗时的环节之一，优化构建速度可以显著提升研发效率。

这一节，我们来学习如何高效地构建和推送镜像。

## Docker 构建基础

### Dockerfile 最佳实践

```dockerfile
# 1. 选择合适的基础镜像
FROM maven:3.8-openjdk-17 AS build

# 2. 利用缓存 - 先复制依赖文件
WORKDIR /app
COPY pom.xml .
RUN mvn dependency:go-offline -B

# 3. 再复制源代码
COPY src ./src
RUN mvn package -DskipTests

# 4. 多阶段构建 - 减小镜像体积
FROM openjdk:17-jre-slim
WORKDIR /app
COPY --from=build /app/target/my-app.jar ./app.jar

# 5. 非 root 用户运行
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
USER appuser

# 6. 健康检查
HEALTHCHECK --interval=30s --timeout=3s \
    CMD curl -f http://localhost:8080/health || exit 1

ENTRYPOINT ["java", "-jar", "app.jar"]
```

## BuildKit 加速构建

### 什么是 BuildKit

BuildKit 是 Docker 18.06+ 内置的新一代构建引擎，相比传统构建有显著优势：

| 特性 | 传统构建 | BuildKit |
|---|---|---|
| 并行构建 | 顺序执行 | 自动并行 |
| 增量构建 | 全量重建 | 智能缓存 |
| 传输效率 | 整层传输 | 按需传输 |
| 构建速度 | 慢 | 快 2-10x |

### 启用 BuildKit

```bash
# 临时启用
DOCKER_BUILDKIT=1 docker build .

# 永久启用
# /etc/docker/daemon.json
{
  "features": {
    "buildkit": true
  }
}
```

### BuildKit 流水线示例

```groovy
stage('构建镜像 (BuildKit)') {
    steps {
        container('docker') {
            sh '''
                # 启用 BuildKit
                export DOCKER_BUILDKIT=1
                export BUILDKIT_PROGRESS=plain
                
                # 构建并推送
                docker build \
                    --output type=image,name=${IMAGE_TAG},push=true \
                    --cache-from type=registry,ref=${CACHE_IMAGE} \
                    --cache-to type=registry,ref=${CACHE_IMAGE},mode=max \
                    .
            '''
        }
    }
}
```

## 多架构构建

### 为什么要多架构

<Badge text="跨平台支持" type="tip" /> 现代应用需要支持多种 CPU 架构（amd64、arm64）和操作系统。Docker Buildx 支持一次性构建多个平台。

### 使用 Buildx 构建多架构镜像

```groovy
stage('多架构构建') {
    steps {
        container('docker') {
            script {
                sh '''
                    # 启用 experimental 功能
                    export DOCKER_CLI_EXPERIMENTAL=enabled
                    
                    # 创建并使用 builder
                    docker buildx create --use --name mybuilder
                    docker buildx inspect --bootstrap
                    
                    # 构建多架构镜像
                    docker buildx build \
                        --platform linux/amd64,linux/arm64 \
                        --tag ${IMAGE_TAG} \
                        --push \
                        --cache-from type=registry,ref=${CACHE_IMAGE} \
                        --cache-to type=registry,ref=${CACHE_IMAGE} \
                        .
                '''
            }
        }
    }
}
```

## 构建缓存策略

### 缓存类型

| 缓存类型 | 说明 | 适用场景 |
|---|---|---|
| Dockerfile 层缓存 | 复用未变化的层 | 网络好、代码变动少 |
| Registry 缓存 | 将缓存镜像推送到仓库 | 分布式构建 |
| 本地缓存 | 使用本地缓存 | 单节点 Docker |

### 缓存最佳实践

```groovy
pipeline {
    environment {
        // 缓存镜像名称
        CACHE_IMAGE = "${DOCKER_REGISTRY}/cache/${APP_NAME}:buildcache"
    }
    
    stages {
        stage('准备缓存') {
            steps {
                script {
                    try {
                        // 尝试拉取之前的缓存
                        sh "docker pull ${CACHE_IMAGE} || true"
                    } catch (Exception e) {
                        echo "缓存拉取失败，使用默认缓存"
                    }
                }
            }
        }
        
        stage('构建') {
            steps {
                container('docker') {
                    sh '''
                        docker build \
                            --cache-from ${CACHE_IMAGE} \
                            --tag ${IMAGE_TAG} \
                            .
                    '''
                }
            }
        }
        
        stage('推送镜像和缓存') {
            steps {
                container('docker') {
                    sh '''
                        # 推送镜像
                        docker push ${IMAGE_TAG}
                        
                        # 推送缓存
                        docker push ${CACHE_IMAGE}
                    '''
                }
            }
        }
    }
}
```

## 分层构建优化

### Maven 项目优化

```dockerfile
# Stage 1: 构建
FROM maven:3.8-openjdk-17 AS builder
WORKDIR /app

# 先复制 pom.xml，利用 Maven 缓存
COPY pom.xml .
RUN mvn dependency:go-offline -B

# 再复制源码
COPY src ./src
RUN mvn package -DskipTests

# Stage 2: 运行
FROM openjdk:17-jre-slim
WORKDIR /app

# 只复制 jar 文件
COPY --from=builder /app/target/*.jar app.jar

EXPOSE 8080
ENTRYPOINT ["java", "-jar", "app.jar"]
```

### Node.js 项目优化

```dockerfile
# Stage 1: 构建
FROM node:18-alpine AS builder
WORKDIR /app

# 先复制 package 文件
COPY package*.json ./
RUN npm ci --only=production

# 再复制源码
COPY . .
RUN npm run build

# Stage 2: 运行
FROM node:18-alpine
WORKDIR /app

# 只复制必要文件
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
COPY package.json .

EXPOSE 3000
CMD ["node", "dist/index.js"]
```

## 镜像标签策略

### 标签生成规则

| 场景 | 标签格式 | 示例 |
|---|---|---|
| Git 分支构建 | `branch-name` | `main`、`feature-xxx` |
| 正式版本 | `v1.0.0` | `v1.0.0` |
| CI 构建 | `build-{number}` | `build-256` |
| Commit Hash | `{hash}` | `abc1234` |
| 日期版本 | `date-{date}` | `date-20231201` |

### 流水线中的标签生成

```groovy
environment {
    // 动态生成标签
    GIT_COMMIT = sh(script: 'git rev-parse --short HEAD', returnStdout: true).trim()
    BUILD_DATE = sh(script: 'date +%Y%m%d', returnStdout: true).trim()
    IMAGE_TAG = "${GIT_COMMIT}-${BUILD_DATE}"
    
    // 主版本
    MAJOR_TAG = "v1.0"
}

// 推送到不同标签
stages {
    stage('推送 latest') {
        steps {
            sh 'docker tag ${IMAGE_TAG} ${REGISTRY}:latest'
            sh 'docker push ${REGISTRY}:latest'
        }
    }
    
    stage('推送版本标签') {
        steps {
            sh 'docker tag ${IMAGE_TAG} ${REGISTRY}:${MAJOR_TAG}'
            sh 'docker push ${REGISTRY}:${MAJOR_TAG}'
        }
    }
}
```

## 安全构建

### 使用 Build Secrets

```dockerfile
# 语法: --mount=type=secret
FROM alpine

RUN --mount=type=secret,id=npmrc \
    cat /run/secrets/npmrc > ~/.npmrc && \
    npm install && \
    rm ~/.npmrc
```

### 安全扫描集成

```groovy
stage('安全扫描') {
    steps {
        container('docker') {
            script {
                sh '''
                    # 扫描镜像漏洞
                    docker run --rm -v /var/run/docker.sock:/var/run/docker.sock \
                        aquasec/trivy:latest \
                        image --severity HIGH,CRITICAL \
                        --exit-code 0 \
                        --no-progress \
                        ${IMAGE_TAG}
                '''
            }
        }
    }
    post {
        failure {
            error "镜像存在高危漏洞，构建终止"
        }
    }
}
```

## 构建时间优化清单

| 优化项 | 效果 | 实施难度 |
|---|---|---|
| 使用 BuildKit | 2-5x | 低 |
| 利用 Docker 缓存 | 5-10x | 低 |
| 多阶段构建 | 减小 50%+ | 中 |
| 并行构建 | 节省 30% | 中 |
| 高性能构建节点 | 节省 50% | 高 |
| 预热缓存 | 节省 40% | 中 |

<Badge text="性能建议" type="tip" /> 最有效的优化是合理利用 Docker 层缓存：把不常变化的步骤（如依赖安装）放在前面，把经常变化的步骤（如代码复制）放在后面。
