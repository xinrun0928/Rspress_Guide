# 部署策略：蓝绿、金丝雀与滚动更新

「如何在不停机的情况下发布新版本？」

这是每个运维工程师都会被问到的问题。答案是：选择正确的部署策略。

在 Kubernetes 环境中，主要有三种部署策略：滚动更新（Rolling Update）、蓝绿部署（Blue-Green）、金丝雀发布（Canary Release）。每种策略都有其适用场景和权衡。

## 三种部署策略对比

| 策略 | 停机时间 | 资源消耗 | 回滚速度 | 适用场景 |
|---|---|---|---|---|
| 滚动更新 | 无 | 低 | 较慢 | 大多数场景 |
| 蓝绿部署 | 无 | 2x | 即时 | 有状态服务、高风险变更 |
| 金丝雀发布 | 无 | 1.x | 渐进 | 大规模系统、特性验证 |

## 滚动更新：Kubernetes 原生支持

滚动更新是 Kubernetes Deployment 的默认策略，逐步替换旧版本 Pod，确保服务持续可用。

### 工作原理

```
时间线：
T0: [v1][v1][v1]     3 个 v1 Pod
T1: [v2][v1][v1]     新增 1 个 v2，逐步替换
T2: [v2][v2][v1]     继续替换
T3: [v2][v2][v2]     全部替换完成
```

### 配置示例

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 3
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1        # 最多超出期望副本数
      maxUnavailable: 0  # 始终保持 3 个可用
  template:
    spec:
      containers:
      - name: my-app
        image: my-app:v2
```

### 关键参数

| 参数 | 说明 | 建议值 |
|---|---|---|
| `maxSurge` | 更新过程中最多超出期望副本数 | 1 或 25% |
| `maxUnavailable` | 更新过程中最多不可用的副本数 | 0 或 25% |
| `minReadySeconds` | 新 Pod 必须保持就绪的最短时间 | 30s |

:::tip 滚动更新调优
- `maxSurge: 1, maxUnavailable: 0`：最保守，服务质量最高，但更新最慢
- `maxSurge: 25%, maxUnavailable: 25%`：平衡策略，适合大多数场景
- `maxSurge: 1, maxUnavailable: 1`：激进策略，更新最快，但服务能力下降明显
:::

## 蓝绿部署：双倍资源的即时切换

蓝绿部署保持两套完整环境（Blue 和 Green），新版本在 Green 环境验证通过后，流量一次性切换到 Green。

### 工作流程

```
┌─────────────────────────────────────────────────────────────┐
│                         Load Balancer                        │
│                           │                                  │
│                     流量切换 ▼                               │
│            Blue (v1)  ←────┼────→  Green (v2)               │
│              [生产]          │         [待验证]              │
│              [服务中]         │         [热备中]              │
└─────────────────────────────────────────────────────────────┘
```

### KubeSphere 蓝绿部署

KubeSphere 通过「无场部署」功能支持蓝绿部署策略：

1. **创建蓝绿部署**：
   - 创建两个 Deployment，一个标签为 `version: blue`，一个为 `version: green`
   - Service 选择其中一个标签作为 selector

2. **验证新版本**：
   - 测试 green 环境的应用是否正常
   - 检查日志、监控指标

3. **切换流量**：
   - 修改 Service selector，从 `version: blue` 改为 `version: green`
   - 切换是瞬时的，几乎零延迟

### 蓝绿部署配置

```yaml
# Blue Deployment (v1)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app-blue
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-app
      version: blue
  template:
    metadata:
      labels:
        app: my-app
        version: blue
    spec:
      containers:
      - name: my-app
        image: my-app:v1
---
# Green Deployment (v2)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app-green
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-app
      version: green
  template:
    metadata:
      labels:
        app: my-app
        version: green
    spec:
      containers:
      - name: my-app
        image: my-app:v2
---
# Service (当前指向 Blue)
apiVersion: v1
kind: Service
metadata:
  name: my-app
spec:
  selector:
    app: my-app
    version: blue  # 切换为 green 即可完成蓝绿切换
  ports:
  - port: 80
    targetPort: 8080
```

### 回滚操作

```bash
# 蓝绿回滚：将 Service selector 改回 blue
kubectl patch service my-app -p '{"spec":{"selector":{"version":"blue"}}}'
```

### 适用场景

| 场景 | 是否适合蓝绿 |
|---|---|
| 有状态服务，状态同步复杂 | ✅ 适合 |
| 需要完整验证后再切换 | ✅ 适合 |
| 数据库 schema 变更 | ⚠️ 需额外处理 |
| 资源成本敏感 | ❌ 不适合（需要双倍资源）|

## 金丝雀发布：渐进式灰度

金丝雀发布将新版本逐步推向用户，就像矿井工人用金丝雀探测瓦斯——先让「金丝雀」试试水，风险可控后再全面推广。

### 工作原理

```
流量分配：
5% ──→ v2 (金丝雀)
95% ──→ v1 (稳定版)

验证通过后：
20% ──→ v2
80% ──→ v1

最终：
100% ──→ v2
```

### KubeSphere 灰度发布

KubeSphere 内置灰度发布功能，支持基于流量比例和基于标签的灰度策略。

#### 方式一：基于流量权重的灰度

```yaml
apiVersion: flagger.app/v1beta1
kind: Canary
metadata:
  name: my-app
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: my-app
  progressDeadlineSeconds: 60
  metrics:
  - name: request-success-rate
    interval: 1m
    thresholdRange:
      min: 99
  - name: request-duration
    interval: 1m
    thresholdRange:
      max: 500
  analysis:
    interval: 1m
    maxWeight: 30
    stepWeight: 10
    # 自动分析指标，符合条件后自动提升权重
```

#### 方式二：基于 HTTP Header 的灰度

```yaml
apiVersion: networking.k8s.io/v1
kind: VirtualService
metadata:
  name: my-app
spec:
  hosts:
  - my-app
  http:
  - match:
    - headers:
        x-canary:          # 带有 x-canary: always 头的请求
          exact: always
    route:
    - destination:
        host: my-app
        subset: v2
    weight: 100
  - route:                 # 其他请求
    - destination:
        host: my-app
        subset: v1
      weight: 100
```

#### 方式三：基于权重的 Istio 灰度

```yaml
apiVersion: networking.istio.io/v1alpha3
kind: DestinationRule
metadata:
  name: my-app
spec:
  host: my-app
  subsets:
  - name: v1
    labels:
      version: v1
  - name: v2
    labels:
      version: v2
---
apiVersion: networking.istio.io/v1alpha3
kind: VirtualService
metadata:
  name: my-app
spec:
  host: my-app
  http:
  - route:
    - destination:
        host: my-app
        subset: v1
      weight: 90
    - destination:
        host: my-app
        subset: v2
      weight: 10   # 10% 流量到 v2
```

### 灰度策略配置

```yaml
# KubeSphere 灰度规则
apiVersion: sleeper.v2.migration.ks.io/v1alpha1
kind: StrategyConfig
metadata:
  name: canary-release
spec:
  canaryStrategy:
    static:
      percentage: 10   # 10% 流量到新版本
    headers:
      - name: x-user-id
        value: VIP-.*  # VIP 用户走新版本
```

### 自动分析与回滚

```yaml

  name: latency
```

## 部署策略选择指南

```
                    发布决策树
                        │
                        ▼
            ┌───────────────────────┐
            │  变更风险等级？         │
            └───────────────────────┘
                │           │           │
            低风险        中风险       高风险
                │           │           │
                ▼           ▼           ▼
        滚动更新      金丝雀发布     蓝绿部署
```

### 决策矩阵

| 问题 | 回答 | 推荐策略 |
|---|---|---|
| 团队成熟度？ | 初学者 | 滚动更新 |
| 资源预算？ | 有限 | 滚动更新 |
| 需要 A/B 测试？ | 是 | 金丝雀/Istio |
| 有状态服务？ | 是 | 蓝绿 |
| 需要即时回滚？ | 是 | 蓝绿 |
| 变更涉及数据库？ | 是 | 蓝绿 + 数据库迁移 |

## 回滚策略

### 滚动更新回滚

```bash
# 查看部署历史
kubectl rollout history deployment/my-app

# 回滚到上一版本
kubectl rollout undo deployment/my-app

# 回滚到指定版本
kubectl rollout undo deployment/my-app --to-revision=2
```

### 蓝绿回滚

```bash
# 即时回滚：修改 Service selector
kubectl patch service my-app -p '{"spec":{"selector":{"version":"blue"}}}'
```

### 金丝雀回滚

```bash
# 减少金丝雀权重
kubectl patch virtualservice my-app -p '{
  "spec": {
    "http": [{
      "route": [
        {"destination": {"host": "my-app", "subset": "v1"}, "weight": 100},
        {"destination": {"host": "my-app", "subset": "v2"}, "weight": 0}
      ]
    }]
  }
}'
```

## 生产环境最佳实践

### 1. 健康检查必须完善

```yaml
spec:
  containers:
  - name: my-app
    livenessProbe:
      httpGet:
        path: /health/live
        port: 8080
      initialDelaySeconds: 30
      periodSeconds: 10
    readinessProbe:
      httpGet:
        path: /health/ready
        port: 8080
      initialDelaySeconds: 5
      periodSeconds: 5
```

### 2. 设置资源限制

```yaml
resources:
  requests:
    memory: "256Mi"
    cpu: "100m"
  limits:
    memory: "512Mi"
    cpu: "500m"
```

### 3. 优雅终止

```yaml
spec:
  terminationGracePeriodSeconds: 60  # 给足时间处理请求
```

### 4. 部署前检查清单

- [ ] 新版本镜像已构建并推送
- [ ] 健康检查接口已实现

- [ ] 回滚方案已验证

## 总结

三种部署策略各有优劣：

- **滚动更新**：资源利用率高，但回滚较慢，适合大多数场景
- **蓝绿部署**：回滚即时，但需要双倍资源，适合高风险变更
- **金丝雀发布**：渐进可控，适合大规模系统的特性验证

<Badge text="最佳实践" type="tip" /> 根据业务需求和技术能力选择合适的策略，并建立完善的监控和回滚机制，确保发布的可靠性。
