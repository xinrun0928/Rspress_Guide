# GitOps 工作流： ArgoCD 集成实践

代码即真理，环境即镜像，部署即推送。

GitOps 是云原生时代的部署理念——用 Git 仓库作为系统状态的单点真相（Single Source of Truth）。当 Git 中的配置变更，自动化工具会自动同步到集群；当集群状态偏离 Git，工具会检测并告警。

## GitOps 核心理念

### 什么是 GitOps？

GitOps 的核心思想可以用一句话概括：**你想要的集群状态，全部写在 Git 仓库里。**

```
传统方式：
开发者 → CI 构建 → 部署工具 (kubectl/helm) → 集群

GitOps 方式：
开发者 → Git Push → ArgoCD 检测变更 → 自动同步 → 集群
                ↑
         Git 仓库是 Single Source of Truth
```

### GitOps 的四大原则

1. **声明式基础设施**：所有基础设施和应用的期望状态都用代码声明
2. **版本化/不可变**：Git 仓库是不可变的版本控制系统
3. **自动拉取**：软件自动从 Git 拉取变更并应用到集群
4. **软件代理**：使用软件代理确保集群状态与 Git 一致

### GitOps vs 传统部署

| 对比维度 | 传统 CI/CD | GitOps |
|---|---|---|
| 状态管理 | 部署命令发出即结束 | Git 是单一真相来源 |
| 审计追溯 | 需要额外的审计日志 | Git commit 即审计记录 |
| 回滚 | 需要手动执行回滚命令 | `git revert` + 自动同步 |
| 权限管理 | CI/CD 平台权限集中 | Git 仓库权限即部署权限 |
| 集群一致性 | 难以保证多集群一致 | Git 推送即全集群同步 |

## ArgoCD 简介

ArgoCD 是 CNCF 的毕业项目，专为 Kubernetes 设计的 GitOps 持续交付工具。

### ArgoCD 特点

| 特性 | 说明 |
|---|---|
| Kubernetes 原生 | 完全基于 K8s CRD 实现 |
|声明式 | 用 Application CRD 定义应用 |
| GitOps | 依赖 Git 作为真相来源 |
| 多集群支持 | 一个 ArgoCD 管理多个集群 |
| 可视化界面 | 提供 Web UI 查看应用状态 |
| 丰富的回滚能力 | 支持回滚到 Git 中的任意版本 |

## ArgoCD 核心概念

### Application

Application 是 ArgoCD 的核心资源，定义了「哪个 Git 仓库的哪个路径」对应「哪个集群的哪个命名空间」。

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: my-app
  namespace: argocd
spec:
  # Git 仓库信息
  source:
    repoURL: https://github.com/myorg/my-app.git
    targetRevision: main
    path: k8s/overlays/production
  # 目标集群信息
  destination:
    server: https://kubernetes.default.svc
    namespace: production
  # 同步策略
  syncPolicy:
    automated:
      prune: true          # 自动删除 Git 中不存在的资源
      selfHeal: true       # 自动修复集群与 Git 的差异
    syncOptions:
    - CreateNamespace=true
```

### Project

Project 用于组织和管理 Application，提供访问控制和资源限制。

```yaml
apiVersion: argoproj.io/v1alpha1
kind: AppProject
metadata:
  name: my-project
  namespace: argocd
spec:
  # 允许访问的仓库
  sourceRepos:
  - https://github.com/myorg/*
  # 允许部署的目标集群和命名空间
  destinations:
  - namespace: production
    server: https://kubernetes.default.svc
  - namespace: staging
    server: https://kubernetes.default.svc
  # 允许创建的 K8s 资源类型
  resourceWhitelist:
  - group: "*"
    kind: "*"
```

### Sync 和 Sync Status

| 状态 | 含义 |
|---|---|
| `Synced` | 集群状态与 Git 一致 |
| `OutOfSync` | 集群状态与 Git 不一致 |
| `Healthy` | 应用运行正常 |
| `Degraded` | 应用不健康 |
| `Missing` | Git 中有但集群中没有 |

## ArgoCD 部署实战

### 方式一：Helm Chart 部署

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: my-app-helm
  namespace: argocd
spec:
  source:
    repoURL: https://charts.bitnami.com/bitnami
    chart: nginx
    targetRevision: 9.0.0
    helm:
      valueFiles:
      - values-prod.yaml
      parameters:
      - name: service.type
        value: LoadBalancer
  destination:
    server: https://kubernetes.default.svc
    namespace: production
  syncPolicy:
    automated: {}
```

### 方式二：Kustomize 多环境

```yaml
# 应用结构
# .
# ├── base/
# │   ├── deployment.yaml
# │   └── service.yaml
# └── overlays/
#     ├── staging/
#     │   ├── kustomization.yaml
#     │   └── replica-count.yaml
#     └── production/
#         ├── kustomization.yaml
#         └── replica-count.yaml
```

```yaml
# overlays/production/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
resources:
- ../../base
patchesStrategicMerge:
- replica-count.yaml
commonLabels:
  environment: production
```

```yaml
# overlays/production/replica-count.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx
spec:
  replicas: 5
```

```yaml
# ArgoCD Application
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: my-app-production
spec:
  source:
    repoURL: https://github.com/myorg/my-app.git
    path: overlays/production
    targetRevision: main
  destination:
    server: https://kubernetes.default.svc
    namespace: production
```

### 方式三：纯 YAML 资源

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: my-app-plain
spec:
  source:
    repoURL: https://github.com/myorg/k8s-resources.git
    targetRevision: HEAD
    directory:
      recurse: true  # 递归扫描目录
  destination:
    server: https://kubernetes.default.svc
    namespace: default
```

## 自动同步策略

### 自动 Sync

```yaml
spec:
  syncPolicy:
    automated:
      prune: true    # 自动删除 Git 中没有的资源
      selfHeal: true # 自动修复与 Git 不一致的变更
      allowEmpty: false  # 是否允许同步后资源为空
```

:::warning 自动化风险
开启 `selfHeal: true` 后，ArgoCD 会自动修复集群中的任何变更，包括手动修改。这可能与某些运维操作冲突。请谨慎使用。
:::

### 同步选项

```yaml
spec:
  syncPolicy:
    syncOptions:
    - CreateNamespace=true      # 自动创建命名空间
    - PruneLast=true             # 最后执行删除操作
    - PrunePropagationPolicy=foreground  # 级联删除策略
    - RespectIgnoreDifferences=true  # 忽略差异
```

### 手动审批同步

```yaml
spec:
  syncPolicy:
    automated: null  # 关闭自动同步
    retry:
      limit: 5
      backoff:
        duration: 5s
        factor: 2
```

## 回滚操作

### Web UI 回滚

1. 登录 ArgoCD Web UI
2. 选择应用
3. 点击「History and Rollback」
4. 选择历史版本
5. 点击「Rollback」

### CLI 回滚

```bash
# 查看应用历史
argocd app history my-app

# 回滚到指定版本
argocd app rollback my-app <revision>

# 示例：回滚到第一个版本
argocd app rollback my-app 1
```

### 强制同步

```bash
# 强制同步，即使检测到差异
argocd app sync my-app --force

# 同步并忽略某些资源
argocd app sync my-app --force \
  --replace \
  --prune \
  --dry-run
```

## ArgoCD + KubeSphere 集成

KubeSphere 内置支持 ArgoCD/Argo Rollouts，可以通过 KubeSphere 的 DevOps 模块直接配置 GitOps 工作流。

### 配置仓库连接

1. 进入 KubeSphere DevOps 模块
2. 选择「代码仓库」
3. 添加 Git 仓库（支持 GitHub/GitLab/Gitea）
4. 配置凭据

### 创建 GitOps 应用

1. 进入「应用管理」→「创建应用」
2. 选择「通过 GitOps 创建」
3. 选择已连接的 Git 仓库
4. 配置同步策略

```yaml

kind: Role

  name: argocd-application-controller

- apiGroups:
  - argoproj.io
```

### 多集群管理

```yaml

kind: Application

  name: my-app-prod

  destination:
    server: https://prod-cluster.k8s.example.com
    namespace: production

kind: Application
metadata:
  name: my-app-staging
spec:
  destination:
    server: https://staging-cluster.k8s.example.com
    namespace: staging
```

## 总结

GitOps + ArgoCD 是云原生持续交付的黄金组合：

- **Git 是单一真相来源**：所有变更通过 Git 管理，可追溯、可回滚
- **声明式部署**：声明即期望，ArgoCD 自动达成期望状态
- **多集群一致性**：一个 Git 仓库，多个集群自动同步
- **可视化运维**：Web UI 让部署状态一目了然

**核心要点：**

- 选择合适的仓库结构（Helm/Kustomize/纯 YAML）
- 谨慎开启 `selfHeal`，避免意外覆盖手动变更
- 建立 Git 分支策略（建议 main/staging/prod）
- 善用回滚功能，快速响应故障
