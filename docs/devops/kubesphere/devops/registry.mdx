# KubeSphere 镜像仓库

"我本地构建的镜像怎么推不上去？认证失败？"

这是很多团队在配置镜像仓库时遇到的第一个问题。KubeSphere 集成了 Harbor 作为默认的镜像仓库，但配置不当就会导致各种推送失败的问题。

这一节，我们详细讲解镜像仓库的配置和使用。

## 镜像仓库概述

### KubeSphere 内置组件

KubeSphere DevOps 内置了以下仓库组件：

| 组件 | 用途 |
|---|---|
| Harbor | Docker 镜像仓库 |
| MinIO/S3 | 二进制制品仓库 |
| SonarQube | 代码质量检查 |

## Harbor 配置

### 访问 Harbor 控制台

```
默认地址: http://{$NODE_IP}:30002
首次登录: admin / Harbor12345
```

### Harbor 项目

Harbor 使用"项目"来组织镜像：

| 项目类型 | 说明 |
|---|---|
| 项目 | 存放镜像，默认私有 |
| 公开项目 | 镜像公开访问，无需登录 |

### 创建项目

```bash
# 通过 Harbor API 创建项目
curl -X POST "http://harbor.example.com/api/v2.0/projects" \
  -H "Content-Type: application/json" \
  -u "admin:Harbor12345" \
  -d '{
    "project_name": "my-app",
    "public": false,
    "storage_limit": -1
  }'
```

## 凭证管理

### 在 KubeSphere 中创建凭证

```
DevOps 项目 → 凭证管理 → 创建凭证
  ├── 类型: 用户名/密码
  ├── 名称: harbor-credential
  ├── 用户名: admin
  └── 密码: Harbor12345
```

### 凭证类型说明

| 类型 | 适用场景 |
|---|---|
| 用户名/密码 | Harbor 登录、Git 仓库登录 |
| SSH 密钥 | Git SSH 拉取 |
| 访问令牌 | GitHub/GitLab Personal Access Token |
| 秘密文本 | API Token、敏感字符串 |

### 凭证命名规范

<Badge text="最佳实践" type="tip" /> 建议使用有意义的凭证名称，如 `harbor-${env}-credential`，便于流水线中识别。

## 镜像拉取

### 在 Pod 中使用 Image Pull Secret

```yaml
# 创建 Secret
kubectl create secret docker-registry harbor-secret \
  --docker-server=harbor.example.com \
  --docker-username=admin \
  --docker-password=Harbor12345 \
  --namespace=demo

# 在 Pod 中引用
apiVersion: v1
kind: Pod
metadata:
  name: my-app
  namespace: demo
spec:
  containers:
  - name: my-app
    image: harbor.example.com/my-app:v1.0
  imagePullSecrets:
  - name: harbor-secret
```

### 在 Deployment 中使用

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
  namespace: demo
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-app
  template:
    spec:
      containers:
      - name: my-app
        image: harbor.example.com/my-app:v1.0
      imagePullSecrets:
      - name: harbor-secret
```

## 镜像推送

### Docker 登录

```bash
# 登录 Harbor
docker login harbor.example.com -u admin -p Harbor12345

# 推送镜像
docker tag my-app:v1.0 harbor.example.com/my-project/my-app:v1.0
docker push harbor.example.com/my-project/my-app:v1.0
```

### 在流水线中推送

```groovy
pipeline {
    agent {
        kubernetes {
            defaultContainer 'docker'
            yaml '''
spec:
  containers:
  - name: docker
    image: docker:20.10
    command: [cat]
    tty: true
    securityContext:
      privileged: true
'''
        }
    }
    
    environment {
        DOCKER_REGISTRY = 'harbor.example.com'
        PROJECT_NAME = 'my-project'
        APP_NAME = 'my-app'
        IMAGE_TAG = "${env.DOCKER_REGISTRY}/${env.PROJECT_NAME}/${env.APP_NAME}:${env.BUILD_NUMBER}"
    }
    
    stages {
        stage('构建镜像') {
            steps {
                container('docker') {
                    script {
                        sh """
                            # 构建镜像
                            docker build -t ${IMAGE_TAG} .
                            
                            # 登录并推送
                            echo ${HARBOR_PASSWORD} | docker login ${DOCKER_REGISTRY} \
                                -u ${HARBOR_USERNAME} --password-stdin
                            docker push ${IMAGE_TAG}
                        """
                    }
                }
            }
        }
    }
}
```

## 二进制制品仓库

### Sonatype Nexus

KubeSphere DevOps 内置了 Nexus，支持 Maven、npm 等制品仓库。

### Nexus 访问

```
默认地址: http://{$NODE_IP}:30001
默认账户: admin / admin123
```

### 配置 Maven 仓库

```xml
<!-- pom.xml -->
<distributionManagement>
    <repository>
        <id>nexus-releases</id>
        <url>http://nexus:8081/repository/maven-releases/</url>
    </repository>
    <snapshotRepository>
        <id>nexus-snapshots</id>
        <url>http://nexus:8081/repository/maven-snapshots/</url>
    </snapshotRepository>
</distributionManagement>
```

```xml
<!-- settings.xml -->
<server>
    <id>nexus-releases</id>
    <username>admin</username>
    <password>admin123</password>
</server>
<server>
    <id>nexus-snapshots</id>
    <username>admin</username>
    <password>admin123</password>
</server>
```

## 镜像管理最佳实践

### 镜像命名规范

<Badge text="命名约定" type="tip" /> 建议使用 `<registry>/<project>/<app-name>:<version>` 的命名格式。

| 格式 | 示例 |
|---|---|
| 正确 | `harbor.example.com/dev/my-app:v1.0` |
| 不推荐 | `harbor.example.com/my-app:latest` |
| 错误 | `harbor.example.com/library/my-app` |

### 版本管理策略

| 类型 | 说明 | 示例 |
|---|---|---|
| Git Commit Hash | 精确追踪 | `v1.0-abc1234` |
| Semver | 语义化版本 | `v1.0.0` |
| Date | 构建日期 | `v20231201` |
| Build Number | CI 构建号 | `v1.0-256` |

### 镜像清理策略

```bash
# 在 Harbor 中配置自动清理
# 项目 → 配置 → 清理策略
- 保留最近 N 个版本
- 删除超过 N 天的镜像
- 按标签规则匹配
```

### 安全扫描

Harbor 内置 Trivy 漏洞扫描：

```
项目 → 镜像 → 选择镜像 → 扫描
```

扫描结果会显示：

| 严重级别 | 颜色 | 说明 |
|---|---|---|
| Critical | 红色 | 必须修复 |
| High | 橙色 | 建议尽快修复 |
| Medium | 黄色 | 适当时候修复 |
| Low | 绿色 | 可以接受 |
