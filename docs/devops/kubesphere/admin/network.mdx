# 网络配置：集群的「血管」

Kubernetes 网络是出了名的复杂——Pod 之间怎么通信？Service 如何被发现？外部流量怎么进入集群？在 KubeSphere 环境中，这些问题变得更加具体。

本章节聚焦 KubeSphere 的网络配置，包括负载均衡器 PorterLB、NetworkPolicy 隔离、Ingress 控制器管理。

## PorterLB：裸金属负载均衡器

在云厂商环境，Kubernetes 可以直接使用云负载均衡器（CLB/ALB）。但在私有化部署的裸金属环境，没有云 LB 可用——这就是 PorterLB 的用武之地。

### PorterLB 工作模式

| 模式 | 说明 | 适用场景 |
|---|---|---|
| Layer 2 模式 | ARP/NDP 广播方式 | 小规模集群，简单易用 |
| BGP 模式 | BGP 路由协议 | 大规模集群，需要交换机配合 |

### 部署 PorterLB

```yaml
apiVersion: v1
kind: Namespace
metadata:
  name: porterlb-system
---
apiVersion: v1
kind: ServiceAccount
metadata:
  name: porterlb-controller
  namespace: porterlb-system
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: porterlb-controller
  namespace: porterlb-system
  labels:
    app: porterlb
    component: controller
spec:
  replicas: 1
  selector:
    matchLabels:
      app: porterlb
      component: controller
  template:
    metadata:
      labels:
        app: porterlb
        component: controller
    spec:
      serviceAccountName: porterlb-controller
      containers:
      - name: porterlb
        image: kubespheredocker/porterlb:v0.5.7
        imagePullPolicy: IfNotPresent
        args:
        - --leader-elect=true
        - --lb-mode=layer2
        env:
        - name: PORTERLB_NAMESPACE
          valueFrom:
            fieldRef:
              fieldPath: metadata.namespace
```

### 使用 PorterLB 暴露服务

```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-app-lb
  annotations:
    layerfit.ksyun.io/service-type: " porterlb"  # 指定使用 PorterLB
spec:
  type: LoadBalancer
  selector:
    app: my-app
  ports:
  - name: http
    protocol: TCP
    port: 80
    targetPort: 8080
```

```bash
# 查看 LoadBalancer 服务
kubectl get svc

# 输出示例
NAME         TYPE           CLUSTER-IP     EXTERNAL-IP     PORT(S)        AGE
my-app-lb    LoadBalancer   10.96.89.123    192.168.1.200   80:31234/TCP   5m
```

:::tip PorterLB 注意事项
- Layer 2 模式下，EXTERNAL-IP 是 Node 的 IP，同一时刻只有一个 Node 响应 ARP
- 如果集群规模大，建议使用 BGP 模式
- 确保物理网络允许 ARP/NDP 广播
:::

## NetworkPolicy：微服务隔离

NetworkPolicy 是 Kubernetes 的网络隔离机制，类似于云安全组。它定义了 Pod 之间的访问规则。

### 基本 NetworkPolicy

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: api-allow-frontend
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: api
  policyTypes:
  - Ingress
  ingress:
  # 只允许带 frontend 标签的 Pod 访问
  - from:
    - podSelector:
        matchLabels:
          app: frontend
    ports:
    - protocol: TCP
      port: 8080
```

### 常见场景

**场景 1：禁止访问数据库的 Pod 访问外网**

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: db-no-egress
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: database
  policyTypes:
  - Egress
  egress: []  # 禁止所有出站流量
```

**场景 2：只允许命名空间内访问**

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: namespace-internal-only
spec:
  podSelector: {}
  policyTypes:
  - Ingress
  ingress:
  - from:
    - namespaceSelector:
        matchLabels:
          name: production
```

:::warning NetworkPolicy 生效条件
NetworkPolicy 需要 CNI 插件支持。Calico 原生支持 NetworkPolicy，Flannel 不支持。在选择 CNI 时要考虑网络隔离需求。
:::

### KubeSphere 可视化配置

KubeSphere 提供了图形化的 NetworkPolicy 配置界面：

1. 进入项目 → 网络隔离
2. 点击「启用网络隔离」
3. 配置入站/出站规则

## Ingress 管理

Ingress 是 HTTP/HTTPS 入口，定义七层路由规则。KubeSphere 默认集成 Traefik 作为 Ingress 控制器。

### 基本 Ingress 配置

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-app-ingress
  annotations:
    kubernetes.io/ingress.class: traefik
spec:
  rules:
  - host: myapp.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: my-app-service
            port:
              number: 80
  tls:
  - hosts:
    - myapp.example.com
    secretName: my-tls-secret
```

### Path 类型

| 类型 | 说明 | 示例 |
|---|---|---|
| `Prefix` | 路径前缀匹配 | `/api` 匹配 `/api`, `/api/users` |
| `Exact` | 精确匹配 | `/api` 只匹配 `/api` |
| `ImplementationSpecific` | 由 IngressClass 决定 | 默认 |

### 常用注解

```yaml
metadata:
  annotations:
    # 重写规则
    traefik.ingress.kubernetes.io/rule-type: "PathPrefixStrip"
    traefik.ingress.kubernetes.io/rewrite-target: "/$1"
    
    # 限流
    traefik.ingress.kubernetes.io/rate-limit: "100"
    
    # 跨域
    nginx.ingress.kubernetes.io/enable-cors: "true"
```

### 配置 HTTPS

```yaml
# 创建 TLS Secret
kubectl create secret tls my-tls-secret \
  --cert=fullchain.pem \
  --key=privkey.pem

# 或使用 cert-manager 自动管理证书
```

## 服务发现

### CoreDNS 原理

Kubernetes 使用 CoreDNS 作为服务发现的 DNS 服务器。

```
Pod → kube-dns → Service DNS → Pod IP
```

- Service DNS：`<service-name>.<namespace>.svc.cluster.local`
- Headless Service：返回所有 Pod IP

```bash
# 在 Pod 内测试 DNS
kubectl exec -it test-pod -- nslookup kubernetes.default
kubectl exec -it test-pod -- nslookup my-service.prod.svc.cluster.local
```

### 外部服务访问

**方式 1：ExternalName Service**

```yaml
apiVersion: v1
kind: Service
metadata:
  name: external-api
  namespace: production
spec:
  type: ExternalName
  externalName: api.external.com
```

**方式 2：Endpoints 手动定义**

```yaml
apiVersion: v1
kind: Service
metadata:
  name: external-db
spec:
  ports:
  - port: 3306
---
apiVersion: v1
kind: Endpoints
metadata:
  name: external-db
subsets:
- addresses:
  - ip: 192.168.1.100
  ports:
  - port: 3306
```

## 网络排错

### 常见问题

| 问题 | 排查命令 |
|---|---|
| Pod 之间不通 | `kubectl exec` 进入 Pod 测试 |
| Service 无法访问 | 检查 Endpoints 是否存在 |
| 外部无法访问 | 检查 Ingress/SLB 配置 |
| DNS 解析失败 | 检查 CoreDNS 是否运行 |

### 诊断命令

```bash
# 查看 CNI 插件状态
kubectl get pods -n kube-system -o wide | grep cni

# 检查 iptables 规则
kubectl exec -it <pod> -- iptables -L -n

# 查看 DNS 解析
kubectl exec -it <pod> -- cat /etc/resolv.conf

# 测试网络连通性
kubectl exec -it <pod> -- ping <target-ip>
kubectl exec -it <pod> -- wget -O- http://<service-name>
```

## 总结

KubeSphere 网络配置的核心要点：

- **负载均衡**：裸金属环境使用 PorterLB，云环境使用云 LB
- **网络隔离**：使用 NetworkPolicy 限制微服务间访问
- **入口管理**：通过 Ingress 统一管理 HTTP/HTTPS 路由
- **服务发现**：CoreDNS 提供集群内 DNS 服务

正确的网络配置是应用可用性的基础，建议在部署应用前规划好网络策略。
