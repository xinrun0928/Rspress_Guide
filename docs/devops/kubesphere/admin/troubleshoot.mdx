# 故障排查：系统化定位问题

凌晨 2 点，你收到告警：「KubeSphere 控制台无法访问」。你的第一反应是什么？打开日志？重启组件？还是慌了？

**故障排查的核心是「系统化」——不是盲目尝试，而是按逻辑逐层排除。**

本章节提供 KubeSphere 故障排查的方法论和实战技巧，帮助你在最短时间内定位问题。

## 排查方法论

### 层级排查法

```
用户层
  ↓ 是不是 UI 问题？
前端层
  ↓ 是不是浏览器/前端问题？
网关层
  ↓ 请求能否到达后端？
API 层
  ↓ 后端服务是否正常？
基础设施层
  ↓ K8s 集群是否正常？
```

### 排查步骤

1. **确认问题范围**：单个用户还是所有用户？单个功能还是全部功能？
2. **检查错误现象**：错误码、错误信息、请求耗时
3. **定位故障层级**：用户端、网络、网关、应用、集群
4. **缩小范围**：排除法，找出最可能的原因
5. **验证假设**：根据假设采取行动，验证是否解决

## 常见安装问题

### 问题 1：ks-installer 启动失败

```bash
# 查看 installer 状态
kubectl get pods -n kubesphere-system
kubectl describe pod -n kubesphere-system ks-installer-xxxxx
kubectl logs -n kubesphere-system ks-installer-xxxxx
```

**可能原因：**

- 镜像拉取失败
- 配置错误
- 资源不足

**解决方案：**

```bash
# 检查镜像是否可拉取
docker pull kubesphere/ks-installer:latest

# 查看详细错误
kubectl logs -n kubesphere-system ks-installer-xxxxx --tail=200

# 如果配置错误，修改 cluster-configuration 后重新 apply
kubectl apply -f cluster-configuration.yaml
```

### 问题 2：组件一直 Pending

```bash
# 查看组件 Pod
kubectl get pods -n kubesphere-system -o wide

# 查看详细状态
kubectl describe pod -n kubesphere-system <pod-name>
```

**常见原因：**

- 资源不足（CPU/内存）
- 调度失败（节点选择器/亲和性）
- 存储问题（PVC 未绑定）

**解决方案：**

```bash
# 检查节点资源
kubectl describe nodes | grep -A5 "Allocated resources"

# 检查 PVC 状态
kubectl get pvc -n kubesphere-system
kubectl describe pvc -n kubesphere-system <pvc-name>
```

### 问题 3：外网无法访问控制台

```bash
# 检查 NodePort
kubectl get svc -n kubesphere-system | grep -E "ks-|console"

# 检查 Ingress（如使用）
kubectl get ingress -n kubesphere-system
kubectl describe ingress -n kubesphere-system ks-console

# 检查网络策略
kubectl get networkpolicies -n kubesphere-system
```

**解决方案：**

```bash
# 使用 NodePort 临时访问
# 编辑 svc，将 NodePort 改为固定值
kubectl edit svc -n kubesphere-system ks-console

# 或使用端口转发测试
kubectl port-forward -n kubesphere-system svc/ks-console 8080:80
```

## 组件健康检查

### 检查所有系统组件

```bash
#!/bin/bash
# health-check.sh

echo "=== KubeSphere 组件状态 ==="
kubectl get pods -n kubesphere-system

echo ""
echo "=== 核心 API 可用性 ==="
kubectl cluster-info

echo ""
echo "=== etcd 健康状态 ==="
kubectl get cs

echo ""
echo "=== 节点状态 ==="
kubectl get nodes

echo ""
echo "=== 控制平面组件 ==="
kubectl get componentstatuses
```

### 逐组件检查

| 组件 | 命名空间 | 检查命令 |
|---|---|---|
| ks-apiserver | kubesphere-system | `kubectl logs -n kubesphere-system ks-apiserver-xxx` |
| ks-controller | kubesphere-system | `kubectl logs -n kubesphere-system ks-controller-xxx` |
| ks-console | kubesphere-system | `kubectl logs -n kubesphere-system ks-console-xxx` |

| Redis | kubesphere-system | `kubectl logs -n kubesphere-system redis-xxx` |
| MinIO | kubesphere-system | `kubectl logs -n kubesphere-system minio-xxx` |

### 组件重启

```bash
# 重启单个组件（谨慎使用）
kubectl rollout restart deployment -n kubesphere-system <component-name>

# 查看重启历史
kubectl rollout history deployment -n kubesphere-system <component-name>

# 回滚到上一版本
kubectl rollout undo deployment -n kubesphere-system <component-name>
```

## 日志收集

### 收集诊断信息

```bash
#!/bin/bash
# collect-diagnostics.sh

DIAG_DIR="kubesphere-diagnostics-$(date +%Y%m%d-%H%M%S)"
mkdir -p $DIAG_DIR

# 收集集群信息
kubectl cluster-info dump > $DIAG_DIR/cluster-info.txt

# 收集 KubeSphere 组件日志
kubectl logs -n kubesphere-system -l app=ks-install --tail=1000 > $DIAG_DIR/ks-installer.log
kubectl logs -n kubesphere-system --all-containers --tail=500 > $DIAG_DIR/all-pods.log

# 收集资源状态
kubectl get all -n kubesphere-system -o yaml > $DIAG_DIR/kubesphere-resources.yaml
kubectl get events -n kubesphere-system > $DIAG_DIR/events.log

# 收集各组件详细日志
for pod in $(kubectl get pods -n kubesphere-system -o name); do
    pod_name=$(echo $pod | cut -d'/' -f2)
    kubectl logs -n kubesphere-system $pod --tail=500 > $DIAG_DIR/$pod_name.log
done

# 打包
tar -czf $DIAG_DIR.tar.gz $DIAG_DIR
echo "诊断信息已保存: $DIAG_DIR.tar.gz"
```

### 关键日志位置

| 组件 | 日志来源 | 日志级别 |
|---|---|---|
| ks-apiserver | Pod logs | INFO |
| ks-controller | Pod logs | DEBUG |
| ks-console | Pod logs | INFO |
| OpenLDAP | Pod logs | statslog |
| Redis | Pod logs | notice |

### 日志分析技巧

```bash
# 搜索错误日志
kubectl logs -n kubesphere-system ks-apiserver-xxx | grep -i error

# 搜索特定时间范围的日志
kubectl logs -n kubesphere-system ks-apiserver-xxx \
  --since=2024-01-15T10:00:00Z \
  --until=2024-01-15T11:00:00Z

# 实时跟踪日志
kubectl logs -n kubesphere-system -f ks-apiserver-xxx

# 统计错误类型
kubectl logs -n kubesphere-system ks-apiserver-xxx | \
  grep -oP 'error=\K\w+' | sort | uniq -c | sort -rn
```

## 诊断工具

### kubectl-debug

```bash
# 安装 kubectl-debug
kubectl plugin install github.com/aylei/kubectl-debug

# 在问题 Pod 中启动诊断容器
kubectl debug -n production myapp-pod --share-processes --copy-to=myapp-debug

# 进入诊断容器
kubectl exec -it -n production myapp-debug -- bash

# 排查完毕后清理
kubectl delete debug myapp-debug -n production
```

### View-cert

```bash
# 检查证书过期时间
kubectl get secret -n kubesphere-system -o jsonpath='{range .items[*]}{.metadata.name}{"\t"}{.metadata.annotations.kubernetes\.io/#{.type}}{"\n"}{end}' | grep -E "tls|certificate"

# 查看特定 Secret 的证书信息
kubectl get secret -n kubesphere-system ks-apiserver-cert -o jsonpath='{.data.tls\.crt}' | base64 -d | openssl x509 -text -noout | grep -E "Subject:|Not After:"
```

### 网络诊断

```bash
# 检查 CoreDNS
kubectl get pods -n kube-system -l k8s-app=kube-dns

# 测试 DNS 解析
kubectl run -it --rm dnsutils --image=tutum/dnsutils --restart=Never -- nslookup kubernetes.default

# 检查网络策略
kubectl get networkpolicies --all-namespaces

# 查看 CNI 状态
kubectl get pods -n kube-system -o wide | grep -E "calico|flannel|cilium"
```

## 常见问题速查

| 问题 | 快速检查 | 常见原因 |
|---|---|---|
| 控制台打不开 | 检查 ks-console Pod | ks-console 崩溃/网络问题 |

| 应用部署失败 | 检查 ks-apiserver | API 服务异常 |

| 监控无数据 | 检查 Prometheus | Prometheus 异常 |
| 日志无数据 | 检查 Fluent Bit | 日志收集器问题 |

## 总结

故障排查是运维的「硬功夫」。系统化的排查方法比经验更重要。

**核心要点：**

- 遵循「用户层 → 基础设施层」的排查顺序
- 善用 `kubectl describe` 和 `kubectl logs`
- 学会收集诊断信息包
- 保持冷静，按逻辑排除

**遇到问题时：先问自己三个问题：**
1. 什么时间点开始出问题？
2. 这个问题影响范围有多大？
3. 最近做了什么变更？
