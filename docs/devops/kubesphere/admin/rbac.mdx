# RBAC 权限控制：多租户的「门禁」

「谁可以查看生产环境的配置？谁可以删除 Pod？谁可以管理其他用户？」

在多租户 Kubernetes 环境中，权限控制是安全的基石。KubeSphere 构建在 Kubernetes RBAC 之上，提供了企业级的多租户权限管理体系。

**RBAC（Role-Based Access Control）不是「加个角色」那么简单——它是一套精确的权限语言，决定了谁能做什么。**

## Kubernetes RBAC 基础

### 核心对象

| 对象 | 作用范围 | 说明 |
|---|---|---|
| Role/ClusterRole | 定义权限 | 「能做什么」 |
| RoleBinding/ClusterRoleBinding | 绑定到主体 | 「谁有这些权限」 |

```
┌─────────────────────────────────────────────────────────────┐
│                        ClusterRoleBinding                    │
│  subjects: [User: admin, Group: dev-team]                   │
│  roleRef: ClusterRole: cluster-admin                        │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                       ClusterRole                            │
│  rules:                                                     │
│    - apiGroups: [*]                                         │
│      resources: [*]                                         │
│      verbs: [*]                                             │
└─────────────────────────────────────────────────────────────┘
```

### 权限动词

| 动词 | 对应操作 |
|---|---|
| `get` | 查看单个资源 |
| `list` | 列出资源列表 |
| `watch` | 监听资源变化 |
| `create` | 创建资源 |
| `update` | 更新资源 |
| `patch` | 部分更新资源 |
| `delete` | 删除资源 |
| `deletecollection` | 删除一组资源 |

### 资源与 API 组

```yaml
rules:
# 非资源型 URL
- nonResourceURLs: ["/healthz", "/version"]
  verbs: ["get"]

# 核心 API 组
- apiGroups: [""]
  resources: ["pods", "services", "configmaps"]
  verbs: ["get", "list"]

# 特定 API 组
- apiGroups: ["apps"]
  resources: ["deployments", "statefulsets"]
  verbs: ["*"]

# 所有 API 组
- apiGroups: ["*"]
  resources: ["*"]
  verbs: ["*"]
```

## KubeSphere 内置角色

KubeSphere 预定义了一套角色，涵盖集群、项目、工作空间三个层级。

### 集群级别角色

| 角色 | 权限范围 | 适用人员 |
|---|---|---|
| `cluster-admin` | 全部集群资源 | 运维管理员 |
| `clusters-admin` | 集群管理 | 集群所有者 |
| `workspace-manager` | 管理所有工作空间 | 平台管理员 |
| `regular-user` | 被邀请加入工作空间 | 普通用户 |
| `anonymous` | 未登录用户 | 访客 |

### 工作空间级别角色

| 角色 | 权限范围 | 说明 |
|---|---|---|
| `workspace-admin` | 整个工作空间 | 工作空间所有者 |
| `workspace-regular` | 工作空间内资源 | 普通成员 |
| `workspace-viewer` | 只读权限 | 观察者 |

### 项目级别角色

| 角色 | 权限范围 | 说明 |
|---|---|---|
| `admin` | 项目全部管理权限 | 项目管理员 |
| `operator` | 创建/管理应用 | 开发者 |
| `viewer` | 只读 | 访客 |
| `self-provisioner` | 可在项目内创建工作负载 | 开发者 |

## 自定义角色

当内置角色无法满足需求时，可以创建自定义角色。

### 创建 ClusterRole

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: monitoring-reader
rules:
# 读取监控相关资源
- apiGroups: ["monitoring.coreos.com"]
  resources: ["prometheuses", "prometheusrules", "servicemonitors"]
  verbs: ["get", "list", "watch"]
# 读取 Pod 日志
- apiGroups: [""]
  resources: ["pods/log"]
  verbs: ["get", "list"]
```

### 创建 Role

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: configmap-editor
  namespace: production
rules:
- apiGroups: [""]
  resources: ["configmaps"]
  verbs: ["get", "list", "create", "update", "patch"]
  resourceNames: ["app-config", "feature-flags"]  # 仅限特定资源
```

:::warning 最小权限原则
**永远使用最小权限集。** 不要给 `*` 权限，不要给 `apiGroups: ["*"]` 除非必要。权限过大是安全漏洞的根源。
:::

## 权限绑定

### RoleBinding：绑定到用户

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: developer-rolebinding
  namespace: production
subjects:
# 用户
- kind: User
  name: alice@example.com
  apiGroup: rbac.authorization.k8s.io
# 组
- kind: Group
  name: developer-team
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: developer
  apiGroup: rbac.authorization.k8s.io
```

### ClusterRoleBinding：集群级别绑定

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: monitoring-admin-binding
subjects:
- kind: ServiceAccount
  name: prometheus
  namespace: monitoring
roleRef:
  kind: ClusterRole
  name: monitoring-admin
  apiGroup: rbac.authorization.k8s.io
```

### KubeSphere Web 控制台绑定

在 KubeSphere 中，权限绑定更加直观：

1. 进入「访问控制」
2. 选择用户/工作空间/项目
3. 点击「编辑权限」
4. 选择角色或自定义权限集

## KubeSphere 多租户权限体系

KubeSphere 的多租户体系由三个层级构成：

```
┌─────────────────────────────────────────────────────────────┐
│                     集群 (Cluster)                            │
│  ├── 层级：集群管理员                                          │
│  └── 权限：管理所有资源                                        │
│         │                                                    │
│         ▼                                                    │
│  ┌─────────────────────────────────────────────────────┐     │
│  │                   工作空间 (Workspace)                │     │
│  │  ├── 层级：工作空间管理员                             │     │
│  │  └── 权限：管理工作空间内的项目和成员                   │     │
│  │         │                                            │     │
│  │         ▼                                            │     │
│  │  ┌─────────────────────────────────────────────┐     │     │
│  │  │                 项目 (Project)                │     │     │
│  │  │  ├── 层级：项目管理员                         │     │     │
│  │  │  └── 权限：管理项目内的资源                     │     │     │
│  │  └─────────────────────────────────────────────┘     │     │
│  └─────────────────────────────────────────────────────┘     │
└─────────────────────────────────────────────────────────────┘
```

### 层级与权限继承

| 层级 | 角色 | 下级权限继承 |
|---|---|---|
| 集群 | cluster-admin | 可管理所有工作空间和项目 |
| 工作空间 | workspace-admin | 可管理工作空间内的所有项目 |
| 项目 | admin | 仅管理当前项目 |

### 跨层级权限

有时需要授予跨层级的权限。例如，工作空间管理员需要查看所有项目的日志：

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: workspace-log-reader
aggregationRule:
  clusterRoleSelectors:
  - matchLabels:
      rbac.authorization.k8s.io/aggregate-to-workspace-viewer: "true"
rules: []  # 由 selector 自动填充
---
# 在每个项目级别授予日志读取权限
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: project-log-reader
  namespace: project-a
  labels:
    rbac.authorization.k8s.io/aggregate-to-workspace-viewer: "true"
rules:
- apiGroups: [""]
  resources: ["pods/log"]
  verbs: ["get", "list"]
```

## 实战场景

### 场景 1：授予开发者「只读项目」权限

```yaml
# 只读角色（基于内置 viewer）
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: read-only
  namespace: production
rules:
- apiGroups: ["*"]
  resources: ["*"]
  verbs: ["get", "list", "watch"]
- apiGroups: [""]
  resources: ["pods/exec", "pods/portforward"]
  verbs: []  # 明令禁止
```

### 场景 2：授予运维「部署管理」权限

```yaml
# 运维角色
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: devops-deployer
rules:
# 管理 Deployments、StatefulSets
- apiGroups: ["apps"]
  resources: ["deployments", "statefulsets", "daemonsets"]
  verbs: ["*"]
# 管理 Services、Ingress
- apiGroups: [""]
  resources: ["services", "ingresses", "configmaps", "secrets"]
  verbs: ["*"]
# 管理 HPA
- apiGroups: ["autoscaling"]
  resources: ["horizontalpodautoscalers"]
  verbs: ["*"]
# 不能删除
- apiGroups: ["*"]
  resources: ["*/finalizers"]
  verbs: ["update"]
```

### 场景 3：ServiceAccount 的权限

```yaml
# 给 CI/CD Pipeline 使用的 ServiceAccount
apiVersion: v1
kind: ServiceAccount
metadata:
  name: ci-deployer
  namespace: production
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: ci-deployer-binding
  namespace: production
subjects:
- kind: ServiceAccount
  name: ci-deployer
  namespace: production
roleRef:
  kind: ClusterRole
  name: edit  # 使用内置的 edit 角色（可管理工作负载）
  apiGroup: rbac.authorization.k8s.io
```

## 权限审计

### 查看用户的权限

```bash
# 查看用户绑定的角色
kubectl get rolebindings,clusterrolebindings --all-namespaces \
  -o custom-columns='NS:metadata.namespace,NAME:metadata.name,SUBJECTS:subjects[*].name'

# 使用 kubesphere CLI（可选）
kubectl ks user info admin
```

### 检查 ClusterRole 聚合

```bash
# 查看 Role 是否有聚合标签
kubectl get role -n production -o yaml | grep -A5 labels

# 查看聚合后的 ClusterRole 规则
kubectl auth can-i --list \
  --as=alice@example.com \
  -n production
```

### 常用审计命令

```bash
# 查看谁有 cluster-admin 权限
kubectl get clusterrolebinding -o json | jq \
  '.items[] | select(.roleRef.name=="cluster-admin") | .subjects'

# 查看特定 API 的访问者
kubectl get clusterrolebinding -o json | jq \
  '.items[] | select(.roleRef.name | contains("admin")) | .subjects[].name'

# 检查 ServiceAccount 权限
kubectl auth can-i create pods \
  --as=system:serviceaccount:production:ci-deployer
```

## 最佳实践

### 1. 避免使用 cluster-admin

```yaml
# 错误：给普通用户 cluster-admin
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
subjects:
- kind: User
  name: alice@example.com
roleRef:
  kind: ClusterRole
  name: cluster-admin  # 太危险！
```

### 2. 使用 Role 而非 ClusterRole

除非必要，不要给集群级别的权限。优先使用 Role 限制在特定命名空间。

### 3. 定期审计权限

- 季度审查用户权限
- 移除离职/转岗人员的权限
- 清理无用的 RoleBinding

### 4. ServiceAccount 权限最小化

Pipeline、CI/CD 组件不应使用 cluster-admin。创建专用的 ServiceAccount 并绑定最小权限。

## 总结

RBAC 是 Kubernetes 安全的核心。KubeSphere 在原生 RBAC 基础上构建了企业级多租户体系，但原理相通。

**核心要点：**

- Role 定义「能做什么」，Binding 定义「谁能做」
- 遵循最小权限原则，永远不用 `*`
- 层级清晰：集群 → 工作空间 → 项目
- 定期审计，清理过期权限
- ServiceAccount 是程序访问 K8s 的最佳方式
