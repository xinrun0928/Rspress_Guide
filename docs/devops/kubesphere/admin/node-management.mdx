# 节点管理：集群的「骨骼」

「kubectl get nodes」看到 Ready，节点就真的健康吗？不一定。

节点的健康管理涉及多个层面：kubelet 是否正常汇报、容器运行时是否可用、网络是否通畅、磁盘是否即将耗尽。本质上，Kubernetes 的「节点」不只是一台虚拟机，它是一个由多个组件构成的复杂系统。

## 添加 Worker 节点

### 准备工作

在添加新节点之前，确保：

- Master 节点可访问
- 新节点已安装 Docker/containerd
- 新节点已下载 kubeadm、kubelet、kubectl
- 新节点与 Master 网络互通

```bash
# 检查节点是否已安装必要组件
which kubeadm
which kubelet
which kubectl

# 检查容器运行时
systemctl status containerd
```

### 获取加入命令

KubeSphere 提供图形化的节点添加界面。登录控制台，进入集群管理页面：

1. 点击「节点管理」→「添加节点」
2. 选择节点角色（Worker）
3. 复制生成的 kubeadm join 命令

```bash
# 示例 join 命令（实际命令以控制台生成的为准）
kubeadm join 192.168.1.100:6443 \
  --token xxxxxxxxxxxxxxxx \
  --discovery-token-ca-cert-hash sha256:xxxxxxxxxx
```

:::warning Token 过期
 kubeadm join 命令中的 token 默认有效期为 24 小时。如果过期，需要重新生成：

```bash
kubeadm token create --print-join-command
```
:::

### 节点角色配置

KubeSphere 支持灵活的节点角色配置：

| 角色 | 说明 | 适用场景 |
|---|---|---|
| `all-in-one` | Master + Worker 合一 | 开发测试环境 |
| `master` | Master 控制平面 | 生产环境独立控制平面 |
| `worker` | 仅运行工作负载 | 扩容 Worker 节点 |
| `etcd` | 独立 etcd 节点 | 大规模集群高可用 etcd |

```bash
# 给节点打标签，指定角色
kubectl label node worker-1 node-role.kubernetes.io/worker=worker
kubectl label node worker-1 node.kubernetes.io/workload=compute-intensive

# 查看节点标签
kubectl get nodes --show-labels
```

## Taint 管理

Taint（污点）和 Toleration（容忍）是 Kubernetes 调度的重要机制。Master 节点默认带有 Taint，防止工作负载调度到控制平面节点。

### 常用 Taint 操作

```bash
# 给节点打 Taint
kubectl taint nodes node1 dedicated=production:NoSchedule

# 解释 Taint 参数
# key=value:effect
#   key   - 污点键名
#   value - 污点值（可省略）
#   effect - 调度策略

# NoSchedule    - 不调度，除非 Pod 有匹配的 Toleration
# PreferNoSchedule - 尽量不调度，但不强制
# NoExecute     - 不调度，且驱逐已有 Pod
```

### 常见场景

**场景 1：专用节点（GPU 节点）**

```yaml
# 给 GPU 节点打污点
kubectl taint nodes gpu-1 nvidia.com/gpu=present:NoSchedule

# Pod 需要声明容忍
apiVersion: v1
kind: Pod
metadata:
  name: ml-training
spec:
  tolerations:
  - key: "nvidia.com/gpu"
    operator: "Exists"
    effect: "NoSchedule"
  containers:
  - name: train
    image: pytorch:latest
    resources:
      limits:
        nvidia.com/gpu: 1
```

**场景 2：临时隔离节点（用于维护）**

```bash
# 标记节点为维护模式
kubectl taint nodes worker-1 maintenance=true:NoExecute

# 查看 Taint 效果
kubectl describe node worker-1 | grep Taints
```

**场景 3：清除 Taint**

```bash
# 清除特定 Taint
kubectl taint nodes worker-1 dedicated-

# 清除所有 Taint
kubectl taint nodes worker-1 dedicated=production:NoSchedule-
# 使用短横线表示删除该 key 下的所有污点
```

## 节点角色与标签体系

KubeSphere 在 Kubernetes 原有角色基础上扩展了企业级节点概念：

### KubeSphere 内置角色

| 角色 | 说明 | 权限范围 |
|---|---|---|
| `node.kubernetes.io/master` | K8s Master 节点 | 控制平面组件 |
| `node.kubernetes.io/worker` | K8s Worker 节点 | 运行工作负载 |
| `kubesphere.io/host` | KubeSphere 托管组件节点 | 存储、网络插件 |

### 自定义节点标签

```yaml
# 示例：标记高性能节点
kubectl label node compute-node-1 \
  compute-type=cpu-intensive \
  environment=production \
  --overwrite  # 覆盖已有标签
```

:::tip 标签命名规范
建议采用 Kubernetes 推荐的 `topology.kubernetes.io/*`、`node.kubernetes.io/*` 标签前缀，自定义标签使用 `company.com/*` 格式，避免与官方标签冲突。
:::

## 节点健康检查

### 基本检查命令

```bash
# 节点状态
kubectl get nodes -o wide

# 节点详情
kubectl describe node <node-name>

# 查看 kubelet 日志
journalctl -u kubelet -n 100 --no-pager

# 检查节点条件
kubectl get nodes -o jsonpath='{range .items[*]}{.metadata.name}{"\t"}{range .status.conditions[*]}{.type}={.status}{" "}{end}{"\n"}{end}'
```

### 常见问题排查

| 问题 | 可能原因 | 排查命令 |
|---|---|---|
| Node NotReady | kubelet 未启动 | `systemctl status kubelet` |
| Node NotReady | 网络不通 | `ping master-ip` |
| Pod 无法调度 | 资源不足 | `kubectl describe node` 查看 allocatable |
| Pod 被驱逐 | 节点压力大 | `kubectl get events --field-selector involvedObject.name=<pod>` |

### KubeSphere 节点监控

在 KubeSphere 控制台中，可以查看节点级别的监控指标：

- CPU 使用率
- 内存使用率
- 磁盘 I/O
- 网络流量
- 进程数量

```bash
# 通过 Prometheus 查看节点指标
kubectl exec -it prometheus-0 -n kubesphere-monitoring-system -- \
  promtool query instant \
  'node_cpu_seconds_total{mode="idle",instance="worker-1"}'
```

## 节点 Drain 与维护

### 安全驱逐 Pod

```bash
# 驱逐前检查
kubectl drain <node-name> \
  --delete-emptydir-pods \
  --ignore-daemonsets \
  --force \
  --dry-run=client

# 实际执行驱逐
kubectl drain <node-name> \
  --delete-emptydir-pods \
  --ignore-daemonsets \
  --force
```

:::danger Drain 会中断服务
StatefulSet 的 Pod 不会自动迁移，Drain StatefulSet 节点需要谨慎操作。对于有状态应用，建议先转移数据再维护节点。
:::

### 节点缩容

```bash
# 从集群移除节点
kubectl delete node <node-name>

# 在目标节点上重置 kubeadm
# （在目标节点上执行）
kubeadm reset -f
rm -rf /etc/kubernetes/manifests/
rm -rf ~/.kube/
```

## 节点调度优化

### 资源预留

KubeSphere 控制台可以设置节点的预留资源，确保系统组件有足够资源运行：

```yaml
apiVersion: v1
kind: Node
metadata:
  name: worker-1
spec:
  config:
    # Kubernetes 1.22+ 使用 kubelet.config
status:
  allocatable:
    cpu: "8"
    memory: 16Gi
  capacity:
    cpu: "8"
    memory: 32Gi
```

### 调度策略示例

```yaml
# Pod 调度到特定节点
apiVersion: v1
kind: Pod
metadata:
  name: webapp
spec:
  nodeSelector:
    disktype: ssd
  affinity:
    podAntiAffinity:
      preferredDuringSchedulingIgnoredDuringExecution:
      - weight: 100
        podAffinityTerm:
          labelSelector:
            matchExpressions:
            - key: app
              operator: In
              values:
              - webapp
          topologyKey: kubernetes.io/hostname
```

## 总结

节点管理是 Kubernetes 运维的基础。掌握添加节点、Taint 管理、健康检查、维护操作，是运维 KubeSphere 集群的必备技能。

**核心要点：**

- 添加节点前确保基础组件就绪
- Taint 是控制调度的利器，谨慎使用 NoExecute
- Drain 前评估对有状态应用的影响
- 定期检查节点健康指标，防患于未然
