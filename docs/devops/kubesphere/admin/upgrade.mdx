# 升级指南：平滑演进的关键

Kubernetes 版本发布频繁，KubeSphere 也在快速迭代。升级是运维工作中风险最高的操作——一旦出问题，可能影响整个集群的可用性。

**升级的核心原则：永远不要在生产环境「试水」。**

本章节介绍 KubeSphere 升级的标准流程、组件顺序、备份策略和验证方法。

## 升级前的准备工作

### 1. 确认当前版本

```bash
# 查看 KubeSphere 版本
kubectl logs -n kubesphere-system ks-installer-xxxxx | grep "KubeSphere"

# 查看 Kubernetes 版本
kubectl version --short

# 查看 KubeSphere 组件版本
helm list -n kubesphere-system
```

### 2. 阅读版本说明

每个版本的 Release Notes 都会说明：

- 支持的 Kubernetes 版本范围
- 新增功能
- 已知问题
- Breaking Changes
- 升级注意事项

:::tip 跳跃式升级限制
KubeSphere 通常不支持跨越大版本升级（如 3.3 → 4.0）。建议逐版本升级，或查看官方是否支持跳过升级。
:::

### 3. 准备目标版本

```bash
# 下载目标版本 installer
wget https://github.com/kubesphere/ks-installer/releases/download/v3.4.0/kubesphere-installer.yaml
wget https://github.com/kubesphere/ks-installer/releases/download/v3.4.0/cluster-configuration.yaml
```

## 备份策略

### 核心数据备份

| 数据类型 | 备份内容 | 工具 |
|---|---|---|
| Kubernetes 资源 | Deployment、Service、ConfigMap 等 | `kubectl get all -o yaml` |
| etcd 数据 | 集群全部状态 | etcdctl snapshot |
| volumes | PVC 数据 | 存储快照 |
| KubeSphere 配置文件 | cluster-configuration | 导出 YAML |

### etcd 备份

```bash
# 使用 etcdctl 备份（需要安装 etcdctl）
ETCDCTL_API=3 etcdctl --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key \
  snapshot save /backup/etcd-snapshot-$(date +%Y%m%d).db

# 验证备份
ETCDCTL_API=3 etcdctl --write-out=table snapshot status /backup/etcd-snapshot-*.db
```

### KubeSphere 资源备份

```bash
#!/bin/bash
# backup-kubesphere.sh

BACKUP_DIR="/backup/kubesphere-$(date +%Y%m%d)"
mkdir -p $BACKUP_DIR

# 备份所有 namespace（排除系统 namespace）
for ns in $(kubectl get ns | awk 'NR>1 && !/kube-|kubesphere|istio/ {print $1}'); do
  echo "Backing up namespace: $ns"
  kubectl get all -n $ns -o yaml > $BACKUP_DIR/$ns-resources.yaml
done

# 备份 KubeSphere 核心配置
kubectl get clusterconfiguration -n kubesphere-system -o yaml > $BACKUP_DIR/cluster-configuration.yaml

# 备份 ks-installer
kubectl get deploy -n kubesphere-system -o yaml > $BACKUP_DIR/ks-installer.yaml

echo "Backup completed: $BACKUP_DIR"
```

## 升级流程

### 方式一：在线升级

适用于有外网访问的环境。

```bash
# 1. 确认备份完成
ls -la /backup/

# 2. 修改 cluster-configuration，指定目标版本
vim cluster-configuration.yaml
# 修改 ks-installer 的 image tag 为目标版本

# 3. 应用升级
kubectl apply -f cluster-configuration.yaml

# 4. 监控升级进度
kubectl logs -n kubesphere-system ks-installer-xxxxx -f

# 5. 验证组件状态
kubectl get pods -n kubesphere-system
```

### 方式二：离线升级

适用于内网环境，需要提前准备镜像。

```bash
# 1. 在有外网的机器上打包镜像
docker pull kubesphere/ks-installer:target-version
docker save kubesphere/ks-installer:target-version | gzip > ks-installer.tar.gz

# 2. 传输到内网环境
scp ks-installer.tar.gz user@k8s-master:/tmp/

# 3. 在内网环境导入镜像
docker load < /tmp/ks-installer.tar.gz

# 4. 推送镜像到私有仓库
docker tag kubesphere/ks-installer:target-version registry.example.com/kubesphere/ks-installer:target-version
docker push registry.example.com/kubesphere/ks-installer:target-version

# 5. 修改配置使用私有仓库
vim cluster-configuration.yaml
# 替换所有镜像地址为私有仓库地址

# 6. 执行升级
kubectl apply -f cluster-configuration.yaml
```

### 组件升级顺序

```
1. Kubernetes (如需升级)
   ↓
2. kubelet/kubectl/kubeadm
   ↓
3. etcd (跟随 Kubernetes)
   ↓
4. CNI 插件 (Calico/Flannel)
   ↓
5. CSI 插件 (存储)
   ↓
6. CoreDNS
   ↓
7. KubeSphere 核心组件 (ks-installer)
   ↓
8. KubeSphere 可插拔组件
   - KS-Account
   - KS-Notifier
   - KS-Console
   - KS-Apiserver
   - KS-Controller-Manager
```

## 升级后验证

### 1. 检查核心组件

```bash
# KubeSphere 系统组件
kubectl get pods -n kubesphere-system

# 所有系统组件健康
kubectl get componentstatuses

# Kubernetes 版本
kubectl version
```

### 2. 检查 API 可用性

```bash
# KubeSphere API
curl -k https://ks-apiserver.kubesphere-system.svc/version

# Kubernetes API
kubectl cluster-info
kubectl get nodes
kubectl get pods --all-namespaces
```

### 3. 登录控制台验证

1. 登录 KubeSphere Web 控制台
2. 检查各个功能模块是否正常
3. 创建测试资源验证功能完整性
4. 检查通知系统（邮件、钉钉等）

### 4. 功能测试清单

| 模块 | 测试内容 |
|---|---|
| 工作负载 | 创建 Deployment，验证扩缩容 |
| 存储 | 创建 PVC，验证持久化 |
| 网络 | 创建 Service，验证网络连通 |
| 监控 | 查看 Dashboard，验证指标采集 |
| 日志 | 搜索日志，验证日志收集 |
| 告警 | 触发告警规则，验证通知 |

## 回滚策略

### 如果升级失败

```bash
# 1. 停止升级进程
kubectl delete -f cluster-configuration.yaml

# 2. 恢复 etcd 备份
systemctl stop kube-apiserver
ETCDCTL_API=3 etcdctl snapshot restore /backup/etcd-snapshot-*.db \
  --data-dir=/var/lib/etcd
systemctl start kube-apiserver

# 3. 恢复 Kubernetes 资源
kubectl apply -f /backup/kubesphere-*/namespace-resources.yaml
```

:::warning 回滚的限制
KubeSphere 升级涉及数据库 schema 变更。回滚可能无法完全恢复到升级前状态。建议：
- 升级前确保备份完整
- 在测试环境充分验证
- 制定好回滚决策的触发条件
:::

## 常见问题

### 问题 1：升级卡住不动

```bash
# 查看 installer 日志
kubectl logs -n kubesphere-system ks-installer-xxxxx -f

# 查看所有 pod 状态
kubectl get pods -n kubesphere-system -o wide

# 重启 installer（慎用）
kubectl delete pod -n kubesphere-system ks-installer-xxxxx
```

### 问题 2：组件镜像拉取失败

```bash
# 查看 pod 事件
kubectl describe pod -n kubesphere-system <pod-name>

# 手动拉取镜像
docker pull <image-address>

# 检查镜像仓库配置
kubectl get configmap -n kubesphere-system
```

### 问题 3：API 不可用

```bash
# 检查 ks-apiserver
kubectl get pods -n kubesphere-system | grep apiserver
kubectl logs -n kubesphere-system ks-apiserver-xxxxx

# 检查证书
kubectl get secret -n kubesphere-system | grep ks-apiserver
```

## 升级检查清单

```
升级前检查：
[ ] 确认当前版本和目标版本
[ ] 阅读 Release Notes
[ ] 完整备份 etcd
[ ] 备份 Kubernetes 资源
[ ] 检查节点资源（CPU/内存/磁盘）
[ ] 确认集群节点状态正常
[ ] 准备回滚方案

升级中监控：
[ ] 监控 etcd 健康
[ ] 监控 API Server 可用性
[ ] 监控核心组件启动状态
[ ] 记录升级日志

升级后验证：
[ ] 所有系统组件 Running
[ ] Web 控制台可登录
[ ] API 可访问
[ ] 功能测试通过
[ ] 监控指标正常
```

## 总结

升级是运维的「大考」。做好充分准备，严格执行流程，留有回滚余地。

**核心要点：**

- 升级前必须完整备份
- 在测试环境先验证
- 按正确顺序升级组件
- 升级后全面验证功能
- 制定清晰的回滚触发条件
