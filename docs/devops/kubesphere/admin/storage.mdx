# 存储管理：持久化的「地基」

「数据库 Pod 重启后，数据还在吗？」

这是每个 Kubernetes 新手都会问的问题。答案是：**默认情况下，不在**。容器的文件系统是临时的，Pod 重建意味着数据丢失。要让数据持久化，必须依赖 Kubernetes 的存储体系——StorageClass、PersistentVolume、PersistentVolumeClaim。

KubeSphere 提供了可视化的存储管理界面，但背后的原理和运维工作，依然需要管理员深入理解。

## 存储架构概述

Kubernetes 存储的核心是 PV/PVC 机制：

```
┌─────────────────────────────────────────────────────────┐
│                      Pod                                 │
│  ┌─────────────────────────────────────────────────┐    │
│  │            PersistentVolumeClaim (PVC)          │    │
│  │  "我需要一个 10Gi 的存储，读写模式为 ReadWriteOnce"│   │
│  └─────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────┘
                         ↓ 绑定
┌─────────────────────────────────────────────────────────┐
│                 PersistentVolume (PV)                   │
│  "这是一个 20Gi 的 NFS 存储，已经准备好被使用了"          │
└─────────────────────────────────────────────────────────┘
                         ↓ 供给
┌─────────────────────────────────────────────────────────┐
│                    StorageClass (SC)                    │
│  "动态供给器，根据 PVC 请求自动创建 PV"                   │
└─────────────────────────────────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────────┐
│              底层存储后端 (NFS/Ceph/Cloud)               │
└─────────────────────────────────────────────────────────┘
```

**三个核心概念：**

- **PVC（PersistentVolumeClaim）**：应用声明需要的存储规格
- **PV（PersistentVolume）**：实际的存储卷，先创建好或动态供给
- **StorageClass**：存储的「类」，定义使用哪种存储后端和供给策略

## StorageClass 管理

### 查看已有 StorageClass

```bash
kubectl get storageclass

# 输出示例
NAME                  PROVISIONER                  RECLAIMPOLICY
standard (default)    k8s.io/minikube-hostpath     Delete
nfs-client            fuseim.pri/ifs               Delete
csi-driver            disk.csi.qingcloud.com       Delete
```

### 创建 NFS StorageClass

```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: nfs-storage
  annotations:
    storageclass.kubernetes.io/is-default-class: "true"
provisioner: fuseim.pri/ifs  # NFS provisioner
parameters:
  archiveOnDelete: "false"  # 删除 PVC 时不保留数据
reclaimPolicy: Retain  # 推荐设置为 Retain，防止误删
volumeBindingMode: Immediate  # Immediate/WaitForFirstConsumer
```

:::warning reclaimPolicy 选择
- **Delete**：删除 PVC 时自动删除 PV 和数据（**危险！**）
- **Retain**：删除 PVC 时保留 PV 和数据（**推荐**）
- **Recycle**：删除 PVC 时清空数据再释放 PV（已废弃）
:::

### WaitForFirstConsumer 模式

对于需要区域亲和性的存储，建议使用 `WaitForFirstConsumer`：

```yaml
volumeBindingMode: WaitForFirstConsumer
# Pod 被调度到某个节点后，才绑定 PV
# 确保 Pod 和 PV 在同一可用区，减少跨区流量
```

## 动态供给

动态供给是 StorageClass 的核心能力——应用申请 PVC 时，自动创建对应的 PV。

### KubeSphere 控制台操作

1. 进入「存储管理」→「存储类」
2. 点击「创建」
3. 选择存储后端（云厂商 CBS、NFS、Ceph 等）
4. 配置参数

### AWS EBS 示例

```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: aws-ebs-sc
provisioner: ebs.csi.aws.com
parameters:
  type: gp3
  csi.storage.k8s.io/fstype: ext4
  encrypted: "true"
reclaimPolicy: Retain
volumeBindingMode: WaitForFirstConsumer
allowVolumeExpansion: true  # 允许扩容
```

## 项目级存储配额

KubeSphere 支持在项目（Namespace）级别设置存储配额，控制资源使用。

### 创建存储配额

```yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: storage-quota
  namespace: my-project
spec:
  hard:
    requests.storage: 100Gi    # 最多申请 100Gi
    persistentvolumeclaims: "10"  # 最多 10 个 PVC
    # 特定 StorageClass 配额
    persistentvolumeclaims.storageclass.storage.k8s.io/requests.storage: 50Gi
```

### 应用存储配额

```yaml
apiVersion: v1
kind: LimitRange
metadata:
  name: storage-limits
  namespace: my-project
spec:
  limits:
  - type: PersistentVolumeClaim
    max:
      storage: 50Gi
    min:
      storage: 1Gi
    default:
      storage: 10Gi  # 未指定时默认大小
```

:::tip 配额设计建议
- 开发环境：限制存储总量，防止资源浪费
- 生产环境：适当放宽，确保业务可用
- 定期 Review：删除无用的 PVC 释放存储
:::

## 常用存储后端

### NFS

```yaml
# nfs-subdir-external-provisioner
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: nfs-client
provisioner: k8s-sigs.io/nfs-subdir-external-provisioner
parameters:
  archiveOnDelete: "false"
  onDelete: "delete"
nfs:
  server: 192.168.1.100
  path: /data/nfs
```

**适用场景**：通用文件存储、开发测试环境

### 云厂商块存储

| 云厂商 | StorageClass 类型 | 特点 |
|---|---|---|
| AWS | gp2/gp3/io1/io2 | AWS EBS |
| GCP | standard/ssd | GCP PD |
| 阿里云 | alicloud-disk-efficiency | ESSD/SSD |
| 腾讯云 | cbs | CBS 高效云盘 |

### Ceph RBD

```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: ceph-rbd
provisioner: rbd.csi.ceph.com
parameters:
  clusterID: my-cluster
  poolName: kube-pool
  csi.storage.k8s.io/controller-expand-secret-name: csi-rbd-secret
  csi.storage.k8s.io/node-stage-secret-name: csi-rbd-secret
  csi.storage.k8s.io/provisioner-secret-name: csi-rbd-secret
```

## PersistentVolume 操作

### 手动创建 PV

```yaml
apiVersion: v1
kind: PersistentVolume
metadata:
  name: pv-manual
  labels:
    type: local
spec:
  capacity:
    storage: 10Gi
  volumeMode: Filesystem
  accessModes:
    - ReadWriteOnce
  persistentVolumeReclaimPolicy: Retain
  storageClassName: manual
  claimRef:
    namespace: default
    name: pvc-manual
  nfs:
    path: /data/pv-manual
    server: 192.168.1.100
```

### 查看 PV 状态

```bash
# 查看所有 PV
kubectl get pv

# 查看 PV 详情
kubectl describe pv <pv-name>

# 查看 PVC 绑定状态
kubectl get pvc --all-namespaces
```

### 常见状态及处理

| 状态 | 含义 | 处理方式 |
|---|---|---|
| Available | 可用，等待 PVC 绑定 | 正常状态 |
| Bound | 已绑定到 PVC | 正常状态 |
| Released | PVC 已删除，PV 待处理 | 检查数据，决定保留或删除 |
| Failed | 供给或删除失败 | 查看 Events 排查原因 |

## 存储扩容

### 扩容 PVC

```bash
# 修改 PVC 大小
kubectl patch pvc my-pvc -p '{"spec":{"resources":{"requests":{"storage":"20Gi"}}}}'
```

:::warning 扩容条件
1. StorageClass 必须设置 `allowVolumeExpansion: true`
2. 扩容只支持「增大」，不支持「缩小」
3. 文件系统类型 ext4/XFS 支持在线扩容
:::

### 在线扩容验证

```bash
# 查看 StorageClass 是否允许扩容
kubectl get storageclass <sc-name> -o jsonpath='{.allowVolumeExpansion}'

# Kubernetes 1.24+ 默认启用 CSI 扩容
# 旧版本需要开启 --allow-expansion 参数
```

## 常见问题

### 问题 1：PVC 一直 Pending

```bash
# 排查步骤
kubectl describe pvc <pvc-name>

# 检查 StorageClass 是否存在
kubectl get storageclass

# 检查供给器是否正常运行
kubectl get pods -n kube-system | grep provisioner

# 查看供给器日志
kubectl logs -n kube-system <provisioner-pod>
```

### 问题 2：Pod 无法挂载 Volume

```bash
# 检查挂载信息
kubectl describe pod <pod-name>

# 查看 kubelet 日志
journalctl -u kubelet | grep -i volume
```

### 问题 3：存储空间不足

```bash
# 在 NFS 服务器上查看
df -h /data/nfs

# 清理无用 PV/PVC
kubectl get pv | grep Released
kubectl delete pv <pv-name>
```

## 总结

存储是 Kubernetes 运维中最容易出问题的领域之一。新手常犯的错误包括：

- 使用默认的 Delete reclaimPolicy，误删重要数据
- 未设置存储配额，导致存储滥用
- 忽视存储性能选型，导致 IO 瓶颈

**最佳实践：**

- 生产环境使用 `Retain` reclaimPolicy
- 启用 `WaitForFirstConsumer` 模式
- 设置合理的存储配额
- 定期清理无用的 PVC 和 PV
