# KubeSphere 应用商店

"这个应用我们团队部署过很多次了，每次都要写 YAML、配置参数，能不能有人帮我封装好，我点一下就用？"

这是很多团队的真实诉求。KubeSphere 的应用商店就是来解决这个问题的——它让应用的部署变得像在手机应用商店安装 App 一样简单。

## 应用商店概述

KubeSphere 的应用商店基于 **OpenPitrix** 构建，这是一个开源的多云应用管理平台。OpenPitrix 的核心理念是：**把复杂的 Helm Chart 封装成可管理的应用商品**。

### 什么是 OpenPitrix

OpenPitrix 最初由 QingCloud（青云）开源，后来捐给了 CNCF。它提供了一套完整的应用生命周期管理方案：

- **应用的标准化描述**：统一的包格式和元信息
- **版本管理**：多版本并存，支持版本回滚
- **审核工作流**：支持应用的审核发布流程
- **多租户支持**：不同租户可以有不同的应用仓库

### 应用商店与 Helm 的关系

<Badge text="核心区别" type="tip" /> 应用商店不是 Helm 的替代品，而是 Helm 的增强。Helm 是打包工具，应用商店是管理和分发平台。

```
Helm                                  应用商店
┌───────────────┐                    ┌───────────────┐
│ helm install  │                    │ 应用浏览      │
│ helm upgrade  │     ───►           │ 一键部署      │
│ helm rollback │                    │ 版本管理      │
│ helm list     │                    │ 审核发布      │
└───────────────┘                    │ 多租户隔离    │
                                      └───────────────┘
```

## 应用商店架构

### 核心组件

| 组件 | 作用 |
|---|---|
| App | 应用元信息（名称、描述、图标等） |
| App Version | 应用的具体版本 |
| Repository | 应用仓库（Helm Repo） |
| Runtime | 应用的运行实例 |
| Category | 应用分类 |

### 应用生命周期

```
创建应用 → 上架版本 → 审核通过 → 发布到商店 → 用户部署 → 运行实例
     │                                                    │
     └────────────────────────────────────────────────────┘
                         版本迭代
```

## 应用分类

### 内置分类

KubeSphere 预置了常用的应用分类：

| 分类 | 典型应用 |
|---|---|
| 基础设施 | MySQL、Redis、PostgreSQL |
| 存储 | MinIO、NFS Client Provisioner |
| 网络 | Ingress Nginx、CoreDNS |
| 监控 | Prometheus、Grafana |
| 日志 | Elasticsearch、Fluentd |
| 安全 | Vault、Cert-Manager |

### 自定义分类

管理员可以创建自定义分类：

```bash
kubectl create category custom -n kubesphere-system \
  --name="业务中间件" \
  --description="业务系统常用的中间件"
```

## 部署应用

### 从商店部署

在控制台部署应用是最简单的方式：

```
步骤 1：进入应用商店
  └── 选择目标应用

步骤 2：配置应用参数
  └── 设置副本数、资源限制、环境变量

步骤 3：确认部署
  └── 选择目标项目
  └── 点击"部署"

步骤 4：等待应用就绪
  └── 查看部署状态
```

### 部署配置示例

部署一个 MySQL 应用需要配置的参数：

```yaml
# 典型 MySQL 部署配置
mysql:
  # 基本配置
  imageTag: "8.0"
  architecture: "replication"
  
  # 认证配置
  root:
    password: "MySecretPassword"
    
  # 持久化配置
  primary:
    persistence:
      enabled: true
      size: 20Gi
      storageClass: "nfs-storage"
      
  # 资源配置
  resources:
    limits:
      cpu: 1000m
      memory: 1Gi
    requests:
      cpu: 250m
      memory: 256Mi
      
  # 服务配置
  service:
    type: ClusterIP
    port: 3306
```

### 部署后的管理

应用部署完成后，可以在项目中管理：

| 功能 | 说明 |
|---|---|
| 升级 | 升级到新版本 |
| 回滚 | 回滚到历史版本 |
| 删除 | 删除应用实例 |
| 详情 | 查看应用的 Pod、Service、ConfigMap 等 |

## 应用版本管理

### 版本兼容性

<Badge text="版本策略" type="info" /> 应用商店支持同一应用的多个版本并存，用户可以选择使用哪个版本。

### 回滚操作

如果新版本出现问题，可以快速回滚：

```bash
# 查看应用版本历史
helm history my-app -n my-project

# 回滚到指定版本
helm rollback my-app 1 -n my-project

# 在 KubeSphere 控制台
# 应用 → 选中应用 → 版本管理 → 选择版本 → 回滚
```

## 创建并发布应用

如果你有自己的 Helm Chart，想发布到应用商店，可以按以下步骤操作。

### 步骤一：准备 Helm Chart

确保你的 Helm Chart 符合以下要求：

- Chart.yaml 完整：name、version、description、keywords
- values.yaml 有合理的默认值
- 有清晰的 README 说明

```yaml
# Chart.yaml 示例
apiVersion: v2
name: my-application
version: 1.0.0
appVersion: "1.0"
description: 我的应用程序
keywords:
  - web
  - microservice
maintainers:
  - name: DevOps Team
    email: devops@example.com
```

### 步骤二：创建应用元信息

在 KubeSphere 控制台创建应用：

```
应用管理 → 创建应用
  ├── 名称: my-app
  ├── 描述: 我的第一个应用
  └── 分类: 基础设施
```

### 步骤三：上传应用版本

```
应用详情 → 版本管理 → 创建版本
  ├── 上传 Chart 文件 (my-app-1.0.0.tgz)
  ├── 或指定 Helm Repo URL
  └── 设置版本说明
```

### 步骤四：应用审核

生产环境的应用通常需要审核流程：

```
提交审核 → 管理员审核 → 审核通过 → 发布到商店
```

审核内容包括：

- Chart 格式是否规范
- 配置参数是否合理
- 是否有安全漏洞
- 文档是否完整

## 私有应用仓库

除了 KubeSphere 内置的商店，还可以配置私有应用仓库。

### 添加私有仓库

```bash
# 通过 kubectl 创建 Repository
kubectl apply -f - << 'EOF'
apiVersion: app.kubesphere.io/v1
kind: AppRepository
metadata:
  name: my-private-repo
  namespace: kubesphere-system
spec:
  name: my-private-repo
  url: https://charts.example.com
  type: helm
  description: 我的私有 Helm 仓库
  syncPeriod: 360
  # 认证信息（可选）
  # secretRef:
  #   name: repo-credential
EOF
```

### 仓库类型

| 类型 | 说明 |
|---|---|
| Helm | 标准 Helm Repo |
| Git | 从 Git 仓库拉取 |
| S3 | 从对象存储拉取 |

## 应用商店最佳实践

### 生产环境建议

| 实践 | 说明 |
|---|---|
| 使用官方认证应用 | 优先使用经过审核的应用 |
| 版本锁定 | 生产环境指定具体版本号 |
| 资源配额 | 为应用设置资源限制 |
| 高可用部署 | 有状态应用使用多副本 |

### 应用治理

<Badge text="治理原则" type="tip" /> 应用商店不仅是部署工具，更是标准化和治理的工具。通过统一的应用来源，可以确保生产环境的应用来源可追溯、版本可管控。

建议制定以下规范：

1. **应用准入标准**：明确什么应用可以进入商店
2. **版本更新策略**：多久更新一次基础镜像和安全补丁
3. **废弃流程**：不再维护的应用如何优雅下线
