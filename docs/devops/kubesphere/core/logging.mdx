# KubeSphere 日志系统

凌晨 3 点，用户反馈「订单失败了」。你登录服务器，翻日志，排查了 30 分钟，终于发现是某个服务的数据库连接超时。

但在 KubeSphere 中，这一切都可以更优雅——统一的日志平台，让你在一个界面里追踪跨服务的完整链路。

## 日志系统架构

### 组件架构

```
Pod 日志
    │
    ├── Fluent Bit（边缘收集）→ Kafka（可选缓冲）
    │
    ▼
Fluentd Operator（聚合处理）
    │
    ▼
Elasticsearch（存储）
    │
    ▼
KubeSphere 日志控制台（可视化）
```

### 日志收集流程

```yaml title="log-flow.yaml"
# 日志从应用容器到 Elasticsearch 的完整路径
flow:
  - source: Pod Container
      ↓
  - collector: Fluent Bit (DaemonSet)
      ↓
  - buffer: Kafka / Elasticsearch
      ↓
  - storage: Elasticsearch
      ↓
  - ui: KubeSphere Console
```

## 日志收集配置

### 开启日志收集

```yaml title="logging.enable.yaml"
apiVersion: logging.kubesphere.io/v1alpha2
kind: Logging
metadata:
  name: logging
spec:
  # 日志存储后端
  logsidecar:
    enabled: true
  
  # 收集范围
  inputs:
  - name: container
    type: container
    container:
      # 排除系统组件
      excludeInputs:
      - kube-system
      - istio-system
```

### 自定义日志收集规则

```yaml title="log-input.yaml"
apiVersion: logging.kubesphere.io/v1alpha2
kind: Input
metadata:
  name: app-logs
  namespace: production
spec:
  # 收集指定命名空间的日志
  application:
    namespaces:
    - production
    
    # 排除特定容器
    excludeKubesphere: true
    excludeSystemNamespaces: true
    
    # 日志路径
    logPaths:
    - /var/log/nginx/*.log
    - /app/logs/*.log
    
    # 日志格式
    parsers:
    - name: json
      type: json
      time_key: timestamp
```

### 多租户日志隔离

```yaml title="tenant-log.yaml"
apiVersion: logging.kubesphere.io/v1alpha2
kind: TenantLoggingConfig
metadata:
  name: tenant-config
  namespace: production
spec:
  # 租户级别日志配置
  tenant:
    name: production-team
    matchLabels:
      tenant: production
  
  # 日志保留策略
  retention:
    days: 30
  
  # 日志标签
  labels:
    environment: production
    team: production
```

```

│  关键词:  [ERROR                           ] 🔍          │
├─────────────────────────────────────────────────────────┤
│  时间                │ 级别  │ 服务            │ 日志内容   │
│  2024-01-15 02:30:01│ ERROR │ order-service  │ Database  │
│  2024-01-15 02:30:02│ WARN  │ order-service  │ Retry... │
│  2024-01-15 02:30:03│ ERROR │ payment-service│ Timeout  │
└─────────────────────────────────────────────────────────┘
```

### 日志查询语法

```txt
# 基础查询
# 查询指定命名空间
kubernetes.namespace_name: production

# 查询指定服务
kubernetes.pod_name: order-service-*

# 关键词搜索
log: "ERROR"
log: "database timeout"

# 组合查询
kubernetes.namespace_name: production AND log: "ERROR"

# 正则匹配
kubernetes.pod_name: /order-.*-v[0-9]+/

# 时间范围
@timestamp: ["2024-01-01" TO "2024-01-15"]
```

### 日志关联分析

```txt
# 追踪特定 Trace ID 的日志
log: "traceId=abc123"

# 查询同一时间窗口的所有日志
@timestamp: "2024-01-15T02:30:00" ± 1m

# 查询同一请求的日志
kubernetes.pod_name: /.*/ AND kubernetes.trace_id: "abc123"
```

## 日志告警

### 基于日志的告警

```yaml title="log-alert.yaml"
apiVersion: alerts.kubesphere.io/v1alpha1
kind: Alert
metadata:
  name: frequent-errors
  namespace: production
spec:
  # 告警规则
  rule:
    query: |
      sum(increase(logging_messages_total{level="error"}[5m])) by (service) > 10
    
    for: 5m
    severity: warning
  
  # 通知配置
  notifyPolicy:
    repeatInterval: 1h
  
  channels:
  - type: dingtalk
    webhook: https://oapi.dingtalk.com/robot/send
```

## 应用日志最佳实践

### 结构化日志

在应用中输出 JSON 格式日志：

```java title="Application.java"
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
import com.fasterxml.jackson.databind.ObjectMapper;

public class OrderService {
    private static final Logger log = LoggerFactory.getLogger(OrderService.class);
    private static final ObjectMapper mapper = new ObjectMapper();
    
    public void createOrder(Order order) {
        try {
            // 业务逻辑
            Order result = orderRepository.save(order);
            
            // 结构化日志
            Map<String, Object> logData = new HashMap<>();
            logData.put("event", "order_created");
            logData.put("orderId", result.getId());
            logData.put("customerId", order.getCustomerId());
            logData.put("amount", order.getAmount());
            logData.put("duration", System.currentTimeMillis() - startTime);
            
            log.info(mapper.writeValueAsString(logData));
            
        } catch (Exception e) {
            Map<String, Object> errorLog = new HashMap<>();
            errorLog.put("event", "order_failed");
            errorLog.put("orderId", order.getId());
            errorLog.put("error", e.getMessage());
            errorLog.put("stackTrace", ExceptionUtils.getStackTrace(e));
            
            log.error(mapper.writeValueAsString(errorLog));
        }
    }
}
```

### 日志脱敏

```java title="LogSanitizer.java"
public class LogSanitizer {
    public static String sanitize(String log) {
        // 脱敏手机号
        log = log.replaceAll("1[3-9]\\d{9}", "1****9");
        
        // 脱敏身份证
        log = log.replaceAll("\\d{17}[\\dXx]", "**************");
        
        // 脱敏银行卡
        log = log.replaceAll("\\d{16,19}", "****");
        
        return log;
    }
}
```

## 总结

KubeSphere 日志系统的核心能力：

| 能力 | 说明 |
|---|---|
| **统一收集** | 收集所有命名空间的日志 |
| **多租户隔离** | 不同租户日志分离 |
| **全文检索** | 快速定位问题 |
| **关联分析** | 跨服务追踪 |
| **告警联动** | 异常日志自动告警 |

最佳实践：

1. **结构化日志**：JSON 格式便于分析
2. **合理日志级别**：避免日志噪音
3. **日志脱敏**：保护敏感信息
4. **保留策略**：平衡成本与需求
5. **告警联动**：异常自动通知
