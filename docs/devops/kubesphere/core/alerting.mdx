# KubeSphere 告警系统

想象一下：你正在看世界杯决赛，服务的告警响了。用户已经开始投诉，但你还在排查到底是哪个环节出了问题。

KubeSphere 的告警系统就是为了避免这种场景而设计的——它让告警更智能、响应更及时。

## 告警系统架构

### 核心组件

```
指标采集（Prometheus）
    │
    ▼
告警引擎（Alert Engine）
    │
    ├── 告警规则（Alert Rule）
    │
    ├── 告警策略（Alert Policy）
    │
    └── 告警通知（Alert Notification）
    │
    ▼
通知渠道（邮件/钉钉/企业微信/Slack）
```

### 告警生命周期

```
触发条件满足 → 告警触发 → 发送通知 → 等待响应 → 解决/升级
```

## 告警规则

### 内置规则模板

KubeSphere 提供常用的告警规则模板：

| 模板名称 | 监控指标 | 阈值 |
|---|---|---|
| **CPU 使用率过高** | cpu_usage | `>` 80% |
| **内存使用率过高** | memory_usage | `>` 85% |
| **磁盘使用率过高** | disk_usage | `>` 90% |
| **Pod 重启频繁** | pod_restart_count | `>` 3 |
| **Pod OOM** | oom_count | `>` 0 |
| **网络丢包** | network_packet_loss | `>` 1% |

### 自定义告警规则

```yaml title="custom-alert-rule.yaml"
apiVersion: monitoring.kubesphere.io/v1alpha2
kind: AlertRule
metadata:
  name: node-high-cpu
  namespace: kubesphere-monitoring-system
spec:
  # 告警条件
  expr: |
    (
      1 - (node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes)
    ) * 100 > 80
  
  # 告警持续时间
  for: 5m
  
  # 告警级别
  severity: warning
  
  # 告警标签
  labels:
    type: node
    severity: warning
  
  # 告警注释
  annotations:
    summary: "Node CPU usage is high"
    description: "Node {{ $labels.node }} CPU usage is {{ $value }}%"
    runbook_url: "https://runbook.example.com/node-cpu"
```

### 应用层告警

```yaml title="app-alert-rule.yaml"
apiVersion: monitoring.kubesphere.io/v1alpha2
kind: AlertRule
metadata:
  name: api-high-latency
  namespace: production
spec:
  expr: |
    histogram_quantile(0.99, 
      sum(rate(http_request_duration_seconds_bucket[5m])) by (le, service)
    ) > 1
  
  for: 3m
  severity: critical
  
  labels:
    type: application
  
  annotations:
    summary: "API P99 latency is high"
    description: "Service {{ $labels.service }} P99 latency is {{ $value }}s"
```

## 告警策略

### 多级别告警

| 级别 | 触发条件 | 通知频率 | 升级策略 |
|---|---|---|---|
| **Critical** | 立即触发 | 每 5 分钟 | 自动升级 |
| **Warning** | 持续 5 分钟 | 每 15 分钟 | 可选升级 |
| **Info** | 持续 10 分钟 | 仅一次 | 无升级 |

### 告警策略配置

```yaml title="alert-policy.yaml"
apiVersion: alerts.kubesphere.io/v1alpha1
kind: AlertingPolicy
metadata:
  name: resource-alert-policy
  namespace: production
spec:
  # 告警规则
  rule:
    expr: |
      (
        1 - (node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes)
      ) * 100 > 80
    
    for: 5m
  
  # 告警级别
  severity: warning
  
  # 通知配置
  notifyPolicy:
    # 是否重复发送
    repeatInterval: 1h
    # 重复次数
    repeatCount: 3
  
  # 通知渠道
  channels:
  - type: email
    recipient: ops-team@example.com
  - type: dingtalk
    webhook: https://oapi.dingtalk.com/robot/send
    secret: ${DINGTALK_SECRET}
```

## 告警管理

### 告警抑制

避免同一问题产生大量告警：

```yaml title="alert-inhibition.yaml"
spec:
  inhibitRules:
  - # 当 Critical 告警触发时，抑制同实例的 Warning 告警
    sourceMatch:
      severity: critical
    targetMatch:
      severity: warning
    equal:
    - instance
    - job
```

### 告警静默

```yaml title="alert-silencing.yaml"
apiVersion: alerts.kubesphere.io/v1alpha1
kind: Silence
metadata:
  name: maintenance-window
spec:
  # 静默时间范围
  matchers:
  - name: instance
    value: node-1
  startTime: "2024-01-15T02:00:00Z"
  endTime: "2024-01-15T06:00:00Z"
  
  # 静默原因
  createdBy: admin@example.com
  comment: "Scheduled maintenance window"
```

### 告警聚合

```yaml title="alert-grouping.yaml"
spec:
  # 按服务聚合告警
  groupBy:
  - service
  - severity
  
  # 聚合时间窗口
  groupWait: 30s
  groupInterval: 5m
  
  # 重复告警间隔
  repeatInterval: 4h
```

## 最佳实践

### 1. 告警分级

| 级别 | 定义 | 响应时间 | 升级时间 |
|---|---|---|---|
| **P1 - 紧急** | 服务不可用 | 立即 | 15 分钟 |
| **P2 - 严重** | 部分功能受损 | 30 分钟 | 1 小时 |
| **P3 - 一般** | 非核心问题 | 4 小时 | 无升级 |
| **P4 - 提示** | 潜在风险 | 下一个工作日 | 无升级 |

### 2. 避免告警风暴

:::tip
- 设置合理的告警阈值，避免过于敏感
- 使用抑制规则避免重复告警
- 合理使用聚合，减少通知数量
- 设置静默窗口，避免计划内维护期间的告警
:::

### 3. 告警处理流程

```
收到告警
    │
    ▼
评估影响 → 确认是否需要立即处理
    │
    ├── 是 → 立即响应 → 紧急处理
    │
    └── 否 → 记录 → 安排时间处理
    │
    ▼
解决后 → 确认告警自动消除 → 记录经验教训
```

## 常见问题

### 告警不触发

检查清单：

1. Prometheus 是否正常运行
2. 告警规则表达式是否正确
3. `for` 持续时间是否合理
4. 通知渠道是否配置

### 告警重复发送

解决：

1. 检查 repeatInterval 配置
2. 设置告警抑制规则
3. 使用告警聚合

### 告警漏发

解决：

1. 检查通知渠道配置
2. 验证 Webhook 是否可达
3. 查看 AlertManager 日志

## 总结

KubeSphere 告警系统的核心要点：

| 组件 | 职责 |
|---|---|
| 告警规则 | 定义何时触发告警 |
| 告警策略 | 定义如何通知 |
| 通知渠道 | 定义通知方式 |
| 抑制/静默 | 避免告警风暴 |

最佳实践：

1. **合理的告警阈值**：避免过多误报
2. **分级管理**：区分紧急程度
3. **值班制度**：确保有人响应
4. **复盘改进**：从告警中学习
