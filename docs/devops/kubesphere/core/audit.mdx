# KubeSphere 审计日志

「谁在凌晨 3 点修改了生产环境的配置？」「为什么要删除这个 Deployment？」

审计日志就是来回答这些问题的。它记录了所有对 Kubernetes 资源的操作，让你的集群操作有据可查。

## 审计日志概述

### 什么是审计日志

审计日志记录了对 Kubernetes API 的所有请求，包括：

- 请求来源（用户、服务账户、IP）
- 请求内容（操作、资源、变更）
- 请求结果（成功、失败）
- 时间戳

### 审计日志的价值

| 场景 | 审计日志的作用 |
|---|---|
| 安全调查 | 追踪谁做了什么操作 |
| 合规审计 | 满足监管要求 |
| 问题排查 | 回溯操作历史 |
| 异常检测 | 发现异常操作模式 |

## 审计策略配置

### 创建审计策略

```yaml title="audit-policy.yaml"
apiVersion: audit.kubesphere.io/v1alpha1
kind: AuditPolicy
metadata:
  name: production-audit
  namespace: production
spec:
  # 审计级别
  level: Metadata
  
  # 审计范围
  stages:
  - RequestReceived
  - ResponseStarted
  - ResponseComplete
  
  # 规则配置
  rules:
  # 记录所有写操作
  - level: RequestResponse
    operations:
    - CREATE
    - UPDATE
    - DELETE
  
  # 元数据级别记录读操作
  - level: Metadata
    operations:
    - GET
    - LIST
    - WATCH
  
  # 忽略只读资源
  - level: None
    resources:
    - group: ""
      resources: ["healthz", "readyz", "livez"]
```

### 多级审计策略

```yaml title="multi-level-audit.yaml"
spec:
  # 生产环境使用严格审计
  level: RequestResponse
  
  rules:
  # 所有命名空间的所有操作
  - level: RequestResponse
    namespaces: ["*"]
  
  # 敏感操作详细记录
  - level: RequestResponse
    resources:
    - group: ""
      resources: ["secrets", "configmaps"]
      resourceNames: ["*"]  # 包括所有 secrets
  
  # 系统组件只记录元数据
  - level: Metadata
    users: ["system:kube-controller-manager", "system:kube-scheduler"]
  
  # 只读操作记录元数据
  - level: Metadata
    operations: ["GET", "LIST", "WATCH"]
  
  # 健康检查忽略
  - level: None
    nonResourceURLs:
    - /healthz
    - /version
```

## 审计规则详解

### 规则级别

| 级别 | 记录内容 |
|---|---|
| **None** | 不记录 |
| **Metadata** | 请求元数据（用户、时间、资源） |
| **Request** | 请求体（不含响应） |
| **RequestResponse** | 请求和响应 |
| **FullRequestResponse** | 完整请求和响应 |

### 规则匹配

```yaml title="audit-rules.yaml"
spec:
  rules:
  # 按用户匹配
  - level: RequestResponse
    users: ["admin", "deployment-bot"]
  
  # 按用户组匹配
  - level: Metadata
    userGroups: ["system:masters"]
  
  # 按操作匹配
  - level: RequestResponse
    operations:
    - CREATE
    - UPDATE
    - DELETE
  
  # 按资源匹配
  - level: RequestResponse
    resources:
    - group: ""
      resources: ["secrets", "configmaps"]
    - group: "apps"
      resources: ["deployments", "statefulsets"]
  
  # 按命名空间匹配
  - level: RequestResponse
    namespaces: ["production", "staging"]
  
  # 按非资源 URL 匹配
  - level: None
    nonResourceURLs:
    - "/healthz"
    - "/metrics"
```

## 合规审计

### SOC2 合规

```yaml title="soc2-audit.yaml"
spec:
  # SOC2 要求：记录所有操作
  level: RequestResponse
  
  # 保留期限（通常 1 年）
  retentionDays: 365
  
  # 不能修改或删除审计日志
  immutable: true
```

### 等保合规

```yaml title="security-audit.yaml"
spec:
  # 等保要求：详细审计
  level: RequestResponse
  
  # 保留期限（通常 6 个月以上）
  retentionDays: 180
  
  # 告警配置
  alerts:
  - name: sensitive_operation
    condition: |
      verb in (CREATE, UPDATE, DELETE) AND
      objectRef.resource in (secrets, configmaps) AND
      objectRef.namespace in (production)
    severity: warning
    channels:
    - email
    - dingtalk
```

## 审计告警

### 异常操作告警

```yaml title="audit-alert.yaml"
apiVersion: alerts.kubesphere.io/v1alpha1
kind: AuditAlert
metadata:
  name: suspicious-delete
spec:
  # 告警条件
  condition: |
    verb == "DELETE" AND
    (
      objectRef.namespace == "production" OR
      objectRef.resource in ["clusterroles", "clusterrolebindings"]
    )
  
  # 持续时间
  for: 0s
  
  # 告警级别
  severity: critical
  
  # 通知
  notify:
  - type: email
    recipients:
    - security@example.com
  - type: dingtalk
```

### 操作频率告警

```yaml title="frequency-alert.yaml"
spec:
  # 检测异常操作频率
  condition: |
    count() by (user.username) > 100
    within 5m
  
  for: 1m
  severity: warning
  
  # 可能表示脚本误操作或攻击
```

## 总结

KubeSphere 审计日志的核心能力：

| 能力 | 说明 |
|---|---|
| **全面记录** | 所有 API 操作都有记录 |
| **灵活策略** | 支持多级别审计规则 |
| **多后端支持** | Elasticsearch、Kafka、Webhook |
| **合规支持** | 满足 SOC2、等保等合规要求 |
| **告警联动** | 异常操作自动告警 |

最佳实践：

1. **合理设置审计级别**：平衡日志量和信息量
2. **配置保留策略**：满足合规要求
3. **启用告警**：异常操作及时发现
4. **定期审计**：主动发现问题
5. **保护审计日志**：防止篡改
