# KubeSphere 多租户体系

凌晨 2 点，你的老板发来消息："明天要给外包团队开一个测试环境，但不能让他们看到我们正式环境的任何东西，能搞定吗？"

这是一个典型的多租户场景。

在 Kubernetes 中，多租户通常意味着创建多个 Namespace，然后用 RBAC 控制权限。但这种方法有两个问题：一是 Namespace 数量多了难以管理；二是权限控制还是太粗粒度，无法实现"工作组"的概念。

KubeSphere 的多租户体系就是来解决这个问题的。它在 Kubernetes 之上构建了一层更符合企业组织架构的抽象。

## 为什么需要多租户

先思考一个问题：Kubernetes 的 Namespace 已经能隔离资源了，为什么还要多租户？

### 原生 Namespace 的局限性

| 问题 | 说明 |
|---|---|
| 缺乏组织抽象 | Namespace 只是容器，缺乏"项目""部门"的概念 |
| 权限管理复杂 | 每个 Namespace 都要单独配置 RBAC |
| 资源配额分散 | 配额只能按 Namespace 设置，无法按业务线汇总 |
| 缺乏统一视图 | 不同 Namespace 的资源无法统一管理 |

### 企业组织结构的映射

<Badge text="核心思想" type="tip" /> KubeSphere 的多租户设计，本质上是在 Kubernetes 中映射企业真实的组织结构。

```
企业组织结构                KubeSphere 多租户
┌─────────────────┐        ┌─────────────────┐
│    公司         │        │    集群         │
├─────────────────┤        ├─────────────────┤
│    事业部 A     │        │    工作空间 A   │
├─────────────────┤        ├─────────────────┤
│  产品线 1      │        │    项目 1       │
│  产品线 2      │        │    项目 2       │
├─────────────────┤        ├─────────────────┤
│    事业部 B     │        │    工作空间 B   │
└─────────────────┘        └─────────────────┘
```

## 租户层级架构

KubeSphere 的多租户体系分为五层，从上到下依次是：

```
┌────────────────────────────────────────────────────────────┐
│                     平台层 (Platform)                       │
│                 整个 KubeSphere 系统                        │
├────────────────────────────────────────────────────────────┤
│                    集群层 (Cluster)                         │
│                 Kubernetes 集群本身                        │
├────────────────────────────────────────────────────────────┤
│                   工作空间层 (Workspace)                    │
│              隔离不同部门/业务线/客户                       │
├────────────────────────────────────────────────────────────┤
│                    项目层 (Project)                         │
│              隔离不同应用或微服务                          │
├────────────────────────────────────────────────────────────┤
│                   DevOps 层 (DevOps Project)                │
│              隔离 CI/CD 流水线                             │
└────────────────────────────────────────────────────────────┘
```

### 层级说明

| 层级 | 对应 Kubernetes 资源 | 典型用途 |
|---|---|---|
| 平台 | - | 系统级管理 |
| 集群 | - | 物理资源管理 |
| 工作空间 | 标签隔离 | 部门/业务线隔离 |
| 项目 | Namespace | 应用隔离 |
| DevOps 项目 | Namespace + CRD | CI/CD 隔离 |

## 核心概念详解

### 工作空间 (Workspace)

工作空间是 KubeSphere 多租户的核心概念，它是"资源池"的概念，不是单个应用。

:::tip 理解工作空间
工作空间相当于一个"虚拟组织"，在这个组织里可以创建多个项目、多个 DevOps 项目，共享某些公共资源。
:::

#### 工作空间的特性

- **资源池化**：工作空间内的项目可以共享某些配额
- **独立配额**：可以给整个工作空间设置资源上限
- **独立成员**：工作空间成员只能访问该工作空间内的资源
- **独立角色**：工作空间管理员、项目管理员等角色

#### 创建工作空间

```yaml
apiVersion: tenant.kubesphere.io/v1alpha2
kind: Workspace
metadata:
  name: sales-dept
  labels:
    workspace-template: default
spec:
  placement:
    cluster: cluster-1
  template:
    spec:
      manager: admin@sales.com
```

### 项目 (Project)

项目对应 Kubernetes 的 Namespace，但在 KubeSphere 中有更丰富的语义。

<Badge text="项目 vs Namespace" type="info" /> 项目就是 Kubernetes Namespace，只是 KubeSphere 在其上封装了更多的业务逻辑和权限控制。

#### 项目能力

- **应用管理**：部署和管理应用
- ** Ingress 管理**：管理外部访问
- **配置管理**：管理 ConfigMap、Secret
- **存储管理**：申请和管理持久化存储
- **权限控制**：项目级别的角色分配

### DevOps 项目

DevOps 项目是专门用于 CI/CD 的工作空间，与普通项目的区别在于：

| 特性 | 普通项目 | DevOps 项目 |
|---|---|---|
| CI/CD 流水线 | 需要集成外部 Jenkins | 内置 Jenkins |
| 凭证管理 | 无 | 有 |
| 二进制制品库 | 无 | 有 (Nexus) |
| 镜像构建 | 手动 | 支持自动化 |

## 角色与权限体系

### 角色继承

KubeSphere 的角色设计遵循"继承"原则：

```
平台管理员
    │
    └──► 集群管理员
            │
            └──► 工作空间管理员
                    │
                    └──► 项目管理员
                            │
                            └──► 项目开发者
```

每个下级角色都继承上级角色的基本权限，再加上自己的特定权限。

### 内置角色

#### 平台级别

| 角色 | 权限范围 |
|---|---|
| platform-admin | 平台最高权限，管理所有集群、工作空间 |
| cluster-admin | 管理指定集群的所有资源 |
| cluster-regular | 查看集群资源，无法修改 |

#### 工作空间级别

| 角色 | 权限范围 |
|---|---|
| workspace-admin | 管理工作空间内的所有资源 |
| workspace-regular | 使用工作空间内的资源 |
| workspace-viewer | 只读访问工作空间 |

#### 项目级别

| 角色 | 权限范围 |
|---|---|
| admin | 项目最高权限 |
| operator | 部署和管理应用 |
| viewer | 只读访问 |

### 自定义角色

如果内置角色无法满足需求，可以创建自定义角色：

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: custom-project-creator
rules:
- apiGroups: ["", "apps", "extensions"]
  resources: ["namespaces", "deployments", "services"]
  verbs: ["get", "list", "create"]
```

## 资源配额管理

### 多级配额

KubeSphere 支持在多个层级设置资源配额：

```
┌────────────────────────────────────────────────────────────┐
│                     集群级配额                              │
│                  集群总容量上限                             │
├────────────────────────────────────────────────────────────┤
│                     工作空间级配额                          │
│              限制该工作空间可用的资源                       │
├────────────────────────────────────────────────────────────┤
│                      项目级配额                              │
│              限制该项目可创建的资源上限                      │
└────────────────────────────────────────────────────────────┘
```

### 配额配置示例

```yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: project-quota
  namespace: demo-project
spec:
  hard:
    requests.cpu: "4"
    limits.cpu: "8"
    requests.memory: 8Gi
    limits.memory: 16Gi
    pods: "100"
    services: "20"
```

### LimitRange

除了 ResourceQuota，还可以使用 LimitRange 设置 Pod 的默认资源限制：

```yaml
apiVersion: v1
kind: LimitRange
metadata:
  name: default-limits
  namespace: demo-project
spec:
  limits:
  - type: Container
    default:
      cpu: 500m
      memory: 512Mi
    defaultRequest:
      cpu: 100m
      memory: 128Mi
```

## 多租户隔离实现

### 认证与鉴权流程

```
用户请求
    │
    ▼
1. 认证 (Authentication)
   └── 验证用户身份 (Token / LDAP / OAuth)
   │
    ▼
2. 租户识别 (Tenant Resolution)
   └── 识别用户属于哪个 Workspace
   │
    ▼
3. 鉴权 (Authorization)
   └── 检查用户角色是否有权限访问资源
   │
    ▼
4. 资源过滤 (Resource Filtering)
   └── 只返回用户有权限访问的资源
   │
    ▼
返回响应
```

### 网络隔离

<Badge text="重要" type="warning" /> KubeSphere 默认不强制网络隔离，项目之间可以相互通信。如需网络隔离，需要手动配置 NetworkPolicy。

```yaml
# 禁止其他命名空间访问当前命名空间
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
  namespace: demo-project
spec:
  podSelector: {}
  policyTypes:
  - Ingress
  - Egress
```

### 存储隔离

每个项目只能使用被管理员分配给该项目的存储类：

```yaml
# 管理员创建存储配额
apiVersion: tenant.kubesphere.io/v1alpha2
kind: WorkspaceStorageQuota
metadata:
  name: sales-quota
  namespace: sales-workspace
spec:
  storageClassName: managed-nfs
  limit: 500Gi
```

## 实际使用场景

### 场景一：外包团队隔离

```
需求：外包团队需要一个独立的测试环境，不能访问正式环境

方案：
1. 创建 "外包团队" 工作空间
2. 在工作空间下创建测试项目
3. 创建外包团队成员账户
4. 分配 workspace-regular 角色
5. 外包团队只能看到和操作自己工作空间的内容
```

### 场景二：部门资源配额

```
需求：销售部门的应用经常需要更多资源，需要单独控制配额

方案：
1. 创建 "sales" 工作空间
2. 设置工作空间级配额：CPU 100核，内存 500G
3. 在工作空间内创建多个项目
4. 各项目共享工作空间配额，由项目管理员自行分配
```
