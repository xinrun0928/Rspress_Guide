# KubeSphere 系统架构

很多人第一次看到 KubeSphere 的架构图时会觉得复杂——确实，它整合了几十个组件。但只要理解了分层设计思想，你会发现它其实很有条理。

这一节，我们从整体到局部，层层拆解 KubeSphere 的架构设计。

## 架构分层总览

KubeSphere 采用经典的四层架构设计：

```
┌─────────────────────────────────────────────────────────────┐
│                      用户层 (ks-console)                     │
│                  Vue.js + Ant Design 单页应用                 │
├─────────────────────────────────────────────────────────────┤
│                      API 网关层 (ks-apiserver)               │
│                     统一认证、统一鉴权                        │
├─────────────────────────────────────────────────────────────┤
│                      核心服务层 (Controller)                  │
│  ks-controller-manager │ ks-scheduler │Notifier │Audit      │
├─────────────────────────────────────────────────────────────┤
│                      Kubernetes 基础层                       │
│    etcd │ kube-apiserver │ kube-controller-manager │ kubelet │
└─────────────────────────────────────────────────────────────┘
```

<Badge text="分层原则" type="tip" /> 每一层都只依赖下层的能力，层与层之间通过定义好的接口通信。这种设计让 KubeSphere 可以很好地与 Kubernetes 解耦，也便于独立升级各个组件。

## 前端控制台：ks-console

KubeSphere 的 Web 控制台基于 Vue.js + Ant Design 构建，运行在独立的 Pod 中，通过 Ingress 对外暴露服务。

### 核心职责

- 提供图形化的运维界面
- 处理用户交互逻辑
- 调用后端 API 完成操作
- 实时响应集群状态变化

### 技术栈

| 技术 | 用途 |
|---|---|
| Vue 3 | 前端框架 |
| Ant Design Vue | UI 组件库 |
| Axios | HTTP 客户端 |
| WebSocket | 实时事件推送 |

### 内部服务调用

```
用户浏览器
    │
    ▼
ks-console (Vue SPA)
    │
    ▼
ks-apiserver (认证/鉴权)
    │
    ├──► ks-controller-manager (业务逻辑)
    │
    ├──► Kubernetes API Server (资源操作)
    │
    └──► 其他后端服务
```

## API 网关层：ks-apiserver

ks-apiserver 是 KubeSphere 的统一入口，类似于 Kubernetes 的 kube-apiserver。它承担了以下职责：

### 统一认证

- 支持本地账户、LDAP、OAuth2 多种认证方式
- Token 颁发和验证
- Session 管理

### 统一鉴权

- 多租户权限校验
- 资源访问控制
- 操作审计

### 代理转发

- 将前端请求路由到对应的后端服务
- 请求参数转换
- 响应数据格式化

### 核心配置

```yaml
# ks-apiserver 核心配置
apiVersion: v1
kind: ConfigMap
metadata:
  name: ks-apiserver
  namespace: kubesphere-system
data:
  # JWT 签名密钥
  jwt_secret: "kubesphere"
  # 认证模式
  authentication_mode: "login"
  # 多租户配置
  multi_tenant_default_level: "workspace"
```

## 核心服务层：ks-controller-manager

这是 KubeSphere 最核心的业务逻辑层，包含多个独立运行的控制器。

### 控制器列表

| 控制器 | 职责 |
|---|---|
| ks-account | 账户和权限管理 |
| ks-s2irun | DevOps 流水线运行管理 |
| ks-devops | CI/CD 任务调度 |
| ks-scheduler | 扩展 Kubernetes 调度策略 |
| ks-monitor | 监控数据聚合 |
| ks-events | 事件收集和存储 |
| notifier | 告警通知发送 |
| audit | 审计日志处理 |

### 控制器协作示例

以创建一个 DevOps 流水线为例，看看各控制器如何协作：

```
1. 用户在 ks-console 点击"创建流水线"
   │
   ▼
2. ks-console 调用 ks-apiserver
   │
   ▼
3. ks-apiserver 写入 Pipeline 资源到 etcd
   │
   ▼
4. ks-devops 控制器监听到新 Pipeline
   │
   ▼
5. ks-devops 调用 Jenkins API 创建 Job
   │
   ▼
6. 用户在 Jenkins 中看到的流水线
```

## Kubernetes 基础层

KubeSphere 依赖的基础设施层，包括 Kubernetes 原生组件和第三方组件。

### Kubernetes 核心组件

| 组件 | KubeSphere 中的作用 |
|---|---|
| kube-apiserver | 资源 CRUD 操作 |
| etcd | 数据存储 |
| kube-controller-manager | Pod/Deployment 管理 |
| kube-scheduler | Pod 调度 |
| kubelet | 节点代理 |

### 扩展组件

| 组件 | 用途 |
|---|---|

| Prometheus | 监控数据收集 |
| Elasticsearch | 日志存储 |
| FluentBit | 日志收集 |
| Istio | 服务网格 |
| Jenkins | CI/CD 引擎 |

## 多租户架构

KubeSphere 的多租户设计是其区别于普通 Kubernetes 管理界面的核心能力。

### 租户层级

```
┌────────────────────────────────────────────────────────────┐
│                    平台管理员 (Platform Admin)              │
│                    最高权限，管理整个集群                    │
├────────────────────────────────────────────────────────────┤
│                      集群管理员 (Cluster Admin)              │
│                      管理集群级别资源                        │
├────────────────────────────────────────────────────────────┤
│                    工作空间管理员 (Workspace Admin)          │
│                    创建和管理项目、DevOps 项目              │
├────────────────────────────────────────────────────────────┤
│                      项目管理员 (Project Admin)              │
│                    管理项目内的应用和资源                    │
├────────────────────────────────────────────────────────────┤
│                      开发者 (Developer)                      │
│                    部署应用、查看日志                        │
└────────────────────────────────────────────────────────────┘
```

### 层级与 Kubernetes 资源映射

<Badge text="映射关系" type="info" /> KubeSphere 的每一层都对应 Kubernetes 中的特定资源，理解这个映射关系是理解多租户的关键。

| KubeSphere 层级 | Kubernetes 资源 |
|---|---|
| 平台 | 整个 Kubernetes 集群 |
| 工作空间 | 无直接对应，使用标签隔离 |
| 项目 | Namespace |
| DevOps 项目 | Namespace + 特定 CRD |

### 权限控制流

```
用户请求 (Token)
    │
    ▼
ks-apiserver 验证 Token
    │
    ▼
ks-apiserver 查询用户角色和 Workspace
    │
    ▼
ks-apiserver 检查 Workspace → Project 关联
    │
    ▼
允许访问，返回数据 / 拒绝请求
```

## 服务间通信

KubeSphere 内部的服务通过 Kubernetes Service 相互发现和通信。

### 服务发现

所有 KubeSphere 组件都通过 Service 名称相互访问：

| 服务名 | 命名空间 | 用途 |
|---|---|---|
| ks-apiserver | kubesphere-system | API 统一入口 |
| ks-controller-manager | kubesphere-system | 业务控制器 |
| jenkins | kubesphere-devops | CI/CD 引擎 |
| minio | kubesphere-storage | 对象存储 |

### 外部依赖

KubeSphere 依赖的外部服务（可选）：

| 服务 | 用途 | 部署模式 |
|---|---|---|
| Elasticsearch | 日志存储 | 内置或外部 |
| Prometheus | 监控数据 | 内置或外部 |
| MySQL | KubeSphere 元数据 | 内置或外部 |
| Redis | 缓存和会话 | 内置或外部 |

## 架构设计亮点

### 可扩展性

<Badge text="设计思想" type="tip" /> KubeSphere 的控制器都是独立运行的，可以根据需要选择性启用或禁用。

```yaml
# 安装时可以选择性启用组件
# KubeKey 配置文件
spec:
  persistence:
    storageClass: ""
  etcd:
    type: kubespary
  common:
    redis:
      enabled: true
    openldap:
      enabled: true
    prometheus:
      enabled: true
    logging:
      enabled: true
    alerting:
      enabled: true
    devops:
      enabled: true
    servicemesh:
      enabled: true
```

### 高可用设计

生产环境的 KubeSphere 应该部署为高可用模式：

```
┌─────────────────────────────────────────────────────────────┐
│                      Load Balancer                          │
│                    (Nginx / 云 LB)                          │
├─────────────────────────────────────────────────────────────┤
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐        │
│  │  ks-console │  │  ks-console │  │  ks-console │        │
│  │    Pod 1    │  │    Pod 2    │  │    Pod 3    │        │
│  └─────────────┘  └─────────────┘  └─────────────┘        │
├─────────────────────────────────────────────────────────────┤
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐        │
│  │  ks-apiserver│  │  ks-apiserver│  │  ks-apiserver│        │
│  │    Pod 1    │  │    Pod 2    │  │    Pod 3    │        │
│  └─────────────┘  └─────────────┘  └─────────────┘        │
└─────────────────────────────────────────────────────────────┘
```
