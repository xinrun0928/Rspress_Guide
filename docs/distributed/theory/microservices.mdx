# 微服务架构设计

<Badge text="分布式系统" type="tip" /> <Badge text="架构设计" type="warning" /> <Badge text="最佳实践" type="info" />

2014 年，Martin Fowler 和 James Lewis 在一篇博文中首次系统地提出了微服务的概念。他们描述了一种将单个应用程序拆分为一组小服务的方法，每个服务运行在自己的进程中，使用轻量级协议通信。

但微服务不是银弹——它带来了分布式系统固有的复杂性。在踏上微服务这条路之前，你需要清楚地知道：**这条路通向哪里，以及路上有哪些坑。**

## 微服务的核心理念

```java
/**
 * 微服务 vs 单体架构
 */

// 单体架构
// - 所有功能在一个应用中
// - 统一部署、统一运行
// - 修改任何功能都需要重新部署整个应用

// 微服务架构
// - 每个功能独立成服务
// - 独立部署、独立运行
// - 可以独立修改和扩展每个服务
```

### 微服务的定义特征

```java
/**
 * 微服务的核心特征
 */

// 1. 单一职责（Single Responsibility）
// 每个服务专注于做好一件事
// 例如：用户服务、订单服务、支付服务

// 2. 进程隔离
// 每个服务运行在独立的进程中
// 可以使用不同技术栈

// 3. 轻量级通信
// 服务间使用 HTTP、gRPC、消息队列等通信
// 不使用重量级的分布式对象

// 4. 去中心化数据管理
// 每个服务拥有自己的数据库
// 避免共享数据库导致的耦合

// 5. 独立部署
// 每个服务可以独立部署
// 支持持续集成和持续交付
```

## 服务间通信

### 同步通信：HTTP/gRPC

```java
/**
 * HTTP 通信
 */
@RestController
public class OrderController {
    
    @Autowired
    private RestTemplate restTemplate;
    
    /**
     * 调用用户服务
     */
    public User getUser(Long userId) {
        return restTemplate.getForObject(
            "http://user-service/users/{id}",
            User.class,
            userId
        );
    }
    
    /**
     * 使用 Feign（声明式 HTTP 客户端）
     */
    @FeignClient("user-service")
    public interface UserClient {
        @GetMapping("/users/{id}")
        User getUser(@PathVariable("id") Long userId);
    }
}

/**
 * gRPC 通信
 */

// 定义 proto 文件
/*
syntax = "proto3";

service UserService {
    rpc GetUser(GetUserRequest) returns (User);
}

message GetUserRequest {
    int64 user_id = 1;
}

message User {
    int64 id = 1;
    string name = 2;
    string email = 3;
}
*/

// 服务实现
public class UserServiceImpl extends UserServiceGrpc.UserServiceImplBase {
    
    @Override
    public void getUser(GetUserRequest req, 
                        StreamObserver<User> responseObserver) {
        User user = userRepository.findById(req.getUserId());
        responseObserver.onNext(user);
        responseObserver.onCompleted();
    }
}
```

### 异步通信：消息队列

```java
/**
 * 消息队列通信
 */
@Service
public class OrderService {
    
    @Autowired
    private MessageQueue mq;
    
    /**
     * 创建订单
     */
    public Order createOrder(OrderRequest request) {
        // 1. 创建订单
        Order order = orderRepository.save(new Order(request));
        
        // 2. 发送消息通知其他服务
        mq.send("order.created", new OrderCreatedEvent(order));
        
        return order;
    }
}

/**
 * 消息消费者
 */
@Service
public class InventoryService {
    
    @Consume(topic = "order.created")
    public void handleOrderCreated(OrderCreatedEvent event) {
        // 扣减库存
        for (OrderItem item : event.getOrder().getItems()) {
            inventoryService.deduct(item.getProductId(), item.getQuantity());
        }
    }
}
```

## 服务治理

### 服务注册与发现

```java
/**
 * 服务注册与发现
 */
public class ServiceDiscovery {
    
    /**
     * 服务注册
     */
    @Service
    public class ServiceRegistry {
        
        public void register(String serviceName, String host, int port) {
            // 1. 注册到注册中心
            consulClient.register(serviceName, host, port);
            
            // 2. 发送心跳
            scheduledExecutor.scheduleAtFixedRate(() -> {
                consulClient.renew(serviceName);
            }, 0, 10, TimeUnit.SECONDS);
        }
    }
    
    /**
     * 服务发现
     */
    @Service
    public class ServiceDiscovery {
        
        public List<Instance> discover(String serviceName) {
            // 从注册中心获取服务实例
            return consulClient.getInstances(serviceName);
        }
        
        public Instance selectOne(String serviceName) {
            List<Instance> instances = discover(serviceName);
            
            // 负载均衡选择
            return loadBalancer.choose(instances);
        }
    }
}
```

 * 限流

@Component
public class RateLimiter {
    
    // 使用令牌桶算法
    private RateLimiter rateLimiter = RateLimiter.create(1000);  // 每秒 1000 个令牌
    
    public void handleRequest(Request request) {
        if (rateLimiter.tryAcquire()) {
            process(request);
        } else {
            throw new RateLimitException("请求过于频繁");
        }
    }
}

## 微服务的挑战

### 1. 分布式事务

```java
/**
 * 微服务中的事务挑战
 */

// 单体架构中，一个事务搞定所有操作
// 微服务架构中，操作分布在多个服务中

/**
 * 解决方案
 */

// 1. 最终一致性
// - 允许短暂不一致
// - 通过补偿机制修复

// 2. 分布式事务框架
// - Seata
// - 2PC/TCC/Saga

// 3. 事件溯源
// - 通过事件驱动保证最终一致
```

### 2. 服务间数据一致性

```java
/**
 * 数据一致性模式
 */

// Saga 模式
public class OrderSaga {
    
    public void createOrder(Order order) {
        // 步骤 1：创建订单
        orderService.create(order);
        
        // 步骤 2：预留库存
        try {
            inventoryService.reserve(order.getItems());
        } catch (Exception e) {
            // 补偿：取消订单
            orderService.cancel(order.getId());
            throw e;
        }
        
        // 步骤 3：处理支付
        try {
            paymentService.pay(order.getPayment());
        } catch (Exception e) {
            // 补偿：释放库存
            inventoryService.release(order.getItems());
            orderService.cancel(order.getId());
            throw e;
        }
    }
}
```

## 总结

微服务架构是一种强大的分布式系统架构模式：

1. **优势**：
   - 独立部署，快速迭代
   - 技术多样性
   - 可扩展性
   - 容错性

2. **挑战**：
   - 分布式系统固有的复杂性
   - 服务间通信和数据一致性
   - 运维复杂度增加

3. **适用场景**：
   - 大型复杂系统
   - 需要快速迭代的互联网产品
   - 需要高可用的服务
