# 分布式共识算法

<Badge text="分布式系统" type="tip" /> <Badge text="一致性核心" type="warning" /> <Badge text="算法综述" type="info" />

想象一个场景：五个人围坐在圆桌旁，需要就「明天中午去哪里吃饭」达成一致。问题是：

1. 有人可能走神（节点故障）
2. 信息传递可能有延迟（网络不可靠）
3. 有人可能给出矛盾的答案（拜占庭故障）

他们能否最终达成一致？

这就是**分布式共识问题**——分布式系统中最核心、最基础的问题之一。

## 什么是共识问题？

共识（Consensus）问题要求一组节点就某个值达成一致：

```java
/**
 * 共识问题的形式化定义
 * 
 * 假设有 N 个进程，每个进程有一个初始值
 * 所有进程需要通过消息传递达成共识
 */
public class Consensus {
    
    /**
     * 终止性（Termination）
     * 每个正确的进程最终都会做出决定
     */
    void termination() {}
    
    /**
     * 协议性（Agreement）
     * 所有做出决定的进程都决定相同的值
     */
    void agreement() {}
    
    /**
     * 有效性（Validity）
     * 决定的值必须是某个进程的提议值
     * 不能凭空产生
     */
    void validity() {}
}
```

### 共识无处不在

共识问题在分布式系统中无处不在：

| 场景 | 共识内容 |
|---|---|
| 分布式锁 | 谁能获得锁 |
| 主节点选举 | 哪个节点成为主节点 |
| 原子提交 | 事务是否提交 |
| 状态机复制 | 执行哪个操作 |
| 分布式配置 | 配置值是什么 |

## 为什么共识问题困难？

共识问题困难的原因在于分布式系统的三个基本挑战：

### 1. 异步通信

```java
// 异步意味着：消息可能延迟到达，可能永远不到达
public class AsyncChallenge {
    
    public void sendMessage(Node target, Message msg) {
        // 发送后立即返回，不等待响应
        // 消息可能在 1ms 后到达
        // 也可能在 10 秒后到达
        // 甚至可能永远不到达（网络故障）
        network.send(target, msg);
    }
    
    public Message receiveMessage() {
        // 可能阻塞等待
        // 可能立即返回（如果没有消息）
        // 可能永远阻塞（如果没有消息且没有超时）
        return network.receive();
    }
}
```

### 2. 节点故障

```java
// 节点可能突然崩溃，不再响应任何消息
public class NodeFailure {
    
    public void processRequest(Request req) {
        if (Math.random() < FAILURE_RATE) {
            // 节点崩溃，不再响应任何后续请求
            throw new NodeCrashException();
        }
        // 正常处理
        return process(req);
    }
}
```

### 3. 无全局时钟

```java
// 没有全局时钟，无法判断事件的先后顺序
public class NoGlobalClock {
    
    // 节点 A 认为：先执行操作 1，再执行操作 2
    // 节点 B 认为：先执行操作 2，再执行操作 1
    // 谁是对的？没有全局时钟，无法判断！
    
    // 解决方案：Lamport 时钟或向量时钟
    LamportClock clock = new LamportClock();
}
```

## 共识算法的分类

根据故障模型，共识算法可以分为两大类：

### 崩溃-容错（Crash Fault Tolerant, CFT）

假设节点只会崩溃，不会产生恶意行为：

```java
/**
 * 崩溃-容错算法
 * 
 * 容忍 f 个节点故障，需要 2f+1 个节点
 */
public class CFTExample {
    
    /**
     * Quorum 机制
     * 
     * N = 3, f = 1
     * 写入需要 W > N/2 节点确认
     * 读取需要 R > N/2 节点确认
     * 
     * W + R > N 保证读写交集
     */
    public void write(String key, String value) {
        // 写入到多数派节点
        int quorum = (nodes.size() / 2) + 1;
        int confirmed = 0;
        
        for (Node node : nodes) {
            if (node.write(key, value)) {
                confirmed++;
            }
        }
        
        if (confirmed < quorum) {
            throw new WriteException("写入多数派失败");
        }
    }
}
```

### 拜占庭-容错（Byzantine Fault Tolerant, BFT）

假设节点可能产生任意错误（被攻击、恶意行为等）：

```java
/**
 * 拜占庭-容错算法
 * 
 * 容忍 f 个拜占庭节点，需要 3f+1 个节点
 * 
 * 为什么需要 3f+1？
 * - 需要 f+1 个正确节点才能产生有效响应
 * - 需要 2f+1 个响应才能覆盖 f 个恶意节点
 * - 合并：3f+1
 */
public class BFTExample {
    
    /**
     * PBFT（Practical Byzantine Fault Tolerance）
     * 
     * 核心思想：三阶段投票
     * 1. Pre-prepare：主节点广播提案
     * 2. Prepare：所有节点广播 Prepare 消息
     * 3. Commit：如果收到 2f+1 个 Prepare，就广播 Commit
     * 
     * 当收到 2f+1 个 Commit 时，达成共识
     */
    public void pbftConsensus(Message message) {
        // Phase 1: Pre-prepare
        if (isPrimary()) {
            broadcastPrePrepare(message);
        }
        
        // Phase 2: Prepare
        waitForPrePrepare();
        broadcastPrepare();
        waitFor2FPlus1Prepare();
        
        // Phase 3: Commit
        broadcastCommit();
        waitFor2FPlus1Commit();
        
        // 共识达成
        execute(message);
    }
}
```

## 经典共识算法

### Paxos

Paxos 是 Leslie Lamport 于 1990 年提出的共识算法，被认为是分布式共识问题的里程碑：

```java
/**
 * Paxos 算法核心概念
 */

// 1. Proposer（提议者）：提出提案
// 2. Acceptor（接受者）：批准提案
// 3. Learner（学习者）：学习被选中的提案

public class PaxosProposer {
    
    private int proposalNumber = 0;
    private int proposalValue = 0;
    
    /**
     * 准备阶段（Prepare）
     */
    public void prepare(int proposalId) {
        proposalNumber = proposalId;
        
        // 向所有 Acceptor 发送 Prepare 请求
        for (Acceptor acceptor : acceptors) {
            acceptor.receivePrepare(proposalId);
        }
        
        // 等待多数派响应
        int promises = waitForPromises(majority);
        
        if (promises >= majority) {
            // 进入接受阶段
            accept();
        }
    }
    
    /**
     * 接受阶段（Accept）
     */
    public void accept() {
        // 向所有 Acceptor 发送 Accept 请求
        for (Acceptor acceptor : acceptors) {
            acceptor.receiveAccept(proposalNumber, proposalValue);
        }
        
        // 等待多数派接受
        int accepted = waitForAccepted(majority);
        
        if (accepted >= majority) {
            // 共识达成
            System.out.println("共识达成: " + proposalValue);
        }
    }
}
```

### Raft

Raft 是 Diego Ongaro 和 John Ousterhout 于 2014 年提出的共识算法，旨在更容易理解和实现：

```java
/**
 * Raft 共识算法的核心组件
 */

// 1. Leader Election（领导者选举）
// 2. Log Replication（日志复制）
// 3. Safety（安全性保证）

public class RaftConsensus {
    
    /**
     * Raft 将共识问题分解为三个独立的子问题：
     * 
     * 1. 领导者选举
     *    - 节点随机超时后发起选举
     *    - 获得多数派投票的节点成为领导者
     *    - 领导者通过心跳维持权威
     * 
     * 2. 日志复制
     *    - 客户端请求发给领导者
     *    - 领导者将日志写入本地并发给Follower
     *    - 多数派确认后提交
     * 
     * 3. 安全性
     *    - 领导者必须包含所有已提交的日志
     *    - 选举时检查候选者日志是否最新
     */
}
```

### ZAB

ZAB（Zookeeper Atomic Broadcast）是 Zookeeper 使用的共识协议：

```java
/**
 * ZAB 协议核心概念
 */

// ZAB 专注于两类消息：
// 1. 事务消息（Transaction）：需要原子广播
// 2. 心跳消息（Heartbeat）：用于领导者选举

public class ZABProtocol {
    
    /**
     * ZAB 的两种模式：
     * 
     * 1. 恢复模式（Recovery）
     *    - 领导者崩溃后，新领导者需要同步数据
     *    - 确保已提交的消息不丢失
     * 
     * 2. 广播模式（Broadcast）
     *    - 正常运行时，使用 ZAB 广播消息
     *    - 类似于两阶段提交，但更高效
     */
    
    /**
     * ZAB 与 Raft 的区别：
     * - Raft 使用任期号，ZAB 使用 epoch
     * - Raft 日志是追加的，ZAB 事务有全局 ID（zxid）
     * - ZAB 的领导者更强：所有写操作必须经过领导者
     */
}
```

## 共识算法对比

| 维度 | Paxos | Raft | ZAB |
|---|---|---|---|
| 提出时间 | 1990 | 2014 | 2007 |
| 可理解性 | 难 | 易 | 中 |
| 领导者 | 弱领导者 | 强领导者 | 强领导者 |
| 成员变更 | 复杂 | 联合共识 | 单节点变更 |
| 应用 | 理论奠基 | 工业主流 | Zookeeper |
| 故障模型 | CFT | CFT | CFT |

## 选择共识算法的考量

选择哪种共识算法，需要考虑以下因素：

```java
// 1. 节点数量
// - 3-5 个节点：大多数共识算法都可以
// - 10+ 个节点：考虑成员变更的复杂性
// - 100+ 个节点：考虑 Paxos 的效率问题

// 2. 故障类型
// - 只容崩溃：Raft、ZAB
// - 容拜占庭：PBFT、需要签名

// 3. 性能需求
// - 强一致、低延迟：单领导者 + 流水线
// - 高吞吐：多领导者、异步复制

// 4. 运维复杂度
// - Raft 实现简单，社区活跃
// - Paxos 理论复杂，但工程实现稳定（Chubby、Spanner）
```

## 总结

共识算法是分布式系统的核心。从 FLP 不可能性原理出发，我们看到了共识问题的理论极限。但工程实践告诉我们：通过合理的假设和机制设计，我们可以在大多数情况下解决共识问题。

Paxos、Raft、ZAB 等算法各有特点：

- **Paxos**：理论基础扎实，但难以理解和实现
- **Raft**：易于理解和实现，工业主流
- **ZAB**：为 Zookeeper 优化，强领导者模型

理解了共识算法，我们就能理解 Raft、Paxos 等算法的具体实现。下一章，让我们深入探讨 [Paxos 算法](/distributed/theory/core/paxos) 的细节。
