# 一致性哈希

<Badge text="分布式系统" type="tip" /> <Badge text="数据分片" type="warning" /> <Badge text="负载均衡" type="info" />

想象一下：你运营着一个图片存储服务，最初有 3 台服务器。用户上传图片时，需要决定「这张图片存到哪台服务器上」。

最简单的方法是哈希：`server = hash(imageId) % 3`。但当服务器从 3 台增加到 4 台时，`hash % 3` 变成了 `hash % 4`，几乎所有图片的存储位置都变了——缓存失效，服务雪崩。

这就是传统哈希的困境：**当服务器数量变化时，大量数据需要重新映射**。

如何解决这个问题？答案是一致性哈希。

## 一致性哈希的核心思想

一致性哈希（Consistent Hashing）由 Karger 等人在 1997 年提出，核心思想是：

1. 将哈希值空间组织成一个环（0 到 2^32-1）
2. 服务器映射到环上的某个位置
3. 数据映射到环上顺时针方向最近的服务器

```java
/**
 * 一致性哈希环
 */
public class ConsistentHashRing {
    
    /**
     * 哈希环结构
     * 
     *        0
     *        │
     *        │
     *   B    │    C
     *  (30%)  │  (40%)
     *        │
     *   A ──────── D
     *  (15%)   │  (15%)
     *          │
     *          │
     *        2^32-1
     * 
     * 数据存储位置：
     * - 落在 0-30% 区间的数据 -> B
     * - 落在 30-70% 区间的数据 -> C
     * - 落在 70-85% 区间的数据 -> A
     * - 落在 85-100% 区间的数据 -> D
     */
    
    private TreeMap<Long, String> ring = new TreeMap<>();
    
    /**
     * 添加服务器
     */
    public void addServer(String serverId) {
        // 将服务器映射到环上的多个位置（虚拟节点）
        for (int i = 0; i < VIRTUAL_NODES; i++) {
            long hash = hash(serverId + "#" + i);
            ring.put(hash, serverId);
        }
    }
    
    /**
     * 查找数据所属的服务器
     */
    public String findServer(String key) {
        long hash = hash(key);
        
        // 找到顺时针方向的第一个服务器
        Map.Entry<Long, String> entry = ring.ceilingEntry(hash);
        
        // 如果超过环的末端，取第一个
        if (entry == null) {
            entry = ring.firstEntry();
        }
        
        return entry.getValue();
    }
}
```

## 为什么需要虚拟节点？

简单的服务器位置映射会导致负载不均：

```java
/**
 * 简单映射的问题
 */
public class SimpleHashRing {
    
    private List<String> servers = Arrays.asList("A", "B", "C");
    
    /**
     * 简单哈希：直接映射到服务器
     */
    public String findServer(String key) {
        long hash = hash(key);
        int index = (int) (hash % servers.size());
        return servers.get(index);
    }
}

/**
 * 问题分析
 */

// 如果哈希不均匀，会导致负载不均
// 如果只有 3 台服务器，理想情况下每台负载 33%
// 但实际可能一台 50%，一台 30%，一台 20%

// 虚拟节点解决方案：
// 每台服务器在环上有多个位置（虚拟节点）
// 虚拟节点越多，负载越均匀
```

```java
/**
 * 虚拟节点实现
 */
public class VirtualNodeHashRing {
    
    private static final int VIRTUAL_NODES = 150;  // 每台服务器的虚拟节点数
    
    private TreeMap<Long, String> virtualRing = new TreeMap<>();
    private Map<String, Integer> realNodeCount = new HashMap<>();
    
    public void addServer(String serverId) {
        realNodeCount.put(serverId, VIRTUAL_NODES);
        
        // 添加 VIRTUAL_NODES 个虚拟节点
        for (int i = 0; i < VIRTUAL_NODES; i++) {
            long hash = hash(serverId + "#VN" + i);
            virtualRing.put(hash, serverId + "#VN" + i);
        }
    }
    
    public String findServer(String key) {
        long hash = hash(key);
        Map.Entry<Long, String> entry = virtualRing.ceilingEntry(hash);
        
        if (entry == null) {
            entry = virtualRing.firstEntry();
        }
        
        // 从虚拟节点名称提取真实服务器 ID
        String virtualNode = entry.getValue();
        return virtualNode.split("#")[0];
    }
}
```

## 添加和删除服务器

一致性哈希的优势在于服务器变更时影响范围小：

```java
/**
 * 服务器变更的影响
 */
public class ServerChangeImpact {
    
    /**
     * 假设有 4 台服务器：S1, S2, S3, S4
     * 
     * 理想分布：每台 25%
     * 
     * 添加新服务器 S5：
     * - 只需要迁移 S5 位置到 S1 之间的数据
     * - 影响范围：约 1/5 = 20% 的数据
     * 
     * 删除服务器 S3：
     * - 只需要迁移 S3 的数据到 S4
     * - 影响范围：约 25% 的数据
     */
    
    /**
     * 计算影响范围
     */
    public double calculateImpact(int totalServers, int change) {
        // 新增/删除一台服务器，影响约 1/N 的数据
        return 1.0 / (totalServers + change);
    }
}
```

### 添加服务器

```java
/**
 * 添加服务器
 */
public class ConsistentHashDemo {
    
    public void addServer(String newServer) {
        // 1. 计算新服务器在环上的位置
        long newHash = hash(newServer);
        
        // 2. 找到顺时针方向的下一个服务器
        String nextServer = findNextServer(newHash);
        
        // 3. 迁移数据
        List<String> dataToMigrate = getDataForServer(nextServer, newHash);
        for (String data : dataToMigrate) {
            // 重新哈希，决定是否迁移到新服务器
            if (shouldMigrateToNew(data, newHash, nextHash)) {
                migrateData(data, newServer);
            }
        }
        
        // 4. 添加到环
        addToRing(newServer, newHash);
    }
    
    /**
     * 判断数据是否应该迁移到新服务器
     */
    private boolean shouldMigrateToNew(String data, long newServerHash, 
                                       long nextServerHash) {
        long dataHash = hash(data);
        
        // 数据在新服务器和下一个服务器之间
        return dataHash >= newServerHash && dataHash < nextServerHash;
    }
}
```

## 一致性哈希的改进

### 带负载上限的一致性哈希

```java
/**
 * 带负载上限的一致性哈希
 */
public class BoundedLoadConsistentHash {
    
    private Map<String, Double> serverLoad = new HashMap<>();
    private Map<String, Integer> serverCapacity = new HashMap<>();
    
    /**
     * 查找服务器（考虑负载）
     */
    public String findServer(String key) {
        long hash = hash(key);
        String server = findServerOnRing(hash);
        
        // 检查服务器是否过载
        double currentLoad = serverLoad.getOrDefault(server, 0.0);
        int capacity = serverCapacity.get(server);
        
        if (currentLoad / capacity > LOAD_THRESHOLD) {
            // 跳过过载服务器
            return findNextAvailableServer(hash);
        }
        
        return server;
    }
    
    /**
     * 找到下一个可用的服务器
     */
    private String findNextAvailableServer(long startHash) {
        for (int i = 0; i < servers.size(); i++) {
            long hash = (startHash + i) % RING_SIZE;
            String server = findServerOnRing(hash);
            
            double currentLoad = serverLoad.getOrDefault(server, 0.0);
            int capacity = serverCapacity.get(server);
            
            if (currentLoad / capacity <= LOAD_THRESHOLD) {
                return server;
            }
        }
        
        // 如果都过载，返回负载最轻的
        return findLeastLoadedServer();
    }
}
```

### 虚拟节点 + 负载均衡

```java
/**
 * 完整的虚拟节点一致性哈希实现
 */
public class CompleteConsistentHash {
    
    private static final int VIRTUAL_NODES = 150;
    private static final int REPLICATION_FACTOR = 3;  // 数据副本数
    
    private TreeMap<Long, String> ring = new TreeMap<>();
    private Map<String, List<String>> serverToVirtual = new HashMap<>();
    
    /**
     * 添加服务器
     */
    public void addServer(String serverId) {
        List<String> virtualNodes = new ArrayList<>();
        
        for (int i = 0; i < VIRTUAL_NODES; i++) {
            String virtualNode = serverId + "#VN" + i;
            long hash = hash(virtualNode);
            
            ring.put(hash, virtualNode);
            virtualNodes.add(virtualNode);
        }
        
        serverToVirtual.put(serverId, virtualNodes);
    }
    
    /**
     * 删除服务器
     */
    public void removeServer(String serverId) {
        List<String> virtualNodes = serverToVirtual.get(serverId);
        
        for (String vn : virtualNodes) {
            long hash = hash(vn);
            ring.remove(hash);
        }
        
        serverToVirtual.remove(serverId);
    }
    
    /**
     * 存储数据（带副本）
     */
    public List<String> put(String key, Object value) {
        List<String> servers = getServers(key);
        List<String> results = new ArrayList<>();
        
        for (String server : servers) {
            storage.put(server, key, value);
            results.add(server);
        }
        
        return results;
    }
    
    /**
     * 获取数据所属的服务器列表（副本）
     */
    public List<String> getServers(String key) {
        List<String> result = new ArrayList<>();
        long hash = hash(key);
        
        Map.Entry<Long, String> entry = ring.ceilingEntry(hash);
        
        for (int i = 0; i < REPLICATION_FACTOR; i++) {
            if (entry == null) {
                entry = ring.firstEntry();
            }
            
            String virtualNode = entry.getValue();
            String serverId = virtualNode.split("#")[0];
            
            if (!result.contains(serverId)) {
                result.add(serverId);
            }
            
            // 移动到下一个虚拟节点
            entry = ring.higherEntry(entry.getKey());
        }
        
        return result;
    }
}
```

    // - 副本位置由 RackAwareStrategy 等决定
    
    /**
     * 示例
     */

    // hash("user123") = 12345678
    // Token 范围：0-2^127
    // 节点 A 负责 Token 0-50000000
    // 节点 B 负责 Token 50000001-100000000
    // 节点 C 负责 Token 100000001-2^127
    // hash("user123") = 12345678，属于节点 A 的范围
    // 数据存储在节点 A 及其副本节点上
}

## 一致性哈希的优缺点

```java
/**
 * 一致性哈希的优缺点
 */

// 优点
// 1. 增删节点影响范围小
//    - 只需要迁移相邻节点之间的数据
//    - 不需要重新哈希所有数据

// 2. 负载均衡
//    - 虚拟节点使负载更均匀
//    - 可以根据服务器性能设置权重

// 3. 扩展性好
//    - 可以渐进增加服务器
//    - 数据迁移平滑

// 缺点
// 1. 实现复杂
//    - 需要维护哈希环
//    - 需要处理虚拟节点

// 2. 仍有数据迁移
//    - 服务器变更时仍有部分数据需要迁移
//    - 虽然范围小了，但不为零

// 3. 难以实现强一致性
//    - 数据在多个副本间的一致性需要额外机制
```

## 总结

一致性哈希是分布式系统中的重要数据结构，它解决了传统哈希在服务器变更时的痛点：

1. **环形结构**：将哈希空间组织成环，服务器和数据都映射到环上
2. **虚拟节点**：通过虚拟节点实现负载均衡
3. **最小影响**：服务器变更时只影响相邻区间的数据

从 Memcached 到 Cassandra，从 Nginx 到 DynamoDB，一致性哈希在众多分布式系统中发挥着重要作用。
