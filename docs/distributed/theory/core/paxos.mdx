# Paxos 算法

|Paxos| |共识算法| |分布式系统|

1990 年，Leslie Lamport 发表了一篇论文，介绍了一种名为 Paxos 的共识算法。这篇论文最初被发表在一个晦涩的希腊小岛上，后来被《TOCS》拒稿，理由是「太难理解」。

直到 1998 年，这篇论文才正式发表。审稿人甚至建议 Lamport 发表一个简化版本。于是诞生了著名的《Paxos Made Simple》——一篇比原版「稍微」简单一点点的论文，开头第一句话就是：

> 「Paxos 算法通常被认为难以理解，尽管它可能是最简单、最明显的共识算法。」

Lamport 曾经半开玩笑地说：**也许 Paxos 算法的难点不在于理解它，而在于让别人相信它是正确的。**

这一章，让我们一起攻克这个「最简单」的算法。

## Paxos 解决什么问题？

Paxos 解决的问题是：**在一组可能发生故障的节点中，如何就某个值达成共识？**

```java
/**
 * Paxos 的典型应用场景
 */

// 1. 分布式锁
// N 个节点需要就「谁获得锁」达成共识
// "节点 3 获得锁"

// 2. 主节点选举
// N 个节点需要就「谁是主节点」达成共识
// "节点 5 是主节点"

// 3. 配置管理
// N 个节点需要就「配置值是什么」达成共识
// "副本数 = 3"

// 4. 状态机复制
// N 个节点需要就「执行哪个操作」达成共识
// "执行操作 42"
```

### 问题的约束

Paxos 假设：

1. **异步通信**：消息可能丢失、延迟、乱序，但最终会送达
2. **节点故障**：节点可能崩溃，但不会产生恶意行为（非拜占庭）
3. **多数派正确**：至少有过半数的节点是正确的

## Paxos 的角色

Paxos 算法中有三种角色：

```java
/**
 * Paxos 的三种角色
 */

// 1. Proposer（提议者）
// 发起提案的节点
// 提出要达成共识的值
class Proposer {
    int pid;              // 提议者 ID
    int proposalNumber;    // 提案编号
    int proposalValue;    // 提案值
}

// 2. Acceptor（接受者）
// 批准提案的节点
// 存储已批准的提案
class Acceptor {
    int highestPromisedNumber = 0;    // 已承诺的最高提案号
    int acceptedNumber = 0;          // 已接受的提案号
    int acceptedValue = null;         // 已接受的值
}

// 3. Learner（学习者）
// 学习被选中的提案
// 不参与决策，只学习结果
class Learner {
    Map<Integer, Integer> acceptedValues;  // 接受的提案
}
```

## Paxos 的两个阶段

Paxos 算法分为两个阶段：**准备阶段（Prepare）和接受阶段（Accept）**。

### 第一阶段：准备阶段（Prepare）

```java
/**
 * Prepare 阶段
 * 
 * Proposer 选择一个提案号 n，向所有 Acceptor 发送 Prepare(n) 请求
 * Acceptor 收到 Prepare(n) 后：
 *   - 如果 n > highestPromisedNumber
 *       承诺不再接受编号 < n 的提案
 *       返回已接受的最高编号提案（如果有）
 *   - 否则，拒绝
 */

// Proposer 端
public class PaxosProposer {
    
    private int proposalNumber = 0;
    private int proposalValue;
    private List<Acceptor> acceptors;
    
    public void prepare() {
        proposalNumber++;  // 生成新的提案编号
        
        // 向所有 Acceptor 发送 Prepare
        Map<Acceptor, PromiseResponse> responses = new HashMap<>();
        
        for (Acceptor acceptor : acceptors) {
            try {
                PromiseResponse resp = acceptor.receivePrepare(proposalNumber);
                responses.put(acceptor, resp);
            } catch (RejectException e) {
                // 被拒绝，需要用更大的提案号重试
            }
        }
        
        // 检查是否收到多数派响应
        if (responses.size() >= majority) {
            // 进入 Accept 阶段
            int highestNumber = 0;
            int highestValue = null;
            
            // 找出已接受的最高编号提案
            for (PromiseResponse resp : responses.values()) {
                if (resp.acceptedNumber > highestNumber) {
                    highestNumber = resp.acceptedNumber;
                    highestValue = resp.acceptedValue;
                }
            }
            
            // 如果有已接受的提案，使用它的值
            // 否则，使用自己的值
            if (highestValue != null) {
                proposalValue = highestValue;
            }
            
            // 进入 Accept 阶段
            accept();
        }
    }
}

// Acceptor 端
public class PaxosAcceptor {
    
    private int highestPromisedNumber = 0;
    private int acceptedNumber = 0;
    private Object acceptedValue = null;
    
    public PromiseResponse receivePrepare(int proposalNumber) {
        if (proposalNumber > highestPromisedNumber) {
            // 承诺不再接受编号更小的提案
            highestPromisedNumber = proposalNumber;
            
            // 返回已接受的提案（如果有）
            return new PromiseResponse(acceptedNumber, acceptedValue);
        } else {
            // 拒绝
            throw new RejectException(highestPromisedNumber);
        }
    }
}
```

### 第二阶段：接受阶段（Accept）

```java
/**
 * Accept 阶段
 * 
 * Proposer 收到多数派 Prepare 响应后
 * 向所有 Acceptor 发送 Accept(n, v) 请求
 * 
 * Acceptor 收到 Accept(n, v) 后：
 *   - 如果 n >= highestPromisedNumber
 *       接受提案，记录提案号和值
 *   - 否则，拒绝
 */

// Proposer 端
public class PaxosProposer {
    
    public void accept() {
        int accepted = 0;
        
        for (Acceptor acceptor : acceptors) {
            try {
                acceptor.receiveAccept(proposalNumber, proposalValue);
                accepted++;
            } catch (RejectException e) {
                // 被拒绝，需要用更大的提案号重新开始
                prepare();  // 重新开始
                return;
            }
        }
        
        // 如果多数派接受，提案被选中
        if (accepted >= majority) {
            System.out.println("提案被选中: number=" + proposalNumber 
                + ", value=" + proposalValue);
        }
    }
}

// Acceptor 端
public class PaxosAcceptor {
    
    public void receiveAccept(int proposalNumber, Object proposalValue) {
        if (proposalNumber >= highestPromisedNumber) {
            // 接受提案
            highestPromisedNumber = proposalNumber;
            acceptedNumber = proposalNumber;
            acceptedValue = proposalValue;
            
            // 通知 Learner
            for (Learner learner : learners) {
                learner.onAccepted(proposalNumber, proposalValue);
            }
        } else {
            throw new RejectException(highestPromisedNumber);
        }
    }
}
```

## Multi-Paxos

基本的 Paxos 算法每次共识都需要两轮 RPC。Multi-Paxos 优化了这一点，允许多次共识复用第一阶段：

```java
/**
 * Multi-Paxos 优化
 * 
 * 当一个 Proposer 连续成为领导者时
 * 可以跳过 Prepare 阶段，直接进行 Accept
 */

// 第一次共识：需要两阶段
public class MultiPaxos {
    
    private int lastExecutedInstance = 0;
    private boolean isLeader = false;
    
    public void propose(int instanceId, Object value) {
        if (!isLeader) {
            // 先尝试成为领导者
            becomeLeader();
        }
        
        if (isLeader) {
            // 领导者可以直接 Accept，跳过 Prepare
            // 一次 RPC 即可
            for (Acceptor acceptor : acceptors) {
                acceptor.receiveAccept(instanceId, value);
            }
        } else {
            // 非领导者使用标准 Paxos
            standardPaxos(instanceId, value);
        }
    }
    
    private void becomeLeader() {
        // 尝试获取租约，获得租约的成为领导者
        int leases = 0;
        for (Acceptor acceptor : acceptors) {
            if (acceptor.grantLease()) {
                leases++;
            }
        }
        
        isLeader = leases >= majority;
    }
}
```

## Paxos 的活性问题

Paxos 存在活性问题：**两个 Proposer 交替prepare，可能导致永远无法达成共识。**

```java
/**
 * 活锁（Livelock）问题
 * 
 * 场景：
 * Proposer A 提出编号 1，收到多数派响应
 * Proposer B 提出编号 2，拒绝了 A 的 Accept
 * Proposer A 提出编号 3，拒绝了 B 的 Prepare
 * Proposer B 提出编号 4，拒绝了 A 的 Accept
 * ...无限循环
 */

// 解决方案：Proposer 随机退避
public class ProposerWithBackoff {
    
    private Random random = new Random();
    
    public void prepare() {
        while (true) {
            int proposalNumber = generateNumber();
            
            int promises = sendPrepare(proposalNumber);
            
            if (promises >= majority) {
                if (accept(promises)) {
                    return;  // 成功
                }
            }
            
            // 随机退避，避免冲突
            try {
                Thread.sleep(random.nextInt(100) + 10);
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        }
    }
}
```

## Paxos 的正确性证明

Paxos 的正确性基于两个核心不变量：

```java
/**
 * Paxos 的两个核心不变量
 */

// 不变量 1：Acceptor 必须首先批准一个提案，才能批准后续编号更大的提案
// 即：如果 Acceptor 批准了 (n, v)
// 那么对于所有 m > n，它只能批准 (m, v)

public class PaxosInvariant {
    
    // 伪代码形式化描述
    /*
    invariant 1:
        if acceptor.accepted(n, v) then
            for all m > n:
                acceptor.state doesn't contain any accept(m, _)
    */
    
    // 这个不变量的意义：
    // 一旦某个值被接受（可能是被多数派），
    // 后续的提案必须使用相同的值（如果知道的话）
}

// 不变量 2：提案被选中后，所有更高编号的提案必须使用相同的值
/*
invariant 2:
    if proposal (n, v) is chosen then
        for all m > n:
            any accepted proposal (m, v') must have v' = v
*/
```

### 证明概要

**命题**：如果提案 (n, v) 被选中，那么所有更高编号的提案都会使用值 v。

**证明**（概要）：

1. 设 (n, v) 被多数派 Acceptor 接受
2. 考虑任意提案 (m, v')，其中 m `>` n
3. Proposer(m) 在 Prepare(m) 时，必然收到至少一个 Acceptor 的响应，该 Acceptor 接受了 (n, v)
4. 因此 Prepare(m) 的响应包含 (n, v)
5. Proposer(m) 必须使用已接受的值 v（或自己的值）
6. 如果 (n, v) 已知，Proposer(m) 必须使用 v

## Paxos 与 Raft 的对比

| 维度 | Paxos | Raft |
|---|---|---|
| 角色划分 | Proposer/Acceptor/Learner | Leader/Follower/Candidate |
| 领导者 | 无强领导者 | 强领导者 |
| 日志连续性 | 允许空洞 | 必须是连续的 |
| 成员变更 | 复杂 | Multi-Paxos |
| 可理解性 | 难 | 较易 |

```java
// Raft 可以看作 Paxos 的工程化实现
// 主要区别：
// 1. Raft 有强领导者：所有写操作必须经过领导者
// 2. Raft 日志是连续的：简化了正确性证明
// 3. Raft 使用任期号：更直观
```

## 总结

Paxos 算法是分布式共识问题的奠基之作。虽然它被认为「难以理解」，但其核心思想其实很简单：

**第一阶段**：Proposer 询问 Acceptor「我可以提议吗？」
**第二阶段**：获得多数派承诺后，Proposer 正式提议

理解 Paxos 的关键在于理解两个核心不变量，它们保证了算法的正确性。

虽然 Raft 等算法更容易理解和实现，但 Paxos 仍然是每个分布式系统工程师必须掌握的基础。
