# CAP 权衡

<Badge text="分布式系统" type="tip" /> <Badge text="架构设计" type="warning" /> <Badge text="实战策略" type="info" />

某视频平台的工程师最近遇到了一个棘手的问题：他们的推荐系统需要在「用户等待时间」和「推荐准确度」之间找到平衡。如果每次推荐都要等待所有相关数据同步完成，用户会感到页面加载缓慢；但如果直接返回已有数据，又可能推荐过期的内容。

这个问题的本质，就是 CAP 权衡的体现。理论告诉我们「鱼和熊掌不可兼得」，但实战中我们需要在「什么时候兼得」「牺牲多少」「如何优雅地降级」这些细节上做出精心设计。

## 为什么需要 CAP 权衡？

CAP 理论告诉我们，在网络分区发生时，必须在一致性和可用性之间做出选择。但现实远比理论复杂：

**分区的发生不是瞬间的**：网络故障往往有一个渐变的过程，可能先出现延迟增大，然后才是完全断连。

**分区的持续时间不确定**：可能是几秒钟，也可能是几小时。

**业务对一致性和可用性的需求不是二元的**：大多数业务既需要一定的一致性保证，又需要一定的可用性保证。

因此，我们需要一套精细化的权衡策略，而不是简单地选择 CP 或 AP。

## 权衡策略一：读写一致性级别调整

一个常见的做法是根据业务的读写特性，动态调整一致性的级别。

### 读多写少场景

对于读多写少的场景，可以采用「主写从读」的策略。写入时保证强一致性，读取时可以有一定程度的一致性让步。

```java
// 主从复制模型
public class ReadWriteStrategy {
    
    // 写入操作：强一致性
    public void write(String key, String value) {
        // 写入主节点
        masterNode.write(key, value);
        // 同步到从节点（强一致）
        for (Node slave : slaveNodes) {
            slave.syncWrite(key, value);
        }
    }
    
    // 读取操作：可以根据需求选择一致性级别
    public String read(String key, ConsistencyLevel level) {
        switch (level) {
            case STRONG:
                // 强一致性读：必须读主节点或多数派
                return masterNode.read(key);
            case QUORUM:
                // Quorum 读：读取多数节点，取最新值
                return quorumRead(key, slaveNodes);
            case EVENTUAL:
                // 最终一致读：读取最近的节点
                return anyNodeRead(key, slaveNodes);
        }
    }
}
```

### 写多读少场景

对于写多读少的场景，可以采用「本地写入 + 异步同步」的策略。写入操作在本地节点完成，然后异步同步到其他节点。

```java
public class WriteHeavyStrategy {
    
    // 本地写入，异步同步
    public void write(String key, String value) {
        // 立即在本地写入（低延迟）
        localNode.write(key, value);
        // 异步同步到其他节点
        asyncSync(key, value);
    }
    
    // 读取操作
    public String read(String key) {
        // 读取本地数据
        String local = localNode.read(key);
        if (local != null) {
            return local;
        }
        // 本地没有，查询其他节点
        return queryRemote(key);
    }
}
```

## 权衡策略二：一致性级别降级

在系统压力较大或发生局部故障时，可以动态降级一致性要求，保证系统整体可用。

```java
public class ConsistencyDegradation {
    
    private AtomicInteger errorCount = new AtomicInteger(0);
    private static final int DEGRADATION_THRESHOLD = 10;
    
    public String read(String key) {
        // 正常情况下强一致读
        if (errorCount.get() < DEGRADATION_THRESHOLD) {
            return strongConsistencyRead(key);
        }
        
        // 触发降级：改为最终一致读
        log.warn("系统进入降级模式，降低一致性要求");
        return eventualConsistencyRead(key);
    }
    
    public void recordError() {
        int errors = errorCount.incrementAndGet();
        if (errors == DEGRADATION_THRESHOLD) {
            // 触发降级告警
            alert("一致性降级已触发");
        }
    }
    
    public void recordSuccess() {
        errorCount.set(0);
    }
}
```

### 降级触发条件

合理的降级策略需要明确的触发条件：

| 触发条件 | 降级策略 |
|---|---|
| 节点响应时间 `>` 500ms | 从强一致降级为 Quorum 一致 |
| 失败节点数 `>` 总数 1/3 | 降级为最终一致读 |
| 分区检测信号触发 | 切换为 AP 模式 |
| 手动触发降级命令 | 根据配置降级到指定级别 |

## 权衡策略三：业务层面的妥协

有时候，一致性和可用性的权衡可以在业务层面优雅地解决，而不是在技术层面硬扛。

### 补偿机制

当无法保证强一致性时，可以引入补偿机制，在事后修复数据一致性问题。

```java
public class CompensationExample {
    
    // 订单支付场景
    public void processPayment(Order order, Payment payment) {
        try {
            // 1. 尝试扣减库存（可能失败）
            boolean stockReserved = inventoryService.reserve(order.getItems());
            if (!stockReserved) {
                throw new StockException("库存不足");
            }
            
            // 2. 处理支付
            boolean paid = paymentService.pay(payment);
            if (!paid) {
                // 补偿：释放库存
                inventoryService.release(order.getItems());
                throw new PaymentException("支付失败");
            }
            
            // 3. 创建订单
            orderService.create(order);
            
        } catch (Exception e) {
            // 记录补偿任务
            compensationService.schedule(new CompensationTask(order));
        }
    }
}
```

### 异步化处理

将强一致性要求高的操作转化为异步消息处理，降低对系统即时响应的要求。

```java
public class AsyncProcessingExample {
    
    private MessageQueue mq = new MessageQueue();
    
    // 写入操作：立即返回，异步处理
    public void submitOrder(Order order) {
        // 1. 立即返回成功（高可用）
        order.setStatus(OrderStatus.SUBMITTED);
        orderRepository.save(order);
        
        // 2. 发送异步消息触发后续处理
        mq.send("order:created", order);
        // 后续的库存扣减、支付处理都是异步的
    }
    
    // 后续处理逻辑
    @Consume("order:created")
    public void handleOrderCreated(Order order) {
        // 扣减库存
        inventoryService.deduct(order.getItems());
        // 通知支付系统
        paymentService.notify(order);
        // 更新订单状态
        order.setStatus(OrderStatus.PROCESSING);
        orderRepository.save(order);
    }
}
```

## 权衡策略四：混合一致性策略

大型系统通常包含多个子系统，每个子系统可以根据其业务特点选择不同的 CP/AP 策略。

```java
public class HybridConsistencyStrategy {
    
    // 用户服务：选择 AP（用户体验优先）
    // 社交信息、动态发布等可以容忍短暂不一致
    public void postActivity(UserId userId, Activity activity) {
        activityService.post(userId, activity);  // 本地写入，高可用
        syncService.asyncSync(activity);          // 异步同步
    }
    
    // 订单服务：选择 CP（交易一致性优先）
    // 订单创建、支付等必须保证强一致
    public Order createOrder(Order order) {
        // 使用分布式事务保证强一致性
        transactionManager.execute(() -> {
            inventoryService.reserve(order.getItems());  // 原子操作
            orderService.create(order);                    // 原子操作
        });
        return order;
    }
    
    // 库存服务：动态选择
    // 根据当前系统状态动态选择 CP 或 AP
    public boolean reserveStock(List<Item> items) {
        if (systemHealthCheck.isHealthy()) {
            // 系统健康：强一致
            return transactionManager.executeWithLock(items);
        } else {
            // 系统承压：最终一致
            return stockService.reserveWithCompensation(items);
        }
    }
}
```

## 一致性级别的工程实现

在实际工程中，我们通常将一致性划分为多个级别，让调用者可以根据需求选择合适的级别：

### 一致性级别定义

```java
public enum ConsistencyLevel {
    /**
     * 强一致性：读取必须读到最新的数据
     */
    STRONG("强一致性", 0),
    
    /**
     * 线性一致性：在强一致性的基础上，保证操作的先后顺序
     */
    LINEARIZABLE("线性一致性", 1),
    
    /**
     * 因果一致性：保证因果相关的操作顺序，不同分支的操作顺序不保证
     */
    CAUSAL("因果一致性", 2),
    
    /**
     * Quorum 一致性：读取多数派节点
     */
    QUORUM("Quorum 一致性", 3),
    
    /**
     * 最终一致性：允许短暂不一致，最终会达到一致
     */
    EVENTUAL("最终一致性", 4);
}
```

### 读写 Quorum 计算

对于使用 Quorum 机制的系统，需要合理计算读写节点数：

```java
public class QuorumCalculator {
    
    /**
     * N: 副本总数
     * R: 读 quorum（读取的节点数）
     * W: 写 quorum（写入的节点数）
     * 
     * 强一致性要求：W + R > N
     */
    public static boolean isStrongConsistent(int N, int R, int W) {
        return R + W > N;
    }
    
    /**
     * 常用配置：
     * - N=3, W=2, R=2: 强一致
     * - N=3, W=1, R=3: 写性能好（快速写入），读性能好（多副本验证）
     * - N=3, W=3, R=1: 读性能好，数据持久性最强
     */
    public static void main(String[] args) {
        // 强一致性配置
        System.out.println(isStrongConsistent(3, 2, 2));  // true
        
        // 快速写入配置
        System.out.println(isStrongConsistent(3, 1, 1));  // false
        
        // 读写均衡配置
        System.out.println(isStrongConsistent(5, 3, 3));  // true
    }
}
```

## 权衡决策框架

面对具体场景，如何做出合理的 CAP 权衡？以下是决策框架：

### 评估维度

| 维度 | 问题 | 权重 |
|---|---|---|
| 数据价值 | 数据不一致带来的损失有多大？ | 高 |
| 响应延迟 | 用户能接受多长的等待时间？ | 中 |
| 服务可用性 | 系统不可用带来的损失有多大？ | 高 |
| 业务场景 | 是核心交易场景还是辅助功能？ | 高 |

### 决策树

```
开始评估
    │
    ├─ 数据不一致代价极高？
    │       │
    │       ├─ 是 ──→ 选择 CP（强一致）
    │       │
    │       └─ 否
    │               │
    │               ├─ 可以容忍短暂不一致？
    │               │       │
    │               │       ├─ 是 ──→ 选择 AP（最终一致）
    │               │       │
    │               │       └─ 否 ──→ 混合策略
    │               │
    │               └─ 响应延迟敏感？
    │                       │
    │                       ├─ 是 ──→ AP + 补偿机制
    │                       │
    │                       └─ 否 ──→ CP + 缓存
```

## 总结

CAP 权衡不是非此即彼的选择题，而是一个需要精细化设计的工程问题：

**在技术层面**：可以通过读写策略调整、一致性级别降级、混合一致性架构等方式，实现灵活的一致性保障。

**在业务层面**：可以通过补偿机制、异步处理、业务规则约束等方式，在业务层面化解一致性问题。

**在架构层面**：可以通过微服务拆分，让不同的服务选择不同的 CP/AP 策略，各取所需。
