# 一致性模型

<Badge text="分布式系统" type="tip" /> <Badge text="一致性级别" type="warning" /> <Badge text="语义对比" type="info" />

你有没有遇到过这种情况：明明在手机上删了一封邮件，打开电脑却发现邮件还在；刚改了个头像，朋友圈里好友看到的还是旧头像；下单时显示有库存，付款时却提示库存不足。

这些场景的本质，都是**一致性**的问题。不同业务场景对一致性的要求不同，有的需要强一致，有的只需要最终一致。理解各种一致性模型的语义，是设计分布式系统的基础。

## 一致性模型是什么？

一致性模型是**系统对数据更新后，观察者能看到什么数据的一种承诺**。它定义了「什么时候读取能保证看到最新的数据」。

```
一致性模型本质上是这样的契约：
"如果你遵守系统规定的规则（比如在写入后等待一段时间），
 我保证你能看到最新写入的数据。"
```

不同的模型提供不同的保证，代价也不同。强一致性提供最强的保证，但性能开销大；弱一致性性能好，但保证弱。

## 一致性模型谱系

一致性模型不是非此即彼的选择，而是一个从强到弱的连续谱系：

```
强 ──────────────────────────────────────────> 弱
线性一致性 > 顺序一致性 > 因果一致性 > 
会话一致性 > PRAM 一致性 > 最终一致性
```

## 线性一致性（Linearizability）

线性一致性是最强的一致性模型。**在一个线性一致性的系统中，任何操作都能在某个时间点上看到之前所有操作的结果。**

### 形式化定义

线性一致性包含两个关键性质：

1. **原子性（Atomicity）**：每个操作都是瞬间完成的，没有重叠
2. **实时性（Real-time Ordering）**：如果操作 A 在操作 B 开始之前完成，那么 A 必须在 B 之前可见

```java
// 线性一致性示例
// 假设有两个进程 P1 和 P2，同时操作共享变量 x
// 时间线：
// T1: P1 写入 x = 1
// T2: P2 读取 x，得到 1（此时 P2 知道 T1 已经完成）
// T3: P1 写入 x = 2
// T4: P2 再次读取 x，得到 2
// 线性一致性保证：
// 如果 T4 的读返回了 2，那么 T4 之前的任何读都不能返回旧值
```

### 线性一致性的实现代价

线性一致性是最强的保证，但实现代价也最高：

```java
public class LinearizabilityExample {
    
    // 线性一致性存储实现
    // 通常需要使用共识算法（如 Paxos、Raft）
    // 或者分布式锁
    
    private final List<Node> nodes;
    private final QuorumStrategy quorum;
    
    public void write(String key, String value) {
        // 写入所有节点（同步）
        for (Node node : nodes) {
            node.write(key, value);
        }
    }
    
    public String read(String key) {
        // 从多数派节点读取
        List<String> values = quorum.readFromQuorum(key);
        return values.get(0);  // 假设所有节点返回相同值
    }
}
```

## 顺序一致性（Sequential Consistency）

顺序一致性比线性一致性稍弱，它**保证所有节点看到的操作顺序是一致的，但不保证这个顺序与实际时间一致**。

### 与线性一致性的区别

```java
// 顺序一致性 vs 线性一致性
// 场景：
// P1: write(x, 1)
// P2: write(x, 2)
// P3: read(x) -> 2
// P4: read(x) -> 2
// 顺序一致性：允许
// 解释：P3 和 P4 可能观察到 P1 先执行，但 P2 后执行
// 即便实际上 P1 在 P2 之前完成
// 线性一致性：不允许
// 解释：因为 P1 在 P2 开始之前完成（P1 写入了 1）
// 所以所有后续的读必须看到 x=2
```

## 因果一致性（Causal Consistency）

因果一致性只保证**因果相关的操作顺序，不同节点对无因果关系的操作顺序可以不同**。

```java
// 因果相关示例
// P1: write(x, 1)           // 因果链开始
// P2: read(x) -> 1
// P3: write(x, 2)           // 依赖于 P2 的读操作
// 在这个例子中：
// - P1 -> P2 -> P3 是因果链
// - 正确的因果顺序必须保持
```

### 因果一致性的实现

```java
public class CausalConsistencyExample {
    
    static class VectorClock {
        Map<NodeId, Long> clock = new HashMap<>();
        
        public void increment(NodeId node) {
            clock.merge(node, 1L, Long::sum);
        }
        
        public void merge(VectorClock other) {
            for (Map.Entry<NodeId, Long> entry : other.clock.entrySet()) {
                clock.merge(entry.getKey(), entry.getValue(), Math::max);
            }
        }
        
        public boolean happensBefore(VectorClock other) {
            boolean allLessOrEqual = true;
            boolean someLess = false;
            
            for (Map.Entry<NodeId, Long> entry : other.clock.entrySet()) {
                Long mine = clock.get(entry.getKey());
                if (mine == null || mine < entry.getValue()) {
                    allLessOrEqual = false;
                }
                if (mine != null && mine < entry.getValue()) {
                    someLess = true;
                }
            }
            
            return allLessOrEqual && someLess;
        }
    }
}
```

## 会话一致性（Session Consistency）

会话一致性保证**在同一个会话内，读操作能看到自己之前写操作的结果**。

```java
// 会话一致性示例
Session s = sessionManager.createSession(userId);
// 1. 用户更新购物车
s.execute("UPDATE cart SET items = ? WHERE user_id = ?", items, userId);
// 2. 立即刷新页面查看购物车
// 会话一致性保证：用户能看到自己刚才添加的商品
Cart cart = s.query("SELECT * FROM cart WHERE user_id = ?", userId);
```

## 最终一致性（Eventual Consistency）

最终一致性是弱一致性模型的典型代表。**它只保证：如果没有新的更新操作，所有节点最终会返回相同的值。**

```java
// 最终一致性示例
Node beijing = nodes.get("beijing");
beijing.write("user:123:nickname", "新昵称");

Node shanghai = nodes.get("shanghai");
String nickname = shanghai.read("user:123:nickname");
// 可能读到「旧昵称」，因为数据还在同步中
// 但只要等待足够长的时间，最终会读到「新昵称」
```

## 一致性模型对比

| 模型 | 保证强度 | 性能 | 实现复杂度 | 适用场景 |
|---|---|---|---|---|
| 线性一致性 | 最强 | 最低 | 最高 | 分布式锁、分布式事务 |
| 顺序一致性 | 强 | 低 | 高 | 多核 CPU、分布式数据库 |
| 因果一致性 | 中强 | 中 | 中高 | 社交网络、评论系统 |
| 会话一致性 | 中 | 中 | 中 | Web 应用、移动端 |
| 最终一致性 | 弱 | 高 | 低 | 缓存、日志、社交 |

## 总结

一致性模型是分布式系统设计的核心考量。选择何种一致性模型，取决于业务需求、性能要求和实现成本。
