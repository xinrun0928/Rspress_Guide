# ACID vs BASE

<Badge text="分布式系统" type="tip" /> <Badge text="理论对比" type="warning" /> <Badge text="设计哲学" type="info" />

1998 年，Jim Gray 在《Transaction Processing: Concepts and Techniques》一书中写道：「事务是计算的基本单位，它将世界从连续的时间流中切分出来，使得每一个时刻只有一个计算在发生。」

二十年后，这句话仍然振聋发聩。ACID 和 BASE 两种事务处理范式，本质上是对「分布式环境下如何保证数据正确性」这一根本问题的不同回答。

## 两种哲学的起源

### ACID 的诞生

ACID 理论可以追溯到 1970 年代。那时候，关系型数据库刚刚兴起，Jim Gray 和他的同事们在研究数据库事务处理时，提出了 ACID 的概念。

ACID 是**单机数据库时代的产物**，它的设计目标是：**在单机环境下，提供一种让程序员可以无视并发和故障的编程模型。**

```sql
-- ACID 哲学：程序员可以像操作单机数据库一样操作事务
BEGIN TRANSACTION;
UPDATE account SET balance = balance - 100 WHERE id = 'A';
UPDATE account SET balance = balance + 100 WHERE id = 'B';
COMMIT;
```

### BASE 的诞生

BASE 理论诞生于 2000 年代初的互联网时代。那时候，Amazon、eBay 等公司面临的数据规模和访问压力是传统数据库无法承受的。

工程师们发现：**在互联网规模下，强一致性是奢侈品。** 与其追求完美的正确性，不如在「基本可用」的前提下，通过其他手段保证系统的可靠性。

```
BASE 哲学：承认不完美，用工程手段弥补
- 系统可能会返回过期数据（软状态）
- 但只要给足够的时间，最终会恢复正确（最终一致）
```

## 核心概念对比

### A - Atomicity vs Basically Available

**ACID 的原子性**：一个事务中的所有操作，要么全部成功，要么全部失败。

**BASE 的基本可用**：系统允许在某些情况下牺牲部分功能，保证核心功能可用。

```java
// ACID 原子性
public void transfer(Account from, Account to, BigDecimal amount) {
    transactionManager.begin();
    try {
        from.withdraw(amount);
        to.deposit(amount);
        transactionManager.commit();
    } catch (Exception e) {
        transactionManager.rollback();
        throw e;
    }
}

// BASE 基本可用
public void handleRequest(Request request) {
    try {
        processCore(request);  // 核心功能必须成功
    } catch (Exception e) {
        return ErrorResponse.CORE_FAILED;
    }
    
    try {
        processSideEffect(request);  // 辅助功能可以降级
    } catch (Exception e) {
        log.warn("辅助处理失败，降级处理", e);
    }
}
```

### C - Consistency vs Soft State

**ACID 的一致性**：事务执行前后，数据库必须始终处于一致状态。

**BASE 的软状态**：系统的状态可以随时间变化，即使没有新的输入。

```java
// BASE 软状态示例
public class AccountSnapshot {
    private String nickname;        // 可能和数据库不一致
    private BigDecimal balance;     // 可能是旧值
    private long lastSyncTime;      // 上次同步时间
    
    public boolean isStale() {
        return System.currentTimeMillis() - lastSyncTime > 30 * 1000;
    }
}
```

### I - Isolation vs Eventually Consistent

**ACID 的隔离性**：并发执行的事务相互隔离，互不干扰。

**BASE 的最终一致性**：系统会达到一致状态，但过程是异步的。

### D - Durability

**ACID 的持久性**：事务提交后，结果必须永久保存。

## 适用场景对比

### ACID 的主战场

ACID 适合**对数据一致性要求极高、不允许任何错误**的场景：

| 场景 | 原因 |
|---|---|
| 银行转账 | 钱不能多也不能少 |
| 库存管理 | 超卖会导致严重损失 |
| 订单处理 | 订单状态必须准确可靠 |

### BASE 的主战场

BASE 适合**可以容忍短暂不一致、追求高可用和性能的互联网场景**：

| 场景 | 原因 |
|---|---|
| 社交媒体发布 | 短暂的不一致用户感知不强 |
| 视频播放量统计 | 允许一定误差 |
| 推荐系统 | 实时性比精确性更重要 |

## 混合使用：Saga 模式

在实际大型系统中，ACID 和 BASE 往往混合使用。Saga 模式就是这种混合的典型代表。

```java
public class SagaOrchestrator {
    
    public Saga createOrderSaga(Order order) {
        return Saga.builder("create-order")
            .step("create-order")
                .action(() -> orderService.create(order))
                .compensation(() -> orderService.cancel(order.getId()))
            .step("reserve-inventory")
                .action(() -> inventoryService.reserve(order.getItems()))
                .compensation(() -> inventoryService.release(order.getItems()))
            .step("payment")
                .action(() -> paymentService.pay(order))
                .compensation(() -> paymentService.refund(order.getId()))
            .build();
    }
}
```

## 总结

ACID 和 BASE 代表了两种不同的设计哲学：

- **ACID** 是「完美主义」：追求强一致性
- **BASE** 是「实用主义」：承认分布式环境的不确定性

没有绝对的好坏之分，只有「是否适合当前场景」的选择。
