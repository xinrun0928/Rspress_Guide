# 2PC vs 3PC

<Badge text="分布式系统" type="tip" /> <Badge text="对比总结" type="warning" /> <Badge text="协议选型" type="info" />

经过前两章的学习，你已经了解了 2PC 和 3PC 的原理和实现。这一章，我们来做一个全面的对比，帮助你在实际项目中做出正确的选择。

## 核心对比

```java
/**
 * 2PC vs 3PC 核心对比
 */

// 阶段对比
// 2PC：投票阶段 + 提交阶段
// 3PC：询问阶段 + 预备提交阶段 + 提交阶段

// 阻塞对比
// 2PC：参与者投票后完全阻塞，等待协调者指令
// 3PC：部分改善，在某些情况下可以超时自主决策

// 协调者崩溃
// 2PC：参与者永久等待
// 3PC：参与者可以尝试自主决策

// 网络分区
// 2PC：可能导致部分参与者提交、部分回滚
// 3PC：可能导致更严重的不一致
```

## 详细对比表格

| 维度 | 2PC | 3PC |
|---|---|---|
| 阶段数 | 2 | 3 |
| 协调者到参与者的通信次数 | 2 | 3 |
| 参与者阻塞时间 | 长（整个提交过程） | 短（只在 PreCommit 阶段） |
| 协调者单点故障影响 | 严重 | 可恢复 |
| 网络分区风险 | 中等 | 较高 |
| 实现复杂度 | 较低 | 较高 |
| 性能开销 | 较低 | 较高 |
| 实际应用 | 广泛（XA、Seata） | 较少 |

## 流程对比

```java
/**
 * 2PC 流程
 */

// 阶段一（投票）
// C → P: CanCommit?
// P → C: Yes/No

// 阶段二（提交/回滚）
// 如果全部 Yes：
//   C → P: DoCommit
//   P → C: Ack
// 如果有 No：
//   C → P: DoRollback
//   P → C: Ack

/**
 * 3PC 流程
 */

// 阶段一（询问）
// C → P: CanCommit?
// P → C: Yes/No

// 阶段二（预备提交）
// 如果全部 Yes：
//   C → P: PreCommit
//   P → C: Ack
// 如果有 No：
//   C → P: DoAbort
//   P → C: Ack

// 阶段三（提交/回滚）
// C → P: DoCommit/DoAbort
// P → C: Ack
```

## 阻塞问题对比

```java
/**
 * 2PC 的阻塞
 */

// 场景：协调者在发送 DoCommit 前崩溃
// 
// Coordinator         Participant
//     |                    |
//     |<-- Yes ------------|
//     |                    |
//     X [崩溃]             |
//                         v
//                    永远等待...
//                    持有锁，无法释放

/**
 * 3PC 的阻塞改进
 */

// 场景：协调者在发送 DoCommit 前崩溃
// 
// Coordinator         Participant
//     |                    |
//     |<-- Yes ------------|
//     |                    |
//     |--- PreCommit ----->|
//     |<-- Ack ------------|
//     |                    |
//     X [崩溃]             |
//                         v
//                    超时后询问其他参与者
//                    得知已 PreCommit，安全提交

/**
 * 3PC 仍然阻塞的场景
 */

// 场景：网络分区发生在 PreCommit 阶段
// 
// P1 + C              P2 + P3
//   |                   |
//   |<-- PreCommit -----|  [消息到达 P1]
//   |--- Ack ----------->|
//   |                   |   [网络分区]
//   |                   X
//   |                   |
//   |<-- DoCommit -------|  [P1 收到]
//   |<-- DoAbort --------|  [P2, P3 收到]
//   |                   |
//   X                   X
//   [P1 永久等待]      [P2, P3 永久等待]
```

## 超时机制对比

```java
/**
 * 2PC 的超时处理
 */

// 参与者超时（等待协调者）
// - 在投票后不知道该提交还是回滚
// - 只能保守地等待或告警
// - 依赖协调者恢复后重新发送指令

/**
 * 3PC 的超时处理
 */

public class TimeoutHandling {
    
    /**
     * 参与者超时状态机
     */
    public enum ParticipantState {
        INIT,
        VOTING,       // 2PC：等待 CanCommit 响应
        PREPARED,     // 2PC：投票后等待指令
        PRE_COMMIT,   // 3PC：收到 PreCommit 等待 DoCommit
        ABORT,
        COMMIT
    }
    
    /**
     * 2PC 超时处理
     */
    public void on2PCTimeout(ParticipantState state) {
        switch (state) {
            case VOTING:
                // 还没投票，可以安全回滚
                rollback();
                break;
            case PREPARED:
                // 已投票，不知道该提交还是回滚
                // 只能等待，持有锁
                // 这是 2PC 的主要问题
                log.warn("Prepared but waiting for coordinator");
                break;
        }
    }
    
    /**
     * 3PC 超时处理
     */
    public void on3PCTimeout(ParticipantState state) {
        switch (state) {
            case VOTING:
                // 还没投票，可以安全回滚
                rollback();
                break;
            case PRE_COMMIT:
                // 已收到 PreCommit，说明协调者决定提交
                // 可以安全提交
                commit();
                break;
        }
    }
}
```

## 实际应用选择

```java
/**
 * 为什么业界更多使用 2PC？
 */

// 1. 协议成熟稳定
// - XA 规范基于 2PC
// - 数据库厂商长期支持
// - 工具链完善

// 2. 补偿机制弥补不足
// - 即使 2PC 失败，可以通过补偿解决
// - TCC、Saga 等模式处理复杂场景
// - 补偿机制更灵活

// 3. 3PC 的改进不够显著
// - 虽然减少了部分阻塞
// - 但增加了复杂度和网络开销
// - 在网络分区时仍有问题

// 4. 实践中的网络分区处理
// - 网络分区是小概率事件
// - 为了小概率事件增加复杂度不划算
// - 可以通过其他机制（如重试）处理

/**
 * 何时选择 3PC？
 */

// 1. 参与者可能长时间不可用
// - 2PC 会导致长时间阻塞
// - 3PC 可以让参与者在某些情况下自主决策

// 2. 对延迟不敏感
// - 3PC 多一轮通信
// - 需要额外 10-30ms 的延迟

// 3. 愿意投入更多开发资源
// - 3PC 状态机复杂
// - 需要仔细实现超时逻辑
```

## 总结建议

```java
/**
 * 选择建议
 */

// 1. 大多数场景选择 2PC
// - XA 事务（Java EE）
// - Seata AT 模式
// - 任何对性能有要求的场景

// 2. 需要处理长时间阻塞时考虑 3PC
// - 跨数据中心事务
// - 参与者可能长时间不可达
// - 愿意投入额外复杂度

// 3. 实际项目中
// - 使用成熟的分布式事务框架
// - 不要自己实现 2PC/3PC
// - 关注业务层面的补偿机制

/**
 * 最佳实践
 */

// 1. 使用成熟框架
// - Seata
// - Atomikos
// - Narayana

// 2. 设计补偿机制
// - 即使事务协议失败，也能通过补偿恢复
// - 业务幂等性设计

// 3. 监控和告警
// - 监控事务执行时间
// - 监控超时和失败
// - 及时处理 in-doubt 事务
```

## 两种协议的本质区别

```java
/**
 * 2PC 的哲学
 * 
 * 简单、实用、牺牲部分可用性换取一致性
 */

// 假设失败是小概率事件
// 一旦发生，依赖人工干预或自动补偿
// 协议本身保持简单

/**
 * 3PC 的哲学
 * 
 * 复杂、更高的可用性、但一致性保证更弱
 */

// 尝试让参与者在协调者失败时自主决策
// 但在异步网络中，这个目标实际上无法完美实现
// 增加了复杂度，却没有完全解决问题

/**
 * 最终结论
 * 
 * 没有任何协议能在异步网络中同时保证：
 * - 强一致性
 * - 高可用性
 * - 分区容错性
 * 
 * 这是 FLP 不可能性原理告诉我们的
 * 2PC 和 3PC 都是在这个约束下的工程权衡
 */
```

## 总结

2PC 和 3PC 是分布式事务的两种基础协议：

1. **2PC**：简单、成熟、广泛使用，但存在阻塞问题
2. **3PC**：改进了阻塞，但增加了复杂度和网络开销，实际应用较少

在实际项目中，推荐使用成熟的分布式事务框架（如 Seata），而不是自己实现这些协议。
