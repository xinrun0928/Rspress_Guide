# 时钟与事件顺序

<Badge text="分布式系统" type="tip" /> <Badge text="时间同步" type="warning" /> <Badge text="事件排序" type="info" />

你有没有想过这个问题：**在没有全局时钟的分布式系统中，如何判断两个事件谁先发生？**

单机系统中这不是问题——操作系统提供了全局时钟，每个事件都有明确的时间戳。但在分布式系统中，每个节点有自己的本地时钟，而本地时钟可能会漂移、漂移、甚至被人为调整。

如何解决「谁先谁后」这个看似简单却至关重要的问题？这就引出了分布式系统中的一个核心概念：**时钟算法**。

## 分布式系统中的时钟问题

```java
/**
 * 分布式时钟的挑战
 */

// 场景：三个节点通过网络通信
Node A, B, C;

// 事件序列：
// 1. A 发送消息 M1 给 B
// 2. B 收到 M1，发送消息 M2 给 C
// 3. C 收到 M2

// 问题：
// - A 的时钟可能比 B 快 10ms
// - B 的时钟可能比 C 快 5ms
// - 我们无法通过物理时钟判断事件顺序
```

## Lamport 时钟

Leslie Lamport 在 1978 年提出了一个简单的解决方案：

> **如果事件 A 导致事件 B 发生，那么 A 必然在 B 之前。**

这就是「happens-before」关系。

```java
/**
 * Lamport 时钟
 * 
 * 规则：
 * 1. 每次处理本地事件，计数器 + 1
 * 2. 发送消息时，携带当前计数器
 * 3. 收到消息时，计数器 = max(本地计数器, 消息计数器) + 1
 */
public class LamportClock {
    
    private AtomicLong counter = new AtomicLong(0);
    private int nodeId;
    
    /**
     * 处理本地事件
     */
    public void localEvent() {
        counter.incrementAndGet();
    }
    
    /**
     * 发送消息
     */
    public long sendMessage() {
        long ts = counter.incrementAndGet();
        return ts;  // 将时间戳附加到消息上
    }
    
    /**
     * 接收消息
     */
    public void receiveMessage(long receivedTs) {
        // 更新为 max(本地计数器, 消息计数器) + 1
        counter.updateAndGet(current -> Math.max(current, receivedTs) + 1);
    }
    
    /**
     * 获取当前时间戳
     */
    public long getTime() {
        return counter.get();
    }
}

/**
 * Lamport 时钟的局限性
 * 
 * 如果 C(a) < C(b)，能说明 a happens-before b 吗？
 * 
 * 答案是：不能！
 * 
 * C(a) < C(b) 只能说明：
 * - 要么 a happens-before b
 * - 要么 a 和 b 是并发的
 * 
 * Lamport 时钟不能判断两个事件是否并发
 */
```

## 向量时钟

为了解决并发事件的判断问题，向量时钟（Vector Clock）应运而生。

```java
/**
 * 向量时钟
 * 
 * 每个节点维护一个向量
 * 向量[i] 表示节点 i 看到的最大事件数
 */
public class VectorClock {
    
    private Map<NodeId, Long> clock = new HashMap<>();
    private int nodeId;
    
    public VectorClock(int nodeId, Set<NodeId> allNodes) {
        this.nodeId = nodeId;
        for (NodeId id : allNodes) {
            clock.put(id, 0L);
        }
    }
    
    /**
     * 处理本地事件
     */
    public void localEvent() {
        clock.merge(nodeId, 1L, Long::sum);
    }
    
    /**
     * 发送消息
     */
    public VectorClock sendMessage() {
        clock.merge(nodeId, 1L, Long::sum);
        return this.copy();  // 复制并发送
    }
    
    /**
     * 接收消息
     */
    public void receiveMessage(VectorClock receivedClock) {
        // 合并：取每个维度最大值
        for (Map.Entry<NodeId, Long> entry : receivedClock.clock.entrySet()) {
            clock.merge(entry.getKey(), entry.getValue(), Math::max);
        }
    }
    
    /**
     * 判断 happens-before 关系
     */
    public boolean happensBefore(VectorClock other) {
        boolean dominated = true;  // 全部 <=
        boolean strictlyLess = false;  // 至少一个 <
        
        for (NodeId id : clock.keySet()) {
            long thisValue = clock.getOrDefault(id, 0L);
            long otherValue = other.clock.getOrDefault(id, 0L);
            
            if (thisValue > otherValue) {
                dominated = false;
            }
            if (thisValue < otherValue) {
                strictlyLess = true;
            }
        }
        
        return dominated && strictlyLess;
    }
    
    /**
     * 判断是否并发
     */
    public boolean isConcurrent(VectorClock other) {
        // 不满足 happensBefore 且 other 不满足 happensBefore
        return !this.happensBefore(other) && !other.happensBefore(this);
    }
    
    private VectorClock copy() {
        VectorClock copy = new VectorClock(nodeId, clock.keySet());
        copy.clock.putAll(this.clock);
        return copy;
    }
}

/**
 * 向量时钟示例
 */

// 初始状态：
// A: [A=0, B=0, C=0]
// B: [A=0, B=0, C=0]
// C: [A=0, B=0, C=0]

// 1. A 发送消息
// A: [A=1, B=0, C=0] -> 发送给 B

// 2. B 收到消息
// B: [A=1, B=1, C=0]

// 3. B 发送消息
// B: [A=1, B=2, C=0] -> 发送给 C

// 4. C 收到消息
// C: [A=1, B=2, C=1]

// 判断：
// A 的状态 [1,0,0] vs B 的状态 [1,2,0]
// B 包含 A 的所有维度且至少一个维度更大
// 所以 A happens-before B

// A 的状态 [1,0,0] vs C 的状态 [1,2,1]
// C 包含 A 的所有维度且至少一个维度更大
// 所以 A happens-before C
```

## 版本向量（Version Vectors）

向量时钟的变种，在 DynamoDB、Cassandra 等系统中广泛使用：

```java
/**
 * 版本向量
 * 
 * 与向量时钟的区别：
 * - 版本向量通常存储在数据库的每条记录中
 * - 用于检测并发更新和冲突
 */
public class VersionVector {
    
    private Map<NodeId, Long> versions = new HashMap<>();
    
    /**
     * 更新版本
     */
    public void update(NodeId nodeId) {
        versions.merge(nodeId, 1L, Long::sum);
    }
    
    /**
     * 合并版本
     */
    public void merge(VersionVector other) {
        for (Map.Entry<NodeId, Long> entry : other.versions.entrySet()) {
            versions.merge(entry.getKey(), entry.getValue(), Math::max);
        }
    }
    
    /**
     * 判断是否包含另一个版本
     */
    public boolean contains(VersionVector other) {
        for (Map.Entry<NodeId, Long> entry : other.versions.entrySet()) {
            long thisValue = this.versions.getOrDefault(entry.getKey(), 0L);
            if (thisValue < entry.getValue()) {
                return false;
            }
        }
        return true;
    }
}

/**
 * DynamoDB 的数据版本控制
 */
public class DynamoDBVersion {
    
    /**
     * DynamoDB 使用向量时钟检测冲突
     */
    VersionVector version;
    Object value;
    
    /**
     * 写入数据
     */
    public void put(Object newValue, VersionVector newVersion) {
        if (this.version.contains(newVersion)) {
            // 现有版本包含新版本，数据已过时
            throw new StaleVersionException();
        }
        
        if (newVersion.contains(this.version)) {
            // 新版本包含现有版本，直接更新
            this.value = newValue;
            this.version = newVersion;
        } else {
            // 版本不可比较，存在冲突
            // 需要冲突解决
            Object resolved = resolveConflict(this.value, newValue);
            this.value = resolved;
            this.version.merge(newVersion);
        }
    }
    
    private Object resolveConflict(Object v1, Object v2) {
        // Last-Write-Wins
        return timestamp(v1) > timestamp(v2) ? v1 : v2;
    }
}
```

## TrueTime（Google Spanner）

Google Spanner 使用硬件时钟实现更强的一致性保证：

```java
/**
 * TrueTime API
 * 
 * Google Spanner 使用 GPS 和原子钟实现 TrueTime
 * TrueTime 返回一个时间区间 [earliest, latest]
 * 实际时间一定在这个区间内
 */
public class TrueTime {
    
    /**
     * TrueTime 返回值
     */
    public static class TTInterval {
        private final long earliest;
        private final long latest;
        
        public TTInterval(long earliest, long latest) {
            this.earliest = earliest;
            this.latest = latest;
        }
        
        public long earliest() { return earliest; }
        public long latest() { return latest; }
        
        /**
         * 判断两个区间是否有交集
         */
        public boolean overlaps(TTInterval other) {
            return this.earliest <= other.latest 
                && other.earliest <= this.latest;
        }
    }
    
    /**
     * 获取当前时间区间
     */
    public TTInterval now() {
        // 返回当前时间的一个小范围区间
        // 例如：当前时间在 [t-4ms, t+4ms] 之间
        long now = hardwareClock.now();
        return new TTInterval(now - ERROR_BOUND, now + ERROR_BOUND);
    }
}

/**
 * Spanner 的提交协议
 */
public class SpannerCommit {
    
    /**
     * 两阶段提交的 Spanner 版本
     */
    public void twoPhaseCommit(Transaction tx) {
        // 1. 准备阶段
        // 参与者锁定数据
        
        // 2. 提交阶段
        // 获取 TrueTime
        TTInterval now = tt.now();
        
        // 等待直到 latest <= now.latest
        // 这样可以保证所有节点的提交时间都不重叠
        tt.waitUntil(now.latest);
        
        // 提交事务
        tx.commit(now.latest);
    }
    
    /**
     * Spanner 保证提交时间不会重叠
     * 
     * 事务 T1 在时间区间 [e1, l1] 提交
     * 事务 T2 在时间区间 [e2, l2] 提交
     * 
     * 如果 l1 < e2，那么 T1 在 T2 之前提交
     */
}
```

## Hybrid Logical Clock（HLC）

结合物理时钟和逻辑时钟的优点：

```java
/**
 * Hybrid Logical Clock (HLC)
 * 
 * 结合：
 * - 物理时钟的直观性
 * - 逻辑时钟的因果关系保证
 */
public class HybridLogicalClock {
    
    private long physical;   // 物理时钟
    private long counter;    // 逻辑计数器
    private int nodeId;
    
    /**
     * 处理本地事件
     */
    public HLCValue localEvent() {
        long pt = physicalClock.now();
        
        if (pt > physical) {
            // 物理时钟前进了
            physical = pt;
            counter = 0;
        } else {
            // 物理时钟没有前进，使用逻辑计数器
            counter++;
        }
        
        return new HLCValue(physical, counter, nodeId);
    }
    
    /**
     * 接收消息
     */
    public HLCValue receiveMessage(HLCValue received) {
        long pt = physicalClock.now();
        long maxPhysical = Math.max(pt, Math.max(this.physical, received.physical));
        
        if (maxPhysical == this.physical && maxPhysical == received.physical) {
            // 物理时钟相同，使用逻辑计数器
            counter = Math.max(this.counter, received.counter) + 1;
        } else if (maxPhysical == this.physical) {
            counter++;
        } else if (maxPhysical == received.physical) {
            counter = 0;
        } else {
            counter = 0;
        }
        
        physical = maxPhysical;
        return new HLCValue(physical, counter, nodeId);
    }
    
    /**
     * HLC 比较
     * 首先比较物理时钟，然后比较逻辑计数器
     */
    public int compare(HLCValue a, HLCValue b) {
        if (a.physical != b.physical) {
            return Long.compare(a.physical, b.physical);
        }
        if (a.counter != b.counter) {
            return Long.compare(a.counter, b.counter);
        }
        return Integer.compare(a.nodeId, b.nodeId);
    }
}

/**
 * HLC 的优势
 * 
 * 1. 保持与物理时钟的关系
 *    - 可以直接用于需要时间戳的场景
 *    - 便于调试和监控
 * 
 * 2. 保持因果关系
 *    - 如果 a happens-before b，那么 HLC(a) < HLC(b)
 * 
 * 3. 有界增长
 *    - counter 不会无限增长
 *    - 如果物理时钟前进，counter 会重置
 */
```

## 时钟算法的对比

| 算法 | 表达能力 | 存储开销 | 适用场景 |
|---|---|---|---|
| Lamport 时钟 | 仅能判断偏序 | O(1) | 简单因果追踪 |
| 向量时钟 | 可判断并发 | O(N) | DynamoDB、Cassandra |
| 版本向量 | 变种向量时钟 | O(N) | 数据版本追踪 |
| TrueTime | 物理时间边界 | O(1) | Spanner |
| HLC | 混合 | O(1) | 分布式数据库 |

## 总结

时钟算法是分布式系统中判断事件顺序的基础工具：

1. **Lamport 时钟**：简单有效，但无法判断并发
2. **向量时钟**：完整表达因果关系，但存储开销大
3. **TrueTime**：通过硬件时钟实现更强的一致性
4. **HLC**：结合物理和逻辑时钟的优点

选择哪种时钟算法，取决于系统的具体需求和约束。
