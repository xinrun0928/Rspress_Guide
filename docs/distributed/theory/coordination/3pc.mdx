# 3PC

<Badge text="分布式系统" type="tip" /> <Badge text="分布式事务" type="warning" /> <Badge text="三阶段提交" type="info" />

2PC 有一个致命的问题：**阻塞**。参与者在投票后必须等待协调者的最终指令，如果协调者崩溃，参与者可能永远等待。

**三阶段提交（3PC）**就是为了解决这个问题——它将 2PC 的投票阶段拆分为两个阶段，并引入了超时机制，让参与者在某些情况下可以自主做出决策。

## 3PC 的核心思想

```java
/**
 * 3PC vs 2PC 的区别
 */

// 2PC：
// - 阶段一：投票（CanCommit?）
// - 阶段二：提交/回滚（DoCommit / DoRollback）

// 3PC：
// - 阶段一：询问（CanCommit?）
// - 阶段二：预备提交（PreCommit）
// - 阶段三：提交（DoCommit）

/**
 * 3PC 的关键改进
 */

// 1. 引入超时机制
// - 参与者可以在等待时超时
// - 超时后自主做出决策

// 2. 拆分了投票阶段
// - 第一个阶段只询问，不锁定资源
// - 第二个阶段才锁定资源并预备提交

// 3. 减少了阻塞
// - 参与者知道协调者的最终决定
// - 在某些情况下可以单方面行动
```

## 3PC 的详细流程

```java
/**
 * 3PC 详细流程
 */

// 阶段一：CanCommit（询问阶段）
// 
// Coordinator                    Participant1  Participant2  Participant3
//     |                               |            |            |
//     |--- CanCommit? --------------> |            |            |
//     |<-- Yes ---------------------- |            |            |
//     |--- CanCommit? ----------------------------> |            |
//     |<-- Yes ---------------------------------- |            |
//     |--- CanCommit? ------------------------------------------------> |
//     |<-- Yes ------------------------------------------------------ |
//     |                               |            |            |
//     v                               |            |            |
//                                  不锁定资源，只检查能否提交          |

// 阶段二：PreCommit（预备提交阶段）
// 
// Coordinator                    Participant1  Participant2  Participant3
//     |                               |            |            |
//     |--- PreCommit -----------------> |            |            |
//     |<-- Ack ---------------------- |            |            |
//     |--- PreCommit ----------------------------> |            |
//     |<-- Ack ---------------------------------- |            |
//     |--- PreCommit ------------------------------------------------> |
//     |<-- Ack ------------------------------------------------------ |
//     |                               |            |            |
//     v                               |            |            |
//                                  锁定资源，预备提交               |

// 阶段三：DoCommit（提交阶段）
// 
// Coordinator                    Participant1  Participant2  Participant3
//     |                               |            |            |
//     |--- DoCommit -----------------> |            |            |
//     |<-- Ack ---------------------- |            |            |
//     |                               v            v            v
//     |                            提交          提交          提交
//     |                               |            |            |
//     v                               |            |            |
```

## 3PC 的超时处理

```java
/**
 * 3PC 的超时机制
 */

// 协调者的超时处理
public class ThreePhaseCommitCoordinator {
    
    /**
     * 在 PreCommit 阶段的超时
     * 
     * 如果在发送 PreCommit 后超时
     * 协调者应该继续重试发送 PreCommit
     */
    public void onPreCommitTimeout() {
        // 所有参与者都应该处于 PreCommit 或 Abort 状态
        // 继续发送 DoAbort，让参与者回滚
        for (Participant p : participants) {
            p.doAbort(transaction);
        }
    }
    
    /**
     * 在等待 PreCommit Ack 时超时
     */
    public void onPreCommitAckTimeout() {
        // 部分参与者可能没有收到 PreCommit
        // 需要重新发送
        // 如果重试多次后仍然超时，当作失败处理
        boolean resendSuccess = resendPreCommit();
        if (!resendSuccess) {
            // 放弃事务
            abort(transaction);
        }
    }
}

/**
 * 参与者的超时处理
 */
public class ThreePhaseCommitParticipant {
    
    private volatile CoordinatorState state = CoordinatorState.INIT;
    
    public enum CoordinatorState {
        INIT,           // 初始状态
        CAN_COMMIT_SENT, // 已发送 CanCommit
        PRE_COMMIT,     // 收到 PreCommit
        ABORT,          // 需要回滚
        COMMIT          // 可以提交
    }
    
    /**
     * 在等待 PreCommit 时超时
     * 
     * 此时已经投票 Yes，说明其他参与者也投票了
     * 协调者可能崩溃，应该询问其他参与者的状态
     */
    public void onPreCommitTimeout() {
        // 向其他参与者询问状态
        for (Participant other : participants) {
            try {
                CoordinatorState otherState = other.queryState();
                if (otherState == CoordinatorState.PRE_COMMIT) {
                    // 其他参与者已经收到 PreCommit
                    // 我们也可以进入 PreCommit 状态
                    enterPreCommit();
                    return;
                }
                if (otherState == CoordinatorState.ABORT) {
                    // 其他参与者已决定回滚
                    selfAbort();
                    return;
                }
            } catch (Exception e) {
                // 继续尝试其他参与者
            }
        }
        
        // 如果所有尝试都失败，自主决定回滚
        // 这是一个保守的选择
        selfAbort();
    }
    
    /**
     * 在等待 DoCommit 时超时
     * 
     * 既然已经收到了 DoCommit（或者在 PreCommit 状态）
     * 说明协调者已经决定提交
     * 可以安全提交
     */
    public void onDoCommitTimeout() {
        if (state == CoordinatorState.PRE_COMMIT 
            || state == CoordinatorState.COMMIT) {
            // 可以安全提交
            doCommit();
        }
    }
}
```

## 3PC 的代码实现

```java
/**
 * 3PC 协调者实现
 */
public class ThreePhaseCommitCoordinator {
    
    private List<Participant> participants;
    private Map<String, TransactionState> transactions = new ConcurrentHashMap<>();
    
    /**
     * 提交事务
     */
    public void commit(Transaction transaction) {
        // 阶段一：询问
        boolean allCanCommit = doCanCommitPhase(transaction);
        
        if (allCanCommit) {
            // 阶段二：预备提交
            doPreCommitPhase(transaction);
            // 阶段三：提交
            doCommitPhase(transaction);
        } else {
            // 回滚
            doAbortPhase(transaction);
        }
    }
    
    /**
     * 阶段一：CanCommit
     */
    private boolean doCanCommitPhase(Transaction transaction) {
        List<Future<Boolean>> futures = new ArrayList<>();
        
        for (Participant p : participants) {
            futures.add(executor.submit(() -> p.canCommit(transaction)));
        }
        
        boolean allYes = true;
        for (Future<Boolean> f : futures) {
            try {
                if (!f.get(TIMEOUT, TimeUnit.SECONDS)) {
                    allYes = false;
                    break;
                }
            } catch (Exception e) {
                allYes = false;
                break;
            }
        }
        
        return allYes;
    }
    
    /**
     * 阶段二：PreCommit
     */
    private void doPreCommitPhase(Transaction transaction) {
        for (Participant p : participants) {
            p.preCommit(transaction);
        }
        
        // 等待所有 Ack
        waitForAcks(participants);
    }
    
    /**
     * 阶段三：DoCommit
     */
    private void doCommitPhase(Transaction transaction) {
        for (Participant p : participants) {
            p.doCommit(transaction);
        }
    }
    
    /**
     * 中止事务
     */
    private void doAbortPhase(Transaction transaction) {
        for (Participant p : participants) {
            p.doAbort(transaction);
        }
    }
}

/**
 * 3PC 参与者实现
 */
public class ThreePhaseCommitParticipant {
    
    public boolean canCommit(Transaction transaction) {
        // 阶段一：只检查能否提交，不锁定资源
        try {
            // 检查业务规则
            if (!checkBusinessRules(transaction)) {
                return false;
            }
            
            // 检查数据一致性
            if (!checkDataConsistency(transaction)) {
                return false;
            }
            
            return true;
            
        } catch (Exception e) {
            log.error("Check failed", e);
            return false;
        }
    }
    
    public void preCommit(Transaction transaction) {
        // 阶段二：锁定资源，预备提交
        try {
            // 1. 获取锁
            acquireLocks(transaction);
            
            // 2. 执行预备提交（写 redo log）
            prepare(transaction);
            
            // 3. 发送 Ack
            coordinator.receiveAck(transaction);
            
        } catch (Exception e) {
            log.error("PreCommit failed", e);
            coordinator.receiveNoAck(transaction);
        }
    }
    
    public void doCommit(Transaction transaction) {
        // 阶段三：正式提交
        try {
            commit(transaction);
            releaseLocks(transaction);
        } catch (Exception e) {
            log.error("Commit failed", e);
            // 重试或告警
        }
    }
    
    public void doAbort(Transaction transaction) {
        // 回滚
        try {
            rollback(transaction);
            releaseLocks(transaction);
        } catch (Exception e) {
            log.error("Abort failed", e);
            // 重试或告警
        }
    }
}
```

## 3PC 的局限性

虽然 3PC 改进了 2PC 的阻塞问题，但它仍然不是完美的：

### 1. 网络分区问题

```java
/**
 * 3PC 的网络分区问题
 */

// 场景：协调者和部分参与者在网络分区的一侧
// 
// Coordinator + P1                    P2 + P3
//     |                                |
//     |--- PreCommit ----------------->P1
//     |<-- Ack ---------------------- P1
//     |                                |
//     |              [网络分区]         |
//     |                                |
//     |--- DoCommit ----------------->P1
//     |   [消息丢失]                   |
//     |                                |
//     X                                v
//                                    P1 等待 DoCommit...
//                                    P2, P3 等待 PreCommit...
//                                    永久等待！
// 
// 问题：在 PreCommit 阶段发生网络分区
// P1 已经收到 PreCommit 并 Ack
// P2, P3 没收到 PreCommit
// 协调者崩溃，P1 永远等待 DoCommit
```

### 2. 异步网络假设

```java
/**
 * 3PC 的假设问题
 */

// 3PC 假设消息传输是有界的（bounded）
// 但在真实的异步网络中，这个假设不成立

// 如果网络延迟无限大：
// 1. 协调者的超时触发
// 2. 参与者 A 的超时也触发
// 3. A 自主决定提交
// 4. 但协调者还在等待其他参与者
// 5. 协调者也可能自主决定提交
// 6. 导致不一致！

/**
 * 结论
 * 
 * 在异步网络中，任何事务提交协议都不可能同时保证：
 * - 安全性（Safety）：要么全提交，要么全回滚
 * - 活跃性（Liveness）：最终必须做出决定
 * 
 * 这就是 FLP 不可能性原理的体现
 */
```

## 2PC vs 3PC

| 维度 | 2PC | 3PC |
|---|---|---|
| 阶段数 | 2 | 3 |
| 阻塞 | 完全阻塞 | 部分改善 |
| 协调者单点 | 严重问题 | 有所改善 |
| 网络分区 | 可能导致不一致 | 更严重的分区问题 |
| 实现复杂度 | 较低 | 较高 |
| 实际应用 | XA、Seata | 很少使用 |

## 3PC 为什么不流行？

```java
/**
 * 为什么 3PC 没有被广泛使用？
 */

// 1. 协议复杂
// - 多一个阶段意味着更多的状态和交互
// - 开发和调试成本高

// 2. 不能完全解决一致性问题
// - 在网络分区时仍然可能出问题
// - 需要额外的机制来处理边缘情况

// 3. 性能开销
// - 多一次网络往返
// - 在高并发场景下性能影响明显

// 4. 工程实现困难
// - 超时机制的实现细节复杂
// - 状态机的正确性难以保证

/**
 * 实际选择
 * 
 * 大多数分布式事务系统选择 2PC + 补偿机制
 * 而不是 3PC
 * 
 * 因为：
 * - 2PC 的问题可以通过其他机制缓解（如超时补偿）
 * - 补偿机制更灵活，可以处理更复杂的业务场景
 * - 业界对 2PC 的理解和实现更成熟
 */
```

## 总结

3PC 是对 2PC 的一次重要改进：

1. **三阶段**：询问 → 预备提交 → 提交
2. **超时机制**：参与者在某些情况下可以自主决策
3. **减少阻塞**：在协调者崩溃时，参与者有机会继续

但 3PC 也有其局限性：

1. **不能完全解决一致性问题**
2. **在网络分区时可能更糟**
3. **实现复杂，实际应用不多**
