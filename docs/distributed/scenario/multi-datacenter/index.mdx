# 多机房部署：同城双活与异地多活

你有没有想过这个问题：

阿里双十一，流量那么大，一个机房肯定扛不住。那多个机房怎么协同？北京机房挂了，上海机房能接管吗？

**这就是多机房部署要解决的问题。**

## 多机房部署的四个层次

```
┌─────────────────────────────────────────────────────────┐
│               多机房部署演进路径                         │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  Level 1：冷备                                          │
│  └─ 主机房对外服务，备机房完全闲置                       │
│     问题：备机房资源浪费，切换时间长                     │
│                                                         │
│  Level 2：热备（同城灾备）                               │
│  └─ 同城两机房同时运行，主机房故障切换到备机房           │
│     问题：只能防机房级故障，不能防城市级灾难             │
│                                                         │
│  Level 3：同城双活                                      │
│  └─ 两机房同时对外服务，流量按比例分发                   │
│     问题：跨机房调用延迟，需要数据同步                   │
│                                                         │
│  Level 4：异地多活                                      │
│  └─ 多地机房同时服务，每个机房都独立                      │
│     问题：复杂度极高，数据一致性挑战大                   │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

## 同城双活 vs 异地多活

| 维度 | 同城双活 | 异地多活 |
|------|---------|---------|
| 距离 | 50-100km | 500km+ |
| 延迟 | 1-3ms | 10-50ms |
| 目的 | 容灾备份 | 性能 + 容灾 |
| 数据同步 | 同步双写 | 异步复制 |
| 复杂度 | 中 | 高 |
| 适用 | 金融、政务 | 互联网头部 |

## 核心挑战

**1. 数据同步延迟**
- 同城双活：网络延迟 1-3ms，可接受同步双写
- 异地多活：延迟 10-50ms，同步双写会严重影响性能

**2. 流量调度**
- 用户就近访问：DNS / Anycast 调度
- 故障切换：健康检查 → 自动切走流量

**3. 数据一致性**
- 强一致：分布式事务（2PC / TCC），性能损耗大
- 最终一致：异步复制，业务接受短暂不一致

## 核心公式

```
多机房部署 = 部署架构选型 + 数据同步方案 + 流量调度 + 故障切换

选型依据：
- 城市级容灾 → 同城双活（主备或双活）
- 全国用户访问 → 异地多活（按地域分流）
- 两地三中心 → 同城双活 + 异地灾备

数据同步：
- 同城：同步双写（RPO=0）
- 异地：异步复制（RPO>0，接受数据丢失窗口）
```
