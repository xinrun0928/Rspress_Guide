# 分布式批量任务：分片与并行

想象一下这个场景：

你需要处理 1000 万条数据迁移，从旧数据库迁移到新数据库。

单机处理需要 10 小时，但业务只给了你 2 小时的窗口期。

**怎么办？**

分布式批量任务。

## 分布式批处理的场景

- **数据迁移**：旧系统到新系统的数据迁移
- **对账**：日终交易对账，批量校验数据一致性
- **报表生成**：日、周、月报表的批量计算
- **数据清洗**：ETL 过程中的数据清洗和转换

这些场景的共同特点是：**数据量大，需要并行加速**。

## 分片任务的核心思想

```
┌─────────────────────────────────────────────────────────┐
│                   分片任务原理                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  原始任务：处理 1000 万条数据                            │
│                                                         │
│  拆分后：                                               │
│    分片 1: 处理 0-250 万条                              │
│    分片 2: 处理 250-500 万条                            │
│    分片 3: 处理 500-750 万条                            │
│    分片 4: 处理 750-1000 万条                           │
│                                                         │
│  4 个节点并行处理 → 时间缩短到 2.5 小时                  │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

## 分片策略对比

| 策略 | 原理 | 适用场景 | 风险 |
|------|------|---------|------|
| Hash 分片 | 按字段 hash 取模 | 数据均匀分布 | 扩缩容需迁移 |
| 范围分片 | 按 ID/时间范围 | 时序数据 | 可能热点 |
| 固定分片 | 任务列表轮询分配 | 任务数量稳定 | 不够灵活 |

## 主流框架

| 框架 | 特点 | 适用场景 |
|------|------|---------|
| ElasticJob | 当当网开源，功能完善 | 企业级批处理 |
| XXL-JOB | 轻量、调度中心分离 | 快速接入 |
| PowerJob | 支持 DAG 任务流 | 复杂任务编排 |

## 核心公式

```
分布式批处理 = 任务拆分（分片）+ 并行执行 + 结果聚合

ElasticJob 核心配置：
- shardingTotalCount：分片数（通常=节点数）
- shardingStrategy：分片策略（按地区/按ID范围/哈希）
- jobListener：分片监听器（收集结果）
- misfireStrategy：错过策略（重执行/忽略）

避坑指南：
- 分片数设置：通常为节点数倍数
- 失败重试：设置重试次数和间隔
- 结果聚合：各分片完成后汇总
- 任务幂等：防止分片重复执行
```
