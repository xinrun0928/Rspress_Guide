# 业务链路黑白盒监控：Metrics + Tracing + Logging

如何知道系统当前是否健康？如果不健康，问题出在哪里？是哪个接口慢？是哪个服务故障？

**可观测性（Observability）** 的三大支柱——Metrics（指标）、Tracing（链路追踪）、Logging（日志）——是分布式系统运维的核心能力。

## 可观测性三支柱概览

```
┌──────────────────────────────────────────────────────────┐
│                      可观测性三支柱                         │
│                                                          │
│   ┌─────────────┐   ┌─────────────┐   ┌─────────────┐    │
│   │  Metrics    │   │   Tracing   │   │   Logging   │    │
│   │   指标监控   │   │   链路追踪   │   │   日志聚合   │    │
│   ├─────────────┤   ├─────────────┤   ├─────────────┤    │
│   │  回答：多少？ │   │  回答：哪里？ │   │  回答：什么？ │    │
│   │  QPS/RT/错误 │   │  TraceId    │   │  错误堆栈   │    │
│   │  吞吐量/错误率│   │  调用链路   │   │  详细上下文  │    │
│   └─────────────┘   └─────────────┘   └─────────────┘    │
│                                                          │
│   Prometheus    Jaeger / SkyWalking     ELK              │
│   + Grafana                                      Loki + ES  │
└──────────────────────────────────────────────────────────┘
```

| 支柱 | 关注点 | 回答的问题 | 数据特征 | 存储 |
|---|---|---|---|---|
| Metrics | 聚合的数值指标 | "现在系统健康吗？" | 预聚合、高吞吐量 | 时序数据库 |
| Tracing | 请求级别的调用链 | "哪个环节慢了？" | 详细、低吞吐量 | Jaeger/SkyWalking |
| Logging | 离散的文本事件 | "为什么会出错？" | 海量、离散 | ES/Loki |

## 黑盒监控：Metrics（指标监控）

黑盒监控不问"为什么"，只问"现在怎么样"。

### 黄金指标（Google SRE）

| 指标 | 含义 | 告警阈值（示例） |
|---|---|---|
| **延迟（Latency）** | 请求响应时间 | P99 `>` 500ms |
| **流量（Traffic）** | QPS / TPS | QPS 突降 `>` 50% |
| **错误（Errors）** | 错误率 | 5xx 占比 `>` 1% |
| **饱和度（Saturation）** | CPU / 内存 / 连接数 | CPU `>` 80% |

### Micrometer + Prometheus 埋点

```java
// 方式一：注解自动埋点（Spring Boot Actuator + Micrometer）
@Configuration
public class MetricsConfig {
    
    @Bean
    public MeterRegistry meterRegistry() {
        return new PrometheusMeterRegistry(PrometheusConfig.DEFAULT);
    }
}

// 方式二：手动埋点
@Service
public class OrderService {
    
    private final MeterRegistry meterRegistry;
    private final Counter orderSuccessCounter;
    private final Counter orderFailCounter;
    private final Timer orderTimer;
    
    public OrderService(MeterRegistry meterRegistry) {
        this.meterRegistry = meterRegistry;
        
        // 订单成功计数器
        this.orderSuccessCounter = Counter.builder("order.created")
            .tag("status", "success")
            .description("订单创建成功数")
            .register(meterRegistry);
        
        // 订单失败计数器
        this.orderFailCounter = Counter.builder("order.created")
            .tag("status", "fail")
            .description("订单创建失败数")
            .register(meterRegistry);
        
        // 订单处理时间
        this.orderTimer = Timer.builder("order.process.time")
            .description("订单处理耗时")
            .publishPercentiles(0.5, 0.95, 0.99)  // 发布 P50/P95/P99
            .register(meterRegistry);
    }
    
    public void createOrder(Order order) {
        Timer.Sample sample = Timer.start(meterRegistry);
        try {
            doCreateOrder(order);
            orderSuccessCounter.increment();
        } catch (Exception e) {
            orderFailCounter.increment();
            throw e;
        } finally {
            sample.stop(orderTimer);
        }
    }
}
```

### Grafana 看板

```yaml
# Prometheus 告警规则
groups:
  - name: order-service-alerts
    rules:
      # QPS 告警
      - alert: OrderQPSLow
        expr: rate(order_created_total[5m]) < 10
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "订单 QPS 过低（当前 {{ $value }} /s）"
          description: "可能是服务异常或流量跌零"
      
      # 错误率告警
      - alert: OrderErrorRateHigh
        expr: |
          rate(order_created{status="fail"}[5m])
          / rate(order_created_total[5m]) > 0.01
        for: 1m
        labels:
          severity: critical
        annotations:
          summary: "订单错误率过高（{{ $value | humanizePercentage }}）"
```

## 白盒监控：Tracing（链路追踪）

白盒监控告诉你"为什么会慢"、"问题出在哪一步"。

### SkyWalking 埋点

SkyWalking 是国内最流行的 APM 工具，对 Java 生态支持非常好：

```java
// SkyWalking 自动埋点（几乎不需要手动代码）
// 只需要引入 agent，SkyWalking 会自动拦截：
// - Spring MVC / WebFlux 控制器
// - Feign / OkHttp / HttpClient
// - JDBC / Redis / MongoDB
// - Kafka / RocketMQ / RabbitMQ
// - 主流 RPC 框架（Dubbo/gRPC/Thrift）

// 手动埋点（自定义业务指标）
@Trace
public OrderDetail getOrderDetail(long orderId) {
    // @Trace 注解：这个方法会作为一个新的 Span
    ActiveSpan.tag("orderId", String.valueOf(orderId));
    
    return orderRepository.findById(orderId);
}
```

## 黑盒 + 白盒联动

当黑盒监控触发告警时，白盒监控帮助定位根因：

```
告警：OrderService P99 > 1s
    ↓
白盒追踪：查看最近 5 分钟的慢调用
    ↓
发现：80% 的慢调用都在"调用支付服务"这一步耗时过长
    ↓
进一步分析：支付服务的 P99 > 2s，且大部分时间是数据库查询慢
    ↓
结论：支付服务的数据库连接池耗尽，导致等待时间长
```

## 总结

可观测性三支柱各有分工：

- **Metrics（黑盒监控）**：告诉你"系统现在好不好"，适合告警和趋势分析
- **Tracing（白盒监控）**：告诉你"哪个地方有问题"，适合性能分析和根因定位
- **Logging**：告诉你"具体发生了什么"，适合 debug 和审计

最佳实践：**三管齐下，互通关联。**
- Prometheus + Grafana：指标采集和可视化
- SkyWalking / Jaeger：链路追踪
- ELK / Loki + ElastAlert：日志聚合和告警
- 三者通过 TraceId 关联：告警时可以通过 TraceId 跳转到完整的调用链和日志
