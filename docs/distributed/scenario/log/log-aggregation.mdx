# 日志聚合方案：ELK + Kafka 分流架构

一个中等规模的互联网系统，每天产生的日志量可能在数百 GB 到 TB 级别。这些日志分散在数百台服务器上，如何快速检索、快速分析、快速告警？

日志聚合系统，是每个成熟互联网公司的标配。

## 日志聚合的核心架构

## FileBeat 配置：轻量级日志采集

FileBeat 是 Elastic 官方推荐的日志采集器，比 Logstash 更轻量，适合部署在每台应用服务器上。

```yaml
# filebeat.yml
filebeat.inputs:
  # 业务日志
  - type: log
    enabled: true
    paths:
      - /var/log/application/*.log
    fields:
      log_type: business
      service: order-service
    fields_under_root: true
    
  # 错误日志（单独采集，方便告警）
  - type: log
    enabled: true
    paths:
      - /var/log/application/error.log
    fields:
      log_type: error
      service: order-service
    fields_under_root: true
    multiline:
      pattern: '^\d{4}-\d{2}-\d{2}'
      negate: true
      match: after  # 合并多行异常堆栈

# 输出到 Kafka（而不是直接到 ES，Kafka 作为缓冲层）
output.kafka:
  hosts: ["kafka1:9092", "kafka2:9092", "kafka3:9092"]
  
  # 按日志类型发送到不同的 Topic
  topic: 'logs-%{[fields.log_type]}'
  
  # 批量发送，提高吞吐量
  bulk_max_size: 2048
  producer.batch.max_messages: 100
  
  # 压缩
  compression: gzip
  
  # ACKs：等待所有副本确认
  required_acks: -1

# 加快采集速度（不做额外处理）
processors:
  - add_host_metadata: ~
  - add_cloud_metadata: ~
```

## Kafka Topic 分流设计

Kafka 在日志聚合架构中起到**缓冲层**的作用，同时支持**分流**——不同类型的日志进入不同的 Topic，Logstash 分别处理。

```yaml
# Kafka Topic 规划
logs-business      # 业务日志 → Logstash A → ES 索引 business-logs-*
logs-error         # 错误日志 → Logstash B → ES 索引 error-logs-*
logs-audit         # 审计日志 → Logstash C → ES 索引 audit-logs-*
logs-access        # 访问日志 → Logstash D → ES 索引 access-logs-*
```

:::tip 推荐
Kafka 分流的好处：
1. 不同类型日志的处理逻辑不同（字段解析、过滤规则）
2. 不同日志的保留策略不同（审计日志保留 1 年，业务日志保留 30 天）
3. 不同日志的查询频率不同（错误日志查询最频繁）
4. Kafka 本身有副本机制，日志不会丢失
:::

## Logstash 消费 + 解析 + 写入 ES

```conf
# logstash.conf - 业务日志处理管道
input {
  kafka {
    bootstrap_servers => "kafka1:9092,kafka2:9092"
    topics_pattern => "logs-business"
    group_id => "logstash-business"
    codec => json
    consumer_threads => 8
  }
}

filter {
  # 解析 JSON 格式的日志
  if [message] =~ /^\{/ {
    json {
      source => "message"
      target => "parsed"
    }
  }
  
  # 提取日志字段
  mutate {
    add_field => {
      "service_name" => "%{[fields][service]}"
      "trace_id" => "%{[mdc][traceId]}"
      "span_id" => "%{[mdc][spanId]}"
    }
  }
  
  # 过滤掉心跳日志
  if [message] =~ /heartbeat|ping/ {
    drop {}
  }
  
  # 转换时间戳
  date {
    match => ["[log][timestamp]", "ISO8601"]
    target => "@timestamp"
  }
  
  # GeoIP 解析 IP 地址
  if [remote_addr] {
    geoip {
      source => "remote_addr"
      target => "geoip"
    }
  }
}

output {
  elasticsearch {
    hosts => ["es1:9200", "es2:9200", "es3:9200"]
    
    # 按天创建索引
    index => "business-logs-%{+YYYY.MM.dd}"
    
    # 模板：定义字段类型
    template_name => "business-logs"
    template_overwrite => true
  }
}
```

## Elasticsearch 索引设计

```bash
# 创建日志索引模板
PUT _ilm/policy/business-logs-policy
{
  "policy": {
    "phases": {
      "hot": {
        "actions": {
          "rollover": {
            "max_size": "50GB",
            "max_age": "7d"
          }
        }
      },
      "warm": {
        "min_age": "7d",
        "actions": {
          "shrink": {
            "number_of_shards": 1
          },
          "forcemerge": {
            "max_num_segments": 1
          }
        }
      },
      "delete": {
        "min_age": "30d",
        "actions": {
          "delete": {}
        }
      }
    }
  }
}
```

:::warning 注意
日志量大的系统，Elasticsearch 存储成本是必须要考虑的问题。建议：
- 业务日志保留 7-30 天
- 错误日志保留 30-90 天
- 审计日志保留 1-3 年（建议用对象存储如 OSS/S3）
- 超过保留期的日志归档到冷存储
:::

## Kibana 查询示例

```json
// 查询特定 TraceId 的所有日志
{
  "query": {
    "bool": {
      "must": [
        { "match": { "traceId": "abc123def456" } }
      ]
    }
  }
}

// 查询最近 5 分钟的错误日志
{
  "query": {
    "bool": {
      "must": [
        { "range": { "@timestamp": { "gte": "now-5m" } } },
        { "match": { "level": "ERROR" } }
      ]
    }
  }
}

// 查询慢请求（响应时间 > 1 秒）
{
  "query": {
    "bool": {
      "filter": [
        { "range": { "cost_time_ms": { "gt": 1000 } } }
      ]
    }
  }
}
```

## 总结

日志聚合的核心架构是 **FileBeat + Kafka + Logstash + Elasticsearch + Kibana**：

1. **FileBeat**：轻量采集，部署在每个节点
2. **Kafka**：缓冲层 + 分流枢纽，支持多 Topic 并行处理
3. **Logstash**：解析、过滤、转换日志数据
4. **Elasticsearch**：分布式存储 + 全文检索
5. **Kibana**：可视化查询 + 监控面板

选型替代：
- 日志量大且成本敏感： Loki（基于对象存储，成本更低）
- 需要 APM 全链路追踪： SkyWalking（更强大的调用链分析）
- 实时告警为主： Elastic + ElastAlert 或 Grafana Loki + Alerting
