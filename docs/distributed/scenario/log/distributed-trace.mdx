# 分布式日志追踪：TraceId + SpanId 全链路串联

一个用户请求，经历了网关 → 认证服务 → 订单服务 → 库存服务 → 支付服务 → 数据库，整整 6 个节点。如果日志散落在 6 台机器上，你怎么追踪一次请求的完整调用链路？

这就是分布式日志追踪要解决的问题。

## 问题分析：日志散落问题

```
用户请求（TraceId: abc123）
    ↓
[192.168.1.10] 网关日志: abc123 | GET /order | 耗时 5ms
    ↓
[192.168.1.11] 认证服务: abc123 | 验证 Token | 耗时 10ms
    ↓
[192.168.1.12] 订单服务: abc123 | 创建订单 | 耗时 50ms
    ↓
[192.168.1.13] 库存服务: ??? | 扣减库存 | 耗时 20ms  ← TraceId 丢了！
    ↓
[192.168.1.14] 支付服务: ??? | 发起支付 | 耗时 200ms ← 链路断了
```

问题根源：**HTTP 请求头中的 TraceId 没有被传递到每个服务。**

## TraceId + SpanId 全链路追踪模型

分布式追踪的标准模型来自 Google Dapper 论文，使用 **TraceId + SpanId** 串联整个调用链路：

```
TraceId: abc123 (整个请求的唯一标识，全链路不变)
├── SpanId: 1.1 (订单服务 → 认证服务)
│   └── SpanId: 1.1.1 (认证服务内部调用)
├── SpanId: 1.2 (订单服务 → 库存服务)
│   └── SpanId: 1.2.1 (库存服务内部)
└── SpanId: 1.3 (订单服务 → 支付服务)
    └── SpanId: 1.3.1 (支付服务内部)
```

- **TraceId**：整个请求链路的唯一标识，从入口服务生成，向下透传
- **SpanId**：当前调用节点的唯一标识，`父 SpanId.子序号` 的层级结构

## Sleuth + Zipkin 实战

Spring Cloud Sleuth 是最常用的分布式追踪组件，它自动为每个请求生成 TraceId 和 SpanId：

### Maven 依赖

```xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-sleuth</artifactId>
</dependency>
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-zipkin</artifactId>
</dependency>
```

### 配置

```yaml
spring:
  zipkin:
    base-url: http://zipkin-server:9411
    sender:
      type: web  # 异步发送 span 数据到 Zipkin
  sleuth:
    sampler:
      probability: 1.0  # 采样率：100%（生产环境可降低到 10-30%）
    propagation:
      # 指定 TraceId 在 HTTP 头中的 Key
      service:
        name: ${spring.application.name}
```

### 手动传递 TraceId

对于非 Spring MVC 的调用（如异步消息、定时任务），需要手动传递 TraceId：

```java
// 方式一：通过消息头传递 TraceId
@Service
public class OrderMessageProducer {
    
    @Autowired
    private StreamBridge streamBridge;
    
    public void sendOrderCreatedEvent(Order order) {
        MessageBuilder builder = MessageBuilder
            .withPayload(order)
            .setHeader("X-B3-TraceId", TraceContext.traceId());
        
        streamBridge.send("order-events-out-0", builder.build());
    }
}

// 消费端接收
@Bean
public Consumer<Order> handleOrderCreated(Message<Order> message) {
    // Sleuth 自动从消息头提取 TraceId 并放入 TraceContext
    log.info("收到订单事件, traceId={}", TraceContext.traceId());
    
    processOrder(message.getPayload());
}
```

```java
// 方式二：异步线程池传递 TraceId
@Configuration
public class AsyncConfig {
    
    @Bean("traceExecutor")
    public Executor traceExecutor() {
        return ContextAwareCallable.create(
            () -> {
                // 异步执行的任务
            },
            TraceContext.get()
        );
    }
}

// 使用
@Async("traceExecutor")
public void asyncProcess(Order order) {
    // 异步任务的日志会自动带上 TraceId
    log.info("异步处理订单, traceId={}", TraceContext.traceId());
}
```

### MDC 集成：让日志自带 TraceId

把 TraceId 和 SpanId 放入 MDC（Mapped Diagnostic Context），日志格式自动带上：

```java
@Configuration
public class TracingMdcConfig {
    
    @Bean
    public Slf4jMDCFilter mdcFilter() {
        return new Slf4jMDCFilter(
            Collections.singletonMap("X-B3-TraceId", "traceId"),
            Collections.singletonMap("X-B3-SpanId", "spanId")
        );
    }
}
```

```xml
<!-- 日志格式中加入 %X{traceId} -->
<PatternLayout pattern="%d{yyyy-MM-dd HH:mm:ss.SSS} [%thread] [%X{traceId}] [%X{spanId}] %-5level %logger{36} - %msg%n"/>
```

输出效果：
```
2024-01-01 12:00:00.001 [http-nio-8080-1] [abc123def456] [1.2] INFO  OrderService - 创建订单成功
2024-01-01 12:00:00.050 [http-nio-8080-2] [abc123def456] [1.3] INFO  StockService - 扣减库存
```

## OpenTelemetry：更现代的追踪标准

Sleuth + Zipkin 是 Spring Cloud 时代的方案。**OpenTelemetry** 是云原生时代的标准：

```java
// OpenTelemetry Java SDK
@Configuration
public class OtelConfig {
    
    @Bean
    public OpenTelemetry openTelemetry() {
        // 创建 OpenTelemetry 实例
        OpenTelemetry openTelemetry = OpenTelemetrySdk.builder()
            .setTracerProvider(
                OpenTelemetrySdkAutoConfiguration.initialize())
            .build();
        
        return openTelemetry;
    }
    
    @Bean
    public Tracer tracer(OpenTelemetry openTelemetry) {
        return openTelemetry.getTracer("order-service");
    }
}

// 手动创建 Span
@Service
public class OrderServiceWithTracing {
    
    @Autowired
    private Tracer tracer;
    
    public void createOrder(Order order) {
        // 创建子 Span
        Span span = tracer.spanBuilder("createOrder")
            .setParent(TraceContext.current())  // 自动继承上游 Context
            .setAttribute("order.id", order.getId())
            .setAttribute("user.id", order.getUserId())
            .startSpan();
        
        try (Scope scope = span.makeCurrentScope()) {
            // 执行业务逻辑
            doCreateOrder(order);
            
            span.setStatus(StatusCode.OK);
        } catch (Exception e) {
            span.setStatus(StatusCode.ERROR, e.getMessage());
            span.recordException(e);
            throw e;
        } finally {
            span.end();  // 结束 Span，数据自动上报到 OTLP Collector
        }
    }
}
```

## 总结

分布式日志追踪的核心是 **TraceId 透传**：

1. **入口服务生成 TraceId**：在网关或第一个服务生成
2. **HTTP 调用时传递**：通过 HTTP 头（`X-B3-TraceId` 等）传递
3. **异步调用时传递**：通过消息头或线程上下文传递
4. **日志中带上 TraceId**：通过 MDC + 日志格式配置
5. **汇聚到追踪系统**：Zipkin / Jaeger / SkyWalking / OpenTelemetry

选型建议：
- Spring Cloud 体系：Sleuth + Zipkin
- 云原生 / Kubernetes 体系：OpenTelemetry + Jaeger
- 国内项目：SkyWalking（对中文友好，有 APM 功能）
