# 同城双活架构设计

"两地三中心" 是过去企业 IT 的标配，但真正能扛住机房级别故障的，是同城双活——两个机房同时对外提供服务，任何一个机房挂了，业务不中断。这就是同

城双活，也是当前头部互联网公司的标准架构。

## 为什么需要同城双活？

传统的"主备"模式，备机房平时是闲置的。主机房故障时，需要等待备机房拉起，这个过程可能需要几分钟到几十分钟。对于核心业务系统，这个窗口期的损失是巨大的。

同城双活的核心目标：**主备机房同时在线，任何一个机房都可以独立承接全部流量。**

## 同城双活的核心设计

### 一、流量分配：谁说了算？

两个机房同时对外服务，流量怎么分配？通常有两种方式：

1. **DNS 解析轮询**：用户请求通过 DNS 解析到不同机房的 VIP（虚拟 IP）。简单，但切换慢，DNS 缓存问题严重。
2. **GSLB（全局负载均衡）**：在 DNS 层面之上增加 GSLB 设备，根据机房健康状态、用户地理位置、实时负载智能分配流量。推荐这种方式。

```bash
# GSLB 的健康检查配置示例
北京机房 VIP: 10.0.1.100 (权重 50%)
廊坊机房 VIP: 10.0.2.100 (权重 50%)

# 当廊坊机房故障时，GSLB 自动将权重调整为 100%/0%
北京机房 VIP: 10.0.1.100 (权重 100%)
廊坊机房 VIP: 10.0.2.100 (权重 0%)
```

### 二、数据同步：双向同步还是单向同步？

这是同城双活最核心的问题，也是最容易出错的地方。

| 同步方式 | 说明 | 优点 | 缺点 |
|---|---|---|---|
| 双向同步 | 两个机房互相同步数据 | 任何机房都能写入 | 存在数据冲突风险 |
| 单向同步 | 只有主机房能写入，备机房只读 | 数据一致性好 | 备机房不能写入 |
| 假双活 | 主机房写，备机房只读，写故障时切备 | 简单可靠 | 写能力不是真正双活 |

大多数金融业务采用"单向同步 + 写 IP 漂移"：主机房负责写入，通过数据库主从同步到备机房。当主机房故障时，写 IP 漂移到备机房，继续服务。

### 三、数据库层：双活的关键

```bash
# MySQL 主从同步架构（单向双活）
主机房 MySQL Master (写) ←→ MySQL Slave (备机房，读)
        ↓ 同步延迟 < 1s
    GTID 半同步复制
```

:::warning 注意
半同步复制（Semi-sync Replication）可以保证：主库的事务提交前，必须等待至少一个从库收到 binlog。这比异步复制可靠，但会增加写操作的延迟（约 1-5ms）。
:::

### 四、缓存层：跨机房缓存一致性

Redis 主从同步通常是异步的，存在一定的数据不一致窗口。常见的处理方式：

- **不跨机房读**：用户请求打到哪个机房，就从哪个机房的 Redis 读，尽量减少跨机房读
- **读主从**：优先读本地 Redis，从库只作为兜底
- **延迟双删**：写操作后延迟一段时间再删除缓存，降低脏读概率

## 同城双活的风险

同城双活解决了机房级故障，但**网络分区**仍然是最大的风险。如果两个机房之间的网络断开，会发生什么？

```bash
# 网络分区场景
主机房 → 写入数据 → 提交成功
     ↘ 网络断开
备机房 → 读取主机房最新数据 → 读不到（脏读窗口）

# 如果此时主机房完全不可用，备机房接管：
备机房 → 认为自己可以写入 → 写入成功
主机房恢复 → 两边数据不一致 → 需要人工介入
```

这就是为什么"真正的双活"（两个机房都能写）是非常困难的，通常需要引入分布式共识算法（Paxos/Raft）来保证数据一致性。

## 总结

同城双活的核心是：**流量能切换、数据能同步、缓存能一致。**

实现难度排序：单向双活 `<` 双向双活 `<` 真正双活。大多数互联网公司做到"单向双活 + 写 IP 漂移"就已经足够应对 99% 的机房故障场景了。

如果你的业务对 RPO（Recovery Point Objective）要求 `<` 1 秒，那必须考虑真正的双向同步方案，这需要引入分布式数据库（如 TiDB、CockroachDB）或在应用层自己实现冲突检测。
