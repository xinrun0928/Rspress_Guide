# 跨机房数据同步方案：MySQL binlog + Canal + Kafka

你有没有遇到过这种场景：主库在北京机房，需要实时将数据同步到上海的备机房做查询。两个机房之间的网络偶尔抖动，数据同步延迟不稳定，甚至可能出现数据不一致。

跨机房数据同步是分布式系统中公认的技术难题之一。本质上，它是在**一致性**和**性能**之间做 trade-off。

## 常见的跨机房数据同步架构

### 一、基于 MySQL 原生主从复制

```bash
# MySQL 原生主从复制（单向同步）
主机房 MySQL Master ──复制──▶ 备机房 MySQL Slave
         │                           │
         │                    数据延迟 0~N秒
         │                           │
         └── binlog 同步（单线程）──┘
```

优点：简单，MySQL 原生支持。
缺点：跨机房网络抖动时复制中断，单线程复制（MySQL 5.7 前）导致从库延迟大。

### 二、基于 Canal + Kafka 的 CDC 方案

Canal 是阿里巴巴开源的 MySQL binlog 增量订阅组件。它的核心原理：

1. Canal 模拟 MySQL slave 的协议，向主库请求 binlog
2. 主库推送 binlog 给 Canal
3. Canal 解析 binlog，转换为 JSON 消息
4. 消息发送到 Kafka（或 RocketMQ）
5. 消费端订阅 Kafka，消费消息，应用到目标数据库

```
MySQL Master ──binlog──▶ Canal Server ──JSON──▶ Kafka ──消费──▶ 目标 MySQL
                              │                    │
                              │              跨城专线
                              ▼                    ▼
                        Canal Client         备机房消费者
```

```java
// Canal 消息格式（JSON）
{
  "table": "orders",
  "pkNames": ["id"],
  "data": {
    "id": 10001,
    "user_id": 12345,
    "amount": 299.00,
    "status": "PAID"
  },
  "old": {
    "status": "PENDING"
  },
  "type": "UPDATE",  // INSERT / UPDATE / DELETE
  "ts": 1704067200000
}
```

### 三、Debezium + Kafka Connect 方案

Debezium 是另一款流行的 CDC 工具，支持 MySQL、PostgreSQL、MongoDB 等。相比 Canal，Debezium 更适合 Kafka 生态：

```bash
# Debezium 部署在 Kafka Connect 集群中
MySQL → Debezium(在KafkaConnect中) → Kafka Topic → 消费应用
```

## Canal + Kafka 的完整实现

### 部署架构

```bash
# canal.conf 配置
[server]
serverId=6754  # 唯一 ID，不能与 MySQL 从库 ID 冲突

[destinations]
canal1:
  canalInstanceManagerAddress: localhost:2181  # ZK 注册中心
  canalDestination: example
  canalServerSyslog: 127.0.0.1:5444
  
[mq]
# Kafka 相关配置
 kafka.bootstrap.servers: kafka1:9092,kafka2:9092
 kafka.topic: binlog_orders
 kafka.partition.hash: true  # 按表名 hash 到不同分区
```

### 消费端幂等处理

跨机房数据同步，最大的问题是**幂等性**：同一消息可能被重复消费。

```java
@Component
public class BinlogConsumer {
    
    @Autowired
    private JdbcTemplate jdbcTemplate;
    
    @KafkaListener(topics = "binlog_orders")
    public void consume(String message) {
        CanalMessage msg = JSON.parseObject(message, CanalMessage.class);
        
        // 幂等处理：基于主键的 UPSERT
        switch (msg.getType()) {
            case "INSERT":
                jdbcTemplate.update(
                    "INSERT IGNORE INTO orders (id, user_id, amount, status) " +
                    "VALUES (?, ?, ?, ?)",
                    msg.getData().get("id"),
                    msg.getData().get("user_id"),
                    msg.getData().get("amount"),
                    msg.getData().get("status")
                );
                break;
                
            case "UPDATE":
                jdbcTemplate.update(
                    "UPDATE orders SET status = ? WHERE id = ?",
                    msg.getData().get("status"),
                    msg.getData().get("id")
                );
                break;
                
            case "DELETE":
                jdbcTemplate.update(
                    "DELETE FROM orders WHERE id = ?",
                    msg.getData().get("id")
                );
                break;
        }
    }
}
```

## 跨机房同步的核心挑战

### 挑战一：网络分区

跨城网络不稳定是常态，不是异常。MySQL 5.7 的半同步复制可以缓解这个问题，但无法完全解决。

**解决思路**：引入消息队列作为缓冲层（Kafka/RocketMQ），在网络抖动时消息堆积在队列中，网络恢复后继续消费。

### 挑战二：数据冲突

双向同步时，同一条记录在两个机房同时被修改，会产生冲突。

**解决思路**：
- 业务层面禁止双向写入（单向双活）
- 使用 Last Write Wins（最后写入胜出）+ 对账机制
- 引入 CRDT（Conflict-free Replicated Data Types）

### 挑战三：延迟尖刺

跨城延迟通常是 20-50ms，但网络抖动时可能飙升到数秒甚至分钟级。

**解决思路**：
- 监控跨城延迟，设置告警阈值（如延迟 `>` 1s）
- 延迟超过阈值时，业务层面降级为本地优先读取
- 定期做数据对账，发现不一致及时修复

## 总结

跨机房数据同步没有银弹，关键是**根据业务对数据一致性的要求，选择合适的方案**：

| 方案 | 一致性 | 延迟 | 复杂度 | 适用场景 |
|---|---|---|---|---|
| MySQL 主从 | 弱一致 | 低 | 低 | 容忍秒级延迟的读场景 |
| Canal + Kafka | 最终一致 | 中 | 中 | 异步计算、缓存更新 |
| Debezium + Kafka | 最终一致 | 中 | 中 | Kafka 生态 |
| 同步双写 | 强一致 | 高 | 高 | 核心金融业务 |

如果你的业务允许最终一致，CDC + Kafka 是目前最成熟的方案。如果需要强一致，考虑业务改造为单向双活。
