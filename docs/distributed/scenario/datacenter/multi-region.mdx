# 异地多活架构：单元化部署与数据同步方案

"我的系统能扛住整个城市断电吗？"

2021 年郑州特大暴雨，导致整个城市大面积停电、断网。无数互联网公司的服务中断，有的甚至停摆了好几天。而那些实现了异地多活架构的公司，几乎没有受到影响。这就是异地多活的价值——在更大的灾难面前，保证业务连续性。

## 什么是异地多活？

异地多活是同城双活的升级版：将业务部署在**地理上相隔较远的多个机房**，每个机房都能独立提供服务。与同城双活相比，异地多活可以应对城市级甚至地区级的灾难。

### 异地多活 vs 同城双活

| 维度 | 同城双活 | 异地多活 |
|---|---|---|
| 机房距离 | 通常 `<` 100km | 通常 `>=` 500km |
| 延迟 | 跨机房延迟 1-5ms | 跨机房延迟 10-100ms |
| 成本 | 较高 | 非常高 |
| 能应对的灾难级别 | 单机房断电/网络故障 | 城市级灾难 |
| 数据一致性挑战 | 较小 | 非常大 |

异地多活的跨机房延迟是不可忽视的。1000km 的物理距离，光速传播也需要约 3.3ms 往返，加上网络设备转发延迟，实际延迟可能在 20-50ms。这意味着：异地多活下，任何跨机房的同步操作都会明显变慢。

## 单元化部署：异地多活的核心架构

异地多活的核心架构思想是**单元化（Cell-based）**：把整个系统拆成多个独立的"单元"，每个单元内包含完整的业务逻辑和数据。

### 单元化的设计原则

```
┌─────────────────────────────────────────────────────┐
│                   全局路由层                        │
│  (根据用户 ID / 请求特征，路由到对应单元)            │
└───────────────────────┬─────────────────────────────┘
                        │
    ┌───────────────────┼───────────────────┐
    ▼                   ▼                   ▼
┌─────────┐        ┌─────────┐        ┌─────────┐
│ 单元 A  │        │ 单元 B  │        │ 单元 C  │
│ 北京    │        │ 上海    │        │ 广州    │
│ 1000万  │        │ 800万   │        │ 600万   │
│ 用户    │        │ 用户    │        │ 用户    │
└────┬────┘        └────┬────┘        └────┬────┘
     │                  │                  │
     ▼                  ▼                  ▼
  本地 MySQL         本地 MySQL         本地 MySQL
  (该单元独享)       (该单元独享)       (该单元独享)
```

**单元化的铁律**：流量在哪里，数据就在哪里。用户的请求必须在同一个单元内完成闭环，不能跨单元读写数据。

### 用户如何路由到单元？

通常使用**全局负载均衡器（GSLB）** 或 **一致性哈希**：

```java
public class CellRouter {
    // 根据用户 ID 的哈希值，决定用户属于哪个单元
    public Cell route(long userId) {
        // 取用户 ID 的 CRC32 哈希值
        long hash = Math.abs(CRC32.of(userId));
        
        // 根据哈希值路由到对应单元
        if (hash % 100 < 40) {
            return Cell.BEIJING;  // 40% 用户
        } else if (hash % 100 < 75) {
            return Cell.SHANGHAI;  // 35% 用户
        } else {
            return Cell.GUANGZHOU;  // 25% 用户
        }
    }
}
```

## 异地多活的数据同步方案

单元化的关键约束：**每个单元只能读写本单元的数据。** 那么，当用户跨城市迁移时（比如从北京出差到广州），他的数据怎么迁移？

### 方案一：热迁移（推荐）

用户进入新城市时，通过 GSLB 将流量路由到新单元，但数据仍然在原单元——只有当用户在新城市产生新数据积累到一定程度后，才触发数据迁移。

```bash
# 热迁移流程
1. 用户从北京出差到广州
2. GSLB 检测到用户地理位置变化，将流量切到广州单元
3. 广州单元需要读取北京单元的数据（跨单元读，通过 HTTP/gRPC）
4. 用户写操作写入广州单元，通过同步服务异步同步到北京
5. 当用户在广州的数据量超过阈值（比如 1000 条），触发正式迁移
6. 迁移期间写双份，迁移完成后只写广州
```

热迁移的核心思想是：**先路由再迁移，不要求数据立即同步。**

### 方案二：基于数据库 CDC 的实时同步

使用 Change Data Capture（CDC）技术，监听数据库的 binlog，实时将数据变化同步到其他机房。

```
MySQL binlog → Canal → Kafka → 跨城专线 → 目标机房 Kafka → 写入目标 MySQL
```

:::tip 推荐
阿里开源的 Canal + RocketMQ 是比较成熟的 MySQL CDC 方案，很多头部互联网公司都在用。
:::

## 异地多活的最大挑战：RPO vs RTO

异地多活面临的最核心矛盾：

- **RPO（Recovery Point Objective）**：能容忍丢失多少数据？
- **RTO（Recovery Time Objective）**：故障后多久能恢复服务？

跨城市同步延迟在 10-100ms 之间，任何同步操作都会增加延迟。如果要求 RPO `=` 0（零数据丢失），意味着所有写操作必须等所有机房都确认，数据写入延迟会非常高。

实际工程中，通常允许一定的 RPO（如 1 秒），通过**异步同步 + 对账机制**来保证最终一致性。

## 总结

异地多活的核心挑战不是技术，而是**业务改造**。单元化要求每个业务模块都能独立部署、独立数据存储。这对业务架构的改造是伤筋动骨的。

典型的成功案例：
- 阿里：单元化改造 + DRC 数据同步，支撑了双十一的流量洪峰
- 蚂蚁金服：SOFA 单元化 + LDC（Logical Data Center）架构

异地多活不是银弹，只有对业务连续性有极端要求的核心系统才值得投入。
