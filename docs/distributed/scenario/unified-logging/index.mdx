# 统一日志与可观测性：TraceId 串联全链路

用户投诉：订单支付失败了。

你去查日志。订单服务说："我调用了库存服务，库存服务超时了。"库存服务说："我收到了请求，但我没收到支付成功的回调。"

**三个人吵架，没有结论。问题在哪？**

答案是：**日志散落在各个服务里，没有串联起来。**

## 可观测性的三根支柱

```
┌─────────────────────────────────────────────────────────┐
│               可观测性三根支柱（Three Pillars）          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  Metrics（指标）                                         │
│  └─ "系统现在怎么样？"                                   │
│     CPU使用率、QPS、响应延迟、P99、错误率                 │
│                                                         │
│  Tracing（链路）                                         │
│  └─ "这个请求经历了什么？"                               │
│     TraceId 串联：网关 → 订单 → 库存 → 支付 → 数据库    │
│                                                         │
│  Logging（日志）                                         │
│  └─ "具体发生了什么？"                                  │
│     错误堆栈、业务日志、审计日志                         │
│                                                         │
│  三者结合：Metrics发现异常 → Tracing定位慢请求 → 日志定位根因 │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

## 链路追踪的核心机制

分布式链路追踪的核心是 **TraceId**：一个请求从入口进入时，生成一个全局唯一的 TraceId，后续所有调用都带着这个 ID。

```
┌─────────────────────────────────────────────────────────┐
│                    TraceId 串联示例                      │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  [请求入口]                                              │
│  生成 TraceId: abc-123                                  │
│                                                         │
│  [服务 A: 订单服务]                                      │
│  traceId: abc-123 │ 耗时: 5ms │ 操作: 创建订单        │
│                                                         │
│  [服务 B: 库存服务]                                      │
│  traceId: abc-123 │ 耗时: 30ms │ 操作: 扣减库存        │
│                                                         │
│  [服务 C: 支付服务]                                      │
│  traceId: abc-123 │ 耗时: 200ms │ 操作: 发起支付       │
│                                                         │
│  [数据库]                                                │
│  traceId: abc-123 │ 耗时: 2ms │ SQL: UPDATE...         │
│                                                         │
│  汇总：abc-123 请求总耗时 237ms，慢 → 支付服务耗时最长   │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

## 日志聚合方案

| 方案 | 特点 | 适用规模 |
|------|------|---------|
| ELK（Elasticsearch + Logstash + Kibana） | 功能全面，查询灵活 | 中大型规模 |
| EFK（Elasticsearch + Fluentd + Kibana） | 轻量级，Fluentd 资源占用低 | 中小型规模 |
| Loki | 简单，存储成本低 | 中小型规模 |
| 自建 + 云日志服务 | 运维成本低 | 不想自建 |

## 核心公式

```
可观测性 = 日志聚合（ELK/EFK）+ 链路追踪（SkyWalking/Jaeger）+ 指标监控（Prometheus）

全链路追踪实现：
- 入口：网关生成 TraceId，放入 HTTP Header
- 传递：每个服务读取 Header，传递给下游
- 输出：TraceId 写入日志，MDC 上下文
- 汇总：日志收集到 ELK，按 TraceId 检索

快速定位问题：Metrics 发现异常 → TraceId 定位请求 → 日志详情排查根因
```
