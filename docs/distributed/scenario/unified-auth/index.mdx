# 微服务统一认证：Session 共享 vs Token

你有没有想过这个问题：

用户在你的电商网站登录了一次，为什么访问商品详情、购物车、订单列表这些不同的微服务，都能保持登录状态？

**这不是因为每个微服务都"认识"这个用户，而是有一个统一的机制，让每个服务都知道"当前请求是谁发来的"。**

这就是微服务认证要解决的核心问题。

## 两种认证方案

```
┌─────────────────────────────────────────────────────────┐
│                   认证方案对比                           │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  方案一：Session 共享                                    │
│  └─ 登录 → 生成 Session → 存入 Redis                   │
│  └─ 请求 → 各服务从 Redis 读取 Session                  │
│  └─ 优点：传统架构改动小                               │
│  └─ 缺点：有状态，扩展困难                             │
│                                                         │
│  方案二：Token（JWT）                                   │
│  └─ 登录 → 生成 Token（签名）→ 返回客户端              │
│  └─ 请求 → 验证 Token 签名 → 解析用户信息              │
│  └─ 优点：无状态，天生可扩展                           │
│  └─ 缺点：撤销困难，Payload 不能放敏感信息              │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

## JWT 的结构

```
Header.Payload.Signature

- Header: {"alg": "HS256", "typ": "JWT"}
- Payload: {"userId": 123, "exp": 1699999999}
- Signature: HMAC-SHA256(Header.Payload, secret)
```

三部分分别 Base64 编码，用 `.` 连接。服务端只存 secret，验证签名即可。

## 方案对比

| 维度 | Session 共享 | Token |
|------|-------------|-------|
| 存储位置 | Redis | 客户端 |
| 状态 | 有状态 | 无状态 |
| 扩展性 | 需要 Redis | 天生可扩展 |
| 跨域支持 | 需特殊处理 | 原生支持 |
| 撤销能力 | 立即生效 | 需额外机制 |
| 适用场景 | 企业内网 | 互联网应用 |

## 核心公式

```
微服务认证 = Session共享（Redis） OR Token（JWT）

选型建议：
- 企业内网、微服务少 → Spring Session + Redis
- 互联网应用、跨域需求 → JWT
- 需要 Token 撤销 → JWT + Redis 黑名单

安全要点：
- Token 有效期不宜过长（access token: 15min ~ 2h）
- refresh token 单独管理
- Payload 不放敏感信息（Base64 可解码）
- 传输过程 HTTPS 加密
```
