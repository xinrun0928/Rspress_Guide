# CAS 单点登录

你一定有过这样的体验：在淘宝登录后，打开天猫发现已经自动登录了；在阿里云登录后，访问钉钉、企业邮箱都无需再次输入密码。这就是 SSO（Single Sign-On，单点登录）的魅力。

但你有没有想过，这背后的技术原理是什么？Cookie 跨域共享？域名不同怎么做？今天我们来深入剖析最经典的 CAS 协议，看看它是如何实现「一次登录，处处通行」的。

## 一、为什么需要 SSO

在企业场景中，往往有多个业务系统：

```
┌────────────────────────────────────────────────────────────────┐
│                        企业内网                                 │
│                                                                │
│   ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐          │
│   │  OA系统  │  │  邮件系统 │  │  考勤系统  │  │  审批系统  │          │
│   └─────────┘  └─────────┘  └─────────┘  └─────────┘          │
│        │            │            │            │                │
│        └────────────┴────────────┴────────────┘                │
│                         都需要登录                               │
└────────────────────────────────────────────────────────────────┘
```

如果没有 SSO：
- 每个系统都要维护独立的用户体系
- 用户需要记住多套账号密码
- 管理员需要在每个系统单独创建/删除账号
- 员工离职后，需要在每个系统逐一禁用账号

SSO 的核心价值是**统一身份管理**，让用户登录一次，所有系统都认识你。

## 二、CAS 协议核心原理

CAS（Central Authentication Service，中心认证服务）是最经典的 SSO 协议，由 Yale 大学开源，目前是 OASIS 标准。

### 2.1 核心概念

| 术语 | 说明 |
|---|---|
| CAS Server | 认证中心，负责用户登录、Token 颁发 |
| CAS Client | 各业务系统，负责验证 Ticket、与 CAS Server 通信 |
| TGT（Ticket Granting Ticket） | 用户在 CAS Server 的登录凭证，存储在 Session 中 |
| ST（Service Ticket） | 一次性票据，CAS Server 颁发给 Client 用于验证 |
| TGC（Ticket Granting Cookie） | 浏览器 Cookie，存储 TGT ID |

### 2.2 CAS 认证流程

```
┌─────────────────────────────────────────────────────────────────────────┐
│                              第一次访问 App1                              │
└─────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────────┐
│  1. App1 发现用户未登录，重定向到 CAS Server                             │
│     GET /app1/service?target=https://app1.example.com                   │
└─────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────────┐
│  2. CAS Server 发现用户未登录（无 TGC Cookie），显示登录页               │
│     GET /cas/login?service=https://app1.example.com                     │
└─────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────────┐
│  3. 用户提交账号密码，CAS Server 验证通过                                │
│     POST /cas/login                                                     │
│     username=zhangsan&password=xxx&service=https://app1.example.com     │
└─────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────────┐
│  4. CAS Server 创建 TGT，生成 TGC Cookie，返回 ST                       │
│     302: https://app1.example.com/?ticket=ST-xxx-xxx                     │
│     Set-Cookie: TGC=TGT-xxx-xxx; Domain=.example.com; HttpOnly           │
└─────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────────┐
│  5. App1 用 ST 向 CAS Server 验证，获取用户身份                          │
│     GET /cas/validate?service=https://app1.example.com&ticket=ST-xxx    │
│     Response: <cas:user>zhangsan</cas:user>                             │
└─────────────────────────────────────────────────────────────────────────┘
```

### 2.3 再次访问 App2

```
┌─────────────────────────────────────────────────────────────────────────┐
│                              访问 App2                                  │
└─────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────────┐
│  1. App2 重定向到 CAS Server                                             │
│     GET /cas/login?service=https://app2.example.com                      │
└─────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────────┐
│  2. 浏览器自动带上 TGC Cookie，CAS Server 发现 TGT 有效                  │
│     无需再次输入密码，直接生成 ST 重定向回 App2                           │
│     302: https://app2.example.com/?ticket=ST-yyy-yyy                     │
└─────────────────────────────────────────────────────────────────────────┘
```

:::tip 核心原理
ST（Service Ticket）是一次性的，用完即废。每次访问不同业务系统，都会生成新的 ST。这保证了即使 ST 被截获，也无法用来访问其他系统。
:::

```properties title="application.properties"
# CAS 服务名称
cas.server.name=https://cas.example.com
cas.server.prefix=${cas.server.name}/cas

# 客户端应用注册（需要预先注册才能访问）

cas.serviceRegistry.json.location=classpath:/services

# 数据库认证
cas.authn.jdbc.query[0].sql=SELECT password FROM users WHERE username=?
cas.authn.jdbc.query[0].passwordField=password
cas.authn.jdbc.query[0].passwordEncoder.type=BCRYPT
cas.authn.jdbc.query[0].driverClass=com.mysql.cj.jdbc.Driver
cas.authn.jdbc.query[0].url=jdbc:mysql://localhost:3306/cas?useSSL=false
cas.authn.jdbc.query[0].user=root
cas.authn.jdbc.query[0].password=xxx

# Ticket 存储（生产环境必须用 Redis 或数据库）
cas.ticket.registry.redis.host=localhost
cas.ticket.registry.redis.port=6379

# TGC Cookie 配置
server.servlet.session.cookie.name=TGC
server.servlet.session.cookie.http-only=true
server.servlet.session.cookie.secure=true
server.servlet.session.timeout=PT8H
```

## 四、CAS Client 集成

### 4.1 Spring Boot 集成

```java title="CAS 配置类"
@Configuration
@EnableWebSecurity
public class CasSecurityConfig {

    @Value("${cas.server.url}")
    private String casServerUrl;

    @Value("${app.server.url}")
    private String appServerUrl;

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .authorizeRequests()
                .antMatchers("/public/**", "/login/**").permitAll()
                .anyRequest().authenticated()
                .and()
            .addFilterBefore(casAuthenticationFilter(), UsernamePasswordAuthenticationFilter.class)
            .logout()
                .logoutSuccessUrl(casServerUrl + "/logout?service=" + appServerUrl)
                .and()
            .csrf()
                .disable();

        return http.build();
    }

    @Bean
    public CasAuthenticationFilter casAuthenticationFilter() {
        CasAuthenticationFilter filter = new CasAuthenticationFilter();
        filter.setAuthenticationManager(authenticationManager());
        filter.setServiceProperties(serviceProperties());
        return filter;
    }

    @Bean
    public ServiceProperties serviceProperties() {
        ServiceProperties properties = new ServiceProperties();
        properties.setService(appServerUrl + "/login/cas");
        properties.setSendRenew(false);
        properties.setArtifactParameter("ticket");
        return properties;
    }

    @Bean
    public CasAuthenticationProvider casAuthenticationProvider() {
        CasAuthenticationProvider provider = new CasAuthenticationProvider();
        provider.setServiceProperties(serviceProperties());
        provider.setTicketValidator(new Cas30ServiceTicketValidator(casServerUrl));
        provider.setUserDetailsService(userDetailsService());
        provider.setKey("CasAuthenticationProvider");
        return provider;
    }
}
```

### 4.2 用户信息获取

```java title="获取当前登录用户"
@Service
public class CurrentUserService {
    @Autowired
    private SecurityContextHolderStrategy securityContextHolderStrategy;

    public String getCurrentUsername() {
        Authentication authentication = securityContextHolderStrategy.getContext().getAuthentication();
        if (authentication != null && authentication.isAuthenticated()) {
            return authentication.getName();
        }
        return null;
    }

    public Map<String, Object> getCurrentUserAttributes() {
        Authentication authentication = securityContextHolderStrategy.getContext().getAuthentication();
        if (authentication instanceof CasAuthenticationToken) {
            CasAuthenticationToken casToken = (CasAuthenticationToken) authentication;
            return casToken.getPrincipal().getAttributes();
        }
        return Collections.emptyMap();
    }
}
```

### 4.3 前端配合

```javascript title="登录页面"
function redirectToCas() {
    const service = encodeURIComponent(window.location.origin + '/login/cas');
    window.location.href = `https://cas.example.com/cas/login?service=${service}`;
}

// 检查 CAS 回调
function handleCasCallback() {
    const urlParams = new URLSearchParams(window.location.search);
    const ticket = urlParams.get('ticket');

    if (ticket) {
        // 有 ticket，说明是从 CAS 重定向回来的
        // 交给后端验证
        fetch(`/api/cas/callback?ticket=${ticket}`)
            .then(res => res.json())
            .then(data => {
                if (data.success) {
                    // 移除 URL 中的 ticket 参数
                    window.history.replaceState({}, '', window.location.pathname);
                    // 刷新页面获取登录状态
                    window.location.reload();
                }
            });
    }
}
```

## 五、CAS 安全性设计

### 5.1 Ticket 安全机制

```java title="Ticket 生成与校验"
@Service
public class TicketService {
    private static final SecureRandom SECURE_RANDOM = new SecureRandom();

    public String generateServiceTicket() {
        byte[] bytes = new byte[32];
        SECURE_RANDOM.nextBytes(bytes);
        return "ST-" + Base64.getUrlEncoder().withoutPadding().encodeToString(bytes);
    }

    public boolean validateTicket(String ticket, String serviceId) {
        // 校验逻辑
        // 1. Ticket 格式正确
        // 2. Ticket 未过期（默认 10 秒）
        // 3. Ticket 未被使用过
        // 4. serviceId 与签发时一致
    }
}
```

| 安全措施 | 说明 |
|---|---|
| 一次性 Ticket | ST 只能用一次，防止重放 |
| 短期有效 | ST 默认 10 秒内必须使用 |
| 服务绑定 | ST 只能用于签发时指定的服务 |
| TGT 长期有效 | 用户登录态可保持 8 小时 |
| HttpOnly Cookie | 防止 XSS 攻击窃取 TGC |
| Secure Cookie | 仅 HTTPS 传输 |

### 5.2 防御 CSRF 攻击

CAS 支持通过 `pgtUrl`（Proxy Callback URL）实现服务间委托认证，但需要额外的 CSRF Token 保护。

```properties
# 启用 CSRF 防护
cas.sso.initiated=true
cas.sso.stateful.enabled=true
```

## 七、与其他 SSO 方案对比

| 特性 | CAS | OAuth 2.0 + OIDC | SAML 2.0 |
|---|---|---|---|
| 协议复杂度 | 中等 | 较低 | 较高 |
| 手机端支持 | 一般 | 优秀 | 一般 |
| 微服务友好度 | 一般 | 优秀 | 一般 |
| 企业级应用 | 优秀 | 良好 | 优秀 |
| 标准程度 | OASIS 标准 | IETF 标准 | OASIS 标准 |
| 国内生态 | 成熟 | 一般 | 一般 |

:::warning 选型建议
CAS 协议适合**同域名体系下的多应用 SSO**，如果你的应用分布在不同域名，建议使用 OAuth 2.0 + OIDC 方案。
:::

## 八、常见问题与排查

### 8.1 Ticket 验证失败

```bash
# 常见原因及排查
# 1. ST 已被使用（一次性）
# 2. ST 已过期（默认 10 秒）
# 3. Service ID 不匹配（注册时配置错误）
# 4. 时钟不同步（NTP 未配置）
```

### 8.2 Cookie 跨域问题

CAS 的 TGC Cookie 只能在同域名下共享。如果业务系统在不同子域：

```properties
# 方案1：设置父域名
cas.tgc.domain=.example.com

# 方案2：使用 JSONP 跨域获取用户信息
# 不推荐，存在安全风险
```

### 8.3 性能问题

```properties
# Ticket 缓存配置
cas.ticket.registry.core.cache.capacity=10000
cas.ticket.registry.core.cache.timeout=PT10M

# 服务注册缓存
cas.serviceRegistry.cache.schedule.repeat.interval=PT2M
```

## 结尾

CAS 协议是 SSO 领域的经典方案，它的核心思想是**「认证中心化，授权分散化」**。用户身份由 CAS Server 统一管理，而每个业务系统只需要验证 CAS 签发的票据即可。

但 CAS 也有其局限性：协议相对复杂、对移动端支持不友好。如果你正在构建面向移动端的系统，或者需要对接第三方登录，OAuth 2.0 可能是更好的选择。
