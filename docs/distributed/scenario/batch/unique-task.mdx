# 分布式唯一任务执行：乐观锁 + 分布式锁双保险

你有没有遇到过这种情况：定时任务部署了两台机器，结果两边同时跑，同一个任务执行了两次。数据重复了，优惠券重复发放了，日志重复写了好几条……

分布式环境下，保证**同一时刻只有一个节点执行同一个任务**，是分布式调度最核心的问题。

## 问题分析：为什么会有重复执行？

```
定时任务触发（每分钟执行一次）
    ↓
两个节点同时收到调度信号
    ↓
节点 A 执行任务，节点 B 也执行任务
    ↓
任务被重复执行
```

根本原因：**调度中心广播了任务触发信号，但多个执行节点都认为自己应该执行。**

## 方案一：抢任务模式（分布式锁）

最简单的方案：用分布式锁抢任务执行权。只有抢到锁的节点才能执行。

```java
public class DistributedTaskExecutor {
    
    @Autowired
    private RedissonClient redisson;
    
    public void executeTask(String taskId, Runnable task) {
        RLock lock = redisson.getLock("task:" + taskId);
        
        // tryLock：尝试获取锁，最多等待 0 秒，锁自动过期 10 秒
        boolean acquired = lock.tryLock(0, 10, TimeUnit.SECONDS);
        
        if (acquired) {
            try {
                // 抢到锁，执行任务
                task.run();
            } finally {
                // 任务执行完释放锁
                if (lock.isHeldByCurrentThread()) {
                    lock.unlock();
                }
            }
        } else {
            log.info("任务 {} 被其他节点抢走了", taskId);
        }
    }
}
```

### 分布式锁的问题

分布式锁看起来简单，但有几个隐藏的坑：

**问题一：锁过期时间设多久？**
```java
// ❌ 错误：锁过期时间太短，任务还没执行完就自动释放了
lock.tryLock(0, 5, TimeUnit.SECONDS);  // 任务要跑 10 秒，锁 5 秒就过期了

// ✅ 正确：根据任务预估时间 + buffer 设置
// 任务预估 10 秒，锁过期时间设为 30 秒（3 倍 buffer）
lock.tryLock(0, 30, TimeUnit.SECONDS);

// ✅ 更可靠：用看门狗自动续期
RLock lock = redisson.getLock("task:xxx");
// 不设置过期时间，Redisson 会自动续期（默认每 10 秒续期 30 秒）
lock.lock();
```

**问题二：任务执行超时**
```java
// 执行任务时，设置超时保护
public void executeWithTimeout(String taskId, Runnable task, long timeoutSeconds) {
    RLock lock = redisson.getLock("task:" + taskId);
    
    if (lock.tryLock(0, timeoutSeconds * 2, TimeUnit.SECONDS)) {
        try {
            Future<?> future = executor.submit(() -> {
                task.run();
            });
            
            future.get(timeoutSeconds, TimeUnit.SECONDS);
        } catch (TimeoutException e) {
            log.error("任务 {} 执行超时（{}s），强制中断", taskId, timeoutSeconds);
            future.cancel(true);
        } finally {
            if (lock.isHeldByCurrentThread()) {
                lock.forceUnlock();  // 强制释放
            }
        }
    }
}
```

## 方案二：乐观锁（数据库版本号）

用数据库的版本号字段实现乐观锁，不需要分布式锁。

```sql
-- 任务调度表
CREATE TABLE scheduled_task (
    id VARCHAR(64) PRIMARY KEY,      -- 任务 ID
    cron_expression VARCHAR(32),      -- cron 表达式
    status VARCHAR(16),              -- IDLE/RUNNING/SUCCESS/FAILED
    lock_version INT DEFAULT 0,     -- 乐观锁版本号
    lock_node VARCHAR(64),          -- 抢到锁的节点 ID
    lock_time BIGINT,                -- 加锁时间
    update_time BIGINT,              -- 更新时间
    INDEX idx_status_update (status, update_time)
);
```

```java
public class OptimisticLockTaskExecutor {
    
    @Autowired
    private JdbcTemplate jdbcTemplate;
    
    public boolean tryAcquire(String taskId) {
        String nodeId = getCurrentNodeId();
        
        // 乐观锁更新：只有当前版本号匹配才能更新成功
        String sql = """
            UPDATE scheduled_task
            SET status = 'RUNNING',
                lock_node = ?,
                lock_version = lock_version + 1,
                lock_time = ?
            WHERE id = ?
              AND status = 'IDLE'
              AND lock_version = ?
            """;
        
        long now = System.currentTimeMillis();
        int updated = jdbcTemplate.update(sql, nodeId, now, taskId, getCurrentVersion(taskId));
        
        return updated > 0;  // updated == 1 表示抢锁成功
    }
    
    public void release(String taskId, String status) {
        String sql = """
            UPDATE scheduled_task
            SET status = ?, lock_node = NULL, lock_version = lock_version + 1
            WHERE id = ?
            """;
        jdbcTemplate.update(sql, status, taskId);
    }
}
```

### 乐观锁的问题

乐观锁在高并发下会有**锁竞争问题**：大量节点同时尝试更新，但只有第一个成功，其他全部失败（白白浪费资源）。

## 方案三：双保险——分布式锁 + 乐观锁

生产环境推荐用**分布式锁做第一道关卡，乐观锁做兜底**。

```java
public class DoubleGuardTaskExecutor {
    
    @Autowired
    private RedissonClient redisson;
    @Autowired
    private JdbcTemplate jdbcTemplate;
    
    private final String taskId;
    
    public void execute(Runnable task) {
        // 第一步：分布式锁（快速过滤大部分节点）
        RLock lock = redisson.getLock("task:lock:" + taskId);
        if (!lock.tryLock(0, 10, TimeUnit.SECONDS)) {
            log.debug("分布式锁未抢到，跳过");
            return;
        }
        
        try {
            // 第二步：乐观锁（数据库层面再确认一次）
            if (!dbTryAcquire(taskId)) {
                log.debug("数据库乐观锁未抢到，跳过");
                return;
            }
            
            // 执行任务
            try {
                task.run();
                dbRelease(taskId, "SUCCESS");
            } catch (Exception e) {
                dbRelease(taskId, "FAILED");
                throw e;
            }
        } finally {
            if (lock.isHeldByCurrentThread()) {
                lock.unlock();
            }
        }
    }
    
    private boolean dbTryAcquire(String taskId) {
        String sql = """
            UPDATE scheduled_task
            SET status = 'RUNNING', lock_node = ?, lock_time = ?
            WHERE id = ? AND status = 'IDLE'
            """;
        return jdbcTemplate.update(sql, getNodeId(), System.currentTimeMillis(), taskId) > 0;
    }
    
    private void dbRelease(String taskId, String status) {
        String sql = "UPDATE scheduled_task SET status = ?, lock_node = NULL WHERE id = ?";
        jdbcTemplate.update(sql, status, taskId);
    }
}
```

## 方案四：任务分片（最推荐的方案）

把任务按执行节点分片，每个节点只执行自己负责的那部分。天然不会有重复执行问题。

```java
public class ShardingTaskExecutor {
    
    // 10 个节点，节点 0 只执行分片 0，节点 1 只执行分片 1...
    public void execute(String taskId, int totalShards, int shardIndex, Runnable task) {
        String nodeId = getCurrentNodeId();
        int nodeShard = calculateShard(nodeId, totalShards);
        
        if (nodeShard != shardIndex) {
            log.debug("分片 {} 不归本节点（{}）执行", shardIndex, nodeShard);
            return;
        }
        
        task.run();
    }
    
    // 节点 ID → 分片序号的映射（一致性哈希，保证节点变化时影响最小）
    private int calculateShard(String nodeId, int totalShards) {
        int hash = Math.abs(nodeId.hashCode());
        return hash % totalShards;
    }
}
```

## 各种方案对比

| 方案 | 实现复杂度 | 可靠性 | 适用场景 |
|---|---|---|---|
| 分布式锁 | 低 | 中（依赖锁可靠性） | 短任务 |
| 乐观锁 | 中 | 高（数据库保证） | 中等并发 |
| 双保险 | 高 | 最高 | 核心任务 |
| 任务分片 | 中 | 最高（天然去重） | 大量数据处理 |

## 总结

分布式唯一任务执行，关键是**让任务执行权和任务本身解耦**：

- **分布式锁**：适合短任务，锁可靠性是关键
- **乐观锁**：适合中等并发场景，数据库层面保证
- **双保险**：适合核心任务，两层保护
- **任务分片**：最适合大批量数据处理，天然无冲突

实际生产中，**任务分片 + 分布式锁** 是最推荐的组合：分片保证任务不重复，锁保证分片内同一时刻只有一个节点执行。
