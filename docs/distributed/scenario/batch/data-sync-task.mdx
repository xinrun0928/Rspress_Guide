# 数据同步任务：全量同步与增量同步方案

数据同步是分布式系统中最常见的任务之一：从 MySQL 同步到 Elasticsearch 做搜索，从业务库同步到数据仓库做分析，从主库同步到从库做读写分离……场景不同，方案也不同。

## 全量同步 vs 增量同步

| | 全量同步 | 增量同步 |
|---|---|---|
| 触发时机 | 首次同步、数据迁移 | 定时任务或实时 |
| 数据量 | 全量数据 | 仅变更数据 |
| 性能影响 | 大 | 小 |
| 复杂度 | 低 | 高 |
| 适用场景 | 首次初始化 | 日常数据同步 |

大多数数据同步系统，初始时做一次全量同步，之后持续做增量同步。

## 全量同步：简单直接的方案

### 基于 JDBC 分批导出

```java
public class FullSyncExecutor {
    
    @Autowired
    private DataSource sourceDataSource;
    
    public void syncToES(String tableName, String indexName) {
        // 1. 获取数据总数
        long totalCount = getTotalCount(tableName);
        int batchSize = 5000;
        int totalPages = (int) Math.ceil((double) totalCount / batchSize);
        
        log.info("全量同步 {}: 共 {} 条，分 {} 页", tableName, totalCount, totalPages);
        
        // 2. 分页查询并写入目标
        for (int page = 0; page < totalPages; page++) {
            List<Map<String, Object>> batch = queryBatch(tableName, page, batchSize);
            
            // 批量写入 ES
            bulkIndexToES(indexName, batch);
            
            log.info("已完成 {} / {} 页", page + 1, totalPages);
        }
        
        log.info("全量同步完成: {} 条数据", totalCount);
    }
    
    private List<Map<String, Object>> queryBatch(String tableName, int page, int batchSize) {
        String sql = String.format(
            "SELECT * FROM %s LIMIT %d OFFSET %d",
            tableName, batchSize, page * batchSize
        );
        return jdbcTemplate.queryForList(sql);
    }
}
```

:::warning 注意
分页查询时，OFFSET 过大会导致性能问题（MySQL 需要跳过大量行）。对于超大数据集，使用游标（keyset pagination）分页代替 OFFSET 分页。
:::

### 并行全量同步

为了加快速度，可以并行分页查询：

```java
public class ParallelFullSyncExecutor {
    
    private final int parallelism = 10;  // 并行度
    private final ExecutorService executor = Executors.newFixedThreadPool(10);
    
    public void parallelSync(String tableName, String indexName) {
        long totalCount = getTotalCount(tableName);
        int batchSize = 5000;
        int totalBatches = (int) Math.ceil((double) totalCount / batchSize);
        
        // 分配偏移量区间
        List<CompletableFuture<Void>> futures = new ArrayList<>();
        
        for (int batch = 0; batch < totalBatches; batch++) {
            int offset = batch * batchSize;
            CompletableFuture<Void> future = CompletableFuture.runAsync(() -> {
                syncBatch(tableName, indexName, offset, batchSize);
            }, executor);
            futures.add(future);
        }
        
        // 等待所有批次完成
        CompletableFuture.allOf(futures.toArray(new CompletableFuture[0])).join();
        log.info("全量同步完成");
    }
}
```

## 增量同步：如何捕获数据变更？

增量同步的核心问题是：**如何捕获数据库中的新增、修改和删除？**

### 方案一：基于时间戳（Timestamp）

最简单的方式：记录上次同步的时间戳，每次只同步时间戳大于该值的数据。

```java
public class TimestampBasedIncrementalSync {
    
    private volatile long lastSyncTimestamp;
    
    public void incrementalSync(String tableName, String indexName) {
        long currentTime = System.currentTimeMillis();
        
        // 查询上次同步之后变更的数据
        String sql = String.format(
            "SELECT * FROM %s WHERE update_time > ? AND update_time <= ?",
            tableName
        );
        
        List<Map<String, Object>> changedData = jdbcTemplate.queryForList(
            sql, new Object[]{lastSyncTimestamp, currentTime}
        );
        
        // 处理新增和修改（UPSERT）
        for (Map<String, Object> record : changedData) {
            upsertToES(indexName, record);
        }
        
        // 更新同步时间戳
        lastSyncTimestamp = currentTime;
        saveCheckpoint(lastSyncTimestamp);  // 持久化，防止重启丢失
    }
}
```

:::tip 经验
时间戳增量同步的问题：
- **无法捕获删除操作**：需要额外维护一张"删除记录表"
- **依赖 update_time 字段**：如果业务代码没有维护该字段，可能漏数据
- **不适用于 UPDATE 不更新时间戳的场景**：如"逻辑删除"

建议配合 CDC 工具处理。
:::

### 方案二：基于 Binlog（CDC）

Binlog 是 MySQL 的变更日志，是最可靠的增量同步数据源。

```java
// Canal 监听 Binlog，实现增量同步
@CanalListener(destination = "example", batch = true)
public class BinlogIncrementalSync {
    
    @Autowired
    private ElasticsearchRestTemplate esTemplate;
    
    @StreamListener(CanalEntry.INNER_BEFORE_ENTRY)
    public void onEvent(CanalEntry.Entry entry) {
        CanalEntry.RowChange rowChange = CanalEntry.RowChange.parseFrom(entry.getStoreValue());
        
        for (CanalEntry.RowData rowData : rowChange.getRowDatasList()) {
            String tableName = entry.getHeader().getTableName();
            CanalEntry.EventType eventType = rowChange.getEventType();
            
            switch (eventType) {
                case INSERT:
                    upsertToES(tableName, rowData.getAfterColumnsList());
                    break;
                case UPDATE:
                    updateToES(tableName, rowData.getAfterColumnsList());
                    break;
                case DELETE:
                    deleteFromES(tableName, rowData.getBeforeColumnsList());
                    break;
            }
        }
    }
}
```

### 方案三：基于 Debezium 的 CDC 方案

Debezium 是更现代的 CDC 工具，支持多种数据库：

```java
// Debezium Embedded 模式
public class DebeziumIncrementalSync {
    
    public void startSync() {
        Configuration config = Configuration.create()
            .with("connector.class", "io.debezium.connector.mysql.MySqlConnector")
            .with("database.hostname", "mysql-host")
            .with("database.port", "3306")
            .with("database.user", "debezium")
            .with("database.password", "password")
            .with("database.server.id", "184054")
            .with("database.server.name", "mysql-source")
            .with("table.include.list", "mydb.orders,mydb.products")
            .with("topic.prefix", "mysql")
            .with("schema.history.internal.kafka.bootstrap.servers", "kafka:9092")
            .with("schema.history.internal.kafka.topic", "schema-changes")
            .build();
        
        EmbeddedEngine engine = EmbeddedEngine.create()
            .using(config)
            .notifying(this::handleEvent)
            .build();
        
        Executors.newSingleThreadExecutor().submit(engine);
    }
    
    private void handleEvent(RecordCommit<SourceRecord> record) {
        SourceRecord src = record.value();
        Struct value = (Struct) src.value();
        Struct after = value.getStruct("after");
        
        // 处理变更数据
        processChange(src.topic(), extractData(after));
    }
}
```

## 同步一致性与幂等保证

增量同步最大的问题是**幂等性**：同一条数据可能被同步多次。

```java
public class IdempotentSyncProcessor {
    
    public void sync(String tableName, Map<String, Object> record) {
        String uniqueKey = getUniqueKey(tableName, record);
        String idempotentKey = tableName + ":" + uniqueKey;
        
        // 幂等键：基于主键的去重
        Boolean success = redisTemplate.opsForValue()
            .setIfAbsent(idempotentKey, "1", Duration.ofHours(24));
        
        if (!success) {
            log.debug("已处理过，跳过: {}", idempotentKey);
            return;
        }
        
        try {
            upsertToTarget(tableName, record);
        } catch (Exception e) {
            // 同步失败时清理幂等键，允许重试
            redisTemplate.delete(idempotentKey);
            throw e;
        }
    }
}
```

## 总结

数据同步方案选型：

| 场景 | 推荐方案 |
|---|---|
| 首次数据初始化 | 全量 JDBC 分批导出 |
| 定时增量同步（`≥` 5 分钟） | 时间戳 + 查询条件 |
| 准实时同步（秒级） | Canal/RocketMQ |
| 高可靠实时同步 | Debezium + Kafka |
| 多源数据汇聚 | Debezium + Kafka Connect |

核心原则：
1. **幂等优先**：任何同步操作都要考虑重复执行的情况
2. **检查点持久化**：同步进度要定期保存，支持断点恢复
3. **监控告警**：同步延迟、数据量突变、错误率都要监控
