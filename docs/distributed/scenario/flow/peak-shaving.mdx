# 接口流量削峰：消息队列 + 令牌桶 / 漏桶算法

双十一零点，百万用户同时下单。如果你的服务器直接承受这些流量，大概率会OOM。但如果把这些请求放进消息队列，服务器按自己的节奏慢慢处理，峰值就被削平了。

这就是削峰填谷的思想。

## 为什么需要削峰？

### 没有削峰的问题

### 削峰策略对比

| 策略 | 原理 | 优点 | 缺点 |
|---|---|---|---|
| 消息队列 | 异步缓冲 | 简单可靠，削峰能力强 | 增加延迟 |
| 令牌桶 | 按固定速率放行 | 支持突发流量 | 需要预热 |
| 漏桶 | 按固定速率处理 | 流量平滑 | 不支持突发 |
| 限流 | 直接拒绝多余请求 | 保护系统 | 用户体验差 |

## 消息队列削峰

### 订单削峰架构

```java
/**
 * 订单服务 - 生产者
 * 将订单请求放入消息队列，实现削峰
 */
@Service
public class OrderProducerService {

    private final RocketMQTemplate rocketMQTemplate;

    /**
     * 提交订单（异步削峰）
     */
    public Response submitOrder(OrderRequest request) {
        // 1. 订单预校验
        if (!validateOrder(request)) {
            return Response.error("订单校验失败");
        }

        // 2. 生成订单号
        String orderId = IdGenerator.nextOrderId();

        // 3. 构建消息
        OrderMessage message = new OrderMessage()
            .setOrderId(orderId)
            .setUserId(request.getUserId())
            .setItems(request.getItems())
            .setTotalAmount(request.getTotalAmount())
            .setSubmitTime(System.currentTimeMillis());

        // 4. 发送到消息队列（同步发送，保证不丢消息）
        try {
            rocketMQTemplate.asyncSend(
                "order-topic:create-order",  // topic:tag
                message,
                new SendCallback() {
                    @Override
                    public void onSuccess(SendResult result) {
                        log.info("订单消息发送成功: orderId={}", orderId);
                    }

                    @Override
                    public void onException(Throwable e) {
                        log.error("订单消息发送失败: orderId={}", orderId, e);
                        // 发送失败，需要补偿或告警
                        handleSendFailure(orderId, message);
                    }
                }
            );

            // 5. 立即返回（告诉用户订单已提交）
            return Response.success()
                .setMessage("订单已提交，请等待处理")
                .setData(new OrderSubmitResult()
                    .setOrderId(orderId)
                    .setStatus("PENDING"));

        } catch (Exception e) {
            log.error("提交订单异常", e);
            throw new BusinessException("系统繁忙，请稍后重试");
        }
    }

    /**
     * 同步确认下单（适合小流量场景）
     */
    public Response submitOrderSync(OrderRequest request) {
        String orderId = IdGenerator.nextOrderId();

        OrderMessage message = new OrderMessage()
            .setOrderId(orderId)
            .setUserId(request.getUserId())
            .setItems(request.getItems())
            .setTotalAmount(request.getTotalAmount());

        // 同步发送，等待处理结果
        SendResult result = rocketMQTemplate.syncSend(
            "order-topic:create-order",
            message,
            3000  // 超时时间
        );

        if ("SEND_OK".equals(result.getSendStatus())) {
            return Response.success()
                .setData(new OrderSubmitResult()
                    .setOrderId(orderId)
                    .setStatus("SUCCESS"));
        } else {
            return Response.error("下单失败");
        }
    }
}
```

### 订单消费者 - 慢速处理

```java
/**
 * 订单服务 - 消费者
 * 按照服务器的处理能力消费消息
 */
@Component
@RocketMQMessageListener(
    topic = "order-topic",
    tag = "create-order",
    consumerGroup = "order-consumer-group",
    consumeThreadMin = 10,    // 最小消费线程
    consumeThreadMax = 50,     // 最大消费线程
    consumeMode = ConsumeMode.CONCURRENTLY,  // 并发消费
    messageModel = MessageModel.CLUSTERING    // 集群模式
)
public class OrderConsumerService implements RocketMQListener<OrderMessage> {

    private final OrderService orderService;

    @Override
    public void onMessage(OrderMessage message) {
        try {
            // 1. 幂等检查
            if (isDuplicateMessage(message.getOrderId())) {
                log.info("重复订单，跳过: orderId={}", message.getOrderId());
                return;
            }

            // 2. 业务处理
            ProcessResult result = orderService.processOrder(message);

            if (result.isSuccess()) {
                log.info("订单处理成功: orderId={}", message.getOrderId());
            } else {
                // 3. 处理失败，重试
                handleProcessFailure(message, result.getErrorCode());
            }

        } catch (Exception e) {
            log.error("处理订单异常: orderId={}", message.getOrderId(), e);
            // 消息会触发重试
            throw new RuntimeException("处理失败，等待重试");
        }
    }

    /**
     * 重复消息检查
     */
    private boolean isDuplicateMessage(String orderId) {
        String key = "order:processed:" + orderId;
        return redisTemplate.hasKey(key);
    }
}
```

## 令牌桶算法

### 算法原理

```
┌──────────────────────────────────────────────────────────────────┐
│                      令牌桶算法原理                               │
├──────────────────────────────────────────────────────────────────┤
│                                                                   │
│                    ┌─────────────┐                               │
│                    │   令牌桶    │  ← 容量 N 个令牌              │
│                    │    ○○○○    │                               │
│                    │   ○○○○○○   │                               │
│                    │  ○○○○○○○○  │                               │
│                    └──────┬──────┘                               │
│                           │                                       │
│              ┌────────────┼────────────┐                        │
│              │  以固定速率 r 放入令牌       │                        │
│              └─────────────────────────┘                        │
│                                                                   │
│  请求到达:                                                        │
│  ┌─────┐                                                         │
│  │请求1│ → 检查桶中是否有令牌 → 有 → 获取令牌 → 放行              │
│  └─────┘                                                         │
│  ┌─────┐                                                         │
│  │请求2│ → 检查桶中是否有令牌 → 无 → 拒绝                        │
│  └─────┘                                                         │
│                                                                   │
│  特性:                                                            │
│  - 支持突发流量（桶中有令牌即可通过）                             │
│  - 令牌以固定速率生成                                              │
│  - 桶满时，令牌溢出                                                │
│                                                                   │
└──────────────────────────────────────────────────────────────────┘
```

### Java 实现

```java
/**
 * 令牌桶限流器
 */
public class TokenBucketRateLimiter {

    private final double rate;              // 每秒生成令牌数
    private final int capacity;              // 桶容量
    private final AtomicLong tokens;        // 当前令牌数
    private volatile long lastRefillTime;   // 上次补充时间

    public TokenBucketRateLimiter(double rate, int capacity) {
        this.rate = rate;
        this.capacity = capacity;
        this.tokens = new AtomicLong(capacity);
        this.lastRefillTime = System.nanoTime();
    }

    /**
     * 尝试获取令牌
     */
    public boolean tryAcquire() {
        return tryAcquire(1);
    }

    /**
     * 尝试获取指定数量令牌
     */
    public boolean tryAcquire(int permits) {
        // 1. 补充令牌
        refill();

        // 2. 尝试获取令牌
        while (true) {
            long current = tokens.get();
            long newValue = current - permits;

            if (newValue < 0) {
                // 令牌不足
                return false;
            }

            if (tokens.compareAndSet(current, newValue)) {
                return true;
            }

            // CAS 失败，重试
            Thread.yield();
        }
    }

    /**
     * 非阻塞获取，获取不到立即返回
     */
    public boolean tryAcquireNonBlock() {
        refill();
        long current = tokens.get();
        return current > 0 && tokens.compareAndSet(current, current - 1);
    }

    /**
     * 补充令牌
     */
    private void refill() {
        long now = System.nanoTime();
        long elapsed = now - lastRefillTime;

        // 计算应该补充的令牌数
        double tokensToAdd = (elapsed / 1_000_000_000.0) * rate;

        if (tokensToAdd >= 1) {
            long tokensToAddLong = (long) tokensToAdd;

            while (true) {
                long current = tokens.get();
                long newValue = Math.min(capacity, current + tokensToAddLong);

                if (tokens.compareAndSet(current, newValue)) {
                    lastRefillTime = now;
                    break;
                }
            }
        }
    }

    /**
     * 获取当前令牌数（用于监控）
     */
    public long getAvailableTokens() {
        refill();
        return tokens.get();
    }
}
```

### Guava RateLimiter 使用

```java
@Service
public class RateLimitedService {

    // 每秒允许 1000 个请求
    private final RateLimiter rateLimiter = RateLimiter.create(1000.0);

    /**
     * 阻塞方式限流
     * 获取不到令牌时，会阻塞等待
     */
    public Response handleRequest1(Request request) {
        // 阻塞获取令牌
        rateLimiter.acquire();

        return doProcess(request);
    }

    /**
     * 非阻塞方式限流
     * 获取不到令牌时，返回 false
     */
    public Response handleRequest2(Request request) {
        // 非阻塞获取
        if (!rateLimiter.tryAcquire()) {
            return Response.error("系统繁忙，请稍后重试", 429);
        }

        return doProcess(request);
    }

    /**
     * 带超时的非阻塞获取
     */
    public Response handleRequest3(Request request, Duration timeout) {
        if (!rateLimiter.tryAcquire(timeout.toMillis(), TimeUnit.MILLISECONDS)) {
            return Response.error("请求超时，请稍后重试", 429);
        }

        return doProcess(request);
    }

    /**
     * 预热模式
     * 适用于流量逐渐上升的场景
     */
    private RateLimiter createWarmUpLimiter() {
        // 初始 qps=10，5 秒预热到 qps=100
        return RateLimiter.create(
            RateLimiter.create(100.0, 5, TimeUnit.SECONDS),
            10.0,
            5,
            TimeUnit.SECONDS
        );
    }
}
```

## 漏桶算法

### 算法原理

```
┌──────────────────────────────────────────────────────────────────┐
│                      漏桶算法原理                                │
├──────────────────────────────────────────────────────────────────┤
│                                                                   │
│         请求                                                          │
│           │                                                           │
│           ▼                                                           │
│    ┌─────────────┐                                                 │
│    │   漏桶      │  ← 桶容量 N                                   │
│    │  ▓▓▓▓▓     │                                                 │
│    │ ▓▓▓▓▓▓▓▓   │                                                 │
│    └──────┬──────┘                                                 │
│           │  以固定速率 r 漏出                                      │
│           ▼                                                          │
│       请求处理                                                         │
│                                                                   │
│  特性:                                                            │
│  - 无论请求多快，输出速率恒定                                      │
│  - 桶满时，请求溢出（被拒绝）                                       │
│  - 流量平滑，不支持突发                                            │
│                                                                   │
└──────────────────────────────────────────────────────────────────┘
```

### Java 实现

```java
/**
 * 漏桶限流器
 */
public class LeakyBucketRateLimiter {

    private final int capacity;              // 桶容量
    private final double leakRate;            // 每秒漏出数量
    private final AtomicLong water;           // 当前水量
    private volatile long lastLeakTime;      // 上次漏水时间

    public LeakyBucketRateLimiter(int capacity, double leakRate) {
        this.capacity = capacity;
        this.leakRate = leakRate;
        this.water = new AtomicLong(0);
        this.lastLeakTime = System.currentTimeMillis();
    }

    /**
     * 尝试加水（请求进入漏桶）
     */
    public synchronized boolean tryAdd() {
        leak();  // 先漏水

        long current = water.get();
        if (current >= capacity) {
            // 水已满，拒绝
            return false;
        }

        water.incrementAndGet();
        return true;
    }

    /**
     * 漏水
     */
    private void leak() {
        long now = System.currentTimeMillis();
        long elapsed = now - lastLeakTime;

        if (elapsed < 1000) {
            return;  // 不到 1 秒，不漏
        }

        // 计算漏水量
        long leaked = (long) (elapsed / 1000.0 * leakRate);

        if (leaked > 0) {
            while (true) {
                long current = water.get();
                long newValue = Math.max(0, current - leaked);

                if (water.compareAndSet(current, newValue)) {
                    lastLeakTime = now;
                    break;
                }
            }
        }
    }
}
```

### 与令牌桶对比

| 特性 | 令牌桶 | 漏桶 |
|---|---|---|
| 输出速率 | 可突发（上限为桶容量） | 平滑（恒定速率） |
| 突发能力 | 支持 | 不支持 |
| 流量形状 | 允许一定突发 | 完全平滑 |
| 适用场景 | 流量有峰值 | 需要严格限流 |

## 组合使用：消息队列 + 令牌桶

### 完整削峰架构

```java
/**
 * 组合限流：消息队列削峰 + 令牌桶限流
 */
@Service
public class PeakShavingService {

    private final RocketMQTemplate rocketMQTemplate;
    private final TokenBucketRateLimiter rateLimiter;
    private final OrderService orderService;

    /**
     * 提交订单 - 削峰处理
     */
    public Response submitOrderWithShaving(OrderRequest request) {
        // 1. 限流检查
        if (!rateLimiter.tryAcquire()) {
            return Response.error("系统繁忙", 429);
        }

        // 2. 发送到消息队列
        String orderId = IdGenerator.nextOrderId();
        OrderMessage message = buildMessage(orderId, request);

        rocketMQTemplate.asyncSend("order-topic:create-order", message);

        // 3. 返回处理中状态
        return Response.success()
            .setData(new OrderSubmitResult()
                .setOrderId(orderId)
                .setStatus("PROCESSING"));
    }

    /**
     * 消息消费 - 逐条处理
     */
    @KafkaListener(topics = "order-topic", groupId = "order-consumer")
    public void consumeOrder(OrderMessage message) {
        // 限流检查（保护下游服务）
        if (!rateLimiter.tryAcquire()) {
            // 放回队列，稍后重试
            throw new RequeueException("限流中，稍后重试");
        }

        orderService.processOrder(message);
    }
}
```

## 总结

削峰填谷的核心是**用空间换时间**：

| 组件 | 作用 |
|---|---|
| 消息队列 | 缓存请求，平滑流量 |
| 令牌桶 | 控制消费速率，支持突发 |
| 漏桶 | 严格限流，流量平滑 |
| 限流器 | 保护系统不被压垮 |

削峰策略的选择：

| 场景 | 推荐策略 |
|---|---|
| 允许延迟 | 消息队列削峰 |
| 不允许延迟 | 令牌桶 + 拒绝 |
| 需要精确限流 | 漏桶 |
| 保护下游 | 双层限流 |

记住：**削峰不是不用限流**，而是把限流的粒度从「拒绝请求」变成「延迟处理」。最终还是要控制总处理量。
