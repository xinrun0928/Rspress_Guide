# 分布式场景面试高频问题汇总

分布式系统是 Java 后端面试的重灾区。本章节汇总高频面试题，帮你理清面试思路。

## 一、分布式理论

### 问题 1：CAP 理论是什么？为什么只能三选二？

**面试官想考察**：你对分布式系统基础理论的掌握程度，以及能否灵活运用。

**标准答案**：

CAP 理论由 Eric Brewer 提出，指分布式系统最多只能同时满足以下三个特性中的两个：

- **C（Consistency，一致性）**：所有节点在同一时刻看到相同的数据
- **A（Availability，可用性）**：每个请求都能在合理时间内得到响应
- **P（Partition Tolerance，分区容错）**：系统能在网络分区故障时继续运行

由于网络分区不可避免（这是分布式系统的基本假设），所以实际上是在 C 和 A 之间做权衡：

- **CP 系统**：优先保证一致性。如 ZooKeeper、etcd、HBase、Redis Cluster。发生分区时，为了保持一致，会牺牲可用性（拒绝服务）。
- **AP 系统**：优先保证可用性。如 Cassandra、Eureka、DynamoDB。发生分区时，为了保持可用，会牺牲一致性（返回可能过期的数据）。

**加分回答**：
"CAP 理论中的 C 和 A 不是非此即彼的关系，而是程度问题。实际系统通常在一致性和可用性之间取一个平衡——比如最终一致性、读我修复、写时复制等技术手段。BASE 理论就是对 AP 方案的系统化描述。"

### 问题 2：2PC 和 TCC 的区别是什么？

**标准答案**：

| 维度 | 2PC | TCC |
|---|---|---|
| 协议层 | 数据库 XA 协议 | 业务层面 |
| 侵入性 | 数据库原生支持 | 需业务代码改造 |
| 锁粒度 | 全局锁，阻塞时间长 | 资源预留，不阻塞 |
| 性能 | 较低 | 较高 |
| 一致性 | 强一致 | 最终一致 |
| 问题 | 协调者单点、阻塞 | 空回滚、悬挂、幂等 |

**核心区别**：2PC 是数据库层面的协议，靠数据库的 XA 事务实现；TCC 是业务层面的补偿，靠 Try-Confirm-Cancel 三个接口实现。

**追问**：TCC 的三个经典问题怎么解决？
- 空回滚：记录 Try 阶段状态，Cancel 时检查
- 悬挂：TCC 服务幂等，Try 执行后不再允许 Cancel
- 幂等：防重表（Redis SETNX）或数据库唯一约束

### 问题 3：Paxos 和 Raft 的核心区别是什么？

**标准答案**：

Raft 的设计初衷就是**让 Paxos 更容易理解**。两者在本质上解决的问题相同，但设计思路不同：

| 维度 | Paxos | Raft |
|---|---|---|
| 可理解性 | 极难，被认为不可实现 | 相对容易，有明确流程 |
| Leader 选举 | 通过 Prepare 阶段隐式选出 | 显式的 Leader 选举（Term + 随机超时） |
| 日志复制 | 允许日志空洞，通过 Paxos 编号保证 | 要求日志连续（Log Matching 属性） |
| 成员变更 | 需要特殊处理 | 联合共识（Joint Consensus）或单步变更 |

**加分回答**：
"Paxos 的论文里只描述了共识算法的核心，没有给出工程实现细节。Raft 把 Paxos 的概念拆成了 Leader 选举、日志复制、安全性三个独立的部分，并给出了明确的算法流程。这是 Raft 最大的价值——让共识算法可以被真正实现。"

## 二、分布式事务

### 问题 4：分布式事务有哪些解决方案？各自的优缺点？

**标准答案**：

| 方案 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| 2PC/XA | 强一致，实现简单 | 性能差，协调者单点 | 对一致性要求极高的金融系统 |
| TCC | 性能好，无全局锁 | 侵入性强，需改业务代码 | 高并发电商交易 |
| Saga | 长事务友好，可视化编排 | 无隔离性，补偿逻辑复杂 | 长流程业务 |
| 可靠消息 | 实现简单 | 最终一致，有延迟 | 非核心异步操作 |
| 最大努力通知 | 实现最简单 | 一致性最弱 | 非核心结果通知 |

### 问题 5：Seata AT 模式的 undolog 是怎么工作的？

**标准答案**：

AT 模式的 undolog 工作流程：

1. **Before Image（前置镜像）**：执行 SQL 前，查询当前数据行，保存为"前置镜像"
2. **执行业务 SQL**：执行业务操作
3. **After Image（后置镜像）**：执行 SQL 后，再次查询当前数据行，保存为"后置镜像"
4. **全局事务提交**：TC 通知所有分支提交，删除 undolog
5. **全局事务回滚**：TC 通知所有分支回滚，用前置镜像覆盖当前数据

```java
// Seata AT 模式原理（伪代码）
// 执行 UPDATE SET amount = amount - 100 WHERE id = 1

// Before Image: SELECT amount FROM account WHERE id = 1 → amount = 1000
// 执行业务: UPDATE SET amount = 900 WHERE id = 1
// After Image: SELECT amount FROM account WHERE id = 1 → amount = 900

// 回滚时: UPDATE SET amount = 1000 WHERE id = 1 (用前置镜像覆盖)
// 无需写补偿 SQL，Seata 自动生成！
```

## 三、分布式锁

### 问题 6：Redis 分布式锁有哪些坑？怎么解决？

**标准答案**：

Redis 分布式锁的四个经典问题：

**问题一：SETNX + EXPIRE 非原子性**
- ❌ `SETNX key value` + `EXPIRE key 10`（两步操作，之间可能崩溃）
- ✅ `SET key value NX EX 10`（Redis 2.6.12 后支持，单条命令原子执行）

**问题二：锁过期，业务未完成**
- 解决：Redisson 看门狗（Watchdog）自动续期，默认每 10 秒续期 30 秒

**问题三：主从切换后锁丢失（RedLock 争议）**
- 解决：RedLock（多 Redis 实例独立加锁），但实际上 RedLock 有争议（Martin Kleppmann 反对）
- 稳妥方案：zookeeper/etcd 分布式锁（CP 保证）

**问题四：可重入性**
- 解决：Redisson RLock 内部维护计数器 + 线程标识

## 四、分布式协调

### 问题 7：ZooKeeper 的 Watch 机制是怎么工作的？

**标准答案**：

Watch 是 ZooKeeper 最重要的特性之一，它是一个**一次性触发器**：

1. 客户端注册 Watch：`getData("/lock/node", watch)`
2. 数据变更时，ZooKeeper 通知客户端：`NodeDataChanged` 事件
3. **Watch 触发后自动失效**，需要客户端重新注册

Watch 机制保证了：
- **轻量级**：ZooKeeper 只存储通知类型（事件），不存储变化的具体数据
- **可靠推送**：ZooKeeper 保证 Watch 事件在客户端断开重连后不会丢失（ZooKeeper 会为断开连接的客户端保留 Watch 状态）

**追问**：Watch 为什么是一次性的，而不是永久的？
- 永久 Watch 会导致 ZooKeeper 存储大量状态，性能下降
- 一次性的设计让 ZooKeeper 保持轻量，同时客户端可以随时重新注册

## 五、服务治理

### 问题 8：Hystrix 和 Sentinel 的区别？

**标准答案**：

| 维度 | Sentinel | Hystrix |
|---|---|---|
| 隔离策略 | 信号量 / 线程池 | 线程池（默认） |
| 熔断策略 | 慢调用比例 / 异常比例 / 异常数 | 错误率 |
| 限流维度 | 服务 / 接口 / 参数 | 服务 |
| 规则配置 | 运行时推送 | 重启生效 |
| 社区活跃度 | 活跃（阿里开源） | 停止维护（Hystrix 进入维护模式） |

**推荐答案**：
"生产环境推荐 Sentinel。相比 Hystrix，Sentinel 功能更丰富，性能更好，社区也更活跃。Hystrix 已经停止维护了。"

## 六、综合场景

### 问题 9：如何设计一个分布式 ID 生成服务？

**标准答案**：

这道题考察候选人对多种 ID 生成方案的理解，以及实际工程能力。

**方案一：雪花算法（Snowflake）**
- 64 位：1 位符号 + 41 位时间戳 + 10 位机器 ID + 12 位序列号
- 优点：趋势递增，性能高
- 问题：依赖机器时钟，时钟回拨会导致 ID 冲突

**方案二：批量号段**
- 从数据库批量获取 ID 段（如 1000 个），本地内存分配
- 优点：性能极高，可用性好
- 缺点：ID 不严格连续

**方案三：数据库自增 + 批量获取**
- 适合小规模（`≤` 10 个分表）

**追问**：时钟回拨怎么处理？
- 美团 Leaf：使用"双 buffer + RingBuffer"预生成 ID，并对时钟回拨做了特殊处理
- 滴滴 TinyID：基于号段 + 奇偶分段规避时钟问题

## 总结

分布式面试题的核心规律：

1. **理论题**重在理解 CAP、BASE、Paxos/Raft 等核心概念
2. **方案题**重在权衡取舍，清楚每种方案的 trade-off
3. **场景题**重在结合实际，如 Seata、Redis 分布式锁、Sentinel 等
4. **追问题**最见功力，原理层面的追问（如 TCC 三个问题、Watch 机制）最能区分候选人水平

面试官最喜欢的回答方式：**先用简洁的语言给出标准答案，再主动补充"但是"后面的 trade-off 和适用场景，最后给出自己的推荐**。
