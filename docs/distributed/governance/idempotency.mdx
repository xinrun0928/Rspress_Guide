# 幂等性设计

用户点击了三次「提交订单」按钮，账户被扣了三次钱。用户刷新了页面，订单被创建了两次。凌晨的网络抖动，让库存服务收到了两条「扣减库存」的请求，商品超卖了。

这些都是幂等性（Idempotency）缺失导致的灾难。

幂等性是分布式系统中最容易被忽视，但后果最严重的特性之一。它解决的问题是：**同一个请求被执行多次，结果和执行一次是一样的。**

## 什么是幂等性

幂等性源自数学概念：一个操作执行一次和执行多次，效果相同。

```yaml
# HTTP 方法的幂等性
GET     - 幂等：查询不会改变资源状态，多次查询结果相同
PUT     - 幂等：更新操作，用相同数据多次 PUT，结果相同
DELETE  - 幂等：删除操作，删除一次和删除多次，效果相同
POST    - 非幂等：创建操作，多次 POST 会创建多个资源
PATCH   - 非幂等：部分更新，结果取决于更新内容
```

但在实际业务中，「幂等」的定义要更宽泛一些：

```yaml
# 业务层面的幂等定义
场景：创建订单
- 严格幂等：同一个请求执行多次，只创建一个订单
- 业务幂等：同一个用户同一时间只创建一个订单（不会出现重复订单）

场景：扣减库存
- 严格幂等：同一个请求执行多次，只扣减一次库存
- 业务幂等：库存不会变成负数（不会出现超卖）
```

## 哪些场景需要幂等

```yaml
# 必须保证幂等的场景
1. 支付类操作
   - 支付宝/微信支付回调
   - 银行卡扣款
   - 积分扣除
   - 优惠券核销

2. 订单类操作
   - 创建订单
   - 取消订单
   - 修改订单状态

3. 库存类操作
   - 扣减库存
   - 回滚库存

4. 消息确认类操作
   - 消息消费确认
   - 定时任务执行

5. 第三方回调
   - 支付成功回调
   - 推送回调
   - webhook
```

## 幂等实现方案

### 方案一：唯一标识（最常用）

给每个请求分配一个唯一 ID，服务端根据 ID 判断是否已处理。

```java
// 唯一标识幂等实现
@Service
public class IdempotentService {

    @Autowired
    private RedisTemplate<String, String> redisTemplate;

    private static final String IDEMPOTENT_PREFIX = "idempotent:";
    private static final long EXPIRE_SECONDS = 86400;  // 24 小时

    /**
     * 执行幂等操作
     * @param idempotentKey 幂等标识（通常由客户端生成）
     * @param supplier 实际业务逻辑
     * @return 业务结果
     */
    public <T> T execute(String idempotentKey, Supplier<T> supplier) {
        String key = IDEMPOTENT_PREFIX + idempotentKey;

        // Step 1: 检查是否已处理
        String existing = redisTemplate.opsForValue().get(key);
        if (existing != null) {
            log.info("请求已处理，直接返回: key={}", idempotentKey);
            return parseResult(existing);
        }

        // Step 2: 尝试加锁，防止并发重复处理
        Boolean locked = redisTemplate.opsForValue()
                .setIfAbsent(key + ":lock", "1", 30, TimeUnit.SECONDS);

        if (Boolean.FALSE.equals(locked)) {
            // 另一个请求正在处理，等待
            waitForProcessing(key);
        }

        try {
            // Step 3: 执行实际业务逻辑
            T result = supplier.get();

            // Step 4: 存储结果
            redisTemplate.opsForValue().set(key, serializeResult(result), EXPIRE_SECONDS, TimeUnit.SECONDS);

            return result;
        } finally {
            // Step 5: 释放锁
            redisTemplate.delete(key + ":lock");
        }
    }

    // 等待另一个请求处理完成
    private <T> T waitForProcessing(String key) {
        for (int i = 0; i < 30; i++) {
            String result = redisTemplate.opsForValue().get(key);
            if (result != null) {
                return parseResult(result);
            }
            try {
                Thread.sleep(100);
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
                throw new RuntimeException("等待被中断");
            }
        }
        throw new RuntimeException("处理超时");
    }
}

// 使用示例
@Service
public class OrderService {

    @Autowired
    private IdempotentService idempotentService;

    public Order createOrder(CreateOrderRequest request) {
        // 生成幂等 ID（客户端传入或从业务数据生成）
        String idempotentKey = generateIdempotentKey(request);

        return idempotentService.execute(idempotentKey, () -> {
            // 实际创建订单的逻辑
            return doCreateOrder(request);
        });
    }

    private String generateIdempotentKey(CreateOrderRequest request) {
        // 方案一：使用客户端传入的幂等 ID
        // return request.getIdempotentId();

        // 方案二：从业务数据生成
        return String.format("create_order:%s:%d",
                request.getUserId(),
                request.getRequestId());
    }
}
```

### 方案二：数据库唯一索引

利用数据库唯一索引防止重复插入。

```java
// 数据库唯一索引幂等
@Entity
@Table(name = "orders", indexes = {
    // 业务唯一索引
    @Index(name = "idx_user_request", columnList = "userId, requestId", unique = true)
})
public class Order {

    @Id
    @GeneratedValue
    private Long id;

    @Column(nullable = false)
    private Long userId;

    @Column(nullable = false)
    private String requestId;  // 客户端请求 ID

    @Column(nullable = false)
    private BigDecimal amount;

    // ...
}

// Service 层
@Service
public class OrderRepository {

    @Autowired
    private OrderJpaRepository orderRepository;

    @Transactional
    public Order createOrder(CreateOrderRequest request) {
        // 检查是否已存在
        Optional<Order> existing = orderRepository.findByUserIdAndRequestId(
                request.getUserId(), request.getRequestId());

        if (existing.isPresent()) {
            log.info("订单已存在，返回现有订单: orderId={}", existing.get().getId());
            return existing.get();
        }

        // 创建新订单
        Order order = new Order();
        order.setUserId(request.getUserId());
        order.setRequestId(request.getRequestId());
        order.setAmount(request.getAmount());
        // ...
        return orderRepository.save(order);
    }
}

// Repository
@Repository
public interface OrderJpaRepository extends JpaRepository<Order, Long> {

    Optional<Order> findByUserIdAndRequestId(Long userId, String requestId);

    // 使用数据库唯一约束时，捕获异常
    @Transactional
    default Order saveOrReturnExisting(CreateOrderRequest request) {
        try {
            return save(convertToEntity(request));
        } catch (DataIntegrityViolationException e) {
            // 唯一键冲突，返回已有记录
            return findByUserIdAndRequestId(request.getUserId(), request.getRequestId())
                    .orElseThrow();
        }
    }
}
```

### 方案三：状态机控制

对于有状态流转的操作，通过状态机保证幂等。

```java
// 订单状态机
public enum OrderStatus {
    CREATED,       // 已创建
    PAID,          // 已支付
    SHIPPED,       // 已发货
    COMPLETED,     // 已完成
    CANCELLED,     // 已取消
    REFUNDED       // 已退款
}

// 状态机转移规则
@Service
public class OrderStatusService {

    // 状态转移矩阵
    private static final Map<OrderStatus, Set<OrderStatus>> ALLOWED_TRANSITIONS = Map.of(
            OrderStatus.CREATED, Set.of(OrderStatus.PAID, OrderStatus.CANCELLED),
            OrderStatus.PAID, Set.of(OrderStatus.SHIPPED, OrderStatus.REFUNDED),
            OrderStatus.SHIPPED, Set.of(OrderStatus.COMPLETED, OrderStatus.REFUNDED),
            OrderStatus.COMPLETED, Set.of(),
            OrderStatus.CANCELLED, Set.of(),
            OrderStatus.REFUNDED, Set.of()
    );

    @Transactional
    public boolean payOrder(Long orderId, Long userId, BigDecimal amount) {
        Order order = orderRepository.findById(orderId)
                .orElseThrow(() -> new OrderNotFoundException(orderId));

        // 幂等检查：如果是已支付状态，直接返回成功
        if (order.getStatus() == OrderStatus.PAID) {
            log.info("订单已支付，直接返回: orderId={}", orderId);
            return true;
        }

        // 状态检查：只有 CREATED 状态才能支付
        if (order.getStatus() != OrderStatus.CREATED) {
            throw new OrderStatusException("订单状态不允许支付: " + order.getStatus());
        }

        // 执行支付
        order.setStatus(OrderStatus.PAID);
        order.setPaidAt(LocalDateTime.now());
        order.setPaidAmount(amount);
        orderRepository.save(order);

        return true;
    }

    @Transactional
    public boolean cancelOrder(Long orderId, Long userId) {
        Order order = orderRepository.findById(orderId)
                .orElseThrow(() -> new OrderNotFoundException(orderId));

        // 幂等检查
        if (order.getStatus() == OrderStatus.CANCELLED) {
            log.info("订单已取消，直接返回: orderId={}", orderId);
            return true;
        }

        // 状态检查
        if (!ALLOWED_TRANSITIONS.get(order.getStatus()).contains(OrderStatus.CANCELLED)) {
            throw new OrderStatusException("订单状态不允许取消: " + order.getStatus());
        }

        // 执行取消
        order.setStatus(OrderStatus.CANCELLED);
        order.setCancelledAt(LocalDateTime.now());
        orderRepository.save(order);

        return true;
    }
}
```

### 方案四：乐观锁

使用版本号实现乐观锁，防止并发更新。

```java
// 乐观锁实现
@Entity
@Table(name = "products")
public class Product {

    @Id
    private Long id;

    private String name;

    private Integer stock;  // 库存

    @Version  // 乐观锁版本号
    private Long version;

    // 扣减库存方法
    public boolean reduceStock(int quantity) {
        if (this.stock < quantity) {
            return false;
        }
        this.stock -= quantity;
        return true;
    }
}

// Service 层
@Service
public class ProductService {

    @Autowired
    private ProductRepository productRepository;

    /**
     * 扣减库存（乐观锁版）
     * 当库存不足时重试最多 3 次
     */
    @Transactional
    public boolean reduceStock(Long productId, int quantity) {
        int retryCount = 0;
        int maxRetries = 3;

        while (retryCount < maxRetries) {
            Optional<Product> optional = productRepository.findById(productId);
            if (optional.isEmpty()) {
                throw new ProductNotFoundException(productId);
            }

            Product product = optional.get();
            if (!product.reduceStock(quantity)) {
                throw new InsufficientStockException(productId);
            }

            try {
                productRepository.save(product);
                return true;
            } catch (OptimisticLockingFailureException e) {
                // 版本冲突，重试
                retryCount++;
                log.warn("库存更新冲突，重试: productId={}, retry={}", productId, retryCount);
                if (retryCount >= maxRetries) {
                    throw new ConcurrentUpdateException("并发更新失败，请重试");
                }
            }
        }
        return false;
    }

    /**
     * 扣减库存（数据库层面实现，更高效）
     */
    @Modifying
    @Query("UPDATE Product p SET p.stock = p.stock - :quantity, p.version = p.version + 1 " +
           "WHERE p.id = :productId AND p.stock >= :quantity AND p.version = :version")
    int reduceStockAtomic(@Param("productId") Long productId,
                          @Param("quantity") int quantity,
                          @Param("version") Long version);
}
```

### 方案五：分布式锁

使用分布式锁确保同一时间只有一个请求在处理。

```java
// 分布式锁幂等实现
@Service
public class DistributedLockIdempotent {

    @Autowired
    private RedissonClient redissonClient;

    /**
     * 使用分布式锁保证幂等
     */
    public <T> T executeWithLock(String lockKey, Supplier<T> supplier) {
        RLock lock = redissonClient.getLock(lockKey);

        try {
            // 尝试获取锁，等待 5 秒，锁自动过期 30 秒
            boolean acquired = lock.tryLock(5, 30, TimeUnit.SECONDS);

            if (!acquired) {
                throw new ConcurrentOperationException("系统繁忙，请稍后重试");
            }

            // 执行业务逻辑
            return supplier.get();

        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            throw new RuntimeException("操作被中断");
        } finally {
            // 释放锁
            if (lock.isHeldByCurrentThread()) {
                lock.unlock();
            }
        }
    }
}

// 使用示例
@Service
public class PaymentService {

    @Autowired
    private DistributedLockIdempotent idempotentService;

    public PaymentResult processPayment(PaymentRequest request) {
        // 生成锁 key
        String lockKey = String.format("payment:order:%d", request.getOrderId());

        return idempotentService.executeWithLock(lockKey, () -> {
            // Step 1: 检查订单状态（幂等）
            Order order = orderRepository.findById(request.getOrderId())
                    .orElseThrow(() -> new OrderNotFoundException(request.getOrderId()));

            if (order.isPaid()) {
                return PaymentResult.success(order.getPaymentId());
            }

            // Step 2: 执行支付
            PaymentResult result = doPay(order, request);

            // Step 3: 更新订单状态
            order.markAsPaid(result.getPaymentId());
            orderRepository.save(order);

            return result;
        });
    }
}
```

## 幂等设计最佳实践

### 实践一：前端 + 后端双重保障

```javascript
// 前端：防止重复提交
class PaymentButton {
    constructor() {
        this.isSubmitting = false;
    }

    async submit() {
        if (this.isSubmitting) {
            return;
        }

        this.isSubmitting = true;
        try {
            // 生成幂等 ID
            const idempotentId = this.generateIdempotentId();
            await this.doSubmit(idempotentId);
        } finally {
            this.isSubmitting = false;
        }
    }

    generateIdempotentId() {
        return `${this.userId}_${Date.now()}_${Math.random().toString(36).substr(2, 9)}`;
    }
}
```

### 实践二：幂等结果的缓存时间要合理

```java
// 幂等结果缓存时间设计
@Service
public class IdempotentConfig {
    // 不同场景的缓存时间不同
    private static final Map<String, Duration> EXPIRE_DURATIONS = Map.of(
            "payment", Duration.ofDays(7)),        // 支付：7 天
            "order", Duration.ofDays(30)),          // 订单：30 天
            "refund", Duration.ofDays(30)),         // 退款：30 天
            "message", Duration.ofHours(24)),       // 消息：24 小时
            "callback", Duration.ofDays(7));         // 回调：7 天

    public Duration getExpireDuration(String businessType) {
        return EXPIRE_DURATIONS.getOrDefault(businessType, Duration.ofDays(1));
    }
}
```

### 实践三：幂等 ID 的生成策略

```java
// 幂等 ID 生成策略
@Service
public class IdempotentIdGenerator {

    /**
     * 客户端生成幂等 ID（推荐）
     * 由调用方保证唯一性
     */
    public String generateByClient(String clientId) {
        return String.format("%s:%d:%s", clientId, System.currentTimeMillis(),
                UUID.randomUUID().toString().substring(0, 8));
    }

    /**
     * 从业务数据生成幂等 ID
     * 适用于：同一用户同一业务不能重复的场景
     */
    public String generateFromBusiness(Long userId, String businessType, Long businessId) {
        return String.format("%s:%d:%s:%d", businessType, userId, businessId);
    }

    /**
     * 使用 Redis INCR 生成全局唯一 ID
     */
    public long generateByRedis(String key) {
        RedisTemplate<String, Long> longTemplate = redisTemplate.opsForValue();
        return longTemplate.increment(key);
    }
}
```

## 总结

幂等性是分布式系统可靠性的基石。没有幂等性保障，重试可能变成灾难。

**核心要点回顾：**

- 幂等性：同一个请求执行一次和执行多次，效果相同
- 支付、订单、库存、回调等场景必须保证幂等
- 幂等实现方案：唯一标识、唯一索引、状态机、乐观锁、分布式锁
- 前端和后端要双重保障
- 幂等结果要缓存，缓存时间要合理设计
