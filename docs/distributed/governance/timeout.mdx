# 服务超时

你遇到过这种情况吗？点了一个按钮，界面一直转圈，30 秒后弹出一个超时错误。更糟糕的是，你不知道请求是成功了还是失败了——可能后端正在处理，可能网络断了，也可能服务器已经挂了。

这就是超时处理不当的代价：**用户的等待是无尽的，而服务器的线程是有限的。**

服务超时（Timeout）的核心思想很简单：**每个请求都要有个期限，超过期限就不再等待。**

## 为什么需要超时控制

### 没有超时的后果

```yaml
# 无超时场景下的连锁反应
用户发起请求
     │
     ▼
请求打到服务 A（正常响应 100ms）
     │
     ▼
请求打到服务 B（网络抖动，需要 30s）
     │
     ▼
服务 A 的线程被阻塞 30s
     │
     ▼
并发 1000 用户 → 1000 个线程被阻塞
     │
     ▼
线程池耗尽 → 新请求无法处理
     │
     ▼
整个系统雪崩
```

### 有超时的保护

```yaml
# 有超时场景下的处理
用户发起请求
     │
     ▼
请求打到服务 A（超时设置 5s）
     │
     ▼
服务 A 调用服务 B
     │
     ▼
3s 后服务 B 未响应，触发超时
     │
     ▼
服务 A 立即返回错误，释放线程
     │
     ▼
线程被复用，处理其他请求
     │
     ▼
系统保持健康
```

## 超时的设计原则

### 原则一：超时时间要合理

```yaml
# 超时时间设置参考
业务场景          推荐超时时间     说明
─────────────────────────────────────────────────
页面 AJAX 调用      3-10s          用户可接受的最长等待
移动端 API          10-30s         网络不稳定，需要更长
后台任务同步调用    30-60s         不阻塞用户，可适当延长
批量处理任务        5-30min        后台任务，允许更长
```

### 原则二：超时要有层级

```yaml
# 层级化超时设计
┌─────────────────────────────────────────────┐
│              全局超时：60s                   │
│  ┌─────────────────────────────────────────┐ │
│  │           接口级超时：10s                │ │
│  │  ┌───────────────────────────────────┐ │ │
│  │  │         调用级超时：3s              │ │ │
│  │  │  ┌─────────────────────────────────┐│ │ │
│  │  │  │      数据库超时：1s             ││ │ │
│  │  │  └─────────────────────────────────┘│ │ │
│  │  └───────────────────────────────────┘ │ │
│  └─────────────────────────────────────────┘ │
└─────────────────────────────────────────────┘
```

### 原则三：超时要有兜底

```java
// 超时后要有降级处理
@Service
public class ProductService {

    @CircuitBreaker(name = "product-service", fallbackMethod = "getProductFallback")
    public Product getProduct(Long productId) {
        return productApi.getProduct(productId);
    }

    public Product getProductFallback(Long productId, Throwable t) {
        if (t instanceof TimeoutException) {
            return productCache.get(productId);
        }
        throw new RuntimeException(t);
    }
}
```

## 超时的实现

### HTTP 客户端超时配置

```java
// OkHttp 超时配置
@Service
public class OkHttpClientService {

    private final OkHttpClient client = new OkHttpClient.Builder()
            .connectTimeout(3, TimeUnit.SECONDS)
            .readTimeout(5, TimeUnit.SECONDS)
            .writeTimeout(5, TimeUnit.SECONDS)
            .build();
}

// RestTemplate 超时配置
@Configuration
public class RestTemplateConfig {

    @Bean
    public RestTemplate restTemplate() {
        SimpleClientHttpRequestFactory factory = new SimpleClientHttpRequestFactory();
        factory.setConnectTimeout(3000);
        factory.setReadTimeout(5000);
        return new RestTemplate(factory);
    }
}
```

### Feign 超时配置

```yaml
# application.yml
feign:
  client:
    config:
      default:
        connectTimeout: 3000
        readTimeout: 5000
      product-service:
        connectTimeout: 5000
        readTimeout: 10000
```

### Dubbo 超时配置

```yaml
# 全局默认超时
dubbo:
  provider:
    timeout: 3000
  consumer:
    timeout: 3000
```

### 数据库超时配置

```yaml
# MySQL 连接超时
jdbc:mysql://localhost:3306/test?connectTimeout=3000&socketTimeout=5000

# HikariCP
spring:
  datasource:
    hikari:
      connection-timeout: 30000
      idle-timeout: 600000
      max-lifetime: 1800000
```

## 总结

超时控制是服务治理中最容易被忽视，但影响最大的机制之一。合理的超时配置可以让系统在故障时快速失败，而不是被慢慢拖垮。

**核心要点回顾：**

- 超时时间要合理：太长失去保护意义，太短容易误判
- 超时要有层级：从全局到接口到调用，逐层递进
- 超时后要有降级：超时不是终点，而是降级的起点
- 超时链路要串联：每个环节都要设置超时
