# 服务发现

凌晨 3 点，你被告警电话叫醒：生产环境某个服务不可用了。你登录控制台，发现原来是下午那次发布把某台机器的 IP 换了，但调用方的配置里还写着旧 IP。

这不是段子，这是微服务时代最真实的痛点：**服务实例的 IP 不是固定的，端口可能变，节点数量随时在变，传统的静态配置早已无法应对。**

服务发现（Service Discovery）正是为了解决这个问题而生的。它的核心目标是：**让服务调用方不需要关心服务提供者的具体位置，只需要知道「服务名」就够了。**

## 为什么需要服务发现

在单体架构时代，服务之间的调用是通过 IP 和端口直接完成的，配置写在代码里或者 Properties 文件里，几乎不会变动。但到了微服务时代，情况完全不同了：

```yaml
# 单体时代：静态配置，简单粗暴
service:
  url: http://192.168.1.100:8080/api

# 微服务时代：实例动态变化，静态配置是噩梦
# - 服务器可能随时扩缩容
# - 容器重启后 IP 会变
# - 灰度发布时会有新版本实例
# - 服务实例数可能从 1 台变 100 台
```

想象一下，如果没有服务发现，你的配置中心可能需要管理成千上万条这样的静态映射，而且每次扩缩容都需要手动更新。这不仅工作量大，更重要的是——**人是会出错的，而机器出错的概率要低得多。**

服务发现的核心价值，就是把「服务名到实例列表」的映射关系管理起来，让这个过程自动化、动态化、可靠化。

## 服务发现的核心角色

服务发现体系中有三个核心角色：

### 1. 服务注册中心（Registry）

注册中心是服务发现的大脑，所有服务实例在启动时都会向它注册自己的地址信息。常见的注册中心有：

| 注册中心 | 开发语言 | 一致性协议 | 特点 |
|---|---|---|---|
| Eureka | Java | AP（最终一致） | Spring Cloud 默认，对称架构，适合小型项目 |
| Consul | Go | CP（强一致） | 支持多数据中心，内置 DNS 功能 |
| Nacos | Java | AP/CP 可切换 | 功能丰富，国货之光，阿里有力竞争者 |
| Zookeeper | Java | CP（强一致） | 老牌选择，但运维复杂，适合做大规模集群选主 |

:::tip 如何选择注册中心
- 小型项目、快速起步：选 Eureka 或 Nacos
- 对一致性要求高（如金融场景）：选 Consul 或 Zookeeper
- 需要多数据中心容灾：选 Consul
- 想一个组件搞定注册+配置+管理界面：选 Nacos
:::

### 2. 服务提供者（Provider）

服务提供者在启动时主动向注册中心注册自己的地址信息，在运行时持续向注册中心发送心跳，证明自己还活着。如果注册中心长时间没收到某个实例的心跳，就会把这个实例从服务列表中剔除。

```java
// 服务提供者注册示例（Nacos SDK）
@Service
public class OrderServiceImpl implements OrderService {

    @Override
    @NacosInjected
    private NamingService namingService;

    public void register() throws NacosException {
        // 注册当前实例：服务名、IP、端口
        namingService.registerInstance("order-service", "192.168.1.101", 8080);
    }

    // 定时发送心跳
    @Scheduled(fixedRate = 5000)
    public void sendHeartbeat() throws NacosException {
        namingService.sendBeat("order-service", "192.168.1.101", 8080, 5000);
    }
}
```

### 3. 服务消费者（Consumer）

服务消费者在发起调用前，先从注册中心获取目标服务的实例列表，然后根据负载均衡策略选择一个具体的实例进行调用。这个过程是透明的，调用方只需要知道「我要调 order-service」，而不需要知道具体 IP。

```java
// 服务消费者发现示例
@Service
public class PaymentService {

    @NacosInjected
    private NamingService namingService;

    public void callOrderService() throws NacosException {
        // 根据服务名获取所有健康实例
        List<Instance> instances = namingService.selectInstances("order-service", true);

        if (instances.isEmpty()) {
            throw new RuntimeException("无可用的 order-service 实例");
        }

        // 负载均衡选择一个实例
        Instance selected = loadBalancer.select(instances);

        // 发起调用
        String url = "http://" + selected.getIp() + ":" + selected.getPort() + "/api/order";
        // ... 执行 HTTP 调用
    }
}
```

## 服务发现的两种模式

服务发现有两种主流模式，各有优劣：

### 客户端发现模式（Client-Side Discovery）

```yaml
┌────────────────────────────────────────────────────┐
│              客户端发现模式                          │
├────────────────────────────────────────────────────┤
│                                                    │
│   服务调用方                                         │
│   ┌──────────┐                                      │
│   │ 服务发现 │◄───── 直接查询注册中心                  │
│   │  客户端  │                                      │
│   └────┬─────┘                                      │
│        │                                            │
│        ▼                                            │
│   ┌────────────┐     ┌────────────┐                 │
│   │ 注册中心   │◄───►│ 实例 A     │                 │
│   │            │     └────────────┘                 │
│   │            │     ┌────────────┐                 │
│   │            │◄───►│ 实例 B     │                 │
│   │            │     └────────────┘                 │
│   └────────────┘                                    │
│                                                    │
│  优点：调用方直接知道所有实例，负载均衡灵活            │
│  缺点：每个服务调用方都要集成发现逻辑                  │
│                                                    │
└────────────────────────────────────────────────────┘
```

客户端发现的典型代表是 Netflix Eureka + Ribbon。服务消费者直接向注册中心查询服务实例列表，然后在本地完成负载均衡。

### 服务端发现模式（Server-Side Discovery）

```yaml
┌────────────────────────────────────────────────────┐
│              服务端发现模式                          │
├────────────────────────────────────────────────────┤
│                                                    │
│   服务调用方                                         │
│   ┌──────────┐                                      │
│   │  简单    │                                      │
│   │  客户端  │                                      │
│   └────┬─────┘                                      │
│        │                                            │
│        ▼                                            │
│   ┌────────────┐     ┌────────────┐                 │
│   │   API      │────►│ 实例 A     │                 │
│   │   Gateway  │     └────────────┘                 │
│   │            │     ┌────────────┐                 │
│   │  (路由器)  │────►│ 实例 B     │                 │
│   └────┬─────┘     └────────────┘                 │
│        │                                            │
│        ▼                                            │
│   ┌────────────┐                                    │
│   │ 注册中心   │                                    │
│   └────────────┘                                    │
│                                                    │
│  优点：调用方无需关心发现逻辑，统一入口                │
│  缺点：多一跳网络开销，Gateway 可能成为瓶颈            │
│                                                    │
└────────────────────────────────────────────────────┘
```

服务端发现的典型代表是 Kubernetes + kube-proxy、Nginx + Consul。服务调用方将请求发到一个固定地址（如 API Gateway），由 Gateway 负责查询注册中心并完成负载均衡。

## 注册中心的高可用设计

注册中心是整个服务发现体系的核心，一旦挂掉，所有服务都无法发现彼此，整个系统将陷入瘫痪。因此，注册中心的高可用设计至关重要。

### 心跳检测机制

```java
// 心跳检测的三种策略

// 1. 主动探测：注册中心定时探测服务实例
// 优点：准确度高
// 缺点：增加网络开销
public class ActiveProbeChecker {
    public boolean check(Instance instance) {
        String url = "http://" + instance.getIp() + ":" + instance.getPort() + "/health";
        try {
            Response response = httpClient.get(url);
            return response.getStatusCode() == 200;
        } catch (Exception e) {
            return false;
        }
    }
}

// 2. 心跳上报：服务实例主动向注册中心发送心跳
// 优点：减少注册中心的探测压力
// 缺点：依赖实例配合
public class HeartbeatReporter {
    public void report(String serviceName, Instance instance) {
        // 定时发送心跳
        scheduledExecutor.scheduleAtFixedRate(() -> {
            registry.heartbeat(serviceName, instance);
        }, 0, 5, TimeUnit.SECONDS);
    }
}

// 3. 混合模式：结合以上两种
// 注册中心定期探测 + 服务实例主动上报心跳
// 任何一方检测到异常都会触发实例下线
```

### 故障隔离

一个成熟的注册中心需要具备故障隔离能力：

```yaml
# Nacos 的临时实例与持久实例
-
  # 临时实例（ephemeral=true）
  # - 基于心跳检测
  # - 注册中心长时间没收到心跳会自动剔除
  # - 适合普通微服务
  -
    serviceName: order-service
    ephemeral: true
    weight: 1.0

# 持久实例（ephemeral=false）
# - 基于主动探测或手动注销
# - 即使不发送心跳也不会被剔除
# - 适合需要保留地址的基础服务
-
  serviceName: config-center
  ephemeral: false
  weight: 1.0
```

## 服务发现的挑战与演进

### 挑战一：注册延迟

从服务启动到被其他服务发现，中间存在一个时间窗口。在这个窗口内，新启动的实例可能收到请求但尚未准备好处理。

```yaml
# 典型问题时间线
T0: 服务实例启动
T1: 服务完成初始化，开始监听端口
T2: 向注册中心注册成功
T3: 其他服务感知到新实例
T4: 开始收到请求

# 问题：如果 T3 > T1，新实例可能在 T2-T3 期间收到请求
# 解决：服务就绪后再注册，或使用「预注册 + 延迟上线」策略
```

### 挑战二：网络分区

当网络出现分区时，注册中心可能无法及时感知某些实例的存活状态，导致请求被发到已经「假死」的实例上。

:::warning 网络分区的影响
在 CAP 理论中，注册中心必须在可用性（Availability）和一致性（Consistency）之间做出选择。Eureka 选择 AP，保证在网络异常时依然可用；Zookeeper 选择 CP，保证强一致但在网络分区时可能不可用。选择哪种，取决于你的业务场景。
:::

### 挑战三：服务实例元数据

除了 IP 和端口，有时还需要传递更多元数据：

```yaml
# 服务实例元数据示例
-
  ip: 192.168.1.101
  port: 8080
  weight: 100                    # 权重，用于负载均衡
  version: v2.1.0               # 版本号，用于灰度发布
  region: shanghai              # 地域，用于同区域优先
  cluster: default              # 集群，用于跨集群容灾
  metadata:
    cpu: 8                       # CPU 核心数
    memory: 16                    # 内存 GB
    disk: 100                     # 磁盘 GB
```

这些元数据在灰度发布、跨机房容灾、金丝雀发布等场景下非常有用。

## 服务发现与负载均衡的配合

服务发现解决的是「有哪些实例可用」的问题，而负载均衡解决的是「选择哪个实例」的问题。两者配合使用：

```java
// 服务发现 + 负载均衡的完整流程
public class ServiceConsumer {

    private NamingService namingService;
    private LoadBalancer loadBalancer;

    public String callService(String serviceName) {
        // Step 1: 从注册中心获取实例列表（服务发现）
        List<Instance> instances = namingService.selectInstances(serviceName, true);

        // Step 2: 根据负载均衡策略选择实例
        Instance selected = loadBalancer.select(serviceName, instances);

        // Step 3: 发起调用
        return httpClient.get("http://" + selected.getIp() + ":" + selected.getPort());
    }
}
```

## 总结

服务发现是微服务架构的基础设施，它让「服务名」替代「IP+端口」成为服务调用的入口。理解服务发现的工作原理，对于构建高可用的微服务系统至关重要。

**核心要点回顾：**

- 服务发现解决的是「服务在哪」的问题
- 注册中心、服务提供者、服务消费者是三大核心角色
- 客户端发现模式灵活，服务端发现模式简单
- 心跳机制是保证实例健康的关键
- 注册中心需要在 AP 和 CP 之间做出权衡
