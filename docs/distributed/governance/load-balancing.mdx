# 负载均衡策略

双十一零点峰值，你的后台日志疯狂滚动。某台服务器已经 CPU 100% 了，但新请求还在源源不断地打过来；而旁边几台服务器却还闲着，CPU 使用率不到 30%。

这不是资源浪费，这是负载不均的灾难。问题出在哪？出在负载均衡策略选错了。

负载均衡（Load Balancing）是分布式系统的流量调度核心，它解决的问题是：**当有多个服务实例时，如何合理分配请求，使每个实例的负载与其能力相匹配。**

## 负载均衡的本质

负载均衡的核心目标有三个：

```yaml
# 负载均衡三大目标
1. 效率最大化：充分利用每台服务器的能力
2. 稳定性保障：避免单点过载导致系统崩溃
3. 故障容错：当某个实例不可用时，自动将流量切换到健康实例
```

但这三个目标有时会相互矛盾。例如，为了追求绝对公平，可能会增加调度开销；为了快速响应，可能会牺牲负载均衡的精确度。因此，实际系统中的负载均衡策略都是在「精确度」和「效率」之间做权衡。

## 常见的负载均衡策略

### 1. 轮询（Round Robin）

最简单也最常用的策略，所有实例轮流处理请求。

```java
// 轮询实现
public class RoundRobinLoadBalancer {

    private final AtomicInteger position = new AtomicInteger(0);

    public Instance select(List<Instance> instances) {
        if (instances == null || instances.isEmpty()) {
            throw new IllegalArgumentException("无可用实例");
        }

        int index = Math.abs(position.getAndIncrement()) % instances.size();
        return instances.get(index);
    }
}
```

```yaml
# 轮询策略特点
优点：
  - 实现简单，开销低
  - 请求分布均匀（在实例性能相近时）

缺点：
  - 不考虑服务器实际负载
  - 不考虑网络延迟差异
  - 不考虑实例性能差异

适用场景：实例性能相近、网络延迟差异小的场景
```

### 2. 加权轮询（Weighted Round Robin）

根据服务器性能分配不同的权重，性能强的处理更多请求。

```java
// 加权轮询实现
public class WeightedRoundRobinLoadBalancer {

    private final AtomicInteger position = new AtomicInteger(0);

    public Instance select(List<Instance> instances) {
        if (instances == null || instances.isEmpty()) {
            throw new IllegalArgumentException("无可用实例");
        }

        // 计算总权重
        int totalWeight = instances.stream()
                .mapToInt(Instance::getWeight)
                .sum();

        // 随机选择一个位置
        int random = ThreadLocalRandom.current().nextInt(totalWeight);

        int cursor = 0;
        for (Instance instance : instances) {
            cursor += instance.getWeight();
            if (random < cursor) {
                return instance;
            }
        }

        return instances.get(0);
    }
}
```

```yaml
# 配置示例
instances:
  - ip: 192.168.1.101
    port: 8080
    weight: 100        # 高配服务器，权重 100

  - ip: 192.168.1.102
    port: 8080
    weight: 50        # 低配服务器，权重 50

# 结果：高配服务器处理 2/3 的流量
```

### 3. 随机（Random）

```java
// 随机负载均衡
public class RandomLoadBalancer {

    public Instance select(List<Instance> instances) {
        if (instances == null || instances.isEmpty()) {
            throw new IllegalArgumentException("无可用实例");
        }

        int index = ThreadLocalRandom.current().nextInt(instances.size());
        return instances.get(index);
    }
}
```

```yaml
# 随机策略特点
优点：
  - 实现简单
  - 在实例数量足够多时，分布接近均匀
  - 比轮询更好的容灾能力（多个随机选择不太可能总选中同一个故障实例）

缺点：
  - 不保证绝对均匀
  - 不考虑服务器负载
```

### 4. 最少连接（Least Connections）

将请求发送到当前活跃连接数最少的实例。

```java
// 最少连接实现
public class LeastConnectionsLoadBalancer {

    // 记录每个实例的当前连接数
    private final ConcurrentHashMap<String, AtomicInteger> connectionCounts = new ConcurrentHashMap<>();

    public Instance select(List<Instance> instances) {
        if (instances == null || instances.isEmpty()) {
            throw new IllegalArgumentException("无可用实例");
        }

        // 选择连接数最少的实例
        Instance selected = instances.get(0);
        AtomicInteger minCount = connectionCounts.computeIfAbsent(
                selected.getId(), k -> new AtomicInteger(0));

        for (Instance instance : instances) {
            AtomicInteger count = connectionCounts.computeIfAbsent(
                    instance.getId(), k -> new AtomicInteger(0));
            if (count.get() < minCount.get()) {
                selected = instance;
                minCount = count;
            }
        }

        // 增加连接计数
        minCount.incrementAndGet();
        return selected;
    }

    // 请求结束后释放连接
    public void releaseConnection(Instance instance) {
        AtomicInteger count = connectionCounts.get(instance.getId());
        if (count != null && count.get() > 0) {
            count.decrementAndGet();
        }
    }
}
```

```yaml
# 最少连接策略特点
优点：
  - 能动态适应实例的实际负载
  - 适合长连接场景（如 RPC、数据库连接池）

缺点：
  - 需要维护连接状态，增加内存开销
  - 实现相对复杂
  - 瞬时连接数波动可能导致不均衡

适用场景：长连接服务、数据库连接池、HTTP Keep-Alive 场景
```

### 5. 最少响应时间（Least Response Time）

选择平均响应时间最短的实例。

```java
// 最短响应时间实现
public class LeastResponseTimeLoadBalancer {

    // 滑动窗口：记录最近 N 次响应时间
    private final ConcurrentHashMap<String, SlidingWindow> responseTimes = new ConcurrentHashMap<>();

    public Instance select(List<Instance> instances) {
        if (instances == null || instances.isEmpty()) {
            throw new IllegalArgumentException("无可用实例");
        }

        Instance selected = instances.get(0);
        double minAvgTime = Double.MAX_VALUE;

        for (Instance instance : instances) {
            SlidingWindow window = responseTimes.computeIfAbsent(
                    instance.getId(), k -> new SlidingWindow(100));

            double avgTime = window.getAverage();
            if (avgTime < minAvgTime) {
                minAvgTime = avgTime;
                selected = instance;
            }
        }

        return selected;
    }

    // 记录响应时间
    public void recordResponseTime(Instance instance, long responseTimeMs) {
        SlidingWindow window = responseTimes.get(instance.getId());
        if (window != null) {
            window.add(responseTimeMs);
        }
    }
}
```

### 6. 一致性哈希（Consistent Hash）

相同请求特征的请求始终发送到同一个实例。

```java
// 一致性哈希实现
public class ConsistentHashLoadBalancer {

    private final TreeMap<Long, Instance> circle = new TreeMap<>();
    private final HashFunction hashFunction = Hashing.murmur3_128();

    public ConsistentHashLoadBalancer(List<Instance> instances, int virtualNodes) {
        for (Instance instance : instances) {
            addNode(instance, virtualNodes);
        }
    }

    public void addNode(Instance instance, int virtualNodes) {
        // 添加物理节点
        circle.put(hashFunction.hashString(instance.getId(), StandardCharsets.UTF_8).asLong(),
                   instance);

        // 添加虚拟节点，实现更均匀的分布
        for (int i = 0; i < virtualNodes; i++) {
            long hash = hashFunction.hashString(instance.getId() + "#vn" + i,
                                                StandardCharsets.UTF_8).asLong();
            circle.put(hash, instance);
        }
    }

    public Instance select(List<Instance> instances, String key) {
        if (instances == null || instances.isEmpty()) {
            throw new IllegalArgumentException("无可用实例");
        }

        long hash = hashFunction.hashString(key, StandardCharsets.UTF_8).asLong();

        // 找到第一个大于等于 hash 的节点
        Map.Entry<Long, Instance> entry = circle.ceilingEntry(hash);

        // 如果没有，说明 hash 超过了所有节点，顺时针回到第一个
        if (entry == null) {
            entry = circle.firstEntry();
        }

        return entry.getValue();
    }
}
```

```yaml
# 一致性哈希特点
优点：
  - 相同 key 的请求打到相同实例，适合有本地缓存的场景
  - 节点变更时，影响范围最小化
  - 有效解决缓存雪崩问题

缺点：
  - 实现复杂
  - 负载分布可能不均匀（需要虚拟节点优化）

适用场景：缓存服务、Session 保持、有状态服务
```

### 7. 源地址哈希（Source IP Hash）

根据客户端 IP 地址计算哈希值，选择对应实例。

```java
// 源地址哈希实现
public class SourceIpHashLoadBalancer {

    public Instance select(List<Instance> instances, String clientIp) {
        if (instances == null || instances.isEmpty()) {
            throw new IllegalArgumentException("无可用实例");
        }

        int hash = clientIp.hashCode();
        int index = Math.abs(hash) % instances.size();
        return instances.get(index);
    }
}
```

## 策略对比

| 策略 | 复杂度 | 资源利用率 | 适用场景 | 缺点 |
|---|---|---|---|---|
| 轮询 | 低 | 中 | 实例性能相近 | 不考虑实际负载 |
| 加权轮询 | 中 | 高 | 实例性能不均 | 需要手动配置权重 |
| 随机 | 低 | 中 | 小规模系统 | 分布不稳定 |
| 最少连接 | 高 | 高 | 长连接服务 | 需要维护状态 |
| 最短响应时间 | 高 | 高 | 对延迟敏感 | 需要收集统计 |
| 一致性哈希 | 高 | 高 | 缓存/有状态 | 实现复杂 |
| 源地址哈希 | 低 | 中 | Session 保持 | 分布不均 |

## 负载均衡的综合实现

实际生产中，负载均衡器通常会结合多种策略：

```java
// 综合负载均衡器
public class CompositeLoadBalancer {

    private final RandomLoadBalancer randomLoadBalancer = new RandomLoadBalancer();
    private final RoundRobinLoadBalancer roundRobinLoadBalancer = new RoundRobinLoadBalancer();
    private final LeastConnectionsLoadBalancer leastConnectionsLoadBalancer = new LeastConnectionsLoadBalancer();

    private volatile String strategy = "weighted";  // 可配置

    public Instance select(LoadBalancerContext context) {
        List<Instance> healthyInstances = context.getHealthyInstances();

        if (healthyInstances.isEmpty()) {
            throw new NoAvailableException("无可用实例");
        }

        // 过滤掉熔断的实例
        List<Instance> availableInstances = healthyInstances.stream()
                .filter(inst -> !circuitBreaker.isOpen(inst.getServiceName()))
                .collect(Collectors.toList());

        if (availableInstances.isEmpty()) {
            throw new NoAvailableException("所有实例都已熔断");
        }

        // 根据配置选择策略
        switch (strategy) {
            case "round-robin":
                return roundRobinLoadBalancer.select(availableInstances);

            case "random":
                return randomLoadBalancer.select(availableInstances);

            case "least-connections":
                return leastConnectionsLoadBalancer.select(availableInstances);

            case "weighted":
                return weightedSelect(availableInstances, context);

            case "consistent-hash":
                return consistentHashSelect(availableInstances, context.getRequestKey());

            default:
                return roundRobinLoadBalancer.select(availableInstances);
        }
    }

    // 加权选择：综合权重、响应时间、健康度
    private Instance weightedSelect(List<Instance> instances, LoadBalancerContext context) {
        // 计算综合得分
        Map<Instance, Double> scores = new HashMap<>();

        for (Instance instance : instances) {
            double weight = instance.getWeight();           // 配置权重
            double healthScore = instance.getHealthScore(); // 健康度
            double latencyScore = 1.0 / (instance.getAvgResponseTime() + 1);  // 延迟得分

            // 综合得分 = 权重 × 健康度 × 延迟得分
            double score = weight * healthScore * latencyScore;
            scores.put(instance, score);
        }

        // 加权随机选择
        double totalScore = scores.values().stream().mapToDouble(Double::doubleValue).sum();
        double random = ThreadLocalRandom.current().nextDouble() * totalScore;

        double cursor = 0;
        for (Map.Entry<Instance, Double> entry : scores.entrySet()) {
            cursor += entry.getValue();
            if (random <= cursor) {
                return entry.getKey();
            }
        }

        return instances.get(0);
    }
}
```

## 负载均衡的健康感知

负载均衡不仅要分配流量，还要感知实例的健康状态：

```java
// 健康感知的负载均衡
public class HealthAwareLoadBalancer {

    public Instance select(LoadBalancerContext context) {
        List<Instance> candidates = context.getAllInstances();

        // 分离健康和不健康的实例
        List<Instance> healthy = candidates.stream()
                .filter(Instance::isHealthy)
                .collect(Collectors.toList());

        // 优先选择健康实例
        List<Instance> available = healthy.isEmpty() ? candidates : healthy;

        // 在可用实例中进行负载均衡
        return doSelect(available, context);
    }

    // 逐步摘除不健康实例的流量
    private Instance doSelect(List<Instance> instances, LoadBalancerContext context) {
        // 根据健康度和权重计算选择概率
        double totalHealth = instances.stream()
                .mapToDouble(Instance::getHealthScore)
                .sum();

        double random = ThreadLocalRandom.current().nextDouble() * totalHealth;

        double cursor = 0;
        for (Instance instance : instances) {
            cursor += instance.getHealthScore();
            if (random <= cursor) {
                return instance;
            }
        }

        return instances.get(0);
    }
}
```

## 总结

负载均衡是分布式系统的流量调度核心，选择合适的策略对系统性能至关重要。

**核心要点回顾：**

- 轮询和随机适合实例性能相近的场景
- 加权策略适合实例性能差异大的场景
- 最少连接适合长连接和耗时任务
- 一致性哈希适合有本地缓存的场景
- 实际生产中，负载均衡需要结合健康检查、熔断降级等多重保障
