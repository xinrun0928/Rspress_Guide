# 服务限流

双十一零点，你盯着监控大屏，眼睁睁看着 QPS 从 10 万飙升到 100 万。服务器开始告警，响应时间从 50ms 飙升到 5s，然后——系统崩溃了。

如果能重来一次，你会不会在流量高峰到来之前，设置一道闸门，把多余的请求挡在外面，让系统「悠着点」？

这就是限流（Rate Limiting）要解决的问题：**当流量超过系统处理能力时，主动拒绝一部分请求，保护系统不被冲垮。**

## 为什么需要限流

### 不限流的后果


## 限流算法

### 1. 固定窗口算法（Fixed Window）

最简单的限流算法，将时间划分为固定长度的窗口，每个窗口内允许固定数量的请求。

```java
// 固定窗口限流器
public class FixedWindowRateLimiter {

    private final int maxRequests;
    private final long windowSizeMs;
    private final AtomicInteger counter = new AtomicInteger(0);
    private volatile long windowStart = System.currentTimeMillis();

    public FixedWindowRateLimiter(int maxRequests, long windowSizeMs) {
        this.maxRequests = maxRequests;
        this.windowSizeMs = windowSizeMs;
    }

    public synchronized boolean allowRequest() {
        long now = System.currentTimeMillis();
        if (now - windowStart >= windowSizeMs) {
            counter.set(0);
            windowStart = now;
        }
        if (counter.get() < maxRequests) {
            counter.incrementAndGet();
            return true;
        }
        return false;
    }
}
```

### 2. 滑动窗口算法（Sliding Window）

对固定窗口的改进，将时间窗口划分为更小的单元，滑动计算。

```java
// 滑动窗口限流器
public class SlidingWindowRateLimiter {

    private final int maxRequests;
    private final int windowSizeInSeconds;
    private final long windowLengthInMs;
    private final Deque<Long> timestamps;

    public SlidingWindowRateLimiter(int maxRequests, int windowSizeInSeconds) {
        this.maxRequests = maxRequests;
        this.windowSizeInSeconds = windowSizeInSeconds;
        this.windowLengthInMs = windowSizeInSeconds * 1000L;
        this.timestamps = new ConcurrentLinkedDeque<>();
    }

    public synchronized boolean allowRequest() {
        long now = System.currentTimeMillis();
        long windowStart = now - windowLengthInMs;

        while (!timestamps.isEmpty() && timestamps.peekFirst() <= windowStart) {
            timestamps.pollFirst();
        }

        if (timestamps.size() < maxRequests) {
            timestamps.addLast(now);
            return true;
        }
        return false;
    }
}
```

### 3. 令牌桶算法（Token Bucket）

最常用的限流算法，以固定速率向桶中添加令牌，请求需要获取令牌才能执行。

```java
// 令牌桶限流器
public class TokenBucketRateLimiter {

    private final double rate;           // 每秒产生的令牌数
    private final long capacity;         // 桶的容量
    private double tokens;                // 当前令牌数
    private long lastRefillTime;         // 上次填充时间

    public TokenBucketRateLimiter(double rate, long capacity) {
        this.rate = rate;
        this.capacity = capacity;
        this.tokens = capacity;
        this.lastRefillTime = System.currentTimeMillis();
    }

    public synchronized boolean allowRequest() {
        refill();
        if (tokens >= 1) {
            tokens -= 1;
            return true;
        }
        return false;
    }

    private void refill() {
        long now = System.currentTimeMillis();
        long elapsed = now - lastRefillTime;
        double tokensToAdd = (elapsed / 1000.0) * rate;
        tokens = Math.min(capacity, tokens + tokensToAdd);
        lastRefillTime = now;
    }
}
```

### 4. 漏桶算法（Leaky Bucket）

以固定速率从桶中「漏」出请求，桶满则拒绝。

```java
// 漏桶限流器
public class LeakyBucketRateLimiter {

    private final double leakRate;
    private final long capacity;
    private double water;
    private long lastLeakTime;

    public LeakyBucketRateLimiter(double leakRate, long capacity) {
        this.leakRate = leakRate;
        this.capacity = capacity;
        this.water = 0;
        this.lastLeakTime = System.currentTimeMillis();
    }

    public synchronized boolean allowRequest() {
        leak();
        if (water < capacity) {
            water += 1;
            return true;
        }
        return false;
    }

    private void leak() {
        long now = System.currentTimeMillis();
        long elapsed = now - lastLeakTime;
        double leaked = (elapsed / 1000.0) * leakRate;
        water = Math.max(0, water - leaked);
        lastLeakTime = now;
    }
}
```

### 算法对比

```yaml
# 算法对比
算法          突发支持   实现复杂度   内存占用   适用场景
────────────────────────────────────────────────────────────
固定窗口        ❌         低          低        简单限流
滑动窗口        ❌         中          中        精确限流
令牌桶          ✅         中          低        API 限流
漏桶            ❌         中          低        平滑限流
```

## 限流的实现

### 单机限流：Guava RateLimiter

```java
// Guava RateLimiter
@RestController
public class ApiController {

    private final RateLimiter rateLimiter = RateLimiter.create(100.0, 200,
            RateLimiter.SleepyTightBounderMode.BURSTY);

    @GetMapping("/api/data")
    public ResponseEntity<String> getData() {
        if (!rateLimiter.tryAcquire(1, 3, TimeUnit.SECONDS)) {
            return ResponseEntity.status(429)
                    .body("请求过于频繁，请稍后重试");
        }
        return ResponseEntity.ok("Success");
    }
}
```

### 分布式限流：Redis + Lua

```lua
-- Redis 滑动窗口限流
local key = KEYS[1]
local windowSize = tonumber(ARGV[1])
local maxRequests = tonumber(ARGV[2])
local now = tonumber(ARGV[3])
local windowStart = now - windowSize

redis.call('ZREMRANGEBYSCORE', key, '-inf', windowStart)
local currentCount = redis.call('ZCARD', key)

if currentCount < maxRequests then
    redis.call('ZADD', key, now, now .. '-' .. math.random())
    redis.call('PEXPIRE', key, windowSize)
    return 1
else
    return 0
end
```

```java
// Redis 限流实现
@Service
public class RedisRateLimiter {

    @Autowired
    private StringRedisTemplate redisTemplate;

    public boolean isAllowed(String key, int maxRequests, int windowSeconds) {
        Long result = redisTemplate.execute(
                script,
                Collections.singletonList(key),
                String.valueOf(windowSeconds * 1000L),
                String.valueOf(maxRequests),
                String.valueOf(System.currentTimeMillis())
        );
        return result != null && result == 1L;
    }
}
```

### Sentinel 限流

```java
// Sentinel 限流
@Configuration
public class SentinelConfig {

    @PostConstruct
    public void initRules() {
        List<FlowRule> rules = new ArrayList<>();

        FlowRule rule = new FlowRule("/api/order")
                .setGrade(RuleConstant.FLOW_GRADE_QPS)
                .setCount(1000)
                .setControlBehavior(RuleConstant.CONTROL_BEHAVIOR_DEFAULT);

        rules.add(rule);
        FlowRuleManager.loadRules(rules);
    }
}

@RestController
public class SentinelController {

    @GetMapping("/api/order")
    @SentinelResource(value = "/api/order",
                     blockHandler = "orderBlockHandler")
    public ResponseEntity<String> createOrder() {
        return ResponseEntity.ok("Order created");
    }

    public ResponseEntity<String> orderBlockHandler(BlockException ex) {
        return ResponseEntity.status(429).body("系统繁忙，请稍后重试");
    }
}
```

## 限流的维度

限流可以从多个维度进行：

```yaml
# 限流维度
1. 全局限流    - 限制整个系统的总请求量
2. 接口限流    - 限制单个接口的请求量
3. 用户维度    - 按用户 ID 限流
4. IP 维度     - 按 IP 地址限流
5. 服务维度    - 按服务名限流
6. 多维度组合  - 用户 + 接口组合
```

## 总结

限流是保护系统的最后一道防线，当流量超过系统处理能力时，通过主动拒绝部分请求来保护系统不被冲垮。

**核心要点回顾：**

- 固定窗口算法简单但有临界问题，滑动窗口更精确
- 令牌桶支持突发流量，漏桶限流更平滑
- 限流维度包括：全局、接口、用户、IP 等
- 分布式限流需要使用 Redis 等中间件实现
