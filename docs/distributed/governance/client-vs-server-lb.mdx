# 客户端 vs 服务端负载均衡

你在架构选型时陷入了纠结：团队里有人说用 Nginx 做负载均衡，简单可靠；也有人说用 Ribbon/Feign 的客户端负载均衡，灵活可控。两种方案看起来都能用，到底该怎么选？

要回答这个问题，先要搞清楚一个基本概念：**负载均衡在哪里做，是有本质区别的。**

## 两种架构的本质区别

```yaml
┌─────────────────────────────────────────────────────────────┐
```

## 客户端负载均衡

### 工作原理

客户端负载均衡器嵌入在服务调用方进程中，它从注册中心获取服务实例列表，然后在本地进行负载均衡决策。

```java
// Ribbon 客户端负载均衡示例
@Configuration
public class RibbonConfig {

    @Bean
    public ILoadBalancer loadBalancer(IClientConfig config) {
        // 配置负载均衡器
        IRule rule = new WeightedResponseTimeRule();  // 加权响应时间策略
        IPing ping = new PingUrl();                    // 健康检查策略

        return new ZoneAwareLoadBalancer(config, rule, ping);
    }
}

// 使用 Ribbon 调用服务
@RestController
public class OrderController {

    @Autowired
    private RestTemplate restTemplate;

    @GetMapping("/order/{id}")
    public Order getOrder(@PathVariable Long id) {
        // 注意：这里的 URL 是服务名，不是具体 IP
        // Ribbon 会自动解析服务名，并进行负载均衡
        return restTemplate.getForObject(
            "http://product-service/products/" + id,
            Order.class
        );
    }
}
```

### 核心组件

Ribbon 的客户端负载均衡包含四大核心组件：

```java
// 1. Rule - 负载均衡策略
public interface IRule {
    Server choose(Object key);  // 选择一个 Server
}

// 2. Ping - 健康检查机制
public interface IPing {
    boolean isAlive(Server server);  // 检查 Server 是否存活
}

// 3. ServerList - 服务实例列表
public interface ServerList<T extends Server> {
    List<T> getInitialListOfServers();     // 初始列表
    List<T> getUpdatedListOfServers();     // 动态更新列表
}

// 4. LoadBalancer - 负载均衡器核心
public interface ILoadBalancer {
    void addServers(List<Server> servers);     // 添加服务器
    Server chooseServer(Object key);           // 选择服务器
    void markServerDown(Server server);        // 标记服务器下线
    List<Server> getReachableServers();         // 获取可达服务器
    List<Server> getAllServers();               // 获取所有服务器
}
```

### Spring Cloud LoadBalancer（Ribbon 的替代者）

Spring Cloud 2020.0 之后，Ribbon 已进入维护模式，官方推荐使用 Spring Cloud LoadBalancer：

```java
// Spring Cloud LoadBalancer 配置
@Configuration
public class LoadBalancerConfig {

    @Bean
    public ReactorLoadBalancer<ServiceInstance> randomLoadBalancer(
            ServiceInstanceListSupplier supplier) {

        return new RandomLoadBalancer(supplier);
    }

    @Bean
    public ReactorLoadBalancer<ServiceInstance> roundRobinLoadBalancer(
            ServiceInstanceListSupplier supplier) {

        return new RoundRobinLoadBalancer(supplier, "round-robin");
    }

    // 自定义负载均衡策略
    @Bean
    public ReactorLoadBalancer<ServiceInstance> weightedLoadBalancer(
            ServiceInstanceListSupplier supplier) {

        return new WeightedLoadBalancer(supplier,
            new DefaultServiceInstanceList(supplier.toString()));
    }
}

// 使用 Spring Cloud LoadBalancer
@RestController
public class ProductController {

    @Autowired
    private LoadBalancerClient loadBalancer;

    @GetMapping("/product/{id}")
    public Product getProduct(@PathVariable Long id) {
        // 手动选择实例
        ServiceInstance instance = loadBalancer.choose("product-service");
        String url = instance.getUri() + "/products/" + id;

        return restTemplate.getForObject(url, Product.class);
    }

    // 或者使用 @LoadBalanced 的 RestTemplate
    @Bean
    @LoadBalanced
    public RestTemplate restTemplate() {
        return new RestTemplate();
    }

    // 直接使用服务名调用，自动负载均衡
    @GetMapping("/product2/{id}")
    public Product getProduct2(@PathVariable Long id) {
        return restTemplate.getForObject(
            "http://product-service/products/" + id,
            Product.class
        );
    }
}
```

## 服务端负载均衡

### Nginx 负载均衡

```nginx
# Nginx 负载均衡配置
upstream backend {
    # 轮询（默认）
    server 192.168.1.101:8080 weight=100;
    server 192.168.1.102:8080 weight=50;
    server 192.168.1.103:8080 backup;  # 备份服务器
}

server {
    listen 80;

    location / {
        proxy_pass http://backend;

        # 健康检查配置
        proxy_connect_timeout 5s;
        proxy_next_upstream error timeout http_500 http_502 http_503;
    }
}
```

```yaml
# Nginx 负载均衡策略
upstream backend {
    # 轮询（默认）
    server 192.168.1.101:8080;
    server 192.168.1.102:8080;

    # 加权轮询
    server 192.168.1.101:8080 weight=5;
    server 192.168.1.102:8080 weight=1;

    # IP 哈希（Session 保持）
    ip_hash;
    server 192.168.1.101:8080;
    server 192.168.1.102:8080;

    # 最少连接
    least_conn;
    server 192.168.1.101:8080;
    server 192.168.1.102:8080;
}
```

## 深度对比

| 维度 | 客户端负载均衡 | 服务端负载均衡 |
|---|---|---|
| **位置** | 嵌入在调用方进程中 | 独立部署或独立硬件 |
| **实例列表获取** | 调用方从注册中心获取 | LB 从配置或注册中心获取 |
| **决策点** | 调用方本地决策 | LB 统一决策 |
| **网络开销** | 少一次跳转 | 多一次跳转 |
| **复杂度** | 调用方逻辑复杂 | 架构简单 |
| **一致性** | 可能短暂不一致 | 强一致 |
| **运维** | 每个调用方都要管理 | 集中管理 |
| **扩展性** | 横向扩展调用方 | 横向扩展 LB |
| **故障影响** | 单个调用方故障不影响其他 | LB 故障影响全局 |

## 优缺点分析

### 客户端负载均衡的优势

```yaml
# 优势一：更低的延迟
# 调用方直接知道所有实例，可以选择最优的
# 省去了请求转发的那一跳

# 优势二：更高的可用性
# 没有单点瓶颈
# 即使某个调用方挂了，其他调用方不受影响

# 优势三：更好的容错性
# 调用方可以根据实际响应判断实例是否可用
# 比服务端的心跳检测更准确

# 优势四：更灵活的策略
# 可以根据调用方的上下文做决策
# 比如选择同机房、同区域的实例
```

```java
// 区域感知的负载均衡示例
public class ZoneAwareLoadBalancer extends DynamicServerListLoadBalancer {

    @Override
    public Server choose(Object key) {
        // 1. 获取当前实例所在的区域
        String currentZone = ZoneUtils.getCurrentZone();

        // 2. 获取同区域的可用实例
        Map<String, ZoneStats> zoneStatsMap = getZoneStats();
        ZoneStats currentZoneStats = zoneStatsMap.get(currentZone);

        // 3. 优先选择同区域的实例
        List<Server> sameZoneServers = currentZoneStats.getServers().stream()
                .filter(Server::isReadyToServe)
                .collect(Collectors.toList());

        if (!sameZoneServers.isEmpty()) {
            return chooseFromZone(sameZoneServers);
        }

        // 4. 同区域无可用，fallback 到其他区域
        log.warn("同区域无可用实例，fallback 到跨区域调用");
        return chooseFromZone(getAllServers());
    }
}
```

### 客户端负载均衡的劣势

```yaml
# 劣势一：每个服务调用方都要集成 SDK
# 增加了开发和维护成本

# 劣势二：升级困难
# SDK 升级需要修改所有调用方
# 服务端升级只需改 LB

# 劣势三：资源占用
# 每个进程都要维护服务实例列表
# 实例数量多时，内存开销可观
```

### 服务端负载均衡的优势

```yaml
# 优势一：架构简单
# 调用方无需关心负载均衡逻辑
# 只需请求固定地址

# 优势二：集中管理
# 所有流量经过 LB，方便监控和限流
# SSL 终结、日志审计都在 LB 完成

# 优势三：易于升级
# LB 版本升级不影响调用方

# 优势四：硬件加速
# F5 等硬件 LB 可以达到极高吞吐量
```

### 服务端负载均衡的劣势

```yaml
# 劣势一：单点瓶颈
# LB 挂了，所有服务都不可用
# 需要部署多台 LB 做高可用

# 劣势二：多一跳网络开销
# 所有请求都要经过 LB
# 增加了网络延迟

# 劣势三：感知不准确
# LB 只能通过心跳检测实例健康
# 无法感知实例的实际处理能力
```

## 如何选择

```yaml
# 选择建议

## 选客户端负载均衡的场景：
- 微服务架构，服务调用链路复杂
- 对延迟敏感，要求尽可能减少网络跳转
- 需要精细化的负载均衡策略（如区域感知）
- 调用方数量多，需要分散决策压力

## 选服务端负载均衡的场景：
- 需要统一的流量入口（如 API Gateway）
- 需要集中处理 SSL 终结、请求限流
- 对安全性要求高，需要日志审计
- 传统架构，服务实例相对固定
```

:::tip 实际推荐：混合使用
很多成熟系统会同时使用两种负载均衡：

1. **在微服务内部**：使用客户端负载均衡（如 Ribbon/Spring Cloud LoadBalancer）
   - 减少网络跳转
   - 支持更精细的路由策略

2. **在系统边界**：使用服务端负载均衡（如 Nginx Ingress Controller）
   - 统一入口
   - SSL 终结
   - 全局限流

这样可以兼顾两者的优势。
:::

## 总结

客户端负载均衡和服务端负载均衡没有绝对的优劣之分，只有适用场景的不同。

**核心要点回顾：**

- 客户端负载均衡：决策在调用方本地，减少网络跳转，但增加调用方复杂度
- 服务端负载均衡：决策在独立组件，架构简单，但可能成为瓶颈
- 实际生产中，常采用混合模式：内部调用用客户端 LB，边界用服务端 LB
- 选择时考虑：延迟敏感度、运维复杂度、扩展性需求
