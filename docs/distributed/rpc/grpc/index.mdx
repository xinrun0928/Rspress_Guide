# gRPC：Protocol Buffers 与 HTTP/2

你有没有想过这个问题：

REST API 用 JSON，Dubbo 用 Hessian/Protobuf。

但如果我告诉你，有一种方案可以比 REST 快 5-10 倍，同时支持双向流式调用，还自动生成多语言代码——你会不会觉得太完美了？

这就是 **gRPC**。

## 为什么选择 gRPC

```
┌─────────────────────────────────────────────────────────┐
│                  gRPC 核心优势                           │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  1. HTTP/2 多路复用                                     │
│     └─ 一条连接并行传输多个请求，解决队头阻塞            │
│                                                         │
│  2. Protocol Buffers 序列化                            │
│     └─ 比 JSON 小 5-20 倍，快 5-10 倍                   │
│                                                         │
│  3. 强类型接口定义                                      │
│     └─ .proto 文件定义，自动生成多语言代码               │
│                                                         │
│  4. 四种 RPC 模式                                       │
│     └─ 一元、流式服务端、流式客户端、双向流              │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

## ProtoBuf vs JSON

```protobuf
// .proto 定义
syntax = "proto3";

service UserService {
    rpc GetUser (UserRequest) returns (UserResponse);
    rpc StreamUsers (stream UserRequest) returns (stream UserResponse);
}

message UserRequest { int64 id = 1; }
message UserResponse {
    int64 id = 1;
    string name = 2;
}
```

| 维度 | JSON | ProtoBuf |
|------|------|---------|
| 体积 | 大（含字段名） | 小（只用编号） |
| 速度 | 中 | 快 5-10 倍 |
| 类型安全 | 弱 | 强 |
| 可读性 | 好 | 差 |

## gRPC vs REST

| 维度 | REST | gRPC |
|------|------|------|
| 协议 | HTTP/1.1 | HTTP/2 |
| 数据格式 | JSON | ProtoBuf |
| 性能 | 中 | 高（5-10 倍） |
| 流式支持 | 不原生 | 支持四种模式 |
| 浏览器支持 | 原生 | 需 gRPC-Web |
| 代码生成 | Swagger | .proto 自动生成 |

## 核心公式

```
gRPC = HTTP/2 + ProtoBuf + 多语言代码生成 + 四种RPC模式

适用场景：
- 微服务内部高性能通信
- 跨语言微服务互通
- 实时流处理（直播、IoT）
- 移动端与后端通信

不适用：
- 浏览器直接访问（需要 gRPC-Web）
- 人类可读数据场景
```
