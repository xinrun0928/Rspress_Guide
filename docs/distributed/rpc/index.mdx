# 服务调用：Dubbo vs Feign vs gRPC

你有没有想过这个问题：

用户点击下单，后端经过订单服务 → 库存服务 → 支付服务 → 物流服务，一共 4 个微服务。

一个 HTTP 请求，怎么准确地找到对应的服务？服务之间怎么知道彼此的地址？

**这就是服务调用要解决的问题。**

## 三种通信模式对比

| 维度 | Dubbo | Feign | gRPC |
|------|-------|-------|------|
| 通信协议 | TCP (Dubbo 协议) | HTTP | HTTP/2 |
| 序列化 | Hessian / Protobuf | JSON | Protobuf |
| 性能 | 最高 | 中 | 高 |
| 侵入性 | 中（注解） | 低（声明式） | 高（IDL） |
| 跨语言 | 困难 | 容易 | 容易 |
| 适用场景 | Java 微服务内部 | Spring Cloud | 跨语言 / 微服务互通 |

## 选型决策

```
┌─────────────────────────────────────────────────────────┐
│                    服务调用选型树                        │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  你的团队用什么技术栈？                                  │
│  ├─ Spring Cloud → OpenFeign（声明式，最简洁）         │
│  ├─ 高性能 Java 服务 → Dubbo（TCP + 自定义协议）        │
│  └─ 跨语言互通 → gRPC（Protobuf + HTTP/2）             │
│                                                         │
│  你追求的是什么？                                        │
│  ├─ 开发效率 → OpenFeign                                │
│  ├─ 极致性能 → Dubbo                                   │
│  └─ 跨平台互通 → gRPC                                  │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

## 核心公式

```
服务调用 = 服务发现（注册中心）+ 负载均衡 + 网络通信 + 序列化

选型依据：
- Spring Cloud 生态 → OpenFeign
- 高性能 Java 服务 → Dubbo
- 跨语言高性能通信 → gRPC
- 外部 RESTful API → OpenFeign
```
