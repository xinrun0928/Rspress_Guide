# 服务调用

想象一下：你正在构建一个电商系统，用户下单需要调用库存服务扣减库存、调用支付服务完成支付、调用物流服务创建配送单。这些服务分布在不同的机器上，甚至可能是不同的技术栈。

单体架构时代，你只需要一个函数调用。但在微服务时代，你需要一种方式，让这个调用**像本地调用一样简单，同时具备分布式的能力**。

这就是服务调用（RPC）要解决的问题。

## 什么是 RPC

RPC（Remote Procedure Call，远程过程调用）是一种让程序能够调用远程服务器上的函数或方法的技术。对调用方来说，这个调用和本地调用几乎没有区别——你不需要关心数据是如何序列化传输的，也不必关心网络连接是如何建立的。

```yaml
# 本地调用 vs 远程调用
# 本地调用
result = userService.getUser(userId)
# 远程调用
result = userService.getUser(userId)  // 看起来完全一样
```

RpcContext context = RpcContext.getContext();
context.setAttachment("index", String.valueOf(i++));

String result = demoService.hello("world" + i);
System.out.println("result:" + result);

### RPC 框架的核心组件

一个完整的 RPC 框架需要包含以下核心组件：

```yaml
# RPC 框架核心架构
┌─────────────────────────────────────────────────────────────┐
│                       RPC 框架核心组件                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 通信框架                                                 │
│     - BIO / NIO / Netty                                     │
│     - 负责网络传输                                           │
│                                                             │
│  2. 序列化框架                                               │
│     - JDK 序列化 / Hessian / JSON / Protobuf                │
│     - 负责对象与字节流的相互转换                               │
│                                                             │
│  3. 代理层                                                  │
│     - JDK 动态代理 / Javassist / CGLIB                      │
│     - 屏蔽网络调用细节，对调用方透明                           │
│                                                             │
│  4. 注册与发现                                               │
│     - Zookeeper / Consul / Nacos / Eureka                   │
│     - 负责服务注册与发现                                      │
│                                                             │
│  5. 负载均衡                                                 │
│     - Random / RoundRobin / LeastActive                     │
│     - 负责选择目标服务实例                                    │
│                                                             │
│  6. 容错机制                                                 │
│     - Failover / Failsafe / Failfast                       │
│     - 处理调用失败的情况                                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 主流 RPC 框架对比

```yaml
# 主流 RPC 框架对比
框架          开发语言   序列化       传输协议      特点
─────────────────────────────────────────────────────────────────
Dubbo          Java      Hessian/Protobuf  TCP/HTTP    功能全面，阿里出品，国内使用广泛
Feign          Java      JSON              HTTP        声明式 API，Spring Cloud 标配
gRPC           多语言     Protobuf          HTTP/2      高性能，Google 背书，跨语言
Thrift         多语言     Thrift            TCP         早期跨语言 RPC 方案
```

:::tip 选择建议
- Java 技术栈 + Spring Cloud：选 Feign
- Java 技术栈 + 高性能要求：选 Dubbo
- 跨语言 + 高性能：选 gRPC
- 老系统维护：选 Thrift
:::
