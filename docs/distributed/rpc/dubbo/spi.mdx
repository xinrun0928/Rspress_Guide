# Dubbo 过滤器链与扩展机制（SPI）

为什么 Spring 可以通过 `@Autowired` 注入 Bean，而 Dubbo 可以通过 `@DubboReference` 注入远程服务？为什么 Dubbo 的序列化、负载均衡、协议都可以替换？

答案都藏在同一个机制里：**SPI**。

## SPI 是什么？

SPI（Service Provider Interface）是 Java 提供的一种扩展机制，全称是「服务提供接口」。它的核心思想是：**接口由框架定义，实现由第三方提供**。

JDK 的 SPI 很简单：

```java
// 1. 定义接口
public interface DataStore {
    void save(String key, String value);
    String get(String key);
}

// 2. 实现接口
public class RedisDataStore implements DataStore {
    @Override
    public void save(String key, String value) {
        // Redis 实现
    }
}

// 3. 在 META-INF/services/ 下创建文件
// 文件名：com.example.DataStore
// 内容：com.example.RedisDataStore

// 4. 使用 ServiceLoader 加载
ServiceLoader<DataStore> loader = ServiceLoader.load(DataStore.class);
DataStore store = loader.iterator().next();
```

但 JDK SPI 有个问题：**会一次性加载所有实现，无法按需加载**。如果你的实现类有几十个，加载就会很慢。

## Dubbo 的 SPI 改进

Dubbo 重写了 SPI 机制，实现了**按需加载**和**自动注入**：

```java
// Dubbo SPI 使用示例
@SPI
public interface Protocol {
    @Adaptive
    <T> Exporter<T> export(Invoker<T> invoker);

    @Adaptive
    <T> Invoker<T> refer(Class<T> type, URL url);
}
```

### 核心注解解析

| 注解 | 作用 |
|---|---|
| `@SPI` | 标记这个接口是 SPI 接口，需要 Dubbo 扩展 |
| `@Adaptive` | 标记在接口方法上，Dubbo 会生成适配器类，根据 URL 参数动态选择实现 |
| `@Adaptive`（类级别） | 标记在实现类上，表示这是默认的适配实现 |

### Dubbo SPI 的配置文件

```properties
# META-INF/dubbo/internal/com.example.Protocol
dubbo=com.apache.dubbo.rpc.protocol.dubbo.DubboProtocol
grpc=com.apache.dubbo.rpc.protocol.grpc.GrpcProtocol
injvm=com.apache.dubbo.rpc.protocol.injvm.InjvmProtocol
```

## 过滤器链：拦截每一次调用

Dubbo 的过滤器链是 SPI 扩展的典型应用。每一个 RPC 调用，都会经过过滤器链：

```java
// 过滤器链的执行流程
public class FilterChain {
    private final List<Filter> filters;
    private final Invoker<T> invoker;

    public Result invoke(Invocation invocation) {
        // 1. 正向执行：过滤器的 preHandle
        Result result = null;
        int i = 0;
        try {
            // 从第一个过滤器开始
            Result preResult = filters.get(i).invoke(invocation, nextFilter(i + 1));
            // ... 依次执行
        } catch (Exception e) {
            // 2. 异常处理
        } finally {
            // 3. 反向执行：过滤器的 postHandle
        }
        return result;
    }
}
```

### 内置过滤器

| 过滤器 | 功能 |
|---|---|
| `EchoFilter` | 回声测试，检测服务是否存活 |
| `ClassLoaderFilter` | 切换线程上下文类加载器 |
| `ContextFilter` | 传递 RpcContext 上下文 |
| `TraceFilter` | 记录调用链路 |
| `TimeoutFilter` | 记录超时调用 |
| `ExceptionFilter` | 统一异常处理 |
| `AccessLogFilter` | 记录访问日志 |
| `LimitFilter` | 限流控制 |

### 自定义过滤器

```java
// 1. 创建过滤器实现
@Activate(group = Constants.PROVIDER, order = -100)
public class CustomFilter implements Filter {

    private final Logger logger = LoggerFactory.getLogger(CustomFilter.class);

    @Override
    public Result invoke(Invoker<?> invoker, Invocation invocation) throws RpcException {
        long startTime = System.currentTimeMillis();
        String methodName = invocation.getMethodName();
        String interfaceName = invoker.getInterface().getName();

        // 1. 前置逻辑：参数校验
        Object[] arguments = invocation.getArguments();
        if (!validateParams(arguments)) {
            return new RpcResult(new IllegalArgumentException("参数校验失败"));
        }

        // 2. 执行调用
        Result result;
        try {
            result = invoker.invoke(invocation);
        } catch (Exception e) {
            // 3. 异常处理
            handleException(e, methodName, interfaceName);
            throw e;
        }

        // 4. 后置逻辑：记录日志
        long cost = System.currentTimeMillis() - startTime;
        logger.info("调用方法: {}.{} 耗时: {}ms", interfaceName, methodName, cost);

        return result;
    }

    private boolean validateParams(Object[] arguments) {
        // 参数校验逻辑
        return true;
    }

    private void handleException(Exception e, String method, String interfaceName) {
        logger.error("调用异常: {}.{}", interfaceName, method, e);
    }
}
```

### 过滤器的执行顺序


### 负载均衡扩展

```java
@SPI(RandomLoadBalance.NAME)
public interface LoadBalance {
    @Adaptive("loadbalance")
    <T> Invoker<T> select(List<Invoker<T>> invokers, URL url, Invocation invocation);
}

// 实现随机负载均衡
public class RandomLoadBalance extends AbstractLoadBalance {
    public static final String NAME = "random";

    @Override
    protected <T> Invoker<T> doSelect(List<Invoker<T>> invokers, URL url, Invocation invocation) {
        int length = invokers.size();
        boolean sameWeight = true;
        int[] weights = new int[length];
        int totalWeight = 0;

        for (int i = 0; i < length; i++) {
            int weight = getWeight(invokers.get(i), invocation);
            totalWeight += weight;
            weights[i] = totalWeight;
            if (sameWeight && i > 0 && weight != weights[i - 1]) {
                sameWeight = false;
            }
        }

        if (totalWeight > 0 && !sameWeight) {
            int offset = ThreadLocalRandom.current().nextInt(totalWeight);
            for (int i = 0; i < length; i++) {
                if (offset < weights[i]) {
                    return invokers.get(i);
                }
            }
        }

        return invokers.get(ThreadLocalRandom.current().nextInt(length));
    }
}
```

## 自适应扩展：动态选择实现

`@Adaptive` 是 Dubbo SPI 最强大的特性。它允许 Dubbo 根据 URL 参数，动态生成适配器代码：

```java
public interface Optimizer {
    // 这个方法会生成适配器，根据 url 中的 optimizer 参数选择实现
    @Adaptive
    Object optimize(URL url, Object data);
}

// Dubbo 生成的伪代码
public class Optimizer$Adaptive implements Optimizer {
    public Object optimize(Invoker<?> invoker, URL url) {
        String optimizerName = url.getParameter("optimizer");

        if (optimizerName == null) {
            optimizerName = "default";
        }

        Optimizer optimizer = ExtensionLoader.getExtensionLoader(Optimizer.class)
            .getExtension(optimizerName);

        return optimizer.optimize(url, data);
    }
}
```

## 扩展加载机制源码解析

```java
public class ExtensionLoader<T> {

    private static final Map<Class<?>, ExtensionLoader<?>> EXTENSION_LOADERS = new ConcurrentHashMap<>();
    private static final Map<Class<?>, Object> EXTENSION_INSTANCES = new ConcurrentHashMap<>();

    public T getExtension(String name) {
        if ("*".equals(name)) {
            return getDefaultExtension();
        }

        // 1. 查找缓存
        T instance = cachedInstances.get(name);
        if (instance == null) {
            synchronized (cachedInstances) {
                instance = cachedInstances.get(name);
                if (instance == null) {
                    // 2. 加载并创建实例
                    instance = createExtension(name);
                    cachedInstances.put(name, instance);
                }
            }
        }
        return instance;
    }

    private T createExtension(String name) {
        // 1. 加载类
        Class<?> clazz = getExtensionClasses().get(name);

        // 2. 创建实例
        T instance = (T) clazz.newInstance();

        // 3. 依赖注入（自动注入其他 SPI 扩展）
        injectExtension(instance);

        return instance;
    }

    // 依赖注入：通过 setter 方法注入其他扩展
    private T injectExtension(T instance) {
        for (Method method : instance.getClass().getMethods()) {
            if (method.getName().startsWith("set")
                    && method.getParameterTypes().length == 1) {
                // 查找对应的 SPI 扩展并注入
                Class<?> pt = method.getParameterTypes()[0];
                Object adaptiveExtension = getAdaptiveExtension();
                method.invoke(instance, adaptiveExtension);
            }
        }
        return instance;
    }
}
```

## 总结

Dubbo 的 SPI 扩展机制，是它区别于传统 RPC 框架的核心优势：

| 特性 | JDK SPI | Dubbo SPI |
|---|---|---|
| 加载时机 | 全量加载 | 按需加载 |
| 依赖注入 | 不支持 | 支持 |
| 自适应 | 不支持 | `@Adaptive` 支持 |
| 配置方式 | 配置文件 | 注解 + 配置文件 |
| 扩展点数量 | 少 | 几十个 |

掌握 SPI，你就掌握了 Dubbo 扩展的大门。想替换序列化？实现 Serialization。想自定义负载均衡？实现 LoadBalance。一切皆可扩展。
