# Dubbo SPI 扩展机制

你需要在 Dubbo 中自定义一个负载均衡算法，或者换一个序列化方式。正常来说需要修改 Dubbo 源码。

但 Dubbo 告诉你：不用。只需要按照约定写一个配置文件，你的实现就会自动被加载。

这就是 SPI（Service Provider Interface）机制的力量。

## 什么是 SPI

SPI 全称是 Service Provider Interface，是一种服务发现机制。它的核心思想是：**定义接口，具体实现由第三方提供。**

```yaml
# SPI 机制原理
┌─────────────────────────────────────────────────────────────┐
│                     SPI 机制原理                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 定义接口                                                │
│     ┌─────────────────────────────────────────────────┐   │
│     │  package com.example;                           │   │
│     │  public interface DataStore {                  │   │
│     │      void save(String key, String value);      │   │
│     │  }                                             │   │
│     └─────────────────────────────────────────────────┘   │
│                                                             │
│  2. 实现接口                                                │
│     ┌─────────────────────────────────────────────────┐   │
│     │  package com.example.impl;                     │   │
│     │  public class RedisDataStore implements DataStore {   │   │
│     │      public void save(String key, String value) {     │   │
│     │          // Redis 实现                           │   │
│     │      }                                          │   │
│     │  }                                              │   │
│     └─────────────────────────────────────────────────┘   │
│                                                             │
│  3. 配置实现类                                              │
│     ┌─────────────────────────────────────────────────┐   │
│     │  META-INF/services/com.example.DataStore        │   │
│     │  内容：com.example.impl.RedisDataStore          │   │
│     └─────────────────────────────────────────────────┘   │
│                                                             │
│  4. 加载使用                                                │
│     ┌─────────────────────────────────────────────────┐   │
│     │  DataStore store = ExtensionLoader              │   │
│     │      .getExtensionLoader(DataStore.class)       │   │
│     │      .getExtension("redis");                   │   │
│     └─────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## Dubbo SPI vs JDK SPI

Dubbo 重新实现了 SPI，弥补了 JDK SPI 的不足：

```yaml
# JDK SPI vs Dubbo SPI
特性              JDK SPI                Dubbo SPI
─────────────────────────────────────────────────────────────────
配置文件位置       META-INF/services/     META-INF/dubbo/,
                                       META-INF/dubbo/internal/
                                       META-INF/services/
实例化           一次性加载所有实现       按需加载
扩展方式         只支持配置              支持注解 + 配置
依赖注入         不支持                  支持
AOP/Decorator    不支持                  支持
自适应扩展       不支持                  支持
```

## Dubbo SPI 核心注解

```java
// @SPI 注解：标记接口是可扩展的
@SPI
public interface LoadBalance {
    <T> Invoker<T> select(List<Invoker<T>> invokers);
}

// @Adaptive 注解：标记实现类是自适应实现
@Adaptive
public class AdaptiveLoadBalance implements LoadBalance {
    @Override
    public <T> Invoker<T> select(List<Invoker<T>> invokers) {
        // 根据 URL 参数动态选择具体实现
        URL url = RpcContext.getContext().getUrl();
        String loadbalanceName = url.getParameter("loadbalance");
        LoadBalance loadbalance = ExtensionLoader
            .getExtensionLoader(LoadBalance.class)
            .getExtension(loadbalanceName);
        return loadbalance.select(invokers);
    }
}

// @Activate 注解：标记实现类会被自动激活
@Activate(group = Constants.PROVIDER, order = 2)
public class CacheFilter implements Filter {
    // ...
}
```

## Dubbo SPI 配置文件

### 配置位置

```yaml
# Dubbo SPI 配置文件位置（按优先级排序）
1. META-INF/dubbo/internal/        # Dubbo 内部实现
2. META-INF/dubbo/                 # Dubbo 扩展（用户定义）
3. META-INF/services/               # 兼容 JDK SPI
```

### 配置文件格式

```properties
# 文件名：接口全限定名
# 内容：key=value

```

## 自定义扩展

### 示例一：自定义序列化

```java
// 1. 定义接口
package com.example.serialization;

@SPI("fastjson")
public interface Serialization {
    void serialize(OutputStream output, Object data) throws IOException;
    Object deserialize(InputStream input) throws IOException;
}

// 2. 实现接口
package com.example.serialization.impl;

public class FastJsonSerialization implements Serialization {
    @Override
    public void serialize(OutputStream output, Object data) throws IOException {
        String json = JSON.toJSONString(data);
        output.write(json.getBytes());
    }

    @Override
    public Object deserialize(InputStream input) throws IOException {
        byte[] bytes = input.readAllBytes();
        return JSON.parseObject(bytes, Object.class);
    }
}

// 3. 创建配置文件
# META-INF/dubbo/com.example.serialization.Serialization
fastjson=com.example.serialization.impl.FastJsonSerialization

// 4. 使用
@Configuration
public class DubboConfig {
    @Bean
    public ProtocolConfig protocol() {
        ProtocolConfig config = new ProtocolConfig();
        config.setName("dubbo");
        config.setSerialization("fastjson");  // 指定使用 fastjson
        return config;
    }
}
```

### 示例二：自定义负载均衡

```java
// 1. 实现接口
package com.example.loadbalance;

public class WeightedResponseTimeLoadBalance extends AbstractLoadBalance {

    private ConcurrentHashMap<String, AtomicLong> responseTimes = new ConcurrentHashMap<>();

    @Override
    protected <T> Invoker<T> doSelect(List<Invoker<T>> invokers, URL url, Invocation invocation) {
        // 根据响应时间加权选择
        long totalWeight = 0;
        long currentTime = System.currentTimeMillis();

        List<Invoker<T>> weightedInvokers = new ArrayList<>();
        for (Invoker<T> invoker : invokers) {
            String key = invoker.getUrl().getAddress();
            AtomicLong time = responseTimes.computeIfAbsent(key, k -> new AtomicLong(0));

            // 响应时间越长，权重越低
            long responseTime = time.get();
            long weight = Math.max(1, 10000 - responseTime);
            totalWeight += weight;

            for (int i = 0; i < weight; i++) {
                weightedInvokers.add(invoker);
            }
        }

        return weightedInvokers.get(
            ThreadLocalRandom.current().nextInt(weightedInvokers.size())
        );
    }

    public void recordResponse(String address, long responseTime) {
        responseTimes.get(address).set(responseTime);
    }
}

// 2. 创建配置文件
# META-INF/dubbo/com.alibaba.dubbo.rpc.cluster.LoadBalance
weighted-response-time=com.example.loadbalance.WeightedResponseTimeLoadBalance

// 3. 使用
@Reference(loadbalance = "weighted-response-time")
private UserService userService;
```

### 示例三：自定义过滤器

```java
// 1. 实现 Filter 接口
@Activate(group = {Constants.CONSUMER, Constants.PROVIDER})
public class TracingFilter implements Filter {

    private Tracer tracer;

    @Override
    public Result invoke(Invoker<?> invoker, Invocation invocation) throws RpcException {
        // 开始Span
        Span span = tracer.startSpan(invocation.getMethodName());
        try (Scope scope = tracer.withSpan(span)) {
            // 设置标签
            span.setTag("service", invoker.getUrl().getServiceInterface());
            span.setTag("method", invocation.getMethodName());

            // 执行调用
            Result result = invoker.invoke(invocation);

            // 记录结果
            if (result.hasException()) {
                span.setTag("error", result.getException().getMessage());
            }

            return result;
        } finally {
            span.finish();
        }
    }
}

// 2. 创建配置文件
# META-INF/dubbo/com.alibaba.dubbo.rpc.Filter
tracing=com.example.filter.TracingFilter

// 3. 配置启用
dubbo:
  provider:
    filter: tracing
  consumer:
    filter: tracing
```

## 自适应扩展（Adaptive Extension）

自适应扩展是 Dubbo SPI 的高级特性，可以根据调用上下文动态选择具体实现。

```java
// 1. 定义接口
@SPI
public interface Protocol {
    @Adaptive
    <T> Exporter<T> export(Invoker<T> invoker);
    <T> Invoker<T> refer(Class<T> type, URL url);
}

// 2. 自适应实现
// Dubbo 会自动生成一个动态适配器，根据 URL 参数选择具体实现
// 例如：url = dubbo://192.168.1.1:20880?protocol=dubbo
// 会自动选择 dubbo 协议的实现

// 3. 手动指定
@Reference(protocol = "rest")
private UserService userService;
```

## 扩展点加载顺序

```yaml
# 扩展点加载顺序
1. 先加载内置扩展
   - META-INF/dubbo/internal/

2. 再加载外部扩展
   - META-INF/dubbo/

3. 最后加载 JDK SPI 兼容扩展
   - META-INF/services/

# 后加载的会覆盖先加载的同名扩展
```

## 最佳实践

### 实践一：使用注解而非硬编码

```java
// 推荐：使用注解
@SPI("fastjson")
public interface Serialization {
    void serialize(OutputStream output, Object data);
}

// 不推荐：硬编码实现类
Serialization serialization = new FastJsonSerialization();
```

### 实践二：合理使用 @Activate

```java
// @Activate 注解控制自动激活条件
@Activate(
    group = Constants.PROVIDER,           // 在 Provider 端激活
    value = {"cache", "validation"},      // 在指定配置值时激活
    order = 1                              // 激活顺序
)
public class CacheFilter implements Filter {
    // ...
}
```

### 实践三：使用自适应扩展

```java
// 使用 @Adaptive 让框架自动选择实现
// 框架会根据 URL 参数生成适配代码
// 无需手动判断协议、负载均衡类型等
public class ClusterInvoker<T> {

    // 框架会自动注入 AdaptiveLoadBalance
    private LoadBalance loadBalance;

    public Invoker<T> select(List<Invoker<T>> invokers) {
        // 根据 URL 参数动态选择
        return loadBalance.select(invokers);
    }
}
```

## 总结

Dubbo SPI 是 Dubbo 可扩展性的基石。通过 SPI 机制，Dubbo 实现了零侵入式的框架扩展。

**核心要点回顾：**

- SPI 是一种服务发现机制，允许第三方提供实现
- Dubbo SPI 相比 JDK SPI 支持按需加载、自适应扩展、Activate 注解
- 扩展点通过配置文件声明
- @SPI 标记接口可扩展，@Adaptive 生成适配代码，@Activate 自动激活
- 常见扩展点：Protocol、Serialization、LoadBalance、Cluster、Filter 等
