# Dubbo 服务导出（Export）与引用（Refer）流程

凌晨 3 点，你正准备上线新版本。服务启动成功了，消费者也连上来了，但调用时却返回 "No provider"。检查了一圈，发现服务确实启动了。那问题出在哪？

大概率是**服务导出（Export）的时机不对**，或者**导出失败了但你没看到错误日志**。

Dubbo 的服务导出与引用是一个复杂的过程，涉及注册中心、协议层、网络通信等多个环节。理解这个流程，是掌握 Dubbo 的基础。

## 服务导出：从何而来

### 导出触发的时机

Dubbo 服务有三种导出方式：

```java
// 方式一：配置声明式导出（推荐）
@DubboService
public class UserServiceImpl implements UserService {
    // 方法实现
}

// 方式二：API 手动导出
public void exportService() {
    ServiceConfig<UserService> service = new ServiceConfig<>();
    service.setInterface(UserService.class);
    service.setRef(new UserServiceImpl());
    service.setRegistryConfig(new RegistryConfig("zookeeper://127.0.0.1:2181"));
    service.export();  // 手动触发导出
}

// 方式三：延迟导出（等待首次访问时导出）
service.setDelay(5000);  // 启动 5 秒后导出
```

### 导出的核心流程

```java
public synchronized void export() {
    // 1. 检查配置，补全默认值
    checkAndUpdateSubConfigs();

    // 2. 构建 URL（这是理解 Dubbo 的关键）
    URL serviceUrl = buildURL();

    // 3. 遍历所有协议，导出服务
    for (Protocol protocol : protocols) {
        // 4. 服务暴露：启动 Netty 服务器，监听端口
        Exporter<?> exporter = protocol.export(
            ProxyFactory.getInvoker(ref, interfaceClass, serviceUrl)
        );
        exporters.add(exporter);
    }

    // 5. 注册到注册中心
    register();
}
```

### URL 是理解 Dubbo 的钥匙

Dubbo 中，一切皆 URL。这个 URL 包含了服务的所有信息：

```java
// 典型的服务 URL
dubbo://192.168.1.100:20880/com.example.UserService?timeout=3000&version=1.0.0&methods=getUser,saveUser

// 拆解后的结构
// protocol: dubbo
// host: 192.168.1.100
// port: 20880
// path: com.example.UserService
// parameters: timeout=3000&version=1.0.0&methods=getUser,saveUser
```

### 服务导出与注册时序

```java
public void register() {
    // 1. 构建注册 URL
    URL registerURL = serviceUrl.setPath(Constants.INTERFACE_PATH)
        .addParameters(Constants.CATEGORY_KEY, Constants.PROVIDERS_CATEGORY);

    // 2. 连接注册中心
    doRegister(registerURL);
}

private void doRegister(URL url) {
    // 注册到 Zookeeper 的流程：
    // /dubbo/com.example.UserService/providers/
    //   dubbo://192.168.1.100:20880/com.example.UserService?...
    registry.register(url);
}
```

## 服务引用：从消费者角度

### 引用的触发时机

```java
// 方式一：声明式引用（推荐）
@DubboReference
private UserService userService;

// 方式二：XML 配置
<dubbo:reference id="userService" interface="com.example.UserService" />

// 方式三：API 手动引用
public void referService() {
    ReferenceConfig<UserService> reference = new ReferenceConfig<>();
    reference.setInterface(UserService.class);
    reference.setRegistryConfig(new RegistryConfig("zookeeper://127.0.0.1:2181"));
    reference.setGroup("${dubbo.group}");
    reference.setVersion("1.0.0");

    // 获取代理对象，这才是真正可调用的
    UserService userService = reference.get();
    userService.getUserById(1L);
}
```

### 引用的核心流程

```java
public synchronized T get() {
    // 1. 检查配置
    checkAndUpdateSubConfigs();

    // 2. 构建引用 URL
    URL refUrl = buildURL();

    // 3. 订阅注册中心，获取提供者列表
    List<URL> providerUrls = subscribe();

    // 4. 遍历协议，创建代理
    for (Protocol protocol : protocols) {
        // 5. 引用服务：创建 Invoker，建立连接
        Invoker<T> invoker = protocol.refer(serviceType, refUrl);

        // 6. 生成代理：封装Invoker，屏蔽底层调用细节
        proxyFactory.getProxy(invoker);
    }
}
```

### 服务订阅与本地缓存

消费者订阅服务后，会把提供者列表缓存在本地：

```java
public class RegistryDirectory<T> implements NotifyListener {

    // 本地缓存：key=方法名，value=调用列表
    private volatile Map<String, List<Invoker<T>>> methodInvokerMap;

    @Override
    public void notify(List<URL> urls) {
        // 1. 按 category 分组（providers, configurators, routers）
        Map<String, List<URL>> categorized = categorize(urls);

        // 2. 更新本地缓存
        List<URL> providerUrls = categorized.get("providers");
        this.methodInvokerMap = toInvokers(providerUrls);

        // 3. 触发重新路由
        routerChain.setInvokers(this.methodInvokerMap.values());
    }
}
```

## 常见问题与排查

### 问题一：服务导出成功但注册失败

```java
// 检查注册中心连接
public boolean isRegistryConnected() {
    return registry != null && registry.isAvailable();
}

// 手动触发重注册
public void reExport() {
    exporters.forEach(Exporter::unexport);
    export();
}
```

### 问题二：引用时获取不到服务

```java
// 调试：打印当前订阅的提供者
public void debugProviders() {
    RegistryDirectory<?> directory = getDirectory();
    System.out.println("Cached invokers: " + directory.getAllInvokers());
    System.out.println("Method invokers: " + directory.getMethodInvokerMap());
```

### 问题三：版本号不匹配

```yaml
# 消费者配置
dubbo:
  reference:
    interface: com.example.UserService
    version: 1.0.0
    group: production

# 提供者配置
dubbo:
  service:
    interface: com.example.UserService
    version: 1.0.0
    group: production
```

### 问题四：启动顺序问题

服务提供者必须在消费者启动前完成导出和注册：

```java
// 确保 Provider 完全就绪后再启动 Consumer
public class ServiceReadyChecker {
    private final Registry registry;

    public void waitForService(String serviceName, int timeoutSeconds) {
        long deadline = System.currentTimeMillis() + timeoutSeconds * 1000L;
        while (System.currentTimeMillis() < deadline) {
            List<URL> providers = registry.lookup(
                URL.valueOf("dubbo://" + serviceName)
            );
            if (!providers.isEmpty()) {
                return;
            }
            Thread.sleep(100);
        }
        throw new RuntimeException("Service not ready: " + serviceName);
    }
}
```

## 异步导出与引用

Dubbo 支持异步导出，提高启动速度：

```yaml
dubbo:
  service:
    export: async  # 异步导出，不阻塞启动
  reference:
    init: false    # 延迟引用，首次调用时才建立连接
```

```java
// 编程式异步导出
public void asyncExport() {
    ServiceConfig<UserService> service = new ServiceConfig<>();
    service.setInterface(UserService.class);
    service.setRef(new UserServiceImpl());

    // 异步导出，不阻塞
    CompletableFuture<Void> exportFuture = service.asyncExport();

    // 主线程继续其他初始化
    doOtherInitialization();

    // 等待导出完成
    exportFuture.join();
}
```

## 总结

Dubbo 的服务导出与引用流程，可以用四个字概括：**注册发现**。

```
Provider 启动 → 导出服务 → 注册到中心
Consumer 启动 → 订阅服务 → 获取列表 → 创建代理
调用时 → 路由筛选 → 负载均衡 → 执行调用
```

理解这个流程，你就掌握了 Dubbo 的一半。另一半，是下一章要讲的 SPI 扩展机制。
