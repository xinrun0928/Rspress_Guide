# Dubbo：高性能 RPC 框架

你有没有注意到一个现象：

同样的微服务架构，用 Feign 调用延迟 10ms，用 Dubbo 调用延迟 2ms。

差 5 倍？

**这就是 RPC 框架和 HTTP 框架的本质区别。**

Dubbo 是阿里巴巴开源的高性能 RPC 框架，核心优势在于：TCP 协议、自定义报头、序列化优化。它是 Java 微服务追求极致性能的首选。

## 核心技术点

| 能力 | 说明 | 性能影响 |
|------|------|---------|
| 自定义协议 | TCP + 报头压缩，比 HTTP 协议更小 | 减少网络开销 |
| 多种序列化 | Hessian / Protobuf / FastJSON | 序列化速度影响延迟 |
| 线程模型 | DubboThread + IO 线程分离 | 提升并发处理能力 |
| 负载均衡 | 5 种策略：随机/轮询/最小活跃/哈希/最短响应 | 流量分配合理性 |
| 集群容错 | 失败自动切换/快速失败/广播 | 系统稳定性 |

## Dubbo vs Feign

| 维度 | Dubbo | Feign |
|------|-------|-------|
| 协议 | TCP (Dubbo 协议) | HTTP |
| 序列化 | Hessian / Protobuf | JSON |
| 性能 | 高（2-5ms） | 中（10-20ms） |
| 侵入性 | 中（注解） | 低（声明式） |
| 生态 | 独立生态 | Spring Cloud |
| 适用场景 | 高性能 Java 服务 | Spring Cloud 生态 |

## 核心公式

```
Dubbo = 自定义TCP协议 + 高效序列化 + Netty通信 + SPI扩展

性能优化方向：
- 协议：Hessian > FastJSON > JDK 序列化
- 连接：长连接复用，避免频繁建连
- 线程：合理配置线程池大小
- 路由：标签路由实现灰度发布

选型建议：
- 高性能 Java 微服务 → Dubbo
- Spring Cloud 生态 → OpenFeign
- 需要跨语言 → gRPC
```
