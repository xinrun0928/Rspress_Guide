# Dubbo 线程模型

你部署的 Dubbo 服务 QPS 只有几千，但 CPU 使用率已经爆表了。线程池满了，请求堆积，甚至开始 OOM。

问题出在哪？线程模型配置不当。

Dubbo 的线程模型直接决定了服务能承受的并发量。理解线程模型，是优化 Dubbo 性能的第一步。

## Dubbo 线程模型概述

Dubbo 采用的是 **Reactor 模式** 的线程模型，通过分层处理，实现高并发。

```yaml
# Dubbo 线程模型分层
┌─────────────────────────────────────────────────────────────┐
│                     Dubbo 线程模型                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                  Boss Group (Acceptor)               │   │
│  │  - 负责 Accept 新连接                               │   │
│  │  - 通常只有 1-2 个线程                              │   │
│  └────────────────────────┬────────────────────────────┘   │
│                           │                                 │
│                           ▼                                 │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                  Worker Group                        │   │
│  │  - 负责 IO 读写                                    │   │
│  │  - 通常是 CPU 核心数的 2 倍                         │   │
│  └────────────────────────┬────────────────────────────┘   │
│                           │                                 │
│                           ▼                                 │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                 Dubbo Thread Pool                   │   │
│  │  - 负责处理业务逻辑                                 │   │
│  │  - 可配置线程池大小                                 │   │

│  ┌─────────────────────────────────────────────────────┐   │
│  │                 Service Implementation              │   │
│  │  - 用户业务代码                                     │   │
│  │  - 执行实际的业务逻辑                               │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 线程池类型

Dubbo 支持多种线程池类型，适用于不同的场景：

```yaml
# Dubbo 线程池类型
类型              类名                       特点
─────────────────────────────────────────────────────────────────
fixed             FixedThreadPool            固定大小线程池
cached            CachedThreadPool           弹性线程池
limited           LimitedThreadPool          固定 + 无界队列
eager             EagerThreadPool           预创建线程池
                   (Dubbo 3.x 新增)          请求过来时优先创建新线程
```

### 1. FixedThreadPool - 固定大小线程池

```java
// FixedThreadPool 实现
public class FixedThreadPool implements ThreadPool {

    @Override
    public Executor getExecutor(URL url) {
        int threads = url.getParameter(THREADS_KEY, 200);
        int queues = url.getParameter(QUEUES_KEY, 0);

        return new ThreadPoolExecutor(
            threads,              // corePoolSize
            threads,              // maximumPoolSize（相同 = 固定大小）
            0,                    // keepAliveTime
            TimeUnit.MILLISECONDS,
            queues > 0 ? new LinkedBlockingQueue<>(queues) : new SynchronousQueue<>(),
            new NamedThreadFactory("DubboServerHandler", true),
            new AbortPolicy()
        );
    }
}
```

**特点**：

- 线程数固定，不会增长或收缩
- 适合 CPU 密集型场景
- 队列大小可配置

**配置**：

```yaml
dubbo:
  protocol:
    threads: 200
    queues: 0
```

### 2. CachedThreadPool - 弹性线程池

```java
// CachedThreadPool 实现
public class CachedThreadPool implements ThreadPool {

    @Override
    public Executor getExecutor(URL url) {
        int threads = url.getParameter(THREADS_KEY, Integer.MAX_VALUE);
        int queues = url.getParameter(QUEUES_KEY, 0);

        return new ThreadPoolExecutor(
            0,                    // corePoolSize（允许为空）
            threads,              // maximumPoolSize
            60L,                  // keepAliveTime
            TimeUnit.SECONDS,
            queues > 0 ? new LinkedBlockingQueue<>(queues) : new SynchronousQueue<>(),
            new NamedThreadFactory("DubboServerHandler", true),
            new AbortPolicy()
        );
    }
}
```

**特点**：

- 线程数可动态增长
- 适合 IO 密集型场景
- 空闲线程 60 秒后被回收

### 3. LimitedThreadPool - 固定 + 无界队列

```java
// LimitedThreadPool 实现
public class LimitedThreadPool implements ThreadPool {

    @Override
    public Executor getExecutor(URL url) {
        int threads = url.getParameter(THREADS_KEY, 200);

        return new ThreadPoolExecutor(
            threads,              // corePoolSize
            threads,              // maximumPoolSize
            0,                    // keepAliveTime（永不回收）
            TimeUnit.MILLISECONDS,
            new LinkedBlockingQueue<>(Integer.MAX_VALUE),  // 无界队列
            new NamedThreadFactory("DubboServerHandler", true),
            new AbortPolicy()
        );
    }
}
```

**特点**：

- 线程数固定
- 队列无界
- 适合流量平稳的场景

### 4. EagerThreadPool - 预创建线程池（Dubbo 3.x 新增）

```java
// EagerThreadPool 实现
public class EagerThreadPool implements ThreadPool {

    @Override
    public Executor getExecutor(URL url) {
        int threads = url.getParameter(THREADS_KEY, 200);
        int queues = url.getParameter(QUEUES_KEY, 100);

        return new ThreadPoolExecutor(
            threads,              // corePoolSize
            Integer.MAX_VALUE,    // maximumPoolSize
            60L,                  // keepAliveTime
            TimeUnit.SECONDS,
            new TaskQueue(queues),  // 自定义队列
            new NamedThreadFactory("DubboServerHandler", true),
            new AbortPolicy()
        );
    }
}
```

**特点**：

- 优先创建新线程，而不是放入队列
- 适合高并发场景
- 性能更好

## 线程模型配置

### 基本配置

```yaml
# application.yml
dubbo:
  protocol:
    name: dubbo
    port: 20880
    threads: 200              # 业务线程池大小
    queues: 0                 # 线程池队列大小
    iothreads: 8              # IO 线程数
    threadpool: fixed         # 线程池类型
```

### 不同场景的配置建议

```yaml
# 线程池配置建议
场景                  推荐配置                    说明
─────────────────────────────────────────────────────────────────
CPU 密集型            threads = CPU * 2          IO 少，CPU 计算多
IO 密集型            threads = CPU * 2 + 内存考虑  等待 IO 时 CPU 空闲
高并发低延迟          threads = CPU * 2 + 高      追求极致吞吐
高并发高吞吐          threads = CPU * 2 + 保守    追求稳定吞吐
```

### 完整配置示例

```yaml
# 完整线程池配置
dubbo:
  protocol:
    name: dubbo
    port: 20880

    # 线程池配置
    threadpool: eager              # 线程池类型
    threads: 200                   # 业务线程池大小
    queues: 100                    # 队列大小
    iothreads: 8                  # IO 线程数（Netty Boss/Worker）
    buffer: 32768                 # 读写缓冲区大小（字节）

  provider:
    # 线程池配置（服务端）
    threadpool: eager
    threads: 200
    queues: 100

  consumer:
    # 消费者线程池配置（通常不需要）
    threadpool: fixed
    threads: 100
```

## 线程模型与调用模式

Dubbo 支持同步调用和异步调用，两者的线程模型不同：

### 同步调用

```java
// 同步调用线程模型
┌─────────────────────────────────────────────────────────────┐
│                     同步调用线程模型                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Consumer                                                   │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ Thread-1 (调用线程)                                  │   │
│  │   userService.getUser(id);  ← 阻塞等待               │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  ────────────────────────────────────────────────────────  │
│                        网络                                  │
│  ────────────────────────────────────────────────────────  │
│                                                             │
│  Provider                                                   │

│  │ DubboThreadPool                                     │   │

│  │     → UserService.getUser()  ← 执行业务逻辑          │   │
│  │   DubboServerHandler-2                              │   │

│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 异步调用

```java
// 异步调用
@Reference
private UserService userService;

public void asyncCall() {
    // 异步调用，立即返回
    CompletableFuture<User> future = userService.getUserAsync(1L);

    future.thenAccept(user -> {
        // 回调处理结果
        System.out.println(user.getName());
    });
}

// 服务端定义异步方法
@Service
public class UserServiceImpl implements UserService {

    @Override
    public CompletableFuture<User> getUserAsync(long id) {
        return CompletableFuture.supplyAsync(() -> {
            return userRepository.findById(id);
        });
    }
}
```

## 线程模型调优

### 调优思路

```yaml
# 线程模型调优思路
1. 分析瓶颈在哪里
   - IO 瓶颈：增加 IO 线程
   - CPU 瓶颈：增加业务线程
   - 网络瓶颈：检查网络带宽

2. 监控关键指标
   - 线程池活跃线程数
   - 队列等待长度
   - 请求处理延迟

3. 配置调优
   - threads：业务线程池大小
   - iothreads：IO 线程池大小
   - queues：队列大小
```

### 常见问题排查

```java
// 问题一：线程池耗尽
// 现象：Thread pool is EXHAUSTED!
// 原因：线程数不足或处理太慢
// 解决：增加 threads 或优化业务逻辑

// 问题二：队列堆积
// 现象：请求堆积在队列中
// 原因：线程池已满，新请求等待
// 解决：增加 threads 或 queues

// 问题三：IO 线程耗尽
// 现象：IO 线程被打满
// 原因：IO 操作阻塞了 IO 线程
// 解决：增加 iothreads 或将 IO 操作移到业务线程
```

### 监控指标

```yaml
# 线程池监控指标
1. 活跃线程数
   - Active Threads: 当前正在执行任务的线程数
   - 接近 corePoolSize = 线程池繁忙
   - 接近 maximumPoolSize = 线程池即将耗尽

2. 队列等待
   - Queue Size: 等待执行的任务数
   - 持续增长 = 生产能力不足

3. 任务执行统计
   - Completed Tasks: 已完成任务数
   - Total Tasks: 总任务数
```

## 最佳实践

### 实践一：避免在业务线程中执行耗时操作

```java
// 错误：IO 操作在业务线程中执行
@Service
public class BadUserServiceImpl implements UserService {

    @Override
    public User getUser(long id) {
        // 同步调用远程服务，业务线程被阻塞
        return remoteService.getUser(id);
    }
}

// 正确：使用异步调用
@Service
public class GoodUserServiceImpl implements UserService {

    @Override
    public CompletableFuture<User> getUserAsync(long id) {
        return remoteService.getUserAsync(id)
            .thenCompose(user -> {
                // 后续处理
                return CompletableFuture.completedFuture(user);
            });
    }
}
```

### 实践二：使用信号量控制并发

```java
// 使用信号量限制并发
@Service
public class ControlledServiceImpl implements ControlledService {

    private final Semaphore semaphore = new Semaphore(100);

    @Override
    public Result process(Request request) {
        if (!semaphore.tryAcquire()) {
            throw new RuntimeException("系统繁忙，请稍后重试");
        }

        try {
            return doProcess(request);
        } finally {
            semaphore.release();
        }
    }
}
```

### 实践三：合理配置线程数

```yaml
# 线程数配置计算公式
# IO 密集型：线程数 = CPU 核心数 * 2 + 内存相关
# CPU 密集型：线程数 = CPU 核心数 + 1

# 示例：
# 16 核 CPU，32GB 内存
IO 密集型服务： threads = 32 + 8 = 40
CPU 密集型服务： threads = 16 + 1 = 17
```

## 总结

Dubbo 的线程模型采用 Reactor 模式，通过分层处理实现高并发。理解线程模型对于优化 Dubbo 性能至关重要。

**核心要点回顾：**

- Dubbo 使用 Netty 实现 NIO 通信
- 线程池类型：fixed、cached、limited、eager
- 同步调用会阻塞调用线程，异步调用不阻塞
- 线程数配置：IO 密集型 = CPU 核心数 * 2，CPU 密集型 = CPU 核心数 + 1
- 避免在业务线程中执行耗时 IO 操作
