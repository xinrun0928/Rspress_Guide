# Dubbo 路由机制：条件路由与脚本路由

想象一下这个场景：你的服务部署在北京和上海两个机房，上海的用户访问到了北京的服务器，延迟从 5ms 飙升到 80ms。Dubbo 的路由机制就是来解决这个问题的——它能根据规则决定「谁应该调用谁」。

## 为什么需要路由？

在没有路由的情况下，消费者会拿到所有提供者的列表，然后随机选一个调用。这在流量小的时候不是问题，但当你的服务规模扩大、跨机房调用增多、或者需要灰度发布时，随机路由就成了噩梦：

- 跨机房调用：增加延迟，浪费带宽
- 灰度发布：新版本没有验证就直接全量上线
- 环境隔离：测试流量跑到生产环境

Dubbo 的路由机制，就是为了让你**精确控制调用路径**。

## 条件路由

条件路由是最常用的路由方式，通过配置规则来决定调用关系。

### 路由规则格式

```yaml
# 格式：scope => condition
# scope: consumer.application 表示消费者端规则
# scope: provider.application 表示提供者端规则
```

### 基础用法：同机房优先

```yaml
# 优先调用同机房的服务
scope: consumer.application = demo-consumer
rule: >-
  if (provider.properties.containsKey("zone")) {
    return provider.properties.get("zone") == consumer.properties.get("zone");
  }
  return true;
```

更常见的写法是用条件路由表达式：

```yaml
# dubbo-router-condition.yaml
scope: consumer
rule:://上海机房优先
host = 10.0.0.* => host = 10.0.0.10-20
host = 192.168.* => host = 192.168.1.*
# 默认规则，放到最后
=> host = *
```

### 标签路由

标签路由是条件路由的简化版本，通过给服务打标签来实现路由：

```java
// 提供者端：设置服务标签
@DubboService(tag = "v1", weight = 100)
public class UserServiceImpl implements UserService {
    // ...
}

// 消费者端：通过 RPCContext 动态设置标签
public void callWithTag() {
    RpcContext.getContext().setAttachment("dubbo.tag", "v1");
    userService.getUserById(1L);
}
```

对应的 XML 配置：

```xml
<!-- 服务级别标签路由 -->
<dubbo:service interface="com.example.UserService" tag="v1" />

<!-- 应用级别标签路由 -->
<dubbo:reference interface="com.example.UserService">
    <dubbo:parameter key="dubbo.tag" value="v1" />
</dubbo:reference>
```

### 条件路由实战：灰度发布

```yaml
# 灰度流量路由规则：10% 的流量走新版本
scope: consumer
rule: >-
  if (provider.version == "1.0.0" && Math.random() > 0.1) {
    return false;
  }
  return true;
```

或者用更简洁的格式：

```yaml
# 只允许调用 version=2.0.0 的服务
scope: provider
rule: version = 2.0.0
```

## 脚本路由

脚本路由支持更复杂的路由逻辑，适合条件路由无法满足的场景。Dubbo 支持 JavaScript、Groovy 等脚本引擎。

### 动态脚本路由

```java
// 实现 Route 接口创建自定义路由
public class DynamicRoute implements RouterFactory {
    @Override
    public Router getRouter(URL url) {
        return (invokers, invokerURL, rule, needingCheck, params) -> {
            // 根据参数动态决定调用哪个服务
            String version = invokerURL.getParameter("version");
            if (version != null) {
                return invokers.stream()
                    .filter(invoker -> version.equals(invoker.getUrl().getParameter("version")))
                    .collect(Collectors.toList());
            }
            return invokers;
        };
    }
}
```

### 使用 JavaScript 脚本

```yaml
# JavaScript 脚本路由
scope: consumer
rule: >-
  function route(invokers, invokerURL, params) {
    var region = params["region"];
    if (!region) return invokers;

    return invokers.filter(function(invoker) {
      return invoker.getUrl().getParameter("region") === region;
    });
  }
  route(invokers, invokerURL, params);
```

### 脚本路由与规则动态更新

```java
// 通过动态配置中心更新路由规则
public class RouteConfigManager {
    private final ConfigCenter configCenter;
    private final AtomicReference<String> currentRule = new AtomicReference<>();

    public void updateRouteRule(String rule) {
        // 1. 解析并验证规则
        RouteParser.parse(rule);

        // 2. 发布到配置中心
        configCenter.publishConfig("router-rule", rule);

        // 3. 更新本地缓存
        currentRule.set(rule);

        // 4. 通知所有消费者刷新路由
        notifyRouteChanged();
    }
}
```

## 路由链的执行流程

理解路由的执行顺序很重要：

```java
// 路由链执行流程
public <T> List<Invoker<T>> route(List<Invoker<T>> invokers, URL url, Invocation invocation) {
    // 1. 标签路由（最优先）
    List<Invoker<T>> result = tagRouter.route(invokers, url, invocation);

    // 2. 条件路由
    result = conditionRouter.route(result, url, invocation);

    // 3. 脚本路由（最后执行，最灵活）
    result = scriptRouter.route(result, url, invocation);

    // 4. 负载均衡（路由选完后，最后一步）
    return loadBalance.select(result, url, invocation);
}
```

## 路由配置的优先级

当多个路由规则同时存在时，Dubbo 按以下优先级执行：

| 优先级 | 路由类型 | 说明 |
|---|---|---|
| 1 | 标签路由 | `dubbo.tag`，用于环境隔离 |
| 2 | 条件路由 | 通过配置文件或动态配置中心设置 |
| 3 | 脚本路由 | 自定义脚本逻辑 |
| 4 | 负载均衡 | 最终从候选节点中选择一个 |

## 路由的常见问题与解决

### 问题一：路由规则不生效

大多数路由规则不生效的问题，都是因为**消费者不知道规则已更新**。确保：

1. 配置中心正确配置
2. 消费者订阅了配置变更
3. 检查规则的语法是否正确

```java
// 调试：打印当前应用的路由链
public void debugRouterChain() {
    RouterChain<UserService> chain = RouterChain.buildChain(referenceConfig);
    System.out.println("Active Routers: " + chain.getRouters());
}
```

### 问题二：路由后无可用节点

```yaml
# 错误示例：规则过于严格，导致没有服务可用
rule: host = 192.168.99.99 && version = 99.99.99

# 正确示例：添加兜底规则
rule: host = 192.168.99.99 => host = 192.168.99.99
     => host = *  # 没有匹配时，走默认路由
```

### 问题三：路由性能问题

当服务实例数量达到数百个时，路由链的执行会成为性能瓶颈：

```java
// 优化：使用索引而非遍历
public class IndexedRouter implements Router {
    private final Map<String, List<Invoker>> indexCache = new ConcurrentHashMap<>();

    @Override
    public List<Invoker> route(List<Invoker> invokers, URL url, Invocation invocation) {
        String key = buildIndexKey(url);
        return indexCache.computeIfAbsent(key, k ->
            doRoute(invokers, url, invocation)
        );
    }
}
```

## 总结

Dubbo 的路由机制是一套完整的服务调用治理方案：

| 路由类型 | 适用场景 | 配置复杂度 |
|---|---|---|
| 标签路由 | 环境隔离、金丝雀发布 | 低 |
| 条件路由 | 同机房优先、版本路由 | 中 |
| 脚本路由 | 复杂业务逻辑 | 高 |

记住一个原则：**路由越简单越可靠**。能用标签路由解决的问题，就不要用条件路由；能用条件路由解决的，就不要用脚本路由。
