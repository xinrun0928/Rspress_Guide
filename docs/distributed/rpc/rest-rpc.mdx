# RESTful vs RPC vs gRPC

你的团队正在设计服务间通信方案，有人说用 RESTful API，有人说用 RPC，还有人说直接上 gRPC。

到底该怎么选？

这一章，我们从协议、性能、适用场景等多个维度，彻底搞清楚这三种方案的优劣。

## RPC 设计

### 接口设计原则

```yaml
# RPC 接口设计原则
1. 面向业务而非技术
   // 好
   rpc CreateOrder(CreateOrderRequest) returns (CreateOrderResponse);
   // 不好
   rpc RPC1(Request) returns (Response);

2. 方法名使用动词
   rpc GetUser, CreateUser, UpdateUser, DeleteUser

3. 请求响应对象明确
   rpc GetUser(GetUserRequest) returns (GetUserResponse);

4. 考虑向后兼容
   - 不删除字段，只废弃
   - 新增字段使用 optional
   - 版本化接口
```

### Dubbo 接口设计

```java
// 服务接口定义
public interface UserService {

    // 查询
    User getUser(Long id);

    // 分页查询
    Page<User> listUsers(int page, int size);

    // 创建
    User createUser(CreateUserRequest request);

    // 更新
    User updateUser(Long id, UpdateUserRequest request);

    // 删除
    void deleteUser(Long id);
}

// 请求对象
@Data
public class CreateUserRequest {
    private String name;
    private String email;
    private Integer age;
}

// 响应对象
@Data
public class User {
    private Long id;
    private String name;
    private String email;
    private Integer age;
    private LocalDateTime createdAt;
}
```

### Feign 接口设计

```java
// Feign Client 设计
@FeignClient(name = "user-service", path = "/api/users")
public interface UserClient {

    @GetMapping("/{id}")
    User getUser(@PathVariable Long id);

    @GetMapping
    Page<User> listUsers(
        @RequestParam int page,
        @RequestParam int size
    );

    @PostMapping
    User createUser(@RequestBody CreateUserRequest request);
}
```

## gRPC 设计

### .proto 文件设计

```protobuf
// 良好设计的 .proto 文件
syntax = "proto3";

package user;

option java_package = "com.example.grpc.user";
option java_multiple_files = true;

// 包级别注释
// 用户管理服务
service UserService {
    // 获取用户详情
    rpc GetUser(GetUserRequest) returns (User);

    // 创建用户
    rpc CreateUser(CreateUserRequest) returns (User);

    // 流式获取用户列表
    rpc ListUsers(ListUsersRequest) returns (stream User);
}

// 请求消息定义
message GetUserRequest {
    int64 id = 1;  // 用户 ID
}

// 枚举类型
enum UserStatus {
    UNKNOWN = 0;      // 未知
    ACTIVE = 1;       // 活跃
    INACTIVE = 2;     // 非活跃
    BANNED = 3;       // 被封禁
}

// 消息定义
message User {
    int64 id = 1;
    string name = 2;
    string email = 3;
    UserStatus status = 4;
    map<string, string> metadata = 5;
}
```

## 实际选型建议

### 微服务内部通信

```yaml

- 性能优先，内部网络可控
- 强类型，编译时检查
- 支持负载均衡、熔断等治理能力
```

```java
    interfaceClass = UserService.class,

    group = "prod"

// 跨语言选 gRPC
// user.proto 定义后自动生成各语言客户端
```

### 微服务外部通信

```yaml

理由：
- 通用性强，所有语言都支持
- 易于调试和测试
- 便于集成第三方服务
- 便于文档生成
```

## 最佳实践

### 实践一：统一错误处理

```java
// RESTful 统一错误响应
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(BusinessException.class)
    public Response<?> handleBusinessException(BusinessException ex) {
        return Response.error(ex.getCode(), ex.getMessage());
    }

    @ExceptionHandler(ValidationException.class)
    public Response<?> handleValidationException(ValidationException ex) {
        return Response.error("VALIDATION_ERROR", ex.getMessage());
    }
}

// RPC 统一异常
public class RpcExceptionHandler {
    public Result handleRpcException(Throwable e) {
        if (e instanceof BusinessException) {
            return Result.fail(((BusinessException) e).getCode());
        }
        return Result.fail("INTERNAL_ERROR");
    }
}
```

### 实践三：超时与重试


## 总结

没有银弹，只有最适合的方案。

**选型建议：**

| 场景 | 推荐方案 |
|---|---|
| 外部 API、公众 API | RESTful API |
| Java 微服务内部通信 | Dubbo / Feign |
| 跨语言微服务通信 | gRPC |
| 高性能场景 | gRPC |
| 需要浏览器调试 | RESTful API |
| 复杂的企业内部系统 | RESTful + RPC 混合 |

**核心要点回顾：**

- RESTful 通用性强，适合外部 API
- RPC 性能高，适合 Java 服务间通信
- gRPC 性能最高，支持跨语言和流式调用
- 实际项目中可以混合使用
- 无论选择哪种，都要做好监控和治理
