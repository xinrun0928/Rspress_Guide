# 分布式锁应满足的条件

在分布式系统中，单机锁（如 `synchronized`、`ReentrantLock`）完全失效。当你的服务部署在多台机器上，每台机器都是独立的 JVM 进程，操作系统层面的锁机制无法跨越进程边界。你需要的是一把能够跨机器、跨进程生效的「锁」——分布式锁。

但什么样的分布式锁才是合格的？市面上的实现那么多，如何判断一个分布式锁方案是否靠谱？本节将深入分析分布式锁必须满足的八个核心条件。

## 核心条件一：互斥性

互斥性是锁的灵魂。在任意时刻，只能有一个持有者占用锁资源。

```java
// Redis SETNX 实现互斥
Boolean success = redisTemplate.opsForValue()
    .setIfAbsent("lock:product:001", "thread-1", 
        Duration.ofSeconds(30));
```

互斥性的实现难点在于**原子性**。获取锁和设置过期时间必须是一个原子操作，否则可能出现「获取锁成功但设置过期时间失败」的尴尬局面。

```java
// 错误写法：非原子操作
if (redisTemplate.opsForValue().setIfAbsent(key, value)) {
    redisTemplate.expire(key, 30, TimeUnit.SECONDS); // 可能失败
}

// 正确写法：原子操作
redisTemplate.opsForValue().setIfAbsent(key, value, 30, TimeUnit.SECONDS);
```

## 核心条件二：可重入性

可重入性是指同一线程在外层函数获取锁之后，内层函数可以直接再次获取该锁，而不会因已持有锁而阻塞自己。

Java 中的 `ReentrantLock` 就是可重入锁的典型代表。在分布式场景下，`Redisson` 通过 `ThreadLocal` 记录持有次数来实现可重入。

```java
// Redisson 可重入锁示例
RLock lock = redissonClient.getLock("product-lock");
lock.lock();      // 第一次获取
lock.lock();      // 第二次获取，可重入
lock.unlock();    // 持有计数 -1
lock.unlock();    // 持有计数 = 0，锁释放
```

可重入性的实现需要记录两个信息：**持有锁的线程标识**和**重入计数**。

## 核心条件三：锁超时机制

锁必须有超时机制，防止持有锁的客户端崩溃后无法释放锁，导致死锁。

超时时间的选择是个技术活。太短，业务还没执行完锁就过期了；太长，万一持锁客户端宕机，其他客户端要等很久才能获取锁。

:::tip 建议
锁超时时间建议设置为业务正常执行时间的 2-3 倍，给异常情况留足缓冲空间。
:::

```java
// 设置 30 秒超时，业务预计 10 秒内完成
RLock lock = redissonClient.getLock("order-lock");
lock.lock(30, TimeUnit.SECONDS); // 自动过期
```

## 核心条件四：非阻塞获取

有些场景下，我们不希望线程一直阻塞等待锁，而是尝试获取后立即返回结果。

```java
// 非阻塞获取锁
RLock lock = redissonClient.getLock("resource-lock");
boolean acquired = lock.tryLock(0, 10, TimeUnit.SECONDS);
if (acquired) {
    try {
        // 执行业务逻辑
    } finally {
        lock.unlock();
    }
} else {
    // 获取锁失败，处理业务降级
}
```

`tryLock()` 方法的关键在于第一个参数 `waitTime`：
- `waitTime = 0`：非阻塞模式，立即返回
- `waitTime > 0`：阻塞模式，最多等待指定时间

## 核心条件五：公平性（可选）

分布式锁分为**公平锁**和**非公平锁**。

| 类型 | 特点 | 实现复杂度 |
|---|---|---|
| 非公平锁 | 先到先得，不保证顺序 | 低 |
| 公平锁 | 按请求顺序依次获取锁 | 高 |

Redis 原生实现是非公平的。`Redisson` 通过 ZSet（有序集合）实现了公平锁，按照请求时间戳排序。

```java
// 非公平锁（默认）
RLock unfairLock = redissonClient.getLock("resource");

// 公平锁
RLock fairLock = redissonClient.getFairLock("resource");
```

:::warning
公平锁虽然保证了请求顺序，但会显著降低系统吞吐量。在高并发场景下，非公平锁通常是更好的选择。
:::

## 核心条件六：性能与可用性

分布式锁的性能取决于两个因素：网络延迟和协调服务的吞吐量。

| 方案 | 性能 | 可用性 | 适用场景 |
|---|---|---|---|
| Redis | 极高 | 中（依赖 Redis 集群） | 追求性能的业务 |
| ZooKeeper | 高 | 高（ZAB 协议） | 强一致性要求 |
| etcd | 高 | 极高（Raft 协议） | 云原生场景 |

在设计系统时，需要在性能和可靠性之间做出权衡。

## 核心条件七：容错能力

分布式锁必须具备容错能力，能够应对各种异常情况。

### 客户端崩溃

通过设置锁超时时间，即使客户端崩溃，锁也会在超时后自动释放。

### 网络分区

网络抖动可能导致锁操作失败。需要设计重试机制，但要注意幂等性。

```java
// 简单的重试机制
int retryCount = 3;
while (retryCount-- > 0) {
    try {
        lock.lock();
        break;
    } catch (Exception e) {
        if (retryCount == 0) throw e;
        Thread.sleep(100);
    }
}
```

### 主从切换

对于 Redis 主从架构，主节点宕机后从节点晋升时，可能丢失尚未同步的锁数据。红锁（RedLock）算法通过在多个独立的 Redis 实例上加锁来解决这个问题。

## 核心条件八：锁释放的安全性

锁释放必须安全，防止误删他人持有的锁。

常见的问题是：客户端 A 获取锁后业务执行时间过长，锁过期自动释放；客户端 B 获取到同一把锁开始执行；客户端 A 业务执行完毕，释放锁——这时释放的是 B 的锁！

```java
// 错误的释放方式：释放前没有验证
public void unlock() {
    redisTemplate.delete("lock:product:001"); // 可能误删他人锁
}

// 正确的释放方式：验证后删除
public void unlock() {
    String currentValue = redisTemplate.opsForValue().get("lock:product:001");
    if (currentValue.equals(Thread.currentThread().getId())) {
        redisTemplate.delete("lock:product:001");
    }
}
```

`Redisson` 的解锁逻辑通过 Lua 脚本保证原子性：

```lua
-- Lua 脚本解锁，保证只删除自己持有的锁
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

## 条件汇总

| 条件 | 说明 | 必须 |
|---|---|---|
| 互斥性 | 任意时刻只能有一个持有者 | 是 |
| 可重入性 | 同线程可重复获取锁 | 是 |
| 超时机制 | 防止死锁 | 是 |
| 非阻塞获取 | 立即返回结果 | 是 |
| 公平性 | 按请求顺序获取 | 否 |
| 高性能 | 低延迟、高吞吐 | 是 |
| 容错能力 | 应对各种异常 | 是 |
| 安全释放 | 只释放自己持有的锁 | 是 |

## 总结

一个合格的分布式锁必须同时满足**互斥性**、**可重入性**、**超时机制**、**非阻塞获取**、**高性能**、**容错能力**和**安全释放**这些条件。公平性是可选的，根据业务场景决定是否需要。

但即使满足了这些条件，分布式锁也不是银弹。它只能保证互斥性，无法解决所有并发问题。在设计系统时，应该优先考虑**业务层面**的幂等性设计，将分布式锁作为最后一道防线，而不是万能解决方案。
