# ZooKeeper 分布式锁实现

Redis 锁靠 TTL 超时，ZooKeeper 锁靠会话断开——两种方案各有取舍，但 ZooKeeper 的实现更优雅。

ZooKeeper 是一个分布式协调服务，它的临时顺序节点（EPHEMERAL_SEQUENTIAL）是实现分布式锁的利器。相比 Redis 锁，ZooKeeper 锁的实现更简洁，可靠性也更高。

本文深入剖析 ZooKeeper 分布式锁的实现原理。

## 一、ZooKeeper 数据模型

### 1.1 ZNode 节点

ZooKeeper 的数据存储单元叫 ZNode：

```
/
├── /locks
│   ├── /locks/lock-0000000001  (临时顺序节点)
│   ├── /locks/lock-0000000002
│   └── /locks/lock-0000000003
├── /config
│   └── /config/settings
└── /services
    └── /services/service-1
```

### 1.2 ZNode 的特性

| 特性 | 说明 |
|---|---|
| **持久节点（PERSISTENT）** | 创建后一直存在，直到主动删除 |
| **临时节点（EPHEMERAL）** | 会话断开后自动删除 |
| **顺序节点（SEQUENTIAL）** | 创建时 ZooKeeper 会自动追加递增序号 |
| **版本号** | 每个 ZNode 都有版本号，用于乐观锁 |

### 1.3 临时顺序节点（EPHEMERAL_SEQUENTIAL）

这是 ZooKeeper 锁的核心：

```
节点类型：EPHEMERAL_SEQUENTIAL

行为：
1. 客户端创建节点时，ZooKeeper 自动追加递增序号
2. 客户端断开连接，节点自动删除
3. 序号全局递增，不会重复

示例：
- 客户端 A 创建：/locks/lock-0000000001
- 客户端 B 创建：/locks/lock-0000000002
- 客户端 C 创建：/locks/lock-0000000003
```

## 二、ZooKeeper 锁实现原理

### 2.1 获取锁的流程

```
步骤 1：创建临时顺序节点
        │
        ▼
    在 ZooKeeper 创建 /locks/lock-0000000001
        │
        ▼
步骤 2：获取所有子节点
        │
        ▼
    获取 /locks 下所有子节点列表
        │
        ▼
步骤 3：判断自己是否是第一个
        │
        ├── 是 → 获取锁成功
        │
        └── 否 → 监听前一个节点
                    │
                    ▼
                等待前一个节点删除
                    │
                    ▼
                收到通知，再次检查
                    │
                    ▼
                如果自己是第一个 → 获取锁成功
```

### 2.2 详细流程

```java
public class ZooKeeperDistributedLock {

    private final ZooKeeper zooKeeper;
    private final String lockPath;
    private final String nodePrefix = "lock-";
    private String currentNode;
    private String waitNode;

    /**
     * 获取锁
     */
    public void lock() throws InterruptedException, KeeperException {
        // 1. 创建临时顺序节点
        currentNode = zooKeeper.create(
            lockPath + "/" + nodePrefix,  // 节点路径
            new byte[0],                   // 数据
            ZooDefs.Ids.OPEN_ACL_UNSAFE,  // ACL
            CreateMode.EPHEMERAL_SEQUENTIAL  // 临时顺序节点
        );

        // 2. 获取所有子节点
        List<String> children = zooKeeper.getChildren(lockPath, false);

        // 3. 排序
        Collections.sort(children);

        // 4. 判断自己是否是第一个
        String nodeName = new File(currentNode).getName();

        if (isFirstNode(children, nodeName)) {
            // 自己就是第一个，获取锁成功
            return;
        } else {
            // 不是第一个，监听前一个节点
            waitNode = findPrevNode(children, nodeName);

            // 5. 等待前一个节点删除
            synchronized (this) {
                while (true) {
                    // 阻塞等待通知
                    wait();
                    break;
                }
            }
        }
    }

    /**
     * 释放锁
     */
    public void unlock() throws InterruptedException, KeeperException {
        // 删除自己创建的节点
        zooKeeper.delete(currentNode, -1);
    }

    /**
     * 判断自己是否是第一个
     */
    private boolean isFirstNode(List<String> children, String nodeName) {
        return children.get(0).equals(nodeName);
    }

    /**
     * 找到前一个节点
     */
    private String findPrevNode(List<String> children, String nodeName) {
        int index = children.indexOf(nodeName);
        if (index > 0) {
            return children.get(index - 1);
        }
        return null;
    }
}
```

### 2.3 Watcher 监听机制

```java
/**
 * 使用 Watcher 监听前一个节点
 */
private void watchPrevNode() throws KeeperException, InterruptedException {
    if (waitNode != null) {
        // 监听前一个节点的删除事件
        String waitNodePath = lockPath + "/" + waitNode;

        Stat stat = zooKeeper.exists(waitNodePath, event -> {
            if (event.getType() == Watcher.Event.EventType.NodeDeleted) {
                // 前一个节点被删除，通知当前线程
                synchronized (ZooKeeperDistributedLock.this) {
                    ZooKeeperDistributedLock.this.notify();
                }
            }
        });

        // 如果节点已经不存在了，直接获取锁
        if (stat == null) {
            // 节点已删除，获取锁
        }
    }
}
```

## 三、Curator Framework 封装

### 3.1 Curator 的 InterProcessMutex

Curator 提供了开箱即用的分布式锁：

```java
@Service
public class CuratorLockService {

    private final CuratorFramework curatorFramework;

    public CuratorLockService(CuratorFramework curatorFramework) {
        this.curatorFramework = curatorFramework;
        // 启动 Curator
        curatorFramework.start();
    }

    /**
     * 获取分布式互斥锁
     */
    public InterProcessMutex getLock(String lockPath) {
        return new InterProcessMutex(curatorFramework, lockPath);
    }

    /**
     * 使用锁执行任务
     */
    public <T> T executeWithLock(String lockPath, Callable<T> task) {
        InterProcessMutex lock = getLock(lockPath);

        try {
            // 获取锁（阻塞等待）
            lock.acquire();

            // 执行任务
            return task.call();

        } catch (Exception e) {
            throw new RuntimeException("执行失败", e);
        } finally {
            try {
                // 释放锁
                lock.release();
            } catch (Exception e) {
                // 忽略释放异常
            }
        }
    }

    /**
     * 尝试获取锁（带超时）
     */
    public boolean tryAcquire(String lockPath, long timeout, TimeUnit unit) {
        InterProcessMutex lock = getLock(lockPath);
        try {
            return lock.acquire(timeout, unit);
        } catch (Exception e) {
            return false;
        }
    }
}
```

### 3.2 完整示例

```java
@Service
public class OrderService {

    private final CuratorLockService lockService;

    public void createOrder(Order order) {
        // 使用 ZooKeeper 锁
        String lockPath = "/locks/order/" + order.getUserId();

        lockService.executeWithLock(lockPath, () -> {
            // 1. 检查库存
            if (!checkStock(order.getProductId())) {
                throw new BusinessException("库存不足");
            }

            // 2. 创建订单
            Order createdOrder = orderMapper.insert(order);

            // 3. 扣减库存
            inventoryService.decreaseStock(order.getProductId());

            return createdOrder;
        });
    }
}
```

## 四、与 Redis 锁的对比

### 4.1 实现机制对比

| 维度 | Redis 锁 | ZooKeeper 锁 |
|---|---|---|
| **核心机制** | SETNX + TTL | 临时顺序节点 + Watcher |
| **锁释放** | TTL 自动过期 | 会话断开自动删除 |
| **锁获取** | 自旋重试 | Watcher 等待 |
| **一致性** | 依赖 Redis 集群 | ZAB 协议保证 |
| **性能** | 更高 | 较低 |

### 4.2 可靠性对比

```
Redis 锁的可靠性风险：
├─ Redis 主节点挂了
│     ├─ 主从同步未完成
│     │     └─ 锁数据丢失
│     └─ 锁可能被其他客户端获取
│
└─ TTL 设置不合理
      ├─ 业务超过 TTL
      │     └─ 锁自动释放，数据不一致
      └─ TTL 太长
            └─ 故障时锁无法快速恢复

ZooKeeper 锁的可靠性：
├─ ZAB 协议保证一致性
│     └─ 所有节点状态一致
│
└─ 会话机制
      ├─ 会话断开，临时节点自动删除
      └─ 锁立即释放
```

### 4.3 性能对比

| 操作 | Redis | ZooKeeper |
|---|---|---|
| 获取锁延迟 | ~0.1ms | ~2-5ms |
| 释放锁延迟 | ~0.1ms | ~1-3ms |
| 并发能力 | 10万 QPS | 1万 QPS |
| 锁等待 | 自旋轮询 | Watcher 机制 |

## 五、ZooKeeper 锁的优缺点

### 5.1 优点

1. **实现优雅**：利用临时顺序节点和 Watcher，实现简洁
2. **可靠性高**：ZAB 协议保证一致性，会话机制保证锁释放
3. **无需续期**：会话断开锁自动释放，不存在超时问题
4. **自然避免问题**：Watcher 机制自然避免空回滚、悬挂等问题
5. **支持公平锁**：顺序节点天然保证 FIFO

### 5.2 缺点

1. **性能较低**：ZooKeeper 的吞吐量不如 Redis
2. **运维复杂**：需要维护 ZooKeeper 集群
3. **网络依赖**：依赖 ZooKeeper 的网络连接
4. **锁竞争开销**：每次获取锁都需要创建节点

## 七、面试高频问题

**Q：ZooKeeper 锁如何实现？**

A：利用 ZooKeeper 的临时顺序节点（EPHEMERAL_SEQUENTIAL）。创建节点时自动追加序号，判断自己是否是序号最小的节点，如果是则获取锁成功；否则监听前一个节点，等待其删除后通知自己。

**Q：ZooKeeper 锁和 Redis 锁的区别？**

A：Redis 锁依赖 TTL 超时释放，ZooKeeper 锁依赖会话断开自动删除。Redis 锁性能更高，ZooKeeper 锁可靠性更高。Redis 锁需要处理超时问题，ZooKeeper 锁天然避免。

**Q：为什么 ZooKeeper 锁不需要续期？**

A：ZooKeeper 锁通过会话机制管理生命周期。会话断开时，临时节点自动删除。客户端通过心跳保持会话，只要客户端存活，锁就不会释放。

**Q：ZooKeeper 锁有什么缺点？**

A：性能比 Redis 锁低；需要维护 ZooKeeper 集群；网络依赖强；每次获取锁都需要创建节点和获取子节点列表，开销较大。

## 总结

ZooKeeper 分布式锁的实现非常优雅：

1. **临时顺序节点**：保证锁的自动释放和公平性
2. **Watcher 机制**：避免轮询，事件驱动
3. **ZAB 协议**：保证分布式一致性

选择 ZooKeeper 还是 Redis，取决于你的需求：
- **追求性能**：选 Redis
- **追求可靠性**：选 ZooKeeper
- **已有 ZooKeeper 集群**：选 ZooKeeper
- **需要超高性能**：选 Redis
