# MySQL 分布式锁实现

不用 Redis 能不能做分布式锁？当然能。MySQL 的唯一索引，就是最朴素的分布式锁实现。

很多人一说分布式锁就想到 Redis，但 MySQL 同样可以实现分布式锁，而且实现简单，不依赖额外的中间件。

本文深入剖析 MySQL 分布式锁的两种实现方案：**唯一索引方案**和**悲观锁方案**。

## 一、唯一索引方案

### 1.1 核心思想

利用数据库的唯一索引特性：
- 插入成功 = 获取锁
- 插入失败 = 锁已被占用
- 删除记录 = 释放锁

### 1.2 表结构设计

```sql
CREATE TABLE `distributed_lock` (
    `id` BIGINT NOT NULL AUTO_INCREMENT,
    `lock_name` VARCHAR(64) NOT NULL COMMENT '锁名称',
    `lock_owner` VARCHAR(128) NOT NULL COMMENT '锁持有者',
    `expire_time` DATETIME NOT NULL COMMENT '过期时间',
    `retry_count` INT DEFAULT 0 COMMENT '重试次数',
    `create_time` DATETIME DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_lock_name` (`lock_name`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8;
```

### 1.3 获取锁

```java
@Service
public class MysqlDistributedLock {

    @Autowired
    private DistributedLockMapper lockMapper;

    /**
     * 尝试获取锁
     */
    public boolean tryLock(String lockName, String owner, long expireSeconds) {
        try {
            DistributedLock lock = new DistributedLock();
            lock.setLockName(lockName);
            lock.setLockOwner(owner);
            lock.setExpireTime(new Date(System.currentTimeMillis() + expireSeconds * 1000));

            // 插入记录，唯一索引保证原子性
            lockMapper.insert(lock);
            return true;

        } catch (DuplicateKeyException e) {
            // 唯一键冲突，说明锁已被占用
            return false;
        }
    }

    /**
     * 释放锁
     */
    public boolean unlock(String lockName, String owner) {
        // 只删除自己持有的锁
        return lockMapper.deleteByLockNameAndOwner(lockName, owner) > 0;
    }
}
```

### 1.4 Mapper 实现

```java
@Mapper
public interface DistributedLockMapper {

    @Insert("INSERT INTO distributed_lock (lock_name, lock_owner, expire_time) " +
            "VALUES (#{lockName}, #{lockOwner}, #{expireTime})")
    void insert(DistributedLock lock);

    @Delete("DELETE FROM distributed_lock " +
            "WHERE lock_name = #{lockName} AND lock_owner = #{lockOwner}")
    int deleteByLockNameAndOwner(@Param("lockName") String lockName,
                                  @Param("lockOwner") String owner);

    @Select("SELECT * FROM distributed_lock " +
            "WHERE lock_name = #{lockName}")
    DistributedLock selectByLockName(@Param("lockName") String lockName);
}
```

### 1.5 问题：锁过期但业务未完成

```java
// 问题场景
public void process() {
    // 获取锁（TTL = 30 秒）
    if (lock.tryLock("order-lock", owner, 30)) {
        try {
            // 执行业务（需要 60 秒）
            doLongRunningTask();  // 40 秒后锁过期
        } finally {
            lock.unlock("order-lock", owner);
        }
    }
}
```

## 二、乐观锁方案（带自动续期）

### 2.1 自动续期机制

MySQL 锁也可以实现自动续期：

```java
@Service
public class MysqlDistributedLockWithRenewal {

    private final ScheduledExecutorService scheduler = Executors.newScheduledThreadPool(1);

    private final Map<String, ScheduledFuture<?>> renewalTasks = new ConcurrentHashMap<>();

    public boolean tryLock(String lockName, String owner, long expireSeconds) {
        try {
            // 插入锁记录
            DistributedLock lock = new DistributedLock();
            lock.setLockName(lockName);
            lock.setLockOwner(owner);
            lock.setExpireTime(new Date(System.currentTimeMillis() + expireSeconds * 1000));
            lockMapper.insert(lock);

            // 启动续期任务
            startRenewalTask(lockName, owner, expireSeconds);

            return true;
        } catch (DuplicateKeyException e) {
            return false;
        }
    }

    private void startRenewalTask(String lockName, String owner, long expireSeconds) {
        // 每 1/3 expireSeconds 执行一次续期
        long renewalInterval = expireSeconds * 1000 / 3;

        ScheduledFuture<?> task = scheduler.scheduleAtFixedRate(() -> {
            try {
                // 更新过期时间
                int rows = lockMapper.renewExpireTime(lockName, owner,
                    new Date(System.currentTimeMillis() + expireSeconds * 1000));

                if (rows == 0) {
                    // 锁已被删除，停止续期
                    cancelRenewalTask(lockName);
                }
            } catch (Exception e) {
                log.error("续期失败", e);
            }
        }, renewalInterval, renewalInterval, TimeUnit.MILLISECONDS);

        renewalTasks.put(lockName, task);
    }

    public void unlock(String lockName, String owner) {
        // 停止续期任务
        cancelRenewalTask(lockName);

        // 删除锁记录
        lockMapper.deleteByLockNameAndOwner(lockName, owner);
    }

    private void cancelRenewalTask(String lockName) {
        ScheduledFuture<?> task = renewalTasks.remove(lockName);
        if (task != null) {
            task.cancel(false);
        }
    }
}
```

### 2.2 Mapper 更新方法


## 三、悲观锁方案

### 3.1 SELECT FOR UPDATE

MySQL 的行级锁可以通过 `SELECT ... FOR UPDATE` 实现悲观锁：

```java
@Service
public class MysqlPessimisticLock {

    @Autowired
    private TransactionTemplate transactionTemplate;

    /**
     * 使用悲观锁获取数据
     */
    public void processWithPessimisticLock(Long id) {
        transactionTemplate.execute(status -> {
            // FOR UPDATE 加行锁
            Account account = accountMapper.selectForUpdate(id);

            if (account == null) {
                throw new RuntimeException("账户不存在");
            }

            // 执行业务逻辑
            account.setBalance(account.getBalance() - 100);
            accountMapper.updateById(account);

            return null;
        });
    }
}
```

### 3.2 Mapper 实现

```java
@Mapper
public interface AccountMapper {

    @Select("SELECT * FROM account WHERE id = #{id} FOR UPDATE")
    Account selectForUpdate(@Param("id") Long id);

    @Update("<script>" +
            "UPDATE account SET balance = #{balance} WHERE id = #{id}" +
            "</script>")
    int updateById(Account account);
}
```

### 3.3 与唯一索引方案的区别

| 维度 | 唯一索引方案 | 悲观锁方案 |
|---|---|---|
| **锁粒度** | 表级别 | 行级别 |
| **使用场景** | 分布式锁 | 数据库事务内 |
| **超时控制** | 可自定义 | 依赖数据库事务超时 |
| **死锁风险** | 低 | 存在（需注意加锁顺序） |

## 四、表锁 vs 行锁

### 4.1 表锁

```sql
-- 手动加表锁
LOCK TABLES account READ;   -- 读锁
LOCK TABLES account WRITE;  -- 写锁

-- 释放表锁
UNLOCK TABLES;
```

特点：简单粗暴，锁定整张表，性能差，不推荐生产使用。

### 4.2 行锁

```sql
-- 行锁通过 WHERE 条件隐式加锁
UPDATE account SET balance = balance - 100 WHERE id = 1;
```

特点：粒度小，并发高，需要注意索引才能生效。

### 4.3 锁粒度选择

| 场景 | 推荐锁粒度 | 原因 |
|---|---|---|
| 全局唯一操作 | 表锁或唯一索引 | 简单可靠 |
| 按 ID 操作 | 行锁 | 减少锁冲突 |
| 批量操作 | 分批行锁 | 避免长事务 |

## 五、MySQL 锁的可重入实现

### 5.1 可重入设计

MySQL 锁通过 version 字段实现可重入：

```sql
CREATE TABLE `distributed_lock_reentrant` (
    `id` BIGINT NOT NULL AUTO_INCREMENT,
    `lock_name` VARCHAR(64) NOT NULL,
    `lock_owner` VARCHAR(128) NOT NULL,
    `lock_count` INT DEFAULT 1 COMMENT '重入次数',
    `version` INT DEFAULT 0 COMMENT '版本号，用于乐观锁',
    `expire_time` DATETIME NOT NULL,
    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_lock_name_owner` (`lock_name`, `lock_owner`),
    KEY `idx_lock_name` (`lock_name`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8;
```

### 5.2 可重入实现

```java
@Service
public class MysqlReentrantLock {

    @Autowired
    private ReentrantLockMapper lockMapper;

    public boolean tryLock(String lockName, String owner, long expireSeconds) {
        return transactionTemplate.execute(status -> {
            // 1. 查询是否已持有锁
            ReentrantLock existingLock = lockMapper.selectByLockNameAndOwner(lockName, owner);

            if (existingLock != null) {
                // 2. 已持有，可重入，计数器 +1
                lockMapper.incrementLockCount(lockName, owner);
                return true;
            }

            // 3. 未持有，尝试获取
            try {
                DistributedLock lock = new DistributedLock();
                lock.setLockName(lockName);
                lock.setLockOwner(owner);
                lock.setLockCount(1);
                lock.setExpireTime(new Date(System.currentTimeMillis() + expireSeconds * 1000));
                lockMapper.insert(lock);
                return true;
            } catch (DuplicateKeyException e) {
                return false;
            }
        });
    }

    public boolean unlock(String lockName, String owner) {
        return transactionTemplate.execute(status -> {
            ReentrantLock existingLock = lockMapper.selectByLockNameAndOwner(lockName, owner);

            if (existingLock == null) {
                return false;
            }

            if (existingLock.getLockCount() > 1) {
                // 计数器 > 1，减 1
                lockMapper.decrementLockCount(lockName, owner);
                return true;
            } else {
                // 计数器 = 1，完全释放
                lockMapper.deleteByLockNameAndOwner(lockName, owner);
                return true;
            }
        });
    }
}
```

## 六、优点与缺点

### 6.1 优点

1. **不依赖额外中间件**：只要有 MySQL 就能用
2. **持久化保证**：锁数据存储在数据库中，不会丢失
3. **实现简单**：不需要复杂的分布式协调
4. **支持可重入**：可以通过表设计实现
5. **成熟稳定**：MySQL 的锁机制经过长期验证

### 6.2 缺点

1. **性能较差**：数据库 QPS 远低于 Redis
2. **网络延迟高**：每次加锁需要一次数据库交互
3. **数据库压力**：高并发下会给数据库带来压力
4. **单点问题**：数据库本身也有高可用问题
5. **锁粒度限制**：无法像 Redis 那样精确控制

### 6.3 性能对比

| 操作 | Redis | MySQL |
|---|---|---|
| 加锁 | ~0.1ms | ~5ms |
| 解锁 | ~0.1ms | ~3ms |
| 续期 | 自动 | 需额外实现 |
| QPS | 10万+ | 1万 |

## 七、实战建议

### 7.1 何时用 MySQL 锁

- **没有 Redis**：项目只用 MySQL
- **低并发场景**：对性能要求不高
- **简单需求**：不需要复杂的锁功能
- **过渡方案**：快速实现，后续迁移

### 7.2 最佳实践

```java
@Service
public class SafeMysqlLock {

    private final MysqlReentrantLock lock;

    /**
     * 执行带锁的任务
     */
    public <T> T executeWithLock(String lockKey, Callable<T> task,
                                  long expireSeconds, long waitSeconds) {
        String owner = UUID.randomUUID().toString();

        long startTime = System.currentTimeMillis();
        long deadline = startTime + waitSeconds * 1000;

        while (System.currentTimeMillis() < deadline) {
            if (lock.tryLock(lockKey, owner, expireSeconds)) {
                try {
                    return task.call();
                } finally {
                    lock.unlock(lockKey, owner);
                }
            }

            try {
                Thread.sleep(100);  // 重试间隔
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
                throw new RuntimeException("等待锁被中断");
            }
        }

        throw new LockAcquisitionException("获取锁超时: " + lockKey);
    }
}
```

## 八、面试高频问题

**Q：MySQL 分布式锁如何实现？**

A：通过唯一索引实现：插入记录成功 = 获取锁，插入失败 = 锁已被占用，删除记录 = 释放锁。通过 `UNIQUE KEY` 保证原子性。

**Q：MySQL 锁和 Redis 锁的区别？**

A：Redis 锁性能高（~0.1ms），但数据存储在内存，可能丢失。MySQL 锁性能较低（~5ms），但数据持久化，不丢失。

**Q：MySQL 锁如何实现可重入？**

A：添加 `lock_count` 字段记录重入次数。同一线程再次获取锁时计数器 +1，释放时计数器 -1，只有归零时才真正释放锁。

**Q：MySQL 锁有哪些问题需要注意？**

A：1. 锁过期但业务未完成，需要实现自动续期；2. 数据库本身也有单点问题，需要配合主从；3. 高并发下会给数据库带来压力。

## 总结

MySQL 分布式锁是最朴素的实现方案：

1. **唯一索引方案**：利用数据库唯一约束，插入成功 = 获取锁
2. **悲观锁方案**：通过 `SELECT FOR UPDATE` 加行锁
3. **可重入**：通过 `lock_count` 字段实现
4. **自动续期**：通过定时任务实现

MySQL 锁的优势是简单、可靠、不依赖额外组件；缺点是性能不如 Redis。适用于低并发、对性能要求不高的场景。
