# Redis 分布式锁

你打开手机准备秒杀一款限量商品，倒计时归零的瞬间，你的手指以 0.01 秒的速度点击下单按钮。页面显示「下单成功」，你还没来得及高兴，刷新一看——订单生成了两笔。同样的商品，你被扣了两次钱。

这不是电商平台的 bug，这是分布式环境下的并发问题。同一个商品，在库存系统里只有一件，但负责扣库存的服务部署在三台机器上。当秒杀请求同时到达三台机器，它们都「看到」库存还有一件，都去执行扣减操作——结果当然是超卖。

单机环境下，用 `synchronized` 或 `ReentrantLock` 就能解决。但三台机器各是一个 JVM 进程，操作系统的锁机制跨不过去。你需要的是分布式锁。

## Redis 实现分布式锁的核心命令

Redis 之所以能实现分布式锁，依赖两个关键命令：

| 命令 | 作用 |
|---|---|
| `SETNX` | SET if Not eXists，键不存在时设置值 |
| `EXPIRE` | 设置键的过期时间 |

`SETNX` 保证互斥性——只有一个客户端能设置成功；`EXPIRE` 防止死锁——即使持有锁的客户端崩溃，锁也会在过期后自动释放。

## 基础实现

### 最简单的版本

```java
public class SimpleRedisLock {

    private RedisTemplate<String, String> redisTemplate;
    private String lockKey;

    public SimpleRedisLock(RedisTemplate<String, String> redisTemplate, String lockKey) {
        this.redisTemplate = redisTemplate;
        this.lockKey = lockKey;
    }

    public boolean lock(String uniqueValue, long expireTime, TimeUnit unit) {
        Boolean result = redisTemplate.opsForValue()
            .setIfAbsent(lockKey, uniqueValue, expireTime, unit);
        return Boolean.TRUE.equals(result);
    }

    public boolean unlock(String uniqueValue) {
        // 简单版本：直接删除
        redisTemplate.delete(lockKey);
        return true;
    }
}
```

这段代码有三个严重问题：

1. **锁无法重入**：同一线程再次调用 `lock()` 会失败
2. **可能误删他人锁**：解锁时没有验证锁是否是自己持有的
3. **非原子操作**：`SETNX` 和 `EXPIRE` 不是原子操作

### 改进版本：解决原子性

Redis 2.6.12 之后，`SET` 命令支持 `NX` 和 `PX` 选项，可以一步完成设置和过期：

```java
public boolean lock(String uniqueValue, long expireTime, TimeUnit unit) {
    Long expireMillis = unit.toMillis(expireTime);
    Boolean result = redisTemplate.opsForValue()
        .setIfAbsent(lockKey, uniqueValue, Duration.ofMillis(expireMillis));
    return Boolean.TRUE.equals(result);
}
```

### 改进版本：解决安全释放

释放锁前必须验证锁的持有者：

```java
public boolean unlock(String uniqueValue) {
    String currentValue = redisTemplate.opsForValue().get(lockKey);
    if (uniqueValue.equals(currentValue)) {
        return redisTemplate.delete(lockKey);
    }
    return false;
}
```

但这又引入了新问题：`GET` 和 `DELETE` 不是原子操作。解决方案是使用 Lua 脚本：

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

```java
public boolean unlock(String uniqueValue) {
    String script = "if redis.call('get', KEYS[1]) == ARGV[1] then " +
                    "return redis.call('del', KEYS[1]) else return 0 end";
    RedisScript<Long> redisScript = RedisScript.of(script, Long.class);
    Long result = redisTemplate.execute(redisScript, 
        Collections.singletonList(lockKey), uniqueValue);
    return result != null && result > 0;
}
```

## 完整实现

将上述改进整合，得到一个相对完善的 Redis 分布式锁实现：

```java
public class RedisDistributedLock {

    private static final String LOCK_PREFIX = "lock:";

    private RedisTemplate<String, String> redisTemplate;

    public RedisDistributedLock(RedisTemplate<String, String> redisTemplate) {
        this.redisTemplate = redisTemplate;
    }

    public String lock(String businessKey, long expireTime, TimeUnit unit) {
        String lockKey = LOCK_PREFIX + businessKey;
        String uniqueValue = UUID.randomUUID().toString() + ":" + Thread.currentThread().getId();

        long expireMillis = unit.toMillis(expireTime);
        long startTime = System.currentTimeMillis();

        // 尝试获取锁，失败则重试
        while (true) {
            Boolean acquired = redisTemplate.opsForValue()
                .setIfAbsent(lockKey, uniqueValue, Duration.ofMillis(expireMillis));

            if (Boolean.TRUE.equals(acquired)) {
                return uniqueValue; // 获取锁成功
            }

            // 检查是否超时
            if (System.currentTimeMillis() - startTime > expireMillis) {
                return null; // 获取锁超时
            }

            // 短暂等待后重试
            try {
                Thread.sleep(50);
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
                return null;
            }
        }
    }

    public boolean unlock(String uniqueValue) {
        String script = "if redis.call('get', KEYS[1]) == ARGV[1] then " +
                        "return redis.call('del', KEYS[1]) else return 0 end";
        RedisScript<Long> redisScript = RedisScript.of(script, Long.class);
        Long result = redisTemplate.execute(redisScript,
            Collections.singletonList(getLockKey(uniqueValue)), uniqueValue);
        return result != null && result > 0;
    }

    private String getLockKey(String uniqueValue) {
        // 从 uniqueValue 中提取 businessKey
        // 这里简化处理，实际应根据 uniqueValue 的格式来解析
        return LOCK_PREFIX + uniqueValue.split(":")[0];
    }
}
```

## 存在的问题

即使做到了以上几点，Redis 分布式锁仍然有几个难以解决的问题。

### 主从切换问题

Redis 通常部署为主从架构：主节点接收写请求，从节点同步主节点数据。当主节点宕机时，从节点晋升为主节点。

问题在于：**数据同步有延迟**。如果主节点宕机时，某个锁的设置操作还没同步到从节点，从节点晋升后这把锁就「消失」了。

```
客户端 A -> 主节点 SET lock uuid1 EX 30   (成功)
                                ↓ 同步延迟
                              从节点 (还没有)
主节点宕机，从节点晋升
客户端 B -> 新主节点 SET lock uuid2 EX 30 (成功，锁不存在！)
```

### 时钟漂移问题

不同 Redis 实例可能运行在不同的物理机器上，这些机器的时钟可能存在微小差异（时钟漂移）。在设置过期时间时，这个问题可能导致锁提前或延后失效。

### 锁续期问题

如果业务执行时间不确定，可能在锁过期后业务还没执行完。此时锁被自动释放，其他客户端获取锁开始执行，导致两个客户端同时操作共享资源。

## 解决方案：Redisson

为了解决这些问题，Redisson 提供了更完善的分布式锁实现：

1. **可重入锁**：通过 `ThreadLocal` 记录持有计数
2. **Watch Dog**：自动续期机制，防止业务未完成锁过期
3. **红锁**：在多个独立的 Redis 实例上加锁，解决主从切换问题

关于 Redisson 的详细实现原理，将在后续章节深入讲解。

## 使用建议

| 场景 | 推荐方案 |
|---|---|
| 对性能要求高，可接受小概率失效 | Redis 单机或主从 |
| 对可靠性要求高 | Redisson 红锁 |
| 需要可重入、公平锁等功能 | Redisson |
| 业务执行时间确定且短 | Redis 原生实现即可 |

:::tip 最佳实践
1. 锁的过期时间要设置合理，通常是业务执行时间的 2-3 倍
2. 唯一值要包含线程标识，用于安全释放
3. 释放锁必须使用 Lua 脚本保证原子性
4. 高可用场景建议使用 Redisson
:::

## 总结

Redis 分布式锁的实现看似简单，但要做到生产级别可靠，需要解决原子性、安全释放、可重入、主从切换等一系列问题。

原生 Redis 锁适合对可靠性要求不高、业务逻辑简单的场景。对于生产环境，特别是涉及金钱交易的场景，推荐使用 Redisson——它已经帮你处理了大部分「坑」。
