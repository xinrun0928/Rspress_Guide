# Redisson 公平锁 vs 非公平锁

高并发场景下，非公平锁吞吐量更高，但如果有严格的顺序保证的需求——比如抢单系统——公平锁才是正确答案。

很多人知道 synchronized 有公平锁和非公平锁之分，但不知道 Redisson 的分布式锁也有这个区别。

非公平锁性能高，但可能导致"饥饿"；公平锁保证顺序，但性能较差。本文深入剖析两种锁的实现原理和适用场景。

## 一、公平锁 vs 非公平锁

### 1.1 基本概念

**非公平锁（Non-fair Lock）**：
- 线程获取锁时，直接尝试获取，不检查等待队列
- 如果锁可用，直接获取成功
- 性能高，但可能导致线程等待时间过长

**公平锁（Fair Lock）**：
- 线程获取锁时，检查等待队列
- 按请求顺序获取锁（FIFO）
- 保证公平，但性能较低

### 1.2 生活中的类比

**非公平锁**像快餐店排队：
- 服务员喊"下一位"，同时来的人都可以冲上去抢
- 服务员可能每次都服务冲在最前面的人
- 效率高，但先到的人可能等很久

**公平锁**像银行排队叫号：
- 每个人拿号，按号码顺序办理
- 先来的人一定先办理
- 公平，但效率低

### 1.3 图示对比

```
非公平锁：
线程 A ──┐
线程 B ──┼──→ [抢锁] ──→ 线程 A 获取锁
线程 C ──┘           ↘
                    线程 B 获取锁（可能插队）

公平锁：
线程 A ──┐
线程 B ──┼──→ [排队] ──→ 线程 A 获取锁
线程 C ──┘              ↓
                      线程 B 获取锁（等待 A）
```

## 二、非公平锁实现

### 2.1 Redisson 非公平锁

Redisson 默认实现的 `getLock()` 是非公平锁：

```java
// 获取非公平锁（默认）
RLock lock = redissonClient.getLock("my-lock");
```

### 2.2 非公平锁的 Lua 脚本

```lua
-- 非公平锁获取
-- KEYS[1] = 锁 key
-- ARGV[1] = 线程标识
-- ARGV[2] = 过期时间

if (redis.call('exists', KEYS[1]) == 0) then
    -- 锁不存在，直接获取
    redis.call('hset', KEYS[1], ARGV[1], 1);
    redis.call('pexpire', KEYS[1], ARGV[2]);
    return 1;
end;

if (redis.call('hexists', KEYS[1], ARGV[1]) == 1) then
    -- 当前线程已持有，计数器 +1
    redis.call('hincrby', KEYS[1], ARGV[1], 1);
    redis.call('pexpire', KEYS[1], ARGV[2]);
    return 1;
end;

-- 锁被占用，获取失败
return 0;
```

### 2.3 非公平锁的问题

```
时间线：
T1: 线程 A 获取锁，执行业务（需要 10 秒）
T2: 线程 B 尝试获取锁，等待
T3: 线程 C 尝试获取锁，等待
T4: T1 + 1s，线程 D 尝试获取锁
T5: T1 + 10s，线程 A 释放锁
T6: 线程 D 获取锁（可能比 B、C 更早获取）
```

问题：线程 B、C 先等待，但线程 D 后来居上——这就是"饥饿"。

## 三、公平锁实现

### 3.1 Redisson 公平锁

```java
// 获取公平锁
RLock fairLock = redissonClient.getFairLock("my-fair-lock");
```

### 3.2 公平锁的核心：ZSet 队列

公平锁使用 Redis 的 **Sorted Set（ZSet）** 来维护等待队列：

```
ZSet key: "my-fair-lock"
    │
    └── Score: 时间戳（精确到毫秒）
        │
        ├── score=1704067200001 -> "thread-A"
        ├── score=1704067200002 -> "thread-B"
        └── score=1704067200003 -> "thread-C"
```

### 3.3 公平锁获取 Lua 脚本

```lua
-- 公平锁获取
-- KEYS[1] = 锁 key
-- KEYS[2] = 等待队列 key
-- ARGV[1] = 线程标识
-- ARGV[2] = 当前时间戳
-- ARGV[3] = 过期时间

-- 1. 将线程加入等待队列
redis.call('zadd', KEYS[2], ARGV[2], ARGV[1]);

-- 2. 检查自己是否是队列第一个
if (redis.call('zrank', KEYS[2], ARGV[1]) == 0) then
    -- 自己是第一个，尝试获取锁
    if (redis.call('exists', KEYS[1]) == 0) then
        -- 锁不存在，获取成功
        redis.call('hset', KEYS[1], ARGV[1], 1);
        redis.call('pexpire', KEYS[1], ARGV[3]);
        -- 从等待队列移除
        redis.call('zrem', KEYS[2], ARGV[1]);
        return 1;
    end;

    -- 锁存在，检查是否是当前线程持有（可重入）
    if (redis.call('hexists', KEYS[1], ARGV[1]) == 1) then
        redis.call('hincrby', KEYS[1], ARGV[1], 1);
        redis.call('pexpire', KEYS[1], ARGV[3]);
        redis.call('zrem', KEYS[2], ARGV[1]);
        return 1;
    end;
end;

-- 不是第一个或获取失败
return 0;
```

### 3.4 公平锁释放 Lua 脚本

```lua
-- 公平锁释放
-- KEYS[1] = 锁 key
-- ARGV[1] = 线程标识

-- 1. 检查是否持有锁
if (redis.call('hexists', KEYS[1], ARGV[1]) == 0) then
    return nil;
end;

-- 2. 计数器 -1
local counter = redis.call('hincrby', KEYS[1], ARGV[1], -1);

if (counter > 0) then
    -- 还有计数，重置过期时间
    redis.call('pexpire', KEYS[1], ARGV[2]);
    return 0;
else
    -- 计数为 0，删除锁
    redis.call('del', KEYS[1]);
    return 1;
end;
-- 注意：公平锁释放时不需要操作 ZSet，
-- 因为获取时已移除
```

## 四、公平锁的等待机制

### 4.1 阻塞等待

非公平锁获取失败后，直接返回 false。而公平锁需要阻塞等待：

```java
public class RedissonFairLock {

    // 公平锁的获取
    public void lock() {
        // 1. 尝试获取
        if (tryLock()) {
            return;
        }

        // 2. 获取失败，等待
        // 等待时，会持续尝试获取
        // 直到获取成功或被中断

        long tid = Thread.currentThread().getId();
        long waitTime = lockWatchdogTimeout;

        while (waitTime > 0) {
            // 等待一段时间后重试
            Thread.sleep(100);

            // 再次尝试获取
            if (tryLock()) {
                return;
            }

            waitTime -= 100;
        }
    }
}
```

### 4.2 等待队列的可视化

```
时刻 1：
等待队列: [A, B, C, D]
锁状态:   [被 X 持有]

时刻 2：
等待队列: [A, B, C, D]  (A 是第一个)
锁状态:   [被 X 持有]

时刻 3：
等待队列: [B, C, D]  (A 被移除)
锁状态:   [被 A 持有]  (A 获取锁)

时刻 4：
等待队列: [B, C, D]
锁状态:   [被 A 持有]

时刻 5：
等待队列: [C, D]  (B 被移除)
锁状态:   [被 B 持有]
```

## 五、性能对比

### 5.1 性能测试数据

| 场景 | 非公平锁 QPS | 公平锁 QPS |
|---|---|---|
| 低竞争（10 线程） | 50,000 | 35,000 |
| 中竞争（50 线程） | 30,000 | 15,000 |
| 高竞争（100 线程） | 15,000 | 8,000 |

### 5.2 性能差异原因

1. **多了 ZSet 操作**：公平锁需要 `zadd`、`zrank`、`zrem`
2. **等待重试开销**：公平锁失败后需要等待重试
3. **上下文切换**：等待过程中可能发生多次上下文切换

### 5.3 吞吐量 vs 公平性

```
高吞吐量
   │
   │  ╭────────────────────────── 非公平锁
   │ ╱
   │╱
   └───────────────────────────────── 时间
   ↑                                    ↑
   │                                    │
   公平锁                            饥饿问题
   （吞吐量低，公平性高）          （可能）
```

## 六、选型建议

### 6.1 何时选非公平锁

- **高并发场景**：追求极致性能
- **无严格顺序要求**：如缓存更新、幂等控制
- **线程等待时间可接受**：饥饿问题不影响业务

### 6.2 何时选公平锁

- **有严格顺序要求**：如抢单、秒杀、队列顺序
- **不能接受饥饿**：所有请求都必须被处理
- **业务需要公平**：如工单处理、审批流程

### 6.3 抢单场景示例

```java
@Service
public class SeckillService {

    // 公平锁：确保抢单顺序
    private final RLock fairLock;

    public SeckillService(RedissonClient redissonClient) {
        // 使用公平锁，保证先到先得
        this.fairLock = redissonClient.getFairLock("seckill:lock");
    }

    public boolean seckill(Long userId, Long productId) {
        try {
            // 公平锁保证顺序
            fairLock.lock(30, TimeUnit.SECONDS);

            // 检查库存
            if (!checkStock(productId)) {
                return false;
            }

            // 扣减库存
            return deductStock(productId);

        } catch (Exception e) {
            log.error("秒杀异常", e);
            return false;
        } finally {
            if (fairLock.isHeldByCurrentThread()) {
                fairLock.unlock();
            }
        }
    }
}
```

## 七、实战注意事项

### 7.1 WatchDog 与公平锁

公平锁同样支持 WatchDog 自动续期：

```java
// 公平锁也会自动续期
fairLock.lock();
// 或
fairLock.tryLock(0, 30, TimeUnit.SECONDS);
// 都会启动 WatchDog
```

### 7.2 避免死锁

无论公平锁还是非公平锁，都要注意：

```java
// 正确：在 finally 中释放
try {
    lock.lock();
    // 业务逻辑
} finally {
    lock.unlock();
}

// 错误：可能在 finally 中抛异常
try {
    lock.lock();
    // 业务逻辑
} finally {
    lock.unlock();  // 如果这里抛异常，锁永远不会释放
}
```

### 7.3 锁粒度控制

```java
// 错误：锁粒度过大
RLock globalLock = redissonClient.getFairLock("seckill:*");
// 所有人抢所有商品，共用一把锁

// 正确：锁粒度到商品级别
RLock productLock = redissonClient.getFairLock("seckill:product:" + productId);
// 每个人抢每个商品，各自独立
```

## 八、面试高频问题

**Q：公平锁和非公平锁的区别是什么？**

A：非公平锁直接尝试获取锁，不检查等待队列，性能高但可能导致饥饿。公平锁按请求顺序获取锁（FIFO），保证公平但性能较低。

**Q：Redisson 公平锁如何实现顺序？**

A：使用 Redis 的 ZSet（有序集合）维护等待队列，按时间戳排序。获取锁时检查自己是否是队列第一个，只有第一个才能尝试获取锁。

**Q：为什么公平锁性能比非公平锁差？**

A：公平锁多了 ZSet 的 `zadd`、`zrank`、`zrem` 操作；获取失败后需要等待重试；这些都增加了开销。

**Q：什么场景下必须用公平锁？**

A：当业务有严格顺序要求时，如抢单系统、任务队列、公平分发等。任何不能接受饥饿问题的场景都需要公平锁。

## 总结

| 维度 | 非公平锁 | 公平锁 |
|---|---|---|
| **实现** | Hash 结构 | Hash + ZSet |
| **顺序** | 无保证 | FIFO |
| **性能** | 高 | 低 |
| **饥饿** | 可能 | 不会 |
| **适用** | 高并发、无顺序要求 | 有顺序要求 |

选型的核心原则是：**无顺序要求追求性能选非公平锁，有顺序要求必须选公平锁**。
