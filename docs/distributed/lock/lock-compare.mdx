# Redis vs ZooKeeper vs etcd 分布式锁对比

分布式锁，Redis、ZooKeeper、etcd 都能实现，但哪个最适合你的场景？

分布式锁是分布式系统中的基础组件，但不同中间件的实现原理和适用场景差异很大。

本文从实现原理、性能、可靠性等多个维度，全面对比 Redis、ZooKeeper 和 etcd 三种分布式锁实现。

## 一、三种方案的实现原理

### 1.1 Redis 分布式锁

**核心机制**：SETNX + EXPIRE + Lua 脚本

```lua
-- Redis 获取锁的 Lua 脚本
if redis.call('exists', KEYS[1]) == 0 then
    redis.call('hset', KEYS[1], ARGV[1], 1)
    redis.call('pexpire', KEYS[1], ARGV[2])
    return 1
end
if redis.call('hexists', KEYS[1], ARGV[1]) == 1 then
    redis.call('hincrby', KEYS[1], ARGV[1], 1)
    redis.call('pexpire', KEYS[1], ARGV[2])
    return 1
end
return 0
```

**特点**：
- 基于内存，性能极高
- 依赖 TTL 超时释放
- 需要考虑锁续期问题

### 1.2 ZooKeeper 分布式锁

**核心机制**：临时顺序节点 + Watcher

```
创建节点：
    /locks/lock-0000000001  (线程 A)
    /locks/lock-0000000002  (线程 B)
    /locks/lock-0000000003  (线程 C)

获取锁：
    线程 A：自己是最小的，获取锁 ✓
    线程 B：监听 lock-0000000001
    线程 C：监听 lock-0000000002

释放锁：
    线程 A 删除节点，通知线程 B
    线程 B 成为最小的，获取锁 ✓
```

**特点**：
- 利用 ZAB 协议保证一致性
- Watcher 机制事件驱动
- 会话断开节点自动删除

### 1.3 etcd 分布式锁

**核心机制**：Raft 共识 + MVCC + 区间锁

```go
// etcd 锁实现（简化）
type Mutex struct {
    key  string
    etcd *Client
}

func (m *Mutex) Lock(ctx context.Context) error {
    // 1. 创建带 revision 的 key
    resp, err := m.etcd.KV.Txn(ctx).Then(
        clientv3.OpPut(m.key, ""),
    ).Commit()

    // 2. 获取当前最小的 revision
    // 3. 如果不是最小，等待前面的 key 删除
}
```

**特点**：
- 基于 Raft 共识，一致性保证
- MVCC 支持，提供 revision 用于排序
- 支持公平锁

## 三、单点问题与高可用

### 3.1 Redis 高可用方案

```
方案一：主从模式
┌─────┐         ┌─────┐
│ Master │ ───→ │ Slave │
└─────┘         └─────┘
问题：主从切换时锁可能丢失

方案二：Sentinel
┌─────────────┐
│  Sentinel   │ (监控 + 故障转移)
└─────────────┘
    │
    ├─────┐
    │     │
┌───┴─┐ ┌─┴───┐
│ M    │ │ S    │
└───┘ └─┘───┘

方案三：Cluster
┌─────────────┐
│   Cluster    │
└─────────────┘
    │
    ├─────┬─────┐
    │     │     │
┌───┴─┐ ┌─┴───┐ ┌─┴───┐
│Slot 0│ │Slot1 │ │Slot2│
└───┘ └─┘───┘ └─┘───┘
```

### 3.2 ZooKeeper 高可用

```
ZooKeeper 集群（ZAB 协议）

┌─────────┐
│  Leader  │ (处理写请求，发起选举)
└─────────┘
    │
    ├─────────┬─────────┐
    ▼         ▼         ▼
┌───────┐ ┌───────┐ ┌───────┐
│Follow1│ │Follow2│ │Follow3│
└───────┘ └───────┘ └───────┘

特点：
- N >= 2N+1 节点才能正常工作
- Leader 故障自动重新选举
- 写请求需要多数节点确认
```

### 3.3 etcd 高可用

```
etcd 集群（Raft 共识）

┌─────────┐
│  Leader │ (处理写请求)
└─────────┘
    │
    ├─────────┬─────────┐
    ▼         ▼         ▼
┌───────┐ ┌───────┐ ┌───────┐
│Follow1│ │Follow2│ │Follow3│
└───────┘ └───────┘ └───────┘

特点：
- Raft 协议保证强一致
- 写请求需要多数节点确认
- 自动选主和故障恢复
```

## 四、安全性对比

### 4.1 RedLock 的争议

Redis 的 RedLock 算法曾引发争议：

**质疑点**：
1. 时钟漂移导致 TTL 计算错误
2. 多个 Redis 实例之间无一致性保证
3. 释放锁时部分失败导致问题

**作者回应**：
1. 合理设置 TTL 和 driftFactor 可以缓解
2. 分布式系统无法完全避免时钟问题
3. 释放失败可以忽略

### 4.2 ZooKeeper vs etcd

| 维度 | ZooKeeper | etcd |
|---|---|---|
| **协议** | ZAB | Raft |
| **API** | 专用协议 | gRPC + Protobuf |
| **使用场景** | 分布式协调 | 服务发现 + 配置 |
| **社区活跃度** | 中等 | 高（Cloud Native） |
| **多语言支持** | 弱 | 强 |

## 五、选型建议

### 5.1 选择 Redis 的场景

- **性能优先**：对锁操作 QPS 要求极高
- **简单场景**：不需要强一致性
- **已有 Redis**：项目中已有 Redis 基础设施
- **成本敏感**：Redis 部署简单，资源占用少

### 5.2 选择 ZooKeeper 的场景

- **强一致性要求**：不允许锁丢失
- **需要成熟方案**：ZooKeeper 在分布式锁领域应用最广
- **已有 ZooKeeper**：项目中已有 ZooKeeper
- **需要 Watcher**：需要事件驱动的协调

### 5.3 选择 etcd 的场景

- **Cloud Native 环境**：使用 Kubernetes
- **配置中心 + 锁**：同时需要配置管理
- **强一致性**：Raft 保证强一致
- **新项目**：etcd 是更现代的选择

### 5.4 选型决策树

```
需要分布式锁？
    │
    ├── 性能优先？
    │     └─→ 选择 Redis
    │
    ├── Cloud Native 环境？
    │     └─→ 选择 etcd
    │
    ├── 已有 ZooKeeper？
    │     └─→ 选择 ZooKeeper
    │
    ├── 强一致性要求？
    │     └─→ 选择 ZooKeeper 或 etcd
    │
    └── 其他情况？
          └─→ 选择 Redis（最简单）
```

## 七、实战建议

### 7.1 不要过度设计

对于大多数场景，普通的 Redis 锁足够用：

```java
// 足够用的简单实现
@Service
public class SimpleLockService {

    private final RedissonClient redisson;

    public void executeWithLock(String resource, Runnable task) {
        RLock lock = redisson.getLock(resource);
        try {
            lock.lock();
            task.run();
        } finally {
            lock.unlock();
        }
    }
}
```

### 7.2 高可靠性场景

对于金融、支付等场景，考虑：

public class ReliableLockService {

    private final RedissonRedLock redLock;

### 7.3 监控告警

无论选择哪种方案，都需要监控：

```java
// 监控锁的获取时间
@Aspect
@Component
public class LockMonitor {

    @Around("@annotation(DistributedLock)")
    public Object monitor(ProceedingJoinPoint point) throws Throwable {
        long start = System.currentTimeMillis();
        try {
            return point.proceed();
        } finally {
            long duration = System.currentTimeMillis() - start;
            if (duration > 1000) {
                // 锁持有时间过长，告警
                alertService.alert("Lock hold too long: " + duration + "ms");
            }
        }
    }
}
```

## 八、面试高频问题

**Q：Redis、ZooKeeper、etcd 锁的实现有什么区别？**

A：Redis 锁基于 SETNX + TTL，通过 Lua 脚本保证原子性；ZooKeeper 锁基于临时顺序节点 + Watcher，天然公平可靠；etcd 锁基于 Raft 共识 + MVCC revision，保证强一致性。

**Q：为什么 Redis 锁可能丢失？**

A：在主从模式下，如果主节点故障且从节点未同步，锁可能丢失。可以通过 RedLock 算法（多 Redis 实例）或使用 Redis Cluster 降低风险。

**Q：ZooKeeper 锁为什么更可靠？**

A：ZooKeeper 使用 ZAB 协议，所有写请求经过 Leader 广播到 Follower，只有多数节点确认才算成功。临时节点与会话绑定，会话断开节点自动删除。

**Q：如何选择分布式锁的实现？**

A：性能优先选 Redis，强一致优先选 ZooKeeper/etcd，Cloud Native 环境选 etcd，大多数场景选 Redis 足够用。

**Q：RedLock 有什么争议？**

A：主要争议是多个 Redis 实例之间缺乏一致性保证，时钟漂移问题可能导致 TTL 计算不准确。但这些问题是分布式系统的固有问题，合理使用是可以接受的。

## 总结

| 维度 | Redis | ZooKeeper | etcd |
|---|---|---|---|
| **性能** | ★★★★★ | ★★★ | ★★★ |
| **可靠性** | ★★★ | ★★★★ | ★★★★★ |
| **一致性** | 最终一致 | 强一致 | 强一致 |
| **复杂度** | 低 | 中 | 中 |
| **适用场景** | 高性能 | 可靠性 | Cloud Native |

选型的核心原则：

1. **性能优先**：Redis
2. **可靠性优先**：ZooKeeper / etcd
3. **Cloud Native**：etcd
4. **简单场景**：Redis

没有最好的方案，只有最适合当前场景的方案。
