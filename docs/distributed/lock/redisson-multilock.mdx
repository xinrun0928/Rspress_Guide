# Redisson 联锁与红锁

单 Redis 实例加锁，万一 Redis 挂了怎么办？Redisson 的 MultiLock 和 RedLock 给出了不同的答案。

分布式锁最怕的是 Redis 本身出问题。一旦 Redis 宕机，所有依赖这把锁的业务都会受到影响。

Redisson 提供了两种方案来解决这个问题：**MultiLock（联锁）**和 **RedLock（红锁）**。它们的设计思路不同，适用场景也不同。

## 一、MultiLock（联锁）

### 1.1 什么是联锁

MultiLock 将多个 RLock 组合成一个逻辑锁。只有当**所有锁都获取成功**时，才算获取成功；释放时也是**释放所有锁**。

```
MultiLock = [Lock-A] + [Lock-B] + [Lock-C]

获取联锁：
    ├─→ 获取 Lock-A → 成功
    ├─→ 获取 Lock-B → 成功
    └─→ 获取 Lock-C → 成功
    = 联锁获取成功

释放联锁：
    ├─→ 释放 Lock-A
    ├─→ 释放 Lock-B
    └─→ 释放 Lock-C
    = 联锁释放完成
```

### 1.2 使用场景

联锁适用于需要**同时锁定多个资源**的场景：

- 订单处理：需要同时锁定库存和优惠券
- 转账业务：需要同时锁定转出账户和转入账户
- 数据同步：需要同时锁定多个数据源

### 1.3 代码示例

```java
@Service
public class OrderService {

    private final RedissonClient redissonClient;

    public OrderService(RedissonClient redissonClient) {
        this.redissonClient = redissonClient;
    }

    public void processOrder(Long orderId) {
        // 创建联锁
        RLock inventoryLock = redissonClient.getLock("inventory:lock");
        RLock couponLock = redissonClient.getLock("coupon:lock");

        // 组合成联锁
        RedissonMultiLock multiLock = new RedissonMultiLock(inventoryLock, couponLock);

        try {
            // 同时获取两把锁
            // 如果任何一个获取失败，所有已获取的都会被释放
            multiLock.lock();

            // 业务逻辑：同时操作库存和优惠券
            deductInventory(orderId);
            useCoupon(orderId);

        } finally {
            // 释放所有锁
            multiLock.unlock();
        }
    }
}
```

### 1.4 获取逻辑

```java
public class RedissonMultiLock implements RLock {

    private final RLock[] locks;

    @Override
    public boolean tryLock(long waitTime, TimeUnit unit) throws InterruptedException {
        // 依次获取每把锁
        for (RLock lock : locks) {
            if (!lock.tryLock(waitTime, unit)) {
                // 获取失败，释放所有已获取的锁
                unlockInner(calculateCompletedLocks(lock));
                return false;
            }
        }
        return true;
    }

    @Override
    public void unlock() {
        // 释放所有锁
        for (RLock lock : locks) {
            try {
                lock.unlock();
            } catch (Exception e) {
                // 记录异常，继续释放其他锁
                log.error("释放锁失败", e);
            }
        }
    }
}
```

### 1.5 联锁的问题

联锁要求所有锁都在同一个 Redis 实例中：
- 如果 Redis-A 挂了，即使 Redis-B 和 Redis-C 可用，也无法获取联锁
- 联锁只能在单 Redis 实例中使用

## 二、RedLock（红锁）

### 2.1 什么是红锁

RedLock 算法由 Redis 作者 Antirez 提出，用于在**多个独立的 Redis 实例**上加锁。只有超过半数（N/2+1）的实例加锁成功，才算整个红锁获取成功。

```
RedLock = 5 个独立的 Redis 实例

加锁过程：
    ├─→ Redis-1: tryLock() → 成功 ✓
    ├─→ Redis-2: tryLock() → 成功 ✓
    ├─→ Redis-3: tryLock() → 成功 ✓
    ├─→ Redis-4: tryLock() → 失败 ✗
    └─→ Redis-5: tryLock() → 失败 ✗
    
结果：3/5 成功，超过半数，红锁获取成功 ✓

释放过程：
    ├─→ Redis-1: unlock() → 成功
    ├─→ Redis-2: unlock() → 成功
    └─→ Redis-3: unlock() → 成功
```

### 2.2 为什么需要红锁

单 Redis 实例的问题：
- Redis 挂了，所有分布式锁失效
- 即使是主从架构，主节点挂了后从节点升级为主节点
- 但锁信息在从节点中（未同步），可能导致锁丢失

RedLock 的设计目标：
- 多个独立的 Redis 实例
- 不依赖主从复制
- 多数实例可用即认为锁有效

### 2.3 Redisson RedLock 实现

```java
// 创建 5 个 Redisson 客户端（对应 5 个 Redis 实例）
RedissonClient redisson1 = Redisson.create(config1);
RedissonClient redisson2 = Redisson.create(config2);
RedissonClient redisson3 = Redisson.create(config3);
RedissonClient redisson4 = Redisson.create(config4);
RedissonClient redisson5 = Redisson.create(config5);

// 创建红锁
RLock lock1 = redisson1.getLock("my-redlock");
RLock lock2 = redisson2.getLock("my-redlock");
RLock lock3 = redisson3.getLock("my-redlock");
RLock lock4 = redisson4.getLock("my-redlock");
RLock lock5 = redisson5.getLock("my-redlock");

RedissonRedLock redLock = new RedissonRedLock(lock1, lock2, lock3, lock4, lock5);

try {
    // 获取红锁
    redLock.lock();
    // 业务逻辑
} finally {
    redLock.unlock();
}
```

### 2.4 红锁算法步骤

```java
public class RedissonRedLock implements RLock {

    private final RLock[] locks;

    @Override
    public boolean tryLock(long waitTime, TimeUnit unit) throws InterruptedException {
        // 等待时间平均分配给每个锁
        long sleepTime = unit.toMillis(waitTime) / locks.length;
        long currentTime = System.currentTimeMillis();
        long deadline = currentTime + unit.toMillis(waitTime);

        // 已成功获取的锁数量
        int successCount = 0;

        // 依次尝试获取锁
        for (RLock lock : locks) {
            // 如果已超时，停止尝试
            if (System.currentTimeMillis() >= deadline) {
                break;
            }

            // 尝试获取锁
            if (lock.tryLock(sleepTime, unit)) {
                successCount++;
            } else {
                // 获取失败，但可能前面已获取成功
                if (successCount >= locks.length / 2 + 1) {
                    // 已超过半数，继续尝试获取剩余的锁
                    break;
                }
            }
        }

        // 检查是否超过半数
        if (successCount < locks.length / 2 + 1) {
            // 未超过半数，释放所有已获取的锁
            unlock(successCount);
            return false;
        }

        return true;
    }
}
```

### 2.5 红锁的释放

```java
@Override
public void unlock() {
    // 释放所有锁
    for (RLock lock : locks) {
        try {
            lock.unlock();
        } catch (Exception e) {
            // 记录异常
        }
    }
}
```

## 三、MultiLock vs RedLock

### 3.1 核心区别

| 维度 | MultiLock | RedLock |
|---|---|---|
| **设计目标** | 同时锁定多个资源 | 提高锁的可靠性 |
| **Redis 实例** | 同一实例 | 多个独立实例 |
| **成功条件** | 所有锁都成功 | 超过半数成功 |
| **容错能力** | 无（单点） | 有（N/2+1 可用） |
| **性能** | 较高 | 较低（多次网络调用） |
| **场景** | 锁定多个资源 | 提高可用性 |

### 3.2 架构对比

```
MultiLock 架构：
┌─────────────────────────────────────┐
│           应用服务                   │
│  ┌─────────────────────────────────┐│
│  │  MultiLock: [A] + [B] + [C]    ││
│  └─────────────────────────────────┘│
└─────────────────────────────────────┘
              │ 全在同一个 Redis
              ▼
        ┌─────────────┐
        │  Redis-1   │
        │ (A, B, C)  │
        └─────────────┘

RedLock 架构：
┌─────────────────────────────────────┐
│           应用服务                   │
│  ┌─────────────────────────────────┐│
│  │  RedLock: 5 个独立锁            ││
│  └─────────────────────────────────┘│
└─────────────────────────────────────┘
     │      │      │      │      │
     ▼      ▼      ▼      ▼      ▼
┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐
│ R-1 │ │ R-2 │ │ R-3 │ │ R-4 │ │ R-5 │
└─────┘ └─────┘ └─────┘ └─────┘ └─────┘
```

## 四、使用建议

### 4.1 何时使用 MultiLock

- 需要同时锁定多个资源
- 这些资源可以共用同一个 Redis
- 对锁的可靠性要求不是特别高

### 4.2 何时使用 RedLock

- 对锁的可靠性要求极高
- 不能容忍单点故障
- 愿意牺牲部分性能换取可靠性

### 4.3 RedLock 的争议

RedLock 自提出以来就存在争议：

**质疑点**：
1. 多个 Redis 实例之间时钟不同步，可能导致 TTL 计算错误
2. 释放锁时，如果部分实例失败，可能导致锁被其他客户端获取
3. 在网络分区场景下，可能出现脑裂问题

**作者回应**：
1. 时钟漂移问题可以通过合理设置 TTL 缓解
2. 释放失败可以忽略（客户端崩溃时也面临同样问题）
3. 分布式系统无法完全避免脑裂

### 4.4 替代方案

如果对可靠性要求极高，考虑使用：
- **ZooKeeper**：强一致性保证
- **etcd**：Raft 共识算法保证
- **Consul**：类似的共识机制

## 五、实战建议

### 5.1 普通锁足够用的场景

对于大多数业务场景，普通的 Redisson 锁足够用：
- 单 Redis 实例本身有 RDB/AOF 持久化
- 主从架构可以保证高可用
- 极端情况下锁丢失的概率很低

### 5.2 需要 RedLock 的场景

- 金融交易、支付等强一致性要求的场景
- 锁失效会导致严重数据问题的场景
- 愿意为可靠性牺牲性能的场景

### 5.3 代码模板

```java
@Service
public class DistributedLockService {

    private final RedissonClient redissonClient;

    // 普通锁（大多数场景）
    public RLock getSimpleLock(String resource) {
        return redissonClient.getLock(resource);
    }

    // 联锁（锁定多个资源）
    public RedissonMultiLock getMultiLock(String... resources) {
        RLock[] locks = Arrays.stream(resources)
            .map(redissonClient::getLock)
            .toArray(RLock[]::new);
        return new RedissonMultiLock(locks);
    }

    // 红锁（高可靠性场景）
    public RedissonRedLock getRedLock(List<RedissonClient> redissonClients, String resource) {
        RLock[] locks = redissonClients.stream()
            .map(client -> client.getLock(resource))
            .toArray(RLock[]::new);
        return new RedissonRedLock(locks);
    }
}
```

## 六、面试高频问题

**Q：MultiLock 和 RedLock 的区别是什么？**

A：MultiLock 将多个锁组合成一个逻辑锁，要求所有锁都获取成功。RedLock 在多个独立的 Redis 实例上加锁，要求超过半数实例加锁成功。

**Q：RedLock 解决了什么问题？**

A：RedLock 提高了分布式锁的可靠性。即使某个 Redis 实例宕机，只要超过半数的实例可用，锁仍然有效。

**Q：RedLock 有哪些争议？**

A：主要争议包括时钟漂移问题、释放失败处理、网络分区脑裂问题等。但这些问题是分布式系统的固有问题，没有完美的解决方案。

**Q：什么场景应该用 RedLock？**

A：对锁的可靠性要求极高、不能容忍单点故障、愿意牺牲性能换取可靠性的场景，如金融交易等。

## 总结

| 方案 | 目的 | 成功条件 | 适用场景 |
|---|---|---|---|
| 普通锁 | 基本分布式锁 | 单实例加锁成功 | 大多数场景 |
| MultiLock | 锁定多个资源 | 所有锁都成功 | 同时锁定多个资源 |
| RedLock | 提高可靠性 | 半数以上成功 | 高可靠性要求 |

选择的关键是：**根据业务对可靠性的要求，选择合适的方案**。大多数场景下，普通锁足够用；只有对可靠性要求极高的场景，才需要考虑 RedLock。
