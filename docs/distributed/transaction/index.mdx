# 分布式事务

一个订单服务调用了库存服务、账户服务、仓储服务。

四个服务，四个数据库，怎么保证「订单创建成功，库存扣减成功，余额扣减成功，库存入库成功」？

**这就是分布式事务要解决的问题。**

## 解决方案对比

| 方案 | 一致性 | 侵入性 | 性能 | 适用场景 |
|------|--------|--------|------|---------|
| XA | 强一致 | 中等 | 差 | 多数据库强一致 |
| AT | 最终一致 | 低 | 中 | 普通业务场景 |
| TCC | 强一致 | 高 | 高 | 资源敏感场景 |
| SAGA | 最终一致 | 中 | 高 | 长事务编排 |
| 可靠消息 | 最终一致 | 低 | 高 | 异步解耦 |

## 选型决策树

```
┌─────────────────────────────────────────────────────────┐
│                   事务方案选型树                         │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  业务代码能改吗？                                        │
│  ├─ 不能 → AT 模式（零侵入）                           │
│  └─ 能 ↓                                                │
│                                                         │
│  资源敏感吗？（库存、资金）                             │
│  ├─ 是 → TCC 模式（高性能）                           │
│  └─ 否 ↓                                                │
│                                                         │
│  事务时间长？（> 1 分钟）                               │
│  ├─ 是 → SAGA 模式（长事务）                          │
│  └─ 否 → AT 模式 / 可靠消息                           │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

## TCC 的三个经典问题

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| 空回滚 | Try 未执行就执行了 Cancel | 记录 Try 执行状态 |
| 幂等 | Confirm/Cancel 重复执行 | 唯一键 + 状态机 |
| 悬挂 | Cancel 先于 Confirm 执行 | 状态转移检查 |

## 核心公式

```
分布式事务 = 方案选型 + 场景匹配 + 补偿机制

Seata 三模式：
- AT：零侵入，通过 undolog 自动补偿（适合大多数场景）
- TCC：高性能，业务控制资源（适合库存/余额等敏感场景）
- SAGA：长事务编排，通过补偿实现最终一致（适合 > 1 分钟事务）

TCC 避坑：
- 空回滚：Cancel 前检查 Try 是否已执行
- 幂等：每个操作加唯一键 or 状态机
- 悬挂：Confirm 前检查状态是否正确

没有银弹：一致性、性能、开发成本，三者只能取其二。
```
