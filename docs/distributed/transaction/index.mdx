# 分布式事务

<Badge text="分布式事务" type="tip" /> <Badge text="解决方案" type="warning" /> <Badge text="Seata" type="info" />

当你打开外卖 App，点一份披萨下单支付时，这个简单的操作背后可能涉及：

- 商家服务：检查商家是否营业
- 库存服务：检查并扣减配料库存
- 订单服务：创建订单
- 支付服务：扣款
- 积分服务：增加用户积分
- 短信服务：发送通知

这些服务可能分布在不同的服务器、不同数据库、甚至不同的数据中心。如何保证「要么全部成功，要么全部失败」？

这就是**分布式事务**要解决的问题。

## 为什么需要分布式事务？

```java
/**
 * 单机事务 vs 分布式事务
 */

// 单机事务（ACID）
// 所有操作在一个数据库中
// BEGIN;
//   UPDATE inventory SET count = count - 1 WHERE id = 1;
//   UPDATE account SET balance = balance - 100 WHERE id = 2;
//   INSERT INTO orders (...) VALUES (...);
// COMMIT;

// 分布式事务
// 操作分布在多个服务/数据库中
// Service A: UPDATE db_a.inventory SET ...;  // 数据库 A
// Service B: UPDATE db_b.account SET ...;    // 数据库 B
// Service C: INSERT INTO db_c.orders ...;   // 数据库 C

// 问题：如何保证这 3 个操作要么全部成功，要么全部失败？
```

## 分布式事务的挑战

```java
/**
 * 分布式事务面临的挑战
 */

// 1. 网络不可靠
// - 消息可能丢失
// - 消息可能延迟
// - 消息可能乱序

// 2. 节点可能故障
// - 服务可能崩溃
// - 数据库可能宕机
// - 网络可能分区

// 3. 时钟不一致
// - 不同节点时钟不同步
// - 难以判断事件先后顺序

// 4. 数据不一致
// - 部分节点成功，部分节点失败
// - 如何修复不一致？
```

## 分布式事务解决方案全景图

分布式事务有多种解决方案，各有优缺点：

```java
/**
 * 解决方案分类
 */

// 一、强一致性方案
// - 2PC（两阶段提交）
// - 3PC（三阶段提交）
// - XA 规范

// 二、最终一致性方案
// - TCC（Try-Confirm-Cancel）
// - SAGA
// - 可靠消息最终一致
// - 最大努力通知

// 三、业务补偿方案
// - 补偿事务
// - 人工介入
```

### 解决方案对比

| 方案 | 一致性 | 性能 | 复杂度 | 适用场景 |
|---|---|---|---|---|
| XA/2PC | 强一致 | 低 | 中 | 对一致性要求高 |
| TCC | 最终一致 | 高 | 高 | 高性能要求 |
| SAGA | 最终一致 | 高 | 中 | 长事务 |
| 可靠消息 | 最终一致 | 中 | 中 | 异步场景 |
| 最大努力通知 | 最终一致 | 高 | 低 | 非关键业务 |

## Seata：分布式事务解决方案

### Seata 是什么？

Seata（Simple Extensible Autonomous Transaction Architecture）是阿里巴巴开源的分布式事务解决方案：

```java
/**
 * Seata 的核心优势
 */

// 1. 对业务零侵入
// - 基于 AT 模式，自动处理回滚
// - 开发者无需关心事务细节

// 2. 高性能
// - 相比传统 XA，性能提升 10 倍以上
// - 支持高并发场景

// 3. 易用性
// - 提供简单的注解 @GlobalTransactional
// - 一键开启分布式事务

// 4. 多模式支持
// - AT 模式：自动处理
// - TCC 模式：手动处理
// - SAGA 模式：长事务
```

### Seata 的三种模式

```java
/**
 * AT 模式（Auto Transaction）
 */

// 原理：
// - 基于 UNDO LOG 实现自动回滚
// - 对业务零侵入
// - 适合大多数场景

/**
 * TCC 模式（Try-Confirm-Cancel）
 */

// 原理：
// - 手动实现 Try、Confirm、Cancel 三个阶段
// - 性能高，但代码侵入
// - 适合对性能要求高的场景

/**
 * SAGA 模式
 */

// 原理：
// - 将长事务拆分为多个子事务
// - 每个子事务有对应的补偿操作
// - 适合长事务场景
```

## 总结

分布式事务是微服务架构中的核心挑战之一。根据业务场景选择合适的解决方案：

- **对一致性要求极高**：选择 XA/2PC
- **对性能要求高，业务简单**：选择 AT 模式
- **对性能要求高，业务复杂**：选择 TCC 模式
- **长事务场景**：选择 SAGA 模式
- **非关键业务**：选择可靠消息或最大努力通知
