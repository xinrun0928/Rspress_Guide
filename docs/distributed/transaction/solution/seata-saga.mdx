# Seata SAGA 模式

<Badge text="分布式事务" type="tip" /> <Badge text="SAGA 模式" type="warning" /> <Badge text="长事务" type="info" />

你有没有遇到过这样的业务场景：一次促销活动涉及数十个步骤，包括创建订单、扣减库存、扣减余额、发放优惠券、发送短信、更新积分、日志记录……

如果用 TCC 模式，你需要为每个步骤实现 Try/Confirm/Cancel，开发成本极高。

**SAGA 模式**就是来解决这个问题的——它将长事务拆分为多个短事务，通过状态机驱动事务的执行和回滚。

## SAGA 模式的核心概念

```java
/**
 * SAGA 的核心思想
 */

// SAGA 将长事务拆分为多个子事务：
// T1 -> T2 -> T3 -> ... -> Tn

// 每个子事务 Ti 都有对应的补偿操作 Ci：
// T1 -> T2 -> T3 -> T4 -> T5
// C1 <- C2 <- C3 <- C4

// 如果某个 Ti 失败：
// 执行 C(i-1), C(i-2), ..., C1
// 回滚之前的所有操作

/**
 * SAGA vs TCC
 */

// TCC：每个操作都有 Try/Confirm/Cancel
// SAGA：每个操作只有正向和补偿操作

// TCC：锁定资源
// SAGA：不锁定资源，通过补偿回滚
```

### 实现 SAGA 服务

```java
/**
 * SAGA 服务实现
 */
@Service
public class OrderSagaService {
    
    @Autowired
    private OrderDao orderDao;
    
    /**
     * 创建订单（正向操作）
     */
    public void create(Map<String, Object> params) {
        Long userId = Long.parseLong(params.get("userId").toString());
        Long productId = Long.parseLong(params.get("productId").toString());
        Integer count = Integer.parseInt(params.get("count").toString());
        
        Order order = new Order();
        order.setUserId(userId);
        order.setProductId(productId);
        order.setCount(count);
        order.setStatus(OrderStatus.CREATED);
        
        orderDao.insert(order);
        
        // 将订单 ID 存入上下文，供后续步骤使用
        params.put("orderId", order.getId());
    }
    
    /**
     * 取消订单（补偿操作）
     */
    public void cancel(Map<String, Object> params) {
        Long orderId = Long.parseLong(params.get("orderId").toString());
        
        Order order = orderDao.selectById(orderId);
        if (order != null) {
            order.setStatus(OrderStatus.CANCELED);
            orderDao.update(order);
        }
    }
}

/**
 * 库存 SAGA 服务
 */
@Service
public class InventorySagaService {
    
    @Autowired
    private InventoryDao inventoryDao;
    
    /**
     * 扣减库存（正向操作）
     */
    public void deduct(Map<String, Object> params) {
        Long productId = Long.parseLong(params.get("productId").toString());
        Integer count = Integer.parseInt(params.get("count").toString());
        
        inventoryDao.deductStock(productId, count);
    }
    
    /**
     * 恢复库存（补偿操作）
     */
    public void restore(Map<String, Object> params) {
        Long productId = Long.parseLong(params.get("productId").toString());
        Integer count = Integer.parseInt(params.get("count").toString());
        
        inventoryDao.restoreStock(productId, count);
    }
}

/**
 * 账户 SAGA 服务
 */
@Service
public class AccountSagaService {
    
    @Autowired
    private AccountDao accountDao;
    
    /**
     * 扣减余额（正向操作）
     */
    public void deduct(Map<String, Object> params) {
        Long userId = Long.parseLong(params.get("userId").toString());
        BigDecimal amount = new BigDecimal(params.get("amount").toString());
        
        accountDao.deductBalance(userId, amount);
    }
    
    /**
     * 退款（补偿操作）
     */
    public void refund(Map<String, Object> params) {
        Long userId = Long.parseLong(params.get("userId").toString());
        BigDecimal amount = new BigDecimal(params.get("amount").toString());
        
        accountDao.addBalance(userId, amount);
    }
}
```

### 启动 SAGA 状态机

```java
/**
 * 启动 SAGA 事务
 */
@Service
public class BusinessService {
    
    @Autowired
    private SagaClusteringStateMachineEngine stateMachineEngine;
    
    public void startSaga(OrderDTO orderDTO) {
        // 1. 构建参数
        Map<String, Object> params = new HashMap<>();
        params.put("userId", orderDTO.getUserId());
        params.put("productId", orderDTO.getProductId());
        params.put("count", orderDTO.getCount());
        params.put("amount", orderDTO.getTotalAmount());
        
        // 2. 启动 SAGA
        String businessKey = UUID.randomUUID().toString();
        SagaContext context = stateMachineEngine.start(
            "createOrderSaga",  // 状态机名称
            businessKey,         // 业务 Key
            params              // 初始参数
        );
        
        // 3. 检查结果
        if (context.getStatus() == SagaStatus.SUCCEEDED) {
            System.out.println("SAGA 执行成功");
        } else if (context.getStatus() == SagaStatus.ROLLING_BACK) {
            System.out.println("SAGA 正在回滚");
        } else {
            System.out.println("SAGA 执行失败");
        }
    }
}
```

## SAGA 模式的状态机

### 状态类型

```java
/**
 * SAGA 状态机支持的状态类型
 */

// 1. ServiceTask（服务任务）
//    调用一个服务执行正向操作
//    可选补偿操作

// 2. Choice（分支选择）
//    根据条件选择执行路径

// 3. SubStateMachine（子状态机）
//    嵌套另一个 SAGA

// 4. CompensationTrigger（补偿触发）
//    触发补偿流程

// 5. Fail（失败）
//    标记状态机失败

// 6. Succeed（成功）
//    标记状态机成功
```

### 状态转换

```java
/**
 * 状态转换示例
 */

// CreateOrder (ServiceTask)
// -> Success: DeductInventory
// -> Fallback: End (失败)

// DeductInventory (ServiceTask)
// -> Success: DeductBalance
// -> Fallback: RestoreInventory -> CancelOrder -> End

// DeductBalance (ServiceTask)
// -> Success: SendNotification
// -> Fallback: RefundBalance -> RestoreInventory -> CancelOrder -> End

// SendNotification (ServiceTask)
// -> Success: End
// -> Fallback: End (通知失败不影响整体)
```

## SAGA 模式的特点

### 优势

```java
/**
 * SAGA 的优势
 */

// 1. 适合长事务
// - 不需要锁定资源
// - 可以处理几十上百个步骤

// 2. 性能高
// - 每个步骤独立执行
// - 不需要等待其他步骤完成

// 3. 易于实现
// - 每个步骤只需实现正向和补偿
// - 不需要 Try 阶段

// 4. 可视化
// - 状态机配置文件可读性强
// - 便于理解和维护
```

### 劣势

```java
/**
 * SAGA 的劣势
 */

// 1. 不保证隔离性
// - 脏写问题
// - 需要业务层面处理

// 2. 补偿可能失败
// - 补偿也可能失败
// - 需要重试或人工介入

// 3. 补偿复杂度
// - 某些操作无法补偿
// - 需要业务设计时考虑

// 4. 事务时间长
// - 长事务执行时间长
// - 中间状态对外可见
```

## SAGA vs TCC vs AT

```java
/**
 * 模式对比
 */

// AT 模式：
// - 适合：SQL 操作简单的场景
// - 优点：代码零侵入
// - 缺点：SQL 有限制，性能稍低

// TCC 模式：
// - 适合：性能要求高，资源预留场景
// - 优点：性能高，资源控制精细
// - 缺点：代码侵入，三个阶段

// SAGA 模式：
// - 适合：长事务，步骤多的场景
// - 优点：不需要锁定资源，可视化
// - 缺点：不保证隔离性
```

## 适用场景

```java
/**
 * SAGA 适用场景
 */

// 1. 业务流程复杂
// - 涉及多个服务
// - 步骤数量多

// 2. 长事务
// - 执行时间长
// - 不能长时间锁定资源

// 3. 需要可视化
// - 状态机配置文件
// - 便于理解和维护

// 4. 允许补偿
// - 业务可以设计补偿逻辑
// - 补偿失败可接受重试
```

## 总结

SAGA 模式是处理长事务的利器：

1. **状态机驱动**：通过配置文件定义事务流程
2. **正向/补偿操作**：每个步骤有正向和补偿操作
3. **不回滚**：通过补偿回滚，而不是逆向操作
4. **适合长事务**：不锁定资源，适合步骤多的场景

SAGA 模式适合业务流程复杂、步骤多的场景，是 TCC 模式的重要补充。
