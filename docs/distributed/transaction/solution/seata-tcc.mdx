# Seata TCC 模式

<Badge text="分布式事务" type="tip" /> <Badge text="TCC 模式" type="warning" /> <Badge text="手动控制" type="info" />

TCC 模式是 Seata 的另一种核心模式。相比 AT 模式的自动处理，TCC 模式需要开发者**手动实现 Try、Confirm、Cancel 三个阶段**。

虽然代码侵入性更强，但 TCC 模式有更高的性能和更大的灵活性。

## TCC 模式的核心概念

```java
/**
 * TCC 的三个阶段
 */

// Try（预留）
// - 检查业务是否允许执行
// - 预留必要的资源
// - 常见的预留操作：冻结、锁定

// Confirm（确认）
// - 执行真正的业务操作
// - 使用 Try 阶段预留的资源
// - 必须成功，否则不断重试

// Cancel（取消）
// - 释放 Try 阶段预留的资源
// - 回滚业务操作
// - 必须成功，否则不断重试

/**
 * TCC vs AT
 */

// AT 模式：
// - 自动生成 Undo Log
// - 自动回滚
// - 代码零侵入

// TCC 模式：
// - 手动实现 Try/Confirm/Cancel
// - 手动控制资源
// - 代码侵入，但更灵活
```

## TCC 模式的代码示例

### 定义 TCC 接口

```java
/**
 * TCC 接口定义
 */
@LocalTCC
public interface InventoryTccService {
    
    /**
     * Try 阶段
     * 
     * @param businessActionContext TCC 上下文
     * @param productId 商品 ID
     * @param count 扣减数量
     * @return 是否成功
     */
    @TwoPhaseBusinessAction(
        name = "inventoryTcc",
        commitMethod = "confirm",
        rollbackMethod = "cancel"
    )
    boolean tryOperation(
        BusinessActionContext businessActionContext,
        @BusinessActionContextParameter(paramName = "productId") Long productId,
        @BusinessActionContextParameter(paramName = "count") Integer count
    );
    
    /**
     * Confirm 阶段
     */
    boolean confirm(BusinessActionContext businessActionContext);
    
    /**
     * Cancel 阶段
     */
    boolean cancel(BusinessActionContext businessActionContext);
}
```

### 实现 TCC 接口

```java
/**
 * TCC 实现
 */
@Service
public class InventoryTccServiceImpl implements InventoryTccService {
    
    @Autowired
    private InventoryDao inventoryDao;
    
    @Override
    public boolean tryOperation(BusinessActionContext context, 
                                Long productId, Integer count) {
        // 1. 检查库存是否充足
        Inventory inventory = inventoryDao.selectByProductId(productId);
        if (inventory == null) {
            throw new BusinessException("商品不存在");
        }
        
        // 2. 检查库存是否足够扣减
        if (inventory.getAvailableStock() < count) {
            // 库存不足
            return false;
        }
        
        // 3. 冻结库存（预留资源）
        int rows = inventoryDao.freezeStock(productId, count);
        return rows > 0;
    }
    
    @Override
    public boolean confirm(BusinessActionContext context) {
        // 1. 获取参数
        Long productId = Long.parseLong(
            context.getActionContext("productId").toString()
        );
        Integer count = Integer.parseInt(
            context.getActionContext("count").toString()
        );
        
        // 2. 扣减冻结的库存（真正的业务操作）
        int rows = inventoryDao.deductFrozenStock(productId, count);
        return rows > 0;
    }
    
    @Override
    public boolean cancel(BusinessActionContext context) {
        // 1. 获取参数
        Long productId = Long.parseLong(
            context.getActionContext("productId").toString()
        );
        Integer count = Integer.parseInt(
            context.getActionContext("count").toString()
        );
        
        // 2. 解冻库存（释放预留的资源）
        int rows = inventoryDao.unfreezeStock(productId, count);
        return rows > 0;
    }
}
```

### 使用 TCC 服务

```java
/**
 * 业务服务使用 TCC
 */
@Service
public class OrderService {
    
    @Autowired
    private InventoryTccService inventoryTccService;
    
    @GlobalTransactional(name = "create-order")
    public void createOrder(OrderDTO orderDTO) {
        // 1. 调用 TCC 服务的 Try 方法
        boolean reserved = inventoryTccService.tryOperation(
            null,  // 由框架传递 BusinessActionContext
            orderDTO.getProductId(),
            orderDTO.getCount()
        );
        
        if (!reserved) {
            throw new BusinessException("库存不足");
        }
        
        // 2. 创建订单
        Order order = new Order();
        order.setProductId(orderDTO.getProductId());
        order.setCount(orderDTO.getCount());
        orderDao.insert(order);
        
        // 3. 如果后续操作失败，TM 会自动调用 Cancel 方法
    }
}
```

### 嵌套事务

```java
/**
 * TCC 嵌套事务
 */

// TCC 支持嵌套事务
// 内层事务失败只会回滚内层
// 外层事务失败会回滚外层和内层

@Service
public class OrderService {
    
    @GlobalTransactional(name = "create-order")
    public void createOrder(OrderDTO orderDTO) {
        // 外层事务
        inventoryTccService.tryOperation(productId, count);
        
        // 嵌套事务
        @GlobalTransactional(name = "nested-payment")
        public void nestedPayment() {
            paymentTccService.tryOperation(userId, amount);
        }
    }
}
```

## TCC 模式的问题处理

### 空回滚

```java
/**
 * 空回滚问题
 */

// 问题：Try 方法还没执行，Cancel 方法就先执行了
// 原因：Try 方法超时返回失败，但实际可能成功

/**
 * 解决方案：幂等控制
 */
@Override
public boolean cancel(BusinessActionContext context) {
    // 1. 获取分支事务 ID
    Long xid = Long.parseLong(context.getXid());
    Long branchId = context.getBranchId();
    
    // 2. 检查是否已经执行过 Cancel
    // 通过分支事务 ID 查询状态表
    BranchStatus status = branchTransactionDao.getStatus(branchId);
    
    if (status == BranchStatus.BRANCH_STATUS_CONFIRMED) {
        // 已经 Confirm，直接返回成功
        return true;
    }
    
    if (status == BranchStatus.BRANCH_STATUS_CANCELED) {
        // 已经 Cancel，直接返回成功
        return true;
    }
    
    // 3. 执行 Cancel
    Long productId = Long.parseLong(context.getActionContext("productId").toString());
    Integer count = Integer.parseInt(context.getActionContext("count").toString());
    
    int rows = inventoryDao.unfreezeStock(productId, count);
    
    // 4. 更新状态
    branchTransactionDao.updateStatus(branchId, BranchStatus.BRANCH_STATUS_CANCELED);
    
    return rows > 0;
}
```

### 幂等控制

```java
/**
 * 幂等控制
 */

// TCC 的 Confirm/Cancel 可能重复执行
// 需要保证幂等性

@Service
public class InventoryTccServiceImpl implements InventoryTccService {
    
    @Autowired
    private BranchTransactionDao branchTransactionDao;
    
    @Override
    public boolean confirm(BusinessActionContext context) {
        Long branchId = context.getBranchId();
        
        // 1. 检查是否已经 Confirm
        BranchStatus status = branchTransactionDao.getStatus(branchId);
        if (status == BranchStatus.BRANCH_STATUS_CONFIRMED) {
            return true;  // 已经执行过，直接返回成功
        }
        
        // 2. 执行 Confirm
        Long productId = Long.parseLong(context.getActionContext("productId").toString());
        Integer count = Integer.parseInt(context.getActionContext("count").toString());
        
        inventoryDao.deductFrozenStock(productId, count);
        
        // 3. 更新状态
        branchTransactionDao.updateStatus(branchId, BranchStatus.BRANCH_STATUS_CONFIRMED);
        
        return true;
    }
}
```

### 悬挂问题

```java
/**
 * 悬挂问题
 */

// 问题：Cancel 方法先执行了，Try 方法才执行
// 原因：Cancel 方法超时返回失败，但实际可能成功

/**
 * 解决方案：检查 Try 是否已执行
 */
@Override
public boolean tryOperation(BusinessActionContext context, 
                           Long productId, Integer count) {
    Long branchId = context.getBranchId();
    
    // 1. 检查是否已经 Cancel 过
    BranchStatus status = branchTransactionDao.getStatus(branchId);
    if (status == BranchStatus.BRANCH_STATUS_CANCELED) {
        // 已经 Cancel，不执行 Try，直接返回失败
        return false;
    }
    
    // 2. 正常执行 Try
    return doTryOperation(productId, count);
}
```

## TCC 模式 vs AT 模式

```java
/**
 * 对比
 */

// AT 模式：
// - 优点：代码零侵入，自动处理
// - 缺点：性能稍低，SQL 有限制
// - 适用：大多数业务场景

// TCC 模式：
// - 优点：性能高，更灵活，资源控制更精细
// - 缺点：代码侵入，开发者工作量大
// - 适用：性能要求高的场景，资源预留场景

/**
 * 选择建议
 */

// 选择 AT 模式：
// - SQL 操作简单
// - 不想改动业务代码
// - 追求开发效率

// 选择 TCC 模式：
// - 需要精细控制资源
// - 对性能要求极高
// - SQL 不支持 AT 模式的场景
```

## 总结

TCC 模式是 Seata 的高性能模式：

1. **三阶段处理**：Try（预留）、Confirm（确认）、Cancel（取消）
2. **手动控制**：开发者实现每个阶段，灵活性高
3. **高性能**：本地事务快速完成，资源锁定时间短
4. **需要处理的问题**：空回滚、幂等、悬挂

TCC 模式适合对性能要求高、需要精细控制资源的场景。
