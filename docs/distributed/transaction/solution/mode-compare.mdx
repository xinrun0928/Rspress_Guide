# AT vs TCC vs SAGA 对比

|分布式事务| |方案对比| |选型指南|

经过前几章的学习，你已经了解了 Seata 的三种主要模式：AT、TCC 和 SAGA。这一章，我们来做一次全面的对比，帮助你在实际项目中做出正确的选择。

## 三种模式总览

```java
/**
 * 三种模式的特点
 */

// AT 模式：
// - 原理：基于 Undo Log 自动回滚
// - 侵入性：零侵入
// - 性能：高
// - 适用：SQL 为主的场景

// TCC 模式：
// - 原理：Try-Confirm-Cancel 三个阶段
// - 侵入性：高侵入
// - 性能：最高
// - 适用：需要精细控制资源的场景

// SAGA 模式：
// - 原理：正向操作 + 补偿操作
// - 侵入性：中侵入
// - 性能：高
// - 适用：长事务、步骤多的场景
```

## 详细对比

### 核心机制对比

```java
/**
 * 核心机制对比
 */

// AT 模式：
// - 回滚机制：基于 Undo Log 自动生成反向 SQL
// - 资源锁定：本地事务提交前锁定，全局锁持有时间短
// - SQL 支持：需要支持 SELECT FOR UPDATE

// TCC 模式：
// - 回滚机制：手动实现 Cancel 方法释放资源
// - 资源锁定：通过 Try 阶段预留资源
// - SQL 支持：不依赖 SQL

// SAGA 模式：
// - 回滚机制：通过补偿操作回滚
// - 资源锁定：不锁定资源
// - SQL 支持：不依赖 SQL
```

### 性能对比

```java
/**
 * 性能对比
 */

// 性能排名：TCC > SAGA > AT > XA

// AT 模式：
// - 高性能：本地事务先提交
// - 全局锁持有时间短
// - 但需要解析 SQL、生成 Undo Log

// TCC 模式：
// - 最高性能：Try 阶段快速完成
// - 全局锁只在 Try 阶段持有
// - Confirm/Cancel 都是本地操作

// SAGA 模式：
// - 高性能：不锁定资源
// - 每个步骤独立执行
// - 补偿操作可以异步执行
```

### 侵入性对比

```java
/**
 * 侵入性对比
 */

// 侵入性排名：AT < SAGA < TCC

// AT 模式：
// - 零侵入：只需要加 @GlobalTransactional 注解
// - 不需要关心事务细节
// - 适合快速开发

// TCC 模式：
// - 高侵入：需要实现 Try/Confirm/Cancel 三个方法
// - 需要处理幂等、空回滚、悬挂等问题
// - 开发成本高

// SAGA 模式：
// - 中侵入：需要实现正向操作和补偿操作
// - 通过配置文件定义流程
// - 比 TCC 简单，但比 AT 复杂
```

### SQL 支持对比

```java
/**
 * SQL 支持对比
 */

// AT 模式支持的 SQL：
// - INSERT
// - UPDATE（WHERE 包含主键或唯一索引）
// - DELETE（WHERE 包含主键或唯一索引）

// AT 模式不支持的 SQL：
// - UPDATE 不带 WHERE 或 WHERE 不包含主键
// - DELETE 不带 WHERE 或 WHERE 不包含主键
// - 批量操作
// - 子查询
// - 跨库操作

// TCC 和 SAGA 模式：
// - 不依赖 SQL
// - 任意业务逻辑都可以实现
```

### 隔离性对比

```java
/**
 * 隔离性对比
 */

// AT 模式：
// - 通过全局锁保证隔离性
// - 同一时刻只有一个全局事务能修改同一行
// - 隔离级别较高

// TCC 模式：
// - 通过资源预留保证隔离性
// - Try 阶段预留后，其他事务无法使用该资源
// - 隔离级别高

// SAGA 模式：
// - 不保证隔离性
// - 可能出现脏写
// - 需要业务层面处理
```

## 场景选择建议

```java
/**
 * 场景选择建议
 */

// 场景 1：用户注册送积分
// - 操作：创建用户、发放积分（两个服务）
// - 特点：简单、步骤少
// - 选择：AT 模式（TCC 也可以但不必要）

// 场景 2：下单扣减库存
// - 操作：创建订单、扣减库存、扣减余额
// - 特点：资源预留
// - 选择：TCC 模式（资源冻结/解冻）

// 场景 3：电商促销流程
// - 操作：创建订单、扣减库存、扣减余额、发放优惠券、
//         发放积分、发送短信、更新统计数据...
// - 特点：步骤多、长事务
// - 选择：SAGA 模式（状态机驱动）

// 场景 4：银行转账
// - 操作：扣减余额、转账、添加余额
// - 特点：强一致性要求高
// - 选择：XA 模式（强一致性）

// 场景 5：订单完成后发送通知
// - 操作：发送短信、发送邮件、推送消息
// - 特点：允许失败
// - 选择：最大努力通知
```

## 实际项目中的选择

```java
/**
 * 实际项目建议
 */

// 1. 优先考虑 AT 模式
// - 开发效率高
// - 代码零侵入
// - 维护成本低

// 2. 性能瓶颈时考虑 TCC
// - AT 性能不够时
// - 需要精细控制资源时
// - 可以牺牲开发效率

// 3. 复杂流程考虑 SAGA
// - 业务流程复杂
// - 步骤数量多
// - 需要可视化流程

// 4. 混合使用
// - 核心业务用 AT 或 TCC
// - 非核心业务用可靠消息
// - 通知类业务用最大努力通知
```

## 模式对比表

| 维度 | AT | TCC | SAGA | XA |
|---|---|---|---|---|
| 一致性 | 最终一致 | 最终一致 | 最终一致 | 强一致 |
| 性能 | 高 | 最高 | 高 | 低 |
| 侵入性 | 无 | 高 | 中 | 中 |
| SQL 支持 | 有限制 | 无 | 无 | 支持 |
| 资源锁定 | 短 | 短 | 无 | 长 |
| 隔离性 | 全局锁 | 资源预留 | 无 | 全局锁 |
| 复杂度 | 低 | 高 | 中 | 中 |
| 适用场景 | 大多数 | 资源预留 | 长事务 | 强一致 |

## 总结

三种模式各有优劣，选择取决于具体场景：

1. **AT 模式**：首选，大多数场景都适用，代码零侵入
2. **TCC 模式**：性能要求极高，或需要精细控制资源的场景
3. **SAGA 模式**：长事务、业务流程复杂的场景

实际项目中，可以根据业务特点灵活选择，甚至混合使用。
