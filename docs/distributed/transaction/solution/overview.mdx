# 分布式事务解决方案总览

<Badge text="分布式事务" type="tip" /> <Badge text="方案对比" type="warning" /> <Badge text="选型指南" type="info" />

在微服务架构中，分布式事务是一个无法回避的问题。不同业务场景对一致性的要求不同，选择一个合适的解决方案至关重要。

这一章，我们来系统性地了解各种分布式事务解决方案的特点、适用场景和选型建议。

## 分布式事务的分类

根据对一致性的保证程度，分布式事务解决方案可以分为三类：

```java
/**
 * 解决方案分类
 */

// 一、强一致性方案
// 严格保证 ACID 特性
// - XA 规范（基于 2PC）
// - 3PC（三阶段提交）

// 二、最终一致性方案
// 允许短暂不一致，最终达到一致
// - TCC（Try-Confirm-Cancel）
// - SAGA
// - 可靠消息
// - 最大努力通知

// 三、业务补偿方案
// 通过业务手段处理不一致
// - 人工补偿
// - 自动补偿
```

## XA 规范

### 原理

XA 是由 X/Open 组织提出的分布式事务处理标准：

```java
/**
 * XA 事务的工作流程
 */

// 应用层
Application {
    // 开启全局事务
    xa.start("transaction-id");
    
    // 执行分支事务
    serviceA.doSomething();
    serviceB.doSomething();
    serviceC.doSomething();
    
    // 提交/回滚全局事务
    xa.end("transaction-id");
    xa.commit("transaction-id");
}

// XA 事务协调者
XA coordinator {
    // 两阶段提交
    // 阶段一：要求所有分支事务 Prepare
    // 阶段二：根据结果 Commit 或 Rollback
}

/**
 * XA 的优势
 */

// 1. 强一致性
// 2. 标准接口，数据库原生支持
// 3. 实现相对简单

/**
 * XA 的劣势
 */

// 1. 性能较低
// - 需要多次网络通信
// - 锁定资源时间长

// 2. 协调者单点
// - 协调者故障会导致事务阻塞
```

### 适用场景

```java
/**
 * XA 适用场景
 */

// 1. 对一致性要求极高的业务
// - 银行转账
// - 证券交易
// - 库存扣减

// 2. 数据量不大
// - 单次事务涉及的数据不多
// - 锁定时间可接受

// 3. 短事务
// - 事务执行时间短
// - 锁定资源时间可接受
```

## TCC 模式

### 原理

TCC 将每个业务操作分为三个阶段：

```java
/**
 * TCC 的三个阶段
 */

// Try（预留）
// - 检查业务是否允许执行
// - 预留必要的资源
// - 不做真正的业务操作

// Confirm（确认）
// - 执行真正的业务操作
// - 使用 Try 阶段预留的资源
// - 必须成功，否则会不断重试

// Cancel（取消）
// - 释放 Try 阶段预留的资源
// - 回滚业务操作
// - 必须成功，否则会不断重试

/**
 * TCC 示例：库存服务
 */
@LocalTCC
public class InventoryTccService implements TccAction {
    
    @Override
    public boolean try(GameTccContext context) {
        // 1. 检查库存是否充足
        int stock = inventoryDao.getStock(productId);
        if (stock < quantity) {
            return false;  // 库存不足，Try 失败
        }
        
        // 2. 冻结库存（预留资源）
        inventoryDao.freezeStock(productId, quantity);
        
        // 3. 保存事务信息，用于后续 Confirm/Cancel
        context.setActionContext("productId", productId);
        context.setActionContext("quantity", quantity);
        
        return true;  // Try 成功
    }
    
    @Override
    public boolean confirm(GameTccContext context) {
        // 1. 扣减冻结的库存
        String productId = context.getActionContext("productId");
        Integer quantity = context.getActionContext("quantity");
        
        inventoryDao.deductFrozenStock(productId, quantity);
        return true;
    }
    
    @Override
    public boolean cancel(GameTccContext context) {
        // 1. 释放冻结的库存
        String productId = context.getActionContext("productId");
        Integer quantity = context.getActionContext("quantity");
        
        inventoryDao.unfreezeStock(productId, quantity);
        return true;
    }
}

/**
 * TCC 的优势
 */

// 1. 性能高
// - 在 Try 阶段就预留资源
// - Confirm/Cancel 都是本地操作

// 2. 锁定资源时间短
// - Try 阶段锁定
// - Confirm/Cancel 立即释放

/**
 * TCC 的劣势
 */

// 1. 代码侵入
// - 每个业务都要实现 Try/Confirm/Cancel
// - 开发成本高

// 2. 幂等性要求
// - Confirm/Cancel 可能重复执行
// - 必须保证幂等

// 3. 空回滚
// - Try 未执行，Cancel 执行了
// - 需要处理这种情况

// 4. 悬挂
// - Cancel 先执行了
// - Try 才执行
// - 需要处理这种情况
```

### 适用场景

```java
/**
 * TCC 适用场景
 */

// 1. 对性能要求高的业务
// - 高并发场景
// - 需要快速响应

// 2. 资源预留场景
// - 库存冻结/解冻
// - 优惠券锁定/解锁

// 3. 跨库操作
// - 不同数据库之间的事务
```

## SAGA 模式

### 原理

SAGA 将长事务拆分为多个短事务：

```java
/**
 * SAGA 的工作流程
 */

// 将一个长事务拆分为：
// T1 -> T2 -> T3 -> ... -> Tn
// 每个 Ti 都有对应的补偿操作：
// C1 <- C2 <- C3 <- ... <- Cn

// 如果某个 Ti 失败：
// 执行 C(i-1), C(i-2), ..., C1
// 回滚之前的所有操作

/**
 * SAGA 示例：订单服务
 */
public class OrderSaga {
    
    public void createOrder(Order order) {
        // 定义 Saga 步骤
        Saga saga = sagaBuilder
            .step("createOrder")
                .invoke(() -> orderService.create(order))
                .onCompensation(() -> orderService.cancel(order.getId()))
            .step("reserveInventory")
                .invoke(() -> inventoryService.reserve(order.getItems()))
                .onCompensation(() -> inventoryService.release(order.getItems()))
            .step("processPayment")
                .invoke(() -> paymentService.pay(order))
                .onCompensation(() -> paymentService.refund(order.getId()))
            .build();
        
        // 执行 Saga
        saga.execute();
    }
}

/**
 * SAGA 的优势
 */

// 1. 适合长事务
// - 不需要锁定资源
// - 每个步骤都是独立的

// 2. 性能好
// - 没有两阶段提交的开销
// - 可以异步执行

/**
 * SAGA 的劣势
 */

// 1. 不保证隔离性
// - 可能出现脏写
// - 需要业务层面处理

// 2. 补偿复杂
// - 需要为每个操作编写补偿
// - 补偿可能失败

// 3. 事务状态机复杂
// - 需要维护事务状态
// - 状态转换逻辑复杂
```

### 适用场景

```java
/**
 * SAGA 适用场景
 */

// 1. 长事务场景
// - 业务流程复杂
// - 执行时间长

// 2. 跨服务调用链
// - 调用多个服务
// - 每个服务独立

// 3. 允许补偿的业务
// - 可以通过补偿回滚
// - 补偿逻辑简单
```

## 可靠消息

### 原理

通过消息队列保证消息的可靠传递：

```java
/**
 * 可靠消息的流程
 */

// 1. 生产者发送消息到消息队列
// 2. 消息队列持久化消息
// 3. 消费者消费消息
// 4. 消费成功后发送确认

/**
 * 可靠消息示例
 */
public class OrderService {
    
    @Autowired
    private TransactionalMessageSender messageSender;
    
    @Transactional
    public void createOrder(Order order) {
        // 1. 本地事务：创建订单
        orderDao.insert(order);
        
        // 2. 发送消息（与本地事务绑定）
        // RocketMQ 的事务消息
        messageSender.sendInTransaction("order-topic", order, () -> {
            // 事务提交后的回调
            // 可以在这里更新消息状态
        });
    }
}

/**
 * 可靠消息的优势
 */

// 1. 异步化
// - 不需要同步等待
// - 性能高

// 2. 解耦
// - 生产者和消费者解耦
// - 可以独立扩展

/**
 * 可靠消息的劣势
 */

// 1. 最终一致
// - 不是强一致
// - 可能有一段时间不一致

// 2. 消息延迟
// - 消息可能延迟到达
// - 需要处理这种情况
```

### 适用场景

```java
/**
 * 可靠消息适用场景
 */

// 1. 允许延迟的业务
// - 通知类消息
// - 日志同步

// 2. 异步处理场景
// - 异步发送短信
// - 异步发送邮件
```

## 最大努力通知

### 原理

尽最大努力通知，但不保证最终成功：

```java
/**
 * 最大努力通知的流程
 */

// 1. 发起方向消息队列发送消息
// 2. 接收方消费消息
// 3. 发送方定期重试
// 4. 超过最大次数后放弃

/**
 * 最大努力通知示例
 */
public class NotificationService {
    
    private static final int MAX_RETRY = 5;
    private static final long RETRY_INTERVAL = 60 * 1000;  // 60 秒
    
    /**
     * 发送通知
     */
    public void sendNotification(Order order) {
        // 1. 发送通知
        boolean success = send(order);
        
        // 2. 如果失败，添加重试任务
        if (!success) {
            scheduleRetry(order, 1);
        }
    }
    
    /**
     * 重试通知
     */
    public void retry(Order order, int retryCount) {
        if (retryCount > MAX_RETRY) {
            // 超过最大次数，放弃
            log.warn("Notification failed after {} retries", MAX_RETRY);
            return;
        }
        
        // 延迟重试
        CompletableFuture.delayedExecutor(RETRY_INTERVAL, TimeUnit.MILLISECONDS)
            .execute(() -> {
                boolean success = send(order);
                if (!success) {
                    scheduleRetry(order, retryCount + 1);
                }
            });
    }
}

/**
 * 最大努力通知的优势
 */

// 1. 实现简单
// 2. 性能高
// 3. 资源消耗少

/**
 * 最大努力通知的劣势
 */

// 1. 不保证成功
// 2. 可能丢失消息
```

### 适用场景

```java
/**
 * 最大努力通知适用场景
 */

// 1. 非关键业务
// - 对一致性要求不高
// - 失败可以接受

// 2. 通知类场景
// - 短信通知
// - 邮件通知

// 3. 对实时性要求不高的场景
```

## 总结

分布式事务没有银弹，需要根据业务场景选择合适的方案：

1. **XA**：强一致，但性能低，适合关键业务
2. **TCC**：最终一致，性能高，但侵入性强
3. **SAGA**：最终一致，适合长事务
4. **可靠消息**：最终一致，适合异步场景
5. **最大努力通知**：实现简单，适合非关键业务

在实际项目中，往往会组合使用多种方案，根据业务特点灵活选择。
