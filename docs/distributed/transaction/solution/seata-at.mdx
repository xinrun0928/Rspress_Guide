# Seata AT 模式

<Badge text="分布式事务" type="tip" /> <Badge text="AT 模式" type="warning" /> <Badge text="自动处理" type="info" />

Seata AT 模式是 Seata 最核心的模式，它的特点是**对业务代码零侵入**。开发者只需要在需要开启分布式事务的方法上加一个注解，Seata 会自动处理回滚。

这听起来很神奇，它是如何做到的？

## AT 模式的核心原理

AT 模式的核心是 **Undo Log**（回滚日志）。在执行业务 SQL 的同时，Seata 会自动记录数据的前镜像（Before Image）和后镜像（After Image），用于回滚：

```java
/**
 * AT 模式的工作流程
 */

// 1. 开启全局事务
//    TC（Transaction Coordinator）生成全局事务 ID

// 2. 执行分支事务
//    - 生成 Undo Log（记录前后镜像）
//    -执行业务 SQL
//    - 提交本地事务

// 3. 提交/回滚全局事务
//    - 全部成功：删除 Undo Log
//    - 部分失败：根据 Undo Log 回滚
```

### 前后镜像

```java
/**
 * 前后镜像示例：扣减库存
 */

// 业务 SQL：
// UPDATE inventory SET count = count - 10 WHERE id = 1;

// 执行前查询（Before Image）：
// SELECT count FROM inventory WHERE id = 1;
// 结果：count = 100

// 执行后查询（After Image）：
// SELECT count FROM inventory WHERE id = 1;
// 结果：count = 90

// 生成 Undo Log：
// {
//   "beforeImage": { "count": 100 },
//   "afterImage": { "count": 90 },
//   "sql": "UPDATE inventory SET count = count - 10 WHERE id = 1"
// }

// 回滚时：
// UPDATE inventory SET count = 100 WHERE id = 1;
```

### 使用分布式事务

```java
/**
 * AT 模式使用示例
 */
@Service
public class OrderService {
    
    @GlobalTransactional(name = "create-order", rollbackFor = Exception.class)
    public void createOrder(OrderDTO orderDTO) {
        // 1. 创建订单
        Order order = new Order();
        order.setUserId(orderDTO.getUserId());
        order.setProductId(orderDTO.getProductId());
        order.setCount(orderDTO.getCount());
        order.setTotalAmount(orderDTO.getTotalAmount());
        order.setStatus(OrderStatus.CREATED);
        orderDao.insert(order);
        
        // 2. 扣减库存（远程调用）
        inventoryFeignClient.deductStock(orderDTO.getProductId(), orderDTO.getCount());
        
        // 3. 扣减余额（远程调用）
        accountFeignClient.deductBalance(orderDTO.getUserId(), orderDTO.getTotalAmount());
    }
}

/**
 * 全局事务注解 @GlobalTransactional
 */

// name: 全局事务名称（唯一标识）
// rollbackFor: 指定哪些异常触发回滚
// timeoutMills: 超时时间（毫秒）
```

### 多个服务的事务处理

```java
/**
 * 库存服务
 */
@Service
public class InventoryService {
    
    @GlobalTransactional(name = "deduct-inventory", rollbackFor = Exception.class)
    public void deductStock(Long productId, Integer count) {
        // AT 模式会自动处理：
        // 1. 生成前后镜像
        // 2. 执行业务 SQL
        // 3. 记录 Undo Log
        inventoryDao.deductById(productId, count);
    }
}

/**
 * 账户服务
 */
@Service
public class AccountService {
    
    @GlobalTransactional(name = "deduct-balance", rollbackFor = Exception.class)
    public void deductBalance(Long userId, BigDecimal amount) {
        // AT 模式会自动处理
        accountDao.deductByUserId(userId, amount);
    }
}
```

## AT 模式的实现原理

### 整体架构

```java
/**
 * AT 模式的架构组件
 */

// 1. TC（Transaction Coordinator）- 事务协调器
//    - Seata Server
//    - 管理全局事务状态
//    - 协调分支事务提交/回滚

// 2. TM（Transaction Manager）- 事务管理器
//    - 业务应用
//    - 开启全局事务
//    - 发起提交或回滚

// 3. RM（Resource Manager）- 资源管理器
//    - Seata 数据源代理
//    - 管理分支事务
//    - 注册到 TC

/**
 * AT 模式处理流程
 */

// 全局事务开启：
// TM -> TC: 开启全局事务（begin），获取 XID
// TC -> TM: 返回全局事务 ID（XID）

// 分支事务执行：
// TM -> RM: 执行分支事务，携带 XID
// RM: 生成 Undo Log，执行业务 SQL
// RM -> TC: 注册分支事务

// 全局事务提交：
// TM -> TC: 提交全局事务
// TC -> RM: 异步删除 Undo Log

// 全局事务回滚：
// TM -> TC: 回滚全局事务
// TC -> RM: 根据 Undo Log 回滚
```

### Undo Log 表结构

```sql
-- Undo Log 表
CREATE TABLE `undo_log` (
    `id` bigint NOT NULL AUTO_INCREMENT,
    `branch_id` bigint NOT NULL,
    `xid` varchar(100) NOT NULL,
    `context` varchar(128) NOT NULL,
    `rollback_info` longblob NOT NULL,
    `log_status` int NOT NULL,
    `log_created` datetime NOT NULL,
    `log_modified` datetime NOT NULL,
    PRIMARY KEY (`id`),
    UNIQUE KEY `ux_undo_log` (`xid`,`branch_id`)
) ENGINE=InnoDB AUTO_INCREMENT=1 DEFAULT CHARSET=utf8mb4;
```

### 全局锁

AT 模式通过全局锁来解决隔离性问题：

```java
/**
 * 全局锁机制
 */

// 问题：两个全局事务同时扣减同一商品的库存
// T1: UPDATE inventory SET count = count - 10 WHERE id = 1;
// T2: UPDATE inventory SET count = count - 20 WHERE id = 1;
// 如果不加控制，可能导致库存变成负数

// AT 模式的解决方案：全局锁
// 1. 分支事务执行前，向 TC 申请全局锁
// 2. 获得全局锁后才能执行业务 SQL
// 3. 本地事务提交后，释放全局锁

/**
 * 全局锁的实现
 */

// TC 端：
// 全局锁管理器维护每个资源的锁信息
// branch_id, xid, lock_key

// RM 端：
// 执行前检查全局锁
// 执行后释放全局锁
```

## AT 模式的使用限制

```java
/**
 * AT 模式支持的 SQL
 */

// 支持：
// - INSERT
// - UPDATE（WHERE 条件包含主键或唯一索引）
// - DELETE（WHERE 条件包含主键或唯一索引）

/**
 * AT 模式不支持的场景
 */

// 1. 批量操作不带主键
// UPDATE inventory SET count = count - 10;  // 不支持
// UPDATE inventory SET count = count - 10 WHERE product_id = 1;  // 支持

// 2. 查询不带 WHERE 条件
// SELECT * FROM inventory;  // 不支持

// 3. 跨库多表操作（AT 模式是单库内的）
// 不同数据库需要使用 TCC 或 XA

// 4. 子查询
// UPDATE inventory SET count = count - 10 
//   WHERE id IN (SELECT id FROM product WHERE status = 1);  // 不支持
```

## AT 模式的高性能秘密

```java
/**
 * AT 模式 vs XA 的性能对比
 */

// XA 模式：
// - 执行 SQL 后不提交，等待 TC 指令
// - 分支事务周期长
// - 锁定资源时间长

// AT 模式：
// - 执行 SQL 后立即提交
// - Undo Log 异步处理
// - 分支事务周期短

/**
 * AT 模式的高性能原因
 */

// 1. 本地事务先提交
//    - 分支事务快速完成
//    - 不阻塞其他事务

// 2. Undo Log 异步删除
//    - 提交时异步删除
//    - 不影响响应时间

// 3. 全局锁持有时间短
//    - 锁在本地事务提交后释放
//    - 不需要等待 TC 指令

// 4. TC 可以异步处理
//    - 分支注册是异步的
//    - 状态同步是异步的
```

## 常见问题

### Q1: AT 模式和 TCC 模式如何选择？

```java
/**
 * 选择建议
 */

// 选择 AT 模式：
// - SQL 支持你的业务场景
// - 希望代码零侵入
// - 对性能要求不是极端高

// 选择 TCC 模式：
// - 需要手动控制资源
// - AT 模式不支持的场景
// - 对性能要求极高
```

### Q2: AT 模式如何保证隔离性？

```java
/**
 * 隔离性保证
 */

// AT 模式通过全局锁保证隔离性

// 全局锁保证：
// 1. 同一时刻只有一个全局事务能修改同一行数据
// 2. 其他全局事务必须等待

// 但 AT 模式的隔离级别是「脏读」
// 在分支事务执行期间，数据可能被修改
// 但由于有 Undo Log，回滚时能恢复
```

### Q3: AT 模式挂了对业务有影响吗？

```java
/**
 * 故障影响
 */

// AT 模式挂了对业务的影响：

// 1. 新事务无法开启
//    - 需要重启 Seata Server
//    - 或者切换到其他模式

// 2. 进行中的事务
//    - 等待 TC 恢复
//    - TC 恢复后会继续处理

// 3. 业务数据
//    - 已经提交的数据不受影响
//    - 未提交的数据会回滚
```

## 总结

AT 模式是 Seata 最核心的模式：

1. **零侵入**：通过注解开启事务，Seata 自动处理回滚
2. **高性能**：本地事务先提交，全局锁持有时间短
3. **基于 Undo Log**：记录前后镜像，支持自动回滚
4. **全局锁**：保证隔离性，防止脏写

AT 模式适合大多数业务场景，是 Seata 的首选模式。
