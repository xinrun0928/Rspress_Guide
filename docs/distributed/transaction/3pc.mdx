## 一、3PC 的基本原理

### 1.1 什么是 3PC

3PC（三阶段提交）在 2PC 的基础上增加了「CanCommit」阶段，用于提前确认参与者是否可以提交，从而避免长时间锁定资源。

```
3PC 的三个阶段：

1. CanCommit：询问是否可提交
   - 协调者询问所有参与者：「可以提交吗？」
   - 参与者返回 Yes/No

2. PreCommit：准备提交
   - 如果所有参与者返回 Yes，协调者发送 preCommit
   - 参与者锁定资源，进入预提交状态

3. DoCommit：真正提交
   - 协调者发送 doCommit
   - 参与者提交或回滚
```

### 1.2 3PC 与 2PC 的对比

```mermaid
graph LR
    subgraph "2PC"
        A1["Prepare"] --> A2["Commit/Rollback"]
    end
    
    subgraph "3PC"
        B1["CanCommit"] --> B2["PreCommit"] --> B3["DoCommit"]
    end
    
    style B1 fill:#48dbfb
    style B2 fill:#feca57
    style B3 fill:#ff6b6b
```

| 阶段 | 2PC | 3PC |
|------|-----|-----|
| **第一阶段** | Prepare（锁定资源） | CanCommit（询问） |
| **第二阶段** | Commit/Rollback | PreCommit（准备） |
| **第三阶段** | - | DoCommit（执行） |

## 二、3PC 的详细流程

### 2.1 第一阶段：CanCommit

```mermaid
sequenceDiagram
    participant C as 协调者
    participant P1 as 参与者1
    participant P2 as 参与者2
    participant P3 as 参与者3
    
    Note over C,P3: 3PC 第一阶段：CanCommit
    
    C->>P1: canCommit?
    C->>P2: canCommit?
    C->>P3: canCommit?
    
    Note over P1: 检查本地资源<br/>返回 Yes/No
    Note over P2: 检查本地资源<br/>返回 Yes/No
    Note over P3: 检查本地资源<br/>返回 Yes/No
    
    P1-->>C: Yes
    P2-->>C: Yes
    P3-->>C: No  ← 参与者3不可提交
```

**关键点**：CanCommit 阶段不锁定资源，只做检查。

### 2.2 第二阶段：PreCommit

```mermaid
sequenceDiagram
    participant C as 协调者
    participant P1 as 参与者1
    participant P2 as 参与者2
    participant P3 as 参与者3
    
    Note over C,P3: 3PC 第二阶段：PreCommit
    
    alt 所有参与者返回 Yes
        C->>P1: preCommit()
        C->>P2: preCommit()
        
        Note over P1: 锁定资源<br/>进入预提交状态
        Note over P2: 锁定资源<br/>进入预提交状态
        
        P1-->>C: ACK
        P2-->>C: ACK
    else 任意参与者返回 No
        C->>P1: abort()
        C->>P2: abort()
        C->>P3: abort()
    end
```

**关键点**：PreCommit 阶段锁定资源，但参与者知道「最终会提交」。

### 2.3 第三阶段：DoCommit

```mermaid
sequenceDiagram
    participant C as 协调者
    participant P1 as 参与者1
    participant P2 as 参与者2
    
    Note over C,P2: 3PC 第三阶段：DoCommit
    
    alt 协调者收到所有 ACK
        C->>P1: doCommit()
        C->>P2: doCommit()
        
        Note over P1: 真正提交<br/>释放资源
        Note over P2: 真正提交<br/>释放资源
        
        P1-->>C: COMMITTED
        P2-->>C: COMMITTED
    end
```

## 三、3PC 如何解决 2PC 的阻塞

### 3.1 协调者宕机场景

```mermaid
sequenceDiagram
    participant C as 协调者
    participant P1 as 参与者1
    participant P2 as 参与者2
    
    Note over C,P2: 2PC 的阻塞
    
    C->>P1: prepare()
    C->>P2: prepare()
    
    P1-->>C: READY
    P2-->>C: READY
    
    C-x C: 宕机！
    
    Note over P1: 无限等待...资源锁定
    Note over P2: 无限等待...资源锁定
    
    Note over C,P2: 3PC 的改进
    
    C->>P1: canCommit()
    C->>P2: canCommit()
    
    P1-->>C: Yes
    P2-->>C: Yes
    
    C->>P1: preCommit()
    C->>P2: preCommit()
    
    P1-->>C: ACK
    P2-->>C: ACK
    
    C-x C: 宕机！← 在发送 doCommit 前
    
    Note over P1: 等待 doCommit 超时<br/>可以主动提交！<br/>因为已进入 preCommit 状态
    Note over P2: 等待 doCommit 超时<br/>可以主动提交！
```

**核心改进**：

```
2PC 问题：
- 参与者不知道协调者的决策
- 只能无限等待

3PC 改进：
- 参与者收到 preCommit 后知道「一定会提交」
- 超时后可以主动提交
```

### 3.2 超时机制的改进

```java
// 3PC 参与者的超时处理
public class Participant3PC {
    
    public void onPreCommit(PreCommitRequest request) {
        // 锁定资源，进入预提交状态
        lockResources();
        saveState("PRE_COMMITTED");
        
        // 发送 ACK
        sendACK(request.getXid());
        
        // 等待 doCommit
        // 如果超时，可以主动提交
        try {
            waitForDoCommit(TIMEOUT_MS);
        } catch (TimeoutException e) {
            // 3PC 改进：超时后可以主动提交
            // 因为已经进入 preCommit 状态
            doCommit();
        }
    }
    
    public void onDoCommit(DoCommitRequest request) {
        doCommit();
    }
    
    public void onAbort(AbortRequest request) {
        // 无论什么状态，收到 abort 都必须回滚
        rollback();
        releaseResources();
    }
}
```

## 四、3PC 的问题与局限性

### 4.1 3PC 的新问题：网络分区

**问题**：3PC 假设网络分区是短暂且可恢复的，但在真正的网络分区下仍可能出问题。

```mermaid
sequenceDiagram
    participant C as 协调者（多数分区）
    participant P1 as 参与者1
    participant P2 as 参与者2
    participant P3 as 参与者3
    participant C2 as 协调者2（少数分区）
    
    Note over C,P3: 协调者所在分区：P1, P2, C
    
    C->>P1: canCommit()
    C->>P2: canCommit()
    
    P1-->>C: Yes
    P2-->>C: Yes
    
    C->>P1: preCommit()
    C->>P2: preCommit()
    
    Note over P3: P3 不可达，状态未知
    Note over C2: 如果备协调者 C2 启动<br/>可能产生不一致
    
    Note over P1,C2: ⚠️ 两组都可能在超时后提交
```

### 4.2 3PC 无法完全解决的问题

| 问题 | 2PC | 3PC | 说明 |
|------|-----|-----|------|
| **协调者宕机** | 完全阻塞 | 部分解决 | 超时后可提交 |
| **网络分区** | 可能不一致 | 可能不一致 | 仍然有问题 |
| **数据不一致** | 可能 | 可能 | 网络分区时都可能出现 |
| **性能开销** | 高 | 更高 | 多了一个阶段 |

### 4.3 为什么实际应用不多

```
实际应用较少的原因：

1. 协议更复杂
   - 多一个阶段意味着更多网络开销
   - 更多状态需要管理

2. 假设不一定成立
   - 3PC 假设网络分区是短暂的
   - 但真正的网络分区可能导致长时间不可达

3. 性能收益不明显
   - 虽然减少了阻塞时间
   - 但增加了网络往返次数

4. 实际系统选择其他方案
   - Raft/Paxos 已经是解决共识的好方案
   - 不需要再使用 3PC
```

## 五、3PC 的状态机

### 5.1 协调者状态机

```mermaid
stateDiagram-v2
    [*] --> Init
    Init --> CanCommit: 发起投票
    CanCommit --> CanCommit: 等待响应
    CanCommit --> PreCommit: 全部 Yes
    CanCommit --> Abort: 任意 No
    PreCommit --> DoCommit: 全部 ACK
    PreCommit --> Abort: 超时/任意 No
    DoCommit --> [*]: 完成
    Abort --> [*]: 完成
```

### 5.2 参与者状态机

```mermaid
stateDiagram-v2
    [*] --> Initial
    Initial --> Ready: 收到 canCommit 且同意
    Initial --> Abort: 收到 abort
    Ready --> PreCommit: 收到 preCommit
    Ready --> Abort: 超时或收到 abort
    PreCommit --> Committed: 收到 doCommit
    PreCommit --> Committed: 超时后可提交
    PreCommit --> Abort: 收到 abort
    Committed --> [*]: 完成
    Abort --> [*]: 完成
```

## 六、3PC 与 2PC 的对比总结

### 6.1 详细对比表

| 维度 | 2PC | 3PC |
|------|-----|-----|
| **阶段数** | 2 | 3 |
| **准备阶段** | 锁定资源 | 检查但不锁定 |
| **阻塞时间** | 长（prepare 到 commit） | 短（preCommit 到 doCommit） |
| **协调者宕机** | 完全阻塞 | 部分解决 |
| **性能开销** | 高 | 更高 |
| **实现复杂度** | 中 | 高 |
| **数据一致性风险** | 有 | 有（网络分区时） |

### 6.2 面试回答模板

```
面试问题：2PC 和 3PC 的区别是什么？

回答要点：
1. 3PC 在 2PC 的基础上增加了 CanCommit 阶段
2. CanCommit 阶段用于提前确认是否可提交，不锁定资源
3. 3PC 的改进：参与者收到 preCommit 后，超时可以主动提交
4. 但 3PC 也有问题：网络分区时仍可能产生数据不一致
5. 实际应用中，3PC 使用较少，因为 Raft/Paxos 是更好的共识方案
```
