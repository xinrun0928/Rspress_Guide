```java
/**
 * 回答要点
 */

// CAP 理论三要素：
// - C (Consistency)：一致性
// - A (Availability)：可用性
// - P (Partition Tolerance)：分区容错性

// CAP 的不可能三角：
// 分布式系统最多只能同时满足两个
```

## 2PC 和 3PC 问题

### 问题四：什么是 2PC？有什么问题？

:::tip 回答思路

1. 解释两阶段提交的流程
2. 说明协调者和参与者的角色
3. 分析 2PC 的问题（单点故障、阻塞、数据不一致）

:::

```java
/**
 * 回答要点
 */

// 2PC 两阶段提交：
// 阶段一（Prepare）：
// - 协调者向所有参与者发送 Prepare 请求
// - 参与者锁定资源，返回 Yes 或 No

// 阶段二（Commit/Rollback）：
// - 所有参与者都返回 Yes：发送 Commit
// - 任何一个返回 No：发送 Rollback

// 2PC 的问题：

// 1. 协调者单点故障：
// - 协调者在发送 Commit 前崩溃
// - 参与者不知道该提交还是回滚
// - 事务处于「未决」状态

// 2. 阻塞问题：
// - 参与者在 Prepare 后必须等待
// - 如果协调者不响应，参与者一直阻塞
// - 持有的锁无法释放

// 3. 数据不一致：
// - 协调者发送 Commit，但部分没送达
// - 部分参与者提交了，部分没有
// - 导致数据不一致
```

### 问题五：什么是 3PC？相比 2PC 有什么改进？

:::tip 回答思路

1. 解释三阶段提交的流程
2. 说明与 2PC 的区别
3. 分析 3PC 的改进和仍然存在的问题

:::

```java
/**
 * 回答要点
 */

// 3PC 三阶段提交：
// 阶段一（CanCommit）：
// - 协调者询问：能否提交？
// - 参与者检查，返回 Yes 或 No

// 阶段二（PreCommit）：
// - 所有返回 Yes：发送 PreCommit
// - 参与者锁定资源，准备提交

// 阶段三（DoCommit）：
// - 协调者发送 DoCommit
// - 参与者正式提交

// 3PC 的改进：
// - 引入超时机制
// - 参与者可以在某些情况下自主决策

// 3PC 的问题：
// - 网络分区时仍可能不一致
// - 性能开销增加
// - 实际上没有被广泛使用
```

## Seata 相关问题

### 问题六：Seata 有哪些模式？有什么区别？

:::tip 回答思路

1. 列举 Seata 的三种模式
2. 简要说明每种模式的特点
3. 对比适用场景

:::

```java
/**
 * 回答要点
 */

// Seata 的三种模式：

// 1. AT 模式：
// - 基于 Undo Log 自动回滚
// - 对业务零侵入
// - SQL 有限制
// - 适合大多数场景

// 2. TCC 模式：
// - Try-Confirm-Cancel 三个阶段
// - 手动实现，代码侵入
// - 性能高，灵活
// - 适合资源预留场景

// 3. SAGA 模式：
// - 正向操作 + 补偿操作
// - 不锁定资源
// - 适合长事务
// - 不保证隔离性
```

### 问题七：AT 模式是如何工作的？

:::tip 回答思路

1. 解释 AT 模式的核心机制（Undo Log）
2. 说明前后镜像的概念
3. 描述完整的执行流程

:::

```java
/**
 * 回答要点
 */

// AT 模式核心：Undo Log

// 执行流程：
// 1. 解析 SQL，获取表名和主键
// 2. 生成 Before Image（查询修改前的数据）
// 3. 执行 SQL
// 4. 生成 After Image（查询修改后的数据）
// 5. 记录 Undo Log（Before Image + After Image + SQL）
// 6. 提交本地事务

// 回滚流程：
// 1. 解析 Undo Log
// 2. 生成反向 SQL
// 3. 执行反向 SQL

// 为什么 AT 性能高：
// - 本地事务先提交
// - Undo Log 可以异步删除
// - 全局锁持有时间短
```

### 问题八：TCC 模式的三个阶段是什么？

:::tip 回答思路

1. 详细解释 Try、Confirm、Cancel 三个阶段
2. 给出具体业务场景的例子
3. 说明 TCC 模式需要注意的问题

:::

```java
/**
 * 回答要点
 */

// TCC 三阶段：

// 1. Try（预留）：
// - 检查业务是否允许执行
// - 预留资源（如冻结库存）
// - 不做真正的业务操作

// 2. Confirm（确认）：
// - 执行真正的业务操作
// - 使用 Try 阶段预留的资源
// - 必须成功，否则不断重试

// 3. Cancel（取消）：
// - 释放 Try 阶段预留的资源
// - 回滚业务操作
// - 必须成功，否则不断重试

// TCC 注意事项：
// - 空回滚：Try 未执行，Cancel 执行了
// - 幂等性：Confirm/Cancel 可能重复执行
// - 悬挂：Cancel 先执行，Try 后执行
```

### 问题九：TCC 的空回滚、幂等、悬挂问题如何解决？

:::tip 回答思路

1. 解释每个问题的原因
2. 给出具体的解决方案
3. 可以用代码示例说明

:::

```java
/**
 * 回答要点
 */

// 1. 空回滚：
// 原因：Try 超时失败，但实际可能成功了

// 解决：幂等表
// - 在 Cancel 前检查 Try 是否已执行
// - 只有 Try 执行过才执行 Cancel

// 2. 幂等性：
// 原因：Confirm/Cancel 可能重复执行

// 解决：状态检查 + 唯一约束
// - 执行前检查状态
// - 使用数据库唯一约束保证幂等

// 3. 悬挂：
// 原因：Cancel 先执行，Try/Confirm 后执行

// 解决：状态检查
// - Cancel 执行后，标记状态为 Canceled
// - Try/Confirm 检查状态，如果已 Cancel，不再执行
```

### 问题十：AT 模式和 XA 模式的区别是什么？

:::tip 回答思路

1. 说明两者的相同点
2. 说明两者的不同点
3. 分析性能差异的原因

:::

```java
/**
 * 回答要点
 */

// 相同点：
// - 都基于两阶段提交
// - 都使用全局锁保证隔离性
// - 都需要数据源支持

// 不同点：

// XA 模式：
// - 数据库原生 XA 支持
// - 数据库完成 Prepare
// - 数据库锁定资源时间长

// AT 模式：
// - Seata 代理数据源
// - Seata 完成 Prepare（写 Undo Log）
// - 数据库立即提交
// - 数据库锁定资源时间短

// 性能差异原因：
// - XA：数据库 Prepare 后等待指令，锁定资源
// - AT：本地事务先提交，全局锁异步获取
```
