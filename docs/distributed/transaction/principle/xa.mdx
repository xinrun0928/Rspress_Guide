# XA 规范与两阶段提交

<Badge text="分布式事务" type="tip" /> <Badge text="原理篇" type="warning" /> <Badge text="XA 规范" type="info" />

在分布式事务的世界里，XA 规范是一个绕不开的话题。它是分布式事务领域的「工业标准」，几乎所有的关系型数据库都支持 XA 规范。

理解 XA 规范，不仅能帮助我们理解 Seata 的 AT 模式，还能帮助我们理解分布式事务的本质问题。

## XA 规范概述

### XA 的历史

```java
/**
 * XA 规范的起源
 */

// XA 来自 X/Open 组织（现在的 Open Group）
// X/Open 是一个制定开放系统标准的联盟

// 1991 年，X/Open 发布 DTP 模型（Distributed Transaction Processing）
// XA 规范是 DTP 模型中的核心部分

/**
 * XA 规范的核心思想
 */

// XA 规范定义了一套标准的接口
// 让应用可以透明地处理分布式事务
// 而不需要关心具体的数据源实现
```

### X/Open DTP 模型

```java
/**
 * DTP 模型的组件
 */

// 1. Application（应用程序）
// - 事务的发起者
// - 使用 TM 提供的接口

// 2. Resource Manager（资源管理器）
// - 管理共享资源
// - 如：数据库、消息队列
// - 实现 XA 接口

// 3. Transaction Manager（事务管理器）
// - 协调和管理分布式事务
// - 与 RM 通信
// - 实现事务的提交或回滚
```

## XA 规范详解

### XA 接口

```java
/**
 * XA 规范的核心接口
 */

// 应用程序调用的接口：
// xa.start(xid, flags)    - 开启一个事务分支
// xa.end(xid, flags)      - 结束一个事务分支
// xa.prepare(xid)         - 准备提交
// xa.commit(xid, onePhase)- 提交事务
// xa.rollback(xid)       - 回滚事务
// xa.recover(flags)       - 恢复未决事务

/**
 * XID 结构
 */

// XID 是全局事务 ID
// 结构：
// - formatID: 格式标识符
// - gtrid: 全局事务 ID
// - bqual: 分支限定符
```

### MySQL XA 实现

```sql
-- MySQL XA 事务示例

-- 开启全局事务
XA START 'gtrid1','bqual1',1;

-- 执行分支事务
UPDATE account SET balance = balance - 100 WHERE id = 1;
INSERT INTO orders (...) VALUES (...);

-- 结束分支
XA END 'gtrid1','bqual1',1;

-- 准备（第一阶段）
XA PREPARE 'gtrid1','bqual1',1;

-- 提交（第二阶段）
XA COMMIT 'gtrid1','bqual1',1;
```

/**
 * Java XA 接口
 */

import javax.transaction.xa.XAResource;
import javax.transaction.xa.Xid;

/**
 * XAResource 接口
 */
public interface XAResource {
    
    /**
     * 开始一个事务分支
     */
    void start(Xid xid, int flags) throws XAException;
    
    /**
     * 结束一个事务分支
     */
    void end(Xid xid, int flags) throws XAException;

     * 两阶段提交的第一阶段：准备

     * 两阶段提交的第二阶段：提交

    void commit(Xid xid, boolean onePhase) throws XAException;
    
    /**
     * 回滚事务
     */
    void rollback(Xid xid) throws XAException;
    
    // ... 其他方法
}

## 两阶段提交详解

### 第一阶段：Prepare（准备）

```java
/**
 * Prepare 阶段
 */

// 协调者向所有参与者发送 Prepare 请求

// 参与者执行以下操作：
// 1. 锁定资源
// 2. 记录 Undo Log
// 3. 执行 SQL
// 4. 返回 Ready 或 No

/**
 * 参与者的响应
 */

// Ready：
// - 已准备好提交
// - 可以提交也可以回滚

// No 或 ReadOnly：
// - 不需要参与提交
// - 资源已释放
```

```java
/**
 * Prepare 阶段的代码示例
 */
public class XACommitPhase1 {
    
    /**
     * 协调者发送 Prepare
     */
    public boolean prepare(String xid, List<XAResource> resources) {
        for (XAResource resource : resources) {
            try {
                // 1. 开启分支事务
                resource.start(xid, XAResource.TMNOFLAGS);
                
                // 2. 执行 SQL（这里由应用执行）
                // doBusinessLogic();
                
                // 3. 结束分支
                resource.end(xid, XAResource.TMSUCCESS);
                
                // 4. 发送 Prepare
                int result = resource.prepare(xid);
                
                // 5. 检查结果
                if (result == XAResource.XA_RDONLY) {
                    // 只读事务，不需要提交
                    continue;
                }
                
                // 等待所有资源的 Prepare 结果
                // 如果有任何一个返回非 OK，执行回滚
                
            } catch (XAException e) {
                // Prepare 失败，发送 Rollback
                return false;
            }
        }
        
        return true;
    }
}
```

### 第二阶段：Commit（提交）

```java
/**
 * Commit 阶段
 */

// 协调者收到所有参与者的 Ready 后
// 向所有参与者发送 Commit

// 参与者执行以下操作：
// 1. 提交本地事务
// 2. 释放锁
// 3. 发送 Ack
```

```java
/**
 * Commit 阶段的代码示例
 */
public class XACommitPhase2 {
    
    /**
     * 协调者发送 Commit
     */
    public void commit(String xid, List<XAResource> resources) {
        for (XAResource resource : resources) {
            try {
                // 提交分支事务
                resource.commit(xid, false);  // false = 两阶段提交
            } catch (XAException e) {
                // 处理提交失败
                // 可能需要重试或记录日志
                handleCommitFailure(resource, xid, e);
            }
        }
    }
}
```

## XA 的问题

### 协调者单点故障

```java
/**
 * 协调者故障问题
 */

// 场景：协调者在发送 Commit 前崩溃

// Coordinator                 Participant
//     |                            |
//     |<-- Ready ------------------|
//     |                            |
//     X [崩溃]                     |
//                                v
//                           永久等待...

// 参与者不知道该提交还是回滚
// 事务处于「未决」状态

/**
 * 解决方案：事务恢复
 */

// 协调者重启后，需要恢复未决事务
// 通过 xa.recover() 获取未决事务
// 重新发送 Commit 或 Rollback
```

### 阻塞问题

```java
/**
 * XA 的阻塞问题
 */

// 在 Prepare 后，参与者必须等待协调者的指令
// 在此期间，参与者持有锁

// 阻塞场景：
// 1. 协调者崩溃
// 2. 网络分区
// 3. 参与者崩溃

// 影响：
// - 参与者持有的锁无法释放
// - 其他事务无法获取锁
// - 可能导致死锁
```

### 性能问题

```java
/**
 * XA 的性能问题
 */

// 1. 多次网络通信
// - Prepare 需要所有参与者参与
// - Commit 需要所有参与者参与

// 2. 锁持有时间长
// - Prepare 后直到 Commit 才能释放锁
// - 时间 = 事务执行时间 + 网络延迟

// 3. 数据库支持有限
// - 不是所有数据库都支持 XA
// - MySQL XA 有一些限制
```

## MySQL XA 的实现细节

```java
/**
 * MySQL XA 的限制
 */

// 1. 只支持 InnoDB 引擎
// 2. XA 事务和普通事务不能混用
// 3. XA 事务不支持 SAVEPOINT
// 4. 对 binlog 格式有要求
// 5. 如果使用 binlog-sync，XA 事务必须一起提交

/**
 * MySQL XA 的性能
 */

// XA 事务比普通事务慢 3-10 倍
// 主要开销：
// - 两阶段提交协议
// - XID 生成和解析
// - 额外的事务日志
```

## Seata AT 与 XA 的关系

```java
/**
 * Seata AT 模式 vs XA
 */

// 相同点：
// - 都基于两阶段提交
// - 都使用全局锁保证隔离性
// - 都需要数据源支持

/**
 * 不同点
 */

// XA 模式：
// - 数据库原生 XA 支持
// - 数据库完成 Prepare
// - 锁定资源时间长

// AT 模式：
// - Seata 代理数据源
// - Seata 完成 Prepare（写 Undo Log）
// - 数据库立即提交
// - 锁定资源时间短
```

## XA 的使用场景

```java
/**
 * XA 适用场景
 */

// 1. 强一致性要求
// - 金融交易
// - 库存扣减
// - 账户转账

// 2. 多数据源场景
// - 同时操作多个数据库

// 3. 异构数据源
// - MySQL + Oracle + PostgreSQL
// - 通过 XA 统一事务管理
```

## 总结

XA 规范是分布式事务的工业标准：

1. **两阶段提交**：Prepare + Commit
2. **标准接口**：应用透明处理分布式事务
3. **强一致性**：通过锁保证
4. **问题**：单点故障、阻塞、性能

虽然 XA 有其局限性，但它是理解分布式事务的基础。Seata AT 模式正是借鉴了 XA 的思想，并针对其性能问题做了优化。
