# TCC 问题分析

<Badge text="分布式事务" type="tip" /> <Badge text="TCC" type="warning" /> <Badge text="问题处理" type="info" />

TCC 模式虽然性能高、灵活性强，但在实际使用中有一些棘手的问题需要处理。这一章，让我们深入分析 TCC 模式中的三大问题：**空回滚**、**幂等性**和**悬挂**，以及 Seata 是如何解决这些问题的。

## TCC 的三大问题

```java
/**
 * TCC 的三大问题
 */

// 1. 空回滚（Blank Rollback）
//    Try 未执行，Cancel 执行了

// 2. 幂等性（Idempotency）
//    Confirm/Cancel 可能重复执行

// 3. 悬挂（Suspension）
//    Cancel 先执行，Try 后执行
```

## 空回滚

### 问题描述

```java
/**
 * 空回滚场景
 */

// 正常流程：
// 1. Try 执行成功，冻结资源
// 2. Confirm 执行，扣减冻结资源
// 3. 完成

// 异常流程 1（超时）：
// 1. Try 执行成功，冻结资源
// 2. TC 没有收到 Confirm 请求（网络超时）
// 3. TC 触发回滚
// 4. Cancel 执行，解冻资源
// 5. 完成

// 空回滚场景：
// 1. Try 执行成功
// 2. TC 没有收到 Try 的结果（网络超时）
// 3. TC 触发回滚
// 4. Cancel 执行（但资源没有被冻结！）
// 5. 空回滚发生
```

### 问题原因

```java
/**
 * 空回滚的原因
 */

// 1. Try 超时返回失败
//    - Try 方法执行了，但返回超时
//    - TC 误以为 Try 失败，发起 Cancel

// 2. 网络分区
//    - Try 请求丢失
//    - TC 收不到 Try 响应
//    - 触发回滚

// 3. Try 方法抛异常
//    - Try 方法内部抛异常
//    - TC 触发回滚
```

### 解决方案

```java
/**
 * 空回滚的解决方案
 */

// 核心思想：检查 Try 是否已经执行过
// 只有 Try 执行过，才执行 Cancel

/**
 * 解决方案：幂等表
 */
@Service
public class InventoryTccServiceImpl implements InventoryTccService {
    
    @Autowired
    private BranchTransactionDao branchTransactionDao;
    
    @Override
    public boolean cancel(BusinessActionContext context) {
        // 1. 获取分支事务 ID
        Long xid = Long.parseLong(context.getXid());
        Long branchId = context.getBranchId();
        
        // 2. 检查 Try 是否已执行
        BranchTransaction tryBranch = branchTransactionDao.findByXidAndBranchId(xid, branchId);
        
        // 3. Try 未执行，空回滚，直接返回成功
        if (tryBranch == null || tryBranch.getStatus() != BranchStatus.TRY_SUCCESS) {
            log.info("空回滚，Try 未执行，branchId={}", branchId);
            return true;
        }
        
        // 4. 执行 Cancel
        Long productId = Long.parseLong(context.getActionContext("productId").toString());
        Integer count = Integer.parseInt(context.getActionContext("count").toString());
        
        inventoryDao.unfreezeStock(productId, count);
        
        // 5. 更新状态
        branchTransactionDao.updateStatus(branchId, BranchStatus.CANCELED);
        
        return true;
    }
}

/**
 * 分支事务状态表
 */
@Entity
@Table(name = "branch_transaction")
public class BranchTransaction {
    
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    
    private String xid;
    private Long branchId;
    private String actionName;        // Try / Confirm / Cancel
    private BranchStatus status;       // 状态
    
    // 其他字段...
}

public enum BranchStatus {
    TRY,           // Try 已执行
    TRY_SUCCESS,   // Try 成功
    CONFIRMED,     // 已 Confirm
    CANCELED,      // 已 Cancel
}
```

## 幂等性

### 问题描述

```java
/**
 * 幂等性问题场景
 */

// 场景 1：网络抖动导致重复请求
// 1. Confirm 请求发送到 RM
// 2. RM 执行 Confirm
// 3. 网络抖动，请求超时
// 4. TC 重试 Confirm
// 5. RM 再次执行 Confirm
// 6. 资源被重复扣减！

// 场景 2：RM 响应超时
// 1. Confirm 执行成功
// 2. RM 发送 Ack
// 3. Ack 丢失
// 4. TC 重试 Confirm
// 5. RM 再次执行 Confirm
// 6. 资源被重复扣减！
```

### 解决方案

```java
/**
 * 幂等性解决方案
 */

// 核心思想：检查是否已经执行过
// 如果已经执行过，直接返回成功

/**
 * 解决方案 1：状态检查
 */
@Override
public boolean confirm(BusinessActionContext context) {
    // 1. 获取分支事务 ID
    Long branchId = context.getBranchId();
    
    // 2. 检查是否已经 Confirm
    BranchTransaction branch = branchTransactionDao.findByBranchId(branchId);
    
    if (branch.getStatus() == BranchStatus.CONFIRMED) {
        // 已经 Confirm 过，直接返回成功
        log.info("幂等，Confirm 已执行，branchId={}", branchId);
        return true;
    }
    
    // 3. 执行 Confirm
    Long productId = Long.parseLong(context.getActionContext("productId").toString());
    Integer count = Integer.parseInt(context.getActionContext("count").toString());
    
    inventoryDao.deductFrozenStock(productId, count);
    
    // 4. 更新状态
    branchTransactionDao.updateStatus(branchId, BranchStatus.CONFIRMED);
    
    return true;
}

/**
 * 解决方案 2：数据库唯一约束
 */
@Entity
@Table(name = "inventory_freeze",
    uniqueConstraints = @UniqueConstraint(columnNames = {"product_id", "order_id"}))
public class InventoryFreeze {
    
    private Long productId;
    private Long orderId;
    private Integer frozenCount;
    private Integer status;  // 1-冻结, 2-已扣减, 3-已解冻
    
    // 其他字段...
}

/**
 * 幂等实现
 */
@Override
public boolean confirm(BusinessActionContext context) {
    Long productId = Long.parseLong(context.getActionContext("productId").toString());
    Long orderId = Long.parseLong(context.getActionContext("orderId").toString());
    Integer count = Integer.parseInt(context.getActionContext("count").toString());
    
    // 使用唯一约束保证幂等
    try {
        // 尝试更新，如果记录不存在会抛异常
        int rows = inventoryDao.deductFrozenWithUniqueConstraint(
            productId, orderId, count, InventoryStatus.FROZEN, InventoryStatus.DEDUCTED);
        
        return rows > 0;
    } catch (DuplicateKeyException e) {
        // 唯一约束冲突，说明已经执行过
        return true;
    }
}
```

## 悬挂

### 问题描述

```java
/**
 * 悬挂问题场景
 */

// 场景：
// 1. Try 执行成功，冻结资源
// 2. Confirm 请求发送到 RM，但网络延迟
// 3. TC 等待 Confirm 超时
// 4. TC 触发 Cancel
// 5. Cancel 执行成功，解冻资源
// 6. Confirm 请求到达
// 7. Confirm 执行，尝试扣减已解冻的资源
// 8. 资源不存在或状态错误，抛出异常
// 9. 悬挂发生：Try 执行了，Confirm 也执行了，但业务逻辑错误
```

### 解决方案

```java
/**
 * 悬挂的解决方案
 */

// 核心思想：检查 Cancel 是否已经执行
// 如果 Cancel 已经执行，不执行 Try/Confirm

/**
 * 解决方案：状态检查
 */
@Override
public boolean tryOperation(BusinessActionContext context) {
    // 1. 获取分支事务 ID
    Long branchId = context.getBranchId();
    
    // 2. 检查 Cancel 是否已执行
    BranchTransaction branch = branchTransactionDao.findByBranchId(branchId);
    
    if (branch != null && branch.getStatus() == BranchStatus.CANCELED) {
        // Cancel 已执行，悬挂，直接返回失败
        log.warn("悬挂，Cancel 已执行，branchId={}", branchId);
        return false;
    }
    
    // 3. 执行 Try
    Long productId = Long.parseLong(context.getActionContext("productId").toString());
    Integer count = Integer.parseInt(context.getActionContext("count").toString());
    
    inventoryDao.freezeStock(productId, count);
    
    // 4. 更新状态
    branchTransactionDao.updateStatus(branchId, BranchStatus.TRY_SUCCESS);
    
    return true;
}

@Override
public boolean confirm(BusinessActionContext context) {
    // 1. 获取分支事务 ID
    Long branchId = context.getBranchId();
    
    // 2. 检查 Cancel 是否已执行
    BranchTransaction branch = branchTransactionDao.findByBranchId(branchId);
    
    if (branch != null && branch.getStatus() == BranchStatus.CANCELED) {
        // Cancel 已执行，悬挂，直接返回成功（避免异常）
        log.warn("悬挂，Cancel 已执行，branchId={}", branchId);
        return true;
    }
    
    // 3. 执行 Confirm
    Long productId = Long.parseLong(context.getActionContext("productId").toString());
    Integer count = Integer.parseInt(context.getActionContext("count").toString());
    
    inventoryDao.deductFrozenStock(productId, count);
    
    // 4. 更新状态
    branchTransactionDao.updateStatus(branchId, BranchStatus.CONFIRMED);
    
    return true;
}
```

### 代码示例

```java
/**
 * Seata TCC 最佳实践
 */
@Service
public class InventoryTccServiceImpl implements InventoryTccService {
    
    @Autowired
    private InventoryDao inventoryDao;
    
    @Override
    public boolean tryOperation(BusinessActionContext context, 
                               Long productId, Integer count) {
        // 1. 冻结库存
        int rows = inventoryDao.freezeStock(productId, count);
        
        return rows > 0;
    }
    
    @Override
    public boolean confirm(BusinessActionContext context) {
        // 1. 获取 Try 阶段的参数
        Long productId = Long.parseLong(
            context.getActionContext("productId").toString());
        Integer count = Integer.parseInt(
            context.getActionContext("count").toString());
        
        // 2. 扣减冻结的库存
        // Seata 会自动处理幂等性
        // 使用 DELETE + RETURNING 或 UPDATE + WHERE 保证幂等
        int rows = inventoryDao.deleteFreezeStock(productId, count);
        
        // 3. 如果 rows == 0，说明已经执行过（幂等）
        // Seata 认为这是正常的
        return true;
    }
    
    @Override
    public boolean cancel(BusinessActionContext context) {
        // 1. 获取 Try 阶段的参数
        Long productId = Long.parseLong(
            context.getActionContext("productId").toString());
        Integer count = Integer.parseInt(
            context.getActionContext("count").toString());
        
        // 2. 解冻库存
        // Seata 会自动处理空回滚
        // 如果没有冻结记录，rows == 0
        int rows = inventoryDao.unfreezeStock(productId, count);
        
        return true;
    }
}
```

### 数据库设计

```sql
-- 冻结表设计
CREATE TABLE inventory_freeze (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    product_id BIGINT NOT NULL,
    order_id VARCHAR(64) NOT NULL COMMENT '订单 ID，也是分支事务 ID',
    frozen_count INT NOT NULL COMMENT '冻结数量',
    status TINYINT NOT NULL DEFAULT 1 COMMENT '1-冻结, 2-已扣减, 3-已解冻',
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    update_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY uk_order_product (order_id, product_id),
    INDEX idx_product_status (product_id, status)
);

-- 冻结时插入记录
INSERT INTO inventory_freeze (product_id, order_id, frozen_count, status)
VALUES (?, ?, ?, 1);

-- Confirm 时删除记录（幂等）
DELETE FROM inventory_freeze 
WHERE order_id = ? AND product_id = ? AND status = 1;

-- Cancel 时更新状态（幂等）
UPDATE inventory_freeze 
SET status = 3 
WHERE order_id = ? AND product_id = ? AND status = 1;
```

## 总结

TCC 的三大问题是实际使用中必须面对的：

1. **空回滚**：Try 未执行，Cancel 执行了
   - 解决：检查 Try 是否已执行

2. **幂等性**：Confirm/Cancel 重复执行
   - 解决：状态检查 + 数据库唯一约束

3. **悬挂**：Cancel 先执行，Try/Confirm 后执行
   - 解决：检查 Cancel 是否已执行

Seata 框架层面已经做了大量处理，但开发者仍需注意：

- 数据库设计要支持幂等
- 状态转换要合理
- 日志记录要完整
