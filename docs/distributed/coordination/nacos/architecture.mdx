# Nacos 架构

理解一个系统的架构，是掌握它如何工作的前提。Nacos 虽然表面上是服务发现和配置管理平台，但其内部架构涉及一致性协议、数据同步、高并发处理等多个方面。

本节将深入剖析 Nacos 的架构设计。

## 核心模块

### Naming Service（命名服务）

负责服务注册与发现：

| 组件 | 说明 |
|---|---|
| `ServiceManager` | 服务管理器，维护服务元数据 |
| `ServiceRegistry` | 服务注册器，处理注册请求 |
| `DistroProtocol` | 分布式协议，支持临时实例 |
| `RaftProtocol` | 一致性协议，支持持久实例 |

### Config Service（配置服务）

负责配置管理：

| 组件 | 说明 |
|---|---|
| `ConfigManager` | 配置管理器 |
| `ConfigInfo` | 配置信息实体 |
| `ConfigFilterChain` | 配置过滤器链 |
| `ConfigChangeHook` | 配置变更钩子 |

## 一致性协议

Nacos 支持两种一致性协议，适配不同场景：

```
┌─────────────────────────────────────────────────────────────┐
│                    Nacos 一致性协议                          │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│                      CP 模式 (Raft)                          │
│                      ──────────────                          │
│                      适用：持久实例、配置                     │
│                                                              │
│  ┌─────────┐                                                 │
│  │ Leader  │ ◀── 选举产生                                    │
│  └────┬────┘                                                 │
│       │                                                      │
│  ┌────┴────┐ ┌─────────┐                                     │
│  │Follower │ │Follower │ ← 过半确认                         │
│  └─────────┘ └─────────┘                                     │
│                                                              │
│                      AP 模式 (Distro)                       │
│                      ──────────────                          │
│                      适用：临时实例                           │
│                                                              │
│  ┌─────────┐   ┌─────────┐   ┌─────────┐                    │
│  │ Node 1  │◀──│ Node 2  │──▶│ Node 3  │                    │
│  │ (写)    │   │ (同步)  │   │ (同步)  │                    │
│  └─────────┘   └─────────┘   └─────────┘                    │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### Raft 协议（CP）

对于需要强一致性的场景（如持久实例、配置），Nacos 使用 Raft 协议：

```java
// Raft 协议核心组件
public class RaftCore {
    // Leader 选举
    public void leaderElect();

    // 日志复制
    public void replicate(LogEntry entry);

    // 状态同步
    public void syncFromLeader();
}
```

### Distro 协议（AP）

对于高可用场景（如临时实例），Nacos 使用 Distro 协议：

```java
// Distro 协议核心组件
public class DistroProtocol {
    // 数据同步
    public void syncData(DataChange dataChange);

    // 健康检查
    public void healthCheck();

    // 数据分片
    public String getResponsibleKey(String key);
}
```

## 核心流程

### 服务注册流程

```java
// 服务注册核心流程
public void registerInstance(ServiceInstance instance) {
    // 1. 检查命名空间
    if (StringUtils.isBlank(instance.getNamespaceId())) {
        instance.setNamespaceId(getDefaultNamespaceId());
    }

    // 2. 创建服务（如果不存在）
    createServiceIfAbsent(instance.getServiceName(), instance.getGroupName());

    // 3. 添加实例
    addInstance(instance);

    // 4. 同步到其他节点（Distro 协议）
    syncServiceInstance(instance);
}
```

### 配置发布流程

```java
// 配置发布核心流程
public boolean publishConfig(String dataId, String group, String content) {
    // 1. 检查参数
    checkTenantInfo(dataId, group);

    // 2. 计算 MD5
    String md5 = MD5Utils.md5Hex(content, Constants.ENCODE);

    // 3. 写入数据库
    ConfigInfo configInfo = new ConfigInfo(dataId, group, content);
    configInfo.setMd5(md5);

    boolean success = configInfoPersistService.insertOrUpdate(configInfo);

    // 4. 通知变更
    if (success) {
        notifyConfigChange(dataId, group);
    }

    return success;
}
```

## 总结

Nacos 的架构设计体现了几个核心理念：

1. **双协议支持**：Raft（CP）用于持久数据，Distro（AP）用于临时数据
2. **存储分离**：单机嵌入式 Derby，集群 MySQL
3. **插件化设计**：Protocol、Storage 都支持插件扩展
4. **运维友好**：内置控制台，监控指标丰富

理解 Nacos 的架构，有助于更好地使用和运维这个平台。
