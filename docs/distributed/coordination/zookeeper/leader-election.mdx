# ZooKeeper Leader 选举

|ZooKeeper| |Leader 选举| |FastLeaderElection|

想象一个没有领导者的团队：每个人都可以做任何决定。当需要协调时，所有人聚在一起讨论，效率低下，偶尔还会出现两个人做了同样的工作。

分布式系统也一样。ZooKeeper 集群需要一个 Leader 来统一协调所有写操作，确保数据一致性。

但问题来了：**谁来当 Leader？** 这就是 Leader 选举要解决的问题。

## 为什么需要 Leader 选举

### ZooKeeper 的架构

ZooKeeper 集群采用主从架构：

```
┌─────────────────────────────────────────────────────────────┐
│                      ZooKeeper 集群                          │
│                                                             │
│   ┌─────────┐                                               │
│   │ Leader  │ ◀── 所有写操作都经过 Leader                    │
│   └────┬────┘                                               │
│        │                                                    │
│   ┌────┴────┐                                               │
│   ▼         ▼                                               │
│ ┌──────┐ ┌──────┐                                           │
│ │Follow│ │Follow│ ← 接收客户端请求，转发给 Leader              │
│ │ er   │ │ er   │ ← 参与投票                                │
│ └──────┘ └──────┘                                           │
│                                                             │
│   ┌─────────┐                                               │
│   │Observer│ ← 只同步数据，不参与投票（可选）                  │
│   └─────────┘                                               │
└─────────────────────────────────────────────────────────────┘
```

### 写操作的流程

```
客户端 ───▶ Follower ────▶ Leader ────▶ 广播到所有服务器
                     │
                     ▼
              过半服务器确认
                     │
                     ▼
              客户端收到响应
```

### Leader 的职责

| 职责 | 说明 |
|---|---|
| 接收写请求 | 所有写操作必须经过 Leader |
| 发起事务 | 将写操作封装为事务 |
| 同步数据 | 将事务同步给所有 Follower |
| 协调操作 | 分布式锁、序列号生成等 |

## FastLeaderElection 算法

ZooKeeper 使用 FastLeaderElection 算法进行 Leader 选举。这个算法的核心思想是：**谁的服务器 ID 最大、数据最新，谁就当 Leader**。

### 选举的触发时机

1. 集群启动时
2. Leader 崩溃时
3. Leader 无法响应时（过半服务器认为 Leader 不可用）

### 选举参数

每个服务器参与选举时，会携带以下信息：

| 参数 | 说明 |
|---|---|
| `serverId` | 服务器 ID（myid 文件中配置，越大越优先） |
| `zxid` | 最后一次事务 ID（越大表示数据越新） |
| `epoch` | 当前投票轮次（也叫 logicalclock） |

### 选举过程

#### 第一轮：自荐投票

```java
// 每个服务器先投票给自己
public Vote startElection(int myId, long myZxid) {
    Vote myVote = new Vote(myId, myZxid, ElectionEpoch, myId);
    return myVote;
}
```

#### 第二轮：交换选票

```
服务器 1 (ID=1)                    服务器 2 (ID=2)
     │                                  │
     │──── Vote(1, 100) ────────────────▶│
     │◀─── Vote(2, 150) ────────────────│
     │                                  │
     │   比较：ID(2) > ID(1)            │
     │   采纳 Vote(2, 150)              │
     │                                  │
```

#### 第三轮：收敛到一致

当一个服务器收到过半相同选票时，选举结束：

```
服务器 1: 收到过半 Vote(2, 150)
         → 选举完成，2 号服务器是 Leader

服务器 2: 收到过半 Vote(2, 150)
         → 选举完成，2 号服务器是 Leader

服务器 3: 收到过半 Vote(2, 150)
         → 选举完成，2 号服务器是 Leader
```

### 选举算法的 Java 伪代码

```java
public class FastLeaderElection {

    // 投票
    static class Vote {
        int serverId;
        long zxid;
        long electionEpoch;
    }

    public Vote lookForLeader() {
        Map<Integer, Vote> votes = new HashMap<>();
        int myId = self.getId();
        long myZxid = self.getCurrentZxid();

        Vote myVote = new Vote(myId, myZxid, 0, myId);
        int updates = 0;

        while (true) {
            // 发送自己的投票
            sendNotification(myVote);

            // 接收其他服务器的投票
            Notification n = recvQueue.poll();

            if (n == null) {
                // 没有收到投票，发送投票
                if (updates > 0) {
                    updates = 0;
                }
                // 重新投票
            } else {
                // 处理收到的投票
                if (n.electionEpoch > electionEpoch) {
                    // 投票更新，回归当前轮次
                    electionEpoch = n.electionEpoch;
                    updates = 0;
                }

                // 收到过半票
                if (isMajority(votes)) {
                    // 选举完成
                    return getWinningVote(votes);
                }

                // 记录投票
                votes.put(n.serverId, n.vote);

                // 比较并更新自己的投票
                if (totalOrder(getWinningVote(votes), myVote) > 0) {
                    myVote = getWinningVote(votes);
                    updates++;
                }
            }
        }
    }
}
```

## 选举的投票规则

投票遵循以下优先级（从高到低）：

1. **epoch**：投票轮次越大，优先级越高
2. **zxid**：事务 ID 越大，数据越新
3. **serverId**：服务器 ID 越大，优先级越高

```
优先级比较：

(epoch=5, zxid=100, serverId=3) > (epoch=5, zxid=100, serverId=2)
                                                │
                                            serverId 大的优先

(epoch=5, zxid=100, serverId=3) > (epoch=4, zxid=200, serverId=3)
                                    │
                                epoch 大的优先

(epoch=5, zxid=100, serverId=3) > (epoch=5, zxid=99, serverId=3)
                                        │
                                    zxid 大的优先
```

## 集群启动时的选举

假设 3 台服务器的集群：

| 服务器 | myid | zxid |
|---|---|---|
| 192.168.1.1 | 1 | 0 |
| 192.168.1.2 | 2 | 0 |
| 192.168.1.3 | 3 | 0 |

```
时间线：

T1: 所有服务器启动，都投自己
    服务器 1: 票 = Vote(1, 0)
    服务器 2: 票 = Vote(2, 0)
    服务器 3: 票 = Vote(3, 0)

T2: 交换选票后，所有服务器都看到 3 号票数过半
    → 选举完成
    → 3 号是 Leader，zxid=0
```

## Leader 崩溃时的选举

假设运行中 Leader 崩溃：

```
初始状态：
  Leader(2): zxid=500
  Follower(1): zxid=480
  Follower(3): zxid=490

T1: Leader(2) 崩溃
T2: Follower(1) 和 Follower(3) 发起选举
    服务器 1: 票 = Vote(1, 480)
    服务器 3: 票 = Vote(3, 490)

T3: 交换选票后
    服务器 1: 看到 3 号票数过半
              → 选举完成，3 号是 Leader

T4: 服务器 3 成为新 Leader
    新 Leader zxid = max(480, 490) = 490
    新 epoch = old_epoch + 1
```

## Observer 的作用

Observer 不参与投票，但同步数据：

```properties
# zoo.cfg
peerType=observer

server.1=192.168.1.1:2888:3888
server.2=192.168.1.2:2888:3888
server.3=192.168.1.3:2888:3888:observer
```

```
                    ┌──────────────┐
                    │   Observer   │
                    └──────┬───────┘
                           │
        客户端 ────▶ 读请求 ──┘
                           │
                           ▼
                      直接返回数据
                      （无需经过 Leader）

                    ┌──────────────┐
                    │    Leader    │
                    └──────┬───────┘
                           │
        写请求 ────▶ 事务处理
                           │
                           ▼
                      过半确认
                           │
                           ▼
                      同步到 Observer
```

## 选举的注意事项

### 集群节点数量

ZooKeeper 集群通常部署奇数台：

| 节点数 | 容错能力 | 适用场景 |
|---|---|---|
| 3 | 允许 1 台故障 | 小规模集群 |
| 5 | 允许 2 台故障 | 生产环境 |
| 7 | 允许 3 台故障 | 大规模高可用 |

:::warning
不要部署偶数台服务器！偶数台不会提高容错能力，反而会浪费资源。如果 4 台服务器中有 2 台故障，集群仍然不可用（没过半）。
:::

### 网络分区（脑裂）

ZooKeeper 通过「过半」原则避免脑裂：

```
情况：4 台服务器分裂为 2+2

  分区 A: 服务器 1, 2          分区 B: 服务器 3, 4
        │                            │
        └──▶ Leader 1                └──▶ Leader 2 (无法选出)

结果：两个分区都无法选出 Leader（都没过半）
      集群对外不可用

      ✓ 避免了脑裂
```

## 总结

ZooKeeper 的 Leader 选举是保证分布式一致性的关键：

1. **FastLeaderElection**：基于服务器 ID 和事务 ID 的选举算法
2. **过半原则**：保证选举结果唯一，避免脑裂
3. **epoch + zxid**：确保选举出的 Leader 数据最新
4. **Observer**：扩展读性能的方案，不参与投票

理解 Leader 选举后，你就可以更好地理解 ZooKeeper 如何保证高可用和强一致性。
