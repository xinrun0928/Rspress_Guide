# ZooKeeper：分布式协调的鼻祖

你有没有想过这个问题：

Kafka 的 Controller 是怎么选出来的？Hadoop 的 NameNode 高可用是怎么做到的？Dubbo 的注册中心除了 Nacos 还能用什么？

**答案都是 ZooKeeper。**

ZooKeeper 是一个老牌（2007 年诞生）的分布式协调服务，经历了无数生产环境的考验。虽然近年来被 etcd、Nacos 抢了不少风头，但在某些场景下，它依然是不可替代的选择。

## ZooKeeper 的三大核心能力

```
┌─────────────────────────────────────────────────────────┐
│                   ZooKeeper 核心架构                   │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  1. 数据模型：ZNode 树                                  │
│     └─ 持久节点、临时节点、顺序节点、版本号              │
│                                                         │
│  2. Watch 机制：事件驱动                                │
│     └─ 监听节点变化，变更时主动通知                      │
│                                                         │
│  3. 选举机制：ZAB 协议                                  │
│     └─ Leader 崩溃后快速选举，保证数据一致性              │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

## ZNode 数据模型

ZooKeeper 的数据存储在树形结构的 ZNode 中：

```
/dubbo
  /providers         ← 临时节点：服务注册
  /consumers         ← 临时节点：消费者订阅
  /config            ← 持久节点：配置信息
  /master            ← 临时节点：Leader 选举
```

| 类型 | 说明 | 典型用途 |
|------|------|---------|
| 持久节点 | 创建后一直存在，直到主动删除 | 配置存储 |
| 临时节点 | 会话结束后自动删除 | 服务注册、心跳 |
| 顺序节点 | 节点名自动带递增序号 | 分布式锁、选举 |

## 适用场景

| 场景 | 说明 | 优势 |
|------|------|------|
| Leader 选举 | 临时节点 + 最小序号 | 实现简单，天然有序 |
| 分布式锁 | 临时顺序节点 + Watch | 公平锁，避免羊群效应 |
| 配置管理 | 持久节点 + Watch | 配置变更实时推送 |
| 服务发现 | 临时节点注册 | 节点下线自动清理 |

## ZooKeeper vs etcd vs Nacos

| 维度 | ZooKeeper | etcd | Nacos |
|------|-----------|------|-------|
| 一致性协议 | ZAB | Raft | Raft |
| Watch 机制 | 一次性触发 | 持续监听 | 持续监听 |
| 数据模型 | ZNode 树 | Key-Value | Key-Value |
| 生态 | Hadoop、KBase、 Kafka | Kubernetes | Spring Cloud |
| 社区活跃度 | 低（停止维护） | 高 | 高 |

## 核心公式

```
ZooKeeper = ZAB一致性 + ZNode树 + 一次性Watch + 临时节点

适用场景：Kafka Controller > Leader选举 > 分布式锁 > 配置中心
不适场景：需要配置热更新（Watch 不支持）、Spring Cloud 生态（Nacos 更合适）
```
