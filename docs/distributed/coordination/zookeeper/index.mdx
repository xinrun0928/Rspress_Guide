# ZooKeeper 简介

在一个繁忙的十字路口，没有红绿灯，没有交警，车辆和行人各走各的——很快就会乱成一锅粥。分布式系统也一样：没有协调机制，服务之间会乱成一团。

ZooKeeper 就是这个十字路口的「交通枢纽」——它是一个分布式协调服务，让分布式系统能够有序地工作。

## ZooKeeper 是什么

Apache ZooKeeper 是一个开源的分布式协调服务，最初是 Hadoop 项目的一部分，后来独立成为 Apache 顶级项目。它被广泛应用于各类分布式系统中：

| 知名项目 | ZooKeeper 用途 |
|---|---|
| Hadoop | NameNode 高可用 |
| HBase | Master 选举、RegionServer 注册 |
| Kafka | Broker 注册、Controller 选举 |
| Dubbo | 服务注册中心 |
| RocketMQ | NameServer 集群协调 |

## 设计目标

ZooKeeper 的设计哲学是**简单**——它只做一件事：保持分布式系统有序。

```
┌─────────────────────────────────────────────────────────────┐
│                    ZooKeeper 设计目标                        │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  简单（Simple）          ──▶  树形命名空间，类似文件系统      │
│                                                              │
│  有序（Ordered）         ──▶  顺序节点保证请求顺序           │
│                                                              │

│  高性能（Performant）    ──▶  每秒处理数万请求                │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

## 核心特性

### 树形命名空间

ZooKeeper 的数据存储在类似文件系统的树形结构中：

```
ZooKeeper Server
        │
        └── / (根)
            ├── /configs
            │   └── /database
            │       └── host = 192.168.1.100
            ├── /services
            │   ├── /order
            │   │   ├── /instance1
            │   │   └── /instance2
            │   └── /inventory
            └── /locks
                └── /distributed-lock
```

每个节点叫 **ZNode**，可以存储数据（最大 1MB），可以有子节点。

### Watch 机制

客户端可以订阅节点变化，当变化发生时收到通知：

```java
// 订阅配置变化
zk.getData("/configs/database", event -> {
    if (event.getType() == EventType.NodeDataChanged) {
        System.out.println("配置已更新！");
    }
}, null);
```

### 集群部署

ZooKeeper 集群通常部署奇数台服务器：

```
┌─────────────────────────────────────────────────────────────┐
│                    ZooKeeper 集群                          │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│    Client                                                   │
│       │                                                     │
│       ├───────────────────────────────┤                    │
│       ▼                               ▼                     │
│  ┌─────────┐                   ┌─────────┐                  │
│  │ Server1 │◀─── Leader ─────▶│ Server2 │                  │
│  │(Follower)│                  │(Follower)│                │
│  └────┬────┘                   └────┬────┘                  │
│       │                               │                      │
│       └──────────────┬────────────────┘                      │
│                      ▼                                       │
│                ┌─────────┐                                  │
│                │ Server3 │                                  │
│                │(Follower)│                                  │
│                └─────────┘                                  │
│                                                              │
└─────────────────────────────────────────────────────────────┘

过半原则：
- 3 台服务器允许 1 台故障
- 5 台服务器允许 2 台故障
- 7 台服务器允许 3 台故障
```

## 典型应用场景

### 服务注册与发现

```java
// 服务启动时注册
String path = zk.create("/services/order/" + instanceId,
    "192.168.1.10:8080".getBytes(),
    ZooDefs.Ids.OPEN_ACL_UNSAFE,
    CreateMode.EPHEMERAL);

// 服务消费者订阅
List<String> instances = zk.getChildren("/services/order", true);
```

### 分布式锁

```java
// 创建临时顺序节点
String lockPath = zk.create("/locks/order-",
    null,
    ZooDefs.Ids.OPEN_ACL_UNSAFE,
    CreateMode.EPHEMERAL_SEQUENTIAL);

// 获取锁（等待更小的节点删除）
while (true) {
    List<String> locks = zk.getChildren("/locks", false);
    if (isSmallest(lockPath, locks)) {
        break;  // 获取锁成功
    }
    zk.exists(preLockPath, watch);
}
```

### 配置管理

```java
// 存储配置
zk.setData("/configs/app/feature-flags",
    "{\"enableNewUI\":true}".getBytes(), -1);

// 监听配置变化
zk.getData("/configs/app/feature-flags", event -> {
    // 配置变化，刷新本地缓存
}, null);
```

### Master 选举

```java
// 竞争创建临时节点
try {
    zk.create("/master",
        instanceId.getBytes(),
        ZooDefs.Ids.OPEN_ACL_UNSAFE,
        CreateMode.EPHEMERAL);
    // 成为 Master
} catch (NodeExistsException e) {
    // 已有 Master，监听其变化
    zk.exists("/master", event -> {
        // Master 宕机，重新竞争
    });
}
```

## 总结

ZooKeeper 是分布式协调领域的老牌选手：

1. **简单可靠**：只提供协调原语，不捆绑服务发现等功能
2. **成熟稳定**：经过大量生产环境验证
3. **生态完善**：Dubbo、Kafka 等都依赖它

如果你需要一个纯粹的分布式协调服务，ZooKeeper 依然是很好的选择。后续章节我们将深入探讨它的数据模型、Watch 机制、Leader 选举等核心内容。
