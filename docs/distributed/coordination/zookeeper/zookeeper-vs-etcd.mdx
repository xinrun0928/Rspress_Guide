# ZooKeeper vs etcd

|ZooKeeper| |etcd| |选型对比|

你的公司正在设计一套新的分布式系统，面临一个选择：ZooKeeper 还是 etcd？两者都能实现分布式协调，但各有特点。

ZooKeeper 历史悠久，Dubbo、Kafka、HBase 都用它；etcd 则是后起之秀，Kubernetes 默认用它。选哪个？

## 核心对比

| 维度 | ZooKeeper | etcd |
|---|---|---|
| **诞生时间** | 2010 年 | 2013 年 |
| **语言** | Java | Go |
| **一致性协议** | ZAB 协议 | Raft 协议 |
| **数据模型** | ZNode 树 | Key-Value + 目录 |
| **API** | 自定义协议（ZK Client） | gRPC/HTTP |
| **Watch** | 一次性 Watch | 长期订阅（流式） |
| **性能** | ~3 万 QPS | ~2 万 QPS |
| **生态** | 成熟（Dubbo、Kafka） | 云原生（K8s、CoreOS） |

## 一致性协议对比

### ZAB vs Raft

两者都是 Paxos 的变种，但设计重点不同：

```
┌─────────────────────────────────────────────────────────────┐
│                      一致性协议对比                          │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  ZAB (ZooKeeper)          Raft (etcd)                       │
│  ─────────────            ────────────                       │
│                                                              │
│  专注于主备复制            通用一致性协议                      │
│  写请求必须经过 Leader     写请求必须经过 Leader              │
│  支持恢复模式              日志复制                           │
│  epoch + zxid             term + index                      │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### ZAB 的特点

```java
// ZAB 协议的核心概念
public class ZAB {
    // Epoch：选举轮次
    long epoch;

    // Zxid：事务 ID（高 32 位 epoch + 低 32 位 counter）
    long zxid;

    // 消息类型
    enum MessageType {
        PROPOSAL,    // 提案
        ACK,         // 确认
        COMMIT       // 提交
    }
}
```

### Raft 的特点

```java
// Raft 协议的核心概念
public class Raft {
    // Term：任期号
    int term;

    // Index：日志索引
    int index;

    // 角色
    enum Role {
        FOLLOWER,    // 跟随者
        CANDIDATE,   // 候选人
        LEADER       // 领导者
    }

    // 日志条目
    static class LogEntry {
        int term;
        int index;
        Object command;
    }
}
```

## 数据模型对比

### ZooKeeper ZNode

```
/services
├── /order
│   ├── /instances
│   │   ├── 192.168.1.10:8080 (临时节点)
│   │   └── 192.168.1.11:8080 (临时节点)
│   └── config (数据节点)
└── /inventory
    └── instances
```

- 树形结构
- 每个节点可存储数据（`<` 1MB）
- 支持五种节点类型
- 原子操作（事务）

### etcd Key-Value

```bash
# etcd 的 Key-Value 模型
# Key 支持层次结构，用 / 分隔

/services/order/instances/192.168.1.10:8080
/services/order/config
/services/inventory/instances/192.168.1.20:8080

# 范围查询
etcdctl get /services/order --prefix
```

- 扁平化的 Key-Value（但支持前缀模拟目录）
- 支持范围查询
- 支持事务
- 支持 Lease（类似临时节点）

## API 对比

### ZooKeeper API

```java
// ZooKeeper 原生 API
public class ZKClient {

    private ZooKeeper zk;

    // 增删改查
    String create(String path, byte[] data, List<ACL> acl, CreateMode mode);
    void delete(String path, int version);
    Stat setData(String path, byte[] data, int version);
    byte[] getData(String path, boolean watch, Stat stat);
    List<String> getChildren(String path, boolean watch);

    // ACL
    List<ACL> getACL(String path, Stat stat);
    Stat setACL(String path, List<ACL> acl, int version);

    // 事务
    Transaction transaction();
}

// 事务操作
Transaction txn = zk.transaction();
txn.create(path1, data1, acl, CreateMode.PERSISTENT);
txn.delete(path2, -1);
txn.setData(path3, data3, -1);
List<OpResult> results = txn.commit();
```

### etcd API

```bash
# etcd 命令行 API
etcdctl put /services/order "order-service:8080"
etcdctl get /services/order
etcdctl del /services/order

# 带条件的更新（CAS）
etcdctl put /config/database "mysql://localhost" --lease=abc123

# 范围查询
etcdctl get /services/ --prefix

# 事务
etcdctl txn --insecure-tnx-url='http://localhost:2379/v3/txn'
```

```java
// etcd Java Client
public class EtcdClient {

    // put 操作
    client.put("/services/order", "order-service:8080")
          .lease(LeaseId)     // 绑定 Lease
          .prevKV()           // 返回旧值
          .sync();

    // get 操作
    GetResponse response = client.get("/services/order")
                                  .prefix()     // 前缀匹配
                                  .countOnly()  // 只返回计数
                                  .sync();

    // 事务
    client.get("/key")
          .then(kv -> {
              if (kv.getValue().equals("expected")) {
                  return client.put("/key", "newValue");
              }
              return client.get("/key");
          });

    // Watch
    client.get("/services/order")
          .watch()
          .putObject((key, value) -> {
              System.out.println("新增: " + key + " = " + value);
          })
          .delete((key) -> {
              System.out.println("删除: " + key);
          });
}
```

## Watch 机制对比

### ZooKeeper 的一次性 Watch

```java
// ZooKeeper Watch：每次变化后需要重新注册
zk.getData("/config/app", event -> {
    if (event.getType() == EventType.NodeDataChanged) {
        System.out.println("配置变化: " + new String(zk.getData("/config/app", true, null)));
        // 需要重新注册 Watch！
    }
}, null);

// 批量监听（使用 NodeCache）
PathChildrenCache cache = new PathChildrenCache(zk, "/services", true);
cache.start();
cache.getListenable().addListener((curatorClient, event) -> {
    // Curator 会自动处理重新注册
    System.out.println("变化: " + event.getData().getPath());
});
```

### etcd 的流式 Watch

```java
// etcd Watch：持续监听，无需重新注册
Watch watch = client.getWatchClient();
watch.watch("/services/order")
     .create();

watch.listen-events().forEach(event -> {
    switch (event.getEventType()) {
        case PUT:
            System.out.println("新增: " + event.getKeyValue().getKey());
            break;
        case DELETE:
            System.out.println("删除: " + event.getKeyValue().getKey());
            break;
    }
});

// 或使用简化 API
client.getWatchClient().watch("/services/order")
      .putObject((key, value) -> { /* 处理新增/更新 */ })
      .delete((key) -> { /* 处理删除 */ })
      .start();
```

## Lease 机制对比

### ZooKeeper 的临时节点

```java
// ZooKeeper 临时节点
// 客户端断开连接后自动删除
zk.create("/services/instance1",
    "192.168.1.10:8080".getBytes(),
    ZooDefs.Ids.OPEN_ACL_UNSAFE,
    CreateMode.EPHEMERAL);
// 客户端断开时自动删除
```

### etcd 的 Lease

```java
// etcd Lease：租约机制
// 可以设置 TTL，到期后 Key 自动删除

// 创建 Lease（TTL = 30 秒）
Lease lease = client.getLeaseClient()
                    .grant(30)
                    .sync();

// 绑定 Key 到 Lease
client.getKVClient()
      .put(ByteSequence.from("/services/instance1"),
           ByteSequence.from("192.168.1.10:8080"))
      .withLease(lease.getID())
      .sync();

// 续约
client.getLeaseClient().keepAlive(lease.getID());

// 查看剩余 TTL
TimeToLiveResponse ttl = client.getLeaseClient()
                                .timeToLive(lease.getID())
                                .sync();
System.out.println("剩余 TTL: " + ttl.getTTL() + "秒");
```

## 选型建议

### 选 ZooKeeper 的场景

```
✓ 已有 Kafka、Dubbo、HBase 等依赖 ZK 的组件
✓ 需要成熟稳定的分布式协调
✓ 团队有 Java 技术栈经验
✓ 需要强一致性的服务发现
```

### 选 etcd 的场景

```
✓ 云原生项目，使用 Kubernetes
✓ 需要 HTTP/gRPC 访问
✓ 团队有 Go 语言能力
✓ 需要 Watch 长期订阅
✓ 配置文件较多，需要范围查询
```

### 不要混用的场景

```
✗ 不要同时使用 ZooKeeper 和 etcd 做同样的事情
✗ 增加系统复杂度
✗ 两套运维体系
✗ 数据同步问题
```

## 性能对比

### 测试数据

| 操作 | ZooKeeper | etcd |
|---|---|---|
| 写入（单节点） | ~3 万/秒 | ~2 万/秒 |
| 读取（单节点） | ~10 万/秒 | ~20 万/秒 |
| 集群写入（3 节点） | ~1 万/秒 | ~1 万/秒 |
| Watch 触发延迟 | ~5-10ms | ~2-5ms |

### 影响因素

```java
// ZooKeeper 性能瓶颈
public class ZKPerformance {
    // 1. Leader 是单点，所有写操作串行
    // 2. 每个写操作需要同步到多数节点
    // 3. Watch 基于网络通知

    // 优化建议：
    // - 使用 Observer 扩展读
    // - 批量写操作
    // - 合理设置 tickTime
}

// etcd 性能瓶颈
public class EtcdPerformance {
    // 1. BoltDB 写入是串行的
    // 2. Raft 日志复制
    // 3. gRPC 编解码开销

    // 优化建议：
    // - 使用 SSD
    // - 批量操作
    // - 合理使用 Lease
}
```

## 总结

| 维度 | ZooKeeper | etcd |
|---|---|---|
| **一致性** | ZAB，强一致 | Raft，强一致 |
| **性能** | 写入更快 | 读取更快 |
| **API** | ZK Client | HTTP/gRPC |
| **Watch** | 一次性，需重新注册 | 流式，持续监听 |
| **生态** | 大数据生态 | 云原生生态 |
| **运维** | 成熟稳定 | 新兴但成熟 |

**没有绝对的好坏，只有适合的场景**。如果你在构建云原生系统，优先选择 etcd；如果你的系统已经依赖 ZooKeeper，没必要切换。
