# ZooKeeper 数据模型

ZooKeeper 是一个分布式协调服务，它的设计哲学是**简单**。与其他复杂的分布式系统不同，ZooKeeper 只提供了一种核心能力——**分层命名空间**，并围绕这个能力构建了一系列分布式协作原语。

理解 ZooKeeper 的数据模型，是掌握分布式协调服务的基础。

## 文件系统与数据模型的类比

如果你熟悉 Linux 文件系统，ZooKeeper 的数据模型会让你感到非常亲切。

```
Linux 文件系统                          ZooKeeper 数据模型
───────────────────────────────────────────────────────────────
/home/user/file.txt        ←────→     /home/user:config
/etc/nginx/nginx.conf                   /services/order:lock
                                        /configs/db:master
───────────────────────────────────────────────────────────────
```

两者都是**树形结构**，每个节点叫**ZNode**，可以存储数据，可以有子节点。

## ZNode 的四种类型

ZooKeeper 中的节点（ZNode）有四种类型，每种类型都有其独特的用途：

| 类型 | 说明 | 应用场景 |
|---|---|---|
| **持久节点** | 创建后一直存在，直到主动删除 | 配置存储、服务注册 |
| **临时节点** | 客户端断开连接后自动删除 | 心跳检测、临时服务 |
| **持久顺序节点** | 持久 + 顺序编号 | 分布式锁 |
| **临时顺序节点** | 临时 + 顺序编号 | 分布式锁、队列 |

### 持久节点（PERSISTENT）

```java
// 创建持久节点
zk.create("/configs/app", "config_data".getBytes(),
    ZooDefs.Ids.OPEN_ACL_UNSAFE, CreateMode.PERSISTENT);

// 节点创建后，即使创建它的客户端退出，节点依然存在
```

### 临时节点（EPHEMERAL）

```java
// 创建临时节点
zk.create("/services/worker1", "alive".getBytes(),
    ZooDefs.Ids.OPEN_ACL_UNSAFE, CreateMode.EPHEMERAL);

// 客户端断开连接后，节点自动删除
// 常用于心跳检测：节点存在 = 服务在线
```

### 顺序节点（SEQUENTIAL）

```java
// 创建持久顺序节点
zk.create("/locks/order_lock_", null,
    ZooDefs.Ids.OPEN_ACL_UNSAFE, CreateMode.PERSISTENT_SEQUENTIAL);
// 可能创建出：/locks/order_lock_0000000001

// 创建临时顺序节点
zk.create("/locks/waiter_", null,
    ZooDefs.Ids.OPEN_ACL_UNSAFE, CreateMode.EPHEMERAL_SEQUENTIAL);
// 可能创建出：/locks/waiter_0000000002
```

## ZNode 的数据结构

每个 ZNode 包含以下信息：

```java
public class Stat {
    // 版本信息
    long czxid;        // 创建时的事务 ID
    long mzxid;        // 最后修改时的事务 ID
    long ctime;        // 创建时间（毫秒）
    long mtime;        // 最后修改时间（毫秒）

    // 节点信息
    int version;        // 数据版本号
    int cversion;       // 子节点版本号
    int aversion;       // ACL 版本号

    // 其他
    long ephemeralOwner; // 如果是临时节点，记录持有者会话 ID
    int dataLength;       // 数据长度
    int numChildren;     // 子节点数量
    long pzxid;          // 最后修改子节点的事务 ID
}
```

### 获取节点详细信息

```java
Stat stat = new Stat();
byte[] data = zk.getData("/configs/app", true, stat);

System.out.println("版本号: " + stat.getVersion());
System.out.println("创建时间: " + stat.getCtime());
System.out.println("数据长度: " + stat.getDataLength());
System.out.println("子节点数: " + stat.getNumChildren());
```

## ZNode 的版本机制

ZooKeeper 使用**乐观锁**的版本控制机制：

| 版本 | 说明 |
|---|---|
| `version` | 数据版本，每次修改 +1 |
| `cversion` | 子节点版本 |
| `aversion` | ACL 版本 |

### 乐观锁的使用

```java
// 设置数据时指定版本
Stat stat = new Stat();
byte[] data = zk.getData("/configs/app", false, stat);
int currentVersion = stat.getVersion();

// 只有版本匹配时才能更新成功
zk.setData("/configs/app", newData.getBytes(), currentVersion);

// 如果其他客户端已经修改过，版本不匹配，更新失败
// 这是 CAS（Compare-And-Swap）思想
```

### 典型应用：配置更新

```java
@Service
public class ConfigService {

    private ZooKeeper zk;

    /**
     * 乐观锁更新配置
     */
    public boolean updateConfig(String path, String newValue) {
        try {
            // 获取当前版本
            Stat stat = new Stat();
            zk.getData(path, false, stat);

            // 尝试更新
            zk.setData(path, newValue.getBytes(), stat.getVersion());
            return true;

        } catch (BadVersionException e) {
            // 版本冲突，需要重试
            log.warn("配置更新冲突，需要重试");
            return false;
        }
    }
}
```

## Watch 机制

Watch 是 ZooKeeper 最强大的特性之一——它允许客户端**订阅**节点变化，当变化发生时**主动推送**给客户端。

### Watch 的工作原理

```
客户端                          ZooKeeper
   │                                │
   │──── getData(path, watch) ────▶│
   │                                │
   │    ◀──── 返回数据和stat ──────│
   │                                │
   │   ... 等待 ...                 │
   │                                │  服务器数据变化
   │                                │
   │◀─── Watch事件通知 ─────────────│
   │                                │
   │   注意：Watch是一次性的！       │
   │   需要再次注册才能继续监听      │
```

### Watch 使用示例

```java
public class ConfigWatcher {

    private ZooKeeper zk;
    private String configPath;

    public void watchConfig() throws Exception {
        // 注册 Watch
        byte[] data = zk.getData(configPath, event -> {
            // Watch 回调
            if (event.getType() == Event.EventType.NodeDataChanged) {
                System.out.println("配置发生变化: " + new String(event.getData()));
                // 重新读取配置
                try {
                    watchConfig();  // 重新注册 Watch
                } catch (Exception e) {
                    e.printStackTrace();
                }
            }
        }, null);

        System.out.println("当前配置: " + new String(data));
    }
}
```

### Watch 的类型

| 事件类型 | 说明 |
|---|---|
| `NodeCreated` | 节点被创建 |
| `NodeDeleted` | 节点被删除 |
| `NodeDataChanged` | 节点数据变化 |
| `NodeChildrenChanged` | 子节点列表变化 |

### Watch 的注意事项

:::warning
**Watch 是一次性的！** 当 Watch 被触发后，需要重新注册才能继续监听。这是 ZooKeeper 的设计取舍，换取的是简单性和可靠性。
:::

```java
// 错误做法：Watch 只触发一次
zk.getData("/path", true, stat);
// 只注册一次，之后的变化监听不到

// 正确做法：每次收到 Watch 后重新注册
zk.getData("/path", event -> {
    if (event.getType() == Event.EventType.NodeDataChanged) {
        // 处理变化
        try {
            // 重新注册 Watch
            zk.getData("/path", this, stat);
        } catch (Exception e) {
            e.printStackTrace();
        }
    }
}, stat);
```

## 典型目录结构设计

在实际项目中，ZooKeeper 的目录结构需要合理设计：

```
/
├── configs/                    # 配置中心
│   ├── database/
│   │   ├── host               # 数据库地址
│   │   └── port               # 数据库端口
│   └── redis/
│       ├── host
│       └── port
│
├── services/                  # 服务注册
│   ├── order/                  # 订单服务
│   │   ├── 192.168.1.10:8080  # 实例1
│   │   └── 192.168.1.11:8080  # 实例2
│   └── inventory/
│       └── 192.168.1.20:8080
│
├── locks/                     # 分布式锁
│   ├── order_lock/
│   └── inventory_lock/
│
├── master/                    # Master 选举
│   └── leader                 # 当前 Leader
│
└── workers/                   # 任务队列
    ├── pending/
    └── completed/
```

## 与 etcd/nacos 的对比

| 维度 | ZooKeeper | etcd | Nacos |
|---|---|---|---|
| 数据模型 | ZNode 树 | Key-Value + 目录 | Key-Value + 目录 |
| 访问协议 | 自定义协议 | gRPC/HTTP | HTTP/DNS |
| Watch 机制 | 一次性 Watch | 长期订阅 | 长期订阅 |
| 监听方式 | 回调通知 | 轮询/流 | WebSocket/轮询 |
| 成熟度 | 非常成熟 | 成熟 | 较新 |

## 总结

ZooKeeper 的数据模型简洁而强大：

1. **树形结构**：类似文件系统，易于理解
2. **四种节点类型**：持久、临时、持久顺序、临时顺序
3. **版本控制**：乐观锁机制，支持 CAS 操作
4. **Watch 机制**：事件订阅，但需要一次性处理

理解这些基础概念后，你就可以更好地理解 ZooKeeper 如何实现分布式锁、服务注册、配置管理等高级功能。
