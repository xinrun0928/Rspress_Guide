## etcd 在 K8s 中的角色

### 唯一数据源

Kubernetes 采用 **声明式 API**，所有集群状态都存储在 etcd 中：

| 数据类型 | 存储路径 | 说明 |
|---|---|---|
| Pod | `/registry/pods/` | Pod 规格和状态 |
| Service | `/registry/services/` | 服务定义 |
| ConfigMap | `/registry/configmaps/` | 配置数据 |
| Secret | `/registry/secrets/` | 敏感信息 |
| Deployment | `/registry/deployments/` | 部署状态 |
| Node | `/registry/minions/` | 节点信息 |

### 数据示例

```bash
# 查看 etcd 中的数据
etcdctl --cacert=/etc/kubernetes/pki/etcd/ca.crt \
        --cert=/etc/kubernetes/pki/etcd/server.crt \
        --key=/etc/kubernetes/pki/etcd/server.key \
        get /registry/pods/default --prefix

# 输出示例

{"kind":"Pod","apiVersion":"v1","metadata":
```

### API Server 源码解析

```go
// API Server 中的 etcd 存储层
// k8s.io/apiserver/pkg/storage/etcd3/store.go

type Store struct {
    client        *clientv3.Client
    codec         runtime.Codec
    versioner     storage.Versioner
    transformer   value.Transformer
}

// 创建资源
func (s *Store) Create(ctx context.Context, key string, obj, out runtime.Object, ttl uint64) error {
    // 1. 编码对象
    data, err := s.codec.Encode(obj)
    if err != nil {
        return err
    }

    // 2. 添加元数据（创建时间、UID 等）
    meta, err := s.objectMeta(obj)
    if err != nil {
        return err
    }

    // 3. 写入 etcd
    txnResp, err := s.client.Txn(ctx).
        If(s.createObjNotFoundErr(key)).
        Then(clientv3.OpPut(key, data, options...))

    return err
}

// 读取资源
func (s *Store) Get(ctx context.Context, key string, objPtr runtime.Object) error {
    // 1. 从 etcd 读取
    getResp, err := s.client.Get(ctx, key)

    // 2. 解码对象
    return s.codec.Decode(getResp.Kvs[0].Value, objPtr)
}
```

### 控制器如何使用 Informer

```go
// Kubernetes 控制器使用 Informer
// 例如：Deployment Controller

type DeploymentController struct {
    // Informer
    deploymentInformer cache.SharedIndexInformer
    rsInformer        cache.SharedIndexInformer
    podInformer       cache.SharedIndexInformer

    // Workqueue
    queue workqueue.RateLimitingInterface
}

func (dc *DeploymentController) Run(threadiness int, stopCh <-chan struct{}) {
    // 启动 Informer
    go dc.deploymentInformer.Run(stopCh)
    go dc.rsInformer.Run(stopCh)
    go dc.podInformer.Run(stopCh)

    // 启动 Worker 处理变化
    for i := 0; i < threadiness; i++ {
        go wait.Until(dc.worker, time.Second, stopCh)
    }
}

func (dc *DeploymentController) enqueueDeployment(obj interface{}) {
    // 将变化放入队列
    key, err := controller.KeyFunc(obj)
    dc.queue.Add(key)
}

func (dc *DeploymentController) worker() {
    // 处理队列中的任务
    for {
        key, shutdown := dc.queue.Get()
        if shutdown {
            return
        }

        defer dc.queue.Done(key)

        // 调谐：确保实际状态匹配期望状态
        err := dc.syncDeployment(key.(string))
        if err != nil {
            dc.queue.AddRateLimited(key)
        }
    }
}
```

### etcd 备份与恢复

```bash
# 1. 备份
etcdctl snapshot save /backup/etcd-snapshot.db \
    --endpoints=https://127.0.0.1:2379 \
    --cacert=/etc/kubernetes/pki/etcd/ca.crt \
    --cert=/etc/kubernetes/pki/etcd/server.crt \
    --key=/etc/kubernetes/pki/etcd/server.key

# 2. 检查备份
etcdctl snapshot status /backup/etcd-snapshot.db

# 3. 恢复（需要停止 kube-apiserver）
etcdctl snapshot restore /backup/etcd-snapshot.db \
    --data-dir=/var/lib/etcd-restore \
    --name=etcd-1 \
    --initial-cluster=etcd-1=https://192.168.1.1:2380,etcd-2=...

# 4. 重启 etcd 和 kube-apiserver
systemctl restart etcd
systemctl restart kube-apiserver
```

## 性能调优

### 常见问题

| 问题 | 症状 | 解决方案 |
|---|---|---|
| etcd 延迟高 | API Server 超时 | 使用 SSD、增强硬件 |
| 选举超时 | 集群频繁重新选主 | 增大 heartbeat-interval |
| 存储空间不足 | etcd 启动失败 | 配置配额、压缩旧数据 |
| Watch 阻塞 | 控制器不工作 | 增加 API Server 副本 |

### 监控指标


## 常见运维场景

### 场景一：增加 etcd 节点

```bash
# 1. 添加新节点到集群
etcdctl member add etcd-4 \
    --peer-urls=https://192.168.1.4:2380

# 2. 在新节点上启动 etcd
etcd --name=etcd-4 \
    --initial-cluster=etcd-1=... \
    --initial-cluster-state=existing \
    --peer-urls=https://192.168.1.4:2380
```

### 场景二：清理过期数据

```bash
# 1. 检查存储使用
etcdctl endpoint status

# 2. 执行 defragmentation（释放未使用的空间）
etcdctl defrag --endpoints=https://127.0.0.1:2379

# 3. 清理历史版本（需要 API Server v1.11+）
kubectl exec -it -n kube-system etcd-<pod-name> -- \
    etcdctl compact 100000
```

### 场景三：灾难恢复

```bash
# 1. 在所有节点上停止 etcd
systemctl stop etcd

# 2. 在一个节点上恢复快照
etcdctl snapshot restore /backup/etcd-snapshot.db \
    --data-dir=/var/lib/etcd \
    --name=etcd-1

# 3. 重启所有 etcd 节点
systemctl restart etcd

# 4. 验证集群健康
etcdctl --endpoints=https://127.0.0.1:2379 endpoint health
```

## 总结

etcd 在 Kubernetes 中扮演着**数据中枢**的角色：

1. **唯一数据源**：所有集群状态都存储在 etcd 中
2. **API Server 依赖**：K8s 组件通过 API Server 与 etcd 交互
3. **高可用设计**：通过 Raft 协议保证数据强一致
4. **Informer 机制**：K8s 控制器通过 Watch 同步集群状态

理解 etcd 的工作原理，对于运维 Kubernetes 集群至关重要。
