# etcd：Kubernetes 的心脏

你知道 Kubernetes 为什么能保持集群状态一致吗？

答案很简单：**因为它把所有的集群状态都存在了 etcd 里。**

etcd 是一个分布式 Key-Value 存储，被 Kubernetes 用作唯一的真相来源（Single Source of Truth）。Master 节点的所有状态——Pod 调度信息、Service 端点、ConfigMap——全都存在 etcd。

这就是 etcd 的核心定位：**高可用的分布式一致性存储**。

## etcd 核心特性

```
┌─────────────────────────────────────────────────────────┐
│                     etcd 核心能力                        │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  1. Raft 一致性协议                                     │
│     └─ 强leader，支持leader选举                         │
│                                                         │
│  2. MVCC 多版本并发控制                                 │
│     └─ 每次写入生成新版本，支持历史读取                   │
│                                                         │
│  3. Watch 机制                                          │
│     └─ 持续监听key变化，实时推送变更                    │
│                                                         │
│  4. Range 查询                                          │
│     └─ 支持前缀查询，模拟目录结构                       │
│                                                         │
│  5. TLS 安全通信                                        │
│     └─ 端到端加密，支持证书认证                         │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

## 适用场景

| 场景 | 说明 |
|------|------|
| Kubernetes 存储 | 服务发现、配置中心、Pod 调度状态 |
| 服务配置中心 | 动态配置管理，支持热更新 |
| 分布式锁 | 基于 RAFT 的强一致锁 |
| Leader 选举 | 协调多节点任务分配 |
| 元数据存储 | 高可靠的小数据存储 |

## etcd vs ZooKeeper vs Consul

| 维度 | etcd | ZooKeeper | Consul |
|------|------|-----------|--------|
| 一致性 | Raft（强一致） | ZAB（强一致） | Raft + gossip |
| 数据模型 | Key-Value + Range | ZNode 树 | Key-Value + Service |
| 健康检查 | TTL lease | KeepAlive | Script/HTTP |
| 监控 | Prometheus metrics | JMX | 内置 UI |
| 生态 | K8s 深度集成 | 大数据生态 | 服务网格 |

## 核心公式

```
etcd = Raft一致性 + MVCC + Watch + Range查询

适用场景：K8s元数据存储 > 配置中心 > Leader选举 > 分布式锁
不适用：大数据生态（Hadoop/Kafka选ZK）、超大数据存储
```
