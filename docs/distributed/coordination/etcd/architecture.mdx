# etcd 架构

Kubernetes 已经成为容器编排的事实标准，而 etcd 就是 Kubernetes 的「记忆」——所有集群状态、配置信息都存储在 etcd 中。如果 etcd 挂了，整个 Kubernetes 集群就会陷入混乱。

但 etcd 不仅仅服务于 Kubernetes，它本身就是一个优秀的分布式键值存储。本节将深入剖析 etcd 的架构设计。

## etcd 是什么

etcd 是一个分布式的、可靠的键值存储，用于存储分布式系统中最关键的数据。它基于 Raft 协议保证一致性，具有高可用、强一致的特点。

| 特性 | 说明 |
|---|---|
| **一致性** | Raft 协议，强一致 |
| **可用性** | 多数派投票，允许少数节点故障 |
| **持久化** | Write-Ahead Log + BoltDB |
| **接口** | gRPC/HTTP，JSON/Protocol Buffers |
| ** Lease** | 租约机制，类似临时节点 |
| **Watch** | 流式监听，支持历史版本 |

## 核心组件

### gRPC Gateway

gRPC Gateway 将 HTTP/JSON 请求转换为 gRPC 调用：

```bash
# HTTP/JSON 请求
curl -X POST http://localhost:2379/v3/kv/put \
  -d '{"key": "aGVsbG8=", "value": "d29ybGQ="}'

# 转换为 gRPC 调用
grpcurl -d '{"key": "aGVsbG8=", "value": "d29ybGQ="} \
  localhost:2379 etcdserverpb.KV.Put
```

### Raft 模块

Raft 模块负责节点间的日志复制和 Leader 选举：

```go
// Raft 核心逻辑简化
type Raft struct {
    id        uint64
    term      uint64
    vote      uint64
    state     StateType
    log       *RaftLog
    peers     map[uint64]Peer
}

// 状态机
type StateType int
const (
    StateFollower  StateType = iota
    StateCandidate
    StateLeader
)
```

### WAL (Write-Ahead Log)

WAL 记录所有写操作，确保数据不丢失：

```bash
# WAL 文件结构
$ ls -la /var/lib/etcd/member/wal/
-rw-r--r--  1 root root  65536 Jan 15 10:00:00  0000000000000000-0000000000.snap
-rw-r--r--  1 root root  65536 Jan 15 10:00:01  0000000000000000-0000000001.snap
-rw-r--r--  1 root root  65536 Jan 15 10:00:02  0000000000000000-0000000002.tmp
```

### BoltDB

BoltDB 是一个嵌入式的键值存储，用于持久化数据：

```go
// BoltDB 操作
db, _ := bolt.Open("etcd.db", 0600, nil)

db.Update(func(tx *bolt.Tx) error {
    b, _ := tx.CreateBucket([]byte("kv"))
    b.Put([]byte("key"), []byte("value"))
    return nil
})
```

## Raft 协议详解

### Leader 选举

```
初始状态：3 台服务器都是 Follower

Follower 1                    Follower 2                    Follower 3
    │                              │                              │
    │ 选举超时（150-300ms 随机）      │                              │
    │ ─────────────────────────────▶│                              │
    │                              │                              │
    │    成为 Candidate，自增 term  │                              │
    │    给自己投票                │                              │
    │                              │  RequestVote RPC              │
    │                              │ ─────────────────────────────▶│
    │                              │                              │
    │                              │              收到多数票        │
    │                              │              成为 Leader       │
    │◀─────────────────────────────│                              │
    │        心跳请求              │                              │
    │◀─────────────────────────────│──────────────────────────────│
    │     Leader 任期内不再选举     │                              │
```

### 日志复制

```go
// Leader 接收客户端请求
type Entry struct {
    Term    uint64  // 任期号
    Index   uint64  // 日志索引
    Data    []byte  // 数据
}

// 日志复制流程
func (r *Raft) sendAppend(to uint64) {
    // 1. 构建 AppendEntries RPC
    prevLogIndex := r.log.LastIndex()
    prevLogTerm := r.log.Term(prevLogIndex)
    entries := r.log.NextEntries()

    rpc := &AppendEntriesRPC{
        Term:         r.term,
        LeaderID:     r.id,
        PrevLogIndex: prevLogIndex,
        PrevLogTerm:  prevLogTerm,
        Entries:      entries,
        LeaderCommit: r.commitIndex,
    }

    // 2. 发送到 Follower
    send(to, rpc)
}
```

### 脑裂与恢复

```
网络分区：节点 1、2 在一侧，节点 3 在另一侧

分区 A: 节点 1 (Leader), 节点 2 (Follower)
分区 B: 节点 3 (Candidate)

分区 A:
  - 继续服务，可以写入
  - term 逐渐增加

分区 B:
  - 选举超时，节点 3 成为 Candidate
  - 自增 term，尝试选举
  - 收不到多数票，无法成为 Leader

网络恢复:
  - 节点 3 收到更高的 term
  - 主动变为 Follower
  - 同步 Leader 的数据
```

## 数据存储结构

### 版本管理

etcd 支持两种版本的键：

```bash
# 创建 key
etcdctl put /config/app "value1"
# OK

# 读取 key
etcdctl get /config/app
# /config/app
# value1

# 读取 key 的创建版本和修改版本
etcdctl --write-out="json" get /config/app
# {
#     "kvs": [{
#         "key": "/config/app",
#         "value": "dmFsdWUx",
#         "create_revision": 3,    # 创建版本
#         "mod_revision": 4        # 修改版本
#     }]
# }
```

### 事务支持

```go
// etcd 事务：比较版本后再写入
// Java Client 写法
kvClient.txn()
    .If(
        newCompare()
            .withKey("/lock/resource")
            .withVersion(1)  // 版本必须为 1
            .withTarget(Compare.CompareResult.EQUAL)
    )
    .Then(
        newPut()
            .withKey("/lock/resource")
            .withValue("locked-by-me")
    )
    .Else(
        newGet()
            .withKey("/lock/resource")
    )
    .commit();
```

## 与 ZooKeeper 对比

| 维度 | etcd | ZooKeeper |
|---|---|---|
| **协议** | Raft | ZAB |
| **语言** | Go | Java |
| **API** | gRPC/HTTP | 自定义协议 |
| **Watch** | 流式 | 一次性 |
| **性能** | 读更快 | 写更快 |
| **生态** | K8s 默认 | 大数据生态 |

## 总结

etcd 的架构设计体现了几个核心理念：

1. **分层设计**：Client Layer → API Layer → Raft Layer → Storage Layer
2. **Raft 一致性**：通过日志复制保证数据一致性
3. **WAL + BoltDB**：WAL 保证故障恢复，BoltDB 提供高效存储
4. **Lease + Watch**：灵活的数据过期和变更监听机制

理解 etcd 的架构，对于深入理解 Kubernetes 的工作原理至关重要。
