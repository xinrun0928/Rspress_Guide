# etcd vs ZooKeeper vs Consul

你的团队正在评估分布式协调服务，市面上有三个主流选择：etcd、ZooKeeper 和 Consul。它们都能实现分布式协调，但各有特点。

etcd 是 Kubernetes 的「御用」存储，ZooKeeper 是久经考验的老将，Consul 则是服务发现的瑞士军刀。选哪个？

## 横向对比

| 维度 | etcd | ZooKeeper | Consul |
|---|---|---|---|
| **诞生** | 2013 年 | 2010 年 | 2014 年 |
| **语言** | Go | Java | Go |
| **一致性** | Raft | ZAB | Raft |
| **数据模型** | Key-Value | ZNode 树 | Key-Value + 服务目录 |
| **API** | gRPC/HTTP | 自定义协议 | HTTP/REST |
| **服务发现** | ✗ | ✗ | ✓ 内置 |
| **健康检查** | ✗ | ✗ | ✓ 内置 |
| **多数据中心** | ✗ | ✗ | ✓ 支持 |
| **ACL** | RBAC | ACL | ACL |
| **性能** | 读 20k+/s | 写 3 万/s | 读 10k+/s |

## API 对比

### ZooKeeper API

```java
// ZooKeeper 使用自定义协议
// 需要使用 ZK Client 库
ZooKeeper zk = new ZooKeeper("localhost:2181", 3000, watcher);

// 增删改查
zk.create("/path", data, acl, CreateMode.PERSISTENT);
zk.getData("/path", watcher, stat);
zk.setData("/path", data, version);
zk.delete("/path", version);

// 批量操作需要事务
Transaction txn = zk.transaction();
txn.create(...);
txn.setData(...);
txn.commit();
```

### etcd API

```bash
# etcd 支持 HTTP API
curl -X POST http://localhost:2379/v3/kv/put \
  -d '{"key": "aGVsbG8=", "value": "d29ybGQ="}'

curl http://localhost:2379/v3/kv/range \
  -d '{"key": "aGVsbG8="}'

# 事务
curl -X POST http://localhost:2379/v3/kv/txn \
  -d '{"compare":[{"target":"VERSION","key":"aGVsbG8=","result":"EQUAL","version":1}],
        "success":[{"requestPut":{"key":"aGVsbG8=","value":"bmV3IHZhbHVl"}}]}'
```

```java
// Java Client for etcd
import io.etcd.jetcd.*;
import io.etcd.jetcd.kv.*;

Client client = Client.builder()
    .endpoint("http://localhost:2379")
    .build();

KV kvClient = client.getKVClient();

// 写入
kvClient.put(ByteSequence.from("key".getBytes()),
             ByteSequence.from("value".getBytes())).get();

// 读取
GetResponse response = kvClient.get(ByteSequence.from("key".getBytes())).get();
System.out.println(new String(response.getKvs().get(0).getValue().getBytes()));
```

### Consul API

## 服务发现对比

### ZooKeeper - 需要自己实现

```java
// ZooKeeper 服务发现（需要自己实现）
public class ZKServiceDiscovery {

    private ZooKeeper zk;

    public List<String> discover(String serviceName) throws Exception {
        String path = "/services/" + serviceName;
        return zk.getChildren(path, false);
    }
}
```

### etcd - 需要自己实现

```java
// etcd 服务发现（需要自己实现）
public class EtcdServiceDiscovery {

    private KV kvClient;

    public List<String> discover(String serviceName) throws Exception {
        String prefix = "/services/" + serviceName + "/";

        GetResponse response = kvClient.get(
            ByteSequence.from(prefix.getBytes()),
            GetOption.newBuilder()
                .withPrefix(ByteSequence.from(prefix.getBytes()))
                .build()
        ).get();

        return response.getKvs().stream()
            .map(kv -> new String(kv.getValue().getBytes()))
            .collect(Collectors.toList());
    }
}
```

### Consul - 内置服务发现

```java
// Consul 服务发现（内置）
// 注册服务
ConsulClient consul = new ConsulClient("localhost", 8500);

NewService newService = new NewService();
newService.setId("order-service-1");
newService.setName("order-service");
newService.setPort(8080);
newService.setCheck(new NewService().setCheck(
    new NewServiceCheck().setHttp("http://localhost:8080/health")
                          .setInterval("10s")
));

consul.agentServiceRegister(newService);

// 发现服务
List<String> services = consul.getCatalogServices().getValue();
List<String> orderServices = consul.getHealthServices(
    "order-service", true, null).getValue()
    .stream()
    .map(HealthService::getService)
    .map(Service::getAddress)
    .collect(Collectors.toList());
```

## 健康检查对比

### ZooKeeper - 无内置健康检查

ZooKeeper 只通过会话超时来检测客户端存活，没有内置的健康检查机制。

### etcd - 无内置健康检查

etcd 也没有内置健康检查，但可以通过 Lease 机制实现简单的存活检测。

```java
// 使用 Lease 实现健康检查
Lease lease = client.getLeaseClient().grant(10).get();

// 绑定 Key 到 Lease
kvClient.put(key, value, PutOption.newBuilder()
    .withLeaseId(lease.getID())
    .build()).get();

// 续约
client.getLeaseClient().keepAlive(lease.getID());
```

### Consul - 强大的健康检查

```java
// Consul 支持多种健康检查

// HTTP 检查
newServiceCheck.setHttp("http://localhost:8080/health");
newServiceCheck.setInterval("10s");

// TCP 检查
newServiceCheck.setTcp("localhost:8080");
newServiceCheck.setInterval("10s");

// Docker 检查
newServiceCheck.setDockerContainerId("container-id");
newServiceCheck.setShell("/bin/check");

// TTL 检查（客户端主动报告）
newServiceCheck.setTtl("30s");  // 30 秒内必须续约

// 脚本检查
newServiceCheck.setScript("/usr/local/bin/check");
newServiceCheck.setInterval("10s");
```

## 多数据中心支持

### ZooKeeper - 不支持

ZooKeeper 集群必须在同一个数据中心，跨数据中心部署需要额外的同步机制。

### etcd - 不支持

etcd 本身不支持多数据中心，但可以通过 federation 模式手动同步数据。

### Consul - 原生支持

```bash
# Consul 多数据中心
#  datacenter = "dc1"
#  primary_datacenter = "dc1"
#  retry_join_wan = ["192.168.1.10:8302"]  # 跨数据中心节点

# 查看数据中心
consul catalog datacenters

# 跨数据中心查询
curl "http://localhost:8500/v1/health/service/order-service?dc=dc2"
```

## 选型决策树

```
需要选择分布式协调服务？

│
├── 需要服务发现 + 健康检查？
│   ├── 是 → Consul
│   └── 否 → 继续判断
│
├── 使用 Kubernetes？
│   ├── 是 → etcd（K8s 默认）
│   └── 否 → 继续判断
│
├── 已有 ZooKeeper 基础设施？
│   ├── 是 → ZooKeeper（继续使用）
│   └── 否 → 继续判断
│
├── 团队 Java 技术栈？
│   ├── 是 → ZooKeeper（生态成熟）
│   └── 否 → etcd 或 Consul
│
└── 需要轻量级方案？
    ├── 是 → etcd（安装简单）
    └── 否 → Consul（功能最全）
```

## 场景分析

### 场景一：Kubernetes 集群

**推荐：etcd**

```
Kubernetes + etcd 是黄金组合：
- K8s 默认使用 etcd
- 官方支持，优化完善
- 与 K8s API Server 无缝集成
```

### 场景二：微服务注册与发现

**推荐：Consul**

```
Consul = 配置中心 + 服务注册 + 健康检查 + DNS

优点：
- 开箱即用，无需额外开发
- 内置健康检查
- 支持多数据中心
- 提供 DNS 接口（service.mesh.service.consul）
```

### 场景三：已有大数据生态

**推荐：ZooKeeper**

```
Hadoop、Kafka、HBase 都用 ZooKeeper：
- 与现有组件兼容
- 团队已有运维经验
- 生态系统完善
```

### 场景四：高并发配置中心

**推荐：etcd**

```
etcd 读取性能更好：
- gRPC 高效序列化
- 内置范围查询
- Watch 机制更高效
```

## 总结

| 维度 | etcd | ZooKeeper | Consul |
|---|---|---|---|
| **服务发现** | 需要自己实现 | 需要自己实现 | **内置** |
| **健康检查** | 无 | 无 | **内置** |
| **多数据中心** | 无 | 无 | **支持** |
| **K8s 集成** | **原生** | 需要适配 | 需要适配 |
| **配置中心** | ✓ | 需要自己实现 | ✓ |
| **Java 生态** | 一般 | **成熟** | 一般 |
| **安装部署** | 简单 | 中等 | **简单** |
| **运维难度** | 低 | 中 | 低 |

**选择建议**：

- **Kubernetes 环境**：etcd
- **微服务 + 服务发现**：Consul
- **大数据生态**：ZooKeeper
- **轻量级配置中心**：etcd
