# 分布式协调服务

分布式系统里，谁来当"裁判"？谁来协调各个节点的步调一致？——这就是分布式协调服务要解决的问题。

在一个分布式系统中，有太多需要协调的事情：
- 多个节点如何选出一个主节点？
- 配置信息如何在集群中同步？
- 分布式锁如何实现？
- 服务发现如何做？

这些问题的答案，都藏在分布式协调服务中。

## 一、为什么需要协调服务

### 1.1 分布式系统的挑战

```
分布式系统面临的问题：
    │
    ├── 一致性问题
    │     └─→ 如何保证多个节点数据一致？
    │
    ├── 协调问题
    │     └─→ 如何协调多个节点的行为？
    │
    ├── 选举问题
    │     └─→ 如何选出一个主节点？
    │
    ├── 配置问题
    │     └─→ 如何同步配置信息？
    │
    └── 服务发现问题
          └─→ 客户端如何找到服务？
```

### 1.2 没有协调服务会怎样

```
场景：没有协调服务的分布式锁
    │
    ├─→ 锁服务 A：SETNX lock = OK
    ├─→ 锁服务 B：SETNX lock = OK (同样的 key)
    │
    └─→ 问题：两个服务都认为自己获取了锁
              数据不一致！
```

### 1.3 协调服务的作用

```
协调服务 = 分布式系统的"神经系统"
    │
    ├── 统一视图：所有节点看到的数据一致
    ├── 选举中心：决定谁是主节点
    ├── 配置中心：同步配置信息
    └── 服务注册：服务发现
```

## 二、主流协调服务介绍

### 2.1 ZooKeeper

ZooKeeper 是最早被广泛使用的分布式协调服务：

```xml
<!-- ZooKeeper 依赖 -->
<dependency>
    <groupId>org.apache.zookeeper</groupId>
    <artifactId>zookeeper</artifactId>
    <version>3.8.0</version>
</dependency>

<!-- Curator（推荐使用） -->
<dependency>
    <groupId>org.apache.curator</groupId>
    <artifactId>curator-recipes</artifactId>
    <version>5.5.0</version>
</dependency>
```

**核心能力**：
- 分布式锁（InterProcessMutex）
- Leader 选举
- 命名服务
- 配置管理
- 服务发现

### 2.2 etcd

etcd 是云原生时代的分布式协调服务：

```yaml
# etcd 部署
version: '3'

services:
  etcd:
    image: quay.io/coreos/etcd:v3.5
    environment:
      - ETCD_NAME=etcd-1
      - ETCD_ADVERTISE_CLIENT_URLS=http://etcd-1:2379
      - ETCD_INITIAL_CLUSTER_STATE=new
```

**核心能力**：
- 键值存储（KV Store）
- 分布式锁（基于 revision）
- Leader 选举
- 监听机制（Watch）
- 事务支持

### 2.3 Nacos

Nacos 是阿里巴巴开源的更易用的协调服务：

```yaml
# Nacos 配置
spring:
  cloud:
    nacos:
      discovery:
        server-addr: nacos-server:8848
```

**核心能力**：
- 服务注册与发现
- 动态配置管理
- DNS 服务
- 流量管理

## 三、核心能力详解

### 3.1 配置管理

**ZooKeeper 配置管理**：

```java
// ZooKeeper 配置管理
@Service
public class ZooKeeperConfigService {

    private final CuratorFramework curator;
    private static final String CONFIG_PATH = "/config/myapp";

    // 监听配置变化
    public void watchConfig() throws Exception {
        curator.getData().usingWatcher(event -> {
            if (event.getType() == Watcher.Event.EventType.NodeDataChanged) {
                // 配置变更，重新加载
                reloadConfig();
            }
        }).forPath(CONFIG_PATH);
    }

    // 更新配置
    public void updateConfig(String config) throws Exception {
        curator.setData().forPath(CONFIG_PATH, config.getBytes());
    }
}
```

**etcd 配置管理**：

```java
// etcd 配置管理
@Service
public class EtcdConfigService {

    private final Client client;
    private static final String CONFIG_KEY = "/config/myapp";

    // 监听配置变化
    public void watchConfig() {
        client.getKVClient().watch(
            ByteSequence.from(CONFIG_KEY, StandardCharsets.UTF_8),
            response -> {
                for (WatchEvent event : response.getEvents()) {
                    if (event.getEventType() == EventType.PUT) {
                        // 配置变更
                        String newValue = event.getKeyValue().getValue().toString(StandardCharsets.UTF_8);
                        reloadConfig(newValue);
                    }
                }
            }
        );
    }
}
```

**Nacos 服务发现**：

```java

@SpringBootApplication
@EnableDiscoveryClient
public class Application {
    public static void main(String[] args) {
        SpringApplication.run(Application.class, args);
    }
}

// Nacos 服务发现
@RestController
public class ServiceController {

    @Autowired
    private NamingService namingService;

    @GetMapping("/instances")
    public List<Instance> getInstances() throws NacosException {
        return namingService.selectInstances("my-service", true);
    }
}
```

### 3.3 Leader 选举

**ZooKeeper Leader 选举**：

```java
// ZooKeeper Leader 选举
LeaderLatch latch = new LeaderLatch(curator, "/leader", instanceId);

latch.addListener(new LeaderLatchListener() {
    @Override
    public void isLeader() {
        // 成为 Leader
        System.out.println("我成为 Leader 了！");
    }

    @Override
    public void notLeader() {
        // 失去 Leader 身份
        System.out.println("我不再是 Leader 了");
    }
});

latch.start();
```

**etcd Leader 选举**：

```java
// etcd Leader 选举
try (Client client = Client.fromEndpoints("http://127.0.0.1:2379")) {
    Election election = client.getElection("/leader");
    election.campaign("candidate-1");

    // 监听 Leader 变化
    election.observe(event -> {
        System.out.println("新的 Leader: " + event.leader().getName());
    });
}
```

## 四、三种协调服务对比

### 4.1 技术对比

| 维度 | ZooKeeper | etcd | Nacos |
|---|---|---|---|
| **一致性协议** | ZAB | Raft | Raft |
| **客户端支持** | Java（Curator） | 多语言 | Java/Go/Python |
| **数据模型** | ZNode | KV | KV/注册中心 |
| **语言** | Java | Go | Java |
| **监控** | JMX | metrics | metrics |
| **运维难度** | 中 | 低 | 低 |

### 4.2 适用场景对比

| 场景 | ZooKeeper | etcd | Nacos |
|---|---|---|---|
| **分布式锁** | ✓ 成熟 | ✓ 推荐 | ✓ 可用 |
| **配置中心** | ✓ 可用 | ✓ 推荐 | ✓ 首选 |
| **服务发现** | ✓ 可用 | ✓ 可用 | ✓ 首选 |
| **Leader 选举** | ✓ 成熟 | ✓ 推荐 | ✓ 可用 |
| **元数据存储** | ✓ 可用 | ✓ 推荐 | ✗ 不适合 |

### 4.3 选型建议

```
选型决策树：

需要服务注册发现？
    └─→ 选择 Nacos

Cloud Native 环境？
    └─→ 选择 etcd

需要成熟稳定的分布式锁？
    └─→ 选择 ZooKeeper

简单的配置同步？
    └─→ 选择 etcd 或 Nacos
```

## 五、实战最佳实践

### 5.1 ZooKeeper 使用建议

```java
// ZooKeeper 连接池配置
@Service
public class CuratorConfig {

    @Bean
    public CuratorFramework curatorFramework() {
        RetryPolicy retryPolicy = new ExponentialBackoffRetry(1000, 3);

        CuratorFramework client = CuratorFrameworkFactory.builder()
            .connectString("zk-server:2181")
            .retryPolicy(retryPolicy)
            .connectionTimeoutMs(5000)
            .sessionTimeoutMs(30000)
            .namespace("myapp")  // 命名空间隔离
            .build();

        client.start();
        return client;
    }
}
```

## 六、面试高频问题

**Q：ZooKeeper 和 etcd 的区别是什么？**

A：ZooKeeper 使用 ZAB 协议，etcd 使用 Raft 协议。etcd 提供 HTTP/JSON API，多语言支持更好。ZooKeeper 社区更成熟，etcd 是云原生时代的产物。

**Q：分布式协调服务需要 CAP 中选择哪个？**

A：协调服务需要强一致性，通常选择 CP（一致性 + 分区容错性）。ZooKeeper 和 etcd 都是 CP 系统，优先保证一致性。

**Q：如何保证分布式锁的安全性？**

A：1. 使用唯一标识区分锁持有者；2. 使用 Lua 脚本保证释放的原子性；3. 设置合理的超时时间；4. 考虑 RedLock 等多实例方案提高可靠性。

**Q：Nacos 和 ZooKeeper 选哪个？**

A：Nacos 提供更丰富的功能（注册中心 + 配置中心），开箱即用。ZooKeeper 更轻量，专注于协调。简单场景选 Nacos，需要深度定制选 ZooKeeper。

## 总结

分布式协调服务是分布式系统的基础设施：

1. **ZooKeeper**：成熟稳定，分布式锁领域应用最广
2. **etcd**：云原生时代的选择，Raft 协议保证强一致
3. **Nacos**：功能丰富，上手简单，适合国内团队

理解协调服务的原理和使用场景，是分布式系统开发者的必备技能。
