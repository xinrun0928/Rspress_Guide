## 一、Heap Dump 生成方法 🔴

### 1.1 问题拆解

堆转储分析是 OOM 排查的最后一环，也是最关键的环节。面试官追问 MAT 分析技巧，是在测试候选人的深度排查能力。

### 1.2 生成方式对比

| 方式 | 命令 | 优点 | 缺点 |
| --- | --- | --- | --- |
| OOM 时自动 | `-XX:+HeapDumpOnOutOfMemoryError` | 自动、无遗漏 | 只能事后分析 |
| jcmd | `jcmd <pid> GC.heap_dump` | JDK 11+ 推荐 | 需要进程活着 |
| jmap | `jmap -dump:format=b,file=dump.hprof <pid>` | 简单 | JDK 8 常用 |
| jmap -F | `jmap -F -dump:format=b,file=dump.hprof <pid>` | 进程挂起时可用 | **生产禁用** |
| Async-profiler | `profiler.sh -d 30 -f dump.hprof <pid>` | 低开销 | 需额外安装 |

### 1.3 最佳实践

```bash
# 生产环境配置（JDK 8+）
# 在启动脚本中添加：
JAVA_OPTS="$JAVA_OPTS -XX:+HeapDumpOnOutOfMemoryError"
JAVA_OPTS="$JAVA_OPTS -XX:HeapDumpPath=/data/logs/heapdump"
JAVA_OPTS="$JAVA_OPTS -XX:+UseGCLogFileRotation"
JAVA_OPTS="$JAVA_OPTS -XX:NumberOfGCLogFiles=5"
JAVA_OPTS="$JAVA_OPTS -XX:GCLogFileSize=10M"

# JDK 11+ 额外配置
# 限制 dump 文件大小
-XX:HeapDumpFullGC=true
```

## 二、MAT 核心功能 🔴

### 2.1 MAT 四大视图

| 视图 | 用途 | 使用场景 |
| --- | --- | --- |
| **Leak Suspects** | 自动检测泄漏点 | 快速定位 |
| **Histogram** | 按类统计对象数量和大小 | 找异常大的类 |
| **Dominator Tree** | 对象支配关系树 | 找泄漏持有者 |
| **Top Consumers** | 占用内存最多的对象 | 快速找到大头 |

### 2.2 Histogram（直方图）

```bash
# Histogram 操作流程：
# 1. 打开 dump 文件
# 2. 选择 "Histogram" 视图
# 3. 选择分组方式：
#    - Class：按类名分组
#    - Class Loader：按类加载器分组
#    - Superclass：按父类分组
# 4. 按 "Shallow Heap" 或 "Retained Heap" 降序

# 关键指标：
# - Shallow Heap：对象自身大小
# - Retained Heap：对象 + 它引用的所有对象的大小
```

**快速定位技巧**：

```java
// 正常情况
java.lang.String[]         Shallow: 24M  Retained: 1.2G
java.lang.String           Shallow: 200M  Retained: 1.2G

// 异常情况（可能泄漏）
com.xxx.CacheObject        Shallow: 50M   Retained: 800M  ← 对象本身不大，但引用的数据巨大
java.util.HashMap          Shallow: 1M    Retained: 500M  ← 持续增长
```

### 2.3 Dominator Tree（支配树）

Dominator Tree 展示对象的支配关系：**如果从 GC Root 到对象 B 的所有路径都必须经过对象 A，则 A 支配 B**。

```
支配树的优势：
  - 直接找到"谁持有了这片内存"
  - 不需要手动追踪引用链
  - 找到根节点就能确定泄漏点
```

**典型泄漏定位流程**：

```
1. Dominator Tree 顶层
   ↓ com.xxx.Cache: Retained=500M
   ↓   ↓ HashMap: Retained=490M
   ↓   ↓   ↓ HashMap$Entry[]: Retained=480M
   ↓   ↓   ↓   ↓ String: Retained=450M

结论：Cache 持有了一个超大 HashMap，HashMap 里的 String 持续增长

2. 右键 → Path To GC Roots → exclude all weak/soft references
   找到强引用链：
   Cache.cache → HashMap → Entry[] → String[] → String

3. 分析引用链：
   - Cache.cache 是静态字段（GC Root 引用）
   - HashMap 没有清理机制
   - 泄漏点：Cache 永不清理的 HashMap
```

### 2.4 ❌ 错误示范

**候选人原话**："Shallow Heap 大的就是泄漏。"

【面试官心理】
这个候选人没理解 Shallow 和 Retained 的区别。Shallow Heap 大只是说明对象本身大（如 char[]），真正的泄漏可能是某个小对象持有了大量数据的引用。区分"大对象"和"泄漏对象"是 MAT 分析的第一步。

## 三、GC Root 分析 🟡

### 3.1 什么是 GC Root

GC Root 是可达性分析的起点。只有从 GC Root 通过强引用链可达的对象，才不会被 GC 回收。

**六大 GC Root 类型**：

```
1. 虚拟机栈（局部变量表中的 reference）
2. 方法区静态属性引用的对象
3. 方法区常量引用的对象
4. 本地方法栈 JNI 引用的对象
5. JVM 内部引用（ClassLoader, Exception）
6. synchronized 持有的对象
```

### 3.2 Path To GC Roots 的选择

```bash
# 选择引用类型过滤：
# - exclude weak references：排除弱引用（弱引用随时可回收，不是真正的泄漏）
# - exclude soft references：排除软引用（内存不足时才回收）
# - exclude phantom references：排除虚引用
# - exclude all: 排除所有弱软虚（只保留强引用）

# 查找泄漏时，通常选 "exclude all weak/soft references"
# 这样剩下的引用链都是强引用（真正的泄漏点）
```

## 四、OQL 查询语言 🟡

### 4.1 OQL 基本语法

OQL（Object Query Language）允许用 SQL-like 语法查询堆：

```sql
-- 查询所有 String 对象
SELECT * FROM java.lang.String

-- 查询大于 1MB 的 byte[]
SELECT * FROM byte[] WHERE length > 1048576

-- 查询指定包下的所有对象
SELECT * FROM com.xxx.* WHERE objectsize > 1000000

-- 查询包含特定字段的对象
SELECT * FROM java.util.HashMap WHERE table.@length > 10000
```

### 4.2 实用 OQL 查询

```sql
-- 查找可疑的大 HashMap
SELECT toString(k), length(k), length(v)
FROM java.util.HashMap
WHERE length(k) + length(v) > 1000

-- 查找未关闭的连接
SELECT * FROM java.sql.Connection WHERE @openHandles > 0

-- 查找泄漏的 ThreadLocal
SELECT s.@value
FROM java.lang.ThreadLocal$ThreadLocalMap$Entry s
WHERE s.value != null

-- 查找重复的 String（可能导致字符串常量泄漏）
SELECT s.value.toString()
FROM java.lang.String s
GROUP BY s.value
HAVING COUNT(*) > 2
```

### 4.3 动态分析技巧

```java
// 在 OQL 中使用正则表达式查找特定模式的类
SELECT * FROM "com\\.xxx\\.cache\\..*" WHERE objectsize > 1000000

// 查找对象年龄（对象在内存中存在的时间）
SELECT obj, obj.@usedObjectAge
FROM java.lang.Object obj
WHERE obj.@usedObjectAge > 100
```

## 五、典型泄漏模式分析 🟡

### 5.1 模式一：静态集合泄漏

```sql
-- 查询所有静态 Map
SELECT m FROM java.util.Map m
WHERE m.class.getDeclaredField('holder') != null
  AND m.class.declaredFields.any(f => f.synthetic == false && f.type == java.util.Map)
```

**特征**：静态 Map 持续增长，永不清理。

**解决方案**：使用 `WeakHashMap`、设置 TTL、或定期清理。

### 5.2 模式二：监听器未注销

```java
// 代码中的泄漏模式
class EventSource {
    private List<EventListener> listeners = new ArrayList<>();

    public void addListener(EventListener l) {
        listeners.add(l); // 只添加，不移除
    }
    // 缺少 removeListener() 方法
}
```

**MAT 中的特征**：ArrayList 持有大量 EventListener 对象，但这些 Listener 的原始引用已经不存在。

### 5.3 模式三：ThreadLocal 泄漏

```sql
-- 查找 ThreadLocalMap 中的 entry
SELECT t.@value
FROM java.lang.ThreadLocal$ThreadLocalMap$Entry e
WHERE e.@value != null
```

**特征**：ThreadLocalMap 中有大量 Entry，但对应的 ThreadLocal 对象已经为 null。

## 六、生产分析实战 🟢

### 6.1 分析流程

```
1. 确认 dump 时间点（OOM 前后）
2. 打开 Leak Suspects 自动报告
3. 分析 Dominator Tree 顶层
4. 对比多次 dump（如果有多次）
   - 增长最快的对象 = 最可疑
5. 找 GC Root 引用链
6. 回到代码定位泄漏点
```

### 6.2 命令行快速分析

```bash
# 使用 jhat 内置 HTTP 服务器分析 dump（JDK 11 移除）
jhat -J-mx2g heap.hprof

# 使用 MAT 命令行（JDK 11+）
mat-parse heap.hprof --analysis-type=leaksuspects

# 使用 haasome（第三方）
java -jar haasome.jar heap.hprof
```

:::tip 💡
**多 dump 对比法**：如果有多次 dump（OOM 前、OOM 时、Full GC 后），对比同一类对象的大小变化。如果某个类在所有 dump 中持续增长，这个类就是泄漏点。
:::
