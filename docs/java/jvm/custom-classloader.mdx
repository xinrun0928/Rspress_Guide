## 一、自定义 ClassLoader 的基本结构 🔴

### 1.1 问题拆解

自定义 ClassLoader 不是背一个模板就完事的。面试官追问"怎么做"，深层在测试候选人的工程实现能力——加密方案、缓存策略、异常处理、内存管理，每个环节都有坑。

### 1.2 标准模板

```java
public class CustomClassLoader extends ClassLoader {

    private final String classPath; // 自定义类加载路径

    public CustomClassLoader(String classPath) {
        this.classPath = classPath;
    }

    @Override
    protected Class<?> findClass(String name) throws ClassNotFoundException {
        // 1. 将类的全限定名转换为文件路径
        String filePath = classPath + name.replace('.', '/') + ".class";

        // 2. 读取 class 文件字节流
        byte[] classData;
        try {
            classData = Files.readAllBytes(Paths.get(filePath));
        } catch (IOException e) {
            throw new ClassNotFoundException("Class not found: " + name, e);
        }

        // 3. 调用 defineClass 将字节流转换为 Class 对象
        return defineClass(name, classData, 0, classData.length);
    }
}
```

### 1.3 ❌ 错误示范

**候选人原话**："重写 loadClass 就可以自定义加载逻辑了。"

【面试官心理】
这个候选人没有理解双亲委派的工程价值。重写 `loadClass` 会破坏整个委派链，导致无法正确复用父类加载器的已加载类。正确的做法是重写 `findClass`，让父类加载器仍然生效。

## 二、加密类加载器 🟡

### 2.1 加密方案

```java
// 加密工具：XOR 简单加密
public class ClassEncryptor {
    private static final byte KEY = (byte) 0x5A;

    public static void encrypt(File input, File output) throws IOException {
        byte[] data = Files.readAllBytes(input.toPath());
        byte[] encrypted = new byte[data.length];
        for (int i = 0; i < data.length; i++) {
            encrypted[i] = (byte) (data[i] ^ KEY);
        }
        Files.write(output.toPath(), encrypted);
    }
}
```

### 2.2 解密 ClassLoader

```java
public class DecryptClassLoader extends ClassLoader {

    private static final byte KEY = (byte) 0x5A;
    private final String classPath;

    public DecryptClassLoader(String classPath) {
        this.classPath = classPath;
    }

    @Override
    protected Class<?> findClass(String name) throws ClassNotFoundException {
        String filePath = classPath + name.replace('.', '/') + ".class";

        try {
            byte[] encrypted = Files.readAllBytes(Paths.get(filePath));
            byte[] decrypted = new byte[encrypted.length];

            // XOR 解密
            for (int i = 0; i < encrypted.length; i++) {
                decrypted[i] = (byte) (encrypted[i] ^ KEY);
            }

            // 使用原始字节数据定义类
            return defineClass(name, decrypted, 0, decrypted.length);

        } catch (IOException e) {
            throw new ClassNotFoundException(name, e);
        }
    }
}
```

### 2.3 ❌ 错误示范

**候选人原话**："密钥直接写在代码里，这样别人反编译也看不出问题。"

【面试官心理】
密钥硬编码在 ClassLoader 中确实不是最佳方案，因为可以被反编译。更好的方案是用 Native 方法隐藏密钥（放到 `.so`/`.dll` 中），或者用远程服务器下发密钥。但更重要的是：这个候选人对安全问题有意识，而不是盲目自信。

## 三、热部署 ClassLoader 🟡

### 3.1 热部署的核心思路

```java
public class HotDeployClassLoader extends URLClassLoader {

    private final Map<String, Long> classFileVersion = new HashMap<>();
    private final String classPath;

    public HotDeployClassLoader(String classPath) {
        super(new URL[]{new File(classPath).toURI().toURL()});
        this.classPath = classPath;
    }

    @Override
    protected Class<?> loadClass(String name, boolean resolve)
            throws ClassNotFoundException {
        // 检查是否有更新
        if (hasUpdated(name)) {
            // 强制重新加载：先从自己的缓存清除，再重新加载
            Class<?> clazz = findLoadedClass(name);
            if (clazz != null) {
                // 从类加载器的命名空间中移除旧类（实现上通常较复杂）
            }
            return findClass(name);
        }

        return super.loadClass(name, resolve);
    }

    private boolean hasUpdated(String name) {
        String filePath = classPath + "/" + name.replace('.', '/') + ".class";
        File file = new File(filePath);
        long newLastModified = file.lastModified();
        Long oldVersion = classFileVersion.get(name);
        return oldVersion == null || newLastModified > oldVersion;
    }
}
```

### 3.2 Tomcat 的实现原理

Tomcat 的热部署通过 `WebappLoader` 实现：
1. 停止时，销毁旧的 `WebappClassLoader`
2. 销毁过程：将所有该 ClassLoader 加载的类从 JVM 永久移除（通过类加载器引用计数归零）
3. 重新创建新的 `WebappClassLoader`
4. 重新加载所有类

## 四、生产避坑 🟡

### 4.1 内存泄漏

```java
// 错误做法：ClassLoader 持有旧类引用，无法卸载
public class BadClassLoader extends ClassLoader {
    private List<Object> holders = new ArrayList<>();

    public void hold(Object obj) {
        holders.add(obj); // 持有对象引用，导致类无法卸载
    }
}
```

### 4.2 类版本冲突

```java
// 错误场景：两个 ClassLoader 加载同一个类的不同版本
ClassLoader cl1 = new CustomClassLoader("lib1");
ClassLoader cl2 = new CustomClassLoader("lib2");

Class<?> A1 = cl1.loadClass("com.example.A"); // lib1 的 A
Class<?> A2 = cl2.loadClass("com.example.A"); // lib2 的 A

// A1 和 A2 的 Class 对象不相等！
A1 != A2; // true

// 如果 A1 被传入需要 A2 的方法，抛出 ClassCastException
Object obj = A1.newInstance();
// (A2) obj → ClassCastException!
```

:::warning ⚠️
**ClassCastException 是自定义 ClassLoader 最常见的坑**。同一个类名但不同 ClassLoader 加载的类，在 JVM 中被视为完全不同的类型。在设计模块化系统时，必须确保类型传递的一致性。
:::

### 5.1 追问：自定义 ClassLoader 的使用场景

| 场景 | 方案 | 核心要点 |

### 5.2 追问：如何检测 ClassLoader 冲突

```java
// 方法1：检查类加载来源
public static String getClassLoaderInfo(Class<?> clazz) {
    ClassLoader cl = clazz.getClassLoader();
    return cl == null ? "Bootstrap" : cl.getClass().getName();
}

// 方法2：检查类对象相等性
public static boolean isSameClass(Class<?> a, Class<?> b) {
    return a.equals(b); // 比较的是 ClassLoader + 类名
}
```
