# CPU 高占用分析

CPU 高占用是生产环境中常见的问题之一。当服务器 CPU 使用率持续居高不下时，会导致应用响应变慢、服务降级甚至崩溃。本篇文章将详细介绍如何诊断和解决 CPU 高占用问题。

## CPU 高占用的常见原因

<Badge>Common Causes</Badge>

### 原因一：死循环或无限递归

```java
public class InfiniteLoop {
    public static void main(String[] args) {
        while (true) {
            // 空循环消耗 CPU
        }
    }
}
```

死循环是导致 CPU 100% 最常见的原因。线程在循环中不断执行，没有任何等待或阻塞。

### 原因二：频繁 GC

<Badge>GC Overhead</Badge>

如果对象分配速度过快，会导致频繁的 GC，GC 线程占用大量 CPU。

```java
public class FrequentAllocation {
    public static void main(String[] args) {
        // 每秒分配大量对象
        while (true) {
            List<byte[]> list = new ArrayList<>();
            for (int i = 0; i < 1000; i++) {
                list.add(new byte[1024]);  // 每秒分配大量内存
            }
            try {
                Thread.sleep(1000);
            } catch (InterruptedException e) {}
        }
    }
}
```

### 原因三：正则表达式过度使用

<Badge>Regex CPU</Badge>

复杂的正则表达式可能导致回溯，造成 CPU 消耗。

```java
public class RegexCPU {
    // 错误：使用贪婪匹配的复杂正则
    public static final String PATTERN = "(a+)+b";
    
    public static void main(String[] args) {
        String input = "aaaaaaaaaaaaaaaaX";  // 触发回溯
        while (true) {
            input.matches(PATTERN);  // CPU 飙升
        }
    }
}
```

### 原因四：加密解密或压缩

<Badge>CPU Intensive</Badge>

CPU 密集型的业务操作也可能导致高 CPU 占用。

## CPU 高占用排查步骤

<Badge>Troubleshooting Steps</Badge>

### 第一步：定位高 CPU 线程

```bash
# 使用 top 命令查看 CPU 占用最高的进程
top -c

# 查看指定进程的线程 CPU 占用
top -Hp <pid>

# 或者使用 ps 命令
ps -mp <pid> -o THREAD,tid,time,%cpu
```

### 第二步：获取线程堆栈

```bash
# 获取线程堆栈
jstack <pid> > thread_dump.txt

# 分析高 CPU 线程的堆栈
# 线程 ID 需要转换为 16 进制
printf "%x\n" <tid>

# 查看指定线程的堆栈
grep -A 50 "nid=0x<tid>" thread_dump.txt
```

### 第三步：分析代码

根据堆栈信息定位问题代码。

```bash
# 示例：定位到某个方法的死循环
"pool-1-thread-1" #42 prio=5 os_prio=0 tid=0x00007f8a8c01a800
    java.lang.Thread.State: RUNNABLE
        at com.example.InfiniteLoop.loop(InfiniteLoop.java:10)
        at com.example.InfiniteLoop.run(InfiniteLoop.java:15)
```

## 使用 Arthas 排查 CPU 问题

<Badge>Arthas Profiling</Badge>

Arthas 提供了强大的性能分析功能，可以快速定位 CPU 热点。

```bash
# 使用 Arthas profiler 生成火焰图
profiler start

# 运行一段时间后，获取采样结果
profiler stop --format html

# 或者使用 trace 命令追踪耗时方法
trace com.example.*.* '#cost > 100'
```

火焰图可以直观显示 CPU 时间在各方法中的分布，火焰尖尖的位置就是消耗 CPU 最多的方法。

## 实战案例：正则表达式回溯导致 CPU 飙升

<Badge>Case Study</Badge>

### 问题现象

某接口在处理特定输入时 CPU 使用率突然飙升到 100%。

### 排查过程

```bash
# 1. 查看 CPU 占用最高的线程
top -Hp <pid>

# 2. 获取线程堆栈
jstack <pid> > dump.txt

# 3. 分析堆栈，发现是 Pattern.matches() 消耗 CPU

# 4. 检查代码
grep -r "matches" src/
# 发现使用了复杂的正则表达式
```

### 解决方案

```java
// 问题代码
public boolean validate(String input) {
    return input.matches("(a+)+b");  // 复杂正则
}

// 优化方案一：简化正则
public boolean validate(String input) {
    return input.matches("a+b");  // 简化
}

// 优化方案二：使用 Matcher.lookingAt() 或限时匹配
public boolean validate(String input, long timeoutMs) {
    Pattern pattern = Pattern.compile("(a+)+b");
    Matcher matcher = pattern.matcher(input);
    long deadline = System.currentTimeMillis() + timeoutMs;
    while (matcher.find()) {
        if (System.currentTimeMillis() > deadline) {
            return false;  // 超时放弃
        }
    }
    return false;
}
```

## 线程状态与 CPU 关系

<Badge>Thread State</Badge>

| 线程状态 | CPU 占用 | 可能原因 |
|----------|----------|----------|
| RUNNABLE | 高 | 死循环、计算密集、正则回溯 |
| RUNNABLE | 低 | 等待 IO、等待锁 |
| BLOCKED | 低 | 线程阻塞，等待锁释放 |
| WAITING | 低 | 等待条件，线程池问题 |
| TIMED_WAITING | 低 | Thread.sleep、sleep 等待 |

## 面试高频考点

**问题一：如何排查 CPU 高占用？** 首先使用 top 或 ps 命令定位高 CPU 的进程或线程；然后使用 jstack 获取线程堆栈；最后分析堆栈定位问题代码。

**问题二：死循环导致的 CPU 高占用有什么特征？** 线程状态为 RUNNABLE，堆栈中显示同一个方法重复调用。

**问题三：正则表达式可能导致 CPU 问题吗？** 是的。复杂的正则表达式可能导致回溯问题，在极端情况下导致 CPU 100%。

**问题四：频繁 GC 会导致 CPU 高占用吗？** 会。频繁 GC 会导致 GC 线程占用大量 CPU，同时也会导致应用线程暂停。

**问题五：如何生成火焰图？** 使用 Arthas 的 profiler 命令，或使用 async-profiler 工具。

**问题六：火焰图怎么看？** 火焰图的每一层代表一个方法调用，宽度代表 CPU 占用时间，越宽的方法消耗 CPU 越多。

CPU 高占用就像汽车引擎过热——如果不及时降温，最终会导致整个系统崩溃。那么，在排查 CPU 问题时，你会从哪个方向开始？
