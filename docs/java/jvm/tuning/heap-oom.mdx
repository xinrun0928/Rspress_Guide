# 堆内存溢出排查

堆内存溢出（java.lang.OutOfMemoryError: Java heap space）是最常见的 OOM 类型。当应用程序创建的对象数量超过堆内存能够容纳的上限时，JVM 就会抛出这个错误。掌握堆内存溢出的排查方法，是每个 Java 开发者的必备技能。

## 堆溢出的典型场景

<Badge>Heap OOM Scenarios</Badge>

堆内存溢出通常发生在以下几种场景中：

**内存泄漏**：程序中存在对象引用未及时释放的问题，导致内存持续增长。这是生产环境中最常见的堆溢出原因。

```java
public class MemoryLeakScenario {
    // 静态集合持续增长
    private static final List<byte[]> CACHE = new ArrayList<>();
    
    public void process() {
        // 每秒添加 1MB 数据
        CACHE.add(new byte[1024 * 1024]);
        System.out.println("缓存大小: " + CACHE.size() + " MB");
    }
}
```

**一次性加载过多数据**：将大文件或数据库结果集全部加载到内存。

```java
public class DataLoadingMistake {
    // 错误：将所有数据一次性加载
    public List<User> getAllUsers() {
        String sql = "SELECT * FROM users";
        return jdbcTemplate.query(sql, (rs, rowNum) -> {
            User user = new User();
            user.setId(rs.getLong("id"));
            user.setName(rs.getString("name"));
            // 千万级数据 = 内存溢出
            return user;
        });
    }
    
    // 正确：分页查询
    public void processUsers() {
        int page = 0;
        int pageSize = 1000;
        List<User> batch;
        do {
            batch = getUsersByPage(page++, pageSize);
            processBatch(batch);  // 处理完释放
        } while (!batch.isEmpty());
    }
}
```

## 堆溢出排查步骤

<Badge>Heap OOM Troubleshooting</Badge>

### 第一步：启用堆转储

排查堆溢出的第一步是生成堆转储文件，以便事后分析。

```bash
# 方式一：OOM 时自动生成堆转储（推荐）
java -Xms512m -Xmx512m \
     -XX:+HeapDumpOnOutOfMemoryError \
     -XX:HeapDumpPath=/var/log/java/ \
     MyApplication

# 方式二：通过 JMX 手动触发
jcmd <pid> GC.heap_dump /var/log/heapdump.hprof

# 方式三：使用 jmap（不推荐在生产环境使用）
jmap -dump:format=b,file=/var/log/heapdump.hprof <pid>
```

### 第二步：分析堆转储文件

堆转储文件可以通过多种工具分析：

**Eclipse MAT（Memory Analyzer Tool）** 是最常用的堆分析工具。

**VisualVM** 是 JDK 自带的可视化分析工具。

**jhat** 是 JDK 自带的命令行分析工具。

```bash
# 使用 jhat 分析堆转储
jhat -J-mx2g /var/log/heapdump.hprof
# 然后通过浏览器访问 http://localhost:7000
```

### 第三步：定位内存泄漏点

使用 MAT 分析时，关注以下视图：

**Leak Suspects Report**：自动分析可能的内存泄漏点。

**Histogram**：按类统计对象数量和内存占用。

**Dominator Tree**：显示对象引用树，定位持有大量内存的对象。

```java
// 示例：定位内存泄漏
public class LeakLocator {
    public static void main(String[] args) throws Exception {
        // 生成堆转储后，在 MAT 中查看：
        
        // 1. 查看 Histogram，按 Shallow Heap 排序
        //    找出占用内存最大的类
        
        // 2. 查看 Dominator Tree
        //    找出内存占用的根源
        
        // 3. 查看 OQL 查询
        //    SELECT * FROM java.util.HashMap WHERE size > 1000
    }
}
```

## 实战案例：缓存未清理导致的 OOM

<Badge>Case Study</Badge>

```java
public class CacheOOMCase {
    // 场景：用户会话缓存未设置上限
    private static Map<String, UserSession> sessionCache = new HashMap<>();
    
    public static void main(String[] args) {
        // 模拟不断创建会话
        int count = 0;
        try {
            while (true) {
                String sessionId = UUID.randomUUID().toString();
                UserSession session = new UserSession(sessionId);
                sessionCache.put(sessionId, session);
                count++;
            }
        } catch (OutOfMemoryError e) {
            System.out.println("OOM 发生前创建了 " + count + " 个会话");
            System.out.println("缓存大小: " + sessionCache.size());
            
            // 堆转储分析会显示：
            // - HashMap 持有大量 Entry 对象
            // - 每个 Entry 包含 key（String）和 value（UserSession）
            // - 泄漏根源是 sessionCache 变量
        }
    }
    
    static class UserSession {
        private String sessionId;
        private byte[] data = new byte[1024 * 100]; // 每个会话 100KB
        
        public UserSession(String sessionId) {
            this.sessionId = sessionId;
        }
    }
}
```

**解决方案**：

```java
// 方案一：使用带清理机制的缓存
private static Map<String, UserSession> sessionCache = 
    new WeakHashMap<>();  // 使用弱引用，自动清理

// 方案二：使用 LRU 缓存
private static Map<String, UserSession> sessionCache = 
    new LinkedHashMap<>(1000, 0.75f, true) {
        @Override
        protected boolean removeEldestEntry(Map.Entry eldest) {
            return size() > 10000;  // 超过 10000 条自动清理
        }
    };

// 方案三：使用成熟缓存框架
private static Cache<String, UserSession> sessionCache = 
    Caffeine.newBuilder()
        .maximumSize(10000)
        .expireAfterAccess(Duration.ofMinutes(30))
        .build();
```

## 堆配置优化建议

<Badge>Heap Configuration</Badge>

```bash
# 合理的堆配置
java -Xms2g -Xmx2g \        # 最小和最大设为相同，避免动态调整
    -XX:+HeapDumpOnOutOfMemoryError \
    -XX:HeapDumpPath=/var/log/ \
    -XX:+UseG1GC \
    -XX:MaxGCPauseMillis=200 \
    MyApplication

# 监控堆使用
jstat -gcutil <pid> 1000
```

## 面试高频考点

**问题一：如何排查堆内存溢出？** 首先启用 -XX:+HeapDumpOnOutOfMemoryError 生成堆转储；然后使用 MAT、VisualVM 或 jhat 分析堆转储文件；通过 Histogram 和 Dominator Tree 定位内存占用最大的对象。

**问题二：什么是内存泄漏？如何解决？** 内存泄漏是对象被错误持有导致无法回收。解决方案包括：使用 WeakHashMap、及时清理集合、使用缓存框架配置清理策略、修复对象持有链。

**问题三：如何预防堆内存溢出？** 合理配置堆大小、避免一次性加载过多数据、使用分页/流式处理、监控内存使用、设置告警。

堆内存溢出就像家里的水管漏水——如果不及时修补，水会一直流直到溢出。那么，在你的代码中，有哪些地方可能存在类似的"漏水"隐患？
