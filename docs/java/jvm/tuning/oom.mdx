# OutOfMemoryError 各类型与原因分析

OutOfMemoryError（OOM）是 Java 程序运行中最令人头疼的问题之一。当 JVM 无法分配足够的内存时，就会抛出这个错误。OOM 不仅仅是内存不足的简单表现，它有多种不同的类型，每种类型都指向不同的内存区域和问题根源。理解这些 OOM 类型的区别，是排查和解决问题的第一步。

## OOM 的本质

<Badge>OOM Mechanism</Badge>

OOM 的本质是 JVM 无法满足内存分配请求。这可能发生在 JVM 达到自身内存限制时（如堆大小、元空间大小），也可能发生在操作系统无法提供足够物理内存时。

OOM 并不总是意味着真的没有内存了。有时候，是因为内存碎片化导致无法找到足够大的连续空间；有时候，是因为分配请求的对象太大，无法在任何可用空间中分配。

OOM 的消息格式通常是：`java.lang.OutOfMemoryError: [具体类型]`。具体类型部分描述了问题的性质，是定位问题的关键。

## 堆内存溢出：Java heap space

<Badge>Heap OOM</Badge>

这是最常见的 OOM 类型，表明堆内存无法为新对象分配空间。

**触发原因**：
- 内存泄漏：程序持有对象引用但不再使用，导致对象无法被回收
- 对象分配过快：程序创建对象的速度超过 GC 回收的速度
- 堆内存设置太小：`-Xmx` 设置的最大堆大小无法满足应用需求

**典型场景**：

```java
// 无限增长的集合
List<byte[]> list = new ArrayList<>();
while (true) {
    list.add(new byte[1024 * 1024]);  // 不断添加 1MB 数组
}
```

**排查方向**：使用 MAT 分析堆转储，找出占用内存最多的对象，分析引用链判断是否是内存泄漏。

## 虚拟机栈溢出：Unable to create new native thread

<Badge>Native Thread OOM</Badge>

当无法创建新线程时抛出此错误，表明系统资源（而非堆内存）耗尽。

**触发原因**：
- 创建了过多线程：每个线程都占用独立的栈空间（`-Xss`），大量线程会耗尽内存
- 栈空间设置过大：`-Xss` 设置的值过大，减少了可创建的线程数

**典型场景**：

```java
// 创建大量线程
ExecutorService executor = Executors.newCachedThreadPool();
for (int i = 0; i < 10000; i++) {
    executor.submit(() -> {
        try {
            Thread.sleep(1000000);
        } catch (InterruptedException e) {}
    });
}
```

**排查方向**：检查线程创建逻辑，优化线程池配置，减少不必要的线程创建。

## GC 开销超限：GC overhead limit exceeded

<Badge>GC Overhead</Badge>

当 GC 花费的时间超过限制（默认 98%），且回收的内存少于 2% 时抛出。

**触发原因**：
- 堆内存接近耗尽：GC 几乎无法回收任何空间
- 内存分配速率过高：对象创建速度太快

**排查方向**：这通常是堆内存不足或内存泄漏的信号。增大堆内存或排查内存泄漏。

## 直接内存溢出：Direct buffer memory

<Badge>Direct Memory OOM</Badge>

当 NIO 使用直接内存（堆外内存）超出限制时抛出。

**触发原因**：
- 大量使用 NIO 直接缓冲区：`ByteBuffer.allocateDirect()` 分配大量直接内存
- `-XX:MaxDirectMemorySize` 设置过小

**排查方向**：检查 NIO 相关的代码和缓冲区使用情况，调整直接内存大小限制。

## 元空间溢出：Metaspace

<Badge>Metaspace OOM</Badge>

JDK 8 及之后，当元空间（Metaspace）无法分配类元数据时抛出。

**触发原因**：
- 大量动态类生成：反射、代理、字节码增强等生成大量类
- 类加载器泄漏：类加载器无法被回收

**典型场景**：

```java
// 动态生成大量类
for (int i = 0; i < 100000; i++) {
    ClassWriter cw = new ClassWriter(ClassWriter.COMPUTE_FRAMES);
    // 动态生成类...
}
```

**排查方向**：使用 `-XX:MaxMetaspaceSize` 设置上限（默认无限制），使用 jstat 监控元空间使用。

## 数组大小超限：Requested array size exceeds VM limit

<Badge>Array Size OOM</Badge>

当尝试分配的数组大小超过 JVM 限制时抛出。

**触发原因**：
- 尝试分配过大的数组
- 整数溢出导致数组大小计算错误

**典型场景**：

```java
// 尝试分配过大的数组
byte[] arr = new byte[Integer.MAX_VALUE];
```

## 本地内存映射溢出：Map failed

<Badge>Memory Map OOM</Badge>

当无法创建内存映射或映射大小超过限制时抛出。

**触发原因**：
- 文件过大：尝试 mmap 过大的文件
- 地址空间耗尽：32 位系统地址空间有限

## 栈溢出：StackOverflowError

<Badge>StackOverflow</Badge>

当线程请求的栈深度超过 JVM 允许的最大深度时抛出。

**触发原因**：
- 无限递归：方法无限调用自身，没有退出条件

**典型场景**：

```java
// 无限递归
public void recursive() {
    recursive();  // StackOverflowError
}
```

## OOM 的排查流程

<Badge>Troubleshooting</Badge>

遇到 OOM 时，应该按照以下流程进行排查。

第一步，保存现场。记录 OOM 发生的环境、时间、应用状态等信息。

第二步，检查 OOM 类型。根据错误消息判断是哪类 OOM。

第三步，生成堆转储。如果可能，在 OOM 之前或之后生成堆转储文件。

第四步，分析内存使用。使用 MAT 等工具分析堆转储，找出占用内存最多的对象。

第五步，追踪引用链。从占用内存最多的对象出发，追踪 GC Root 引用链，找出内存泄漏的原因。

第六步，修复问题。根据分析结果修复代码或调整 JVM 参数。

## 常见 OOM 场景总结

| OOM 类型 | 内存区域 | 常见原因 | 排查工具 |
|----------|----------|----------|----------|
| Java heap space | 堆 | 内存泄漏、分配过快 | MAT、jmap |
| Unable to create new native thread | 线程栈 | 线程过多 | jstack |
| GC overhead limit exceeded | 堆 | 堆不足 | jstat、jmap |
| Direct buffer memory | 直接内存 | NIO 过多 | jcmd |
| Metaspace | 元空间 | 动态类生成 | jstat |
| StackOverflowError | 虚拟机栈 | 递归过深 | jstack |

## 面试高频考点

**问题一：OOM 有哪些类型？** 堆内存溢出、元空间溢出、虚拟机栈溢出、直接内存溢出、GC 开销超限等。

**问题二：如何判断是否是内存泄漏导致的 OOM？** 观察 GC 日志，如果每次 GC 后内存持续增长无法降低，说明可能存在内存泄漏。使用 MAT 分析堆转储找出泄漏点。

**问题三：StackOverflowError 和 OutOfMemoryError: Unable to create new native thread 的区别是什么？** StackOverflowError 是单个线程栈深度超限；Unable to create new native thread 是整个进程的线程数或内存超限。

**问题四：Metaspace OOM 的原因是什么？** 大量动态类生成、类加载器泄漏、频繁的反射和代理操作。

**问题五：GC overhead limit exceeded 是什么含义？** JVM 认为 GC 效率太低，花了太多时间但回收不了多少内存，通常是堆内存不足的信号。

OOM 是 JVM 给我们发出的警告信号，理解每种 OOM 的含义，才能对症下药。那么，当遇到 OOM 时，你首先会关注哪个信息？
