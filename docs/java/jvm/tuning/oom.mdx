# OutOfMemoryError 各类型与原因分析

OutOfMemoryError（OOM）是 Java 程序运行中最令人头疼的问题之一。当 JVM 无法分配足够的内存时，就会抛出这个错误。OOM 不仅仅是内存不足的简单表现，它有多种不同的类型，每种类型都指向不同的内存区域和问题根源。理解这些 OOM 类型的区别，是排查和解决问题的第一步。

## OOM 的本质

|OOM 机制|

OOM 的本质是 JVM 无法满足内存分配请求。这可能发生在 JVM 达到自身内存限制时（如堆大小、元空间大小），也可能发生在操作系统无法提供足够物理内存时。

OOM 并不总是意味着真的没有内存了。有时候，是因为内存碎片化导致无法找到足够大的连续空间；有时候，是因为分配请求的对象太大，无法在任何可用空间中分配。

OOM 的消息格式通常是：`java.lang.OutOfMemoryError: [具体类型]`。具体类型部分描述了问题的性质，是定位问题的关键。

## 堆内存溢出：Java heap space

|堆溢出|

这是最常见的 OOM 类型，表明堆内存无法为新对象分配空间。

**触发原因**：

- 内存泄漏：程序持有对象引用但不再使用，导致对象无法被回收
- 对象分配过快：程序创建对象的速度超过 GC 回收的速度
- 堆内存设置太小：`-Xmx` 设置的最大堆大小无法满足应用需求

**典型场景**：

```java
// 无限增长的集合
List<byte[]> list = new ArrayList<>();
while (true) {
    list.add(new byte[1024 * 1024]);  // 不断添加 1MB 数组
}
```

**排查方向**：使用 MAT 分析堆转储，找出占用内存最多的对象，分析引用链判断是否是内存泄漏。

## 虚拟机栈溢出：Unable to create new native thread

|本地线程|

当无法创建新线程时抛出此错误，表明系统资源（而非堆内存）耗尽。

**触发原因**：

- 创建了过多线程：每个线程都占用独立的栈空间（`-Xss`），大量线程会耗尽内存
- 栈空间设置过大：`-Xss` 设置的值过大，减少了可创建的线程数

**典型场景**：

```java
// 创建大量线程
ExecutorService executor = Executors.newCachedThreadPool();
while (true) {
    executor.submit(() -> {
        try {
            Thread.sleep(Long.MAX_VALUE);
        } catch (InterruptedException e) {
            e.printStackTrace();
        }
    });
}
```

**排查方向**：检查线程创建逻辑，使用线程池限制线程数量，考虑减小 `-Xss` 参数。

## 虚拟机栈溢出：StackOverflowError

|栈溢出|

当线程请求的栈深度超过 JVM 允许的范围时抛出此错误。

**触发原因**：

- 递归调用没有正确终止：递归深度过大
- 方法调用链过深：大量方法嵌套调用
- 局部变量占用过多栈空间：大数组或大对象作为局部变量

**典型场景**：

```java
// 无限递归
public class RecursiveDemo {
    public static int count = 0;
    
    public static void recursive() {
        count++;
        recursive();  // 栈深度无限增加
    }
    
    public static void main(String[] args) {
        try {
            recursive();
        } catch (StackOverflowError e) {
            System.out.println("递归次数: " + count);
            e.printStackTrace();
        }
    }
}
```

**排查方向**：检查递归调用是否有终止条件，使用 `-Xss` 参数调整栈大小（通常不推荐，调大只是延缓问题）。

## 元空间溢出：Metaspace

|元空间|

JDK 8 之前使用永久代（PermGen），JDK 8 之后使用元空间（Metaspace）。当元空间无法容纳类的元数据时抛出此错误。

**触发原因**：

- 动态类生成过多：如使用 CGLIB、ASM 动态生成大量类
- 大量使用反射：反射会生成代理类和缓存的 Class 对象
- 第三方库问题：如某些 ORM 框架会动态生成类

**典型场景**：

```java
// 使用 CGLIB 动态生成大量类
Enhancer enhancer = new Enhancer();
enhancer.setSuperclass(MyClass.class);
while (true) {
    enhancer.create();  // 每次创建都生成新的代理类
}
```

**排查方向**：检查动态类生成的代码，使用 `-XX:MaxMetaspaceSize` 限制元空间大小（不推荐，应该找到内存泄漏的根源）。

## 直接内存溢出：Direct buffer memory

|直接内存|

当使用 NIO 的直接内存（DirectByteBuffer）时，如果分配的直接内存超过限制，会抛出此错误。

**触发原因**：

- 大量使用直接内存：如使用 Netty、Zookeeper 等高性能网络框架
- 直接内存设置过小：`-XX:MaxDirectMemorySize` 设置的值过小
- 直接内存泄漏：没有正确释放直接内存

**典型场景**：

```java
// 不断分配直接内存
List<ByteBuffer> buffers = new ArrayList<>();
while (true) {
    ByteBuffer buffer = ByteBuffer.allocateDirect(1024 * 1024);  // 每次分配 1MB 直接内存
    buffers.add(buffer);
}
```

**排查方向**：使用 `-XX:MaxDirectMemorySize` 参数调整直接内存大小，检查代码是否正确释放直接内存。

## GC 开销限制：GC overhead limit exceeded

|GC 开销|

当 GC 花费的时间超过限制（默认是 98%），并且回收的内存少于 2% 时抛出此错误。这表明程序在不断进行 GC 但几乎回收不到任何空间。

**触发原因**：

- 内存泄漏：持续创建对象但无法回收
- 堆内存太小：可用空间不足
- 对象生命周期过长：大部分对象都是长期存活

```bash
# 关闭这个检查（不推荐）
-XX:-UseGCOverheadLimit
```

## OOM 的排查工具

常用的 OOM 排查工具包括：

**JVM 参数**：

```bash
# 发生 OOM 时生成堆转储
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/path/to/dump

# 示例
java -Xms256m -Xmx256m \
     -XX:+HeapDumpOnOutOfMemoryError \
     -XX:HeapDumpPath=/var/log/heapdump.hprof \
     MyApplication
```

**MAT（Memory Analyzer Tool）**：专业的堆内存分析工具，可以找出内存泄漏的根源。

**Arthas**：阿里开源的 Java 诊断工具，支持动态查看内存使用情况。

```bash
# 使用 Arthas 查看内存
dashboard        # 查看内存区域使用情况
heapdump        # 生成堆转储文件
```

## OOM 预防策略

预防 OOM 的关键在于：

1. 合理设置堆大小和元空间大小
2. 避免内存泄漏：使用弱引用/软引用管理缓存，及时清理资源
3. 使用线程池限制线程数量
4. 监控内存使用：使用 JMX、Prometheus 等工具监控 JVM 内存
5. 定期分析 GC 日志和堆转储

```bash
# 推荐的 JVM 内存参数配置
java -Xms256m -Xmx256m \                # 初始堆和最大堆设置为相同，避免动态调整
   -XX:MetaspaceSize=128m \             # 元空间初始大小
   -XX:MaxMetaspaceSize=256m \          # 元空间最大大小
   -XX:MaxDirectMemorySize=256m \       # 直接内存最大大小
   -XX:+HeapDumpOnOutOfMemoryError \   # OOM 时生成堆转储
   -XX:HeapDumpPath=/var/log/ \        # 堆转储路径
   MyApplication
```
