# Metaspace 溢出排查

Metaspace 是 JDK 8 引入的元空间，用于存储类的元数据信息。与堆内存不同，Metaspace 使用本地内存（Native Memory），不再占用堆空间。但这并不意味着 Metaspace 可以无限使用——当 Metaspace 空间不足时，同样会抛出 OutOfMemoryError。理解 Metaspace 溢出对于排查类加载相关问题至关重要。

## Metaspace 的前世今生

<Badge>Metaspace History</Badge>

在 JDK 7 及之前，类的元信息存储在永久代（PermGen）中。永久代是堆内存的一部分，有固定的大小限制。这带来了很多问题：永久代空间难以调整、容易发生 OOM、影响堆内存使用等。

JDK 8 彻底废弃了永久代，改用 Metaspace。Metaspace 使用本地内存，默认只受可用本地内存的限制，大大降低了 OOM 的风险。但 JDK 也提供了参数来限制 Metaspace 的大小，防止无限增长。

## Metaspace 溢出的典型场景

<Badge>Metaspace OOM Scenarios</Badge>

### 场景一：动态生成大量类

使用字节码框架动态生成类是导致 Metaspace 溢出的主要原因。

```java
import javassist.*;
import java.lang.reflect.Method;

public class MetaspaceOOMCase {
    public static void main(String[] args) throws Exception {
        int count = 0;
        while (true) {
            // 使用 Javassist 动态生成类
            ClassPool pool = ClassPool.getDefault();
            CtClass ctClass = pool.makeClass("DynamicClass" + count);
            
            // 添加一个方法
            ctClass.addMethod(CtNewMethod.make(
                "public String getMessage() { return \"Hello\"; }",
                ctClass
            ));
            
            // 生成类
            Class<?> clazz = ctClass.toClass();
            System.out.println("已生成类: " + count++);
        }
    }
}
```

### 场景二：JSP 较多的 Web 应用

<Badge>JSP</Badge>

在传统的 JSP 页面较多的 Web 应用中，每个 JSP 页面编译后都会生成一个 Servlet 类。如果 JSP 文件数量过多，或者频繁修改 JSP 文件，可能导致 Metaspace 溢出。

### 场景三：OSGi 插件化应用

<Badge>OSGi</Badge>

OSGi 框架支持动态加载和卸载 bundle。每个 bundle 都有自己的类加载器，卸载 bundle 时需要同时卸载其类加载器。如果卸载不彻底，可能导致类元信息无法释放。

### 场景四：大量使用反射和代理

<Badge>Reflection</Badge>

CGLIB、ASM 等字节码增强库会动态生成代理类或子类。每个动态生成的类都会占用 Metaspace 空间。

## Metaspace 溢出排查步骤

<Badge>Troubleshooting</Badge>

### 第一步：确认是 Metaspace OOM

```bash
# 查看错误信息
java.lang.OutOfMemoryError: Metaspace
    at java.lang.ClassLoader.defineClass1(Native Method)
    at java.lang.ClassLoader.defineClass(ClassLoader.java:810)
    ...

# 确认是 Metaspace 问题，不是 heap space 问题
```

### 第二步：监控 Metaspace 使用

```bash
# 使用 jstat 监控
jstat -gcutil <pid> 1000

# 输出中的 MC 和 MU 列表示 Metaspace
# MC = Metaspace Capacity（元空间容量）
# MU = Metaspace Used（元空间使用）

# 或者使用 jcmd
jcmd <pid> VM.native_memory summary
```

### 第三步：生成类直方图并分析

<Badge>Class Histogram</Badge>

虽然 Metaspace 不在 heap 中，但分析 ClassLoader 和加载的类数量仍然有帮助。

```bash
# 生成类直方图
jcmd <pid> GC.class_histogram > class_histogram.txt

# 查看类加载器统计
jcmd <pid> VM.classloader_stats > classloader_stats.txt
```

### 第四步：分析代码

检查是否存在以下问题：无限动态生成类、没有正确卸载类加载器、反射使用不当。

## Metaspace 配置优化

<Badge>Metaspace Configuration</Badge>

```bash
# 设置 Metaspace 初始大小
-XX:MetaspaceSize=128m

# 设置 Metaspace 最大大小（强烈建议设置）
-XX:MaxMetaspaceSize=512m

# 设置元空间增长步长
-XX:MinMetaspaceFreeRatio=40
-XX:MaxMetaspaceFreeRatio=70

# 完整配置示例
java -Xms2g -Xmx2g \
     -XX:MetaspaceSize=256m \
     -XX:MaxMetaspaceSize=512m \
     MyApplication
```

## 面试高频考点

**问题一：Metaspace 和 PermGen 的区别是什么？** PermGen 是堆内存的一部分，有固定大小限制；Metaspace 使用本地内存，默认只受可用内存限制。JDK 8 用 Metaspace 替代了 PermGen。

**问题二：什么情况下会触发 Metaspace OOM？** 动态生成大量类、JSP 较多的 Web 应用、OSGi 插件化应用、大量使用反射和代理。

**问题三：如何排查 Metaspace OOM？** 监控 Metaspace 使用情况、分析类加载统计、检查代码中是否存在无限动态生成类、启用类卸载。

**问题四：如何预防 Metaspace OOM？** 设置 MaxMetaspaceSize、使用成熟的缓存框架、正确卸载不再使用的类加载器、限制动态类生成数量。

从 PermGen 到 Metaspace，JVM 经历了一次重要的架构演进。这次演进解决了什么问题？又带来了什么新的挑战？
