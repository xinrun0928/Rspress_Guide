# JVM 调优总览

JVM 调优是 Java 工程师必备的技能之一。在生产环境中，合理的 JVM 参数配置可以让应用运行得更加稳定、高效；而不当的配置可能导致频繁的 GC 停顿、内存不足、甚至 OOM 崩溃。本系列文章将带你系统地学习 JVM 调优的知识，从 OOM 问题的成因分析，到各种排查工具的使用，再到实际的调优经验，帮助你成为 JVM 调优的高手。

## 为什么需要 JVM 调优

<Badge>JVM Tuning</Badge>

在讨论具体的调优方法之前，我们需要理解为什么 JVM 需要调优。JVM 的默认配置是针对通用场景设计的，适用于大多数简单应用。但生产环境中的应用往往有更复杂的需求和更严格的要求。

想象一下，如果你开一家餐厅，厨房设备都是按"普通家庭使用"的标准配置的，突然要接待 1000 人的大型宴会，肯定会出现各种问题。JVM 也是如此——默认配置无法满足所有场景的需求，需要根据实际情况进行调整。

JVM 调优的核心目标是：减少 GC 停顿时间、提高吞吐量、避免 OOM、让资源利用更合理。这些目标有时是相互矛盾的，需要根据应用特点做出取舍。

## JVM 调优的核心原则

<Badge>Core Principles</Badge>

JVM 调优有几个核心原则需要牢记。

第一，不要过早优化。在没有发现实际问题之前，不要随意调整 JVM 参数。过早优化可能引入新的问题，而且没有数据支撑的优化往往是盲目的。

第二，基于数据调优。调优决策应该基于监控数据、GC 日志、性能指标等实际数据，而不是猜测或经验直觉。

第三，一次只改一个参数。调优时应该一次只调整一个参数，观察效果，然后决定是否需要进一步调整。多个参数同时调整无法判断哪个参数起了作用。

第四，理解权衡。JVM 参数之间存在权衡关系。比如，增大堆内存可以减少 GC 频率，但会增加 GC 停顿时间；减小堆内存可以缩短停顿时间，但会增加 GC 频率。

## JVM 调优的关键指标

<Badge>Key Metrics</Badge>

进行 JVM 调优时，需要关注以下几个关键指标。

吞吐量是指应用运行时间占总时间的比例。公式为：吞吐量 = 应用代码运行时间 / (应用代码运行时间 + GC 时间)。对于批处理应用，吞吐量是首要指标。

停顿时间是指每次 GC 造成的应用暂停时间。对于 Web 应用、在线交易系统等对响应时间敏感的应用，停顿时间是首要指标。

内存占用是指应用运行所需的内存空间。在内存资源有限的环境中，内存占用是重要指标。

GC 频率是指 GC 发生的频率。频繁的 GC 即使每次停顿时间短，累积起来也会影响性能。

## 常见的调优场景

<Badge>Common Scenarios</Badge>

不同的应用场景有不同的调优策略。

对于 Web 应用和在线服务，响应时间是首要指标。应该选择停顿时间可控的收集器（如 G1、ZGC），设置合理的停顿时间目标，监控 GC 日志，确保停顿时间在可接受范围内。

对于批处理和大数据计算，吞吐量是首要指标。应该选择吞吐量优先的收集器（如 Parallel Scavenge + Parallel Old），最大化 CPU 利用率，接受较长的 GC 停顿。

对于微服务架构，每个服务实例的内存可能较小，应该选择内存占用低的收集器（如 Serial），合理设置堆大小，避免资源浪费。

对于容器化部署，内存限制是硬约束。应该精确设置堆大小、元空间大小、直接内存大小等，确保不超出容器限制。

## JVM 调优的基本流程

<Badge>Tuning Process</Badge>

JVM 调优应该遵循以下基本流程。

第一步，确定目标。根据应用特点确定调优目标：降低停顿时间、提高吞吐量、还是减少内存占用。

第二步，收集数据。开启 GC 日志，监控系统资源使用，记录应用性能指标。收集足够的数据后再开始调优。

第三步，分析问题。分析 GC 日志和监控数据，找出性能瓶颈。是 GC 停顿时间过长，还是频率过高，还是内存不足。

第四步，制定方案。根据分析结果，制定调优方案。一次只改一个参数，记录修改内容和效果。

第五步，验证效果。上线后持续监控，验证调优效果是否达到预期。如果未达到，继续分析调整。
