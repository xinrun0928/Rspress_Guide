# JVM 虚拟机

双十一零点，订单系统 CPU 告警突然拉到 95%，持续了整整 8 分钟。

DBA 发现数据库连接池正常，Redis 正常，网络正常。排查了三个小时，最后定位到是 Java 进程在疯狂 GC——stop-the-world 从 50ms 飙升到 2s。

问题根源？一个开发同学在初始化 HashMap 时没设置容量，用默认构造函数，初始容量只有 16，每次扩容都要重新计算所有元素的 hash 位置。流量一上来，HashMap 扩容触发了 GC STW。

这就是 JVM——它藏在所有 Java 程序的底层，你平时感知不到它，但一旦出问题，就是大问题。

这一模块，带你把 JVM 从"听说过"升级到"能定位生产问题、能做性能调优"。

## 学习路径指引

### P5 候选人（校招/初级社招）

JVM 的入门关是**理解运行时数据区和基本的垃圾回收算法**。

**P5 必过关卡**：

1. JVM 的运行时数据区有哪些？哪些是线程共享的，哪些是线程私有的？
2. 什么是垃圾判断算法？引用计数和可达性分析各有什么优缺点？
3. 常见的 GC 算法有哪些？各自的优缺点是什么？
4. 什么是分代收集？为什么要分代？年轻代和老年代的比例一般是多少？

这四道题能流畅答出来的，JVM 基础就算过关。

### P6 候选人（中级社招）

P6 的门槛在**能讲清楚垃圾收集器的选型和 G1 的核心原理**，以及**能回答类加载器相关的追问**。

**拉开差距的追问链（GC）**：

```
CMS 和 G1 的区别是什么？
  → G1 为什么叫 Garbage First？它的 Region 是什么概念？
    → Remembered Set 是干什么用的？怎么避免全堆扫描？
      → G1 的 GC 阶段有哪些？Mixed GC 什么时候触发？
        → G1 的参数怎么调？什么场景下 G1 比 CMS 好？
          → 什么场景下 CMS 比 G1 好？
```

这条追问链能走完的，基本都对线上 GC 调优有实战经验。

**拉开差距的追问链（类加载）**：

```
为什么要设计双亲委派模型？
  → 如果不双亲委派，会出现什么问题？（类被覆盖、安全问题）
    → 哪些场景需要打破双亲委派？（Tomcat、JDBC 驱动、SPI）
      → JDBC 的 DriverManager 是怎么加载不同数据库驱动的？（Thread Context ClassLoader）
        → Tomcat 的 WebAppClassLoader 怎么实现隔离的？
```

**生产调优追问**：

```
线上的 GC 停顿时间太长，怎么排查？
  → 怎么分析 GC 日志？用什么工具？
    → 如果频繁 Young GC，是什么问题？（ Eden 区太小/对象太多）
      → 如果频繁 Full GC，是什么问题？（老年代满了/大对象直接进老年代）
        → 怎么用 Arthas/MAT 定位内存泄漏？
```

【面试官心理】
JVM 这道题，我通常从"运行时数据区"或"GC算法"切入。如果候选人基础扎实，我会追问到 G1、ZGC 的细节，或者类加载器的双亲委派。JVM 是个非常吃实战经验的领域——没在线上排查过 GC 问题的，基本答不出"怎么定位"和"用什么工具"。

### P7 候选人（高级社招/架构方向）

P7 的 JVM 是在考**生产调优能力、问题定位能力、以及对底层原理的深度理解**。

**典型的 P7 级问题**：

- G1 和 ZGC 的适用场景分别是什么？你们的线上服务用的哪种？为什么？
- 对象头在不同 GC 算法下有什么区别？Mark Word 的结构是什么？
- 为什么 JDK8 把永久代替换成了元空间？元空间会 OOM 吗？怎么排查？
- JIT 编译优化：逃逸分析、锁消除、标量替换——这些优化在生产上怎么验证？

:::tip 💡
G1 的目标是把停顿时间控制在用户指定的范围内（通过 `-XX:MaxGCPauseMillis` 参数），它通过动态调整年轻代和老年代的比例来实现这一点。但这个参数只是个"目标"，JVM 不保证一定达成。
:::

## 生产避坑指南

JVM 是生产事故的高发区，很多问题都是"平时感知不到，一出就是大事故"：

- **Full GC 频繁**：老年代空间不足、大对象直接进入老年代、Metaspace 满了。排查工具：GC 日志 + MAT + Arthas。
- **Young GC 频繁**：Eden 区太小、对象分配率过高。排查工具：GC 日志里的分配速率分析。
- **OOM 无处不在**：堆内 OOM（对象泄漏）、元空间 OOM（类加载器泄漏）、栈溢出（递归太深）。
- **GC 停顿时间过长**：G1 的 Mixed GC 阶段会 stop-the-world，停顿时间过长需要调整参数。
- **类加载器泄漏**：Tomcat 热部署场景下，旧的 WebAppClassLoader 没有被回收，导致内存泄漏。

:::warning ⚠️
JDK8 的永久代替换成元空间后，很多人以为元空间不会 OOM。实际上元空间使用的是本地内存，默认上限是"无限制"，但受 `MaxMetaspaceSize` 参数限制。如果动态类加载过多（比如热部署场景），元空间 OOM 会直接导致进程崩溃。
:::
