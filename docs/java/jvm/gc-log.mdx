## 一、GC 日志格式 🔴

### 1.1 问题拆解

GC 日志是排查 GC 性能问题的基础数据。面试官追问 GC 日志格式，不是在考记忆力，是在测试候选人是否有实际排查经验。

### 1.2 开启 GC 日志

```bash
# JDK 8 及之前
-XX:+PrintGCDetails
-XX:+PrintGCDateStamps
-Xloggc:/path/to/gc.log
-XX:+UseGCLogFileRotation
-XX:GCLogFileSize=10M
-XX:NumberOfGCLogFiles=5

# JDK 9+
-Xlog:gc*:/path/to/gc.log:time,filecount=5,filesize=10M
```

### 1.3 新生代 GC 日志（Minor GC）

```
2024-01-15T10:23:45.123+0800: 1234.567: [GC (Allocation Failure)
  PSYoungGen: 655360K->65536K(958464K)] 1048576K->950272K(3145728K)
  , 0.023s] [Times: user=0.05 sys=0.01, real=0.02s]
```

**字段解析**：

| 字段 | 含义 |
| --- | --- |
| `Allocation Failure` | 触发原因：Eden 区分配失败 |
| `PSYoungGen: 655360K->65536K(958464K)` | 新生代：GC 前使用 -> GC 后使用(总大小) |
| `1048576K->950272K(3145728K)` | 堆：GC 前 -> GC 后(总大小) |
| `0.023s` | 停顿时间 |
| `user=0.05 sys=0.01, real=0.02` | CPU 用户态/内核态/实际时间 |

### 1.4 老年代 GC 日志（Full GC）

```
2024-01-15T10:23:46.789+0800: 1235.890: [Full GC (Ergonomics)
  [PSYoungGen: 65536K->0K(958464K)]
  [ParOldGen: 875520K->875512K(2097152K)]
  941056K->875512K(3145728K)
  [Metaspace: 52488K->52488K(106496K)]
  , 0.156s]
```

**关键信息**：
- `Full GC`：这是 Full GC（不是 Minor GC）
- `Ergonomics`：由 JVM 自适应策略触发
- `ParOldGen`：老年代空间变化
- `Metaspace`：元空间变化（这里是 `->` 不变，说明没有类卸载）

### 1.5 CMS GC 日志 🟡

```
# CMS 初始标记
```

## 二、G1 GC 日志解读 🟡

### 2.1 G1 Young GC

```
2024-01-15T10:30:00.123: [GC pause (G1 Evacuation Pause) (young)
  Initial Mark: 0.123ms
  Root Region Scan Waiting: 0.000ms
  Concurrent Marking: 0.000ms
  Post Cleaning: 0.000ms
  Evacuation Pause: 1.234ms
  512M->512M(4096M), 0.002s]
  [Times: user=0.008 real=0.002 secs]
```

### 2.2 G1 Mixed GC

```
2024-01-15T10:35:00.456: [GC pause (G1 Evacuation Pause) (mixed)
  Initial Mark: 0.100ms
  Root Region Scan Waiting: 0.000ms
  Concurrent Marking: 0.000ms
  Post Cleaning: 0.000ms
  Evacuation Pause: 15.678ms
  2048M->1536M(4096M), 0.016s]
  [Ext Root Scanning: 2.345ms]
  [Update RS: 1.234ms]
  [Scan RS: 0.567ms]
  [Object Copy: 10.123ms]
  [Termination: 0.456ms]
```

**关键阶段耗时**：
- `Ext Root Scanning`：外部根扫描
- `Update RS`：更新 RSet
- `Scan RS`：扫描 RSet
- `Object Copy`：对象复制（最耗时的阶段）

## 三、GC 日志分析指标 🔴

### 3.1 核心分析指标

```bash
# 使用 gc Easy 分析工具或手动统计
# 1. 吞吐量 = (总运行时间 - GC时间) / 总运行时间
# 2. GC 频率 = Full GC次数 / 运行时间
# 3. 每次 GC 回收量 = 回收前后差值
# 4. 停顿时间分布
```

**GC 正常标准**：

| 指标 | 正常范围 | 异常信号 |
| --- | --- | --- |
| Full GC 频率 | < 1次/分钟 | > 5次/分钟 |
| Full GC 停顿时间 | < 1s | > 3s |
| Young GC 停顿时间 | < 100ms | > 500ms |
| 堆使用率（稳定态） | < 70% | > 90% |
| 元空间使用率 | < 80% | > 90% |

### 3.2 常见 GC 异常模式

**模式一：频繁 Young GC，存活率低**

```
现象：Young GC 每秒触发多次，但每次回收量很少
原因：对象分配速率过高，Eden 区太小
解决：增大 Eden 区 -XX:NewSize 或 -Xmn
```

**模式二：频繁 Full GC，老年代持续增长**

```
现象：Full GC 频繁触发，每次回收量几乎等于晋升量
原因：内存泄漏 或 一次性加载过多数据
解决：用 MAT 分析堆转储，找泄漏对象
```

**模式三：CMS 并发模式失败**

```
[CGC][CMS: abort preclean due to time]
[Full GC (Allocation Failure) ...]

现象：CMS 还没完成并发标记，老年代就满了
原因：并发阶段太快，老年代增长太快
解决：降低 CMSInitiatingOccupancyFraction 或增大堆
```

### 3.3 ❌ 错误示范

**候选人原话**："GC 日志我从来不用看，JConsole 和 JVisualVM 够用了。"

【面试官心理】
这个候选人没有生产环境经验。JConsole 和 JVisualVM 是实时监控工具，无法分析历史数据。而且它们在生产环境中对应用有性能影响，不能长期开启。GC 日志是最轻量、最可靠的分析手段。

## 四、GC 日志分析工具 🟡

### 4.1 常用工具

| 工具 | 用途 | 优缺点 |
| --- | --- | --- |
| **gceasy.io** | 在线 GC 日志分析 | 直观，自动生成报告 |
| **GCViewer** | 本地 GC 日志分析 | 开源，功能全面 |
| **JProfiler** | 专业性能分析工具 | 全面，但收费 |
| **命令行统计** | `grep`/`awk` 统计 | 快速，适合紧急排查 |

### 4.2 命令行快速统计

```bash
# 统计 Full GC 次数
grep "Full GC" gc.log | wc -l

# 统计 Young GC 总耗时
grep "GC (Allocation Failure)" gc.log | awk -F'[,[:space:]]+' '{sum += $NF} END {print sum}'

# 统计 Full GC 总耗时
grep "Full GC" gc.log | awk -F'[,[:space:]]+' '{sum += $NF} END {print sum}'

# 查看最慢的 10 次 GC
grep -E "(GC|Full GC)" gc.log | sort -t, -kNF -r | head -10

# 分析停顿时间分布
awk '/\[Times:/{print $NF}' gc.log | sort -n | uniq -c
```

## 五、生产排查实战 🟡

### 5.1 场景：GC 时间占总时间 15%

```bash
# 第一步：统计 GC 分布
grep -E "(GC|Full GC)" gc.log | awk '{print $NF}' | sort -n

# 第二步：识别是 Young GC 还是 Full GC 慢
grep "Full GC" gc.log | wc -l  # Full GC 次数
grep "GC (Allocation Failure)" gc.log | wc -l  # Young GC 次数

# 第三步：如果是 Young GC 慢
# 查看 Young GC 后 Survivor 区占用
grep "Allocation Failure" gc.log | grep -oP '\d+K->\d+K\(\d+K\)' | tail -20

# 如果是老年代持续增长
grep "Full GC" gc.log | awk '{print $4}' | sort -n  # 查看堆使用
```

:::tip 💡
**黄金法则**：如果 Full GC 频繁且回收量等于晋升量，说明老年代增长 > 回收量，核心问题是**内存泄漏或一次性加载**。如果 Young GC 频繁且每次回收量少，说明**对象分配速率过高**，核心问题是**代码中的临时对象**。
:::

【面试官心理】
能说出一套 GC 日志分析流程的候选人，说明他有实际的生产排查经验。这在 P6+ 面试中是很大的加分项。
