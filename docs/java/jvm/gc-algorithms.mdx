面试官问："垃圾回收有哪几种算法？它们的优缺点是什么？"

候选人小孙说："有标记清除、复制和标记整理三种算法。标记清除会产生碎片，复制算法浪费空间，标记整理没有碎片但效率低。"

面试官追问："那为什么新生代用复制算法，老年代用标记整理？Eden 和 Survivor 的比例为什么是 8:1？"

小孙支支吾吾答不上来。

## 一、三种基础算法 🔴

### 1.1 问题拆解

三种 GC 算法是理解所有现代垃圾收集器的基础。面试官不是在考你背概念，而是在测你是否能理解"为什么这么选型"——新生代用复制算法不是拍脑袋，是和对象生命周期分布高度相关的工程决策。

### 1.2 算法一：标记-清除（Mark-Sweep）

**算法流程**：
1. **标记阶段**：从 GC Roots 出发，遍历所有可达对象，标记为"存活"
2. **清除阶段**：遍历整个堆，回收所有未标记对象（垃圾）的内存

```mermaid
graph TD
    subgraph 标记阶段
        R[GC Roots] --> A[存活对象A]
        R --> B[存活对象B]
        A --> C[存活对象C]
    end
    subgraph 清除阶段
        D[垃圾D] -.x E[垃圾E]
        F[垃圾F] -.x
    end
    style D fill:#ff6b6b
    style E fill:#ff6b6b
    style F fill:#ff6b6b
```

**优点**：
- 实现简单，不需要移动对象
- 适合老年代（对象存活率高，移动成本高）

**缺点**：
- **产生内存碎片**：清除后内存不连续，导致大对象分配失败
- **效率低**：标记和清除都需要遍历整个堆

### 1.3 算法二：复制（Copying）

**算法流程**：将内存分为两块，每次只使用一块。GC 时，将存活对象复制到另一块，然后整块清除。

```mermaid
graph TD
    subgraph 复制前
        A[区域A<br/>使用中] -->|复制存活对象| B[区域B<br/>空闲]
        C1[存活C] --> B
        C2[垃圾D] -.x C1
    end
    subgraph 复制后
        B2[区域B<br/>使用中]
        C3[存活C]
    end
    A -->|清除| B2
    style C2 fill:#ff6b6b
```

**优点**：
- 无内存碎片：分配只需要移动指针（bump-the-pointer）
- 效率高：复制 + 清除一次完成

**缺点**：
- **空间浪费**：可用内存减半
- **存活率高时效率低**：如果存活对象多，复制成本高

### 1.4 算法三：标记-整理（Mark-Compact）

**算法流程**：
1. **标记阶段**：和标记-清除一样
2. **整理阶段**：将存活对象向一端移动，紧凑排列，然后清理边界外内存

```mermaid
graph TD
    subgraph 整理前
        A1[存活] -->|移动| B1[存活]
        A2[存活] -->|移动| B2[存活]
        G1[垃圾] -.x
        G2[垃圾] -.x
    end
    style G1 fill:#ff6b6b
    style G2 fill:#ff6b6b
```

**优点**：
- 无内存碎片
- 空间利用率 100%

**缺点**：
- **移动成本高**：需要更新所有引用（指针更新）
- **需要 stop-the-world**：整理期间用户线程不能运行
- **适合老年代**：存活率高，移动成本相对可控

## 二、为什么新生代用复制算法 🟡

### 2.1 核心原因：对象生命周期分布

根据 IBM 的研究数据：
- **98% 的对象朝生夕死**：创建后很快变成垃圾
- 存活对象中，大多数也只经历 1~2 次 Minor GC

这意味着：**新生代中需要复制的对象很少，复制成本很低**。

```
Eden : Survivor = 8 : 1 时：
- Eden 区满了，触发 Minor GC
- 存活对象 < 10%（大多数对象已死亡）
- 只需要复制 10% 的对象到 Survivor，浪费的不过是一个 Survivor 区
- 空间利用率 = 80% + 10% = 90%
```

### 2.2 ❌ 错误示范

**候选人原话**："新生代用复制算法是因为复制算法快。"

【面试官心理】
这个答案只说对了一半。复制算法快是在存活对象少的前提下。如果存活对象多，复制算法的成本反而更高。面试官想知道的是候选人是否理解"为什么新生代存活对象少"这个前提假设。

**候选人原话 2**："标记整理比标记清除好，所以新生代也应该用标记整理。"

面试官追问："标记整理需要移动对象，如果对象多，移动成本是多少？"

候选人："...应该还行？"

如果候选人不理解"整理需要更新所有引用"这个开销，就无法理解为什么新生代不选标记整理。

## 三、三种算法的对比 🟡

| 维度 | 标记-清除 | 复制 | 标记-整理 |
| --- | --- | --- | --- |
| **速度** | 中等 | 最快（无碎片） | 最慢（移动+更新引用） |
| **空间利用率** | 100%（有碎片） | 50%（两半） | 100% |
| **碎片问题** | 有碎片 | 无碎片 | 无碎片 |
| **移动对象** | 否 | 是 | 是 |
| **适用区域** | 老年代 | 新生代 | 老年代 |
| **STW 时间** | 中等 | 短（仅复制存活对象） | 长（需更新所有引用） |
| **引用更新** | 不需要 | 需要 | 需要 |

## 四、实际 GC 中的算法组合 🟡

### 4.1 新生代的 Minor GC

```
Eden 区 → Minor GC → 存活对象 → Survivor To 区
                               → 直接晋升老年代（Survivor 放不下时）
Survivor From 和 To 角色互换
```

### 4.2 老年代的 Full GC

```
老年代满了 → Full GC（串行/并行 CMS/G1 ZGC）
  → 标记-清除（CMS）
  → 标记-整理（Serial Old / Parallel Old / G1）
  → 染色指针+读屏障（ZGC）
```

### 4.3 面试高频追问

**追问**：CMS 用的是标记-清除，为什么不用标记-整理？

CMS 的目标是"最短停顿时间"。标记-整理需要移动对象并更新所有引用，这会引发长停顿。CMS 选择标记-清除，用碎片换时间。但碎片会导致后续分配失败——这就是 CMS 的"并发模式失败"问题：当老年代碎片化严重，无法分配大对象时，触发 Full GC。

【面试官心理】
能说出 CMS 和标记-清除的"交易"的候选人，说明他理解 GC 算法的核心矛盾：时间、空间、停顿——三者不可兼得。这是 P7 级别的系统设计思维。
