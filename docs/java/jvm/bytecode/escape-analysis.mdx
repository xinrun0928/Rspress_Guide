# 逃逸分析

逃逸分析（Escape Analysis）是 JIT 编译器进行的一项关键优化技术。它分析对象的动态作用域，判断对象是否会"逃逸"到方法外部。如果对象不逃逸，JIT 就可以对其进行一系列激进优化，包括栈上分配、标量替换等。这些优化可以显著减少堆分配和垃圾回收的压力，提升程序性能。

## 什么是对象逃逸

|逃逸定义|

对象逃逸是指对象的引用被传递到创建它的方法或线程之外。当对象逃逸后，它就成为了在堆上分配的理由，GC 必须跟踪它的生命周期。

```java
public class EscapeExamples {
    // 情况一：对象不逃逸
    public int compute() {
        Point p = new Point(1, 2);  // 局部对象
        return p.x + p.y;           // 只在方法内使用
    }
    // 优化机会：对象可以栈上分配或标量替换
    
    // 情况二：对象逃逸到方法返回
    public Point createPoint() {
        Point p = new Point(1, 2);  // 局部对象
        return p;                     // 返回引用，对象逃逸
    }
    // 优化机会：不能栈上分配，必须堆分配
    
    // 情况三：对象逃逸到其他线程
    public void sharePoint() {
        Point p = new Point(1, 2);
        sharedRef = p;  // 逃逸到其他线程
    }
    // 优化机会：不能栈上分配，必须堆分配
}
```

## 逃逸分析的三个级别

|逃逸级别|

JDK 6 引入了逃逸分析，但默认是关闭的（直到 JDK 8 默认开启）。逃逸分析分为三个级别：

**不分析（No Escape）**：对象不逃逸，仅在创建它的方法内使用。

**参数逃逸（Arg Escape）**：对象作为参数传递给其他方法，或者在方法返回时返回引用。

**全局逃逸（Global Escape）**：对象逃逸到其他线程，或者被存储在静态变量中。

```java
public class EscapeLevels {
    static Object shared;  // 全局逃逸
    
    // 全局逃逸
    public static void globalEscape() {
        Object obj = new Object();
        shared = obj;  // 逃逸到静态变量
    }
    
    // 参数逃逸
    public void argEscape(Object obj) {
        // obj 是参数传入的，可能已经逃逸
    }
    
    // 不逃逸
    public int compute() {
        int x = 10;
        int y = 20;
        return x + y;  // 只有基本类型，不涉及对象
    }
}
```

## 栈上分配：对象的栈内存

|栈上分配|

如果逃逸分析判断一个对象不会逃逸，JIT 编译器可以选择在栈上分配对象，而不是堆上分配。

栈上分配的优势是巨大的。栈内存的分配和回收只需要移动栈指针，不需要垃圾回收器的介入。而且栈上分配的对象会随着方法结束自动销毁，不需要额外的清理工作。

```java
// 没有逃逸分析优化的情况
public class WithoutEscapeAnalysis {
    public static void main(String[] args) {
        long start = System.currentTimeMillis();
        for (int i = 0; i < 10000000; i++) {
            allocate();
        }
        long end = System.currentTimeMillis();
        System.out.println("耗时: " + (end - start) + "ms");
    }
    
    static void allocate() {
        // 如果没有逃逸分析优化，对象在堆上分配
        // 需要 GC 定期回收这些短期对象
        Point p = new Point(1, 2);
        p.x = 3;
    }
}

class Point {
    int x;
    int y;
    Point(int x, int y) { this.x = x; this.y = y; }
}
```

启用逃逸分析后，`Point` 对象可能直接在栈上分配，不需要进入堆，GC 的压力大大减轻。

## 标量替换：拆解对象

|标量替换|

标量替换是另一种基于逃逸分析的优化。当对象不逃逸时，JIT 编译器可以将对象"拆解"为其成员变量（标量），直接在寄存器或栈上操作，而不需要真正创建对象。

```java
// 标量替换前
public int compute() {
    Point p = new Point(1, 2);
    return p.x + p.y;
}

// JIT 优化后（标量替换）
public int compute() {
    // Point 对象被拆解为两个 int 变量
    int x = 1;
    int y = 2;
    return x + y;
}
```

标量替换完全消除了对象分配的开销。对象不需要分配内存，不需要 GC 回收，成员变量的访问也变成了直接操作寄存器或局部变量。

## 同步锁消除

|锁消除|

逃逸分析的另一个重要优化是同步锁消除。如果逃逸分析判断一个对象只在一个线程内访问，JIT 编译器会消除该对象上的同步锁。

```java
public class LockElision {
    public void compute() {
        // 编译器可能判断 sb 不会逃逸
        // 因此同步锁被消除
        StringBuilder sb = new StringBuilder();
        synchronized (sb) {
            sb.append("hello");
        }
    }
}
```

同步锁消除通过 `-XX:+DoEscapeAnalysis` 参数控制（默认开启）。

## 逃逸分析参数配置

```bash
# 逃逸分析开关（JDK 8 默认开启）
-XX:+DoEscapeAnalysis     # 开启逃逸分析
-XX:-DoEscapeAnalysis     # 关闭逃逸分析

# 逃逸分析结果是渐进的，JIT 编译后才会生效
# 可以通过以下参数查看逃逸分析信息
-XX:+PrintEscapeAnalysis  # 打印逃逸分析结果
-XX:+PrintOptimization    # 打印 JIT 优化信息

# 示例
java -XX:+DoEscapeAnalysis \
     -XX:+PrintEscapeAnalysis \
     -Xmx256m \
     MyApplication
```

## 逃逸分析的实际效果

逃逸分析的优化效果取决于代码的具体写法。以下场景更容易获得优化：

```java
// 场景一：短期局部对象
public void process() {
    // 对象不逃逸，可以栈上分配
    Request req = new Request();
    req.setData("data");
    handle(req);
    // 方法结束时自动销毁，不需要 GC
}

// 场景二：方法返回数组
public int[] compute() {
    // 返回数组可能逃逸，但如果不返回则可优化
    int[] arr = new int[1000];
    for (int i = 0; i < 1000; i++) {
        arr[i] = i * i;
    }
    return arr;  // 返回则逃逸
}

// 场景三：StringBuilder 拼接
public String buildString() {
    StringBuilder sb = new StringBuilder();
    // 如果 sb 不逃逸，append 可能被优化
    sb.append("hello");
    sb.append("world");
    return sb.toString();
}
```

## 逃逸分析的限制

逃逸分析虽然强大，但也有其局限性：

**保守性**：JIT 编译器倾向于保守判断，只有在确定不逃逸时才进行优化。如果分析不确定，对象仍然会在堆上分配。

**复杂度**：逃逸分析需要分析整个调用图，对于多线程、反射、动态类加载等场景，分析的复杂度很高。

**性能开销**：逃逸分析本身也有开销。如果分析方法太多，可能会影响编译速度。

**JIT 门槛**：逃逸分析是在 JIT 编译时进行的，只有热点代码才会触发 JIT 编译。首次运行时仍然使用解释执行，没有优化。

```bash
# 设置 JIT 编译阈值
-XX:CompileThreshold=10000  # 方法调用次数达到这个值时触发 JIT 编译

# 设置编译线程数
-XX:CICompilerCount=2       # JIT 编译线程数
```

## 逃逸分析与性能调优

理解逃逸分析对于性能调优很有帮助。以下是一些实践建议：

```java
// 建议一：优先在方法内创建对象
public void good() {
    // 对象不逃逸，可能被优化
    List<Item> items = new ArrayList<>();
    for (int i = 0; i < 10; i++) {
        items.add(new Item(i));
    }
    process(items);  // items 本身可能逃逸
}

// 建议二：谨慎使用返回值
public Point createPoint(int x, int y) {
    return new Point(x, y);  // 返回则逃逸
}

// 建议三：使用基本类型包装类
public long compute() {
    // 可能被优化为标量替换
    Long sum = 0L;  // 尝试使用基本类型 long
    for (int i = 0; i < 1000; i++) {
        sum += i;
    }
    return sum;
}
```
