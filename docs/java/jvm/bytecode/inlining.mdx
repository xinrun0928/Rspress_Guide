# 方法内联与热点代码探测

方法内联（Method Inlining）是 JIT 编译器最重要的优化技术之一，也是解锁其他优化的钥匙。没有方法内联，许多其他优化（如常量折叠、死代码消除）就无法进行。同时，方法内联也为虚方法调用、锁消除等优化奠定了基础。本篇文章深入探讨方法内联的原理、策略以及热点代码探测机制。

## 方法内联的基本原理

|内联原理|

方法内联是将目标方法的调用点替换为方法体本身的优化技术。这一优化消除了方法调用的开销（参数传递、栈帧创建、返回值处理），更重要的是，内联后 JIT 编译器可以看到更大的代码上下文，从而进行更多优化。

```java
// 内联前
public class BeforeInlining {
    public int compute(int a, int b) {
        return add(a, b) * add(b, a);  // 两次方法调用
    }
    
    private int add(int x, int y) {
        return x + y;
    }
}

// JIT 优化后（内联）
public int compute(int a, int b) {
    // add(a, b) 被替换为 a + b
    // add(b, a) 被替换为 b + a
    return (a + b) * (b + a);  // 无方法调用
}
```

方法内联的收益来自几个方面：消除调用成本（方法调用涉及压栈、出栈、跳转等操作）；解锁更多优化（内联后可以看到完整的调用上下文）；提高缓存命中率（热代码集中，减少指令缓存失效）。

## 内联的条件

|内联条件|

JIT 编译器不会无条件地内联所有方法。内联需要满足一定条件：

**热点代码**：只有被频繁调用的方法才值得内联，因为内联本身也有编译开销。

**方法大小**：太大的方法不适合内联，因为会显著增加代码体积。

**调用点数量**：调用次数少的方法不值得内联。

```bash
# JIT 内联决策因素
# - 方法调用计数器（热点探测）
# - 方法字节码大小（默认 < 35 字节可内联）
# - 调用点的调用频率
# - 虚方法调用的类型推断

# 控制内联大小的参数
-XX:MaxInlineLevel=9           # 最大内联嵌套层数
-XX:MaxInlineMethodSize=35      # 可内联方法的最大字节码大小
-XX:MinInliningThreshold=250   # 触发内联的最小调用次数
```

## 热点代码探测

|热点探测|

JIT 编译器需要识别哪些代码是"热点代码"，只对热点代码进行编译优化。热点探测使用两种策略：基于采样的探测和基于计数的探测。

HotSpot JVM 使用的是基于计数器的探测，为每个方法维护两个计数器：调用计数器（Invocation Counter）和回边计数器（Backedge Counter）。

调用计数器记录方法被调用的次数。当调用次数达到阈值（默认 10000）时，方法会被标记为热点，触发 JIT 编译。

回边计数器记录方法中的循环体被执行次数。循环体被执行多次也会触发 JIT 编译。

```java
public class HotSpotDetection {
    public long sum = 0;
    
    // 热点方法：循环中的调用
    public void compute() {
        for (int i = 0; i < 100000; i++) {
            sum += add(i);  // 循环体被频繁执行
        }
    }
    
    private long add(int x) {
        return x * x;  // 这个方法可能被内联
    }
}
```

## 分层编译

|JIT 分层|

现代 JVM 采用分层编译（Tiered Compilation）策略，分为多个编译层级：

**解释执行（Tier 0）**：程序开始时，所有代码都在解释器中执行。解释执行启动快，但速度慢。

**C1 编译（Tier 1-3）**：快速但轻量的优化。编译速度快，但优化程度较低。适用于启动阶段。

**C2 编译（Tier 4）**：深度优化。编译速度慢，但生成的代码性能高。适用于长期运行的热代码。

```
分层编译流程:

代码执行
    │
    ▼
┌─────────────────┐
│  Tier 0: 解释执行 │
└────────┬────────┘
         │ (调用计数器达到阈值)
         ▼
┌─────────────────┐
│ Tier 3: C1 编译  │ ← 快速编译
└────────┬────────┘
         │ (调用计数继续增加)
         ▼
┌─────────────────┐
│ Tier 2: C1 编译  │ ← 完整优化
└────────┬────────┘
         │ (触发分层阈值)
         ▼
┌─────────────────┐
│ Tier 1: C2 编译  │ ← 激进优化
└─────────────────┘
```

分层编译可以通过 `-XX:+TieredCompilation` 参数启用（JDK 8+ 默认开启）。

```bash
# 启用分层编译（默认开启）
-XX:+TieredCompilation

# 关闭分层编译
-XX:-TieredCompilation

# 分层阈值参数
-XX:TierXInvocationThreshold=50000    # Tier 3 -> Tier 2 的阈值
-XX:TierXMinInvocationThreshold=1000 # Tier 3 的最低调用次数
-XX:TIERED_COMPILATION=1            # 启用分层编译（JDK 7 之前需要）
```

## 虚方法的内联

|虚方法内联|

虚方法调用（如多态调用）是方法内联的一个挑战。因为在编译时无法确定具体调用哪个实现，所以理论上虚方法不能内联。

但 JIT 编译器有"虚化"技术来解决这个问题。通过类型推断（Type Profile），JIT 编译器可以发现某些虚方法的实际类型是稳定的，此时可以进行虚化（Devirtualization）并内联。

```java
public class VirtualInlining {
    public static void main(String[] args) {
        Animal animal = new Dog();  // 类型稳定
        for (int i = 0; i < 100000; i++) {
            animal.speak();  // 虚方法调用
        }
    }
}

abstract class Animal {
    abstract void speak();
}

class Dog extends Animal {
    @Override
    void speak() {
        // JIT 可能推断出是 Dog 类型
        // 进行虚化并内联
    }
}
```

如果 JIT 编译器发现 `animal` 总是指向 `Dog` 对象，就可以将 `animal.speak()` 虚化为 `Dog.speak()` 并进行内联。

## 内联与锁消除

|锁消除|

方法内联为另一个重要优化——锁消除（Lock Elision）奠定了基础。锁消除发生在逃逸分析确认对象不会逃逸到其他线程后。

```java
public class LockCoarsening {
    private final Object lock = new Object();
    
    // JIT 可能进行锁消除和内联
    public void process() {
        synchronized (lock) {
            // 操作
        }
        synchronized (lock) {
            // 操作
        }
    }
}
```

如果 JIT 判断 `lock` 对象不会逃逸，同步块可能被完全消除，这就是锁消除。

## 内联与常量折叠

|常量折叠|

方法内联后，可以进行常量折叠等优化。内联让 JIT 编译器看到完整的调用上下文，能够进行更多激进的优化。

```java
// 内联前
public class BeforeFolding {
    public static final int CONST = 100;
    
    public int compute() {
        return multiply(CONST);
    }
    
    private int multiply(int x) {
        return x * 2;
    }
}

// 内联 + 常量折叠后
public int compute() {
    return 100 * 2;  // 内联后进一步折叠为 200
}
```

## 内联参数

```bash
# 内联相关参数

-XX:MaxInlineLevel=9                     # 最大内联嵌套层数
-XX:MaxInlineMethodSize=35              # 可内联方法的最大字节码大小
-XX:MinInliningThreshold=250             # 触发内联的最小调用次数
-XX:InlineSynchronizedMethods=true       # 允许内联同步方法（默认开启）

# 查看内联信息
-XX:+PrintCompilation                   # 打印编译信息
-XX:+PrintInlining                      # 打印内联信息

# 示例

     -XX:+PrintCompilation \
     -XX:+PrintInlining \
     -Xmx256m \
     MyApplication
```

## 内联的实际影响

方法内联对性能的影响是显著的。以下是一些实验数据（仅供参考，实际效果取决于具体场景）：

```java
// 热点方法的内联优化
public class InlineDemo {
    // 优化前：每次调用都有方法调用开销
    public long computeSum(int n) {
        long sum = 0;
        for (int i = 0; i < n; i++) {
            sum += square(i);  // 每次调用都有开销
        }
        return sum;
    }
    
    private long square(int x) {
        return (long) x * x;
    }
}

// JIT 优化后：square 方法被内联
public long computeSum(int n) {
    long sum = 0;
    for (int i = 0; i < n; i++) {
        sum += (long) i * i;  // 无方法调用
    }
    return sum;
}
```
