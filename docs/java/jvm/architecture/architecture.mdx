# JVM 整体架构图解

凌晨 2 点，线上服务突然 OOM 了。你登录服务器，查看日志，发现是堆内存不足。但为什么堆内存会不足？是因为代码有内存泄漏，还是流量太大导致对象创建过多？如果不了解 JVM 的整体架构，这些问题都很难回答。JVM 不是黑盒子——理解它的架构，是排查生产问题的必备技能。

## 类加载子系统：字节码的"搬运工"

类加载子系统负责将 `.class` 文件加载到 JVM 中。但这个过程远非简单的文件读取，它包括五个严格的阶段：

### 加载阶段

"加载"的核心任务是"找到并读取"字节码文件：

```java
// JVM 需要根据类的全限定名找到对应的 .class 文件
// 例如：java.lang.String -> <JAVA_HOME>/lib/rt.jar 中的 String.class
// 然后以二进制字节流的形式读取进来
```

### 验证阶段

验证阶段是 JVM 的"安检部门"：

- **文件格式验证**：魔数（`0xCAFEBABE`）、版本号等
- **元数据验证**：语法层面的语义检查
- **字节码验证**：控制流、数据流分析
- **符号引用验证**：类、字段、方法的访问权限

如果一个不怀好意的 `.class` 文件试图攻击 JVM，验证阶段会毫不留情地拦截。

### 准备阶段

为类变量分配内存并设置初始值。**关键点**：这个阶段设置的是"零值"，而不是代码中显式赋予的值：

```java
public class PrepareDemo {
    // 在准备阶段，value 会被初始化为 0，而不是 123
    public static int value = 123;

    // 在准备阶段，reference 会被初始化为 null，而不是 "initialized"
    public static String reference = "initialized";
}
```

### 解析阶段

将符号引用替换为直接引用：

```java
// 符号引用：java.lang.Object 只是一个字符串
// 直接引用：指向方法区中 Object 类实际内存地址的指针
```

### 初始化阶段

类加载过程的最后一步，真正执行 Java 代码：

```java
public class InitDemo {
    static {
        // <clinit>() 方法会执行这里的代码
        // 所有静态变量的赋值操作都在这里
        value = 200;
    }

    public static int value = 100;  // 实际执行顺序决定最终值
}
```

## 运行时数据区：Java 程序的"家"

运行时数据区是 JVM 管理的内存区域，也是 Java 程序员最关心的部分。

### 线程私有区域

这些区域与线程同生共死，不需要考虑并发问题：

| 区域 | 作用 | 大小 |
|------|------|------|
| 程序计数器 | 记录当前执行的字节码指令地址 | 固定大小，几乎不占内存 |
| 虚拟机栈 | 存储方法调用的栈帧 | 可通过 `-Xss` 调整 |
| 本地方法栈 | 为 native 方法服务 | HotSpot 与虚拟机栈合并 |

### 线程共享区域

这些区域被所有线程共同访问，需要考虑线程安全：

| 区域 | 作用 | 特点 |
|------|------|------|
| 堆 | 存储对象实例和数组 | GC 的主战场 |
| 方法区 | 存储类元信息 | JDK 7 PermGen，JDK 8 Metaspace |

### 各区域与异常类型的关系

理解这种对应关系对于排查问题至关重要：

| 区域 | 异常类型 |
|------|---------|
| 堆 | `OutOfMemoryError: Java heap space` |
| 虚拟机栈 | `StackOverflowError`、`OutOfMemoryError` |
| 方法区 | `OutOfMemoryError: Metaspace`（JDK 8+） |
| 直接内存 | `OutOfMemoryError: Direct buffer memory` |

## 执行引擎：字节码的"翻译官"

执行引擎是 JVM 最核心的组件，它负责执行 Java 线程的字节码指令。

### 解释器 vs JIT 编译器

```
解释器                          JIT 编译器
   │                                │
   ↓                                ↓
逐条解释执行                   热点代码探测
   │                                │
   ↓                                ↓
启动快，执行慢                  编译慢，执行快
（每条指令都需要解释）          （生成本地机器码）
```

### 分层编译策略

HotSpot JVM 采用了分层编译：

- **C0 编译**：轻量级优化，编译速度快，代码质量一般
- **C1 编译**：更激进的优化，代码质量更高
- **C2 编译**：最激进的优化，可能启用实验性优化技术

```bash
# JVM 参数控制编译级别
# -Xint：纯解释模式
# -Xcomp：纯编译模式（禁用解释器）
# -Xmixed：混合模式（默认）
```

### 热点代码探测

JIT 编译器不是对所有代码都编译，它只会编译"热点代码"：

- **方法调用计数器**：统计方法被调用的次数
- **回边计数器**：统计循环体执行的次数

```java
// 热点代码示例
for (int i = 0; i < 1000000; i++) {
    sum += i;  // 这段循环代码会被 JIT 编译
}
```

## 垃圾回收：内存的"清洁工"

垃圾回收是 JVM 自动内存管理的核心，负责回收不再使用的对象，释放内存空间。

### GC 算法演进

| 算法 | 特点 | 代表收集器 |
|------|------|----------|
| 标记-清除 | 简单，但会产生内存碎片 | Serial GC |
| 复制 | 无碎片，但浪费一半空间 | Serial GC（年轻代） |
| 标记-整理 | 无碎片，整理有开销 | Parallel Old |
| 分代收集 | 根据对象生命周期选择策略 | G1、ZGC |

### GC 调优核心参数

```bash
# 堆大小
-Xms512m      # 最小堆
-Xmx1024m     # 最大堆
-Xmn256m      # 年轻代大小

# 垃圾收集器
-XX:+UseG1GC          # 使用 G1 收集器
-XX:+UseZGC           # 使用 ZGC（低延迟）

# GC 日志
-Xlog:gc*:file=gc.log # 打印 GC 日志
```

## 本地接口与本地方法库

本地接口（JNI，Java Native Interface）是 Java 与本地代码交互的标准接口。

### 为什么需要本地方法？

- 访问操作系统级别的特性（文件操作、网络编程）
- 调用遗留的 C/C++ 库
- 提升性能关键代码的执行效率
- 与硬件设备交互

```java
public class NativeMethodDemo {
    // 声明一个 native 方法
    public native void nativeOperation();

    static {
        // 加载包含 native 方法实现的动态库
        System.loadLibrary("nativelib");
    }
}
```

### 常见 native 方法

- `Object.clone()` — 对象克隆
- `System.arraycopy()` — 数组复制
- `Thread.start()` — 启动线程
- `ClassLoader.loadClass()` — 加载类

## 组件协作示例：一次方法调用的完整旅程

理解了各个组件的功能后，让我们通过一个具体的例子看看它们是如何协作的：

```java
String s = new String("hello");
```

1. **类加载子系统**：检查 `String` 类是否已加载，如果没有，执行类加载过程
2. **堆内存分配**：在堆中分配 `String` 对象的内存空间
3. **执行引擎**：执行 `new` 指令，调用构造函数
4. **本地接口**：如果涉及 native 方法，调用本地方法库
5. **垃圾回收**：当对象不再被引用时，回收其占用的内存

## 面试高频考点

**问题一：JVM 的主要组成部分有哪些？**

类加载子系统负责加载字节码；执行引擎负责执行字节码指令；运行时数据区管理内存；本地接口支持与本地代码交互。

**问题二：类加载的过程是怎样的？**

加载（读取字节码）、验证（检查安全性）、准备（分配内存初始化零值）、解析（符号引用转直接引用）、初始化（执行赋值代码）。

**问题三：JVM 采用什么执行引擎？有什么特点？**

现代 JVM 采用解释器 + JIT 编译器的混合架构。解释器启动快但效率低，JIT 编译器将热点代码编译为本地机器码执行。

**问题四：堆和栈的区别是什么？**

堆是线程共享的，存储对象实例和数组，受 GC 管理；栈是线程私有的，存储局部变量和方法调用信息，不受 GC 管理。
