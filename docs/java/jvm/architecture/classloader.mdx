# 类加载子系统与类加载器分类

Java 程序的运行始于类加载，而类加载器就是负责这项关键工作的"搬运工"。理解类加载器的工作机制，不仅能帮助我们理解 Java 的动态性——运行时加载、依赖注入框架的热加载、OSGi 的模块化——更能让我们在遇到类找不到、类冲突等疑难杂症时有诊断的依据。

## 类加载器的角色与职责

类加载器（ClassLoader）是 JVM 架构中类加载子系统的核心执行者。它的核心职责说起来很简单：**根据类的全限定名，找到对应的 .class 文件，将其字节流转换为 JVM 内部的 java.lang.Class 对象**。但这个简单的职责背后，隐藏着复杂的设计考量和精妙的技术实现。

首先，类加载器是 Java 动态性的基石。在 C/C++ 等编译型语言中，程序在编译时就已经确定了所有要执行的代码；而在 Java 中，类可以在运行时被动态加载。这意味着你可以在程序运行过程中根据条件决定加载哪个类，实现插件机制、热部署、依赖注入等高级特性。Spring 框架的依赖注入、Tomcat 的热部署、Jenkins 的插件系统，背后都依赖于类加载器的动态加载能力。

其次，类加载器天然具有层次结构。JVM 中存在多个类加载器，它们以父子关系组织，形成一个树形结构。最顶层是 Bootstrap ClassLoader（启动类加载器），往下是 Extension ClassLoader（扩展类加载器），再往下是 Application ClassLoader（应用程序类加载器）。每个类加载器都有自己的职责范围，加载类的请求会沿着这个层级向上传递。

第三，不同的类加载器加载的类是"不同"的。这听起来有些违反直觉，但在 Java 的世界里，两个类是否"相等"，不仅取决于类的全限定名，还取决于加载它们的类加载器。假设我们用两个不同的类加载器加载同一个类 `com.example.Test`，得到的两个 Class 对象在 `instanceof` 比较时将返回 false。这不是 JVM 的 bug，而是一种精心设计的安全机制。

## Bootstrap ClassLoader：最底层的守护者

Bootstrap ClassLoader 是 JVM 最早创建的类加载器，也是整个类加载器层级的最顶层。它不是用 Java 代码实现的（至少在 HotSpot JVM 中是这样），而是由 C++ 编写的 JVM 内部代码实现的。正因为如此，在 Java 代码中试图获取 Bootstrap ClassLoader 通常会得到 null。

Bootstrap ClassLoader 负责加载 Java 核心类库，具体来说就是加载 `<JAVA_HOME>/lib/rt.jar`、`<JAVA_HOME>/lib/tools.jar`、`<JAVA_HOME>/lib/ext/*.jar` 等核心 jar 包中的类。这些类是 Java 运行时环境的基础，包括 java.lang.String、java.lang.Object、java.lang.Thread 等你每天都在使用的类。

如果你好奇 Bootstrap ClassLoader 实际加载了哪些类，可以通过以下方式查看：

```java
import java.net.URL;
import java.security.CodeSource;

public class BootstrapClassLoaderDemo {
    public static void main(String[] args) {
        // 获取 String 类的类加载器
        ClassLoader classLoader = String.class.getClassLoader();
        System.out.println("String class loader: " + classLoader);
        // 输出: String class loader: null (表示 Bootstrap ClassLoader)
        
        // 查看 Bootstrap ClassLoader 的类路径
        CodeSource source = Object.class.getProtectionDomain().getCodeSource();
        System.out.println("Object class loaded from: " + source);
    }
}
```

由于 Bootstrap ClassLoader 是用 C++ 实现的，它加载的类在 JVM 内部有着特殊的地位。这些类的元数据存储在 Metaspace 中，而不是堆内存中。理解这一点对于排查 PermGen/Metaspace 相关的问题非常重要。

## Extension ClassLoader：JDK 扩展的门户

Extension ClassLoader 是 Java SE 扩展机制的产物。它的职责是加载 `<JAVA_HOME>/lib/ext` 目录下的 jar 包中的类，以及通过 `java.ext.dirs` 系统属性指定的目录下的类。

Extension ClassLoader 是由纯 Java 代码实现的，具体实现类是 `sun.misc.Launcher.ExtClassLoader`。它直接继承了 `java.lang.ClassLoader` 类，是 Application ClassLoader 的父加载器。

JDK 为什么要设计扩展类加载器呢？想象这样一个场景：JDK 核心类库提供了一套接口，而某组织开发了一套实现类。如果这些实现类必须放在核心 jar 包中，就会破坏 JDK 的完整性；如果必须放在应用类路径中，用户使用起来又不方便。扩展类加载器机制提供了一种折中方案——实现类放在指定的扩展目录中，由 JVM 自动加载，对应用程序透明。

在 JDK 9 之后，Extension ClassLoader 的概念发生了变化。由于 JDK 9 引入了模块化系统（Project Jigsaw），扩展机制被重新设计。`java.ext.dirs` 系统属性被废弃，扩展类加载器加载的类现在由平台类加载器（Platform ClassLoader）负责加载。但从功能上讲，平台类加载器继承了扩展类加载器的职责。

## Application ClassLoader：应用程序的大管家

Application ClassLoader（也称为 System ClassLoader）是应用程序默认的类加载器。它的职责是加载应用程序类路径（ClassPath）上的类，包括 `java.class.path` 系统属性指定的目录和 jar 包，以及通过 `-cp` 或 `-classpath` 命令行参数指定的类路径。

与 Extension ClassLoader 一样，Application ClassLoader 也是由纯 Java 代码实现的，具体实现类是 `sun.misc.Launcher.AppClassLoader`。它继承了 `URLClassLoader` 类，后者是处理文件 URL 和 jar 包 URL 的标准实现。

日常开发中，我们编写的代码、打包的第三方依赖，基本上都由 Application ClassLoader 负责加载。如果你没有显式指定自定义的类加载器，`Thread.currentThread().getContextClassLoader()` 默认返回的就是 Application ClassLoader。

```java
public class ApplicationClassLoaderDemo {
    public static void main(String[] args) {
        ClassLoader appClassLoader = ApplicationClassLoaderDemo.class.getClassLoader();
        System.out.println("Current class loader: " + appClassLoader);
        System.out.println("Parent class loader: " + appClassLoader.getParent());
        System.out.println("Grandparent: " + appClassLoader.getParent().getParent());
        
        // Context ClassLoader
        ClassLoader contextClassLoader = Thread.currentThread().getContextClassLoader();
        System.out.println("Context class loader: " + contextClassLoader);
    }
}
```

运行这段代码，你会看到输出类似：

```
Current class loader: sun.misc.Launcher$AppClassLoader@18b4aac2
Parent class loader: sun.misc.Launcher$ExtClassLoader@1ee0005
Grandparent: null
```

注意，Application ClassLoader 的父加载器是 Extension ClassLoader，而 Extension ClassLoader 的父加载器显示为 null——这并不意味着 Extension ClassLoader 没有父加载器，而是 Bootstrap ClassLoader 用 null 表示。继续向上追溯，Extension ClassLoader 的 loadClass 方法会委托给 Bootstrap ClassLoader，只是后者用 null 来标识自己。

## 自定义类加载器：打破常规的利器

除了 JDK 内置的三个类加载器，Java 还支持自定义类加载器。自定义类加载器是实现热部署、隔离、加密等高级功能的关键。

要创建自定义类加载器，只需要继承 `java.lang.ClassLoader` 类，并重写 `findClass(String name)` 方法。`findClass` 方法是自定义类加载器的核心入口，它负责根据类的全限定名找到对应的字节码文件，并将其转换为 byte 数组。

```java
import java.io.*;

public class CustomClassLoader extends ClassLoader {
    private String classPath;
    
    public CustomClassLoader(String classPath) {
        this.classPath = classPath;
    }
    
    @Override
    protected Class<?> findClass(String name) throws ClassNotFoundException {
        // 获取类的字节码
        byte[] classData = getClassData(name);
        if (classData == null) {
            throw new ClassNotFoundException("Cannot find class: " + name);
        }
        // 调用父类的 defineClass 方法将字节码转换为 Class 对象
        return defineClass(name, classData, 0, classData.length);
    }
    
    private byte[] getClassData(String className) {
        // 将包名转换为路径
        String path = className.replace('.', File.separatorChar) + ".class";
        try (FileInputStream fis = new FileInputStream(new File(classPath, path));
             ByteArrayOutputStream baos = new ByteArrayOutputStream()) {
            byte[] buffer = new byte[4096];
            int bytesRead;
            while ((bytesRead = fis.read(buffer)) != -1) {
                baos.write(buffer, 0, bytesRead);
            }
            return baos.toByteArray();
        } catch (IOException e) {
            return null;
        }
    }
    
    // 测试
    public static void main(String[] args) throws Exception {
        CustomClassLoader loader = new CustomClassLoader("/Users/demo/classes");
        Class<?> clazz = loader.loadClass("com.example.HelloWorld");
        System.out.println("Class loaded by: " + clazz.getClassLoader());
        Object obj = clazz.getDeclaredConstructor().newInstance();
        System.out.println("Object created: " + obj);
    }
}
```

这个简单的自定义类加载器从指定目录加载 .class 文件。但实际应用中，自定义类加载器通常用于更复杂的场景：

**热部署场景**：在 Web 应用服务器中，当应用重新部署时，旧的类加载器及其加载的所有类都需要被卸载，新的类加载器需要加载新版本的类。由于类的 Class 对象、Method 对象等都与类加载器绑定，更换类加载器就相当于使用了新版本的类。Tomcat、OSGi 等框架都采用这种方式实现热部署。

**类库隔离场景**：在大型应用中，不同模块可能依赖同一类库的不同版本。如果没有类加载器隔离，JVM 只会加载第一个遇到的类版本。使用独立的类加载器加载不同模块，可以实现真正的版本隔离。JDBC 驱动加载就使用了这种模式——每个驱动由对应的类加载器加载，驱动之间互不干扰。

**加密保护场景**：可以将 .class 文件加密，在自定义类加载器中解密后再加载。这样即使用反编译工具拿到字节码，也只能看到加密后的乱码，增加了逆向工程的难度。

## 类加载器的双亲委派模型

类加载器之间不是平等的关系，而是按照严格的父子层次组织的。当一个类加载器收到加载请求时，它首先会委派给自己的父加载器处理，只有父加载器无法完成时，它才会尝试自己加载。这就是著名的"双亲委派模型"（Parents Delegation Model）。

双亲委派模型的核心代码在 `ClassLoader.loadClass(String name, boolean resolve)` 方法中：

```java
protected Class<?> loadClass(String name, boolean resolve) throws ClassNotFoundException {
    // 首先检查类是否已经被加载
    Class<?> c = findLoadedClass(name);
    if (c == null) {
        try {
            // 委派给父加载器
            if (parent != null) {
                c = parent.loadClass(name, false);
            } else {
                // 没有父加载器，委托给 Bootstrap ClassLoader
                c = findBootstrapClassOrNull(name);
            }
        } catch (ClassNotFoundException e) {
            // 父加载器找不到此类
        }
        if (c == null) {
            // 自己尝试加载
            c = findClass(name);
        }
    }
    if (resolve) {
        resolveClass(c);
    }
    return c;
}
```

这个模型的设计有着深远的安全考量。想象这样一个攻击场景：攻击者编写一个恶意类 `java.lang.String`，放在自己应用的类路径下。如果没有双亲委派机制，这个恶意类会被 Application ClassLoader 加载，`java.lang.String` 的安全性就被完全破坏了。有了双亲委派模型，`java.lang.String` 的加载请求会首先委派给 Bootstrap ClassLoader，而 Bootstrap ClassLoader 只会加载它自己目录下的 `java.lang.String`，攻击者的恶意代码永远不会被加载。

另一个重要的设计目的是避免类的重复加载。当父加载器已经加载过某个类时，子加载器不需要再次加载——它可以直接使用父加载器加载的结果。这不仅提高了效率，也保证了类的唯一性。

但双亲委派模型也不是银弹，它也有自己的局限性。例如，JNDI 服务需要由启动类加载器加载，但 JNDI 的实现类可能由厂商提供，这些类需要能够访问由应用类加载器加载的类。JDBC、XML 解析器等都存在类似的需求。Java 通过线程上下文类加载器（Thread Context ClassLoader）来解决这个问题——每个线程都有一个上下文类加载器，默认继承自父线程，主线程的默认是 Application ClassLoader。JNDI 等服务可以通过 `Thread.currentThread().getContextClassLoader()` 获取这个类加载器，加载需要的类。

## 类加载器与命名空间

每个类加载器都有自己独立的命名空间，由它加载的所有类组成。两个类来自同一个命名空间，当且仅当它们由同一个类加载器加载。理解命名空间的概念对于理解类的唯一性和隔离性至关重要。

命名空间的一个重要应用是理解 OSGi 和 Java EE 容器的类加载机制。在这些系统中，每个模块/应用都有自己的类加载器，形成独立的命名空间。A 模块中定义的 `com.example.Service` 和 B 模块中定义的 `com.example.Service` 是两个完全不同的类，它们的方法签名、实现逻辑可以完全不同，模块之间完全隔离。

但这也带来了一个问题：如果 A 模块需要调用 B 模块的类，这种跨命名空间的调用如何实现？通常的做法是通过接口+反射：两个模块共享一组接口定义（由公共类加载器加载），A 模块通过反射创建 B 模块的实现类实例，然后通过接口调用方法。这样既保证了模块隔离，又实现了模块间通信。

## 面试高频考点

**问题一：类加载器的层级结构是怎样的？** 需要按顺序说明：Bootstrap ClassLoader（启动类加载器，负责加载核心类库，C++实现）、Extension ClassLoader（扩展类加载器，负责加载扩展类库，Java实现）、Application ClassLoader（应用类加载器，负责加载应用类路径的类）。

**问题二：什么是双亲委派模型？为什么要使用这个模型？** 双亲委派模型是指类加载器收到加载请求时，首先委派给父加载器处理，只有父加载器无法完成时才自己加载。好处有两个：一是安全性，防止核心类库被篡改；二是避免重复加载，保证类的唯一性。

**问题三：如何打破双亲委派模型？** 可以重写 loadClass 方法，不遵循父加载器优先的策略。Tomcat、JDBC 等都采用了打破双亲委派的策略来实现自己的加载逻辑。

**问题四：ClassLoader 的 findClass 和 loadClass 方法有什么区别？** loadClass 是类加载的入口方法，实现了双亲委派逻辑；findClass 是实际的查找逻辑，由子类重写。正确的自定义类加载器应该重写 findClass，而不是 loadClass。

**问题五：类的主动加载和被动加载有什么区别？** 主动加载会触发类的初始化，被动加载不会。被动加载的场景包括：通过子类引用父类的静态字段（只会触发父类初始化）、通过数组引用类（不会触发类初始化）、通过常量引用类（编译时常量，不触发加载）。

## 总结与展望

类加载器是 JVM 中连接编译时和运行时的桥梁，它的设计体现了 Java 动态性的精髓。从 Bootstrap ClassLoader 到 Application ClassLoader，再到自定义类加载器，每一层都有其独特的职责和适用场景。双亲委派模型是类加载器设计的基石，它保证了 Java 安全性和类的唯一性，但在某些场景下也需要被打破。
