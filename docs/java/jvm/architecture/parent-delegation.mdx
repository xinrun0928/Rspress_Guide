# 双亲委派模型与打破方式

双亲委派模型是 Java 类加载器的核心设计哲学，理解它不仅能帮助我们理解 Java 的类加载机制，更能让我们在遇到类冲突、热部署、模块化等高级话题时有清晰的思路。但任何模型都不是绝对的，了解如何以及为什么要打破双亲委派，同样是高级 Java 开发者必须掌握的知识。

## 双亲委派模型的工作原理

双亲委派模型（Parents Delegation Model）的核心思想很简单：**永远优先使用父加载器加载类，只有父加载器无法完成时，才由自己加载**。这个模型通过 `ClassLoader.loadClass()` 方法实现，其逻辑可以概括为四步：检查是否已加载、尝试父加载器加载、尝试自己加载、返回或抛出异常。

让我们用更直观的方式理解这个模型。想象一个公司的工作流程：当一个任务分配到部门时，部门经理不会自己处理，而是先问处长；处长不会自己处理，而是先问总监；总监不会自己处理，而是问总经理；总经理作为最高负责人，如果能处理就处理，不能处理就一层层返回，最后还是回到部门经理手里处理。这种层层向上汇报、又层层向下执行的方式，就是双亲委派模型的核心。

在 JVM 中，这个模型的具体实现如下：应用程序类加载器收到加载 `java.lang.Object` 的请求后，首先检查是否已加载——没有；然后委派给扩展类加载器；扩展类加载器同样检查——没有，委派给启动类加载器；启动类加载器检查自己的加载记录，发现 `java.lang.Object` 已经在 Bootstrap ClassLoader 的加载范围内，于是加载并返回。应用程序类加载器和扩展类加载器都收到加载成功的通知，就这样，`java.lang.Object` 总是由启动类加载器加载，无论应用程序类路径下有多少个同名类，都不会被加载。

这个模型保证了关键核心类的安全性。如果允许随意覆盖核心类，一个恶意的 `java.lang.String` 类可能导致整个 Java 安全性体系崩溃。同时，它也保证了类的唯一性——无论有多少个类加载器，同一个类只会被加载一次，所有地方引用的都是同一个 Class 对象。

## 双亲委派模型的技术实现

深入到源码层面，双亲委派模型是由 `java.lang.ClassLoader` 类的 `loadClass` 方法实现的：

```java
protected Class<?> loadClass(String name, boolean resolve) throws ClassNotFoundException {
    synchronized (getClassLoadingLock(name)) {
        // 第一步：检查类是否已经被当前类加载器加载过
        Class<?> c = findLoadedClass(name);
        if (c != null) {
            return c;
        }
        
        // 第二步：尝试委派给父类加载器
        ClassLoader parent = this.parent;
        if (parent != null) {
            try {
                c = parent.loadClass(name, false);
            } catch (ClassNotFoundException e) {
                // 父类加载器无法加载此类，这不是错误，只是继续尝试
            }
        } else {
            // parent 为 null，表示已经到了最顶层，委托给 Bootstrap ClassLoader
            c = findBootstrapClassOrNull(name);
        }
        
        // 第三步：父类加载器无法加载，由当前类加载器加载
        if (c == null) {
            c = findClass(name);
        }
        
        // 第四步：如果 resolve 为 true，则解析类
        if (resolve) {
            resolveClass(c);
        }
        return c;
    }
}
```

这个方法有几个关键点值得注意。首先，`findLoadedClass` 是本地方法，直接查询 JVM 内部的已加载类记录，这保证了即使多个线程同时加载同一个类，类也只会被加载一次。其次，`findBootstrapClassOrNull` 也是本地方法，它尝试使用 Bootstrap ClassLoader 加载，如果返回 null 并不会抛出异常，而是继续让子类尝试加载。第三，子类应该重写 `findClass` 方法而不是 `loadClass` 方法，这样才能正确地参与双亲委派流程。

## 为什么要打破双亲委派

双亲委派模型虽然保证了安全性和唯一性，但在某些场景下却成为限制。Java 规范允许打破这个模型，实际上许多知名框架和组件都这样做了。

**Tomcat 的类加载器打破**：Tomcat 是 Java Web 应用服务器的标杆，它必须解决一个根本性问题：同一个 Tomcat 实例可能运行多个 Web 应用，每个应用都依赖不同版本的同一个类库。例如，应用 A 需要 Spring 5.0，应用 B 需要 Spring 3.0，如果只有一套类加载器，只能加载一个版本。Tomcat 为每个 Web 应用创建了独立的 WebAppClassLoader，它的加载顺序是：首先尝试自己加载，自己加载不了才委派给父加载器。这实际上是一个"反双亲委派"模型。更进一步，Tomcat 还实现了类库优先级：WebAppClassLoader -> SharedClassLoader -> CommonClassLoader，允许应用覆盖容器的类库。

**JDBC 的打破**：JDBC（Java Database Connectivity）是 Java 操作数据库的标准 API。当我们使用 `DriverManager.getConnection()` 获取数据库连接时，你有没有想过驱动是怎么加载的？JDBC 驱动实现类（如 MySQL 的 `com.mysql.cj.jdbc.Driver`）放在各厂商的 jar 包中，放在应用的类路径下，应该由 Application ClassLoader 加载。但 `DriverManager` 的代码是由 Bootstrap ClassLoader 加载的。如果不打破双亲委派，`DriverManager` 将无法"看到"由 Application ClassLoader 加载的驱动类。

JDBC 的解决方案是"线程上下文类加载器"：在 `DriverManager` 初始化时，会从当前线程的上下文类加载器开始，依次尝试加载驱动类。应用程序的入口代码通常由 Application ClassLoader 加载，所以当前线程的上下文类加载器就是 Application ClassLoader。这样，即使 `DriverManager` 本身由 Bootstrap ClassLoader 加载，它也能加载并使用应用程序类路径下的 JDBC 驱动。

**JNDI 的打破**：JNDI（Java Naming and Directory Interface）是 Java 的命名和目录服务接口。与 JDBC 类似，JNDI 的接口由 JDK 提供，实现由第三方厂商提供。如果不打破双亲委派，JDK 提供的 JNDI 接口将无法访问厂商实现的类。JNDI 同样使用线程上下文类加载器来解决这个问题。

**OSGi 的打破**：OSGi（Open Service Gateway Initiative）是 Java 模块化标准，每个 OSGi bundle（模块）都有自己的类加载器。OSGi 的类加载策略更加复杂：可以声明导出哪些包供其他 bundle 使用，声明导入哪些包，并指定导入的包的来源。这种精细的类加载控制是通过完全自定义的类加载器实现的，它打破了传统的树形委派结构，采用了网状的类加载关系。

## 打破双亲委派的方式

打破双亲委派主要有以下几种方式：

**方式一：重写 loadClass 方法**

最直接的方式是重写 `ClassLoader.loadClass()` 方法，改变委派逻辑：

```java
public class CustomClassLoader extends ClassLoader {
    @Override
    protected Class<?> loadClass(String name, boolean resolve) throws ClassNotFoundException {
        // 检查是否已经加载
        Class<?> c = findLoadedClass(name);
        if (c != null) {
            return c;
        }
        
        // 判断是否需要打破委派
        // 例如，只对特定包下的类使用自定义加载策略
        if (name.startsWith("com.myapp.")) {
            // 对于 com.myapp 包下的类，尝试自己加载
            byte[] classData = loadClassData(name);
            if (classData != null) {
                return defineClass(name, classData, 0, classData.length);
            }
            // 自己加载失败，尝试父加载器
            return super.loadClass(name, resolve);
        }
        
        // 其他类使用标准的双亲委派
        return super.loadClass(name, resolve);
    }
}
```

但这种方式需要非常小心。如果完全去掉双亲委派逻辑，可能会导致核心类被覆盖，安全风险极大。正确的做法是在保留安全检查的前提下，对特定场景进行定制。

**方式二：使用线程上下文类加载器**

线程上下文类加载器是 Java 引入的一种更优雅的机制。它允许在不修改 JDK 代码的情况下，让高层代码访问低层的类：

```java
public class ContextClassLoaderDemo {
    public static void main(String[] args) {
        // 获取当前线程的上下文类加载器
        ClassLoader contextClassLoader = Thread.currentThread().getContextClassLoader();
        
        // 设置新的上下文类加载器
        ClassLoader newClassLoader = new CustomClassLoader();
        Thread.currentThread().setContextClassLoader(newClassLoader);
        
        // 在新的上下文中执行操作
        doWorkWithNewClassLoader();
    }
}
```

线程上下文类加载器的典型应用场景是 SPI（Service Provider Interface）机制。Java 的许多扩展功能都采用 SPI 实现，例如 JDBC 驱动、XML 解析器、日志框架等。SPI 的核心思想是：接口由 JDK 提供，实现由第三方厂商提供。实现类的 jar 包放在应用的类路径下，由 Application ClassLoader 加载。但接口的代码可能需要加载实现类，如果不做特殊处理，就会遇到"接口看不到实现类"的问题。

`java.util.ServiceLoader` 是 SPI 的标准实现。它的加载逻辑是：使用线程上下文类加载器加载实现类，而不是使用定义接口的类的类加载器。这样，即使用 Bootstrap ClassLoader 加载的接口，也能使用 Application ClassLoader 加载的实现类。

```java
public class ServiceLoader<S> implements Iterable<S> {
    // ...
    public Iterator<S> iterator() {
        return new Iterator<S>() {
            Iterator<ProviderAndType> providers = lazyIterator;
            public S next() {
                return providers.next().provider.get();
            }
        };
    }
    
    private S loadNextService() {
        // 关键：使用线程上下文类加载器
        ClassLoader cl = Thread.currentThread().getContextClassLoader();
        ServiceLoader.Provider<S> p = loadOneInstance(cl);
        // ...
    }
}
```

**方式三：组合使用多个类加载器**

在实际应用中，往往需要组合使用多个类加载器，每个类加载器负责不同的职责：

```java
public class ModularClassLoader extends ClassLoader {
    // 负责加载核心模块
    private ClassLoader coreClassLoader;
    // 负责加载业务模块
    private ClassLoader businessClassLoader;
    // 负责加载第三方库
    private ClassLoader libClassLoader;
    
    @Override
    protected Class<?> findClass(String name) throws ClassNotFoundException {
        // 根据类名判断使用哪个类加载器
        if (name.startsWith("com.myapp.core.")) {
            return coreClassLoader.loadClass(name);
        } else if (name.startsWith("com.myapp.biz.")) {
            return businessClassLoader.loadClass(name);
        } else {
            return libClassLoader.loadClass(name);
        }
    }
}
```

## SPI 与双亲委派的冲突

Service Provider Interface 机制是理解双亲委派限制的经典案例。考虑一个常见的使用场景：应用使用 SLF4J 作为日志门面，具体实现是 Logback。

```java
// SLF4J API（假设由 Bootstrap ClassLoader 加载）
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

public class MyService {
    Logger logger = LoggerFactory.getLogger(MyService.class);
}
```

问题来了：`LoggerFactory` 由 Bootstrap ClassLoader 加载，它需要加载 `org.slf4j.impl.StaticLoggerBinder` 类。这个类在 Logback 的 jar 包中，由 Application ClassLoader 加载。如果不打破双亲委派，`LoggerFactory` 根本看不到 `StaticLoggerBinder`。

解决方案是 `LoggerFactory` 使用线程上下文类加载器：

```java
public class LoggerFactory {
    public static Logger getLogger(Class<?> clazz) {
        // 使用线程上下文类加载器
        ClassLoader cl = Thread.currentThread().getContextClassLoader();
        // 使用 cl 加载 org.slf4j.impl.StaticLoggerBinder
        // ...
    }
}
```

这解释了为什么在某些容器环境中（如 OSGi），如果线程上下文类加载器设置不正确，SPI 机制可能无法正常工作。

## 类加载器的退化

还有一个值得关注的概念：类加载器的退化。当一个类加载器无法找到要加载的类时，它会抛出 `ClassNotFoundException`，但抛出之前，它已经"尽力"了。这个过程可能是这样的：自定义类加载器收到加载请求，委派给父加载器，父加载器继续向上委派，最终到达 Bootstrap ClassLoader。Bootstrap ClassLoader 找不到此类，逐层返回，最后回到自定义类加载器，它尝试自己加载，如果也失败，则抛出异常。

这个层层委派、又层层返回的过程，可能会导致性能问题。为了减少这种开销，JVM 和类库都做了优化：已加载的类会被缓存，`findLoadedClass` 方法会快速判断类是否已经加载。如果类已经加载过，后续的 `loadClass` 调用会直接返回缓存的结果，不会再次执行委派流程。

## 面试高频考点

**问题一：双亲委派模型的工作流程是什么？** 当类加载器收到加载请求时，首先检查类是否已被加载；若没有，则委派给父加载器加载；若父加载器无法加载，则自己尝试加载。这个过程保证了类的唯一性和安全性。

**问题二：为什么要打破双亲委派模型？** 主要原因有三个：一是某些场景需要加载同名但不同版本的类（如 Tomcat 的多应用隔离）；二是 JDK 提供的 SPI 接口需要访问第三方实现类（如 JDBC 驱动）；三是模块化系统需要更精细的类加载控制（如 OSGi）。

**问题三：如何打破双亲委派？** 三种方式：重写 loadClass 方法（直接改变委派逻辑）、使用线程上下文类加载器（间接影响委派结果）、组合使用多个类加载器（针对不同类型的类使用不同策略）。

**问题四：线程上下文类加载器是什么？为什么需要它？** 线程上下文类加载器是每个线程关联的类加载器，默认继承自父线程。它是 JDK 解决 SPI 加载问题的标准方案，使得上层代码可以访问下层实现的类。

**问题五：Tomcat 如何实现类隔离？** Tomcat 为每个 Web 应用创建独立的 WebAppClassLoader，并设置其 parent 为 null（或者设置为 System ClassLoader），从而打破双亲委派。每个 Web 应用使用自己的类加载器加载类，不同应用的同名类互不影响。

**问题六：JDBC 为什么需要打破双亲委派？** JDBC 接口由 Bootstrap ClassLoader 加载（因为它们在 rt.jar 中），但 JDBC 驱动由 Application ClassLoader 加载。如果不打破双亲委派，JDBC 接口看不到驱动实现类。JDBC 通过线程上下文类加载器解决此问题。

## 总结与展望

双亲委派模型是 Java 类加载器的基石，它的设计体现了安全性和唯一性的追求。但任何模型都不是万能的，当面对类隔离、SPI 加载、模块化等更复杂的需求时，打破双亲委派就成了必然选择。

理解双亲委派模型及其打破方式，不仅能帮助我们理解框架的内部机制，更能在遇到类加载相关的疑难问题时快速定位原因。下一篇文章我们将深入探讨类加载器的源码实现，学习如何编写一个真正安全可靠的自定义类加载器。
