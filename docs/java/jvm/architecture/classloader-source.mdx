# ClassLoader 源码解析与自定义类加载器

前面两篇文章我们从宏观层面探讨了类加载器的分类和双亲委派模型，这篇文章让我们深入到源码层面，看看 `ClassLoader` 类是如何实现的，以及如何编写一个功能完善的自定义类加载器。理解源码不仅能帮助我们更好地使用类加载器，更能在排查问题时有的放矢。

## ClassLoader 源码解析

`java.lang.ClassLoader` 是所有类加载器的基类，它的源码设计精巧，充分考虑了扩展性、安全性和性能。让我们逐一分析其核心方法。

### loadClass 方法：委派逻辑的入口

```java
protected Class<?> loadClass(String name, boolean resolve) throws ClassNotFoundException {
    // 使用类名作为锁对象，保证并发场景下类只被加载一次
    synchronized (getClassLoadingLock(name)) {
        // 第一步：检查类是否已经被当前类加载器加载过
        // findLoadedClass 是本地方法，直接查询 JVM 内部的类缓存
        Class<?> c = findLoadedClass(name);
        if (c != null) {
            return c;
        }

        long t0 = System.nanoTime();

        try {
            // 第二步：委派给父加载器
            // parent 字段在构造函数中设置，形成类加载器链
            ClassLoader parent = this.parent;
            if (parent != null) {
                // 递归调用父加载器的 loadClass
                c = parent.loadClass(name, false);
            } else {
                // parent 为 null，说明当前已经是顶层
                // 调用 Bootstrap ClassLoader
                c = findBootstrapClassOrNull(name);
            }
        } catch (ClassNotFoundException e) {
            // 父加载器抛出异常，说明它无法加载此类
            // 注意：这里只是捕获异常，不会立即失败
            // 因为某些场景下，子加载器可能有意愿尝试加载
        }

        // 第三步：如果父加载器无法加载，则由当前类加载器加载
        if (c == null) {
            // 如果仍然没有找到类，说明确实无法加载
            // 调用 findClass 方法，让子类实现具体的查找逻辑
            c = findClass(name);

            // this is the defining class loader; record the stats
            PerfCounter.getParentDelegationTime().addElapsedTimeFrom(t0);
        }

        // 第四步：如果 resolve 为 true，则解析类
        // 解析阶段会解析符号引用为直接引用
        if (resolve) {
            resolveClass(c);
        }
        return c;
    }
}
```

这段代码有几个值得关注的点。首先，`getClassLoadingLock(name)` 方法返回的锁对象是与类名相关的，这允许不同类的加载并行执行，减少了锁竞争。其次，`PerfCounter` 是性能计数器类，用于记录类加载的统计数据，便于 JVM 调优和问题诊断。第三，子类应该重写 `findClass` 方法而不是 `loadClass` 方法，这样才能正确参与双亲委派。

### findClass 方法：子类实现的入口

```java
protected Class<?> findClass(String name) throws ClassNotFoundException {
    // ClassLoader 的默认实现是抛出异常
    // 这意味着：如果没有任何子类重写此方法，
    // 所有 loadClass 请求最终都会失败
    throw new ClassNotFoundException(name);
}
```

`findClass` 方法是子类必须重写的核心方法。它的职责是根据类的全限定名，找到对应的字节码数据，然后调用 `defineClass` 方法将字节码转换为 Class 对象。

### defineClass 方法：字节码到 Class 的转换

```java
protected final Class<?> defineClass(String name, byte[] b, int off, int len)
    throws ClassFormatError {
    // 调用本地方法，将字节数组转换为 Class 对象
    return defineClassInternal(name, b, off, len, null);
}

private native Class<?> defineClassInternal(String name, byte[] b, int off, int len,
                                            ProtectionDomain protectionDomain)
    throws ClassFormatError;
```

`defineClass` 是将字节码转换为 Class 对象的关键方法。它是 `final` 的，不能被子类重写。内部调用了 JVM 的本地方法 `defineClassInternal`，完成实际的转换工作。这个方法会做以下事情：

1. 验证字节码的格式是否正确
2. 在方法区分配内存，存储类的元数据
3. 在堆中创建 Class 对象
4. 返回 Class 对象的引用

### getClassLoadingLock 方法：并发控制的艺术

```java
protected Object getClassLoadingLock(String className) {
    Object lock = this;
    // 如果启用了并行加载（通过 ClassLoader.registerAsParallelCapable() 注册）
    // 则使用专门的锁对象，而不是当前的 ClassLoader 实例作为锁
    if (parallelLockMap != null) {
        // 使用 ConcurrentHashMap 存储类名到锁对象的映射
        // 每个类有自己的锁对象，不同类的加载可以并行
        lock = parallelLockMap.computeIfAbsent(className, k -> new Object());
    }
    return lock;
}
```

这个方法体现了 JVM 对并发性能的考虑。早期版本的 ClassLoader 使用 `synchronized(this)` 来保证同一时刻只有一个线程加载类，但这导致即使加载不同的类也要互斥等待，非常低效。后来引入了并行加载机制，通过 `registerAsParallelCapable()` 方法注册后，每个类使用独立的锁对象，显著提升了并发性能。

## URLClassLoader：标准实现的精髓

`sun.misc.Launcher.URLClassLoader` 是最常用的类加载器实现，Application ClassLoader 和 Extension ClassLoader 都是它的子类。理解 URLClassLoader 的实现，对理解 Java 的类路径机制至关重要。

### URLClassLoader 的构造

```java
public class URLClassLoader extends SecureClassLoader {
    // 存储类搜索路径的 URL 数组
    private final URL[] urls;

    // 父类加载器
    private final URLClassPath ucp;

    public URLClassLoader(URL[] urls, ClassLoader parent) {
        super(parent);
        this.urls = urls;
        // URLClassPath 负责管理 URL 列表和加载资源
        this.ucp = new URLClassPath(urls);
    }
}
```

`URLClassPath` 是 URLClassLoader 的核心组件，它管理了所有的类搜索路径，并能根据路径类型（目录、jar 文件、网络 URL）选择合适的加载策略。

### findResource 和 findResources 方法

```java
public URL findResource(String name) {
    return ucp.findResource(name, false);
}

public Enumeration<URL> findResources(String name) throws IOException {
    return ucp.findResources(name, false);
}
```

这两个方法用于查找资源文件（如配置文件、图片等），而不加载类。`getResource` 和 `getResources` 方法会首先委派给父加载器，如果父加载器找不到，再调用这两个方法。

### 构造 Main ClassLoader

当我们查看 `sun.misc.Launcher` 的源码时，可以发现 Application ClassLoader 和 Extension ClassLoader 是如何被创建的：

```java
public class Launcher {
    private static URLStreamHandlerFactory factory;
    private static Launcher launcher;
    private ClassLoader booster;

    public Launcher() {
        // 首先创建扩展类加载器
        ExtClassLoader extcl;
        try {
            extcl = ExtClassLoader.getExtClassLoader();
        } catch (IOException e) {
            throw new InternalError("Could not create extension class loader", e);
        }

        // 然后创建应用类加载器，将扩展类加载器作为父加载器
        loader = AppClassLoader.getAppClassLoader(extcl);

        // 设置线程上下文类加载器
        Thread.currentThread().setContextClassLoader(loader);
    }
}
```

## 自定义类加载器的最佳实践

现在我们理解了 ClassLoader 的工作原理，接下来实现一个功能完善的自定义类加载器。这个类加载器将支持：指定加载目录、热加载功能、类缓存机制、异常处理。

```java
import java.io.*;
import java.lang.reflect.Method;
import java.security.CodeSource;
import java.security.ProtectionDomain;
import java.util.HashMap;
import java.util.Map;
import java.util.concurrent.ConcurrentHashMap;

/**
 * 支持热加载的自定义类加载器
 * 当类文件发生变化时，自动重新加载类
 */
public class HotSwapClassLoader extends ClassLoader {

    // 类搜索路径
    private final File classPath;

    // 类缓存：类名 -> (类文件最后修改时间, Class对象)
    private final Map<String, CacheEntry> classCache;

    // 是否打印调试信息
    private final boolean debug;

    // 缓存条目
    private static class CacheEntry {
        final long lastModified;
        final Class<?> clazz;
        final byte[] bytecode;

        CacheEntry(long lastModified, Class<?> clazz, byte[] bytecode) {
            this.lastModified = lastModified;
            this.clazz = clazz;
            this.bytecode = bytecode;
        }
    }

    public HotSwapClassLoader(String classPath, boolean debug) throws FileNotFoundException {
        // 设置父加载器为 Application ClassLoader
        super(HotSwapClassLoader.class.getClassLoader().getParent());
        File path = new File(classPath);
        if (!path.exists() || !path.isDirectory()) {
            throw new FileNotFoundException("Class path directory not found: " + classPath);
        }
        this.classPath = path;
        this.classCache = new ConcurrentHashMap<>();
        this.debug = debug;

        // 注册为支持并行加载
        registerAsParallelCapable();
    }

    public HotSwapClassLoader(String classPath) throws FileNotFoundException {
        this(classPath, false);
    }

    @Override
    protected Class<?> findClass(String name) throws ClassNotFoundException {
        if (debug) {
            System.out.println("[HotSwapClassLoader] findClass: " + name);
        }

        // 检查是否需要热加载
        Class<?> clazz = checkForHotSwap(name);
        if (clazz != null) {
            return clazz;
        }

        // 正常加载流程
        byte[] bytecode = loadClassData(name);
        if (bytecode == null) {
            throw new ClassNotFoundException(name);
        }

        // 定义类
        return defineClass(name, bytecode, 0, bytecode.length);
    }

    /**
     * 检查是否需要热加载类
     */
    private Class<?> checkForHotSwap(String name) {
        String fileName = name.replace('.', File.separatorChar) + ".class";
        File classFile = new File(classPath, fileName);

        if (!classFile.exists()) {
            // 文件不存在，移除缓存（如果有）
            classCache.remove(name);
            return null;
        }

        long currentModified = classFile.lastModified();

        // 检查缓存
        CacheEntry entry = classCache.get(name);
        if (entry != null && entry.lastModified == currentModified) {
            // 文件未修改，返回缓存的类
            if (debug) {
                System.out.println("[HotSwapClassLoader] Using cached: " + name);
            }
            return entry.clazz;
        }

        // 文件已修改或有新的类需要加载
        if (debug) {
            System.out.println("[HotSwapClassLoader] Loading/Reloading: " + name);
        }

        try {
            byte[] bytecode = loadClassDataFromFile(classFile);
            if (bytecode == null) {
                return null;
            }

            // 对于重新加载的类，使用新的类加载器实例来加载
            // 这是热加载的关键：旧版本的类不会被新代码"污染"

            // 首先检查是否是由当前类加载器加载的
            Class<?> existing = findLoadedClass(name);
            if (existing != null) {
                // 如果是已经加载的类，不能重新定义
                // 需要使用新的类加载器实例
                if (debug) {
                    System.out.println("[HotSwapClassLoader] Class already loaded, " +
                            "needs new loader instance for hot swap");
                }
                return existing;
            }

            // 定义新类
            Class<?> newClass = defineClass(name, bytecode, 0, bytecode.length);

            // 更新缓存
            classCache.put(name, new CacheEntry(currentModified, newClass, bytecode));

            return newClass;

        } catch (IOException e) {
            if (debug) {
                System.out.println("[HotSwapClassLoader] Error loading: " + name + ", " + e);
            }
            return null;
        }
    }

    /**
     * 从文件加载字节码
     */
    private byte[] loadClassData(String name) {
        String fileName = name.replace('.', File.separatorChar) + ".class";
        File classFile = new File(classPath, fileName);

        if (!classFile.exists()) {
            return null;
        }

        try {
            return loadClassDataFromFile(classFile);
        } catch (IOException e) {
            if (debug) {
                System.out.println("[HotSwapClassLoader] Error reading: " + classFile);
            }
            return null;
        }
    }

    private byte[] loadClassDataFromFile(File file) throws IOException {
        try (FileInputStream fis = new FileInputStream(file);
             ByteArrayOutputStream baos = new ByteArrayOutputStream()) {
            byte[] buffer = new byte[8192];
            int bytesRead;
            while ((bytesRead = fis.read(buffer)) != -1) {
                baos.write(buffer, 0, bytesRead);
            }
            return baos.toByteArray();
        }
    }

    /**
     * 清除所有缓存的类
     */
    public void clearCache() {
        classCache.clear();
    }

    /**
     * 重新加载指定的类
     */
    public Class<?> reloadClass(String name) throws ClassNotFoundException {
        classCache.remove(name);
        return loadClass(name);
    }
}
```

## 热加载的完整示例

上面的类加载器支持热加载，但真正的热加载还需要一些额外的代码。关键点在于：当一个类被修改后，需要使用新的类加载器实例来加载它，而不是重用旧的实例。这是因为 Class 对象与其加载的类加载器紧密绑定，旧实例持有的旧类引用不会自动更新。

```java
import java.io.File;
import java.lang.reflect.Method;

/**
 * 热加载管理器
 * 负责管理多个类加载器实例，实现真正的类热加载
 */
public class HotSwapManager {

    private final String classPath;
    private ClassLoader currentLoader;
    private Class<?> currentServiceClass;

    public HotSwapManager(String classPath) throws Exception {
        this.classPath = classPath;
        this.currentLoader = createNewClassLoader();
        loadInitialClass();
    }

    private ClassLoader createNewClassLoader() {
        // 每次都创建新的类加载器实例
        // 这是实现热加载的关键
        return new HotSwapClassLoader(classPath);
    }

    private void loadInitialClass() throws Exception {
        String className = "com.example.HotSwapService";
        currentServiceClass = Class.forName(className, true, currentLoader);
    }

    /**
     * 检测并重新加载类（如果文件已修改）
     */
    public void checkAndReload() throws Exception {
        String className = "com.example.HotSwapService";
        String fileName = className.replace('.', File.separatorChar) + ".class";
        File classFile = new File(classPath, fileName);

        if (classFile.exists() && hasChanged(classFile)) {
            System.out.println("Detected class change, reloading...");

            // 创建新的类加载器
            ClassLoader newLoader = createNewClassLoader();

            // 使用新类加载器加载类
            Class<?> newClass = Class.forName(className, true, newLoader);

            // 更新引用
            this.currentLoader = newLoader;
            this.currentServiceClass = newClass;

            System.out.println("Reload completed.");
        }
    }

    private boolean hasChanged(File classFile) {
        // 简单实现：每次都返回 true，实际应该检查文件修改时间
        return true;
    }

    /**
     * 调用服务方法
     */
    public void execute() throws Exception {
        if (currentServiceClass != null) {
            Method mainMethod = currentServiceClass.getMethod("execute");
            mainMethod.invoke(null);
        }
    }

    public static void main(String[] args) throws Exception {
        HotSwapManager manager = new HotSwapManager("/path/to/classes");

        // 模拟持续运行和检测
        while (true) {
            manager.checkAndReload();
            manager.execute();
            Thread.sleep(5000); // 每5秒检测一次
        }
    }
}
```

## 类加载器的常见问题与排查

理解类加载器的源码后，我们可以更好地诊断和解决相关问题。

**问题一：NoClassDefFoundError vs ClassNotFoundException**

这两个异常常常被混淆。`ClassNotFoundException` 是检查时异常，当 `Class.forName()`、`ClassLoader.loadClass()` 等方法找不到类时抛出，通常是因为类不在类路径中。`NoClassDefFoundError` 是错误，当类在编译时存在但运行时找不到时抛出，通常是因为静态初始化失败、类文件被删除、或者类加载器不一致。

**问题二：LinkageError**

`LinkageError` 发生在类已经被加载后，再次尝试加载不兼容版本的同一类时。这在容器环境（如 Tomcat）中很常见，当应用和容器都依赖同一个库的不同版本时。解决方案是确保类库版本一致，或者使用正确的类加载器隔离。

**问题三：ClassCastException**

即使两个对象类型看起来相同，`ClassCastException` 也可能发生。这是因为它们是由不同的类加载器加载的。在类加载器隔离的场景下，同一个类名被不同的类加载器加载后，得到的 Class 对象不兼容。

## 面试高频考点

**问题一：loadClass 和 findClass 方法的区别是什么？**

`loadClass` 是类加载的入口方法，包含了委派逻辑，定义了类加载的整体流程；`findClass` 是具体实现类需要重写的方法，负责根据类名查找字节码。正确的方式是重写 findClass 而不是 loadClass。

**问题二：defineClass 有什么限制？**

`defineClass` 是 final 方法，不能被子类重写。它负责将字节数组转换为 Class 对象，这个过程涉及 JVM 内部的类解析和验证，是不透明的关键步骤。

**问题三：如何实现热加载？**

热加载的关键是：当类文件修改后，创建新的类加载器实例来加载新版本的类。旧的类加载器及其加载的类会在不再被引用后被垃圾回收。注意，同一个类加载器实例不能重复定义同一个类。

**问题四：并行加载是如何实现的？**

通过 `registerAsParallelCapable()` 方法注册后，每个类的加载使用独立的锁对象，而不是整个类加载器实例作为锁。这允许多个线程同时加载不同的类，提升并发性能。

**问题五：类加载器的父子关系是如何建立的？**

父类加载器是在构造函数中通过 `super(parent)` 传入的。这种关系是组合关系而不是继承关系，子类加载器"持有"父类加载器的引用，但父类加载器"不知道"子类的存在。

通过对 `ClassLoader` 源码的深入分析，我们理解了类加载器的核心设计：委派模型保证了安全性和唯一性，`findClass` 和 `defineClass` 的分离设计提供了良好的扩展性，并行加载机制提升了并发性能。掌握了这些知识，我们就有了编写高质量自定义类加载器的能力。
