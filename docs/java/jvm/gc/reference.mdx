# 四种引用：强引用、软引用、弱引用、虚引用

Java 从 JDK 1.2 开始，引入了四种引用类型的概念，这四种引用代表了对象从"活着"到"死了但还能被利用"的不同状态。理解这四种引用，是理解 Java 内存管理、设计高效缓存、以及排查内存问题的重要基础。在实际的 Java 应用中，合理使用软引用和弱引用，可以让程序在内存紧张时自动回收无用对象，避免 OOM 的发生。

## 为什么需要不同的引用类型

<Badge>Reference Type</Badge>

在传统的 Java 内存管理中，对象只有两种状态：被引用和不被引用。被引用就存活，不被引用就死亡。这种二元划分对于大多数场景是够用的，但它过于粗糙，无法表达"想要保留但可以牺牲"或"想观察回收但不想阻止"等更精细的需求。

想象一个图片缓存的场景：如果缓存中的图片都是强引用，那即使内存紧张，图片也不会被回收，最终可能导致 OOM。但如果使用软引用，内存紧张时缓存图片可以被回收，程序仍然可以正常运行，只是需要重新加载图片。这就是软引用的价值——表达"非必需但有价值"的对象。

弱引用的应用场景有所不同。比如，在使用 `ThreadLocal` 时，每个线程都有一个 Map 存储线程本地的变量。如果用强引用存储这些变量，即使线程已经结束不再使用，这些变量也无法被回收，导致内存泄漏。使用弱引用，线程结束时 Map 中的弱引用可以被回收，变量也随之被清理。

虚引用是更特殊的存在，它完全不影响对象的生命周期。虚引用的唯一作用是在对象被回收时收到通知。这种机制可以用于精确控制对象的销毁时机。

## 强引用：最普通的引用

<Badge>Strong Reference</Badge>

强引用（Strong Reference）是 Java 中最常见的引用类型，也是默认的引用方式。当我们写 `Object obj = new Object()` 时，创建的就是强引用。只要强引用还存在，垃圾回收器就永远不会回收被引用的对象。

```java
Object obj = new Object();  // 强引用
obj = null;  // 取消强引用，现在 Object 实例可以被回收
```

强引用是内存泄漏的主要原因之一。当对象被错误地持有强引用，即使它们已经不再被需要，也无法被垃圾回收。这种情况在静态集合中尤为常见——如果一个静态集合持有了大量不再使用的对象引用，内存会持续增长。

```java
public class MemoryLeakDemo {
    // 静态集合持有强引用，可能导致内存泄漏
    private static final List<Object> CACHE = new ArrayList<>();
    
    public static void addToCache(Object obj) {
        CACHE.add(obj);  // 即使 obj 不再被其他地方使用，仍然无法回收
    }
}
```

## 软引用：内存敏感的最后防线

<Badge>Soft Reference</Badge>

软引用（Soft Reference）描述一些还有用但非必需的对象。在内存足够时，软引用对象不会被回收；只有在内存快要耗尽、即将发生 OOM 之前，软引用对象才会被回收。

软引用非常适合用于实现内存敏感的缓存。当内存充足时，缓存命中率高，性能好；当内存紧张时，缓存对象被自动回收，释放内存，避免 OOM。

```java
import java.lang.ref.SoftReference;

public class SoftReferenceCache<K, V> {
    private final Map<K, SoftReference<V>> cache = new HashMap<>();
    
    public V get(K key) {
        SoftReference<V> ref = cache.get(key);
        if (ref == null) {
            return null;
        }
        V value = ref.get();  // 可能被 GC 回收，返回 null
        if (value == null) {
            cache.remove(key);  // 清理空引用
        }
        return value;
    }
    
    public void put(K key, V value) {
        cache.put(key, new SoftReference<>(value));
    }
}
```

软引用的回收策略由 JVM 实现决定。在 HotSpot JVM 中，软引用对象在 OutOfMemoryError 之前被回收。可以通过 `-XX:SoftRefLRUPolicyMSPerMB` 参数调整软引用的回收时机。

## 弱引用：比软引用更弱

<Badge>Weak Reference</Badge>

弱引用（Weak Reference）比软引用更弱。无论内存是否充足，下一次垃圾回收时，弱引用对象都会被回收。弱引用持有的对象，在下一次 GC 时必定被清除。

弱引用的典型应用是 `WeakHashMap`。`WeakHashMap` 的 Entry 继承自 WeakReference，当 key 不再被其他强引用持有时，即使 Entry 还在 Map 中，下一次 GC 时 Entry 也会被自动清除。

```java
import java.lang.ref.WeakReference;

public class WeakReferenceDemo {
    public static void main(String[] args) {
        // 创建弱引用
        WeakReference<byte[]> ref = new WeakReference<>(new byte[1024 * 1024]);
        
        System.out.println("GC 前: " + ref.get() + " 不为 null");
        
        // 手动触发 GC
        System.gc();
        
        // 弱引用对象被回收
        System.out.println("GC 后: " + ref.get() + " 可能为 null");
    }
}
```

ThreadLocal 是弱引用的经典应用场景。ThreadLocalMap 中 Entry 的 key 是 ThreadLocal 对象的弱引用，value 是实际值的强引用。当 ThreadLocal 对象不再被外部强引用时，下一次 GC 时 ThreadLocal 对象被清除，Entry 的 key 变为 null。但由于 value 仍然被 Entry 持有，value 不会被回收——这就是 ThreadLocal 内存泄漏的根本原因。

```java
// ThreadLocal 内部实现简化
static class ThreadLocalMap {
    static class Entry extends WeakReference<ThreadLocal<?>> {
        Object value;  // 强引用
        
        Entry(ThreadLocal<?> k, Object v) {
            super(k);  // 弱引用 key
            value = v;  // 强引用 value
        }
    }
}
```

正确的 ThreadLocal 使用方式是：在使用完毕后调用 `remove()` 方法清除 Entry，避免 value 无法回收导致的内存泄漏。

## 虚引用：死后的观察者

<Badge>Phantom Reference</Badge>

虚引用（Phantom Reference）是最弱的引用类型，也称为幽灵引用或幻影引用。一个对象是否有虚引用的存在，完全不会影响它的生命周期。虚引用不能通过 `get()` 方法获取被引用的对象——它始终返回 null。

虚引用的唯一作用是在对象被垃圾收集器回收时，得到一个系统通知。虚引用必须与 ReferenceQueue 联合使用。当垃圾回收器准备回收一个对象时，如果发现它还有虚引用，会在回收之前把这个虚引用加入 ReferenceQueue。

```java
import java.lang.ref.PhantomReference;
import java.lang.ref.ReferenceQueue;

public class PhantomReferenceDemo {
    public static void main(String[] args) {
        ReferenceQueue<Object> queue = new ReferenceQueue<>();
        Object obj = new Object();
        
        // 创建虚引用，必须关联 ReferenceQueue
        PhantomReference<Object> ref = new PhantomReference<>(obj, queue);
        
        // 虚引用的 get() 总是返回 null
        System.out.println(ref.get());  // null
        
        // 取消强引用
        obj = null;
        
        // 触发 GC
        System.gc();
        
        // 虚引用被加入 ReferenceQueue
        try {
            // 带超时地等待 ReferenceQueue 中的引用
            Reference<?> r = queue.remove(1000);
            if (r != null) {
                System.out.println("对象被回收，虚引用进入 ReferenceQueue");
            }
        } catch (InterruptedException e) {}
    }
}
```

虚引用的应用场景包括：精确控制对象的销毁时机、在对象回收前进行资源清理、追踪对象的回收活动。虚引用最常见的用途是在 NIO 中管理直接内存——DirectByteBuffer 对象持有虚引用到 Cleaner，当 DirectByteBuffer 被回收时，Cleaner 会被加入 ReferenceQueue，后台线程调用 Cleaner 释放直接内存。

## ReferenceQueue：引用的回收站

<Badge>Reference Queue</Badge>

ReferenceQueue 是四种引用类型共用的一个重要组件。它用于存储待处理的引用对象，垃圾收集器会在适当的时机将引用加入队列。

```java
import java.lang.ref.Reference;
import java.lang.ref.ReferenceQueue;

public class ReferenceQueueDemo {
    public static void main(String[] args) throws InterruptedException {
        ReferenceQueue<Object> queue = new ReferenceQueue<>();
        
        // 创建软引用、弱引用或虚引用时关联 ReferenceQueue
        SoftReference<Object> softRef = new SoftReference<>(new Object(), queue);
        WeakReference<Object> weakRef = new WeakReference<>(new Object(), queue);
        
        // 清除引用对象
        System.gc();
        
        // 轮询 ReferenceQueue
        Reference<Object> ref;
        while ((ref = queue.remove(1000)) != null) {
            if (ref == softRef) {
                System.out.println("软引用被回收");
            } else if (ref == weakRef) {
                System.out.println("弱引用被回收");
            }
            // 处理虚引用的逻辑类似
        }
    }
}
```

## WeakHashMap 实战：缓存的优雅实现

<Badge>WeakHashMap</Badge>

WeakHashMap 是四种引用最直接的应用之一。它是一种 Map 实现，其中的键（Key）是弱引用的。当 key 不再被其他强引用持有时，即使 Entry 仍然在 Map 中，也会在下一次 GC 时被自动清除。

```java
import java.util.Map;
import java.util.WeakHashMap;

public class CacheDemo {
    // 使用 WeakHashMap 实现自动清理的缓存
    private static final Map<String, ExpensiveObject> CACHE = new WeakHashMap<>();
    
    public static ExpensiveObject get(String key) {
        return CACHE.get(key);
    }
    
    public static void put(String key, ExpensiveObject value) {
        CACHE.put(key, value);
    }
    
    public static void main(String[] args) {
        // 当 key 不再被使用时，Entry 会在 GC 时自动清除
        String key1 = new String("key1");  // new String 确保不在字符串常量池
        put(key1, new ExpensiveObject());
        
        // key1 不再被外部引用
        key1 = null;
        
        // 触发 GC
        System.gc();
        
        // key1 对应的 Entry 可能已被清除
    }
}
```

WeakHashMap 适合用于缓存场景，其中 key 的生命周期应该与对象的生命周期一致。比如，在缓存 Web 请求的响应时，使用 URL 作为 key，当 URL 对象不再被使用时，缓存 Entry 可以自动清除。

## 面试高频考点

**问题一：四种引用的区别是什么？** 强引用是普通引用，永远不会被回收；软引用在内存不足时回收；弱引用在下次 GC 时回收；虚引用不影响生命周期，仅用于接收回收通知。

**问题二：软引用适合什么场景？** 软引用适合实现内存敏感的缓存。在内存充足时缓存命中率高，内存紧张时自动回收，避免 OOM。

**问题三：ThreadLocal 为什么可能导致内存泄漏？** ThreadLocalMap 中 Entry 的 key 是弱引用，但 value 是强引用。当 ThreadLocal 对象不再被外部引用时，key 被清除，但 value 仍然被 Entry 持有，无法回收。解决方法是在使用完毕后调用 `remove()`。

**问题四：虚引用的 get() 为什么总是返回 null？** 虚引用的设计目的就是不影响对象的生命周期，无法通过虚引用获取对象。虚引用的唯一作用是在对象回收时收到通知。

**问题五：ReferenceQueue 的作用是什么？** ReferenceQueue 用于接收被垃圾回收的引用对象。软引用、弱引用、虚引用可以关联 ReferenceQueue，当引用的对象被回收时，引用本身会被加入队列，供程序处理。

**问题六：DirectByteBuffer 如何释放直接内存？** DirectByteBuffer 对象持有虚引用到 Cleaner。当 DirectByteBuffer 被 GC 时，虚引用进入 ReferenceQueue，ReferenceHandler 线程调用 Cleaner 释放直接内存。

四种引用类型就像给对象设置了不同的"存活等级"，从"永不死亡"到"随时可回收"。理解它们的区别和应用场景，能帮助我们设计出更优雅、更高效的 Java 程序。那么，如果让你设计一个缓存系统，你会选择哪种引用类型来实现？为什么？
