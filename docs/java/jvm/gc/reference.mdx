# 四种引用：强引用、软引用、弱引用、虚引用

Java 从 JDK 1.2 开始，引入了四种引用类型的概念，这四种引用代表了对象从"活着"到"死了但还能被利用"的不同状态。理解这四种引用，是理解 Java 内存管理、设计高效缓存、以及排查内存问题的重要基础。在实际的 Java 应用中，合理使用软引用和弱引用，可以让程序在内存紧张时自动回收无用对象，避免 OOM 的发生。

## 为什么需要不同的引用类型

|引用类型|

在传统的 Java 内存管理中，对象只有两种状态：被引用和不被引用。被引用就存活，不被引用就死亡。这种二元划分对于大多数场景是够用的，但它过于粗糙，无法表达"想要保留但可以牺牲"或"想观察回收但不想阻止"等更精细的需求。

想象一个图片缓存的场景：如果缓存中的图片都是强引用，那即使内存紧张，图片也不会被回收，最终可能导致 OOM。但如果使用软引用，内存紧张时缓存图片可以被回收，程序仍然可以正常运行，只是需要重新加载图片。这就是软引用的价值——表达"非必需但有价值"的对象。

弱引用的应用场景有所不同。比如，在使用 `ThreadLocal` 时，每个线程都有一个 Map 存储线程本地的变量。如果用强引用存储这些变量，即使线程已经结束不再使用，这些变量也无法被回收，导致内存泄漏。使用弱引用，线程结束时 Map 中的弱引用可以被回收，变量也随之被清理。

虚引用是更特殊的存在，它完全不影响对象的生命周期。虚引用的唯一作用是在对象被回收时收到通知。这种机制可以用于精确控制对象的销毁时机。

## 强引用：最普通的引用

|强引用|

强引用（Strong Reference）是 Java 中最常见的引用类型，也是默认的引用方式。当我们写 `Object obj = new Object()` 时，创建的就是强引用。只要强引用还存在，垃圾回收器就永远不会回收被引用的对象。

```java
Object obj = new Object();  // 强引用
obj = null;  // 取消强引用，现在 Object 实例可以被回收
```

强引用是内存泄漏的主要原因之一。当对象被错误地持有强引用，即使它们已经不再被需要，也无法被垃圾回收。这种情况在静态集合中尤为常见——如果一个静态集合持有了大量不再使用的对象引用，内存会持续增长。

```java
public class MemoryLeakDemo {
    // 静态集合持有强引用，可能导致内存泄漏
    private static final List<Object> CACHE = new ArrayList<>();
    
    public static void addToCache(Object obj) {
        CACHE.add(obj);  // 即使 obj 不再被其他地方使用，仍然无法回收
    }
}
```

## 软引用：内存敏感的最后防线

|软引用|

软引用（Soft Reference）描述一些还有用但非必需的对象。在内存足够时，软引用对象不会被回收；只有在内存快要耗尽、即将发生 OOM 之前，软引用对象才会被回收。

软引用非常适合用于实现内存敏感的缓存。当内存充足时，缓存命中率高，性能好；当内存紧张时，缓存对象被自动回收，释放内存，避免 OOM。

```java
import java.lang.ref.SoftReference;

public class SoftReferenceCache<K, V> {
    private final Map<K, SoftReference<V>> cache = new HashMap<>();
    
    public V get(K key) {
        SoftReference<V> ref = cache.get(key);
        if (ref == null) {
            return null;
        }
        V value = ref.get();  // 可能被 GC 回收，返回 null
        if (value == null) {
            cache.remove(key);  // 清理空引用
        }
        return value;
    }
    
    public void put(K key, V value) {
        cache.put(key, new SoftReference<>(value));
    }
}
```

软引用的回收策略由 JVM 实现决定。在 HotSpot JVM 中，软引用对象在 OutOfMemoryError 之前被回收。可以通过 `-XX:SoftRefLRUPolicyMSPerMB` 参数调整软引用的回收时机。

```bash
# 软引用在每次 GC 后检查是否需要回收
# -XX:SoftRefLRUPolicyMSPerMB=1000 表示每 MB 空闲空间保留 1000ms
java -XX:SoftRefLRUPolicyMSPerMB=1000 -Xmx256m MyApplication
```

## 弱引用：比软引用更弱

|弱引用|

弱引用（Weak Reference）比软引用更弱。无论内存是否充足，下一次垃圾回收时，弱引用对象都会被回收。弱引用持有的对象，在下一次 GC 时必定被清除。

弱引用的典型应用是 `WeakHashMap`。`WeakHashMap` 的 Entry 继承自 WeakReference，当 key 不再被其他强引用持有时，即使 Entry 还在 Map 中，下一次 GC 时 Entry 也会被自动清除。

```java
import java.lang.ref.WeakReference;

public class WeakReferenceDemo {
    public static void main(String[] args) {
        // 创建弱引用
        WeakReference<byte[]> ref = new WeakReference<>(new byte[1024 * 1024]);
        
        System.out.println("GC 前: " + ref.get() + " 不为 null");
        
        // 手动触发 GC
        System.gc();
        
        // 弱引用对象被回收
        System.out.println("GC 后: " + ref.get() + " 可能为 null");
    }
}
```

WeakHashMap 的经典应用场景是缓存元数据。例如，存储类加载器与类信息的映射：

```java
WeakHashMap<ClassLoader, ClassInfo> classInfoMap = new WeakHashMap<>();

// 当 ClassLoader 不再被使用时，即使仍然在 Map 中，
// 下次 GC 时 Entry 也会被自动清除，避免类加载器泄漏
```

## 虚引用：幽灵般的存在

|虚引用|

虚引用（Phantom Reference）是 Java 中最弱的引用类型，完全不影响对象的生命周期。虚引用的 `get()` 方法永远返回 null，这也是它与其他引用的根本区别。

虚引用的唯一作用是在对象被回收时收到一个通知。这个机制叫做引用队列（Reference Queue）机制。当对象被回收时，JVM 会将虚引用对象本身（不是它引用的对象）加入关联的引用队列。我们可以通过监控引用队列来追踪对象的回收。

```java
import java.lang.ref.PhantomReference;
import java.lang.ref.ReferenceQueue;

public class PhantomReferenceDemo {
    public static void main(String[] args) {
        // 创建引用队列
        ReferenceQueue<Object> queue = new ReferenceQueue<>();
        
        // 创建虚引用
        Object obj = new Object();
        PhantomReference<Object> phantomRef = new PhantomReference<>(obj, queue);
        
        // obj 可以被回收了
        obj = null;
        
        System.out.println("虚引用的 get() 永远返回 null: " + phantomRef.get());
        
        // 手动触发 GC
        System.gc();
        
        // 等待垃圾回收
        try {
            Thread.sleep(1000);
        } catch (InterruptedException e) {
            e.printStackTrace();
        }
        
        // 检查引用队列
        System.out.println("虚引用被加入队列: " + (queue.poll() != null));
    }
}
```

虚引用的典型应用是精确定位对象的销毁时机，比如用于 NIO 中的直接内存（DirectByteBuffer）回收。当直接内存不再使用时，虚引用机制可以确保堆外内存被及时释放。

## 四种引用的对比总结

|引用类型 | 回收时机 | `get()` 返回值 | 典型应用 |
|---|---|---|---|
| 强引用 | 永不回收（除非手动置 null） | 对象本身 | 普通对象引用 |
| 软引用 | 内存不足时回收 | 对象本身或 null | 内存敏感缓存 |
| 弱引用 | 下次 GC 时回收 | 对象本身或 null | WeakHashMap、ThreadLocal |
| 虚引用 | 对象回收后加入队列 | 永远 null | 对象销毁追踪、直接内存管理 |

```java
import java.lang.ref.PhantomReference;
import java.lang.ref.ReferenceQueue;
import java.lang.ref.SoftReference;
import java.lang.ref.WeakReference;

public class ReferenceComparison {
    public static void main(String[] args) {
        Object obj = new Object();
        
        // 强引用
        StrongRef strongRef = new StrongRef(obj);
        
        // 软引用
        SoftReference<Object> softRef = new SoftReference<>(obj);
        
        // 弱引用
        WeakReference<Object> weakRef = new WeakReference<>(obj);
        
        // 虚引用
        ReferenceQueue<Object> queue = new ReferenceQueue<>();
        PhantomReference<Object> phantomRef = new PhantomReference<>(obj, queue);
        
        // 取消强引用
        obj = null;
        
        System.gc();
        
        // 强引用对象仍存活（如果有强引用指向）
        // 软引用对象取决于内存情况
        // 弱引用对象已被回收
        // 虚引用对象被回收，加入队列
    }
    
    static class StrongRef {
        private final Object referent;
        StrongRef(Object obj) { this.referent = obj; }
    }
}
```

## ThreadLocal 中的引用实战

理解四种引用，最经典的案例是 `ThreadLocal`。`ThreadLocal` 的内部实现完美展示了弱引用的价值。

```java
public class ThreadLocalDemo {
    static final ThreadLocal<User> userThreadLocal = new ThreadLocal<>();
    
    public static void main(String[] args) {
        // 设置用户信息
        userThreadLocal.set(new User("张三", 25));
        
        // 获取用户信息
        User user = userThreadLocal.get();
        System.out.println(user);
        
        // 手动清除
        userThreadLocal.remove();
        
        // 如果不手动清除，当线程结束时：
        // ThreadLocalMap.Entry 持有对 ThreadLocal 的弱引用
        // ThreadLocal 本身可以被回收
        // 但如果 Entry 持有的是强引用，则会导致内存泄漏
    }
}

class User {
    String name;
    int age;
    User(String name, int age) { this.name = name; this.age = age; }
}
```

`ThreadLocal` 的 Entry 继承自 `WeakReference<ThreadLocal<?>>`，使用弱引用持有 ThreadLocal 对象。这是 JVM 设计者精心考虑的结果：如果 Entry 持有强引用，即使 ThreadLocal 不再被使用，Entry 和它持有的值也无法被回收，导致内存泄漏。

但弱引用只是解决方案的一部分。即使 ThreadLocal 对象被回收了，Entry 的值仍然持有强引用，如果不调用 `remove()`，这个值仍然无法被回收。这就是为什么 `ThreadLocal` 内存泄漏的场景时有发生。
