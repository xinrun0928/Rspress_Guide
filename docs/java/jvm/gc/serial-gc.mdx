# Serial / Serial Old：单线程收集器

Serial 收集器是 JVM 历史上最早的垃圾收集器，也是最简单、最基础的选择。它使用单线程进行垃圾回收，在回收期间暂停所有应用线程。虽然 Serial 收集器在今天看来效率不高，但它简洁的实现和低内存开销使其在某些场景下仍然有价值。理解 Serial 收集器的工作原理，是理解其他更复杂收集器的基础。

## Serial 收集器的工作原理

|Serial GC|

Serial 收集器如其名字所示，使用单线程进行垃圾回收。当 Serial 收集器工作时，它会暂停所有应用线程（Stop-The-World），然后在后台使用唯一的 GC 线程执行垃圾回收任务。

Serial 收集器采用复制算法回收年轻代。在 Minor GC 时，Serial 收集器将 Eden 区和当前 Survivor 区中存活的对象复制到另一个 Survivor 区，年龄达标的对象晋升到老年代。复制完成后，Eden 和原来的 Survivor 区被清空。

Serial Old 收集器是 Serial 收集器的老年代版本，使用标记-整理算法。在 Full GC 时，Serial Old 收集器首先标记所有存活对象，然后将存活对象向一端移动，最后清除边界外的内存。

```
Serial GC 工作示意:

┌─────────────────────────────────────────┐
│           Stop-The-World                 │
│                                          │
│    应用线程暂停 ←──── GC 线程执行        │
│    (所有线程)      (单线程回收)          │
│                                          │
└─────────────────────────────────────────┘
```

## Serial 收集器的特点

Serial 收集器有几个显著的特点。第一是简单高效。Serial 的实现非常简洁，没有复杂的并发控制机制，GC 线程的额外开销很小。对于单核 CPU 或极小内存的场景，Serial 的效率可能反而更高。

第二是内存开销小。Serial 只需要一个 GC 线程，不需要额外的线程同步开销。对于内存受限的环境（如嵌入式系统），Serial 是合理的选择。

第三是停顿时间长。由于是单线程串行执行，Serial 的停顿时间与堆大小和存活对象数量成正比。在大内存场景下，停顿时间可能达到数秒。

```bash
# 启用 Serial 收集器
java -XX:+UseSerialGC -Xms256m -Xmx256m MyApplication

# 启用 Serial GC + PrintGCDetails 查看 GC 日志
java -XX:+UseSerialGC -XX:+PrintGCDetails -Xms256m -Xmx256m MyApplication
```

## Serial 收集器的适用场景

|Client VM|

Serial 收集器并非一无是处，在某些场景下它是合理甚至最优的选择。

第一种场景是客户端模式（Client VM）。在 JDK 8 及之前的客户端 JVM 中，Serial 是默认的垃圾收集器。这是因为客户端应用的内存通常较小，停顿时间的影响不明显，而 Serial 的低开销和简单性更适合客户端环境。

第二种场景是单核 CPU 环境。在单核 CPU 上，Serial 的多线程版本（ParNew、Parallel Scavenge）反而需要额外的线程切换开销，效率不如 Serial。

第三种场景是极小内存环境。对于几十 MB 的堆内存，Serial 的简单性优势明显，停顿时间也在可接受范围内。

第四种场景是资源受限的嵌入式系统。嵌入式系统的内存和 CPU 资源有限，Serial 的低开销使其成为合理选择。

```java
// Serial 收集器的典型应用场景
public class ClientApplication {
    public static void main(String[] args) {
        // 客户端应用通常内存需求不大
        // Serial 收集器足以应对
        System.out.println("Running with Serial GC");
    }
}
```

## Serial 收集器的配置参数

Serial 收集器有几个重要的配置参数：

`-XX:+UseSerialGC`：启用 Serial 收集器。这是启用 Serial 的必需参数。

`-XX:+UseSerialOldGC`：启用 Serial Old 收集器（通常与 Serial GC 一起自动启用）。

`-XX:NewSize`：设置年轻代的初始大小。

`-XX:MaxNewSize`：设置年轻代的最大大小。

`-XX:NewRatio`：设置年轻代与老年代的比例。

```bash
# Serial 收集器的完整配置示例
java -XX:+UseSerialGC \
     -Xms128m -Xmx128m \
     -XX:NewSize=64m -XX:MaxNewSize=64m \
     -XX:NewRatio=1 \
     -XX:+PrintGCDetails \
     -XX:+PrintGCDateStamps \
     -Xloggc:gc.log \
     MyApplication
```

## Serial Old 收集器的特点

|Serial Old|

Serial Old 是 Serial 收集器的老年代版本，同样使用单线程和标记-整理算法。它的特点和 Serial 收集器类似：简单、低开销、但停顿时间长。

Serial Old 主要在以下场景使用：作为 Serial 收集器的老年代搭档、使用 CMS 收集器的备选方案（当 CMS 发生并发模式失败时）、大对象直接进入老年代的分配担保。

CMS 在并发模式失败时，会回退到 Serial Old 进行单线程 Full GC，这是 CMS 的一个重大缺陷。当发生 "Concurrent Mode Failure" 时，停顿时间可能非常长。

## Serial 收集器的优缺点总结

|优点 |缺点 |
|---|---|
| 实现简单，稳定性高 | 单线程，停顿时间长 |
| 无额外内存开销 | 不适合多核 CPU |
| 对小内存应用友好 | 不适合大内存应用 |
| 适合客户端模式 | 吞吐量低 |

Serial 收集器的设计哲学是"简单即美"。虽然它的性能不如现代收集器，但它的高稳定性和低开销使其在特定场景下仍然有价值。
