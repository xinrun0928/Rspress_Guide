# Serial / Serial Old：单线程收集器

Serial 收集器是 JVM 历史上最早的垃圾收集器，也是最简单、最基础的选择。它使用单线程进行垃圾回收，在回收期间暂停所有应用线程。虽然 Serial 收集器在今天看来效率不高，但它简洁的实现和低内存开销使其在某些场景下仍然有价值。理解 Serial 收集器的工作原理，是理解其他更复杂收集器的基础。

## Serial 收集器的工作原理

<Badge>Serial GC</Badge>

Serial 收集器如其名字所示，使用单线程进行垃圾回收。当 Serial 收集器工作时，它会暂停所有应用线程（Stop-The-World），然后在后台使用唯一的 GC 线程执行垃圾回收任务。

Serial 收集器采用复制算法回收年轻代。在 Minor GC 时，Serial 收集器将 Eden 区和当前 Survivor 区中存活的对象复制到另一个 Survivor 区，年龄达标的对象晋升到老年代。复制完成后，Eden 和原来的 Survivor 区被清空。

Serial Old 收集器是 Serial 收集器的老年代版本，使用标记-整理算法。在 Full GC 时，Serial Old 收集器首先标记所有存活对象，然后将存活对象向一端移动，最后清除边界外的内存。

```
Serial GC 工作示意:
┌─────────────────────────────────────────┐
│           Stop-The-World                 │
│                                          │
│    应用线程暂停 ←──── GC 线程执行        │
│    (所有线程)      (单线程回收)          │
│                                          │
└─────────────────────────────────────────┘
```

## Serial 收集器的特点

Serial 收集器有几个显著的特点。第一是简单高效。Serial 的实现非常简洁，没有复杂的并发控制机制，GC 线程的额外开销很小。对于单核 CPU 或极小内存的场景，Serial 的效率可能反而更高。

第二是内存开销小。Serial 只需要一个 GC 线程，不需要额外的线程同步开销。对于内存受限的环境（如嵌入式系统），Serial 是合理的选择。

第三是停顿时间长。由于是单线程串行执行，Serial 的停顿时间与堆大小和存活对象数量成正比。在大内存场景下，停顿时间可能达到数秒。

```bash
# 启用 Serial 收集器
java -XX:+UseSerialGC -Xms256m -Xmx256m MyApplication

# 启用 Serial GC + PrintGCDetails 查看 GC 日志
java -XX:+UseSerialGC -XX:+PrintGCDetails -Xms256m -Xmx256m MyApplication
```

## 何时使用 Serial 收集器

<Badge>Client VM</Badge>

Serial 收集器并非一无是处，在某些场景下它是合理甚至最优的选择。

第一种场景是客户端模式（Client VM）。在 JDK 8 及之前的客户端 JVM 中，Serial 是默认的垃圾收集器。这是因为客户端应用的内存通常较小，停顿时间的影响不明显，而 Serial 的低开销和简单性更适合客户端环境。

第二种场景是单核 CPU 环境。在单核 CPU 上，Parallel Scavenge 等多线程收集器无法发挥并行优势，反而因为线程切换和同步开销而效率更低。Serial 在单核环境下的表现可能更好。

第三种场景是极小内存（`<100MB`）环境。在极小的堆内存下，Serial 的停顿时间本身就很短，而其低内存开销成为优势。

第四种场景是调试和测试环境。由于 Serial 的行为简单可预测，在调试 GC 行为或测试 GC 相关功能时，Serial 是常用的选择。

## Serial 收集器的常用参数

<Badge>JVM 参数</Badge>

```bash
# 年轻代使用 Serial（复制算法）
-XX:+UseSerialGC

# 老年代使用 Serial Old（标记-整理算法）
# Serial Old 通常与 Serial 或 ParNew 配合

# 堆大小设置
-Xms256m    # 最小堆
-Xmx256m    # 最大堆

# 年轻代大小
-Xmn128m    # 年轻代大小
-XX:SurvivorRatio=8  # Eden:Survivor 比例

# 打印 GC 日志
-XX:+PrintGCDetails
-XX:+PrintGCDateStamps
-Xloggc:gc.log
```

## Serial 收集器的 GC 日志分析

```bash
# Serial GC 日志示例
33.125: [GC (Allocation Failure)  166K->96K(62M), 0.0013879 secs]
34.562: [Full GC (System.gc())  1024K->512K(62M), 0.0056789 secs]
```

日志解读：`33.125` 是 JVM 启动后的秒数。`GC` 表示这是一个年轻代 GC（Minor GC）。`Allocation Failure` 是 GC 触发原因（Eden 区空间不足）。`166K->96K` 表示 GC 前占用 166KB，GC 后占用 96KB。`(62M)` 表示年轻代总大小为 62MB。`0.0013879 secs` 表示 GC 停顿时间为 1.39 毫秒。

Full GC 日志中，`Full GC` 表示整个堆的 GC。`(System.gc())` 表示触发原因是显式调用。`1024K->512K` 表示堆内存从 1024KB 回收到了 512KB。

## Serial Old 的作用

<Badge>CMS Backup</Badge>

Serial Old 在现代 JVM 中的主要作用有两个。第一是作为老年代收集器与 Serial 配合使用，主要用于客户端模式或极小内存场景。第二是在 CMS 收集器的并发模式失败时作为后备方案。

当 CMS 在并发标记阶段发生空间不足时，它会放弃并发标记，回退到 Serial Old 进行串行的 Full GC。这种情况称为"并发模式失败"（Concurrent Mode Failure），是 CMS 使用中需要尽量避免的问题。

## Serial 收集器的演进

<Badge>JDK 10</Badge>

Serial 收集器虽然是最古老的收集器，但它并非一成不变。HotSpot JVM 在持续优化 Serial 的实现，包括更快的对象分配、更好的卡片表优化、以及与逃逸分析的配合等。

在 JDK 10 中，HotSpot 引入了 Serial GC 的改进版本 Epsilon GC。Epsilon GC 只分配内存，不回收内存，适用于短生命周期程序或测试场景。虽然 Epsilon 不是 Serial 的直接改进，但它体现了 JVM 在极简方向上的探索。

## 面试高频考点

**问题一：Serial 收集器的特点是什么？** Serial 是单线程收集器，使用复制算法回收年轻代，标记-整理算法回收老年代。特点是简单高效、内存开销小，但停顿时间长。

**问题二：什么时候适合使用 Serial 收集器？** 客户端模式、单核 CPU 环境、极小内存环境、调试测试环境等场景适合使用 Serial。

**问题三：Serial Old 的作用是什么？** Serial Old 是 Serial 的老年代版本，使用标记-整理算法。另外，在 CMS 并发模式失败时作为后备方案使用。

**问题四：Serial 收集器的停顿时间与什么因素有关？** 停顿时间与堆大小成正比，与存活对象数量成正比。由于是单线程，停顿时间通常比并行收集器长。

**问题五：Serial 和 Parallel Scavenge 的区别是什么？** Serial 是单线程，Parallel Scavenge 是多线程并行。Parallel Scavenge 追求高吞吐量，使用自适应调节策略。Serial 追求简单和低开销。

Serial 收集器虽然古老，但它简洁的设计理念在今天仍有价值。在追求极致简单的场景下，Serial 仍然是不可替代的选择。那么，在什么场景下你会选择 Serial 而不是更现代的收集器？
