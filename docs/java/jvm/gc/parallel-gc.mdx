# Parallel Scavenge / Parallel Old：吞吐量优先

Parallel Scavenge 和 Parallel Old 是 JVM 中追求高吞吐量的收集器组合。如果说 Serial 是单兵作战，CMS 是精兵突袭，那么 Parallel Scavenge 就是大兵团协同作战。它充分利用多核 CPU 的优势，通过并行处理最大化垃圾回收的吞吐量。理解这个收集器组合，是理解高吞吐量场景下 JVM 调优的关键。

## 吞吐量优先的设计理念

|吞吐量优先|

Parallel Scavenge 收集器的设计目标与其他收集器不同。其他收集器（如 CMS、G1）追求减少停顿时间，而 Parallel Scavenge 追求最大化应用的运行时间占比——即吞吐量。

**吞吐量** = **运行应用代码时间** / (运行应用代码时间 + GC 停顿时间)

如果一个应用每运行 100 秒，GC 停顿 1 秒，那么吞吐量就是 99%。Parallel Scavenge 的目标就是最大化这个比率。

```
吞吐量 vs 停顿时间:

吞吐量优先 (Parallel Scavenge + Parallel Old)
┌─────────────────────────────────────────────┐
│███████████████ 应用运行 ████████████████│
└─────────────────────────────────────────────┘
                           ↑
                      较长停顿

停顿优先 (CMS / G1)
┌────────┐┌────────┐┌────────┐┌────────┐
│  停顿  ││  停顿  ││  停顿  ││  停顿  │
└────────┘└────────┘└────────┘└────────┘
     ↑         ↑         ↑         ↑
   短停顿    短停顿    短停顿    短停顿
```

## Parallel Scavenge 的工作原理

|Parallel Scavenge|

Parallel Scavenge 在年轻代使用复制算法，与 Serial 和 ParNew 类似，但使用多线程并行执行。多个 GC 线程同时扫描和复制年轻代的存活对象，充分利用多核 CPU 的并行能力。

Parallel Scavenge 的并行实现经过精心优化。使用了线程本地分配缓冲区（TLAB）减少分配时的锁竞争。使用了对称多线程处理技术，多个 GC 线程分工协作，效率更高。

```bash
# Parallel Scavenge 的核心参数
-XX:+UseParallelGC          # 启用 Parallel Scavenge + Parallel Old 组合
-XX:+UseParallelOldGC       # 启用 Parallel Old（如果只启用 UseParallelGC，默认会启用）
```

## Parallel Old 的配合

|Parallel Old|

Parallel Old 是 Parallel Scavenge 的老年代搭档，使用标记-整理算法。与 Parallel Scavenge 一样，Parallel Old 也是并行执行的，多个 GC 线程同时进行老年代的标记和整理工作。

Parallel Scavenge + Parallel Old 的组合是 JDK 6 之后追求高吞吐量的标准选择。这个组合能够最大化多核 CPU 的利用率，是批处理、大数据处理等场景的理想选择。

```
Parallel Scavenge + Parallel Old 组合示意:

┌─────────────────────────────────────────────────────────────┐
│                      多核 CPU 并行工作                         │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐      │
│  │  GC线程1  │ │  GC线程2  │ │  GC线程3  │ │  GC线程4  │      │
│  └────┬─────┘ └────┬─────┘ └────┬─────┘ └────┬─────┘      │
│       │            │            │            │              │
│       └────────────┴─────┬──────┴────────────┘              │
│                          │                                   │
│                     并行扫描和复制                             │
└─────────────────────────────────────────────────────────────┘
```

## 自适应调节策略

Parallel Scavenge 有一个独特的功能：自适应调节策略。这个策略会自动调整年轻代的大小、Eden 和 Survivor 区的比例、以及对象晋升到老年代的年龄阈值，以达到设定的吞吐量目标。

这个功能通过 `-XX:+UseAdaptiveSizePolicy` 参数启用（默认启用）。启用后，JVM 会根据运行时的统计数据自动调整这些参数，减轻调优的负担。

```bash
# 自适应调节策略
-XX:+UseAdaptiveSizePolicy    # 启用自适应调节（默认启用）
-XX:+UseParallelGC           # 启用并行 GC（启用此参数后自动启用）

# 如果关闭自适应调节，需要手动设置各参数：
-XX:-UseAdaptiveSizePolicy   # 关闭自适应调节
-XX:NewSize=128m             # 手动设置年轻代大小
-XX:MaxNewSize=128m
-XX:SurvivorRatio=8          # Eden:Survivor = 8:1
-XX:PretenureSizeThreshold=1m # 大对象直接进入老年代
```

## 吞吐量相关参数

Parallel Scavenge 提供了一些专门用于控制吞吐量和停顿时间的参数：

`-XX:GCTimeRatio`：设置 GC 时间与应用运行时间的比例。默认是 19，表示 GC 时间占总时间的 1/(1+19) = 5%。如果设置为 99，则 GC 时间占 1%。

`-XX:MaxGCPauseMillis`：设置期望的最大 GC 停顿时间（毫秒）。JVM 会尽量满足这个目标，但不保证一定能达到。

```bash
# 设置吞吐量目标：99% = GC时间占1%
java -XX:+UseParallelGC -XX:GCTimeRatio=99 -Xmx256m MyApplication

# 设置最大停顿时间目标：200毫秒
java -XX:+UseParallelGC -XX:MaxGCPauseMillis=200 -Xmx256m MyApplication

# 组合使用（停顿时间优先）
java -XX:+UseParallelGC \
     -XX:MaxGCPauseMillis=200 \
     -XX:GCTimeRatio=19 \
     -Xmx256m MyApplication
```

需要注意的是，MaxGCPauseMillis 和 GCTimeRatio 可能会有冲突。如果同时设置这两个参数，JVM 可能会优先满足 MaxGCPauseMillis。

## Parallel Scavenge 的适用场景

Parallel Scavenge + Parallel Old 组合最适合以下场景：

第一种是批处理任务。离线批处理、数据处理、ETL 等任务不需要实时响应，可以容忍较长的 GC 停顿，追求的是单位时间内处理更多任务。

第二种是科学计算。科学计算应用通常计算量大、对象生命周期长，需要高吞吐量。

第三种是对吞吐量要求高的后台服务。某些后台服务追求的是高并发处理能力，对单次请求的响应时间要求不高。

```
典型的不适合场景:
├── 低延迟应用（如实时交易系统）← 需要使用 CMS / G1 / ZGC
├── 内存受限环境 ← Serial 可能更合适
└── 单核 CPU ← Serial 可能效率更高
```

## JDK 8 的默认配置

JDK 8 的 Server 模式下，默认使用 Parallel Scavenge + Parallel Old 组合。这是因为在服务器环境中，多核 CPU 是主流，高吞吐量是大多数应用的目标。

```bash
# JDK 8 Server 模式的默认 GC 配置
# 等价于：
-XX:+UseParallelGC
-XX:+UseParallelOldGC
-XX:+UseAdaptiveSizePolicy
```

如果使用 JDK 9+，默认收集器变成了 G1，不再是 Parallel Scavenge + Parallel Old。
