# Parallel Scavenge / Parallel Old：吞吐量优先

Parallel Scavenge 和 Parallel Old 是 JVM 中追求高吞吐量的收集器组合。如果说 Serial 是单兵作战，CMS 是精兵突袭，那么 Parallel Scavenge 就是大兵团协同作战。它充分利用多核 CPU 的优势，通过并行处理最大化垃圾回收的吞吐量。理解这个收集器组合，是理解高吞吐量场景下 JVM 调优的关键。

## 吞吐量优先的设计理念

<Badge>Throughput</Badge>

Parallel Scavenge 收集器的设计目标与其他收集器不同。其他收集器（如 CMS、G1）追求减少停顿时间，而 Parallel Scavenge 追求最大化应用的运行时间占比——即吞吐量。

吞吐量 = 运行应用代码时间 / (运行应用代码时间 + GC 停顿时间)

如果一个应用每运行 100 秒，GC 停顿 1 秒，那么吞吐量就是 99%。Parallel Scavenge 的目标就是最大化这个比率。

## Parallel Scavenge 的工作原理

<Badge>Parallel Scavenge</Badge>

Parallel Scavenge 在年轻代使用复制算法，与 Serial 和 ParNew 类似，但使用多线程并行执行。多个 GC 线程同时扫描和复制年轻代的存活对象，充分利用多核 CPU 的并行能力。

Parallel Scavenge 的并行实现经过精心优化。使用了线程本地分配缓冲区（TLAB）减少分配时的锁竞争。使用了对称多线程处理技术，多个 GC 线程分工协作，效率更高。

```bash
# Parallel Scavenge 的核心参数
-XX:+UseParallelGC          # 启用 Parallel Scavenge + Parallel Old 组合
-XX:+UseParallelOldGC       # 启用 Parallel Old（如果只启用 UseParallelGC，默认会启用）
```

## Parallel Old 的配合

<Badge>Parallel Old</Badge>

Parallel Old 是 Parallel Scavenge 的老年代搭档，使用标记-整理算法。与 Parallel Scavenge 一样，Parallel Old 也是并行执行的，多个 GC 线程同时进行老年代的标记和整理工作。

Parallel Scavenge + Parallel Old 的组合是 JDK 6 之后追求高吞吐量的标准选择。这个组合能够最大化多核 CPU 的利用率，是批处理、大数据处理等场景的理想选择。

## 自适应大小策略

<Badge>Adaptive Size Policy</Badge>

Parallel Scavenge 最独特的特性是自适应大小策略（Adaptive Size Policy）。这个特性使得 Parallel Scavenge 能够根据应用的实际运行情况自动调节各代的大小，以达到最佳的吞吐量目标。

当自适应策略开启时，JVM 会自动调整年轻代大小、Survivor 区大小、晋升阈值等参数。应用运行过程中，JVM 持续监控 GC 统计数据，分析吞吐量是否达到目标，如果未达到则进行调整。

```bash
# 自适应策略相关参数
-XX:+UseAdaptiveSizePolicy    # 开启自适应策略（默认开启）
-XX:MaxGCPauseMillis=500     # 设置期望的最大停顿时间（毫秒）
-XX:GCTimeRatio=19           # 设置 GC 时间与应用时间的比例（默认 19）
```

`-XX:GCTimeRatio` 参数的含义是：如果设置为 19，表示 GC 时间占总时间的 1/(19+1) = 5%。也就是说，吞吐量目标是 95%。

## 吞吐量优先 vs 停顿优先

<Badge>GC Tuning</Badge>

选择吞吐量优先还是停顿优先，取决于应用的特点。对于批处理作业、大数据计算、离线数据处理等场景，吞吐量是首要目标，Parallel Scavenge 是最佳选择。对于 Web 应用、在线交易系统、游戏服务器等对响应时间敏感的场景，停顿优先的收集器（如 G1、ZGC）更合适。

```bash
# 吞吐量优先场景
java -XX:+UseParallelGC -XX:+UseParallelOldGC \
     -XX:GCTimeRatio=19 \
     -Xms4g -Xmx4g \
     -jar batch-processor.jar

# 停顿优先场景
java -XX:+UseG1GC \
     -XX:MaxGCPauseMillis=200 \
     -Xms4g -Xmx4g \
     -jar webapp.jar
```

## Parallel Scavenge 的常用参数

<Badge>JVM 参数</Badge>

```bash
# 基本参数
-XX:+UseParallelGC          # 年轻代使用 Parallel Scavenge
-XX:+UseParallelOldGC        # 老年代使用 Parallel Old
# 注意：UseParallelGC 已经隐含启用 Parallel Old

# 吞吐量相关
-XX:GCTimeRatio=19          # GC 时间占比 = 1/(19+1) = 5%
-XX:MaxGCPauseMillis=500    # 最大停顿时间目标（毫秒）

# 内存大小设置
-Xms4g                       # 最小堆
-Xmx4g                       # 最大堆
-Xmn2g                       # 年轻代大小
-XX:NewRatio=2               # 年轻代:老年代 = 1:2

# Survivor 区设置
-XX:SurvivorRatio=8         # Eden:Survivor = 8:1
-XX:MaxTenuringThreshold=15   # 最大年龄阈值

# GC 线程数
-XX:ParallelGCThreads=8      # GC 线程数（默认 = CPU 核心数）
```

## Parallel Scavenge 的 GC 日志分析

```bash
# Parallel Scavenge GC 日志示例
2023-01-15T10:23:45.123-0800: [GC (Allocation Failure) 
    [PSYoungGen: 524288K->65536K(6291456K)] 
    1048576K->589824K(16777216K), 
    0.0234567 secs] 
    [Times: user=0.15 sys=0.02, real=0.02 secs]

2023-01-15T10:23:50.456-0800: [Full GC (System.gc()) 
    [PSYoungGen: 65536K->0K(6291456K)] 
    [ParOldGen: 1572864K->786432K(10485760K)] 
    1638400K->786432K(16777216K), 
    0.1234567 secs] 
    [Times: user=0.50 sys=0.10, real=0.12 secs]
```

日志解读：`PSYoungGen` 表示 Parallel Scavenge 年轻代。`ParOldGen` 表示 Parallel Old 老年代。`Times` 部分显示了 CPU 时间的消耗：`user` 是用户态 CPU 时间，`sys` 是内核态 CPU 时间，`real` 是实际经过的时间。如果 `user + sys > real * 线程数`，说明 GC 是并行的。

## 调优策略

<Badge>Performance Tuning</Badge>

使用 Parallel Scavenge 时，调优的核心是平衡吞吐量和停顿时间。以下是一些常用的调优策略。

第一，设定吞吐量目标。使用 `-XX:GCTimeRatio` 设定目标吞吐量。比率越高，目标吞吐量越高，但可能导致更长的停顿时间。

第二，设定停顿时间目标。使用 `-XX:MaxGCPauseMillis` 设定最大停顿时间目标。JVM 会尽量满足这个目标，但可能通过增大年轻代等方式实现。

第三，手动设置各代大小。如果自适应策略无法满足需求，可以手动设置 `-Xmn`、`-XX:NewRatio` 等参数。

第四，观察 GC 日志。使用 `-XX:+PrintGCDetails` 观察 GC 行为，根据实际情况调整参数。

## 面试高频考点

**问题一：Parallel Scavenge 的设计目标是什么？** 目标是最大化吞吐量，即最大化应用运行时间占总时间的比例。

**问题二：Parallel Scavenge 和 ParNew 有什么区别？** 两者都是年轻代并行收集器，但 Parallel Scavenge 追求高吞吐量，使用自适应策略；ParNew 追求低停顿，专为配合 CMS 设计。

**问题三：什么是自适应大小策略？** 自适应策略让 JVM 根据应用运行情况自动调整各代大小、Survivor 区比例、晋升阈值等参数，以达到吞吐量目标。

**问题四：`-XX:GCTimeRatio=19` 是什么意思？** 表示 GC 时间占总时间的 1/(19+1) = 5%，即目标吞吐量是 95%。

**问题五：Parallel Scavenge 适合什么场景？** 适合批处理、大数据计算、离线数据处理等吞吐量优先的场景。不适合对响应时间敏感的应用。

Parallel Scavenge 是追求极致吞吐量的利器，它的设计理念是"让 CPU 尽可能跑应用代码"。那么，在什么场景下，吞吐量比停顿时间更重要？
