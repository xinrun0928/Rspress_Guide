# Full GC 问题排查与调优

Full GC 是 JVM 中最需要关注的 GC 类型。Minor GC 停顿时间通常只有几十毫秒，对应用影响有限；而 Full GC 停顿时间可能达到数秒甚至更长，严重影响应用的响应时间。本节将深入探讨 Full GC 的成因分析、排查方法、以及调优策略。

## Full GC 的常见成因

<Badge>Full GC Causes</Badge>

理解 Full GC 的触发原因，是排查问题的第一步。

第一种是老年代空间不足。当老年代无法容纳从年轻代晋升的对象时，触发 Full GC。这是最常见的 Full GC 触发原因。可能是年轻代设置太小，导致对象过快晋升；也可能是老年代本身太小。

第二种是空间分配担保失败。在 Minor GC 之前，JVM 会检查老年代是否有足够的空间容纳年轻代晋升的对象。如果空间不足，先触发一次 Full GC 再进行 Minor GC。

第三种是显式调用 `System.gc()`。虽然 `System.gc()` 是建议而非强制，但 JVM 通常会执行 Full GC。如果代码中频繁调用 `System.gc()`，会导致 Full GC 频繁。

第四种是元空间（方法区）不足。JDK 8 及之后，如果元空间使用达到阈值，会触发 Full GC 来回收元空间中的死亡类。

第五种是 CMS 并发模式失败。CMS 在并发标记阶段，如果老年代空间不足，会放弃并发模式，回退到 Serial Old 进行 Full GC。

## 排查 Full GC 的方法

### 第一步：分析 GC 日志

<Badge>GC Log</Badge>

GC 日志是排查 Full GC 问题的第一手资料。通过分析 GC 日志，可以了解 Full GC 的频率、触发原因、以及回收效果。

```bash
# 开启详细 GC 日志
java -XX:+PrintGCDetails -XX:+PrintGCDateStamps \
     -XX:+PrintGCTimeStamps -Xloggc:gc.log \
     MyApplication
```

观察日志中的以下信息：Full GC 的触发原因（`Allocation Failure`、`System.gc()`、`concurrent mode failure` 等）、Full GC 前后老年代的内存变化、Full GC 的停顿时间。

### 第二步：使用 jstat 观察内存趋势

```bash
# 持续监控 GC 状态
jstat -gcutil <pid> 1000
```

观察 O（老年代使用率）是否持续偏高。如果老年代使用率始终在 90% 以上，说明老年代空间确实不足。

### 第三步：生成堆转储文件

```bash
# 使用 jmap 生成堆转储
jmap -dump:format=b,file=heap.hprof <pid>

# JDK 11+ 推荐使用 jcmd
jcmd <pid> GC.heap_dump heap.hprof
```

生成堆转储后，使用 MAT（Memory Analyzer Tool）分析内存占用情况，找出占用内存最多的对象。

## 常见 Full GC 问题与解决方案

<Badge>GC Problems</Badge>

### 问题一：老年代空间不足

**表现**：Full GC 后老年代空间仍然不足，内存持续增长。

**排查**：使用 MAT 分析堆转储，找出占用老年代最多的对象类型。

**解决**：增大老年代（`-Xmx`）、减少长期存活对象、检查是否存在内存泄漏。

### 问题二：年轻代太小导致频繁晋升

**表现**：频繁发生 Minor GC，且每次 Minor GC 都有对象晋升老年代。

**排查**：观察 GC 日志中年轻代和老年代的变化关系。

**解决**：增大年轻代（`-Xmn`）、增大 Survivor 区（`-XX:SurvivorRatio`）、调大晋升阈值（`-XX:MaxTenuringThreshold`）。

```bash
# 示例：增大年轻代
java -Xms4g -Xmx4g -Xmn2g MyApplication
# 年轻代 2GB，老年代 2GB
```

### 问题三：对象分配速率过高

**表现**：Minor GC 非常频繁，每次 Minor GC 后 Eden 区很快又满了。

**排查**：观察 Minor GC 频率和每次 GC 的效果。如果 Minor GC 每秒发生多次，说明对象分配速率过高。

**解决**：优化代码减少对象创建、使用对象池、减少大对象分配。

### 问题四：内存泄漏

**表现**：GC 后堆内存持续增长，最终导致频繁 Full GC。

**排查**：使用 MAT 分析堆转储，找出 GC Root 引用链。

**解决**：修复内存泄漏代码。常见的内存泄漏原因包括：静态集合持有对象引用、监听器未注销、缓存未清理、ThreadLocal 未清理。

### 问题五：CMS 并发模式失败

**表现**：日志中出现 `concurrent mode failure`。

**原因**：CMS 并发标记期间，老年代空间不足以容纳新晋升的对象。

**解决**：降低 CMS 触发阈值（`-XX:CMSInitiatingOccupancyFraction`）、增大堆内存、切换到 G1 或 ZGC。

```bash
# 降低 CMS 触发阈值
java -XX:+UseConcMarkSweepGC \
     -XX:CMSInitiatingOccupancyFraction=60 \
     MyApplication
```

## Full GC 调优参数总结

<Badge>JVM 参数</Badge>

```bash
# 年轻代调优
-Xmn512m              # 年轻代大小
-XX:NewRatio=2        # 年轻代:老年代 = 1:2
-XX:SurvivorRatio=8   # Eden:Survivor = 8:1
-XX:MaxTenuringThreshold=15  # 最大晋升年龄

# 老年代调优 (CMS)
-XX:CMSInitiatingOccupancyFraction=70  # 触发阈值
-XX:+UseCMSCompactAtFullCollection     # Full GC 后压缩

# G1 调优
-XX:+UseG1GC
-XX:MaxGCPauseMillis=200    # 停顿时间目标
-XX:G1HeapRegionSize=8m     # Region 大小

# ZGC 调优
-XX:+UseZGC
-XX:MaxGCPauseMillis=5      # 停顿时间目标
```

## 调优案例

<Badge>Case Study</Badge>

**案例：电商系统 Full GC 频繁**

某电商系统使用 CMS 收集器，Full GC 每小时发生 10+ 次，每次停顿 3 秒。

**排查过程**：分析 GC 日志，发现 Full GC 都是 `Allocation Failure`；jstat 显示老年代使用率持续在 90% 以上；MAT 分析发现大量 `Order` 对象在老年代；代码审查发现订单缓存使用 `HashMap`，只添加不删除。

**解决方案**：将 `HashMap` 替换为带过期机制的 `LinkedHashMap`；添加缓存大小限制；增大年轻代，调整晋升阈值为 8。

**效果**：Full GC 降到每小时 1-2 次。

## 面试高频考点

**问题一：Full GC 的触发原因有哪些？** 老年代空间不足、空间分配担保失败、显式 System.gc()、元空间不足、CMS 并发模式失败。

**问题二：如何排查 Full GC 问题？** 分析 GC 日志、使用 jstat 监控、生成堆转储用 MAT 分析。

**问题三：CMS 并发模式失败的原因是什么？** CMS 并发标记期间，老年代空间不足以容纳新晋升的对象。

**问题四：如何减少 Full GC 的频率？** 增大堆内存、调整年轻代大小、优化代码减少对象创建、避免内存泄漏。

**问题五：Full GC 时间过长怎么办？** 使用 G1 或 ZGC 等低延迟收集器、减小堆大小、减少长期存活对象。

Full GC 是 JVM 调优中最需要关注的指标之一，它直接反映了应用的内存健康状况。那么，当 Full GC 无法避免时，如何让它的影响最小化？
