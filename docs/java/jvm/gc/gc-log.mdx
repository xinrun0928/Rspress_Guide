# GC 日志分析实战

GC 日志是排查 JVM 内存问题和调优 GC 性能的重要工具。通过分析 GC 日志，可以了解 GC 的频率、停顿时间、内存变化趋势，从而判断 GC 配置是否合理，是否存在内存泄漏等问题。本节将详细介绍如何分析 GC 日志，以及如何从日志中发现问题。

## 开启 GC 日志

<Badge>GC Log</Badge>

默认情况下，JVM 不输出 GC 日志。需要通过参数开启。

```bash
# JDK 8 及之前
java -XX:+PrintGCDetails -XX:+PrintGCDateStamps -Xloggc:gc.log MyApplication

# JDK 9+ 推荐使用新的日志系统
java -Xlog:gc*:file=gc.log:time,uptime MyApplication

# 更详细的配置
java -Xlog:gc*=debug,gc+heap=trace:file=gc_debug.log:time,uptime,level MyApplication
```

`-Xlog` 参数格式为：`日志标签:日志级别:输出配置`。gc* 表示所有 gc 相关的日志，*=debug 表示 debug 及以上级别。

## GC 日志格式解读

<Badge>Log Format</Badge>

### 年轻代 GC 日志（ParNew）

```bash
2023-01-15T10:23:45.123-0800: [ParNew: 524288K->65536K(6291456K), 0.0156789 secs] 
1234567K->234567K(16777216K), 0.0157890 secs]
```

- `2023-01-15T10:23:45.123-0800`: GC 发生的时间戳
- `[ParNew:`: 使用的年轻代收集器是 ParNew
- `524288K->65536K`: GC 前年轻代占用 512MB，GC 后占用 64MB
- `(6291456K)`: 年轻代总大小 6GB
- `0.0156789 secs`: 年轻代 GC 停顿时间 15.7ms
- `1234567K->234567K`: 整个堆 GC 前 1.2GB，GC 后 230MB
- `16777216K`: 堆总大小 16GB
- `0.0157890 secs`: 整个 GC 停顿时间

### CMS GC 日志

```bash
2023-01-15T10:23:50.456-0800: [GC (CMS Initial Mark) 
    [CMS: 1572864K->1572864K(2097152K), 0.0000000 secs] 
    2097152K->1572864K(2097152K), 0.0001234 secs]
```

- `CMS Initial Mark`: CMS 初始标记阶段
- 后续是 CMS 的并发标记、预清理、重新标记等阶段

### Full GC 日志

```bash
2023-01-15T10:24:00.789-0800: [Full GC (Allocation Failure) 
    [CMS: 1572864K->786432K(2097152K), 2.3456789 secs] 
    2097152K->786432K(2097152K), 2.3456789 secs]
```

- `Full GC`: Full GC
- `(Allocation Failure)`: 触发原因是分配失败（老年代空间不足）
- `[CMS: ...]`: CMS 清理后的状态
- `2.3456789 secs`: Full GC 停顿 2.35 秒，这是一个很长的停顿

### G1 GC 日志

```bash
2023-01-15T10:25:00.123: [GC pause (G1 Evacuation Pause) (young), 0.0123456 secs]
2023-01-15T10:25:10.456: [GC pause (G1 Evacuation Pause) (mixed), 0.0234567 secs]
2023-01-15T10:25:20.789: [GC pause (G1 Humongous Allocation) (young), 0.0056789 secs]
```

- `G1 Evacuation Pause (young)`: G1 年轻代回收
- `G1 Evacuation Pause (mixed)`: G1 混合回收
- `G1 Humongous Allocation`: 大对象分配

## 使用工具分析 GC 日志

<Badge>GC Analysis Tools</Badge>

手动分析 GC 日志效率低下，推荐使用专业工具。

### GCEasy

GCEasy（https://gceasy.io）是一个在线 GC 日志分析工具，支持所有主流收集器的日志格式。上传日志文件后，可以自动分析 GC 行为，生成可视化报告，指出潜在问题。

### GCViewer

GCViewer 是一个开源的本地 GC 日志分析工具，支持图表展示 GC 趋势、停顿时间分布等。

### jstat 查看实时 GC 状态

```bash
# 查看 GC 统计信息（每 1 秒刷新一次）
jstat -gcutil <pid> 1000

# 输出示例
  S0     S1     E      O      M     CCS    YGC     YGCT    FGC    FGCT     GCT   
  0.00  12.34  45.67  78.90  89.12  90.23     12    0.234  3      1.234    1.468
```

- S0, S1: Survivor 区使用率
- E: Eden 区使用率
- O: 老年代使用率
- M: Metaspace 使用率
- YGC: Young GC 次数
- YGCT: Young GC 总时间
- FGC: Full GC 次数
- FGCT: Full GC 总时间
- GCT: GC 总时间

## GC 问题分析

<Badge>GC Problems</Badge>

### 问题一：频繁 Full GC

如果 Full GC 频繁发生，说明老年代空间不足或对象晋升过快。

**表现**：FGCT（Full GC 总时间）持续增长，FGC（Full GC 次数）快速增加。

**原因**：老年代太小、创建了大量长期存活对象、内存泄漏。

**解决**：增大堆内存、减少长期存活对象、检查内存泄漏。

### 问题二：GC 停顿时间过长

**表现**：GCT（GC 总时间）过长，特定 GC 停顿时间超过预期。

**原因**：堆太大、存活对象太多、收集器选择不当。

**解决**：选择低延迟收集器（如 G1、ZGC）、调整各代大小。

### 问题三：内存持续增长

**表现**：GC 后内存持续增长，最终达到堆上限。

**原因**：内存泄漏、分配速率过高、大对象问题。

**解决**：排查内存泄漏、使用 MAT 分析堆转储。

## GC 日志分析实战案例

<Badge>Case Study</Badge>

```bash
# 模拟问题场景的日志
2023-01-15T10:00:00.000: [GC [ParNew: 102400K->10240K(460800K), 0.0051234 secs] 
    204800K->184320K(2097152K), 0.0052345 secs]

2023-01-15T10:00:01.000: [Full GC (Allocation Failure) 
    [CMS: 1843200K->1572864K(2097152K), 3.4567890 secs] 
    2097152K->1572864K(2097152K), 3.4567890 secs]

2023-01-15T10:00:05.000: [Full GC (Allocation Failure) 
    [CMS: 1572864K->1572864K(2097152K), 2.3456789 secs] 
    2097152K->1572864K(2097152K), 2.3456789 secs]
```

从这段日志可以看出：ParNew GC 停顿只有 5ms，正常；CMS Full GC 停顿 3.5 秒，太长；紧接着又发生一次 Full GC，说明老年代空间恢复不足；两次 Full GC 都是 `Allocation Failure`，说明对象分配过快或老年代太小。

**建议**：增大老年代空间、降低对象分配速率、或切换到 G1/ZGC 收集器。

## 面试高频考点

**问题一：如何开启 GC 日志？** JDK 8 使用 `-XX:+PrintGCDetails -XX:+PrintGCDateStamps -Xloggc:gc.log`。JDK 9+ 使用 `-Xlog:gc*:file=gc.log`。

**问题二：Full GC 频繁的原因是什么？** 老年代空间不足、对象晋升过快、内存泄漏、大对象问题等。

**问题三：如何分析 GC 日志发现内存泄漏？** 观察 GC 后内存是否持续增长，如果 GC 后内存始终无法降到合理水平，可能是内存泄漏。

**问题四：YGC 和 FGC 次数很多正常吗？** YGC 频繁是正常的（取决于年轻代大小）。FGC 频繁不正常，说明老年代有问题需要排查。
