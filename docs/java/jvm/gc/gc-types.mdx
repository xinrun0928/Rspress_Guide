# Minor GC vs Major GC vs Full GC

在 JVM 调优和问题排查中，我们经常听到 Minor GC、Major GC、Full GC 这些术语。它们代表了不同范围和不同代价的垃圾回收操作。理解这三种 GC 的区别、触发条件、以及对应用的影响，是进行 JVM 调优的基础。很多性能问题的根源，都与这三种 GC 的不当配置有关。

## GC 分类的本质

<Badge>GC Types</Badge>

从名字上看，Minor、Major、Full 似乎是按照"规模"或"重要程度"来分类的，但实际上这种分类是基于垃圾回收的作用范围。Minor GC 作用于年轻代，Major GC 作用于老年代，Full GC 作用于整个堆（包括年轻代、老年代、有时还包括方法区）。

这种分类背后是分代收集理论的应用。年轻代和老年代的对象有不同的生命周期特点，需要不同的回收策略。Minor GC 频繁但代价低，主要回收大量死亡对象；Major GC 和 Full GC 频率低但代价高，涉及更复杂的对象图扫描。

需要注意的是，"Major GC" 这个术语在不同的收集器中含义不同。在某些语境下，Major GC 等同于 Full GC；在另一些语境下，Major GC 特指老年代的回收，而 Full GC 是完整的堆回收。理解具体语境才能正确理解含义。

## Minor GC：年轻代的日常清理

<Badge>Minor GC</Badge>

Minor GC 是回收年轻代的垃圾收集操作。当 Eden 区空间不足以分配新对象时触发 Minor GC。这是 Java 程序运行中最频繁的 GC 操作，也是大部分对象的最终归宿——它们在 Minor GC 中被回收。

Minor GC 的工作流程相对简单：首先标记 Eden 和当前 Survivor 区（From Survivor）中存活的对象，然后将存活对象复制到另一个 Survivor 区（To Survivor）。如果 Survivor 区空间不足，超出的对象或者年龄达标的对象被晋升到老年代。最后，清空 Eden 和 From Survivor 区。

```java
// Minor GC 触发的示例
public class MinorGCDemo {
    public static void main(String[] args) {
        // 分配大量对象
        List<byte[]> list = new ArrayList<>();
        for (int i = 0; i < 100; i++) {
            list.add(new byte[1024 * 1024]);  // 每次分配 1MB
            // 当 Eden 区满时触发 Minor GC
        }
    }
}
```

Minor GC 的特点包括：频率高（根据年轻代大小和对象分配速率，可能每秒发生多次）、停顿时间短（通常在几十毫秒以内）、主要回收死亡对象（存活对象较少，复制成本低）。由于 Minor GC 停顿时间短，对应用响应的影响通常较小。但频繁的 Minor GC 本身也可能影响性能，如果对象分配速率过高。

Minor GC 是 Stop-The-World 事件，会暂停所有应用线程。但由于年轻代的存活对象通常很少，扫描和复制的速度很快，停顿时间可以接受。Serial GC 和 Parallel GC 等使用单线程或少量线程进行 Minor GC，在高并发场景下可能成为瓶颈。

## Major GC 和 Full GC：重量级回收

<Badge>Major GC</Badge> <Badge>Full GC</Badge>

Major GC 和 Full GC 是更重量级的垃圾回收操作，涉及老年代的垃圾收集。顾名思义，Major GC 通常指老年代的回收，Full GC 指整个堆的回收。在实际使用中，Major GC 和 Full GC 经常混用，因为 Full GC 通常包括 Major GC。

Full GC 的触发条件有多种：

第一种是老年代空间不足。当老年代无法容纳大对象或无法容纳从年轻代晋升的对象时，触发 Full GC。这是最常见的 Full GC 触发原因。

第二种是显式调用 `System.gc()`。虽然这是建议而非强制，但 JVM 通常会执行 Full GC。不建议依赖这种方式进行垃圾回收。

第三种是元空间（方法区）不足。JDK 8 及之后，如果元空间接近上限，可能触发 Full GC 同时回收元空间。

第四种是 Minor GC 前的安全检查。某些收集器在执行 Minor GC 前会检查老年代是否有足够的空间容纳年轻代晋升的对象，如果空间不足，触发 Full GC 作为替代（这通常意味着空间分配担保失败）。

```
Full GC 触发场景:

场景 1: 老年代空间不足
┌──────────────────────────────────┐
│ 老年代无法容纳晋升对象            │
│ → 触发 Full GC                   │
└──────────────────────────────────┘

场景 2: System.gc() 调用
┌──────────────────────────────────┐
│ 应用显式调用 System.gc()         │
│ → JVM 执行 Full GC（建议性）     │
└──────────────────────────────────┘

场景 3: 元空间不足 (JDK 8+)
┌──────────────────────────────────┐
│ 元空间使用达到阈值               │
│ → 触发 Full GC 回收元空间        │
└──────────────────────────────────┘

场景 4: 空间分配担保失败
┌──────────────────────────────────┐
│ Minor GC 前检查老年代空间不足    │
│ → 先执行 Full GC 再 Minor GC     │
└──────────────────────────────────┘
```

Full GC 的特点与 Minor GC 形成鲜明对比：频率低（可能数分钟甚至数小时才发生一次）、停顿时间长（可能数百毫秒甚至数秒）、涉及整个堆的扫描（需要扫描年轻代、老年代、所有对象）。长时间的 Full GC 停顿会对应用响应造成严重影响，尤其是对延迟敏感的应用。

## GC 组合与行为差异

<Badge>GC Collector</Badge>

不同的垃圾收集器对 Minor GC、Major GC、Full GC 的处理方式不同。在 Serial GC、Parallel GC 中，Minor GC 和 Major GC/Full GC 是连续执行的，先 Minor GC 再根据需要执行 Major GC/Full GC。

CMS 收集器的行为更为复杂。CMS 追求低停顿时间，采用并发收集策略。CMS 的 Full GC 有几种情况：并发模式失败（concurrent mode failure）、碎片化导致的压缩（promotion failure）、以及显式 Full GC。

G1 收集器引入了混合回收（Mixed GC）的概念。G1 不再严格区分年轻代和老年代，而是将堆划分为多个 Region。G1 的年轻代回收（Young GC）只回收年轻代 Region，混合回收（Mixed GC）回收所有年轻代 Region 和部分老年代 Region。

ZGC 和 Shenandoah 的目标是实现停顿时间与堆大小无关。它们使用着色指针和并发压缩技术，在标记、压缩过程中保持极短的停顿时间。

```
不同收集器的 GC 行为:

Serial GC:
  Minor GC → (必要时) Major GC → (必要时) Full GC
  全部 STW，单线程

Parallel GC:
  Minor GC → (必要时) Major GC → (必要时) Full GC
  全部 STW，多线程并行

CMS GC:
  Minor GC (STW) → Major GC (并发)
  可能并发模式失败导致 STW Full GC

G1:
  Young GC → Mixed GC
  可控停顿时间

ZGC/Shenandoah:
  所有阶段并发
  停顿时间 < 1ms
```

## GC 停顿对应用的影响

<Badge>Stop-The-World</Badge>

GC 停顿（Stop-The-World）是影响应用响应的重要因素。用户请求在 GC 期间必须等待，因此 GC 停顿直接影响用户体验。Minor GC 停顿时间短，影响通常可接受。Full GC 停顿时间长，是需要重点关注和优化的对象。

停顿时间与堆大小成正比。堆越大，扫描和回收的时间越长。对于追求低停顿时间的应用，可以考虑使用 G1、ZGC、Shenandoah 等收集器。

停顿时间与存活对象数量成正比。年轻代中存活对象越少，Minor GC 停顿越短。可以通过调整年轻代大小、Survivor 区比例等方式优化。

停顿时间与收集器实现有关。并发收集器通过将部分工作与应用线程并发执行来减少停顿时间，但需要额外的 CPU 资源。

## 何时需要 Full GC

<Badge>GC Tuning</Badge>

Full GC 是我们最不希望看到的 GC 类型，但不是所有 Full GC 都是问题。了解何时发生 Full GC 以及如何优化，是 JVM 调优的核心内容。

合理的 Full GC 包括：应用启动时的初始 Full GC（加载所有类）、老年代满了之后的正常 Full GC（但频率不应过高）、显式 Full GC（但应尽量避免）。

异常的 Full GC 包括：频繁的 Full GC（说明年轻代太小或对象分配速率太高）、长时间停顿的 Full GC（说明堆太大或收集器选择不当）、并发模式失败（CMS 的常见问题）、碎片化导致的 Full GC（CMS 压缩）。

```java
// 分析 Full GC 原因的示例代码
public class GCAnalysis {
    public static void main(String[] args) {
        // 模拟频繁触发 Full GC 的场景
        // 1. 创建大量长期存活对象
        List<byte[]> cache = new ArrayList<>();
        
        // 2. 创建大量短期对象
        for (int i = 0; i < 100000; i++) {
            // 短期对象填满年轻代
            byte[] temp = new byte[10 * 1024];  // 10KB
            // temp 在循环结束时应该被回收
        }
        
        // 3. 长期存活对象占用老年代
        for (int i = 0; i < 100; i++) {
            cache.add(new byte[1024 * 1024]);  // 1MB 大对象直接进老年代
        }
    }
}
```

## 面试高频考点

**问题一：Minor GC、Major GC、Full GC 的区别是什么？** Minor GC 回收年轻代，频率高、停顿短；Major GC 回收老年代，频率低、停顿长；Full GC 回收整个堆，是最重量级的回收操作。

**问题二：什么时候会触发 Minor GC？** 当 Eden 区空间不足以分配新对象时触发 Minor GC。

**问题三：什么时候会触发 Full GC？** 老年代空间不足时、元空间不足时、显式调用 System.gc() 时、以及某些收集器的特定条件下。

**问题四：为什么 Full GC 停顿时间比 Minor GC 长？** Full GC 需要扫描整个堆的所有对象，包括年轻代和老年代；存活对象多，需要更多的复制或整理操作；涉及更大的内存区域。

**问题五：CMS 的并发模式失败是什么？** CMS 在并发标记阶段，应用线程可能产生新的对象，导致老年代空间不足。这种情况下，CMS 会放弃并发标记，进入 STW 模式的 Full GC。

**问题六：如何减少 Full GC 的频率？** 增大堆大小或年轻代大小、优化对象分配模式减少大对象、选择合适的垃圾收集器、避免显式调用 System.gc()。

从 Minor GC 到 Full GC，垃圾回收的"代价"递增。理解它们的区别和触发条件，是进行 GC 调优的第一步。那么，如果让你设计一个 GC 系统，你会如何平衡 GC 频率和停顿时间？
