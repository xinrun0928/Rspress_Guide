# Minor GC vs Major GC vs Full GC

在 JVM 调优和问题排查中，我们经常听到 Minor GC、Major GC、Full GC 这些术语。它们代表了不同范围和不同代价的垃圾回收操作。理解这三种 GC 的区别、触发条件、以及对应用的影响，是进行 JVM 调优的基础。很多性能问题的根源，都与这三种 GC 的不当配置有关。

## GC 分类的本质

|GC 类型|

从名字上看，Minor、Major、Full 似乎是按照"规模"或"重要程度"来分类的，但实际上这种分类是基于垃圾回收的作用范围。Minor GC 作用于年轻代，Major GC 作用于老年代，Full GC 作用于整个堆（包括年轻代、老年代、有时还包括方法区）。

这种分类背后是分代收集理论的应用。年轻代和老年代的对象有不同的生命周期特点，需要不同的回收策略。Minor GC 频繁但代价低，主要回收大量死亡对象；Major GC 和 Full GC 频率低但代价高，涉及更复杂的对象图扫描。

需要注意的是，"Major GC" 这个术语在不同的收集器中含义不同。在某些语境下，Major GC 等同于 Full GC；在另一些语境下，Major GC 特指老年代的回收，而 Full GC 是完整的堆回收。理解具体语境才能正确理解含义。

## Minor GC：年轻代的日常清理

|Minor GC|

Minor GC 是回收年轻代的垃圾收集操作。当 Eden 区空间不足以分配新对象时触发 Minor GC。这是 Java 程序运行中最频繁的 GC 操作，也是大部分对象的最终归宿——它们在 Minor GC 中被回收。

Minor GC 的工作流程相对简单：首先标记 Eden 和当前 Survivor 区（From Survivor）中存活的对象，然后将存活对象复制到另一个 Survivor 区（To Survivor）。如果 Survivor 区空间不足，超出的对象或者年龄达标的对象被晋升到老年代。最后，清空 Eden 和 From Survivor 区。

```java
// Minor GC 触发的示例
public class MinorGCDemo {
    public static void main(String[] args) {
        // 分配大量对象
        List<byte[]> list = new ArrayList<>();
        for (int i = 0; i < 100; i++) {
            list.add(new byte[1024 * 1024]);  // 每次分配 1MB
            // 当 Eden 区满时触发 Minor GC
        }
    }
}
```

Minor GC 的特点包括：频率高（根据年轻代大小和对象分配速率，可能每秒发生多次）、停顿时间短（通常在几十毫秒以内）、主要回收死亡对象（存活对象较少，复制成本低）。由于 Minor GC 停顿时间短，对应用响应的影响通常较小。但频繁的 Minor GC 本身也可能影响性能，如果对象分配速率过高。

Minor GC 是 Stop-The-World 事件，会暂停所有应用线程。但由于年轻代的存活对象通常很少，扫描和复制的速度很快，停顿时间可以接受。Serial GC 和 Parallel GC 等使用单线程或少量线程进行 Minor GC，在高并发场景下可能成为瓶颈。

## Major GC 和 Full GC：重量级回收

|Major GC| |Full GC|

Major GC 和 Full GC 是更重量级的垃圾回收操作，涉及老年代的垃圾收集。顾名思义，Major GC 通常指老年代的回收，Full GC 指整个堆的回收。在实际使用中，Major GC 和 Full GC 经常混用，因为 Full GC 通常包括 Major GC。

Full GC 的触发条件有多种：

第一种是老年代空间不足。当老年代的空间不足以容纳从年轻代晋升的对象时，会触发 Full GC。这是最常见的 Full GC 触发原因。

第二种是显式调用 `System.gc()`。虽然调用 `System.gc()` 并不一定会触发 Full GC（取决于 JVM 实现和参数配置），但它是 Full GC 的一个触发条件。建议使用 `-XX:+DisableExplicitGC` 参数禁用显式 GC。

第三种是元空间（方法区）不足。当元空间达到设定的阈值时，会触发 Full GC（同时回收堆和元空间）。这通常发生在大量动态类生成的应用中，如使用 CGLIB、GraalVM 等技术的场景。

第四种是分配担保失败。如果 Survivor 区空间不足，需要将对象晋升到老年代，但老年代空间也不足，会触发 Full GC。这是 Minor GC 失败后的兜底策略。

```bash
# 完整的 GC 日志示例
[GC (Allocation Failure) [PSYoungGen: 8192K->512K(9216K)] 12288K->6144K(20480K), 0.0156789 secs]
#                  ↑ 年轻代清理 ↑ 年轻代清理后/前(年轻代总大小) 堆清理后/前(堆总大小) 耗时
```

## 三种 GC 的对比

|类型 | 作用范围 | 触发条件 | 停顿时间 | 发生频率 |
|---|---|---|---|---|
| Minor GC | 年轻代 | Eden 区满 | 短（毫秒级） | 高 |
| Major GC | 老年代 | 老年代满 | 中等 | 低 |
| Full GC | 整个堆 + 方法区 | 多种条件 | 长（秒级可能） | 很低 |

Minor GC 和 Full GC 的一个重要区别是：Minor GC 几乎总是正常的、预期的行为，而 Full GC 往往是问题的信号。如果 Full GC 频繁发生，说明应用配置或代码存在问题，需要排查。

## GC 与应用响应时间

对于 Web 服务而言，GC 造成的停顿直接影响了用户体验。不同类型的 GC 对响应时间的影响程度不同。

Minor GC 由于停顿时间短（通常小于 50ms），对单次请求的影响可能不明显。但如果 Minor GC 过于频繁（如每秒数次），累积的影响可能导致整体吞吐量下降。

Full GC 的停顿时间可能达到数百毫秒甚至数秒。对于需要高响应速度的应用，如在线交易系统、游戏服务器等，这是不可接受的。即使是几百毫秒的停顿，也可能导致用户体验明显下降。

```
GC 停顿对响应时间的影响:

请求1 ─────┐
请求2 ───────────┐      ← 正常响应
请求3 ───────────────┐
                        ↓  Full GC 停顿 2 秒
请求4 ─────────────────────┐   ← 长时间无响应
请求5 ───────────────────────────┐
```

## 调优目标：减少 Full GC

JVM 调优的核心目标之一是减少 Full GC 的频率和持续时间。Full GC 频繁发生通常意味着以下问题之一：

年轻代空间过小会导致对象频繁晋升到老年代，加快老年代的填充速度。可以通过增大年轻代（`-Xmn`）或调整 `-XX:NewRatio` 来改善。

老年代空间过小会导致晋升失败，触发 Full GC。需要合理设置堆大小（`-Xmx`）和年轻代/老年代比例。

对象分配速率过高会导致 Minor GC 频繁发生，可能触发大量的对象晋升。需要优化代码，减少临时对象的创建。

内存泄漏会导致老年代持续增长，最终触发 Full GC。需要使用 MAT 等工具排查内存泄漏。

```bash
# 推荐的 GC 调优参数
java -Xms256m -Xmx256m \          # 堆初始值和最大值相等，避免动态调整
   -Xmn128m \                     # 年轻代 128MB
   -XX:NewRatio=1 \               # 年轻代:老年代 = 1:1
   -XX:+UseParallelGC \           # 使用 Parallel GC
   -XX:+PrintGCDetails \          # 打印详细 GC 日志
   -XX:+PrintGCDateStamps \       # 打印 GC 时间戳
   -Xloggc:gc.log \               # GC 日志输出到文件
   MyApplication
```

## GC 日志分析

通过分析 GC 日志，可以了解应用的 GC 情况，判断是否需要调优。常用的 GC 日志分析工具包括 GCEasy、GCViewer、JMeter 的 GC 插件等。

```bash
# 关键 GC 日志指标
# 1. GC 频率：Minor GC 和 Full GC 每分钟发生多少次
# 2. GC 耗时：每次 GC 花费多长时间
# 3. GC 前后内存变化：内存回收效率
# 4. 对象晋升速率：每秒有多少对象进入老年代

# 示例分析
[Full GC (System.gc()) [PSYoungGen: 0K->0K(133120K)] 
 [ParOldGen: 68912K->68912K(137216K)] 68912K->68912K(270336K), [Metaspace: 32768K->32768K(33280K)], 0.0234567 secs]
#          ↑ 显式调用 System.gc() 触发  ↑ 老年代内存未变化  ↑ 堆内存未变化   ↑ 元空间未变化    耗时
```

分析 GC 日志时，重点关注 Full GC 的原因。日志中会显示是哪种原因触发的 Full GC，如 `Allocation Failure`（分配失败）、`System.gc()`（显式调用）、`Last ditch collection`（最后一次 GC）等。
