# ParNew：多线程年轻代收集器

ParNew 收集器是年轻代收集器中的多线程版本，它在 Serial 收集器的基础上引入了多线程并行回收的能力。在多核 CPU 环境下，ParNew 的回收效率通常远高于 Serial。ParNew 在历史上扮演了重要角色，它曾是 CMS 收集器的唯一搭档，在追求低停顿时间的应用中被广泛使用。

## ParNew 的工作原理

<Badge>ParNew GC</Badge>

ParNew 收集器在年轻代使用复制算法，工作原理与 Serial 类似，但利用了多核 CPU 的并行能力。ParNew 启动多个 GC 线程并行工作，将年轻代的存活对象复制到 Survivor 区或晋升到老年代。

ParNew 的多线程实现使用了"stop-the-world"的并行算法。在 GC 开始时，所有应用线程被暂停（Stop-The-World）。然后，ParNew 启动多个 GC 线程，这些线程分工协作，同时扫描和复制年轻代中的存活对象。最后，所有应用线程恢复运行。

```
ParNew vs Serial 对比:
Serial:                              ParNew:
┌─────────────────────┐             ┌─────────────────────┐
│    STW               │             │    STW               │
│  ┌─────────────────┐│             │  ┌─────┬─────┬─────┐│
│  │   GC Thread     ││             │  │ GC1 │ GC2 │ GC3 ││
│  │   (单线程)      ││             │  │     │     │     ││
│  └─────────────────┘│             │  │  (多线程并行)     ││
└─────────────────────┘             └─────────────────────┘
```

## ParNew 的线程数量

<Badge>ParallelGCThreads</Badge>

ParNew 的线程数量默认等于 CPU 核心数。可以通过 `-XX:ParallelGCThreads` 参数调整。需要注意的是，这个参数不仅影响 ParNew，也影响其他并行收集器（Parallel Scavenge、Parallel Old）。

```bash
# 查看默认 GC 线程数
java -XX:+UseParNewGC -XX:+PrintFlagsFinal -version | grep ParallelGCThreads

# 设置 ParNew GC 线程数为 4
java -XX:+UseParNewGC -XX:ParallelGCThreads=4 MyApplication
```

GC 线程数量并非越多越好。过多的线程会导致线程切换开销增加，反而降低效率。对于 4 核或 8 核的 CPU，通常设置 4-6 个 GC 线程是比较合理的。

## ParNew 与 CMS 的配合

<Badge>CMS</Badge>

ParNew 最重要的作用是作为 CMS 收集器的年轻代收集器。CMS（Concurrent Mark Sweep）是追求低停顿时间的收集器，它的老年代回收是并发进行的，但年轻代回收必须是 STW 的。ParNew 就是为这个场景设计的。

CMS + ParNew 的组合是 JDK 5 到 JDK 8 早期追求低停顿时间应用的主流选择。CMS 并发回收老年代，ParNew 并行回收年轻代，两者配合实现了相对可控的停顿时间。

```bash
# 使用 CMS + ParNew 组合
java -XX:+UseConcMarkSweepGC -XX:+UseParNewGC MyApplication

# 注意：JDK 7u4 之后，UseConcMarkSweepGC 会自动启用 ParNew
# JDK 9 之后，CMS 被标记为废弃
# JDK 14 正式移除了 CMS
```

## ParNew 与 Parallel Scavenge 的区别

<Badge>Parallel Scavenge</Badge>

ParNew 和 Parallel Scavenge 都是年轻代并行收集器，但它们的设计目标不同。Parallel Scavenge 追求高吞吐量，使用自适应大小策略，不与其他收集器配合。ParNew 追求低停顿时间，专为配合 CMS 设计。

两者的另一个区别是对象晋升规则。ParNew 与 CMS 配合时，对象晋升老年代的规则与 Serial 类似。而 Parallel Scavenge 使用自己的自适应策略。

```bash
# ParNew 特点：配合 CMS 使用
# Parallel Scavenge 特点：追求吞吐量

# 两者都使用复制算法，但设计目标不同
```

## ParNew 的常用参数

<Badge>JVM 参数</Badge>

```bash
# 启用 ParNew
-XX:+UseParNewGC

# 设置 GC 线程数
-XX:ParallelGCThreads=4

# 年轻代大小设置
-Xmn256m           # 年轻代大小
-XX:NewRatio=2     # 年轻代:老年代比例
-XX:SurvivorRatio=8  # Eden:Survivor 比例

# 年龄阈值
-XX:MaxTenuringThreshold=15
```

## ParNew 的 GC 日志分析

```bash
# ParNew GC 日志示例
28.456: [ParNew: 524288K->65536K(589824K), 0.0156789 secs] 12345K->6789K(2097152K), 0.0157890 secs]
```

日志解读：`28.456` 是 JVM 启动后的秒数。`[ParNew:` 表示使用 ParNew 收集器。`524288K->65536K` 表示年轻代 GC 前占用 512MB，GC 后占用 64MB。`(589824K)` 表示年轻代总大小为 576MB。第一个 `0.0156789 secs` 是年轻代 GC 停顿时间。`12345K->6789K(2097152K)` 是整个堆的内存变化。第二个 `0.0157890 secs` 是整个 GC 停顿时间。

## ParNew 的调优要点

<Badge>GC Tuning</Badge>

ParNew 的调优主要关注年轻代大小和 GC 频率。年轻代太小会导致频繁的 Minor GC，每次 GC 的停顿虽然短，但累积起来也会影响响应。年轻代太大可能导致 Minor GC 时存活对象过多，停顿时间变长。

```bash
# 调优建议
# 1. 年轻代大小应该能够容纳大部分短期对象
# 2. Survivor 区比例应该足够，避免对象过早晋升老年代
# 3. 如果 ParNew GC 时间过长，考虑增大年轻代
# 4. 如果 Full GC 频繁，考虑调整年龄阈值
```

## ParNew 的历史地位

<Badge>JDK History</Badge>

ParNew 在 JVM 历史上扮演了重要角色。它是最早的年轻代并行收集器之一，为后续更先进的收集器奠定了基础。

随着 CMS 被标记为废弃，ParNew 的使用场景也大幅减少。JDK 9 之后，G1 成为默认收集器，ParNew + CMS 的组合被 G1 所取代。但在 JDK 8 及之前的项目中，ParNew + CMS 仍然是最常见的低停顿收集器组合之一。

## 面试高频考点

**问题一：ParNew 和 Serial 的区别是什么？** ParNew 是多线程版本，使用多个 GC 线程并行回收；Serial 是单线程版本。两者都使用复制算法，但 ParNew 在多核环境下效率更高。

**问题二：ParNew 主要与哪个收集器配合使用？** ParNew 主要与 CMS 收集器配合使用，作为 CMS 的年轻代收集器。

**问题三：如何设置 ParNew 的线程数量？** 通过 `-XX:ParallelGCThreads` 参数设置。这个参数影响所有并行收集器。

**问题四：ParNew 和 Parallel Scavenge 有什么区别？** 两者都是年轻代并行收集器，但设计目标不同。Parallel Scavenge 追求高吞吐量，使用自适应策略；ParNew 追求低停顿，专为配合 CMS 设计。

**问题五：什么情况下 ParNew GC 时间会变长？** 年轻代太大导致存活对象多、Survivor 区太小导致对象过早晋升、GC 线程数设置不当等情况都可能导致 ParNew GC 时间变长。

ParNew 作为 CMS 的黄金搭档，在 Java GC 历史上留下了浓墨重彩的一笔。虽然它已经被 G1 所取代，但它的设计理念——并行回收年轻代——仍然影响着现代收集器的设计。那么，为什么现代收集器不再需要专门的年轻代收集器了呢？
