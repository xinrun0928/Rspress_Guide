# G1：Region 化分代收集器

G1（Garbage First）收集器是 JVM 历史上最具创新性的收集器之一。它彻底重新定义了堆内存的分区方式，不再将堆严格划分为年轻代和老年代，而是将堆划分为大小相等的 Region。这种设计使得 G1 能够预测停顿时间，同时保持良好的吞吐量。G1 从 JDK 7 Update 4 开始支持，JDK 9 将其设为默认收集器，代表了现代垃圾收集器的发展方向。

## G1 的核心创新：Region

<Badge>G1</Badge>

G1 收集器的核心创新是将堆划分为大小相等的 Region。每个 Region 大小为 2 的幂次方，通常在 1MB 到 32MB 之间。一个 4GB 的堆大约有 2000-4000 个 Region。

Region 可以根据需要被配置为不同的角色：Eden Region（年轻代 Eden 区）、Survivor Region（年轻代 Survivor 区）、Old Region（老年代 Region）、Humongous Region（大对象区）。

```
G1 Region 划分示意:
┌────────────────────────────────────────────────────────────────────┐
│                        G1 Heap (4GB)                                │
├────────────────────────────────────────────────────────────────────┤
│  Region 1 │ Region 2 │ Region 3 │ Region 4 │ Region 5 │ ...      │
│   Eden     │  Eden     │ Survivor │  Old     │ Humongous │         │
├────────────────────────────────────────────────────────────────────┤
│  Region N │ Region N+1│ Region N+2│ Region N+3│ Region N+4│       │
│    Old     │  Old      │  Eden     │ Survivor  │   Eden   │        │
└────────────────────────────────────────────────────────────────────┘
特点: Region 大小相等，可动态配置角色
```

## G1 的工作原理

<Badge>Region</Badge>

G1 的工作过程与传统的分代收集器有显著不同。G1 根据期望的停顿时间目标，动态选择需要回收的 Region。

G1 的收集分为两种类型：Young GC（年轻代回收）和 Mixed GC（混合回收）。

Young GC 发生在年轻代 Region 空间不足时。G1 回收所有年轻代 Region（Eden 和 Survivor），将存活对象复制到 Survivor Region 或晋升到 Old Region。

Mixed GC 发生在老年代 Region 占用比例超过阈值时。G1 回收所有年轻代 Region 和部分老年代 Region。G1 会优先回收垃圾比例高的 Region（这就是"Garbage First"的含义）。

## Remembered Set

<Badge>Remembered Set</Badge>

G1 引入了 Remembered Set（记忆集）来解决 Region 间引用的问题。在传统的分代收集中，跨代引用只需要扫描脏卡片。但在 G1 中，Region 之间的引用更加复杂。

每个 Region 都有一个 Remembered Set，记录了其他 Region 对该 Region 中对象的引用。当 GC 需要扫描某个 Region 时，只需要扫描 Remembered Set 中记录的 Region，而不需要扫描整个堆。

Remembered Set 使用了写屏障技术来维护。当应用线程修改引用时，JVM 会自动更新相关 Region 的 Remembered Set。

```bash
# Remembered Set 相关的参数
-XX:+UseG1GC                    # 启用 G1
-XX:G1HeapRegionSize=4m         # Region 大小（1MB, 2MB, 4MB, 8MB, 16MB, 32MB）
```

## G1 的停顿时间预测

<Badge>Pause Prediction</Badge>

G1 使用停顿时间模型（Pause Prediction Model）来预测停顿时间。G1 维护了每个 Region 的回收成本估算，模型会根据期望的停顿时间目标，选择合适的 Region 组合进行回收。

```bash
# 停顿时间控制
-XX:MaxGCPauseMillis=200       # 设置期望的最大停顿时间（毫秒，默认 200）

# G1 会尽量满足这个目标，但并不保证
```

## Humongous Regions

<Badge>Humongous</Badge>

G1 为大对象设计了特殊的 Humongous Region。当对象大小超过 Region 大小的 50% 时，G1 会在 Humongous Region 中分配。Humongous Region 是连续的多个 Region，专门用于存储大对象。

Humongous 对象可能影响 G1 的性能。如果 Humongous 对象过多，会导致 Region 碎片化。G1 会在 Old GC 时检查 Humongous Region，回收不再使用的 Humongous 对象。

```bash
# 大对象阈值
-XX:G1HeapRegionSize=4m        # Region 大小
# 超过 Region 大小 50% 的对象视为 Humongous
```

## G1 的收集周期

<Badge>GC Cycle</Badge>

G1 的收集周期包括以下阶段：

1. **年轻代回收（Young GC）**：回收年轻代 Region。停顿时间通常较短。
2. **并发标记周期（Concurrent Marking）**：与 CMS 类似，G1 也进行并发标记。标记完成后，G1 计算每个 Region 的回收价值。
3. **混合回收（Mixed GC）**：回收年轻代和部分老年代 Region。
4. **Full GC**：如果混合回收无法跟上对象分配速度，G1 会触发 Full GC。

```
G1 收集周期:
┌─────────────────────────────────────────────────────────────────┐
│                                                                  │
│   Young GC ──→ (并发标记) ──→ Mixed GC ──→ Young GC ──→ ...    │
│     ↑                    │                    │                  │
│     │                    ↓                    │                  │
│     │              标记完成                   │                  │
│     │                                           │                  │
│     └────────────────────────── (Full GC) ←───┘                  │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## G1 的优势

<Badge>GC Advantage</Badge>

G1 相比 CMS 有几个显著优势。

第一，停顿时间可控。G1 通过停顿时间模型预测停顿时间，用户可以设置期望的最大停顿时间，G1 会尽量满足。

第二，无内存碎片。G1 使用复制算法回收 Region，回收后 Region 是紧凑的，不存在碎片问题。

第三，良好的整体性能。G1 既考虑了吞吐量，也考虑了停顿时间，在大多数场景下表现均衡。

## G1 的常用参数

<Badge>JVM 参数</Badge>

```bash
# 基本参数
-XX:+UseG1GC                   # 启用 G1

# 停顿时间控制
-XX:MaxGCPauseMillis=200       # 最大停顿时间目标
-XX:GCPauseIntervalMillis=5000 # 停顿间隔目标

# Region 大小
-XX:G1HeapRegionSize=4m        # Region 大小

# 年轻代比例
-XX:G1NewSizePercent=5         # 年轻代最小比例（默认 5%）
-XX:G1MaxNewSizePercent=60     # 年轻代最大比例（默认 60%）

# 老年代回收阈值
-XX:InitiatingHeapOccupancyPercent=45  # 老年代占用比例达到此值时触发并发标记

# 混合回收
-XX:G1MixedGCLiveThresholdPercent=85  # Old Region 存活对象比例低于此值时参与回收
-XX:G1HeapWastePercent=10            # 可容忍的浪费空间比例
```

## G1 的调优策略

<Badge>GC Tuning</Badge>

G1 的调优通常从设置停顿时间目标开始。根据应用的响应时间要求，设置合理的 `-XX:MaxGCPauseMillis` 值。

```bash
# G1 调优示例
java -XX:+UseG1GC \
     -XX:MaxGCPauseMillis=100 \      # 期望 100ms 内完成 GC
     -XX:G1HeapRegionSize=8m \        # Region 大小 8MB
     -XX:InitiatingHeapOccupancyPercent=50 \
     -Xms4g -Xmx4g \
     MyApplication
```

如果停顿时间目标无法满足，可以考虑增大堆大小、减小 Region 大小（更多 Region，粒度更细）、减少并发标记周期等策略。

## G1 与 CMS 的对比

<Badge>CMS vs G1</Badge>

| 特性 | CMS | G1 |
|------|-----|-----|
| 内存模型 | 传统分代 | Region 化 |
| 停顿时间 | 不可预测 | 可预测 |
| 内存碎片 | 有碎片 | 无碎片 |
| 内存占用 | 较低 | 较高（Remembered Set） |
| 吞吐量 | 中等 | 较好 |
| 适用场景 | 低延迟 | 低延迟 + 吞吐量平衡 |

## 面试高频考点

**问题一：G1 收集器的核心创新是什么？** G1 将堆划分为大小相等的 Region，不再严格区分年轻代和老年代，每个 Region 可以根据需要配置角色。

**问题二：什么是 Remembered Set？** Remembered Set 记录了其他 Region 对当前 Region 中对象的引用，用于优化 Region 间的引用扫描，避免扫描整个堆。

**问题三：G1 如何控制停顿时间？** G1 使用停顿时间预测模型，根据期望的最大停顿时间，动态选择需要回收的 Region 组合。

**问题四：Humongous Region 是什么？** 超过 Region 大小 50% 的对象称为 Humongous 对象，存储在连续的 Humongous Region 中。Humongous 对象过多可能影响性能。

**问题五：G1 和 CMS 的主要区别是什么？** G1 无内存碎片，停顿时间可控；CMS 有碎片问题，停顿时间不可预测。G1 是 JDK 9 之后的默认收集器。

**问题六：Young GC 和 Mixed GC 的区别是什么？** Young GC 只回收年轻代 Region；Mixed GC 回收年轻代和部分老年代 Region。

**问题七：G1 适合什么场景？** 适合大内存（>6GB）、需要可控停顿时间、追求吞吐量与延迟平衡的场景。

G1 代表着从传统分代到区域化的转变，它的设计理念深刻影响了后续的 ZGC 和 Shenandoah。那么，G1 的 Region 化设计相比传统分代有什么优势？又有什么新的挑战？
