# ZGC / Shenandoah：低延迟收集器

ZGC 和 Shenandoah 是 JVM 历史上划时代的低延迟垃圾收集器。它们代表了垃圾收集技术的最新成就，能够在保持高吞吐量的同时，将 GC 停顿时间控制在毫秒级甚至亚毫秒级。对于对延迟极其敏感的应用，如金融交易、游戏服务器、实时系统等，ZGC 和 Shenandoah 是改变游戏规则的选择。

## 停顿时间问题的本质

<Badge>Low Latency</Badge>

在 ZGC 和 Shenandoah 出现之前，所有垃圾收集器都有一个共同的问题：停顿时间与堆大小成正比。当堆内存达到数十 GB 时，一次 Full GC 可能导致数秒甚至更长的停顿。

问题的根源在于传统的垃圾回收算法需要在回收过程中暂停应用线程，以便安全地遍历对象图和移动对象。堆越大，需要处理的对象越多，停顿时间越长。

ZGC 和 Shenandoah 通过不同的技术手段解决了这个问题。它们实现了并发标记和并发压缩，使得停顿时间与堆大小无关，只与活跃对象数量有关。

## ZGC 的着色指针技术

<Badge>Colored Pointers</Badge>

ZGC 使用了着色指针（Colored Pointers）技术来实现并发操作。着色指针是在对象指针上附加额外的信息位，用于标记对象的 GC 状态。

ZGC 使用了 4 位颜色位（从 64 位指针中借用）：Marked0 / Marked1 用于标记位和并发标记、Remapped 表示重定位位以指示对象是否已移动、Marked0 和 Marked1 交替使用以避免 STAB 写屏障。

```
64位对象指针结构:
┌────────────────────────────────────────────────┐
│                    有效指针位                    │  ← 对象在堆中的地址
│────────────────────────────────────────────────│
│                    颜色位                       │  ← GC 状态
│  [45-46]   [47]      [48]      [49-63]        │
│  Marked0   Marked1  Remapped   (未使用)        │
└────────────────────────────────────────────────┘
```

当 GC 线程修改对象引用时，通过读屏障（Load Barrier）检查颜色位。当对象已被移动时，ZGC 会自动更新引用。这就是 ZGC 能够并发移动对象而不会导致引用失效的关键。

## ZGC 的工作阶段

<Badge>ZGC Phases</Badge>

ZGC 的收集周期包括三个主要阶段，全部都可以与应用线程并发执行。

第一阶段是并发标记（Concurrent Mark）。GC 线程遍历对象图，标记所有存活对象。应用线程在运行过程中可能创建新对象或修改引用，ZGC 使用 STAB 写屏障保证标记的准确性。

第二阶段是并发重定位（Concurrent Relocate）。GC 线程将存活对象移动到新的位置，更新对象引用的颜色位。应用线程通过读屏障自动发现并更新引用。

第三阶段是并发引用更新（Concurrent Reference Processing）。GC 线程更新各种引用类型（软引用、弱引用、虚引用等）的引用。

```
ZGC 的阶段示意:
┌─────────────────────────────────────────────────────────────────┐
│  初始标记 (STW)    ─── 极短，通常 < 1ms                         │
│  ↓                                                               │
│  并发标记        ─── 与应用并发，不停顿                         │
│  ↓                                                               │
│  再标记 (STW)    ─── 极短，通常 < 1ms                         │
│  ↓                                                               │
│  并发重定位      ─── 与应用并发，不停顿                         │
│  ↓                                                               │
│  并发引用处理    ─── 与应用并发，不停顿                         │
└─────────────────────────────────────────────────────────────────┘
```

## ZGC 的优势

<Badge>ZGC Advantage</Badge>

ZGC 的停顿时间通常在毫秒级以内，最大停顿时间不超过配置的阈值。这意味着即使在数百 GB 的堆内存下，应用也能保持流畅的响应。

ZGC 的吞吐量损失很小，通常在 10-15% 以内。相比 CMS 和 G1 的长时间停顿，这个代价是值得的。

ZGC 的内存占用也很优秀。使用着色指针技术，ZGC 不需要额外的记忆集等数据结构。

## Shenandoah 的 Brooks 指针

<Badge>Brooks Pointer</Badge>

Shenandoah 使用了与 ZGC 不同的技术方案。Shenandoah 在每个对象头部添加了一个转发指针（Forwarding Pointer），也称为 Brooks 指针。

转发指针指向对象的新位置。当对象被移动时，只需要更新转发指针。应用线程在访问对象时，首先检查转发指针，如果对象已移动，则通过转发指针访问新位置。

```
Brooks 指针示意:
旧位置:  [转发指针] → [对象数据]
              ↓
新位置:  [对象数据]
```

Shenandoah 的优点是实现相对简单，缺点是每个对象都额外占用 8 字节（64 位系统），增加了内存开销。

## Shenandoah 与 ZGC 的对比

<Badge>Shenandoah</Badge>

| 特性 | ZGC | Shenandoah |
|------|-----|------------|
| 指针技术 | 着色指针 | Brooks 指针 |
| 内存开销 | 无额外开销 | 对象头 +8 字节 |
| 堆内存上限 | 16TB (JDK 21+) | ~128GB |
| 支持的 JDK 版本 | JDK 11+ | JDK 12+ (JDK 15+ 正式) |
| 非堆内存回收 | 部分支持 | 支持 |
| 并发类卸载 | JDK 15+ 支持 | 支持 |
| 停顿时间 | < 1ms | < 1ms |

## ZGC 的常用参数

<Badge>JVM 参数</Badge>

```bash
# 启用 ZGC (JDK 11+)
java -XX:+UseZGC MyApplication

# 最大停顿时间目标
-XX:MaxGCPauseMillis=1        # 期望最大停顿 1 毫秒

# 堆大小
-Xms16g -Xmx16g               # ZGC 适合大内存场景

# GC 线程数
-XX:ConcGCThreads=8           # 并发 GC 线程数

# 启用类卸载 (JDK 15+)
-XX:+ClassUnloadingWithConcurrentMark  # ZGC 支持并发类卸载

# 日志
-XX:+PrintGCDetails
-Xlog:gc*:gc.log
```

## Shenandoah 的常用参数

<Badge>JVM 参数</Badge>

```bash
# 启用 Shenandoah (JDK 12+)
java -XX:+UseShenandoahGC MyApplication

# 最大停顿时间目标
-XX:MaxGCPauseMillis=1

# 堆大小
-Xms8g -Xmx8g

# 调优模式
-XX:ShenandoahGCMode=adaptive   # 自适应模式
-XX:ShenandoahGCHeuristics=interactive  # 交互模式

# 压缩模式
-XX:ShenandoahAlwaysPreTouch     # 预触摸堆
```

## ZGC 的实战调优

<Badge>GC Tuning</Badge>

ZGC 的调优相对简单，因为它本身就是为自动化设计。以下是一些实战建议。

```bash
# ZGC 调优示例
java -XX:+UseZGC \
     -XX:MaxGCPauseMillis=5 \      # 目标停顿 5ms
     -Xms32g -Xmx32g \             # 大内存场景
     -XX:+ZCollectionInterval=10 \ # 每 10 秒进行一次 GC
     -XX:+ClassUnloadingWithConcurrentMark \
     -XX:+PrintGCDetails \
     -Xlog:gc*:/var/log/gc.log \
     MyApplication
```

如果需要更精细的调优，可以调整并发 GC 线程数。对于 CPU 密集型应用，可以减少并发线程数以留出更多 CPU 给应用。

## 何时选择 ZGC / Shenandoah

<Badge>GC Selection</Badge>

ZGC 和 Shenandoah 适合以下场景。

第一，超大内存场景。堆内存超过 32GB 时，G1 的停顿时间可能超过可接受范围。ZGC 可以处理 TB 级别的堆，停顿时间仍然很短。

第二，极低延迟要求。停顿时间要求在毫秒级甚至更低时，ZGC 和 Shenandoah 是仅有的选择。

第三，对吞吐量也有要求。ZGC 和 Shenandoah 的吞吐量损失相对较小，可以同时满足延迟和吞吐量要求。

```bash
# 选择建议
# < 6GB 堆: G1 足够
# 6-32GB 堆: G1 或 ZGC 均可
# > 32GB 堆: 强烈建议使用 ZGC
# 极端低延迟: ZGC
```

## 面试高频考点

**问题一：ZGC 如何实现并发移动对象？** ZGC 使用着色指针技术，在对象指针中附加 GC 状态位。通过读屏障自动发现并更新引用，实现并发移动对象而不会导致引用失效。

**问题二：ZGC 和 Shenandoah 的停顿时间是多少？** 两者都可以实现 < 1ms 的停顿时间，且停顿时间与堆大小无关。

**问题三：着色指针和 Brooks 指针有什么区别？** 着色指针将信息编码在对象指针中，不增加对象大小。Brooks 指针在对象头部添加转发指针，增加 8 字节开销。

**问题四：ZGC 支持多大的堆内存？** JDK 15 及之前支持 8TB，JDK 16-20 支持 16TB，JDK 21+ 支持 16TB（实际可达 128TB+）。

**问题五：ZGC 的吞吐量如何？** ZGC 的吞吐量损失通常在 10-15% 以内，相比长时间停顿的代价是值得的。

**问题六：什么场景下应该使用 ZGC？** 超大内存（> 32GB）、极低延迟要求（< 10ms）、需要同时满足延迟和吞吐量的场景。

**问题七：JDK 哪个版本开始支持 ZGC？** ZGC 从 JDK 11 开始支持。JDK 15 之前是实验特性，JDK 15 开始正式支持。

ZGC 和 Shenandoah 代表了 GC 技术的最新成就，它们将停顿时间推向了极限。那么，既然 ZGC 如此强大，为什么还需要 Shenandoah？它们各有什么适用场景？
