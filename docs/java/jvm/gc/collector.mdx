# 垃圾收集器分类与组合关系图

如果说垃圾收集算法是理论，那么垃圾收集器就是实现这些理论的具体方案。JVM 发展至今，已经诞生了多款垃圾收集器，每款收集器都有其独特的设计理念、适用场景和性能特点。理解这些收集器以及它们之间的组合关系，是进行 JVM 调优的前提。

## 年轻代收集器

### Serial 收集器

<Badge>Serial GC</Badge>

Serial（串行）收集器是最基础、历史最悠久的收集器。它使用单线程进行垃圾收集，收集时必须暂停所有用户线程（Stop The World）。

**特点**：单线程收集、简单高效（没有线程切换开销）、适合单核 CPU 或小内存应用。

**参数**：`-XX:+UseSerialGC`

```
Serial 收集器工作示意:

用户线程: ──────────────────────────────────────────────────▶
                    │                              │
                    ▼                              ▼
              [GC 开始]                      [GC 结束]
              (Stop The World)               (继续执行)
                       ↑__________________________↑
                              GC 线程执行
```

### ParNew 收集器

<Badge>ParNew GC</Badge>

ParNew（并行新生代）收集器是 Serial 收集器的多线程版本。它使用多个线程并行进行垃圾收集，显著提升了收集效率。

**特点**：多线程并行收集、Server 模式下年轻代首选收集器、可以与 CMS 收集器配合使用。

**参数**：`-XX:+UseParNewGC`

```
ParNew 收集器工作示意:

用户线程: ──────────────────────────────────────────────────▶
           │     │     │                              │
           │     │     │                              │
GC 线程:    │     │     │                              │
(线程1) ────┼─────┼─────┼──────────────────────────────┼────▶
(线程2) ────┼─────┼─────┼──────────────────────────────┼────▶
(线程3) ────┼─────┼─────┼──────────────────────────────┼────▶
           ↑     ↑     ↑_______________________________↑
           │     │            多线程并行
           └──┬──┘
              │
         [Stop The World]
```

### Parallel Scavenge 收集器

<Badge>Parallel Scavenge</Badge>

Parallel Scavenge（并行清理）收集器专注于提高吞吐量（Throughput），即用户线程运行时间占总时间的比例。

**特点**：多线程并行收集、可控制吞吐量、自适应调节策略（不需要手动设置年轻代比例）、适合后台计算任务。

**关键参数**：`-XX:MaxGCPauseMillis`（最大停顿时间）、`-XX:GCTimeRatio`（GC 时间占总时间的比例）、`-XX:+UseAdaptiveSizePolicy`（启用自适应调节）。

**参数**：`-XX:+UseParallelGC`

## 老年代收集器

### Serial Old 收集器

<Badge>Serial Old</Badge>

Serial Old 是 Serial 收集器的老年代版本，使用标记-整理算法。

**特点**：单线程，标记-整理算法，适合小内存应用。

### Parallel Old 收集器

<Badge>Parallel Old</Badge>

Parallel Old 是 Parallel Scavenge 收集器的老年代版本，使用标记-整理算法，与 Parallel Scavenge 搭配形成"吞吐量优先"组合。

**特点**：多线程并行，标记-整理算法，JDK 1.6 开始支持。

**参数**：`-XX:+UseParallelOldGC`

```
吞吐量优先组合 (Parallel Scavenge + Parallel Old):

        年轻代                    老年代
    ┌──────────┐            ┌──────────────┐
    │Parallel  │            │ Parallel Old │
    │Scavenge  │ 晋升 ──▶   │              │
    │(复制算法) │            │(标记-整理)   │
    └──────────┘            └──────────────┘
```

### CMS 收集器

<Badge>CMS</Badge>

CMS（Concurrent Mark Sweep）收集器目标是获取最短回收停顿时间，采用标记-清除算法。

**特点**：并发收集（大部分过程与用户线程并发）、低停顿时间、标记-清除算法，会产生碎片、CPU 敏感（会占用 CPU 资源）。

**CMS 执行阶段**：初始标记（Initial Mark，STW）标记 GC Roots 直接引用的对象、并发标记（Concurrent Mark）从 GC Roots 出发标记存活对象、重新标记（Remark，STW）修正并发标记期间的变化、并发清除（Concurrent Sweep）清除死亡对象。

**参数**：`-XX:+UseConcMarkSweepGC`

```
CMS 收集器阶段示意:

时间: ──────────────────────────────────────────────────▶

用户线程: ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓
              │                          │                    │
              ▼                          ▼                    ▼
CMS:     [初始标记] [并发标记] [重新标记] [并发清除]
         (STW)                   (STW)

         停顿        并发            停顿        并发
```

### G1 收集器

<Badge>G1</Badge>

G1（Garbage First）收集器是 JDK 7 引入的全新收集器，JDK 9 开始成为默认收集器。它将堆划分为多个大小相等的 Region，采用混合收集策略。

**特点**：区域化设计（Region）、并发并行收集、可预测的停顿时间、优先回收垃圾最多的 Region、适合大内存、多核环境。

**参数**：`-XX:+UseG1GC`

```

┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  ┌────┐ ┌────┐ ┌────┐ ┌────┐ ┌────┐ ┌────┐ ┌────┐ ┌────┐    │
│  │Eden│ │Eden│ │S0  │ │Eden│ │Old │ │Old │ │Eden│ │S1  │    │
│  └────┘ └────┘ └────┘ └────┘ └────┘ └────┘ └────┘ └────┘    │
│                                                              │
│  ┌────┐ ┌────┐ ┌────┐ ┌────┐ ┌────┐ ┌────┐ ┌────┐ ┌────┐    │
│  │Old │ │S0  │ │Old │ │Old │ │Humg│ │Eden│ │Humg│ │Old │    │
│  └────┘ └────┘ └────┘ └────┘ └────┘ └────┘ └────┘ └────┘    │
│                                                              │
│  ┌────────────────────────────────────────────────────────┐  │
│  │               Humongous 区域（大对象）                   │  │
│  └────────────────────────────────────────────────────────┘  │
│                                                              │

```

## 低延迟收集器

### ZGC

<Badge>ZGC</Badge>

ZGC（Z Garbage Collector）是 JDK 11 引入的低延迟垃圾收集器，设计目标是将停顿时间控制在 10ms 以内。

**特点**：着色指针技术（Colored Pointers）、并发收集、极低停顿时间、停顿时间不随堆大小增加、适合超大内存应用。

**参数**：`-XX:+UseZGC`（JDK 11+）

### Shenandoah

<Badge>Shenandoah</Badge>

Shenandoah 是 Red Hat 开发的低延迟收集器，目标与 ZGC 类似。

**特点**：转发指针（Brooks Pointer）、并发整理、停顿时间短、与 G1 类似的 Region 设计。

**参数**：`-XX:+UseShenandoahGC`（JDK 12+，需要 OpenJDK）

## 收集器组合关系

<Badge>GC Combination</Badge>

不同的年轻代收集器和老年代收集器可以组合使用：

```
┌─────────────┬────────────────┬──────────────────────┐
│  年轻代     │    老年代       │      说明             │
├─────────────┼────────────────┼──────────────────────┤
│ Serial     │ Serial Old     │ 串行组合，最简单       │
│ ParNew     │ CMS           │ 并行+并发，最常见     │
│ Parallel   │ Parallel Old   │ 吞吐量优先组合        │
│ Scavenge   │                │                      │
│ G1         │ G1             │ 统一收集器，独立使用  │
│ ZGC        │ ZGC            │ 低延迟，独立使用      │
│ Shenandoah │ Shenandoah     │ 低延迟，独立使用      │
└─────────────┴────────────────┴──────────────────────┘

推荐组合（根据应用场景）:

低延迟优先 ──▶ G1 / ZGC / Shenandoah
高吞吐优先 ──▶ Parallel Scavenge + Parallel Old
简单稳定优先 ──▶ Serial + Serial Old
```

## 如何选择收集器

选择收集器需要根据应用的特点：

**如果是小内存（`<100MB`）或单核**：Serial + Serial Old，简单高效。

**如果追求高吞吐量**：Parallel Scavenge + Parallel Old，适合后台批处理任务。

**如果追求低延迟**：G1（中等延迟）或 ZGC/Shenandoah（极低延迟），适合 Web 应用。

**如果是大内存**：G1 或 ZGC，适合几十 GB 以上堆内存的应用。

**考虑因素**：停顿时间要求、吞吐量要求、内存大小、CPU 核心数、JDK 版本。

## 面试高频考点

**问题一：有哪些垃圾收集器？** Serial、ParNew、Parallel Scavenge（年轻代），Serial Old、Parallel Old、CMS（老年代），G1、ZGC、Shenandoah（统一收集器）。

**问题二：CMS 收集器的工作过程是什么？** 四个阶段：初始标记（STW）、并发标记、重新标记（STW）、并发清除。

**问题三：G1 收集器的特点是什么？** Region 化设计、优先回收垃圾最多的 Region、可预测停顿时间、JDK 9+ 默认收集器。

**问题四：Serial 和 ParNew 的区别是什么？** Serial 单线程，ParNew 多线程并行收集。在多核环境下 ParNew 效率更高。

**问题五：吞吐量优先用什么收集器组合？** Parallel Scavenge + Parallel Old，通过 -XX:MaxGCPauseMillis 和 -XX:GCTimeRatio 控制吞吐量。

从 Serial 到 ZGC，垃圾收集器经历了从串行到并行、从停顿到并发的演进。每种收集器都有其适用场景，选择时需要权衡停顿时间、吞吐量、内存大小等因素。那么，如果让你为一个在线游戏服务器选择垃圾收集器，你会如何权衡？
