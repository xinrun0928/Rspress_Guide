# 垃圾收集器分类与组合关系图

如果说垃圾收集算法是理论，那么垃圾收集器就是实现这些理论的具体方案。JVM 发展至今，已经诞生了多款垃圾收集器，每款收集器都有其独特的设计理念、适用场景和性能特点。理解这些收集器以及它们之间的组合关系，是进行 JVM 调优的前提。

## 年轻代收集器

### Serial 收集器

|Serial GC|

Serial（串行）收集器是最基础、历史最悠久的收集器。它使用单线程进行垃圾收集，收集时必须暂停所有用户线程（Stop The World）。

**特点**：单线程收集、简单高效（没有线程切换开销）、适合单核 CPU 或小内存应用。

**参数**：`-XX:+UseSerialGC`

### ParNew 收集器

|ParNew GC|

ParNew 是 Serial 的多线程版本，使用多个线程并行进行年轻代的垃圾收集。它是 CMS 收集器在 JDK 7 之前的默认年轻代收集器。

**特点**：多线程并行收集、可以与 CMS 配合使用、停顿时间比 Serial 更短。

**参数**：`-XX:+UseParNewGC`

### Parallel Scavenge 收集器

|Parallel Scavenge|

Parallel Scavenge 收集器也是多线程并行收集，但它的设计目标与其他收集器不同。其他收集器追求减少停顿时间，而 Parallel Scavenge 追求高吞吐量。

**特点**：吞吐量优先、自适应调节策略（GC 自动优化）、适合后台批处理任务。

**参数**：`-XX:+UseParallelGC`

## 老年代收集器

### Serial Old 收集器

|Serial Old|

Serial Old 是 Serial 的老年代版本，使用标记-整理算法。它是 Serial 收集器的老年代搭档。

**特点**：单线程、简单高效、适合客户端模式。

### Parallel Old 收集器

|Parallel Old|

Parallel Old 是 Parallel Scavenge 的老年代搭档，使用标记-整理算法。它是多线程并行收集，追求高吞吐量。

**特点**：吞吐量优先、与 Parallel Scavenge 组合使用、JDK 8 的默认老年代收集器。

**参数**：`-XX:+UseParallelOldGC`

### CMS 收集器

|CMS|

CMS（Concurrent Mark Sweep）收集器追求最短停顿时间，采用并发标记-清除算法。它是第一个实现并发收集的老年代收集器。

**特点**：并发收集、停顿时间短、会产生内存碎片、需要额外空间存储浮动垃圾。

**参数**：`-XX:+UseConcMarkSweepGC`

## 整堆收集器

### G1 收集器

|G1|

G1（Garbage First）收集器是 JDK 7 引入的新一代收集器，在 JDK 9 中成为默认收集器。它采用分区（Region）的概念，将堆划分为多个大小相等的 Region。

**特点**：分区收集、停顿时间可预测、可并发执行、适合大内存应用。

**参数**：`-XX:+UseG1GC`

### ZGC 收集器

|ZGC|

ZGC（Z Garbage Collector）是 JDK 11 引入的低延迟收集器，设计目标是停顿时间不超过 10ms，且停顿时间不随堆大小增加而增加。

**特点**：亚毫秒级停顿、染色指针、读屏障、可扩展到 TB 级堆。

**参数**：`-XX:+UseZGC`

### Shenandoah 收集器

|Shenandoah|

Shenandoah 是 Red Hat 开发的低延迟收集器，与 ZGC 类似，但实现方式不同。它采用转发指针（Forwarding Pointer）而不是染色指针。

**特点**：停顿时间短、OpenJDK 特有、与 G1 类似但停顿时间更短。

**参数**：`-XX:+UseShenandoahGC`

## 收集器的组合关系

不同的年轻代收集器和老年代收集器可以组合使用，但不是所有组合都是合法的：

|年轻代 | 老年代 | 是否支持 | 说明 |
|---|---|---|---|
| Serial | Serial Old | 支持 | 最经典的组合，适合客户端模式 |
| Serial | CMS | 不支持 | CMS 不支持 Serial 作为年轻代收集器 |
| ParNew | CMS | 支持 | 经典组合，JDK 7 之前的默认配置 |
| ParNew | Serial Old | 支持 | 不推荐，会导致 Serial Old 单线程整理 |
| Parallel Scavenge | Serial Old | 支持 | 不推荐，Parallel Scavenge 的自适应策略与 Serial Old 不兼容 |
| Parallel Scavenge | Parallel Old | 支持 | JDK 8 默认组合，吞吐量优先 |
| G1 | G1 | 支持 | JDK 9+ 默认，无需指定老年代收集器 |
| 任何 | ZGC | 支持 | ZGC 是整堆收集器，独立工作 |
| 任何 | Shenandoah | 支持 | Shenandoah 是整堆收集器，独立工作 |

```
常见组合示意:

Serial + Serial Old:
┌────────────────┐      ┌────────────────┐
│  Serial (年轻代) │ + │ Serial Old (老年代) │
└────────────────┘      └────────────────┘

ParNew + CMS (已废弃):
┌────────────────┐      ┌────────────────┐
│  ParNew (年轻代) │ + │  CMS (老年代)   │
└────────────────┘      └────────────────┘

Parallel Scavenge + Parallel Old:
┌────────────────────┐   ┌────────────────────┐
│ Parallel Scavenge │ + │ Parallel Old (老年代)│
│     (年轻代)       │   └────────────────────┘
└────────────────────┘

G1 (整堆收集):
┌────────────────────────┐
│         G1             │
│  (年轻代 + 老年代)      │
└────────────────────────┘
```

## 如何选择收集器

选择收集器需要根据应用的特点和需求：

如果应用是客户端模式或小内存应用（小于 100MB），使用 Serial + Serial Old 就足够了。

如果应用追求高吞吐量，是批处理或科学计算类应用，使用 Parallel Scavenge + Parallel Old。

如果应用对停顿时间敏感，是 Web 应用或交互式应用，使用 ParNew + CMS 或 G1。

如果应用是大内存应用（大于 6GB）且对停顿时间要求极高，使用 ZGC 或 G1。

如果应用是 JDK 11+，且追求低延迟，推荐使用 ZGC。

```bash
# 选择不同收集器的示例

# 客户端模式（默认配置）
java -Xmx256m MyApplication

# 吞吐量优先
java -XX:+UseParallelGC -XX:+UseParallelOldGC -Xmx256m MyApplication

# 停顿时间优先
java -XX:+UseG1GC -XX:MaxGCPauseMillis=100 -Xmx256m MyApplication

# 低延迟（需要 JDK 11+）
java -XX:+UseZGC -XX:MaxGCPauseMillis=10 -Xmx8g MyApplication
```
