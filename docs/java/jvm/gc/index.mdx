# JVM 垃圾回收：内存管理的艺术

在 Java 程序运行过程中，对象被不断创建，也应该被适时销毁。在 C/C++ 中，内存管理完全由程序员负责，手动 `malloc` 和 `free` 容易导致内存泄漏和野指针问题。Java 引入垃圾回收（Garbage Collection，GC）机制，让程序员从繁琐的内存管理中解放出来，专注于业务逻辑的实现。然而，理解 GC 原理对于写出高效、少 GC 的代码，以及排查生产环境中的内存问题至关重要。

## 为什么要垃圾回收？

<Badge>GC Roots</Badge>

在讨论 GC 之前，先思考一个问题：哪些对象需要被回收？

当一个对象不再被任何"活着的"引用指向时，它就变成了"垃圾"——不再有用，但仍然占据着内存空间。这些垃圾如果不清理，会逐渐耗尽可用内存，导致 OutOfMemoryError。

```java
public void process() {
    Object obj = new Object();  // 创建对象
    // ... 使用 obj
    obj = null;  // 不再引用，变为垃圾
    // obj 指向的对象现在可以被回收
}
```

GC 的任务就是：自动识别并回收这些无用的对象，释放内存空间。

## 如何判断对象已死？

<Badge>Reachability Analysis</Badge>

GC 需要判断哪些对象是"活着的"（可达），哪些是"死去的"（不可达）。有两种主要算法：

**引用计数法**：每个对象有一个引用计数器，引用增加时 +1，引用消失时 -1。计数器为 0 时，对象已死。

```java
Object obj1 = new Object();  // refCount = 1
Object obj2 = obj1;           // refCount = 2
obj1 = null;                  // refCount = 1
obj2 = null;                  // refCount = 0，可以回收
```

优点：实现简单，判定效率高。缺点：无法处理循环引用。

```
A 引用 B，B 引用 A，但都没有其他外部引用
引用计数法认为 A、B 的计数都是 1，无法回收
但实际上 A、B 都已经不可达了
```

**可达性分析算法**：这是 Java 使用的主流算法，从"GC Roots"出发，沿着引用链向下搜索，能到达的对象是活的，不能到达的是垃圾。

```
GC Roots 包括：
- 虚拟机栈（栈帧中的本地变量表）中引用的对象
- 方法区中类静态属性引用的对象
- 方法区中常量引用的对象
- 本地方法栈中 JNI（Native 方法）引用的对象
- JVM 内部引用（Class 对象、异常对象、系统类加载器等）
- 同步锁持有的对象
- 反映 JVM 内部情况的 JMXBean、Callback 等
```

## 四种引用类型

<Badge>Strong Reference</Badge> <Badge>Soft Reference</Badge> <Badge>Weak Reference</Badge> <Badge>Phantom Reference</Badge>

Java 提供了四种引用强度不同的引用类型，让开发者可以更灵活地控制对象回收时机：

**强引用**（Strong Reference）是最常见的引用类型，只要强引用存在，对象就不会被回收。

```java
Object obj = new Object();  // 强引用
// obj 存在，Object 对象不会被回收
```

**软引用**（Soft Reference）关联的对象，在内存不足时会被回收。适合实现内存敏感的缓存。

```java
SoftReference<Object> softRef = new SoftReference<>(new Object());
Object obj = softRef.get();  // 可能返回 null
```

**弱引用**（Weak Reference）关联的对象，只能存活到下一次 GC。无论内存是否充足，下次 GC 时都会被回收。

```java
WeakReference<Object> weakRef = new WeakReference<>(new Object());
// 下次 GC 时，弱引用引用的对象就会被回收
```

`ThreadLocal` 的实现就使用了弱引用来避免内存泄漏。

**虚引用**（Phantom Reference）几乎不影响对象的生命周期，唯一的作用是在对象被回收时收到通知。

```java
ReferenceQueue<Object> queue = new ReferenceQueue<>();
PhantomReference<Object> phantomRef = new PhantomReference<>(new Object(), queue);
// 对象被回收后，phantomRef 会被加入 queue
```

虚引用常用于追踪对象的销毁。

## 分代收集理论：经验与实践的结晶

<Badge>Generational GC</Badge>

### 为什么需要分代？

不同对象的生命周期差异很大：大多数对象是朝生夕死的（比如方法中的局部变量）；少数对象会存活很长时间（比如缓存对象、单例对象）。

如果对所有对象采用统一的回收策略，要么效率低下（扫描所有对象），要么浪费空间（长生命对象被反复回收）。

### 分代假说

JVM 采用了分代收集理论，基于两个经验假说：

1. **弱分代假说**：大多数对象朝生夕死
2. **强分代假说**：熬过越多次 GC 的对象，越难死亡

基于这些假说，JVM 将堆内存分为新生代和老年代：

```
+---------------------------------------------------+
|                      堆内存                        |
+------------------------+--------------------------+
|         新生代          |          老年代          |
+------------+-----------++------------+-----------+
|  Eden  | S0  | S1     |            |            |
|        |     |        |            |            |
+------------+-----------+            +------------+
```

- **新生代**：存放新创建的对象，采用复制算法
- **老年代**：存放经过多次 GC 仍然存活的对象，采用标记-清除或标记-整理算法

## 垃圾收集算法

<Badge>GC Algorithm</Badge>

### 复制算法（Copying）

将内存分为两块，每次只使用一块。GC 时，将存活的对象复制到另一块，然后清空原来的那块。

```
|  空间 A   |  空间 B  |
| [存活对象] |          |
    GC 时：
|           | [存活对象]|
    交换指针
|           | [存活对象]|
```

优点：实现简单，没有内存碎片。缺点：可用内存减半。

现代 JVM 将新生代分为一个 Eden 区和两个 Survivor 区（S0、S1），使用率是 8:1:1。

### 标记-清除算法（Mark-Sweep）

分为标记和清除两个阶段：

```
标记阶段：遍历所有 GC Roots，标记存活对象
清除阶段：遍历整个堆，清除未标记的对象

[已标记] [未标记] [已标记] [未标记]
         ↓ 清除
[已标记]          [已标记]
```

优点：不需要额外空间。缺点：产生内存碎片，导致连续大对象分配失败。

### 标记-整理算法（Mark-Compact）

在标记-清除的基础上，增加了整理阶段：

```
[已标记] [未标记] [已标记] [未标记]
         ↓ 整理
[已标记] [已标记]
```

优点：没有内存碎片。缺点：整理需要移动大量对象，开销较大。

## 常见垃圾收集器

<Badge>GC Collector</Badge>

### Serial GC：单线程的怀旧派

最古老、最简单的垃圾收集器，使用单线程进行垃圾收集。在用户线程运行时，必须暂停所有其他线程（Stop The World，STW）。

```java
-XX:+UseSerialGC
```

适用于客户端应用或小型应用，堆内存不大（几十MB）。

### ParNew GC：多线程的新生代

Serial 的多线程版本，与 CMS 配合使用新生代。

```java
-XX:+UseParNewGC
```

### Parallel GC：追求吞吐量

专注于提高系统吞吐量（用户代码时间 / 总时间）。

```java
-XX:+UseParallelGC  // 新生代 Parallel Scavenge
-XX:+UseParallelOldGC  // 老年代
```

适合后台计算任务，不需要太多交互。

### CMS GC：追求低延迟

Concurrent Mark Sweep，以低延迟为目标的收集器。GC 过程尽可能与用户线程并发执行。

```java
-XX:+UseConcMarkSweepGC
```

```
初始标记 (STW) -> 并发标记 -> 重新标记 (STW) -> 并发清除
```

缺点：产生内存碎片；无法处理浮动垃圾；并发阶段占用 CPU。

### G1 GC：面向未来的收集器

<Badge>G1</Badge>

Garbage First，将堆划分为多个大小相等的 Region，优先回收垃圾最多的区域。

```java
-XX:+UseG1GC
```

适合大内存（6GB+）应用，追求高概率的停顿时间控制。

### ZGC 和 Shenandoah：新时代低延迟收集器

<Badge>JDK 11</Badge> <Badge>JDK 12</Badge>

JDK 11+ 引入的 ZGC 和 JDK 12+ 的 Shenandoah，都以极低的停顿时间为目标（< 1ms 或 < 10ms）。

```java
-XX:+UseZGC
-XX:+UseShenandoahGC
```

```

```

GC 日志是排查内存问题的关键工具。

## 本章小结

理解 GC 原理对于写出高效 Java 代码至关重要：

1. **理解对象存活判断**：可达性分析 + 四种引用类型
2. **理解分代收集**：新生代（复制算法）+ 老年代（标记-整理）
3. **理解 GC 算法**：复制、标记-清除、标记-整理
4. **理解收集器选择**：根据应用特点选择合适的收集器

你是否曾被频繁的 Full GC 困扰过？理解 GC 原理是解决这类问题的第一步。那么，如果让你设计一个垃圾收集器，你会采用什么样的策略？
