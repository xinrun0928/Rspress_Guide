# GC 判断对象存活算法：引用计数 vs 可达性分析

判断一个对象是否还活着，是垃圾回收的第一步。这个看似简单的问题，背后却蕴含着精妙的设计思想和无数次的技术演进。从引用计数到可达性分析，JVM 设计者们走了一条曲折但正确的道路。理解这两种算法的原理和差异，是理解整个垃圾回收系统的基础。

## 引用计数法：简单但有致命缺陷

<Badge>Reference Counting</Badge>

引用计数法（Reference Counting）是最早也是最直觉的存活判断算法。它的核心思想非常直接：每个对象都有一个引用计数器，每当有一个地方引用它时计数器加 1，每当一个引用失效时计数器减 1。当计数器为 0 时，表示没有任何引用指向该对象，对象死亡，可以回收。

引用计数法有一个显著的优点：判定效率极高。计数器在引用创建和销毁时同步更新，不需要遍历整个内存区域。当计数器归零时，可以立即回收对象，不需要像其他算法那样等待 GC 时刻。

然而，引用计数法有一个致命的缺陷：无法处理循环引用。考虑这样的场景：对象 A 引用对象 B，对象 B 引用对象 A，但除此之外没有任何其他引用指向 A 和 B。从程序逻辑上讲，A 和 B 都是不可达的，应该被回收。但由于它们互相引用，各自的计数器都是 1，无法被回收。这就是循环引用导致的内存泄漏。

```
对象A ←──── 引用计数: 1
  ↓                  （对象B引用A）
  └→ 对象B ←─── 引用计数: 1
              （对象A引用B）

外部引用: 无
但A、B的引用计数都不是0，无法回收！
```

Python 就使用了引用计数法来管理对象内存。为了解决循环引用的问题，Python 额外引入了标记-清除的机制来处理循环引用。每隔一段时间，Python 的 GC 线程会遍历所有对象，标记出不可达的对象，然后清除它们。这就是为什么 Python 对象有时会出现非确定性的延迟回收。

## 可达性分析算法：现代 JVM 的选择

<Badge>Reachability Analysis</Badge> <Badge>GC Roots</Badge>

现代 JVM 普遍采用可达性分析（Reachability Analysis）算法来判断对象的存活状态。这个算法的核心思想是：通过一系列"GC Roots"作为起始点，从这些点出发，沿着引用链向下搜索，能够被搜索到的对象就是存活的，无法被搜索到的对象就是垃圾。

GC Roots 是一组特殊的对象，它们被认为是必然存活的对象，不会被回收。以 GC Roots 为起点进行搜索，所有能够到达的对象都是可达的，都是存活对象；无法到达的对象都是不可达的，都是垃圾。

GC Roots 的选择有严格的标准。它们必须是逻辑上不可能被回收的对象。在 JVM 规范中，GC Roots 包括以下几类：

第一类是虚拟机栈（栈帧中的本地变量表）中引用的对象。虚拟机栈中的局部变量表存储了正在运行的方法中使用到的对象引用。这些对象正在被方法使用，显然不能被回收。例如，正在执行的方法中的局部变量、引用的对象数组等。

第二类是方法区中类静态属性引用的对象。静态属性属于类本身，而不是某个对象。类可能被加载后一直存在，静态属性引用的对象也一直存活。例如，类的静态 final 引用、基本类型的包装类对象等。

第三类是方法区中常量引用的对象。常量池中存储的字符串常量、类名、方法名等，这些常量引用的对象通常也是长期存在的。

第四类是本地方法栈中 JNI（即本地方法接口）引用的对象。当 Java 代码调用 native 方法时，native 方法可能持有 Java 对象的引用，这些引用同样是 GC Roots 的一部分。

第五类是 JVM 内部引用的对象，如系统类加载器、常驻类对象（异常类、系统类等）、用于监视的 Java 对象（JVMTI 代理、JVMTI 监听器等）。

第六类是所有被同步锁（synchronized 关键字）持有的对象。正在等待一个对象锁的线程持有的对象引用，也是 GC Roots 的一部分。

```
                    GC Roots
                       │
         ┌─────────────┼─────────────┐
         ↓             ↓             ↓
      虚拟机栈      方法区静态      本地方法栈
      局部变量       属性引用        JNI引用
         │             │             │
         ↓             ↓             ↓
      对象A ←────────┐  对象C         对象E
         │          │    │             │
         ↓          │    ↓             ↓
      对象B ←───────┼→ 对象D          ...
         │          │
         └──────────┘
         
能到达GC Roots的对象: A, B, C, D (存活)
不能到达GC Roots的对象: E (或其他未显示的对象) (垃圾)
```

## 可达性分析的标记过程

<Badge>Marking</Badge>

可达性分析的标记过程可以分为几个阶段。首先是初始标记阶段（Initial Mark），这个阶段需要 Stop-The-World。GC 线程从 GC Roots 出发，标记所有直接可达的对象。这个阶段通常很快，因为只标记一层。

然后是并发标记阶段（Concurrent Mark），这个阶段与应用线程并发运行。GC 线程从初始标记阶段标记的对象出发，沿着引用链遍历所有可达的对象。这个阶段是并发的，不需要 Stop-The-World，所以应用线程可以继续运行。

接下来是重新标记阶段（Remark），这个阶段需要 Stop-The-World。并发标记阶段中，应用线程可能修改了引用关系，导致有些对象被遗漏或错误标记。重新标记阶段需要修正这些变化。

最后是并发清除阶段（Concurrent Sweep），这个阶段与应用线程并发运行。GC 线程清除所有未被标记的对象，回收它们的内存空间。

这种多阶段的标记过程是现代垃圾收集器（如 CMS、G1）常用的模式。并发标记减少了 Stop-The-World 的时间，但同时也增加了实现的复杂性和 CPU 开销。

## 对象的引用类型与可达性

<Badge>Reference Type</Badge>

在讨论可达性分析时，必须理解 Java 中引用的不同类型。Java 提供了四种引用强度不同的引用类型，从强到弱依次是：强引用（Strong Reference）、软引用（Soft Reference）、弱引用（Weak Reference）、虚引用（Phantom Reference）。

强引用是日常生活中最常见的引用类型，如 `Object obj = new Object()`。只要强引用还存在，垃圾回收器就永远不会回收被引用的对象。

软引用描述一些还有用但非必需的对象。在内存足够时，软引用对象不会被回收；只有在内存快要耗尽时，软引用对象才会被回收。软引用可以用于实现内存敏感的缓存。

弱引用比软引用更弱，只能存活到下一次垃圾回收之前。无论内存是否充足，下次 GC 时弱引用对象都会被回收。弱引用常用于实现规范化映射（如 WeakHashMap）。

虚引用是最弱的引用，完全不影响对象的生存时间。虚引用的唯一作用是在对象被回收时收到一个系统通知。虚引用主要用于追踪对象的回收活动。

## 不可达对象的回收过程

<Badge>Finalization</Badge>

即使一个对象通过可达性分析被判定为不可达，它也不会立即被回收。对象从不可达到真正被回收，还需要经历 finalization 的过程。

当对象被判定为不可达时，如果对象重写了 `Object` 的 `finalize()` 方法且该方法尚未被调用，GC 会将该对象放入一个 F-Queue 中。稍后，GC 会触发 Finalizer 线程执行 F-Queue 中对象的 finalize 方法。这是对象最后一次拯救自己的机会——如果在 finalize 方法中，对象将自己或引用链上的任何一个对象与 GC Roots 关联起来，对象就可以逃脱被回收的命运。

但 finalize 方法的执行时机不确定，且一个对象只能执行一次 finalize 方法。finalize 方法是 Java 早期版本的遗产，现代编程中强烈不推荐使用。

```java
public class FinalizeEscapeDemo {
    public static FinalizeEscapeDemo SAVE_HOOK = null;
    
    @Override
    protected void finalize() throws Throwable {
        super.finalize();
        System.out.println("finalize 方法被执行");
        // 将自己重新与 GC Roots 关联
        SAVE_HOOK = this;
    }
    
    public static void main(String[] args) throws InterruptedException {
        SAVE_HOOK = new FinalizeEscapeDemo();
        
        // 第一次 GC
        SAVE_HOOK = null;
        System.gc();
        Thread.sleep(500);
        
        if (SAVE_HOOK != null) {
            System.out.println("对象逃脱了第一次回收");
        }
        
        // 第二次 GC
        SAVE_HOOK = null;
        System.gc();
        Thread.sleep(500);
        
        if (SAVE_HOOK != null) {
            System.out.println("对象逃脱了第二次回收");
        } else {
            System.out.println("对象被回收了");
        }
    }
}
```

## 引用计数与可达性分析的实际应用

<Badge>Python</Badge> <Badge>Redis</Badge>

虽然现代 JVM 主要使用可达性分析，但引用计数在某些场景下仍有应用。Redis 就使用了引用计数来管理内存——每个 Redis 对象都有一个引用计数器，计数器归零时立即释放内存。引用计数的即时性对于 Redis 这种对延迟敏感的系统非常有价值。

Python 结合了引用计数和标记-清除两种方法。引用计数提供即时回收，标记-清除定期处理循环引用。这种混合策略在 Python 的具体实现中取得了很好的平衡。

对于大多数 Java 程序员来说，理解可达性分析就足够了。但了解引用计数的原理有助于理解为什么 Java 选择可达性分析，以及为什么 Python 需要额外的循环引用处理机制。

## 面试高频考点

**问题一：为什么现代 JVM 使用可达性分析而不是引用计数？** 引用计数无法处理循环引用问题。当两个对象互相引用但不再被外部引用时，引用计数不为 0，导致内存泄漏。可达性分析从 GC Roots 出发，能够正确处理循环引用。

**问题二：GC Roots 包括哪些？** 虚拟机栈中引用的对象、方法区静态属性引用的对象、方法区常量引用的对象、本地方法栈 JNI 引用的对象、JVM 内部引用（系统类加载器等）、同步锁持有的对象。

**问题三：对象从创建到回收经历哪些阶段？** 创建阶段（分配内存、初始化、设置对象头）→ 应用阶段（对象被使用）→ 不可达阶段（通过可达性分析判定不可达）→ Finalization 阶段（可选，执行 finalize 方法）→ 回收阶段（内存被回收）。

**问题四：为什么重写了 finalize 方法后对象可能逃脱回收？** 因为 finalize 方法给了对象一次将自己与 GC Roots 重新建立关联的机会。如果在 finalize 中将 this 赋值给静态变量或局部变量，对象就重新变成可达的了。但一个对象只能逃脱一次回收。

**问题五：四种引用类型有什么区别？** 强引用永远不会回收；软引用在内存不足时回收；弱引用在下次 GC 时回收；虚引用不影响回收，仅用于接收通知。

**问题六：Stop-The-World 是什么？** 在进行 GC 时，需要暂停所有应用线程（Stop-The-World），然后才能安全地进行内存扫描和回收。CMS 和 G1 等并发收集器试图减少 STW 的时间。

从引用计数到可达性分析，JVM 的设计者们经历了一次重要的技术演进。这次演进解决了什么问题？又带来了什么新的挑战？理解这段历史，能帮助我们更好地理解现代 GC 系统的设计哲学。那么，为什么有些系统（如 Python）仍然选择使用引用计数，而不是完全采用可达性分析呢？
