# 垃圾收集算法

垃圾收集算法是 GC 的核心。从 1960 年 Lisp 语言诞生至今，垃圾收集领域发展出了多种算法，包括标记-清除、复制、标记-整理等。每种算法都有其优势和局限，适用于不同的场景。理解这些算法的原理，是理解各种垃圾收集器设计的基础。

## 标记-清除算法：最基础的算法

|标记-清除|

标记-清除（Mark-Sweep）算法是最早的垃圾收集算法，分为"标记"和"清除"两个阶段。

**标记阶段**：从 GC Roots 出发，标记所有存活的对象。

**清除阶段**：遍历整个堆内存，回收所有未被标记的对象。

```
标记-清除算法执行过程:

标记前:
┌────────────────────────────────────────────────┐
│  ██ ░░ ██ ██ ░░ ░░ ██ ░░ ██ ░░ ██ ░░ ██ ░░ │
│  存活 死亡 存活 存活 死亡 死亡 存活 死亡 存活 │
└────────────────────────────────────────────────┘
        ↓ 标记阶段
        ↓
标记后:
┌────────────────────────────────────────────────┐
│  ██ ░░ ██ ██ ░░ ░░ ██ ░░ ██ ░░ ██ ░░ ██ ░░ │
│   ↑      ↑  ↑       ↑     ↑     ↑      ↑     │
│  存活标记存活存活  存活 存活 存活 存活 存活   │
└────────────────────────────────────────────────┘
        ↓ 清除阶段
        ↓
清除后:
┌────────────────────────────────────────────────┐
│  ██ ░░ ░░ ██ ░░ ░░ ░░ ░░ ██ ░░ ██ ░░ ██ ░░ │
│  存活    存活     存活     存活 存活 存活     │
│           ↑              ↑              ↑     │
│        死亡回收        死亡回收        死亡回收│
└────────────────────────────────────────────────┘

问题：产生内存碎片
```

**优点**：实现相对简单、不需要额外的内存空间（与复制算法相比）。

**缺点**：**内存碎片化**：清除后产生大量不连续的内存碎片，可能导致明明总空闲内存足够，却无法分配大对象的情况、**效率问题**：标记和清除过程的效率都不高。

## 复制算法：解决碎片问题

|复制算法|

复制（Copying）算法将内存划分为两个相等的半区，每次只使用其中一个半区。当这个半区空间用完时，将存活的对象复制到另一个半区，然后一次性清除当前半区的所有空间。

```
复制算法执行过程:

初始状态:
┌─────────────────────┬─────────────────────┐
│       From 区        │        To 区        │
│   存活 死亡 存活     │        空闲         │
│    ↑      ↑         │                     │
│   存活标记存活       │                     │
└─────────────────────┴─────────────────────┘
        ↓ 复制阶段
        ↓
复制后:
┌─────────────────────┬─────────────────────┐
│       From 区        │        To 区        │
│                      │   存活 存活          │
│                      │    ↑      ↑         │
│                      │   复制   复制        │
└─────────────────────┴─────────────────────┘
        ↓ 清除阶段
        ↓
清除后:
┌─────────────────────┬─────────────────────┐
│       From 区        │        To 区        │
│      全部清空        │    存活 存活         │
└─────────────────────┴─────────────────────┘
```

**优点**：实现简单、运行高效（只需移动指针）、没有内存碎片。

**缺点**：**内存利用率低**：每次只能使用一半的内存、**存活对象多时复制成本高**：如果存活对象较多，复制开销会很大。

复制算法非常适合年轻代。因为年轻代中的对象大部分都是"朝生夕灭"的，存活对象通常很少（默认只有 10%），复制成本很低。这也是为什么年轻代使用复制算法的原因。

## 标记-整理算法：兼顾效率与空间

|标记-整理|

标记-整理（Mark-Compact）算法是在标记-清除算法的基础上发展而来的，解决了内存碎片化的问题。它的执行过程分为三个阶段：标记阶段（与标记-清除相同）、整理阶段（将存活对象向一端移动）、清除阶段（清除边界外的内存）。

**标记阶段**：从 GC Roots 出发，标记所有存活的对象。

**整理阶段**：将所有存活对象向一端移动，紧凑排列。

**清除阶段**：清除边界外的内存。

```
标记-整理算法执行过程:

整理前:
┌────────────────────────────────────────────────┐
│  ██ ░░ ██ ██ ░░ ░░ ██ ░░ ██ ░░ ██ ░░ ██ ░░ │
│  存活 死亡 存活 存活 死亡 死亡 存活 死亡 存活 │
└────────────────────────────────────────────────┘
        ↓ 整理阶段（移动存活对象）
        ↓
整理后:
┌────────────────────────────────────────────────┐
│  ██ ██ ██ ██ ░░ ░░ ░░ ░░ ░░ ░░ ░░ ░░ ░░ ░░ │
│  存活 存活 存活 存活                           │
│           ↑                                    │
│        边界（存活对象紧凑排列）                 │
└────────────────────────────────────────────────┘
        ↓ 清除阶段
        ↓
清除后:
┌────────────────────────────────────────────────┐
│  ██ ██ ██ ██ ░░ ░░ ░░ ░░ ░░ ░░ ░░ ░░ ░░ ░░ │
│                           ↑                   │
│                    全部空间变为可用            │
└────────────────────────────────────────────────┘
```

**优点**：没有内存碎片、内存利用率高（100%）、适合老年代。

**缺点**：**整理阶段需要移动存活对象**：增加了复杂度、**停顿时间较长**：需要移动大量对象。

标记-整理算法的关键在于"整理"阶段如何高效地移动对象。常见的实现方式有多种，如滑动整理（Lisp 2 算法）、表格整理（Cheney 算法的改进）等。不同的收集器可能有不同的实现。

## 三种算法的对比

|算法 | 内存利用率 | 碎片问题 | 停顿时间 | 适用场景 |
|---|---|---|---|---|
| 标记-清除 | 高 | 有碎片 | 中等 | 老年代（CMS） |
| 复制 | 50% | 无碎片 | 短 | 年轻代 |
| 标记-整理 | 100% | 无碎片 | 较长 | 老年代 |

## 分代收集与算法的结合

现代 JVM 不是单一使用某种算法，而是根据不同代际的特点选择最合适的算法。

年轻代使用复制算法。因为年轻代中对象存活率低，复制成本低，而且可以保证无碎片。

老年代使用标记-清除或标记-整理算法。老年代对象存活率高，复制成本高，所以选择标记-清除或标记-整理。

```
分代收集的算法选择:

┌────────────────────────────────────────────────────────┐
│                         Heap                             │
│                                                        │
│  ┌──────────────────────────────────────────────────┐ │
│  │                  Old Generation                   │ │
│  │                  标记-清除 / 标记-整理              │ │
│  │                  存活率高，适合整理式算法           │ │
│  └──────────────────────────────────────────────────┘ │
│                                                        │
│  ┌──────────────────────────────────────────────────┐ │
│  │                  Young Generation                 │ │
│  │                      复制算法                     │ │
│  │                  存活率低，复制成本低              │ │
│  │  ┌────────────────┐ ┌──────┐ ┌──────┐  │ │
│  │  │     Eden       │ │  S0  │ │  S1  │  │ │
│  │  └────────────────┘ └──────┘ └──────┘  │ │
│  └──────────────────────────────────────────────────┘ │
└────────────────────────────────────────────────────────┘
```

CMS 收集器是一个例外，它在老年代使用标记-清除算法。CMS 的设计目标是追求低停顿时间，所以采用了并发清除策略，避免了 Stop-The-World。但标记-清除产生的碎片问题，最终需要一次 Stop-The-World 的 Full GC 来解决。

## 算法的演进与优化

垃圾收集算法经过几十年的发展，不断演进和优化。

早期 JVM 主要使用标记-清除算法，简单但有碎片问题。Serial 收集器（年轻代复制、老年代标记-整理）是最经典的实现。

后来出现了复制算法的优化版本，将内存按比例划分（如 8:1:1），进一步提高内存利用率。这就是现代年轻代的 Eden + Survivor + Survivor 布局。

现代收集器如 G1，引入了"分区"的概念，将堆划分为多个大小相等的 Region，每个 Region 可以独立作为年轻代或老年代。G1 根据每个 Region 的回收价值（回收能释放的空间和回收需要的时间）优先回收价值最高的 Region。

更先进的收集器如 ZGC 和 Shenandoah，采用着色指针和读屏障技术，实现了亚毫秒级的停顿时间，虽然它们使用的仍然是标记-整理算法的变体。
