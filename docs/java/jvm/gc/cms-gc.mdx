# CMS：并发标记-清除收集器

CMS（Concurrent Mark Sweep）收集器是 JVM 历史上第一个追求低停顿时间的并发收集器。它的出现改变了 Java 应用在低延迟场景下的表现，使得 Java 能够应用于更多对响应时间敏感的场景。虽然 CMS 在 JDK 14 中被正式移除，但它的设计理念和技术创新对后续的 G1、ZGC 等收集器产生了深远影响。理解 CMS 的工作原理，对于理解现代低延迟收集器的发展脉络至关重要。

## CMS 的设计理念

|CMS|

在 CMS 出现之前，垃圾收集器的停顿时间是一个难以逾越的障碍。当堆内存达到数 GB 时，一次 Full GC 可能导致数秒的停顿，这对于 Web 应用、在线交易系统等场景是不可接受的。

CMS 的核心创新是"并发"。传统的垃圾收集器在回收时需要暂停所有应用线程，而 CMS 尝试将大部分工作与应用线程并发执行，从而大幅减少停顿时间。CMS 将 Full GC 的停顿时间从数秒降低到数百毫秒，这是一个质的飞跃。

## CMS 的工作阶段

|CMS 工作阶段|

CMS 的回收过程分为四个阶段：初始标记、并发标记、重新标记、并发清除。这种多阶段的设计是并发收集器的典型模式。

```
CMS 工作阶段示意:

┌─────────────────────────────────────────────────────────────┐
│  阶段1: 初始标记 (STW)                                       │
│  ────────────────────────────────────────────────────────  │
│  从 GC Roots 直接可达的对象                                    │
│  停顿时间: 短                                                │
│                                                              │
│  阶段2: 并发标记 (并发)                                       │
│  ────────────────────────────────────────────────────────  │
│  遍历对象图，标记所有存活对象                                  │
│  应用线程继续运行                                             │
│                                                              │
│  阶段3: 重新标记 (STW)                                        │
│  ────────────────────────────────────────────────────────  │
│  修正并发标记期间产生的变化                                    │
│  停顿时间: 较短                                              │
│                                                              │
│  阶段4: 并发清除 (并发)                                        │
│  ────────────────────────────────────────────────────────  │
│  清扫垃圾对象                                                 │
│  应用线程继续运行                                             │
└─────────────────────────────────────────────────────────────┘
```

第一阶段是初始标记（Initial Mark）。这个阶段需要 Stop-The-World。GC 线程从 GC Roots 出发，标记所有直接可达的对象。这个阶段只标记一层，所以停顿时间很短。

第二阶段是并发标记（Concurrent Mark）。这个阶段与应用线程并发运行。GC 线程从初始标记阶段标记的对象出发，遍历整个对象图，标记所有存活对象。应用线程继续运行，可能不断修改对象引用，导致漏标或错标。

第三阶段是重新标记（Remark）。这个阶段需要 Stop-The-World。GC 线程需要修正并发标记期间由于应用线程运行导致的漏标和错标。这个阶段是并行执行的，停顿时间通常比初始标记长，但远短于完整的标记-清除。

第四阶段是并发清除（Concurrent Sweep）。这个阶段与应用线程并发运行。GC 线程清除所有未被标记的对象，回收它们的内存空间。

## CMS 的核心参数

```bash
# 启用 CMS 收集器
-XX:+UseConcMarkSweepGC

# 常用参数
-XX:CMSInitiatingOccupancyFraction=70   # 老年代使用率达到 70% 时开始回收
-XX:+UseCMSInitiatingOccupancyOnly      # 只使用设定阈值，不自适应调整
-XX:ParallelGCThreads=4                 # 并行 GC 线程数
-XX:ConcGCThreads=2                     # 并发 GC 线程数
```

`-XX:CMSInitiatingOccupancyFraction` 控制 CMS 收集器开始回收的时机。默认值是 -1，表示由 JVM 自动计算。如果设置得太低，会导致频繁的 CMS 回收；如果设置得太高，可能导致并发模式失败。

## CMS 的问题与解决方案

CMS 虽然大幅减少了停顿时间，但也引入了一些新问题。

第一个问题是 CPU 敏感。并发阶段会占用 CPU 资源，导致应用吞吐量下降。CMS 默认启动的并发线程数是 `(ParallelGCThreads + 3) / 4`，在 CPU 资源紧张时可能影响应用性能。

第二个问题是内存碎片。CMS 使用标记-清除算法，会产生内存碎片。当碎片过多时，无法为大对象分配连续空间，会触发一次 "Full GC" 进行碎片整理。这次 Full GC 是 Stop-The-World 的，停顿时间可能很长。

第三个问题是浮动垃圾。并发标记阶段，应用线程可能产生新的垃圾（称为浮动垃圾）。这些垃圾只能在下次 GC 时才能回收。如果浮动垃圾增长太快，可能导致 "Concurrent Mode Failure"。

```
Concurrent Mode Failure 示意:

┌─────────────────────────────────────────────────────────────┐
│  并发标记期间产生的新对象成为"浮动垃圾"                       │
│                                                              │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  已标记的存活对象         浮动垃圾                   │   │
│  │      ■ ■ ■                   ○ ○ ○                 │   │
│  │      ■ ■ ■                   ○ ○                   │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                              │
│  如果老年代空间不足容纳晋升对象 → Concurrent Mode Failure   │
└─────────────────────────────────────────────────────────────┘
```

## 并发模式失败

当 CMS 在并发阶段应用线程创建对象的速度快于回收速度时，会发生并发模式失败（Concurrent Mode Failure）。这时 CMS 会回退到 Serial Old 收集器进行一次单线程的 Full GC，停顿时间可能非常长。

解决这个问题的方法是：

1. 降低 `-XX:CMSInitiatingOccupancyFraction`，让 CMS 更早开始回收。
2. 增加堆大小，给浮动垃圾更多空间。
3. 使用 `-XX:+UseCMSCompactAtFullCollection` 在 Full GC 后进行碎片整理。
4. 使用 `-XX:CMSFullGCsBeforeCompaction=n` 设置多少次 Full GC 后进行一次碎片整理。

```bash
# 优化 CMS 配置示例
java -XX:+UseConcMarkSweepGC \
     -XX:CMSInitiatingOccupancyFraction=68 \
     -XX:+UseCMSInitiatingOccupancyOnly \
     -XX:+UseCMSCompactAtFullCollection \
     -XX:CMSFullGCsBeforeCompaction=5 \
     -Xms512m -Xmx512m \
     MyApplication
```

## CMS 的前世今生

CMS 收集器在 JDK 9 中被标记为废弃（@Deprecated），在 JDK 14 中被移除。它的继任者是 G1 收集器。

G1 继承了 CMS 的并发收集理念，但解决了 CMS 的主要问题：G1 采用分区模型，可以更好地控制停顿时间；G1 使用标记-整理算法，不会产生内存碎片；G1 的回收价值模型可以优先回收垃圾最多的区域。

```bash
# 从 CMS 迁移到 G1
# CMS（已废弃）
java -XX:+UseConcMarkSweepGC -Xmx512m MyApplication

# G1（推荐）
java -XX:+UseG1GC -XX:MaxGCPauseMillis=200 -Xmx512m MyApplication
```
