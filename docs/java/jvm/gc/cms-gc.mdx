# CMS：并发标记-清除收集器

CMS（Concurrent Mark Sweep）收集器是 JVM 历史上第一个追求低停顿时间的并发收集器。它的出现改变了 Java 应用在低延迟场景下的表现，使得 Java 能够应用于更多对响应时间敏感的场景。虽然 CMS 在 JDK 14 中被正式移除，但它的设计理念和技术创新对后续的 G1、ZGC 等收集器产生了深远影响。理解 CMS 的工作原理，对于理解现代低延迟收集器的发展脉络至关重要。

## CMS 的设计理念

<Badge>CMS</Badge>

在 CMS 出现之前，垃圾收集器的停顿时间是一个难以逾越的障碍。当堆内存达到数 GB 时，一次 Full GC 可能导致数秒的停顿，这对于 Web 应用、在线交易系统等场景是不可接受的。

CMS 的核心创新是"并发"。传统的垃圾收集器在回收时需要暂停所有应用线程，而 CMS 尝试将大部分工作与应用线程并发执行，从而大幅减少停顿时间。CMS 将 Full GC 的停顿时间从数秒降低到数百毫秒，这是一个质的飞跃。

## CMS 的工作阶段

<Badge>Concurrent Mark Sweep</Badge>

CMS 的回收过程分为四个阶段：初始标记、并发标记、重新标记、并发清除。这种多阶段的设计是并发收集器的典型模式。

```
CMS 工作阶段示意:
┌─────────────────────────────────────────────────────────────┐
│  阶段1: 初始标记 (STW)                                       │
│  ────────────────────────────────────────────────────────  │
│  从 GC Roots 直接可达的对象                                    │
│  停顿时间: 短                                                │
│                                                              │
│  阶段2: 并发标记 (并发)                                       │
│  ────────────────────────────────────────────────────────  │
│  遍历对象图，标记所有存活对象                                  │
│  应用线程继续运行                                             │
│                                                              │
│  阶段3: 重新标记 (STW)                                        │
│  ────────────────────────────────────────────────────────  │
│  修正并发标记期间产生的变化                                    │
│  停顿时间: 较短                                              │
│                                                              │
│  阶段4: 并发清除 (并发)                                        │
│  ────────────────────────────────────────────────────────  │
│  清扫垃圾对象                                                 │
│  应用线程继续运行                                             │
└─────────────────────────────────────────────────────────────┘
```

第一阶段是初始标记（Initial Mark）。这个阶段需要 Stop-The-World。GC 线程从 GC Roots 出发，标记所有直接可达的对象。这个阶段只标记一层，所以停顿时间很短。

第二阶段是并发标记（Concurrent Mark）。这个阶段与应用线程并发运行。GC 线程从初始标记阶段标记的对象出发，遍历整个对象图，标记所有存活对象。应用线程继续运行，可能不断修改对象引用，导致漏标或错标。

第三阶段是重新标记（Remark）。这个阶段需要 Stop-The-World。GC 线程需要修正并发标记期间由于应用线程运行导致的漏标和错标。这个阶段是并行执行的，停顿时间通常比初始标记长，但远短于完整的标记-清除。

第四阶段是并发清除（Concurrent Sweep）。这个阶段与应用线程并发运行。GC 线程遍历堆内存，清除所有未被标记的对象。应用线程继续运行。

## CMS 的并发模式失败

<Badge>Concurrent Mode Failure</Badge>

CMS 最著名的问题是并发模式失败（Concurrent Mode Failure）。当 CMS 的并发标记阶段还未完成时，老年代空间已经不足，CMS 无法继续并发回收，必须放弃并发模式，触发一次 Stop-The-World 的 Full GC。

并发模式失败通常发生在以下情况：老年代空间增长过快、年轻代对象大量晋升老年代、并发标记阶段应用产生了大量新对象。

```bash
# 并发模式失败的日志表现
2023-01-15T10:23:45.123: [Full GC (Allocation Failure) 
    [CMS: abort preclean due to time  ...
    (concurrent mode failure): 1048576K->786432K(1048576K), 
    2.3456789 secs]
```

当发生并发模式失败时，CMS 会回退到 Serial Old 进行串行的 Full GC。这会导致长时间的停顿，是需要尽量避免的情况。

## CMS 的内存碎片问题

<Badge>Memory Fragmentation</Badge>

CMS 使用标记-清除算法，不进行内存整理。这意味着长期运行后，老年代会产生内存碎片。当需要分配一个大对象时，即使老年代总的空闲空间足够，也可能找不到足够大的连续空间。

CMS 使用`-XX:CMSFullGCsBeforeCompaction`参数来控制何时进行内存压缩。当 Full GC 次数达到这个值后，下一次 Full GC 会进行内存整理。

```bash
# CMS 压缩相关参数
-XX:+UseCMSCompactAtFullCollection  # Full GC 后进行内存压缩（默认开启）
-XX:CMSFullGCsBeforeCompaction=5    # 经过 5 次 Full GC 后进行压缩
```

## CMS 的预清理阶段

<Badge>Preclean</Badge>

CMS 为了减少重新标记阶段的停顿时间，引入了预清理（Preclean）阶段。在预清理阶段，CMS 处理在并发标记阶段由于应用线程运行导致的引用变化。

预清理阶段将部分工作提前进行，减少重新标记时需要处理的变化量。但预清理阶段本身也会消耗一定的时间。

## CMS 与年轻代收集器的配合

<Badge>ParNew</Badge>

CMS 只负责老年代的回收，年轻代由其他收集器负责。CMS 通常与 ParNew 配合使用，ParNew 是唯一能够与 CMS 配合的年轻代收集器。

```
CMS + ParNew 组合示意:
┌─────────────────────────────────────────────────────────────┐
│  年轻代 (ParNew)              老年代 (CMS)                  │
│  ┌──────────────────────┐    ┌──────────────────────┐      │
│  │    Minor GC          │    │    Major GC           │      │
│  │    (STW, 并行)       │    │    (并发)             │      │
│  └──────────────────────┘    └──────────────────────┘      │
│                                                              │
│  Minor GC 触发 → 对象晋升老年代                               │
│  老年代满 → CMS 并发回收                                      │
└─────────────────────────────────────────────────────────────┘
```

## CMS 的常用参数

<Badge>JVM 参数</Badge>

```bash
# 基本参数
-XX:+UseConcMarkSweepGC     # 启用 CMS
-XX:+UseParNewGC            # 年轻代使用 ParNew（通常自动启用）

# 并发线程数
-XX:ConcGCThreads=4         # 并发 GC 线程数（默认约为 ParallelGCThreads/4）

# 触发阈值
-XX:CMSInitiatingOccupancyFraction=70  # 老年代使用率达到 70% 时触发回收
-XX:+UseCMSInitiatingOccupancyOnly     # 只使用上述阈值，不动态调整

# 内存碎片控制
-XX:+UseCMSCompactAtFullCollection      # Full GC 后压缩
-XX:CMSFullGCsBeforeCompaction=5       # Full GC 次数后压缩

# 日志
-XX:+PrintGCDetails
-XX:+PrintGCDateStamps
-Xloggc:gc.log
```

## CMS 的调优要点

<Badge>GC Tuning</Badge>

CMS 的调优主要关注以下几点。

第一，老年代触发阈值。如果 CMS 启动太晚，可能导致空间不足引发并发模式失败。通常建议将 `-XX:CMSInitiatingOccupancyFraction` 设置在 60-70% 之间。

第二，并发线程数。并发线程数默认约为 CPU 核心数的 1/4。如果应用是多线程且 CPU 资源充足，可以适当增加并发线程数。

第三，内存碎片控制。如果碎片化严重，需要调整 `-XX:CMSFullGCsBeforeCompaction` 参数，增加压缩频率。

第四，年轻代大小。年轻代太小会导致频繁的 Minor GC，对象过快晋升老年代。年轻代太大会导致 Minor GC 停顿时间变长。需要根据应用特点进行调整。

## CMS 的历史地位

<Badge>JDK History</Badge>

CMS 在 JVM 历史上有重要地位。它是第一个实现并发回收的收集器，将 Java 应用的 GC 停顿时间从秒级降低到毫秒级。

然而，CMS 也有明显的局限性：CPU 资源消耗大（并发执行需要占用 CPU）、内存碎片问题、并发模式失败的风险。这些问题在后续的 G1 和 ZGC 中得到了改进。

JDK 9 将 CMS 标记为废弃，JDK 14 正式移除 CMS。新的项目应该使用 G1 或 ZGC。

## 面试高频考点

**问题一：CMS 收集器的工作阶段有哪些？** CMS 包括四个阶段：初始标记（STW）、并发标记、重新标记（STW）、并发清除（并发）。

**问题二：什么是并发模式失败？** 当 CMS 在并发标记阶段老年代空间不足时，无法继续并发回收，必须触发一次 Stop-The-World 的 Full GC。这种情况称为并发模式失败。

**问题三：CMS 为什么会有内存碎片问题？** CMS 使用标记-清除算法，不移动存活对象。长期运行后会产生内存碎片，可能导致分配大对象失败。

**问题四：CMS 的优点和缺点是什么？** 优点是并发执行，停顿时间短。缺点是 CPU 资源消耗大、内存碎片问题、可能发生并发模式失败。

**问题五：CMS 适合什么场景？** 适合对停顿时间敏感但 CPU 资源充足的应用，如 Web 应用、在线交易系统等。不适合对 CPU 敏感或碎片化问题严重的应用。

**问题六：CMS 在 JDK 哪个版本被移除？** JDK 14 正式移除了 CMS。JDK 9 将其标记为废弃。

CMS 虽然已经退出历史舞台，但它开创的并发收集理念深深影响了后继者。G1 吸收了 CMS 的并发标记技术，ZGC 和 Shenandoah 则将并发发挥到极致。从这个角度看，CMS 的精神仍在延续。那么，为什么 CMS 最终被 G1 取代？G1 相比 CMS 有哪些改进？
