# 方法区：JDK 7 PermGen vs JDK 8 Metaspace

方法区是 JVM 中一个容易被忽视但至关重要的内存区域。它不像堆那样存储大量的对象实例，也不像栈那样频繁地进行入栈出栈操作，但它是 Java 程序能够运行的幕后英雄。存储在方法区中的类元信息，是 JVM 识别和执行 Java 代码的基础。理解方法区的演进历程，对于理解 JDK 版本差异、排查 PermGen/Metaspace OOM 问题至关重要。

## 方法区的核心职能

方法区（Method Area）在 JVM 规范中定义为存储以下几类信息的区域：类的元数据（类的修饰符、类名、包名、父类名、实现的接口等）、字段信息（字段名、字段类型、字段修饰符等）、方法信息（方法名、返回类型、参数列表、方法修饰符、字节码等）、常量（运行时常量池）、静态变量（虽然 JDK 8 后静态变量的引用存在堆中，但静态变量的元信息仍在方法区）、即时编译器编译后的代码缓存。

方法区是被所有线程共享的区域。一个类被加载后，它的元信息就存储在方法区中，所有线程都可以访问这些信息。这种共享设计节省了内存——每个类只需要一份元数据，而不是每个对象都持有一份。

方法区的生命周期与 JVM 进程相同。当 JVM 启动时，方法区被创建；当 JVM 关闭时，方法区被销毁。方法区的大小可以是固定的，也可以是动态扩展的。

## JDK 7 的永久代：历史的局限

在 JDK 7 及之前的版本中，方法区使用永久代（Permanent Generation，简称 PermGen）实现。永久代是堆内存的一部分，位于堆的年轻代和老年代之外，拥有自己的内存空间。

永久代的设计初衷是存储类的元信息，避免这些信息占用堆内存导致频繁的垃圾回收。在当时看来，这是一个合理的设计——类的元信息不会像普通对象那样频繁创建和销毁，放在非堆区域可以减少垃圾回收的开销。

然而，永久代的设计从一开始就埋下了隐患。永久代的大小是固定的，通过 `-XX:PermSize` 和 `-XX:MaxPermSize` 参数设置。如果应用程序加载的类过多，或者使用了大量动态代理、反射生成的类，永久代可能耗尽空间，导致 `java.lang.OutOfMemoryError: PermGen space`。

这个问题在当时的 Web 应用服务器中尤为普遍。Tomcat、Jetty 等服务器需要加载大量的 JSP 编译后的类和框架类，很容易触发 PermGen OOM。开发者不得不小心翼翼地调整永久代大小，甚至编写脚本来定期重启服务器。

永久代还带来了另一个问题：字符串常量池的存储位置。在 JDK 7 及之前，字符串常量池存储在永久代中。这意味着字符串常量池的大小受永久代限制，而且字符串的垃圾回收与普通对象分离，可能导致永久代空间利用效率不高。

## JDK 8 的元空间：革命性的改变

JDK 8 彻底废弃了永久代，引入了元空间（Metaspace）作为方法区的新实现。这是一次革命性的架构变革，彻底解决了永久代的诸多问题。

元空间不再占用堆内存，而是使用本地内存（Native Memory）。本地内存是 JVM 进程直接向操作系统申请的内存，不受 JVM 堆大小限制。元空间的默认大小只受可用本地内存的限制，只要机器有足够的物理内存，元空间就可以动态扩展。

```
JDK 7 内存结构:
┌─────────────────────────────────────┐
│              Heap                    │
│  ┌─────────┐ ┌───────┐ ┌─────────┐  │
│  │  Eden    │ │ Survivor│ │  Old    │  │
│  └─────────┘ └───────┘ └─────────┘  │
│  ┌────────────────┐                   │
│  │  PermGen       │ ← 堆的一部分      │
│  │  (方法区)      │                   │
│  └────────────────┘                   │
└─────────────────────────────────────┘

JDK 8 内存结构:
┌─────────────────────────────────────┐
│              Heap                    │
│  ┌─────────┐ ┌───────┐ ┌─────────┐  │
│  │  Eden    │ │ Survivor│ │  Old    │  │
│  └─────────┘ └───────┘ └─────────┘  │
└─────────────────────────────────────┘
┌─────────────────────────────────────┐
│         Metaspace (元空间)           │
│  ← 本地内存，不占用堆                │
└─────────────────────────────────────┘
```

元空间相比永久代有以下优势：首先，元空间大小可以动态扩展，不再受固定大小限制，大大降低了 OOM 的风险。其次，元空间使用本地内存，不再消耗堆内存，堆可以完全用于对象存储。再次，元空间的垃圾回收更简单——当类加载器死亡时，其加载的所有类都可以被回收。

## 元空间的内部结构

元空间并非铁板一块，它的内部结构同样精妙。在 HotSpot JVM 中，元空间由以下几部分组成：类元数据虚拟机（Class Metadata VM）负责管理元数据的分配和回收、空间管理使用块（Chunk）分配机制，类似 Linux 内核的内存管理方式，效率更高、元数据分为多种类型，每种类型有不同的生命周期和回收策略。

元空间中存储的类元数据包括：Klass 结构体（指向类的 C++ 内部结构）、方法元数据（方法的字节码、异常表、本地变量表等）、常量池（运行时常量池的数据）、注解（类的注解信息）、方法计数器（用于 JIT 热点代码探测）等。

元空间的内存分配单位是 Chunk，而非字节。JVM 向操作系统申请内存时，一次性申请较大的内存块，然后分割成小块分配给类使用。当类被卸载或类加载器死亡时，其占用的 Chunk 会被标记为空闲，可以被后续的类加载复用。

## 类加载器与元空间回收

元空间垃圾回收的核心是类加载器的生命周期。当一个类加载器被垃圾回收时，它加载的所有类都变成了不可达的，可以被元空间回收。这意味着理解类加载器的生命周期，是理解元空间回收机制的关键。

在典型的 Web 应用中，每个 Web 应用使用独立的类加载器。当应用重新部署时，旧的应用类加载器被销毁，其加载的所有类（包括 JSP 编译后的类）都会被元空间回收。这是热部署的基础。

然而，在某些场景下，类加载器泄漏可能导致元空间 OOM。如果类加载器被错误地持有引用（比如静态集合中存储了类加载器的引用），导致类加载器无法被回收，其加载的所有类也无法被回收。随着时间推移，越来越多的类加载器和类占据元空间，最终导致 OOM。

## 元空间的调优参数

虽然元空间使用本地内存且默认可以动态扩展，但仍然建议设置上限以防止无限制增长导致系统内存耗尽。`-XX:MetaspaceSize` 设置元空间的初始大小，当元空间使用达到这个阈值时，会触发第一次 GC，之后这个阈值会动态调整。`-XX:MaxMetaspaceSize` 设置元空间的最大大小，设置为 0 表示无限制。

`-XX:MinMetaspaceFreeRatio` 和 `-XX:MaxMetaspaceFreeRatio` 分别设置元空间空闲比例的最小值和最大值，用于控制 GC 后元空间的利用率。

```bash
# 设置元空间初始大小为 256MB
java -XX:MetaspaceSize=256m MyApplication

# 设置元空间最大大小为 512MB
java -XX:MaxMetaspaceSize=512m MyApplication

# 监控元空间使用
jstat -gcutil <pid> 1000
```

元空间 OOM 的常见原因包括：大量动态类生成（Java 动态代理、CGLIB、ASM 生成的类）、频繁的类加载和卸载循环（可能导致类加载器泄漏）、使用了会生成大量内部类的框架（如某些 ORM 框架）。

## 字符串常量池的位置变迁

字符串常量池的存储位置是 JDK 版本变迁中另一个值得关注的点。在 JDK 6 及之前，字符串常量池存储在永久代中。这意味着字符串常量池的大小受永久代限制，字符串的回收与普通对象分离。

JDK 7 将字符串常量池移入堆内存中，与普通对象一起管理。这一改变有几个原因：首先，字符串是对象，应该由堆的垃圾回收器统一管理；其次，永久代的 GC 效率较低，不适合字符串这种频繁创建和回收的对象；再次，字符串常量池移入堆后，堆大小调整更加灵活。

JDK 8 之后，由于永久代被废弃，字符串常量池理所当然地存储在堆中。`String.intern()` 方法的行为在 JDK 7/8 中保持一致：首次调用时，如果字符串常量池中没有该字符串，会将其复制到常量池并返回引用；后续调用直接返回常量池中的引用。

## 运行时常量池详解

运行时常量池（Runtime Constant Pool）是方法区的一部分，是 Class 文件中常量池表在运行时的表示。Class 文件中的常量池表存储了编译期可以确定的各类常量和符号引用，这些信息在类加载后进入运行时常量池。

运行时常量池存储的信息包括：字面量（字符串字面量、基本类型常量值等）、符号引用（类的全限定名、字段名和描述符、方法名和描述符等）、方法引用（指向方法和字段的直接引用）等。

与 Class 文件常量池相比，运行时常量池的重要特点是具有动态性。Java 不是只有编译期才能产生常量，运行期间也可以将新的常量放入常量池。使用 `String.intern()` 可以将字符串放入常量池，动态代理也会在运行时常量池中添加类和方法信息。

## 面试高频考点

**问题一：JDK 7 和 JDK 8 的方法区有什么区别？**

JDK 7 使用永久代（PermGen）实现方法区，位于堆内存中，大小固定。JDK 8 废弃永久代，使用元空间（Metaspace）替代，位于本地内存中，大小可以动态扩展。

**问题二：为什么 JDK 8 要移除永久代？**

永久代大小固定，容易发生 OOM；永久代占用堆内存，影响堆的可用空间；永久代的 GC 效率较低。元空间使用本地内存，可以动态扩展，降低了 OOM 风险。

**问题三：Metaspace OOM 的原因是什么？**

大量动态类生成（反射、代理、字节码增强）、频繁的类加载和卸载导致类加载器泄漏、静态集合持有类加载器引用导致无法回收。

**问题四：字符串常量池在哪个版本移入了堆？**

JDK 7 将字符串常量池从永久代移入堆内存，与普通对象一起管理。

**问题五：方法区存储哪些信息？**

类的元数据（修饰符、类名、包名、父类、接口）、字段信息、方法信息、运行时常量池、静态变量引用、即时编译器编译后的代码缓存。

从永久代到元空间，JVM 经历了一次重要的架构演进。这次演进解决了什么问题？又带来了什么新的挑战？理解这段历史，能帮助我们更好地理解 JVM 的设计哲学。
