# 堆内存：年轻代、老年代、TLAB

堆内存是 JVM 中最大的一块内存区域，也是 Java 程序运行中最为核心的内存空间。如果说 Java 程序是一个繁忙的物流中心，堆内存就是那座巨大的仓库——所有通过 `new` 关键字创建的对象都存放在这里，等待被使用或回收。理解堆内存的分区设计、分配策略和垃圾回收机制，是掌握 JVM 内存管理的核心所在。

## 堆内存的分区哲学

<Badge>GC</Badge> <Badge>Generational GC</Badge>

为什么 JVM 要将堆内存划分为年轻代和老年代？这要从对象生命周期的分布规律说起。大量实际项目的内存分析表明：绝大多数对象都是朝生夕灭的，它们在创建后很短的时间内就会变得不可达，随即被垃圾回收。只有极少数对象会存活较长时间，跨越多次垃圾回收仍然存在。

根据这一观察，JVM 设计者提出了"弱分代假说"——大多数对象都在年轻代死亡，只有存活时间长的对象才应该进入老年代。这一假说奠定了现代垃圾收集器分区设计的理论基础。基于这一假说设计的分代收集策略，可以用较低的代价完成垃圾回收：频繁地对年轻代进行回收（Minor GC），偶尔才对老年代进行回收（Major/Full GC）。

## 年轻代的内部结构

<Badge>Eden</Badge> <Badge>Survivor</Badge>

年轻代（Young Generation）是堆内存的第一道防线，用于存储新创建的对象。年轻代进一步细分为三个区域：Eden 区和两个 Survivor 区（通常称为 S0 和 S1 或 From 和 To）。

```
┌──────────────────────────────────────────────────────┐
│                   Young Generation                    │
│  ┌──────────────────┐  ┌───────────┐ ┌───────────┐  │
│  │                    │  │ Survivor  │ │ Survivor  │  │
│  │       Eden         │  │  Space 0  │ │  Space 1  │  │
│  │      Space         │  │   (S0)    │ │   (S1)    │  │
│  │                    │  │           │ │           │  │
│  │                    │  │           │ │           │  │
│  └──────────────────┘  └───────────┘ └───────────┘  │
│                                                      │
│  Eden : Survivor0 : Survivor1 = 8 : 1 : 1 (默认)     │
└──────────────────────────────────────────────────────┘
```

当一个对象首次分配时，它被放在 Eden 区。如果 Eden 区空间不足，会触发 Minor GC。Minor GC 会检查 Eden 区和当前使用的 Survivor 区，将存活的对象复制到另一个 Survivor 区，并增加对象的年龄计数器。经过多次 Minor GC 后，年龄达到阈值（默认是 15，JVM 参数 `-XX:MaxTenuringThreshold` 设置）的对象会被晋升到老年代。

两个 Survivor 区的作用是避免内存碎片化。如果只使用一个 Survivor 区，存活对象在区内的分布会变得不连续，造成碎片。双向复制的设计确保任何一个时刻都有一个完全空闲的 Survivor 区可以使用。Minor GC 发生时，存活对象从 Eden 和当前 Survivor 复制到另一个 Survivor，如果目标 Survivor 空间不足，剩余的对象直接晋升老年代。

## 老年代的存储策略

<Badge>Tenured Gen</Badge>

老年代（Old Generation 或 Tenured Generation）存储生命周期较长的对象。这些对象通常是应用程序的骨干对象，如缓存对象、连接池对象、单例对象等。老年代的回收频率远低于年轻代，但单次回收的成本更高。

对象晋升老年代的条件有多种。第一种是年龄达到阈值：`-XX:MaxTenuringThreshold` 控制对象在年轻代经历多少次 Minor GC 后晋升，默认值为 15。需要注意的是，这个值最大只能设置为 15，因为对象头中的年龄字段只有 4 位。

第二种是 Survivor 空间不足：当 Survivor 区无法容纳存活对象时，这些对象会直接晋升老年代。这通常发生在大量短期对象同时存活的情况下。

第三种是大对象直接分配：`-XX:PretenureSizeThreshold` 参数可以设置大对象的阈值，超过这个大小的对象直接在老年代分配，避免在年轻代频繁复制。

## TLAB：线程本地分配缓冲区

<Badge>TLAB</Badge>

在多线程程序中，对象分配是最频繁的操作之一。如果所有线程都竞争同一个锁来管理堆内存的分配，锁竞争将成为严重的性能瓶颈。TLAB（Thread Local Allocation Buffer）就是为了解决这一问题而设计的。

TLAB 为每个线程预分配一小块 Eden 区作为该线程私有的分配缓冲区。当线程需要分配新对象时，首先在 TLAB 中分配，只有当 TLAB 空间不足时才需要向 Eden 区申请新的 TLAB。TLAB 分配只需要移动指针，不需要任何锁操作，极大地提高了分配效率。

TLAB 的大小可以通过 `-XX:TLABSize` 参数设置，也可以让 JVM 自动调整。`-XX:+UseTLAB` 开启 TLAB（默认开启），`-XX:TLABRefillWasteFraction` 设置 TLAB 允许的浪费比例。当 TLAB 剩余空间不足以分配新对象时，JVM 有两个选择：直接分配在 Eden 区（失去 TLAB 优化），或者重新分配一个 TLAB。

```
线程A的TLAB:  [已用区域][剩余空间]████████████[可分配]
线程B的TLAB:  [已用区域]████████████████████[少量剩余]
线程C的TLAB:  [完全未使用]██████████████████████████
                                        ↓
                    Eden区共享空间（需要加锁）
```

## 自适应调整策略

<Badge>Adaptive Size Policy</Badge>

JVM 的堆内存并非静态不变的，它会根据应用程序的运行状况自动调整各区域的大小。`-XX:+UseAdaptiveSizePolicy` 开启自适应大小策略（默认开启）后，JVM 会根据 GC 的统计数据动态调整各代的大小、Eden 与 Survivor 的比例等参数。

自适应调整虽然方便，但也可能导致问题。在追求停顿时间的场景下，自适应策略可能频繁调整各区域大小，导致 GC 行为不稳定。因此，很多性能调优场景会选择关闭自适应策略，手动指定各区域大小。

`-XX:NewRatio` 设置年轻代与老年代的比例，默认值为 2，表示年轻代占堆的 1/3，老年代占 2/3。`-XX:NewSize` 和 `-XX:MaxNewSize` 分别设置年轻代的最小和最大大小。`-XX:SurvivorRatio` 设置 Eden 区与单个 Survivor 区的比例，默认值为 8，表示 Eden:Survivor = 8:1:1。

## 对象分配的实际流程

<Badge>Object Allocation</Badge>

一个对象从创建到回收，经历了怎样的旅程？让我们追踪对象的生命周期。

当执行 `new Object()` 时，JVM 首先检查这个对象是否能在当前线程的 TLAB 中分配。如果能（TLAB 有足够的剩余空间），对象头指针直接移动到新对象位置，分配完成。如果不能，JVM 尝试在 Eden 区分配新 TLAB。如果 Eden 区有足够空间，分配新 TLAB 并在新 TLAB 中创建对象。如果 Eden 区空间不足，触发 Minor GC。

Minor GC 发生时，垃圾收集器首先标记 Eden 区和当前 Survivor 区中存活的对象，然后将这些对象复制到另一个 Survivor 区。年龄达标的对象被复制到老年代。Minor GC 完成后，Eden 区和原来的 Survivor 区变为空闲状态。

如果 Minor GC 后老年代空间也不足，或者对象是大于 Survivor 空间的大对象，对象直接分配在老年代。如果老年代也无法容纳，触发 Major GC 或 Full GC。如果仍然无法回收足够空间，最终抛出 OutOfMemoryError。

## 堆内存的常用调优参数

<Badge>JVM 参数</Badge>

堆内存调优是 JVM 性能优化的基础。`-Xms` 和 `-Xmx` 设置堆的最小和最大大小，通常建议设置为相同值以避免运行时调整开销。`-Xmn` 设置年轻代大小，是控制 GC 行为的关键参数。

对于追求低停顿时间的应用，可以增大年轻代、减小老年代，因为 Minor GC 的停顿时间通常远短于 Full GC。但这会增加 Minor GC 的频率和对象晋升老年代的年龄。

对于追求高吞吐量的应用，可以减小年轻代、增大老年代，让更多对象在年轻代被回收，减少 GC 总次数。但这会增加每次 Minor GC 的停顿时间。

对于大内存应用（几十 GB 以上的堆），G1 收集器是默认选择。G1 的 `-XX:MaxGCPauseMillis` 设置期望的最大停顿时间，G1 会尽量调整回收策略以满足这个目标。

## 堆内存与垃圾收集器的配合

不同的垃圾收集器对堆内存的分区有不同的实现和使用方式。Serial 收集器使用单线程进行 Minor GC 和 Full GC，简洁但效率低。Parallel 收集器使用多线程并行回收，追求高吞吐量。CMS 和 G1 收集器追求低停顿时间，采用并发收集策略。

G1 收集器完全重新定义了堆内存的分区方式。它不再将堆严格划分为年轻代和老年代，而是将堆划分为大小相等的 Region。每个 Region 可以独立地作为 Eden、Survivor 或老年代使用。G1 的 `-XX:G1NewSizePercent` 和 `-XX:G1MaxNewSizePercent` 控制年轻代占堆的比例。

ZGC 和 Shenandoah 等新一代收集器更进一步，实现了与堆大小无关的停顿时间。它们使用着色指针（ZGC）或Brooks指针（Shenandoah）技术，在并发标记和压缩过程中保持极短的停顿时间。

## 面试高频考点

**问题一：堆内存为什么要分代？** 分代基于"弱分代假说"——大多数对象朝生夕灭，只有少数对象存活时间长。分代收集可以用不同的策略处理不同生命周期的对象：年轻代频繁回收、复制成本低；老年代回收成本高、但频率低。

**问题二：Minor GC 和 Major GC 有什么区别？** Minor GC 回收年轻代，速度快、停顿时间短。Major GC（部分收集器中）或 Full GC 回收老年代，速度慢、停顿时间长。Full GC 通常指整个堆的垃圾收集。

**问题三：对象在 Survivor 区之间的复制过程是怎样的？** Minor GC 时，存活对象从 Eden 和当前 Survivor 复制到另一个 Survivor，年龄达标的对象晋升老年代。如果目标 Survivor 空间不足，超出部分也晋升老年代。

**问题四：TLAB 的作用是什么？** TLAB（线程本地分配缓冲区）为每个线程预分配一小块 Eden 区，线程在自己的 TLAB 中分配对象，无需加锁，提高了多线程环境下的分配效率。

**问题五：大对象如何分配？** 大对象（超过 `-XX:PretenureSizeThreshold` 阈值）直接在老年代分配，避免在年轻代频繁复制。因为大对象复制成本高，且容易占满 Survivor 区导致其他小对象提前晋升。

堆内存是 Java 内存管理的核心，分代设计是 JVM 数十年经验的结晶。理解堆的结构和分配机制，是学习 GC 和进行性能调优的基础。那么，如果所有对象都是朝生夕灭的，JVM 还需要老年代吗？分代设计的边界在哪里？
