# 运行时常量池与字符串常量池

如果说方法区是类的档案馆，运行时常量池就是档案馆中最精彩的那本百科全书。它不仅是类的元信息仓库，更是 Java 语言动态性的体现——常量可以不仅仅在编译期定义，还可以在运行期间动态生成。理解运行时常量池和字符串常量池的工作原理，是理解 Java 内存管理和性能优化的重要一环。

## 常量池的两生两世

理解运行时常量池，首先要理解 Class 文件常量池与运行时常量池的关系。Class 文件常量池是 Class 文件中的一张表，存储编译期能够确定的字面量和符号引用。这张表在编译阶段生成，反映了源代码中的常量定义。运行时常量池是方法区的一部分，是 Class 文件常量池在运行时的表现形式。两者的关系如同建筑图纸与实际建筑——图纸是蓝图，实际建筑是蓝图的实现。

运行时常量池相比 Class 文件常量池，有一个关键特点：动态性。运行时常量池允许在运行期间添加新的常量，而不局限于编译期的定义。这种设计使得 Java 语言能够在运行期间创建和更新常量，提供了更大的灵活性。

## 运行时常量池的存储内容

运行时常量池存储的信息可以分为几大类。第一类是字面量，包括字符串字面量（`"Hello World"`）、数值字面量（`42`、`3.14`）、布尔字面量（`true`、`false`）以及 null 字面量。字面量是源代码中直接写出的常量值。

第二类是符号引用，包括类的全限定名（`java.lang.Object`）、字段的名称和描述符（`name:Ljava/lang/String;`）、方法的名称和描述符（`doSomething:()V`）等。符号引用是一种间接引用，在类加载阶段会被解析为直接引用（内存地址或偏移量）。

第三类是方法信息，包括方法的字节码、异常表、本地变量表等。这些信息在编译后生成，存储在方法区的其他部分，运行时常量池中存储的是指向这些信息的引用。

```java
public class ConstantPoolDemo {
    // 字符串字面量 -> 进入 Class 文件常量池 -> 运行时常量池
    public static final String GREETING = "Hello";

    // 数值字面量
    public static final int MAX_SIZE = 100;

    // 方法引用
    public void doSomething() {
        System.out.println("doing something");
    }
}
```

## 符号引用的解析过程

符号引用需要被解析为直接引用才能使用。解析过程发生在类加载的初始化阶段。当 JVM 遇到符号引用时，会查找运行时常量池中对应的条目，如果尚未解析，则触发该条目的解析过程。

类符号引用解析为类直接引用：查找或加载对应的类/接口。字段引用解析为字段的直接引用：查找字段所属的类，验证字段的访问权限。方法引用解析为方法的直接引用：查找方法所属的类，如果方法是虚方法，需要绑定到具体的实现。

符号引用的解析可能触发其他类的加载和初始化，这是一个连锁过程。理解这一过程对于理解类加载器的委托模型和避免循环依赖至关重要。

## 字符串常量池的演进史

字符串常量池是 Java 内存管理中最独特的区域之一，它的存储位置在 JDK 版本演进中发生了多次变化。理解这些变化的背后原因，有助于深入理解 JVM 的设计哲学。

JDK 6 时期，字符串常量池存储在永久代中。字符串常量作为 `String.intern()` 方法的返回值存储在永久代。这个设计的局限很快暴露：永久代大小固定，容易 OOM；永久代的 GC 效率低，字符串作为高频创建销毁的对象，不适合在永久代管理。

JDK 7 将字符串常量池移入堆内存中。这一改变解决了两个问题：字符串常量池的大小可以随着堆大小调整，不受固定大小限制；字符串可以与普通对象一起被垃圾回收，回收效率更高。

JDK 8 及之后，字符串常量池继续存储在堆内存中。由于永久代被移除，字符串常量池理所当然地与堆共存。这一设计延续至今，成为 Java 内存管理的基础设施。

## String.intern() 的魔法

`String.intern()` 是字符串常量池的核心方法。调用该方法时，如果常量池中已经包含一个等于此 String 对象的字符串（通过 `equals(Object)` 方法判断），则返回常量池中的字符串；否则，将此 String 对象添加到常量池中，并返回此 String 对象的引用。

理解 `intern()` 的行为对于理解字符串内存管理至关重要。在 JDK 6 中，`intern()` 会将字符串复制到常量池。这意味着常量池中的字符串与堆中的字符串是不同的对象。在 JDK 7/8 中，`intern()` 将字符串的引用添加到常量池，常量池中的字符串与堆中的字符串是同一个对象。

```java
public class InternDemo {
    public static void main(String[] args) {
        String s1 = new String("hello");
        String s2 = s1.intern();
        String s3 = "hello";

        System.out.println(s1 == s3);  // false: s1 是堆对象，s3 是常量池
        System.out.println(s2 == s3);  // true: s2 和 s3 都是常量池字符串

        // 字符串拼接
        String s4 = new String("a") + new String("b");
        String s5 = s4.intern();
        String s6 = "ab";

        System.out.println(s4 == s6);  // false (JDK 7/8)
        System.out.println(s5 == s6);  // true: intern 后指向常量池
    }
}
```

## 字符串常量池与性能优化

字符串常量池的设计是为了节省内存和提高性能。当我们使用字符串字面量时，JVM 会先在常量池中查找是否已存在相同的字符串。如果存在，直接返回常量池中的引用；如果不存在，将新字符串加入常量池并返回引用。

这种设计避免了创建重复的字符串对象。在一个典型的 Java 应用中，字符串字面量的使用极其频繁，如果没有常量池，每个字面量都会创建一个新的 String 对象，造成严重的内存浪费。

`intern()` 方法可以用于手动优化内存使用。当存在大量重复字符串时，将这些字符串 `intern()` 可以让所有引用共享同一个对象，显著减少内存占用。但需要注意，`intern()` 并非没有代价：调用 `intern()` 需要搜索或写入常量池，在高并发场景下可能有性能开销。

```java
public class StringPoolOptimization {
    private static final Set<String> INTERNALIZED = new ConcurrentHashMap<>();

    public static String optimize(String input) {
        // 对于可能大量重复的字符串，使用 intern 优化
        return input.intern();
    }

    // 适用场景：
    // 1. 存在大量重复的字符串值
    // 2. 字符串会被长期持有
    // 3. 字符串集合占用的内存成为问题
}
```

## 常量池与垃圾回收的关系

运行时常量池中的常量是否会被垃圾回收？这取决于 JVM 的具体实现和运行参数。在 HotSpot JVM 中，运行时常量池作为方法区（元空间）的一部分，其垃圾回收与元空间的回收策略一致。

当一个类加载器被回收时，该类加载器加载的所有类都会被回收，类的运行时常量池也随之回收。这意味着常量池的垃圾回收与类加载器的生命周期紧密相关。

字符串常量池的情况不同。由于 JDK 7 后字符串常量池在堆中，它遵循堆的垃圾回收策略。当字符串不再被任何引用持有时，它会被垃圾回收，包括常量池中的字符串（前提是它是通过 `intern()` 添加的，且不再被任何引用持有）。

需要注意的是，字面量字符串（如 `"hello"`）直接存储在 Class 文件常量池中，不受运行时 GC 影响。它们随类的加载而加载，随类的卸载而卸载。只有通过 `intern()` 添加到运行时常量池的字符串，才受堆的 GC 管理。

## 常量池大小与 OOM

运行时常量池本身是方法区的一部分，受方法区大小限制（MetaspaceSize/MaxMetaspaceSize）。但运行时常量池的大小问题通常不会直接导致 OOM，因为常量池表的大小在编译时就确定了。

真正可能导致 OOM 的是动态常量生成。`String.intern()` 会向字符串常量池添加字符串。如果程序动态生成了大量不同的字符串并调用 `intern()`，字符串常量池可能占用大量内存。

```java
public class ConstantPoolOOM {
    public static void main(String[] args) {
        // 模拟不断生成的唯一字符串
        List<String> list = new ArrayList<>();
        int i = 0;
        try {
            while (true) {
                // 每个唯一字符串都会被 intern，积累在常量池
                list.add(String.valueOf(i++).intern());
            }
        } catch (OutOfMemoryError e) {
            System.out.println("添加到常量池的数量: " + i);
        }
    }
}
```

在 JDK 6 中，这种代码很容易触发 PermGen OOM。在 JDK 8 中，虽然 Metaspace 可以动态扩展，但通过 `-XX:MaxMetaspaceSize` 设置上限后，仍然可能触发 OOM。

## 面试高频考点

**问题一：Class 文件常量池和运行时常量池有什么区别？**

Class 文件常量池是编译期的概念，存储在 Class 文件中，包含字面量和符号引用。运行时常量池是方法区的一部分，是 Class 文件常量池在运行时的表现形式，支持运行时的动态扩展。

**问题二：字符串常量池在 JDK 6、7、8 中的位置变化是什么？**

JDK 6 在永久代中；JDK 7 移入堆内存中，与普通对象一起管理；JDK 8 延续这一设计，因为永久代被元空间替代。

**问题三：`String.intern()` 在 JDK 6 和 JDK 7/8 中的行为有什么区别？**

JDK 6 中，`intern()` 会将字符串复制到常量池，常量池和堆中是两个不同的对象。JDK 7/8 中，`intern()` 将字符串引用添加到常量池，常量池和堆中是同一个对象。

**问题四：运行时常量池可能抛出 OOM 吗？**

在 JDK 7 及之前，运行时常量池属于永久代，可能抛出 PermGen OOM。JDK 8 之后，运行时常量池属于元空间，在 `-XX:MaxMetaspaceSize` 未设置或设置过大时，不会直接 OOM；但动态生成大量字符串并 `intern()` 可能间接导致问题。

**问题五：什么情况下字符串会被添加到字符串常量池？**

使用字符串字面量时自动添加；调用 `String.intern()` 方法时手动添加。

**问题六：常量池中的符号引用何时被解析？**

在类加载的准备阶段或初始化阶段。某些符号引用可能在第一次使用时才被解析，这称为懒解析。

字符串常量池是 Java 内存管理中最独特的优化之一。从 JDK 6 到 JDK 8，字符串常量池的位置变化反映了 JVM 对内存管理的不断思考。
