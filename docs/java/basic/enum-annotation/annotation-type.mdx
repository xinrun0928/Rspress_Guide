# 注解的分类与保留策略：SOURCE、CLASS、RUNTIME 的深度解析

注解（Annotation）是 Java 5 引入的特性，它提供了一种将元数据直接嵌入到 Java 代码中的方式。表面上，注解只是一个以 `@` 开头的标记，如 `@Override`、`@Autowired`、`@Entity`。但实际上，注解的内部机制远比这复杂。理解注解的保留策略（Retention Policy），是理解注解如何工作的关键。

当我们定义一个注解时，我们需要指定它的保留策略。这个策略决定了注解在什么阶段可见——是在源代码中、在编译后的 class 文件中，还是在运行时。下面让我们深入探讨这三种保留策略。

## 注解的本质

在深入讨论保留策略之前，我们需要理解注解的本质。从字节码层面看，注解实际上是一种特殊的接口。

```java
public @interface Deprecated {
    String value();
}
```

编译后，这个"注解"会变成：

```java
public interface Deprecated extends Annotation {
    String value();
}
```

是的，你没有看错——`@interface` 在编译后变成了 `extends Annotation` 的接口。这揭示了注解与接口的本质联系：注解是一种特殊的接口，用于携带类型化的元数据。

## RetentionPolicy.SOURCE：源代码级别的注解

SOURCE 策略的注解只在源代码中存在，编译后完全消失，不会进入 class 文件。这种注解主要用于以下场景。

### 编译时检查

最常见的 SOURCE 级别注解是 `@Override`。这个注解告诉编译器"接下来的方法应该覆盖父类的方法"。如果方法签名不匹配任何父类方法，编译器会报错：

```java
public class Parent {
    public void doSomething() {}
}

public class Child extends Parent {
    @Override  // 如果方法名拼写错误，编译器会报错
    public void doSomthing() {}  // 拼写错误：少了 'e'
}
```

`@Override` 的保留策略是 SOURCE，这意味着在 class 文件和运行时都没有这个注解的信息。编译器在解析源代码时识别这个标记，然后丢弃它。

### 注解处理

另一个 SOURCE 注解的重要用途是与其他注解处理器配合。例如，Java 的 `@Generated` 注解（JDK 9+）用于标记由代码生成器生成的代码：

```java
@Generated(value = "com.example.CodeGenerator", 
           date = "2024-01-15T10:30:00")
public class GeneratedClass {
    // ...
}
```

工具可以读取这个注解来决定是否重新生成代码，或者在构建报告中记录代码的来源。

### Lombok 的早期版本

早期的 Lombok（< 1.12）在 JDK 8 之前使用 SOURCE 级别的注解处理。Lombok 的注解（如 `@Data`）在编译时被注解处理器读取，然后生成相应的辅助方法。

## RetentionPolicy.CLASS：类文件级别的注解

CLASS 策略的注解会被编译到 class 文件中，但 JVM 不会在运行时加载这些信息。这是 JVM 的"默认"保留策略（当注解没有显式指定 @Retention 时）。

```java
@Retention(RetentionPolicy.CLASS)
public @interface MyClassAnnotation {
    String value();
}
```

这种策略的注解主要用于需要在编译后、类加载前读取注解的工具。例如，一些字节码增强库会在类加载时读取 CLASS 级别的注解，而不是依赖反射机制（这要求 RUNTIME 级别）。

### 字节码增强工具

ASM 和 Javassist 等字节码操作库可以在类加载时修改 class 文件。这些库可以读取 CLASS 级别的注解，用于决定如何修改字节码。例如，一个性能监控库可能使用 CLASS 级别的注解来标记需要增强的方法：

```java
@Retention(RetentionPolicy.CLASS)
public @interface Profiled {}

public class UserService {
    @Profiled
    public void processOrder() {
        // ...
    }
}
```

字节码增强工具在加载类时读取 @Profiled 注解，然后为这些方法添加性能监控逻辑。

### 为什么要使用 CLASS 而非 RUNTIME？

使用 CLASS 级别而非 RUNTIME 级别有几个优势。首先，性能更好——不需要在运行时使用反射读取注解。其次，兼容性更好——即使在启用模块化的 JDK 9+ 环境中，CLASS 级别的注解也能正常工作。第三，信息更少泄露——注解信息不会出现在运行时的堆栈跟踪或异常消息中。

## RetentionPolicy.RUNTIME：运行时可访问的注解

RUNTIME 策略的注解会被保留到运行时，可以通过反射 API 在程序运行时读取。这是 Spring、Hibernate、JUnit 等框架大量使用的机制。

```java
@Retention(RetentionPolicy.RUNTIME)
public @interface Column {
    String name() default "";
    boolean nullable() default true;
}
```

通过反射读取 RUNTIME 注解的代码如下：

```java
public class AnnotationReader {
    public static void main(String[] args) throws Exception {
        Class<?> clazz = User.class;
        
        // 检查类是否有特定注解
        if (clazz.isAnnotationPresent(Entity.class)) {
            Entity entity = clazz.getAnnotation(Entity.class);
            System.out.println("实体名称: " + entity.name());
        }
        
        // 获取所有字段注解
        for (Field field : clazz.getDeclaredFields()) {
            if (field.isAnnotationPresent(Column.class)) {
                Column column = field.getAnnotation(Column.class);
                System.out.println("字段: " + field.getName() + 
                                   ", 列名: " + column.name());
            }
        }
        
        // 获取所有方法注解
        for (Method method : clazz.getDeclaredMethods()) {
            if (method.isAnnotationPresent(Init.class)) {
                System.out.println("初始化方法: " + method.getName());
            }
        }
    }
}
```

### Target：注解可以应用的目标

除了 @Retention，每个注解都应该有 @Target 元注解，指定注解可以应用于哪些程序元素：

```java
@Target(ElementType.TYPE)           // 类、接口、枚举
@Target(ElementType.METHOD)          // 方法
@Target(ElementType.FIELD)           // 字段
@Target(ElementType.PARAMETER)      // 方法参数
@Target(ElementType.CONSTRUCTOR)    // 构造函数
@Target(ElementType.LOCAL_VARIABLE) // 局部变量
@Target(ElementType.ANNOTATION_TYPE) // 注解类型
@Target(ElementType.PACKAGE)         // 包声明
```

可以组合多个目标类型：

```java
@Target({ElementType.TYPE, ElementType.METHOD})
public @interface MyAnnotation {
    String value();
}
```

## 内置注解一览

Java 提供了一组内置注解，它们的保留策略各有不同：

**@Override** (SOURCE)：标记方法覆盖。编译器会验证这个方法确实覆盖了父类的方法。

**@Deprecated** (RUNTIME)：标记已废弃的元素。编译器会生成警告，反射可以检测到这个标记。

**@SuppressWarnings** (SOURCE)：告诉编译器抑制特定类型的警告。

**@SafeVarargs** (RUNTIME)：标记不会对可变参数执行危险操作的方法或构造函数。

**@FunctionalInterface** (RUNTIME)：标记函数式接口（只有一个抽象方法的接口）。

**@Repeatable** (RUNTIME)（JDK 8+）：标记可以在同一位置多次应用的注解。

## @Repeatable：可重复注解

JDK 8 引入了 @Repeatable，允许同一个注解在同一个位置多次应用：

```java
@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.TYPE)
@Repeatable(AuthorGroups.class)
public @interface Author {
    String name();
}

@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.TYPE)
public @interface AuthorGroups {
    Author[] value();
}

@Author(name = "张三")
@Author(name = "李四")
public class MyClass {
    // 这个类由张三和李四编写
}
```

@Repeatable 的容器注解必须有一个名为 value 的数组属性，类型是被容器的可重复注解。

## 注解与继承

注解默认不会继承到子类。考虑这种情况：

```java
@Deprecated
public class Parent {}

public class Child extends Parent {}
```

Child 类不会自动拥有 @Deprecated 注解。如果你想让注解被子类继承，需要在注解定义中使用 @Inherited：

```java
@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.TYPE)
@Inherited
public @interface MyInheritedAnnotation {}
```

## 实际应用场景

注解在实际项目中有无数应用场景。

**配置管理**：Spring 的 @Configuration、@Bean、@ComponentScan 等用于定义配置和组件扫描规则。

**依赖注入**：@Autowired、@Inject 用于声明依赖注入点。

**请求处理**：@GetMapping、@PostMapping、@RequestParam 用于 REST API 路由。

**数据验证**：@NotNull、@Size、@Email 用于 Bean Validation。

**ORM 映射**：@Entity、@Table、@Column 用于 JPA/Hibernate 实体映射。

理解注解的保留策略和作用机制，是理解这些框架工作原理的基础。

## 总结

注解是 Java 强大的元编程特性的核心。理解 SOURCE、CLASS、RUNTIME 三种保留策略的区别，对于正确使用注解至关重要。SOURCE 级别用于编译时检查，CLASS 级别用于字节码增强，RUNTIME 级别用于反射处理。选择正确的保留策略，可以让你的注解更加精确和高效。

你是否想过，为什么 Java 不默认使用 RUNTIME 保留策略，而是使用 CLASS？这种设计选择反映了什么权衡？当你设计自己的注解时，应该如何选择保留策略？这些问题将帮助你在实践中更好地使用注解。
