# 抽象类与接口：代码的契约设计

在建筑设计领域，建筑师会给施工队一份图纸，上面详细规定了建筑的结构、管道走向、电线布置等。施工队不需要知道这份图纸是如何绘制的，他们只需要按照图纸施工。而监理工程师则根据另一份规范文档来检查建筑是否符合标准——这份规范可能来自政府或行业协会。

在 Java 的世界里，抽象类和接口就扮演着类似图纸和规范的角色。它们定义了"应该做什么"，而把"如何做"留给具体的实现类。但它们之间有什么区别？什么时候该用抽象类，什么时候该用接口？这个问题困惑了无数 Java 开发者。

## 抽象类：不完全的蓝图

### 什么是抽象类？

当你在设计一个类层次结构时，有时候你会发现某些类是"概念上存在，但不应该直接实例化"的。比如，"动物"是一个概念，你可以实例化"狗"或"猫"，但"动物"本身没有具体的形态，无法实例化。

抽象类就是为这种场景设计的。使用 `abstract` 关键字修饰的类就是抽象类：

```java
public abstract class Animal {
    protected String name;
    protected int age;

    public Animal(String name, int age) {
        this.name = name;
        this.age = age;
    }

    // 抽象方法：没有实现，只有签名
    public abstract void makeSound();

    // 具体方法：可以有自己的实现
    public void sleep() {
        System.out.println(name + "正在睡觉");
    }

    public void eat() {
        System.out.println(name + "正在吃东西");
    }
}
```

抽象类可以有抽象方法（没有方法体的方法），也可以有具体方法（有方法体的方法）。抽象类本身不能被实例化，但可以引用子类对象：

```java
Animal animal = new Dog("旺财", 3);  // 合法
Animal animal2 = new Animal();       // 编译错误！
```

### 抽象方法：强制子类实现

抽象方法是一种特殊的声明，它只有方法签名，没有方法体。子类必须实现所有抽象方法，除非子类自己也是抽象的：

```java
public abstract class Shape {
    public abstract double getArea();
    public abstract double getPerimeter();
}

public class Circle extends Shape {
    private double radius;

    public Circle(double radius) {
        this.radius = radius;
    }

    @Override
    public double getArea() {
        return Math.PI * radius * radius;
    }

    @Override
    public double getPerimeter() {
        return 2 * Math.PI * radius;
    }
}
```

<Badge text="重要提醒" type="warning" /> 如果子类没有实现父类的所有抽象方法，那么子类也必须声明为抽象类。这是 Java 强制执行的规则，确保抽象类定义的契约最终会被实现。

### 抽象类的构造函数

抽象类可以有构造函数，而且通常都会定义构造函数供子类调用。这允许子类在创建对象时初始化从父类继承的状态：

```java
public abstract class Person {
    protected String name;
    protected int age;

    public Person(String name, int age) {
        this.name = name;
        this.age = age;
    }

    public abstract String getRole();
}

public class Student extends Person {
    private String studentId;

    public Student(String name, int age, String studentId) {
        super(name, age);  // 调用父类构造函数
        this.studentId = studentId;
    }

    @Override
    public String getRole() {
        return "学生";
    }
}
```

## 接口：纯粹的行为契约

### 接口的定义

如果说抽象类是"不完整的类"，那么接口就是"纯粹的行为规范"。接口只定义方法签名，不提供任何实现（在 JDK 8 之前），而且不能有任何实例字段：

```java
public interface Flyable {
    void fly();
}

public interface Swimmable {
    void swim();
}

public interface Runnable {
    void run();
}
```

接口中的方法默认是 `public abstract`（JDK 7 及之前），所以你可以简写为：

```java
public interface Flyable {
    void fly();  // 等价于 public abstract void fly();
}
```

### 接口的实现

类使用 `implements` 关键字来实现接口：

```java
public class Bird implements Flyable {
    @Override
    public void fly() {
        System.out.println("鸟儿展翅飞翔");
    }
}

public class Fish implements Swimmable {
    @Override
    public void swim() {
        System.out.println("鱼儿水中游");
    }
}
```

一个类可以实现多个接口，这弥补了 Java 单继承的限制：

```java
public class Duck implements Flyable, Swimmable, Runnable {
    @Override
    public void fly() {
        System.out.println("鸭子飞起来了");
    }

    @Override
    public void swim() {
        System.out.println("鸭子浮在水面上");
    }

    @Override
    public void run() {
        System.out.println("鸭子在地上跑");
    }
}
```

### JDK 8 的重大改进：默认方法

<Badge text="JDK 8+" type="tip" /> JDK 8 为接口引入了默认方法，使用 `default` 关键字修饰。这允许接口提供方法的有默认实现，实现类可以选择覆盖或者使用默认实现。

```java
public interface Readable {
    void read();

    // 默认实现
    default void close() {
        System.out.println("关闭阅读器");
    }
}
```

为什么要引入默认方法？主要是为了向后兼容。当给接口添加新方法时，如果没有默认方法，所有实现类都会编译失败。有了默认方法，旧代码可以正常运行，新代码可以覆盖默认实现。

最典型的例子是 `java.util.List` 接口新增的 `sort` 方法：

```java
default void sort(Comparator<? super E> c) {
    Collections.sort(this, c);
}
```

如果不使用默认方法，每个 `List` 的实现类都需要单独实现这个方法，这将是一场灾难。

### JDK 9 的改进：私有方法

<Badge text="JDK 9+" type="tip" /> JDK 9 允许接口定义私有方法。私有方法可以复用默认方法中的逻辑，但不能被外部访问：

```java
public interface Service {
    default void doSomething() {
        validate();
        // ... 业务逻辑
    }

    default void doAnotherThing() {
        validate();
        // ... 其他业务逻辑
    }

    private void validate() {
        // 共同的验证逻辑
        System.out.println("验证中...");
    }
}
```

### JDK 8+ 的静态方法

<Badge text="JDK 8+" type="tip" /> JDK 8 还允许接口定义静态方法。接口的静态方法属于接口本身，不属于实现类：

```java
public interface StringUtil {
    static boolean isBlank(String str) {
        return str == null || str.trim().isEmpty();
    }

    static String defaultIfBlank(String str, String defaultValue) {
        return isBlank(str) ? defaultValue : str;
    }
}

// 调用静态方法
String result = StringUtil.defaultIfBlank(input, "N/A");
```

这为接口提供了一种组织相关工具方法的方式，比如 `Comparator`、`Function` 等接口都有丰富的静态工具方法。

## 抽象类与接口的抉择

### 核心区别

| 特性 | 抽象类 | 接口 |
|------|--------|------|
| 继承/实现 | 单继承 | 多实现 |
| 字段 | 可以有任何类型的字段 | 只能有常量（static final） |
| 方法 | 抽象方法 + 具体方法 | JDK 7 前只能是抽象方法；JDK 8+ 支持默认方法和静态方法；JDK 9+ 支持私有方法 |
| 构造函数 | 可以有构造函数 | 不能有构造函数 |
| 访问修饰符 | 任何修饰符 | 方法默认 public；JDK 9+ 接口方法可以是 private |

### 选择的原则

**使用抽象类的场景**：

当你需要表示"是...的特例"关系时，抽象类是更好的选择。比如"狗是动物的一种特例"，这用抽象类表达就很自然。

当你需要子类共享状态或行为时，抽象类中的具体方法可以实现代码复用。

当你需要控制版本——抽象类可以添加非抽象方法而不破坏现有代码，而旧接口添加抽象方法会让所有实现类编译失败。

```java
public abstract class Vehicle {
    protected String brand;

    public abstract void start();
    public abstract void stop();

    // 所有车辆都有的通用行为
    public void addFuel(double amount) {
        System.out.println("添加 " + amount + " 升燃油");
    }
}
```

**使用接口的场景**：

当你需要定义"能做什么"的能力时，接口更合适。比如"能飞"、"能打印"、"能序列化"。

当你需要多继承时，接口是唯一的选择。

当你正在设计一个公用的 API，而且未来可能有很多实现者时，接口提供了最大的灵活性。

```java
public interface Serializable {
    void serialize(OutputStream out) throws IOException;
}

public interface Comparable<T> {
    int compareTo(T o);
}

public interface Cloneable {
    // 空接口，只作为标记
}
```

### 实际应用的例子

考虑一个图形编辑器的设计：

```java
// 定义图形的基本属性
public abstract class Shape {
    protected double x;
    protected double y;

    public Shape(double x, double y) {
        this.x = x;
        this.y = y;
    }

    public abstract double getArea();
    public abstract double getPerimeter();

    public void move(double dx, double dy) {
        x += dx;
        y += dy;
    }
}

// 定义可绘制的契约
public interface Drawable {
    void draw(GraphicsContext ctx);
}

// 定义可拖拽的契约
public interface Draggable {
    void dragTo(double x, double y);
    void dragBy(double dx, double dy);
}

// 圆形：是一种图形，并且可以绘制和拖拽
public class Circle extends Shape implements Drawable, Draggable {
    private double radius;

    public Circle(double x, double y, double radius) {
        super(x, y);
        this.radius = radius;
    }

    @Override
    public double getArea() {
        return Math.PI * radius * radius;
    }

    @Override
    public double getPerimeter() {
        return 2 * Math.PI * radius;
    }

    @Override
    public void draw(GraphicsContext ctx) {
        ctx.drawCircle(x, y, radius);
    }

    @Override
    public void dragTo(double newX, double newY) {
        x = newX;
        y = newY;
    }

    @Override
    public void dragBy(double dx, double dy) {
        x += dx;
        y += dy;
    }
}
```

在这个设计中，`Shape` 是抽象类，表达了"图形是一种几何概念"的本质关系；`Drawable` 和 `Draggable` 是接口，表达了图形"能做什么"的能力。这种组合设计让代码既有清晰的继承结构，又有灵活的组合能力。

## 函数式接口：接口的新角色

<Badge text="JDK 8+" type="tip" /> JDK 8 引入了函数式接口的概念：只包含一个抽象方法的接口。可以用 `@FunctionalInterface` 注解标记，编译器会验证这个接口是否符合要求。

函数式接口是 Lambda 表达式的基础。Java 标准库中提供了大量有用的函数式接口：

```java
@FunctionalInterface
public interface Predicate<T> {
    boolean test(T t);
}

@FunctionalInterface
public interface Supplier<T> {
    T get();
}

@FunctionalInterface
public interface Function<T, R> {
    R apply(T t);
}

@FunctionalInterface
public interface Consumer<T> {
    void accept(T t);
}
```

## 常见设计模式与接口

接口在实际开发中有很多经典应用：

**模板方法模式**：抽象类定义算法的骨架，具体步骤由子类实现。

**策略模式**：接口定义算法的接口，不同实现代表不同策略。

**适配器模式**：接口作为目标接口，适配器类实现接口并持有被适配对象的引用。

**工厂模式**：工厂接口定义创建产品的方法，具体工厂实现创建具体产品。

理解了抽象类和接口的区别与联系，你就掌握了 Java 面向对象设计的精髓。它们不是对立的工具，而是互补的设计手段。下一章我们将探讨内部类与 Lambda 表达式，看看 Java 如何让代码更加简洁和表达力强。
