# 面向对象：封装、继承、多态

想象一个精致的瑞士手表。数不清的精密齿轮相互咬合，协同工作，但佩戴者只需要转动表冠，就能精准地设置时间。手表的内部机械对用户是隐藏的，用户不需要了解齿轮的咬合方式，也不需要知道发条是如何储存能量的。这种"隐藏复杂性，只暴露接口"的哲学，正是面向对象编程中封装的核心思想。

Java 作为一门纯正的面向对象语言，将这种思想发挥到了极致。在这一章节中，我们将深入探讨面向对象编程的三大支柱：封装、继承和多态。这三个概念不仅仅是语法规则，更是解决复杂软件问题的思维框架。

## 封装：信息的艺术

### 为什么需要封装？

在早期的面向过程编程中，程序就像一碗面条——数据和操作数据的函数混在一起，没有任何组织。当程序规模变大时，这种混乱变得无法维护。想象一下，如果全局变量可以被任何函数随意修改，你怎么知道程序为什么会突然出问题？

封装就像是给这碗面条加上了一个包装盒。你把相关的功能和数据打包在一起，隐藏内部实现，只通过精心设计的接口与外界交互。这带来的好处是：

**数据保护**：通过访问修饰符，你可以控制哪些数据可以被外部访问，哪些只能被内部使用。就像银行的保险柜，用户只能通过柜员操作，而不能直接拿取里面的现金。

**实现灵活性**：封装允许你改变类的内部实现，而不影响使用它的代码。比如，你一开始用数组存储数据，后来发现数组效率不高，改成了链表——只要接口不变，所有使用这个类的代码都不需要修改。

**降低复杂度**：使用类的人不需要理解复杂的内部逻辑，只需要知道怎么调用接口。这大大降低了学习和使用成本。

### 访问修饰符：代码的门卫

Java 提供了四个级别的访问控制，从最严格到最宽松：

```java
public class AccessDemo {
    public String publicField = "任何地方都可以访问";
    protected String protectedField = "同类、同包及子类可以访问";
    String defaultField = "同包可以访问";  // 没有修饰符，默认访问级别
    private String privateField = "只有本类可以访问";
}
```

**private** 是最严格的访问级别。被 `private` 修饰的成员只能在当前类内部访问。这通常用于封装类的内部实现细节，比如存储数据的变量。

**default**（无修饰符）允许同包内的其他类访问。这在设计一个包内的协作类时很有用，但对包外的代码是隐藏的。

**protected** 主要用于继承场景。它允许子类访问父类的成员，即使子类在不同的包中。

**public** 是最开放的访问级别。被 `public` 修饰的成员可以从任何地方访问。

<Badge text="最佳实践" type="tip" /> 作为一个经验法则，成员变量应该尽可能使用 `private`，然后通过 `public` 方法提供受控的访问。这就是 Java Beans 规范的核心：通过 getter 和 setter 封装数据。

### getter 和 setter：优雅的接口

```java
public class User {
    private String name;
    private int age;

    public String getName() {
        return name;
    }

    public void setName(String name) {
        this.name = name;
    }

    public int getAge() {
        return age;
    }

    public void setAge(int age) {
        if (age < 0 || age > 150) {
            throw new IllegalArgumentException("年龄必须在 0-150 之间");
        }
        this.age = age;
    }
}
```

注意 `setAge` 方法中的验证逻辑。通过封装，我们可以在 setter 中添加验证，确保对象始终处于有效状态。如果直接暴露 `age` 字段，任何人都可以设置一个负数或超出范围的年龄，导致数据不一致。

## 继承：基因的传递

### 继承的本质

"龙生龙，凤生凤，老鼠的儿子会打洞。"这句俗语完美地诠释了继承的本质：子类自动拥有父类的特征，同时还可以添加自己特有的属性和行为。

在 Java 中，使用 `extends` 关键字表示继承关系：

```java
public class Animal {
    protected String name;
    protected int age;

    public void eat() {
        System.out.println(name + "正在吃东西");
    }

    public void sleep() {
        System.out.println(name + "正在睡觉");
    }
}

public class Dog extends Animal {
    private String breed;  // 品种，狗特有的属性

    public void bark() {
        System.out.println(name + "汪汪叫");
    }

    @Override
    public void eat() {
        System.out.println(name + "正在啃骨头");
    }
}
```

在这个例子中，`Dog` 自动拥有了 `Animal` 的 `name`、`age` 属性以及 `eat()` 和 `sleep()` 方法，同时添加了自己的 `breed` 属性和 `bark()` 方法。这就是继承的力量：代码复用。

### 方法重写：超越父辈

`Dog` 重写了 `eat()` 方法。当调用 `dog.eat()` 时，执行的是 `Dog` 版本的 `eat()`，而不是 `Animal` 的版本。这就是多态的一种体现。

<Badge text="重要提醒" type="warning" /> 重写方法时，必须保持方法签名完全一致（参数类型、个数、顺序），否则就变成了重载而不是重写。同时，`private` 方法不能被重写，`static` 方法也不能被重写（只能被隐藏）。

使用 `@Override` 注解是一个好习惯，它告诉编译器这个方法应该覆盖父类的方法。如果由于疏忽导致方法签名不匹配，编译器会报错：

```java
@Override
public void eat() {  // 如果拼写错误，编译器会报错
    // ...
}
```

### super 关键字：与父辈对话

有时候，子类需要调用父类的方法或访问父类的字段。`super` 关键字就是为此而生的：

```java
public class Cat extends Animal {
    private String color;

    public Cat(String name, int age, String color) {
        super(name, age);  // 调用父类的构造函数
        this.color = color;
    }

    @Override
    public void eat() {
        super.eat();  // 先调用父类的 eat 方法
        System.out.println(getName() + "然后舔了舔嘴唇");
    }

    public void purr() {
        System.out.println(getName() + "发出呼噜声");
    }
}
```

### 构造函数的调用顺序

当创建一个子类对象时，构造函数的调用顺序是一个经常被误解的点。正确的顺序是：

1. 递归调用父类构造函数，直到 `Object`
2. 按声明顺序执行实例变量初始化
3. 执行当前类的构造函数体

```java
public class Parent {
    public Parent() {
        System.out.println("Parent 构造函数");
    }
}

public class Child extends Parent {
    public Child() {
        System.out.println("Child 构造函数");
    }
}

public class GrandChild extends Child {
    public GrandChild() {
        System.out.println("GrandChild 构造函数");
    }
}

new GrandChild();
// 输出：
// Parent 构造函数
// Child 构造函数
// GrandChild 构造函数
```

<Badge text="重要提醒" type="warning" /> 如果父类没有无参构造函数，子类必须在构造函数中显式调用父类的其他构造函数。调用必须是在子类构造函数的第一行。

### Java 的单继承限制

与 C++ 支持多继承不同，Java 只允许单继承——一个类只能有一个直接父类。这个设计决策是有意为之的，目的是避免多继承带来的复杂性（如菱形继承问题）。但 Java 通过接口提供了一种间接的多继承能力，我们会在后续章节详细讨论。

## 多态：同一个面孔，不同的表现

### 多态是什么？

"一千个读者眼中有一千个哈姆雷特。" 多态就是这样一种能力：同一个方法调用，在不同对象上产生不同的行为。

多态主要有两种形式：编译时多态（通过方法重载）和运行时多态（通过方法重写和向上转型）。我们主要讨论运行时多态。

### 向上转型：子到父的安全转换

把子类的引用赋值给父类类型的变量，称为向上转型。这是安全的，因为子类必然是父类的一种：

```java
Animal dog = new Dog();  // 向上转型
dog.eat();  // 调用的是 Dog 版本的 eat()
```

这个 `dog` 变量的类型是 `Animal`，但实际指向的是一个 `Dog` 对象。当调用 `eat()` 方法时，Java 会根据实际对象的类型来决定调用哪个版本。这就是运行时多态。

### 向下转型：谨慎的操作

向下转型是把父类引用转回子类类型，这需要显式强制转换，而且可能失败：

```java
Animal animal = new Dog();  // 向上转型
Dog dog = (Dog) animal;     // 向下转型，安全，因为 animal 确实是 Dog

Animal cat = new Cat();
Dog dog2 = (Dog) cat;  // 运行时抛出 ClassCastException！
```

<Badge text="重要提醒" type="warning" /> 向下转型是不安全的。为了避免 `ClassCastException`，在转型之前应该使用 `instanceof` 检查：

```java
if (animal instanceof Dog) {
    Dog dog = (Dog) animal;
    dog.bark();
}
```

<Badge text="JDK 14+" type="tip" /> JDK 14 引入了模式匹配的 `instanceof`，可以简化这种写法：

```java
if (animal instanceof Dog dog) {
    dog.bark();  // dog 在这个块内可用
}
```

### 多态的实际应用

多态最常见的应用场景是：定义一个方法接受基类类型，实际传入的是不同的子类类型。

```java
public class Shape {
    public void draw() {
        System.out.println("绘制图形");
    }
}

public class Circle extends Shape {
    @Override
    public void draw() {
        System.out.println("绘制圆形 ○");
    }
}

public class Rectangle extends Shape {
    @Override
    public void draw() {
        System.out.println("绘制矩形 □");
    }
}

public class Triangle extends Shape {
    @Override
    public void draw() {
        System.out.println("绘制三角形 △");
    }
}

// 多态的威力：同一个方法，处理不同类型的对象
public void render(Shape shape) {
    shape.draw();  // 自动调用对应类型的 draw 方法
}

render(new Circle());      // 输出：绘制圆形 ○
render(new Rectangle());   // 输出：绘制矩形 □
render(new Triangle());     // 输出：绘制三角形 △
```

这种设计的好处是：添加新的图形类型时，不需要修改 `render` 方法。只需要创建新的类继承 `Shape`，重写 `draw` 方法即可。这就是著名的"开闭原则"：对扩展开放，对修改封闭。

### 虚方法表：多态的底层机制

你可能会好奇，Java 是如何在运行时确定应该调用哪个方法的。这涉及到 Java 的虚方法表（vtable）机制。

每个类都有一个虚方法表，其中列出了该类所有可重写的方法以及它们的实现地址。当创建对象时，这个表会被复制到对象头中。方法调用时，JVM 查表确定实际要执行的方法。

这个过程发生在运行时，所以称为运行时绑定或动态分派。相比之下，私有方法、静态方法、构造方法和 `final` 方法是在编译时绑定的，称为静态绑定。

## 三大原则的协同

封装、继承和多态并不是孤立的概念，它们协同工作，构成了 Java 面向对象编程的完整体系。

继承让代码得以复用，建立类与类之间的"是...的特例"关系。封装保护了继承来的内部实现，通过受控的接口与外界交互。多态则让我们可以用统一的方式处理不同子类的对象，编写出灵活可扩展的代码。

一个典型的场景：设计一个支付系统：

```java
public abstract class Payment {
    protected String orderId;

    public abstract void pay(double amount);
    public abstract void refund(double amount);

    public void validateOrder() {
        // 通用的订单验证逻辑
    }
}

public class Alipay extends Payment {
    @Override
    public void pay(double amount) {
        // 支付宝支付逻辑
    }

    @Override
    public void refund(double amount) {
        // 支付宝退款逻辑
    }
}

public class WechatPay extends Payment {
    @Override
    public void pay(double amount) {
        // 微信支付逻辑
    }

    @Override
    public void refund(double amount) {
        // 微信退款逻辑
    }
}

// 使用
public class PaymentService {
    public void processPayment(Payment payment, double amount) {
        payment.validateOrder();  // 多态调用父类方法
        payment.pay(amount);      // 多态调用子类实现
    }
}
```

这个例子展示了三种技术的完美配合：抽象类 `Payment` 封装了通用逻辑并定义了支付接口；`Alipay` 和 `WechatPay` 继承 `Payment` 并实现具体逻辑；调用方通过 `Payment` 类型处理所有支付方式，无需关心具体是哪一种。

## 常见误区与最佳实践

**滥用继承**：继承应该表示"是...的特例"关系，而不是"包含"关系。如果只是想复用代码，考虑使用组合（has-a）而不是继承（is-a）。

**脆弱的基类问题**：当父类改变时，可能破坏子类。这就是为什么Effective Java 建议"组合优于继承"。

**过度使用多态**：多态增加了代码的灵活性，但也增加了理解难度。在简单场景下，直接使用具体类型可能更清晰。

**忘记重写方法的 `@Override`**：虽然不是必须的，但这是一个好习惯，可以捕获拼写错误和签名不匹配的问题。

封装、继承和多态是 Java 语言的三大支柱。掌握它们，你就能编写出结构清晰、易于维护的代码。但这只是开始，下一章我们将探讨抽象类与接口的区别，这是 Java 中另一个重要的设计工具。
