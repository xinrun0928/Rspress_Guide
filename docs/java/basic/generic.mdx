小郑："泛型就是让集合在编译时就能检查类型，不用手动强转。"

面试官点点头，写下代码：

```java
List<String> list = new ArrayList<>();
```

小郑说："因为泛型会自动帮我们转型。"

小郑愣住了。

## 一、泛型基础 🔴

### 1.1 问题拆解

**第一层：怎么用？**
面试官问："Java 泛型是什么？为什么要引入泛型？"

**第二层：底层实现**
追问："`List<String>` 和 `List<Integer>` 在运行时有什么区别？"

**第三层：边界缺陷**
追问："为什么 Java 泛型要用类型擦除而不是像 C# 那样用 reified types？"

**第四层：选型 trade-off**
追问："泛型有什么限制？什么时候不能用泛型？"

### 1.2 标准回答

**泛型是 JDK 5 引入的参数化类型机制，让代码在编译时就能进行类型检查，避免强制类型转换**。

```java
// 没有泛型：需要手动强转，容易出错
List list = new ArrayList();
list.add("hello");
String s = (String) list.get(0);  // 必须强转，错了只能运行时才发现

// 有泛型：编译器自动检查和转换
List<String> list = new ArrayList<>();
list.add("hello");
String s = list.get(0);  // 不需要强转，编译时就检查类型
```

### 1.3 泛型的三大好处

| 好处 | 解释 | 示例 |
| --- | --- | --- |
| **类型安全** | 编译时检查，减少 ClassCastException | `list.add(123)` 编译报错 |
| **消除强转** | 不需要手动转型，代码更简洁 | `list.get(0)` 返回值直接是 String |
| **代码复用** | 一个类/方法可以处理多种类型 | `Collections.sort()` 可以排序任意类型的 List |

## 二、类型擦除：Java 泛型的本质 🔴

### 2.1 问题拆解

面试官追问："Java 泛型在编译之后是什么样的？"

### 2.2 类型擦除的过程

```java
// 源代码
public class Container<T> {
    private T value;

    public T getValue() {
        return value;
    }

    public void setValue(T value) {
        this.value = value;
    }
}
```

编译后的字节码（简化）：

```java
// 编译后，所有 T 都被替换为 Object（如果没有上限）
public class Container {
    private Object value;  // T 被替换为 Object

    public Object getValue() {
        return value;
    }

    public void setValue(Object value) {
        this.value = value;
    }
}
```

```java
// 如果有上限
public class Container<T extends Number> {
    private T value;
    public T getValue() { return value; }
}
```

编译后：

```java
// T extends Number，编译后 T 被替换为 Number
public class Container {
    private Number value;
    public Number getValue() { return value; }
}
```

:::tip 💡
**类型擦除**是 Java 泛型的核心机制：泛型信息只在编译期存在，用于类型检查；编译后，JVM 完全不知道泛型的存在，所有泛型类型参数都被替换为它们的上限（默认是 `Object`）。
:::

### 2.3 类型擦除的规则

| 泛型声明 | 上限 | 擦除后的类型 |
| --- | --- | --- |
| `<T>` | 无显式上限 → `Object` | `Object` |
| `<T extends Number>` | `Number` | `Number` |
| `<T extends Comparable>` | `Comparable` | `Comparable` |
| `<T extends A & B>` | 多上限 → 第一个 | `A` |
| `<?>` | 无限制 → `Object` | `Object` |

### 2.4 类型擦除带来的限制

```java
// 限制1：不能用 new T()
public <T> T createInstance(Class<T> clazz) {
    return new T();  // 编译错误！因为 T 在运行时是 Object
    // 正确做法：
    return clazz.newInstance();
}

// 限制2：不能用 new T[]
public <T> T[] createArray(int size) {
    return new T[size];  // 编译错误！
    // 正确做法：使用反射
    return (T[]) Array.newInstance(componentType, size);

// 限制3：不能用 instanceof T
if (obj instanceof T) {  // 编译错误！
    // 因为 T 在运行时不存在
}
```

## 三、通配符 🔴

### 3.1 问题拆解

面试官问："`List<Object>` 和 `List<?>` 有什么区别？"

### 3.2 无限定通配符 `?`

```java
// ? 表示未知类型
void printList(List<?> list) {
    for (Object item : list) {
        System.out.println(item);
    }
}

// 可以传入任意类型的 List
printList(Arrays.asList(1, 2, 3));
printList(Arrays.asList("a", "b", "c"));
printList(Arrays.asList(true, false));
```

### 3.3 上限通配符 `? extends T`

```java
// ? extends Number 表示 Number 或 Number 的子类
void printNumbers(List<? extends Number> list) {
    // 可以安全读取为 Number
    for (Number n : list) {
        System.out.println(n.doubleValue());
    }
    // 不能添加任何元素！因为不知道具体是什么类型
    // list.add(Integer.valueOf(1));  // 编译错误！
}

printNumbers(Arrays.asList(1, 2, 3));  // List<Integer> ✓
printNumbers(Arrays.asList(1.0, 2.0));  // List<Double> ✓
```

### 3.4 下限通配符 `? super T`

```java
// ? super Integer 表示 Integer 或 Integer 的父类
void addNumbers(List<? super Integer> list) {
    // 可以添加 Integer 或其子类
    list.add(1);  // ✓ Integer
    list.add(Integer.valueOf(1));  // ✓ Integer
    // list.add(Double.valueOf(1));  // ✗ Double 不是 Integer 的子类

    // 读取时只能作为 Object
    Object obj = list.get(0);  // ✓
    // Integer i = list.get(0);  // ✗ 编译错误！
}

addNumbers(new ArrayList<Integer>());  // ✓
addNumbers(new ArrayList<Number>());  // ✓ Number 是 Integer 的父类
addNumbers(new ArrayList<Object>());  // ✓ Object 是 Integer 的父类
```

### 3.5 PECS 原则：生产者/消费者

| 场景 | 通配符 | 原因 |
| --- | --- | --- |
| 生产数据（读取） | `? extends T` | 只能保证读出来的类型是 T |
| 消费数据（写入） | `? super T` | 只能写入 T 或 T 的子类 |
| 既生产又消费 | 不用通配符 | 用具体类型 |

```java
// Collections.copy 的签名
public static <T> void copy(List<? super T> dest, List<? extends T> src) {
    // dest 是消费者（写入），所以用 ? super T
    // src 是生产者（读取），所以用 ? extends T
    for (T item : src) {
        dest.add(item);
    }
}
```

## 四、泛型方法 🔴

### 4.1 基本用法

```java
// 泛型方法：在返回类型前声明类型参数
public static <T> T firstElement(List<T> list) {
    if (list == null || list.isEmpty()) {
        return null;
    }
    return list.get(0);
}

// 调用时自动推断类型
String s = firstElement(Arrays.asList("a", "b", "c"));
Integer i = firstElement(Arrays.asList(1, 2, 3));
```

### 4.2 泛型方法 vs 泛型类

```java
// 泛型类：类型参数在类声明时确定
class Container<T> {
    private T value;
    public T get() { return value; }
}

// 泛型方法：类型参数在方法调用时确定
class Utils {
    // 泛型方法，不需要类声明泛型
    public static <T> T firstElement(List<T> list) {
        return list.get(0);
    }

    // 静态泛型方法
    public static <T extends Comparable<T>> T max(T a, T b) {
        return a.compareTo(b) > 0 ? a : b;
    }
}
```

## 五、类型擦除的桥接方法 🔴

### 5.1 子类与父类的类型桥接

```java
// 源代码
class Parent<T> {
    public T method(T t) { return t; }
}

class Child extends Parent<String> {
    @Override
    public String method(String t) { return t; }
}
```

编译后（JVM 字节码层面）：

```java
class Parent {
    public Object method(Object t) { return t; }  // T 被擦除为 Object
}

class Child extends Parent {
    // JVM 生成的桥接方法
    public Object method(Object t) {
        return this.method((String) t);  // 调用我们的 String 版本
    }

    // 我们的原始方法
    public String method(String t) {
        return t;
    }
}
```

:::tip 💡
**桥接方法**是 JVM 为了保证多态正确性而自动生成的方法。当子类继承泛型父类并指定具体类型时，JVM 会生成一个桥接方法，将擦除后的签名桥接到原始方法。这是 Java 泛型的实现细节，也是面试中"为什么我重写的方法签名有 `@Override` 注解"的原因。
:::

## 六、生产避坑 🔴

### 6.1 坑：泛型数组

```java
// ❌ 危险：创建泛型数组
List<String>[] array = new ArrayList<String>[10];  // 编译警告，运行时出问题

// ✅ 正确：使用通配符数组
List<?>[] array = new ArrayList[10];
array[0] = new ArrayList<String>();
array[1] = new ArrayList<Integer>();

// ✅ 更优：避免数组，用 List
List<List<String>> listOfLists = new ArrayList<>();
```

### 6.2 坑：静态方法中的泛型

```java
class Container<T> {
    private static T staticValue;  // ❌ 编译错误！静态成员不能使用类泛型

    // ❌ 编译错误
    public static T getValue() {
        return staticValue;
    }

    // ✅ 正确：静态方法有自己的泛型参数
    public static <T> T create() {
        return null;
    }
}
```

### 6.3 坑：catch 中不能用泛型

```java
// ❌ 编译错误
try {
    process();
} catch (T e) {  // catch 中不能用类型参数
    // ...
}

// ✅ 正确：catch 只能用具体类型
try {
    process();
} catch (IOException e) {
    // ...
}
```

最后出一道综合题：

```java
public static void main(String[] args) {
    List<Integer> intList = new ArrayList<>();
    List<Number> numList = intList;  // 编译错误还是运行成功？
}
```

答案是**编译错误**。`List<Integer>` 不是 `List<Number>` 的子类型——即使 Integer 是 Number 的子类。这叫**泛型的不可变性**。如果 Java 让 `List<Integer>` 自动成为 `List<Number>` 的子类型，就可以通过 `numList.add(3.14)` 往里面加一个 Double，破坏 Integer 集合的类型安全。

:::tip 💡
理解"**Integer 是 Number，但 `List<Integer>` 不是 `List<Number>`**"——这叫"泛型的逆变性"，这是 Java 泛型最核心也最容易混淆的概念之一。
:::
