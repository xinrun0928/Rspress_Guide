## 一、为什么要引入泛型？ 🔴

### 1.1 问题拆解

**第一层：怎么用？**
```java
// 不用泛型 - 强制类型转换，容易 ClassCastException
List list = new ArrayList();
list.add("hello");
String s = (String) list.get(0); // 没问题

list.add(123);
String s2 = (String) list.get(1); // 运行时 ClassCastException!

// 用泛型 - 编译时类型检查
List<String> list2 = new ArrayList<>();
list2.add("hello");
// list2.add(123); // 编译错误！
String s3 = list2.get(0); // 不需要强转
```

**第二层：为什么需要泛型？**

面试官追问："有了泛型，为什么还要 ClassCastException 这个异常？"

你应该这样回答：
- **类型安全**：泛型在编译时检查类型，避免了 `ClassCastException` 在运行时才暴露
- **消除强转**：`List<String>` 的 `get()` 直接返回 `String`，不需要手动 `(String)` 强转
- **代码复用**：一套泛型代码可以支持多种类型，如 `List<T>`、`Pair<K, V>`

**第三层：翻车现场**
```java
// 90% 候选人踩过的坑
Object[] arr = new String[3];
arr[0] = "hello";
arr[1] = 123; // 编译通过，运行时 ArrayStoreException
```

【面试官心理】
我问他数组和泛型的区别，其实是想看他知不知道"数组是协变的、泛型是不变的"这个本质区别。知道这一层的人，大概有50%。

### 1.2 ❌ 错误示范

**候选人小李的原话：**
> "泛型就是让我们写代码更方便，不用强制类型转换。类型擦除就是泛型信息在编译后就没了。"

**问题诊断：**
1. 只答出了泛型的表象（不用强转），没答出本质（类型安全）
2. 知道类型擦除，但不知道具体怎么擦、擦成什么
3. 完全没提到通配符和边界

**面试官内心 OS：** 又是一个背八股的。

## 二、类型擦除深度剖析 🔴

### 2.1 什么是类型擦除？

类型擦除是 Java 泛型的核心机制：**泛型信息只存在于编译时，编译后进入 class 文件的只是裸类型（Raw Type）**。

```java
// 编译前
List<String> list = new ArrayList<>();
list.add("hello");
String s = list.get(0);

// 编译后（等价于）
List list = new ArrayList();
list.add("hello");
String s = (String) list.get(0);
```

### 2.2 擦除规则

**规则一：无界泛型擦除为 Object**
```java
List<String>      →  List (Object)
List<Integer>     →  List (Object)
List<T>           →  List (Object)
```

**规则二：有界泛型擦除为上界**
```java
List<? extends Number>  →  List (Number)
List<T extends Comparable> →  List (Comparable)
```

**规则三：类型参数擦除后，如果需要，编译器会自动插入类型转换指令**

### 2.3 擦除后源码验证

```java
// 这段代码的字节码等价于：
public static void main(String[] args) {
    List list = new ArrayList();
    list.add("hello");
    String s = (String) list.get(0); // 编译器自动插入的强转
}
```

### 2.4 ❌ 错误示范

**候选人小张：**
> "类型擦除就是把泛型信息删掉了，所以 `List<String>` 和 `List<Integer>` 在运行时分不出区别。"

**问题诊断：**
- 说对了一半，但完全忽略了**字节码中还有 Signature 属性**存储泛型信息
- 没提到 `getClass()` 返回的是同一个 `List.class`
- 不知道这个 Signature 属性是反射能够获取泛型信息的根本原因

:::tip 💡
`List<String>.class` 和 `List<Integer>.class` 在运行時是同一个 `List.class`，但通过反射可以获取泛型参数类型。
```java
List<String> list = new ArrayList<>();
Type type = list.getClass().getGenericSuperclass(); // 获取泛型类型信息
```
:::

### 2.5 类型擦除的代价

**代价一：泛型类型不可实例化**
```java
// ❌ 编译错误
T t = new T(); // 编译时根本不知道 T 是什么，无法实例化

// ✅ 正确做法：传入 Class 对象
<T> void createInstance(Class<T> clazz) {
    T t = clazz.newInstance();
}
```

**代价二：泛型静态变量共享问题**
```java
class Container<T> {
    static T value; // ❌ 编译错误 - 静态变量是所有对象共享的，但泛型类型不确定
}
```

**代价三：泛型数组无法直接创建**
```java
// ❌ 编译错误
T[] arr = new T[10];

// ✅ 正确做法：传入数组类型
<T> T[] createArray(Class<T> clazz, int size) {
    return (T[]) java.lang.reflect.Array.newInstance(clazz, size);
}
```

【面试官心理】
类型擦除这个点，我会连环追问：第一，擦成什么？第二，为什么擦成 Object 而不是 String？第三，擦除后怎么保证类型安全？这三个问题能完整答出来的，大概有40%。

## 三、通配符详解 🟡

### 3.1 三种通配符

```java
?                   // 无界通配符
? extends T          // 上界通配符（Producer）
? super T           // 下界通配符（Consumer）
```

### 3.2 PECS 原则：Producer-Extends, Consumer-Super

**场景一：只读取（生产数据）→ 用 extends**
```java
// ✅ 正确：读取 Number，但不知道具体是什么子类型
List<? extends Number> list = new ArrayList<Integer>();
Number num = list.get(0); // 安全，能读取为 Number

// ❌ 错误：不能写入
list.add(123); // 编译错误！list 可能是 ArrayList<Double>
```

**场景二：只写入（消费数据）→ 用 super**
```java
// ✅ 正确：可以写入 Integer 及其父类型
List<? super Integer> list = new ArrayList<Number>();
list.add(123); // 安全，Integer 肯定是 list 的某种父类型

// ❌ 错误：读取出来是 Object
Object obj = list.get(0); // 安全，但类型是 Object，没太大意义
```

**场景三：既读又写？→ 不用通配符**
```java
// 如果既要从 list 读又要写，用具体类型
List<Number> list = new ArrayList<Number>();
list.add(3.14); // 写 Number
Number num = list.get(0); // 读 Number
```

### 3.3 ❌ 错误示范

**候选人小王被追问时回答：**
> "`? extends T` 是上限，`? super T` 是下限，上限不能写、下限不能读。"

**问题诊断：**
- 背对了结论，但不理解原因
- 不知道"读"和"写"的具体含义
- 被追问"为什么"就卡住

**标准理解应该是：**
- `extends` 让变量变成**生产者**，只能往外吐数据（读），因为不知道具体子类型，写入会危险
- `super` 让变量变成**消费者**，只能往里塞数据（写），因为不知道具体父类型，读出来只能是 Object

### 3.4 PECS 实战应用

```java
// JDK 源码中的典型应用
public class Collections {
    // 从 source 读，往 dest 写
    public static <T> void copy(List<? super T> dest, List<? extends T> src) {
        for (T item : src) { // src 是生产者，extends T
            dest.add(item);  // dest 是消费者，super T
        }
    }
}
```

```java
// 日常开发中的例子
// 计算 List<Integer> 的总和
public double sum(List<? extends Number> list) {
    double sum = 0;
    for (Number n : list) { // 读出来是 Number
        sum += n.doubleValue();
    }
    return sum;
}
```

【面试官心理】
PECS 原则是我最爱追问的扩展问题。能完整解释 PECS 的是20%，能说出"不变和协变"的是10%，能把这个原则和日常代码结合的是5%。

## 四、上下界约束 🟡

### 4.1 上界 `<? extends T>`

```java
// Integer 的上界是 Number
List<? extends Integer> list1;  // 可能是 List<Integer> 或 List<Double>(Number的子类)

List<? extends Number> list2 = list1; // ✅ 合法，Integer extends Number
// list2.add(3.14); // ❌ 编译错误
```

### 4.2 下界 `<? super T>`

```java
// Integer 的下界是 Integer 本身，Integer 的上界是 Number
List<? super Integer> list1; // 可能是 List<Integer>、List<Number>、List<Object>

List<? super Integer> list2 = new ArrayList<Number>();
list2.add(123); // ✅ 合法，Integer 肯定能加入 Number 的列表
list2.add(3.14); // ❌ 编译错误，Double 不一定是 Integer 的父类型
```

### 4.3 协变与不变

| 类型 | 协变？ | 示例 |
|------|--------|------|
| 数组 | ✅ 协变 | `Number[] arr = new Integer[10];` |
| 泛型 | ❌ 不变 | `List<Number> = List<Integer>` ❌ |
| 通配符 | ✅ 有条件协变 | `List<? extends Number> = List<Integer>` ✅ |

```java
// 数组的协变陷阱
Number[] arr = new Integer[3];
arr[0] = 3.14; // 运行时 ArrayStoreException！

// 泛型的不变安全性
List<Number> list1 = new ArrayList<Number>();
// List<Integer> list2 = list1; // 编译错误！泛型是不变的
```

【面试官心理】
数组协变和泛型不变的对比，是我用来试探候选人对 Java 类型系统理解深度的问题。知道协变但不知道不变的占60%，知道两者区别的占30%。

## 五、追问升级：面试官的连环追问

### 5.1 第一轮追问：类型擦除细节

**面试官：** "类型擦除后，泛型信息真的完全消失了吗？"

**参考答案：**
不是完全消失。字节码中的 **Signature 属性** 存储了泛型信息，这是反射能够获取泛型类型的根本原因。

```java
// 通过反射获取泛型信息
Type type = ((ParameterizedType) list.getClass().getGenericSuperclass()).getActualTypeArguments()[0];
```

### 5.2 第二轮追问：泛型约束

**面试官：** "为什么不能用 `new T()`？为什么静态方法不能引用泛型变量？"

**参考答案：**
因为类型擦除后，T 被替换成 Object 或上界，但 Object 不能直接实例化。静态变量是所有对象共享的，而泛型类型是实例级别的，两者矛盾。

```java
class Container<T> {
    static T value; // ❌ 编译错误
    static void foo(T t) {} // ❌ 编译错误
}
```

### 5.3 第三轮追问：综合应用

**面试官：** "写一个方法，接收一个 `List<? extends Number>`，返回所有元素的和。"

**参考答案：**
```java
public static double sum(List<? extends Number> list) {
    double sum = 0;
    for (Number n : list) {
        if (n != null) {
            sum += n.doubleValue();
        }
    }
    return sum;
}
```

**面试官追问：** "为什么 `for (Number n : list)` 是安全的？"

因为 `? extends Number` 保证了列表中的元素一定是 Number 的子类，所以可以安全地当作 Number 处理。

## 六、生产避坑

### 6.1 泛型与反射组合翻车

```java
// 翻车现场：Spring 的 ClassUtils
// 有些旧代码通过反射获取泛型类型，但类型擦除后拿不到
public class UserService extends GenericService<User> {
    // 错误做法：通过反射获取泛型父类
    Class<?> actualType = (Class<?>) ((ParameterizedType)
        getClass().getGenericSuperclass()).getActualTypeArguments()[0];
    // 如果 UserService 被多次继承，类型擦除会导致拿到的是 Object
}
```

### 6.2 泛型数组的坑

```java
// 翻车现场：旧版 Gson 的 TypeToken
// 为了获取泛型数组，必须用匿名内部类"骗过"类型擦除
TypeToken<List<String>> token = new TypeToken<List<String>>(){};
Type type = token.getType(); // 匿名内部类能保留泛型信息！
```

**原理：** 匿名内部类的 superclass 会保留泛型信息，因为子类的 signature 包含了父类的类型参数。

### 6.3 泛型与 RPC 序列化

```java
// 翻车现场：Dubbo/FastJSON 泛型反序列化
// FastJSON 通过 ASM 获取泛型类型，但某些场景下类型擦除会导致反序列化失败
Type type = PersonService.class.getField("friends").getGenericType();
// 如果 friends 是 List<String>，这里拿到的是 ParameterizedType
// 如果泛型嵌套多层，ASM 可能解析失败
```

## 七、工程选型：什么场景用什么方案

| 场景 | 推荐方案 | 为什么 |
|------|----------|--------|
| 只读数据 | `List<? extends T>` | 生产者模式，安全读取 |
| 只写数据 | `List<? super T>` | 消费者模式，安全写入 |
| 既读又写 | 具体类型 `List<T>` | 需要明确类型 |
| 返回未知类型 | `<?>` 无界通配符 | 灵活但失去类型安全 |
| RPC 泛型反序列化 | 匿名内部类 + TypeToken | 利用子类保留泛型信息 |

:::tip 💡
阿里内部封装泛型工具时，通常会强制要求传入 `Class<T>` 参数，避免类型擦除带来的困扰。
:::
