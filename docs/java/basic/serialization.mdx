# Java 序列化

你有没有遇到过这种情况：程序在本地跑得好好的，一部署到服务器就开始报错——`NotSerializableException`。或者，你序列化了一个对象到文件，隔了半年后想反序列化回来，却发现类已经变了，JVM 报了一个 `InvalidClassException`。

这些问题的根源，都在于 Java 序列化机制的复杂性。

## 序列化的本质

Java 序列化（Serialization）是将对象转换为字节流的过程，反序列化（Deserialization）则是将字节流恢复为对象的过程。

```java
// 序列化：将对象写入文件
User user = new User("Alice", 25);
try (ObjectOutputStream oos = new ObjectOutputStream(
        new FileOutputStream("user.ser"))) {
    oos.writeObject(user);
}

// 反序列化：从文件读取对象
try (ObjectInputStream ois = new ObjectInputStream(
        new FileInputStream("user.ser"))) {
    User restored = (User) ois.readObject();
}
```

这个看似简单的机制，背后却涉及对象结构、版本控制、继承关系、安全性等多个维度。

## Serializable 接口：序列化的门槛

一个类要想被序列化，必须实现 `Serializable` 接口：

```java
public class User implements Serializable {
    private static final long serialVersionUID = 1L;

    private String name;
    private int age;

    // 必须有无参构造器
    public User() {}

    public User(String name, int age) {
        this.name = name;
        this.age = age;
    }
}
```

### serialVersionUID：版本控制的秘密

`serialVersionUID` 是序列化版本号，用于确保序列化和反序列化时使用同一个类结构。

```java
// 版本号不一致，会导致反序列化失败
private static final long serialVersionUID = 1L;
```

当你修改了类的结构（比如添加新字段），**必须更新版本号**，否则旧序列化的数据将无法恢复：

```java
// User 类升级，添加了新字段
public class User implements Serializable {
    private static final long serialVersionUID = 2L;  // 版本号必须更新

    private String name;
    private int age;
    private String email;  // 新增字段
}
```

JVM 在反序列化时，会比较序列化数据中的版本号和当前类的版本号。如果不一致，会抛出 `InvalidClassException`：

```text
java.io.InvalidClassException: com.example.User;
  local class incompatible: stream classdesc serialVersionUID = 1,
  local class serialVersionUID = 2
```

### transient：让敏感数据「隐身」

`transient` 关键字可以排除不想序列化的字段：

```java
public class User implements Serializable {
    private static final long serialVersionUID = 1L;

    private String name;
    private transient String password;  // 不会被序列化
    private transient int sessionId;    // 不会被序列化

    public User(String name, String password) {
        this.name = name;
        this.password = password;
    }
}
```

常见的 transient 应用场景：

| 字段类型 | 序列化影响 | 说明 |
|---|---|---|
| `transient` 修饰的字段 | 不序列化 | 值为 `null` 或默认值 |
| `static` 字段 | 不序列化 | 属于类，不属于对象 |
| `final` 基本类型 | 序列化 | 值会被保留 |
| `final` 引用类型 | 序列化 | 引用对象会被递归序列化 |

## 复杂对象的序列化

### 集合的序列化

集合类（`ArrayList`、`HashMap` 等）已经实现了 `Serializable`，但需要注意的是：**集合中存储的对象也必须实现 `Serializable`**。

```java
public class UserList implements Serializable {
    private static final long serialVersionUID = 1L;

    private List<User> users;

    public UserList() {
        this.users = new ArrayList<>();
    }

    public void addUser(User user) {
        users.add(user);
    }
}
```

### 继承关系的序列化

当父类实现了 `Serializable`，子类会自动继承序列化能力：

```java
public class Animal implements Serializable {
    private static final long serialVersionUID = 1L;
    protected String name;
}

public class Dog extends Animal {
    private static final long serialVersionUID = 1L;
    private String breed;

    public Dog(String name, String breed) {
        this.name = name;
        this.breed = breed;
    }
}
```

**但如果父类没有实现 `Serializable`，会有一个陷阱**：父类的字段不会被序列化，反序列化时会调用父类的无参构造器初始化。

```java
public class Animal {
    protected String name;

    // 如果没有无参构造器，反序列化会失败
    public Animal() {}
}

public class Dog extends Animal implements Serializable {
    private static final long serialVersionUID = 1L;
    private String breed;
}
```

## 自定义序列化逻辑

有时候，需要对序列化过程进行精细控制。可以通过重写 `writeObject` 和 `readObject` 方法来实现：

```java
public class User implements Serializable {
    private static final long serialVersionUID = 1L;

    private String name;
    private String password;
    private String email;

    // 自定义序列化逻辑
    private void writeObject(ObjectOutputStream oos) throws IOException {
        oos.defaultWriteObject();

        // 密码加密后存储
        oos.writeObject(encrypt(password));
    }

    // 自定义反序列化逻辑
    private void readObject(ObjectInputStream ois) throws IOException, ClassNotFoundException {
        oos.defaultReadObject();

        // 密码解密后恢复
        this.password = decrypt((String) ois.readObject());
    }

    private String encrypt(String password) {
        // 加密逻辑
        return Base64.getEncoder().encodeToString(password.getBytes());
    }

    private String decrypt(String encrypted) {
        // 解密逻辑
        return new String(Base64.getDecoder().decode(encrypted));
    }
}
```

### Externalizable：强制自定义序列化

`Externalizable` 接口比 `Serializable` 更进一步，它**完全控制序列化逻辑**：

```java
public class User implements Externalizable {
    private String name;
    private int age;

    public User() {}  // Externalizable 要求必须有公有无参构造器

    public User(String name, int age) {
        this.name = name;
        this.age = age;
    }

    @Override
    public void writeExternal(ObjectOutput out) throws IOException {
        out.writeObject(name);
        out.writeInt(age);
    }

    @Override
    public void readExternal(ObjectInput in) throws IOException, ClassNotFoundException {
        this.name = (String) in.readObject();
        this.age = in.readInt();
    }
}
```

## 序列化的性能问题

Java 原生序列化有几个公认的性能缺陷：

| 问题 | 说明 | 解决方案 |
|---|---|---|
| 序列化速度慢 | 需要递归遍历对象图 | 使用 Kryo、FST 等第三方库 |
| 序列化体积大 | 包含类元数据、header 等 | 使用 Protobuf、Avro 等二进制格式 |
| 无法跨语言 | Java 专用格式 | 使用 JSON、Protocol Buffers |

### 使用 Kryo 提升性能

Kryo 是一个高效的 Java 序列化库，性能比原生序列化快 10 倍以上：

```java
public class KryoSerializer {

    private static final ThreadLocal<Kryo> kryoThreadLocal = ThreadLocal.withInitial(Kryo::new);

    public static byte[] serialize(Object obj) {
        Kryo kryo = kryoThreadLocal.get();
        try (ByteArrayOutputStream baos = new ByteArrayOutputStream();
             Output output = new Output(baos)) {
            kryo.writeObject(output, obj);
            return baos.toByteArray();
        }
    }

    public static <T> T deserialize(byte[] bytes, Class<T> clazz) {
        Kryo kryo = kryoThreadLocal.get();
        try (ByteArrayInputStream bais = new ByteArrayInputStream(bytes);
             Input input = new Input(bais)) {
            return kryo.readObject(input, clazz);
        }
    }
}
```

## 序列化与安全

序列化是安全漏洞的高发区。攻击者可以通过构造恶意的序列化数据，在反序列化时执行任意代码。

这就是著名的 **反序列化漏洞**（Deserialization Vulnerability）。

防护措施：

1. **不要反序列化不可信的数据**
2. **使用白名单机制**：`ObjectInputStream` 的 `filter` 方法可以限制允许反序列化的类
3. **优先使用 JSON、Protocol Buffers** 等安全的替代方案

```java
ObjectInputStream ois = new ObjectInputStream(inputStream);
ois.setObjectInputFilter(new ObjectInputFilter() {
    @Override
    public ObjectFilterInfo checkInput(ObjectFilterInfo info) {
        // 白名单：只允许 User 和相关基础类型
        if (info.referencedType() == User.class) {
            return ObjectFilterInfo.Status.ALLOWED;
        }
        return ObjectFilterInfo.Status.REJECTED;
    }
});
```

## 总结

Java 序列化看似简单，实则暗藏玄机：

- `Serializable` 是序列化的门槛，`serialVersionUID` 是版本控制的纽带
- `transient` 可以排除敏感字段，但要注意继承关系带来的坑
- 自定义序列化逻辑通过 `writeObject`/`readObject` 实现
- 原生序列化性能堪忧，生产环境推荐 Kryo 或 Protobuf
- 永远不要反序列化不可信的数据

理解了这些，你就能在面试中从容应对序列化相关的问题，也能写出更安全的序列化代码。
