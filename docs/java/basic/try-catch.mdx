美团P6面试间里，面试官看着候选人小孙，问了一道看似简单的题：

"try-catch-finally 的执行顺序是什么？"

小孙说："先执行 try，如果有异常就跳到 catch，finally 最后执行。"

面试官点点头，写下了这段代码：

```java
public static int test() {
    int x = 1;
    try {
        return x;
    } finally {
        x = 2;
    }
}
```

"这个方法返回什么？"

小孙说："返回 1，因为 return 执行了。"

面试官追问："但 finally 里把 x 改成了 2，为什么还是返回 1？"

小孙："...因为 return 已经保存了返回值？"

面试官没说话，又写：

```java
public static int test() {
    try {
        throw new RuntimeException();
    } finally {
        return 3;
    }
}
```

"这个返回什么？异常还在吗？"

小孙彻底卡住了。

【面试官心理】
这道题我用来测试候选人对 JVM 执行流程的理解深度。知道"finally 总是执行"只是第一层，知道 finally 中的修改对返回值的影响是第二层，知道 finally 中的 return 会覆盖异常是第三层。能答到第三层的人，说明真正动手写过代码、研究过字节码。

## 一、基本执行顺序 🔴

### 1.1 问题拆解

**第一层：怎么用？**
面试官问："try-catch-finally 的执行顺序是什么？"

**第二层：底层实现**
追问："finally 是在什么时候执行的？是在 return 之前还是之后？"

**第三层：边界缺陷**
追问："如果 catch 里有 return，finally 什么时候执行？"

**第四层：选型 trade-off**
追问："finally 一定会执行吗？有没有办法让它不执行？"

### 1.2 标准回答：基本执行流程

```java
try {
    System.out.println("1. try 块开始");
    int x = 10 / 0;  // 抛出 ArithmeticException
    System.out.println("2. try 块结束（不会执行）");
} catch (ArithmeticException e) {
    System.out.println("3. catch 捕获到异常");
} finally {
    System.out.println("4. finally 块（总是执行）");
}

System.out.println("5. 方法结束");
```

输出顺序：`1 → 3 → 4 → 5`

### 1.3 finally 什么时候执行？

finally 的执行时机是**在方法返回之前**。但这个"之前"的含义很深：

```java
public static int test() {
    int x = 1;
    try {
        return x;  // 1. 先计算返回值 = 1
    } finally {
        x = 2;     // 2. 然后执行 finally
    }             // 3. 返回值已经是 1 了，所以返回 1
}
```

**finally 是在 return 表达式执行之后、真正返回之前执行的**。

:::tip 💡
理解这个的关键是：return x 不是一步完成的。JVM 实际上是两步：**第一步**计算 return 表达式的值（此时 x=1），**第二步**返回。如果 return 的是对象引用，则引用值（地址）先被保存，然后 finally 再修改对象内容。
:::

## 二、return 与 finally 的博弈 🔴

### 2.1 问题拆解

面试官追问："如果在 try 和 catch 里都有 return，finally 什么时候执行？"

### 2.2 规则一：finally 在 return 之前执行

```java
public static int test() {
    try {
        return 1;
    } catch (Exception e) {
        return 2;
    } finally {
        System.out.println("finally 执行");
    }
}
```

流程：
1. try 块执行到 `return 1`
2. return 表达式计算，返回值 = 1（暂存）
3. **执行 finally**
4. **返回之前暂存的 1**

输出：`finally 执行`，返回值 `1`

### 2.3 规则二：finally 中的修改对基本类型返回值无效

```java
public static int test() {
    int x = 1;
    try {
        return x;  // 返回值 = 1（已计算并暂存）
    } finally {
        x = 2;     // 这里修改 x，但返回值已经是 1 了
    }
}
```

返回：`1`，而不是 `2`

**为什么？** 因为 return 表达式 `x` 的值在返回之前就被计算并保存了。finally 修改 `x` 不会影响已经暂存的返回值。

### 2.4 规则三：finally 中的修改对引用类型返回值有效

```java
public static StringBuilder test() {
    StringBuilder sb = new StringBuilder("hello");
    try {
        return sb;  // 返回值 = sb 的引用（地址）
    } finally {
        sb.append(" world");  // 这里修改了 sb 的内容
    }              // sb 的引用没变，但内容变了
}
```

返回：`hello world`

**为什么？** 因为 `return sb` 保存的是**引用值（对象的内存地址）**，而不是对象的副本。finally 中修改了对象的内容，返回的对象已经被改变了。

:::warning ⚠️
这是一个经典的面试陷阱：基本类型在 finally 中修改不会影响返回值，但引用类型会。能解释清楚这个区别的候选人，说明真正理解 Java 的值传递机制。
:::

## 三、finally 中的 return 🔴

### 3.1 问题拆解

面试官追问："如果 finally 中也有 return，会发生什么？"

### 3.2 规则四：finally 中的 return 会覆盖 try/catch 的返回值

```java
public static int test() {
    try {
        return 1;
    } catch (Exception e) {
        return 2;
    } finally {
        return 3;  // 这个 return 会覆盖 try 和 catch 的返回值！
    }
}
```

返回：`3`

### 3.3 规则五：finally 中的 return 会静默吞掉异常

这是生产中最危险的坑：

```java
public static int test() {
    try {
        throw new RuntimeException("原始异常");
    } catch (Exception e) {
        throw new RuntimeException("catch 异常");
    } finally {
        return 3;  // 异常被吞掉了！
    }
}
```

调用方：
```java
try {
    int result = test();
    System.out.println("返回: " + result);
} catch (Exception e) {
    System.out.println("捕获到异常: " + e.getMessage());
}
```

输出：`返回: 3`——**异常完全消失了！**

:::warning ⚠️
finally 中的 return 是 Java 语法允许的，但这是最糟糕的异常处理方式。它会静默吞掉 try 和 catch 中抛出的异常，导致调试地狱。**永远不要在 finally 中写 return 语句。**
:::

## 四、异常传播与 finally 🔴

### 4.1 规则六：finally 中的异常会覆盖原有异常

```java
public static void test() {
    try {
        throw new RuntimeException("原始异常");
    } finally {
        throw new RuntimeException("finally 异常");
    }
}
```

finally 中的异常会**覆盖** try 中的异常：

```java
public static void main(String[] args) {
    try {
        test();
    } catch (Exception e) {
        System.out.println(e.getMessage());  // 打印 "finally 异常"
    }
}
```

原始异常被吞掉了！

### 4.2 正确做法：捕获并处理

```java
public static void test() {
    Exception original = null;
    try {
        throw new RuntimeException("原始异常");
    } catch (Exception e) {
        original = e;
    } finally {
        try {
            throw new RuntimeException("finally 异常");
        } catch (Exception e) {
            // 如果有原始异常，优先记录原始异常
            if (original != null) {
                original.addSuppressed(e);  // JDK7+ 抑制异常
                throw original;
            }
            throw e;
        }
    }
}
```

JDK 7 引入了**抑制异常（Suppressed Exception）**机制：

```java
public class Throwable {
    public synchronized void addSuppressed(Throwable exception) {
        // 将 finally 中的异常标记为"被抑制的异常"
        // 可以通过 getSuppressed() 获取
    }

    public Throwable[] getSuppressed() {
        // 返回被抑制的异常数组
    }
}
```

## 五、finally 不执行的情况 🔴

### 5.1 JVM 退出

```java
try {
    System.out.println("try");
    System.exit(0);  // JVM 直接退出
} finally {
    System.out.println("finally");  // 不会执行！
}
```

### 5.2 其他 JVM 停止方式

```java
try {
    Runtime.getRuntime().halt(0);  // 强制停止 JVM
} finally {
    System.out.println("finally");  // 不会执行！
}
```

### 5.3 死循环（很少见但确实存在）

```java
try {
    System.out.println("try");
    while (true) {
        // 无限循环
    }
} finally {
    System.out.println("finally");  // 不会执行！
}
```

## 六、面试高频追问 🔴

### 6.1 追问：try-with-resources

Java 7 引入的 try-with-resources 是处理资源的最佳实践：

```java
// 传统写法
BufferedReader br = null;
try {
    br = new BufferedReader(new FileReader("file.txt"));
    return br.readLine();
} finally {
    if (br != null) {
        try {
            br.close();
        } catch (IOException e) {
            e.printStackTrace();
        }
    }
}

// try-with-resources 写法
try (BufferedReader br = new BufferedReader(new FileReader("file.txt"))) {
    return br.readLine();
}  // 自动调用 close()，并且在 finally 之后才返回
```

**执行顺序**：
1. 资源初始化（括号内）
2. try 块
3. 自动 close()
4. 返回

如果 close() 也抛异常？会被**抑制异常**机制处理，不会覆盖原有异常。

:::tip 💡
Java 7+ 应该优先使用 try-with-resources 而不是传统的 try-finally。自动资源管理（ARM）更简洁、更安全，异常处理也更合理。
:::

### 6.2 追问：catch 块的选择

```java
// ❌ 错误：顺序错误
try {
    process();
} catch (Exception e) {        // 先捕获 Exception
    handle(e);
} catch (ArithmeticException e) {  // 这行永远不会执行！
    handleArithmetic(e);
}

// ✅ 正确：从小到大排序
try {
    process();
} catch (ArithmeticException e) {  // 先捕获具体异常
    handleArithmetic(e);
} catch (Exception e) {              // 再捕获通用异常
    handle(e);
}
```

## 七、生产避坑 🔴

### 7.1 坑：finally 中的资源关闭失败

```java
// ❌ 危险写法
try {
    conn = getConnection();
    process(conn);
} finally {
    conn.close();  // 如果 close() 抛异常，原有异常会被覆盖
}

// ✅ 正确做法：嵌套 try
try {
    conn = getConnection();
    process(conn);
} finally {
    try {
        if (conn != null) conn.close();
    } catch (SQLException e) {
        // 记录但不要抛，否则覆盖原异常
        log.error("close connection failed", e);
    }
}
```

### 7.2 坑：finally 中的异常掩盖业务异常

```java
// ❌ 危险
public int getUserAge(User user) {
    try {
        return user.getAge();
    } finally {
        System.out.println("finally");  // 正常
        // 假设这里有个隐藏的 bug 会抛异常
    }
}

// 假设 user 为 null，getAge() 抛 NPE
// 但 finally 的异常会覆盖 NPE，调试时只能看到 finally 的异常
```

最后出一道综合题：

```java
public static String test() {
    String s = "hello";
    try {
        return s;
    } finally {
        s = s + " world";
    }
}
```

这个方法返回什么？答案是 `"hello"`，因为 return 时 s 的引用指向的是 `"hello"` 这个 String 对象（String 是不可变的），finally 中 s = s + " world" 创建了一个新的 String 对象 `"hello world"`，但返回的还是原来暂存的 `"hello"`。

:::tip 💡
如果把 String 换成 StringBuilder，那么返回的就是 `"hello world"`，因为 StringBuilder 是可变的，finally 中 append 修改的是同一个对象。
:::
