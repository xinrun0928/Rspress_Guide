美团P6面试间里，面试官看着候选人小马的简历，上面写着"熟悉Java核心机制"。

"问个简单的，"面试官说，"`==` 和 `equals` 有什么区别？"

小马答道："`==` 比较的是引用地址，equals 比较的是内容。"

面试官点点头，写下一行代码：

```java
String a = new String("hello");
String b = new String("hello");
System.out.println(a == b);
System.out.println(a.equals(b));
```

"这两个输出什么？"

小马："false，true。"

"为什么 `equals` 是 true？"

小马："因为 String 重写了 equals 方法。"

面试官又写：

```java
String c = "hello";
String d = "hello";
System.out.println(c == d);
System.out.println(c.equals(d));
```

"这个呢？"

小马开始犹豫了："...c == d 是 true？不对，是 false？"

面试官："你刚才不是说 `==` 比较地址吗？a 和 b 是两个 new 出来的对象，地址不同。c 和 d 呢？"

小张彻底卡住了。

【面试官心理】
这道题看似简单，但能答到最后一层的人不超过 30%。我想知道他有没有真正理解 Java 的内存模型、String 池的机制，以及 equals 方法的默认行为。为什么面试官爱问 String？因为 String 是 Java 中被重写得最彻底的类，它完美地展示了 == 和 equals 的区别。

## 一、`==` 运算符 🔴

### 1.1 问题拆解

**第一层：怎么用？**
面试官问："`==` 可以比较什么？"

**第二层：底层实现**
追问："`==` 比较的是值还是地址？基本类型和引用类型一样吗？"

**第三层：边界缺陷**
追问："`==` 能比较两个数组吗？`[1,2] == [1,2]` 有意义吗？"

**第四层：选型 trade-off**
追问："什么时候用 `==` 而不是 equals？"

### 1.2 标准回答

`==` 运算符比较的是**值**。但基本类型和引用类型"值"的含义不同：

```java
// 基本类型：== 比较的是具体的数值
int a = 10;
int b = 10;
System.out.println(a == b);  // true，比较的是 10 == 10

// 引用类型：== 比较的是内存地址（引用）
String x = new String("hello");
String y = new String("hello");
System.out.println(x == y);  // false，new 出来的是两个不同对象，地址不同

// 特殊情况：字面量 String
String p = "hello";   // 字面量，JVM 会优先从 String Pool 查找
String q = "hello";   // 直接复用 String Pool 中的同一个对象
System.out.println(p == q);  // true！因为 p 和 q 指向 String Pool 中的同一个对象
```

#### 基本类型 vs 引用类型

| 类型 | `==` 比较的是什么 | 说明 |
| --- | --- | --- |
| byte, short, int, long | 数值 | 基本类型没有引用的概念 |
| float, double | 数值 | 基本类型没有引用的概念 |
| char, boolean | 数值 | 基本类型没有引用的概念 |
| 引用类型 | 地址（引用） | 比较两个引用是否指向同一个对象 |

:::warning ⚠️
很多人说"`==`比较地址"——这个说法不够准确。更准确的说法是：**基本类型比较值，引用类型比较引用（地址）**。如果非要说地址，那是因为引用类型存储的就是地址。
:::

## 二、equals 方法 🔴

### 2.1 问题拆解

**第一层：怎么用？**
面试官问："equals 和 == 有什么区别？"

**第二层：底层实现**
追问："Object 的 equals 默认实现是什么？为什么需要重写？"

**第三层：边界缺陷**
追问："equals 和 hashCode 有什么关系？不一起重写会出什么问题？"

**第四层：选型 trade-off**
追问："你平时是怎么重写 equals 的？用 IDE 自动生成还是自己写？"

### 2.2 Object 的 equals 默认实现

这是 P6 高频翻车点。`Object` 的 equals 默认实现：

```java
// Object.java（JDK 源码）
public boolean equals(Object obj) {
    return this == obj;  // 默认就是 == 比较地址
}
```

所以，如果不重写 equals，**equals 和 == 没有任何区别**！这就是为什么 String、Integer 这些类都要重写 equals。

### 2.3 String 的 equals 重写

```java
// String.java（JDK 源码，简化）
public boolean equals(Object anObject) {
    if (this == anObject) {           // 第一步：先比较地址，地址相同直接 true
        return true;
    }
    if (anObject instanceof String) { // 第二步：检查类型
        String anotherString = (String)anObject;
        int n = value.length;         // value 是 char[]，存储字符串内容
        if (n == anotherString.value.length) {
            char v1[] = value;
            char v2[] = anotherString.value;
            int i = 0;
            while (n-- != 0) {        // 第三步：逐字符比较内容
                if (v1[i] != v2[i])
                    return false;
                i++;
            }
            return true;
        }
    }
    return false;
}
```

String equals 的三步走策略：**地址比较（快速路径）→ 类型检查 → 逐字符比较**。第一步用 `==` 做快速短路，避免不必要的字符比较。

### 2.4 为什么 String 字面量 `==` 是 true？

这就要说到 String Pool 了：

```java
String a = "hello";   // 字面量，JVM 从 String Pool 查找
String b = "hello";   // 直接复用 String Pool 中的对象
String c = new String("hello");  // new 出来的一定是新对象

System.out.println(a == b);  // true：a 和 b 指向 String Pool 中同一个对象
System.out.println(a == c);  // false：a 在 String Pool，c 在堆
System.out.println(a.equals(b));  // true
System.out.println(a.equals(c));  // true
```

:::tip 💡
String Pool 是 JVM 对 String 的一个优化：相同内容的字面量 String 共享同一个对象，节省内存。但 `new String()` 一定会在堆中创建新对象。
:::

### 2.5 equals 的正确重写方式

这道题是 P6+ 的高频考察点。IDE 自动生成的 equals 模板：

```java
@Override
public boolean equals(Object o) {
    // 1. 自反性：自己和自己比一定是 true
    if (this == o) return true;

    // 2. 类型检查：必须是同一个类（注意 instanceof vs getClass()）
    if (o == null || getClass() != o.getClass()) return false;

    // 3. 类型转换
    Student student = (Student) o;

    // 4. 逐字段比较：基本类型用 ==，引用类型用 Objects.equals()
    return age == student.age &&
           Objects.equals(name, student.name);
}
```

#### instanceof vs getClass()

| 方法 | 行为 | 适用场景 |
| --- | --- | --- |
| `o instanceof Student` | 允许子类对象通过父类 equals 检查 | 需要父子类在语义上相等 |
| `o.getClass() == Student.class` | 必须类型完全一致 | 需要严格类型匹配 |

:::warning ⚠️
`instanceof` 和 `getClass()` 的选择会影响 equals 的对称性和传递性。阿里巴巴规范手册推荐用 `getClass()` 进行严格类型检查，除非你明确需要父类 equals 在子类上也能返回 true。
:::

## 三、equals 和 hashCode 的关系 🔴

### 3.1 问题拆解

**第一层：怎么用？**
面试官问："重写 equals 时为什么要重写 hashCode？"

**第二层：底层实现**
追问："hashCode 相等的对象，equals 一定相等吗？反过来呢？"

**第三层：边界缺陷**
追问："把对象放到 HashMap 里但不重写 hashCode，会发生什么？"

**第四层：选型 trade-off**
追问："equals 和 hashCode 的重写有什么规范？"

### 3.2 标准回答

**hashCode 和 equals 的契约**：

```java
// Object 规范定义的契约
// 1. 同一个对象，hashCode 必须相同
obj.hashCode() == obj.hashCode();  // 永远为 true

// 2. equals 返回 true 的两个对象，hashCode 必须相同
if (a.equals(b)) {
    a.hashCode() == b.hashCode();  // 必须为 true
}

// 3. hashCode 相同的对象，equals 不一定为 true（这就是哈希碰撞）
// 4. equals 返回 false 的对象，hashCode 可以相同也可以不同
```

违反契约会导致什么问题？来看一个真实翻车案例：

```java
public class Student {
    private String name;
    private int age;

    // 只重写了 equals，没重写 hashCode
    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (o == null || getClass() != o.getClass()) return false;
        Student s = (Student) o;
        return age == s.age && Objects.equals(name, s.name);
    }
    // hashCode 没重写！默认返回的是对象的内存地址相关的值
}

public static void main(String[] args) {
    Student s1 = new Student("张三", 20);
    Student s2 = new Student("张三", 20);

    System.out.println(s1.equals(s2));   // true
    System.out.println(s1.hashCode());   // 某个数，比如 123456
    System.out.println(s2.hashCode());  // 另一个数，比如 789012

    // 放到 HashMap 里试试
    Map<Student, String> map = new HashMap<>();
    map.put(s1, "一班");

    System.out.println(map.get(s2));  // null！！！因为 s1 和 s2 的 hashCode 不同
    // map 用 hashCode 找到桶的位置，但 s2 的 hashCode 和 s1 不同，
    // 所以根本不会去同一个桶里找，自然就找不到了
}
```

:::warning ⚠️
**equals 为 true 的两个对象，hashCode 必须相等**——这是 HashMap、HashSet 等哈希结构的底层要求。违反这条，会导致 HashMap 找不到明明"相等"的对象。
:::

### 3.3 hashCode 的重写规范

IDE 自动生成的模板：

```java
@Override
public int hashCode() {
    int result = age;
    result = 31 * result + (name != null ? name.hashCode() : 0);
    return result;
}
```

为什么用 31？

| 选择 | 原因 |
| --- | --- |
| 31 是质数 | 质数在乘法中能更好地分散哈希值，减少碰撞 |
| 31 的位运算优化 | `31 * i` 可以写成 `i * 31 = i << 5 - i`，现代 JVM 会自动优化 |
| 31 不大不小 | 太大容易溢出，太小容易碰撞 |

## 四、高频面试追问 🔴

### 4.1 追问：为什么 String 用 `==` 比较有时候对有时候错？

```java
String a = "hello";
String b = "hello";
String c = new String("hello");
String d = new String("hello");

System.out.println(a == b);        // true：字面量在 String Pool 中是同一个对象
System.out.println(c == d);        // false：new 出来的是两个不同对象
System.out.println(a == c);        // false：a 在 String Pool，c 在堆
System.out.println("hello" == a);  // true：字面量和 String 变量都是同一个 Pool 对象
```

**记忆口诀**：String 的 `==`，字面量对字面量才靠谱，其他情况全部是坑。

### 4.2 追问：包装类的缓存

```java
Integer a = 127;
Integer b = 127;
System.out.println(a == b);  // true！因为 Integer 缓存了 -128~127

Integer c = 128;
Integer d = 128;
System.out.println(c == d);  // false！超出缓存范围，每次都 new

// 验证
Integer e = new Integer(127);
System.out.println(a == e);  // false！new 出来的对象不在缓存里
```

Java 的包装类（Integer、Byte、Short、Long、Character、Boolean）都有缓存机制，但范围不同。**Integer 缓存 -128~127，Boolean 只有 true/false 两个值**。

:::tip 💡
阿里巴巴 Java 开发手册明确规定：**禁止用 `==` 比较包装类型**，应该用 `equals`。因为缓存范围不可预测，用 `==` 极容易写出隐蔽的 bug。
:::

## 五、生产避坑

### 5.1 坑：HashSet/HashMap 的 key 用了未重写 equals/hashCode 的类

```java
// 自定义类做 HashMap 的 key
class Order {
    private String orderId;

    // 只重写了 equals，没重写 hashCode
    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (o == null || getClass() != o.getClass()) return false;
        Order order = (Order) o;
        return Objects.equals(orderId, order.orderId);
    }
    // hashCode 漏了！
}

public static void main(String[] args) {
    Map<Order, String> map = new HashMap<>();
    Order o1 = new Order("O001");
    Order o2 = new Order("O001");

    map.put(o1, "待支付");
    System.out.println(map.get(o1));  // "待支付"
    System.out.println(map.get(o2));  // null！明明 o1 和 o2 是同一个订单
}
```

**排查方法**：如果 HashMap/HashSet 的 get() 莫名返回 null，先检查 key 类的 equals 和 hashCode 是否正确重写。

### 5.2 坑：equals 的对称性/传递性/自反性被破坏

```java
class A {
    int x;
    @Override
    public boolean equals(Object o) {
        if (o instanceof B) return x == ((B) o).y;
        if (o instanceof A) return x == ((A) o).x;
        return false;
    }
}

class B {
    int y;
    @Override
    public boolean equals(Object o) {
        if (o instanceof B) return y == ((B) o).y;
        if (o instanceof A) return y == ((A) o).x;
        return false;
    }
}
```

A 和 B 的 equals 实现不对称——A.equals(B) 和 B.equals(A) 可能返回不同结果。这种 bug 在 HashMap 里会导致极其诡异的行为。**永远用 IDE 自动生成 equals/hashCode，除非你完全理解对称性、传递性、自反性的要求**。
