## 一、反射是什么 🔴

### 1.1 问题拆解

**第一层：怎么用？**
面试官问："Java 反射是什么？为什么需要反射？"

**第二层：底层实现**
追问："Class 对象是怎么来的？Class.forName() 和 .class 语法的区别是什么？"

**第三层：边界缺陷**
追问："反射为什么慢？慢在哪里？有什么优化手段？"

**第四层：选型 trade-off**
追问："你在项目里用过反射吗？什么场景下用反射而不是直接调用？"

### 1.2 标准回答

**反射是 Java 在运行时动态获取类的信息（如方法、字段、构造函数）和动态调用对象方法的能力**。

正常调用 vs 反射调用：

```java
// 正常调用：编译时就知道要调用 UserService 的 addUser 方法
UserService service = new UserService();
service.addUser(user);

// 反射调用：运行时才知道要调用哪个类的哪个方法
Class<?> clazz = Class.forName("com.example.UserService");
Object service = clazz.getDeclaredConstructor().newInstance();
Method method = clazz.getDeclaredMethod("addUser", User.class);
method.invoke(service, user);
```

### 1.3 为什么需要反射？

| 场景 | 不用反射的问题 | 反射解决了什么 |
| --- | --- | --- |
| Spring IOC | 编译时不知道需要创建哪些对象 | 运行时扫描并创建 Bean |
| JDBC 驱动加载 | 编译时不知道要用哪个数据库驱动 | 运行时动态加载驱动类 |
| 序列化/反序列化 | 不知道有哪些字段要序列化 | 运行时获取所有字段 |
| 通用框架 | 每个类都要写一遍 | 通过注解标记，自动处理 |

## 二、Class 对象：反射的核心 🔴

### 2.1 问题拆解

面试官追问："Class 对象是什么？Class.forName() 和 .class 语法有什么区别？"

### 2.2 三种获取 Class 对象的方式

```java
// 方式1：.class 语法（编译时确定，最快）
Class<?> clazz1 = String.class;

// 方式2：Object.getClass()
Class<?> clazz2 = "hello".getClass();

// 方式3：Class.forName()（运行时加载，可能抛 ClassNotFoundException）
Class<?> clazz3 = Class.forName("java.lang.String");

// 三种方式获取的是同一个 Class 对象
System.out.println(clazz1 == clazz2);  // true
System.out.println(clazz1 == clazz3);  // true
```

### 2.3 Class.forName() vs .class 语法的本质区别

```java
// .class 语法：只触发 Class 对象的加载和链接，不触发初始化
Class<?> c1 = int.class;  // Class 对象已存在，JVM 不会重新加载

// Class.forName()：会触发 Class 的加载、链接、初始化（clinit）
Class<?> c2 = Class.forName("java.lang.String");  // 完整初始化

// 区别在哪里？看静态代码块
public class Dog {
    static {
        System.out.println("Dog 类初始化");
    }
}

// 测试
Class<?> c1 = Dog.class;  // 不打印！因为只加载不初始化
Class<?> c2 = Class.forName("Dog");  // 打印！触发了完整初始化
```

:::tip 💡
`.class` 语法只做加载和链接，**不执行类的初始化**（static 块不执行）。`Class.forName()` 默认执行完整初始化。如果只想加载类而不初始化，用 `Class.forName(name, false, classLoader)` 第二个参数设为 `false`。
:::

### 2.4 Class 对象的结构

```mermaid
graph TD
    A[Class 对象] --> B[类的基本信息<br/>名称、父类、接口、修饰符]
    A --> C[Constructor[]<br/>所有构造函数]
    A --> D[Method[]<br/>所有方法]
    A --> E[Field[]<br/>所有字段]
    A --> F[Annotation[]<br/>注解]
```

## 三、反射的核心操作 🔴

### 3.1 获取 Constructor：创建对象

```java
// 获取无参构造函数
Constructor<?> constructor = clazz.getDeclaredConstructor();
Object obj = constructor.newInstance();

// 获取有参构造函数
Constructor<?> constructor = clazz.getDeclaredConstructor(String.class, int.class);
Object obj = constructor.newInstance("张三", 20);

// 访问私有构造函数
Constructor<?> privateConstructor = clazz.getDeclaredConstructor(String.class);
privateConstructor.setAccessible(true);  // 绕过访问检查
Object obj = privateConstructor.newInstance("李四");
```

### 3.2 获取 Method：调用方法

```java
// 获取公开方法
Method setName = clazz.getDeclaredMethod("setName", String.class);
setName.invoke(obj, "新名字");

// 获取私有方法
Method privateMethod = clazz.getDeclaredMethod("process", null);
privateMethod.setAccessible(true);  // 绕过访问检查
privateMethod.invoke(obj);

// 获取静态方法
Method staticMethod = clazz.getDeclaredMethod("getVersion", null);
staticMethod.invoke(null);  // 静态方法不需要对象实例
```

### 3.3 获取 Field：读写字段

```java
// 获取公开字段
Field nameField = clazz.getDeclaredField("name");
nameField.set(obj, "王五");

// 获取私有字段
Field privateField = clazz.getDeclaredField("password");
privateField.setAccessible(true);
String password = (String) privateField.get(obj);

// 读取/写入静态字段
Field versionField = clazz.getDeclaredField("VERSION");
Object value = versionField.get(null);  // 静态字段不需要对象实例
```

## 四、反射的性能开销 🔴

### 4.1 问题拆解

面试官追问："反射为什么慢？慢在哪里？"

### 4.2 性能测试

```java
// 普通调用 vs 反射调用
public class PerformanceTest {
    public int add(int a, int b) { return a + b; }
}

// 基准测试结果（JMH）
// 普通调用：1000万次调用 ≈ 10ms
// 反射调用（无优化）：1000万次调用 ≈ 3000ms（慢 300 倍）
// 反射调用（MethodHandle）：1000万次调用 ≈ 50ms
// 反射调用（setAccessible + 缓存）：1000万次调用 ≈ 80ms
```

### 4.3 反射慢的三个原因

| 原因 | 解释 | 性能影响 |
| --- | --- | --- |
| **运行时类型检查** | 每次调用都要检查参数类型、访问权限 | 约 2-3 倍开销 |
| **Native 方法调用** | `Method.invoke()` 内部调用 JVM 的 native 方法 | 约 2 倍开销 |
| **无法 JIT 优化** | 反射调用的目标在运行时才知道，JIT 编译器无法内联 | 约 10 倍开销 |

```java
// Method.invoke() 的内部实现（简化）
public Object invoke(Object obj, Object... args) {
    // 1. 参数类型检查
    if (args == null) args = EmptyArray.OBJECT;
    // 2. 权限检查（setAccessible）
    if (override) {
        // 如果设置了 override，跳过访问检查
    } else {
        checkAccess();  // 检查方法访问权限
    }
    // 3. 调用 native 方法
    return invoke0(method, obj, args);
}
```

### 4.4 反射的优化方案

```java
// 优化1：方法缓存（最重要的优化）
public class ReflectCache {
    private static final Map<String, Method> METHOD_CACHE = new ConcurrentHashMap<>();

    public static Object invoke(Object target, String methodName, Object... args) {
        String key = target.getClass().getName() + "#" + methodName;
        Method method = METHOD_CACHE.computeIfAbsent(key, k -> {
            try {
                Method m = target.getClass().getDeclaredMethod(methodName);
                m.setAccessible(true);
                return m;
            } catch (NoSuchMethodException e) {
                throw new RuntimeException(e);
            }
        });
        return method.invoke(target, args);
    }
}

// 优化2：使用 MethodHandle（JDK7+，比反射更快的动态调用）
public Object invokeWithMethodHandle(Object target, String methodName) throws Throwable {
    MethodHandles.Lookup lookup = MethodHandles.lookup();
    MethodType mt = MethodType.methodType(int.class, int.class, int.class);
    MethodHandle mh = lookup.findVirtual(Target.class, "add", mt);
    return mh.invoke(target, 1, 2);
}

// 优化3：避免 setAccessible（访问检查有额外开销）
// 如果方法是 public 的，不需要 setAccessible
Method publicMethod = clazz.getMethod("publicMethod");  // 不需要 setAccessible
```

## 五、生产避坑 🔴

### 5.1 坑：反射破坏封装

```java
public class SecureData {
    private String password = "123456";
}

public static void main(String[] args) throws Exception {
    SecureData data = new SecureData();
    Field field = SecureData.class.getDeclaredField("password");
    field.setAccessible(true);
    String pwd = (String) field.get(data);
    System.out.println(pwd);  // 打印 "123456"，密码被偷了！
}
```

`setAccessible(true)` 绕过了 Java 的访问控制机制。这是安全漏洞的常见来源——攻击者可以通过反射访问私有字段。

:::warning ⚠️
在 Java 9+ 中，模块系统对反射有更严格的限制。如果你的类在模块化系统中，`setAccessible(true)` 可能会失败。应该使用 `MethodHandles.Lookup` 的 `privateLookup` 来获取私有方法的访问权限。
:::

### 5.2 坑：Spring 中的反射：循环依赖

```java
// Spring 反射创建 Bean 的简化流程
public Object createBean(Class<?> beanClass) {
    try {
        Constructor<?> constructor = beanClass.getDeclaredConstructor();
        constructor.setAccessible(true);
        Object bean = constructor.newInstance();
        // 然后继续填充属性（@Autowired 字段）
        // 这里会递归创建依赖的 Bean
        return bean;
    } catch (Exception e) {
        throw new BeanCreationException(beanClass.getName(), e);
    }
}
```

Spring 的循环依赖检测不到所有情况，当 Bean A 依赖 Bean B、Bean B 依赖 Bean A 时，反射创建过程中可能会出现 StackOverflowError。

### 5.3 坑：反射与序列化

```java
// 使用反射做序列化/反序列化
public <T> T deserialize(String json, Class<T> clazz) {
    T obj = clazz.getDeclaredConstructor().newInstance();
    // 反射设置每个字段
    return obj;
}
```

这把双刃剑——反射让序列化框架通用，但性能开销也很大。Fastjson、Jackson 等框架都大量使用反射做序列化，这是它们的性能瓶颈之一。

## 六、反射的应用场景 🔴

### 6.1 Spring IOC：Bean 的创建

```java
// Spring 反射创建 Bean 的简化源码
public Object createBean(Class<?> beanClass, String beanName) {
    Object beanInstance = null;
    try {
        Constructor<?>[] constructors = beanClass.getDeclaredConstructors();
        // 找到带 @Autowired 注解的构造函数
        Constructor<?> autowiredConstructor = findAutowiredConstructor(constructors);
        if (autowiredConstructor != null) {
            // 使用 autowiredConstructor 创建
            beanInstance = autowiredConstructor.newInstance(resolveDependencies(autowiredConstructor));
        } else {
            // 使用默认构造函数
            beanInstance = beanClass.getDeclaredConstructor().newInstance();
        }
    } catch (Exception e) {
        throw new BeanCreationException(beanClass.getName(), e);
    }
    return beanInstance;
}
```

### 6.2 JDBC 驱动加载

```java
// JDBC 4.0 之前：必须用 Class.forName 加载驱动
Class.forName("com.mysql.cj.jdbc.Driver");

// JDBC 4.0+：驱动通过 META-INF/services 自动加载
// 不再需要 Class.forName()，但 Class.forName 仍然有效
// SPI 机制：ServiceLoader.load(Driver.class)
```

### 6.3 通用对象映射

```java
// 简化版对象映射器
public class ObjectMapper {
    public <S, T> T map(S source, Class<T> targetClass) {
        try {
            T target = targetClass.getDeclaredConstructor().newInstance();
            for (Field sourceField : source.getClass().getDeclaredFields()) {
                Field targetField = findMatchingField(targetClass, sourceField.getName());
                if (targetField != null) {
                    targetField.setAccessible(true);
                    sourceField.setAccessible(true);
                    targetField.set(target, sourceField.get(source));
                }
            }
            return target;
        } catch (Exception e) {
            throw new RuntimeException("Mapping failed", e);
        }
    }
}
```

最后出一道实战题：为什么 Spring 选择用反射而不是 `new` 来创建 Bean？这背后有什么权衡？

答案是：**灵活性 vs 性能**。反射允许 Spring 在运行时决定创建哪个类的实例、支持循环依赖检测、支持 AOP 代理。但代价是性能开销——比直接 `new` 慢 10~100 倍。Spring 选择用反射，是因为 IOC 容器的初始化只在启动时执行一次，性能损失可接受，但获得的灵活性价值巨大。
