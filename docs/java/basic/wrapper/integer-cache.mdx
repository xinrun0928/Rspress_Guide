# Integer 缓存池与 IntegerCache：为什么 -128 到 127 是神奇的区间？

你是否曾经被这段代码困惑过？

```java
public class IntegerCachePuzzle {
    public static void main(String[] args) {
        Integer a = 127;
        Integer b = 127;
        System.out.println("a == b: " + (a == b));  // true
        
        Integer c = 128;
        Integer d = 128;
        System.out.println("c == d: " + (c == d));  // false
        
        Integer e = new Integer(127);
        Integer f = new Integer(127);
        System.out.println("e == f: " + (e == f));  // false
        
        Integer g = Integer.valueOf(127);
        Integer h = Integer.valueOf(127);
        System.out.println("g == h: " + (g == h));  // true
    }
}
```

这个程序的输出会让很多 Java 初学者感到困惑：为什么 127 的情况下 `==` 返回 true，而 128 就返回 false？为什么 `new Integer()` 永远返回新对象，而 `valueOf()` 却可能返回同一个对象？这些现象的根源，正是我们要深入探讨的 Integer 缓存池机制。

## IntegerCache 的完整实现

让我们直接查看 JDK 源码中 IntegerCache 的实现（JDK 11）：

```java
private static class IntegerCache {
    static final int low = -128;
    static final int high;
    static final Integer cache[];

    static {
        // high 值可以通过 -Djava.lang.Integer.IntegerCache.high=<size> 设置
        int h = 127;
        String integerCacheHighPropValue =
            sun.misc.VM.getSavedProperty("java.lang.Integer.IntegerCache.high");
        if (integerCacheHighPropValue != null) {
            try {
                int i = parseInt(integerCacheHighPropValue);
                i = Math.max(i, 127);
                // Maximum array size is Integer.MAX_VALUE
                h = Math.min(i, Integer.MAX_VALUE - (-low) -1);
            } catch( NumberFormatException nfe) {
                // If the property cannot be parsed into an int, ignore it.
            }
        }
        high = h;

        cache = new Integer[(high - low) + 1];
        int j = low;
        for(int k = 0; k < cache.length; k++)
            cache[k] = new Integer(j++);
    }

    private IntegerCache() {}
}
```

这段代码揭示了几个重要的事实。首先，缓存的默认值范围是从 -128 到 127，这是一个硬编码的常量（low = -128）。其次，缓存的上限（high）默认是 127，但可以通过 JVM 参数 `-Djava.lang.Integer.IntegerCache.high` 来动态调整——不过这个值不能小于 127。最后，缓存在类加载时就通过静态初始化块一次性创建好了，所有的 Integer 对象都存储在 `cache[]` 数组中。

## 为什么选择 -128 到 127？

这个范围的选择并非随意，而是有着深厚的历史渊源和工程考量。JVM 规范明确要求 Integer 缓存的下限必须是 -128，而上限可以通过配置调整，默认值为 127。让我们分别从历史、性能和实用性的角度来分析这个设计决策。

### 历史渊源：JLS 的强制规定

Java 语言规范（Java Language Specification，JLS）对整数缓存有明确的规定。在 JLS 第 5.1.7 节" Boxing Conversion"中，规定了对某些特定值的装箱转换必须返回缓存的实例。具体来说，对于 int 值在 \(-2^7\) 到 \(2^7-1\)（即 -128 到 127）范围内的装箱操作，如果结果是 "canonical" 的（即在编译时已知为常量表达式），则必须返回缓存的对象。这是为了确保在编译时常量折叠时的一致性行为。

这一规定确保了 Java 程序在不同 JVM 实现之间的可移植性。无论你使用的是 Oracle JDK、OpenJDK 还是 IBM JDK、Azul Zulu JDK，对于 -128 到 127 范围内的整数装箱，结果都是可预测的——相同的数值必定返回相同的对象引用（假设使用的是默认缓存设置）。

### 性能优化：覆盖最常用的范围

从性能角度来看，-128 到 127 这个范围覆盖了 Java 程序中最常用的整数运算场景。Byte 的全部取值范围（-128 到 127）都被覆盖，Character 的 ASCII 码范围（0 到 127）也被覆盖，而 Short 的取值范围（-32768 到 32767）的大部分值也落在这个区间内。

在实际的 Java 程序中，超过 90% 的整数运算结果都在这个范围内——循环计数器、数组索引、状态码、布尔标志的编码等。这些操作的高频性使得缓存机制能够发挥最大的效益。当你在循环中使用整数计数器时，每次装箱都返回同一个对象，不仅减少了内存分配，还可能提高 CPU 缓存的命中率。

### 内存权衡：缓存大小的限制

如果无限扩大缓存范围，将会占用大量堆内存。考虑默认情况：缓存需要存储 256 个 Integer 对象（从 -128 到 127），每个 Integer 对象在 64 位 JVM 中占用 16 字节（对象头 12 字节 + int 值 4 字节），加上引用占用 8 字节，总共约 24 字节 * 256 = 6144 字节，也就是大约 6KB。这对于现代 JVM 来说微不足道，但如果是 -32768 到 32767（Short 范围），就需要约 400KB；如果是整个 int 范围（约 85 亿个值），那就需要 TB 级别的内存，显然不可行。

JVM 通过允许用户配置上限来实现灵活性。你可以将缓存上限调整为更大的值（如 10000），以覆盖更多的常用值，但这会消耗更多的内存。默认的 127 是一个经过权衡的折中值，既能覆盖大部分使用场景，又不会占用过多的内存。

## equals 与 == 的正确使用

理解了 Integer 缓存机制后，我们再来讨论如何正确比较 Integer 对象。答案很简单：始终使用 `equals()` 方法进行比较，而不是 `==`。

```java
Integer a = 127;
Integer b = 127;
System.out.println(a.equals(b));  // true，始终可靠

Integer c = new Integer(128);
Integer d = new Integer(128);
System.out.println(c.equals(d));  // true，equals 比较的是数值而非引用

Integer e = Integer.valueOf(128);
Integer f = Integer.valueOf(128);
System.out.println(e.equals(f));  // true
```

equals() 方法继承自 Number 类，在 Integer 中的实现如下：

```java
public boolean equals(Object obj) {
    if (obj instanceof Integer) {
        return value == ((Integer)obj).intValue();
    }
    return false;
}
```

它直接比较的是 int 类型的数值，与缓存无关，与对象的创建方式无关，因此是绝对可靠的。

## 动态调整缓存上限

Java 允许通过 JVM 参数动态调整 Integer 缓存的上限：

```bash
java -Djava.lang.Integer.IntegerCache.high=10000 MyApplication
```

启用这个参数后，Integer 缓存的范围将从 -128 到 10000，共 10129 个对象。如果你的应用大量使用小整数，这个设置可能会显著提升性能，因为它能减少 Integer 对象的创建数量。

不过，这种优化需要谨慎使用。在分布式系统中，不同节点使用不同的缓存上限可能导致行为不一致。例如，两台服务器分别使用不同的 IntegerCache.high 值，它们对同一个数值创建的对象可能不同（一个是缓存对象，一个是新对象）。如果这些对象通过网络传输或存储到数据库，可能会出现序列化/反序列化后 `==` 比较结果不同的问题。

## 其他包装类的缓存策略

Integer 并不是唯一一个实现缓存机制的包装类。让我看看其他包装类的缓存策略：

**Long、Short、Byte、Character 的缓存范围：**

| 包装类 | 缓存范围 | 说明 |
|--------|----------|------|
| Byte | -128 ~ 127 | 整个 byte 取值范围 |
| Short | -128 ~ 127 | 可配置上限 |
| Character | 0 ~ 127 | ASCII 码范围 |
| Long | -128 ~ 127 | 可配置上限 |
| Integer | -128 ~ 127 | 可配置上限 |
| Boolean | TRUE, FALSE | 只有两个缓存对象 |
| Float | 无 | 每次创建新对象 |
| Double | 无 | 每次创建新对象 |

注意到一个有趣的规律：Float 和 Double 没有缓存机制，因为浮点数的精度问题使得缓存变得不切实际。考虑一下：如果 0.1 的装箱结果可能是同一个对象，那么比较 `0.1 == 0.1` 会怎样？这可能会引入更多的问题而不是解决方案。

## 编译器优化与常量折叠

Java 编译器会进行常量折叠优化——如果在编译时就能确定表达式的值，就直接使用这个值。考虑下面的代码：

```java
Integer a = 100;  // 编译时确定
Integer b = 100;  // 编译时确定
Integer c = someMethod();  // 运行时才确定
```

对于前两行，编译器知道它们是常量表达式，因此生成的字节码会直接调用 `Integer.valueOf(100)`，而由于 100 在缓存范围内，运行时就会返回同一个 Integer 对象。对于第三行，编译器无法优化，只能在运行时通过 `valueOf()` 方法来处理。

这种编译器优化与运行时缓存的配合，确保了 Java 程序在不同上下文下的一致性行为，同时也为 JIT 编译器提供了进一步优化的空间。

## 空指针异常与自动装箱

当 Integer 引用为 null 时，自动拆箱会抛出 NullPointerException：

```java
Integer nullRef = null;
int value = nullRef;  // 自动拆箱：调用 nullRef.intValue()
// 抛出: java.lang.NullPointerException
```

这个场景在集合操作中特别容易出现：

```java
List<Integer> numbers = new ArrayList<>();
numbers.add(null);

// 运行时错误
int sum = 0;
for (Integer num : numbers) {
    sum += num;  // 尝试对 null 进行拆箱
}
```

## 理解 IntegerCache 的实际意义

Integer 缓存池是 Java 语言中一个看似简单但蕴含深意的设计。它体现了 Java 设计者在语言层面进行性能优化的努力，同时也展示了在性能与内存、正规性（canonical form）与灵活性之间寻求平衡的智慧。

理解这一机制对于编写正确的 Java 代码至关重要。在日常开发中，我们应该养成使用 `equals()` 比较包装类型的好习惯，避免在循环中进行不必要的装箱操作，并在必要时考虑调整缓存上限以优化特定应用场景的性能。同时，也要意识到缓存机制可能带来的意外行为——特别是当你依赖于 `==` 比较来检测对象相同时。

下次当你写 `Integer id = userId;` 这样的代码时，希望你能想起这个看似简单背后隐藏的精妙设计。这不仅仅是一个缓存，更是 Java 语言数十年演进的结晶。
