# new Integer(127) vs Integer.valueOf(127)：两种创建方式的本质差异

在 Java 中，创建 Integer 对象有两种主要方式：使用构造方法 `new Integer(int)` 和使用静态工厂方法 `Integer.valueOf(int)`。很多 Java 程序员可能觉得这两种方式没有区别——它们不都是创建一个 Integer 对象吗？然而，事实远比这复杂。让我们通过一个简单的实验来揭示它们之间的本质差异。

```java
public class IntegerCreationDemo {
    public static void main(String[] args) {
        // 方式一：使用构造方法
        Integer constructor1 = new Integer(127);
        Integer constructor2 = new Integer(127);
        
        // 方式二：使用 valueOf 方法
        Integer valueOf1 = Integer.valueOf(127);
        Integer valueOf2 = Integer.valueOf(127);
        
        System.out.println("constructor1 == constructor2: " + (constructor1 == constructor2));
        System.out.println("valueOf1 == valueOf2: " + (valueOf1 == valueOf2));
        System.out.println("constructor1 == valueOf1: " + (constructor1 == valueOf1));
        
        // 对于 128，超出缓存范围
        Integer valueOf3 = Integer.valueOf(128);
        Integer valueOf4 = Integer.valueOf(128);
        System.out.println("valueOf(128) 1 == valueOf(128) 2: " + (valueOf3 == valueOf4));
        
        Integer constructor3 = new Integer(128);
        Integer constructor4 = new Integer(128);
        System.out.println("new Integer(128) 1 == new Integer(128) 2: " + (constructor3 == constructor4));
    }
}
```

运行结果会清晰地展示这些差异：

```
constructor1 == constructor2: false
valueOf1 == valueOf2: true
constructor1 == valueOf1: false
valueOf(128) 1 == valueOf(128) 2: false
new Integer(128) 1 == new Integer(128) 2: false
```

## 构造方法：每次都是新对象

`new Integer(int)` 构造方法几乎在所有情况下都会创建一个全新的 Integer 对象，无论传入的值是多少。查看 JDK 源码可以验证这一点：

```java
public Integer(int value) {
    this.value = value;
}
```

这个方法简单到令人惊讶——它只是将传入的 int 值存储到实例变量中，没有任何缓存逻辑。这就是为什么 `new Integer(127) == new Integer(127)` 始终返回 false 的原因：每次调用构造方法都会在堆内存中分配一个新对象，它们的引用自然不同。

更值得注意的是，Java 在 JDK 9 就将 Integer 的构造方法标记为 `@Deprecated(since="9")` 了。官方文档明确建议使用 `valueOf()` 方法而不是构造方法来创建 Integer 对象。构造方法虽然功能正常，但完全绕过了缓存机制，在性能和内存使用上都不是最优选择。

## valueOf 方法：智能缓存

`Integer.valueOf(int i)` 方法的实现我们已经见过：

```java
public static Integer valueOf(int i) {
    if (i >= IntegerCache.low && i <= IntegerCache.high)
        return IntegerCache.cache[i + (-IntegerCache.low)];
    return new Integer(i);
}
```

当值在缓存范围内时（默认 -128 到 127），它返回缓存数组中的已有对象；当值超出缓存范围时，才会创建新对象。这种设计带来几个重要的优势。

首先是内存效率。在大量使用小整数的场景中，缓存机制能显著减少 Integer 对象的创建数量。假设你的程序中有 10000 个 Integer 变量都存储值 42，如果没有缓存，就需要创建 10000 个对象；而有了缓存，就只需要创建 1 个对象（42 在缓存范围内），其他 9999 次都复用同一个引用。

其次是身份比较的可靠性。在 -128 到 127 范围内，两个相同值的 Integer 对象实际上是同一个引用，这意味着 `==` 比较是可预测的（虽然仍然推荐使用 equals）。但超出这个范围，即使是相同的值也无法通过 `==` 比较为 true。

## 自动装箱的行为差异

理解了这两种创建方式的差异后，我们就能理解 Java 中一个容易混淆的行为：自动装箱到底使用哪一种方式？答案是自动装箱使用 `valueOf()` 方法。

```java
Integer a = 127;        // 编译后等价于 Integer.valueOf(127)
Integer b = new Integer(127);  // 显式调用构造方法
```

当你写 `Integer a = 127;` 时，编译器会生成 `Integer a = Integer.valueOf(127);` 的字节码。由于 127 在缓存范围内，a 实际上指向的是缓存对象。而 b 则明确地通过构造方法创建了一个新对象。

这解释了为什么下面的代码有时会让人困惑：

```java
public class BoxingSurprise {
    public static void main(String[] args) {
        Integer a = 127;
        Integer b = new Integer(127);
        
        System.out.println("a == 127: " + (a == 127));    // true：拆箱比较
        System.out.println("b == 127: " + (b == 127));    // true：拆箱比较
        System.out.println("a == b: " + (a == b));        // false：引用比较
    }
}
```

当 Integer 与 int 进行 `==` 比较时，实际上是将 Integer 拆箱为 int 后再进行比较。因此，无论 a 是来自缓存还是新创建的对象，它与 int 的比较都只涉及数值本身。

## 实际开发中的建议

基于以上分析，在实际开发中有几点建议值得关注。

第一，优先使用自动装箱或 valueOf()，避免使用构造方法。由于 `new Integer(int)` 已被标记为废弃，我们应该使用更简洁的自动装箱语法（`Integer a = 127;`）或明确的 `Integer.valueOf(127)` 方法。

第二，始终使用 equals() 比较 Integer 对象。由于缓存行为与值的大小相关，`==` 比较的结果在不同值范围内可能不同：

```java
Integer a = 127;
Integer b = 127;
Integer c = 128;
Integer d = 128;

System.out.println(a.equals(b));  // true
System.out.println(c.equals(d));  // true
System.out.println(a == b);       // true（巧合，在缓存范围内）
System.out.println(c == d);       // false（缓存范围外）
```

第三，在高性能场景下注意装箱成本。虽然自动装箱很方便，但在循环中频繁装箱可能造成性能问题。如果你在编写计算密集型代码，应该优先使用基本类型。

## 反序列化中的对象创建

还有一个容易忽略的场景是对象的反序列化。当使用 ObjectInputStream 反序列化一个 Integer 对象时，它会使用反射调用 Integer 的构造方法或特殊方法来创建对象，这可能绕过了 valueOf() 的缓存逻辑：

```java
ByteArrayOutputStream baos = new ByteArrayOutputStream();
ObjectOutputStream oos = new ObjectOutputStream(baos);
oos.writeObject(Integer.valueOf(127));

ByteArrayInputStream bais = new ByteArrayInputStream(baos.toByteArray());
ObjectInputStream ois = new ObjectInputStream(bais);
Integer deserialized = (Integer) ois.readObject();

Integer cached = Integer.valueOf(127);
System.out.println("deserialized == cached: " + (deserialized == cached));  // 可能是 false
```

这意味着序列化前后的同一个值，在反序列化后可能不再指向同一个对象实例。对于大多数应用来说这不是问题，但如果你依赖于 Integer 对象的身份语义（例如用作 Map 的键且使用 identity semantics），就需要特别注意了。

## 总结与最佳实践

理解 `new Integer()` 和 `Integer.valueOf()` 的区别，本质上是理解 Java 对象创建与缓存机制的关系。`new Integer()` 每次创建新对象，绕过缓存；`Integer.valueOf()` 智能地使用缓存，在 -128 到 127 范围内复用对象。这种设计在保证功能正确性的同时，优化了内存使用和性能。

在现代 Java 开发中，构造方法已被废弃，我们应该：
- 使用自动装箱语法或 valueOf() 方法创建 Integer 对象
- 始终使用 equals() 进行相等性比较
- 在性能敏感的代码中注意装箱操作的成本
- 在涉及对象身份比较的场景中特别小心
