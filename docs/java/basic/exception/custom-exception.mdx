# 自定义异常：设计与实践

Java 内置的异常类虽然功能完善，但在实际项目中，通常需要创建自定义异常来更好地表达业务逻辑中的错误情况。想象一个银行转账系统——如果转账失败，抛出 `NullPointerException` 或 `IllegalArgumentException` 能准确表达问题吗？显然不能。我们需要一个专门的 `InsufficientBalanceException` 来告诉调用者余额不足，需要一个 `TransferLimitExceededException` 来表示超过了单笔转账限额。

自定义异常不仅仅是给异常起一个好听的名字，更重要的是它应该携带足够的上下文信息，让调试和问题定位变得更加容易。

## 异常层次结构的设计

在设计自定义异常之前，首先需要考虑异常类的层次结构。一个好的异常层次结构应该与业务领域模型相对应。

```java
// 基础业务异常
public class BusinessException extends RuntimeException {
    private String errorCode;
    private Map<String, Object> details;
    
    public BusinessException(String message) {
        super(message);
    }
    
    public BusinessException(String message, Throwable cause) {
        super(message, cause);
    }
    
    public BusinessException(String errorCode, String message) {
        super(message);
        this.errorCode = errorCode;
    }
    
    // getter 和 setter
}
```

然后我们可以从这个基础类派生出更具体的异常：

```java
// 账户相关异常
public class AccountException extends BusinessException {
    private String accountId;
    
    public AccountException(String message, String accountId) {
        super(message);
        this.accountId = accountId;
    }
}

// 余额不足
public class InsufficientBalanceException extends AccountException {
    private BigDecimal currentBalance;
    private BigDecimal requiredAmount;
    
    public InsufficientBalanceException(String accountId, 
            BigDecimal currentBalance, BigDecimal requiredAmount) {
        super("账户余额不足", accountId);
        this.currentBalance = currentBalance;
        this.requiredAmount = requiredAmount;
    }
    
    public BigDecimal getCurrentBalance() {
        return currentBalance;
    }
    
    public BigDecimal getRequiredAmount() {
        return requiredAmount;
    }
}
```

这样的设计有几个好处：首先，异常类按照业务领域组织，层次清晰；其次，每个异常都携带了该领域相关的属性（如 accountId、currentBalance）；最后，通过继承关系，调用者可以选择捕获具体的异常或更通用的父类异常。

## 异常消息的最佳实践

异常消息是异常最重要的属性之一。一个好的异常消息应该能够回答以下问题：发生了什么错误？在哪里发生的？为什么出错？用户应该如何处理？

```java
// 不好的异常消息
throw new Exception("错误");

// 更好的异常消息
throw new BusinessException("SYS001", 
    String.format("用户[%s]尝试转账[%s]元，但账户余额为[%s]元",
        userId, transferAmount, balance));
```

在设计异常消息时，应该包含足够的关键信息，使用清晰的语言，并且避免暴露敏感信息（如完整的密码、日志中的私密数据）。

## 异常链的建立

当自定义异常是由其他异常引起时，应该建立异常链：

```java
public class AccountService {
    
    public void transfer(String fromAccountId, String toAccountId, BigDecimal amount) {
        try {
            Account from = accountRepository.findById(fromAccountId);
            Account to = accountRepository.findById(toAccountId);
            
            if (from.getBalance().compareTo(amount) < 0) {
                throw new InsufficientBalanceException(
                    fromAccountId, from.getBalance(), amount);
            }
            
            // 执行转账
            from.withdraw(amount);
            to.deposit(amount);
            accountRepository.save(from);
            accountRepository.save(to);
            
        } catch (SQLException e) {
            // 将数据库异常转换为业务异常
            throw new DataAccessException(
                "数据库操作失败，无法完成转账", e);
        }
    }
}
```

通过异常链，最终的日志或错误报告中会包含完整的调用栈，从最上层的业务异常一直追溯到最底层的原始异常（如 SQLException、IOException），帮助开发者快速定位问题根源。

## 异常与日志记录的平衡

在捕获和重新抛出异常时，需要注意不要重复记录日志：

```java
// 不好的做法：日志记录了两次
try {
    doSomething();
} catch (IOException e) {
    logger.error("操作失败", e);
    throw new BusinessException("操作失败", e);  // 这会再次打印堆栈
}

// 好的做法：只在合适的位置记录日志
try {
    doSomething();
} catch (IOException e) {
    throw new BusinessException("操作失败", e);  // 只抛异常，不记录日志
}

// 或者
try {
    doSomething();
} catch (IOException e) {
    logger.error("操作失败", e);
    throw new BusinessException("操作失败", e);  // 调用者不会再次记录
}
```

一个常见的模式是在应用的最上层（如 Controller 或 Filter）统一记录日志，而在中间层只负责抛出异常。这样可以避免重复的日志输出，也可以确保日志记录的一致性。

## 全局异常处理器

在 Spring MVC 或类似的框架中，可以使用全局异常处理器来统一处理异常：

```java
@RestControllerAdvice
public class GlobalExceptionHandler {
    
    @ExceptionHandler(InsufficientBalanceException.class)
    public ResponseEntity<ErrorResponse> handleInsufficientBalance(
            InsufficientBalanceException e) {
        ErrorResponse error = new ErrorResponse(
            "INSUFFICIENT_BALANCE",
            e.getMessage(),
            Map.of(
                "accountId", e.getAccountId(),
                "currentBalance", e.getCurrentBalance(),
                "requiredAmount", e.getRequiredAmount()
            )
        );
        return ResponseEntity.status(HttpStatus.BAD_REQUEST).body(error);
    }
    
    @ExceptionHandler(BusinessException.class)
    public ResponseEntity<ErrorResponse> handleBusinessException(
            BusinessException e) {
        ErrorResponse error = new ErrorResponse(
            e.getErrorCode(),
            e.getMessage()
        );
        return ResponseEntity.status(HttpStatus.BAD_REQUEST).body(error);
    }
    
    @ExceptionHandler(Exception.class)
    public ResponseEntity<ErrorResponse> handleGenericException(Exception e) {
        logger.error("未预期的错误", e);
        ErrorResponse error = new ErrorResponse(
            "INTERNAL_ERROR",
            "系统内部错误，请稍后重试"
        );
        return ResponseEntity.status(HttpStatus.INTERNAL_SERVER_ERROR).body(error);
    }
}
```

通过全局异常处理器，可以将异常的 HTTP 响应格式统一化，同时根据异常类型返回适当的 HTTP 状态码。

## 异常类的性能考虑

创建异常对象是有成本的——它需要分配内存并填充堆栈跟踪。在高频代码路径中频繁创建异常可能导致性能问题：

```java
// 性能较差：频繁创建异常
public void process(String input) {
    if (input == null || input.isEmpty()) {
        throw new IllegalArgumentException("输入不能为空");  // 每次都创建新异常
    }
    // ...
}

// 更好的做法：使用预定义的异常实例
public class ProcessException {
    public static final IllegalArgumentException NULL_INPUT = 
        new IllegalArgumentException("输入不能为空");
    public static final IllegalArgumentException EMPTY_INPUT = 
        new IllegalArgumentException("输入不能为空字符串");
}

public void process(String input) {
    if (input == null) {
        throw ProcessException.NULL_INPUT;
    }
    if (input.isEmpty()) {
        throw ProcessException.EMPTY_INPUT;
    }
    // ...
}
```

但这种优化通常只对高频代码路径有意义。对于大多数业务代码来说，可读性和可维护性比微优化更重要。

## 总结

自定义异常是 Java 异常处理中的重要组成部分。通过合理设计异常层次结构、包含有意义的消息和上下文信息、建立清晰的异常链，我们可以让错误处理变得更加清晰和高效。

设计良好的异常系统应该能够回答三个问题：发生了什么（异常类型）、在哪里发生的（堆栈跟踪）、如何处理（异常消息和文档）。当你的异常能够清晰地传达这些信息时，调试和问题定位就会变得更加高效。
