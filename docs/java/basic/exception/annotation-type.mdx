# 注解的分类与保留策略

你有没有想过，为什么有些注解能写在源代码里，有些能编译进 class 文件，但运行时却找不到它们的身影？

同样的 `@Override`，为什么 IDEA 能识别它、`javac` 能检查它、反射却无法读取？这背后，是 Java 设计者精心设计的一套「生命周期分层」机制。

## 三个注解的生命周期

Java 中的注解按照「保留策略」分为三类，决定了它们能存活到哪个阶段：

| 保留策略 | 枚举值 | 生命周期 | 典型场景 |
|---|---|---|---|
| SOURCE | `RetentionPolicy.SOURCE` | 仅存在于源码阶段 | `@Override`、`@SuppressWarnings` |
| CLASS | `RetentionPolicy.CLASS` | 编译进 class 文件，运行时不可见 | Lombok 的大部分注解 |
| RUNTIME | `RetentionPolicy.RUNTIME` | 贯穿全程，可用反射读取 | Spring 的 `@Component`、`@Autowired` |

### SOURCE：源码级别的「临时工」

`SOURCE` 级别的注解是最「短命」的，它们只在源码阶段存在，编译完成后就被丢弃。

```java
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.SOURCE)
public @interface Override {
    // 没有任何属性
}
```

`@Override` 是最典型的 SOURCE 级别注解。它的作用是告诉编译器：「我要覆盖父类的方法，你帮我检查一下」。

编译器在编译阶段看到 `@Override`，会验证这个方法是否真的在父类中存在。如果存在，编译完成后，`@Override` 注解就完成了使命，不会在 class 文件中留下任何痕迹。

这也解释了为什么：**编译后的 class 文件中，`@Override` 注解消失了。**

### CLASS：编译产物中的「沉睡者」

`CLASS` 级别的注解会保留到 class 文件中，但不会被 JVM 加载到内存。

这是 Java 编译器的「中间层」，专门给那些需要在编译时处理注解的工具使用。

Lombok 是 CLASS 级别注解的最佳实践：

```java
@Target(ElementType.TYPE)
@Retention(RetentionPolicy.CLASS)
public @interface Data {
    // Lombok 在编译时读取此注解
    // 生成 getter/setter/equals/toString 等方法
}
```

javac 编译器在处理 Lombok 注解时，会在编译阶段识别这些注解，然后生成对应的方法。生成完成后，这些注解本身不会进入运行时的内存空间——它们只是编译时的「指令」。

### RUNTIME：贯穿始终的「常驻者」

`RUNTIME` 级别是注解的「完整生命周期」，它们从源码一路走到 JVM 运行阶段。

这是 Spring、Hibernate 等框架能够工作的基础——**只有 RUNTIME 级别的注解，才能通过反射被读取**。

```java
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface Autowired {
    boolean required() default true;
}
```

Spring 在启动时，会扫描所有被 `@Autowired` 标记的字段或方法，然后通过反射读取这个注解，将对应的 Bean 注入进来。

## 自定义注解的完整模板

理解了三种保留策略后，来看看自定义注解的标准写法：

```java
package com.example.annotation;

import java.lang.annotation.*;

@Target({
    ElementType.TYPE,          // 可以用在类上
    ElementType.METHOD,         // 可以用在方法上
    ElementType.FIELD,          // 可以用在字段上
    ElementType.CONSTRUCTOR     // 可以用在构造器上
})
@Retention(RetentionPolicy.RUNTIME)  // 运行时可见，可反射读取
@Documented                          // 加入 Javadoc
@Inherited                           // 子类继承父类的此注解
public @interface MyAnnotation {
    String value() default "";       // 注解属性

    String name() default "default";

    int priority() default 1;
}
```

使用这个注解：

```java
@MyAnnotation(value = "test", name = "MyService", priority = 10)
public class MyService {
    @MyAnnotation(value = "method annotation")
    public void doSomething() {
        // ...
    }
}
```

## 反射读取注解

只有 `RUNTIME` 级别的注解才能被反射读取：

```java
public class AnnotationReader {
    public static void main(String[] args) throws Exception {
        Class<?> clazz = MyService.class;

        // 读取类上的注解
        if (clazz.isAnnotationPresent(MyAnnotation.class)) {
            MyAnnotation annotation = clazz.getAnnotation(MyAnnotation.class);
            System.out.println("Class annotation: " + annotation.value());
        }

        // 读取方法上的注解
        for (java.lang.reflect.Method method : clazz.getDeclaredMethods()) {
            if (method.isAnnotationPresent(MyAnnotation.class)) {
                MyAnnotation annotation = method.getAnnotation(MyAnnotation.class);
                System.out.println("Method " + method.getName() + ": " + annotation.value());
            }
        }
    }
}
```

输出：

```text
Class annotation: test
Method doSomething: method annotation
```

## 注解的类型分类

除了保留策略，注解还可以按「来源」和「是否包含属性」分类：

### 标准注解 vs 元注解

**标准注解**是 Java 提供给我们使用的，如 `@Override`、`@Deprecated`、`@FunctionalInterface`。

**元注解**是专门用来注解其他注解的注解：

| 元注解 | 作用 |
|---|---|
| `@Target` | 限制注解能用在哪些元素上 |
| `@Retention` | 指定注解的保留策略 |
| `@Documented` | 注解信息会出现在 Javadoc 中 |
| `@Inherited` | 子类会继承父类的注解 |

### 标记注解 vs 数据注解

**标记注解**：不包含任何属性，只起到「标记」作用。

```java
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.SOURCE)
public @interface Override {
}
```

**数据注解**：包含属性，可以携带更多信息。

```java
@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
public @interface Table {
    String name() default "";
    String schema() default "public";
}
```

## 为什么 `@Override` 用 SOURCE 而不是 RUNTIME？

一个常见的疑问是：`@Override` 既然可以标记方法，为什么不设计成 RUNTIME 级别，让开发者通过反射来检查一个方法是否覆盖了父类？

答案是：**没有意义**。

在运行时，方法覆盖是语言层面的事实。如果你在代码里写了一个与父类方法签名相同的方法，它就是覆盖了父类方法，不需要额外标记。`@Override` 的价值在于**编译期的错误检查**，而不是运行时的运行时行为。

让 `@Override` 成为 RUNTIME 级别，只会让 class 文件更大、反射遍历更慢，却没有任何实际收益。

## 实战场景：自定义校验注解

理解了注解的分类后，来实现一个自定义校验注解：

```java
@Target(ElementType.FIELD)
@Retention(RetentionPolicy.RUNTIME)
@Constraint(validatedBy = NotBlankValidator.class)
public @interface NotBlank {
    String message() default "字段不能为空";
    Class<?>[] groups() default {};
    Class<? extends Payload>[] payload() default {};
}
```

校验器实现：

```java
public class NotBlankValidator implements ConstraintValidator<NotBlank, String> {
    @Override
    public void initialize(NotBlank constraintAnnotation) {
        // 初始化逻辑
    }

    @Override
    public boolean isValid(String value, ConstraintValidatorContext context) {
        return value != null && !value.trim().isEmpty();
    }
}
```

使用：

```java
public class User {
    @NotBlank(message = "用户名不能为空")
    private String username;

    @NotBlank(message = "邮箱不能为空")
    private String email;
}
```

## 总结

注解的保留策略决定了它们在哪个阶段「消失」，理解这一点是掌握 Spring、Hibernate、Lombok 等框架的基础：

- **SOURCE**：给编译器看，编译后消失
- **CLASS**：给编译工具看，运行前消失
- **RUNTIME**：给反射看，全程可见

下次当你用 `@Autowired` 注入 Bean、用 `@Data` 生成代码、用 `@Override` 检查覆盖时，希望你能想起这篇文章——注解虽小，设计却精妙。
