## 二、核心原理：注解到底是什么？

### 2.1 注解的本质

**注解（Annotation）** 本质上是一个 **接口**，它继承自 `java.lang.annotation.Annotation`。

```java title="java.lang.annotation.Annotation"
public interface Annotation {
    boolean equals(Object obj);
    String toString();
    int hashCode();
    Class<? extends Annotation> annotationType();
}
```

当你定义一个注解时：

```java
public @interface MyAnnotation {
    String value() default "";
}
```

编译器会自动生成一个实现 `Annotation` 接口的 **代理类**。这个代理类由 JVM 在运行时动态生成。

### 2.2 元注解：注解的注解

Java 定义了 5 个元注解（meta-annotation），用于修饰其他注解：

| 元注解 | 作用 | 关键参数 |
|--------|------|----------|
| `@Target` | 指定注解可以作用在哪些元素上 | `ElementType[]` |
| `@Retention` | 指定注解的保留策略 | `RetentionPolicy` |
| `@Documented` | 标记是否出现在 Javadoc 中 | 无 |
| `@Inherited` | 标记是否允许子类继承 | 无 |
| `@Repeatable` | 标记是否可重复使用 | `Class<? extends Annotation>` |

## 三、源码级解析：@Retention 三种保留策略

`@Retention` 是最常被问到的元注解，它决定了注解的**生命周期**。

```java title="java.lang.annotation.Retention"
@Documented
@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.ANNOTATION_TYPE)
public @interface Retention {
    /**
     * Returns the retention policy.
     */
    RetentionPolicy value();
}
```

`RetentionPolicy` 是一个枚举，有三个取值：

```java title="java.lang.annotation.RetentionPolicy"
public enum RetentionPolicy {
    SOURCE,    // 源码阶段，编译后丢弃
    CLASS,     // 编译时阶段，存储在 .class 文件中，运行时丢弃
    RUNTIME    // 运行时阶段，可通过反射读取
}
```

### 3.1 三种保留策略对比

| 策略 | 源码阶段 | 编译阶段 | 类加载阶段 | 运行阶段 | 典型用途 |
|------|----------|----------|------------|----------|----------|
| `SOURCE` | ✅ 可见 | ❌ 丢弃 | ❌ 丢弃 | ❌ 丢弃 | `@Override`、`@SuppressWarnings` |
| `CLASS` | ✅ 可见 | ✅ 保留在 .class | ❌ 丢弃 | ❌ 丢弃 | 编译时检查（如 Lombok 某些功能） |
| `RUNTIME` | ✅ 可见 | ✅ 保留在 .class | ✅ 保留在内存 | ✅ 可反射读取 | Spring DI、JUnit、反射框架 |

### 3.2 源码解析：@Override 为什么是 SOURCE 级别？

```java title="java.lang.Override"
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.SOURCE)
public @interface Override {
}
```

`@Override` 是 **SOURCE** 级别，这意味着：

1. **编译时生效**：编译器在编译阶段检查方法签名是否与父类/接口匹配
2. **编译后消失**：编译后的 .class 文件不包含 `@Override` 信息
3. **运行不可见**：JVM 运行时无法通过反射获取此注解

:::tip
**为什么 @Override 要用 SOURCE 级别？**

因为它的目的就是在**编译时强制校验**，帮助开发者发现"重写失败"的 bug（比如父类方法签名变了，但你写错了方法名）。运行时根本不需要这个信息，保留它只会浪费内存。
:::

### 3.3 @Deprecated 的 RUNTIME 保留策略

```java title="java.lang.Deprecated"
@Documented
@Retention(RetentionPolicy.RUNTIME)
@Target(value={CONSTRUCTOR, FIELD, LOCAL_VARIABLE, METHOD, PACKAGE, PARAMETER, TYPE})
public @interface Deprecated {
}
```

`@Deprecated` 使用 **RUNTIME** 级别，因为：

- **编译时**：编译器会生成警告
- **运行时**：JVM 可以通过反射检测到 `@Deprecated` 标记，对调用方给出警告

## 四、@Target：注解能作用在哪里？

`@Target` 使用 `ElementType` 枚举定义作用目标：

```java title="java.lang.annotation.ElementType"
public enum ElementType {
    TYPE,               // 类、接口、枚举
    FIELD,              // 字段（包括枚举常量）
    METHOD,             // 方法
    PARAMETER,          // 参数
    CONSTRUCTOR,        // 构造器
    LOCAL_VARIABLE,     // 局部变量
    ANNOTATION_TYPE,    // 注解类型
    PACKAGE,            // 包
    TYPE_PARAMETER,     // 类型参数（JDK 8+）
    TYPE_USE,           // 类型使用（JDK 8+）
    MODULE,             // 模块（JDK 9+）
    RECORD_COMPONENT    // 记录组件（JDK 14+）
}
```

### 常见注解的作用目标组合

```java
// @Override 只能作用于方法
@Target(ElementType.METHOD)

// @Deprecated 可以作用于多种元素
@Target({TYPE, FIELD, METHOD, PARAMETER, CONSTRUCTOR, LOCAL_VARIABLE, PACKAGE, TYPE})

// @Retention 是元注解，只能作用于注解
@Target(ElementType.ANNOTATION_TYPE)
```

## 五、运行时注解处理：反射读取

### 5.1 反射获取注解的核心 API

只有 `@Retention(RetentionPolicy.RUNTIME)` 的注解才能被反射获取：

```java title="AnnocationReflection.java"
public class AnnotationReflection {

    // 检查类上是否有注解
    public static boolean hasClassAnnotation(Class<?> clazz, Class<? extends Annotation> annotationClass) {
        return clazz.isAnnotationPresent(annotationClass);
    }

    // 获取类上的注解
    public static <A extends Annotation> A getClassAnnotation(Class<?> clazz, Class<A> annotationClass) {
        return clazz.getAnnotation(annotationClass);
    }

    // 获取方法上的注解
    public static <A extends Annotation> A getMethodAnnotation(
            Class<?> clazz, String methodName, Class<A> annotationClass) {
        try {
            Method method = clazz.getMethod(methodName);
            return method.getAnnotation(annotationClass);
        } catch (NoSuchMethodException e) {
            return null;
        }
    }

    // 获取所有注解（包括重复注解）
    public static Annotation[] getAllMethodAnnotations(
            Class<?> clazz, String methodName) {
        try {
            Method method = clazz.getMethod(methodName);
            return method.getAnnotations();
        } catch (NoSuchMethodException e) {
            return new Annotation[0];
        }
    }
}
```

### 5.2 面试追问：getAnnotation vs getAnnotations

```java
public class GetAnnotationDemo {

    @MyAnnotation("test")
    @MyRepeatableAnnotation("repeat1")
    @MyRepeatableAnnotation("repeat2")
    public void demo() {}
}
```

| 方法 | 返回值 | 说明 |
|------|--------|------|
| `getAnnotation(Class)` | `T` 或 `null` | 获取指定类型的注解（重复注解只返回第一个） |
| `getAnnotations()` | `Annotation[]` | 获取所有注解（包括继承的） |
| `getDeclaredAnnotation(Class)` | `T` 或 `null` | 获取直接存在的注解（不包括继承） |
| `getDeclaredAnnotations()` | `Annotation[]` | 获取直接存在的所有注解 |

### 5.3 @Inherited：子类继承父类注解

```java title="java.lang.annotation.Inherited"
@Documented
@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.ANNOTATION_TYPE)
public @interface Inherited {
}
```

:::warning
**⚠️ @Inherited 的继承机制有严格条件限制！**

1. 只有**类**的注解才会被子类继承（接口、方法、字段的注解不会）
2. 子类通过 `getAnnotations()` 继承父类的 `@Inherited` 注解
3. 通过 `getDeclaredAnnotations()` 获取的注解**不包括**继承的
4. `@Inherited` 是**标记注解**，没有属性
:::

```java
@Inherited
@Retention(RetentionPolicy.RUNTIME)
@interface MyInheritedAnnotation {
    String value();
}

@MyInheritedAnnotation("parent")
class Parent {}

class Child extends Parent {}  // Child 继承了 @MyInheritedAnnotation

// 验证
public class InheritedTest {
    public static void main(String[] args) {
        // Child 类自动拥有 @MyInheritedAnnotation
        MyInheritedAnnotation annotation = Child.class.getAnnotation(MyInheritedAnnotation.class);
        System.out.println(annotation.value());  // 输出: parent
    }
}
```

## 六、@Repeatable：重复使用注解

JDK 8 引入了 `@Repeatable`，允许同一个元素使用多次同一种注解。

```java title="java.lang.annotation.Repeatable"
@Documented
@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.ANNOTATION_TYPE)
public @interface Repeatable {
    /**
     * 指向容纳注解的容器注解类型
     */
    Class<? extends Annotation> value();
}
```

### 使用示例

```java title=" repeatable/UserLog.java"
// 第一步：定义容器注解
@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.METHOD)
@interface UserLogs {
    UserLog[] value();
}

// 第二步：定义可重复的注解，并指定容器
@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.METHOD)
@Repeatable(UserLogs.class)
@interface UserLog {
    String action();
    String target();
}

// 第三步：直接使用多次
class UserService {

    @UserLog(action = "登录", target = "用户")
    @UserLog(action = "下单", target = "订单")
    public void createOrder() {
        // 业务逻辑
    }
}
```

### 获取重复注解

```java
public class RepeatableTest {
    public static void main(String[] args) throws Exception {
        Method method = UserService.class.getMethod("createOrder");

        // 获取所有 @UserLog 注解
        UserLog[] logs = method.getAnnotationsByType(UserLog.class);
        for (UserLog log : logs) {
            System.out.println(log.action() + " -> " + log.target());
        }

        // 输出:
        // 登录 -> 用户
        // 下单 -> 订单
    }
}
```

## 七、注解 vs XML 配置：深度对比

| 维度 | 注解配置 | XML 配置 |
|------|----------|----------|
| **内聚性** | 配置与代码同处一处 | 配置与代码分离 |
| **可读性** | 代码即文档，直观 | 集中管理，适合复杂配置 |
| **维护性** | 修改需要重新编译 | 可独立修改，无需重启 |
| **复用性** | 复用困难，继承受限 | 可复用，import 引用 |
| **适用场景** | 简单绑定、框架内置 | 跨模块、业务配置 |
| **IDE 支持** | 编译时即检查 | 需要额外验证工具 |
| **灵活性** | 固定于元素 | 可配置任意元素 |

:::warning
**⚠️ 为什么 Spring 不完全用注解替代 XML？**

1. **循环依赖**：注解配置在类内部，无法解决循环依赖问题
2. **外部化配置**：有些配置需要从数据库/Nacos 等外部源读取
3. **运行时变更**：XML 可以实现运行时的动态 Bean 注册
4. **历史包袱**：老项目迁移成本巨大

但趋势是：**Spring Boot 之后，注解+Java Config 已成为主流**。
:::

## 八、经典面试追问

### Q1：注解能被继承吗？

**普通注解不能被继承**，除非同时满足以下条件：

1. 父类的注解标记了 `@Inherited`
2. 父类的注解 `Retention` 是 `RUNTIME`
3. 通过 `getAnnotations()` 获取（不是 `getDeclaredAnnotations()`）

```java
// 普通注解不会被继承
@MyNormalAnnotation  // 子类看不到这个
class Parent {}
```

### Q2：注解的默认属性值是什么？

```java
@interface MyAnnotation {
    String value() default "default";  // 有默认值，可省略
    int count() default 0;              // 有默认值，可省略
    String name();                     // 没有默认值，必须指定
}

// 使用
@MyAnnotation(name = "test")  // value 和 count 使用默认值
class Demo {}
```

### Q3：为什么注解的属性不能是包装类型？

```java
// 错误示例
@interface WrongAnnotation {
    Integer value();  // 编译错误！
}

// 正确示例
@interface CorrectAnnotation {
    int value();      // 基本类型
    String name();    // String
    Class<?> type();  // Class
    Annotation[] nested();  // 注解数组
}
```

:::tip
**💡 为什么注解属性有这些限制？**

因为注解在编译后存储在 **Constant_pool** 中，常量池只支持基本类型、String、Class、枚举、注解类型本身。包装类型（如 `Integer`）不是常量池的基本类型，所以不被支持。
:::

## 九、总结：高频考点速记

| 考点 | 答案要点 |
|------|----------|
| `@Override` 的 Retention | `SOURCE`，编译时检查，运行后丢弃 |
| `@Deprecated` 的 Retention | `RUNTIME`，运行时可通过 `getAnnotation()` 获取 |
| `@Inherited` 限制 | 只对类生效，不对接口/方法生效 |
| 可重复注解 | 需要 `@Repeatable` + 容器注解 |
| 反射获取注解条件 | `Retention` 必须是 `RUNTIME` |
| 注解继承条件 | `@Inherited` + `RUNTIME` + 类 + `getAnnotations()` |

## 十、应用场景：框架中的注解

### 10.1 JUnit 4/5 注解体系

```java
public class JunitDemo {

    @BeforeClass    // 类加载前执行一次
    public static void setupClass() {}

    @Before         // 每个测试方法前执行
    public void setup() {}

    @Test          // 标记测试方法
    public void testMethod() {}

    @After         // 每个测试方法后执行
    public void teardown() {}

    @AfterClass    // 类卸载前执行一次
    public static void teardownClass() {}
}
```

### 10.2 Spring 核心注解

| 注解 | 作用 | 保留策略 |
|------|------|----------|
| `@Component` | 定义 Spring Bean | RUNTIME |
| `@Autowired` | 依赖注入 | RUNTIME |
| `@RequestMapping` | 请求映射 | RUNTIME |
| `@Transactional` | 事务管理 | RUNTIME |

### 10.3 Lombok 注解

Lombok 的部分注解是 **SOURCE** 级别，通过编译时注解处理器生成代码：

| 注解 | 生成代码 | 保留策略 |
|------|----------|----------|
| `@Getter` | `getXxx()` 方法 | SOURCE |
| `@Setter` | `setXxx()` 方法 | SOURCE |
| `@Data` | getter/setter/equals/hashCode/toString | SOURCE |
| `@Builder` | Builder 模式 | SOURCE |
| `@Slf4j` | `Logger` 字段 | SOURCE |
