阿里P6面试间里，面试官看着候选人小钱，抛出了一道"简单"的问题：

"Java 里 checked exception 和 unchecked exception 的区别是什么？"

小钱："checked 是编译器检查的，需要强制处理；unchecked 不需要强制处理。"

面试官："那为什么 `Thread.sleep()` 抛的是 checked exception，而 `String.charAt()` 越界抛的是 unchecked exception？它们不都是在运行时才发生的吗？"

小钱停顿了几秒："因为...sleep 是 IO 操作？"

面试官追问："那 `Class.forName()` 也是 IO 吗？它为什么也是 checked exception？"

小钱答不上来了。

面试官继续："你平时写代码，更倾向于用 checked exception 还是 unchecked exception？为什么？"

小钱说："我一般用 unchecked...因为 checked 太麻烦了，每次都要 try-catch。"

面试官在笔记本上写了几行字。

【面试官心理】
这道题我用来测试候选人是否理解"设计意图"而非"表面规则"。能答出"编译器检查"只是第一层，能答出"可预期 vs 程序员错误"是第二层，能讨论 Checked Exception 的设计争议是第三层。大多数候选人卡在第一层，只有 P6+ 才能答到第三层。

## 一、本质区别 🔴

### 1.1 问题拆解

**第一层：怎么用？**
面试官问："Checked 和 Unchecked 异常的本质区别是什么？"

**第二层：底层实现**
追问："编译器是怎么处理这两种异常的？为什么会产生不同的编译行为？"

**第三层：边界缺陷**
追问："Java 为什么设计 Checked Exception？这是一个好的设计吗？"

**第四层：选型 trade-off**
追问："你在项目里怎么选？自定义异常用哪种？为什么？"

### 1.2 ❌ 错误示范

**候选人原话**："Checked Exception 是在编译时检查的，Unchecked Exception 是在运行时才发生的。"

**问题诊断**：
- 这个说法经不起推敲——IOException 难道不是运行时才发生的？文件存在不存在只有运行时才知道
- 混淆了"编译器检查"（表现）和"设计意图"（本质）
- 没有理解"可预期"和"不可预期"的区别

**面试官内心 OS**："知道'编译器检查'只是知道规则，不知道'为什么'才是理解设计意图。这个候选人可能只是背过，没思考过。"

### 1.3 标准回答：设计意图决定类型

Checked Exception 和 Unchecked Exception 的唯一区别是**设计意图**：

| 类型 | 设计意图 | 编译器行为 | 典型代表 |
| --- | --- | --- | --- |
| Checked Exception | 调用方的责任——这个操作可能失败，调用方必须处理 | 必须 try-catch 或 throws | IOException, SQLException |
| Unchecked Exception | 程序员的错误——这是 bug，不是业务流程的一部分 | 不强制处理 | NullPointerException, IndexOutOfBoundsException |

```java
// Checked Exception 的典型场景：外部依赖可能失败
// 文件可能不存在，网络可能断开，数据库可能宕机
// 这些都是业务的一部分，调用方必须考虑
public void loadConfig(String path) throws IOException {
    FileReader reader = new FileReader(path);  // 编译器强制要求处理
    // 为什么不强制？因为文件不存在是正常的业务流程
}

// Unchecked Exception 的典型场景：程序员错误
// null 应该被检查，参数应该被验证，类型应该正确
// 这些是程序员的责任，不应该由调用方处理
public void process(String name) {
    if (name == null) {
        throw new NullPointerException("name 不能为 null");  // 不需要声明
    }
}
```

### 1.4 编译器是怎么检查的？

```java
// 方法签名中的 throws 声明
public void readFile(String path) throws IOException, SQLException {
    // 如果不 catch IOException 或 SQLException，代码无法编译
    // 这是 JVM 层面的强制规范
}

// 违反规则：编译器报错
public void readFile(String path) {
    new FileReader(path);  // 编译错误！IOException 未处理
}

// 正确处理
public void readFile(String path) {
    try {
        new FileReader(path);
    } catch (IOException e) {
        // 处理或重新抛出
    }
}
```

## 二、为什么 Thread.sleep 抛的是 Checked Exception 🔴

### 2.1 问题拆解

面试官的这道追问非常经典：`Thread.sleep()` 明显是"运行时"的行为，为什么用 checked exception？

### 2.2 标准回答

`InterruptedException` 是 checked exception，**不是因为它"运行时才发生"，而是因为它代表的是一种"可被中断"的等待状态**：

```java
try {
    Thread.sleep(1000);
} catch (InterruptedException e) {
    // 收到中断信号——这是可预期的外部事件
    // 调用方需要知道线程被中断了，可能需要做清理工作
    Thread.currentThread().interrupt();  // 重新设置中断标志
    return;  // 或者做其他处理
}
```

`InterruptedException` 的语义是：**"当前线程被要求中断"**。这是一个有意义的信号，调用方应该知道并做出响应。

而 `String.charAt(-1)` 抛 `StringIndexOutOfBoundsException`，**是因为这是程序员的 bug**——你在调用 `charAt` 前应该检查索引，而不是让调用方 catch 一个"索引越界"的异常。

:::tip 💡
判断标准：**异常应该由谁处理？** 如果是**调用方**应该考虑并处理 → Checked。如果是**当前方法**自己的 bug → Unchecked。
:::

### 2.3 InterruptedException 的正确处理

这是生产中的高频踩坑点：

```java
// ❌ 错误：吞掉中断信号
try {
    Thread.sleep(1000);
} catch (InterruptedException e) {
    // 什么也不做——中断信号被吞掉了！
    // 调用方以为线程还在正常运行，实际上已经被要求中断了
}

// ✅ 正确：重新设置中断标志
try {
    Thread.sleep(1000);
} catch (InterruptedException e) {
    Thread.currentThread().interrupt();  // 重要！
    // 然后要么 return，要么做清理工作
}

// ✅ 正确：立即处理并返回
try {
    Thread.sleep(1000);
} catch (InterruptedException e) {
    // 收到中断请求，立即停止当前操作
    log.info("Operation interrupted");
    return;
}
```

## 三、Checked Exception 的设计争议 🟡

### 3.1 问题拆解

面试官追问："很多语言（C#、Python、Kotlin）都不支持 Checked Exception，你怎么看？"

这道题考察的是候选人对 Java 设计哲学的批判性思考。

### 3.2 Checked Exception 的问题

```java
// 问题1：方法签名污染
public void saveUser(User user)
    throws IOException, SQLException, TimeoutException {
    // 层层传递，每个方法都要声明 throws
    // 最终方法签名变成异常列表，而不是业务逻辑
}

// 问题2：过度处理
try {
    readFile(path);
} catch (IOException e) {
    // 不知道怎么处理，干脆 catch 然后 continue
    // 把有价值的异常信号白白浪费了
}

// 问题3：异常与返回值的混淆
// Checked Exception 试图强制调用方处理错误
// 但错误处理的方式有很多种：返回错误码、try-catch、rethrow
// 强制其中一种，限制了灵活性
```

### 3.3 Checked Exception 的价值

但 Checked Exception 也有价值：

```java
// 价值1：强制处理——不让错误静默传播
// 编译器逼着你考虑失败情况，减少"忘记处理"的错误
public void loadConfig(String path) throws IOException {
    // 如果不处理 IOException，代码根本编译不过
    // 这强迫开发者认真考虑文件不存在的场景
}

// 价值2：方法签名即文档
public void connect(String host, int port) throws TimeoutException, IOException {
    // 从方法签名就能看出这个方法可能失败的原因
    // 比注释更可靠，因为注释可能过时
}
```

| 观点 | 支持 Checked Exception | 反对 Checked Exception |
| --- | --- | --- |
| 好处 | 强制处理，减少静默错误；方法签名即文档 | 代码简洁；异常传播更自然 |
| 坏处 | 代码膨胀；方法签名污染；异常强制处理导致过度 try-catch | 异常可能被忽略；调用方不知道方法可能失败 |
| 代表语言 | Java | C#, Python, Kotlin, JavaScript |

## 四、工程选型 🔴

### 4.1 问题拆解

面试官追问："你在项目里自定义异常时，怎么决定用 Checked 还是 Unchecked？"

### 4.2 标准答案：通常选 Unchecked

阿里巴巴 Java 开发手册的推荐：**自定义业务异常通常继承 RuntimeException**。

原因：
1. **避免异常污染**：业务层异常层层传递会让方法签名变得不可读
2. **灵活性**：调用方可以选择处理或不处理
3. **与 Spring 生态兼容**：Spring 的声明式事务默认对 unchecked exception 回滚，对 checked exception 不会回滚

```java
// 推荐：自定义业务异常继承 RuntimeException
public class BusinessException extends RuntimeException {
    private String code;

    public BusinessException(String code, String message) {
        super(message);
        this.code = code;
    }
}

// 使用
public void transfer(Account from, Account to, BigDecimal amount) {
    if (from.getBalance().compareTo(amount) < 0) {
        // 余额不足，不需要强制 try-catch
        // 调用方可以选择处理这个异常，也可以让它传播
        throw new BusinessException("BALANCE_NOT_ENOUGH", "余额不足");
    }
}
```

### 4.3 什么时候用 Checked Exception？

| 场景 | 推荐 | 原因 |
| --- | --- | --- |
| 直接与外部 IO 交互（文件、网络、进程） | Checked Exception | 这些失败是可预期的，调用方必须考虑 |
| 需要强迫调用方处理 | Checked Exception | 比如 Spring 的 `IOException` |
| 业务异常（订单不存在、余额不足） | Unchecked Exception | 业务异常不应该强制处理，应该让框架统一处理 |
| 程序员错误（null、空指针） | Unchecked Exception | 永远用 Unchecked，这是 bug，不是业务流程 |

## 五、生产避坑 🔴

### 5.1 坑：把 Checked Exception 当 Unchecked 抛

```java
// ❌ 错误：吞掉 Checked Exception 的类型
public void processFile(String path) {
    try {
        new FileReader(path);
    } catch (IOException e) {
        throw new RuntimeException(e);  // 原始异常信息丢失了！
        // 应该用带 cause 的构造函数，或者记录原始异常信息
    }
}

// ✅ 正确
public void processFile(String path) {
    try {
        new FileReader(path);
    } catch (IOException e) {
        throw new BusinessException("FILE_READ_ERROR", "读取文件失败: " + path, e);
    }
}
```

### 5.2 坑：异常与 Spring 事务的误解

```java
// Spring 的默认事务行为：
// Unchecked Exception（RuntimeException 及其子类）→ 自动回滚
// Checked Exception → 默认不回滚（除非配置 rollbackFor）

@Transactional
public void saveOrder(Order order) {
    try {
        orderMapper.insert(order);
        inventoryService.deduct(order.getItems());
    } catch (Exception e) {
        // 如果你 catch 了但不重新抛出，事务不会回滚！
        log.error("保存订单失败", e);
    }
    // 方法正常结束，Spring 看到没有异常，提交事务
    // 但实际上订单保存成功了但库存没扣减！
}
```

:::warning ⚠️
Spring 的声明式事务依赖异常来判断是否回滚。如果你在方法内部 catch 了异常但没有重新抛出，Spring 认为事务"成功"了，会正常提交。这是生产中非常隐蔽的 bug。
:::
