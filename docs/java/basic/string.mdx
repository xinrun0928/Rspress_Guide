## 一、String 不可变性 🔴

### 1.1 问题拆解

**第一层：怎么用？**
面试官问："String 为什么是不可变的？"

**第二层：底层实现**
追问："String 真的完全不可变吗？有没有办法突破？"

**第三层：边界缺陷**
追问："String 不可变带来了哪些问题？比如大量字符串操作时的性能？"

**第四层：选型 trade-off**
追问："既然不可变有性能问题，为什么 JDK 还要把 String 设计成不可变？"

### 1.2 String 的底层结构

```java
// String.java（JDK 源码，JDK9 之后改为 byte[]）
public final class String
    implements java.io.Serializable, Comparable<String>, CharSequence {

    // JDK8 及之前：char[] 存储字符
    private final char value[];

    // JDK9 及之后：byte[] + 编码标记，更省内存
    private final byte[] value;

    // 哈希值缓存，String 不可变所以可以缓存
    private int hash;

    // 其他字段...
}
```

String 被 `final` 修饰，说明**类不能被继承**。`value` 数组被 `final` 修饰，说明**数组引用不能变**。但等等——数组内容可以变吗？

```java
String s = "hello";
// s.charAt(0) = 'H';  // 编译错误！String 没有提供修改内容的方法

// 但是，反射可以突破这个限制
Field field = String.class.getDeclaredField("value");
field.setAccessible(true);
char[] val = (char[]) field.get(s);
val[0] = 'H';  // 破坏了 String 的不可变性！
System.out.println(s);  // 打印 "Hello"
```

:::warning ⚠️
String 的不可变是由**设计约定**和**访问控制**共同保障的，不是 JVM 的硬性限制。反射可以破坏它，这也是为什么 String 不应该存储敏感信息。
:::

### 1.3 String 不可变的五大好处

| 好处 | 解释 | 生产意义 |
| --- | --- | --- |
| **字符串常量池** | 相同内容的字符串可以共享对象，节省内存 | 大量重复字符串时内存占用大幅降低 |
| **线程安全** | 不可变对象天然线程安全，不需要同步 | 多线程环境下直接共享，无需担心并发问题 |
| **哈希缓存** | String 可以缓存自己的 hashCode，因为内容不变 | HashMap 的 key 用 String 性能更好 |
| **安全保证** | 不会被意外/恶意修改 | 网络连接参数、文件路径、类加载器依赖 String 的不可变性 |
| **类加载器依赖** | 类加载时用的 String 不能被篡改 | 如果 String 可变，类加载机制可能被破坏 |

:::tip 💡
String 不可变最重要的意义是**支持字符串常量池**。JVM 维护一个 String Pool，相同内容的字面量字符串共享同一个对象——这在大量使用字符串的系统中能节省 30%~50% 的内存。
:::

### 1.4 String 的"假不可变"：+ 拼接

这是面试官最爱的追问点。String 明明不可变，为什么能拼接？

```java
String s = "a" + "b" + "c";  // 编译后是什么？
```

编译器会优化这段代码，**在编译时就计算好了**：

```java
// 编译器优化后
String s = "abc";
```

但如果拼接涉及变量呢？

```java
String a = "hello";
String b = "world";
String c = a + b;  // 编译后是什么？
```

编译器**无法优化**这种运行时才知道结果的拼接，编译后的字节码：

```java
// 编译器生成的是：
new StringBuilder().append(a).append(b).toString();
```

:::warning ⚡
在循环中拼接字符串是经典的性能杀手：
```java
String result = "";
for (int i = 0; i < 10000; i++) {
    result += i;  // 每次循环都 new 一个 StringBuilder！
}
```
这会在循环中创建 10000 个 StringBuilder 对象。正确做法是**在循环外创建一个 StringBuilder**。
:::

## 二、StringBuilder vs StringBuffer 🔴

### 2.1 问题拆解

**第一层：怎么用？**
面试官问："StringBuilder 和 StringBuffer 有什么区别？"

**第二层：底层实现**
追问："它们是怎么实现可变的？底层数据结构是什么？"

**第三层：边界缺陷**
追问："StringBuilder 是线程安全的吗？什么情况下会被多线程影响？"

**第四层：选型 trade-off**
追问："你在项目里什么时候用 StringBuilder，什么时候用 StringBuffer？"

### 2.2 标准回答：线程安全

```java
// StringBuilder：非线程安全
public class StringBuilder {
    @Override
    public StringBuilder append(String str) {
        super.append(str);  // 没有 synchronized
        return this;
    }
}

// StringBuffer：线程安全
public class StringBuffer {
    @Override
    public synchronized StringBuffer append(String str) {
        super.append(str);
        return this;
    }
}
```

**StringBuilder 的方法全是普通方法，没有 `synchronized` 修饰**。
**StringBuffer 的关键方法都有 `synchronized` 修饰**。

| 维度 | StringBuilder | StringBuffer |
| --- | --- | --- |
| 线程安全 | ❌ 否 | ✅ 是 |
| 性能 | 更高（无锁开销） | 较低（synchronized 有锁开销） |
| 使用场景 | 单线程、局部变量、拼接频繁 | 多线程共享、字符串构建器需要同步 |
| 引入版本 | JDK 1.5 | JDK 1.0 |

### 2.3 底层实现：可变的字符数组

```java
// AbstractStringBuilder 是 StringBuilder 和 StringBuffer 的共同父类
abstract class AbstractStringBuilder {
    char[] value;  // 可变的字符数组

    AbstractStringBuilder(int capacity) {
        value = new char[capacity];
    }
}

public class StringBuilder extends AbstractStringBuilder {
    public StringBuilder() {
        super(16);  // 默认容量 16
    }

    public StringBuilder(int capacity) {
        super(capacity);
    }

    @Override
    public StringBuilder append(String str) {
        if (str == null) {
            appendNull();
        } else {
            int len = str.length();
            ensureCapacityInternal(count + len);  // 扩容检查
            str.getChars(0, len, value, count);   // 追加到数组末尾
            count += len;
        }
        return this;
    }

    // 扩容机制：容量翻倍 + 2
    private void ensureCapacityInternal(int minimumCapacity) {
        if (minimumCapacity > value.length) {
            expandCapacity(minimumCapacity);
        }
    }

    void expandCapacity(int minimumCapacity) {
        int newCapacity = value.length * 2 + 2;  // 扩容公式
        value = Arrays.copyOf(value, newCapacity);
    }
}
```

**StringBuilder/StringBuffer 的扩容策略**：容量不够时，**翻倍 + 2**。比如初始容量 16，追加后不够了，就扩展到 `16 * 2 + 2 = 34`，再不够就 `34 * 2 + 2 = 70`...

### 2.4 扩容的坑

```java
StringBuilder sb = new StringBuilder();
// 默认容量只有 16，每 append 一次都检查扩容
// 如果你 append 一个很长的字符串（超过 16 个字符），
// 扩容会触发数组复制，性能急剧下降

// 正确做法：预估容量
StringBuilder sb2 = new StringBuilder(1000);  // 预分配容量，避免扩容
```

:::warning ⚠️
在高频拼接场景下，如果能预估最终字符串的大致长度，**一定要在构造时指定初始容量**。否则每次扩容都要 `Arrays.copyOf()` 复制整个数组，GC 压力也会增大。
:::

## 三、生产避坑 🔴

### 3.1 坑1：循环内字符串拼接

```java
// ❌ 错误写法
String sql = "";
for (String field : fields) {
    sql += field + ",";  // 每次 + 都会 new StringBuilder
}

// ✅ 正确写法
StringBuilder sb = new StringBuilder(fields.length * 10);
for (String field : fields) {
    sb.append(field).append(",");
}
String sql = sb.toString();

// ✅ 更好的写法（Java 8+）
String sql = String.join(",", fields);
```

### 3.2 坑2：StringBuilder 在多线程环境下被共享

```java
// ❌ 危险：StringBuilder 不是线程安全的
public class RequestHandler implements Runnable {
    private StringBuilder sb = new StringBuilder();  // 共享的！

    public void run() {
        sb.append("request-").append(id).append("\n");
        // 多线程同时 append，会出现数据错乱、字符丢失
    }
}

// ✅ 正确做法1：ThreadLocal
private ThreadLocal<StringBuilder> tl = ThreadLocal.withInitial(() -> new StringBuilder(256));

// ✅ 正确做法2：用 StringBuffer（如果一定要共享）
private StringBuffer sb = new StringBuffer(256);  // synchronized 安全

// ✅ 正确做法3：用局部变量
public void run() {
    StringBuilder sb = new StringBuilder(256);  // 每个线程独立一份
    sb.append("request-").append(id).append("\n");
}
```

### 3.3 坑3：toString() 创建新对象

```java
StringBuilder sb = new StringBuilder("hello");
String s1 = sb.toString();
String s2 = sb.toString();

System.out.println(s1 == s2);  // false！每次 toString() 都 new String()
// toString() 的实现：
public String toString() {
    return new String(value, 0, count);  // 每次都 new String
}
```

:::warning ⚠️
不要用 `sb.toString() == sb.toString()` 来判断内容是否变化。StringBuilder 不提供内容变化的检测机制，每次 toString() 都是一个新的 String 对象。
:::

### 3.4 坑4：String 在 HashMap 中被误用做 key

```java
Map<String, Integer> map = new HashMap<>();
String key = "user";
map.put(key, 1);

// 如果某段代码用反射或其他方式改变了 key 这个 String 的内容
// 那么 map.get(key) 就会返回 null，因为 String 的 hashCode 也变了
```

但实际上 String 被 final 修饰且没有暴露修改方法，所以正常使用下不会出现这个问题。String 不可变使得它**天然适合做 HashMap 的 key**。

## 四、工程选型 🔴

| 场景 | 推荐 | 原因 |
| --- | --- | --- |
| 字符串常量 | String | 不可变 + 常量池，性能最佳 |
| 单线程拼接（局部变量） | StringBuilder | 非线程安全但性能最好 |
| 多线程共享字符串构建器 | StringBuffer | synchronized 保证线程安全 |
| 大量字符串拼接（循环） | StringBuilder（预容量） | 避免频繁扩容 |
| 字符串列表拼接 | String.join / Arrays.stream | Java 8+ 最佳实践 |

```java
// Java 8+ 字符串拼接最佳实践
// 1. 列表拼接
List<String> names = Arrays.asList("张三", "李四", "王五");
String result = String.join(", ", names);  // "张三, 李四, 王五"

// 2. Stream 流拼接
String streamResult = names.stream()
    .collect(Collectors.joining(", "));

// 3. 预容量 StringBuilder（性能敏感场景）
StringBuilder sb = new StringBuilder(estimatedSize);
for (Item item : items) {
    sb.append(item.toString());
}
```

## 五、面试总结

| 级别 | 期望回答 | 判分标准 |
| --- | --- | --- |
| P5 | 知道 String 不可变，StringBuilder 非线程安全，StringBuffer 线程安全 | 基础概念对即可 |
| P6 | 能讲清 String 不可变的底层实现（final 类 + final 数组），能说出 `+` 拼接被编译器优化为 StringBuilder，理解扩容机制 | 追问 2-3 轮不崩 |
| P7 | 能讲清 String 为什么被设计成不可变（常量池、安全性、哈希缓存），能讲清 StringBuilder/StringBuffer 的 synchronized 细节，能结合生产案例讲性能调优 | 有架构视角，有生产踩坑经验 |

最后出一道实战题，检验你是否真的理解了这三者的区别：

```java
// 下面的代码有什么性能问题？
public static String buildUrl(String base, String... paths) {
    String url = base;
    for (String path : paths) {
        url = url + "/" + path;
    }
    return url;
}
```

这个方法在路径很多时（比如有 100 个 path），每次循环都会 new 一个 StringBuilder + toString，性能极差。正确做法是用 StringBuilder 在循环外初始化。

:::tip 💡
回答 String 相关问题时，如果能加上一句"阿里的 Java 开发手册明确要求：循环内禁止使用 `+` 进行字符串拼接，必须使用 StringBuilder"，会让面试官眼前一亮——这说明你不只研究过源码，还关注过工程规范。
:::
