# JDK 8 vs JDK 9+ String 实现差异

你知道吗？Java 的 String 类型在 JDK 9 迎来了一次重大变革。从 JDK 1.0 到 JDK 8，String 一直使用 `char[]` 来存储字符；但从 JDK 9 开始，String 根据内容类型可以选择使用 `byte[]` 或 `char[]`。这个看似微小的变化，实际上是 Java 对现代应用需求的回应——在大量使用英文的互联网时代，用一个字节存储英文字符比用两个字节更高效。今天，让我们深入探索这次变革的背景、实现细节以及对性能的影响。

## 变革的背景：为什么需要改变

### Unicode 的演进与存储效率

在 JDK 8 及之前，Java 使用 UTF-16 编码来表示字符串，每个字符占用 2 个字节（16 位）。这种设计的初衷是 UTF-16 可以表示所有 Unicode 字符，包括那些需要超过 2 字节的字符（通过代理对 surrogate pairs）。

然而，现实是残酷的：

- **英文字符主导的互联网**：统计分析显示，大多数 Java 应用中，超过 90% 的字符串只包含 Latin-1 字符（ASCII + 西欧字符）
- **Latin-1 vs UTF-16**：Latin-1 只需要 1 字节就可以表示所有字符，但 UTF-16 总是使用 2 字节，浪费了一半的空间
- **内存压力**：在大型应用中，String 对象占用的内存可能达到堆内存的 25-30%。如果能节省一半的 String 存储空间，将显著减少内存占用

### 一个具体的例子

考虑一个典型的 HTTP 请求，包含以下头部：

```
Content-Type: text/html; charset=UTF-8
Content-Length: 12345
Server: Apache-Coyote/1.1
```

这些值大部分都是 ASCII/Latin-1 字符。在 JDK 8 的实现中，即使是 "Content-Type: text/html" 这样的纯 ASCII 字符串，每个字符也要占用 2 字节。而在 JDK 9+ 的实现中，这类字符串可以使用 1 字节存储，内存占用减半。

## JDK 8 的 String 内部实现

### 核心字段

```java
// JDK 8 String.java
public final class String
    implements java.io.Serializable, Comparable<String>, CharSequence {
    
    /** 存储字符串内容的字符数组，始终是 UTF-16 编码 */
    private final char[] value;
    
    /** 字符串的哈希码，默认为 0 */
    private int hash;
    
    /** 用于序列化的序列化 ID */
    private static final long serialVersionUID = -6849794470754667710L;
}
```

### char[] vs byte[] 的选择

`char` 在 Java 中是 16 位无符号整数，可以表示 Unicode 码点（0x0000 到 0xFFFF）。对于 Basic Multilingual Plane（BMP）中的字符，直接用单个 char 表示；对于 Supplemental Characters（辅助字符），需要使用代理对（surrogate pairs），即两个 char。

这种设计的问题是：
- 对于只需要 Latin-1 范围的字符（0x00 到 0xFF），浪费了一半空间
- 代理对增加了字符串处理的复杂度

## JDK 9+ 的 String 内部实现

### 核心字段

```java
// JDK 9+ String.java
public final class String
    implements java.io.Serializable, Comparable<String>, CharSequence {
    
    /** 存储字符串内容的字节数组 */
    @Stable
    private final byte[] value;
    
    /** 编码标识：LATIN1 或 UTF16 */
    private final byte coder;
    
    /** 字符串的哈希码，默认为 0 */
    private int hash;
    
    /** 编码常量 */
    static final byte LATIN1 = 0;
    static final byte UTF16 = 1;
}
```

### coder 字段的作用

coder 字段是一个字节（0 或 1），用于标识 value 数组使用的编码方式：
- `LATIN1`（0）：value 使用 ISO-8859-1（Latin-1）编码，每个字符 1 字节
- `UTF16`（1）：value 使用 UTF-16 编码，每个字符 2 字节

通过这种方式，String 可以根据内容自动选择最紧凑的编码。

### 自动选择编码的逻辑

```java
// JDK 9+ String 构造方法中的编码选择
public String(char[] value, int off, int len, Void cotag) {
    if (len == 0) {
        this.coder = LATIN1;
        this.value = StringUTF16.compress(value, off, len);
        // ...
    } else {
        // 检查是否所有字符都在 Latin-1 范围内
        this.coder = StringUTF16.canEncode(len) ? LATIN1 : UTF16;
        this.value = (this.coder == LATIN1)
            ? StringUTF16.compress(value, off, len)
            : StringUTF16.toBytes(value, off, len);
    }
}

// 静态方法：检查是否所有字符都可用 Latin-1 表示
static boolean canEncode(int length) {
    // 在 ASCII 范围内 (0-127) 的字符可以用 Latin-1 表示
    // 但实际上 UTF-16 需要处理更多情况
    // ...
    return true;
}
```

当创建字符串时，JVM 会检查所有字符是否都在 Latin-1 范围内：
- 如果是，选择 `LATIN1` 编码，使用 `StringUTF16.compress()` 压缩为 1 字节/字符
- 如果不是（包含中文字符或特殊 Unicode 字符），选择 `UTF16` 编码，使用 2 字节/字符

### UTF16 为什么仍然存在

既然 String 可以用 Latin-1，为什么还要保留 UTF16？因为 Java 需要处理所有 Unicode 字符，包括那些超过 U+FFFF 的字符（如 emoji 表情）。

```java
// emoji 字符需要使用代理对
String emoji = "\uD83D\uDE00";  // 😀 的 UTF-16 代理对
// 在 Latin-1 中无法表示
```

这些字符必须使用 UTF-16 编码存储，即使它们在 Latin-1 字符串中表现为"不可编码"的字符。

## 性能影响分析

### 内存节省

在典型的 Java Web 应用中，String 的内存节省效果显著：

```java
// 测试代码
public class StringMemoryTest {
    public static void main(String[] args) {
        // 纯英文字符串
        String english = "Hello World, this is a test string!";
        
        // 包含中文的字符串
        String chinese = "你好，世界！这是一个测试字符串！";
        
        // 混合字符串
        String mixed = "Hello 你好 World 世界";
        
        // 统计内存使用（需要通过反射或工具获取）
        printMemoryInfo(english);
        printMemoryInfo(chinese);
        printMemoryInfo(mixed);
    }
}
```

理论上的内存节省：

| 字符串类型 | JDK 8 | JDK 9+ | 节省比例 |
|-----------|-------|--------|---------|
| 纯 ASCII/Latin-1 | 2N 字节 | N 字节 | 50% |
| 包含 BMP 外字符 | 2N 字节 | 2N 字节 | 0% |
| 混合（典型应用） | 2N 字节 | ~1.2N 字节 | ~40% |

### GC 压力降低

String 对象的内存节省直接影响到垃圾收集的压力：

- 更少的 String 对象意味着更少的对象分配
- 更少的对象意味着更少的 GC 扫描和处理
- 更紧凑的堆使用意味着更好的缓存局部性

<Badge text="性能提示" type="tip" /> 紧凑字符串（Compact Strings）对性能的影响是全方位的：内存减少、GC 压力降低、缓存命中率提高。对于内存敏感的应用（如微服务、Serverless 函数），这尤其重要。

## 代码兼容性与注意事项

### 内部 API 的变化

JDK 9+ 的 String 内部变化主要在 JVM 层面，对 Java 源代码基本透明。但有一些依赖内部实现的代码可能会受影响：

```java
// 可能的兼容性问题
// JDK 8
String s = "hello";
char[] chars = s.toCharArray();  // 返回 char[]

// JDK 9+ 仍然返回 char[]，但内部实现已经变化
// 这个方法会进行必要的转换，不会影响用户代码
```

### 反射访问的差异

```java
// 在 JDK 8 中，可以通过反射访问 char[] value
Field valueField = String.class.getDeclaredField("value");
valueField.setAccessible(true);
char[] value = (char[]) valueField.get(s);

// 在 JDK 9+ 中，value 是 byte[]
// 类型检查会失败
try {
    byte[] bytes = (byte[]) valueField.get(s);
} catch (ClassCastException e) {
    // 需要处理类型转换问题
}
```

<Badge text="警告" type="warning" /> 通过反射修改 String 内部状态一直是不良实践。在 JDK 9+ 中，这种做法更加危险，因为内部表示可能随时变化。强烈建议不要使用反射来修改 String 对象。

## 字符串长度计算的变化

### charLength vs byteLength vs codePointLength

```java
String s = "Hello你好";

System.out.println(s.length());           // 字符数（代码单元）
System.out.println(s.codePointCount(0, s.length()));  // 码点数

// JDK 9+ 新增方法
System.out.println(s.isLatin1() ? s.getBytes().length 
                                : s.getBytes().length * 2);  // 实际字节数
```

### 索引与字符访问

```java
String emoji = "😀";

// JDK 8：emoji.length() == 2（代理对）
// JDK 9+：emoji.length() == 2，行为一致

// 正确处理代理对的方式
emoji.codePoints().forEach(cp -> {
    System.out.println("Code point: " + cp);
});
```

## JEP 277: 增强弃用

与紧凑字符串相关的是 JEP 277，它改进了 `@Deprecated` 注解，允许标记 API 的"移除"版本：

```java
@Deprecated(since = "9", forRemoval = true)
public String getBytes(int charset) {
    // ...
}
```

这意味着某些与字符编码相关的老旧方法可能在未来版本中被移除。

## 未来展望

紧凑字符串是 Java 持续优化的一部分。展望未来，我们可能看到更多类似的优化：

- **字符串去重**（G1 GC 的字符串去重特性）已经是生产可用的
- **值类型**（Valhalla 项目）可能为 String 提供更进一步的优化
- **新的字符串拼接机制**（JEP 280）允许 JVM 动态优化拼接策略

## 总结

JDK 9 引入的紧凑字符串是 Java 演化史上的一个重要里程碑。它不改变 String 的外部行为，但显著降低了内存占用，提升了整体性能。这次变化体现了 Java 团队对现代应用需求的回应——在保持向后兼容的同时，持续优化核心类型的实现。

理解这些底层变化，能帮助我们更好地理解 Java 的演进方向。作为开发者，我们不需要记住每一个内部细节，但理解 String 为什么这样设计，能让我们写出更加高效的代码。
