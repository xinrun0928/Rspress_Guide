# StringBuilder vs StringBuffer vs String

在 Java 中拼接字符串，你有多少种选择？`+` 运算符、`StringBuilder`、`StringBuffer`、甚至古老的 `StringBuffer` 的兄弟 `String`……这些选项让人眼花缭乱。什么时候该用哪一种？它们的性能差异有多大？为什么 `StringBuffer` 几乎被遗忘了？这些问题看似简单，却困扰着不少 Java 开发者。今天，让我们通过实际案例和性能测试，彻底弄清楚这个问题。

## 一个常见的问题场景

让我们从一个经典的面试题开始：

```java
// 方式1：使用 + 拼接
String result1 = "";
for (int i = 0; i < 100; i++) {
    result1 += i;
}

// 方式2：使用 StringBuilder
StringBuilder sb = new StringBuilder();
for (int i = 0; i < 100; i++) {
    sb.append(i);
}
String result2 = sb.toString();
```

这两种方式看起来功能完全相同，但你知道它们背后的区别吗？

## String 的拼接之谜

当你写下 `str += "hello"` 这样的代码时，Java 编译器实际上做了什么？

```java
// 源代码
str += "hello";

// 编译器生成的（大致）等效代码
str = new StringBuilder(str).append("hello").toString();
```

没错！编译器会把所有的 `+` 拼接操作转换为 `StringBuilder` 的操作。这意味着 `+` 拼接并不是什么神奇的语法糖，而是编译器帮你自动创建了 StringBuilder。

但是，这里有一个陷阱：编译器只在单个表达式中使用同一个 StringBuilder。考虑以下代码：

```java
String result = "";
for (int i = 0; i < 100; i++) {
    result += i;  // 每次循环都创建新的 StringBuilder！
}
```

编译器无法跨表达式优化，所以每次循环都会创建一个新的 StringBuilder，然后立即调用 `toString()` 转换为 String。这导致大量中间对象的创建，严重影响性能。

## StringBuilder：可变字符串的首选

`StringBuilder` 是 Java 5 引入的，替代了早期的 `StringBuffer`。它的核心特点是：**可变、非同步、线程不安全**。

```java
// 基本用法
StringBuilder sb = new StringBuilder();
sb.append("hello");
sb.append(" ");
sb.append("world");
String result = sb.toString();  // "hello world"

// 链式调用
String result2 = new StringBuilder()
    .append("hello")
    .append(" ")
    .append("world")
    .toString();

// 插入和删除
StringBuilder sb2 = new StringBuilder("HelloWorld");
sb2.insert(5, " ");      // "Hello World"
sb2.delete(5, 6);         // "HelloWorld"
sb2.replace(5, 11, " Java"); // "Hello Java"
```

### StringBuilder 的内部机制

`StringBuilder` 内部维护着一个字符数组（`char[]` 或 `byte[]` + 编码器），当容量不足时会自动扩容：

```java
// StringBuilder 的简化实现
public final class StringBuilder
    extends AbstractStringBuilder
    implements java.io.Serializable, Appendable {
    
    public StringBuilder() {
        super(16);  // 默认容量 16
    }
    
    public StringBuilder(String str) {
        super(str.length() + 16);  // 容量 = 字符串长度 + 16
    }
    
    public StringBuilder append(String str) {
        super.append(str);
        return this;  // 支持链式调用
    }
}
```

**扩容机制**：当需要添加的字符长度超过当前容量时，StringBuilder 会创建一个新的更大的数组（通常是原容量的 2 倍 + 2），然后复制原有内容。这是一个相对昂贵的操作，所以如果能预估最终大小，在构造时就指定容量可以避免频繁扩容：

```java
// 预估容量，避免扩容
StringBuilder sb = new StringBuilder(1000);
for (int i = 0; i < 1000; i++) {
    sb.append(someData[i]);
}
```

## StringBuffer：线程安全的代价

`StringBuffer` 是 StringBuilder 的"老大哥"，在 Java 1.0 就已经存在。它与 StringBuilder 的核心区别在于：**所有关键方法都使用 `synchronized` 关键字同步**。

```java
// StringBuffer 的 append 方法
@Override
public synchronized StringBuffer append(String str) {
    toStringCache = null;
    super.append(str);
    return this;
}

@Override
public synchronized StringBuffer delete(int start, int end) {
    toStringCache = null;
    super.delete(start, end);
    return this;
}

// 其他修改方法同样都是 synchronized
```

这种同步机制使得 `StringBuffer` 可以在多线程环境下安全使用，但代价是性能开销。

<Badge text="性能提示" type="warning" /> 在单线程环境下或仅在局部作用域内使用时，`StringBuffer` 的同步开销是不必要的。使用 `StringBuilder` 通常是更好的选择。

## 性能对比：数字说话

让我们通过实际测试来比较三种方式的性能：

```java
public class StringConcatBenchmark {
    private static final int ITERATIONS = 100000;
    
    public static void main(String[] args) {
        // 测试1：少量拼接
        benchmarkPlus(100);
        benchmarkBuilder(100);
        benchmarkBuffer(100);
        
        // 测试2：大量拼接
        benchmarkPlus(10000);
        benchmarkBuilder(10000);
        benchmarkBuffer(10000);
    }
    
    private static void benchmarkPlus(int count) {
        long start = System.nanoTime();
        String result = "";
        for (int i = 0; i < count; i++) {
            result += "item" + i;
        }
        long duration = System.nanoTime() - start;
        System.out.printf("+ 拼接 (%d): %d ms%n", count, duration / 1_000_000);
    }
    
    private static void benchmarkBuilder(int count) {
        long start = System.nanoTime();
        StringBuilder sb = new StringBuilder();
        for (int i = 0; i < count; i++) {
            sb.append("item").append(i);
        }
        String result = sb.toString();
        long duration = System.nanoTime() - start;
        System.out.printf("StringBuilder (%d): %d ms%n", count, duration / 1_000_000);
    }
    
    private static void benchmarkBuffer(int count) {
        long start = System.nanoTime();
        StringBuffer sb = new StringBuffer();
        for (int i = 0; i < count; i++) {
            sb.append("item").append(i);
        }
        String result = sb.toString();
        long duration = System.nanoTime() - start;
        System.out.printf("StringBuffer (%d): %d ms%n", count, duration / 1_000_000);
    }
}
```

典型的测试结果（具体数值因机器而异）：

| 拼接方式 | 100次 | 10000次 |
|---------|-------|---------|
| `+` 拼接 | 0.5 ms | ~5000 ms |
| StringBuilder | 0.1 ms | 1 ms |
| StringBuffer | 0.2 ms | 2 ms |

从结果可以看出：
- `+` 拼接在大循环中性能极差，因为每次迭代都创建新的 StringBuilder 对象
- StringBuilder 比 StringBuffer 略快，因为没有同步开销
- 在小规模拼接时，差异不明显，但养成使用 StringBuilder 的习惯是好的

## 实际应用场景分析

### 场景1：日志拼接

```java
// 不推荐：每次调用都创建新的 StringBuilder
public String buildLogMessage(String userId, String action) {
    return "User " + userId + " performed " + action + " at " + new Date();
}

// 推荐：使用 StringBuilder
public String buildLogMessage(String userId, String action) {
    return new StringBuilder()
        .append("User ")
        .append(userId)
        .append(" performed ")
        .append(action)
        .append(" at ")
        .append(new Date())
        .toString();
}

// 更推荐：如果是单次使用，Java 14+ 的文本块更清晰
public String buildLogMessage(String userId, String action) {
    return """
            User %s performed %s at %s
            """.formatted(userId, action, new Date());
}
```

<Badge text="JDK 14+" type="tip" /> Java 14 引入了文本块（Text Blocks），使用 `"""` 语法，可以更自然地编写多行字符串。结合 `String.formatted()` 或 `StringTemplate`，拼接体验大大提升。

### 场景2：构建 SQL 查询

```java
// 使用 StringBuilder 构建动态 SQL
public String buildSelectQuery(String table, List<String> columns, 
                                Map<String, Object> conditions) {
    StringBuilder sql = new StringBuilder("SELECT ");
    
    // 添加列名
    sql.append(String.join(", ", columns));
    sql.append(" FROM ").append(table);
    
    // 添加条件
    if (!conditions.isEmpty()) {
        sql.append(" WHERE ");
        int count = 0;
        for (Map.Entry<String, Object> entry : conditions.entrySet()) {
            if (count > 0) sql.append(" AND ");
            sql.append(entry.getKey()).append(" = ?");
            count++;
        }
    }
    
    return sql.toString();
}
```

### 场景3：多线程环境下的字符串构建

在多线程环境下，如果确实需要构建共享的字符串内容，`StringBuffer` 仍然是一个选择：

```java
// 多个线程协作构建一个字符串
public class SharedStringBuilder {
    private final StringBuffer buffer = new StringBuffer();
    
    public void appendFromMultipleThreads(String data) {
        synchronized (buffer) {  // 手动同步
            buffer.append(data);
        }
    }
    
    public String getResult() {
        synchronized (buffer) {
            return buffer.toString();
        }
    }
}
```

但更现代的做法可能是使用 `CopyOnWriteArrayList` 或其他并发结构，而不是依赖 `StringBuffer` 的同步特性。

## JDK 9+ 的字符串拼接优化

JDK 9 引入了一个名为 InvokeDynamic（indy）的新的字符串拼接机制，称为「动态拼接」：

```java
// 编译后不再转换为 StringBuilder
// 而是使用 invokedynamic 调用 StringConcatFactory
String result = "hello" + " " + "world";
```

这种新机制的优势在于：
1. 允许 JVM 动态选择最优的拼接策略
2. 可以根据具体场景（如少量拼接 vs 大量拼接）生成不同的字节码
3. 未来可以通过改变实现而不改变源码来优化性能

<Badge text="JDK 9+" type="tip" /> 虽然编译器不再自动使用 StringBuilder，但代码的可读性和意图仍然重要。对于简单拼接，`+` 仍然是可以接受的；但对于复杂或循环中的拼接，推荐显式使用 StringBuilder。

## 选择指南

| 场景 | 推荐选择 | 原因 |
|------|---------|------|
| 单次简单拼接 | `+` 或 `String.formatted()` | 可读性好，性能已优化 |
| 循环中拼接 | `StringBuilder` | 避免创建过多中间对象 |
| 多线程共享 | `StringBuffer` | 线程安全（但考虑其他方案） |
| 单线程高性能 | `StringBuilder` | 最优性能 |

## 常见错误避免

### 错误1：在循环中使用字符串拼接

```java
// 错误
String result = "";
for (Item item : items) {
    result += item.getName() + ",";
}

// 正确
StringBuilder sb = new StringBuilder(items.size() * 16);
for (Item item : items) {
    sb.append(item.getName()).append(",");
}
String result = sb.toString();
```

### 错误2：忘记调用 toString()

```java
StringBuilder sb = new StringBuilder();
sb.append("hello");
// 错误：直接使用 sb，Java 会自动调用 toString()
// 但这样每次使用都会调用 toString()，性能很差
log.debug("Message: " + sb);  // 这里 sb 会被转换为 String

// 正确：明确调用 toString()
log.debug("Message: " + sb.toString());
```

### 错误3：过度优化

```java
// 过度优化：不必要的容量预分配
StringBuilder sb = new StringBuilder(10000000);  // 10MB 容量
// 只有在使用接近这个大小的情况下才有意义
// 对于大多数场景，默认的 16 容量就足够了
```

## 总结

StringBuilder 和 StringBuffer 是 Java 中处理可变字符串的两种主要方式。StringBuilder 以其非同步的设计提供了更好的性能，是大多数场景下的首选；StringBuffer 虽然线程安全，但同步开销使其在现代 Java 应用中逐渐被边缘化。

理解它们的工作原理，能帮助你在日常开发中做出更好的选择。在单线程环境下，总是优先选择 StringBuilder；在需要线程安全时，考虑使用其他更现代的并发机制，而不是依赖 StringBuffer。

你是否曾在项目中遇到过字符串拼接导致的性能问题？当时是如何解决的？对于现代 Java 开发，你认为 StringBuilder 会被完全取代吗？这些问题的答案可能会随着 Java 版本的演进而改变，但理解基础原理永远是最好的起点。
