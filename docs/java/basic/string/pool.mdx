# 字符串常量池与 intern()

你有没有遇到过这样的困惑：两个字符串内容完全相同，但 `==` 比较却返回 false？或者听说 `String.intern()` 可以节省内存，却不知道什么时候该用？这些问题都指向一个 Java 中不太被充分理解的概念——字符串常量池。这个看似简单的机制，实际上是 Java 内存管理中一个精妙的优化设计。今天，让我们揭开字符串常量池的神秘面纱，理解它的工作原理和最佳实践。

## 什么是字符串常量池

字符串常量池（String Constant Pool）是 Java 堆内存中的一块特殊区域，用于存储字符串字面量和通过 `intern()` 方法驻留的字符串对象。它的核心思想是：如果两个字符串内容相同，为什么要在内存中存储两份副本？共享同一份副本不是更节省空间吗？

这个想法看似简单，但要正确实现却需要解决很多问题。首先，字符串可能被传递给不可信的代码，如果返回的是可修改的对象，后果不堪设想。其次，字符串的哈希码可能被缓存，如果对象被修改，缓存的哈希码就失效了。这些问题因为 String 的不可变性而迎刃而解——正是因为 String 不可变，才使得字符串常量池的实现成为可能。

### 字符串常量池的位置变迁

字符串常量池的位置在 Java 发展历程中发生过变化：

- **JDK 7 之前**：字符串常量池位于永久代（PermGen）中，这是一个固定的内存区域，大小有限
- **JDK 7**：字符串常量池被移到了堆内存中，因为永久代的固定大小导致了 "OutOfMemoryError: PermGen space" 错误
- **JDK 8+**：彻底移除了永久代，使用原生内存（Metaspace）替代，字符串常量池继续留在堆内存中

这一变化使得字符串常量池的大小可以动态增长，不再受固定内存限制的影响。但这也带来了一些新的考量：如果常量池无限增长，同样会导致内存问题。

## 字面量字符串的工作原理

当你写出 `String s = "hello";` 这样的代码时，背后发生了什么？

### 编译期的优化

Java 源代码在编译期就会被处理。编译器会将所有字符串字面量收集起来，放在 class 文件的"常量池"中。运行时，当类被加载时，这些字面量会被加载到 JVM 的运行时常量池，然后可能进一步被添加到字符串常量池。

```java
String s1 = "hello";
String s2 = "hello";
```

对于上述代码，编译器在编译时就知道 "hello" 这个字面量只会存在一份。因此，在编译后的字节码中，两个变量引用的是同一个常量池入口。运行时会确保 `s1` 和 `s2` 指向同一个 String 对象。

### 验证字符串常量池的行为

让我们通过一些实验来验证字符串常量池的行为：

```java
public class StringPoolDemo {
    public static void main(String[] args) {
        String s1 = "hello";
        String s2 = "hello";
        String s3 = new String("hello");
        
        System.out.println(s1 == s2);           // true
        System.out.println(s1 == s3);           // false
        System.out.println(s1.equals(s3));      // true
        System.out.println(s1.intern() == s3);  // false
        System.out.println(s1.intern() == s3.intern());  // true
    }
}
```

这个实验揭示了几个重要事实：

1. **字面量字符串来自常量池**：`s1` 和 `s2` 直接指向字符串常量池中的同一个对象
2. **`new String()` 会在堆中创建新对象**：`s3` 是一个新创建的对象，即使内容相同，也不是同一个引用
3. **`intern()` 方法的作用**：`intern()` 会返回字符串常量池中的"官方"版本

### intern() 方法详解

`String.intern()` 是字符串常量池的核心方法。让我们深入理解它的工作原理：

```java
public native String intern();
```

这是一个本地方法（native method），意味着它用其他语言（通常是 C 或 C++）实现，因为需要直接操作 JVM 的内部数据结构。

**intern() 的工作流程**：

1. 如果字符串常量池中已经存在一个字符串，其内容与当前字符串相等（通过 `equals()` 方法判断），则返回常量池中的那个字符串
2. 如果常量池中不存在，则将当前字符串添加到常量池，并返回对这个驻留字符串的引用

```java
String s1 = new String("hello");    // 创建了两个对象：
                                     // 1. 堆中的新 String 对象
                                     // 2. 常量池中的 "hello" 字面量（可能已存在）

String s2 = s1.intern();            // 返回常量池中的 "hello"

String s3 = "hello";                // 直接从常量池获取

System.out.println(s2 == s3);       // true
```

### 何时使用 intern()

`intern()` 看起来很美好，但并不是所有场景都适合使用。以下是一些最佳实践：

**适合使用 intern() 的场景**：

```java
// 场景1：处理大量重复的字符串
// 例如：处理日志文件中的重复单词
public class WordCounter {
    private Set<String> uniqueWords = new HashSet<>();
    
    public void addWord(String word) {
        // 对于频繁出现的单词，intern() 可以节省内存
        uniqueWords.add(word.intern());
    }
}

// 场景2：作为 Map 的键时
// 如果 Map 会长期存在，使用 intern() 可以让比较更快
Map<String, Integer> wordCount = new HashMap<>();
wordCount.put(userInput.intern(), 1);
```

**不适合使用 intern() 的场景**：

```java
// 场景1：短期存在的字符串
// intern() 的引用会被永久保存在常量池中，如果字符串是临时的，会浪费内存
void processRequest(String input) {
    // 不应该在这里 intern()
    // 因为请求结束后，input 可能不再需要
    // 但常量池会一直保留它
}

// 场景2：用户输入或不可控的字符串
// 如果处理恶意输入，故意制造大量不同的字符串，可能导致 OOM
void handleInput(String userInput) {
    // 危险！用户可以构造大量不同的字符串
    // 这些字符串都会被永久保存在常量池中
    cache.add(userInput.intern());
}
```

## 字符串常量池的实现细节

### JDK 6-8 的实现

在 JDK 8 及之前，字符串常量池的实现基于一个 `StringTable`——一个固定大小的哈希表。哈希表的大小可以通过 `-XX:StringTableSize` 参数调整。

```java
// 简化版的实现逻辑
private String intern(String s) {
    // 查找常量池中是否已有
    String existing = table.get(s);
    if (existing != null) {
        return existing;
    }
    // 添加到常量池
    table.put(s);
    return s;
}
```

默认的 table 大小在不同版本中有所不同，早期版本只有 1009 个桶，在大量使用字符串的应用中可能成为瓶颈。

### JDK 9+ 的改进

JDK 9 对字符串常量池进行了显著改进，特别是引入了更高效的数据结构和动态调整大小的能力。

<Badge text="JDK 9+" type="tip" /> JDK 9+ 中，字符串常量池的大小可以动态增长，不再受限于固定大小的哈希表。这使得大规模应用能够更高效地使用常量池。

### G1 GC 下的字符串去重

JDK 8u20 引入了 G1 垃圾收集器下的字符串去重特性：

```bash
# 启用字符串去重
-XX:+UseG1GC -XX:+UseStringDeduplication
```

这个特性会自动检测堆中具有相同内容的 String 对象，并将它们指向同一个共享的字符数组，从而节省内存。字符串去重与字符串常量池的区别在于：

- **字符串常量池**：只包含字面量和手动 intern() 的字符串
- **字符串去重**：自动检测所有 String 对象，包括运行时创建的

## 常见误区与陷阱

### 误区1：intern() 总是节省内存

`intern()` 确实可以减少重复字符串对象的数量，但代价是字符串常量池本身会占用内存。如果使用不当，可能导致常量池过度膨胀。

```java
// 糟糕的用法：处理数百万不同的 URL
List<String> urls = parseUrlsFromFile("urls.txt");
for (String url : urls) {
    // 危险！每个不同的 URL都会被永久保存在常量池
    process(url.intern());
}
```

### 误区2：字面量一定在常量池中

对于通过 `+` 连接的常量表达式，编译器会进行折叠优化：

```java
String s1 = "hello" + " world";  // 编译时优化，s1 直接引用 "hello world"
String s2 = "hello world";       // 与 s1 相同
System.out.println(s1 == s2);    // true
```

但对于包含变量的表达式，则不会进行折叠：

```java
String hello = "hello";
String world = "world";
String s1 = hello + world;        // 运行时拼接，不在常量池
String s2 = "helloworld";         // 在常量池
System.out.println(s1 == s2);    // false
```

### 陷阱：字符串驻留与内存泄漏

在某些场景下，不当使用 `intern()` 可能导致内存泄漏：

```java
// 模拟问题场景：爬虫程序
public class WebCrawler {
    public void processUrl(String url) {
        // 如果每次访问不同的 URL，都调用 intern()
        // 所有访问过的 URL 都会被永久保存
        visitedUrls.add(url.intern());
    }
}
```

正确的做法是：

```java
public class WebCrawler {
    public void processUrl(String url) {
        // 不使用 intern()，让垃圾收集器正常工作
        visitedUrls.add(url);
        
        // 或者，只对确定会重复出现的字符串使用 intern()
        if (isKnownPattern(url)) {
            visitedUrls.add(url.intern());
        }
    }
}
```

## 性能调优建议

### 调整常量池大小

对于大型应用，可以通过调整哈希表大小来优化性能：

```bash
# JDK 8
java -XX:StringTableSize=600000 -jar application.jar

# JDK 9+
# 在某些版本中不再需要手动调整，但仍可以通过以下方式诊断
jcmd <pid> VM.stringtable -stats
```

### 监控常量池使用

使用 JVM 工具监控字符串常量池的状态：

```java
// 使用 JMX 监控
MBeanServer mbs = ManagementFactory.getPlatformMBeanServer();
StringTableMXBean stringTable = JMX.newMBeanProxy(
    mbs, 
    new ObjectName("java.lang:type=StringTable"),
    StringTableMXBean.class
);

System.out.println("String table size: " + stringTable.size());
System.out.println("Bucket count: " + stringTable.bucketCount());
```

## 总结与思考

字符串常量池是 Java 内存管理中一个看似简单却极其精妙的设计。它建立在 String 不可变性的基础之上，通过共享字符串内容来节省内存。理解它的工作原理，对于编写高效的 Java 代码至关重要。

但同时，我们也要警惕过度使用 `intern()` 可能带来的问题。字符串常量池是 JVM 级别的全局资源，不加节制地使用可能导致内存膨胀。
