# String 不可变性与实现原理

你有没有想过这样一个问题：为什么 Java 的 String 被设计成不可变的？这看似一个简单的设计决策，背后却蕴含着深刻的思考。安全考虑、性能优化、多线程安全、字符串常量池的实现——这一切都建立在 String 不可变性的基础之上。如果 String 是可变的，很多我们习以为常的 Java 特性将不复存在，或者需要用完全不同的方式实现。今天，让我们深入探索 String 不可变性的前世今生，理解这一设计决策的精妙之处。

## 什么是不可变性

在编程语言中，不可变性（Immutability）指的是对象创建后，其状态不能被改变。对于 String 来说，就是一旦创建了一个 String 对象，你无法修改它的内容。任何看似"修改" String 的操作，实际上都是创建了一个新的 String 对象。

```java
String s = "hello";
s = s + " world";  // 这里并没有修改原来的 String
                   // 而是创建了一个新的 String "hello world"
                   // 原来的 "hello" 仍然存在
```

这种设计看似浪费（每次修改都要创建新对象），但实际上带来了巨大的好处：线程安全、无需同步、可以安全地共享、哈希码可以被缓存……让我们逐一分析这些好处。

## 不可变性的五大好处

### 1. 线程安全：天然的无锁世界

由于 String 对象不可被修改，多个线程可以同时访问同一个 String 对象而无需任何同步机制。想象一下，如果 String 是可变的，那么在读取一个字符串的同时，可能另一个线程正在修改它，这将导致数据不一致和各种竞态条件。而不可变性让这一切问题烟消云散——因为没有任何线程能修改它，所以读取永远是安全的。

在并发编程中，对象的可变性是导致线程安全问题的主要元凶之一。不可变对象天生就是线程安全的，这使得 String 成为多线程环境下的理想选择。Map 的键、Set 的元素、方法的参数——这些场景中都大量使用 String，就是因为它们需要被多个线程安全地共享。

### 2. 安全性：防止被恶意篡改

考虑这样一个场景：一个方法接收 String 类型的用户输入作为文件路径。如果 String 是可变的，那么恶意代码可能在你检查路径之后、实际使用路径之前修改这个 String，导致安全漏洞。这就是所谓的 TOCTOU（Time-of-check to time-of-use）攻击。

```java
// 假设这是一个检查权限的方法
boolean checkPermission(String path) {
    // 检查 path 是否有权限
    // ...
}

// 如果 String 是可变的，这里就存在安全隐患
// 恶意代码可能在 checkPermission 之后修改 path
```

String 的不可变性确保了一旦传入某个方法，调用者可以确信这个字符串不会被修改。这在安全敏感的应用中尤为重要，比如文件路径验证、类加载、网络 URL 验证等场景。

### 3. 字符串常量池的实现基础

字符串常量池是 Java 为了节省内存而设计的一块特殊区域，它保存着所有字符串字面量。正是因为 String 是不可变的，字符串常量池才能正常工作——如果 String 是可变的，一个地方的"修改"会导致所有引用该字符串的地方都受到影响，这将是灾难性的。

```java
String s1 = "hello";
String s2 = "hello";
// s1 和 s2 指向同一个对象
// 因为字符串常量池检测到 "hello" 已经存在
// 就直接返回了已有的引用
```

这个优化被称为驻留（Interning），它依赖于 String 的不可变性。如果 String 是可变的，那么共享同一个对象将是不安全的。

### 4. 哈希码的可缓存性

String 类经常被用作 HashMap 的键，因为 String 的哈希码被频繁计算。不可变性使得 String 的哈希码可以在第一次计算后缓存起来，以后每次使用时直接返回缓存的值，而无需重新计算。

```java
// String 类的 hashCode 实现
private int hash;  // 缓存的哈希码，默认为 0

public int hashCode() {
    int h = hash;
    if (h == 0 && value.length > 0) {
        // 只计算一次，之后直接返回缓存值
        hash = h = isLatin1() ? StringLatin1.hashCode(value)
                              : StringUTF16.hashCode(value);
    }
    return h;
}
```

这个缓存机制在 String 被广泛用作 HashMap 键的场景中提供了显著的性能提升。如果 String 是可变的，每次哈希计算前都需要重新计算，而且一旦 String 被修改，之前缓存的哈希码就失效了。

### 5. 类加载器的正确性

Java 的类加载机制依赖于 String 的不可变性。当一个类被加载时，它的名称（包括包名和类名）都是 String 对象。如果 String 是可变的，那么类的身份标识就可能发生变化，这将导致类加载、依赖管理和类对象身份比较等一系列问题。

## 不可变性的实现机制

既然不可变性如此重要，Java 是如何保证 String 不可变的呢？答案是：final 关键字 + 私有字段 + 不暴露修改方法。

### 第一道防线：final 关键字

```java
public final class String
    implements java.io.Serializable, Comparable<String>, CharSequence {
    
    // value 是存储字符串内容的数组
    private final byte[] value;
    
    // ...
}
```

String 类被声明为 `final`，这意味着它不能被继承，防止子类修改其行为。更重要的是，存储字符串内容的 `value` 数组也被声明为 `final`。在 JDK 9 之前，`value` 是 `char[]` 类型；JDK 9 之后，根据字符串内容选择 `byte[]` 或 `char[]`。

`final` 数组意味着一旦数组引用被初始化，就不能指向另一个数组。但这里有一个微妙的问题：`final` 只保证引用不变，不保证数组内容不变。数组内的元素仍然可以被修改。

### 第二道防线：内容的保护

为了防止数组内容被修改，String 类采取了以下措施：

1. **没有提供公开的修改方法**：String 类中没有任何方法可以修改 `value` 数组的内容。所有返回 String 的方法（如 `substring`、`concat`、`replace` 等）都是创建并返回一个新的 String 对象。

2. **value 数组的来源受控**：外部无法直接传入数组来初始化 String。所有的构造方法要么复制传入的数组内容，要么从其他来源构建新的数组。

3. **JDK 9+ 的额外保护**：在 JDK 9 中，String 内部使用了更底层的实现，引入了 `coder` 字段来标记编码方式，并且相关的修改操作被封装在更底层的类中，进一步降低了意外修改的风险。

### String 构造方法的细节

让我们看看 String 类的一些关键构造方法，了解它是如何确保内容不被外部影响的：

```java
// 从另一个 String 复制内容
public String(String original) {
    this.value = original.value;
    this.coder = original.coder;
    this.hash = original.hash;
}

// 从 char 数组复制内容（而非直接引用）
public String(char value[]) {
    this.value = Arrays.copyOf(value, value.length);
    this.coder = CompactStringSupport.put(this.value, -1);
}

// JDK 9+ 的 byte 数组构造方法
String(byte[] value, byte coder) {
    this.value = value;  // 注意：这里直接赋值，但外部无法访问
    this.coder = coder;
}
```

在 char 数组版本的构造方法中，我们可以看到它使用 `Arrays.copyOf()` 复制了一份数组，而不是直接引用传入的数组。这意味着即使调用者后来修改了传入的数组，String 对象内部的内容也不会受影响。

但为什么 String(String original) 构造方法没有复制数组呢？这是因为源 String 也是不可变的，所以不存在修改风险。

## 不可变性的"副作用"

String 的不可变性虽然带来了诸多好处，但也造成了一些"奇怪"的行为，新手可能会感到困惑。

### 字符串拼接的效率问题

```java
String result = "";
for (int i = 0; i < 1000; i++) {
    result += "item" + i;  // 每次拼接都创建新对象！
}
```

在循环中频繁拼接字符串是非常低效的，因为每次 `+=` 操作都会创建一个新的 String 对象。这是因为 String 不可变，无法在原字符串上追加内容。很多新手会在循环中这样写，导致性能问题。

正确的做法是使用 StringBuilder：

```java
StringBuilder sb = new StringBuilder();
for (int i = 0; i < 1000; i++) {
    sb.append("item").append(i);
}
String result = sb.toString();
```

### 字符串反转的问题

由于 String 是不可变的，反转字符串需要创建新对象：

```java
String original = "hello";
String reversed = new StringBuilder(original).reverse().toString();
```

String 类没有提供 `reverse()` 方法，因为那样做会违反不可变性原则。

## 反射可以破坏不可变性？

你可能听说过通过反射可以修改 String 的内容：

```java
String s = "hello";
Field valueField = String.class.getDeclaredField("value");
valueField.setAccessible(true);
byte[] value = (byte[]) valueField.get(s);
value[0] = 'H';  // 修改为大写
System.out.println(s);  // 输出 "Hello"！
```

这确实是真的！通过反射可以绕过封装，修改 String 内部 byte/char 数组的内容。但这并不是 String 设计的缺陷，而是一种必要的权衡。反射在框架、调试、序列化等场景中有重要用途，完全禁止反射会失去很大的灵活性。而且，这种破坏不可变性的行为本身就是不良实践，应该避免。

<Badge text="最佳实践" type="tip" /> 虽然反射可以修改 String，但这应该被视为"危险操作"而非"设计缺陷"。在实际开发中，永远不要依赖或假设反射会修改 String。String 的不可变性仍然是值得信赖的契约。

## 不可变性的哲学思考

String 的不可变性设计体现了一种深刻的软件工程哲学：通过限制灵活性来获得确定性和安全性。这是一种"opinionated"的设计——String 的设计者认为不可变性比可变性更重要，因此做出了明确的选择。

在现代 Java 编程中，不可变对象越来越受到重视。从 Java 14 的 records 到现代函数式编程中的数据类，不可变性已经成为一种被广泛认可的最佳实践。String 作为 Java 中最古老的不可变类型之一，可以说开创了这一趋势的先河。

当我们讨论 String 的不可变性时，我们不仅仅是在讨论一个技术细节，更是在讨论一种编程思想：通过限制可能的状态变化，让程序更容易理解和推理。
