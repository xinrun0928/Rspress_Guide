# Java 核心

候选人小王坐在字节跳动的面试间，面试官翻了翻他的简历，问了一句：

"看你简历上写了精通 Java，那 HashMap 的 put 流程说一下。"

小王背出了四步：计算 hash、找到桶、插入链表/红黑树、判断扩容。面试官点点头，继续追问："JDK8 为什么引入红黑树？链表达到多少长度会树化？"

小王说："好像是 8？"

面试官没说话，又问："什么时候树化，什么时候退链表？红黑树的查找时间复杂度是多少？"

小王停顿了三秒，开始语无伦次...

**这一幕每天都在发生。** HashMap 能背出来的人占 80%，能答出树化阈值的占 30%，能说清楚为什么选择 8 而不是 16 的只有 5%。

这道题，就是 Java 面试最好的筛选器——它能一眼看出你是"背八股"还是"真做过"。

【面试官心理】
我问他 put 流程，其实不是想听背书。我想知道的是：他有没有亲手看过源码，能不能理解 JDK8 为什么要引入红黑树。链表阈值 8、树化条件、链表退化的平衡——这才是 P6 和 P5 的差距所在。

## 模块详解

### Java 基础 `java/basic/`

很多人觉得 Java 基础太简单，不值得花时间。但我告诉你，**越基础的东西越容易被忽略，越容易被忽略的问题越容易在面试中翻车。**

比如：
- `String` 为什么是不可变的？`StringBuilder` 和 `StringBuffer` 的区别是什么？
- `==` 和 `equals` 的区别？`hashCode` 和 `equals` 的关系？
- Java 是值传递还是引用传递？
- `try`、`catch`、`finally` 的执行顺序？`finally` 里 return 会发生什么？

这些问题看起来简单，但面试官的追问可以让你原形毕露。

核心文章：
- [String 源码解析](/java/basic/string) —— 不可变、设计原因、拼接优化
- [equals 与 hashCode](/java/basic/equals) —— 二者关系与集合中的实际作用
- [值传递还是引用传递](/java/basic/pass-by-value) —— 这个话题能刷掉 50% 的候选人
- [OOP 四大特性](/java/basic/oop) —— 封装/继承/多态/抽象的深度理解
- [异常体系](/java/basic/exception) —— Checked/Unchecked/Runtime 的区别
- [泛型](/java/basic/generic) —— 类型擦除、协变逆变、泛型擦出补偿

### 集合框架 `java/collection/` 🔴

集合是 Java 面试的半壁江山。HashMap 家族、List 家族、Set 家族、Queue 家族——每个家族都有自己的设计哲学和性能特点。

HashMap 单独拿出来说：这道题我平均每场面试都会问，因为它足够简单到每个人都能说两句，又足够复杂到能追问到源码细节。

**HashMap 追问链（能答到第四层的都是 P6+）**：
1. put 流程是什么？
2. JDK8 为什么引入红黑树？链表阈值是多少？
3. 什么时候树化？什么时候退链表？为什么退链表阈值是 6 不是 8？
4. 你在项目里用过什么替代方案？什么场景下 HashMap 不是最优解？

核心文章：
- [HashMap 源码深度解析](/java/collection/hashmap) —— put/get/扩容/树化，全流程逐行解读
- [HashMap 扩容机制](/java/collection/hashmap-resize) —— JDK7 vs JDK8 扩容差异，死循环问题
- [HashMap 树化与退化](/java/collection/hashmap-treeify) —— 红黑树阈值 8/6 的设计原因
- [ConcurrentHashMap](/java/collection/concurrent-hashmap) —— JDK7 分段锁 vs JDK8 CAS + synchronized
- [ConcurrentHashMap put 流程](/java/collection/concurrent-hashmap-put) —— 线程安全的实现细节
- [ConcurrentHashMap 扩容](/java/collection/concurrent-hashmap-resize) —— JDK8 并发扩容核心机制
- [ArrayList vs LinkedList](/java/collection/list-comparison) —— 随机访问 vs 插入删除的性能差异
- [Map 横向对比](/java/collection/map-comparison) —— HashMap/TreeMap/LinkedHashMap/ConcurrentHashMap 选型

### 并发编程 `java/concurrent/` 🔴

并发编程是 P6 和 P5 的分水岭。能说清楚线程安全问题的候选人占 40%，能说清楚 JMM 和 happens-before 的占 20%，能写好一个线程池并且说出核心参数的占 10%。

**并发编程三座大山**：
1. **线程安全**：synchronized/CAS/volatile 的底层实现
2. **JMM**：happens-before 规则、内存屏障、指令重排
3. **并发工具**：AQS/ReentrantLock/ThreadLocal/线程池

核心文章：
- [synchronized 原理](/java/concurrent/synchronized) —— 锁升级、Mark Word、偏向锁/轻量级锁/重量级锁
- [JMM 与 happens-before](/java/concurrent/happens-before) —— 理解可见性、有序性、原子性的本质
- [CAS 原理](/java/concurrent/cas) —— ABA 问题、乐观锁的优缺点
- [AQS 核心原理](/java/concurrent/aqs) —— 队列同步器的设计哲学
- [ReentrantLock](/java/concurrent/reentrantlock) —— 可重入锁、公平锁/非公平锁
- [ThreadLocal](/java/concurrent/threadlocal) —— 线程本地变量、内存泄漏问题
- [线程池参数](/java/collection/threadpool-params) —— 核心参数、拒绝策略、线程池调优
- [CompletableFuture](/java/concurrent/completable-future) —— 异步编程与编排

### IO/NIO `java/io/`

IO 模型这块，很多后端工程师用了很久 Netty，却说不清楚 Epoll 和 Select 的区别。说不出 Epoll 的三种事件模式，说不出 LT 和 ET 的差异——这种水平去面字节、拼多多这种对底层要求高的公司，基本就是送命题。

核心文章：
- [IO 模型对比](/java/io/io-comparison) —— 阻塞/非阻塞/异步，Linux 五种 IO 模型全解析
- [NIO 核心组件](/java/io/nio) —— Buffer/Channel/Selector 的设计与使用
- [Netty Reactor 模型](/java/io/netty-reactor) —— 单线程/多线程/主从多线程模型对比
- [Netty 核心组件](/java/io/netty-components) —— ByteBuf、ChannelHandler、Pipeline
- [零拷贝](/java/io/zero-copy) —— mmap/SendFile/DMA 拷贝，Kafka 为什么要用零拷贝

### JVM `java/jvm/` 🔴

JVM 是 Java 面试的最后一关。GC 算法、类加载、运行时数据区、OOM 排查——每一条都能追问到生产级别。

**JVM 经典追问链**：
1. 对象的创建过程？（内存分配 + 字节码执行）
2. 对象的内存布局？（对象头 + 实例数据 + 对齐填充）
3. GC roots 有哪些？
4. 常见的垃圾回收算法？
5. G1 和 ZGC 的区别？分代收集和分区收集的本质差异？
6. 线上 OOM 怎么排查？用什么工具？

核心文章：
- [JVM 运行时数据区](/java/jvm/runtime-data-area) —— 堆/栈/方法区/本地方法栈完整解析
- [对象的创建过程](/java/jvm/object-creation) —— 字节码执行、内存分配、对象头布局
- [GC Roots 与可达性分析](/java/jvm/gc-roots) —— 哪些对象不能被回收
- [垃圾回收算法](/java/jvm/gc-algorithms) —— 标记清除/复制/标记整理，各代际的权衡
- [G1 垃圾收集器](/java/jvm/g1) —— 分区收集、并发标记、RSet 记忆集
- [ZGC](/java/jvm/zgc) —— 着色指针、读屏障、亚毫秒级停顿
- [类加载机制](/java/jvm/class-loading) —— 双亲委派、破坏双亲委派、自定义类加载器
- [OOM 排查](/java/jvm/oom) —— Heap/Stack/Metaspace/-direct-buffer-memory 的排查方法

### Java 新特性 `java/new-features/`

Lambda 和 Stream 是 JDK8 的标志性特性，不会用的直接扣分。JDK17 的 Sealed Class、Record、Pattern Matching 则是 P7 面试的加分项。

核心文章：
- [Lambda 表达式](/java/new-features/lambda) —— 函数式接口、类型推断、闭包
- [Stream API](/java/new-features/stream) —— 惰性求值、并行流、内部迭代
- [Optional](/java/new-features/optional) —— 消灭 NPE 的函数式方案
- [Record](/java/new-features/record) —— JDK16 正式版，数据载体的简洁写法
- [Sealed Class](/java/new-features/sealed-class) —— JDK17 限制类的继承层级

## 面试关联与进阶

Java 核心知识和分布式/数据库知识紧密相连：

- **HashMap + 红黑树** → [MySQL InnoDB 索引](/database/mysql/index) —— B+ 树是 HashMap 树化的工程级扩展
- **线程安全 + JMM** → [Redis 线程模型](/redis/threading) —— Redis 为什么是单线程又高性能
- **GC + 内存布局** → [Redis 内存淘汰策略](/redis/eviction) —— 主动淘汰 vs 被动淘汰
- **类加载 + 反射** → [Spring IOC 原理](/spring/ioc) —— Bean 的实例化与依赖注入
- **NIO + Epoll** → [Netty 架构](/java/io/netty-reactor) —— 高性能网络框架的底层支撑

## 面试官忠告

学习 Java 核心，有一个最基本的原则：**看源码，不要背答案。**

HashMap 的 putVal 方法不过 100 行，你花一个小时把它读一遍，比看十篇"HashMap 面试题大全"都有用。

JDK 设计者写的每一行代码，背后都有权衡和取舍。能说出"为什么"的，才是真正理解了的。

> **记住**：面试官不是在考你记忆力，是在考你理解深度。能用自己的话把原理讲清楚，还能在源码里找到对应行——这才是 P6+ 的标准。
