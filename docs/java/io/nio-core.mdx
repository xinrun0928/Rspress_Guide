# NIO 核心组件：Channel、Buffer、Selector

# 如果把NIO比作一个厨房，Channel是传送带，Buffer是操作台，Selector是大堂经理——传送带负责把原材料送来，操作台负责加工，大堂经理负责知道什么时候该处理哪份订单。

BIO的I/O模型是「流」——InputStream、OutputStream，像水管一样单向流动。NIO的核心创新是**通道（Channel）**和**缓冲区（Buffer）**的抽象，把单向流变成了双向通道，把「流动的数据」变成了「可以随机访问的内存块」。

这个转变看似简单，实际上带来了一场范式革命。BIO中，数据是「流过」的，必须边读边处理，如果想回头重读同一段数据，不可能——水已经流走了。但在NIO中，数据先进入Buffer，你可以停在任意位置，任意回退，反复读取同一块数据。这就是「缓冲区」的意义：**数据的中转站，让你有机会停下来思考和处理**。

Selector则是NIO处理高并发的关键。它让你能用一个线程监控成千上万个连接的状态变化，而不是为每个连接分配一个线程。

## Channel：数据的双向高速公路

Channel（通道）是NIO的基础，几乎所有I/O操作都离不开它。与BIO的Stream不同，Channel是**双向的**——一个Channel可以同时用于读写，不需要分别创建输入流和输出流。

### Channel的类型

Java NIO的Channel体系有几个关键类型：

```
Channel
├── SelectableChannel（可选择通道，用于Selector）
│   ├── SocketChannel（TCP套接字，客户端/服务端）
│   ├── ServerSocketChannel（服务端监听通道）
│   └── DatagramChannel（UDP套接字）
└── FileChannel（文件I/O）
```

**SocketChannel**：代表TCP连接的客户端一侧，用于连接远程服务器、读写数据。

```java
SocketChannel channel = SocketChannel.open();
channel.connect(new InetSocketAddress("localhost", 8080));
ByteBuffer buffer = ByteBuffer.allocate(1024);
channel.read(buffer);   // 读数据
channel.write(buffer);  // 写数据
```

**ServerSocketChannel**：服务端监听通道，绑定端口，接受连接请求。

```java
ServerSocketChannel serverChannel = ServerSocketChannel.open();
serverChannel.socket().bind(new InetSocketAddress(8080));
serverChannel.configureBlocking(false);  // 非阻塞模式
SocketChannel clientChannel = serverChannel.accept();  // 返回null如果无连接
```

**FileChannel**：文件I/O通道，支持文件锁、内存映射等高级特性。

```java
FileInputStream fis = new FileInputStream("data.txt");
FileChannel channel = fis.getChannel();
ByteBuffer buffer = ByteBuffer.allocate(1024);
channel.read(buffer);
```

### FileChannel的特殊能力

FileChannel不只是「能用Selector的文件Channel」，它还有一些独特能力：

**文件锁**：

```java
FileChannel channel = new RandomAccessFile("data.txt", "rw").getChannel();
FileLock lock = channel.lock();  // 获取独占锁
// ... 操作文件 ...
lock.release();  // 释放锁
```

文件锁可以用来协调多进程对同一文件的访问。注意：`lock()`会阻塞，如果文件已被其他进程持有独占锁，会等待。

**内存映射文件**：

```java
FileChannel channel = new RandomAccessFile("data.txt", "rw").getChannel();
MappedByteBuffer mappedBuffer = channel.map(
    FileChannel.MapMode.READ_WRITE,  // 读写模式
    0,                                 // 文件起始位置
    channel.size()                     // 映射区域大小
);
mappedBuffer.put(0, (byte) 'X');  // 像操作普通ByteBuffer一样
mappedBuffer.force();  // 强制刷盘
```

内存映射将文件直接映射到内存，避免了传统的read/write系统调用。OS会自动管理哪些页在内存中、哪些在磁盘上。适合处理大文件，但要注意32位系统的寻址空间限制。

### Channel的connect、read、write

Channel的核心操作只有三个，但理解它们与Buffer的配合至关重要：

```java
// 连接（客户端）
channel.connect(new InetSocketAddress("example.com", 80));

// 读取：数据从Channel进入Buffer
int bytesRead = channel.read(buffer);  // 返回读取的字节数，-1表示EOF

// 写入：数据从Buffer进入Channel
int bytesWritten = channel.write(buffer);
```

注意`read()`和`write()`都返回实际传输的字节数。在非阻塞模式下，这个数字可能小于请求的数量，你需要自己处理「部分读写」的情况。

## Buffer：可读写的数据容器

Buffer是NIO的核心数据结构。它是一个**可以写入数据、然后从中读取数据的内存块**。与普通数组不同，Buffer有位置（position）、限制（limit）、容量（capacity）三个指针，这让它能高效地在「写入模式」和「读取模式」之间切换。

### Buffer的三个指针

```
容量(Capacity)：缓冲区最大容量，创建后不变
    +-------------------------------------------------+
    |                                                 |
    v                                                 v
    +-----+-----+-----+-----+-----+-----+-----+-----+
    |  H  |  e  |  l  |  l  |  o  |     |     |     |
    +-----+-----+-----+-----+-----+-----+-----+-----+
              ^
              |
        位置(Position)：下一个读写的位置
              ^
              |
        限制(Limit)：第一个不能读/写的位置
```

- **capacity（容量）**：Buffer能容纳的最大数据量，创建后不变
- **position（位置）**：当前读写位置。写入时position前进，读取时也从position开始
- **limit（限制）**：在写模式下，limit等于capacity；在读模式下，limit表示能读取多少数据

这个设计看起来有点绕，但背后的逻辑很优雅：**同一个Buffer可以复用，不需要创建两个独立的对象**。

### Buffer的生命周期：写入 -> 翻转 -> 读取 -> 清空

```java
ByteBuffer buffer = ByteBuffer.allocate(1024);

// 阶段1：写入数据
buffer.put((byte) 'H');
buffer.put((byte) 'i');
buffer.put((byte) '!');
// position = 3, limit = 1024

// 阶段2：翻转（flip）- 从写切换到读
buffer.flip();
// position = 0, limit = 3（只能读到之前写入的3个字节）

// 阶段3：读取数据
while (buffer.hasRemaining()) {
    byte b = buffer.get();
    System.out.print((char) b);
}
// position = 3, limit = 3

// 阶段4：清空（clear）- 准备重新写入
buffer.clear();
// position = 0, limit = 1024
```

**flip()** 做了什么？`limit = position; position = 0`。它记住了「写入到哪里」，然后把position重置为0，把limit设置到写入位置。这样读取时就只会读到之前写入的数据，而不是整个capacity。

**clear()** 做了什么？`position = 0; limit = capacity`。它不删除数据，只是重置指针，让你重新写入同一块Buffer。之前写入的3个字节其实还在那里，只是下次写入时会覆盖它们。

### Buffer的类型：不只是ByteBuffer

Java NIO提供了多种Buffer类型，对应不同的原始数据类型：

- `ByteBuffer`：最常用，可以手动分配（`allocate()`）或直接分配（`allocateDirect()`）
- `CharBuffer`、`ShortBuffer`、`IntBuffer`、`LongBuffer`、`FloatBuffer`、`DoubleBuffer`：对应其他原始类型
- `MappedByteBuffer`：内存映射文件

此外，还有`HeapByteBuffer`和`DirectByteBuffer`之分。HeapByteBuffer在JVM堆上分配，GC会管理；DirectByteBuffer在堆外内存分配，不受GC影响，但创建和销毁成本更高。DirectByteBuffer的详细内容，我们会在专门的文章中讨论。

### 创建Buffer的方式

```java
// 堆上Buffer（推荐大多数场景）
ByteBuffer heapBuffer = ByteBuffer.allocate(1024);

// 直接Buffer（适合I/O操作，减少一次内存复制）
ByteBuffer directBuffer = ByteBuffer.allocateDirect(1024);

// 从已有数组包装（不复制数据）
byte[] array = new byte[1024];
ByteBuffer wrappedBuffer = ByteBuffer.wrap(array);
```

### Buffer的压缩：compact()

有时候你只读取了部分数据，还想保留剩余数据继续使用。`compact()`方法帮你做到这一点：

```java
// 假设已经读取了buffer中前5个字节
buffer.flip();
buffer.get(new byte[5]);

// 现在想继续读取剩余数据
buffer.compact();  // 将未读数据复制到buffer开头
// position = limit - 5（未读数据的长度）
// limit = capacity
```

`compact()`会把你没读完的数据「压缩」到Buffer开头，然后你可以在后面继续写入新数据。这对于边读边写的场景非常有用。

## Selector：单线程管理多连接的秘密

Selector（选择器）是NIO处理高并发的核心。它允许**一个线程监控多个Channel的I/O状态**（可读、可写、连接完成等），当某个Channel就绪时才处理，避免了轮询的开销和阻塞的低效。

### Selector的创建与注册

```java
Selector selector = Selector.open();  // 创建Selector

SocketChannel channel = SocketChannel.open();
channel.configureBlocking(false);  // 必须是非阻塞模式
channel.register(selector, SelectionKey.OP_READ);  // 注册监控事件
```

**关键约束**：Selector只能监控非阻塞的Channel。如果一个Channel处于阻塞模式，`register()`会抛出`IllegalBlockingModeException`。

### SelectionKey：代表注册关系

当你把一个Channel注册到Selector时，会返回一个`SelectionKey`。这个key代表了「Channel + Selector + 感兴趣的事件」的组合：

```java
SelectionKey key = channel.register(selector, SelectionKey.OP_READ);
```

**SelectionKey可以附加一个对象**：

```java
key.attach(new MyHandler());  // 附加业务对象
MyHandler handler = (MyHandler) key.attachment();  // 获取附加对象
```

这是Reactor模式实现的关键：你可以在注册时就附加一个处理器对象，当事件就绪时取出处理。

### 事件类型

SelectionKey定义了四种事件：

| 事件 | 值 | 含义 |
|---|---|---|
| OP_READ | `1 << 0` | Channel中有数据可读 |
| OP_WRITE | `1 << 2` | Channel可以写数据 |
| OP_CONNECT | `1 << 3` | TCP连接完成（客户端） |
| OP_ACCEPT | `1 << 4` | 有新连接等待接受（服务端） |

可以组合多个事件：

```java
channel.register(selector, SelectionKey.OP_READ | SelectionKey.OP_WRITE);
```

### select()方法：等待事件

Selector的核心方法是`select()`系列：

```java
int readyCount = selector.select();      // 阻塞直到有事件就绪
int readyCount = selector.select(1000);  // 阻塞最多1秒
int readyCount = selector.selectNow();    // 非阻塞，立即返回
```

返回值为就绪事件的数量。**注意**：返回0不代表没有事件，可能只是超时。

### 处理事件：完整示例

```java
while (true) {
    int ready = selector.select();  // 阻塞等待
    if (ready == 0) continue;

    Set<SelectionKey> keys = selector.selectedKeys();
    Iterator<SelectionKey> iter = keys.iterator();

    while (iter.hasNext()) {
        SelectionKey key = iter.next();

        if (key.isAcceptable()) {
            // OP_ACCEPT：接受新连接
            ServerSocketChannel ssc = (ServerSocketChannel) key.channel();
            SocketChannel sc = ssc.accept();
            sc.configureBlocking(false);
            sc.register(selector, SelectionKey.OP_READ);
        }

        if (key.isReadable()) {
            // OP_READ：读取数据
            SocketChannel sc = (SocketChannel) key.channel();
            ByteBuffer buffer = ByteBuffer.allocate(1024);
            int read = sc.read(buffer);
            if (read > 0) {
                buffer.flip();
                // 处理数据...
            } else if (read == -1) {
                sc.close();
            }
        }

        if (key.isWritable()) {
            // OP_WRITE：写入数据
            // 通常不需要显式注册OP_WRITE
            // 当Channel可写时通常直接写
        }

        iter.remove();  // 处理完必须移除，否则会重复处理
    }
}
```

### wakeUp()：强制select()返回

有时候你需要在另一个线程中唤醒Selector：

```java
// 线程A：正在select()阻塞
Selector selector = ...;
selector.select();

// 线程B：通知线程A有新任务
selector.wakeup();  // 让select()立即返回
selector.select();  // 重新进入阻塞
```

`wakeup()`会唤醒一个阻塞的`select()`调用，即使实际上没有事件发生。之后再调用`select()`会重新阻塞。

### close()：正确关闭Selector

使用完Selector后必须关闭：

```java
selector.close();
```

这会关闭Selector，并使其上注册的SelectionKey全部失效。但**不会关闭相关的Channel**——Channel需要你自己关闭。

## Selector的内部原理：epoll在Java中的化身

Selector在不同操作系统上有不同的实现：

- Linux 2.6+：epoll（高效，O(1)复杂度）
- macOS/BSD：kqueue
- Windows：IOCP（通过选择性的模拟层）

<Badge text="JDK 11+" type="tip" /> 在Linux上默认使用epoll，之前版本使用poll。

从JDK 11开始，使用epoll时默认启用**EPOLLLEVEL_TRIGGERED**（水平触发）和**EPOLLLONESHOT**。当Channel数据就绪时，会通知多次，知道数据被完全消费。之前的JDK版本使用poll机制，行为类似但效率略低。

## 常见陷阱与最佳实践

### 陷阱1：忘记移除SelectionKey

```java
iter.remove();  // 这行必须写！
```

如果不移除，下次`select()`返回时，这个key还在集合里，会被重复处理。

### 陷阱2：在多线程环境下错误地使用Selector

Selector本身是线程安全的，但**SelectionKey的cancel()方法是在Selector线程中延迟执行的**。如果你在一个线程中`key.cancel()`，Selector.close()需要等到下一次select才会真正注销这个key。这可能导致资源泄漏。

### 陷阱3：ByteBuffer没有flip就读取

```java
ByteBuffer buffer = ByteBuffer.allocate(1024);
channel.read(buffer);
// 你以为现在能读了？错！
// position还在0，数据写在buffer里但你读不出来
buffer.flip();  // 必须先flip
byte b = buffer.get();  // 现在才能读到第一个字节
```

### 最佳实践：用DirectBuffer减少复制

对于网络I/O，使用DirectByteBuffer可以**跳过堆外内存到堆内内存的一次复制**：

```java
// 普通Buffer：Channel -> 堆外 -> 堆内
ByteBuffer heapBuffer = ByteBuffer.allocate(1024);
channel.read(heapBuffer);  // 先读入堆外，然后复制到堆内

// 直接Buffer：Channel -> 堆外
ByteBuffer directBuffer = ByteBuffer.allocateDirect(1024);
channel.read(directBuffer);  // 直接读入堆外，少一次复制
```

不过DirectBuffer的创建成本高，应该**复用**而不是每次读写都创建：

```java
// 复用DirectBuffer
ByteBuffer buffer = directBuffer.get();
buffer.clear();  // 重置position，不是释放
```

### 最佳实践：用SelectionKey附加业务状态

```java
// 注册时附加处理器
SelectionKey key = channel.register(selector, SelectionKey.OP_READ);
key.attach(new HttpHandler());

// 事件处理时取出
HttpHandler handler = (HttpHandler) key.attachment();
handler.handle(buffer);
```

这样每个连接都有自己的处理器对象，代码更清晰。

## Channel、Buffer、Selector的协作流程

```
                    +-----------------+
                    |    Selector     |
                    |   (事件监控)     |
                    +--------+--------+
                             |
           +-----------------+-----------------+
           |                 |                 |
      OP_ACCEPT         OP_READ          OP_WRITE
           |                 |                 |
           v                 v                 v
    +------------+   +-------------+   +----------------+
    | ServerSocket|   | SocketChannel|   | SocketChannel  |
    |  Channel    |   |  (读缓冲区)   |   |  (写缓冲区)     |
    +-----+------+   +------+------+   +--------+-------+
          |                 |                     |
          +-----------------+---------------------+
                             |
                             v
                    +-----------------+
                    |    Buffer      |
                    | (数据操作台)    |
                    +-----------------+
```

Channel负责与外部（网络、文件）交换数据；Buffer是数据暂存的容器；Selector负责监控多个Channel的状态，事件就绪时通知应用处理。
