## 一、序列化的本质与必要性 🔴

### 1.1 什么是序列化

序列化：**将内存中的对象结构转换成字节流**，用于网络传输或持久化存储。反序列化是逆过程。

```
对象（内存结构）  ──序列化──>  字节流（网络/磁盘）
对象（内存结构）  <──反序列化──  字节流（网络/磁盘）
```

**为什么需要序列化？**
- 网络传输：对象→字节流→网络→字节流→对象
- 持久化：对象→字节流→磁盘→字节流→对象

### 1.2 序列化与编码的区别

很多人混淆"序列化"和"编码"：

- **序列化**：对象结构→字节序列（决定怎么把对象"拍平"成字节）
- **编码**：字节→可读字符（UTF-8、GBK等，决定字节如何表示字符）

```
JSON序列化："name" + ":" + "张三"
           ↓ 编码（UTF-8）
           [110, 97, 109, 101, 58, 229, 133, 133, 229, 156, 168]
```

## 二、五大序列化协议对比 🔴

### 2.1 核心维度对比

| 维度 | Java序列化 | JSON | ProtoBuf | Hessian | Kryo |
| --- | --- | --- | --- | --- | --- |
| **序列化后体积** | 大（含元数据） | 大（文本冗余） | **极小**（二进制紧凑） | 小 | 小 |
| **序列化速度** | 慢 | 中等 | **极快** | 快 | **极快** |
| **反序列化速度** | 慢 | 中等 | **极快** | 快 | **极快** |
| **跨语言支持** | 仅Java | 全部语言 | 全部语言 | Java/Flash/C++ | Java为主 |
| **可读性** | ❌ 不可读 | ✅ 人类可读 | ❌ 二进制 | ❌ 二进制 | ❌ 二进制 |
| **IDL支持** | ❌ 无 | ❌ 无 | ✅ .proto定义 | ❌ 无 | ❌ 无 |
| **版本兼容性** | ❌ 差 | ✅ 好 | ✅ 好 | ✅ 好 | ⚠️ 一般 |
| **大小敏感场景** | ❌ 不适用 | ❌ 带宽浪费 | ✅ 首选 | ✅ 推荐 | ✅ 推荐 |
| **调试友好场景** | ❌ | ✅ 首选 | ❌ | ❌ | ❌ |

### 2.2 量化性能数据

以下是典型性能对比（仅供参考，取决于具体实现和数据结构）：

| 操作 | Java序列化 | JSON | ProtoBuf | Hessian |
| --- | --- | --- | --- | --- |
| 序列化1000次耗时 | ~500ms | ~200ms | ~50ms | ~80ms |

| 序列化后大小 | 100% | 150% | **30%** | **40%** |

面试加分点：能说出JSON的"Human-readable"不是免费的——文本格式意味着大量冗余字符（引号、冒号、花括号），序列化后体积是ProtoBuf的3-5倍，在高QPS场景下浪费大量带宽。

## 三、Java原生序列化 🟡

### 3.1 使用方式

```java
// 必须实现 Serializable 接口
public class User implements Serializable {
    private static final long serialVersionUID = 1L;

    private String name;
    private int age;

    // 序列化
    ByteArrayOutputStream baos = new ByteArrayOutputStream();
    ObjectOutputStream oos = new ObjectOutputStream(baos);
    oos.writeObject(user);
    byte[] bytes = baos.toByteArray();

    // 反序列化
    ObjectInputStream ois = new ObjectInputStream(new ByteArrayInputStream(bytes));
    User u = (User) ois.readObject();
}
```

### 3.2 ❌ 错误示范

**候选人原话**："Java序列化简单，ObjectOutputStream直接用就行了。"

**问题诊断**：
- Java序列化的serialVersionUID如果不声明，JVM会自动生成，类结构一变（如加字段），UID就变了，反序列化直接抛`InvalidClassException`。
- 序列化后包含类元数据（类名、包名、字段签名），体积大、跨语言不兼容。
- 性能差，不适合高性能场景。

【面试官心理】
Java序列化是JDK自带的最方便的方案，但也是生产中最不推荐的方案。90%的性能问题都与它有关。能说出serialVersionUID机制、writeObject/readObject自定义序列化、Java序列化的安全风险的候选人，才是真的用过。

### 3.3 Java序列化的深坑

**坑1：serialVersionUID不一致**

```java
// ❌ 不声明UID：类改动后反序列化必崩
public class User implements Serializable {
    private String name;
    private int age;
    // JVM自动生成UID = hash(类结构)
    // 加一个字段 → UID变了 → InvalidClassException
}

// ✅ 声明UID：向前兼容
public class User implements Serializable {
    private static final long serialVersionUID = 1L;
    // 加字段后UID不变，可以兼容
    private String name;
    private int age;
    private String email; // 新增字段，旧数据反序列化时email为null
}
```

**坑2：安全风险——反序列化漏洞**

```java
// ❌ 危险：未校验的反序列化可以执行任意代码
ObjectInputStream ois = new ObjectInputStream(inputStream);
Object obj = ois.readObject(); // 恶意构造的字节流可以执行任意代码！

// ✅ 正确：使用白名单或禁用Java序列化
ObjectInputFilter filter = ObjectInputFilter.Config.createFilter("com.example.*;! *");
ois.setObjectInputFilter(filter);
```

## 四、JSON序列化 🟡

### 4.1 常见实现与对比

| 实现 | 特点 | 适用场景 |
| --- | --- | --- |
| Jackson | 功能最全，性能好 | Spring默认 |
| Fastjson | 快但安全漏洞多 | 内部系统 |
| Gson | 谷歌出品，稳定 | 通用场景 |

```java
// Jackson（Spring默认）
ObjectMapper mapper = new ObjectMapper();
String json = mapper.writeValueAsString(user);
User u = mapper.readValue(json, User.class);

// Fastjson
String json = JSON.toJSONString(user);
User u = JSON.parseObject(json, User.class);

// Gson
Gson gson = new Gson();
String json = gson.toJson(user);
User u = gson.fromJson(json, User.class);
```

### 4.2 JSON的优缺点

**优点**：
- Human-readable，调试友好
- 跨语言，生态完善
- Schema-less，字段增减灵活

**缺点**：
- 序列化后体积大（大量引号、冒号、花括号）
- 性能相对较差
- 日期类型没有标准格式
- 没有IDL，需要手写字段映射

:::warning ⚠️
Fastjson安全漏洞：Fastjson历史上出现过多个反序列化漏洞（RCE），生产环境建议使用Jackson或Gson，或升级到Fastjson 2.x。

## 五、ProtoBuf——性能标杆 🔴

### 5.1 ProtoBuf的核心原理

ProtoBuf（Protocol Buffers）使用IDL（接口定义语言）定义数据结构，编译生成代码：

```protobuf
// user.proto
syntax = "proto3";

message User {
    string name = 1;     // tag=1
    int32 age = 2;       // tag=2
    string email = 3;    // tag=3 (新增字段，tag=3)
}
```

### 5.2 ProtoBuf的编码原理——为什么这么快

ProtoBuf使用**TLV（Tag-Length-Value）**编码，每个字段的格式：

```
[Tag] [Length] [Value]
 ↑       ↑        ↑
 字段ID + 类型   数据长度   实际数据
```

**核心优化1：Tag-Length-Value紧凑编码**

```java
// ProtoBuf的int32不需要完整的4字节
// 使用varint编码：小数字用1字节，大数字用更多字节
// 示例：age=10 → [0x0A] (1字节)
// Java序列化：age=10 → [0x00 0x00 0x00 0x0A] (4字节)

Tag编码：
  field_number(3 bits) + wire_type(2 bits) = 1字节
  name: tag = 1 << 3 | 2 = 0x0A (varint类型，string/bytes用wire_type=2)
```

**核心优化2：字段顺序写入，无需字段名**

```
Java序列化后的字节：
  [类名][字段名1][值1][字段名2][值2]...  ← 包含大量字符串

ProtoBuf序列化后的字节：
  [tag1][value1][tag2][value2]...  ← 只有tag和值，没有字段名！
```

### 5.3 ProtoBuf的版本兼容性

这是ProtoBuf最强大的特性：

```protobuf
// 初始版本
message User {
    string name = 1;
    int32 age = 2;
}

// 升级版本：添加email字段，删除phone字段
message User {
    string name = 1;
    int32 age = 2;
    string email = 3;  // 新增，tag=3，tag不能复用！
    // phone字段被删除，tag=4不能再用（除非等所有老数据消失）
}
```

**ProtoBuf版本兼容性规则**：
1. **新增字段**：tag必须用新的数字，旧代码会忽略新字段
2. **删除字段**：tag不能复用（否则旧代码会把新数据当旧字段解析）
3. **字段改名**：不影响序列化（序列化时不写字段名）
4. **tag-number不能改变**：改变tag会破坏兼容性

### 5.4 ProtoBuf在gRPC中的应用

```java
// Maven引入（从.proto文件生成）
// mvn generate-sources -> 生成 UserProto.java

// 序列化
UserProto.User user = UserProto.User.newBuilder()
    .setName("张三")
    .setAge(30)
    .build();
byte[] bytes = user.toByteArray();

// 反序列化
UserProto.User u = UserProto.User.parseFrom(bytes);
```

:::tip 💡
面试加分点：能说出ProtoBuf的tag-number机制——字段用数字标记而非名称，数字对应固定的类型和位置。这样序列化时只写tag+值，接收方根据tag就知道字段含义，比JSON的字段名字符串高效得多。

## 六、Hessian序列化 🟢

### 6.1 Hessian的特点

Hessian是Adobe公司开源的序列化协议，体积小、速度快，主要用于Java生态：

```java
// Hessian序列化
ByteArrayOutputStream baos = new ByteArrayOutputStream();
Hessian2Output output = new Hessian2Output(baos);
output.writeObject(user);
byte[] bytes = baos.toByteArray();

// Hessian反序列化
Hessian2Input input = new Hessian2Input(new ByteArrayInputStream(bytes));
User u = (User) input.readObject();
```

**优势**：
- 比Java序列化体积小、速度快
- 跨语言支持（Java、C++、Flash、Python等）
- Dubbo默认序列化协议
- 无需IDL，但支持基础版本兼容

**劣势**：
- 二进制格式不可读
- 不如ProtoBuf性能好
- 大鳄（Hessian 2 vs Hessian 3有差异）

## 七、生产选型决策树 🟡

```mermaid
graph TD
    A{需要跨语言?} -->|是| B{需要极致性能?}
    A -->|否| C{Human-readable?}
    B -->|是| D[ProtoBuf + gRPC]
    B -->|否| E[Hessian / Kryo]
    C -->|是| F[JSON (Jackson)]
    C -->|否| G[ProtoBuf / Hessian]
    D --> H[带宽节省3-5倍<br/>QPS提升2-5倍]
    E --> I[无需IDL<br/>Java生态无缝]
    F --> J[调试友好<br/>生态完善]
    G --> K[高性能二进制]
```

### 7.1 典型选型

| 场景 | 推荐方案 | 原因 |
| --- | --- | --- |
| 微服务内部通信 | Dubbo + Hessian | 成熟、Java原生、性能好 |
| 跨语言RPC | gRPC + ProtoBuf | 跨语言、高性能、IDL标准化 |
| 对外HTTP API | JSON (Jackson) | 通用、调试方便、SDK生态 |
| 高性能存储 | ProtoBuf | 体积小、版本兼容 |
| 消息队列 | ProtoBuf/Hessian | 高性能 |

### 7.2 Dubbo序列化配置

```java
// Dubbo 配置 Hessian2 序列化
// dubbo:
protocol:
  name: dubbo
  serialization: hessian2

// 切换到 ProtoBuf
// dubbo:
protocol:
  name: grpc
  serialization: protobuf
```

## 八、面试通关话术 🟢

**开场**：
> 序列化本质是把对象结构转成字节流。选择序列化协议要权衡：体积（影响带宽）、速度（影响延迟）、跨语言（影响生态）、版本兼容（影响升级）。JSON可读性好但体积大速度慢；Java序列化简单但体积大、版本差、不安全；ProtoBuf体积最小速度最快，支持IDL和版本兼容，是高性能场景首选；Hessian体积小速度快，Java生态无缝，适合Dubbo等内部RPC。

**被追问ProtoBuf为什么快**：
> ProtoBuf用varint编码（小数字1字节、大数字多字节），不用写字段名（只写tag-number），TLV紧凑编码，每个字段位置固定。这三个机制加起来，序列化后体积是JSON的1/5到1/3，速度是JSON的5-10倍。

**被追问版本兼容性**：
> ProtoBuf的版本兼容性靠tag-number机制：字段用数字标记而不是名称，新增字段用新tag，旧代码读到新tag会忽略；删除字段不能复用tag，避免老数据被错误解析。加字段不影响，减字段要慎重。
