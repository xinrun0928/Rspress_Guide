# IO 模型：阻塞 IO vs 非阻塞 IO vs IO 多路复用 vs 异步 IO

当你在浏览器输入一个网址，按下回车的那一刻，数据是如何穿越层层协议，最终到达服务器的？答案藏在这个看似简单的问题里：你的线程，在等待什么？

凌晨三点，你被报警铃声吵醒。线上监控系统显示，订单服务的延迟突然飙升，从正常的 50 毫秒飙到了 5 秒。慌忙登录服务器查看，发现 CPU 使用率并不高，内存也够用，但请求就是堆积在那里无法处理。你打开线程 dump，发现大量线程都阻塞在同一个地方——`SocketInputStream.read()`。

这是典型的 I/O 阻塞问题。当你的服务采用 BIO（Blocking I/O）模型时，每个连接都需要一个独立的线程来处理。而 I/O 操作的特点是：**大部分时间，你的线程都在等待**。等待数据从网络到达，等待磁盘完成读写，等待对端发送确认信号。如果有 10000 个连接，你就需要 10000 个线程，即使其中 9999 个都在「睡觉」。

这个问题困扰了开发者很久，直到人们开始思考：能不能让一个线程同时管理多个连接？这个想法催生了后来一系列 I/O 模型的演进。

## 为什么需要非阻塞？

要理解非阻塞 I/O 的价值，我们先要理解传统 I/O 模型的问题在哪里。

想象你去餐厅吃饭。**BIO 模式**下，每位顾客进门都配备一个专属服务员。这个服务员站在顾客旁边，从顾客坐下开始就一直等待：等顾客看菜单、等顾客点菜、等厨师做菜、等上菜。等啊等，直到顾客吃完离开，这个服务员才能去服务下一位客人。如果有 100 位顾客，就需要 100 个服务员，即使大部分时间他们只是站在那里干等。

这显然是资源的巨大浪费。服务员的「等待」时间（I/O 等待）远超「工作」时间（数据处理）。但是在计算机早期，由于每个连接都很「活跃」，这种模式勉强能接受。

然而，互联网时代颠覆了这个假设。一个 HTTP 服务器可能同时维护着数万个连接，但其中 99% 的连接在 99% 的时间里都是空闲的——用户在思考、在阅读内容、在比较商品。一个请求的处理时间可能只有 100 毫秒，但两次请求之间的间隔可能是 10 秒甚至更长。如果每个连接都绑死一个线程，资源浪费将极其严重。

这就是非阻塞 I/O 模型诞生的背景：**让一个线程能够高效管理大量连接，而不是被单个连接的等待时间绑架。**

## 五种 I/O 模型：UNIX 的网络 I/O 世界

在深入 Java 之前，我们需要理解操作系统层面的 I/O 模型。Richard Stevens 的经典著作《UNIX 网络编程》详细描述了 POSIX 标准定义的五种 I/O 模型，理解它们是理解 Java I/O 的基础。

### 同步阻塞 I/O（BIO）

这是最传统、最容易理解的模型。用户进程调用 `recvfrom` 系统调用后，内核开始准备数据。在数据到达之前，进程被阻塞，什么也做不了，直到数据复制到用户空间并返回结果。

```
用户进程                                    内核
    |                                        |
    |--- recvfrom() ----------------------->|
    |         [进程阻塞，等待数据]           |
    |                                        |
    |         [数据从网络到达]                |
    |                                        |
    |         [内核复制数据到用户空间]        |
    |<-- 返回，数据在用户空间 ----------------|
    |                                        |
```

对于网络编程来说，BIO 模式意味着每个连接需要一个独立线程。当线程在 `read()` 上阻塞时，它不消耗 CPU，但也不做任何有用的事。线程的创建和销毁有开销，线程栈占用内存，如果并发连接数很高，系统会被大量阻塞线程拖垮。

```java
// BIO 模式的典型代码
public class BioServer {
    public static void main(String[] args) throws IOException {
        ServerSocket server = new ServerSocket(8080);
        while (true) {
            // 阻塞：等待连接
            Socket client = server.accept();
            // 每个连接一个线程
            new Thread(() -> {
                try {
                    handle(client);
                } catch (IOException e) {
                    e.printStackTrace();
                }
            }).start();
        }
    }
    
    private static void handle(Socket client) throws IOException {
        InputStream in = client.getInputStream();
        OutputStream out = client.getOutputStream();
        byte[] buffer = new byte[1024];
        // 阻塞：等待数据
        int n = in.read(buffer);
        // 处理数据...
        out.write(buffer, 0, n);
    }
}
```

### 同步非阻塞 I/O（NIO）

非阻塞模式下，当数据没有准备好时，`recvfrom()` 调用会立即返回，并设置 `errno` 为 `EWOULDBLOCK`（或 `EAGAIN`），而不是阻塞进程。用户进程需要不断轮询检查数据是否就绪。

```
用户进程                                    内核
    |                                        |
    |--- recvfrom() ----------------------->|
    |<-- EWOULDBLOCK，立即返回 ---------------|
    |         [数据未就绪]                    |
    |                                        |
    |--- recvfrom() ----------------------->|
    |<-- EWOULDBLOCK，立即返回 ---------------|
    |         [数据仍未就绪]                  |
    |                                        |
    |--- recvfrom() ----------------------->|
    |<-- 返回数据 ----------------------------|
    |         [数据已就绪并复制完成]          |
```

非阻塞 I/O 解决了阻塞问题，但引入了新的问题：CPU 忙等待。如果不断轮询，大部分轮询都是无意义的，浪费 CPU 资源。

```java
// 非阻塞 I/O 示例
ServerSocketChannel server = ServerSocketChannel.open();
server.socket().bind(new InetSocketAddress(8080));
server.configureBlocking(false);  // 设置为非阻塞

while (true) {
    SocketChannel client = server.accept();  // 非阻塞，立即返回
    if (client != null) {
        // 处理连接
    }
}
```

### IO 多路复用：select / poll / epoll

IO 多路复用是解决忙等待问题的关键方案。多个连接共用一个系统调用，内核负责监听所有连接的数据就绪状态。当某个连接的数据就绪时，内核通知用户进程，用户进程再去处理。

```
IO 多路复用示意:

用户进程                                    内核
    |                                        |
    |--- select()/poll() ------------------>|
    |         [阻塞，等待连接数据就绪]        |
    |                                        |
    |<-- 返回，可读连接列表 ------------------|
    |         [某个连接数据已就绪]            |
    |                                        |
    |--- recvfrom(连接1) ------------------->|
    |<-- 返回数据 ----------------------------|
    |                                        |
    |--- select()/poll() ------------------>|
    |         [继续等待]                      |
```

select、poll、epoll 是三种 IO 多路复用的实现。select 有限制（最多监听 1024 个 fd），需要遍历所有 fd；poll 没有数量限制，但仍然是遍历；epoll 是 Linux 高效实现，使用回调机制，不需要遍历。

```java
// Java NIO Selector 示例
Selector selector = Selector.open();
ServerSocketChannel server = ServerSocketChannel.open();
server.socket().bind(new InetSocketAddress(8080));
server.configureBlocking(false);
server.register(selector, SelectionKey.OP_ACCEPT);

while (true) {
    selector.select();  // 阻塞，直到有事件就绪
    Set<SelectionKey> keys = selector.selectedKeys();
    for (SelectionKey key : keys) {
        if (key.isAcceptable()) {
            // 处理 accept
        } else if (key.isReadable()) {
            // 处理 read
        }
    }
    keys.clear();
}
```

### 信号驱动 I/O

信号驱动 I/O 允许内核在数据就绪时发送 SIGIO 信号通知进程。这种模式在某些场景下很有用，但 Java 一直没有提供直接的支持。

```
信号驱动 I/O 示意:

用户进程                                    内核
    |                                        |
    |--- sigaction(SIGIO) ----------------->|
    |         [注册信号处理函数]              |
    |                                        |
    |<-- 立即返回 ----------------------------|
    |                                        |
    |         [数据从网络到达]                |
    |<-- SIGIO 信号 -------------------------|
    |         [通知进程数据就绪]              |
    |                                        |
    |--- recvfrom() ------------------------>|
    |<-- 返回数据 ----------------------------|
```

### 异步 I/O：真正的异步

异步 I/O（POSIX AIO / Windows IOCP）是真正的异步 I/O 模型。当进程发起异步 I/O 操作后，操作立即返回，进程可以继续执行其他任务。当 I/O 完成时，内核会通知进程。

```
异步 I/O 示意:

用户进程                                    内核
    |                                        |
    |--- aio_read() ----------------------->|
    |<-- 立即返回 ----------------------------|
    |         [操作已提交，立即返回]          |
    |                                        |
    |         [继续执行其他任务]              |
    |                                        |
    |         [数据从网络到达]                |
    |         [内核复制数据到用户空间]        |
    |<-- 信号/回调通知 -----------------------|
    |         [IO 完成通知]                   |
```

Java 7 引入了 `AsynchronousChannel`，提供了对异步 I/O 的支持。在 Linux 上使用 epoll，在 Windows 上使用 IOCP。

```java
// Java 异步 I/O 示例
AsynchronousServerSocketChannel server = AsynchronousServerSocketChannel.open();
server.bind(new InetSocketAddress(8080));
server.accept(null, new CompletionHandler<AsynchronousSocketChannel, Object>() {
    @Override
    public void completed(AsynchronousSocketChannel client, Object attachment) {
        // 处理连接
    }
    
    @Override
    public void failed(Throwable exc, Object attachment) {
        // 处理失败
    }
});
```

## 五种模型的对比

|模型 | 阻塞 | 同步/异步 | 轮询方式 | 适用场景 |
|---|---|---|---|---|
| 阻塞 I/O | 阻塞 | 同步 | - | 连接数少、低并发场景 |
| 非阻塞 I/O | 非阻塞 | 同步 | 忙轮询 | 基本不用，浪费 CPU |
| IO 多路复用 | 阻塞 | 同步 | 内核通知 | 高并发场景（Java NIO） |
| 信号驱动 I/O | 非阻塞 | 同步 | 信号通知 | 特殊场景，Java 不支持 |
| 异步 I/O | 非阻塞 | 异步 | 回调通知 | 高性能场景（Java AIO） |

```
性能对比（理论）:

阻塞 I/O:     线程 = 连接数，大量阻塞
非阻塞 I/O:   线程 = 连接数，CPU 忙等待
IO 多路复用:   线程 << 连接数，单线程管理万级连接
异步 I/O:      线程 << 连接数，真正异步处理
```

## Java I/O 模型的演进

Java 的 I/O 模型也经历了类似的演进：

**Java 1.0-1.3**：只有阻塞 I/O（`InputStream`/`OutputStream`）

**Java 1.4**：引入 NIO（`java.nio`），提供非阻塞 I/O 和 IO 多路复用（`Selector`）

**Java 7**：引入 AIO（`AsynchronousChannel`），提供真正的异步 I/O

```java
// BIO
InputStream in = client.getInputStream();
in.read(buffer);  // 阻塞

// NIO (IO 多路复用)
selector.select();  // 阻塞，但一个 selector 管理多个通道
channel.read(buffer);  // 非阻塞

// AIO (异步 I/O)
channel.read(buffer, null, completionHandler);  // 完全异步，立即返回
```

## 选择合适的 I/O 模型

选择哪种 I/O 模型取决于应用的特点：

**BIO**：适用于连接数少、低并发的简单应用。代码简单，易于理解和维护。

**NIO（IO 多路复用）**：适用于高并发、中等负载的应用。是目前主流的高性能网络编程选择（Netty 基于此）。

**AIO**：适用于高并发、需要极致性能的应用。但生态不如 NIO 成熟，Netty 等框架最终也选择了基于 NIO 的实现。
