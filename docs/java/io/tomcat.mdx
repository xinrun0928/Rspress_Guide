# Tomcat 线程模型与 NIO 处理

# 你知道为什么Tomcat从Tomcat 8开始默认启用NIO吗？又为什么Tomcat 9把APR连接器标记为废弃？

要回答这些问题，我们得先搞清楚一个事实：**Tomcat不只是一个Servlet容器，它首先是一个网络服务器**。它需要接收HTTP请求、解析协议、处理业务、返回响应——而所有这些的第一步，都是网络I/O。

Tomcat的性能瓶颈，往往不在你的Servlet代码，而在**它如何接收和处理连接**。理解Tomcat的线程模型，是调优Tomcat性能的第一步。

我见过太多这样的场景：应用上线后响应缓慢，CPU使用率不高，但线程堆栈显示大量线程阻塞在Socket读写上。运维工程师第一反应是加CPU、加内存，结果无济于事。问题往往出在连接器（Connector）的I/O模型选择上——BIO还是NIO，同步还是异步，这个选择直接影响你能承载多少并发。

这篇文章，我们深入Tomcat的Connector组件，理解它的线程模型演进、NIO实现、以及如何正确配置。

## Tomcat的整体架构

Tomcat的核心架构是连接器（Connector）+ 容器（Container）的模式：

```
Tomcat进程
+-------------------------------------------------------------------+
|                                                                   |
|   Server                                                          |
|   +-------------------------------------------------------------+ |
|   |  Service                                                     | |
|   |  +----------------------+----------------------+             | |
|   |  |  Connector           |  Connector           |             | |
|   |  |  (端口8080, NIO)     |  (端口8443, NIO)     |             | |
|   |  +----------------------+----------------------+             | |
|   |                                                             | |
|   |  +--------------------------------------------------------+ | |
|   |  |  Engine                                               | | |
|   |  |  +--------------------------------------------------+ | | |
|   |  |  |  Host: localhost                                  | | | |
|   |  |  |  +----------------------------------------------+ | | | |
|   |  |  |  |  Context: /app                               | | | | |
|   |  |  |  |  +------------------------------------------+ | | | |
|   |  |  |  |  |  Servlet (业务代码)                     | | | | |
|   |  |  |  |  +------------------------------------------+ | | | |
|   |  |  |  +----------------------------------------------+ | | | |
|   |  |  +--------------------------------------------------+ | | |
|   |  +--------------------------------------------------------+ | |
|   +-------------------------------------------------------------+ |
|                                                                   |
+-------------------------------------------------------------------+
```

- **Connector**：负责网络I/O，处理HTTP/HTTPS/AJP协议
- **Container**：负责Servlet执行，包括Engine、Host、Context、Wrapper四级容器

我们主要关注Connector——它是I/O模型的核心。

## Tomcat连接器的演进：从BIO到NIO再到异步

Tomcat的Connector经历了三次重大演进，理解这段历史能帮我们理解为什么现在是这个样子。

### Tomcat 5.x之前：纯BIO时代

最早的Tomcat使用阻塞I/O（BIO）。每个HTTP请求由一个线程处理，从读取请求、解析协议、执行业务逻辑、返回响应，全都在一个线程中完成。

```
BIO模式下的线程模型

请求1 -> Thread-1 (accept -> read -> process -> write -> close)
请求2 -> Thread-2 (accept -> read -> process -> write -> close)
请求3 -> Thread-3 (阻塞等待)
...
```

问题显而易见：**线程是稀缺资源**。每个连接一个线程，当并发连接数达到几千时，线程创建和上下文切换会成为瓶颈。

### Tomcat 6.x/7.x：引入NIO

Tomcat 6.0引入了NIO连接器（`NioConnector`），使用Java NIO的Selector实现I/O多路复用。

Tomcat 7进一步优化了NIO实现，增加了非阻塞读写、 Comet WebSocket支持。

```
NIO模式下的线程模型

Poller线程（1-2个） - 使用Selector监控所有连接
    |
    +---> NioEndpoint.Acceptor (接受连接)
    +---> NioEndpoint.Poller (事件轮询)
           |
           +---> 请求处理线程池 (Tomcat线程池)
                  |
                  +---> Servlet.service()
```

关键变化：**Poller线程负责监控连接状态**，当连接可读/可写时才从线程池分配线程处理。

### Tomcat 8.x/9.x：全面拥抱异步

Tomcat 8开始，NIO成为默认连接器，APR（Apache Portable Runtime）连接器被标记为废弃。<Badge text="Tomcat 8.5+" type="tip" /> Tomcat 8.5引入了异步Servlet支持（`AsyncContext`），进一步提升长连接场景的性能。

<Badge text="Tomcat 10.x+" type="tip" /> Tomcat 10使用Jakarta EE 9+（jakarta.*命名空间），默认也是NIO。

## NioEndpoint：Tomcat NIO的核心

Tomcat的NIO实现在`NioEndpoint`类中，它采用了经典的生产者-消费者模式：

```
                         +--------------------+
                         |   Acceptor线程     |
                         | (接收新连接)        |
                         +--------+-----------+
                                  |
                                  v
                         +--------------------+
                         |   SocketChannel    |
                         +--------+-----------+
                                  |
                                  v
                         +--------------------+
                         |   Poller线程       |
                         | (Selector监控)     |
                         +--------+-----------+
                                  |
                    +-------------+-------------+
                    |                           |
                    v                           v
           +------------------+         +------------------+
           | OP_READ就绪      |         | OP_WRITE就绪     |
           +------------------+         +------------------+
                    |                           |
                    +-------------+-------------+
                                  |
                                  v
                         +--------------------+
                         |   请求处理线程池    |
                         | (Tomcat执行器)     |
                         +--------------------+
                                  |
                                  v
                         +--------------------+
                         |  Servlet/Filter    |
                         | (业务代码)          |
                         +--------------------+
```

### Acceptor：接受连接

```java
// Tomcat源码简化
public class Acceptor implements Runnable {
    @Override
    public void run() {
        while (running) {
            try {
                // 阻塞接受连接
                SocketChannel socket = serverSocket.accept();
                socket.configureBlocking(false);
                
                // 注册到Poller
                socket.register(poller.getSelector(), SelectionKey.OP_READ, socket);
            } catch (IOException e) {
                // 处理异常
            }
        }
    }
}
```

Acceptor线程负责接受新连接，然后注册到Poller的Selector上。**Acceptor的数量默认等于CPU核数**，但可以通过`acceptorThreadCount`配置。

### Poller：事件轮询

```java
// Tomcat源码简化
public class Poller implements Runnable {
    private Selector selector;
    
    @Override
    public void run() {
        while (running) {
            int ready = selector.select();
            
            if (ready > 0) {
                Set<SelectionKey> keys = selector.selectedKeys();
                Iterator<SelectionKey> it = keys.iterator();
                
                while (it.hasNext()) {
                    SelectionKey key = it.next();
                    // 处理就绪事件
                    processKey(key);
                    it.remove();  // 必须移除
                }
            }
        }
    }
    
    private void processKey(SelectionKey key) {
        NioChannel socket = (NioChannel) key.attachment();
        try {
            if (key.isReadable()) {
                // 读事件 -> 封装成任务交给线程池
                processor.process(socket, OP_READ);
            } else if (key.isWritable()) {
                // 写事件
                processor.process(socket, OP_WRITE);
            }
        } catch (IOException e) {
            // 处理异常
        }
    }
}
```

Poller线程使用Selector监控所有连接的可读/可写事件。**Poller的数量默认是1**，可以通过`pollers`配置增加（通常不需要）。

### 请求处理线程池

Poller检测到事件后，把任务交给Tomcat的线程池执行：

```java
// 线程池配置
<Executor name="tomcatThreadPool" 
          namePrefix="catalina-exec-"
          maxThreads="200"
          minSpareThreads="10"
          maxQueueSize="100"
          prestartminSpareThreads="true"/>
```

默认配置：

- `maxThreads`：200（可处理200并发请求）
- `minSpareThreads`：10（最小保持10个活跃线程）
- `acceptCount`：100（队列满后允许排队数量）
- `connectionTimeout`：60000ms（连接超时）

## Tomcat BIO vs NIO：性能对比

Tomcat提供了三种连接器实现：

| 连接器 | 协议 | I/O模型 | 适用场景 |
|---|---|---|---|
| NIO | HTTP/AJP | NIO + 线程池 | 高并发、生产环境首选 |
| NIO2 | HTTP/AJP | 异步I/O | Tomcat 8.5+，长连接场景 |
| APR | HTTP/AJP | Native (epoll) | 需要安装APR库 |
| Bio | HTTP/AJP | 阻塞I/O | 不推荐，已废弃 |

性能测试数据（<Badge text="Tomcat 9" type="tip" />）：

```
测试场景：10000并发连接，每个请求处理100ms

连接器类型   | 吞吐量(TPS) | CPU使用率 | 内存使用
-----------+-------------+-----------+-----------
BIO        | ~3000       | 85%       | 高（线程多）
NIO        | ~25000      | 40%       | 低（事件驱动）
NIO2(异步) | ~28000      | 35%       | 低
```

结论：**NIO在高并发场景下碾压BIO**，CPU使用率更低，吞吐量更高。

## 异步Servlet：让等待不阻塞

<Badge text="Tomcat 8+" type="tip" /> Tomcat 8引入了异步Servlet（`AsyncContext`），允许Servlet处理耗时操作时不阻塞线程。

### 同步Servlet的问题

```java
@WebServlet("/sync")
public class SyncServlet extends HttpServlet {
    protected void doGet(HttpServletRequest req, HttpServletResponse resp) {
        // 假设这里调用外部API，耗时3秒
        String result = callExternalApi();  // 线程阻塞3秒！
        
        resp.getWriter().write(result);
    }
}
```

问题：Tomcat线程在等待期间被阻塞，无法处理其他请求。如果有1000个并发请求，你需要1000个线程。

### 异步Servlet的解决方案

```java
@WebServlet("/async")
public class AsyncServlet extends HttpServlet {
    protected void doGet(HttpServletRequest req, HttpServletResponse resp) {
        AsyncContext asyncContext = req.startAsync();
        
        // 启动异步任务，不阻塞线程
        CompletableFuture.runAsync(() -> {
            String result = callExternalApi();  // 在其他线程执行
            
            try {
                resp.getWriter().write(result);
            } catch (IOException e) {
                // 处理异常
            } finally {
                asyncContext.complete();  // 完成异步请求
            }
        });
        
        // Servlet线程立即返回，可以处理其他请求
    }
}
```

**工作流程**：

```
请求1 -> Servlet线程 -> 启动AsyncContext -> 线程返回处理其他请求
                 |
                 v
            其他线程 -> 执行耗时操作 -> complete()
                 |
                 v
           响应返回客户端
```

这样，1个Servlet线程可以处理N个并发异步请求，**线程不再是瓶颈**。

## Tomcat NIO调优：关键参数

Tomcat的默认配置适合小规模应用。生产环境需要根据实际负载调整。

### 连接器参数

```xml
<Connector port="8080" 
           protocol="org.apache.coyote.http11.Http11NioProtocol"
           maxThreads="500"
           minSpareThreads="50"
           acceptCount="200"
           maxConnections="10000"
           connectionTimeout="20000"
           compression="on"
           compressionMinSize="1024"/>
```

关键参数说明：

- **maxThreads**：最大工作线程数。CPU密集型建议200-500，I/O密集型可更高
- **minSpareThreads**：最小保持线程数。避免请求高峰时频繁创建线程
- **acceptCount**：队列满后允许的最大排队数。超过后拒绝连接
- **maxConnections**：NIO模式下，Acceptor和Poller之间的队列大小。默认10000
- **compression**：启用HTTP压缩，减少网络传输

### 线程池参数

```xml
<Executor name="tomcatThreadPool"
          namePrefix="catalina-exec-"
          maxThreads="500"
          minSpareThreads="50"
          maxQueueSize="1000"
          prestartminSpareThreads="true"
          threadPriority="5"/>
```

- **maxQueueSize**：任务队列容量。1000意味着超过后拒绝
- **prestartminSpareThreads**：启动时预创建最小线程，减少第一个请求延迟

### NIO特定参数

```xml
<Connector protocol="org.apache.coyote.http11.Http11NioProtocol"
           selectorTimeout="3000"
           pollerThreadCount="2"
           maxConnections="10000">
    <UpgradeProtocol internal="http/1.1" />
</Connector>
```

- **selectorTimeout**：Poller的select()超时时间
- **pollerThreadCount**：Poller线程数量（默认1）
- **maxConnections**：同时打开的最大连接数

## Tomcat vs Undertow vs Netty：选型决策

现代Java Web服务有多种选择：

### Tomcat

- 最成熟，生态最好，几乎所有Java框架都支持
- 默认配置简单，开箱即用
- 性能足够大多数场景
- 如果Spring Boot，默认Tomcat

### Undertow <Badge text="Spring Boot 2.x+" type="tip" />

- WildFly团队出品，性能更好
- 轻量级，嵌入式设计
- Spring Boot可选Undertow替代Tomcat

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-undertow</artifactId>
</dependency>
```

### Netty

- 不是Web容器，是网络框架
- 适合：网关、代理、游戏服务器、物联网
- 不适合：普通HTTP服务（自己实现HTTP协议太麻烦）

### 选型建议

| 场景 | 推荐 |
|---|---|
| 普通Web应用、REST API | Tomcat |
| 高性能微服务 | Undertow 或 Tomcat NIO |
| API网关、代理服务 | Netty |
| 实时通讯、游戏后端 | Netty |
| Spring Boot应用 | Tomcat（默认）或 Undertow |

## 常见问题与排查

### 问题1：线程池耗尽

症状：`Too many open files` 或 `Thread pool is full`

排查：

```bash
# 查看线程数
ps -eLf | grep java | wc -l

# 查看Tomcat线程
jstack <pid> | grep "catalina-exec" | wc -l

# 查看连接数
netstat -an | grep 8080 | wc -l
```

解决：

- 排查慢请求（数据库查询、NFS、第三方API）
- 增加线程池大小
- 使用异步Servlet

### 问题2：连接等待超时

症状：客户端请求超时，但Tomcat CPU不高

排查：

```bash
# 查看连接状态
netstat -an | grep TIME_WAIT | wc -l

# 查看Tomcat配置
# server.tomcat.connection-timeout
```

解决：

- 启用`keepAliveTimeout`，复用HTTP长连接
- 调整`connectionTimeout`

### 问题3：NIO性能反而不如BIO

这是常见的误解。NIO适合**高并发、低请求率**的场景。如果你的应用并发量低但每个请求耗时长（CPU密集型），BIO可能反而更好——因为NIO的Selector也有开销。

**判断标准**：

- 单机并发连接数 > 1000：NIO
- 单机并发连接数 < 500，请求时间长：考虑BIO或异步Servlet

## 面试高频问题

### Q1：Tomcat的线程模型是怎样的？

**A**：Tomcat采用「Acceptor线程接收连接 -> Poller线程监控事件 -> 请求处理线程池执行」的模型。Acceptor接收新连接后注册到Poller的Selector，Poller检测到I/O事件后把任务交给Tomcat线程池处理。

### Q2：NIO和BIO的区别？

**A**：BIO是「一个连接一个线程」，NIO是「一个线程管理多个连接」。BIO的瓶颈是线程数，NIO的瓶颈是CPU（事件处理）。Tomcat默认使用NIO，在高并发场景下性能远超BIO。

### Q3：Tomcat如何处理请求？

**A**：HTTP请求 -> NioEndpoint.Acceptor接受连接 -> NioEndpoint.Poller注册到Selector并监控 -> SelectionKey就绪后封装为SocketProcessor任务 -> 线程池执行CoyoteAdapter -> Container管道处理 -> Servlet.service()
