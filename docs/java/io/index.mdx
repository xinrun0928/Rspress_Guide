# Java IO/NIO

面试官问："解释一下 NIO 的三大核心组件。"

候选人小刘说："是 Buffer、Channel 和 Selector。"

面试官追问："Selector 底层是怎么实现的？为什么单线程能处理多个连接？"

小刘说："...用 epoll？"

面试官："Linux 下用 epoll，那 JDK 的 epoll 实现有什么坑，你知道吗？"

小刘愣住了。

---

这道题我在阿里和字节的面试里都问过。NIO 看似概念简单，但真正能讲清楚 Selector 底层实现、BIO/NIO/AIO 对比、以及零拷贝技术的人，不超过 20%。

更别提 Netty 了——这玩意儿是现在高性能网络编程的事实标准，但很多人只会"听说过"，真正懂的人太少了。

这一模块，带你把 Java IO 从"知道有 BIO 和 NIO"升级到"能讲清楚演进逻辑、底层实现、以及 Netty 的核心设计"。

## 学习路径指引

### P5 候选人（校招/初级社招）

IO 的入门关是**理解 BIO 和 NIO 的核心区别**，以及**为什么会演进到 NIO**。

**P5 必过关卡**：

1. BIO 的线程模型是什么？为什么 BIO 在高并发下会有性能问题？
2. NIO 的三大核心组件是什么？Selector 解决了什么问题？
3. 什么是零拷贝？Linux 下有哪些零拷贝技术？
4. Netty 是什么？为什么高性能网络编程都用 Netty？

这四道题能流畅答出来，IO 基础就算过关。

### P6 候选人（中级社招）

P6 的门槛在**能讲清楚 Linux epoll 的底层原理，以及 Netty Reactor 模型的演进**。

**拉开差距的追问链（Selector）**：

```
Selector 底层是怎么实现的？
  → Linux 下用 epoll，那 JDK 怎么封装 epoll 的？（epoll_create/ctl/wait）
    → JDK 的 epoll 实现有什么坑？（epoll 空轮询 bug、空转 CPU 100%）
      → 这个问题在哪个 JDK 版本修好的？怎么修复的？（selector 空轮询次数阈值）
```

这条追问链能走完的，基本都是踩过坑或者深入研究过源码的。

**拉开差距的追问链（Netty）**：

```
Netty 为什么用 Reactor 模型？
  → 单线程 Reactor 有什么问题？（单线程负担过重）
    → 多线程 Reactor 呢？（boss 和 worker 分离）
      → 主从 Reactor 呢？（Accept 和读写分离，参考 Nginx/Redis）
        → 你的项目中用的是哪种模式？为什么？
```

**零拷贝追问链**：

```
什么是零拷贝？
  → 传统 read/write 需要几次拷贝？（4 次：磁盘→内核→用户→Socket→网卡）
    → sendfile 需要几次？（3 次，消除了用户态拷贝）
      → 支持 SG-DMA 的 sendfile 呢？（2 次，DMA 直接从磁盘到网卡）
        → Netty 里怎么实现零拷贝的？（ByteBuf、CompositeByteBuf、slice）
```

【面试官心理】
IO 这道题，我从 NIO 的 Selector 开始问。如果候选人能说清楚 epoll 的原理，我会追问 Netty 的 Reactor 模型；如果卡住了，我会换个角度问零拷贝。NIO 和零拷贝是两个独立的深度追问方向，看候选人在哪个方向有积累。

### P7 候选人（高级社招/架构方向）

P7 的 IO 是在考**性能优化能力和框架选型能力**。

**典型的 P7 级问题**：

- 你的项目为什么选择 Netty 而不是 MINA？选型依据是什么？
- Netty 的 ByteBuf 为什么要分堆内和堆外？什么场景用哪种？
- epoll 空轮询 bug 在生产上怎么规避？有没有遇到过？
- RPC 框架里怎么设计序列化和反序列化协议？为什么选择你用的方案？

:::tip 💡
Netty 的 ByteBuf 分池化和非池化两种。JDK9 引入的堆外内存（DirectByteBuffer）和 Netty 的池化 ByteBuf 都能减少 GC 压力，但堆外内存的分配和释放成本较高，适合长连接场景。
:::

## 生产避坑指南

IO 相关的问题在生产上往往是性能瓶颈的根源：

- **BIO 连接数爆炸**：没限制最大连接数，上游流量突增时线程池耗尽，服务雪崩。
- **Netty 内存泄漏**：ByteBuf 没释放、没用 ReferenceCounted、handler 里忘记释放 ctxalloc。最典型的表现是内存不断增长直到 OOM。
- **拆包粘包问题**：自定义协议没有处理 TCP 粘包，上线后数据错乱。
- **NIO epoll 空轮询**：JDK Bug 导致 Selector 空转，CPU 100% 但实际没处理任何事件。
- **序列化性能问题**：用 Java 原生序列化做 RPC，性能极差；换了 Protobuf 后 QPS 翻了几倍。

:::warning ⚠️
Netty ByteBuf 的内存泄漏是最常见的生产问题之一。使用 PooledByteBufAllocator 时，如果 Handler 中使用 `buf.toString()` 或 `buf.array()` 获取了数组引用后没释放，内存会慢慢耗尽。务必使用 try-with-resources 或手动释放。
:::
