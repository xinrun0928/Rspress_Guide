# Socket 通信与三次握手

# 当你在浏览器输入`www.example.com`并按下回车时，数据包经历了什么？三次握手为什么是三次，四次挥手为什么不是三次？

这是一个看似简单却蕴含深意的问题。几乎所有计算机网络面试都会问「TCP三次握手」，但大多数人只背住了「Client发SYN、Server回SYN+ACK、Client回ACK」这个流程，却不理解**为什么需要三次**、**半连接队列是什么**、**SYN Flood攻击是怎么回事**。

更少有人知道：**三次握手过程中，操作系统为每个连接分配了什么资源？TCP的TIME_WAIT状态会持续多久，对服务端有什么影响？为什么连接成功建立后，数据传输仍然可能失败？**

理解Socket通信，不仅是为了应付面试，更是为了写出健壮的网络代码。比如，你知道为什么使用短连接的HTTP服务会遇到「Address already in use」错误吗？你知道为什么高并发服务端需要调整`tcp_tw_reuse`参数吗？

这篇文章，我们从Socket的底层原理出发，深入TCP连接的建立和关闭过程，理解三次握手和四次挥手的真正含义。

## Socket是什么：从API到内核

Socket（套接字）是网络通信的抽象接口。它最早出现在Berkeley UNIX中，后来成为POSIX标准的一部分。Java的`java.net.Socket`就是对操作系统Socket的封装。

### Socket的诞生背景

在Socket出现之前，网络编程需要直接操作网络协议细节：封装数据包、处理校验和、管理重传……这太复杂了，普通应用开发者无法承受。

Socket的设计哲学是：**提供一个简单的API，把协议细节隐藏起来**。你不需要知道TCP是怎么建连的，只需要`connect()`；你不需要知道数据是怎么可靠传输的，只需要`read()`/`write()`。

```
应用层
+---------------------------------------------------------------+
| Socket API                                                    |
| socket() -> bind() -> listen() -> accept() -> connect()        |
|             read() -> write() -> close()                      |
+---------------------------------------------------------------+
传输层（TCP/UDP）
+---------------------------------------------------------------+
| TCP: 可靠连接、流量控制、拥塞控制                              |
| UDP: 无连接、不可靠、最大努力交付                              |
+---------------------------------------------------------------+
网络层（IP）
+---------------------------------------------------------------+
| IP: 路由转发、分组封装                                         |
+---------------------------------------------------------------+
链路层（Ethernet）
+---------------------------------------------------------------+
| ARP: IP到MAC地址解析                                           |
| Ethernet: 局域网传输                                            |
+---------------------------------------------------------------+
```

### Java Socket的基本使用

**服务端**：

```java
// 1. 创建ServerSocket
ServerSocket serverSocket = new ServerSocket(8080);

// 2. 监听连接
while (true) {
    Socket clientSocket = serverSocket.accept();  // 阻塞等待
    new Thread(() -> {
        try {
            // 3. 处理请求
            InputStream in = clientSocket.getInputStream();
            OutputStream out = clientSocket.getOutputStream();
            
            // 读写数据
            byte[] buf = new byte[1024];
            int len = in.read(buf);
            
            // 4. 关闭连接
            clientSocket.close();
        } catch (IOException e) {
            e.printStackTrace();
        }
    }).start();
}
```

**客户端**：

```java
// 1. 创建Socket
Socket socket = new Socket("example.com", 8080);

// 2. 获取流
OutputStream out = socket.getOutputStream();
InputStream in = socket.getInputStream();

// 3. 发送请求
out.write("Hello".getBytes());
out.flush();

// 4. 接收响应
byte[] buf = new byte[1024];
int len = in.read(buf);

// 5. 关闭连接
socket.close();
```

这段代码的背后，隐藏着TCP三次握手的全过程。

## 三次握手：建立可靠连接的艺术

### 为什么需要握手？

TCP是**面向连接**的协议。在传输数据之前，通信双方需要确认对方「在线」且「能接收数据」。这就是连接建立过程——**三次握手**。

```
Client                                                    Server
   |                                                        |
   |  [CLOSED]                                         [CLOSED]
   |                                                        |
   |--------------------- SYN ------------------------------>|
   |  [SYN_SENT]                                      [LISTEN]
   |                                                        |
   |<------------------- SYN + ACK -------------------------|
   |  [SYN_SENT]                                      [SYN_RCVD]
   |                                                        |
   |--------------------- ACK ------------------------------>|
   |  [ESTABLISHED]                                  [ESTABLISHED]
   |                                                        |
   |================= 数据传输 =============================>|
```

### 三次握手的具体过程

**第一次握手**（Client → Server）：

- Client发送TCP段，`SYN=1, seq=x`
- `SYN=1`表示这是一个连接请求
- `seq=x`是Client的初始序列号（ISN）
- Client进入`SYN_SENT`状态

**第二次握手**（Server → Client）：

- Server收到SYN后，发送`SYN=1, ACK=1, seq=y, ack=x+1`
- `SYN=1`表示Server也请求建立连接
- `ACK=1`确认收到了Client的SYN
- `ack=x+1`表示Server期望收到的下一个字节
- Server进入`SYN_RCVD`状态

**第三次握手**（Client → Server）：

- Client发送`ACK=1, seq=x+1, ack=y+1`
- `ack=y+1`确认收到了Server的SYN
- Client进入`ESTABLISHED`状态
- Server收到ACK后也进入`ESTABLISHED`状态

### 为什么必须是三次？

这是面试中最常被问到的问题。理解答案需要深入TCP的设计哲学。

**目的1：同步双方的序列号**

TCP使用序列号来标识数据字节。每个方向都需要知道对方的初始序列号（ISN）。三次握手中的每次握手都携带序列号：

```
Client -> Server: SYN, seq=x          (Client告诉Server自己的ISN)
Server -> Client: SYN, seq=y, ack=x+1 (Server告诉Client自己的ISN，并确认Client的ISN)
Client -> Server: ACK, ack=y+1         (Client确认Server的ISN)
```

只有通过三次交换，双方才能同步两个ISN。

**目的2：验证双方都有发送和接收能力**

- 第一次握手：Client证明自己能发送
- 第二次握手：Server证明自己能收（收到Client的SYN）、能发（发送自己的SYN）
- 第三次握手：Client证明自己能收（收到Server的SYN+ACK）

如果只有两次握手，Server无法确认自己的SYN是否被Client收到。

**目的3：防止历史连接初始化**

这是最关键的理由。考虑这个场景：

```
Client发送了SYN = 100，但网络延迟，这个包很晚才到达Server
Client已经取消了这个连接，重新发了新的SYN = 200

如果Server只收到一个SYN，它无法区分这是新的连接还是延迟的老连接
通过三次握手，Client的最后一个ACK确认了Server的SYN是「当前」的
```

RFC 793明确指出：**TCP三次握手的目的之一是防止旧连接的重复分组被误认为是新连接的一部分**。

### 半连接队列与SYN Flood

Server收到Client的SYN后，会在`SYN_RCVD`状态等待Client的ACK。此时的连接称为**半连接**（half-open connection）。

Server会为每个半连接分配内存，如果半连接数量太多，会耗尽Server资源。

```
SYN Flood攻击原理：

攻击者发送大量SYN，但不完成第三次握手
Server维护的半连接队列被填满
正常用户的连接请求被拒绝

防御措施：
- SYN Cookie：不保存半连接，校验Client的ACK后才分配资源
- 减少SYN+ACK重试次数
- 增大半连接队列
```

Linux参数：

```bash
# 查看半连接队列大小
cat /proc/sys/net/ipv4/tcp_max_syn_backlog

# 查看当前半连接数
ss -s

# 开启SYN Cookie
echo 1 > /proc/sys/net/ipv4/tcp_syncookies
```

### TCP选项：握手中的协商

三次握手时可以协商TCP选项：

```
TCP Header中的Options字段可以携带：
- MSS (Maximum Segment Size)：最大报文段长度
- Window Scale：窗口扩大因子
- SACK：选择性确认
- Timestamp：时间戳
```

```bash

cat /proc/sys/net/ipv4/tcp_window_scaling
```

这些选项在握手时协商好，之后的数据传输会使用这些参数。

## 四次挥手：优雅关闭的艺术

### 为什么是四次？

与建立连接不同，关闭连接需要**每个方向都单独关闭**。这是因为TCP是全双工的——同一时刻，双方都可以发送数据。

```
Client                                                    Server
   |                                                        |
   |================= 数据传输 =============================>|
   |================= 数据传输 <=============================|
   |                                                        |
   |--------------------- FIN ------------------------------>|
   |  [FIN_WAIT_1]                                     [CLOSE_WAIT]
   |                                                        |
   |<-------------------- ACK -------------------------------|
   |  [FIN_WAIT_2]                                       [数据处理中]
   |                                                        |
   |                                            (Server处理完数据)
   |                                                        |
   |<-------------------- FIN ------------------------------|
   |  [TIME_WAIT]                                      [LAST_ACK]
   |                                                        |
   |-------------------- ACK ------------------------------>|
   |  [CLOSED]                                         [CLOSED]
   |                                                        |
   |                    2MSL后                              |
   |  [CLOSED]                                         
```

**第一次挥手**：主动关闭方发送FIN，表示「我这边数据发完了」
**第二次挥手**：被动关闭方发送ACK，确认收到FIN
**第三次挥手**：被动关闭方发送FIN，表示「我这边数据也发完了」
**第四次挥手**：主动关闭方发送ACK，确认收到FIN

### TIME_WAIT状态：最容易被忽视的坑

主动关闭方在发送最后一个ACK后，会进入`TIME_WAIT`状态，持续**2MSL**（Maximum Segment Lifetime，一个报文段在网络中的最大生存时间，通常为60秒）。

TIME_WAIT存在的意义：

1. **保证最后的ACK能到达对端**：如果ACK丢失，Server会重发FIN，Client需要处于可接收状态
2. **让旧连接的报文在网络中消失**：防止新连接收到旧连接的延迟数据

```
2MSL = 60秒 × 2 = 120秒

这意味着：
- 主动关闭方在关闭后，还有约2分钟无法使用相同的端口
- 高并发短连接服务可能遇到「端口耗尽」问题
```

### 常见问题：Address already in use

这是高并发服务的经典问题。当Server快速重启时，会遇到端口被TIME_WAIT占用的情况：

```java
// Server重启时可能遇到
ServerSocket server = new ServerSocket(8080);
// 异常：Address already in use
```

解决方案：

1. **设置SO_REUSEADDR**：

```java
ServerSocket server = new ServerSocket();
server.setReuseAddress(true);  // 允许地址复用
server.bind(new InetSocketAddress(8080));
```

2. **调整TIME_WAIT时长**（Linux）：

```bash
# 减少MSL（不推荐，影响可靠性）
echo 30 > /proc/sys/net/ipv4/tcp_fin_timeout

# 开启TIME_WAIT重用
echo 1 > /proc/sys/net/ipv4/tcp_tw_reuse
```

3. **使用长连接**：避免频繁建立和关闭连接

### CLOSE_WAIT状态：资源泄漏的信号

被动关闭方在收到FIN后进入`CLOSE_WAIT`状态，等待自己的数据发送完毕后发送FIN。

如果代码中忘记关闭Socket，会导致大量连接处于`CLOSE_WAIT`状态：

```bash
# 查看连接状态
ss -ant | grep CLOSE_WAIT

# 正常情况下，CLOSE_WAIT数量应该为0或接近0
```

问题代码示例：

```java
// 问题：读取完数据后没有关闭连接
Socket socket = serverSocket.accept();
InputStream in = socket.getInputStream();
byte[] buf = new byte[1024];
int len = in.read(buf);
// 忘记调用 socket.close();
// 导致连接永远处于CLOSE_WAIT
```

## Socket选项：调优的关键

### SO_REUSEADDR

允许多个Socket绑定到同一个地址，解决TIME_WAIT端口占用问题：

```java
ServerSocket server = new ServerSocket();
server.setReuseAddress(true);
server.bind(new InetSocketAddress(8080));
```

### SO_KEEPALIVE

开启TCP保活机制，检测连接是否存活：

```java
socket.setKeepAlive(true);
```

如果连接空闲2小时（默认），会发送探测包。如果对方无响应，连接会被关闭。

### SO_LINGER

控制close()行为：

```java
// 不等待，直接关闭
socket.setSoLinger(false, 0);

// 等待最多30秒发送剩余数据
socket.setSoLinger(true, 30);
```

### TCP_NODELAY

禁用Nagle算法，立即发送小数据包：

```java
socket.setTcpNoDelay(true);  //适合实时交互
```

### SO_RCVBUF / SO_SNDBUF

调整接收/发送缓冲区大小：

```java
socket.setReceiveBufferSize(1024 * 1024);  // 1MB
socket.setSendBufferSize(1024 * 1024);
```

## Java NIO Socket：正确处理连接

使用Java NIO处理Socket时，需要注意Channel的配置和事件处理：

```java
ServerSocketChannel serverChannel = ServerSocketChannel.open();
serverChannel.socket().bind(new InetSocketAddress(8080));
serverChannel.configureBlocking(false);  // 非阻塞模式

Selector selector = Selector.open();
serverChannel.register(selector, SelectionKey.OP_ACCEPT);

while (true) {
    selector.select();  // 阻塞直到有事件
    
    Set<SelectionKey> keys = selector.selectedKeys();
    for (SelectionKey key : keys) {
        if (key.isAcceptable()) {
            // 处理新连接
            ServerSocketChannel ssc = (ServerSocketChannel) key.channel();
            SocketChannel sc = ssc.accept();
            sc.configureBlocking(false);
            
            // 注册读事件
            sc.register(selector, SelectionKey.OP_READ);
        } else if (key.isReadable()) {
            // 处理读事件
            SocketChannel sc = (SocketChannel) key.channel();
            ByteBuffer buffer = ByteBuffer.allocate(1024);
            int read = sc.read(buffer);
            
            if (read > 0) {
                buffer.flip();
                // 处理数据...
                buffer.clear();
            } else if (read == -1) {
                // 连接关闭
                sc.close();
            }
        } else if (key.isWritable()) {
            // 处理写事件
            // 注意：通常不需要显式注册写事件
            // 直接写即可，除非写缓冲区已满
        }
    }
    keys.clear();
}
```

## 常见面试问题

### Q1：三次握手过程中，Server端发生了什么？

**A**：

1. Server收到Client的SYN，状态变为`SYN_RCVD`
2. Server分配资源（TCB，Transmission Control Block）保存连接状态
3. Server将连接加入半连接队列（SYN Queue）
4. Server回复SYN+ACK，等待Client的确认

### Q2：为什么连接是三次，挥手是四次？

**A**：连接时Server的SYN和ACK可以一起发送（第二次握手），因为此时Server还没有数据要发送。关闭时，Server可能还有数据要发送，所以先回复ACK，等数据发送完再单独发FIN。

### Q3：TIME_WAIT状态持续多久？为什么需要它？

**A**：TIME_WAIT持续2MSL（约60秒×2=120秒）。目的有两个：1）保证最后的ACK能到达对端，如果丢失，Server会重发FIN；2）让旧连接的报文在网络中完全消失，防止影响新连接。

### Q4：SYN Flood攻击原理和防御？

**A**：攻击者发送大量SYN但不完成握手，填满Server的半连接队列，导致正常用户无法连接。防御措施包括：SYN Cookie（不保存半连接，校验ACK后才分配资源）、减少SYN+ACK重试、增加半连接队列长度。

## 总结：Socket编程的实践建议

1. **使用长连接**：避免频繁建立/关闭连接，减少TIME_WAIT问题
2. **正确关闭连接**：确保FIN能正常发送和接收
3. **处理异常**：网络不稳定时，连接可能意外断开，需要心跳检测
4. **合理配置Socket选项**：根据业务特点调整缓冲区大小、超时时间等
5. **监控连接状态**：关注CLOSE_WAIT、TIME_WAIT等异常状态

现在，你已经理解了Java I/O与网络编程的全貌。从最底层的Socket通信，到NIO、Selector，再到Netty、Tomcat，每一层都是对上一层的封装和优化。选择合适的抽象层次，是写出高性能网络应用的关键。
