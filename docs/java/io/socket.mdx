# Socket 通信与三次握手

凌晨 3 点，你被告警吵醒：线上服务连接数暴涨，响应时间飙升。你登录服务器，`netstat` 一看，发现大量 `CLOSE_WAIT` 状态的连接。

这不是玄学，这是 TCP 连接管理的经典问题。

Socket 编程是 Java 网络通信的根基，但很多开发者只会用封装好的框架，却不理解底层发生了什么。当连接泄漏、`CLOSE_WAIT` 堆积、响应超时这些问题出现时，如果不懂 TCP 的状态机，就只能靠玄学 debug。

这篇文章会从 Socket 的本质讲起，重点剖析三次握手、连接状态，以及 BIO 模型下服务端开发的常见坑。

## Socket 的本质

### 什么是 Socket

Socket（套接字）是 TCP/IP 协议栈的抽象，提供了应用程序与网络协议栈之间的接口。从内核角度，Socket 是一个文件描述符；从应用角度，Socket 是双方通信的端点。

```
应用程序
    ↓ 写入 Socket
+---------+
|  Socket |  <-- 文件描述符，read/write 操作的对象
+---------+
    ↓ 系统调用
内核协议栈
    ↓
TCP/UDP 层
    ↓
IP 层
    ↓
网卡驱动
```

### Java 中的 Socket 编程

Java 提供了两套 Socket API：

| API | 特点 | 适用场景 |
|---|---|---|
| `java.net.Socket` / `ServerSocket` | BIO（阻塞 I/O） | 传统服务端，低并发场景 |
| `java.nio.channels.SocketChannel` | NIO（非阻塞） | 高并发服务，如 Netty |
| `java.nio.channels.ServerSocketChannel` | NIO | 高并发服务 |

本文重点讲解 BIO 模型，因为它是理解其他模型的基础。

## TCP 三次握手详解

### 为什么需要三次握手

TCP 是一个**面向连接**的协议，"连接"本质上是双方协商一系列参数的过程。三次握手的目的是：

1. **确认双方的发送和接收能力正常**
2. **协商初始序列号（ISN）**
3. **避免历史连接干扰新连接**

```
客户端                        服务端
   |                             |
   |------- SYN (seq=x) -------->|  客户端：我可以发送
   |                             |
   |<----- SYN-ACK (seq=y, ack=x+1) -----|  服务端：我收到，我能发送
   |                             |
   |------- ACK (ack=y+1) ------>|  客户端：我收到，连接建立
   |                             |
   |======== 数据传输开始 ========>|
```

**为什么不是两次握手？**

假设网络延迟很高，客户端发送的第一个 SYN 很久才到达服务端，而此时客户端已经重传了新的 SYN。如果只有两次握手，服务端会建立两个连接，但客户端只认可一个。

**为什么不是四次握手？**

服务端对 SYN 的确认（ACK）和自己的 SYN 可以合并成一次发送（SYN-ACK），所以三次是理论最优。

### 序列号（ISN）的秘密

每个 TCP 连接的初始序列号（Initial Sequence Number，ISN）不是从 0 或 1 开始的，而是一个随机值：

```java
// Linux 中 ISN 的生成算法（简化）
ISN = nonce + (timestamp << 6) + flags
// nonce 是一个安全的随机数
// timestamp 防止旧连接的包干扰新连接
```

为什么 ISN 要随机？

```
场景：客户端连接服务器，发送了数据后断开

错误假设 ISN = 0：

客户端 ----[seq=0, data="hello"]----> 服务端
客户端 断开连接
客户端 重新连接 ----[seq=0, data="world"]----> 服务端

问题：如果旧的数据包延迟到达服务端，
它可能被错误地认为是新连接的数据！
```

随机 ISN 可以防止这种情况，因为旧连接的序列号空间已经失效。

### Socket 状态变化

TCP 连接有 11 种状态，理解它们是排查网络问题的关键：

```
CLOSED
   |
   | socket()
   v
LISTEN  <-------> ESTABLISHED <-------> CLOSE_WAIT
   |                  ^                  |
   |                  |                  |
   | accept()        |                  | close()
   v                  |                  v
   |            ESTABLISHED <-------> LAST_ACK
   |                  |                  ^
   |                  |                  |
   |<--- SYN_RCVD ---|                  |
   |                  |                  |
   |<------- FIN_WAIT_2 <------ CLOSING
   |                  |
   |<----- FIN_WAIT_1 --|------|
   |                  |      |
   |             TIME_WAIT      |
   |<----------------------|
```

关键状态说明：

| 状态 | 含义 | 常见问题 |
|---|---|---|
| `LISTEN` | 服务端监听中 | |
| `SYN_SENT` | 客户端已发送 SYN | |
| `SYN_RCVD` | 服务端收到 SYN，回复 SYN-ACK | |
| `ESTABLISHED` | 三次握手完成，可以传数据 | 正常状态 |
| `FIN_WAIT_1` | 已发送 FIN，等待 ACK | |
| `FIN_WAIT_2` | 收到 ACK，等待对方 FIN | |
| `TIME_WAIT` | 等待 2MSL 后关闭 | 端口占用 |
| `CLOSE_WAIT` | 收到 FIN，等待应用关闭 | 连接泄漏 |
| `LAST_ACK` | 最后一次 ACK 发送中 | |
| `CLOSING` | 双方同时关闭 | |

## BIO Socket 服务端开发

### 基础模型

BIO（Blocking I/O）是传统的 Socket 编程模型，特点是**阻塞**：

```java
public class BioServer {
    public static void main(String[] args) throws IOException {
        // 1. 创建 ServerSocket，绑定端口
        ServerSocket serverSocket = new ServerSocket(8080);
        System.out.println("服务器启动，监听 8080 端口");

        while (true) {
            // 2. 阻塞等待客户端连接
            // 线程在这里卡住，直到有客户端连接
            Socket clientSocket = serverSocket.accept();
            System.out.println("新连接: " + clientSocket.getRemoteSocketAddress());

            // 3. 处理请求
            handleRequest(clientSocket);
        }
    }

    private static void handleRequest(Socket clientSocket) {
        try {
            // 获取输入流，读取客户端数据
            InputStream in = clientSocket.getInputStream();
            BufferedReader reader = new BufferedReader(
                new InputStreamReader(in));

            // 获取输出流，发送响应
            OutputStream out = clientSocket.getOutputStream();

            // 4. 阻塞读取：直到客户端发送数据或关闭连接
            String line;
            while ((line = reader.readLine()) != null) {
                System.out.println("收到: " + line);

                // 5. 阻塞写入：直到数据发送完成
                out.write(("Echo: " + line + "\n").getBytes());
                out.flush();
            }

            System.out.println("客户端断开: " + clientSocket);
        } catch (IOException e) {
            e.printStackTrace();
        } finally {
            try {
                clientSocket.close();
            } catch (IOException ignored) {}
        }
    }
}
```

### 问题：每个连接一个线程

BIO 模型的最大问题是**无法处理高并发**：

```
客户端1 -----> 服务端线程1 -----> 处理
客户端2 -----> 服务端线程2 -----> 处理
客户端3 -----> 服务端线程3 -----> 处理
...
客户端10000 --> 服务端线程10000 --> 等待...
```

如果 10000 个连接同时存在，就需要 10000 个线程，每个线程栈占用 1MB 内存，光栈空间就是 10GB。

### 多线程优化方案

**方案一：线程池**

```java
public class ThreadPoolServer {
    private static final int THREAD_POOL_SIZE = 100;

    public static void main(String[] args) throws IOException {
        ServerSocket serverSocket = new ServerSocket(8080);

        // 创建线程池
        ExecutorService executor = Executors.newFixedThreadPool(THREAD_POOL_SIZE);

        while (true) {
            Socket clientSocket = serverSocket.accept();

            // 提交到线程池处理，而不是每个连接一个线程
            executor.submit(() -> handleRequest(clientSocket));
        }
    }
}
```

**方案二：每个连接一个线程（传统方式）**

```java
public class PerConnectionServer {
    public static void main(String[] args) throws IOException {
        ServerSocket serverSocket = new ServerSocket(8080);

        while (true) {
            final Socket clientSocket = serverSocket.accept();

            // 为每个连接创建一个新线程
            new Thread(() -> {
                try {
                    handleRequest(clientSocket);
                } catch (IOException e) {
                    e.printStackTrace();
                }
            }).start();
        }
    }
}
```

### 连接泄漏：CLOSE_WAIT 的罪魁祸首

这是线上最常见的问题。

**问题复现**：

```java
public class LeakServer {
    public static void main(String[] args) throws IOException {
        ServerSocket serverSocket = new ServerSocket(8080);

        while (true) {
            Socket clientSocket = serverSocket.accept();

            // 问题：这里如果抛出异常，socket 不会被关闭
            // 连接会进入 CLOSE_WAIT 状态，永远不会释放
            handleRequest(clientSocket);
        }
    }
}

private static void handleRequest(Socket clientSocket) {
    try {
        // 如果在这里抛出异常，下面的 close() 不会执行
        throw new RuntimeException("处理出错");

        // 这行代码不会执行
        // clientSocket.close();
    } catch (Exception e) {
        // 只打印日志，不关闭连接
        e.printStackTrace();
    }
}
```

**正确的做法：finally 块关闭**

```java
private static void handleRequest(Socket clientSocket) {
    try {
        InputStream in = clientSocket.getInputStream();
        OutputStream out = clientSocket.getOutputStream();

        // 业务处理
        byte[] buffer = new byte[1024];
        int len = in.read(buffer);

        if (len > 0) {
            out.write(buffer, 0, len);
            out.flush();
        }
    } catch (IOException e) {
        e.printStackTrace();
    } finally {
        // 必须在这里关闭 socket
        try {
            if (clientSocket != null) {
                clientSocket.close();
            }
        } catch (IOException ignored) {}
    }
}
```

### 完整的多线程服务端示例

```java
public class BioServerDemo {
    private static final int PORT = 8080;
    private static final int THREAD_POOL_SIZE = 100;

    public static void main(String[] args) throws IOException {
        // 创建带线程池的服务端
        ExecutorService threadPool = Executors.newFixedThreadPool(THREAD_POOL_SIZE);

        ServerSocket serverSocket = new ServerSocket(PORT);
        System.out.println("BIO 服务器启动，端口: " + PORT);

        try {
            while (true) {
                Socket clientSocket = null;
                try {
                    // 阻塞等待连接
                    clientSocket = serverSocket.accept();
                    System.out.println("连接建立: " + clientSocket.getRemoteSocketAddress());

                    // 设置超时，防止客户端恶意连接
                    clientSocket.setSoTimeout(30000);  // 30秒超时

                    // 提交到线程池处理
                    threadPool.submit(() -> handleClient(clientSocket));
                } catch (IOException e) {
                    // 连接接受失败
                    System.err.println("接受连接失败: " + e.getMessage());
                }
            }
        } finally {
            threadPool.shutdown();
            serverSocket.close();
        }
    }

    private static void handleClient(Socket clientSocket) {
        try (clientSocket) {
            InputStream in = clientSocket.getInputStream();
            OutputStream out = clientSocket.getOutputStream();

            // 使用 ByteBuffer 处理数据
            ByteBuffer buffer = ByteBuffer.allocate(1024);

            while (true) {
                int read = in.read(buffer.array());

                if (read == -1) {
                    // 客户端关闭连接
                    System.out.println("客户端关闭: " + clientSocket.getRemoteSocketAddress());
                    break;
                }

                buffer.flip();
                out.write(buffer.array(), 0, read);
                out.flush();
                buffer.clear();
            }
        } catch (SocketTimeoutException e) {
            System.out.println("读取超时: " + clientSocket.getRemoteSocketAddress());
        } catch (IOException e) {
            System.err.println("处理异常: " + e.getMessage());
        }
    }
}
```

## Socket 选项与调优

### 常用选项

```java
Socket socket = serverSocket.accept();

// 设置读写缓冲区大小
socket.setReceiveBufferSize(64 * 1024);  // 接收缓冲区 64KB
socket.setSendBufferSize(64 * 1024);     // 发送缓冲区 64KB

// SO_TIMEOUT：读取超时时间
socket.setSoTimeout(30000);  // 30秒无数据抛 SocketTimeoutException

// SO_KEEPALIVE：TCP 保活
socket.setKeepAlive(true);  // 检测对端是否存活（2小时探测一次）

// SO_REUSEADDR：端口复用
// 解决 TIME_WAIT 状态下端口无法立即使用的问题
serverSocket.setReuseAddress(true);

// TCP_NODELAY：禁用 Nagle 算法
socket.setTcpNoDelay(true);  // 小数据立即发送，不等待合并
```

### TIME_WAIT 问题

当服务端主动关闭连接时，会进入 `TIME_WAIT` 状态，持续 2MSL（Maximum Segment Lifetime，通常 60 秒）：

```
客户端                        服务端
   |                             |
   |========= FIN_WAIT =========>|
   |<======== ACK ===============|
   |<======== FIN ===============|
   |========= ACK ===============>|
   |                             |
   |     TIME_WAIT (等待 2MSL)    |
   |                             |
```

在这段时间内，服务器端口无法被绑定。如果服务端频繁重启，会出现 `Address already in use` 错误。

**解决方案**：

```java
ServerSocket serverSocket = new ServerSocket(PORT);

// 开启端口复用
serverSocket.setReuseAddress(true);
```

## TCP 连接状态排查命令

### netstat 查看连接状态

```bash
# 查看所有连接，按状态分组
netstat -ant | awk '{print $6}' | sort | uniq -c | sort -rn

# 结果示例
#    100 ESTABLISHED
#     50 CLOSE_WAIT
#     20 TIME_WAIT
#      5 LISTEN
```

### ss 命令（更现代）

```bash
# 查看连接状态统计
ss -s

# 查看特定端口的连接
ss -ant | grep 8080

# 查看 CLOSE_WAIT 连接
ss -ant state close-wait | head -20
```

### 分析 CLOSE_WAIT 的原因

```bash
# 找出 CLOSE_WAIT 连接的 PID
ss -antp | grep CLOSE_WAIT

# 结果示例
# CLOSE-WAIT 192.168.1.100:54321 192.168.1.10:8080 users:(("java",pid=12345,fd=45))

# 查看这个进程的堆栈
jstack 12345
```

## 总结：BIO Socket 开发 checklist

| 检查项 | 说明 |
|---|---|
| 连接关闭 | 是否在 finally 块中关闭 Socket？ |
| 超时设置 | 是否设置了合理的 SO_TIMEOUT？ |
| 缓冲区大小 | 读写缓冲区是否足够？ |
| 线程池 | 是否使用线程池而非每连接一线程？ |
| 异常处理 | 异常是否正确处理，不吞掉？ |
| 资源泄漏 | 是否有 CLOSE_WAIT 堆积？ |

BIO Socket 是理解网络编程的基础，虽然在高性能场景下已被 NIO/AIO 取代，但其中的原理和问题是理解其他模型的前提。下一节我们会看到 NIO 如何通过多路复用解决 BIO 的并发问题。
