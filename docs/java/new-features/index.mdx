# Java 新特性

面试官问："Java 8 都用过哪些新特性？"

候选人小王说："用过 Lambda 表达式和 Stream API。"

面试官追问："Stream 和 for 循环比性能差多少？什么场景下不推荐用？"

小王说："...应该差不多吧？"

面试官："那 Optional 呢？用过吗？"

小王说："用过，但感觉没什么用，还是喜欢 if(obj != null) ..."

面试官没说话，在本子上写了点什么。

---

Java 新特性不是"新版本出了哪些 API"，而是要理解**每一个特性背后的设计动机和适用场景**。

Lambda 不只是语法糖，它开启了函数式编程的大门；Stream 不只是 for 循环的替代品，它带来的是思维方式的变化；虚拟线程更是一次底层模型的革命，它改变了 Java 并发编程的基本范式。

这一模块，带你把 Java 新特性从"听说过"升级到"能讲清楚演进原因、适用场景和工程取舍"。

## 学习路径指引

### P5 候选人（校招/初级社招）

Java 新特性的入门关是**会用 JDK8 的核心特性，并理解背后的设计动机**。

**P5 必过关卡**：

1. Lambda 表达式的核心是什么？函数式接口是什么？常见的四大函数式接口分别是什么？
2. Stream 和 for 循环的区别是什么？Stream 的惰性求值是怎么实现的？
3. Optional 解决了什么问题？常用方法有哪些？有哪些误用方式？
4. 什么是接口默认方法？为什么引入它？

这四道题能流畅答出来，JDK8 新特性就算过关。

### P6 候选人（中级社招）

P6 的门槛在**能讲清楚新特性的底层实现、性能影响、以及选型权衡**。

**拉开差距的追问链（Stream）**：

```
Stream 和 for 循环哪个性能更好？
  → Stream 的惰性求值是怎么实现的？pipeline 是怎么连接的？
    → 什么时候用串行流，什么时候用并行流？
      → 并行流的原理是什么？（ForkJoinPool）有什么坑？
        → parallelStream 一定比 stream 快吗？什么场景下反而更慢？
```

这条追问链能走完的，基本都深入看过 Stream 的实现。

**拉开差距的追问链（Optional）**：

```
Optional 相比 null 检查有什么优势？
  → Optional 的常用方法：map/flatMap/filter/orElse/orElseGet/orElseThrow
    → Optional 有哪些误用方式？（Optional 作为字段、Optional.ofNullable 滥用）
      → Optional 内部是怎么实现的？Optional.empty() 是单例吗？
```

**拉开差距的追问链（虚拟线程）**：

```
虚拟线程和传统线程有什么区别？
  → 虚拟线程为什么能大幅提升并发数量？原理是什么？（Continuation）
    → 虚拟线程和 Goroutine 有什么区别？各自的优势是什么？
      → 虚拟线程的调度器是什么？Carrier Thread 是怎么管理的？
        → 虚拟线程适合什么场景？不适合什么场景？
          → ThreadLocal 在虚拟线程里有什么变化？
```

【面试官心理】
Java 新特性的题，我从 JDK8 的 Lambda 和 Stream 开始。如果候选人基础扎实，我会追问虚拟线程这个最新的重磅特性。Lambda 和 Stream 考的是对函数式编程的理解，虚拟线程考的是对并发模型演进的认知。

### P7 候选人（高级社招/架构方向）

P7 的新特性是在考**技术判断力和前瞻性思维**。

**典型的 P7 级问题**：

- Java 的模块化系统为什么没有被广泛采用？它的设计目标是什么？遇到了哪些阻力？
- Record 和 Lombok 的 @Data 有什么区别？为什么 JDK 要引入 Record？
- Sealed 类配合 Pattern Matching，是不是在向函数式靠拢？Java 的函数式之路走得怎么样？
- 虚拟线程出来后，ThreadPoolExecutor 会被替代吗？未来的并发编程模型会怎么演进？

:::tip 💡
虚拟线程并不是银弹。它适合 IO 密集型任务（大量等待网络请求、数据库查询），但不适合 CPU 密集型任务。Carrier Thread 仍然是普通线程，CPU 密集型任务用虚拟线程反而会增加调度开销。
:::

## 生产避坑指南

Java 新特性用错了，性能反而更差：

- **Stream 滥用**：在循环内用 Stream（每次创建流对象）、并行流用在非线程安全集合上、链式太长可读性极差。
- **Optional 误用**：用 Optional 包装 null、用 Optional 作为方法参数、用 Optional.map().map() 嵌套多层。
- **虚拟线程误用**：在虚拟线程里再用线程池、用 synchronized 阻塞操作（会 pin 住 Carrier Thread）、ThreadLocal 的共享问题。
- **接口默认方法冲突**：多重继承时方法冲突解决规则复杂，容易出 bug。
- **旧日期 API**：继续用 java.util.Date（线程不安全、月份从0开始）而不是 java.time API。

:::warning ⚠️
虚拟线程里有一个经典陷阱：如果在 synchronized 块中阻塞（比如调用 `object.wait()` 或阻塞式 IO），虚拟线程会 pin 住底层的 Carrier Thread，导致可用线程数骤减。建议用 `ReentrantLock` + `Condition` 替代 synchronized wait/notify。
:::
