# JDK 12~16 新特性

你有没有注意到一个问题——Java 的 switch 语句，已经几十年没有变化了？

从 C 和 C++ 继承来的 switch 语法，陪伴了 Java 开发者无数个日夜。但这套语法的局限性也越来越明显：无法作为表达式返回值、每个 case 都需要 break 防止穿透、只能匹配常量不能匹配模式……当 Kotlin 和 Scala 早已拥有更优雅的模式匹配时，Java 的 switch 显得格外老旧。

好消息是：从 JDK 12 开始，Java 开启了一场"现代化改造"运动。Switch 表达式、Records、文本块、密封类……每隔半年，一个新特性就会与开发者见面。这一节，我们来梳理 JDK 12 到 JDK 16 期间那些改变编码体验的重要新特性。

## Switch 表达式：告别 break 地狱

Switch 的 case 穿透（fall-through）行为，大概是 Java 最容易出 bug 的语法特性之一。忘记写 break？恭喜你，代码"愉快地"执行了下一个 case 的逻辑。这种 bug 在遗留代码中尤其隐蔽，常常在代码审查中被忽略，直到上了生产才被发现。

JDK 12 引入的 Switch 表达式（预览，JDK 14 正式），从根本上解决了这个问题：

```java
// 传统 switch（容易漏写 break）
int days = 0;
switch (month) {
    case 1:
    case 3:
    case 5:
    case 7:
    case 8:
    case 10:
    case 12:
        days = 31;
        break;
    case 4:
    case 6:
    case 9:
    case 11:
        days = 30;
        break;
    case 2:
        days = isLeapYear ? 29 : 28;
        break;
}

// 新的 switch 表达式——箭头语法，不会 fall-through
int days = switch (month) {
    case 1, 3, 5, 7, 8, 10, 12 -> 31;
    case 4, 6, 9, 11 -> 30;
    case 2 -> isLeapYear ? 29 : 28;
    default -> 0;
};
```

箭头语法（`->`）的每个 case 不会穿透，逻辑清晰。同时，多个常量可以合并到一个 case（`case 1, 3, 5`），大大减少了重复代码。而 switch 本身可以作为表达式返回值使用，赋值给变量，这使得函数式编程风格的处理更加自然。

更重要的是，JDK 14 引入的**模式匹配**让 switch 不再只能匹配常量——它可以匹配类型、匹配 null、匹配结构。这为后续的模式匹配增强埋下了伏笔（后文详细讲解）。

## 文本块：从字符串拼接地狱中解脱

你有没有写过这样的代码？

```java
String json = "{\n" +
    "  \"name\": \"Alice\",\n" +
    "  \"age\": 30,\n" +
    "  \"city\": \"Beijing\"\n" +
"}";
```

一行 JSON 需要大量 `+` 和 `\n`，代码可读性极差。而 SQL 查询、HTML 片段、日志格式等场景，文本拼接的痛苦有过之而无不及。

JDK 15 引入的**文本块**（Text Blocks），用三引号 `"""` 包裹原始字符串，让格式化文本终于可以"所见即所得"：

```java
String json = """
    {
      "name": "Alice",
      "age": 30,
      "city": "Beijing"
    }
    """;
```

这个改进看起来简单，但实际使用中大幅提升了代码可读性和维护效率。编译器会自动处理行尾换行，最终生成的字符串与手动拼接的结果完全一致。

文本块还有一个强大特性：**字符串格式化**。传统上，多行字符串拼接后替换占位符非常繁琐。JDK 15 引入的 `String::formatted` 方法和 `%s` 占位符语法，让模板填充变得优雅：

```java
String query = """
    SELECT * FROM users
    WHERE name = '%s'
      AND age > %d
    ORDER BY created_at
    """.formatted(username, age);
```

不过需要注意：虽然 `%s` 可以处理字符串格式化，但在 SQL 场景下仍然应该使用 PreparedStatement 的参数化查询来防止 SQL 注入——`formatted` 只是语法层面的便利，不是安全的替代方案。

## Records：告别 POJO 模板代码

如果让你写一个"数据传输对象"（DTO），比如一个 `Point` 类包含 x 和 y 坐标，然后还要 equals()、hashCode()、toString()、getter 方法……你能忍住不复制粘贴吗？

```java
public class Point {
    private final int x;
    private final int y;

    public Point(int x, int y) {
        this.x = x;
        this.y = y;
    }

    public int x() { return x; }  // getter，不是 getX()
    public int y() { return y; }

    @Override
    public boolean equals(Object o) {
        if (!(o instanceof Point)) return false;
        Point p = (Point) o;
        return x == p.x && y == p.y;
    }

    @Override
    public int hashCode() {
        return Objects.hash(x, y);
    }

    @Override
    public String toString() {
        return "Point[x=" + x + ", y=" + y + "]";
    }
}
```

JDK 14 预览、JDK 16 正式发布的 **Records**，把这件事压缩成了一行：

```java
public record Point(int x, int y) {}
```

这行代码自动提供了：构造函数、`x()` 和 `y()` 两个 accessor 方法（注意不是 `getX()`）、`equals()` 和 `hashCode()`（基于数据内容比较）、`toString()`（打印所有字段）。Record 天生是**不可变**的——所有字段都是 final 的，没有 setter。

Record 的引入不仅仅是省代码。背后更深的含义是：**明确区分"值对象"和"行为对象"**。Record 代表的是纯数据，不应该有复杂行为逻辑；而传统 class 则保留给需要封装复杂业务逻辑的对象。这种区分在领域驱动设计（DDD）中早有概念，Java 终于在语法层面提供了工具支持。

Record 还可以定义自定义构造函数、添加额外方法和静态字段：

```java
public record Point(int x, int y) {
    // 自定义构造函数：校验逻辑
    public Point {
        if (x < 0 || y < 0) {
            throw new IllegalArgumentException("Coordinates must be non-negative");
        }
    }

    // 添加方法
    public double distanceFromOrigin() {
        return Math.sqrt(x * x + y * y);
    }

    // 静态字段和方法
    public static Point ORIGIN = new Point(0, 0);
}
```

## instanceof 模式匹配：告别强制类型转换

又一个令人窒息的 Java 语法——强制类型转换。`instanceof` 检查之后，如果不转换就无法使用子类的特有方法。代码写起来繁琐，阅读起来更繁琐：

```java
// JDK 14 之前：先 instanceof 检查，再强制转换
if (obj instanceof String) {
    String s = (String) obj;
    System.out.println(s.length()); // 终于可以用 String 的方法了
}
```

这有什么问题？问题在于：这种模式太常见了，每次都要写两遍类型名称。更重要的是，如果你在 if 块中忘记使用 `s`，而误用了 `obj`，编译器不会报错——因为 `obj` 依然是 `Object` 类型，只不过你能确定它是 `String`。

JDK 14 预览、JDK 16 正式的 **instanceof 模式匹配**，让这个模式彻底优雅化：

```java
// JDK 16+：模式变量直接可用
if (obj instanceof String s) {
    System.out.println(s.length()); // s 在这个作用域内就是 String 类型
}
```

注意这里的 `s` 不是普通变量——它是**模式变量**（Pattern Variable）。编译器在 `instanceof` 检查通过后，会将 `s` 的类型收窄为 `String`，无需手动转换。这种"检查 + 绑定"合二为一的模式，大幅减少了样板代码，同时提升了类型安全性。

结合 switch 表达式，instanceof 模式匹配可以写出非常优雅的类型分发逻辑：

```java
static String formatter(Object obj) {
    return switch (obj) {
        case Integer i -> String.format("int %d", i);
        case String s -> String.format("String '%s'", s);
        case null -> "null";
        case CaseFormat cf -> cf.format();
        default -> obj.toString();
    };
}
```

## 密封类：类型系统的最后一块拼图

想象一个支付系统的场景：`Payment` 接口有多种实现：`CreditCardPayment`、`AlipayPayment`、`WechatPayPayment`。现在有个处理逻辑要根据具体支付类型做不同处理：

```java
void process(Payment payment) {
    if (payment instanceof CreditCardPayment) {
        // 处理信用卡支付
    } else if (payment instanceof AlipayPayment) {
        // 处理支付宝
    } else if (payment instanceof WechatPayPayment) {
        // 处理微信支付
    } else {
        // 未来如果再加一个新支付方式，这个分支永远不会命中
    }
}
```

问题来了：随着业务发展，新的支付方式不断加入。如果某个新同事添加了一个 `UnionPayPayment` 但忘记在这个 switch 里加分支，会发生什么？编译通过，运行时不报错，但新支付方式的处理逻辑就"静默消失"了——用户付款成功了，但处理逻辑没执行，后果不堪设想。

JDK 15 预览、JDK 17 正式的**密封类**（Sealed Classes），就是为了解决这个问题：

```java
public sealed class Shape permits Circle, Rectangle, Square {}

// 哪些类可以继承/实现 Shape？
final class Circle extends Shape {}           // final：不可被继承
non-sealed class Rectangle extends Shape {}  // non-sealed：自由继承
sealed class Square extends Shape permits ColoredSquare {}
```

密封类通过 `sealed` 修饰符和 `permits` 子句，**明确声明了所有被允许的子类**。这带来了两个核心优势：第一，穷尽性检查——编译器可以确保 switch 表达式覆盖了所有可能的情况，遗漏任何子类都会触发编译错误；第二，可控继承——通过 `final`、`sealed`、`non-sealed` 三种修饰符的组合，父类可以精确控制子类体系的扩展性，防止未授权的扩展。

这个特性与 Record 配合使用格外优雅。密封接口+记录类型+模式匹配，构成了一套完整的**代数数据类型**（ADT）体系，这在函数式编程语言中早已是标准配置：

```java
// JDK 17+：密封类 + Record + 模式匹配
public sealed interface Expr permits Number, Add, Multiply {}

// 纯数据表达式
public record Number(int value) implements Expr {}
public record Add(Expr left, Expr right) implements Expr {}
public record Multiply(Expr left, Expr right) implements Expr {}

// 编译器保证穷尽：所有 Expr 类型都被覆盖
int eval(Expr e) {
    return switch (e) {
        case Number n -> n.value();
        case Add a -> eval(a.left()) + eval(a.right());
        case Multiply m -> eval(m.left()) * eval(m.right());
    };
}
```

---

JDK 12 到 JDK 16 这五年，Java 的语法现代化程度超过了此前二十年的总和。Switch 表达式让分支逻辑更清晰，文本块让格式化字符串不再痛苦，Records 让数据类告别模板代码，instanceof 模式匹配让类型检查和绑定一步到位，密封类让类型系统更安全。

但这只是开始。JDK 17 之后的虚拟线程、完整的模式匹配、字符串模板……Java 正在以惊人的速度追赶着现代编程语言的步伐。接下来，让我们把目光投向 LTS 版本 JDK 17~21，看看这些特性如何在生产环境中发挥作用。
