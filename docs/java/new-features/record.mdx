## 一、Record 是什么？🔴

### 1.1 问题拆解

Record是Java 16引入的**预览特性**（JEP 395），用于简化**值对象**（Value Object）的创建。这道题考察候选人对Java语言演进的理解深度。

**第一层：基本概念**
- 面试官问："Record是用来做什么的？"
- 考察点：不可变数据载体、减少样板代码

**第二层：语法与限制**
- 面试官追问："Record有什么限制？不能做什么？"
- 考察点：不能继承、无setter、隐式final

**第三层：底层实现**
- 面试官继续追问："Record的equals/hashCode/toString是怎么生成的？和手写比有区别吗？"
- 考察点：字节码内幕、invokedynamic

**第四层：性能与选型**
- 面试官问："Record和Lombok的@Data比，有什么区别？什么时候用Record？"
- 考察点：不可变性、性能、工具链兼容性

### 1.2 ❌ 错误示范

**错误回答1："Record就是Java的data class，和Kotlin一样"**

不完全对。Record和Kotlin data class有相似之处，但关键区别是：Record**不能继承**，不能有可变字段，是**彻底不可变**的。

**错误回答2："Record自动生成的equals和手写的一模一样"**

不完全一样。Record的equals/hashCode/toString是编译器**强制的标准化实现**，而且equals的比较方式是**组件逐个比较**，和手写的通常一致。

**错误回答3："Record比普通类性能更好"**

性能基本相同，但Record由于是隐式final + 不可变，JIT编译器可以做更多优化（比如激进内联）。不过这个优势在实践中非常有限。

【面试官心理】
Record这道题我通常用来试探候选人对Java 14~16新特性的了解程度，以及他对"不可变性"这个重要设计原则的理解。能说出Record限制（不能继承、无setter）和适用场景的占30%，能说出和Lombok对比的占15%，能说出底层实现的占5%。

### 1.3 标准回答

#### 第一步：Record解决什么问题

```java
// Java 16之前：定义一个简单的值对象需要大量样板代码
public final class Point {
    private final int x;
    private final int y;

    public Point(int x, int y) {
        this.x = x;
        this.y = y;
    }

    public int x() { return x; } // getter不叫getX()，叫x()
    public int y() { return y; }

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof Point)) return false;
        Point point = (Point) o;
        return x == point.x && y == point.y;
    }

    @Override
    public int hashCode() {
        return Objects.hash(x, y);
    }

    @Override
    public String toString() {
        return "Point{x=" + x + ", y=" + y + "}";
    }
}
```

```java
// Record：同样的功能，一行搞定
public record Point(int x, int y) { }

// 编译器自动生成：
// - 构造器 Point(int x, int y)
// - getter方法 x() 和 y()（不是getX()，是x()）
// - equals/hashCode/toString
// - 私有final字段 x, y
```

#### 第二步：Record的隐式声明

Record自动生成的组件：

| 组件 | Record自动生成 | 说明 |
| --- | --- | --- |
| 字段 | ✅ `private final T name` | 每一个record组件对应一个字段 |
| 构造器 | ✅ 紧凑构造器 | 参数列表等于组件列表 |
| getter | ✅ `T name()` | 方法名就是组件名，不是`getName()` |
| equals | ✅ 逐组件比较 | `this.x == o.x && this.y == o.y` |
| hashCode | ✅ 逐组件hash | `Objects.hash(x, y)` |
| toString | ✅ 格式化的字符串 | `Point[x=1, y=2]` |

#### 第三步：Record的限制

```java
// 限制1：不能继承——Record是隐式final的
public record BasePoint(int x, int y) {}
public record ExtendedPoint(int x, int y, int z) {}
// 可以implements接口，但不能extends任何类
public record IntPoint(int x, int y) implements Comparable<IntPoint> {}

// 限制2：字段不可变
public record Point(int x, int y) {
    public void setX(int x) {  // ❌ 编译错误
        this.x = x;
    }
}

// 限制3：字段不能有初始化器
public record Point(int x, int y) {
    private int z = 0; // ❌ 编译错误：record不能有实例字段
}

// 限制4：record组件（参数）不能是可变类型（警告）
public record Holder(List<String> list) {} // ⚠️ 警告：list本身不可变，但引用类型内容可变
```

#### 第四步：紧凑构造器——validation

```java
// 标准构造器写法
public record Point(int x, int y) {
    public Point(int x, int y) {
        if (x < 0 || y < 0) {
            throw new IllegalArgumentException("坐标不能为负");
        }
        this.x = x;
        this.y = y;
    }
}

// 紧凑构造器写法（Java 16+）
public record Point(int x, int y) {
    // 无参数，无主体，隐式赋值给字段
    // 但可以加validation块
    public Point {
        if (x < 0 || y < 0) {
            throw new IllegalArgumentException("坐标不能为负");
        }
        // 隐式：this.x = x; this.y = y;
    }
}
```

## 二、Record vs Lombok vs 普通POJO 🟡

### 2.1 对比表

| 特性 | Record | Lombok @Data | 普通POJO |
| --- | --- | --- | --- |
| 不可变性 | ✅ 隐式final | ❌ 可变（需@Value） | ❌ 手动控制 |
| equals/hashCode | ✅ 自动标准实现 | ✅ 自动（但不够标准） | ❌ 手动 |
| toString | ✅ 自动 | ✅ 自动 | ❌ 手动 |
| 继承 | ❌ 不支持 | ✅ 支持 | ✅ 支持 |
| 泛型 | ✅ 支持 | ⚠️ 部分支持 | ✅ 支持 |
| 性能 | ✅ JIT优化友好 | ✅ 同普通类 | ✅ 同Record |
| 编译时依赖 | ❌ JDK 16+ | ❌ 需要注解处理器 | ✅ 无 |

### 2.2 Record vs Lombok @Value

```java
// Lombok @Value：效果类似Record，但字段可变
@Value
public class Point {
    int x;
    int y;
    // 会生成：final字段 + 全参数构造器 + equals/hashCode/toString + getters
    // 但Point对象仍然可以通过反射修改（如果构造器不检查的话）
}

// Record：彻底不可变
public record Point(int x, int y) {
    // 隐式final字段
    // 构造器可以加validation
    // 不可变：JVM级别保证
}
```

:::warning ⚠️
**Record的反序列化注意**：Record的构造器参数顺序很重要。Jackson默认按JSON字段名匹配，JDK 17+的JSON provider也支持Record的序列化。但如果用第三方JSON库，需要确认兼容性。
:::

### 2.3 Record的字节码实现

```java
public record Point(int x, int y) {}

// 编译后等价的字节码（简化）
public final class Point extends Record {
    private final int x;
    private final int y;

    // 构造器
    public Point(int x, int y) {
        this.x = x;
        this.y = y;
    }

    // getter
    public int x() { return this.x; }
    public int y() { return this.y; }

    // equals（标准化实现）
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof Point p)) return false;
        return this.x == p.x && this.y == p.y;
    }

    // hashCode（标准化实现）
    public int hashCode() {
        return Integer.hashCode(x) * 31 + Integer.hashCode(y);
    }

    // toString（标准化实现）
    public String toString() {
        return "Point[x=" + x + ", y=" + y + "]";
    }
}
```

:::tip 💡
**Record的equals使用模式匹配**：`o instanceof Point p`在字节码层面就是JDK 16引入的`pattern matching for instanceof`（JEP 394），这意味着Record的equals实现比手写的`instanceof + cast`更高效。
:::

## 三、Record 适用场景与最佳实践 🟡

### 3.1 最佳适用场景

| 场景 | 推荐 | 原因 |
| --- | --- | --- |
| DTO（数据传输对象） | ✅ | 天然不可变，适合跨边界传递 |
| 函数返回值 | ✅ | 多个值一起返回 |
| Map.entry | ✅ | 键值对天然适合record |
| 配置对象 | ✅ | 只读配置 |
| 模式匹配（JEP 439+） | ✅ | Record可以和switch/sealed配合 |
| 实体类（JPA） | ❌ | JPA需要默认构造器和setter |
| 有可变状态的业务对象 | ❌ | Record不适合 |

### 3.2 Record在模式匹配中的价值

```java
// Java 21+：Record可以用于switch的模式匹配
sealed interface Result permits Success, Failure {}

record Success(Object data) implements Result {}
record Failure(String message) implements Result {}

String describe(Result r) {
    return switch (r) {
        case Success(Object data) -> "Data: " + data;
        case Failure(String msg) -> "Error: " + msg;
    };
}
```

这里`Success(Object data)`就是**record的模式匹配**——可以直接解构Record的组件作为匹配条件。

### 3.3 序列化支持

```java
// Jackson序列化（需要jackson-modules-java8）
ObjectMapper mapper = new ObjectMapper();
mapper.registerModule(new JavaTimeModule());

Point p = new Point(1, 2);
String json = mapper.writeValueAsString(p);
// {"x":1,"y":2}

// 反序列化
Point p2 = mapper.readValue(json, Point.class);
```
