## 一、为什么需要新的日期时间API？🔴

### 1.1 问题拆解

这道题是**P6高频题**，考察候选人对Java历史遗留问题的理解深度。

**第一层：线程安全问题**
- 面试官问："SimpleDateFormat为什么不是线程安全的？"
- 考察点：Calendar内部状态、共享可变状态

**第二层：新API设计**
- 面试官追问："Java 8的新日期API是怎么解决这个问题的？"
- 考察点：不可变对象、线程安全、职责分离

**第三层：API差异**
- 面试官继续问："LocalDate、LocalDateTime、ZonedDateTime有什么区别？什么时候用哪个？"
- 考察点：时区概念、业务场景

**第四层：格式化与解析**
- 面试官问："DateTimeFormatter怎么用？和SimpleDateFormat比有什么优势？"
- 考察点：线程安全、格式化选项、ISO标准

### 1.2 ❌ 错误示范

**错误回答1："SimpleDateFormat线程不安全是因为它内部用了Calendar"**

这个答案只说对了一半。更准确的说法是：SimpleDateFormat的内部状态（Calendar实例）是**可变共享**的——多个线程调用同一个SimpleDateFormat实例会相互覆盖Calendar状态。

**错误回答2："LocalDate比Date更好，所以Date要被废弃了"**

Java 8引入新API不是为了"废弃"旧API，而是因为旧API**设计上就有缺陷**。LocalDate/DateTimeFormatter的核心优势在于**不可变性和线程安全**，但Date仍然在Java生态中大量使用（尤其是JDBC、序列化场景）。

**错误回答3："DateTimeFormatter就是线程安全的SimpleDateFormat"**

不完全对。DateTimeFormatter的线程安全性来自于**不可变性**（所有字段都是final的），而不是像SimpleDateFormat那样依赖ThreadLocal来掩盖问题。

### 1.3 标准回答

#### 第一步：SimpleDateFormat的线程安全问题根源

SimpleDateFormat内部维护了一个`Calendar`实例，这个Calendar是**可变对象**：

```java
// SimpleDateFormat内部（简化）
public class SimpleDateFormat {
    private Calendar calendar; // 可变状态！

    public Date parse(String source) {
        // 使用calendar解析...
    }

    public StringBuffer format(Date date, StringBuffer toAppendTo, FieldDelegate delegate) {
        // 使用calendar格式化...
    }
}
```

当多个线程共享同一个SimpleDateFormat实例时：

```java
// 反面教材：共享SimpleDateFormat
public class DateUtil {
    private static final SimpleDateFormat SDF = new SimpleDateFormat("yyyy-MM-dd");

    public static Date parse(String dateStr) {
        return SDF.parse(dateStr); // 线程A在用calendar，线程B也用了
        // 结果：日期错乱、数据污染，甚至抛异常
    }
}
```

并发场景下，线程A刚刚设置好`calendar.setTime(dateA)`，线程B立刻调用`parse()`把自己的时间设进去，线程A再使用时calendar里的值已经被覆盖了。

**三种解决方案对比**：

| 方案 | 优点 | 缺点 |
| --- | --- | --- |
| ThreadLocal包装 | 简单、改动小 | 每个线程都有自己的SDF实例，内存开销 |
| 每次new | 线程安全 | 频繁创建对象，GC压力大 |
| DateTimeFormatter | 不可变、线程安全、设计合理 | 需要迁移代码，依赖Java 8+ |

#### 第二步：Java 8新日期API的设计哲学

Java 8引入的`java.time`包有三大核心设计原则：

**1. 不可变性**：所有核心类都是不可变的，线程安全

```java
// 不可变性示例
LocalDate date = LocalDate.of(2024, 1, 1);
date.plusDays(1); // 不修改date，返回新的LocalDate
System.out.println(date); // 仍然是2024-01-01
```

**2. 职责分离**：不同的概念用不同的类表示

| 类 | 概念 | 例子 |
| --- | --- | --- |
| `LocalDate` | 日期（年月日） | 2024-01-01 |
| `LocalTime` | 时间（时分秒） | 13:45:30 |
| `LocalDateTime` | 日期+时间 | 2024-01-01 13:45:30 |
| `ZonedDateTime` | 带时区的日期时间 | 2024-01-01T13:45:30+08:00 |
| `Instant` | 时间戳（UTC） | 2024-01-01T05:45:30Z |
| `Duration` | 时间段（秒/纳秒） | PT1H30M |
| `Period` | 日期段（年/月/日） | P1M2D |

```java
// 职责分离的好处
LocalDate date = LocalDate.now();              // 只有日期
LocalTime time = LocalTime.now();              // 只有时间
LocalDateTime dateTime = LocalDateTime.now(); // 日期+时间
ZonedDateTime zoned = ZonedDateTime.now(ZoneId.of("America/New_York")); // 带时区
```

**3. ISO 8601标准**：默认遵循国际标准

```java
// ISO 8601格式是默认的
LocalDate.parse("2024-01-01");                 // ISO date
LocalTime.parse("13:45:30");                   // ISO time
Instant.parse("2024-01-01T05:45:30Z");        // ISO timestamp
```

#### 第三步：DateTimeFormatter——线程安全的格式化

```java
// SimpleDateFormat线程安全问题
// 每次parse/format都要操作共享的Calendar，线程间相互覆盖

// DateTimeFormatter：不可变，线程安全
DateTimeFormatter formatter = DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm:ss");
String str = LocalDateTime.now().format(formatter); // 线程安全！

// 可以安全地定义为static final
public static final DateTimeFormatter DATE_TIME_FORMATTER =
    DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm:ss");
```

**格式化模式字符差异**：

| SimpleDateFormat | DateTimeFormatter | 说明 |
| --- | --- | --- |
| `y` | `y` | 年 |
| `M` | `M` | 月 |
| `d` | `d` | 日 |
| `H` | `H` | 24小时制 |
| `h` | `h` | 12小时制 |
| `m` | `m` | 分 |
| `s` | `s` | 秒 |

:::tip 💡
**注意**：`SimpleDateFormat`的`M`是月份，`DateTimeFormatter`的`M`也是月份。但SimpleDateFormat中`y`代表year（西方习惯），而中文环境下有时需要特别注意年份格式。
:::

```java
// DateTimeFormatter的常用模式
DateTimeFormatter d1 = DateTimeFormatter.ofPattern("yyyy-MM-dd");
DateTimeFormatter d2 = DateTimeFormatter.ofPattern("yyyy年MM月dd日");
DateTimeFormatter d3 = DateTimeFormatter.ofPattern("HH:mm:ss");
DateTimeFormatter d4 = DateTimeFormatter.ofPattern("yyyy-MM-dd'T'HH:mm:ssXXX"); // ISO with timezone
```

## 二、时区处理——最容易翻车的点 🟡

### 2.1 时区问题的本质

面试官经常追问："用户在北京创建了一条订单，传到纽约的服务器上，显示时间会变吗？"

这个问题考的是**时区和时间戳的区别**：

```java
// 时间戳：UTC毫秒数，全球唯一
Instant instant = Instant.now(); // 2024-01-01T05:00:00Z（UTC时间）

// 同一时刻，北京时间 = UTC + 8小时
ZonedDateTime beijingTime = instant.atZone(ZoneId.of("Asia/Shanghai"));
// 2024-01-01T13:00+08:00

// 同一时刻，纽约时间 = UTC - 5小时
ZonedDateTime nyTime = instant.atZone(ZoneId.of("America/New_York"));
// 2024-01-01T08:00-05:00

// 同一个Instant.toEpochMilli()：值完全相同，只是表示方式不同
System.out.println(beijingTime.toInstant().toEpochMilli() == nyTime.toInstant().toEpochMilli()); // true
```

:::warning ⚠️
**最常见的时区bug**：后端存储Date/Long类型的时间戳时没有问题，但当需要展示时，如果服务器时区和用户时区不一致，就会出现"早8晚8"的问题。正确的做法是：**后端存储UTC时间戳，前端根据用户时区转换展示**。
:::

### 2.2 旧API与新API的时区交互

```java
// JDBC中的日期类型
PreparedStatement ps = conn.prepareStatement("INSERT INTO orders(created_at) VALUES(?)");
ps.setObject(1, LocalDateTime.now()); // JDBC 4.2+，直接用LocalDateTime

// 反面教材：混用Date和LocalDateTime
Date date = new Date();
LocalDateTime ldt = date.toInstant()
    .atZone(ZoneId.systemDefault())
    .toLocalDateTime();
// 正确做法：统一使用Instant作为中间转换点
Instant instant = date.toInstant();
LocalDateTime ldt2 = LocalDateTime.ofInstant(instant, ZoneId.systemDefault());
```

### 2.3 生产避坑

**坑1：时区不一致导致数据错乱**

```java
// 反面教材：在不同时区服务器间直接传LocalDateTime
LocalDateTime orderTime = LocalDateTime.of(2024, 11, 11, 0, 0, 0); // 北京时间的零点
// 如果另一台服务器时区是UTC，存入数据库时会被解析为UTC的零点
// 实际存储的毫秒值少了8小时

// 正确做法：统一使用Instant或ZonedDateTime
Instant orderTime = Instant.parse("2024-11-11T00:00:00+08:00"); // 明确时区
ZonedDateTime zonedOrderTime = orderTime.atZone(ZoneId.of("Asia/Shanghai"));
```

**坑2：格式化时忘记设置时区**

```java
// 反面教材：格式化时丢失时区信息
LocalDateTime ldt = LocalDateTime.now();
DateTimeFormatter fmt = DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm:ss");
String str = ldt.format(fmt); // 只格式化日期时间，不包含时区

// 正确做法：需要时区时用ZonedDateTime
ZonedDateTime zdt = ZonedDateTime.now();
String str2 = zdt.format(DateTimeFormatter.ISO_ZONED_DATE_TIME);
// 2024-01-01T13:45:30+08:00[Asia/Shanghai]
```

**坑3：夏令时（DST）问题**

```java
// 美国夏令时切换日：2024-03-10 02:00 -> 03:00
// 这意味着 02:30 这个时间实际上不存在！
LocalDateTime dst = LocalDateTime.of(2024, 3, 10, 2, 30);
// DateTimeFormatter会抛出DateTimeException，因为2:30不存在
ZonedDateTime zdt = dst.atZone(ZoneId.of("America/New_York"));
// 自动调整到 03:30
System.out.println(zdt); // 2024-03-10T03:30-04:00[America/New_York]
```

## 三、实战场景选型 🟢

### 3.1 选型决策表

| 场景 | 推荐类型 | 原因 |
| --- | --- | --- |
| 存储数据库时间戳 | `Instant`或`Long` | 数据库通常存UTC毫秒值 |
| 用户出生日期 | `LocalDate` | 只有日期，无时区概念 |
| 用户下单时间（展示用） | `LocalDateTime` + 时区字段 | 展示本地时间 |
| 消息队列时间戳 | `Instant` | 全局唯一，无歧义 |
| 定时任务调度 | `ZonedDateTime` | 需要明确时区 |
| 业务周期计算 | `Period` | "加一个月"等操作 |
| 性能计时 | `Duration` | 毫秒/纳秒精度 |

### 3.2 日期计算——Period vs Duration

```java
LocalDate date1 = LocalDate.of(2024, 1, 1);
LocalDate date2 = LocalDate.of(2024, 3, 1);

// Period：按日历单位计算
Period period = Period.between(date1, date2); // P2M（2个月）
LocalDate date3 = date1.plus(period); // 2024-03-01

// Duration：按时间长度计算
LocalDateTime dt1 = LocalDateTime.of(2024, 1, 1, 0, 0);
LocalDateTime dt2 = LocalDateTime.of(2024, 1, 1, 12, 0);
Duration duration = Duration.between(dt1, dt2); // PT12H（12小时）

// 关键区别：加一个月（Period）vs 加30天（Duration）
LocalDate d = LocalDate.of(2024, 1, 31);
d.plusMonths(1); // 2024-02-29（自动调整，因为2月没有31号）
d.plusDays(30);   // 2024-03-01
```
