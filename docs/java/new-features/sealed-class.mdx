# Sealed Class 密封类

一个支付系统上线了。开发团队定义了一个 `Payment` 接口，三种支付方式：信用卡、支付宝、微信。业务逻辑根据支付类型做不同处理，一切看起来很美好。

然后，有人添加了第四种支付方式——数字人民币。没有更新处理逻辑，导致这种支付方式静默失败：用户付款成功，但系统状态不一致。排查了整整两天，才定位到这个"看不见"的分支。

这不只是代码问题，这是一个**类型系统问题**——Java 的继承机制允许"意外的扩展"，而 Java 编译器对此一无所知。

密封类（Sealed Classes），就是来解决这个问题的。

## 继承的失控：问题的根源

要理解密封类为什么重要，我们先要理解它试图解决的问题的本质。

Java 的继承体系，从设计之初就推崇"开放扩展"——任何 class 都可以被子类化，任何 interface 都可以被实现。这种开放性是面向对象设计的核心原则之一：**对扩展开放，对修改关闭**（开闭原则）。

但问题在于：**有些扩展是我们不希望看到的**。

考虑一个图形编辑器的场景：

```java
public interface Shape {
    double area();
}

public class Circle implements Shape { ... }
public class Rectangle implements Shape { ... }
```

这两类图形是我们已知的，代码里写了处理逻辑。突然有一天，某个第三方库引入了一个 `Triangle` 类，实现了 `Shape` 接口。你的代码会怎样？编译通过，运行时不报错，但面积计算逻辑对三角形完全失效——`area()` 方法返回了错误的值，或者更糟，返回了默认值。

这不是代码 bug，这是**架构漏洞**——Java 的类型系统无法表达"只有这些类实现了这个接口，其他类不行"这一约束。

## 密封类的引入：类型系统的契约

密封类通过 `sealed` 修饰符和 `permits` 子句，为继承关系引入了**受控的封闭性**：

```java
public sealed class Shape permits Circle, Rectangle {}

// 只有 Circle 和 Rectangle 可以继承 Shape
final class Circle extends Shape {}      // final：不能再被继承
sealed class Rectangle extends Shape permits ColoredRectangle {}
non-sealed class ColoredRectangle extends Rectangle {}
```

这个声明的含义是：**`Shape` 类只能被 `Circle` 和 `Rectangle` 继承，除此之外的任何类都不能继承它**。如果你试图写 `class Triangle extends Shape {}`，编译器会直接报错。

这不是简单的语法约束，这是**类型级别的契约**。它向编译器、工具、以及所有代码阅读者宣告：这个类的继承层次是封闭的，所有可能的子类型都已在这里列明。

### 核心设计：三种继承修饰符

密封类的被许可子类必须选择三种修饰符之一：

**`final`**：最终类，不可被继承。这是最严格的控制——一旦声明为 final，这个类型的继承链就彻底封死了。

```java
public sealed class Expr permits Number, Add, Multiply {}
final class Number extends Expr {}  // Number 的继承链到此为止
```

**`sealed`**：继续密封，允许进一步受限的继承。这允许你构建**层次化的密封结构**——父类封闭了直接子类，每个直接子类又可以定义自己的密封子层次。

```java
public sealed class Expr permits Number, BinaryExpr {}

sealed class BinaryExpr extends Expr permits Add, Multiply {}
final class Add extends BinaryExpr {}
final class Multiply extends BinaryExpr {}
```

这种层次化设计非常适合**递归数据结构**——表达式树就是典型例子。`Expr` 是密封的顶级接口，它的子类型分为两类：`Number`（基础情况）和 `BinaryExpr`（递归情况）。`BinaryExpr` 本身又是密封的，只能被 `Add` 和 `Multiply` 继承。通过这种层层约束，编译器可以精确追踪整个继承树的结构。

**`non-sealed`**：解除密封，恢复普通的可继承行为。这提供了一种**受控的出口**——你可以在密封层次中允许某些"开放"的分支。

```java
public sealed class Shape permits Circle, Polygon {}

non-sealed class Polygon extends Shape {}
// 现在任何类都可以继承 Polygon
class IrregularShape extends Polygon {}
```

`non-sealed` 解决了"大部分情况封闭，但某些情况需要扩展"的现实需求。比如 `Shape` 中大部分已知图形是固定的，但多边形这个类别本身是开放的——不规则多边形、三角形等都应该被允许。

## 密封接口：面向接口编程的密封支持

密封不仅适用于 class，也适用于 interface。密封接口的实现类同样必须被标记为 `final`、`sealed` 或 `non-sealed`：

```java
public sealed interface Operation permits Add, Subtract, Multiply {}

// Add 是 final，只能被最终类实现
final class Add implements Operation {
    public double apply(double a, double b) { return a + b; }
}

// Subtract 继续密封，可以有自己的密封实现
sealed class Subtract implements Operation permits IntSubtract {}
final class IntSubtract extends Subtract {}

// Multiply 解除密封
non-sealed class Multiply implements Operation {}
class CustomMultiply extends Multiply {}
```

接口的密封规则与类完全一致——任何实现或继承密封接口的类型，都必须显式声明其继承可扩展性。

## 穷尽性检查：编译器的守护

密封类最重要的实际意义，是让编译器能够进行**穷尽性检查**（Exhaustiveness Checking）。

想象一下，你用密封接口定义了一组有限的操作，然后写了一个处理这些操作的 switch 语句：

```java
sealed interface Expr permits Number, Add, Subtract, Multiply {}

int eval(Expr e) {
    return switch (e) {
        case Number n -> n.value();
        case Add a -> eval(a.left()) + eval(a.right());
        case Subtract s -> eval(s.left()) - eval(s.right());
        // 等等……Multiply 呢？
    };
}
```

如果编译器知道 `Expr` 只有 `Number`、`Add`、`Subtract`、`Multiply` 四种子类型，而你只处理了三种，编译器会报错：**Switch 不穷尽！** 这意味着你在某个分支没有处理的情况，运行时可能会遇到未预期的类型，导致错误或异常。

这种检查对于 sealed class + switch 表达式的组合尤其有价值——你可以在编译期就发现"我遗漏了对某种情况的处理"，而不是等到生产环境出现奇怪的 bug。

穷尽性检查的应用不仅限于 switch，还包括：

```java
// 密封类的 equals 实现——编译器确保穷尽
boolean equals(Object o) {
    return switch (this) {
        case Number n -> o instanceof Number m && n.value() == m.value();
        case Add a -> o instanceof Add m && a.left().equals(m.left())
                                      && a.right().equals(m.right());
        case Subtract s -> /* ... */;
        case Multiply m -> /* ... */;
    };
}
```

## 密封类的实际应用场景

### 领域模型中的状态机

许多业务场景可以用有限状态机来描述。密封类让状态机的所有可能状态在编译期就是已知的：

```java
public sealed interface OrderStatus
    permits OrderStatus.Pending,
           OrderStatus.Paid,
           OrderStatus.Shipped,
           OrderStatus.Delivered,
           OrderStatus.Cancelled {}

public final class Pending extends OrderStatus {}
public final class Paid extends OrderStatus {}
public final class Shipped extends OrderStatus {}
public final class Delivered extends OrderStatus {}
public non-sealed class Cancelled extends OrderStatus {}
```

所有订单状态一目了然。如果有人在代码中引入了一个未定义的 `Refunded` 状态并忘记更新相关处理逻辑，编译器会立即发现。

### 解析树和 AST

编译器前端和解析器是密封类最典型的应用领域。语法树的结构是固定的，每个节点类型是明确的：

```java
public sealed interface ASTNode permits Program, Declaration, Statement, Expression {}

public sealed interface Expression extends ASTNode permits Literal, Variable, BinaryOp, UnaryOp {}
public sealed interface Statement extends ASTNode permits IfStmt, WhileStmt, ReturnStmt, Block {}

public sealed interface BinaryOp extends Expression permits Add, Subtract, Multiply, Divide {}

// 每种运算符的实现是 final
public final class Add extends BinaryOp {
    public final Expression left, right;
    public Add(Expression left, Expression right) {
        this.left = left; this.right = right;
    }
}
```

编译器对 AST 的处理逻辑，可以通过穷尽的 switch 表达式覆盖所有情况，确保没有遗漏任何节点类型。

### API 的受控扩展点

在为库或框架设计 API 时，密封类提供了"受控扩展点"的优雅实现方式：

```java
public sealed class Validator<T>
    permits StringValidator, NumberValidator, CustomValidator {}

// 预定义的验证器是 final
public final class StringValidator extends Validator<String> {}
public final class NumberValidator extends Validator<Number> {}

// 用户自定义验证器通过 non-sealed 扩展
public non-sealed class CustomValidator<T> extends Validator<T> {}
```

框架可以安全地 switch 所有 Validator 类型——因为它知道所有可能的类型。而用户依然可以通过 CustomValidator 扩展功能，但这种扩展是显式的、受框架知晓的。

## 密封类的内部实现

理解密封类的内部实现，有助于深入理解其行为和安全保证。

每个密封类都有一个 `sealed` 和 `permits` 元数据，JVM 在字节码层面记录了这些信息：

```java
// 编译后的 Shape.class 会包含这样的属性：
// EnclosingMethod_attribute 和 InnerClasses_attribute
// 标记了 sealed 修饰和 permits 列表
```

这种元数据存储在 class 文件的 `SealedAttribute` 中，使得**即使源码不可用，反射和序列化框架也能识别密封约束**。

反射 API 也因此得到了增强。`Class.getPermittedSubclasses()` 方法可以查询一个密封类允许的所有子类：

```java
// 通过反射查询密封类的被许可子类
Class<?>[] permitted = Shape.class.getPermittedSubclasses();
for (Class<?> subclazz : permitted) {
    System.out.println("Shape 的合法子类: " + subclazz.getName());
}
```

这个 API 对于编写**通用的密封类处理器**特别有价值——框架代码可以自动发现并处理所有密封子类型，而无需硬编码类型列表。

## 与 Kotlin sealed class 的比较

作为对比，Kotlin 在 2015 年（Kotlin 1.0）就引入了 sealed class，比 Java 早了整整六年：

```kotlin
// Kotlin
sealed class Result
class Success(val data: String) : Result()
class Error(val message: String) : Result()
object Loading : Result()  // object 用于单例
```

Kotlin 的 sealed class 有几个值得注意的设计差异：

- Kotlin 的 sealed class 默认子类必须在同一个文件内（除非是嵌套类），Java 则允许在同一 module 内任何位置定义。
- Kotlin 不需要 `permits` 子句——只要在同一个文件中声明，就视为被允许。这是一种更简洁但更隐式的设计。
- Kotlin 的 sealed class 本质上是抽象的（implicitly abstract），不能直接实例化。Java 的 sealed class 默认不是抽象的，除非同时被 abstract 修饰。

Java 的 `permits` 声明方式虽然更显式，但也更冗长。两种设计各有取舍——Java 的方式更严格、更适合大型项目和跨文件继承，Kotlin 的方式更简洁、更适合小型项目和单一文件内的类型定义。

## 与枚举的比较

既然密封类和枚举都涉及"有限的可能性集合"，它们之间的选择就值得探讨。

**枚举**适合表示**固定的命名常量**——一个有限的、静态的集合，每个常量有唯一身份。枚举最适合表示状态或配置选项：

```java
enum Color { RED, GREEN, BLUE; }
enum Direction { NORTH, SOUTH, EAST, WEST; }
```

**密封类**适合表示**具有行为差异的类型层次**——每个子类型可能有不同的数据结构、不同的方法实现。密封类最适合表示语法树、解析结果或多元数据结构：

```java
sealed interface Shape permits Circle, Rectangle {}
sealed interface Payment permits CreditCard, Alipay, WechatPay {}
```

一个实用的判断原则：**如果子类型只是不同的常量值，用枚举；如果子类型有不同的数据结构和行为，用密封类**。

当然，两者也可以结合使用——枚举可以实现密封接口：

```java
public sealed interface ErrorCode permits HttpError, BizError {}
public enum HttpError implements ErrorCode {
    NOT_FOUND(404), SERVER_ERROR(500), UNAUTHORIZED(401)
}
public enum BizError implements ErrorCode {
    INVALID_INPUT, DUPLICATE_KEY, EXPIRED_TOKEN
}
```

## 密封类的局限性

密封类虽好，但也有其局限性。

**不能跨模块密封**。Java 的密封约束基于 module 或 package 的可见性。如果一个 sealed class 是 public 的，它的所有 permitted 子类也必须是同一个 module 内的类。这意味着第三方无法扩展你的 sealed class，除非你将它们放在同一个 module 中。

**密封层次不能跨越 JAR 边界**。如果一个 sealed class 位于 JAR A 中，其 permitted 子类在 JAR B 中，那么 JAR B 依赖 JAR A 时必须同时满足密封约束。这在实际项目中可能带来模块化设计的复杂性。

**sealed 和 record 的组合需要谨慎**。虽然 `sealed record` 看起来很优雅（代数数据类型），但如果 sealed record 需要被序列化，要注意 Java Record 没有无参构造函数，某些序列化框架可能需要额外配置。

---

密封类解决的，是一个看似简单却影响深远的问题：**如何让类型层次在编译期就成为封闭的集合**。

通过 `sealed` + `permits` + 三种继承修饰符的组合，Java 终于具备了表达"这个类型的扩展是受控的"的能力。这种能力不是语法糖——它是类型系统的底层支撑，让编译器能够自动检查穷尽性，让代码审查者一眼看出类型的约束边界，让未来的维护者不会在不知不觉中引入不一致的扩展。

从密封类、Record 到模式匹配，Java 正在一步步构建一套完整的代数数据类型体系。这套体系的最终愿景是：**让编译器能够理解你的代码结构，并帮助你检查那些以前只有运行时才能发现的错误。**

如果你对密封类在类型推导和模式匹配中的高级应用感兴趣，推荐进一步阅读 JEP 409（Sealed Classes 的正式 JEP）以及 Project Amber 的相关提案。
