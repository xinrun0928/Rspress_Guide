## 一、密封类解决什么问题？🔴

### 1.1 问题拆解

密封类是Java 17引入的重要特性，它和**模式匹配**配合才能发挥最大威力。这道题是**P7前瞻题**——考察候选人对Java类型系统演进的理解。

**第一层：设计意图**
- 面试官问："密封类是解决什么问题的？"
- 考察点：穷举检查、类型安全

**第二层：permits语法**
- 面试官追问："`sealed permits A, B`是什么意思？哪些类可以继承密封类？"
- 考察点：sealed/non-sealed/final修饰符

**第三层：密封接口**
- 面试官继续问："密封接口和密封类有什么区别？"
- 考察点：接口天然支持多实现

**第四层：与模式匹配的配合**
- 面试官问："密封类在模式匹配里怎么用？"
- 考察点：穷举性检查、exhaustiveness

### 1.2 ❌ 错误示范

**错误回答1："密封类就是限制别人继承，跟final一样"**

密封类比final更灵活：final禁止继承，密封类**有条件地允许继承**。密封类允许你声明"只有这几种类型可以继承/实现"。

**错误回答2："密封类在JDK源码里见过，主要是`java.lang.Enum`"**

`Enum`是`final class`，不是密封类。但`java.lang.Number`、`java.lang.Thread`等是密封类的应用例子。

**错误回答3："密封类就是给设计模式用的"**

密封类确实适合策略模式、工厂模式，但核心价值是**类型穷举检查**——让编译器帮你验证switch/instanceof的穷举性。

【面试官心理】
密封类是Java 17的重量级特性，和模式匹配一起构成了更强大的类型系统。问这道题的面试官通常是在试探候选人是否跟进Java新版本。能说出穷举检查价值的占30%，能说出sealed permits和non-sealed/final修饰符区别的占15%，能说出和模式匹配配合的占5%。

### 1.3 标准回答

#### 第一步：穷举检查——密封类的核心价值

```java
// 没有密封类时
sealed interface Shape permits Circle, Rectangle, Triangle {}

abstract sealed class Shape permits Circle, Rectangle, Triangle {}

final class Circle extends Shape {}
sealed class Rectangle extends Shape {}
non-sealed class Triangle extends Shape {}
```

当一个类被声明为`sealed`时，只有在`permits`列表中显式声明的类可以继承它。这带来一个关键优势：**编译器可以穷举检查**。

```java
// 密封类配合switch（Java 21+）
sealed interface Expr permits Constant, Variable, Add, Multiply {}

record Constant(int value) implements Expr {}
record Variable(String name) implements Expr {}
record Add(Expr left, Expr right) implements Expr {}
record Multiply(Expr left, Expr right) implements Expr {}

int eval(Expr expr) {
    return switch (expr) {
        case Constant(int v) -> v;
        case Variable(String name) -> 42; // 假设所有变量=42
        case Add(Expr left, Expr right) -> eval(left) + eval(right);
        case Multiply(Expr left, Expr right) -> eval(left) * eval(right);
        // 如果漏掉任何一个case，编译器直接报错！
        // 即使是Triangle，也逃不过编译器的穷举检查
    };
}
```

:::tip 💡
**穷举检查的价值**：当你新增一个`Expr`实现（比如`Subtract`），所有使用sealed接口的switch都会**在编译期**报错，提醒你补全处理逻辑。这是运行时`ClassCastException`的克星。
:::

#### 第二步：sealed/non-sealed/final三元组

密封类的子类必须使用以下三个修饰符之一：

```java
public sealed class Shape permits Circle, Rectangle, Triangle {}

// Circle：final，禁止进一步继承
public final class Circle extends Shape {}

// Rectangle：sealed，可以继续继承，但仍然受限
public sealed class Rectangle extends Shape permits Square {}

// Triangle：non-sealed，取消继承限制
public non-sealed class Triangle extends Shape {}

// Square：normal类，可以继承Rectangle
public class Square extends Rectangle {}
```

```
Shape (sealed)
├── Circle (final)     → 完全密封，不能再继承
├── Rectangle (sealed) → 可以继续继承
│   └── Square         → 恢复普通类
└── Triangle (non-sealed) → 解除密封，可以再继承
```

**关键规则**：
- permits列表中的子类必须在同一个**模块**或**包**中
- 如果在同一个源文件定义，可以省略permits（JDK 17改进）

```java
// 同一文件中定义，可以省略permits
public sealed class Shape {
    public final class Circle extends Shape { }
    public sealed class Rectangle extends Shape { }
    public non-sealed class Triangle extends Shape { }
}
```

## 二、密封接口 🟡

### 2.1 密封接口的独特之处

```java
// 密封接口：implements替代extends
public sealed interface Operate permits Add, Subtract, Multiply, Divide {}

// 类实现接口
public final class Add implements Operate { }
public sealed class Subtract implements Operate permits SubOfSub { }
public non-sealed class Multiply implements Operate { }

// 接口继承接口
public sealed interface BinaryOperate extends Operate permits Mod { }
public final class Mod implements BinaryOperate { }
```

接口的sealed特性更强大，因为：
- 一个类可以实现**多个密封接口**
- 接口天然支持多继承，配合sealed可以构建复杂的类型层级

### 2.2 JDK中的密封类应用

```java
// java.lang.Number是sealed class（Java 17+）
public abstract sealed class Number permits Byte, Double, Float,
                                        Integer, Long, Short, BigDecimal, BigInteger {}

// java.lang.Thread.State是enum（enum天然是sealed的）
// 所有枚举都是隐式final和sealed的
```

## 三、密封类与模式匹配的配合 🟡

### 3.1 穷举性检查（Exhaustiveness）

```java
// 普通接口：switch可以不穷举，不报错
interface Color {}
record Red() implements Color {}
record Blue() implements Color {}

// 漏掉Blue不会报错
String describe(Color c) {
    return switch (c) {
        case Red r -> "红色";
        // 忘了处理Blue，运行时抛MatchException
    };
}
```

```java
// 密封接口：switch必须穷举
sealed interface Color permits Red, Blue {}
record Red() implements Color {}
record Blue() implements Color {}

// 编译器强制穷举
String describe(Color c) {
    return switch (c) {
        case Red r -> "红色";
        case Blue b -> "蓝色";
        // 漏掉任何case，编译直接报错
    };
}
```

### 3.2 Record在密封类中的黄金组合

```java
// 最佳实践：sealed + record + pattern matching
sealed interface Tree permits Leaf, Branch {}

record Leaf(int value) implements Tree {}
record Branch(Tree left, Tree right) implements Tree {}

// 编译器保证：所有Tree的实现都被处理
int sum(Tree t) {
    return switch (t) {
        case Leaf(int v) -> v;
        case Branch(Tree l, Tree r) -> sum(l) + sum(r);
    };
}
```

:::warning ⚠️
**向后兼容性注意**：密封类需要所有permits的类在同一个模块或包内。如果你的开源库使用了密封类，调用方必须将实现类放在正确的位置。在模块化项目中，这可能带来迁移成本。
:::

## 四、密封类与设计模式 🟢

### 4.1 策略模式的类型安全版本

```java
// 普通策略模式：无法穷举
interface DiscountStrategy {
    BigDecimal apply(BigDecimal price);
}

// 密封类策略模式：穷举性保证
sealed interface DiscountStrategy
    permits FixedDiscount, PercentageDiscount, NoDiscount {}

public final class FixedDiscount implements DiscountStrategy {
    private final BigDecimal amount;
    public FixedDiscount(BigDecimal amount) { this.amount = amount; }
    @Override
    public BigDecimal apply(BigDecimal price) {
        return price.subtract(amount);
    }
}

public final class PercentageDiscount implements DiscountStrategy {
    private final double rate;
    // ...
}

// 新增折扣类型时，编译器强制检查所有switch
BigDecimal calcPrice(BigDecimal price, DiscountStrategy strategy) {
    return switch (strategy) {
        case FixedDiscount f -> price.subtract(f.amount());
        case PercentageDiscount p -> price.multiply(BigDecimal.valueOf(1 - p.rate()));
        case NoDiscount n -> price;
    };
}
```

### 4.2 工厂模式的改进

```java
// 传统工厂：返回类型需要转型
Shape createShape(String type) {
    return switch (type) {
        case "circle" -> new Circle();
        case "rect" -> new Rectangle();
        default -> throw new IllegalArgumentException();
    };
}

// sealed + record：类型信息保留
sealed interface Shape permits Circle, Rectangle, Triangle {}
record Circle(double radius) implements Shape {}
record Rectangle(double width, double height) implements Shape {}
record Triangle(double base, double height) implements Shape {}
```
