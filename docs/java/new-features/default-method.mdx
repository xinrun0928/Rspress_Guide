## 一、为什么要给接口加默认方法？🔴

### 1.1 问题拆解

这道题考察候选人对Java 8设计决策的理解，是**P6进阶题**。

**第一层：历史背景**
- 面试官问："Java 8之前接口能不能有方法实现？为什么不行？"
- 考察点：抽象契约、类单继承

**第二层：引入动机**
- 面试官追问："Java 8为什么要打破这个规则？"
- 考察点：Collection.forEach的演进、向后兼容

**第三层：多继承冲突**
- 面试官继续追问："如果两个接口有同名的默认方法，实现类怎么办？"
- 考察点：菱形继承问题、方法解析规则

**第四层：细节规则**
- 面试官问："接口静态方法和类静态方法有什么区别？默认方法能不能访问private？"
- 考察点：Java 9的private接口方法、设计演进

### 1.2 ❌ 错误示范

**错误回答1："接口默认方法是为了减少实现类的工作量"**

这只是表面原因。更深层的原因是**向后兼容性**——Java 8引入Lambda时，需要给Collection接口增加`forEach`方法。如果不允许接口有默认实现，所有已有的Collection实现类都会编译失败，Java生态会崩溃。

**错误回答2："多继承冲突时，编译器会自动选一个"**

编译器不会自动选，而是**强制要求程序员显式解决**。如果两个接口都有同名默认方法，代码根本**无法编译**，必须由程序员在实现类中显式覆盖。

**错误回答3："接口静态方法和类静态方法用法完全一样"**

有区别：接口静态方法**不能被继承**（子类不能通过`SubInterface.staticMethod()`调用），只能通过`InterfaceName.staticMethod()`调用。而类静态方法可以被继承。

【面试官心理】
我问他默认方法，其实是在试探他有没有理解Java 8的设计动机。Lambda是Java 8最重要的特性，但为了引入Lambda，Java团队必须改造Collection API——这需要给接口添加默认方法。知道这个背景故事的占20%，能说清菱形继承规则的占15%，能说清private接口方法演进的占5%。

### 1.3 标准回答

#### 第一步：历史动机——向后兼容的无奈之举

Java 8引入Lambda表达式，但Lambda的基础是**函数式接口**。为了支持Lambda，Java团队想让Collection API支持`forEach`方法：

```java
// Java 8想给Collection加forEach
public interface Collection<E> {
    default void forEach(Consumer<? super E> action) {
        for (E e : this) {
            action.accept(e);
        }
    }
}
```

如果接口不允许有默认实现，那么所有已有的`ArrayList`、`HashSet`、`LinkedList`等实现类都会编译失败——JDK团队不可能修改所有开源库。这就是**向后兼容**的压力。

#### 第二步：默认方法规则

**基本语法**：

```java
public interface Readable {
    // 抽象方法：实现类必须提供
    void read(ByteBuffer dst);

    // 默认方法：实现类可以继承，也可以覆盖
    default int read(ByteBuffer dst, byte[] bytes) {
        // 默认实现
        return 0;
    }
}
```

**默认方法继承规则**：

```java
public interface A {
    default void hello() {
        System.out.println("A.hello");
    }
}

public interface B {
    default void hello() {
        System.out.println("B.hello");
    }
}

// 如果C只实现A和B，不覆盖hello()——编译错误！
public class C implements A, B {
    // 编译器报错：class C inherits unrelated defaults for hello()
}

// 必须显式覆盖
public class C implements A, B {
    @Override
    public void hello() {
        // 选择A的版本
        A.super.hello();
        // 或者B的版本
        // B.super.hello();
        // 或者自定义
        System.out.println("C.hello");
    }
}
```

:::tip 💡
**菱形继承问题的解决**：Java使用"C的选择"规则——如果一个类实现的多个接口有同名的默认方法，类必须显式覆盖该方法，并在覆盖方法中通过`InterfaceName.super.method()`选择使用哪个父接口的默认实现。
:::

#### 第三步：默认方法解析规则（优先级）

当类继承了一个类并实现了一个或多个接口时，优先级规则如下：

```java
public class Parent {
    default void hello() { System.out.println("Parent.hello"); }
}

public interface MyInterface {
    default void hello() { System.out.println("MyInterface.hello"); }
}

public class Child extends Parent implements MyInterface {
    // 不用覆盖！父类方法优先于接口默认方法
    // 输出：Parent.hello
}
```

**优先级顺序**：类方法 > 接口默认方法 > 更具体的接口默认方法

如果一个接口继承另一个接口，两个接口都有同名默认方法，子接口的默认方法**覆盖**父接口的。

```java
public interface GrandParent {
    default void hello() { System.out.println("GrandParent"); }
}

public interface Parent2 extends GrandParent {
    @Override
    default void hello() { System.out.println("Parent2"); }
}

public interface Child2 extends GrandParent {
    @Override
    default void hello() { System.out.println("Child2"); }
}

public interface GrandChild extends Parent2, Child2 {
    // GrandChild现在有两个hello的候选：Parent2和Child2
    // 但都不冲突，因为它们都override了GrandParent
    // 输出：Parent2.hello（优先规则：子接口覆盖父接口）
}
```

## 二、接口静态方法与私有方法 🟡

### 2.1 接口静态方法

```java
public interface StringUtils {
    // 接口静态方法：通过接口名直接调用
    static boolean isBlank(String str) {
        return str == null || str.trim().isEmpty();
    }

    // 不能被子接口继承
    // 不能被子类继承
    // 属于接口自身，不属于实现类
}

// 调用方式
StringUtils.isBlank("hello"); // ✅ 正确
// MyClass.isBlank("hello"); // ❌ 编译错误，除非MyClass显式实现并重写了
```

**设计动机**：Java 8之前，工具方法通常放在**工具类**中（如`Collections`、`Arrays`）。引入接口静态方法后，可以让**相关功能内聚**在接口中，减少工具类的数量。

### 2.2 Java 9的私有接口方法

Java 9进一步允许接口有**私有方法**：

```java
public interface MyInterface {
    default void method1() {
        // 可以调用私有方法
        helper();
    }

    default void method2() {
        // 多个默认方法共享代码
        helper();
    }

    // Java 9+：私有方法，default方法可以调用，但不能被实现类访问
    private void helper() {
        // 共享逻辑
    }
}
```

:::warning ⚠️
**为什么需要私有方法**？当多个默认方法有共同的辅助逻辑时，如果没有私有方法，这些辅助逻辑必须写成`static`方法才能共享。但`static`方法可以被外部调用，不够封装。私有方法解决了这个问题——只供接口内部使用。
:::

## 三、生产避坑与工程选型 🟡

### 3.1 默认方法的最大坑——打破开闭原则

```java
// 场景：JDK引入了新的默认方法，导致你的实现类行为改变
public class MyList<E> implements List<E> {
    // Java 8的List接口增加了sort()默认方法
    // 现在你的MyList实例可以直接调用.sort()了
    // 但你原本没考虑过这个场景！
}
```

这是一个经典的**隐性行为变更**：接口添加默认方法后，所有实现类都会继承这个方法，即使实现类从未声明过。这可能打破原有的设计假设。

### 3.2 默认方法与Lambda的关系

```java
public interface Iterable<T> {
    default void forEach(Consumer<? super T> action) {
        for (T t : this) {
            action.accept(t);
        }
    }
}

// 所以任何实现Iterable的类都可以直接用forEach
List<String> list = Arrays.asList("a", "b", "c");
list.forEach(System.out::println); // 直接用，不需要手动遍历
```

### 3.3 最佳实践

| 场景 | 建议 |
| --- | --- |
| 需要向后兼容的API扩展 | ✅ 用默认方法 |
| 需要工具方法 | ✅ 接口静态方法（Java 8+） |
| 多默认方法共享逻辑 | ✅ 私有方法（Java 9+） |
| 核心契约 | ❌ 避免用默认方法，优先抽象方法 |
| 需要状态 | ❌ 接口不能有实例字段 |

**接口 vs 抽象类**：

| 特性 | 接口 | 抽象类 |
| --- | --- | --- |
| 多继承 | ✅ 支持 | ❌ 不支持 |
| 字段 | ❌ 只能是`static final` | ✅ 任意类型 |
| 方法实现 | ✅ 默认方法（Java 8+） | ✅ 任意方法 |
| 构造器 | ❌ 不能有实例构造器 | ✅ 可以有 |
| 状态 | ❌ 无实例状态 | ✅ 可以有实例字段 |
