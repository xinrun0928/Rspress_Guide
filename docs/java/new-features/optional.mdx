## 一、Optional 到底解决什么问题？🔴

### 1.1 问题拆解

这道题是**P6区分题**，85%的候选人会死在三个问题上：设计意图、正确用法、反模式。

**第一层：基本使用**
- 面试官问："Optional怎么用？"
- 考察点：of/empty/ofNullable、map/flatMap/filter

**第二层：设计意图**
- 面试官追问："Optional是为了解决NPE吗？那它和直接判空有什么区别？"
- 考察点：类型系统、显式建模、可组合性

**第三层：API误用**
- 面试官问："`Optional.get()`和直接判空相比，有什么区别？"
- 考察点：get()本身也会NPE、正确的终端操作

**第四层：性能与选型**
- 面试官问："Optional有什么性能问题？什么场景下不适合用？"
- 考察点：内存开销、装箱开销、不适合序列化

### 1.2 ❌ 错误示范

**错误回答1："Optional解决了NPE问题"**

Optional**不能**解决NPE，它只是把NPE的时机从运行时推迟到了使用`get()`时。如果不配合正确的API，Optional.get()同样会抛出`NoSuchElementException`。

**错误回答2："用Optional替代所有null判断"**

这是过度使用的反模式。Optional是**值包装器**，会增加内存开销（每个Optional至少多16字节），在高性能场景中要慎用。

**错误回答3："Optional可以通过`isPresent()` + `get()`组合判断"**

这是最经典的反模式——`isPresent() + get()`等于什么都没做，和`if (obj != null)`完全等价，而且可读性更差。

【面试官心理】
我问他Optional，其实是在试探他有没有**函数式编程的思维**。知道Optional API的占80%，能说出设计意图的占40%，能正确使用map/flatMap组合的占20%，能指出反模式的占10%。这道题能答到最后的，基本都有函数式编程的实战经验。

### 1.3 标准回答

#### 第一步：Optional的设计意图——类型系统层面的null建模

Optional的**核心设计意图**不是"解决NPE"，而是**在类型层面显式建模"可能有值"这个概念**。

```java
// 传统写法：null的语义隐藏在类型中
String findName(Long id); // 返回null是什么意思？查不到？异常？

// Optional写法：null的语义在返回类型中显式表达
Optional<String> findName(Long id); // 明确表示：可能有值，可能没有
```

这个区别的意义在于：**编译器能帮你在调用处强制处理空值**。当你看到返回类型是`Optional<String>`，你就被迫要处理"可能为空"的情况。

#### 第二步：正确使用——链式API组合

```java
// 错误：isPresent + get = 和null判断等价
Optional<User> user = findUser(id);
if (user.isPresent()) {
    return user.get().getName();
}
return "Anonymous";

// 正确：map链式调用
return findUser(id)
    .map(User::getName)
    .orElse("Anonymous");

// 更复杂的例子
return userRepository.findById(id)
    .flatMap(User::getDepartment)
    .flatMap(Department::getManager)
    .map(Manager::getName)
    .orElseThrow(() -> new BusinessException("未找到负责人"));
```

`map()`用于**转换值**，`flatMap()`用于**变换Optional本身**：

```java
// map：用函数T -> U转换值
Optional<User> user = findUser(id);
Optional<String> name = user.map(User::getName); // Optional<String>

// flatMap：用函数T -> Optional<U>变换Optional
Optional<Department> dept = user.flatMap(User::getDepartment);
// 如果user为空，dept也是empty，不会执行getDepartment

// 关键区别：如果转换函数返回的是Optional，用flatMap
//            如果转换函数返回的是普通值，用map
```

#### 第三步：三大终端操作——orElse vs orElseGet vs orElseThrow

```java
Optional<String> opt = findName(id);

// orElse：始终计算默认值（即使Optional有值）
String result = opt.orElse(computeDefault()); // 无论opt是否为空，computeDefault()都会被调用

// orElseGet：延迟计算默认值（只在Optional为空时才调用）
String result = opt.orElseGet(() -> computeDefault()); // 推荐：避免不必要的计算

// orElseThrow：为空时抛异常
String result = opt.orElseThrow(() -> new UserNotFoundException(id));
```

:::tip 💡
**orElse vs orElseGet的性能差异**：如果默认值是一个昂贵的计算或数据库查询，`orElse`会无条件执行，而`orElseGet`只在Optional为空时才执行。生产中推荐优先使用`orElseGet`。
:::

#### 第四步：Optional的filter组合

```java
// 查找年龄大于18岁的用户
return userRepository.findById(id)
    .filter(user -> user.getAge() >= 18)
    .map(User::getName)
    .orElse("未成年用户");
```

## 二、Optional 反模式与避坑 🟡

### 2.1 七大反模式

**反模式1：作为字段类型**

```java
// ❌ 反模式：Optional作为POJO字段
public class User {
    private Optional<String> nickname; // 序列化问题！ORM框架不友好
}

// ✅ 正确做法：字段本身可以是null，getter返回Optional
public class User {
    private String nickname; // 普通字段

    public Optional<String> getNickname() {
        return Optional.ofNullable(nickname);
    }
}
```

Optional**不支持序列化**（`Serializable`接口是空的），很多ORM框架（如MyBatis、Hibernate）对Optional字段的处理也不一致。在POJO中使用Optional是严重反模式。

**反模式2：作为方法参数**

```java
// ❌ 反模式：Optional作为参数
public void process(Optional<User> user) {
    if (user.isPresent()) {
        // ...
    }
}

// ✅ 正确做法：重载或使用多个参数
public void process(User user) {
    // 直接处理null情况
}
public void process() {
    // 无参数版本
}
```

Optional作为参数会**强制调用者包装参数**，增加了调用方的负担。正确的设计是让调用方决定是否传值。

**反模式3：Boolean判空**

```java
// ❌ 反模式：Optional布尔化
if (userRepository.findById(id).isPresent()) {
    // do something
}

// ✅ 正确：直接使用
userRepository.findById(id).ifPresent(u -> {
    // do something
});
```

**反模式4：链式判空不够用时的粗暴处理**

```java
// ❌ 反模式：多层ifPresent嵌套
userOpt.ifPresent(user -> {
    user.getDepartment().ifPresent(dept -> {
        dept.getManager().ifPresent(mgr -> {
            // 地狱回调
        });
    });
});

// ✅ 正确：flatMap扁平化
userOpt
    .flatMap(User::getDepartment)
    .flatMap(Department::getManager)
    .ifPresent(Manager::sendEmail);
```

**反模式5：在Stream中使用Optional判断后再get**

```java
// ❌ 反模式：先Optional再get
return list.stream()
    .filter(x -> x > 0)
    .findFirst()
    .get(); // 可能抛NoSuchElementException

// ✅ 正确：orElse/ifPresent/orElseThrow
return list.stream()
    .filter(x -> x > 0)
    .findFirst()
    .orElse(0);
```

**反模式6：忽略Optional的副作用**

```java
// ❌ 反模式：在map中做副作用
userOpt.map(user -> {
    log.info("Found user: {}", user); // map的lambda不应该有副作用
    return user.getName();
});

// ✅ 正确：使用ifPresent处理副作用
userOpt.ifPresent(user -> log.info("Found user: {}", user));
```

**反模式7：滥用OptionalStream互转**

```java
// ❌ 反模式：频繁Stream转Optional
List<User> list = getUsers();
list.stream()
    .filter(u -> u.getAge() > 18)
    .findFirst()
    .stream() // 又转回Stream，完全没必要
    .map(User::getName)
    .collect(toList());

// ✅ 正确：直接用Optional
Optional<User> opt = list.stream()
    .filter(u -> u.getAge() > 18)
    .findFirst();
```

### 2.2 Optional与Stream的无缝衔接

Optional在Java 9引入了`stream()`方法，与Stream的无缝衔接是其设计最精妙的地方：

```java
// Java 9+：Optional转Stream
Optional<User> userOpt = userRepository.findById(id);
Stream<User> stream = userOpt.stream(); // 空Optional返回空Stream

// 实用场景：处理可能为空的列表
List<User> users = optionalUserList
    .stream()                        // Optional<List<User>> -> Stream<List<User>>
    .flatMap(List::stream)          // flatMap打平：Stream<User>
    .filter(u -> u.getAge() > 18)
    .collect(toList());
```

```java
// 更复杂的场景：多个Optional合并
Optional<String> result = Stream.of(
        getName1(),
        getName2(),
        getName3()
    )
    .flatMap(Optional::stream)       // 转成Stream<String>
    .filter(s -> !s.isEmpty())
    .findFirst();
```

## 三、Optional 性能与工程选型 🟢

### 3.1 性能真相

Optional在HotSpot中是一个简单的包装类：

```java
// Optional的内存布局（简化）
public final class Optional<T> {
    private final T value; // 非空时存储值，空时为null
    // 额外开销：对象头 + 引用 = 约16字节
}
```

在**高性能场景**下，Optional的额外对象分配是个问题：

| 场景 | 推荐方案 |
| --- | --- |
| DTO/Entity字段 | ❌ 不用Optional |
| 方法返回值 | ✅ 推荐：显式建模空值语义 |
| 集合元素 | ❌ 不需要：Collections.emptyList()足够 |
| 业务逻辑中间变量 | ✅ 推荐：链式API |
| 热路径循环（百万次+） | ⚠️ 慎用：评估GC压力 |

:::warning ⚠️
**Optional的GC压力**：在高频交易系统中，每次API调用产生的Optional对象可能达到数十个。如果QPS是10万，这些临时对象的GC压力不可忽视。某量化交易平台曾因过度使用Optional导致Young GC频率翻倍，最终通过JMH定位后改为直接判空。
:::

### 3.2 选型决策树

```
返回值可能为空？
├── 是 → 用Optional<T>
│   ├── 需要链式转换 → map/flatMap
│   ├── 需要默认值 → orElse/orElseGet
│   └── 需要抛异常 → orElseThrow
└── 否 → 直接返回T，不加Optional包装
```

**能用null表示的，就不要用Optional**。Optional适合在**API边界**上强制处理空值，不适合在内部逻辑中传递。
