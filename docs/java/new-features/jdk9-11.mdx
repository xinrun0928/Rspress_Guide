# JDK 9~11 新特性

凌晨三点，你刚准备睡觉，手机突然震动。监控系统报警：生产环境的微服务启动失败了。不是代码问题，是依赖冲突——两个第三方库分别依赖不同版本的 Spring Framework，版本拉扯之间，整个依赖树崩溃了。

这不是故事，这是 Java 开发者过去二三十年间反复经历的噩梦。而这个噩梦，最终催生了 Java 历史上最具争议的变革之一：**模块化系统**。

## 模块化系统：Java 的"楼房设计图"

Java 诞生之初，靠的是"类路径"（Classpath）来组织代码。所有 JAR 包一股脑扔进 Classpath，运行时由类加载器负责查找。这种方式简单粗暴，但随着项目规模增长，问题接踵而至：依赖不透明（不知道一个 JAR 依赖了哪些其他 JAR）、版本冲突难以解决（多个版本同时存在时行为不可预测）、运行时才知道的类缺失错误（编译时完全不检查）。

Java 9 引入的模块化系统（Project Jigsaw），从根本上改变了这种局面。每个模块（Module）现在有了一个明确的"设计图"——`module-info.java`，在其中声明：这个模块叫什么名字、依赖哪些其他模块、对外暴露哪些包。

```java
module com.myapp {
    requires com.example.utils;       // 声明依赖
    exports com.myapp.api;            // 暴露对外接口
}
```

模块化的好处是全方位的。编译时，编译器会检查依赖完整性，再也不会等到运行时才发现少了一个类。运行时，强封装（Strong Encapsulation）确保模块只能访问它明确依赖的其他模块的公开 API，内部实现被彻底隐藏。这意味着 JDK 自身也被模块化了——你不再需要加载整个 JRE 运行环境，只需要加载应用真正需要的模块。一个精简的 Java 运行时镜像可以只有几十 MB，而不是传统 JRE 的两百多 MB。

但模块化也带来了"迁移的痛苦"。大量历史项目在升级到 JDK 9+ 后遭遇了类访问被拒绝的错误，因为它们以前可以"偷偷"访问 JDK 内部的 `sun.*` 包，而这些包现在被封装起来了。反射、字节码操作等黑魔法也受到了更多限制。社区对此褒贬不一：有人认为这是 Java 走向成熟的必经之路，有人则认为这种breaking change伤害了生态。

## 局部变量类型推断：var 来了

Java 10 引入的 `var`，大概是这个版本最被热议的特性了。它让局部变量声明时可以省略类型，由编译器自动推断。

```java
// 传统写法
Map<String, List<Transaction>> cache = new HashMap<>();

// var 写法
var cache = new HashMap<String, List<Transaction>>();
```

初看起来，这不过是省了几个字。但它的意义远不止语法糖。`var` 实际上改变了我们编写代码的方式——当你不再需要预先声明变量类型时，**代码的流畅度和可读性大幅提升**。特别是在处理泛型嵌套、链式调用、方法引用等复杂类型时，`var` 能让代码更聚焦于"做什么"而不是"是什么类型"。

```java
var result = users.stream()
    .filter(u -> u.getAge() > 18)
    .map(User::getName)
    .collect(Collectors.toList());
// 类型一目了然：List<String>
```

但 `var` 也带来了争议。过度使用会让代码难以理解——想象一下满屏的 `var`，阅读者需要反复推敲每个变量的实际类型。最佳实践是：`var` 适合类型名称过长或者类型从右侧明显可见的场景，不适合类型本身有重要语义的场景（比如 boolean 标志位用 `var flag = checkSomething()` 就很隐晦）。

## HTTP Client API：从 HttpURLConnection 解放出来

如果让你列举 Java 最"历史悠久"的 API，`HttpURLConnection` 绝对榜上有名。这个从 JDK 1.1 就存在的类，二十多年来几乎没有变化。同步阻塞的 API 设计、无法处理响应体流、缺乏对现代 HTTP 特性的支持（HTTP/2、WebSocket、异步请求），让它在实际项目中几乎无人问津。开发者纷纷投向 Apache HttpClient、OkHttp 等第三方库。

JDK 11 终于带来了官方的现代 HTTP Client。从第一天起，它就是为现代互联网设计的：

```java
// 同步调用
HttpClient client = HttpClient.newHttpClient();
HttpRequest request = HttpRequest.newBuilder()
    .uri(URI.create("https://api.example.com/data"))
    .header("Accept", "application/json")
    .build();
HttpResponse<String> response = client.send(request,
    HttpResponse.BodyHandlers.ofString());
System.out.println(response.body());
```

```java
// 异步调用——完全非阻塞
client.sendAsync(request, HttpResponse.BodyHandlers.ofString())
    .thenApply(HttpResponse::body)
    .thenAccept(System.out::println);
```

支持 HTTP/2（自动协商，兼容 HTTP/1.1）、内置 WebSocket 客户端、支持同步和异步两种模式。底层的 `CompletableFuture` 驱动让异步编程变得自然流畅。结合 Java 15+ 的密封类和 Java 21 的虚拟线程，HTTP Client 在高并发场景下的表现将更加出色。

## JDK 9~11 的其他亮点

JDK 9 还带来了**接口私有方法**——接口现在可以有私有方法了，用于在默认方法之间共享代码，告别了辅助类的需求。**进程 API** 也大幅增强，可以直接获取进程信息、CPU 使用率等，以前这些信息需要调用原生代码或依赖第三方库。

JDK 10 的**集合增强**让 `List`、`Set`、`Map` 拥有了 `copyOf()` 方法，可以轻松创建不可变集合的副本。`Stream` 也新增了 `orElseThrow()` 方法，为空时直接抛异常，省去了 `orElseThrow(() -> new RuntimeException("..."))` 的模板代码。

JDK 11 的**单文件源码程序**运行是一个容易被忽视的实用特性：直接 `java HelloWorld.java` 就能运行单文件程序，无需先编译再运行，极大提升了脚本化和快速实验的体验。**字符串增强**则带来了 `isBlank()`、`lines()`、`strip()`、`repeat()` 等方法，告别了用第三方库处理字符串的日常。

---

从 JDK 9 到 JDK 11，Java 在保持向后兼容的同时，迈出了模块化的第一步。这段时期的变革更多是"夯实基础"，为后续更激进的语法增强铺平道路。下一节，我们将看到 Java 如何在 JDK 12~16 期间，通过一系列预览特性，大步走向现代化。
