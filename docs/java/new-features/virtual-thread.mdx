# Virtual Thread（虚拟线程）原理与适用场景

想象一个餐厅场景。

传统模式下，餐厅只有 10 个服务员。每个服务员一次只能为一桌客人服务——点菜时必须等待厨房出菜，等待期间这个服务员什么都不能干，只能干等着。直到菜做好了，服务员才能继续上菜、结账。10 个服务员，最多同时服务 10 桌。如果来了 100 桌客人，就得排队等。

这就是 Java 传统线程模型面临的问题：**线程是昂贵的资源**。每个线程拥有独立的调用栈（通常 1MB），创建、销毁、上下文切换都有显著开销。在传统 I/O 密集型应用中，大量的时间都浪费在"等待"上——等数据库返回、等 HTTP 响应、等文件读取——而线程却在空转，白白消耗着内存和 CPU。

虚拟线程的出现，彻底改变了这个局面。

## 从"服务员"到"座位"：一个新的心智模型

还是那个餐厅。但这次，餐厅改成了开放式座位。客人自己找座位坐下来，扫码点餐。服务员不需要"专属服务"——他们只需要在厨房和餐桌之间穿梭送菜。50 个座位，5 个服务员，足够了。客人在等菜的时候依然坐在座位上，不会占用任何服务员的精力。

在虚拟线程的世界里，**虚拟线程就是客人，平台线程（Carrier Thread）就是服务员**。

- 1000 个虚拟线程并发执行，但只需要十几个平台线程来承载。
- 虚拟线程在等待 I/O 时会"暂停"（`WAITING` / `BLOCKING`），底层载体线程被释放出来，去执行其他虚拟线程。
- 上下文切换发生在虚拟线程层面，开销远小于平台线程的上下文切换。

这是怎么做到的？答案是：**协作式调度**。

## 虚拟线程的实现原理

### 延续（Continuation）：暂停的艺术

传统线程的调度是**抢占式**的——操作系统随时可以打断正在运行的线程，切换到另一个线程。这种方式保证了公平性，但也带来了频繁的上下文切换开销（保存/恢复寄存器、切换页表等）。

虚拟线程采用**协作式调度**。当一个虚拟线程需要等待时，它会主动"让出"载体线程，而不是被动被操作系统抢占。这个主动让出的机制，就是通过**延续（Continuation）**实现的。

延续是一个可以被暂停和恢复的计算单元。听起来很神奇，但原理并不复杂：

```java
// 伪代码：Continuation 的工作方式
ContinuationScope scope = new ContinuationScope("demo");
Continuation continuation = new Continuation(scope, () -> {
    while (true) {
        // 任务 A：调用阻塞方法
        doBlockingOperation();
        // 这里会 yield，让出载体线程
        Continuation.yield(scope);
        // 恢复后继续执行
        doNextTask();
    }
});

while (!continuation.isDone()) {
    continuation.run();  // 运行，直到 yield 或完成
}
```

当 `Continuation.yield()` 被调用时，延续会暂停执行，状态被保存（局部变量、程序计数器等），载体线程被释放回线程池，可以去执行其他虚拟线程。当这个虚拟线程需要恢复时（例如 I/O 完成），调度器会找一个载体线程，把保存的状态恢复出来，继续执行。

这个过程发生在 JVM 层面，不涉及操作系统级的上下文切换，开销大幅降低。

### ForkJoinPool：虚拟线程的搬运工

虚拟线程的调度由 JDK 的 **ForkJoinPool** 实现——但不是普通的 ForkJoinPool，而是一个专门为虚拟线程优化的版本，称为 *FJP for Virtual*。

这个调度器的核心设计是"工作窃取"（Work Stealing）：每个载体线程有一个本地队列，当虚拟线程 yield 后空出来的载体线程，会尝试从其他线程的队列中"偷"任务来执行，最大化 CPU 利用率。

```java
// 虚拟线程的调度器配置
System.setProperty("jdk.virtualThreadScheduler.parallelism", "10");
// 载体线程池大小，默认是 CPU 核心数
```

调度器创建的载体线程数量，由 `jdk.virtualThreadScheduler.parallelism` 控制。如果你的应用是 I/O 密集型的，可以适当调高；如果 CPU 密集型的，`parallelism` 等于 CPU 核心数即可。

### 挂起 vs 阻塞：两种等待的本质区别

理解虚拟线程，有一个关键概念必须厘清：**挂起（Suspend）和阻塞（BLOCKING）的区别**。

**阻塞（Thread.sleep / synchronized）**：当一个平台线程在 `synchronized` 锁上阻塞时，这个线程会彻底"卡死"，操作系统不会调度它运行。但它依然占用着操作系统级的线程资源（1MB 栈空间、文件描述符等）。1000 个线程同时在 synchronized 上等待，就是 1000 个操作系统线程同时空转。

**挂起（虚拟线程的等待）**：虚拟线程的等待是通过 `Continuation.yield()` 实现的。挂起后，虚拟线程不消耗任何载体线程的资源——载体线程被释放回调度池，去执行其他虚拟线程。1000 个虚拟线程同时等待 I/O，底层只需要 10 个载体线程。

```java
// 这段代码在虚拟线程中执行时
URL url = new URL("https://api.example.com/data");
HttpURLConnection conn = (HttpURLConnection) url.openConnection();
// openConnection() 底层会 yield，不会阻塞载体线程
```

JDK 的核心类库（`java.net`、NIO、File I/O 等）已经改造为"虚拟线程友好"的实现——所有阻塞操作都会触发 yield，而非真正的 OS 阻塞。

### 为什么虚拟线程不能替代平台线程

这里有一个容易被忽视的陷阱：**synchronized 同步块**。

`java.lang.Thread` 的 `synchronized` 关键字，内部是通过 `ObjectMonitor` 实现的。当虚拟线程在 synchronized 块上等待时，它实际上会进入**重量级锁**（JDK 6+ 的自适应锁）状态，导致**载体线程被阻塞**而非挂起。

```java
// 这种用法会让虚拟线程"退化"为平台线程
synchronized (sharedResource) {
    // 当大量虚拟线程同时争用这个锁时，
    // 它们的载体线程会被阻塞，虚拟线程的优势荡然无存
    process(sharedResource);
}
```

这就是为什么 JDK 21 引入了**结构化并发**（Structured Concurrency）和**Scoped Values**——它们提供了更安全的并发原语，避免了 synchronized 在虚拟线程环境下的性能陷阱。

另一个不适用于虚拟线程的场景是**CPU 密集型任务**。如果你的任务是纯计算（不涉及 I/O），虚拟线程无法提供加速——因为 CPU 无法被"共享"，计算必须真正在某个 CPU 核心上执行。此时，应该使用平台线程（传统线程池）来处理 CPU 密集型工作。

## 如何使用虚拟线程

### 创建虚拟线程

JDK 21 中，创建虚拟线程非常简单：

```java
// 方式一：Thread.ofVirtual() 工厂方法（JDK 21）
Thread virtualThread = Thread.ofVirtual().name("my-vt").start(() -> {
    System.out.println("Running in a virtual thread!");
});

// 方式二：Thread.startVirtualThread() 快捷方法
Thread vt = Thread.startVirtualThread(() -> {
    System.out.println("Quick start virtual thread");
});

// 方式三：ExecutorService 工厂方法
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    Future<String> future = executor.submit(() -> {
        return "result from virtual thread";
    });
    System.out.println(future.get());
}
```

`newVirtualThreadPerTaskExecutor()` 是 JDK 21 推荐的方式——每个任务创建一个虚拟线程，任务完成后虚拟线程自动销毁，**无需线程池管理**。这彻底改变了"线程池 + 任务队列"的老套路。

### 迁移现有代码

如果你有基于传统线程池的代码，迁移到虚拟线程的改动通常很小：

```java
// 传统模式：线程池 + Runnable
ExecutorService executor = Executors.newFixedThreadPool(100);
executor.submit(() -> fetchData(url));

// 虚拟线程模式：每任务一线程，无需池
try (ExecutorService executor = Executors.newVirtualThreadPerTaskExecutor()) {
    executor.submit(() -> fetchData(url));
}
```

但迁移时必须注意以下几点：

第一，检查 synchronized 使用。分析所有 synchronized 块，评估并发争用的程度。对于高争用的场景，改用 `java.util.concurrent` 包中的显式锁（`ReentrantLock`、`ReadWriteLock` 等）——这些锁的等待是公平的、虚拟线程友好的，不会阻塞载体线程。

第二，避免 ThreadLocal 滥用。ThreadLocal 绑定到平台线程，在虚拟线程环境中容易导致内存泄漏——虚拟线程用完即销毁，但 ThreadLocal 值不会被及时清理。如果必须使用 ThreadLocal，用 `Thread.ofVirtual()` 创建虚拟线程时，通过 `inheritInheritableThreadLocals(false)` 禁用继承。

第三，线程局部句柄限制。虚拟线程的内部实现使用了大量线程局部变量（如用于调试信息的 `java.lang.Thread.threadLocalHashCode`），某些极端场景下可能导致句柄耗尽。不过这在正常业务场景中几乎不会遇到。

### 虚拟线程的性能边界

说了这么多优势，虚拟线程的极限在哪里？

**吞吐量基准测试**（来自 JEP 444 的性能数据）：在典型的 I/O 密集型基准测试中，虚拟线程相比传统线程：

- 吞吐量（任务/秒）：提升 2~5 倍（取决于 I/O 等待时间占比）
- 内存占用：降低 10~50 倍（假设每个平台线程 1MB 栈，虚拟线程仅 ~1KB）
- 延迟：抖动更低（避免了线程池排队等待）

但这些数字取决于具体场景。对于 CPU 密集型任务，虚拟线程与平台线程性能相当——因为没有 I/O 等待可以"共享"。

## 虚拟线程的适用场景

那么，什么时候应该使用虚拟线程？

**强烈推荐使用虚拟线程的场景：**

高并发 I/O 密集型服务是最典型的场景。HTTP API 调用、数据库查询、消息队列消费——这些任务大部分时间都在等待 I/O 完成。使用虚拟线程，可以用极低的资源成本创建大量并发连接。

微服务架构中的**请求处理**也非常适合虚拟线程。每个请求一个虚拟线程，请求间的 I/O 等待相互隔离，代码风格依然是同步阻塞的（比 CompletableFuture 回调链可读性更好），但资源消耗接近异步。

**不推荐使用虚拟线程的场景：**

CPU 密集型任务（如图像处理、加密计算、复杂数学运算）不适合虚拟线程。此时应该使用平台线程配合 `ForkJoinPool.common()`，或者直接使用 Java 的 Vector API 进行 SIMD 并行化。

高度争用的 synchronized 代码块在迁移到虚拟线程后可能性能下降。如果无法重构为显式锁，应该保持使用平台线程。

## 展望：结构化并发与虚拟线程的未来

虚拟线程不是孤立的技术。它是 JDK 21 整体并发模型升级的一部分。

**结构化并发**（Structured Concurrency，JDK 21 Incubating）：将多个在不同的虚拟线程中运行的任务视为一个单元——一个线程失败时，其他相关线程自动取消。这让"多个任务协同完成一个操作"的场景，代码结构更清晰：

```java
try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {
    String user = scope.fork(() -> findUser(id));
    String orders = scope.fork(() -> fetchOrders(id));
    scope.join();           // 等待所有任务完成
    scope.throwIfFailed();  // 任何一个失败就抛异常
    System.out.println(user.result() + " : " + orders.result());
}
```

**Scoped Values**（JDK 21 Incubating）：替代 ThreadLocal 的虚拟线程友好方案。当数据需要在调用链中传递时，Scoped Values 比 ThreadLocal 更安全——它会自动在虚拟线程结束时清理，不会导致内存泄漏。

---

线程的诞生，是为了在单个 CPU 上模拟"同时执行多个任务"。经过三十年的演进，Java 的线程模型经历了从绿色线程（JDK 1.0 的用户级线程，后来被放弃）到平台线程（JDK 1.2+），再到如今的虚拟线程。

但虚拟线程的意义远不止性能提升。它重新定义了我们对"并发"的理解：**并发不应该是稀缺资源，而应该是随手可得的工具**。

当一百万个并发连接变得稀松平常，当"按需创建、随手丢弃"成为编写高并发代码的标准方式，Java 才真正进入了现代化的并发时代。
