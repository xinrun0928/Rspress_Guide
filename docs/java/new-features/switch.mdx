# Switch 表达式增强

switch 语句大概是 Java 语法中最"复古"的存在了。

从 JDK 1.0 诞生之初，switch 就几乎是现在的模样——标签式的语法、fall-through 的默认行为、只能匹配常量的 case。这种设计继承自 C 和 C++，在那个年代是合理的，但三十年过去了，Java 世界早已天翻地覆，switch 却好像被时间遗忘了。

直到 JDK 12，一切开始改变。

## 传统 switch 的"七宗罪"

在深入新特性之前，我们先来盘点一下传统 switch 的问题。这些问题你可能早已习以为常，但它们确实是 Java 语法中最容易出错的角落。

**第一宗：fall-through 陷阱**。这是传统 switch 最臭名昭著的问题。忘记在 case 末尾加 break？代码会"穿透"到下一个 case，执行不属于它的逻辑。这种 bug 极难发现——逻辑上看起来没问题，编译器也不报错，只有运行后才能看到症状。

**第二宗：重复代码噩梦**。当多个 case 需要相同的处理逻辑时，传统写法只能把代码堆在一起：

```java
switch (month) {
    case 1: case 3: case 5: case 7:
    case 8: case 10: case 12:
        days = 31;
        break;
    case 4: case 6: case 9: case 11:
        days = 30;
        break;
}
```

`case 1: case 3:` 这种写法语义正确，但读起来像"乱码"，代码可读性极差。

**第三宗：无法作为表达式**。switch 只能作为语句使用，不能返回值。这意味着如果你需要根据枚举值返回一个计算结果，不得不借助临时变量或者三目运算符的嵌套地狱。

**第四宗：只能匹配常量**。case 标签只能是编译时常量，无法匹配范围、类型或任意条件。

**第五宗：null 导致 NPE**。如果被 switch 的对象是 null，抛 NullPointerException，没有显式的 null 处理。

**第六宗：缺少 default 分支的警告**。对于有限枚举的 switch，如果遗漏了 default，编译器虽然会警告，但如果你新增了一个枚举值而忘记更新 switch，警告也不会自动消失。

**第七宗：重复 case 检测不完整**。编译器能检测字面量重复的 case（如 `case 1: case 1:`），但对于涉及常量的表达式，检测能力有限。

## 箭头语法：消除 fall-through

JDK 12 引入的**箭头语法**（`->`），从根本上解决了 fall-through 问题：

```java
int days = switch (month) {
    case 1, 3, 5, 7, 8, 10, 12 -> 31;
    case 4, 6, 9, 11 -> 30;
    case 2 -> isLeapYear ? 29 : 28;
    default -> 0;
};
```

每个 `case L ->` 箭头右侧可以是一个**表达式**（返回值）、**语句块**或**throw 语句**。箭头语法的核心保证：**不会发生 fall-through**。一旦匹配，立即从 switch 中返回，不会继续执行后续 case。

`case 1, 3, 5` 的多值合并写法也是新引入的——多个常量可以写在一个 case 标签后，用逗号分隔，进一步减少了重复代码。

## switch 作为表达式：返回值的新用法

箭头语法让 switch 可以作为**表达式**使用，返回一个值。这使得"匹配并计算"的场景更加自然：

```java
// JDK 12 之前：临时变量 + switch 语句
int grade;
switch (score) {
    case 10, 9 -> grade = 1;    // 优良
    case 8, 7 -> grade = 2;     // 良好
    case 6, 5, 4 -> grade = 3; // 及格
    default -> grade = 0;      // 不及格
}

// JDK 12+：switch 作为表达式直接返回值
int grade = switch (score) {
    case 10, 9 -> 1;
    case 8, 7 -> 2;
    case 6, 5, 4 -> 3;
    default -> 0;
};
```

这个变化不仅仅是语法的便利——它让**穷尽性检查**成为可能。如果一个 switch 表达式没有覆盖所有可能的情况（例如枚举的某个值没有对应 case），编译器会报错。这在编译期就能捕获遗漏，而不是等到运行时才发现 bug。

## 模式匹配 for switch：不仅仅是常量

从 JDK 17 开始预览、JDK 21 正式转正的**模式匹配 for switch**，将 switch 的匹配能力从"常量比较"扩展到了"模式匹配"。

### 类型模式

传统 switch 只能匹配固定值。模式匹配让 switch 可以匹配类型：

```java
// JDK 17 之前：多重 instanceof + 类型转换
if (obj instanceof Integer) {
    int i = (Integer) obj;
    System.out.println(i * 2);
} else if (obj instanceof String) {
    String s = (String) obj;
    System.out.println(s.length());
}

// JDK 21：类型模式 + switch = 清晰的结构
switch (obj) {
    case Integer i -> System.out.println(i * 2);
    case String s -> System.out.println(s.length());
    case null -> System.out.println("null");
    default -> System.out.println("unknown");
}
```

### Guarded Pattern：when 子句

`case Integer i` 匹配所有 Integer。如果只想匹配正整数呢？**Guarded Pattern**（带条件的模式）通过 `when` 子句解决这个问题：

```java
switch (obj) {
    case Integer i when i > 0 -> "正整数: " + i;
    case Integer i -> "非正整数: " + i;
    case String s when s.length() > 10 -> "长字符串: " + s;
    case String s -> "短字符串: " + s;
    case null -> "null";
    default -> "未知类型";
}
```

注意顺序：更具体的模式（正整数）必须写在更通用的模式（任何整数）之前。Java 的 switch 不保证 case 的求值顺序，所以这种排序必须手动控制。

### null 的显式处理

传统 switch 对 null 的处理是"一冲了事"——直接抛 NPE。模式匹配 for switch 允许显式处理 null：

```java
switch (str) {
    case null -> System.out.println("字符串为 null");
    case "" -> System.out.println("空字符串");
    default -> System.out.println("内容: " + str);
}
```

显式处理 null 不仅避免了 NPE，更是一种**意图声明**——告诉代码阅读者："null 是预期的输入，这里有对应的处理逻辑。"

### Record 模式

模式匹配 for switch 与 Record 结合，可以优雅地解构嵌套数据结构：

```java
sealed interface Shape permits Circle, Rectangle {}
record Circle(double radius) implements Shape {}
record Rectangle(double width, double height) implements Shape {}

double area = switch (shape) {
    case Circle(double r) -> Math.PI * r * r;
    case Rectangle(double w, double h) -> w * h;
};
```

`Circle(double r)` 就是 Record 模式——检查 shape 是 Circle 类型，同时将 radius 字段解构到变量 `r` 中。

## 实际应用：从枚举分发到多态处理

### 替代策略模式

传统的策略模式需要维护策略映射表：

```java
// 策略模式传统实现
Map<String, PaymentStrategy> strategies = Map.of(
    "credit", new CreditCardStrategy(),
    "alipay", new AlipayStrategy(),
    "wechat", new WechatPayStrategy()
);
strategies.get(type).pay(amount);
```

使用 switch + 密封类，可以达到类似效果且更直观：

```java
sealed interface Payment permits CreditCard, Alipay, WechatPay {}

double pay(Payment payment, double amount) {
    return switch (payment) {
        case CreditCard cc -> creditPay(cc, amount);
        case Alipay a -> alipayPay(a, amount);
        case WechatPay w -> wechatPay(w, amount);
    };
}
```

编译器会检查是否穷尽了所有 Payment 类型——如果新增了一个 `UnionPay` 但忘记在 switch 中处理，编译直接失败。

### 替代 if-else 链

对于多层 if-else 的类型检查，switch + 模式匹配提供了更清晰的替代方案：

```java
// 之前：深层嵌套的 if-else
public String describe(Object obj) {
    if (obj == null) return "null";
    if (obj instanceof Integer) {
        Integer i = (Integer) obj;
        if (i > 0) return "正整数";
        else return "非正整数";
    }
    if (obj instanceof String) {
        String s = (String) obj;
        if (s.isEmpty()) return "空字符串";
        else return "非空字符串";
    }
    return "其他";
}

// 现在：switch + guarded pattern，更清晰
public String describe(Object obj) {
    return switch (obj) {
        case null -> "null";
        case Integer i when i > 0 -> "正整数";
        case Integer i -> "非正整数";
        case String s when s.isEmpty() -> "空字符串";
        case String s -> "非空字符串";
        default -> "其他";
    };
}
```

switch 的演进史，就是 Java 语法现代化的一个缩影。从 C 风格的标签跳转，到表达式化的类型匹配，switch 已经从一个"勉强能用"的控制结构，演变成了一个真正现代化的多路分发工具。

但 switch 增强并非孤军奋战——它与密封类、Record、模式匹配共同构成了 Java 类型系统的三驾马车。下一节，我们就来深入了解密封类，看看 Java 如何通过它实现了"继承的精确控制"。
