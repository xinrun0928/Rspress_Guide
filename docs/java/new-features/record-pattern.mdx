# Record 类型与模式匹配

写一个数据传输对象（DTO），在 Java 中大概需要多少行代码？

我们来做个实验。假设需要一个表示二维坐标点的 `Point` 类，包含 x 和 y 两个字段，还需要 equals()、hashCode()、toString() 以及 getter 方法。你大概会写出这样的代码——三十多行，可能更多。如果还需要构造函数校验？那就奔着五十行去了。

```java
public class Point {
    private final int x;
    private final int y;

    public Point(int x, int y) {
        this.x = x;
        this.y = y;
    }

    public int x() { return x; }
    public int y() { return y; }

    @Override
    public boolean equals(Object o) {
        if (!(o instanceof Point)) return false;
        Point p = (Point) o;
        return x == p.x && y == p.y;
    }

    @Override
    public int hashCode() {
        return Objects.hash(x, y);
    }

    @Override
    public String toString() {
        return "Point[x=" + x + ", y=" + y + "]";
    }
}
```

而这一切，在 Java 16 之后，一行代码就能搞定：

```java
public record Point(int x, int y) {}
```

这就是 Record。但 Record 的意义远不止"省代码"这么简单。它的背后是一整套关于**值对象**、**代数数据类型**、以及**模式匹配**的设计哲学。

## Record 的本质：数据作为一等公民

要理解 Record，首先要理解 Java 设计者为什么要引入它。

Java 诞生之初，everything is object（一切皆对象）的哲学贯穿始终。但随着编程范式的演进，一个问题逐渐浮出水面：**并不是所有对象都需要行为**。

试想一下：`Point` 类的 `x()` 和 `y()` getter 方法，是"行为"吗？不是，它们只是暴露数据。`equals()` 和 `hashCode()` 是"行为"吗？也不是，它们是数据的等价性契约。`toString()` 是"行为"吗？更不是，它是数据的字符串表示。这些方法都服务于一个目的——**描述数据**。

但长期以来，Java 的 class 是唯一的抽象机制。你想描述一个数据结构，就必须用 class，就必须接受它带来的全部复杂性——可变状态、继承体系、访问控制。**Record 的引入，本质上是将"数据"从"行为"中解放出来，让 Java 也有能力将数据作为一等公民来对待。**

这在函数式编程语言中早有先例。Scala 的 case class、Kotlin 的 data class、Haskell 的 Algebraic Data Type（代数数据类型），都体现了"数据应该被平等对待"的理念。Java Record 是 Java 世界对这一理念的迟到但坚定的回应。

## Record 的核心特性

### 自动生成的方法

当你声明 `public record Point(int x, int y) {}` 时，编译器会自动生成：

```java
// 1. 构造函数（参数即为字段）
public Point(int x, int y) { this.x = x; this.y = y; }

// 2. accessor 方法（注意：不是 getX()/getY()，而是 x()/y()）
public int x() { return x; }
public int y() { return y; }

// 3. equals()：基于内容比较，两个 Point相等当且仅当 x和 y都相等
public boolean equals(Object o) {
    return o instanceof Point p && x == p.x && y == p.y;
}

// 4. hashCode()：基于内容的哈希
public int hashCode() { return Objects.hash(x, y); }

// 5. toString()：包含所有字段值的字符串表示
public String toString() {
    return "Point[x=" + x + ", y=" + y + "]";
}
```

注意 accessor 方法的命名：**`x()` 和 `y()`，而不是 `getX()` 和 `getY()`**。这是刻意的设计决策。Record 代表的是数据的"投影"，不是 JavaBean 的属性。JavaBean 的 getter/setter 模式是为可封装、可变对象设计的，而 Record 的 accessor 直接以字段名为方法名，更简洁，也更符合数据优先的理念。

### 不可变性

Record 的所有字段都是隐式 final 的。一旦创建，就无法修改。

```java
Point p = new Point(1, 2);
p.x();   // 1
// p.x = 3;  // 编译错误！Record 不可变
```

这种不可变性是 Record 的核心约束之一。不可变对象天然线程安全——没有可变状态，就不存在竞态条件；不可推导性（referential transparency）使得 Record 在并发和函数式编程场景下格外安全。

### 自定义成员

Record 可以添加额外的方法、静态字段和静态方法：

```java
public record Point(int x, int y) {
    // 自定义构造函数：添加校验逻辑
    public Point {
        if (x < 0 || y < 0) {
            throw new IllegalArgumentException(
                "Coordinates must be non-negative");
        }
    }

    // 添加计算方法
    public double distanceFromOrigin() {
        return Math.sqrt(x * x + y * y);
    }

    // 添加静态工厂方法
    public static Point origin() {
        return new Point(0, 0);
    }

    // 添加静态字段
    public static final Point[] UNIT_VECTORS = {
        new Point(1, 0), new Point(0, 1)
    };
}
```

注意自定义构造函数的语法：`public Point { ... }`，而不是 `public Point(int x, int y) { ... }`。这个紧凑构造函数语法用于对参数进行校验或转换，而无需重新声明参数列表。

### Record 与泛型

Record 完全支持泛型，可以定义带类型参数的 Record：

```java
public record Pair<K, V>(K key, V value) {}

Pair<String, Integer> pair = new Pair<>("age", 30);
String key = pair.key();   // "age"
Integer value = pair.value(); // 30
```

### Record 的继承

Record 有一些特殊的继承规则，与普通 class 有显著不同：

**Record 不能继承其他类**（但可以实现接口）：

```java
// 合法：Record 实现接口
public record Point(int x, int y) implements Serializable {}

// 非法：Record 不能继承类
// public record ColoredPoint(int x, int y, Color color) extends Point {} // 编译错误！
```

**Record 的继承通过接口实现**：

```java
public interface Identified {
    long getId();
}

public record User(long id, String name) implements Identified {
    public long getId() { return id; }
}
```

**Record 可以继承其他 Record 吗？** 也不行。Record 的继承层次是扁平的——每个 Record 都是独立的最终实现，不存在"父子 Record"的继承链。这是因为 Record 强调的是**数据的组合**而非**类型的派生**，继承会引入复杂性和不确定性。

### Record 与本地类

Record 是 `java.lang.Record` 的隐式最终子类。这意味着：
- Record 属于引用类型，存储在堆内存中（不是值类型）
- Record 可以被序列化（实现了 `Serializable`）
- Record 可以用于类型强制转换和 `instanceof` 检查
- Record 可以用 `==` 和 `!=` 比较（基于 equals()，即内容相等性）

## Record 与模式匹配

Record 真正大放异彩的舞台，是与**模式匹配**的结合。

JDK 16 引入的 instanceof 模式匹配，允许在 instanceof 检查的同时解构对象：

```java
// JDK 16 之前
if (obj instanceof Point) {
    Point p = (Point) obj;
    double dist = Math.sqrt(p.x() * p.x() + p.y() * p.y());
}

// JDK 16+：模式变量自动可用
if (obj instanceof Point p) {
    double dist = Math.sqrt(p.x() * p.x() + p.y() * p.y());
    // 编译器确保：在这个分支内，p 一定是 Point 类型
}
```

JDK 21 正式引入的 **Record 模式**，更进一步——`instanceof` 不再只检查类型，还能直接解构 Record 的字段：

```java
// JDK 21 Record 模式：类型检查 + 字段解构 一步到位
if (obj instanceof Point(int x, int y)) {
    System.out.println(x + y);  // x 和 y 直接可用，无需 p.x() 调用
}
```

注意这里的 `int x, int y`——这是 Record 的**解构模式**（Deconstructor Pattern）。编译器会将 Record 的 accessor 方法与解构模式自动对应起来。

结合 switch，Record 模式的威力更加明显：

```java
String describeShape(Object shape) {
    return switch (shape) {
        case null -> "null";
        case Point p -> "点(x=%d, y=%d)".formatted(p.x(), p.y());
        case Circle(int radius) -> "圆形，半径=" + radius;
        case Rectangle(int width, int height) ->
            "矩形，宽=" + width + "，高=" + height;
        case Shape s -> "其他形状: " + s;
    };
}
```

当 `shape instanceof Circle(int radius)` 匹配成功时，`radius` 自动绑定为圆的半径值。这不是字符串替换，不是反射，而是**编译器的静态类型推导**——在编译时就确定了类型的正确性，性能与手写分支完全一致。

## Record 在实际项目中的应用

### DTO 和值对象

Record 最自然的用途就是替代传统的 DTO：

```java
// 之前：漫长的 POJO
public class UserDTO {
    private final Long id;
    private final String name;
    private final String email;
    private final LocalDateTime createdAt;
    // ... 构造函数、getters、equals、hashCode、toString
}

// 之后：清晰简洁
public record UserDTO(
    Long id,
    String name,
    String email,
    LocalDateTime createdAt
) {}
```

### 复合键

Map 的 key 需要同时考虑 equals() 和 hashCode()，Record 的自动生成让复合键的定义变得极为简洁：

```java
public record CompositeKey(String category, String id) {}

Map<CompositeKey, Product> productMap = new HashMap<>();
productMap.put(new CompositeKey("electronics", "SKU001"), product);
// 自动 equals/hashCode，无需手动实现
```

### 函数式返回复合值

方法需要返回多个值时，Record 替代了手动创建 Pair 或自定义 Result 类的繁琐：

```java
public record SearchResult<T>(
    List<T> items,
    int totalCount,
    boolean hasMore
) {}

// 使用
SearchResult<User> result = userService.search(query);
result.items().forEach(System.out::println);
```

### 密封类与 Record 的结合

Record 与密封类是天作之合，共同构成代数数据类型的完整体系：

```java
// 定义密封层次
public sealed interface Expr permits Number, Add, Subtract, Multiply {}

// 用 Record 表示每种表达式变体
public record Number(int value) implements Expr {}
public record Add(Expr left, Expr right) implements Expr {}
public record Subtract(Expr left, Expr right) implements Expr {}
public record Multiply(Expr left, Expr right) implements Expr {}

// 求值：穷尽性检查保证没有遗漏
int evaluate(Expr e) {
    return switch (e) {
        case Number n -> n.value();
        case Add a -> evaluate(a.left()) + evaluate(a.right());
        case Subtract s -> evaluate(s.left()) - evaluate(s.right());
        case Multiply m -> evaluate(m.left()) * evaluate(m.right());
    };
}
```

这里编译器可以验证：所有 `Expr` 的可能实现都被 switch 覆盖了。如果你添加了一个新的 `Divide` 实现但忘记更新 switch，编译器会报错。

## Record 的局限性

Record 不是万能的。理解它的局限性，才能正确地使用它。

**状态不可变意味着不适合可变对象**。如果你的对象需要被修改（比如 Builder 模式、可变计数器等），Record 帮不上忙，传统的 class 仍然是首选。

**Record 不能表示实体**。实体对象有身份认同——两个 id 相同的 `User` 是同一个用户，即使其他字段不同。这种"身份 vs 值"的区分是领域驱动设计中的重要概念，Record 只能覆盖"值"这一半。

**Record 序列化有特殊注意事项**。虽然 Record 实现了 `Serializable`，但它没有无参构造函数。在使用 Kryo、FST 等依赖反射序列化的框架时，可能需要额外配置。

**Record 不适合需要动态扩展的场景**。Record 一旦声明，字段就固定了。如果你的数据结构需要运行时动态添加字段，普通 class 或者 `Map<String, Object>` 更合适。

## Record 与 Kotlin data class 的比较

作为对比，Kotlin 的 data class 在 2016 年就引入了类似功能：

```kotlin
// Kotlin
data class Point(val x: Int, val y: Int)
```

Java Record 与 Kotlin data class 在核心理念上高度一致，但有一些差异值得注意：

- Kotlin 的 accessor 使用 `x`、`y`（属性语法），Java 使用 `x()`、`y()`（方法语法）。Java 的方式保持了方法的纯粹性，不引入新概念。
- Kotlin 支持解构声明 `val (a, b) = point`，Java 通过 Record 模式实现类似功能。
- Kotlin 的 data class 默认可以继承（除非标记为 final），Java Record 不可继承类，体现了更彻底的数据不可变哲学。

---

回到开头的那个问题：Record 的出现，真的只是让我们少写几行代码吗？

不完全是。它的更深层含义是：**承认"数据"与"行为"是不同的抽象，给它们配备不同的工具**。当一个对象的本质是一组数据的集合时，Record 让这种本质得以显式化，而不是被包裹在一个什么都做的 class 里。

这个转变的影响是深远的。它让代码更简洁、更安全（不可变意味着没有意外的修改），也让类型系统更强大（密封类 + Record + 模式匹配构成了代数数据类型的基础）。
