候选人小刘在阿里P7面试间，面试官翻到他的简历——上面没有写模块化的内容，但面试官随口问了一句：

"Java 9的模块化你了解吗？"

小刘说："知道一点，就是加了一个module-info.java文件。"面试官追问："`requires`和`requires static`有什么区别？为什么会有`exports`？那`opens`呢？"

小刘愣住。

## 一、JPMS 到底是什么？🔴

### 1.1 问题拆解

这道题是**P7区分题**，Java 9模块化是Java历史上最大的架构变更之一，但大多数候选人只停留在"知道有这么回事"的程度。

**第一层：核心概念**
- 面试官问："JPMS是什么？解决什么问题？"
- 考察点：强封装、模块路径 vs 类路径、可读性

**第二层：module-info语法**
- 面试官追问："module-info怎么写？exports/requires/opens有什么区别？"
- 考察点：exports限制导出、opens允许反射

**第三层：迁移策略**
- 面试官问："老项目怎么迁移到模块化？会遇到什么问题？"
- 考察点：JAR地狱、未命名模块、自动模块

**第四层：生产适用性**
- 面试官问："你们项目用了模块化吗？为什么不用？"
- 考察点：迁移成本、收益评估、Spring Boot兼容性

### 1.2 ❌ 错误示范

**错误回答1："模块化就是加个module-info.java文件，让项目结构更清晰"**

这完全没有理解模块化的核心价值。模块化不是为了"结构清晰"，而是为了**强封装**——控制哪些包可以被外部访问。如果没有这个需求，模块化就是过度设计。

**错误回答2："exports就是导出包，requires就是依赖模块，很简单"**

`requires static` vs `requires transitive`的区别、`opens` vs `exports`的差异，这些细节才是面试官想追问的。

**错误回答3："模块化很好用，我们项目已经全量迁移了"**

大多数项目没有迁移模块化。Spring Boot 3才勉强支持模块路径，迁移成本极高。这个回答要么是吹牛，要么说明他没踩过迁移的坑。

【面试官心理】
Java模块化是P7的深度题。我不是在问他会不会用module-info.java，我是在试探他有没有理解Java平台的**强封装**设计意图，以及有没有实际迁移过老项目的经验。能说出"自动模块"、"未命名模块"、"层叠依赖"这些概念的，基本都研究过模块化的迁移方案。

### 1.3 标准回答

#### 第一步：JPMS解决的核心问题——强封装

在JPMS之前，Java的访问控制只有`public`、`protected`、`private`和包私有四级。`public`意味着**无条件公开**——任何人都能访问。

```java
// 模块化前的访问控制
public class InternalService {
    // public意味着全世界都能访问
    public void dangerousMethod() { }
}
```

模块化引入了第五级访问控制——**模块级访问控制**：

```java
// module-info.java
module com.myapp.core {
    // 导出com.myapp.core.api包，允许其他模块依赖
    exports com.myapp.core.api;

    // 只在模块内部可见，不导出
    // exports com.myapp.core.internal; // 没有这行，internal包对外不可见

    // 依赖其他模块
    requires com.myapp.utils;
}
```

关键概念：
- **`exports`**：显式声明哪些包可以被其他模块访问
- **`requires`**：声明依赖哪些模块
- **未导出的包即使`public`，其他模块也无法访问**

#### 第二步：module-info语法详解

```java
module com.myapp.api {
    // 1. 依赖声明
    requires java.sql;                        // 运行时必需
    requires static org.slf4j;               // 编译时必需，运行时可选

    // 2. 传递依赖（transitive）
    requires transitive com.myapp.utils;     // 使用本模块的地方也自动获得utils
    // 如果A requires B, B requires transitive C，则A自动可以访问C

    // 3. 导出控制
    exports com.myapp.api.service;           // 导出service包
    exports com.myapp.api.dto;               // 导出dto包
    // com.myapp.api.internal包不导出，外部无法访问

    // 4. 服务提供（ServiceLoader机制）
    provides com.myapp.api.spi.Plugin        // 本模块提供Plugin接口的实现
        with com.myapp.api.impl.FilePlugin;

    uses com.myapp.api.spi.Plugin;           // 本模块使用Plugin接口
}
```

**requires static vs requires**：

```java
// requires：运行时必需，模块找不到会直接失败
requires java.logging; // 应用代码依赖日志，打印日志必须

// requires static：编译时必需，运行时可选
// 编译时需要检查类型，但运行时如果没有也不报错
requires static lombok; // 编译工具，运行时不需要
```

**requires transitive**：

```java
// 不使用transitive时
// A requires B, B exports C
// A不能直接访问C，必须A也requires C

// 使用transitive时
// A requires B, B requires transitive C
// A自动获得对C的读取权限，不用再显式requires
// 避免A的开发者在不知情的情况下用不到C
```

#### 第三步：opens——反射访问控制

```java
// 严格模式：未导出的包，reflection也访问不了
// 被强制访问时，抛出InaccessibleObjectException

// 解决方案：opens
module com.myapp {
    opens com.myapp.entity;                 // 允许reflection访问
    opens com.myapp.entity to com.fasterxml.databind; // 只对特定模块开放reflection
}
```

`opens`的三个级别：

| 语法 | 说明 |
| --- | --- |
| `opens pkg` | 允许所有模块reflection |
| `opens pkg to target1, target2` | 只对指定模块开放reflection |
| 无opens | 严格限制reflection访问（Java 9+默认） |

:::warning ⚠️
**Spring/Hibernate的模块化兼容问题**：Spring大量使用反射访问Bean属性，Java 9模块化默认禁止未导出包的反射访问。Spring 5/6通过`opens`指令和`--add-opens` JVM参数来解决，但迁移到完整模块化仍然困难重重。
:::

## 二、模块路径 vs 类路径 🟡

### 2.1 核心区别

```
类路径（Classpath）：
- JAR文件散落存放，无结构性
- 所有public类型都可以被访问（除非你明确用了Class.forName黑科技）
- JAR地狱：多个JAR包含同名类，取决于classpath顺序

模块路径（Module Path）：
- JAR被组织成模块，有声明的依赖关系
- 未导出的类型不可访问
- 依赖关系有向无环图，可被JVM验证
```

```bash
# 类路径模式启动（Java 8及以下）
java -cp "lib/*:app.jar" com.myapp.Main

# 模块路径模式启动（Java 9+）
java -p mods:app.jar -m com.myapp/com.myapp.Main
# -p: 模块路径
# -m: 主模块
```

### 2.2 三类模块

| 类型 | 说明 | 特点 |
| --- | --- | --- |
| 正式模块（Named Module） | 有module-info.java的JAR | 严格遵守exports/requires规则 |
| 自动模块（Automatic Module） | 无module-info.java但放在模块路径 | 自动exports所有包，requires所有JAR |
| 未命名模块（Unnamed Module） | 放在类路径中的代码 | 可以访问所有模块，但自身不被命名 |

```bash
# 场景：混用模块路径和类路径
java -p mymodule.jar -cp legacy.jar com.myapp.Main
# mymodule.jar: 正式模块（module-info）
# legacy.jar: 类路径中的未命名模块
```

**自动模块的规则**：
- 自动导出所有包：`exports *`
- 自动依赖所有JAR：`requires`所有放在模块路径上的JAR
- JAR文件名变成模块名：比如`commons-lang3-3.12.jar` → 模块名`commons.lang.lang3`

## 三、迁移策略与生产选型 🟢

### 3.1 迁移路径

迁移一个老项目到模块化是**高成本低收益**的事情：

```
步骤1：给所有JAR添加module-info.java
  ↓ 每个JAR都要分析exports和requires
步骤2：处理传递依赖
  ↓ A requires B, B requires C → A需要知道C吗？
步骤3：处理反射代码
  ↓ Spring/hibernate的reflection调用
步骤4：测试所有模块组合
  ↓ 模块间依赖关系复杂
```

### 3.2 为什么大多数项目不用模块化？

| 项目 | 模块化 | 原因 |
| --- | --- | --- |
| JDK自身 | ✅ 全量 | 平台级，需要强封装 |
| 大型企业内部框架 | ⚠️ 部分 | 核心模块迁移，其他兼容 |
| Spring Boot应用 | ❌ | 迁移成本高，收益低 |
| 开源库 | ⚠️ 部分 | 提供module-info以便用户模块化，但保持无info兼容 |

Spring Boot 3.x（Spring Framework 6）通过`Automatic-Module-Name` manifest实现了**增量兼容**——既可以在模块路径上使用，也可以继续在类路径上使用。

### 3.3 `--add-opens` JVM参数

当代码需要reflection访问未导出包时（典型场景是JUnit、Spring、Hibernate），可以用JVM参数临时"破解"：

```bash
# --add-opens：开放指定模块的指定包给reflection
java --add-opens java.base/java.lang=ALL-UNNAMED \
      --add-opens java.base/java.util=ALL-UNNAMED \
      -jar myapp.jar
```

:::tip 💡
**生产建议**：模块化对于大多数应用项目来说是**过度设计**。如果团队规模小于10人、没有强封装需求，迁移模块化的成本远大于收益。但如果是在做平台级/SDK产品，模块化能提供清晰的API边界。
:::
