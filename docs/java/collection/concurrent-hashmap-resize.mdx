## 一、扩容触发条件 🔴

### 1.1 问题拆解

先搞清楚一个问题：ConcurrentHashMap 在什么情况下会触发扩容？

大部分候选人会把 HashMap 的扩容逻辑直接套用过来，这恰恰是第一个坑。

### 1.2 ❌ 错误示范

**背题型**：

> "当 size 超过 capacity * loadFactor 时触发扩容。"

**问题诊断**：
- 把 HashMap 的扩容逻辑直接套用
- 没有意识到 ConcurrentHashMap 是分段统计的
- 不理解 `sizeCtl` 的多重语义
- 忽略了"树化前扩容"这个特殊场景

【面试官心理】
> "这个候选人看过一些文章，但没理解 ConcurrentHashMap 和 HashMap 的本质区别。"

### 1.3 标准回答

ConcurrentHashMap 的扩容触发有**三种情况**，不是一种：

**第一种：put 后检查**

```java
private final void addCount(long x, int check) {
    CounterCell[] as = counterCells;
    long b = baseCount;

    // 尝试 CAS 更新 baseCount（热点竞争）
    if (as != null || !casBaseCount(b, b + x)) {
        // baseCount 更新失败，进入 CounterCell 数组计数
    }

    // check > 0 说明需要检查是否扩容
    if (check >= 0) {
        Node<K,V>[] tab, nt;
        int n, sc;
        // [!code highlight] 触发扩容的条件：sizeCtl > 数组长度
        while (sizeCtl > (n = tab.length)) {
            // 触发扩容
        }
    }
}
```

**第二种：链表树化时检查**

```java
// treeifyBin 方法
private final void treeifyBin(Node<K,V>[] tab, int index) {
    Node<K,V> b = tabAt(tab, index);
    if (b != null) {
        if (b.length >= TREEIFY_THRESHOLD) {
            // 如果桶内节点 >= 8，尝试树化
            // 但如果数组太小，先扩容再树化
            if (tab == null || (n = tab.length) < MIN_TREEIFY_CAPACITY)
                // [!code highlight] 数组 < 64 时优先扩容，不是树化
                tryPresize(n << 1);
        }
    }
}
```

:::warning ⚠️
**面试必考**：当数组大小小于 64 时，即使链表长度达到 8，也会**优先扩容而不是树化**。这是 JDK 的一个优化策略——小数组扩容能更快分散元素，比直接树化代价更小。
:::

**第三种：协助扩容（helpTransfer）**

```java
final V putVal(K key, V value, boolean onlyIfAbsent) {
    // ...
    else if ((fh = f.hash) == MOVED)
        // [!code highlight] 发现正在扩容，协助完成后再重试
        tab = helpTransfer(tab, f);
    // ...
}
```

### 1.4 【面试官心理】

面试官问扩容触发条件时，他真正想知道的是：

1. 你有没有注意到 `tryPresize` 方法——JDK8 新增的预扩容机制
2. 你知不知道数组小于 64 时的特殊处理——这是高频追问点
3. 你能不能说清楚"协助扩容"这个机制——这是 JDK8 的核心创新
4. 你分不分得清"分段扩容"是 JDK7 的设计——JDK8 已经完全不同

### 1.5 追问升级

**追问1**：为什么数组小于 64 时优先扩容？

**更优回答**：
> "因为小数组扩容的代价更小。假设数组长度是 16，链表长度是 8，扩容到 32 后，每个桶平均只有 4 个元素，不需要树化就能保证查询性能。树化需要额外的红黑树维护开销。"

**追问2**：ConcurrentHashMap 的 size() 方法怎么实现的？

**更优回答**：
> "JDK8 用 baseCount + CounterCell 数组实现并行计数，避免了全局 CAS 竞争。put 元素时优先 CAS 更新 baseCount，失败后才竞争 CounterCell。size() 是 baseCount + sum(counterCells)。"

## 二、JDK7 vs JDK8 扩容机制对比 🟡

### 2.1 问题拆解

JDK7 和 JDK8 的扩容机制**完全不同**，这不是简单的优化，而是架构级的重写。

### 2.2 【架构权衡】

JDK8 为什么要废除 Segment？

| 维度 | JDK7 Segment 分段 | JDK8 CAS + synchronized |
|------|-------------------|-------------------------|
| 内存开销 | 每个 Segment 独立对象，开销大 | 整个数组一次扩容 |
| 并发度 | 固定（创建时确定） | 动态，多线程协助 |
| 热门桶 | 锁住后仍是瓶颈 | 可被其他线程协助迁移 |
| 实现复杂度 | 两层 Map | 一层 Map，更简单 |

**核心原因**：JDK7 的 Segment 分段锁虽然保证了并发度，但"热门桶"（比如 hash 碰撞严重的桶）仍然是瓶颈。JDK8 改成全场并发，任何线程都可以帮助迁移任何桶。

### 2.3 标准回答

**JDK7：Segment 分段扩容**

```java
public class ConcurrentHashMap<K, V> {
    final Segment<K,V>[] segments;

    static final class Segment<K,V> extends ReentrantLock {
        transient volatile HashEntry<K,V>[] table;
    }

    // 扩容只影响单个 Segment
    final void rehash(HashEntry<K,V> node) {
        // 锁住这个 Segment（其他 Segment 不受影响）
        lock();
        try {
            Entry<K,V>[] newTable = new Entry[newCapacity];
            // 迁移数据...
        } finally {
            unlock();
        }
    }
}
```

JDK7 的扩容是**分段**的，每个 Segment 独立扩容，锁住一个 Segment 时其他 Segment 仍可用。

**JDK8：全场并发扩容**

```java
// 核心状态变量
private transient volatile int sizeCtl;      // 主控标志
private transient volatile int transferIndex; // 下一个待迁移桶的索引
private transient int cellsBusy;              // 初始化 CounterCells 的锁
```

## 三、JDK8 并发扩容核心流程 🔴

### 3.1 问题拆解

JDK8 的并发扩容是怎么实现的？多个线程如何分工？

### 3.2 【直观类比】

可以把 ConcurrentHashMap 扩容比作**仓库搬迁**：

- 旧仓库有 N 个货架，新仓库有 2N 个货架
- 每个货架（桶）有一队货物（链表节点）
- 搬迁时，先锁住一个货架，把货物分成两队：`hash & N == 0` 的留在原位附近，`!= 0` 的搬到对面货架
- 多个搬迁队同时工作，每队负责一段货架
- 搬迁完成后，在旧货架放一个"已搬迁"牌子（ForwardingNode）

### 3.3 扩容入口：tryPresize

```java
private final void tryPresize(int size) {
    // 计算目标容量：>= size * 1.5
    int c = (size >= (MAXIMUM_CAPACITY >>> 1)) ? MAXIMUM_CAPACITY :
            tableSizeFor(size + (size >>> 1) + 1);

    int sc;
    while ((sc = sizeCtl) >= 0) {
        Node<K,V>[] tab = table;
        int n = tab.length;

        if (c <= sc || n >= MAXIMUM_CAPACITY)
            break;  // 不需要扩容

        if (tab == table) {
            // 计算扩容戳记
            int rs = resizeStamp(n);

            if (sc < 0) {
                // 已经有线程在扩容
                Node<K,V>[] nt;
                if (sc >>> RESIZE_STAMP_SHIFT != rs)
                    break;  // 戳记不匹配，不参与
                if (nt = nextTable)
                    break;  // 正在迁移中
            }
            // [!code highlight] 首次触发：设置 sizeCtl 为负数
            else if (casSizeCtl(sc, (rs << RESIZE_STAMP_SHIFT) + 2))
                // 开始扩容，transferIndex = n（从后往前迁移）
                transfer(tab, null);
        }
    }
}
```

### 3.4 transfer 核心流程

```java
private final void transfer(Node<K,V>[] tab, Node<K,V>[] nextTab) {
    int n = tab.length, stride;

    // 每个线程最少迁移 16 个桶
    if ((stride = (NCPU > 1) ? (n >>> 3) / NCPU : n) < MIN_TRANSFER_STRIDE)
        stride = MIN_TRANSFER_STRIDE;

    // nextTab == null 说明是首个扩容线程
    if (nextTab == null) {
        try {
            @SuppressWarnings("unchecked")
            Node<K,V>[] nt = (Node<K,V>[]) new Node[n << 1];  // 2倍容量
            nextTab = nt;
        } catch (Throwable ex) {
            sizeCtl = Integer.MAX_VALUE;
            return;
        }
        nextTable = nextTab;
        transferIndex = n;  // 从后往前迁移
    }

    int nextn = nextTab.length;

    // 遍历所有桶进行迁移
    for (int i = 0, bound = 0;;) {
        // [!code highlight] 每个线程认领一段 [bound, i) 的桶
        while (i >= bound || i < 0) {
            if (i >= n) {
                // 迁移完成
                if (finishing) {
                    table = nextTable;
                    sizeCtl = n + (n >>> 1);
                    return;
                }
                finishing = true;
                i = n;
            }
            else if (i < 0) {
                // 任务被分配完了，重新认领
                int m = 0;
                // [!code highlight] 原子性地获取一段任务
                if ((m = transferIndex) <= 0)
                    i = -1;
                else if (U.compareAndSwapInt(this, TRANSFERINDEX, m,
                        m > stride ? m - stride : 0)) {
                    bound = m > stride ? m - stride : 0;
                    i = m - 1;
                }
            }
            else {
                // 处理单个桶
                Node<K,V> f = tabAt(tab, i);
                // ...
            }
        }
    }
}
```

### 3.5 桶迁移逻辑

```java
// 单个桶的迁移
private final void transfer(Node<K,V>[] tab, Node<K,V>[] nextTab) {
    // ...
    else {
        Node<K,V> f = tabAt(tab, i);
        if (f != null) {
            int fh = f.hash;
            if (fh == MOVED) {
                // 已经被其他线程迁移过了
                i = -1;
            }
            else {
                synchronized (f) {
                    // [!code highlight] 锁住桶，开始迁移
                    if (tabAt(tab, i) == f) {
                        Node<K,V> ln, hn;  // low node, high node
                        if (fh >= 0) {
                            // 普通链表节点
                            int runBit = fh & n;
                            // 分裂链表
                            for (Node<K,V> p = f; p != null; p = p.next) {
                                int b = p.hash & n;
                                if (b != runBit) {
                                    runBit = b;
                                    hn = p;
                                }
                                else {
                                    ln = p;
                                }
                            }
                            // 迁移到 nextTab（尾插法，防止链表反转）
                            setTabAt(nextTab, i, ln);
                            setTabAt(nextTab, i + n, hn);
                        }
                        // 红黑树迁移类似...
                        setTabAt(tab, i, new ForwardingNode<K,V>(nextTab));
                    }
                }
            }
        }
    }
}
```

### 3.6 sizeCtl 的多重语义

`sizeCtl` 是 ConcurrentHashMap 最核心的状态变量，不同阶段代表不同含义：

| 值 | 语义 | 说明 |
|---|---|---|
| `0` | 未初始化 | 使用默认容量 16 |
| `> 0` | 扩容阈值 | `capacity * loadFactor` |
| `-1` | 正在初始化 | 只有一个线程能进入 |
| `< -1` | 正在扩容 | `-(n+1)` 表示有 n 个线程在扩容 |

```java
// resizeStamp 计算
static final int resizeStamp(int n) {
    // Integer.numberOfLeadingZeros(n) 返回 n 前面的零位数
    // 例如 n=16，二进制是 10000，前面有 27 个零
    // 返回 0x80010000 | 27 = 0x8001001b
    return Integer.numberOfLeadingZeros(n) | (1 << (RESIZE_STAMP_SHIFT - 1));
}
```

## 四、生产避坑 🟡

### 4.1 扩容期间的性能问题

:::warning ⚠️
**典型事故**：某电商服务在促销高峰期大量请求超时，排查发现是 ConcurrentHashMap 扩容导致的请求阻塞。
:::

**问题分析**：

- ConcurrentHashMap 扩容期间，get 操作可以正常进行（读旧表或新表）
- 但 put 操作在发现 `MOVED` 标记后，必须等待协助扩容完成
- 如果扩容线程不足，请求会堆积

**排查路径**：

```bash
# 1. 查看线程状态
jstack <pid> | grep "ConcurrentHashMap"

# 2. 反解 sizeCtl 值
# 如果是负数，说明正在扩容
# 例如 sizeCtl = -98278145
# 98278145 >>> 16 = 1499，说明有 1499 个线程在协助扩容

# 3. Arthas 监控
dashboard -n 5  # 查看线程面板
trace com.xx.ConcurrentHashMap *  # 跟踪方法调用
```

### 4.2 初始化并发问题

```java
private final Node<K,V>[] initTable() {
    Node<K,V>[] tab; int sc;
    while ((tab = table) == null || tab.length == 0) {
        if ((sc = sizeCtl) < 0)
            Thread.yield();  // 让出 CPU，等待其他线程初始化
        else if (U.compareAndSwapInt(this, SIZECTL, sc, -1)) {
            // 只有一个线程能进入这里初始化
            // ...
        }
    }
    return tab;
}
```

:::warning ⚠️
初始化时如果多个线程同时竞争，会使用 `Thread.yield()` 让出 CPU，可能导致**性能抖动**。这是很多候选人忽略的点。
:::

### 4.3 扩容与树化的临界点

```java
// 数组大小 < 64 时，优先扩容
if (tab == null || (n = tab.length) < MIN_TREEIFY_CAPACITY)
    tryPresize(n << 1);  // 扩容到 2 倍
```

**生产建议**：

- 如果预知数据量大，初始化时设置足够的容量
- 避免频繁的小容量扩容（每次扩容有复制开销）

## 五、工程选型

### 5.1 什么场景用什么方案

| 场景 | 推荐方案 | 原因 |
|------|----------|------|
| 单线程本地缓存 | `HashMap` | 无锁开销，性能最高 |
| 低并发（< 100线程） | `Collections.synchronizedMap` | 简单，无额外学习成本 |
| 高并发读多写少 | `ConcurrentHashMap` | 分段锁/CAS，读不加锁 |
| 高并发读多写多 | `ConcurrentHashMap` + 预分区 | 减少热点竞争 |
| 需要保序 | `ConcurrentSkipListMap` | 基于跳表，有序 |

### 5.2 如何说服面试官

当你被问到"为什么选 ConcurrentHashMap"时，可以这样回答：

> "ConcurrentHashMap 在 JDK8 采用了 CAS + synchronized 的组合，比 JDK7 的 Segment 分段锁更高效。主要优势有三点：第一，内存开销更小，不需要为每个 Segment 创建独立对象；第二，并发度更高，任何线程都可以协助迁移任何桶；第三，实现更简单，维护成本低。"
