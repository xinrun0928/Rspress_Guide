# Java 集合框架

候选人小李坐在字节跳动的面试间，面试官看了一眼简历，开口问：

"HashMap 的 put 流程说一下。"

小李背出了四步：计算 hash、找到桶、插入链表/红黑树、判断扩容。面试官点点头，继续追问：

"JDK8 为什么引入红黑树？链表达到多少长度会树化？"

小李说："好像是 8？"

面试官没说话，又问："什么时候树化，什么时候退链表？"

小李停顿了三秒，开始语无伦次。

---

这道题，90% 的候选人会死在第三层追问上。不是因为 HashMap 难，而是因为网上资料太乱、太多人只背流程不读源码。

这一模块，带你把 Java 集合从"会用"升级到"能讲清楚底层设计与演进逻辑"。

## 学习路径指引

### P5 候选人（校招/初级社招）

集合框架是 Java 面试的重灾区，因为太多人把它当成"背了就能答"的知识点。

**P5 必过关卡**：

1. ArrayList 的 add() 流程是什么？扩容机制是什么？默认容量是多少？
2. HashMap 的 put() 流程是什么？JDK7 和 JDK8 有什么区别？
3. HashMap 的容量为什么是 2 的幂次？hash 算法是怎么设计的？
4. ConcurrentHashMap 为什么比 Hashtable 性能好？JDK7 和 JDK8 有什么区别？

这四道题是基本盘。答不清楚的，面试直接挂一半。

### P6 候选人（中级社招）

P6 的区分点在**扩容机制和树化/退化条件**——这两点是网上资料最乱、最容易被忽略的。

**拉开差距的追问链**：

```
HashMap put 流程
  → 为什么容量必须是 2 的幂次？（hash & (length-1) 代替取模）
    → 那 hashCode 已经是 int 了，为什么还要再扰动一次？
      → 扩容时 rehash 的过程是什么？JDK7 和 JDK8 的扩容有什么区别？
        → JDK7 在并发扩容时有什么问题？（环形链表、死循环）
          → JDK8 怎么解决这个问题的？
```

这整条追问链能走下来的，基本都是看过源码的。

**红黑树追问链**：

```
为什么引入红黑树？
  → 链表长度达到多少会树化？（8）
    → 为什么是 8 不是 16？（泊松分布、空间时间权衡）
      → 红黑树什么时候退化成链表？（6）
        → 为什么是 6 不是 8？（避免频繁树化退化）
```

这条链也是 P5/P6 分水岭。能答到最后的，基本都能拿到 offer。

### P7 候选人（高级社招/架构方向）

P7 的集合框架不是在考实现细节，而是在考**方案选型和性能调优**。

**典型的 P7 级问题**：

- 如果你需要存 100 万个 key-value，HashMap 初始容量设多少最合适？为什么？
- HashMap 的 loadFactor 为什么是 0.75？这个数字是怎么来的？
- 什么场景下 HashMap 不是最优解？有没有替代方案？
- ConcurrentHashMap 在高并发写入时的性能瓶颈在哪里？如何排查？

【面试官心理】
我出 HashMap 这道题，不是在考背书。我想知道的是：他有没有亲手看过源码，能不能理解 JDK8 为什么要引入红黑树，链表阈值 8 和退化条件 6 的设计原因是什么。知道结论的人很多，能解释为什么的才是真正理解的。

## 生产避坑指南

集合框架的生产事故，往往来得很突然：

- **HashMap 死循环**：JDK7 并发扩容时，链表头插法导致环形链表，get() 永远找不到数据或死循环。JDK8 用尾插法修复了这个问题。
- **ArrayList 扩容 OOM**：循环内不断 add()，没有预设容量，扩容到一定规模直接 OOM。
- **HashMap 预热不足**：根据数据量预设容量，避免频繁扩容触发的 stop-the-world。
- **ConcurrentHashMap 误用**：在需要强一致性的场景用 ConcurrentHashMap，但实际上单个操作也不是原子的。

:::warning ⚠️
HashMap 的死循环问题在 JDK8 里修好了，但很多人还在 JDK7 的认知里。另外，ConcurrentHashMap 的单操作不是原子的——它保证的是分段 CAS 原子性，不是整个操作的事务性。
:::

:::tip 💡
生产上如果预估数据量是 N，推荐的初始容量是 `N / 0.75 + 1`，这样可以避免自动扩容带来的性能损耗。
:::
