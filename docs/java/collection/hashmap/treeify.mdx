# 树化条件：链表长度 > 8 且容量 >= 64

JDK 8 引入了红黑树优化 HashMap 的哈希冲突。但为什么是「链表长度 > 8」？为什么还要「容量 >= 64」？

这不是拍脑袋的决定，而是基于**泊松分布的概率分析**和**性能权衡**。

## 问题背景

当多个 Key 哈希到同一个桶时，HashMap 会用链表存储冲突的元素。

```
┌────────────────────────────────────────────────────┐
│                    链表结构                          │
├────────────────────────────────────────────────────┤
│   hashCode % n = 3 的桶：                           │
│                                                    │
│   ┌─────┬─────┬─────┬─────┬─────┐                  │
│   │Node1│ ──> │Node2│ ──> │Node3│ ──> ...          │
│   └─────┴─────┴─────┴─────┴─────┘                  │
│                                                    │
│   查找时间：O(k)，k = 链表长度                       │
└────────────────────────────────────────────────────┘
```

当链表过长时，查找性能急剧下降。JDK 8 的解决方案是：**将链表转换为红黑树**，查找时间从 O(k) 降为 O(log k)。

## 树化条件

```java
// JDK 8 HashMap 源码
static final int TREEIFY_THRESHOLD = 8;   // 链表转红黑树的阈值
static final int UNTREEIFY_THRESHOLD = 6; // 红黑树转链表的阈值
static final int MIN_TREEIFY_CAPACITY = 64; // 最小树化容量

final void treeifyBin(Node<K,V>[] tab, int hash) {
    int n, index;
    Node<K,V> e;

    if (tab == null || (n = tab.length) < MIN_TREEIFY_CAPACITY)
        resize();  // 容量 < 64，先扩容，不树化
    else if ((e = tab[index = (n - 1) & hash]) != null) {
        if (e.hash == hash &&
            (k = e.key) == key || (key != null && key.equals(k)))) {
            // 链表转红黑树
            // ...
        }
    }
}
```

两个条件必须同时满足：

1. **链表长度 > 8**（实际上是 `>= 7` 时触发）
2. **数组容量 >= 64**

## 为什么是 8？

### 泊松分布分析

JDK 源码中有这样一段注释：

```java
/*
 * Tree bins are allocated primarily when there are hash collisions,
 * but in the absence of hash coding problems (which should be rare).
 * Despite collisions, the expected size of bins in a tree will
 * be relatively short (average less than 2) and are unlikely to
 * become depth-long. The variance for bin count is high, but the
 * probabilities of tree vs list for a bin of size k are:
 *
 * k=0:    (e^(-0.5) * 0.5^0) / 0! = 0.6065
 * k=1:    (e^(-0.5) * 0.5^1) / 1! = 0.3032
 * k=2:    (e^(-0.5) * 0.5^2) / 2! = 0.0758
 * k=3:    (e^(-0.5) * 0.5^3) / 3! = 0.0126
 * k=4:    (e^(-0.5) * 0.5^4) / 4! = 0.0015
 * k=5:    (e^(-.05) * 0.5^5) / 5! = 0.0001
 * ...
 * k=8:    about 0.0000016
 */
```

这里的 `0.5` 是负载因子为 `0.75` 时的泊松参数 `λ`。

计算链表长度为 8 的概率：

```java
// P(k) = (λ^k * e^(-λ)) / k!
// P(8) = (0.5^8 * e^(-0.5)) / 8!
//      = (0.0039 * 0.6065) / 40320
//      = 0.0000016
//      ≈ 百万分之一
```

**链表长度达到 8 的概率只有百万分之一。** 正常情况下，HashMap 不应该出现这么长的链表。

### 8 vs 6：为什么不是 6？

树化的阈值是 8，但退树化的阈值是 6。

这是一个**避免频繁切换**的设计：

```
链表长度增长：1 → 2 → ... → 6 → 7 → 8（树化）
链表长度减少：8 → 7 → 6（退树化）
```

如果两个阈值都是 8，那么在链表长度来回波动时，会频繁触发树化和退树化，性能开销很大。

设置为 6 和 8，形成了一个「缓冲带」，避免抖动：

```
[1, 2, 3, 4, 5, 6]   → 链表
[7, 8]               → 可能树化
[回到 6]             → 可能退树化
```

## 为什么容量必须 >= 64？

### 避免过早树化

如果容量很小（比如 16），扩容阈值只有 `16 * 0.75 = 12`。

这时候如果发生哈希碰撞，链表长度达到 8 就触发树化，但数组还有很多空位。

更好的策略是：**先扩容，把元素分散开，而不是急着树化**。

```java
// 容量 16，负载因子 0.75
// 扩容阈值 = 12

// 情况 A：容量足够，扩容分散
// 扩容后容量 = 32，可以容纳更多元素

// 情况 B：容量过小就树化
// 16 个桶，每个桶都很稀疏，树化意义不大
```

### 树化是有成本的

红黑树虽然查找快，但也有额外开销：

1. **空间开销**：每个节点需要存储额外的颜色、左右指针
2. **维护成本**：插入/删除需要旋转和着色操作
3. **常数因子**：小数据量下，红黑树不如链表快

当容量很小时，链表本身就很短（平均不到 1 个元素），树化的收益几乎为零，反而增加了开销。

### 数据验证

```java
// 容量 = 64，负载因子 = 0.75
// 扩容阈值 = 48
// 平均每个桶的元素数 = 元素总数 / 64

// 如果元素总数 = 48（扩容前）
// 平均每个桶 = 48 / 64 = 0.75
// 链表长度超过 8 的概率 ≈ 0.0000016

// 如果元素总数 = 32（正常情况）
// 平均每个桶 = 32 / 64 = 0.5
// 链表长度超过 8 的概率更低
```

在 64 容量的 HashMap 中，只有在极端哈希碰撞下，链表才可能达到长度 8。

## 完整的树化流程

```java
public class HashMapDemo {

    public static void main(String[] args) {
        HashMap<String, Integer> map = new HashMap<>();

        // 模拟哈希碰撞：故意让多个元素 hash 到同一个桶
        // 这里用反射或自定义哈希来模拟
        for (int i = 0; i < 20; i++) {
            map.put("key" + i, i);
        }

        // 正常情况下，HashMap 不会触发树化
        // 需要特定的哈希碰撞场景才会触发
        System.out.println("Map size: " + map.size());
    }

    // 触发树化的条件：
    // 1. 同一个桶内有 >= 8 个元素
    // 2. HashMap 容量 >= 64
    // 3. 通常需要精心构造的哈希碰撞攻击
}
```

## 容量 < 64 时的行为

```java
// 当链表长度 > 8，但容量 < 64 时
// HashMap 会选择扩容，而不是树化

HashMap<String, Integer> map = new HashMap<>(32);  // 初始容量 32

// 往同一个桶里塞大量元素（通过哈希碰撞）
// 当链表长度达到 8，但容量仍 < 64
// 触发 resize()，容量翻倍（32 -> 64）
// 重新分布后，链表长度变短，避免了树化
```

## 性能对比

| 链表长度 | 查找复杂度 | 红黑树查找 | 链表遍历 |
|---|---|---|---|
| 1 | O(1) | - | O(1) |
| 4 | O(4) | O(log 4) = O(2) | O(4) |
| 8 | O(8) | O(log 8) = O(3) | O(8) |
| 16 | O(16) | O(log 16) = O(4) | O(16) |
| 100 | O(100) | O(log 100) = O(7) | O(100) |

链表长度超过 8 时，红黑树的 O(log n) 明显优于链表的 O(n)。

## 面试追问

### Q1：为什么 JDK 7 用的是头插法，而 JDK 8 用尾插法？

JDK 7 的头插法在并发扩容时可能导致环形链表，造成死循环。JDK 8 改用尾插法，避免了这个问题。

### Q2：HashMap 的红黑树是什么版本？

HashMap 使用的是标准的红黑树实现，节点类型是 `TreeNode`，继承自 `LinkedHashMap.Entry`。

### Q3：能通过构造哈希碰撞来攻击 HashMap 吗？

可以，这叫 **HashDoS 攻击**。JDK 8 通过引入红黑树大幅提升了抗攻击能力，但极端情况下仍可能影响性能。Servlet 容器通常会限制请求参数数量来防护。

## 总结

HashMap 的树化条件是经过精心设计的：

1. **链表长度 > 8**：基于泊松分布，此时概率只有百万分之一
2. **容量 >= 64**：避免过早树化，让扩容优先
3. **退树化阈值 = 6**：与树化阈值 8 形成缓冲带，避免频繁切换
4. **工程权衡**：树化有成本，只有在链表确实过长时才有收益

理解了这些设计决策，你才能真正掌握 HashMap 的内部机制。
