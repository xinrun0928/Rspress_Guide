# 树化条件：链表长度 > 8 且容量 >= 64

HashMap 在 JDK 8 中引入红黑树来解决长链表查询性能退化的问题。但红黑树并非在链表一长就被触发——它需要满足两个条件：链表长度超过 8，且数组容量至少达到 64。为什么是这两个数字？它们背后有着怎样的概率分析和工程权衡？

## 树化的触发条件

在 JDK 8 的 HashMap 中，树化由 `treeifyBin()` 方法处理：

```java
static final int TREEIFY_THRESHOLD = 8;
static final int MIN_TREEIFY_CAPACITY = 64;

final void treeifyBin(Node<K,V>[] tab, int hash) {
    int n, index; Node<K,V> e;
    if (tab == null || (n = tab.length) < MIN_TREEIFY_CAPACITY)
        resize();
    else if ((e = tab[index = (n - 1) & hash]) != null) {
        TreeNode<K,V> hd = null, tl = null;
        do {
            TreeNode<K,V> p = replacementTreeNode(e, null);
            if (tl == null)
                hd = p;
            else {
                p.prev = tl;
                tl.next = p;
            }
            tl = p;
        } while ((e = e.next) != null);
        if ((tab[index] = hd) != null)
            hd.treeify(tab);
    }
}
```

树化需要同时满足两个条件：

1. 链表长度 >= TREEIFY_THRESHOLD（8）
2. 数组容量 >= MIN_TREEIFY_CAPACITY（64）

如果容量小于 64，HashMap 会选择扩容而非树化。这是因为在小容量情况下，扩容的成本比树化更低，而且扩容能够直接减少冲突。

## 为什么是 8：泊松分布的概率分析

JDK 8 的作者选择 8 作为树化阈值，是经过严格的概率分析的。在负载因子 0.75 的条件下，每个桶中元素的数量服从泊松分布。

泊松分布的概率质量函数是：

```
P(X = k) = (λ^k * e^(-λ)) / k!
```

其中 λ 是平均每个桶的元素数量，在负载因子 0.75 下，λ = 0.75。

根据这个公式，我们可以计算一个桶中出现 k 个元素的概率：

```
k=0: 0.4721 (47.2%)
k=1: 0.3541 (35.4%)
k=2: 0.1329 (13.3%)
k=3: 0.0332 (3.3%)
k=4: 0.0062 (0.6%)
k=5: 0.0009 (0.09%)
k=6: 0.0001 (0.01%)
k=7: 0.00001 (0.001%)
k=8: 0.00000006 (十亿分之六)
k>=9: 更低
```

从概率分布可以看出，当链表长度达到 8 时，概率已经低到几乎不可能发生（十亿分之六）。这意味着，在正常的非恶意输入下，一个桶中出现 8 个元素是非常罕见的情况。

## 泊松分布在 JDK 8 源码中的体现

JDK 8 的源码注释中直接写出了泊松分布的概率数据，这可能是 HashMap 文档中最详细的数学分析了：

```
* Ideally, under random hashCodes, the frequency of
* nodes in bins follows a Poisson distribution
* (http://en.wikipedia.org/wiki/Poisson_distribution) with a
* parameter of about 0.5 on average for the default resizing
* threshold of 0.75, although with a large variance because of
* resizing granularity. Ignoring variance, the expected
* occurrences of list size k are (exp(-0.5) * pow(0.5, k) /
* factorial(k)). The first values are:
*
* 0:    0.60681566
* 1:    0.30340783
* 2:    0.07585196
* 3:    0.01263683
* 4:    0.00157952
* 5:    0.00015795
* 6:    0.00001316
* 7:    0.00000094
* 8:    0.00000006
```

虽然注释中的 λ 参数（0.5）与实际负载因子（0.75）对应的 λ（0.75）略有不同，但数量级的结论是一致的：链表长度达到 8 的概率极低。

## 为什么不是更低的阈值

既然链表长了对查询性能有影响，为什么不把阈值设为 4 或 5？

答案是：**权衡**。红黑树虽然查询性能好，但插入和删除时需要维护树的平衡（旋转和着色），这些操作本身有开销。对于长度较短的链表，遍历比较的代价并不高。

具体来说：

- 链表长度 1-7：直接遍历，最坏 7 次比较
- 链表长度 8：需要 8 次比较
- 红黑树：O(log n)，对于 8 个节点最多 3 次比较

如果阈值太低，会导致不必要的树化，增加插入和删除的成本。只有当链表真正"太长"时，树化的收益才大于成本。

JDK 的选择是经过大量实验和调优的结果，8 这个阈值在大多数场景下表现最优。

## 为什么容量阈值是 64

第二个条件是容量必须达到 64 才会树化。为什么需要这个额外的限制？

**原因一：小容量时扩容更划算**

当 HashMap 容量较小（如 16、32）时，扩容的成本相对较低。扩容可以将所有元素重新分布到更大的数组中，从根本上减少冲突的可能性。

例如，假设一个桶中有 8 个元素，容量是 16。如果触发树化，每个节点的树操作都需要更多的内存和计算。但如果选择扩容到 32，容量翻倍，负载因子下降，大多数桶会变空，查询性能自然提升。

**原因二：避免频繁树化-拆分循环**

如果容量小于 64 时允许树化，而删除元素后又可能拆分回链表，会导致频繁的树化和拆分，增加不必要的开销。

**原因三：泊松分布的概率保证**

在正常情况下（无哈希攻击），当数组容量较小时，由于负载因子的控制，长链表的概率极低。只有当数组容量较大（>= 64）时，才更可能出现真正的长链表。

## 反向树化：链表拆分阈值 6

与树化对应的，有一个反向过程：拆分回链表。JDK 8 使用 UNTREEIFY_THRESHOLD = 6 作为拆分阈值：

```java
static final int UNTREEIFY_THRESHOLD = 6;
```

当红黑树中的节点被删除到只剩下 6 个或更少时，会转换回链表。这个阈值比树化阈值低 2，形成了一个"迟滞"区域（hysteresis region）。

```
链表长度:  6 ----> 7 ----> 8 ----> ...
          |      |      |
          |      |      +----> 树化
          |      |
          |      +----> 可能树化（binCount >= 7）
          |
          +----> 拆分回链表
```

设置迟滞区域的原因是避免在临界点反复树化和拆分。例如，如果阈值的上下限相同（都是 8），那么插入第 9 个元素时树化，删除第 8 个元素时拆分回链表，再插入又树化...这种反复会影响性能。

## 哈希攻击与树化

红黑树机制还有一个重要意义：防止哈希碰撞攻击（Hash Collision Attack）。

恶意攻击者可以通过精心设计大量哈希值相同的字符串来触发 HashMap 的最坏情况性能。如果 HashMap 只使用链表，攻击者可以使所有查询退化为 O(n)，造成服务拒绝攻击（DoS）。

有了红黑树机制，即使面对恶意输入，HashMap 也能将最坏情况从 O(n) 降到 O(log n)，大大提高了系统的鲁棒性。

理解树化条件的设计，你不仅理解了 HashMap 的一个重要特性，也理解了一个经典的工程权衡案例：没有绝对的最优，只有特定场景下的最优。
