# HashMap 家族全景图

HashMap 是 Java 集合框架中使用频率最高的数据结构之一。从 JDK 1.2 诞生至今，它经历了多次重大演进。理解 HashMap 的前世今生，不仅能帮助我们更好地使用它，还能在面试中展现深厚的技术功底。

## HashMap 的诞生背景

在 JDK 1.2 之前，Java 只有 Dictionary 类来存储键值对。Dictionary 是一个抽象类，提供了基于键的存储和检索功能。但是，它的方法都是同步的，在单线程环境下性能很差。

HashMap 的出现解决了这个问题。它继承自 AbstractMap，实现了 Map 接口，提供了非同步、高性能的键值对存储能力。从诞生的那一刻起，HashMap 就以其卓越的性能成为 Java 开发者的首选。

## 演进历程：从简单到复杂

HashMap 的发展历程可以分为三个阶段：

**JDK 1.2 - JDK 7**：数组 + 链表 + 头插法时代。这一时期的 HashMap 结构相对简单，采用数组加单向链表存储冲突的元素，使用头插法插入新元素。这个设计在单线程环境下工作良好，但在并发环境下暴露出了严重的问题——扩容时可能导致环形链表，引发死循环。

**JDK 8**：数组 + 链表/红黑树 + 尾插法时代。JDK 8 对 HashMap 进行了重大改进：引入红黑树优化长链表查询，使用尾插法避免扩容死循环，同时优化了扰动函数使哈希分布更加均匀。这一版本的 HashMap 更加健壮和高效。

**JDK 11+**：继续优化与完善。后续版本主要进行性能优化和 bug 修复，包括对 forEach 方法的优化、键冲突处理的改进等。

## 核心数据结构

无论哪个版本，HashMap 的核心都是"数组 + 链表/红黑树"的结构。数组提供 O(1) 的随机访问能力，链表/红黑树解决哈希冲突问题。

```java
transient Node<K,V>[] table;
transient int size;
transient int modCount;
int threshold;
final float loadFactor;
```

- `table`：存储元素的数组，每个位置是一个桶
- `size`：实际存储的键值对数量
- `modCount`：结构性修改的次数，用于 fail-fast 机制
- `threshold`：扩容阈值，等于 `capacity * loadFactor`
- `loadFactor`：负载因子，默认 0.75

## 重要概念解析

理解 HashMap 需要掌握几个核心概念：

**哈希冲突**：当两个不同的键计算出相同的哈希值时，就会发生哈希冲突。HashMap 通过链表或红黑树来解决冲突。

**寻址公式**：`index = (n - 1) & hash`。由于 HashMap 的容量始终是 2 的幂次方，这个位运算等价于 `hash % n`，但速度更快。

**扰动函数**：`hash = (h = key.hashCode()) ^ (h >>> 16)`。它将高位信息混合到低位，使哈希分布更加均匀。

**负载因子**：控制扩容时机的关键参数。默认 0.75 意味着当 75% 的桶被占用时触发扩容。

**树化**：当链表长度超过 8 且数组容量达到 64 时，链表会转换成红黑树，以提升查询性能。

## 各版本对比

| 特性 | JDK 7 | JDK 8+ |
|------|-------|--------|
| 数据结构 | 数组 + 链表 | 数组 + 链表/红黑树 |
| 插入方式 | 头插法 | 尾插法 |
| 扩容死循环 | 可能 | 不可能 |
| 扰动函数 | 4次位运算+异或 | 1次异或 |
| 查询复杂度 | O(n) 最坏 | O(log n) 最坏 |

## 为什么需要了解 HashMap 内部原理

很多开发者觉得 HashMap 使用简单，`put` 和 `get` 就够了，何必了解内部原理？但在实际开发中，理解 HashMap 的原理能帮助你：

1. 写出更高性能的代码：避免哈希冲突，选择合适的初始容量
2. 排查生产问题：OOM、死循环等问题的根本原因分析
3. 面试必备：HashMap 几乎是 Java 面试的必考题

HashMap 不仅仅是一个键值对存储工具，它的设计蕴含着丰富的数据结构知识和工程实践经验。
