# JDK 7 扩容死循环问题

JDK 7 的 HashMap 在并发环境下进行扩容操作时，可能导致环形链表，进而引发死循环。这个问题被称为"HashMap 扩容死循环"，是 Java 并发领域的经典问题。虽然 JDK 8 已经解决了这个问题，但理解其根本原因对于深入理解并发编程和 HashMap 设计至关重要。

## 问题根源：头插法与扩容重分布

JDK 7 HashMap 使用头插法插入新元素，同时在扩容时使用头插法重新分配元素。这个组合在并发环境下是致命的。

先回顾 JDK 7 的扩容代码：

```java
void transfer(Entry[] newTable, boolean rehash) {
    int newCapacity = newTable.length;
    for (Entry<K,V> e : oldTable) {
        while(null != e) {
            Entry<K,V> next = e.next;
            if (rehash) {
                e.hash = null == e.key ? 0 : hash(e.key);
            }
            int i = indexFor(e.hash, newCapacity);
            e.next = newTable[i];
            newTable[i] = e;
            e = next;
        }
    }
}
```

关键在于 `e.next = newTable[i]; newTable[i] = e;` 这两行代码，它们实现了头插法：将新节点插入到链表头部，新节点的 next 指向原来的头节点。

## 单线程扩容：链表反转

在单线程环境下，头插法会导致链表在扩容后反转。假设原链表的顺序是 A -> B -> C（按插入顺序），扩容过程如下：

1. e = A，next = B
   - e.next = null（newTable[i] 为空）
   - newTable[i] = A
   - e = B

2. e = B，next = C
   - e.next = A（原来的 newTable[i]）
   - newTable[i] = B
   - e = C

3. e = C，next = null
   - e.next = B（原来的 newTable[i]）
   - newTable[i] = C
   - e = null

扩容后，链表顺序变成了 C -> B -> A，顺序完全反转。这本身不是问题，但在并发环境下，这个过程会变得混乱。

## 并发场景：环形链表的诞生

假设两个线程 A 和 B 同时对 HashMap 进行扩容。线程 A 和线程 B 都持有相同的 oldTable 引用，都看到了相同的初始状态。

**初始状态**：
- oldTable 中有一个桶，链表是 A -> B -> C
- A.next = B，B.next = C，C.next = null

**线程 A 的执行过程**：

1. 线程 A 遍历链表，e = A
2. 线程 A 计算 A 的新位置
3. 线程 A 执行 `e.next = newTable[i]`（newTable[i] = null）
4. 线程 A 执行 `newTable[i] = A`
5. 此时：newTable[i] = A，A.next = null
6. e = B

**线程 A 被挂起，线程 B 开始执行**：

1. 线程 B 遍历链表，e = A（与线程 A 看到的是同一个 A）
2. 线程 B 计算 A 的新位置（与线程 A 相同，因为 hash 没变）
3. 线程 B 执行 `e.next = newTable[i]`（newTable[i] 现在是 A）
4. 此时：A.next = A（自己指向自己！）
5. 线程 B 执行 `newTable[i] = A`（没有变化）
6. e = B

**线程 B 继续执行**：
7. 线程 B 遍历，e = B
8. 线程 B 执行 `e.next = newTable[i]`（newTable[i] 是 A）
9. 此时：B.next = A
10. 线程 B 执行 `newTable[i] = B`
11. 此时：newTable[i] = B，B.next = A，A.next = A
12. e = C

**线程 B 继续执行**：
13. 线程 B 遍历，e = C
14. 线程 B 执行 `e.next = newTable[i]`（newTable[i] 是 B）
15. 此时：C.next = B
16. 线程 B 执行 `newTable[i] = C`
17. 此时：newTable[i] = C，C.next = B，B.next = A，A.next = A
18. e = null

**线程 A 恢复执行**：

但线程 A 之前已经将 A.next 设为了 null，现在它继续处理 B：
19. 线程 A 继续，e = B（还记得 B 吗？但 B.next 已经被线程 B 改成了 A）
20. 线程 A 执行 `e.next = newTable[i]`（newTable[i] 是 B）
21. B.next = B（自己指向自己！）
22. newTable[i] = B

结果是：newTable[i] = B，B.next = B（自环）。

## 死循环的触发

当有线程尝试从 HashMap 中获取数据时：

```java
public V get(Object key) {
    Entry<K,V> e = null;
    int i = indexFor(hash(key), table.length);
    int n = i;
    Entry<K,V> first = table[i];
    for (e = first; e != null; e = e.next) {
        // 遍历链表查找
    }
}
```

如果链表形成了环形（B -> B），遍历永远不会结束——for 循环中的 `e != null` 永远为真，`e = e.next` 永远在原地打转。

这会导致 CPU 100%，线程陷入死循环，无法响应任何请求。

## 详细的并发时序分析

让我们更精确地分析一个典型的死循环场景：

**T1 时刻**：线程 A 开始扩容，正在处理 A 节点：
- `Entry<K,V> next = e.next` 执行完毕，next = B
- `e.next = newTable[i]` 执行完毕（newTable[i] 为 null，所以 A.next = null）
- 线程 A 被 OS 挂起

**T2 时刻**：线程 B 开始扩容：
- 线程 B 看到 oldTable 仍然是 A -> B -> C
- 线程 B 处理 A：`A.next = null`（newTable[i] 仍为 null）
- 线程 B 处理 B：`B.next = A`
- 线程 B 处理 C：`C.next = B`
- 扩容完成，线程 B 退出

**T3 时刻**：线程 A 恢复，继续扩容：
- 线程 A 从 B 继续（之前 next = B）
- `Entry<K,V> next = e.next` 执行完毕，next = A（注意！此时 B.next 已经被线程 B 改成了 A）
- `e.next = newTable[i]` 执行，B.next = B（newTable[i] 是 B）

此时，B.next = B，环形链表形成。

## JDK 8 的解决方案

JDK 8 通过两个关键改变彻底解决了这个问题：

**改变一：尾插法**

JDK 8 不再在扩容时反转链表顺序，而是保持原顺序：

```java
Node<K,V> loHead = null, loTail = null;
Node<K,V> hiHead = null, hiTail = null;
do {
    next = e.next;
    if ((e.hash & oldCap) == 0) {
        if (loTail == null)
            loHead = e;
        else
            loTail.next = e;
        loTail = e;
    } else {
        if (hiTail == null)
            hiHead = e;
        else
            hiTail.next = e;
        hiTail = e;
    }
} while ((e = next) != null);
if (loTail != null) {
    loTail.next = null;
    newTab[j] = loHead;
}
```

新节点被追加到链表尾部，保持了元素的原始顺序。

**改变二：元素分组**

JDK 8 将元素分成两组（低位和高位），每组内部保持顺序。即使在并发环境下，两个线程可能处理不同的元素组，不会相互干扰。

更重要的是，由于顺序保持不变，即使并发执行，也不会形成环形引用。

## 真实案例：JDK 7 HashMap 导致的线上事故

HashMap 扩容死循环问题在生产环境中并不罕见。典型的场景是：

1. 使用 HashMap 在多线程环境下作为缓存
2. 高并发写入触发扩容
3. 扩容过程中线程被调度
4. 某个线程进入死循环
5. CPU 100%，服务不可用

这个问题的严重性使得 JDK 8 的改动成为必然。同时，它也提醒我们：在并发环境下，不要使用非线程安全的集合类。

## 正确的并发替代方案

如果需要在并发环境下使用键值对存储，应该使用：

- `ConcurrentHashMap`：专门为并发设计，分段锁/无锁机制
- `Collections.synchronizedMap()`：对 HashMap 加全局锁，性能较差
- `Hashtable`：历史遗留类，不推荐使用

理解 JDK 7 HashMap 的死循环问题，你不仅理解了一个经典的并发 bug，也理解了在并发编程中数据竞争（data race）和内存可见性（memory visibility）的重要性。
