# JDK 7：数组 + 链表 + 头插法

JDK 7 是 HashMap 发展历程中的重要版本。虽然它已经在 2020 年停止更新，但它采用的"数组 + 链表 + 头插法"设计揭示了 HashMap 设计的演进历程。更重要的是，理解 JDK 7 的问题能帮助我们更好地理解 JDK 8 的改进。

## JDK 7 HashMap 的核心结构

JDK 7 中 HashMap 的核心数据结构是 Entry 数组加单向链表：

```java
public class HashMap<K,V> extends AbstractMap<K,V> 
    implements Map<K,V>, Cloneable, Serializable {
    
    transient Entry<K,V>[] table;
    transient int size;
    transient int modCount;
    int threshold;
    final float loadFactor;
    
    static class Entry<K,V> implements Map.Entry<K,V> {
        final K key;
        V value;
        Entry<K,V> next;
        int hash;
        
        Entry(int h, K k, V v, Entry<K,V> n) {
            value = v;
            next = n;
            key = k;
            hash = h;
        }
        // ...
    }
}
```

每个 Entry 包含四个字段：`key` 存储键，`value` 存储值，`next` 指向链表中的下一个节点，`hash` 存储扰动后的哈希值。注意 `next` 是指向链表下一个节点的引用，这形成了一条单向链表。

## 扰动函数的演进

JDK 7 的扰动函数与 JDK 8 有所不同，经历了多次优化。早期版本使用：

```java
// JDK 7 早期版本
static int hash(int h) {
    h ^= (h >>> 20) ^ (h >>> 12);
    return h ^ (h >>> 7) ^ (h >>> 4);
}
```

这个扰动函数进行了 4 次位运算和异或操作。后来 JDK 7u40 版本简化为：

```java
// JDK 7u40 之后
static final int hash(Object key) {
    int h;
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}
```

这个版本与 JDK 8 一致了。位运算的减少提高了性能，同时保持了良好的哈希分布。

## 头插法：简洁但危险

JDK 7 插入新元素时使用的是头插法：

```java
void addEntry(int hash, K key, V value, int bucketIndex) {
    if ((size >= threshold) && (null != table[bucketIndex])) {
        resize(2 * table.length);
        hash = (null != key) ? hash(key) : 0;
        bucketIndex = indexFor(hash, table.length);
    }
    createEntry(hash, key, value, bucketIndex);
}

void createEntry(int hash, K key, V value, int bucketIndex) {
    Entry<K,V> e = table[bucketIndex];
    table[bucketIndex] = new Entry<>(hash, key, value, e);
    size++;
}
```

`createEntry` 方法揭示了头插法的精髓：`Entry<K,V> e = table[bucketIndex]` 先保存当前桶中的链表头，然后 `table[bucketIndex] = new Entry<>(hash, key, value, e)` 将新节点插入到链表头部，新节点的 `next` 指向原来的头节点。

头插法的优点是插入效率高，O(1) 时间复杂度，只需要修改一个引用。但它的缺点在并发环境下会暴露无遗。

## 扩容与元素重分布

当 HashMap 容量不足时，会触发扩容：

```java
void resize(int newCapacity) {
    Entry<K,V>[] oldTable = table;
    int oldCapacity = oldTable.length;
    
    if (oldCapacity == MAXIMUM_CAPACITY) {
        threshold = Integer.MAX_VALUE;
        return;
    }
    
    Entry<K,V>[] newTable = new Entry[newCapacity];
    transfer(newTable, initHashSeedAsNeeded(oldCapacity));
    table = newTable;
    threshold = (int)Math.min(newCapacity * loadFactor, MAXIMUM_CAPACITY + 1);
}

void transfer(Entry<K,V>[] newTable, boolean rehash) {
    int newCapacity = newTable.length;
    for (Entry<K,V> e : oldTable) {
        while(null != e) {
            Entry<K,V> next = e.next;
            if (rehash) {
                e.hash = null == e.key ? 0 : hash(e.key);
            }
            int i = indexFor(e.hash, newCapacity);
            e.next = newTable[i];
            newTable[i] = e;
            e = next;
        }
    }
}
```

扩容过程的核心是 `transfer` 方法。它遍历旧数组的每个桶，然后遍历桶中的链表，将每个元素重新计算位置并插入新数组。

**关键问题**：在单链表遍历过程中，JDK 7 使用头插法重新插入元素。这意味着链表中的元素顺序在扩容后会**反转**。

## 头插法导致的扩容死循环

为什么头插法会导致扩容死循环？让我们模拟一个场景：

假设 HashMap 中有三个元素 A、B、C，它们的哈希值相同，被放在同一个桶中，链表顺序是 A -> B -> C（先插入 A，再插入 B，最后插入 C）。

扩容时，容量翻倍，元素需要重新分配。假设 A、B、C 重新分配后仍然在同一个桶中。

在 `transfer` 过程中：

1. `e = A`，`next = B`
2. 计算 A 的新位置 i
3. `e.next = newTable[i]`（此时 newTable[i] 为 null）
4. `newTable[i] = A`
5. `e = B`

此时新链表中只有 A。继续：

6. `e = B`，`next = C`
7. 计算 B 的新位置 i
8. `e.next = newTable[i]`（此时 newTable[i] 是 A）
9. `newTable[i] = B`
10. `e = C`

现在新链表是 B -> A，顺序已经反转。继续：

11. `e = C`，`next = null`
12. 计算 C 的新位置 i
13. `e.next = newTable[i]`（此时 newTable[i] 是 B）
14. `newTable[i] = C`
15. `e = null`

最终新链表是 C -> B -> A，顺序完全反转。

**在并发环境下**，两个线程同时扩容会发生什么？

假设线程 A 执行到一半被挂起：
- oldTable 中链表是 A -> B -> C
- 线程 A 已经将 A 和 B 移动到新表
- 此时 newTable[i] = B, B.next = A, A.next = null
- 但 A.next 实际上仍然指向 B（因为还没来得及修改）

线程 B 开始扩容：
- 它看到的是 oldTable 中的原始链表 A -> B -> C
- 线程 B 正常遍历和转移元素
- 但由于某些元素已经被线程 A 部分修改...

这就是经典的 JDK 7 HashMap 并发扩容死循环问题。两条链表相互引用，形成环形链表，永远无法遍历结束。

## JDK 7 的历史贡献与局限

JDK 7 HashMap 的设计奠定了现代 HashMap 的基础。它的核心思想——数组加链表解决哈希冲突、扩容翻倍、扰动函数优化——都被 JDK 8 继承和发扬。

但 JDK 7 的头插法设计在并发场景下的致命缺陷，促使 JDK 8 做出了重大改进。理解 JDK 7，不仅是为了面试，更是为了理解技术演进背后的逻辑。
