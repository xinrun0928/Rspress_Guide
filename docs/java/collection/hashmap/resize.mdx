# HashMap 扩容时机与 resize 过程

HashMap 的扩容机制是其高性能的重要保障之一。通过动态调整容量，HashMap 能够在元素增加时保持较低的负载因子，从而维持高效的查找性能。但扩容也是一把双刃剑——它需要重新分配数组并迁移所有元素，是一项昂贵的操作。理解扩容的时机和过程，是掌握 HashMap 的关键。

## 扩容时机：threshold 的计算

HashMap 并非在装满时才扩容，而是在达到容量与负载因子的乘积时就开始扩容。threshold（阈值）的计算公式是：`capacity * loadFactor`。

以默认参数为例：初始容量 16，负载因子 0.75，初始阈值是 16 * 0.75 = 12。这意味着当你向 HashMap 添加第 13 个元素时，就会触发扩容。

```java
public HashMap(int initialCapacity, float loadFactor) {
    if (initialCapacity < 0)
        throw new IllegalArgumentException("Illegal initial capacity: " + initialCapacity);
    if (initialCapacity > MAXIMUM_CAPACITY)
        initialCapacity = MAXIMUM_CAPACITY;
    if (loadFactor <= 0 || Float.isNaN(loadFactor))
        throw new IllegalArgumentException("Illegal load factor: " + loadFactor);
    this.loadFactor = loadFactor;
    this.threshold = tableSizeFor(initialCapacity);
}

static final int tableSizeFor(int cap) {
    int n = -1 >>> Integer.numberOfLeadingZeros(cap - 1);
    return (n < 0) ? 1 : (n >= MAXIMUM_CAPACITY) ? MAXIMUM_CAPACITY : n + 1;
}
```

`tableSizeFor()` 方法确保初始容量是 2 的幂次方。例如，如果传入 10，会返回 16；传入 17，会返回 32。

## resize 方法全解析

`resize()` 方法承担了两个职责：初始化 table 数组和扩容（当 table 已存在时）：

```java
final Node<K,V>[] resize() {
    Node<K,V>[] oldTab = table;
    int oldCap = (oldTab == null) ? 0 : oldTab.length;
    int oldThr = threshold;
    int newCap, newThr = 0;
    
    if (oldCap > 0) {
        // 情况一：已达最大容量，不再扩容
        if (oldCap >= MAXIMUM_CAPACITY) {
            threshold = Integer.MAX_VALUE;
            return oldTab;
        }
        // 情况二：容量翻倍，阈值也翻倍
        else if ((newCap = oldCap << 1) < MAXIMUM_CAPACITY &&
                 oldCap >= DEFAULT_INITIAL_CAPACITY)
            newThr = oldThr << 1;
    }
    else if (oldThr > 0)
        // 情况三：初始化时，使用阈值作为容量
        newCap = oldThr;
    else {
        // 情况四：零参数构造，使用默认值
        newCap = DEFAULT_INITIAL_CAPACITY; // 16
        newThr = (int)(DEFAULT_LOAD_FACTOR * DEFAULT_INITIAL_CAPACITY); // 12
    }
    
    if (newThr == 0) {
        float ft = (float)newCap * loadFactor;
        newThr = (newCap < MAXIMUM_CAPACITY && ft < (float)MAXIMUM_CAPACITY ?
                  (int)ft : Integer.MAX_VALUE);
    }
    threshold = newThr;
    
    @SuppressWarnings({"rawtypes","unchecked"})
    Node<K,V>[] newTab = (Node<K,V>[]) new Node[newCap];
    table = newTab;
    
    if (oldTab != null) {
        // 核心：迁移所有元素到新数组
        for (int j = 0; j < oldCap; ++j) {
            Node<K,V> e;
            if ((e = oldTab[j]) != null) {
                oldTab[j] = null;
                if (e.next == null)
                    // 单节点，直接移动
                    newTab[e.hash & (newCap - 1)] = e;
                else if (e instanceof TreeNode)
                    // 红黑树，特殊处理
                    ((TreeNode<K,V>)e).split(this, newTab, j, oldCap);
                else {
                    // 链表，分成两部分
                    Node<K,V> loHead = null, loTail = null;
                    Node<K,V> hiHead = null, hiTail = null;
                    Node<K,V> next;
                    do {
                        next = e.next;
                        // 关键判断：hash & oldCap
                        if ((e.hash & oldCap) == 0) {
                            if (loTail == null)
                                loHead = e;
                            else
                                loTail.next = e;
                            loTail = e;
                        } else {
                            if (hiTail == null)
                                hiHead = e;
                            else
                                hiTail.next = e;
                            hiTail = e;
                        }
                    } while ((e = next) != null);
                    if (loTail != null) {
                        loTail.next = null;
                        newTab[j] = loHead;
                    }
                    if (hiTail != null) {
                        hiTail.next = null;
                        newTab[j + oldCap] = hiHead;
                    }
                }
            }
        }
    }
    return newTab;
}
```

## 扩容的四种情况

`resize()` 需要处理四种不同的情况：

**情况一**：oldCap `>=` MAXIMUM_CAPACITY（已达最大容量 2^30）。不再扩容，只将阈值设为 Integer.MAX_VALUE。这个条件实际上永远不会在扩容时触发，因为容量最大只能是 2^30，而 `newCap < MAXIMUM_CAPACITY` 会阻止继续扩容。

**情况二**：oldCap > 0，容量翻倍。新容量是 oldCap 的两倍，只要不超过最大容量且原容量不少于默认初始容量（16），阈值也翻倍。

**情况三**：oldThr > 0，使用阈值作为新容量。这发生在使用带参数的构造方法时，例如 `new HashMap<>(32)`。此时 threshold 已经被 `tableSizeFor()` 计算好了，直接用它作为容量。

**情况四**：零参数构造，使用默认值。newCap = 16，newThr = 12。

## 链表拆分：高位链表与低位链表

扩容时最精彩的部分是链表的拆分逻辑。当 oldCap 翻倍变成 newCap 时，元素的位置有两种可能：

- **低位链表（lo）**：保持在原位置 j
- **高位链表（hi）**：移动到新位置 j + oldCap

判断依据是 `(e.hash & oldCap) == 0`。为什么这个位运算能决定位置？

oldCap 是 2 的幂次方，例如 16 的二进制是 `10000`。当 oldCap = 16 时，`e.hash & 16` 只检查 hash 值的第 5 位（从右往左数，从 0 开始）。如果第 5 位是 0，元素留在原位；如果是 1，元素移动到新位。

考虑一个具体例子：假设 oldCap = 16（`10000`），newCap = 32（`100000`）。

- 元素 A 的 hash = `01101`（第 5 位是 0）
  - `01101 & 10000 = 0`，留在原位置 13（`01101`）
- 元素 B 的 hash = `11101`（第 5 位是 1）
  - `11101 & 10000 = 10000`，移动到新位置 29（`11101`）

原位置 13 与新位置 29 的关系是 `13 + 16 = 29`，正是 oldCap 的偏移量。

## 红黑树的拆分

红黑树的拆分逻辑与链表类似，但更复杂：

```java
final void split(HashMap<K,V> map, Node<K,V>[] tab, int index, int bit) {
    TreeNode<K,V> b = this;
    TreeNode<K,V> loHead = null, loTail = null;
    TreeNode<K,V> hiHead = null, hiTail = null;
    int lc = 0, hc = 0;
    for (TreeNode<K,V> e = b.next; e != null; e = e.next) {
        if ((e.hash & bit) == 0) {
            if ((e.prev = loTail) == null)
                loHead = e;
            else
                loTail.next = e;
            loTail = e;
            ++lc;
        } else {
            if ((e.prev = hiTail) == null)
                hiHead = e;
            else
                hiTail.next = e;
            hiTail = e;
            ++hc;
        }
    }
    if (loHead != null) {
        if (lc <= UNTREEIFY_THRESHOLD)
            tab[index] = loHead.untreeify(map);
        else {
            tab[index] = loHead;
            if (hiHead != null)
                loHead.treeify(tab);
        }
    }
    if (hiHead != null) {
        if (hc <= UNTREEIFY_THRESHOLD)
            tab[index + bit] = hiHead.untreeify(map);
        else {
            tab[index + bit] = hiHead;
            if (loHead != null)
                hiHead.treeify(tab);
        }
    }
}
```

红黑树拆分成两个链表后，如果某个链表的节点数过少（`<=` 6），会转换回链表；否则保持红黑树结构。这样既保证了扩容效率，又避免了红黑树节点过少时的性能退化。

## 扩容的成本与优化

扩容的成本是 O(n)，其中 n 是 HashMap 中的元素总数。扩容时需要遍历所有元素，并为每个元素重新计算位置。

幸运的是，扩容操作不是频繁发生的。由于每次扩容容量翻倍，扩容次数与元素数量是对数关系。假设初始容量 16，最多容纳 2^30 个元素，扩容次数最多 30 次左右。

JDK 8 的扩容优化使得元素迁移更加高效。通过 `(e.hash & oldCap)` 的判断，元素不需要重新计算哈希，只需要检查一位即可确定新位置。这比 JDK 7 的头插法（需要反转链表）高效得多。

理解扩容机制，你就能理解为什么预设置合理的初始容量可以提升性能——减少扩容次数，减少元素迁移。
