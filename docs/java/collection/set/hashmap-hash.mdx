# HashMap 的扰动函数与哈希冲突

在上一篇文章中，我们简要提到了扰动函数的存在。但这个看似简单的 `(h = key.hashCode()) ^ (h >>> 16)` 背后隐藏着怎样的设计智慧？更重要的是，当哈希冲突不可避免地发生时，HashMap 是如何应对的？让我们深入探讨这些问题。

## 深入理解扰动函数的设计

扰动函数的全貌是这样的：

```java
static final int hash(Object key) {
    int h;
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}
```

首先，为什么需要扰动？想象一下，如果直接使用 `hashCode()` 的返回值进行寻址 `hash % n`，会有什么问题？问题在于，当 `n`（HashMap 容量）远小于 `hashCode()` 的取值范围时，哈希值的高位信息被完全丢弃了。

考虑一个具体场景：两个对象的 `hashCode()` 分别是 `0xABCDEF01` 和 `0x12345601`。它们的低 16 位相同（都是 `0x0001`），但高位不同。如果在一个容量为 16 的 HashMap 中直接取模，这两个对象会被放入同一个桶——尽管它们的哈希值完全不同。

扰动函数的设计正是为了解决这个问题。通过将原始哈希值与右移 16 位后的值进行异或，扰动函数将高位信息"混合"到了低位中。这样，即使最终参与寻址的只是低位，高位信息的影响也被保留了下来。

异或运算的选择也经过了深思熟虑。与 `&`（与运算）和 `|`（或运算）相比，异或 `^` 有一个独特的性质：相同为 0，不同为 1。这意味着异或运算能够在保留信息的同时，使结果更加分散。

## 哈希冲突：理想与现实的差距

即使有了扰动函数，在海量数据面前，哈希冲突仍然是不可避免的。这不是 HashMap 的缺陷，而是哈希表这一数据结构的固有特性。数学上可以证明，当 n 个元素放入 m 个桶中时，冲突的发生概率与 n/m（负载因子）成正比。

假设我们有 10 个元素放入一个容量为 8 的 HashMap 中。生日悖论告诉我们，即使元素数量远小于桶的数量，冲突仍然很可能发生。事实上，当 HashMap 的负载因子为 0.75 时，平均每个桶会有 0.75 个元素；超过这个值，冲突的概率就会显著上升。

## 解决哈希冲突的两大流派

计算机科学家们为解决哈希冲突提出了两大流派：开放地址法（Open Addressing）和链地址法（Chaining）。

**开放地址法**的核心思想是：当发生冲突时，通过某种探测策略寻找下一个可用的空槽。常见的探测策略包括：

- 线性探测：依次检查相邻的槽位
- 二次探测：探测间隔为二次方序列
- 双重哈希：使用第二个哈希函数计算探测间隔

开放地址法的优点是所有元素都存储在数组中，内存利用率高。但它的缺点也很明显：当负载因子较高时，探测序列会变得很长，查找效率急剧下降。此外，删除操作也很复杂，不能简单地置空，否则会切断后续元素的探测路径。

**链地址法**是 HashMap 采用的方案。当发生冲突时，所有哈希值相同的元素被组织成一个链表（或红黑树）。新元素可以插入链表头部（头插法）或尾部（尾插法），查找时遍历链表即可。

链地址法的优点是实现简单，删除操作容易，负载因子可以超过 1。缺点是当链表过长时，查找效率会退化为 O(n)。

## JDK 7 vs JDK 8：头插法与尾插法的较量

JDK 7 采用头插法插入新元素：

```java
e.next = table[index];
table[index] = e;
```

这种方式的特点是最新插入的元素在链表头部。在单线程环境下，头插法和尾插法没有功能上的区别。但在并发环境下，头插法会导致一个严重的问题——环形链表的产生。

JDK 8 改用尾插法：

```java
Node<K,V> lastAdded = tail;
lastAdded.next = e;
tail = e;
```

尾插法确保了元素按照插入顺序排列。更重要的是，在扩容重新分布元素时，JDK 8 保持了链表原本的顺序（从头到尾遍历，原位置和新位置各一个链表），避免了 JDK 7 中可能出现的环形链表问题。

## 红黑树：长链表的救赎

当链表长度过长时，查找效率会急剧下降。JDK 8 引入红黑树来解决这个问题。当链表长度超过 8 且数组容量达到 64 时，链表会被转换成红黑树。红黑树是一种自平衡的二叉搜索树，查找、插入、删除的时间复杂度都是 O(log n)。

为什么选择 8 作为树化阈值？这涉及到泊松分布的概率分析。在负载因子为 0.75 的情况下，一个桶中出现 k 个元素的概率遵循泊松分布。根据计算，一个桶中出现 8 个元素的概率约为 0.00000006（千万分之六），出现 8 个以上元素的概率更低。因此，8 是一个比较合理的阈值——既能在链表真正过长时进行转换，又能避免因为偶然的哈希碰撞而频繁树化。

## 哈希冲突的工程实践

理解了哈希冲突的原理后，我们可以在工程实践中做出更好的决策。

首先，合理的初始容量和负载因子设置可以减少冲突。默认初始容量是 16，负载因子是 0.75。如果你能预估元素数量，可以预先设置合适的容量，避免频繁扩容。

其次，选择合适的 key 类型也很重要。String、Integer 等不可变类型作为 key 是最安全的，因为它们的哈希值在创建后不会改变。自定义对象作为 key 时，需要确保 `equals()` 和 `hashCode()` 的实现是一致的——如果两个对象 equals，它们必须有相同的 hashCode。

最后，避免设计糟糕的哈希函数。如果自定义对象作为 key，确保哈希值的分布是均匀的，避免大量对象集中到少数几个桶里。

哈希冲突是哈希表这一伟大发明的"甜蜜负担"。正是因为有了冲突处理机制，我们才能在 O(1) 的平均时间复杂度下享受快速查找的便利。
