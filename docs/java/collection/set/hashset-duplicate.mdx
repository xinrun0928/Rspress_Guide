# HashSet 如何判断元素重复

HashSet 声称自己不允许重复元素，这个承诺是如何实现的？当你向 HashSet 添加一个元素时，它如何判断这个元素是否已经存在？让我们深入 HashSet 的内部，揭开元素去重的神秘面纱。

## HashSet 与 HashMap 的秘密关系

HashSet 之所以能够判断元素重复，秘密在于它并非真正的"独立实现"。HashSet 实际上是 HashMap 的"外壳"，它将所有操作都委托给内部的 HashMap 来完成。

```java
public class HashSet<E> extends AbstractSet<E> implements Set<E>, Cloneable, java.io.Serializable {
    private transient HashMap<E,Object> map;
    private static final Object PRESENT = new Object();
    
    public HashSet() {
        map = new HashMap<>();
    }
    
    public HashSet(Collection<? extends E> c) {
        map = new HashMap<>(Math.max((int) (c.size()/.75f) + 1, 16));
        addAll(c);
    }
    
    public boolean add(E e) {
        return map.put(e, PRESENT) == null;
    }
}
```

这个设计简洁而优雅。HashSet 内部维护了一个 `HashMap<E,Object>`，其中 E 是元素的类型。所有的元素都被当作 map 的 key 存储，而 value 则使用一个固定的哨兵对象 `PRESENT`（一个普通的 Object 实例）。

为什么选择 `PRESENT = new Object()` 作为哨兵？因为 Object 对象本身没有任何意义，它只是一个占位符，用来满足 Map 接口"键值对"的形式要求。HashSet 只关心 key 是否存在，value 的具体值并不重要。

## add 方法的执行流程

当你调用 `hashSet.add(element)` 时，HashSet 实际上是调用了 `map.put(element, PRESENT)`。这个方法返回一个值：

- 如果返回 `null`，说明 map 中原本没有这个 key，添加成功
- 如果返回一个非 null 的值（也就是之前的 PRESENT），说明 map 中已经存在这个 key，添加失败

HashSet 的 add 方法只需要检查返回值是否为 null 即可判断是否添加成功：

```java
public boolean add(E e) {
    return map.put(e, PRESENT) == null;
}
```

这就是为什么 HashSet 能够判断元素是否重复——它底层依赖 HashMap 的 key 唯一性机制。

## 判断重复的两个关键方法

HashMap（以及 HashSet）判断两个元素是否"相同"，依赖于两个核心方法：`hashCode()` 和 `equals()`。

**hashCode()** 决定元素应该放到哪个桶里。如果两个元素的 hashCode 不同，HashMap 可以确定它们一定不相等，直接放到不同的桶中。如果 hashCode 相同，它们会被放到同一个桶里，形成链表或红黑树。

**equals()** 决定在同一个桶里的元素是否"真正相等"。当 hashCode 相同时，HashMap 需要逐个调用 equals() 来比较元素。如果 equals() 返回 true，说明两个元素确实相等，后者会覆盖前者的 value。

这个判断流程可以用伪代码表示：

```java
if (hash1 == hash2) {
    if (equals(key1, key2)) {
        // 元素相等，key2 覆盖 key1
        return oldValue;
    } else {
        // hashCode 相同但 equals 不同，是哈希冲突
        // 在链表/红黑树中继续查找
    }
} else {
    // hashCode 不同，一定不相等
    // 直接放到对应桶中
}
```

## equals() 和 hashCode() 的契约

Java 为 equals() 和 hashCode() 定义了明确的契约，任何正确实现的类都必须遵守：

1. **自反性**：对于任何非 null 的引用值 x，`x.equals(x)` 必须返回 true
2. **对称性**：对于任何非 null 的引用值 x 和 y，`x.equals(y)` 必须与 `y.equals(x)` 返回相同的结果
3. **传递性**：对于任何非 null 的引用值 x、y 和 z，如果 `x.equals(y)` 返回 true 且 `y.equals(z)` 返回 true，那么 `x.equals(z)` 也必须返回 true
4. **一致性**：对于任何非 null 的引用值 x 和 y，只要对象没有被修改，多次调用 `x.equals(y)` 必须返回一致的结果
5. **非空性**：对于任何非 null 的引用值 x，`x.equals(null)` 必须返回 false

同时，还有 hashCode() 的契约：

1. 在同一个应用执行期间，只要对象的 equals() 比较所用到的信息没有被修改，那么对这个对象调用多次 hashCode() 必须返回相同的整数
2. 如果两个对象根据 equals() 方法是相等的，那么这两个对象调用 hashCode() 必须产生相同的整数
3. 如果两个对象根据 equals() 方法是不相等的，不要求两个对象调用 hashCode() 必须产生不同的整数（但这是良好的实践，有助于提高哈希表的性能）

## 自定义对象作为 HashSet 元素的注意事项

如果你希望将自定义对象添加到 HashSet 中，必须正确实现 `equals()` 和 `hashCode()` 方法。假设我们有一个 Person 类：

```java
public class Person {
    private String name;
    private int age;
    
    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (o == null || getClass() != o.getClass()) return false;
        Person person = (Person) o;
        return age == person.age && Objects.equals(name, person.name);
    }
    
    @Override
    public int hashCode() {
        return Objects.hash(name, age);
    }
}
```

在这个例子中，两个 Person 对象被认为相等当且仅当它们的 name 和 age 都相同。这意味着如果你向 HashSet 添加两个姓名和年龄都相同的 Person 对象，第二个添加会失败（返回 false）。

如果不实现 equals() 和 hashCode()，Person 对象会使用 Object 类的默认实现，即通过内存地址判断相等性。这会导致即使内容完全相同的两个 Person 对象，也会被视为不同的元素，从而都成功添加到 HashSet 中——这显然违背了 HashSet"不允许重复"的语义。

## 常见的坑：可变对象作为 HashSet 元素

一个容易被忽视的陷阱是：使用可变对象作为 HashSet 的元素。当一个对象被添加到 HashSet 后，如果修改了它的关键属性（影响 equals() 或 hashCode() 计算的属性），可能会导致：

1. 无法再找到这个对象（因为 hashCode 变了，原来的桶位置不正确）
2. 可能出现"幽灵元素"（在另一个位置出现一个看似重复的元素）
3. HashSet 的行为变得不可预测

因此，强烈建议使用不可变对象（如 String、Integer 等）或确保对象在 HashSet 中不会被修改关键属性。

理解了 HashSet 的去重机制，你会发现它的设计是如此简洁而强大——只需要借助 HashMap 的 key 唯一性，就实现了 Set 的核心功能。
