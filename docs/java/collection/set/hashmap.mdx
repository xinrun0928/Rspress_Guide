# HashMap 底层结构与寻址算法

如果你曾经打开过 HashMap 的源码，一定见过 `Node<K,V>[] table` 这个神秘的数组。它就像一个神奇的柜子，能够在 O(1) 的时间内找到任何你想要的东西。这个柜子是如何工作的？让我们一起揭开它的神秘面纱。

## 核心数据结构：数组 + 链表 + 红黑树

HashMap 的底层数据结构经历了一个优雅的演进过程。在 JDK 8 之前，它采用纯粹的数组加链表结构；而从 JDK 8 开始，引入红黑树来优化长链表场景下的查询性能。现在的 HashMap 可以说是"三位一体"的数据结构：数组用于快速寻址，链表用于解决哈希冲突，红黑树用于优化长链表的查找效率。

```java
transient Node<K,V>[] table;
transient int size;
transient int modCount;
int threshold;
final float loadFactor;
```

这段代码揭示了 HashMap 的核心组成。`table` 是一个 Node 类型的数组，每个 Node 既可以是普通的链表节点，也可以是红黑树的根节点。`size` 记录着键值对的数量，`threshold` 是决定是否需要扩容的临界值，而 `loadFactor`（负载因子）则是控制扩容时机的重要参数。

## Node 节点：HashMap 的基本单元

Node 是 HashMap 内部最基本的数据结构，它本质上是一个链表节点：

```java
static class Node<K,V> implements Map.Entry<K,V> {
    final int hash;
    final K key;
    V value;
    Node<K,V> next;
    
    Node(int hash, K key, V value, Node<K,V> next) {
        this.hash = hash;
        this.key = key;
        this.value = value;
        this.next = next;
    }
}
```

每个 Node 包含四个关键字段：`hash` 是 key 经过扰动函数处理后的哈希值，`key` 是存储的键，`value` 是对应的值，`next` 是指向下一个节点的指针。当多个 key 的哈希值相同时，它们会被组织成链表，`next` 指针串联起所有冲突的元素。

## 寻址算法：如何找到正确的位置

HashMap 能够在 O(1) 时间内定位元素，关键在于它的寻址算法。给定一个 key，HashMap 通过以下步骤确定它在数组中的位置：

```java
index = (n - 1) & hash
```

这个看似简单的公式蕴含着深刻的数学原理。`n` 是数组的容量（也就是 `table.length`），`hash` 是 key 经过扰动后的哈希值。位运算 `&`（与运算）的效率远高于传统的取模运算 `%`，这使得寻址过程几乎可以在常数时间内完成。

为什么 HashMap 的容量必须是 2 的幂次方？答案就在这个公式里。当 `n` 是 2 的幂次方时，`n - 1` 的二进制形式是一系列连续的 1。例如，当 `n = 16` 时，`n - 1 = 15` 的二进制是 `1111`。这时，`hash & (n-1)` 的结果正好等于 `hash % n`，但计算速度却快得多。

## 扰动函数：为什么哈希值需要"扰动"

你可能注意到了，在计算 index 之前，HashMap 会对 key 的原始哈希值进行特殊处理，这个处理过程就是"扰动函数"：

```java
static final int hash(Object key) {
    int h;
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}
```

`hashCode()` 方法是 Object 类提供的，每个对象都有。这个方法返回的是一个 32 位的整数。但是，如果直接使用这个原始哈希值进行寻址，会有什么问题呢？

问题在于哈希分布的不均匀性。假设我们有一个容量为 16 的 HashMap，如果直接使用原始哈希值的后 4 位进行寻址，那么只有 `hash % 16` 的低位参与运算。当两个对象的哈希值只是高位不同但低位相同时，它们就会被放到同一个桶里，导致哈希分布不均匀。

扰动函数的设计就是为了解决这个问题。它将原始哈希值 `h` 与 `h >>> 16`（h 无符号右移 16 位）进行异或运算，使得高位和低位都能参与到最终的寻址过程中。这样，即使两个对象的 `hashCode()` 值相近，只要高位有所不同，最终的哈希分布也会更加均匀。

## 哈希冲突：不可避免的甜蜜负担

虽然扰动函数能够使哈希分布更加均匀，但在海量数据面前，哈希冲突仍然不可避免。当两个不同的 key 计算出相同的哈希值时，它们会被放到同一个桶里，通过链表或红黑树来存储。

HashMap 处理冲突的方式非常巧妙。当新元素插入时，如果桶为空，直接将新节点放入桶中；如果桶不为空，则遍历链表或红黑树。如果找到相同的 key，就更新对应的 value；如果没找到，就将新节点插入链表或红黑树中。

对于链表，JDK 7 采用头插法，新节点会被插入到链表头部；JDK 8 采用尾插法，新节点被追加到链表尾部。这两种方式在单线程环境下功能相同，但在并发环境下，头插法可能导致环形链表的问题。

## 容量与寻址效率的数学关系

HashMap 的容量必须是 2 的幂次方这一约束，不仅仅是实现上的便利，更有着深刻的数学意义。假设 HashMap 容量为 n，那么寻址公式 `hash & (n-1)` 的结果范围是 0 到 n-1。

当 n 是 2 的幂次方时，`n-1` 的二进制形式是连续的 1。这意味着 `hash & (n-1)` 能够充分利用哈希值的所有低位。假设 n = 8（`n-1 = 7 = 0111`），一个哈希值 `h = 30`（`11110`）的寻址过程是：`11110 & 00111 = 110 = 6`。可以看到，只有哈希值的低 3 位参与了运算。

当 HashMap 扩容时，容量翻倍，n 变成 2n。此时 `n-1` 的二进制形式多了一个 1。扩容前后的寻址结果要么保持不变，要么增加 `oldCap`（旧容量）的偏移量。这使得扩容时元素的重新寻址变得非常高效，只需要判断一个额外的二进制位即可。
