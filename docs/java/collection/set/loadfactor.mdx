# 负载因子 0.75 的深层原因

当你 new 一个 HashMap 时，是否曾疑惑过为什么默认的负载因子是 0.75 而不是 0.5 或 1.0？这个看似随意的数字背后，隐藏着 Java 设计者对时间与空间平衡的精妙把控。让我们一起揭开这个数字的秘密。

## 什么是负载因子

负载因子（Load Factor）是 HashMap 的一个关键配置参数，定义为"元素数量 n"除以"数组容量 m"，即 `n/m`。它衡量了 HashMap 被"填满"的程度。当 `size > capacity * loadFactor` 时，HashMap 会触发扩容操作。

```java
public HashMap(int initialCapacity, float loadFactor) {
    // ...
    this.loadFactor = loadFactor;
    this.threshold = tableSizeFor(initialCapacity);
}

final float loadFactor;
int threshold;
```

HashMap 的扩容阈值 `threshold` 初始值等于 `capacity * loadFactor`。当元素数量超过这个阈值时，HashMap 会将容量翻倍，并重新分布所有元素。

## 为什么不是 0.5

一个直观的猜想是：使用 0.5 的负载因子，每个桶平均只有 0.5 个元素，冲突概率很低，查找速度很快。这个想法在理论上是对的，但忽略了内存的代价。

假设你需要存储 100 万个元素。如果负载因子是 0.5，HashMap 的容量至少需要 200 万，这意味着底层数组的大小是 200 万。这会产生巨大的内存开销——仅仅是为了存储一个指针数组，你需要分配一个能容纳 200 万个引用（64 位 JVM 上每个引用 8 字节）的数组，额外消耗约 16MB 内存。

更重要的是，负载因子 0.5 意味着你浪费了一半的空间来换取并不显著的查询性能提升。哈希表的查询性能从 O(n) 退化到 O(n) 并不是在负载因子略高时发生的——只有在链表非常长时才会有显著影响。

## 为什么不是 1.0

反过来想，如果负载因子是 1.0，会发生什么？HashMap 会在数组完全填满时才开始扩容。这听起来很节省空间，但问题在于：当负载因子接近 1.0 时，哈希冲突会急剧增加。

考虑泊松分布的概率分析。在负载因子为 0.75 时，每个桶中恰好有 k 个元素的概率可以用泊松分布计算：

```
P(X=k) = (λ^k * e^(-λ)) / k!
```

其中 λ = 0.75（平均每个桶的元素数）。根据计算：

- 0 个元素的概率：约 47%
- 1 个元素的概率：约 35%
- 2 个元素的概率：约 13%
- 4 个元素的概率：约 1.8%
- 8 个元素的概率：约 0.00000006

如果负载因子升到 1.0，λ 变成 1.0，链表的平均长度翻倍。这意味着查找时需要遍历更多的节点，查询性能明显下降。

## 0.75：时间与空间的黄金分割

Java 设计者选择 0.75 是经过深思熟虑的时间-空间权衡。这个数值既不会因为负载因子过低而浪费空间，也不会因为过高而导致严重的哈希冲突和查询性能下降。

从内存利用率角度看，0.75 意味着平均每个元素占用约 1.33 个槽位（1/0.75），空间利用率约为 75%。这是一个相当高的利用率，同时保留了足够的缓冲空间来避免频繁的哈希冲突。

从查询性能角度看，0.75 确保了平均链表长度保持在较短的水平。在负载因子 0.75 的条件下，绝大多数桶只有 0 到 2 个元素，查找几乎是 O(1) 的。只有极少数桶会有较长链表，这些长链表会在达到阈值时被转换成红黑树。

## 泊松分布的概率支撑

0.75 的选择得到了泊松分布的概率分析支持。泊松分布描述了在固定时间/空间内随机事件发生次数的概率分布，非常适合描述哈希表中每个桶的元素数量分布。

在负载因子 0.75 的条件下，λ = 0.75，这意味着平均每个桶有 0.75 个元素。根据泊松公式，我们可以计算出各种情况的概率：

- 链表长度为 0 的概率：47.2%
- 链表长度为 1 的概率：35.4%
- 链表长度为 2 的概率：13.3%
- 链表长度为 4 的概率：1.7%
- 链表长度为 8 的概率：0.00000006（十亿分之六）

这个 0.00000006 的概率非常重要。JDK 8 使用 8 作为链表树化的阈值，正是因为在正常（非攻击）情况下，一个桶中出现 8 个元素的概率极低。如果真的出现了 8 个元素的链表，说明哈希函数可能存在问题或者是恶意攻击，这时将其转换成红黑树是合理的优化。

## 扩容成本与频率的平衡

负载因子还决定了扩容的频率。假设我们有 n 个元素需要存储：

- 如果负载因子是 0.5，容量为 2n，需要 log₂(n/16) 次扩容（假设初始容量 16）
- 如果负载因子是 0.75，容量约为 1.33n，需要 log₁.₅(n/16) 次扩容
- 如果负载因子是 1.0，容量为 n，不需要扩容（理论上）

扩容是有成本的：需要重新分配数组、遍历所有元素、重新计算哈希和位置。这个过程是 O(n) 的。如果扩容过于频繁，会导致性能抖动；如果从不扩容，查询性能会退化。

0.75 在这个平衡点上找到了一个合适的折中。它使得扩容的频率相对较低，同时保持了较好的查询性能。

## 空间与时间的精妙平衡

HashMap 的设计哲学完美地体现了计算机科学中"时空权衡"（Space-Time Tradeoff）的基本原理。0.75 这个数字不是拍脑袋想出来的，而是基于概率论、性能分析和工程实践的综合考量。

它让我们意识到，在系统设计中，没有绝对的"最优解"，只有特定场景下的"最优权衡"。选择 0.75，是因为在大多数应用场景下，这个数值能够在内存效率和查询性能之间达到最佳平衡点。

理解了这个数字背后的原理，你在面试中面对"为什么 HashMap 的负载因子是 0.75"这个问题时，就能给出令人印象深刻的答案了。

如果让你针对一个特定的应用场景调整负载因子，你会考虑哪些因素？是一个需要极致查询性能的场景，还是一个内存受限的嵌入式环境？
