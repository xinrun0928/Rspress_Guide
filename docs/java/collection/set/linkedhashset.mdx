# LinkedHashSet 与访问顺序 LRU

HashSet 的元素是无序的，遍历结果完全取决于哈希分布。但有时候，我们需要保持元素的插入顺序，或者实现一个"最近最少使用"的缓存。LinkedHashSet 正是为解决这些问题而生的。

## LinkedHashSet 的诞生

LinkedHashSet 继承自 HashSet，它在 HashSet 的基础上增加了一条双向链表来维护元素的插入顺序（或访问顺序）。这个设计既简单又优雅：

```java
public class LinkedHashSet<E> extends HashSet<E> 
    implements Set<E>, Cloneable, Serializable {
    
    public LinkedHashSet(int initialCapacity, float loadFactor) {
        super(initialCapacity, loadFactor, true); // 第三个参数 true 表示 accessOrder
    }
    
    public LinkedHashSet(int initialCapacity) {
        super(initialCapacity, .75f, true);
    }
    
    public LinkedHashSet() {
        super(16, .75f, true);
    }
    
    public LinkedHashSet(Collection<? extends E> c) {
        super(Math.max(c.size()*2, 16), .75f, true);
        addAll(c);
    }
}
```

注意所有构造方法都调用了 `super(..., true)`，第三个参数 `true` 意味着启用了 accessOrder（访问顺序模式）。这使得 LinkedHashSet 不仅能维护插入顺序，还能维护访问顺序。

## LinkedHashMap：真正的幕后英雄

LinkedHashSet 的所有能力都来自它调用的父类构造方法中传入的 `LinkedHashMap`。HashSet 的一个关键构造方法是这样的：

```java
HashSet(int initialCapacity, float loadFactor, boolean dummy) {
    map = new LinkedHashMap<>(initialCapacity, loadFactor);
}
```

这个 dummy 参数只是一个标记，没有实际意义。关键是创建了一个 `LinkedHashMap` 而非普通的 `HashMap`。LinkedHashSet 添加元素时，实际上是操作内部的 LinkedHashMap。

## 双向链表：如何记住顺序

LinkedHashMap 在 HashMap 的基础上增加了双向链表机制。每个节点（称为 Entry）不仅有 `next` 指针指向同桶的下一个节点，还有 `before` 和 `after` 指针串联起所有元素：

```java
static class Entry<K,V> extends HashMap.Node<K,V> {
    Entry<K,V> before, after;
    Entry(int hash, K key, V value, Node<K,V> next) {
        super(hash, key, value, next);
    }
}
```

在 LinkedHashMap 中，有两个特殊的指针：`header.before` 指向第一个插入的元素，`header.after` 指向最后一个插入的元素。当我们按顺序遍历 LinkedHashMap 时，实际上是沿着 `before/after` 指针前进，而非 `table` 数组。

这意味着，即使两个元素哈希到同一个桶里，它们的遍历顺序也不会受哈希影响——遍历顺序完全由 `before/after` 链表决定。

## 访问顺序：LRU 的秘密

默认情况下，LinkedHashMap 按插入顺序维护元素。但当构造方法传入 `accessOrder=true` 时，LinkedHashMap 会切换到访问顺序模式。在这种模式下，每次访问一个元素（`get()` 或 `put()` 已存在的 key），该元素会被移动到链表的末尾。

这个特性使得 LinkedHashMap 天生适合实现 LRU（Least Recently Used，最近最少使用）缓存。LRU 缓存的淘汰策略是：当缓存满时，淘汰最长时间未被访问的元素。

```java
public class LRUCache<K, V> extends LinkedHashMap<K, V> {
    private final int capacity;
    
    public LRUCache(int capacity) {
        super(16, 0.75f, true); // accessOrder = true
        this.capacity = capacity;
    }
    
    @Override
    protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
        return size() > capacity;
    }
}
```

这段代码展示了如何用 LinkedHashMap 实现一个线程不安全的 LRU 缓存。`removeEldestEntry()` 方法会在每次 `put()` 或 `putAll()` 时被调用，当返回 true 时，最老的元素（即 `header.after`，链表第一个元素）会被自动删除。

## 链表操作：维护顺序的代价

LinkedHashMap 的双向链表维护是有代价的。首先，空间开销增加了——每个 Entry 需要额外的 `before` 和 `after` 指针。其次，每次插入或访问元素时，都需要更新这些指针。

插入新元素时，LinkedHashMap 需要：

1. 调用父类 HashMap 的插入逻辑
2. 将新节点加入双向链表的末尾（更新 `before` 和 `after` 指针）

访问元素时（accessOrder=true），需要：

1. 将被访问的节点从当前位置移除
2. 将节点重新插入链表末尾

这些额外的操作使得 LinkedHashMap 在纯粹的性能上不如 HashMap。但如果你需要维护顺序或实现 LRU，这些开销是完全值得的。

## LinkedHashSet vs HashSet：如何选择

选择 LinkedHashSet 还是 HashSet，取决于你的具体需求。

选择 **HashSet** 的场景：追求极致性能，不需要维护任何顺序。例如存储临时变量、去重处理、Set 运算等。

选择 **LinkedHashSet** 的场景：需要保持元素插入顺序，需要实现 LRU 缓存，或者需要可预测的遍历顺序。例如实现访问历史、任务队列、配置项管理等。

值得注意的是，LinkedHashSet 的迭代性能是 O(n)，与元素总数成正比，而非与桶数量相关。这在某些场景下反而是优势——HashSet 的迭代顺序几乎是随机的，而 LinkedHashSet 可以保证稳定的遍历顺序。

理解 LinkedHashSet 的双向链表机制，你会发现它不仅仅是一个"有序的 HashSet"，更是一个能够追踪访问历史的强大数据结构。
