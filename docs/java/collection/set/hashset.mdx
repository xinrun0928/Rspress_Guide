# HashSet 实现原理

如果你在 Java 世界里寻找一种"快速判断元素是否存在"的数据结构，`HashSet` 就是答案。它是 `Set` 接口最常用的实现，以 O(1) 的时间复杂度傲视群雄。但在这令人惊叹的性能背后，隐藏着一个鲜为人知的秘密：`HashSet` 根本不是一种独立的数据结构——它只是一个"披着 Set 外衣的 HashMap"。

## HashSet 的真实身份

让我们看看 `HashSet` 的源代码：

```java
public class HashSet<E> extends AbstractSet<E>
        implements Set<E>, Cloneable, java.io.Serializable {
    
    private transient HashMap<E, Object> map;
    
    // 一个所有 value 都共享的占位对象
    private static final Object PRESENT = new Object();
    
    public HashSet() {
        map = new HashMap<>();
    }
    
    public boolean add(E e) {
        return map.put(e, PRESENT) == null;
    }
    
    public boolean contains(Object o) {
        return map.containsKey(o);
    }
    
    public boolean remove(Object o) {
        return map.remove(o) == PRESENT;
    }
}
```

这就是 `HashSet` 的全部秘密！它内部持有一个 `HashMap` 对象，所有元素都被当作 key 存储，而所有 value 都是同一个对象——一个名为 `PRESENT` 的占位符。

## 为什么 HashSet 要基于 HashMap？

这个设计决策看似奇怪，实际上蕴含着深刻的设计哲学：**复用已有实现，避免代码重复**。

`HashSet` 需要实现的核心功能——元素的唯一性——`HashMap` 已经完美实现。`HashMap` 的 key 天然不允许重复，因为每次 `put()` 操作都会检查 key 是否已存在。如果 key 已存在，`put()` 会返回旧值；如果 key 不存在，返回 `null`。

于是，`HashSet` 的设计者做了一个优雅的决定：不重新发明轮子，而是复用 `HashMap` 的能力。具体来说：

1. **`add()` 方法**：调用 `map.put(e, PRESENT)`，返回值如果是 `null` 表示添加成功（key 不存在），返回非 `null` 表示添加失败（key 已存在）
2. **`contains()` 方法**：调用 `map.containsKey(o)`，直接利用 `HashMap` 的 key 查找
3. **`remove()` 方法**：调用 `map.remove(o)`，删除指定的 key

## HashSet 的时间复杂度

得益于 `HashMap` 的哈希表实现，`HashSet` 的各种操作都有出色的时间复杂度：

| 操作 | 平均时间复杂度 | 最坏时间复杂度 |
|------|---------------|---------------|
| add() | O(1) | O(n) |
| remove() | O(1) | O(n) |
| contains() | O(1) | O(n) |
| 遍历 | O(n) | O(n) |

"最坏情况 O(n)" 发生在所有元素都有相同哈希码时（如大量重写了 `hashCode()` 返回固定值的对象）。此时所有元素都被放入同一个桶，哈希表退化成链表，查找变成了线性搜索。

## HashSet 的构造函数

`HashSet` 提供了多个构造函数，提供了不同的定制能力：

```java
// 默认构造函数，初始容量16，负载因子0.75
public HashSet() {
    map = new HashMap<>();
}

// 指定初始容量
public HashSet(int initialCapacity) {
    map = new HashMap<>(initialCapacity);
}

// 指定初始容量和负载因子
public HashSet(int initialCapacity, float loadFactor) {
    map = new HashMap<>(initialCapacity, loadFactor);
}

// 从现有集合创建
public HashSet(Collection<? extends E> c) {
    map = new HashMap<>(Math.max((int) (c.size() / .75f) + 1, 16));
    addAll(c);
}
```

最后一个构造函数使用了 `Math.max((int) (c.size() / .75f) + 1, 16)` 来计算初始容量。这是因为 `HashMap` 的负载因子是 0.75，所以要容纳 c.size() 个元素，需要的最小容量是 `c.size() / 0.75`。

## HashSet 不是线程安全的

`HashSet` 继承自 `AbstractSet` → `AbstractCollection` → `Collection` → `Iterable`，整个继承链上没有任何同步措施。如果在多线程环境下使用 `HashSet`，需要外部同步或使用并发版本：

```java
// 方法一：使用 Collections 包装
Set<String> syncSet = Collections.synchronizedSet(new HashSet<>());

// 方法二：使用 CopyOnWriteArraySet
Set<String> concurrentSet = new CopyOnWriteArraySet<>();

// 方法三：使用 ConcurrentHashMap（推荐）
Set<String> concurrentSet2 = ConcurrentHashMap.newKeySet();
```

`ConcurrentHashMap.newKeySet()` 是 JDK 8+ 提供的创建并发 Set 的便捷方法，它返回一个由 `ConcurrentHashMap` 支持的 Set，性能远优于 `synchronizedSet` 包装的 `HashSet`。

## HashSet 的序列化

`HashSet` 实现了 `Serializable` 接口，但它有一个有趣的设计——`elementData` 数组被声明为 `transient`：

```java
private transient HashMap<E, Object> map;
```

虽然 `HashMap` 本身不是 transient，但当我们序列化 `HashSet` 时，它实际上序列化的是 `map` 的内容。`transient` 关键字在这里的作用是告诉序列化机制"这个字段不需要默认序列化"——因为 `HashMap` 的序列化有特殊实现。

## HashSet 与数据去重

`HashSet` 最常见的应用场景是**数据去重**：

```java
// 去除 List 中的重复元素
List<Integer> numbers = Arrays.asList(1, 2, 3, 2, 1, 4, 3, 5);
Set<Integer> uniqueNumbers = new HashSet<>(numbers);
System.out.println(uniqueNumbers);  // [1, 2, 3, 4, 5]

// 或者直接就地去重
List<Integer> list = new ArrayList<>(numbers);
Set<Integer> set = new HashSet<>(list);
list.clear();
list.addAll(set);
```

对于基本类型和不可变对象（如 `String`、`Integer`），这种方式工作得很好。但对于自定义对象，需要正确实现 `hashCode()` 和 `equals()` 方法。

## 自定义对象的去重

如果使用 `HashSet` 存储自定义对象，必须重写 `hashCode()` 和 `equals()` 方法：

```java
public class Person {
    private String name;
    private int age;
    
    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (o == null || getClass() != o.getClass()) return false;
        Person person = (Person) o;
        return age == person.age && Objects.equals(name, person.name);
    }
    
    @Override
    public int hashCode() {
        return Objects.hash(name, age);
    }
}
```

关于这两个方法的正确实现，有几个黄金法则：

1. **一致性**：如果两个对象相等，它们的 `hashCode()` 必须返回相同的值
2. **效率**：如果两个对象的 `hashCode()` 不同，它们**可能**不相等（这是哈希表高效的基础）
3. **规范性**：对于同一个对象，多次调用 `hashCode()` 应返回相同的值

## LinkedHashSet：带顺序的 HashSet

如果需要在去重的同时保持元素的插入顺序，可以使用 `LinkedHashSet`：

```java
Set<String> orderedSet = new LinkedHashSet<>();
orderedSet.add("First");
orderedSet.add("Second");
orderedSet.add("Third");
orderedSet.add("First");  // 不会重复添加

System.out.println(orderedSet);  // [First, Second, Third]
```

`LinkedHashSet` 继承自 `HashSet`，但它内部维护了一个双向链表。代价是更多的内存开销和略微的性能损失（约 5-10%）。

## 性能调优建议

使用 `HashSet` 时，以下建议可以帮助你优化性能：

### 预估容量，避免扩容

```java
// 反例：频繁扩容
Set<String> set1 = new HashSet<>();
for (String s : largeCollection) {
    set1.add(s);
}

// 正例：预估容量
Set<String> set2 = new HashSet<>(largeCollection.size());
for (String s : largeCollection) {
    set2.add(s);
}
```

### 选择合适的负载因子

默认负载因子 0.75 是一个时间和空间的平衡点。对于读多写少的场景，可以适当降低负载因子以提高查询性能：

```java
// 更低的负载因子，更快的查询，更大的空间开销
Set<String> set = new HashSet<>(initialCapacity, 0.5f);
```

### 保证哈希函数的质量

自定义对象的 `hashCode()` 实现应该让不同对象有尽可能不同的哈希值，避免"哈希冲突"：

```java
// 糟糕的 hashCode 实现
@Override
public int hashCode() {
    return 1;  // 所有对象都有相同的哈希值，哈希表退化成链表
}

// 良好的 hashCode 实现
@Override
public int hashCode() {
    return Objects.hash(field1, field2, field3);
}
```
