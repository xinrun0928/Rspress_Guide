# HashSet 如何判断元素重复：hashCode 与 equals 的契约

你知道这段代码的输出吗？

```java
Set<User> set = new HashSet<>();
User u1 = new User("Alice", 18);
User u2 = new User("Alice", 18);

set.add(u1);
set.add(u2);

System.out.println(set.size());  // 2？还是 1？
```

答案是 **2**——两个内容完全相同的对象，HashSet 居然认为它们不是重复的。

问题出在哪？大多数人会怀疑 HashSet 的实现，但真相是：**你可能没有正确重写 `hashCode()` 和 `equals()`**。

:::tip 剧透警告

这不是 HashSet 的 bug，而是 Java 设计哲学的体现。HashSet 判断重复依赖一套精确的契约，而大多数「奇怪」行为都可以追溯到对这个契约的误解或忽视。

:::

---

## 两种相等：== 和 equals

在 Java 中，相等有两种：

| 类型 | 说明 |
|---|---|
| `==` | 引用相等——比较的是内存地址，同一个对象则为 true |
| `equals()` | 对象相等——比较的是内容，由你来定义什么是「相等」 |

```java
String s1 = new String("hello");
String s2 = new String("hello");

s1 == s2;              // false，s1 和 s2 是两个不同的对象
s1.equals(s2);         // true，内容相同，String 重写了 equals()
```

:::note 一个常见的误解

初学者经常混淆这两种相等：

- `==` 比较的是「是不是同一个东西」（内存地址）
- `equals()` 比较的是「内容是否相同」（由类自己定义规则）

String 重写了 `equals()`，所以内容相同就返回 true。但自定义类默认的 `equals()` 继承自 `Object`，等价于 `==`——也就是比较地址。

:::

## HashSet 内部原理：桶结构

在理解 hashCode 和 equals 的关系之前，需要先知道 HashSet 内部是怎么存储数据的。

HashSet 的底层实现其实是 `HashMap`——它把每个元素同时作为 key 和 value 存入一个内部 map。这句理解透了，后面的所有内容就顺了。

```java
// HashSet 内部是这样实现的
private transient HashMap<E, Object> map;

// add 方法等价于：
public boolean add(E e) {
    return map.put(e, PRESENT) == null;
}
```

而 HashMap 的内部，是一个**数组 + 链表（+ 红黑树）**的结构，称为**桶（bucket）数组**：

```
桶数组（HashMap 内部）
┌─────┬─────┬─────┬─────┬─────┬─────┐
│ 0   │ 1   │ 2   │ 3   │ 4   │ ... │
└──┬──┴──┬──┴──┬──┴──┬──┴─────┴─────┘
   │     │     │
   ▼     ▼     ▼
 ┌───┐ ┌───┐ ┌───┐
 │A  │ │B  │ │C  │    ← 链表节点（冲突的元素串在一起）
 └───┘ └───┘ └───┘
```

当你调用 `set.add(element)` 时，HashSet 做了以下事情：

1. 计算 `element.hashCode()`，得到一个整数
2. 用这个整数决定元素应该落在哪个「桶」（bucket）里
3. 在那个桶里，用 `equals()` 逐一比较是否已存在
4. 不存在则添加，存在则忽略

:::tip 核心理解

HashSet 内部是一个桶数组，每个桶里可以放一个或多个元素。

- hashCode 决定元素落在哪个桶
- equals 决定桶里的元素是否「真的相等」

两级缺一不可。

:::

## HashSet 的两步验证

HashSet 判断两个元素是否重复，需要**两步验证**：

```text title="hashset-lookup-flow.txt"
第一步：计算 hashCode()
        ↓
第二步：根据 hashCode 找到对应桶
        ↓
     hash 不同？
    ↙          ↘
  是             否（hash 相同）
  ↓                  ↓
 不重复          在桶内遍历链表/红黑树
 不添加           ↓
              equals() 返回 true？
             ↙              ↘
          是                 否
          ↓                  ↓
        重复              不重复
        不添加             正常添加
```

为什么不能直接用 equals()？因为性能。

100 万个元素的集合，如果每次 add 都比较 100 万次 equals()，时间复杂度是 O(n)，性能会惨不忍睹。hashCode() 的作用是**快速过滤**：不同 hash 的元素，一定不相等，直接跳过，不需要比较 equals()。

---

## 为什么 hash 相同还要 equals？

因为哈希碰撞是正常现象，不是错误。

hashCode() 返回的是 int（32 位整数），而世界上可能有无限多的对象。所以**不同的对象产生相同的 hashCode，是必然会发生的事情**，这叫哈希碰撞（或哈希冲突）。

两个不同的字符串，它们的 hashCode() 可能恰好相同：

```java
// 大多数情况下，hash 不同，直接跳过 equals
"hello".hashCode() = 99162322
"world".hashCode() = 113318802  // hash 不同，不需要比较 equals

// 但极端情况下会发生碰撞
"hello".hashCode()  = 99162322
"hello2".hashCode() = 99162322  // 碰撞了！hash 相同
"hello".equals("hello2") = false  // equals 来兜底，判定不相等
```

> [!WARNING]
> hash 相同 ≠ 对象相等。hash 相同只是说明**可能相等**，equals() 才是最终的仲裁者。

## hashCode() 和 equals() 的契约

Java 对这两个方法有一个强制契约（来自 `Object` 类的 Javadoc 规范）：

```
契约一（一致性）：equals() 返回 true 的两个对象，hashCode() 必须返回相同的值
契约二（可重复性）：同一个对象多次调用 hashCode()，必须返回相同的值
契约三（反向不成立）：hashCode() 相同，equals() 不一定返回 true
```

第三点尤其重要：hash 碰撞是合法的，不需要避免。

### 违反契约的后果

只重写 equals()，不重写 hashCode()——这是最常见的错误：

```java
public class Student {
    private String name;
    private int age;

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (o == null || getClass() != o.getClass()) return false;
        Student student = (Student) o;
        return age == student.age &&
               Objects.equals(name, student.name);
    }
    // ❌ 错误：没有重写 hashCode()
}
```

```java
Set<Student> set = new HashSet<>();
Student s1 = new Student("Alice", 18);
Student s2 = new Student("Alice", 18);

set.add(s1);
System.out.println(set.contains(s2));  // false？！！
```

为什么会这样？

因为 JVM 默认的 hashCode() 基于内存地址，s1 和 s2 是两个不同的对象，内存地址不同，hashCode() 大概率也不同。HashSet 看到 hash 不同，根本不会调用 equals()，直接判定为不重复。所以 `contains(s2)` 返回 false——明明内容完全相同，却被认定不存在。

### 反过来：只重写 hashCode()，不重写 equals()

这种情况更隐蔽，错误更严重：

```java
public class Broken {
    private String name;

    @Override
    public int hashCode() {
        return name.hashCode();  // 重写了 hashCode
        // 但没有重写 equals()，默认比较地址
    }
}
```

```java
Set<Broken> set = new HashSet<>();
Broken b1 = new Broken("test");
Broken b2 = new Broken("test");

set.add(b1);
set.add(b2);  // 两次 add 都成功！因为 equals() 永远返回 false（地址不同）
System.out.println(set.size());  // 2 —— HashSet 里有两个 "相同" 的对象
```

> [!DANGER]
> 只重写 hashCode 不重写 equals，会导致 HashSet **无法去重**，相同的元素会被重复添加。

## 正确的实现方式

hashCode() 和 equals() 必须配对重写，缺一不可：

```java
public class Student {
    private String name;
    private int age;

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (o == null || getClass() != o.getClass()) return false;
        Student student = (Student) o;
        return age == student.age &&
               Objects.equals(name, student.name);
    }

    @Override
    public int hashCode() {
        return Objects.hash(name, age);
    }
}
```

IDEA 自动生成的版本（等价但更底层）：

```java
@Override
public int hashCode() {
    int result = name != null ? name.hashCode() : 0;
    result = 31 * result + age;
    return result;
}
```

---

## 为什么用 31 作为乘数？

IDEA 和 JDK 源码都喜欢用 `31 * result + age` 这种形式。这不是随便选的，有两个原因：

**原因一：质数减少碰撞**

哈希碰撞的分布与乘数有关。质数作为乘数能让哈希值分布更均匀。31 是一个被验证过效果不错的质数。

**原因二：JVM 会优化**

```
31 * i = (i << 5) - i
```

31 = 2^5 - 1，JVM 可以将乘法优化为移位和减法，在老一代 CPU 上性能更好。JDK 的 String.hashCode() 也用的是 31：

```java
public int hashCode() {
    int h = 0;
    for (int i = 0; i < value.length; i++) {
        h = 31 * h + value[i];
    }
    return h;
}
```

:::tip Objects.hash() 底层

JDK 的 `Objects.hash()` 底层其实就是循环计算 `31 * result + o.hashCode()`：

```java
public static int hash(Object... values) {
    return Arrays.hashCode(values);
}

public static int deepHashCode(Object a[]) {
    int result = 1;
    for (Object element : a)
        result = 31 * result +
                (element == null ? 0 : deepHashCode0(element));
    return result;
}
```

所以 `Objects.hash(name, age)` 等价于手写的 `31 * name.hashCode() + age`，只是更简洁。

:::

## 为什么 String 的 hashCode 表现优秀？

String 重写了 hashCode()，确保内容相同的字符串 hashCode 也相同：

```java
"hello".hashCode() = 99162322  // 每次都是这个值，可重复
"hello".hashCode() = 99162322  // 相同输入 → 相同输出
```

所以 `HashSet<String>` 的 contains() 非常快：先根据 hashCode 找到对应桶，由于 String 的 hash 碰撞极少，命中 bucket 后基本不需要调用 equals()，时间复杂度接近 O(1)。

这正是 HashMap 用 String 作为 key 时性能出色的原因之一。

## HashMap 的 key 也是同样的道理

HashSet 依赖 HashMap，而 HashMap 的 key 也有完全相同的规则。

如果你自定义类作为 HashMap 的 key，同样必须同时重写 hashCode() 和 equals()：

```java
Map<Student, String> map = new HashMap<>();

Student s1 = new Student("Alice", 18);
Student s2 = new Student("Alice", 18);

map.put(s1, "Alice's score");
System.out.println(map.get(s2));  // 如果没重写 hashCode，返回 null！
```

### 一道经典的面试改编题

```java
Map<String, Integer> map = new HashMap<>();

map.put("语文", 90);
map.put("数学", 80);

String k1 = new String("语文");
String k2 = new String("语文");

System.out.println(map.get(k1));  // 90，k1 == k2 为 false，但 equals() 为 true
System.out.println(map.get(k2));  // 90
```

为什么 String 可以？因为 String 重写了 hashCode() 和 equals()，内容相同即为相等。

## 实战场景：避免元素丢失

理解了 hashCode 和 equals 的关系之后，你在实际开发中需要注意一个致命陷阱：**不要在元素放入 HashSet 后修改其参与 hashCode 计算的属性**。

```java
Set<Student> set = new HashSet<>();
Student s = new Student("Alice", 18);

set.add(s);
System.out.println(set.size());  // 1

// ❌ 危险操作：修改了参与 hashCode/equals 的属性
s.setAge(20);

System.out.println(set.contains(s));  // false！
System.out.println(set.size());       // 仍然是 1，但 s 找不到了！
```

发生了什么？s 放入 HashSet 时，落在 bucket A 中。修改 age 后，s 的 hashCode 变了，可能落在 bucket B。但 HashSet 内部只维护了 bucket A 中的引用，它去 bucket B 找不到 s。

> [!DANGER]
> 放入 HashSet/HashMap 后修改参与 hashCode 计算的属性，会导致元素「丢失」——contains() 永远返回 false。这在实际项目中是极其隐蔽的 bug。

最佳实践：**如果类要作为 HashSet/HashMap 的 key，用 final 字段参与 hashCode/equals 计算**。

## 面试追问方向

### Q：为什么重写 equals() 时必须重写 hashCode()？

A：HashSet/HashMap 依赖「相等的对象必须有相同的 hashCode」这一契约（contract 1）。如果只重写 equals() 不重写 hashCode()，相等的对象可能 hash 不同。HashSet 在查找时根据 hash 定位桶，hash 不同则直接跳过了 equals()，导致明明相等的元素也被判定为不存在，contains() 永远返回 false。

### Q：hashCode() 相同，equals() 一定返回 true 吗？

A：不一定。hash 相同只是说明元素**可能**相等（落在同一个桶里），需要 equals() 来最终确认。这是哈希碰撞（Hash Collision），是正常现象，不是错误。反过来，equals() 返回 true，hashCode() 则**必须**相同——这是强制契约。

### Q：如果 hashCode() 总是返回相同的值会怎样？

A：HashMap 会退化为链表，所有元素都存在同一个桶里。put 和 get 的时间复杂度从 O(1) 退化为 O(n)。虽然功能上正确（因为 equals() 最终会兜底），但性能会严重下降。这就是为什么 String 和 Integer 这样的常用类型，重写 hashCode 时会尽量让 hash 值分布均匀。

### Q：为什么 Integer 的 hashCode() 就是自己的值？

A：因为 Integer 是值类型，相等意味着值相同，值相同则 hashCode 相同。直接返回自己是最优选择——无需额外计算，分布均匀。Short、Long、Byte 等包装类型也都遵循这一原则。

### Q：hashCode 的实现可以用浮点数吗？

A：可以，但要注意。浮点数的 hashCode 基于其二进制表示计算。例如 `Float.hashCode(1.0f)` 和 `Float.hashCode(1.0f + 0.0f)` 返回相同值（因为 1.0f 和 1.0f + 0.0f 在二进制上是相等的）。这和浮点数本身的特性一致：NaN 与自身不相等，NaN 的 hashCode 为 0。

### Q：equals() 里为什么要先判断 this == o？

A：`this == o` 是在比较地址。如果两个引用指向同一个对象（同一个内存地址），那么它们一定相等，不需要再比较字段。这是一个性能优化——如果对象已经相等，最快的判断方式就是 `==`。这也是为什么 `String.intern()` 有时会提升 HashMap 的性能：intern 后的 String 引用相同，`this == o` 直接命中，不需要走 equals()。

---

## 总结

HashSet 判断元素是否重复，靠的是 hashCode() 和 equals() 的配合：

```
hashCode() 快速定位桶
equals()    最终仲裁是否相等
```

两者缺一不可，而且必须遵循 Java 强制的契约：

```
相等的对象  →  hashCode() 必须相同（强契约）
hashCode 相同  →  equals() 不一定返回 true（正常碰撞）
```

理解了这个契约，你就能解释所有「奇怪」的 HashSet 行为，也能写出高质量的 value object。
