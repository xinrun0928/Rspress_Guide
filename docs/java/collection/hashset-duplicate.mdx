# HashSet 如何判断元素重复：hashCode 与 equals 的契约

你知道这段代码的输出吗？

```java
Set<User> set = new HashSet<>();
User u1 = new User("Alice", 18);
User u2 = new User("Alice", 18);

set.add(u1);
set.add(u2);

System.out.println(set.size());  // 2？还是 1？
```

答案是 **2**——两个内容完全相同的对象，HashSet 居然认为它们不是重复的。

问题出在哪？大多数人会怀疑 HashSet 的实现，但真相是：**你可能没有正确重写 `hashCode()` 和 `equals()`**。

---

## 两种相等：== 和 equals

在 Java 中，相等有两种：

| 类型 | 说明 |
|---|---|
| `==` | 引用相等——同一个内存地址 |
| `equals()` | 对象相等——内容相同 |

```java
String s1 = new String("hello");
String s2 = new String("hello");

s1 == s2;              // false，不同对象
s1.equals(s2);         // true，内容相同
```

HashSet 判断重复，靠的是 `equals()`——但只靠它，还不够。

---

## HashSet 的两步验证

HashSet 判断两个元素是否重复，需要**两步验证**：

1. **先比 hashCode()** — hash 不同，直接判定为不重复
2. **hash 相同时，再比 equals()** — equals 返回 true 才判定为重复

```
hash 不同 → 一定不重复
hash 相同 → 调用 equals()
            equals 返回 true → 重复，不添加
            equals 返回 false → 不重复，添加
```

为什么不能直接用 equals()？因为 100 万个元素的集合，如果每次 add 都比较 100 万次 equals()，性能会惨不忍睹。hashCode() 的作用是**快速过滤**：不同 hash 的元素，一定不相等，不需要比较 equals()。

---

## 为什么 hash 相同还要 equals？

因为哈希碰撞是正常现象。

两个不同的字符串，它们的 hashCode() 可能恰好相同：

```java
// 模拟哈希碰撞
"hello".hashCode() = 99162322
"world".hashCode() = 113318802  // hash 不同，无需比较 equals

// 但极端情况下
"hello".hashCode() = 99162322
"hello2".hashCode() = 99162322  // 碰撞了！
"hello".equals("hello2") = false  // equals 来兜底
```

hash 相同 ≠ 对象相等。equals() 是最终的**仲裁者**。

---

## hashCode() 和 equals() 的契约

Java 对这两个方法有一个强制契约（来自 `Object` 类的规范）：

```
1. 一致性：equals() 返回 true 的两个对象，hashCode() 必须返回相同的值
2. 可重复性：同一个对象多次调用 hashCode()，必须返回相同的值
3. 反过来不成立：hashCode() 相同，equals() 不一定返回 true
```

### 违反契约的后果

只重写 equals()，不重写 hashCode()：

```java
public class Student {
    private String name;
    private int age;

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (o == null || getClass() != o.getClass()) return false;
        Student student = (Student) o;
        return age == student.age &&
               Objects.equals(name, student.name);
    }
    // 错误：没有重写 hashCode()
}
```

```java
Set<Student> set = new HashSet<>();
Student s1 = new Student("Alice", 18);
Student s2 = new Student("Alice", 18);

set.add(s1);
System.out.println(set.contains(s2));  // false？！！
```

因为 JVM 默认的 hashCode() 基于内存地址，s1 和 s2 是不同对象，hash 很可能不同。HashSet 看到 hash 不同，根本不会调用 equals()，直接判定为不重复。

---

## 正确的实现方式

hashCode() 和 equals() 必须配对重写：

```java
public class Student {
    private String name;
    private int age;

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (o == null || getClass() != o.getClass()) return false;
        Student student = (Student) o;
        return age == student.age &&
               Objects.equals(name, student.name);
    }

    @Override
    public int hashCode() {
        return Objects.hash(name, age);
        // 等价于：
        // int result = name != null ? name.hashCode() : 0;
        // result = 31 * result + age;
        // return result;
    }
}
```

IDEA 自动生成的版本：

```java
@Override
public int hashCode() {
    int result = name != null ? name.hashCode() : 0;
    result = 31 * result + age;
    return result;
}
```

为什么用 31？因为它是质数，能减少哈希碰撞；而且 31 = (1 << 5) - 1，JVM 会优化为移位运算，性能好。

---

## 为什么 String 的 hashCode 表现优秀？

String 重写了 hashCode()，确保内容相同的字符串 hashCode 也相同：

```java
"hello".hashCode() = 99162322  // 每次都是这个值
"hello".hashCode() = 99162322  // 相同
```

所以 HashSet<String> 的 contains() 非常快，命中 bucket 后基本不需要调用 equals()。

这正是 HashMap 用 String 作为 key 时性能出色的原因之一。

## 面试追问方向

**Q：为什么重写 equals() 时必须重写 hashCode()？**

A：HashSet/HashMap 依赖「相等的对象必须有相同的 hashCode」这一契约。如果只重写 equals() 不重写 hashCode()，相等的对象可能 hash 不同，导致集合操作失败。

**Q：hashCode() 相同，equals() 一定返回 true 吗？**

A：不一定。hash 相同只是说明元素可能相等，需要 equals() 来最终确认。这是哈希碰撞，不是错误。

**Q：如果 hashCode() 总是返回相同的值会怎样？**

A：HashMap 会退化为链表，所有元素都存在同一个桶里。put 和 get 的复杂度从 O(1) 退化为 O(n)。

**Q：为什么 Integer 的 hashCode() 就是自己的值？**

A：因为它是值对象，相等意味着值相同，值相同意味着 hashCode 相同。直接返回自己是最优选择。
