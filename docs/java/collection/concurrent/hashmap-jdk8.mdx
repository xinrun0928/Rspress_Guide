# ConcurrentHashMap JDK 8+：CAS + synchronized

JDK 8 对 ConcurrentHashMap 进行了彻底的重新设计。放弃了 JDK 7 的 Segment 分段锁，转而采用更细粒度的实现：数组加链表/红黑树，配合 CAS 进行无锁读操作，使用 synchronized 锁保护写操作。这种设计与普通 HashMap 趋于一致，但增加了线程安全的保护。

## JDK 8 的核心数据结构

```java
public class ConcurrentHashMap<K,V> extends AbstractMap<K,V> 
    implements ConcurrentMap<K,V>, Serializable {
    
    transient volatile Node<K,V>[] table;
    private static final int MAXIMUM_CAPACITY = 1 << 30;
    private static final int DEFAULT_CAPACITY = 16;
    private static final float LOAD_FACTOR = 0.75f;
    static final int TREEIFY_THRESHOLD = 8;
    static final int UNTREEIFY_THRESHOLD = 6;
    private transient volatile int sizeCtl;
    
    static class Node<K,V> implements Map.Entry<K,V> {
        final int hash;
        final K key;
        volatile V value;
        volatile Node<K,V> next;
    }
    
    static final class TreeNode<K,V> extends Node<K,V> {
        TreeNode<K,V> parent;
        TreeNode<K,V> left;
        TreeNode<K,V> right;
        TreeNode<K,V> prev;
        boolean red;
    }
}
```

与 JDK 7 的主要区别：

1. **没有 Segment**：直接使用 Node 数组，与 HashMap 结构一致
2. **sizeCtl**：替代了 Segment 数组，控制数组初始化和扩容
3. **TreeNode**：红黑树节点，与 HashMap 的 TreeNode 类似

## CAS 与 Unsafe 类

JDK 8 的 ConcurrentHashMap大量使用 CAS（Compare-And-Swap）进行无锁操作。CAS 是现代 CPU 提供的一种原子指令，它包含三个操作数：

- **内存位置 V**：要更新的变量地址
- **预期值 A**：变量当前的期望值
- **新值 B**：要写入的新值

当且仅当 V 的当前值等于 A 时，CAS 才将 V 更新为 B；否则，不做任何操作并返回当前值。整个操作是原子的。

JDK 通过 `Unsafe` 类提供 CAS 支持：

```java
private static final sun.misc.Unsafe U = sun.misc.Unsafe.getUnsafe();

private static final long SIZECTL = U.objectFieldOffset(
    ConcurrentHashMap.class.getDeclaredField("sizeCtl"));

// CAS 操作
U.compareAndSwapInt(this, SIZECTL, c, v)
U.compareAndSwapObject(this, TABLE, null, node)
```

Unsafe 类提供了直接操作内存的能力，是 JDK 内部实现并发工具的基础。但它不是面向公众的 API，不应该在应用代码中使用。

## sizeCtl：控制并发操作的核心变量

sizeCtl 是一个 volatile 变量，承担了多重职责：

```java
private transient volatile int sizeCtl;

// sizeCtl 的含义：
// - 未初始化时：sizeCtl = -1，表示正在初始化
// - 初始化完成后：sizeCtl = 实际容量（threshold）
// - 扩容时：sizeCtl = -(1 + 参与扩容的线程数)
// - 其他时候：sizeCtl < 0 时表示有线程在修改表
```

这种设计使得各种状态都能通过一个变量协调，减少了锁的使用。

## 初始化：ensureSegment

数组的初始化是延迟的，只有在第一次 put 时才会初始化：

```java
private Segment<K,V> ensureSegment(int k) {
    final Segment<K,V>[] segments = this.segments;
    long u = (k << SSHIFT) + SBASE;
    if ((s = (Segment<K,V>)U.getObjectVolatile(segments, u)) == null) {
        Segment<K,V> s = new Segment<K,V>(lf, threshold);
        if (U.compareAndSwapObject(segments, u, null, s))
            return s;
    }
    return s;
}
```

使用 CAS 确保只有一个线程真正创建 Segment，其他线程通过自旋等待。

## putVal：CAS + synchronized

JDK 8 的 put 操作使用两阶段策略：先用 CAS 尝试插入，失败后使用 synchronized：

```java
final V putVal(K key, V value, boolean onlyIfAbsent) {
    if (key == null || value == null) throw new NullPointerException();
    int hash = spread(key.hashCode());
    int binCount = 0;
    for (Node<K,V>[] tab = table;;) {
        Node<K,V> f; int n, i, fh;
        if (tab == null || (n = tab.length) == 0)
            tab = initTable();
        else if ((f = tabAt(tab, i = (n - 1) & hash)) == null) {
            // 桶为空，使用 CAS 尝试插入
            if (casTabAt(tab, i, null, new Node<K,V>(hash, key, value, null)))
                break;
        }
        else if ((fh = f.hash) == MOVED)
            tab = helpTransfer(tab, f);
        else {
            // 桶不为空，使用 synchronized 锁定桶
            synchronized (f) {
                if (tabAt(tab, i) == f) {
                    if (fh >= 0) {
                        binCount = 1;
                        for (Node<K,V> e = f;; ++binCount) {
                            K ek;
                            if (e.hash == hash &&
                                ((ek = e.key) == key ||
                                 (key != null && key.equals(ek)))) {
                                V oldVal = e.value;
                                if (!onlyIfAbsent || oldVal == null)
                                    e.value = value;
                                break;
                            }
                            Node<K,V> pred = e;
                            if ((e = e.next) == null) {
                                pred.next = new Node<K,V>(hash, key, value, null);
                                break;
                            }
                        }
                    }
                    else if (f instanceof TreeBin) {
                        // 红黑树插入
                        binCount = 2;
                        TreeNode<K,V> p = ((TreeBin<K,V>)f).putTreeVal(hash, key, value);
                        // ...
                    }
                }
            }
            if (binCount != 0) {
                if (binCount >= TREEIFY_THRESHOLD)
                    treeifyBin(tab, i);
                break;
            }
        }
    }
    addCount(1L, binCount);
    return null;
}
```

关键流程：

1. **桶为空**：使用 `casTabAt()` CAS 尝试插入，成功则结束
2. **桶正在迁移**（hash == MOVED）：协助扩容
3. **桶已有元素**：使用 `synchronized (f)` 锁定桶头节点，然后遍历链表或红黑树进行插入/更新

## tabAt：volatile 读取

```java
static final <K,V> Node<K,V> tabAt(Node<K,V>[] tab, int i) {
    return (Node<K,V>)U.getObjectVolatile(tab, ((long)i << ASHIFT) + ABASE);
}
```

`tabAt()` 使用 volatile 读取（通过 Unsafe），确保看到其他线程的最新写入。

## helpTransfer：协助扩容

```java
final Node<K,V>[] helpTransfer(Node<K,V>[] tab, Node<K,V> f) {
    Node<K,V>[] nextTab; int sc;

```

当一个线程发现要操作的桶正在被迁移（桶头是 ForwardingNode），它会调用 `helpTransfer()` 协助迁移，而不是阻塞等待。

## JDK 8 的优势

相比 JDK 7，JDK 8 的设计有以下优势：

**更高的并发度**：锁的粒度从 Segment 缩小到单个桶。16 个线程可以同时操作 16 个不同的桶，而 JDK 7 最多也是 16 个（由 Segment 数量决定），但 JDK 8 更容易达到这个并发度。

**更低的内存开销**：没有 Segment 数组，每个桶独立加锁，内存使用更高效。

**更好的扩容支持**：ForwardingNode 机制允许其他线程协助扩容，加快扩容进度。

**实现更简单**：不再需要两层数据结构（Segment + HashEntry），代码更易维护。
