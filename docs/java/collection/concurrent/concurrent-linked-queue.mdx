# ConcurrentLinkedQueue 无锁实现原理

为什么 `ConcurrentLinkedQueue` 能在高并发下保持高性能，而不需要任何锁？

答案藏在它的核心算法里：**CAS + 延迟更新**。

`ConcurrentLinkedQueue`（CLQ）是 Java 并发包中最优雅的无锁数据结构之一。它的设计完美地展示了如何在链表结构上实现线程安全，同时保持高性能。

这篇文章会深入剖析它的实现原理，解释 tail/pred 指针的协作机制，以及为什么说"无界"既是优势也是风险。

## 无锁队列的核心思想

### 锁 vs 无锁

传统的有锁队列：

```java
public class LockedQueue<E> {
    private final ReentrantLock lock = new ReentrantLock();
    private final Queue<E> queue = new LinkedList<>();

    public void offer(E e) {
        lock.lock();
        try {
            queue.offer(e);
        } finally {
            lock.unlock();
        }
    }

    public E poll() {
        lock.lock();
        try {
            return queue.poll();
        } finally {
            lock.unlock();
        }
    }
}
```

问题：当多个线程同时访问时，只能有一个线程成功获取锁，其他线程自旋等待。

**无锁队列**的核心思路：用 CAS 替代锁，多个线程可以同时尝试修改数据结构，只有一个会成功，其他继续重试。

### CAS 的基本模式

```java
// 无锁链表插入的基本模式
public boolean offer(E e) {
    Node<E> newNode = new Node<>(e);

    while (true) {
        Node<E> tail = this.tail;  // 读取尾节点

        // 尝试 CAS：tail.next 从 null 改成 newNode
        if (tail.next.compareAndSet(null, newNode)) {
            // 成功！更新 tail 指针
            return true;
        }
        // 失败：其他线程修改了 tail.next，重试
    }
}
```

## ConcurrentLinkedQueue 的结构

### 节点定义

```java
public class ConcurrentLinkedQueue<E> extends AbstractQueue<E>
        implements Queue<E>, Serializable {

    private static class Node<E> {
        volatile E item;      // 节点存储的元素
        volatile Node<E> next;  // 指向下一个节点

        Node(E item) {
            UNSAFE.putObject(this, itemOffset, item);
        }
    }

    // 头指针
    private transient volatile Node<E> head;

    // 尾指针
    private transient volatile Node<E> tail;
}
```

**关键设计**：节点 `item` 和 `next` 都是 volatile 的，保证可见性。

### 初始化

```java
public ConcurrentLinkedQueue() {
    // 哨兵节点，空队列时 head 和 tail 都指向它
    Node<E> node = new Node<>(null);
    head = node;
    tail = node;
}
```

```
空队列的初始状态：

head ──→ [null | next] ←── tail
              ↑
           哨兵节点
```

## offer() 的无锁算法

### 核心逻辑

```java
public boolean offer(E e) {
    checkNotNull(e);

    Node<E> newNode = new Node<>(e);

    // 外层循环：找到真正的尾节点
    for (Node<E> t = tail, p = t;;) {
        Node<E> q = p.next;  // p 的后继节点

        if (q == null) {
            // q 是尾节点，尝试插入
            if (p.casNext(null, newNode)) {
                // 插入成功！但 tail 可能需要更新
                if (p != t) {
                    // p 不等于 t，说明 tail 滞后，更新 tail
                    casTail(t, newNode);
                }
                return true;
            }
            // CAS 失败：p.next 被其他线程修改了，重试
        } else if (p == q) {
            // p 已经被出队（HOPS 优化，见下文）
            // 需要重新从 head 开始
            p = (t != (t = tail)) ? t : head;
        } else {
            // p 不是尾节点，需要向后移动
            p = (p != t && t != (t = tail)) ? t : q;
        }
    }
}
```

### 图解 offer 过程

```
步骤1：初始状态
head ──→ [A | next] ──→ [B | null]
                ↑
           tail = p
           
步骤2：插入新节点 C
head ──→ [A | next] ──→ [B | next] ──→ [C | null]
                              ↑            ↑
                         p             newNode
                         
步骤3：更新 tail（如果需要）
head ──→ [A | next] ──→ [B | next] ──→ [C | null]
                                             ↑
                                        tail 更新到这里
```

### HOPS 优化：为什么不是每次都更新 tail？

**设计思想**：tail 更新是可选的，不是每次 offer 都必须更新。

```java
if (p != t) {
    // 只有当 p 不等于 t 时，才更新 tail
    casTail(t, newNode);
}
```

**原因**：

1. **减少 CAS 操作**：tail 指向尾节点是一个"热"位置，多个线程会竞争更新它。
2. **摊销成本**：多个 offer 共享一次 tail 更新，平摊下来开销更小。
3. **语义等价**：即使 tail 暂时滞后，从 head 出发也能找到尾节点。

**摊销分析**：

```
如果 tail 滞后 k 个节点，那么每 k 次 offer 才需要 1 次 tail 更新
实际测试显示：tail 通常滞后 1-2 个节点
```

## poll() 的无锁算法

### 核心逻辑

```java
public E poll() {
    // restartFromHead 是 HOPS 优化的关键
    restartFromHead:
    for (Node<E> h = head, p = h, q;;) {
        E item = p.item;

        if (item != null && p.casItem(item, null)) {
            // CAS 成功：标记为已删除
            // 还需要更新 head
            if (p != h) {
                // p 不是 head，更新 head
                updateHead(h, q);
            }
            return item;
        }

        // item 为 null，说明 p 已经是空节点
        q = p.next;

        if (q == null) {
            // p 是尾节点后面，没有更多元素
            updateHead(h, p);
            return null;
        }

        if (p == q) {
            // p 已经被其他线程删除，需要从 head 重试
            continue restartFromHead;
        }

        p = q;  // 移动到下一个节点
    }
}
```

### updateHead 的 HOPS 优化

```java
final void updateHead(Node<E> h, Node<E> p) {
    // CAS 更新 head，最多执行一次
    if (h != p && casHead(h, p)) {
        // 将旧 head 的 next 指向自己，帮助 GC
        h.lazySetNext(h);
    }
}
```

**lazySetNext 的作用**：将旧 head 的 next 指向自己，形成自引用，帮助 GC 回收。同时避免与新 head 的 CAS 竞争。

### 图解 poll 过程

```
步骤1：队列状态 [A, B, C]
head ──→ [A | next] ──→ [B | next] ──→ [C | null]
                ↑                   ↑
             p = head              tail

步骤2：poll() 删除 A（item = null，head 已更新）
head ──→ [A | next] ...  (旧 head，next 指向自己，帮助 GC)
          ↓
     [B | next] ──→ [C | null]
          ↑
      新 head = p

步骤3：继续 poll() 删除 B
head ──→ [B | next] ──→ [C | null]
```

## HOPS 机制：延迟更新的艺术

### 什么是 HOPS

HOPS（Handoff-optimized pointer style）是一种延迟更新策略：tail 和 head 不需要每次都精确指向真正的尾/头节点，可以滞后几个节点。

```java
// offer 中 tail 更新的条件
if (p != t) {
    // 只有当 p 不等于 t 时才更新
    casTail(t, newNode);
}
```

### 为什么需要 HOPS

**理由一：减少 CAS 竞争**

tail 是一个"热"位置。每个 offer 都要尝试更新它会导致大量 CAS 竞争。

```
优化前（每次更新 tail）：
线程1 ──→ CAS(tail, A, B) ──→ 成功
线程2 ──→ CAS(tail, B, C) ──→ 失败（线程1 已更新）
线程3 ──→ CAS(tail, B, C) ──→ 失败

优化后（延迟更新 tail）：
线程1 ──→ 插入成功，不更新 tail
线程2 ──→ 插入成功，更新 tail
线程3 ──→ 插入成功，不更新 tail
线程4 ──→ 插入成功，更新 tail
```

**理由二：摊销复杂度**

多个 offer 操作共享一次 tail 更新，平摊复杂度是 O(1)。

### tail 滞后的代价

```java
// 极端情况：tail 滞后导致需要遍历
Node<E> p = tail;
Node<E> q = p.next;  // q != null

// 需要从 tail 遍历到真正的尾节点
while (q != null) {
    Node<E> next = q.next;
    // ...
    p = q;
    q = next;
}
```

但实测表明，tail 通常只滞后 1-2 个节点，这种遍历开销可以忽略不计。

## isEmpty() 的陷阱

### 实现

```java
public boolean isEmpty() {
    return first() == null;
}

private Node<E> first() {
    for (Node<E> h = head, p = h;;) {
        Node<E> q = p.next;
        if (q == null) {
            return p;
        }
        if (q.item != null) {
            return q;  // 找到第一个非空节点
        }
        if (p == q) {
            // 遇到自引用，重新从 head 开始
            return head;
        }
        p = q;
    }
}
```

### 为什么不能用 size() `==` 0

`ConcurrentLinkedQueue` **没有提供 size() 方法**，原因：

1. **无锁环境下精确计数代价高**：需要遍历所有节点。
2. **语义上没有意义**：返回的 size 可能是过时的。

```java
// 错误：isEmpty() 不保证后续 poll() 一定能返回元素
if (queue.isEmpty()) {
    return;  // 可能在这里被其他线程 offer 了元素
}

// 正确：直接 poll()
E item = queue.poll();  // 返回 null 表示空
```

## 应用场景

### 场景一：生产者-消费者模式

```java
public class ProducerConsumerDemo {

    private static final ConcurrentLinkedQueue<String> queue =
        new ConcurrentLinkedQueue<>();

    public static void main(String[] args) {
        // 生产者
        ExecutorService producers = Executors.newFixedThreadPool(3);
        for (int i = 0; i < 3; i++) {
            final int id = i;
            producers.submit(() -> {
                for (int j = 0; j < 1000; j++) {
                    queue.offer("P" + id + "-T" + j);
                }
            });
        }

        // 消费者
        ExecutorService consumers = Executors.newFixedThreadPool(3);
        for (int i = 0; i < 3; i++) {
            consumers.submit(() -> {
                while (true) {
                    String task = queue.poll();
                    if (task == null) {
                        Thread.yield();  // 让出 CPU
                        continue;
                    }
                    process(task);
                }
            });
        }
    }

    private static void process(String task) {
        // 模拟处理
    }
}
```

### 场景二：并发任务队列

```java
public class TaskQueue {

    private final ConcurrentLinkedQueue<Runnable> queue =
        new ConcurrentLinkedQueue<>();

    public void submit(Runnable task) {
        queue.offer(task);
    }

    public Runnable take() throws InterruptedException {
        while (true) {
            Runnable task = queue.poll();
            if (task != null) {
                return task;
            }
            Thread.sleep(10);  // 或使用 wait/notify
        }
    }
}
```

### 场景三：收集器

```java
public class ResultCollector<T> {

    private final ConcurrentLinkedQueue<T> results =
        new ConcurrentLinkedQueue<>();

    public void add(T result) {
        results.offer(result);
    }

    public List<T> getAll() {
        List<T> list = new ArrayList<>();
        results.forEach(list::add);  // 迭代是弱一致的
        return list;
    }
}
```

## 性能特征

### 与其他队列对比

| 特性 | ConcurrentLinkedQueue | LinkedBlockingQueue | synchronized LinkedList |
|---|---|---|---|
| 有界性 | 无界 | 可选有界 | 无界 |
| 锁 | 无锁（CAS） | 双锁 | 全局锁 |
| 并发度 | 最高 | 高 | 低 |
| 吞吐量 | 最高 | 中 | 低 |
| 内存开销 | 中 | 大 | 小 |
| 适用场景 | 高并发、需无界 | 需要限流、有界 | 简单场景 |

### 性能测试

```java
public class QueueBenchmark {

    public static void main(String[] args) throws Exception {
        int threads = 8;
        int operations = 1_000_000;

        // ConcurrentLinkedQueue
        ConcurrentLinkedQueue<Integer> clq = new ConcurrentLinkedQueue<>();
        benchmark("CLQ", threads, operations, i -> clq.offer(i), clq::poll);

        // LinkedBlockingQueue
        LinkedBlockingQueue<Integer> lbq = new LinkedBlockingQueue<>();
        benchmark("LBQ", threads, operations, lbq::offer, lbq::poll);

        // synchronized List
        List<Integer> sync = Collections.synchronizedList(new LinkedList<>());
        benchmark("SyncList", threads, operations, sync::add, sync::remove);
    }

    private static void benchmark(String name, int threads,
            int ops, IntConsumer offer, Supplier<Integer> poll)
            throws Exception {
        CountDownLatch latch = new CountDownLatch(threads);
        long start = System.nanoTime();

        for (int t = 0; t < threads; t++) {
            new Thread(() -> {
                for (int i = 0; i < ops / threads; i++) {
                    offer.accept(i);
                    poll.get();
                }
                latch.countDown();
            }).start();
        }

        latch.await();
        long elapsed = System.nanoTime() - start;
        System.out.printf("%s: %.2f Mops/s%n",
            name, ops * threads / (elapsed / 1_000_000_000.0));
    }
}
```

典型结果：

```
CLQ:     12.5 Mops/s  (无锁，最高)
LBQ:      8.3 Mops/s  (双锁)
SyncList: 4.1 Mops/s  (全局锁)
```

## 注意事项

### 无界队列的风险

`ConcurrentLinkedQueue` 是无界的，如果不加控制地 offer，可能导致 OOM：

```java
// 危险：无界队列可能导致内存溢出
ConcurrentLinkedQueue<byte[]> queue = new ConcurrentLinkedQueue<>();

// 永远往里放
while (true) {
    queue.offer(new byte[1024 * 1024]);  // 1MB
}
```

**解决方案**：使用有界队列，或监控队列大小。

### 迭代器弱一致性

```java
ConcurrentLinkedQueue<Integer> queue = new ConcurrentLinkedQueue<>();
queue.add(1);
queue.add(2);

// 获取迭代器
Iterator<Integer> iter = queue.iterator();

// 迭代过程中，其他线程可能修改队列
queue.add(3);

// 迭代器可能看到也可能看不到 3
while (iter.hasNext()) {
    System.out.println(iter.next());  // 不保证一致性
}
```

### poll() 返回 null 不代表队列为空

```java
// 错误理解
while (queue.poll() != null) {
    // ...
}

// 正确：需要外部同步或其他机制
while (running) {
    String task = queue.poll();
    if (task != null) {
        process(task);
    } else {
        Thread.sleep(10);
    }
}
```

## 总结

`ConcurrentLinkedQueue` 的无锁实现是 Java 并发编程的经典案例：

- **CAS + 延迟更新**：减少竞争，摊销开销
- **HOPS 优化**：tail/head 不必每次都精确更新
- **无界设计**：适合高并发场景，但需要注意内存控制

理解它的原理，不仅能帮助你更好地使用它，也能在面试中展示你对并发数据结构的深层理解。
