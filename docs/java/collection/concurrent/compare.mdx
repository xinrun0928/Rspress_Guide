# 各并发容器对比与选型

Java 并发包提供了多种并发容器，每种都有其特定的设计目标和适用场景。选择合适的并发容器需要理解它们的特性、性能特点和适用条件。

## 并发 Map 对比

| 容器 | 线程安全方式 | 并发度 | null 支持 | 适用场景 |
|------|------------|--------|----------|----------|
| Hashtable | synchronized 全局锁 | 低 | 否 | 不推荐 |
| Collections.synchronizedMap | synchronized 全局锁 | 低 | 是 | 不推荐 |
| ConcurrentHashMap | CAS + synchronized 桶锁 | 高 | 否 | 通用高并发 |
| ConcurrentSkipListMap | CAS + 跳表 | 高 | 否 | 需要排序的高并发 |

**ConcurrentHashMap** 是首选，适用于大多数需要线程安全 Map 的场景。

**ConcurrentSkipListMap** 在需要按键排序且要求高并发时使用，其性能与 ConcurrentHashMap 相当。

## 并发 List 对比

| 容器 | 线程安全方式 | 读性能 | 写性能 | 适用场景 |
|------|------------|--------|--------|----------|
| Collections.synchronizedList | synchronized 全局锁 | 差 | 差 | 不推荐 |
| CopyOnWriteArrayList | ReentrantLock + 写时复制 | 极好 | 差 | 读多写少 |
| ConcurrentLinkedQueue | CAS 无锁 | 好 | 好 | 队列语义 |

**CopyOnWriteArrayList** 适合读多写少的场景，如配置列表、监听器列表。

**ConcurrentLinkedQueue** 是无界的，不适合需要阻塞操作的场景。

## 并发 Queue 对比

| 容器 | 有界性 | 阻塞支持 | 内部结构 | 典型用途 |
|------|--------|----------|----------|----------|
| ConcurrentLinkedQueue | 无界 | 否 | 链表 CAS | 高性能无阻塞 |
| LinkedBlockingQueue | 可选有界 | 是 | 链表双锁 | 生产者-消费者 |
| ArrayBlockingQueue | 有界 | 是 | 数组单锁 | 固定容量队列 |
| PriorityBlockingQueue | 无界 | 是 | 堆 | 优先级任务 |
| DelayQueue | 无界 | 是 | 优先队列 | 定时任务 |

**LinkedBlockingQueue** 是最常用的选择，可选有界、支持阻塞，适合大多数生产者-消费者场景。

**ArrayBlockingQueue** 在需要精确控制内存使用时使用，吞吐量略低于 LinkedBlockingQueue。

**PriorityBlockingQueue** 用于需要按优先级处理任务的场景。

**DelayQueue** 用于定时任务调度。

## 并发 Deque 对比

| 容器 | 有界性 | 阻塞支持 | 适用场景 |
|------|--------|----------|----------|
| ConcurrentLinkedDeque | 无界 | 否 | 高性能双端队列 |
| LinkedBlockingDeque | 可选有界 | 是 | 工作窃取模式 |

**ConcurrentLinkedDeque** 用于需要高效双端操作的场景。

**LinkedBlockingDeque** 可用于实现工作窃取（Work-Stealing）模式。

## 选型决策树

```
需要键值对存储？
├── 不需要排序 → ConcurrentHashMap
└── 需要排序 → ConcurrentSkipListMap

需要列表存储？
├── 读多写少 → CopyOnWriteArrayList
└── 通用场景 → ConcurrentLinkedQueue（队列语义）

需要队列？
├── 需要阻塞 → 生产者-消费者
│   ├── 需要优先级 → PriorityBlockingQueue
│   ├── 需要延迟 → DelayQueue
│   ├── 固定容量 → ArrayBlockingQueue
│   └── 可选有界 → LinkedBlockingQueue
└── 不需要阻塞 → ConcurrentLinkedQueue

需要双端队列？
├── 需要阻塞 → LinkedBlockingDeque
└── 高性能 → ConcurrentLinkedDeque
```

## 性能测试参考

在 8 核机器上，1 百万次并发操作的测试结果（仅供参考）：

| 操作 | ConcurrentHashMap | Hashtable | synchronizedMap |
|------|------------------|-----------|----------------|
| 纯读 | ~50ms | ~2000ms | ~1800ms |
| 纯写 | ~150ms | ~3000ms | ~2800ms |
| 混合读写 | ~120ms | ~2500ms | ~2200ms |

ConcurrentHashMap 的性能是 Hashtable 和 synchronizedMap 的 10-20 倍。

## 常见误区

**误区一：使用了并发容器就一定线程安全**

并发容器本身是线程安全的，但如果业务逻辑中存在复合操作（如 check-then-act），仍需要额外同步。

```java
// 不安全的复合操作
ConcurrentHashMap<String, Integer> map = new ConcurrentHashMap<>();
if (map.containsKey("counter")) {
    map.put("counter", map.get("counter") + 1); // 不是原子操作
}

// 正确的做法：使用原子操作
map.compute("counter", (k, v) -> v == null ? 1 : v + 1);
```

**误区二：CopyOnWriteArrayList 适合大量写操作**

CopyOnWriteArrayList 的写操作需要复制整个数组，写多时性能极差。

**误区三：无界队列不会满**

ConcurrentLinkedQueue、PriorityBlockingQueue（无界模式）等无界队列可能导致 OOM，需要监控队列大小或使用有界队列。

## 最佳实践

1. **优先使用专用并发容器**：如 ConcurrentHashMap、ConcurrentLinkedQueue
2. **选择合适的队列类型**：生产者-消费者用 BlockingQueue，直接传递用 SynchronousQueue
3. **监控队列大小**：防止无界队列导致 OOM
4. **避免复合操作的竞态条件**：使用并发容器提供的原子方法
5. **根据实际负载测试**：不同场景下的性能表现可能不同

理解各并发容器的特性和适用场景，你就能在多线程编程中做出正确的选择。
