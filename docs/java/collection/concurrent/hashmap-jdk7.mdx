# ConcurrentHashMap JDK 7：Segment 分段锁

ConcurrentHashMap 是 Java 并发容器中最核心的组件。它的设计经历了从 JDK 7 的分段锁到 JDK 8 的 CAS + synchronized 的重大演进。理解这两个版本的设计，能帮助我们深入理解并发编程的核心问题——锁竞争。

## 为什么 HashMap 不是线程安全的

在讨论 ConcurrentHashMap 之前，先理解普通 HashMap 为什么不是线程安全的。

HashMap 的非线程安全体现在多个方面：

**并发 put 导致数据覆盖**：两个线程同时 put 到同一个位置，后面的会覆盖前面的。

**并发扩容导致死循环**：JDK 7 中，头插法加并发扩容会导致环形链表。

**可见性问题**：一个线程对 HashMap 的修改，可能对其他线程不可见。

Java 的早期解决方案是使用 `Collections.synchronizedMap()` 包装，但这只是简单地在每个方法上加锁，性能很差。

## Segment 分段锁的设计

JDK 7 的 ConcurrentHashMap 采用了"分段锁"的设计思想：

```java
public class ConcurrentHashMap<K, V> extends AbstractMap<K, V> 
    implements ConcurrentMap<K, V>, Serializable {
    
    final Segment<K,V>[] segments;
    
    final class Segment<K,V> extends ReentrantLock 
        implements Serializable {
        
        transient volatile HashEntry<K,V>[] table;
        transient int count;
        transient int modCount;
        transient int threshold;
        final float loadFactor;
    }
    
    static final class HashEntry<K,V> {
        final int hash;
        final K key;
        volatile V value;
        volatile HashEntry<K,V> next;
    }
}
```

ConcurrentHashMap 将数据分成多个 Segment（段），每个 Segment 持有一个独立的 HashMap。默认情况下，有 16 个 Segment，可以同时支持 16 个线程并发写入（理论上）。

## 寻址与锁的获取

```java
private int hash(Object key) {
    int h = hashSeed;
    if ((h != 0) || (key != null)) {
        h ^= key.hashCode();
        h ^= (h >>> 20) ^ (h >>> 12);
        return h ^ (h >>> 7) ^ (h >>> 4);
    }
    return 0;
}

private Segment<K,V> segmentFor(int hash) {
    return segments[(hash >>> segmentShift) & segmentMask];
}

public V put(K key, V value) {
    Segment<K,V> s;
    if (value == null)
        throw new NullPointerException();
    int hash = hash(key);
    int j = (hash >>> segmentShift) & segmentMask;
    s = (Segment<K,V>)UNSAFE.getObjectVolatile(segments, 
        (long)(j << SSHIFT) + SBASE);
    if (s != null && s.count != 0) {
        s.lock();
        try {
            HashEntry<K,V> e = s.table[j];
            // ... 遍历或插入
        } finally {
            s.unlock();
        }
    }
}
```

首先通过 hash 计算应该访问哪个 Segment（`j`），然后只锁定这个 Segment。不同 Segment 之间互不干扰，可以并发操作。

## ReentrantLock 的使用

Segment 继承自 `ReentrantLock`，使用可重入锁：

```java
static final class Segment<K,V> extends ReentrantLock 
    implements Serializable {
    
    V put(K key, int hash, V value, boolean onlyIfAbsent) {
        lock();
        try {
            // ...
        } finally {
            unlock();
        }
    }
    
    V remove(Object key, int hash, Object value) {
        lock();
        try {
            // ...
        } finally {
            unlock();
        }
    }
}
```

使用 ReentrantLock 而不是 synchronized，有几个好处：

- 可以响应中断
- 支持超时
- 支持非阻塞获取锁（tryLock）

## 并发度的控制

Segment 的数量（并发度）可以通过构造函数指定：

```java
public ConcurrentHashMap(int initialCapacity, float loadFactor, int concurrencyLevel) {
    if (loadFactor <= 0 || initialCapacity < 0 || concurrencyLevel <= 0)
        throw new IllegalArgumentException();
    
    if (concurrencyLevel > MAX_SEGMENTS)
        concurrencyLevel = MAX_SEGMENTS;
    
    int sshift = 0;
    int ssize = 1;
    while (ssize < concurrencyLevel) {
        ++sshift;
        ssize <<= 1;
    }
    
    this.segmentShift = 32 - sshift;
    this.segmentMask = ssize - 1;
    this.segments = createSegment(ssize);
}
```

默认情况下，`concurrencyLevel = 16`，创建 16 个 Segment。Segment 数量必须是 2 的幂次方。

## get 操作的无锁化

ConcurrentHashMap 的 get 操作完全不需要加锁：

```java
public V get(Object key) {
    Segment<K,V> s;
    HashEntry<K,V>[] tab;
    int h = hash(key);
    long u = (((h >>> segmentShift) & segmentMask) << SSHIFT) + SBASE;
    if ((s = (Segment<K,V>)UNSAFE.getObjectVolatile(tab = segments, u)) != null &&
        (tab = s.table) != null) {
        for (HashEntry<K,V> e = (HashEntry<K,V>) 
                 UNSAFE.getObjectVolatile(tab, 
                     (((long)(((tab.length - 1) & h)) << TSIZE_SHIFT) + TBASE));
             e != null; e = e.next) {
            if (e.hash == h && key.equals(e.key))
                return e.value;
        }
    }
    return null;
}
```

使用 UNSAFE.getObjectVolatile() 保证可见性，遍历链表查找。由于 HashEntry 的 value 和 next 都是 volatile 的，读取线程总能获取到最新写入的值。

## size 方法的挑战

计算 ConcurrentHashMap 的 size 是一个棘手的问题。简单地在每个 Segment 上加锁然后求 count 会很慢，而且可能与其他操作冲突。

JDK 7 的实现使用了两轮不加锁的尝试：

```java
public int size() {
    long sum = 0;
    long check = 0;
    int[] segments = this.segments;
    for (int k = 0; k < retries; ++k) {
        check = 0;
        sum = 0;
        for (int i = 0; i < segments.length; ++i) {
            sum += segments[i].count;
            if (segments[i].count != check) {
                check = segments[i].count;
            }
        }
        if (sum == check) {
            break;
        }
    }
    return (int) sum;
}
```

第一轮读取所有 Segment 的 count，如果读取过程中有 Segment 被修改，count 会变化。继续重试直到两次读取的结果一致（没有修改发生）。

## 分段锁的局限性

JDK 7 的分段锁设计虽然比全局锁好很多，但仍然有局限性：

**热点问题**：如果所有操作都集中在同一个 Segment，这个 Segment 仍然是热点。

**内存开销**：每个 Segment 都有独立的锁、计数器、阈值等，16 个 Segment 的开销比单个 HashMap 大。

**复杂度**：Segment 数组增加了实现复杂度，也增加了调试难度。

JDK 8 放弃了分段锁，转而使用更细粒度的锁（桶锁）配合 CAS，实现了更高的并发度。

理解 JDK 7 ConcurrentHashMap 的分段锁设计，你就能理解为什么需要更细粒度的并发控制。
