# CopyOnWriteArrayList 原理与适用场景

在并发编程中，有一个经典的问题：读多写少场景下的线程安全集合应该如何设计？如果使用 synchronized 或锁来保护 ArrayList，读操作会阻塞等待写操作完成；如果不加锁，读操作可能读到不一致的数据。CopyOnWriteArrayList 提供了一个优雅的解决方案——写操作时复制整个数组，读操作完全无锁。

## 核心思想：读写分离

CopyOnWriteArrayList 的核心思想是"写入时复制"（Copy-On-Write）：

- **读操作**：完全无锁，直接访问数组
- **写操作**：先复制整个数组，在副本上修改，然后原子地替换引用

```java
public class CopyOnWriteArrayList<E> implements List<E>, RandomAccess, Cloneable, Serializable {
    
    private transient volatile Object[] array;
    final transient ReentrantLock lock = new ReentrantLock();
    
    final Object[] getArray() {
        return array;
    }
    
    private void setArray(Object[] a) {
        array = a;
    }
}
```

注意两个关键点：

1. **数组是 volatile 的**：保证了写入后对其他线程立即可见
2. **有 ReentrantLock**：虽然读不需要锁，但写需要锁来保证线程安全

## 读写操作的实现

**get 操作：完全无锁**

```java
public E get(int index) {
    return (E) getArray()[index];
}

private E getArray() {
    return array;
}
```

读操作直接返回数组指定位置的元素，没有任何锁或同步。由于数组引用是 volatile 的，读线程总能看到最新的数组内容。

**set 操作：复制 + 替换**

```java
public E set(int index, E element) {
    final ReentrantLock lock = this.lock;
    lock.lock();
    try {
        Object[] elements = getArray();
        E oldValue = (E) elements[index];
        if (oldValue != element) {
            int len = elements.length;
            Object[] newElements = Arrays.copyOf(elements, len);
            newElements[index] = element;
            setArray(newElements);
        } else {
            setArray(elements);
        }
        return oldValue;
    } finally {
        lock.unlock();
    }
}
```

写操作需要：

1. 获取锁
2. 复制整个数组（`Arrays.copyOf`）
3. 在副本上修改
4. 原子地替换引用（`setArray`）
5. 释放锁

## add 操作的实现

```java
public boolean add(E e) {
    final ReentrantLock lock = this.lock;
    lock.lock();
    try {
        Object[] elements = getArray();
        int len = elements.length;
        Object[] newElements = Arrays.copyOf(elements, len + 1);
        newElements[len] = e;
        setArray(newElements);
        return true;
    } finally {
        lock.unlock();
    }
}

public void add(int index, E element) {
    final ReentrantLock lock = this.lock;
    lock.lock();
    try {
        Object[] elements = getArray();
        int len = elements.length;
        if (index > len || index < 0)
            throw new IndexOutOfBoundsException("Index: " + index + ", Size: " + len);
        Object[] newElements = new Object[len + 1];
        System.arraycopy(elements, 0, newElements, 0, index);
        System.arraycopy(elements, index, newElements, index + 1, len - index);
        newElements[index] = element;
        setArray(newElements);
    } finally {
        lock.unlock();
    }
}
```

每次 add 都会复制整个数组，时间复杂度是 O(n)。

## 迭代器：快照迭代器

CopyOnWriteArrayList 的迭代器是一个"快照"：

```java
public Iterator<E> iterator() {
    Object[] elements = getArray();
    return new COWIterator<E>(elements, elements.length);
}

private static class COWIterator<E> implements ListIterator<E> {
    private final Object[] snapshot;
    private int cursor;
    
    COWIterator(Object[] array, int length) {
        snapshot = array;
        cursor = length;
    }
    
    public E next() {
        return (E) snapshot[cursor++];
    }
}
```

迭代器持有创建时数组的快照。即使后续有其他线程修改了列表，迭代器仍然遍历原始数组。这意味着：

- **迭代期间不会抛出 ConcurrentModificationException**
- **迭代器不反映后续的修改**
- **迭代器不支持 remove、set、add 等修改操作**（会抛出 UnsupportedOperationException）

## 适用场景

CopyOnWriteArrayList 的特性决定了它最适合的场景：

**场景一：读多写少的并发列表**

```java
CopyOnWriteArrayList<Config> configs = new CopyOnWriteArrayList<>();

// 大量读取操作
public Config getConfig(String name) {
    for (Config c : configs) {
        if (c.getName().equals(name)) {
            return c;
        }
    }
    return null;
}

// 偶尔的更新操作
public void updateConfig(Config newConfig) {
    configs.add(newConfig);
}
```

配置列表通常读远多于写，非常适合 CopyOnWriteArrayList。

**场景二：事件监听器列表**

```java
CopyOnWriteArrayList<EventListener> listeners = new CopyOnWriteArrayList<>();

public void addListener(EventListener listener) {
    listeners.add(listener);
}

public void removeListener(EventListener listener) {
    listeners.remove(listener);
}

// 在事件触发时遍历
public void fireEvent(Event e) {
    for (EventListener l : listeners) {
        l.onEvent(e);
    }
}
```

事件监听器列表在遍历时不需要加锁，不会抛出 ConcurrentModificationException。

**场景三：白名单/黑名单**

```java
CopyOnWriteArrayList<String> whitelist = new CopyOnWriteArrayList<>();

public boolean isAllowed(String item) {
    return whitelist.contains(item);
}

// 管理员偶尔更新白名单
public void updateWhitelist(List<String> newList) {
    whitelist.clear();
    whitelist.addAll(newList);
}
```

## 不适合的场景

CopyOnWriteArrayList 不适合以下场景：

**写操作频繁**：每次写入都复制整个数组，写多时性能极差。

**数据量较大**：复制大数组会消耗大量内存，可能导致 OOM。

**需要实时数据**：迭代器持有的是快照，可能读到过期数据。

## 性能考量

CopyOnWriteArrayList 的性能特点：

| 操作 | 时间复杂度 | 说明 |
|------|-----------|------|
| get | O(1) | 完全无锁 |
| set/add/remove | O(n) | 需要复制数组 |
| iterator | O(1) 创建 | 持有快照 |
| 迭代遍历 | O(n) | 遍历快照 |

对于 1000 个元素的列表，每次写操作需要复制 1000 个引用（8KB 左右）。如果每秒有 1000 次写操作，会消耗 8MB 内存用于复制操作。

## 与 Collections.synchronizedList 的对比

| 特性 | CopyOnWriteArrayList | Collections.synchronizedList |
|------|----------------------|------------------------------|
| 读性能 | O(1)，无锁 | O(1)，需获取锁 |
| 写性能 | O(n)，复制数组 | O(1)，只需加锁 |
| 迭代器 | 快照，不会 CME | 快速失败，会 CME |
| 内存占用 | 较高（每次复制） | 较低 |

理解 CopyOnWriteArrayList 的读写分离设计，你就能在合适的场景（读多写少）选择这个优雅的并发工具。
