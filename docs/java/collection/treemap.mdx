# TreeMap 与红黑树

小李面试字节时，被问到："TreeMap 和 HashMap 有什么区别？"

小李说："TreeMap 是树结构，HashMap 是哈希表。"

面试官追问："TreeMap 用的是什么树？"

小李说："红黑树。"

面试官点点头："红黑树是怎么保证平衡的？为什么不需要严格的平衡？"

小李支支吾吾："就是...通过旋转和变色？"

面试官："那你说说，什么情况下会左旋，什么情况下会右旋？"

小张彻底卡住了。

【面试官心理】

这道题我用来筛选 P6 候选人。能说出 TreeMap 用红黑树的占 70%，能讲出左旋右旋的占 30%，能解释红黑树为什么选择"近似平衡"而不是严格平衡的占 10%。TreeMap 不像 HashMap 那么高频，但它是 Java 中唯一实现了 NavigableMap 的类，支持有序操作，是面试中的加分项。

## 一、为什么需要 TreeMap 🔴

### 1.1 HashMap 的局限

HashMap 只能通过 key 精确查找，无法支持：
- 按 key 排序遍历
- 查找比给定 key 小/大的最近元素
- 范围查询（如：查找所有年龄在 20`~`30 之间的用户）

```java
Map<String, Integer> map = new HashMap<>();
map.put("zhang", 30);
map.put("wang", 25);
map.put("li", 28);

// HashMap：遍历顺序是哈希顺序，不可预测
for (String key : map.keySet()) {
    System.out.println(key);  // 顺序不确定
}

// TreeMap：遍历顺序是 key 的自然顺序或比较器顺序
Map<String, Integer> treeMap = new TreeMap<>(map);
for (String key : treeMap.keySet()) {
    System.out.println(key);  // li, wang, zhang（字典序）
}
```

### 1.2 红黑树的引入

TreeMap 使用**红黑树**（Red-Black Tree）作为底层数据结构。红黑树是一种自平衡的二叉搜索树，通过节点着色（红/黑）和旋转操作，保证最长路径不超过最短路径的 2 倍。

```java
public class TreeMap<K, V> extends AbstractMap<K, V>
        implements NavigableMap<K, V>, Cloneable, java.io.Serializable {

    // 红黑树的根节点
    private transient Entry<K, V> root;

    // 树中元素个数
    private transient int size = 0;

    // 红黑树约束：根节点必须是黑色
    private static final boolean RED = false;
    private static final boolean BLACK = true;

    // 节点定义
    static final class Entry<K, V> implements Map.Entry<K, V> {
        K key;
        V value;
        Entry<K, V> left;
        Entry<K, V> right;
        Entry<K, V> parent;
        boolean color = BLACK;  // 默认为黑色

        Entry(K key, V value, Entry<K, V> parent) {
            this.key = key;
            this.value = value;
            this.parent = parent;
        }
    }
}
```

## 二、红黑树的平衡机制 🔴

### 2.1 红黑树的五条规则

```
1. 每个节点要么是红色，要么是黑色
2. 根节点必须是黑色
3. 所有叶子节点（NIL/空节点）必须是黑色
4. 红色节点的子节点必须都是黑色（不能有两个连续的红色）
5. 从任一节点到其每个叶子的路径上，黑高相同
```

这五条规则保证了红黑树的平衡：最长路径不超过最短路径的 2 倍。

### 2.2 左旋和右旋

红黑树的平衡通过**旋转**来维护：

```java
// 左旋：以 p 为支点，其右子节点 r 成为新的父节点
//        p                    r
//       / \        →        / \
//      a   r              p   c
//         / \            / \
//        b   c          a   b
private void rotateLeft(Entry<K, V> p) {
    if (p != null) {
        Entry<K, V> r = p.right;
        p.right = r.left;
        if (r.left != null)
            r.left.parent = p;
        r.parent = p.parent;
        if (p.parent == null)
            root = r;
        else if (p.parent.left == p)
            p.parent.left = r;
        else
            p.parent.right = r;
        r.left = p;
        p.parent = r;
    }
}

// 右旋：以 p 为支点，其左子节点 l 成为新的父节点
//        p                    l
//       / \        →        / \
//      l   c              a   p
//     / \                    / \
//    a   b                  b   c
private void rotateRight(Entry<K, V> p) {
    if (p != null) {
        Entry<K, V> l = p.left;
        p.left = l.right;
        if (l.right != null)
            l.right.parent = p;
        l.parent = p.parent;
        if (p.parent == null)
            root = l;
        else if (p.parent.right == p)
            p.parent.right = l;
        else
            p.parent.left = l;
        l.right = p;
        p.parent = l;
    }
}
```

### 2.3 插入修复：fixAfterInsertion

红黑树插入新节点时，如果破坏了平衡，需要修复：

```java
private void fixAfterInsertion(Entry<K, V> x) {
    x.color = RED;  // 新节点默认红色

    while (x != null && x != root && x.parent.color == RED) {
        if (parentOf(x) == leftOf(parentOf(parentOf(x)))) {
            // 父节点是祖父节点的左子节点
            Entry<K, V> y = rightOf(parentOf(parentOf(x)));
            if (colorOf(y) == RED) {
                // 情况1：叔叔节点是红色 → 重新着色
                setColor(parentOf(x), BLACK);
                setColor(y, BLACK);
                setColor(parentOf(parentOf(x)), RED);
                x = parentOf(parentOf(x));
            } else {
                // 情况2：叔叔节点是黑色，且当前节点是右子节点 → 左旋
                if (x == rightOf(parentOf(x))) {
                    x = parentOf(x);
                    rotateLeft(x);
                }
                // 情况3：叔叔节点是黑色，且当前节点是左子节点 → 右旋 + 重新着色
                setColor(parentOf(x), BLACK);
                setColor(parentOf(parentOf(x)), RED);
                rotateRight(parentOf(parentOf(x)));
            }
        } else {
            // 对称情况：父节点是祖父节点的右子节点
            // ...
        }
    }
    root.color = BLACK;  // 保证根节点是黑色
}
```

:::tip 💡
红黑树选择"近似平衡"（最长路径 `<=` 2倍最短路径）而不是 AVL 树的严格平衡（左右子树高度差 `<=` 1），是因为：**近似平衡的插入/删除维护成本更低**。红黑树的插入最多旋转 2 次，删除最多旋转 3 次；而 AVL 树在插入/删除时可能需要多次旋转。对于需要频繁插入/删除的场景，红黑树的性能更稳定。
:::

## 三、TreeMap 核心操作 🔴

### 3.1 put 方法

```java
public V put(K key, V value) {
    Entry<K, V> t = root;
    if (t == null) {
        // 第一个节点，成为根节点
        root = new Entry<>(key, value, null);
        size = 1;
        modCount++;
        return null;
    }

    int cmp;
    Entry<K, V> parent;
    Comparator<? super K> cpr = comparator;
    if (cpr != null) {
        // 使用自定义比较器
        do {
            parent = t;
            cmp = cpr.compare(key, t.key);
            if (cmp < 0)
                t = t.left;
            else if (cmp > 0)
                t = t.right;
            else
                return t.setValue(value);  // key 相等，覆盖 value
        } while (t != null);
    } else {
        // 使用 key 的自然顺序（必须实现 Comparable）
        if (key == null)
            throw new NullPointerException();
        Comparable<? super K> k = (Comparable<? super K>) key;
        do {
            parent = t;
            cmp = k.compareTo(t.key);
            if (cmp < 0)
                t = t.left;
            else if (cmp > 0)
                t = t.right;
            else
                return t.setValue(value);
        } while (t != null);
    }

    // 创建新节点并插入
    Entry<K, V> e = new Entry<>(key, value, parent);
    if (cmp < 0)
        parent.left = e;
    else
        parent.right = e;

    fixAfterInsertion(e);  // 修复红黑树平衡
    size++;
    modCount++;
    return null;
}
```

### 3.2 NavigableMap 特色方法

TreeMap 实现了 `NavigableMap`，提供了丰富的有序操作：

```java
NavigableMap<String, Integer> map = new TreeMap<>();

map.put("a", 1);
map.put("c", 3);
map.put("e", 5);
map.put("g", 7);

// 查找小于 key 的最大元素（floor）
map.floorEntry("d");  // 返回 c=3

// 查找大于 key 的最小元素（ceiling）
map.ceilingEntry("d");  // 返回 e=5

// 查找小于 key 的最大元素（lower）
map.lowerEntry("d");  // 返回 c=3（不包括等于的情况）

// 查找大于 key 的最小元素（higher）
map.higherEntry("d");  // 返回 e=5（不包括等于的情况）

// 降序 Map
NavigableMap<String, Integer> descMap = map.descendingMap();  // g=7, e=5, c=3, a=1

// 子 Map
map.subMap("b", "f");  // 返回 c=3, e=5（不包括 f）
map.subMap("b", true, "f", true);  // 包括 b 和 f
```

## 四、Comparable vs Comparator 🟡

### 4.1 自然顺序（Comparable）

```java
public class User implements Comparable<User> {
    String name;
    int age;

    @Override
    public int compareTo(User o) {
        // 按年龄升序
        return this.age - o.age;
    }
}

TreeMap<User, String> map = new TreeMap<>();
map.put(new User("张三", 25), "data");
```

### 4.2 比较器顺序（Comparator）

```java
TreeMap<User, String> map = new TreeMap<>(new Comparator<User>() {
    @Override
    public int compare(User u1, User u2) {
        return u1.age - u2.age;
    }
});

// JDK8 简化
TreeMap<User, String> map = new TreeMap<>(Comparator.comparingInt(u -> u.age));
```

:::warning ⚠️
使用自然顺序时，key 必须实现 `Comparable` 接口；使用比较器时，key 不需要实现 `Comparable`。**两者不要混用**，否则可能导致排序不一致，甚至抛出 `ClassCastException`。
:::

## 五、常见翻车现场 🟡

### ❌ 翻车点一：key 没有实现 Comparable

```java
// ❌ 错误：没有指定 Comparator，key 也没有实现 Comparable
TreeMap<Point, String> map = new TreeMap<>();
map.put(new Point(1, 2), "data");  // ClassCastException!

class Point {
    int x, y;
    // 没有实现 Comparable
}

// ✅ 正确：提供 Comparator
TreeMap<Point, String> map = new TreeMap<>(Comparator.comparingInt(p -> p.x));
```

### ❌ 翻车点二：混用自然顺序和比较器

```java
class User implements Comparable<User> {
    int age;
    @Override
    public int compareTo(User o) {
        return this.age - o.age;
    }
}

// ❌ 错误：同时指定了 Comparator 和自然顺序，可能导致不一致
TreeMap<User, String> map = new TreeMap<>(Comparator.comparing(u -> u.name));
// 比较器按 name 排序，自然顺序按 age 排序，行为不一致
```

### ❌ 翻车点三：null 作为 key

```java
TreeMap<String, Integer> map = new TreeMap<>();
map.put(null, 1);  // 正常，因为 String 实现了 Comparable

TreeMap<Integer, Integer> map2 = new TreeMap<>();
map2.put(null, 1);  // NullPointerException！因为 Integer.compareTo(null) 会 NPE
```

## 六、生产避坑指南 🟢

### 6.1 场景：排行榜系统

```java
public class Leaderboard {
    private final TreeMap<Integer, List<String>> rankMap;
    private final Map<String, Integer> playerScores;

    public Leaderboard() {
        this.rankMap = new TreeMap<>(Collections.reverseOrder());  // 降序
        this.playerScores = new HashMap<>();
    }

    public void updateScore(String player, int newScore) {
        // 移除旧分数
        Integer oldScore = playerScores.get(player);
        if (oldScore != null) {
            rankMap.get(oldScore).remove(player);
        }

        // 添加新分数
        playerScores.put(player, newScore);
        rankMap.computeIfAbsent(newScore, k -> new ArrayList<>()).add(player);
    }

    // 获取 Top N
    public List<String> getTopN(int n) {
        List<String> result = new ArrayList<>();
        for (Map.Entry<Integer, List<String>> entry : rankMap.entrySet()) {
            result.addAll(entry.getValue());
            if (result.size() >= n) break;
        }
        return result;
    }
}
```

## 七、TreeMap vs HashMap 🟢

| 维度 | TreeMap | HashMap |
| --- | --- | --- |
| 底层结构 | 红黑树 | 哈希表（数组+链表/红黑树） |
| 查找复杂度 | O`(`log n`)` | O`(`1`)` 均摊 |
| 排序 | 有序（key 自然顺序或比较器顺序） | 无序 |
| 范围查询 | 支持 | 不支持 |
| null 作为 key | 需要比较器或实现 Comparable | 允许一个 null key |
| 性能 | 插入/查找 O`(`log n`)`，稳定 | O`(`1`)`，但有扩容开销 |
| 适用场景 | 需要排序、有序遍历、范围查询 | 纯粹的 key-value 查找 |

【面试官心理】

问到 TreeMap 的候选人，通常是对集合框架有进阶追求的。这道题的关键不在于背出左旋右旋的代码，而在于理解红黑树为什么选择近似平衡而不是严格平衡。AVL 树的查询性能略高于红黑树（因为更平衡），但插入/删除成本更高。TreeMap 选择红黑树，是因为在大部分场景下，插入/删除的频率高于查询，而红黑树在动态场景下的性能更稳定。
