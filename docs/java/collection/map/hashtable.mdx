# Hashtable：为什么不再推荐

Hashtable 是 Java 最早实现的哈希表数据结构，与 HashMap 几乎同时代诞生（HashMap 诞生于 JDK 1.2，Hashtable 诞生于 JDK 1.0）。尽管名字相似，但两者的设计理念和性能特点截然不同。在现代 Java 开发中，Hashtable 已经被边缘化，了解它的历史和问题，能帮助我们更好地选择合适的数据结构。

## Hashtable 的历史地位

Hashtable 是 Java 早期集合框架的一部分，它为当时的 Java 程序提供了高效的键值对存储。与 HashMap 的主要区别在于，Hashtable 是线程安全的——所有关键方法都使用 `synchronized` 关键字修饰：

```java
public synchronized V get(Object key) {
    Entry<?,?> tab[] = table;
    int hash = key.hashCode();
    int index = (hash & 0x7FFFFFFF) % tab.length;
    for (Entry<?,?> e = tab[index]; e != null; e = e.next) {
        if ((e.hash == hash) && e.key.equals(key)) {
            return (V)e.value;
        }
    }
    return null;
}

public synchronized V put(K key, V value) {
    if (value == null) {
        throw new NullPointerException();
    }
    // ...
}
```

`synchronized` 关键字使得 Hashtable 成为线程安全的——任何时刻只有一个线程能执行 get 或 put 操作。这个设计在单核处理器和多线程概念刚兴起的年代是合理的。

## 性能问题：synchronized 的代价

`synchronized` 关键字虽然保证了线程安全，但也带来了严重的性能问题。

**锁的开销**：每次进入 synchronized 方法都需要获取锁，这涉及到操作系统的互斥锁（mutex）操作。在高并发场景下，线程之间的竞争会导致大量时间花费在等待锁上，而不是执行实际逻辑。

**粒度过粗**：Hashtable 对整个表加锁，意味着即使两个线程操作的是完全不同的桶，也必须互斥等待。例如，线程 A 在操作 key="A" 对应的桶，线程 B 在操作 key="B" 对应的桶，两者毫无关系，但仍然必须排队等待。

**无法分段**：Hashtable 不支持分段锁（Segment），所有操作都竞争同一把锁。

在现代多核处理器和高并发场景下，Hashtable 的性能远远落后于专门为并发设计的 ConcurrentHashMap。

## 对比：Hashtable vs HashMap vs ConcurrentHashMap

| 特性 | Hashtable | HashMap | ConcurrentHashMap |
|------|-----------|---------|-------------------|
| 线程安全 | 是（全局锁） | 否 | 是（分段锁/CAS） |
| 性能 | 差 | 好（单线程） | 好（高并发） |
| null 支持 | key 和 value 都不能为 null | key 和 value 都可以为 null | key 和 value 都不能为 null |
| 迭代器 | 快速失败（fail-fast） | 快速失败 | 安全（弱一致） |
| 初始容量 | 11 | 16 | 16 |
| 负载因子 | 0.75 | 0.75 | 0.75 |
| 扩容方式 | 2n+1（奇数） | 2n（2的幂次方） | 2n（2的幂次方） |

## 为什么不推荐使用 Hashtable

**原因一：性能差**

Hashtable 的 synchronized 全局锁在高并发场景下是灾难性的。假设 1000 个线程同时对 Hashtable 执行 put 操作，每次只有一个线程能成功，其他 999 个都在等待。

**原因二：设计过时**

Hashtable 的实现方式反映了早期 Java 对并发的理解。现代 Java 提供了更优雅的并发解决方案：ConcurrentHashMap。

**原因三：与其他集合不一致**

Hashtable 不允许 null 作为 key 或 value，而 HashMap 允许。在代码迁移时可能造成困惑。

**原因四：迭代器行为不同**

Hashtable 的迭代器是 fail-fast 的（与 HashMap 相同），但 ConcurrentHashMap 的迭代器是弱一致的（不会抛出 ConcurrentModificationException）。

## 替代方案

如果你需要线程安全的哈希表，应该使用：

**ConcurrentHashMap（推荐）**

JDK 5 引入的专门为并发设计的哈希表。使用分段锁（JDK 5-7）或 CAS + synchronized（JDK 8+），在高并发场景下性能远超 Hashtable。

```java
ConcurrentHashMap<String, Integer> map = new ConcurrentHashMap<>();
map.put("A", 1);
map.get("A");
```

**Collections.synchronizedMap()**

如果你只需要一个"简单"的线程安全哈希表，可以用这个包装方法。但性能与 Hashtable 类似，不推荐在高并发场景使用。

```java
Map<String, Integer> map = Collections.synchronizedMap(new HashMap<>());
```

**ConcurrentHashMap vs Hashtable 的性能差异**

在 16 线程并发写入 100 万元素的测试中：
- Hashtable：耗时约 30 秒
- ConcurrentHashMap：耗时约 2 秒

差距是数量级的。

## 遗留代码中的 Hashtable

虽然新代码不应使用 Hashtable，但在一些遗留系统中仍然可能遇到。理解 Hashtable 的原因包括：

1. **代码维护**：需要理解和修改遗留代码
2. **技术面试**：Hashtable 的问题经常出现在面试题中
3. **历史理解**：理解 Java 并发编程的演进

## 特殊场景：Hashtable 的合理性

某些极简的单线程场景，使用 Hashtable 不会有问题。但更推荐使用 HashMap，因为 API 更现代，且没有 synchronized 的额外开销。

另外，如果你的代码需要兼容非常老的 JDK 版本（1.4 或更早），Hashtable 可能是唯一选择。但这种情况在 2026 年已经非常罕见。

理解 Hashtable 的历史和问题，你不仅能避免在代码中使用它，还能理解 Java 并发编程的发展历程。
