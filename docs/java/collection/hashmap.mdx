## 一、核心问题

### 🔴 HashMap 的 put 流程是怎样的？

**JDK8 put 流程源码**：

```java title="HashMap.java (JDK 8)"
public V put(K key, V value) {
    return putVal(hash(key), key, value, false, true);
}

/**
 * @param hash         key 的 hash 值
 * @param onlyIfAbsent 如果为 true，不覆盖已有值
 * @param evict        用于 LinkedHashMap 的回调
 */
final V putVal(int hash, K key, V value, boolean onlyIfAbsent,
               boolean evict) {
    Node<K,V>[] tab; Node<K,V> p; int n, i;
    
    // 步骤 1：table 为空或长度为 0，进行扩容
    if ((tab = table) == null || (n = tab.length) == 0)
        n = (tab = resize()).length;
    
    // 步骤 2：(n - 1) & hash 计算索引位置
    // 如果该位置为空，直接创建节点
    if ((p = tab[i = (n - 1) & hash]) == null)
        tab[i] = newNode(hash, key, value, null);
    
    // 步骤 3：该位置已有元素（哈希冲突）
    else {
        Node<K,V> e; K k;
        
        // 3a：第一个节点的 key 完全匹配
        if (p.hash == hash &&
            ((k = p.key) == key || (key != null && key.equals(k))))
            e = p;
        
        // 3b：第一个节点是红黑树节点，调用树化插入
        else if (p instanceof TreeNode)
            e = ((TreeNode<K,V>)p).putTreeVal(this, tab, hash, key, value);
        
        // 3c：遍历链表
        else {
            for (int binCount = 0; ; ++binCount) {
                if ((e = p.next) == null) {
                    // 尾插法添加新节点
                    p.next = newNode(hash, key, value, null);
                    
                    // 链表长度 >= 8，转红黑树
                    if (binCount >= TREEIFY_THRESHOLD - 1) // -1 for 1st
                        treeifyBin(tab, hash);
                    break;
                }
                
                // 链表中找到相同 key，覆盖 value
                if (e.hash == hash &&
                    ((k = e.key) == key || (key != null && key.equals(k))))
                    break;
                
                p = e;
            }
        }
        
        // 步骤 4：key 已存在，覆盖旧值
        if (e != null) {
            V oldValue = e.value;
            if (!onlyIfAbsent || oldValue == null)
                e.value = value;
            afterNodeAccess(e);
            return oldValue;
        }
    }
    
    // 步骤 5：修改计数，扩容检查
    ++modCount;
    if (++size > threshold)
        resize();
    afterNodeInsertion(evict);
    return null;
}
```

## 二、哈希算法

### 🔴 HashMap 如何计算 key 的 hash？

**JDK8 哈希算法源码**：

```java title="HashMap.java"
static final int hash(Object key) {
    int h;
    // key.hashCode() 触发方法分发
    // 高低位异或运算，使 hash 值分布更均匀
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}
```

### 为什么不直接用 hashCode？

```java
// hashCode() 返回 32 位 int 值
// 假设 hashCode = 0xABCD1234
// 直接取模：(n-1) & hashCode

// 当 n = 16 时：(n-1) = 0x0000000F
// 结果只取决于 hashCode 的低 4 位
// 0xABCD1234 & 0x0000000F = 0x4

// 高位信息完全丢失！
```

**解决方案**：让高位参与运算

```java
// JDK 采用：(h = key.hashCode()) ^ (h >>> 16)
// 将 hashCode 高 16 位右移到低 16 位，再与原 hashCode 异或
// 这样计算索引时，高位信息也能参与进来

// 示例：
// h = 0xABCD1234
// h >>> 16 = 0x0000ABCD
// h ^ (h >>> 16) = 0xABCD1234 ^ 0x0000ABCD = 0xABCD1234
// 最终：(0xABCD1234) & 15 分布更均匀
```

### 索引计算与容量

```java
// 计算桶数组索引
i = (n - 1) & hash

// 为什么容量必须是 2 的幂次？
// 因为 (n - 1) 的二进制全是 1，如 15 = 0b1111
// & 运算等价于取模，但速度更快
// n 为 2 的幂次时，hash % n == (n-1) & hash
```

:::tip 容量与 hash 分布
当 n = 16 时，`(n-1) & hash` 等价于 `hash % 16`，但位运算比取模快 10 倍。当 n = 15 时，`(n-1) = 0b1110`，某些 hash 值永远映射不到某些槽。
:::

## 三、扩容机制

### 🔴 HashMap 什么时候会扩容？

两个触发条件：

1. **元素数量超过阈值**（`threshold = capacity * loadFactor`）
2. **链表长度 >= 8 但容量 < 64 时**（调用 `treeifyBin`）

### 扩容源码解析

```java title="HashMap.java"
final Node<K,V>[] resize() {
    Node<K,V>[] oldTab = table;
    int oldCap = (oldTab == null) ? 0 : oldTab.length;
    int oldThr = threshold;
    int newCap, newThr = 0;
    
    // 原容量 > 0，计算新容量和新阈值
    if (oldCap > 0) {
        // 超过最大容量，不再扩容
        if (oldCap >= MAXIMUM_CAPACITY) {
            threshold = Integer.MAX_VALUE;
            return oldTab;
        }
        // 新容量 = 原容量 * 2
        else if ((newCap = oldCap << 1) < MAXIMUM_CAPACITY &&
                 oldCap >= DEFAULT_INITIAL_CAPACITY)
            newThr = oldThr << 1; // 新阈值也翻倍
    }
    
    // 原容量 = 0 但阈值 > 0（指定容量构造）
    else if (oldThr > 0)
        newCap = oldThr;
    
    // 原容量 = 0 且阈值 = 0（默认构造）
    else {
        newCap = DEFAULT_INITIAL_CAPACITY; // 16
        newThr = DEFAULT_LOAD_FACTOR * DEFAULT_INITIAL_CAPACITY; // 12
    }
    
    threshold = newThr;
    
    // 创建新数组
    @SuppressWarnings("unchecked")
    Node<K,V>[] newTab = (Node<K,V>[]) new Node[newCap];
    table = newTab;
    
    // 迁移元素
    if (oldTab != null) {
        for (int j = 0; j < oldCap; ++j) {
            Node<K,V> e;
            if ((e = oldTab[j]) != null) {
                oldTab[j] = null;
                
                // 只有一个节点，直接重新计算索引
                if (e.next == null)
                    newTab[e.hash & (newCap - 1)] = e;
                
                // 红黑树节点，拆分迁移
                else if (e instanceof TreeNode)
                    ((TreeNode<K,V>)e).split(this, newTab, j, oldCap);
                
                // 链表，拆分迁移（JDK8 优化）
                else {
                    Node<K,V> loHead = null, loTail = null;
                    Node<K,V> hiHead = null, hiTail = null;
                    Node<K,V> next;
                    do {
                        next = e.next;
                        // 关键：判断节点应该放到低位桶还是高位桶
                        if ((e.hash & oldCap) == 0) {
                            if (loTail == null)
                                loHead = e;
                            else
                                loTail.next = e;
                            loTail = e;
                        } else {
                            if (hiTail == null)
                                hiHead = e;
                            else
                                hiTail.next = e;
                            hiTail = e;
                        }
                    } while ((e = next) != null);
                    
                    // 低位桶保持原位置
                    if (loTail != null) {
                        loTail.next = null;
                        newTab[j] = loHead;
                    }
                    // 高位桶放到新位置（j + oldCap）
                    if (hiTail != null) {
                        hiTail.next = null;
                        newTab[j + oldCap] = hiHead;
                    }
                }
            }
        }
    }
    return newTab;
}
```

### JDK8 vs JDK7 扩容区别

```mermaid
flowchart LR
    subgraph JDK7
        J7_1[遍历旧数组] --> J7_2[遍历每个桶的链表]
        J7_2 --> J7_3[对每个节点重新计算 hash]
        J7_3 --> J7_4[插入新数组对应位置]
    end
    
    subgraph JDK8
        J8_1[遍历旧数组] --> J8_2[遍历每个桶]
        J8_2 --> J8_3{节点类型}
        J8_3 --> J8_4[单个节点<br/>用 (n-1)&hash]
        J8_3 --> J8_5[链表<br/>拆分高低位]
        J8_3 --> J8_6[红黑树<br/>拆分树节点]
    end
    
    style J8_4 fill:#90EE90
    style J8_5 fill:#90EE90
```

| 特性 | JDK7 | JDK8 |
|------|------|------|
| **rehash** | 每个节点重新计算 hash | 不重新计算，用 `(e.hash & oldCap)` 判断 |
| **时间复杂度** | O(n * k) | O(n)，k 是链表长度 |
| **原因** | JDK7 只用链表 | JDK8 利用容量为 2 的幂次特性 |

### 💡 JDK8 扩容优化原理

```java
// oldCap = 16 = 0b00010000
// newCap = 32 = 0b00100000

// 判断节点应该放在原位置还是原位置 + oldCap：
// (e.hash & oldCap) == 0 → 原位置（低位桶）
// (e.hash & oldCap) != 0 → 原位置 + oldCap（高位桶）

// 示例：
// hash1 = 0b00000101 = 5,  hash1 & 16 = 0  → 位置 5
// hash2 = 0b00010101 = 21, hash2 & 16 = 16 → 位置 5 + 16 = 21

// 只需要判断新增的那一位（bit）！
```

## 四、面试题精讲

### 🔴 第一层：HashMap put 流程

> **问题**：请描述 HashMap 的 put 流程。

> **参考答案**：
>
> 1. 计算 key 的 hash 值（`hash(key)`）
> 2. 计算数组索引：`i = (n - 1) & hash`
> 3. 如果桶为空，直接插入新节点
> 4. 如果桶不为空：
>    - key 相同 → 覆盖 value
>    - 红黑树节点 → 调用树化插入
>    - 链表节点 → 遍历查找或尾插新节点
> 5. 链表长度 >= 8 且容量 >= 64 时，链表转红黑树
> 6. 元素数量超过阈值时，触发扩容

### 🟡 第二层：为什么容量必须是 2 的幂次？

> **参考答案**：
>
> 1. **位运算优化**：`i = (n - 1) & hash` 比 `hash % n` 更快
> 2. **扩容优化**：JDK8 中 `e.hash & oldCap` 判断节点位置，无需重新计算 hash
> 3. **分布均匀**：`(n-1)` 二进制全为 1，能充分利用 hash 的所有位

> **追问**：如果容量不是 2 的幂次会怎样？
>
> - `(n-1) & hash` 不再等价于 `hash % n`
> - 某些 hash 值无法映射到有效位置
> - 扩容时需要重新计算所有节点的 hash

### 🟡 第三层：JDK7 和 JDK8 的 put 区别

| 特性 | JDK7 | JDK8 |
|------|------|------|
| **插入方式** | 头插法 | 尾插法 |
| **扩容 rehash** | 重新计算每个节点的 hash | 用 `& oldCap` 判断位置 |
| **链表→红黑树** | 无 | 有（JDK8 新增） |
| **并发问题** | 扩容时可能死循环 | 尾插法避免但仍有并发不安全 |

### ⚠️ 面试官挖坑点

| 陷阱 | 错误回答 | 正确回答 |
|------|---------|----------|
| 「JDK8 put 比 JDK7 快」 | 只因为用了红黑树 | 红黑树只在极端情况启用，主要优化是扩容时不 rehash |
| 「容量翻倍后 rehash」 | 重新计算 hash | 用 `e.hash & oldCap` 判断，不重新计算 |
| 「链表转红黑树阈值是 8」 | 忽略容量条件 | 必须同时满足：链表 >= 8 **且** 容量 >= 64 |
| 「HashMap 是线程安全的」 | 认为是安全集合 | HashMap 线程不安全，并发用 ConcurrentHashMap |

## 五、常见错误与陷阱

### ⚠️ 死循环问题（JDK7）

JDK7 头插法 + 扩容时可能导致死循环：

```java
// JDK7 扩容代码
void transfer(Entry[] newTable) {
    Entry[] src = table;
    int newCapacity = newTable.length;
    for (int j = 0; j < src.length; j++) {
        Entry<K,V> e = src[j];
        if (e != null) {
            src[j] = null;
            do {
                Entry<K,V> next = e.next;  // 记录下一个
                // 头插法：节点插入到新数组头部
                int i = indexFor(e.hash, newCapacity);
                e.next = newTable[i];
                newTable[i] = e;
                e = next;
            } while (e != null);
        }
    }
}
```

**死循环场景**：线程 A 和线程 B 同时扩容

1. 线程 A 遍历到 A1 → A2 → null，暂停
2. 线程 B 完成扩容，新链表顺序变成 A2 → A1
3. 线程 A 恢复，继续遍历
4. 遍历到 A2.next = A1，A1.next = A2 → 死循环

**JDK8 解决方案**：尾插法保证顺序，避免死循环，但仍然不是线程安全的。

### ⚠️ 数据丢失问题

并发 put 可能导致数据覆盖：

```java
// 线程 A 计算索引 i，暂停
// 线程 B 计算索引 i，完成插入
// 线程 A 继续，可能覆盖线程 B 的数据
```

## 六、核心源码手写版

### 简化版 HashMap put

```java title="SimpleHashMap.java"
public class SimpleHashMap<K, V> {
    private static final int DEFAULT_CAPACITY = 16;
    private static final float LOAD_FACTOR = 0.75f;
    
    private Node<K, V>[] table;
    private int size;
    private int threshold;
    
    @SuppressWarnings("unchecked")
    public SimpleHashMap() {
        table = new Node[DEFAULT_CAPACITY];
        threshold = (int) (DEFAULT_CAPACITY * LOAD_FACTOR);
    }
    
    static class Node<K, V> {
        final int hash;
        final K key;
        V value;
        Node<K, V> next;
        
        Node(int hash, K key, V value, Node<K, V> next) {
            this.hash = hash;
            this.key = key;
            this.value = value;
            this.next = next;
        }
    }
    
    private int hash(K key) {
        return key == null ? 0 : key.hashCode() ^ (key.hashCode() >>> 16);
    }
    
    public V put(K key, V value) {
        int hash = hash(key);
        int index = (table.length - 1) & hash;
        
        // 桶为空，直接插入
        if (table[index] == null) {
            table[index] = new Node<>(hash, key, value, null);
        } else {
            // 桶不为空，遍历链表
            Node<K, V> node = table[index];
            while (node != null) {
                if (node.hash == hash && 
                    (node.key == key || key.equals(node.key))) {
                    // key 相同，覆盖
                    V oldValue = node.value;
                    node.value = value;
                    return oldValue;
                }
                node = node.next;
            }
            // 尾插法添加新节点
            Node<K, V> last = table[index];
            while (last.next != null) {
                last = last.next;
            }
            last.next = new Node<>(hash, key, value, null);
        }
        
        size++;
        if (size > threshold) {
            resize();
        }
        return null;
    }
    
    @SuppressWarnings("unchecked")
    private void resize() {
        Node<K, V>[] oldTable = table;
        int oldCap = oldTable.length;
        int newCap = oldCap << 1; // 容量翻倍
        
        Node<K, V>[] newTable = new Node[newCap];
        threshold = (int) (newCap * LOAD_FACTOR);
        table = newTable;
        
        // JDK8 优化：不重新计算 hash，用 & oldCap 判断
        for (int i = 0; i < oldCap; i++) {
            Node<K, V> node = oldTable[i];
            if (node != null) {
                if (node.next == null) {
                    // 单个节点，直接计算新位置
                    newTable[(newCap - 1) & node.hash] = node;
                } else {
                    // 链表需要拆分
                    Node<K, V> loHead = null, loTail = null;
                    Node<K, V> hiHead = null, hiTail = null;
                    Node<K, V> next;
                    
                    do {
                        next = node.next;
                        // 判断是否在原位置
                        if ((node.hash & oldCap) == 0) {
                            if (loHead == null) loHead = node;
                            else loTail.next = node;
                            loTail = node;
                        } else {
                            if (hiHead == null) hiHead = node;
                            else hiTail.next = node;
                            hiTail = node;
                        }
                    } while ((node = next) != null);
                    
                    // 挂载到新数组
                    if (loHead != null) {
                        loTail.next = null;
                        newTable[i] = loHead;
                    }
                    if (hiHead != null) {
                        hiTail.next = null;
                        newTable[i + oldCap] = hiHead;
                    }
                }
            }
        }
    }
}
```

## 七、对比总结表

| 维度 | JDK7 | JDK8 |
|------|------|------|
| **数据结构** | 数组 + 链表 | 数组 + 链表 + 红黑树 |
| **插入方式** | 头插法 | 尾插法 |
| **扩容 rehash** | 重新计算 hash | `e.hash & oldCap` 判断 |
| **并发问题** | 扩容可能死循环 | 避免死循环但仍不安全 |
| **红黑树化** | 无 | 链表 >= 8 且容量 >= 64 |
| **查询效率** | O(n) 极端情况 | O(log n) 极端情况 |
| **极端场景** | 哈希冲突严重时性能差 | 有保底机制 |

## 八、扩展思考

> **追问**：如果让你实现一个自定义的 HashMap，你会考虑哪些优化？

1. **预扩容**：根据预期大小预分配容量，避免频繁扩容
2. **缓存 hashCode**：避免重复计算 hash
3. **延迟删除**：标记删除而不是立即删除，减少结构变动
4. **一致性 hash**：分布式场景下减少 rehash
5. **多种哈希函数**：解决哈希碰撞攻击（HashDoS）

> **追问**：HashMap 的负载因子为什么默认是 0.75？

空间与时间的权衡：
- 太小（0.5）：空间浪费，扩容频繁
- 太大（0.9）：空间利用率高，但哈希冲突增多
- 0.75 是经过数学计算的最优平衡点
