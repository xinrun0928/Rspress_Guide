# Collection 接口结构与分类

Collection 接口是 Java 集合框架的根基，它位于整个集合体系的最顶层（排除 `Iterable` 接口）。理解 Collection 的设计哲学，就等于握住了一把钥匙，可以打开整个集合框架的大门。

## 从 Iterable 到 Collection

在深入 Collection 之前，我们需要先理解它的父接口 `Iterable`。这个接口的出现，让"迭代"成为集合的统一行为标准。

```java
public interface Iterable<T> {
    Iterator<T> iterator();
    default void forEach(Consumer<? super T> action);
    default Spliterator<T> spliterator();
}
```

`Iterable` 的核心是 `iterator()` 方法，它返回一个 `Iterator` 对象。`Iterator` 是 Java 迭代器模式的核心，屏蔽了底层数据结构的差异——无论你是遍历数组、链表还是树，只需要调用 `hasNext()` 和 `next()` 这两个简单的方法。

JDK 8 为 `Iterable` 添加了 `forEach()` 默认方法，配合 Lambda 表达式可以让遍历更加简洁：

```java
List<String> names = Arrays.asList("Alice", "Bob", "Charlie");
names.forEach(name -> System.out.println(name));
```

`Spliterator` 则是 JDK 8 引入的"可分割迭代器"，专为并行遍历设计。它是 Stream 并行处理的基础。

## Collection 接口的设计哲学

`Collection` 接口直接继承了 `Iterable`，这意味着所有 Collection 子类都是可迭代的。`Collection` 接口定义了集合操作的基本规范：

```java
public interface Collection<E> extends Iterable<E> {
    // 查询操作
    int size();
    boolean isEmpty();
    boolean contains(Object o);
    Iterator<E> iterator();
    Object[] toArray();
    <T> T[] toArray(T[] a);
    
    // 修改操作
    boolean add(E e);
    boolean remove(Object o);
    
    // 批量操作
    boolean containsAll(Collection<?> c);
    boolean addAll(Collection<? extends E> c);
    boolean removeAll(Collection<?> c);
    boolean retainAll(Collection<?> c);
    void clear();
    
    // 查询操作（ JDK 8+）
    boolean removeIf(Predicate<? super E> filter);
    
    // Stream 相关
    Stream<E> stream();
    Stream<E> parallelStream();
    
    // 比较与哈希
    boolean equals(Object o);
    int hashCode();
}
```

这些方法看似简单，但每一组都承担着特定的功能：

**查询方法**（`size()`、`isEmpty()`、`contains()`）用于获取集合的基本信息。这些方法在各种实现中都有默认的时间复杂度，应该尽量避免不必要的调用。

**修改方法**（`add()`、`remove()`、`clear()`）是集合的灵魂。`add()` 方法的行为因具体实现而异——`List` 实现要求允许重复元素，`Set` 实现则需要保证元素唯一性。

**批量操作**（`addAll()`、`removeAll()`、`retainAll()`）是集合代数的体现。`retainAll()` 实现了"交集"的语义，`removeAll()` 实现了"差集"，而 `addAll()` 则对应"并集"。

## Collection 的三大分支

虽然 Collection 接口是一个统一的整体，但它分化出了三个主要分支，每个分支都有鲜明的特点：

### List：有序的序列

`List` 接口强调**有序性**和**可重复性**。它继承自 Collection，增加了基于索引的操作：

```java
public interface List<E> extends Collection<E> {
    E get(int index);
    E set(int index, E element);
    void add(int index, E element);
    E remove(int index);
    int indexOf(Object o);
    int lastIndexOf(Object o);
    ListIterator<E> listIterator();
    List<E> subList(int fromIndex, int toIndex);
}
```

`List` 的"有序"体现在两个层面：首先是插入顺序——元素按照你添加的顺序排列；其次是索引顺序——每个元素都有一个确定的位置，可以通过 `get(index)` 快速访问。

`ListIterator` 是 `Iterator` 的增强版本，支持双向遍历和元素修改。`subList()` 方法则提供了一个"视图"的机制，对子列表的操作会影响原列表（这是一个经常被误解的特性）。

### Set：唯一性的守护者

`Set` 接口的数学定义是"不包含重复元素的集合"。这听起来简单，但背后的实现却蕴含着精妙的设计。

```java
public interface Set<E> extends Collection<E> {
    // 没有添加新方法，所有约束都在实现中体现
}
```

有意思的是，`Set` 接口本身并没有添加任何新方法！它的独特性完全体现在实现类如何解释 `add()` 方法：`add()` 方法会首先检查元素是否已存在，只有在不存在时才真正添加。这种"添加时检查"的策略是 `Set` 实现的核心逻辑。

`Set` 的三个主要实现体现了三种不同的唯一性判断方式：`HashSet` 基于哈希码，`TreeSet` 基于自然顺序或比较器，`LinkedHashSet` 则在 `HashSet` 基础上维护插入顺序。

### Queue：排队的艺术

`Queue` 接口引入了**队列**这种经典数据结构，它遵循 FIFO（First In First Out）原则——最先进入的元素最先离开。

```java
public interface Queue<E> extends Collection<E> {
    boolean add(E e);        // 添加元素，队列满时抛异常
    boolean offer(E e);      // 添加元素，队列满时返回 false
    
    E remove();              // 移除并返回队首，队列空时抛异常
    E poll();                // 移除并返回队首，队列空时返回 null
    
    E element();             // 查看队首，队列空时抛异常
    E peek();                // 查看队首，队列空时返回 null
}
```

这里有一个微妙的设计：`add()`/`remove()`/`element()` 在操作失败时会抛出异常，而 `offer()`/`poll()`/`peek()` 则会返回特定值。这种设计让调用者可以选择处理方式——是想确保操作成功（用前者）还是想优雅地处理失败（用后者）。

`Deque` 接口是 `Queue` 的扩展，支持双端操作：

```java
public interface Deque<E> extends Queue<E> {
    void addFirst(E e);
    void addLast(E e);
    boolean offerFirst(E e);
    boolean offerLast(E e);
    
    E removeFirst();
    E removeLast();
    E pollFirst();
    E pollLast();
    
    E getFirst();
    E getLast();
    E peekFirst();
    E peekLast();
}
```

`Deque` 既可以作为普通队列使用，也可以作为栈使用（`push()`/`pop()`）。这使得 `ArrayDeque` 和 `LinkedList` 成为实现栈的推荐选择——比传统的 `Stack` 类更加高效。

## 实现类的选择艺术

面对如此多的实现类，如何做出正确的选择？这需要理解每种实现的时间和空间复杂度权衡。

| 实现类 | 随机访问 | 头部操作 | 尾部操作 | 查找 | 线程安全 |
|--------|----------|----------|----------|------|----------|
| ArrayList | O(1) | O(n) | O(1) amortized | O(n) | 否 |
| LinkedList | O(n) | O(1) | O(1) | O(n) | 否 |
| HashSet | - | - | O(1) avg | O(1) avg | 否 |
| TreeSet | - | - | O(log n) | O(log n) | 否 |
| LinkedHashSet | - | - | O(1) | O(1) avg | 否 |
| ArrayDeque | - | O(1) | O(1) amortized | O(n) | 否 |

选择的基本原则是：**根据你的主要操作模式选择实现**。

如果你需要通过索引快速访问元素，`ArrayList` 是首选。如果你需要频繁在头部插入删除，`LinkedList` 或 `ArrayDeque` 更合适。如果你需要保证元素唯一性并且不关心顺序，`HashSet` 最优。如果需要有序集合，`TreeSet` 是不二之选。

## 不可变集合的崛起

JDK 9+ 引入了`List.of()`、`Set.of()`、`Map.of()` 等工厂方法，可以便捷地创建不可变集合：

```java
List<String> immutable = List.of("a", "b", "c");
// immutable.add("d"); // UnsupportedOperationException
```

不可变集合在并发场景下特别有价值——它们天然线程安全，不需要任何同步措施。创建后就不会改变的集合特别适合作为常量集合或共享数据。

## 写在最后

Collection 接口的设计体现了"接口与实现分离"的经典设计原则。它定义了行为的契约，而将具体实现的选择权交给了开发者。这种设计给了我们灵活性，也让我们承担了做出正确选择的责任。

理解 Collection 的层次结构，不只是为了应对面试，更是为了在实际开发中做出明智的技术决策。每当你需要选择一个集合类型时，都应该问自己：我主要做什么操作？需要保证什么特性？有什么性能要求？只有清楚地回答这些问题，才能选出最合适的实现。
