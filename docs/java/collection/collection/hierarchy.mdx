# Collection 接口结构与分类

在 Java 的浩瀚代码宇宙中，Collection 接口是那颗恒星——所有集合类型的源头，一切容器类设计的根基。理解了这个接口的层次结构，就等于手握一张藏宝图，可以按图索骥，找到每一种集合的独特定位和最佳使用场景。

## 为什么需要集合框架？

在 Java 诞生之初的 JDK 1.0 版本中，开发者只能用数组来存储多个对象。数组虽快，但局限性明显：长度固定、缺乏灵活的操作方法、没有统一的使用方式。想象一下，每次插入一个元素都要手动移动后面的所有元素，每次查找一个元素都要写一段循环代码——这是多么痛苦的体验。

Java 1.2 带来了革命性的变化：集合框架（Collections Framework）横空出世。它不仅仅是一堆类的集合，更是一套精心设计的设计模式、语言规范和最佳实践的总和。这套框架解决了三个核心问题：**统一接口、丰富实现、性能优化**。

## 从 Iterable 到 Collection：迭代的抽象

一切始于 `Iterable` 接口——这是 Java 集合框架的最顶层抽象：

```java
public interface Iterable<T> {
    Iterator<T> iterator();
    
    default void forEach(Consumer<? super T> action) {
        Objects.requireNonNull(action);
        for (T t : this) {
            action.accept(t);
        }
    }
    
    default Spliterator<T> spliterator() {
        return Spliterators.spliteratorUnknownSize(iterator(), 0);
    }
}
```

`Iterable` 的出现让"遍历"成为所有容器的共同能力。`Iterator` 模式封装了遍历逻辑，使得调用者无需关心底层是数组还是链表——这种**接口与实现分离**的设计，正是面向对象编程的精髓。

`forEach()` 方法是 JDK 8 引入的默认方法，它让我们可以用 Lambda 表达式优雅地遍历集合：

```java
List<String> languages = Arrays.asList("Java", "Python", "Go");
languages.forEach(lang -> System.out.println(lang));

// 或者使用方法引用
languages.forEach(System.out::println);
```

`Spliterator`（可分割迭代器）是 JDK 8 引入的另一项创新，专为并行遍历设计。Stream API 的并行处理就建立在 `Spliterator` 之上，它可以将一个大集合分割成多个小片段，让多个线程并行处理。

## Collection 接口：集合的契约

`Collection` 直接继承了 `Iterable`，是所有集合类型的根接口：

```java
public interface Collection<E> extends Iterable<E> {
    // 基本操作
    int size();
    boolean isEmpty();
    boolean contains(Object o);
    boolean add(E e);
    boolean remove(Object o);
    
    // 批量操作
    boolean containsAll(Collection<?> c);
    boolean addAll(Collection<? extends E> c);
    boolean removeAll(Collection<?> c);
    boolean retainAll(Collection<?> c);
    void clear();
    
    // 数组转换
    Object[] toArray();
    <T> T[] toArray(T[] a);
    
    // 比较与哈希
    boolean equals(Object o);
    int hashCode();
    
    // Stream 操作（JDK 8+）
    default boolean removeIf(Predicate<? super E> filter);
    Stream<E> stream();
    Stream<E> parallelStream();
}
```

这些方法可以分为几组，每组承担特定的功能：

**查询方法**（`size()`、`isEmpty()`、`contains()`）返回集合的状态信息。值得注意的是，`contains()` 方法的时间复杂度取决于具体实现——`HashSet` 是 O(1)，`TreeSet` 是 O(log n)，`LinkedList` 是 O(n)。

**修改方法**（`add()`、`remove()`、`clear()`）改变集合的内容。`add()` 的返回值表示操作是否成功——对于 `Set` 实现，如果元素已存在，`add()` 会返回 `false`；对于 `List` 实现，`add()` 通常返回 `true`。

**批量操作**（`addAll()`、`removeAll()`、`retainAll()`）实现集合代数。这些操作在数学上有明确的含义：`A.addAll(B)` 相当于 A = A ∪ B，`A.removeAll(B)` 相当于 A = A - B，`A.retainAll(B)` 相当于 A = A ∩ B。

## Collection 的三大分支

`Collection` 接口分化出三个主要子接口，每个都有鲜明的特点：

### List：有序的序列

`List` 接口是 `Collection` 的第一个主要分支，它的核心特征是**有序性**和**可重复性**：

```java
public interface List<E> extends Collection<E> {
    // 基于索引的操作
    E get(int index);
    E set(int index, E element);
    void add(int index, E element);
    E remove(int index);
    
    // 搜索操作
    int indexOf(Object o);
    int lastIndexOf(Object o);
    
    // 迭代器
    ListIterator<E> listIterator();
    ListIterator<E> listIterator(int index);
    
    // 视图
    List<E> subList(int fromIndex, int toIndex);
}
```

"有序"有两层含义。首先是**插入顺序**——元素按照添加的顺序排列，这是 `List` 与 `Set` 的根本区别。其次是**索引顺序**——每个元素都有一个整数索引，可以直接通过 `get(index)` 访问。

`ListIterator` 是 `Iterator` 的增强版本，它允许双向遍历和元素修改：

```java
List<String> list = new ArrayList<>(Arrays.asList("a", "b", "c"));
ListIterator<String> it = list.listIterator();

while (it.hasNext()) {
    System.out.println(it.next());
}
while (it.hasPrevious()) {
    System.out.println(it.previous());
}
```

`subList()` 方法返回一个原列表的"视图"：

```java
List<String> list = new ArrayList<>(Arrays.asList("a", "b", "c", "d", "e"));
List<String> sub = list.subList(1, 4);  // ["b", "c", "d"]

sub.set(0, "x");  // 修改子列表
System.out.println(list);  // [a, x, c, d, e] —— 原列表也变了！

sub.add("y");  // 在子列表中添加
System.out.println(list);  // [a, x, c, d, y, e] —— 原列表也变了！
```

`subList()` 的这个特性既是优点也是陷阱。如果你不理解这个"视图"机制，可能会在无意中修改原列表；如果正确利用它，可以用简洁的代码实现复杂的列表操作。

### Set：唯一性的守护者

`Set` 接口在数学上是"不包含重复元素的集合"。有趣的是，`Set` 接口本身并没有添加任何新方法——它的独特性完全体现在实现类对 `add()` 方法的解释上：

```java
public interface Set<E> extends Collection<E> {
    // 没有添加任何新方法！
    // 所有约束都在实现类中体现
}
```

为什么 `Set` 接口不添加任何方法？这是因为**唯一性**不是一个可以通过方法签名表达的特性，而是一种**行为约束**。`add()` 方法的签名在 `Collection` 中是 `boolean add(E e)`，但在 `Set` 实现中，这个方法会先检查元素是否已存在，只有不存在时才真正添加。

`Set` 的三个主要实现体现了三种不同的唯一性保证方式：

`HashSet` 是最快的查找实现，它使用哈希表来组织元素。元素的唯一性通过 `hashCode()` 和 `equals()` 方法判断。适合对顺序没有要求的场景。

`TreeSet` 实现了有序集合，它使用红黑树结构。元素按照自然顺序或自定义比较器排序。适合需要有序遍历的场景。

`LinkedHashSet` 是 `HashSet` 的变体，它维护了一个双向链表来记录插入顺序。元素的唯一性判断与 `HashSet` 相同，但遍历时按插入顺序返回。适合需要保持插入顺序的场景。

### Queue：排队的艺术

`Queue` 接口引入了队列这种经典数据结构，遵循 FIFO（First In First Out）原则：

```java
public interface Queue<E> extends Collection<E> {
    // 添加元素
    boolean add(E e);      // 队列满时抛 IllegalStateException
    boolean offer(E e);    // 队列满时返回 false
    
    // 移除元素
    E remove();            // 队列空时抛 NoSuchElementException
    E poll();              // 队列空时返回 null
    
    // 查看元素
    E element();           // 队列空时抛 NoSuchElementException
    E peek();              // 队列空时返回 null
}
```

这里有一个精妙的设计：**成对的方法提供不同的失败处理语义**。`add()`/`remove()`/`element()` 在操作失败时抛出异常，适合需要"必须成功"的场景；`offer()`/`poll()`/`peek()` 在失败时返回特定值，适合需要"优雅处理失败"的场景。

`Deque` 接口扩展了 `Queue`，支持从两端操作：

```java
public interface Deque<E> extends Queue<E> {
    // 双端操作
    void addFirst(E e);
    void addLast(E e);
    boolean offerFirst(E e);
    boolean offerLast(E e);
    
    E removeFirst();
    E removeLast();
    E pollFirst();
    E pollLast();
    
    E getFirst();
    E getLast();
    E peekFirst();
    E peekLast();
    
    // 作为栈使用
    void push(E e);  // 等价于 addFirst
    E pop();         // 等价于 removeFirst
}
```

`Deque` 可以作为**栈**使用，`push()` 和 `pop()` 方法与 `Stack` 类的方法功能相同，但性能更好（因为 `ArrayDeque` 的底层是数组）。`LinkedList` 也是一个常用的 `Deque` 实现。

## 体系结构图

```
Iterable (可迭代)
├── Collection (集合根接口)
│   ├── List (有序、可重复)
│   │   ├── ArrayList (数组实现)
│   │   ├── LinkedList (链表实现，也实现Deque)
│   │   └── Vector (古老同步实现)
│   │       └── Stack (栈，继承Vector)
│   │
│   ├── Set (无序、唯一)
│   │   ├── HashSet (哈希表)
│   │   │   └── LinkedHashSet (维护插入顺序)
│   │   └── TreeSet (红黑树)
│   │
│   └── Queue (队列，FIFO)
│       ├── Deque (双端队列)
│       │   ├── ArrayDeque (数组实现)
│       │   └── LinkedList (链表实现)
│       └── PriorityQueue (优先级队列)
│
└── Map (键值对映射，单独体系)
    ├── HashMap
    │   ├── LinkedHashMap
    │   └── WeakHashMap
    ├── TreeMap
    ├── Hashtable
    └── ConcurrentHashMap
```

## 实现类的选择指南

面对如此多的选择，如何做出正确的决策？下面是实用的选择原则：

| 需求 | 推荐实现 | 原因 |
|------|----------|------|
| 通用列表 | `ArrayList` | 随机访问快，缓存友好 |
| 频繁头部操作 | `ArrayDeque` | 优于 `LinkedList` |
| 唯一性检查 | `HashSet` | O(1) 时间复杂度 |
| 有序唯一集合 | `TreeSet` | 自动排序 |
| 保持插入顺序 | `LinkedHashSet` | 唯一性 + 插入顺序 |
| 优先级处理 | `PriorityQueue` | 基于堆结构 |
| 线程安全列表 | `CopyOnWriteArrayList` | 读多写少场景 |
| 线程安全映射 | `ConcurrentHashMap` | 高并发场景首选 |

## 不可变集合：JDK 9+ 的新选择

JDK 9 引入了集合工厂方法，可以创建简洁的不可变集合：

```java
// 不可变列表
List<String> immutable = List.of("a", "b", "c");

// 不可变集合
Set<Integer> numbers = Set.of(1, 2, 3);

// 不可变映射
Map<String, Integer> ages = Map.of("Alice", 30, "Bob", 25);

// 不可变映射（条目较多时）
Map<String, Integer> moreAges = Map.ofEntries(
    Map.entry("Charlie", 35),
    Map.entry("Diana", 28)
);
```

不可变集合是**真正的不可变**——任何修改操作都会抛出 `UnsupportedOperationException`。它们特别适合以下场景：常量集合、并发共享数据、防御性编程。

## 写在最后

Collection 接口的设计体现了软件工程的黄金原则：**接口与实现分离，多态与封装并重**。`List`、`Set`、`Queue` 三个分支各有使命：`List` 守护有序序列，`Set` 保证元素唯一，`Queue` 维护处理顺序。

理解这套体系，不只是为了面试，更是为了在日常开发中做出正确的技术选择。每当你声明一个集合变量时，都应该先问自己：我需要保持顺序吗？我需要唯一性吗？我需要什么时间复杂度？这些问题的答案会指引你找到最适合的实现。
