## 一、ArrayList 到底是什么 🔴

### 1.1 最简实现：数组 + 长度

ArrayList 的本质就是一个**可以自动扩容的数组**。先看一个最简实现：

```java
public class SimpleArrayList<E> {
    private Object[] elementData;
    private int size;

    public SimpleArrayList() {
        this.elementData = new Object[10];  // 初始容量10
        this.size = 0;
    }

    public void add(E e) {
        if (size == elementData.length) {
            Object[] newArray = new Object[elementData.length * 2];  // 2倍扩容
            System.arraycopy(elementData, 0, newArray, 0, size);
            elementData = newArray;
        }
        elementData[size++] = e;
    }
}
```

这段代码能跑，但有几个致命问题：

- 初始容量写死了，不灵活
- 扩容倍数写死了，不智能
- 没有边界检查，不安全

JDK 的 ArrayList 就是在这基础上一步步打磨出来的。

### 1.2 JDK 源码核心属性

```java
public class ArrayList<E> extends AbstractList<E>
        implements List<E>, RandomAccess, Cloneable, java.io.Serializable {

    // 真正存储元素的数组，注意是 transient，不参与序列化
    transient Object[] elementData;

    // 实际元素个数，不是数组长度
    private int size;

    // 空数组实例，第一次 add 时会变成容量10
    private static final Object[] EMPTY_ELEMENTDATA = {};

    // 默认空数组，用于无参构造
    private static final Object[] DEFAULTCAPACITY_EMPTY_ELEMENTDATA = {};

    // 默认初始容量
    private static final int DEFAULT_CAPACITY = 10;

    // 最大数组长度，受 JVM 限制
    private static final int MAX_ARRAY_SIZE = Integer.MAX_VALUE - 8;
}
```

:::tip 💡
`transient Object[] elementData` 这个细节很重要。ArrayList 重写了 `writeObject` 和 `readObject` 方法，只序列化有值的元素，不浪费空间。所以你存的 `new ArrayList<>()` 序列化后不会有一堆 null。
:::

### 1.3 三种构造方法

```java
// 1. 无参构造 —— 创建空数组，第一次 add 时容量变成 10
public ArrayList() {
    super();
    this.elementData = DEFAULTCAPACITY_EMPTY_ELEMENTDATA;
}

// 2. 指定初始容量 —— 性能敏感场景必用
public ArrayList(int initialCapacity) {
    if (initialCapacity > 0) {
        this.elementData = new Object[initialCapacity];
    } else if (initialCapacity == 0) {
        this.elementData = EMPTY_ELEMENTDATA;
    } else {
        throw new IllegalArgumentException("Illegal Capacity: " + initialCapacity);
    }
}

// 3. 从集合复制 —— 需要复制数据时用
public ArrayList(Collection<? extends E> c) {
    elementData = c.toArray();
    if ((size = elementData.length) != 0) {
        if (elementData.getClass() != Object[].class)
            elementData = Arrays.copyOf(elementData, size, Object[].class);
    } else {
        elementData = EMPTY_ELEMENTDATA;
    }
}
```

:::warning ⚠️
无参构造创建的 ArrayList，`elementData` 长度是 0，不是 10！容量变成 10 是在第一次 add 时由 `grow()` 方法完成的。这是 90% 候选人翻车的第一个坑。
:::

## 二、扩容机制：grow 方法 🔴

### 2.1 扩容入口：add 方法

```java
public boolean add(E e) {
    // 先确保容量够用，再添加元素
    ensureCapacityInternal(size + 1);
    elementData[size++] = e;
    return true;
}
```

关键在 `ensureCapacityInternal`，这个方法会决定是否触发扩容。

### 2.2 容量计算：两步走

```java
private void ensureCapacityInternal(int minCapacity) {
    // 如果是默认空数组，取 DEFAULT_CAPACITY(10) 和 minCapacity 的较大值
    if (elementData == DEFAULTCAPACITY_EMPTY_ELEMENTDATA) {
        minCapacity = Math.max(DEFAULT_CAPACITY, minCapacity);
    }
    ensureExplicitCapacity(minCapacity);
}

private void ensureExplicitCapacity(int minCapacity) {
    // 只有需要的容量 > 当前数组长度时才扩容
    if (minCapacity - elementData.length > 0)
        grow(minCapacity);
}
```

**第一次 add 时的计算过程**：

- `size = 0`，调用 `add(e)` → `ensureCapacityInternal(1)`
- `elementData == DEFAULTCAPACITY_EMPTY_ELEMENTDATA` → `minCapacity = Math.max(10, 1) = 10`
- `minCapacity(10) - elementData.length(0) > 0` → 触发 `grow(10)`

所以第一次 add 后，容量变成 10，不是 16。

### 2.3 核心：grow 方法

```java
private void grow(int minCapacity) {
    // 旧容量
    int oldCapacity = elementData.length;
    // 新容量 = 旧容量 + 旧容量/2 = 旧容量的1.5倍
    int newCapacity = oldCapacity + (oldCapacity >> 1);

    // 如果1.5倍还不够，直接用 minCapacity
    if (newCapacity - minCapacity < 0)
        newCapacity = minCapacity;

    // 如果新容量超出上限，做最大值限制
    if (newCapacity - MAX_ARRAY_SIZE > 0)
        newCapacity = hugeCapacity(minCapacity);

    // 核心操作：创建新数组，复制旧数据
    elementData = Arrays.copyOf(elementData, newCapacity);
}
```

:::tip 💡
这里有个细节：`hugeCapacity` 方法里，如果 minCapacity 超过 `MAX_ARRAY_SIZE`，会抛出 `OutOfMemoryError`。因为 JVM 允许的最大数组长度就是 `Integer.MAX_VALUE - 8`（约21亿），超过这个值要么OOM，要么被拒绝。
:::

### 2.4 为什么是 1.5 倍扩容？

这是面试官最爱追问的点。能答出"1.5倍"的占 60%，能解释"为什么不是2倍"的占 20%。

**1.5 倍的权衡**：

| 扩容倍数 | 优点 | 缺点 |
| --- | --- | --- |
| 1.5倍 | 内存利用率高，扩容不激进 | 扩容次数相对多 |
| 2倍 | 扩容次数少 | 内存浪费严重（50%空桶） |

假设初始容量 10：
- 1.5倍扩容序列：10 → 15 → 22 → 33 → 49 → 73 → ...
- 2倍扩容序列：10 → 20 → 40 → 80 → 160 → ...

2倍扩容到 160 时，只有 73 个元素，浪费了 87 个位置。1.5 倍更节省空间，但代价是扩容次数更多。

**还有一个隐藏原因**：容量通常要经历多次扩容，1.5 倍增长可以让容量在某个地方"碰巧"和预期容量重合，减少浪费。

## 三、常见翻车现场 🟡

### ❌ 翻车点一：死抱着"默认容量16"不放

```java
// 错误认知
List<String> list = new ArrayList<>();  // ❌ 认为默认容量是16
list.add("a");  // ❌ 认为此时容量是16

// 正确认知
List<String> list = new ArrayList<>();   // 默认容量0
list.add("a");  // 第一次add后，容量变成10
```

面试官追问："那如果我提前知道要存100万个元素，该怎么初始化？"

如果回答"直接 add 就行，ArrayList 会自动扩容"，基本gg。正确答案是**预设容量**：

```java
List<String> list = new ArrayList<>(1000000);  // 预设容量100万
```

### ❌ 翻车点二：在循环里无限 add

```java
// 生产级翻车代码
List<Order> orders = new ArrayList<>();
for (Order order : orderList) {  // orderList 有100万条
    orders.add(order);  // 触发多次扩容，每次都要数组复制
}
```

100万条数据，ArrayList 扩容多少次？

10 → 15 → 22 → 33 → 49 → 73 → 109 → 163 → 244 → 366 → 549 → 823 → 1234 → 1851 → 2776 → 4164 → 6247 → 9370 → 14055 → 21083 → 31624 → 47437 → 71155 → 106733 → 160100 → 240150 → 360225 → 540338 → 810507 → 1215761

**扩容了29次**，每次都要 `System.arraycopy` 复制整个数组，白白浪费 CPU。

:::warning ⚠️
线上环境，100万次 `System.arraycopy`，每次复制数据量从几十字节到几十MB递增，这是什么概念？GC 压力暴涨，CPU 打满，还可能触发 full GC。生产环境这种写法，分分钟给你表演什么叫"服务雪崩"。
:::

### ❌ 翻车点三：subList 不是视图

```java
List<String> list = new ArrayList<>(Arrays.asList("a", "b", "c", "d"));
List<String> sub = list.subList(1, 3);
sub.clear();  // 改了 sub，原始 list 也被改了
System.out.println(list);  // [a, d]
```

`subList` 返回的不是新数组，而是原数组的**视图**，操作会影响原 list。这个坑在 JDBC 结果集处理时特别容易踩。

## 四、生产避坑指南 🔴

### 4.1 场景一：接口入参用 ArrayList

```java
// ❌ 错误：每次调用都创建新 ArrayList
public void processOrders(List<Order> orders) {
    List<Order> list = new ArrayList<>(orders);  // 又复制一遍
    // ...
}

// ✅ 正确：根据实际情况判断是否需要复制
public void processOrders(List<Order> orders) {
    if (orders instanceof ArrayList) {
        // 已经是 ArrayList，不需要复制
        ((ArrayList<Order>) orders).sort(Comparator.naturalOrder());
    } else {
        // 非 ArrayList，需要复制避免副作用
        List<Order> list = new ArrayList<>(orders);
        list.sort(Comparator.naturalOrder());
    }
}
```

### 4.2 场景二：大数据量写入

```java
// ❌ 生产翻车写法
List<LogEvent> events = new ArrayList<>();
for (LogEvent event : eventStream) {
    events.add(event);  // 无限扩容，内存爆炸
}

// ✅ 正确：预估容量或分批处理
List<LogEvent> events = new ArrayList<>(estimatedSize);
events.addAll(eventBatch);
```

### 4.3 场景三：序列化场景

ArrayList 的 `writeObject` 重写了序列化逻辑，只写有值的元素：

```java
private void writeObject(java.io.ObjectOutputStream s)
        throws java.io.IOException {
    s.defaultWriteObject();
    s.writeInt(size);
    for (int i = 0; i < size; i++) {
        s.writeObject(elementData[i]);  // 只序列化前 size 个
    }
}
```

所以一个 `new ArrayList<>()` 序列化后很小，但一个 `new ArrayList<>(1000000)` 即使只有10个元素，序列化后也包含10个元素（不是100万个）。

## 五、面试追问链 🟡

### 追问一：ArrayList 是线程安全的吗？

不是。ArrayList 的所有操作都没有同步，在多线程环境下会有：

1. **数据覆盖**：`add` 操作不是原子的，两个线程同时 add 可能覆盖彼此数据
2. **size 错乱**：`size++` 不是原子操作，volatile 也救不了
3. **数组越界**：扩容过程中一个线程在读、一个线程在扩容，可能读到不完整数据

解决方案：`Collections.synchronizedList(new ArrayList<>())` 或直接用 `CopyOnWriteArrayList`。

### 追问二：remove 操作会不会缩容？

**不会**。ArrayList 只扩容不缩容，`remove` 后数组容量不变。这意味着如果 list 先 add 了100万个元素，再 remove 掉99万个，数组容量还是100万，内存浪费严重。

如果需要缩容，手动 `trimToSize()`：

```java
list.removeAll(Collections.nCopies(999000, null));  // 删除99万
list.trimToSize();  // 缩容到实际大小
```

### 追问三：为什么 max array size 是 Integer.MAX_VALUE - 8？

因为 JVM 在数组对象头部会存储数组长度，HotSpot 的实现里头部是 12 字节（对象头 8 字节 + 数组长度 4 字节），为了避免计算溢出，留了 8 字节余量。不同 JVM 实现可能不同，所以 JDK 保守地用了 `MAX_ARRAY_SIZE = Integer.MAX_VALUE - 8`。

## 六、工程选型建议 🟢

| 场景 | 推荐方案 | 原因 |
| --- | --- | --- |
| 读多写少 | ArrayList | O(1) 随机访问 |
| 写多读少 | LinkedList | O(1) 头尾插入 |
| 需要线程安全 | CopyOnWriteArrayList | 读不加锁，写复制 |
| 需要去重 | HashSet | O(1) 查找 |
| 需要保序去重 | LinkedHashSet | 保持插入顺序 |
| 大数据量固定窗口 | 预分配容量 | 避免扩容开销 |

ArrayList 不是万能的，但它是最常用的。在选型之前，先问自己一个问题：你的主要操作是什么？
