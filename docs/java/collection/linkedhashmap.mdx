## 一、LinkedHashMap 是什么 🔴

### 1.1 问题拆解

在回答"如何实现 LRU"之前，先问自己：LinkedHashMap 和 HashMap 到底有什么区别？

**一句话总结**：LinkedHashMap = HashMap + 双向链表

```java
// LinkedHashMap 继承 HashMap，但多了一个双向链表
public class LinkedHashMap<K,V> extends HashMap<K,V> {
    // 头结点（最久未访问）
    transient LinkedHashMap.Entry<K,V> head;
    // 尾结点（最近访问）
    transient LinkedHashMap.Entry<K,V> tail;
    // true: 按访问顺序; false: 按插入顺序（默认）
    final boolean accessOrder;
}
```

**Entry 节点多了 before 和 after 指针**：

```java
static class Entry<K,V> extends HashMap.Node<K,V> {
    Entry<K,V> before;  // 前驱
    Entry<K,V> after;   // 后继
}
```

### 1.2 【直观类比】

可以把 LinkedHashMap 想象成一个**带顺序的抽屉柜**：

- HashMap 只能告诉你"第3个抽屉里有什么"
- LinkedHashMap 还能告诉你"第3个被用过的是哪个抽屉"

这对于 LRU 缓存来说太重要了——我们需要知道**谁是最近最少使用的**。

### 1.3 三种遍历顺序

LinkedHashMap 支持三种遍历模式，很多候选人面试时会被问到：

| 模式 | `accessOrder` | 遍历顺序 | 使用场景 |
|------|----------------|----------|----------|
| 插入顺序 | `false`（默认） | 按插入顺序 | 普通 Map，保持顺序 |
| 访问顺序 | `true` | 最近访问的排后面 | LRU 缓存 |
| 插入顺序 | `true` + 重写 `afterNodeAccess` | 访问会移动位置 | 有序 LRU |

## 二、LRU 缓存核心实现 🟡

### 2.1 最简实现

```java
// java:LRUCache.java
public class LRUCache<K, V> extends LinkedHashMap<K, V> {

    private final int capacity;

    public LRUCache(int capacity) {
        // initialCapacity: 16, loadFactor: 0.75, accessOrder: true 表示按访问顺序
        super(16, 0.75f, true);
        this.capacity = capacity;
    }

    @Override
    protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
        // 当 size > capacity 时，删除最老的元素
        return size() > capacity;
    }
}
```

**调用示例**：

```java
LRUCache<String, Integer> cache = new LRUCache<>(3);
cache.put("A", 1);
cache.put("B", 2);
cache.put("C", 3);

System.out.println(cache); // {A=1, B=2, C=3}

cache.get("A");  // 访问 A，将其移到链表末尾

cache.put("D", 4);  // 此时需要淘汰 B（最久未访问）
System.out.println(cache); // {A=1, C=3, D=4}
```

**问题诊断**：

- 不知道 `removeEldestEntry` 的调用时机
- 不知道 LinkedHashMap 的双向链表是怎么维护的

### 2.3 追问升级

面试官可能这样追问：

**追问1**：`accessOrder = true` 时，put 操作会触发排序吗？

```java
cache.put("A", 1);  // 插入，不影响顺序
cache.put("A", 100);  // 更新，已存在，触发 afterNodeAccess，移动到最后
```

**追问2**：`removeEldestEntry` 是在什么时机被调用的？

```java
// HashMap.putVal 源码
public V put(K key, V value) {
    return putVal(hash(key), key, value, false, true);
}

final V putVal(int hash, K key, V value, boolean onlyIfAbsent, boolean evict) {
    // ...
    afterNodeInsertion(evict);  // 这里调用 removeEldestEntry
    // evict = true 表示插入新节点，可能需要删除老节点
    return null;
}
```

**追问3**：为什么 LinkedHashMap 的 removeEldestEntry 默认返回 false？

```java
// LinkedHashMap 默认实现
protected boolean removeEldestEntry(Map.Entry<K,V> eldest) {
    return false;  // 不删除任何元素
}
```

## 三、双向链表如何维护 🟡

### 3.1 put 操作

```java
// HashMap.Node -> LinkedHashMap.Entry
static class Entry<K,V> extends HashMap.Node<K,V> {
    Entry<K,V> before, after;
}

// put 时会调用这些钩子
void afterNodeAccess(Node<K,V> e) { }
void afterNodeInsertion(boolean evict) { }
```

当 `accessOrder = true` 且 key 已存在时，调用 `afterNodeAccess`：

```java
// LinkedHashMap.afterNodeAccess
void afterNodeAccess(Node<K,V> e) {
    LinkedHashMap.Entry<K,V> last;
    // 将当前节点移动到链表末尾
    if (accessOrder && (last = tail) != e) {
        LinkedHashMap.Entry<K,V> p = (LinkedHashMap.Entry<K,V>)e,
            b = p.before, a = p.after;
        p.after = null;
        if (b == null) head = a;
        else b.after = a;
        if (a != null) a.before = b;
        else last = b;

        if (last == null) head = p;
        else {
            p.before = last;
            last.after = p;
        }
        tail = p;
        ++modCount;
    }
}
```

**图示**：

```
访问前: head -> A -> B -> C -> tail
访问B: head -> A -> C -> B -> tail  (B被移到末尾)
```

### 3.2 get 操作

```java
public V get(Object key) {
    Node<K,V> e;
    if ((e = getNode(hash(key), key)) == null) return null;
    if (accessOrder)
        afterNodeAccess(e);  // 如果 accessOrder=true，访问后移到末尾
    return e.value;
}
```

:::tip 💡
**面试加分点**：get 操作在 `accessOrder = true` 时也会修改链表！这是很多人忽略的点。
:::

### 3.3 remove 操作

```java
public V remove(Object key) {
    Node<K,V> e;
    if ((e = removeNode(hash(key), key, null, false, true)) == null) return null;
    afterNodeRemoval(e);  // 从双向链表中摘除
    return e.value;
}

void afterNodeRemoval(Node<K,V> e) {
    LinkedHashMap.Entry<K,V> p = (LinkedHashMap.Entry<K,V>)e,
        b = p.before, a = p.after;
    p.before = p.after = null;
    if (b == null) head = a;
    else b.after = a;
    if (a == null) tail = b;
    else a.before = b;
}
```

## 四、生产级 LRU 实现 🟢

### 4.1 简单版的问题

继承 LinkedHashMap 的方式有个致命问题：**不是线程安全的**。

在高并发场景下，可能出现：

- 链表结构被破坏
- size() 和实际元素数量不一致
- removeEldestEntry 判断出错

### 4.2 线程安全版

```java
// java:ThreadSafeLRUCache.java
public class ThreadSafeLRUCache<K, V> {

    private final int capacity;
    private final LinkedHashMap<K, V> cache;

    public ThreadSafeLRUCache(int capacity) {
        this.capacity = capacity;
        // 覆写 removeEldestEntry，不让它自动删除
        this.cache = new LinkedHashMap<>(16, 0.75f, true) {
            @Override
            protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
                return false;  // 我们手动控制删除
            }
        };
    }

    public V get(K key) {
        synchronized (cache) {
            return cache.get(key);
        }
    }

    public void put(K key, V value) {
        synchronized (cache) {
            cache.put(key, value);
            // 手动淘汰超出容量的元素
            while (cache.size() > capacity) {
                Map.Entry<K, V> eldest = cache.entrySet().iterator().next();
                cache.remove(eldest.getKey());
            }
        }
    }

    public int size() {
        synchronized (cache) {
            return cache.size();
        }
    }

    public void clear() {
        synchronized (cache) {
            cache.clear();
        }
    }
}
```

### 4.3 【架构权衡】

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| 继承 LinkedHashMap | 简单，几行代码 | 非线程安全 | 单线程、本地缓存 |
| synchronized 包装 | 简单安全 | 性能差，并发度低 | 低并发场景 |
| ReentrantLock 分段 | 并发度高 | 实现复杂 | 高并发场景 |
| ConcurrentHashMap + 双向链表 | 线程安全+高性能 | 实现复杂 | 生产级缓存 |

:::warning ⚠️
生产环境推荐使用 `Guava Cache` 或 `Caffeine`，它们提供了更完善的 LRU/W-TinyLFU 等淘汰策略，比手写更安全、更高效。
:::

## 五、面试标准回答

### 5.1 P5 回答（基本概念）

> "LinkedHashMap 继承自 HashMap，比 HashMap 多维护了一个双向链表。通过 `accessOrder` 参数可以控制是按插入顺序还是访问顺序遍历。LRU 缓存可以通过继承 LinkedHashMap，重写 `removeEldestEntry` 方法实现。"

### 5.2 P6 回答（原理理解）

> "LinkedHashMap 在 HashMap 的桶结构基础上，每个节点多了 before 和 after 指针，串成双向链表。当 `accessOrder = true` 时，每次 get/put 都会调用 `afterNodeAccess`，把访问的节点移到链表末尾。`removeEldestEntry` 在 `putVal` 最后被调用，返回 true 就会删除最老的节点（head）。"

### 5.3 P7 回答（工程视角）

> "继承 LinkedHashMap 的方式实现简单，但有几个坑：第一，非线程安全，高并发下链表可能断裂；第二，`removeEldestEntry` 是基于 size() 判断的，如果你在 put 后手动删除了元素，可能误删；第三，LinkedHashMap 的迭代器是弱一致的。生产环境建议用 Caffeine 或 Guava Cache，它们不仅线程安全，还支持 W-TinyLFU 等更智能的淘汰策略。"

## 七、常见翻车点

### ❌ 翻车点 1：分不清 accessOrder

```java
// ❌ 错误：默认 false，按插入顺序，永远不会触发 LRU 淘汰
new LinkedHashMap<>(capacity);

// ✅ 正确：设为 true，按访问顺序
new LinkedHashMap<>(capacity, 0.75f, true);
```

### ❌ 翻车点 2：忘记 removeEldestEntry 的调用条件

```java
// ❌ 错误理解：每次 put 都会调用
// ✅ 正确理解：只有 put 新元素时才会调用，更新已存在的 key 不会触发
cache.put("A", 1);   // 触发
cache.put("A", 100); // 不触发 removeEldestEntry，因为 key 已存在
```

### ❌ 翻车点 3：在 removeEldestEntry 中做复杂操作

```java
// ❌ 危险：removeEldestEntry 可能被频繁调用，不适合做重操作
protected boolean removeEldestEntry(Map.Entry<K,V> eldest) {
    // 不要在这里做 IO、计算等重操作
    return size() > capacity;
}
```

## 八、延伸问题

### Q1：LinkedHashMap 的迭代器是 fail-fast 的吗？

不是。LinkedHashMap 的迭代器是**弱一致**的，而 HashMap 是 **fail-fast** 的。

- HashMap 迭代器：结构性修改会抛出 ConcurrentModificationException
- LinkedHashMap 迭代器：可以一边遍历一边删除，不会抛异常

```java
// LinkedHashMap 可以这样安全地边遍历边删除
for (Map.Entry<String, Integer> entry : cache.entrySet()) {
    if (entry.getValue() < 0) {
        cache.remove(entry.getKey());  // 安全
    }
}
```

### Q2：LinkedHashMap 能否用于实现 FIFO 缓存？

可以。只需设置 `accessOrder = false`（默认），配合 `removeEldestEntry` 即可：

```java
public class FIFOCache<K, V> extends LinkedHashMap<K, V> {
    private final int capacity;

    public FIFOCache(int capacity) {
        super(16, 0.75f, false);  // 按插入顺序
        this.capacity = capacity;
    }

    @Override
    protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
        return size() > capacity;
    }
}
```
