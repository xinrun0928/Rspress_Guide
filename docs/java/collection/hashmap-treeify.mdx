# HashMap 红黑树阈值设计

小张答："因为链表太长了查询慢。"

小张说："应该是性能考虑吧..."

【面试官心理】

## 一、为什么需要红黑树 🔴

### 1.1 链表退化的灾难

HashMap 的最坏情况是所有 key 都 hash 碰撞到一个桶里，退化成链表：

```java
// 恶意构造的 hash 碰撞
public class CollisionKey {
    int value;
    public int hashCode() {
        return 0;  // 所有实例的 hashCode 都返回 0
    }
    public boolean equals(Object o) {
        return o instanceof CollisionKey;
    }
}
```

100万个 CollisionKey 放入 HashMap，查询从 O(1) 退化成 O(n)，百万次查询变成十亿次操作。

### 1.2 红黑树的性能保障

| 数据结构 | 查询复杂度 | 最坏情况（100万元素） |
| --- | --- | --- |
| 链表 | O(n) | 1,000,000 次比较 |
| 红黑树 | O(log n) | 约 20 次比较 |

红黑树是一种自平衡的二叉搜索树，通过旋转和着色保持左右子树高度差不超过 2 倍，保证查找、插入、删除都是 O(log n)。

## 二、树化阈值：8 不是拍脑袋 🔴

### 2.1 JDK 源码注释

```java
/**
 * 链表转红黑树的阈值。链表长度达到 8 时转红黑树。
 *
 * 这个值是通过泊松分布计算得出的。
 * 假设 hash 分布均匀，负载因子 0.75，
 * 链表长度为 k 的概率为：
 * P(k) = (e^(-λ) * λ^k) / k!
 * 其中 λ = 0.5（负载因子相关参数）
 *
 * P(8) = 0.00000006
 *
 * 也就是说，hash 均匀分布时，链表长度达到 8 的概率只有千万分之六。
 * 正常情况下几乎不会触发，只有在恶意攻击或 hash 函数设计不当时才会出现。
 */
static final int TREEIFY_THRESHOLD = 8;
```

### 2.2 泊松分布推导

```java
// 泊松分布公式：P(k) = (e^(-λ) * λ^k) / k!
// 当负载因子 = 0.75 时，碰撞参数 λ = 0.5

// 链表长度为 0 的概率
P(0) = e^(-0.5) * 0.5^0 / 0! ≈ 0.607  (60.7%的桶是空的)

// 链表长度为 1 的概率
P(1) = e^(-0.5) * 0.5^1 / 1! ≈ 0.303  (30.3%的桶只有1个元素)

// 链表长度为 2 的概率
P(2) = e^(-0.5) * 0.5^2 / 2! ≈ 0.076  (7.6%的桶有2个元素)

// 链表长度为 3 的概率
P(3) ≈ 0.013  (1.3%的桶有3个元素)

// 链表长度为 4 的概率
P(4) ≈ 0.0015  (0.15%的桶有4个元素)

// 链表长度为 5 的概率
P(5) ≈ 0.00015  (0.015%的桶有5个元素)

// 链表长度为 6 的概率
P(6) ≈ 0.000013  (0.0013%的桶有6个元素)

// 链表长度为 7 的概率
P(7) ≈ 0.0000009  (0.00009%的桶有7个元素)

// 链表长度为 8 的概率
P(8) ≈ 0.00000006  (千万分之六)

// 链表长度为 9 的概率
P(9) ≈ 0.000000003  (亿分之三)
```

:::tip 💡
从概率可以看出：正常情况下，99.9% 的桶内元素不超过 6 个。链表长度达到 8 是一个极小概率事件，只有在 hash 函数被攻击或者设计失败时才会发生。JDK 用 8 作为阈值，在正常场景下几乎不会触发红黑树，而在极端场景下能保护性能。
:::

### 2.3 为什么不是 16？

面试官追问："那为什么不用 16 作为阈值？"

答案很简单：**16 太大了**。如果阈值设为 16，意味着在链表长度达到 16 之前，HashMap 都要忍受 O(n) 的查询。

| 阈值 | 正常场景触发概率 | 最坏查询次数（16个元素） |
| --- | --- | --- |
| 6 | 极低，但可能触发 | 6 次 |
| 8 | 千万分之六 | 8 次 |
| 16 | 几乎不可能触发 | 16 次 |

8 是一个平衡点：
- 不太激进：不会因为正常碰撞就触发红黑树（红黑树有额外开销）
- 不过于保守：不会让链表太长导致性能退化

## 三、退链表阈值：6 不是 8 🟡

### 3.1 为什么不是 8？

面试官又问："红黑树退化到链表，阈值为什么是 6 而不是 8？"

这是**留了两个元素的缓冲空间**，避免在临界点频繁切换。

假设退链表阈值也是 8：
```
树化：>=8 个元素
退链表：<=8 个元素

场景：桶内元素在 7 和 8 之间来回变化
- add 第8个元素 → 触发树化
- remove 1个元素 → 触发退链表
- add 1个元素 → 又触发树化
→ 无限切换，疯狂消耗 CPU
```

6 作为退链表阈值，留了 2 个元素的缓冲（6、7），只有当元素减少到 6 个以下时才退链表，避免频繁切换。

### 3.2 红黑树的额外开销

红黑树不是免费的午餐：
- 每个节点多 3 个字段（parent、left、right）vs 链表只有 1 个（next）
- 插入/删除时需要 O(log n) 的旋转操作来维持平衡
- 代码复杂度更高，调试困难

所以 JDK 的策略是：**能用链表就不用树**。只有在链表"太长"（>=8）时，才升级为红黑树。

## 四、容量阈值：64 🟡

### 4.1 为什么需要 MIN_TREEIFY_CAPACITY？

```java
static final int MIN_TREEIFY_CAPACITY = 64;
```

`treeifyBin` 方法中：

```java
final void treeifyBin(Node<K, V>[] tab, int hash) {
    int n, index;
    Node<K, V> e;

    if (tab == null || (n = tab.length) < MIN_TREEIFY_CAPACITY)
        // 容量 < 64 时，优先扩容而不是树化
        resize();
    // ...
}
```

**设计逻辑**：

如果数组容量很小（< 64），与其把链表树化，不如直接扩容。扩容后数组容量翻倍，每个桶的平均元素数量会减半，链表自然会变短。

```
容量 16 → 链表长度 >= 8 → 扩容到 32
扩容后每个桶平均元素数量减半，链表退化到 4 个元素以下
```

### 4.2 扩容 vs 树化对比

| 策略 | 容量 < 64 | 容量 >= 64 |
| --- | --- | --- |
| 链表长度 >= 8 | 扩容 | 树化 |
| 扩容后链表长度 | 约 4 个元素 | 不变 |

## 五、红黑树的分裂处理 🟡

### 5.1 split 方法

当 HashMap 扩容时，红黑树需要"分裂"到新数组中：

```java
final void split(HashMap<K, V> map, Node<K, V>[] newTab, int bit, int oldCap) {
    TreeNode<K, V> b = this;
    TreeNode<K, V> loHead = null, loTail = null;  // 留在原桶的节点
    TreeNode<K, V> hiHead = null, hiTail = null;  // 移动到新桶的节点

    // 遍历红黑树，按扩容规则分配节点
    for (TreeNode<K, V> e = b; e != null; e = e.prev) {
        int ei = e.hash & oldCap;
        if ((ei & bit) == 0) {  // bit = oldCap
            // 留在原位置
            if ((e.prev = loTail) == null)
                loHead = e;
            else
                loTail.next = e;
            loTail = e;
        } else {
            // 移动到新位置
            if ((e.prev = hiTail) == null)
                hiHead = e;
            else
                hiTail.next = e;
            hiTail = e;
        }
    }

    if (loHead != null) {
        // 原位置节点：如果数量 <= 6，退链表；否则保持红黑树
        newTab[j] = (size < UNTREEIFY_THRESHOLD) ?
            nodeTab(loHead) : (loHead.treeify() ? loHead : loHead);
    }
    if (hiHead != null) {
        // 新位置节点：同上
        int hiBit = j + oldCap;
        newTab[hiBit] = (size < UNTREEIFY_THRESHOLD) ?
            nodeTab(hiHead) : (hiHead.treeify() ? hiHead : hiHead);
    }
}
```

### 5.2 分裂后的判断

扩容时，红黑树的分裂也会遵循退链表规则：
- 如果分裂后某一边节点数 `<= 6`，退化为链表
- 如果节点数 `> 6`，保持红黑树

## 六、生产避坑指南 🟢

### 6.1 场景：识别恶意 hash 攻击

如果线上发现某个桶的链表特别长（> 8），可能是：
1. **hash 函数设计有问题**：自定义 key 的 hashCode 实现不均匀
2. **恶意攻击**：攻击者故意构造 hash 碰撞

```java
// 监控代码
public class HashMapMonitor {
    public static void detectAnomaly(HashMap<?, ?> map) {
        // 可以通过反射获取内部 table
        // 统计每个桶的链表/树长度
        // 如果某个桶 > 8，发出告警
    }
}
```

### 6.2 场景：自定义 key 的 hashCode

```java
// ❌ 错误：hashCode 不均匀
class BadKey {
    int id;
    public int hashCode() {
        return id % 10;  // 只返回 0~9，大量碰撞
    }
}

// ✅ 正确：使用 Objects.hash()
class GoodKey {
    String name;
    int age;
    public int hashCode() {
        return Objects.hash(name, age);  // JDK 自带的均匀 hash
    }
}

// ✅ 更优：直接用 Integer / String 等已有类型作为 key
Map<String, User> users = new HashMap<>();  // String 的 hashCode 经过优化
```
