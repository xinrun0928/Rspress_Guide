# 线程池执行流程与状态转换

凌晨 3 点，你的监控系统报警：接口响应时间从 10ms 飙升到 5 秒。排查发现，是某个定时任务突然提交了 10000 个任务到线程池，核心线程都在忙，非核心线程被回收，新任务全部堆积在队列里等待。

这是线程池的「失控」时刻。要避免它，必须深入理解线程池的**执行流程**和**状态转换**。

## 线程池的七种状态

线程池不是简单的「提交任务、执行任务」，它有五种运行状态：

| 状态 | 高3位值 | 说明 |
| --- | --- | --- |
| RUNNING | 111 | 接收新任务，处理队列任务 |
| SHUTDOWN | 000 | 不接收新任务，但处理完队列剩余任务 |
| STOP | 001 | 不接收新任务，不处理队列任务，中断正在执行的任务 |
| TIDYING | 010 | 所有任务已终止，workerCount = 0，即将执行 terminated() |
| TERMINATED | 011 | terminated() 已执行完毕 |

:::tip 状态值的设计
高三位用于存储状态，低 29 位存储 workerCount。这种设计让状态和线程数可以原子性地存储在一个 `int` 变量中，避免了加锁的开销。
:::

```java
private final AtomicInteger ctl = new AtomicInteger(ctlOf(RUNNING, 0));

private static int ctlOf(int rs, int wc) { return rs | wc; }
```

状态转换是**单向不可逆**的：

```
RUNNING -> SHUTDOWN -> TIDYING -> TERMINATED
   |          |
   v          v
  STOP     (不可逆)
   |
   v
 TIDYING
```

## 任务执行的核心流程

当你调用 `execute(Runnable command)` 时，线程池内部经历了怎样的旅程？

```java
public void execute(Runnable command) {
    if (command == null)
        throw new NullPointerException();

    int c = ctl.get();

    // 步骤1：核心线程池是否满了？
    if (workerCountOf(c) < corePoolSize) {
        // 添加核心线程
        if (addWorker(command, true))
            return;
        c = ctl.get();
    }

    // 步骤2：线程池是否在运行？任务是否成功入队？
    if (isRunning(c) && workQueue.offer(command)) {
        int recheck = ctl.get();
        // 二次检查：入队后线程池可能被关闭了
        if (!isRunning(recheck) && remove(command))
            reject(command);
        // 核心线程为0时，创建一个非核心线程
        else if (workerCountOf(recheck) == 0)
            addWorker(null, false);
        return;
    }

    // 步骤3：尝试创建非核心线程
    else if (!addWorker(command, false))
        reject(command);
}
```

**三步决策树**：

```
提交任务
    │
    ├── workerCount < corePoolSize ──> 创建核心线程
    │
    ├── 入队成功 ──> 二次检查（防止入队后被关闭）
    │         ├── 线程池已关闭？移除任务并拒绝
    │         └── workerCount = 0？创建非核心线程保活
    │
    └── 入队失败 ──> 尝试创建非核心线程
              └── 创建失败？执行拒绝策略
```

## addWorker：线程创建的细节

`addWorker` 是线程池的核心方法，它负责创建工作线程：

```java
private boolean addWorker(Runnable firstTask, boolean core) {
    retry:
    for (;;) {
        int c = ctl.get();
        int rs = runStateOf(c);

        // 检查状态：非 RUNNING 或 SHUTDOWN + 有任务 + 队列非空 时才允许
        if (rs >= SHUTDOWN &&
            !(rs == SHUTDOWN && firstTask == null && !workQueue.isEmpty()))
            return false;

        for (;;) {
            int wc = workerCountOf(c);
            // 核心线程限制 or 非核心线程限制
            if (wc >= (core ? corePoolSize : maximumPoolSize))
                return false;
            if (compareAndIncrementWorkerCount(c))
                break retry;
            c = ctl.get();
            if (runStateOf(c) != rs)
                continue retry;
        }
    }

    // Worker 对象创建
    Worker w = null;
    w = new Worker(firstTask);
    final Thread t = w.thread;

    // 添加到 workers 集合
    workers.add(w);
    t.start();
    workerAdded = true;
}
```

## Worker：Runnable 还是 Thread？

Worker 继承自 AbstractQueuedSynchronizer（AQS），实现了 Runnable 接口：

```java
private final class Worker extends AbstractQueuedSynchronizer implements Runnable {
    final Thread thread;
    Runnable firstTask;

    Worker(Runnable firstTask) {
        this.firstTask = firstTask;
    }

    public void run() {
        runWorker(this);
    }
}
```

为什么要用 AQS？因为 Worker 本身就是一个**不可重入的锁**：

```java
// 尝试获取锁
protected boolean tryAcquire(int unused) {
    if (compareAndSetState(0, 1)) {
        setExclusiveOwnerThread(Thread.currentThread());
        return true;
    }
    return false;
}

// 释放锁
protected boolean tryRelease(int unused) {
    setExclusiveOwnerThread(null);
    setState(0);
    return true;
}
```

:::warning Worker 锁的用途
Worker 持有自己的锁，用于中断正在执行的任务。当线程池调用 `shutdownNow()` 时，会中断所有正在执行任务的 Worker。
:::

## runWorker：任务的真正执行者

```java
final void runWorker(Worker w) {
    Thread wt = Thread.currentThread();
    Runnable task = w.firstTask;
    w.firstTask = null;

    // 解锁，允许中断
    w.unlock();

    boolean completedAbruptly = true;
    try {
        // 循环：优先取 firstTask，然后从队列中获取
        while (task != null || (task = getTask()) != null) {
            w.lock();
            try {
                // 任务执行前的钩子
                beforeExecute(wt, task);
                try {
                    task.run();
                } finally {
                    task = null;
                }
                // 任务执行后的钩子
                afterExecute(task, thrown);
            } finally {
                w.unlock();
            }
        }
        completedAbruptly = false;
    } finally {
        // 清理：移除 Worker
        processWorkerExit(w, completedAbruptly);
    }
}
```

**关键设计**：Worker 通过 `getTask()` 从队列中获取任务，这是一个**阻塞操作**：

```java
private Runnable getTask() {
    boolean timedOut = false;

    for (;;) {
        int c = ctl.get();
        int rs = runStateOf(c);

        // 队列空，线程应该退出
        if (rs >= SHUTDOWN && workQueue.isEmpty())
            return null;

        int wc = workerCountOf(c);

        // 是否需要超时控制？
        boolean timed = allowCoreThreadTimeOut || wc > corePoolSize;

        if ((wc > maximumPoolSize || (timed && timedOut))
            && wc > 1 && workQueue.isEmpty())
            return null;

        try {
            // 阻塞获取，带超时
            Runnable r = timed ?
                workQueue.poll(keepAliveTime, TimeUnit.NANOSECONDS) :
                workQueue.take();
            if (r != null) return r;
            timedOut = true;
        } catch (InterruptedException retry) {
            timedOut = false;
        }
    }
}
```

## 状态转换的实际意义

```
RUNNING -> SHUTDOWN
触发：shutdown()
影响：不接受新任务，等待队列中的任务执行完毕

SHUTDOWN -> STOP
触发：shutdownNow()
影响：不接受新任务，中断正在执行的任务

STOP -> TIDYING
触发：所有任务已终止
影响：准备执行 terminated() 钩子

TIDYING -> TERMINATED
触发：terminated() 执行完毕
影响：线程池彻底关闭
```

**实际应用**：如果你需要在线程池关闭后做一些清理工作，可以重写 `terminated()` 方法：

```java
public class CustomThreadPool extends ThreadPoolExecutor {
    @Override
    protected void terminated() {
        // 释放资源、记录日志、通知监控
        System.out.println("线程池已关闭，开始清理资源");
    }
}
```

## 总结

线程池的执行流程是一个精心设计的状态机：

- **三步决策**：核心线程满 → 入队 → 非核心线程
- **Worker 作为执行单元**：既是 Runnable 又是锁
- **阻塞获取任务**：空闲线程在队列上等待，支持超时回收
- **单向状态转换**：一旦关闭，无法重启

理解这些细节，才能在生产环境中正确配置线程池参数，避免任务堆积和线程饥饿的问题。
