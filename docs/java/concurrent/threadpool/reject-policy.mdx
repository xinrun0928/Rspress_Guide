# 线程池拒绝策略对比

当线程池无法接受新任务时，JDK 提供了四种内置的拒绝策略。理解每种策略的特性和适用场景，能帮助我们在不同业务场景下做出正确的选择。

## 四种内置拒绝策略

### AbortPolicy（默认策略）

`AbortPolicy` 是线程池的默认拒绝策略。当任务无法被接受时，直接抛出 `RejectedExecutionException`。

```java
new ThreadPoolExecutor.AbortPolicy()
```

```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    2, 4, 60, TimeUnit.SECONDS,
    new ArrayBlockingQueue<>(2),
    new ThreadPoolExecutor.AbortPolicy()
);

try {
    executor.execute(() -> System.out.println("Task"));
} catch (RejectedExecutionException e) {
    // 任务被拒绝，需要处理
    System.out.println("Task rejected!");
}
```

**适用场景**：默认策略适用于大多数场景。如果任务无法执行，我们需要知道这一点并采取补救措施。抛出异常是最直接的失败通知机制。

### CallerRunsPolicy（调用者执行策略）

`CallerRunsPolicy` 不会丢弃任务，而是由提交任务的线程自己执行。

```java
new ThreadPoolExecutor.CallerRunsPolicy()
```

```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    2, 4, 60, TimeUnit.SECONDS,
    new ArrayBlockingQueue<>(2),
    new ThreadPoolExecutor.CallerRunsPolicy()
);
```

**执行流程**：

1. 线程 A 提交任务 T
2. 线程池队列已满，无法接受任务
3. 任务 T 被返回给线程 A 执行
4. 线程 A 在执行 T 的同时，无法继续提交新任务

**适用场景**：适合对任务完整性要求高的场景。这个策略实际上是让调用者"背锅"，自己执行任务。虽然会增加调用者的负担，但能确保任务不会丢失。

### DiscardPolicy（丢弃策略）

`DiscardPolicy` 直接丢弃无法接受的任务，不抛出异常，也不通知。

```java
new ThreadPoolExecutor.DiscardPolicy()
```

```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    2, 4, 60, TimeUnit.SECONDS,
    new ArrayBlockingQueue<>(2),
    new ThreadPoolExecutor.DiscardPolicy()
);

// 任务被静默丢弃，没有任何通知
executor.execute(() -> System.out.println("This task is silently discarded"));
```

**适用场景**：适合对任务丢失不敏感的场景。比如日志收集、监控数据上报等，即使丢失几条数据也不会影响核心业务。

### DiscardOldestPolicy（丢弃最老策略）

`DiscardOldestPolicy` 丢弃队列中最老的任务，然后尝试重新提交新任务。

```java
new ThreadPoolExecutor.DiscardOldestPolicy()
```

```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    2, 4, 60, TimeUnit.SECONDS,
    new ArrayBlockingQueue<>(2),
    new ThreadPoolExecutor.DiscardOldestPolicy()
);
```

**执行流程**：

1. 线程池队列已满
2. 移除队列中最老的任务（队首）
3. 尝试提交新任务
4. 如果仍然无法提交，新任务也被丢弃

**适用场景**：适合"尽量执行最新任务"的场景。比如股票行情数据，我们只关心最新的价格，丢弃旧数据比丢弃新数据更有价值。

## 策略对比

| 策略 | 任务丢失 | 线程池压力 | 调用者压力 | 通知机制 |
|------|---------|-----------|-----------|---------|
| AbortPolicy | 是 | 无 | 无 | 抛异常 |
| CallerRunsPolicy | 否 | 是 | 是 | 无（自己执行） |
| DiscardPolicy | 是 | 无 | 无 | 无 |
| DiscardOldestPolicy | 是（老的） | 无 | 无 | 无 |

## 自定义拒绝策略

除了四种内置策略，我们还可以实现 `RejectedExecutionHandler` 接口来自定义策略。

```java
public class LoggingRejectedHandler implements RejectedExecutionHandler {
    private static final Logger log = LoggerFactory.getLogger(LoggingRejectedHandler.class);
    
    @Override
    public void rejectedExecution(Runnable r, ThreadPoolExecutor executor) {
        log.warn("Task {} rejected from {}", r, executor);
    }
}
```

更复杂的策略可以记录日志、持久化任务、发送告警等：

```java
public class PersistentRejectedHandler implements RejectedExecutionHandler {
    private final TaskRepository repository;
    
    @Override
    public void rejectedExecution(Runnable r, ThreadPoolExecutor executor) {
        if (r instanceof PersistentTask) {
            // 持久化任务以便稍后重试
            repository.save(((PersistentTask) r).toTaskEntity());
        }
        // 发送告警
        alertService.send("Thread pool saturated", AlertLevel.WARNING);
    }
}
```

## 总结

四种拒绝策略各有特点：AbortPolicy 直接抛异常，适用于需要明确失败通知的场景；CallerRunsPolicy 让调用者执行，适用于任务不能丢失的场景；DiscardPolicy 静默丢弃，适用于丢失无所谓的场景；DiscardOldestPolicy 丢弃最老的，适用于优先执行新任务的场景。

选择合适的拒绝策略，需要根据业务对任务完整性、及时性的要求，以及系统的承载能力来综合考虑。
