# volatile：既熟悉又陌生的可见性关键字

候选人小王在美团三面的高并发场景题中，遇到了这样一个问题：

面试官问："单例模式的双重检查锁定中，instance为什么要加volatile？不加行不行？"

小王说："因为volatile保证可见性，可以防止指令重排。"

面试官追问："那你说说volatile是怎么保证可见性的？底层是什么？"

小王说："...就是每次都从主内存读写？"

面试官继续追问："那在x86架构下，volatile写和普通写有什么区别？"

小王开始支支吾吾。

【面试官心理】
volatile这个关键字，80%的候选人知道它是"保证可见性"的，但只有30%能说出内存屏障，10%能讲清楚x86和ARM下的不同实现。面试官追问volatile底层，不是在炫技，而是在确认这个候选人是否真正理解了自己用的技术。毕竟volatile的可见性保证在某些场景下是反直觉的——比如x86下volatile读几乎零开销，但ARM下volatile读开销显著。

## 一、volatile的可见性保证：从一个bug说起 🔴

### 1.1 经典的可见性bug

```java
public class VisibilityBug {
    private static boolean running = true;

    public static void main(String[] args) throws InterruptedException {
        Thread counter = new Thread(() -> {
            int count = 0;
            while (running) { // ①
                count++;
            }
            System.out.println("Count: " + count);
        });

        counter.start();
        Thread.sleep(1000);
        running = false; // ②
        System.out.println("Set running to false");
    }
}
```

这段代码的行为在理论上应该是：主线程设置`running=false`后，counter线程的while循环应该立即退出。

**但实际上呢？**

在某些JIT编译优化下，counter线程的while循环可能被优化成：

```java
// JIT优化后的代码
if (running) {  // 只读一次
    while (true) {
        count++;
    }
}
```

JIT认为`running`是`while`循环的不变条件（Loop Invariant），把它提到循环外面，只读一次。设置`running=false`后，counter线程永远看不到这个变化——**程序永远不会退出**。

这就是**可见性**问题：`running=false`写入了主内存，但counter线程的CPU缓存和工作内存中还保存着`running=true`的值。

### 1.2 volatile如何修复可见性

```java
private static volatile boolean running = true;
```

加上`volatile`后，JIT和CPU必须遵守volatile变量的happens-before规则：
- 每次读取`running`时，必须从主内存读取最新值（不能使用缓存）
- 每次写入`running`时，必须立即刷新到主内存并通知其他核心

这样，②写入的值对①的读取**保证可见**，程序会正常退出。

## 二、volatile的禁止重排序保证 🔴

### 2.1 什么叫做"重排序"

重排序是编译器、解释器、CPU为了优化性能，对指令顺序进行调整的行为。重排序分为两种：

**编译期重排序**：JIT编译器调整字节码顺序
**运行时重排序**：CPU在执行时调整指令流水线顺序

重排序的前提是：**不改变单线程的执行结果**。但多线程环境下，重排序会导致严重的bug。

### 2.2 volatile在x86下的四条屏障规则

JMM通过在volatile读写前后插入内存屏障来实现禁止重排序。具体的规则：

| 屏障插入位置 | 屏障类型 | x86实现 |
|---|---|---|
| volatile写**之前** | StoreStore | 不需要额外指令（x86天然满足） |
| volatile写**之后** | StoreLoad | MFENCE指令 |
| volatile读**之前** | LoadLoad | 不需要额外指令 |
| volatile读**之后** | LoadStore | 不需要额外指令 |

x86架构下，**最关键的是StoreLoad屏障**——volatile写之后必须加MFENCE，然后再执行后续的读操作。这个屏障保证volatile写对其他核心的可见性。

### 2.3 实际效果：禁止了哪些重排序

```java
// volatile写之前的重排序被禁止
int a = 1;          // 普通写
volatile boolean flag = true;  // volatile写

// 禁止重排: flag = true 不能排到 a = 1 之前
// 即: a = 1 happens-before flag = true

// volatile读之后的重排序被禁止
volatile boolean ready = false; // volatile读
int x = 42;             // 普通读

// 禁止重排: x = 42 不能排到 ready 之前
// 即: ready = false happens-before x = 42
```

### 2.4 用字节码验证

```java
public class VolatileExample {
    volatile int x;

    public void writer() {
        x = 100;           // volatile写
    }

    public void reader() {
        int r = x;         // volatile读
    }
}
```

用`javap -c`查看字节码：

```
public void writer() ...
  0: aload_0
  1: bipush 100
  3: putfield #2   <-- volatile字段，ACC_VOLATILE标记
  6: return

public void reader() ...
  0: aload_0
  1: getfield #2   <-- volatile字段，ACC_VOLATILE标记
  4: istore_1
  5: return
```

字节码上看不出太多差异——volatile的魔法发生在**JIT编译时**，不是在字节码层面。JIT编译器看到`ACC_VOLATILE`标记后，在生成的机器码中插入对应的内存屏障。

## 三、volatile vs synchronized：边界在哪里 🔴

### 3.1 核心区别

| 维度 | volatile | synchronized |
|------|----------|-------------|
| 原子性 | 仅保证volatile读写本身原子 | 保证整个临界区原子 |
| 可见性 | 写→后续对同一变量的读 | 解锁→后续对同一锁的加锁 |
| 有序性 | 禁止与volatile变量的重排 | 禁止临界区内的重排 |
| 性能 | 极低开销（x86读几乎零开销） | 有锁竞争开销 |
| 使用限制 | 仅适用于单个变量的读写 | 适用于任意复杂的多步操作 |

### 3.2 ❌ 错误示范

**候选人原话**："volatile可以保证复合操作的原子性，比如i++。"

**问题诊断**：
- i++包含三步：读取→加1→写入。volatile只能保证每一步单独是原子的
- 两个线程同时执行i++时，可能发生：线程A读到0，线程B也读到0，各自加1后写回，结果是1而不是2
- 这是read-modify-write竞态条件，volatile解决不了

**候选人原话**："volatile比synchronized性能好，所以应该尽量用volatile。"

**问题诊断**：
- volatile只适用于非常有限的场景（单个volatile变量的读写）
- 如果需要保证复合操作的原子性，volatile无能为力
- 用volatile替代synchronized是在用错误的方案解决问题

**候选人原话**："volatile写会立即刷到主内存，所以可以保证数据一致。"

**问题诊断**：
- 忽略了Store Buffer的存在——在x86下volatile写先写入Store Buffer，然后异步刷新
- 其他核心看到volatile写的时间点取决于缓存失效信号的传播速度
- 严格来说volatile写是"最终一致"的，不是严格的"立即"

【面试官心理】
volatile这道题，面试官最想确认三件事：第一，他知不知道volatile只能保证单个变量的可见性，不能保证复合操作的原子性；第二，他有没有看过底层实现，知道内存屏障的作用；第三，他能不能准确区分volatile和synchronized的边界。能答到第三层的候选人，基本在并发编程上已经有了实战经验。

## 四、volatile的适用场景与典型模式 🟡

### 4.1 适用场景一：状态标志

```java
volatile boolean shutdownRequested = false;

public void shutdown() {
    shutdownRequested = true;
}

public void doWork() {
    while (!shutdownRequested) {
        // 执行业务逻辑
    }
}
```

这是volatile最标准的使用场景——**一个线程修改状态，多个线程读取该状态，且修改和读取都是简单的赋值/条件判断**。

### 4.2 适用场景二：双重检查锁定

```java
private static volatile Singleton instance;

public static Singleton getInstance() {
    if (instance == null) {
        synchronized (Singleton.class) {
            if (instance == null) {
                instance = new Singleton(); // 禁止重排的关键点
            }
        }
    }
    return instance;
}
```

`instance = new Singleton()`在字节码层面是三个操作：分配内存、调用构造函数、将引用赋值给instance。没有volatile时，JIT可能重排为：分配内存→赋值引用→调用构造函数。第三步（构造）还没执行完时，另一个线程看到instance!=null就拿去用了，导致NPE。

volatile通过StoreLoad屏障禁止了这个重排。

### 4.3 适用场景三：保证引用对象的可见性

```java
volatile Person person;

public void update(Person newPerson) {
    person = newPerson; // 引用赋值本身是原子的
}

public void print() {
    Person p = person;
    if (p != null) {
        System.out.println(p.getName());
    }
}
```

volatile可以保证引用赋值对其他线程立即可见。但注意：**person指向的对象内部的字段不受volatile保护**，如果其他线程同时修改person指向的对象的内容，仍需要额外的同步。

### 4.4 不适用场景

```java
// ❌ 错误：复合操作
volatile int counter = 0;
counter++; // 不安全！需要 AtomicInteger

// ❌ 错误：读-改-写序列
if (map.containsKey(key)) {  // volatile读
    map.get(key);             // volatile读
} else {
    map.put(key, value);      // volatile写，但前后没有原子性
}

// ❌ 错误：需要同步多个变量
volatile int a = 0;
volatile int b = 0;

public void update() {
    a = 1;
    b = 2;
}

public void check() {
    if (b == 2 && a == 0) { // 可能看到a=0但b=2的状态（单volatile无跨变量保证）
        // ...
    }
}
```

## 五、生产避坑：volatile的隐藏陷阱 ⚠️

### 5.1 long和double的volatile

JMM规定，对非volatile的`long`和`double`的读写**不是原子**的（因为它们是64位，CPU可能分两次32位操作）。

```java
volatile long bigValue = 0L; // 原子读写
long anotherValue = 0L;       // 非原子读写
```

如果`anotherValue`被两个线程并发读写，可能出现"只写了一半"的情况——一个线程写入了高32位，另一个写入了低32位，导致一个完全错误的值。

### 5.2 JIT优化的"惊喜"

如前所述，JIT可能在运行时做激进的优化。`volatile`虽然能阻止某些重排，但JIT的循环不变式外提是另一个维度的问题。

在高频读写volatile变量的场景下（比如一个纳秒级的高频交易系统），volatile的StoreLoad屏障开销会成为瓶颈。

### 5.3 x86 vs ARM的差异

x86的StoreLoad屏障需要MFENCE指令，代价不小。但在x86上volatile读几乎无额外开销（因为x86的MESI是强一致性模型）。

ARM架构下，volatile读也需要显式的加载屏障（DMB LD），volatile读写都有开销。所以在高并发+跨平台（移动端/ARM服务器）的场景下，volatile的性能影响需要更谨慎地评估。

## 六、面试应对脚本

**面试官**："volatile能保证原子性吗？"
**标准回答**："volatile只保证volatile变量本身的读写是原子的，但不保证复合操作的原子性。比如i++，它包含读取→加1→写入三步，volatile只能保证每一步单独是原子的，但两步之间可能被其他线程插入。所以i++这种read-modify-write操作必须用AtomicInteger.getAndIncrement()或synchronized来保证原子性。"

**面试官追问**："volatile和synchronized的区别是什么？"
**标准回答**："volatile是轻量级的同步，它只保证可见性和有序性，不保证原子性；synchronized是重量级的同步，它保证原子性、可见性和有序性。从性能角度，volatile的读写开销极低（x86上volatile读几乎没有额外开销），但synchronized在有锁竞争时有显著的上下文切换开销。使用场景上，volatile只适合单个变量的简单读写，synchronized适合复杂的复合操作。"

**面试官追问**："为什么DCL单例一定要加volatile？"
**标准回答**："因为new Singleton()在字节码层面是三个操作——分配内存、调用构造函数、将引用赋值给instance。JIT可能重排为分配内存→赋值引用→调用构造函数。如果另一个线程在构造函数执行前就看到了非null的引用，它会拿到一个未初始化完成的对象。加上volatile后，StoreLoad屏障禁止了构造函数和instance赋值的重排，保证其他线程只有在构造函数完成后才能看到instance的非null值。"

【面试官心理】
volatile这道题，能从可见性讲到指令重排、再讲到内存屏障、再讲到DCL单例的候选人，已经建立了一个完整的知识链条。这个链条的完整性决定了候选人处理并发问题的深度——只背API的人遇到真实的可见性bug时会手足无措，但理解原理的人会直接定位到Store Buffer或JIT优化的根因。
