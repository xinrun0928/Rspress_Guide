# 线程池execute流程：任务是如何被调度执行的

候选人小李在面试中被问到这样一个问题：

面试官问："线程池的execute()方法是怎么执行的？"

小李说："新任务提交后，线程池会创建新线程或者把任务放到队列里。"

面试官追问："那具体是什么流程？线程是怎么被创建和复用的？"

小李说："...有一个worker来获取任务？"

面试官继续追问："那你说说worker是如何实现线程复用的？它是怎么阻塞等待任务的？"

小李答不上来。

【面试官心理】
这道题考核的是候选人对线程池底层执行流程的理解深度。execute()的流程是Java并发编程中最核心的流程之一——它涉及线程创建、任务调度、队列排队、线程复用的完整链路。能把这个流程讲清楚的候选人，在处理线程池调优、问题排查时会有更强的分析能力。

## 一、execute()的完整执行流程 🔴

### 1.1 流程总览

```java
public void execute(Runnable command) {
    if (command == null) throw new NullPointerException();

    int c = ctl.get(); // 获取线程池状态

    // ① 核心线程数检查
    if (workerCountOf(c) < corePoolSize) {
        // 添加核心线程
        if (addWorker(command, true)) // core=true
            return;
        c = ctl.get();
    }

    // ② 队列检查
    if (isRunning(c) && workQueue.offer(command)) {
        // 入队成功，再次检查状态
        int recheck = ctl.get();
        if (!isRunning(recheck) && remove(command))
            reject(command); // 非运行状态，移除任务
        else if (workerCountOf(recheck) == 0)
            addWorker(null, false); // 没有worker，创建临时线程
        return;
    }

    // ③ 尝试创建非核心线程
    else if (!addWorker(command, false)) // core=false
        reject(command); // 线程池已关闭或达到上限，拒绝任务
}
```

### 1.2 三段式流程图

```
任务到达
    ↓
workerCount < corePoolSize？
    ├─ 是 → addWorker(command, true) → 创建核心线程执行
    └─ 否
        ↓
    队列offer成功？
        ├─ 是 → 再次检查状态（防止状态变化）
        │         ├─ 非运行状态 → remove + reject
        │         └─ 没有worker → addWorker(null) 防止线程泄漏
        └─ 否
            ↓
        addWorker(command, false)？
            ├─ 成功 → 创建非核心线程执行
            └─ 失败 → reject() 触发拒绝策略
```

## 二、Worker线程的创建与复用 🔴

### 2.1 Worker是什么

Worker是ThreadPoolExecutor的内部类，封装了一个工作线程：

```java
private final class Worker extends AbstractQueuedSynchronizer
    implements Runnable {

    final Thread thread;        // 实际执行任务的线程
    Runnable firstTask;          // 第一个任务（可以为空）
    long completedCount;         // 该worker完成的任务数

    Worker(Runnable firstTask) {
        setState(-1); // 禁止在构造函数中获取锁
        this.firstTask = firstTask;
        this.thread = getThreadFactory().newThread(this); // 创建线程
    }

    public void run() {
        runWorker(this); // 线程启动后执行的逻辑
    }
}
```

### 2.2 addWorker()的实现

```java
private boolean addWorker(Runnable firstTask, boolean core) {
    retry:
    for (;;) {
        int c = ctl.get();
        int rs = runStateOf(c); // 线程池状态

        // ① 状态检查：非Running状态不接受新任务
        if (rs >= SHUTDOWN && !(rs == SHUTDOWN && firstTask == null && !workQueue.isEmpty()))
            return false;

        // ② 线程数量检查
        for (;;) {
            int wc = workerCountOf(c);
            // core=true时检查corePoolSize，core=false时检查maximumPoolSize
            if (wc >= CAPACITY || wc >= (core ? corePoolSize : maximumPoolSize))
                return false;

            // CAS增加线程计数
            if (compareAndIncrementWorkerCount(c))
                break retry;
            c = ctl.get();
        }
    }

    // ③ 创建Worker并启动
    Worker w = null;
    try {
        w = new Worker(firstTask);
        final Thread t = w.thread;
        if (t != null) {
            mainLock.lock();
            try {
                // 添加到workers集合
                workers.add(w);
                int s = workers.size();
                if (s > largestPoolSize)
                    largestPoolSize = s;
            } finally {
                mainLock.unlock();
            }
            t.start(); // 启动线程
        }
    } finally {
        if (w == null || w.thread == null)
            fail ...
    }
    return true;
}
```

### 2.3 runWorker()：线程复用的核心循环

```java
final void runWorker(Worker w) {
    Thread wt = Thread.currentThread();
    Runnable task = w.firstTask;
    w.firstTask = null;
    w.unlock(); // 允许中断

    boolean completedAbruptly = true;
    try {
        // 关键：while循环 + getTask()
        while (task != null || (task = getTask()) != null) {
            // 获取任务后，执行前
            w.lock();

            // 确保线程池未被关闭
            if ((runStateAtLeast(ctl.get(), STOP) ||
                 (Thread.interrupted() && runStateAtLeast(ctl.get(), STOP)))
                 && !wt.isInterrupted())
                wt.interrupt();

            try {
                beforeExecute(wt, task); // 钩子方法
                Throwable thrown = null;
                try {
                    task.run(); // 执行任务！
                } catch (RuntimeException x) {
                    thrown = x;
                    throw x;
                } finally {
                    afterExecute(task, thrown);
                }
            } finally {
                task = null;
                w.completedCount++;
                w.unlock();
            }
        }
        completedAbruptly = false;
    } finally {
        processWorkerExit(w, completedAbruptly); // 处理worker退出
    }
}
```

### 2.4 getTask()：阻塞获取任务

```java
private Runnable getTask() {
    boolean timedOut = false; // 是否刚刚超时

    for (;;) {
        int c = ctl.get();
        int rs = runStateOf(c);

        // 队列为空且线程池正在停止
        if (rs >= SHUTDOWN && (rs >= STOP || workQueue.isEmpty())) {
            decrementWorkerCount();
            return null; // worker退出
        }

        int wc = workerCountOf(c);

        // 是否需要超时回收
        boolean timed = allowCoreThreadTimeOut || wc > corePoolSize;

        if ((wc > maximumPoolSize || (timed && timedOut))
            && (wc > 1 || workQueue.isEmpty())) {
            if (compareAndDecrementWorkerCount(c))
                return null; // worker退出
            continue;
        }

        try {
            // 关键：从队列中取任务
            Runnable r = timed ?
                workQueue.poll(keepAliveTime, TimeUnit.NANOSECONDS) : // 超时等待
                workQueue.take(); // 阻塞等待
            if (r != null)
                return r;
            timedOut = true; // poll超时了
        } catch (InterruptedException retry) {
            timedOut = false;
        }
    }
}
```

**线程复用的关键**：`while(task != null || (task = getTask()) != null)` 循环。只要`getTask()`能返回任务，worker就不会退出，线程就会被复用。

**阻塞机制**：
- 核心线程（未开启allowCoreThreadTimeOut）：调用`workQueue.take()`，阻塞等待直到有任务
- 非核心线程（或开启allowCoreThreadTimeOut）：调用`workQueue.poll(keepAliveTime)`，超时等待

## 三、拒绝策略的触发时机 🟡

### 3.1 何时触发拒绝

```java
// 拒绝策略的触发时机只有两个：

// 时机一：线程池非Running状态，addWorker失败
if (!isRunning(c) && remove(command))
    reject(command);

// 时机二：线程数已达maximumPoolSize，队列已满
else if (!addWorker(command, false))
    reject(command); // 线程池已满且队列已满
```

### 3.2 拒绝的执行过程

```java
final void reject(Runnable command) {
    handler.rejectedExecution(command, this);
}
```

四种内置策略：
- `AbortPolicy`：抛出RejectedExecutionException
- `CallerRunsPolicy`：由调用者线程执行
- `DiscardOldestPolicy`：丢弃最老的任务
- `DiscardPolicy`：丢弃新任务

## 四、生产避坑：execute的常见问题 ⚠️

### 4.1 任务抛出异常导致线程退出

```java
// ❌ 危险：未捕获异常导致线程消失
executor.execute(() -> {
    if (true) throw new RuntimeException("Oops!");
});
// 如果workerCountOf(c) >= corePoolSize，这个异常会导致worker突然退出
// 但线程池会立即创建新的worker，所以线程数不会减少

// ✅ 正确：用Future捕获异常
Future<?> future = executor.submit(() -> {
    // 任务代码
});
try {
    future.get();
} catch (ExecutionException e) {
    log.error("任务执行失败", e);
}
```

### 4.2 核心线程不会被回收的陷阱

```java
// 默认：核心线程即使空闲也不会被回收
executor.allowCoreThreadTimeOut(true); // 显式开启
// 开启后，核心线程也会在keepAliveTime超时后被回收
```

## 五、❌ 错误示范

**候选人原话**："线程池用完后就自动关闭了。"

**问题诊断**：
- 线程池默认不会自动关闭
- 需要调用shutdown()或shutdownNow()才会关闭
- 忘记shutdown是内存泄漏的常见原因

**候选人原话**："addWorker成功后线程会立即执行任务。"

**问题诊断**：
- addWorker创建Worker并启动线程
- 但线程在runWorker中通过getTask()获取任务
- 如果队列为空，线程会阻塞在getTask()中

【面试官心理】
能回答runWorker的while循环和getTask()阻塞机制的候选人，说明他对线程复用有了深入理解。线程复用的核心就是worker线程从队列中不断获取任务——只要队列有任务，worker就不会退出。这和协程的"挂起/恢复"有异曲同工之妙，但线程复用的是"阻塞/唤醒"。

## 六、面试应对脚本

**面试官**："线程池的execute()流程是什么？"
**标准回答**："execute()分为三个阶段。第一阶段：检查workerCount是否小于corePoolSize，如果是则addWorker(command, true)创建核心线程执行任务。第二阶段：如果核心线程已满，尝试将任务offer到阻塞队列。如果入队成功，再次检查线程池状态——如果非Running则remove任务并拒绝；如果核心线程为0则创建临时线程。第三阶段：如果队列已满或入队失败，尝试addWorker(command, false)创建非核心线程。如果创建失败，触发拒绝策略。"

**面试官追问**："worker线程是如何复用执行多个任务的？"
**标准回答**："worker线程在runWorker中用while循环不断调用getTask()获取任务。核心线程调用workQueue.take()阻塞等待，非核心线程调用workQueue.poll(keepAliveTime)超时等待。只要getTask()返回任务，worker就一直执行；getTask()返回null时，worker退出循环，调用processWorkerExit()。这就是线程复用的核心：worker不退出，通过getTask()从队列中不断取任务执行。"

【面试官心理】
能回答execute流程和worker复用机制的候选人，对线程池的理解已经非常深入了。这类候选人在排查"为什么线程池的线程数没有达到最大线程数"、"为什么核心线程空闲时不会被回收"等问题时，能直接定位到getTask()的阻塞逻辑和allowCoreThreadTimeOut的配置。
