# CAS 原理与三大问题

在 Java 并发编程中，`synchronized` 和 `Lock` 是最常用的锁机制，但它们都是**悲观锁**——假设并发一定会产生冲突，所以在操作前先加锁。**CAS（Compare-And-Swap）**则是一种**乐观锁**——假设没有冲突，直接进行操作，只有在检测到冲突时才重试。CAS 是 Java 并发包的基石，`AtomicInteger`、`AtomicReference`、`ConcurrentHashMap` 等线程安全类都依赖 CAS 实现。今天我们就来深入理解 CAS 的原理及其三大问题。

## CAS 的基本概念

### 什么是 CAS？

CAS 操作包含三个参数：
- **内存位置**（V）：要操作的变量地址
- **期望值**（E）：认为当前应该的值
- **新值**（N）：要设置的新值

```java
// CAS 的语义：
// 如果内存位置 V 的值等于期望值 E
// 则将内存位置 V 的值设置为新值 N
// 否则，不做任何操作
// 返回是否设置成功
```

### CAS 的伪代码

```java
public class CASConcept {
    
    // CAS 的逻辑
    public boolean compareAndSwap(Object obj, long offset, 
                                 int expected, int newValue) {
        // 读取当前值
        int current = getInt(obj, offset);
        
        // 比较
        if (current == expected) {
            // 相等，设置为新值
            setInt(obj, offset, newValue);
            return true;
        } else {
            // 不相等，不做任何操作
            return false;
        }
    }
    
    // 使用 CAS 实现原子递增
    public int incrementAndGet(Object obj, long offset) {
        int oldValue, newValue;
        do {
            oldValue = getInt(obj, offset);
            newValue = oldValue + 1;
        } while (!compareAndSwap(obj, offset, oldValue, newValue));
        
        return newValue;
    }
}
```

### CAS 的 CPU 指令支持

CAS 是 CPU 提供的原子指令：

| 架构 | 指令 |
|------|------|
| x86/x64 | `lock cmpxchg` |
| ARM | `ldaxr` + `stlxr` |
| PowerPC | `lwarx` + `stwcx` |

```asm
; x86 架构的 CAS 实现
lock cmpxchg [esi], edi
; 如果 [esi] == eax，用 edi 替换 [esi]
; 否则，将 [esi] 加载到 eax
```

## CAS 在 Java 中的使用

### Java 对象字段的偏移量

```java
import sun.misc.Unsafe;
import java.lang.reflect.Field;

public class UnsafeDemo {
    
    private static Unsafe unsafe;
    private volatile long valueOffset;
    
    static {
        try {
            Field f = Unsafe.class.getDeclaredField("theUnsafe");
            f.setAccessible(true);
            unsafe = (Unsafe) f.get(null);
            
            // 获取 value 字段在 AtomicLong 中的偏移量
            valueOffset = unsafe.objectFieldOffset(
                AtomicLong.class.getDeclaredField("value")
            );
        } catch (Exception e) {
            throw new RuntimeException(e);
        }
    }
    
    public static void main(String[] args) {
        AtomicLong atomic = new AtomicLong(100);
        System.out.println("Value offset: " + valueOffset);
        
        // 使用 unsafe 进行 CAS 操作
        long expected = 100L;
        long newValue = 200L;
        boolean success = unsafe.compareAndSwapLong(
            atomic, valueOffset, expected, newValue
        );
        System.out.println("CAS success: " + success);
        System.out.println("Value: " + atomic.get());
    }
}
```

### AtomicInteger 的实现

```java
public class AtomicInteger extends Number implements java.io.Serializable {
    
    private volatile int value;
    
    // CAS 操作
    private static final long valueOffset;
    
    static {
        try {
            valueOffset = unsafe.objectFieldOffset(
                AtomicInteger.class.getDeclaredField("value")
            );
        } catch (Exception e) {
            throw new Error(e);
        }
    }
    
    // 获取并递增
    public final int incrementAndGet() {
        return unsafe.getAndAddInt(this, valueOffset, 1) + 1;
    }
    
    // compareAndSet
    public final boolean compareAndSet(int expectedValue, int newValue) {
        return unsafe.compareAndSwapInt(
            this, valueOffset, expectedValue, newValue
        );
    }
}
```

## CAS 的三大问题

### 问题一：ABA 问题

#### 什么是 ABA 问题？

CAS 只检查"值是否变化"，但不能检测值是否被修改后再改回来：

```java
public class ABAProblem {
    
    public static void main(String[] args) {
        AtomicInteger ref = new AtomicInteger(100);
        
        // 线程 A 读取 value = 100
        int valueA = ref.get();
        
        // 线程 B 修改：100 → 101 → 100
        new Thread(() -> {
            ref.compareAndSet(100, 101);  // 100 → 101
            ref.compareAndSet(101, 100);  // 101 → 100
        }).start();
        
        // 线程 A 尝试修改：100 → 200
        // CAS 检查：期望 100，当前 100，相等！修改成功！
        ref.compareAndSet(100, 200);  // 成功，但中间经历了变化！
    }
}
```

#### ABA 问题的危害

```java
public class ABAProblemScenario {
    
    // 栈的实现（简化版）
    static class Stack<E> {
        AtomicReference<Node<E>> top = new AtomicReference<>();
        
        public boolean push(E item) {
            Node<E> newNode = new Node<>(item);
            Node<E> oldTop;
            do {
                oldTop = top.get();
                newNode.next = oldTop;
            } while (!top.compareAndSet(oldTop, newNode));  // CAS
            return true;
        }
    }
    
    // ABA 问题导致的问题：
    // 如果线程 A 读取 top = Node1
    // 线程 B: Node1 → Node2 → Node1（又变回来了）
    // 线程 A 以为栈没变化，继续执行 CAS，成功了
    // 但实际上栈已经被修改过了！
}
```

#### 解决方案：版本号

```java
public class AtomicStampedReferenceDemo {
    
    // 使用版本号解决 ABA 问题
    public void solveABA() {
        // 每个操作都附带版本号
        AtomicStampedReference<Integer> ref = 
            new AtomicStampedReference<>(100, 1);
        
        int[] stamp = new int[1];
        
        // 读取当前值和版本号
        Integer value = ref.get(stamp);
        int currentStamp = stamp[0];
        
        // 修改：只有版本号匹配时才能成功
        ref.compareAndSet(
            100,          // 期望值
            200,          // 新值
            currentStamp, // 期望版本号
            currentStamp + 1  // 新版本号
        );
    }
}
```

### 问题二：循环开销大

#### 长时间自旋

在高竞争环境下，CAS 可能长时间失败，导致 CPU 空转：

```java
public class CASRetryProblem {
    
    public void highContention() {
        AtomicInteger counter = new AtomicInteger(0);
        
        // 1000 个线程同时竞争
        ExecutorService executor = Executors.newFixedThreadPool(1000);
        
        for (int i = 0; i < 10000; i++) {
            executor.submit(() -> {
                // CAS 失败时，会一直重试
                counter.incrementAndGet();  // getAndAddInt 内部自旋
            });
        }
        
        // 在高竞争下，大量 CAS 操作会失败
        // 导致线程不断自旋，浪费 CPU 资源
    }
}
```

#### 解决方案

```java
public class ReduceRetry {
    
    // 解决方案1：使用 LongAdder（分段锁）
    public void useLongAdder() {
        LongAdder adder = new LongAdder();
        adder.increment();  // 分段计数，减少竞争
    }
    
    // 解决方案2：限制重试次数
    public void limitedRetry() {
        AtomicInteger counter = new AtomicInteger(0);
        int maxRetries = 10;
        int retries = 0;
        
        while (retries < maxRetries) {
            int current = counter.get();
            if (counter.compareAndSet(current, current + 1)) {
                break;
            }
            retries++;
        }
        
        // 超过重试次数，执行备选方案
        if (retries >= maxRetries) {
            // fallback
        }
    }
}
```

### 问题三：只能保证单个变量的原子性

#### 复合操作的问题

CAS 只能保证单个变量的原子性，不能保证复合操作的原子性：

```java
public class SingleVariableProblem {
    
    // 问题：CAS 不能保证复合操作的原子性
    public void wrongUsage() {
        AtomicInteger balance = new AtomicInteger(100);
        
        // 复合操作：检查余额 → 取款
        // 不是原子操作！
        if (balance.get() >= 100) {
            balance.addAndGet(-100);  // 两个线程可能都通过检查
        }
    }
    
    // 解决方案1：使用 synchronized
    public void synchronizedSolution() {
        AtomicInteger balance = new AtomicInteger(100);
        
        synchronized (balance) {
            if (balance.get() >= 100) {
                balance.addAndGet(-100);
            }
        }
    }
    
    // 解决方案2：使用死循环 CAS
    public void casLoopSolution() {
        AtomicInteger balance = new AtomicInteger(100);
        
        while (true) {
            int current = balance.get();
            if (current >= 100) {
                if (balance.compareAndSet(current, current - 100)) {
                    break;  // 成功
                }
                // 失败，重试
            } else {
                break;  // 余额不足
            }
        }
    }
}
```

## CAS 与 synchronized 的对比

```java
public class CASvsSynchronized {
    
    // synchronized：悲观锁
    public synchronized void syncIncrement() {
        counter++;
    }
    
    // CAS：乐观锁
    public void casIncrement() {
        int current;
        do {
            current = counter.get();
        } while (!counter.compareAndSet(current, current + 1));
    }
}
```

| 特性 | CAS | synchronized |
|------|-----|--------------|
| 并发模型 | 乐观锁 | 悲观锁 |
| 实现方式 | CPU 原子指令 | OS 互斥量 |
| 性能（低竞争） | 高（无阻塞） | 中等 |
| 性能（高竞争） | 低（大量重试） | 高（线程阻塞） |
| 适用场景 | 简单操作 | 复杂复合操作 |
| ABA 问题 | 有 | 无 |

## 总结

CAS 是 Java 并发的基础：

- **原理**：比较并交换，CPU 原子指令支持
- **优点**：无锁实现，避免线程阻塞
- **ABA 问题**：解决方案是版本号（AtomicStampedReference）
- **循环开销**：高竞争下性能下降，可考虑 LongAdder
- **单一变量**：只能保证单个变量的原子性
