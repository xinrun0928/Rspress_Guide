# CAS 原理与三大问题：ABA、循环开销、范围限制

线程 A 和线程 B 同时修改同一个变量 `value`，初始值为 `0`。

线程 A 读取 `value = 0`，准备改成 `1`，但就在它准备写入之前，线程 B 先一步把 `value` 改成了 `1`，然后又改回了 `0`。

线程 A 继续写入，它看到 `value` 还是 `0`，写入成功。

**这就是经典的 ABA 问题。**

在 Java 并发编程中，CAS（Compare-And-Swap）是实现无锁算法的核心。理解 CAS 的原理和问题，是掌握 `AtomicInteger`、`AtomicReference` 等原子类的必经之路。

## CAS 的原理

### 硬件级别的支持

CAS 是一条 CPU 原子指令，不同架构有不同的实现：

| 架构 | CAS 指令 |
| --- | --- |
| x86/ARM | `cmpxchg`（x86）、`ldrex/strex`（ARM） |
| 多核系统 | 需要 `lock` 前缀保证缓存一致性 |
| JVM | 在 HotSpot 中封装为 `Atomic::cmpxchg` |

```cpp
// HotSpot 中的 CAS 实现（简化）
int Atomic::cmpxchg(int exchange_value, volatile int* dest,
                   int compare_value) {
    int oldv = *dest;
    if (oldv == compare_value) {
        *dest = exchange_value;
    }
    return oldv;
}
```

关键在于：`compare` 和 `swap` 是**不可分割的原子操作**。

### Java 中的 CAS

Java 通过 `sun.misc.Unsafe` 提供 CAS 操作：

```java
public final class AtomicInteger extends Number {
    private volatile int value;

    // 获取 Unsafe 实例
    private static final Unsafe unsafe = Unsafe.getUnsafe();

    // value 字段的内存偏移量
    private static final long valueOffset;

    static {
        valueOffset = unsafe.objectFieldOffset(AtomicInteger.class.getDeclaredField("value"));
    }

    // CAS 操作
    public final boolean compareAndSet(int expectedValue, int newValue) {
        return unsafe.compareAndSwapInt(this, valueOffset, expectedValue, newValue);
    }

    // 自增
    public final int incrementAndGet() {
        return unsafe.compareAndSwapInt(this, valueOffset, value, value + 1) ? value + 1 : incrementAndGet();
    }
}
```

### incrementAndGet 的自旋实现

```java
public final int incrementAndGet() {
    for (;;) {
        int current = get();           // 1. 获取当前值
        int next = current + 1;        // 2. 计算新值
        if (compareAndSet(current, next)) {  // 3. CAS 原子更新
            return next;              // 4. 成功，返回新值
        }
        // 失败，循环重试
    }
}
```

这是一种典型的**乐观锁**策略：假设没有冲突，直接更新；如果失败，重试。

## CAS 的三大问题

### 问题一：ABA 问题

**问题描述**：变量的值从 A 变成 B，又变回 A，CAS 认为没有变化，但实际发生了变化。

```java
public class ABADemo {
    public static void main(String[] args) {
        AtomicInteger ref = new AtomicInteger(1);

        // 线程 A
        new Thread(() -> {
            int value = ref.get();       // value = 1
            System.out.println("线程A读取: " + value);

            // 模拟其他线程的修改
            try { Thread.sleep(100); } catch (InterruptedException ignored) {}

            // 线程 A 尝试 CAS
            boolean success = ref.compareAndSet(value, 2);
            System.out.println("线程A CAS结果: " + success);
        }).start();

        // 线程 B
        new Thread(() -> {
            ref.compareAndSet(1, 2);     // 1 -> 2
            System.out.println("线程B修改: 1 -> 2");

            ref.compareAndSet(2, 1);     // 2 -> 1
            System.out.println("线程B修改: 2 -> 1");
        }).start();
    }
}
```

**潜在风险**：在某些场景下，ABA 会导致逻辑错误。

```java
// 栈的 push 操作，ABA 问题导致链表断裂
Stack stack = new Stack();
stack.push(nodeA);  // A.next = null

// 线程1：读取 top = A，准备 push B
// 线程2：A -> B，然后 B -> A（ABA）
// 线程1：CAS(top, A, B)，成功
// 但此时 A.next 已经被其他线程修改！
```

**解决方案**：使用版本号或时间戳。`AtomicStampedReference` 和 `AtomicMarkableReference` 就是为此设计的。

### 问题二：循环开销大

**问题描述**：在高并发场景下，CAS 失败率很高，导致大量线程自旋重试，浪费 CPU 资源。

```java
// 高并发下的自旋
public class CASContention {
    public static void main(String[] args) throws InterruptedException {
        AtomicInteger counter = new AtomicInteger(0);

        long start = System.currentTimeMillis();

        // 1000 个线程同时竞争
        IntStream.range(0, 1000).parallel().forEach(i ->
            IntStream.range(0, 10000).forEach(j ->
                counter.incrementAndGet()  // 可能有大量重试
            )
        );

        long end = System.currentTimeMillis();
        System.out.println("耗时: " + (end - start) + "ms");
        System.out.println("结果: " + counter.get());
    }
}
```

**性能影响**：

```
并发度    CAS 失败率    自旋次数    CPU 消耗
低        ~10%         少          低
中        ~30%         中          中
高        ~70%         多          高
```

**优化策略**：

1. **使用 LongAdder 替代 AtomicLong**

```java
// JDK 8 引入，专门解决高并发下的 CAS 竞争
LongAdder adder = new LongAdder();
for (int i = 0; i < 100000; i++) {
    adder.increment();  // 内部数组分段，减少竞争
}
long result = adder.sum();
```

2. **减少竞争粒度**

```java
// 差：单一计数器
AtomicInteger global = new AtomicInteger(0);

// 好：分段计数器
LongAdder local = new LongAdder();
```

### 问题三：范围限制

**问题描述**：CAS 只能保证单个变量的原子性，对于复合操作无能为力。

```java
// CAS 无法保证这些操作的原子性
counter.incrementAndGet();        // i++（读取-修改-写入）
map.putIfAbsent(key, value);     // 检查后插入
list.remove(element);            // 查找后删除
```

**例子**：`AtomicInteger` 的 `compareAndSet` 只能基于当前值判断：

```java
public class RangeLimitDemo {
    public static void main(String[] args) {
        AtomicInteger balance = new AtomicInteger(100);

        // 场景：余额扣款，不能为负
        // CAS 方式（有问题）
        boolean tryDeduct(int amount) {
            int current = balance.get();
            if (current >= amount) {
                // 问题：检查和更新之间，balance 可能已被其他线程修改
                return balance.compareAndSet(current, current - amount);
            }
            return false;
        }
    }
}
```

**正确的做法**：使用循环 CAS 或改用锁：

```java
public class SafeDeduct {
    private final AtomicInteger balance;

    public SafeDeduct(AtomicInteger balance) {
        this.balance = balance;
    }

    // 循环 CAS + 重试
    public boolean tryDeduct(int amount) {
        for (;;) {
            int current = balance.get();
            if (current < amount) {
                return false;
            }
            int newValue = current - amount;
            if (balance.compareAndSet(current, newValue)) {
                return true;
            }
            // CAS 失败，重试
        }
    }

    // 或者：使用 synchronized（简单但有性能开销）
    public synchronized boolean deduct(int amount) {
        if (balance.get() >= amount) {
            balance.set(balance.get() - amount);
            return true;
        }
        return false;
    }
}
```

## CAS 与 synchronized 的对比

| 特性 | CAS | synchronized |
| --- | --- | --- |
| 实现方式 | 硬件指令 | JVM 内置锁 |
| 原理 | 乐观并发 | 悲观并发 |
| 阻塞 | 非阻塞，自旋重试 | 阻塞线程 |
| 公平性 | 非公平 | 可配置 |
| 粒度 | 单变量 | 代码块/方法 |
| 适用场景 | 低竞争、高并发 | 高竞争、数据复杂 |

## 总结

CAS 是 Java 并发编程的基石：

- **原理**：硬件级别的 compare-and-swap 指令，保证读-改-写的原子性
- **ABA 问题**：通过 `AtomicStampedReference` 或 `AtomicMarkableReference` 解决
- **循环开销**：高并发下可考虑 `LongAdder` 或减少竞争
- **范围限制**：只能保证单变量复合操作的原子性

理解 CAS 的本质和局限性，才能在合适的场景选择合适的并发工具。
