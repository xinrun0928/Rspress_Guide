# Java 异步编程模式总结

从 Callback 到 CompletableFuture，Java 异步编程经历了怎样的演进？

如果你是 Java 新手，可能觉得 CompletableFuture 已经够用了。但如果你是 Java 老兵，可能还记得那些 Callback 地狱的日子。

每种异步模式都有它的适用场景，没有银弹。理解它们的优缺点，才能在合适的场景选择合适的方案。

这篇文章会对比四种主流异步编程模式：Callback、Future、CompletableFuture、响应式编程（Reactive），帮助你建立完整的异步编程知识体系。

## 为什么需要异步编程

### 同步 vs 异步

```java
// 同步：调用方等待被调用方完成
String result = syncCall();  // 阻塞在这里
System.out.println(result);

// 异步：调用方不等待，继续执行
asyncCall(result -> {        // 立即返回，结果通过回调通知
    System.out.println(result);
});
System.out.println("我继续执行");
```

**为什么需要异步？**

| 场景 | 同步的问题 | 异步的优势 |
|---|---|---|
| I/O 操作 | 线程阻塞，浪费 CPU | 线程不阻塞，提高吞吐 |
| RPC 调用 | 调用链长，总耗时 = sum(每步) | 可以并行，总耗时 = max(每步) |
| 用户界面 | 界面卡顿 | 界面流畅响应 |
| 微服务调用 | 阻塞等待超时 | 超时控制、资源保护 |

## 模式一：Callback

### 基本模式

Callback 是最早的异步编程模式：

```java
public interface Callback<T> {
    void onSuccess(T result);
    void onError(Throwable error);
}

public class AsyncService {
    public void asyncOperation(String input, Callback<String> callback) {
        new Thread(() -> {
            try {
                String result = doOperation(input);
                callback.onSuccess(result);
            } catch (Exception e) {
                callback.onError(e);
            }
        }).start();
    }
}

// 使用
asyncService.asyncOperation("input", new Callback<String>() {
    @Override
    public void onSuccess(String result) {
        System.out.println("成功: " + result);
    }

    @Override
    public void onError(Throwable error) {
        System.out.println("失败: " + error.getMessage());
    }
});
```

### Lambda 简化

```java
asyncService.asyncOperation("input",
    result -> System.out.println("成功: " + result),
    error -> System.out.println("失败: " + error.getMessage())
);
```

### Callback 的问题：回调地狱

当多个异步操作需要顺序执行时：

```java
// 回调地狱
asyncService.getUser(userId, userResult -> {
    asyncService.getOrders(userResult.getId(), ordersResult -> {
        asyncService.getProducts(ordersResult, productsResult -> {
            asyncService.calculateRecommendations(productsResult, recResult -> {
                // 终于到了业务逻辑！
                display(recResult);
            }, e -> handleError(e));
        }, e -> handleError(e));
    }, e -> handleError(e));
}, e -> handleError(e));

// 问题：
// 1. 代码嵌套层数多，难以阅读
// 2. 错误处理需要重复
// 3. 变量作用域混乱
// 4. 难以测试
```

### Callback 的适用场景

| 优点 | 缺点 |
|---|---|
| 简单直观 | 回调地狱 |
| 内存开销小 | 错误处理复杂 |
| 最早的支持 | 无法组合 |
| 事件驱动友好 | 堆栈追踪困难 |

## 模式二：JDK Future

### 基本模式

JDK 5 引入了 `Future`，将异步结果封装为一个容器：

```java
public interface Future<V> {
    V get() throws InterruptedException, ExecutionException;
    V get(long timeout, TimeUnit unit) throws ...;
    boolean isDone();
    boolean isCancelled();
    boolean cancel(boolean mayInterruptIfRunning);
}
```

### 使用示例

```java
ExecutorService executor = Executors.newFixedThreadPool(4);

Future<String> future = executor.submit(() -> {
    // 模拟耗时操作
    Thread.sleep(1000);
    return "Result";
});

// 阻塞等待
try {
    String result = future.get();  // 阻塞
    System.out.println(result);
} catch (ExecutionException e) {
    System.out.println("失败: " + e.getCause());
}
```

### Future 的问题

**问题一：阻塞式获取结果**

```java
Future<String> future = executor.submit(() -> getFromDatabase());
Future<String> future2 = executor.submit(() -> getFromCache());

// 串行获取，不能并行！
String result = future.get();   // 阻塞在这里
String result2 = future2.get();  // 如果这个先完成，还是要等

// 正确的并行方式（但很复杂）
// 需要自己管理线程、等待
```

**问题二：无法组合**

```java
// 想做：future1 完成后，用结果调用 future2
Future<String> result1 = executor.submit(() -> getUserId());
Future<String> result2 = executor.submit(() -> {
    // 无法获取 result1 的结果来计算
    return getUserName();
});

// 需要手动管理依赖
```

**问题三：没有异常传播**

```java
Future<String> future = executor.submit(() -> {
    throw new RuntimeException("error");
});

try {
    future.get();  // 异常被包装在 ExecutionException 中
} catch (ExecutionException e) {
    System.out.println(e.getCause().getMessage());  // 需要手动处理
}
```

### Future 的适用场景

| 优点 | 缺点 |
|---|---|
| 标准接口 | 阻塞式 get() |
| 任务取消 | 无法组合 |
| 简单场景够用 | 无回调机制 |
| 广泛支持 | 无法链式调用 |

## 模式三：CompletableFuture

### 为什么需要 CompletableFuture

Future 的问题催生了 CompletableFuture。JDK 8 引入的 CompletableFuture 解决了这些问题：

```java
CompletableFuture<String> future = CompletableFuture
    .supplyAsync(() -> "Hello")          // 异步执行
    .thenApply(String::toUpperCase)      // 链式组合
    .exceptionally(ex -> "Error");       // 异常处理

System.out.println(future.join());
```

### 核心能力

**1. 非阻塞获取结果**

```java
CompletableFuture<String> future = CompletableFuture
    .supplyAsync(() -> {
        Thread.sleep(1000);
        return "Result";
    });

// 不阻塞主线程，可以继续做其他事
System.out.println("我继续执行");

// 完成后自动回调
future.thenAccept(result -> System.out.println("收到: " + result));
```

**2. 链式组合**

```java
CompletableFuture<User> userFuture = CompletableFuture
    .supplyAsync(() -> getUserId())
    .thenCompose(id -> getUser(id));  // 使用结果

CompletableFuture<String> resultFuture = CompletableFuture
    .supplyAsync(() -> getUser())
    .thenCombine(
        CompletableFuture.supplyAsync(() -> getSettings()),
        (user, settings) -> user.getName() + ":" + settings.getTheme()
    );
```

**3. 并行执行**

```java
CompletableFuture<User> userFuture = CompletableFuture
    .supplyAsync(() -> getUser(id));

CompletableFuture<List<Order>> ordersFuture = CompletableFuture
    .supplyAsync(() -> getOrders(id));

// 等待两个都完成
CompletableFuture<UserProfile> profileFuture = userFuture
    .thenCombine(ordersFuture, (user, orders) ->
        new UserProfile(user, orders)
    );
```

**4. 异常处理**

```java
CompletableFuture<String> future = CompletableFuture
    .supplyAsync(() -> callService())
    .exceptionally(ex -> {
        log.error("服务调用失败", ex);
        return "DEFAULT_VALUE";
    })
    .handle((result, ex) -> {
        if (ex != null) {
            return "HANDLED_VALUE";
        }
        return result;
    });
```

### CompletableFuture vs Future

| 特性 | Future | CompletableFuture |
|---|---|---|
| 阻塞 get | 是 | 可选 |
| 组合 | 否 | 是 |
| 回调 | 否 | 是 |
| 异常处理 | 手动 | thenCatch / handle |
| 多结果合并 | 否 | thenCombine / allOf / anyOf |
| 超时控制 | 手动 | orTimeout |

### CompletableFuture 的适用场景

| 优点 | 缺点 |
|---|---|
| 强大的组合能力 | 复杂场景代码可读性差 |
| 非阻塞 | 需要理解函数式编程 |
| 丰富的 API | 线程池管理 |
| 异常处理友好 | 并行度控制 |

## 模式四：响应式编程（Reactive）

### 什么是响应式

响应式编程是一种声明式的异步编程范式，核心是"数据流"和"传播变化"：

```java
// 使用 Reactor（Spring 5 响应式栈）
Mono<User> userMono = userRepository.findById(id);

userMono
    .map(User::getName)
    .subscribe(
        name -> System.out.println(name),
        error -> System.out.println(error),
        () -> System.out.println("完成")
    );
```

### 核心概念

**1. 数据流（Stream）**

```java
// Flux: 0~N 个元素的数据流
Flux<String> flux = Flux.just("A", "B", "C");

// Mono: 0~1 个元素的数据流
Mono<String> mono = Mono.just("A");
```

**2. 操作符（Operators）**

```java
flux
    .filter(s -> s.length() > 1)     // 过滤
    .map(String::toUpperCase)        // 转换
    .flatMap(this::fetchData)       // 扁平化
    .buffer(10)                      // 分批
    .window(Duration.ofSeconds(1))   // 窗口
    .onErrorResume(ex -> fallback()) // 错误处理
```

**3. 背压（Backpressure）**

```java
// 生产者快速发送，消费者慢速处理
// 响应式提供背压机制，消费者告诉生产者自己能处理多少

flux
    .onBackpressureBuffer(100)       // 缓冲区大小
    .onBackpressureDrop(dropped -> log.info("丢弃: " + dropped))
```

### 响应式 vs CompletableFuture

| 特性 | CompletableFuture | 响应式 |
|---|---|---|
| 数据模型 | 单次结果 | 数据流 |
| 背压 | 不支持 | 支持 |
| 资源清理 | 手动 | 自动 |
| 适用场景 | 单次异步操作 | 流式数据处理 |
| 生态 | JDK | Spring Reactor / RxJava |
| 学习曲线 | 中等 | 陡峭 |

### 响应式的适用场景

| 优点 | 缺点 |
|---|---|
| 处理数据流 | 学习曲线陡峭 |
| 声明式 | 调试困难 |
| 背压支持 | 额外开销 |
| 背压传播 | 场景有限 |
| 非阻塞 I/O | 复杂性 |

## 模式对比与选择

### 横向对比

| 特性 | Callback | Future | CompletableFuture | 响应式 |
|---|---|---|---|---|
| 诞生时间 | 最早 | JDK 5 | JDK 8 | JDK 9+ |
| 组合能力 | 差 | 无 | 好 | 非常好 |
| 错误处理 | 复杂 | 手动 | 丰富 API | 丰富 API |
| 学习曲线 | 低 | 低 | 中 | 高 |
| 适用场景 | 简单事件 | 简单异步 | 复杂异步 | 数据流处理 |
| 内存开销 | 低 | 中 | 中 | 高 |
| 线程安全 | 用户负责 | 是 | 是 | 是 |

### 场景选择指南

```
选择模式：

1. 简单的异步任务？
   └─→ Future 即可

2. 需要链式调用、并行执行？
   └─→ CompletableFuture

3. 大量异步任务需要组合？
   └─→ CompletableFuture

4. 处理数据流（数据库查询、消息队列）？
   └─→ 响应式（Reactor/RxJava）

5. 旧代码，只能用 Callback？
   └─→ 尽量包装成 Future 或 CompletableFuture
```

### 实战选择

**场景一：REST API 调用**

```java
// CompletableFuture：适合
CompletableFuture<User> userFuture = CompletableFuture
    .supplyAsync(() -> userClient.getUser(userId));

CompletableFuture<List<Order>> ordersFuture = CompletableFuture
    .supplyAsync(() -> orderClient.getOrders(userId));

userFuture.thenCombine(ordersFuture, UserProfile::new)
    .thenAccept(this::sendResponse);
```

**场景二：微服务编排**

```java
// CompletableFuture：适合
List<CompletableFuture<ServiceResult>> futures = services.stream()
    .map(service ->
        CompletableFuture.supplyAsync(() -> service.call())
            .exceptionally(ex -> ServiceResult.fallback(ex))
    )
    .collect(Collectors.toList());

CompletableFuture.allOf(futures.toArray(new CompletableFuture[0]))
    .thenApply(v -> futures.stream()
        .map(CompletableFuture::join)
        .collect(Collectors.toList())
    );
```

**场景三：数据库查询**

```java
// 响应式：适合（流式数据）
Flux<User> users = userRepository.findAll();

users
    .filter(user -> user.isActive())
    .map(user -> enrichWithPermissions(user))
    .buffer(100)
    .flatMap(this::saveBatch)
    .subscribe(
        count -> log.info("保存了 {} 条", count),
        error -> log.error("处理失败", error),
        () -> log.info("全部完成")
    );
```

**场景四：实时数据处理**

```java
// 响应式：适合
Flux<StockQuote> quotes = stockService.subscribe(symbol);

quotes
    .window(Duration.ofSeconds(5))
    .flatMap(window -> window.reduce((a, b) -> {
        // 计算 5 秒内的最高价
        return a.getPrice().compareTo(b.getPrice()) > 0 ? a : b;
    }))
    .subscribe(highest -> {
        System.out.println("最高价: " + highest.getPrice());
    });
```

## 常见问题

### Q1：CompletableFuture 和线程池的关系？

```java
// 默认使用 ForkJoinPool.commonPool()
CompletableFuture.supplyAsync(() -> task());

// 使用自定义线程池
ExecutorService executor = Executors.newFixedThreadPool(10);
CompletableFuture.supplyAsync(() -> task(), executor);

// 如果是 I/O 密集型，线程数应该大于 CPU 核心数
// 公式：线程数 = CPU 核心数 * (1 + I/O 时间 / CPU 时间)
```

### Q2：响应式什么时候用 Mono，什么时候用 Flux？

```java
// Mono: 返回 0~1 个结果
Mono<User> findById(String id);
Mono<Void> save(User user);

// Flux: 返回 0~N 个结果
Flux<User> findAll();
Flux<Order> findByUserId(String userId);
```

### Q3：如何处理超时？

```java
// CompletableFuture
CompletableFuture<String> future = CompletableFuture
    .supplyAsync(() -> callService())
    .orTimeout(3, TimeUnit.SECONDS)
    .exceptionally(ex -> {
        if (ex instanceof TimeoutException) {
            return "超时降级";
        }
        throw new RuntimeException(ex);
    });

// 响应式
flux
    .timeout(Duration.ofSeconds(5))
    .onErrorResume(TimeoutException.class, e -> fallback());
```

### Q4：响应式适合什么场景？

```java
// 适合：
// - 大量并发 I/O（HTTP 请求、数据库查询）
// - 事件流处理（WebSocket、消息队列）
// - 需要背压的数据管道
// - 非阻塞 REST API

// 不适合：
// - 计算密集型任务
// - 小数据量、简单逻辑
// - 需要同步等待的场景
```

## 总结

| 模式 | 适用场景 | 推荐指数 |
|---|---|---|
| Callback | 简单事件回调、老代码 | ⭐⭐ |
| Future | 简单异步任务 | ⭐⭐⭐ |
| CompletableFuture | 复杂异步编排、微服务调用 | ⭐⭐⭐⭐⭐ |
| 响应式 | 数据流处理、高并发 I/O | ⭐⭐⭐⭐（特定场景） |

记住：**没有银弹**。根据业务场景选择合适的模式，比追求最新技术更重要。
