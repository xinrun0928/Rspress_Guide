# 并行流 ParallelStream 原理

`stream().parallel()` 真的能让代码变快吗？

不一定。

并行流是 Java 8 引入的特性，它基于 ForkJoinPool 实现，号称能让你的代码"自动"变快。但现实很残酷：很多人用了并行流，反而发现性能更差了。

问题在于并行流不是万能药。它有自己的适用场景和性能陷阱。理解它的底层原理，才能用好它。

这篇文章会深入剖析并行流的工作机制，解释 ForkJoinPool 的设计，以及那些让并行流变慢的坑。

## 并行流的基础

### 从串行到并行

```java
// 串行流
List<Integer> result = list.stream()
    .map(String::toUpperCase)
    .collect(Collectors.toList());

// 并行流
List<Integer> result = list.parallelStream()
    .map(String::toUpperCase)
    .collect(Collectors.toList());
```

就这么简单？从 `.stream()` 变成 `.parallelStream()`，代码就能并行执行？

是的，但代价是引入了复杂的并行逻辑和数据分片机制。

### 并行流的执行模型

```
串行执行：
[T1] ──→ [T2] ──→ [T3] ──→ [T4] ──→ [T5]
  |       |       |       |       |
  ↓       ↓       ↓       ↓       ↓
顺序执行

并行执行：
[T1] ──→ [T2] ──→ [T3] ──→ [T4] ──→ [T5]
  |       |       |       |       |
  ↓       ↓       ↓       ↓       ↓
[CPU0] [CPU1] [CPU2] [CPU3] [CPU0] [CPU1]
同时执行，乱序完成
```

## ForkJoinPool 原理

### 核心思想：分治

ForkJoinPool 基于"分治"算法：

1. **Fork**：把大任务拆成小任务
2. **Join**：等小任务完成，合并结果
3. **RecursiveTask**：有返回值的任务

```java
// ForkJoinPool 的基本用法
ForkJoinPool pool = ForkJoinPool.commonPool();

ForkJoinTask<Long> task = new RecursiveTask<Long>() {
    @Override
    protected Long compute() {
        if (rangeSize < THRESHOLD) {
            // 小任务，直接计算
            return computeSum(start, end);
        }

        // 大任务，拆成两半
        int mid = (start + end) / 2;
        RecursiveTask<Long> left = new RecursiveTask<Long>() {
            @Override
            protected Long compute() {
                return compute(start, mid);
            }
        };
        RecursiveTask<Long> right = new RecursiveTask<Long>() {
            @Override
            protected Long compute() {
                return compute(mid, end);
            }
        };

        // Fork 拆分，Join 合并
        left.fork();
        right.fork();
        return left.join() + right.join();
    }
};

Long result = pool.invoke(task);
```

### 工作窃取（Work Stealing）

ForkJoinPool 的核心优化是**工作窃取**：

```
普通线程池：
[Worker1] [Worker2] [Worker3] [Worker4]
   |        |        |        |
 busy      busy     idle     idle
            ↑ 任务不均匀，Worker3 和 Worker4 空闲

ForkJoinPool（工作窃取）：
[Worker1] [Worker2] [Worker3] [Worker4]
   |        |        |        |
 busy      busy     busy     busy
   ↑                    ↑    ↑
   └──── 窃取任务 ──────┘    │
                            └──── 窃取任务
```

当某个 Worker 完成了自己的任务，它会从其他 Worker 的队列中"窃取"任务来执行。这样可以让所有 CPU 核心都保持忙碌状态。

### 工作窃取的源码解析

```java
// ForkJoinWorkerThread 的 run 方法
public void run() {
    while (running) {
        ForkJoinTask<?> task = tryPoll();  // 尝试从其他线程队列窃取任务
        if (task == null) {
            task = localQueue.poll();  // 窃取不到就执行自己的任务
        }
        if (task != null) {
            task.doExec();
        }
    }
}
```

**窃取策略**：从其他线程的队列尾部窃取任务，因为尾部是最可能还有未完成任务的区域。

## 并行流的默认线程池

### commonPool

Java 8 引入了 `ForkJoinPool.commonPool()`，所有并行流默认使用这个共享线程池：

```java
public static ForkJoinPool commonPool() {
    // 返回共享的公共池，默认大小 = CPU 核心数 - 1
    return common;
}

// 默认并行度 = max(1, Runtime.getRuntime().availableProcessors() - 1)
// 例如：8 核 CPU → 7 个线程
```

### 配置并行度

```bash
# 设置公共池大小
-Djava.util.concurrent.ForkJoinPool.common.parallelism=4

# 或者禁用公共池，每个任务用自己的池
-Djava.util.concurrent.ForkJoinPool.common.parallelism=0
```

### 自定义线程池

```java
// 创建自定义线程池
ForkJoinPool customPool = new ForkJoinPool(8);

// 使用自定义池执行并行流
customPool.submit(() ->
    list.parallelStream()
        .map(String::toUpperCase)
        .collect(Collectors.toList())
).join();
```

## 并行流的性能陷阱

### 陷阱一：数据量太小

```java
// 错误：100 个元素并行，开销可能比串行还大
List<String> list = Arrays.asList("a", "b", "c", ...);
list.parallelStream()
    .map(String::toUpperCase)
    .collect(Collectors.toList());

// 并行开销：
// - ForkJoinPool 线程切换
// - 数据分片
// - 结果合并
// - 如果计算本身很快，开销可能超过计算本身
```

**何时并行有优势**：元素数量大、每个元素处理耗时长、计算相互独立。

### 陷阱二：串行部分过长

```java
// 错误：并行部分很短，串行部分很长
list.parallelStream()
    .map(x -> {         // 并行（短暂）
        expensiveOperation(x);
    })
    .collect(Collectors.toList())
    .stream()           // ← 又变成串行了！
    .forEach(System.out::println);
```

### 陷阱三：数据源分片成本高

```java
// LinkedList 的并行流很慢，因为分片成本高
LinkedList<Integer> linked = new LinkedList<>();
linked.parallelStream()
    .map(x -> x * 2)
    .collect(Collectors.toList());

// ArrayList 的并行流很快，因为分片成本低
ArrayList<Integer> array = new ArrayList<>();
array.parallelStream()
    .map(x -> x * 2)
    .collect(Collectors.toList());
```

**原因**：LinkedList 的 Spliterator 需要遍历链表才能分片，而 ArrayList 可以直接通过索引计算分片边界。

### 陷阱四：顺序依赖

```java
// 错误：后一步依赖前一步的结果，无法并行
list.parallelStream()
    .reduce(0, (acc, x) -> {
        // 这里 acc 可能被多个线程同时修改，导致错误
        acc += expensiveOperation(x);
        return acc;
    });
```

### 陷阱五：共享状态

```java
// 错误：使用共享变量
List<String> results = Collections.synchronizedList(new ArrayList<>());
list.parallelStream()
    .map(String::toUpperCase)
    .forEach(results::add);  // 同步开销大，并行优势被抵消
```

## 性能对比测试

```java
public class ParallelStreamBenchmark {

    public static void main(String[] args) {
        // 测试不同数据量
        int[] sizes = {10, 100, 1000, 10000, 100000, 1000000};

        for (int size : sizes) {
            List<Integer> list = new ArrayList<>();
            for (int i = 0; i < size; i++) {
                list.add(i);
            }

            System.out.println("\n数据量: " + size);

            // 串行
            long serialTime = measure(() ->
                list.stream()
                    .map(ParallelStreamBenchmark::process)
                    .collect(Collectors.toList())
            );
            System.out.println("串行: " + serialTime + "ms");

            // 并行
            long parallelTime = measure(() ->
                list.parallelStream()
                    .map(ParallelStreamBenchmark::process)
                    .collect(Collectors.toList())
            );
            System.out.println("并行: " + parallelTime + "ms");

            System.out.println("加速比: " + (double) serialTime / parallelTime);
        }
    }

    // 模拟耗时操作
    private static int process(int x) {
        try {
            Thread.sleep(1);  // 1ms
        } catch (InterruptedException ignored) {}
        return x * 2;
    }

    private static long measure(Runnable task) {
        // 预热
        for (int i = 0; i < 3; i++) task.run();

        // 测量
        long start = System.nanoTime();
        task.run();
        return TimeUnit.NANOSECONDS.toMillis(System.nanoTime() - start);
    }
}
```

典型结果：

```
数据量: 10
串行: 11ms, 并行: 8ms, 加速比: 1.4x

数据量: 100
串行: 101ms, 并行: 25ms, 加速比: 4.0x

数据量: 1000
串行: 1005ms, 并行: 155ms, 加速比: 6.5x

数据量: 10000
串行: 10023ms, 并行: 1420ms, 加速比: 7.1x
```

**结论**：数据量越大、单个元素处理时间越长，并行优势越明显。

## 并行流与线程安全

### 收集器的线程安全性

```java
// 错误：ConcurrentHashMap 作为收集目标，但逻辑有问题
Map<String, Integer> map = new ConcurrentHashMap<>();
list.parallelStream()
    .forEach(x -> map.merge(key(x), 1, Integer::sum));

// 正确：使用正确的收集器
Map<String, Integer> map = list.parallelStream()
    .collect(Collectors.groupingByConcurrent(
        Function.identity(),
        Collectors.counting()
    ));
```

### 常见错误

```java
// 错误 1：使用非线程安全集合
List<String> results = new ArrayList<>();
list.parallelStream()
    .map(String::toUpperCase)
    .forEachOrdered(results::add);  // 不安全！

// 错误 2：在并行流中使用 ThreadLocal
list.parallelStream()
    .forEach(x -> {
        String threadLocal = ThreadLocalRandom.current().nextInt();
        // ThreadLocal 在并行流中效果不可预测
    });
```

## 最佳实践

### 何时使用并行流

| 场景 | 推荐 | 原因 |
|---|---|---|
| 大数据量 + 独立计算 | parallelStream | 计算密集型，可充分并行 |
| CPU 密集型任务 | parallelStream | 充分利用多核 |
| 简单映射/过滤 | parallelStream | 分片成本低 |
| 小数据量 | stream | 并行开销不划算 |
| I/O 密集型任务 | CompletableFuture | 并行流受公共池限制 |
| 有顺序要求 | stream | forEachOrdered 牺牲并行性 |
| 有共享状态 | stream 或手动同步 | 并行安全很难保证 |

### 何时不用并行流

```java
// 场景 1：数据量小
list.parallelStream()  // 不要
    .map(String::toUpperCase)
    .collect(Collectors.toList());

// 场景 2：I/O 密集型
list.parallelStream()  // 不要，会占用公共池
    .map(this::fetchFromRemote)
    .collect(Collectors.toList());

// 场景 3：有状态依赖
list.parallelStream()  // 不要
    .reduce(0, (acc, x) -> acc + x);

// 应该用 CompletableFuture
CompletableFuture.allOf(list.stream()
    .map(x -> CompletableFuture.supplyAsync(() -> fetchFromRemote(x)))
    .toArray(CompletableFuture[]::new)
).join();
```

### 性能调优

```java
// 技巧 1：选择合适的分片大小
list.parallelStream()
    .unordered()  // 允许无序处理，可能提高并行度
    .map(expensiveOperation)
    .collect(Collectors.toList());

// 技巧 2：避免自动装箱
IntStream.range(0, 1000000)  // 基本类型流，没有装箱开销
    .parallel()
    .map(x -> x * 2)
    .sum();

// 技巧 3：使用自定义收集器
Collector<T, ?, R> myCollector = Collector.of(
    MyCollector::new,
    MyCollector::accumulate,
    MyCollector::combine,  // 并行合并
    MyCollector::finish
);
```

## 常见问题

### Q1：parallelStream 使用多少线程？

默认 = `Runtime.getRuntime().availableProcessors() - 1`。8 核 CPU 使用 7 个线程。

### Q2：parallelStream 线程安全吗？

并行流本身是线程安全的，但不安全的代码在并行流中会出现并发问题。

### Q3：parallelStream 会阻塞主线程吗？

不会，它是异步的。但 `collect()` 和 `reduce()` 会等待所有子任务完成。

### Q4：可以同时运行多个 parallelStream 吗？

可以，但它们共享同一个公共池。如果一个长时间运行的流占满公共池，其他流会被阻塞。

### Q5：如何为每个 parallelStream 使用独立线程池？

```java
ForkJoinPool pool = new ForkJoinPool(4);
pool.submit(() -> {
    list.parallelStream()
        .map(String::toUpperCase)
        .collect(Collectors.toList());
});
```

## 总结

并行流是一个强大的工具，但需要谨慎使用：

| 要素 | 说明 |
|---|---|
| 原理 | ForkJoinPool + 工作窃取 |
| 默认并行度 | CPU 核心数 - 1 |
| 优势 | 大数据量 + 独立计算 |
| 劣势 | 小数据量 + 复杂分片 + 共享状态 |
| 线程安全 | 需要使用线程安全的收集器或同步机制 |

记住：**并行不等于自动优化**。只有在合适的场景下使用，才能真正提升性能。
