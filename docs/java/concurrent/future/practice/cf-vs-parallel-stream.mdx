# CompletableFuture vs ParallelStream vs 响应式编程对比

Java 8 引入的 `CompletableFuture` 和 `ParallelStream`，加上后来的响应式编程（Reactor、RxJava），让 Java 异步编程的选择变得丰富起来。面对一个具体的业务场景，应该选择哪种方式？它们各自的优劣是什么？本文将从多个维度进行深入对比。

## 三种方案的概述

### CompletableFuture：灵活的异步编排

```java
CompletableFuture.supplyAsync(() -> queryFromDB())
    .thenApply(this::transform)
    .thenAccept(this::process)
    .exceptionally(ex -> {
        log.error("处理失败", ex);
        return null;
    });
```

`CompletableFuture` 提供了丰富的 API 来组合异步操作，支持任务依赖、异常处理、超时控制等。适合复杂的业务逻辑编排。

### ParallelStream：简单的数据并行

```java
List<Result> results = data.parallelStream()
    .map(this::process)
    .filter(this::shouldInclude)
    .collect(Collectors.toList());
```

`ParallelStream` 是对 Java Stream API 的并行扩展，适合对集合数据进行并行处理。使用简单，但控制能力较弱。

### 响应式编程：声明式的数据流处理

```java
monoService.getData()
    .map(this::transform)
    .flatMap(this::processAsync)
    .onErrorReturn(defaultValue)
    .subscribe(result -> process(result));
```

响应式编程（Reactive Programming）强调数据的流动和变换，适合处理异步事件流和背压场景。

## 核心维度对比

| 维度 | CompletableFuture | ParallelStream | 响应式编程 |
|------|------------------|----------------|-----------|
| 线程池 | 可自定义 | ForkJoinPool.common | 可自定义 |
| 任务依赖 | 强支持 | 不支持 | 强支持 |
| 数据并行 | 不直接支持 | 强支持 | 支持 |
| 背压控制 | 不支持 | 不支持 | 支持 |
| 异常处理 | 丰富 API | 有限 | 丰富 |
| 学习曲线 | 中等 | 低 | 高 |
| 适用场景 | 业务编排 | 数据处理 | 事件流 |

## 并行能力对比

### ParallelStream 的并行优势

对于 CPU 密集型的数据处理任务，`ParallelStream` 是最简单有效的选择：

```java
// 串行处理
List<Integer> results = data.stream()
    .map(this::computeIntensive)
    .collect(Collectors.toList());

// 并行处理，只需要换一个方法
List<Integer> parallelResults = data.parallelStream()
    .map(this::computeIntensive)
    .collect(Collectors.toList());
```

`ParallelStream` 默认使用 `ForkJoinPool.commonPool()`，它的大小默认为 `Runtime.getRuntime().availableProcessors() - 1`。对于 8 核机器，线程池大小是 7。

### CompletableFuture 的并行能力

`CompletableFuture` 默认也使用 `ForkJoinPool.commonPool()`，但支持自定义线程池：

```java
// 使用自定义线程池
ExecutorService executor = Executors.newFixedThreadPool(10);
CompletableFuture.supplyAsync(() -> task(), executor)
    .thenApplyAsync(this::nextTask, executor)
    .thenAcceptAsync(this::process, executor);
```

这让 `CompletableFuture` 在需要精细控制线程池的场景下更有优势。

### 响应式编程的并行能力

响应式库通常有自己的调度器：

```java
// 指定调度器
monoService.getData()
    .subscribeOn(Schedulers.parallel())  // 并行执行
    .publishOn(Schedulers.elastic())    // 结果发布到弹性线程
    .subscribe(result -> process(result));
```

## 任务依赖处理

### CompletableFuture 的依赖处理

`CompletableFuture` 提供了丰富的依赖处理 API：

```java
CompletableFuture<A> futureA = CompletableFuture.supplyAsync(() -> queryA());
CompletableFuture<B> futureB = CompletableFuture.supplyAsync(() -> queryB());

// thenCompose：处理依赖
futureA.thenCompose(a -> futureB.thenApply(b -> combine(a, b)));

// thenCombine：合并独立任务
futureA.thenCombine(futureB, this::combine);

// allOf：等待所有任务
CompletableFuture.allOf(futureA, futureB).thenApply(v -> {
    A a = futureA.join();
    B b = futureB.join();
    return combine(a, b);
});
```

### 响应式编程的依赖处理

响应式编程用操作符处理依赖：

```java
Mono.just(input)
    .flatMap(this::queryA)                    // thenCompose
    .flatMap(a -> queryB(a).map(b -> combine(a, b)))  // thenCombine
    .zipWith(queryC())                        // 合并另一个 Mono
    .map(this::finalCombine);
```

## 异常处理对比

### CompletableFuture 的异常处理

```java
CompletableFuture.supplyAsync(() -> riskyOperation())
    .exceptionally(ex -> {                    // 处理异常
        log.error("操作失败", ex);
        return defaultValue;
    })
    .handle((result, ex) -> {                 // 同时处理成功和异常
        if (ex != null) {
            return defaultValue;
        }
        return result;
    })
    .whenComplete((result, ex) -> {           // 观察结果（不修改）
        if (ex != null) {
            log.error("完成但有错误", ex);
        }
    });
```

### ParallelStream 的异常处理

`ParallelStream` 的异常处理能力有限，必须用 try-catch 包裹：

```java
List<Result> results = data.parallelStream()
    .map(item -> {
        try {
            return process(item);
        } catch (Exception e) {
            log.error("处理失败", e);
            return defaultResult;
        }
    })
    .collect(Collectors.toList());
```

### 响应式编程的异常处理

```java
monoService.getData()
    .map(this::riskyTransform)
    .onErrorReturn(defaultValue)              // 返回默认值
    .onErrorResume(ex -> fallbackMono)        // 切换到另一个 Mono
    .onErrorMap(ex -> new BusinessException(ex))  // 转换异常类型
    .retry(3)                                 // 重试
    .doOnError(ex -> log.error("最终失败", ex))  // 观察错误
    .subscribe();
```

## 性能对比

### 线程开销

```java
// ParallelStream：为每个任务创建任务
data.parallelStream()
    .map(task -> {
        // 任务被分割成多个子任务
        return compute(task);
    });

// CompletableFuture：更精细的控制
CompletableFuture.supplyAsync(() -> compute(data));
```

### 内存使用

响应式编程在背压场景下可以显著降低内存使用，因为它采用"拉取"模式而非"推送"模式。但对于小数据量，响应式的开销可能更大。

### 基准测试建议

对于性能敏感的场景，建议进行实际测试：

```java
public class PerformanceBenchmark {

    public void benchmark(String name, Runnable task) {
        // 预热
        for (int i = 0; i < 10; i++) {
            task.run();
        }

        // 计时
        long start = System.nanoTime();
        for (int i = 0; i < 1000; i++) {
            task.run();
        }
        long duration = System.nanoTime() - start;

        System.out.printf("%s: %.2f ms%n", name, duration / 1_000_000.0);
    }
}
```

## 场景选择指南

### 选择 CompletableFuture 的场景

- 业务逻辑需要多个服务调用编排
- 需要精细控制线程池
- 需要复杂的错误处理和重试逻辑
- 需要超时控制

```java
public CompletableFuture<OrderResult> createOrder(OrderRequest request) {
    return validateRequest(request)
        .thenCompose(this::checkInventory)
        .thenCompose(this::calculatePrice)
        .thenCompose(this::createOrderRecord)
        .exceptionally(ex -> {
            log.error("订单创建失败", ex);
            return OrderResult.failed(ex.getMessage());
        });
}
```

### 选择 ParallelStream 的场景

- 对集合数据进行 CPU 密集型处理
- 任务之间没有依赖
- 简单的 map-filter-reduce 操作
- 代码简洁性优先

```java
List<MonthlyReport> reports = months.parallelStream()
    .map(this::generateReport)
    .filter(Report::isValid)
    .collect(Collectors.toList());
```

### 选择响应式编程的场景

- 处理异步事件流（用户点击、传感器数据）
- 需要背压控制
- 多个数据源需要合并
- 长时间运行的流处理

```java
fluxService.getEvents()
    .buffer(100, Duration.ofSeconds(1))        // 批量
    .flatMap(this::processBatch)              // 处理批次
    .onBackpressureBuffer()                   // 背压
    .subscribe(this::onNext);
```

## 混合使用

实际项目中，三种方案可以混合使用：

```java
public CompletableFuture<Dashboard> getDashboard(String userId) {
    // 使用 CompletableFuture 编排多个异步服务
    CompletableFuture<User> userFuture = userService.getUser(userId);
    CompletableFuture<List<Order>> ordersFuture = orderService.getOrders(userId);

    return CompletableFuture.allOf(userFuture, ordersFuture)
        .thenApply(v -> {
            User user = userFuture.join();
            List<Order> orders = ordersFuture.join();

            // 使用 parallelStream 处理数据
            List<PopularProduct> recommendations = orders.parallelStream()
                .flatMap(order -> order.getItems().stream())
                .collect(Collectors.groupingBy(Product::getId, Collectors.counting()))
                .entrySet().stream()
                .sorted(Map.Entry.<ProductId, Long>comparingByValue().reversed())
                .limit(10)
                .map(e -> popularProductService.get(e.getKey()))
                .collect(Collectors.toList());

            return new Dashboard(user, orders, recommendations);
        });
}
```

## 面试高频考点

**问题一：ParallelStream 和 CompletableFuture 使用的线程池是什么？**

默认都使用 `ForkJoinPool.commonPool()`。`ParallelStream` 无法自定义（除非使用 `ForkJoinPool` 构造函数创建自定义池）。`CompletableFuture` 可以通过 `supplyAsync(Supplier, Executor)` 自定义线程池。

**问题二：ParallelStream 的并行度如何调整？**

默认并行度为 `Runtime.getRuntime().availableProcessors() - 1`。可以设置系统属性 `java.util.concurrent.ForkJoinPool.common.parallelism` 调整，但会影响所有并行流。

**问题三：什么时候不适用 ParallelStream？**

- 任务之间有依赖关系
- 需要精细的异常处理
- 需要取消任务
- I/O 密集型任务（容易耗尽线程）

**问题四：响应式编程的优势是什么？**

- 背压控制
- 声明式的数据流处理
- 丰富的操作符
- 适合处理异步事件流

**问题五：如何选择三种方案？**

简单数据并行选 `ParallelStream`，复杂业务编排选 `CompletableFuture`，异步事件流选响应式编程。

理解三种方案的优劣，才能在实际项目中做出正确的技术选型。没有最好的方案，只有最适合的方案。
