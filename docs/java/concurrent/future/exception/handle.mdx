# handle：携带异常信息的统一处理

`handle` 和 `exceptionally` 有什么区别？

这是一个看似简单但实际有深层区别的问题。很多开发者会混用它们，但这种混用可能导致异常信息丢失、代码逻辑混乱。

`handle` 的核心优势是：**无论成功还是失败，都能处理；无论有无异常，都能访问结果**。这种"统一处理"的能力，让它在某些场景下比 `exceptionally` 更好用。

这篇文章会深入对比 `handle` 和 `exceptionally`，并通过实战案例展示如何正确使用它们。

## exceptionally 的局限性

### 回顾 exceptionally

`exceptionally` 只在异常发生时执行，可以将异常转换为正常结果：

```java
CompletableFuture<String> future = CompletableFuture
    .supplyAsync(() -> {
        throw new RuntimeException("任务失败");
    })
    .exceptionally(ex -> {
        // 只有异常时才会执行这里
        System.out.println("捕获异常: " + ex.getMessage());
        return "降级结果";  // 返回降级值
    });

System.out.println(future.join());  // 输出: 降级结果
```

### exceptionally 的问题

**问题一：异常被吞掉后，无法区分是降级结果还是正常结果**

```java
CompletableFuture<User> future = getUserFromCache(userId)
    .exceptionally(ex -> {
        log.error("缓存失败", ex);
        return getUserFromDatabase(userId);  // 从数据库获取
    });

// 这里的 user 可能是缓存返回的，也可能是数据库返回的
User user = future.join();
```

在这个场景下，我们需要区分"从缓存获取"和"从数据库获取"，以便后续统计缓存命中率。但 `exceptionally` 无法提供这种区分能力。

**问题二：无法访问原始结果（如果成功的话）**

```java
CompletableFuture<List<String>> future = fetchData()
    .exceptionally(ex -> {
        // ex 发生了，但无法访问 fetchData() 原本可能返回的结果
        // 只能返回一个固定值
        return Collections.emptyList();
    });
```

如果 fetchData() 已经获取了部分数据，`exceptionally` 无法利用这些数据，只能返回固定值。

**问题三：异常之后无法继续组合**

```java
future
    .exceptionally(ex -> {
        return "fallback";  // 返回降级值
    })
    .thenApply(result -> {
        // 这里的 result 是字符串，不是原始类型
        // 如果原类型是 User，这里就需要强制转换
        return ((String) result).length();
    });
```

类型被破坏了，链式调用变得困难。

## handle 的优势

### 核心定义

`handle` 的签名是：

```java
public <U> CompletableFuture<U> handle(
    BiFunction<? super T, Throwable, ? extends U> fn
)
```

关键点：

1. **接收两个参数**：`result`（成功时的结果）和 `ex`（异常，可能为 null）
2. **总是执行**：无论成功还是异常，handle 都会执行
3. **可以返回任何类型**：`U` 可以与 `T` 不同

### 基本用法

```java
CompletableFuture<String> future = CompletableFuture
    .supplyAsync(() -> {
        throw new RuntimeException("任务失败");
    })
    .handle((result, ex) -> {
        // 成功时：result = 值，ex = null
        // 异常时：result = null，ex = 异常对象

        if (ex != null) {
            System.out.println("异常: " + ex.getMessage());
            return "降级结果";
        }

        System.out.println("成功: " + result);
        return result;
    });

System.out.println(future.join());  // 输出: 降级结果
```

## handle vs exceptionally：深度对比

### 对比表

| 特性 | handle | exceptionally |
|---|---|---|
| 执行时机 | 无论成功/异常都执行 | 只在异常时执行 |
| 参数 | `(result, ex)` | `(ex)` |
| 能否访问成功结果 | 能（result 参数） | 不能 |
| 异常为 null 时 | 能判断 | 不会进入 |
| 返回类型 | 可与原类型不同 | 必须与原类型相同 |
| 类型安全性 | 更好 | 需要强制转换 |

### 实战对比：缓存降级

**用 exceptionally 实现（有问题）**

```java
public CompletableFuture<User> getUser(String userId) {
    return getFromCache(userId)
        .exceptionally(ex -> {
            // 问题1：无法区分是缓存失败还是降级结果
            // 问题2：无法统计缓存命中率
            log.warn("缓存失败，回源数据库", ex);
            return getFromDatabase(userId);
        });
}
```

**用 handle 实现（更好）**

```java
public CompletableFuture<User> getUser(String userId) {
    return getFromCache(userId)
        .handle((user, ex) -> {
            if (ex != null) {
                // 缓存失败，降级到数据库
                log.warn("缓存失败，回源数据库", ex);
                cacheMisses.incrementAndGet();  // 统计缓存未命中
                return getFromDatabase(userId);
            }

            // 缓存成功
            cacheHits.incrementAndGet();  // 统计缓存命中
            return CompletableFuture.completedFuture(user);
        })
        .thenCompose(userFuture -> userFuture);  // 展开嵌套的 Future
}
```

### 实战对比：日志记录 + 降级

**用 exceptionally 实现**

```java
CompletableFuture<Result> future = processData()
    .exceptionally(ex -> {
        log.error("处理失败: {}", ex.getMessage(), ex);
        return Result.fallback();  // 只能返回固定降级值
    });
```

**用 handle 实现**

```java
CompletableFuture<Result> future = processData()
    .handle((result, ex) -> {
        if (ex != null) {
            log.error("处理失败: {}", ex.getMessage(), ex);
            return Result.fallback();
        }

        log.info("处理成功: {}", result);
        return result;
    });
```

## 实战案例

### 案例一：组合多个数据源

```java
public CompletableFuture<UserProfile> getUserProfile(String userId) {
    // 同时查询多个数据源
    CompletableFuture<User> userFuture = getUser(userId);
    CompletableFuture<UserSettings> settingsFuture = getSettings(userId);
    CompletableFuture<UserActivity> activityFuture = getActivity(userId);

    return userFuture
        .handle((user, ex) -> {
            if (ex != null) {
                log.warn("获取用户信息失败", ex);
                return User.ANONYMOUS;  // 降级为匿名用户
            }
            return user;
        })
        .thenCombine(
            settingsFuture.exceptionally(ex -> {
                log.warn("获取设置失败", ex);
                return UserSettings.DEFAULT;
            }),
            UserProfile::new
        )
        .handle((profile, ex) -> {
            if (ex != null) {
                log.error("组合用户资料失败", ex);
                return UserProfile.defaultProfile();
            }
            return profile;
        });
}
```

### 案例二：重试 + 降级

```java
public CompletableFuture<String> fetchWithRetry(String url, int maxRetries) {
    CompletableFuture<String> future = CompletableFuture
        .supplyAsync(() -> httpClient.get(url));

    for (int i = 0; i < maxRetries; i++) {
        final int retryCount = i + 1;

        future = future.handle((result, ex) -> {
            if (ex == null) {
                // 成功，直接返回
                return result;
            }

            // 失败，检查是否还有重试次数
            if (retryCount >= maxRetries) {
                log.error("重试{}次后仍失败", maxRetries, ex);
                return "DEFAULT_RESPONSE";
            }

            log.warn("第{}次重试", retryCount, ex);
            throw new RuntimeException("需要重试", ex);
        }).orTimeout(5, TimeUnit.SECONDS);
    }

    return future;
}
```

### 案例三：统计 + 降级

```java
public CompletableFuture<Order> createOrder(OrderRequest request) {
    return validateOrder(request)
        .handle((isValid, ex) -> {
            if (ex != null || !isValid) {
                orderMetrics.increment("validation_failures");
                return false;
            }
            orderMetrics.increment("validation_successes");
            return true;
        })
        .thenCompose(isValid -> {
            if (!isValid) {
                return CompletableFuture.failedFuture(
                    new OrderValidationException("订单验证失败"));
            }
            return processPayment(request);
        })
        .handle((order, ex) -> {
            if (ex != null) {
                orderMetrics.increment("payment_failures");
                return Order.unpaid(request);  // 返回未支付订单
            }
            orderMetrics.increment("payment_successes");
            return order;
        });
}
```

## handle 的高级用法

### 嵌套 Future 处理

当 handle 返回一个 Future 时，需要用 `thenCompose` 展开：

```java
CompletableFuture<CompletableFuture<String>> nested =
    getFromCache(id)
        .handle((result, ex) -> {
            if (ex != null) {
                return getFromDatabase(id);  // 返回 Future
            }
            return CompletableFuture.completedFuture(result);
        });

// 需要展开
CompletableFuture<String> flat = nested.thenCompose(f -> f);
```

更简洁的写法：

```java
CompletableFuture<String> result = getFromCache(id)
    .handle((result, ex) ->
        ex != null ? getFromDatabase(id) : CompletableFuture.completedFuture(result))
    .thenCompose(f -> f);
```

### 异常类型判断

```java
future.handle((result, ex) -> {
    if (ex == null) {
        return result;
    }

    // 分类处理不同类型的异常
    if (ex.getCause() instanceof TimeoutException) {
        return "超时";
    } else if (ex.getCause() instanceof IOException) {
        return "IO错误";
    } else if (ex.getCause() instanceof BusinessException be) {
        return "业务错误: " + be.getCode();
    }

    return "未知错误";
});
```

### 组合多个 handle

```java
CompletableFuture<User> userFuture = getUser(id);

// 第一层：数据验证
userFuture = userFuture
    .handle((user, ex) -> {
        if (user == null && ex == null) {
            throw new IllegalArgumentException("用户不存在");
        }
        return user;
    })
    .exceptionally(ex -> {
        if (ex.getCause() instanceof IllegalArgumentException) {
            return null;  // 用户不存在，返回 null
        }
        throw new RuntimeException(ex);  // 其他异常继续抛出
    });

// 第二层：权限检查
userFuture = userFuture
    .handle((user, ex) -> {
        if (ex != null) {
            throw new CompletionException(ex);
        }
        if (!hasPermission(user)) {
            throw new SecurityException("无权限访问");
        }
        return user;
    })
    .exceptionally(ex -> {
        if (ex.getCause() instanceof SecurityException) {
            log.warn("权限检查失败: {}", ex.getMessage());
        }
        return null;
    });
```

## 常见错误

### 错误一：在 handle 中重新抛出异常

```java
// 错误：可能导致异常丢失
future.handle((result, ex) -> {
    if (ex != null) {
        throw new RuntimeException(ex);  // 不要这样做！
    }
    return result;
});

// 正确：用 exceptionally 或 whenComplete
future.exceptionally(ex -> {
    throw new CompletionException(ex);  // 正确包装
});
```

### 错误二：忘记处理异常为 null 的情况

```java
// 错误：假设总是有异常
future.handle((result, ex) -> {
    return ex.getMessage();  // ex 可能为 null！
});

// 正确：显式判断
future.handle((result, ex) -> {
    if (ex != null) {
        return ex.getMessage();
    }
    return result.toString();
});
```

### 错误三：handle 返回 null 导致 NPE

```java
// 错误：返回 null
future.handle((result, ex) -> {
    if (ex != null) {
        log.error("失败", ex);
        return null;  // 后续操作可能 NPE
    }
    return result;
});

// 正确：返回有意义的默认值
future.handle((result, ex) -> {
    if (ex != null) {
        log.error("失败", ex);
        return "DEFAULT_VALUE";
    }
    return result;
});
```

## 面试问题

**Q1：handle 和 exceptionally 的区别？**

`handle` 接收两个参数 `(result, ex)`，无论成功还是异常都会执行；`exceptionally` 只接收 `ex`，只在异常时执行。`handle` 可以访问成功结果，而 `exceptionally` 不能。

**Q2：什么时候用 handle？什么时候用 exceptionally？**

需要访问成功结果或需要统一处理成功/异常时用 `handle`；只需要简单降级且不需要访问结果时用 `exceptionally`。

**Q3：handle 中抛出异常会怎样？**

`handle` 中抛出的异常会被包装在 `CompletionException` 中，继续向后传播。

**Q4：handle 能替代 try-catch 吗？**

不能完全替代。`handle` 用于异步场景，try-catch 用于同步场景。如果同步代码需要降级，应该用 try-catch。

## 总结

`handle` 是 CompletableFuture 中最灵活的异常处理方法：

- **统一处理**：成功和异常都能处理
- **访问结果**：可以同时访问成功结果和异常信息
- **类型安全**：返回类型可以与原类型不同

| 场景 | 推荐用法 |
|---|---|
| 简单降级 | `exceptionally` |
| 需要访问结果 | `handle` |
| 需要统计/监控 | `handle` |
| 需要组合多个降级路径 | `handle` |
| 需要区分异常类型 | `handle` |

理解 `handle` 和 `exceptionally` 的区别，才能写出更健壮的异步代码。
