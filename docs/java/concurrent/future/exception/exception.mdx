# unchecked 异常与 CancellationException 处理

当 `CompletableFuture` 中的代码抛出异常时，JVM 并不是简单地把异常原封不动地抛出来。Java 的异常体系被精心设计过，`CompletableFuture` 对不同类型的异常有不同的包装策略。理解这些策略，是掌握异步编程异常处理的必经之路。很多开发者在这个问题上踩过坑——在 `exceptionally` 中直接打印异常消息，却发现输出的是一个 `CompletionException`，而不是自己抛出的原始异常。

## Java 异常体系回顾

在深入 `CompletableFuture` 之前，先回顾一下 Java 的异常分类：

```java
public class ExceptionHierarchy {
    public void showExceptionTypes() {
        // Throwable 是所有错误和异常的基类
        // Error 表示 JVM 层面的严重问题，如 OutOfMemoryError
        // Exception 表示应用程序层面的异常

        // RuntimeException 是未检查异常（Unchecked Exception）
        // 编译器不要求强制捕获，可以选择在方法声明处抛出
        RuntimeException unchecked = new RuntimeException("未检查异常");

        // IOException 是已检查异常（Checked Exception）
        // 编译器要求必须捕获或声明抛出
        IOException checked = new IOException("已检查异常");
    }
}
```

**已检查异常（Checked Exception）**：继承自 `Exception` 但不是 `RuntimeException`。编译器会检查代码是否处理了这些异常，典型代表是 `IOException`、`SQLException`。

**未检查异常（Unchecked Exception）**：包括 `RuntimeException` 及其子类。编译器不强制处理，典型代表是 `NullPointerException`、`IllegalArgumentException`。

## CompletableFuture 的异常包装策略

这是理解 `CompletableFuture` 异常处理的关键：`CompletableFuture` 对不同类型的异常有不同的包装行为。

### unchecked 异常：包装在 CompletionException 中

当 `supplyAsync` 或 `thenApply` 等方法中抛出 `RuntimeException` 或 `Error` 时，异常会被包装在 `CompletionException` 中：

```java
CompletableFuture.supplyAsync(() -> {
    throw new RuntimeException("原始unchecked异常");
})
.exceptionally(ex -> {
    // ex 的类型是 CompletionException
    System.out.println(ex.getClass().getName());
    // 输出: java.util.concurrent.CompletionException

    // 原始异常需要通过 getCause() 获取
    System.out.println(ex.getCause().getMessage());
    // 输出: 原始unchecked异常

    return null;
});
```

这种包装策略的原因是：异步任务可能在线程池的另一个线程中执行，当异常传播到主线程时，需要一种方式将异步上下文中的异常与主线程的异常区分开来。`CompletionException` 就扮演了这个"容器"的角色。

### checked 异常：直接抛出或包装？

对于 `supplyAsync`（它接收的是 `Supplier<T>`，可以抛出任何异常），checked 异常和 unchecked 异常的处理方式相同——都会被包装在 `CompletionException` 中：

```java
CompletableFuture.supplyAsync(() -> {
    // 即使是 checked 异常，也被包装在 CompletionException 中
    throw new IOException("IO操作失败");
})
.exceptionally(ex -> {
    // ex 是 CompletionException，getCause() 是 IOException
    if (ex.getCause() instanceof IOException) {
        System.out.println("捕获到IO异常");
    }
    return null;
});
```

但是，如果你使用的是 `thenApply` 等方法接收 `Function`，情况略有不同：

```java
CompletableFuture.supplyAsync(() -> "input")
    .thenApply(result -> {
        // 如果这里抛出 checked 异常
        throw new IOException("IO错误");  // 编译错误！

        // 编译器会要求你处理或声明抛出
        return result;
    });
```

因为 `thenApply` 的函数签名是 `Function<? super T, ? extends U>`，它的 `apply` 方法签名是 `U apply(T t) throws Exception`，这意味着 checked 异常必须被显式处理。

### 异常包装的源码解读

理解异常包装的最好方式是看源码。`CompletableFuture` 的 `completeExceptionally` 方法展示了包装逻辑：

```java
public boolean completeExceptionally(Throwable ex) {
    Object r;
    // 将原始异常包装在 CompletionException 中
    // 注意：如果是 Error 或 RuntimeException，直接包装
    // 如果已经是 CompletionException，保留原样（避免双重包装）
    Throwable cause = (ex instanceof RuntimeException || ex instanceof Error)
        ? ex
        : new CompletionException(ex);

    // 后续的 complete 操作会处理异常的传播
    return completeWithEmpty(cause);
}
```

这里的逻辑是：如果是 `RuntimeException` 或 `Error`，直接保留；如果是其他异常（包括 checked 异常），包装在 `CompletionException` 中。这个设计确保了异步代码抛出的异常总能用一致的方式处理。

## CancellationException：特殊的取消异常

`CancellationException` 是一个特殊的异常，它表示任务被主动取消。在 `CompletableFuture` 中，取消任务有两种方式：

### 方式一：调用 cancel() 方法

```java
CompletableFuture<String> future = new CompletableFuture<>();

// 取消任务
boolean cancelled = future.cancel(true);

System.out.println("是否取消成功: " + cancelled);
System.out.println("isDone: " + future.isDone());
System.out.println("isCancelled: " + future.isCancelled());

// 尝试获取结果会抛出 CancellationException
try {
    future.join();
} catch (CompletionException e) {
    System.out.println("捕获到异常: " + e.getClass().getName());
    // 输出: java.util.concurrent.CancellationException
}
```

### 方式二：在回调中抛出 CancellationException

```java
CompletableFuture.supplyAsync(() -> {
    // 模拟业务逻辑
    if (/* 某些条件 */) {
        throw new CancellationException("业务取消");
    }
    return "result";
})
.exceptionally(ex -> {
    if (ex instanceof CancellationException ||
        ex.getCause() instanceof CancellationException) {
        System.out.println("任务被取消");
    }
    return null;
});
```

### CancellationException 与普通异常的区分

在处理异常时，正确区分 `CancellationException` 和其他异常很重要：

```java
future.exceptionally(ex -> {
    Throwable cause = ex.getCause();

    if (cause instanceof CancellationException) {
        // 任务被取消，不算错误
        return null;
    } else if (cause instanceof TimeoutException) {
        // 超时，返回降级值
        return getFallbackValue();
    } else {
        // 其他错误，记录并返回降级值
        log.error("任务执行失败", cause);
        return getFallbackValue();
    }
});
```

## 异常的穿透与中断

默认情况下，如果异步任务抛出异常且没有被处理，调用 `get()` 或 `join()` 会抛出 `CompletionException`。但有时候，我们希望在异常发生时做一些清理工作，或者让异常继续传播。

### 使用 whenComplete 观察异常

`whenComplete` 允许你在异常发生时执行清理代码，同时不阻止异常的传播：

```java
CompletableFuture<String> future = CompletableFuture
    .supplyAsync(() -> {
        throw new RuntimeException("任务失败");
    })
    .whenComplete((result, ex) -> {
        // 无论成功还是失败，都会执行这里
        System.out.println("清理资源...");

        if (ex != null) {
            System.out.println("异常: " + ex.getMessage());
            // 注意：这里不能吞掉异常
        }
    });

// future 仍然会抛出异常
try {
    future.join();
} catch (CompletionException e) {
    System.out.println("最终抛出的异常: " + e.getMessage());
    // 输出: 任务失败
}
```

### 使用 handle 转换异常

`handle` 方法允许你同时处理成功和异常的情况，并返回一个新值：

```java
String result = CompletableFuture
    .supplyAsync(() -> {
        throw new RuntimeException("失败");
    })
    .handle((res, ex) -> {
        if (ex != null) {
            // 异常情况，返回降级值
            return "fallback";
        }
        // 正常情况，返回原值
        return res;
    })
    .join();

System.out.println(result);  // 输出: fallback
```

## 异常处理的最佳实践

### 实践一：总是使用 getCause() 获取原始异常

```java
// 错误：可能看到的是包装类的消息
.exceptionally(ex -> {
    log.error(ex.getMessage());  // 可能不是原始错误信息
    return null;
})

// 正确：获取原始异常
.exceptionally(ex -> {
    Throwable cause = ex instanceof CompletionException
        ? ex.getCause()
        : ex;
    log.error(cause.getMessage(), cause);
    return null;
})
```

### 实践二：考虑使用 handle 统一处理

如果你的业务逻辑需要对成功和异常情况做不同的处理，`handle` 比 `thenApply` + `exceptionally` 更清晰：

```java
// 不推荐：thenApply 和 exceptionally 分开处理
future.thenApply(this::process)
      .exceptionally(ex -> {
          log.error(ex.getMessage());
          return null;
      });

// 推荐：handle 统一处理
future.handle((result, ex) -> {
    if (ex != null) {
        log.error(ex.getMessage());
        return null;
    }
    return process(result);
});
```

### 实践三：不要在 exception 块中再次抛出异常

```java
// 错误：可能造成双重包装
.exceptionally(ex -> {
    throw new RuntimeException(ex);  // 不要这样做！
})

// 正确：返回降级值或记录后返回 null
.exceptionally(ex -> {
    log.error("任务失败", ex);
    return null;  // 或者返回降级值
})
```

## 面试高频考点

**问题一：CompletableFuture 中 unchecked 异常是如何包装的？**

RuntimeException 和 Error 会被包装在 CompletionException 中。当调用 `getCause()` 时才能获取原始异常。

**问题二：为什么要在异步代码中包装异常？**

异步任务可能在不同的线程中执行，异常需要一种方式从子线程传播到主线程。CompletionException 作为容器，将异步上下文中的异常与调用线程的异常区分开来。

**问题三：CancellationException 与普通异常有什么区别？**

CancellationException 表示任务被主动取消，是任务生命周期正常结束的一种方式。通过 `future.cancel()` 或在代码中抛出 `CancellationException` 触发。

**问题四：如何区分不同类型的异常？**

通过 `instanceof` 判断异常类型。对于 CompletionException，需要先调用 `getCause()` 获取原始异常，再做判断。

**问题五：whenComplete 和 exceptionally 的区别是什么？**

`exceptionally` 只在异常发生时执行，可以返回降级值；`whenComplete` 在成功和异常时都会执行，但不会改变结果或阻止异常传播。

理解 `CompletableFuture` 的异常包装策略，是写出健壮异步代码的基础。异常不是敌人，而是程序告诉我们"出了问题"的方式。学会优雅地处理异常，才能让异步程序真正可靠。
