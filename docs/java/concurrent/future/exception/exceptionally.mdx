# exceptionally：异常时返回默认值

`CompletableFuture.exceptionally()` 是处理异步异常最简洁的方式。当异步任务抛出异常时，它允许你返回一个默认值，让程序继续正常执行，而不会因为异常而中断业务流程。在分布式系统中，这种能力是实现服务降级的基石——当某个服务不可用时，返回一个缓存值或默认值，比让整个系统崩溃要好得多。

## 为什么需要 exceptionally

传统的同步代码中，我们可以使用 try-catch 捕获异常：

```java
public String getData() {
    try {
        return remoteService.call();
    } catch (Exception e) {
        return "default";  // 降级返回
    }
}
```

但在异步编程中，异常的处理方式完全不同。当 `supplyAsync` 中的代码抛出异常时，这个异常会被包装在 `CompletionException` 中，最终传递给 future 的回调。如果你不在回调中处理异常，异常就会"静默消失"，你的程序可能永远等不到结果。

`exceptionally` 的出现，让异步异常的处理变得和同步代码一样直观。它接收一个函数，这个函数的参数是抛出的异常，返回值是降级后的默认值。

## 基本用法详解

```java
CompletableFuture.supplyAsync(() -> {
    // 模拟可能失败的操作
    if (Math.random() > 0.5) {
        throw new RuntimeException("随机错误");
    }
    return "success";
})
.exceptionally(ex -> {
    // ex 是 CompletionException，getCause() 才是原始异常
    System.err.println("发生错误: " + ex.getCause().getMessage());
    return "fallback";  // 返回默认值
})
.thenAccept(result -> {
    System.out.println("最终结果: " + result);
});
```

这里有一个关键点需要特别注意：`exceptionally` 接收的参数 `ex` 实际上是 `CompletionException`，而非原始异常。原始异常需要通过 `ex.getCause()` 获取。这是一个容易踩坑的地方——如果你直接打印 `ex.getMessage()`，可能看不到你期望的错误信息。

```java
.exceptionally(ex -> {
    // 正确做法：获取原始异常
    Throwable cause = ex.getCause();
    System.err.println("原始错误: " + cause.getMessage());
    return "fallback";
})
```

## 实战场景：远程服务调用降级

在微服务架构中，调用远程服务是家常便饭，但远程服务不可用也是家常便饭。`exceptionally` 让我们能够优雅地处理这种情况：

```java
public class ProductService {

    private final Map<String, Product> cache = new ConcurrentHashMap<>();

    public CompletableFuture<Product> getProduct(String productId) {
        return CompletableFuture.supplyAsync(() -> {
            // 尝试从远程服务获取
            return remoteProductService.fetch(productId);
        })
        .exceptionally(ex -> {
            // 远程服务调用失败，记录日志
            log.error("获取商品 {} 失败: {}", productId, ex.getCause().getMessage());

            // 尝试从本地缓存获取
            Product cached = cache.get(productId);
            if (cached != null) {
                log.info("使用缓存商品 {}", productId);
                return cached;
            }

            // 返回默认商品
            return getDefaultProduct(productId);
        });
    }

    private Product getDefaultProduct(String productId) {
        return new Product(productId, "默认商品", BigDecimal.ZERO);
    }
}
```

这个例子展示了 `exceptionally` 的典型用法：记录错误日志、尝试备用方案、返回默认值。整个过程是链式的，调用方不需要关心底层的降级逻辑。

## exceptionally 与 thenApply 的组合

`exceptionally` 不仅仅可以返回默认值，它返回的值会继续在链中传递。这意味着你可以在 `exceptionally` 之后继续使用 `thenApply`、`thenAccept` 等方法：

```java
CompletableFuture<Integer> future = CompletableFuture
    .supplyAsync(() -> {
        if (Math.random() > 0.5) {
            throw new RuntimeException("计算失败");
        }
        return 100;
    })
    .exceptionally(ex -> {
        System.err.println("发生错误: " + ex.getMessage());
        return 50;  // 默认返回 50
    })
    .thenApply(value -> {
        // value 可能是 100（正常）或 50（降级）
        return value * 2;  // 最终结果是 200 或 100
    });

System.out.println(future.join());  // 打印 200 或 100
```

这种设计非常强大——降级值可以继续参与后续的计算，就像没有发生过错误一样。

## 异常类型的精确处理

有时候，我们希望根据异常类型做不同的处理。`exceptionally` 的参数函数可以判断异常类型并返回不同的值：

```java
CompletableFuture<User> future = CompletableFuture
    .supplyAsync(() -> queryUser(userId))
    .exceptionally(ex -> {
        if (ex.getCause() instanceof UserNotFoundException) {
            // 用户不存在，返回 null 或特定对象
            return null;
        } else if (ex.getCause() instanceof NetworkException) {
            // 网络错误，返回缓存用户或离线数据
            return getCachedUser(userId);
        } else {
            // 其他未知错误，记录并返回默认值
            log.error("未知错误", ex);
            return User.GUEST;
        }
    });
```

## 避免 exceptionally 的过度使用

虽然 `exceptionally` 很好用，但也不是万能的。以下几个场景需要考虑其他方案：

**场景一：需要记录异常但不降级**

如果你只是想在异常发生时记录日志，而不想返回默认值，`handle()` 方法更合适：

```java
.future.handle((result, ex) -> {
    if (ex != null) {
        log.error("操作失败", ex);
        // 不返回默认值，让异常继续传播
        throw new RuntimeException(ex);
    }
    return result;
})
```

**场景二：需要在异常和正常情况下都执行清理逻辑**

使用 `whenComplete()` 方法，无论成功还是失败都会执行：

```java
.future.whenComplete((result, ex) -> {
    // 清理资源，无论成功还是失败
    releaseResource();

    if (ex != null) {
        log.error("操作失败", ex);
    }
})
```

**场景三：需要在异常时执行副作用操作**

`exceptionally` 适合返回降级值，但不适合执行与结果无关的副作用（比如发送监控告警）。这时候应该用 `whenComplete()` 或在 `exceptionally` 中同时处理：

```java
.exceptionally(ex -> {
    // 记录日志
    log.error("服务调用失败", ex);

    // 发送告警
    alertService.send("service_call_failed", ex);

    // 返回降级值
    return "fallback";
})
```

## 与 CompletableFuture 其他方法的对比

`exceptionally` 只是 `CompletableFuture` 提供的异常处理方式之一。根据不同场景，还有 `handle()` 和 `whenComplete()` 可供选择：

| 方法 | 入参 | 返回值 | 适用场景 |
| --- | --- | --- | --- |
| exceptionally | 异常 | T（降级值） | 只处理异常，返回降级结果 |
| handle | (结果, 异常) | T | 同时处理成功和异常 |
| whenComplete | (结果, 异常) | void | 只做副作用，不改变结果 |

选择建议：只需要返回降级值时用 `exceptionally`，需要根据结果和异常综合计算时用 `handle`，只需要执行清理或日志等副作用时用 `whenComplete`。

## 总结

`exceptionally` 是 `CompletableFuture` 中最简洁的异常处理方式，特别适合以下场景：

- 远程服务调用失败时的服务降级
- 缓存穿透后的默认值返回
- 异常情况的优雅恢复

关键要点：记住 `exceptionally` 接收的是 `CompletionException`，需要用 `getCause()` 获取原始异常；降级值会继续在链中传递；不要在 `exceptionally` 中执行过于复杂的逻辑，保持降级逻辑的简洁和快速。
