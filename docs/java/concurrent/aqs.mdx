## 一、AQS是什么：并发框架的"骨架" 🔴

### 1.1 AQS的定义

AbstractQueuedSynchronizer（AQS）是Doug Lea在JDK 1.5引入的抽象同步框架。它用**一个volatile的state变量** + **一个FIFO双向队列** 实现了两大类同步器：

- **独占模式（Exclusive）**：同一时刻只有一个线程能获取锁，如ReentrantLock
- **共享模式（Shared）**：同一时刻可以有多个线程获取锁，如Semaphore、CountDownLatch

AQS只定义了模板方法，把具体的获取/释放逻辑留给子类实现。这就是著名的**模板方法设计模式**。

### 1.2 AQS的核心结构

```java
public abstract class AbstractQueuedSynchronizer
    extends AbstractOwnableSynchronizer {

    // 核心状态变量：volatile int，锁的计数
    private volatile int state;

    // 等待队列：CLH队列的变体
    private transient volatile Node head;
    private transient volatile Node tail;

    // Node是队列中的等待节点
    static final class Node {
        volatile int waitStatus;       // 等待状态
        volatile Node prev;            // 前驱
        volatile Node next;            // 后继
        volatile Thread thread;        // 等待的线程
        int nextWaiter;               // 指向下一个共享模式节点
        // ...
    }
}
```

### 1.3 state变量的语义

`state`是AQS最核心的变量，它的含义由子类定义：

- **ReentrantLock**：state表示锁的重入次数（0=未被持有，n=被持有n次）
- **ReentrantReadWriteLock**：state的高16位表示读锁计数，低16位表示写锁计数
- **Semaphore**：state表示剩余许可数
- **CountDownLatch**：state表示初始计数

## 二、CLH队列：AQS的等待队列 🔴

### 2.1 CLH队列的起源

CLH队列（Craig, Landin, Hagersten队列）是一种**无锁链表**，最初用于SMP系统中的锁实现。它的核心思想是：用CAS保证链表操作的原子性，避免使用重量级的锁。

AQS的CLH队列与原始CLH略有不同——它是**变体版本**，支持**取消**和**超时**。

### 2.2 队列节点结构

```java
static final class Node {
    // 等待状态
    static final int CANCELLED =  1;   // 已取消
    static final int SIGNAL    = -1;   // 后继节点需要被唤醒
    static final int CONDITION = -2;   // 节点在条件队列中等待
    static final int PROPAGATE = -3;    // 共享模式传播

    volatile int waitStatus;
    volatile Node prev;
    volatile Node next;
    volatile Thread thread;
    Node nextWaiter; // SHARED时为SHARED_MARKER，否则是条件队列的next

    final boolean isShared() {
        return nextWaiter == SHARED_MARKER;
    }
}
```

### 2.3 等待状态（waitStatus）的语义

| 状态值 | 含义 | 说明 |
|--------|------|------|
| 0 | 初始化 | 节点刚创建 |
| 1 | CANCELLED | 已取消，等待超时或被中断 |
| -1 | SIGNAL | 后继节点需要被唤醒 |
| -2 | CONDITION | 节点在条件队列中等待 |
| -3 | PROPAGATE | 共享模式，头节点需要向后传播 |

**SIGNAL状态是AQS最重要的状态之一**。一个节点在阻塞前，必须确保前驱节点释放锁时会唤醒它。前驱节点在释放锁时，会将自己的waitStatus从-1改为0，并唤醒后继节点。

## 三、独占模式：acquire/release 🔴

### 3.1 acquire流程

```java
public final void acquire(int arg) {
    if (!tryAcquire(arg) &&               // ① 子类实现：尝试获取锁
        acquireQueued(
            addWaiter(Node.EXCLUSIVE), arg)) {  // ② 获取失败，加入队列
        selfInterrupt();                  // ③ 如果被中断，补中断标记
    }
}
```

流程分解：

**① tryAcquire(arg)**：子类实现，尝试获取锁。
- ReentrantLock.NonfairSync的tryAcquire：直接CAS修改state
- ReentrantLock.FairSync的tryAcquire：先检查队列中是否有更早等待的线程

**② addWaiter(Node.EXCLUSIVE)**：创建新节点，加入队列尾部
```java
private Node addWaiter(Node mode) {
    Node node = new Node(Thread.currentThread(), mode);
    Node pred = tail;
    if (pred != null) {
        node.prev = pred;
        if (compareAndSetTail(pred, node)) {
            pred.next = node;
            return node;
        }
    }
    enq(node); // 队列为空或CAS失败，自旋入队
    return node;
}
```

**③ acquireQueued(node, arg)**：节点在队列中自旋等待
```java
final boolean acquireQueued(Node node, int arg) {
    boolean failed = true;
    try {
        boolean interrupted = false;
        for (;;) {
            Node p = node.predecessor(); // 获取前驱节点
            if (p == head && tryAcquire(arg)) { // 只有前驱是head才尝试获取
                setHead(node);
                p.next = null; // help GC
                failed = false;
                return interrupted;
            }
            // 前驱不是head，或者获取锁失败，检查是否需要阻塞
            if (shouldParkAfterFailedAcquire(p, node) &&
                parkAndCheckInterrupt()) {
                interrupted = true;
            }
        }
    } finally {
        if (failed)
            cancelAcquire(node);
    }
}
```

**关键点**：节点只在**自己是队列第一个有效节点（prev==head）** 时才尝试获取锁。这是公平锁的实现基础。

### 3.2 release流程

```java
public final boolean release(int arg) {
    if (tryRelease(arg)) {       // ① 子类实现：尝试释放锁
        Node h = head;
        if (h != null && h.waitStatus != 0) {
            unparkSuccessor(h);  // ② 唤醒后继节点
        }
        return true;
    }
    return false;
}

private void unparkSuccessor(Node node) {
    int ws = node.waitStatus;
    if (ws < 0)
        compareAndSetWaitStatus(node, ws, 0); // 清除SIGNAL状态

    Node s = node.next; // 后继节点
    if (s == null || s.waitStatus > 0) { // 后继不存在或已取消
        s = null;
        // 从tail向前找到最前边的有效后继
        for (Node t = tail; t != null && t != node; t = t.prev) {
            if (t.waitStatus <= 0)
                s = t;
        }
    }
    if (s != null)
        LockSupport.unpark(s.thread); // 唤醒线程
}
```

## 四、共享模式：acquireShared/releaseShared 🟡

### 4.1 共享模式与独占模式的核心区别

- **独占模式**：同一时刻只有一个线程能持有锁，锁释放后唤醒**一个**后继节点
- **共享模式**：同一时刻可以有多个线程持有锁，锁释放后唤醒**所有**后继节点

### 4.2 acquireShared流程

```java
public final void acquireShared(int arg) {
    if (tryAcquireShared(arg) < 0) {  // ① 子类实现，返回剩余许可数
        doAcquireShared(arg);          // ② 获取失败，加入队列
    }
}

private void doAcquireShared(int arg) {
    Node node = addWaiter(Node.SHARED);
    boolean failed = true;
    try {
        boolean interrupted = false;
        for (;;) {
            Node p = node.predecessor();
            if (p == head) {
                int r = tryAcquireShared(arg);
                if (r >= 0) {          // 获取成功
                    setHeadAndPropagate(node, r); // 关键：传播给后继
                    p.next = null;
                    failed = false;
                    if (interrupted)
                        selfInterrupt();
                    return;
                }
            }
            if (shouldParkAfterFailedAcquire(p, node) &&
                parkAndCheckInterrupt()) {
                interrupted = true;
            }
        }
    } finally {
        if (failed)
            cancelAcquire(node);
    }
}
```

**setHeadAndPropagate**：获取成功后，将自己设为head，并**传播给后继节点**——继续唤醒更多等待的共享模式节点。这就是CountDownLatch和Semaphore能同时唤醒多个线程的机制。

### 4.3 releaseShared流程

```java
public final boolean releaseShared(int arg) {
    if (tryReleaseShared(arg)) {  // 子类实现，增加许可数
        doReleaseShared();         // 唤醒所有后继
        return true;
    }
    return false;
}
```

## 五、如何用AQS实现自定义同步器 🟡

### 5.1 实现一个Mutex（互斥锁）

```java
public class Mutex implements Lock {

    private final Sync sync = new Sync();

    private static class Sync extends AbstractQueuedSynchronizer {
        // 尝试获取锁：只有state==0时才能获取
        protected boolean tryAcquire(int ignores) {
            return compareAndSetState(0, 1);
        }

        // 尝试释放锁：重置state为0
        protected boolean tryRelease(int ignores) {
            setState(0);
            return true;
        }

        // 是否持有锁
        protected boolean isHeldExclusively() {
            return getState() == 1;
        }

        // 创建Condition
        public Condition newCondition() {
            return newConditionObject();
        }
    }

    public void lock() { sync.acquire(1); }
    public boolean tryLock() { return sync.tryAcquire(1); }
    public void unlock() { sync.release(1); }
    public Condition newCondition() { return sync.newCondition(); }
    public void lockInterruptibly() throws InterruptedException {
        sync.acquireInterruptibly(1);
    }
    public boolean tryLock(long timeout, TimeUnit unit)
            throws InterruptedException {
        return sync.tryAcquireNanos(1, unit.toNanos(timeout));
    }
}
```

整个Mutex只需要覆写三个方法：`tryAcquire`、`tryRelease`、`isHeldExclusively`。AQS处理了所有队列管理、等待、唤醒的逻辑。

## 六、❌ 错误示范

**候选人原话**："AQS是一个队列，存储等待锁的线程。"

**问题诊断**：
- 只说对了一半——AQS确实有一个队列（CLH队列），但AQS本身是抽象同步框架，不只是队列
- 没有理解state变量的核心作用
- 没有区分独占模式和共享模式

**候选人原话**："CLH队列是一种无锁队列，所以不需要CAS操作。"

**问题诊断**：
- CLH队列虽然避免了传统的锁（mutex），但它的入队和出队仍然需要CAS
- CLH队列的"无锁"指的是不需要互斥锁来保护队列本身，而不是说不需要原子操作

## 七、面试应对脚本

**面试官**："AQS的acquire流程是什么？"
**标准回答**："acquire调用tryAcquire尝试获取锁。获取成功则返回；获取失败则创建独占模式的节点，加入CLH队列尾部。然后在acquireQueued中自旋：节点只有在前驱是head时才尝试获取锁。获取失败后，通过shouldParkAfterFailedAcquire判断是否需要阻塞——如果前驱节点的waitStatus是SIGNAL，则阻塞当前线程。这个流程的关键是：公平性由'只有前驱是head才尝试获取'来实现，锁的释放通过unparkSuccessor唤醒后继节点实现。"

**面试官追问**："AQS的state变量有什么作用？"
**标准回答**："state是AQS的核心状态变量，volatile修饰保证可见性。它的语义由子类定义：ReentrantLock用它表示重入次数，Semaphore用它表示剩余许可数，CountDownLatch用它表示倒计时值。tryAcquire和tryRelease方法负责原子地修改state，所有对state的修改都通过compareAndSet保证原子性。"

**面试官追问**："AQS如何实现可中断和超时等待？"
**标准回答**："acquireInterruptibly()在acquire的基础上检查interrupted标记，中断时抛InterruptedException。tryAcquireNanos()使用LockSupport.parkNanos()实现限时阻塞——超时后节点从队列中移除（cancelAcquire），方法返回false。这两个方法都是通过acquire()的模板变体实现的。"

【面试官心理】
能回答AQS的候选人，说明他对Doug Lea的设计哲学有认同感。Doug Lea在写AQS时，把所有复杂的等待队列管理、取消、超时、传播机制都封装在框架内部，子类只需要实现三个方法就能得到一个完整的同步器。这种设计思想本身就是最好的架构教科书。
