# AQS 核心思想：state + FIFO 队列

在 Java 并发包里，有一半的同步工具类都是基于同一个「骨架」构建的。这个骨架，就是 **AbstractQueuedSynchronizer**，简称 **AQS**。

`ReentrantLock`、`Semaphore`、`CountDownLatch`、`ReentrantReadWriteLock`、`CyclicBarrier`……它们背后的实现逻辑，都离不开 AQS 的两个核心概念：**state** 和 **FIFO 队列**。

理解 AQS，就理解了 Java 并发工具的半壁江山。

## 为什么需要 AQS？

在 AQS 出现之前，JDK 的同步器都是各自为战：`synchronized` 是 JVM 内部实现，`Object.wait/notify` 是 C 语言级别的底层 API。如果你要写一个「信号量」或「倒计时门栓」，只能自己从头实现。

JDK 开发者 Doug Lea 意识到：**所有的同步器本质上都是在回答同一个问题——「资源够不够，能不能用」**。

他把这个共性抽离出来，形成了 AQS：

```
资源计数（state） + 等待队列（CLH 队列） + 模板方法（acquire/release）
```

## AQS 的两大核心

### 核心一：state 状态变量

`state` 是一个 `volatile int` 类型的变量，代表资源的可用数量：

```java
private volatile int state;

protected final int getState() {
    return state;
}

protected final void setState(int newState) {
    state = newState;
}

protected final boolean compareAndSetState(int expect, int update) {
    return STATE.compareAndSet(this, expect, update);
}
```

不同的同步器对 `state` 有不同的解读：

| 同步器 | state 的含义 |
| --- | --- |
| `ReentrantLock` | 持有锁的线程的重入次数（0 表示未锁定） |
| `Semaphore` | 可用许可证数量 |
| `CountDownLatch` | 倒计时初始值 |
| `ReentrantReadWriteLock` | 高 16 位读锁计数 + 低 16 位写锁计数 |

### 核心二：CLH 队列（FIFO 队列）

CLH（Craig, Landin, Hagersten）队列是一种虚拟的双向链表队列：

```java
// AQS 内部的 Node 类
static final class Node {
    // 共享模式
    static final Node SHARED = new Node();
    // 独占模式
    static final Node EXCLUSIVE = null;

    // 等待状态
    volatile int waitStatus;
    // 前驱节点
    volatile Node prev;
    // 后继节点
    volatile Node next;
    // 关联的线程
    volatile Thread thread;
    // 下一个等待节点
    Node nextWaiter;
}
```

队列的入队和出队：

```
入队：tail ──> [Node(t3)] ──> [Node(t2)] ──> [Node(t1)] <── head
                      新节点插入到 tail 位置

出队：head ──> [Node(t1)] ──> [Node(t2)] ──> [Node(t3)] <── tail
           (已取消)
           head 指向下一个有效节点
```

## 独占模式：acquire/release

### acquire：获取资源（不可中断）

```java
public final void acquire(int arg) {
    if (!tryAcquire(arg) &&
        acquireQueued(addWaiter(Node.EXCLUSIVE), arg)) {
        selfInterrupt();
    }
}
```

流程图：

```
acquire(arg)
    │
    ├── tryAcquire(arg) ── 成功？直接返回
    │                      │
    │                      └── 失败
    │
    ├── addWaiter(Node.EXCLUSIVE) ── 创建 Node，加入队列尾部
    │
    └── acquireQueued(node, arg) ── 自旋 + 阻塞，直到获取成功
```

**`tryAcquire` 是模板方法**，由子类实现：

```java
// ReentrantLock.FairSync 的实现
protected final boolean tryAcquire(int acquires) {
    final Thread current = Thread.currentThread();
    int c = getState();
    if (c == 0) {
        // 无人持有，检查是否是队首（公平锁）
        if (!hasQueuedPredecessors() &&
            compareAndSetState(0, acquires)) {
            setExclusiveOwnerThread(current);
            return true;
        }
    } else if (current == getExclusiveOwnerThread()) {
        // 可重入，state++
        int nextc = c + acquires;
        setState(nextc);
        return true;
    }
    return false;
}
```

### release：释放资源

```java
public final boolean release(int arg) {
    if (tryRelease(arg)) {
        Node h = head;
        if (h != null && h.waitStatus != 0) {
            unparkSuccessor(h);  // 唤醒后继节点
        }
        return true;
    }
    return false;
}

protected boolean tryRelease(int arg) {
    throw new UnsupportedOperationException();
}
```

## 共享模式：acquireShared/releaseShared

### acquireShared：获取资源（可被多个线程同时持有）

```java
public final void acquireShared(int arg) {
    if (tryAcquireShared(arg) < 0) {
        doAcquireShared(arg);  // 入队并阻塞
    }
}

// Semaphore 的实现
protected int tryAcquireShared(int reduces) {
    while (true) {
        int available = getState();
        int remaining = available - reduces;
        if (remaining < 0 ||
            compareAndSetState(available, remaining)) {
            return remaining;
        }
    }
}
```

### releaseShared：释放资源

```java
public final boolean releaseShared(int arg) {
    if (tryReleaseShared(arg)) {
        doReleaseShared();  // 唤醒所有等待的共享节点
        return true;
    }
    return false;
}
```

## 等待状态的演进

Node 的 `waitStatus` 有 5 种值：

| 状态 | 值 | 含义 |
| --- | --- | --- |
| `CANCELLED` | 1 | 线程已取消，等待超时或被中断 |
| `SIGNAL` | -1 | 后继节点需要被唤醒 |
| `CONDITION` | -2 | 节点在条件队列中等待 |
| `PROPAGATE` | -3 | 共享模式下，释放需要传播 |
| `INITIAL` | 0 | 初始状态 |

状态转换图：

```
INITIAL (0)
    │
    ├── 等待中 ──> SIGNAL (-1) ──> 被唤醒，继续竞争
    │
    ├── 取消 ──> CANCELLED (1)  ──> 移出队列
    │
    └── 条件等待 ──> CONDITION (-2)
                        │
                        └── signal() ──> 移入同步队列
```

## 自旋与阻塞的平衡

AQS 的核心问题：**线程应该自旋等待还是阻塞等待？**

| 策略 | 优点 | 缺点 |
| --- | --- | --- |
| 自旋等待 | 响应快（不用上下文切换） | 浪费 CPU |
| 阻塞等待 | 不浪费 CPU | 上下文切换开销大 |

AQS 的解决方案：**自适应自旋**：

```java
final boolean acquireQueued(Node node, int arg) {
    boolean failed = true;
    try {
        boolean interrupted = false;
        for (;;) {
            Node p = node.predecessor();
            if (p == head && tryAcquire(arg)) {
                setHead(node);
                p.next = null;
                failed = false;
                return interrupted;
            }
            // shouldParkAfterFailedAcquire：
            // 如果前驱是 SIGNAL，则阻塞
            // 如果前驱已取消，则跳过
            // 否则设置为 SIGNAL，继续自旋
            if (shouldParkAfterFailedAcquire(p, node)) {
                LockSupport.park(this);  // 阻塞
                interrupted |= Thread.interrupted();
            }
        }
    } finally {
        if (failed)
            cancelAcquire(node);
    }
}
```

## 公平锁 vs 非公平锁

AQS 支持两种模式，只需实现 `tryAcquire` 即可：

```java
// 公平锁：检查队列
static final class FairSync extends Sync {
    protected final boolean tryAcquire(int acquires) {
        final Thread current = Thread.currentThread();
        int c = getState();
        if (c == 0) {
            // 关键：检查是否有等待更久的线程
            if (!hasQueuedPredecessors() &&
                compareAndSetState(0, acquires)) {
                setExclusiveOwnerThread(current);
                return true;
            }
        } else if (current == getExclusiveOwnerThread()) {
            int nextc = c + acquires;
            setState(nextc);
            return true;
        }
        return false;
    }
}

// 非公平锁：直接抢
static final class NonfairSync extends Sync {
    protected final boolean tryAcquire(int acquires) {
        final Thread current = Thread.currentThread();
        int c = getState();
        if (c == 0) {
            // 关键：不检查队列，直接 CAS
            if (compareAndSetState(0, acquires)) {
                setExclusiveOwnerThread(current);
                return true;
            }
        } else if (current == getExclusiveOwnerThread()) {
            int nextc = c + acquires;
            setState(nextc);
            return true;
        }
        return false;
    }
}
```

**非公平锁的性能通常更好**——新来的线程如果运气好，可能在队列线程被唤醒之前就获取到锁（锁刚被释放，缓存还没刷新）。

## 手写一个简单的信号量

理解 AQS 之后，实现一个信号量其实很简单：

```java
public class SimpleSemaphore {
    private final Sync sync;

    private static abstract class Sync extends AbstractQueuedSynchronizer {
        Sync(int permits) {
            setState(permits);
        }

        final int getPermits() {
            return getState();
        }

        protected final boolean tryReleaseShared(int releases) {
            for (;;) {
                int current = getState();
                int next = current + releases;
                if (compareAndSetState(current, next)) {
                    return true;
                }
            }
        }
    }

    private static final class NonfairSync extends Sync {
        NonfairSync(int permits) {
            super(permits);
        }

        protected int tryAcquireShared(int acquires) {
            while (true) {
                int available = getState();
                int remaining = available - acquires;
                if (remaining < 0 ||
                    compareAndSetState(available, remaining)) {
                    return remaining;
                }
            }
        }
    }

    public SimpleSemaphore(int permits) {
        sync = new NonfairSync(permits);
    }

    public void acquire() throws InterruptedException {
        sync.acquireSharedInterruptibly(1);
    }

    public void release() {
        sync.releaseShared(1);
    }

    public int availablePermits() {
        return sync.getPermits();
    }
}
```

## 总结

AQS 的设计哲学是**模板方法模式**：

- **state**：统一的资源计数抽象
- **FIFO 队列**：统一的等待队列抽象
- **tryAcquire / tryRelease**：子类实现的具体策略
- **acquire / release**：AQS 实现的通用流程

```
AQS 骨架
    │
    ├── tryAcquire(state) ── 子类实现：获取资源的策略
    ├── tryRelease(state) ── 子类实现：释放资源的策略
    ├── acquire() ── AQS 实现：获取失败则入队阻塞
    └── release() ── AQS 实现：释放时唤醒后继节点
```

理解 AQS，就理解了 `ReentrantLock` 的公平/非公平、`Semaphore` 的许可证、`CountDownLatch` 的倒计时、`CyclicBarrier` 的屏障——这一切的共同基础。
