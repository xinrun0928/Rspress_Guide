# 线程池7个核心参数：阿里不允许你不知道的配置

候选人小陈在面试中遇到了这样一个问题：

面试官问："你用过线程池吗？线程池有哪些参数？"

小陈说："有核心线程数、最大线程数、队列。"

面试官追问："那阻塞队列用的是哪个？拒绝策略呢？"

小陈说："用的是LinkedBlockingQueue。"

面试官继续追问："为什么用LinkedBlockingQueue而不是ArrayBlockingQueue？它们的区别是什么？"

小张开始语无伦次。

【面试官心理】
线程池的配置是Java后端开发中最常见的面试题之一。80%的候选人能说出核心线程数和最大线程数，但只有50%能说出全部7个参数，只有30%能讲清楚不同队列的区别和拒绝策略的选择。阿里P6以上的面试，这道题几乎是必问的——因为线程池配置不当导致的线上故障太常见了。

## 一、线程池的7个核心参数 🔴

### 1.1 完整参数列表

```java
public ThreadPoolExecutor(
    int corePoolSize,          // 核心线程数
    int maximumPoolSize,        // 最大线程数
    long keepAliveTime,         // 空闲线程存活时间
    TimeUnit unit,              // keepAliveTime的单位
    BlockingQueue<Runnable> workQueue,  // 阻塞队列
    ThreadFactory threadFactory, // 线程工厂
    RejectedExecutionHandler handler    // 拒绝策略
) { ... }
```

### 1.2 每个参数的作用

**① corePoolSize（核心线程数）**

线程池在正常情况下保持的线程数量。即使线程空闲，也不会被回收（除非设置allowCoreThreadTimeOut）。

```java
// 默认：核心线程不会被回收
pool.prestartAllCoreThreads(); // 预启动所有核心线程

// JDK 1.6+：允许核心线程超时回收
pool.allowCoreThreadTimeOut(true); // 核心线程也会被回收
```

**② maximumPoolSize（最大线程数）**

线程池允许的最大线程数量。任务过多时，线程池会创建新线程直到达到最大线程数。

**③ keepAliveTime + TimeUnit（空闲存活时间）**

非核心线程的最大空闲时间。超过这个时间没有任务执行，非核心线程会被回收。

```
核心线程数 ←─────── 最大线程数
  ↓
  核心线程（不会被回收）
         ↓
  非核心线程（空闲keepAliveTime后被回收）
```

**④ workQueue（阻塞队列）**

用于存放等待执行任务的队列。当线程数达到核心线程数后，新任务进入队列等待。

**⑤ ThreadFactory（线程工厂）**

用于创建新线程的工厂。可以自定义线程名称、优先级、是否为守护线程等。

```java
ThreadFactory factory = new ThreadFactoryBuilder()
    .setNameFormat("worker-%d")
    .setDaemon(false)
    .setPriority(Thread.NORM_PRIORITY)
    .build();
```

**⑥ RejectedExecutionHandler（拒绝策略）**

当线程池和队列都满了，无法接受新任务时的处理策略。

## 二、四种阻塞队列的对比 🔴

### 2.1 ArrayBlockingQueue vs LinkedBlockingQueue

| 维度 | ArrayBlockingQueue | LinkedBlockingQueue |
|------|-------------------|--------------------|
| 底层结构 | 有界数组 | 无界链表 |
| 容量 | 必须指定（有界） | 可选（默认Integer.MAX_VALUE，无界） |
| 吞吐量 | 较低（有界数组操作） | 较高（链表CAS操作） |
| 内存开销 | 固定（创建时分配） | 动态（按需分配节点） |
| 公平性 | 支持公平/非公平 | 非公平 |

**ArrayBlockingQueue适合场景**：需要严格的队列容量限制，防止内存溢出。

**LinkedBlockingQueue适合场景**：任务量可控，希望高吞吐量（但要小心Integer.MAX_VALUE的陷阱）。

### 2.2 SynchronousQueue

```java
// 没有任何缓冲，直接传递
new SynchronousQueue<>();

// 使用场景：任务必须被立即消费，不允许积压
new ThreadPoolExecutor(1, Integer.MAX_VALUE, 60L, TimeUnit.SECONDS,
    new SynchronousQueue<>());
```

SynchronousQueue不存储元素——每个put操作必须等待一个take操作。JDK的CachedThreadPool使用了这种队列。

**适用场景**：
- 任务耗时短、执行快、需要快速响应的场景
- 不允许任务积压，必须立即处理

### 2.3 PriorityBlockingQueue

```java
// 按优先级排序的队列
new PriorityBlockingQueue<>(10, Comparator.comparingInt(Task::getPriority));
```

**适用场景**：任务有优先级，高优先级任务需要优先执行（如任务调度系统）。

## 三、常见线程池的配置 🔴

### 3.1 FixedThreadPool：固定大小线程池

```java
Executors.newFixedThreadPool(nThreads);
// 底层等价于：
new ThreadPoolExecutor(nThreads, nThreads, 0L,
    TimeUnit.MILLISECONDS, new LinkedBlockingQueue<>());
```

**特点**：
- 核心线程数 = 最大线程数 = n
- 无界队列（Integer.MAX_VALUE）
- 没有线程会被回收
- **危险**：队列无限增长，可能导致OOM

### 3.2 CachedThreadPool：缓存线程池

```java
Executors.newCachedThreadPool();
// 底层等价于：
new ThreadPoolExecutor(0, Integer.MAX_VALUE, 60L,
    TimeUnit.SECONDS, new SynchronousQueue<>());
```

**特点**：
- 核心线程数=0，最大线程数=无限制
- 60秒空闲后回收线程
- 使用SynchronousQueue，不存储任务
- **危险**：任务量巨大时，可能创建Integer.MAX_VALUE个线程，导致OOM

### 3.3 SingleThreadExecutor：单线程池

```java
Executors.newSingleThreadExecutor();
// 底层等价于：
new ThreadPoolExecutor(1, 1, 0L,
    TimeUnit.MILLISECONDS, new LinkedBlockingQueue<>());
```

**特点**：保证任务顺序执行，不会并发。

### 3.4 ScheduledThreadPool：定时线程池

```java
Executors.newScheduledThreadPool(corePoolSize);
// 底层使用 DelayedWorkQueue
new ScheduledThreadPoolExecutor(corePoolSize);
```

**特点**：支持延迟任务和周期性任务。

### 3.5 为什么不推荐Executors创建线程池

```java
// ❌ 阿里规范禁止：FixedThreadPool和CachedThreadPool都有OOM风险
ExecutorService fixed = Executors.newFixedThreadPool(100);
// 队列容量是Integer.MAX_VALUE → 任务堆积 → OOM

ExecutorService cached = Executors.newCachedThreadPool();
// 最大线程数是Integer.MAX_VALUE → 创建过多线程 → OOM

// ✅ 正确做法：手动配置线程池
new ThreadPoolExecutor(
    corePoolSize, maximumPoolSize,
    keepAliveTime, TimeUnit.SECONDS,
    new ArrayBlockingQueue<>(1000), // 有界队列，防止OOM
    new ThreadFactoryBuilder().setNameFormat("biz-worker-%d").build(),
    new ThreadPoolExecutor.AbortPolicy() // 拒绝策略
);
```

## 四、线程工厂（ThreadFactory）的实战配置 🟡

### 4.1 自定义ThreadFactory

```java
public class NamingThreadFactory implements ThreadFactory {
    private final AtomicInteger poolNumber = new AtomicInteger(1);
    private final ThreadFactory defaultFactory = Executors.defaultThreadFactory();

    public Thread newThread(Runnable r) {
        Thread thread = defaultFactory.newThread(r);
        thread.setName("biz-pool-" + poolNumber.getAndIncrement()
            + "-thread-" + thread.getId());
        return thread;
    }
}
```

### 4.2 设置守护线程的注意事项

```java
// ❌ 危险：线程池的线程是守护线程
new ThreadFactory() {
    public Thread newThread(Runnable r) {
        Thread t = new Thread(r);
        t.setDaemon(true); // JVM退出时所有守护线程都会被强制终止
        return t;
    }
};

// ✅ 正确：非守护线程
t.setDaemon(false);
```

**注意**：Spring Boot的异步任务默认使用`SimpleAsyncTaskExecutor`，可以通过配置改为线程池。

## 五、❌ 错误示范

**候选人原话**："线程池的参数不重要，直接用Executors.newFixedThreadPool(10)就行。"

**问题诊断**：
- 阿里内部规范明确禁止使用Executors创建线程池
- FixedThreadPool使用无界队列，有OOM风险
- 说明候选人没有处理过线上生产问题

**候选人原话**："keepAliveTime是核心线程的存活时间。"

**问题诊断**：
- keepAliveTime是非核心线程的存活时间
- 核心线程默认不会被回收（除非设置allowCoreThreadTimeOut=true）
- 混淆了概念

【面试官心理】
线程池的7个参数是Java后端开发的基础知识。能完整说出7个参数的候选人，说明他有配置线程池的经验。能讲清楚不同队列区别和拒绝策略的候选人，说明他对线程池的原理有深入理解。这类候选人在处理线上高并发问题时，不会犯"线程池配置错误导致OOM"这种低级错误。

## 六、生产避坑：线程池配置的血泪教训 ⚠️

### 6.1 OOM的第一大元凶：线程池队列配置错误

```java
// ❌ 致命错误：无界队列 + 大任务 → OOM
ThreadPoolExecutor pool = new ThreadPoolExecutor(
    10, 20, 60L, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(), // 无界队列！
    new ThreadFactoryBuilder().setNameFormat("pool-%d").build()
);
// 任务堆积 → 队列无限增长 → OOM

// ✅ 正确：有界队列
new LinkedBlockingQueue<>(1000); // 限制队列大小
```

### 6.2 拒绝策略的配置

```java
// ✅ 根据业务选择拒绝策略

// AbortPolicy：抛出异常（默认，适合需要严格控制的场景）
new ThreadPoolExecutor.AbortPolicy()

// CallerRunsPolicy：由调用者线程执行（削峰填谷）
new ThreadPoolExecutor.CallerRunsPolicy()

// DiscardOldestPolicy：丢弃最老的任务
new ThreadPoolExecutor.DiscardOldestPolicy()

// DiscardPolicy：丢弃最新任务
new ThreadPoolExecutor.DiscardPolicy()

// ✅ 自定义：记录日志并报警
new RejectedExecutionHandler() {
    public void rejectedExecution(Runnable r, ThreadPoolExecutor e) {
        log.error("任务被拒绝: {}", r.toString());
        alert();
    }
}
```

## 七、面试应对脚本

**面试官**："线程池有哪7个参数？你是怎么配置的？"
**标准回答**："7个参数是：corePoolSize（核心线程数）、maximumPoolSize（最大线程数）、keepAliveTime（空闲存活时间）、TimeUnit（时间单位）、workQueue（阻塞队列）、ThreadFactory（线程工厂）、RejectedExecutionHandler（拒绝策略）。我的配置原则是：IO密集型任务用2倍CPU核心数作为核心线程数；CPU密集型用CPU核心数+1；队列用ArrayBlockingQueue（有界）而不是LinkedBlockingQueue（无界）；拒绝策略根据业务选择，CallerRunsPolicy适合需要削峰填谷的场景。所有配置都要避免Executors创建线程池，必须手动配置以防OOM。"

【面试官心理】
能回答完整7个参数并说出配置原则的候选人，说明他有实际配置线程池的经验。这类候选人不会犯"Executors.newFixedThreadPool导致OOM"的错误，因为他们知道LinkedBlockingQueue的默认容量是Integer.MAX_VALUE。
