# 活锁与饥饿

死锁是线程互相等待，谁都不肯放手。活锁呢？线程倒是没有等待——它们一直在「忙碌」，但忙的都是无用功，谁也没前进半步。

还有饥饿——某些线程永远得不到 CPU 时间片或资源，一直处于「吃不饱」的状态。

这三种问题，常常被混为一谈，但实际上有本质区别。

## 活锁：一直在动，却没进展

### 什么是活锁？

活锁的特征：
1. 线程没有被阻塞，一直在「活动」
2. 但线程无法继续执行，因为一直在响应彼此的动作
3. 看似在忙碌，实则在做无用功

**经典场景**：两个人在走廊相遇，互相让路，结果都往同一边让，又卡住了，再让，再卡住——永远过不去。

### 活锁的代码示例

```java
public class LivelockDemo {
    private static final int MAX_RETRIES = 3;

    public static void main(String[] args) {
        final Resource resourceA = new Resource("A");
        final Resource resourceB = new Resource("B");

        Thread t1 = new Thread(() -> {
            int retries = 0;
            while (retries < MAX_RETRIES) {
                try {
                    if (resourceA.tryAcquire(100)) {
                        System.out.println("线程1 获取到资源 A");
                        Thread.sleep(50);  // 模拟处理

                        if (resourceB.tryAcquire(50)) {
                            System.out.println("线程1 获取到资源 B");
                            // 干活...
                            resourceB.release();
                        } else {
                            // 获取 B 失败，释放 A，重试
                            System.out.println("线程1 获取 B 失败，释放 A");
                            resourceA.release();
                            retries++;
                            Thread.sleep(10);
                            continue;
                        }
                    }
                    resourceA.release();
                    break;  // 成功完成
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                    break;
                }
            }
            if (retries >= MAX_RETRIES) {
                System.out.println("线程1 重试次数过多，放弃");
            }
        });

        Thread t2 = new Thread(() -> {
            int retries = 0;
            while (retries < MAX_RETRIES) {
                try {
                    if (resourceB.tryAcquire(100)) {
                        System.out.println("线程2 获取到资源 B");
                        Thread.sleep(50);

                        if (resourceA.tryAcquire(50)) {
                            System.out.println("线程2 获取到资源 A");
                            // 干活...
                            resourceA.release();
                        } else {
                            System.out.println("线程2 获取 A 失败，释放 B");
                            resourceB.release();
                            retries++;
                            Thread.sleep(10);
                            continue;
                        }
                    }
                    resourceB.release();
                    break;
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                    break;
                }
            }
            if (retries >= MAX_RETRIES) {
                System.out.println("线程2 重试次数过多，放弃");
            }
        });

        t1.start();
        t2.start();
    }

    static class Resource {
        private final String name;
        private final Lock lock = new ReentrantLock();

        public Resource(String name) {
            this.name = name;
        }

        public boolean tryAcquire(long timeoutMs) {
            return lock.tryLock(timeoutMs, TimeUnit.MILLISECONDS);
        }

        public void release() {
            lock.unlock();
        }
    }
}
```

### 活锁的常见原因

| 原因 | 描述 | 示例 |
| --- | --- | --- |
| 锁获取策略冲突 | 多个线程采用相同的重试策略 | 两个人同时让同一侧 |
| 状态检查冲突 | 检查-行动之间状态发生变化 | 乐观锁重试冲突 |
| 资源竞争重试 | 多个线程同时释放并重试同一资源 | 网络超时重试 |

### 解决活锁的方法

**方法一：随机退避**

不让所有线程都用相同的策略，加入随机性：

```java
public class RandomizedLivelockFix {
    private final Random random = new Random();

    public void doWithBackoff() {
        int retries = 0;
        while (!success()) {
            // 随机等待时间，避免同步
            int backoff = 1 + random.nextInt(100);  // 1-100ms
            try {
                Thread.sleep(backoff);
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
                break;
            }
            retries++;
            if (retries > 100) break;
        }
    }
}
```

**方法二：设置最大重试次数**

```java
private static final int MAX_LIVELOCK_RETRIES = 5;

public boolean tryAcquireWithLivelockProtection() {
    int retries = 0;
    while (retries < MAX_LIVELOCK_RETRIES) {
        if (acquireLock()) {
            return true;
        }
        retries++;
        // 指数退避
        try {
            Thread.sleep((long) Math.pow(2, retries));
        } catch (InterruptedException e) {
            return false;
        }
    }
    // 多次失败后，主动放弃或降级
    return false;
}
```

**方法三：引入「礼让」机制**

让某些线程主动等待更长时间：

```java
public class PolitenessLivelockFix {
    private final AtomicInteger turn = new AtomicInteger(0);

    public void process(int threadId) {
        while (true) {
            // 只有轮到自己才能获取锁
            while (turn.get() != threadId) {
                Thread.yield();  // 让出 CPU
            }

            if (tryAcquireResources()) {
                // 干活...
                break;
            } else {
                // 获取失败，切换到对方
                turn.set(1 - threadId);
                Thread.sleep(10);
            }
        }
    }
}
```

## 饥饿：永远得不到资源

### 什么是饥饿？

饥饿的特征：
1. 线程需要资源（比如 CPU 时间片、锁、I/O）
2. 资源被其他线程长期占用或优先分配给其他线程
3. 该线程永远无法获得所需的资源

**经典场景**：食堂排队，有人一直插队，你永远排不到窗口。

### 饥饿的常见原因

| 原因 | 描述 | 示例 |
| --- | --- | --- |
| 线程优先级 | 高优先级线程抢占了所有资源 | 实时线程饿死后台线程 |
| 持锁时间过长 | 持有锁的线程执行时间过长 | 某个任务占着锁不释放 |
| 调度器偏见 | 操作系统对某些线程有调度偏好 | 某些 JVM 版本对特定线程不公平 |
| 非公平锁 | 新线程可能抢占正在等待的线程 | ReentrantLock 非公平模式 |

### 饥饿的代码示例

```java
public class StarvationDemo {
    public static void main(String[] args) {
        // 高优先级线程
        Thread highPriority = new Thread(() -> {
            while (true) {
                // 模拟繁忙计算
                System.out.println("高优先级执行");
            }
        }, "HighPriority");
        highPriority.setPriority(Thread.MAX_PRIORITY);

        // 低优先级线程
        Thread lowPriority = new Thread(() -> {
            while (true) {
                // 模拟繁忙计算
                System.out.println("低优先级执行");
            }
        }, "LowPriority");
        lowPriority.setPriority(Thread.MIN_PRIORITY);

        highPriority.start();
        lowPriority.start();
    }
}
```

### 饥饿 vs 活锁

| 特征 | 饥饿 | 活锁 |
| --- | --- | --- |
| 线程状态 | 等待状态（Blocked/Waiting） | 运行状态（Runnable） |
| CPU 消耗 | 不消耗 CPU | 消耗 CPU（可能在做无用功） |
| 是否在动 | 不动（等待资源） | 在动（但无进展） |
| 恢复条件 | 资源被释放 | 改变策略或条件 |

### 解决饥饿的方法

**方法一：公平锁**

```java
// 非公平锁：新线程可能插队
ReentrantLock unfairLock = new ReentrantLock();

// 公平锁：先来先服务
ReentrantLock fairLock = new ReentrantLock(true);  // true = 公平模式

// 公平锁的使用
fairLock.lock();
try {
    // 临界区
} finally {
    fairLock.unlock();
}
```

:::warning 公平锁的性能代价
公平锁需要维护等待队列，每次获取锁都需要检查队列，性能通常比非公平锁差 2-10 倍。除非对公平性有严格要求，否则慎用。
:::

**方法二：限制持锁时间**

```java
public class TimeBoundedLock {
    private final ReentrantLock lock = new ReentrantLock();
    private static final long MAX_HOLD_TIME_MS = 100;

    public void process() {
        long start = System.currentTimeMillis();
        lock.lock();
        try {
            while (System.currentTimeMillis() - start < MAX_HOLD_TIME_MS) {
                doChunk();  // 分批处理，每次处理一部分
            }
        } finally {
            lock.unlock();
        }
    }

    private void doChunk() { /* 处理一小部分 */ }
}
```

**方法三：优先级提升（防止长期饥饿）**

```java
public class PriorityInheritanceDemo {
    public void handleWithPriorityBoost() {
        // 某些场景下，可以让持锁线程临时提升优先级
        Thread current = Thread.currentThread();
        int originalPriority = current.getPriority();
        try {
            current.setPriority(Math.min(originalPriority + 2, Thread.MAX_PRIORITY));
            // 快速执行并释放锁
        } finally {
            current.setPriority(originalPriority);
        }
    }
}
```

**方法四：使用 `Thread.yield()` 或 `LockSupport`**

```java
public class YieldDemo {
    public void cooperativeScheduling() {
        synchronized (this) {
            // 干一部分活
            doPartialWork();

            // 让出 CPU，让其他线程有机会运行
            Thread.yield();

            // 继续干活
            doRestWork();
        }
    }
}
```

## 三种问题的对比

| 特征 | 死锁 | 活锁 | 饥饿 |
| --- | --- | --- | --- |
| 线程状态 | Blocked（阻塞） | Runnable（运行） | Blocked/Waiting |
| CPU 消耗 | 不消耗 | 消耗 | 不消耗 |
| 是否在动 | 不动 | 在动 | 不动 |
| 资源状态 | 持有资源并等待 | 释放并重试 | 等待资源 |
| 恢复方式 | 外部干预 | 改变策略 | 资源分配改善 |

## 实际开发建议

### 避免活锁

1. **不要用完全相同的重试策略**：加入随机性或轮转机制
2. **设置最大重试次数**：多次失败后主动放弃或降级
3. **减少锁粒度**：缩小持有锁的时间

### 避免饥饿

1. **慎用线程优先级**：生产环境优先级效果不可靠
2. **使用公平锁时谨慎**：考虑性能影响
3. **持锁时间要短**：锁内只做必要的操作
4. **监控线程状态**：如果发现某些线程长期处于 Waiting 状态，排查饥饿问题

```java
// 监控线程状态
ThreadMXBean bean = ManagementFactory.getThreadMXBean();
long[] deadlockedIds = bean.findDeadlockedThreads();
if (deadlockedIds != null) {
    // 发现死锁线程
}

long[] blockedIds = bean.findBlockedMonitorThreads();
if (blockedIds != null) {
    // 发现阻塞线程
}
```

## 总结

活锁和饥饿是比死锁更隐蔽的问题：

- **活锁**：线程一直在动，但因为互相冲突而没有进展。解决方法是加入随机退避、限制重试次数、或引入礼让机制。
- **饥饿**：线程永远得不到资源。解决方法是使用公平锁、限制持锁时间、或调整资源分配策略。
- **三者对比**：死锁不动不消耗 CPU，活锁在动消耗 CPU，饥饿不动不消耗 CPU。

理解这三种问题的区别，才能在排查线上问题时准确定位根因。
