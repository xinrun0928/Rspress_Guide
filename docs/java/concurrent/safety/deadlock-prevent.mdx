# 如何避免死锁：破坏四大条件

死锁一旦发生，线程互相等待，谁都无法前进。

上一节我们讲了死锁产生的四个必要条件：**互斥、持有并等待、不可抢占、循环等待**。既然叫「必要条件」，那就意味着——**只要破坏其中任意一个，死锁就不会发生**。

这一节，我们来详细讲解四种破坏死锁的策略，并给出 Java 代码实现。

## 策略一：破坏互斥条件

**思路**：把只能一个线程使用的资源，改为可以多个线程同时使用。

**典型例子**：使用 `CopyOnWriteArrayList` 替代普通的 `ArrayList`。

```java
// 互斥：多个线程同时读写同一个 ArrayList，会出问题
List<String> list = new ArrayList<>();
list.add("a");  // 可能抛出 ConcurrentModificationException

// 非互斥：CopyOnWriteArrayList 每次写入都复制一份，读取不需要加锁
List<String> safeList = new CopyOnWriteArrayList<>();
safeList.add("a");  // 安全
```

**适用场景**：
- 读多写少的场景
- 数据结构本身支持「复制」语义

**局限性**：
- 不是所有资源都能复制，比如锁、打印机
- 复制开销大，不适合频繁写入的场景

:::warning 互斥是悲观策略
互斥是操作系统层面的基本保护机制。破坏互斥意味着改变资源本身的设计，这种方法局限性很大。
:::

## 策略二：破坏持有并等待条件

**思路**：线程一次性获取所有需要的资源，要么全拿到，要么一个都不拿。

**实现方式**：

```java
// 错误的做法：分步获取锁
public void transfer(Account from, Account to, int amount) {
    synchronized (from) {      // 先锁 from
        // 模拟操作延迟，放大死锁概率
        try { Thread.sleep(100); } catch (InterruptedException ignored) {}
        synchronized (to) {   // 再锁 to —— 可能死锁！
            from.debit(amount);
            to.credit(amount);
        }
    }
}

// 正确的做法：一次性获取所有锁
public void transfer(Account from, Account to, int amount) {
    // 定义锁的顺序，确保全局一致
    Account first = from.id < to.id ? from : to;
    Account second = from.id < to.id ? to : from;

    synchronized (first) {
        synchronized (second) {
            from.debit(amount);
            to.credit(amount);
        }
    }
}
```

**另一种方案：Lock 的 tryLock**

```java
import java.util.concurrent.TimeUnit;
import java.util.concurrent.locks.ReentrantLock;

public boolean transferWithTryLock(Account from, Account to, int amount) {
    ReentrantLock lock1 = from.id < to.id ? from.lock : to.lock;
    ReentrantLock lock2 = from.id < to.id ? to.lock : from.lock;

    boolean gotFirst = false;
    boolean gotSecond = false;

    try {
        gotFirst = lock1.tryLock(1, TimeUnit.SECONDS);
        if (gotFirst) {
            gotSecond = lock2.tryLock(1, TimeUnit.SECONDS);
        }
        if (gotSecond) {
            from.debit(amount);
            to.credit(amount);
            return true;
        }
        return false;
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
        return false;
    } finally {
        if (gotSecond) lock2.unlock();
        if (gotFirst) lock1.unlock();
    }
}
```

**核心原则**：锁的顺序必须全局一致，不能有分叉。

## 策略三：破坏不可抢占条件

**思路**：如果获取不到锁，就主动释放已持有的资源，而不是一直等待。

**实现方式**：使用 `Lock.tryLock()` + 超时机制。

```java
public class ResourceHolder {
    private final ReentrantLock lock = new ReentrantLock();

    public void work() {
        while (true) {
            // 尝试获取锁，最多等待 1 秒
            if (lock.tryLock(1, TimeUnit.SECONDS)) {
                try {
                    // 执行任务
                    doWork();
                } finally {
                    lock.unlock();
                }
                break;
            } else {
                // 获取失败，释放已持有的资源
                releaseResources();
                // 等待一段时间后重试
                try {
                    Thread.sleep(100);
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                    break;
                }
            }
        }
    }

    private void doWork() { /* ... */ }
    private void releaseResources() { /* ... */ }
}
```

**synchronized 无法破坏不可抢占条件**，因为 synchronized 的锁不能被中断或超时。

:::tip Lock 的优势
`ReentrantLock` 提供了 `tryLock()` 和 `tryLock(timeout)` 方法，可以主动放弃获取锁，从而破坏不可抢占条件。
:::

## 策略四：破坏循环等待条件

**思路**：对所有资源编号，线程按固定顺序获取锁，避免循环等待。

这是**最常用**的死锁避免策略。

### 银行家算法思想

银行家算法是破坏循环等待的经典方法：

```java
public class BankAccount implements Comparable<BankAccount> {
    private final int id;
    private final ReentrantLock lock = new ReentrantLock();
    private int balance = 0;

    public BankAccount(int id, int balance) {
        this.id = id;
        this.balance = balance;
    }

    @Override
    public int compareTo(BankAccount other) {
        return Integer.compare(this.id, other.id);
    }

    public void debit(int amount) {
        balance -= amount;
    }

    public void credit(int amount) {
        balance += amount;
    }

    public ReentrantLock getLock() {
        return lock;
    }
}

public class SafeBankTransfer {
    // 按 id 排序后统一加锁，破坏循环等待
    public static void transfer(BankAccount from, BankAccount to, int amount) {
        // 确保锁的顺序一致
        BankAccount first = from.compareTo(to) < 0 ? from : to;
        BankAccount second = from.compareTo(to) < 0 ? to : from;

        first.getLock().lock();
        try {
            second.getLock().lock();
            try {
                from.debit(amount);
                to.credit(amount);
            } finally {
                second.getLock().unlock();
            }
        } finally {
            first.getLock().unlock();
        }
    }
}
```

### 统一资源获取顺序

```java
// 定义资源枚举，统一编号
public enum Resource {
    PRINTER(1),
    SCANNER(2),
    MODEM(3),
    DISK(4);

    private final int order;

    Resource(int order) {
        this.order = order;
    }

    public int getOrder() {
        return order;
    }
}

// 按编号顺序获取资源
public class UnifiedResourceAcquisition {
    public void useResources(Resource... resources) {
        // 排序确保顺序一致
        Arrays.sort(resources, Comparator.comparingInt(Resource::getOrder));

        Lock[] locks = new Lock[resources.length];
        for (int i = 0; i < resources.length; i++) {
            locks[i] = getLock(resources[i]);
        }

        for (Lock lock : locks) {
            lock.lock();
        }

        try {
            // 使用资源
            for (Resource r : resources) {
                process(r);
            }
        } finally {
            for (int i = locks.length - 1; i >= 0; i--) {
                locks[i].unlock();
            }
        }
    }

    private Lock getLock(Resource r) {
        // 返回对应资源的锁
        return new ReentrantLock();
    }

    private void process(Resource r) { /* ... */ }
}
```

## 四种策略的对比

| 策略 | 破坏条件 | 常用程度 | 实现难度 | 适用场景 |
| --- | --- | --- | --- | --- |
| 破坏互斥 | 互斥条件 | 低 | 低 | 读多写少、无状态数据 |
| 破坏持有并等待 | 持有并等待 | 中 | 中 | 资源数量少、可以预知 |
| 破坏不可抢占 | 不可抢占 | 中 | 高 | 需要使用显式锁 |
| 破坏循环等待 | 循环等待 | **高** | 低 | **最常用** |

## 实际开发建议

### 最佳实践

1. **统一锁顺序**：如果必须获取多个锁，确保所有代码路径都按相同顺序获取。

2. **使用 Lock.tryLock()**：在高并发场景下，使用带超时的 tryLock 避免永久等待。

3. **减少锁的粒度**：拆分大锁为多个小锁，降低死锁概率。

4. **使用并发工具类**：`ConcurrentHashMap`、`BlockingQueue` 等并发工具类内部已经处理了同步问题。

```java
// 推荐：使用并发工具类
BlockingQueue<Task> queue = new LinkedBlockingQueue<>(100);
producer.submit(() -> queue.put(task));
consumer.submit(() -> queue.take());

// 不推荐：自己管理复杂的锁
```

### 代码审查清单

```
□ 代码中是否有多层嵌套的 synchronized？
□ 获取多个锁的顺序是否一致？
□ 是否有获取锁后长时间阻塞的操作？
□ 是否可以使用并发工具类替代？
□ 是否有 tryLock 的超时设置？
```

## 总结

破坏死锁的四大条件：

- **破坏互斥**：使用无锁数据结构（如 CopyOnWriteArrayList）
- **破坏持有并等待**：一次性获取所有资源，或使用 tryLock
- **破坏不可抢占**：使用 Lock.tryLock() 主动放弃
- **破坏循环等待**：统一锁顺序（最常用）

实际开发中，**破坏循环等待**是最简单有效的方法，值得优先考虑。
