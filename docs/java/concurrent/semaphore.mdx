# Semaphore：信号量，控制并发访问的令牌桶

候选人小吴在面试中遇到了这样的问题：

面试官问："如果你的系统有100个并发请求，但数据库连接池只有10个，该怎么处理？"

小吴说："可以加一个限流器。"

面试官追问："那限流器怎么实现？"

小吴说："可以用信号量Semaphore。"

面试官继续追问："那Semaphore是怎么实现的？acquire和release的流程是什么？"

小吴停顿了半天，说不出细节。

【面试官心理】
Semaphore是最直观的并发控制工具之一——它模拟了PV操作（信号量）的概念。但大部分候选人只知道Semaphore"限制并发数"，不知道它底层依赖AQS的共享模式。能讲清楚acquire/release流程的候选人，说明他对AQS的共享模式有深入理解——而这正是CountDownLatch、CyclicBarrier（部分）和Semaphore共同的底层基础。

## 一、Semaphore是什么：信号量模型 🔴

### 1.1 PV操作的起源

信号量（Semaphore）是由荷兰计算机科学家Edsger Dijkstra在1965年提出的并发原语：
- **P()操作**（Proberen = try）：申请一个令牌，如果令牌用完则等待
- **V()操作**（Verhogen = increment）：释放一个令牌，唤醒一个等待者

### 1.2 Semaphore的使用场景

**场景一：数据库连接池限流**

```java
// 数据库连接池只有10个连接，但有100个并发请求
Semaphore semaphore = new Semaphore(10); // 10个许可证

for (int i = 0; i < 100; i++) {
    final int id = i;
    new Thread(() -> {
        try {
            semaphore.acquire(); // 获取一个许可证（拿不到就等）
            try {
                // 使用数据库连接
                useConnection(id);
            } finally {
                semaphore.release(); // 释放许可证
            }
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    }).start();
}
```

**场景二：令牌桶限流**

```java
// 每秒最多处理1000个请求
Semaphore rateLimiter = new Semaphore(1000);

public void handleRequest(Request req) {
    try {
        if (!rateLimiter.tryAcquire(1, 100, TimeUnit.MILLISECONDS)) {
            reject(req); // 超时拒绝
            return;
        }
        try {
            process(req);
        } finally {
            rateLimiter.release();
        }
    } catch (InterruptedException e) {
        reject(req);
    }
}
```

### 1.3 核心概念

- **许可证（permit）**：Semaphore内部维护一个许可证计数
- **acquire()**：获取一个或多个许可证，计数减1，计数为0则阻塞
- **release()**：释放许可证，计数加1，唤醒等待者
- **公平性**：支持公平模式和非公平模式

## 二、Semaphore的底层实现 🔴

### 2.1 AQS共享模式

Semaphore内部有两个实现类：NonfairSync和FairSync，都继承Sync：

```java
private final Sync sync;

// 非公平模式
static final class NonfairSync extends Sync {
    NonfairSync(int permits) { super(permits); }
    protected int tryAcquireShared(int acquires) {
        return nonfairTryAcquireShared(acquires);
    }
}

// 公平模式
static final class FairSync extends Sync {
    FairSync(int permits) { super(permits); }
    protected int tryAcquireShared(int acquires) {
        return fairTryAcquireShared(acquires);
    }
}
```

**state变量的语义**：在Semaphore中，state = 剩余许可证数量。

### 2.2 acquire()的实现

```java
public void acquire() throws InterruptedException {
    sync.acquireSharedInterruptibly(1);
}

public final void acquireSharedInterruptibly(int arg) {
    if (Thread.interrupted())
        throw new InterruptedException();
    if (tryAcquireShared(arg) < 0) // < 0 表示许可不足
        doAcquireSharedInterruptibly(arg); // 进入AQS队列等待
}
```

**非公平模式的tryAcquireShared**：

```java
final int nonfairTryAcquireShared(int acquires) {
    for (;;) {
        int available = getState();
        int remaining = available - acquires;
        if (remaining < 0 ||                   // 许可不足
            compareAndSetState(available, remaining))  // CAS修改
            return remaining;
    }
}
```

**公平模式的tryAcquireShared**：

```java
protected int tryAcquireShared(int acquires) {
    for (;;) {
        if (hasQueuedPredecessors())  // 关键：检查队列！
            return -1;
        int available = getState();
        int remaining = available - acquires;
        if (remaining < 0 ||
            compareAndSetState(available, remaining))
            return remaining;
    }
}
```

**核心差异**：公平模式多了`hasQueuedPredecessors()`检查——如果队列中有人在等待，就不能抢（必须排队）。

### 2.3 release()的实现

```java
public void release() {
    sync.releaseShared(1);
}

public final boolean releaseShared(int arg) {
    if (tryReleaseShared(arg)) { // CAS增加许可
        doReleaseShared();        // 唤醒等待者
        return true;
    }
    return false;
}

protected final boolean tryReleaseShared(int releases) {
    for (;;) {
        int current = getState();
        int next = current + releases;
        if (next < current) // 溢出检查
            throw new Error("Maximum permit count exceeded");
        if (compareAndSetState(current, next))
            return true;
    }
}
```

## 三、非公平模式的"插队"现象 🟡

### 3.1 非公平Semaphore的效率

非公平Semaphore的acquire()在lock()阶段就尝试CAS抢许可证：

```java
// NonfairSync
final void lock() {
    if (compareAndSetState(0, 1)) // 直接CAS抢，不检查队列！
        setExclusiveOwnerThread(Thread.currentThread());
    else
        acquire(1);
}
```

这意味着一个新来的线程可能"插队"到等待队列头部的线程之前拿到许可证。

### 3.2 公平vs非公平的吞吐量对比

| 场景 | 非公平Semaphore | 公平Semaphore |
|------|----------------|--------------|
| 低竞争 | 几乎无差别 | 几乎无差别 |
| 高竞争 | 吞吐量大（允许插队） | 吞吐量低（严格FIFO） |
| 公平性 | 不保证 | 保证FIFO |
| 饥饿风险 | 极低（吞吐量高，许可流转快） | 存在（等待时间长） |

## 四、生产避坑：Semaphore的陷阱 ⚠️

### 4.1 许可证泄漏

```java
// ❌ 危险：忘记release
Semaphore semaphore = new Semaphore(10);
semaphore.acquire();
try {
    doSomething();
    if (error) return; // 忘记release！许可证泄漏！
} finally {
    semaphore.release();
}
```

### 4.2 acquire和release不匹配

```java
// ❌ 错误：release次数多于acquire
semaphore.acquire(5);
semaphore.release(10); // 释放的比获取的多！
```

### 4.3 公平Semaphore的性能陷阱

```java
// ❌ 高并发场景下使用公平Semaphore
Semaphore fairSemaphore = new Semaphore(10, true); // 灾难性性能
```

在高频场景下，公平Semaphore的hasQueuedPredecessors()检查会成为瓶颈。

## 五、❌ 错误示范

**候选人原话**："Semaphore是一种锁，所以可以保证原子性。"

**问题诊断**：
- Semaphore不是锁，是**信号量**
- Semaphore控制的是"许可数量"，不是互斥
- 多个线程可以同时持有同一把Semaphore的多个许可证

**候选人原话**："Semaphore.acquire()会阻塞直到有许可证。"

**问题诊断**：
- `acquire()`会阻塞
- 但`tryAcquire()`会立即返回（拿到返回true，拿不到返回false）
- 还有带超时的`tryAcquire(timeout, unit)`版本

【面试官心理】
Semaphore是JUC中最直接体现"信号量"概念的组件。能回答Semaphore的acquire/release流程的候选人，说明他对AQS的共享模式有了系统化的理解。信号量是一个被发明了60年的并发原语，理解它的原理对于设计并发控制方案有重要的参考价值。

## 六、面试应对脚本

**面试官**："Semaphore是怎么实现的？acquire和release的流程是什么？"
**标准回答**："Semaphore内部有一个Sync类继承AQS的共享模式。state表示剩余许可证数量。acquire()调用acquireSharedInterruptibly()，在tryAcquireShared()中用CAS将state减1。如果state>=acquires则CAS成功返回，线程继续执行；如果state<acquires则返回负数，线程进入AQS共享模式等待队列被阻塞。release()调用releaseShared()，在tryReleaseShared()中用CAS将state加1，然后调用doReleaseShared()唤醒等待队列中的一个线程。公平和非公平的区别在于：公平模式在tryAcquireShared中多了hasQueuedPredecessors()检查，非公平模式直接CAS抢许可证。"

**面试官追问**："Semaphore的acquire(5)和release(5)是什么含义？"
**标准回答**："acquire(5)表示尝试获取5个许可证——如果剩余许可证>=5则成功，state减5；否则阻塞直到获取到5个。release(5)表示释放5个许可证，state加5，并唤醒等待队列中的线程。这常用于限制对批量资源的访问，比如一次获取5个数据库连接执行批量操作。"

【面试官心理】
能回答Semaphore的候选人，说明他对AQS的共享模式有了完整的理解。Semaphore、CountDownLatch、ReentrantReadWriteLock的读锁——这些都用AQS的共享模式实现。能把这三个组件串联起来的候选人，对JUC的理解已经到了体系化的层面。
