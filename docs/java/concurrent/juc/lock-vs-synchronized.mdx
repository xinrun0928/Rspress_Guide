# ReentrantLock vs synchronized：该如何选择？

在 Java 并发编程中，`synchronized` 和 `ReentrantLock` 都能实现线程同步，但它们有各自的特点和适用场景。很多面试官喜欢问这个问题："synchronized 和 ReentrantLock 有什么区别？"今天我们就来深入对比这两者，帮助你在实际开发中做出正确的选择。

## 基本特性对比

| 特性 | synchronized | ReentrantLock |
|------|-------------|---------------|
| **锁获取方式** | 阻塞式，不可中断 | 支持可中断、超时、非阻塞 |
| **公平性** | 非公平 | 公平/非公平可选 |
| **条件变量** | 只能配合 Object.wait/notify | 支持多个 Condition |
| **可重入** | 支持 | 支持 |
| **锁粒度** | 方法级、代码块级 | 代码块级 |
| **性能** | JVM 优化，接近 Lock | JDK 优化，高并发优秀 |
| **异常处理** | 自动释放锁 | 必须在 finally 中手动释放 |

## synchronized 的优势

### 1. 简洁易用

```java
// synchronized：简单直接
public class SynchronizedSimple {
    
    private final Object lock = new Object();
    
    public void doSomething() {
        synchronized (lock) {
            // 业务逻辑
        }
    }
    
    // 方法级别的同步，更简洁
    public synchronized void anotherMethod() {
        // 自动获取 this 锁
    }
    
    // 静态方法同步
    public static synchronized void staticMethod() {
        // 获取类锁
    }
}
```

`synchronized` 不需要担心锁的释放——当离开 synchronized 代码块时，锁会自动释放。即使抛出异常，JVM 也会确保锁被释放。

### 2. JVM 原生支持，深度优化

```java
public class JVMPromotion {
    
    private final Object lock = new Object();
    
    public void optimized() {
        synchronized (lock) {
            // JVM 会进行以下优化：
            // 1. 锁粗化：将连续的同步块合并
            // 2. 锁消除：通过逃逸分析移除不必要的锁
            // 3. 偏向锁：无竞争时消除同步开销
            // 4. 轻量级锁：短时竞争用 CAS 代替阻塞
        }
    }
}
```

`synchronized` 是 JVM 的"亲儿子"，JVM 对它进行了大量的运行时优化。从 JDK 6 开始，synchronized 的性能已经非常接近，甚至在某些场景下超过 ReentrantLock。

### 3. 内存语义

```java
public class MemorySemantics {
    
    private int value = 0;
    private final Object lock = new Object();
    
    public void increment() {
        synchronized (lock) {
            // synchronized 保证：
            // 1. 原子性：同一时刻只有一个线程执行
            // 2. 可见性：锁释放前对共享变量的修改对其他线程可见
            // 3. 有序性：防止指令重排序
            value++;
        }
    }
}
```

`synchronized` 的内存语义是 JVM 内置保证的，不需要开发者理解底层细节。

## ReentrantLock 的优势

### 1. 尝试获取锁

```java
public class TryLockDemo {
    
    private final ReentrantLock lock = new ReentrantLock();
    
    // 场景：限时操作
    public boolean doSomethingWithTimeout() {
        try {
            // 尝试获取锁，最多等待3秒
            if (lock.tryLock(3, TimeUnit.SECONDS)) {
                try {
                    // 业务逻辑
                    return true;
                } finally {
                    lock.unlock();
                }
            } else {
                // 获取锁超时
                return false;
            }
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            return false;
        }
    }
    
    // 场景：可中断等待
    public void interruptibleOperation() throws InterruptedException {
        lock.lockInterruptibly();  // 可响应中断
        try {
            // 业务逻辑
        } finally {
            lock.unlock();
        }
    }
}
```

当需要实现超时获取或可中断获取时，synchronized 无能为力。

### 2. 公平锁

```java
public class FairLockDemo {
    
    // 公平锁：严格按照等待顺序
    private final ReentrantLock fairLock = 
        new ReentrantLock(true);
    
    // 非公平锁（默认）：允许插队
    private final ReentrantLock unfairLock = 
        new ReentrantLock(false);
    
    public void fairAccess() {
        fairLock.lock();
        try {
            // 按 FIFO 顺序执行
        } finally {
            fairLock.unlock();
        }
    }
}
```

`synchronized` 总是非公平的，无法实现公平锁。在需要严格 FIFO 的场景（如数据库连接池），ReentrantLock 是更好的选择。

### 3. 多个条件变量

```java
public class MultipleConditions {
    
    private final ReentrantLock lock = new ReentrantLock();
    private final Condition notEmpty = lock.newCondition();
    private final Condition notFull = lock.newCondition();
    
    private final Object[] items = new Object[10];
    private int count = 0;
    
    public void put(Object item) throws InterruptedException {
        lock.lock();
        try {
            while (count == items.length) {
                notFull.await();  // 等待"不满"条件
            }
            items[count++] = item;
            notEmpty.signal();   // 通知"非空"条件
        } finally {
            lock.unlock();
        }
    }
    
    public Object take() throws InterruptedException {
        lock.lock();
        try {
            while (count == 0) {
                notEmpty.await();  // 等待"非空"条件
            }
            Object item = items[--count];
            notFull.signal();      // 通知"不满"条件
            return item;
        } finally {
            lock.unlock();
        }
    }
}
```

`Object.wait/notify` 只有一套等待队列，无法区分多个条件。使用 Condition 可以为不同的等待条件创建独立的队列。

## 性能对比

### 低竞争场景

```java
public class LowContentionBenchmark {
    
    private final Object syncLock = new Object();
    private final ReentrantLock lock = new ReentrantLock();
    
    // synchronized（偏向锁优化）
    public void synchronizedMethod() {
        synchronized (syncLock) {
            // 偏向锁生效，几乎零开销
        }
    }
    
    // ReentrantLock（非公平）
    public void lockMethod() {
        lock.lock();
        try {
            // 有一定的 CAS 开销
        } finally {
            lock.unlock();
        }
    }
}
```

在低竞争、无竞争场景下，synchronized 的偏向锁几乎零开销，而 ReentrantLock 有一定的 CAS 开销。

### 高竞争场景

```java
public class HighContentionBenchmark {
    
    private final ReentrantLock lock = new ReentrantLock();
    
    public void highContentionMethod() throws InterruptedException {
        // 公平锁在高竞争下吞吐量低
        // 非公平锁在高竞争下吞吐量高
        lock.lock();
        try {
            // 业务逻辑
        } finally {
            lock.unlock();
        }
    }
}
```

在高竞争下，ReentrantLock 的非公平锁吞吐量更高，因为线程可以避免无意义的阻塞和唤醒。

## 异常处理对比

### synchronized：自动释放

```java
public class SyncExceptionHandling {
    
    private final Object lock = new Object();
    
    public void withException() {
        synchronized (lock) {
            // 即使抛出异常，锁也会自动释放
            throw new RuntimeException("error");
        }
    }
    
    // 等价于：
    public void equivalent() {
        lock.lock();
        try {
            throw new RuntimeException("error");
        } finally {
            lock.unlock();  // JVM 确保 finally 执行
        }
    }
}
```

### ReentrantLock：必须手动释放

```java
public class LockExceptionHandling {
    
    private final ReentrantLock lock = new ReentrantLock();
    
    public void correctUsage() {
        lock.lock();
        try {
            // 业务逻辑
            if (someCondition()) {
                throw new RuntimeException("error");
            }
        } finally {
            // 必须手动释放，遗忘会导致死锁
            lock.unlock();
        }
    }
    
    // 错误用法
    public void wrongUsage() {
        lock.lock();
        try {
            if (someCondition()) {
                // 忘记 finally，直接返回
                return;
            }
        } finally {
            // 如果上面的 if return，这里也会执行
            // 但如果忘记写 finally...
            lock.unlock();
        }
    }
}
```

## 实战选择指南

### 推荐使用 synchronized 的场景

```java
public class ChooseSynchronized {
    
    // 场景1：简单的同步需求
    private int counter = 0;
    
    public synchronized void increment() {
        counter++;
    }
    
    // 场景2：不需要超时、可中断等高级特性
    public void simpleSync() {
        synchronized (this) {
            // 简单的互斥
        }
    }
    
    // 场景3：方法级别的同步
    public synchronized Object getValue() {
        return value;
    }
    
    // 场景4：与 wait/notify 配合使用
    public synchronized void waitForSomething() throws InterruptedException {
        while (!condition) {
            wait();  // 等待
        }
        // 继续执行
    }
    
    public synchronized void signal() {
        condition = true;
        notify();  // 通知
    }
}
```

### 推荐使用 ReentrantLock 的场景

```java
public class ChooseReentrantLock {
    
    private final ReentrantLock lock = new ReentrantLock();
    
    // 场景1：需要超时获取
    public boolean tryExecuteWithTimeout() {
        try {
            if (lock.tryLock(5, TimeUnit.SECONDS)) {
                try {
                    // 限时操作
                    return true;
                } finally {
                    lock.unlock();
                }
            }
            return false;  // 超时
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            return false;
        }
    }
    
    // 场景2：需要公平锁
    private final ReentrantLock fairLock = 
        new ReentrantLock(true);
    
    // 场景3：需要多个条件变量
    private final Condition notEmpty = lock.newCondition();
    private final Condition notFull = lock.newCondition();
    
    // 场景4：需要 lockInterruptibly
    public void interruptibleWait() throws InterruptedException {
        lock.lockInterruptibly();
        try {
            // 可中断的操作
        } finally {
            if (lock.isHeldByCurrentThread()) {
                lock.unlock();
            }
        }
    }
}
```

## 总结建议

**优先选择 synchronized**：
- 简单的互斥需求
- 不需要超时、可中断、公平锁等特性
- 代码更简洁，更易维护
- 享受 JVM 原生优化

**选择 ReentrantLock**：
- 需要超时获取锁
- 需要可中断等待
- 需要公平锁保证
- 需要多个条件变量
- 需要查看锁的状态信息

在大多数日常开发中，`synchronized` 足够使用。只有在需要特定功能时，才考虑 `ReentrantLock`。记住：**最简单的解决方案往往是最优的**。
