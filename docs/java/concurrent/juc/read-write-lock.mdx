# ReadWriteLock：读写锁原理

一个缓存系统，有大量并发读请求，偶尔有写请求。

如果用普通的 `synchronized` 或 `ReentrantLock`，读和写都要排队，即使两个线程都在读，也要互相等待。

这是对性能的浪费——**读操作不会改变数据，为什么不能并发读？**

读写锁就是为了解决这个问题。

## 读写锁的核心思想

读写锁允许多个线程同时读，但写操作必须独占：

```
┌─────────────────────────────────────────────────────────┐
│           读写锁的访问规则                               │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  读-读：并发  ✓  → 多个线程可以同时读取                  │
│  读-写：互斥  ✗  → 读操作必须等待写完成                  │
│  写-写：互斥  ✗  → 写操作必须独占                        │
│  写-读：互斥  ✗  → 写操作期间不允许读取                  │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

## ReadWriteLock 接口

```java
public interface ReadWriteLock {
    // 获取读锁
    Lock readLock();

    // 获取写锁
    Lock writeLock();
}
```

### 基本使用

```java
public class CacheDemo {
    private final Map<String, Object> cache = new HashMap<>();
    private final ReadWriteLock rwLock = new ReentrantReadWriteLock();

    // 读取：使用读锁
    public Object get(String key) {
        rwLock.readLock().lock();
        try {
            return cache.get(key);
        } finally {
            rwLock.readLock().unlock();
        }
    }

    // 写入：使用写锁
    public void put(String key, Object value) {
        rwLock.writeLock().lock();
        try {
            cache.put(key, value);
        } finally {
            rwLock.writeLock().unlock();
        }
    }

    // 清空：使用写锁
    public void clear() {
        rwLock.writeLock().lock();
        try {
            cache.clear();
        } finally {
            rwLock.writeLock().unlock();
        }
    }
}
```

## ReentrantReadWriteLock 实现

`ReentrantReadWriteLock` 是 `ReadWriteLock` 的标准实现，同时支持：

- **可重入**：持有锁的线程可以再次获取锁
- **公平/非公平**：支持两种模式
- **锁降级**：写锁可以降级为读锁

### 状态存储

`ReentrantReadWriteLock` 用一个 `Sync` 类存储锁状态：

```java
// 使用一个整数的高 16 位存储读锁计数
// 低 16 位存储写锁计数（或写锁重入次数）

┌─────────────────────────────────────────────────────────┐
│          32 位整数的状态分配                             │
├─────────────────────────────────────────────────────────┤
│                                                          │
│   32 位整数：0b SSSSSSSS SSSSSSSS RRRRRRRR RRRRRRRR    │
│                                                          │
│   高 16 位（S）：写锁状态                                 │
│   - 0：无人持有写锁                                      │
│   - > 0：写锁重入次数                                    │
│                                                          │
│   低 16 位（R）：读锁状态                                 │
│   - 0：无人持有读锁                                      │
│   - > 0：持有读锁的线程数量                              │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

### 写锁的获取

```java
// 写锁获取
protected final boolean tryAcquire(int acquires) {
    Thread current = Thread.currentThread();
    int c = getState();
    int w = exclusiveCount(c);  // 获取写锁计数

    if (c != 0) {
        // 已经有线程持有锁
        if (w == 0 || current != getExclusiveOwnerThread()) {
            // 有读锁存在，或写锁被其他线程持有
            return false;
        }
        // 写锁重入
        if (w + acquires > EXCLUSIVE_MASK) {
            throw new Error("Maximum lock count exceeded");
        }
    }

    // CAS 尝试获取写锁
    if (compareAndSetState(c, c + acquires)) {
        setExclusiveOwnerThread(current);
        return true;
    }
    return false;
}
```

### 读锁的获取

```java
// 读锁获取
protected final int tryAcquireShared(int unused) {
    for (;;) {
        int c = getState();
        int nextc = c + SHARED_UNIT;  // SHARED_UNIT = 1 << 16
        if (nextc < 0) {
            throw new Error("Maximum lock count exceeded");
        }
        if (compareAndSetState(c, nextc)) {
            // 增加读锁持有者计数
            if (firstReader == null) {
                firstReader = current;
                readHolds.set(current, 1);
            } else if (firstReader == current) {
                readHolds.get(current).value++;
            } else {
                // ...
            }
            return 1;
        }
    }
}
```

## 读写锁的特殊规则

### 锁的公平性

```java
// 非公平模式（默认）
ReadWriteLock rwLock = new ReentrantReadWriteLock();

// 公平模式
ReadWriteLock fairLock = new ReentrantReadWriteLock(true);
```

| 模式 | 规则 |
|---|---|
| 非公平 | 写锁优先获取，可能「饥饿」读者 |
| 公平 | 按等待时间顺序，可能「饥饿」写者 |

### 锁降级（Lock Demotion）

写锁可以降级为读锁，但读锁不能升级为写锁。

```java
public void demotionDemo() {
    rwLock.writeLock().lock();
    try {
        // 写入数据
        updateState();

        // 锁降级：写锁 -> 读锁
        rwLock.readLock().lock();
        try {
            // 此时仍持有写锁，但其他线程可以获取读锁
            readState();
        } finally {
            rwLock.readLock().unlock();  // 释放读锁
        }
        // 继续持有写锁
    } finally {
        rwLock.writeLock().unlock();  // 释放写锁
    }
}
```

**为什么读锁不能升级为写锁？**

```java
// 这样的操作可能导致死锁
rwLock.readLock().lock();
try {
    // 两个线程同时持有读锁
    // 线程 A 尝试升级为写锁，等待线程 B 释放读锁
    // 线程 B 也尝试升级为写锁，等待线程 A 释放读锁
    // → 死锁！
    rwLock.writeLock().lock();
} finally {
    rwLock.readLock().unlock();
}
```

## 读写锁的等待策略

### 写锁饥饿问题

在非公平模式下，读锁可能「饿死」写锁：

```
时间 ──────────────────────────────────────────────────────>

线程 A: [持有读锁]
线程 B: [读锁等待] [读锁等待] [读锁等待] [读锁等待]
线程 C: ──────────────────[持有写锁]─────────────────────

问题：读锁不断涌入，写锁长时间得不到执行
```

### 解决方案：降序策略

`ReentrantReadWriteLock` 支持 **writer preference**：

```java
// 写锁优先
ReentrantReadWriteLock rwLock = new ReentrantReadWriteLock(true);
```

## 实战：缓存实现

```java
public class ReadWriteCache<K, V> {
    private final Map<K, V> cache = new HashMap<>();
    private final ReadWriteLock rwLock = new ReentrantReadWriteLock();

    private final Lock readLock = rwLock.readLock();
    private final Lock writeLock = rwLock.writeLock();

    // 缓存查找（高频读取）
    public V get(K key) {
        readLock.lock();
        try {
            V value = cache.get(key);
            if (value == null) {
                // 缓存未命中，释放读锁，尝试获取写锁
                readLock.unlock();
                writeLock.lock();
                try {
                    // 双重检查
                    if (cache.get(key) == null) {
                        value = loadFromDatabase(key);  // 模拟从 DB 加载
                        cache.put(key, value);
                    } else {
                        value = cache.get(key);
                    }
                } finally {
                    // 锁降级
                    readLock.lock();
                    writeLock.unlock();
                }
            }
            return value;
        } finally {
            readLock.unlock();
        }
    }

    // 更新缓存（低频写入）
    public void put(K key, V value) {
        writeLock.lock();
        try {
            cache.put(key, value);
        } finally {
            writeLock.unlock();
        }
    }

    // 清空缓存
    public void clear() {
        writeLock.lock();
        try {
            cache.clear();
        } finally {
            writeLock.unlock();
        }
    }

    private V loadFromDatabase(K key) {
        // 模拟数据库查询
        return (V) ("value_for_" + key);
    }
}
```

## StampedLock：更高效的读写锁

JDK 8 引入了 `StampedLock`，它使用「戳」而不是引用计数，性能更好。

```java
public class StampedLockDemo {
    private final StampedLock stampedLock = new StampedLock();
    private double x, y;

    // 乐观读
    public double distanceFromOrigin() {
        long stamp = stampedLock.tryOptimisticRead();
        double currentX = x;
        double currentY = y;

        // 验证戳是否有效
        if (!stampedLock.validate(stamp)) {
            // 戳失效，需要获取读锁重读
            stamp = stampedLock.readLock();
            try {
                currentX = x;
                currentY = y;
            } finally {
                stampedLock.unlockRead(stamp);
            }
        }
        return Math.sqrt(currentX * currentX + currentY * currentY);
    }

    // 写入
    public void move(double deltaX, double deltaY) {
        long stamp = stampedLock.writeLock();
        try {
            x += deltaX;
            y += deltaY;
        } finally {
            stampedLock.unlockWrite(stamp);
        }
    }
}
```

`StampedLock` 的优势：

- **乐观读**：不需要真正加锁，只验证戳
- **无重入**：更轻量
- **支持锁升级**：读锁可以升级为写锁

## 总结

读写锁是读多写少场景的利器：

1. **读写分离**：读操作可以并发，写操作独占
2. **ReentrantReadWriteLock**：标准实现，支持重入和公平模式
3. **锁降级**：写锁可以降级为读锁
4. **StampedLock**：更高效的读写锁，支持乐观读

| 特性 | ReentrantReadWriteLock | StampedLock |
|---|---|---|
| 重入 | 支持 | 不支持 |
| 公平模式 | 支持 | 不支持 |
| 乐观读 | 不支持 | 支持 |
| 锁升级 | 不支持 | 支持 |

选择哪种读写锁，取决于你的具体场景。
