# JUC 并发工具类：Java 并发编程的瑞士军刀

Java 并发包（java.util.concurrent，简称 JUC）是 JDK 5 引入的核心扩展库，提供了大量用于并发编程的高性能工具。与 `synchronized` 相比，JUC 工具类提供了更灵活、更强大的同步能力。本章将带你全面了解这些工具类的设计理念和使用场景。

## JUC 工具类的分类

JUC 提供了多种并发工具，大致可以分为以下几类：

```
JUC 并发工具类
├── 同步器 (Synchronizers)
│   ├── ReentrantLock
│   ├── ReadWriteLock
│   ├── StampedLock
│   ├── Semaphore
│   ├── CountDownLatch
│   ├── CyclicBarrier
│   └── Condition
├── 并发集合
│   ├── ConcurrentHashMap
│   ├── CopyOnWriteArrayList
│   ├── BlockingQueue
│   └── ...
└── 原子类 (Atomic)
    ├── AtomicInteger
    ├── AtomicLong
    └── AtomicReference
```

## ReentrantLock：可重入的互斥锁

`ReentrantLock` 是 `synchronized` 的替代方案，提供了更灵活的锁操作：

```java
public class ReentrantLock implements Lock, Serializable {
    private final Sync sync;
    
    abstract static class Sync extends AbstractQueuedSynchronizer {
        // 核心实现
    }
    
    public static class NonfairSync extends Sync { /* 非公平锁 */ }
    public static class FairSync extends Sync { /* 公平锁 */ }
}
```

### 核心特性

**可重入**：同一个线程可以多次获取锁，每获取一次，内部计数器加一，每释放一次减一：

```java
ReentrantLock lock = new ReentrantLock();

lock.lock();
try {
    // 第一次获取锁
    lock.lock();  // 同一个线程再次获取
    try {
        // 临界区
    } finally {
        lock.unlock();  // 第二次释放
    }
} finally {
    lock.unlock();  // 第一次释放
}
```

**公平 vs 非公平**：构造函数可以指定是否为公平锁。公平锁按照等待顺序获取锁，而非公平锁允许"插队"——新获取锁的线程可能比等待队列中的线程更早获得锁：

```java
// 非公平锁（默认）- 性能更好
ReentrantLock unfairLock = new ReentrantLock();

// 公平锁 - 先到先得
ReentrantLock fairLock = new ReentrantLock(true);
```

### 高级特性

```java
// 尝试获取锁（非阻塞）
if (lock.tryLock()) {
    try {
        // 获取到锁
    } finally {
        lock.unlock();
    }
} else {
    // 获取失败，继续其他逻辑
}

// 尝试获取锁（超时）
if (lock.tryLock(5, TimeUnit.SECONDS)) {
    try {
        // 5秒内获取到锁
    } finally {
        lock.unlock();
    }
} else {
    // 等待超时
}

// 可中断获取
try {
    lock.lockInterruptibly();
    try {
        // 临界区
    } finally {
        lock.unlock();
    }
} catch (InterruptedException e) {
    // 线程被中断
}
```

## ReadWriteLock：读写分离的锁

`ReadWriteLock` 将锁分为读锁和写锁，允许多个线程同时读，但写操作需要独占锁：

```java
ReadWriteLock rwLock = new ReentrantReadWriteLock();
Lock readLock = rwLock.readLock();
Lock writeLock = rwLock.writeLock();

// 读操作 - 可以并发
readLock.lock();
try {
    String value = cache.get(key);
} finally {
    readLock.unlock();
}

// 写操作 - 独占
writeLock.lock();
try {
    cache.put(key, value);
} finally {
    writeLock.unlock();
}
```

读写锁适用于"读多写少"的场景：读操作可以并行执行，提高并发度；写操作虽然独占，但发生频率较低。

## StampedLock：乐观读的升级版

JDK 8 引入的 `StampedLock` 是读写锁的进一步优化。它引入了"戳"（Stamp）概念，支持乐观读模式：

```java
StampedLock sl = new StampedLock();

// 乐观读 - 不阻塞读操作
long stamp = sl.tryOptimisticRead();
V value = cache.get(key);
if (!sl.validate(stamp)) {
    // 有写操作发生，需要升级为悲观读
    stamp = sl.readLock();
    try {
        value = cache.get(key);
    } finally {
        sl.unlockRead(stamp);
    }
}

// 写操作
long stamp = sl.writeLock();
try {
    cache.put(key, value);
} finally {
    sl.unlockWrite(stamp);
}
```

乐观读不阻塞写操作，只在读取后验证是否有写操作发生。如果没有写操作，避免了获取读锁的开销；如果有写操作，则升级为悲观读。这在大多数读操作不会遇到写操作的场景下能显著提升性能。

## Semaphore：信号量控制并发数

`Semaphore` 控制同时访问某个资源的线程数量：

```java
Semaphore permits = new Semaphore(3);  // 3 个许可

// 获取许可
permits.acquire();
try {
    // 临界区 - 最多 3 个线程同时执行
} finally {
    permits.release();  // 释放许可
}

// 批量获取
permits.acquire(2);  // 获取 2 个许可

// 非阻塞获取
if (permits.tryAcquire()) {
    try {
        // 获取成功
    } finally {
        permits.release();
    }
}

// 带超时
if (permits.tryAcquire(5, TimeUnit.SECONDS)) {
    try {
        // 5 秒内获取成功
    } finally {
        permits.release();
    }
}
```

`Semaphore` 常用于限制对连接池的并发访问、实现资源池、限流等场景。

## CountDownLatch：倒计时门栓

`CountDownLatch` 让一个或多个线程等待，直到其他线程完成操作：

```java
CountDownLatch latch = new CountDownLatch(3);

// 工作线程
for (int i = 0; i < 3; i++) {
    new Thread(() -> {
        try {
            // 执行任务
            doWork();
        } finally {
            latch.countDown();  // 计数减一
        }
    }).start();
}

// 主线程等待所有任务完成
latch.await();  // 阻塞直到计数为 0
System.out.println("所有任务完成");
```

典型的应用场景包括：等待多个服务启动完成、等待多个计算任务完成、等待所有玩家加载完毕等。

## CyclicBarrier：循环栅栏

`CyclicBarrier` 让一组线程相互等待，直到所有线程都到达某个屏障点后一起继续执行。它与 `CountDownLatch` 的区别在于：可以重用，线程到达后可以再次等待：

```java
CyclicBarrier barrier = new CyclicBarrier(3, () -> {
    // 所有线程到达后执行（可选）
    System.out.println("所有玩家已就位，开始游戏");
});

for (int i = 0; i < 3; i++) {
    new Thread(() -> {
        loadResources();           // 加载资源
        barrier.await();           // 等待其他玩家
        startGame();              // 所有玩家都加载完毕，开始游戏
    }).start();
}
```

`CyclicBarrier` 适合用于多线程计算的最后一步汇总、各阶段结束后的同步等待等场景。

## Condition：条件变量

`Condition` 提供了比 `Object.wait/notify` 更灵活的条件等待机制：

```java
ReentrantLock lock = new ReentrantLock();
Condition notEmpty = lock.newCondition();
Condition notFull = lock.newCondition();

// 生产者
lock.lock();
try {
    while (isFull()) {
        notFull.await();  // 等待队列不满
    }
    add(item);
    notEmpty.signal();   // 通知消费者
} finally {
    lock.unlock();
}

// 消费者
lock.lock();
try {
    while (isEmpty()) {
        notEmpty.await();  // 等待队列不空
    }
    remove(item);
    notFull.signal();     // 通知生产者
} finally {
    lock.unlock();
}
```

一个 `ReentrantLock` 可以创建多个 `Condition`，比 `synchronized` 的单一 wait 集合更灵活。

## 如何选择合适的并发工具？

```
需要独占锁？
├── 简单互斥：synchronized
├── 需要高级特性：ReentrantLock
├── 读多写少：ReadWriteLock
└── 乐观读为主：StampedLock

需要限制并发数？
└── Semaphore

需要线程同步？
├── 一次性等待：CyclicBarrier
├── 倒计时等待：CountDownLatch
└── 条件等待：Condition

需要高性能并发集合？
└── ConcurrentHashMap / CopyOnWriteArrayList / BlockingQueue
```
