# volatile 有序性原理：禁止指令重排序

`volatile` 不仅保证了可见性，还保证了**有序性**——禁止指令重排序。为什么需要禁止重排序？因为编译器和 CPU 为了提升性能，会调整指令的执行顺序。但在多线程环境下，这种重排序可能导致灾难性的后果。今天我们就来深入剖析 `volatile` 如何通过内存屏障禁止指令重排序。

## 指令重排序的问题

### 什么是重排序？

重排序是编译器和 CPU 为了优化性能而调整指令执行顺序的现象。

```java
public class ReorderingProblem {
    
    int a = 0;
    int b = 0;
    
    public void method() {
        a = 1;      // 指令 1
        b = 2;      // 指令 2
    }
}
```

单线程下，无论指令 1 和 2 谁先执行，最终 a = 1, b = 2。但多线程下可能出现问题：

```java
public class MultiThreadReordering {
    
    int a = 0;
    int b = 0;
    
    // 线程 A
    public void writer() {
        a = 1;      // 指令 1
        b = 2;      // 指令 2
    }
    
    // 线程 B
    public void reader() {
        if (b == 2) {        // 读取 b
            int r = a;        // 读取 a
            // r 可能等于 0！即使 b == 2
        }
    }
}
```

如果编译器重排序为 `b = 2` 先于 `a = 1`：
1. 线程 A 执行 `b = 2`，刷新到主内存
2. 线程 B 读取 `b == 2`，然后读取 `a`，得到 `a = 0`

## volatile 如何禁止重排序？

### volatile 写操作的屏障

```java
public class VolatileWriteReordering {
    
    int a = 0;
    volatile int b = 0;
    
    public void writer() {
        a = 1;           // Store A
        // StoreStore 屏障 ← 禁止 A 与 B 重排序
        b = 2;           // Store B (volatile)
        // StoreLoad 屏障 ← 禁止 B 与后续读重排序
    }
}
```

### volatile 读操作的屏障

```java
public class VolatileReadReordering {
    
    volatile int a = 0;
    int b = 0;
    
    public void reader() {
        // LoadLoad 屏障 ← 禁止后续读与 A 重排序
        int r1 = a;           // Load A (volatile)
        // LoadStore 屏障 ← 禁止 B 与 A 重排序
        int r2 = b;           // Load B
    }
}
```

## 重排序规则表

JMM 定义了**happens-before 内存屏障插入规则**：

| 第一个操作 | 第二个操作 | 第二个操作 | 第二个操作 |
|-----------|-----------|-----------|-----------|
| **第一个操作** | 普通读/写 | volatile 读 | volatile 写 |
| 普通读/写 | | | **StoreLoad** |
| volatile 读 | **LoadLoad** | **LoadLoad** | **LoadStore** |
| volatile 写 | | **StoreStore** | **StoreStore** |

### 详细说明

```java
public class ReorderingRules {
    
    int a = 0;           // 普通变量
    volatile int b = 0;  // volatile 变量
    
    // 规则 1：volatile 写 vs 普通写 → StoreLoad
    public void rule1() {
        b = 1;           // volatile 写
        a = 2;           // 普通写
        // 需要 StoreLoad 屏障
        // 禁止 b = 1 与 a = 2 重排序
    }
    
    // 规则 2：volatile 写 vs volatile 写 → StoreStore
    public void rule2() {
        b = 1;           // volatile 写
        // StoreStore 屏障
        int x = a;       // 普通读（不是写）
        // 等等，这里是读，不是写
    }
    
    // 规则 3：volatile 写 vs volatile 写
    public void rule3() {
        b = 1;           // volatile 写
        // StoreStore 屏障
        int c = 2;       // volatile 写 ← 需要 StoreStore
    }
    
    // 规则 4：普通读 vs volatile 读 → LoadLoad
    public void rule4() {
        int r1 = a;      // 普通读
        // LoadLoad 屏障
        int r2 = b;      // volatile 读
        // 禁止 r1 与 r2 重排序
    }
    
    // 规则 5：volatile 读 vs 普通读 → LoadLoad
    public void rule5() {
        int r1 = b;      // volatile 读
        // LoadLoad 屏障
        int r2 = a;      // 普通读
    }
    
    // 规则 6：volatile 读 vs volatile 读 → LoadLoad
    public void rule6() {
        int r1 = b;      // volatile 读
        // LoadLoad 屏障
        int r2 = a;      // volatile 读
    }
    
    // 规则 7：volatile 读 vs 普通写 → LoadStore
    public void rule7() {
        int r1 = b;      // volatile 读
        // LoadStore 屏障
        a = 2;           // 普通写
        // 禁止 r1 与 a = 2 重排序
    }
    
    // 规则 8：volatile 读 vs volatile 写
    public void rule8() {
        int r1 = b;      // volatile 读
        // LoadStore 屏障
        a = 2;           // volatile 写 ← 需要 LoadStore
    }
}
```

## 完整示例：单例模式的指令重排序

### DCL 双重检查锁定的问题

```java
public class DCLSingleton {
    
    private static DCLSingleton instance;
    
    public static DCLSingleton getInstance() {
        if (instance == null) {  // 第一次检查
            synchronized (DCLSingleton.class) {
                if (instance == null) {  // 第二次检查
                    instance = new Singleton();  // 问题所在！
                }
            }
        }
        return instance;
    }
}
```

### `instance = new Singleton()` 的实际步骤

`new Singleton()` 在字节码层面分为三步：

```java
// 编译器/JIT 可能重排序这三条指令：
// 
// 1. 分配内存
// 2. 调用构造函数（初始化）
// 3. 写入引用（赋值给 instance）
//
// 可能被重排序为：
// 1. 分配内存
// 2. 写入引用（instance != null，但未初始化）
// 3. 调用构造函数
```

### 问题场景

```java
public class DCLProblem {
    
    private static DCLSingleton instance;
    
    // 线程 A
    public static void writer() {
        if (instance == null) {
            synchronized (lock) {
                if (instance == null) {
                    // 这三步可能被重排序：
                    // 1. memory = allocate()      // 分配内存
                    // 2. instance = memory        // 写入引用 ← 此时 instance != null
                    // 3. ctor = Singleton()        // 调用构造函数
                    instance = new DCLSingleton();
                }
            }
        }
    }
    
    // 线程 B
    public static void reader() {
        if (instance != null) {  // 读取到 instance != null
            instance.doSomething();  // 但对象未初始化！
        }
    }
}
```

### 线程 B 可能看到的情况


### 解决方案：volatile 禁止重排序

```java
public class DCLSingletonFixed {
    
    // 使用 volatile 禁止重排序
    private static volatile DCLSingleton instance;
    
    public static DCLSingleton getInstance() {
        if (instance == null) {
            synchronized (DCLSingleton.class) {
                if (instance == null) {
                    // volatile 的 StoreLoad 屏障确保：
                    // 1. 构造函数一定在写入引用之前完成
                    // 2. instance = new Singleton() 不会与其他操作重排序
                    instance = new DCLSingleton();
                }
            }
        }
        return instance;
    }
}
```

### volatile 如何保证顺序？

```java
public class VolatileOrderGuarantee {
    
    private static volatile DCLSingleton instance;
    
    public static void writer() {
        // 没有 volatile 时，可能重排序为：
        // temp = allocate()      // 1. 分配内存
        // instance = temp         // 2. 写入引用（instance != null）
        // temp.ctor()             // 3. 构造函数
        
        // 有 volatile 后，步骤 2 和 3 不能重排序：
        instance = new DCLSingleton();
        // 等价于：
        // temp = allocate()
        // StoreStore 屏障 ← 禁止重排序
        // temp.ctor()
        // StoreLoad 屏障  ← 禁止重排序
        // instance = temp
        
        // 任何读取 instance 的线程都会在步骤 3 完成后才读取
    }
}
```

## happens-before 规则的保证

```java
public class HappensBeforeGuarantee {
    
    volatile DCLSingleton instance;
    
    // 线程 A
    public void init() {
        DCLSingleton temp = new DCLSingleton();
        // 构造函数 Happens-Before 写入
        instance = temp;
    }
    
    // 线程 B
    public void use() {
        DCLSingleton local = instance;
        // 读取 Happens-Before 使用
        if (local != null) {
            local.doSomething();  // 一定已初始化
        }
    }
}
```

由于：
1. 构造函数 Happens-Before `instance = temp`（程序顺序）
2. `instance = temp` Happens-Before 线程 B 的 `local = instance`（volatile 规则）
3. 传递性：构造函数 Happens-Before 线程 B 的读取

## 总结

`volatile` 的有序性通过内存屏障实现：

1. **StoreStore 屏障**：volatile 写之前，之前的写操作不会与 volatile 写重排序
2. **StoreLoad 屏障**：volatile 写之后，后续的读/写不会与 volatile 写重排序
3. **LoadLoad 屏障**：volatile 读之后，后续的读操作不会与 volatile 读重排序
4. **LoadStore 屏障**：volatile 读之后，后续的写操作不会与 volatile 读重排序

这些屏障确保了：
- volatile 写之前的操作一定在 volatile 写之前执行
- volatile 读之后的操作一定在 volatile 读之后执行

这就是 `volatile` 能够解决 DCL 单例模式重排序问题的根本原因。
