# Java 内存模型：主内存与工作内存

在多核处理器时代，CPU 缓存和编译器的指令优化在带来性能提升的同时，也引入了一个微妙的问题：**可见性**。你修改了一个共享变量，其他线程可能看不到这个修改。`volatile` 关键字、`synchronized` 的内存语义，这些并发编程的基础概念，都源于 Java 内存模型（JMM）。理解 JMM，是理解 Java 并发的第一步。

## 为什么需要 JMM？

### CPU 缓存带来的问题

现代 CPU 为了提升性能，引入了多级缓存：

```
┌────────────────────────────────────────────────────────────────┐
│                         CPU 缓存结构                             │
│                                                                 │
│                    ┌──────────┐                                 │
│                    │   CPU    │                                 │
│                    └─────┬────┘                                 │
│                          │                                      │
│              ┌───────────┴───────────┐                         │
│              │                       │                          │
│         ┌────┴────┐             ┌────┴────┐                    │
│         │   L1    │             │   L2    │                     │
│         │ (核心独享)│            │ (核心独享)│                    │
│         └─────────┘             └─────────┘                    │
│                                                                 │
│                         ┌───────┴───────┐                       │
│                         │               │                       │
│                    ┌────┴────┐      ┌────┴────┐                │
│                    │   L3    │      │   L3    │                 │
│                    │ (共享)   │      │ (共享)   │                 │
│                    └─────────┘      └─────────┘                  │
│                                                                 │
│                         ┌───────────────┐                        │
│                         │     主内存    │                        │
│                         │   (RAM)      │                        │
│                         └───────────────┘                        │
└────────────────────────────────────────────────────────────────┘
```

当多个 CPU 核心同时访问同一块内存时，就会出现可见性问题：线程 A 在 CPU 核心 1 上修改了变量 X，但这个修改还在 CPU 核心 1 的缓存中，线程 B 在 CPU 核心 2 上读取 X，读到的还是旧值。

### 编译器的重排序

现代编译器会进行各种优化来提升性能，其中就包括**指令重排序**：

```java
// 原始代码
int a = 1;
int b = 2;
a = 3;

// 编译器可能优化为（性能更好）
int b = 2;
a = 3;
int a = 1;  // 赋值操作提前
```

这种重排序在单线程环境下没有问题，但在多线程环境下可能导致程序行为不符合预期。

### JMM 的设计目标

JMM 的设计目标：
1. **定义可见性的语义**：规定什么样的修改对其他线程可见
2. **禁止特定重排序**：保证程序在多线程下的正确性
3. **提供同步原语**：提供 `volatile`、`synchronized`、`final` 等关键字

## JMM 的核心概念

### 主内存与工作内存

JMM 将内存划分为主内存和工作内存：

```
┌────────────────────────────────────────────────────────────────┐
│                      JMM 内存模型                                │
│                                                                 │
│   主内存（Main Memory）                                          │
│   ┌────────────────────────────────────────────────────────┐   │
│   │                                                         │   │
│   │     ┌─────┐   ┌─────┐   ┌─────┐   ┌─────┐             │   │
│   │     │  X  │   │  Y  │   │  Z  │   │ ... │   共享变量   │   │
│   │     └─────┘   └─────┘   └─────┘   └─────┘             │   │
│   │                                                         │   │
│   └────────────────────────────────────────────────────────┘   │
│                         ▲                                       │
│                         │                                       │
│            ┌────────────┴────────────┐                         │
│            │                         │                          │
│            ▼                         ▼                          │
│   ┌─────────────────┐     ┌─────────────────┐                  │
│   │   工作内存 1     │     │   工作内存 2     │                  │
│   │  (CPU 核心 1)   │     │  (CPU 核心 2)   │                  │
│   │                 │     │                 │                  │
│   │   线程A的工作空间 │     │   线程B的工作空间 │                  │
│   └─────────────────┘     └─────────────────┘                  │
│                                                                 │
│   线程A ──▶ 工作内存1 ──▶ 主内存                                │
│   线程B ──▶ 工作内存2 ──▶ 主内存                                │
│                                                                 │
└────────────────────────────────────────────────────────────────┘
```

**主内存**：所有线程共享的内存区域，对应于 Java 堆中的对象实例

**工作内存**：每个线程独有的内存区域，存储该线程读/写的共享变量的副本

线程对变量的读写都在自己的工作内存中进行，然后同步到主内存。

### 八大原子操作

JMM 定义了八种主内存与工作内存之间的原子操作：

| 操作 | 说明 |
|------|------|
| **lock** | 作用于主内存，将变量标识为线程独占 |
| **unlock** | 作用于主内存，释放线程独占的变量 |
| **read** | 作用于主内存，将变量值传输到工作内存 |
| **load** | 作用于工作内存，将 read 的值放入变量副本 |
| **use** | 作用于工作内存，将变量值传递给执行引擎 |
| **assign** | 作用于工作内存，将执行引擎的值赋给变量 |
| **store** | 作用于工作内存，将变量值传输到主内存 |
| **write** | 作用于主内存，将 store 的值放入主内存变量 |

### 原子操作的规则

```java
// read -> load 必须成对出现
// store -> write 必须成对出现
// assign 修改后必须同步回主内存
// use 前必须 load
```

## Happens-Before 原则

JMM 使用 **Happens-Before** 原则来定义**可见性**。如果操作 A Happens-Before 操作 B，那么 A 的结果对 B 可见。

### 常见的 Happens-Before 规则

```java
public class HappensBeforeDemo {
    
    // 1. 程序顺序规则
    // 同一线程中，前面的操作 Happens-Before 后面的操作
    int a = 1;       // Happens-Before
    int b = a + 1;   // 这个操作
    // 即 a = 1 对 b = a + 1 可见
    
    // 2. 监视器锁规则
    // 解锁 Happens-Before 后续对这个锁的加锁
    synchronized (lock) {
        a = 2;  // Happens-Before
    }          // 这个解锁
    
    synchronized (lock) {
        System.out.println(a);  // 读取到 2
    }
    
    // 3. volatile 规则
    // volatile 写 Happens-Before volatile 读
    volatile boolean ready = false;  // Happens-Before
    
    writer() {
        data = 42;    // Happens-Before
        ready = true;
    }
    
    reader() {
        if (ready) {        // 这个读取
            System.out.println(data);  // 能看到 data = 42
        }
    }
    
    // 4. 线程启动规则
    // Thread.start() Happens-Before 被启动线程中的任何操作
    Thread t = new Thread(() -> {
        // 可以看到主线程中设置的值
    });
    sharedVar = 100;      // Happens-Before
    t.start();            // 这个操作
    
    // 5. 线程终止规则
    // 线程中所有操作 Happens-Before 其他线程检测到该线程终止
    sharedVar = 200;
    t.join();            // 等待线程结束
    System.out.println(sharedVar);  // 看到 200
    
    // 6. 传递性
    // A Happens-Before B，B Happens-Before C，则 A Happens-Before C
}
```

## 内存屏障

内存屏障是 JVM 层面的指令，用于禁止特定类型的指令重排序：

| 屏障类型 | 说明 |
|---------|------|
| **LoadLoad** | 禁止 Load1 和 Load2 重排序 |
| **StoreStore** | 禁止 Store1 和 Store2 重排序 |
| **LoadStore** | 禁止 Load 和 Store 重排序 |
| **StoreLoad** | 禁止 Store 和 Load 重排序 |

### 内存屏障与 volatile

```java
// volatile 写的内存屏障
public void volatileWrite() {
    // StoreStore 屏障：禁止之前的写与 volatile 写重排序
    ready = true;  // volatile write
    // StoreLoad 屏障：禁止 volatile 写与后续的读重排序
}

// volatile 读的内存屏障
public void volatileRead() {
    // LoadLoad 屏障：禁止后续的读与 volatile 读重排序
    boolean r = ready;  // volatile read
    // LoadStore 屏障：禁止后续的写与 volatile 读重排序
}
```

## volatile 的内存语义

### volatile 写的语义

```java
public class VolatileWrite {
    
    volatile int x = 0;
    
    public void write() {
        x = 10;  // volatile write
        
        // 等价于：
        // 1. StoreStore 屏障
        // 2. 将 x=10 写入主内存
        // 3. StoreLoad 屏障
    }
}
```

### volatile 读的语义

```java
public class VolatileRead {
    
    volatile int x = 0;
    
    public int read() {
        int r = x;  // volatile read
        
        // 等价于：
        // 1. LoadLoad 屏障
        // 2. LoadStore 屏障
        // 3. 从主内存读取 x
    }
}
```

### 完整示例

```java
public class VolatileComplete {
    
    volatile int a = 0;
    volatile int b = 0;
    
    // 线程 A
    public void writer() {
        a = 1;           // 写 a
        // StoreStore 屏障
        b = 2;           // 写 b（volatile）
        // StoreLoad 屏障
    }
    
    // 线程 B
    public void reader() {
        // LoadLoad 屏障
        int r2 = b;      // 读 b（volatile）
        // LoadStore 屏障
        int r1 = a;      // 读 a
        
        // 结果：
        // 如果 r2 = 2，则 r1 一定 = 1
        // 因为 b=2 Happens-Before r2
        // b=2 Happens-Before a=1（程序顺序）
        // a=1 Happens-Before r1
        // 传递性：b=2 Happens-Before r1
    }
}
```

## 先行发生原则的应用

### 先行发生不等于时间先后

```java
public class HappensBeforeMisconception {
    
    int x = 0;
    int y = 0;
    
    // 线程 A
    public void A() {
        x = 1;      // 时间上先发生
        y = 2;
    }
    
    // 线程 B
    public void B() {
        // 线程 B 可能看到 y=2，但 x=0
        // 因为没有 Happens-Before 关系
        // 编译器/CPU 可能重排序
    }
}
```

### 正确的同步

```java
public class CorrectSync {
    
    volatile int x = 0;
    volatile int y = 0;
    
    // 线程 A
    public void A() {
        x = 1;
        y = 2;  // volatile
    }
    
    // 线程 B
    public void B() {
        // y=2 Happens-Before r2
        // x=1 Happens-Before y=2
        // 所以 x=1 Happens-Before r2
        int r2 = y;  // volatile
        int r1 = x;
        
        // 如果 r2 = 2，则 r1 = 1
    }
}
```

## JMM 与线程安全

### 线程安全与 JMM

线程安全的三种特性：
1. **原子性**：操作不可分割
2. **可见性**：一个线程的修改对其他线程可见
3. **有序性**：指令执行顺序符合预期

```java
public class ThreadSafetyAndJMM {
    
    // 原子性
    private int counter = 0;  // 非原子性
    private AtomicInteger atomicCounter = new AtomicInteger(0);  // 原子性
    
    // 可见性
    private volatile boolean ready = false;  // 可见
    
    // 有序性
    private volatile int a = 0;
    private volatile int b = 0;
    
    // 有序性通过 volatile 的内存屏障保证
}
```

## 总结

JMM 的核心概念：

1. **主内存与工作内存**：线程在各自工作内存中操作变量副本
2. **八种原子操作**：read、load、use、assign、store、write、lock、unlock
3. **Happens-Before 规则**：定义可见性的边界
4. **内存屏障**：禁止特定类型的重排序

理解 JMM 能帮助我们：
- 正确使用 `volatile`、`synchronized`
- 理解并发问题的根源
- 编写正确的多线程代码
