# volatile 能保证原子性吗？

volatile 是 Java 并发编程中最常用的关键字之一，但很多人对它的理解存在误区。最常见的问题就是：**volatile 能保证原子性吗？** 答案是：**不能**。volatile 只保证**可见性**和**有序性**，不保证**原子性**。今天我们就来详细分析 volatile 与原子性的关系。

## 原子性的概念

### 什么是原子性？

原子性是指一个操作是不可分割的，要么全部执行，要么全部不执行。

```java
public class AtomicityConcept {
    
    // 原子操作：不可分割
    int a = 1;           // 原子
    boolean flag = true; // 原子
    Object obj = new Object();  // 原子（引用赋值）
    
    // 非原子操作：可分割
    int b = a + 1;       // 读取 + 加法 + 赋值
    a++;                  // 读取 + 加1 + 赋值
}
```

### 常见操作的原子性

| 操作 | 是否原子 | 原因 |
|------|---------|------|
| `int a = 1;` | ✓ | 基本类型赋值 |
| `boolean flag = true;` | ✓ | boolean 赋值 |
| `Object o = new Object();` | ✓ | 引用赋值（64位 JVM 中） |
| `int i = a;` | ✓ | 读取基本类型 |
| `a = 2;` | ✓ | 写入基本类型 |
| `a++;` | ✗ | 读-改-写 |
| `i = a++;` | ✗ | 读-改-写 |
| `a += b;` | ✗ | 读-改-写 |

## volatile 不能保证原子性

### volatile ++ 的问题

```java
public class VolatileAtomicityProblem {
    
    volatile int counter = 0;
    
    public void increment() {
        counter++;  // 不是原子操作！
    }
    
    // counter++ 实际包含三步：
    // 1. 读取：temp = counter
    // 2. 修改：temp = temp + 1
    // 3. 写入：counter = temp
}
```

### 多线程下的 race condition

```java
public class RaceConditionDemo {
    
    volatile int counter = 0;
    
    public static void main(String[] args) throws InterruptedException {
        RaceConditionDemo demo = new RaceConditionDemo();
        
        // 创建 1000 个线程，每个线程增加 counter 1000 次
        Thread[] threads = new Thread[1000];
        for (int i = 0; i < 1000; i++) {
            threads[i] = new Thread(demo::increment);
        }
        
        for (Thread t : threads) {
            t.start();
        }
        
        for (Thread t : threads) {
            t.join();
        }
        
        // 期望：1,000,000
        // 实际：往往小于 1,000,000（volatile 不能保证原子性）
        System.out.println("Counter = " + demo.counter);
    }
    
    public void increment() {
        for (int i = 0; i < 1000; i++) {
            counter++;  // 非原子操作
        }
    }
}
```

### 问题分析

```
┌────────────────────────────────────────────────────────────────┐
│                   counter++ 的竞态条件                            │
│                                                                 │
│   线程 A                 线程 B                                  │
│   ┌──────────┐           ┌──────────┐                          │
│   │ temp=0   │ ◄─────────│ 读取 counter=0│                       │
│   │ temp=1   │           │            │                         │
│   │ counter=1│ ─────────►│ (读取的还是 0)│                       │
│   └──────────┘           │ temp=1   │                          │
│                           │ counter=1│ ─────────► 覆盖！         │
│                           └──────────┘                          │
│                                                                 │
│   期望：counter = 2                                               │
│   实际：counter = 1                                               │
│                                                                 │
└────────────────────────────────────────────────────────────────┘
```

虽然 `volatile` 确保了 `counter` 的修改对其他线程可见，但由于 `counter++` 本身不是原子操作，仍然会产生丢失更新。

## volatile 的正确使用场景

### 场景1：标志位

```java
public class VolatileFlagUsage {
    
    volatile boolean running = true;
    
    public void start() {
        running = true;
    }
    
    public void stop() {
        running = false;  // volatile 保证其他线程立即看到
    }
    
    public void run() {
        while (running) {  // volatile 保证读取最新值
            // 业务逻辑
        }
    }
}
```

这个场景是安全的，因为**只涉及一次读写操作**。

### 场景2：一次性初始化

```java
public class VolatileInitUsage {
    
    volatile boolean initialized = false;
    volatile Object data;
    
    public void init() {
        data = loadData();
        initialized = true;  // 写入标记
    }
    
    public void use() {
        if (initialized) {  // 读取标记
            // 此时 data 一定已初始化
            doSomething(data);
        }
    }
}
```

### 场景3：独立的状态变量

```java
public class IndependentState {
    
    volatile int temperature;     // 温度（独立读取）
    volatile int humidity;         // 湿度（独立读取）
    volatile boolean sensorActive;  // 传感器状态
    
    public void updateTemp(int temp) {
        temperature = temp;  // 原子写入
    }
    
    public int readTemp() {
        return temperature;  // 原子读取
    }
}
```

## 需要原子性的场景

### 场景1：计数器

```java
public class AtomicCounter {
    
    // 错误：volatile 不能保证原子性
    volatile int counter = 0;
    
    // 正确：使用 AtomicInteger
    private final AtomicInteger atomicCounter = new AtomicInteger(0);
    
    public void increment() {
        atomicCounter.incrementAndGet();  // 原子操作
    }
    
    public int get() {
        return atomicCounter.get();
    }
}
```

### 场景2：check-then-act

```java
public class CheckThenAct {
    
    volatile User user = null;
    
    // 错误：非原子操作
    public void wrongInit() {
        if (user == null) {      // 检查
            user = new User();   // 初始化
        }
    }
    
    // 正确：使用 synchronized
    private final Object lock = new Object();
    
    public void correctInit() {
        synchronized (lock) {
            if (user == null) {
                user = new User();
            }
        }
    }
    
    // 或者：使用 AtomicReference
    private final AtomicReference<User> userRef = new AtomicReference<>();
    
    public void atomicInit() {
        userRef.compareAndSet(null, new User());
    }
}
```

### 场景3：复合操作

```java
public class CompoundOperation {
    
    volatile int value = 0;
    
    // 错误：复合操作非原子
    public void wrongCompound() {
        value += 10;  // value = value + 10
    }
    
    // 正确：使用同步
    private final ReentrantLock lock = new ReentrantLock();
    
    public void correctCompound() {
        lock.lock();
        try {
            value += 10;
        } finally {
            lock.unlock();
        }
    }
}
```

## volatile vs synchronized vs Atomic

```java
public class ComparisonTable {
    
    // 1. volatile：只保证可见性和有序性
    volatile int volatileValue = 0;
    
    // 2. synchronized：保证原子性、可见性和有序性
    synchronized void syncMethod() {
        // ...
    }
    
    // 3. AtomicXXX：保证原子性和可见性
    AtomicInteger atomicValue = new AtomicInteger(0);
}
```

### 选择指南

| 场景 | 推荐方案 |
|------|---------|
| 单一读写操作，boolean/int/long/引用 | volatile |
| 计数器（++/--） | AtomicInteger/AtomicLong |
| check-then-act | synchronized 或 CAS |
| 复合操作 | synchronized |
| 需要公平性 | ReentrantLock |

## volatile 的进阶特性：64位变量的拆分写入

### 64位 Java 的特殊问题

在 32位 JVM 中，64位的 long 和 double 的读写不是原子的：

```java
public class LongSplitWrite {
    
    // 32位 JVM：写入可能只写入低32位
    // 其他线程可能读到"半写入"的值
    volatile long value;
    
    // 64位 JVM：volatile long 的读写是原子的
    // 但不保证 64位变量的可见性（拆分写入问题）
}
```

### 解决方案

```java
public class LongVisibilityProblem {
    
    // 问题：long 是 64 位，可能被拆分写入
    volatile long value = 0;
    
    // 解决方案1：volatile double（64位 JVM 中原子）
    volatile double dValue = 0.0;
    
    // 解决方案2：使用 AtomicLong
    AtomicLong atomicValue = new AtomicLong();
}
```

## 总结

`volatile` 的能力边界：

- ✓ **可见性**：保证一个线程的修改对其他线程立即可见
- ✓ **有序性**：禁止指令重排序
- ✗ **原子性**：不保证复合操作的原子性

什么时候用 `volatile`：
- 只有一个线程写，多个线程读的场景
- 状态标志、开关
- 独立变量的读写

什么时候不用 `volatile`：
- 计数器、累加器（用 AtomicInteger）
- check-then-act 模式（用 synchronized）
- 复合操作（用 synchronized 或 Atomic）
