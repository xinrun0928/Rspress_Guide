# DCL 单例模式与 volatile

单例模式是 Java 设计模式中最常用的一种，但它的实现却暗藏玄机。`DCL`（Double-Checked Locking）双重检查锁定是实现延迟初始化单例的经典方式，但直接使用它却会导致严重的问题。`volatile` 关键字正是解决这个问题的关键。理解这个问题，是理解 Java 内存模型（JMM）和并发编程的绝佳案例。

## 单例模式的多种实现

### 饿汉式（立即加载）

```java
public class HungrySingleton {
    
    // 类加载时就创建实例，没有延迟加载的优势
    private static final HungrySingleton INSTANCE = new HungrySingleton();
    
    private HungrySingleton() {}
    
    public static HungrySingleton getInstance() {
        return INSTANCE;
    }
}
```

### 懒汉式（延迟加载）

```java
public class LazySingleton {
    
    private static LazySingleton instance;
    
    private LazySingleton() {}
    
    // 每次调用都需要获取锁，性能差
    public static synchronized LazySingleton getInstance() {
        if (instance == null) {
            instance = new LazySingleton();
        }
        return instance;
    }
}
```

### DCL 双重检查锁定

```java
public class DCLSingleton {
    
    private static DCLSingleton instance;
    
    private DCLSingleton() {}
    
    public static DCLSingleton getInstance() {
        if (instance == null) {           // 第一次检查
            synchronized (DCLSingleton.class) {
                if (instance == null) {   // 第二次检查
                    instance = new DCLSingleton();
                }
            }
        }
        return instance;
    }
}
```

DCL 看似完美：
- 第一次检查：避免不必要的同步
- 第二次检查：避免重复创建实例

但实际上它有**致命的问题**。

## DCL 的问题：重排序导致的灾难

### `instance = new DCLSingleton()` 的真相

`new` 操作在 JVM 层面并非原子操作，它包含三个步骤：

```java
// 实际的字节码步骤：
memory = allocate();      // 1. 分配内存
instance = memory;        // 2. 将引用赋值给 instance（此时 instance != null）
Singleton.<init>();       // 3. 调用构造函数（初始化）
```

### 问题：指令重排序

编译器和 CPU 可能进行指令重排序，将其变为：

```java
memory = allocate();      // 1. 分配内存
Singleton.<init>();       // 2. 调用构造函数
instance = memory;        // 3. 将引用赋值给 instance
```

### 多线程下的灾难场景

```java
public class DCLDisaster {
    
    private static DCLSingleton instance;
    
    // 线程 A：获取单例
    public static DCLSingleton getInstance() {
        if (instance == null) {           // 第一次检查
            synchronized (DCLSingleton.class) {
                if (instance == null) {
                    // 发生重排序：
                    // 1. memory = allocate()
                    // 2. instance = memory   ← instance != null！
                    // 3. Singleton.<init>()
                    instance = new DCLSingleton();
                }
            }
        }
        return instance;
    }
}
```

```
└────────────────────────────────────────────────────────────────┘
```

### 灾难后果

线程 B 可能拿到一个**未完成初始化**的对象：
- 对象的字段是默认值（0, null, false）
- 调用对象方法可能导致 NPE 或其他异常
- 程序行为完全不可预测

## DCL 问题的根源

### happens-before 规则的缺失

没有 volatile 时：
```
线程 A：
  构造函数 Happens-Before instance = memory
  
线程 B：
  instance = memory Happens-Before 使用 instance
  
没有 happens-before 关系，线程 B 可能看到错误的初始化顺序
```

## 解决方案：volatile

### 正确实现

```java
public class DCLSingletonFixed {
    
    // 使用 volatile
    private static volatile DCLSingleton instance;
    
    private DCLSingleton() {}
    
    public static DCLSingleton getInstance() {
        if (instance == null) {
            synchronized (DCLSingleton.class) {
                if (instance == null) {
                    instance = new DCLSingleton();
                }
            }
        }
        return instance;
    }
}
```

### volatile 如何禁止重排序？

```java
public class VolatileOrderGuarantee {
    
    // instance = new DCLSingleton() 变成：
    
    // 1. memory = allocate()
    // 2. Singleton.<init>()
    // 3. instance = memory
    
    // volatile 插入屏障：
    // 1. memory = allocate()
    // 2. Singleton.<init>()
    //    ↓ StoreStore 屏障
    //    ↓ 禁止步骤 2 和 3 重排序
    // 3. instance = memory
    //    ↓ StoreLoad 屏障
    //    ↓ 禁止步骤 3 与后续读重排序
}
```

### happens-before 关系的建立

```java
public class HappensBeforeGuarantee {
    
    private static volatile DCLSingleton instance;
    
    // 线程 A
    public static void writer() {
        instance = new DCLSingleton();
        // 等价于：
        // temp = allocate()
        // temp.ctor()
        // StoreStore 屏障
        // instance = temp
        // StoreLoad 屏障
    }
    
    // 线程 B
    public static void reader() {
        DCLSingleton local = instance;
        // 等价于：
        // LoadLoad 屏障
        // LoadStore 屏障
        // local = instance
        
        if (local != null) {
            local.doSomething();  // local 一定已完全初始化
        }
    }
}
```

**保证**：
- 构造函数 Happens-Before `instance = temp`（volatile write）
- `instance = temp` Happens-Before `local = instance`（volatile read）
- 传递性：构造函数 Happens-Before `local.doSomething()`

## 完整分析：DCL 为什么要两次检查？

### 第一次检查

```java
public class FirstCheck {
    
    // 第一次检查的目的：
    // 避免每次调用都获取锁，提升性能
    
    public static DCLSingleton getInstance() {
        if (instance == null) {  // 第一次检查
            // 如果 instance != null，直接返回
            // 避免进入 synchronized 块
            // 99.9% 的调用走这里
        }
    }
}
```

### 第二次检查

```java
public class SecondCheck {
    
    // 第二次检查的目的：
    // 避免多个线程同时通过第一次检查后重复创建实例
    
    public static DCLSingleton getInstance() {
        if (instance == null) {           // 第一次检查
            synchronized (DCLSingleton.class) {
                if (instance == null) {   // 第二次检查
                    // 只有第一个获取锁的线程创建实例
                    // 后续线程通过第二次检查，不会重复创建
                    instance = new DCLSingleton();
                }
            }
        }
        return instance;
    }
}
```

### 为什么第一次检查不能用 synchronized？

```java
public class WhyNotSyncFirst {
    
    // 如果只在 synchronized 块内检查：
    public static synchronized DCLSingleton getInstance() {
        if (instance == null) {  // 每次调用都要获取锁！
            instance = new DCLSingleton();
        }
        return instance;
    }
    
    // 问题：每次调用都要获取 synchronized 锁
    // 性能极差，失去了 DCL 的意义
}
```

## 替代方案

### 方案1：静态内部类（推荐）

```java
public class StaticInnerClassSingleton {
    
    private StaticInnerClassSingleton() {}
    
    // 静态内部类只有在被引用时才会加载
    private static class Holder {
        private static final StaticInnerClassSingleton INSTANCE = 
            new StaticInnerClassSingleton();
    }
    
    public static StaticInnerClassSingleton getInstance() {
        return Holder.INSTANCE;
    }
}
```

**优点**：延迟加载 + 线程安全 + 简洁

### 方案2：枚举（最简洁）

```java
public enum EnumSingleton {
    
    INSTANCE;
    
    public void doSomething() {
        // ...
    }
}

// 使用
EnumSingleton instance = EnumSingleton.INSTANCE;
```

**优点**：线程安全 + 防反射 + 防反序列化

## 完整示例：各种单例实现对比

```java
public class SingletonComparison {
    
    // 方案1：饿汉式（立即加载）
    static class Hungry {
        private static final Hungry INSTANCE = new Hungry();
        private Hungry() {}
        public static Hungry getInstance() { return INSTANCE; }
    }
    
    // 方案2：懒汉式（同步方法）
    static class LazySync {
        private static LazySync instance;
        private LazySync() {}
        public static synchronized LazySync getInstance() {
            if (instance == null) instance = new LazySync();
            return instance;
        }
    }
    
    // 方案3：DCL（有问题）
    static class DCLBroken {
        private static DCLBroken instance;
        private DCLBroken() {}
        public static DCLBroken getInstance() {
            if (instance == null) {
                synchronized (DCLBroken.class) {
                    if (instance == null) {
                        instance = new DCLBroken();
                    }
                }
            }
            return instance;
        }
    }
    
    // 方案4：DCL + volatile（正确）
    static class DCLFixed {
        private static volatile DCLFixed instance;
        private DCLFixed() {}
        public static DCLFixed getInstance() {
            if (instance == null) {
                synchronized (DCLFixed.class) {
                    if (instance == null) {
                        instance = new DCLFixed();
                    }
                }
            }
            return instance;
        }
    }
    
    // 方案5：静态内部类（推荐）
    static class StaticInner {
        private StaticInner() {}
        private static class Holder {
            private static final StaticInner INSTANCE = new StaticInner();
        }
        public static StaticInner getInstance() { return Holder.INSTANCE; }
    }
    
    // 方案6：枚举（最佳）
    enum EnumSingleton {
        INSTANCE;
    }
}
```

## 总结

DCL 单例模式的 volatile 问题：

1. **问题根源**：`instance = new Singleton()` 可能被重排序
2. **灾难后果**：其他线程可能拿到未初始化的对象
3. **解决方案**：使用 `volatile` 禁止重排序
4. **最佳实践**：静态内部类或枚举

理解这个问题，需要掌握：
- JVM 指令重排序
- Java 内存模型（JMM）
- volatile 的内存屏障
- happens-before 规则
