# Java 内存模型（JMM）概述

在 Java 的并发编程中，有一个概念虽然抽象，却至关重要——Java 内存模型（Java Memory Model，简称 JMM）。如果说线程是 Java 并发的基础，那么 JMM 就是理解线程安全、volatile、synchronized 等并发工具的理论基石。很多看似奇怪的并发问题，比如可见性问题、指令重排序问题，根源都在 JMM。

## 为什么需要 JMM

早期的 Java 试图让程序员不必关心内存访问的细节。"一次编写，到处运行"是 Java 的口号，JVM 试图屏蔽底层硬件和操作系统的差异。然而，随着多核处理器的普及和并发编程的兴起，这个抽象变得不够用了。

考虑一个看似简单的问题：为什么 `volatile` 能保证可见性？为什么 `synchronized` 既保证原子性又保证可见性？为什么即使在单核处理器上运行良好的代码，在多核处理器上却可能出现 bug？

这些问题无法用"Java 是跨平台的"来解释。JMM 就是为了回答这些问题而设计的。它定义了 Java 程序中各种变量（线程共享变量）的访问规则，以及在并发环境下这些规则如何保证可见性、有序性和原子性。

## JMM 与硬件内存模型

要理解 JMM，首先要理解现代多核处理器的内存架构。

每个 CPU 核心都有自己的寄存器和缓存（L1、L2 缓存），所有核心共享主内存。当 CPU 核心需要访问数据时，它首先在本地缓存中查找，如果没有找到，再去主内存获取。

这种架构带来了一个有趣的问题：如果两个核心同时修改同一份数据，会发生什么？CPU 缓存一致性协议（MESI）会保证每个核心看到的缓存行状态是一致的。但这种一致性是有代价的——它需要消息传递和等待，耗时可能比直接访问主内存还要长。

JMM 并不是简单地映射到硬件内存结构。它是 Java 虚拟机的规范，定义了一套抽象的规则，让程序员可以不用关心底层硬件的复杂性。

## JMM 的核心概念

JMM 的核心可以概括为以下几点：

**主内存与工作内存**：JMM 把内存划分为主内存和工作内存。主内存是所有线程共享的，类似于物理内存；每个线程有自己的工作内存，保存了主内存中变量的副本，类似于 CPU 缓存。

**内存间交互**：线程不能直接操作主内存，必须通过工作内存中转。变量从主内存到工作内存需要经过"read"、"load"；从工作内存到主内存需要经过"assign"、"write"。

** Happens-Before 规则**：JMM 定义了一套 Happens-Before 规则，规定了对同一变量的操作，哪些必须按顺序执行，哪些可以重排序。如果一个操作 Happens-Before 另一个操作，那么第一个操作的结果对第二个操作可见。
