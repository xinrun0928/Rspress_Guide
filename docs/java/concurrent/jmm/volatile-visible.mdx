# volatile 可见性原理：内存屏障

执行下面的代码，你觉得会输出什么？

```java
public class VisibilityDemo {
    private static boolean ready = false;
    private static int number = 0;

    public static void main(String[] args) throws InterruptedException {
        Thread reader = new Thread(() -> {
            while (!ready) {
                // 等待 ready 变为 true
            }
            System.out.println(number);
        });

        Thread writer = new Thread(() -> {
            number = 42;
            ready = true;
        });

        writer.start();
        reader.start();

        reader.join();
        System.out.println("程序结束");
    }
}
```

大多数人会说：输出 `42`，然后程序结束。

**但实际上，这个程序可能会永久运行下去，或者输出 `0`！**

这就是 **volatile** 要解决的问题。

## 可见性问题：为什么需要 volatile？

### 缓存一致性

现代 CPU 有多级缓存架构：

```
CPU 0 ──────> L1 Cache ──────> L2 Cache ──────> L3 Cache ──────> 主内存
CPU 1 ──────> L1 Cache ──────> L2 Cache ──────> L3 Cache ──────> 主内存
```

每个 CPU 核心都有自己的 L1、L2 缓存，共享 L3 缓存。

当 writer 线程修改 `number = 42` 和 `ready = true` 时：
- 修改被写入 CPU 0 的 L1 缓存
- L3 缓存和主内存中的值可能还是旧的
- CPU 1 的 L1 缓存中看到的 `number` 仍然是 `0`，`ready` 仍然是 `false`

### JMM 的 happens-before 规则

Java 内存模型（JMM）定义了 8 条 happens-before 规则，其中有一条：

> **volatile 变量规则**：对 volatile 变量的写操作 happens-before 后续对该变量的读操作。

```java
// 线程 A 写入
writer.start();
number = 42;      // 操作 1
ready = true;     // 操作 2（volatile 写）

// 线程 B 读取
while (!ready) {  // 操作 3（volatile 读）
    // spin
}
System.out.println(number);  // 操作 4
```

由于 `ready` 是 volatile，根据规则：
- 操作 2 happens-before 操作 3
- 操作 3 happens-before 操作 4

传递得到：操作 1 happens-before 操作 4，所以线程 B 一定能读取到 `number = 42`。

## volatile 的底层实现：内存屏障

### 什么是内存屏障？

内存屏障（Memory Barrier）是一条 CPU 指令，用于：
1. **禁止编译器和 CPU 重排序**
2. **强制刷新/读取缓存**

```
Store Barrier (写屏障)
  │
  ├── 确保屏障前的所有 Store 操作刷新到主内存
  └── 发送信号使缓存失效

Load Barrier (读屏障)
  │
  ├── 确保屏障后的所有 Load 操作从主内存读取
  └── 使 CPU 缓存失效，强制重新读取
```

### volatile 写操作

当执行 volatile 写时，JIT 编译器会插入一条 **Store Barrier**：

```java
// 源代码
volatile boolean ready = false;
ready = true;  // volatile 写

// 伪汇编（x86）
mov    $0x1,%eax        // 将 1 加载到寄存器
mov    %eax,0x10(%esp)  // Store 到内存
lock addl $0x0,(%esp)   // 内存屏障：刷新缓存
```

`lock addl` 指令的作用：
- 将写缓冲区的内容刷新到 L3 缓存/主内存
- 使其他 CPU 核心的缓存行无效
- 作为 CPU 级别的内存屏障

### volatile 读操作

当执行 volatile 读时，JIT 编译器会插入一条 **Load Barrier**：

```java
// 源代码
while (!ready) {  // volatile 读
    // spin
}

// 伪汇编（x86）
mov    0x10(%esp),%eax   // Load 从内存读取
test   %eax,%eax
jne    loop              // 条件跳转

// 隐含的 Load Barrier：CPU 缓存失效
```

### 四种内存屏障

JMM 定义了四种内存屏障：

| 屏障类型 | 指令 | 作用 |
| --- | --- | --- |
| LoadLoad | 两条 load 指令之间 | 确保前面的 load 都读取完毕 |
| StoreStore | 两条 store 指令之间 | 确保前面的 store 都刷新到内存 |
| LoadStore | load 和 store 之间 | 确保 load 读取完毕后再 store |
| StoreLoad | store 和 load 之间 | 最强屏障，确保 store 刷新后再 load |

volatile 的读写组合：

```java
volatile int x;

// 读：LoadLoad + LoadStore
int r1 = x;   // LoadLoad 屏障
// 后续所有 load 和 store

// 写：StoreStore + StoreLoad
x = 1;        // StoreStore 屏障
// StoreLoad 屏障，刷新所有写
```

## volatile 的双重语义

### 可见性语义

volatile 确保**一个线程的修改对其他线程立即可见**：

```java
public class VolatileCounter {
    private volatile int count = 0;

    public void increment() {
        count++;  // 原子操作，但 volatile 只保证可见性
    }

    public int get() {
        return count;
    }
}
```

:::warning count++ 不是原子操作
`count++` 包含三个操作：读取、递增、写入。volatile 只能保证读取和写入的可见性，不能保证递增的原子性。如果需要原子递增，应该使用 `AtomicInteger.incrementAndGet()`。
:::

### 禁止重排序语义

volatile 禁止编译器重排序，使得：

```java
// 双重检查锁定（单例模式）
public class Singleton {
    private static volatile Singleton instance;

    public static Singleton getInstance() {
        if (instance == null) {
            synchronized (Singleton.class) {
                if (instance == null) {
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
```

`instance = new Singleton()` 分解为三个操作：
1. 分配内存
2. 调用构造函数
3. 将引用赋值给 instance

如果没有 volatile，步骤 2 和 3 可能重排序，导致其他线程看到一个未构造完成的对象。volatile 禁止了这种重排序。

## happens-before 规则的推导

volatile 的写-读关系可以推导出其他 happens-before 关系：

```java
int a = 0;
volatile boolean flag = false;

Thread A:
    a = 1;           // 操作 1
    flag = true;    // 操作 2（volatile 写）

Thread B:
    if (flag) {     // 操作 3（volatile 读）
        int b = a;  // 操作 4
        System.out.println(b);  // 一定是 1
    }
```

推导过程：
- 操作 1 happens-before 操作 2（程序顺序）
- 操作 2 happens-before 操作 3（volatile 规则）
- 操作 3 happens-before 操作 4（程序顺序）

因此：操作 1 happens-before 操作 4，Thread B 一定能读取到 `a = 1`。

## volatile vs synchronized

| 特性 | volatile | synchronized |
| --- | --- | --- |
| 原子性 | 仅保证单个变量的读写原子性 | 块原子性 |
| 可见性 | 立即同步到主内存 | 释放锁时刷新到主内存 |
| 重排序 | 禁止重排序 | 块内禁止重排序 |
| 性能 | 略低于普通变量 | 有一定开销 |

```java
// volatile 适用场景
volatile boolean shutdownRequested;

// 场景1：状态标志
// 多个线程读取状态，一个线程修改状态

// 场景2：一次性发布
// 构造函数完成后，设置 initialized = true
```

## 总结

volatile 是 Java 提供的轻量级同步机制：

- **底层通过内存屏障实现**，强制刷新写缓冲区和失效读缓存
- **提供两种语义**：可见性保证和禁止重排序
- **遵循 happens-before 规则**：volatile 写 happens-before volatile 读
- **适用场景**：状态标志、一次性发布、低频更新字段

理解 volatile 的本质是理解 Java 内存模型的关键，也是理解 Java 并发编程的基础。
