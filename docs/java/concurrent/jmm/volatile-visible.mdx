# volatile 可见性原理：内存屏障

`volatile` 是 Java 并发编程中最容易误解的关键字之一。很多人知道它能保证"可见性"，但为什么能保证？底层是怎么实现的？`volatile` 修饰的变量在读写时，CPU 和编译器会插入特殊的**内存屏障**指令，这些屏障确保了操作的**有序性**和**可见性**。今天我们就来深入剖析 volatile 的内存屏障原理。

## volatile 的两大特性

`volatile` 保证了两大特性：
1. **可见性**：一个线程对 volatile 变量的修改，对其他线程立即可见
2. **有序性**：禁止指令重排序

```java
public class VolatileBasic {
    
    volatile int x = 0;
    
    public void writer() {
        x = 100;  // 写 volatile
    }
    
    public void reader() {
        int r = x;  // 读 volatile
        // 一定能读到 100
    }
}
```

## 内存屏障的概念

### 什么是内存屏障？

内存屏障（Memory Barrier）是一种 CPU 指令，用于：
1. **禁止在屏障前后的指令重排序**
2. **强制将写缓冲区/缓存中的数据刷新到主内存**
3. **使 CPU 缓存中的数据失效**

```
┌────────────────────────────────────────────────────────────────┐
│                      内存屏障示意                                  │
│                                                                 │
│   普通操作：编译器/CPU 可能重排序                                  │
│   ─────────────────────────────────────                        │
│   Store A                                                         │
│   Store B   ────────►  可以重排序为  ────────► Store B          │
│   Store C                        Store A                        │
│                        Store C                                  │
│                                                                 │
│   内存屏障：禁止重排序                                            │
│   ─────────────────────────────────────                        │
│   Store A                                                         │
│   │ Memory Barrier │ ← StoreStore 屏障                         │
│   Store B                                                         │
│   │ Memory Barrier │ ← StoreLoad 屏障                           │
│   Load C                                                          │
│                                                                 │
└────────────────────────────────────────────────────────────────┘
```

### 四种内存屏障

| 屏障类型 | 说明 |
|---------|------|
| **LoadLoad** | 确保屏障前的 Load 先于屏障后的 Load |
| **StoreStore** | 确保屏障前的 Store 先于屏障后的 Store |
| **LoadStore** | 确保屏障前的 Load 先于屏障后的 Store |
| **StoreLoad** | 确保屏障前的 Store 先于屏障后的 Load |

## volatile 写操作的内存屏障

### 写入 volatile 的过程

```java
public class VolatileWriteBarrier {
    
    volatile int x = 0;
    
    public void write() {
        x = 100;  // volatile write
        
        // 编译器/CPU 会插入内存屏障：
        // 
        // 1. StoreStore 屏障
        //    确保 x = 100 之前的写操作都已刷新到主内存
        //    防止：x = 100 先于其他写操作执行
        // 
        // 2. 将 x = 100 写入主内存（而不是仅在 CPU 缓存中）
        // 
        // 3. StoreLoad 屏障
        //    确保 x = 100 之后才执行后续的读操作
        //    防止：后续的读操作先于 x = 100 执行
    }
}
```

### StoreStore 屏障的作用

```java
public class StoreStoreBarrierDemo {
    
    int a = 0;              // 普通变量
    volatile int b = 0;      // volatile 变量
    
    public void method() {
        a = 1;           // Store A
        // StoreStore 屏障
        b = 2;           // Store B (volatile)
        // StoreLoad 屏障
    }
}
```

**没有 StoreStore 屏障时**，可能发生：
```
执行顺序：b = 2 → a = 1  // 重排序
结果：b = 2 对其他线程可见，但 a = 1 仍为旧值
```

**有 StoreStore 屏障后**：
```
执行顺序：a = 1 → b = 2  // 不会重排序
结果：b = 2 对其他线程可见时，a = 1 一定已完成
```

## volatile 读操作的内存屏障

### 读取 volatile 的过程

```java
public class VolatileReadBarrier {
    
    volatile int x = 0;
    
    public int read() {
        int r = x;  // volatile read
        
        // 编译器/CPU 会插入内存屏障：
        // 
        // 1. LoadLoad 屏障
        //    确保 x 的读取在屏障前的操作之后
        //    防止：读取 x 先于前面的读取操作
        // 
        // 2. LoadStore 屏障
        //    确保 x 的读取在屏障前的写操作之后
        //    防止：后续的写操作先于读取 x 执行
    }
}
```

### StoreLoad 屏障的作用

```java
public class StoreLoadBarrierDemo {
    
    volatile int b = 0;      // volatile 变量
    int c = 0;               // 普通变量
    
    public void method() {
        int r = b;           // Load B (volatile)
        // LoadStore 屏障
        c = 3;           // Store C
    }
}
```

**没有 LoadStore 屏障时**，可能发生：
```
执行顺序：c = 3 → r = b  // 重排序
结果：线程 B 可能读到 c = 3，但 b 是旧值
```

**有 LoadStore 屏障后**：
```
执行顺序：r = b → c = 3  // 不会重排序
结果：读取 b 后，c 的值一定对当前线程可见
```

## volatile 的完整语义

### 读写配对

```java
public class ReadWritePair {
    
    volatile int x = 0;
    volatile int y = 0;
    
    // 线程 A
    public void writer() {
        x = 10;           // Volatile Write
        // StoreStore 屏障
        // 写入 x
        // StoreLoad 屏障
        
        y = 20;           // Volatile Write
        // StoreStore 屏障
        // 写入 y
        // StoreLoad 屏障
    }
    
    // 线程 B
    public void reader() {
        // LoadLoad 屏障
        // 读取 y
        // LoadStore 屏障
        
        int r2 = y;
        int r1 = x;
        
        // 语义保证：
        // - 如果读到 y = 20，则一定也能读到 x = 10
        // - 因为 y 的写 Happens-Before y 的读
        // - y 的写 Happens-Before x 的写（程序顺序）
        // - x 的写 Happens-Before x 的读（volatile 规则）
        // - 传递性：y 的写 Happens-Before x 的读
    }
}
```

### StoreLoad 屏障的特殊性

StoreLoad 屏障是最"重"的屏障，因为它：
1. 需要将 Store Buffer 中的所有数据刷新到主内存
2. 需要让所有 CPU 核心的缓存失效
3. 可能导致 CPU 流水线停滞

```java
public class StoreLoadCost {
    
    volatile long x = 0;
    
    // 写操作
    public void write() {
        x = 1;  // 包含 StoreLoad 屏障，可能较慢
    }
    
    // 读操作
    public void read() {
        long r = x;  // 不包含 StoreLoad 屏障
    }
}
```

## volatile 在 JVM 层面的实现

### x86 架构的内存屏障

| 屏障类型 | x86 实现 |
|---------|---------|
| StoreStore | `mfence` 或空操作 |
| LoadLoad | 空操作 |
| LoadStore | 空操作 |
| StoreLoad | `mfence` 或 `lock` 指令 |

### x86 的特殊情况

在 x86/x64 架构下：
- StoreLoad 是唯一真正需要插入指令的屏障
- 其他三种屏障通常是空操作

```java
public class X86Barrier {
    
    volatile int x = 0;
    
    // x86 架构下的 volatile write:
    // mov $1, [%esp]     ; 写入 x
    // mfence             ; StoreLoad 屏障
    // 或者使用 lock 指令：
    // lock mov $1, [%esp]  ; lock 前缀隐含 StoreLoad 语义
    
    // x86 架构下的 volatile read:
    // mov [%esp], %eax    ; 读取 x
    // (无额外指令)
}
```

## volatile 的可见性保证

### 缓存一致性协议

当一个 CPU 核心写入 volatile 变量时：

```java
public class CacheCoherencyDemo {
    
    volatile int x = 0;
    
    public void write() {
        x = 100;  // CPU 核心 1 写入
    }
}
```

1. **CPU 核心 1** 将 x = 100 写入自己的 Store Buffer
2. **StoreLoad 屏障**强制将 Store Buffer 刷写到主内存
3. **缓存一致性协议（MESI）**通知其他 CPU 核心：
   - CPU 核心 2 的 L1/L2 缓存中的 x 副本变为 **无效（Invalid）**
   - CPU 核心 3 的 L1/L2 缓存中的 x 副本变为 **无效（Invalid）**
4. 当 **CPU 核心 2** 读取 x 时，发现缓存无效，从主内存重新读取

```
┌────────────────────────────────────────────────────────────────┐
│                   MESI 协议与 volatile                           │
│                                                                 │
│   CPU 核心 1         CPU 核心 2         CPU 核心 3               │
│   ┌─────────┐       ┌─────────┐       ┌─────────┐            │
│   │ Store   │       │ Cache   │       │ Cache   │            │
│   │ Buffer  │       │ (x=旧值) │       │ (x=旧值) │            │
│   │ x=100   │──────▶│ Invalid │──────▶│ Invalid │            │
│   └─────────┘ 通知   └─────────┘ 通知   └─────────┘            │
│         │                                                       │
│         │ 刷新到主内存                                           │
│         ▼                                                       │
│   ┌─────────────────────────────────────────┐                   │
│   │              主内存                       │                   │
│   │                                         │                   │
│   │              x = 100                     │                   │
│   │                                         │                   │
│   └─────────────────────────────────────────┘                   │
│                                                                 │
└────────────────────────────────────────────────────────────────┘
```

## 实战：验证 volatile 的可见性

```java
public class VolatileVisibilityTest {
    
    private static final int ITERATIONS = 1_000_000;
    
    // 测试 volatile 的可见性
    public static void testVolatile() throws InterruptedException {
        VolatileFlag flag = new VolatileFlag();
        
        Thread writer = new Thread(() -> {
            for (int i = 0; i < ITERATIONS; i++) {
                flag.value = i;
            }
            flag.stopped = true;
        });
        
        Thread reader = new Thread(() -> {
            int lastValue = -1;
            while (!flag.stopped) {
                // 一定能读取到 writer 线程写入的最新值
                lastValue = flag.value;
            }
            System.out.println("Volatile: 最后读取的值 = " + lastValue);
        });
        
        writer.start();
        reader.start();
        writer.join();
        reader.join();
    }
    
    // 对比测试：普通变量
    public static void testNormal() throws InterruptedException {
        NormalFlag flag = new NormalFlag();
        
        Thread writer = new Thread(() -> {
            for (int i = 0; i < ITERATIONS; i++) {
                flag.value = i;
            }
            flag.stopped = true;
        });
        
        Thread reader = new Thread(() -> {
            while (!flag.stopped) {
                // 可能永远看不到 stopped = true
            }
            System.out.println("Normal: 检测到停止");
        });
        
        writer.start();
        reader.start();
        writer.join();
        reader.join();
    }
    
    // 普通变量（可能不可见）
    static class NormalFlag {
        volatile int value = 0;  // 保持 stopped 是 volatile
        volatile boolean stopped = false;
    }
    
    // volatile 变量（保证可见性）
    static class VolatileFlag {
        volatile int value = 0;
        volatile boolean stopped = false;
    }
}
```

## 总结

`volatile` 的可见性通过内存屏障实现：

1. **volatile 写**：
   - StoreStore 屏障：禁止之前的普通写操作与 volatile 写重排序
   - StoreLoad 屏障：禁止 volatile 写与后续的读操作重排序

2. **volatile 读**：
   - LoadLoad 屏障：禁止后续的读操作与 volatile 读重排序
   - LoadStore 屏障：禁止后续的写操作与 volatile 读重排序

3. **底层机制**：
   - x86 架构主要依赖 StoreLoad 屏障
   - MESI 缓存一致性协议确保其他 CPU 核心的缓存失效
