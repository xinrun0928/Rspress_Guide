## 一、为什么不能随意配置线程数 🔴

### 1.1 线程数的误区

```
配置太大：
  ├─ 线程切换开销增大（上下文切换消耗CPU）
  ├─ 内存消耗增加（每个线程栈1MB）
  └─ 资源竞争加剧（锁竞争、缓存竞争）

配置太小：
  ├─ CPU利用率低（大量时间在等待IO）
  └─ 吞吐量低（任务串行执行）
```

**线程数不是越大越好，也不是越小越好**——需要精确计算。

### 1.2 性能曲线

```
吞吐量
  ↑
  │     ┌─ 合理配置区
  │   ╱ └─
  │ ╱     ←─ 线程数太少：利用率低
  │X        ←─ 最优点
  │ ╲     ←─ 线程数太多：上下文切换开销
  │   ╲
  └──────→ 线程数
```

## 二、核心计算公式 🔴

### 2.1 CPU密集型任务

**定义**：任务主要是CPU计算，如加密、压缩、矩阵运算、AI推理。

**核心公式**：

```
线程数 = CPU核心数 + 1
```

**为什么+1？** CPU密集型任务几乎100%占用CPU。保留一个额外的线程是为了当一个线程因为内存页错误或等待硬件中断而暂停时，其他线程可以使用CPU。

**验证**：

```java
// 4核CPU的服务器
// CPU密集型：线程数 = 4 + 1 = 5

// 在4核CPU上运行5个线程：
// - 4个线程在执行计算
// - 1个线程在等待（如等待内存分配）
// CPU利用率 ≈ 100%
```

### 2.2 IO密集型任务

**定义**：任务大部分时间在等待IO，如HTTP请求、数据库查询、文件读写。

**核心公式**：

```
线程数 = CPU核心数 × CPU利用率 × (1 + IO等待时间 / IO执行时间)
```

**简化公式**：

```
线程数 = CPU核心数 × 2  （经验值）
```

**或者更精确**：

```
线程数 = CPU核心数 / (1 - 阻塞系数)
阻塞系数 = 阻塞时间 / (阻塞时间 + CPU执行时间)
```

**验证**：

```java
// 4核CPU，任务中IO占比70%（即阻塞系数=0.7）
// 线程数 = 4 × (1 / (1 - 0.7)) = 4 × 3.33 ≈ 13.2

// 或者用简化公式：4 × 2 = 8
// 两种公式在IO密集型场景下都接近
```

### 2.3 混合型任务

实际项目中，任务通常是CPU和IO混合的：

```java
public class HybridTask implements Runnable {
    public void run() {
        // CPU计算部分
        computeHash();
        // IO部分
        dbQuery();
        // CPU计算部分
        processResult();
    }
}

// 配置策略：
// 1. 估算CPU时间和IO时间的比例
// 2. 根据比例调整线程数
```

## 三、availableProcessors()的坑 🟡

### 3.1 容器环境下的陷阱

```java
// ❌ 危险：在容器（Docker/K8s）中错误获取CPU核数
int ncpus = Runtime.getRuntime().availableProcessors();
// 问题：Java 8返回的是宿主机的CPU核数，而不是容器限制的CPU核数

// 例如：宿主机64核，容器限制为2核
// availableProcessors()返回64，但容器只有2核
// 配置 64/2+1 ≈ 33 个线程 → 创建33个线程 → 容器CPU疯狂调度 → 服务雪崩
```

**JDK版本对比**：

| JDK版本 | availableProcessors()在容器中的行为 |
|---------|----------------------------------|
| JDK 8 (u191之前) | 返回宿主机CPU核数（错误） |
| JDK 8 (u191+) | 返回容器CPU限制（正确） |
| JDK 10+ | 返回容器CPU限制（正确） |
| JDK 11+ | 更准确地返回容器限制 |

### 3.2 正确的获取方式

```java
// ✅ 方案一：使用容器环境变量（适用于所有JDK版本）
int cpuLimit = Integer.parseInt(
    System.getenv("CPU_LIMIT") != null ?
    System.getenv("CPU_LIMIT") :
    System.getenv("CONTAINER_CPU_LIMIT") != null ?
    System.getenv("CONTAINER_CPU_LIMIT") :
    "1");

int ncpus = Runtime.getRuntime().availableProcessors();
if (isInContainer()) {
    ncpus = Math.min(ncpus, cpuLimit); // 取较小值
}

// ✅ 方案二：JDK 10+，直接使用availableProcessors()
int ncpus = Runtime.getRuntime().availableProcessors(); // 容器感知

// ✅ 方案三：通过cgroup读取
try {
    String cpuQuota = new String(
        Files.readAllBytes(Paths.get("/sys/fs/cgroup/cpu/cpu.cfs_quota_us")));
    String cpuPeriod = new String(
        Files.readAllBytes(Paths.get("/sys/fs/cgroup/cpu/cpu.cfs_period_us")));
    int containerCpus = Integer.parseInt(cpuQuota) /
                         Integer.parseInt(cpuPeriod);
    ncpus = containerCpus;
} catch (Exception e) {
    // 使用默认值
}
```

## 四、不同场景的线程池配置参考 🟡

### 4.1 场景一：HTTP API服务

```java
// IO密集型（假设：4核CPU，任务IO占比70%）
int corePoolSize = (int) (4 / (1 - 0.7)); // ≈ 13
int maxPoolSize = corePoolSize; // IO密集型通常不需要太大max
int queueCapacity = 1000;

ThreadPoolExecutor executor = new ThreadPoolExecutor(
    corePoolSize, maxPoolSize,
    60L, TimeUnit.SECONDS,
    new ArrayBlockingQueue<>(queueCapacity),
    new ThreadFactoryBuilder().setNameFormat("http-worker-%d").build(),
    new CallerRunsPolicy()
);
```

### 4.2 场景二：数据库批量导入

```java
// CPU密集型
int corePoolSize = Runtime.getRuntime().availableProcessors() + 1;
int maxPoolSize = corePoolSize;
int batchSize = 1000;

ThreadPoolExecutor executor = new ThreadPoolExecutor(
    corePoolSize, maxPoolSize,
    0L, TimeUnit.SECONDS, // CPU密集型不需要keepAliveTime
    new LinkedBlockingQueue<>(batchSize * 2),
    new ThreadFactoryBuilder().setNameFormat("db-worker-%d").build(),
    new CallerRunsPolicy()
);
```

### 4.3 场景三：异步任务处理（高并发）

```java
// IO密集型（高并发）
int ncpus = Runtime.getRuntime().availableProcessors();
int corePoolSize = ncpus * 2; // 经验值
int maxPoolSize = ncpus * 4; // 峰值时可扩展

ThreadPoolExecutor executor = new ThreadPoolExecutor(
    corePoolSize, maxPoolSize,
    60L, TimeUnit.SECONDS,
    new ArrayBlockingQueue<>(10000),
    new ThreadFactoryBuilder().setNameFormat("async-worker-%d").build(),
    new RejectedExecutionHandler() {
        public void rejectedExecution(Runnable r, ThreadPoolExecutor e) {
            metrics.increment("task.rejected");
            log.warn("任务被拒绝，等待队列已满");
        }
    }
);
```

## 五、生产调优实践 ⚠️

### 5.1 监控指标

必须监控以下指标：

```java
// 核心监控指标
executor.getPoolSize();         // 当前线程数
executor.getActiveCount();     // 正在执行任务的线程数
executor.getQueue().size();     // 队列中的任务数
executor.getCompletedTaskCount(); // 已完成任务总数
executor.getLargestPoolSize();  // 历史最大线程数
```

### 5.2 动态调整线程池

```java
// Spring Boot 动态线程池
@Configuration
public class ThreadPoolConfig {
    @Bean
    public ThreadPoolTaskExecutor taskExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(8);
        executor.setMaxPoolSize(16);
        executor.setQueueCapacity(1000);
        executor.setKeepAliveSeconds(60);
        executor.setThreadNamePrefix("biz-");
        executor.setRejectedExecutionHandler(new CallerRunsPolicy());
        executor.setWaitForTasksToCompleteOnShutdown(true);
        executor.setAwaitTerminationSeconds(60);
        executor.initialize();
        return executor;
    }
}

// 通过 actuator 动态调整
@RestController
public class ThreadPoolController {

    @Autowired
    private ThreadPoolTaskExecutor executor;

    @GetMapping("/pool/resize")
    public String resizePool(@RequestParam int core, @RequestParam int max) {
        executor.setCorePoolSize(core);
        executor.setMaximumPoolSize(max);
        return "OK: core=" + core + ", max=" + max;
    }
}
```

## 六、❌ 错误示范

**候选人原话**："线程池大小设置为10最稳定。"

**问题诊断**：
- "10"是最经典的拍脑袋配置
- 10个线程在1核CPU上会疯狂上下文切换
- 在64核CPU上可能严重浪费资源
- 必须根据任务类型和机器配置来计算

**候选人原话**："IO密集型任务用2倍CPU核心数，CPU密集型用4倍。"

**问题诊断**：
- 搞反了：CPU密集型应该是核心数+1，IO密集型才需要更多线程
- 2倍/4倍只是经验值，不够精确

【面试官心理】
线程池大小配置的题目，考核的是候选人对系统资源利用的理解。能正确说出CPU密集型和IO密集型配置差异的候选人，说明他有性能优化的实战经验。能指出availableProcessors()在容器中的陷阱的候选人，说明他有在生产环境中处理过真实问题的经历。

## 七、面试应对脚本

**面试官**："线程池大小应该怎么配置？"
**标准回答**："线程池大小的配置取决于任务类型。CPU密集型任务（大量计算）的配置是：CPU核心数+1。比如4核CPU配置5个线程。原因是CPU密集型任务几乎100%占用CPU，+1是为了在某个线程等待时其他线程可以使用CPU。IO密集型任务的配置公式是：CPU核心数 × CPU利用率 × (1 + IO等待时间/IO执行时间)，简化为CPU核心数/(1-阻塞系数)。阻塞系数是阻塞时间占总时间的比例。如果不知道精确数值，可以用经验值：CPU核心数×2。还需要注意容器环境下availableProcessors()可能返回错误的值，需要根据容器CPU限制来调整。"

【面试官心理】
能回答出线程池配置公式并指出容器陷阱的候选人，说明他对性能优化有系统化的理解。这类候选人在配置线程池时不会"拍脑袋"，而是会通过监控和计算来确定最优配置。
