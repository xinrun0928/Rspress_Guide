## 一、CountDownLatch是什么：解决什么问题 🔴

### 1.1 场景引入：多线程汇合

想象一个场景：主线程需要等待N个子任务全部完成后才能继续执行。

传统方案：
```java
// ❌ 笨拙：轮询检查
Thread[] threads = new Thread[N];
for (int i = 0; i < N; i++) {
    threads[i] = new Thread(() -> doTask());
    threads[i].start();
}
while (true) {
    boolean allDone = true;
    for (Thread t : threads) {
        if (t.isAlive()) allDone = false;
    }
    if (allDone) break;
}
```

CountDownLatch的方案：
```java
// ✅ 优雅：倒计时门栓
CountDownLatch latch = new CountDownLatch(N);
for (int i = 0; i < N; i++) {
    new Thread(() -> {
        doTask();
        latch.countDown(); // 完成一个任务，倒计时-1
    }).start();
}
latch.await(); // 等待倒计时归零
System.out.println("所有任务完成");
```

### 1.2 CountDownLatch的核心特性

- **倒计时**：初始值N，每调用一次`countDown()`减1
- **一次性**：计数器归零后，不能重置，不能再次使用
- **阻塞等待**：`await()`阻塞直到计数器归零
- **不可中断的等待**：不响应中断，直到超时或计数归零

### 1.3 典型使用场景

**场景一：多线程预加载数据**

```java
// 主程序等待所有数据加载完成
CountDownLatch startLatch = new CountDownLatch(1); // 门控
CountDownLatch doneLatch = new CountDownLatch(10);  // 倒计时

for (int i = 0; i < 10; i++) {
    final int id = i;
    new Thread(() -> {
        try {
            startLatch.await(); // 等待开始信号
            loadData(id);
            doneLatch.countDown();
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    }).start();
}

// 10个线程都准备好后，同时开始
startLatch.countDown(); // 开门！
doneLatch.await();       // 等待所有数据加载完成
System.out.println("所有数据加载完毕");
```

**场景二：压力测试的并发门控**

```java
// 100个并发请求同时发起
CountDownLatch latch = new CountDownLatch(100);
ExecutorService executor = Executors.newFixedThreadPool(100);
long start = System.nanoTime();

for (int i = 0; i < 100; i++) {
    executor.submit(() -> {
        try {
            barrier.await(); // 所有线程到位
            httpClient.send(); // 同时发起请求
            barrier.countDown();
        } finally {
            latch.countDown();
        }
    });
}
latch.await();
long duration = System.nanoTime() - start;
System.out.println("100并发请求总耗时: " + duration / 1_000_000 + "ms");
```

## 二、CountDownLatch的底层实现 🔴

### 2.1 AQS共享模式

CountDownLatch内部有一个Sync类继承AQS：

```java
private static final class Sync extends AbstractQueuedSynchronizer {
    private final int count;

    Sync(int count) {
        setState(count); // AQS的state = 倒计时初始值
    }

    // 共享模式：tryAcquireShared返回负数表示获取失败
    protected int tryAcquireShared(int acquires) {
        return getState() == 0 ? 1 : -1; // state==0时才能获取
    }

    // 共享模式：countDown时调用
    protected boolean tryReleaseShared(int releases) {
        for (;;) {
            int c = getState();
            if (c == 0) return false; // 已经归零，不能再减
            int nextc = c - 1;
            if (compareAndSetState(c, nextc)) // CAS减1
                return nextc == 0; // 归零时返回true，触发唤醒
        }
    }
}
```

### 2.2 await()的实现

```java
public void await() throws InterruptedException {
    sync.acquireSharedInterruptibly(1);
}

public final void acquireSharedInterruptibly(int arg) {
    if (Thread.interrupted())
        throw new InterruptedException();
    if (tryAcquireShared() < 0)  // state==0？返回1（成功）；否则返回-1（失败）
        doAcquireSharedInterruptibly(arg); // 进入AQS共享模式等待队列
}
```

在`doAcquireSharedInterruptibly`中，线程被park。当`countDown()`使得state归零时，共享模式的`releaseShared`触发`doReleaseShared`，唤醒等待队列中的所有线程。

### 2.3 countDown()的实现

```java
public void countDown() {
    sync.releaseShared(0);
}

public final boolean releaseShared(int arg) {
    if (tryReleaseShared(arg)) { // CAS递减，返回true时唤醒
        doReleaseShared();       // 唤醒所有等待线程
        return true;
    }
    return false;
}
```

### 2.4 关键点：CAS + 自旋

`tryReleaseShared`使用**for循环+CAS**来递减计数器：

```java
protected boolean tryReleaseShared(int releases) {
    for (;;) {
        int c = getState();
        if (c == 0) return false; // 归零后不能再减
        int nextc = c - 1;
        if (compareAndSetState(c, nextc)) // CAS保证原子性
            return nextc == 0; // 归零时返回true
    }
}
```

**为什么用for循环？** 因为`compareAndSetState`可能失败（其他线程同时countDown），失败后重试直到成功。

## 三、CountDownLatch vs CyclicBarrier：核心区别 🟡

### 3.1 一句话区分

| 工具 | 特点 | 能否重置 |
|------|------|----------|
| CountDownLatch | "主线程等子线程"（等待N个事件完成） | ❌ 不能重置 |
| CyclicBarrier | "子线程互相等"（N个线程互相等待） | ✅ 可以重置 |

### 3.2 流程对比

```
CountDownLatch:
  子线程: countDown() countDown() countDown()
                               ↓         ↓
  主线程:                  await() ← 归零时唤醒

CyclicBarrier:
  线程1: 到达屏障 → 等待
  线程2: 到达屏障 → 等待
  线程3: 到达屏障 → 全部放行 ← 最后一个到达的线程执行回调
```

### 3.3 典型场景对比

| 场景 | 选哪个 | 原因 |
|------|--------|------|
| 主线程等待子任务完成 | CountDownLatch | 主线程不需要参与任务 |
| 线程A和B互相等待对方完成后才能继续 | CyclicBarrier | 两个线程互相等待 |
| 多个线程都准备好后同时开始 | CyclicBarrier + await | 所有线程都到达屏障后才放行 |
| 统计N个异步操作的完成时间 | CountDownLatch | 主线程统计，任务线程独立执行 |

## 四、生产避坑：CountDownLatch的陷阱 ⚠️

### 4.1 计数器放错位置

```java
// ❌ 错误：countDown放错位置
CountDownLatch latch = new CountDownLatch(1);
new Thread(() -> {
    doSomething();
    // 异常时不会countDown
    doAnotherThing();
    latch.countDown();
}).start();
latch.await();

// ✅ 正确：countDown放在finally中
new Thread(() -> {
    try {
        doSomething();
        doAnotherThing();
    } finally {
        latch.countDown();
    }
}).start();
```

### 4.2 线程池 + CountDownLatch的陷阱

```java
// ❌ 危险：主线程await，但子线程在lambda中捕获了InterruptedException
CountDownLatch latch = new CountDownLatch(1);
executor.submit(() -> {
    try {
        doSomething();
    } catch (InterruptedException e) {
        // 捕获了中断但没有重新设置中断标记！
        Thread.currentThread().interrupt();
    } finally {
        latch.countDown();
    }
});
latch.await(); // 可能永远阻塞

// ✅ 正确：使用CountDownLatch的await(long, TimeUnit)重载
latch.await(30, TimeUnit.SECONDS);
```

### 4.3 线程安全问题

`countDown()`可以并发调用，内部使用CAS保证计数器的线程安全。但要注意：

```java
// ❌ 错误：计数器初始值不能为0
CountDownLatch latch = new CountDownLatch(0);
latch.await(); // 立即返回，因为计数器已经是0
latch.countDown(); // 无效操作
```

## 五、❌ 错误示范

**候选人原话**："CountDownLatch可以重用，调用reset()就能重置。"

**问题诊断**：
- CountDownLatch**不能重置**！一旦计数器归零，就永远归零
- 想要可重置的屏障，应该用CyclicBarrier
- 这是CountDownLatch和CyclicBarrier的核心区别

**候选人原话**："CountDownLatch比CyclicBarrier性能更好。"

**问题诊断**：
- 两者不是竞争关系，是不同的使用场景
- 强行比较性能没有意义，因为功能不同
- 说明候选人没有理解两个工具的定位

【面试官心理】
CountDownLatch和CyclicBarrier的区别是Java并发包中最经典的对比题。能准确说出"一次性vs可重用"的候选人，说明他有过实际的并发场景设计经验。但更重要的是能说出"主线程等子线程"vs"子线程互相等"——这种场景化的描述比死记硬背更说明理解深度。

## 六、面试应对脚本

**面试官**："CountDownLatch的底层是怎么实现的？"
**标准回答**："CountDownLatch内部有一个Sync类继承AQS的共享模式。构造函数设置AQS的state为倒计时初始值。await()调用acquireSharedInterruptibly()，在tryAcquireShared()中检查state是否为0——为0则返回1表示成功，线程通过；否则返回-1，线程进入AQS共享模式等待队列被阻塞。countDown()调用releaseShared(0)，在tryReleaseShared()中用for循环+CAS将state减1。如果减到0，tryReleaseShared返回true，触发doReleaseShared()唤醒等待队列中的所有线程。"

**面试官追问**："CountDownLatch和CyclicBarrier的区别是什么？"
**标准回答**："核心区别是使用场景和是否可重用。CountDownLatch用于'主线程等待N个子任务完成'——子任务执行countDown()，主线程执行await()。它是一次性的，计数器归零后不能重置。CyclicBarrier用于'N个线程互相等待，然后同时继续执行'——所有线程都到达屏障点后才能继续，它支持重置（通过reset()方法）。打个比方：CountDownLatch像是发令枪，子运动员听到枪响后各跑各的；CyclicBarrier像是集合点，所有运动员到齐后一起出发。"

【面试官心理】
CountDownLatch是最简单的JUC同步工具之一，但能把它和AQS的共享模式串联起来的候选人，说明他对JUC的底层实现有体系化的理解。这类候选人在阅读JUC源码时，不会只停留在"知道用法"层面，而是会去理解Doug Lea为什么要这样设计。
