# ReentrantLock公平锁 vs 非公平锁：谁在背后操控你的线程

候选人小孙在面试中遇到了这样一个问题：

面试官问："ReentrantLock的new ReentrantLock(true)和new ReentrantLock(false)有什么区别？"

小孙说："true是公平锁，false是非公平锁。公平锁按请求顺序，非公平锁不排队。"

面试官追问："那你说说，公平锁和非公平锁的acquire流程有什么区别？"

小孙说："公平锁要检查队列。"

面试官继续追问："具体怎么检查？hasQueuedPredecessors()返回true意味着什么？非公平锁为什么不检查队列？"

小孙开始语无伦次。

【面试官心理】
这道题考核的是候选人对ReentrantLock底层实现的理解深度。公平锁和非公平锁的差异不在于API，而在于tryAcquire的ACQUIRE流程中是否调用hasQueuedPredecessors()。理解这个差异，才能理解为什么JDK默认选择非公平锁，才能在生产环境中正确选型。

## 一、公平锁 vs 非公平锁：概念定义 🔴

### 1.1 定义

**非公平锁（默认）**：线程竞争锁时，不排队，直接用CAS抢锁。只有当锁被占用时，才进入等待队列。

```java
ReentrantLock lock = new ReentrantLock(); // 默认非公平
ReentrantLock lock = new ReentrantLock(false); // 非公平
```

**公平锁**：线程竞争锁时，必须检查等待队列。如果队列中有更早等待的线程，则乖乖排队，不插队。

```java
ReentrantLock lock = new ReentrantLock(true); // 公平
```

### 1.2 表面现象

非公平锁允许"插队"——一个新到来的线程可能在队列头部线程之前获得锁。

```java
ReentrantLock fairLock = new ReentrantLock(true);  // 公平
ReentrantLock unfairLock = new ReentrantLock(false); // 非公平

// 线程A持有锁
// 线程B在等待队列中
// 线程C新来，试图获取锁

// 公平锁：B先于C获取锁
// 非公平锁：C可能先于B获取锁（如果A此时刚好释放）
```

## 二、源码对比：公平锁和非公平锁的差异 🔴

### 2.1 NonfairSync.tryAcquire

```java
// ReentrantLock.NonfairSync
static final class NonfairSync extends Sync {
    final void lock() {
        // 直接尝试CAS抢锁，不检查队列
        if (compareAndSetState(0, 1))
            setExclusiveOwnerThread(Thread.currentThread());
        else
            acquire(1); // CAS失败才走AQS标准流程
    }

    protected final boolean tryAcquire(int acquires) {
        // 非公平锁的tryAcquire：直接获取，不检查队列
        return nontryAcquire(acquires);
    }
}
```

关键点：**非公平锁在lock()阶段就尝试CAS抢锁**，完全不检查等待队列。只有在CAS失败后，才通过acquire()进入AQS队列。

### 2.2 FairSync.tryAcquire

```java
// ReentrantLock.FairSync
static final class FairSync extends Sync {
    final void lock() {
        acquire(1); // 直接走AQS流程
    }

    protected final boolean tryAcquire(int acquires) {
        final Thread current = Thread.currentThread();
        int c = getState();
        if (c == 0) {
            // 关键区别：hasQueuedPredecessors()检查队列
            if (!hasQueuedPredecessors() &&
                compareAndSetState(0, acquires)) {
                setExclusiveOwnerThread(current);
                return true;
            }
        } else if (current == getExclusiveOwnerThread()) {
            // 可重入：同一线程再次获取
            int nextc = c + acquires;
            setState(nextc);
            return true;
        }
        return false;
    }
}
```

**核心差异**：`!hasQueuedPredecessors()`。

### 2.3 hasQueuedPredecessors()的实现

```java
public final boolean hasQueuedPredecessors() {
    Node t = tail;
    Node h = head;
    Node s;
    // 队列非空 且 第一个等待线程不是当前线程
    return h != t &&
           ((s = h.next) == null || s.thread != Thread.currentThread());
}
```

这个方法检查两件事：
1. **队列是否非空**：`h != t`
2. **第一个有效节点是否是自己**：`s.thread != Thread.currentThread()`

如果队列非空且第一个等待线程不是当前线程，返回true——表示有更早的等待者，不能插队。

### 2.4 acquire流程图对比

```
公平锁 acquire(1):
  ① tryAcquire(1) ← hasQueuedPredecessors() 检查队列
  ② 如果失败，addWaiter(Node.EXCLUSIVE)
  ③ acquireQueued 自旋等待

非公平锁 acquire(1):
  ① lock() → 直接 CAS 抢锁（不检查队列！）
  ② 如果CAS失败，tryAcquire(1)（同样不检查队列）
  ③ 如果失败，addWaiter(Node.EXCLUSIVE)
  ④ acquireQueued 自旋等待
```

## 三、为什么默认选择非公平锁 🔴

### 3.1 性能原因：减少线程切换

非公平锁的优势在于：**允许新来的线程插队，利用锁释放和锁获取之间的时间窗口**。

```java
// 场景：线程A持有锁，线程B在等待队列中，线程C新来

// 如果线程A释放锁的瞬间，线程C刚好尝试获取锁：
// 非公平锁：线程C的CAS可能在线程B的阻塞线程被唤醒之前成功
// 节省了一次线程切换（用户态→内核态→用户态）
```

如果线程A刚释放锁，线程C立即来抢，非公平锁让C拿到锁，然后C释放，A又来拿——**线程C和线程A形成了一个"交接棒"式的流水线**，而线程B（队列中的等待者）暂时被跳过。

### 3.2 吞吐量数据

理论分析和实验数据都表明：
- 非公平锁的吞吐量约为公平锁的**2-5倍**
- 在高竞争场景下，差距更明显
- 公平锁因为要检查队列，每次获取锁都有额外的O(n)操作

### 3.3 公平锁的饥饿问题

公平锁严格的FIFO顺序，在高竞争场景下可能导致**线程饥饿（Starvation）**。

场景：
1. 线程A持有锁执行长时间操作
2. 线程B、C、D...Z依次到达，全部进入等待队列
3. 如果新线程不断到来，它们会全部排在队列尾部
4. B在队列头等了很久，但不断有新线程插队

等等，非公平锁才是"插队"的，为什么公平锁会饥饿？

**公平锁的饥饿发生在**：如果持有锁的线程执行时间很长，而新线程以高频到达，每次释放锁后，新线程的CAS可能比等待队列头部的线程更快地抢到锁——因为持有锁的线程释放锁到等待队列头部的线程被唤醒之间，有一个**唤醒延迟**（内核态切换、调度延迟），新线程的CAS可能在这个窗口内成功。

**非公平锁在这种情况下反而避免了饥饿**：新线程可以直接插队，不需要等待队列头部的线程被唤醒，锁的流转速度更快。

## 四、饥饿场景分析 🟡

### 4.1 什么场景下必须用公平锁

**场景1：需要严格的请求顺序**

比如某些金融交易系统，必须按照请求到达顺序执行，保证数据一致性。

```java
// 场景：订单处理必须按到达顺序
ReentrantLock fairLock = new ReentrantLock(true);
```

**场景2：需要保证所有线程都有执行机会**

在某些测试或模拟场景中，需要确保每个线程都能获得锁，避免某些线程被"饿死"。

### 4.2 饥饿vs吞吐量：权衡的艺术

| 维度 | 非公平锁 | 公平锁 |
|------|---------|--------|
| 吞吐量 | 高（2-5倍） | 低 |
| 响应时间 | 低（减少线程切换） | 高（需要排队） |
| 饥饿风险 | 低（新线程和持有者形成流水线） | 中（唤醒延迟导致插队） |
| 请求顺序 | 不保证 | 保证FIFO |
| 适用场景 | 绝大多数生产场景 | 严格顺序要求 |

## 五、生产避坑：公平锁的性能陷阱 ⚠️

### 5.1 高竞争场景下公平锁的灾难

```java
// ❌ 错误配置：高频HTTP服务使用公平锁
ReentrantLock lock = new ReentrantLock(true); // 灾难性的性能

// 线程A持有锁处理请求，100个新请求同时到达
// 每个请求都要检查hasQueuedPredecessors() → O(n)
// 100个线程都检查完队列，发现队列是空的（因为A还没释放）
// 全部CAS抢锁失败，100个线程全部进入等待队列
// 噩梦开始了...
```

在HTTP服务这种高频场景下，公平锁的性能灾难是灾难性的。

### 5.2 正确的选择

```java
// ✅ 默认：非公平锁
ReentrantLock lock = new ReentrantLock(); // 推荐

// ✅ 需要严格顺序时：公平锁
ReentrantLock fairLock = new ReentrantLock(true); // 仅在必要时使用
```

### 5.3 ReentrantReadWriteLock的公平性

```java
// ReentrantReadWriteLock默认也是非公平
ReentrantReadWriteLock rwLock = new ReentrantReadWriteLock();
ReentrantReadWriteLock rwLockFair = new ReentrantReadWriteLock(true);

// 注意：公平模式下，写锁优先于读锁
// 非公平模式下，写锁可能被后续到来的读锁插队（写饥饿）
```

ReentrantReadWriteLock的公平性有更复杂的权衡——公平模式下可能导致写饥饿，非公平模式下可能导致读饥饿。

## 六、❌ 错误示范

**候选人原话**："公平锁比非公平锁好，因为公平锁更安全。"

**问题诊断**：
- 把"公平"等同于"安全"，但对于并发性能来说，非公平锁反而更安全（避免饥饿）
- 没有理解饥饿和公平的权衡
- 混淆了"请求顺序的正确性"和"系统整体的吞吐量"

**候选人原话**："非公平锁会导致线程饥饿，所以不应该用。"

**问题诊断**：
- 实际上非公平锁的饥饿风险更低（因为吞吐量高，锁流转快）
- 公平锁在持有锁时间长的场景下，反而更容易导致等待线程长时间无法执行

【面试官心理】
问到公平锁和非公平锁的候选人，能说出hasQueuedPredecessors()这个方法的，说明他真的看过ReentrantLock的源码。但更难得的是能讲清楚背后的权衡——为什么JDK默认选择非公平锁，什么时候应该选公平锁。这种权衡思维才是P6/P7候选人和P5候选人的核心差距。

## 七、面试应对脚本

**面试官**："公平锁和非公平锁的acquire流程有什么区别？"
**标准回答**："核心区别在于是否检查等待队列。非公平锁的lock()方法直接用CAS抢锁，不检查队列。只有CAS失败后才走acquire()进入AQS流程。公平锁的lock()方法直接走acquire()，在tryAcquire()中调用hasQueuedPredecessors()检查队列——如果队列非空且第一个等待线程不是自己，就不能获取锁，返回false。hasQueuedPredecessors()通过比较tail和head节点，以及head.next的线程是否是当前线程来判断。"

**面试官追问**："为什么JDK默认选择非公平锁？"
**标准回答**："有三个原因。第一是吞吐量：非公平锁减少了线程切换次数——当锁刚释放时，新来的线程可以立即插队抢到锁，不需要先唤醒等待队列头部的线程再切换回来，这个窗口期的利用是非公平锁性能高的关键。第二是饥饿风险：公平锁在持有锁时间长的场景下，等待队列头部的线程可能因为唤醒延迟而一直被新线程插队，反而更容易饥饿。第三是实际场景：大多数业务场景不需要严格的请求顺序，而非公平锁的性能优势是实实在在的。"

**面试官追问**："什么场景下应该用公平锁？"
**标准回答**："需要严格FIFO顺序的场景，或者需要保证所有线程都有执行机会的场景。比如某些任务调度器需要严格按照优先级执行，或者在模拟/测试环境中需要确保每个线程都能获得锁。但说实话，这类场景在生产环境中非常少见。如果你不确定，就用非公平锁。"

【面试官心理】
能回答这道题的候选人，对ReentrantLock的理解已经到了"知其然更知其所以然"的层次。他们不只会调用API，还能理解JDK为什么要这样设计。更难得的是，他们能讲清楚背后的权衡——吞吐量vs公平性，唤醒延迟vs线程切换。这种权衡思维，是高级工程师和初级工程师的核心差距。
