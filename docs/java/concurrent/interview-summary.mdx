# Java 并发面试高频问题汇总

Java 并发编程是面试中的重中之重，也是区分候选人水平的关键领域。本文汇总了高频面试问题及参考答案，帮你系统梳理并发知识体系。无论你是面试官还是候选人，这份指南都能帮你事半功倍。

## synchronized 相关

### Q1：synchronized 和 Lock 的区别？

这是最常被问到的问题，表面是 API 层面的区别，深层考察的是对锁机制的理解。

| 特性 | synchronized | Lock |
|------|-------------|------|
| 获取方式 | 自动获取和释放 | 手动获取和释放 |
| 公平性 | 非公平锁 | 可选择公平或非公平 |
| 中断响应 | 不可中断等待 | 可中断等待（tryLock） |
| 条件变量 | Object.wait/notify | Condition |
| 尝试获取 | 不支持 | tryLock() |
| 底层实现 | JVM 内置 | AQS + CAS |

```java
// synchronized 示例：自动释放
public synchronized void syncMethod() {
    // 自动获取锁，方法结束自动释放
}

// Lock 示例：手动控制
private final Lock lock = new ReentrantLock();

public void lockMethod() {
    lock.lock();
    try {
        // 业务逻辑
    } finally {
        lock.unlock();  // 必须手动释放
    }
}
```

**深层理解**：synchronized 在 JDK 6 之后经历了大量优化，包括偏向锁、轻量级锁、锁消除、锁粗化等。在大多数场景下，两者性能差异不大。但 Lock 的灵活性是 synchronized 无法替代的。

### Q2：synchronized 锁的是什么？

synchronized 锁的是对象的 Monitor（监视器锁），与对象的 Mark Word 密切相关：

```java
public class SynchronizedDemo {
    // 锁的是 this 对象
    public synchronized void method1() {}

    // 锁的是当前类的 Class 对象
    public static synchronized void method2() {}

    // 锁的是指定的对象
    public void method3() {
        Object lock = new Object();
        synchronized (lock) {
            // 锁的是 lock 对象
        }
    }
}
```

**关键点**：实例方法锁的是 this 对象，静态方法锁的是 Class 对象，普通代码块锁的是指定对象。

### Q3：synchronized 的锁升级过程？

synchronized 采用了无锁 → 偏向锁 → 轻量级锁 → 重量级锁的升级策略，这个过程是不可逆的：

```
无锁 → 偏向锁 → 轻量级锁 → 重量级锁
         ↑          ↑           ↑
      只有一个    多线程竞争    竞争激烈
      线程访问    但自旋可以    或自旋过久
                   解决
```

- **偏向锁**：第一次由一个线程获取，Mark Word 记录线程 ID，后续该线程直接进入临界区
- **轻量级锁**：其他线程竞争时，升级为轻量级锁，通过 CAS 自旋尝试获取
- **重量级锁**：自旋超过阈值后升级，线程阻塞，涉及内核态切换

## JMM 相关

### Q7：volatile 保证了什么？

volatile 是 Java 面试的重点，它保证了两大特性：

```java
public class VolatileDemo {
    volatile boolean flag = false;

    // 线程 A
    public void writer() {
        flag = true;  // 写
    }

    // 线程 B
    public void reader() {
        if (flag) {  // 读
            // 一定能看到线程 A 写入的值
        }
    }
}
```

- **可见性**：一个线程对 volatile 变量的修改，对其他线程立即可见
- **有序性**：禁止指令重排序（内存屏障）

**注意**：volatile 不保证原子性！`volatile++` 不是原子操作。

### Q8：happens-before 规则？

happens-before 是 JMM 的核心概念，定义了操作之间的可见性保证：

1. **程序顺序规则**：同一个线程中，前面的操作 happens-before 后面的操作
2. **监视器锁规则**：unlock happens-before lock
3. **volatile 规则**：对 volatile 变量的写 happens-before 读
4. **线程启动规则**：Thread.start() happens-before 线程内的任何操作
5. **线程终止规则**：线程内的所有操作 happens-before 其他线程检测到终止
6. **传递性**：如果 A happens-before B，B happens-before C，则 A happens-before C

## 并发工具类

### Q9：CountDownLatch vs CyclicBarrier？

| 特性 | CountDownLatch | CyclicBarrier |
|------|----------------|---------------|
| 计数方向 | 递减到 0 | 递增到 parties |
| 可重用 | 否 | 是 |
| 等待线程 | 任意数量 | 固定数量 |
| 典型用法 | 主线程等待多个子任务 | 多个线程相互等待 |

```java
// CountDownLatch：主线程等待子任务
CountDownLatch latch = new CountDownLatch(3);
for (int i = 0; i < 3; i++) {
    new Thread(() -> {
        doWork();
        latch.countDown();
    }).start();
}
latch.await();  // 主线程等待

// CyclicBarrier：线程间相互等待
CyclicBarrier barrier = new CyclicBarrier(3, () -> System.out.println("所有线程到达"));
for (int i = 0; i < 3; i++) {
    new Thread(() -> {
        doWork();
        barrier.await();  // 等待其他线程
    }).start();
}
```

### Q10：ReentrantLock 可重入原理？

ReentrantLock 底层使用 AQS，通过 state 计数器实现可重入：

```java
// 获取锁：state + 1
if (compareAndSetState(0, 1)) {
    setExclusiveOwnerThread(Thread.currentThread());
} else if (Thread.currentThread() == getExclusiveOwnerThread()) {
    setState(getState() + 1);  // 重入，state + 1
}

// 释放锁：state - 1
if (Thread.currentThread() == getExclusiveOwnerThread()) {
    int state = getState() - 1;
    if (state == 0) {
        setExclusiveOwnerThread(null);
        setState(state);
        return true;
    }
    setState(state);
}
```

## CAS 相关

### Q11：CAS 原理？

CAS（Compare-And-Swap）是 CPU 提供的原子指令：

```java
// CAS 的语义
boolean compareAndSwap(Object obj, long expected, long newValue) {
    if (obj.value == expected) {
        obj.value = newValue;
        return true;
    }
    return false;
}
```

三个操作数：内存位置（V）、预期原值（A）和新值（B）。如果 V 符合 A，则用 B 更新 V，否则不更新。更新成功返回 true，失败返回 false。

### Q12：CAS 的 ABA 问题？

ABA 问题：变量从 A 变成 B 再变回 A，CAS 检查通过但实际上发生了变化。

**解决方案**：使用 AtomicStampedReference 带版本号：

```java
AtomicStampedReference<Integer> ref = new AtomicStampedReference<>(100, 0);

public boolean compareAndSet(int expected, int newValue,
                             int expectedStamp, int newStamp) {
    return ref.compareAndSet(expected, newValue, expectedStamp, newStamp);
}
```

## 实战问题

### Q13：如何实现线程安全的单例？

```java
// 方案一：饿汉式（类加载时就创建）
class HungrySingleton {
    private static final HungrySingleton INSTANCE = new HungrySingleton();
    private HungrySingleton() {}
    public static HungrySingleton getInstance() {
        return INSTANCE;
    }
}

// 方案二：双重检查 + volatile
class DCLSingleton {
    private static volatile DCLSingleton instance;
    private DCLSingleton() {}

    public static DCLSingleton getInstance() {
        if (instance == null) {  // 第一次检查
            synchronized (Singleton.class) {
                if (instance == null) {  // 第二次检查
                    instance = new Singleton();
                    // 创建对象不是原子操作：
                    // 1. 分配内存
                    // 2. 调用构造函数
                    // 3. 赋值给引用
                    // volatile 防止指令重排序
                }
            }
        }
        return instance;
    }
}

// 方案三：静态内部类
class StaticInnerSingleton {
    private StaticInnerSingleton() {}

    private static class Holder {
        private static final StaticInnerSingleton INSTANCE = new StaticInnerSingleton();
    }

    public static StaticInnerSingleton getInstance() {
        return Holder.INSTANCE;
    }
}

// 方案四：枚举（推荐）
enum EnumSingleton {
    INSTANCE;
    public void doSomething() {}
}
```

### Q14：如何避免死锁？

死锁产生的四个必要条件：互斥、占有并等待、非抢占、循环等待。破坏任意一个即可避免死锁：

```java
public class DeadLockPrevention {

    // 方案一：固定锁获取顺序
    public void transfer1(Account from, Account to, int amount) {
        // 始终按账户 ID 的顺序获取锁
        Account first = from.id < to.id ? from : to;
        Account second = from.id < to.id ? to : from;

        synchronized (first) {
            synchronized (second) {
                from.balance -= amount;
                to.balance += amount;
            }
        }
    }

    // 方案二：使用 tryLock 超时
    private final Lock lock = new ReentrantLock();

    public void transfer2(Account from, Account to, int amount) {
        while (true) {
            if (lock.tryLock(1, TimeUnit.SECONDS)) {
                try {
                    from.balance -= amount;
                    to.balance += amount;
                    break;
                } finally {
                    lock.unlock();
                }
            }
            // 获取失败，稍后重试
            LockSupport.parkNanos(1000000);
        }
    }

    // 方案三：减少锁的范围
    public void reduceLockScope(Account from, Account to, int amount) {
        int amountToTransfer = calculateAmount(amount);  // 锁外计算
        synchronized (from) {
            synchronized (to) {
                from.balance -= amountToTransfer;
                to.balance += amountToTransfer;
            }
        }
    }
}
```

### Q17：fork/join 框架原理？

ForkJoinPool 使用工作窃取算法：

```
线程1: [task1] [task2] [task3]           线程1: [task2] [task3]
线程2: [task4] [task5]        →          线程2: [task5] [task1(窃取)]
线程3: [task6] [task7]                    线程3: [task6] [task7]
```

- 每个线程有自己的双端队列
- 空闲线程从其他线程队列尾部"窃取"任务
- 减少线程空闲等待时间
- 适合计算密集型、分治策略的任务

## 面试技巧

### 如何回答"你用过哪些并发工具"？

不要只列举工具名称，要说明使用场景和踩过的坑：

> "我用过 ConcurrentHashMap 优化缓存查询。之前用 synchronized + HashMap，QPS 只有 3000，换成 ConcurrentHashMap 后 QPS 提升到 8000。我还用过 CountDownLatch 实现主线程等待多个初始化任务完成，用 CyclicBarrier 实现游戏房间匹配。"

### 如何回答"遇到过什么并发问题"？

STAR 法则（Situation, Task, Action, Result）：

> "场景：订单服务高峰期大量超时。任务：排查超时原因。行动：发现是 synchronized 锁粒度过大，所有订单操作都锁了整个方法。优化后改用分段锁，按订单号 hash 取模分 16 个锁实例。结果：超时率从 5% 降到 0.1%。"

### 如何展示深度理解？

不仅知道"是什么"，还要知道"为什么"和"怎么做"：

> "synchronized 在 JDK 6 之后做了大量优化，包括偏向锁（消除无竞争开销）、轻量级锁（CAS 自旋）、锁消除（ JIT 分析后移除不必要的锁）。在低并发场景，偏向锁让获取锁几乎没有开销；在高并发场景，竞争激烈时会升级为重量级锁，涉及操作系统 mutex 调用。"
