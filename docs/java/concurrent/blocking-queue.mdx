【面试官心理】

## 一、阻塞队列的核心概念 🔴

### 1.1 什么是阻塞队列

阻塞队列（BlockingQueue）在普通队列基础上增加了两个特性：
- **阻塞获取**：队列为空时，获取操作阻塞等待
- **阻塞插入**：队列满时，插入操作阻塞等待

```java
public interface BlockingQueue<E> extends Queue<E> {
    // 阻塞插入：队列满时等待
    void put(E e) throws InterruptedException;

    // 阻塞获取：队列空时等待
    E take() throws InterruptedException;

    // 超时阻塞插入
    boolean offer(E e, long timeout, TimeUnit unit) throws InterruptedException;

    // 超时阻塞获取
    E poll(long timeout, TimeUnit unit) throws InterruptedException;
}
```

### 1.2 阻塞队列与普通队列的对比

| 操作 | 普通队列 | 阻塞队列 |
|------|---------|---------|
| 添加 | add/offer → 队列满则异常/返回false | put/offer(timeout) → 队列满则阻塞/超时 |
| 获取 | poll → 队列空返回null | take/poll(timeout) → 队列空则阻塞/超时 |
| 线程模型 | 消费者需要轮询 | 生产者-消费者自动协调 |

### 1.3 核心使用场景

**生产者-消费者模式**：

```java
BlockingQueue<Integer> queue = new ArrayBlockingQueue<>(100);

// 生产者
new Thread(() -> {
    while (true) {
        try {
            queue.put(produce()); // 队列满则阻塞
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    }
}).start();

// 消费者
new Thread(() -> {
    while (true) {
        try {
            Integer item = queue.take(); // 队列空则阻塞
            consume(item);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    }
}).start();
```

## 二、ArrayBlockingQueue vs LinkedBlockingQueue 🔴

### 2.1 ArrayBlockingQueue

```java
public class ArrayBlockingQueue<E> extends AbstractQueue<E>
    implements BlockingQueue<E> {

    private final Object[] items;
    private int takeIndex, putIndex, count; // 无volatile，用putIndex/takeIndex间接保证

    public ArrayBlockingQueue(int capacity) {
        this(capacity, false); // 默认非公平
    }

    public ArrayBlockingQueue(int capacity, boolean fair) {
        if (capacity <= 0) throw new IllegalArgumentException();
        items = new Object[capacity]; // 有界数组
        putLock = new ReentrantLock(fair); // 支持公平/非公平
        notEmpty = putLock.newCondition();
        notFull = putLock.newCondition();
    }
}
```

**关键特性**：
- **有界**：容量必须在创建时指定，不可变
- **一把锁**：putLock和takeLock共用同一把ReentrantLock（公平模式下可指定FIFO）
- **循环数组**：takeIndex和putIndex循环使用，count间接保证线程安全

### 2.2 LinkedBlockingQueue

```java
public class LinkedBlockingQueue<E> extends AbstractQueue<E>
    implements BlockingQueue<E> {

    private final int capacity; // 默认Integer.MAX_VALUE（无界！）
    private final AtomicInteger count = new AtomicInteger();

    private Node<E> head, last;

    private final ReentrantLock takeLock = new ReentrantLock();
    private final ReentrantLock putLock = new ReentrantLock();
    private final Condition notEmpty = takeLock.newCondition();
    private final Condition notFull = putLock.newCondition();

    static class Node<E> {
        E item;
        Node<E> next;
    }
}
```

**关键特性**：
- **可选有界**：默认无界（Integer.MAX_VALUE）
- **两把锁**：takeLock和putLock分离，允许put和take并行操作
- **链表结构**：Node节点动态分配

### 2.3 核心对比

| 维度 | ArrayBlockingQueue | LinkedBlockingQueue |
|------|-------------------|--------------------|
| 容量 | 必须指定（有界） | 可选，默认Integer.MAX_VALUE（无界） |
| 锁 | 一把锁（put+take共用） | 两把锁（put和take分离） |
| 吞吐量 | 较低（竞争激烈） | 较高（put/take可并行） |
| 内存 | 预分配（固定开销） | 动态分配（节点开销） |
| 公平性 | 支持公平/非公平 | 只支持非公平 |

### 2.4 为什么线程池用LinkedBlockingQueue

```java
// ThreadPoolExecutor默认队列
new LinkedBlockingQueue<>(Integer.MAX_VALUE);

// 原因一：高吞吐量
// LinkedBlockingQueue的put和take可以用不同锁并行
// ArrayBlockingQueue的put和take共用一把锁

// 原因二：默认容量足够大
// 大多数场景下，队列积压到Integer.MAX_VALUE是不可能的
// 但注意：如果任务产生速度 > 处理速度，会OOM！
```

## 三、SynchronousQueue：没有缓冲的传递 🔴

### 3.1 什么是SynchronousQueue

SynchronousQueue**不存储任何元素**。每个put操作必须等待一个take操作，反之亦然。

```java
SynchronousQueue<Integer> queue = new SynchronousQueue<>();

// 生产者
new Thread(() -> {
    try {
        queue.put(42); // 阻塞，直到有消费者取走
        System.out.println("生产完成");
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
    }
}).start();

// 消费者
new Thread(() -> {
    try {
        Thread.sleep(1000);
        Integer val = queue.take(); // 阻塞，直到有生产者放入
        System.out.println("消费: " + val);
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
    }
}).start();
```

### 3.2 适用场景

```java
// JDK的CachedThreadPool使用了SynchronousQueue
new ThreadPoolExecutor(0, Integer.MAX_VALUE, 60L,
    TimeUnit.SECONDS, new SynchronousQueue<>());
```

**适用场景**：
- **任务必须立即被消费**：不允许任何积压
- **生产速度和消费速度匹配**：如果消费太慢，生产者会阻塞
- **需要直接传递**：如线程间的任务交接

### 3.3 SynchronousQueue vs ArrayBlockingQueue(1)

```java
// 效果类似，但实现不同
ArrayBlockingQueue<Integer> abq = new ArrayBlockingQueue<>(1);
SynchronousQueue<Integer> sq = new SynchronousQueue<>();

// ArrayBlockingQueue(1) 可以存放一个元素
abq.offer(42); // 成功，队列中有1个元素
// 消费者take()后才能再放

// SynchronousQueue 不能存放任何元素
sq.offer(42); // 失败！（队列已满）
// 必须有消费者在等待，才能放进去
```

**关键区别**：SynchronousQueue没有存储能力——它只是生产者和消费者之间的直接"握手"。

## 四、DelayQueue：延迟任务调度 🟡

### 4.1 什么是DelayQueue

DelayQueue中的元素必须实现`Delayed`接口，只有当元素的延迟过期后才能被取出：

```java
public interface Delayed extends Comparable<Delayed> {
    long getDelay(TimeUnit unit); // 剩余延迟时间
}

public class DelayQueue<E extends Delayed> extends AbstractQueue<E>
    implements BlockingQueue<E> {
    private final transient ReentrantLock lock = new ReentrantLock();
    private final PriorityQueue<E> q = new PriorityQueue<>(); // 优先队列
    private final Condition available = lock.newCondition();
}
```

### 4.2 实现一个延迟任务

```java
public class DelayTask implements Delayed {
    private final String task;
    private final long expireTime; // 过期时间（纳秒）

    public DelayTask(String task, long delayMillis) {
        this.task = task;
        this.expireTime = System.nanoTime() + TimeUnit.NANOSECONDS.toNanos(delayMillis);
    }

    @Override
    public long getDelay(TimeUnit unit) {
        return unit.convert(expireTime - System.nanoTime(), TimeUnit.NANOSECONDS);
    }

    @Override
    public int compareTo(Delayed o) {
        DelayTask other = (DelayTask) o;
        return Long.compare(this.expireTime, other.expireTime);
    }
}

// 使用
DelayQueue<DelayTask> delayQueue = new DelayQueue<>();

delayQueue.put(new DelayTask("延迟1秒", 1000));
delayQueue.put(new DelayTask("延迟2秒", 2000));

while (true) {
    DelayTask task = delayQueue.take(); // 阻塞直到延迟过期
    System.out.println("执行: " + task.task);
}
```

### 4.3 适用场景

- **定时任务调度**：如订单30分钟未支付则自动取消
- **缓存过期**：如本地缓存的TTL管理
- **超时重试**：如接口调用超时后自动重试

## 五、阻塞队列的选型指南 💡

### 5.1 选型决策树

```
需要限制队列容量？
├─ 是 → ArrayBlockingQueue（有界数组，一把锁）
│        └─ 需要高吞吐量？ → LinkedBlockingQueue（有界链表，两把锁）
└─ 否 → 需要直接传递？
          ├─ 是 → SynchronousQueue（无缓冲，直接握手）
          └─ 否 → DelayQueue（按延迟排序）
```

### 5.2 各队列适用场景速查

| 队列 | 适用场景 |
|------|---------|
| ArrayBlockingQueue | 需要精确容量控制，公平性要求 |
| LinkedBlockingQueue | 吞吐量优先，允许无界积压（但要防OOM） |
| SynchronousQueue | 生产-消费速度匹配，不允许积压 |
| DelayQueue | 延迟任务调度，定时处理 |
| PriorityBlockingQueue | 优先级任务调度 |

## 六、❌ 错误示范

**候选人原话**："LinkedBlockingQueue是无界的，所以不会满。"

**问题诊断**：
- 虽然LinkedBlockingQueue默认容量是Integer.MAX_VALUE
- 但当任务积压超过Integer.MAX_VALUE时，仍然会OOM
- 无界不是"无限"，是"容量足够大但有限"

**候选人原话**："ArrayBlockingQueue比LinkedBlockingQueue慢是因为用数组。"

**问题诊断**：
- ArrayBlockingQueue慢的核心原因是**一把锁**（put和take共用）
- LinkedBlockingQueue快的原因是**两把锁**（put和take可并行）

## 七、面试应对脚本

**面试官**："ArrayBlockingQueue和LinkedBlockingQueue的区别是什么？"
**标准回答**："ArrayBlockingQueue使用有界数组，一把ReentrantLock同时保护put和take操作。LinkedBlockingQueue使用链表，默认容量是Integer.MAX_VALUE（无界），用两把锁分别保护put和take操作，允许put和take并行执行。因为锁分离，LinkedBlockingQueue的吞吐量比ArrayBlockingQueue高。但LinkedBlockingQueue的无界容量在任务积压过多时可能导致OOM，需要注意。"

【面试官心理】
能回答LinkedBlockingQueue两把锁设计的候选人，说明他对并发数据结构的内部实现有深入理解。这种理解不仅来自源码阅读，还来自于实际生产环境中的性能调优经验。
