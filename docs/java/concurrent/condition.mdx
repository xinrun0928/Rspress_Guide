# Condition条件队列：比wait/notify更强大的同步原语

候选人小陈在面试中遇到了这样的问题：

面试官问："synchronized的wait/notify有什么局限？"

小陈说："wait必须在synchronized块里调用，而且只有一个等待队列。"

面试官追问："那Condition是怎么解决这些问题的？"

小陈说："Condition是配合ReentrantLock使用的，可以创建多个队列。"

面试官继续追问："那你说说Condition的await和signal是怎么实现的？底层是什么？"

小陈答不上来。

【面试官心理】
Condition是JUC并发包中最容易被忽略的组件之一。大多数候选人知道Condition可以创建多个等待队列，但不知道它是如何与AQS配合的。理解Condition的await/signal原理，是理解Java多线程协作的核心——生产者/消费者、缓冲区、屏障，都离不开条件队列。

## 一、为什么需要Condition：wait/notify的局限性 🔴

### 1.1 Object.wait/notify的三个问题

**问题一：只有一个等待队列**

synchronized只有一个内置的等待队列，无法满足多条件等待的需求。

```java
// 场景：生产者-消费者问题，需要两个条件：
// 1. 队列满时，生产者等待；队列空时，消费者等待
// 用wait/notify实现很别扭

synchronized (buffer) {
    while (buffer.isFull()) {
        buffer.wait(); // 等待"不满"条件
    }
    buffer.add(item);
    buffer.notify(); // 唤醒一个生产者或消费者，无法精确控制
}
```

notify()无法指定唤醒谁——唤醒的可能是不应该被唤醒的线程。

**问题二：必须在synchronized块中调用**

```java
// ❌ 错误：在Lock.lock()和unlock()之间调用wait
lock.lock();
try {
    while (conditionNotMet) {
        // 错误：ConditionObject是ReentrantLock创建的，
        // 只能在Lock的保护下使用
        condition.await();
    }
} finally {
    lock.unlock();
}
```

虽然看起来类似，但Condition和wait/notify的语义完全不同。

**问题三：虚假唤醒（Spurious Wakeup）**

`Object.wait()`的Javadoc明确指出：**wait可能虚假唤醒**——即线程从wait()返回但没有收到notify。

JLS规范要求必须用while循环包裹wait：

```java
// ✅ 正确：必须用while循环，防止虚假唤醒
while (conditionIsFalse) {
    object.wait();
}
```

### 1.2 Condition的解决方案

```java
ReentrantLock lock = new ReentrantLock();
// 创建两个独立条件队列
Condition notEmpty = lock.newCondition(); // 队列非空条件
Condition notFull = lock.newCondition();  // 队列非满条件
```

## 二、Condition的底层实现：AQS的条件队列 🔴

### 2.1 ConditionObject的结构

ConditionObject是AQS的内部类，实现了Condition接口：

```java
public class ConditionObject implements Condition, java.io.Serializable {
    // 条件队列：单向链表
    private transient Node firstWaiter;
    private transient Node lastWaiter;

    // 等待状态：CONDITION = -2
    static final int CONDITION = 0;
}
```

**关键设计**：Condition有一个独立的等待队列（单向链表），和AQS的CLH同步队列（双向链表）是**完全独立的两个队列**。

### 2.2 await()的流程

```java
public final void await() throws InterruptedException {
    if (Thread.interrupted())
        throw new InterruptedException();

    // ① 创建节点，加入条件队列
    Node node = addConditionWaiter();

    // ② 释放持有的锁（必须！await前必须持有锁）
    int savedState = fullyRelease(node);

    // ③ 阻塞当前线程，直到被signal或中断
    int interruptMode = 0;
    while (!isOnSyncQueue(node)) {
        LockSupport.park(this);  // 挂起线程
        // 检查中断...
    }

    // ④ 重新获取锁（被唤醒后）
    acquireQueued(node, savedState);
    // ...
}
```

**关键步骤分解**：

**① 加入条件队列**：创建CONDITION状态的节点，加入单向链表尾部
```java
private Node addConditionWaiter() {
    Node t = lastWaiter;
    if (t != null && t.waitStatus != Node.CONDITION) {
        // 清除非CONDITION节点（取消等待的节点）
        unlinkCancelledWaiters();
        t = lastWaiter;
    }
    Node node = new Node(Thread.currentThread(), Node.CONDITION);
    if (t == null)
        firstWaiter = lastWaiter = node;
    else
        t.nextWaiter = node;
    return node;
}
```

**② 释放锁**：调用`fullyRelease`将锁完全释放（重入多少次就释放多少次），并保存释放前的state值，用于后续恢复。

**③ 阻塞**：在while循环中`park`当前线程，直到被signal唤醒。

### 2.3 signal()的流程

```java
public final void signal() {
    if (!isHeldExclusively())  // 必须持有独占锁才能signal
        throw new IllegalMonitorStateException();

    Node first = firstWaiter;
    if (first != null)
        doSignal(first);  // 唤醒第一个等待者
}

private void doSignal(Node first) {
    do {
        // 将first从条件队列中移除
        firstWaiter = first.nextWaiter;
        first.nextWaiter = null;  // 断开引用

        // ② 关键：将节点从条件队列转移到同步队列（CLH队列）
        Transferer.transfer(this, first, false);
    } while (!transferForSignal(first) && (first = firstWaiter) != null);
}
```

**signal()的关键是transfer操作**：将节点从**条件队列（单向链表）**转移到**同步队列（双向CLH链表）**。

### 2.4 signalAll()：唤醒所有等待者

```java
public final void signalAll() {
    Node first = firstWaiter;
    if (first != null)
        doSignalAll(first);
}

private void doSignalAll(Node first) {
    lastWaiter = firstWaiter = null;
    do {
        Node next = first.nextWaiter;
        first.nextWaiter = null;
        transferForSignal(first);  // 逐个转移所有节点
        first = next;
    } while (first != null);
}
```

## 三、await/signal的完整生命周期 🟡

### 3.1 四个状态的流转

```
等待线程（线程A）：
  调用await()
    → 创建CONDITION节点，加入条件队列
    → 完全释放锁（fullyRelease）
    → park() 挂起，等待signal

唤醒线程（线程B）：
  调用signal()
    → 取出条件队列第一个节点
    → transferForSignal：转移到同步队列，改为0状态
    → LockSupport.unpark(线程A)  唤醒

被唤醒线程（线程A）：
  被unpark唤醒
    → 自旋：在同步队列中尝试acquire
    → acquireQueued：排队等待获取锁
    → 获取到锁，await()返回
```

### 3.2 Transfer的机制

`transferForSignal()`是Condition最核心的操作：

```java
final boolean transferForSignal(Node node) {
    // 尝试将节点状态从CONDITION改为0
    if (!compareAndSetWaitStatus(node, Node.CONDITION, 0))
        return false; // 失败（节点已取消）

    // 节点加入同步队列（CLH队列尾部）
    enq(node);

    // 检查是否需要unpark前驱节点
    int ws = node.prev.status;
    if (ws < 0 ||  // 前驱节点在同步队列中
        !compareAndSetWaitStatus(node.prev, ws, Node.SIGNAL)) {
        LockSupport.unpark(node.thread); // 唤醒当前节点
    }
    return true;
}
```

**设计精妙之处**：节点被signal后，立即加入同步队列尾部。然后检查前驱节点——如果前驱节点不是SIGNAL状态，说明前驱在同步队列中的acquireQueued即将完成，立即unpark当前节点让它抢锁。

## 四、生产者-消费者：Condition的经典应用 🔴

### 4.1 有界阻塞队列

```java
public class BoundedBlockingQueue<E> {
    private final Object[] items;
    private int count, takeIndex, putIndex;
    private final ReentrantLock lock = new ReentrantLock();
    private final Condition notEmpty = lock.newCondition();  // 队列非空
    private final Condition notFull = lock.newCondition();   // 队列非满

    public BoundedBlockingQueue(int capacity) {
        items = new Object[capacity];
    }

    public void put(E e) throws InterruptedException {
        lock.lockInterruptibly();
        try {
            while (count == items.length) {
                notFull.await(); // 队列满，等待"非满"
            }
            items[putIndex] = e;
            putIndex = (putIndex + 1) % items.length;
            count++;
            notEmpty.signal(); // 唤醒一个消费者
        } finally {
            lock.unlock();
        }
    }

    public E take() throws InterruptedException {
        lock.lockInterruptibly();
        try {
            while (count == 0) {
                notEmpty.await(); // 队列空，等待"非空"
            }
            @SuppressWarnings("unchecked")
            E item = (E) items[takeIndex];
            items[takeIndex] = null;
            takeIndex = (takeIndex + 1) % items.length;
            count--;
            notFull.signal(); // 唤醒一个生产者
            return item;
        } finally {
            lock.unlock();
        }
    }
}
```

**相比wait/notify的优势**：
- `notEmpty.await()`精确等待"队列非空"条件
- `notFull.await()`精确等待"队列非满"条件
- `signal()`可以精确唤醒对应的等待线程

## 五、Condition与wait/notify的对比 🟡

| 维度 | Object.wait/notify | Condition |
|------|--------------------|-----------|
| 数量 | 只有一个内置等待队列 | 可创建多个独立条件队列 |
| 调用要求 | 必须在synchronized块中 | 必须在Lock保护下 |
| 唤醒控制 | notify随机唤醒一个，notifyAll唤醒所有 | signal精确唤醒，signalAll全部唤醒 |
| 公平性 | 不支持 | 支持（FIFO唤醒） |
| 中断支持 | 有限 | 支持中断和超时等待 |
| 超时等待 | 不支持 | awaitNanos()支持超时 |
| 线程中断 | wait()抛InterruptedException | await()可中断/不中断 |

## 六、❌ 错误示范

**候选人原话**："Condition比wait/notify好，因为Condition更现代。"

**问题诊断**：
- 没有理解Condition必须配合ReentrantLock使用
- 没有理解Condition的await前必须持有对应的Lock
- 两者是互补关系，不是替代关系

**候选人原话**："signal()会立即唤醒await()的线程。"

**问题诊断**：
- signal()只是将节点从条件队列转移到同步队列，并尝试unpark
- 被唤醒的线程还需要在同步队列中重新竞争锁
- await()真正返回，是在重新获取锁之后

【面试官心理】
Condition这道题，能把await/signal的流程讲清楚的候选人，对AQS的内部机制有深入理解。这类候选人在设计复杂的多线程协作场景（如消息队列、生产者-消费者、屏障）时，不会只依赖JUC的高层API，而是能从底层机制出发设计出更精确、更高效的协作方案。

## 七、面试应对脚本

**面试官**："Condition的await()和signal()是怎么实现的？"
**标准回答**："Condition是AQS的内部类ConditionObject实现的。await()首先创建CONDITION状态的节点加入条件队列，然后完全释放当前持有的锁（fullyRelease，释放所有重入次数），最后park当前线程。signal()首先检查当前线程是否持有独占锁，然后取出条件队列的第一个节点，调用transferForSignal将它转移到AQS的同步队列（CLH队列）中，同时将节点状态从CONDITION改为0，并尝试unpark被signal的线程。被唤醒的线程需要重新在同步队列中竞争锁，acquireQueued获取锁后await()才返回。"

**面试官追问**："Condition和wait/notify的核心区别是什么？"
**标准回答**："Condition的核心优势是可以创建多个独立的条件队列。比如一个有界队列需要'队列非空'和'队列非满'两个条件，用wait/notify只能共用一个队列，signal()无法区分要唤醒哪类等待线程。用Condition可以创建notEmpty和notFull两个独立队列，signal()精确唤醒对应的等待者。此外Condition支持更丰富的等待方式（限时等待、可中断等待），而Object.wait()的超时实现更复杂。"

【面试官心理】
能把Condition的await/signal流程讲清楚的候选人，对AQS的理解已经到了非常深的层次。Doug Lea在Condition中使用的"两个独立队列"（条件队列+同步队列）设计，是理解Java多线程协作的关键。这种设计思想——分离关注点、独立管理不同状态的等待者——在操作系统和网络编程中有广泛应用。
