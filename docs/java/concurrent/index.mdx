# Java 并发编程

面试官问："synchronized 和 ReentrantLock 有什么区别？"

候选人小张说："synchronized 是 JVM 层面的，ReentrantLock 是 JDK 层面的。"

面试官点点头："那 synchronized 的锁升级过程是什么？"

小张说："...从偏向锁开始？"

面试官追问："为什么要设计偏向锁？Mark Word 的结构是什么？"

小张开始擦汗。

---

这道题我在字节和美团都问过，超过 70% 的候选人会卡在"锁升级过程"或"Mark Word 结构"上。不是并发编程太难，而是太多人只背结论，不看底层实现。

这一模块，带你把 Java 并发从"会用 synchronized"升级到"能讲清楚 JMM、CAS、AQS、线程池的全套底层逻辑"。

## 学习路径指引

### P5 候选人（校招/初级社招）

并发编程的入门关是**理解 JMM 和 synchronized 的基本用法**。

**P5 必过关卡**：

1. 什么是可见性、原子性、有序性？Java 是怎么保证的？
2. synchronized 的作用是什么？它修饰方法/代码块有什么区别？
3. volatile 和 synchronized 的区别是什么？
4. 线程池有哪些核心参数？拒绝策略是什么？

这四道题能流畅答出来的，P5 基础就算过关。

### P6 候选人（中级社招）

P6 的门槛在**能回答 synchronized 的锁升级过程和 AQS 核心原理**。

**拉开差距的追问链（锁机制）**：

```
synchronized 的锁升级过程
  → 偏向锁的作用是什么？什么时候撤销？（其他线程竞争）
    → 轻量级锁是怎么加锁的？（CAS 替换 Mark Word）
      → 什么情况下轻量级锁会膨胀为重量级锁？（自旋超过阈值）
        → 重量级锁的底层是什么？（Monitorenter）
          → Mark Word 在不同锁状态下的结构是什么？
```

这条追问链，P5 候选人基本走不完。能完整走下来的，基本都看过源码。

**拉开差距的追问链（AQS）**：

```
AQS 的核心思想是什么？（FIFO 队列 + CAS 状态控制）
  → 独占模式和共享模式的区别是什么？
    → ReentrantLock 是怎么实现可重入的？（state 计数）
      → 公平锁和非公平锁的实现有什么区别？（多一步 hasQueuedPredecessors 检查）
```

【面试官心理】
我出并发编程的题，通常从 synchronized 开始。如果候选人能说清楚锁升级和 Mark Word，我会往 AQS 方向深挖；如果卡住了，我会换个方向问线程池。并发这一块，能说清楚原理的人不超过 30%。

### P7 候选人（高级社招/架构方向）

P7 的并发不是在考实现，而是在考**方案选型和性能调优能力**。

**典型的 P7 级问题**：

- 你的线上服务 CPU 100%，怎么定位问题？是不是锁竞争导致的？怎么验证？
- 线程池的大小怎么估算？IO 密集型和 CPU 密集型的公式各是什么？
- ConcurrentHashMap 的 size() 在 JDK8 里为什么不加锁了？这种方式有什么代价？
- 生产环境线程池参数怎么调？有没有通用的经验公式？

:::tip 💡
CPU 密集型线程池大小公式：`核心线程数 = CPU 核心数 + 1`。IO 密集型：`核心线程数 = CPU 核心数 * 2` 或 `CPU 核心数 / (1 - 阻塞系数)`。
:::

## 生产避坑指南

并发问题是最难排查的生产事故类型之一，因为问题往往不可复现：

- **线程池拒绝策略没配**：线上流量突增时，线程池队列满了触发拒绝策略，如果没做日志记录，问题排查起来非常困难。
- **ThreadLocal 内存泄漏**：线程池复用线程时，ThreadLocal 变量没有 remove()，导致内存泄漏。最典型的场景是 Tomcat 线程池。
- **synchronized 锁的范围太大**：锁住了整个方法，高并发下性能严重退化。
- **ABA 问题**：使用 AtomicReference 时忽略 ABA 问题，在某些场景（栈、链表）下会导致严重 bug。
- **CountDownLatch 用错**：用 await() 等待，但忘记调用 countDown()，线程永远不往下走。

:::warning ⚠️
ThreadLocal 在线程池场景下最容易翻车。线程复用时，如果不在 finally 里调用 remove()，ThreadLocalMap 的 Entry 会一直持有 value 引用，导致内存泄漏。这不是 JDK 的 bug，是很多人没意识到的使用陷阱。
:::
