## 一、synchronized的底层实现：对象锁的本质 🔴

### 1.1 反汇编看真相

写一段最简单的synchronized代码：

```java
public class SyncDemo {
    private final Object lock = new Object();

    public void doSomething() {
        synchronized (lock) {
            System.out.println("Hello");
        }
    }
}
```

用`javap -v -c SyncDemo.class`反汇编，关键字节码：

```
public void doSomething();
  Code:
     0: aload_1
     1: dup
     2: astore_2
     3: monitorenter      // 获取监视器锁（进入同步块）
     4: getstatic #2
     7: ldc
     9: invokevirtual #3
    12: monitorexit       // 释放监视器锁（退出同步块）
    13: goto 20
    16: astore_3
    17: aload_2
    18: monitorexit       // 异常路径的释放锁
    19: athrow
    20: return
```

`synchronized`的底层就是两个指令：
- **`monitorenter`**：获取对象的监视器锁
- **`monitorexit`**：释放对象的监视器锁

### 1.2 监视器锁（Monitor）的原理

HotSpot中，每个对象都关联一个Monitor对象。`monitorenter`尝试获取锁，实际上是修改对象的**Mark Word**字段。

Mark Word是对象头的核心部分，它记录了对象的哈希码、GC分代年龄和**锁状态**。锁状态不同，Mark Word的存储内容也不同。

这就是synchronized的底层机制——**通过修改对象头的Mark Word来实现加锁**。

### 1.3 synchronized的三大特性

| 特性 | 说明 |
|------|------|
| 原子性 | monitorenter/monitorexit成对出现，保证临界区原子 |
| 可见性 | 解锁时将工作内存刷新到主内存（Store Barrier） |
| 有序性 | 临界区内禁止重排序 |

## 二、锁升级四阶段：偏向锁→轻量级锁→重量级锁 🔴

### 2.1 为什么要分级？

synchronized在JDK 1.6之前是纯重量级锁——线程竞争锁失败后直接进入内核态，等待OS调度。这个开销大约是**几十微秒**级别。

但现实中，大部分synchronized代码块的竞争并不激烈。很多场景下，锁只被一个线程反复获取和释放（比如单线程循环内的synchronized块）。

针对这个现象，JVM引入了**锁分级**机制：
- 无锁 → 偏向锁：消除无竞争时的同步开销
- 偏向锁 → 轻量级锁：处理短时间的多线程竞争
- 轻量级锁 → 重量级锁：处理长时间竞争

### 2.2 阶段一：偏向锁（Biased Locking）

**适用场景**：只有一个线程反复获取同一把锁。

**原理**：在Mark Word中记录获取锁的线程ID。以后这个线程再次进入同步块时，不需要任何同步操作，直接比较Mark Word中的线程ID即可。

```
Mark Word结构（偏向锁状态）：
| unused:25 | hashcode:31 | biased_lock:1 | lock:2 | thread:54 | epoch:2 | age:4 | lock:2 |
```

**偏向锁的获取**：当一个线程访问同步块时，在Mark Word中CAS写入自己的线程ID。写入成功后，Mark Word中保存的就是这个线程的ID。以后该线程进入/退出同步块时，只需要检查Mark Word中的线程ID是否是自己。

**偏向锁的撤销**：当另一个线程尝试获取偏向锁时，JVM需要撤销偏向锁。撤销是一个**全局安全点（Safe Point）**的操作，需要暂停持有偏向锁的线程。如果原线程不在同步块中，则可以直接撤销；如果在，则需要升级到轻量级锁。

**注意**：JDK 15开始默认禁用了偏向锁（JEP 374），因为在现代高并发场景下，偏向锁反而会带来额外的复杂度。

### 2.3 阶段二：轻量级锁（Lightweight Locking）

**适用场景**：多个线程交替获取锁，但不会同时竞争（无实际冲突，或冲突时间很短）。

**原理**：在栈帧中创建一个**锁记录（Lock Record）**，然后通过CAS将Mark Word修改为指向Lock Record的指针。

```
轻量级锁获取过程（线程A）：
1. 在自己的栈帧中创建锁记录（Lock Record）
2. 将对象的Mark Word复制到锁记录中（Displaced Mark Word）
3. CAS尝试将对象的Mark Word替换为指向锁记录的指针
   - 成功：获得轻量级锁，Mark Word的锁标志位变为00
   - 失败：说明有竞争，升级到重量级锁

轻量级锁释放：
1. CAS将Displaced Mark Word放回对象的Mark Word
2. 如果成功，无竞争释放
3. 如果失败，说明有竞争，释放锁并唤醒等待的线程
```

**轻量级锁的本质**：用CAS自旋替代了线程阻塞。对于短时间的竞争，自旋比阻塞/唤醒的开销更小。

**轻量级锁升级重量级锁的触发**：自旋超过阈值（默认10次，可自适应调整）后，或者有超过1个线程在自旋等待时，锁膨胀为重量级锁。

### 2.4 阶段三：重量级锁（Heavyweight Locking）

**适用场景**：多线程同时竞争同一把锁。

**原理**：Mark Word中保存指向Monitor对象的指针。竞争线程进入内核态，被OS的等待队列管理，CPU在用户态和内核态之间切换。

**性能问题**：重量级锁的获取涉及：
1. 用户态→内核态切换（系统调用）
2. OS将当前线程挂起，加入等待队列
3. 锁释放后，OS唤醒等待线程
4. 内核态→用户态切换

这个过程大约需要**几十微秒**，期间线程无法做其他事情。

**自旋优化**：为了避免频繁的线程切换，重量级锁引入了**自适应自旋**：如果这个锁刚刚被释放，CPU就让竞争线程自旋几个周期再进入等待队列。JDK 1.6默认开启，次数不固定（JIT自适应）。

### 2.5 锁状态速查表

| 锁状态 | 锁标志位 | Mark Word存储内容 | 适用场景 | 获取开销 |
|--------|----------|------------------|----------|----------|
| 无锁 | 01 | 对象的hashCode、分代年龄 | 无竞争 | 无 |
| 偏向锁 | 101 | 线程ID | 单线程反复获取 | 极低（CAS比较） |
| 轻量级锁 | 00 | 指向栈帧Lock Record的指针 | 短时间多线程交替 | 低（CAS自旋） |
| 重量级锁 | 10 | 指向Monitor的指针 | 多线程激烈竞争 | 高（系统调用+线程阻塞） |

## 三、锁优化：JVM的骚操作 🟡

### 3.1 锁粗化（Lock Coarsening）

如果同一个线程反复在同一个对象上加锁/解锁，JIT会把这几次锁合并成一次：

```java
// JIT优化前
for (int i = 0; i < 1000; i++) {
    synchronized (list) {
        list.add(i);
    }
}

// JIT优化后
synchronized (list) {  // 锁只加了一次
    for (int i = 0; i < 1000; i++) {
        list.add(i);
    }
}
```

减少了加锁/解锁的次数，提升性能。

### 3.2 锁消除（Lock Elimination）

JIT通过逃逸分析（Escape Analysis），如果判断一个锁对象不会逃逸出方法或线程，就直接消除这个锁：

```java
public void safe(List<Integer> list) {
    // 编译后发现vector是方法内局部对象，不会逃逸
    // 锁被消除
    synchronized (vector) {
        for (Integer i : list) {
            vector.add(i);
        }
    }
}
```

### 3.3 自适应自旋（Adaptive Spinning）

JDK 1.6引入的自旋不再固定次数，而是根据**上一次自旋的成功率**动态调整。如果上次在这个锁上自旋成功了，下一次就多自旋几次；如果上次失败了，就减少自旋甚至不自旋。

## 五、生产避坑：synchronized的线上陷阱 ⚠️

### 5.1 锁对象的改变

```java
// ❌ 错误：锁对象被改变后，锁失效
private Object lock = new Object();

public void doSomething() {
    synchronized (lock) {
        lock = new Object(); // 改变了锁对象！
        // 临界区内锁变了，等于没锁
    }
}
```

### 5.2 字符串常量作为锁

```java
// ❌ 错误：字符串常量池导致的"伪锁"
public void doSomething() {
    synchronized ("LOCK") { // 所有使用"LOCK"的地方共享同一把锁！
        // ...
    }
}
```

### 5.3 类对象作为锁的不一致

```java
// ❌ 错误：锁粒度过大
synchronized (MyClass.class) { // 所有MyClass实例共享同一把锁
    // 影响所有实例的并发
}

// ✅ 正确：锁粒度要精确
private final Object lock = new Object();
synchronized (lock) {
    // 只影响当前实例
}
```

### 5.4 死锁：两个线程互相等待

```java
// ❌ 危险代码
Object A = new Object();
Object B = new Object();

Thread 1:
synchronized (A) {
    synchronized (B) {
        // 业务逻辑
    }
}

Thread 2:
synchronized (B) {  // 如果Thread 1持有A等待B，Thread 2持有B等待A
    synchronized (A) {  // 死锁！
        // 业务逻辑
    }
}
```

**排查方法**：`jstack <pid>` 查看线程堆栈，搜索`Found one Java-level deadlock`，会显示具体的死锁位置。

## 六、面试应对脚本

**面试官**："synchronized的锁升级过程是什么？"
**标准回答**："synchronized的锁有四种状态，从无锁到偏向锁到轻量级锁到重量级锁，依次升级（不可降级）。偏向锁在只有一个线程反复获取锁时消除同步开销；轻量级锁在多个线程交替获取锁但不同时竞争时，用CAS自旋替代阻塞；重量级锁在多线程激烈竞争时使用，让OS管理等待队列。锁升级的触发条件是：偏向锁在有另一个线程尝试获取锁时撤销；轻量级锁在CAS自旋超过阈值时升级；轻量级锁在自旋失败时升级为重量级锁。"

**面试官追问**："Mark Word的结构是什么？"
**标准回答**："Mark Word是64位（或者32位在32位JVM中），不同锁状态下存储内容不同。无锁状态存储hashCode（31位）和分代年龄（4位）；偏向锁状态存储线程ID（54位）和epoch（2位）；轻量级锁状态存储指向栈帧锁记录的指针；重量级锁状态存储指向Monitor对象的指针。锁标志位（2位）决定了当前是哪种锁状态。"
