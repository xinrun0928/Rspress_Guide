# 进程、线程与协程：并发三巨头的恩怨情仇

候选人小周坐在字节跳动1-4级别的面试间，面试官翻到简历上"熟悉多线程编程"这一行，开口问道：

"进程和线程的区别是什么？"

小周说："进程是资源分配的基本单位，线程是CPU调度的基本单位。"

面试官点点头："那协程呢？"

小周愣了一秒："协程...也是一种并发方案？"

面试官追问："JDK 21引入了虚拟线程，你知道它和普通线程的区别吗？为什么说协程解决了线程的哪些问题？"

小周开始擦汗。

【面试官心理】
我问他进程、线程、协程，其实不是在考概念。我最想确认的是：他知不知道线程的痛点在哪里，协程为什么能解决这个问题，以及JDK 21的虚拟线程到底做了什么。知道"是什么"的占80%，能说出"为什么"的只有20%。

## 一、进程、线程、协程三者的本质区别 🔴

### 1.1 先说进程：隔离的代价

进程是操作系统分配资源的基本单位。每个进程都有自己独立的内存空间——代码段、数据段、堆、栈，全部隔离。

这带来了一个天然的好处：**进程之间不会相互踩踏**。一个进程崩溃了，不会直接搞挂另一个进程。

但代价是什么？**贵**。

进程切换（Context Switch）需要切换整个地址空间，涉及内核态切换、TLB刷新、缓存失效。根据Linux内核的数据，一次进程切换大约需要消耗 **2-5微秒**，而且切换时CPU的L1/L2缓存基本全废。

更致命的是进程间的通信（IPC）。管道、消息队列、共享内存、Socket...每一种都需要额外的系统调用，数据在内核空间和用户空间之间来回拷贝。一顿操作下来，跨进程通信的延迟轻松达到毫秒级。

### 1.2 线程：共享的甜头与代价

线程是进程内的执行单元。**同一个进程内的线程共享地址空间**——代码、数据、文件句柄、堆全部共享，只有栈和寄存器是线程私有的。

这带来了两个好处：
- **创建/切换快**：不需要切换地址空间，用户态就能完成线程切换，开销只有进程切换的1/10甚至更低
- **通信方便**：共享地址空间意味着数据共享不需要额外的系统调用

但共享本身就是一把双刃剑。**线程之间会相互踩踏**——一个线程越界写了别人的内存，整个进程都可能崩掉。而且线程切换虽然比进程快，但也需要内核参与：系统调用、模式切换、调度器介入，一次线程切换大约消耗 **0.5-2微秒**。

更关键的问题来了：**线程的数量不能太多**。

线程的栈空间默认 **1MB**（JDK默认1MB，可通过`-Xss`调整，但即使降到256KB，1万个线程也要2.5GB内存）。而且操作系统调度线程时，随着线程数增加，调度开销呈指数级增长。超过一定阈值后，增加线程不仅不能提升性能，反而会拖垮系统。

这就是线程模型的根本痛点：**并发度受制于系统资源，而非业务逻辑的并发需求**。

### 1.3 协程：用户态的轻量级并发

协程（Coroutine）是什么？**协程是用户态的轻量级执行单元，由程序员自己调度，不依赖操作系统的调度器。**

这是理解协程最关键的一句话。

线程的调度由操作系统内核控制，抢到CPU时间片才能执行。协程的调度完全在用户态完成——没有系统调用，没有模式切换，协程主动让出（yield）执行权给调度器，下一个协程接着跑。

带来的效果是：
- **切换开销极低**：协程切换只需要保存/恢复少量寄存器数据，开销在 **纳秒级**，是线程切换的100-1000倍
- **可以创建百万级别**：因为不需要独立栈空间（协程栈是动态伸缩的），可以在单线程内承载海量协程
- **摆脱了内核调度的限制**：不需要OS给时间片，协程自己决定什么时候让出、什么时候恢复

但协程也有代价：**必须有协作式的让出机制**。如果某个协程执行了一个CPU密集型任务而一直不让出，整个线程都会被卡住。所以协程必须主动调用`yield`，不像线程可以被OS强制抢走。

【面试官心理】
面试官追问协程，通常是想确认两件事：第一，他知不知道线程模型的根本痛点（资源消耗大、调度受限）；第二，他有没有跟进JDK 21的新特性。只懂传统线程不懂虚拟线程的，说明他的知识可能还停留在2-3年前。

## 二、JDK协程方案：Project Loom与虚拟线程 🔴

### 2.1 为什么JDK要引入虚拟线程

JDK 21正式发布的虚拟线程（Virtual Threads），是Project Loom的产物。这个项目的目标很简单：**让Java能够轻松创建数百万个并发任务，而不需要背线程模型的包袱。**

传统线程模型中，一个请求对应一个线程。电商场景下，假设QPS是10万，那就需要10万个线程。内存开销、调度开销直接爆掉。

异步编程（CompletableFuture、WebFlux）是当时的主流解法——用少量线程处理大量IO操作。但异步编程的代码是反直觉的，一整条同步链路的代码会被切成碎片，调试困难，生态割裂。

虚拟线程的出现，给了第三条路：**保留同步编程的模型，但用协程的方式实现**。一行`Thread.ofVirtual().start(...)`创建的就是虚拟线程，底层映射到少量OS线程上，复用同一个Carrier Thread。

### 2.2 虚拟线程的原理

虚拟线程的实现有几个核心设计：

**1. 虚拟线程不持有固定栈空间**

普通线程的栈是固定大小的（通常1MB），在JVM启动时就分配好了。虚拟线程的栈是**动态伸缩的**，初始很小（几百字节），随着需要增长到MB级别。`StackWalker`机制负责在需要时扩展栈空间，不需要时释放。

**2. 虚拟线程挂起时不占用OS线程**

当虚拟线程执行阻塞IO操作时，JDK会将其**挂起**（park），释放底层Carrier Thread，让同一个OS线程去执行其他虚拟线程。这就是虚拟线程能做到"百万并发"的核心——不是同时在跑，而是随时可以被调度恢复。

**3. 挂起/恢复通过Continuation实现**

Continuation是Loom的核心抽象。一个Continuation保存了挂起点处的执行状态（寄存器、栈帧），下次恢复时从挂起点继续执行。对程序员透明，但对JVM内部至关重要。

```java
// 创建虚拟线程
Thread virtualThread = Thread.ofVirtual().start(() -> {
    // 这个虚拟线程可以被挂起，不阻塞底层OS线程
    try {
        Thread.sleep(Duration.ofSeconds(10)); // 模拟IO等待
    } catch (InterruptedException e) {
        throw new RuntimeException(e);
    }
});
```

### 2.3 ❌ 错误示范

**候选人原话**："虚拟线程就是更轻量的线程，所以性能肯定更好。"

**问题诊断**：
- 没有理解虚拟线程的本质是**协作式调度**，不是并行执行
- 混淆了"并发"和"并行"——虚拟线程解决的是并发问题，不是并行问题
- 没有意识到虚拟线程在CPU密集型场景下可能没有优势

**候选人原话**："协程比线程好，因为线程太慢了。"

**问题诊断**：
- 忽略了协程需要主动让出（yield）的约束
- CPU密集型任务用协程反而更慢，因为协程没有真正的并行能力
- 没有区分IO密集型（协程有优势）和CPU密集型（线程/进程有优势）

【面试官心理】
问到虚拟线程时，我最怕听到两种回答：要么是把虚拟线程吹上天（"协程吊打线程"），要么是一问三不知（"没听说过"）。理想的回答应该承认虚拟线程的优势场景（IO密集型、海量并发），同时指出它的局限性（CPU密集型、不能替代线程池）。

## 三、追问升级：线程数规划与实战对比 🟡

### 3.1 追问链：线程模型选型

**第一层追问**：
面试官问："那你说说线程池和虚拟线程该怎么选？"
候选人答："...看场景？"

**第二层追问**：
面试官追问："IO密集型和CPU密集型任务，分别用什么？为什么？"
候选人答：...

这是P6/P7的分水岭。IO密集型任务的核心瓶颈是IO等待时间，不是CPU计算。大量线程等待IO时，用虚拟线程可以让少量OS线程服务大量IO任务，CPU利用率反而更高。CPU密集型任务的核心是计算资源，虚拟线程没有额外的CPU给你，反而多了一层调度开销。

**第三层追问**：
面试官追问："那如果我现在有一个同步HTTP客户端，怎么改成虚拟线程友好的写法？"
候选人答：...

这道题考的是实战落地能力。关键是**不要在虚拟线程中调用阻塞的同步方法**——那会挂起虚拟线程本身。正确做法是使用`java.net.http.HttpClient`（JDK 11+）或支持虚拟线程的HTTP客户端，配合`Future`/`CompletableFuture`。

### 3.2 三者对比

| 维度 | 进程 | 线程 | 协程/虚拟线程 |
|------|------|------|---------------|
| 调度者 | 操作系统内核 | 操作系统内核 | 用户态（协程调度器） |
| 切换开销 | 2-5微秒 | 0.5-2微秒 | 纳秒级 |
| 内存开销 | 独立地址空间（MB-GB级） | 独立栈（通常1MB） | 动态栈（初始几百字节） |
| 并发规模 | 百级 | 千级 | 百万级 |
| 通信方式 | IPC（管道/消息队列/共享内存） | 共享地址空间（需同步） | 共享地址空间（需同步） |
| 隔离性 | 进程级隔离，最安全 | 共享地址空间，有踩踏风险 | 共享地址空间，有踩踏风险 |
| 适用场景 | 进程隔离需求（容器隔离） | CPU/IO混合型并行任务 | 海量IO并发（HTTP/DB访问） |

## 四、生产避坑：线程池用错了会怎样 ⚠️

### 4.1 线程池配置不当的线上惨案

我们有个订单服务，部署在8核16G的机器上。开发同学配了`newFixedThreadPool(200)`，理由是"并发量高，得多开点线程"。

压测的时候QPS只有800，但CPU利用率只有15%。看起来CPU闲得发慌，但服务就是跑不快。

问题在哪？线程池大小和CPU利用率没有必然关系。如果任务都是IO密集型（数据库查询、HTTP调用），200个线程大部分时间都在等待IO，CPU当然跑不满。而且线程数太多，上下文切换的开销反而成了瓶颈。

**教训**：IO密集型任务，线程池大小公式是 `核心线程数 = CPU核心数 × 2`，或者更精确地 `核心线程数 = CPU核心数 × CPU利用率 × (1 + IO等待时间/IO执行时间)`。CPU密集型任务，线程数 = CPU核心数 + 1。

### 4.2 虚拟线程的"陷阱"

虚拟线程引入了几个容易踩的坑：

**坑1：在虚拟线程中用synchronized锁住共享资源**

synchronized在虚拟线程场景下可能引发**虚拟线程 pinned 到 Carrier Thread**的问题。当虚拟线程持有synchronized锁并执行阻塞操作时，JDK无法挂起它，因为它持有内部监视器锁。这种情况下底层OS线程被占用，无法调度其他虚拟线程。

解法：优先使用`java.util.concurrent.locks.ReentrantLock`替代synchronized，它在设计时就考虑了虚拟线程的挂起。

**坑2：在虚拟线程中创建大量ThreadLocal**

ThreadLocal每个线程一份。虚拟线程数量巨大时，如果每个虚拟线程都持有ThreadLocal，实际内存消耗可能比预想的要大。需要注意ThreadLocal的生命周期管理。

## 五、工程选型：什么场景用什么方案 💡

### 5.1 决策树

```
业务场景
├── 海量IO并发（HTTP请求、DB查询、RPC调用）
│   ├── 并发量 < 1000 → 普通线程池（IO密集型配置）
│   ├── 并发量 1000~10000 → 虚拟线程池
│   └── 并发量 > 10000 → 虚拟线程 + 异步框架
├── CPU密集型（计算密集、加密、压缩）
│   └── → 线程池（CPU核心数 + 1），不用虚拟线程
└── 需要进程隔离（安全性、容错）
    └── → 进程/容器隔离，不选线程
```

### 5.2 面试加分点

说到线程模型选型时，如果能补充这些，面试官会眼前一亮：

**加分点1：协程与Project Loom的演进脉络**
"JDK 19开始预览虚拟线程，21 LTS正式发布。这之前Java社区的协程方案主要是Quarkus的Kotlin协程和阿里开源的Quarkus协程框架。Loom的优势是语言层面的支持，不需要额外的第三方库。"

**加分点2：虚拟线程与Web框架的结合**
"Spring Boot 3.3开始支持虚拟线程，Tomcat 10默认启用虚拟线程模式。如果业务是同步IO模型迁移到虚拟线程，改动极小，性能提升明显。"

**加分点3：道法术器的层次**
"进程是安全边界，线程是并行单元，协程是并发单元。并行是同一时刻同时执行，并发是交替执行。虚拟线程解决的是海量并发的资源问题，不是并行问题——CPU核心数才是并行的天花板。"

【面试官心理】
这道题能答到最后的候选人，通常有三个特点：第一，能把线程模型的痛点讲清楚（不只是背结论）；第二，跟进了JDK 21的新特性；第三，有自己的工程判断力，能说清楚什么时候用什么方案。能做到第三步的，基本都是P6+了。

## 六、核心概念速记

**进程**：操作系统分配的最小资源单位，拥有独立地址空间，隔离性最好但开销最大。

**线程**：进程内的执行单元，共享地址空间，调度由内核完成，开销中等。

**协程**：用户态的轻量级执行单元，程序员自己调度，挂起/恢复开销极低，适合IO密集型海量并发。

**虚拟线程**：JDK 21引入的协程实现，保留同步编程模型，底层复用少量OS线程，可创建百万级并发。

**核心公式**：
- IO密集型线程数 = `CPU核心数 × 2` 或 `CPU核心数 × CPU利用率 × (1 + IO等待时间/IO执行时间)`
- CPU密集型线程数 = `CPU核心数 + 1`
