# 重量级锁：Monitor 机制与等待队列

当线程竞争变得激烈，轻量级锁会膨胀为**重量级锁**。

这是一种「真正的」锁机制，依赖操作系统底层的Mutex（互斥量）实现。代价是线程会进入阻塞状态，需要从用户态切换到内核态。

今天，我们来彻底理解重量级锁的工作原理。

## 为什么叫「重量级」？

轻量级锁失败后自旋，线程不会阻塞。重量级锁则不同：

```
┌─────────────────────────────────────────────────────────┐
│          用户态 vs 内核态切换                             │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  用户态（User Mode）：                                    │
│  - Java 代码执行区域                                      │
│  - 执行速度快，切换成本低                                 │
│                                                          │
│  内核态（Kernel Mode）：                                   │
│  - 操作系统内核执行区域                                   │
│  - 可以访问硬件、管理线程                                 │
│  - 切换成本高（约 1000-10000 CPU 周期）                   │
│                                                          │
│  轻量级锁自旋：用户态内完成                                │
│  重量级锁阻塞：用户态 ──> 内核态 ──> 用户态               │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

重量级锁的「重量」指的是线程状态切换的开销。

## Monitor 的本质

在 HotSpot 中，每个对象都关联一个 **Monitor**（监视器锁）。

```cpp
// Monitor 结构（简化）
class ObjectMonitor {
    // 头部
    ObjectHeader*  _header;      // 对象头
    volatile void* _object;       // 被监控的对象

    // 持有者信息
    volatile Thread * _owner;     // 持有锁的线程
    volatile intptr_t _count;     // 锁的递归计数

    // 等待队列
    ObjectWaiter * _WaitSet;      // 调用 wait() 的线程队列
    ObjectWaiter * _EntryList;    // 阻塞等待锁的线程队列

    // 其他状态
    volatile intptr_t _recursions; // 递归次数
};
```

## 重量级锁的获取

```java
synchronized (object) {
    // 临界区代码
}
```

执行过程：

```
┌─────────────────────────────────────────────────────────┐
│           重量级锁获取流程                                │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  线程 A 执行 synchronized (object)：                      │
│                                                          │
│  1. 检查对象头的 Mark Word：                              │
│     - 如果是无锁状态：CAS 加锁，进入临界区                │
│     - 如果是轻量级锁：膨胀为重量级锁                      │
│     - 如果是重量级锁：进入 EntryList 等待                 │
│              ↓                                           │
│  2. 成功获取锁：                                         │
│     - _owner = 当前线程                                  │
│     - _count++                                          │
│     - 进入临界区执行                                      │
│              ↓                                           │
│  3. 释放锁：                                             │
│     - _owner = null                                     │
│     - _count = 0                                        │
│     - 从 EntryList 唤醒一个线程                          │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

### 获取锁的源码

```cpp
// objectMonitor.cpp
void ObjectMonitor::enter(TRAPS) {
    Thread * const Self = THREAD;

    // 尝试 CAS 获取锁
    if (Try_Increment(&_count) != 0) {
        // 成功，锁计数加 1
        return;
    }

    // CAS 失败，进入自旋重试
    for (;;) {
        // 自旋获取
        if (Try_Spin(Self)) {
            return;
        }

        // 自旋失败，进入系统调用，阻塞当前线程
        EnterI(THREAD);
    }
}

// 进入等待队列
void ObjectMonitor::EnterI(TRAPS) {
    // 创建等待节点
    ObjectWaiter node(Self);
    node._prev = nullptr;
    node.TState = ObjectWaiter::TS_CXQ;

    // 加入等待队列（CX 队列，插入头部）
    for (;;) {
        ObjectWaiter * n = _CXQ;
        node._next = n;
        if (Atomic::cmpxchg(&_CXQ, &n, &node) == n) {
            break;
        }
        // 自旋等待...
    }

    // 挂起线程
    for (;;) {
        if (Try_Acquire(Self)) {
            return;
        }
        // park 挂起
        Self->_ParkEvent->park((use_absolute, 0));
    }
}
```

## 等待队列与阻塞机制

### 两个队列

Monitor 有两个重要的等待队列：

```
┌─────────────────────────────────────────────────────────┐
│           Monitor 的等待队列                              │
├─────────────────────────────────────────────────────────┤
│                                                          │
│   EntryList ──────────────────────────────────────────┐ │
│   ┌──────┐ ┌──────┐ ┌──────┐                         │ │
│   │ T1   │ │ T2   │ │ T3   │  ← 等待获取锁的线程      │ │
│   └──────┘ └──────┘ └──────┘                         │ │
│                                                          │ │
│   WaitSet ──────────────────────────────────────────── │ │
│   ┌──────┐ ┌──────┐ ┌──────┐                         │ │
│   │ W1   │ │ W2   │ │ W3   │  ← 调用 wait() 的线程  │ │
│   └──────┘ └──────┘ └──────┘                         │ │
│                                                          │ │
│   _owner ──────────────> Thread A ← 当前持有锁的线程   │
│                                                          │ │
└─────────────────────────────────────────────────────────┘
```

| 队列 | 存储的线程 | 进入条件 | 唤醒方式 |
|---|---|---|---|
| EntryList | 等待获取锁的线程 | CAS 获取锁失败 | 锁持有者释放时唤醒 |
| WaitSet | 调用 wait() 的线程 | 调用 `Object.wait()` | 调用 `notify()`/`notifyAll()` |

### 线程状态转换

```
线程 T1, T2, T3 竞争锁（锁被线程 A 持有）

T1: RUNNABLE ──(CAS 失败)──> BLOCKED ──(A 释放)──> RUNNABLE（获得锁）
T2: RUNNABLE ──(CAS 失败)──> BLOCKED ──(A 释放)──> BLOCKED ──(T1 释放)──> RUNNABLE
T3: RUNNABLE ──(CAS 失败)──> BLOCKED ──(A 释放)──> BLOCKED ──(T1 释放)──> BLOCKED
```

## wait/notify 机制

### wait()

```java
public class WaitNotifyDemo {
    private static final Object lock = new Object();

    public static void main(String[] args) throws InterruptedException {
        Thread waiter = new Thread(() -> {
            synchronized (lock) {
                try {
                    System.out.println("线程等待信号...");
                    lock.wait();  // 进入 WaitSet，释放锁
                    System.out.println("收到信号！");
                } catch (InterruptedException e) {
                    e.printStackTrace();
                }
            }
        });

        Thread notifier = new Thread(() -> {
            synchronized (lock) {
                System.out.println("发送信号...");
                lock.notify();  // 唤醒 WaitSet 中的一个线程
                // 注意：notify 后，当前线程不会立即释放锁
                // 需要执行完 synchronized 代码块才释放
                System.out.println("已发送信号，锁仍持有");
            }
        });

        waiter.start();
        Thread.sleep(100);
        notifier.start();
    }
}
```

### wait/notify 的流程

```
┌─────────────────────────────────────────────────────────┐
│           wait/notify 完整流程                            │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  线程 A（持有锁）调用 wait()：                            │
│  1. 将 _owner 置为 null                                  │
│  2. 将线程 A 加入 WaitSet                               │
│  3. 线程 A 进入 BLOCKED/WAITING 状态                    │
│              ↓                                           │
│  线程 A 释放锁                                            │
│              ↓                                           │
│  线程 B（从 EntryList 被唤醒）获得锁                      │
│              ↓                                           │
│  线程 B 调用 notify()：                                   │
│  1. 从 WaitSet 中取出线程 A                              │
│  2. 将线程 A 加入 EntryList（尾部）                      │
│  3. 线程 B 继续执行...直到释放锁                         │
│              ↓                                           │
│  线程 C（EntryList 头部）获得锁                          │
│  线程 A 从 wait() 返回，重新竞争锁                       │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

## notify 的「虚假唤醒」问题

### 问题描述

`notify()` 只唤醒一个线程，但**被唤醒的线程不一定是最早等待的线程**。

```java
// WaitSet 的实现可能是链表、栈或队列
// 不同 JVM 实现不同
```

### 解决方案

`wait()` 必须始终在循环中调用：

```java
// 错误写法
synchronized (lock) {
    while (condition 不满足) {
        lock.wait();  // 虚假唤醒后可能继续执行
    }
    // 处理
}

// 正确写法
synchronized (lock) {
    while (!condition) {
        lock.wait();
    }
    // 处理
}
```

## 生产者-消费者模式

```java
public class ProducerConsumer {
    private static final int CAPACITY = 10;
    private static final List<Integer> buffer = new ArrayList<>();

    public static void main(String[] args) {
        Object lock = new Object();

        // 生产者
        Thread producer = new Thread(() -> {
            for (int i = 0; i < 100; i++) {
                synchronized (lock) {
                    while (buffer.size() >= CAPACITY) {
                        try {
                            lock.wait();
                        } catch (InterruptedException e) {
                            Thread.currentThread().interrupt();
                        }
                    }
                    buffer.add(i);
                    System.out.println("Produced: " + i);
                    lock.notify();  // 唤醒消费者
                }
            }
        });

        // 消费者
        Thread consumer = new Thread(() -> {
            while (true) {
                synchronized (lock) {
                    while (buffer.isEmpty()) {
                        try {
                            lock.wait();
                        } catch (InterruptedException e) {
                            Thread.currentThread().interrupt();
                        }
                    }
                    int item = buffer.remove(0);
                    System.out.println("Consumed: " + item);
                    lock.notify();  // 唤醒生产者
                }
            }
        });

        producer.start();
        consumer.start();
    }
}
```

## 性能问题与优化

### 重量级锁的痛点

| 问题 | 原因 | 解决 |
|---|---|---|
| 线程阻塞 | 用户态/内核态切换 | 使用轻量级锁/偏向锁 |
| 唤醒延迟 | 操作系统调度 | 控制等待时间 |
| 竞争热点 | 单一锁 | 拆分锁/分段锁 |

### 解决方案：分段锁

ConcurrentHashMap 的分段锁设计：

```
┌─────────────────────────────────────────────────────────┐
│           ConcurrentHashMap 分段锁                       │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  ┌─────────┬─────────┬─────────┬─────────┐            │
│  │ Segment │ Segment │ Segment │ Segment │            │
│  │   0     │   1     │   2     │   3     │            │
│  └────┬────┴────┬────┴────┬────┴────┬────┘            │
│       │         │         │         │                  │
│       ▼         ▼         ▼         ▼                  │
│   ┌───────┐ ┌───────┐ ┌───────┐ ┌───────┐            │
│   │ Hash  │ │ Hash  │ │ Hash  │ │ Hash  │            │
│   │ Bucket│ │ Bucket│ │ Bucket│ │ Bucket│            │
│   └───────┘ └───────┘ └───────┘ └───────┘            │
│                                                          │
│  不同 Segment 的操作可以并行                              │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

JDK 8 的 ConcurrentHashMap 进一步优化，使用 **CAS + synchronized** 代替分段锁。

## 总结

重量级锁是 Java 并发的「兜底方案」：

1. **Monitor 机制**：每个对象关联一个 Monitor，Monitor 持有锁信息和等待队列
2. **EntryList**：竞争锁失败的线程在此等待
3. **WaitSet**：调用 wait() 的线程在此等待
4. **用户态/内核态切换**：线程阻塞和唤醒需要上下文切换
5. **wait/notify 必须成对使用**：且必须在循环中调用 wait()

理解重量级锁的原理，才能理解为什么 Java 要设计锁升级机制——**尽量用轻量级方案解决问题，只在必要时才动用「重量级」武器**。
