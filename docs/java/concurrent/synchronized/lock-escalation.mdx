# 锁升级机制：偏向锁 → 轻量级锁 → 重量级锁

你是否曾好奇，为什么 JDK 6 引入了偏向锁？为什么 synchronized 的性能从 JDK 6 开始有了质的飞跃？答案藏在这套精心设计的**锁升级机制**中。JVM 根据锁的竞争程度，动态地调整锁的策略，让 synchronized 在不同场景下都能表现出色。这不是魔法，而是 JVM 工程师们多年智慧的结晶。

## 锁升级：JVM 的自适应策略

锁升级（Lock Escalation）是 JDK 6 引入的重要优化，全称是"偏向锁→轻量级锁→重量级锁"的升级路径。这条路径是**单向的、不可逆的**——锁只能升级，不能降级。

```
┌─────────────────────────────────────────────────────────────────┐
│                        锁状态升级路径                              │
│                                                                  │
│   偏向锁 ────────────────────────────► 轻量级锁 ─────────────► 重量级锁  │
│    (Biased)    竞争发生                (Thin Lock)    自旋超过阈值   │
│                    │                                        │    │
│                    ▼                                        ▼    │
│              ◄──────────────────────◄───────────────────────    │
│                    偏向锁撤销，重新竞争                             │
└─────────────────────────────────────────────────────────────────┘
```

这种设计的核心思想是：**让无竞争的代码跑得更快，让有竞争的代码正确地等待**。

## 为什么要设计锁升级？

在讨论具体的升级机制之前，我们需要理解为什么需要锁升级。

### 偏向锁：无竞争时的极致优化

大多数 synchronized 代码块在整个程序生命周期内，只会被同一个线程访问。典型的场景：

```java
public class Singleton {
    private static volatile Singleton instance;
    
    public static Singleton getInstance() {
        if (instance == null) {           // 第一次检查
            synchronized (Singleton.class) {  // 只有一个线程会执行这里
                if (instance == null) {      // 第二次检查
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
```

在单例模式中，一旦 instance 创建完成，后续所有调用 `getInstance()` 的操作都只是读取引用，完全不需要同步。但如果每次调用都要走重量级锁的等待/唤醒流程，性能损失巨大。

**偏向锁**应运而生：第一个获取锁的线程，将自己的线程 ID 记录在 Mark Word 中，之后这个线程进入同步块时，只需要检查 Mark Word 中的线程 ID 是否是自己即可，完全不需要任何原子操作。

### 轻量级锁：短时竞争的高效处理

当偏向锁被撤销，或者多个线程交替执行同步块时，轻量级锁登场。轻量级锁的核心是 **CAS + 自旋**：

- **CAS（Compare-And-Swap）**：尝试把 Mark Word 原子地替换为指向栈帧中 Lock Record 的指针
- **自旋**：如果 CAS 失败，线程不阻塞，而是不停地重试

这种方式避免了线程的上下文切换开销，特别适合**锁持有时间很短**的场景。

### 重量级锁：长时间竞争的正确选择

如果锁竞争激烈，持有时间长，自旋就会浪费大量 CPU 资源。这时候就需要升级到重量级锁，让等待的线程真正阻塞。

## 锁升级的详细流程

### 第一阶段：无锁 → 偏向锁

对象创建时处于无锁状态，Mark Word 结构如下：

```
unused:25 | hashcode:31 | age:4 | biased_lock:1 | lock:2
       0     0 (lazy)        0     0              01
```

当第一个线程执行 `synchronized(obj)` 时：

1. 在当前线程的栈帧中创建 **Lock Record**，将 Mark Word 复制到 Lock Record 的 **Displaced Header**
2. 使用 **CAS** 将 Mark Word 替换为：`线程ID + 偏向标志位(1) + 锁标志位(01)`

```
thread_id:54 | epoch:2 | age:4 | biased_lock:1 | lock:2
    线程A的ID          0     1              01
```

偏向锁的"偏向"二字就体现在这里：**锁偏向了某个线程**。

### 第二阶段：偏向锁 → 轻量级锁

当有其他线程尝试获取偏向锁时，JVM 会撤销偏向锁。根据线程是否在安全点，撤销方式不同：

**场景一：线程在安全点**

```java
// 安全点撤销（Stop The World）
void revoke_bias_at_safe_point() {
    // 1. JVM 进入安全点，所有线程停止
    // 2. 遍历所有 Java 线程的栈，移除所有与该对象相关的 Lock Record
    // 3. 如果持有偏向锁的线程不在同步代码中，升级为无锁
    // 4. 如果持有偏向锁的线程正在同步代码中，升级为轻量级锁
    // 5. JVM 退出安全点，线程恢复执行
}
```

安全点撤销会导致**所有 Java 线程停顿（Stop The World）**，这是偏向锁的缺点之一。

**场景二：线程不在安全点**

```java
// 非安全点撤销（使用原子操作）
void revoke_bias_inline() {
    // 使用 CAS 原子地将 Mark Word 恢复到无锁状态
    // 请求线程会自旋等待偏向锁持有线程到达安全点后重新竞争
    mark_word.compare_and_set(
        biased_mark_of_thread_A,
        unlocked_mark
    );
}
```

撤销偏向锁后，对象处于无锁状态。请求线程会再次尝试获取锁，这次会走轻量级锁的流程：

```
┌──────────────────────────────────────────────────────┐
│  线程A（偏向锁持有者）           线程B（请求锁）        │
│         │                        │                  │
│         ▼                        ▼                  │
│  ┌─────────────┐         ┌─────────────┐            │
│  │  偏向锁撤销  │◄───────│ CAS 失败     │            │
│  │  (到达安全点) │         │ 自旋等待     │            │
│  └─────────────┘         └──────┬──────┘            │
│         │                        │                    │
│         ▼                        ▼                    │
│  ◄─────────────────────────────────────              │
│                                                      │
│         ◄───────  线程A 到达安全点后释放锁            │
│                                                      │
│         ▼                                           │
│  线程B CAS 成功，获取轻量级锁                          │
└──────────────────────────────────────────────────────┘
```

### 第三阶段：轻量级锁 → 重量级锁

轻量级锁的自旋不是无限的。当自旋次数超过阈值（默认 10 次），或者自旋线程超过 CPU 核心数的 1/2 时，轻量级锁会膨胀为重量级锁。

```java
// 轻量级锁膨胀的简化逻辑
void inflate_and_exit() {
    // 1. 创建 Monitor 对象
    ObjectMonitor monitor = new ObjectMonitor();
    
    // 2. 使用 CAS 将 Mark Word 设置为指向 Monitor 的指针
    //    锁标志位从 00 变为 10
    mark_word.compare_and_set(
        current_mark,         // 当前轻量级锁的 Mark Word
        monitor_address       // Monitor 地址
    );
    
    // 3. 将当前线程加入 Monitor 的 EntryList
    monitor.enter();
    
    // 4. 当前线程释放 Lock Record，自旋线程被唤醒后获取重量级锁
}

// 自旋线程的视角
void spin_and_acquire(Object obj) {
    for (int i = 0; i < spin_limit; i++) {
        if (obj.mark_word.try_lock()) {
            return;  // CAS 成功，获取锁
        }
        cpu_pause();  // CPU 空转
    }
    
    // 自旋超过阈值，放弃轻量级锁
    // 膨胀为重量级锁后阻塞
    inflate_and_block(obj);
}
```

## JVM 参数配置

可以通过 JVM 参数控制锁升级行为：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `-XX:+UseBiasedLocking` | true（JDK 6+） | 是否启用偏向锁 |
| `-XX:BiasedLockingStartupDelay` | 4秒 | JVM 启动后延迟多久启用偏向锁 |
| `-XX:+UseThinScaling` | - | 启用轻量级锁（隐式，JVM 自适应） |
| `-XX:+UseHeavyMonitors` | - | 禁用偏向锁，直接使用重量级锁 |
| `-XX:PreBlockSpin` | 10 | 自旋次数阈值（JDK 6 之前，之后自适应） |

```bash
# 关闭偏向锁
java -XX:-UseBiasedLocking -jar app.jar

# 调整偏向锁启动延迟
java -XX:BiasedLockingStartupDelay=0 -jar app.jar

# 高并发场景，直接使用重量级锁
java -XX:+UseHeavyMonitors -jar app.jar
```

## 逃逸分析的影响

JVM 的**逃逸分析（Escape Analysis）**会对锁升级产生重大影响。如果 JIT 编译器通过逃逸分析发现一个对象不会逃逸出方法或线程，它会：

1. **锁消除（Lock Elision）**：完全消除不必要的锁
2. **栈上分配（Stack Allocation）**：对象在栈上分配而非堆上
3. **标量替换（Scalar Replacement）**：将对象拆解为其成员变量

```java
public void escapeAnalysis() {
    // 如果 JIT 证明 obj 不会逃逸，可能完全不需要加锁
    synchronized (new Object()) {
        // ...
    }
    
    // 或者标量替换后：
    // int x = 0;  // 原 obj.x
    // int y = 0;  // 原 obj.y
    // ... 不需要任何同步
}
```

逃逸分析默认启用（`-XX:+DoEscapeAnalysis`），但只在方法 JIT 编译后才生效。

## 实战：观察锁升级

```java
import java.util.concurrent.TimeUnit;

public class LockEscalationDemo {
    
    public static void main(String[] args) throws InterruptedException {
        Object obj = new Object();
        
        // 1. 首先让对象进入偏向锁状态
        synchronized (obj) {
            System.out.println("偏向锁状态");
        }
        
        // 2. 等待偏向锁启用延迟
        TimeUnit.SECONDS.sleep(5);
        
        // 3. 模拟两个线程竞争
        Thread t1 = new Thread(() -> {
            synchronized (obj) {
                System.out.println("线程1获取锁");
            }
        });
        
        Thread t2 = new Thread(() -> {
            synchronized (obj) {
                System.out.println("线程2获取锁");
            }
        });
        
        // 先启动线程1，再启动线程2制造竞争
        t1.start();
        Thread.sleep(100);
        t2.start();
        
        t1.join();
        t2.join();
        
        System.out.println("锁已释放");
    }
}
```

使用 `-XX:+PrintFlagsFinal` 可以查看 JVM 的锁相关参数。

## 总结

锁升级机制是 JVM 为 synchronized 提供的一套自适应优化策略：

- **偏向锁**：消除无竞争时的同步开销，第一个获取锁的线程会"偏向"它
- **轻量级锁**：用 CAS + 自旋处理短时竞争，避免线程阻塞
- **重量级锁**：用 Monitor 处理长时竞争，让线程真正等待

理解锁升级机制，不仅能帮助我们写出更高性能的代码，还能在调试并发问题时提供思路。你有没有想过，在什么场景下应该禁用偏向锁？答案是：高并发、锁竞争激烈的场景，比如中间件服务器、RPC 框架等。
