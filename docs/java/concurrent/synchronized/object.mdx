# synchronized 锁的到底是什么？对象头结构详解

在 Java 中，`synchronized` 是最常用的同步机制，但很多人对它的理解仅限于"加锁"这个层面。深入思考一下，`synchronized` 锁的到底是什么？为什么加在同一对象上的锁会互斥？答案藏在 Java 对象的内存布局中——具体来说，是对象头（Object Header）的 Mark Word。理解 Mark Word 的结构，就理解了 synchronized 锁的本质。

## Java 对象的内存布局

在 HotSpot JVM 中，一个 Java 对象的内存布局分为三部分：

```
+------------------+------------------+------------------+
|   对象头 (Header)  |   实例数据 (Instance Data) |   对齐填充 (Padding) |
+------------------+------------------+------------------+
| Mark Word (64位=8字节) | Klass Pointer (压缩=4字节) |                   |
+------------------+------------------+------------------+
```

**对象头**包含两部分：
- **Mark Word**：存储对象的哈希码、GC 年龄、锁状态等信息
- **Klass Pointer**：指向类元数据的指针，JVM 通过它确定对象是哪个类的实例

对齐填充是为了让对象大小是 8 字节的倍数，提高内存访问效率。

## Mark Word：锁信息的核心载体

Mark Word 是 synchronized 锁机制的关键。它在不同状态下存储不同的信息：

| 锁状态 | 25 位 | 31 位 | 1 位 | 4 位 | 1 位 | 2 位 |
|--------|-------|-------|------|------|------|------|
| **无锁** | 对象哈希码 | 分代年龄 |偏向标志(0) | 锁标志位(01) |
| **偏向锁** | 线程 ID | Epoch | 分代年龄 | 偏向标志(1) | 锁标志位(01) |
| **轻量级锁** | 指向栈中锁记录的指针 | 锁标志位(00) |
| **重量级锁** | 指向互斥量(Monitor)的指针 | 锁标志位(10) |
| **GC 标记** | 空 | 锁标志位(11) |

可以看到，Mark Word 在不同锁状态下，用同一块内存存储了完全不同的信息。这种设计被称为"无状态设计"——JVM 根据锁标志位判断当前处于哪种状态，然后解释 Mark Word 的内容。

## 无锁状态

对象刚创建时，处于无锁状态。此时 Mark Word 存储：

```
25 bits: hashcode | 4 bits: age | 1 bit:偏向标志(0) | 2 bits:锁标志(01)
```

- **hashcode**：对象的 identity hash code（懒加载，调用 `hashCode()` 时计算）
- **age**：对象在 Minor GC 中存活次数，达到阈值后进入老年代
- **偏向标志**：0 表示未偏向
- **锁标志**：01 表示无锁或轻量级锁（需结合偏向标志判断）

```java
public class MarkWordDemo {
    private int value = 42;
    
    public static void main(String[] args) {
        Object obj = new Object();
        System.out.println("无锁状态: " + Integer.toBinaryString(
            System.identityHashCode(obj)));
        // 调用 hashCode 后，Mark Word 会存储哈希码
        obj.hashCode();
    }
}
```

## 偏向锁状态

当启用了偏向锁（`-XX:+UseBiasedLocking`，JDK 6+ 默认开启），对象第一次被线程获取时，会进入偏向锁状态：

```
54 bits: 线程ID | 2 bits: Epoch | 4 bits: age | 1 bit:偏向标志(1) | 2 bits:锁标志(01)
```

- **线程 ID**：持有偏向锁的线程 ID
- **Epoch**：偏向时间戳，用于批量重偏向
- **偏向标志**：1 表示已偏向
- **锁标志**：01 表示偏向锁

偏向锁的核心思想是：**如果一个线程获取了对象的锁，后续同一个线程再次进入同步块，不需要任何同步操作**。这是因为 Mark Word 中已经记录了锁的持有者。

```java
public class BiasedLockDemo {
    public synchronized void doSomething() {
        // 第一次获取偏向锁
    }
    
    public synchronized void doOtherThing() {
        // 同一个线程再次进入，不需要 CAS 操作
    }
}
```

### 偏向锁的撤销

偏向锁虽好，但并非没有代价。当有其他线程尝试获取一个偏向锁的对象时，JVM 需要"撤销"偏向锁。这个过程发生在**安全点（Safe Point）**：

```java
// 安全点撤销偏向锁的简化流程
void revoke_bias(Object obj, Thread requesting_thread) {
    // 1. 确保请求线程不在安全点执行
    if (requesting_thread.is_at_safe_point()) {
        // 2. 将 Mark Word 恢复到无锁状态
        obj.mark_word = unlocked_mark;
        // 3. 其他线程可以重新竞争
    } else {
        // 使用原子操作直接撤销
        obj.mark_word.compare_and_set(
            biased_mark(requesting_thread),
            unlocked_mark
        );
    }
}
```

安全点是 JVM 生成的字节码位置，GC 、安全点检查等都在安全点进行。在安全点撤销偏向锁需要等待所有线程到达安全点，这可能导致"偏向锁撤销的停顿"。

## 轻量级锁状态

当有多个线程竞争同一把锁，但竞争程度不激烈时，JVM 会使用轻量级锁。轻量级锁的核心是 **CAS 操作 + 自旋**。

### 加锁过程

```java
// 轻量级锁加锁（在线程栈中创建 Lock Record）
void lock(Object obj) {
    // 1. 在当前线程的栈帧中创建 Lock Record
    BasicObjectLock* lock_record = new BasicObjectLock();
    lock_record->set_obj(obj);
    
    // 2. 将 Mark Word 复制到 Lock Record 的 Displaced Header
    lock_record->set_displaced_header(obj.mark_word());
    
    // 3. 使用 CAS 尝试将 Mark Word 替换为指向 Lock Record 的指针
    if (obj.mark_word().compare_and_set(
            obj.original_mark(),     // 期望值：无锁状态的 Mark Word
            lock_record)) {          // 更新值：指向 Lock Record 的指针
        // CAS 成功，获取轻量级锁
        return;
    }
    
    // 4. CAS 失败，说明有竞争，进入自旋等待
    if (current_thread.is_owned_by(other_thread)) {
        // 当前线程是锁的持有者（可重入）
        // 在 Displaced Header 中记录重入次数
        lock_record->set_displaced_header(
            markOopDesc::unused_mark()
        );
    } else {
        // 自旋等待或膨胀为重量级锁
        self_spin();
    }
}
```

加锁后，Mark Word 的结构变为：

```
62 bits: 指向栈中锁记录的指针 | 2 bits: 锁标志(00)
```

### 解锁过程

```java
void unlock(Object obj) {
    // 1. 获取栈帧中的 Lock Record
    BasicObjectLock* lock_record = current_thread.last_frame().lock();
    
    // 2. 使用 CAS 将 Displaced Header 恢复到 Mark Word
    if (lock_record->displaced_header().is_neutral()) {
        // 非重入：直接将 Displaced Header 写回
        obj.mark_word.compare_and_set(
            lock_record,           // 期望值：指向 Lock Record 的指针
            lock_record->displaced_header()  // 更新值：原始的 Mark Word
        );
    } else {
        // 重入：减少重入计数，不释放锁
    }
}
```

## 重量级锁状态

当轻量级锁的自旋次数超过阈值，或者有大量线程竞争同一把锁时，轻量级锁会**膨胀**为重量级锁。

重量级锁依赖操作系统底层的 Mutex 实现，不需要自旋消耗 CPU，但会导致线程阻塞和唤醒，存在用户态和内核态的切换开销。

膨胀后的 Mark Word 结构：

```
62 bits: 指向互斥量(Monitor)的指针 | 2 bits: 锁标志(10)
```

## Monitor：重量级锁的实体

`Monitor`（管程）是重量级锁的具体实现，每个对象都关联一个 Monitor：

```cpp
// HotSpot Monitor 结构（简化版）
class ObjectMonitor {
    volatile _owner = NULL;        // 持有锁的线程
    volatile _WaitSet = NULL;      // 等待队列（调用 wait() 的线程）
    volatile _EntryList = NULL;    // 阻塞队列（等待获取锁的线程）
    volatile _Recursion = 0;       // 重入次数
    
    volatile _cxq = NULL;         // 竞争队列（单向链表）
    volatile _succ = NULL;         // 继承锁的候选线程
}
```

Monitor 的等待机制将在后续章节详细讨论。

## 锁标志位：判断锁状态的关键

锁标志位（Lock Flags）是 Mark Word 的最后两位，是判断对象当前锁状态的最终依据：

| 锁标志位 | 锁状态 | 说明 |
|---------|--------|------|
| **01** | 无锁 / 偏向锁 | 需要结合偏向标志位区分 |
| **00** | 轻量级锁 | CAS 加锁，Mark Word 指向栈帧 |
| **10** | 重量级锁 | Monitor 加锁，Mark Word 指向 Monitor |
| **11** | GC 标记 | 垃圾回收时使用 |

## 锁状态的转换流程

完整的锁状态转换如下：

```
无锁状态
    │
    │ 偏向锁启用 + 首次线程获取
    ▼
偏向锁状态 ──────────────────┐
    │                       │
    │ 其他线程竞争（安全点）   │ 其他线程竞争（非安全点）
    ▼                       ▼
    ▼ 无锁状态 ◄─────────────┘
    │                       ▲
    │                       │ 撤销偏向
    │ 轻量级锁（CAS）        │
    │                       │
    ▼ 自旋超过阈值           │
    │ 大量线程竞争           │
    ▼                       │
重量级锁 ◄──────────────────┘
    │
    │ 释放锁
    ▼
无锁状态（也可能保持偏向）
```

需要注意的是：**锁的升级是单向的**，从偏向锁到轻量级锁再到重量级锁，这个过程不可逆。这是 JVM 的设计决策，目的是减少加锁的开销。

## 实战：用代码验证 Mark Word

可以使用 JDK 自带的工具查看对象头信息：

```java
import sun.misc.Unsafe;
import java.lang.reflect.Field;

public class MarkWordPrinter {
    
    private static Unsafe unsafe;
    
    static {
        try {
            Field f = Unsafe.class.getDeclaredField("theUnsafe");
            f.setAccessible(true);
            unsafe = (Unsafe) f.get(null);
        } catch (Exception e) {
            throw new RuntimeException(e);
        }
    }
    
    public static void main(String[] args) throws Exception {
        Object obj = new Object();
        System.out.println("对象大小: " + unsafe.objectFieldOffset(
            Object.class.getDeclaredField("markWord")) + " bytes");
        
        // 获取 Mark Word（64位系统，前56位）
        long markWord = unsafe.getLong(obj, 0L);
        System.out.println("Mark Word: " + Long.toBinaryString(markWord));
        System.out.println("锁状态: 无锁或偏向锁（最后3位为 101）");
    }
}
```

## 总结

`synchronized` 锁的实际上是**对象的 Mark Word**。JVM 通过 Mark Word 中的锁标志位和状态信息，动态地选择最适合当前场景的加锁方式：

- **偏向锁**：消除同一线程的同步开销
- **轻量级锁**：用 CAS + 自旋避免线程阻塞
- **重量级锁**：依赖 OS Mutex，避免大量自旋消耗 CPU

理解 Mark Word 的结构，就理解了 synchronized 锁演进的本质。你有没有想过，为什么 JDK 6 要引入锁升级机制？答案就在 Mark Word 的巧妙设计中。
