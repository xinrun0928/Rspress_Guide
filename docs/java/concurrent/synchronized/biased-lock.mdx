# 偏向锁的获取、撤销与批量重偏向

JDK 6 引入了一项革命性的优化：**偏向锁**（Biased Locking）。

它的核心思想简单到令人发指——「如果一个线程反复进入同一把锁，干嘛还要每次都竞争？」

但这项优化在 JDK 15 后被废弃了。

今天，我们来聊聊偏向锁的设计原理、撤销机制，以及为什么它最终被放弃。

## 偏向锁的设计初衷

在没有偏向锁的时代，即使只有一个线程反复访问同步块，JVM 也需要进行锁竞争：

```
┌─────────────────────────────────────────────────────────┐
│              无偏向锁时代的每次竞争                        │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  线程 A 获取锁 ──> 执行 ──> 释放锁                        │
│       ↑______________|______________↓                   │
│                                                          │
│  线程 A 再次获取锁：                                       │
│       竞争锁 ──> 获取 ─> 执行 ──> 释放 ──> 竞争           │
│                                                          │
│  即使是同一个线程，也要走完整的 CAS 流程                   │
└─────────────────────────────────────────────────────────┘
```

偏向锁的目标：**如果一个锁始终只被一个线程访问，就把这个锁「偏向」给这个线程，后续访问直接跳过竞争**。

## Mark Word 与偏向锁

理解偏向锁，需要先理解对象的内存布局。在 64 位 JVM 中，对象头（Mark Word）有 64 位：

```java
// 64 位 Mark Word 结构（无偏向锁状态）
┌──────────────────────────────────────┐
│  无锁状态                              │
│  25 位: 对象哈希码                      │
│  4 位: 对象分代年龄                     │
│  1 位: 偏向锁位 (0)                    │
│  2 位: 锁状态 (01)                    │
└──────────────────────────────────────┘

┌──────────────────────────────────────┐
│  偏向锁状态                            │
│  54 位: 线程 ID                        │
│  2 位: epoch                           │
│  4 位: 对象分代年龄                     │
│  1 位: 偏向锁位 (1)                    │
│  2 位: 锁状态 (01)                    │
└──────────────────────────────────────┘
```

**偏向锁开启后，对象头的最后两位会标记为 `01`**。

## 偏向锁的获取

### 场景：线程 A 首次进入同步块

```java
public class BiasedLockDemo {
    public static void main(String[] args) {
        Object lock = new Object();

        synchronized (lock) {  // 线程 A 首次进入
            // 临界区代码
        }
    }
}
```

执行过程：

```
┌─────────────────────────────────────────────────────────┐
│           偏向锁获取流程（线程 A 首次进入）               │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  1. 检查对象头：                                         │
│     - 偏向锁位 = 0（未偏向）                            │
│     - 锁状态 = 01（轻量级锁）                           │
│              ↓                                          │
│  2. 使用 CAS 将 Mark Word 设置为：                      │
│     - 线程 ID = 线程 A 的 ID                            │
│     - epoch = 0                                        │
│     - 偏向锁位 = 1                                      │
│     - 锁状态 = 01                                      │
│              ↓                                          │
│  3. 偏向成功！线程 A 获得偏向锁                          │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

### 代码层面

```java
// HotSpot 伪代码：偏向锁获取
if (mark->has_bias_pattern()) {
    // 当前线程 ID
    Thread* current = Thread::current();
    // 对象头中记录的偏向线程
    mark = mark->Bias();
    // 相同线程，直接进入临界区
    if (mark->Bias() == current) {
        return true;
    }
    // 不同线程，尝试 CAS 偏向
    if (mark->Age() != 0) {
        // 有年龄，进入批量重偏向逻辑
        return attempt_biased_revoke();
    }
    // CAS 替换线程 ID
    if (mark->CasBias(current)) {
        return true;
    }
}
```

## 偏向锁的撤销

当其他线程尝试获取偏向锁时，偏向锁会被撤销。撤销有两种类型：

### 场景一：安全点撤销

当有其他线程尝试竞争偏向锁时，JVM 会在「安全点」（Safe Point）撤销偏向锁。

```java
public class BiasedLockRevoke {
    public static void main(String[] args) throws InterruptedException {
        Object lock = new Object();

        // 线程 A 获取偏向锁
        synchronized (lock) {
            System.out.println("Thread A got biased lock");
        }
        // 线程 A 退出同步块，偏向锁仍保留

        // 线程 B 尝试获取锁
        Thread threadB = new Thread(() -> {
            synchronized (lock) {
                System.out.println("Thread B got lock");
            }
        });
        threadB.start();
    }
}
```

执行流程：

```
┌─────────────────────────────────────────────────────────┐
│           偏向锁撤销流程（线程 B 竞争）                   │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  线程 A 持有偏向锁，线程 B 请求同一把锁：                  │
│                                                          │
│  线程 B: "我想获取锁"                                    │
│              ↓                                          │
│  JVM: "等一下，让线程 A 到达安全点"                       │
│              ↓                                          │
│  [线程 A 到达安全点，挂起]                               │
│              ↓                                          │
│  JVM: "好了，我来撤销偏向锁"                             │
│              ↓                                          │
│  - 检查线程 A 是否在临界区内                             │
│  - 如果不在：将对象头恢复为无锁状态                      │
│  - 如果在：升级为轻量级锁                                │
│              ↓                                          │
│  线程 B: 尝试 CAS 获取轻量级锁                          │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

### 场景二：调用 hashCode() 撤销

```java
Object lock = new Object();
System.out.println(lock.hashCode());  // 计算哈希码
// 此时对象已经无法偏向（因为 Mark Word 被哈希码占用）
```

**哈希码占用了 Mark Word 的大部分空间，没有空间存储偏向线程信息**。

## 批量重偏向（Bulk Rebiasing）

当一个类的对象大量被同一个线程偏向，然后被另一个线程竞争时，JVM 不会逐个撤销，而是批量处理：

```java
// 类 A 的对象被批量创建，全部偏向线程 A
for (int i = 0; i < 1000; i++) {
    locks[i] = new Object();
    synchronized (locks[i]) { /* 线程 A */ }
}

// 然后线程 B 竞争这些锁
for (int i = 0; i < 1000; i++) {
    new Thread(() -> {
        synchronized (locks[i]) { /* 线程 B */ }
    }).start();
}
```

### 批量重偏向的触发条件

| 条件 | 值 |
|---|---|
| 偏向锁批量撤销阈值 | `BiasedLockingStartupDelay = 4000ms`（JVM 启动 4 秒后生效） |
| 批量重偏向阈值 | `BiasedLockingBulkRebiasThreshold = 20` |
| 批量撤销阈值 | `BiasedLockingBulkRevokeThreshold = 40` |

### 批量重偏向的过程

```
┌─────────────────────────────────────────────────────────┐
│           批量重偏向示意图                                │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  类 A 的对象被偏向线程 A，然后被线程 B 竞争：              │
│                                                          │
│  epoch 机制：                                            │
│  ┌─────────────────────────────────────────────┐        │
│  │ Class A 中有一个共享的 epoch 计数器           │        │
│  │ 每次批量重偏向时，epoch++                     │        │
│  │ 对象头中存储的是自己的 epoch                   │        │
│  └─────────────────────────────────────────────┘        │
│                                                          │
│  线程 B 到达安全点：                                      │
│  1. 将 Class A 的 epoch++                               │
│  2. 遍历所有偏向线程 A 的对象：                          │
│     - 如果对象在安全点：重新偏向到线程 B                  │
│     - 如果对象不在安全点：等到安全点再处理               │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

## 批量撤销（Bulk Revoke）

当竞争过于激烈时，JVM 会批量撤销整个类的偏向状态：

```
批量重偏向阈值: 20 次
批量撤销阈值: 40 次

竞争次数分布：
0-20: 正常竞争，每次重偏向
20-40: 批量重偏向
>40:  批量撤销，后续对象不再偏向
```

## 偏向锁的废弃

JDK 15 废弃了偏向锁，原因包括：

### 1. 撤销成本高

```java
// 偏向锁撤销需要挂起线程
// 这在低延迟场景下是不可接受的
while (true) {
    // 高频同步操作
    synchronized (this) {
        process();
    }
}
```

### 2. 安全问题

偏向锁的撤销发生在安全点，如果应用长时间停顿在安全点，会导致延迟毛刺。

### 3. 维护成本

偏向锁的代码路径复杂，bug 难以排查，且收益在现代 CPU 上越来越小。

## 总结

偏向锁的核心机制：

1. **获取**：首次进入时，CAS 将线程 ID 写入对象头
2. **维持**：同一线程再次进入，直接跳过竞争
3. **撤销**：其他线程竞争时，在安全点撤销
4. **批量重偏向**：当竞争模式固定时，批量调整偏向目标
5. **批量撤销**：竞争过于激烈时，禁用整个类的偏向

| 概念 | 说明 |
|---|---|
| 偏向锁状态 | `01`，但偏向位 = 1 |
| 撤销触发 | 其他线程竞争 / 调用 hashCode() / GC |
| 撤销代价 | 需要到达安全点，线程暂停 |
| 批量重偏向 | 同一类 20+ 次竞争后，批量调整 |
| 批量撤销 | 同一类 40+ 次竞争后，禁用偏向 |

虽然偏向锁在 JDK 15 后被废弃，但它代表了 JVM 对锁优化的经典思路：**消除不必要的竞争**。
