# 偏向锁的获取、撤销与批量重偏向

偏向锁是 JDK 6 引入的重量级优化，它的核心思想简单而优雅——如果一个同步块从未被多个线程竞争过，为什么要让后续的每次进入都要做同步操作呢？偏向锁让第一个获取锁的线程"记住"自己的身份，后续再进入时无需任何原子操作。但这把双刃剑也有它的复杂性：偏向锁的获取、撤销、批量重偏向，每个环节都藏着精妙的设计。

## 偏向锁的获取：一次 CAS，永久偏向

当一个线程首次执行 `synchronized(obj)` 时，JVM 会尝试将对象设置为偏向锁状态。这个过程只需要一次 CAS 操作：

```java
// 偏向锁获取的简化流程
void bias_revive(Object obj, Thread requesting_thread) {
    // 1. 获取对象当前的 Mark Word（应该是无锁状态）
    markOop mark = obj.mark_word();
    
    // 2. 检查是否可以偏向
    //    - 锁标志位必须是 01（无锁）
    //    - 偏向标志位必须是 0（未偏向）
    if (mark.is_biased() && !mark.has_biased_locker()) {
        // 3. 构造偏向锁的 Mark Word
        //    - 线程ID：requesting_thread 的线程ID
        //    - Epoch：类的 Epoch 值
        //    - 偏向标志：1
        //    - 锁标志：01
        biasedMark = mark.with_biased_info(requesting_thread, epoch);
        
        // 4. CAS 设置偏向锁
        //    这是唯一一次原子操作！
        if (obj.mark_word.compare_and_set(mark, biasedMark)) {
            // 偏向成功，线程永久持有偏向
            requesting_thread.increment_biased_lock_count();
            return ACQUIRED;
        }
        
        // CAS 失败，说明有竞争，撤销偏向
        revoke_and_retry(obj);
    }
}
```

偏向锁的核心就是这次 **CAS**。一旦成功，Mark Word 中就记录了当前线程的 ID，后续该线程进入同步块时，只需要比较 Mark Word 中的线程 ID 是否是自己。

## 偏向锁的快速路径：零开销同步

偏向锁获取后，线程再次进入同步块的路径变得极其高效：

```java
// 偏向锁的快速路径（伪代码）
void fast_path_enter(Object obj) {
    markOop mark = obj.mark_word();
    
    // 检查：是否是我偏向的锁？
    if (mark.has_biased_locker() && 
        mark.biased_thread() == Thread.currentThread()) {
        // 完全是零开销！
        // 不需要任何 CAS 或同步操作
        return ACQUIRED;
    }
    
    // 不是我的偏向锁，走慢速路径
    slow_path_enter(obj);
}
```

这种"零开销"是偏向锁最大的优势。在锁竞争很少的场景下，性能提升显著。

## 偏向锁的撤销：竞争的代价

偏向锁虽好，但一旦有其他线程尝试获取，JVM 必须撤销偏向。这个过程比获取复杂得多。

### 为什么要撤销偏向？

考虑这个场景：

```java
Thread A:  // 持有偏向锁
    synchronized (obj) {
        // 业务操作
    }

Thread B:  // 尝试获取锁
    synchronized (obj) {  // 偏向锁被撤销
        // ...
    }
```

Thread B 不能简单地"覆盖" Thread A 的偏向，因为这样会导致数据不一致。JVM 必须确保 Thread A 完全释放锁后，Thread B 才能获取。

### 撤销的时机：安全点

偏向锁的撤销发生在**安全点（Safe Point）**。安全点是 JVM 生成的字节码位置，此时所有 Java 线程都已暂停，JVM 可以安全地执行一些需要"全局视角"的操作。

```java
// 安全点撤销偏向锁的完整流程
void revoke_bias_at_safe_point(Object obj) {
    // 1. JVM 触发 Stop The World，所有线程停止
    
    // 2. 判断持有偏向锁的线程状态
    Thread biased_thread = obj.mark_word.biased_thread();
    
    if (biased_thread.is_executing_sync()) {
        // 线程正在同步块中
        // 升级为轻量级锁
        upgrade_to_thin_lock(obj);
    } else {
        // 线程不在同步块中（已释放锁）
        // 直接恢复为无锁状态
        obj.mark_word = UNLOCKED_MARK;
    }
    
    // 3. JVM 恢复所有线程
    // 4. Thread B 尝试获取锁（现在是轻量级锁）
}
```

**安全点撤销的代价**：需要 Stop The World，所有 Java 线程都会停顿。对于大型应用，这可能导致数百毫秒的停顿。这是偏向锁的致命缺点，也是 JDK 15 废弃偏向锁的原因之一。

### 偏向撤销的四种情况

```java
enum BiasRevokeReason {
    // 1. 同类竞争：同一个类的不同实例被不同线程使用
    CLASS_DEMOTION,
    
    // 2. 批量操作：HashMap 等批量操作触发
    BULK_REBIAS,
    
    // 3. 批量撤销：大量实例被多个线程竞争
    BULK_REVOKE,
    
    // 4. 哈希码请求：调用 hashCode() 需要无锁状态
    HASH_CODE_REQUEST
}
```

### 哈希码与偏向锁的冲突

这是一个容易被忽视的点：**对象获取哈希码后，就无法进入偏向锁状态了**。

```java
Object obj = new Object();
obj.hashCode();  // 计算并缓存哈希码

// 此时 Mark Word 结构：
// unused:25 | hashcode:31 | age:4 | biased_lock:0 | lock:01
//                                        ↑ 偏向标志为0
// 对象无法再偏向！
```

原因：偏向锁状态下，Mark Word 存储的是线程 ID，没有空间存储哈希码。而 `hashCode()` 需要读取对象的哈希码。

```java
// hash() 方法的实现
public int hashCode() {
    // 获取 Mark Word
    int hash = mark_word.identity_hash();
    if (hash == 0) {
        // 懒加载：第一次调用时计算
        hash = compute_identity_hash();
        mark_word.set_hash(hash);
    }
    return hash;
}
```

## 批量重偏向：批量处理的艺术

JVM 对偏向锁的撤销和重偏向做了批量优化。当一个类的多个实例被反复撤销偏向时，JVM 会"批量重偏向"，而不是逐个处理。

### Epoch：偏向锁的时间戳

Epoch 是偏向锁的"版本号"，存储在两处：

1. **对象头**：每个对象的 Mark Word 中
2. **Class 元数据**：每个类的 `class` 对象中

```
┌─────────────────────────────────────────────────────────┐
│                      Class 元数据                        │
│  ┌─────────────────────────────────────────────────┐   │
│  │  biased_lock_statistics                          │   │
│  │  ├── total_biased: 1000  (偏向次数)              │   │
│  │  ├── revoked_biased: 950  (撤销次数)              │   │
│  │  └── last_epoch: 2        (当前 Epoch)            │   │
│  └─────────────────────────────────────────────────┘   │
│                                                         │
│                      对象头                              │
│  ┌─────────────────────────────────────────────────┐   │
│  │  Mark Word                                       │   │
│  │  ├── thread_id: xxx                              │   │
│  │  ├── epoch: 2        ◄── 与 Class 的 epoch 对比   │   │
│  │  └── biased_lock: 1                              │   │
│  └─────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
```

### 批量重偏向的触发条件

```java
// 批量重偏向的触发条件（JVM 参数可调）
if (revoked_biased > (total_biased * BiasedLockingBulkRebiasThreshold)) {
    // 撤销次数超过阈值（默认20），触发批量重偏向
    
    // 遍历类的所有实例
    for (Object instance : class.instances()) {
        // 如果对象处于偏向锁状态
        if (instance.mark_word.is_biased()) {
            // 直接更新 epoch，不需要遍历所有线程的栈
            instance.mark_word.set_epoch(class.last_epoch);
        }
    }
}
```

### 批量重偏向的原理

当某个类的实例在**多个线程间被反复转移**时（如生产者-消费者场景中的任务对象），批量重偏向可以避免大量的安全点停顿：

```java
// 场景：生产者-消费者模式
public class TaskQueue {
    Queue<Task> tasks = new LinkedBlockingQueue<>();
    
    public void put(Task t) {
        synchronized (t) {  // 线程A：生产者
            // ...
        }
    }
    
    public Task get() {
        synchronized (task) {  // 线程B：消费者
            // ...
        }
    }
}

// 未优化前：每个 task 对象都会触发偏向锁撤销
// 优化后：批量重偏向后，task 直接偏向消费者线程
```

### 批量撤销

如果撤销次数过多，JVM 会认为这个类"不适合偏向"，触发**批量撤销**：

```java
if (revoked_biased > (total_biased * BiasedLockingBulkRevokeThreshold)) {
    // 批量撤销：该类的所有实例都不能再偏向
    
    // 遍历类的所有实例
    for (Object instance : class.instances()) {
        // 将对象恢复为无锁状态
        instance.mark_word = instance.mark_word.remove_biased_info();
    }
    
    // 更新 Class：禁用该类的偏向锁
    class.set_biased_lock_enabled(false);
}
```

## JVM 参数调优

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `-XX:+UseBiasedLocking` | true | 是否启用偏向锁（JDK 15+ 默认 false） |
| `-XX:BiasedLockingStartupDelay` | 4000ms | JVM 启动后延迟启用偏向锁的时间 |
| `-XX:BiasedLockingBulkRebiasThreshold` | 20 | 批量重偏向阈值 |
| `-XX:BiasedLockingBulkRevokeThreshold` | 40 | 批量撤销阈值 |
| `-XX:BiasedLockingDecayTime` | 25000ms | 衰减时间，撤销计数重置的间隔 |

```bash
# 禁用偏向锁（高并发场景推荐）
java -XX:-UseBiasedLocking -jar app.jar

# 调整批量重偏向阈值
java -XX:BiasedLockingBulkRebiasThreshold=50 -jar app.jar

# 延迟偏向锁启用
java -XX:BiasedLockingStartupDelay=10000 -jar app.jar
```

## 实战：观察偏向锁行为

```java
import java.util.concurrent.TimeUnit;

public class BiasedLockDemo {
    
    public static void main(String[] args) throws Exception {
        Object obj = new Object();
        
        System.out.println("=== 测试偏向锁获取 ===");
        synchronized (obj) {
            System.out.println("偏向锁已获取（快速路径，无 CAS）");
        }
        
        System.out.println("\n=== 等待 JVM 偏向锁启用 ===");
        TimeUnit.SECONDS.sleep(5);
        
        System.out.println("\n=== 同一线程再次获取 ===");
        synchronized (obj) {
            System.out.println("偏向锁命中（零开销）");
        }
        
        System.out.println("\n=== 制造竞争，观察撤销 ===");
        Thread competitor = new Thread(() -> {
            synchronized (obj) {
                System.out.println("竞争对手获取了锁");
            }
        });
        
        competitor.start();
        competitor.join();
        
        System.out.println("\n=== 撤销后再次获取 ===");
        synchronized (obj) {
            System.out.println("已升级为轻量级锁");
        }
    }
}
```

## 总结

偏向锁是 JVM 为低竞争场景设计的优化，它用一次 CAS 操作换取后续的零开销同步。但这把双刃剑也有其复杂性：

- **获取**：只需要一次 CAS，之后完全零开销
- **撤销**：需要在安全点执行，可能导致 STW 停顿
- **批量重偏向**：避免重复处理同一类对象的偏向撤销
- **哈希码冲突**：一旦对象调用 hashCode()，就无法偏向

从 JDK 15 开始，偏向锁已被废弃。JVM 团队认为在现代高并发场景下，偏向锁的开销大于收益。但理解其原理，仍然有助于我们深入理解 JVM 的锁机制。你有没有想过，在什么场景下偏向锁仍然有价值？
