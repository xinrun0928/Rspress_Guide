# 锁消除与锁粗化优化

你已经知道 synchronized 有四种状态：无锁、偏向锁、轻量级锁、重量级锁。

但 JVM 还有两个「隐藏」优化：**锁消除**和**锁粗化**。

这两个优化由 JIT 编译器在运行时自动完成，你写的 synchronized 代码，可能比想象中更快。

## 锁消除（Lock Elision）

### 什么是锁消除？

锁消除是 JIT 编译器分析代码后，发现**某个锁不可能被多线程竞争**，于是把这个锁完全消除掉。

```java
public class LockElisionDemo {
    // StringBuffer 是线程安全的，但在这个方法中...
    public String concat(String a, String b) {
        StringBuffer sb = new StringBuffer();
        sb.append(a);
        sb.append(b);
        return sb.toString();
    }
}
```

表面上看，`sb.append()` 使用了 synchronized，但实际上：

- `sb` 是方法内的局部变量
- 局部变量不可能被其他线程访问
- **JIT 会消除这个「无效」的锁**

### 逃逸分析（Escape Analysis）

锁消除的前提是**逃逸分析**。

```java
public class EscapeAnalysis {
    // 情况一：不逃逸
    public void method1() {
        Object obj = new Object();  // 局部变量
        synchronized (obj) {
            // obj 不会逃逸出方法
        }
    }

    // 情况二：逃逸
    public Object method2() {
        Object obj = new Object();
        synchronized (obj) {
            return obj;  // obj 逃逸了
        }
    }

    // 情况三：逃逸
    static Object shared;
    public void method3() {
        shared = new Object();  // 逃逸到堆
        synchronized (shared) {
            // 不能消除锁
        }
    }
}
```

逃逸分析判断对象的引用是否会「逃逸」出当前线程的执行范围：

```
┌─────────────────────────────────────────────────────────┐
│           逃逸分析：对象的引用去向                        │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  不逃逸（可以锁消除）：                                   │
│  ├── 仅作为局部变量                                       │
│  ├── 作为方法返回值                                      │
│  ├── 作为参数传递但未存储                                 │
│  └── 仅在当前线程内使用                                   │
│                                                          │
│  逃逸（不能锁消除）：                                     │
│  ├── 作为静态变量                                        │
│  ├── 作为其他对象的字段                                   │
│  ├── 作为集合元素                                        │
│  └── 作为类的 public/protected 成员                      │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

### 逃逸分析的源码

```cpp
// HotSpot 的逃逸分析相关代码（简化）
bool EscapeAnalysis::compute_escape(CallGenerator* gen) {
    for (Graph& g = gen->graph(); !g.is_done(); g.next()) {
        Node* n = g.current();

        if (n->is_Allocate()) {
            // 检查对象是否逃逸
            PointsToNode* alloc = PointsTo(n);
            if (!alloc->escapes_to_global()) {
                // 不逃逸到全局，可以进行锁消除
                return true;
            }
        }
    }
    return false;
}
```

### 实战验证

```java
public class LockElisionTest {
    // 开启逃逸分析和锁消除：默认开启
    // 关闭：-XX:-DoEscapeAnalysis

    public static void main(String[] args) {
        long start = System.currentTimeMillis();
        for (int i = 0; i < 10_000_000; i++) {
            lockElisionTest();
        }
        long end = System.currentTimeMillis();
        System.out.println("Time: " + (end - start) + "ms");
    }

    private static void lockElisionTest() {
        StringBuffer sb = new StringBuffer();
        sb.append("hello");
        sb.append("world");
    }
}
```

开启逃逸分析后，`StringBuffer.append()` 的锁会被消除。

## 锁粗化（Lock Coarsening）

### 什么是锁粗化？

JIT 发现多次连续的 synchronized 操作，实际上可以合并成一次更大的同步块。

```java
// 优化前：多次加锁/解锁
public String concat(String a, String b, String c) {
    StringBuffer sb = new StringBuffer();
    synchronized (sb) {
        sb.append(a);
    }
    synchronized (sb) {
        sb.append(b);
    }
    synchronized (sb) {
        sb.append(c);
    }
    return sb.toString();
}

// 优化后：合并为一次加锁
public String concat(String a, String b, String c) {
    StringBuffer sb = new StringBuffer();
    synchronized (sb) {
        sb.append(a);
        sb.append(b);
        sb.append(c);
    }
    return sb.toString();
}
```

### 为什么需要锁粗化？

每一次加锁和解锁都有开销：

| 操作 | 代价 |
|---|---|
| 加锁 | 检查锁状态、CAS（轻量级） |
| 解锁 | 更新锁状态、CAS |
| 锁竞争 | 等待队列操作（重量级） |

如果多个操作使用同一把锁，合并后只加锁和解锁一次，性能更好。

### 锁粗化的触发条件

```cpp
// HotSpot 的锁粗化逻辑（简化）
bool LockCoarsening::should_coarsen(Graph& g, Node* lock1, Node* lock2) {
    // 检查两个锁是否相同
    if (lock1->obj_node() != lock2->obj_node()) {
        return false;  // 不是同一把锁，不能合并
    }

    // 检查锁之间是否有空隙
    if (!are_adjacent_locks(lock1, lock2)) {
        return false;  // 有其他代码，不能合并
    }

    // 检查粗化次数是否超限
    if (++coarsen_count > MaxLockCoarsening) {
        return false;  // 粗化太多，可能影响并发
    }

    return true;
}
```

### 锁粗化的阈值

```java
// JVM 参数控制
-XX:+UseLockCoarsening   // 开启锁粗化（默认）
-XX:MaxLockCoarsening=N  // 最大粗化次数（默认 5）
```

### 实际场景

```java
// 场景一：StringBuilder 的 append
// StringBuilder 是非线程安全的，JVM 不会加锁
// 但如果是 StringBuffer，JIT 会进行锁粗化

StringBuffer sb = new StringBuffer();
sb.append("a");
sb.append("b");
sb.append("c");
sb.append("d");
// 可能会被粗化为：
// synchronized (sb) {
//     sb.append("a");
//     sb.append("b");
//     sb.append("c");
//     sb.append("d");
// }

// 场景二：循环中的锁
for (int i = 0; i < 1000; i++) {
    synchronized (this) {
        process(i);
    }
}
// 可能被粗化为：
// synchronized (this) {
//     for (int i = 0; i < 1000; i++) {
//         process(i);
//     }
// }
```

## 两种优化的关系

```
┌─────────────────────────────────────────────────────────┐
│           锁消除 vs 锁粗化                               │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  锁消除（Lock Elision）：                                │
│  - 目标：完全消除不必要的锁                              │
│  - 依据：逃逸分析证明对象不逃逸                          │
│  - 结果：零开销                                          │
│                                                          │
│  锁粗化（Lock Coarsening）：                             │
│  - 目标：减少锁的加锁/解锁次数                          │
│  - 依据：多次使用同一把锁的相邻操作                      │
│  - 结果：降低锁开销                                      │
│                                                          │
│  执行顺序：                                              │
│  1. 先逃逸分析 → 可能锁消除                             │
│  2. 对剩余的锁 → 尝试锁粗化                             │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

## 实际影响

### StringBuilder vs StringBuffer

```java
// StringBuilder（非线程安全）
StringBuilder sb = new StringBuilder();
sb.append("hello");
sb.append("world");
// 不会有任何锁操作

// StringBuffer（线程安全）
StringBuffer sb = new StringBuffer();
sb.append("hello");
sb.append("world");
// 可能有锁操作（但会被粗化）
```

**结论**：在单线程环境下，`StringBuffer` 和 `StringBuilder` 性能差异不大，因为锁会被粗化或消除。

### 集合类的选择

```java
// ArrayList（非线程安全）
ArrayList<String> list = new ArrayList<>();
// 无锁操作

// Vector（线程安全）
Vector<String> list = new Vector<>();
// 有锁操作，但会被优化

// 最佳实践
// 单线程：用 ArrayList
// 多线程：用 Collections.synchronizedList() 或 CopyOnWriteArrayList
```

## 面试追问

### Q1：锁消除安全吗？

安全。锁消除是 JIT 编译器基于逃逸分析的结果做出的优化。如果对象真的不逃逸，消除锁后程序行为不会改变。

### Q2：锁粗化会影响并发吗？

不会明显影响。锁粗化只是合并了相邻的同步块，不会改变锁的持有时间或并发度。如果两个操作之间有真正的数据依赖，JIT 不会进行粗化。

### Q3：如何观察这些优化？

```bash
# 打印编译后的汇编代码
-XX:+UnlockDiagnosticVMOptions
-XX:+PrintAssembly

# 打印 JIT 编译日志
-XX:+PrintCompilation

# 打印锁粗化信息
-XX:+PrintLockCoarsening
```

## 总结

JIT 的两个隐藏优化：

1. **锁消除**：通过逃逸分析，判断锁是否必要，如果对象不逃逸，直接消除锁
2. **锁粗化**：将多个相邻的同步块合并为一个，减少加锁/解锁次数

这两个优化是 JIT 编译器「润物细无声」的性能提升。你写的 synchronized 代码，可能比想象中更高效。

| 优化 | 触发条件 | 效果 |
|---|---|---|
| 锁消除 | 逃逸分析证明对象不逃逸 | 零开销 |
| 锁粗化 | 多次使用同一把锁的相邻操作 | 减少锁操作次数 |

理解这些底层优化，才能写出真正高效的并发代码。
