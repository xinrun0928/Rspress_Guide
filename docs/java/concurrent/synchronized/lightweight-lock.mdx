# 轻量级锁：CAS + 自旋

面试官问：「synchronized 的锁升级过程是什么？」

你回答：「无锁 → 偏向锁 → 轻量级锁 → 重量级锁。」

面试官点点头：「那轻量级锁是怎么工作的？」

你：「用 CAS...然后自旋...」

面试官追问：「具体是怎么用 CAS 的？自旋多少次？」

你沉默了。

今天，我们来彻底理解轻量级锁的原理。

## 为什么需要轻量级锁？

偏向锁解决了「同一线程反复进入同步块」的问题。但如果有多个线程交替进入同步块呢？

```
┌─────────────────────────────────────────────────────────┐
│         偏向锁失效的场景                                  │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  时间线：                                                │
│  T1: 线程 A 获取偏向锁，执行                             │
│  T2: 线程 A 释放锁                                       │
│  T3: 线程 B 获取偏向锁（偏向撤销）                       │
│  T4: 线程 B 释放锁                                       │
│  T5: 线程 A 获取偏向锁（偏向撤销）                       │
│  ...                                                    │
│                                                          │
│  线程 A 和 B 交替进入同步块：                             │
│  每次都要撤销偏向锁，代价很大！                          │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

这种情况叫做「交错竞争」或「多线程交替执行」。偏向锁会频繁撤销，性能反而下降。

**轻量级锁的设计目标**：在多线程交替执行时，避免频繁的线程阻塞和用户态/内核态切换。

## 轻量级锁的获取

### Mark Word 的变化

轻量级锁状态下，Mark Word 存储的是「指向栈中锁记录的指针」：

```java
// 64 位 Mark Word - 轻量级锁状态
┌──────────────────────────────────────┐
│  62 位: 指向栈中锁记录的指针           │
│  2 位: 锁状态 = 00（轻量级锁）         │
└──────────────────────────────────────┘
```

### 获取流程

```java
public class LightweightLockDemo {
    static final Object lock = new Object();

    public static void main(String[] args) {
        synchronized (lock) {
            // 临界区
        }
    }
}
```

线程进入同步块时：

```
┌─────────────────────────────────────────────────────────┐
│           轻量级锁获取流程                                │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  线程 A 进入同步块：                                      │
│                                                          │
│  1. 在线程 A 的栈帧中创建锁记录（Lock Record）            │
│     ┌─────────────────────────────┐                    │
│     │  Lock Record                 │                    │
│     │  ┌─────────────────────────┐│                   │
│     │  │ 对象头的 Mark Word (COPY) ││                   │
│     │  └─────────────────────────┘│                    │
│     │  │ 对象 reference            ││                   │
│     │  └─────────────────────────┘│                   │
│     └─────────────────────────────┘                    │
│              ↓                                           │
│  2. CAS 尝试将对象头的 Mark Word 替换为：                │
│     - 指向当前线程栈帧中锁记录的指针                      │
│     - 锁状态位 = 00                                      │
│              ↓                                           │
│  3. 替换成功：线程 A 获得轻量级锁                         │
│  4. 替换失败：说明有其他线程竞争，升级为重量级锁          │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

### HotSpot 源码解析

```cpp
// interpreterRuntime.cpp 中的获取锁逻辑
IRT_ENTRY_NO_ASYNC(Object*, InterpreterRuntime::monitorenter(JavaThread* thread, BasicObjectLock* elem))
    // elem 是栈帧中的锁记录
    // obj 是被锁对象

    if (UseBiasedLocking) {
        // 尝试偏向锁（如果偏向可用）
        ObjectSynchronizer::fast_enter(h_obj, elem, attempt_biased_locking, THREAD);
    } else {
        // 直接尝试轻量级锁
        ObjectSynchronizer::slow_enter(h_obj, elem, THREAD);
    }
IRT_END

// slow_enter 实现
bool ObjectSynchronizer::slow_enter(Handle obj, BasicObjectLock* lock, JavaThread* thread) {
    markOop mark = obj()->mark();

    if (mark->is_neutral()) {
        // 对象无锁，尝试 CAS 替换
        lock->set_displaced_header(mark);  // 保存原 Mark Word
        if (mark == obj()->cas_set_mark(mark, lock->address())) {
            // CAS 成功，获取轻量级锁
            return true;
        }
    }

    // CAS 失败，对象已被锁定或偏向，升级为重量级锁
    return false;
}
```

## 轻量级锁的释放

```java
synchronized (lock) {
    // 临界区代码
}
// 锁在这里释放
```

线程退出同步块时：

```
┌─────────────────────────────────────────────────────────┐
│           轻量级锁释放流程                                │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  线程 A 释放轻量级锁：                                    │
│                                                          │
│  1. 从当前栈帧中取出锁记录（Lock Record）                 │
│  2. 检查锁记录的 displaced_header：                      │
│     - 如果 Mark Word 指向当前栈帧                        │
│       （说明是轻量级锁持有者）                           │
│              ↓                                           │
│  3. CAS 尝试将对象头的 Mark Word 恢复为：                │
│     - 锁记录中保存的原始 Mark Word                       │
│              ↓                                           │
│  4. 恢复成功：释放轻量级锁成功                            │
│  5. 恢复失败：说明有其他线程在等待，升级为重量级锁        │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

### 释放代码

```cpp
bool ObjectSynchronizer::fast_exit(oop obj, BasicObjectLock* lock, JavaThread* thread) {
    markOop displaced = lock->displaced_header();

    // 获取对象头的当前 Mark Word
    markOop mark = obj->mark();

    if (mark == lock->addr()) {
        // 对象头指向当前锁记录，说明是轻量级锁持有者
        // CAS 恢复原始 Mark Word
        obj->cas_set_mark(displaced);
        return true;
    }

    // CAS 失败，需要膨胀为重量级锁
    return ObjectSynchronizer::inflate(...)->exit(...);
}
```

## 自旋优化

### 为什么需要自旋？

轻量级锁获取失败时，并不立即升级为重量级锁，而是**自旋等待**。

```
┌─────────────────────────────────────────────────────────┐
│           自旋等待示意图                                  │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  线程 A 持有轻量级锁，正在临界区执行                       │
│  线程 B 尝试获取同一把锁：                                │
│                                                          │
│  线程 B: "锁被占用了"                                    │
│          ↓                                              │
│  线程 B: "等一下，也许很快就释放了"（自旋）               │
│          ↓                                              │
│  线程 A 释放锁                                          │
│          ↓                                              │
│  线程 B: "拿到锁了！"（CAS 成功）                        │
│          ↓                                              │
│  线程 B 进入临界区                                       │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

### 自旋的代价

自旋会占用 CPU 时间。如果自旋时间过长或锁持有者一直不释放，自旋就是浪费。

```java
// 自旋参数（JDK 6 之前）
-XX:PreBlockSpin = 10  // 默认自旋 10 次

// JDK 6 之后：自适应自旋
// 根据上次自旋结果动态调整
-XX:PreBlockSpin  // 不再推荐设置
```

### 自适应自旋

```cpp
// HotSpot 的自适应自旋策略
int ObjectSynchronizer::TrySpin(JavaThread* thread) {
    // 记录连续自旋次数
    int spin_count = 0;

    for (;;) {
        // 执行 PAUSE 指令（提示 CPU 缓存预取）
        if (Supports_cx8() && Atomic::cmpxchg(1, &spin_count, spin_count + 1) != spin_count) {
            // 自旋次数达到上限或被其他线程唤醒
            return 0;
        }

        // 检查锁是否已释放
        if (lock->is_unlocked()) {
            return spin_count;  // 自旋成功
        }

        // 如果是其他线程持有（不可偏向），退出自旋
        if (!UseBiasedLocking || mark->biased_lock_locked()) {
            return 0;
        }

        // 继续自旋...
    }
}
```

自适应策略：

| 场景 | 策略 |
|---|---|
| 锁持有时间短 | 增加自旋次数 |
| 锁持有时间长 | 减少自旋次数 |
| 多核 CPU | 增加自旋 |
| 单核 CPU | 减少自旋 |

## 轻量级锁的升级

轻量级锁在以下情况会升级为重量级锁：

```
┌─────────────────────────────────────────────────────────┐
│           轻量级锁升级条件                                │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  1. 自旋超过阈值                                          │
│     - 单核：1-2 次                                       │
│     - 多核：默认 10 次                                    │
│     - 自适应：根据历史调整                                │
│                                                          │
│  2. 多个线程竞争同一把锁                                  │
│     - CAS 替换失败                                        │
│     - 说明有并发冲突                                      │
│                                                          │
│  3. 持有锁的线程阻塞                                      │
│     - 线程切换代价大                                       │
│     - 直接升级重量级锁                                    │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

## 实战代码验证

```java
public class LightweightLockTest {

    public static void main(String[] args) throws InterruptedException {
        Object lock = new Object();

        // 两个线程交替进入同步块
        Thread t1 = new Thread(() -> {
            for (int i = 0; i < 100; i++) {
                synchronized (lock) {
                    System.out.println("T1: " + i);
                }
            }
        }, "T1");

        Thread t2 = new Thread(() -> {
            for (int i = 0; i < 100; i++) {
                synchronized (lock) {
                    System.out.println("T2: " + i);
                }
            }
        }, "T2");

        t1.start();
        t2.start();
        t1.join();
        t2.join();

        System.out.println("Done");
    }
}
```

在这个场景下，偏向锁会被频繁撤销，轻量级锁会接管。如果竞争加剧，最终会升级为重量级锁。

## 面试追问

### Q1：轻量级锁和偏向锁的 Mark Word 区别？

- 偏向锁 Mark Word：存储线程 ID
- 轻量级锁 Mark Word：存储栈帧中锁记录的指针

### Q2：为什么轻量级锁用栈帧指针而不是线程 ID？

因为轻量级锁需要**快速释放**。用指针指向栈帧，释放时只需要 CAS 恢复原值。如果用线程 ID，还需要额外的线程 ID 查找。

### Q3：轻量级锁在什么场景下性能最好？

- 多线程交替执行（错开执行临界区）
- 临界区执行时间短
- 锁竞争不激烈

## 总结

轻量级锁的核心机制：

1. **CAS 替换**：将对象头的 Mark Word 替换为栈帧指针
2. **自旋等待**：获取失败后，自旋等待锁释放
3. **自适应调整**：根据历史数据动态调整自旋次数
4. **升级条件**：自旋超时或多线程竞争时，升级为重量级锁

| 锁状态 | Mark Word 内容 | 适用场景 |
|---|---|---|
| 无锁 | 哈希码 + 年龄 + 偏向位 = 0 |
| 偏向锁 | 线程 ID + epoch | 单线程反复进入 |
| 轻量级锁 | 栈帧锁记录指针 | 多线程交替执行 |
| 重量级锁 | Monitor 指针 | 多线程竞争 |
