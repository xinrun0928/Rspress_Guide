# synchronized 关键字：对象锁的全面解析

在多线程编程中，最核心的问题之一是如何保证多个线程对共享资源的正确访问。如果没有任何同步机制，两个线程同时修改同一个变量可能导致数据损坏；如果过度同步，又可能导致性能急剧下降甚至死锁。Java 提供了多种同步机制，而 `synchronized` 关键字是最基础、最常用的一种。

## synchronized 的本质：内置的锁

`synchronized` 是 Java 内置的锁机制，不需要显式获取和释放。当线程进入 `synchronized` 修饰的代码块或方法时，会自动获取对象的锁；当线程离开 `synchronized` 区域时，会自动释放锁。这种"自动档"的设计让同步编程变得简单，同时避免了开发者忘记释放锁的问题。

```java
// 修饰实例方法：锁住的是 this 对象
public synchronized void increment() {
    count++;
}

// 修饰静态方法：锁住的是 Class 对象
public static synchronized void staticMethod() {
    staticCount++;
}

// 修饰代码块：锁住指定的对象
public void blockMethod() {
    synchronized (this) {
        count++;
    }
    // 锁外的代码可以被其他线程并发访问
}
```

`synchronized` 的锁是**可重入**的。这意味着同一个线程可以多次获取同一个对象的锁：

```java
public synchronized void outer() {
    inner();  // 可以直接调用，因为是同一个线程
}

public synchronized void inner() {
    // 同一个线程可以重入
}
```

可重入锁避免了同一个线程反复进入自己已经持有锁的代码块导致的死锁。如果锁不可重入，线程可能会永久等待自己持有的锁。

## synchronized 的底层实现：对象头的秘密

`synchronized` 的高效实现依赖于 JVM 对对象头的精心设计。每个 Java 对象在内存中都有一个对象头（Object Header），其中包含了对象的元数据和锁信息：

```java
// 32 位 JVM 对象头结构
+------------------+----------------+
|    Mark Word (32 bits)           |    Klass Word (32 bits)    |
+------------------+----------------+
| hashcode:25 | age:4 | biased:0 | 01 |    pointer to klass     |
+------------------+----------------+
| ptr_to_lock:30            | 00 |    pointer to klass     |
+------------------+----------------+
```

在 JDK 6 之前，`synchronized` 只有重量级锁一种实现，每次获取和释放锁都需要操作系统级别的用户态与内核态切换，开销巨大。JDK 6 引入了多种锁优化技术，根据竞争程度自动升级，这就是"锁升级"机制。

## 锁升级：无锁到重量级的进化

JDK 6 实现了"锁膨胀"机制，锁会经历四种状态：偏向锁、轻量级锁、重量级锁。这个过程是单向的，只能升级不能降级。

### 偏向锁：消除同步的基础开销

当一个线程第一次进入 `synchronized` 代码块时，对象头的 Mark Word 会记录该线程的 ID。之后，这个线程进入代码块时，只需要简单地检查 Mark Word 中的线程 ID 是否指向自己：

```java
// 偏向锁获取过程
if (mark == biasedMark && thread == BiasedLocking.PROTOTYPE) {
    // 同一个线程，直接进入
} else {
    // 撤销偏向锁，升级为轻量级锁
}
```

偏向锁消除了同一线程反复进入同步代码块的性能开销。但如果另一个线程尝试获取锁，偏向锁会被撤销，升级为轻量级锁。

### 轻量级锁：用户态的自旋

如果没有锁竞争，轻量级锁使用 CAS（Compare-And-Swap）操作在用户态完成加锁过程，避免了进入内核态的开销：

```java
// 轻量级锁获取
void Lock(Object obj) {
    // 在当前线程的栈帧中创建锁记录
    LockRecord *lr = allocateLockRecord();
    
    // 将对象的 Mark Word 复制到锁记录
    lr->setDisplacedMarkWord(obj.mark);
    
    // 使用 CAS 尝试将对象的 Mark Word 指向锁记录
    if (obj.mark == CAS(lr)) {
        // 加锁成功
        return;
    }
    
    // 如果 CAS 失败，说明存在竞争，升级为重量级锁
    inflateAndLock(obj);
}
```

轻量级锁适用于"锁竞争不激烈且锁持有时间短"的场景。如果锁持有时间较长，自旋等待会浪费 CPU 资源，此时应该升级为重量级锁。

### 重量级锁：操作系统的互斥量

重量级锁使用操作系统的互斥量（Mutex）实现。当获取重量级锁失败时，线程会被挂起，进入阻塞状态，需要等待锁持有者释放锁并唤醒：

```java
// 重量级锁获取 - 操作系统介入
void lock() {
    if (tryLock()) {
        return;  // 尝试成功
    }
    
    // 尝试自旋等待
    int spin = 0;
    while (spin < MAX_SPIN) {
        if (tryLock()) {
            return;
        }
        spin++;
    }
    
    // 自旋失败，线程进入阻塞队列，等待唤醒
    park();
}
```

## 锁优化技术：超越基础 synchronized

### 自适应自旋：智能的等待策略

JDK 6 引入的自旋锁会根据上一次自旋成功率动态调整自旋次数。如果同一个锁的上次自旋很快成功了，认为这次也可能成功，增加自旋次数；如果经常失败，减少自旋甚至取消自旋。

### 锁消除：不必要的锁可以消除

如果 JIT 编译器通过逃逸分析发现某个锁只能被一个线程访问，就会消除这个锁：

```java
public String concatString(String s1, String s2, String s3) {
    // StringBuilder 是局部变量，不会逃逸
    // synchronized 可以被安全消除
    StringBuilder sb = new StringBuilder();
    sb.append(s1);
    sb.append(s2);
    sb.append(s3);
    return sb.toString();
}
```

### 锁粗化：合并相邻的锁区域

如果一系列操作都对同一个对象加锁，JIT 编译器可能会将它们合并为一次加锁，减少加锁次数：

```java
// 优化前
for (int i = 0; i < 1000; i++) {
    synchronized (this) {
        list.add(i);
    }
}

// 优化后 - 锁粗化
synchronized (this) {
    for (int i = 0; i < 1000; i++) {
        list.add(i);
    }
}
```

## synchronized 的使用最佳实践

1. **尽量缩小同步范围**：只对真正需要保护的代码加锁，减少锁持有时间
2. **避免锁嵌套**：嵌套的 synchronized 容易导致死锁
3. **选择合适的锁粒度**：过粗的锁降低并发度，过细的锁增加复杂度
4. **注意锁对象的选择**：实例方法锁 this，静态方法锁 Class 对象
5. **使用并发工具替代**：在 JDK 5+ 时代，优先考虑 `ReentrantLock`、`ReadWriteLock` 等更灵活的锁

## synchronized vs Lock：何时选择谁

| 特性 | synchronized | ReentrantLock |
|------|--------------|----------------|
| 锁获取 | 自动 | 手动 |
| 尝试非阻塞获取 | 不支持 | `tryLock()` |
| 超时获取 | 不支持 | `tryLock(timeout)` |
| 中断获取 | 不支持 | `lockInterruptibly()` |
| 公平锁 | 否（偏向锁优化） | 可选 |
| 多条件队列 | 不支持 | `newCondition()` |

`synchronized` 是 JVM 内置的，使用简单且经过了大量优化；`ReentrantLock` 提供了更多高级特性，适合复杂的同步场景。对于大多数场景，`synchronized` 已经足够好。

理解 synchronized 的锁升级机制，你是否对 JVM 的锁优化有了更深的认识？这些优化让 Java 并发编程在保持简洁的同时，也能在各种场景下发挥出色的性能。
