# synchronized 修饰方法 vs 代码块：深入对比

在日常开发中，我们经常需要决定是使用 synchronized 修饰方法，还是使用 synchronized 块来包裹代码。这个选择看似简单，实际上涉及到锁粒度、性能、以及代码可维护性等多个方面。

## 修饰方法的两种形式

### 修饰实例方法

```java
public class InstanceMethod {
    private int count = 0;

    public synchronized void increment() {
        count++;
    }

    public synchronized int get() {
        return count;
    }
}
```

当 synchronized 修饰实例方法时，锁住的是 `this` 对象。任何线程调用这个对象的任何 synchronized 实例方法，都需要获取同一个锁。

### 修饰静态方法

```java
public class StaticMethod {
    private static int count = 0;

    public static synchronized void increment() {
        count++;
    }
}
```

当 synchronized 修饰静态方法时，锁住的是 Class 对象。由于 Class 对象在 JVM 中只有一个，所以所有线程调用这个类的任何 synchronized 静态方法都需要竞争同一把锁。

## synchronized 代码块的三种形式

### 锁住 this

```java
public class ThisBlock {
    private int count = 0;

    public void increment() {
        synchronized (this) {
            count++;
        }
    }
}
```

这与 synchronized 修饰实例方法等效。

### 锁住其他对象

```java
public class OtherObjectBlock {
    private final Object lock = new Object();
    private int count = 0;

    public void increment() {
        synchronized (lock) {
            count++;
        }
    }
}
```

### 锁住 Class 对象

```java
public class ClassBlock {
    private static int count = 0;

    public static void increment() {
        synchronized (ClassBlock.class) {
            count++;
        }
    }
}
```

## 核心区别：锁粒度

### 方法级锁 vs 代码块级锁

synchronized 修饰方法时，锁的粒度是整个方法体：

```java
// 锁粒度：整个方法
public synchronized void method() {
    prepare();           // 需要同步
    doSomething();       // 需要同步
    cleanup();           // 需要同步
}
```

而 synchronized 代码块允许你只锁住需要同步的部分：

```java
// 锁粒度：只锁住关键部分
public void method() {
    prepare();           // 不需要同步
    synchronized (this) {
        doSomething();   // 只锁住这个
    }
    cleanup();           // 不需要同步
}
```

### 性能差异

锁的粒度越小，持有锁的时间越短，并发性能越好。但这不是绝对的——如果代码块太小，可能导致过多的锁竞争。

```java
public class LockGranularity {
    private int a = 0;
    private int b = 0;
    private int c = 0;

    // 粗粒度锁：整个方法锁住
    public synchronized void method1() {
        a++;
        b++;
        c++;
    }

    // 细粒度锁：每个操作独立加锁
    public void method2() {
        synchronized (this) { a++; }
        synchronized (this) { b++; }
        synchronized (this) { c++; }
    }

    // 分离锁：使用不同的锁对象
    private final Object lockA = new Object();
    private final Object lockB = new Object();
    private final Object lockC = new Object();

    public void method3() {
        synchronized (lockA) { a++; }
        synchronized (lockB) { b++; }
        synchronized (lockC) { c++; }
    }
}
```

## 实践中的选择

### 选择代码块的场景

**场景一：只需要锁住部分操作**

```java
public class UserService {
    private Map<String, User> users = new ConcurrentHashMap<>();

    public void createUser(String name, User user) {
        validateUser(user);          // 验证不需要锁
        synchronized (users) {
            users.put(name, user);   // 只有这个需要锁
        }
        sendNotification(user);     // 发通知不需要锁
    }
}
```

**场景二：与多个锁配合实现更复杂的同步**

```java
public class BankAccount {
    private final Object balanceLock = new Object();
    private double balance;

    private final Object historyLock = new Object();
    private List<String> history = new ArrayList<>();

    public void deposit(double amount) {
        synchronized (balanceLock) {
            balance += amount;
        }
        synchronized (historyLock) {
            history.add("Deposited: " + amount);
        }
    }

    public void withdraw(double amount) {
        synchronized (balanceLock) {
            if (balance >= amount) {
                balance -= amount;
            }
        }
        synchronized (historyLock) {
            history.add("Withdrew: " + amount);
        }
    }
}
```

**场景三：避免锁住整个方法导致其他无关操作阻塞**

```java
public class OrderProcessor {
    private final Map<String, Order> orders = new ConcurrentHashMap<>();

    public void processOrder(Order order) {
        // 验证订单
        validateOrder(order);           // 快速，不需要锁

        // 数据库操作，耗时长
        synchronized (orders) {
            saveToDatabase(order);     // 只有这个需要锁
            orders.put(order.getId(), order);
        }

        // 发送通知
        sendNotification(order);       // 异步，不阻塞
    }
}
```

### 选择方法级 synchronized 的场景

**场景一：方法体本身就很小**

```java
public class Counter {
    private int count = 0;

    public synchronized int incrementAndGet() {
        return ++count;
    }
}
```

方法体只有一条语句，没必要用代码块。

**场景二：所有操作都需要原子性**

```java
public class Stack {
    private final Object[] elements;
    private int top = -1;

    public synchronized void push(Object e) {
        if (top < elements.length - 1) {
            elements[++top] = e;
        }
    }

    public synchronized Object pop() {
        if (top >= 0) {
            return elements[top--];
        }
        return null;
    }
}
```

push 和 pop 都需要同时检查和修改 `top`，必须原子执行。

## 特殊场景：死锁风险

### 嵌套锁导致死锁

使用代码块时更容易出现嵌套锁导致的死锁：

```java
public class TransferService {
    private final Object lockA = new Object();
    private final Object lockB = new Object();

    // 方法A：从账户A转给账户B
    public void transferAtoB() {
        synchronized (lockA) {
            System.out.println("持有 lockA");
            synchronized (lockB) {
                System.out.println("持有 lockB");
                // 转账操作
            }
        }
    }

    // 方法B：从账户B转给账户A
    public void transferBtoA() {
        synchronized (lockB) {
            System.out.println("持有 lockB");
            synchronized (lockA) {
                System.out.println("持有 lockA");
                // 转账操作
            }
        }
    }
}
```

如果两个线程分别调用这两个方法，就可能发生死锁。

### 解决死锁

**方案一：固定加锁顺序**

```java
public class TransferServiceFixed {
    private final Object lockA = new Object();
    private final Object lockB = new Object();

    public void transferAtoB() {
        synchronized (lockA) {
            synchronized (lockB) {
                doTransfer();
            }
        }
    }

    public void transferBtoA() {
        synchronized (lockA) {  // 统一使用 lockA 在前
            synchronized (lockB) {
                doTransfer();
            }
        }
    }
}
```

**方案二：使用 Lock 锁（tryLock）**

```java
import java.util.concurrent.locks.Lock;
import java.util.concurrent.locks.ReentrantLock;

public class TransferServiceWithTryLock {
    private final Lock lockA = new ReentrantLock();
    private final Lock lockB = new ReentrantLock();

    public void transfer() {
        while (true) {
            try {
                if (lockA.tryLock() && lockB.tryLock()) {
                    doTransfer();
                    return;
                }
            } finally {
                if (lockA.isHeldByCurrentThread()) lockA.unlock();
                if (lockB.isHeldByCurrentThread()) lockB.unlock();
            }
            // 退避重试
            Thread.sleep(10);
        }
    }
}
```

## 可重入性

无论使用 synchronized 方法还是 synchronized 块，synchronized 都是可重入的。这意味着同一个线程可以多次获取同一把锁：

```java
public class ReentrantDemo {
    private final Object lock = new Object();

    public void methodA() {
        synchronized (lock) {
            System.out.println("methodA 进入临界区");
            methodB();  // 可以重入
        }
    }

    public void methodB() {
        synchronized (lock) {
            System.out.println("methodB 进入临界区");
        }
    }
}
```

synchronized 修饰方法同样可重入：

```java
public class ReentrantMethod {
    public synchronized void methodA() {
        System.out.println("methodA");
        methodB();  // 同一线程调用同一个锁的方法
    }

    public synchronized void methodB() {
        System.out.println("methodB");
    }
}
```

## 性能对比

在 JDK 1.6 之前，synchronized 是重量级锁，性能很差。但在 1.6 之后引入了锁升级机制，性能大幅提升。

### 无竞争时的性能

```java
public class PerformanceTest {
    private int count = 0;

    // synchronized 方法
    public synchronized void incrementMethod() {
        count++;
    }

    // synchronized 代码块
    private final Object lock = new Object();

    public void incrementBlock() {
        synchronized (lock) {
            count++;
        }
    }

    // 无锁实现
    private AtomicInteger atomicCount = new AtomicInteger();

    public void incrementAtomic() {
        atomicCount.incrementAndGet();
    }
}
```

在无竞争情况下：
- synchronized 代码块（轻量级/偏向锁）性能接近无锁
- synchronized 方法会稍微慢一点（因为需要隐式获取 this 锁）
- AtomicInteger 在高并发下可能有优势

## 最佳实践

1. **优先使用代码块**：缩小锁的粒度，减少锁的持有时间

2. **避免锁住整个方法**：如果方法中只有一小部分需要同步，只锁住那部分

3. **不要在代码块中调用未知代码**：可能导致死锁或性能问题

```java
// 错误示例：调用外部方法可能导致死锁
public void method() {
    synchronized (this) {
        externalService.call();  // 危险！
    }
}

// 正确做法：先释放锁，调用完再重新获取
public void method() {
    Object result;
    synchronized (this) {
        result = compute();
    }
    externalService.call(result);  // 安全
}
```

4. **选择合适的锁对象**：不要使用 String 常量或 Integer 等可能被缓存的对象作为锁

5. **保持锁的对称性**：如果有多把锁，操作多个资源时要保持一致的加锁顺序

## 面试中的常见问题

**问题 1**：synchronized 修饰方法和 synchronized(this) 代码块有什么区别？

**答案**：从效果上看，两者都锁住 this 对象，区别在于锁的粒度。修饰方法会锁住整个方法，可能导致不需要同步的操作也被阻塞；代码块可以只锁住需要同步的部分，提高并发性能。另外，synchronized 代码块可以指定任意对象作为锁，而方法只能锁住 this。

**问题 2**：synchronized 是可重入锁吗？

**答案**：是的。synchronized 是可重入锁，同一个线程可以多次获取同一把锁。可重入是通过计数器实现的，每次获取锁计数器加 1，每次释放锁计数器减 1，计数器为 0 时锁完全释放。

**问题 3**：synchronized 代码块中使用 String 作为锁有什么问题？

**答案**：String 有常量池机制，使用字面量 String（如 "lock"）可能使多个类/实例共享同一把锁。另外，String 对象可能被 JVM 内部缓存或重用，导致意外的锁竞争。

**问题 4**：如何选择 synchronized 和 Lock？

**答案**：synchronized 是 JVM 内置的语法糖，自动释放锁，适合简单同步场景。Lock（ReentrantLock）功能更丰富：可中断、可超时、公平锁、多条件变量。复杂场景推荐使用 Lock，简单场景用 synchronized 即可。
