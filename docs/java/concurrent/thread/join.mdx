# join()：让主线程等待的艺术

想象一个场景：主线程启动了一群子线程去执行任务，然后主线程需要等待所有子线程完成后才能汇总结果。就像项目经理启动了多个员工去调查不同的区域，必须等所有人回来汇报后才能做总结。

`Thread.join()` 就是 Java 提供的这种等待机制。理解 `join()` 的用法和原理，对于编写正确的并发代码至关重要。

## join() 的基本用法

`join()` 方法使当前线程等待指定线程执行完毕。只有当指定线程终止（正常或异常）后，等待的线程才会继续执行。

### 简单示例

```java
public class JoinBasic {
    public static void main(String[] args) throws InterruptedException {
        Thread task = new Thread(() -> {
            try {
                Thread.sleep(2000);
                System.out.println("子线程执行完成");
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
        }, "Task-Thread");

        System.out.println("启动子线程");
        task.start();

        System.out.println("主线程等待子线程...");
        task.join(); // 主线程在这里等待

        System.out.println("主线程继续执行");
    }
}
```

输出顺序是确定的：

```
启动子线程
主线程等待子线程...
子线程执行完成
主线程继续执行
```

主线程调用 `task.join()` 后，会进入 WAITING 状态，直到 task 执行完毕。

## join() 的三种形式

```java
public class JoinForms {
    public static void main(String[] args) throws InterruptedException {
        Thread task = new Thread(() -> {
            // 执行任务
        });

        // 无参数：无限期等待
        task.join();

        // 带毫秒参数：限时等待
        task.join(1000);

        // 带毫秒和纳秒参数：精确超时
        task.join(1000, 500000);
    }
}
```

### join(long millis)

等待指定毫秒数后，即使线程还未执行完毕，也会继续执行。

```java
public class JoinWithTimeout {
    public static void main(String[] args) throws InterruptedException {
        Thread task = new Thread(() -> {
            try {
                Thread.sleep(10000); // 任务需要10秒
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
        });

        long start = System.currentTimeMillis();
        task.start();

        boolean finished = task.join(3000); // 最多等3秒

        long elapsed = System.currentTimeMillis() - start;
        System.out.println("等待耗时: " + elapsed + "ms");
        System.out.println("线程是否完成: " + finished);
    }
}
```

输出：

```
等待耗时: 3000ms
线程是否完成: false
```

`join()` 返回布尔值，表示是否在超时前完成。

## join() 的底层原理

`join()` 的实现其实依赖于 `wait()` 和 `notify()` 机制：

```java
// Thread 类中的 join() 实现（简化版）
public final void join() throws InterruptedException {
    wait(0); // 调用 wait(0)，即无限期等待
}

public final synchronized void join(long millis) throws InterruptedException {
    if (millis == 0) {
        // 循环等待目标线程终止
        while (isAlive()) {
            wait(0);
        }
    } else {
        // 限时等待
        long start = System.nanoTime();
        long remaining = millis;
        while (isAlive()) {
            long delay = Math.min(remaining, Integer.MAX_VALUE);
            wait(delay);
            long elapsed = System.nanoTime() - start;
            remaining = millis - elapsed;
        }
    }
}
```

关键点在于：

1. `join()` 是 `synchronized` 方法，会获取当前线程的监视器锁
2. `join(0)` 通过循环检查 `isAlive()` 来判断目标线程是否终止
3. 如果线程还活着，就调用 `wait(0)` 无限期等待
4. 当目标线程执行完毕后，JVM 会自动 `notifyAll()` 所有等待该线程对象的线程

这是一个很少有人注意到的细节：**是 JVM 在线程终止时调用 `notifyAll()`，而不是被等待的线程本身**。

## 实际应用场景

### 场景一：并行任务汇总

```java
public class TaskAggregation {
    private static final int THREAD_COUNT = 5;
    private static int[] results = new int[THREAD_COUNT];

    public static void main(String[] args) throws InterruptedException {
        Thread[] threads = new Thread[THREAD_COUNT];

        // 启动所有子线程
        for (int i = 0; i < THREAD_COUNT; i++) {
            final int index = i;
            threads[i] = new Thread(() -> {
                results[index] = compute(index);
            }, "Worker-" + i);
            threads[i].start();
        }

        // 等待所有子线程完成
        for (Thread thread : threads) {
            thread.join();
        }

        // 汇总结果
        int total = 0;
        for (int result : results) {
            total += result;
        }

        System.out.println("最终结果: " + total);
    }

    private static int compute(int index) {
        // 模拟计算
        return (index + 1) * 100;
    }
}
```

### 场景二：超时控制

```java
public class TimeoutControl {
    private static volatile boolean taskCompleted = false;

    public static void main(String[] args) {
        Thread task = new Thread(() -> {
            try {
                Thread.sleep(5000);
                taskCompleted = true;
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
        });

        task.start();

        try {
            boolean finished = task.join(2000);
            if (finished) {
                System.out.println("任务在2秒内完成");
            } else {
                System.out.println("任务超时（超过2秒）");
                // 可以选择继续等待或放弃
                task.join(); // 继续等待完成
            }
        } catch (InterruptedException e) {
            e.printStackTrace();
        }
    }
}
```

### 场景三：优雅关闭

```java
public class GracefulShutdown {
    public static void main(String[] args) throws InterruptedException {
        System.out.println("开始关闭...");

        Thread cleanup = new Thread(() -> {
            System.out.println("开始清理资源...");
            try {
                Thread.sleep(2000);
            } catch (InterruptedException e) {
                System.out.println("清理被中断");
                return;
            }
            System.out.println("清理完成");
        });

        cleanup.start();

        // 等待清理线程完成
        cleanup.join();
        System.out.println("程序退出");
    }
}
```

## join() 的陷阱

### 陷阱一：死锁风险

```java
public class JoinDeadlock {
    public static void main(String[] args) throws InterruptedException {
        Thread thread1 = new Thread(() -> {
            System.out.println("Thread1 等待 Thread2");
            try {
                // 错误：在 thread1 中等待 thread2
                // 假设 thread2 也在等待 thread1...
                Thread.sleep(100);
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
        });

        Thread thread2 = new Thread(() -> {
            System.out.println("Thread2 等待 Thread1");
            try {
                thread1.join(); // Thread2 等待 Thread1
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
        });

        thread1.start();
        thread2.start();

        thread1.join(); // main 等待 thread1
        System.out.println("Main 继续");
    }
}
```

### 陷阱二：忘记检查任务状态

```java
public class ForgetStateCheck {
    public static void main(String[] args) throws InterruptedException {
        Thread task = new Thread(() -> {
            // 什么都不做
        });

        task.start();
        task.join(); // 等待完成

        // 错误：假设任务一定成功
        // 应该检查任务的实际结果或状态
        System.out.println("任务完成了");
    }
}
```

更好的方式是使用 `Future` 或 `CompletableFuture` 来获取任务结果和状态。

### 陷阱三：join() 在同步块外

```java
public class JoinOutsideSync {
    private static final Object lock = new Object();
    private static boolean ready = false;

    public static void main(String[] args) throws InterruptedException {
        Thread setter = new Thread(() -> {
            synchronized (lock) {
                ready = true;
            }
        });

        setter.start();
        setter.join(); // 这里 join() 是可以的

        // 但是如果想等待 ready 变为 true
        // 不能这样写：
        // while (!ready) { Thread.sleep(100); } // 忙等待，浪费 CPU
    }
}
```

## join() 与 CountDownLatch 对比

`join()` 和 `CountDownLatch` 都可以用来等待线程完成，但各有优劣：

```java
// 使用 join() 等待多个线程
public class JoinMulti {
    public static void main(String[] args) throws InterruptedException {
        Thread t1 = new Thread(() -> System.out.println("t1 done"));
        Thread t2 = new Thread(() -> System.out.println("t2 done"));
        Thread t3 = new Thread(() -> System.out.println("t3 done"));

        t1.start();
        t2.start();
        t3.start();

        t1.join();
        t2.join();
        t3.join();

        System.out.println("所有线程完成");
    }
}

// 使用 CountDownLatch 等待多个线程
public class CountDownLatchMulti {
    public static void main(String[] args) throws InterruptedException {
        CountDownLatch latch = new CountDownLatch(3);

        new Thread(() -> { System.out.println("t1 done"); latch.countDown(); }).start();
        new Thread(() -> { System.out.println("t2 done"); latch.countDown(); }).start();
        new Thread(() -> { System.out.println("t3 done"); latch.countDown(); }).start();

        latch.await();
        System.out.println("所有线程完成");
    }
}
```

**CountDownLatch 的优势**：

- 可以让任意线程调用 `await()`，而 `join()` 必须是被等待线程的调用者
- 可以重置（通过 `new CountDownLatch(n)`），`join()` 不行
- 可以等待任意数量的线程，而 `join()` 需要持有线程引用

**join() 的优势**：

- API 简单直观
- 不需要额外引入 `CountDownLatch`

## 面试问题

**问题 1**：`join()` 等待的线程结束后，为什么等待的线程会被唤醒？

**答案**：这是 JVM 的特殊处理。当一个线程执行完毕后（`run()` 方法结束或抛出未捕获的异常），JVM 会自动调用该线程对象的 `notifyAll()`，唤醒所有等待这个线程对象的线程。这就是为什么 `join()` 能让等待的线程继续执行。

**问题 2**：`join()` 和 `wait()` 有什么区别？

**答案**：虽然 `join()` 内部使用 `wait()` 实现，但两者有本质区别：

- `wait()` 是 `Object` 的方法，等待的是某个对象被 `notify()`
- `join()` 是 `Thread` 的方法，等待的是指定线程执行完毕
- `wait()` 需要在同步块中调用，`join()` 不需要
- `wait()` 可以被任意线程 `notify()`，`join()` 的唤醒是线程终止自动触发的

**问题 3**：如何实现 join() 的超时功能？

**答案**：可以使用 `join(long millis)` 方法，或者使用 `Future.get(long timeout)`，或者 `CountDownLatch.await(long timeout, TimeUnit)`。
