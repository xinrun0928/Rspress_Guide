# ThreadLocal：线程封闭的利器与内存泄漏的陷阱

候选人小张在面试中遇到了这样的问题：

面试官问："ThreadLocal是什么？用它做什么？"

小张说："ThreadLocal给每个线程提供独立的变量副本，可以实现线程隔离。"

面试官追问："那ThreadLocal的底层是怎么实现的？"

小张说："每个Thread有一个ThreadLocalMap。"

面试官继续追问："那ThreadLocalMap的Entry为什么要用弱引用？内存泄漏是怎么产生的？"

小张停顿了半天，说不太清楚。

【面试官心理】
ThreadLocal是Java面试中的高频问题，但90%的候选人只知道"线程隔离"。能回答出ThreadLocalMap的实现细节、Entry用弱引用的原因、以及内存泄漏的完整链条的，只有10%。ThreadLocal的内存泄漏问题是一个经典的"设计权衡"——JDK选择了用弱引用来避免内存泄漏，但引入了另一层复杂性。理解这个问题，需要对JVM内存管理和弱引用机制有深入理解。

## 一、ThreadLocal的核心思想 🔴

### 1.1 解决什么问题

在没有ThreadLocal的时代，如何在线程内共享一个变量但不在线程间共享？

```java
// ❌ 错误方案：普通成员变量
public class UserService {
    private String userId; // 所有请求共享？不安全！

    public void handleRequest(HttpServletRequest req) {
        userId = req.getUserId(); // 设置用户ID
        doSomething(); // 其他方法读取userId
    }
}

// ✅ 正确方案：ThreadLocal
public class UserService {
    private static final ThreadLocal<String> userId = new ThreadLocal<>();

    public void handleRequest(HttpServletRequest req) {
        userId.set(req.getUserId()); // 只在当前线程设置
        doSomething(); // 其他方法读取userId
    }
}
```

### 1.2 典型使用场景

**场景一：保存线程上下文**

```java
public class RequestContext {
    private static final ThreadLocal<RequestContext> CONTEXT =
        ThreadLocal.withInitial(RequestContext::new);

    private String userId;
    private String traceId;
    private Map<String, Object> attributes = new HashMap<>();

    public static RequestContext current() {
        return CONTEXT.get();
    }

    public static void clear() {
        CONTEXT.remove();
    }
}

// 使用：任何地方都能获取当前请求的上下文
String traceId = RequestContext.current().getTraceId();
```

**场景二：数据库连接的线程隔离**

```java
public class DBConnectionHolder {
    private static final ThreadLocal<Connection> localConn =
        ThreadLocal.withInitial(() -> {
            try {
                return DriverManager.getConnection(url, user, pwd);
            } catch (SQLException e) {
                throw new RuntimeException(e);
            }
        });

    public static Connection get() {
        return localConn.get();
    }
}
```

**场景三：SimpleDateFormat的线程安全问题**

```java
// ❌ 错误：SimpleDateFormat不是线程安全的
private static final SimpleDateFormat sdf = new SimpleDateFormat("yyyy-MM-dd");

// ✅ 正确：ThreadLocal为每个线程提供独立的SimpleDateFormat
private static final ThreadLocal<SimpleDateFormat> dateFormat =
    ThreadLocal.withInitial(() -> new SimpleDateFormat("yyyy-MM-dd"));

public String format(Date date) {
    return dateFormat.get().format(date);
}
```

## 二、ThreadLocal的底层实现 🔴

### 2.1 核心结构

```java
public class ThreadLocal<T> {
    // ThreadLocalMap的哈希表容量（必须是2的幂）
    private static final int INITIAL_CAPACITY = 16;

    // ThreadLocalMap是ThreadLocal的内部类
    static class ThreadLocalMap {
        // Entry继承WeakReference<ThreadLocal<?>>
        static class Entry extends WeakReference<ThreadLocal<?>> {
            Object value;
            Entry(ThreadLocal<?> k, Object v) {
                super(k); // 弱引用！key是ThreadLocal对象
                value = v;
            }
        }

        private Entry[] table;
        private int size = 0;
        private int threshold; // 扩容阈值
    }

    // Thread类中持有ThreadLocalMap引用
    // ThreadLocal.ThreadLocalMap threadLocals = null;
}
```

### 2.2 三层引用结构

```
Thread (强引用)
  └─ threadLocals (强引用)
       └─ ThreadLocalMap.table (强引用)
            └─ Entry[] (强引用)
                 └─ Entry (WeakReference<ThreadLocal>)
                      └─ get() → ThreadLocal对象（弱引用，可被GC）
```

### 2.3 get()的实现

```java
public T get() {
    Thread t = Thread.currentThread();
    ThreadLocalMap map = t.threadLocals;

    if (map != null) {
        ThreadLocalMap.Entry e = map.getEntry(this);
        if (e != null) {
            @SuppressWarnings("unchecked")
            T result = (T)e.value;
            return result;
        }
    }
    return setInitialValue(); // 首次调用，创建map
}

private T setInitialValue() {
    T value = initialValue();
    Thread t = Thread.currentThread();
    ThreadLocalMap map = t.threadLocals;

    if (map != null)
        map.set(this, value);
    else
        createMap(t, value); // 创建ThreadLocalMap

    return value;
}
```

### 2.4 set()的实现

```java
public void set(T value) {
    Thread t = Thread.currentThread();
    ThreadLocalMap map = t.threadLocals;

    if (map == null)
        createMap(t, value);
    else
        map.set(this, value);
}
```

## 三、弱引用与内存泄漏：JDK的设计权衡 🔴

### 3.1 为什么要用弱引用

**Entry的key用弱引用的原因**：避免ThreadLocal对象无法被GC回收。

```java
static class Entry extends WeakReference<ThreadLocal<?>> {
    Object value;
    Entry(ThreadLocal<?> k, Object v) {
        super(k); // 弱引用
        value = v;
    }
}
```

如果Entry用**强引用**持有ThreadLocal：
```
Thread → ThreadLocalMap → Entry[] → Entry → ThreadLocal（强引用）
```

当ThreadLocal变量不再被使用（方法栈中没有引用了），但Thread还在运行，ThreadLocal对象无法被GC——因为ThreadLocalMap还持有强引用。这会导致**内存泄漏**。

用**弱引用**后：
```
ThreadLocal变量（栈上引用）→ 强引用
ThreadLocal变量 → 弱引用 → Entry（可被GC）

当栈上的强引用消失后，GC可以回收ThreadLocal对象。
Entry中的弱引用变成null，value仍然存在，但key已经回收了。
```

### 3.2 内存泄漏的完整链条

即使使用弱引用，**仍然存在内存泄漏风险**：

```
场景：ThreadLocal实例不再使用，但Thread仍在运行

Timeline:
T1: threadLocal.set(value) → Entry{key=threadLocal, value=obj}
T2: threadLocal = null（栈上的强引用消失）
T3: GC运行 → threadLocal对象被回收（因为Entry持有的是弱引用）
    → Entry.key = null, Entry.value = obj（value仍然被Entry持有！）
T4: Thread仍在运行，ThreadLocalMap.entry[].value 永远无法被清理
```

**内存泄漏的根本原因**：Entry的**value是强引用**，即使key（ThreadLocal）被GC了，value仍然无法被清理。

### 3.3 JDK的缓解措施：expungeStaleEntry

ThreadLocalMap在每次get/set时，会触发**探测式清理**：

```java
private int expungeStaleEntry(int staleSlot) {
    Entry[] tab = table;
    int len = tab.length;

    // 清理当前槽位
    tab[staleSlot].value = null;
    tab[staleSlot].referent = null;
    tab[staleSlot] = null;
    size--;

    // 重新hash周围的槽位，清理可能产生的空洞
    // ...
}
```

但这只清理当前操作遇到的过期Entry，**不能保证所有过期Entry都被清理**。

### 3.4 正确的做法：手动remove()

```java
try {
    userId.set(userId);
    doSomething();
} finally {
    userId.remove(); // 手动清理！
}
```

**唯一可靠的内存泄漏预防方案**：在使用完ThreadLocal后，**显式调用remove()**。

## 四、JDK 11后的改进与InheritableThreadLocal 🟡

### 4.1 InheritableThreadLocal

ThreadLocal的值默认不继承给子线程。如果需要在子线程中继承父线程的值，使用InheritableThreadLocal：

```java
private static final InheritableThreadLocal<String> traceId =
    new InheritableThreadLocal<>();

// 父线程设置
traceId.set("abc123");

// 子线程自动继承
Thread child = new Thread(() -> {
    System.out.println(traceId.get()); // 输出: abc123
});
child.start();
```

**原理**：Thread在创建子线程时，父Thread的ThreadLocalMap会被复制到子线程：

```java
Thread(ThreadGroup group, Runnable target, String name,
       long stackSize, AccessControlContext acc,
       boolean inheritThreadLocals) {
    // ...
    if (inheritThreadLocals && parent.inheritableThreadLocals != null)
        this.inheritableThreadLocals =
            ThreadLocal.createInheritedMap(parent.inheritableThreadLocals);
}
```

### 4.2 InheritableThreadLocal的坑

**坑1：线程池场景下的问题**

线程池中的线程是**复用**的。当一个任务完成后，ThreadLocal的值不会自动清除，下一个任务可能读到上一个任务的值。

```java
// ❌ 危险：线程池中的InheritableThreadLocal值不会自动清理
ExecutorService pool = Executors.newFixedThreadPool(2);
InheritableThreadLocal<String> ctx = new InheritableThreadLocal<>();

pool.submit(() -> {
    ctx.set("任务A的值");
    // 任务A完成，线程归还线程池
    // ctx的值仍然存在！
});

pool.submit(() -> {
    System.out.println(ctx.get()); // 可能读到"任务A的值"！
});
```

**解决方案**：在任务开始时显式remove，或使用阿里开源的TransmittableThreadLocal库。

### 4.3 TransmittableThreadLocal（阿里TTL）

阿里巴巴开源的TTL解决了InheritableThreadLocal在线程池场景下的值泄漏问题：

```java
// 使用TTL
TransmittableThreadLocal<String> context = new TransmittableThreadLocal<>();

// 线程池提交时，自动处理值的传递和清理
TtlRunnable ttlRunnable = TtlRunnable.get(runnable);
executorService.submit(ttlRunnable);
```

## 五、生产避坑：ThreadLocal的致命陷阱 ⚠️

### 5.1 线程池 + ThreadLocal的内存泄漏

```java
// ❌ 危险：Web服务中使用ThreadLocal但忘记清理
public class UserContextFilter implements Filter {
    private static final ThreadLocal<User> currentUser = new ThreadLocal<>();

    public void doFilter(...) {
        User user = authenticate(request);
        currentUser.set(user);
        try {
            chain.doFilter(request, response);
        } finally {
            // ❌ 忘记remove！每个请求都会泄漏一个User对象
        }
    }
}

// ✅ 正确：必须remove
try {
    currentUser.set(user);
    chain.doFilter(request, response);
} finally {
    currentUser.remove(); // 关键！
}
```

### 5.2 ThreadLocal在ThreadPoolExecutor中的注意事项

```java
// ❌ 错误配置
new ThreadPoolExecutor(10, 20, 0L, TimeUnit.MILLISECONDS,
    new LinkedBlockingQueue<>(),
    new ThreadFactory() {
        public Thread newThread(Runnable r) {
            Thread t = new Thread(r);
            // 没有继承父线程的InheritableThreadLocal
            // 但线程池中的值泄漏问题仍然存在
            return t;
        }
    });
```

## 六、❌ 错误示范

**候选人原话**："ThreadLocalMap的Entry用弱引用是为了防止内存泄漏。"

**问题诊断**：
- 弱引用只能防止key（ThreadLocal）本身泄漏
- value仍然可能泄漏
- 说"防止内存泄漏"是不准确的

**候选人原话**："ThreadLocal是线程安全的，因为每个线程有独立的变量副本。"

**问题诊断**：
- ThreadLocal本身是线程安全的
- 但ThreadLocal中的对象如果是可变对象（List、Map等），修改它们仍然是不安全的
- ThreadLocal只隔离了变量的引用，不隔离变量指向的对象

【面试官心理】
ThreadLocal的内存泄漏问题是Java面试中最经典的"设计权衡"题之一。JDK用弱引用保护ThreadLocal对象本身，但value的泄漏需要程序员手动处理。这种权衡背后的哲学是：框架承担一部分责任（用弱引用），但最终的清理工作由开发者负责。理解这个权衡的候选人，在设计API和框架时也会考虑到责任边界的划分。

## 七、面试应对脚本

**面试官**："ThreadLocalMap的Entry为什么要用弱引用？"
**标准回答**："Entry用弱引用是为了避免ThreadLocal对象本身无法被GC。如果用强引用，即使ThreadLocal变量不再使用（栈上无引用），ThreadLocalMap仍然持有对它的强引用，导致内存泄漏。用弱引用后，ThreadLocal对象的引用消失时，GC可以回收它，Entry.key变成null。但注意，value仍然被Entry持有（强引用），不会因为key被GC而自动清理。所以仍然需要手动调用remove()来清理value，避免value泄漏。"

**面试官追问**："ThreadLocal的内存泄漏是怎么产生的？如何避免？"
**标准回答**："内存泄漏的场景是：ThreadLocal变量设置后不再使用（但Thread仍在运行）。GC运行时ThreadLocal对象被回收（因为弱引用），但Entry.value没有被清理。泄漏链条是：Thread → ThreadLocalMap → Entry → value。JDK通过expungeStaleEntry()在get/set时清理过期的Entry，但这不能保证所有过期Entry都被清理。唯一可靠的方案是在使用完ThreadLocal后显式调用remove()。在Web应用中，Filter或Interceptor是清理ThreadLocal的最佳位置。"

【面试官心理】
ThreadLocal这道题能回答到内存泄漏链条的候选人，说明他对JVM内存管理和弱引用机制有实战理解。这类候选人在使用ThreadLocal时，不会忘记在finally中调用remove()——因为他知道不这么做的代价是什么。这种"知其然更知其所以然"的习惯，是区分P6和P7候选人的关键。
