# ThreadLocal 原理与内存泄漏

`ThreadLocal` 是个好用的工具，但用不好就会导致内存泄漏。

很多人知道 ThreadLocal 使用了弱引用，但很少有人真正理解为什么有了弱引用还会有泄漏。

这篇文章会深入剖析 ThreadLocal 的实现原理，解释弱引用与内存泄漏的关系，以及如何正确使用 ThreadLocal。

## ThreadLocal 是什么

### 基本概念

`ThreadLocal` 提供了线程本地的变量。每个线程访问的是自己的独立副本，互不干扰：

```java
ThreadLocal<String> threadLocal = new ThreadLocal<>();

// 线程 A
new Thread(() -> {
    threadLocal.set("A's value");
    System.out.println(threadLocal.get());  // A's value
}).start();

// 线程 B
new Thread(() -> {
    threadLocal.set("B's value");
    System.out.println(threadLocal.get());  // B's value
}).start();

// 主线程
System.out.println(threadLocal.get());  // null
```

### 典型使用场景

```java
// 场景 1：存储用户上下文
public class UserContext {
    private static final ThreadLocal<User> currentUser = new ThreadLocal<>();

    public static void set(User user) {
        currentUser.set(user);
    }

    public static User get() {
        return currentUser.get();
    }

    public static void clear() {
        currentUser.remove();
    }
}

// 使用
public void handleRequest(User user) {
    UserContext.set(user);
    try {
        // 业务逻辑中随时获取当前用户
        User current = UserContext.get();
    } finally {
        UserContext.clear();  // 重要！
    }
}

// 场景 2：线程安全的时间格式化
private static final ThreadLocal<DateFormat> dateFormat =
    ThreadLocal.withInitial(() -> new SimpleDateFormat("yyyy-MM-dd"));

String date = dateFormat.get().format(new Date());
```

## ThreadLocal 的实现原理

### 核心结构

每个线程都有一个 `ThreadLocalMap`，是 `ThreadLocal` 的核心数据结构：

```java
public class Thread implements Runnable {
    // Thread 类内部有一个 ThreadLocalMap
    ThreadLocal.ThreadLocalMap threadLocals = null;

    // ...
}

public class ThreadLocal<T> {

    // ThreadLocalMap 的 Entry
    static class Entry extends WeakReference<ThreadLocal<?>> {
        Object value;

        Entry(ThreadLocal<?> k, Object v) {
            super(k);  // key 是弱引用！
            value = v;  // value 是强引用！
        }
    }

    // ThreadLocalMap 的结构（类似 HashMap）
    static class ThreadLocalMap {
        private Entry[] table;
        // ...
    }
}
```

**关键点**：

1. 每个线程有一个 `ThreadLocalMap`
2. `ThreadLocalMap` 的 Entry 继承 `WeakReference<ThreadLocal<?>>`
3. Entry 的 key（ThreadLocal 对象）是**弱引用**
4. Entry 的 value（实际值）是**强引用**

### 工作流程

```java
public class ThreadLocal<T> {

    public void set(T value) {
        Thread t = Thread.currentThread();
        ThreadLocalMap map = getMap(t);

        if (map != null) {
            map.set(this, value);  // 以 this（ThreadLocal 对象）为 key
        } else {
            createMap(t, value);
        }
    }

    public T get() {
        Thread t = Thread.currentThread();
        ThreadLocalMap map = getMap(t);

        if (map != null) {
            ThreadLocalMap.Entry e = map.getEntry(this);
            if (e != null) {
                @SuppressWarnings("unchecked")
                T result = (T) e.value;
                return result;
            }
        }
        return setInitialValue();  // 没有则初始化
    }
}
```

### 图解数据存储

```
线程内部结构：

Thread
  |
  └── threadLocals: ThreadLocalMap
                        |
                        └── table: Entry[]
                              |
        +--------------------+--------------------+
        |                    |                    |
    Entry(ThreadLocal1, value1)  Entry(ThreadLocal2, value2)
    key=WeakRef(TL1)              key=WeakRef(TL2)
    value=强引用                   value=强引用
```

## 弱引用与内存泄漏

### 为什么使用弱引用

弱引用的设计是为了解决这样的问题：

```
如果没有弱引用会发生什么？

ThreadLocal<UserContext> tl = new ThreadLocal<>();
tl.set(userContext);

// 业务代码不再持有 tl 的引用
tl = null;

// 此时：
// - ThreadLocal 对象还有谁引用？
// - ThreadLocalMap.entry.value 还引用着 userContext！
// - 结果：userContext 永远无法被 GC 回收 → 内存泄漏！
```

使用弱引用后：

```
如果 key（ThreadLocal）是弱引用：

ThreadLocal<UserContext> tl = new ThreadLocal<>();
tl.set(userContext);

// tl 置为 null
tl = null;

// 此时只有 ThreadLocalMap.entry.key 引用 ThreadLocal
// GC 时，如果只有弱引用，ThreadLocal 对象会被回收！

// 但 value 呢？
// value 仍然被 entry.value 强引用，不会被回收！
```

### 内存泄漏的真正原因

有了弱引用，**ThreadLocal 对象本身不会泄漏**，但**value 仍然可能泄漏**：

```
泄漏场景：

1. ThreadLocal tl = new ThreadLocal<>();
   tl.set(largeObject);

2. 业务代码不再使用 tl
   tl = null;

3. ThreadLocal 对象被 GC 回收（弱引用）

4. 但 value 呢？
   entry = { key=null, value=largeObject }

   key 为 null，但 value 仍然存在！

5. 如果这个线程是池化复用的（如线程池）
   线程不会结束，ThreadLocalMap 不会清理
   → value 永远无法回收 → 内存泄漏！
```

### 泄漏的触发条件

```java
// 场景 1：线程池 + ThreadLocal
ExecutorService pool = Executors.newFixedThreadPool(1);

pool.submit(() -> {
    ThreadLocal<byte[]> data = new ThreadLocal<>();
    data.set(new byte[1024 * 1024 * 100]);  // 100MB

    // 线程复用，ThreadLocalMap 中的数据不会被清理
});

// 线程池中的线程不会销毁，ThreadLocalMap 永久存在
// 如果不手动 remove，100MB 内存泄漏
```

```java
// 场景 2：没有手动 remove
public class UserContext {
    private static final ThreadLocal<User> currentUser = new ThreadLocal<>();

    public static void set(User user) {
        currentUser.set(user);  // 忘记 remove
    }
}

// 用户 A 请求完成，切换到用户 B
// 但 User A 的数据可能还在内存中
```

## ThreadLocalMap 的清理机制

### 探测式清理

ThreadLocalMap 提供了 `expungeStaleEntry()` 方法清理过期的 Entry：

```java
private int expungeStaleEntry(int staleSlot) {
    Entry[] tab = table;
    int len = tab.length;

    // 删除过期 entry
    tab[staleSlot].value = null;
    tab[staleSlot] = null;
    size--;

    // 重新哈希，清理链表中的过期 entry
    Entry e = tab[staleSlot];
    while (e != null) {
        ThreadLocal<?> k = e.get();
        if (k == null) {
            // 清理过期 entry
            e.value = null;
            tab[i] = null;
        }
        // ...
    }
    return nextIndex;
}
```

### 启发式清理

`get()` 和 `set()` 时会触发启发式清理：

```java
private Entry getEntry(ThreadLocal<?> key) {
    int i = key.threadLocalHashCode & (table.length - 1);
    Entry e = table[i];

    if (e != null && e.get() == key) {
        return e;
    }
    // 探测式清理
    return getEntryAfterMiss(key, i, e);
}

private Entry getEntryAfterMiss(ThreadLocal<?> key, int i, Entry e) {
    Entry[] tab = table;
    int len = tab.length;

    while (e != null) {
        ThreadLocal<?> k = e.get();
        if (k == key)
            return e;

        if (k == null) {
            // 发现过期 entry，触发清理
            expungeStaleEntry(i);
        } else {
            i = nextIndex(i, len);
        }
        e = tab[i];
    }
    return null;
}
```

### 扩容机制

当 ThreadLocalMap 中的元素过多时，会触发扩容：

```java
private void rehash() {
    // 清理所有过期 entry
    expungeStaleEntries();

    // 如果清理后 size >= threshold * 3/4，扩容
    if (size >= threshold - threshold / 4) {
        resize();
    }
}
```

## 正确使用 ThreadLocal

### 必须手动 remove

```java
// 正确用法
public class UserContext {
    private static final ThreadLocal<User> currentUser = new ThreadLocal<>();

    public static void set(User user) {
        currentUser.set(user);
    }

    public static User get() {
        return currentUser.get();
    }

    public static void remove() {
        currentUser.remove();
    }
}

// 在请求处理完成后必须调用
public void handleRequest(HttpServletRequest request) {
    User user = authenticate(request);
    UserContext.set(user);
    try {
        // 业务逻辑
    } finally {
        UserContext.remove();  // 必须！
    }
}
```

### 使用 try-finally 确保清理

```java
public <T> T executeWithContext(T value, Supplier<T> action) {
    ThreadLocal<T> context = threadLocals.get();
    T previous = context.get();
    try {
        context.set(value);
        return action.get();
    } finally {
        if (previous != null) {
            context.set(previous);
        } else {
            context.remove();
        }
    }
}
```

### 框架集成

Spring 框架早就考虑了这个问题：

```java
// Spring 的 RequestContextHolder
// 内部使用 ThreadLocal，在请求结束时自动清理

// Spring MVC 的 Filter
public class OncePerRequestFilter {
    protected void doFilterInternal(...) {
        try {
            doFilter(request, response, filterChain);
        } finally {
            // 清理 ThreadLocal
        }
    }
}
```

## 常见问题

### Q1：为什么 Entry 用弱引用？

用弱引用是为了防止 ThreadLocal 对象本身泄漏。如果没有弱引用，ThreadLocal 对象被 ThreadLocalMap 强引用，即使外部不再使用也无法回收。

### Q2：有了弱引用为什么还有泄漏？

弱引用只能保护 ThreadLocal 对象本身。Entry 的 value 仍然是强引用，如果线程长期存活（线程池），value 不会被清理。

### Q3：为什么推荐使用 static ThreadLocal？

```java
// 推荐：static
private static final ThreadLocal<User> currentUser = new ThreadLocal<>();

// 不推荐：非 static
public class MyService {
    private ThreadLocal<User> user = new ThreadLocal<>();
    // 问题：如果创建多个 MyService 实例，每个实例的 ThreadLocal 不同
}
```

`static` 确保整个应用只创建一个 ThreadLocal 对象，与线程数量无关。

### Q4：ThreadLocal 适合什么场景？

```java
// 适合：
// - 线程上下文传递（如用户信息、事务 ID）
// - 线程安全的对象缓存（如 DateFormat）
// - 跨层参数传递

// 不适合：
// - 需要跨线程共享的数据
// - 大量数据的缓存（会泄漏）
// - 长期存储
```

## 总结

ThreadLocal 的内存泄漏机制：

| 引用类型 | 持有对象 | GC 后的状态 |
|---|---|---|
| 强引用 | ThreadLocalMap.entry.key | ThreadLocal 对象被回收 |
| 强引用 | ThreadLocalMap.entry.value | 如果线程存活，value 仍然存活 |

**避免泄漏的方法**：

1. **手动 remove**：每次使用后调用 `threadLocal.remove()`
2. **使用 try-finally**：确保清理一定执行
3. **框架集成**：利用 Spring 等框架的自动清理
4. **避免大对象**：不要在 ThreadLocal 中存储大对象

记住：**弱引用保护 ThreadLocal 对象，但保护不了 value**。线程池环境下，ThreadLocal 的生命周期与线程相同，必须手动清理才能避免泄漏。
