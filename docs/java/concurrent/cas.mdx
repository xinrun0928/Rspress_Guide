# CAS原理与ABA问题：乐观锁的双刃剑

候选人小赵在面试中被问到这样一个问题：

面试官问："AtomicInteger的incrementAndGet()是怎么实现的？"

小赵说："它用的是CAS操作，compareAndSwapInt。"

面试官追问："那CAS是怎么保证原子性的？底层是什么？"

小赵说："...Unsafe类的本地方法？"

面试官继续追问："那ABA问题是什么？JDK怎么解决的？"

小赵停顿了半天："ABA...是指A变成B再变回A吗？"

面试官："对，然后呢？"

小赵彻底答不上来。

【面试官心理】
这道题我用来筛选候选人对并发底层原理的理解深度。知道CAS是"比较并交换"的占60%，能说出它是CPU指令的占30%，能讲清楚LOCK CMPXCHGL指令在总线级别保证原子的只有10%。ABA问题更是一个经典陷阱——90%的候选人听说过这个词，但只有20%能说清楚JDK的解决方案（AtomicStampedReference）。能回答到最后的，基本都是对并发编程有深度实践的候选人。

## 一、CAS的原理：一条CPU指令的魔法 🔴

### 1.1 从问题开始

在多线程环境下，如何安全地给一个整数加1？

`synchronized`的方案是加锁——同一时刻只有一个线程能执行加法操作。

但`synchronized`太重了——每次加1都要进内核态，开销太大。

有没有更轻量的方案？

**CAS（Compare-And-Swap）**。

### 1.2 CAS的核心思想

CAS(V, A, B) 的语义是：**如果当前值等于期望值A，则将其更新为B；否则什么都不做，返回当前值**。

整个过程是**原子**的——不会有任何线程在比较和交换之间插进来。

```java
// incrementAndGet的伪实现
public final int incrementAndGet() {
    int current;
    do {
        current = get();       // 读取当前值
    } while (!compareAndSwapInt(this, valueOffset, current, current + 1));
    // CAS成功时返回，失败时重试（自旋）
    return current + 1;
}
```

### 1.3 底层实现：一条CPU指令

在x86架构下，CAS对应一条指令：

```asm
LOCK CMPXCHGL eax, [ebx]   ; 如果eax == [ebx]，则[ebx] = eax+1
```

关键点是**LOCK前缀**：
- LOCK前缀会锁住**总线**或**缓存行**，确保在CAS操作期间，其他CPU核心无法访问同一个内存地址
- 这个锁是由硬件（CPU）保证的，**不需要操作系统介入**

因此，CAS操作是**硬件级别的原子操作**，开销比synchronized小得多——大约是10-100纳秒级别。

### 1.4 CAS vs synchronized

| 维度 | CAS（乐观锁） | synchronized（悲观锁） |
|------|-------------|---------------------|
| 核心思想 | 乐观地假设没有冲突 | 悲观地假设一定有冲突 |
| 实现方式 | CPU原子指令 | OS互斥量 |
| 线程模型 | 自旋重试 | 阻塞/唤醒 |
| 开销 | 无竞争时极低 | 每次都要加锁/解锁 |
| 适用场景 | 冲突概率低的场景 | 冲突概率高的场景 |

## 二、CAS的三大问题 🔴

### 2.1 ABA问题：最经典的陷阱

**什么是ABA问题？**

线程A读取到值V，线程B将V改成A'再改回V，线程A再次执行CAS时，发现V==V，于是CAS成功。

**问题在哪？** 线程A无法感知值在中间被修改过。

**一个ABA的致命场景**：

```java
public class StackWithCAS {
    AtomicReference<Node> top = new AtomicReference<>();

    public void push(Node newNode) {
        Node oldTop;
        do {
            oldTop = top.get();      // ① 读取oldTop = Node A
            newNode.next = oldTop;   // ② newNode.next = Node A
        } while (!top.compareAndSet(oldTop, newNode)); // ③ CAS
        // 如果线程A在①之后、③之前被挂起...
    }
}
```

考虑这个场景：
1. 线程A执行push：读取oldTop（Node A），挂起（还没CAS）
2. 线程B执行pop：弹出Node A，push Node B，此时top = Node B
3. 线程B继续pop：弹出Node B，push Node A'（新的Node A），此时top = Node A'
4. 线程A恢复：CAS比较top==oldTop（Node A == Node A'？不对，但CAS比较的是引用...实际上引用值相同，所以CAS成功了）
5. **但Node A的next指针已经被改过了**——形成了**链表环**

这就是ABA问题的经典危害：**导致数据结构损坏**。

### 2.2 自旋开销

CAS不成功时，线程会**自旋重试**。如果冲突概率很高，自旋次数会很多，浪费CPU。

```
// 自旋的代价
while (!cas(addr, expect, update)) {
    // 什么也不做，纯粹空转
}
```

在高竞争场景下（1000个线程同时修改同一个值），自旋的开销会超过synchronized的阻塞/唤醒开销。

**解法**：JDK 8引入了`LongAdder`，用分段锁的思想替代单一CAS变量，减少竞争。

### 2.3 只能保证单个变量的原子性

synchronized可以锁住一段代码逻辑，保护多个变量的复合操作。

CAS只能保证**一个变量的原子性**。

```java
// ❌ 不能用CAS保证复合操作的原子性
if (map.containsKey(key)) {    // CAS 1
    map.get(key);               // CAS 2
} else {
    map.put(key, value);        // CAS 3
}
// check-then-act不是原子操作，CAS保证不了
```

## 三、ABA问题的JDK解决方案：AtomicStampedReference 🟡

### 3.1 版本号机制

JDK提供了`AtomicStampedReference<V>`，它在每个值旁边附带一个**版本号（stamp）**：

```java
// 结构：(reference, stamp)
// reference: 值
// stamp: 版本号（int）

private Pair<V> pair;

// compareAndSet的签名
public boolean compareAndSet(V expectedReference,
                               V newReference,
                               int expectedStamp,
                               int newStamp)
```

修改时不仅要比较值，还要比较版本号：

```java
public boolean compareAndSet(V expectedReference,
                               V newReference,
                               int expectedStamp,
                               int newStamp) {
    Pair<V> current = pair;
    // 值相等但版本号变了？说明中间被修改过，CAS失败
    return expectedReference == current.reference &&
           expectedStamp == current.currentStamp &&
           ((newReference == current.reference &&
             newStamp == newStamp) ||
            casPair(current, Pair.of(newReference, newStamp)));
}
```

### 3.2 解决链表ABA问题

```java
public class SafeStack {
    private AtomicStampedReference<Node> top =
        new AtomicStampedReference<>(null, 0);

    public void push(Node newNode) {
        int[] stamp = new int[1];
        Node oldTop;
        do {
            oldTop = top.getStamp(stamp);      // 获取当前值和版本号
            newNode.next = oldTop;
        } while (!top.compareAndSet(oldTop, newNode, stamp[0], stamp[0] + 1));
    }
}
```

即使另一个线程修改了链表结构并让值"变回"A，版本号已经+1了，CAS会失败，触发重试。

### 3.3 AtomicMarkableReference

如果不需要递增的版本号，只需要一个"标记位"（boolean），可以用`AtomicMarkableReference`：

```java
AtomicMarkableReference<String> ref =
    new AtomicMarkableReference<>("value", false);

ref.compareAndSet("value", "newValue", false, true);
```

它的内存开销比`AtomicStampedReference`小（一个bit vs 一个int），适合只需要"是否被修改过"的场景。

## 四、CAS在JDK中的应用 🔴

### 4.1 AtomicInteger/AtomicLong

```java
// incrementAndGet()
public final int incrementAndGet() {
    for (int;;) {
        int current = get();
        int next = current + 1;
        if (compareAndSet(current, next))
            return next;
    }
}
```

### 4.2 ConcurrentHashMap中的CAS

JDK 8的ConcurrentHashMap在大量操作中使用CAS：

```java
// putVal中的CAS应用
if (casTabAt(tab, i, null, new Node(hash, key, value, null))) {
    break; // 插入成功
}
```

只有当槽位为空时才能用CAS（避免了锁竞争）。如果槽位非空，JDK 8改用`synchronized`锁住单个桶（锁粒度细化）。

### 4.3 LongAdder：分段CAS减少竞争

JDK 8引入的`LongAdder`是应对高竞争场景的利器：

```java
LongAdder adder = new LongAdder();
adder.increment(); // 多个线程分散写入不同的cell，减少CAS竞争
long result = adder.sum(); // 汇总所有cell的值
```

内部维护了一个`Cell[]`数组（类似分段锁的思想，但用CAS替代锁）。每个线程累加时Hash到不同的Cell，最后求和时将所有Cell的值相加。冲突越多，分段越多，性能越好。

## 五、❌ 错误示范

**候选人原话**："CAS是Java提供的原子操作，所以它比synchronized快。"

**问题诊断**：
- 混淆了"CAS是硬件指令"和"Java封装了CAS"的概念
- CAS在没有竞争时很快，但高竞争时自旋开销可能超过synchronized
- 没有理解CAS的局限性（ABA问题、复合操作、只能保证单变量）

**候选人原话**："ABA问题不重要，因为最终值还是A，没有数据损坏。"

**问题诊断**：
- 只理解了ABA的字面意思，没有理解ABA在数据结构中的实际危害
- 在栈/队列等链表操作中，ABA会导致链表环
- 说明对并发数据结构的理解还停留在表面

**候选人原话**："Unsafe类可以直接操作内存，性能最好，所以应该尽量用Unsafe。"

**问题诊断**：
- Unsafe是JDK内部类，不应该在业务代码中使用
- JDK 9之后Unsafe被限制访问（需要--add-opens）
- 正确做法是使用java.util.concurrent.atomic包中的原子类

【面试官心理】
问到CAS时，我最想确认候选人对"乐观锁"这个范式的理解。CAS不是万能的——它解决不了高竞争场景下的性能问题，也解决不了复合操作的原子性问题。能正确使用CAS的候选人，会知道在什么时候用CAS，什么时候用synchronized，什么时候用LongAdder。这种判断力来自对并发底层原理的深度理解。

## 六、生产避坑：CAS的线上陷阱 ⚠️

### 6.1 伪共享：缓存行的隐形杀手

在高并发的CAS操作中，即使多个线程修改的是不同变量，如果它们恰好在**同一个缓存行**（64字节），MESI协议也会让它们相互invalidate对方的缓存行。

`LongAdder`的`Cell[]`数组为什么用`@Contended`注解填充缓存行？就是为了避免伪共享——让每个Cell独占一个缓存行。

```java
@sun.misc.Contended
static final class Cell {
    volatile long value;
    Cell(long x) { value = x; }
    // ...
}
```

### 6.2 JOL验证缓存行

```java
import org.openjdk.jol.info.ClassLayout;

System.out.println(ClassLayout.parseInstance(new Cell[4]).toPrintable());
// 查看Cell数组的内存布局，确认是否有填充
```

### 6.3 循环CAS的CPU占用

在死循环中进行CAS自旋：

```java
while (!atomicRef.compareAndSet(expected, newValue)) {
    expected = atomicRef.get(); // 重新读取重试
}
```

如果竞争激烈，这条代码会吃掉一个CPU核心的100%资源。在容器化环境（CPU限制严格）下，可能影响其他容器的调度。

**解法**：JDK 9+的VarHandle API提供了`acquireFence()`等内存屏障工具，可以更细粒度地控制内存顺序。

## 七、面试应对脚本

**面试官**："CAS是怎么保证原子性的？"
**标准回答**："CAS是一条CPU指令，在x86上是LOCK CMPXCHGL。这条指令通过LOCK前缀锁住总线或缓存行，确保在比较和交换操作之间，没有其他CPU核心能访问同一个内存地址。这个锁是由硬件级别保证的，不需要OS介入，所以开销比synchronized小得多——大约10-100纳秒级别。"

**面试官追问**："ABA问题是什么？为什么会有问题？"
**标准回答**："ABA问题发生在：线程A读取值V，线程B将V改成A'再改回V，线程A的CAS发现值还是V就成功了。但线程A不知道中间被修改过。在链表数据结构中，这会导致严重的结构损坏。比如一个用CAS实现的栈，ABA问题可能导致链表形成环，后续pop操作陷入死循环。解决方案是使用AtomicStampedReference或AtomicMarkableReference，在值旁边附带版本号或标记位。"

**面试官追问**："LongAdder和AtomicInteger怎么选？"
**标准回答**："LongAdder适合高并发累加场景。它的核心思想是分段：每个线程累加到自己的Cell中而不是同一个变量，减少CAS竞争。但LongAdder的sum()需要汇总所有Cell，不是lock-free的。AtomicInteger适合低竞争、单线程累加、或需要立即获取最终值的场景。选型原则是：如果竞争激烈且只需要最终结果，用LongAdder；如果竞争不激烈或需要立即拿到当前值，用AtomicInteger。"

【面试官心理】
能回答ABA问题和LongAdder选型的候选人，对并发编程已经有了体系化的理解。ABA问题不是一道孤立的概念题——它背后反映了乐观锁的核心缺陷。只有真正理解乐观锁适用范围的人，才能在实战中正确选择CAS、synchronized或LongAdder。
