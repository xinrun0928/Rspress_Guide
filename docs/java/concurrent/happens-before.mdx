# Happens-Before原则：JMM的8条铁律与面试破解

候选人小李在阿里面P6面试时，遇到了这样一个场景：

面试官画了两段代码，问："这段代码输出什么？"

```java
// 线程A
int x = 1;          // ①
boolean ready = true; // ②

// 线程B
if (ready) {         // ③
    System.out.println(x); // ④
}
```

小李说："如果线程B执行到③时线程A已经执行完②，那就会输出1。"

面试官追问："那如果线程B执行到③时，线程A的①还没执行完呢？"

小李说："那可能输出0。"

面试官点点头："那如果给ready加上volatile，结果会怎样？"

小李停顿了一下："...那应该就输出了1？"

面试官："为什么？"

小李彻底卡住了。

【面试官心理】
这道题考的是候选人对happens-before规则的理解程度。知道"volatile保证可见性"的人很多，但能说出"volatile写happens-before volatile读"这条具体规则，并用它来解释为什么x一定被正确初始化的，就少得多。happens-before不是一条规则，而是一套规则树——只有把这8条规则串联起来，才能真正理解JMM的有序性和可见性保证。

## 一、Happens-Before是什么：序的哲学 🔴

### 1.1 定义：时间的先后与因果

在JMM中，happens-before不是"时间上的先后"。它是**因果关系**的保证：**如果操作A happens-before操作B，那么A的执行结果对B可见，且A的执行顺序在B之前**。

这句话有三个关键词：
- **因果**：A是B的因，B能看到A的结果
- **可见性**：A的修改对B可见
- **顺序**：A的执行在B之前

但注意：happens-before**不要求物理时间上的先后**。在不同的CPU核心上，物理时间可能不一样。happens-before定义的是**程序顺序上的可见性关系**，是JMM层面的抽象。

### 1.2 为什么需要happens-before

假设没有happens-before规则，JIT编译器和CPU可以随意重排指令。那意味着什么？

```java
public class NoHappensBefore {
    int a = 0;
    int b = 0;

    public void writer() {
        a = 1;  // ①
        b = 1;  // ②
    }

    public void reader() {
        if (b == 1) {  // ③
            System.out.println(a); // ④
        }
    }
}
```

如果没有happens-before约束：
- 编译器可以重排成：先写b，再写a
- CPU可以重排成：b先刷回主内存，a还在Store Buffer里
- 线程B看到b==1时，a可能还是0

有了happens-before规则——**程序顺序规则**（同一线程内，前面的操作happens-before后面的操作）——我们就知道① happens-before ②，③ happens-before ④。但①②和③④之间没有happens-before关系，所以这个场景下④可能看到a==0。

**结论**：没有同步措施的情况下，④看到a==0是**合法的**。这不是bug，是JMM允许的行为。

## 二、8条Happens-Before规则 🔴

这是JMM最核心的内容，面试必考。

### 2.1 规则一：程序顺序规则（Program Order Rule）

> 一个线程中，每个操作happens-before该线程中后续的操作。

```java
int a = 1;     // ①
int b = 2;     // ②
a = a + 1;     // ③
```

① happens-before ② happens-before ③。在同一个线程内，重排只能在这些操作的**结果不被后续操作观测到**的范围内进行。

### 2.2 规则二：volatile变量规则（Volatile Variable Rule）

> 对volatile变量的写，happens-before后续对该volatile变量的读。

```java
volatile boolean ready = false;

writer() {
    x = 42;        // ①
    ready = true;  // ②
}

reader() {
    if (ready) {   // ③
        System.out.println(x); // ④
    }
}
```

② happens-before ③。那么根据**传递性**——① happens-before ② happens-before ③，所以① happens-before ④。

**这就是前面那道面试题的标准答案**：`volatile ready`使得线程B在看到`ready==true`时，一定能看到`x==42`。

### 2.3 规则三：监视器锁规则（Monitor Lock Rule）

> 对一个锁的解锁，happens-before后续对这个锁的加锁。

```java
synchronized (lock) {
    // 临界区
}
```

线程A释放锁happens-before线程B获取同一把锁。这意味着线程A在临界区内对共享变量的所有修改，对线程B都是可见的。

**这是synchronized保证可见性和有序性的根本原因**。

### 2.4 规则四：线程启动规则（Thread Start Rule）

> `Thread.start()`调用happens-before被启动线程中的任何操作。

```java
Thread B = new Thread(() -> {
    // 线程B的代码
});
B.start();
sharedVar = 100;
```

`sharedVar = 100` happens-before B线程内的所有代码。主线程在启动子线程之前对共享变量的修改，子线程一定能看到。

### 2.5 规则五：线程终止规则（Thread Termination Rule）

> 线程中的所有操作happens-before其他线程检测到该线程终止。

```java
Thread B = new Thread(() -> {
    result = compute();
});
B.start();
B.join();
System.out.println(result);
```

`B.join()`返回后，主线程能看到B线程对result的所有修改。`result = compute()` happens-before join()返回。

### 2.6 规则六：中断规则（Thread Interruption Rule）

> 对线程`interrupt()`的调用happens-before被中断线程检测到中断事件。

`Thread.interrupted()`和`Thread.isInterrupted()`能检测到中断，happens-before主线程调用`interrupt()`。

### 2.7 规则七：对象终结规则（Finalizer Rule）

> 对象的构造函数happens-before该对象的`finalize()`方法。

`finalize()`开始执行时，构造函数中的所有操作都已经完成。这保证了对象在finalize时已经完全初始化。

### 2.8 规则八：传递性规则（Transitivity）

> 如果A happens-before B，B happens-before C，则A happens-before C。

这是最常用的推导规则。**大部分复杂的happens-before推导，都是通过传递性把多条规则串联起来的**。

### 2.9 规则速查表

| 规则 | 描述 | 典型场景 |
|------|------|----------|
| 程序顺序 | 同线程内，前→后 | 普通单线程代码 |
| volatile | 写→读同一volatile | volatile变量通信 |
| 监视器锁 | 解锁→加锁同一锁 | synchronized |
| 线程启动 | start()→子线程代码 | 主线程→子线程数据传递 |
| 线程终止 | 子线程→join()返回 | 等待子线程结果 |
| 线程中断 | interrupt()→检测中断 | 中断响应 |
| 对象终结 | 构造→finalize() | 资源清理 |
| 传递性 | A→B, B→C → A→C | 规则串联推导 |

## 三、❌ 错误示范

### 3.1 混淆"时间先后"和"happens-before"

**候选人原话**："因为线程A先执行①和②，线程B后执行③和④，所以①一定在③之前执行，x应该是1。"

**问题诊断**：
- 把物理时间上的"先执行"当成了happens-before
- 忽略了CPU缓存、Store Buffer、JIT重排的影响
- 没有理解JMM的定义——happens-before是因果关系，不是时间关系

### 3.2 以为volatile是"全能钥匙"

**候选人原话**："只要加了volatile，所有线程都能立即看到最新值，所以所有变量都不需要同步了。"

**问题诊断**：
- volatile只保证自己和`ready`相关的读写有序
- 如果另一个线程在写一个**非volatile**变量，其他线程仍然可能看不到最新值
- volatile不能替代synchronized

【面试官心理】
问到happens-before时，我最常见的错误回答有两种：第一种是把happens-before理解成物理时间顺序，说明没有理解JMM的抽象本质；第二种是只记得volatile规则，但忘了其他7条规则。能完整说出8条规则并且能举例的，基本都是认真看过JMM规范的人，这类候选人在并发问题的分析上往往有体系化的思考方式。

## 四、用Happens-Before规则分析面试题 🟡

### 4.1 经典题：双重检查锁定

```java
public class Singleton {
    private static volatile Singleton instance;

    public static Singleton getInstance() {
        if (instance == null) {
            synchronized (Singleton.class) {
                if (instance == null) {
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
```

用happens-before分析：

**问题**：为什么instance必须是volatile？不加volatile会怎样？

**分析**（不加volatile）：
1. `instance = new Singleton()` 分解为三个操作：
   - ① 分配内存
   - ② 调用构造函数初始化
   - ③ 将引用赋值给instance

2. 编译器和CPU**可能重排**为：① → ③ → ②（JIT认为这样效率更高，且不影响单线程语义）

3. 线程A执行到③（instance已非null，但还未初始化），线程B在第一次检查时发现instance != null，直接返回了**未初始化完成的对象**——导致NPE或访问未初始化的字段。

**加volatile后的分析**：
- 根据volatile变量规则，volatile写的③ happens-before volatile读的读取
- 传递性：volatile写的③ happens-before构造函数中的所有操作（因为构造函数在volatile写之前执行）
- 这就禁止了构造函数和instance赋值的重排

### 4.2 追问升级：JMM规则下的数据初始化问题

```java
public class SafePublication {
    static String[] ARR;

    public static void init() {
        ARR = new String[10];
        for (int i = 0; i < 10; i++) {
            ARR[i] = "item" + i;
        }
    }

    public static void print() {
        if (ARR != null) {
            System.out.println(ARR[0]);
        }
    }
}
```

**不加同步**：线程A调用init()，线程B调用print()。ARR[0]可能输出null或"item0"，两种都是可能的。因为`ARR = new String[10]`和`ARR[i] = ...`之间没有happens-before关系。

**加volatile后**：ARR声明为`volatile static String[] ARR`。根据volatile规则，init()中所有对ARR的写入 happens-before print()中对ARR的读取。线程B保证看到完整的初始化后的数组。

## 五、生产避坑：看不见的Happens-Before陷阱 ⚠️

### 5.1 集合的线程安全问题

```java
Map<String, Object> cache = new HashMap<>();

// 线程A
cache.put("key", value);

// 线程B
Object val = cache.get("key");
```

`HashMap`不是线程安全的。即使线程A完全写完了，线程B看到的可能是不完整的状态（扩容到一半的中间状态）。这不是happens-before能解决的问题——需要用`ConcurrentHashMap`或`Collections.synchronizedMap`。

### 5.2 Future的隐含Happens-Before

`Future.get()`方法包含了happens-before保证——submit()的任务执行 happens-before get()返回。这意味着：

```java
Future<Integer> future = executor.submit(() -> {
    return compute();
});
// submit和get之间有隐含的happens-before
Integer result = future.get(); // 一定能拿到submit中的返回值
```

## 六、面试应对脚本

**面试官**："happens-before是什么？解决什么问题？"
**标准回答**："happens-before是JMM中定义的一套因果关系规则。它规定了如果操作A happens-before操作B，那么A的结果对B可见，且A的执行顺序在B之前。注意，happens-before不是物理时间上的先后，而是程序顺序上的可见性保证。这8条规则是JMM有序性和可见性的基石。"

**面试官追问**："你能把这8条规则都说出来吗？"
**标准回答**："可以。程序顺序规则——同线程内前→后；volatile变量规则——volatile写→后续volatile读；监视器锁规则——解锁→后续加锁；线程启动规则——start()→子线程代码；线程终止规则——子线程→join()返回；中断规则——interrupt()→检测中断；对象终结规则——构造→finalize()；传递性规则——A→B, B→C则A→C。其中前三条最重要，也最容易在面试中被追问。"

**面试官追问**："volatile和synchronized在happens-before上的区别是什么？"
**标准回答**："volatile只保证和该volatile变量相关的读写有序——写→后续对同一个变量的读。synchronized的锁规则更全面——它保证临界区内所有操作对后续获取同一把锁的线程可见，且禁止临界区内外的重排。从这个角度说，synchronized是比volatile更强的同步手段，volatile只是轻量级的可见性保证。"
