# 30-60-90 天计划

拿到 Offer 不是终点，而是新起点。

入职后的前 90 天，是决定你能否在这家公司站稳脚跟的关键期。本章帮你制定一个清晰的入职计划。

## 面试官最关心的 3 个问题

> **问题 1**：入职第一天应该做什么？
> **问题 2**：如何快速融入团队？
> **问题 3**：如何在前 90 天展示价值？

## 30-60-90 天框架

```
┌─────────────────────────────────────────────┐
│              30-60-90 天成长计划            │
├─────────────────────────────────────────────┤
│                                             │
│  30天：适应期                              │
│  ├── 熟悉环境、认识人、了解业务            │
│  └── 目标：不出错                           │
│                                             │
│  60天：融入期                              │
│  ├── 参与项目、产出价值、建立关系          │
│  └── 目标：做出贡献                        │
│                                             │
│  90天：突破期                              │
│  ├── 主导项目、展现能力、赢得信任          │
│  └── 目标：展示潜力                        │
│                                             │
└─────────────────────────────────────────────┘
```

## 第一个月（1-30天）：适应期

### 目标

```
核心目标：熟悉环境，不出错

具体目标：
- 了解公司的组织架构
- 熟悉团队的工作流程
- 掌握核心技术栈
- 完成 onboarding 任务
```

### 关键行动

#### Week 1

| 行动 | 说明 |
|------|------|
| 熟悉环境 | 认识团队成员、IT 系统、办公环境 |
| 了解业务 | 阅读业务文档、了解产品线 |
| 技术准备 | 配置开发环境、阅读代码 |
| 建立沟通 | 和领导 1:1、认识同事 |

#### Week 2-4

| 行动 | 说明 |
|------|------|
| 深入了解业务 | 跟项目、参与会议、阅读文档 |
| 技术学习 | 学习核心技术栈、完成小任务 |
| 主动请教 | 遇到问题及时请教，不要闷头做事 |
| 记录总结 | 记录学到的东西，建立知识库 |

### 注意事项

```
✅ 应该做：
- 主动沟通，不懂就问
- 记录学到的东西
- 按时完成任务
- 尊重现有流程和规范

❌ 不应该做：
- 太急于表现
- 质疑现有方案
- 独来独往
- 闷头做事不沟通
```

## 第二个月（31-60天）：融入期

### 目标

```
核心目标：融入团队，做出贡献

具体目标：
- 独立完成小任务
- 参与项目开发
- 建立合作关系
- 了解团队痛点
```

### 关键行动

#### 工作层面

| 行动 | 说明 |
|------|------|
| 独立任务 | 独立完成分配的任务 |
| 参与项目 | 参与核心项目的开发 |
| 代码质量 | 注意代码规范和质量 |
| 主动沟通 | 及时同步进度，遇到问题主动沟通 |

#### 关系层面

| 行动 | 说明 |
|------|------|
| 建立信任 | 按时完成任务，建立信任 |
| 跨团队沟通 | 认识其他团队的同事 |
| 参与活动 | 参加团队的技术分享、活动 |
| 寻求反馈 | 定期和领导 1:1，接受反馈 |

### 注意事项

```
✅ 应该做：
- 主动承担任务
- 展示学习能力
- 建立技术影响力
- 及时同步进度

❌ 不应该做：
- 只顾自己
- 不主动沟通
- 回避问题
- 过度表现
```

## 第三个月（61-90天）：突破期

### 目标

```
核心目标：展示价值，赢得认可

具体目标：
- 主导小项目
- 产出技术成果
- 获得领导认可
- 建立技术影响力
```

### 关键行动

#### 工作层面

| 行动 | 说明 |
|------|------|
| 主导项目 | 主导一个小项目或模块 |
| 解决问题 | 主动发现并解决问题 |
| 技术优化 | 提出并推动技术优化 |
| 沉淀输出 | 产出技术文档、最佳实践 |

#### 影响力层面

| 行动 | 说明 |
|------|------|
| 技术分享 | 做一个技术分享 |
| 帮助他人 | 主动帮助新人或同事 |
| 参与决策 | 参与技术方案讨论 |
| 建立口碑 | 在团队中建立技术形象 |

### 注意事项

```
✅ 应该做：
- 主动争取机会
- 展示独特价值
- 持续学习
- 维护关系

❌ 不应该做：
- 过度承诺
- 推卸责任
- 固步自封
- 只做分配的任务
```

## 试用期避坑指南

### 常见坑点

| 坑点 | 说明 | 避免方法 |
|------|------|----------|
| 太急于表现 | 还没了解情况就提建议 | 先观察再行动 |
| 不主动沟通 | 遇到问题不说不问 | 及时沟通汇报 |
| 踩红线 | 违反公司规定 | 了解并遵守规则 |
| 固步自封 | 只做分配的任务 | 主动寻找机会 |
| 急于融入 | 过度社交或过度沉默 | 保持平衡 |

### 与领导相处

```
✅ 应该做：
- 定期 1:1
- 主动汇报进度
- 寻求反馈
- 理解领导的期望

❌ 不应该做：
- 不主动沟通
- 隐瞒问题
- 只听命令不动脑
- 当面一套背后一套
```

### 与同事相处

```
✅ 应该做：
- 主动认识同事
- 乐于助人
- 尊重他人
- 积极参与团队活动

❌ 不应该做：
- 背后议论他人
- 抢他人功劳
- 独来独往
- 过度表现
```

## 常见问题

**Q：试用期被辞退的原因有哪些？**

A：能力不达标、态度问题、融入问题、违反公司红线等。

**Q：试用期表现好有什么表现？**

A：按时完成任务、主动承担、获得信任、建立影响力。

**Q：试用期需要注意什么？**

A：多沟通、多请教、多观察、少表态、不踩红线。
