# 面试记录与复盘模板

面试不总结，等于白面试。

每一场面试都是一次学习的机会，但很多人面试完就「放飞自我」，没有复盘，导致同样的错误反复出现。

## 面试官最关心的 3 个问题

> **问题 1**：面试后应该记录哪些内容？
> **问题 2**：如何做有效的面试复盘？
> **问题 3**：复盘的结果如何应用到下次面试？

## 面试记录模板

### 面试记录表

```markdown
## 面试基本信息

| 项目 | 内容 |
|------|------|

| 级别 | P6 |

| 面试日期 | 2024-03-15 |

| 面试官 | 张三（P7） |

## 面试流程

1. 自我介绍（5 分钟）
2. 项目介绍（15 分钟）
3. 技术追问（20 分钟）
4. 算法题（15 分钟）
5. 反问环节（5 分钟）

## 面试题目

### 技术基础

1. HashMap 的底层结构是什么？
2. JDK7 和 JDK8 的 HashMap 有什么区别？
3. ConcurrentHashMap 的实现原理是什么？

### 项目问题

1. 请介绍一下你负责的 XX 项目
2. 项目中遇到的最大技术挑战是什么？
3. 项目中的性能优化是怎么做的？

### 算法题

1. 合并两个有序链表
   - 我的思路：使用双指针
   - 完成情况：✅ 10 分钟完成
   - 复杂度：O(n)

## 反馈与问题

### 面试官反馈
- 项目深度可以再深挖
- 有些技术细节需要更清晰

### 暴露的问题
1. 项目中某些技术选型的理由不够清晰
2. 性能优化的数据不够详细

## 下次改进

- [ ] 补充项目中的技术选型理由
- [ ] 准备更详细的数据（QPS、响应时间等）
```

## 复盘方法论

### 复盘时间

| 时机 | 复盘内容 |
|------|----------|
| 面试结束后 30 分钟内 | 趁记忆清晰，快速记录 |
| 面试结束后 24 小时内 | 详细复盘，分析得失 |
| 面试季结束后 | 总结规律，优化策略 |

### 复盘框架（KISS）

| 维度 | 问题 |
|------|------|
| **K - Keep** | 哪些做得好，下次继续保持？ |
| **I - Improve** | 哪些需要改进？ |
| **S - Stop** | 哪些不应该再做？ |
| **S - Start** | 下次要开始做什么？ |

## 复盘示例

### 示例：算法表现复盘

```
## 算法表现复盘

### Keep（保持）
- 拿到题目先和面试官确认题意
- 先说思路再写代码
- 注意边界条件处理

### Improve（改进）
- 代码写完后没有检查，可以加一个测试用例的验证
- 写代码时没有边写边解释，可以边写边说思路

### Stop（停止）
- 不要再写完代码才想起来检查
- 不要在没想清楚时就动手

### Start（开始）
- 每次写代码前，先在草稿纸上画出思路
- 代码写完后，主动提出几个测试用例验证
```

### 示例：项目表现复盘

```
## 项目表现复盘

### Keep（保持）
- 用 STAR 法则组织项目描述
- 用数据说话，展示项目成果
- 主动说明自己在项目中的角色

### Improve（改进）
- 项目中的一些技术细节不够清晰
- 被追问时有些紧张

### Stop（停止）
- 不要过度准备导致不自然
- 不要把团队成果说成个人成果

### Start（开始）
- 准备一个「万能项目故事」可以应对多个问题
- 练习被追问时的应对技巧
```

## 面试复盘检查表

### 技术面复盘

- [ ] 技术基础问题回答得怎么样？
- [ ] 项目细节是否清晰？
- [ ] 算法题是否完成？
- [ ] 有没有答不上来的问题？
- [ ] 暴露了哪些知识盲点？

### BQ面复盘

- [ ] BQ 回答是否流畅？
- [ ] 是否用了 STAR 法则？
- [ ] 有没有暴露负面情绪？
- [ ] 是否展示了价值观匹配？

### HR面复盘

- [ ] 离职原因是否合理？
- [ ] 薪资期望是否表达清楚？
- [ ] 有没有踩坑？
- [ ] 是否展示了稳定性？

## 常见问题

**Q：面试记录要保存多久？**

A：建议保存至少 6 个月，方便回顾和对比。

**Q：面试没通过还要复盘吗？**

A：是的。没通过更应该复盘，找出原因，下次改进。

**Q：复盘后如何应用？**

A：将复盘结果转化为具体的改进行动，更新简历或准备计划。
