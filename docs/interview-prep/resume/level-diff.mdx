# 不同级别简历差异

「工作 3 年写的简历，和工作 5 年、7 年的简历有什么不同？」这是很多候选人都会困惑的问题。

答案很简单：**级别越高，简历越需要展示「超越代码」的能力**。

## 面试官最关心的 3 个问题

> **问题 1**：P5、P6、P7 的简历有什么区别？
> **问题 2**：级别提升后，简历应该增加哪些内容？
> **问题 3**：如果想从 P6 升 P7，简历应该怎么调整？

## 三个级别的核心差异

| 维度 | P5 | P6 | P7 |
|------|-----|-----|-----|
| **核心定位** | 执行者 | 独立贡献者 | 技术引领者 |
| **项目角色** | 参与 | 负责 | 主导/定义 |
| **技术深度** | 会用 | 理解原理 | 能做技术选型 |
| **团队协作** | 协作 | 协调 | 影响力 |
| **业务理解** | 知道做什么 | 知道为什么做 | 决定做什么 |

## P5 级别简历（初/中级工程师）

### 核心定位

- 能够独立完成分配的任务
- 在指导下能完成复杂任务
- 技术栈正在建立中

### 简历特点

```
✅ 重点内容：
- 技术栈名称（Java、Spring、MySQL、Redis）
- 项目中参与的具体功能
- 解决的具体技术问题
- 使用的工具和方法

❌ 不需要写：
- 架构设计（级别不够）
- 团队管理（没有）
- 技术选型（没到那个层级）
```

### 简历示例

```
个人信息
姓名：张三 | 3 年 Java 开发经验 | 手机：138xxxx | 邮箱：zhangsan@email.com

技能
Java、Spring Boot、MySQL、Redis、Git、Docker

工作经历
2021.07 - 至今  XX科技  Java 开发工程师
- 参与订单系统开发，负责订单查询、状态流转等核心功能开发
- 使用 Redis 实现订单状态缓存，查询 QPS 提升 3 倍
- 编写单元测试，核心模块覆盖率从 40% 提升至 80%
- 参与系统重构，将响应时间从 500ms 优化至 100ms

项目经历
订单查询性能优化（参与）
- 背景：订单查询接口日均调用 100 万次，响应时间 500ms
- 行动：使用 Arthas 定位慢查询，优化 SQL 索引，引入 Redis 缓存
- 结果：查询时间从 500ms 降至 100ms，QPS 提升 3 倍
```

### P5 简历常见问题

| 问题 | 表现 | 正确做法 |
|------|------|----------|
| 角色模糊 | 「参与」「负责」混淆 | 明确写「独立负责」或「参与」 |
| 成果空洞 | 「运行良好」「效果不错」 | 用数据说话 |
| 技术栈太杂 | 堆砌所有会的技术 | 只写与项目相关的 |

## P6 级别简历（高级工程师）

### 核心定位

- 能独立完成复杂任务
- 能设计方案并推动落地
- 能指导 P5

### 简历特点

```
✅ 重点内容：
- 主导的项目和模块
- 技术方案设计能力
- 性能优化和架构改进
- 量化的工作成果
- 一定的团队协作能力

❌ 不需要写：
- 公司级战略（级别不够）
- 完整的技术体系规划（偏架构师）
- 大量的人员管理（偏 TL）
```

### 简历示例

```
个人信息
姓名：李四 | 5 年 Java 开发经验 | 手机：138xxxx | 邮箱：lisi@email.com | GitHub：github.com/lisi

技能
- 后端：Java、Golang、Spring Cloud、Dubbo
- 数据库：MySQL（分库分表、读写分离）、Redis、Kafka
- 架构：微服务、分布式、缓存、高并发
- 工具：Docker、K8s、Prometheus、Grafana

工作经历
2019.07 - 至今  XX科技  高级 Java 开发工程师
- 主导交易系统高可用架构改造，系统可用性从 99.9% 提升至 99.99%
- 设计并实现多级缓存方案，缓存命中率 95%，数据库压力降低 80%
- 主导服务拆分项目，将单体应用拆分为 12 个微服务，发布周期缩短 80%
- 作为 team lead，培养 2 名 P5 工程师

项目经历
交易中台系统（技术负责人）
- 背景：日均 100 万订单，峰值 QPS 10000，原有系统响应时间 > 2s
- 技术栈：Java、Redis、Kafka、MySQL、Spring Cloud Alibaba
- 架构设计：
  - 分层缓存：JVM 本地缓存 + Redis 分布式缓存 + MySQL
  - 异步化：Kafka 实现订单履约链路异步解耦
  - 高可用：服务降级、熔断、限流多重保障
- 结果：核心接口 P99 < 100ms，QPS 提升 5 倍，SLA 99.99%
```

### P6 vs P5 简历差异对比

| 维度 | P5 | P6 |
|------|------|------|
| **项目角色** | 参与 | 负责/主导 |
| **项目描述** | 做了什么功能 | 怎么设计的、为什么 |
| **技术深度** | 会用 | 理解原理、能优化 |
| **成果表达** | 功能上线 | 量化指标（性能提升 X%） |
| **协作能力** | 协作完成 | 指导他人、跨团队沟通 |

## P7 级别简历（技术专家/架构师）

### 核心定位

- 能定义技术方向
- 能主导跨团队项目
- 能带团队做技术决策
- 对业务有深刻理解

### 简历特点

```
✅ 重点内容：
- 技术战略和方向
- 跨团队/跨部门项目
- 团队规模和技术影响力
- 业务价值和社会效益
- 技术体系设计能力

❌ 不需要写：
- 太多 CRUD 细节（级别太高）
- 纯执行层面的内容
```

### 简历示例

```
个人信息
姓名：王五 | 8 年 Java 开发经验 | 手机：138xxxx | 邮箱：wangwu@email.com

技能
- 架构：高并发、高可用、微服务、云原生、领域驱动设计
- 技术栈：Java/Go、Spring Cloud、K8s、Istio、Prometheus
- 业务：电商交易、金融支付、供应链
- 管理：技术团队培养、技术委员会运作

工作经历
2017.07 - 至今  XX科技  技术专家 / 架构师
- 主导公司级微服务架构升级项目，涉及 5 个部门、20+ 服务、30+ 人
- 制定微服务、云原生、DevOps 三大技术方向，规划 3 年技术演进路线
- 建立技术评审委员会，制定技术规范和最佳实践
- 带领 8 人团队，培养 P7 2 人、P6 4 人

项目经历
全站微服务架构升级（技术负责人）
- 背景：单体应用代码量 80 万行，发布周期 2 周，严重制约业务迭代
- 挑战：
  - 涉及 12 个业务域、5 个技术中台、30+ 微服务
  - 需要在不停服情况下完成平滑迁移
  - 跨 5 个部门协调，需要统一技术规范
- 架构设计：
  - 领域驱动设计（DDD）划分业务边界
  - 服务间通信：同步 HTTP + 异步消息
  - 分布式事务：Seata AT 模式
  - 可观测性：Skywalking + ELK + Prometheus
- 结果：
  - 发布周期从 2 周缩短至 1 天，迭代效率提升 10 倍
  - 系统可用性从 99.9% 提升至 99.99%
  - 获得公司年度技术革新奖
```

### P6 vs P7 简历差异对比

| 维度 | P6 | P7 |
|------|------|------|
| **项目规模** | 单模块/单系统 | 跨团队/公司级 |
| **职责描述** | 负责XX系统设计 | 定义技术方向 |
| **影响范围** | 本团队 | 多个团队/全公司 |
| **技术决策** | 执行方案 | 定义标准 |
| **团队贡献** | 指导 P5 | 培养 P6/P7 |
| **成果表达** | 系统性能提升 | 业务价值/技术影响力 |

## ⚠️ 常见问题

### 问题 1：级别和实际能力不匹配

**表现**：工作 3 年写 P6 的内容，面试被问倒。

**正确做法**：诚实评估自己的能力级别，不要为了「好看」而夸大。

### 问题 2：P6 写的内容太像 P5

**表现**：P6 了，项目描述还是「做了什么功能」「用了什么技术」。

**正确做法**：P6 要展示「独立负责」的能力，从「执行者」转变为「设计者」。

### 问题 3：P7 写的内容太偏执行

**表现**：P7 了，还在详细描述怎么优化 SQL、怎么配置 Redis。

**正确做法**：P7 要展示「定义问题」和「影响范围」，而不是「解决问题」。

## 简历升级路径

### 从 P5 到 P6 的关键转变

| 维度 | P5 思维 | P6 思维 |
|------|---------|---------|
| **项目** | 参与功能开发 | 主导模块设计 |
| **问题** | 别人安排问题 | 发现并解决问题 |
| **技术** | 会用工具 | 选择工具、解决问题 |
| **协作** | 被指导 | 指导别人 |

### 从 P6 到 P7 的关键转变

| 维度 | P6 思维 | P7 思维 |
|------|---------|---------|
| **项目** | 主导单个系统 | 定义技术方向 |
| **问题** | 解决技术问题 | 发现业务问题 |
| **技术** | 选择实现方案 | 定义技术标准 |
| **协作** | 协调团队 | 影响多个团队 |
| **价值** | 系统性能提升 | 业务价值/技术影响力 |

## 常见问题

**Q：工作年限短，能跳级写高级别的简历吗？**

A：不建议。简历写得再好，面试时也会被问倒。但如果能力确实达到了，可以在简历中体现对应的能力。

**Q：P6 投 P7 的岗位，简历怎么写？**

A：重点突出「超出 P6 职责」的内容，如主导跨团队项目、定义技术规范、培养团队等。如果没有这些经历，建议先在当前级别积累。

**Q：简历需要区分「工作经历」和「项目经历」吗？**

A：需要。工作经历是「纵向」的时间线，项目经历是「横向」的能力展示。两者结合，能让面试官全面了解你。
