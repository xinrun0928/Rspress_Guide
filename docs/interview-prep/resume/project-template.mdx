# 项目描述模板与案例

简历上的项目描述，是面试官判断你技术水平的第一依据。好的项目描述能让你从「码农」变成「技术人」，差的描述则让人昏昏欲睡。

## 面试官最关心的 3 个问题

> **问题 1**：项目描述应该写几行字？太长和太短有什么问题？
> **问题 2**：如何在项目描述中展示「个人价值」而不是「团队功劳」？
> **问题 3**：不同级别的候选人，项目描述应该有什么差异？

## 项目描述的基本结构

### 简历项目结构黄金模板

```
【项目名称】（可选：项目类型/所属公司）
【项目背景】（1-2句话，数字说话）
【技术栈】（用到的核心技术）
【你的职责】（你在项目中的角色）
【核心成果】（用数据说话的结果）
【技术亮点】（可选：1-2个技术亮点）
```

### 各部分详解

| 部分 | 字数 | 核心要点 | 示例 |
|------|------|----------|------|
| 项目名称 | 5-15 字 | 简洁、有辨识度 | 交易中台系统、订单履约中心 |
| 项目背景 | 1-2 句 | 用数字说明挑战 | 日均 50 万订单，峰值 QPS 5000 |
| 技术栈 | 3-5 项 | 与项目相关 | Java、Redis、Kafka、MySQL |
| 你的职责 | 1 句 | 明确角色 | 独立负责、高并发模块设计 |
| 核心成果 | 1-2 句 | 量化结果 | QPS 提升 3 倍，响应时间降低 60% |
| 技术亮点 | 0-2 项 | 展示深度 | 自研限流算法、零故障迁移 |

## 优秀项目描述案例

### 案例 1：性能优化类项目

**原始写法**（❌）：

```
订单系统性能优化项目
对公司订单系统进行性能优化，使用缓存，减少数据库查询，提升了系统性能。
```

**优化写法**（✅）：

```
订单系统性能优化（独立负责）
- 背景：订单系统日均处理 50 万订单，峰值 QPS 5000+，响应时间 > 2s，用户下单成功率仅 85%
- 技术栈：Java、Redis、MySQL、Kafka
- 职责：独立负责性能优化全流程
- 行动：
  - 使用 Arthas 定位瓶颈，定位到慢查询集中在订单查询接口
  - 设计三级缓存方案：JVM 本地缓存 + Redis 分布式缓存 + MySQL 旁路缓存
  - 优化 MySQL 索引，查询时间从 800ms 降至 20ms
  - 引入 Kafka 异步处理非核心流程，削峰填谷
- 结果：响应时间从 2s 降至 300ms，QPS 提升 3 倍，订单成功率提升至 99.5%
```

### 案例 2：架构重构类项目

**原始写法**（❌）：

```
参与公司微服务改造项目，用 Spring Cloud 重构了原有系统。
```

**优化写法**（✅）：

```
微服务架构演进项目（技术负责人）
- 背景：单体应用代码量 50 万行，每次发布需要 3 人值守，开发周期 2 周，严重制约业务迭代速度
- 技术栈：Spring Cloud Alibaba、Nacos、Seata、Redis、Kafka
- 职责：作为核心架构师，主导服务拆分方案设计和技术选型
- 行动：
  - 分析业务边界，采用领域驱动设计（DDD）划分服务，拆分为 12 个微服务
  - 设计服务间通信方案，HTTP 调用用于同步接口，消息队列用于异步解耦
  - 引入 Seata 分布式事务解决方案，解决跨服务数据一致性
  - 设计灰度发布方案，实现流量无损切换
- 结果：发布周期从 2 周缩短至 1 天，团队开发效率提升 40%，系统可用性 99.99%
```

### 案例 3：业务增长类项目

**原始写法**（❌）：

```
做了推荐系统，提升了用户体验。
```

**优化写法**（✅）：

```
个性化推荐系统（核心开发）
- 背景：App 首页推荐转化率仅 1.2%，远低于行业平均水平 2.5%，每日流失潜在订单约 5000 单
- 技术栈：Python、TensorFlow、Redis、Spark
- 职责：作为算法工程师，主导推荐算法设计和工程落地
- 行动：
  - 设计基于用户行为序列的协同过滤算法，实现实时推荐
  - 构建特征工程 pipeline，提取用户、商品、上下文三维特征 200+ 个
  - 引入 A/B 测试框架，支持算法快速迭代验证
  - 工程优化：模型推理耗时从 50ms 降至 8ms，支持日均 1 亿次调用
- 结果：推荐点击率从 1.2% 提升至 3.8%，提升 3.2 倍，带来日均 GMV 增长 15 万
```

### 案例 4：高可用保障类项目

**原始写法**（❌）：

```
做了监控系统，保障系统稳定运行。
```

**优化写法**（✅）：

```
全链路监控系统（独立负责）
- 背景：系统故障定位耗时 2-4 小时，严重影响 SLA，SLO 达成率仅 95%
- 技术栈：Prometheus、Grafana、ELK、Kafka、Spring Boot Actuator
- 职责：从 0 到 1 设计并落地全链路监控体系
- 行动：
  - 设计 metrics、logging、tracing 三位一体监控方案
  - 接入 200+ 微服务，覆盖核心链路 50 条，告警规则 300+ 条
  - 实现故障定位自动化，异常链路自动标注，平均定位时间降至 10 分钟
  - 设计「红黄绿」分级告警机制，减少无效告警 80%
- 结果：SLO 达成率提升至 99.9%，故障平均定位时间从 2 小时降至 10 分钟
```

## ⚠️ 项目描述常见问题

### 问题 1：项目描述太空洞

**错误示例**：

```
负责系统开发，使用先进技术，提升系统性能，得到领导和客户好评。
```

**问题诊断**：没有具体信息，看不出任何价值。

**正确写法**：用数字、具体技术、明确结果替换。

### 问题 2：把所有项目都写成一个样

**错误示例**：

```
项目A：负责XX系统开发，使用Java、MySQL、Redis，系统运行良好。
项目B：负责XX系统开发，使用Java、MySQL、Redis，系统运行良好。
```

**问题诊断**：看不出每个项目的独特价值。

**正确写法**：每个项目要有独特的技术挑战和成果，如果两个项目太相似，考虑合并。

### 问题 3：项目描述太长

**错误示例**：每个项目写 500 字，5 个项目写了 2500 字。

**问题诊断**：HR 平均看简历的时间只有 6-7 秒，太长的描述会被跳过。

**正确写法**：核心项目写 100-150 字，其他项目写 50-80 字。

### 问题 4：忽略个人贡献

**错误示例**：

```
团队使用微服务架构，成功上线系统，系统稳定运行。
```

**问题诊断**：「团队」做了什么，你做了什么？

**正确写法**：用「我」开头，明确你的职责和贡献。

## 不同级别的项目描述差异

### P5 级别（初/中级）

**描述重点**：

- 参与的具体功能
- 使用的技术栈
- 解决的具体问题

**示例**：

```
用户权限系统（独立负责）
- 为公司内部系统设计基于 RBAC 的权限模型，支持细粒度权限控制
- 使用 Spring Security + JWT 实现认证授权，接口响应时间 < 50ms
- 接入 10+ 业务系统，累计服务 5000+ 用户
```

### P6 级别（高级）

**描述重点**：

- 主导的模块或系统
- 技术方案设计能力
- 性能优化或架构改进
- 量化结果

**示例**：

```
交易中台系统（技术负责人）
- 主导设计交易中台架构，支持日均 100 万订单、峰值 QPS 10000+
- 设计多级缓存、分库分表、异步消息等高性能方案
- 核心接口 P99 响应时间 < 100ms，系统可用性 99.99%
- 沉淀通用组件 10+ 个，供其他团队复用
```

### P7 级别（专家/架构师）

**描述重点**：

- 主导的跨团队项目
- 技术战略和方向把控
- 团队规模和技术影响力
- 业务价值和社会效益

**示例**：

```
平台技术架构升级（架构师）
- 主导公司级技术架构升级项目，涉及 5 个部门、20+ 服务、30+ 人
- 制定微服务、云原生、DevOps 三大技术方向，规划 3 年技术演进路线
- 推动落地后，研发效能提升 50%，资源成本降低 30%
- 建立公司级技术委员会，培养核心技术人才 10+ 人
```

## 💡 加分技巧：项目描述的「钩子」设计

好的项目描述就像「钩子」，能吸引面试官追问。

### 钩子 1：技术挑战钩子

在项目描述中留下技术细节的「钩子」：

```
使用 Redis Cluster + Codis 双写方案解决缓存高可用问题……
```

面试官可能会追问：「为什么用双写？主从同步不行吗？」

### 钩子 2：数字钩子

留下可追问的数字：

```
QPS 从 1000 提升至 5000……
```

面试官可能会追问：「具体做了哪些优化？」

### 钩子 3：架构钩子

留下架构设计的「钩子」：

```
采用 CQRS 模式分离读写流量……
```

面试官可能会追问：「为什么用 CQRS？带来了什么好处？」

## 项目描述检查清单

- [ ] 项目名称简洁有辨识度
- [ ] 项目背景用数字说明挑战
- [ ] 技术栈与项目相关
- [ ] 明确你的角色（独立负责/核心参与）
- [ ] 用 STAR 法则组织描述
- [ ] 核心成果用数字量化
- [ ] 没有「团队完成」「系统运行良好」等空洞表述
- [ ] 每个项目有独特性，不重复
- [ ] 总字数控制在 800-1200 字（3-4 个项目）

## 常见问题

**Q：简历上有几个项目比较好？**

A：建议写 3-4 个核心项目。太多显得没有重点，太少显得经历单调。优先写最能体现你技术深度的项目。

**Q：项目经历不够怎么办？**

A：如果项目经历少，可以写得更详细，把每个项目的技术细节展开。也可以写一些技术实践（如参与开源项目、自己做的 side project）。

**Q：项目描述和 STAR 法则有什么关系？**

A：STAR 法则适合面试时的口头表达，简历上的项目描述是 STAR 的「浓缩版」。简历上不用写完整的 STAR，只需要把核心要素（背景、行动、结果）写清楚即可。
