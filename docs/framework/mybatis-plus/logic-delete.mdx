# MyBatis Plus 逻辑删除：删除不等于消失

<Badge text="MyBatis Plus" type="tip" /> <Badge text="核心功能" type="info" />

在业务系统中，"删除"数据往往不是真的从数据库抹掉，而是打上一个"已删除"的标记。

这就是**逻辑删除**——数据还在，只是查询时不显示；必要时可以恢复。

MyBatis Plus 的 `@TableLogic` 注解，让你用一行配置实现逻辑删除。

## 物理删除 vs 逻辑删除

### 物理删除

```sql
-- 物理删除：数据真的没了
DELETE FROM user WHERE id = 1;

-- 数据永久丢失，无法恢复
```

### 逻辑删除

```sql
-- 逻辑删除：数据还在，只是标记为已删除
UPDATE user SET deleted = 1 WHERE id = 1;

-- 查询时自动加上 deleted = 0
SELECT * FROM user WHERE deleted = 0;
```

| 特性 | 物理删除 | 逻辑删除 |
|------|---------|---------|
| 数据 | 永久删除 | 保留 |
| 可恢复 | 否 | 是 |
| 查询 | 直接查询 | 需加条件 |
| 性能 | 快 | 稍慢（需要索引） |
| 审计 | 难 | 易 |

## @TableLogic 注解

### 基本用法

```java
@TableName("user")
public class User {

    @TableId(type = IdType.AUTO)
    private Long id;

    private String name;

    @TableLogic  // 逻辑删除字段
    private Integer deleted;
}
```

**效果**：
- 删除操作：`UPDATE user SET deleted = 1 WHERE id = ? AND deleted = 0`
- 查询操作：`SELECT ... FROM user WHERE deleted = 0`

### 实体类配置

```java
@TableName("user")
public class User {

    @TableId(type = IdType.AUTO)
    private Long id;

    private String name;

    // 逻辑删除字段
    @TableLogic
    private Integer deleted;
}
```

**使用**：

```java
// 删除（变成 UPDATE）
userMapper.deleteById(1L);
// UPDATE user SET deleted = 1 WHERE id = 1 AND deleted = 0

// 查询（自动加上 deleted = 0）
userMapper.selectById(1L);
// SELECT * FROM user WHERE id = 1 AND deleted = 0
```

## 全局配置

### application.yml

```yaml
mybatis-plus:
  global-config:
    db-config:
      # 逻辑删除字段名
      logic-delete-field: deleted
      # 删除后的值
      logic-delete-value: 1
      # 未删除的值
      logic-not-delete-value: 0
```

### 注解 vs 全局配置

| 配置方式 | 说明 |
|---------|------|
| `@TableLogic` 注解 | 优先级高，覆盖全局配置 |
| 全局配置 | 所有表共用一套配置 |

```java
// 注解配置（优先级高）
@TableLogic(value = "1", delval = "0")
private Integer deleted;

// 全局配置（优先级低）
mybatis-plus:
  global-config:
    db-config:
      logic-delete-value: 1
      logic-not-delete-value: 0
```

## 逻辑删除值自定义

### 常见场景

```java
// 方式一：使用 Integer
@TableLogic
private Integer deleted;  // 1 = 已删除，0 = 未删除

// 方式二：使用 String
@TableLogic
private String deleted;  // 'Y' = 已删除，'N' = 未删除

// 方式三：使用 LocalDateTime（软删除时间）
@TableLogic
private LocalDateTime deletedTime;  // 非空 = 已删除
```

### 不同类型配置

```java
// Integer 类型
@TableLogic(value = "1", delval = "0")
private Integer deleted;

// String 类型
@TableLogic(value = "'Y'", delval = "'N'")
private String deleted;

// LocalDateTime 类型（删除时设置为当前时间）
@TableField(fill = FieldFill.INSERT)
private LocalDateTime deletedTime;  // 非空即为删除
```

## 查询方法的影响

### 自动添加条件

| 方法 | 效果 |
|------|------|
| `selectById()` | `WHERE id = ? AND deleted = 0` |
| `selectList()` | `WHERE deleted = 0` |
| `selectCount()` | `WHERE deleted = 0` |
| `update()` | `WHERE deleted = 0` |
| `deleteById()` | `UPDATE SET deleted = 1 WHERE id = ? AND deleted = 0` |

### 不受影响的场景

```java
// 使用 Wrapper 指定条件时，不会自动添加 deleted = 0
LambdaQueryWrapper<User> wrapper = new LambdaQueryWrapper<>();
wrapper.eq("deleted", 1);  // 查询已删除的记录
List<User> deletedUsers = userMapper.selectList(wrapper);
```

## 恢复已删除数据

### 方法一：手动更新

```java
// 恢复已删除的数据
userMapper.update(null,
    new LambdaUpdateWrapper<User>()
        .set(User::getDeleted, 0)
        .eq(User::getId, 1L)
);
```

### 方法二：自定义方法

```java
@Mapper
public interface UserMapper extends BaseMapper<User> {

    // 恢复删除
    @Update("UPDATE user SET deleted = 0 WHERE id = #{id}")
    int restore(@Param("id") Long id);

    // 彻底删除（物理删除）
    @Delete("DELETE FROM user WHERE id = #{id}")
    int physicallyDelete(@Param("id") Long id);
}
```

## 逻辑删除与唯一索引

### 问题

当字段配置了唯一索引时，逻辑删除可能出问题：

```sql
-- 表结构
CREATE TABLE user (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(50) NOT NULL UNIQUE,  -- 唯一索引
    deleted TINYINT DEFAULT 0
);
```

```java
// 插入数据
User user = new User();
user.setName("张三");
userMapper.insert(user);  // deleted = 0

// 删除数据
userMapper.deleteById(1L);  // deleted = 1

// 再次插入同名数据
user.setName("张三");
userMapper.insert(user);
// 报错！唯一索引冲突（deleted = 0 的行已经存在）
```

### 解决方案

#### 方案一：使用唯一索引 + deleted

```sql
-- 复合唯一索引
CREATE TABLE user (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(50) NOT NULL,
    deleted TINYINT DEFAULT 0,
    UNIQUE KEY uk_name_deleted (name, deleted)
);
```

#### 方案二：使用时间戳代替 deleted

```sql
-- 使用删除时间代替状态
CREATE TABLE user (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(50) NOT NULL,
    deleted_at DATETIME  -- NULL = 未删除，非空 = 已删除
);
```

```java
@TableName("user")
public class User {

    @TableId(type = IdType.AUTO)
    private Long id;

    private String name;

    @TableField(fill = FieldFill.INSERT)
    private LocalDateTime deletedAt;  // 删除时间
}
```

#### 方案三：不使用唯一索引

如果必须保证 `name` 唯一，可以：
1. 不使用逻辑删除
2. 使用 UUID + name 组合
3. 软删除后再插入时报错提醒用户

## 多租户场景

### 配置

```java
@Configuration
public class MyBatisPlusConfig {

    @Bean
    public MybatisPlusInterceptor mybatisPlusInterceptor() {
        MybatisPlusInterceptor interceptor = new MybatisPlusInterceptor();

        // 多租户插件
        interceptor.addInnerInterceptor(new TenantLineInnerInterceptor(new TenantLineHandlerImpl()));

        // 分页插件（多租户之后添加）
        interceptor.addInnerInterceptor(new PaginationInnerInterceptor());

        return interceptor;
    }
}
```

### SQL 效果

```sql
-- 查询
SELECT * FROM user WHERE deleted = 0 AND tenant_id = 1

-- 删除
UPDATE user SET deleted = 1 WHERE id = 1 AND deleted = 0 AND tenant_id = 1
```

## 注意事项

### 1. 索引问题

```java
// deleted 字段必须加索引
@TableField("deleted")
private Integer deleted;
```

建议为 `deleted` 字段添加索引：

```sql
CREATE INDEX idx_deleted ON user(deleted);
```

### 2. 批量删除

```java
// 批量删除
List<Long> ids = Arrays.asList(1L, 2L, 3L);
userMapper.deleteBatchIds(ids);
// UPDATE user SET deleted = 1 WHERE id IN (1, 2, 3) AND deleted = 0
```

### 3. Service 层删除

```java
@Service
public class UserServiceImpl extends ServiceImpl<UserMapper, User> implements UserService {

    @Override
    public boolean removeById(Long id) {
        // 逻辑删除
        return super.removeById(id);
    }

    @Override
    public boolean physicallyRemoveById(Long id) {
        // 物理删除
        return baseMapper.deleteById(id) > 0;
    }
}
```

## 常见问题

### Q1：为什么查询不到已删除的数据？

因为 `BaseMapper` 的查询方法会自动添加 `deleted = 0` 条件。

### Q2：如何查询已删除的数据？

```java
// 使用 Wrapper 指定 deleted 值
LambdaQueryWrapper<User> wrapper = new LambdaQueryWrapper<>();
wrapper.eq(User::getDeleted, 1);
List<User> deletedUsers = userMapper.selectList(wrapper);
```

### Q3：物理删除和逻辑删除如何共存？

```java
// 逻辑删除（默认）
userMapper.deleteById(1L);

// 物理删除
userMapper.delete(new LambdaQueryWrapper<User>()
    .eq(User::getDeleted, 1)
    .eq(User::getId, 1L)
);
```

### Q4：逻辑删除字段是 String 类型怎么配置？

```java
@TableLogic(value = "'Y'", delval = "'N'")
private String deleted;
```

注意：字符串值需要用单引号包裹。

## 总结

### 配置步骤

1. **添加 deleted 字段**到数据库表
2. **实体类添加 `@TableLogic` 注解**
3. **可选：配置全局默认值**

### 配置选项

| 选项 | 说明 | 默认值 |
|------|------|--------|
| `value` | 未删除的值 | `0` |
| `delval` | 已删除的值 | `1` |

### 效果

| 操作 | SQL 变化 |
|------|---------|
| `deleteById()` | `DELETE` → `UPDATE SET deleted = 1 WHERE ... AND deleted = 0` |
| `selectById()` | `WHERE id = ?` → `WHERE id = ? AND deleted = 0` |
| `selectList()` | 自动添加 `WHERE deleted = 0` |

逻辑删除是业务系统中非常重要的功能，合理使用可以保护数据安全，便于审计和恢复。
