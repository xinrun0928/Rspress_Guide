# MyBatis 缓存：一级缓存与二级缓存的秘密

MyBatis 有缓存吗？有，而且有两级。

很多人在使用 MyBatis 时，并不知道它内置了缓存机制。或者知道有一级缓存、二级缓存，但不清楚它们的工作原理，导致在某些场景下出现"数据不一致"的问题。

这篇文章，让你彻底搞懂 MyBatis 的缓存设计。

## 为什么需要缓存？

缓存的核心思想是：**用空间换时间**。

```
没有缓存：
    查询请求 → 执行 SQL → 返回结果
    查询请求 → 执行 SQL → 返回结果
    查询请求 → 执行 SQL → 返回结果

有缓存：
    查询请求 → 执行 SQL → 存入缓存 → 返回结果
    查询请求 → 查缓存 → 返回结果（跳过 SQL）
    查询请求 → 查缓存 → 返回结果（跳过 SQL）
```

数据库查询是系统中最耗时的操作之一。通过缓存，可以显著减少数据库的访问次数，提升系统性能。

## MyBatis 缓存的两级架构

```
┌─────────────────────────────────────────────────────────────┐
│                        SqlSession                            │
│  ┌───────────────────────────────────────────────────────┐  │
│  │                   一级缓存 (Local Cache)                │  │
│  │                                                       │  │
│  │   ┌─────────────────────────────────────────────┐    │  │
│  │   │  PerpetualCache (HashMap 实现)              │    │  │
│  │   │  作用域：同一 SqlSession 内                   │    │  │
│  │   └─────────────────────────────────────────────┘    │  │
│  │                                                       │  │
│  └───────────────────────────────────────────────────────┘  │
│                         ↓                                   │
│  ┌───────────────────────────────────────────────────────┐  │
│  │                   二级缓存 (Second Level Cache)        │  │
│  │                                                       │  │
│  │   ┌─────────────────────────────────────────────┐    │  │
│  │   │  CachingExecutor (装饰器模式)                │    │  │
│  │   │  作用域：Mapper 级别（namespace）             │    │  │
│  │   └─────────────────────────────────────────────┘    │  │
│  │                                                       │  │
│  └───────────────────────────────────────────────────────┘  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 一级缓存：SqlSession 级别

### 工作原理

一级缓存是 MyBatis **默认开启**的，它存在于 `BaseExecutor` 中，使用 `PerpetualCache` 实现，本质是一个 `HashMap`：

```java
public class PerpetualCache implements Cache {
    private final String id;
    private final Map<Object, Object> cache = new HashMap<>();

    @Override
    public void putObject(Object key, Object value) {
        cache.put(key, value);
    }

    @Override
    public Object getObject(Object key) {
        return cache.get(key);
    }

    @Override
    public Object removeObject(Object key) {
        return cache.remove(key);
    }

    @Override
    public void clear() {
        cache.clear();
    }
}
```

### 缓存命中条件

一级缓存的 key 由以下因素组成：

```java
public class CacheKey {

    private final List<Object> updateList;

    @Override
    public boolean equals(Object object) {
        // CacheKey 相等的条件：
        // 1. MappedStatement 的 id（SQL 语句）
        // 2. RowBounds 的 offset 和 limit
        // 3. SQL 的参数值
        // 4. Environment 的 id
    }
}
```

**命中缓存的条件**：

1. **相同的 StatementId**：同一个 SQL 语句
2. **相同的参数**：查询条件完全一致
3. **相同的 RowBounds**：分页参数一致
4. **同一个 SqlSession**：会话未关闭

```java
// 示例：两次查询同一用户
// 第一次：执行 SQL，存入缓存
User user1 = userMapper.selectById(1L);
// SQL: SELECT * FROM user WHERE id = 1

// 第二次：命中缓存，不会执行 SQL
User user2 = userMapper.selectById(1L);
// 直接从缓存返回
```

### 缓存的查询流程

```java
// BaseExecutor.query()
public <E> List<E> query(MappedStatement ms, Object parameter, RowBounds rowBounds,
                          ResultHandler resultHandler, CacheKey key, BoundSql boundSql) {
    // 1. 检查是否有嵌套查询（association/collection）
    if (ms.hasNestedResultMaps()) {
        // 有嵌套查询，不使用缓存
        return doQuery(ms, parameter, rowBounds, resultHandler, boundSql);
    }

    // 2. 从一级缓存查询
    List<E> list = queryFromCache(key);

    if (list == null) {
        // 3. 缓存未命中，查询数据库
        list = doQuery(ms, parameter, rowBounds, resultHandler, boundSql);

        // 4. 存入一级缓存
        addToCache(key, list);
    }

    return list;
}
```

### 何时清空缓存？

一级缓存在以下情况会被清空：

| 操作 | 是否清空缓存 |
|------|-------------|
| `sqlSession.close()` | 不清空（缓存随 SqlSession 关闭） |
| `sqlSession.commit()` | 清空（事务提交，数据可能变化） |
| `sqlSession.rollback()` | 清空（事务回滚，数据恢复） |
| `sqlSession.clearCache()` | 清空（显式清空） |
| 执行增删改操作 | 清空（数据可能变化） |

```java
// BaseExecutor.update()
@Override
public int update(MappedStatement ms, Object parameter) throws SQLException {
    // 清空一级缓存
    clearLocalCache();
    return doUpdate(ms, parameter);
}
```

### 作用域配置

```xml
<settings>
    <!-- SESSION：缓存存活于 SqlSession 生命周期（默认） -->
    <!-- STATEMENT：每条语句执行后清空缓存 -->
    <setting name="localCacheScope" value="SESSION"/>
</settings>
```

**SESSION vs STATEMENT**：

| 作用域 | 缓存时机 | 适用场景 |
|--------|---------|----------|
| SESSION | SqlSession 生命周期内有效 | 大多数场景，数据一致性要求不高 |
| STATEMENT | 每条语句执行后清空 | 高并发，数据一致性要求高的场景 |

## 二级缓存：Mapper 级别

### 工作原理

二级缓存由 `CachingExecutor` 实现，它是一个**装饰器**，包装了其他执行器：

```java
public class CachingExecutor implements Executor {

    private final Executor delegate;

    @Override
    public <E> List<E> query(MappedStatement ms, Object parameter, RowBounds rowBounds,
                              ResultHandler resultHandler, CacheKey key, BoundSql boundSql) {
        // 1. 检查是否启用二级缓存
        if (ms.isCacheEnabled()) {
            // 2. 获取缓存对象
            Cache cache = ms.getCache();

            if (cache != null) {
                // 3. 尝试从缓存获取
                CacheKey cacheKey = createCacheKey(ms, parameter, rowBounds, boundSql);

                if (cache.containsKey(cacheKey)) {
                    // 缓存命中！返回缓存数据
                    return delegate.<E>query(ms, parameter, rowBounds,
                                              resultHandler, boundSql, cacheKey);
                }
            }
        }

        // 4. 缓存未命中，查询数据库
        List<E> list = delegate.<E>query(ms, parameter, rowBounds,
                                          resultHandler, boundSql, key);

        // 5. 存入二级缓存
        if (ms.isCacheEnabled()) {
            tcm.putObject(cache, cacheKey, list);
        }

        return list;
    }
}
```

### 二级缓存的查询流程

```
查询请求
    ↓
先查二级缓存（CachingExecutor）
    ↓ 命中
返回缓存结果

    ↓ 未命中
再查一级缓存（BaseExecutor）
    ↓ 命中
返回一级缓存结果

    ↓ 未命中
执行数据库查询
    ↓
结果存入一级缓存
    ↓
结果存入二级缓存
    ↓
返回结果
```

### 如何开启二级缓存？

**第一步：配置开启**

```xml
<settings>
    <!-- 开启二级缓存（默认 true） -->
    <setting name="cacheEnabled" value="true"/>
</settings>
```

**第二步：在 Mapper XML 中声明**

```xml
<mapper namespace="com.example.UserMapper">
    <!-- 开启当前 Mapper 的二级缓存 -->
    <cache/>

    <!-- 或者自定义缓存配置 -->
    <cache eviction="LRU" flushInterval="60000" size="256" readOnly="true"/>
</mapper>
```

### 缓存配置参数

| 参数 | 说明 | 可选值 |
|------|------|--------|
| `eviction` | 缓存淘汰策略 | `LRU`（默认）、`FIFO`、`SOFT`、`WEAK` |
| `flushInterval` | 刷新间隔（毫秒） | 默认不刷新 |
| `size` | 缓存容量 | 默认 1024 |
| `readOnly` | 是否只读 | `true`（只读，性能更好）、`false`（可读写） |

### 事务缓存：TransactionalCache

二级缓存有一个特殊问题：**脏读**。

考虑这个场景：

```java
SqlSession sqlSession = sqlSessionFactory.openSession();
UserMapper mapper = sqlSession.getMapper(UserMapper.class);

User user = mapper.selectById(1L);  // 查询，存入缓存
System.out.println(user.getName()); // 张三

mapper.updateName(1L, "李四");       // 更新，缓存应该清空吗？

User user2 = mapper.selectById(1L);  // 这次查缓存还是数据库？
System.out.println(user2.getName()); // ???
```

MyBatis 使用 `TransactionalCache` 解决这一问题：

```java
public class TransactionalCache implements Cache {

    // 待提交到缓存的数据
    private final Map<Object, Object> entriesToAddOnCommit = new HashMap<>();

    // 需要清除的缓存 key
    private final Set<Object> entriesToRemoveOnCommit = new HashSet<>();

    @Override
    public void putObject(Object key, Object value) {
        // 不直接放入缓存，而是放入待提交队列
        entriesToAddOnCommit.put(key, value);
    }

    @Override
    public Object getObject(Object key) {
        // 先检查是否需要移除
        if (entriesToRemoveOnCommit.contains(key)) {
            return null;
        }
        // 再从待提交队列获取
        return entriesToAddOnCommit.get(key);
    }

    public void commit() {
        // 提交时，将待提交数据真正放入缓存
        for (Map.Entry<Object, Object> entry : entriesToAddOnCommit.entrySet()) {
            delegate.putObject(entry.getKey(), entry.getValue());
        }

        // 清除需要移除的数据
        for (Object key : entriesToRemoveOnCommit) {
            delegate.removeObject(key);
        }
    }

    public void rollback() {
        // 回滚时，清空所有待提交数据
        clear();
    }
}
```

**工作流程**：

```
增删改操作 → 标记缓存失效（entriesToRemoveOnCommit）
    ↓
commit() 执行时
    ↓
移除标记的缓存
    ↓
添加新数据到缓存
```

## 缓存淘汰策略

### LRU（Least Recently Used）

最近最少使用，**默认策略**。当缓存满时，移除最长时间未使用的缓存项。

```java
public class LruCache implements Cache {
    private final Cache delegate;
    private final Map<Object, Object> keyMap = new LinkedHashMap<>(16, 0.75f, true);

    @Override
    public void putObject(Object key, Object value) {
        delegate.putObject(key, value);
        cycleKeyList(key);
    }

    private void cycleKeyList(Object key) {
        keyMap.put(key, key);
        if (keyMap.size() > size) {
            // 移除最老的 key
            Object oldestKey = keyMap.keySet().iterator().next();
            delegate.removeObject(oldestKey);
            keyMap.remove(oldestKey);
        }
    }
}
```

### FIFO（First In First Out）

先进先出，当缓存满时，移除最早进入的缓存项。

### SOFT（Soft Reference）

软引用，当内存不足时，GC 会回收软引用指向的对象。

### WEAK（Weak Reference）

弱引用，每次 GC 都会回收弱引用指向的对象。

## 序列化问题

当 `readOnly="false"` 时，缓存对象需要**序列化**：

```xml
<cache readOnly="false"/>
```

```java
public class SerializedCache implements Cache {
    @Override
    public void putObject(Object key, Object value) {
        // 序列化后存储
        delegate.putObject(key, serialize(value));
    }

    @Override
    public Object getObject(Object key) {
        // 反序列化后返回
        return deserialize(delegate.getObject(key));
    }
}
```

**实体类需要实现 Serializable**：

```java
public class User implements Serializable {
    private static final long serialVersionUID = 1L;

    private Long id;
    private String name;
    private Integer age;
    // ...
}
```

## 一级缓存 vs 二级缓存对比

| 特性 | 一级缓存 | 二级缓存 |
|------|---------|---------|
| 作用域 | SqlSession | Mapper（namespace） |
| 存储位置 | 内存（HashMap） | 内存/磁盘（可配置） |
| 开启方式 | 默认开启 | 需显式配置 |
| 缓存存储 | `BaseExecutor` | `CachingExecutor` |
| 失效时机 | commit/rollback/clear/update | 增删改操作（同一 namespace） |
| 序列化需求 | 无 | `readOnly=false` 时需要 |

## 延伸思考

### 为什么一级缓存需要事务保护？

考虑以下场景：

```java
SqlSession sqlSession = sqlSessionFactory.openSession();

User u1 = mapper.selectById(1L);  // 缓存：id=1 → User("张三")
System.out.println(u1.getName());  // 张三

mapper.updateName(1L, "李四");     // 更新数据库，但未提交

User u2 = mapper.selectById(1L);  // 查缓存，返回"张三"，还是查数据库，返回"李四"？
```

在**自动提交**模式下（`openSession(true)`），MyBatis 会在每次增删改后自动 `commit()`，这会清空一级缓存，所以 u2 会查询数据库。

在**手动提交**模式下，commit() 之前，u2 会命中一级缓存，返回"张三"。

### 二级缓存的脏读问题

二级缓存的脏读问题更严重：

```java
// 会话 1
SqlSession s1 = sqlSessionFactory.openSession();
UserMapper m1 = s1.getMapper(UserMapper.class);
User u1 = m1.selectById(1L);  // 缓存：id=1 → User("张三")

// 会话 2（不同 SqlSession）
SqlSession s2 = sqlSessionFactory.openSession();
UserMapper m2 = s2.getMapper(UserMapper.class);
m2.updateName(1L, "李四");      // 更新了数据库，但二级缓存还是"张三"
s2.commit();

// 会话 1（继续使用原来的 SqlSession）
User u2 = m1.selectById(1L);   // 查二级缓存，返回"张三"（脏数据）
```

**解决方案**：

1. 不要在有并发修改的场景下使用二级缓存
2. 使用 Redis 等分布式缓存代替
3. 增删改时显式清空缓存

### 你可能想问的问题

**Q1：二级缓存为什么不适合分布式环境？**

因为二级缓存默认存在本地内存中，不同服务器的缓存无法共享。分布式场景下，应该使用 Redis 等分布式缓存。

**Q2：如何查看缓存是否命中？**

开启 MyBatis 日志：

```xml
<settings>
    <setting name="logImpl" value="STDOUT_LOGGING"/>
</settings>
```

日志中会显示缓存命中情况。

**Q3：如何禁用某个查询的二级缓存？**

```xml
<select id="selectById" resultType="User" useCache="false">
    SELECT * FROM user WHERE id = #{id}
</select>
```

理解 MyBatis 的缓存设计，你就能在性能和数据一致性之间找到平衡。一级缓存是默认的、透明的优化；二级缓存则需要谨慎使用，因为它引入了数据一致性的复杂性。
