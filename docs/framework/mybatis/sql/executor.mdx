# MyBatis Executor：三种执行器的选择

<Badge text="MyBatis" type="tip" /> <Badge text="核心原理" type="info" />

SimpleExecutor、ReuseExecutor、BatchExecutor——我该用哪个？

大多数人在使用 MyBatis 时，几乎不会去配置执行器，默认的 `SimpleExecutor` 就够了。但当数据量上来，或者需要批量操作时，执行器的选择就变得至关重要。

这篇文章，让你彻底搞懂 MyBatis 执行器的设计哲学。

## 为什么需要执行器？

在 JDBC 中，每一次 SQL 执行都需要：

1. 创建 Connection
2. 创建 Statement/PreparedStatement
3. 执行 SQL
4. 关闭 Statement
5. 关闭 Connection

这个过程有两个关键问题：

- **Connection 创建开销大**：建立 TCP 连接、三次握手、TLS 握手（如果是 HTTPS）等，都是耗时操作
- **Statement 重复创建**：如果执行 1000 条相似的 INSERT 语句，会创建 1000 个 PreparedStatement 对象

MyBatis 的执行器，就是为了解决这些问题。执行器封装了 Statement 的管理和复用逻辑，让你可以根据场景选择最优的执行策略。

## BaseExecutor：模板方法模式

三个执行器都继承自 `BaseExecutor`，它采用了**模板方法模式**，定义了一套完整的执行骨架：

```java
public abstract class BaseExecutor implements Executor {

    // 一级缓存，SqlSession 级别
    protected PerpetualCache localCache;
    protected PerpetualCache localOutputParameterCache;

    protected Configuration configuration;
    protected Transaction transaction;

    @Override
    public <E> List<E> query(MappedStatement ms, Object parameter, RowBounds rowBounds,
                              ResultHandler resultHandler, CacheKey cacheKey, BoundSql boundSql) {
        // 1. 检查一级缓存
        if (configuration.getLocalCacheScope() == LocalCacheScope.STATEMENT) {
            clearLocalCache();
        }

        List<E> list;
        try {
            // 2. 先查缓存
            list = queryFromCache(cacheKey);
            if (list == null) {
                // 3. 缓存未命中，调用子类方法执行查询
                list = doQuery(ms, parameter, rowBounds, resultHandler, boundSql);
                // 4. 结果存入缓存
                addToCache(cacheKey, list);
            }
        } catch (Exception e) {
            throw ExceptionUtil.unwrapThrowable(e);
        } finally {
            // 5. 清理缓存（如果配置了 STATEMENT 级别）
        }
        return list;
    }

    @Override
    public int update(MappedStatement ms, Object parameter) throws SQLException {
        // 清空缓存
        clearLocalCache();
        // 调用子类方法
        return doUpdate(ms, parameter);
    }

    @Override
    public <E> Cursor<E> queryCursor(MappedStatement ms, Object parameter,
                                      RowBounds rowBounds, BoundSql boundSql) {
        return doQueryCursor(ms, parameter, rowBounds, boundSql);
    }

    // flushStatements 用于批量操作，清理暂存的语句
    @Override
    public List<BatchResult> flushStatements() throws SQLException {
        return flushStatements(false);
    }

    // 模板方法，子类必须实现
    protected abstract <E> List<E> doQuery(MappedStatement ms, Object parameter,
                                             RowBounds rowBounds, ResultHandler resultHandler,
                                             BoundSql boundSql) throws SQLException;

    protected abstract <E> Cursor<E> doQueryCursor(MappedStatement ms, Object parameter,
                                                    RowBounds rowBounds, BoundSql boundSql);

    protected abstract int doUpdate(MappedStatement ms, Object parameter) throws SQLException;
}
```

模板方法模式的精髓在于：**父类定义算法骨架，子类负责具体实现**。`BaseExecutor` 定义了缓存管理、超时控制、异常处理等通用逻辑，而 `doQuery()`、`doUpdate()` 等核心方法由子类实现。

## SimpleExecutor：每次都是新的开始

`SimpleExecutor` 是最简单的执行器，也是 MyBatis 的默认选择。每次执行 SQL，都会创建新的 `PreparedStatement`：

```java
public class SimpleExecutor extends BaseExecutor {

    @Override
    public <E> List<E> doQuery(MappedStatement ms, Object parameter,
                                 RowBounds rowBounds, ResultHandler resultHandler,
                                 BoundSql boundSql) {
        Statement stmt = null;
        try {
            // 1. 获取配置
            Configuration configuration = ms.getConfiguration();

            // 2. 创建 StatementHandler
            StatementHandler handler = configuration.newStatementHandler(
                this, ms, parameter, rowBounds, resultHandler, boundSql);

            // 3. 准备 Statement
            stmt = prepareStatement(handler, ms.getStatementType());

            // 4. 执行查询
            return handler.<E>query(stmt, resultHandler);
        } catch (Exception e) {
            throw ExceptionFactory.wrapException("query error", e);
        } finally {
            // 5. 关闭 Statement
            closeStatement(stmt);
        }
    }

    @Override
    protected Statement prepareStatement(StatementHandler handler, String statementType) {
        // 每次都创建新的 Connection
        Connection connection = transaction.getConnection();

        // 每次都创建新的 PreparedStatement
        Statement stmt = handler.prepare(connection, ms.getTimeout());

        // 设置参数
        handler.parameterize(stmt);

        return stmt;
    }
}
```

**SimpleExecutor 的特点：**

| 优点 | 缺点 |
|------|------|
| 实现简单，不存在状态共享问题 | 每次都创建新的 PreparedStatement，有一定开销 |
| 线程安全，无需考虑并发问题 | 无法利用预编译缓存 |
| 适用于大多数业务场景 | 不适合批量执行相似 SQL |

## ReuseExecutor：复用预编译语句

`ReuseExecutor` 的核心思想是：**复用同一个 SQL 的 PreparedStatement**。它维护了一个 `statementMap`，以 SQL 字符串为 key 缓存 PreparedStatement。

```java
public class ReuseExecutor extends BaseExecutor {

    // 缓存 PreparedStatement，key 是 SQL 字符串
    private final Map<String, Statement> statementMap = new ConcurrentHashMap<>();

    @Override
    public <E> List<E> doQuery(MappedStatement ms, Object parameter,
                                 RowBounds rowBounds, ResultHandler resultHandler,
                                 BoundSql boundSql) {
        Configuration configuration = ms.getConfiguration();
        StatementHandler handler = configuration.newStatementHandler(
            this, ms, parameter, rowBounds, resultHandler, boundSql);

        // 1. 先尝试从缓存获取
        String sql = boundSql.getSql();
        Statement statement = statementMap.get(sql);

        if (statement == null) {
            // 2. 缓存不存在，创建新的并放入缓存
            Connection connection = transaction.getConnection();
            statement = handler.prepare(connection, ms.getTimeout());
            statementMap.put(sql, statement);
        }

        // 3. 设置参数（复用已有的 Statement）
        handler.parameterize(statement);

        return handler.<E>query(statement, resultHandler);
    }

    @Override
    public int doUpdate(MappedStatement ms, Object parameter) {
        String sql = boundSql.getSql();
        Statement statement = statementMap.get(sql);

        if (statement == null) {
            Connection connection = transaction.getConnection();
            statement = handler.prepare(connection, ms.getTimeout());
            statementMap.put(sql, statement);
        }

        return handler.execute(statement);
    }

    @Override
    public List<BatchResult> flushStatements(boolean isRollback) {
        // 关闭所有缓存的 Statement
        for (Statement stmt : statementMap.values()) {
            stmt.close();
        }
        statementMap.clear();
        return Collections.emptyList();
    }
}
```

**ReuseExecutor 的特点：**

| 优点 | 缺点 |
|------|------|
| 复用预编译语句，减少编译开销 | 占用更多内存（缓存了 Statement 对象） |
| 适用于执行大量相似 SQL | 线程安全问题（需要用 ConcurrentHashMap） |
| 比 SimpleExecutor 性能更好 | 需要手动清理缓存（flushStatements） |

## BatchExecutor：批量执行的利器

`BatchExecutor` 是为批量操作而生的。它会将多个 SQL 累积起来，一次性发送给数据库执行：

```java
public class BatchExecutor extends BaseExecutor {

    // 批量结果列表
    private final List<Statement> statementList = new ArrayList<>();
    private final List<BatchResult> batchResultList = new ArrayList<>();

    @Override
    public <E> List<E> doQuery(MappedStatement ms, Object parameter,
                                 RowBounds rowBounds, ResultHandler resultHandler,
                                 BoundSql boundSql) {
        // 批量模式下，每次查询前先刷新之前的批量语句
        flushStatements();

        // 然后执行普通的查询
        return super.doQuery(ms, parameter, rowBounds, resultHandler, boundSql);
    }

    @Override
    public int doUpdate(MappedStatement ms, Object parameter) {
        // 获取 BoundSql
        String sql = boundSql.getSql();
        Statement stmt = null;

        try {
            // 尝试获取已有的 Statement
            BatchStatementWrapper wrapper = getBatchStatement(ms, sql);
            stmt = wrapper.statement;

            // 设置参数
            handler.parameterize(stmt);

            // 添加到批处理
            handler.addBatch(stmt);

            return BATCH_UPDATE_RETURN_VALUE;
        } catch (Exception e) {
            throw ExceptionFactory.wrapException("update error", e);
        }
    }

    @Override
    public List<BatchResult> flushStatements(boolean isRollback) {
        // 关键方法：执行所有累积的批量语句
        for (Statement stmt : statementList) {
            try {
                // 执行批量
                stmt.executeBatch();

                // 获取更新计数
                int[] updateCounts = ((PreparedStatement)stmt).getUpdateCounts();

                BatchResult br = new BatchResult(ms, boundSql, updateCounts);
                batchResultList.add(br);
            } catch (SQLException e) {
                // 处理异常
            } finally {
                stmt.close();
            }
        }

        // 清空
        statementList.clear();
        batchResultList.clear();

        return batchResultList;
    }
}
```

**BatchExecutor 的核心优势：**

| 优势 | 说明 |
|------|------|
| 减少数据库往返次数 | N 条 SQL 只需 1 次网络往返 |
| 事务原子性 | 可以在一个事务中批量执行 |
| JDBC 原生支持 | `PreparedStatement.addBatch()` + `executeBatch()` |

**使用 BatchExecutor 的典型场景：**

```java
try (SqlSession sqlSession = sqlSessionFactory.openSession(ExecutorType.BATCH)) {
    UserMapper mapper = sqlSession.getMapper(UserMapper.class);

    for (User user : users) {
        mapper.insert(user);
    }

    sqlSession.commit();  // 触发批量执行
}
```

执行时，MyBatis 会将所有 INSERT 语句添加到同一个 `PreparedStatement` 的批处理队列，然后一次性执行：

```sql
-- 一次网络往返完成所有插入
INSERT INTO user (name, age) VALUES ('张三', 20);
INSERT INTO user (name, age) VALUES ('李四', 21);
INSERT INTO user (name, age) VALUES ('王五', 22);
```

## CachingExecutor：二级缓存的装饰器

`CachingExecutor` 并不是真正的执行器，它是一个**装饰器**，用于包装其他执行器并添加二级缓存功能：

```java
public class CachingExecutor implements Executor {

    private final Executor delegate;

    public CachingExecutor(Executor delegate) {
        this.delegate = delegate;
    }

    @Override
    public <E> List<E> query(MappedStatement ms, Object parameter, RowBounds rowBounds,
                              ResultHandler resultHandler, CacheKey cacheKey, BoundSql boundSql) {
        // 1. 判断是否启用二级缓存
        if (ms.isCacheEnabled()) {
            // 2. 尝试从二级缓存获取
            Cache cache = ms.getCache();
            if (cache != null) {
                CacheKey key = createCacheKey(ms, parameter, rowBounds, boundSql);
                if (cache.containsKey(key)) {
                    return delegate.<E>query(ms, parameter, rowBounds,
                                              resultHandler, boundSql, key);
                }
            }
        }

        // 3. 缓存未命中，调用被包装的执行器
        List<E> list = delegate.<E>query(ms, parameter, rowBounds,
                                          resultHandler, boundSql, cacheKey);

        // 4. 结果存入二级缓存
        if (ms.isCacheEnabled()) {
            tcm.putObject(cache, cacheKey, list);
        }

        return list;
    }
}
```

`CachingExecutor` 的设计采用了**装饰器模式**：

```
用户代码
    ↓
CachingExecutor (二级缓存)
    ↓
BaseExecutor (一级缓存 + 模板方法)
    ↓
SimpleExecutor/ReuseExecutor/BatchExecutor (具体执行)
```

这种设计的好处是：**二级缓存功能和执行策略可以自由组合**。

## 执行器的选择策略

### 默认配置

在 MyBatis 配置文件中，可以指定执行器类型：

```xml
<settings>
    <!-- SIMPLE | REUSE | BATCH -->
    <setting name="defaultExecutorType" value="SIMPLE"/>
</settings>
```

### Spring 集成中的默认选择

在使用 Spring 与 MyBatis 集成时，`SqlSessionFactoryBean` 会根据配置创建执行器：

```java
// org.mybatis.spring.SqlSessionFactoryBean
protected SqlSessionFactory buildSqlSessionFactory() throws Exception {
    // ...
    ExecutorType executorType = this.executorType;
    if (executorType == null) {
        executorType = ExecutorType.SIMPLE;
    }
    // ...
}
```

**Spring 推荐的策略：**

| 场景 | 推荐执行器 | 原因 |
|------|-----------|------|
| 普通业务查询 | `SIMPLE` | 无需缓存 Statement，释放后立即归还连接池 |
| 批量插入/更新 | `BATCH` | 减少网络往返，提升批量操作性能 |
| 大量相似查询 | `REUSE` | 减少 SQL 预编译开销 |

### 手动指定执行器类型

```java
// 手动指定 Batch 执行器
try (SqlSession batchSession = sqlSessionFactory.openSession(ExecutorType.BATCH)) {
    UserMapper mapper = batchSession.getMapper(UserMapper.class);
    // 批量操作
    batchSession.commit();
}
```

## 四种 Statement 类型

执行器和 Statement 类型是两个不同的概念。执行器关注的是**如何管理 Statement 的生命周期**，而 Statement 类型关注的是**使用什么 JDBC 级别的 Statement**。

```java
public enum StatementType {
    STATEMENT,    // 普通 Statement，字符串拼接
    PREPARED,    // 预编译 Statement（默认）
    CALLABLE      // 存储过程 CallableStatement
}
```

| Statement 类型 | 对应的 Handler | SQL 拼接方式 | SQL 注入风险 |
|---------------|---------------|-------------|-------------|
| `STATEMENT` | `StatementHandler` | 字符串拼接 | 高 |
| `PREPARED` | `PreparedStatementHandler` | 占位符 `?` | 低 |
| `CALLABLE` | `CallableStatementHandler` | 存储过程调用 | 低 |

```xml
<!-- 默认使用 PREPARED，可防止 SQL 注入 -->
<select id="selectById" resultType="User">
    SELECT * FROM user WHERE id = #{id}
</select>

<!-- 显式指定使用 STATEMENT（不推荐） -->
<select id="selectById" resultType="User" statementType="STATEMENT">
    SELECT * FROM user WHERE id = ${id}
</select>
```

## 总结与延伸思考

### 执行器对比总结

| 特性 | SimpleExecutor | ReuseExecutor | BatchExecutor |
|------|----------------|---------------|---------------|
| Statement 管理 | 每次新建 | 复用 | 复用 + 批量 |
| 性能 | 一般 | 较好 | 最好（批量场景） |
| 内存占用 | 低 | 中 | 中 |
| 线程安全 | 安全 | 安全 | 安全 |
| 适用场景 | 普通查询 | 大量相似 SQL | 批量插入/更新 |

### 你可能想问的问题

**Q1：为什么 Spring 默认使用 SIMPLE 而不是 REUSE？**

因为 Spring 的 `SqlSession` 是线程绑定的（每个请求一个 SqlSession），请求结束后立即关闭。在这种情况下，同一个 SQL 在同一请求中很少重复执行，REUSE 的优势无法体现，反而增加缓存管理的复杂度。

**Q2：BatchExecutor 真的比循环执行快吗？**

是的，但有前提条件：
- 批量操作需要在同一个事务内
- 数据库连接池需要足够大
- 批量大小需要合理（建议 100-1000 条/批）

**Q3：二级缓存和一级缓存同时存在时，查询顺序是什么？**

```
查询请求
    ↓
先查二级缓存（CachingExecutor）
    ↓ 未命中
再查一级缓存（BaseExecutor）
    ↓ 未命中
执行数据库查询
    ↓
结果存入一级缓存
    ↓
结果存入二级缓存
```

理解了这个顺序，你就明白了为什么有时候查询结果不符合预期——可能是一级缓存或二级缓存中的旧数据导致的。

3. **ConcurrentHashMap 的并发原理**：ReuseExecutor 如何保证线程安全？

这些问题，值得你在源码中寻找答案。
