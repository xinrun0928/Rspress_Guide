# MyBatis 二级缓存失效：你的缓存为什么没有生效

二级缓存明明配了，为什么每次查出来的不一样？

很多人在配置完 MyBatis 二级缓存后，发现缓存"不生效"——要么每次都查数据库，要么数据不一致。问题的根源，往往在于对二级缓存的工作机制理解不够深入。

这篇文章，帮你排查二级缓存的各种"失效"场景。

## 二级缓存失效的常见场景

### 场景一：增删改操作清空缓存

这是最常见的原因。MyBatis 会自动清空对应 namespace 的二级缓存：

```xml
<!-- UserMapper.xml -->
<mapper namespace="com.example.UserMapper">
    <cache/>

    <select id="selectById" resultType="User">
        SELECT * FROM user WHERE id = #{id}
    </select>

    <!-- 增删改操作会清空 UserMapper 的二级缓存 -->
    <insert id="insert">
        INSERT INTO user (name, age) VALUES (#{name}, #{age})
    </insert>

    <update id="update">
        UPDATE user SET name = #{name} WHERE id = #{id}
    </update>

    <delete id="delete">
        DELETE FROM user WHERE id = #{id}
    </delete>
</mapper>
```

**执行流程**：

```
执行 INSERT/UPDATE/DELETE
        ↓
清空 UserMapper namespace 的二级缓存
        ↓
后续 SELECT 重新查询数据库
```

### 场景二：跨 namespace 访问的脏读

这是最容易被忽视的问题。考虑以下场景：

```xml
<!-- UserMapper.xml -->
<mapper namespace="com.example.UserMapper">
    <cache/>
    <select id="selectById" resultType="User">
        SELECT u.*, d.name AS dept_name
        FROM user u
        LEFT JOIN department d ON u.dept_id = d.id
        WHERE u.id = #{id}
    </select>
</mapper>

<!-- OrderMapper.xml -->
<mapper namespace="com.example.OrderMapper">
    <cache/>
    <update id="updateDeptName">
        UPDATE department SET name = #{name} WHERE id = #{deptId}
    </update>
</mapper>
```

**问题分析**：

```
SqlSession A：
    selectById(1) → 缓存：User(deptName="研发部")
            ↓
SqlSession B（不同会话）：
    updateDeptName() → 更新数据库 deptName="销售部" → 清空 OrderMapper 缓存
            ↓
            （UserMapper 的缓存没被清空！）
            ↓
SqlSession A：
    selectById(1) → 命中缓存 → 返回 User(deptName="研发部") ← 脏数据！
```

**问题根源**：UserMapper 的缓存依赖于 department 表的数据，但 department 的更新在 OrderMapper 中，无法触发 UserMapper 缓存清空。

**解决方案**：

1. **方案一：禁用跨表查询的二级缓存**

```xml
<select id="selectById" resultType="User" useCache="false">
    SELECT u.*, d.name AS dept_name
    FROM user u
    LEFT JOIN department d ON u.dept_id = d.id
    WHERE u.id = #{id}
</select>
```

2. **方案二：使用 flushCache 属性**

```xml
<update id="updateDeptName" flushCache="true">
    UPDATE department SET name = #{name} WHERE id = #{deptId}
</update>
```

### 场景三：flushCache="true" 强制清空

每个查询都可以配置 `flushCache` 属性：

```xml
<select id="selectById" resultType="User" flushCache="true">
    SELECT * FROM user WHERE id = #{id}
</select>
```

| flushCache | 效果 |
|-----------|------|
| `true` | 每次查询前清空一级、二级缓存 |
| `false` | 正常查询，不清空缓存 |

### 场景四：useCache="false" 禁用缓存

```xml
<select id="selectById" resultType="User" useCache="false">
    SELECT * FROM user WHERE id = #{id}
</select>
```

| useCache | 效果 |
|---------|------|
| `true` | 结果存入二级缓存 |
| `false` | 不使用二级缓存，直接查询数据库 |

### 场景五：flushInterval 定时清空

```xml
<mapper namespace="com.example.UserMapper">
    <!-- 每 60 秒清空一次缓存 -->
    <cache flushInterval="60000"/>
</mapper>
```

**注意事项**：

- `flushInterval` 是在**缓存管理器级别**定时清空，不是自动刷新
- 它不会主动刷新缓存，只是定时移除过期数据
- 适合数据变化不频繁的场景

### 场景六：事务未提交

二级缓存的写入发生在事务提交时：

```java
SqlSession sqlSession = sqlSessionFactory.openSession();
UserMapper mapper = sqlSession.getMapper(UserMapper.class);

User user = mapper.selectById(1L);  // 查询，但不提交
// 此时二级缓存中没有数据

// 错误的做法：没有 commit
sqlSession.close();

// 正确的做法：需要 commit
sqlSession.commit();
sqlSession.close();
```

`TransactionalCache` 的实现决定了这一点：

```java
public class TransactionalCache implements Cache {

    private final Map<Object, Object> entriesToAddOnCommit = new HashMap<>();

    @Override
    public void putObject(Object key, Object value) {
        // 数据不直接写入缓存，而是放入待提交队列
        entriesToAddOnCommit.put(key, value);
    }

    public void commit() {
        // commit 时才真正写入缓存
        for (Map.Entry<Object, Object> entry : entriesToAddOnCommit.entrySet()) {
            delegate.putObject(entry.getKey(), entry.getValue());
        }
        entriesToAddOnCommit.clear();
    }
}
```

### 场景七：实体类未序列化

当 `readOnly="false"` 时，二级缓存需要对对象进行序列化：

```xml
<cache readOnly="false"/>
```

```java
// 错误：实体类没有实现 Serializable
public class User {
    private Long id;
    private String name;
}

// 正确：实现 Serializable
public class User implements Serializable {
    private static final long serialVersionUID = 1L;

    private Long id;
    private String name;
}
```

**序列化失败会导致缓存写入失败**，看起来就是缓存"不生效"。

## 分布式环境下的缓存问题

### 问题分析

在单机环境下，二级缓存是进程级别的（存在 JVM 堆内存中）。但在分布式环境下：

```
服务器 A：缓存 key=1 → User("张三")
服务器 B：数据库 id=1 → User("李四")  ← 数据不一致！
```

不同服务器的缓存完全隔离，无法感知其他服务器的数据库更新。

### 解决方案

1. **方案一：不用二级缓存**

   对于数据一致性要求高的场景，直接禁用二级缓存。

2. **方案二：使用分布式缓存**

   将缓存存储在 Redis、Memcached 等分布式缓存中：

   ```xml
   <mapper namespace="com.example.UserMapper">
       <!-- 使用 Redis 缓存 -->
       <cache type="org.mybatis.caches.redis.RedisCache"/>
   </mapper>
   ```

3. **方案三：主动清空缓存**

   增删改时，通过 MQ 或其他机制通知所有服务器清空缓存。

## 缓存失效排查清单

遇到缓存问题，按照以下清单逐项排查：

| 检查项 | 检查内容 | 修复方法 |
|--------|---------|---------|
| cache 配置 | 是否在 XML 中声明了 `<cache/>` | 添加 `<cache/>` 声明 |
| cacheEnabled | 全局是否开启 | 确认 `cacheEnabled=true` |
| useCache | 查询是否启用缓存 | 确认 `useCache="true"` |
| flushCache | 查询是否强制清空缓存 | 确认 `flushCache="false"` |
| 实体类序列化 | 是否实现 Serializable | 实现 `Serializable` 接口 |
| 事务提交 | 是否正确 commit | 调用 `sqlSession.commit()` |
| namespace | 增删改和查询是否在同一 namespace | 确认 namespace 一致 |
| 缓存范围 | 是否跨 namespace 查询 | 考虑禁用缓存或使用分布式缓存 |

## 缓存配置最佳实践

### 基础配置

```xml
<!-- mybatis-config.xml -->
<settings>
    <!-- 开启二级缓存 -->
    <setting name="cacheEnabled" value="true"/>
</settings>

<!-- UserMapper.xml -->
<mapper namespace="com.example.UserMapper">
    <!-- 使用 LRU 策略，容量 512，只读缓存 -->
    <cache eviction="LRU" flushInterval="0" size="512" readOnly="true"/>
</mapper>
```

### 高一致性配置

```xml
<!-- 高一致性场景：禁用二级缓存或使用 flushCache -->
<select id="selectById" resultType="User" useCache="false">
    SELECT * FROM user WHERE id = #{id}
</select>

<!-- 或者每次查询强制清空缓存 -->
<select id="selectById" resultType="User" flushCache="true">
    SELECT * FROM user WHERE id = #{id}
</select>
```

### 高性能配置

```xml
<!-- 高性能场景：读写缓存 -->
<mapper namespace="com.example.UserMapper">
    <!-- 读写缓存，需要实体类实现 Serializable -->
    <cache eviction="LRU" flushInterval="300000" size="1024" readOnly="false"/>
</mapper>
```

## 常见问题解答

### Q1：为什么增删改会自动清空缓存？

MyBatis 在执行增删改时，会调用 `CachingExecutor` 的 `update()` 方法：

```java
@Override
public int update(MappedStatement ms, Object parameter) throws SQLException {
    // 清空对应 namespace 的缓存
    clearLocalCache();
    return delegate.update(ms, parameter);
}
```

这是设计上的保护机制，防止数据不一致。

### Q2：查询结果没有存入缓存？

检查以下几点：

1. 是否有 `useCache="false"`
2. 是否正确 commit 了事务
3. 实体类是否实现了 `Serializable`（当 `readOnly="false"` 时）
4. namespace 是否正确配置了 `<cache/>`

### Q3：缓存中的数据和数据库不一致？

这是典型的脏读问题：

1. 检查是否有跨 namespace 的关联查询
2. 检查是否有并发更新
3. 考虑使用分布式缓存或禁用二级缓存

### Q4：Redis 缓存如何配置？

```xml
<!-- 引入 mybatis-redis 依赖 -->
<dependency>
    <groupId>org.mybatis.caches</groupId>
    <artifactId>mybatis-redis</artifactId>
    <version>1.0.0-beta2</version>
</dependency>
```

```xml
<!-- 配置 Redis 缓存 -->
<cache type="org.mybatis.caches.redis.RedisCache"/>

<!-- redis.properties -->
redis.host=localhost
redis.port=6379
```

## 延伸思考

### 缓存一致性和性能的两难

缓存的核心矛盾：

```
强一致性 ←————————————————————————→ 高性能
    ↑                                       ↑
  不用缓存                                  缓存不刷新
    ↑                                       ↑
每次都查数据库                              脏读风险
```

没有完美的解决方案，只有根据业务场景的权衡：

| 场景 | 推荐策略 |
|------|---------|
| 订单、支付等核心业务 | 禁用二级缓存 |
| 用户、配置等读多写少 | 开启缓存 + 短过期时间 |
| 数据报表、统计分析 | 开启缓存 + 定时刷新 |
| 分布式环境 | 使用 Redis 等分布式缓存 |

### 缓存穿透、缓存击穿、缓存雪崩

这三个问题在实际生产中经常遇到：

**缓存穿透**：查询不存在的数据，每次都查数据库。

```java
// 解决方案：布隆过滤器 或 缓存空值
if (result == null) {
    cache.put(key, NULL_VALUE);  // 缓存空值
}
```

**缓存击穿**：热点 key 过期瞬间，大量请求涌入数据库。

```java
// 解决方案：加锁 或 永不过期
public User getUser(Long id) {
    User user = cache.get(id);
    if (user == null) {
        synchronized (this) {
            user = cache.get(id);
            if (user == null) {
                user = database.query(id);
                cache.put(id, user);  // 永不过期
            }
        }
    }
    return user;
}
```

**缓存雪崩**：大量缓存同时过期。

```java
// 解决方案：过期时间加随机值
cache.put(key, value, 3600 + random.nextInt(600));
```

理解这些缓存问题的本质，你就能在实际项目中做出正确的架构决策。缓存是优化性能的重要手段，但不是银弹——了解它的边界，才能用好它。
