# MyBatis 注解方式 vs XML 方式：各有各的用武之地

<Badge text="MyBatis" type="tip" /> <Badge text="对比分析" type="info" />

注解写 SQL 简单，但 XML 能做的事注解不一定能做到。

MyBatis 同时支持两种 SQL 定义方式：XML 映射和注解映射。两者各有优劣，如何选择是很多开发者关心的问题。

这篇文章，让你清楚了解两种方式的能力边界，做出正确的技术决策。

## 注解方式：@Select、@Insert、@Update、@Delete

### 基本用法

```java
public interface UserMapper {

    @Select("SELECT * FROM user WHERE id = #{id}")
    User selectById(Long id);

    @Insert("INSERT INTO user (name, email) VALUES (#{name}, #{email})")
    @Options(useGeneratedKeys = true, keyProperty = "id")
    int insert(User user);

    @Update("UPDATE user SET name = #{name} WHERE id = #{id}")
    int update(User user);

    @Delete("DELETE FROM user WHERE id = #{id}")
    int delete(Long id);
}
```

### 注解方式的优势

| 优势 | 说明 |
|------|------|
| 代码紧凑 | SQL 和接口放在一起，不用切换文件 |
| 快速开发 | 简单的 CRUD 不需要写 XML |
| 类型安全 | 编译器可以检查参数类型 |
| 重构友好 | 方法名改变时，IDE 可以自动更新 |

### 适用场景

```java
// 单表 CRUD，最适合注解方式
@Select("SELECT * FROM user WHERE id = #{id}")
User selectById(Long id);

@Select("SELECT * FROM user WHERE name = #{name} AND status = #{status}")
List<User> selectByNameAndStatus(@Param("name") String name, @Param("status") Integer status);
```

## 结果映射注解

### @Results 和 @Result

```java
@Select("SELECT id, username, email, status FROM user WHERE id = #{id}")
@Results({
    @Result(id = true, property = "id", column = "id"),
    @Result(property = "username", column = "username"),
    @Result(property = "email", column = "email"),
    @Result(property = "status", column = "status")
})
User selectById(Long id);
```

### 驼峰映射

```java
// 开启驼峰映射后，不需要逐个指定
@Select("SELECT * FROM user WHERE id = #{id}")
@Results({
    @Result(id = true, property = "id", column = "id")
})
User selectById(Long id);
```

```xml
<!-- mybatis-config.xml -->
<settings>
    <setting name="mapUnderscoreToCamelCase" value="true"/>
</settings>
```

### 一对一关联

```java
@Select("SELECT * FROM user WHERE id = #{id}")
@Results({
    @Result(id = true, property = "id", column = "id"),
    @Result(property = "username", column = "username"),

    // 一对一关联
    @Result(property = "department", column = "dept_id",
            one = @One(select = "com.example.DepartmentMapper.selectById"))
})
User selectById(Long id);
```

### 一对多关联

```java
@Select("SELECT * FROM user")
@Results({
    @Result(id = true, property = "id", column = "id"),

    // 一对多关联
    @Result(property = "orders", column = "id",
            many = @Many(select = "com.example.OrderMapper.selectByUserId"))
})
List<User> selectAll();
```

## 构造器映射

### @ConstructorArgs

```java
@Select("SELECT id, username, email FROM user WHERE id = #{id}")
@ConstructorArgs({
    @Arg(column = "id", javaType = Long.class, id = true),
    @Arg(column = "username", javaType = String.class),
    @Arg(column = "email", javaType = String.class)
})
User selectById(Long id);
```

### @Arg（简化写法）

```java
@Select("SELECT id, username, email FROM user WHERE id = #{id}")
@Results({
    @Result(id = true, property = "id", column = "id")
})
@Arg(column = "id", javaType = Long.class, id = true)
@Arg(column = "username", javaType = String.class)
@Arg(column = "email", javaType = String.class)
User selectById(Long id);
```

## 注解方式的局限性

### 动态 SQL 困难

```java
// 错误：注解中不能使用 <if> 标签
@Select("SELECT * FROM user WHERE
        <if test="name != null">
            name = #{name}
        </if>")
List<User> search(@Param("name") String name);
```

**解决方案**：使用 `<script>` 标签（不推荐）

```java
@Select("<script>" +
        "SELECT * FROM user WHERE 1=1 " +
        "<if test='name != null'>" +
        "AND name = #{name}" +
        "</if>" +
        "</script>")
List<User> search(@Param("name") String name);
```

### SQL 字符串拼接不直观

```java
// 简单的 IN 查询
@Select("SELECT * FROM user WHERE id IN (${ids})")
List<User> selectByIds(@Param("ids") String ids);
```

### 多表关联映射复杂

```java
// JOIN 查询，手动映射所有字段
@Select("SELECT u.id, u.username, d.name AS dept_name " +
        "FROM user u LEFT JOIN department d ON u.dept_id = d.id " +
        "WHERE u.id = #{id}")
@Results({
    @Result(id = true, property = "id", column = "id"),
    @Result(property = "username", column = "username"),
    @Result(property = "department", column = "dept_name",
            one = @One(select = "com.example.DepartmentMapper.selectByName"))
})
User selectById(Long id);
```

## XML 方式的优势

### 动态 SQL 强大

```xml
<select id="search" resultType="User">
    SELECT * FROM user
    <where>
        <if test="name != null">
            name = #{name}
        </if>
        <if test="status != null">
            AND status = #{status}
        </if>
    </where>
</select>
```

### SQL 可读性好

```xml
<select id="selectComplexReport" resultMap="reportResultMap">
    SELECT
        u.id,
        u.username,
        d.name AS dept_name,
        COUNT(o.id) AS order_count,
        SUM(o.amount) AS total_amount,
        AVG(o.amount) AS avg_amount
    FROM user u
    LEFT JOIN department d ON u.dept_id = d.id
    LEFT JOIN `order` o ON u.id = o.user_id
    WHERE u.status = 1
    GROUP BY u.id, u.username, d.name
    HAVING COUNT(o.id) > 5
    ORDER BY total_amount DESC
    LIMIT 100
</select>
```

### resultMap 强大

```xml
<resultMap id="userResultMap" type="User">
    <id property="id" column="id"/>

    <!-- 一对多 -->
    <collection property="orders" ofType="Order">
        <id property="id" column="order_id"/>
    </collection>
</resultMap>
```

### 团队协作

- SQL 在独立的 XML 文件中，易于版本管理
- 可以单独测试 SQL 是否正确
- 可以复用 SQL 片段

## 混用策略

### 最佳实践

| SQL 类型 | 推荐方式 | 原因 |
|---------|---------|------|
| 单表 CRUD | 注解 | 简单、快速 |
| 动态条件查询 | XML | 动态 SQL 更强大 |
| 复杂 JOIN | XML | resultMap 更清晰 |
| 简单 JOIN | 注解 | 也能接受 |
| 批量操作 | XML | foreach 更直观 |

### 实际项目中的选择

```java
// 简单的单表操作 → 注解
@Select("SELECT * FROM user WHERE id = #{id}")
User selectById(Long id);

@Insert("INSERT INTO user (name, email) VALUES (#{name}, #{email})")
int insert(User user);

// 复杂的查询逻辑 → XML
```

```xml
<!-- UserMapper.xml -->
<select id="search" resultType="User">
    <bind name="namePattern" value="'%' + name + '%'"/>
    SELECT * FROM user
    <where>
        <if test="name != null">
            AND name LIKE #{namePattern}
        </if>
        <if test="status != null">
            AND status = #{status}
        </if>
    </where>
</select>
```

### 注解 + resultMap 分离

如果想在注解中复用 resultMap：

```xml
<!-- UserMapper.xml -->
<resultMap id="userResultMap" type="User">
    <id property="id" column="id"/>
    <result property="username" column="username"/>
    <association property="department" javaType="Department">
        <id property="id" column="dept_id"/>
        <result property="name" column="dept_name"/>
    </association>
</resultMap>
```

```java
// 接口中引用 XML 的 resultMap
@Select("SELECT u.id, u.username, u.dept_id, d.name AS dept_name " +
        "FROM user u LEFT JOIN department d ON u.dept_id = d.id " +
        "WHERE u.id = #{id}")
@ResultMap("com.example.mapper.UserMapper.userResultMap")
User selectById(Long id);
```

## 性能对比

### 启动性能

| 方式 | 启动时解析 | 运行时解析 |
|------|-----------|-----------|
| XML | MyBatis 启动时解析所有 XML | 无额外开销 |
| 注解 | Mapper 注册时解析注解 | 首次调用时解析 |

**结论**：启动性能差异不大。

### 运行性能

**结论**：没有性能差异。SQL 最终都会被解析成 `MappedStatement`，执行方式完全一样。

### 可维护性

| 方面 | XML | 注解 |
|------|-----|------|
| SQL 修改 | 需要改 XML | 需要改 Java |
| 重构 | 依赖 IDE 支持 | 编译器检查 |
| 版本管理 | 独立文件，易于 diff | 与代码混在一起 |

## 总结对比

| 特性 | 注解方式 | XML 方式 |
|------|---------|---------|
| 动态 SQL | 弱（需要 `<script>`） | 强（9 个标签） |
| SQL 可读性 | 差（单行或拼接） | 好（格式化） |
| resultMap | 基本映射 | 强大映射 |
| 团队协作 | 一般 | 好 |
| 修改频率 | 低（SQL 稳定时） | 高（SQL 频繁变更） |
| 学习曲线 | 低 | 中 |

### 推荐策略

1. **小项目**：全部用注解，开发快
2. **中等项目**：简单 SQL 用注解，复杂 SQL 用 XML
3. **大项目**：统一用 XML，便于管理和维护

### 你可能想问的问题

**Q1：注解方式可以完全替代 XML 吗？**

不能。动态 SQL 是注解方式的硬伤，复杂的业务场景下必须使用 XML。

**Q2：可以同时使用注解和 XML 吗？**

可以。MyBatis 支持在同一个 Mapper 接口上混用：
- 在接口上使用 `@Select`、`@Insert` 等注解
- 在 XML 中定义 resultMap 和动态 SQL

**Q3：MyBatis Plus 对注解方式有什么增强？**

MyBatis Plus 大幅简化了注解方式的使用：
- `BaseMapper<T>` 提供通用 CRUD，无需写 SQL
- `@Select` 注解支持更强大的功能
- Lambda 表达式支持

理解两种方式的特点和适用场景，你就能根据项目实际情况做出正确的技术决策。记住：**没有最好的方式，只有最适合的方式**。
