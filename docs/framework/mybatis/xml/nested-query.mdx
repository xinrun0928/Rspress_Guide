# N+1 问题：MyBatis 最容易踩的坑

<Badge text="MyBatis" type="tip" /> <Badge text="性能优化" type="warning" />

查了一个订单列表，为什么执行了一百多条 SQL？

N+1 问题，是 MyBatis 使用过程中最常见的性能陷阱之一。理解它的成因和解决思路，是 MyBatis 进阶的必经之路。

## 什么是 N+1 问题？

当使用嵌套 select 方式查询关联对象时，会产生 N+1 次数据库查询：

```
1 次查询主表（1）
+
N 次查询关联表（N）
= N + 1 次查询
```

### 具体场景

假设有 100 个用户，每个用户有 5 个订单：

```xml
<resultMap id="userResultMap" type="User">
    <id property="id" column="id"/>
    <result property="username" column="username"/>

    <!-- 嵌套 select 方式 -->
    <collection property="orders"
                ofType="Order"
                column="id"
                select="com.example.mapper.OrderMapper.selectByUserId"/>
</resultMap>

<select id="selectAllUsers" resultMap="userResultMap">
    SELECT * FROM user LIMIT 100
</select>
```

**执行的 SQL**：

```sql
-- 第 1 次：查询用户列表
SELECT * FROM user LIMIT 100

-- 第 2-101 次：查询每个用户的订单（N = 100）
SELECT * FROM order WHERE user_id = 1
SELECT * FROM order WHERE user_id = 2
SELECT * FROM order WHERE user_id = 3
...
SELECT * FROM order WHERE user_id = 100
```

**问题**：查 100 个用户，执行了 101 次 SQL！

## N+1 的性能分析

### 时间复杂度

```
T(1+N) = T(1) + N × T(1)
       = (N + 1) × T(1)
```

当 N = 1000 时：
- 单次查询时间 = 1ms
- 总时间 = 1001 × 1ms = 1.001 秒

### 网络开销

每次数据库查询，都需要：
- 建立/获取连接
- 发送 SQL 语句
- 数据库执行
- 返回结果
- 关闭连接（归还连接池）

N+1 问题意味着 N 倍的网络往返开销。

### 连接池压力

假设连接池有 10 个连接：
- 查询 1 个用户：需要 1 个连接
- 查询 100 个用户：需要 100 个连接 × 多次使用 = 连接争抢

## 两种查询方式对比

### 嵌套 select（会产生 N+1）

```xml
<resultMap id="userResultMap" type="User">
    <id property="id" column="id"/>
    <result property="username" column="username"/>

    <collection property="orders"
                ofType="Order"
                column="id"
                select="com.example.mapper.OrderMapper.selectByUserId"/>
</resultMap>

<select id="selectAll" resultMap="userResultMap">
    SELECT id, username FROM user
</select>
```

```java
// 执行
List<User> users = mapper.selectAll();
// 实际执行：1 + N 次 SQL
```

### 嵌套结果（JOIN 一次查完）

```xml
<resultMap id="userResultMap" type="User">
    <id property="id" column="id"/>
    <result property="username" column="username"/>

    <collection property="orders" ofType="Order">
        <id property="id" column="order_id"/>
        <result property="orderNo" column="order_no"/>
    </collection>
</resultMap>

<select id="selectAll" resultMap="userResultMap">
    SELECT
        u.id, u.username,
        o.id AS order_id, o.order_no
    FROM user u
    LEFT JOIN `order` o ON u.id = o.user_id
</select>
```

```java
// 执行
List<User> users = mapper.selectAll();
// 实际执行：1 次 SQL（带 JOIN）
```

## 嵌套 select 的 lazy loading 配置

### 全局延迟加载

```xml
<settings>
    <!-- 开启延迟加载（默认 true） -->
    <setting name="lazyLoadingEnabled" value="true"/>

    <!-- aggressiveLazyLoading：false = 按需加载 -->
    <setting name="aggressiveLazyLoading" value="false"/>
</settings>
```

### 局部加载方式

```xml
<resultMap id="userResultMap" type="User">
    <id property="id" column="id"/>

    <!-- 嵌套 select + 延迟加载 -->
    <collection property="orders"
                ofType="Order"
                column="id"
                select="com.example.mapper.OrderMapper.selectByUserId"
                fetchType="lazy"/>
</resultMap>
```

### 立即加载

```xml
<!-- 嵌套 select + 立即加载（会产生 N+1） -->
<collection property="orders"
            ofType="Order"
            column="id"
            select="com.example.mapper.OrderMapper.selectByUserId"
            fetchType="eager"/>
```

## 嵌套结果（JOIN）详解

### 一对多映射

```xml
<resultMap id="userResultMap" type="User">
    <id property="id" column="id"/>
    <result property="username" column="username"/>

    <!-- 一对多：订单列表 -->
    <collection property="orders" ofType="Order">
        <id property="id" column="order_id"/>
        <result property="orderNo" column="order_no"/>
        <result property="totalAmount" column="total_amount"/>
    </collection>
</resultMap>

<select id="selectAll" resultMap="userResultMap">
    SELECT
        u.id, u.username,
        o.id AS order_id, o.order_no, o.total_amount
    FROM user u
    LEFT JOIN `order` o ON u.id = o.user_id
</select>
```

**JOIN 结果**：

| id | username | order_id | order_no | total_amount |
|----|----------|----------|----------|--------------|
| 1 | 张三 | 101 | ORD001 | 100.00 |
| 1 | 张三 | 102 | ORD002 | 200.00 |
| 2 | 李四 | 103 | ORD003 | 150.00 |

**MyBatis 处理**：
- 按 `id` 字段去重
- 将相同的 `id` 聚合到同一个 User 对象
- 将 `order_id`、`order_no`、`total_amount` 聚合到 Orders 列表

### 一对一映射

```xml
<resultMap id="userResultMap" type="User">
    <id property="id" column="id"/>
    <result property="username" column="username"/>

    <!-- 一对一：部门 -->
    <association property="department" javaType="Department">
        <id property="id" column="dept_id"/>
        <result property="name" column="dept_name"/>
    </association>
</resultMap>

<select id="selectAll" resultMap="userResultMap">
    SELECT
        u.id, u.username, u.dept_id,
        d.id AS dept_id, d.name AS dept_name
    FROM user u
    LEFT JOIN department d ON u.dept_id = d.id
</select>
```

## fetchType：eager vs lazy

### 全局配置

```xml
<settings>
    <!-- 默认延迟加载（默认 true） -->
    <setting name="lazyLoadingEnabled" value="true"/>

    <!-- 激进加载（默认 true）设为 false 实现按需加载 -->
    <setting name="aggressiveLazyLoading" value="false"/>
</settings>
```

| aggressiveLazyLoading | 行为 |
|-----------------------|------|
| `true` | 访问对象的任一属性，加载所有延迟加载属性 |
| `false` | 只有访问延迟加载属性时，才触发加载 |

### 局部配置

```xml
<resultMap id="userResultMap" type="User">
    <id property="id" column="id"/>

    <!-- 强制延迟加载（覆盖全局配置） -->
    <association property="department" javaType="Department" fetchType="lazy">
        <id property="id" column="dept_id"/>
    </association>

    <!-- 强制立即加载 -->
    <collection property="orders" ofType="Order" fetchType="eager">
        <id property="id" column="order_id"/>
    </collection>
</resultMap>
```

## 使用延迟加载的正确姿势

### 场景一：列表页，不需要关联数据

```java
// 列表查询，只显示用户基本信息
List<User> users = userMapper.selectAll();
// 不访问 orders，不触发延迟加载
```

### 场景二：详情页，需要关联数据

```java
// 详情查询
User user = userMapper.selectById(1L);
// 访问关联数据
System.out.println(user.getDepartment().getName());  // 触发延迟加载
System.out.println(user.getOrders().size());          // 触发延迟加载
```

### 场景三：性能优先

```java
// 使用 JOIN 一次查完，不产生额外查询
List<User> users = userMapper.selectAllWithDetails();
```

## 如何发现 N+1 问题

### 方法一：查看 SQL 日志

```xml
<settings>
    <setting name="logImpl" value="STDOUT_LOGGING"/>
</settings>
```

或使用 Log4j：

```xml
<logger name="org.apache.ibatis">
    <level value="DEBUG"/>
</logger>
```

**N+1 的日志特征**：

```
-- 第一次查询
DEBUG: ==>  Preparing: SELECT * FROM user
DEBUG: ==> Parameters: 

-- 之后 N 次查询
DEBUG: ==>  Preparing: SELECT * FROM order WHERE user_id = ?
DEBUG: ==> Parameters: 1(Integer)
DEBUG: ==>  Preparing: SELECT * FROM order WHERE user_id = ?
DEBUG: ==> Parameters: 2(Integer)
...
```

### 方法二：使用 MyBatis Generator 插件

MyBatis Generator 可以生成带 resultMap 的 XML，避免手写错误。

### 方法三：使用拦截器统计 SQL

```java
@Intercepts({
    @Signature(type = StatementHandler.class, method = "query", args = {Statement.class, ResultHandler.class})
})
public class SqlCountInterceptor implements Interceptor {

    @Override
    public Object intercept(Invocation invocation) throws Throwable {
        long start = System.currentTimeMillis();
        Object result = invocation.proceed();
        long end = System.currentTimeMillis();

        String sql = invocation.getArgs()[0].toString();
        if (sql.contains("SELECT")) {
            System.out.println("SQL: " + sql + " | 耗时: " + (end - start) + "ms");
        }

        return result;
    }
}
```

## 解决方案总结

### 避免 N+1 的最佳实践

| 场景 | 推荐方案 |
|------|---------|
| 一对一关联 | `association` + 嵌套 resultMap |
| 一对多关联 | `collection` + 嵌套 resultMap |
| 列表查询 | JOIN 一次查完 |
| 详情查询 | 可用延迟加载，但要注意 |

### 代码层面优化

```java
// 避免这样做：循环中访问延迟加载属性
List<User> users = userMapper.selectAll();
for (User user : users) {
    // 触发 N 次查询！
    System.out.println(user.getOrders().size());
}

// 正确做法：使用 JOIN 一次性加载
List<User> users = userMapper.selectAllWithOrders();
for (User user : users) {
    // 不触发额外查询
    System.out.println(user.getOrders().size());
}
```

### 分页与 N+1

分页查询时，N+1 问题会放大：

```xml
<!-- 查 10 个用户，产生 11 次 SQL -->
<select id="selectPage" resultMap="userResultMap">
    SELECT * FROM user LIMIT 10
</select>
```

**解决方案**：分页 + JOIN

```xml
<select id="selectPage" resultMap="userResultMap">
    SELECT
        u.id, u.username,
        o.id AS order_id
    FROM user u
    LEFT JOIN `order` o ON u.id = o.user_id
    WHERE ...（查询条件）
    LIMIT 10
</select>
```

## 延伸思考

### N+1 vs JOIN 的权衡

| 方面 | N+1（嵌套 select） | JOIN |
|------|-------------------|------|
| SQL 数量 | N+1 | 1 |
| 单次查询时间 | 短 | 长（数据量大时） |
| 网络开销 | 高 | 低 |
| 内存占用 | 低 | 高（JOIN 结果集大） |
| 适用场景 | 数据量小 | 数据量大 |

### MyBatis Plus 的解决方案

MyBatis Plus 提供了 `plusJoin` 等优化，减少 N+1 问题：

```java
// 使用 Plus 的联表查询
IPage<User> page = userService.page(page,
    new LambdaQueryWrapper<User>()
        .selectAll(User.class)
        .plusJoin(Order.class, (user, order) -> user.getId().eq(order.getUserId()))
);
```

### 你可能想问的问题

**Q1：延迟加载一定比立即加载慢吗？**

不一定。如果关联数据大部分情况下不需要访问，延迟加载可以节省很多不必要的查询。但如果有大量关联查询，JOIN 可能更快。

**Q2：association 和 collection 都会产生 N+1 吗？**

是的。只要使用 `select` 属性（嵌套 select），就会产生额外查询。

**Q3：有没有自动检测 N+1 的工具？**

- MyBatis Debug 日志
- MyBatis Log Plugin（IntelliJ IDEA 插件）
- SkyWalking 等 APM 工具

理解 N+1 问题的本质，你就能在性能和代码简洁性之间做出正确的权衡。记住一个原则：**能用 JOIN 解决的，不用嵌套 select**。
