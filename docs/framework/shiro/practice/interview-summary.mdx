# Shiro 面试高频问题：核心问题一网打尽

"Shiro 和 Spring Security 的区别是什么？"、"Shiro 的认证流程是什么？"、"Shiro 怎么实现分布式 Session？"

这些问题，是面试官考察候选人 Java 安全框架掌握程度的常用问题。本节将逐一解答，帮你把 Shiro 的核心知识点梳理清楚。

## 一、Shiro 的三大核心组件

### 问：Shiro 的三大核心组件是什么？

Shiro 有三个最核心的概念：**Subject、SecurityManager、Realm**。

- **Subject**：当前操作用户，代表任何与系统交互的实体（用户、第三方服务）
- **SecurityManager**：Shiro 的核心，管理所有 Subject，协调认证器、授权器、会话管理器
- **Realm**：连接 SecurityManager 和数据源的桥梁，提供用户、角色、权限信息

```
Subject ←── 委托 ──→ SecurityManager ←── 查询 ──→ Realm（数据源）
```

### 问：Shiro 的架构图是什么样的？

```
┌─────────────────────────────────────────────────────────────┐
│                         Subject                             │
│  (SecurityUtils.getSubject())                              │
│  login() / logout() / isPermitted() / getSession()        │
└───────────────────────────┬─────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│                     SecurityManager                         │
│  ┌──────────────┐ ┌──────────────┐ ┌────────────────┐    │
│  │ Authenticator │ │  Authorizer  │ │ SessionManager  │    │
│  └───────┬──────┘ └───────┬──────┘ └───────┬────────┘    │
│          └────────────────┼────────────────┘              │
│                           ▼                                 │
│  ┌─────────────────────────────────────────────────────┐  │
│  │                    Realm(s)                           │  │
│  │  doGetAuthenticationInfo() → 认证                    │  │
│  │  doGetAuthorizationInfo()  → 授权                    │  │
│  └─────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

## 二、认证流程

### 问：Shiro 的认证流程是什么？

```
1. Subject.login(token) 发起登录
         │
         ▼
2. SecurityManager.login() 接收请求
         │
         ▼
3. Authenticator.authenticate() 执行认证
         │
         ▼
4. ModularRealmAuthenticator 遍历 Realm
         │
         ▼
5. Realm.doGetAuthenticationInfo() 查询用户信息
         │
         ▼
6. CredentialsMatcher 验证密码
         │
         ▼
7. 认证成功，创建已认证的 Subject
```

### 问：Shiro 有哪些认证异常？

| 异常 | 说明 |
|---|---|
| `UnknownAccountException` | 用户不存在 |
| `IncorrectCredentialsException` | 密码错误 |
| `LockedAccountException` | 账户被锁定 |
| `ExpiredCredentialsException` | 凭证过期 |

### 问：ModularRealmAuthenticator 在多 Realm 场景下如何工作？

按顺序尝试每个 Realm，只要有一个 Realm 返回认证信息成功，就认为认证通过。如果所有 Realm 都失败，才抛出异常。

```java
protected AuthenticationInfo doMultiRealmAuthentication(
        Collection<Realm> realms, AuthenticationToken token) {
    AuthenticationInfo aggregatedAccountInfo = null;
    for (Realm realm : realms) {
        if (realm.supports(token)) {
            AuthenticationInfo info = realm.getAuthenticationInfo(token);
            if (info != null) {
                return info;  // 第一个成功就返回
            }
        }
    }
    throw new AuthenticationException();  // 全部失败
}
```

## 三、授权流程

### 问：Shiro 的授权流程是什么？

```
1. Subject.isPermitted("user:delete") 发起权限检查
         │
         ▼
2. SecurityManager 委托给 Authorizer
         │
         ▼
3. Realm.doGetAuthorizationInfo() 查询权限信息
         │
         ▼
4. PermissionResolver 解析权限字符串
         │
         ▼
5. WildcardPermission.implies() 匹配权限
         │
         ▼
6. 返回权限检查结果
```

### 问：hasRole() 和 isPermitted() 的区别？

| 方法 | 检查内容 | 适用场景 |
|---|---|---|
| `hasRole(role)` | 角色 | 按角色粗粒度划分 |
| `isPermitted(permission)` | 权限 | 按权限细粒度控制资源 |

### 问：WildcardPermission 如何匹配？

```java
"user:*"  implies "user:create"   // true
"user:*"  implies "user:delete"   // true
"*:read"  implies "order:read"    // true
"user:create" implies "user:*"    // false
```

## 四、Shiro vs Spring Security

### 问：Shiro 和 Spring Security 的区别？

| 对比项 | Shiro | Spring Security |
|---|---|---|
| 学习曲线 | 平缓 | 陡峭 |
| 配置复杂度 | 简单 | 复杂 |
| 依赖 | 无 Spring 依赖 | 深度耦合 Spring |
| 功能 | 基础安全够用 | OAuth2/LDAP 等企业级 |
| 适用场景 | 中小型项目 | 企业级项目 |
| 微服务 | 需改造 | 内置支持 |

### 问：什么时候选 Shiro？

- 项目需要快速交付、人员安全知识储备不足
- 只做基础的认证授权
- 非 Spring 项目

### 问：什么时候选 Spring Security？

- 需要 OAuth2、LDAP、CAS 等企业级认证
- 已在使用 Spring 全家桶
- 微服务架构，需要 Spring Cloud 安全集成

## 五、Session 管理

### 问：Shiro 的 Session 和 Servlet 的 Session 有什么关系？

`DefaultWebSessionManager` 会自动与 Servlet 容器集成，从请求中获取 Session ID。但 Shiro 使用自己的 Session 机制，不依赖容器。

- Web 环境：`DefaultWebSecurityManager` + `DefaultWebSessionManager`
- 普通 Java：`DefaultSecurityManager` + `DefaultSessionManager`

### 问：如何实现分布式 Session？

使用 `SessionDAO` 将 Session 存储到 Redis：

```java
// 配置 RedisSessionDAO
DefaultWebSessionManager sessionManager = new DefaultWebSessionManager();
sessionManager.setSessionDAO(new RedisSessionDAO(redisManager));
```

### 问：Subject.getSession(true) 和 getSession(false) 的区别？

- `getSession(true)`：如果 Session 不存在，则创建新的
- `getSession(false)`：如果 Session 不存在，返回 null

## 六、RememberMe

### 问：Shiro 怎么实现 RememberMe？

1. 登录时设置 `token.setRememberMe(true)`
2. `SimpleRememberMeManager` 生成加密 Cookie
3. 写入浏览器 Cookie，格式为 `Base64(AES(principal))`
4. 下次请求时，过滤器解析 Cookie，自动创建 Subject

### 问：isAuthenticated() 和 isRemembered() 的区别？

| 方法 | 状态 | 能做什么 |
|---|---|---|
| `isAuthenticated()` | 已认证（登录过） | 所有功能 |
| `isRemembered()` | RememberMe | 只能访问公开资源 |

:::warning 安全注意
RememberMe 用户不能执行敏感操作，敏感操作前必须检查 `isAuthenticated()`。
:::

## 七、自定义 Realm

### 问：自定义 Realm 需要重写哪些方法？

继承 `AuthorizingRealm`，重写两个核心方法：

```java
public class UserRealm extends AuthorizingRealm {

    // 认证
    @Override
    protected AuthenticationInfo doGetAuthenticationInfo(
            AuthenticationToken token) {
        // 返回 AuthenticationInfo
    }

    // 授权
    @Override
    protected AuthorizationInfo doGetAuthorizationInfo(
            PrincipalCollection principals) {
        // 返回 AuthorizationInfo
    }
}
```

### 问：密码加密用什么？

推荐使用 `HashedCredentialsMatcher`：

```java
HashedCredentialsMatcher matcher = new HashedCredentialsMatcher();
matcher.setHashAlgorithmName("SHA-256");
matcher.setHashIterations(1024);
realm.setCredentialsMatcher(matcher);
```

### 问：为什么要加盐？

防止彩虹表攻击。每个用户使用独立的随机盐值，即使密码相同，存储结果也不同。

## 八、过滤器链

### 问：Shiro 的过滤器链顺序是什么？

按配置顺序从上到下匹配，**第一个匹配的过滤器生效**：

```
/logout      → LogoutFilter      → 先处理登出
/login       → AnonymousFilter    → 公开路径
/static/**   → AnonymousFilter    → 静态资源
/admin/**    → RolesFilter       → 角色验证
/api/**      → PermissionsFilter → 权限验证
/**          → AuthcFilter       → 需要认证
```

### 问：如何自定义过滤器？

继承 `AccessControlFilter`，重写两个方法：

```java
public class IpBlacklistFilter extends AccessControlFilter {

    @Override
    protected boolean isAccessAllowed(ServletRequest request,
                                     ServletResponse response,
                                     Object mappedValue) {
        // 允许访问的逻辑，返回 true 表示允许
    }

    @Override
    protected boolean onAccessDenied(ServletRequest request,
                                    ServletResponse response) {
        // 拒绝访问的逻辑
    }
}
```

## 九、缓存机制

### 问：Shiro 的缓存机制是什么？

Shiro 内置了两层缓存：

- **AuthenticationCache**：认证信息缓存
- **AuthorizationCache**：授权信息缓存

缓存实现：

- `MemoryConstrainedCacheManager`：内存缓存（开发环境）
- `EhCacheManager`：EhCache（单节点）
- `RedisCacheManager`：Redis（分布式）

### 问：权限变更后如何清除缓存？

```java
// 方式一：退出登录
subject.logout();

// 方式二：手动清除
realm.getAuthorizationCache().remove(username);

// 方式三：清除所有
realm.getAuthorizationCache().clear();
```

## 十、注解

### 问：Shiro 注解有哪些？

| 注解 | 说明 |
|---|---|
| `@RequiresAuthentication` | 需要已认证 |
| `@RequiresUser` | 需要已认证或 RememberMe |
| `@RequiresGuest` | 需要匿名 |
| `@RequiresRoles` | 需要特定角色 |
| `@RequiresPermissions` | 需要特定权限 |

### 问：注解为什么不生效？

1. 未配置 `AuthorizationAttributeSourceAdvisor`
2. 未配置 `DefaultAdvisorAutoProxyCreator`
3. 同类内部调用不走代理（AOP 限制）
4. Spring Boot 3.x 缺少 `aspectjweaver` 依赖

## 十一、CSRF 防护

### 问：Shiro 有 CSRF 防护吗？

没有内置。需要自定义 `OncePerRequestFilter` 实现：

```java
public class CsrfFilter extends OncePerRequestFilter {

    @Override
    protected boolean isAccessAllowed(ServletRequest request,
                                     ServletResponse response,
                                     Object mappedValue) {
        // 验证 CSRF Token
    }
}
```

### 问：CSRF 攻击的原理是什么？

攻击者诱导用户访问恶意页面，该页面自动向目标站点发起请求，由于用户已登录，请求会携带认证 Cookie，导致未授权操作被执行。

## 十二、JWT 无状态

### 问：Shiro 如何实现无状态认证？

1. 自定义 `JwtToken` 封装 Token
2. 自定义 `JwtFilter` 验证 Token，构建 Subject
3. 自定义 `JwtRealm` 从 Token 获取用户信息
4. 关闭 Shiro Session

### 问：Shiro Session 和 JWT Token 如何选择？

| 场景 | 推荐方案 |
|---|---|
| 传统 Web 应用 | Shiro Session |
| 前后端分离 | JWT Token |
| 微服务架构 | JWT Token |
| 需要实时注销 | Shiro Session |
