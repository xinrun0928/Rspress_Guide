# Shiro 权限数据模型：用户-角色-权限树状结构

权限系统的表结构怎么设计？这是每一个权限系统都要面对的问题。

设计得好，后续扩展灵活、维护简单；设计得差，改一个小需求都要改表结构。常见的方案有三种：**用户-角色-权限**、**树状权限**、**ACL（访问控制列表）**。

## 三种权限模型

### 1. ACL（Access Control List）

最简单直接：用户直接关联权限。

```sql
-- 用户-权限表
CREATE TABLE user_permission (
    user_id BIGINT,
    permission VARCHAR(100),
    PRIMARY KEY (user_id, permission)
);
```

**优点**：灵活
**缺点**：权限分散，难以批量管理

### 2. RBAC（Role-Based Access Control）

引入角色作为中间层：用户→角色→权限。

```sql
-- 用户表
CREATE TABLE sys_user (
    id BIGINT PRIMARY KEY,
    username VARCHAR(50),
    password VARCHAR(100)
);

-- 角色表
CREATE TABLE sys_role (
    id BIGINT PRIMARY KEY,
    role_code VARCHAR(50),
    role_name VARCHAR(100)
);

-- 权限表
CREATE TABLE sys_permission (
    id BIGINT PRIMARY KEY,
    permission_code VARCHAR(100),
    permission_name VARCHAR(100)
);

-- 用户-角色关联表
CREATE TABLE sys_user_role (
    user_id BIGINT,
    role_id BIGINT,
    PRIMARY KEY (user_id, role_id)
);

-- 角色-权限关联表
CREATE TABLE sys_role_permission (
    role_id BIGINT,
    permission_id BIGINT,
    PRIMARY KEY (role_id, permission_id)
);
```

**优点**：通过角色批量管理权限
**缺点**：角色是静态的，不够灵活

### 3. 树状权限（Tree-Based Permission）

权限组织成树状结构，支持继承。

```
├── 系统管理
│   ├── 用户管理
│   │   ├── user:create
│   │   ├── user:read
│   │   ├── user:update
│   │   └── user:delete
│   └── 角色管理
│       ├── role:create
│       ├── role:read
│       └── role:*
└── 业务管理
    ├── 订单管理
    │   └── order:*
    └── 报表管理
        └── report:*
```

**优点**：权限可继承，支持细粒度控制
**缺点**：实现复杂

## 推荐方案：RBAC + 树状权限

实际项目中，推荐使用 **RBAC + 树状权限** 的混合方案：

```sql
-- 用户表
CREATE TABLE sys_user (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    username VARCHAR(50) UNIQUE NOT NULL,
    password VARCHAR(100) NOT NULL,
    salt VARCHAR(50),
    locked TINYINT(1) DEFAULT 0,
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 角色表
CREATE TABLE sys_role (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    role_code VARCHAR(50) UNIQUE NOT NULL,
    role_name VARCHAR(100),
    description VARCHAR(255),
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 权限表（树状结构）
CREATE TABLE sys_permission (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    parent_id BIGINT DEFAULT 0,          -- 父权限 ID，0 表示根节点
    permission_code VARCHAR(100) UNIQUE,
    permission_name VARCHAR(100),
    permission_type TINYINT(1),          -- 1=菜单, 2=按钮, 3=接口
    url VARCHAR(255),                     -- 对应的 URL（可选）
    icon VARCHAR(50),                     -- 菜单图标（可选）
    sort_order INT DEFAULT 0,
    status TINYINT(1) DEFAULT 1,
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 用户-角色关联表
CREATE TABLE sys_user_role (
    user_id BIGINT NOT NULL,
    role_id BIGINT NOT NULL,
    PRIMARY KEY (user_id, role_id)
);

-- 角色-权限关联表
CREATE TABLE sys_role_permission (
    role_id BIGINT NOT NULL,
    permission_id BIGINT NOT NULL,
    PRIMARY KEY (role_id, permission_id)
);
```

## 权限树查询

### MyBatis 查询用户的所有权限

```xml title="PermissionMapper.xml"
<!-- 查询用户的所有权限（包含父权限，用于继承） -->
<select id="selectPermissionsByUserId" resultMap="PermissionResultMap">
    WITH RECURSIVE permission_tree AS (
        -- 从用户关联的角色和权限开始
        SELECT p.*
        FROM sys_permission p
        INNER JOIN sys_role_permission rp ON p.id = rp.permission_id
        INNER JOIN sys_user_role ur ON rp.role_id = ur.role_id
        WHERE ur.user_id = #{userId}

        UNION ALL

        -- 递归获取父权限
        SELECT p.*
        FROM sys_permission p
        INNER JOIN permission_tree pt ON p.id = pt.parent_id
    )
    SELECT DISTINCT * FROM permission_tree
</select>

<resultMap id="PermissionResultMap" type="Permission">
    <id property="id" column="id"/>
    <result property="parentId" column="parent_id"/>
    <result property="permissionCode" column="permission_code"/>
    <result property="permissionName" column="permission_name"/>
    <result property="permissionType" column="permission_type"/>
    <result property="url" column="url"/>
</resultMap>
```

### Java 实现递归查询

```java
@Service
public class PermissionServiceImpl implements PermissionService {

    @Autowired
    private PermissionMapper permissionMapper;

    @Override
    public Set<String> findPermissionCodesByUserId(Long userId) {
        // 1. 获取用户关联的权限
        Set<String> directPermissions = permissionMapper
            .selectPermissionCodesByUserId(userId);

        // 2. 递归获取父权限
        Set<String> allPermissions = new HashSet<>();
        for (String code : directPermissions) {
            allPermissions.add(code);
            // 查询父权限链
            Set<String> parentPermissions =
                findParentPermissions(code);
            allPermissions.addAll(parentPermissions);
        }

        return allPermissions;
    }

    private Set<String> findParentPermissions(String permissionCode) {
        Set<String> parents = new HashSet<>();

        // 查询权限对应的菜单 ID
        Long permissionId = permissionMapper
            .selectIdByCode(permissionCode);

        // 递归向上查询父节点
        queryParentNodes(permissionId, parents);

        return parents;
    }

    private void queryParentNodes(Long permissionId, Set<String> parents) {
        Permission permission = permissionMapper.selectById(permissionId);
        if (permission != null && permission.getParentId() != 0) {
            // 添加父权限
            Permission parent = permissionMapper.selectById(permission.getParentId());
            if (parent != null) {
                parents.add(parent.getPermissionCode());
                // 继续向上查询
                queryParentNodes(parent.getId(), parents);
            }
        }
    }
}
```

## 权限缓存设计

### 多级缓存

```java
@Service
public class PermissionCacheService {

    @Autowired
    private CacheManager cacheManager;

    private static final String USER_PERMISSION_CACHE = "userPermissions";
    private static final String ROLE_PERMISSION_CACHE = "rolePermissions";
    private static final long CACHE_TTL = 1800;  // 30 分钟

    /**
     * 获取用户权限（带缓存）
     */
    public Set<String> getUserPermissions(Long userId) {
        String cacheKey = "user:" + userId;
        Cache<Object, Object> cache = cacheManager.getCache(USER_PERMISSION_CACHE);

        // 1. 查缓存
        @SuppressWarnings("unchecked")
        Set<String> cached = (Set<String>) cache.get(cacheKey);
        if (cached != null) {
            return cached;
        }

        // 2. 查数据库
        Set<String> permissions = permissionService
            .findPermissionCodesByUserId(userId);

        // 3. 写入缓存
        cache.put(cacheKey, permissions);

        return permissions;
    }

     * 清除用户权限缓存
     */
    public void evictUserCache(Long userId) {
        Cache<Object, Object> cache = cacheManager.getCache(USER_PERMISSION_CACHE);
        cache.remove("user:" + userId);
```

## Shiro 中的权限注册

### 动态权限加载

```java
@Component
public class ShiroFilterChainManager {

    @Autowired
    private ShiroFilterFactoryBean shiroFilterFactoryBean;

    @Autowired
    private PermissionService permissionService;

    /**
     * 动态更新权限过滤器链
     */
    public void updateFilterChain() {
        // 1. 获取现有过滤器链
        Map<String, Filter> filters = shiroFilterFactoryBean.getFilters();

        // 2. 从数据库加载 URL-权限映射
        List<Permission> urlPermissions = permissionService
            .findUrlPermissions();

        // 3. 构建新的过滤器链
        Map<String, String> filterChain = new LinkedHashMap<>();

        // 静态资源（公开）
        filterChain.put("/static/**", "anon");
        filterChain.put("/login", "anon");

        // 动态权限
        for (Permission perm : urlPermissions) {
            if (perm.getUrl() != null) {
                filterChain.put(
                    perm.getUrl(),
                    "perms[" + perm.getPermissionCode() + "]"
                );
            }
        }

        // 默认：需要认证
        filterChain.put("/**", "authc");

        // 4. 更新过滤器链
        shiroFilterFactoryBean.setFilterChainDefinitionMap(filterChain);
    }
}
```

## 数据权限设计（行级权限）

### 场景：用户只能查看自己的数据

```sql
-- 数据权限表
CREATE TABLE sys_data_permission (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id BIGINT,
    role_id BIGINT,
    data_scope VARCHAR(50),        -- ALL=全部, DEPT=本部门, SELF=本人
    dept_ids VARCHAR(500),        -- 允许访问的部门 ID 列表
    create_time DATETIME
);
```

### 实现

### DataScope 注解实现

```java
@Aspect
@Component
public class DataScopeAspect {

    @Autowired
    private DataPermissionMapper dataPermissionMapper;

    @Around("execution(* com.example.mapper.*Mapper.*(..))")
    public Object addDataScope(ProceedingJoinPoint point) throws Throwable {
        // 1. 获取当前用户的数据权限
        Subject subject = SecurityUtils.getSubject();
        String username = subject.getPrincipal().toString();

        DataScope dataScope = dataPermissionMapper.selectDataScopeByUsername(username);

        // 2. 根据数据权限修改 SQL
        if (dataScope != null) {
            switch (dataScope.getDataScope()) {
                case "ALL":
                    // 无限制
                    break;
                case "DEPT":
                    // 本部门数据
                    setParameter(point, "deptIds", dataScope.getDeptIds());
                    break;
                case "SELF":
                    // 仅本人数据
                    setParameter(point, "createBy", username);
                    break;
            }
        }

        return point.proceed();
    }
}
```

## 权限数据初始化

```java
@Component
public class PermissionInitializer implements ApplicationRunner {

    @Autowired

    @Override
    public void run(ApplicationArguments args) {
        // 初始化系统管理权限

        // 初始化业务管理权限

    }

        // 菜单
        Permission systemMenu = new Permission();
        systemMenu.setParentId(0L);
        systemMenu.setPermissionCode("system");

        systemMenu.setPermissionType(1);

        insertButton(userMenu.getId(), "user:delete", "删除用户", "/admin/users/*/delete");

        button.setPermissionCode(code);

        button.setPermissionType(2);  // 按钮类型

    }
```

## 面试高频问题

**问：权限表如何设计？**

推荐使用用户-角色-权限的 RBAC 模型，权限表使用树状结构支持继承。

**问：如何实现数据权限？**

在 SQL 层面注入数据范围条件，如 `WHERE create_by = #{currentUser}` 或 `WHERE dept_id IN (#{visibleDepts})`。

**问：权限变更后如何刷新缓存？**

在修改权限时，主动清除相关用户的权限缓存。可以使用 Redis 的发布订阅机制实现多实例间缓存同步。

**问：WildcardPermission 怎么匹配？**

`user:*` 匹配 `user:create`、`user:delete` 等所有以 `user:` 开头的权限。`*` 可以匹配任意字符串。

## 总结

权限数据模型设计要点：

```
权限模型核心：
├── 用户-角色-权限（RBAC）：标准模型
├── 树状权限：支持权限继承
├── 多级缓存：减少数据库查询
├── 数据权限：行级权限控制
└── 动态刷新：权限变更时清除缓存
```

好的权限模型设计，让系统安全又灵活。
