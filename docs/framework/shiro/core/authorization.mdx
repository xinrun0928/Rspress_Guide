# Shiro 授权流程：isPermitted() 的权限判断过程

认证回答的是"你是谁"的问题，授权回答的是"你能做什么"的问题。

用户登录成功了，但这不意味着他可以做任何事情。普通用户不能删除其他用户的账户，财务人员不能访问技术文档——这就是授权（Authorization）存在的意义。

Shiro 的授权流程比认证更复杂，因为它涉及**权限模型的设计**、**权限的解析**、**权限的匹配**等多个环节。理解了这套机制，你就能设计出灵活的权限系统。

## 授权的核心概念

在 Shiro 中，有三个关键概念需要分清：

- **Authentication**：认证，确认用户身份
- **Authorization**：授权，确认用户权限
- **Permission**：权限，一个资源+操作的组合，如 `user:create`、`order:read`

```java
// 认证成功后，才能进行授权检查
subject.login(token);

// 角色检查：有没有这个角色？
boolean hasRole = subject.hasRole("admin");

// 权限检查：有没有这个权限？
boolean isPermitted = subject.isPermitted("user:delete");
```

## hasRole() vs isPermitted()：什么时候用哪个？

这是两个最容易混淆的方法。

| 方法 | 检查内容 | 使用场景 |
|---|---|---|
| `hasRole(roleName)` | 用户是否拥有某个角色 | 按角色粗粒度划分权限 |
| `isPermitted(permission)` | 用户是否拥有某个权限 | 按权限细粒度控制资源 |
| `hasRoles(list)` | 批量检查多个角色 | 需要同时判断多个角色 |
| `isPermittedAll(permissions)` | 是否拥有所有权限 | 需要同时满足多个权限 |

```java
// 角色检查：判断是否有"管理员"角色
if (subject.hasRole("admin")) {
    // 可以访问管理后台
}

// 权限检查：判断是否有"删除用户"权限
if (subject.isPermitted("user:delete")) {
    // 可以删除用户
}

// 批量权限检查：需要同时拥有"订单查看"和"订单导出"权限
if (subject.isPermittedAll("order:read", "order:export")) {
    // 可以导出订单
}
```

:::tip 选型建议
在企业级应用中，推荐使用**权限检查**而非**角色检查**。角色是静态的，权限是动态的。当组织架构调整或业务流程变化时，改权限比改角色更灵活。
:::

## 权限模型：基于资源的权限设计

Shiro 使用**基于资源的权限模型**（Resource-Based Permission），权限表示为"资源:操作"的组合。

```
user:create     → 创建用户
user:read       → 查看用户
user:update     → 修改用户
user:delete     → 删除用户
order:create    → 创建订单
order:read      → 查看订单
order:*         → 所有订单相关操作
*:read          → 所有读操作
```

这种设计有几个优势：

1. **直观**：权限名称本身就是操作描述
2. **可组合**：可以用通配符表达批量权限
3. **可继承**：通过树状结构实现权限继承

## WildcardPermission：通配符权限

Shiro 内置了 `WildcardPermission` 类来解析和匹配权限字符串：

```java
// 精确匹配
Permission p1 = new WildcardPermission("user:create");
p1.implies(new WildcardPermission("user:create"));  // true

// 通配符匹配
Permission p2 = new WildcardPermission("user:*");
p2.implies(new WildcardPermission("user:create"));  // true
p2.implies(new WildcardPermission("user:delete"));  // true

// 多级通配符
Permission p3 = new WildcardPermission("user:create:*");
p3.implies(new WildcardPermission("user:create:batch"));  // true

// 根级别通配符
Permission p4 = new WildcardPermission("*");
p4.implies(new WildcardPermission("user:create"));  // true
p4.implies(new WildcardPermission("order:read"));   // true
```

### WildcardPermission 的匹配规则

```
权限字符串: user:create
拆分: ["user", "create"]

权限字符串: user:create:1
拆分: ["user", "create", "1"]

user:*
拆分: ["user", "*"]
```

匹配逻辑：

| 权限 A | 权限 B | A.implies(B) | 说明 |
|---|---|---|---|
| `user:*` | `user:create` | true | `*` 匹配任何操作 |
| `user:create` | `user:*` | false | `*` 不能被具体权限匹配 |
| `*:read` | `user:read` | true | 根级通配符匹配所有资源 |
| `user:create` | `user:create` | true | 完全相同 |

## isPermitted() 的完整流程

当调用 `subject.isPermitted("user:create")` 时，Shiro 内部经历了以下步骤：

### 第一步：SecurityManager 委托 Authorizer

```java
// org.apache.shiro.subject.Subject
public boolean isPermitted(String permission) {
    return this.securityManager.isPermitted(this.getPrincipals(), permission);
}

public boolean isPermitted(PrincipalCollection principals, String permission) {
    return this.authorizer.isPermitted(principals, permission);
}
```

### 第二步：Authorizer 调用 Realm

```java
// org.apache.shiro.realm.AuthorizingRealm
public boolean isPermitted(PrincipalCollection principals, String permission) {
    Permission p = this.permissionResolver.resolvePermission(permission);
    return this.isPermitted(principals, p);
}

public boolean isPermitted(PrincipalCollection principals, Permission permission) {
    AuthorizationInfo info = this.getAuthorizationInfo(principals);
    return this.isPermitted(info, permission);
}

private boolean isPermitted(AuthorizationInfo info, Permission permission) {
    Collection<Permission> perms = this.getPermissions(info);
    for (Permission perm : perms) {
        if (perm.implies(permission)) {
            return true;
        }
    }
    return false;
}
```

### 第三步：Realm 从数据库获取权限

```java
// 自定义 Realm 中实现
@Override
protected AuthorizationInfo doGetAuthorizationInfo(PrincipalCollection principals) {
    String username = (String) principals.getPrimaryPrincipal();

    // 从数据库查询用户权限
    Set<String> permissions = userMapper.selectPermissionsByUsername(username);

    // 从数据库查询用户角色
    Set<String> roles = userMapper.selectRolesByUsername(username);

    SimpleAuthorizationInfo info = new SimpleAuthorizationInfo();
    info.setRoles(roles);
    info.setStringPermissions(permissions);
    return info;
}
```

## 权限解析器：PermissionResolver

`PermissionResolver` 负责将字符串权限转换为 `Permission` 对象：

```java
// Shiro 默认使用 WildcardPermissionResolver
public interface PermissionResolver {
    Permission resolvePermission(String permissionString);
}

// 自定义权限解析器
public class CustomPermissionResolver implements PermissionResolver {
    @Override
    public Permission resolvePermission(String permissionString) {
        // 自定义解析逻辑
        if (permissionString.startsWith("regex:")) {
            return new RegexPermission(permissionString.substring(6));
        }
        return new WildcardPermission(permissionString);
    }
}

// 配置到 Realm
((AuthorizingRealm) realm).setPermissionResolver(customPermissionResolver);
```

## 角色权限解析器：RolePermissionResolver

有时候权限不是直接赋予用户的，而是通过角色间接赋予的。`RolePermissionResolver` 负责从角色解析出权限：

```java
public interface RolePermissionResolver {
    Collection<Permission> resolvePermissionsInRole(String roleString);
}

// 例如：角色 "admin" 对应以下权限
public class AdminRolePermissionResolver implements RolePermissionResolver {
    @Override
    public Collection<Permission> resolvePermissionsInRole(String role) {
        if ("admin".equals(role)) {
            return Arrays.asList(
                new WildcardPermission("user:*"),
                new WildcardPermission("order:*"),
                new WildcardPermission("system:*")
            );
        }
        return Collections.emptyList();
    }
}
```

## isPermitted vs isPermittedAll：单权限 vs 多权限

```java
// 方式一：isPermitted（任意一个满足即可）
if (subject.isPermitted("user:read") || subject.isPermitted("user:write")) {
    // 任一权限满足即可
}

// 方式二：isPermittedAll（必须全部满足）
if (subject.isPermittedAll("user:read", "user:write")) {
    // 两个权限都必须满足
}
```

`isPermittedAll` 的内部实现是短路求值，一旦发现某个权限不满足就立即返回：

```java
// ModularRealmAuthorizer.isPermittedAll()
public boolean isPermittedAll(PrincipalCollection principals, String... permissions) {
    if (permissions != null && permissions.length `>` 0) {
        for (String perm : permissions) {
            if (!isPermitted(principals, perm)) {
                return false;  // 短路
            }
        }
    }
    return true;
}
```

## 授权缓存：AuthorizationCache

每次权限检查都查数据库？性能撑不住。

Shiro 内置了授权缓存，认证通过后，权限信息会被缓存起来：

```java
// 认证成功后，调用 Realm.getAuthorizationInfo() 获取权限
// 结果会被存入 AuthorizationCache

// 下次权限检查时，先查缓存
AuthorizationInfo info = getCachedAuthorizationInfo(principals);
if (info == null) {
    // 缓存未命中，从数据库加载
    info = doGetAuthorizationInfo(principals);
    // 存入缓存
    cacheAuthorizationInfo(info);
}
```

### 缓存失效

当用户权限变更时，需要清除缓存：

```java
// 方式一：调用 Subjectlogout()
subject.logout();  // 清除当前 Subject 的缓存

// 方式二：清除特定用户的缓存
Realm realm = (Realm) securityManager.getRealms().iterator().next();
Cache<Object, AuthorizationInfo> authzCache = realm.getAuthorizationCache();
authzCache.remove(username);

// 方式三：使用 CacheManager 清除
cacheManager.getCache("shiro-activeSessionCache").remove(principal);
```

## 自定义权限检查：自定义 Permission 接口

除了字符串权限，Shiro 还支持自定义权限接口：

```java
// 定义权限接口
public interface SystemPermission extends Permission {
    boolean isSuperAdmin();
    boolean hasAccessToModule(String module);
}

// 实现自定义权限
public class SystemPermissionImpl implements SystemPermission {
    private final Set<String> modules;
    private final boolean superAdmin;

    @Override
    public boolean isSuperAdmin() {
        return superAdmin;
    }

    @Override
    public boolean hasAccessToModule(String module) {
        return superAdmin || modules.contains(module);
    }

    @Override
    public boolean implies(Permission permission) {
        if (permission instanceof SystemPermission) {
            SystemPermission sp = (SystemPermission) permission;
            return this.isSuperAdmin() || sp.hasAccessToModule(module);
        }
        return false;
    }
}
```

## 授权流程总结

```
用户调用 isPermitted("user:create")
         │
         ▼
Subject.isPermitted()
         │
         ▼
SecurityManager.isPermitted()
         │
         ▼
Authorizer.isPermitted()
         │
         ▼
Realm 从 AuthorizationCache 查缓存
         │
         ├── 缓存命中 → 直接返回权限列表
         │
         └── 缓存未命中 → 调用 doGetAuthorizationInfo()
              │
              ▼
         从数据库查询权限
              │
              ▼
         存入 AuthorizationCache
              │
              ▼
         返回权限列表
              │
              ▼
权限列表中的每个 Permission.implies()
         │
         ▼
任意一个 implies() 返回 true → isPermitted() 返回 true
```

## 面试追问

**问**：Shiro 的权限匹配是精确匹配还是包含匹配？

**答**：是包含匹配。`WildcardPermission.implies()` 方法会检查调用者的权限是否**包含**请求的权限。例如 `user:*` 包含 `user:create`，`*:read` 包含 `user:read`。

**问**：如何实现行级权限控制（如用户只能查看自己的订单）？

**答**：Shiro 的标准权限模型不支持行级控制。需要配合自定义 `Permission` 实现：

```java
public class OrderPermission implements Permission {
    private String orderId;
    private String ownerId;

    @Override
    public boolean implies(Permission permission) {
        if (permission instanceof OrderPermission) {
            OrderPermission op = (OrderPermission) permission;
            // 只有订单所有者或管理员可以访问
            return this.ownerId.equals(op.ownerId) || "admin".equals(this.ownerId);
        }
        return false;
    }
}
```

**问**：Shiro 和 Spring Security 的权限模型有什么不同？

**答**：Spring Security 使用基于角色的访问控制（RBAC），权限检查最终也是角色检查。Shiro 使用基于资源的权限模型（Permission-Based），权限本身就是资源+操作的组合，更灵活。
