# Shiro 架构：Subject、SecurityManager、Realm 三大核心

做安全功能，选 Shiro 还是 Spring Security？这个问题没有标准答案，但有一个值得参考的维度：**Spring Security 太重了**。它深度耦合 Spring 生态，学习曲线陡峭，配置项繁多。而 Shiro，用几百行配置就能跑通一个完整的权限系统。

这不是说 Shiro 不专业——恰恰相反，它的架构设计非常清晰，只需要理解三个核心概念，就能看懂整个框架的工作方式。

## Subject：当前操作用户

`Subject` 是 Shiro 中最"接地气"的接口，它代表当前与系统交互的任何实体——可以是一个用户、一个第三方服务、一个定时任务。所有的安全操作，最终都落实到 Subject 上。

```java
// 获取当前登录用户
Subject currentUser = SecurityUtils.getSubject();

// 判断是否认证
if (currentUser.isAuthenticated()) {
    // 获取用户名
    String username = (String) currentUser.getPrincipal();
}

// 权限检查
if (currentUser.isPermitted("user:create")) {
    // 有创建用户的权限
}
```

Shiro 的设计哲学是**面向接口编程**，`Subject` 有多种实现：

- `SecurityManager` 创建的 `Subject` 实例，用于普通的 Web 应用
- `Subject.Builder` 用于在无状态场景下手动构建 Subject
- 在测试环境或特殊场景下，甚至可以获取"系统 Subject"来执行特权操作

> Subject 本身不存储任何数据，它只是一个**门面（Facade）**，真正的安全逻辑都委托给 SecurityManager 处理。

## SecurityManager：Shiro 的心脏

如果说 Subject 是前台接待，那 SecurityManager 就是后台总管——它管理着 Shiro 所有与安全相关的组件，包括认证器、授权器、会话管理器、缓存管理器。

```java
// Shiro 的默认实现
DefaultSecurityManager securityManager = new DefaultSecurityManager();

// 注入 Realm
securityManager.setRealm(myRealm);

// 注入会话管理器（Web 环境用 DefaultWebSecurityManager）
DefaultWebSecurityManager webSecurityManager = new DefaultWebSecurityManager(realm);
```

SecurityManager 的继承体系值得深入理解：

```
SecurityManager (顶级接口)
├── AuthenticatingSecurityManager
│   └── AuthorizingSecurityManager
│       ├── SessionSecurityManager
│       │   └── DefaultSecurityManager (普通 Java 应用)
│       │   └── DefaultWebSecurityManager (Web 应用)
```

每一层都扩展了特定能力：

| 层级 | 组件 | 职责 |
|---|---|---|
| `AuthenticatingSecurityManager` | Authenticator | 处理认证逻辑 |
| `AuthorizingSecurityManager` | Authorizer | 处理授权逻辑 |
| `SessionSecurityManager` | SessionManager | 管理会话生命周期 |
| `DefaultWebSecurityManager` | CacheManager | 管理认证/授权缓存 |

这种分层设计让 SecurityManager 的职责清晰：它自己不做具体的事，而是协调各个子组件完成任务。

## Realm：安全数据的桥梁

Realm（域）是 Shiro 连接应用数据源的桥梁。它负责从数据库、配置文件或其他地方获取用户、角色、权限信息，交给 SecurityManager 使用。

```java
public interface Realm {
    // 获取 Realm 的名称
    String getName();

    // 判断是否支持指定的 Token 类型
    boolean supports(AuthenticationToken token);

    // 获取认证信息
    AuthenticationInfo getAuthenticationInfo(AuthenticationToken token)
        throws AuthenticationException;

    // 获取授权信息
    AuthorizationInfo getAuthorizationInfo(PrincipalCollection principals);
}
```

Realm 有多个实现，复杂度从低到高：

- `IniRealm`：从 INI 文件加载用户和权限，适合演示
- `JdbcRealm`：从数据库加载，SQL 固定，适合简单场景
- `AuthorizingRealm`：抽象类，需要自定义实现 doGetAuthenticationInfo 和 doGetAuthorizationInfo，是生产环境的标配

```
AuthenticatingRealm
└── AuthorizingRealm (抽象类，需要扩展)
    ├── JdbcRealm
    ├── IniRealm
    └── 自定义 Realm（实现 doGetAuthenticationInfo + doGetAuthorizationInfo）
```

### Realm 的认证流程

当调用 `subject.login(token)` 时，Shiro 的认证流程如下：

```java
// 1. Subject 发起登录请求
subject.login(new UsernamePasswordToken(username, password));

// 2. SecurityManager 接收请求，委托给 Authenticator
AuthenticationInfo info = authenticator.authenticate(token);

// 3. Authenticator 遍历所有 Realm，找到支持该 Token 的 Realm
for (Realm realm : realms) {
    if (realm.supports(token)) {
        info = realm.getAuthenticationInfo(token);
        break;
    }
}

// 4. Realm 从数据源获取用户信息，与 Token 中的凭证对比
// 如果匹配，返回 SimpleAuthenticationInfo
```

### Realm 的授权流程

授权发生在权限检查时：

```java
// 调用 isPermitted
if (subject.isPermitted("user:create")) { ... }

// SecurityManager 委托给 Authorizer
Authorizer authorizer = securityManager.getAuthorizer();

// Authorizer 从 Realm 获取权限信息
AuthorizationInfo info = realm.getAuthorizationInfo(subject.getPrincipals());

// 比对权限
boolean permitted = info.getStringPermissions().contains("user:create");
```

## 认证流程源码解析

让我们深入 Shiro 认证的核心流程，看看各组件是如何协作的。

### 入口：Subject.login()

```java
// org.apache.shiro.subject.Subject
public class Subject {
    private final SecurityManager securityManager;

    public void login(AuthenticationToken token) throws AuthenticationException {
        // 委托给 SecurityManager
        Subject subject = securityManager.login(this, token);
        // ... 省略缓存和状态同步代码
    }
}
```

### SecurityManager 的认证入口

```java
// org.apache.shiro.mgt.DefaultSecurityManager
public class DefaultSecurityManager extends AuthorizingSecurityManager {

    public Subject login(Subject subject, AuthenticationToken token) {
        AuthenticationInfo info;
        try {
            // 调用 Authenticator 的 authenticate 方法
            info = this.authenticate(token);
        } catch (AuthenticationException e) {
            // 认证失败处理
            throw e;
        }

        // 创建已认证的 Subject
        Subject loggedIn = createSubjectContext();
        loggedIn.setAuthenticated(true);
        loggedIn.mergePrincipalInfo(info);
        return loggedIn;
    }
}
```

### Authenticator 的认证实现

```java
// org.apache.shiro.mgt.ModularRealmAuthenticator
public class ModularRealmAuthenticator extends AuthenticatingSecurityManager {

    protected AuthenticationInfo doAuthenticate(AuthenticationToken token) {
        // 1. 获取所有配置的 Realm
        Collection<Realm> realms = getRealms();

        // 2. 单 Realm 场景：直接认证
        if (realms.size() == 1) {
            return doSingleRealmAuthentication(realms.iterator().next(), token);
        }

        // 3. 多 Realm 场景：遍历所有 Realm
        return doMultiRealmAuthentication(realms, token);
    }

    protected AuthenticationInfo doSingleRealmAuthentication(
            Realm realm, AuthenticationToken token) {
        // 1. 检查 Realm 是否支持该 Token 类型
        if (!realm.supports(token)) {
            throw new UnsupportedTokenException();
        }

        // 2. 调用 Realm 获取认证信息
        AuthenticationInfo info = realm.getAuthenticationInfo(token);

        // 3. 验证凭证
        return info;
    }
}
```

### Realm 的凭证验证

```java
// org.apache.shiro.realm.AuthenticatingRealm
public abstract class AuthenticatingRealm extends CachingRealm {

    public final AuthenticationInfo getAuthenticationInfo(AuthenticationToken token) {
        // 1. 先检查缓存
        AuthenticationInfo info = getCachedAuthenticationInfo(token);
        if (info != null) {
            return info;
        }

        // 2. 调用子类的认证实现
        info = doGetAuthenticationInfo(token);

        // 3. 凭证匹配（如果配置了 CredentialsMatcher）
        if (info != null) {
            assertCredentialsMatch(token, info);
        }

        // 4. 写入缓存
        if (info != null) {
            cacheAuthenticationInfoIfPossible(token, info);
        }

        return info;
    }
}
```

> **面试追问**：ModularRealmAuthenticator 在多 Realm 场景下是如何工作的？答案是按顺序尝试每个 Realm，只要有一个 Realm 返回认证信息成功，就认为认证通过。如果所有 Realm 都失败，才抛出异常。

## 总结：三个核心，各司其职

Shiro 的架构设计非常符合"门面+协调器"的经典模式：

- **Subject** 是用户操作的入口，提供 login、logout、isPermitted、getSession 等 API
- **SecurityManager** 是核心协调器，管理认证器、授权器、会话管理器等子组件
- **Realm** 是数据桥梁，连接 SecurityManager 和你的数据库/配置文件

理解了这三者的关系，就抓住了 Shiro 的核心。后续的认证授权、会话管理、缓存优化，都建立在这个架构之上。
