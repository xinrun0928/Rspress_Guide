# Shiro vs Spring Security：安全框架的选型之争

"Shiro 和 Spring Security 选哪个？"这是面试和项目选型时都会被问到的问题。

表面上，这是在比较两个框架的功能差异。但往深了想，这个问题背后其实是在问：**你的项目到底需要什么样的安全能力？**

## 功能丰富度：Spring Security 更全，但 Shiro 够用

Spring Security 脱胎于 Spring 生态，几乎涵盖了企业级安全的所有场景：

| 功能 | Spring Security | Shiro |
|---|---|---|
| 认证（Authentication） | 支持 | 支持 |
| 授权（Authorization） | 支持 | 支持 |
| 会话管理 | 支持 | 支持 |
| 密码加密 | 支持 | 支持 |
| OAuth2.0 | 内置支持 | 需第三方库 |
| SAML | 内置支持 | 不支持 |
| LDAP | 内置支持 | 需自定义 Realm |
| CAS | 内置支持 | 不支持 |
| 单点登录（SSO） | 支持 | 不支持 |
| 方法级安全（@PreAuthorize） | 支持 | 支持 |
| Web 安全过滤器 | 支持 | 支持 |
| CSRF 防护 | 内置 | 需自定义 |

从这个表格可以看出，Shiro 的功能覆盖了**最常用的安全场景**，但对于 SSO、OAuth2、SAML 这些企业级协议，Spring Security 有原生支持，Shiro 则需要借助第三方库或自行实现。

:::tip 选型建议
如果你只需要"用户登录 + 权限控制"的基础功能，Shiro 完全够用。如果你的系统需要对接 OAuth2、LDAP、CAS 等企业级认证协议，Spring Security 更适合。
:::

## 学习曲线：Shiro 更平缓，Spring Security 陡峭

这是 Shiro 最大的优势。

Shiro 的核心概念只有三个：**Subject、SecurityManager、Realm**。理解了这三者，就能看懂整个框架的工作方式。再花半小时了解过滤器链和注解配置，就能上手开发了。

```java
// Shiro 的认证，三行代码搞定
Subject subject = SecurityUtils.getSubject();
subject.login(new UsernamePasswordToken(username, password));
subject.isPermitted("user:create");
```

Spring Security 则完全不同。它有大量概念需要理解：

- AuthenticationManager / AuthenticationProvider
- UserDetailsService / UserDetails
- SecurityFilterChain
- MethodSecurityExpressionHandler
- @PreAuthorize / @PostAuthorize
- CSRF Token 管理
- Session 并发控制
- 记住我（RememberMe）

```java
// Spring Security 的认证，理解成本更高
@Configuration
@EnableWebSecurity
public class SecurityConfig {
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/login").permitAll()
                .anyRequest().authenticated()
            )
            .formLogin(form -> form
                .loginPage("/login")
                .permitAll()
            );
        return http.build();
    }
}
```

Spring Security 的配置看似简单，但背后隐藏着大量的 Spring IoC 机制和过滤器链知识。出了问题，调试成本高。

## 社区生态：Spring 生态完胜

Spring Security 有 Spring 团队背书，文档完善、社区活跃、版本迭代稳定。任何 Spring Security 的问题，Google 一搜一大把解决方案。

Shiro 呢？2010 年进入 Apache 基金会，2019 年后活跃度明显下降。GitHub 上的 issues 回复变慢，Stack Overflow 上的问题很多无人应答。

:::warning 注意
Spring Boot 2.x 之后，官方移除了对 Shiro 的自动集成支持。需要手动配置 SecurityManager 和 Realm，增加了集成成本。
:::

## 性能对比：Shiro 更轻量

Shiro 核心包只有几百 KB，不依赖 Spring，可以独立运行。在资源受限的环境下，Shiro 的启动时间和内存占用都更低。

Spring Security 则依赖 Spring Framework，整个依赖链较长。对于不需要 Spring 生态的项目，Shiro 的轻量优势明显。

## 微服务场景下的选型

微服务架构下，Shiro 和 Spring Security 的表现差异更大：

| 维度 | Shiro | Spring Security |
|---|---|---|
| 无状态认证 | 需改造（见 JWT 实战） | 可直接配合 Spring Security OAuth2 |
| 服务间调用 | 无内置方案 | 可用 Spring Cloud Security |
| 配置中心集成 | 无 | 可与 Spring Cloud Config 集成 |
| 与网关集成 | 无特殊支持 | 可与 Spring Cloud Gateway 集成 |

在微服务场景下，如果你的服务需要对接 Spring Cloud 全家桶，Spring Security 的集成成本更低。如果你的服务是独立的 PHP/Python/Node 服务，Shiro 只适用于 Java 服务。

## 选型决策树

```
项目需要 SSO/OAuth2/SAML？
├── 是 → 选择 Spring Security
└── 否 → 项目是否依赖 Spring Framework？
    ├── 否 → 选择 Shiro（或选择项目对应的安全框架）
    └── 是 → 用户量级是否 `<=` 1000？
        ├── 是 → 选择 Shiro（简单够用）
        └── 否 → 选择 Spring Security（功能更全）
```

## 总结：没有最好的，只有最适合的

Shiro 和 Spring Security 的选择，本质上是在问：**你的项目愿意为安全模块投入多少成本？**

- 如果你的项目需要快速交付、人员安全知识储备不足、只需基础的认证授权，**选 Shiro**
- 如果你的项目需要 OAuth2、LDAP、企业级 SSO，或者已经在使用 Spring 全家桶，**选 Spring Security**
- 如果你的项目未来可能扩展到微服务、对接第三方登录、需要 Spring Cloud 安全集成，**选 Spring Security**

记住：**框架是工具，不是护城河**。真正保护系统安全的，是合理的权限设计、完善的日志审计、以及对安全漏洞的快速响应能力。
