# Spring Boot 面试高频问题汇总

大多数面试官不会直接问你「Spring Boot 是什么」，而是会通过一系列追问，考察你对 Spring Boot 核心原理的理解程度。从自动配置到条件装配，从启动流程到外部化配置，每一个环节都可能是考察点。

## 一、自动配置原理

### @SpringBootApplication 包含哪几个注解？

这是 Spring Boot 面试的开场题，也是送分题。

**参考答案：**

`@SpringBootApplication` 是一个组合注解，包含三个核心注解：

1. `@Configuration`：标注当前类是配置类，Spring IOC 容器会将其作为 Bean 定义的来源
2. `@EnableAutoConfiguration`：启用自动配置，让 Spring Boot 自动加载 `META-INF/spring.factories` 或 `AutoConfiguration.imports` 中定义的自动配置类
3. `@ComponentScan`：开启组件扫描，默认扫描当前类所在包及其子包中的 `@Component`、`@Service`、`@Repository`、`@Controller` 等注解

**追问：@EnableAutoConfiguration 是如何工作的？**

`@EnableAutoConfiguration` 通过 `@Import` 导入了 `AutoConfigurationImportSelector`，该类实现了 `ImportSelector` 接口。在 `selectImports()` 方法中，会读取 `META-INF/spring.factories`（JDK 8 之前）或 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`（JDK 9+）文件中的自动配置类，并交由 Spring 容器管理。

### 自动配置是什么时候触发的？

自动配置的触发发生在 Spring 容器刷新（refresh）阶段。具体来说，是在 `ConfigurationClassPostProcessor` 处理 `@Configuration` 类的过程中，通过 `@EnableAutoConfiguration` 导入的 `AutoConfigurationImportSelector` 加载并注册自动配置类。

### spring.factories 和 AutoConfiguration.imports 的区别？

这是 Spring Boot 2.7.x 升级到 3.x 时的重大变化：

| 对比项 | spring.factories | AutoConfiguration.imports |
|---|---|---|
| 首次出现 | Spring Boot 1.5.x | Spring Boot 2.7.x |
| 正式支持 | JDK 8 及以下 | JDK 9+ |
| 文件位置 | `META-INF/spring.factories` | `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` |
| 格式 | Properties 格式，key 是接口，value 是逗号分隔的类名 | 每行一个类名 |

Spring Boot 3.x（对应 Spring Framework 6.x）只支持 `AutoConfiguration.imports` 文件，不再支持 `spring.factories`。

## 二、条件装配

### @ConditionalOnClass 的作用是什么？

`@ConditionalOnClass` 用于控制配置类是否生效。当且仅当 classpath 中存在指定的类时，对应的 `@Configuration` 类才会被加载。

**追问：底层是如何实现的？**

`@ConditionalOnClass` 的实现依赖于 `Condition` 接口的 `matches()` 方法。Spring Boot 提供了多个 `Condition` 的实现类，其中 `ConditionalOnClassCondition` 通过 `ClassUtils.isPresent()` 判断类是否存在于 classpath 中：

```java
public class ConditionalOnClassCondition extends SpringBootCondition {

    @Override
    public ConditionOutcome getMatchOutcome(
            ConditionContext context, 
            AnnotatedTypeMetadata metadata) {
        // 解析 @ConditionalOnClass 中的类名
        MultiValueMap<String, Object> classes = 
            metadata.getAllAnnotationAttributes(ConditionalOnClass.class.getName());
        
        for (String className : classes.get("value")) {
            // 判断类是否存在
            if (!ClassUtils.isPresent(className, getClass().getClassLoader())) {
                return ConditionOutcome.noMatch(
                    "required class not present: " + className);
            }
        }
        return ConditionOutcome.match();
    }
}
```

### 常用的条件注解有哪些？

| 注解 | 作用 |
|---|---|
| `@ConditionalOnClass` | classpath 中存在指定类时生效 |
| `@ConditionalOnMissingClass` | classpath 中不存在指定类时生效 |
| `@ConditionalOnBean` | 容器中存在指定 Bean 时生效 |
| `@ConditionalOnMissingBean` | 容器中不存在指定 Bean 时生效 |
| `@ConditionalOnProperty` | 配置属性满足条件时生效 |
| `@ConditionalOnWebApplication` | 是 Web 应用时生效 |

### @ConditionalOnBean 和 @ConditionalOnMissingBean 的坑？

这两个注解容易导致配置类的加载顺序问题。如果一个 `@ConditionalOnMissingBean` 的配置类先于实际 Bean 定义执行，可能会错误地覆盖用户自定义的 Bean。

Spring Boot 的解决方案是：**自动配置类通常会使用 `@AutoConfigureAfter` 或 `@AutoConfigureOrder` 来控制加载顺序**，确保用户定义的 Bean 先于自动配置类处理。

## 三、启动流程

### SpringApplication.run() 做了哪些事情？

`SpringApplication.run()` 的执行流程可以分为以下几个阶段：

**1. 创建 SpringApplication 实例**

- 判断应用类型（Servlet、Reactive 或 None）
- 初始化 `bootstrapRegistryInitializers`
- 设置 `initializers`（ApplicationContextInitializer）
- 设置 `listeners`（ApplicationListener）
- 推断主配置类（包含 `@SpringBootApplication` 的类）

**2. 执行 run() 方法**

- 创建 `DefaultBootstrapContext`
- 注册 `BootstrapRegistryInitializer`
- 构造 `SpringApplicationRunListeners`
- 调用 `starting()` 事件
- 准备 `ConfigurableEnvironment`（加载 application.yml 等配置）
- 打印 Banner
- 创建 `ConfigurableApplicationContext`（根据应用类型创建不同的容器）
- 准备容器（调用 `applyInitializers()`）
- 刷新容器（调用 `context.refresh()`）
- 刷新后处理（查找并调用 `ApplicationRunner` 和 `CommandLineRunner`）

**追问：为什么需要 BootstrapContext？**

`BootstrapContext` 是 Spring Boot 2.4 引入的，主要用于在主应用上下文启动之前，临时注册一些 Bean。在 Spring Cloud 场景下，`BootstrapContext` 用于加载 `bootstrap.yml` 配置。

## 四、外部化配置

### application.yml 的加载顺序？

Spring Boot 的配置加载顺序（从低到高，后面的覆盖前面的）：

1. classpath 根目录的 application.yml
2. classpath 根目录的 application.properties
3. classpath 下 config 子目录的 application.yml
4. 当前目录的 application.yml（项目根目录）
5. 当前目录的 config 子目录的 application.yml
6. 命令行参数 `--spring.config.additional-location`

**追问：Spring Boot 3.x 的变化？**

Spring Boot 3.0 之后，`application.yml` 的加载顺序略有调整，新增了 `spring.config.import` 配置，可以导入外部配置文件。

### @ConfigurationProperties 和 @Value 的区别？

| 对比项 | @ConfigurationProperties | @Value |
|---|---|---|
| 绑定方式 | 批量绑定一组配置属性 | 逐个注入单个属性 |
| 功能 | 支持宽松绑定（Relaxed Binding） | 不支持 |
| 元数据 | 支持 JSR-303 校验（`@Validated`） | 不支持 |
| SPEL | 不支持 | 支持 `#{}` 表达式 |
| 适用场景 | 配置类 POJO | 简单属性注入 |

**追问：什么是宽松绑定？**

宽松绑定允许配置属性名与 POJO 字段名不完全匹配。例如，`my-app-name` 可以绑定到 `myAppName`、`myAppName` 或 `MY_APP_NAME`：

```java
@ConfigurationProperties(prefix = "app")
public class AppProperties {
    private String appName;  // 可以绑定 app.app-name、app.appName、app.APP_NAME
}
```

## 五、Starter 机制

### 自定义 Starter 的开发流程？

一个标准的 Spring Boot Starter 由两个模块组成：

1. **xxx-starter**：仅包含 POM 依赖，依赖 auto-configure 模块
2. **xxx-autoconfigure**：包含自动配置类、spring.factories 或 AutoConfiguration.imports

```xml
<!-- xxx-starter/pom.xml -->
<dependency>
    <groupId>com.example</groupId>
    <artifactId>xxx-autoconfigure</artifactId>
    <version>1.0.0</version>
</dependency>
```

### 为什么 Spring Boot 的 Starter 能够自动配置？

这是因为 Spring Boot 的自动配置机制：

1. Starter 的 auto-configure 模块包含 `@Configuration` 类
2. 该类使用 `@ConditionalOnClass` 判断依赖是否存在
3. 配置类注册到 `META-INF/spring.factories` 或 `AutoConfiguration.imports`
4. 当添加 Starter 依赖后，条件满足，配置类自动生效

## 六、综合应用

### 如何排查 Spring Boot 自动配置不生效的问题？

1. 启动应用时添加 `--debug` 参数，查看自动配置报告
2. 查找 `ConditionEvaluationReport`（自动配置条件评估报告）
3. 定位未匹配的条件及其原因
4. 检查是否被 `@ConditionalOnMissingBean` 覆盖

### Spring Boot 如何实现多环境配置？

Spring Boot 支持通过 `spring.profiles.active` 激活不同的环境配置：

```yaml
# application-dev.yml
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/dev

# application-prod.yml
spring:
  datasource:
    url: jdbc:mysql://prod:3306/prod
```

激活方式：
- 命令行：`--spring.profiles.active=prod`
- 环境变量：`SPRING_PROFILES_ACTIVE=prod`
- JVM 参数：`-Dspring.profiles.active=prod`

### Spring Boot 的自动配置是否会干扰用户自定义的 Bean？

Spring Boot 的设计原则是「用户自定义优先」。对于大多数场景，自动配置类中的 `@ConditionalOnMissingBean` 确保了用户自定义的 Bean 优先生效。但需要注意：

1. 自动配置类的加载顺序（通过 `@AutoConfigureAfter` 控制）
2. `@ConditionalOnMissingBean` 的生效前提是扫描到该条件注解
3. 某些场景下可能需要使用 `@EnableConfigurationProperties` 显式启用

## 总结

Spring Boot 面试的核心在于**理解其设计理念和实现原理**。从自动配置到条件装配，从启动流程到外部化配置，每一个环节都体现了「约定优于配置」的思想。
