# @ConditionalOnClass、@ConditionOnBean 等条件注解

你有没有想过，为什么 Spring Boot 的自动配置能够「智能」地判断该配置哪些 Bean？

当你添加 `spring-boot-starter-data-redis` 依赖后，Redis 相关的自动配置就生效了；当你添加 `HikariCP` 依赖后，数据源就自动配置为 Hikari 连接池。这个「智能」的背后，就是 `@Conditional` 系列注解在起作用。

## 一、条件注解概述

Spring Boot 提供了丰富的条件注解，用于控制配置类或 `@Bean` 方法是否生效。这些注解都基于 Spring Framework 的 `@Conditional` 注解实现。

### 条件注解家族

| 注解 | 作用 | 使用场景 |
|---|---|---|
| `@ConditionalOnClass` | classpath 中存在指定类时生效 | 检测依赖是否存在 |
| `@ConditionalOnMissingClass` | classpath 中不存在指定类时生效 | 避免冲突 |
| `@ConditionalOnBean` | 容器中存在指定 Bean 时生效 | 依赖其他 Bean |
| `@ConditionalOnMissingBean` | 容器中不存在指定 Bean 时生效 | 用户自定义优先 |
| `@ConditionalOnProperty` | 配置属性满足条件时生效 | 功能开关 |
| `@ConditionalOnWebApplication` | 是 Web 应用时生效 | Web/普通应用区分 |
| `@ConditionalOnExpression` | SPEL 表达式为 true 时生效 | 复杂条件 |

## 二、@ConditionalOnClass 详解

### 基本用法

```java
@ConditionalOnClass(Gson.class)
@Configuration
public class GsonAutoConfiguration {
    @Bean
    public Gson gson() {
        return new Gson();
    }
}
```

只有当 classpath 中存在 `Gson` 类时，`GsonAutoConfiguration` 才会被加载。

### 多个条件

```java
@ConditionalOnClass({ DataSource.class, JdbcTemplate.class })
@Configuration
public class JdbcTemplateAutoConfiguration {
    // ...
}
```

只有当 classpath 中同时存在 `DataSource` 和 `JdbcTemplate` 时才生效。

### 条件不满足时的处理

当条件不满足时，Spring Boot 不会报错，只是静静地跳过这个配置类。这正是自动配置「智能」的体现——有则配置，无则跳过。

## 三、条件注解的底层实现

### SpringBootCondition 抽象类

Spring Boot 的条件注解都继承自 `SpringBootCondition`，它实现了 `Condition` 接口：

```java
public abstract class SpringBootCondition implements Condition {
    @Override
    public final boolean matches(ConditionContext context, AnnotatedTypeMetadata metadata) {
        String classOrMethodName = getClass().getSimpleName();
        // 获取 @Conditional 注解的信息
        ConditionAndSpec matchSpec = getConditionAndSpec(metadata);
        // 执行匹配逻辑
        return matchSpec.match(context, metadata);
    }

    protected abstract ConditionAndSpec getConditionAndSpec(
            ConditionContext context, AnnotatedTypeMetadata metadata);
}
```

### ConditionalOnClassCondition 实现

```java
public class ConditionalOnClassCondition extends SpringBootCondition {

    @Override
    protected ConditionAndSpec getConditionAndSpec(
            ConditionContext context, AnnotatedTypeMetadata metadata) {
        // 解析 @ConditionalOnClass 注解
        MultiValueMap<String, Object> classes = metadata.getAllAnnotationAttributes(
            ConditionalOnClass.class.getName());

        List<String> classNames = new ArrayList<>();
        if (classes != null) {
            // 收集类名
            for (Object value : classes.get("value")) {
                classNames.add((String) value);
            }
        }

        // 构建条件规格
        ConditionSpecBuilder builder = new ConditionSpecBuilder();
        return builder.whenCondition(OnClassCondition.class, classNames,
            (configurations, autoConfigurationMetadata) -> {
                // 实际的匹配逻辑
                return getMatchOutcome(configurations, autoConfigurationMetadata);
            }).toConditionAndSpec();
    }

    private ConditionOutcome getMatchOutcome(
            ConditionAndSpec conditionAndSpec, AutoConfigurationMetadata metadata) {
        // 收集所有缺失的类
        List<String> missing = new ArrayList<>();
        for (String className : conditionAndSpec.getCondition().getRequiredClasses()) {
            if (!ClassUtils.isPresent(className, getClass().getClassLoader())) {
                missing.add(className);
            }
        }

        if (!missing.isEmpty()) {
            return ConditionOutcome.noMatch(
                "required class" + (missing.size() > 1 ? "s" : "") + " not present: " + missing);
        }

        return ConditionOutcome.match();
    }
}
```

**关键点：`ClassUtils.isPresent()`**

这个方法尝试通过类加载器加载指定类，如果 classpath 中不存在该类，会返回 false：

```java
public static boolean isPresent(String className, ClassLoader classLoader) {
    if (classLoader == null) {
        classLoader = ClassUtils.class.getClassLoader();
    }
    try {
        classLoader.loadClass(className);
        return true;
    } catch (ClassNotFoundException ex) {
        return false;
    }
}
```

## 四、@ConditionalOnMissingBean 详解

### 基本用法

```java
@Configuration
public class DataSourceAutoConfiguration {

    @Bean
    @ConditionalOnMissingBean
    public DataSource dataSource() {
        return new HikariDataSource();
    }
}
```

只有当容器中不存在 `DataSource` 类型的 Bean 时，才会创建这个 Bean。

### 防止覆盖用户配置

`@ConditionalOnMissingBean` 是 Spring Boot「用户自定义优先」原则的核心实现：

```java
@Configuration
public class WebMvcAutoConfiguration {

    @Bean
    @ConditionalOnMissingBean(name = "viewResolver")
    public InternalResourceViewResolver defaultViewResolver() {
        // 只有当用户没有自定义 viewResolver 时才生效
        InternalResourceViewResolver resolver = new InternalResourceViewResolver();
        resolver.setPrefix("/WEB-INF/views/");
        resolver.setSuffix(".jsp");
        return resolver;
    }
}
```

### 潜在问题

`@ConditionalOnMissingBean` 有一个常见的坑：**加载顺序问题**。

如果用户的 `@Bean` 定义先于自动配置类处理，`@ConditionalOnMissingBean` 会错误地认为 Bean 不存在：

```java
// 用户的配置类
@Configuration
public class MyConfig {
    @Bean
    public UserService userService() {
        return new UserService();
    }
}

// 自动配置类
@Configuration
public class DefaultAutoConfiguration {
    @Bean
    @ConditionalOnMissingBean
    public UserService defaultUserService() {
        // 如果先于 MyConfig 处理，这里会创建 Bean
        return new DefaultUserService();
    }
}
```

**Spring Boot 的解决方案：**

1. `@ComponentScan` 会先扫描并注册所有组件（包括用户配置类）
2. 然后才执行 `@EnableAutoConfiguration` 导入的自动配置类
3. 通过 `initializer` 机制确保正确的加载顺序

## 五、@ConditionalOnProperty 详解

### 基本用法

```java
@ConditionalOnProperty(
    prefix = "spring.datasource",
    name = "enabled",
    havingValue = "true",
    matchIfMissing = false
)
@Configuration
public class DataSourceAutoConfiguration {
    // ...
}
```

### 属性说明

| 属性 | 说明 |
|---|---|
| `prefix` | 配置属性的前缀 |
| `name` | 配置属性名（相对于 prefix） |
| `havingValue` | 期望的值 |
| `matchIfMissing` | 缺少属性时的默认行为 |

### 示例

```java
// 启用缓存
@ConditionalOnProperty(
    name = "app.cache.enabled",
    havingValue = "true",
    matchIfMissing = true  // 默认为 true
)
@Configuration
public class CacheAutoConfiguration {
    // ...
}
```

## 六、@ConditionalOnWebApplication 详解

### 基本用法

```java
@ConditionalOnWebApplication(type = Type.SERVLET)
@Configuration
public class ServletWebAutoConfiguration {
    // ...
}

@ConditionalOnWebApplication(type = Type.REACTIVE)
@Configuration
public class ReactiveWebAutoConfiguration {
    // ...
}
```

### 类型判断

```java
enum Type {
    SERVLET,       // 传统 Servlet 应用
    REACTIVE,      // 响应式 Web 应用
    ANY            // 任意 Web 应用
}
```

## 七、条件注解的组合使用

### @ConditionalOnBean 和 @ConditionalOnMissingBean 组合

```java
@Bean
@ConditionalOnBean(DataSource.class)
@ConditionalOnMissingBean(PlatformTransactionManager.class)
public DataSourceTransactionManager transactionManager(DataSource dataSource) {
    return new DataSourceTransactionManager(dataSource);
}
```

### 排除特定 Bean

```java
@Bean
@ConditionalOnMissingBean(type = "io.r2dbc.spi.ConnectionFactory")
@Configuration
public class DataSourceAutoConfiguration {
    // ...
}
```

### 条件注解在方法级别

条件注解也可以标注在 `@Bean` 方法上：

```java
@Configuration
public class CacheConfiguration {

    @Bean
    @ConditionalOnClass(name = "com.github.benmanes.caffeine.cache.Caffeine")
    public Cache<Object, Object> caffeineCache() {
        return Caffeine.newBuilder().build();
    }

    @Bean
    @ConditionalOnClass(name = "com.google.common.cache.Cache")
    @ConditionalOnMissingBean(name = "caffeineCache")
    public Cache<Object, Object> guavaCache() {
        return CacheBuilder.newBuilder().build();
    }
}
```

```java
        });
    }

```

## 九、面试高频追问

### Q1：@ConditionalOnClass 的性能问题？

`@ConditionalOnClass` 在每次启动时都会检查 classpath，理论上会有性能开销。但实际上这个开销很小，因为只是简单的类加载器查找。

Spring Boot 2.7 之后引入了 `@AutoConfiguration` 和 `AutoConfigurationMetadata`，可以通过预编译的元数据加速条件判断。

### Q2：@ConditionalOnMissingBean 的加载顺序？

Spring Boot 通过 `BeanFactoryPostProcessor` 的执行顺序确保自动配置类在用户配置类之后处理。具体来说：

1. `@ComponentScan` 先执行，注册所有用户配置类
2. 然后 `@EnableAutoConfiguration` 导入的自动配置类才执行
3. 这确保了 `userService` Bean 已经存在，`@ConditionalOnMissingBean` 会正确判断

### Q3：如何自定义条件注解？

可以继承 `SpringBootCondition` 实现自定义条件：

```java
@Conditional(MyCustomCondition.class)
@Retention(RetentionPolicy.RUNTIME)
public @interface ConditionalOnCustomCondition {
}

public class MyCustomCondition extends SpringBootCondition {
    @Override
    protected ConditionAndSpec getConditionAndSpec(
            ConditionContext context, AnnotatedTypeMetadata metadata) {
        // 自定义条件逻辑
        if (someCondition) {
            return ConditionOutcome.match();
        }
        return ConditionOutcome.noMatch("custom condition not met");
    }
}
```

## 十、总结

条件注解是 Spring Boot 自动配置的核心机制：

```
┌─────────────────────────────────────────────────────────────────┐
│                    条件注解工作原理                               │
├─────────────────────────────────────────────────────────────────┤
│  1. 加载配置类时，扫描 @Conditional 注解                          │
│                                                                  │
│  2. 获取条件信息（如需要哪些类、哪些 Bean）                       │
│                                                                  │
│  3. 执行条件判断                                                 │
│     ├── @ConditionalOnClass → ClassUtils.isPresent()           │
│     ├── @ConditionalOnBean → BeanFactory.containsBean()        │
│     └── @ConditionalOnProperty → Environment.getProperty()       │
│                                                                  │
│  4. 根据判断结果决定是否加载配置                                  │
│                                                                  │
│  5. 发布 AutoConfigurationImportEvent（可监听）                  │
└─────────────────────────────────────────────────────────────────┘
```

理解条件注解的底层实现，你就能理解 Spring Boot 自动配置的「智能」从何而来。下次遇到自动配置不生效的问题，你就能快速定位是哪个条件不满足。
