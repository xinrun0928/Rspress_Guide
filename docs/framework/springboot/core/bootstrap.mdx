# Bootstrap.yml 与 application.yml 的加载顺序

如果你使用 Spring Cloud，你会发现除了 `application.yml` 之外，还有一个 `bootstrap.yml` 文件。这两个文件有什么区别？加载顺序是怎样的？为什么 Spring Boot 2.4 之后不再支持 `bootstrap.yml`？

这一节，我们来深入理解 `bootstrap.yml` 的设计初衷和使用场景。

## 一、为什么需要 Bootstrap.yml？

### Bootstrap 上下文的设计

在 Spring Cloud 架构中，应用启动时需要首先从配置中心（如 Spring Cloud Config、Nacos、Apollo 等）获取配置。但问题是：**如何知道配置中心的地址？**

`Bootstrap.yml` 的设计就是为了解决这个问题：

1. **Bootstrap 上下文**先于主应用上下文创建
2. Bootstrap 上下文用于加载外部配置
3. 主应用上下文使用 Bootstrap 上下文中加载的配置

```
┌─────────────────────────────────────────────────────────────────┐
│                    Bootstrap 上下文加载流程                       │
├─────────────────────────────────────────────────────────────────┤
│  1. 创建 Bootstrap 上下文                                       │
│     └── 加载 bootstrap.yml / bootstrap.properties              │
│                                                                  │
│  2. 连接到配置中心                                               │
│     └── 从 Config Server 获取配置                                │
│                                                                  │
│  3. 创建主应用上下文                                           │
│     └── 使用 Bootstrap 上下文中加载的配置                       │
│     └── 加载 application.yml / application.properties           │
└─────────────────────────────────────────────────────────────────┘
```

### Bootstrap.yml 的典型用途

```yaml
# bootstrap.yml
spring:
  application:
    name: user-service
  cloud:
    config:
      uri: http://config-server:8888
      profile: dev
      fail-fast: true
```

这个配置告诉应用：配置中心在 `http://config-server:8888`，当前服务是 `user-service`，使用 `dev` profile。

## 二、Spring Boot 2.4 之前

### Bootstrap.yml 加载顺序

在 Spring Boot 2.4 之前，`bootstrap.yml` 的加载顺序如下：

| 顺序 | 配置文件 | 说明 |
|---|---|---|
| 1 | `bootstrap.yml` (classpath) | 最高优先级 |
| 2 | `bootstrap.properties` (classpath) | |
| 3 | `bootstrap.yml` (file) | |
| 4 | `bootstrap.properties` (file) | |
| 5 | `application.yml` (classpath) | |
| 6 | `application.properties` (file) | |

**重要**：`bootstrap.yml` 的优先级**高于** `application.yml`。这确保了配置中心的连接信息能够最先加载。

### 启用 Bootstrap

Spring Boot 2.4 之前，需要显式启用 `bootstrap.yml`：

```xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-bootstrap</artifactId>
</dependency>
```

## 三、Spring Boot 2.4 之后的变化

### 移除内置支持

Spring Boot 2.4 移除了对 `bootstrap.yml` 的内置支持。原因如下：

1. **配置加载机制改进**：Spring Boot 2.4 引入了 `spring.config.import`，可以导入外部配置文件
2. **简化架构**：减少配置文件数量，降低复杂度
3. **统一配置**：所有配置统一使用 `application.yml`

### 迁移方案

如果你的项目还在使用 `bootstrap.yml`，有两种迁移方案：

**方案一：合并到 application.yml**

```yaml
# application.yml
spring:
  application:
    name: user-service
  config:
    import: optional:file:./config/${spring.application.name}.yml

# 原 bootstrap.yml 中的配置中心配置
  cloud:
    config:
      uri: http://config-server:8888
      profile: dev
```

**方案二：使用 spring-cloud-starter-bootstrap**

```xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-bootstrap</artifactId>
</dependency>
```

这个依赖会恢复 `bootstrap.yml` 的加载行为。

## 四、spring.config.import 机制

Spring Boot 2.4 引入了 `spring.config.import` 作为配置扩展机制。

### 基本用法

```yaml
# application.yml
spring:
  config:
    import: optional:file:./config/${spring.application.name}.yml
```

### 支持的位置

```yaml
spring:
  config:
    import:
      # 本地文件
      - optional:file:./config/custom.yml
      # 类路径文件
      - optional:classpath:custom.yml
      # URL
      - optional:https://example.com/config.yml
      # Vault
      - optional:vault://secret/mysecret
      # Consul
      - optional:consul://localhost:8500/config
```

### 配置中心集成

对于 Spring Cloud Config：

```yaml
spring:
  config:
    import: optional:configserver:http://config-server:8888
```

对于 Nacos：

```yaml
spring:
  config:
    import: optional:nacos:${spring.application.name}?profile=${spring.profiles.active}
```

## 五、配置加载顺序（Spring Boot 2.4+）

Spring Boot 2.4 之后的完整配置加载顺序：

```
┌─────────────────────────────────────────────────────────────────┐
│              Spring Boot 2.4+ 配置加载顺序                        │
├─────────────────────────────────────────────────────────────────┤
│  1. spring.config.location（额外位置）                          │
│  2. spring.config.import（导入的配置）                          │
│  3. classpath:application.yml                                  │
│  4. classpath:application.properties                           │
│  5. classpath:config/application.yml                           │
│  6. classpath:config/application.properties                     │
│  7. file:./application.yml                                      │
│  8. file:./application.properties                              │
│  9. file:./config/application.yml                              │
│  10. file:./config/application.properties                       │
│  11. 环境变量                                                   │
│  12. 命令行参数                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**注意**：`bootstrap.yml` 不再是默认加载的文件。

## 六、Profile 与 spring.config.import

### Profile 特定配置

```yaml
# application-dev.yml
spring:
  config:
    import: optional:file:./config/config-server-dev.yml
```

### 多配置导入

```yaml
spring:
  config:
    import:
      - optional:file:./config/${spring.application.name}-common.yml
      - optional:file:./config/${spring.application.name}-${spring.profiles.active}.yml
```

## 七、常见问题排查

### 问题 1：bootstrap.yml 不生效

**原因**：Spring Boot 2.4 移除了内置支持。

**解决方案**：

```xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-bootstrap</artifactId>
</dependency>
```

### 问题 2：配置中心连接失败

**原因**：`spring.cloud.config.uri` 配置错误或 Config Server 不可用。

**排查步骤**：

1. 检查 `bootstrap.yml` 或 `application.yml` 中的配置
2. 确认 Config Server 地址是否正确
3. 检查网络连通性
4. 确认 Config Server 是否正常运行

### 问题 3：配置覆盖问题

**场景**：同时存在 `bootstrap.yml` 和 `application.yml`，同名属性哪个生效？

**答案**：取决于加载顺序。在 Spring Boot 2.4 之前，`bootstrap.yml` 优先生效；在 Spring Boot 2.4 之后，如果使用 `spring.config.import`，导入的配置优先生效。

## 八、面试高频追问

### Q1：为什么 Spring Boot 2.4 移除 bootstrap.yml？

主要原因是简化架构。`bootstrap.yml` 的核心功能是「在主应用上下文创建之前加载配置」，这个功能可以通过 `spring.config.import` 更优雅地实现。

### Q2：spring.config.import 的 optional 是什么含义？

`optional:file:./config.yml` 表示如果配置文件不存在，应用不会报错，继续启动。如果去掉 `optional`，配置文件不存在时会抛出异常。

### Q3：如何在 Spring Boot 3.x 中使用 Spring Cloud Config？

```yaml
# application.yml
spring:
  application:
    name: user-service
  cloud:
    config:
      uri: http://config-server:8888
      profile: dev
```

需要确保使用了兼容 Spring Boot 3.x 的 Spring Cloud 版本。

## 九、总结

`bootstrap.yml` 和 `application.yml` 的关系：

```
┌─────────────────────────────────────────────────────────────────┐
│              bootstrap.yml vs application.yml                   │
├─────────────────────────────────────────────────────────────────┤
│  bootstrap.yml（Spring Boot 2.4 之前）                          │
│  ├── 用途：加载配置中心地址                                     │
│  ├── 时机：主应用上下文创建之前                                 │
│  ├── 优先级：高于 application.yml                               │
│  └── 状态：Spring Boot 2.4 移除内置支持                         │
│                                                                  │
│  application.yml                                                │
│  ├── 用途：所有应用配置                                        │
│  ├── 时机：Bootstrap 上下文创建之后                            │
│  └── 状态：Spring Boot 3.x 唯一标准                            │
│                                                                  │
│  spring.config.import（Spring Boot 2.4+）                      │
│  ├── 用途：替代 bootstrap.yml 的部分功能                        │
│  ├── 支持：本地文件、URL、配置中心                              │
│  └── 状态：Spring Boot 3.x 推荐方式                             │
└─────────────────────────────────────────────────────────────────┘
```

理解配置加载顺序，是解决配置相关问题的关键。在实际项目中，建议：

1. **Spring Boot 2.4+**：使用 `application.yml` + `spring.config.import`
2. **Spring Cloud 项目**：使用 `spring-cloud-starter-bootstrap` 或迁移到 `spring.config.import`
