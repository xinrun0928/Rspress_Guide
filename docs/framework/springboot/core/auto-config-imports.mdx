# spring.factories vs AutoConfiguration.imports

如果你的项目从 Spring Boot 2.7.x 升级到 3.x，可能会遇到一个问题：自动配置不生效了。错误日志显示 `No auto-configured beans` 或类似的信息。

这很可能是因为你使用的 Starter 还在使用旧的 `spring.factories` 格式，而 Spring Boot 3.x 已经不再支持它。

## 一、两种配置方式的历史演变

### Spring Boot 1.5.x ~ 2.6.x：spring.factories

`META-INF/spring.factories` 是 Spring Boot 早期的自动配置注册方式：

```
META-INF/
└── spring.factories
```

文件格式是 Properties 格式：

```properties
# META-INF/spring.factories
org.springframework.boot.autoconfigure.EnableAutoConfiguration=\
com.example.autoconfig.MyAutoConfiguration
```

### Spring Boot 2.7.x ~ 3.x：AutoConfiguration.imports

Spring Boot 2.7 引入 `AutoConfiguration.imports` 作为 `spring.factories` 的替代方案：

```
META-INF/spring/
└── org.springframework.boot.autoconfigure.AutoConfiguration.imports
```

文件格式是每行一个类名：

```text
# META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports
com.example.autoconfig.MyAutoConfiguration
```

### Spring Boot 3.x：完全迁移

Spring Boot 3.x（对应 Spring Framework 6.x）**完全移除了对 `spring.factories` 的支持**，只支持 `AutoConfiguration.imports`。

## 二、核心区别对比

| 对比项 | spring.factories | AutoConfiguration.imports |
|---|---|---|
| 文件位置 | `META-INF/spring.factories` | `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` |
| 文件格式 | Properties 格式 | 每行一个类名 |
| 首次引入 | Spring Boot 1.5.x | Spring Boot 2.7.x |
| JDK 兼容性 | JDK 8 ~ 17 | JDK 9+（推荐 JDK 17+） |
| Spring Boot 3.x | 不支持 | 支持 |
| IDE 支持 | 一般 | 更好 |

### 格式对比

**spring.factories 格式：**

```properties
# 多个类可以用逗号分隔或分行（反斜杠续行）
org.springframework.boot.autoconfigure.EnableAutoConfiguration=\
com.example.autoconfig.ConfigA,\
com.example.autoconfig.ConfigB
```

**AutoConfiguration.imports 格式：**

```text
# 每行一个类名
com.example.autoconfig.ConfigA
com.example.autoconfig.ConfigB
```

`AutoConfiguration.imports` 的格式更简洁，消除了续行符的困扰。

## 三、迁移指南

### 从 spring.factories 迁移到 AutoConfiguration.imports

**Step 1：创建新文件**

在 `META-INF/spring/` 目录下创建 `org.springframework.boot.autoconfigure.AutoConfiguration.imports` 文件。

**Step 2：迁移内容**

将 `spring.factories` 中的 `EnableAutoConfiguration` 条目迁移到新文件：

```properties
# spring.factories（旧）
org.springframework.boot.autoconfigure.EnableAutoConfiguration=\
com.example.autoconfig.ConfigA,\
com.example.autoconfig.ConfigB
```

```text
# AutoConfiguration.imports（新）
com.example.autoconfig.ConfigA
com.example.autoconfig.ConfigB
```

**Step 3：删除旧文件**

迁移完成后，删除 `spring.factories` 文件（或保留但 Spring Boot 3.x 会忽略）。

### 同时兼容两种格式

如果你的库需要兼容 Spring Boot 2.7.x 和 3.x，可以同时保留两个文件：

```
META-INF/
├── spring.factories                    # Spring Boot 2.7.x 读取
└── spring/
    └── org.springframework.boot.autoconfigure.AutoConfiguration.imports  # Spring Boot 3.x 读取
```

Spring Boot 会根据版本自动选择读取哪个文件。

## 四、Spring Boot 2.7.x 的变化

Spring Boot 2.7.x 是一个重要的过渡版本，引入了以下变化：

### 1. 新增自动配置导入机制

```xml
<!-- Spring Boot 2.7.x -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-autoconfigure</artifactId>
</dependency>
```

### 2. @AutoConfiguration 注解

Spring Boot 2.7.x 引入了 `@AutoConfiguration` 注解，作为 `@Configuration` 的替代：

```java
@AutoConfiguration
@ConditionalOnClass(ExampleService.class)
public class ExampleAutoConfiguration {
    // ...
}
```

`@AutoConfiguration` 默认 `proxyBeanMethods = false`，更适合自动配置场景。

### 3. auto-configure 专用依赖

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-autoconfigure-processor</artifactId>
    <optional>true</optional>
</dependency>
```

这个处理器会生成 `spring-autoconfigure-metadata.properties` 文件，用于加速条件判断。

## 五、AutoConfiguration.imports 的优势

### 1. 消除解析复杂性

`spring.factories` 的 Properties 格式需要处理：
- 逗号分隔的类名
- 反斜杠续行
- 多值展开

`AutoConfiguration.imports` 每行一个类名，解析更简单：

```java
// Spring Boot 内部解析逻辑
try (BufferedReader reader = new BufferedReader(new InputStreamReader(inputStream))) {
    String line;
    while ((line = reader.readLine()) != null) {
        line = line.trim();
        if (!line.isEmpty() && !line.startsWith("#")) {
            configurations.add(line);
        }
    }
}
```

### 2. 更好的 IDE 支持

`spring.factories` 是全局配置文件，IDE 难以解析具体的 key。`AutoConfiguration.imports` 每个自动配置模块独立，IDE 可以更好地跟踪。

### 3. 支持分层配置

`AutoConfiguration.imports` 支持 `spring.config.on-profile` 前缀，用于指定配置类在特定 profile 下生效：

```text
# 只在 dev profile 下生效
@ifdev
com.example.autoconfig.DevAutoConfiguration
@endifdev
```

### 4. 支持自动配置排序

可以在 `AutoConfiguration.imports` 中使用 `#` 注释实现分组：

```text
# Default auto-configuration
com.example.autoconfig.ConfigA
com.example.autoconfig.ConfigB

# Additional auto-configuration
com.example.autoconfig.AdditionalConfig
```

## 六、常见问题

### Q1：spring-boot-starter 的配置放在哪里？

`spring-boot-starter` 的 auto-configure 模块应该包含自动配置类：

```
xxx-autoconfigure/
├── src/main/java/
│   └── com/example/autoconfigure/
│       └── XxxAutoConfiguration.java
└── src/main/resources/
    └── META-INF/
        └── spring/
            └── org.springframework.boot.autoconfigure.AutoConfiguration.imports
```

### Q2：自动配置类需要 @AutoConfigureAfter 吗？

如果自动配置类依赖其他自动配置类，需要使用 `@AutoConfigureAfter`：

```java
@AutoConfiguration
@AutoConfigureAfter(DataSourceAutoConfiguration.class)
public class XxxAutoConfiguration {
    // ...
}
```

### Q3：Spring Boot 3.x 是否完全移除 spring.factories 支持？

是的。Spring Boot 3.x 的 `SpringFactoriesLoader` 不再读取 `spring.factories` 中的 `EnableAutoConfiguration` 条目。如果你的 Starter 还在使用 `spring.factories`，需要迁移到 `AutoConfiguration.imports`。

## 七、面试高频追问

### Q1：为什么 Spring Boot 要废弃 spring.factories？

1. **性能**：`spring.factories` 是一个全局文件，Spring Boot 需要扫描所有 JAR 包查找。`AutoConfiguration.imports` 按模块分离，扫描更快
2. **可维护性**：`spring.factories` 容易冲突，多个模块可能定义相同的 key
3. **简洁性**：`AutoConfiguration.imports` 格式更简洁，减少了解析错误

### Q2：迁移时需要注意什么？

1. **清理旧配置**：迁移完成后删除 `spring.factories`，避免混淆
2. **检查第三方依赖**：确保所有依赖的 Starter 都已迁移到 `AutoConfiguration.imports`
3. **测试**：在升级后运行 `--debug` 模式，检查自动配置报告

### Q3：spring.factories 还有其他用途吗？

`spring.factories` 不仅仅用于自动配置，还用于：

```properties
# ApplicationContextInitializer
org.springframework.context.ApplicationContextInitializer=\
com.example.init.MyInitializer

# ApplicationListener
org.springframework.context.ApplicationListener=\
com.example.listener.MyListener

# EnvironmentPostProcessor
org.springframework.boot.env.EnvironmentPostProcessor=\
com.example.env.MyEnvironmentPostProcessor
```

这些场景仍然使用 `spring.factories`，且在 Spring Boot 3.x 中仍然支持。

## 八、总结

两种配置方式的对比：

```
┌─────────────────────────────────────────────────────────────────┐
│                    配置方式对比                                   │
├─────────────────────────────────────────────────────────────────┤
│  spring.factories                                               │
│  ├── 格式：Properties                                           │
│  ├── 位置：META-INF/spring.factories                            │
│  ├── 问题：全局冲突、解析复杂                                    │
│  └── 状态：Spring Boot 3.x 不再支持                             │
│                                                                  │
│  AutoConfiguration.imports                                       │
│  ├── 格式：每行一个类名                                         │
│  ├── 位置：META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports │
│  ├── 优势：模块分离、解析简单、IDE 友好                          │
│  └── 状态：Spring Boot 2.7.x+ 推荐，3.x 必须                    │
└─────────────────────────────────────────────────────────────────┘
```

迁移到 `AutoConfiguration.imports` 是 Spring Boot 3.x 升级的必经之路。理解两种配置方式的差异，有助于你在遇到兼容性问题时快速定位和解决。
