# Spring Boot 启动流程源码解析

凌晨 2 点，你正准备上线新功能，Spring Boot 应用突然启动失败。错误日志显示 `BeanDefinitionStoreException`。你开始怀疑是自动配置出了问题，但不知道从哪查起。

要解决这类问题，你必须理解 `SpringApplication.run()` 的完整执行流程。**只有知道每一步做了什么，才能在出问题的时候准确定位。**

## 一、整体流程概览

`SpringApplication.run()` 的执行流程可以分为两大部分：

1. **创建 SpringApplication 实例**：初始化各种组件
2. **执行 run() 方法**：创建并刷新容器

```
┌─────────────────────────────────────────────────────────────┐
│                    SpringApplication.run()                  │
├─────────────────────────────────────────────────────────────┤
│  1. 创建 SpringApplication 实例                             │
│     ├── 判断应用类型（Servlet / Reactive / None）           │
│     ├── 设置 bootstrapRegistryInitializers                  │
│     ├── 初始化 initializers（ApplicationContextInitializer） │
│     ├── 初始化 listeners（ApplicationListener）             │
│     └── 推断主配置类                                        │
├─────────────────────────────────────────────────────────────┤
│  2. 执行 run() 方法                                          │
│     ├── 创建 BootstrapContext                                │
│     ├── 构造 SpringApplicationRunListeners                  │
│     ├── 调用 starting() 事件                                 │
│     ├── 准备 ConfigurableEnvironment                        │
│     ├── 打印 Banner                                         │
│     ├── 创建 ConfigurableApplicationContext                 │
│     ├── 准备容器（applyInitializers）                        │
│     ├── 刷新容器（context.refresh()）                        │
│     └── 执行 runners（ApplicationRunner / CommandLineRunner）│
└─────────────────────────────────────────────────────────────┘
```

## 二、创建 SpringApplication 实例

### 1. 判断应用类型

Spring Boot 根据 classpath 中的类判断应用类型：

```java
// SpringApplication.java
private WebApplicationType deduceWebApplicationType() {
    if (ClassUtils.isPresent("org.springframework.web.reactive.DispatcherHandler", ...)
            && !ClassUtils.isPresent("org.springframework.web.servlet.DispatcherServlet", ...)) {
        return WebApplicationType.REACTIVE;
    }
    for (String className : SERVLET_INDICATIVE_CLASSES) {
        if (!ClassUtils.isPresent(className, null)) {
            return WebApplicationType.NONE;
        }
    }
    return WebApplicationType.SERVLET;
}
```

| 类型 | 条件 | 说明 |
|---|---|---|
| `SERVLET` | 存在 `spring-webmvc` 相关类 | 传统 Servlet 应用 |
| `REACTIVE` | 存在 `spring-webflux` 但不存在 `spring-webmvc` | 响应式 Web 应用 |
| `NONE` | 不存在上述任何类 | 非 Web 应用（如 CLI 工具） |

### 2. 初始化 initializers 和 listeners

```java
// 初始化 ApplicationContextInitializer
this.initializers = getSpringFactoriesInstances(ApplicationContextInitializer.class);

// 初始化 ApplicationListener
this.listeners = getSpringFactoriesInstances(ApplicationListener.class);
```

这两个方法都通过 `SpringFactoriesLoader` 加载 `META-INF/spring.factories` 中配置的类。

**ApplicationContextInitializer 的作用是什么？**

`ApplicationContextInitializer` 是在容器刷新之前调用的回调接口，常用于：

- 设置容器环境
- 注册临时 Bean
- 修改 Bean 定义

Spring Boot 内部的 `ContextIdApplicationContextInitializer`、`CloudFoundryVcapEnvironmentPostProcessor` 等都是这个接口的实现。

**ApplicationListener 的作用是什么？**

`ApplicationListener` 是 Spring 的事件监听机制。Spring Boot 在启动过程中的各个阶段会发布不同的事件：

| 事件 | 触发时机 |
|---|---|
| `ApplicationStartingEvent` | 应用启动，run() 方法刚执行 |
| `ApplicationEnvironmentPreparedEvent` | Environment 准备完成 |
| `ApplicationPreparedEvent` | 容器准备完成，Bean 定义加载完成 |
| `ContextRefreshedEvent` | 容器刷新完成 |
| `ApplicationStartedEvent` | 应用启动完成 |
| `ApplicationReadyEvent` | 应用就绪，可以接收请求 |

### 3. 推断主配置类

Spring Boot 会根据堆栈信息推断包含 `@SpringBootApplication` 的主类：

```java
private Class<?> deduceMainApplicationClass() {
    StackTraceElement[] stackTrace = new RuntimeException().getStackTrace();
    for (StackTraceElement stackTraceElement : stackTrace) {
        if ("main".equals(stackTraceElement.getMethodName())) {
            return ClassUtils.forName(stackTraceElement.getClassName());
        }
    }
    return null;
}
```

这个推断过程是为了后续创建 `ApplicationContext` 和设置 Banner。

## 三、执行 run() 方法

### 1. 创建 BootstrapContext

```java
private BootstrapContext createBootstrapContext() {
    DefaultBootstrapContext bootstrapContext = new DefaultBootstrapContext();
    this.bootstrapRegistryInitializers.forEach(
        initializer -> initializer.initialize(bootstrapContext)
    );
    return bootstrapContext;
}
```

`BootstrapContext` 是 Spring Boot 2.4 引入的，主要用于在主应用上下文启动之前，临时注册一些 Bean。在 Spring Cloud 场景下，`BootstrapContext` 用于加载 `bootstrap.yml` 配置。

### 2. 准备 ConfigurableEnvironment

```java
private ConfigurableEnvironment prepareEnvironment(
        SpringApplicationRunListeners listeners,
        ApplicationArguments applicationArguments) {
    // 创建并配置 Environment
    ConfigurableEnvironment environment = getOrCreateEnvironment(...);
    configureEnvironment(environment, applicationArguments.getSourceArgs());

    // 触发事件
    listeners.environmentPrepared(bootstrapContext, environment);

    // 绑定 SpringApplication 到 Environment
    bindToSpringApplication(environment);

    // 转换配置
    if (!this.activeProfiles.isEmpty()) {
        activeProfiles.forEach(environment::addActiveProfile);
    }

    // 加载额外配置源
    configureAdditionalProfiles(environment);

    // 动态处理配置（如处理 ${random.*}）
    ConfigurationPropertySources.attach(environment);

    return environment;
}
```

**这一步做了什么？**

1. 根据应用类型创建不同的 `Environment`（`StandardServletEnvironment`、`StandardReactiveWebEnvironment` 等）
2. 加载 `application.yml` 或 `application.properties` 配置
3. 处理命令行参数
4. 发布 `ApplicationEnvironmentPreparedEvent` 事件
5. 绑定随机值 `${random.*}`

### 3. 打印 Banner

```java
private Banner printBanner(ConfigurableEnvironment environment) {
    if (this.bannerMode == Banner.Mode.OFF) {
        return null;
    }

    ResourceLoader resourceLoader = (this.resourceLoader != null) ?
        this.resourceLoader : new DefaultResourceLoader();
    SpringApplicationBannerPrinter bannerPrinter = new SpringApplicationBannerPrinter(
        environment, resourceLoader);

    if (this.bannerMode == Banner.Mode.LOG) {
        return bannerPrinter.printToLog();
    }
    return bannerPrinter.printToTTY();
}
```

Banner 支持三种模式：

| 模式 | 说明 |
|---|---|
| `CONSOLE` | 打印到控制台（默认） |
| `LOG` | 打印到日志 |
| `OFF` | 禁用 Banner |

Banner 的加载顺序：

1. `classpath:banner.txt`（文本 Banner）
2. `classpath:banner.gif/png/jpg`（图片 Banner）
3. `spring.banner.image.location` 指定的位置

### 4. 创建 ApplicationContext

```java
private ConfigurableApplicationContext createApplicationContext() {
    ApplicationContextFactory factory = ApplicationContextFactory.DEFAULT;
    ConfigurableApplicationContext context = factory.create(this.webApplicationType);
    // ...
    return context;
}
```

根据应用类型创建不同的容器：

| 应用类型 | 容器类型 |
|---|---|
| `SERVLET` | `AnnotationConfigServletWebServerApplicationContext` |
| `REACTIVE` | `AnnotationConfigReactiveWebServerApplicationContext` |
| `NONE` | `AnnotationConfigApplicationContext` |

### 5. 准备容器（prepareContext）

```java
private void prepareContext(
        ConfigurableApplicationContext context,
        ConfigurableEnvironment environment,
        SpringApplicationRunListeners listeners,
        ResourceLoader resourceLoader,
        ConfigurableListableBeanFactory beanFactory) {
    // 设置 Environment
    context.setEnvironment(environment);

    // 后置处理（设置 resourceLoader、classLoader 等）
    postProcessApplicationContext(context);

    // 应用 initializers
    applyInitializers(context);

    // 发布 ApplicationPreparedEvent
    listeners.contextPrepared(context);

    // 注册 springApplicationArguments 和 springBootBanner
    beanFactory.registerSingleton("springApplicationArguments", applicationArguments);
    if (banner != null) {
        beanFactory.registerSingleton("springBootBanner", banner);
    }

    // 加载 Bean 定义（主配置类）
    if (beanFactory instanceof DefaultListableBeanFactory) {
        beanFactory.setAllowBeanDefinitionOverriding(this.allowBeanDefinitionOverriding);
    }
    Set<Object> sources = getSources();
    load(context, sources.toArray(new Object[0]));

    // 发布 ApplicationPreparedEvent
    listeners.contextLoaded(context);
}
```

**关键点：加载 Bean 定义**

`load()` 方法会解析主配置类及其 `@ComponentScan` 扫描到的类，将它们转换为 `BeanDefinition` 并注册到容器中。

```java
protected void load(ApplicationContext context, Object[] sources) {
    BeanDefinitionLoader loader = createBeanDefinitionLoader(
        context, getSources());
    loader.load();
}
```

### 6. 刷新容器（refreshContext）

这是整个启动流程中最关键的一步：

```java
private void refreshContext(ConfigurableApplicationContext context) {
    // Spring Boot 2.4 之前：直接调用 refresh()
    // Spring Boot 2.4 之后：通过 StartupStep API 记录启动步骤
    if (this.activeProfiles.isEmpty()) {
        Set<String> profiles = Arrays.asList(ActivatablePropertyProvider
            .findProfiles(environment.getPropertySources()));
        // ...
    }
    refresh(context);
}

@Override
public void refresh() throws BeansException {
    // 1. 准备 BeanFactory
    prepareBeanFactory(beanFactory);

    // 2. 后置处理 BeanFactory
    postProcessBeanFactory(beanFactory);

    // 3. 执行 BeanFactoryPostProcessor
    invokeBeanFactoryPostProcessors(beanFactory);

    // 4. 注册 BeanPostProcessor
    registerBeanPostProcessors(beanFactory);

    // 5. 初始化 MessageSource
    initMessageSource();

    // 6. 初始化事件广播器
    initApplicationEventMulticaster();

    // 7. 初始化其他特殊 Bean（Web 服务器等）
    onRefresh();

    // 8. 注册监听器
    registerListeners();

    // 9. 实例化所有非懒加载的单例 Bean
    finishBeanFactoryInitialization(beanFactory);

    // 10. 刷新完成
    finishRefresh();
}
```

**refresh() 中的关键步骤：**

1. **invokeBeanFactoryPostProcessors()**：执行 `BeanFactoryPostProcessor`，包括 `ConfigurationClassPostProcessor`（解析 `@Configuration` 类）
2. **onRefresh()**：对于 Servlet 应用，这一步会创建并启动嵌入式服务器（Tomcat/Jetty/Undertow）
3. **finishBeanFactoryInitialization()**：实例化所有非懒加载的单例 Bean，包括 `@Component`、`@Service`、`@Repository`、`@Controller` 等注解的类

### 7. 执行 runners

```java
private void callRunners(ApplicationContext context, ApplicationArguments args) {
    List<Object> runners = new ArrayList<>();
    runners.addAll(context.getBeansOfType(ApplicationRunner.class).values());
    runners.addAll(context.getBeansOfType(CommandLineRunner.class).values());
    AnnotationAwareOrderComparator.sort(runners);

    for (Object runner : runners) {
        if (runner instanceof ApplicationRunner) {
            ((ApplicationRunner) runner).run(args);
        } else {
            ((CommandLineRunner) runner).run(args.getSourceArgs());
        }
    }
}
```

`ApplicationRunner` 和 `CommandLineRunner` 都是在容器刷新完成后执行的回调，常用于：

- 应用启动后执行初始化任务
- 预热缓存
- 打印启动信息

## 四、常见问题排查

### 1. 启动失败：BeanDefinitionStoreException

这种情况通常是 Bean 定义加载出错，可能的原因：

- 主配置类没有正确指定
- `@ComponentScan` 扫描路径配置错误
- 存在循环依赖

### 2. 启动失败：NoSuchBeanDefinitionException

这种情况通常是自动配置未生效，可能的原因：

- `@ConditionalOnClass` 条件不满足
- `@ConditionalOnMissingBean` 被用户自定义 Bean 覆盖
- 自动配置类被排除

### 3. 启动成功但请求报错：404

这种情况通常是：

- `@RequestMapping` 路径配置错误
- Controller 未被扫描到
- 包扫描路径问题（默认扫描主类所在包及其子包）

## 五、面试追问

### Q1：SpringApplication 和 SpringApplicationBuilder 的区别？

`SpringApplication` 是传统的编程式启动方式，`SpringApplicationBuilder` 提供了流式 API（Fluent API）：

```java
// SpringApplication
SpringApplication.run(App.class, args);

// SpringApplicationBuilder
new SpringApplicationBuilder(App.class)
    .bannerMode(Banner.Mode.OFF)
    .run(args);
```

`SpringApplicationBuilder` 允许配置多个源（sources），适用于微服务架构。

### Q2：为什么 Spring Boot 2.4 之后不再支持 bootstrap.yml？

Spring Boot 2.4 改进了配置加载机制，`bootstrap.yml` 的功能被整合到 `application.yml` 中。Spring Cloud 需要使用 `spring-cloud-starter-bootstrap` 依赖才能继续使用 `bootstrap.yml`。

### Q3：refresh() 抛出异常后，容器状态是怎样的？

Spring Boot 在 `refresh()` 失败后会尝试销毁已创建的 Bean 和关闭容器：

```java
try {
    refresh(context);
} catch (Throwable ex) {
    stopWatch.stop();
    // 停止容器
    destroy(context);
    throw ex;
}
```

## 总结

`SpringApplication.run()` 的执行流程可以简化为：

1. **创建实例**：初始化 initializers、listeners，推断主类
2. **准备环境**：加载配置，发布事件
3. **创建容器**：根据应用类型创建不同的容器
4. **刷新容器**：执行 BeanFactoryPostProcessor、实例化 Bean、启动服务器
5. **执行回调**：调用 ApplicationRunner 和 CommandLineRunner

理解了这个流程，你就掌握了 Spring Boot 启动的每一个细节。下次遇到启动问题，你就能快速定位问题所在。
