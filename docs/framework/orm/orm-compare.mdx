# JPA/Hibernate vs MyBatis：两条 ORM 道路的哲学

<Badge text="ORM" type="tip" /> <Badge text="框架对比" type="info" />

同样是 ORM，Hibernate 和 MyBatis 的设计理念完全不同。

这是两种哲学的碰撞：一方追求"零 SQL"，另一方坚持"掌控 SQL"。理解它们的差异，才能做出正确的技术选型。

## 设计理念对比

### Hibernate / JPA：配置即映射

```
┌─────────────────────────────────────────────────────────────┐
│                    Hibernate / JPA                           │
│                                                             │
│  开发者写对象                                                │
│       ↓                                                     │
│  框架自动生成 SQL                                            │
│       ↓                                                     │
│  执行 SQL，返回对象                                           │
│                                                             │
│  "开发者只关心业务，SQL 交给框架"                              │
└─────────────────────────────────────────────────────────────┘
```

**核心理念**：
- 用面向对象的方式操作数据库
- 不需要手写 SQL
- 配置即映射

### MyBatis：SQL + 映射

```
┌─────────────────────────────────────────────────────────────┐
│                         MyBatis                             │
│                                                             │
│  开发者手写 SQL                                              │
│       ↓                                                     │
│  配置对象映射规则                                             │
│       ↓                                                     │
│  执行 SQL，返回对象                                           │
│                                                             │
│  "SQL 应该由开发者掌控"                                       │
└─────────────────────────────────────────────────────────────┘
```

**核心理念**：
- SQL 完全可控
- 映射规则可配置
- SQL 和代码分离

## 实际代码对比

### 查询操作

#### JPA 方式

```java
// 实体类
@Entity
@Table(name = "user")
public class User {
    @Id
    @GeneratedValue(strategy = GenerationType.AUTO)
    private Long id;

    @Column(name = "user_name")
    private String name;

    private String email;
}

// Repository
public interface UserRepository extends JpaRepository<User, Long> {
}

// 查询
Optional<User> user = userRepository.findById(1L);
List<User> users = userRepository.findByNameContaining("张");
List<User> users = userRepository.findByNameAndEmail(String name, String email);
```

#### MyBatis 方式

```java
// 实体类（普通 POJO）
public class User {
    private Long id;
    private String name;
    private String email;
}

// Mapper
public interface UserMapper extends BaseMapper<User> {
    User selectById(Long id);
}

// XML
<select id="selectById" resultType="User">
    SELECT * FROM user WHERE id = #{id}
</select>

// 查询
User user = userMapper.selectById(1L);
```

### 复杂查询

#### JPA 方式（JPQL）

```java
// 使用 JPQL
@Query("SELECT u FROM User u WHERE u.name LIKE %:name%")
List<User> searchByName(@Param("name") String name);

// 使用 Criteria API
CriteriaBuilder cb = entityManager.getCriteriaBuilder();
CriteriaQuery<User> query = cb.createQuery(User.class);
Root<User> root = query.from(User.class);
query.where(cb.like(root.get("name"), "%" + name + "%"));
List<User> users = entityManager.createQuery(query).getResultList();
```

#### MyBatis 方式（原生 SQL）

```xml
<select id="searchByName" resultType="User">
    SELECT * FROM user
    WHERE name LIKE CONCAT('%', #{name}, '%')
</select>
```

### 关联查询

#### JPA 方式

```java
@Entity
public class User {
    @Id
    private Long id;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "dept_id")
    private Department department;
}

// 访问时自动加载
User user = userRepository.findById(1L).get();
String deptName = user.getDepartment().getName();  // 自动查询部门
```

#### MyBatis 方式

```xml
<resultMap id="userResultMap" type="User">
    <id property="id" column="id"/>
    <result property="name" column="name"/>
    <association property="department" javaType="Department">
        <id property="id" column="dept_id"/>
        <result property="name" column="dept_name"/>
    </association>
</resultMap>

<select id="selectUserWithDept" resultMap="userResultMap">
    SELECT u.id, u.name, d.id AS dept_id, d.name AS dept_name
    FROM user u
    LEFT JOIN department d ON u.dept_id = d.id
    WHERE u.id = #{id}
</select>
```

```
陡峭

  │          ***
  │       ***
  │    ***

需要学习：

- 关联映射（@OneToMany、@ManyToOne）

- 事务传播
```

### MyBatis 学习曲线

```
平缓
  ↑
  │   *****
  │  ***
  │ **
  │*
  └──────────────────────────→ 时间
    入门   深入   精通

需要学习：
- XML 配置（简单的）
- SQL 标签（9 个动态 SQL 标签）
- resultMap 映射
- 类型处理器
- 缓存配置
```

## 灵活性对比

### SQL 控制

| 场景 | JPA | MyBatis |
|------|-----|---------|
| 简单 CRUD | 方便 | 方便 |
| 动态 SQL | 困难 | 方便 |
| 复杂 JOIN | 一般 | 方便 |
| 存储过程 | 支持但不推荐 | 支持 |
| SQL 优化 | 受限 | 完全掌控 |

### 映射控制

| 场景 | JPA | MyBatis |
|------|-----|---------|
| 字段映射 | 自动 + 注解 | XML 或注解 |
| 继承映射 | 支持 | 不直接支持 |
| 多态映射 | 支持 | 需要鉴别器 |
| 嵌套结果 | 一般 | 强大 |

## 性能对比

### 缓存策略

| 缓存 | JPA/Hibernate | MyBatis |
|------|---------------|---------|
| 一级缓存 | Session 级别，自动 | SqlSession 级别，需配置 |
| 二级缓存 | 可配置，复杂 | 可配置，简单 |
| 查询缓存 | 支持 | 支持 |

### SQL 执行

| 方面 | JPA | MyBatis |
|------|-----|---------|
| SQL 生成 | 自动，可能不优化 | 完全可控 |
| N+1 问题 | 存在，需配置 | 存在，需注意 |
| 批量操作 | 支持，但不推荐 | 支持，推荐使用 |
| 预编译 | 自动 | 自动 |

### 性能调优

```java
// JPA：需要深入理解内部机制
@Entity
public class User {
    @OneToMany(mappedBy = "user", fetch = FetchType.LAZY)
    private List<Order> orders;
}

// MyBatis：直接优化 SQL
<select id="selectUserWithOrders" resultMap="userResultMap">
    SELECT u.*, o.* FROM user u
    LEFT JOIN order o ON u.id = o.user_id
</select>
```

## 维护性对比

### 表结构变更

```java
// JPA：修改实体类
@Entity
public class User {
    private String newField;  // 新增字段
    private String oldField;   // 要删除的字段
}

// Hibernate 会同步更新数据库（需要配置）
```

```xml
<!-- MyBatis：修改 SQL 和 resultMap -->
<select id="selectById" resultType="User">
    SELECT id, name, new_field AS newField FROM user WHERE id = #{id}
</select>
```

### SQL 调试

| 方面 | JPA | MyBatis |
|------|-----|---------|
| SQL 可见性 | 不直观，需要开启日志 | 直接可见 |
| 调试难度 | 高 | 低 |
| SQL 优化 | 困难 | 容易 |

## 适用场景总结

### 适合使用 JPA/Hibernate 的场景

| 场景 | 原因 |
|------|------|
| 业务简单 | 不需要复杂 SQL |
| 快速开发 | 不需要写 SQL |
| 团队 SQL 能力弱 | 框架自动生成 SQL |
| OO 优先 | 重视面向对象设计 |

### 适合使用 MyBatis 的场景

| 场景 | 原因 |
|------|------|
| 业务复杂 | 需要精细 SQL 控制 |
| 性能要求高 | 需要 SQL 调优 |
| 团队 SQL 能力强 | 掌控 SQL |
| 多表复杂查询 | SQL 更直观 |
| SQL 频繁变更 | XML 便于修改 |

## 混合使用策略

### 合理分工

```java
// 简单 CRUD → MyBatis Plus BaseMapper
userMapper.selectById(1L);

    <!-- 复杂的 JOIN 和聚合 -->
```

## 总结对比

| 维度 | JPA/Hibernate | MyBatis |
|------|---------------|---------|
| 设计理念 | 全自动 | 半自动 |
| SQL 控制 | 弱 | 强 |
| 学习曲线 | 陡峭 | 平缓 |
| 灵活性 | 低 | 高 |
| 性能调优 | 困难 | 容易 |
| 维护性 | 较好 | 好 |
| 适用场景 | 简单 CRUD | 复杂业务 |

### 选择建议

1. **小型项目**：两者都可以，MyBatis Plus 更简单
2. **中型项目**：MyBatis 更灵活，便于后期优化
3. **大型项目**：MyBatis + 复杂 SQL 团队
4. **快速原型**：JPA/Hibernate 开发更快

### 核心观点

> JPA 是"面向对象优先"，MyBatis 是"SQL 优先"。

没有绝对的好坏，只有适合与不适合。选择时要考虑团队能力、项目特点和业务复杂度。
