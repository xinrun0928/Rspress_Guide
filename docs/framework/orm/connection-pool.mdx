# 数据库连接池：HikariCP、Druid、C3P0

每次查数据库都要创建连接？太慢了。

数据库连接是一种昂贵的资源。建立一次 TCP 连接需要经历三次握手、SSL 握手等操作，耗时可能高达几十甚至上百毫秒。如果每次查询都创建新连接，性能会严重下降。

连接池就是为了解决这个问题——**复用连接，减少创建销毁的开销**。

## 连接池的核心思想

### 没有连接池

```
请求 1：创建连接 → 查询 → 关闭连接
请求 2：创建连接 → 查询 → 关闭连接
请求 3：创建连接 → 查询 → 关闭连接
...
每次都创建和关闭连接，开销巨大
```

### 有连接池

```
请求 1：从池中获取连接 → 查询 → 归还连接
请求 2：从池中获取连接 → 查询 → 归还连接
请求 3：从池中获取连接 → 查询 → 归还连接
...
复用已有连接，性能大幅提升
```

## 性能对比

### 基准测试数据

| 连接池 | 吞吐量（事务/秒） | 获取连接耗时 | 归还连接耗时 |
|--------|-----------------|-------------|-------------|
| HikariCP | 15,000+ | `<` 1ms | `<` 1ms |
| Druid | 10,000+ | `<` 2ms | `<` 2ms |
| C3P0 | 3,000+ | 5-10ms | 5-10ms |

### 性能差异原因

| 连接池 | 核心数据结构 | 并发控制 |
|--------|-------------|---------|
| HikariCP | FastConnectionList | 无锁链表 + CAS |
| Druid | DruidConnectionHolder | synchronized |
| C3P0 | PooledConnection | synchronized |

## 连接池参数配置

### 核心参数

| 参数 | 说明 | 推荐值 |
|------|------|--------|
| `minimumIdle` | 最小空闲连接数 | `=` `maximumPoolSize` 或 `5-10` |
| `maximumPoolSize` | 最大连接数 | `=` CPU 核心数 × 2 + 磁盘数 |
| `connectionTimeout` | 获取连接超时 | `30,000ms` |
| `idleTimeout` | 空闲超时 | `600,000ms`（10 分钟） |
| `maxLifetime` | 连接最大生命周期 | `1,800,000ms`（30 分钟） |
| `connectionTestQuery` | 连接测试 SQL | `SELECT 1` |

### 连接池大小计算

**公式**：

```
连接数 = (CPU 核心数 × 2) + 有效磁盘数
```

**示例**：

| CPU | 磁盘 | 推荐连接数 |
|-----|------|-----------|
| 4 核 | 1 SSD | 9 |
| 8 核 | 2 SSD | 18 |
| 16 核 | 4 SSD | 36 |

**注意事项**：
- 连接数不是越大越好
- 连接数过多会增加数据库压力
- 业务特点决定最优值

### 配置示例

#### HikariCP

```yaml
spring:
  datasource:
    hikari:
      # 连接池大小
      maximum-pool-size: 20
      minimum-idle: 5
      # 超时配置
      connection-timeout: 30000
      idle-timeout: 600000
      max-lifetime: 1800000
      # 连接测试
      connection-test-query: SELECT 1
```

#### Druid

```yaml
spring:
  datasource:
    druid:
      # 连接池大小
      initial-size: 5
      min-idle: 5
      max-active: 20
      # 超时配置
      max-wait: 60000
      # 连接检测
      validation-query: SELECT 1
      test-while-idle: true
      test-on-borrow: false
      test-on-return: false
```

## 连接池不是越大越好

### 连接竞争问题

```
连接池太小：
    请求 → 等待连接 → 获取连接 → 查询 → 归还
    请求 → 等待连接 → 等待连接 → 等待连接 → ...

连接池太大：
    连接 1：空闲
    连接 2：空闲
    连接 3：执行查询（CPU 等待磁盘 I/O）
    连接 4：执行查询（CPU 等待磁盘 I/O）
    ...
    问题：CPU 大部分时间在等待，真正在计算的连接很少
```

### 正确理解

| 场景 | CPU 利用率 | 最佳连接数 |
|------|-----------|-----------|
| CPU 密集型 | 高 | `=` CPU 核心数 |
| I/O 密集型 | 低 | `>` CPU 核心数 |

### 经验公式

```
┌─────────────────────────────────────────────────────────────┐
│                     连接池大小经验公式                        │
│                                                             │
│  connections = ((core_count * 2) + effective_spindle_count)│
│                                                             │
│  core_count = CPU 核心数                                     │
│  spindle_count = 磁盘数（机械硬盘）或 1（SSD）               │
│                                                             │
│  SSD 场景：connections = core_count * 2 + 1                 │
│  HDD 场景：connections = core_count * 2 + 磁盘数            │
└─────────────────────────────────────────────────────────────┘
```

## 常见问题

### Q1：连接池大小怎么设置？

根据业务特点调整：
- I/O 密集型：可以大一些
- CPU 密集型：不宜过大
- 简单原则：**宁可小而够用，不要大而浪费**

### Q2：连接超时怎么办？

```yaml
hikari:
  connection-timeout: 30000  # 30 秒超时
```

检查：
1. 连接池是否已满
2. 数据库是否响应慢
3. 网络是否有问题

### Q3：连接泄漏怎么办？

连接泄漏：获取连接后忘记归还。

```yaml
hikari:
  leak-detection-threshold: 60000  # 1 分钟检测
```

开启泄漏检测后，如果连接被占用超过阈值，会打印警告日志。

### Q4：数据库重启后连接池怎么办？

HikariCP 有连接验证机制，会自动处理：

```yaml
hikari:
  validation-timeout: 3000  # 验证超时
  connection-test-query: SELECT 1
```

每次获取连接前，会执行验证 SQL，确认连接可用。

### 配置要点

1. **连接数不宜过大**：`=` CPU 核心数 × 2
2. **设置合理超时**：防止无限等待
3. **配置连接验证**：确保连接可用
4. **开启泄漏检测**：及时发现问题

连接池是数据库性能优化的重要一环。选择合适的连接池，配置合理的参数，可以让数据库操作性能提升数倍。
