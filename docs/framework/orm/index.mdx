# ORM 与数据库访问：从 JDBC 到 ORM 的演进

<Badge text="ORM" type="tip" /> <Badge text="基础概念" type="info" />

ORM（Object Relational Mapping），对象关系映射——这是现代 Java 后端开发中不可或缺的技术。

从最初的 JDBC，到 Hibernate、JPA，再到 MyBatis、MyBatis Plus，ORM 技术经历了怎样的演进？为什么现在 MyBatis 如此流行？JPA 和 MyBatis 各自适合什么场景？

这篇文章，带你理解 ORM 的来龙去脉。

## 什么是 ORM？

### 概念定义

ORM 是一种程序设计技术，用于实现面向对象编程语言中不同类型系统（如 Java）之间的数据转换。

```
┌─────────────────────────────────────────────────────────────┐
│                        面向对象                                │
│                                                             │
│   ┌──────────────┐   ┌──────────────┐   ┌──────────────┐    │
│   │    User      │   │   Order      │   │ Department   │    │
│   │  - id        │   │  - id        │   │  - id        │    │
│   │  - name      │   │  - userId    │   │  - name      │    │
│   │  - email     │   │  - amount    │   │  - code      │    │
│   └──────────────┘   └──────────────┘   └──────────────┘    │
│                           ↓                                   │
│                     对象之间的关联                              │
└─────────────────────────────────────────────────────────────┘
                            ↓  ORM 转换
┌─────────────────────────────────────────────────────────────┐
│                        关系型数据库                            │
│                                                             │
│   ┌──────────────┐   ┌──────────────┐   ┌──────────────┐    │
│   │   user 表     │   │   order 表   │   │ department 表 │    │
│   │  id          │   │  id          │   │  id          │    │
│   │  name        │   │  user_id     │   │  name        │    │
│   │  email       │   │  amount      │   │  code        │    │
│   └──────────────┘   └──────────────┘   └──────────────┘    │
│                           ↓                                   │
│                     表之间的关联（外键）                        │
└─────────────────────────────────────────────────────────────┘
```

### 为什么需要 ORM？

没有 ORM 时，我们需要手写 SQL 来操作数据库：

```java
// JDBC 原生方式
Connection conn = dataSource.getConnection();
PreparedStatement ps = conn.prepareStatement(
    "SELECT id, name, email FROM user WHERE id = ?"
);
ps.setLong(1, 1L);
ResultSet rs = ps.executeQuery();

User user = null;
if (rs.next()) {
    user = new User();
    user.setId(rs.getLong("id"));
    user.setName(rs.getString("name"));
    user.setEmail(rs.getString("email"));
}

rs.close();
ps.close();
conn.close();
```

有了 ORM，我们可以这样写：

```java
// MyBatis Plus
User user = userMapper.selectById(1L);
```

## ORM 的演进历程

### 第一代：JDBC

1997 年，JDBC（Java Database Connectivity）诞生。它是 Java 操作数据库的标准 API，但使用起来非常繁琐。

```java
// JDBC 的问题：
// 1. 代码冗长
// 2. 需要手动管理资源
// 3. 没有对象映射
// 4. SQL 和代码耦合
```

### 第二代：Hibernate

2001 年，Hibernate 诞生。它是第一款主流的 ORM 框架，实现了全自动的对象-表映射。

```java
// Hibernate：全自动映射
User user = session.get(User.class, 1L);
user.setName("新名字");
session.save(user);
```

**问题**：
- SQL 是自动生成的，不够灵活
- 复杂查询需要使用 HQL，调试困难
- 学习曲线陡峭

### 第三代：JPA

2002006 年，JPA（Java Persistence API）诞生。它是一套规范，由多家厂商实现（Hibernate、 EclipseLink 等）。

```java
// JPA：标准化 API
@Entity
public class User {
    @Id
    private Long id;
    private String name;
}

@Repository
public interface UserRepository extends JpaRepository<User, Long> {
}
```

### 第四代：MyBatis

2010 年，MyBatis 诞生。它是半自动化的 ORM 框架，需要手写 SQL，但提供了强大的映射能力。

```java
// MyBatis：半自动化
User user = sqlSession.selectOne("UserMapper.selectById", 1L);

// MyBatis Plus：更简单
User user = userMapper.selectById(1L);
```

### 第五代：MyBatis Plus

2016 年，MyBatis Plus 诞生。它在 MyBatis 基础上做了大量增强，进一步简化了 CRUD 操作。

## 主流 ORM 框架对比

| 框架 | 类型 | SQL 控制 | 学习曲线 | 灵活性 |
|------|------|---------|---------|--------|
| JDBC | 原生 | 完全控制 | 低 | 高 |
| Hibernate | 全自动 | 不可控 | 高 | 低 |
| JPA | 全自动 | 部分可控 | 中 | 中 |
| MyBatis | 半自动 | 完全控制 | 中 | 高 |
| MyBatis Plus | 半自动增强 | 完全控制 | 低 | 高 |

## 两条技术路线

```
┌─────────────────────────────────────────────────────────────┐
│                        ORM 框架                               │
│                                                             │
│  ┌─────────────────────────┐    ┌─────────────────────────┐ │
│  │      全自动化路线          │    │      半自动化路线         │ │
│  │                         │    │                         │ │
│  │  Hibernate → JPA        │    │  MyBatis → MyBatis Plus │ │
│  │                         │    │                         │ │
│  │  配置即映射               │    │  SQL + 映射             │ │
│  │  对象即查询               │    │  映射 + 自动生成          │ │
│  │  开发者写对象             │    │  开发者写 SQL             │ │
│  │                         │    │                         │ │
│  └─────────────────────────┘    └─────────────────────────┘ │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 全自动化路线（JPA/Hibernate）

**核心理念**：开发者只需要关注对象，不需要关注 SQL。

```java
// 一切皆对象
User user = new User();
user.setName("张三");
userRepository.save(user);

// 自动生成 SQL
// INSERT INTO user (name) VALUES ('张三')
```

**适合场景**：
- 业务简单，以 CRUD 为主
- 团队 SQL 能力较弱
- 快速原型开发

### 半自动化路线（MyBatis）

**核心理念**：SQL 由开发者掌控，映射由框架处理。

```java
// SQL 由开发者手写
@Select("SELECT * FROM user WHERE name = #{name}")
List<User> selectByName(String name);
```

**适合场景**：
- 业务复杂，需要精细 SQL 优化
- 团队 SQL 能力强
- 对性能要求高

## 延伸思考

### ORM 的本质

ORM 的本质是**解决阻抗失配（Impedance Mismatch）**问题：

```
面向对象          关系型数据库
─────────        ────────────
类                表
属性              字段
对象引用          外键
继承              无直接对应
多态              无直接对应
```

ORM 就是在这两种模型之间建立桥梁。

### 为什么 MyBatis 更流行？

1. **SQL 控制**：国内团队普遍重视 SQL 调优
2. **学习曲线**：比 Hibernate 简单
3. **灵活性**：可以写任何 SQL
4. **生态完善**：MyBatis Plus 大幅简化开发

### NoSQL 的冲击

随着 NoSQL 的兴起，ORM 的重要性有所下降：

| 数据库类型 | 特点 | ORM 需求 |
|-----------|------|---------|
| 关系型 | 强一致性 | 强 |
| MongoDB | 文档存储 | 中 |
| Redis | KV 存储 | 弱 |
| Neo4j | 图数据库 | 弱 |

但关系型数据库仍然是主流，ORM 技术仍然是 Java 后端开发的核心技能。
