# 表单登录：UsernamePasswordAuthenticationFilter 的完整流程

输入用户名密码，点击登录——Spring Security 验证了什么？

大部分人的答案是：「密码验证通过了。」

这个答案没有错，但太浅了。表单是怎么提交的？密码是怎么比对的？验证通过后跳转到哪里？验证失败后又发生了什么？

这一节，就是要回答这些问题。

---

## 表单登录的整体流程

```
用户提交表单（POST /login，username + password）
    ↓
UsernamePasswordAuthenticationFilter 拦截
    ↓
生成 UsernamePasswordAuthenticationToken（未认证）
    ↓
交给 AuthenticationManager
    ↓
ProviderManager 找到 DaoAuthenticationProvider
    ↓
DaoAuthenticationProvider 调用 UserDetailsService.loadUserByUsername()
    ↓
DaoAuthenticationProvider 调用 PasswordEncoder.matches() 校验密码
    ↓
校验通过 → Authentication.authenticated = true
    ↓
SecurityContextHolder 保存 Authentication
    ↓
AuthenticationSuccessHandler 处理成功（重定向到目标页面）
    ↓
校验失败 → AuthenticationFailureHandler 处理失败（重定向到登录页）
```

## UsernamePasswordAuthenticationFilter 源码解析

### 过滤器的工作原理

`UsernamePasswordAuthenticationFilter` 是处理表单登录的核心过滤器。让我们看一下它的 `doFilter` 方法：

```java
public class UsernamePasswordAuthenticationFilter extends
        AbstractAuthenticationProcessingFilter {

    // 表单中用户名和密码的字段名
    public static final String SPRING_SECURITY_FORM_USERNAME_KEY = "username";
    public static final String SPRING_SECURITY_FORM_PASSWORD_KEY = "password";

    private String usernameParameter = SPRING_SECURITY_FORM_USERNAME_KEY;
    private String passwordParameter = SPRING_SECURITY_FORM_PASSWORD_KEY;

    public Authentication doFilter(ServletRequest request, ServletResponse response,
            FilterChain chain) throws IOException, ServletException {

        // 只处理 POST 请求
        if (!requiresAuthentication(request, response)) {
            chain.doFilter(request, response);
            return;
        }

        try {
            // 从请求中提取用户名和密码
            String username = obtainUsername(request);
            String password = obtainPassword(request);

            // 如果用户名已空（可能已被其他过滤器处理过），尝试从 SecurityContext 获取
            if (username == null) {
                username = "";
            }
            if (password == null) {
                password = "";
            }

            // 去除首尾空格
            username = username.trim();

            // 生成未认证的 Authentication Token
            UsernamePasswordAuthenticationToken authRequest =
                new UsernamePasswordAuthenticationToken(username, password);

            // 设置详细信息（WebAuthenticationDetails，包含 SessionId、RemoteAddress 等）
            setDetails(request, authRequest);

            // 核心：委托给 AuthenticationManager 进行认证
            Authentication authResult =
                this.getAuthenticationManager().authenticate(authRequest);

            // 认证成功后的处理
            successfulAuthentication(request, response, chain, authResult);

        } catch (AuthenticationException failed) {
            // 认证失败后的处理
            unsuccessfulAuthentication(request, response, failed);
        }

        return;
    }
}
```

### 认证成功的处理

```java
protected void successfulAuthentication(HttpServletRequest request,
        HttpServletResponse response, FilterChain chain, Authentication authResult)
        throws IOException, ServletException {

    // 将认证结果存入 SecurityContextHolder
    SecurityContext context = SecurityContextHolder.createEmptyContext();
    context.setAuthentication(authResult);
    SecurityContextHolder.setContext(context);

    // 如果配置了 Remember-Me，生成 Remember-Me Token
    if (rememberMeServices != null) {
        rememberMeServices.loginSuccess(request, response, authResult);
    }

    // 发布认证成功事件
    // 可以在这里添加监听器来处理登录成功事件（如记录登录日志）
    publishEvent(new InteractiveAuthenticationSuccessEvent(
        authResult, getClass()));

    // 调用 SuccessHandler，决定登录成功后的跳转
    successHandler.onAuthenticationSuccess(request, response, authResult);
}
```

### 认证失败的处理

```java
protected void unsuccessfulAuthentication(HttpServletRequest request,
        HttpServletResponse response, AuthenticationException failed)
        throws IOException, ServletException {

    // 清除 SecurityContext
    SecurityContextHolder.clearContext();

    // 如果配置了 Remember-Me，清除 Remember-Me Token
    if (rememberMeServices != null) {
        rememberMeServices.loginFail(request, response);
    }

    // 发布认证失败事件
    publishEvent(new AuthenticationFailureFailedEvent(
        this, request, response, failed));

    // 调用 FailureHandler，决定登录失败后的跳转
    failureHandler.onAuthenticationFailure(request, response, failed);
}
```

## Authentication 对象的构建过程

### UsernamePasswordAuthenticationToken

这是一个两步构建的过程：

```java
// 第一步：未认证状态（只有 principal 和 credentials）
UsernamePasswordAuthenticationToken authRequest =
    new UsernamePasswordAuthenticationToken(username, password);
// authRequest.isAuthenticated() == false

// 第二步：认证成功后，变成已认证状态（增加 authorities）
// 注意：这个转换通常由 AuthenticationManager 完成
// authRequest.setAuthenticated(true);
// authRequest.setAuthorities(authorities);
```

为什么设计成两步？**因为未认证和已认证的 Authentication 包含的信息不同。**

| 状态 | principal | credentials | authorities |
|---|---|---|---|
| 未认证 | 用户名（String） | 密码（String） | 空集合 |
| 已认证 | UserDetails | null | 权限列表 |

## AuthenticationManager 的认证决策

### ProviderManager

最常用的 `AuthenticationManager` 实现是 `ProviderManager`，它内部维护一个 `AuthenticationProvider` 列表：

```java
public class ProviderManager implements AuthenticationManager {
    private List<AuthenticationProvider> providers;

    public Authentication authenticate(Authentication authentication)
            throws AuthenticationException {

        Class<?> authenticationType = authentication.getClass();
        AuthenticationException lastException = null;
        Authentication result = null;

        // 遍历所有 Provider，找一个能处理这个 Authentication 的
        for (AuthenticationProvider provider : providers) {
            if (!provider.supports(authenticationType)) {
                continue;
            }

            try {
                // 调用 Provider 的认证方法
                result = provider.authenticate(authentication);

                // 认证成功，复制认证结果
                if (result != null) {
                    copyDetails(authentication, result);
                    break;
                }
            } catch (AccountStatusException | InternalAuthenticationServiceException ex) {
                // 账户状态异常（如账户被禁用）直接抛出
                prepareException(ex, authentication);
                throw ex;
            } catch (AuthenticationException ex) {
                lastException = ex;
                // 继续尝试其他 Provider
            }
        }

        if (result == null && lastException != null) {
            throw lastException;
        }

        return result;
    }
}
```

**为什么一个 Provider 认证失败后还要尝试其他的？**

因为可能有多个 Provider 支持同一种认证方式。比如你配置了两个 `AuthenticationProvider`，它们都支持 `UsernamePasswordAuthenticationToken`，但分别对应不同的用户数据源（管理员用户 vs 普通用户）。

## DaoAuthenticationProvider 的认证过程

### 完整的认证流程

```java
public class DaoAuthenticationProvider extends AbstractUserDetailsAuthenticationProvider {

    private PasswordEncoder passwordEncoder;
    private UserDetailsService userDetailsService;

    // 实际执行认证的方法
    protected void additionalAuthenticationChecks(
            UserDetails userDetails,
            UsernamePasswordAuthenticationToken authentication)
            throws AuthenticationException {

        // 如果没有提供密码（密码为空），而用户有密码，抛出异常
        if (authentication.getCredentials() == null
                || userDetails.getPassword() == null) {
            throw new BadCredentialsException(
                "密码不能为空");
        }

        // 获取用户提交的密码
        String presentedPassword = authentication.getCredentials().toString();

        // 核心：用 PasswordEncoder 校验密码
        if (!passwordEncoder.matches(presentedPassword, userDetails.getPassword())) {
            throw new BadCredentialsException("密码错误");
        }
    }

    // 获取用户信息
    protected UserDetails retrieveUser(String username,
            UsernamePasswordAuthenticationToken authentication)
            throws AuthenticationException {

        // 调用 UserDetailsService 加载用户
        UserDetails loadedUser = this.userDetailsService.loadUserByUsername(username);

        if (loadedUser == null) {
            throw new UsernameNotFoundException(
                "用户不存在: " + username);
        }

        return loadedUser;
    }

    // 认证成功后的额外检查
    protected void doAfterAuthenticationChecks(UserDetails userDetails,
            UsernamePasswordAuthenticationToken authentication)
            throws AuthenticationException {

        // 检查账户是否被锁定
        if (!userDetails.isAccountNonLocked()) {
            throw new LockedException("账户已被锁定");
        }

        // 检查账户是否启用
        if (!userDetails.isEnabled()) {
            throw new DisabledException("账户已被禁用");
        }

        // 检查账户是否过期
        if (!userDetails.isAccountNonExpired()) {
            throw new AccountExpiredException("账户已过期");
        }

        // 检查凭证是否过期
        if (!userDetails.isCredentialsNonExpired()) {
            throw new CredentialsExpiredException("凭证已过期");
        }
    }
}
```

### 认证顺序

```
1. retrieveUser() → UserDetailsService.loadUserByUsername()
                      ↓
2. additionalAuthenticationChecks() → PasswordEncoder.matches()
                      ↓
3. doAfterAuthenticationChecks() → isAccountNonLocked/isEnabled/isAccountNonExpired/isCredentialsNonExpired
                      ↓
4. 返回已认证的 Authentication
```

## SuccessHandler 和 FailureHandler

### 默认行为

- **SuccessHandler 默认**：`SavedRequestCache` 保存用户原始请求，认证成功后重定向到原始请求或主页
- **FailureHandler 默认**：重定向到登录页，并带上错误参数

### 自定义 SuccessHandler

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests(auth -> auth
                .anyRequest().authenticated()
            )
            .formLogin(form -> form
                .loginProcessingUrl("/login")
                // 方式一：使用 defaultSuccessUrl
                .defaultSuccessUrl("/home", true)  // true: 强制跳转，false: 优先跳转原始请求
                // 方式二：自定义 SuccessHandler
                .successHandler((request, response, authentication) -> {
                    response.setContentType("application/json");
                    response.setStatus(200);
                    response.getWriter().write(
                        "{\"message\": \"登录成功\", \"username\": \""
                        + authentication.getName() + "\"}"
                    );
                })
            );

        return http.build();
    }
}
```

### defaultSuccessUrl 的第二个参数

```java
.defaultSuccessUrl("/home", true)   // true = 强制跳转到 /home
.defaultSuccessUrl("/home", false)  // false = 优先跳转原始请求，没有原始请求才跳转到 /home
```

`false` 的场景：用户直接访问 `/admin`，被拦截到登录页，登录成功后应该跳转到 `/admin` 而不是 `/home`。

### 自定义 FailureHandler

```java
.failureHandler((request, response, exception) -> {
    response.setContentType("application/json");
    response.setStatus(401);

    String message = "登录失败";
    if (exception instanceof BadCredentialsException) {
        message = "用户名或密码错误";
    } else if (exception instanceof LockedException) {
        message = "账户已被锁定";
    } else if (exception instanceof DisabledException) {
        message = "账户已被禁用";
    }

    response.getWriter().write("{\"message\": \"" + message + "\"}");
})
```

### SavedRequest 的工作原理

```
用户访问 /admin（未登录）
    ↓
SecurityContextPersistenceFilter 保存当前请求到 Session（SavedRequest）
    ↓
重定向到 /login
    ↓
用户登录成功
    ↓
SavedRequestAwareAuthenticationSuccessHandler 从 Session 取出 SavedRequest
    ↓
重定向到 /admin
```

## 完整配置示例

### 基础配置

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            // CSRF 配置（表单登录需要 CSRF Token）
            .csrf(csrf -> csrf
                .ignoringRequestMatchers("/api/**")  // API 接口关闭 CSRF
            )
            // 授权配置
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/login", "/register").permitAll()
                .requestMatchers("/admin/**").hasRole("ADMIN")
                .anyRequest().authenticated()
            )
            // 表单登录配置
            .formLogin(form -> form
                .loginPage("/login")                  // 登录页面
                .loginProcessingUrl("/doLogin")        // 登录处理接口
                .defaultSuccessUrl("/home")           // 默认成功页
                .failureUrl("/login?error")           // 失败重定向
                .usernameParameter("username")         // 用户名字段名
                .passwordParameter("password")        // 密码字段名
                .permitAll()
            )
            // 登出配置
            .logout(logout -> logout
                .logoutUrl("/logout")
                .logoutSuccessUrl("/login?logout")
                .invalidateHttpSession(true)
                .deleteCookies("JSESSIONID")
            )
            // Session 管理
            .sessionManagement(session -> session
                .sessionCreationPolicy(SessionCreationPolicy.IF_REQUIRED)
                .maximumSessions(1)
            );

        return http.build();
    }

    @Bean
    public UserDetailsService userDetailsService() {
        InMemoryUserDetailsManager manager = new InMemoryUserDetailsManager();
        manager.createUser(
            User.withUsername("admin")
                .password(passwordEncoder().encode("123456"))
                .roles("ADMIN")
                .build()
        );
        return manager;
    }

    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
    }
}
```

## 面试追问预判

### Q1：表单登录和 HTTP Basic 认证的区别是什么？

| 维度 | 表单登录 | HTTP Basic |
|---|---|---|
| 凭证传递方式 | POST 表单，密码在请求体中 | 放在 Authorization 头中（Base64 编码） |
| 会话管理 | 有 Session，支持 Remember-Me | 无状态，每次请求都要带凭证 |
| 登录页面 | 可自定义 | 浏览器原生弹窗 |
| 适用场景 | Web 应用 | RESTful API、测试环境 |

### Q2：为什么表单登录需要 CSRF 防护？

因为表单提交是 POST 请求，如果没有 CSRF 防护，攻击者可以诱导用户访问恶意页面，恶意页面自动提交表单，以用户的身份执行操作（如转账）。CSRF Token 确保表单是用户主动提交的，而不是被恶意页面诱导的。

### Q3：`/login` 是 Spring Security 的默认路径吗？

是的。Spring Security 默认的登录 URL 是 `/login`。如果你想改，需要在配置中指定：

```java
.formLogin(form -> form
    .loginProcessingUrl("/my-login")  // 改掉默认的 /login
)
```

但 `/login` 仍然会被 `formLogin()` 自动启用（作为登录页面 URL）。如果想完全禁用表单登录，用 `.disable()` 而不是不调用 `.formLogin()`。

## 总结

表单登录的完整流程：

1. **过滤器拦截**：`UsernamePasswordAuthenticationFilter` 拦截 POST `/login` 请求
2. **生成 Token**：提取用户名和密码，生成未认证的 `UsernamePasswordAuthenticationToken`
3. **认证决策**：`ProviderManager` 委托给 `DaoAuthenticationProvider`
4. **加载用户**：`UserDetailsService.loadUserByUsername()` 从数据源加载用户
5. **校验密码**：`PasswordEncoder.matches()` 比对密码
6. **保存结果**：`SecurityContextHolder` 保存认证结果
7. **处理成功/失败**：`SuccessHandler` 或 `FailureHandler` 处理后续
