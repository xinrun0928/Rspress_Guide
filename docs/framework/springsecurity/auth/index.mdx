# 认证机制：Spring Security 的第一道关

用户登录，Spring Security 怎么知道用户名和密码对不对？登录成功后，后续请求怎么知道用户已经登录了？不同场景下的认证（表单登录、短信登录、OAuth2 登录）是怎么实现的？

这些问题，都在这一章回答。

## 认证的本质是什么？

认证（Authentication），本质上是一个回答问题的过程：

> **问题**：你是谁？
> **回答**：我是 zhangsan，密码是 123456。

Spring Security 的认证，就是接收这个回答，然后验证它是否正确。

但「回答」的方式有很多种：

- 表单登录：用户名 + 密码
- 短信登录：手机号 + 验证码
- OAuth2 登录：第三方平台授权
- CAS 单点登录：票据验证
- LDAP 登录：企业目录服务

每一种「回答」方式，都是一种认证机制。Spring Security 通过 `AuthenticationProvider` 支持这些不同的认证机制。

## 认证机制全景图

```
┌─────────────────────────────────────────────────────────────┐
│                     认证过滤器链                             │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  FilterChainProxy                                            │
│       ↓                                                      │
│  SecurityContextPersistenceFilter（从 Session 恢复认证状态）  │
│       ↓                                                      │
│  ┌─────────────────────────────────────────────────────┐    │
│  │         认证过滤器（处理不同的认证方式）                │    │
│  │                                                       │    │
│  │   UsernamePasswordAuthenticationFilter  ← 表单登录     │    │
│  │   SmsCodeAuthenticationFilter         ← 短信登录       │    │
│  │   OAuth2LoginAuthenticationFilter     ← OAuth2 登录    │    │
│  │   CasAuthenticationFilter             ← CAS 单点登录   │    │
│  │   BasicAuthenticationFilter          ← HTTP Basic     │    │
│  │   ...                                                   │    │
│  │                                                       │    │
│  └─────────────────────────────────────────────────────┘    │
│       ↓                                                      │
│  FilterSecurityInterceptor（权限决策）                        │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 过滤器的工作流程

1. **捕获请求**：认证过滤器拦截登录请求
2. **生成 Token**：将请求信息封装成 `AuthenticationToken`
3. **委托认证**：调用 `AuthenticationManager`
4. **验证身份**：通过 `AuthenticationProvider` 验证
5. **存储结果**：将认证结果存入 `SecurityContext`
6. **处理成功/失败**：调用 `SuccessHandler` 或 `FailureHandler`

## 认证机制分类

### 凭证型认证

依靠用户「知道什么」（密码）或「拥有什么」（手机）来验证身份。

| 认证方式 | 凭证类型 | 适用场景 |
|---|---|---|
| 表单登录 | 用户名 + 密码 | 普通 Web 应用 |
| 短信登录 | 手机号 + 验证码 | 移动端 |
| 邮箱登录 | 邮箱 + 验证码 | 备用登录方式 |
| API Key | 密钥字符串 | 服务间调用 |

### 授权型认证

用户授权第三方访问自己的资源，认证过程由第三方完成。

| 认证方式 | 说明 | 适用场景 |
|---|---|---|
| OAuth2 登录 | QQ、微信、GitHub 授权 | 社交登录 |
| CAS | 中央认证服务 | 企业内部多系统 |
| SAML | 安全断言标记语言 | 企业间单点登录 |

### 无状态认证

不依赖 Session，Token 携带认证信息。

| 认证方式 | 说明 | 适用场景 |
|---|---|---|
| JWT | 自包含的 Token | RESTful API |
| API Key | 静态密钥 | 服务间调用 |
| OAuth2 Access Token | 授权令牌 | 微服务架构 |

## 认证流程的核心组件

### Authentication（认证信息）

```java
public interface Authentication extends Principal, Serializable {
    Collection<? extends GrantedAuthority> getAuthorities();
    Object getCredentials();
    Object getDetails();
    Object getPrincipal();
    boolean isAuthenticated();
    void setAuthenticated(boolean isAuthenticated) throws IllegalArgumentException;
}
```

**认证前**：`principal` = 用户名，`credentials` = 密码，`authorities` = 空

**认证后**：`principal` = UserDetails，`credentials` = null，`authorities` = 权限列表

### AuthenticationManager（认证管理器）

认证的入口，接收 `Authentication`，返回已认证的 `Authentication`。

```java
public interface AuthenticationManager {
    Authentication authenticate(Authentication authentication)
        throws AuthenticationException;
}
```

### AuthenticationProvider（认证提供者）

具体的认证逻辑，每个 `AuthenticationProvider` 支持一种认证方式。

```java
public interface AuthenticationProvider {
    Authentication authenticate(Authentication authentication)
        throws AuthenticationException;
    boolean supports(Class<?> authentication);
}
```

## 会话与会话管理

认证成功后，用户状态需要保持。这就是 Session 的作用。

### Session 的生命周期

```
用户登录 → 创建 Session → Session ID 写入 Cookie
                              ↓
                    后续请求携带 Cookie
                              ↓
                    Session ID 找到 Session
                              ↓
                    Session 中取出 SecurityContext
                              ↓
                    恢复用户认证状态
```

### Session 管理的关键配置

```java
http.sessionManagement(session -> session
    // 会话创建策略
    .sessionCreationPolicy(SessionCreationPolicy.IF_REQUIRED)
    // 并发控制：一个用户最多几个会话
    .maximumSessions(1)
    // 会话固定攻击防护
    .sessionFixation().newSession()
    // 会话超时
    .maximumSessions(1).expiredUrl("/session-expired")
);
```

### Remember-Me：自动登录

用户勾选「记住我」，Spring Security 在登录成功后生成一个 Token，写入 Cookie。下次访问时，自动验证 Token，恢复登录状态。

```
┌─────────────────────────────────────────────────────────────┐
│                 Remember-Me 流程                             │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  用户登录，勾选「记住我」                                      │
│       ↓                                                      │
│  Spring Security 生成 Token（序列号 + 签名）                  │
│       ↓                                                      │
│  Token 写入 Cookie                                           │
│       ↓                                                      │
│  用户关闭浏览器，再次访问                                      │
│       ↓                                                      │
│  RememberMeFilter 拦截请求                                    │
│       ↓                                                      │
│  读取 Cookie 中的 Token                                       │
│       ↓                                                      │
│  验证 Token 签名（服务端有密钥）                               │
│       ↓                                                      │
│  Token 有效 → 自动登录 → 设置 SecurityContext                  │
│       ↓                                                      │
│  Token 无效/过期 → 忽略，正常访问                              │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

## 认证与业务数据的对接

### UserDetailsService

Spring Security 不知道你的用户存在哪里——可能在 MySQL，可能在 Redis，可能在 LDAP。`UserDetailsService` 就是这个抽象：

```java
public interface UserDetailsService {
    UserDetails loadUserByUsername(String username) throws UsernameNotFoundException;
}
```

实现这个接口，你的认证逻辑就可以接入 Spring Security。

### PasswordEncoder

密码不能明文存储。`PasswordEncoder` 定义了密码的加密和验证方式：

```java
public interface PasswordEncoder {
    String encode(CharSequence rawPassword);
    boolean matches(CharSequence rawPassword, String encodedPassword);
}
```

Spring Security 推荐使用 `BCryptPasswordEncoder`——自带盐值，防彩虹表攻击。

## 面试核心问题

### Q1：Spring Security 的认证流程是什么？

从请求进入过滤器链开始：FilterSecurityInterceptor 前的认证过滤器（UsernamePasswordAuthenticationFilter 等）从请求中提取认证信息，生成 Authentication Token，交给 AuthenticationManager。AuthenticationManager 委托给 AuthenticationProvider，Provider 调用 UserDetailsService 加载用户数据，然后 PasswordEncoder 校验密码。校验通过后，SecurityContextHolder 保存 Authentication，最后 SecurityContextPersistenceFilter 将其存入 Session。

### Q2：表单登录和 OAuth2 登录有什么区别？

表单登录是用户直接向应用提供凭证，应用自己验证。OAuth2 登录是用户向第三方平台授权，第三方返回授权码，应用用授权码换取用户信息。两者在 Spring Security 中的入口过滤器不同，但认证结果的存储方式是一样的——都存入 SecurityContext。

### Q3：Session 和 Token 的区别是什么？

Session 的认证信息存在服务端，客户端只持有 Session ID。Token 的认证信息编码在 Token 本身中，服务端只需要验证签名。Session 适合有状态的 Web 应用，Token 适合无状态的 RESTful API。

## 延伸思考

认证解决了「你是谁」的问题，但认证之后还有授权。同一个用户，可能是普通用户，也可能是管理员——他们能访问的资源不同。这就是下一章要解决的问题。

但你有没有想过：同一个用户在不同设备上登录，算一个会话还是两个？如果允许并发登录，怎么控制？如果不允许，怎么踢人下线？
