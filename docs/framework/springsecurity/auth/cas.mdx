# CAS 单点登录：一次登录，处处访问

你有没有想过，在企业内部系统中，员工登录 OA 系统后，为什么访问财务系统、HR 系统也不需要再登录？

这就是 CAS（Central Authentication Service）——中央认证服务。

---

## CAS 的核心原理

### 什么是 CAS？

CAS 是一个开源的单点登录协议，目的是实现「一次登录，处处访问」。用户只需要在一个地方登录，就可以访问所有信任 CAS 的应用系统。

```
┌─────────────────────────────────────────────────────────────┐
│                    CAS 工作原理                                │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  用户浏览器                                                    │
│       ↓                                                      │
│  访问应用 A（未登录）                                          │
│       ↓                                                      │
│  重定向到 CAS Server                                          │
│       ↓                                                      │
│  CAS Server 显示登录页面                                       │
│       ↓                                                      │
│  用户输入用户名密码，CAS Server 验证                           │
│       ↓                                                      │
│  CAS Server 生成 TGC（Ticket Granting Cookie），写入浏览器     │
│       ↓                                                      │
│  CAS Server 生成 ST（Service Ticket），重定向回应用 A          │
│       ↓                                                      │
│  应用 A 用 ST 向 CAS Server 验证                               │
│       ↓                                                      │
│  验证通过，应用 A 创建本地 Session                              │
│       ↓                                                      │
│  用户访问应用 B（已登录，无需再次认证）                         │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 核心概念

| 概念 | 说明 |
|---|---|
| TGT（Ticket Granting Ticket） | CAS Server 生成的全局会话，存储在 Server 端 |
| TGC（Ticket Granting Cookie） | 浏览器存储的 Cookie，用于识别 TGT |
| ST（Service Ticket） | CAS Server 生成的一次性票据，用于验证用户身份 |
| Service | CAS Client（应用系统） |
| Proxy Ticket | 用于服务间调用的代理票据 |

## CAS 的认证流程详解

### 第一次访问应用

```
用户 → 访问应用 A（http://app-a.com）
    ↓
应用 A 发现用户未登录
    ↓
应用 A 生成 serviceUrl（需要回调的地址）
    ↓
重定向到 CAS Server：
  https://cas.example.com/login?service=http://app-a.com/cas/callback
    ↓
CAS Server 发现没有 TGC
    ↓
显示登录页面
    ↓
用户输入用户名密码
    ↓
CAS Server 验证通过
    ↓
创建 TGT（存在 CAS Server 内存或数据库）
    ↓
写入 TGC 到浏览器：Set-Cookie: CASTGC=TGT-xxx
    ↓
生成 ST（一次性票据）
    ↓
重定向到应用 A 的回调地址：
  http://app-a.com/cas/callback?ticket=ST-xxx
    ↓
应用 A 获取 ST
    ↓
应用 A 用 ST 向 CAS Server 验证：
  https://cas.example.com/p3/serviceValidate?service=...&ticket=ST-xxx
    ↓
CAS Server 验证 ST 有效
    ↓
返回用户信息（username）
    ↓
应用 A 创建本地 Session
    ↓
用户访问应用 A 成功
```

### 第二次访问其他应用

```
用户 → 访问应用 B（http://app-b.com）
    ↓
应用 B 发现用户未登录
    ↓
重定向到 CAS Server：
  https://cas.example.com/login?service=http://app-b.com/cas/callback
    ↓
CAS Server 发现有 TGC
    ↓
读取 TGC，对应 TGT 存在且有效
    ↓
生成 ST（一次性票据）
    ↓
重定向到应用 B 的回调地址
    ↓
应用 B 验证 ST
    ↓
应用 B 创建本地 Session
    ↓
用户访问应用 B 成功
```

## Spring Security CAS 配置

### 添加依赖

```xml
<dependency>
    <groupId>org.springframework.security</groupId>
    <artifactId>spring-security-cas</artifactId>
</dependency>
```

### CAS Client 配置

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Value("${cas.server.url}")
    private String casServerUrl;

    @Value("${cas.client.service-url}")
    private String clientServiceUrl;

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/cas/callback").permitAll()
                .anyRequest().authenticated()
            )
            .oauth2Login()  // 先用 OAuth2 模拟，真实 CAS 配置见下方
            .and()
            // 实际 CAS 配置
            .addFilterBefore(casAuthenticationFilter(),
                UsernamePasswordAuthenticationFilter.class);

        return http.build();
    }

    @Bean
    public CasAuthenticationFilter casAuthenticationFilter() {
        CasAuthenticationFilter filter = new CasAuthenticationFilter();
        filter.setAuthenticationManager(authenticationManager());
        return filter;
    }
}
```

### 完整的 CAS 配置

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Value("${cas.server.url}")
    private String casServerUrl;

    @Value("${cas.server.prefix-url}")
    private String casServerPrefixUrl;

    @Value("${cas.client.service-url}")
    private String clientServiceUrl;

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .csrf(csrf -> csrf.disable())
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/cas/callback").permitAll()
                .anyRequest().authenticated()
            )
            .addFilter(casAuthenticationFilter())
            .addFilter(casLogoutFilter())
            .addFilter(singleLogoutFilter());

        return http.build();
    }

    @Bean
    public CasAuthenticationFilter casAuthenticationFilter() {
        CasAuthenticationFilter filter = new CasAuthenticationFilter();
        filter.setAuthenticationManager(authenticationManager());
        filter.setServiceProperties(serviceProperties());
        return filter;
    }

    @Bean
    public ServiceProperties serviceProperties() {
        ServiceProperties properties = new ServiceProperties();
        properties.setService(clientServiceUrl + "/cas/callback");
        properties.setSendRenew(false);  // 是否每次都需要登录
        properties.setAuthenticateAllArtifacts(false);
        return properties;
    }

    @Bean
    public AuthenticationManager authenticationManager() {
        CasAuthenticationProvider provider = new CasAuthenticationProvider();
        provider.setKey("casProvider");
        provider.setServiceProperties(serviceProperties());
        provider.setTicketValidator(new Cas30ServiceTicketValidator(casServerUrl));
        provider.setUserDetailsService(userDetailsService());
        return new ProviderManager(Arrays.asList(provider));
    }

    @Bean
    public UserDetailsService userDetailsService() {
        return username -> {
            // 从 CAS 验证通过后，username 就是登录用户名
            // 根据用户名加载用户信息
            return User.withUsername(username)
                .password("")
                .roles("USER")
                .build();
        };
    }
}
```

### 单点登出配置

```java
@Bean
public SecurityContextLogoutHandler securityContextLogoutHandler() {
    return new SecurityContextLogoutHandler();
}

@Bean
public LogoutFilter singleLogoutFilter() {
    // 应用登出时，同步登出 CAS
    return new LogoutFilter(
        casServerUrl + "/logout?service=" + clientServiceUrl,
        securityContextLogoutHandler()
    );
}

@Bean
public CasLogoutFilter casLogoutFilter() {
    // CAS 登出时，清除应用的 Session
    return new CasLogoutFilter(
        (request, response, authentication) -> {
            // 清除本地 Session
            HttpSession session = ((HttpServletRequest) request).getSession(false);
            if (session != null) {
                session.invalidate();
            }
        },
        new SecurityContextLogoutHandler()
    );
}
```

## CAS 的安全性

### ST 的一次性

ST（Service Ticket）只能用一次。验证后立即失效，防止重放攻击。

### Ticket 的时效性

ST 有默认的 TTL（Time To Live），通常 10 秒。超过时间未使用则失效。

### HTTPS 强制

生产环境中，CAS Server 必须使用 HTTPS，确保 Cookie 和 Ticket 安全传输。

### 代理票据（Proxy Ticket）

CAS 支持服务间代理：A 应用代表用户调用 B 应用时，可以用 Proxy Ticket。

```
用户 → 应用 A → CAS Server（获取 Proxy Granting Ticket）
    ↓
应用 A → 应用 B（携带 Proxy Ticket）
    ↓
应用 B → CAS Server（验证 Proxy Ticket）
    ↓
验证通过，允许访问
```

## CAS vs OAuth2

| 维度 | CAS | OAuth2 |
|---|---|---|
| 定位 | 单点登录协议 | 授权协议 |
| 用途 | 登录多个应用 | 获取第三方资源访问权限 |
| 票据 | ST（一次性） | Access Token |
| 用户信息 | CAS Server 返回 | 需要额外调用 API |
| 复杂度 | 较低 | 较高 |
| 适用场景 | 企业内部多系统 | 第三方应用授权 |

## 常见问题与解决方案

### 问题一：TGC 过期后需要重新登录

CAS Server 的 TGT 有默认过期时间（通常是 8 小时）。过期后需要重新登录。

### 问题二：多个 CAS Client 的 Session 不同步

CAS 保证的是认证状态的同步，而不是 Session 的同步。每个应用都有自己的 Session。

### 问题三：服务间调用的认证

A 应用调用 B 应用的 API 时，A 应用需要用 Proxy Ticket 或 Basic Auth 认证。

## 面试追问预判

### Q1：CAS 和 OAuth2 登录的区别？

CAS 是单点登录协议，解决的是「一次登录，多系统访问」的问题。OAuth2 是授权协议，解决的是「用户授权第三方访问自己资源」的问题。两者解决的问题不同，但在某些场景下可以互相替代。

### Q2：CAS 的 ST 为什么只能用一次？

防止重放攻击。如果 ST 可以重复使用，攻击者截获 ST 后就可以反复使用冒充用户。

### Q3：CAS 的单点登出是怎么实现的？

CAS Server 维护了所有已登录用户的 TGT。当用户从任一应用登出时，CAS Server 会通知所有其他应用登出。

---

## 总结

CAS 是企业级单点登录的标准方案：

1. **核心机制**：TGT（全局会话）+ ST（一次性票据）
2. **认证流程**：重定向登录 → 验证 ST → 创建本地 Session
3. **Spring Security 集成**：通过 `CasAuthenticationFilter` 和 `CasAuthenticationProvider`
4. **单点登出**：CAS Server 统一管理登出状态
