# PasswordEncoder：BCrypt 密码加密与盐值

你可能听说过「密码不能明文存储」，但为什么不能？怎样才算「安全存储」？

很多人会说：「MD5 摘要一下就行了。」但这是**错误的**。MD5 太快了，GPU 破解每秒几十亿次；MD5 没有盐，容易被彩虹表攻击。

Spring Security 的 `PasswordEncoder` 接口，就是为了解决「如何安全存储密码」这个问题。它支持多种加密算法，最常用的是 BCrypt——一个专为密码设计的哈希算法。

这一节，彻底讲清楚密码加密的原理，以及 BCrypt 为什么是目前最好的选择之一。

## 为什么不能明文存储密码？

### 明文存储的危害

2019 年，Stack Exchange 被拖库，超过 4 亿用户的密码被泄露。如果密码是明文存储的：
- 攻击者直接拿到所有密码
- 用户在其他网站使用的相同密码也会被尝试
- 公司声誉严重受损

### 简单哈希的问题

有人说：「用 MD5 或 SHA-1 哈希一下就行了。」

**问题一：太快了**

现代 GPU 可以每秒计算数十亿次 MD5/SHA-1 哈希。一个 8 位字符的密码，字符集 96 个（大小写字母+数字+符号），组合数 `96^8 = 7.2 万亿`。用 GPU 破解，大约需要 **100 秒**。

**问题二：没有盐**

如果两个用户密码相同，MD5 哈希结果也相同。攻击者可以预计算所有常见密码的哈希值（彩虹表），直接查表破解。

**问题三：无法防止暴力破解**

哈希计算太快，暴力破解成本极低。

## 盐值（Salt）的作用

### 什么是盐？

盐是一个随机生成的字符串，和密码一起哈希：

```java
// 没有盐
hash("123456") = md5("123456")

// 有盐
salt = randomString()
hash("123456" + salt)
```

### 盐的作用

1. **相同密码产生不同哈希**：彩虹表失效
2. **增加破解难度**：攻击者必须为每个盐值单独计算
3. **不可预测**：盐值是随机的，攻击者无法预知

```java
// 两次存储相同密码，产生不同的哈希值
String salt1 = "a3f8c9d2";
String salt2 = "e7b4f1a8";
String hash1 = md5("123456" + salt1);  // ≠ hash2
String hash2 = md5("123456" + salt2);  // ≠ hash1
```

**但 MD5 + 盐仍然不够安全**，因为 MD5 太快了。

## BCrypt 算法原理

### 为什么选择 BCrypt？

BCrypt 是专门为密码设计的哈希算法，有以下特点：

| 特性 | 说明 |
|---|---|
| 内置盐 | 盐值自动生成并嵌入哈希结果中 |
| 可配置强度 | 通过 cost 参数控制迭代次数 |
| 计算速度适中 | 足够慢防止暴力破解，但不至于影响用户体验 |
| 防 GPU 加速 | 设计时就考虑了抵御硬件加速 |

### 工作原理

BCrypt 的哈希结果格式：

```
$2a$10$N9qo8uLOickgx2ZMRZoMyeIjZAgcfl7p92ldGxad68LJZdL17lhWy
 ↑↑  ↑↑ ↑
 |   |  └── 盐值（22字符）+ 哈希值（31字符）
 |   └───── Cost 参数（计算轮数 = 2^cost）
 └───────── 算法版本（2a）
```

**解读**：
- `$2a$`：BCrypt 算法标识
- `10`：cost 参数，计算轮数 `2^10 = 1024` 次
- `N9qo8uLOickgx2ZMRZoMye`：22 字符的盐值（Base64 编码）
- `IjZAgcfl7p92ldGxad68LJZdL17lhWy`：31 字符的哈希值

### 为什么内置盐是安全的？

BCrypt 的盐是**随机生成的**，并且**嵌入在哈希结果中**。验证密码时，从存储的哈希中提取盐值，然后重新计算哈希比对。

```java
// 存储时：密码 + 随机盐 → 哈希
String rawPassword = "123456";
String salt = BCrypt.gensalt();
String hashedPassword = BCrypt.hashpw(rawPassword, salt);
// 存储：hashedPassword（包含盐值）

// 验证时：输入密码 + 存储的盐 → 哈希 → 比对
String inputPassword = "123456";
String storedHash = "$2a$10$...";
boolean matches = BCrypt.checkpw(inputPassword, storedHash);
```

**每次调用 `gensalt()` 都会生成新的随机盐**，因此相同密码每次存储结果都不同。

## PasswordEncoder 接口

### 接口定义

```java
public interface PasswordEncoder {
    // 加密密码
    String encode(CharSequence rawPassword);

    // 验证密码
    boolean matches(CharSequence rawPassword, String encodedPassword);

    // 判断是否需要重新编码（用于升级旧密码）
    default boolean upgradeEncoding(String encodedPassword) {
        return false;
    }
}
```

### 常用实现

| 实现类 | 说明 | 安全性 |
|---|---|---|
| BCryptPasswordEncoder | BCrypt 算法，推荐使用 | 高 |
| Argon2PasswordEncoder | Argon2 算法，最新标准 | 最高 |
| SCryptPasswordEncoder | SCrypt 算法 | 高 |
| Pbkdf2PasswordEncoder | PBKDF2 算法 | 中高 |
| NoOpPasswordEncoder | 不加密，仅用于测试 | 无 |
| PlainPasswordEncoder | 明文，不加密 | 无 |

### BCryptPasswordEncoder 使用

```java
@Bean
public PasswordEncoder passwordEncoder() {
    // 方式一：默认强度（10）
    return new BCryptPasswordEncoder();

    // 方式二：指定强度（4-31）
    return new BCryptPasswordEncoder(12);

    // 方式三：使用 SecureRandom
    return new BCryptPasswordEncoder(12, new SecureRandom());
}
```

## 密码存储的完整流程

### 用户注册

```java
@Service
public class UserService {

    @Autowired
    private PasswordEncoder passwordEncoder;

    @Autowired
    private UserMapper userMapper;

    public void register(UserRegisterRequest request) {
        // 1. 检查用户名是否已存在
        if (userMapper.findByUsername(request.getUsername()) != null) {
            throw new UserAlreadyExistsException();
        }

        // 2. 加密密码
        String hashedPassword = passwordEncoder.encode(request.getPassword());

        // 3. 保存用户
        UserEntity user = new UserEntity();
        user.setUsername(request.getUsername());
        user.setPassword(hashedPassword);
        userMapper.insert(user);
    }
}
```

### 用户登录

```java
@Service
public class CustomUserDetailsService implements UserDetailsService {

    @Autowired
    private UserMapper userMapper;

    @Override
    public UserDetails loadUserByUsername(String username)
            throws UsernameNotFoundException {
        UserEntity user = userMapper.findByUsername(username);
        if (user == null) {
            throw new UsernameNotFoundException("用户不存在");
        }

        // 返回 UserDetails，密码字段是已加密的哈希值
        return User.withUsername(user.getUsername())
            .password(user.getPassword())
            .authorities("ROLE_USER")
            .build();
    }
}
```

**注意**：登录时**不需要手动调用 `passwordEncoder.matches()`**，`DaoAuthenticationProvider` 会自动校验。

## 密码强度与 Cost 参数

### Cost 参数的影响

BCrypt 的 cost 参数决定了计算复杂度：

| Cost | 迭代次数 | 大约耗时（单次哈希） |
|---|---|---|
| 10 | 1024 | ~50ms |
| 12 | 4096 | ~200ms |
| 14 | 16384 | ~800ms |
| 16 | 65536 | ~3s |

**推荐值**：12-14。太小不安全，太大影响用户体验。

### 为什么 cost `=` 10 合适？

- 50ms 的计算时间对用户登录没有感知影响
- 但对攻击者来说，每秒只能尝试 20 次，暴力破解成本极高
- 比 MD5（每秒亿次）慢了 **200 万倍**

## 密码迁移与升级

### 场景：旧系统使用 MD5，需要迁移

有些老系统用 MD5 存储密码，迁移到 BCrypt 时需要「渐进式升级」。

```java
@Bean
public PasswordEncoder passwordEncoder() {
    // 委托链：先尝试新算法，再尝试旧算法
    return PasswordEncoderFactories.createDelegatingPasswordEncoder();
}
```

`spring-security-core` 提供的 `DelegatingPasswordEncoder` 会根据哈希前缀判断算法：

```java
public class DelegatingPasswordEncoder implements PasswordEncoder {
    private Map<String, PasswordEncoder> encoders;
    private String defaultId;

    @Override
    public boolean matches(CharSequence rawPassword, String storedPassword) {
        // 从存储的密码中提取算法 ID
        String algorithmId = extractAlgorithmId(storedPassword);

        // 找到对应的编码器
        PasswordEncoder encoder = encoders.get(algorithmId);

        // 验证密码
        return encoder.matches(rawPassword, storedPassword);
    }

    @Override
    public String encode(CharSequence rawPassword) {
        // 使用默认算法（BCrypt）编码
        return defaultEncoder.encode(rawPassword);
    }
}
```

**存储格式**：`{bcrypt}$2a$10$...` 或 `{md5}5f4dcc3b5aa765d61d8327deb882cf99`

### 渐进式升级策略

用户下次登录时，如果旧密码校验通过，自动升级为新算法：

```java
public class CustomUserDetailsService implements UserDetailsService {

    private PasswordEncoder passwordEncoder;
    private UserMapper userMapper;

    @Override
    public UserDetails loadUserByUsername(String username) {
        UserEntity user = userMapper.findByUsername(username);

        // 检查是否需要升级
        if (needsRehash(user.getPassword())) {
            String newHash = passwordEncoder.encode(user.getPassword());
            userMapper.updatePassword(user.getId(), newHash);
            user.setPassword(newHash);
        }

        return User.withUsername(user.getUsername())
            .password(user.getPassword())
            .authorities("ROLE_USER")
            .build();
    }

    private boolean needsRehash(String password) {
        // BCrypt 不需要升级，其他算法需要
        return !password.startsWith("{bcrypt}");
    }
}
```

## 常见问题与解决方案

### 问题一：如何生成安全的随机盐？

BCrypt 的 `gensalt()` 已经内置了安全的随机盐生成，使用 `java.security.SecureRandom`。

**不要自己生成盐**，直接用 `BCrypt.hashpw(password, BCrypt.gensalt())`。

### 问题二：密码校验失败但密码是对的

检查点：
1. 数据库字段是否够长？BCrypt 哈希结果约 60 字符
2. 数据库字符集是否是 UTF-8？否则某些字符可能乱码
3. 是否有前后空格？`matches()` 会自动 trim，但最好前端就处理

### 问题三：忘记密码怎么办？

**不要告诉用户原密码**。正确做法是：
1. 生成随机临时密码，发送邮件/SMS
2. 或者发送重置链接（带 Token），用户点击后设置新密码

## 面试追问预判

### Q1：BCrypt 和 MD5/SHA 的区别是什么？

| 特性 | BCrypt | MD5/SHA |
|---|---|---|
| 速度 | 可配置，慢 | 快（不安全） |
| 内置盐 | 是 | 否 |
| GPU 加速防护 | 有 | 无 |
| 适用场景 | 密码存储 | 数字签名、文件校验 |

### Q2：BCrypt 的内置盐是怎么工作的？

BCrypt 的盐是**嵌入在哈希结果中的**，不是单独存储。格式为 `$2a$10$salt$hash`，验证时自动提取。

### Q3：为什么 BCrypt 的哈希结果每次都不同？

因为每次调用 `gensalt()` 或 `encode()` 都会生成新的随机盐。相同密码 + 不同盐 = 不同哈希，这是**正常且正确的**行为。

验证时，系统从存储的哈希中提取盐值，重新计算并比对，所以能正确验证。

## 总结

密码安全存储的核心是：

1. **不能用明文**：数据库泄露 = 密码泄露
2. **不能用快速哈希**：MD5/SHA 太快，GPU 破解无压力
3. **必须加盐**：防止彩虹表攻击
4. **BCrypt 是最佳选择**：内置盐、可配置强度、防 GPU

Spring Security 的 `PasswordEncoder` 接口，将密码加密的复杂性封装了起来。只需要：

```java
// 存储
String hashed = passwordEncoder.encode(rawPassword);

// 验证
boolean matches = passwordEncoder.matches(rawPassword, hashed);
```
