# Spring Security：企业级安全的完整方案

当你在招聘网站上搜索「Java 高级工程师」时，几乎每个职位描述都会写上一句「熟悉 Spring Security」。但 Spring Security 到底是什么？它能解决哪些问题？

一句话概括：**Spring Security 是一个功能强大、可高度定制的企业级安全框架，专注于身份认证（Authentication）和权限授权（Authorization）。**

但这只是官方说法的翻版。你真正需要知道的是：Spring Security 能做什么？它的核心机制是什么？为什么它能成为 Java 安全领域的事实标准？

这一章，就是来回答这些问题的。

## 为什么需要 Spring Security？

在没有 Spring Security 的年代，Java Web 应用的安全是怎么做的？

- 在 Filter 里手动检查 Session
- 在每个 Controller 里写 `if (user == null) return 401`
- 用拦截器做权限判断
- 用加密库手动处理密码

这些方案有什么问题？

1. **代码重复**：每个项目都要写一遍
2. **安全漏洞**：手写的安全逻辑容易有疏漏
3. **难以维护**：业务逻辑和安全逻辑混在一起
4. **无法统一**：不同项目安全策略不一致

Spring Security 解决了这些问题：把安全逻辑抽成框架，让业务代码专注于业务。

## Spring Security 能做什么？

### 认证（Authentication）

验证用户身份，确认「你是谁」。

| 认证方式 | 适用场景 |
|---|---|
| 表单登录 | 普通 Web 应用 |
| HTTP Basic | RESTful API、测试环境 |
| HTTP Digest | 很少使用 |
| Remember-Me | 免登录功能 |
| 短信登录 | 移动端应用 |
| 社交登录 | OAuth2（QQ、微信、GitHub） |
| CAS 单点登录 | 企业内部多系统 |
| LDAP | 企业内网 |
| SAML | 企业间单点登录 |

### 授权（Authorization）

判断用户权限，决定「你能做什么」。

| 授权方式 | 说明 |
|---|---|
| URL 权限 | 基于 URL 路径的访问控制 |
| 方法级权限 | `@PreAuthorize`、`@Secured` |
| 领域级安全 | 对象访问控制（ACL） |
| 按钮级权限 | 前端根据权限显示/隐藏按钮 |
| 数据权限 | 行级权限（只能看自己的数据） |

### 防护机制

| 机制 | 说明 |
|---|---|
| CSRF 防护 | 防止跨站请求伪造 |
| CORS | 跨域资源共享控制 |
| XSS 防护 | 跨站脚本攻击防护 |
| 点击劫持防护 | `X-Frame-Options` 响应头 |
| 会话固定攻击防护 | Session 换 ID |
| HTTP 安全响应头 | `X-XSS-Protection`、`Content-Security-Policy` |

### Core 模块

核心接口和类，包括 `Authentication`、`Authorization`、`GrantedAuthority` 等。这些是 Spring Security 的基础，所有其他模块都依赖它。

### Web 模块

处理 HTTP 请求的安全逻辑，基于 Servlet Filter 实现。`SecurityFilterChain` 定义了过滤器链的组成，`FilterChainProxy` 是过滤器链的入口。

### Config 模块

配置相关的类，包括 `@EnableWebSecurity`、`SecurityFilterChain` 的配置等。

### Remember-Me 模块

「记住我」功能，支持基于 Token 的自动登录。

### LDAP 模块

集成 LDAP 目录服务，用于企业内网认证。

### OAuth2 模块

支持 OAuth2 登录（社交登录）、OAuth2 授权服务器。

### ACL 模块

Access Control List，领域级访问控制，可以精细控制到每个对象的访问权限。

### Method 模块

方法级安全，支持 `@PreAuthorize`、`@Secured` 等注解。

## Spring Security 的版本演进

### Spring Security 1.x / 2.x

早期版本，功能相对简单，以命名空间配置为主。

### Spring Security 3.x

支持 Spring 3，引入了注解驱动的安全配置。

### Spring Security 4.x

这是最经典、使用最广泛的版本。支持 Java Config，引入了 `WebSecurityConfigurerAdapter`。

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig extends WebSecurityConfigurerAdapter {

    @Override
    protected void configure(HttpSecurity http) throws Exception {
        http
            .authorizeRequests()
                .anyRequest().authenticated()
                .and()
            .formLogin()
                .and()
            .httpBasic();
    }
}
```

### Spring Security 5.x

引入 OAuth2 客户端支持，`WebSecurityConfigurerAdapter` 被标记为 `@Deprecated`。

### Spring Security 6.x（当前）

`WebSecurityConfigurerAdapter` 完全移除，`Lambda DSL` 成为唯一推荐配置方式。CSRF 默认启用不可关闭。

## Spring Security 在 Java 安全生态中的位置

Spring Security 不是孤立存在的，它与其他技术协同工作：

```
┌─────────────────────────────────────────┐
│              Spring Security              │
│         Authentication + Authorization    │
└───────────────────┬─────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────┐
│          Spring Framework                │
│  IoC / AOP / Transaction / Web MVC      │
└───────────────────┬─────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────┐
│             Servlet 容器                  │
│           Tomcat / Jetty                  │
└─────────────────────────────────────────┘
```

### 与 Spring Session 的关系

Spring Security 管理认证和授权，Spring Session 管理 Session 存储。

| 场景 | Session 存储 | 适用 |
|---|---|---|
| 单机部署 | HttpSession（内存） | 开发环境 |
| 集群部署 | Spring Session + Redis | 生产环境 |
| 无状态 | JWT Token | RESTful API |

### 与 Spring Cloud Gateway 的关系

微服务架构下，Gateway 做统一鉴权，Spring Security 负责服务内部的安全控制。

```
用户请求
    ↓
Gateway（统一鉴权 + JWT 校验）
    ↓
内部服务（Spring Security：细粒度权限控制）
```

## 面试核心问题

### Q1：Spring Security 的核心原理是什么？

Spring Security 的核心是基于 Servlet Filter 的过滤器链。所有请求在到达 Controller 之前，都要经过这串过滤器。认证过滤器负责识别用户身份，授权过滤器负责判断是否有权限访问。

### Q2：Spring Security 和 Shiro 的区别？

| 维度 | Spring Security | Apache Shiro |
|---|---|---|
| 出身 | Spring 官方 | Apache 基金会 |
| 依赖 | 必须配合 Spring 使用 | 独立框架 |
| 权限模型 | RBAC + ACL | RBAC + Realm |
| 学习曲线 | 陡峭 | 平缓 |
| 功能完整性 | 更完整 | 较完整 |
| 社区活跃度 | 非常活跃 | 一般 |

### Q3：Spring Security 如何自定义认证逻辑？

1. 实现 `UserDetailsService` 接口，自定义用户加载逻辑
2. 实现 `AuthenticationProvider` 接口，自定义认证逻辑
3. 注册为 Spring Bean，注入到 Security 配置中

## 为什么选择 Spring Security？

1. **Spring 官方维护**：背靠 Spring 生态，质量有保证
2. **功能完善**：认证、授权、防护、加密……开箱即用
3. **高度可定制**：几乎每个组件都可以替换
4. **社区活跃**：遇到问题容易找到解决方案
5. **持续更新**：跟随 Spring 框架迭代，不断引入新特性

但 Spring Security 也有缺点：

- **学习曲线陡峭**：概念多，配置复杂
- **文档不够友好**：官方文档组织混乱
- **升级风险大**：版本之间 Breaking Changes 较多

这一系列文档，就是要帮你跨越这些障碍，真正掌握 Spring Security。
