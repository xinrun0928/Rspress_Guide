# Spring Security

你知道吗？每年因为权限控制不当导致的数据泄露事件中，有相当一部分是因为没有正确配置安全框架。

Spring Security 是 Spring 生态中最核心的安全框架，它不仅仅是「登录功能」那么简单——它解决的是 **「谁可以做什么」** 这个最根本的问题。

从表单登录到 OAuth2，从 Session 会话到 JWT 无状态认证，从 CSRF 防护到细粒度权限控制——Spring Security 几乎涵盖了企业级应用安全的方方面面。

但正因为它功能强大、配置复杂，很多人对它是「既爱又恨」：爱它的功能完善，恨它的学习曲线陡峭。

这个模块，就是要把 Spring Security 讲透。

## 版本差异

| 版本 | 重大变化 | 说明 |
|------|---------|------|
| 5.x | 密码编码器重构 | 推出 `PasswordEncoder` 接口 |
| 6.0 | 移除 WebSecurityConfigurerAdapter | 全面转向 Lambda DSL |
| 6.1+ | 默认禁用 CSRF 表单登录 | 更安全的默认配置 |

> Spring Security 6.x 有大量 breaking changes，如果你在维护旧项目，建议先看 [6.x 新特性](/framework/springsecurity/v6)。

## 核心公式

```yaml
# Spring Security 核心公式

# 1. 过滤器链执行顺序
顺序:
  1. SecurityContextPersistenceFilter → 上下文持久化
  2. WebAsyncManagerIntegrationFilter → 异步上下文
  3. SecurityContextHolderAwareRequestFilter → 请求包装
  4. HeaderWriterFilter → 响应头
  5. CorsFilter → 跨域
  6. CsrfFilter → CSRF防护
  7. LogoutFilter → 登出
  8. UsernamePasswordAuthenticationFilter → 账号密码登录
  9. RequestCacheAwareFilter → 请求缓存
  10. SecurityContextHolderAwareRequestFilter → 授权
  11. AnonymousAuthenticationFilter → 匿名认证
  12. SessionManagementFilter → 会话管理
  13. ExceptionTranslationFilter → 异常处理
  14. FilterSecurityInterceptor → 最终拦截

# 2. 认证流程
流程:
  请求 → UsernamePasswordAuthenticationFilter
       → Authentication (username + password)
         → AuthenticationManager.authenticate()
           → ProviderManager.authenticate()
             → AuthenticationProvider实现认证
               → UserDetailsService.loadUserByUsername()
               → PasswordEncoder.matches()
             → 返回 Authentication（已认证）
         → SecurityContextHolder.getContext().setAuthentication()

# 3. 授权注解对比
注解:
  @Secured: 简单角色检查，不支持SpEL
  @RolesAllowed: @Secured的替代，支持JDK标准
  @PreAuthorize: 方法前检查，支持SpEL
  @PostAuthorize: 方法后检查，支持SpEL

  表达式:
    hasRole('ADMIN') → ROLE_ADMIN
    hasAuthority('user:read')
    permitAll() / denyAll()

# 4. Session vs JWT
Session:
  → 服务端存储，状态管理
  → 支持会话保持、Remember-Me
  → 分布式需Session共享

JWT:
  → 无状态，Token自包含用户信息
  → 可扩展，支持跨域
  → 适合RESTful API
  → 需处理Token刷新、黑名单

# 5. OAuth2 四种授权模式
模式:
  授权码(Authorization Code): Web服务端，最安全
  隐式(Implicit): SPA，不推荐（已被废弃）
  密码凭证(Resource Owner): 信任的应用
  客户端凭证(Client Credentials): 服务间调用

# 6. 密码加密
加密:
  旧: BCryptPasswordEncoder（推荐）
      → 工作因子（strength）10-31
      → 每次生成不同的盐值
      → 自动验证
  新: DelegatingPasswordEncoder（推荐）
      → 支持多算法兼容
      → {bcrypt}xxx 前缀标识算法
```
