# JWT 与无状态认证：Spring Security 的无状态化改造

传统的 Spring Security 使用 Session 存储认证信息。但 Session 有状态——需要 Session 存储介质（内存、Redis），需要 Session 同步（分布式部署时）。

JWT（JSON Web Token）解决了这个问题。

---

## Session 认证 vs JWT 认证

### Session 认证

```
用户登录
    ↓
服务端创建 Session，Session ID 返回给客户端（Cookie）
    ↓
后续请求带上 Session ID
    ↓
服务端根据 Session ID 查找 Session
    ↓
获取认证信息
```

**问题**：
- 需要 Session 存储介质
- 分布式部署时需要 Session 同步
- 扩展性差

### JWT 认证

```
用户登录
    ↓
服务端生成 JWT（包含用户信息 + 签名）
    ↓
JWT 返回给客户端（通常放在 Authorization 头）
    ↓
后续请求带上 JWT
    ↓
服务端验证 JWT 签名（无需存储）
    ↓
解析 JWT 获取用户信息
```

**优势**：
- 无状态，服务端无需存储
- 天然适合分布式
- 扩展性好

## JWT 的结构

### 三部分组成

```
Header.Payload.Signature
```

### Header（头部）

```json
{
  "alg": "HS256",
  "typ": "JWT"
}
```

### Payload（载荷）

```json
{
  "sub": "1234567890",
  "name": "John Doe",
  "iat": 1516239022,
  "exp": 1516242622
}
```

### Signature（签名）

```
HMACSHA256(
  base64UrlEncode(header) + "." +
  base64UrlEncode(payload),
  secret
)
```

## 为什么选择 JWT？

### JWT 的优势

| 优势 | 说明 |
|---|---|
| 无状态 | 服务端不需要存储 Token |
| 可扩展 | 适合分布式和微服务 |
| 跨域 | 可以在多个域名间共享 |
| 性能 | 减少数据库查询 |

### JWT 的劣势

| 劣势 | 说明 |
|---|---|
| 无法撤销 | Token 过期前无法主动失效 |
| 存储限制 | Payload 大小有限制 |
| 敏感信息 | Payload 是 Base64 编码，可被解码 |

## JWT 与 Spring Security

Spring Security 可以与 JWT 配合，实现无状态认证：

```
请求 → JWT 过滤器 → 验证 Token → 设置 SecurityContext → 业务处理
```

### 核心组件

| 组件 | 说明 |
|---|---|
| JwtTokenProvider | Token 生成和验证 |
| JwtAuthenticationFilter | 拦截请求，提取 Token，验证，设置 SecurityContext |
| JwtAuthenticationToken | 封装认证信息 |

## 面试核心问题

### Q1：JWT 为什么能防篡改？

JWT 的 Signature 部分使用签名算法（如 HMAC-SHA256）生成。如果篡改了 Payload，重新签名需要密钥，服务端验证签名时会发现不一致。

### Q2：JWT 和 Session 哪个更安全？

两者安全性取决于具体实现。JWT 的优势在于无状态，Session 的优势在于可以主动撤销。选择取决于业务场景。

### Q3：如何让 JWT Token 失效？

由于 JWT 无状态，无法直接让 Token 失效。解决方案：
- 使用短期 Token
- 使用 Refresh Token + 黑名单
- 维护 Token 版本号
