# 授权与权限：Spring Security 的权限决策引擎

用户登录成功，只能说明身份验证通过了。但这不意味着用户可以访问所有资源——管理员可以管理用户，财务可以查看报表，普通员工只能查看自己的考勤。

这就是授权（Authorization）要解决的问题：**认证回答「你是谁」，授权回答「你能做什么」。**

---

## 授权的本质

Spring Security 的授权机制，核心是回答一个问题：

> **当前用户有权访问这个资源吗？**

这个问题的答案，取决于三个因素：

1. **用户是谁**：`Authentication` 中的 `principal`
2. **用户有什么权限**：`Authentication` 中的 `GrantedAuthority` 列表
3. **资源需要什么权限**：访问目标配置的 `ConfigAttribute`

```
┌─────────────────────────────────────────────────────────────┐
│                    权限决策三要素                             │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  Authentication（用户）                                       │
│  ├── principal: zhangsan                                     │
│  └── authorities: [ROLE_ADMIN, ROLE_USER]                   │
│                                                              │
│                    ↓ 比对 ↓                                   │
│                                                              │
│  ConfigAttribute（资源需求）                                  │
│  ├── /admin/** → ROLE_ADMIN                                 │
│  └── /user/** → ROLE_USER                                    │
│                                                              │
│                    ↓ 决策 ↓                                   │
│                                                              │
│  AccessDecisionManager                                        │
│  └── 投票决定：通过 / 拒绝                                     │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

## 授权的两种级别

### URL 级别授权

基于 URL 路径的权限控制，是最常用的一种方式。

```java
http.authorizeHttpRequests(auth -> auth
    .requestMatchers("/admin/**").hasRole("ADMIN")
    .requestMatchers("/user/**").hasAnyRole("USER", "ADMIN")
    .requestMatchers("/public/**").permitAll()
    .anyRequest().authenticated()
)
```

优点：简单直观，配置在 Security 配置中
缺点：粒度较粗，无法控制到按钮级别

### 方法级别授权

基于注解的权限控制，可以精确控制到方法。

```java
@PreAuthorize("hasRole('ADMIN')")
public void deleteUser(Long userId) {
    // 只有 ADMIN 才能删除用户
}
```

优点：粒度细，可以控制到具体操作
缺点：需要在每个方法上加注解

## 授权的核心组件

### AccessDecisionManager

权限决策的核心，投票决定是否放行。

```java
public interface AccessDecisionManager {
    void decide(Authentication authentication,
                 object object,
                 Collection<ConfigAttribute> configAttributes)
        throws AccessDeniedException, InsufficientAuthenticationException;

    boolean supports(ConfigAttribute attribute);

    boolean supports(Class<?> clazz);
}
```

### AccessDecisionVoter

投票器，每个投票器独立判断，给出投票结果。

```java
public interface AccessDecisionVoter<S> {
    int ACCESS_GRANTED = 1;    // 赞成
    int ACCESS_DENIED = -1;    // 反对
    int ACCESS_ABSTAIN = 0;    // 弃权

    int vote(Authentication authentication,
             S object,
             Collection<ConfigAttribute> attributes);
}
```

### 三种决策策略

| 策略 | 决策规则 |
|---|---|
| `AffirmativeBased` | 任一投票器赞成即通过（默认） |
| `ConsensusBased` | 多数票决定 |
| `UnanimousBased` | 全票赞成才通过 |

## Authority 与 Role

Spring Security 用 `GrantedAuthority` 表示权限，`Role` 是其中最常用的一种。

### 两者的关系

```
GrantedAuthority（接口）
    ↑
    └── SimpleGrantedAuthority
    └── RoleHierarchyVoter$RoleHierarchyImpl
    └── ...其他实现

Role（概念）
    └── 实际上是 GrantedAuthority 的一种
    └── 命名约定：ROLE_ 前缀
```

### `ROLE_` 前缀的约定

Spring Security 有两个辅助方法：

| 方法 | 说明 |
|---|---|
| `hasRole('ADMIN')` | 检查是否有 `ROLE_ADMIN` 权限 |
| `hasAuthority('ADMIN')` | 检查是否有 `ADMIN` 权限 |

```java
.hasRole("ADMIN")    // 自动加上 ROLE_ 前缀，检查 ROLE_ADMIN
.hasAuthority("ADMIN")  // 不加前缀，检查 ADMIN
```

## 权限模型

### RBAC 模型

Role-Based Access Control，基于角色的访问控制。

```
┌─────────────────────────────────────────────────────────────┐
│                    RBAC 权限模型                             │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  用户 ────── 关联 ────── 角色 ────── 关联 ────── 权限         │
│                                                              │
│  user          user_role       role          role_permission │
│  ├── id        ├── user_id     ├── id        ├── role_id    │
│  ├── name      └── role_id     └── name      └── perm_id     │
│  └── ...                       └── ...      └── perm_id     │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 四种 RBAC 级别

| 级别 | 说明 |
|---|---|
| RBAC0 | 最基础的用户-角色-权限 |
| RBAC1 | 支持角色继承（A 继承 B，A 有 B 的所有权限） |
| RBAC2 | 支持角色互斥、基数约束 |
| RBAC3 | RBAC1 + RBAC2 |

## 面试核心问题

### Q1：Spring Security 的授权流程是什么？

请求到达 `FilterSecurityInterceptor` 后，获取当前用户的 `Authentication`，获取访问资源需要的 `ConfigAttribute`，然后交给 `AccessDecisionManager` 投票决定。投票器根据 `GrantedAuthority` 和 `ConfigAttribute` 判断是否匹配。

### Q2：Authority 和 Role 的区别？

两者都是 `GrantedAuthority` 的实现。没有本质区别，只有命名约定的不同：Role 用 `ROLE_` 前缀，Authority 不加前缀。`hasRole()` 会自动加 `ROLE_` 前缀，`hasAuthority()` 不加。

### Q3：`@PreAuthorize` 为什么不生效？

可能原因：没有添加 `@EnableMethodSecurity` 注解，或者在 `@PreAuthorize` 的 SpEL 表达式中使用了 `hasRole()` 但方法没有加 `@EnableMethodSecurity(prePostEnabled = true)`。

- 权限可以精确到按钮级别（增删改查），怎么设计？
- 数据权限（只能看自己的数据）怎么实现？
