# Authority 与 Role：Spring Security 权限模型的两个维度

`hasRole('ADMIN')` 和 `hasAuthority('ROLE_ADMIN')`，到底有什么区别？

这是 Spring Security 中最容易被混淆的问题之一。很多人会用错，导致权限校验失败。

## 问题的起源

Spring Security 的权限系统有两个核心接口：

- **Authority**：权限
- **Role**：角色

但实际上，它们都实现了同一个接口——`GrantedAuthority`。没有本质区别，只有命名约定的不同。

---

## GrantedAuthority 接口

```java
public interface GrantedAuthority extends Serializable {
    String getAuthority();
}
```

这是 Spring Security 权限系统的基石。所有权限，无论是角色还是普通权限，都通过这个接口表示。

### 唯一的抽象方法

```java
String getAuthority();
```

返回权限字符串。Spring Security 用这个字符串来比较权限。

---

## Authority 与 Role 的本质

### 没有本质区别

```java
// 角色
new SimpleGrantedAuthority("ROLE_ADMIN")

// 权限
new SimpleGrantedAuthority("user:read")
new SimpleGrantedAuthority("system:config")
```

两者都是 `SimpleGrantedAuthority` 的实例，没有任何区别。

### 区别在于语义和命名约定

| 维度 | Authority | Role |
|---|---|---|
| 语义 | 具体的权限（如 `user:read`） | 角色（如 `ADMIN`） |
| 前缀 | 不加前缀 | 加 `ROLE_` 前缀 |
| 粒度 | 细粒度 | 粗粒度 |

### 命名约定

**Role 必须加 `ROLE_` 前缀**：

```java
// 错误
SimpleGrantedAuthority("ADMIN")  // 这个权限叫 "ADMIN"

// 正确
SimpleGrantedAuthority("ROLE_ADMIN")  // 这个角色叫 "ADMIN"
```

**Authority 通常不加前缀**：

```java
SimpleGrantedAuthority("user:read")
SimpleGrantedAuthority("user:write")
SimpleGrantedAuthority("system:config")
```

## hasRole vs hasAuthority

### 方法签名

```java
// 内部自动加上 ROLE_ 前缀
boolean hasRole(String role)

// 不加前缀
boolean hasAuthority(String authority)
```

### 使用示例

```java
// 配置 URL 权限
http.authorizeHttpRequests(auth -> auth
    // 用 hasRole：传入 "ADMIN"，实际检查 "ROLE_ADMIN"
    .requestMatchers("/admin/**").hasRole("ADMIN")
    // 用 hasAuthority：传入 "user:read"，实际检查 "user:read"
    .requestMatchers("/user/**").hasAuthority("user:read")
)
```

```java
// 方法级权限
@PreAuthorize("hasRole('ADMIN')")           // 检查 ROLE_ADMIN
@PreAuthorize("hasAuthority('user:read')")  // 检查 user:read
```

### 混用的坑

```java
// 错误用法
.hasRole("ROLE_ADMIN")  // 实际检查 "ROLE_ROLE_ADMIN"，永远不匹配

// 正确用法
.hasRole("ADMIN")  // 实际检查 "ROLE_ADMIN"
```

## 数据库中的存储设计

### 方案一：Role 模式

```sql
-- 用户表
CREATE TABLE user (
    id BIGINT PRIMARY KEY,
    username VARCHAR(50),
    password VARCHAR(100)
);

-- 角色表
CREATE TABLE role (
    id BIGINT PRIMARY KEY,
    name VARCHAR(50)  -- 存储 "ADMIN", "USER", "MANAGER"
);

-- 用户角色关联表
CREATE TABLE user_role (
    user_id BIGINT,
    role_id BIGINT
);
```

权限存储时不带 `ROLE_` 前缀：

```java
// 加载权限时
List<GrantedAuthority> authorities = roles.stream()
    .map(role -> new SimpleGrantedAuthority("ROLE_" + role.getName()))
    .collect(Collectors.toList());
```

### 方案二：纯 Authority 模式

```sql
-- 用户表
CREATE TABLE user (
    id BIGINT PRIMARY KEY,
    username VARCHAR(50),
    password VARCHAR(100)
);

-- 权限表
CREATE TABLE permission (
    id BIGINT PRIMARY KEY,
    name VARCHAR(50)  -- 存储 "user:read", "user:write", "system:config"
);

-- 用户权限关联表
CREATE TABLE user_permission (
    user_id BIGINT,
    permission_id BIGINT
);
```

权限直接存储，不加前缀：

```java
// 加载权限时
List<GrantedAuthority> authorities = permissions.stream()
    .map(p -> new SimpleGrantedAuthority(p.getName()))
    .collect(Collectors.toList());
```

### 方案三：RBAC 混合模式（推荐）

```sql
-- 用户表
CREATE TABLE user (
    id BIGINT PRIMARY KEY,
    username VARCHAR(50),
    password VARCHAR(100)
);

-- 角色表
CREATE TABLE role (
    id BIGINT PRIMARY KEY,
    name VARCHAR(50),          -- "ADMIN", "USER"
    description VARCHAR(200)
);

-- 权限表
CREATE TABLE permission (
    id BIGINT PRIMARY KEY,
    name VARCHAR(50),          -- "user:read", "user:write"
    description VARCHAR(200)
);

-- 角色权限关联表
CREATE TABLE role_permission (
    role_id BIGINT,
    permission_id BIGINT
);

-- 用户角色关联表
CREATE TABLE user_role (
    user_id BIGINT,
    role_id BIGINT
);
```

用户拥有角色，角色拥有权限：

```java
// 加载权限时：用户角色 + 角色权限
Set<String> authorities = new HashSet<>();

// 用户的角色
user.getRoles().forEach(role -> {
    authorities.add("ROLE_" + role.getName());  // 角色
    role.getPermissions().forEach(p -> {
        authorities.add(p.getName());  // 权限
    });
});
```

## 权限的继承

### Role Hierarchy（角色层级）

如果角色有继承关系，可以用 `RoleHierarchyVoter` 实现：

```java
@Bean
public RoleHierarchyVoter roleHierarchyVoter() {
    return new RoleHierarchyVoter(roleHierarchy());
}

@Bean
public RoleHierarchyImpl roleHierarchy() {
    RoleHierarchyImpl hierarchy = new RoleHierarchyImpl();
    // ADMIN 继承 USER 的所有权限
    hierarchy.setHierarchy("ROLE_ADMIN > ROLE_USER");
    return hierarchy;
}
```

### 多级继承

```java
hierarchy.setHierarchy(
    "ROLE_SUPER_ADMIN > ROLE_ADMIN > ROLE_MANAGER > ROLE_USER"
);
```

这样 `ROLE_SUPER_ADMIN` 拥有所有角色的权限。

### 自定义角色层级

```java
@Bean
public RoleHierarchyImpl roleHierarchy() {
    RoleHierarchyImpl hierarchy = new RoleHierarchyImpl();
    // 从数据库或配置文件加载层级关系
    hierarchy.setHierarchy("ROLE_ADMIN > ROLE_USER");
    return hierarchy;
}
```

## 常见用法对比

### 简单场景：用 Role

用户只有角色，没有细粒度权限：

```java
// Security 配置
.requestMatchers("/admin/**").hasRole("ADMIN")
.requestMatchers("/user/**").hasRole("USER")

// UserDetailsService
UserDetails user = User.withUsername("zhangsan")
    .password(passwordEncoder.encode("123456"))
    .roles("USER", "MANAGER")  // 直接用 roles()，自动加 ROLE_ 前缀
    .build();
```

### 复杂场景：用 Authority

需要细粒度权限控制：

```java
// Security 配置
.requestMatchers(HttpMethod.GET, "/api/users").hasAuthority("user:read")
.requestMatchers(HttpMethod.POST, "/api/users").hasAuthority("user:create")
.requestMatchers(HttpMethod.PUT, "/api/users/**").hasAuthority("user:update")
.requestMatchers(HttpMethod.DELETE, "/api/users/**").hasAuthority("user:delete")

// UserDetailsService
List<GrantedAuthority> authorities = permissions.stream()
    .map(p -> new SimpleGrantedAuthority(p.getName()))
    .collect(Collectors.toList());

UserDetails user = User.withUsername("zhangsan")
    .password(passwordEncoder.encode("123456"))
    .authorities(authorities)  // 用 authorities()
    .build();
```

### hasAnyRole / hasAnyAuthority

多条件判断：

```java
// 任一角色匹配即可
.hasAnyRole("ADMIN", "MANAGER")

// 任一权限匹配即可
.hasAnyAuthority("user:read", "user:write")
```

## 面试追问预判

### Q1：Authority 和 Role 哪个性能更好？

没有性能差异，两者底层都是 `SimpleGrantedAuthority`。选择取决于业务场景：角色够用就用 Role，需要细粒度控制就用 Authority。

### Q2：为什么 Role 要加 `ROLE_` 前缀？

主要是为了避免和 Authority 混淆。如果不加前缀，`hasRole('user:read')` 和 `hasAuthority('user:read')` 会产生歧义——它们检查的是同一个权限，但语义不同。

### Q3：如何动态修改用户权限？

权限信息存在 `Authentication` 的 `authorities` 中。如果需要动态修改：

```java
// 创建新的 Authentication
UsernamePasswordAuthenticationToken newAuth =
    new UsernamePasswordAuthenticationToken(
        authentication.getPrincipal(),
        authentication.getCredentials(),
        newAuthorities  // 新的权限列表
    );

// 更新 SecurityContext
SecurityContextHolder.getContext().setAuthentication(newAuth);
```

## 总结

Authority 和 Role 没有本质区别，都是 `GrantedAuthority` 的实现。区别在于：

1. **语义**：`ROLE_` 前缀用于角色，不带前缀用于细粒度权限
2. **方法**：`hasRole()` 会自动加 `ROLE_`，`hasAuthority()` 不会
3. **存储**：数据库中可以只存角色名，加载时自动加前缀；也可以直接存储带前缀的权限
