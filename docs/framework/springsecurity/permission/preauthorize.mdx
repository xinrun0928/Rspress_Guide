# @PreAuthorize：方法级别的权限注解

`@PreAuthorize` 和 `@Secured`，到底谁更强？

这是方法级权限控制的两大主力注解。它们各有特点，但 `@PreAuthorize` 更加强大和灵活。

## 为什么需要方法级权限控制？

URL 级别的权限控制，只能控制到路径级别。但有时候，我们需要更细粒度的控制：

```java
// 路径 /admin/users 可能是管理员才能访问
// 但删除用户的操作，可能只有超级管理员才能执行

@DeleteMapping("/admin/users/{id}")
public void deleteUser(Long id) {
    // 如果只配 URL 权限，这个方法谁都能访问
    // 需要方法级权限控制
}
```

方法级注解可以精确控制到每个方法。

## @PreAuthorize 与 @PostAuthorize

### @PreAuthorize

方法执行**之前**进行权限检查。如果权限不足，直接拒绝执行。

```java
@PreAuthorize("hasRole('ADMIN')")
public void deleteUser(Long userId) {
    // 只有 ADMIN 才能执行
}
```

### @PostAuthorize

方法执行**之后**进行权限检查。可以根据返回值决定是否放行。

```java
@PostAuthorize("returnObject.owner == authentication.name")
public User getUser(Long userId) {
    // 只有返回对象的 owner 等于当前用户名，才能看到结果
}
```

**使用场景**：修改返回结果，或者需要在方法执行后才知道是否有权访问的场景。

## SpEL 表达式详解

`@PreAuthorize` 支持强大的 SpEL 表达式：

### 基本表达式

```java
@PreAuthorize("hasRole('ADMIN')")
@PreAuthorize("hasAuthority('user:delete')")
@PreAuthorize("hasAnyRole('ADMIN', 'USER')")
```

### 参数引用

```java
// 使用 # 参数名 引用方法参数
@PreAuthorize("#userId == authentication.principal.userId")
public User getUser(Long userId) {
    // 只能查看自己的信息
}

@PreAuthorize("#amount <= 10000")
public void transfer(BigDecimal amount, String to) {
    // 转账金额不能超过 10000
}
```

### 组合条件

```java
// AND 条件
@PreAuthorize("hasRole('ADMIN') and hasAuthority('user:delete')")

// OR 条件
@PreAuthorize("hasRole('ADMIN') or hasRole('SUPER_ADMIN')")

// NOT 条件
@PreAuthorize("!hasRole('GUEST')")
```

### 方法调用

```java
// 调用 Spring Bean 的方法
@PreAuthorize("@permissionService.hasPermission(#userId, 'user:delete')")
public void deleteUser(Long userId) {
    // 通过 Bean 方法判断权限
}
```

```java
@Component("permissionService")
public class PermissionService {
    public boolean hasPermission(Long userId, String permission) {
        // 自定义权限判断逻辑
        return userRepository.hasPermission(userId, permission);
    }
}
```

### 变量注入

```java
// 注入当前 Authentication
@PreAuthorize("authentication.name == 'admin'")

// 注入当前用户信息
@PreAuthorize("principal.username == 'admin'")

// 注入当前请求
@PreAuthorize("request.method == 'GET'")
```

## 开启方法级安全

使用 `@PreAuthorize` 之前，必须先开启方法级安全：

```java
@Configuration
@EnableWebSecurity
@EnableMethodSecurity(
    prePostEnabled = true,   // 启用 @PreAuthorize 和 @PostAuthorize
    securedEnabled = true,   // 启用 @Secured
    jsr250Enabled = true     // 启用 @RolesAllowed 等 JSR-250 注解
)
public class SecurityConfig {
    // ...
}
```

### 参数说明

| 参数 | 说明 | 默认值 |
|---|---|---|
| `prePostEnabled` | 启用 `@PreAuthorize` / `@PostAuthorize` | `true` |
| `securedEnabled` | 启用 `@Secured` | `false` |
| `jsr250Enabled` | 启用 JSR-250 注解 | `false` |

## @Secured 与 @RolesAllowed

### @Secured

Spring 2.x 引入的注解，不支持 SpEL：

```java
@Secured({"ROLE_ADMIN", "ROLE_USER"})
public void doSomething() {
    // 任一角色即可访问
}
```

### @RolesAllowed

JSR-250 标准注解，与 `@Secured` 类似：

```java
@RolesAllowed({"ROLE_ADMIN", "ROLE_USER"})
public void doSomething() {
    // 任一角色即可访问
}
```

### 三者对比

| 特性 | @PreAuthorize | @Secured | @RolesAllowed |
|---|---|---|---|
| SpEL 支持 | 是 | 否 | 否 |
| 条件组合 | 丰富 | 简单 | 简单 |
| 方法参数引用 | 是 | 否 | 否 |
| 返回值引用 | `@PostAuthorize` | 否 | 否 |
| 标准 | Spring | Spring | JSR-250 |

## 自定义安全表达式

### 实现 SecurityExpressionRoot

```java
public class CustomMethodSecurityExpressionRoot
        extends MethodSecurityExpressionOperations {

    private final FilterInvocation invocation;

    public CustomMethodSecurityExpressionRoot(
            Authentication authentication,
            FilterInvocation invocation) {
        this.setAuthentication(authentication);
        this.invocation = invocation;
    }

    // 自定义权限方法
    public boolean hasPermission(Object target, Object permission) {
        return permissionService.hasPermission(
            authentication, target, permission);
    }

    public boolean isMe(Long userId) {
        return authentication.getName()
            .equals(userRepository.findById(userId).getUsername());
    }
}
```

### 实现 MethodSecurityExpressionHandler

```java
@Component
public class CustomMethodSecurityExpressionHandler
        extends DefaultMethodSecurityExpressionHandler {

    @Autowired
    private PermissionService permissionService;

    @Override
    protected MethodSecurityExpressionOperations createSecurityExpressionRoot(
            Authentication authentication,
            MethodInvocation invocation) {
        CustomMethodSecurityExpressionRoot root =
            new CustomMethodSecurityExpressionRoot(
                authentication,
                new FilterInvocation(
                    request, response));
        root.setPermissionService(permissionService);
        root.setTrustResolver(trustResolver);
        root.setRoleHierarchy(roleHierarchy);
        root.setDefaultRolePrefix(defaultRolePrefix);
        return root;
    }
}
```

### 使用自定义表达式

```java
@PreAuthorize("@customSecurity.isMe(#userId)")
public User getUser(Long userId) {
    // 只有当前用户才能访问自己的信息
}

@PreAuthorize("hasPermission(#document, 'read')")
public Document getDocument(Document document) {
    // 检查对文档的读权限
}
```

## 方法级安全的执行流程

### AOP 拦截

```
┌─────────────────────────────────────────────────────────────┐
│                  方法级安全执行流程                           │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  方法调用                                                    │
│       ↓                                                      │
│  AOP 拦截（MethodSecurityInterceptor）                        │
│       ↓                                                      │
│  获取方法上的 @PreAuthorize 注解                              │
│       ↓                                                      │
│  解析 SpEL 表达式                                            │
│       ↓                                                      │
│  评估表达式（调用 Authentication.getAuthorities()）            │
│       ↓                                                      │
│  结果为 true → 执行方法 → 返回结果                            │
│  结果为 false → 抛出 AccessDeniedException → 拒绝访问          │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### MethodSecurityInterceptor

```java
public class MethodSecurityInterceptor implements MethodInterceptor {

    private MethodSecurityExpressionHandler expressionHandler;

    public Object invoke(MethodInvocation invocation) throws Throwable {
        // 1. 获取方法上的安全注解
        Collection<ConfigAttribute> attributes =
            SecurityAnnotationUtils.getAttributes(invocation.getMethod());

        // 2. 如果没有安全注解，直接执行
        if (attributes.isEmpty()) {
            return invocation.proceed();
        }

        // 3. 获取当前 Authentication
        Authentication authenticated = SecurityContextHolder
            .getContext().getAuthentication();

        // 4. 评估权限
        try {
            this.accessDecisionManager.decide(
                authenticated, invocation, attributes);
        } catch (AccessDeniedException ex) {
            throw ex;  // 拒绝访问
        }

        // 5. 执行方法
        return invocation.proceed();
    }
}
```

## @PreAuthorize 与 URL 权限的关系

### 执行顺序

```
请求 → FilterSecurityInterceptor（URL 权限）→ Controller 方法
                                          ↓
                              MethodSecurityInterceptor（@PreAuthorize）
                                          ↓
                                          Service 方法
```

### 两者都配置时的关系

URL 权限和 `@PreAuthorize` 是**叠加关系**，不是替代关系：

```java
// URL 配置：只有 ADMIN 能访问 /admin/**
.requestMatchers("/admin/**").hasRole("ADMIN")

// 方法配置：只有 SUPER_ADMIN 能执行 deleteUser
@PreAuthorize("hasRole('SUPER_ADMIN')")
@DeleteMapping("/admin/users/{id}")
public void deleteUser(Long id) { }
```

用户访问 `/admin/users/123`（DELETE）时：
1. 先经过 URL 权限检查（需要 ADMIN 角色）
2. 再经过方法权限检查（需要 SUPER_ADMIN 角色）
3. 两个检查都通过，才能执行方法

## 常见问题与解决方案

### 问题一：@PreAuthorize 不生效

排查步骤：

1. 确认添加了 `@EnableMethodSecurity`
2. 确认 `@EnableMethodSecurity(prePostEnabled = true)` 启用了 prePost
3. 确认注解放在正确的方法上（不是接口方法）

### 问题二：SpEL 表达式中引用参数失败

```java
// 错误：参数名可能被混淆
@PreAuthorize("#userId == 1")

// 正确：保留参数名
@PreAuthorize("#userId == 1")  // 需要 -parameters 编译参数
// 或
@PreAuthorize("#p0 == 1")  // 使用参数索引
```

### 问题三：方法内部调用不触发拦截

同一个类内部的方法调用，不会走 AOP 拦截：

```java
@Service
public class UserService {
    public void methodA() {
        methodB();  // 直接调用，不触发 @PreAuthorize
    }

    @PreAuthorize("hasRole('ADMIN')")
    public void methodB() {
        // 不会被拦截
    }
}
```

**解决**：注入自身或通过 Bean 调用：

```java
@Autowired
private UserService self;

public void methodA() {
    self.methodB();  // 通过代理调用，触发拦截
}
```

## 面试追问预判

### Q1：@PreAuthorize 和 @Secured 的区别？

@PreAuthorize 支持 SpEL 表达式，可以组合复杂条件、引用方法参数；@Secured 只支持简单的角色列表。如果需要复杂的权限控制，用 @PreAuthorize。

### Q2：方法级安全对性能的影响？

每次方法调用都要经过 AOP 拦截和权限评估，有一定性能损耗。但对于大多数应用来说，这个损耗可以忽略。如果追求极致性能，可以在方法内部手动判断权限。

### Q3：@PostAuthorize 的使用场景？

适合「方法执行后才知道是否有权访问」的场景。比如：返回当前登录用户自己的订单列表，可以在方法返回后过滤掉不属于当前用户的订单。

---

## 总结

`@PreAuthorize` 是 Spring Security 方法级权限控制的核心：

1. **方法执行前检查**：权限不足直接拒绝，不执行方法
2. **强大的 SpEL**：支持条件组合、参数引用、方法调用
3. **开启方式**：`@EnableMethodSecurity(prePostEnabled = true)`
4. **与 URL 权限叠加**：两层防护，更安全
