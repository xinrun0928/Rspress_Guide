# CORS 跨域与 Security：Security 过滤器链中的 CORS 处理

跨域请求被浏览器拦截，是很多前后端分离项目遇到的第一个坑。

`Access-Control-Allow-Origin` 配了，但请求还是被拦截？Spring Security 的 CORS 处理，在过滤器链的哪个位置？

---

## CORS 基础回顾

### 什么是跨域请求？

当一个请求的协议、域名、端口三者中任意一个与当前页面不同时，就是跨域请求。

| 当前页面 | 请求目标 | 是否跨域 |
|---|---|---|
| `http://example.com/` | `http://example.com/api` | 否 |
| `http://example.com/` | `https://example.com/api` | 是（协议不同） |
| `http://example.com/` | `http://api.example.com/api` | 是（域名不同） |
| `http://example.com/` | `http://example.com:8080/api` | 是（端口不同） |

### CORS 的工作原理

```
┌─────────────────────────────────────────────────────────────┐
│                    CORS 请求流程                               │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  【简单请求】                                                │
│  浏览器发起请求                                              │
│  服务器返回 + Access-Control-Allow-Origin 头                  │
│  浏览器检查，响应给 JS                                       │
│                                                              │
│  【非简单请求（预检请求）】                                   │
│  浏览器先发 OPTIONS 预检请求                                  │
│  服务器返回允许的方法和头                                      │
│  浏览器再发实际请求                                          │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 简单请求 vs 非简单请求

| 类型 | 条件 |
|---|---|
| 简单请求 | GET/HEAD/POST + 特定 Content-Type + 无自定义头 |
| 非简单请求 | 其他所有请求 |

## Spring Security 中的 CORS

### CORS 在过滤器链中的位置

```
┌─────────────────────────────────────────────────────────────┐
│              Spring Security 过滤器链                          │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  1. ChannelProcessingFilter                                  │
│  2. SecurityContextPersistenceFilter                         │
│  3. CorsFilter                    ← CORS 处理在这里          │
│  4. CsrfFilter                                             │
│  5. LogoutFilter                                            │
│  6. UsernamePasswordAuthenticationFilter                     │
│  ...                                                        │
│  14. FilterSecurityInterceptor                               │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 为什么 CORS Filter 在前面？

因为 CORS 是一种「浏览器安全策略」，需要在认证之前处理。如果跨域请求被 CORS 拦截，就不需要进行后续的认证了。

## Spring Security CORS 配置

### 基本配置

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            // CORS 配置
            .cors(cors -> cors
                .configurationSource(corsConfigurationSource())
            )
            // 其他配置
            .authorizeHttpRequests(auth -> auth
                .anyRequest().authenticated()
            );

        return http.build();
    }

    @Bean
    public CorsConfigurationSource corsConfigurationSource() {
        CorsConfiguration configuration = new CorsConfiguration();
        configuration.setAllowedOrigins(Arrays.asList(
            "http://localhost:3000",
            "https://example.com"
        ));
        configuration.setAllowedMethods(Arrays.asList("GET", "POST", "PUT", "DELETE"));
        configuration.setAllowedHeaders(Arrays.asList("*"));
        configuration.setAllowCredentials(true);  // 允许携带 Cookie
        configuration.setMaxAge(3600L);  // 预检请求缓存时间

        UrlBasedCorsConfigurationSource source = new UrlBasedCorsConfigurationSource();
        source.registerCorsConfiguration("/**", configuration);
        return source;
    }
}
```

### 使用 Spring Boot 的 CORS 配置

Spring Boot 提供了更简单的 CORS 配置方式：

```java
@Configuration
public class CorsConfig implements WebMvcConfigurer {

    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/api/**")
            .allowedOrigins("http://localhost:3000")
            .allowedMethods("GET", "POST", "PUT", "DELETE")
            .allowedHeaders("*")
            .allowCredentials(true)
            .maxAge(3600);
    }
}
```

但注意：Spring Security 的 CORS 配置会覆盖 Spring MVC 的配置。

## Spring Security 6.x 的变化

### allowedOriginPatterns

Spring Security 6.x 中，`allowedOrigins` 被标记为过时，推荐使用 `allowedOriginPatterns`：

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public CorsConfigurationSource corsConfigurationSource() {
        CorsConfiguration configuration = new CorsConfiguration();

        // Spring Security 6.x 推荐
        configuration.setAllowedOriginPatterns(Arrays.asList(
            "http://localhost:*",       // 本地开发
            "https://*.example.com"    // 生产环境
        ));

        // 仍然有效，但不推荐
        // configuration.setAllowedOrigins(Arrays.asList("http://localhost:3000"));

        configuration.setAllowedMethods(Arrays.asList("GET", "POST", "PUT", "DELETE"));
        configuration.setAllowedHeaders(Arrays.asList("*"));
        configuration.setAllowCredentials(true);

        UrlBasedCorsConfigurationSource source = new UrlBasedCorsConfigurationSource();
        source.registerCorsConfiguration("/**", configuration);
        return source;
    }
}
```

### 为什么废弃 allowedOrigins？

`allowedOrigins` 不能使用通配符。当你有多个域名时，必须一个个列出来：

```java
// 旧方式：每个域名都要列出来
configuration.setAllowedOrigins(Arrays.asList(
    "https://www.example.com",
    "https://m.example.com",
    "https://app.example.com"
));
```

`allowedOriginPatterns` 支持通配符：

```java
// 新方式：使用通配符
configuration.setAllowedOriginPatterns(Arrays.asList(
    "https://*.example.com"
));
```

## 常见问题与解决方案

### 问题一：配置了 CORS 但还是被拦截

排查步骤：

1. 确认 CORS 配置已生效
2. 确认 `allowedOriginPatterns` / `allowedOrigins` 包含请求来源
3. 如果使用 `allowCredentials(true)`，`allowedOriginPatterns` 不能使用 `*`

### 问题二：预检请求（OPTIONS）被拦截

预检请求是 CORS 的一部分，必须放行：

```java
http
    .cors(cors -> cors.configurationSource(corsConfigurationSource()))
    .csrf(csrf -> csrf
        .ignoringRequestMatchers("/api/**")  // API 关闭 CSRF 或忽略 OPTIONS
    )
```

### 问题三：Cookie 无法跨域传递

跨域传递 Cookie 需要满足：

1. 服务器设置 `Access-Control-Allow-Credentials: true`
2. 服务器设置 `Access-Control-Allow-Origin` 不能是 `*`
3. 客户端设置 `withCredentials = true`

```java
configuration.setAllowCredentials(true);
configuration.setAllowedOriginPatterns(Arrays.asList("https://*.example.com"));
```

```javascript
fetch('https://api.example.com/data', {
    credentials: 'include'  // 携带 Cookie
});
```

## CORS 与 CSRF 的关系

### CORS 是浏览器安全策略

CORS 是浏览器强制执行的安全策略，防止恶意网站读取其他网站的响应。

### CSRF 是服务端防护机制

CSRF 是服务端的安全机制，防止恶意网站以用户身份发起请求。

### 两者配合

```
┌─────────────────────────────────────────────────────────────┐
│                  双重防护机制                                 │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  CORS：防止恶意网站读取响应                                    │
│  ├── 浏览器检查 Origin                                       │
│  └── 跨域请求被拦截或只返回部分数据                          │
│                                                              │
│  CSRF：防止恶意网站发起请求                                   │
│  ├── 服务端验证 CSRF Token                                   │
│  └── 无 Token 的请求被拒绝                                  │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

## 微服务架构中的 CORS

### 网关层统一处理

在微服务架构中，通常在网关层统一处理 CORS：

```yaml
# application.yml
spring:
  cloud:
    gateway:
      globalcors:
        cors-configurations:
          '[/**]':
            allowedOrigins: "https://example.com"
            allowedMethods:
              - GET
              - POST
              - PUT
              - DELETE
            allowedHeaders: "*"
            allowCredentials: true
            maxAge: 3600
```

### 后端服务不再处理 CORS

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            // 信任网关层的 CORS 配置，后端不再重复配置
            .cors(cors -> cors.disable())
            .csrf(csrf -> csrf.disable())
            .authorizeHttpRequests(auth -> auth.anyRequest().authenticated());

        return http.build();
    }
}
```

## 面试追问预判

### Q1：CORS 预检请求是什么？

预检请求是浏览器在发起非简单请求之前，先发送的 OPTIONS 请求。用于询问服务器是否允许跨域请求，包含允许的方法、头等信息。

### Q2：为什么 `Access-Control-Allow-Origin` 不能用 `*` 当 `credentials` 为 `true`？

因为 `credentials` 为 `true` 时，浏览器需要知道具体允许哪个来源（用于验证 Cookie）。`*` 不能和 `credentials` 同时使用。

### Q3：Spring Security 和 Spring MVC 的 CORS 配置哪个生效？

Spring Security 的 CORS 配置会覆盖 Spring MVC 的配置。如果想用 Spring MVC 的配置，可以禁用 Spring Security 的 CORS：

```java
http.cors(cors -> cors.disable())
```

## 总结

Spring Security 中的 CORS 处理：

1. **位置**：在 Security 过滤器链的前面，在认证之前处理
2. **配置**：`CorsConfigurationSource` Bean 配置 CORS 参数
3. **Spring Security 6.x**：推荐使用 `allowedOriginPatterns` 替代 `allowedOrigins`
4. **预检请求**：OPTIONS 请求也需要放行
5. **微服务**：在网关层统一处理，后端服务禁用
