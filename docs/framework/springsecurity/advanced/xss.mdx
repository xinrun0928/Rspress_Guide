# XSS 防护：HttpFirewall 与内容安全策略

XSS 攻击是如何注入恶意脚本的？

用户评论中输入了一段 `<script>alert('XSS')</script>`，这段代码被执行了。这就是 XSS 攻击。

Spring Security 如何防护 XSS？

## XSS 攻击类型

### 反射型 XSS

恶意脚本通过 URL 参数传入，服务器直接将参数反射到响应中，浏览器执行脚本。

```
用户点击链接：http://example.com/search?q=<script>alert('XSS')</script>
    ↓
服务器从 URL 获取 q 参数，反射到页面
    ↓
浏览器执行脚本
```

### 存储型 XSS

恶意脚本被存储到数据库中，用户访问页面时脚本被执行。

```
攻击者提交评论：<script>alert('XSS')</script>
    ↓
评论存储到数据库
    ↓
其他用户访问页面
    ↓
页面加载评论，执行脚本
```

### DOM 型 XSS

恶意脚本通过操作 DOM 执行，不经过服务器。

```javascript
// 恶意页面
const params = new URLSearchParams(window.location.search);
document.write("搜索结果: " + params.get('q'));
```

## HttpFirewall

### 什么是 HttpFirewall？

`HttpFirewall` 是 Spring Security 的 HTTP 防火墙，用于过滤恶意请求。它会拒绝不符合规范的请求，防止潜在的 XSS 和注入攻击。

### StrictHttpFirewall

Spring Security 默认使用 `StrictHttpFirewall`，它会拒绝包含可疑字符的请求：

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        // Spring Security 5.x 默认已使用 StrictHttpFirewall
        // 可以自定义配置
        http
            .authorizeHttpRequests(auth -> auth.anyRequest().authenticated());

        return http.build();
    }
}
```

### 防护规则

```java
@Configuration
public class CustomFirewallConfig {

    @Bean
    public HttpFirewall strictHttpFirewall() {
        StrictHttpFirewall firewall = new StrictHttpFirewall();

        // 允许特定字符
        firewall.setAllowedHttpMethods(Arrays.asList("GET", "POST", "PUT", "DELETE"));

        // 允许特定请求头
        firewall.setAllowedHeaderNames(pattern -> true);  // 允许所有请求头
        firewall.setAllowedHeaderValues(pattern -> true);  // 允许所有请求头值

        // 拒绝特殊字符
        firewall.setAllowSemicolon(false);   // 拒绝分号
        firewall.setAllowBackslash(false);   // 拒绝反斜杠
        firewall.setAllowUrlEncodedSlash(false);  // 拒绝 URL 编码的斜杠

        // 拒绝代理头
        firewall.setAllowProxyHeaders(false);

        return firewall;
    }
}
```

### StrictHttpFirewall 的规则

| 规则 | 说明 |
|---|---|
| 拒绝 `;` | 防止会话固定攻击 |
| 拒绝 `\0` | 防止空字节注入 |
| 拒绝 URL 编码 | 防止编码绕过 |
| 拒绝 `/` 双斜杠 | 防止路径遍历 |
| 拒绝 `.` 点号 | 防止路径遍历（可选） |

## 内容安全策略（CSP）

### 什么是 CSP？

Content-Security-Policy 是 HTTP 响应头，用于告诉浏览器只允许加载特定来源的资源。这可以有效防止 XSS 攻击。

### CSP 配置

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .headers(headers -> headers
                .contentSecurityPolicy(csp -> csp
                    .policyDirectives("script-src 'self'; " +
                        "style-src 'self' 'unsafe-inline'; " +
                        "img-src 'self' data: https:;")
                )
            );

        return http.build();
    }
}
```

### 常用 CSP 指令

| 指令 | 说明 |
|---|---|
| `default-src` | 默认来源 |
| `script-src` | JavaScript 来源 |
| `style-src` | CSS 来源 |
| `img-src` | 图片来源 |
| `connect-src` | AJAX、WebSocket 来源 |
| `font-src` | 字体来源 |
| `frame-src` | iframe 来源 |
| `report-uri` | 违规报告地址 |

### CSP 值

| 值 | 说明 |
|---|---|
| `'self'` | 同源资源 |
| `'none'` | 禁止所有 |
| `'unsafe-inline'` | 内联脚本（不安全） |
| `'unsafe-eval'` | eval()（不安全） |
| `data:` | Data URL |
| `https:` | HTTPS 任意来源 |

## Spring Security 的安全响应头

### 默认启用的响应头

```java
http.headers(headers -> headers
    .frameOptions(frame -> frame.deny())                    // X-Frame-Options
    .xssProtection(xss -> xss.enable())                     // X-XSS-Protection
    .contentTypeOptions(content -> content.enable())       // X-Content-Type-Options
    .httpStrictTransportSecurity(hsts -> hsts               // Strict-Transport-Security
        .includeSubDomains(true)
        .maxAgeInSeconds(31536000)
    )
)
```

### X-Frame-Options

防止页面被 iframe 嵌套：

```java
http.headers(headers -> headers
    .frameOptions(frame -> frame
        .deny())  // 完全禁止被 iframe 嵌套
        // .sameOrigin())  // 只允许同源嵌套
)
```

### X-Content-Type-Options

防止浏览器 MIME 类型嗅探：

```java
http.headers(headers -> headers
    .contentTypeOptions(content -> content
        .and()
    )
)
```

### Strict-Transport-Security（HSTS）

强制使用 HTTPS：

```java
http.headers(headers -> headers
    .httpStrictTransportSecurity(hsts -> hsts
        .includeSubDomains(true)
        .maxAgeInSeconds(31536000)  // 1 年
        .preload(true)  // 支持 HSTS preload
    )
)
```

## Spring MVC 的 XSS 防护

### HTML 转义

Spring 提供了 `HtmlUtils.htmlEscape()` 方法：

```java
@Controller
public class ArticleController {

    @GetMapping("/article/{id}")
    public String article(@PathVariable Long id, Model model) {
        Article article = articleService.findById(id);
        // 自动 HTML 转义
        model.addAttribute("title", HtmlUtils.htmlEscape(article.getTitle()));
        model.addAttribute("content", HtmlUtils.htmlEscape(article.getContent()));
        return "article";
    }
}
```

### Thymeleaf 自动转义

Thymeleaf 默认开启 HTML 转义：

```html
<!-- Thymeleaf 默认转义 HTML -->
<p th:text="${article.content}"></p>

<!-- 如果需要输出原始 HTML，使用 th:utext -->
<p th:utext="${article.content}"></p>  <!-- 危险！ -->
```

### @RequestParam 转义

```java
@GetMapping("/search")
public String search(@RequestParam String q, Model model) {
    // 自动转义
    String safeQuery = HtmlUtils.htmlEscape(q);
    model.addAttribute("query", safeQuery);
    return "search";
}
```

## XSS 防护的最佳实践

### 输入层面

1. **验证输入**：检查数据类型、长度、格式
2. **转义输出**：将特殊字符转为 HTML 实体

```java
// Spring 的 HTML 转义映射
'<' → '&lt;'
'>' → '&gt;'
'&' → '&amp;'
'"' → '&quot;'
''' → '&#39;'
```

### 输出层面

1. **HTML 转义**：对用户输入的内容进行 HTML 转义
2. **使用模板引擎**：Thymeleaf、FreeMarker 默认转义
3. **避免内联脚本**：不使用 `th:utext` 输出用户输入

### 配置层面

1. **CSP**：配置内容安全策略，限制脚本来源
2. **HttpFirewall**：过滤恶意请求
3. **安全响应头**：使用 X-Frame-Options、X-Content-Type-Options

## 常见问题与解决方案

### 问题一：HTML 转义影响正常 HTML 内容

**场景**：用户想发布博客文章，包含正常的 HTML 格式

**解决**：
1. 使用富文本编辑器，自动清理恶意脚本
2. 使用白名单 HTML 库（如 OWASP Java HTML Sanitizer）
3. 使用 Markdown 代替 HTML

### 问题二：CSP 阻止了合法脚本

**解决**：根据实际需要调整 CSP 策略

```javascript
// 允许加载同域和 CDN 的脚本
Content-Security-Policy: script-src 'self' https://cdn.example.com;
```

### 问题三：HttpFirewall 拦截了合法请求

**解决**：根据业务需要调整规则

```java
StrictHttpFirewall firewall = new StrictHttpFirewall();
// 如果业务需要允许分号
firewall.setAllowSemicolon(true);
```

## 面试追问预判

### Q1：XSS 和 CSRF 的区别？

XSS 是注入恶意脚本到页面中执行，CSRF 是利用用户身份发起恶意请求。XSS 可以获取页面内容（包括 CSRF Token），CSRF 只能发起请求。两者可以配合：XSS 获取 Token 后发起 CSRF 攻击。

### Q2：如何防止 DOM 型 XSS？

DOM 型 XSS 不经过服务器，需要在前端进行防护：
1. 使用安全的 DOM 操作方法
2. 避免使用 `document.write()`
3. 对输入进行转义

### Q3：CSP 能完全防止 XSS 吗？

CSP 大大降低了 XSS 的风险，但不是完全免疫。如果攻击者找到绕过 CSP 的方法（如内联事件处理器），仍然可能执行脚本。因此，XSS 防护需要多层防御。

---

## 总结

Spring Security 的 XSS 防护是一个系统工程：

1. **HttpFirewall**：过滤恶意请求
2. **CSP**：内容安全策略，限制资源来源
3. **安全响应头**：X-Frame-Options、X-Content-Type-Options、HSTS
4. **HTML 转义**：在输出时转义特殊字符
5. **多层防护**：输入验证、输出转义、配置安全策略
