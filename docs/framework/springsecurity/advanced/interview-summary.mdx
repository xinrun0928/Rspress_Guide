# Spring Security 核心面试题

Spring Security 是 Java 后端面试的重点内容。这一节汇总了高频面试题，帮你快速准备面试。

## 基础概念

### Q1：Spring Security 的核心原理是什么？

Spring Security 的核心是基于 Servlet Filter 的过滤器链。所有请求在到达 Controller 之前，都要经过这串过滤器。

```
FilterChainProxy → 认证过滤器 → ... → FilterSecurityInterceptor → Controller
```

过滤器链中：
- **认证过滤器**负责识别用户身份（表单登录过滤器、OAuth2 过滤器等）
- **授权过滤器**负责判断是否有权限访问（FilterSecurityInterceptor）
- **SecurityContextPersistenceFilter**负责在请求间传递认证状态

### Q2：Authentication 和 Authorization 的区别？

| 概念 | 英文 | 问题 | Spring Security 体现 |
|---|---|---|---|
| 认证 | Authentication | 你是谁？ | `Authentication` 接口 |
| 授权 | Authorization | 你能做什么？ | `AccessDecisionManager` |

### Q3：Spring Security 的核心接口有哪些？

| 接口 | 说明 |
|---|---|
| `Authentication` | 认证信息，封装用户和权限 |
| `SecurityContext` | 持有 Authentication 的容器 |
| `SecurityContextHolder` | 操作 SecurityContext 的入口 |
| `UserDetails` | 用户信息接口 |
| `UserDetailsService` | 加载用户信息的服务 |
| `GrantedAuthority` | 权限接口 |
| `AuthenticationProvider` | 认证提供者 |
| `AccessDecisionManager` | 权限决策管理器 |

## 认证流程

### Q4：Spring Security 的认证流程是什么？

1. 用户提交登录请求（用户名 + 密码）
2. `UsernamePasswordAuthenticationFilter` 拦截，生成 `UsernamePasswordAuthenticationToken`
3. 交给 `AuthenticationManager`（通常是 `ProviderManager`）
4. `ProviderManager` 找到支持该 Token 的 `AuthenticationProvider`（`DaoAuthenticationProvider`）
5. `DaoAuthenticationProvider` 调用 `UserDetailsService.loadUserByUsername()` 加载用户
6. `PasswordEncoder.matches()` 校验密码
7. 认证通过，设置 `SecurityContextHolder`
8. `AuthenticationSuccessHandler` 处理成功（重定向）
9. 认证失败，`AuthenticationFailureHandler` 处理失败

### Q5：SecurityContextHolder 有几种存储策略？

| 策略 | 说明 |
|---|---|
| ThreadLocal（默认） | 仅当前线程可见 |
| InheritableThreadLocal | 父子线程共享 |
| Global | 全局静态变量（不推荐） |

### Q6：Session 和 JWT Token 的区别？

| 维度 | Session | JWT Token |
|---|---|---|
| 存储位置 | 服务端 | 客户端 |
| 状态 | 有状态 | 无状态 |
| 撤销 | 可实时撤销 | 只能等过期 |
| 适用场景 | 传统 Web 应用 | RESTful API、微服务 |
| 依赖 | Session 存储介质 | 无 |

## 过滤器链

### Q7：Spring Security 有哪些核心过滤器？

| 过滤器 | 作用 |
|---|---|
| `SecurityContextPersistenceFilter` | 从 Session 恢复/保存 SecurityContext |
| `LogoutFilter` | 处理登出 |
| `UsernamePasswordAuthenticationFilter` | 表单登录认证 |
| `AnonymousAuthenticationFilter` | 匿名认证（未登录用户） |
| `SessionManagementFilter` | 会话管理 |
| `ExceptionTranslationFilter` | 异常转换 |
| `FilterSecurityInterceptor` | 权限决策 |

### Q8：过滤器的执行顺序如何保证？

通过 `FilterRegistrationBean` 的 `setOrder()` 方法控制。数字越小，优先级越高，执行越靠前。

### Q9：自定义过滤器如何插入到合适位置？

```java
// 在某个过滤器之前
http.addFilterBefore(customFilter, UsernamePasswordAuthenticationFilter.class);

// 在某个过滤器之后
http.addFilterAfter(customFilter, UsernamePasswordAuthenticationFilter.class);

// 替换某个过滤器
http.addFilterAt(customFilter, UsernamePasswordAuthenticationFilter.class);
```

## 权限控制

### Q10：Authority 和 Role 的区别？

没有本质区别。Role 是 Authority 的一种，约定使用 `ROLE_` 前缀。

| 方法 | 说明 |
|---|---|
| `hasRole('ADMIN')` | 自动加上 `ROLE_` 前缀，检查 `ROLE_ADMIN` |
| `hasAuthority('ADMIN')` | 不加前缀，检查 `ADMIN` |

### Q11：AccessDecisionManager 的三种决策策略？

| 策略 | 规则 |
|---|---|
| `AffirmativeBased` | 任一投票器赞成即通过（默认） |
| `ConsensusBased` | 多数票决定 |
| `UnanimousBased` | 全票赞成才通过 |

### Q12：@PreAuthorize 和 @Secured 的区别？

| 特性 | @PreAuthorize | @Secured |
|---|---|---|
| SpEL 支持 | 是 | 否 |
| 条件组合 | 丰富 | 简单 |
| 方法参数引用 | 是 | 否 |
| 返回值引用 | @PostAuthorize | 否 |

## 安全机制

### Q13：CSRF 的原理是什么？

攻击者诱使已登录用户访问恶意页面，恶意页面自动发送请求到目标网站，由于浏览器会自动携带 Cookie，服务器无法区分是用户主动发起的请求还是恶意页面伪造的请求。

### Q14：如何防范 CSRF？

1. 使用 Synchronizer Token Pattern（Spring Security 默认）
2. 检查 SameSite Cookie
3. 验证请求来源

### Q15：XSS 和 CSRF 的区别？

| 攻击 | 原理 | 防护 |
|---|---|---|
| XSS | 在页面中注入恶意脚本 | 输入转义、CSP |
| CSRF | 利用用户身份发起恶意请求 | CSRF Token |

XSS 可以获取页面内容（包括 CSRF Token），两者可以配合攻击。

### Q16：Session 固定攻击是什么？如何防范？

攻击者先获取一个 Session ID，然后诱使受害者使用这个 Session ID 登录。登录成功后，攻击者使用同样的 Session ID 冒充受害者。

防范：Spring Security 提供 `sessionFixation().changeSessionId()` 配置。

## 配置与扩展

### Q17：如何自定义 UserDetailsService？

```java
@Service
public class CustomUserDetailsService implements UserDetailsService {

    @Override
    public UserDetails loadUserByUsername(String username)
            throws UsernameNotFoundException {
        // 1. 查询用户
        UserEntity user = userRepository.findByUsername(username);
        if (user == null) {
            throw new UsernameNotFoundException("用户不存在");
        }

        // 2. 构建权限
        List<GrantedAuthority> authorities = user.getRoles().stream()
            .map(role -> new SimpleGrantedAuthority("ROLE_" + role.getName()))
            .collect(Collectors.toList());

        // 3. 返回 UserDetails
        return new User(user.getUsername(), user.getPassword(), authorities);
    }
}
```

### Q18：Spring Security 6.x 最大的变化是什么？

`WebSecurityConfigurerAdapter` 被完全移除。所有配置必须使用 `SecurityFilterChain` @Bean + Lambda DSL 方式。

### Q19：Spring Security 6.x 如何配置？

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/public/**").permitAll()
                .anyRequest().authenticated()
            )
            .formLogin(Customizer.withDefaults())
            .logout(Customizer.withDefaults());

        return http.build();
    }
}
```

## 微服务相关

### Q20：微服务架构下如何实现认证？

两种方案：

1. **分散认证**：每个服务都集成 Spring Security 验证 Token
2. **统一鉴权**：网关层统一认证，后端服务只做授权

推荐统一鉴权方案：网关验证 Token，提取用户信息注入请求头，后端服务从请求头获取用户信息。

### Q21：JWT 为什么能防篡改？

JWT 的 Signature 部分使用密钥对 Header + Payload 计算签名。如果 Payload 被篡改，重新签名需要密钥，服务端验证签名时会发现不一致。

### Q22：JWT 的缺点是什么？

1. **无法主动撤销**：Token 过期前无法使其失效
2. **Payload 不安全**：Base64 编码，可被解码
3. **大小限制**：Payload 不宜过大

## 高频追问

### Q23：SecurityContext 怎么在线程间传递？

默认使用 ThreadLocal，子线程无法继承。如果需要传递：

1. 使用 `DelegatingSecurityContextRunnable`
2. 使用 `DelegatingSecurityContextTaskExecutor`
3. 配置 `TaskDecorator`

### Q24：如何实现多数据源认证？

```java
@Bean
public AuthenticationManager authenticationManager(
        AuthenticationConfiguration config) throws Exception {
    ProviderManager providerManager =
        (ProviderManager) config.getAuthenticationManager();
    providerManager.getProviders().add(customAuthenticationProvider());
    return providerManager;
}
```

### Q25：Remember-Me 的原理？

登录时生成 Token，写入 Cookie。下次访问时，`RememberMeFilter` 读取 Cookie，验证 Token 有效性，自动恢复登录状态。

## 面试技巧

### 如何回答「Spring Security 的认证流程」？

不要背书式的回答。按照请求的路径，一步一步说：

```
用户提交表单 → Filter 拦截 → 生成 Token → ProviderManager → Provider → UserDetailsService → 密码校验 → SecurityContext → SuccessHandler
```

每一步都简单说明，既清晰又显得理解深入。

### 如何回答「Spring Security 和 Shiro 的区别」？

从出身、依赖、配置方式、权限模型、社区活跃度等维度对比。突出 Spring Security 与 Spring 生态的深度集成，以及更活跃的社区。

### 如何回答「如何实现 XXX 功能」？

先说原理，再说实现步骤，最后给代码示例。三段式结构，原理让人信服，实现步骤让人理解，代码让人验证。
