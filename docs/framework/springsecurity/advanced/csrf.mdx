# CSRF 防护：Synchronizer Token Pattern

Cross-Site Request Forgery，攻击者是怎么绕过你的？

你登录了银行网站，正看着自己的余额。这时你收到一封邮件，点开一看，是一个「晒账单赢大奖」的活动链接。你随手点了进去。

这个页面背后，悄悄向银行转账 API 发了一个请求。由于你登录过银行网站，Cookie 还在有效期内，转账成功了。

这就是 CSRF 攻击。

## CSRF 攻击原理

### 什么是 CSRF？

CSRF（Cross-Site Request Forgery，跨站请求伪造）是一种利用用户已登录的身份，在用户不知情的情况下，以用户的身份发起恶意请求的攻击。

```
┌─────────────────────────────────────────────────────────────┐
│                    CSRF 攻击流程                              │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  1. 用户登录银行网站，获取有效 Session                        │
│       ↓                                                      │
│  2. 用户收到钓鱼邮件，打开恶意页面                            │
│       ↓                                                      │
│  3. 恶意页面自动发起请求：                                   │
│     POST /bank/transfer?to=hacker&amount=10000              │
│       ↓                                                      │
│  4. 浏览器自动携带 Cookie                                     │
│       ↓                                                      │
│  5. 服务器认为是用户本人操作，转账成功                         │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### CSRF 的特点

1. **利用用户身份**：攻击者不能获取 Cookie，但浏览器会自动携带
2. **用户不知情**：请求在用户不知情的情况下发起
3. **绕过同源策略**：攻击页面和目标网站不同源，但 Cookie 会被携带

### 为什么 Cookie 会被携带？

因为 CSRF 攻击的请求通常使用 `<form>` 或 `Fetch API`，这些请求不会携带自定义请求头，但会携带 Cookie（除非 SameSite 属性限制）。

## Synchronizer Token Pattern

### 原理

```
┌─────────────────────────────────────────────────────────────┐
│              Synchronizer Token Pattern                       │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  【正常请求】                                                │
│  服务器在表单中嵌入随机 Token                                 │
│  <form>                                                     │
│    <input type="hidden" name="_csrf" value="abc123...">     │
│    ...                                                       │
│  </form>                                                     │
│       ↓                                                      │
│  用户提交表单，Token 随请求发送                              │
│       ↓                                                      │
│  服务器验证 Token，匹配则通过                                │
│                                                              │
│  【CSRF 攻击】                                              │
│  恶意页面无法获取目标网站的 Token                             │
│  POST /bank/transfer?_csrf=xxx&to=hacker                   │
│       ↓                                                      │
│  服务器验证 Token，不匹配则拒绝                               │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 为什么有效？

1. **随机性**：Token 是随机生成的，攻击者无法猜测
2. **一次性绑定**：Token 与用户 Session 关联
3. **不可预测**：Token 不在 Cookie 中，无法被浏览器自动发送

## Spring Security 的 CSRF 防护

### 默认启用

Spring Security 默认启用 CSRF 防护：

```
http.authorizeHttpRequests(auth -> auth
    .anyRequest().authenticated()
)
.formLogin()
```

等同于：

```
http.csrf(csrf -> csrf.enable())  // 默认启用
```

### CSRF Token 的使用

#### Thymeleaf 模板

```html
<form th:action="@{/transfer}" method="post">
    <!-- Spring Security 自动注入 CSRF Token -->
    <input type="hidden" th:name="${_csrf.parameterName}"
           th:value="${_csrf.token}" />

    收款账户：<input type="text" name="to" />
    转账金额：<input type="number" name="amount" />
    <button type="submit">转账</button>
</form>
```

#### Spring MVC 表单标签

```txt
<%@ taglib prefix="form" uri="http://www.springframework.org/tags/form" %>

<form:form action="/transfer" method="post">
    <form:hidden path="*"/>  <!-- 自动包含 CSRF Token -->
    <!-- ... -->
</form:form>
```

### AJAX 请求中的 CSRF Token

```javascript
// 方式一：从 Cookie 中读取
const csrfToken = document.cookie
    .split('; ')
    .find(row => row.startsWith('XSRF-TOKEN='))
    ?.split('=')[1];

fetch('/api/data', {
    method: 'POST',
    headers: {
        'Content-Type': 'application/json',
        'X-XSRF-TOKEN': csrfToken
    },
    body: JSON.stringify({ data: 'value' })
});

// 方式二：从 meta 标签中读取
const csrfToken = document.querySelector('meta[name="_csrf"]').content;
```

Spring Security 会将 CSRF Token 写入 Cookie（名称为 `XSRF-TOKEN`），前端可以读取并发送。

## CSRF 配置详解

### Cookie 请求参数读取器

```java
http
    .csrf(csrf -> csrf
        // Cookie 请求参数读取器（读取 Cookie 中的 Token）
        .cookieCsrfRepository()
    )
```

这会自动将 CSRF Token 写入 Cookie，前端从 Cookie 读取并通过请求头发送。

### 自定义 Token 存储

```java
http
    .csrf(csrf -> csrf
        // 使用 HTTP Session 存储 Token（传统方式）
        .httpSessionCsrfRepository()
    )
```

### 忽略特定路径

```java
http
    .csrf(csrf -> csrf
        // API 接口通常使用 Token 认证，忽略 CSRF
        .ignoringRequestMatchers("/api/**")
    )
```

### 自定义 Token 仓库

```java
@Component
public class CustomCsrfTokenRepository implements CsrfTokenRepository {

    @Override
    public CsrfToken generateToken(HttpServletRequest request) {
        return new DefaultCsrfToken("X-CSRF-TOKEN", "_csrf", generateToken());
    }

    @Override
    public void saveToken(CsrfToken token, HttpServletRequest request,
                          HttpServletResponse response) {
        // 自定义存储逻辑
    }

    @Override
    public CsrfToken loadToken(HttpServletRequest request) {
        // 自定义加载逻辑
    }
}
```

## POST vs GET

### GET 请求不应该有副作用

CSRF 攻击只能发起 GET 以外的请求（除非使用 `<img>` 等标签发起 GET）。因此：

```java
// GET 请求应该只读取数据，不修改数据
@GetMapping("/user/profile")
public UserProfile getProfile() {
    return userService.getProfile();
}

// POST、PUT、DELETE 请求会修改数据，需要 CSRF 防护
@PostMapping("/user/profile")
public void updateProfile(@RequestBody ProfileUpdate update) {
    userService.updateProfile(update);
}
```

### RESTful API 的 CSRF

对于纯 RESTful API，通常使用 Token（如 JWT）认证，可以禁用 CSRF：

```java
http
    .csrf(csrf -> csrf
        .ignoringRequestMatchers("/api/**")
    )
```

## CSRF 与 CORS 的关系

### CORS 不会阻止 CSRF

```
┌─────────────────────────────────────────────────────────────┐
│              CORS vs CSRF                                   │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  CORS（跨域资源共享）                                         │
│  控制哪些来源可以访问 API                                     │
│  Access-Control-Allow-Origin: https://malicious.com         │
│  → 允许跨域访问（但如果有 CSRF Token，攻击仍然失败）          │
│                                                              │
│  CSRF（跨站请求伪造）                                        │
│  防止恶意页面发起的请求                                       │
│  服务器验证 CSRF Token → 拒绝恶意请求                        │
│                                                              │
│  两者配合使用：CORS 限制来源 + CSRF 验证 Token                │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### SameSite Cookie

现代浏览器提供 SameSite Cookie 属性，可以部分防止 CSRF：

```java
http
    .sessionManagement(session -> session
        .cookieSerializer(cookie ->
            cookie.cookieSerializer(new DefaultCookieSerializer())
                .setSameSite("Strict")  // Strict / Lax / None
        )
    )
```

| 值 | 说明 |
|---|---|
| `Strict` | Cookie 只在同站请求中发送（最安全） |
| `Lax` | GET 请求或导航请求会发送（默认值） |
| `None` | 任何请求都发送（需要 HTTPS + Secure） |

## 常见问题与解决方案

### 问题一：CSRF Token 过期

**解决**：页面加载时获取最新 Token，超时时提示用户刷新页面。

### 问题二：多 Tab 页面 Token 不同步

**解决**：使用 CookieCsrfTokenRepository，Token 自动同步。

### 问题三：第三方登录后的 CSRF

**解决**：第三方登录回调地址需要 CSRF 验证。

### 问题四：SPA 应用如何处理 CSRF

**解决**：

```java
http
    .csrf(csrf -> csrf
        .ignoringRequestMatchers("/api/**")  // 忽略 API
    )
.sessionManagement(session -> session
    .sessionCreationPolicy(SessionCreationPolicy.STATELESS)
)
```

## 面试追问预判

### Q1：CSRF 和 XSS 的区别？

CSRF 是跨站请求伪造，攻击者利用用户的身份发起请求。XSS 是跨站脚本注入，攻击者在页面中注入恶意脚本。两者可以配合使用：XSS 获取页面内容（包括 CSRF Token），然后发起 CSRF 攻击。

### Q2：为什么 GET 请求不需要 CSRF 防护？

因为 GET 请求不能修改数据，只能获取数据。如果你的 GET 请求会修改数据，那是一个设计问题，应该改成 POST。

### Q3：Spring Security 如何禁用 CSRF？

```java
http.csrf(csrf -> csrf.disable())
```

但禁用后，必须确保使用其他认证机制（如 JWT Token）。

## 总结

CSRF 防护是 Web 应用安全的基础：

1. **攻击原理**：利用 Cookie 自动携带特性，伪造用户请求
2. **Synchronizer Token**：在表单中嵌入随机 Token，验证来源
3. **Spring Security 支持**：自动生成、验证 Token
4. **配置选项**：忽略特定路径、自定义 Token 仓库
5. **与 CORS 配合**：双重防护
