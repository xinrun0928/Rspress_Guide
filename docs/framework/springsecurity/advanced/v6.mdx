# Spring Security 6.x：Breaking Changes 全解析

从 Spring Security 5 升级到 6，哪些配置会报错？

Spring Security 6.x 是近年来变化最大的版本。很多在 5.x 中习以为常的配置，在 6.x 中已经不再支持。

这一节，我们来全面了解 6.x 的 Breaking Changes。

## WebSecurityConfigurerAdapter 的移除

### 这是最大的变化

Spring Security 6.x 完全移除了 `WebSecurityConfigurerAdapter`：

```java
// Spring Security 5.x（已废弃）
@Configuration
@EnableWebSecurity
public class SecurityConfig extends WebSecurityConfigurerAdapter {

    @Override
    protected void configure(HttpSecurity http) throws Exception {
        http
            .authorizeRequests()
                .anyRequest().authenticated()
                .and()
            .formLogin()
                .and()
            .httpBasic();
    }
}
```

### Spring Security 6.x 的写法

```java
// Spring Security 6.x
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests(auth -> auth
                .anyRequest().authenticated()
            )
            .formLogin(Customizer.withDefaults())
            .httpBasic(Customizer.withDefaults());

        return http.build();
    }
}
```

### 主要变化

| 变化 | 说明 |
|---|---|
| 不再继承 | 不再 extends WebSecurityConfigurerAdapter |
| Lambda DSL | 推荐使用 Lambda DSL 配置 |
| @Bean 返回 | `SecurityFilterChain` 作为 @Bean 返回 |
| 移除方法覆写 | 不再有 `configure(HttpSecurity)` 方法覆写 |

## Lambda DSL 配置

### 为什么使用 Lambda DSL？

Lambda DSL 使配置更简洁，更易读：

```java
// 传统写法
http
    .authorizeRequests()
        .antMatchers("/public/**").permitAll()
        .anyRequest().authenticated()
        .and()
    .formLogin()
        .loginPage("/login")
        .and()
    .logout();

// Lambda DSL 写法
http
    .authorizeHttpRequests(auth -> auth
        .requestMatchers("/public/**").permitAll()
        .anyRequest().authenticated()
    )
    .formLogin(form -> form
        .loginPage("/login")
    )
    .logout(logout -> logout
        .logoutUrl("/logout")
    );
```

### 方法名变化

| 5.x 方法 | 6.x 方法 |
|---|---|
| `authorizeRequests()` | `authorizeHttpRequests()` |
| `antMatchers()` | `requestMatchers()` |
| `csrf().disable()` | `csrf(Customizer.withDefaults())` |

## CSRF 默认启用

### Spring Security 5.x

默认启用 CSRF，但可以通过继承 `WebSecurityConfigurerAdapter` 并覆写 `configure(HttpSecurity)` 方法来禁用。

### Spring Security 6.x

CSRF 默认启用，不可通过继承关闭：

```java
// Spring Security 5.x（现在会报错）
@Override
protected void configure(HttpSecurity http) throws Exception {
    http.csrf().disable();  // 不再允许
}

// Spring Security 6.x 正确方式
http
    .csrf(csrf -> csrf.disable())  // 必须显式配置
```

### 为什么这样变化？

为了安全。禁用 CSRF 是一个危险的操作，容易被攻击者利用。

## CORS 配置变化

### allowedOrigins 被废弃

Spring Security 6.x 中，`setAllowedOrigins()` 被废弃，推荐使用 `setAllowedOriginPatterns()`：

```java
// 旧方式（废弃）
configuration.setAllowedOrigins(Arrays.asList(
    "http://localhost:3000",
    "https://example.com"
));

// 新方式（推荐）
configuration.setAllowedOriginPatterns(Arrays.asList(
    "http://localhost:*",
    "https://*.example.com"
));
```

### 为什么？

`allowedOrigins` 不能使用通配符，当你有多个域名时必须一个个列出来。`allowedOriginPatterns` 支持通配符，更灵活。

## AuthorizationDeniedException

### Spring Security 6.1+

Spring Security 6.1 引入了 `AuthorizationDeniedException`：

```java
// Spring Security 6.0
public class AccessDeniedException extends RuntimeException {
    // ...
}

// Spring Security 6.1+
public class AuthorizationDeniedException extends AccessDeniedException {
    // ...
}
```

### 处理

```java
@ControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(AuthorizationDeniedException.class)
    public ResponseEntity<?> handleAuthorizationDenied(AuthorizationDeniedException ex) {
        return ResponseEntity.status(403)
            .body(Map.of("message", "权限不足"));
    }

    // 保留对旧异常的兼容
    @ExceptionHandler(AccessDeniedException.class)
    public ResponseEntity<?> handleAccessDenied(AccessDeniedException ex) {
        return ResponseEntity.status(403)
            .body(Map.of("message", "访问被拒绝"));
    }
}
```

## ClientDefaultsLoginPageGeneratingFilter 的移除

### Spring Security 6.x 之前

自动生成默认登录页面。

### Spring Security 6.x

移除 `ClientDefaultsLoginPageGeneratingFilter`。如果需要默认登录页面，需要手动配置：

```java
// Spring Security 6.x
http
    .formLogin(form -> form
        .loginPage("/login")  // 必须指定登录页面
    )
    .oauth2Login(oauth2 -> oauth2
        .loginPage("/oauth2/login")  // OAuth2 登录也需要指定
    );
```

## 其他废弃和移除

### 完全移除

| 移除项 | 替代方案 |
|---|---|
| `WebSecurityConfigurerAdapter` | `SecurityFilterChain` @Bean |
| `WebSecurityConfigurer` | `SecurityFilterChain` @Bean |
| `GlobalAuthenticationConfigurerAdapter` | `AuthenticationManager` @Bean |
| `ClientDefaultsLoginPageGeneratingFilter` | 手动配置登录页 |

### 标记为废弃

| 废弃项 | 替代方案 |
|---|---|
| `allowedOrigins()` | `allowedOriginPatterns()` |
| `antMatchers()` | `requestMatchers()` |
| `regexMatchers()` | `requestMatchers()` |
| `authorizeRequests()` | `authorizeHttpRequests()` |

## 迁移指南

### 从 5.x 迁移到 6.x

#### 步骤一：移除继承

```java
// 之前
@Configuration
@EnableWebSecurity
public class SecurityConfig extends WebSecurityConfigurerAdapter {
    // ...
}

// 之后
@Configuration
@EnableWebSecurity
public class SecurityConfig {
    // ...
}
```

#### 步骤二：将 configure 方法改为 @Bean

```java
// 之前
@Override
protected void configure(HttpSecurity http) throws Exception {
    http.authorizeRequests()...
}

// 之后
@Bean
public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    http.authorizeHttpRequests()...
    return http.build();
}
```

#### 步骤三：使用 Lambda DSL

```java
// 之前
http
    .authorizeRequests()
        .antMatchers("/public/**").permitAll()
        .anyRequest().authenticated()
        .and()
    .formLogin()
        .loginPage("/login")
        .and();

// 之后
http
    .authorizeHttpRequests(auth -> auth
        .requestMatchers("/public/**").permitAll()
        .anyRequest().authenticated()
    )
    .formLogin(form -> form
        .loginPage("/login")
    );
```

#### 步骤四：更新 CORS 配置

```java
// 之前
configuration.setAllowedOrigins(Arrays.asList("http://localhost:3000"));

// 之后
configuration.setAllowedOriginPatterns(Arrays.asList("http://localhost:*"));
```

## 常见问题与解决方案

### 问题一：启动报错 No Bean of type SecurityFilterChain

检查是否正确返回 `@Bean`：

```java
@Bean
public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    // ...
    return http.build();
}
```

### 问题二：CSRF 配置不生效

Spring Security 6.x CSRF 默认启用。如果确实需要禁用：

```java
http.csrf(csrf -> csrf.disable())
```

但要注意这会带来安全风险。

### 问题三：`requestMatchers` 找不到

确保导入了正确的类：

```java
import static org.springframework.security.web.util.matcher.AntPathRequestMatcher.antMatcher;
import static org.springframework.security.web.util.matcher.RegexRequestMatcher.regexMatcher;
import static org.springframework.security.web.util.matcher.IpAddressMatcher.ipAddressMatcher;
```

## 面试追问预判

### Q1：Spring Security 6.x 最大的变化是什么？

`WebSecurityConfigurerAdapter` 的完全移除。所有配置必须使用 `SecurityFilterChain` @Bean + Lambda DSL 方式。

### Q2：为什么要移除 WebSecurityConfigurerAdapter？

为了简化 API、减少歧义、让配置更清晰。Lambda DSL 比继承 + 方法覆写更简洁。

### Q3：Spring Security 6.x 还有什么其他变化？

- CSRF 默认启用，不可关闭
- `allowedOrigins` 废弃，改用 `allowedOriginPatterns`
- 引入 `AuthorizationDeniedException`
- 移除 `ClientDefaultsLoginPageGeneratingFilter`

---

## 总结

Spring Security 6.x 的 Breaking Changes：

1. **WebSecurityConfigurerAdapter 移除**：必须使用 `SecurityFilterChain` @Bean
2. **Lambda DSL 成为唯一推荐**：告别继承和覆写
3. **CSRF 默认启用**：安全第一
4. **CORS 配置变化**：`allowedOriginPatterns` 替代 `allowedOrigins`
5. **新异常类型**：`AuthorizationDeniedException`

迁移虽然有工作量，但新 API 更清晰、更安全。理解这些变化，你就能顺利升级到 Spring Security 6.x。
