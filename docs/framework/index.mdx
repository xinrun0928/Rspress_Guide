# 框架原理：深入 Spring、MyBatis 与安全生态

import { Badge } from '@rspress/core/theme';

你有没有想过，当你写下 `@Autowired private UserService userService;` 时，Spring 到底做了什么？一个 `new HashMap<>()` 能解决的场景，为什么 MyBatis 要写 XML 映射文件？密码存到数据库里，框架是怎么「验证」的？

这些问题，停留在「会用」的层面无法回答。只有深入框架内部，理解它的设计思路和实现原理，才能真正做到游刃有余。

本模块聚焦 Java 生态中最核心的框架：**Spring Framework**、**Spring Boot**、**MyBatis**、**Spring Cloud**、**Spring Security** 和 **Shiro**。每一章都不止于「怎么用」，更深入「为什么这样设计」「源码是怎么实现的」。

## 二、核心框架的核心能力

### 2.1 Spring Framework：一切的基础

Spring Framework 是整个 Java 后端生态的基石。理解 Spring，就等于理解了一半的 Java 后端开发。

| 核心能力 | 解决什么问题 | 原理深度要求 |
|---|---|---|
| IoC 容器 | 对象创建与依赖管理 | <Badge text="必须掌握" type="danger" /> |
| AOP 代理 | 横切关注点分离 | <Badge text="必须掌握" type="danger" /> |
| Bean 生命周期 | Bean 的完整创建过程 | <Badge text="高频" type="warning" /> |
| 循环依赖 | 三级缓存解决依赖链 | <Badge text="高频" type="warning" /> |

### 2.2 Spring Boot：约定优于配置

Spring Boot 不是全新的框架，而是 Spring 的「最佳实践封装」。它解决的是 Spring Framework 配置繁琐的问题。

| 核心能力 | 解决什么问题 | 原理深度要求 |
|---|---|---|
| 自动配置 | 消除 XML / JavaConfig 配置 | <Badge text="必须掌握" type="danger" /> |
| 起步依赖 | 一行 POM 引入全套依赖 | <Badge text="必须掌握" type="danger" /> |
| 外部化配置 | 多环境配置管理 | <Badge text="高频" type="warning" /> |
| Actuator | 生产级监控端点 | <Badge text="了解" type="info" /> |

### 2.3 MyBatis：半自动 ORM 的优雅

MyBatis 和 Hibernate 代表了 ORM 的两条路线。MyBatis 控制 SQL 的灵活性，使其在高并发、大数据量场景下有独特的优势。

| 核心能力 | 解决什么问题 | 原理深度要求 |
|---|---|---|
| 四大核心组件 | Executor/StatementHandler/ParameterHandler/ResultSetHandler | <Badge text="必须掌握" type="danger" /> |
| 缓存机制 | 一级缓存 vs 二级缓存 | <Badge text="必须掌握" type="danger" /> |
| 动态 SQL | if/where/foreach 等标签 | <Badge text="高频" type="warning" /> |
| 嵌套查询 | N+1 问题与解决 | <Badge text="高频" type="warning" /> |

### 2.4 Spring Cloud：微服务架构的全家桶方案

Spring Cloud 不是单一产品，而是一套微服务落地的完整方案，涵盖了注册中心、配置中心、网关、负载均衡、熔断、链路追踪等所有关键能力。

| 核心能力 | 解决什么问题 | 原理深度要求 |
|---|---|---|
| 服务注册与发现 | 服务实例动态管理 | <Badge text="必须掌握" type="danger" /> |
| 声明式 HTTP 客户端 | 微服务间调用 | <Badge text="必须掌握" type="danger" /> |
| API 网关路由 | 统一入口与过滤 | <Badge text="高频" type="warning" /> |
| 熔断降级 | 故障隔离与容错 | <Badge text="高频" type="warning" /> |

### 2.5 Spring Security：企业级安全框架

Spring Security 是 Java 生态中最完善的安全框架，提供了认证、授权、Session 管理、加密、OAuth2 等全套能力。

| 核心能力 | 解决什么问题 | 原理深度要求 |
|---|---|---|
| 过滤器链 | 请求层面的统一安全控制 | <Badge text="必须掌握" type="danger" /> |
| 认证与授权 | 用户身份验证与权限判断 | <Badge text="必须掌握" type="danger" /> |
| JWT 无状态认证 | Token 化的身份验证方案 | <Badge text="必须掌握" type="danger" /> |
| OAuth2 社会化登录 | 第三方账号登录 | <Badge text="高频" type="warning" /> |

### 2.6 Shiro：轻量级安全框架

Shiro 以简单易用著称，适合不需要 Spring Security 全部能力的中小型项目。它的设计理念是「简单但不简陋」。

### 3.1 入门路线：Spring 基础 → Spring Boot → MyBatis

```
第一步：Spring Framework 核心概念（IoC + AOP）

  └─ 理解 @SpringBootApplication、@EnableAutoConfiguration

  └─ 理解四大核心组件、缓存机制、动态 SQL

  └─ 理解 DispatcherServlet 流程、参数绑定、视图解析
```

### 3.2 进阶路线：微服务 → 安全 → 原理深挖

  └─ @EnableAutoConfiguration、ConditionOn、spring.factories

## 五、面试核心问答

### 5.1 Spring 经典面试题

**问**：BeanFactory 和 ApplicationContext 的区别是什么？

> BeanFactory 是 Spring 底层的 IoC 容器，提供了基础的依赖注入能力，按需加载 Bean（lazy-load）。ApplicationContext 是 BeanFactory 的子接口，在启动时就创建所有单例 Bean，并额外提供国际化、事件传播、资源加载等企业级能力。日常开发中，ApplicationContext 是更常用的选择。

**问**：Spring 是如何解决循环依赖的？

> Spring 通过三级缓存解决构造器注入之外的循环依赖。一级缓存存成品 Bean，二级缓存存早期暴露的 Bean 引用，三级缓存存 Bean 工厂。原型 Bean 的循环依赖无法解决，构造器注入的循环依赖也无法解决（因为在实例化阶段就报错了）。

### 5.2 MyBatis 经典面试题

**问**：MyBatis 为什么能执行自定义 SQL？

> MyBatis 在初始化时通过 SqlSessionFactoryBuilder 读取 XML 配置，构建 Configuration 对象，其中包含了所有 Mapper 接口与 SQL 语句的映射关系。执行时，通过 SqlSession 获取 Mapper 代理对象（基于 JDK 动态代理），代理对象的invoke方法根据方法签名找到对应 SQL，通过 StatementHandler 执行。

### 5.3 Spring Security 经典面试题

**问**：Spring Security 的过滤器链是如何工作的？

> 客户端请求到达后，首先经过 FilterChainProxy（这是 Security 的大门），它维护了一个有序的过滤器链。每个过滤器负责特定的职责：WebAsyncManagerIntegrationFilter 负责异步上下文、SecurityContextPersistenceFilter 负责 SecurityContext 的读写、HeaderWriterFilter 负责响应头……最终到达 FilterSecurityInterceptor，由它决定是否放行。

## 总结

框架原理的学习，不是背 API、不是背配置，而是理解设计者为什么要这样设计、这样设计解决了什么问题、有什么代价和局限。

Spring 的 IoC 解决了对象创建和依赖管理的耦合问题；AOP 解决了横切关注点散落的问题；MyBatis 解决了 SQL 控制的灵活性问题；Spring Security 解决了企业级安全的标准化问题。理解了这些「为什么」，框架的使用和源码就变得自然而然了。
