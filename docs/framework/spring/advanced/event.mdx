# Spring 事件机制：ApplicationEvent 与 @EventListener

import { Badge } from '@rspress/core/theme';

<Badge text="核心" type="tip" />

你有没有遇到过这种场景：用户注册成功后，需要同时发送欢迎邮件、初始化积分、记录操作日志……如果把这些逻辑都塞在 `UserService.register()` 方法里，代码会越来越臃肿。

更好的方式是：**发布一个「用户注册成功」的事件，然后让各个模块监听这个事件，自己决定要做什么**。这就是 Spring 的事件机制。

Spring 的事件机制基于**观察者模式**，实现了组件间的**松耦合通信**。当一个 Bean 需要通知其他 Bean 时，不需要直接持有它们的引用，只需要发布一个事件，Spring 会负责把事件分发给所有监听者。

## 一、基本概念

### 三个核心角色

| 角色 | 说明 | 接口/类 |
| --- | --- | --- |
| 事件 | 要传递的消息 | `ApplicationEvent` |
| 发布者 | 发送事件的组件 | `ApplicationEventPublisher` |
| 监听器 | 处理事件的组件 | `ApplicationListener` / `@EventListener` |

### 事件类

自定义事件需要继承 `ApplicationEvent`：

```java
public class UserRegisteredEvent extends ApplicationEvent {
    
    private final User user;
    
    public UserRegisteredEvent(Object source, User user) {
        super(source);
        this.user = user;
    }
    
    public User getUser() {
        return user;
    }
}
```

### 发布事件

通过 `ApplicationEventPublisher` 发布事件：

```java
@Service
public class UserService {
    
    @Autowired
    private ApplicationEventPublisher eventPublisher;
    
    public void register(User user) {
        // 1. 保存用户
        userRepository.save(user);
        
        // 2. 发布事件
        eventPublisher.publishEvent(
            new UserRegisteredEvent(this, user)
        );
    }
}
```

### 监听事件

```java
@Component
public class EmailService {
    
    @EventListener
    public void handleUserRegistered(UserRegisteredEvent event) {
        User user = event.getUser();
        System.out.println("发送欢迎邮件给: " + user.getEmail());
    }
}

@Component
public class PointService {
    
    @EventListener
    public void handleUserRegistered(UserRegisteredEvent event) {
        User user = event.getUser();
        System.out.println("给用户增加积分: " + user.getId());
    }
}
```

当 `UserService.register()` 被调用时，`EmailService` 和 `PointService` 的处理方法会自动被调用。

## 二、@EventListener 详解

### 基本用法

`@EventListener` 可以标注在任意 Spring Bean 的方法上：

```java
@Component
public class OrderService {
    
    @EventListener
    public void handleOrderCreated(OrderCreatedEvent event) {
        // 处理订单创建事件
    }
}
```

Spring 会自动检测所有标注了 `@EventListener` 的方法，将对应类型的事件分发给它们。

### 条件监听

`@EventListener` 支持 `condition` 属性，可以设置监听条件：

```java
@EventListener(
    condition = "#event.order.totalAmount > 100"
)
public void handleHighValueOrder(OrderCreatedEvent event) {
    // 只处理订单金额 > 100 的订单
    System.out.println("高价值订单: " + event.getOrder().getId());
}
```

SpEL 表达式 `#event` 指向事件对象。

### 泛型支持

Spring 支持泛型事件监听：

```java
@EventListener
public void handleUserEvent(UserEvent<?> event) {
    // 可以处理所有 UserEvent 的子类
}

@EventListener
public void handleStringEvent(GenericApplicationEvent<String> event) {
    // 处理泛型为 String 的事件
}
```

## 三、@TransactionalEventListener

普通 `@EventListener` 监听器**在事务提交后执行**，而 `@TransactionalEventListener` 允许你**控制监听器在事务的哪个阶段执行**。

### 四种事务阶段

| 阶段 | 说明 |
| --- | --- |
| `BEFORE_COMMIT` | 事务提交前执行 |
| `COMMIT` | 事务提交时执行 |
| `ROLLBACK` | 事务回滚时执行 |
| `AFTER_COMMIT` | 事务提交后执行（默认） |
| `AFTER_ROLLBACK` | 事务回滚后执行 |
| `AFTER_COMPLETION` | 事务完成后执行（无论提交还是回滚）|

### 使用示例

```java
@Service
public class PaymentService {
    
    @Autowired
    private ApplicationEventPublisher publisher;
    
    @Transactional
    public void pay(Order order) {
        // 1. 扣款
        accountService.deduct(order.getUserId(), order.getAmount());
        
        // 2. 标记订单已支付
        order.setStatus(OrderStatus.PAID);
        orderRepository.save(order);
        
        // 3. 发布支付成功事件
        publisher.publishEvent(new PaymentSuccessEvent(this, order));
    }
}

@Component
public class InvoiceService {
    
    @TransactionalEventListener(phase = AFTER_COMMIT)
    public void handlePaymentSuccess(PaymentSuccessEvent event) {
        // 只有事务成功提交后，才生成发票
        // 如果事务回滚，这个方法不会被执行
        invoiceGenerator.generate(event.getOrder());
    }
}
```

> [!TIP]
> `@TransactionalEventListener` 是处理**事务相关业务**的首选。比如支付成功后发短信、邮件，生成发票等——这些操作只有在事务真正提交后才应该执行。

## 四、同步与异步事件

### 普通事件的同步执行

默认情况下，事件监听器是**同步执行**的：

```java
@EventListener
public void handleEvent(MyEvent event) {
    // 发布者在主线程执行
    // 所有监听器按顺序在主线程执行
}
```

发布者需要等待所有监听器执行完毕后，才能继续。

### 异步事件的实现

通过 `@Async` 注解实现异步监听：

```java
@Component
public class NotificationService {
    
    @Async
    @EventListener
    public void handleUserRegistered(UserRegisteredEvent event) {
        // 这个方法会在新的线程池中执行
        // 发布者不会等待这个方法执行完毕
        sendNotification(event.getUser());
    }
}
```

需要确保主类或配置类上标注了 `@EnableAsync`：

```java
@SpringBootApplication
@EnableAsync
public class Application {
}
```

### 异步事件的问题

异步事件虽然提升了性能，但也带来了问题：

1. **事务问题**：异步方法在事务提交后才执行，此时 `EntityManager` 已关闭，无法进行懒加载
2. **异常处理**：异步方法的异常不会传播给发布者
3. **顺序问题**：无法保证多个异步监听器的执行顺序

### 自定义异步执行器

```java
@Configuration
@EnableAsync
public class AsyncConfig implements AsyncConfigurer {
    
    @Override
    public Executor getAsyncExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(5);
        executor.setMaxPoolSize(10);
        executor.setQueueCapacity(25);
        executor.setThreadNamePrefix("event-");
        executor.initialize();
        return executor;
    }
}

@Component
public class HeavyProcessingService {
    
    @Async("taskExecutor")  // 指定使用哪个线程池
    @EventListener
    public void handleEvent(MyEvent event) {
        // 使用自定义线程池执行
    }
}
```

## 五、SmartApplicationListener

对于更复杂的监听需求，可以实现 `SmartApplicationListener` 接口：

```java
@Component
public class OrderedListener implements SmartApplicationListener {
    
    @Override
    public boolean supportsEventType(Class<? extends ApplicationEvent> eventType) {
        // 指定监听的事件类型
        return UserCreatedEvent.class.isAssignableFrom(eventType);
    }
    
    @Override
    public boolean supportsSourceType(Class<?> sourceType) {
        // 指定监听的事件来源类型
        return UserService.class.isAssignableFrom(sourceType);
    }
    
    @Override
    public int getOrder() {
        // 指定监听器的执行顺序，值越小越先执行
        return 0;
    }
    
    @Override
    public void onApplicationEvent(ApplicationEvent event) {
        UserCreatedEvent e = (UserCreatedEvent) event;
        System.out.println("处理事件: " + e.getUser());
    }
}
```

## 六、事件传递机制

### 事件在容器中的传播

Spring 事件的传播分为两步：

1. **多播（Multicast）**：`ApplicationEventMulticaster` 将事件分发给所有匹配的监听器
2. **广播（Broadcast）**：如果事件是 `ApplicationContextEvent`，会向上传播到父容器

```java
// AbstractApplicationEventMulticaster 伪代码
public void multicastEvent(ApplicationEvent event, 
                           @Nullable ResolvableType eventType) {
    // 1. 获取所有匹配的监听器
    Collection<ApplicationListener<?>> listeners = 
        getApplicationListeners(event, eventType);
    
    // 2. 依次调用监听器
    for (ApplicationListener<?> listener : listeners) {
        listener.onApplicationEvent(event);
    }
}
```

### 父子容器的事件传播

如果存在父子容器，事件会向上传播：

```java
// 父容器发布事件
((ClassPathXmlApplicationContext) parentContext)
    .publishEvent(new MyEvent(this));

// 事件会向上传播，父容器的监听器也会收到
```

## 七、实际应用场景

### 场景一：用户操作日志

```java
// 事件定义
public class BusinessLogEvent extends ApplicationEvent {
    private final String action;
    private final Long userId;
    private final String detail;
    
    public BusinessLogEvent(Object source, String action, 
                           Long userId, String detail) {
        super(source);
        this.action = action;
        this.userId = userId;
        this.detail = detail;
    }
}

// 发布
public class UserService {
    public void disable(Long userId) {
        userRepository.disable(userId);
        eventPublisher.publishEvent(
            new BusinessLogEvent(this, "disable_user", userId, "禁用用户")
        );
    }
}

// 监听
@Component
public class AuditLogService {
    @TransactionalEventListener(phase = AFTER_COMMIT)
    public void handle(BusinessLogEvent event) {
        auditLogRepository.save(
            new AuditLog(event.getAction(), 
                        event.getUserId(), 
                        event.getDetail())
        );
    }
}
```

### 场景二：缓存清除

```java
@Component
public class CacheInvalidationService {
    
    @EventListener
    @Async
    public void handleDataChange(DataChangeEvent event) {
        // 数据变更时，异步清除相关缓存
        cacheManager.evict(event.getCacheName());
    }
}
```

## 总结

Spring 事件机制的核心要点：

1. **`ApplicationEvent`** 定义事件内容
2. **`ApplicationEventPublisher.publishEvent()`** 发布事件
3. **`@EventListener`** 监听事件，最简洁的方式
4. **`@TransactionalEventListener`** 控制监听器在事务的哪个阶段执行
5. **`@Async` + `@EventListener`** 实现异步监听
6. **`SmartApplicationListener`** 实现更精细的控制（顺序、来源类型）

---

下次当你需要在业务方法中调用多个无关操作时，不要用 `service1.method(); service2.method(); service3.method();` 这种方式——发布一个事件，让监听器去处理，代码会更优雅，也更易于维护。
