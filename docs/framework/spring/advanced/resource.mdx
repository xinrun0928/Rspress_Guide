# Spring Resource：抽象一切可读的资源

import { Badge } from '@rspress/core/theme';

<Badge text="基础" type="tip" />

你有没有想过，Spring 如何统一处理这些不同的资源：

- classpath 下的配置文件：`classpath:application.yml`
- 文件系统上的文件：`file:///opt/app/config.json`
- HTTP 远程资源：`https://example.com/data.json`
- jar 包中的资源：`classpath:META-INF/spring.factories`

如果让你来实现，你会写一堆 `if-else` 来判断前缀，然后用不同的 API 去读取。Spring 做了同样的事，但它把这个抽象封装成了 **`Resource` 接口**。

## 一、Resource 接口

### 接口定义

```java
public interface Resource extends InputStreamSource {
    
    // 判断资源是否实际存在
    boolean exists();
    
    // 是否可读
    boolean isReadable();
    
    // 是否处于打开状态（流已被消耗）
    boolean isOpen();
    
    // 是否是文件
    boolean isFile();
    
    // 获取 URL
    URL getURL() throws IOException;
    
    // 获取 URI
    URI getURI() throws IOException;
    
    // 获取 File
    File getFile() throws IOException;
    
    // 获取内容长度
    long contentLength() throws IOException;
    
    // 获取最后修改时间
    long lastModified() throws IOException;
    
    // 创建相对资源
    Resource createRelative(String relativePath) throws IOException;
    
    // 获取文件名
    String getFilename();
    
    // 获取描述
    String getDescription();
}
```

### InputStreamSource

```java
public interface InputStreamSource {
    // 获取输入流
    InputStream getInputStream() throws IOException;
}
```

这是 Resource 接口的父接口，提供了最基本的"获取输入流"能力。

## 二、Resource 的实现类

Spring 提供了多种 Resource 实现，每种对应一种资源类型：

| 实现类 | 前缀 | 说明 |
| --- | --- | --- |
| `UrlResource` | `http://`、`ftp://`、`file://` | URL 资源 |
| `ClassPathResource` | `classpath:` | classpath 资源 |
| `FileSystemResource` | 无前缀（文件路径） | 文件系统资源 |
| `ServletContextResource` | 无前缀（Web 应用） | ServletContext 资源 |
| `ByteArrayResource` | 无 | 字节数组资源 |
| `InputStreamResource` | 无 | 输入流资源 |

### ClassPathResource

从 classpath 读取资源：

```java
Resource resource = new ClassPathResource("application.yml");

// 读取内容
InputStream is = resource.getInputStream();
Properties props = new Properties();
props.load(is);

// 读取为 File（可能失败）
File file = resource.getFile();

// 获取描述
String desc = resource.getDescription(); 
// "class path resource [application.yml]"
```

### FileSystemResource

操作文件系统：

```java
Resource resource = new FileSystemResource("/opt/app/config.yml");

// 获取文件
File file = resource.getFile();

// 判断是否存在
if (resource.exists()) {
    // 处理文件
}
```

### UrlResource

访问 URL 资源：

```java
Resource httpResource = new UrlResource(
    "https://example.com/config.json"
);

Resource ftpResource = new UrlResource(
    "ftp://ftp.example.com/data.csv"
);
```

### ByteArrayResource

内存中的字节数组作为资源：

```java
byte[] data = "Hello World".getBytes();
Resource resource = new ByteArrayResource(data);
```

## 三、ResourceLoader

`ResourceLoader` 是 Spring 用来加载 Resource 的接口：

```java
public interface ResourceLoader {
    
    // 通用的资源路径前缀
    String CLASSPATH_URL_PREFIX = "classpath:";
    
    // 根据路径加载资源
    Resource getResource(String location);
    
    // 获取类加载器
    ClassLoader getClassLoader();
}
```

### DefaultResourceLoader

默认实现，智能判断路径前缀：

```java
Resource resource = resourceLoader.getResource("classpath:application.yml");
// -> ClassPathResource

resource = resourceLoader.getResource("file:///opt/app/config.json");
// -> FileSystemResource

resource = resourceLoader.getResource("http://example.com/config.json");
// -> UrlResource
```

### ApplicationContext 与 ResourceLoader

所有 `ApplicationContext` 都实现了 `ResourceLoader`：

```java
// Spring 容器本身就实现了 ResourceLoader
ApplicationContext ctx = new ClassPathXmlApplicationContext();

Resource resource = ctx.getResource("classpath:config.xml");
Resource resource2 = ctx.getResource("file:///opt/data.json");
Resource resource3 = ctx.getResource("http://example.com/api.json");
```

所以可以直接在 Bean 中注入 `ApplicationContext` 来加载资源：

```java
@Component
public class ConfigLoader {
    
    @Autowired
    private ApplicationContext ctx;
    
    public void loadConfig() {
        Resource resource = ctx.getResource("classpath:config.yml");
        // 读取资源
    }
}
```

## 四、资源路径通配符

Spring 支持在资源路径中使用通配符：

### classpath*:

`classpath*:` 可以匹配多个 classpath 下的同名资源：

```java
// 加载所有 classpath 下的 META-INF/spring.factories
Resource[] resources = ctx.getResources(
    "classpath*:META-INF/spring.factories"
);
```

这在处理**多个 jar 包**中的资源时特别有用。比如 Spring Boot 的自动配置，会扫描所有 jar 包中的 `META-INF/spring.factories` 文件。

### Ant 风格通配符

| 模式 | 说明 | 示例 |
| --- | --- | --- |
| `?` | 匹配单个字符 | `???` 匹配 `abc` |
| `*` | 匹配路径中的零或多个目录 | `*/config.yml` 匹配 `a/config.yml`、`b/config.yml` |
| `**` | 匹配任意路径 | `**/config.yml` 匹配 `a/b/config.yml` |

```java
// 加载所有子目录下的配置文件
Resource[] resources = ctx.getResources(
    "classpath*:META-INF/*-config.xml"
);

// 加载所有 jar 包中的 properties 文件
Resource[] resources = ctx.getResources(
    "classpath*:META-INF/**/*.properties"
);
```

## 五、@Value 注入 Resource

Spring 支持通过 `@Value` 直接注入 Resource：

```java
@Component
public class FileProcessor {
    
    @Value("classpath:data.json")
    private Resource jsonFile;
    
    @Value("file:///opt/data/input.csv")
    private Resource csvFile;
    
    public void process() throws IOException {
        InputStream is = jsonFile.getInputStream();
        // 读取文件内容
    }
}
```

### 带默认值的注入

```java
@Value("classpath:default-config.yml")
private Resource configFile;  // 不存在时抛出异常

@Value("classpath:optional.properties")
private Resource optionalFile;  // 不存在时为 null
```

## 六、PathMatchingResourcePatternResolver

Spring 用来解析路径模式的核心类：

```java
PathMatchingResourcePatternResolver resolver = 
    new PathMatchingResourcePatternResolver();

Resource[] resources = resolver.getResources(
    "classpath*:mapper/**/*.xml"
);
```

### 内部实现原理

```java
// 简化版源码
public class PathMatchingResourcePatternResolver {
    
    public Resource[] getResources(String locationPattern) 
            throws IOException {
        
        if (locationPattern.startsWith(CLASSPATH_ALL_URL_PREFIX)) {
            // classpath*: 开头的模式
            if (getPathMatcher().isPattern(
                    locationPattern.substring(CLASSPATH_ALL_URL_PREFIX.length()))) {
                // 有通配符，需要遍历
                return findPathMatchingResources(locationPattern);
            } else {
                // 没有通配符，直接加载
                return getResourcesFromClassPath(locationPattern);
            }
        } else {
            // 普通路径
            return new Resource[] { getResourceLoader().getResource(locationPattern) };
        }
    }
    
    private Resource[] findPathMatchingResources(String locationPattern) 
            throws IOException {
        // 1. 去除通配符，获取根路径
        String rootDirPath = determineRootDir(locationPattern);
        
        // 2. 获取根路径下的所有资源
        Resource[] rootResources = 
            findAllMatchingResources(rootDirPath);
        
        // 3. 对每个资源应用通配符匹配
        // ...
    }
}
```

## 七、实际应用场景

### 场景一：加载多个 Mapper XML

```java
PathMatchingResourcePatternResolver resolver = 
    new PathMatchingResourcePatternResolver();

Resource[] mapperLocations = resolver.getResources(
    "classpath*:mapper/**/*.xml"
);

for (Resource mapperLocation : mapperLocations) {
    System.out.println("加载 Mapper: " + mapperLocation.getFilename());
}
```

### 场景二：读取配置模板

```java
@Component
public class TemplateLoader {
    
    @Value("classpath:templates/*.ftl")
    private Resource[] templates;
    
    public void loadTemplates() {
        for (Resource template : templates) {
            String name = template.getFilename();
            InputStream is = template.getInputStream();
            // 处理模板
        }
    }
}
```

### 场景三：动态加载资源

```java
@Component
public class DynamicResourceLoader {
    
    @Autowired
    private ApplicationContext ctx;
    
    public void loadExternalConfig(String path) {
        Resource resource = ctx.getResource(path);
        if (!resource.exists()) {
            throw new IllegalArgumentException(
                "Resource not found: " + path);
        }
        
        // 根据不同类型处理
        if (resource.isFile()) {
            File file = resource.getFile();
            // 处理文件
        } else {
            InputStream is = resource.getInputStream();
            // 处理输入流
        }
    }
}
```

## 总结

Spring Resource 的核心要点：

1. **`Resource` 接口**统一了所有资源的访问方式
2. **多种实现类**对应不同资源来源：classpath、file、URL、ServletContext
3. **`ResourceLoader`** 是加载资源的入口，`ApplicationContext` 实现了该接口
4. **`classpath*:`** 可以匹配多个 classpath 下的同名资源
5. **Ant 风格通配符**：`?`、`*`、`**` 支持灵活的资源匹配
6. **`PathMatchingResourcePatternResolver`** 是解析路径模式的核心类

---

下次当你需要读取配置文件时，不要再写 `new FileInputStream("...")` 了——用 `Resource` 和 `ApplicationContext.getResource()`，代码会更简洁，也更易于测试和扩展。
