# BeanFactory vs ApplicationContext

你知道你在 Spring Boot 项目中注入的 `ApplicationContext` 是什么吗？它和 `BeanFactory` 是什么关系？两者在性能、功能、使用场景上有什么区别？

很多开发者只会用 `ApplicationContext`，甚至不知道 `BeanFactory` 的存在。但如果你能说清楚两者的区别，面试官对你的印象分会高不少。

## 一、继承关系：ApplicationContext 扩展了 BeanFactory

```
BeanFactory（接口）
├── ListableBeanFactory
│   └── AbstractBeanFactory
│       └── DefaultListableBeanFactory
│
└── ApplicationContext（接口）
    ├── ConfigurableApplicationContext
    │   └── AbstractApplicationContext
    │       ├── ClassPathXmlApplicationContext
    │       ├── FileSystemXmlApplicationContext
    │       ├── AnnotationConfigApplicationContext
    │       └── ...（各种具体实现）
    │
    ├── WebApplicationContext（接口）
    │   └── AnnotationConfigWebApplicationContext
    │   └── XmlWebApplicationContext
    │
    └── ConfigurableWebApplicationContext（接口）
```

从继承图可以看出：
- `BeanFactory` 是顶层接口，定义了容器的基本功能
- `ApplicationContext` 继承了多个接口，在 `BeanFactory` 基础上扩展了大量功能

## 二、BeanFactory：最基础的 IoC 容器

### 2.1 BeanFactory 的核心方法

```java
public interface BeanFactory {
    // 根据名称获取 Bean
    Object getBean(String name);

    // 根据名称和类型获取 Bean
    <T> T getBean(String name, Class<T> requiredType);

    // 根据类型获取 Bean
    <T> T getBean(Class<T> requiredType);

    // 带参数的 getBean（工厂方法场景）
    Object getBean(String name, Object... args);

    // 检查容器中是否包含该 Bean
    boolean containsBean(String name);

    // 判断 Bean 是否为单例/原型
    boolean isSingleton(String name);
    boolean isPrototype(String name);

    // 获取 Bean 的类型
    Class<?> getType(String name);

    // 获取 Bean 的别名
    String[] getAliases(String name);
}
```

### 2.2 BeanFactory 的特点

**懒加载（Lazy Loading）**

`BeanFactory` 默认采用懒加载策略：只有调用 `getBean()` 时，才会真正创建 Bean。

```java
BeanFactory factory = new XmlBeanFactory(new ClassPathResource("beans.xml"));
// 此时所有 Bean 都不会创建
UserService userService = factory.getBean(UserService.class);
// 只有调用 getBean 时，UserService 才会被实例化
```

**只提供基本功能**

- Bean 的创建（通过工厂方法或构造器）
- 依赖注入
- 基本的生命周期管理

## 三、ApplicationContext：功能增强的 IoC 容器

### 3.1 ApplicationContext 的扩展功能

`ApplicationContext` 在 `BeanFactory` 基础上扩展了以下功能：

| 扩展功能 | 说明 | 相关接口/类 |
|---|---|---|
| 资源加载 | 从 classpath、文件系统、URL 加载资源 | `ResourceLoader` |
| 事件发布 | 支持应用内事件机制 | `ApplicationEventPublisher` |
| 国际化 | 支持多语言消息 | `MessageSource` |
| Environment |  profiles 和 properties | `EnvironmentCapable` |
| 生命周期管理 | refresh() 完整初始化 | `ConfigurableApplicationContext` |

### 3.2 ApplicationContext 的核心方法

```java
public interface ApplicationContext extends EnvironmentCapable,
        ListableBeanFactory, HierarchicalBeanFactory,
        MessageSource, ApplicationEventPublisher, ResourcePatternResolver {

    // 获取应用 ID
    String getId();

    // 获取应用名称
    String getApplicationName();

    // 获取父容器
    ApplicationContext getParent();

    // 获取 Bean 工厂
    ConfigurableBeanFactory getBeanFactory();

    // 获取环境
    Environment getEnvironment();

    // 获取资源加载器
    ResourcePatternResolver getResourceLoader();
}
```

### 3.3 ApplicationContext 的两种加载方式

**立即加载（默认）**

`ApplicationContext` 默认会在容器初始化时**立即创建所有单例 Bean**：

```java
ApplicationContext context = new ClassPathXmlApplicationContext("application.xml");
// 此时所有 singleton Bean 都已创建完成
```

**延迟加载**

可以通过配置实现懒加载：

```xml
<bean id="userService" class="com.example.UserService" lazy-init="true"/>
```

或者注解方式：

```java
@Lazy // 加在类上，整个 Bean 懒加载
@Service
public class UserService {
    @Lazy // 加在字段上，只该字段懒加载
    private SomeHeavyComponent component;
}
```

## 四、两者对比总结

| 对比维度 | BeanFactory | ApplicationContext |
|---|---|---|
| **加载方式** | 懒加载（按需创建） | 立即加载（启动时创建） |
| **启动速度** | 快 | 慢 |
| **内存占用** | 启动时占用少 | 启动时占用多 |
| **资源加载** | 不支持 | 支持 ResourceLoader |
| **事件机制** | 不支持 | 支持 ApplicationEvent |
| **国际化** | 不支持 | 支持 MessageSource |
| **profiles** | 不支持 | 支持 |
| **使用场景** | 资源受限环境 | 大多数企业应用 |

## 五、为什么大多数场景用 ApplicationContext？

### 5.1 功能更全面

`ApplicationContext` 提供了 `BeanFactory` 没有的功能：

**事件机制**

```java
@Service
public class OrderService implements ApplicationEventPublisherAware {
    private ApplicationEventPublisher publisher;

    @Override
    public void setApplicationEventPublisher(ApplicationEventPublisher publisher) {
        this.publisher = publisher;
    }

    public void createOrder(Order order) {
        // 业务逻辑
        publisher.publishEvent(new OrderCreatedEvent(this, order));
    }
}

// 监听器
@Component
public class OrderEventListener {
    @EventListener
    public void handleOrderCreated(OrderCreatedEvent event) {
        System.out.println("订单创建事件: " + event.getOrder());
    }
}
```

**国际化**

```java
@Configuration
public class AppConfig {
    @Bean
    public MessageSource messageSource() {
        ResourceBundleMessageSource source = new ResourceBundleMessageSource();
        source.setBasename("messages");
        return source;
    }
}

// 使用
@Service
public class UserService {
    @Autowired
    private MessageSource messageSource;

    public void sendEmail(String email) {
        String content = messageSource.getMessage(
            "email.template",
            new Object[]{email},
            Locale.getDefault()
        );
    }
}
```

### 5.2 配置更灵活

`ApplicationContext` 支持多种配置方式：

| 配置方式 | 说明 |
|---|---|
| XML 配置 | 传统的 `<bean>` 方式 |
| 注解配置 | `@Configuration` + `@Bean` |
| Java 配置 | 更灵活的编程式配置 |
| Groovy 配置 | 支持 Groovy 脚本配置 |

### 5.3 更适合企业应用

企业应用需要：
- 启动时发现配置问题（立即加载）
- 事件驱动的业务逻辑
- 多环境配置管理
- 资源国际化

这些 `ApplicationContext` 都原生支持，而 `BeanFactory` 需要手动扩展。

## 六、特殊场景：为什么还需要 BeanFactory？

### 6.1 资源受限环境

在移动端、嵌入式设备等资源受限的环境中，`BeanFactory` 的懒加载特性可以减少启动时的资源占用。

### 6.2 超大应用

如果应用有上万个 Bean，且不是所有 Bean 都会被用到，`BeanFactory` 可以加快启动速度。

### 6.3 轻量级框架集成

一些轻量级框架只需要 `BeanFactory` 的基本功能，不需要 `ApplicationContext` 的扩展特性。

## 七、源码解析：ApplicationContext .refresh() 做了什么？

`ApplicationContext` 的初始化过程是通过 `refresh()` 方法完成的：

```java
// AbstractApplicationContext.refresh() 核心流程
public void refresh() throws BeansException {
    synchronized (this.startupShutdownMonitor) {
        // 1. 准备上下文：设置启动日期、激活状态
        prepareRefresh();

        // 2. 获取 Bean 工厂
        ConfigurableListableBeanFactory beanFactory = obtainFreshBeanFactory();

        // 3. 准备 Bean 工厂：添加后置处理器、注册默认单例
        prepareBeanFactory(beanFactory);

        try {
            // 4. 子类扩展：允许 BeanFactoryPostProcessor
            postProcessBeanFactory(beanFactory);

            // 5. 执行 BeanFactoryPostProcessor
            invokeBeanFactoryPostProcessors(beanFactory);

            // 6. 注册 BeanPostProcessor
            registerBeanPostProcessors(beanFactory);

            // 7. 初始化消息源
            initMessageSource();

            // 8. 初始化事件广播器
            initApplicationEventMulticaster();

            // 9. 子类扩展：初始化特殊 Bean
            onRefresh();

            // 10. 注册监听器
            registerListeners();

            // 11. 初始化所有非懒加载的单例 Bean
            finishBeanFactoryInitialization(beanFactory);

            // 12. 完成刷新：发布事件
            finishRefresh();
        } catch (BeansException ex) {
            destroyBeans();
            cancelRefresh(ex);
            throw ex;
        }
    }
}
```

这个方法非常关键，后续的 [IoC 容器原理](../ioc-principle/startup-flow) 会详细分析。

## 八、面试追问方向

**「BeanFactory 是怎么实现懒加载的？」**

关键在于 `AbstractBeanFactory.getBean()` 方法：Bean 不会在容器启动时创建，只有调用 `getBean()` 时才触发创建逻辑。

**「ApplicationContext 启动很慢，有什么优化方法？」**

- 减少 `@ComponentScan` 范围
- 使用 `@Lazy` 延迟加载非必要 Bean
- 排除不需要的自动配置（`excludeFilters`）
- 使用 Spring Boot 的懒初始化：`spring.main.lazy-initialization=true`

**「ApplicationContext 继承的 HierarchicalBeanFactory 是什么？」**

表示容器可以有父子层级关系，子容器可以访问父容器的 Bean，但父容器不能访问子容器的 Bean。

---

## 总结

`BeanFactory` 和 `ApplicationContext` 的关系：

- `BeanFactory` 是基础接口，提供了 IoC 容器的基本功能
- `ApplicationContext` 扩展了 `BeanFactory`，增加了资源加载、事件机制、国际化等企业级功能
- 日常开发中，我们使用 `ApplicationContext`，它几乎满足所有场景需求
- `BeanFactory` 主要在资源受限或特殊定制场景下使用

理解两者的区别，有助于你在不同场景下做出正确的选择。
