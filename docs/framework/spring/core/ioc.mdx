# Spring IoC 容器：控制反转与依赖注入

你有没有想过，当你在代码里写 `@Autowired UserService userService` 时，Spring 到底做了什么？它是怎么知道要创建哪个 `UserService` 实例的？又是怎么把这个实例「变」出来的？

更关键的问题是：**为什么我们需要把对象的创建权交给容器？自己 new 一个不好吗？**

这一节，我们来彻底搞清楚 Spring 最核心的思想——**控制反转（IoC）** 和 **依赖注入（DI）**。

---

## 一、从「主动获取」到「被动注入」

### 1.1 传统写法：一个类自己负责创建依赖

假设我们有一个订单服务 `OrderService`，它依赖于用户服务 `UserService`：

```java
public class UserService {
    public User findById(Long id) {
        return new User(id, "张三");
    }
}

public class OrderService {
    private UserService userService;

    public OrderService() {
        // 自己创建依赖
        this.userService = new UserService();
    }

    public Order createOrder(Long userId, BigDecimal amount) {
        User user = userService.findById(userId);
        return new Order(user, amount);
    }
}
```

这段代码看起来没问题，但实际上埋下了几个隐患：

**问题一：紧耦合**

`OrderService` 直接依赖 `UserService` 的实现类。如果 `UserService` 需要改用 `UserServiceImpl`（比如加了缓存实现），`OrderService` 也要跟着改。

**问题二：难以测试**

单元测试时，你没法轻易替换 `UserService` 的实现。总不能为了测试，专门写一个 mock 类，还得改构造函数？

**问题三：违背单一职责**

`OrderService` 的职责是处理订单，但它还要操心依赖对象怎么创建。这不是它该管的事。

### 1.2 控制反转：把创建权交给容器

Spring 的解决方案是：**不要自己 new，把创建对象的权力「反转」给容器。**

```java
public class OrderService {
    private UserService userService;

    // 不再自己创建，而是通过构造器接收
    public OrderService(UserService userService) {
        this.userService = userService;
    }

    public Order createOrder(Long userId, BigDecimal amount) {
        User user = userService.findById(userId);
        return new Order(user, amount);
    }
}
```

现在，`OrderService` 不再负责创建 `UserService`，它只需要「接受」一个 `UserService` 实例即可。这个「接受」的动作，就是**依赖注入（DI）**。

## 二、依赖注入的三种方式

### 2.1 构造器注入（Constructor Injection）

通过构造函数注入依赖，Spring 推荐的方式。

```java
@Service
public class OrderService {
    private final UserService userService;
    private final OrderDao orderDao;

    // 构造器注入
    public OrderService(UserService userService, OrderDao orderDao) {
        this.userService = userService;
        this.orderDao = orderDao;
    }
}
```

**优点**：
- 依赖不可变（`final` 字段）
- 强制要求依赖必须存在（没有 null 的风险）
- 便于测试（直接传入 mock 对象）

**缺点**：
- 如果依赖较多，构造函数会很长

### 2.2 Setter 注入（Setter Injection）

通过 setter 方法注入依赖。

```java
@Service
public class OrderService {
    private UserService userService;
    private OrderDao orderDao;

    // setter 方法注入
    @Autowired
    public void setUserService(UserService userService) {
        this.userService = userService;
    }

    @Autowired
    public void setOrderDao(OrderDao orderDao) {
        this.orderDao = orderDao;
    }
}
```

**优点**：
- 可选依赖（setter 方法可以省略）
- 部分更新（可以单独替换某个依赖）

**缺点**：
- 依赖可以是 null
- 对象创建后状态不完整，需要检查依赖是否注入

### 2.3 字段注入（Field Injection）

直接注入到字段上，最简洁但不推荐。

```java
@Service
public class OrderService {
    @Autowired
    private UserService userService;

    @Autowired
    private OrderDao orderDao;
}
```

**优点**：
- 代码简洁

**缺点**：
- 依赖不可见（看不到构造函数，不知道有哪些依赖）
- 难以单元测试（需要反射或容器）
- 违背单一职责原则

> **最佳实践**：优先使用构造器注入。对于可选依赖，可以使用 `@Autowired(required = false)` 结合 setter 注入。

## 三、IoC 容器是如何工作的？

### 3.1 Bean 注册与扫描

当 Spring 启动时，它会扫描指定的包，把带有 `@Component`、`@Service`、`@Repository`、`@Controller` 注解的类注册为 Bean。

```java
// 配置类：指定扫描路径
@Configuration
@ComponentScan("com.example")
public class AppConfig {
    // 或者在 XML 中配置
    // <context:component-scan base-package="com.example"/>
}
```

扫描过程：

1. `ComponentScan` 指定要扫描的包路径
2. 对于每个类，检查是否标注了 stereotype 注解
3. 如果标注了，就创建一个 `BeanDefinition`，注册到容器中

**Stereotype 注解**（Spring 提供的四类组件注解）：

| 注解 | 语义 | 使用场景 |
|---|---|---|
| `@Component` | 通用组件 | 通用 Bean |
| `@Service` | 服务层组件 | Service 层 |
| `@Repository` | 数据访问层组件 | DAO 层 |
| `@Controller` | 控制器 | MVC 控制器 |

```java
// 这四个注解在功能上没有区别，都是 @Component 的别名
@Target({ElementType.TYPE})
@Retention(RetentionPolicy.RUNTIME)
@Documented
@Component
public @interface Service {
    @AliasFor(annotation = Component.class)
    String value() default "";
}
```

### 3.2 Bean 的创建过程

容器启动后，`BeanFactory` 会按需创建 Bean。以单例 Bean 为例：

```java
// 伪代码：getBean 的简化逻辑
public Object getBean(String beanName) {
    // 1. 检查是否已创建
    if (singletonObjects.containsKey(beanName)) {
        return singletonObjects.get(beanName);
    }

    // 2. 获取 BeanDefinition
    BeanDefinition bd = beanDefinitionMap.get(beanName);

    // 3. 创建 Bean
    Object bean = createBean(bd);

    // 4. 存入缓存
    singletonObjects.put(beanName, bean);

    return bean;
}
```

完整的创建过程包括：实例化 → 属性填充 → 初始化，具体细节见 [Bean 创建流程](../ioc-principle/bean-create)。

### 3.3 自动注入（Autowiring）

Spring 提供了多种自动注入的方式：

**byType**：按类型匹配

```java
// 如果容器中只有一个 UserService 类型的 Bean，就会自动注入
@Autowired
private UserService userService;
```

**byName**：按名称匹配

```java
// 注入名为 userService 的 Bean
@Qualifier("userService")
private UserService userService;

// 或者通过字段名匹配（不使用 @Qualifier）
@Autowired
private UserService userService; // 匹配名为 userService 的 Bean
```

**constructor**：构造器注入

```java
// Spring 会选择参数最多的构造器进行注入
@Service
public class OrderService {
    private final UserService userService;

    public OrderService(UserService userService) { // 自动注入
        this.userService = userService;
    }
}
```

## 四、Bean 的命名规则

每个 Bean 都有一个名称，用于在容器中唯一标识它。

**默认命名规则**：类名的首字母小写

| 类名 | 默认 Bean 名称 |
|---|---|
| `UserService` | `userService` |
| `OrderDAO` | `orderDAO` |
| `URLHelper` | `URLHelper`（三个连续大写不转小写） |

**自定义命名**：使用 `@Service("customName")`

```java
@Service("myUserService")
public class UserServiceImpl implements UserService {
    // Bean 名称为 myUserService
}
```

## 五、Bean 的作用域

Spring 默认创建的 Bean 是**单例**的，即容器中只有一个实例。

| 作用域 | 说明 | 使用场景 |
|---|---|---|
| `singleton` | 单例，整个容器一个实例 | 默认，共享的无状态 Bean |
| `prototype` | 多例，每次获取创建新实例 | 有状态的 Bean |
| `request` | 每次 HTTP 请求一个实例 | Web 应用 |
| `session` | 每次 HTTP Session 一个实例 | Web 应用 |
| `application` | ServletContext 生命周期一个实例 | Web 应用 |

详细内容见 [Spring Bean 作用域](./bean-scope)。

## 六、面试追问方向

**「IoC 和 DI 的区别是什么？」**

这是常见的概念辨析题：
- **IoC（控制反转）** 是一种设计思想，强调把对象的创建和依赖获取从应用代码转移到外部容器
- **DI（依赖注入）** 是 IoC 的一种具体实现方式，指容器在创建对象时主动将依赖注入到对象中
- 关系：DI 是 IoC 最常见的实现方式，但 IoC 不等于 DI（比如还有 DL（Dependency Lookup，依赖查找））

**「Spring 为什么推荐构造器注入？」**

- 依赖不可变：`final` 字段只能在构造时赋值
- 防止 null：构造器注入要求依赖必须存在
- 便于测试：直接调用构造器传入 mock 对象
- 线程安全：构造完成后，对象状态不会再变化

**「字段注入为什么不好？」**

- 不符合单一职责：类的依赖不可见
- 难以测试：需要反射或容器辅助
- 违背 OOP 封装：直接访问私有字段

---

## 总结

Spring IoC 容器的核心思想是**控制反转**——把对象的创建和依赖管理从应用代码转移到容器。

依赖注入的三种方式：
- **构造器注入**：推荐，强制依赖不可变
- **Setter 注入**：适合可选依赖
- **字段注入**：简洁但不可取
