# 构造器注入 vs Setter 注入 vs 字段注入

你在代码里是怎么注入依赖的？是 `@Autowired private UserService userService`？还是 `public UserService(UserService userService)`？

别小看这个问题——不同的注入方式，在代码可读性、可测试性、循环依赖处理上都有巨大差异。

这一节，我们来对比三种注入方式，搞清楚什么时候该用哪种。

## 一、三种注入方式一览

| 注入方式 | 代码示例 | 推荐程度 |
|---|---|---|
| **构造器注入** | `public UserService(UserDao userDao) { this.userDao = userDao; }` | ⭐⭐⭐⭐⭐ |
| **Setter 注入** | `@Autowired public void setUserDao(UserDao userDao) { ... }` | ⭐⭐⭐ |
| **字段注入** | `@Autowired private UserDao userDao;` | ⭐ |

---

## 二、构造器注入（Constructor Injection）

### 2.1 基本用法

```java
@Service
public class OrderService {
    private final UserService userService;
    private final OrderDao orderDao;

    // 构造器注入
    public OrderService(UserService userService, OrderDao orderDao) {
        this.userService = userService;
        this.orderDao = orderDao;
    }
}
```

### 2.2 Spring 如何选择构造器？

当类有多个构造器时，Spring 会：

1. **如果有只有一个构造器且参数 `>= 1`，直接使用**
2. **如果有多个构造器，使用参数最多的构造器**
3. **如果有 `@Autowired` 注解的构造器，使用该构造器**

```java
@Service
public class ExampleService {
    private DependencyA depA;
    private DependencyB depB;
    private DependencyC depC;

    // Spring 会选择这个构造器（参数最多）
    public ExampleService(DependencyA depA) {
        this.depA = depA;
    }

    // 不会被选中
    public ExampleService(DependencyA depA, DependencyB depB) {
        this.depA = depA;
        this.depB = depB;
    }

    // 优先使用（因为有 @Autowired）
    @Autowired
    public ExampleService(DependencyA depA, DependencyB depB, DependencyC depC) {
        this.depA = depA;
        this.depB = depB;
        this.depC = depC;
    }
}
```

### 2.3 构造器注入的优点

**1. 依赖不可变（final）**

```java
@Service
public class OrderService {
    private final UserService userService; // 只能是 final

    public OrderService(UserService userService) {
        this.userService = userService; // 构造时赋值
    }

    // userService 永远不为 null，不可能被重新赋值
    public void createOrder() {
        // 放心使用，不用担心 NPE
        userService.findById(1L);
    }
}
```

**2. 强制依赖必须存在**

构造器参数必须有，否则 Bean 创建失败。这比运行时 NPE 好多了。

**3. 便于测试**

```java
// 单元测试：直接传入 mock 对象
@Test
public void testOrderService() {
    UserService mockUserService = mock(UserService.class);
    OrderDao mockOrderDao = mock(OrderDao.class);

    // 直接 new，无需 Spring 容器
    OrderService orderService = new OrderService(mockUserService, mockOrderDao);

    // 测试逻辑
    orderService.createOrder(1L, BigDecimal.TEN);

    verify(mockUserService).findById(1L);
    verify(mockOrderDao).save(any(Order.class));
}
```

**4. 清晰的依赖关系**

看构造器签名，就知道这个类依赖什么。一目了然。

**5. 支持循环依赖解决**

Setter/字段注入的循环依赖可以通过三级缓存解决，但构造器注入不行。构造器注入逼着你设计好依赖关系。

### 2.4 构造器注入的缺点

**1. 参数过多时代码冗长**

```java
// 依赖 10 个服务的构造器
public UserService(
        UserDao userDao,
        RoleDao roleDao,
        PermissionDao permissionDao,
        CacheService cacheService,
        LogService logService,
        ConfigService configService,
        MessageService messageService,
        EmailService emailService,
        SmsService smsService,
        PushService pushService) {
    this.userDao = userDao;
    this.roleDao = roleDao;
    // ... 10 行赋值代码
}
```

**解决方案**：使用 Lombok `@RequiredArgsConstructor` 自动生成构造器

```java
@Service
@RequiredArgsConstructor // Lombok 自动生成构造器
public class UserService {
    private final UserDao userDao;
    private final RoleDao roleDao;
    private final PermissionDao permissionDao;
    // ... 更多依赖
    // 构造器由 Lombok 自动生成，无需手动编写
}
```

**2. 可选依赖处理不便**

如果某个依赖是可选的（可能为 null），构造器注入就不太方便。

## 三、Setter 注入（Setter Injection）

### 3.1 基本用法

```java
@Service
public class OrderService {
    private UserService userService;
    private OrderDao orderDao;

    @Autowired
    public void setUserService(UserService userService) {
        this.userService = userService;
    }

    @Autowired
    public void setOrderDao(OrderDao orderDao) {
        this.orderDao = orderDao;
    }
}
```

### 3.2 适用场景

**可选依赖**

```java
@Service
public class EmailService {
    private SmsService smsService; // 可选

    @Autowired(required = false)
    public void setSmsService(SmsService smsService) {
        this.smsService = smsService;
    }

    public void send(String message) {
        // smsService 可能为 null
        if (smsService != null) {
            smsService.send(message);
        }
        // 总是发邮件
        sendEmail(message);
    }
}
```

**需要运行时替换依赖**

```java
@Service
public class PaymentService {
    private PaymentGateway gateway;

    @Autowired
    public void setGateway(PaymentGateway gateway) {
        this.gateway = gateway;
    }

    // 可以在测试时替换为 mock
    // 或运行时根据配置切换实现
}
```

### 3.3 Setter 注入的缺点

**1. 依赖可能为 null**

对象创建后，依赖可能还没注入，使用时容易 NPE。

**2. 对象状态可能不完整**

业务方法可能在依赖注入之前被调用。

**3. 可读性差**

不知道哪些依赖是必需的，哪些是可选的。

## 四、字段注入（Field Injection）

### 4.1 基本用法

```java
@Service
public class OrderService {
    @Autowired
    private UserService userService;

    @Autowired
    private OrderDao orderDao;
}
```

### 4.2 字段注入的问题

**1. 依赖不可见**

看构造器不知道依赖什么，看字段也不直观。

```java
// 这个类依赖了什么？不知道！
@Service
public class SomeService {
    @Autowired
    private DependencyA depA;

    @Autowired
    private DependencyB depB;

    @Autowired
    private DependencyC depC;
}
```

**2. 难以单元测试**

```java
// 字段注入的类：测试困难
@Service
public class OrderService {
    @Autowired
    private UserService userService; // 无法直接赋值

    public void createOrder() {
        userService.findById(1L); // 可能 NPE
    }
}

// 测试：必须用反射或 Spring Test
@Test
public void testWithReflection() throws Exception {
    OrderService service = new OrderService();
    ReflectionTestUtils.setField(service, "userService", mockUserService);
    service.createOrder();
}
```

**3. 违背单一职责**

类的依赖关系隐藏在内部，不符合 OOP 的封装原则。

**4. 无法处理 final 字段**

```java
// final 和 @Autowired 不能同时使用
@Service
public class OrderService {
    @Autowired // 编译错误！
    private final UserService userService;
}
```

### 4.3 字段注入的唯一「优点」

代码简洁。但这个优点在上述缺点面前，不值一提。

---

## 五、三种方式对比总结

| 特性 | 构造器注入 | Setter 注入 | 字段注入 |
|---|---|---|---|
| **依赖不可变** | ✅（final） | ❌ | ❌ |
| **强制依赖必须存在** | ✅ | ❌ | ❌ |
| **便于测试** | ✅（直接 new） | ✅（可以替换） | ❌（需要反射） |
| **可选依赖支持** | ❌（需要重载） | ✅（required=false） | ✅ |
| **循环依赖支持** | ❌ | ✅ | ✅ |
| **代码可读性** | ✅（依赖清晰） | ⚠️（一般） | ❌ |
| **IDE 支持** | ✅（自动补全） | ✅ | ❌ |
| **推荐程度** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐ |

## 六、最佳实践

### 6.1 优先使用构造器注入

```java
@Service
@RequiredArgsConstructor // Lombok
public class UserService {
    private final UserDao userDao;
    private final RoleDao roleDao;
    private final PermissionDao permissionDao;
    private final CacheService cacheService;
}
```

### 6.2 可选依赖使用 Setter 注入

```java
@Service
@RequiredArgsConstructor
public class NotificationService {
    private final UserDao userDao;

    @Autowired(required = false)
    private SmsService smsService;

    @Autowired(required = false)
    private PushService pushService;
}
```

或者单独使用 Setter：

```java
@Service
public class EmailService {
    private SmsService smsService;

    @Autowired
    public void setSmsService(SmsService smsService) {
        this.smsService = smsService;
    }
}
```

### 6.3 完全不用字段注入

字段注入是「看起来简单」，实际上后患无穷。在代码审查时，应该明确禁止字段注入。

### 6.4 使用 @RequiredArgsConstructor 管理构造器

```java
// 不需要手动写构造器
@Service
@RequiredArgsConstructor
public class OrderService {
    private final UserService userService;
    private final OrderDao orderDao;
    private final PaymentService paymentService;
    // Lombok 自动生成构造器
}
```

## 七、与循环依赖的关系

### 7.1 为什么构造器注入不能解决循环依赖？

```java
@Service
public class ServiceA {
    private final ServiceB serviceB;

    public ServiceA(ServiceB serviceB) { // 构造器执行时，serviceB 必须存在
        this.serviceB = serviceB;
    }
}
```

Spring 创建 A 时，必须先创建 B（作为构造器参数），但创建 B 又需要 A……死循环。

### 7.2 循环依赖时的选择

如果确实存在循环依赖，可以：

**方案一：重构代码**

最好的解决方案是重构，消除循环依赖。

**方案二：使用 @Lazy**

```java
@Service
public class ServiceA {
    private final ServiceB serviceB;

    public ServiceA(@Lazy ServiceB serviceB) {
        this.serviceB = serviceB;
    }
}
```

**方案三：Setter 注入**

```java
@Service
public class ServiceA {
    private ServiceB serviceB;

    @Autowired
    public void setServiceB(ServiceB serviceB) {
        this.serviceB = serviceB;
    }
}
```

## 八、面试追问方向

**「Spring 为什么推荐构造器注入？」**

主要四个原因：依赖不可变、强制依赖必须存在、便于测试、代码可读性好。

**「字段注入为什么不推荐？」**

依赖不可见、难以测试、违背 OOP 封装、无法使用 final。

**「构造器注入参数太多怎么办？」**

说明类职责太多，应该拆分。或者使用 Lombok `@RequiredArgsConstructor` 自动生成。

**「有循环依赖怎么办？」**

重构消除循环依赖，或使用 `@Lazy` 延迟注入，或改用 Setter 注入。

---

## 总结

依赖注入的三种方式各有优劣：

- **构造器注入**：最佳选择，适合必需依赖，代码清晰、安全、可测试
- **Setter 注入**：适合可选依赖，或需要在运行时替换依赖的场景
- **字段注入**：不推荐，测试困难，隐藏依赖关系

在实际项目中，**强烈建议使用构造器注入 + Lombok** 的组合，既保证了代码质量，又保持了简洁性。
