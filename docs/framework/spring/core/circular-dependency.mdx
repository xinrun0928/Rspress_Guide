# Bean 循环依赖：三级缓存与解决过程

import { Badge } from '@rspress/core/theme';

<Badge text="核心原理" type="danger" /> <Badge text="三级缓存" type="tip" /> <Badge text="面试高频" type="warning" />

你有没有遇到过这样的错误：`BeanCurrentlyInCreationException: Error creating bean with name 'xxx': Requested bean is currently in creation: Is there an unresolvable circular reference?`

这就是循环依赖——A 依赖 B，B 依赖 A，Spring 在创建 A 时发现需要 B，开始创建 B，又发现 B 需要 A……死循环了。

但奇怪的是，有时候循环依赖能解决，有时候不能。这是为什么？

这一节，我们来彻底搞清楚 Spring 的三级缓存机制，理解循环依赖是如何被解决的，以及哪些情况下循环依赖无法处理。

---

## 一、什么是循环依赖？

### 1.1 循环依赖的三种形态

循环依赖是指两个或多个 Bean 之间相互依赖，形成闭环：

```
// 形态一：直接循环依赖
A → B → A

// 形态二：间接循环依赖
A → B → C → A

// 形态三：自我依赖（极少数场景）
A → A
```

### 1.2 循环依赖的发生场景

循环依赖通常发生在以下几种情况：

**构造器注入循环依赖**

```java
@Service
public class ServiceA {
    private final ServiceB serviceB;

    public ServiceA(ServiceB serviceB) {
        this.serviceB = serviceB;
    }
}

@Service
public class ServiceB {
    private final ServiceA serviceA;

    public ServiceB(ServiceA serviceA) {
        this.serviceA = serviceA;
    }
}
```

**Setter 注入循环依赖（单例模式下可以解决）**

```java
@Service
public class ServiceA {
    private ServiceB serviceB;

    @Autowired
    public void setServiceB(ServiceB serviceB) {
        this.serviceB = serviceB;
    }
}

@Service
public class ServiceB {
    private ServiceA serviceA;

    @Autowired
    public void setServiceA(ServiceA serviceA) {
        this.serviceA = serviceA;
    }
}
```

**prototype 作用域循环依赖（无法解决）**

```java
@Scope("prototype")
@Service
public class ServiceA {
    @Autowired
    private ServiceB serviceB;
}

@Scope("prototype")
@Service
public class ServiceB {
    @Autowired
    private ServiceA serviceA;
}
```

## 二、三级缓存机制详解

### 2.1 三级缓存的数据结构

Spring 使用三个 `Map` 来解决单例 Bean 的循环依赖：

```java
// DefaultSingletonBeanRegistry 中的三个缓存

/** 第一级缓存：已经完全初始化好的 Bean，可以直接使用 */
private final Map<String, Object> singletonObjects = new ConcurrentHashMap<>(256);

/** 第二级缓存：已实例化但未完成属性填充和初始化的 Bean（早期引用） */
private final Map<String, Object> earlySingletonObjects = new ConcurrentHashMap<>(16);

/** 第三级缓存：Bean 的工厂对象，用于创建早期引用 */
private final Map<String, ObjectFactory<?>> singletonFactories = new ConcurrentHashMap<>(16);
```

### 2.2 三级缓存的作用

| 缓存 | 内容 | 用途 |
|---|---|---|
| **singletonObjects** | 完整的 Bean | 正式交付给应用使用 |
| **earlySingletonObjects** | 半成品 Bean | 解决循环依赖 |
| **singletonFactories** | 工厂对象 | 延迟创建早期引用 |

### 2.3 为什么需要三级？两级不够吗？

这是一个经典的追问。想象一下：

**如果只有两级**（singletonObjects + earlySingletonObjects）：

```
场景：A 依赖 B，B 依赖 A

1. 创建 A，开始实例化 A（半成品）
2. 发现 A 依赖 B，尝试创建 B
3. 创建 B，开始实例化 B（半成品）
4. 发现 B 依赖 A，尝试从 earlySingletonObjects 获取 A
5. 如果 A 还没放入 earlySingletonObjects，就拿不到
```

**关键问题**：A 实例化后，什么时候放入 earlySingletonObjects？

- 如果实例化后立即放入，那么当 B 获取 A 时，确实能拿到 A 的早期引用
- 但问题是，A 还没完成初始化，B 拿到的是「半成品 A」，调用 A 的方法可能会有问题

**三级缓存的意义**：

```
singletonFactories 解决的是「提前暴露引用」和「可能需要代理」的矛盾：

1. A 实例化后，立刻把「创建 A 的工厂」放入 singletonFactories
2. B 需要 A 时，从 singletonFactories 获取工厂，工厂可以创建 A 的代理（如果有 AOP）
3. 代理对象放入 earlySingletonObjects，B 拿到的是代理对象
4. 最终，A 初始化完成后，放入 singletonObjects，替换掉半成品
```

## 三、循环依赖解决流程

### 3.1 场景分析：Setter 注入的循环依赖

```
A (@Service) → B (@Service)，B (@Service) → A (@Service)
```

### 3.2 详细解决过程

```java
// getSingleton(beanName) 的核心逻辑
protected Object getSingleton(String beanName, boolean allowEarlyReference) {
    // 1. 检查第一级缓存：是否已经初始化完成
    Object singletonObject = singletonObjects.get(beanName);
    if (singletonObject == null && isSingletonCurrentlyInCreation(beanName)) {
        // 2. 检查第二级缓存：是否有早期引用
        synchronized (earlySingletonObjects) {
            singletonObject = earlySingletonObjects.get(beanName);
            if (singletonObject == null && allowEarlyReference) {
                // 3. 检查第三级缓存：是否有工厂
                ObjectFactory<?> singletonFactory = singletonFactories.get(beanName);
                if (singletonFactory != null) {
                    // 工厂存在，创建早期引用
                    singletonObject = singletonFactory.getObject();
                    // 放入第二级缓存
                    earlySingletonObjects.put(beanName, singletonObject);
                    // 从第三级缓存移除
                    singletonFactories.remove(beanName);
                }
            }
        }
    }
    return singletonObject;
}
```

**完整流程**：

```
时间线：
────────────────────────────────────────────────────────────────────────

T1: 创建 A
    ├─ doCreateBean("A")
    ├─ 标记 A 为「正在创建」：singletonsCurrentlyInCreation.add("A")
    ├─ createBeanInstance("A") → 创建 A 实例（半成品）
    ├─ 添加到三级缓存：singletonFactories.put("A", A的工厂)
    └─ populateBean("A") 开始填充属性

T2: 填充 A 的属性时，发现依赖 B
    ├─ getBean("B") → B 不存在，开始创建 B
    │
    │  创建 B
    ├─ doCreateBean("B")
    ├─ 标记 B 为「正在创建」：singletonsCurrentlyInCreation.add("B")
    ├─ createBeanInstance("B") → 创建 B 实例（半成品）
    ├─ 添加到三级缓存：singletonFactories.put("B", B的工厂)
    └─ populateBean("B") 开始填充属性

T3: 填充 B 的属性时，发现依赖 A
    ├─ getSingleton("A")
    │  ├─ 第一级缓存没有
    │  ├─ 第二级缓存没有
    │  └─ 第三级缓存有 A 的工厂！
    │     ├─ 调用工厂.getObject() → 获取 A 的早期引用
    │     ├─ 放入第二级缓存：earlySingletonObjects.put("A", 早期A)
    │     └─ 从第三级缓存移除：singletonFactories.remove("A")
    └─ B 拿到了 A 的早期引用（半成品），完成填充

T4: B 初始化完成
    ├─ B 放入第一级缓存：singletonObjects.put("B", 完成的B)
    ├─ 从正在创建集合移除：singletonsCurrentlyInCreation.remove("B")
    └─ 返回完成的 B

T5: A 继续填充属性
    └─ 拿到完成的 B，完成填充

T6: A 初始化完成
    ├─ A 放入第一级缓存：singletonObjects.put("A", 完成的A)
    ├─ 从正在创建集合移除：singletonsCurrentlyInCreation.remove("A")
    └─ 返回完成的 A

完成！
```

## 四、三级缓存与 AOP 的关系

### 4.1 为什么需要 singletonFactories？

你可能会问：为什么不直接在半成品 Bean 创建后放入 earlySingletonObjects，而是要用工厂？

答案：**为了支持 AOP 代理**。

考虑这样的场景：

```java
@Service
public class ServiceA {
    @Autowired
    private ServiceB serviceB;
}

@Service
public class ServiceB {
    @Autowired
    private ServiceA serviceA;
}

// 假设 ServiceA 需要被 AOP 代理（比如加事务）
@Service
public class ServiceA {
    @Transactional
    public void doSomething() { }
}
```

如果没有工厂（singletonFactories），当 B 获取 A 的早期引用时，只能拿到原始对象（半成品），而不是 AOP 代理对象。这意味着 B 持有的 A 引用是未代理的，`@Transactional` 不会生效！

**解决方案**：使用 `ObjectFactory` 工厂：

```java
// singletonFactories 存储的是这样的工厂
singletonFactories.put("A", () -> {
    // 在这里可以应用后置处理器，生成代理对象
    return applyBeanPostProcessorAfterInitialization(new A(), "A");
});
```

当 B 需要 A 时，调用工厂方法，工厂会应用 `BeanPostProcessor`（包括 `AutoProxyCreator`），生成正确的代理对象。

### 4.2 什么时候放入哪一级？

```java
// AbstractAutowireCapableBeanFactory.doCreateBean()
protected Object doCreateBean(String beanName, RootBeanDefinition mbd, Object[] args) {
    // 1. 实例化
    BeanWrapper instanceWrapper = createBeanInstance(beanName, mbd, args);
    Object bean = instanceWrapper.getWrappedInstance();

    // 2. 放入三级缓存（实例化后，填充前）
    addSingletonFactory(beanName, () -> getEarlyBeanReference(beanName, mbd, bean));

    // 3. 填充属性（这里可能触发循环依赖解决）
    populateBean(beanName, mbd, instanceWrapper);

    // 4. 初始化
    initializeBean(beanName, bean, mbd);

    // 5. 检查循环依赖
    // ...

    return bean;
}
```

## 五、哪些循环依赖无法解决？

### 5.1 构造器循环依赖——无法解决

Spring 的三级缓存解决循环依赖依赖于**先实例化、后填充**的策略。对于构造器注入，依赖在实例化时就需要传入：

```java
@Service
public class ServiceA {
    private final ServiceB serviceB;

    // 构造器依赖，实例化时就必须拿到 B
    public ServiceA(ServiceB serviceB) {
        this.serviceB = serviceB; // 此时 B 还不存在！
    }
}
```

Spring 无法在调用构造器之前就创建依赖，因此构造器循环依赖会直接失败。

**解决方案**：
1. 改用 Setter 注入
2. 使用 `@Lazy` 延迟注入
3. 使用 `ObjectFactory` 延迟获取

```java
// 解决方案：@Lazy 延迟注入
@Service
public class ServiceA {
    private final ServiceB serviceB;

    public ServiceA(@Lazy ServiceB serviceB) {
        this.serviceB = serviceB;
    }
}
```

### 5.2 prototype 作用域循环依赖——无法解决

prototype Bean 每次获取都会创建新实例，Spring 不会缓存它们：

```java
@Scope("prototype")
@Service
public class ServiceA {
    @Autowired
    private ServiceB serviceB; // 每次获取 A 时，都会尝试创建新的 B
}
```

**原因**：三级缓存只作用于 singleton 作用域，prototype Bean 不走缓存。

### 5.3 多例 Bean 之间的循环依赖——无法解决

与 prototype 类似，多例 Bean 不会被缓存，因此无法通过三级缓存解决。

### 5.4 @Async 注解的 Bean 循环依赖——可能无法解决

`@Async` 会生成 `AsyncAnnotationBeanPostProcessor`，它会替换掉 `BeanPostProcessor` 的默认行为，可能导致早期引用获取不到正确的代理。

## 六、@Lazy 注解的原理

`@Lazy` 可以延迟 Bean 的初始化，包括解决循环依赖：

```java
@Service
public class ServiceA {
    private final ServiceB serviceB;

    public ServiceA(@Lazy ServiceB serviceB) {
        this.serviceB = serviceB;
    }
}
```

**原理**：Spring 会为 `@Lazy` 注解的依赖创建一个代理对象（代理到真正的 Bean）。只有当第一次调用代理方法时，才会真正触发 Bean 的创建，从而打破循环。

**源码位置**：`AutowiredAnnotationBeanPostProcessor.AutowiredFieldElement`

```java
// 注入 @Lazy 依赖时的处理
private void inject(Object bean, String beanName, PropertyValue pv) {
    Object value;
    if (this.cached) {
        value = resolvedCachedArgument(beanName, this.cachedFieldValue);
    } else {
        DependencyDescriptor descriptor = new DependencyDescriptor(...);
        value = beanFactory.resolveDependency(descriptor, beanName, autowiredBeanNames, typeConverter);

        // @Lazy 场景下，value 是代理对象
        // 真正的 Bean 在第一次调用时才创建
    }
    // 注入到字段
    ReflectionUtils.makeAccessible(field);
    field.set(bean, value);
}
```

## 七、面试追问方向

**「为什么第三级缓存要用 ObjectFactory，直接放 Bean 不行吗？」**

因为要解决 AOP 代理的问题。如果直接放 Bean，B 获取到的就是原始对象，而不是代理对象，注解（如 `@Transactional`）不会生效。通过工厂，在获取早期引用时可以动态生成代理。

**「循环依赖时，B 获取的 A 是什么状态？」**

是**半成品 A**，可能没有完成属性填充和初始化。这可能导致一些问题，所以 Spring 设计了 earlySingletonObjects 缓存，确保 B 不会重复创建 A。

**「如果 A 有多个构造器参数都依赖 B，B 也有多个构造器参数都依赖 A，Spring 会怎么处理？」**

这种情况肯定是构造器循环依赖，Spring 无法解决，会抛出 `BeanCurrentlyInCreationException`。

**「singletonFactories 什么时候被清理？」**

当 Bean 完全初始化完成后（放入 singletonObjects），会从 singletonFactories 和 earlySingletonObjects 中移除该 Bean 的记录。

---

## 总结

Spring 通过三级缓存解决单例 Bean 的 Setter 注入循环依赖：

- **第一级缓存（singletonObjects）**：完整 Bean，可直接使用
- **第二级缓存（earlySingletonObjects）**：早期引用，解决循环依赖
- **第三级缓存（singletonFactories）**：工厂，支持 AOP 代理

**无法解决的三种情况**：
1. 构造器注入循环依赖
2. prototype 作用域循环依赖
3. 多例 Bean 循环依赖

理解三级缓存的设计思想，是理解 Spring 循环依赖机制的关键。
