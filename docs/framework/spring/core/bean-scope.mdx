# Spring Bean 作用域

当你在一个 Service 中注入另一个 Service 时，有没有想过：这个被注入的 Bean 是每次都新建一个，还是大家共用同一个？

Spring 的 Bean 作用域机制回答了这个问题。理解不同作用域的区别，是避免潜在 bug 的前提。

## 一、作用域类型一览

Spring Framework 支持 6 种作用域：

| 作用域 | 说明 | 使用场景 |
|---|---|---|
| `singleton` | 单例，整个容器一个实例 | 默认，无状态 Bean |
| `prototype` | 多例，每次获取创建新实例 | 有状态 Bean |
| `request` | 每次 HTTP 请求一个实例 | Web 请求上下文 |
| `session` | 每次 HTTP Session 一个实例 | 用户会话数据 |
| `application` | ServletContext 生命周期一个实例 | 全局共享数据 |
| `websocket` | WebSocket 生命周期一个实例 | WebSocket 通信 |

---

## 二、singleton：默认也是最常用的作用域

`singleton` 是 Spring 默认的作用域，意味着容器中只有一个 Bean 实例。

```java
@Service
public class UserService {
    private String name = UUID.randomUUID().toString();

    public String getName() {
        return name;
    }
}

// 测试
@Test
public void testSingleton() {
    ApplicationContext context = new AnnotationConfigApplicationContext(UserService.class);

    UserService bean1 = context.getBean(UserService.class);
    UserService bean2 = context.getBean(UserService.class);

    System.out.println(bean1 == bean2); // true
    System.out.println(bean1.getName().equals(bean2.getName())); // true
}
```

**特点**：
- 在容器启动时创建（除非设置 `lazy-init="true"`）
- 容器存在期间，Bean 始终是同一个实例
- 销毁时也会被容器统一销毁

## 三、prototype：每次获取新实例

`prototype` 作用域下，每次 `getBean()` 都会创建一个新实例。

```java
@Scope("prototype")
@Service
public class OrderService {
    private String orderId = UUID.randomUUID().toString();

    public String getOrderId() {
        return orderId;
    }
}

// 测试
@Test
public void testPrototype() {
    ApplicationContext context = new AnnotationConfigApplicationContext(OrderService.class);

    OrderService bean1 = context.getBean(OrderService.class);
    OrderService bean2 = context.getBean(OrderService.class);

    System.out.println(bean1 == bean2); // false
    System.out.println(bean1.getOrderId().equals(bean2.getOrderId())); // false
}
```

**特点**：
- **懒加载**：只有在第一次获取时才创建
- **容器不管理销毁**：容器不会调用 prototype Bean 的销毁方法，需要手动管理
- **每次都是新对象**：适合有状态的 Bean

**重要提醒**：Spring 不会管理 prototype Bean 的完整生命周期！

```java
// 如果 OrderService 有 @PreDestroy 注解
@PreDestroy
public void cleanup() {
    System.out.println("清理资源");
}

// 容器关闭时，这个方法不会被调用！
```

## 四、Web 作用域：request、session、application

在 Web 应用中，Spring 提供了三种 Web 作用域。

### 4.1 request：每次请求一个实例

```java
@Scope("request")
@Service
public class RequestContextService {
    private String requestId = UUID.randomUUID().toString();

    public String getRequestId() {
        return requestId;
    }
}
```

**特点**：
- 每个 HTTP 请求创建一个新实例
- 请求结束，Bean 销毁
- 需要 `RequestContextListener` 或 `RequestContextFilter` 支持

### 4.2 session：每次会话一个实例

```java
@Scope("session")
@Service
public class ShoppingCartService {
    private List<Product> cartItems = new ArrayList<>();

    public void addItem(Product product) {
        cartItems.add(product);
    }

    public List<Product> getCartItems() {
        return cartItems;
    }
}
```

**特点**：
- 每个 HTTP Session 创建一个新实例
- Session 过期，Bean 销毁
- 需要 `SessionAttributeListener` 支持

### 4.3 application：应用级别一个实例

```java
@Scope("application")
@Service
public class ApplicationStatsService {
    private AtomicInteger visitCount = new AtomicInteger(0);

    public void recordVisit() {
        visitCount.incrementAndGet();
    }

    public int getVisitCount() {
        return visitCount.get();
    }
}
```

**特点**：
- 整个 ServletContext 生命周期一个实例
- 类似于 `singleton`，但不在 Spring IoC 容器层面单例
- 适合需要在 Web 应用级别共享的数据

## 五、作用域注解与配置

### 5.1 注解方式

```java
// 直接使用 @Scope 注解
@Scope("prototype")
@Service
public class MyService { }

// 或者使用注解的 value 属性
@Service
@Scope(scopeName = "prototype")
public class MyService { }

// 原型 + Web 请求
@Scope(value = "request", proxyMode = ScopedProxyMode.TARGET_CLASS)
@Service
public class RequestService { }
```

### 5.2 XML 配置方式

```xml
<bean id="myService" class="com.example.MyService" scope="prototype"/>
```

## 六、作用域代理（Scoped Proxy）

### 6.1 为什么要用代理？

考虑这样一个场景：

```java
@Service
public class OrderService {
    @Autowired
    private ShoppingCartService shoppingCart; // session 作用域
}
```

问题：`OrderService` 是 singleton，每次调用都使用同一个 `shoppingCart` 引用。但 `ShoppingCartService` 是 session 作用域的，不同用户的 session 不同。

**解决方案**：Spring 创建一个代理对象，每次调用代理时，根据当前请求的 session 返回正确的实例。

### 6.2 proxyMode 选项

| 模式 | 说明 |
|---|---|
| `ScopedProxyMode.DEFAULT` | 默认，不使用代理（等价于 NO） |
| `ScopedProxyMode.NO` | 不使用代理 |
| `ScopedProxyMode.INTERFACES` | 基于接口的 JDK 动态代理 |
| `ScopedProxyMode.TARGET_CLASS` | 基于类的 CGLIB 代理 |

### 6.3 完整示例

```java
@Service
public class OrderService {
    // 使用代理，每次调用都获取当前 session 的 ShoppingCartService
    @Autowired
    @Scope(value = "session", proxyMode = ScopedProxyMode.TARGET_CLASS)
    private ShoppingCartService shoppingCart;

    public void checkout() {
        // 每次调用 shoppingCart 都指向当前 session 的实例
        List<Product> items = shoppingCart.getCartItems();
        // ...
    }
}
```

**执行原理**：

```java
// Spring 实际创建的是这样的代理类
public class ShoppingCartServiceProxy implements ShoppingCartService {
    private ScopedObjectAccess scopedObjectAccess;

    @Override
    public List<Product> getCartItems() {
        // 获取当前 request 对应的 session 中的真实实例
        ShoppingCartService target = scopedObjectAccess.getTarget();
        return target.getCartItems();
    }
}
```

## 七、作用域与 Bean 生命周期

不同作用域的 Bean，容器的管理方式也不同：

| 作用域 | 实例化时机 | 销毁管理 | 懒加载 |
|---|---|---|---|
| `singleton` | 容器启动时 | 容器统一销毁 | 可配置 |
| `prototype` | 首次 getBean 时 | 不管理 | 默认懒加载 |
| `request` | 每次请求时 | 请求结束 | — |
| `session` | 首次访问 Session 时 | Session 过期 | — |
| `application` | 容器启动时 | ServletContext 销毁 | — |

## 八、常见问题与避坑

### 8.1 单例 Bean 注入多例 Bean

```java
@Service
public class SingletonService {
    @Autowired
    private PrototypeService prototypeService; // 问题！
}
```

**问题**：`SingletonService` 只会被创建一次，所以 `prototypeService` 也只会注入一次，违背了 prototype 的初衷。

**解决方案**：使用 `ObjectFactory` 或 `ObjectProvider`：

```java
@Service
public class SingletonService {
    @Autowired
    private ObjectFactory<PrototypeService> prototypeServiceFactory;

    public void doSomething() {
        // 每次调用 getObject() 都获取新实例
        PrototypeService prototypeService = prototypeServiceFactory.getObject();
    }
}
```

或者使用 `@Lookup` 方法注入：

```java
@Service
public abstract class SingletonService {
    @Lookup
    public abstract PrototypeService createPrototypeService();

    public void doSomething() {
        PrototypeService prototypeService = createPrototypeService();
    }
}
```

### 8.2 prototype Bean 的循环依赖

**结论**：prototype 作用域的 Bean **不支持循环依赖**。

原因：当 A 构造器依赖 B，B 构造器依赖 A 时，容器尝试创建 A，但 A 的构造器需要 B，容器尝试创建 B，但 B 的构造器需要 A……无限循环，抛出 `BeanCurrentlyInCreationException`。

```java
@Service
@Scope("prototype")
public class ServiceA {
    public ServiceA(ServiceB serviceB) { // 构造器循环依赖
    }
}

@Service
@Scope("prototype")
public class ServiceB {
    public ServiceB(ServiceA serviceA) { // 构造器循环依赖
    }
}
```

## 九、面试追问方向

**「singleton Bean 中可以注入 prototype Bean 吗？有什么问题？」**

可以注入，但存在生命周期问题：singleton Bean 只会被创建一次，所以 prototype Bean 也只会注入一次，违背了 prototype 的设计初衷。

解决方案：
1. 使用 `ObjectFactory` / `ObjectProvider`
2. 使用 `@Lookup` 方法注入
3. 实现 `ApplicationContextAware` 接口手动获取

**「Spring 默认 Bean 作用域为什么是 singleton？」**

因为大多数 Bean 是无状态的（Service、DAO），使用单例可以减少对象创建开销，提高性能。如果每个请求都创建新的 Service 实例，会造成大量对象创建和垃圾回收压力。

**「request 和 session 作用域的 Bean 有什么限制？」**

必须通过代理访问，因为这些 Bean 与当前线程（request）或会话（session）绑定。如果直接注入到 singleton Bean，调用时会获取不到正确的上下文。

---

## 总结

Spring Bean 的作用域决定了 Bean 实例的生命周期和使用范围：

- **singleton**：默认，容器唯一实例，适合无状态 Bean
- **prototype**：每次获取新实例，适合有状态 Bean，但容器不管理销毁
- **request / session / application**：Web 场景专用，需要代理才能在 singleton Bean 中使用

理解作用域的差异和适用场景，能够帮助你在设计中做出正确的选择。
