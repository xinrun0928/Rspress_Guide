# Spring 启动流程源码解析

你有没有想过，当你调用 `new AnnotationConfigApplicationContext(AppConfig.class)` 时，Spring 到底做了什么？几行启动代码背后，容器是如何一步步完成初始化的？

这一节，我们来深入 `AbstractApplicationContext.refresh()` 方法，搞清楚 Spring 启动的完整流程。

---

## 一、为什么是 refresh()？

在讲解启动流程之前，先回答一个常见问题：**为什么叫 `refresh()` 而不是 `start()`？**

`refresh()` 的设计体现了 Spring 的一个重要思想：**容器可以被刷新和重启**。

对比一下：
- `BeanFactory` 是不可变的，一旦创建就不能再添加 Bean
- `ApplicationContext` 支持刷新——可以重新加载配置、重新扫描组件

这个设计允许：
- 热加载配置（如 Spring Cloud Config）
- 动态注册 Bean
- 在不重启 JVM 的情况下更新应用

所以叫 `refresh()`（刷新），而不是 `start()`（启动），是有深意的。

---

## 二、refresh() 核心流程

`refresh()` 方法是 `AbstractApplicationContext` 的核心，整个 Spring 容器的初始化都在这个方法里完成。

### 2.1 十二步骤总览

```java
// AbstractApplicationContext.refresh() 源码
public void refresh() throws BeansException, IllegalStateException {
    synchronized (this.startupShutdownMonitor) {
        // 步骤 1：准备上下文
        prepareRefresh();

        // 步骤 2：获取 Bean 工厂
        ConfigurableListableBeanFactory beanFactory = obtainFreshBeanFactory();

        // 步骤 3：准备 Bean 工厂
        prepareBeanFactory(beanFactory);

        try {
            // 步骤 4：子类扩展（空实现，留给子类钩子）
            postProcessBeanFactory(beanFactory);

            // 步骤 5：执行 BeanFactoryPostProcessor
            invokeBeanFactoryPostProcessors(beanFactory);

            // 步骤 6：注册 BeanPostProcessor
            registerBeanPostProcessors(beanFactory);

            // 步骤 7：初始化消息源
            initMessageSource();

            // 步骤 8：初始化事件广播器
            initApplicationEventMulticaster();

            // 步骤 9：子类扩展（初始化特殊 Bean）
            onRefresh();

            // 步骤 10：注册监听器
            registerListeners();

            // 步骤 11：初始化所有非懒加载的单例 Bean
            finishBeanFactoryInitialization(beanFactory);

            // 步骤 12：完成刷新
            finishRefresh();

        } catch (BeansException ex) {
            destroyBeans();
            cancelRefresh(ex);
            throw ex;
        }
    }
}
```

### 2.2 步骤详解

#### 步骤 1：prepareRefresh() —— 准备上下文

```java
protected void prepareRefresh() {
    // 1. 设置启动日期
    this.startupDate = System.currentTimeMillis();

    // 2. 设置容器关闭标志为 false
    this.closed.set(false);
    this.active.set(true);

    // 3. 初始化 propertySources（空实现，留给子类扩展）
    initPropertySources();

    // 4. 验证必需的属性
    getEnvironment().validateRequiredProperties();
}
```

**关键点**：
- `initPropertySources()` 是空实现，Spring Boot 在这里加载 `application.properties`
- `validateRequiredProperties()` 验证必需属性，如果缺失会抛出异常

#### 步骤 2：obtainFreshBeanFactory() —— 获取 Bean 工厂

```java
protected ConfigurableListableBeanFactory obtainFreshBeanFactory() {
    // 刷新 BeanFactory
    refreshBeanFactory();
    // 返回 BeanFactory
    return getBeanFactory();
}
```

对于 `AnnotationConfigApplicationContext`，`refreshBeanFactory()` 的实现是：

```java
// GenericApplicationContext.refreshBeanFactory()
protected final void refreshBeanFactory() throws BeansException {
    // 如果已存在 BeanFactory，销毁其中的 Bean
    if (this.beanFactory != null) {
        this.beanFactory.destroySingletons();
    }
    // 创建新的 DefaultListableBeanFactory
    this.beanFactory = new DefaultListableBeanFactory();
    // 自定义 BeanFactory（空实现）
    customizeBeanFactory(this.beanFactory);
}
```

**关键点**：
- 每次 `refresh()` 都会创建一个**全新的** `DefaultListableBeanFactory`
- 旧的 Bean 会被销毁

#### 步骤 3：prepareBeanFactory() —— 准备 Bean 工厂

```java
protected void prepareBeanFactory(ConfigurableListableBeanFactory beanFactory) {
    // 1. 设置类加载器
    beanFactory.setBeanClassLoader(getClassLoader());

    // 2. 设置 SpEL 表达式解析器
    beanFactory.setBeanExpressionResolver(new StandardBeanExpressionResolver());

    // 3. 添加属性编辑器注册器（处理属性转换）
    beanFactory.addPropertyEditorRegistrar(new ResourceEditorRegistrar(this));

    // 4. 添加 ApplicationContextAwareProcessor
    beanFactory.addBeanPostProcessor(new ApplicationContextAwareProcessor(this));

    // 5. 设置忽略依赖注入的接口
    beanFactory.ignoreDependencyInterface(EnvironmentAware.class);
    beanFactory.ignoreDependencyInterface(EmbeddedValueResolverAware.class);
    beanFactory.ignoreDependencyInterface(ResourceLoaderAware.class);
    beanFactory.ignoreDependencyInterface(ApplicationEventPublisherAware.class);
    beanFactory.ignoreDependencyInterface(MessageSourceAware.class);
    beanFactory.ignoreDependencyInterface(ApplicationContextAware.class);

    // 6. 注册可解析的依赖
    beanFactory.registerResolvableDependency(BeanFactory.class, beanFactory);
    beanFactory.registerResolvableDependency(ResourceLoader.class, this);
    beanFactory.registerResolvableDependency(ApplicationEventPublisher.class, this);
    beanFactory.registerResolvableDependency(ApplicationContext.class, this);

    // 7. 添加 LoadTimeWeaverAwareProcessor
    beanFactory.addBeanPostProcessor(new LoadTimeWeaverAwareProcessor(beanFactory));

    // 8. 注册临时类加载器
    beanFactory.setTempClassLoader(new ContextTypeMatchClassLoader(beanFactory.getBeanClassLoader()));
}
```

**关键点**：
- `ApplicationContextAwareProcessor`：处理 `Aware` 接口回调
- `ignoreDependencyInterface()`：告诉 Spring 哪些接口不需要通过依赖注入填充
- `registerResolvableDependency()`：注册可解析的依赖，如 `BeanFactory.class` 注入 `beanFactory` 实例

#### 步骤 4：postProcessBeanFactory() —— 子类扩展

```java
protected void postProcessBeanFactory(ConfigurableListableBeanFactory beanFactory) {
    // 空实现，留给子类扩展
}
```

Spring Boot 在这里做了很多扩展，如：
- `EmbeddedServletContainerAutoConfiguration`
- `DispatcherServletAutoConfiguration`

#### 步骤 5：invokeBeanFactoryPostProcessors() —— 执行 BFPP

这是非常重要的步骤！`BeanFactoryPostProcessor` 可以在所有 Bean 创建之前修改 `BeanDefinition`。

```java
public static void invokeBeanFactoryPostProcessors(
        ConfigurableListableBeanFactory beanFactory,
        List<BeanFactoryPostProcessor> beanFactoryPostProcessors) {

    // 1. 首先执行 BeanDefinitionRegistryPostProcessor
    // （它是 BeanFactoryPostProcessor 的子类，可以注册新的 BeanDefinition）
    invokeBeanFactoryPostProcessors(
        getBeanFactoryPostProcessors(), null);

    // 2. 找到所有 BeanDefinitionRegistryPostProcessor
    // 3. 按优先级排序并执行
    String[] postProcessorNames =
        beanFactory.getBeanNamesForType(BeanDefinitionRegistryPostProcessor.class, true, false);
    for (String ppName : postProcessorNames) {
        if (beanFactory.isBeanNameInUse(ppName)) {
            continue;
        }
        BeanDefinitionRegistryPostProcessor pp =
            beanFactory.getBean(ppName, BeanDefinitionRegistryPostProcessor.class);
        // 执行
        pp.postProcessBeanDefinitionRegistry(beanFactory);
    }

    // 4. 找到所有普通的 BeanFactoryPostProcessor 并执行
    postProcessorNames = beanFactory.getBeanNamesForType(BeanFactoryPostProcessor.class, true, false);
    // 按优先级排序并执行
}
```

**关键点**：
- `BeanDefinitionRegistryPostProcessor`（如 `ConfigurationClassPostProcessor`）可以注册新的 `BeanDefinition`
- `BeanFactoryPostProcessor`（如 `PropertyPlaceholderConfigurer`）可以修改已有的 `BeanDefinition`

#### 步骤 6：registerBeanPostProcessors() —— 注册 BPP

`BeanPostProcessor` 会在每个 Bean 创建时调用，用于对 Bean 进行后置处理（如生成代理）。

```java
public static void registerBeanPostProcessors(
        ConfigurableListableBeanFactory beanFactory,
        List<BeanPostProcessor> beanFactoryPostProcessors) {

    // 1. 获取所有 BeanPostProcessor 的名称
    String[] postProcessorNames = beanFactory.getBeanNamesForType(
        BeanPostProcessor.class, true, false);

    // 2. 注册 PriorityOrdered 类型的 BPP
    // 3. 注册 Ordered 类型的 BPP
    // 4. 注册普通的 BPP
    // 5. 注册 MergedBeanDefinitionPostProcessor

    // 6. 创建一个 ApplicationListenerDetector 放到最后
    beanFactory.addBeanPostProcessor(new ApplicationListenerDetector());
}
```

**Spring 内置的 BPP**：

| BPP | 作用 |
|---|---|
| `AutowiredAnnotationBeanPostProcessor` | 处理 `@Autowired`、`@Value` |
| `CommonAnnotationBeanPostProcessor` | 处理 `@PostConstruct`、`@PreDestroy`、`@Resource` |
| `RequiredAnnotationBeanPostProcessor` | 处理 `@Required`（已废弃） |
| `ApplicationContextAwareProcessor` | 处理 Aware 接口回调 |
| `ConfigurationClassPostProcessor` | 处理 `@Configuration` 类 |
| `AopAutoProxyConfigurer` | 处理 AOP 代理 |

#### 步骤 7：initMessageSource() —— 初始化消息源

```java
protected void initMessageSource() {
    ConfigurableListableBeanFactory beanFactory = getBeanFactory();

    // 1. 如果容器中有 messageSource Bean，使用它
    if (beanFactory.containsLocalBean("messageSource")) {
        this.messageSource = beanFactory.getBean("messageSource", MessageSource.class);
    } else {
        // 2. 否则使用默认的 DelegatingMessageSource
        DelegatingMessageSource dps = new DelegatingMessageSource();
        this.messageSource = dps;
    }

    // 3. 注册到容器中
    if (this.parent != null) {
        this.messageSource = getParentMessageSource();
    }
}
```

#### 步骤 8：initApplicationEventMulticaster() —— 初始化事件广播器

```java
protected void initApplicationEventMulticaster() {
    ConfigurableListableBeanFactory beanFactory = getBeanFactory();

    // 1. 如果容器中有 applicationEventMulticaster Bean，使用它
    if (beanFactory.containsLocalBean("applicationEventMulticaster")) {
        this.applicationEventMulticaster = beanFactory.getBean(
            "applicationEventMulticaster", ApplicationEventMulticaster.class);
    } else {
        // 2. 否则使用默认的 SimpleApplicationEventMulticaster
        this.applicationEventMulticaster = new SimpleApplicationEventMulticaster(beanFactory);
    }
}
```

#### 步骤 9：onRefresh() —— 子类扩展

```java
protected void onRefresh() throws BeansException {
    // 空实现，留给子类扩展

```

Spring Boot 在这里初始化嵌入式容器（Tomcat、Jetty 等）。

#### 步骤 10：registerListeners() —— 注册监听器

```java
protected void registerListeners() {
    // 1. 注册静态指定的监听器
    for (ApplicationListener<?> listener : getApplicationListeners()) {
        getApplicationEventMulticaster().addApplicationListener(listener);
    }

    // 2. 从容器中获取所有监听器 Bean 并注册
    String[] listenerBeanNames = getBeanNamesForType(ApplicationListener.class, true, false);
    for (String listenerBeanName : listenerBeanNames) {
        getApplicationEventMulticaster().addApplicationListenerBean(listenerBeanName);
    }
}
```

#### 步骤 11：finishBeanFactoryInitialization() —— 初始化所有单例 Bean

这是**最重要**的步骤！所有非懒加载的单例 Bean 都在这里被创建。

```java
protected void finishBeanFactoryInitialization(ConfigurableListableBeanFactory beanFactory) {
    // 1. 初始化 ConversionService（类型转换服务）
    if (beanFactory.containsBean("conversionService")) {
        beanFactory.setConversionService(
            beanFactory.getBean("conversionService", ConversionService.class));
    }

    // 2. 注册自定义编辑器（如 StringArrayPropertyEditor）
    StringValueResolver valueResolver = new EmbeddedValueResolver(beanFactory);

    // 3. 初始化 LoadTimeWeaverAware Bean
    // 4. 冻结配置，禁止再修改 BeanDefinition
    beanFactory.freezeConfiguration();

    // 5. 初始化所有非懒加载的单例 Bean
    beanFactory.preInstantiateSingletons();
}
```

**`preInstantiateSingletons()` 的核心逻辑**：

```java
// DefaultListableBeanFactory.preInstantiateSingletons()
public void preInstantiateSingletons() {
    // 1. 获取所有 BeanDefinition 的名称
    List<String> beanNames = new ArrayList<>(this.beanDefinitionNames);

    // 2. 遍历创建所有 Bean
    for (String beanName : beanNames) {
        RootBeanDefinition bd = getMergedLocalBeanDefinition(beanName);

        // 跳过抽象Bean、懒加载Bean、非单例Bean
        if (bd.isAbstract() || bd.isLazyInit() || !bd.isSingleton()) {
            continue;
        }

        // 3. 处理 FactoryBean（获取 FactoryBean 本身，不是它创建的对象）
        if (isFactoryBean(beanName)) {
            FactoryBean<?> factory = getBean("&" + beanName);
            // ...
        } else {
            // 4. 创建普通 Bean
            getBean(beanName);
        }
    }

    // 5. 最后处理 SmartInitializingSingleton
    for (String beanName : beanNames) {
        Object singletonInstance = getSingleton(beanName);
        if (singletonInstance instanceof SmartInitializingSingleton) {
            SmartInitializingSingleton smartSingleton =
                (SmartInitializingSingleton) singletonInstance;
            smartSingleton.afterSingletonsInstantiated();
        }
    }
}
```

#### 步骤 12：finishRefresh() —— 完成刷新

```java
protected void finishRefresh() {
    // 1. 清除上下文级别的缓存
    clearResourceCaches();

    // 2. 初始化生命周期处理器
    initLifecycleProcessor();

    // 3. 通知所有 LifecycleProcessor 开始
    getLifecycleProcessor().onRefresh();

    // 4. 发布 ContextRefreshedEvent 事件
    publishEvent(new ContextRefreshedEvent(this));

    // 5. 注册 JMX MBean
    LiveBeansView.registerApplicationContext(this);
}
```

## 三、为什么是这个顺序？

理解了每个步骤后，你可能会有疑问：**为什么要按这个顺序？**

### 3.1 BeanFactoryPostProcessor 为什么要先于 BeanPostProcessor 执行？

因为 BFFP 操作的是 `BeanDefinition`，如果 BPP 先执行，BFFP 对 `BeanDefinition` 的修改就不会被 BPP 感知到。

### 3.2 监听器为什么要先注册，再初始化 Bean？

因为初始化过程中可能发布事件（如 `ContextRefreshedEvent`），需要确保所有监听器都已注册。

### 3.3 MessageSource 为什么要先初始化？

因为其他 Bean 的初始化过程中可能需要国际化消息（如 `@Value` 从属性文件读取）。

### 3.4 BeanFactory 为什么要先冻结配置？

冻结后不能再修改 `BeanDefinition`，确保 Bean 创建过程中配置不会被意外改变。

## 五、面试追问方向

**「refresh() 和 close() 有什么区别？」**

`refresh()` 创建一个全新的 `BeanFactory`，销毁旧的 Bean；`close()` 只是关闭容器，不刷新。

**「BeanFactory 和 ApplicationContext 的初始化有什么区别？」**

`BeanFactory` 懒加载，只有调用 `getBean()` 时才创建 Bean；`ApplicationContext` 在 `refresh()` 时就创建所有单例 Bean。

**「Spring Boot 的自动配置是在哪个步骤执行的？」**

`postProcessBeanFactory()` 步骤。Spring Boot 定义了 `AutoConfigurationImportSelector`，它实现了 `BeanDefinitionRegistryPostProcessor`，在 `invokeBeanFactoryPostProcessors()` 阶段被调用。

**「如果想在容器启动前做点什么，该怎么做？」**

实现 `BeanFactoryPostProcessor` 或 `BeanDefinitionRegistryPostProcessor`，在 `invokeBeanFactoryPostProcessors()` 阶段执行。

## 总结

Spring 的 `refresh()` 方法包含 12 个核心步骤，每个步骤都有其存在的意义：

- **prepareRefresh()**：准备环境
- **obtainFreshBeanFactory()**：获取 Bean 工厂
- **prepareBeanFactory()**：配置 Bean 工厂
- **invokeBeanFactoryPostProcessors()**：执行 BFPP，解析 `@ComponentScan`、`@Bean` 等
- **registerBeanPostProcessors()**：注册 BPP，准备处理 Bean
- **initMessageSource()**：初始化国际化
- **initApplicationEventMulticaster()**：初始化事件机制
- **finishBeanFactoryInitialization()**：创建所有单例 Bean
- **finishRefresh()**：发布刷新完成事件
