# JDK 动态代理 vs CGLIB 动态代理

AOP 的底层实现有两种：JDK 动态代理和 CGLIB 动态代理。面试中经常问到它们的区别，但很多人只停留在「JDK 基于接口，CGLIB 基于继承」这个层面。

这一节，我们来深入理解两种代理的原理和适用场景。

## 一、为什么需要动态代理？

### 1.1 静态代理的问题

在讲动态代理之前，先看看静态代理有什么问题：

```java
// 接口
public interface UserService {
    void create();
    void update();
    void delete();
}

// 目标对象
public class UserServiceImpl implements UserService {
    @Override
    public void create() { /* 业务逻辑 */ }
    @Override
    public void update() { /* 业务逻辑 */ }
    @Override
    public void delete() { /* 业务逻辑 */ }
}

// 静态代理（每个方法都要写一遍）
public class UserServiceProxy implements UserService {
    private UserService target;

    public UserServiceProxy(UserService target) {
        this.target = target;
    }

    @Override
    public void create() {
        System.out.println("日志：create 开始");
        target.create();
        System.out.println("日志：create 结束");
    }

    @Override
    public void update() {
        System.out.println("日志：update 开始");
        target.update();
        System.out.println("日志：update 结束");
    }

    @Override
    public void delete() {
        System.out.println("日志：delete 开始");
        target.delete();
        System.out.println("日志：delete 结束");
    }
}
```

**问题**：每个接口都要写一个代理类，每个方法都要手动添加增强逻辑。如果有 100 个方法，就要写 100 次。

### 1.2 动态代理的优势

动态代理可以在运行时动态生成代理类，不需要手动编写代理类。方法的增强逻辑只需要写一次。

---

## 二、JDK 动态代理

### 2.1 核心接口

JDK 动态代理需要两个核心组件：

1. **`InvocationHandler` 接口**：定义增强逻辑
2. **`Proxy.newProxyInstance()` 方法**：生成代理对象

### 2.2 基本用法

```java
public interface UserService {
    void create();
    void update();
    void delete();
}

public class UserServiceImpl implements UserService {
    @Override
    public void create() {
        System.out.println("创建用户");
    }

    @Override
    public void update() {
        System.out.println("更新用户");
    }

    @Override
    public void delete() {
        System.out.println("删除用户");
    }
}

// InvocationHandler：定义增强逻辑
public class MyInvocationHandler implements InvocationHandler {
    private Object target; // 目标对象

    public MyInvocationHandler(Object target) {
        this.target = target;
    }

    @Override
    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
        // 前置增强
        System.out.println("日志：方法 " + method.getName() + " 开始");

        // 执行目标方法
        Object result = method.invoke(target, args);

        // 后置增强
        System.out.println("日志：方法 " + method.getName() + " 结束");

        return result;
    }
}

// 生成代理对象
public class Main {
    public static void main(String[] args) {
        UserService target = new UserServiceImpl();

        UserService proxy = (UserService) Proxy.newProxyInstance(
            target.getClass().getClassLoader(),  // 类加载器
            target.getClass().getInterfaces(),     // 实现的接口
            new MyInvocationHandler(target)        // InvocationHandler
        );

        // 调用代理方法
        proxy.create();  // 会自动调用 InvocationHandler.invoke()
        proxy.update();
        proxy.delete();
    }
}
```

### 2.3 原理分析

**生成的代理类结构**（反编译后）：

```java
public final class $Proxy0 extends Proxy implements UserService {
    private InvocationHandler h;

    public $Proxy0(InvocationHandler h) {
        this.h = h;
    }

    @Override
    public void create() {
        // 实际上是调用 h.invoke(this, ...)
        h.invoke(this, m3, null); // m3 是 create 方法的引用
    }

    @Override
    public void update() {
        h.invoke(this, m1, null);
    }

    @Override
    public void delete() {
        h.invoke(this, m2, null);
    }
}
```

**关键特点**：
1. 代理类继承 `Proxy`，实现目标接口
2. 所有方法调用都委托给 `InvocationHandler.invoke()`
3. 目标对象必须有接口

### 2.4 为什么必须是接口？

因为 `Proxy.newProxyInstance()` 的第二个参数是 `Class<?>[] interfaces`，生成的代理类必须实现这些接口。

## 三、CGLIB 动态代理

### 3.1 核心接口

CGLIB 需要两个核心组件：

1. **`MethodInterceptor` 接口**：定义增强逻辑
2. **`Enhancer` 类**：生成代理对象

### 3.2 基本用法

```java
public class UserServiceImpl {
    public void create() {
        System.out.println("创建用户");
    }

    public void update() {
        System.out.println("更新用户");
    }

    public void delete() {
        System.out.println("删除用户");
    }
}

// MethodInterceptor：定义增强逻辑
public class MyMethodInterceptor implements MethodInterceptor {
    @Override
    public Object intercept(Object obj, Method method, Object[] args,
                           MethodProxy proxy) throws Throwable {
        // 前置增强
        System.out.println("日志：方法 " + method.getName() + " 开始");

        // 执行目标方法
        // 注意：使用 proxy.invoke() 而不是 method.invoke()
        // 因为 proxy.invoke() 可以避免反射的性能问题
        Object result = proxy.invokeSuper(obj, args);

        // 后置增强
        System.out.println("日志：方法 " + method.getName() + " 结束");

        return result;
    }
}

// 生成代理对象
public class Main {
    public static void main(String[] args) {
        // 创建 Enhancer
        Enhancer enhancer = new Enhancer();

        // 设置目标类
        enhancer.setSuperclass(UserServiceImpl.class);

        // 设置回调
        enhancer.setCallback(new MyMethodInterceptor());

        // 生成代理对象
        UserServiceImpl proxy = (UserServiceImpl) enhancer.create();

        // 调用代理方法
        proxy.create();
        proxy.update();
        proxy.delete();
    }
}
```

### 3.3 原理分析

**生成的代理类结构**（反编译后）：

```java
public class UserServiceImpl$$EnhancerByCGLIB$$ extends UserServiceImpl {
    private MethodInterceptor CGLIB$CALLBACK_0;

    public UserServiceImpl$$EnhancerByCGLIB$$() {
        CGLIB$CALLBACK_0 = ...;
    }

    @Override
    public void create() {
        // 实际上是调用 MethodInterceptor.intercept()
        CGLIB$CALLBACK_0.intercept(this, ...);
    }

    @Override
    public void update() {
        CGLIB$CALLBACK_0.intercept(this, ...);
    }

    @Override
    public void delete() {
        CGLIB$CALLBACK_0.intercept(this, ...);
    }
}
```

**关键特点**：
1. 代理类继承目标类
2. 所有方法调用都委托给 `MethodInterceptor.intercept()`
3. 目标对象可以有接口，也可以没有

### 3.4 为什么 CGLIB 可以代理任何类？

因为 CGLIB 是基于继承实现的。生成的代理类是目标类的子类，所以可以代理任何非 `final` 的类。

## 四、两种代理的对比

### 4.1 核心区别

| 对比维度 | JDK 动态代理 | CGLIB 动态代理 |
|---|---|---|
| **实现方式** | 实现目标接口 | 继承目标类 |
| **代理类结构** | `extends Proxy implements 接口` | `extends 目标类` |
| **目标要求** | 必须有接口 | 可以没有接口 |
| **性能** | 稍慢（每次反射调用） | 较快（FastClass 机制） |
| **final 处理** | 接口的 final 方法无法代理 | final 方法无法重写，无法代理 |
| **Spring 默认** | 有接口时使用 | 无接口时使用 |

### 4.2 性能对比

**JDK 动态代理**：
```java
// 每次调用都通过反射
method.invoke(target, args);
```

**CGLIB 动态代理**：
```java
// 使用 FastClass 避免反射
proxy.invokeSuper(obj, args);
// FastClass 原理：为每个方法生成一个索引，
// 调用时直接通过索引调用，避免 Method.invoke() 的反射开销
```

### 4.3 无法代理的场景

**JDK 动态代理无法代理的情况**：
```java
// 没有接口的类
public class UserService { } // JDK 代理无法处理
```

**CGLIB 动态代理无法代理的情况**：
```java
// final 类
public final class String { } // 无法继承

// final 方法
public class UserService {
    public final void method() { } // 无法重写
}
```

## 五、Spring AOP 中的选择规则

Spring AOP 会根据目标类的情况自动选择代理方式：

```
目标类
  │
  ├─ 有接口
  │    └─ 默认使用 JDK 动态代理
  │         └─ 可配置为强制使用 CGLIB
  │
  └─ 无接口
       └─ 使用 CGLIB 动态代理
```

### 5.1 Spring Boot 的默认行为

Spring Boot 2.0 开始，**如果目标类有接口，默认也使用 CGLIB 代理**：

```properties
# 强制使用 CGLIB（Spring Boot 默认行为）
spring.aop.proxy-target-class=true

# 使用 JDK 动态代理
spring.aop.proxy-target-class=false
```

### 5.2 强制使用 CGLIB 的原因

1. **避免接口方法代理问题**：JDK 代理不能代理 `equals()`、`hashCode()`、`toString()` 等 Object 方法
2. **更好的兼容性**：类代理比接口代理更符合直觉
3. **性能考虑**：CGLIB 的 FastClass 机制在某些场景下性能更好

## 六、实战选择建议

### 6.1 选择 JDK 动态代理

- 目标类有明确的接口，且接口方法不需要特殊处理
- 对性能要求不高
- 依赖 Java 标准（方便切换到其他实现）

```java
public interface UserService {
    void create();
}

// Spring 默认使用 JDK 代理
@Service
public class UserServiceImpl implements UserService { }
```

### 6.2 选择 CGLIB 动态代理

- 目标类没有接口
- 需要代理 Object 方法（如 `equals()`、`hashCode()`）
- 需要更好的性能
- 需要代理 `final` 类的方法（开玩笑的，final 类无法代理）

```java
// Spring 自动使用 CGLIB
@Service
public class UserService { } // 没有接口
```

### 6.3 强制指定代理方式

```java
// 强制使用 CGLIB（推荐）
@EnableAspectJAutoProxy(proxyTargetClass = true)

// 使用 JDK 动态代理
@EnableAspectJAutoProxy(proxyTargetClass = false)
```

## 七、面试追问方向

**「JDK 动态代理为什么必须要有接口？」**

因为 `Proxy.newProxyInstance()` 生成的代理类需要实现某个接口。如果目标类没有接口，就无法生成代理类。这是 Java 语言本身的限制，不是设计缺陷。

**「CGLIB 为什么可以代理任何类？」**

因为 CGLIB 是基于继承实现的。生成的代理类是目标类的子类，所以可以代理任何非 `final` 的类。

**「Spring 为什么推荐使用 CGLIB？」**

Spring Boot 2.0 之后默认使用 CGLIB，主要原因是：1) JDK 代理无法代理 Object 方法；2) CGLIB 性能更好；3) 类代理比接口代理更符合直觉。

**「FastClass 是什么？」**

CGLIB 为了解决反射调用性能问题，生成一个 FastClass 类，存储方法的索引。调用时直接通过索引调用，避免 `Method.invoke()` 的反射开销。

---

## 总结

JDK 动态代理和 CGLIB 动态代理是 AOP 的两种实现方式：

| 维度 | JDK 动态代理 | CGLIB 动态代理 |
|---|---|---|
| 实现 | 实现接口 | 继承目标类 |
| 要求 | 必须有接口 | 可以没有接口 |
| 性能 | 反射调用 | FastClass 机制 |
| Spring 默认 | 1.x 版本 | 2.x 版本 |

理解两种代理的原理，能够帮助你在实际开发中做出正确的选择，并排查相关的代理问题。
