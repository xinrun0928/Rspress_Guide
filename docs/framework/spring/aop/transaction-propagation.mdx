# Spring 事务传播行为 7 种详解

import { Badge } from '@rspress/core/theme';

<Badge text="核心原理" type="danger" /> <Badge text="传播行为" type="tip" /> <Badge text="面试高频" type="warning" />

当你调用一个带有 `@Transactional` 的方法时，如果当前已经在一个事务中了，会发生什么？是加入已有事务，还是创建新事务？

这个问题的答案取决于**事务传播行为**（Propagation）。

这一节，我们来全面讲解 Spring 的 7 种事务传播行为，搞清楚每种行为的含义和使用场景。

---

## 一、什么是事务传播行为？

### 1.1 问题场景

```java
@Service
public class OrderService {
    @Transactional
    public void createOrder(Order order) {
        // 1. 保存订单
        orderDao.save(order);

        // 2. 调用库存服务（也有事务）
        inventoryService.reduceStock(order.getItems());

        // 3. 调用支付服务（也有事务）
        paymentService.pay(order.getPayment());
    }
}
```

当 `createOrder()` 调用 `inventoryService.reduceStock()` 时，如果 `reduceStock()` 也有 `@Transactional` 注解，会发生什么？

### 1.2 传播行为的定义

事务传播行为定义了**当一个事务方法被另一个事务方法调用时，应该如何处理事务**。

Spring 定义了 7 种传播行为：

| 传播行为 | 说明 |
|---|---|
| `REQUIRED` | 支持当前事务，如果没有则创建新事务（默认） |
| `SUPPORTS` | 支持当前事务，如果没有则以非事务执行 |
| `MANDATORY` | 必须支持当前事务，如果没有则抛出异常 |
| `REQUIRES_NEW` | 创建新事务，挂起当前事务 |
| `NOT_SUPPORTED` | 以非事务执行，挂起当前事务 |
| `NEVER` | 必须以非事务执行，否则抛出异常 |
| `NESTED` | 如果当前存在事务，则在嵌套事务中执行 |

## 二、七种传播行为详解

### 2.1 REQUIRED（默认）

```
当前有事务 → 加入该事务
当前无事务 → 创建新事务
```

**场景**：最常用的传播行为，适用于大多数业务场景。

```java
@Service
public class AccountService {
    @Transactional(propagation = Propagation.REQUIRED)  // 默认，可省略
    public void transfer(Account from, Account to, BigDecimal amount) {
        from.withdraw(amount);    // 在同一事务中
        to.deposit(amount);       // 在同一事务中
    }
}
```

**执行流程**：

```
调用 transfer()
    ↓
已有事务？
    ├─ 是 → 加入已有事务，执行所有操作
    └─ 否 → 创建新事务，执行所有操作
```

### 2.2 SUPPORTS

```
当前有事务 → 加入该事务
当前无事务 → 以非事务执行
```

**场景**：适用于查询方法，不强制要求事务。

```java
@Service
public class UserService {
    @Transactional(propagation = Propagation.SUPPORTS)
    public User findById(Long id) {
        // 查询不需要事务，但如果有事务可以加入
        return userDao.findById(id);
    }
}
```

### 2.3 MANDATORY

```
当前有事务 → 加入该事务
当前无事务 → 抛出 IllegalTransactionStateException
```

**场景**：强制要求在事务中执行。

```java
@Service
public class ValidationService {
    @Transactional(propagation = Propagation.MANDATORY)
    public void validateBusinessRule(Order order) {
        // 必须被事务方法调用，否则抛出异常
        if (!validate(order)) {
            throw new BusinessException("业务规则验证失败");
        }
    }
}

// 正确调用
@Service
public class OrderService {
    @Transactional
    public void createOrder(Order order) {
        validationService.validateBusinessRule(order);  // 正常
    }
}

// 错误调用
@Service
public class OtherService {
    public void doSomething() {
        validationService.validateBusinessRule(order);  // 抛出异常！
    }
}
```

### 2.4 REQUIRES_NEW

```
当前有事务 → 挂起当前事务，创建新事务
当前无事务 → 创建新事务
```

**场景**：新事务与当前事务完全独立。

```java
@Service
public class NotificationService {
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void sendEmail(Order order) {
        // 即使订单创建失败，邮件也要发送
        emailService.send(order);
    }
}

@Service
public class OrderService {
    @Transactional
    public void createOrder(Order order) {
        orderDao.save(order);
        notificationService.sendEmail(order);  // 独立事务
        // 即使后面失败，邮件已发送不会回滚
    }
}
```

**执行流程**：

```
调用 createOrder()
    ↓
开启事务 T1
    ↓
保存订单
    ↓
调用 sendEmail()
    ├─ 挂起事务 T1
    ├─ 开启新事务 T2
    ├─ 发送邮件
    ├─ 提交事务 T2
    └─ 恢复事务 T1
    ↓
...（如果这里抛异常）
    ↓
回滚事务 T1（订单保存被回滚，但邮件已发送）
```

### 2.5 NOT_SUPPORTED

```
当前有事务 → 挂起当前事务，以非事务执行
当前无事务 → 以非事务执行
```

**场景**：某些操作不应在事务中执行。

```java
@Service
public class ReportService {
    @Transactional(propagation = Propagation.NOT_SUPPORTED)
    public void generateReport() {
        // 生成报表可能很慢，不应在事务中执行
        // 因为事务会占用数据库连接，影响其他操作
        reportGenerator.generate();
    }
}
```

### 2.6 NEVER

```
当前有事务 → 抛出 IllegalTransactionStateException
当前无事务 → 以非事务执行
```

**场景**：确保方法不在事务中执行。

```java
@Service
public class LegacyService {
    @Transactional(propagation = Propagation.NEVER)
    public void callExternalApi() {
        // 外部 API 调用不应在事务中，避免长时间占用连接
    }
}
```

### 2.7 NESTED

```
当前有事务 → 在嵌套事务中执行（使用 Savepoint）
当前无事务 → 创建新事务
```

**场景**：部分操作需要独立回滚，但不影响整体事务。

```java
@Service
public class OrderService {
    @Transactional
    public void createOrder(Order order) {
        // 1. 保存订单（主操作）
        orderDao.save(order);

        // 2. 发送通知（嵌套操作，可独立回滚）
        try {
            transactionTemplate.executeWithoutResult(status -> {
                notificationService.send(order);  // NESTED 事务
            });
        } catch (Exception e) {
            // 通知失败不影响订单创建
            log.warn("通知发送失败", e);
        }
    }
}
```

**NESTED vs REQUIRES_NEW**：

| 对比 | NESTED | REQUIRES_NEW |
|---|---|---|
| 事务关系 | 嵌套在当前事务中 | 创建独立新事务 |
| 回滚影响 | 嵌套操作回滚不影响主事务 | 互不影响 |
| 数据库支持 | 需要支持 Savepoint | 无要求 |
| 性能 | 较好（同一个事务） | 较差（需要创建新事务） |

## 三、传播行为与异常处理

### 3.1 异常对事务的影响

```java
@Transactional
public void methodA() {
    try {
        methodB();  // REQUIRES_NEW
    } catch (Exception e) {
        // methodB 的事务已提交，这里的 catch 不影响
    }
}

@Transactional(propagation = Propagation.REQUIRES_NEW)
public void methodB() {
    // 如果这里抛异常，事务会回滚
    throw new RuntimeException();
}
```

### 3.2 回滚规则

```java
@Transactional(rollbackFor = Exception.class)  // 指定回滚异常
public void method() {
    // 默认：RuntimeException 和 Error 会回滚
    // 指定后：所有 Exception 都会回滚
}
```

## 四、传播行为对比表

| 传播行为 | 当前有事务 | 当前无事务 | 典型场景 |
|---|---|---|---|
| `REQUIRED` | 加入 | 创建 | 默认，业务操作 |
| `SUPPORTS` | 加入 | 非事务 | 查询操作 |
| `MANDATORY` | 加入 | 抛异常 | 必须在事务中 |
| `REQUIRES_NEW` | 挂起/新建 | 新建 | 日志记录、独立操作 |
| `NOT_SUPPORTED` | 挂起 | 非事务 | 非事务操作 |
| `NEVER` | 抛异常 | 非事务 | 确保非事务 |
| `NESTED` | 嵌套 | 新建 | 可独立回滚的操作 |

## 五、实战选择指南

### 5.1 业务操作：REQUIRED

```java
@Transactional  // 默认 REQUIRED
public void createOrder(Order order) {
    orderDao.save(order);
    inventoryService.reduceStock(order.getItems());
}
```

### 5.2 查询操作：SUPPORTS

```java
@Transactional(propagation = Propagation.SUPPORTS)
public User findById(Long id) {
    return userDao.findById(id);
}
```

### 5.3 日志记录：REQUIRES_NEW

```java
@Transactional(propagation = Propagation.REQUIRES_NEW)
public void logOperation(String operation) {
    logDao.save(operation);  // 即使业务回滚，日志也要保存
}
```

### 5.4 发送通知：REQUIRES_NEW

```java
@Transactional(propagation = Propagation.REQUIRES_NEW)
public void sendNotification(Order order) {
    notificationService.send(order);  // 即使订单创建失败，也要发送通知
}
```

### 5.5 验证逻辑：MANDATORY

```java
@Transactional(propagation = Propagation.MANDATORY)
public void validate(Order order) {
    // 必须在事务中被调用
}
```

## 六、常见问题

### 6.1 REQUIRED 和 NESTED 的区别

```java
// REQUIRED
@Transactional
public void methodA() {
    methodB();  // 在同一事务中
}

@Transactional
public void methodB() {
    // 异常时，整个事务回滚
}

// NESTED
@Transactional
public void methodA() {
    methodB();  // 在嵌套事务中
}

@Transactional(propagation = Propagation.NESTED)
public void methodB() {
    // 异常时，可以只回滚 B 的操作
}
```

### 6.2 REQUIRES_NEW 的性能问题

`REQUIRES_NEW` 会创建新事务，需要获取新连接，性能开销较大。在高并发场景下应谨慎使用。

## 七、面试追问方向

**「Spring 默认的事务传播行为是什么？」**

`REQUIRED`。当一个方法调用另一个 `@Transactional` 方法时，默认会在同一个事务中执行。

**「REQUIRES_NEW 和 REQUIRED 的区别？」**

`REQUIRED` 加入当前事务，`REQUIRES_NEW` 创建新事务。前者共享事务，后者完全独立。

**「NESTED 是如何实现的？」**

通过数据库的 Savepoint 机制。嵌套事务在当前事务中创建一个 Savepoint，如果嵌套部分失败，可以回滚到 Savepoint 位置，不影响主事务。

**「什么场景下需要用 REQUIRES_NEW？」**

当某个操作需要独立提交，不受主事务回滚影响时，如日志记录、发送通知等。

---

## 总结

Spring 的 7 种事务传播行为：

| 行为 | 当前有事务 | 当前无事务 | 推荐场景 |
|---|---|---|---|
| `REQUIRED` | 加入 | 创建 | 业务操作（默认） |
| `SUPPORTS` | 加入 | 非事务 | 查询 |
| `MANDATORY` | 加入 | 抛异常 | 验证 |
| `REQUIRES_NEW` | 新建 | 新建 | 日志、通知 |
| `NOT_SUPPORTED` | 挂起 | 非事务 | 非事务操作 |
| `NEVER` | 抛异常 | 非事务 | 确保非事务 |
| `NESTED` | 嵌套 | 新建 | 可独立回滚 |

理解每种传播行为的含义，才能正确处理复杂业务场景中的事务问题。
