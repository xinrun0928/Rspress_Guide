# AspectJ 与 Spring AOP 对比

import { Badge } from '@rspress/core/theme';

<Badge text="对比分析" type="info" /> <Badge text="AspectJ" type="tip" /> <Badge text="选型指南" type="warning" />

提到 AOP，很多人会混淆 AspectJ 和 Spring AOP。它们都能实现 AOP，但原理和应用场景完全不同。

这一节，我们来全面对比 AspectJ 和 Spring AOP，搞清楚什么时候该用哪个。

---

## 一、基本概念对比

### 1.1 什么是 AspectJ？

AspectJ 是一个完整的 AOP 框架，包含自己的编译器（ajc）和织入器。它是最强大、最完整的 AOP 实现。

### 1.2 什么是 Spring AOP？

Spring AOP 是 Spring 框架提供的 AOP 实现，基于动态代理（ JDK 和 CGLIB）。它是 Spring 框架的一部分，不能独立使用。

### 1.3 核心区别

| 维度 | Spring AOP | AspectJ |
|---|---|---|
| **织入时机** | 运行时织入 | 编译时/加载时织入 |
| **连接点** | 仅方法执行 | 支持所有连接点 |
| **代理方式** | 动态代理 | 编译/加载时修改字节码 |
| **依赖** | 必须依赖 Spring | 可以独立使用 |
| **性能** | 略低（运行时创建代理） | 更高（编译时已织入） |
| **学习曲线** | 较平缓 | 较陡峭 |
| **功能** | 有限 | 完整 |

## 二、织入时机对比

### 2.1 Spring AOP：运行时织入

Spring AOP 在运行时通过动态代理创建代理对象：

```
编译 → 打包 → 部署 → 运行
                        ↓
                 Spring 容器启动
                        ↓
              创建 Bean 时动态生成代理
                        ↓
                   方法调用时增强
```

**优点**：无需特殊编译步骤
**缺点**：运行时开销，无法代理构造器等

### 2.2 AspectJ：编译时织入

AspectJ 使用 `ajc` 编译器，在编译时将切面织入字节码：

```
编写 .java 文件和 .aj 文件
           ↓
        ajc 编译（同时编译和织入）
           ↓
     织入后的 .class 文件
           ↓
     直接运行，无需容器
```

**优点**：性能更好，功能更完整
**缺点**：需要特殊编译步骤

### 2.3 AspectJ：加载时织入（LTW）

在类加载时织入字节码：

```
编译 → 打包 → 部署
           ↓
       类加载时
           ↓
    Java Agent 拦截类加载
           ↓
    织入切面字节码
           ↓
    运行增强后的类
```

## 三、连接点支持对比

### 3.1 Spring AOP 支持的连接点

| 连接点 | 支持 |
|---|---|
| 方法调用（调用方） | ❌ |
| 方法执行 | ✅ |
| 构造器调用 | ❌ |
| 构造器执行 | ❌ |
| 字段读写 | ❌ |
| 异常处理 | ❌ |

Spring AOP 只支持**方法执行**作为连接点，因为它是基于代理的。

### 3.2 AspectJ 支持的连接点

| 连接点 | 支持 |
|---|---|
| 方法调用（调用方） | ✅ |
| 方法执行 | ✅ |
| 构造器调用 | ✅ |
| 构造器执行 | ✅ |
| 字段读写 | ✅ |
| 静态初始化块 | ✅ |
| 异常处理 | ✅ |

AspectJ 支持所有连接点，功能更完整。

## 四、切点表达式对比

### 4.1 Spring AOP 切点表达式

```java
// Spring AOP 只支持 execution
@Pointcut("execution(* com.example.service.*.*(..))")
public void serviceLayer() { }

// 支持 within（类型匹配）
@Pointcut("within(com.example.service..*)")
public void serviceLayer() { }

// 支持 this、target、args
@Pointcut("this(com.example.service.UserService)")
public void thisUserService() { }
```

### 4.2 AspectJ 切点表达式

```java
// AspectJ 支持更丰富的切点
// call：方法调用
@Pointcut("call(* com.example.service..*(..))")
public void methodCall() { }

// initialization：构造器执行
@Pointcut("initialization(* com.example.service..new(..))")
public void constructorExecution() { }

// set：字段赋值
@Pointcut("set(* com.example.service.User.name)")
public void fieldSet() { }

// get：字段读取
@Pointcut("get(* com.example.service.User.name)")
public void fieldGet() { }

// handler：异常处理
@Pointcut("handler(java.io.IOException+)")
public void ioExceptionHandler() { }
```

## 五、性能对比

### 5.1 Spring AOP 性能开销

```
方法调用流程（Spring AOP）：

调用方法
    ↓
代理对象（JDK/CGLIB）
    ↓
InvocationHandler/MethodInterceptor
    ↓
前置通知
    ↓
目标方法（反射调用）
    ↓
后置通知
    ↓
返回结果
```

每次方法调用都有额外的代理开销。

### 5.2 AspectJ 性能优势

```
方法调用流程（AspectJ 编译时织入）：

调用方法
    ↓
织入后的方法（直接调用）
    ↓
前置通知（编译时织入）
    ↓
原逻辑
    ↓
后置通知（编译时织入）
    ↓
返回结果
```

AspectJ 编译时织入后，没有额外的代理开销。

## 六、实际应用场景

### 6.1 什么时候用 Spring AOP？

**场景一：企业应用中的通用功能**

```java
// 日志记录、事务管理、安全控制
@Service
public class UserService {
    @Transactional
    public void createUser(User user) {
        // 事务会自动开启和提交
    }
}
```

**场景二：方法级别的监控**

```java
@Aspect
@Component
public class PerformanceMonitor {
    @Around("execution(* com.example.service.*.*(..))")
    public Object monitor(ProceedingJoinPoint pjp) throws Throwable {
        long start = System.currentTimeMillis();
        Object result = pjp.proceed();
        long duration = System.currentTimeMillis() - start;
        System.out.println(pjp.getSignature() + " took " + duration + "ms");
        return result;
    }
}
```

### 6.2 什么时候用 AspectJ？

**场景一：需要对构造器进行增强**

```java
// AspectJ 可以增强构造器
public aspect UserServiceAspect {
    // 构造器执行时
    pointcut construction() : initialization(new(..));

    after() returning(UserService us) : construction() {
        us.initialize(); // 初始化逻辑
    }
}
```

**场景二：需要对字段进行监控**

```java
// AspectJ 可以监控字段访问
public aspect UserFieldAspect {
    pointcut setName() : set(String User.name);

    before() : setName() {
        System.out.println("Setting name to: " + User.name);
    }
}
```

**场景三：高性能要求的场景**

如果对性能要求极高，AspectJ 编译时织入是更好的选择。

## 七、Spring Boot 集成 AspectJ

### 7.1 启用 AspectJ

Spring Boot 虽然使用 Spring AOP，但可以通过配置开启 AspectJ 支持：

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-aop</artifactId>
</dependency>
```

### 7.2 加载时织入配置

```properties
# 启用加载时织入
spring.aop.auto=true
spring.aop.proxy-target-class=true

# 配置 AspectJ 加载时织入
-Djavaagent=spring-instrument.jar
```

### 7.3 aop.xml 配置

```xml
<aspectj>
    <aspects>
        <aspect>com.example.aspect.MyAspect</aspect>
    </aspects>
    <weaver>
        <include within="com.example..*"/>
    </weaver>
</aspectj>
```

## 八、混合使用

### 8.1 在 Spring AOP 中使用 AspectJ 切点

Spring AOP 可以使用 AspectJ 的切点表达式：

```java
@Aspect
@Component
public class LoggingAspect {
    // 使用 AspectJ 切点表达式
    @Around("execution(* com.example.service..*(..)) && " +
            "!execution(* toString())")
    public Object around(ProceedingJoinPoint pjp) throws Throwable {
        // 增强逻辑
    }
}
```

### 8.2 Spring AOP 无法替代 AspectJ 的场景

```java
// Spring AOP 无法这样做
@Aspect
@Component
public class ConstructorAspect {
    // 无法监控构造器
    @Around("initialization(com.example..new(..))") // Spring AOP 不支持
    public Object aroundInit(ProceedingJoinPoint pjp) throws Throwable {
        // ...
    }
}
```

## 九、面试追问方向

**「AspectJ 和 Spring AOP 哪个性能更好？」**

AspectJ 性能更好。Spring AOP 使用运行时代理，每次方法调用都有额外开销；AspectJ 编译时织入后，没有额外的代理开销。

**「Spring AOP 为什么只支持方法执行？」**

因为 Spring AOP 基于动态代理实现。动态代理只能代理方法调用，无法直接代理构造器或字段访问。

**「什么时候需要用 AspectJ？」**

当需要增强构造器、监控字段访问、或者对性能要求极高时，AspectJ 是更好的选择。

**「Spring Boot 默认使用哪种 AOP？」**

默认使用 Spring AOP（基于动态代理），但引入了 `spring-boot-starter-aop` 后，可以更好地与 AspectJ 集成。

---

## 总结

| 维度 | Spring AOP | AspectJ |
|---|---|---|
| **织入方式** | 运行时 | 编译时/加载时 |
| **连接点** | 仅方法执行 | 所有连接点 |
| **复杂度** | 简单 | 复杂 |
| **性能** | 一般 | 更好 |
| **学习曲线** | 平缓 | 陡峭 |
| **适用场景** | 企业应用 | 框架开发、高性能需求 |

在实际项目中，**大多数场景使用 Spring AOP 就足够了**。只有当需要增强构造器、监控字段访问，或者对性能有极高要求时，才需要考虑 AspectJ。
