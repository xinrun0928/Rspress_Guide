# @Configuration 与 @Bean

`@Configuration` 是 Spring 配置类的核心注解，但它的实现原理很多人并不清楚。

为什么 `@Configuration` 类会被 CGLIB 代理？`@Bean` 方法是如何被调用的？

这一节，我们来深入理解 `@Configuration` 的实现原理。

---

## 一、@Configuration vs @Component

### 1.1 基本用法

```java
@Configuration
public class AppConfig {
    @Bean
    public UserService userService() {
        return new UserService();
    }
}
```

表面上看，`@Configuration` 和 `@Component` 没什么区别——都会被 Spring 扫描到，都会注册为 Bean。

但实际上，`@Configuration` 有一个关键特性：**它会被 CGLIB 代理**。

### 1.2 为什么需要代理？

考虑这样一个场景：

```java
@Configuration
public class AppConfig {
    @Bean
    public UserService userService() {
        return new UserService();
    }

    @Bean
    public OrderService orderService() {
        // 调用 userService()
        return new OrderService(userService());
    }
}
```

如果 `orderService()` 直接调用 `userService()`，每次都会创建新的 `UserService` 实例。

但 Spring 希望所有 Bean 都是单例的，怎么办？

**解决方案**：CGLIB 代理。

---

## 二、CGLIB 代理的实现

### 2.1 代理后的行为

`@Configuration` 类被 CGLIB 代理后，方法调用会被拦截：

```java
// 代理后的 AppConfig（伪代码）
public class AppConfig$$EnhancerBySpringCGLIB$$ extends AppConfig {
    private Map<String, Object> singletonCache = new HashMap<>();

    @Override
    public UserService userService() {
        if (!singletonCache.containsKey("userService")) {
            singletonCache.put("userService", super.userService());
        }
        return (UserService) singletonCache.get("userService");
    }
}
```

这样，无论 `userService()` 被调用多少次，都只返回同一个实例。

### 2.2 源码解析

`ConfigurationClassPostProcessor` 负责处理 `@Configuration` 类：

```java
public class ConfigurationClassPostProcessor implements BeanDefinitionRegistryPostProcessor {
    @Override
    public void postProcessBeanFactory(ConfigurableListableBeanFactory beanFactory) {
        // 增强配置类
        enhanceConfigurationClasses(beanFactory);
    }
}
```

## 三、@Configuration 代理的条件

### 3.1 默认情况

Spring 4.x 之后，默认对所有 `@Configuration` 类使用 CGLIB 代理。

### 3.2 禁用代理

```java
@Configuration(proxyBeanMethods = false)  // 不使用代理
public class AppConfig {
    @Bean
    public UserService userService() {
        return new UserService();
    }

    @Bean
    public OrderService orderService() {
        return new OrderService(userService()); // 每次创建新实例
    }
}
```

### 3.3 Lite 模式

`proxyBeanMethods = false` 称为 Lite 模式，适用于纯配置类（不需要拦截 `@Bean` 方法调用）。

---

## 四、@Configuration vs @Component 使用场景

| 场景 | 推荐注解 |
|---|---|
| 配置类，@Bean 方法互相调用 | `@Configuration` |
| 纯配置类，无依赖调用 | `@Configuration(proxyBeanMethods = false)` 或 `@Component` |
| 组件类，需要注册为 Bean | `@Component` |

---

## 五、面试追问方向

**「@Configuration 为什么默认是 CGLIB 代理？」**

为了确保 `@Bean` 方法调用返回的是容器中的单例 Bean，而不是每次创建新实例。

**「@Configuration 和 @Component 有什么区别？」**

功能上没区别，都会注册为 Bean。但 `@Configuration` 会被 CGLIB 代理，确保 @Bean 方法调用返回单例。

**「proxyBeanMethods = false 是什么意思？」**

不创建 CGLIB 代理，`@Bean` 方法调用会直接执行，每次返回新实例。
