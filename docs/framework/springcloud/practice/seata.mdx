# Seata AT 模式：让分布式事务像本地事务一样简单

import { Badge } from '@rspress/core/theme';

跨服务的数据库操作，怎么保证一致性？

订单服务和库存服务分别在不同的数据库。当用户下单时，需要同时扣减库存和创建订单——这两个操作必须同时成功或同时失败。如果库存扣减成功了，订单创建失败了，怎么办？

这就是**分布式事务**问题。

## 一、分布式事务的问题

### 1.1 本地事务 vs 分布式事务

**本地事务**：所有操作在同一个数据库中，数据库可以保证 ACID。

```sql
BEGIN;
UPDATE inventory SET stock = stock - 1 WHERE product_id = 1;  -- 扣库存
INSERT INTO orders (product_id, quantity) VALUES (1, 1);       -- 创建订单
COMMIT;
```

**分布式事务**：操作分布在多个数据库中，传统的 ACID 事务无法保证。

```
服务 A（数据库 A）
  │
  └── UPDATE orders ...
  │
服务 B（数据库 B）
  │
  └── UPDATE inventory ...

问题：服务 A 的事务成功了，服务 B 的事务失败了，怎么回滚？
```

### 1.2 CAP 定理与 BASE 理论

分布式系统有一个著名的 CAP 定理：

- **C（Consistency）**：一致性
- **A（Availability）**：可用性
- **P（Partition tolerance）**：分区容错性

**CAP 定理**：三个特性只能同时满足两个。

由于网络分区不可避免，分布式系统必须保证 P，因此在 C 和 A 之间权衡。

**BASE 理论**是对 CAP 的补充：

- **Basically Available**：基本可用
- **Soft state**：软状态
- **Eventually consistent**：最终一致

Seata AT 模式正是基于 BASE 理论，通过**补偿机制**实现**最终一致性**。

## 二、Seata 的三种模式

### 2.1 AT 模式

AT（Automatic Transaction）模式是 Seata 最推荐的模式，对业务无侵入：

- 自动分析 SQL，生成逆向 SQL
- 通过 Undolog 实现回滚
- 对业务代码无侵入

### 2.2 TCC 模式

TCC（Try-Confirm-Cancel）模式是手动补偿模式：

- **Try**：预留资源
- **Confirm**：确认执行
- **Cancel**：取消执行

### 2.3 Saga 模式

Saga 模式是长事务编排模式：

- 每个参与者都是独立的服务
- 通过状态机编排事务流程
- 支持正向和逆向操作

## 三、AT 模式的两阶段提交

### 3.1 核心角色

Seata AT 模式有三个核心角色：

| 角色 | 说明 | 英文缩写 |
| --- | --- | --- |
| **事务协调器（TC）** | 管理整个事务的提交和回滚 | Transaction Coordinator |
| **事务管理器（TM）** | 定义事务的边界，开启/提交/回滚事务 | Transaction Manager |
| **资源管理器（RM）** | 管理分支事务，连接数据库，执行 SQL | Resource Manager |

### 3.2 两阶段提交流程

**第一阶段（解析 SQL，生成 Undolog）**：

```
TM 向 TC 申请开启全局事务
        ↓
TC 生成 XID（全局事务 ID）
        ↓
TM 将 XID 传递给各微服务
        ↓
各 RM 执行 SQL，生成 Undolog
        ↓
各 RM 向 TC 注册分支事务
        ↓
各 RM 提交本地事务（自动）
        ↓
TC 通知各 RM 提交/回滚
```

**第二阶段（提交或回滚）**：

```
情况 1：所有分支都成功
TC 通知所有 RM 提交
各 RM 删除 Undolog
全局事务提交完成

情况 2：任意分支失败
TC 通知所有 RM 回滚
各 RM 根据 Undolog 生成逆向 SQL，执行回滚
删除 Undolog
全局事务回滚完成
```

### 3.3 Undolog 的记录

Undolog 记录了修改前后的数据镜像：

```sql
-- 原始数据
UPDATE inventory SET stock = stock - 1 WHERE id = 1;

-- Undolog 记录
{
    "beforeImage": {"id": 1, "stock": 100},
    "afterImage": {"id": 1, "stock": 99}
}
```

回滚时，根据 `beforeImage` 恢复数据。

## 四、AT 模式的使用

### 4.1 引入依赖

```xml
<dependency>
    <groupId>com.alibaba.cloud</groupId>
    <artifactId>spring-cloud-starter-alibaba-seata</artifactId>
</dependency>
<dependency>
    <groupId>io.seata</groupId>
    <artifactId>seata-spring-boot-starter</artifactId>
</dependency>
```

### 4.2 配置

```yaml title="application.yml"
seata:
  tx-service-group: my_tx_group
  registry:
    type: nacos
    nacos:
      server-addr: nacos-server:8848
      group: SEATA_GROUP
  config:
    type: nacos
    nacos:
      server-addr: nacos-server:8848
      group: SEATA_GROUP
  client:
    rm:
      async-commit-buffer-size: 1000
      report-success-enable: false
```

### 4.3 @GlobalTransactional 注解

在开启分布式事务的方法上添加注解：

```java
@Service
public class OrderServiceImpl implements OrderService {
    
    @Autowired
    private OrderMapper orderMapper;
    
    @Autowired
    private InventoryFeignClient inventoryFeignClient;
    
    @GlobalTransactional(rollbackFor = Exception.class)  // 开启分布式事务
    @Override
    public void createOrder(OrderDTO orderDTO) {
        // 1. 创建订单（本地事务，自动纳入全局事务）
        Order order = new Order();
        order.setProductId(orderDTO.getProductId());
        order.setQuantity(orderDTO.getQuantity());
        orderMapper.insert(order);
        
        // 2. 扣减库存（远程调用，自动纳入全局事务）
        inventoryFeignClient.deductStock(orderDTO.getProductId(), orderDTO.getQuantity());
    }
}
```

### 4.4 效果

```
全局事务开始
        ↓
订单服务：创建订单（自动注册分支事务）
        ↓
库存服务：扣减库存（自动注册分支事务）
        ↓
分支 1 成功 ✓
        ↓
分支 2 成功 ✓
        ↓
全局事务提交
        ↓
所有分支一起提交
```

如果库存服务调用失败：

```
全局事务开始
        ↓
订单服务：创建订单 ✓
        ↓
库存服务：扣减库存 ✗（抛出异常）
        ↓
全局事务回滚
        ↓
订单服务：根据 Undolog 回滚
        ↓
所有分支一起回滚
```

## 五、AT 模式的适用场景

### 5.1 适用场景

- 对一致性要求较高，但可以接受最终一致性的业务
- 需要对业务代码无侵入
- 数据量适中，不涉及大量数据迁移

### 5.2 不适用场景

- 对强一致性有要求的业务（如金融转账）
- 数据量非常大，需要自定义分库分表
- 需要长事务的业务

## 六、面试追问方向

- **Seata AT 模式的两阶段提交是怎样的？** 第一阶段解析 SQL、生成 Undolog、提交本地事务；第二阶段根据全局事务结果，通知各分支提交或回滚。
- **Undolog 的作用是什么？** 记录修改前后的数据镜像，用于回滚时恢复数据。
- **AT 模式和 TCC 模式的区别是什么？** AT 模式对业务无侵入，自动生成逆向 SQL；TCC 模式需要手动实现 Try/Confirm/Cancel。
- **Seata 如何保证分布式事务的隔离性？** 通过全局锁机制，确保在全局事务提交前，其他全局事务无法修改相同数据。

---

## 七、总结

Seata AT 模式通过**两阶段提交**和**Undolog**，实现了对业务代码无侵入的分布式事务解决方案：

- **TC（事务协调器）**：管理全局事务状态
- **TM（事务管理器）**：通过 `@GlobalTransactional` 定义事务边界
- **RM（资源管理器）**：管理分支事务，生成 Undolog
- **两阶段提交**：第一阶段提交本地事务，第二阶段统一提交或回滚

相比传统的 XA 方案，AT 模式性能更高；相比 TCC 方案，AT 模式对业务无侵入。
