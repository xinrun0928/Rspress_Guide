# 微服务统一认证：OAuth2 + JWT 的实战方案

import { Badge } from '@rspress/core/theme';

微服务架构下，认证服务怎么设计？

在单体应用中，认证很简单——用户登录后，把信息存在 Session 中，后续请求从 Session 读取。但在微服务架构中，Session 模式遇到了挑战：

- 服务分布在多台机器上，Session 怎么共享？
- 服务间调用也需要认证，Session 怎么传递？
- 网关需要统一鉴权，Session 在哪？

**OAuth2 + JWT，是目前微服务架构下最流行的认证方案。**

## 一、OAuth2 的四种授权模式

### 1.1 OAuth2 的核心概念

OAuth2 有几个核心概念：

| 概念 | 说明 |
| --- | --- |
| **Resource Owner** | 资源所有者，即用户 |
| **Client** | 客户端应用，请求访问令牌 |
| **Authorization Server** | 授权服务器，颁发访问令牌 |
| **Resource Server** | 资源服务器，验证访问令牌 |

### 1.2 四种授权模式

| 模式 | 说明 | 适用场景 |
| --- | --- | --- |
| **授权码模式（Authorization Code）** | 最安全，适合有后端的应用 | Web 应用 |
| **密码模式（Password）** | 用户直接把密码给客户端 | 可信的应用 |
| **客户端凭证模式（Client Credentials）** | 不涉及用户，应用自己的凭证 | 服务间通信 |
| **简化模式（Implicit）** | 不推荐，已废弃 | 已废弃 |

### 1.3 授权码模式

最安全的授权模式，适合有后端服务器的 Web 应用：

```
用户 → 应用 → 授权服务器 → 用户登录授权
        ↓
   授权码返回
        ↓
   用授权码换令牌
        ↓
   获取访问令牌
```

### 1.4 客户端凭证模式

用于服务间的认证，不需要用户参与：

```java
// 获取客户端凭证令牌
@RestController
public class AuthController {
    
    @PostMapping("/oauth/token")
    public TokenResponse getToken(@RequestBody TokenRequest request) {
        // 验证客户端凭证
        if (!validateClient(request.getClientId(), request.getClientSecret())) {
            throw new UnauthorizedException("无效的客户端凭证");
        }
        
        // 生成访问令牌
        String accessToken = jwtTokenProvider.generateToken(
            Map.of("client_id", request.getClientId())
        );
        
        return new TokenResponse(accessToken, "bearer", 3600);
    }
}
```

## 二、授权码模式在微服务中的使用

### 2.1 架构设计

```
                    ┌──────────────┐
                    │   前端应用    │
                    └──────┬───────┘
                           │ 1. 跳转授权
                           ↓
┌─────────────────────────────────────────────────────┐
│                 授权服务器                             │
│   ┌─────────────────────────────────────────┐       │
│   │  用户登录页面                             │       │
│   │  授权确认页面                            │       │
│   └─────────────────────────────────────────┘       │
└──────────────────────┬───────────────────────────────┘
                       │ 2. 返回授权码
                       ↓
┌─────────────────────────────────────────────────────┐
│                     网关                              │
│   ┌─────────────────────────────────────────┐       │
│   │  统一鉴权                                │       │
│   │  Token 验证                             │       │
│   └─────────────────────────────────────────┘       │
└─────────────────────────────────────────────────────┘
                       │
          ┌────────────┼────────────┐
          ↓            ↓            ↓
    ┌──────────┐ ┌──────────┐ ┌──────────┐
    │ 用户服务  │ │ 订单服务  │ │ 商品服务  │
    └──────────┘ └──────────┘ └──────────┘
```

### 2.2 授权流程

```java
// 1. 用户访问受保护资源，被网关拦截
// 2. 网关检查 Token，无效则跳转到授权服务器
// 3. 用户登录授权，授权服务器返回授权码
// 4. 后端用授权码换取 Token
// 5. Token 存储在客户端，后续请求携带 Token
```

## 三、JWT 令牌的结构

### 3.1 JWT 的三部分

JWT 由三部分组成，用点号分隔：

```
Header.Payload.Signature
```

**Header（头部）**：
```json
{
  "alg": "HS256",
  "typ": "JWT"
}
```

**Payload（载荷）**：
```json
{
  "sub": "user123",           // 用户 ID
  "name": "张三",              // 用户名
  "roles": ["admin", "user"], // 角色
  "exp": 1704067200,          // 过期时间
  "iat": 1704063600,           // 签发时间
  "iss": "auth-server"         // 签发者
}
```

**Signature（签名）**：
```
HMACSHA256(
  base64UrlEncode(header) + "." + base64UrlEncode(payload),
  secret_key
)
```

### 3.2 JWT 的优势

| 优势 | 说明 |
| --- | --- |
| **无状态** | 令牌包含认证信息，服务端无需存储 Session |
| **可验证** | 签名确保令牌不被篡改 |
| **跨域认证** | 令牌可以在任何域使用 |
| **性能** | 验证令牌只需解密，无需查库 |

### 3.3 JWT 生成与验证

```java
// 生成 JWT
public String generateToken(String userId, List<String> roles) {
    Map<String, Object> claims = new HashMap<>();
    claims.put("userId", userId);
    claims.put("roles", roles);
    
    return Jwts.builder()
        .setClaims(claims)
        .setSubject(userId)
        .setIssuedAt(new Date())
        .setExpiration(new Date(System.currentTimeMillis() + 3600000))
        .signWith(SignatureAlgorithm.HS256, secretKey)
        .compact();
}

// 验证 JWT
public Claims parseToken(String token) {
    return Jwts.parser()
        .setSigningKey(secretKey)
        .parseClaimsJws(token)
        .getBody();
}
```

## 四、Resource Server 的 JWT 验证

### 4.1 引入依赖

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-oauth2-resource-server</artifactId>
</dependency>
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-oauth2-resource-server</artifactId>
</dependency>
```

### 4.2 配置 Resource Server

```yaml title="application.yml"
spring:
  security:
    oauth2:
      resourceserver:
        jwt:
          # JWT 的公钥或密钥
          # 生产环境通常使用公钥（从授权服务器获取）
          public-key-location: classpath:public.pem
          # 或者使用 JWK Set URI
          # jwk-set-uri: https://auth-server/oauth2/jwks
```

### 4.3 安全配置

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {
    
    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        http
            .authorizeRequests()
                .antMatchers("/public/**", "/actuator/**").permitAll()
                .antMatchers("/admin/**").hasAuthority("ROLE_ADMIN")
                .anyRequest().authenticated()
            .and()
            .oauth2ResourceServer()
                .jwt()
                    .jwtAuthenticationConverter(jwtAuthenticationConverter());
        
        return http.build();
    }
    
    @Bean
    public JwtAuthenticationConverter jwtAuthenticationConverter() {
        JwtGrantedAuthoritiesConverter grantedAuthoritiesConverter = 
            new JwtGrantedAuthoritiesConverter();
        // 从 JWT 的 roles 字段提取权限
        grantedAuthoritiesConverter.setAuthoritiesClaimName("roles");
        grantedAuthoritiesConverter.setAuthorityPrefix("ROLE_");
        
        JwtAuthenticationConverter jwtAuthenticationConverter = 
            new JwtAuthenticationConverter();
        jwtAuthenticationConverter.setJwtGrantedAuthoritiesConverter(
            grantedAuthoritiesConverter);
        
        return jwtAuthenticationConverter;
    }
}
```

## 五、微服务间 Token 传递

### 5.1 网关统一鉴权

在 Gateway 层统一验证 Token：

```java
@Component
public class JwtAuthenticationFilter implements GlobalFilter {
    
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        // 1. 从请求头获取 Token
        String token = exchange.getRequest().getHeaders()
            .getFirst("Authorization");
        
        if (StringUtils.isBlank(token)) {
            // 无 Token，放行（让下游服务自行处理）
            return chain.filter(exchange);
        }
        
        // 2. 验证 Token
        try {
            Claims claims = jwtService.verify(token);
            
            // 3. 将用户信息传递给下游服务
            ServerHttpRequest modifiedRequest = exchange.getRequest().mutate()
                .header("X-User-Id", claims.getUserId())
                .header("X-User-Roles", String.join(",", claims.getRoles()))
                .build();
            
            return chain.filter(
                exchange.mutate().request(modifiedRequest).build()
            );
        } catch (JwtException e) {
            exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
            return exchange.getResponse().setComplete();
        }
    }
}
```

### 5.2 Feign Token 传递

服务间调用时，需要传递 Token：

```java
@Configuration
public class FeignConfig {
    
    @Bean
    public RequestInterceptor requestInterceptor() {
        return template -> {
            // 从当前请求上下文获取 Token
            Authentication authentication = 
                SecurityContextHolder.getContext().getAuthentication();
            
            if (authentication != null && authentication.getCredentials() != null) {
                String token = authentication.getCredentials().toString();
                template.header("Authorization", token);
            }
        };
    }
}
```

## 六、Token 黑名单

### 6.1 为什么需要黑名单

JWT 是无状态的，签发后无法撤销。需要黑名单机制来"撤销"令牌。

### 6.2 Redis 黑名单实现

```java
@Service
public class TokenBlacklistService {
    
    @Autowired
    private StringRedisTemplate redisTemplate;
    
    private static final String BLACKLIST_PREFIX = "token:blacklist:";
    
    // 添加到黑名单
    public void addToBlacklist(String token) {
        Claims claims = jwtService.parseClaims(token);
        Date expiration = claims.getExpiration();
        
        // 计算剩余有效期
        long ttl = (expiration.getTime() - System.currentTimeMillis()) / 1000;
        if (ttl > 0) {
            String key = BLACKLIST_PREFIX + token;
            redisTemplate.opsForValue().set(key, "1", ttl, TimeUnit.SECONDS);
        }
    }
    
    // 检查是否在黑名单
    public boolean isBlacklisted(String token) {
        String key = BLACKLIST_PREFIX + token;
        return Boolean.TRUE.equals(redisTemplate.hasKey(key));
    }
}
```

### 6.3 网关拦截黑名单

```java
@Override
public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
    String token = extractToken(exchange.getRequest());
    
    if (token != null && tokenBlacklistService.isBlacklisted(token)) {
        exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
        return exchange.getResponse().setComplete();
    }
    
    return chain.filter(exchange);
}
```

## 七、刷新令牌机制

### 7.1 Access Token vs Refresh Token

| 类型 | 有效期 | 用途 |
| --- | --- | --- |
| Access Token | 短（15 分钟 ~ 1 小时） | 访问受保护资源 |
| Refresh Token | 长（7 天 ~ 30 天） | 换取新的 Access Token |

### 7.2 刷新流程

```java
@PostMapping("/oauth/refresh")
public TokenResponse refreshToken(@RequestBody RefreshTokenRequest request) {
    // 1. 验证 Refresh Token
    Claims claims = jwtService.verifyRefreshToken(request.getRefreshToken());
    
    // 2. 检查 Refresh Token 是否在黑名单
    if (tokenBlacklistService.isBlacklisted(request.getRefreshToken())) {
        throw new UnauthorizedException("Refresh Token 已失效");
    }
    
    // 3. 生成新的 Access Token 和 Refresh Token
    String newAccessToken = jwtService.generateAccessToken(claims.getSubject());
    String newRefreshToken = jwtService.generateRefreshToken(claims.getSubject());
    
    // 4. 将旧的 Refresh Token 加入黑名单
    tokenBlacklistService.addToBlacklist(request.getRefreshToken());
    
    return new TokenResponse(newAccessToken, newRefreshToken);
}
```

## 八、面试追问方向

- **OAuth2 的四种授权模式是什么？** 授权码模式、密码模式、客户端凭证模式、简化模式。
- **JWT 的结构是怎样的？** Header.Payload.Signature，由 Base64 编码的 JSON 组成。
- **JWT 为什么不需要在服务端存储？** 因为 JWT 包含签名，签名验证通过即可确认其合法性，无需查库。
- **如何实现 Token 的撤销？** 可以使用 Redis 黑名单，在 Token 有效期内检查黑名单。

## 九、总结

OAuth2 + JWT 是微服务认证的最佳实践：

- **OAuth2**：提供标准的授权框架，支持多种授权模式
- **JWT**：无状态的令牌格式，携带用户信息
- **Resource Server**：验证 JWT 的组件
- **网关鉴权**：在网关层统一验证 Token
- **Token 黑名单**：实现 Token 撤销
- **刷新令牌**：延长用户登录状态

这套方案在微服务架构中被广泛采用，是每个微服务开发者必须掌握的技能。
