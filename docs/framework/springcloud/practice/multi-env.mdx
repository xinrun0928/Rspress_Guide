# Spring Cloud 多环境：Bootstrap + Nacos Namespace

import { Badge } from '@rspress/core/theme';

dev、test、prod 三套环境，配置怎么管理？

当你的微服务数量达到几十个时，环境管理成了一个头疼的问题：
- 怎么确保 dev 的配置不会影响到 prod？
- 怎么快速切换环境？
- 不同环境的公共配置怎么复用？

**Spring Cloud + Nacos Namespace，提供了一套完整的多环境配置管理方案。**

## 一、Nacos Namespace 的多环境隔离

### 1.1 Namespace 的作用

Namespace 是 Nacos 配置中心的顶层隔离机制：

```
Nacos Server
  │
  ├── Namespace: dev（开发环境）
  │     ├── Group: DEFAULT_GROUP
  │     │     └── order-service.yaml
  │     └── Group: ORDER-SUIT
  │           └── order-service.yaml
  │
  ├── Namespace: test（测试环境）
  │     └── ...
  │
  └── Namespace: prod（生产环境）
        └── ...
```

每个 Namespace 有独立的配置，互不影响。

### 1.2 创建 Namespace

在 Nacos 控制台创建：

```
配置管理 → 命名空间 → 新建命名空间
  命名空间ID: dev (或 UUID)
  命名空间名: 开发环境
```

> [!TIP]
> 命名空间 ID 是客户端使用的标识，不是显示名称。记住：**客户端配置使用 ID，不是名称**。

## 二、bootstrap.yml 的优先级

### 2.1 为什么使用 bootstrap.yml

Spring Cloud 使用 bootstrap.yml 来加载配置中心的配置，原因：

1. **加载顺序**：bootstrap.yml 在 application.yml 之前加载
2. **用途**：基础设施配置（配置中心、服务注册）需要"提前"加载
3. **隔离**：业务配置（application.yml）依赖基础设施配置

```yaml title="bootstrap.yml"
spring:
  application:
    name: order-service
  profiles:
    active: prod
  cloud:
    nacos:
      config:
        server-addr: nacos-server:8848
        namespace: dev  # 在这里配置 Namespace ID
        group: ORDER-SUIT
        file-extension: yaml
```

### 2.2 bootstrap 与 application 的区别

| 维度 | bootstrap.yml | application.yml |
| --- | --- | --- |
| 加载顺序 | 先加载 | 后加载 |
| 用途 | 基础设施配置 | 业务配置 |
| 参与配置中心 | 是 | 是 |
| 可使用占位符 | 部分 | 全部 |

### 2.3 加载顺序验证

```
配置加载顺序：
1. bootstrap.yml → spring.application.name
2. bootstrap.yml → spring.cloud.nacos.config.*
3. Nacos: ${application.name}.yaml
4. Nacos: ${application.name}-${profile}.yaml
5. application.yml → spring.application.name
6. application.yml → 业务配置
```

## 三、spring.cloud.nacos.config.namespace 配置

### 3.1 配置项

```yaml title="bootstrap.yml"
spring:
  cloud:
    nacos:
      config:
        # Nacos Server 地址
        server-addr: ${NACOS_SERVER:127.0.0.1:8848}
        
        # Namespace ID（必填）
        namespace: ${NACOS_NAMESPACE:dev}
        
        # Group 名称
        group: ${NACOS_GROUP:DEFAULT_GROUP}
        
        # 配置文件格式
        file-extension: yaml
        
        # 是否开启配置刷新
        refresh-enabled: true
```

### 3.2 通过环境变量切换

生产环境中，通常通过环境变量切换 Namespace：

```yaml title="bootstrap.yml"
spring:
  cloud:
    nacos:
      config:
        server-addr: ${NACOS_SERVER}
        namespace: ${NACOS_NAMESPACE}  # 环境变量注入
        group: ${NACOS_GROUP}
```

部署脚本：

```bash
# 开发环境部署
NACOS_SERVER=nacos-dev:8848 \
NACOS_NAMESPACE=dev \
java -jar order-service.jar

# 生产环境部署
NACOS_SERVER=nacos-prod:8848 \
NACOS_NAMESPACE=prod \
java -jar order-service.jar
```

## 四、Group 的使用

### 4.1 Group 的作用

Group 是 Namespace 内的第二层隔离：

```
Namespace: prod
  │
  ├── Group: DEFAULT_GROUP
  │     └── order-service.yaml
  │
  ├── Group: ORDER-SUIT
  │     ├── order-service.yaml
  │     └── payment-service.yaml
  │
  └── Group: INFRASTRUCTURE
        ├── database.yaml
        └── redis.yaml
```

### 4.2 典型使用场景

**场景 1：按业务线隔离**

```yaml
# 订单业务线
spring.cloud.nacos.config.group: ORDER-SUIT

# 用户业务线
spring.cloud.nacos.config.group: USER-CENTER
```

**场景 2：按项目隔离**

```yaml
# 项目 A
spring.cloud.nacos.config.group: PROJECT-A

# 项目 B
spring.cloud.nacos.config.group: PROJECT-B
```

### 4.3 共享配置使用 Group

公共配置可以通过 Group 共享：

```yaml title="bootstrap.yml"
spring:
  cloud:
    nacos:
      config:
        # 使用共享配置
        shared-dataids: database.yaml,redis.yaml
        # 但共享配置的 Group 不是 DEFAULT_GROUP？
        # 需要使用 extension-configs
        extension-configs:
          - data-id: database.yaml
            group: INFRASTRUCTURE
            refresh: true
```

## 五、配置文件的 Data ID 命名规范

### 5.1 命名规范

推荐的 Data ID 命名规范：

```text
${spring.application.name}-${spring.profiles.active}.${file-extension}
```

| 环境 | spring.profiles.active | Data ID |
| --- | --- | --- |
| 开发 | dev | order-service-dev.yaml |
| 测试 | test | order-service-test.yaml |
| 生产 | prod | order-service-prod.yaml |

### 5.2 完整的配置示例

```yaml title="bootstrap.yml"
spring:
  application:
    name: order-service
  profiles:
    active: prod
  cloud:
    nacos:
      server-addr: nacos-cluster:8848
      config:
        namespace: prod
        group: ORDER-SUIT
        file-extension: yaml
        refresh-enabled: true
```

对应的 Nacos 配置：

```
Namespace: prod
Group: ORDER-SUIT
  ├── order-service.yaml       # 默认配置（所有环境共享）
  ├── order-service-prod.yaml  # 生产环境配置（会覆盖默认配置）
  ├── database.yaml            # 数据库配置
  └── redis.yaml              # Redis 配置
```

### 5.3 配置加载顺序

```java
// 加载顺序（后面的覆盖前面的）
1. order-service.yaml              // 默认配置
2. order-service-prod.yaml        // 环境配置
3. shared-dataids: database.yaml   // 共享配置
4. extension-configs              // 扩展配置
```

## 六、Bootstrap 加载顺序

### 6.1 为什么在服务注册前加载

配置中心有一个特点：**它需要在服务注册之前加载**。

原因：
- 数据库连接池大小可能来自配置中心
- 如果配置中心连不上，服务可能无法正常启动
- 服务注册不依赖配置中心，但业务 Bean 依赖

### 6.2 Bootstrap 上下文

Spring Cloud 引入了 Bootstrap 上下文：

```java
// Bootstrap 上下文加载顺序
1. BootstrapApplicationListener 监听 ApplicationEnvironmentPreparedEvent
2. 创建 Bootstrap 上下文
3. 加载 bootstrap.yml / bootstrap.properties
4. 初始化 PropertySourceLocator
5. 从配置中心加载配置
6. 主上下文加载 application.yml
```

### 6.3 禁用 Bootstrap

如果不需要配置中心，可以禁用：

```yaml title="application.yml"
spring:
  cloud:
    bootstrap:
      enabled: false
```

## 七、多环境部署实践

### 7.1 Kubernetes 部署

```yaml title="deployment.yaml"
apiVersion: apps/v1
kind: Deployment
metadata:
  name: order-service
spec:
  containers:
    - name: order-service
      env:
        - name: NACOS_SERVER
          value: "nacos-cluster:8848"
        - name: NACOS_NAMESPACE
          value: "prod"
        - name: NACOS_GROUP
          value: "ORDER-SUIT"
        - name: SPRING_PROFILES_ACTIVE
          value: "prod"
```

### 7.2 Docker Compose 部署

```yaml title="docker-compose.yml"
services:
  order-service:
    image: order-service:latest
    environment:
      - NACOS_SERVER=nacos:8848
      - NACOS_NAMESPACE=prod
      - SPRING_PROFILES_ACTIVE=prod
```

### 7.3 Shell 脚本部署

```bash
#!/bin/bash
# deploy.sh

ENV=$1  # dev / test / prod
JAR_FILE="order-service.jar"

# 根据环境设置变量
case $ENV in
  dev)
    NACOS_NAMESPACE=dev
    NACOS_GROUP=DEFAULT_GROUP
    ;;
  test)
    NACOS_NAMESPACE=test
    NACOS_GROUP=DEFAULT_GROUP
    ;;
  prod)
    NACOS_NAMESPACE=prod
    NACOS_GROUP=ORDER-SUIT
    ;;
esac

# 启动应用
java -jar $JAR_FILE \
  --spring.profiles.active=$ENV \
  --spring.cloud.nacos.config.namespace=$NACOS_NAMESPACE \
  --spring.cloud.nacos.config.group=$NACOS_GROUP
```

## 八、面试追问方向

- **Namespace 和 Group 的区别是什么？** Namespace 是最外层隔离，用于区分环境；Group 是第二层隔离，用于区分业务线或项目。
- **bootstrap.yml 和 application.yml 的区别是什么？** bootstrap.yml 在 application.yml 之前加载，用于配置中心等基础设施配置。
- **配置加载的优先级是怎样的？** Data ID 越具体优先级越高：`${name}-${profile}.yaml > ${name}.yaml > shared-dataids > extension-configs`。
- **如何通过环境变量切换 Nacos Namespace？** 通过 `spring.cloud.nacos.config.namespace` 配置项指定，支持 `${环境变量名:默认值}` 语法。

## 九、总结

Nacos + Bootstrap 提供了一套完整的多环境配置管理方案：

- **Namespace**：环境隔离，dev / test / prod 各有独立空间
- **Group**：业务隔离，同一环境内按业务线或项目分组
- **bootstrap.yml**：在服务注册前加载配置中心配置
- **Data ID 规范**：`${name}-${profile}.yaml` 格式，优先级清晰
- **环境变量**：支持通过环境变量切换环境配置

这套方案让多环境配置管理变得清晰、可控、易维护。
