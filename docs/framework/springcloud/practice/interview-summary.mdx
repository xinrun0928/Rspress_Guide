## 一、服务注册与发现

### 1.1 服务注册流程

**问：Nacos 的服务注册流程是怎样的？**

答：

1. 应用启动时，`NacosAutoServiceRegistration` 监听 `ApplicationReadyEvent`
2. 调用 `NamingService.registerInstance()` 向 Nacos Server 发送注册请求
3. 注册成功后，启动心跳任务定期发送心跳
4. Nacos Server 通过心跳机制检查服务健康状态

---

### 1.2 临时实例与永久实例

**问：临时实例和永久实例有什么区别？**

答：

| 类型 | ephemeral | 心跳机制 | 适用场景 |
| --- | --- | --- | --- |
| 临时实例 | `true` | 客户端主动心跳，15 秒不健康，30 秒剔除 | 无状态服务 |
| 永久实例 | `false` | 服务端主动探测，不自动剔除 | 需要保留实例信息的场景 |

---

### 1.3 宕机后的处理

**问：如果 Nacos Server 全部宕机，服务还能调用吗？**

答：可以。Ribbon/Feign 在本地缓存了服务实例列表，可以继续调用已有实例。但无法感知新的实例变化，无法注册新实例。

## 二、服务调用

### 2.1 Ribbon 和 OpenFeign 的区别

**问：Ribbon 和 OpenFeign 的区别是什么？**

答：

| 维度 | Ribbon | OpenFeign |
| --- | --- | --- |
| 编程方式 | 代码调用 RestTemplate | 声明式接口 |
| 负载均衡 | 内置多种策略 | 底层集成 Ribbon |
| 代码量 | 多（URL 拼接等） | 少（只需定义接口） |
| 可读性 | 一般 | 高 |

---

### 2.2 Feign 的工作原理

**问：Feign 的工作原理是什么？**

答：

1. 通过 `@FeignClient` 定义接口，Spring 为其创建 JDK 动态代理对象
2. 请求时调用 `FeignInvocationHandler.invoke()`，解析注解获取目标 URL 和请求参数
3. 通过 Ribbon 获取目标服务实例（负载均衡）
4. 由 HTTP Client 发送请求并解析响应

---

### 2.3 超时配置

**问：Feign 的超时配置项有哪些？**

答：

| 配置项 | 说明 | 默认值 |
| --- | --- | --- |
| `connectTimeout` | 连接超时 | 10 秒 |
| `readTimeout` | 读取超时 | 60 秒 |

## 三、网关

### 3.1 Gateway 的请求处理流程

**问：Gateway 的请求处理流程是什么？**

答：

```
请求进入
    ↓
RoutePredicateHandlerMapping（匹配路由）
    ↓
DispatcherHandler（分发请求）
    ↓
GatewayFilterChain（执行过滤器链）
    ↓
ProxyExchange（转发请求到后端服务）
    ↓
PostFilter（后处理）
    ↓
响应返回
```

---

### 3.2 Filter 的执行顺序

**问：Gateway 中 Filter 的执行顺序如何确定？**

答：通过实现 `Ordered` 接口的 `getOrder()` 方法或 `@Order` 注解。数值越小，优先级越高。

- Pre 过滤器：按 order 从小到大执行
- Post 过滤器：按 order 从大到小执行

---

### 3.3 限流算法

**问：Gateway 限流使用的是什么算法？**

答：令牌桶算法。

| 参数 | 说明 |
| --- | --- |
| `replenishRate` | 令牌补充速率（每秒补充的令牌数） |
| `burstCapacity` | 令牌桶容量（最大存储的令牌数） |

Redis + Lua 脚本保证限流的原子性。

## 四、配置中心

### 4.1 配置加载优先级

**问：Nacos 配置的加载优先级是怎样的？**

答：Namespace `>` Group `>` Data ID。

同一级别内，后加载的配置覆盖先加载的：

```
${name}-${profile}.yaml > ${name}.yaml > shared-dataids > extension-configs
```

---

### 4.2 热更新原理

**问：@RefreshScope 的热更新原理是什么？**

答：

1. 配置变化时，`NacosConfigListener` 收到推送
2. `ContextRefresher` 发布 `EnvironmentChangeEvent`
3. `RefreshScope` 清空 Bean 缓存
4. 下次访问 `@RefreshScope` Bean 时，重新创建实例（使用新配置）

---

### 4.3 配置一致性

**问：Nacos 配置中心如何保证数据一致性？**

答：

- 单机模式：使用本地文件存储
- 集群模式：使用 Raft 协议保证一致性
- 2.0+ 版本：使用集中式日志复制

## 五、容错治理

### 5.1 熔断和降级的区别

**问：熔断和降级的区别是什么？**

答：

| 维度 | 熔断（Circuit Breaker） | 降级（Fallback） |
| --- | --- | --- |
| 触发条件 | 失败率达到阈值 | 单次调用失败 |
| 行为 | 拒绝后续请求 | 返回兜底数据 |
| 恢复 | 等待探测后恢复 | 立即恢复 |

---

### 5.2 Sentinel 滑动窗口

**问：Sentinel 的滑动窗口算法是什么？**

答：Sentinel 使用 `BucketLeapArray` 实现滑动窗口统计：

- 将时间分成多个 Bucket（默认 10 个）
- 每个 Bucket 记录一段时间内的请求数、响应时间等
- 计算时，将最近 N 个 Bucket 的数据汇总

这种设计保证了统计的精确性和性能的平衡。

---

### 5.3 Hystrix 和 Sentinel 对比

**问：Hystrix 和 Sentinel 有什么区别？**

答：

| 维度 | Hystrix | Sentinel |
| --- | --- | --- |
| 维护状态 | 停止维护 | 活跃维护 |
| 流量控制 | 不支持 | 支持 |
| 隔离方式 | 线程池隔离 | 信号量隔离 |
| 规则配置 | 代码配置 | Dashboard 可视化配置 |
| 系统自适应 | 不支持 | 支持 |

## 六、分布式事务

### 6.1 Seata AT 模式原理

**问：Seata AT 模式的工作原理是什么？**

答：

**第一阶段（自动完成）**：
1. TM 开启全局事务，获取 XID
2. 各 RM 执行 SQL，生成 Undolog
3. 各 RM 提交本地事务（自动）
4. 向 TC 注册分支事务

**第二阶段**：
- TC 判断全局是否成功
- 成功：删除 Undolog
- 失败：根据 Undolog 生成逆向 SQL，回滚

---

### 6.2 全局锁

**问：Seata 如何保证隔离性？**

答：Seata 通过**全局锁**保证隔离性：

- 本地事务提交前，获取该行的**全局锁**
- 获取成功才提交，失败则回滚
- 其他全局事务无法修改持有全局锁的行

## 七、综合问题

### 7.1 微服务拆分原则

**问：微服务拆分的原则是什么？**

答：

- **高内聚低耦合**：同一业务域的功能放在一起
- **单一职责**：每个服务只负责一件事
- **团队边界**：按团队划分服务
- **独立部署**：服务可以独立部署和扩缩容

---

### 7.2 服务间通信方式

**问：微服务间的通信方式有哪些？**

答：

| 方式 | 协议 | 特点 |
| --- | --- | --- |
| 同步通信 | REST、gRPC | 实时响应 |
| 异步通信 | 消息队列 | 解耦、削峰 |

---

### 7.3 Spring Cloud 和 Dubbo 的区别

**问：Spring Cloud 和 Dubbo 有什么区别？**

答：

| 维度 | Spring Cloud | Dubbo |
| --- | --- | --- |
| 通信协议 | HTTP（REST） | RPC（Dubbo 协议） |
| 生态 | 完整生态 | 主要关注服务治理 |
| 社区 | 活跃 | 活跃 |
| 学习成本 | 较高 | 中 |

## 八、面试技巧

### 8.1 回答问题的技巧

- **结构化回答**：使用"总-分-总"结构
- **结合原理**：不仅要会使用，还要理解原理
- **实践经验**：如果有实践经验，可以补充说明
- **举一反三**：可以适当延伸，比如对比其他方案

### 8.2 常见陷阱

- **只背答案**：面试官会追问原理
- **过于理论**：要结合实际使用场景
- **不懂装懂**：不会的问题可以诚实说不会
