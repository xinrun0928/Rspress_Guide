# Spring Cloud 版本命名与组件选型策略

你有没有注意过 Spring Cloud 的版本号不是 `1.0`、`2.0` 这样递增的，而是 `Angel.SR6`、`Brixton.SR7`、`Greenwich.SR2` 这样看起来像地铁站名的命名方式？

这套命名规则不是随意设计的，它背后有清晰的逻辑。

---

## 版本命名规则：从地铁线到年份

Spring Cloud 的版本号经历了两个阶段：

### 第一阶段：地铁线路命名法（2015-2020）

从 2015 年开始，Spring Cloud 使用**伦敦地铁站名**作为版本号，按字母序排列：

| 版本系列 | 发布时间 | 代表组件 |
|---|---|---|
| **Angel** | 2015.12 | 首个版本，集成 Eureka 1.x |
| **Brixton** | 2016.05 | 集成 Feign、Hystrix |
| **Camden** | 2016.09 | 引入 Turbine、Ribbon |
| **Dalston** | 2017.01 | 引入 Config Server |
| **Edgware** | 2017.11 | 引入 Gateway |
| **Finchley** | 2018.01 | 支持 Spring Boot 2.0 |
| **Greenwich** | 2019.02 | 引入 OpenFeign |
| **Hoxton** | 2019.12 | 引入 LoadBalancer |

这套命名有几个特点：

- **按字母序排列**：从 A 到 H，后来版本号越来越大，不是因为功能变复杂了，而是因为字母序往后走了
- **每个版本都有 SR 修正版**：如 `Finchley.SR4` 表示 Finchley 系列的第 4 次稳定修正
- **相邻版本不兼容**：从 Edgware 升级到 Finchley，可能需要同时升级 Spring Boot 版本

### 第二阶段：年份命名法（2020 至今）

2020 年之后，Spring Cloud 改用**年份命名**：

| 版本系列 | 发布时间 | 说明 |
|---|---|---|
| **2020.0** | 2020.12 | 首个年份版本，代号 "Ilford" |
| **2021.0** | 2021.01 | 引入 Spring Cloud CircuitBreaker |
| **2022.0** | 2022.01 | 支持 Spring Boot 3.0 |
| **2023.0** | 2023.01 | 最新版本 |

年份版本的格式更清晰：`年份.次版本`，如 `2023.0.0`，其中第三位是补丁版本号。

## SR 版本的含义

Spring Cloud 的版本号后缀如 `SR1`、`SR2`、`SR6` 是什么意思？

**SR = Service Release**，即服务发布版本。这是 Spring Cloud 对依赖库的补丁合集，确保版本之间的兼容性。

```
Greenwich.SR2
    │
    ├── Greenwich：主版本（伦敦地铁站名）
    │
    └── SR2：Service Release 2，第 2 次补丁版本
```

当你看到 `Greenwich.SR2` 时，意味着这是 Greenwich 主版本的第 2 个补丁包，通常包含：

- 依赖库的 Bug 修复
- 安全漏洞修补
- 与其他 Spring 项目的版本对齐

**实战建议**：生产环境不要用 `Greenwich.RELEASE`，至少要用 `SR2` 或更高版本。RELEASE 版本可能包含未修复的问题。

## 版本选择策略

### 生产环境选型

选版本要考虑三个因素：

1. **Spring Boot 兼容性**：这是硬约束，不匹配直接启动不了
2. **组件维护状态**：有些组件已经停止维护，选择时要避开
3. **社区活跃度**：优先选择活跃社区支持的版本

推荐的生产环境版本选择：

| Spring Cloud | Spring Boot | 适用场景 |
|---|---|---|
| **2023.0** | 3.2.x | 最新项目，Java 17+ |
| **2022.0** | 3.0.x | 需要 Spring Boot 3.0 |
| **Hoxton.SR12** | 2.4.x | 稳定项目，Java 8/11 |
| **Greenwich.SR6** | 2.1.x | 遗留项目，维护模式 |

### 版本升级检查清单

从低版本升级到高版本时，必须检查：

- [ ] 依赖库的版本是否兼容
- [ ] 注解是否仍然有效（如 `@EnableDiscoveryClient`）
- [ ] 配置文件格式是否有变化
- [ ] API 是否发生了变化

## 组件的演进与停更

Spring Cloud 的组件来源有三类：

### Netflix 组件（部分已停更）

| 组件 | 状态 | 替代方案 |
|---|---|---|
| Eureka | **停止维护**（仍可用） | Nacos |
| Hystrix | **停止维护** | Sentinel |
| Ribbon | **停止维护** | Spring Cloud LoadBalancer |
| Feign | **进入维护模式** | OpenFeign |
| Zuul | **停止维护** | Gateway |

### Alibaba 组件（活跃）

| 组件 | 特点 |
|---|---|
| Nacos | 注册中心 + 配置中心一体化 |
| Sentinel | 流量控制 + 熔断降级 |
| Seata | 分布式事务 |
| Dubbo | RPC 框架 |

### Spring Cloud 原生组件（活跃）

| 组件 | 用途 |
|---|---|
| Gateway | API 网关 |
| LoadBalancer | 客户端负载均衡 |
| CircuitBreaker | 熔断抽象层 |
| Stream | 消息中间件抽象 |
| Bus | 消息总线 |
