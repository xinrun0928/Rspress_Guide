# Spring Cloud 基础

当你第一次接触 Spring Cloud 时，面对一堆版本号（Angel、Brixton、Greenwich、Hoxton）和组件名（Eureka、Feign、Hystrix），可能会感到困惑——这个生态到底是怎么组织的？版本之间有什么区别？

这一模块解决的是最基础但最重要的问题：**Spring Cloud 是什么，它的版本是怎么命名的，和 Spring Boot 是什么关系，市面上有哪些组件可以选择**。

很多人跳过这些「看似简单」的内容，直接去写代码，结果在生产环境遇到版本冲突、依赖不兼容的问题，才后悔没有先搞清楚这些基础概念。

## 为什么版本关系如此重要？

Spring Cloud 并不是一个独立的框架，它是一组规范的集合。每个 Spring Cloud 版本都明确声明了兼容的 Spring Boot 版本。如果版本不匹配，最常见的问题包括：

- 启动时报 `NoSuchMethodError`、`ClassNotFoundException`
- 配置不生效，Bean 创建失败
- 某些特性无法使用（如 Spring Boot 2.6+ 对 Path Pattern 的变化影响了 Spring MVC 路由匹配）

所以在引入 Spring Cloud 依赖之前，**先查清楚版本对应关系**是标准操作，而不是可选项。

## 面试考察分析

Spring Cloud 基础在面试中通常不是单独的问题，而是穿插在其他问题的回答中。面试官想考察的是：

1. **你有没有被版本坑过**——能讲出实际遇到的兼容性问题，说明有踩坑经验
2. **你能不能做技术选型**——面对 Alibaba 和 Netflix 两套生态，你是怎么选择的，为什么
3. **你对生态演进有没有了解**——知不知道 Eureka 已经停止维护，Nacos 是怎么崛起的

这些问题的答案不在官方文档里，在实际项目中。建议在学习和工作中多记录遇到的问题，这些才是面试中最有说服力的素材。
