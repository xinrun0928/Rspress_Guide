# Spring Cloud 与 Spring Boot 版本对应关系

为什么你的 Spring Cloud 项目启动失败，报了一堆 `NoSuchMethodError`？

大概率是因为 Spring Cloud 和 Spring Boot 的版本不匹配。

Spring Cloud 是一个「规范层」，它依赖 Spring Boot 提供基础能力。每个 Spring Cloud 版本都明确声明了它支持的 Spring Boot 版本范围。如果你的依赖里写的是 `Spring Cloud 2020.0` + `Spring Boot 2.2.x`，恭喜你，你会有一堆奇怪的报错。

## 版本对应表（重点记忆）

以下对应关系是面试和实际开发中的必考内容，建议重点记忆：

### 年份版本系列（2020+）

| Spring Cloud | Spring Boot | Java 要求 |
|---|---|---|
| **2023.0** | 3.2.x | Java 17+ |
| **2022.0** | 3.0.x | Java 17+ |
| **2021.0** | 2.6.x | Java 8+ |
| **2020.0** | 2.4.x / 2.5.x | Java 8+ |

### 伦敦地铁系列

| Spring Cloud | Spring Boot | Java 要求 |
|---|---|---|
| **Hoxton** | 2.2.x / 2.3.x | Java 8+ |
| **Greenwich** | 2.1.x | Java 8+ |
| **Finchley** | 2.0.x | Java 8+ |
| **Edgware** | 1.5.x | Java 7/8 |
| **Dalston** | 1.5.x | Java 7/8 |

> **注意**：Edgware 和 Dalston 只支持 Spring Boot 1.5.x，这是 legacy 版本，现在新项目不应该再使用了。

## 版本约束的本质

为什么 Spring Cloud 和 Spring Boot 的版本必须严格对应？

### 依赖传递的连锁反应

当你引入 `spring-cloud-starter-alibaba-nacos-discovery` 时，Maven 会下载一系列依赖：

```
nacos-discovery
  ├── spring-cloud-starter-netflix-ribbon (假设有)
  │     └── ribbon-core 2.x
  └── spring-cloud-context
        └── spring-boot 2.x
```

如果 Spring Boot 版本和预期不一致，某些库的版本可能不兼容，导致 `NoSuchMethodError`。

### Spring Cloud BOM 的作用

Spring Cloud 官方提供了 `spring-cloud-dependencies` BOM（Bill of Materials），用来**统一管理所有依赖版本**：

```xml
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>org.springframework.cloud</groupId>
            <artifactId>spring-cloud-dependencies</artifactId>
            <version>2023.0.0</version>
            <type>pom</type>
            <scope>import</scope>
        </dependency>
    </dependencies>
</dependencyManagement>
```

**正确做法**：在 `dependencyManagement` 中声明 Spring Cloud BOM，让它帮你统一版本。你只需要声明 `spring-cloud-starter` 的版本，其他依赖的版本会自动对齐。

## 常见版本冲突场景

### 场景一：Spring Boot 版本高于 Spring Cloud 支持范围

```xml
<!-- 错误配置：Spring Cloud 2020.0 不支持 Spring Boot 2.6+ -->
<spring-cloud.version>2020.0.0</spring-cloud.version>
<spring-boot.version>2.6.0</spring-boot.version>
```

报错：

```
NoSuchMethodError: org.springframework.boot.context.event.SpringApplicationRunListener
```

### 场景二：自定义依赖版本覆盖了 BOM

```xml
<dependencies>
    <dependency>
        <groupId>org.springframework.cloud</groupId>
        <artifactId>spring-cloud-starter-openfeign</artifactId>
    </dependency>
    <!-- 错误：手动指定了不兼容的版本 -->
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-web</artifactId>
        <version>2.0.0</version>
    </dependency>
</dependencies>
```

### 场景三：Spring Boot 3.0 的变化

Spring Boot 3.0 是一个重大里程碑，它带来了：

- **最低 Java 版本要求提升到 17**
- **javax 包迁移到 jakarta**：如 `javax.servlet` 变成 `jakarta.servlet`
- **Spring 6 和 Hibernate 6**：API 有较大变化

如果你的项目用 Spring Cloud 2022.0（支持 Spring Boot 3.0），而代码里还在用 `javax.servlet`，会直接编译失败。

## 版本选择建议

### 新项目（2024+）

直接用最新组合：

```xml
<spring-cloud.version>2023.0.0</spring-cloud.version>
<spring-boot.version>3.2.0</spring-boot.version>
```

要求：JDK 17+

### 过渡期项目（需要兼容 Java 8）

用 Hoxton 或 Greenwich 系列：

```xml
<spring-cloud.version>Hoxton.SR12</spring-cloud.version>
<spring-boot.version>2.4.13</spring-boot.version>
```

### 遗留项目（Spring Boot 1.5）

尽快升级。Spring Boot 1.5 已经在 2019 年停止维护，继续使用会有安全风险。

## 面试高频问题

### Q1：Spring Cloud 和 Spring Boot 是什么关系？

**参考答案**：Spring Boot 是快速构建单个 Spring 应用的脚手架，专注于应用本身的开发和配置。Spring Cloud 是构建分布式系统的全家桶，它定义了一套分布式系统的标准接口（服务注册、负载均衡、熔断等），而 Spring Boot 应用通过集成 Spring Cloud 的组件来实现这些接口。简单说，Spring Cloud 是规范，Spring Boot 是实现基础。

### Q2：为什么 Spring Cloud 的版本和 Spring Boot 必须对应？

**参考答案**：Spring Cloud 的组件（如 Gateway、Feign）底层依赖 Spring MVC、Spring Boot 的自动配置等能力。这些能力在不同 Spring Boot 版本中的 API 可能有变化。如果版本不匹配，会出现方法找不到、配置不生效等问题。Spring Cloud BOM 通过统一版本管理来解决这个问题。

### Q3：Spring Boot 2.4 之后，bootstrap.yml 还有效吗？

**参考答案**：从 Spring Boot 2.4 开始，bootstrap.yml 的加载机制有变化。`spring.cloud.bootstrap.enabled` 不再默认开启。如果使用 Spring Cloud Config，需要额外引入 `spring-cloud-starter-bootstrap` 依赖，或者考虑使用 Nacos 等配置中心来替代。

## 总结

版本对应关系的核心要点：

- **Spring Cloud 2020.0+ 对应 Spring Boot 2.4+**
- **Spring Cloud 2022.0+ 对应 Spring Boot 3.0+，需要 Java 17+**
- **使用 BOM 统一版本管理，不要手动指定冲突的版本**
- **Legacy 版本（Edgware/Dalston）尽快升级**

记住：版本问题是 80% 的 Spring Cloud 启动问题的根源。在抱怨组件不好用之前，先检查一下版本是否匹配。
