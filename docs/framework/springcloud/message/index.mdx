# 消息与异步：Spring Cloud 的消息驱动

import { Badge } from '@rspress/core/theme';

微服务之间，有时候不需要同步等待结果。

当用户下单后，需要发送短信通知、发送邮件通知、更新统计数据——这些事情不需要等待完成，可以异步处理。消息队列，就是实现异步通信的最佳方式。

## 一、消息队列的核心价值

### 1.1 同步 vs 异步

**同步调用**：
```java
// 下单后同步发送短信（阻塞等待）
orderService.createOrder(order);
smsService.sendNotify(order.getUserId());  // 等待短信发送完成
return "下单成功";
```

问题：
- 短信发送可能很慢，影响用户体验
- 如果短信服务挂了，下单也会失败
- 短信服务压力会影响核心业务

**异步消息**：
```java
// 下单后发送消息到队列，立即返回
orderService.createOrder(order);
messageQueue.send("order-created", order);  // 立即返回
return "下单成功";

// 短信服务异步消费消息
@RabbitListener
public void handleOrderCreated(Order order) {
    smsService.sendNotify(order.getUserId());
}
```

优势：
- 下单立即返回，用户体验好
- 短信服务挂了不影响下单
- 可以削峰填谷

### 1.2 消息队列的应用场景

| 场景 | 说明 |
| --- | --- |
| 异步处理 | 下单后发送通知、记录日志等 |
| 流量削峰 | 突发流量时，先写入队列，慢慢消费 |
| 系统解耦 | 微服务间通过消息通信，降低耦合 |
| 事件驱动 | 基于事件的业务流程 |

## 二、Spring Cloud Stream

Spring Cloud Stream 提供了统一的消息编程模型，屏蔽了底层 MQ 的差异。

### 2.1 核心概念

- **Binder**：MQ 的抽象层，支持 Kafka、RabbitMQ 等
- **Channel**：消息通道，对应 MQ 的 Topic/Queue
- **Binding**：通道与 MQ 的绑定关系

### 2.2 使用方式

```java
@EnableBinding(Processor.class)
public class StreamProcessor {
    
    @StreamListener(Processor.INPUT)
    public void handle(Object payload) {
        // 处理消息
    }
    
    @SendTo(Processor.OUTPUT)
    public Object process(Object payload) {
        return payload;
    }
}
```

## 三、Spring Cloud Bus

Spring Cloud Bus 用于在集群内广播事件，最典型的应用是配合配置中心实现配置的批量刷新。

### 3.1 配置刷新

当配置中心的配置发生变化时，通过 Bus 广播到所有实例：

```bash
curl -X POST http://localhost:8080/actuator/bus-refresh
```

### 3.2 应用场景

- 配置变更的批量刷新
- 环境变量的批量更新
- 集群内的任意事件广播

## 五、总结

消息与异步是微服务架构的重要组成部分：

- **Stream**：统一的消息编程模型，屏蔽底层 MQ 差异
- **Bus**：集群内的事件广播，用于配置刷新等场景

掌握这些组件，可以让微服务架构更加灵活和可靠。
