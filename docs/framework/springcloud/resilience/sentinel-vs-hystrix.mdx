# Sentinel vs Hystrix：熔断器的两代方案

Hystrix 停更了，Sentinel 为什么成了新宠？

2018 年，Netflix 宣布 Hystrix 进入维护模式，不再开发新功能。这个曾经的熔断器"老大哥"逐渐退出历史舞台，取而代之的是阿里巴巴开源的 Sentinel。

理解这两代方案的差异，是面试中经常被问到的话题，也是理解服务容错演进历程的关键。

---

## 一、Hystrix 停更背景

### 1.1 Hystrix 的辉煌历史

Hystrix 是 Netflix 在 2012 年开源的熔断器，一经推出就成为微服务架构的标配：

```
2012: Hystrix 开源
2013: Spring Cloud Netflix 集成 Hystrix
2014-2017: Hystrix 成为微服务容错的事实标准
2018: Netflix 宣布 Hystrix 进入维护模式
```

### 1.2 为什么停更

Hystrix 的设计在当时的背景下是先进的，但随着微服务架构的演进，它的问题逐渐暴露：

| 问题 | 说明 |
| --- | --- |
| 线程池隔离开销大 | 每个依赖维护一个线程池，资源消耗高 |
| 配置复杂 | Hystrix 的配置项非常多，学习成本高 |
| 无法自适应 | 规则需要手动配置，不能根据系统负载自动调整 |
| 无流量控制 | Hystrix 只关注熔断，不关注流量控制 |

### 1.3 生态迁移

Netflix 停更 Hystrix 后，社区形成了几个方向：

| 方向 | 项目 | 说明 |
| --- | --- | --- |
| 继续维护 | OpenFeign+Hystrix | OpenFeign 仍在集成 Hystrix |
| 完全重写 | Resilience4j | 轻量级熔断器，兼容 Hystrix 注解 |
| 功能扩展 | Sentinel | 流量控制+熔断降级+系统自适应 |

## 二、设计理念对比

### 2.1 信号量 vs 线程池隔离

**Hystrix 的线程池隔离**：

```java
// Hystrix 的隔离方式
@HystrixCommand(
    commandProperties = {
        @HystrixProperty(name = "execution.isolation.strategy", value = "THREAD")
    },
    threadPoolProperties = {
        @HystrixProperty(name = "coreSize", value = "10")
    }
)
public Order getOrder(Long id) {
    return orderRepository.findById(id);
}
```

**Sentinel 的信号量隔离**：

```java
// Sentinel 的隔离方式（基于并发线程数）
@SentinelResource(value = "getOrder")
public Order getOrder(Long id) {
    return orderRepository.findById(id);
}
```

| 维度 | Hystrix 线程池隔离 | Sentinel 信号量隔离 |
| --- | --- | --- |
| 资源消耗 | 高（每个依赖独立线程池） | 低（基于计数器） |
| 隔离效果 | 强（完全隔离） | 中（统计隔离） |
| 支持异步 | 是 | 否 |
| 超时控制 | 是 | 否 |

### 2.2 设计理念差异

| 维度 | Hystrix | Sentinel |
| --- | --- | --- |
| 核心问题 | 如何在故障时快速失败 | 如何控制流量 + 如何在故障时快速失败 |
| 隔离方式 | 线程池隔离（强隔离） | 信号量隔离（轻量） |
| 规则类型 | 熔断规则 | 流量控制 + 熔断规则 + 系统规则 |
| 自适应能力 | 无 | 系统自适应 |

## 三、熔断策略对比

### 3.1 Hystrix 的熔断策略

Hystrix 使用"半开"机制自动恢复：

```java
// Hystrix 熔断配置
@HystrixCommand(
    commandProperties = {
        // 熔断器跳闸的阈值：10 秒内 50% 请求失败
        @HystrixProperty(name = "circuitBreaker.requestVolumeThreshold", value = "20"),
        @HystrixProperty(name = "circuitBreaker.sleepWindowInMilliseconds", value = "5000"),
        @HystrixProperty(name = "circuitBreaker.errorThresholdPercentage", value = "50")
    }
)
public Order getOrder(Long id) {
    return orderRepository.findById(id);
}
```

### 3.2 Sentinel 的熔断策略

Sentinel 支持三种熔断策略：

```java
// 慢调用比例熔断
DegradeRule rule = new DegradeRule("getOrder")
    .setGrade(CircuitBreakerStrategy.SLOW_REQUEST_RATIO)
    .setCount(1000)              // 阈值：1000ms
    .setSlowRatioThreshold(0.5)  // 50% 慢调用比例
    .setMinRequestAmount(5);     // 最小请求数

// 异常比例熔断
DegradeRule rule = new DegradeRule("getOrder")
    .setGrade(CircuitBreakerStrategy.ERROR_RATIO)
    .setCount(0.5)               // 50% 异常比例
    .setMinRequestAmount(5);

// 异常数熔断
DegradeRule rule = new DegradeRule("getOrder")
    .setGrade(CircuitBreakerStrategy.ERROR_COUNT)
    .setCount(10);               // 10 个异常
```

### 3.3 对比

| 维度 | Hystrix | Sentinel |
| --- | --- | --- |
| 熔断触发条件 | 失败率 + 最小请求数 | 慢调用比例 / 异常比例 / 异常数 |
| 恢复机制 | 半开试探 | 半开试探 |
| 半开状态 | 固定请求数 | 可配置请求数 |
| 熔断粒度 | Command | Resource |

## 四、资源模型对比

### 4.1 Hystrix Command 模型

Hystrix 使用 Command 模式：

```java
public class GetOrderCommand extends HystrixCommand<Order> {
    
    private Long orderId;
    
    public GetOrderCommand(Long orderId) {
        super(Settera
            .withGroupKey(HystrixCommandGroupKey.Factory.asKey("OrderGroup"))
            .andCommandKey(HystrixCommandKey.Factory.asKey("GetOrder"))
            .andThreadPoolKey(HystrixThreadPoolKey.Factory.asKey("OrderPool"))
        );
        this.orderId = orderId;
    }
    
    @Override
    protected Order run() throws Exception {
        return orderRepository.findById(orderId);
    }
    
    @Override
    protected Order getFallback() {
        return Order.fallback();
    }
}

// 使用
Order order = new GetOrderCommand(id).execute();
```

### 4.2 Sentinel Resource 模型

Sentinel 使用 Resource + Slot 链模式：

```java
@SentinelResource(
    value = "getOrder",
    blockHandler = "getOrderBlock",
    fallback = "getOrderFallback"
)
public Order getOrder(Long id) {
    return orderRepository.findById(id);
}
```

### 4.3 模型对比

| 维度 | Hystrix Command | Sentinel Resource |
| --- | --- | --- |
| 编程模型 | Command 模式 | 注解 + AOP |
| 资源定义 | 显式创建 Command | 注解标注方法 |
| 降级方式 | 重写 getFallback() | blockHandler + fallback |
| 配置方式 | 代码中配置 | 注解属性或 Dashboard |

## 五、Dashboard 对比

### 5.1 Hystrix Dashboard

Hystrix Dashboard 监控单个服务：

```yaml
hystrix:
  dashboard:
    proxy-stream-allow-list: "*"
```

需要 Turbine 聚合多个服务的指标：

```yaml
turbine:
  app-config: order-service,user-service
  aggregator:
    cluster-config:
      - DEFAULT
```

### 5.2 Sentinel Dashboard

Sentinel Dashboard 提供更完善的可视化：

- 实时监控
- 流控规则配置
- 熔断规则配置
- 系统规则配置
- 机器列表

```
Dashboard 功能对比：
├─ 实时监控
│   ├─ Hystrix: 支持
│   └─ Sentinel: 支持
├─ 规则配置
│   ├─ Hystrix: 不支持，需要通过代码或 Apollo 配置
│   └─ Sentinel: 支持，Dashboard 直接配置
└─ 系统自适应
    ├─ Hystrix: 不支持
    └─ Sentinel: 支持
```

## 六、迁移方案

### 6.1 Hystrix → Resilience4j

Resilience4j 是 Hystrix 的轻量级替代：

```xml
<!-- 移除 Hystrix -->
<!-- <dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-netflix-hystrix</artifactId>
</dependency> -->

<!-- 添加 Resilience4j -->
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-circuitbreaker-resilience4j</artifactId>
</dependency>
```

**注解迁移**：

```java
// Hystrix
@HystrixCommand(
    fallbackMethod = "getOrderFallback",
    commandProperties = {
        @HystrixProperty(name = "execution.isolation.thread.timeoutInMilliseconds", value = "2000")
    }
)
public Order getOrder(Long id) {
    return orderRepository.findById(id);
}

// Resilience4j
@CircuitBreaker(name = "backendA", fallbackMethod = "getOrderFallback")
@TimeLimiter(name = "backendA")
public Order getOrder(Long id) {
    return orderRepository.findById(id);
}
```

### 6.2 Hystrix → Sentinel

迁移到 Sentinel：

```xml
<dependency>
    <groupId>com.alibaba.csp</groupId>
    <artifactId>sentinel-spring</artifactId>
</dependency>
```

**注解迁移**：

```java
// Hystrix
@HystrixCommand(
    fallbackMethod = "getOrderFallback"
)
public Order getOrder(Long id) {
    return orderRepository.findById(id);
}

// Sentinel
@SentinelResource(
    value = "getOrder",
    fallback = "getOrderFallback"
)
public Order getOrder(Long id) {
    return orderRepository.findById(id);
}
```

## 七、Sentinel 在 Spring Cloud Alibaba 中的生态

### 7.1 完整生态

Sentinel 不只是熔断器，它是完整的流量控制平台：

```
Sentinel 生态
├── 流量控制（Flow Control）
├── 熔断降级（Degrade）
├── 系统自适应（System Adaptive）
├── 热点参数限流（Param Flow）
├── 授权控制（Authority）
├── 实时监控（Metrics）
└── 规则管理（Rule Management）
```

### 7.2 与 Spring Cloud 的集成

Spring Cloud Alibaba 提供了 Sentinel 的自动配置：

```xml
<dependency>
    <groupId>com.alibaba.cloud</groupId>
    <artifactId>spring-cloud-starter-alibaba-sentinel</artifactId>
</dependency>
```

自动集成：
- Sentinel Starter 自动配置
- `@SentinelResource` 注解生效
- RestTemplate 和 Feign 集成
- Sentinel Dashboard 接入

## 八、面试追问方向

- **为什么 Hystrix 停止维护了？** Hystrix 的线程池隔离开销大，配置复杂，且缺少流量控制等现代微服务需要的能力。
- **Hystrix 和 Sentinel 的核心区别是什么？** Sentinel 不仅支持熔断，还支持流量控制、系统自适应、热点参数限流等高级功能。
- **如何从 Hystrix 迁移到 Sentinel？** 移除 Hystrix 依赖，添加 Sentinel 依赖，将 `@HystrixCommand` 改为 `@SentinelResource`。
- **Resilience4j 和 Sentinel 的区别是什么？** Resilience4j 是轻量级熔断器，只关注熔断；Sentinel 是完整的流量控制平台，功能更全面。

---

## 九、总结

Sentinel 相比 Hystrix，是一个更现代、更全面的服务保护方案：

- **更轻量**：信号量隔离比线程池隔离资源消耗更低
- **更全面**：流量控制 + 熔断降级 + 系统自适应
- **更易用**：注解驱动，Dashboard 可视化配置
- **更自研**：Sentinel 是阿里巴巴多年生产实践的结晶

理解这两代方案的差异，不仅是为了面试，更是为了理解微服务容错技术的演进历程。
