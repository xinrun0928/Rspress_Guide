# @SentinelResource：资源的灵魂注解

import { Badge } from '@rspress/core/theme';

Sentinel 的资源是怎么被统计的？

当你用 `@SentinelResource` 标注一个方法时，Sentinel 会自动统计这个方法的调用次数、响应时间、异常次数等指标。这些统计结果决定了什么时候触发限流，什么时候触发熔断。

但 `@SentinelResource` 远不止"统计"这么简单——它还定义了资源的名称、限流后的处理、异常后的降级、以及哪些异常需要忽略。这个注解，是 Sentinel 使用的核心。

## 一、@SentinelResource 的四个核心参数

### 1.1 参数概览

`@SentinelResource` 有四个核心参数：

```java
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface SentinelResource {
    // 1. 资源名称（必填）
    String value();
    
    // 2. blockHandler：限流/熔断处理方法
    Class<?>[] blockHandler() default {};
    String[] blockHandlerClass() default {};
    
    // 3. fallback：业务异常处理方法
    String fallback() default "";
    Class<?>[] fallbackClass() default {};
    
    // 4. exceptionsToIgnore：忽略的异常
    Class<? extends Throwable>[] exceptionsToIgnore() default {};
}
```

### 1.2 value：资源名称

资源名称是 Sentinel 中最重要的概念——它决定了统计数据归属于哪个资源。

```java
@SentinelResource(value = "getOrder")
public Order getOrder(Long id) {
    return orderRepository.findById(id);
}
```

> [!WARNING]
> 资源名称必须唯一。如果两个方法使用相同的资源名，它们的统计数据会合并，导致限流/熔断逻辑混乱。

### 1.3 blockHandler 和 blockHandlerClass：限流处理

blockHandler 处理限流和熔断：

```java
@SentinelResource(value = "getOrder", blockHandler = "getOrderBlock")
public Order getOrder(Long id) {
    return orderRepository.findById(id);
}

// 必须是 static 方法
public static Order getOrderBlock(Long id, BlockException e) {
    log.warn("请求被限流: {}", e.getMessage());
    return Order.fallback();
}
```

blockHandlerClass 允许在单独类中定义处理方法：

```java
@SentinelResource(
    value = "getOrder", 
    blockHandler = "handleBlock",
    blockHandlerClass = SentinelBlockHandler.class
)
public Order getOrder(Long id) {
    return orderRepository.findById(id);
}
```

```java title="SentinelBlockHandler.java"
public class SentinelBlockHandler {
    public static Order handleBlock(Long id, BlockException e, BlockHandler handler) {
        return Order.fallback();
    }
}
```

### 1.4 fallback 和 fallbackClass：异常处理

fallback 处理业务异常：

```java
@SentinelResource(
    value = "getOrder", 
    fallback = "getOrderFallback"
)
public Order getOrder(Long id) {
    return orderRepository.findById(id);
}

public Order getOrderFallback(Long id, Throwable e) {
    log.error("业务异常: {}", e.getMessage());
    return Order.fallback();
}
```

### 1.5 exceptionsToIgnore：忽略异常

对于某些异常，应该直接抛出而不触发降级：

```java
@SentinelResource(
    value = "getOrder", 
    exceptionsToIgnore = {IllegalArgumentException.class}
)
public Order getOrder(Long id) {
    if (id < 0) {
        throw new IllegalArgumentException("订单ID不能为负数");
    }
    return orderRepository.findById(id);
}
```

## 二、blockHandler 和 fallback 的区别

### 2.1 触发条件对比

| 参数 | 触发条件 | 场景 |
| --- | --- | --- |
| blockHandler | 触发限流或熔断时调用 | 请求被 Sentinel 拒绝了 |
| fallback | 业务方法抛出异常时调用 | 业务代码发生了异常 |

### 2.2 执行流程

```java
@SentinelResource(
    value = "getOrder",
    blockHandler = "getOrderBlock",
    fallback = "getOrderFallback"
)
public Order getOrder(Long id) {
    // 业务逻辑
    return orderRepository.findById(id);
}

public static Order getOrderBlock(Long id, BlockException e) {
    // 被限流或熔断
    return Order.fallback();
}

public static Order getOrderFallback(Long id, Throwable e) {
    // 业务异常
    return Order.fallback();
}
```

**执行顺序**：

```
请求进入
    ↓
Sentinel 检查限流/熔断
    ↓
  ├─ 触发限流/熔断 → blockHandler → 返回
  │
  └─ 未触发 → 执行 getOrder()
                ↓
              抛出异常 → fallback → 返回
                ↓
              无异常 → 正常返回
```

## 三、fallbackClass：静态方法降级

### 3.1 为什么需要 fallbackClass

如果所有降级方法都写在业务类中，代码会变得臃肿。可以将降级方法抽取到单独的类中：

```java
// 订单服务的降级处理类
public class OrderFallback {
    
    public static Order getOrderFallback(Long id, Throwable e) {
        log.error("订单服务异常: id={}, error={}", id, e.getMessage());
        return Order.builder()
            .id(id)
            .status(OrderStatus.UNKNOWN)
            .build();
    }
    
    public static List<Order> listOrdersFallback(Throwable e) {
        log.error("订单列表查询异常: {}", e.getMessage());
        return Collections.emptyList();
    }
}
```

### 3.2 配置使用

```java
public class OrderService {
    
    @SentinelResource(
        value = "getOrder",
        fallback = "getOrderFallback",
        fallbackClass = OrderFallback.class
    )
    public Order getOrder(Long id) {
        return orderRepository.findById(id);
    }
    
    @SentinelResource(
        value = "listOrders",
        fallback = "listOrdersFallback",
        fallbackClass = OrderFallback.class
    )
    public List<Order> listOrders() {
        return orderRepository.findAll();
    }
}
```

## 四、exceptionsToIgnore：忽略指定异常

### 4.1 使用场景

有些异常不应该触发降级，比如：

- 业务校验异常（输入参数不合法）
- 认证异常（Token 过期）
- 权限异常（没有权限访问）

这些异常应该直接向上抛出，交由调用方处理。

### 4.2 配置示例

```java
@SentinelResource(
    value = "getOrder",
    exceptionsToIgnore = {
        IllegalArgumentException.class,    // 参数异常
        UnauthorizedException.class,      // 未授权
        AccessDeniedException.class       // 拒绝访问
    }
)
public Order getOrder(Long id, String userId) {
    // 业务逻辑
    if (id == null) {
        throw new IllegalArgumentException("订单ID不能为空");
    }
    if (!hasPermission(userId, id)) {
        throw new AccessDeniedException("没有访问权限");
    }
    return orderRepository.findById(id);
}
```

### 4.3 注意事项

> [!WARNING]
> `exceptionsToIgnore` 中的异常即使被忽略，它们的统计仍然会被记录到 Sentinel 中，但不会触发降级。如果需要完全不统计，需要在规则中配置。

## 五、热点参数限流（ParamFlowRule）

### 5.1 什么是热点参数限流

热点参数限流是 Sentinel 的一种高级功能——不仅限制资源的整体 QPS，还限制特定参数值的 QPS。

### 5.2 使用示例

```java
@SentinelResource(value = "getOrder")
public Order getOrder(
    @ParamFlowKey Long orderId,  // 标注为热点参数
    String userId
) {
    return orderRepository.findById(orderId);
}
```

### 5.3 配置热点规则

```java
ParamFlowRule rule = new ParamFlowRule("getOrder")
    .setParamIdx(0)  // 第一个参数（orderId）
    .setCount(10)    // 每个 orderId 最多 10 QPS
    .setDurationInSec(1);

ParamFlowRuleManager.loadRules(Collections.singletonList(rule));
```

### 5.4 参数值维度限流

热点规则还支持按参数值限流：

```java
ParamFlowRule rule = new ParamFlowRule("getOrder")
    .setParamIdx(0)
    .setCount(10)
    // 对特定参数值单独限流
    .setParamFlowItemList(Arrays.asList(
        new ParamFlowItem().setObject("VIP_ORDER").setCount(100)
    ));
```

## 六、系统规则（SystemRule）

### 6.1 系统规则简介

Sentinel 支持从系统维度进行限流：

| 规则 | 说明 | 阈值类型 |
| --- | --- | --- |
| `highestCpuUsage` | CPU 使用率 | 0~1 |
| `highestSystemLoad` | 系统负载 | 正数 |
| `avgRt` | 平均响应时间 | 毫秒 |
| `maxThread` | 最大并发线程数 | 正数 |
| `qps` | 所有入口 QPS | 正数 |

### 6.2 配置示例

```java
SystemRule rule = new SystemRule()
    .setAvgRt(1000)        // 平均响应时间超过 1 秒
    .setMaxThread(100)     // 最大并发数超过 100
    .setQps(1000)          // QPS 超过 1000
    .setHighestSystemLoad(10.0);  // 系统负载超过 10

SystemRuleManager.loadRules(Collections.singletonList(rule));
```

## 七、授权规则（AuthorityRule）：黑名单/白名单

### 7.1 规则说明

Sentinel 支持黑白名单控制：

| 类型 | 说明 |
| --- | --- |
| 白名单 | 只有在名单中的调用方才允许访问 |
| 黑名单 | 在名单中的调用方不允许访问 |

### 7.2 配置示例

```java
AuthorityRule rule = new AuthorityRule("getOrder")
    .setStrategy(RuleConstant.AUTHORITY_WHITE)  // 白名单模式
    .setLimitApp("app1,app2");  // 允许 app1 和 app2 访问

AuthorityRuleManager.loadRules(Collections.singletonList(rule));
```

### 7.3 来源标识

调用方的标识通过 `ContextUtil.enter()` 方法设置：

```java
// 在网关或入口处设置来源
public void gatewayFilter() {
    String clientId = getClientId();
    ContextUtil.enter("order-api", clientId);
    try {
        // 调用服务
    } finally {
        ContextUtil.exit();
    }
}
```

## 八、面试追问方向

- **blockHandler 和 fallback 的区别是什么？** blockHandler 处理限流/熔断，fallback 处理业务异常。
- **fallbackClass 的作用是什么？** 将降级方法抽取到单独的类中，保持业务代码整洁。
- **exceptionsToIgnore 的使用场景是什么？** 用于忽略不需要降级的异常，如参数校验异常、认证异常。
- **热点参数限流是什么？** 不仅限制资源的整体 QPS，还限制特定参数值的 QPS。

---

## 九、总结

`@SentinelResource` 是 Sentinel 的核心注解：

- **value**：定义资源名称
- **blockHandler**：处理限流/熔断
- **fallback**：处理业务异常
- **exceptionsToIgnore**：忽略指定异常
- **热点参数限流**：基于参数值的精细化限流
- **系统规则**：从系统维度限流
- **授权规则**：黑白名单控制

掌握这些特性，才能用好 Sentinel 的高级功能。
