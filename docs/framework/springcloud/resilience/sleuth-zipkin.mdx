# Sleuth 链路追踪：从 TraceId 还原一次请求的全貌

import { Badge } from '@rspress/core/theme';

微服务之间的调用链路越来越长，一个请求失败，怎么快速定位？

A 服务 → B 服务 → C 服务 → D 服务

如果 D 服务报错了，你怎么知道是哪个环节出了问题？靠日志吗？A 的日志在 A 服务器，B 的日志在 B 服务器，C 的日志在 C 服务器...

**链路追踪，就是为了解决这个问题。**

## 一、TraceId 和 SpanId 的概念

### 1.1 什么是链路追踪

链路追踪的核心是**分布式追踪**技术。它给每次请求分配一个唯一的 ID，这个 ID 会随着请求在整个调用链中传递，记录每个服务的处理时间，最终形成一个完整的调用链。

### 1.2 核心概念

| 概念 | 说明 | 类比 |
| --- | --- | --- |
| **Trace** | 一次完整的请求 | 一条河流 |
| **Span** | 一次服务调用 | 河流中的一段 |
| **TraceId** | .trace 的唯一标识 | 河流的 ID |
| **SpanId** | 当前 Span 的标识 | 河段的编号 |
| **ParentId** | 父 Span 的 ID | 上游河段的编号 |

### 1.3 形象理解

```
A 服务（TraceId=abc123）
  │
  ├── A 服务处理 Span (SpanId=1)
  │
  └── B 服务调用
        │
        ├── B 服务处理 Span (SpanId=2, ParentId=1)
        │
        └── C 服务调用
              │
              ├── C 服务处理 Span (SpanId=3, ParentId=2)
              │
              └── D 服务调用
                    │
                    └── D 服务处理 Span (SpanId=4, ParentId=3)

调用树：
abc123
└── [1] A 服务
    └── [2] B 服务 (parent:1)
        └── [3] C 服务 (parent:2)
            └── [4] D 服务 (parent:3)
```

### 1.4 追踪信息的结构

```java
// Span 的核心信息
public class Span {
    private String traceId;     // Trace 唯一标识
    private String spanId;      // 当前 Span 标识
    private String parentId;    // 父 Span 标识
    private String name;        // Span 名称（如 "order-service/getOrder"）
    private long timestamp;     // 开始时间戳
    private long duration;      // 持续时间（毫秒）
    private List<Tag> tags;     // 标签（key-value）
    private List<Log> logs;     // 日志（异常信息等）
}
```

## 二、Brave（OpenZipkin Brave）的埋点机制

### 2.1 Brave 简介

Sleuth 内部使用 Brave（OpenZipkin Brave）进行埋点。Brave 是 Zipkin 项目的一部分，提供了低开销的追踪库。

### 2.2 自动埋点

引入 Sleuth 后，以下组件会自动埋点：

| 组件 | 自动埋点 | 说明 |
| --- | --- | --- |
| `RestTemplate` | 是 | HTTP 调用自动追踪 |
| `WebClient` | 是 | 响应式 HTTP 调用自动追踪 |
| `Feign` | 是 | OpenFeign 调用自动追踪 |
| `Zuul` | 是 | Zuul 网关自动追踪 |
| `RabbitMQ` | 是 | 消息发送/接收自动追踪 |
| `Kafka` | 是 | Kafka 生产/消费自动追踪 |

### 2.3 手动埋点

如果需要追踪特定的代码段，可以手动埋点：

```java
@Autowired
private Tracer tracer;

public void someMethod() {
    // 创建新的 Span
    Span span = tracer.nextSpan().name("customOperation");
    
    try (Tracer.SpanInScope scope = tracer.withSpanInScope(span)) {
        // 执行业务逻辑
        doSomething();
        
        // 添加标签
        span.tag("userId", "123");
        span.tag("orderId", "456");
    } catch (Exception e) {
        // 记录异常
        span.error(e);
    } finally {
        // 结束 Span
        span.end();
    }
}
```

### 2.4 传递追踪上下文

Sleuth 使用 `Tracer` 在请求间传递追踪上下文：

```java
// 发送方：在请求头中添加追踪信息
Span currentSpan = tracer.currentSpan();
String traceId = currentSpan.context().traceIdString();
String spanId = currentSpan.context().spanIdString();

// 将 traceId、spanId 添加到 HTTP 头、消息头中
httpHeaders.set("X-B3-TraceId", traceId);
httpHeaders.set("X-B3-SpanId", spanId);

// 接收方：从请求头中提取追踪信息
// Sleuth 会自动完成这个过程
```

## 三、Sleuth 的自动埋点

### 3.1 RestTemplate 埋点

使用 `@LoadBalanced RestTemplate` 调用其他服务时，Sleuth 会自动埋点：

```java
@Autowired
private RestTemplate restTemplate;

public Order getOrder(Long id) {
    // Sleuth 自动添加追踪头
    // X-B3-TraceId: abc123
    // X-B3-SpanId: xyz789
    return restTemplate.getForObject(
        "http://order-service/api/orders/" + id,
        Order.class
    );
}
```

### 3.2 OpenFeign 埋点

Sleuth 集成了 OpenFeign，Feign 调用会自动埋点：

```java
@FeignClient(name = "order-service")
public interface OrderClient {
    
    @GetMapping("/api/orders/{id}")
    Order getOrder(@PathVariable("id") Long id);
}

// 使用时自动埋点
@Service
public class OrderService {
    
    @Autowired
    private OrderClient orderClient;
    
    public Order getOrder(Long id) {
        // Sleuth 自动传递追踪上下文
        return orderClient.getOrder(id);
    }
}
```

### 3.3 异步任务埋点

异步任务需要特殊处理，确保 Span 正确传递：

```java
@Configuration
public class AsyncConfig {
    
    @Bean
    public Executor asyncExecutor() {
        // 使用 Sleuth 的 Executor
        return SleuthExecutors.configuration(new SimpleAsyncTaskExecutor())
            .propagate()
            .build();
    }
}

@Service
public class AsyncService {
    
    @Autowired
    private Executor executor;
    
    public void asyncTask() {
        // 异步执行，但 Span 会正确传递
        executor.execute(() -> {
            // 这个任务会被正确追踪
            doSomething();
        });
    }
}
```

## 四、Sleuth + Zipkin 的集成

### 4.1 为什么需要 Zipkin

Sleuth 只负责**埋点**和**数据采集**，不负责**存储**和**展示**。Zipkin 是分布式追踪系统，负责收集、存储和展示追踪数据。

```
Sleuth（埋点） → Zipkin（收集存储） → Zipkin UI（展示）
```

### 4.2 引入依赖

```xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-sleuth</artifactId>
</dependency>
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-zipkin</artifactId>
</dependency>
```

### 4.3 配置 Zipkin

```yaml title="application.yml"
spring:
  zipkin:
    # Zipkin Server 地址
    base-url: http://zipkin-server:9411
    # 是否开启（默认 true）
    enabled: true
    # 采样率：1.0 = 100%，0.1 = 10%
    sampler:
      probability: 1.0
  sleuth:
    # 是否开启追踪（默认 true）
    enabled: true
```

### 4.4 传输方式

Sleuth 支持多种方式将数据发送到 Zipkin：

| 方式 | 说明 | 配置 |
| --- | --- | --- |
| HTTP | 同步 HTTP 发送 | 默认方式 |
| Kafka | 异步消息发送 | 需要 kafka 依赖 |
| RabbitMQ | 异步消息发送 | 需要 rabbitmq 依赖 |

```yaml title="application.yml"
# HTTP 方式（默认）
spring.zipkin.base-url: http://zipkin-server:9411

# Kafka 方式
spring.zipkin.sender.type: kafka
spring.kafka.bootstrap-servers: kafka-server:9092
```

## 六、Sleuth 采样策略

### 6.1 为什么要采样

在生产环境中，请求量非常大，如果每个请求都完整记录，会产生大量数据。采样是减少数据量的手段。

### 6.2 AlwaysSampler：全部采样

```java
// 全部采样（用于开发环境）
@Bean
public Sampler alwaysSampler() {
    return Sampler.ALWAYS_SAMPLE;
}
```

### 6.3 ProbabilitySampler：概率采样

```java
// 概率采样（用于生产环境）
@Bean
public Sampler probabilitySampler() {
    // 采样率：1.0 = 100%，0.1 = 10%
    return ProbabilityBasedSampler.newBuilder()
        .probability(0.1)
        .build();
}
```

### 6.4 RateLimitingSampler：限流采样

```java
// 限流采样（每秒最多 100 个请求）
@Bean
public Sampler rateLimitingSampler() {
    return RateLimitingSampler.create(100);
}
```

### 6.5 基于错误的采样

```java
// 只采样错误请求
@Bean
public Sampler errorSampler() {
    return new Sampler() {
        @Override
        public boolean isSampled(SamplingFlags samplingFlags) {
            // 只采样包含错误的请求
            return true;
        }
    };
}
```

## 八、面试追问方向

- **TraceId 和 SpanId 的区别是什么？** TraceId 是整个请求链路的唯一标识，SpanId 是当前调用的标识。TraceId 在整个链路中不变，SpanId 在每次调用时变化。
- **Sleuth 如何在服务间传递追踪上下文？** 通过 HTTP 头（X-B3-TraceId、X-B3-SpanId）或消息头传递追踪信息。
- **为什么需要采样？** 在高并发场景下，全量采样会产生大量数据，增加存储压力和网络开销。采样可以有效减少数据量。
- **Sleuth 和 Zipkin 的关系是什么？** Sleuth 负责埋点和数据采集，Zipkin 负责数据的收集、存储和展示。

## 九、总结

Sleuth + Zipkin 构成了完整的分布式追踪方案：

- **TraceId**：整个请求链路的唯一标识
- **SpanId**：每次调用的唯一标识
- **Brave**：低开销的追踪埋点库
- **自动埋点**：RestTemplate、Feign、RabbitMQ 等自动追踪
- **Zipkin**：收集、存储、展示追踪数据
- **采样策略**：概率采样、限流采样等

通过链路追踪，可以快速定位微服务架构中的性能瓶颈和故障点，是微服务运维的必备工具。
