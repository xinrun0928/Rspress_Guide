# Sentinel 流量控制：滑动窗口算法的工程实现

Hystrix 停更了，Sentinel 为什么成了新宠？

因为 Sentinel 不仅仅是熔断器，它还提供了**流量控制**这个更强大的能力。在微服务架构中，流量控制比熔断更重要——与其等到系统崩溃后再熔断，不如在流量过大时就限制它。

Sentinel 的核心是**滑动窗口算法**，它让流量统计既精确又高效。

## 一、滑动窗口算法（BucketLeapArray）的实现

### 1.1 为什么需要滑动窗口

传统的计数方式有两种：**固定窗口**和**滑动窗口**。

**固定窗口**：把时间切成固定大小的块，每块独立计数。

```
固定窗口（每 1 秒一个窗口）：

窗口 1 (0-1秒): ████████████ 10 个请求
窗口 2 (1-2秒): ██████████ 8 个请求
窗口 3 (2-3秒): ██████████████████████████████ 30 个请求

问题：如果限制是 20QPS，在 1.9 秒和 2.1 秒之间可能有 30+8=38 个请求！
```

**滑动窗口**：窗口是"滑动"的，任何时刻都能看到最近 N 秒的统计。

```
滑动窗口（窗口大小 1 秒，步进 0.1 秒）：

时间点 1.0: 统计 0.0-1.0 秒的请求数
时间点 1.1: 统计 0.1-1.1 秒的请求数
时间点 1.2: 统计 0.2-1.2 秒的请求数
...

任何时刻，统计的都是最近 1 秒内的请求，精确控制 QPS
```

### 1.2 Sentinel 的 BucketLeapArray

Sentinel 使用 `BucketLeapArray` 实现滑动窗口：

```java
// 简化后的 BucketLeapArray 结构
public class BucketLeapArray extends LeapArray<MetricBucket> {
    
    public BucketLeapArray(int sampleCount, int intervalInMs) {
        super(sampleCount, intervalInMs);
    }
    
    @Override
    public MetricBucket newEmptyBucket(TimeInterval win) {
        return new MetricBucket();
    }
    
    @Override
    protected WindowWrap<MetricBucket> resetWindowTo(WindowWrap<MetricBucket> w, TimeInterval startTime) {
        MetricBucket bucket = new MetricBucket();
        w.setWindowStart(startTime.get());
        w.value().reset();
        return w;
    }
}
```

### 1.3 时间窗口的结构

Sentinel 将时间分成多个"Bucket"（桶）：

```java
// 窗口结构
public class WindowWrap<_metric> {
    // 窗口开始时间
    private long windowStart;
    
    // 窗口数据
    private _metric value;
}

// Bucket 中的统计数据
public class MetricBucket {
    // 每个时间戳对应的计数器
    private LongAdder[] counters;
    
    // 请求总数
    private LongAdder total;
    
    // 异常数
    private LongAdder exception;
    
    // 响应时间之和
    private LongAdder rt;
}
```

### 1.4 窗口的计算逻辑

```java
// 获取当前应该落在哪个窗口
public class LeapArray<T> {
    
    protected final int windowLengthInMs;    // 单个窗口长度（毫秒）
    protected final int sampleCount;         // 窗口数量
    protected final long intervalInMs;       // 总时间跨度
    
    // 计算当前时间应该落在哪个窗口
    public WindowWrap<T> currentWindow() {
        long now = Time.currentTimeMs();
        long windowStart = calculateWindowStart(now);
        
        // 获取或创建窗口
        return data.currentWindow(windowStart);
    }
    
    // 计算窗口开始时间
    protected long calculateWindowStart(long time) {
        // time - time % windowLengthInMs
        return time - time % windowLengthInMs;
    }
}
```

假设配置是 1 秒内最多 100 个请求，Sentinel 会：
- 将 1 秒分成 10 个窗口，每个窗口 100ms
- 记录每个窗口内的请求数
- 计算时，将最近 10 个窗口的请求数相加

## 二、资源名与入口流量

### 2.1 资源的定义

在 Sentinel 中，**资源**是受保护的对象。定义资源有两种方式：

**方式一：@SentinelResource 注解**

```java
public class OrderService {
    
    @SentinelResource(value = "getOrder", blockHandler = "getOrderBlock")
    public Order getOrder(Long id) {
        return orderRepository.findById(id);
    }
    
    // 限流/熔断时的处理方法
    public Order getOrderBlock(Long id, BlockException e) {
        return Order.fallback();
    }
}
```

**方式二：Sentinel API**

```java
public class SentinelDemo {
    
    public void callService() {
        // 定义资源
        Entry entry = null;
        try {
            entry = SphU.entry("getOrder");
            
            // 业务逻辑
            doSomething();
            
        } catch (BlockException e) {
            // 被限流或熔断
            handleBlock(e);
        } finally {
            if (entry != null) {
                entry.exit();
            }
        }
    }
}
```

### 2.2 入口流量的概念

Sentinel 的流量控制分为两种：

| 类型 | 说明 | 控制对象 |
| --- | --- | --- |
| **入口流量** | 从外部进入系统的流量 | 来自外部的请求 |
| **出口流量** | 从系统发出的流量 | 对外部的调用 |

**入口流量控制**是最常用的场景——限制进入系统的请求数量。

## 三、流量控制的 threshold 和 grade

### 3.1 流控模式（grade）

Sentinel 有三种流控模式：

| 模式 | grade 值 | 说明 |
| --- | --- | --- |
| QPS | 1 | 按每秒请求数限流 |
| 并发线程数 | 0 | 按同时处理的线程数限流 |

```java
// QPS 模式：每秒最多 100 个请求
FlowRule rule = new FlowRule("getOrder");
rule.setGrade(RuleConstant.FLOW_GRADE_QPS);
rule.setCount(100);
rule.setControlBehavior(RuleConstant.CONTROL_BEHAVIOR_DEFAULT);

// 并发线程数模式：同时最多 10 个线程
FlowRule rule = new FlowRule("getOrder");
rule.setGrade(RuleConstant.FLOW_GRADE_THREAD);
rule.setCount(10);
```

### 3.2 QPS vs 并发线程数

| 维度 | QPS 模式 | 并发线程数模式 |
| --- | --- | --- |
| 控制方式 | 控制每秒请求数 | 控制同时处理的请求数 |
| 适用场景 | 限制外部请求量 | 限制资源占用 |
| 线程阻塞 | QPS 限制住，但线程可能阻塞 | 直接限制线程数 |
| 响应时间 | 请求进入后，线程可能等待 | 线程直接处理 |

**并发线程数模式**更适合保护慢接口——一个慢接口会占用线程很长时间，如果线程池被占满，系统会变得卡顿。

## 四、流量控制效果

### 4.1 三种流控效果

| 效果 | 说明 | 适用场景 |
| --- | --- | --- |
| **直接拒绝** | 超出阈值的请求直接拒绝 | 保护系统不被冲垮 |
| **冷启动** | 慢慢增加通过的请求数 | 预热场景 |
| **匀速排队** | 请求匀速通过 | 削峰填谷 |

### 4.2 直接拒绝（CONTROL_BEHAVIOR_DEFAULT）

最常用的模式，超出阈值的请求直接拒绝：

```java
rule.setControlBehavior(RuleConstant.CONTROL_BEHAVIOR_DEFAULT);
```

```
QPS 限制：10
请求模式：每秒 20 个请求

结果：
实际通过：10 个请求
被拒绝：10 个请求
返回：429 Too Many Requests
```

### 4.3 冷启动（CONTROL_BEHAVIOR_WARM_UP）

让系统"热身"后再允许大流量进入：

```java
rule.setControlBehavior(RuleConstant.CONTROL_BEHAVIOR_WARM_UP);
rule.setWarmUpPeriodSec(10);  // 预热时间 10 秒
```

```
QPS 限制：100
冷启动时间：10 秒

结果：
第 1 秒：允许 10 个请求
第 2 秒：允许 20 个请求
...
第 10 秒：允许 100 个请求
之后：允许 100 个请求

曲线：
通过的请求数
    100|_______________________________
       |/
       |/
       |/
       |/
       0-------------------------------
       0    5    10   秒
```

### 4.4 匀速排队（CONTROL_BEHAVIOR_RATE_LIMITER）

请求匀速通过，超时则被拒绝：

```java
rule.setControlBehavior(RuleConstant.CONTROL_BEHAVIOR_FLOW_LIMIT);
rule.setMaxQueueingTimeMs(500);  // 最大排队时间
```

```
QPS 限制：10
最大排队时间：500ms

请求 1 (0ms): 通过
请求 2 (100ms): 通过
请求 3 (200ms): 通过
...
请求 11 (1000ms): 排队，等待 100ms
请求 12 (1100ms): 排队失败，被拒绝（排队超时）
```

## 五、熔断降级的三个策略

### 5.1 慢调用比例（SLOW_REQUEST_RATIO）

当平均响应时间超过阈值时触发熔断：

```java
DegradeRule rule = new DegradeRule("getOrder");
rule.setGrade(RuleConstant.DEGRADE_GRADE_RT);
rule.setCount(1000);              // 阈值：1000ms
rule.setSlowRatioThreshold(0.5);  // 50% 慢调用比例
rule.setMinRequestAmount(5);      // 最小请求数
rule.setStatIntervalMs(1000);     // 统计时间窗口：1 秒
```

```
触发条件：
- 在 1 秒内，至少有 5 个请求
- 其中 50% 的请求响应时间超过 1000ms

触发结果：
熔断器打开，后续请求被拒绝
```

### 5.2 异常比例（ERROR_RATIO）

当异常比例超过阈值时触发熔断：

```java
DegradeRule rule = new DegradeRule("getOrder");
rule.setGrade(RuleConstant.DEGRADE_GRADE_EXCEPTION_RATIO);
rule.setCount(0.5);               // 异常比例：50%
rule.setMinRequestAmount(5);      // 最小请求数
```

### 5.3 异常数（ERROR_COUNT）

当异常数量超过阈值时触发熔断：

```java
DegradeRule rule = new DegradeRule("getOrder");
rule.setGrade(RuleConstant.DEGRADE_GRADE_EXCEPTION_COUNT);
rule.setCount(10);  // 10 个异常后熔断
```

## 六、熔断的状态转换

### 6.1 状态机

```
CLOSE → OPEN → HALF_OPEN → CLOSE
  ↑                       │
  └───────────────────────┘
          失败
```

### 6.2 状态转换详解

```java
// 熔断器状态
public enum CircuitBreakerState {
    CLOSED,   // 关闭状态：正常处理请求
    OPEN,     // 打开状态：拒绝所有请求
    HALF_OPEN // 半开状态：尝试处理请求
}

// 状态转换逻辑
public class CircuitBreaker {
    
    public void onSuccess() {
        if (state == HALF_OPEN) {
            // 半开状态下成功，恢复到 CLOSED
            state = CLOSED;
        }
    }
    
    public void onError() {
        if (state == HALF_OPEN) {
            // 半开状态下失败，重新打开
            state = OPEN;
        }
    }
    
    public boolean shouldOpen() {
        if (state == CLOSED) {
            // 检查是否需要打开
            return failureRate > threshold;
        }
        return false;
    }
}
```

## 七、@SentinelResource 的 blockHandler 和 fallback

### 7.1 区别

| 参数 | 触发条件 | 用途 |
| --- | --- | --- |
| `blockHandler` | 触发限流、熔断等规则 | 处理 Sentinel 的流控/熔断 |
| `fallback` | 业务方法抛出异常 | 处理业务异常 |

### 7.2 使用示例

```java
public class OrderService {
    
    @SentinelResource(
        value = "getOrder",
        blockHandler = "getOrderBlock",
        fallback = "getOrderFallback"
    )
    public Order getOrder(Long id) {
        if (id < 0) {
            throw new IllegalArgumentException("无效的订单ID");
        }
        return orderRepository.findById(id);
    }
    
    // blockHandler：处理限流/熔断
    public Order getOrderBlock(Long id, BlockException e) {
        log.warn("请求被限流/熔断: {}", e.getClass().getSimpleName());
        return Order.fallback();
    }
    
    // fallback：处理业务异常
    public Order getOrderFallback(Long id, Throwable e) {
        log.error("业务异常: {}", e.getMessage());
        return Order.fallback();
    }
}
```

## 八、面试追问方向

- **Sentinel 的滑动窗口算法是什么？** 将时间分成多个 Bucket，每个 Bucket 记录一段时间内的统计数据，通过组合多个 Bucket 计算任意时间窗口的统计值。
- **QPS 和并发线程数限流的区别是什么？** QPS 限制每秒请求数，并发线程数限制同时处理的请求数；前者适合控制外部流量，后者适合保护慢接口。
- **Sentinel 的熔断策略有哪三种？** 慢调用比例、异常比例、异常数。
- **blockHandler 和 fallback 的区别是什么？** blockHandler 处理 Sentinel 的流控/熔断，fallback 处理业务异常。

## 九、总结

Sentinel 的滑动窗口算法是流量控制的核心：

- **BucketLeapArray**：将时间分成多个 Bucket，记录每个窗口的统计数据
- **流量控制**：通过 QPS 或并发线程数限制流量
- **三种流控效果**：直接拒绝、冷启动、匀速排队
- **三种熔断策略**：慢调用比例、异常比例、异常数
- **@SentinelResource**：定义资源和处理方法

理解滑动窗口算法，才能理解 Sentinel 为什么能做到精确的流量控制。
