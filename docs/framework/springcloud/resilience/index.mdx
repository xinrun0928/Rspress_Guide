# 服务容错与治理：微服务的保护伞

import { Badge } from '@rspress/core/theme';

微服务架构的最大敌人，是级联故障。

一个服务出问题，如果不做任何处理，这个故障会像滚雪球一样越滚越大，最终导致整个系统崩溃。服务容错与治理，就是给这个雪球装上"刹车"——在故障发生时及时止损，在故障恢复后自动恢复。

## 一、为什么需要服务容错

### 1.1 微服务故障的级联效应

假设服务调用链是：A → B → C → D

如果服务 D 变得极慢或完全不可用：

```
正常情况：
A → B → C → D → 响应 100ms

D 变慢后（超时 3 秒）：
A → B → C → [等待 D 超时] → 请求堆积

A、B、C 的线程池被占满：
A: 线程池满，拒绝请求
B: 线程池满，拒绝请求
C: 线程池满，拒绝请求

最终：整个调用链全部崩溃
```

这就是**雪崩效应**。

### 1.2 服务容错的核心手段

防止雪崩，有几种核心手段：

| 手段 | 作用 | 说明 |
| --- | --- | --- |
| **熔断（Circuit Breaker）** | 快速失败，阻止故障蔓延 | 当下游服务失败率过高时，打开熔断器 |
| **降级（Fallback）** | 返回兜底数据 | 服务不可用时，返回预设的默认值 |
| **限流（Rate Limiting）** | 控制请求速率 | 限制进入系统的流量 |
| **隔离（Bulkhead）** | 资源隔离 | 不同服务使用不同的线程池 |
| **超时（Timeout）** | 防止无限等待 | 设置请求的最大等待时间 |

## 二、流量控制与熔断降级

### 2.1 Sentinel 的定位

Sentinel（哨兵）是阿里巴巴开源的流量控制与熔断降级组件，是 Hystrix 的替代方案：

| 维度 | Sentinel | Hystrix |
| --- | --- | --- |
| 流量控制 | 支持 | 不支持 |
| 熔断降级 | 支持 | 支持 |
| 系统自适应 | 支持 | 不支持 |
| 动态规则 | 支持 | 支持 |
| 社区活跃度 | 高 | 停止维护 |
| 文档完善度 | 高 | 中 |

### 2.2 Sentinel 的核心概念

Sentinel 有几个核心概念：

```
Resource（资源）→ Rule（规则）→ Slot（插槽）
```

- **Resource**：被保护的资源，可以是一个接口、一个方法、一个服务
- **Rule**：流量控制规则、熔断降级规则、系统规则等
- **Slot**：Sentinel 的核心处理器，串联成一条链

### 2.3 Slot 链的执行顺序

Sentinel 的处理链如下：

```java
// 简化后的 Slot 链
ProcessorSlotChain chain = new ProcessorSlotChain();
chain.addLast(new NodeSelectorSlot());      // 1. 构建调用树
chain.addLast(new ClusterBuilderSlot());    // 2. 构建集群节点
chain.addLast(new LogSlot());              // 3. 记录日志
chain.addLast(new StatisticSlot());         // 4. 统计数据
chain.addLast(new AuthoritySlot());         // 5. 授权规则
chain.addLast(new SystemSlot());           // 6. 系统规则
chain.addLast(new FlowSlot());             // 7. 流量控制
chain.addLast(new DegradeSlot());          // 8. 熔断降级
```

## 五、延伸思考

### 5.1 什么时候该用熔断？什么时候该用限流？

| 场景 | 推荐方案 |
| --- | --- |
| 保护服务不被瞬时流量冲垮 | 限流 |
| 保护服务不被下游故障拖垮 | 熔断 |
| 下游服务不可用，但有备用数据 | 降级 |
| 不同服务使用不同资源池 | 隔离 |

### 5.2 降级 vs 熔断

| 维度 | 降级（Fallback） | 熔断（Circuit Breaker） |
| --- | --- | --- |
| 触发条件 | 单次调用失败 | 失败率达到阈值 |
| 行为 | 返回兜底数据 | 拒绝后续请求 |
| 恢复 | 立即恢复 | 需要等待或试探 |

降级是**主动**的——你知道某个功能可以"牺牲"；熔断是**被动**的——故障发生后才触发。

---

## 六、总结

服务容错与治理是微服务架构的"保护伞"：

- **熔断**：在故障蔓延前"断电"，保护系统不被拖垮
- **降级**：在服务不可用时返回"可用"的替代方案
- **限流**：控制流量，防止瞬时流量冲垮系统
- **隔离**：资源隔离，避免一个服务的问题影响其他服务
- **链路追踪**：快速定位问题，了解系统调用全貌

理解这些容错手段，才能在微服务架构中构建出稳定可靠的系统。
