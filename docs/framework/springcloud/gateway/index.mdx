# 服务调用与网关：微服务的入口与出口

在微服务架构中，服务之间的调用就像是一场精心组织的"接力赛"——每个服务只知道把接力棒交给下一个服务，而不需要关心整个赛道的全貌。

但这场接力赛需要一个"调度中心"：它知道每个运动员在哪里，知道接力棒该怎么传，知道哪些运动员受伤了该换人。**服务调用和网关，就是这个调度中心的核心组件。**

## 一、服务调用的演进之路

### 1.1 从 RestTemplate 到 OpenFeign

在 Spring Cloud 的早期，服务的远程调用主要依赖 `RestTemplate` + `Ribbon`：

```java
// RestTemplate + Ribbon 的调用方式
@Service
public class OrderService {
    
    @Autowired
    private RestTemplate restTemplate;
    
    public Order getOrder(Long id) {
        // 硬编码服务名，Ribbon 负责负载均衡
        return restTemplate.getForObject(
            "http://order-service/api/orders/" + id, 
            Order.class
        );
    }
}
```

这种方式有几个问题：

1. **代码冗长**：每次调用都要写 `restTemplate.getForObject()`
2. **URL 拼接麻烦**：字符串拼接容易出错
3. **异常处理复杂**：HTTP 层面的异常需要手动处理
4. **接口不直观**：无法一眼看出这个方法调的是哪个服务

OpenFeign 的出现，解决了这些问题：

```java
// OpenFeign 的调用方式
@FeignClient(name = "order-service")
public interface OrderClient {
    
    @GetMapping("/api/orders/{id}")
    Order getOrder(@PathVariable("id") Long id);
}

// 使用
@Service
public class OrderService {
    
    @Autowired
    private OrderClient orderClient;
    
    public Order getOrder(Long id) {
        // 像调用本地方法一样调用远程服务
        return orderClient.getOrder(id);
    }
}
```

### 1.2 为什么要引入网关

随着微服务数量增长，问题随之而来：

- **安全问题**：每个服务都要独立实现认证和鉴权，代码重复
- **跨域问题**：前端需要调用多个服务，跨域配置复杂
- **统一入口**：前端不想记住每个服务的地址
- **日志追踪**：请求经过多个服务，日志需要关联

网关的作用就是：**作为所有请求的统一入口**，在网关层面统一处理这些横切关注点。

## 二、服务调用的核心组件

### 2.1 OpenFeign 的工作原理

OpenFeign 的核心是**声明式接口** + **动态代理**：

```java
@FeignClient(name = "order-service")
public interface OrderClient {
    
    @GetMapping("/api/orders/{id}")
    Order getOrder(@PathVariable("id") Long id);
}
```

当你注入 `OrderClient` 时，Spring 返回给你的实际上是一个**代理对象**：

```java
// 简化后的代理逻辑
public class FeignClientFactory {
    
    public <T> T create(Class<T> clazz) {
        // 1. 使用 JDK 动态代理
        return (T) Proxy.newProxyInstance(
            clazz.getClassLoader(),
            new Class[]{clazz},
            new FeignInvocationHandler(clazz)
        );
    }
}

public class FeignInvocationHandler implements InvocationHandler {
    
    @Override
    public Object invoke(Object proxy, Method method, Object[] args) {
        // 2. 解析注解，构建请求
        RequestTemplate template = buildTemplate(method, args);
        
        // 3. 执行 HTTP 请求
        Response response = client.execute(template);
        
        // 4. 解析响应
        return decode(method, response);
    }
}
```

### 2.2 Ribbon 的负载均衡集成

OpenFeign 底层集成了 Ribbon，实现了负载均衡：

```yaml title="application.yml"
# Ribbon 配置
order-service:
  ribbon:
    # 负载均衡策略
    NFLoadBalancerRuleClassName: com.netflix.loadbalancer.RandomRule
    # 连接超时
    ConnectTimeout: 1000
    # 读取超时
    ReadTimeout: 3000
    # 重试次数
    MaxAutoRetries: 3
```

常用的负载均衡策略：

| 策略 | 类名 | 说明 |
| --- | --- | --- |
| 轮询 | `RoundRobinRule` | 按顺序逐一选择 |
| 随机 | `RandomRule` | 随机选择一个 |
| 权重 | `WeightedResponseTimeRule` | 根据响应时间分配权重 |
| 最少连接 | `BestAvailableRule` | 选择连接数最少的 |
| 重试 | `RetryRule` | 先用轮询，失败后重试 |

## 三、网关：请求的统一入口

### 3.1 Spring Cloud Gateway 的定位

Spring Cloud Gateway 是 Spring Cloud 的新一代网关，目标是替代 Netflix 的 Zuul（已停止维护）：

| 维度 | Zuul 1.x | Zuul 2.x | Gateway |
| --- | --- | --- | --- |
| 线程模型 | 同步阻塞 | 同步阻塞 | 响应式（非阻塞） |
| 性能 | 一般 | 较好 | 好 |
| 维护状态 | 停止维护 | 维护中 | 活跃维护 |
| 编程模型 | 过滤器链 | 过滤器链 | 过滤器链 + Predicate |

### 3.2 Gateway 的核心概念

Gateway 有三个核心概念：

```
Request → Route（路由）→ Predicate（谓词）→ Filter（过滤器）→ Backend
```

- **Route（路由）**：路由的基本信息，包括 ID、目标 URL、Predicate、Filter
- **Predicate（谓词）**：匹配条件，决定哪些请求会被这个路由处理
- **Filter（过滤器）**：请求和响应的预处理/后处理

## 六、延伸思考

### 6.1 为什么需要统一的网关层

想象一个没有网关的微服务架构：

```
前端 → 认证服务 → 用户服务
       ↓
前端 → 订单服务 → 商品服务 → 库存服务
       ↓
前端 → 支付服务 → 积分服务 → 消息服务
```

问题：
- 前端需要记住 5 个服务的地址
- 每个服务都要独立实现登录校验
- 跨域配置重复 5 次
- 某天订单服务地址变了，前端要改代码

加入网关后：

```
前端 → [Gateway] → 路由到对应服务
             ↓
          统一认证
          统一日志
          统一限流
          统一跨域
```

### 6.2 Gateway vs OpenFeign 的分工

| 组件 | 位置 | 职责 | 处理对象 |
| --- | --- | --- | --- |
| Gateway | 网关层 | 路由、认证、限流、日志 | 进入系统的外部请求 |
| OpenFeign | 服务间 | 服务调用、负载均衡 | 服务间的内部调用 |

两者是互补关系，不是替代关系：
- **外部请求**经过 Gateway 统一入口
- **内部服务间调用**使用 OpenFeign（或其他 RPC 框架）

---

## 七、总结

服务调用与网关是微服务架构的两大支柱：

- **OpenFeign**：让服务间的调用像本地方法调用一样简单，底层通过动态代理实现
- **Gateway**：作为统一入口，处理认证、限流、日志等横切关注点

理解这两个组件的工作原理，是掌握 Spring Cloud 微服务通信的关键。
