# Spring Cloud Gateway 过滤器：请求的预处理和后处理

网关不仅路由请求，还能修改请求、响应、统计日志。

如果说路由决定了"请求去哪里"，那么过滤器决定了"请求怎么去"。在请求发往前端服务之前，在响应返回给客户端之前，都可以插入各种处理逻辑。这就是 Gateway 过滤器的魔力。

---

## 一、过滤器类型概览

### 1.1 GatewayFilter vs GlobalFilter

Gateway 的过滤器分为两种类型：

| 类型 | 作用范围 | 配置方式 | 使用场景 |
| --- | --- | --- | --- |
| GatewayFilter | 作用于单个路由 | 在路由的 `filters` 中配置 | 特定路由的特殊处理 |
| GlobalFilter | 作用于所有路由 | 全局生效，无需单独配置 | 统一的横切关注点 |

### 1.2 内置过滤器一览

Gateway 内置了大量过滤器：

**请求处理类：**

| 过滤器 | 功能 |
| --- | --- |
| `AddRequestHeader` | 添加请求头 |
| `AddRequestParameter` | 添加请求参数 |
| `RemoveRequestHeader` | 移除请求头 |
| `RemoveRequestParameter` | 移除请求参数 |
| `SetRequestHeader` | 设置请求头（覆盖） |
| `SetRequestHostHeader` | 设置 Host 请求头 |

**响应处理类：**

| 过滤器 | 功能 |
| --- | --- |
| `AddResponseHeader` | 添加响应头 |
| `RemoveResponseHeader` | 移除响应头 |
| `SetResponseHeader` | 设置响应头（覆盖） |

**路径处理类：**

| 过滤器 | 功能 |
| --- | --- |
| `StripPrefix` | 去掉 URL 前缀 |
| `PrefixPath` | 添加 URL 前缀 |
| `SetPath` | 修改 URL 路径 |

**其他：**

| 过滤器 | 功能 |
| --- | --- |
| `Retry` | 请求重试 |
| `RequestSize` | 限制请求大小 |
| `SetStatus` | 修改响应状态码 |
| `RedirectTo` | 重定向 |

## 二、过滤器链的执行顺序

### 2.1 顺序的重要性

Gateway 的过滤器链有严格的执行顺序：

```
客户端请求
    ↓
Gateway Handler Mapping → 匹配路由
    ↓
Gateway Web Handler → 构建过滤器链
    ↓
┌─────────────────────────────────────┐
│         Pre 过滤器链（正向）           │
│  GlobalFilter_1 → GlobalFilter_2 → ... │
│  GatewayFilter_1 → GatewayFilter_2 → ... │
└─────────────────────────────────────┘
    ↓
     目标服务
    ↓
┌─────────────────────────────────────┐
│        Post 过滤器链（反向）           │
│  GatewayFilter_2 ← GatewayFilter_1 ← ... │
│  GlobalFilter_2 ← GlobalFilter_1 ← ... │
└─────────────────────────────────────┘
    ↓
客户端响应
```

### 2.2 顺序控制：Ordered 接口

所有过滤器都实现 `Ordered` 接口，通过 `getOrder()` 返回执行顺序：

```java
public interface Ordered {
    int HIGHEST_PRECEDENCE = Integer.MIN_VALUE;  // 最高优先级
    int LOWEST_PRECEDENCE = Integer.MAX_VALUE;   // 最低优先级
    int getOrder();
}
```

数值越小，执行越靠前：
- `-100`：最高优先级
- `0`：默认优先级
- `100`：较低优先级
- `Integer.MAX_VALUE`：最低优先级

### 2.3 @Order 注解

可以使用 `@Order` 注解简化顺序控制：

```java
@Order(1)
@Component
public class FirstFilter implements GlobalFilter {
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        System.out.println("第一个过滤器");
        return chain.filter(exchange);
    }
}

@Order(2)
@Component
public class SecondFilter implements GlobalFilter {
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        System.out.println("第二个过滤器");
        return chain.filter(exchange);
    }
}
```

## 三、常用 GlobalFilter

### 3.1 过滤器链的默认顺序

Gateway 内置的 GlobalFilter 按以下顺序执行（部分）：

| 顺序 | 过滤器 | 功能 |
| --- | --- | --- |
| -10000 | `AdaptCachedBody` | 缓存请求体 |
| -5000 | `RemoveCachedBody` | 移除缓存的请求体 |
| -2500 | `RouteToRequestUrl` | 将请求路由到目标 URL |
| -1000 | `WebSocketRoute` | WebSocket 路由 |
| 0 | `LoadBalancerClient` | 负载均衡客户端 |
| 1 | `Routing` | 实际发起请求 |
| -1 | `WebClientHttpRouting` | WebClient 路由 |

### 3.2 NettyRoutingFilter

使用 Netty 发起 HTTP 请求的过滤器：

```java
// 简化逻辑
public class NettyRoutingFilter implements GlobalFilter, Ordered {
    
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        // 1. 获取目标 URL
        String url = exchange.getAttribute(GATEWAY_REQUEST_URL_ATTR);
        
        // 2. 使用 Netty 发起请求
        Mono<HttpClientResponse> responseMono = httpClient.get(url);
        
        return responseMono
            .doOnSuccess(response -> {
                // 3. 处理响应
                exchange.getResponse().setStatusCode(response.status());
            })
            .then(chain.filter(exchange));
    }
}
```

### 3.3 ForwardRoutingFilter

处理 `forward://` 前缀的路由，直接在网关内部转发：

```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: forward-route
          uri: forward:/local/endpoint
          predicates:
            - Path=/gateway/local/**
```

## 四、常用 GatewayFilter 配置

### 4.1 AddRequestHeader：添加请求头

```yaml title="application.yml"
spring:
  cloud:
    gateway:
      routes:
        - id: add-header-route
          uri: lb://order-service
          predicates:
            - Path=/api/orders/**
          filters:
            # 添加固定的请求头
            - AddRequestHeader=X-Gateway,SpringCloudGateway
            # 添加动态请求头
            - AddRequestHeader=X-Request-Time,{T:System}.currentTimeMillis()
```

### 4.2 AddRequestParameter：添加请求参数

```yaml title="application.yml"
spring:
  cloud:
    gateway:
      routes:
        - id: add-param-route
          uri: lb://order-service
          predicates:
            - Path=/api/orders/**
          filters:
            # 添加查询参数
            - AddRequestParameter=source,gateway
            - AddRequestParameter=version,2.0
```

### 4.3 StripPrefix：去掉前缀

```yaml title="application.yml"
spring:
  cloud:
    gateway:
      routes:
        - id: strip-prefix-route
          uri: lb://order-service
          predicates:
            - Path=/api/orders/**
          filters:
            # 去掉路径的前 1 个部分
            # /api/orders/123 → /orders/123
            - StripPrefix=1
```

多级前缀：
```yaml
filters:
  # 去掉路径的前 2 个部分
  # /api/v1/orders/123 → /orders/123
  - StripPrefix=2
```

### 4.4 SetStatus：修改状态码

```yaml title="application.yml"
spring:
  cloud:
    gateway:
      routes:
        - id: status-route
          uri: lb://order-service
          predicates:
            - Path=/api/orders/**
          filters:
            # 修改状态码：认证失败时返回 401
            - SetStatus=401
```

### 4.5 RetryGatewayFilter：重试过滤器

```yaml title="application.yml"
spring:
  cloud:
    gateway:
      routes:
        - id: retry-route
          uri: lb://order-service
          predicates:
            - Path=/api/orders/**
          filters:
            - name: Retry
              args:
                retries: 3                    # 重试次数
                statuses: SERVER_ERROR        # 对 5xx 错误重试
                methods: GET,POST             # 只对 GET 和 POST 重试
                backoff:
                  firstBackoff: 100ms         # 首次重试间隔
                  maxBackoff: 500ms           # 最大重试间隔
                  factor: 2                   # 间隔倍数
```

支持的配置项：

| 参数 | 说明 | 示例 |
| --- | --- | --- |
| `retries` | 重试次数 | `3` |
| `statuses` | 重试的状态码 | `SERVER_ERROR, GATEWAY_TIMEOUT` |
| `methods` | 重试的 HTTP 方法 | `GET,POST` |
| `series` | 重试的状态码系列 | `5xx` |
| `exceptions` | 重试的异常类型 | `IOException, TimeoutException` |
| `backoff` | 退避策略 | 见上 |

## 五、自定义 GlobalFilter

### 5.1 认证过滤器

最常见的自定义过滤器：检查请求是否有有效的 Token：

```java
@Component
public class AuthenticationFilter implements GlobalFilter, Ordered {
    
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        // 1. 放行登录接口
        String path = exchange.getRequest().getPath().value();
        if (path.startsWith("/auth/")) {
            return chain.filter(exchange);
        }
        
        // 2. 检查 Token
        String token = exchange.getRequest().getHeaders().getFirst("Authorization");
        if (StringUtils.isEmpty(token)) {
            exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
            return exchange.getResponse().setComplete();
        }
        
        // 3. 验证 Token 有效性
        if (!validateToken(token)) {
            exchange.getResponse().setStatusCode(HttpStatus.FORBIDDEN);
            return exchange.getResponse().setComplete();
        }
        
        // 4. 将用户信息传递下去
        ServerWebExchange modifiedExchange = exchange.mutate()
            .request(builder -> builder.header("X-User-Id", getUserId(token)))
            .build();
        
        return chain.filter(modifiedExchange);
    }
    
    @Override
    public int getOrder() {
        return -100;  // 优先级较高，在路由之前执行
    }
    
    private boolean validateToken(String token) {
        // Token 验证逻辑
        return true;
    }
    
    private String getUserId(String token) {
        // 从 Token 中提取用户 ID
        return "user123";
    }
}
```

### 5.2 限流过滤器

结合 Redis 实现限流：

```java
@Component
public class RateLimitFilter implements GlobalFilter, Ordered {
    
    private final StringRedisTemplate redisTemplate;
    
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        // 1. 获取限流 Key（基于 IP）
        String ip = exchange.getRequest().getRemoteAddress().getAddress().getHostAddress();
        String key = "rate_limit:" + ip;
        
        // 2. 检查是否超过限制
        Long current = redisTemplate.opsForValue().increment(key);
        if (current != null && current == 1) {
            // 第一次访问，设置过期时间
            redisTemplate.expire(key, 1, TimeUnit.SECONDS);
        }
        
        if (current != null && current > 100) {
            // 超过限制，返回 429 Too Many Requests
            exchange.getResponse().setStatusCode(HttpStatus.TOO_MANY_REQUESTS);
            return exchange.getResponse().setComplete();
        }
        
        // 3. 添加限流相关响应头
        exchange.getResponse().getHeaders().add("X-RateLimit-Remaining", String.valueOf(Math.max(0, 100 - current)));
        
        return chain.filter(exchange);
    }
    
    @Override
    public int getOrder() {
        return -200;  // 较早执行
    }
}
```

### 5.3 日志过滤器

记录请求日志：

```java
@Component
@Slf4j
public class LoggingFilter implements GlobalFilter, Ordered {
    
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        ServerHttpRequest request = exchange.getRequest();
        
        // 记录请求开始时间
        long startTime = System.currentTimeMillis();
        String path = request.getPath().value();
        String method = request.getMethod().name();
        
        log.info("请求开始: {} {}", method, path);
        
        return chain.filter(exchange)
            .then(Mono.fromRunnable(() -> {
                // 请求结束后记录
                long duration = System.currentTimeMillis() - startTime;
                int status = exchange.getResponse().getStatusCode().value();
                
                log.info("请求结束: {} {} -> {} ({}ms)", 
                    method, path, status, duration);
            }));
    }
    
    @Override
    public int getOrder() {
        return Ordered.LOWEST_PRECEDENCE;  // 最后执行
    }
}
```

## 六、Combined Filter 的执行顺序

### 6.1 单向过滤器链

Gateway 的过滤器链是**单向**执行的：

```text
Pre:  Filter_1 → Filter_2 → Filter_3 → 目标服务
                                    ↓
Post: Filter_3 ← Filter_2 ← Filter_1 ← 响应返回
```

### 6.2 Pre 和 Post 的区分

通过 `chain.filter(exchange)` 的位置来区分：

```java
// Pre 过滤器（在调用 chain.filter 之前执行）
public Mono<Void> preFilter(ServerWebExchange exchange, GatewayFilterChain chain) {
    // 预处理逻辑
    exchange.getRequest().getHeaders().add("X-Pre", "true");
    
    // 继续执行后续过滤器
    return chain.filter(exchange);  // ← 在这里，Pre 部分结束
}

// Post 过滤器（在 chain.filter 完成之后执行）
public Mono<Void> postFilter(ServerWebExchange exchange, GatewayFilterChain chain) {
    return chain.filter(exchange)  // 先执行后续过滤器
        .then(Mono.fromRunnable(() -> {
            // 后处理逻辑
            exchange.getResponse().getHeaders().add("X-Post", "true");
        }));
}
```

### 6.3 双向过滤器的实际例子

```java
@Component
public class RequestTimingFilter implements GlobalFilter, Ordered {
    
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        // Pre 部分：记录开始时间
        exchange.getAttributes().put("startTime", System.currentTimeMillis());
        
        return chain.filter(exchange)  // 执行后续过滤器，到达目标服务
            .then(Mono.fromRunnable(() -> {
                // Post 部分：计算耗时
                Long startTime = exchange.getAttribute("startTime");
                if (startTime != null) {
                    long duration = System.currentTimeMillis() - startTime;
                    exchange.getResponse().getHeaders().add("X-Response-Time", duration + "ms");
                }
            }));
    }
}
```

## 七、面试追问方向

- **GatewayFilter 和 GlobalFilter 的区别是什么？** GatewayFilter 作用于单个路由，需要在路由中显式配置；GlobalFilter 作用于所有路由，全局生效。
- **过滤器的执行顺序是如何确定的？** 通过 `Ordered.getOrder()` 返回的数值确定，数值越小越靠前执行。
- **Pre 过滤器和 Post 过滤器如何区分？** 通过 `chain.filter()` 的位置区分：在调用之前的代码是 Pre 部分，在 `then()` 中的是 Post 部分。
- **如何自定义过滤器？** 实现 `GlobalFilter` 接口（或 `GatewayFilter`），重写 `filter()` 方法，在 `@Configuration` 类中注册。

---

## 八、总结

Gateway 的过滤器机制是处理横切关注点的核心：

- **GatewayFilter**：作用于单个路由，适合特定路由的特殊处理
- **GlobalFilter**：作用于所有路由，适合统一的横切关注点
- **Ordered 接口**：控制过滤器执行顺序
- **Pre 和 Post**：过滤器分为预处理和后处理两个阶段
- **自定义过滤器**：通过实现 `GlobalFilter` 接口

掌握过滤器机制，才能在 Gateway 中实现认证、限流、日志等通用功能，同时为特定路由添加特殊处理逻辑。
