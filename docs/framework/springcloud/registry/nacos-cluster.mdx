# Nacos 集群：Raft 协议下的高可用

生产环境中，Nacos 如果只有单机部署，万一宕机了怎么办？

对于微服务架构来说，服务注册中心和配置中心是整个系统的"神经中枢"——如果它挂了，服务之间无法发现彼此，配置无法读取，整个系统将陷入瘫痪。

**Nacos 集群的高可用设计，就是为了解决这个问题。**

## 一、为什么需要集群

### 1.1 单机部署的风险

Nacos 单机部署时，一旦进程崩溃或服务器宕机：

- **服务注册失败**：新启动的服务无法注册，已经注册的服务实例列表停止更新
- **配置读取失败**：应用无法获取配置，如果应用启动时拉取不到关键配置（如数据库连接），将无法正常启动
- **数据丢失风险**：如果使用的是嵌入式数据库（Derby），重启后数据可能丢失

### 1.2 集群的核心目标

Nacos 集群的核心目标是实现：

1. **故障转移**：任意一个节点宕机，其他节点继续提供服务
2. **数据一致性**：所有节点存储相同的数据，任意一个节点的数据变更都能同步到其他节点
3. **负载均衡**：客户端可以连接任意一个节点，分散单点压力

```text
# 单机 vs 集群
单机：
  客户端 → [Nacos Server] ← 挂了，全完了

集群：
  客户端 → [Node1] ─┬─→ [Node2]  ← 挂了 Node1，Node2 继续服务
         └─→ [Node3]  ← 挂了 Node1、2，Node3 继续服务（过半即可）
```

## 二、Nacos 集群架构

### 2.1 典型的集群拓扑

生产环境中，Nacos 集群通常由 3 个或 5 个节点组成：

```
                    ┌─────────────┐
                    │   客户端    │
                    └──────┬──────┘
                           │
              ┌────────────┼────────────┐
              ↓            ↓            ↓
        ┌──────────┐ ┌──────────┐ ┌──────────┐
        │  Node 1  │ │  Node 2  │ │  Node 3  │
        │ (Leader)  │ │ (Follow) │ │ (Follow) │
        └─────┬────┘ └────┬────┘ └────┬────┘
              └────────────┼────────────┘
                           ↓
                    ┌─────────────┐
                    │   数据库     │
                    │  (MySQL)    │
                    └─────────────┘
```

### 2.2 节点数量的选择

| 节点数 | 可容忍宕机数 | 适用场景 |
| --- | --- | --- |
| 1 | 0 | 开发测试 |
| 3 | 1 | 生产推荐最小规模 |
| 5 | 2 | 对可用性要求极高的场景 |
| 7+ | 3+ | 大型互联网公司核心系统 |

**为什么推荐 3 节点或 5 节点？**

因为 Nacos 使用 Raft 协议，需要**过半节点存活**才能正常工作：
- 3 节点集群：可以容忍 1 节点宕机
- 5 节点集群：可以容忍 2 节点宕机
- 节点数必须是奇数（1、3、5、7...），偶数没有意义（4 节点和 3 节点的容错能力一样，但 4 节点更浪费）

## 三、Raft 协议详解

### 3.1 Raft 的核心思想

Raft 是一个共识算法（Consensus Algorithm），它将分布式共识问题分解为三个子问题：

1. **领导者选举（Leader Election）**：在集群中选出一个领导者
2. **日志复制（Log Replication）**：领导者接收客户端请求，将日志复制到其他节点
3. **安全性（Safety）**：保证每个状态机按相同顺序执行相同命令

Nacos 使用 Raft 协议来保证配置数据在集群节点间的一致性。

### 3.2 三种角色

Raft 集群中的节点有三种角色：

| 角色 | 说明 | 职责 |
| --- | --- | --- |
| Leader（领导者） | 集群中唯一 | 处理所有客户端请求，复制日志到 Follower |
| Follower（跟随者） | 可以有多个 | 响应 Leader 的日志复制请求，响应候选人的投票请求 |
| Candidate（候选人） | 选举时出现 | 发起投票请求，试图成为新的 Leader |

```
选举过程：
Follower → Candidate（选举超时）→ 发起投票 → 获得多数票 → 成为 Leader
```

### 3.3 领导者选举

领导者选举是 Raft 最核心的机制之一：

```java
// Raft 选举的核心参数
public class RaftInfo {
    // 当前任期号（Term），每选举一次递增
    private long currentTerm = 0;
    
    // 投票给哪个节点（节点 ID）
    private String votedFor;
    
    // 选举超时时间（随机 150~300ms），防止所有节点同时发起选举
    private long electionTimeout;
    
    // 心跳间隔（必须小于选举超时）
    private long heartbeatInterval = 150;
}
```

**选举流程**：

1. 节点启动时默认为 Follower
2. 如果在 `electionTimeout` 时间内没收到 Leader 的心跳，就变成 Candidate
3. Candidate 增加自己的 Term，给自己投票，然后向其他节点发起投票请求
4. 获得集群**多数节点**（`> n/2`）投票的 Candidate 成为新的 Leader
5. Leader 定期向 Follower 发送心跳，维持自己的领导地位

```text
# 选举时间线
时间 →
T1: Node1 超时 → 成为 Candidate → Term=1 → 发起投票
T2: Node2 收到投票请求 → 投票给 Node1
T3: Node3 收到投票请求 → 投票给 Node1
T4: Node1 获得 2 票（过半数）→ 成为 Leader
T5: Node1 定期发送心跳给 Node2、Node3
```

> [!TIP]
> **为什么需要随机选举超时？** 如果所有节点的选举超时时间相同，它们会同时发起选举，导致票数分散，谁都拿不到多数。通过随机化超时时间，可以确保在大多数情况下只有一个节点先发起选举并获胜。

### 3.4 日志复制

一旦选出了 Leader，它就可以处理客户端请求了：

```java
// 日志条目的结构
public class LogEntry {
    private long term;      // 收到请求时的任期号
    private long index;     // 日志索引
    private String key;     // 配置的 key
    private String value;   // 配置的 value
    private long timestamp; // 时间戳
}
```

**日志复制流程**：

```
1. 客户端发送配置更新请求给 Leader
   PUT /api/configs -d {key: "timeout", value: "5000"}

2. Leader 将请求写入本地日志（此时还未提交）
   日志：[{term:5, index:100, key:"timeout", value:"5000"}]

3. Leader 并行向所有 Follower 发送 AppendEntries 请求
   ┌─────────┐     AppendEntries     ┌─────────┐
   │  Leader │ ──────────────────────→│ Follower│
   └─────────┘                        └─────────┘
   发送日志条目（term:5, index:100）

4. Follower 收到请求，将日志写入本地
   日志：[{term:5, index:100, key:"timeout", value:"5000"}]

5. Follower 回复 Leader：写入成功
   ┌─────────┐     Success            ┌─────────┘
   │  Leader │ ←──────────────────────┘
   └─────────┘

6. Leader 收到多数节点的成功回复后，提交日志（应用到状态机）
   提交：index:100 已提交

7. Leader 回复客户端：写入成功

8. Leader 通知 Follower 提交日志
   ┌─────────┐     Commit            ┌─────────┐
   │  Leader │ ──────────────────────→│ Follower│
   └─────────┘                        └─────────┘
```

### 3.5 脑裂问题与处理

**什么是脑裂？**

网络分区时，可能会出现两个 Leader 同时存在的情况：

```text
正常状态：
  [Node1:Leader] - [Node2:Follower] - [Node3:Follower]

网络分区后：
  区域 A: [Node1:Leader] - [Node2:Follower]
  区域 B: [Node3:Leader] ← Node3 重新选举成为 Leader
```

**Raft 如何解决脑裂？**

Raft 的关键保证是：**只有包含多数节点的那一侧可以继续工作**。

- 区域 A（2 节点）：可以继续写入，因为包含了多数（2 > 1.5）
- 区域 B（1 节点）：无法写入，因为不包含多数，Leader 变成 Follower

```java
// Leader 处理 AppendEntries 时的检查
if (prevLogIndex > followerLog.lastIndex()) {
    // Follower 日志太短，拒绝写入
    return false;
}

// Leader 检查自己是否是多数派
if (committedIndex > majority) {
    // 只有获得多数节点确认后，才能提交
    // 不包含多数的分区，Leader 无法提交任何操作
}
```

当网络恢复后：
- Term 较小的 Leader（区域 B 的 Node3）发现自己的 Term 比多数派 Leader 小
- Node3 自动变成 Follower，同步区域 A 的日志

## 四、数据持久化配置

### 4.1 Derby vs MySQL

Nacos 支持两种数据持久化方式：

| 方式 | 特点 | 适用场景 |
| --- | --- | --- |
| Derby（嵌入式） | 开箱即用，无需额外安装 | 开发测试 |
| MySQL | 需要额外安装 MySQL | 生产环境 |

```yaml title="cluster.conf"
# 集群节点列表
192.168.1.101:8848
192.168.1.102:8848
192.168.1.103:8848
```

### 4.2 MySQL 持久化配置

```yaml title="application.properties"
# 使用 MySQL 持久化
spring.datasource.platform=mysql

# 数据库 URL
db.url.0=jdbc:mysql://mysql-server:3306/nacos?characterEncoding=utf8&connectTimeout=1000&socketTimeout=3000&autoReconnect=true&useUnicode=true&useSSL=false&serverTimezone=UTC

# 数据库用户名
db.user.0=nacos

# 数据库密码
db.password.0=nacos_password

# 连接池配置
db.pool.config.minimum-idle=5
db.pool.config.maximum-pool-size=20
db.pool.config.connection-timeout=30000
```

### 4.3 初始化数据库

首次部署时，需要执行 Nacos 提供的 SQL 脚本初始化数据库：

```bash
# 下载 Nacos 安装包，解压后找到 sql 目录
cd nacos/conf

# 执行 SQL 脚本
mysql -u root -p nacos < nacos-mysql.sql
```

SQL 脚本会创建以下表：

| 表名 | 用途 |
| --- | --- |
| `config_info` | 配置信息主表 |
| `config_info_aggr` | 配置信息聚合表 |
| `config_info_beta` | 配置信息 Beta 发布 |
| `config_info_tag` | 配置标签表 |
| `config_tags_relation` | 配置标签关联表 |
| `group_capacity` | 分组容量表 |
| `his_config_info` | 配置历史记录表 |

## 五、集群节点的通信

### 5.1 gRPC 通信

Nacos 2.0 开始，使用 **gRPC** 替代了原有的 HTTP 协议进行节点间通信：

```yaml title="application.properties"
# gRPC 配置（2.0+）
nacos.server.grpc ports for clusters=9848
nacos.server.ip=192.168.1.101
```

| 端口 | 协议 | 用途 |
| --- | --- | --- |
| 8848 | HTTP | 客户端 API 端口 |
| 9848 | gRPC | 集群间数据同步（gRPC） |
| 9849 | gRPC | 选举通信（gRPC） |

### 5.2 节点发现与健康检查

集群节点之间会相互发送心跳：

```java
// 节点健康检查（简化逻辑）
public class MemberLookUp {
    
    private List<String> clusterNodes;
    
    public void checkMembers() {
        for (String node : clusterNodes) {
            try {
                // 发送 HTTP 请求检查节点状态
                Response response = httpClient.get("http://" + node + ":8848/v1/operator/metrics");
                if (response.isSuccess()) {
                    // 节点健康
                    updateNodeHealth(node, true);
                }
            } catch (Exception e) {
                // 节点不可达
                updateNodeHealth(node, false);
            }
        }
    }
}
```

## 六、Nginx + Nacos 高可用方案

### 6.1 架构设计

在生产环境中，通常会在 Nacos 集群前加一层 Nginx 做负载均衡：

```
                    ┌─────────────┐
                    │   客户端    │
                    └──────┬──────┘
                           ↓
                    ┌─────────────┐
                    │    Nginx    │  ← 统一入口，负载均衡
                    └──────┬──────┘
                           ↓
         ┌────────────────┼────────────────┐
         ↓                ↓                ↓
   ┌──────────┐     ┌──────────┐     ┌──────────┐
   │  Node 1  │     │  Node 2  │     │  Node 3  │
   │          │ ←→ │          │ ←→ │          │
   └──────────┘     └──────────┘     └──────────┘
```

### 6.2 Nginx 配置

```nginx title="nginx.conf"
upstream nacos-cluster {
    server 192.168.1.101:8848 weight=1;
    server 192.168.1.102:8848 weight=1;
    server 192.168.1.103:8848 weight=1;
}

server {
    listen 80;
    server_name nacos.example.com;

    location / {
        proxy_pass http://nacos-cluster;
        
        # 头信息转发
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        
        # 连接超时
        proxy_connect_timeout 10s;
        proxy_read_timeout 60s;
    }
}
```

### 6.3 客户端配置

客户端只需要连接 Nginx 的地址：

```yaml title="application.yml"
spring:
  cloud:
    nacos:
      discovery:
        server-addr: nacos.example.com:80  # 只需配置 Nginx 地址
      config:
        server-addr: nacos.example.com:80
```

Nginx 会自动将请求分发到后端的某个 Nacos 节点。

## 七、面试追问方向

- **Raft 协议的三种角色是什么？** Leader（领导者）、Follower（跟随者）、Candidate（候选人）。
- **Raft 如何解决脑裂问题？** 只有包含多数节点的那一侧可以继续工作，不包含多数的分区无法提交任何操作。
- **为什么 Nacos 集群节点数推荐奇数？** 因为 Raft 需要多数节点存活才能工作，偶数节点（如 4 节点）和少一个的奇数节点（如 3 节点）容错能力相同，但偶数更浪费。
- **Derby 和 MySQL 持久化的区别是什么？** Derby 是嵌入式数据库，开箱即用但数据不共享；MySQL 是独立数据库，支持数据共享，适合生产环境。

## 八、总结

Nacos 集群通过 Raft 协议实现了高可用和数据一致性：

- **领导者选举**：Raft 协议确保集群中只有一个 Leader
- **日志复制**：Leader 将数据同步到 Follower，只有获得多数确认才算成功
- **故障恢复**：节点宕机后，剩余节点自动重新选举
- **脑裂处理**：只有包含多数节点的分区能继续工作，网络恢复后自动合并

配合 Nginx 负载均衡和 MySQL 持久化，Nacos 集群可以支撑企业级生产环境的高可用需求。
