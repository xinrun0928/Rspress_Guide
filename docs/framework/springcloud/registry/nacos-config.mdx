# Nacos 配置中心：三层隔离让配置更安全

dev、test、prod 三套配置，怎么做到互不干扰？

如果问你"如何管理配置"，你可能会说：搞三套配置文件不就行了？但当你的微服务数量从 10 个增长到 100 个，每个服务都有 dev、test、prod 三套配置，总共就是 300 份配置文件。更要命的是，这些配置之间还有依赖关系——服务 A 的某个配置需要引用服务 B 的某个值——你怎么保证它们的一致性？

**Nacos 配置中心的三层隔离机制，就是为了解决这个问题的。**

---

## 一、从"配置文件"到"配置中心"

### 1.1 传统配置管理的困境

在传统的单体应用中，配置管理相对简单：所有的配置都在一个 `application.properties` 或 `application.yml` 文件中，改完配置，重启应用，完事。

但微服务架构下，这个模式行不通了：

- **100 个服务 = 100 份配置文件**，改一个公共配置要改 100 次
- **环境差异大**，dev 环境可能用 H2 内存数据库，prod 环境用 MySQL
- **配置安全性问题**，prod 的数据库密码不能出现在代码仓库里
- **配置变更困难**，改一个配置需要登录服务器、找到文件、修改、重启

配置中心应运而生。它的核心理念是：**配置是独立于代码的，应该集中管理、动态发布、实时生效**。

### 1.2 Nacos 配置中心的优势

相比 Apollo、Consul 等配置中心，Nacos 的优势在于"二合一"——服务注册和配置中心共用一套基础设施，运维成本更低。

| 维度 | Nacos | Apollo | Consul K/V |
| --- | --- | --- | --- |
| 服务注册发现 | 支持 | 不支持 | 支持 |
| 配置管理 | 支持 | 支持 | 支持 |
| 部署复杂度 | 中 | 高 | 中 |
| 功能完整性 | 高 | 高 | 中 |
| 社区活跃度 | 高（SCA） | 高 | 高 |

## 二、三层隔离：Namespace、Group、Data ID

这是 Nacos 配置中心的核心概念。理解了这三层的含义和优先级，才能正确地使用配置中心。

### 2.1 Namespace：环境隔离

Namespace 是最外层的隔离维度，通常用来区分不同的环境：

```
dev 命名空间 → 开发人员使用，数据可以随意折腾
test 命名空间 → 测试环境，模拟生产但数据独立
prod 命名空间 → 正式环境，权限严格控制
```

在 Nacos 控制台创建命名空间后，会得到一个命名空间 ID（不是名字，是一串 UUID）。这个 ID 是在客户端配置中使用的：

```yaml title="bootstrap.yml"
spring:
  application:
    name: order-service
  cloud:
    nacos:
      config:
        server-addr: 127.0.0.1:8848
        namespace: dev  # 这里填命名空间 ID，不是名字
```

> [!WARNING]
> 很多新手会犯的错误是：在这里填了命名空间的"名字"（如 `dev`），而不是"命名空间 ID"。结果是配置没有加载到正确的环境。记住：**客户端用的是 ID，不是名字**。

### 2.2 Group：分组隔离

Group 是第二层隔离，通常用于区分不同的业务线、项目或模块：

```
命名空间（prod）
  ├── Group: payment-suit（支付业务线）
  │     ├── order-service.yaml
  │     └── payment-service.yaml
  ├── Group: user-center（用户中心）
  │     ├── user-service.yaml
  │     └── auth-service.yaml
  └── Group: infrastructure（基础设施）
        ├── gateway.yaml
        └── monitoring.yaml
```

Group 的命名通常使用公司内部的项目代号，如 `BAT`、`YZD`、`ORDER-SUIT` 等。Group 名默认是 `DEFAULT_GROUP`，如果不需要额外隔离，可以不配置。

```yaml title="bootstrap.yml"
spring:
  cloud:
    nacos:
      config:
        group: ORDER-SUIT  # 指定分组，默认 DEFAULT_GROUP
```

Group 的典型使用场景：

1. **多业务线隔离**：支付业务线和电商业务线各自维护自己的配置
2. **多租户隔离**：SAAS 模式下，不同租户的配置放在不同 Group
3. **环境内部再分区**：prod 环境内部，按机房或业务分区

### 2.3 Data ID：配置归属

Data ID 是最细粒度的隔离，对应一份具体的配置文件。在 Nacos 中，配置是按 Data ID 组织的：

```
Data ID: order-service.yaml
Data ID: order-service-dev.yaml
Data ID: order-service-prod.yaml
```

Data ID 的命名规范，通常遵循：`${spring.application.name}-${profile}.properties`

| Data ID | 含义 |
| --- | --- |
| `order-service.yaml` | order-service 的主配置，所有环境共享 |
| `order-service-dev.yaml` | order-service 的 dev 环境配置 |
| `order-service-test.yaml` | order-service 的 test 环境配置 |
| `order-service-prod.yaml` | order-service 的 prod 环境配置 |

这种命名规范的好处是：**配置加载顺序清晰，覆盖关系明确**。

## 三、三层隔离的优先级

当同一个配置项在三处都有定义时，优先级是怎样的？

```
Namespace > Group > Data ID
```

更准确地说，是一个**层级覆盖**的关系：

1. **Data ID 覆盖 shared-dataids 和 extension-configs**
2. **Group 按配置顺序依次覆盖**
3. **Namespace 决定配置的整体归属**

```yaml title="bootstrap.yml"
spring:
  cloud:
    nacos:
      config:
        # 1. 共享配置（shared-dataids），所有环境共享
        shared-dataids: common.yaml,logging.yaml
        
        # 2. 扩展配置（extension-configs），可指定 Group
        extension-configs:
          - data-id: database.yaml
            group: INFRASTRUCTURE
            refresh: true
          - data-id: redis.yaml
            group: INFRASTRUCTURE
            refresh: true
        
        # 3. 自身配置（${spring.application.name}-${profile}.yaml）
        # 优先级最高，会覆盖上面的同名配置
```

| 配置类型 | 优先级 | 说明 |
| --- | --- | --- |
| 自身配置 | 最高 | `${spring.application.name}-${profile}.yaml` |
| extension-configs | 次高 | 扩展配置，可指定 Group |
| shared-dataids | 次低 | 共享配置，默认 DEFAULT_GROUP |
| 默认配置 | 最低 | `${spring.application.name}.yaml` |

### 实际加载顺序

假设你配置了 `spring.application.name=order-service`，`spring.profiles.active=prod`，Nacos 会按以下顺序加载配置：

```
1. ${spring.application.name}.yaml        → order-service.yaml
2. ${spring.application.name}.${profile}.yaml → order-service-prod.yaml
3. shared-dataids 中的配置
4. extension-configs 中的配置（按数组顺序）
```

**后面的配置会覆盖前面的同名配置**。这个机制使得：

- 在公共配置中定义默认值
- 在扩展配置中覆盖公共配置
- 在自身配置中覆盖所有配置

## 四、配置的加载过程

### 4.1 bootstrap.yml 的作用

Spring Cloud 使用 `bootstrap.yml`（或 `bootstrap.properties`）来加载配置中心的相关配置。这个文件的特点是：**在 application.yml 之前加载**。

```yaml title="bootstrap.yml"
spring:
  application:
    name: order-service
  profiles:
    active: prod
  cloud:
    nacos:
      config:
        server-addr: 127.0.0.1:8848
        namespace: prod
        group: ORDER-SUIT
        file-extension: yaml
        refresh-enabled: true
```

> [!TIP]
> bootstrap.yml 之所以在 application.yml 之前加载，是因为配置中心的配置需要"提前"获取——很多 Spring Bean 的初始化依赖配置中心的值，如果等到 application.yml 加载完成才开始读取配置中心，很多 Bean 可能已经用错误的配置初始化了。

### 4.2 bootstrap 与 application 的区别

| 维度 | bootstrap.yml | application.yml |
| --- | --- | --- |
| 加载顺序 | 先加载 | 后加载 |
| 用途 | 配置中心、服务注册等基础设施配置 | 应用自身的业务配置 |
| 优先级 | 低（会被 application 覆盖同名配置） | 高（但不能覆盖 bootstrap 中的基础设施配置） |
| 是否参与配置中心 | 是 | 是 |

### 4.3 配置的读取机制

当 Spring Boot 启动时，Nacos Config 会按照以下步骤加载配置：

```java
// 1. 构建 Nacos 配置服务
NacosConfigService configService = new NacosConfigService(properties);

// 2. 按优先级获取配置
String config = configService.getConfig(dataId, group, timeout);

// 3. 解析配置（支持 properties / yaml / xml / json）
Properties properties = ConfigFactory.parse(config);

// 4. 注入 Spring Environment
environment.getPropertySources().addFirst(properties);
```

## 五、共享配置与扩展配置

### 5.1 shared-dataids：共享配置

`shared-dataids` 用于定义**所有服务共享的配置**。这些配置通常包含：

- 日志配置
- 监控配置
- 通用工具类配置

```yaml title="bootstrap.yml"
spring:
  cloud:
    nacos:
      config:
        # 共享配置，所有服务都会加载
        shared-dataids: common.yaml,logging.yaml
        # 共享配置的可覆盖性（默认 true）
        # 如果设置为 false，shared-dataids 中的配置不允许被覆盖
        # allow-composite-mutation: false
```

### 5.2 extension-configs：扩展配置

`extension-configs` 用于定义**可选的扩展配置**，可以指定不同的 Group：

```yaml title="bootstrap.yml"
spring:
  cloud:
    nacos:
      config:
        extension-configs:
          # 第一个扩展配置
          - data-id: database.yaml
            group: INFRASTRUCTURE  # 指定 Group
            refresh: true  # 支持热更新
          # 第二个扩展配置
          - data-id: redis.yaml
            group: INFRASTRUCTURE
            refresh: true
```

`extension-configs` 的 `refresh: true` 非常关键——只有设置为 true 的配置，才支持热更新（配置变更后自动刷新）。

### 5.3 典型配置组合

一个典型的微服务 bootstrap.yml 配置：

```yaml title="bootstrap.yml"
spring:
  application:
    name: order-service
  profiles:
    active: prod
  cloud:
    nacos:
      config:
        server-addr: nacos-cluster.example.com:8848
        namespace: prod
        group: ORDER-SUIT
        
        # 加载顺序：
        # 1. order-service.yaml（默认配置）
        # 2. order-service-prod.yaml（环境配置）
        # 3. common.yaml（共享配置）
        # 4. extension-configs 中的配置
        file-extension: yaml
        
        shared-dataids: common.yaml
        
        extension-configs:
          - data-id: database.yaml
            group: INFRASTRUCTURE
            refresh: true
          - data-id: redis.yaml
            group: INFRASTRUCTURE
            refresh: true
```

## 六、Data ID 命名最佳实践

### 6.1 命名规范

```text
# 标准格式
{spring.application.name}-{profile}.{file-extension}

# 示例
order-service.yaml              # 默认配置
order-service-dev.yaml          # 开发环境
order-service-test.yaml         # 测试环境
order-service-prod.yaml         # 生产环境
```

### 6.2 推荐的配置分层

| 类型 | Data ID | 说明 | 加载时机 |
| --- | --- | --- | --- |
| 应用默认配置 | `order-service.yaml` | 应用的通用配置 | 所有环境 |
| 环境配置 | `order-service-{profile}.yaml` | 环境特定的配置 | 对应环境 |
| 共享基础设施配置 | `database.yaml` | 数据库通用配置 | 所有应用 |
| 共享业务配置 | `common.yaml` | 业务通用配置 | 所有应用 |

## 七、面试追问方向

- **Namespace、Group、Data ID 三者的优先级是什么？** 答：`Namespace > Group > Data ID`。更准确地说，Data ID 的配置优先级最高，会覆盖同名的 Group 和 Namespace 级别配置。
- **配置中心的配置是如何保证一致性的？** Nacos 使用 Raft 协议保证配置在集群节点间的一致性。配置变更时，只有超过半数节点确认后，才算成功。
- **shared-dataids 和 extension-configs 的区别是什么？** shared-dataids 是共享配置，所有服务都会加载，不支持指定 Group；extension-configs 是扩展配置，可以指定 Group 和是否支持热更新。
- **bootstrap.yml 和 application.yml 的区别是什么？** bootstrap.yml 在 application.yml 之前加载，用于配置中心、服务注册等基础设施配置。

---

## 八、总结

Nacos 配置中心的三层隔离机制，为微服务配置管理提供了清晰的组织结构：

- **Namespace**：环境隔离，dev / test / prod 各有各的空间
- **Group**：业务隔离，同一环境内按业务线或项目分组
- **Data ID**：配置隔离，每份配置都有唯一的标识

这种设计使得配置管理既能做到环境的严格分离，又能在同一环境内实现灵活的业务隔离。理解三层优先级的覆盖关系，是正确使用配置中心的关键。
