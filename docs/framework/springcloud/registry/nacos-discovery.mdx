# Nacos 服务注册与发现：微服务的第一步

import { Badge } from '@rspress/core/theme';

服务 A 要调用服务 B，但服务 B 有多个实例——怎么知道该调哪个？

这是微服务架构面临的第一个问题。传统的做法是"硬编码地址"，把服务 B 的 IP 和端口直接写在服务 A 的代码里。但这种方式有几个致命问题：

- 服务 B 扩容时，所有调用方都要改代码
- 服务 B 故障迁移时，所有调用方都要改代码
- 测试环境、预发环境、生产环境的地址完全不同，要维护三套代码

**Nacos 的服务注册与发现机制，正是为了解决这个问题。**

## 一、服务注册：应用启动的第一件事

### 1.1 自动注册原理

当 Spring Boot 应用引入了 `spring-cloud-starter-alibaba-nacos-discovery` 依赖后，应用启动时会自动完成服务注册。这个过程是 Spring Boot 自动配置的典型应用——不需要写一行代码，框架帮你做好了一切。

注册流程如下：

```java
// NacosAutoServiceRegistration 类负责处理注册逻辑
// 核心流程：应用启动 → 自动注册 → 心跳上报
```

1. **应用启动**：Spring Boot 容器初始化完成，触发 `ApplicationReadyEvent` 事件
2. **NacosAutoServiceRegistration**：监听该事件，调用 `register()` 方法
3. **构建 Instance 信息**：收集当前实例的 IP、端口、权重、元数据
4. **HTTP 请求**：向 Nacos Server 发送 POST 请求，注册服务实例
5. **心跳启动**：注册成功后，启动心跳任务定期上报健康状态

```java title="NacosAutoServiceRegistration.java 核心流程"
@Override
public void register(Registration registration) {
    if (StringUtils.isEmpty(registration.getServiceId())) {
        return;
    }
    // 1. 获取 Nacos Client 的命名服务
    NamingService namingService = namingService();
    // 2. 构建实例信息
    NamingUtils ...
    // 3. 注册实例
    namingService.registerInstance(registration.getServiceId(), instance);
}
```

### 1.2 @EnableDiscoveryClient 的作用

在 Spring Cloud 的旧版本中，需要显式使用 `@EnableDiscoveryClient` 注解来启用服务注册。但在 Spring Cloud Commons 2020.0 之后（即 Spring Cloud 2020 版本），**这个注解已经不再必需**。

```java
// Spring Cloud 2020 之前：需要显式标注
@SpringBootApplication
@EnableDiscoveryClient
public class Application {
    public static void main(String[] args) {
        SpringApplication.run(Application.class, args);
    }
}

// Spring Cloud 2020 之后：不再需要（自动装配）
@SpringBootApplication
public class Application {
    public static void main(String[] args) {
        SpringApplication.run(Application.class, args);
    }
}
```

这个变化背后的原因是：Spring Cloud 改变了服务注册的触发机制。在旧版本中，`@EnableDiscoveryClient` 会修改 classpath 检查逻辑，选择性地加载 Eureka 或 Consul 的自动配置类。在新版本中，所有服务注册组件的自动配置类都会被加载，然后通过 `spring.cloud.discovery.enabled` 配置来决定启用哪个。

### 1.3 注册信息的构建

当服务向 Nacos 注册时，会携带以下关键信息：

| 字段 | 说明 | 示例 |
| --- | --- | --- |
| serviceName | 服务名称（应用名） | `order-service` |
| ip | 实例 IP 地址 | `192.168.1.100` |
| port | 实例端口 | `8080` |
| clusterName | 所属集群 | `DEFAULT` |
| weight | 权重 | `1.0` |
| healthy | 健康状态 | `true` |
| enabled | 是否启用 | `true` |
| ephemeral | 临时实例标识 | `true` |
| metadata | 元数据（KV 对） | `{version: v1}` |

其中，`metadata` 是一个非常灵活的设计，可以存放任意的键值对信息，比如：

- 应用版本号
- 机房信息
- 灰度标识
- 自定义标签

这些元数据在服务发现时可以用来做**条件筛选**，比如只调用某个版本的服务实例。

## 二、心跳机制：健康的守护者

### 2.1 临时实例 vs 永久实例

Nacos 支持两种实例类型，这个区别对理解服务注册的行为至关重要：

| 类型 | ephemeral | 心跳机制 | 适用场景 |
| --- | --- | --- | --- |
| 临时实例 | `true` | 客户端主动心跳，Nacos 超过 15 秒未收到则标记为不健康，超过 30 秒则剔除 | 常见场景，如无状态服务 |
| 永久实例 | `false` | Nacos 服务端主动探测（TCP 或 HTTP），不主动剔除 | 服务端能探测的场景，或需要保留实例信息的场景 |

```yaml title="application.yml"
spring:
  cloud:
    nacos:
      discovery:
        # 默认是临时实例
        ephemeral: true  # true: 临时实例 false: 永久实例
```

临时实例的设计理念是：**客户端负责报告自己的存活状态**。如果客户端不再发送心跳，说明它已经"死"了，Nacos 应该清理掉这个实例。

永久实例的设计理念是：**Nacos 服务端负责探测实例的存活状态**。这种方式适用于 Nacos 无法依赖客户端心跳的场景，比如某些特殊协议的服务。

### 2.2 心跳的具体实现

对于临时实例，Nacos 客户端会启动一个定时任务，定期向服务端发送心跳：

```java
// 心跳任务的核心逻辑（简化）
public class BeatTask implements Runnable {
    @Override
    public void run() {
        // 1. 构建心跳请求
        BeatRequest beatRequest = new BeatRequest();
        beatRequest.setServiceName(serviceName);
        beatRequest.setIp(instanceIp);
        beatRequest.setPort(instancePort);
        beatRequest.setClusterName(clusterName);
        beatRequest.setPeriod(instance.getInstanceHeartInterval());
        
        // 2. 发送心跳
        boolean success = namingProxy.sendBeat(beatRequest);
        
        // 3. 如果失败，记录日志或触发重连
        if (!success) {
            // 心跳失败，可能是网络问题或服务端不可用
        }
    }
}
```

心跳间隔默认是 5 秒一次（Nacos 客户端常量 `NAMING_HEART_BEAT_INTERVAL`）。服务端如果超过 15 秒没收到某个实例的心跳，会将其标记为不健康；超过 30 秒没收到，则将其从服务列表中剔除。

### 2.3 健康检查的策略

Nacos 支持多种健康检查策略，这个是在 Nacos 服务端配置的：

| 检查策略 | 说明 | 原理 |
| --- | --- | --- |
| TCP | TCP 端口探测 | Nacos 向实例的注册端口发起 TCP 连接，能连上说明健康 |
| HTTP | HTTP 请求探测 | Nacos 向指定路径发送 GET 请求，2xx 响应码表示健康 |
| MySQL | MySQL 探测 | Nacos 执行一个 MySQL 命令，验证数据库连接 |
| 自定义 | 自定义 Agent | 通过自定义的 Agent 脚本判断健康状态 |

对于大多数 Spring Boot 应用，TCP 检查是最简单有效的——只要应用能正常监听端口，就说明它活着。

## 三、服务发现：调用方的寻址之旅

### 3.1 主动查询与订阅推送

服务发现有两种模式：**主动查询**和**订阅推送**。

**主动查询**：调用方主动向 Nacos Server 发起请求，获取目标服务的实例列表。这种方式简单直接，但每次调用都要查询，有额外的网络开销。

```java
// 主动查询示例
NamingService namingService = NacosFactory.createNamingService(properties);
List<Instance> instances = namingService.selectInstances("order-service", true);
```

**订阅推送**：调用方订阅某个服务，当该服务的实例列表发生变化时，Nacos Server 主动推送更新。这种方式更高效，但实现更复杂。

```java
// 订阅服务变更
namingService.subscribe("order-service", event -> {
    // 当 order-service 实例列表发生变化时，会触发这个回调
    List<Instance> instances = event.getInstances();
    // 更新本地的服务缓存
});
```

在实际使用中，Ribbon 和 OpenFeign 等客户端都采用了"订阅 + 缓存"的策略：订阅服务变更获得推送，同时在本地缓存一份实例列表，用于快速响应调用请求。

### 3.2 服务发现的核心代码

`NamingService.getAllInstances()` 是服务发现的核心方法：

```java
// 简化后的逻辑
public List<Instance> getAllInstances(String serviceName) {
    // 1. 从本地缓存获取
    ServiceInfo serviceInfo = serviceInfoHolder.getServiceInfoMap().get(serviceName);
    
    // 2. 如果缓存不存在或过期，向服务端查询
    if (serviceInfo == null || !cacheService.isValidService(serviceInfo)) {
        serviceInfo = queryService(serviceName);
    }
    
    // 3. 返回健康的实例
    return selectInstances(serviceInfo, true);
}
```

这里的"缓存 + 查询"的策略，平衡了性能和实时性。本地缓存保证了调用链路的低延迟，而定期刷新保证了数据的最终一致性。

### 3.3 负载均衡策略

获取到实例列表后，接下来要选择一个具体的实例。Nacos 提供了多种负载均衡策略：

| 策略 | 实现类 | 说明 |
| --- | --- | --- |
| 权重 | `NacosWeightedRandom` | 根据权重值随机选择，权重越高被选中的概率越大 |
| 轮询 | `NacosRoundRobinRule` | 按顺序逐个选择 |
| 最少连接 | `NacosLeastActiveRule` | 选择当前活跃请求数最少的实例 |
| 一致性哈希 | `NacosConsistentHashLBRule` | 相同参数的请求路由到同一实例 |

默认的负载均衡策略是**权重随机**（`NacosWeightedRandom`），如果没有配置权重，所有实例的权重相同，实际效果就是**加权随机**。

```yaml title="application.yml"
spring:
  cloud:
    nacos:
      discovery:
        # 配置负载均衡权重
        weight: 1.0
        # 配置集群名
        cluster-name: HZ
```

## 四、客户端调用：@LoadBalanced + RestTemplate / OpenFeign

### 4.1 @LoadBalanced + RestTemplate

在 Spring Cloud 中，最简单的服务调用方式是使用 `@LoadBalanced` 注解修饰 RestTemplate：

```java
@Configuration
public class RestTemplateConfig {
    
    @Bean
    @LoadBalanced  // 开启负载均衡
    public RestTemplate restTemplate() {
        return new RestTemplate();
    }
}
```

`@LoadBalanced` 的作用是向 RestTemplate 中注入一个 `LoadBalancerInterceptor`，这个拦截器会在发送请求之前，拦截请求 URL 中的"服务名"，将其替换为具体的实例地址：

```java
// LoadBalancerInterceptor 的核心逻辑
@Override
public ClientHttpResponse intercept(final HttpRequest request, final byte[] body,
        final ClientHttpRequestExecution execution) throws IOException {
    // 1. 提取 URL 中的服务名
    URI originalUri = request.getURI();
    String serviceName = originalUri.getHost();
    
    // 2. 根据服务名查询实例列表
    ServiceInstance serviceInstance = loadBalancer(serviceName);
    
    // 3. 替换服务名为实际地址
    URI newUri = UriComponentsBuilder.fromUri(request.getURI())
            .host(serviceInstance.getHost())
            .port(serviceInstance.getPort())
            .build()
            .toUri();
    
    // 4. 继续执行请求
    return execution.execute(request, body);
}
```

所以下面的代码：

```java
restTemplate.getForObject("http://order-service/api/orders", String.class);
```

实际会被解析为类似：

```java
restTemplate.getForObject("http://192.168.1.100:8080/api/orders", String.class);
```

### 4.2 OpenFeign 的服务调用

相比 RestTemplate，OpenFeign 的写法更加声明式，也更符合"像调用本地方法一样调用远程服务"的理念：

```java
@FeignClient(name = "order-service")
public interface OrderClient {
    
    @GetMapping("/api/orders")
    List<Order> getOrders();
}

// 使用
@Service
public class OrderServiceImpl {
    
    @Autowired
    private OrderClient orderClient;
    
    public List<Order> findOrders() {
        return orderClient.getOrders();  // 像调用本地方法一样
    }
}
```

OpenFeign 底层集成了 Ribbon，会自动从 Nacos 获取服务实例列表，并完成负载均衡。开发者不需要关心服务发现的细节，只需要定义好接口即可。

## 五、集群环境下的服务注册

### 5.1 集群节点配置

在生产环境中，Nacos 通常以集群模式部署，客户端需要知道多个 Nacos Server 的地址：

```yaml title="application.yml"
spring:
  cloud:
    nacos:
      discovery:
        server-addr: nacos-server-1:8848,nacos-server-2:8848,nacos-server-3:8848
        # 或者使用域名（推荐，便于后续运维）
        # server-addr: nacos-cluster.example.com:8848
```

Nacos 客户端会从这份列表中随机选一个进行注册，如果某个节点不可用，会自动切换到其他节点。这种设计保证了服务注册的高可用。

### 5.2 注册到指定集群

如果你的 Nacos 集群按机房或区域划分，可以通过 `cluster-name` 配置让服务注册到指定集群：

```yaml title="application.yml"
spring:
  cloud:
    nacos:
      discovery:
        server-addr: nacos-cluster.example.com:8848
        cluster-name: HZ  # 注册到杭州集群
```

然后在调用时，可以配置**同集群优先**的路由策略：

```yaml title="application.yml"
spring:
  cloud:
    nacos:
      discovery:
        # 优先调用同集群的实例
        enabled: true
```

同集群优先的好处是：跨机房调用延迟更低，而且可以避免不必要的跨机房流量。

## 六、spring.cloud.nacos.discovery 的常用配置

下面是一些在生产环境中经常会用到的配置项：

```yaml title="application.yml"
spring:
  cloud:
    nacos:
      discovery:
        # Nacos Server 地址
        server-addr: 127.0.0.1:8848
        
        # 命名空间（用于环境隔离）
        namespace: dev
        
        # 集群名称
        cluster-name: DEFAULT
        
        # 实例权重
        weight: 1.0
        
        # 是否注册到 Nacos
        enabled: true
        
        # 是否使用 Nacos 注册中心
        register-enabled: true
        
        # 元数据
        metadata:
          version: v1
          region: hangzhou
        
        # 心跳间隔（毫秒）
        heart-beat-interval: 5000
        
        # 心跳超时时间（毫秒）
        heart-beat-timeout: 15000
        
        # 多少秒后删除不健康的实例
        ip-delete-timeout: 30000
```

| 配置项 | 默认值 | 说明 |
| --- | --- | --- |
| `server-addr` | `127.0.0.1:8848` | Nacos Server 地址 |
| `namespace` | 空字符串（public） | 命名空间 ID |
| `cluster-name` | `DEFAULT` | 集群名称 |
| `weight` | `1.0` | 实例权重 |
| `register-enabled` | `true` | 是否注册到 Nacos |
| `heart-beat-interval` | `5000` | 心跳间隔（毫秒） |
| `heart-beat-timeout` | `15000` | 心跳超时（毫秒） |

## 七、面试追问方向

- **Nacos 的注册表数据结构是怎样的？** 注册表本质是一个 `Map<serviceName, Service>` 的结构，每个 Service 包含多个 Instance。
- **Nacos 是如何实现高可用的？** 客户端配置多个 Server 地址，失败时自动切换；服务端通过 Raft 协议保证一致性。
- **服务下线时 Nacos 能否及时感知？** 临时实例通过心跳机制感知（15 秒不健康，30 秒剔除）；永久实例通过健康检查感知。
- **如果 Nacos Server 全部宕机，服务还能调用吗？** 可以，因为 Ribbon/Feign 会在本地缓存实例列表，只是无法感知新的实例变化。

## 八、总结

Nacos 的服务注册与发现机制，是整个 Spring Cloud Alibaba 生态的基础。通过这一机制：

- **服务提供方**不需要关心调用方是谁，只需要告诉 Nacos"我在这里"
- **服务调用方**不需要关心提供方有几个实例，只需要问 Nacos"这个服务在哪"
- **Nacos** 作为中间人，维护着一份动态的、可用的服务清单

理解这个"注册 → 心跳 → 发现 → 负载均衡"的完整链路，是掌握 Spring Cloud Alibaba 的第一步。
