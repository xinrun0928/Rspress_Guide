# Nacos 热更新：配置变了，代码不用动

改了个配置，还要重启应用？

这是传统配置管理最大的痛点。在过去，修改一个配置（比如数据库连接池大小）的流程是：修改配置文件 → 登录服务器 → 重启应用。整个过程可能需要 10 分钟，而这 10 分钟里，服务是不可用的。

**Nacos 的热更新机制，让配置变更在秒级生效，而不需要重启任何服务。**

## 一、@RefreshScope 的工作原理

### 1.1 为什么需要 @RefreshScope

Spring Boot 的 Bean 在创建时会"固化"配置值。以 `@Value` 为例：

```java
@Component
public class PaymentService {
    
    @Value("${payment.timeout:5000}")
    private int timeout;
    
    public void pay() {
        // timeout 在 Bean 创建时就已经固化了
        // 即使 Nacos 中的配置变了，这里的 timeout 也不会变
    }
}
```

如果不加任何处理，Bean 一旦创建完成，它持有的配置值就"锁死"了。要让配置变化生效，必须重新创建 Bean 实例。

`@RefreshScope` 就是来解决这个问题的——它让 Bean 能够在配置变化时重新创建。

### 1.2 @RefreshScope 的实现机制

`@RefreshScope` 的原理可以概括为：**配置变化时，销毁旧 Bean，创建新 Bean**。

```java
// @RefreshScope 的简化实现逻辑
@Scope(scopeName = "refresh", proxyMode = ScopedProxyMode.TARGET_CLASS)
public @interface RefreshScope {
    // 底层依赖 refresh 作用域
}
```

核心机制：

1. **Scope 扩展**：Spring 新增了一个 `refresh` 作用域，区别于默认的 `singleton` 和 `prototype`
2. **代理模式**：每个 `@RefreshScope` Bean 实际上是一个**代理对象**，内部持有对真正 Bean 的引用
3. **配置刷新时**：代理指向的目标 Bean 被销毁，新的 Bean 被创建，代理指向新的目标

```java
// 简化后的刷新逻辑
public class RefreshScopeBeanCreator {
    
    public Object createBean(String beanName, BeanDefinition bd) {
        // 1. 创建增强的 BeanPostProcessor
        // 2. 当配置变化时，这个 Processor 会收到通知
        // 3. 通知到达时，清除本地缓存
        // 4. 再次访问这个 Bean 时，从头创建（使用新配置）
    }
}
```

### 1.3 @RefreshScope 的使用方式

只需要在需要热更新的 Bean 上添加注解：

```java
@RefreshScope
@Configuration
public class NacosConfig {
    
    @Value("${payment.timeout:5000}")
    private int timeout;
    
    @Bean
    public RestTemplate restTemplate() {
        return new RestTemplateBuilder()
                .setConnectTimeout(timeout)  // 这个值会热更新
                .build();
    }
}
```

> [!WARNING]
> `@RefreshScope` 有一个重要的副作用：**每次刷新都会创建新的 Bean 实例**。如果你的 Bean 中有非线程安全的操作（比如持有可变状态），要小心刷新时可能产生的并发问题。

## 二、ContextRefresher 与 EnvironmentChangeEvent

### 2.1 完整的刷新链路

配置热更新的完整链路是这样的：

```
1. 用户在 Nacos 控制台修改配置
        ↓
2. Nacos Server 主动推送给客户端（长轮询 / HTTP Long Polling）
        ↓
3. Nacos Config Client 收到推送，调用 ConfigListener
        ↓
4. ContextRefresher.refreshEnvironment() 更新 Spring Environment
        ↓
5. Spring 发布 EnvironmentChangeEvent 事件
        ↓
6. RefreshScope 清空缓存的 Bean
        ↓
7. 下次访问 @RefreshScope Bean 时，创建新实例
```

### 2.2 关键组件：ContextRefresher

`ContextRefresher` 是 Spring Cloud 提供的刷新上下文的工具类：

```java
// ContextRefresher 的核心方法
public class ContextRefresher {
    
    private ConfigurableApplicationContext context;
    private AtomicInteger refreshCount = new AtomicInteger(0);
    
    public synchronized Set<String> refreshEnvironment() {
        // 1. 刷新 Spring Environment
        Map<String, Object> before = extract(
            this.context.getEnvironment().getPropertySources());
        
        // 2. 重新加载配置
        this.context.getEnvironment().getPropertySources().addFirst(...);
        
        Map<String, Object> after = extract(
            this.context.getEnvironment().getPropertySources());
        
        // 3. 找出变化的配置项
        Set<String> keys = changedKeys(before, after);
        
        // 4. 发布事件
        this.context.publishEvent(new EnvironmentChangeEvent(keys));
        
        return keys;
    }
    
    public synchronized Set<String> refresh() {
        // 刷新 Environment
        Set<String> keys = refreshEnvironment();
        
        // 清空 @RefreshScope 缓存
        refreshScope.refreshAll();
        
        return keys;
    }
}
```

### 2.3 EnvironmentChangeEvent 的发布

`EnvironmentChangeEvent` 包含变化的配置项列表：

```java
// 当配置变化时，Spring 会发布这个事件
applicationContext.publishEvent(new EnvironmentChangeEvent(changedKeys));

// 任何 Spring Bean 都可以监听这个事件
@EventListener
public void onEnvironmentChange(EnvironmentChangeEvent event) {
    Set<String> keys = event.getKeys();
    System.out.println("配置发生了变化：" + keys);
    // 执行业务逻辑，比如关闭旧连接、刷新缓存等
}
```

## 三、@ConfigurationProperties 的热更新

### 3.1 @ConfigurationProperties 的优势

相比 `@Value`，`@ConfigurationProperties` 更适合配置对象的绑定：

```java
@RefreshScope
@ConfigurationProperties(prefix = "payment")
public class PaymentProperties {
    
    private int timeout = 5000;
    private String gatewayUrl;
    private Map<String, String> channels;
    
    // getters and setters
}
```

```yaml title="Nacos 配置"
payment:
  timeout: 5000
  gateway-url: https://pay.example.com
  channels:
    alipay: https://alipay.example.com
    wechat: https://wechatpay.example.com
```

### 3.2 @ConfigurationProperties 的刷新机制

`@ConfigurationProperties` 与 `@RefreshScope` 结合时，会自动刷新绑定值：

```java
@RefreshScope
@ConfigurationProperties(prefix = "payment")
public class PaymentProperties {
    
    private int timeout = 5000;
    
    public int getTimeout() {
        return timeout;
    }
    
    public void setTimeout(int timeout) {
        this.timeout = timeout;
    }
}

// 使用方
@RefreshScope
@Service
public class PaymentService {
    
    @Autowired
    private PaymentProperties properties;  // properties 本身会刷新
    
    public void pay() {
        // properties.getTimeout() 会返回最新的值
        int timeout = properties.getTimeout();
    }
}
```

## 四、@Value 的刷新限制

### 4.1 @Value 为什么不能自动刷新

`@Value` 的一个重要限制是：**默认情况下不支持自动刷新，必须配合 @RefreshScope 使用**。

```java
@RefreshScope
public class MyService {
    
    @Value("${payment.timeout:5000}")
    private int timeout;
    
    public void doSomething() {
        // 当 @RefreshScope 触发刷新时，这个 timeout 会更新
    }
}
```

如果不加 `@RefreshScope`，即使 Nacos 配置变了，这里的 `timeout` 也不会变。

### 4.2 @Value 与 @ConfigurationProperties 的对比

| 维度 | @Value | @ConfigurationProperties |
| --- | --- | --- |
| 批量绑定 | 不支持，需要逐个字段写 | 支持，一次性绑定整个对象 |
| 类型转换 | 自动转换 | 自动转换 |
| 热更新 | 需要配合 @RefreshScope | 需要配合 @RefreshScope |
| 默认值 | 支持 `${key:default}` | 支持字段默认值 |
| 校验 | 不支持 | 支持 @Validated |
| 松散绑定 | 不支持 | 支持（camelCase 可匹配 snake_case） |

## 五、Spring Cloud Bus 的集群广播

### 5.1 为什么需要 Spring Cloud Bus

在单实例场景下，Nacos 的配置推送直接送达应用，一切都很好。但如果你的服务有 10 个实例呢？

```
Nacos Server
    ├──→ 应用实例 1：收到配置变更
    ├──→ 应用实例 2：收到配置变更
    ├──→ 应用实例 3：收到配置变更
    ├──→ ...
    └──→ 应用实例 10：收到配置变更
```

Nacos 本身的配置推送是**单点推送**——每个实例都会收到。但问题是：如果有 100 个实例，或者 Nacos Server 重启了，哪些实例已经刷新了？哪些还没有？

Spring Cloud Bus 的作用是：**在应用集群内部广播刷新事件**，确保所有实例同时刷新。

### 5.2 Spring Cloud Bus 的架构

```
用户 → Nacos 控制台 → Nacos Server → 触发配置变更
                                    ↓
                          Nacos Config Client
                                    ↓
                          消息队列（RabbitMQ / Kafka）
                                    ↓
                    ┌─────────┬─────────┬─────────┐
                    ↓         ↓         ↓         ↓
              应用实例1  应用实例2  应用实例3  应用实例4
```

流程：

1. 任意一个应用实例收到 Nacos 的配置变更推送
2. 该实例向消息队列发布 `RefreshRemoteApplicationEvent` 事件
3. 所有应用实例订阅消息队列，收到事件
4. 所有实例同时执行刷新逻辑

### 5.3 Spring Cloud Bus 的配置

```yaml title="application.yml"
spring:
  cloud:
    bus:
      # 开启总线追踪
      trace:
        enabled: true
    stream:
      # RabbitMQ 绑定
      rabbit:
        binder:
          nodes: rabbitmq-server:5672
```

然后通过 actuator 端点触发刷新：

```bash
# 刷新所有实例
curl -X POST http://localhost:8080/actuator/bus-refresh

# 刷新指定服务（定向刷新）
curl -X POST http://localhost:8080/actuator/bus-refresh/order-service:prod

# 刷新指定实例
curl -X POST http://localhost:8080/actuator/bus-refresh/order-service:prod:192.168.1.100:8080
```

## 六、Nacos 原生的配置监听

### 6.1 @NacosConfigurationProperties

Nacos 提供了原生的配置监听注解，不依赖 Spring Cloud：

```java
@NacosConfigurationProperties(prefix = "payment", autoRefreshed = true)
public class PaymentProperties {
    
    private int timeout = 5000;
    
    // getters and setters
}
```

`autoRefreshed = true` 使得配置变更时自动刷新，无需手动触发。

### 6.2 @NacosValue

Nacos 还提供了 `@NacosValue` 注解：

```java
@NacosValue(value = "${payment.timeout:5000}", autoRefreshed = true)
private int timeout;
```

相比 Spring 的 `@Value`，`@NacosValue` 原生支持 `autoRefreshed` 属性。

### 6.3 Nacos 原生监听器

更底层的方式是直接注册监听器：

```java
@NacosInjected
private ConfigService configService;

public void init() {
    configService.addListener("payment.properties", "DEFAULT_GROUP", 
        new Listener() {
            @Override
            public void receiveConfigInfo(String configInfo) {
                // 配置变化时，这里会被调用
                System.out.println("配置变了：" + configInfo);
            }
            
            @Override
            public Executor getExecutor() {
                return Executors.newSingleThreadExecutor();
            }
        });
}
```

## 七、动态数据源切换

### 7.1 典型场景

热更新最典型的应用场景之一是**动态数据源切换**：

```java
@RefreshScope
@Configuration
public class DataSourceConfig {
    
    @Value("${spring.datasource.url}")
    private String url;
    
    @Value("${spring.datasource.username}")
    private String username;
    
    @Value("${spring.datasource.password}")
    private String password;
    
    @Bean
    public DataSource dataSource() {
        // 当配置变化时，这个 DataSource 会被重新创建
        HikariDataSource dataSource = new HikariDataSource();
        dataSource.setJdbcUrl(url);
        dataSource.setUsername(username);
        dataSource.setPassword(password);
        return dataSource;
    }
}
```

### 7.2 ShardingSphere + Nacos 的动态数据源

在分库分表场景下，配合 ShardingSphere 可以实现更复杂的动态切换：

```java
@RefreshScope
@Configuration
public class ShardingSphereDataSourceConfig {
    
    @Autowired
    private NacosConfigProperties nacosConfig;
    
    @Bean
    @Primary
    public DataSource dataSource() {
        // 读取 Nacos 配置，构建 ShardingSphere DataSource
        return DataSourceBuilder.create()
                .type(ShardingSphereDataSource.class)
                .build();
    }
}
```

## 八、面试追问方向

- **@RefreshScope 的原理是什么？** 答：基于 Spring 的自定义 Scope，实现了一个 refresh 作用域。配置变化时，清空 Bean 缓存，下次访问时创建新实例。
- **@Value 为什么不能自动刷新？** 答：@Value 注入的值在 Bean 创建时就固化了，必须配合 @RefreshScope 才能热更新。
- **Spring Cloud Bus 的作用是什么？** 答：实现集群内配置刷新的广播，确保所有实例同时生效。
- **Nacos 的配置推送机制是什么？** 答：基于 HTTP 长轮询（Long Polling），客户端定时向服务端拉取配置，服务端在配置变化时立即返回。

---

## 九、总结

Nacos 的热更新机制，让配置管理真正做到了"改完即生效"：

- **@RefreshScope**：让 Bean 在配置变化时重新创建
- **ContextRefresher**：负责刷新 Spring Environment
- **EnvironmentChangeEvent**：通知所有监听者配置发生了变化
- **Spring Cloud Bus**：在集群内广播刷新事件，确保所有实例同步

这套机制使得微服务的配置管理从"重启生效"进化到了"秒级生效"，大大提升了系统的灵活性和运维效率。
