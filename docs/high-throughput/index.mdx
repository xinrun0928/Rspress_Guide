# 高并发高性能高可用：Java 后端工程师的技术脊梁

凌晨两点，你被一条告警叫醒：「线上服务响应超时，错误率突破 5%。」

你打开监控大盘，看到 QPS 曲线正以近乎垂直的角度飙升，数据库连接池已耗尽，Redis 开始拒绝请求，系统正在滑向崩溃的边缘。

这是每一个后端工程师都可能经历的夜晚。而那些在类似场景中依然能让系统稳如泰山的工程师，他们靠的不是运气，是**三高架构的扎实内功**——高性能、高并发、高可用。

高性能让你知道**怎么让系统跑得更快**，高并发让你掌握**怎么让系统同时服务更多人**，高可用让你明白**怎么让系统在故障中依然存活**。三者缺一不可，共同构成 Java 后端工程师的技术脊梁。

## 一、三高的关系：不是选择题，而是都要会

很多人有一个误解：「高并发、高性能、高可用是三个独立的方向，可以挑一个深入。」

这个理解是错的。**三者相互关联、相互支撑，共同决定了一个系统的技术水位。**

```
高性能（能跑多快）
    ↑
    │  基础能力
    │
高并发（能扛多少）
    ↑
    │  场景压力
    │
高可用（挂了怎么办）
    ↓
    兜底能力
```

- **高性能是基础**：系统连「快」都做不到，高并发就没有意义
- **高并发是场景**：系统能「快」，但同时 10 万人访问会怎样？
- **高可用是底线**：系统能「快」能「扛」，但挂了怎么办？

三高的关系就像一座大楼的**地基、楼层、消防设施**——你可以不盖摩天大楼，但不能不装消防设施。

## 二、高性能优化：让系统跑得更快

高性能优化的核心指标有两个：**RT（响应时间）** 和 **QPS/TPS（吞吐量）**。

- **RT 关注的是「快不快」**：用户点一个按钮，1 秒响应和 5 秒响应，体验天差地别
- **QPS 关注的是「多不多」**：同样的硬件条件，QPS 越高说明系统处理效率越高

高性能优化的关键原则是**木桶原理**：系统的性能瓶颈永远在最短的那块板上。你花了大量精力优化 Redis，但 MySQL 只能扛 1 万 QPS，那整个系统的上限就是 1 万 QPS。**找准瓶颈，比优化重要。**

本模块覆盖的内容：

- **性能优化方法论**：木桶原理、测量驱动、瓶颈定位、性能优化清单
- **数据库性能优化**：SQL 优化（慢查询、EXPLAIN、JOIN、分页）、MySQL 调优（连接池、MVCC）、Redis 优化（大 key、热 key、持久化）
- **JVM 性能调优**：GC 算法（CMS、G1、ZGC）、GC 日志分析、OOM 排查、Arthas 实战
- **IO 模型与零拷贝**：BIO/NIO/AIO、IO 多路复用（epoll）、零拷贝（mmap/SendFile）、Netty
- **池化技术**：线程池、数据库连接池、Redis 连接池、HTTP 连接池
- **接口优化**：序列化选型（JSON/Protobuf/Kryo）、接口幂等性、HTTP Keep-Alive、响应压缩

## 三、高并发设计：让系统扛住更大的压力

高并发不是「并发编程」，而是**在真实流量压力下，系统如何保证稳定运行**。

高并发的核心矛盾是：**资源有限，需求无限。** CPU、内存、磁盘、网络带宽都是有限的，但用户的请求是无限的。怎么在有限资源下服务更多请求，是高并发要解决的核心问题。

本模块覆盖的内容：

- **并发基础优化**：线程池调优、锁升级、CAS 原理、分段锁、ThreadLocal 泄漏、虚拟线程
- **缓存架构**：多级缓存（本地缓存/Caffeine、分布式缓存/Redis）、缓存策略（Cache Aside、Write Through）、缓存问题（穿透/击穿/雪崩）、缓存一致性
- **异步与消息处理**：Spring @Async、CompletableFuture、WebFlux、Kafka 性能优化、消息队列选型
- **分布式高并发核心**：分布式锁（Redis/ZooKeeper/etcd）、分布式事务（Seata/可靠消息）、分布式 ID（雪花算法）

## 四、高可用架构：让系统活得更久

高可用回答的问题是：**当系统出现故障时，如何保证服务依然可用？**

高可用的核心手段是**冗余 + 切换**——当一个组件挂了，立即切换到备用组件。但切换需要时间，这段时间怎么办？这就需要**降级、限流、熔断**等手段来保护系统。

高可用还有一个关键指标：**几个 9**。

| 可用性 | 每年停机时间 | 典型场景 |
|---|---|---|
| 99%（2个9） | 3.65 天 | 个人博客、内部系统 |
| 99.9%（3个9） | 8.76 小时 | 普通企业应用 |
| 99.99%（4个9） | 52 分钟 | 主流互联网应用 |
| 99.999%（5个9） | 5.26 分钟 | 金融支付、电信核心 |

每提升一个 9，难度是指数级增长。

本模块覆盖的内容：

- **负载均衡**：L4/L7 负载均衡、DNS 负载均衡、Nginx 负载均衡、客户端负载均衡、健康检查
- **限流与熔断**：限流算法（令牌桶/漏桶/滑动窗口）、Sentinel 流量控制、断路器模式、舱壁隔离
- **降级与容错**：服务降级、重试机制、超时处理、幂等性、多级降级、自适应限流
- **高可用架构设计**：故障转移（Failover）、多活架构（同城双活/两地三中心）、灰度发布、备份恢复
- **分布式高可用核心**：分布式协调服务（ZooKeeper/etcd/Nacos）、微服务高可用、网关高可用、全链路监控

## 五、面试专项：把知识变成面试战斗力

三高知识学了，但面试中怎么考？这就是面试专项模块存在的意义。

面试专项分为三个方向：

- **架构设计题汇总**：系统设计思维、分布式系统设计原则、高可用/高并发/高性能设计框架
- **高频场景实战**：秒杀系统、Feed 流系统、短链系统、高并发计数器、搜索系统
- **典型故障排查**：OOM 排查、CPU 高占用、GC 停顿、消息积压、网络延迟

面试专项的核心不是背答案，而是建立**从问题到方案**的思维能力。

## 六、学习路线建议

### 入门阶段（1-2 个月）

先掌握高性能优化的核心：

- MySQL 索引原理与 SQL 优化
- Redis 缓存设计与缓存问题
- JVM 内存模型与 GC 基础
- 线程池原理与参数配置

### 进阶阶段（2-3 个月）

深入高并发和高可用：

- 分布式锁、分布式事务
- 限流、熔断、降级
- 负载均衡架构
- 消息队列应用

### 实战阶段（持续）

结合面试场景深化理解：

- 系统设计题练习
- 故障排查能力
- 项目经验的深度梳理

## 七、模块核心公式

```
高性能 = 低 RT + 高 QPS + 低错误率

高并发 = 并发控制 + 缓存 + 异步 + 池化

高可用 = 冗余 + 切换 + 降级 + 熔断 + 限流

三高综合 = 性能根基 × 并发支撑 × 可用底线
```
