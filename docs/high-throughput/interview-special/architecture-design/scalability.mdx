# 可扩展性设计：水平扩展与垂直扩展

凌晨四点，你的手机响了。运营同学发来一条消息：「双十一活动提前开始了，流量是平时的 20 倍！」

你睡眼惺忪地打开监控大屏，看到服务器 CPU 打满、响应时间飙升、错误率飞涨。你紧急联系运维，得到的答复是：「机器都满了，加不了机器。」

这是每一个快速增长的产品都会遇到的问题：系统撑不住了。

但如果你的系统从一开始就设计好了扩展性，这个问题就不会发生。你只需要点点鼠标或者敲几行命令，系统就能自动扩容，从容应对流量洪峰。

可扩展性，是区分「能用」和「好用」系统的关键指标。这篇文章，我会从可扩展性的本质讲起，带你掌握水平扩展与垂直扩展的核心原理，最后给出实战中的设计模式与避坑指南。

## 一、可扩展性的本质：加机器能解决问题吗？

### 1.1 什么是可扩展性

很多人对可扩展性有一个误解：以为可扩展性就是「能不能加机器」。加机器能解决问题，就说系统可扩展；加机器解决不了，就说系统不可扩展。

这个理解只对了一半。

**真正的可扩展性，是系统应对负载增长的能力**。负载增长可能来自用户量增加、数据量增长、功能复杂度提升等多个方面。系统能否通过增加资源（机器、带宽、存储）来应对这些增长，是衡量可扩展性的核心标准。

但「加机器」这件事，并没有听起来那么简单。

### 1.2 加机器为什么不一定能解决问题

假设你的系统是一个简单的单体应用，部署在一台 4 核 8G 的服务器上。某天流量暴涨，你加了一台同样的服务器。

问题来了：**这两台服务器如何协作？**

- 两台服务器的请求如何分发？需要负载均衡器
- 两台服务器的数据如何同步？需要共享存储或者数据复制
- 两台服务器的 session 如何管理？需要外部 session 存储
- 如果数据量太大，一台数据库服务器放不下怎么办？需要分库分表

每加一台服务器，都会引入新的复杂性。如果这些复杂性没有提前考虑，加机器不仅不能解决问题，反而可能引入更多问题。

:::warning 扩展性设计的核心矛盾

扩展性设计的核心矛盾是：**单机性能 vs 分布式复杂性**。

单机性能再好，也有上限。分布式可以突破单机上限，但会引入协调复杂性：数据一致性、故障检测、服务发现、网络分区……

好的扩展性设计，是在「单机性能」和「分布式复杂性」之间找到平衡点——既能通过加机器突破瓶颈，又能控制分布式带来的复杂性。

:::

### 1.3 扩展性的三个层次

可扩展性可以从三个层次来理解：

| 层次 | 描述 | 举例 |
|---|---|---|
| **垂直扩展（Scale Up）** | 增加单机的硬件资源 | CPU 升级、内存扩大、磁盘换成 SSD |
| **水平扩展（Scale Out）** | 增加机器数量 | 从 1 台扩展到 10 台、100 台 |
| **功能扩展（Scale，功能拆分）** | 通过拆分功能来提升系统能力 | 微服务拆分、模块化设计 |

垂直扩展和水平扩展是资源层面的扩展，功能扩展是架构层面的扩展。三者往往需要配合使用。

## 二、垂直扩展：简单粗暴，但有上限

### 2.1 垂直扩展的原理

垂直扩展（Vertical Scaling）是最简单直接的扩展方式：给单机增加更多的 CPU、内存、磁盘、网络带宽。

垂直扩展的好处是：

- **实现简单**：不需要修改应用代码，只需要升级硬件
- **兼容性良好**：单机架构下，所有组件都在同一台机器上，不存在分布式问题
- **性能提升显著**：硬件升级往往能带来 2-4 倍的性能提升

垂直扩展的典型场景：

- 数据库服务器：从 16 核 64G 升级到 64 核 256G
- 内存数据库：从 32G 升级到 128G，缓存命中率大幅提升
- 存储服务器：从机械盘换成 NVMe SSD，IOPS 从几千提升到几十万

### 2.2 垂直扩展的局限

垂直扩展听起来很美好，但有三个致命的局限：

**第一，成本非线性增长**

硬件性能不是线性提升的。CPU 从 4 核升级到 8 核，价格可能是 2 倍；但从 64 核升级到 128 核，价格可能是 4 倍甚至更多。

| CPU 核心数 | 参考价格（相对） | 每核成本（相对） |
|---|---|---|
| 4 核 | 1x | 1x |
| 8 核 | 1.8x | 0.9x |
| 16 核 | 4x | 1x |
| 32 核 | 10x | 1.25x |
| 64 核 | 30x | 1.88x |

可以看到，从 4 核到 64 核，每核成本从 1x 涨到 1.88x。这就是垂直扩展的经济学困境。

**第二，存在物理上限**

即使你有钱，也买不到「无限大」的机器。目前消费级服务器的最大配置大约是 256 核 2TB 内存，而企业级服务器的最大配置也有限。对于互联网公司动不动就需要几千核的场景，垂直扩展根本不可能满足。

**第三，单点故障**

垂直扩展的核心问题是：**所有资源都在一台机器上**。这台机器挂了，整个系统就挂了。这违背了高可用的基本原则。

:::tip 垂直扩展的适用场景

垂直扩展最适合以下场景：

- 系统早期，用户量不大，单机性能足够
- 需要快速上线，来不及做分布式改造
- 系统瓶颈明确在单机资源（CPU、内存、IO），且升级成本可接受
- 作为水平扩展的过渡方案

:::

### 2.3 垂直扩展的实战建议

即使你的系统最终要做水平扩展，垂直扩展仍然是重要的基础。以下是实战中的建议：

**适度超配，留有余量**

采购服务器时，不要刚好满足当前需求。建议预留 50-100% 的余量，给未来的增长留出空间。

**分离关注点，为水平扩展做准备**

即使当前是单机部署，也要做好架构上的分离：

- 应用和数据库分离（不用同一台机器）
- 读写分离（主库负责写，从库负责读）
- Session 外部化（不用本地内存存储 Session）

这样，当需要水平扩展时，改动最小。

## 三、水平扩展：突破单机上限

### 3.1 水平扩展的原理

水平扩展（Horizontal Scaling）通过增加机器数量来提升系统能力。与垂直扩展相比，水平扩展的核心优势是：

- **理论上无限扩展**：只要加机器，系统能力就能提升
- **成本线性增长**：机器数量和系统能力成线性关系
- **天然高可用**：多台机器互为备份，单点故障不影响整体

水平扩展的典型架构：

```bash
                    ┌─────────────────┐
                    │   负载均衡器     │
                    │  (Nginx/HAProxy)│
                    └────────┬────────┘
                             │
         ┌───────────────────┼───────────────────┐
         │                   │                   │
         ▼                   ▼                   ▼
   ┌──────────┐        ┌──────────┐        ┌──────────┐
   │ 应用服务器│        │ 应用服务器│        │ 应用服务器│
   │   Node 1  │        │   Node 2  │        │   Node 3  │
   └──────────┘        └──────────┘        └──────────┘
```

### 3.2 无状态服务：水平扩展的前提

水平扩展的前提是：**服务无状态**。

如果服务是有状态的（比如 Session 存储在本地内存），那么加机器就意味着数据不一致。用户请求 A 服务器登录，Session 存在 A 的内存里；下一次请求被路由到 B 服务器，B 服务器没有这个 Session，用户就「掉线」了。

无状态服务的特点是：**每次请求包含所有必要信息，服务本身不存储请求相关的数据**。

实现无状态服务的关键：

**1. Session 外部化**

不要把 Session 存在应用服务器的内存里，而是存在外部存储中：

```java
// 错误写法：Session 存在本地内存
public class UserService {
    private Map<String, Session> sessionMap = new HashMap<>();
}

// 正确写法：Session 存在 Redis
public class UserService {
    @Autowired
    private RedisTemplate<String, Session> redisTemplate;
    
    public void saveSession(String userId, Session session) {
        redisTemplate.opsForValue().set("session:" + userId, session);
    }
}
```

**2. 请求包含认证信息**

Token 方案比 Session 方案更符合无状态原则：

```java
// 基于 Token 的认证（JWT 为例）
public String login(String username, String password) {
    // 验证用户密码
    User user = userService.verify(username, password);
    // 生成 Token，包含用户信息
    String token = JWT.create()
        .withClaim("userId", user.getId())
        .withClaim("username", user.getUsername())
        .withExpiresAt(Date.from(LocalDateTime.now().plusHours(2).toInstant(ZoneOffset.UTC)))
        .sign(Algorithm.HMAC256("secret"));
    return token;
}

// 后续请求直接解析 Token，不需要查 Session 存储
public User getCurrentUser(String token) {
    JWTVerifier verifier = JWT.require(Algorithm.HMAC256("secret")).build();
    DecodedJWT jwt = verifier.verify(token);
    String userId = jwt.getClaim("userId").asString();
    return userService.findById(userId);
}
```

**3. 避免本地缓存**

本地缓存（如 JVM 堆缓存）是有状态的，会导致数据不一致。如果需要缓存，使用分布式缓存（Redis）：

```java
// 错误写法：本地缓存
private static Map<String, Object> cache = new HashMap<>();

// 正确写法：分布式缓存
private static Cache<String, Object> cache = Caffeine.newBuilder()
    .maximumSize(10000)
    .expireAfterWrite(10, TimeUnit.MINUTES)
    .build();
```

:::tip 无状态 vs 有状态

无状态服务不是「没有状态」，而是「不在服务内存中存储状态」。状态存储在外部（数据库、缓存、文件系统），服务只负责处理逻辑。

无状态服务的好处：

- 任何机器都可以处理任何请求
- 扩容时不需要迁移数据
- 缩容时不需要担心数据丢失
- 故障时请求可以自动路由到其他机器

:::

### 3.3 负载均衡：请求如何分发

水平扩展后，请求需要分发到多台机器上。负载均衡器就是这个分发的入口。

常见的负载均衡算法：

| 算法 | 原理 | 优点 | 缺点 |
|---|---|---|---|
| **轮询（Round Robin）** | 依次分发 | 简单、公平 | 不考虑服务器负载差异 |
| **加权轮询（Weighted Round Robin）** | 按权重分发 | 可以差异化配置 | 需要手动配置权重 |
| **随机（Random）** | 随机选择 | 简单 | 不保证均匀 |
| **加权随机（Weighted Random）** | 按权重随机 | 灵活性高 | 权重配置复杂 |
| **最少连接（Least Connections）** | 选择连接数最少的 | 适合长连接场景 | 需要维护连接数 |
| **一致性哈希（Consistent Hashing）** | 按 Key 哈希 | 最小化数据迁移 | 实现复杂 |
| **IP 哈希（IP Hash）** | 按客户端 IP 哈希 | 同一 IP 路由到同一服务器 | 不够均匀 |

对于 HTTP 服务，Nginx 是最常用的负载均衡器：

```nginx
upstream backend {
    server 192.168.1.101:8080 weight=5;
    server 192.168.1.102:8080 weight=3;
    server 192.168.1.103:8080 weight=2;
    
    # 开启健康检查
    check interval=3000 rise=2 fall=3 timeout=1000 type=http;
    check_http_send "HEAD /health HTTP/1.0\r\n\r\n";
    check_http_expect_alive http_2xx;
}

server {
    listen 80;
    
    location / {
        proxy_pass http://backend;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
```

## 四、数据分片：突破单机存储上限

### 4.1 为什么要分片

水平扩展应用服务器很容易，但数据存储呢？

一台数据库服务器的磁盘空间是有限的。即使使用最大的企业级数据库服务器，存储几十 TB 的数据也会遇到瓶颈。更重要的是，数据库的写入能力也有上限——单台 MySQL 的写入 QPS 最多也就几万，而互联网公司的写入需求可能达到几十万甚至更高。

分片（Sharding）是解决这个问题的关键：**将数据分散到多个数据库实例中，每个实例只负责一部分数据**。

### 4.2 分片策略：如何决定数据去哪儿

分片策略决定了数据如何分配到不同的分片。常见的分片策略有：

**1. 哈希分片（Hash Sharding）**

将数据的某个字段（如用户 ID）做哈希运算，根据哈希值决定数据落在哪个分片：

```java
// 哈希分片的实现
public int getShardId(String userId, int shardCount) {
    return Math.abs(userId.hashCode()) % shardCount;
}

// 示例：4 个分片
// userId = "user_001" -> hash = 12345678 -> 12345678 % 4 = 2 -> 分片 2
// userId = "user_002" -> hash = 23456789 -> 23456789 % 4 = 1 -> 分片 1
```

哈希分片的优点是**数据分布均匀**，缺点是**扩缩容困难**——如果增加一个分片，所有数据都需要重新哈希，之前的数据几乎都要迁移。

**2. 范围分片（Range Sharding）**

根据数据的某个范围（如时间、ID 区间）来决定数据落在哪个分片：

```sql
-- 按时间范围分片
SELECT * FROM orders WHERE created_at >= '2024-01-01' AND created_at < '2024-02-01';

-- 按 ID 范围分片
SELECT * FROM user WHERE id >= 1000000 AND id < 2000000;
```

范围分片的优点是**查询友好**（范围查询可以快速定位分片），缺点是**可能产生热点**（最新数据都在最新的分片上）。

**3. 目录分片（Directory-Based Sharding）**

维护一个查找表（Lookup Table），记录每个数据对应哪个分片：

```sql
-- 分片映射表
CREATE TABLE shard_map (
    user_id VARCHAR(50) PRIMARY KEY,
    shard_id INT NOT NULL
);

-- 查询时先查映射表
SELECT shard_id FROM shard_map WHERE user_id = 'user_001';
-- 然后根据 shard_id 连接对应分片
```

目录分片的优点是**灵活性高**，缺点是**多一次查询**，且映射表本身可能成为瓶颈。

### 4.3 一致性哈希：优雅的扩缩容

哈希分片最大的问题是扩缩容困难。一致性哈希（Consistent Hashing）是一种更优雅的解决方案。

**传统哈希的问题**：

```java
// 假设原来有 3 个分片
shardId = hash(key) % 3;

// 如果增加到 4 个分片
shardId = hash(key) % 4;  // 大部分 key 对应的分片都会变！
```

增加一个分片，几乎所有数据都要重新分布。这就是「雪崩效应」。

**一致性哈希的原理**：

一致性哈希将哈希值空间组织成一个环（0 ~ 2^32-1），每个分片对应环上的一个位置。数据根据哈希值落在顺时针方向的第一个分片上。

```bash
# 一致性哈希环示意
# hash(分片A) = 1000
# hash(分片B) = 3000
# hash(分片C) = 5000
# hash(分片D) = 8000
#
#                    0 (2^32)
#                     │
#                     │
#                  分片D (8000)
#                     │
#                     │
# 分片A (1000) ──────►│◄──────── 分片C (5000)
#                     │
#                     │
#                  分片B (3000)
#
# key1 的 hash = 4000 → 落在分片C（顺时针最近）
# key2 的 hash = 6000 → 落在分片D（顺时针最近）
# key3 的 hash = 9000 → 落在分片A（环绕到起点）
```

**一致性哈希的优势**：

当增加或删除一个分片时，只有少量数据需要迁移：

- 新加分片 D：只影响 D 和 A 之间的数据
- 删除分片 B：只有 B 的数据需要迁移到 C

**虚拟节点：解决数据不均匀问题**

实际应用中，每个分片的哈希值可能集中在某个区域，导致数据分布不均匀。解决方案是引入虚拟节点（Virtual Node）：

```java
// 每个物理分片对应多个虚拟节点
// 分片A：vnode_A1, vnode_A2, vnode_A3, ... vnode_An
// 分片B：vnode_B1, vnode_B2, vnode_B3, ... vnode_Bn
// ...
// 每个虚拟节点都在环上有自己的位置，数据分布更加均匀
```

主流的 Redis 客户端（如 Jedis、Redisson）都支持一致性哈希和虚拟节点。

:::tip 分片策略的选择

**哈希分片**：适合数据分布均匀、查询无明显规律的OLTP场景。

**范围分片**：适合按时间或ID范围查询的OLAP场景，但要注意热点问题。

**一致性哈希**：适合需要频繁扩缩容的弹性系统，如Redis Cluster。

**目录分片**：适合需要灵活路由策略的特殊场景，但实现复杂。

:::

## 五、微服务拆分：功能的水平扩展

### 5.1 什么时候需要微服务

微服务是功能层面的「水平扩展」。当单体应用变得太大，开发团队扩张到几十人甚至上百人时，微服务拆分就成了必然选择。

微服务拆分的典型场景：

- **开发团队扩张**：一个 100 人的团队开发一个单体应用，代码合并、部署协调会变成噩梦
- **技术栈异构**：某些模块需要使用 Python 做机器学习，其他模块用 Java 做交易处理
- **独立扩展**：某个模块是 CPU 密集型（图片处理），另一个模块是 IO 密集型（文件上传）
- **故障隔离**：一个模块的故障不应该影响其他模块

### 5.2 拆分原则：如何划分微服务边界

微服务拆分的核心原则是**高内聚、低耦合**。

**高内聚**：相关的功能放在一起，一次需求变更只需要修改一个服务

**低耦合**：服务之间的依赖最小化，服务可以独立修改、独立部署

常见的拆分维度：

| 维度 | 说明 | 示例 |
|---|---|---|
| **业务能力拆分** | 按业务能力划分 | 订单服务、用户服务、支付服务 |
| **数据边界拆分** | 每个服务拥有自己的数据库 | 用户库、订单库、库存库 |
| **变更频率拆分** | 频繁变更的模块单独拆出 | 营销活动、推荐算法 |
| **团队边界拆分** | 按团队划分 | 每个团队负责 1-2 个服务 |

### 5.3 服务间通信：微服务的挑战

微服务拆分后，服务之间如何通信？

**同步通信：REST / gRPC**

```java
// REST 风格
@RestController
public class OrderController {
    @Autowired
    private RestTemplate restTemplate;
    
    public Order getOrder(String orderId) {
        // 调用用户服务获取用户信息
        User user = restTemplate.getForObject(
            "http://user-service/users/" + order.getUserId(), 
            User.class
        );
        order.setUser(user);
        return order;
    }
}

    .getUser(GetUserRequest.newBuilder().setUserId(userId).build())
    .getUser();
```

## 六、自动扩缩容：让系统「聪明」地扩展

### 6.1 为什么需要自动扩缩容

手动扩缩容的问题是：**人反应不够快**。

当流量突然上涨时，人工响应需要时间：发现告警、分析原因、决定扩容、执行操作——这个过程可能需要几十分钟甚至更久。而流量洪峰可能只持续几分钟，等你扩容完，流量已经下去了。

自动扩缩容（Auto Scaling）让系统根据预设的规则自动调整资源，响应时间从「分钟级」缩短到「秒级」。

### 6.2 扩缩容策略

常见的扩缩容策略：

**基于指标（Metrics-Based）**

根据 CPU、内存、网络等指标自动扩缩容：

```yaml
# Kubernetes HPA 示例
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: order-service-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: order-service
  minReplicas: 3
  maxReplicas: 100
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
  - type: Resource
    resource:
      name: memory
      target:
        type: Utilization
        averageUtilization: 80
```

**基于时间（Schedule-Based）**

在可预测的流量高峰前，提前扩容：

```yaml
# 基于 Cron 的定时扩缩容
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: order-service-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: order-service
  minReplicas: 3
  maxReplicas: 100
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 0
      policies:
      - type: Percent
        value: 100
        periodSeconds: 15
    scaleDown:
      stabilizationWindowSeconds: 300
      policies:
      - type: Percent
        value: 10
        periodSeconds: 60
```

**预测式扩缩容（Predictive Scaling）**

基于历史数据和机器学习，预测未来的流量，提前扩容。AWS、Google Cloud 等云厂商都提供这种能力。

### 6.3 扩缩容的避坑指南

**1. 防止「抖动」**

如果扩容阈值设置不合理，可能出现：扩容 → 流量下降 → 缩容 → 流量上涨 → 扩容……

解决方法是设置**冷却时间（Stabilization Window）**：

```yaml
# 扩容后至少等待 5 分钟，才能缩容
scaleDown:
  stabilizationWindowSeconds: 300
```

**2. 防止「惊群效应」**

多个应用同时扩容，可能压垮底层的共享资源（数据库、消息队列）。

解决方法是**分级扩容**：先扩容非核心服务，再扩容核心服务。

**3. 预留「 buffer」**

不要等到资源用尽才开始扩容。建议在资源使用率达到 70-80% 时就开始扩容，留有余量应对突发流量。

## 七、案例：Twitter 的扩展之路

### 7.1 Twitter 的规模挑战

2010 年，Twitter 每天处理 5000 万条推文，峰值 QPS 达到 30 万。用户抱怨「推文发送后，朋友的 Feed 要等很久才更新」。

这不是一个简单的性能问题，而是一个系统设计问题。

### 7.2 Pull vs Push：时间线设计的经典权衡

Twitter 的核心问题是「用户的时间线（Timeline）如何生成？」

**Push 模型（写时扩散）**

用户发推时，主动推送给所有粉丝：

```bash
用户A 发推 → 系统 → 推送给 A 的 100 万粉丝
                   ↓
            每个粉丝的时间线存储这条推文
```

Push 模型的优点是**读取快**（时间线已经预计算好），缺点是**写入慢**（发一条推要推送给所有粉丝）。

**Pull 模型（读时聚合）**

用户查看时间线时，实时聚合关注对象的推文：

```bash
用户A 查看时间线 → 系统 → 查询 A 关注的 1000 人的最新推文
                            ↓
                      聚合排序后返回
```

Pull 模型的优点是**写入快**（只需要存一条推文），缺点是**读取慢**（每次都要聚合大量数据）。

### 7.3 Twitter 的实际方案

Twitter 最终采用了**混合方案**：

- **普通用户**：Pull 模型，粉丝少（几百人），读取开销可接受
- **大 V 用户**：Push 模型，粉丝多（几百万人），读请求量巨大，写入一次、读取受益多次

同时，引入**缓存层**：

```bash
Timeline Cache（Redis Cluster）
├── user:123:timeline → [推文ID列表，最多 800 条]
└── 缓存过期时间：24 小时

用户查看时间线时：
1. 先查缓存
2. 缓存 miss，再去聚合
3. 结果回填缓存
```

这就是为什么你看到的 Twitter 时间线可能是「旧的」——系统优先保证可用性，而不是强一致性。

:::tip 案例启示

Twitter 的设计告诉我们：

- **没有完美的方案，只有最适合的权衡**
- **规模是检验架构的唯一标准**
- **缓存是性能优化的第一利器**
- **混合方案往往比单一方案更优**

:::

## 八、面试追问

### 问题一：水平扩展和垂直扩展如何选择？

**参考答案**：

选择取决于具体的业务场景和约束：

**优先垂直扩展**：

- 业务早期，用户量不大
- 单机性能还有提升空间
- 团队技术储备不足（分布式改造需要成本）
- 预算有限（分布式系统运维成本更高）

**优先水平扩展**：

- 业务规模大，单机无法满足需求
- 对可用性要求高（水平扩展天然支持高可用）
- 技术储备充足（团队有分布式运维能力）
- 成本可控（云服务让水平扩展成本更低）

**实践中**：往往两者结合。先做垂直扩展，当单机成为瓶颈后，逐步改造为水平扩展架构。

### 问题二：分库分表的最佳实践是什么？

**参考答案**：

**分库分表前的考量**：

- 先尝试读写分离（大部分场景读多写少）
- 再尝试缓存优化（热点数据放缓存）
- 最后才考虑分库分表（复杂度最高）

**分库分表的原则**：

- 按业务ID（而不是时间）分片，避免热点分片
- 预留 2-3 倍的扩容余量
- 分片数最好是 2 的幂次，方便后续扩容
- 分片键不要选择有业务语义的字段（可能变化）

**分库分表的工具选择**：

- Apache ShardingSphere：功能完善，社区活跃
- MyCat：老牌方案，配置灵活
- Vitess：MySQL 水平扩展方案，兼容 MySQL 协议

### 问题三：如何设计一个支持动态扩缩容的缓存集群？

**参考答案**：

核心挑战是**扩缩容时数据迁移的平滑性**。

**使用一致性哈希**：

- 数据根据 Key 的哈希值分布在多个节点上
- 增加/删除节点时，只有部分数据需要迁移
- 使用虚拟节点减少数据倾斜

**数据迁移策略**：

- **灰度迁移**：先迁移冷数据，再迁移热数据
- **双写策略**：新节点和老节点同时写入，切换完成后删除老节点
- **预设容量**：提前预留「buffer」节点，用于扩容

**Redis Cluster 方案**：

```bash
# Redis Cluster 自动处理槽迁移
# 16384 个槽分布在多个节点上
# 增加节点时，自动迁移部分槽

redis-cli --cluster add-node new_node:6379 existing_node:6379
redis-cli --cluster reshard new_node:6379 --cluster-from <node-id> --cluster-to <new-node-id> --cluster-slots 1000
```

## 总结

可扩展性是系统设计的核心能力，决定了系统能否应对业务的增长。

关键要点回顾：

- **垂直扩展**：简单直接，但有成本上限和单点故障问题
- **水平扩展**：突破单机上限，但需要无状态设计和负载均衡
- **数据分片**：哈希分片、范围分片、一致性哈希各有优劣
- **微服务拆分**：功能层面的扩展，但引入服务间通信的复杂性
- **自动扩缩容**：让系统「聪明」地应对流量波动
- **分片策略没有标准答案**，需要根据业务场景选择

记住：**扩展性不是架构的目标，而是业务增长的保障**。好的架构不是「一步到位」的完美设计，而是能够随着业务发展平滑演进的设计。
