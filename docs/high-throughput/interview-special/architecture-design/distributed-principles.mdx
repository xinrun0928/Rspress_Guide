# 分布式系统设计原则：CAP、BASE 与一致性模型

想象这样一个场景：你在电商 App 上买了一本书，支付成功后，页面显示「库存不足」。你明明已经付了钱，系统却告诉你没货了。

这不是段子，这是分布式系统的「经典困境」：你的支付请求到达了服务器 A，你的库存查询请求到达了服务器 B，服务器 A 说「已支付」，服务器 B 说「没库存」。两个服务器之间的数据还没来得及同步，你的请求就被处理了。

这就是分布式系统的本质：**数据分散在多个节点上，节点之间需要协调，而协调需要时间**。在这个时间窗口里，数据可能不一致。

很多程序员写单机代码写得很好，但一进入分布式环境就「水土不服」：要么追求强一致导致性能下降，要么追求高性能导致数据不一致。问题的根源在于：没有真正理解分布式系统的设计原则。

CAP 定理、BASE 理论、一致性模型——这些概念你可能听过，但不一定真正理解它们背后的权衡。这篇文章，我会从这些理论出发，带你理解分布式系统的设计哲学，最后给出实战中的选型建议。

## 一、CAP 定理：分布式系统的铁律

### 1.1 CAP 定理的来源

CAP 定理由加州大学伯克利分校的 Eric Brewer 教授在 2000 年提出，并在 2002 年被 Seth Gilbert 和 Nancy Lynch 证明。

**CAP** 指的是分布式系统的三个特性：

- **C（Consistency，一致性）**：所有节点在同一时刻看到的数据是相同的
- **A（Availability，可用性）**：每个请求都能在合理的时间内得到响应
- **P（Partition Tolerance，分区容错）**：系统能够容忍网络分区（即节点之间的网络中断）

**CAP 定理的核心结论**：在分布式系统中，这三个特性不可能同时满足，最多只能同时满足两个。

```
        ┌──────────────┐
        │              │
    ────┤     CAP      ├────
        │              │
        └──────┬───────┘
               │
       ┌───────┴───────┐
       │               │
       ▼               ▼
  ┌─────────┐     ┌─────────┐
  │   CA    │     │   CP    │
  │ 单机系统 │     │ ZooKeeper │
  └─────────┘     └─────────┘

       ┌─────────┐
       │   AP    │
       │ Cassandra│
       └─────────┘
```

### 1.2 为什么 P 是必然的选择

在 CAP 定理中，很多人会问：「那我能不能放弃 P，选择 CA？」

答案是：**在真实世界中，P 是必然的选择**。

网络分区（Partition）是不可避免的——光缆可能被挖断、路由器可能故障、网络可能拥堵。任何跨数据中心的分布式系统，都必须考虑网络分区的问题。

所以，在实际设计中，我们不是在「C、A、P 三个都要」和「三个都不要」之间选择，而是在 **「选择 CP」** 和 **「选择 AP」** 之间选择。

### 1.3 CP vs AP：两个世界的选择

**CP 系统（一致性优先）**

CP 系统在发生网络分区时，为了保证一致性，会拒绝服务（即牺牲可用性）。

典型代表：

- **ZooKeeper**：分布式协调服务，一致性优先
- **etcd**：Kubernetes 的元数据存储，强一致
- **HBase**：列式数据库，保证数据一致
- **Redis Cluster**：主从切换时可能短暂不可用

**AP 系统（可用性优先）**

AP 系统在发生网络分区时，为了保证可用性，允许短暂的数据不一致。

典型代表：

- **Cassandra**：NoSQL 数据库，最终一致
- **DynamoDB**：AWS 的 NoSQL，最终一致
- **Eureka**：服务注册中心，保证可用性
- **DNS**：域名系统，最终一致

```java
// CP 系统示例：ZooKeeper
// 当 ZooKeeper 集群发生分区时，少数派（Minority）会停止服务
// 只有多数派（Majority）能继续服务

// AP 系统示例：Cassandra
// 当 Cassandra 集群发生分区时，两边都能继续写入
// 但数据会不一致，需要后续修复

// Cassandra 的写入策略
@Data
public class CassandraWriteConfig {
    // ALL：写入所有副本，强一致，但分区时不可用
    // QUORUM：写入法定数量副本，平衡一致性和可用性
    // ONE：写入最近的一个节点，高可用，可能不一致
    ConsistencyLevel consistencyLevel = ConsistencyLevel.QUORUM;
}
```

### 1.4 CAP 定理的补充：PACELC

CAP 定理有一个重要的补充理论：**PACELC**（Partition + Availability + C + E + L + C）。

PACELC 提出了一个 CAP 没有回答的问题：**在没有发生分区的时候，我们应该选择一致性还是延迟？**

```
PACELC = "If there is a Partition (P), how does the system trade off 
          Availability (A) vs Consistency (C)? 
          Else (E), how does the system trade off Latency (L) vs Consistency (C)?"
```

| 数据库 | PACELC 分类 | 说明 |
|---|---|---|
| **DynamoDB** | PA/EC | 分区时优先可用；否则优先延迟 |
| **Cassandra** | PA/EL | 分区时优先可用；否则优先延迟 |
| **HBase** | PC/EC | 始终优先一致性；否则优先一致性 |
| **ZooKeeper** | PC/EC | 始终优先一致性；否则优先一致性 |
| **MongoDB** | PC/EC | 始终优先一致性；否则优先一致性 |

:::tip PACELC 的启示

CAP 定理告诉我们「分区时怎么选」，PACELC 告诉我们「不分区时怎么选」。

对于大多数互联网应用，我们更关心的其实是 **「不分区时能不能快点」**。因为网络分区是小概率事件，而延迟是每次请求都要面对的问题。

所以，很多 AP 系统（追求高可用）的本质选择是：**牺牲一致性来换取低延迟**。

:::

## 二、BASE 理论：实用主义的分布式哲学

### 2.1 BASE 的诞生

CAP 定理告诉我们：强一致性和高可用性不可兼得。但对于大多数互联网应用，100% 的强一致性并不是必须的——用户可以接受「订单状态晚几秒才更新」，但不能接受「订单支付了却显示失败」。

BASE 理论就是在这种背景下提出的，它是 **Basically Available, Soft state, Eventually consistent** 的缩写，代表了一种**实用主义的分布式设计哲学**。

### 2.2 BASE 的三个特性

**1. Basically Available（基本可用）**

系统保证核心功能可用，非核心功能可以降级。

```java
// 基本可用的设计示例
@Service
public class ProductService {
    
    public ProductDetail getProductDetail(Long productId) {
        try {
            // 尝试获取实时数据
            ProductDetail detail = realTimeService.getDetail(productId);
            return detail;
        } catch (Exception e) {
            // 实时服务不可用，返回缓存数据
            log.warn("Real-time service unavailable, return cached data");
            return cacheService.getCachedDetail(productId);
        }
    }
}
```

**2. Soft State（软状态）**

系统数据可以存在中间状态，不要求强一致。

```
强一致：订单状态要么「待支付」，要么「已支付」，没有中间状态
软状态：订单状态可以是「支付中」，表示支付请求已发出但未确认
```

**3. Eventually Consistent（最终一致性）**

系统在一段时间后（通常很短），最终会达到一致状态。

```
时间线：
T=0: 用户下单，订单状态「创建」
T=1: 用户支付，支付状态「支付中」
T=2: 支付回调，订单状态「已支付」，支付状态「已支付」
T=3: 所有系统同步完成，所有人看到「已支付」

在 T=1 到 T=3 之间，不同的人可能看到不同的状态
但最终，在 T=3 之后，所有人看到的状态是一致的
```

### 2.3 BASE vs ACID

BASE 和 ACID 是两种对立的事务处理哲学：

| 特性 | ACID（传统数据库） | BASE（分布式系统） |
|---|---|---|
| **一致性** | 强一致 | 最终一致 |
| **可用性** | 低（牺牲可用保一致） | 高 |
| **事务** | 原子事务 | 大事务（多个操作） |
| **隔离性** | 严格隔离 | 不强调隔离 |
| **典型代表** | MySQL、PostgreSQL | Cassandra、MongoDB |
| **适用场景** | 金融、订单、库存 | 社交、评论、日志 |

**什么时候用 ACID，什么时候用 BASE？**

| 场景 | 推荐方案 | 理由 |
|---|---|---|
| 金融交易 | ACID | 金额不能错，必须强一致 |
| 订单系统 | ACID | 订单状态必须准确 |
| 库存扣减 | ACID + 分布式锁 | 超卖不可接受 |
| 社交 Feed | BASE | 用户可以接受 Feed 晚几秒更新 |
| 日志系统 | BASE | 日志允许短暂不一致 |
| 配置中心 | BASE | 配置可以短暂不一致 |

### 2.4 BASE 实践：如何实现最终一致性

最终一致性不是「不管一致性」，而是「延迟一致」。在实际实现中，需要考虑以下问题：

**1. 如何检测不一致？**

```java
// 方案一：定时对账
@Scheduled(cron = "0 0 3 * * ?")  // 每天凌晨 3 点
public void dailyReconciliation() {
    List<Order> orders = orderService.getYesterdayOrders();
    for (Order order : orders) {
        Payment payment = paymentService.getPayment(order.getId());
        if (!order.getStatus().equals(payment.getStatus())) {
            log.error("Inconsistency detected: order {} vs payment {}", 
                order.getStatus(), payment.getStatus());
            reconcile(order, payment);
        }
    }
}

// 方案二：消息队列重试
@KafkaListener(topics = "payment.callback")
public void handlePaymentCallback(PaymentCallback callback) {
    try {
        orderService.updateStatus(callback.getOrderId(), "PAID");
    } catch (Exception e) {
        // 失败，消息会被重新投递
        throw e;
    }
}
```

**2. 如何修复不一致？**

```java
// 补偿事务（Compensating Transaction）
@Service
public class ReconciliationService {
    
    public void reconcile(Order order, Payment payment) {
        // 根据「先到」原则修复
        // 如果订单状态是「已创建」，支付状态是「已支付」
        // 说明订单状态更新慢了，更新订单状态
        if (order.getStatus().equals("CREATED") && 
            payment.getStatus().equals("PAID")) {
            orderService.updateStatus(order.getId(), "PAID");
            return;
        }
        
        // 其他情况需要人工介入
        log.error("Manual reconciliation required: order {} vs payment {}", 
            order, payment);
        alertSlack("需要人工对账");
    }
}
```

## 三、一致性模型：不同程度的「一致」

### 3.1 一致性模型的层次

一致性不是一个「有无」的问题，而是一个「程度」的问题。从强到弱，一致性可以分为多个层次：

```
强一致 → 顺序一致 → 因果一致 → 会话一致 → 读己之所写 → 单调读 → 最终一致
   │         │          │          │          │         │        │
   │         │          │          │          │         │        │
   最严格 ──────────────────────────────────────────────────▶ 最弱
```

**1. 强一致性（Linearizability）**

最强的一致性模型。任何节点的读写操作都像是在单一节点上执行，所有节点看到操作的顺序都是一样的。

```java
// 强一致性的例子
// 所有节点看到：x=1 → x=2 → x=3，操作顺序完全一致
Node A: Write(x, 1) → Write(x, 2) → Write(x, 3)
Node B: 看到 Write(x, 1) → 看到 Write(x, 2) → 看到 Write(x, 3)
Node C: 看到 Write(x, 1) → 看到 Write(x, 2) → 看到 Write(x, 3)
```

**2. 顺序一致性（Sequential Consistency）**

每个节点的读写操作按某种全局顺序执行，但这个顺序不需要和真实时间一致。

**3. 因果一致性（Causal Consistency）**

只有因果相关的操作需要按顺序，无因果关系的操作可以乱序。

```java
// 因果一致性的例子
// 「发朋友圈」和「看到朋友圈」有因果关系，必须按顺序
// 「A 发朋友圈」和「B 发朋友圈」无因果关系，可以乱序

// B 发朋友圈（无因果依赖）
// A 发朋友圈
// C 看到：A 的朋友圈一定在 B 之前或之后，但两人都必须按相同顺序看到
```

**4. 最终一致性（Eventual Consistency）**

最弱的一致性模型。系统保证如果没有新的更新，最终所有节点会看到相同的数据。

### 3.2 客户端视角的一致性

从客户端的角度，一致性有以下几种常见模型：

**1. 读己之所写（Read Your Writes）**

用户写入的数据，立即能被该用户读取到。

```java
// 读己之所写的实现
// 用户下单后，立即查看订单，应该能看到刚下的订单

// 方案：写入后，记录写入的版本号
public class Session {
    private Set<Long> writtenKeys = new HashSet<>();  // 用户写入的 key
    
    public void write(String key, Object value) {
        storage.put(key, value);
        writtenKeys.add(key.hashCode());
    }
    
    public Object read(String key) {
        if (writtenKeys.contains(key.hashCode())) {
            // 用户写过这个 key，直接从本地读
            return storage.get(key);
        }
        // 用户没写过，走正常的读流程
        return remoteStorage.get(key);
    }
}
```

**2. 单调读（Monotonic Reads）**

用户读取的数据不会「倒退」。如果用户看到了某个版本的数据，之后不会看到更旧版本的数据。

```java
// 单调读的实现
// 用户刷新页面，不应该看到「更旧」的订单列表

public class MonotonicReadService {
    private Map<String, Long> lastSeenVersion = new ConcurrentHashMap<>();
    
    public List<Order> getOrders(String userId) {
        // 获取当前版本
        long currentVersion = getCurrentVersion();
        
        // 如果是单调读，不能返回比上次更旧的版本
        Long previousVersion = lastSeenVersion.get(userId);
        if (previousVersion != null && currentVersion < previousVersion) {
            // 版本倒退了，需要等一下，或者返回缓存
            currentVersion = previousVersion;
        }
        
        lastSeenVersion.put(userId, currentVersion);
        return fetchOrdersByVersion(currentVersion);
    }
}
```

## 四、分布式一致性算法

### 4.1 Paxos 算法

Paxos 是 Leslie Lamport 提出的分布式一致性算法，被认为是分布式系统领域的「圣经」。

**Paxos 的两个阶段**

**第一阶段：Prepare（准备阶段）**

```
提案者（Proposer）：
  向所有接受者发送 Prepare(n) 请求
  n 是提案编号（全局唯一递增）

接受者（Acceptor）：
  如果 n 大于之前接收到的任何提案编号
    → 承诺不再接受编号小于 n 的提案
    → 返回之前接受的编号最大的提案
  否则
    → 拒绝
```

**第二阶段：Accept（接受阶段）**

```
提案者（Proposer）：
  如果收到多数派的响应
    → 选择响应中编号最大的提案的值（或自己的值）
    → 向所有接受者发送 Accept(n, value)

接受者（Acceptor）：
  如果 n 大于等于之前承诺的最小提案编号
    → 接受这个提案
  否则
    → 拒绝
```

**Paxos 的问题**

- 实现复杂，理论上正确但工程实现难度大
- 多 Paxos（Multi-Paxos）可以优化，但更复杂

:::warning Paxos 的工程化

Google 的 Chubby、ZooKeeper 的 ZAB 协议等都是 Paxos 的变体。

如果你要自己实现 Paxos，建议使用现有的库，而不是从头造轮子。Paxos 的理论简单，但细节很多，很容易出错。

:::

### 4.2 Raft 算法

Raft 是 Diego Ongaro 和 John Ousterhout 提出的共识算法，被认为是 Paxos 的「简化版」。

Raft 把一致性问题分解为三个子问题：

- **领导选举（Leader Election）**
- **日志复制（Log Replication）**
- **安全性（Safety）**

**Raft 的角色**

```
    ┌─────────┐
    │ Leader  │ ← 接收客户端请求，复制日志到跟随者
    └────┬────┘
         │ 心跳
         ▼
    ┌─────────┐        ┌─────────┐
    │ Follower│        │ Follower│ ← 接收 Leader 的日志
    └─────────┘        └─────────┘

    ┌─────────┐
    │Candidate │ ← 发起选举时变为候选者
    └─────────┘
```

**领导选举**

```
1. 初始状态：所有节点都是 Follower
2. Follower 收不到 Leader 的心跳，等待 electionTimeout（150-300ms 随机）
3. 超时后，Follower 变为 Candidate，发起选举
4. Candidate 投自己一票，向其他节点发送 RequestVote
5. 如果获得多数派投票，成为 Leader
6. 如果其他人成为 Leader，Candidate 变回 Follower
```

```java
// Raft Leader 选举的简化实现
public class RaftNode {
    private volatile State state = State.FOLLOWER;
    private long currentTerm = 0;
    private String votedFor = null;
    private long lastHeartbeat = 0;
    
    public void runElection() {
        if (state == State.LEADER) {
            // Leader 发送心跳
            sendHeartbeat();
        } else {
            // 检查是否超时
            if (System.currentTimeMillis() - lastHeartbeat > electionTimeout) {
                // 开始选举
                startElection();
            }
        }
    }
    
    private void startElection() {
        state = State.CANDIDATE;
        currentTerm++;
        votedFor = myId;  // 投自己
        
        int votes = 1;  // 自己的一票
        
        // 向其他节点发送 RequestVote
        for (String peer : peers) {
            RequestVoteResponse response = sendRequestVote(peer, currentTerm);
            if (response.voteGranted) {
                votes++;
            }
        }
        
        // 获得多数派投票，成为 Leader
        if (votes > peers.size() / 2) {
            state = State.LEADER;
            log.info("Elected as leader for term {}", currentTerm);
        }
    }
}
```

**日志复制**

```
Leader 接收客户端请求：
1. 将请求追加到本地日志
2. 向所有 Follower 发送 AppendEntries
3. 如果多数派成功写入，成为 committed
4. 通知 Follower 将日志项 applied
```

```java
// Raft 日志复制的简化实现
public class LogEntry {
    long term;
    Object command;
}

public class RaftNode {
    private List<LogEntry> log = new ArrayList<>();
    private int commitIndex = 0;
    
    public void replicateLog(Command command) {
        // 1. 追加到本地日志
        LogEntry entry = new LogEntry();
        entry.term = currentTerm;
        entry.command = command;
        log.add(entry);
        
        // 2. 复制到所有 Follower
        for (String peer : peers) {
            sendAppendEntries(peer);
        }
        
        // 3. 如果多数派成功，提交
        if (getNumReplicated() > peers.size() / 2) {
            commitIndex = log.size() - 1;
            applyLogEntries();
        }
    }
}
```

### 4.3 两阶段提交（2PC）

2PC 是一种传统的分布式事务协议。

**协议流程**

```
                    ┌─────────────────────────────────────┐
                    │           Coordinator（协调者）       │
                    └─────────────────────────────────────┘
                              │
                              │ Phase 1: Prepare
                              ▼
              ┌───────────────┬───────────────┐
              │               │               │
              ▼               ▼               ▼
        ┌──────────┐    ┌──────────┐    ┌──────────┐
        │ Participant│   │ Participant│   │ Participant│
        └─────┬────┘    └─────┬────┘    └─────┬────┘
              │               │               │
              ▼               ▼               ▼
        ┌──────────┐    ┌──────────┐    ┌──────────┐
        │  Prepared │    │  Prepared │    │  Prepared │
        └──────────┘    └──────────┘    └──────────┘
              │               │               │
              │◀──────────────┼──────────────│
              │               │               │
              │     Phase 2: Commit/Rollback   │
              │               │               │
              ▼               ▼               ▼
        ┌──────────┐    ┌──────────┐    ┌──────────┐
        │ Committed│    │ Committed│    │ Committed│
        └──────────┘    └──────────┘    └──────────┘
```

**2PC 的问题**

- **同步阻塞**：协调者等待参与者响应时，所有参与者都在等待
- **单点故障**：协调者挂了，参与者不知道该 commit 还是 rollback
- **数据不一致**：协调者 commit 后，部分参与者 commit 失败

### 4.4 三阶段提交（3PC）

3PC 是 2PC 的改进，试图解决阻塞问题。

```
Phase 1: CanCommit？
  Coordinator → Participants: "Can you commit?"
  Participants → Coordinator: "Yes/No"

Phase 2: PreCommit
  Coordinator → Participants: "PreCommit"
  Participants 锁定资源，准备提交

Phase 3: DoCommit
  Coordinator → Participants: "DoCommit"
  Participants 提交，释放资源
```

**3PC 的改进**

- 增加了「CanCommit」阶段，提前确认参与者是否可以提交
- 如果参与者超时未收到协调者指令，自动 commit（而非 rollback）

**3PC 仍然不是完美的**：网络分区时，仍然可能导致数据不一致。

### 4.5 Saga 模式

Saga 是处理长事务（Long-Running Transaction）的一种模式，适用于跨服务的业务流程。

**Saga 的思想**：把大事务拆成多个小事务，每个小事务都有对应的补偿事务。任何一步失败，执行前面步骤的补偿。

```
订单流程的 Saga：
1. 创建订单（CreateOrder）
2. 扣减库存（DeductInventory）     ← 如果失败，补偿：增加库存
3. 扣减余额（DeductBalance）        ← 如果失败，补偿：增加余额
4. 发送通知（SendNotification）     ← 如果失败，补偿：重试发送

任何一步失败，逆序执行补偿操作
```

```java
// Saga 模式的实现
@Service
public class OrderSaga {
    
    @SagaTransaction(startStep = 1)
    public Order createOrder(OrderRequest request) {
        return orderService.create(request);
    }
    
    @SagaTransaction(step = 2, compensate = "compensateInventory")
    public void deductInventory(Order order) {
        inventoryService.deduct(order.getItemId(), order.getQuantity());
    }
    
    @Compensable(compensationMethod = "compensateInventory")
    public void compensateInventory(Order order) {
        inventoryService.increase(order.getItemId(), order.getQuantity());
    }
    
    @SagaTransaction(step = 3, compensate = "compensateBalance")
    public void deductBalance(Order order) {
        balanceService.deduct(order.getUserId(), order.getAmount());
    }
    
    @Compensable(compensationMethod = "compensateBalance")
    public void compensateBalance(Order order) {
        balanceService.increase(order.getUserId(), order.getAmount());
    }
}
```

## 五、Leader 选举：谁说了算

### 5.1 为什么需要 Leader

在分布式系统中，很多操作需要「一个说了算」的人。如果每个节点都各自为政，就会出现「脑裂」（Split Brain）问题。

Leader 选举解决的问题：

- **数据一致性**：写入只需要 Leader 处理，避免冲突
- **协调效率**：避免多个节点同时做同一件事
- **简化逻辑**：其他节点只需要「听 Leader 的」

### 5.2 常见的 Leader 选举方案

**1. ZooKeeper Leader 选举**

ZooKeeper 使用 ZAB 协议进行 Leader 选举。核心思想是：**谁的数据最新，谁就能当选**。

```bash
# ZooKeeper 选举过程
# 节点 A、B、C 同时发起选举
# A 的 ZXID（事务ID）最大
# A 成为 Leader，B、C 成为 Follower
```

**2. etcd Leader 选举**

etcd 使用 Raft 协议进行 Leader 选举。核心思想是：**谁获得多数派投票，谁就当选**。

```bash
# etcd 选举过程
# 节点 A 发起选举，获得 B、C 的投票
# A 成为 Leader
# 如果 A 挂了，B 发起选举，获得 C 的投票
# B 成为 Leader
```

**3. Redis Sentinel / Cluster**

Redis 使用类似于 Raft 的协议进行 Leader 选举。

```bash
# Redis Sentinel 选举
# 主库挂了，Sentinal 投票
# 获得多数票的从库成为新的主库
```

## 六、向量时钟：解决冲突的艺术

### 6.1 什么是向量时钟

向量时钟（Vector Clock）是解决分布式系统中数据冲突的一种方法。

在分布式系统中，不同节点可能同时修改同一份数据。当网络分区恢复时，如何合并这些修改？

**向量时钟的思想**：每个节点维护一个「时间戳向量」，记录每个节点的「逻辑时间」。

```java
// 向量时钟示例
public class VectorClock {
    private Map<String, Long> clock = new HashMap<>();  // 节点ID -> 逻辑时间
    
    public void increment(String nodeId) {
        clock.put(nodeId, clock.getOrDefault(nodeId, 0L) + 1);
    }
    
    public VectorClock merge(VectorClock other) {
        VectorClock result = new VectorClock();
        for (String key : clock.keySet()) {
            result.clock.put(key, Math.max(
                this.clock.getOrDefault(key, 0L),
                other.clock.getOrDefault(key, 0L)
            ));
        }
        return result;
    }
    
    // 比较两个向量时钟
    public Relation compare(VectorClock other) {
        // 返回：相同、因果、非因果
    }
}
```

**向量时钟的比较**

```
时钟 A: {A:1, B:2, C:1}
时钟 B: {A:1, B:3, C:1}

比较：
- A <= B (A 的每个分量 <= B 的对应分量)
- A < B (有分量严格小于)
结论：B 是 A 的「后代」，A 可以接受 B 的更新
```

### 6.2 Cassandra 的向量时钟

Cassandra 使用向量时钟解决冲突。写入时带上自己的向量时钟，读取时合并多个版本。

```java
// Cassandra 冲突解决
public class ConflictResolver {
    
    public Data resolve(List<Version> versions) {
        // Cassandra 使用「最后写入者胜出」（LWW）
        // 实际使用的是时间戳，而非向量时钟
        
        Version latest = versions.get(0);
        for (Version v : versions) {
            if (v.timestamp > latest.timestamp) {
                latest = v;
            }
        }
        return latest.data;
    }
}
```

## 七、实战选型：什么场景用什么方案

### 7.1 一致性需求的分类

在实际项目中，根据一致性需求，可以分为几类：

| 场景 | 一致性要求 | 推荐方案 |

## 八、面试追问

### 问题一：CAP 和 BASE 的区别是什么？

**参考答案**：

**CAP 定理**是一个理论，证明了你不可能同时拥有强一致性、高可用和分区容错。它告诉你「不可能三角」。

**BASE 理论**是一个实践指导，告诉你既然 CAP 只能选两个，那就选择高可用和分区容错，然后用最终一致性来弥补。

关系：

- CAP 是理论基础，BASE 是实践指南
- CAP 告诉你「不能什么」，BASE 告诉你「应该怎么做」
- BASE 是对 CAP 的补充和妥协

### 问题二：ZooKeeper 是 CP 还是 AP？

**参考答案**：

**ZooKeeper 是 CP 系统**。

ZooKeeper 在发生分区时，少数派（Minority）会停止服务。只有多数派（Majority）能继续处理请求。

但要注意的是，ZooKeeper 的「可用性」是相对的：

- 分区期间，少数派确实不可用
- 但多数派仍然可用
- 所以「整体系统」仍然可用，只是「部分节点」不可用

### 问题三：如何实现「读己之所写」？

**参考答案**：

读己之所写（Read Your Writes）有几种实现方式：

**1. Session 绑定**

用户写入后，记录写入的 key。后续读取时，先检查用户是否写过这个 key，如果写过，直接从本地读取。

**2. 版本号同步**

写入后，返回数据版本号。读取时带上版本号，如果版本号比本地旧，等待或从远端读取。

**3. 管道化写入**

写入操作通过管道异步复制到读取节点。写入返回后，等待一定时间再让用户读取。

## 思考题

### 题目一：强一致 vs 最终一致

假设你在设计一个银行转账系统。用户 A 向用户 B 转账 100 元。你会选择强一致还是最终一致？为什么？

**提示**：考虑资金损失的风险、用户体验、实现的复杂度。

### 题目二：2PC 的改进

2PC 有同步阻塞和单点故障的问题。如果让你改进 2PC，你会怎么做？

**提示**：可以考虑增加超时机制、引入观察者角色、改进协调者选举等。

### 题目三：CAP 的边界

假设你的系统在一个数据中心内部，所有节点都在同一个局域网中。网络分区的概率极低。

这个时候，你是否可以选择 CA（即忽略 P）？

**提示**：考虑「概率很低」和「不可能发生」的区别，以及 CAP 定理的假设。

## 总结

分布式系统设计的核心是在「一致性」「可用性」「分区容错」之间做权衡。

关键要点回顾：

- **CAP 定理**：三者不可兼得，最多同时满足两个
- **PACELC 补充**：没有分区时，还要在延迟和一致性之间权衡
- **BASE 理论**：选择高可用 + 最终一致，放弃强一致
- **一致性模型**：从强一致到最终一致，有多个层次
- **一致性算法**：Paxos、Raft 用于强一致选举；2PC/3PC 用于分布式事务
- **Saga 模式**：适合长事务，用补偿代替回滚
- **向量时钟**：解决分布式冲突的方法

记住：**没有银弹，只有权衡**。根据业务场景选择合适的一致性模型和算法，才是分布式系统设计的正确姿势。
