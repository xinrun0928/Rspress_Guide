# 高性能架构设计：如何设计低延迟系统

你打开一个网页，等了 3 秒，还在转圈圈。你点击一个按钮，等了 5 秒，没有任何反应。你刷新一下，结果页面报错：「服务器内部错误」。

这不是技术问题，这是用户体验问题。每一秒的延迟，都在「杀死」用户。

Google 的研究表明，页面加载时间超过 3 秒，53% 的用户会离开。Amazon 的数据更直接：页面加载每增加 1 秒，转化率下降 7%。对于电商平台，这可能是几千万的营业额差距。

延迟不是技术指标，而是**业务指标**。每降低 100ms 的延迟，可能带来的是用户体验的提升、转化率的提高、乃至公司营收的增长。

这就是为什么「高性能架构设计」如此重要。这篇文章，我会从延迟的本质讲起，带你掌握低延迟系统的设计方法，最后给出一个 p99 < 100ms 的 API 设计实战。

## 一、延迟的本质：时间都去哪儿了

### 1.1 延迟 vs 吞吐量

在讨论性能之前，先要搞清楚两个核心概念：**延迟（Latency）** 和 **吞吐量（Throughput）**。

**延迟**是单次请求从发起到完成的时间。比如你点击一个按钮，1 秒后看到响应，这就是 1 秒的延迟。

**吞吐量**是单位时间内系统处理的请求数量。比如服务器每秒处理 1000 个请求，这就是 1000 QPS 的吞吐量。

两者的关系可以用「高速公路」来类比：

- **延迟** = 从出发点到目的地的时间
- **吞吐量** = 单位时间内通过收费站的车辆数量

一条车道很宽（单车道处理能力高），但收费站效率低（延迟高），吞吐量不一定高。反过来，收费站效率高（延迟低），但车道太少（并发能力低），吞吐量也不高。

好的性能优化，是同时优化**延迟**和**吞吐量**，让系统又快又能扛。

### 1.2 延迟的组成：一个请求的生命周期

一个 HTTP 请求从发出到响应返回，时间都花在哪儿了？

```
总延迟 = DNS 解析 + TCP 连接 + TLS 握手 + 服务器处理 + 数据库查询 + 响应传输
       ↓            ↓          ↓           ↓            ↓              ↓
       ~5ms         ~10ms      ~30ms       ~50ms         ~20ms           ~5ms
       
       总计：~120ms（理想情况下）
```

**各环节详解**

| 环节 | 平均耗时 | 优化手段 |
|---|---|---|
| **DNS 解析** | 5-20ms | DNS 缓存、HTTP DNS、智能 DNS |
| **TCP 连接** | 10-50ms | 连接复用、TCP 优化 |
| **TLS 握手** | 30-100ms | TLS 1.3、会话复用 |
| **服务器处理** | 10-100ms | 异步处理、代码优化 |
| **数据库查询** | 5-50ms | 索引优化、SQL 优化 |
| **响应传输** | 5-20ms | 压缩、CDN |

:::tip 延迟的「木桶原理」

系统延迟由最慢的环节决定。就像木桶装水，最短的那块木板决定了容量。

优化性能的第一步，是找出**瓶颈在哪里**。不要在已经很快的环节（比如网络传输）上优化，而要在最慢的环节（比如数据库查询）上投入资源。

:::

### 1.3 延迟的分布：P50、P95、P99 的含义

平均值会骗人。考虑两个系统：

- **系统 A**：99% 的请求 10ms，1% 的请求 1000ms，平均延迟 20ms
- **系统 B**：100% 的请求 20ms，平均延迟 20ms

平均值都是 20ms，但用户体验完全不同。系统 B 所有用户都觉得还行，系统 A 有 1% 的用户觉得「卡死了」。

这就是为什么要关注延迟的**分布**，而不是平均值。

| 指标 | 含义 | 应用场景 |
|---|---|---|
| **P50（中位数）** | 50% 的请求在此时限内完成 | 衡量「普通用户」体验 |
| **P95** | 95% 的请求在此时限内完成 | 衡量「大多数用户」体验 |
| **P99** | 99% 的请求在此时限内完成 | 衡量「长尾用户」体验 |
| **P999** | 99.9% 的请求在此时限内完成 | 衡量「极端用户」体验 |

对于大多数互联网应用，**P99** 是最重要的指标**。99% 的用户满意，但 1% 的用户「炸了」——这 1% 的用户可能正好是你的大客户，或者正好在投诉。

```java
// 统计延迟分布（使用 Micrometer + Prometheus）
@Configuration
public class MetricsConfig {
    @Bean
    public Timer requestTimer() {
        return Timer.builder("http.request.latency")
            .publishPercentiles(0.5, 0.95, 0.99, 0.999)  // P50, P95, P99, P999
            .register(meterRegistry);
    }
}

// 在请求处理中记录
@GetMapping("/api/users/{id}")
public User getUser(@PathVariable Long id, Timer.Sample sample) {
    try {
        User user = userService.getUser(id);
        return user;
    } finally {
        sample.stop(requestTimer);  // 自动记录延迟
    }
}
```

## 二、缓存：性能优化的第一利器

### 2.1 为什么缓存这么快

缓存之所以快，是因为它跳过了「计算」和「I/O」两个最耗时的环节。

**从磁盘读取 vs 从内存读取**

| 数据位置 | 访问时间 | 数量级 |
|---|---|---|
| CPU 缓存（L1/L2/L3） | 1-10ns | 纳秒级 |
| 内存 | 100ns | 百纳秒级 |
| SSD | 100μs | 百微秒级 |
| HDD | 10ms | 毫秒级 |
| 远程服务 | 10ms | 毫秒级 |

内存比 SSD 快 1000 倍，比网络存储快 10 万倍。把热点数据放在内存里，是性能优化的第一步。

### 2.2 多级缓存架构

实际生产环境中，通常会部署多级缓存：

```bash
用户请求 → L1 本地缓存 → L2 分布式缓存 → 数据源
           ↓命中            ↓命中
           ~0.01ms          ~0.5ms
           
           容量：~100MB      容量：~100GB
           命中率：~60%       命中率：~30%
```

**L1 本地缓存（Caffeine）**

本地缓存是最快的，访问延迟在纳秒级，适合存放**访问极其频繁、数据量小**的数据。

```java
@Configuration
public class LocalCacheConfig {
    @Bean
    public Cache<Long, User> userCache() {
        return Caffeine.newBuilder()
            .maximumSize(10000)                    // 最大 1 万条
            .expireAfterWrite(5, TimeUnit.MINUTES) // 写入 5 分钟后过期
            .recordStats()                          // 开启统计
            .build();
    }
}

// 使用
@Service
public class UserService {
    @Autowired
    private Cache<Long, User> userCache;
    
    public User getUser(Long id) {
        return userCache.get(id, () -> userMapper.selectById(id));
    }
}
```

**L2 分布式缓存（Redis）**

分布式缓存容量大，支持跨进程共享，适合存放**数据量大、访问频繁**的数据。

```java
@Service
public class UserCacheService {
    @Autowired
    private RedisTemplate<String, Object> redisTemplate;
    
    private static final String USER_KEY_PREFIX = "user:";
    private static final long CACHE_EXPIRE_SECONDS = 1800; // 30 分钟
    
    public User getUser(Long userId) {
        String key = USER_KEY_PREFIX + userId;
        
        // 先查本地缓存（省略，本地缓存 miss 才查 Redis）
        User user = (User) redisTemplate.opsForValue().get(key);
        if (user != null) {
            return user;
        }
        
        // Redis miss，查数据库
        user = userMapper.selectById(userId);
        if (user != null) {
            redisTemplate.opsForValue().set(key, user, CACHE_EXPIRE_SECONDS, TimeUnit.SECONDS);
        }
        return user;
    }
}
```

### 2.3 缓存策略选择

| 策略 | 读流程 | 写流程 | 一致性 | 适用场景 |
|---|---|---|---|---|
| **Cache Aside** | 读：miss 后回填 | 写：先更新 DB，再删缓存 | 较高 | 大多数读多写少场景 |
| **Read Through** | 读：缓存代理回填 | 写：直接写 DB | 较高 | 缓存透明场景 |
| **Write Through** | 读：从缓存读 | 写：缓存和 DB 同步更新 | 高 | 需要强一致性场景 |
| **Write Behind** | 读：从缓存读 | 写：先写缓存，异步写 DB | 低 | 写入性能要求高场景 |

**Cache Aside 是最常用的策略**：

```java
// Cache Aside 模式
public User getUser(Long userId) {
    // 1. 先读缓存
    User user = cache.get(userId);
    if (user != null) {
        return user;
    }
    
    // 2. 缓存 miss，读数据库
    user = userMapper.selectById(userId);
    if (user != null) {
        // 3. 回填缓存
        cache.set(userId, user);
    }
    return user;
}

public void updateUser(User user) {
    // 1. 先更新数据库
    userMapper.updateById(user);
    
    // 2. 删除缓存（不是更新，避免并发问题）
    cache.evict(user.getId());
}
```

:::warning 为什么 Cache Aside 写操作是「删缓存」而不是「更新缓存」？

**并发场景下的问题**：

1. 线程 A 更新数据 → 更新数据库
2. 线程 B 读取数据 → 缓存 miss → 读旧数据 → 回填缓存
3. 线程 A 更新缓存

结果：缓存里是旧数据。

**解决方案**：先删缓存，再更新数据库。但删缓存和更新数据库之间，可能又有请求回填旧数据。所以更稳妥的方案是**延迟双删**或**设置较短的过期时间**。

:::

## 三、连接复用：减少网络开销

### 3.1 HTTP Keep-Alive

HTTP/1.0 时代，每个请求都要建立一次 TCP 连接。HTTP/1.1 引入了 Keep-Alive，**一个 TCP 连接可以复用处理多个请求**，大幅减少了连接建立的开销。

```nginx
# Nginx Keep-Alive 配置
upstream backend {
    server 127.0.0.1:8080;
    keepalive 32;  # 保持 32 个空闲连接
}

server {
    location /api/ {
        proxy_pass http://backend;
        proxy_http_version 1.1;  # 开启 HTTP/1.1
        proxy_set_header Connection "";  # 清除 Connection 头
    }
}
```

**HTTP/2 的多路复用**更进一步：在同一个 TCP 连接上，可以同时发送多个请求和响应，彻底解决了 HTTP/1.1 的「队头阻塞」问题。

```nginx
# Nginx HTTP/2 配置
server {
    listen 443 ssl http2;
    ssl_certificate cert.pem;
    ssl_certificate_key key.pem;
}
```

### 3.2 数据库连接池

数据库连接是「重量级」资源，建立一次连接需要 TCP 三次握手、TLS 握手（如果是加密连接）、认证等，开销在毫秒级。

连接池通过复用连接，让请求「借用」已有的连接，避免了反复建立连接的开销。

```java
// Druid 连接池核心配置
@Configuration
public class DataSourceConfig {
    @Bean
    public DataSource dataSource() {
        DruidDataSource dataSource = new DruidDataSource();
        
        // 连接池大小
        dataSource.setInitialSize(10);   // 初始连接数
        dataSource.setMinIdle(10);       // 最小空闲连接数
        dataSource.setMaxActive(100);    // 最大连接数
        
        // 连接获取超时
        dataSource.setMaxWait(30000);    // 获取连接最大等待 30 秒
        
        // 连接检测
        dataSource.setTestWhileIdle(true);
        dataSource.setValidationQuery("SELECT 1");
        dataSource.setTimeBetweenEvictionRunsMillis(60000);
        
        return dataSource;
    }
}
```

**连接池大小如何设置**

连接池太小，并发请求会排队等待；连接池太大，数据库压力增加。

```
连接池大小 = (核心数 * 2) + 有效磁盘数

示例：
- 4 核 CPU，SSD：连接池 = (4 * 2) + 1 = 9 ≈ 10
```

### 3.3 数据库查询优化

数据库是延迟的主要来源之一。优化数据库查询，能显著降低系统延迟。

**索引优化**

```sql
-- 为查询条件创建索引
CREATE INDEX idx_user_id ON orders(user_id);
CREATE INDEX idx_created_at ON orders(created_at);
CREATE INDEX idx_user_status ON orders(user_id, status);  -- 组合索引

-- 避免索引失效
-- ❌ SELECT * FROM orders WHERE status = 'paid' AND created_at > '2024-01-01';
-- ✓ SELECT * FROM orders WHERE created_at > '2024-01-01' AND status = 'paid';
-- 组合索引的最左前缀原则：idx(user_id, status)，如果查询条件缺少 user_id，索引失效
```

**分页优化**

深分页（`OFFSET` 值很大）是性能杀手：

```sql
-- ❌ 深分页：MySQL 需要跳过前 100000 条数据
SELECT * FROM orders LIMIT 100000, 10;

-- ✓ 游标分页：基于上一页最后一条记录的 ID
SELECT * FROM orders WHERE id > #{lastId} ORDER BY id LIMIT 10;

-- ✓ 延迟关联：先查 ID，再关联详情
SELECT * FROM orders 
INNER JOIN (SELECT id FROM orders ORDER BY id LIMIT 100000, 10) AS t 
ON orders.id = t.id;
```

## 四、异步处理：不让请求干等

### 4.1 同步 vs 异步

同步处理中，请求需要等待所有操作完成才返回。异步处理中，请求可以立即返回，实际操作在后台执行。

```
同步处理：
请求 → 操作1(10ms) → 操作2(20ms) → 操作3(15ms) → 响应
                                    总耗时：45ms

异步处理：
请求 → 响应（立即）→ 后台：操作1 → 操作2 → 操作3
              总耗时：0ms（对用户来说）
```

### 4.2 消息队列：异步解耦

消息队列是异步处理的核心组件，生产者和消费者通过消息队列解耦。

```java
// 生产者：发送消息
@Service
public class OrderService {
    @Autowired
    private KafkaTemplate<String, String> kafkaTemplate;
    
    public String createOrder(OrderRequest request) {
        Order order = orderMapper.insert(request);
        
        // 发送消息，异步通知其他服务
        kafkaTemplate.send("order.created", order.getId(), 
            JSON.toJSONString(order));
        
        return order.getId();  // 立即返回订单 ID
    }
}

// 消费者：处理消息
@Service
public class InventoryConsumer {
    @KafkaListener(topics = "order.created", groupId = "inventory-group")
    public void handleOrderCreated(ConsumerRecord<String, String> record) {
        Order order = JSON.parseObject(record.value(), Order.class);
        inventoryService.deductStock(order);  // 扣减库存
    }
}
```

### 4.3 CompletableFuture：Java 并发利器

`CompletableFuture` 是 Java 8 引入的异步编程利器，可以方便地将多个异步任务组合。

```java
@Service
public class UserDetailService {
    
    // 串行执行：总耗时 = t1 + t2 + t3
    public UserDetail getUserDetail(Long userId) {
        User user = userService.getUser(userId);          // 10ms
        List<Order> orders = orderService.getOrders(userId); // 20ms
        List<Product> favorites = favoriteService.getFavorites(userId); // 15ms
        
        return new UserDetail(user, orders, favorites);
    }
    
    // 并行执行：总耗时 = max(t1, t2, t3)
    public UserDetail getUserDetailAsync(Long userId) {
        CompletableFuture<User> userFuture = 
            CompletableFuture.supplyAsync(() -> userService.getUser(userId));
        CompletableFuture<List<Order>> ordersFuture = 
            CompletableFuture.supplyAsync(() -> orderService.getOrders(userId));
        CompletableFuture<List<Product>> favoritesFuture = 
            CompletableFuture.supplyAsync(() -> favoriteService.getFavorites(userId));
        
        // 等待所有任务完成
        CompletableFuture.allOf(userFuture, ordersFuture, favoritesFuture).join();
        
        return new UserDetail(
            userFuture.get(),
            ordersFuture.get(),
            favoritesFuture.get()
        );
    }
}
```

## 五、代码优化：消除性能瓶颈

### 5.1 减少对象创建

对象创建是有开销的，尤其是大对象。在高频调用的路径上，尽量复用对象。

```java
// ❌ 错误：在循环中创建对象
for (String item : items) {
    List<String> list = new ArrayList<>();  // 每次循环都创建新对象
    process(item, list);
}

// ✓ 正确：循环外创建对象
List<String> list = new ArrayList<>();
for (String item : items) {
    list.clear();  // 复用，循环使用
    process(item, list);
}

// ✓ 正确：使用对象池（高并发场景）
public class OrderPool {
    private static final ConcurrentLinkedQueue<Order> POOL = new ConcurrentLinkedQueue<>();
    
    public static Order acquire() {
        Order order = POOL.poll();
        return order != null ? order : new Order();
    }
    
    public static void release(Order order) {
        order.reset();  // 重置状态
        POOL.offer(order);
    }
}
```

### 5.2 减少锁竞争

锁是并发程序性能的大敌。减少锁竞争的方法：

**1. 缩小锁的范围**

```java
// ❌ 错误：锁住整个方法
public synchronized void process(Order order) {
    validate(order);           // 10ms
    saveToDatabase(order);      // 50ms
    sendNotification(order);   // 20ms
}

// ✓ 正确：只锁必要的部分
public void process(Order order) {
    validate(order);  // 不需要锁
    synchronized (this) {
        saveToDatabase(order);  // 只锁数据库操作
    }
    sendNotification(order);    // 不需要锁
}
```

**2. 读写分离**

读多写少场景，使用读写锁：

```java
private final ReadWriteLock rwLock = new ReentrantReadWriteLock();
private final Map<String, Object> cache = new HashMap<>();

public Object get(String key) {
    rwLock.readLock().lock();
    try {
        return cache.get(key);
    } finally {
        rwLock.readLock().unlock();
    }
}

public void put(String key, Object value) {
    rwLock.writeLock().lock();
    try {
        cache.put(key, value);
    } finally {
        rwLock.writeLock().unlock();
    }
}
```

### 5.3 优化集合操作

```java
// ❌ 低效：频繁的装箱拆箱
List<int[]> list = new ArrayList<>();
for (int i = 0; i < 1000; i++) {
    list.add(new int[]{i, i * 2});  // int[] 是对象
}

// ✓ 高效：使用基本类型集合
IntArrayList list = new IntArrayList();  // Trove/HPPC 库
for (int i = 0; i < 1000; i++) {
    list.add(i);
}

// ❌ 低效：HashMap 迭代
for (Map.Entry<String, Object> entry : map.entrySet()) {
    // ...
}

// ✓ 高效：直接遍历 values 或 keys
for (Object value : map.values()) {
    // ...
}
```

## 六、JVM 调优：让垃圾回收不拖后腿

### 6.1 延迟与吞吐量

JVM 的垃圾回收（GC）是 Java 应用延迟的主要来源之一。GC 调优的目标是**减少 GC 停顿时间**。

**吞吐量优先 vs 低延迟优先**

| 策略 | GC 收集器 | 适用场景 |
|---|---|---|
| **吞吐量优先** | Parallel GC（吞吐量优先） | 后台批处理，不关心延迟 |
| **低延迟优先** | G1 GC、ZGC、Shenandoah | Web 服务，要求低延迟 |

### 6.2 G1 GC 调优

G1（Garbage First）是目前主流的 GC 收集器，适合大多数场景。

**核心参数**

```bash
# 堆内存配置
-Xms4g -Xmx4g  # 建议设置相同，避免运行时调整

# G1 特有参数
-XX:MaxGCPauseMillis=100      # 最大 GC 停顿时间目标（100ms）
-XX:G1HeapRegionSize=4m       # Region 大小（1MB-32MB）
-XX:InitiatingHeapOccupancyPercent=45  # 触发并发标记的堆使用率
```

**调优思路**

```bash
# 场景一：GC 停顿时间过长
# 原因：老年代满了，Full GC 时间长
# 解决：增大堆内存，或调整 MaxGCPauseMillis

# 场景二：GC 频率太高
# 原因：对象创建太快，Minor GC 来不及回收
# 解决：增大年轻代，或优化代码减少对象创建

# 场景三：G1 预测不准
# 原因：Region 大小不合适
# 解决：调整 G1HeapRegionSize
```

### 6.3 ZGC：亚毫秒级延迟

对于追求极致低延迟的应用，ZGC 是一个选择：

```bash
# ZGC 配置
-Xms4g -Xmx4g
-XX:+UseZGC
-XX:MaxGCPauseMillis=10  # 最大停顿时间目标 10ms
```

ZGC 的特点：

- 停顿时间 < 1ms（无论堆多大）
- 停顿时间不随堆大小增加而增加
- 但吞吐量比 G1 低 10-20%

## 八、实战：设计一个 p99 < 100ms 的 API

### 8.1 需求分析

设计一个用户信息查询 API：

- QPS：10,000
- 延迟要求：P99 < 100ms
- 可用性：99.9%

### 8.2 架构设计

```
用户请求 → CDN（静态资源）→ API 网关 → 应用服务 → 多级缓存 → 数据库
                                    ↓
                            (本地缓存 10ms)
                                    ↓
                            (Redis 1ms)
                                    ↓
                            (MySQL 10ms)
```

### 8.3 关键设计点

**1. 多级缓存**

```java
@Service
public class UserApiService {
    // L1: 本地缓存（Caffeine）
    private final Cache<Long, User> localCache = Caffeine.newBuilder()
        .maximumSize(10000)
        .expireAfterWrite(1, TimeUnit.MINUTES)
        .build();
    
    // L2: Redis 缓存
    @Autowired
    private RedisTemplate<String, User> redisTemplate;
    
    public User getUser(Long userId) {
        // 1. 查本地缓存
        User user = localCache.getIfPresent(userId);
        if (user != null) {
            return user;
        }
        
        // 2. 查 Redis
        String redisKey = "user:" + userId;
        user = redisTemplate.opsForValue().get(redisKey);
        if (user != null) {
            localCache.put(userId, user);
            return user;
        }
        
        // 3. 查数据库
        user = userMapper.selectById(userId);
        if (user != null) {
            redisTemplate.opsForValue().set(redisKey, user, 30, TimeUnit.MINUTES);
            localCache.put(userId, user);
        }
        return user;
    }
}
```

**2. 连接池优化**

```java
// 数据库连接池
dataSource.setMaxActive(50);          // QPS 10000，单机 200 并发

// HTTP 连接池

    .connectionPool(new ConnectionPool(50, 5, TimeUnit.MINUTES))
```

**3. 异步处理非核心逻辑**

```java
@Service
public class UserApiService {
```

**5. 限流保护**

```java
@Configuration
public class RateLimitConfig {
    @Bean
    public FilterRegistrationBean<RateLimitFilter> rateLimitFilter() {
        FilterRegistrationBean<RateLimitFilter> bean = new FilterRegistrationBean<>();
        bean.setFilter(new RateLimitFilter());
        bean.addUrlPatterns("/api/users/*");
        return bean;
    }
}

public class RateLimitFilter implements Filter {
    private final RateLimiter rateLimiter = RateLimiter.create(12000); // 允许 12000 QPS
    
    @Override
    public void doFilter(ServletRequest request, ServletResponse response, FilterChain chain) {
        if (!rateLimiter.tryAcquire()) {
            HttpServletResponse resp = (HttpServletResponse) response;
            resp.setStatus(429);
            resp.getWriter().write("Too Many Requests");
            return;
        }
        chain.doFilter(request, response);
    }
}
```

## 九、面试追问

### 问题一：如何排查接口延迟高的问题？

**参考答案**：

**第一步：确认问题**

- 确认是偶发问题还是持续问题
- 确认是单机问题还是集群问题
- 确认是所有接口慢还是某个接口慢

**第二步：定位层级**

- 网络延迟：ping、traceroute
- 应用延迟：APM 工具（SkyWalking、Arthas）
- 数据库延迟：慢查询日志、EXPLAIN

**第三步：定位代码**

- Arthas trace：追踪方法调用链路耗时
- Arthas profiler：生成火焰图，定位 CPU 热点
- 日志分析：找到慢请求的日志

**第四步：修复验证**

- 修复后压测验证
- 持续监控，防止回归

### 问题二：如何设计一个高并发的计数器？

**参考答案**：

**方案一：Redis INCR**

```java
String key = "counter:" + date;
redisTemplate.opsForValue().increment(key);
```

优点：简单高性能，天然支持分布式

**方案二：Redis + 本地缓存混合**

```java
// 本地缓存汇总，定期同步到 Redis
private final LongAdder counter = new LongAdder();

// 高频更新，只更新本地
public void increment() {
    counter.increment();
}

// 定期同步到 Redis
@Scheduled(fixedRate = 1000)
public void syncToRedis() {
    long sum = counter.sumThenReset();
    if (sum > 0) {
        redisTemplate.opsForValue().increment("counter:daily", sum);
    }
}
```

### 问题三：如何避免缓存雪崩？

**参考答案**：

缓存雪崩是大量缓存同时过期，导致大量请求击穿到数据库。

**解决方案**：

- **过期时间随机化**：`TTL = base + random(0, 5分钟)`
- **热点数据永不过期**：设置 `-1` 或很长的 TTL，配合逻辑过期
- **多级缓存**：本地缓存 + Redis，即使 Redis 过期，本地缓存还能撑一下
- **服务降级**：缓存不可用时，返回默认值或友好提示

```java
// 过期时间随机化
int baseTTL = 1800; // 30 分钟
int randomTTL = ThreadLocalRandom.current().nextInt(300); // 0-5 分钟随机
redisTemplate.opsForValue().set(key, value, baseTTL + randomTTL, TimeUnit.SECONDS);
```

## 思考题

### 题目一：缓存 vs 数据库的一致性

假设你使用 Cache Aside 策略，读请求缓存 miss 后回填缓存，写请求更新数据库后删除缓存。如果在「删除缓存」和「更新数据库」之间，有另一个请求查到了旧数据并回填缓存，会导致缓存和数据库不一致。这个问题如何解决？

**提示**：考虑延迟双删、消息队列、订阅 binlog 等方案。

### 题目二：延迟和吞吐量的权衡

假设你的 API 目前 P99 延迟是 50ms，QPS 是 5000。如果业务要求 QPS 提升到 10000，你会如何优化？

**提示**：考虑水平扩展（加机器）、垂直优化（提性能）、异步化（削峰）等方案。

### 题目三：ZGC vs G1

ZGC 的停顿时间很短，但吞吐量比 G1 低 10-20%。什么场景下应该选 ZGC？什么场景下应该选 G1？

**提示**：考虑业务对延迟的敏感程度、对吞吐量的要求、JVM 版本等因素。

## 总结

高性能架构设计的核心是**减少延迟、提高吞吐量**，让系统「又快又稳」。

关键要点回顾：

- **延迟分解**：找出瓶颈在哪里，优化最慢的环节
- **P99 指标**：关注长尾延迟，不只是平均值
- **多级缓存**：本地缓存 → Redis → 数据库，热点数据快速响应
- **连接复用**：HTTP Keep-Alive、连接池，减少网络开销
- **异步处理**：消息队列、CompletableFuture，解放线程资源
- **JVM 调优**：G1、ZGC，根据业务需求选择合适的 GC 策略
- **性能测试**：压测验证，持续监控

记住：**性能优化的本质是「让快的更快，让慢的变快」**。不要在已经很快的环节过度优化，而是在瓶颈环节精准发力。
