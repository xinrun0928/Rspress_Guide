# 高并发架构设计：应对百万级并发请求

2019 年双十一，京东商城的订单峰值达到 **每秒 54 万笔**，是 2018 年的两倍。这意味着什么？

每秒 54 万笔订单，意味着每秒有 54 万次数据库写入、54 万次库存扣减、54 万次支付回调。如果你的系统是单体架构，一台服务器每秒能处理 1000 笔订单，那需要 540 台服务器才能撑住。更重要的是，这 540 台服务器之间还需要协调——库存不能超卖、订单不能重复、支付必须成功。

这就是高并发架构设计的核心挑战：**不是单机能跑多快，而是如何让一群机器协同工作，应对巨大的并发压力**。

很多人对高并发有一个误解：以为它只是「把 QPS 做大」。但 QPS 只是表象，真正的高并发问题，是资源竞争、数据一致性、系统稳定性这些深层问题。双十一的 54 万 QPS 背后，是一套经过多年演进、无数踩坑打磨出来的高并发架构。

这篇文章，我会从高并发的本质讲起，带你掌握高并发架构设计的核心套路，最后结合京东、美团等大厂的实际案例，让你真正理解高并发系统是怎么「炼」成的。

## 一、高并发的本质：不是快，是有序

### 1.1 什么是高并发

高并发（High Concurrency）是指系统在同一时间段内处理大量请求的能力。衡量高并发的核心指标是 **QPS（Queries Per Second，每秒查询数）** 和 **TPS（Transactions Per Second，每秒事务数）**。

| 并发规模 | QPS 范围 | 典型场景 |
|---|---|---|
| 普通并发 | < 1,000 QPS | 中小型网站、内部系统 |
| 高并发 | 1,000 - 100,000 QPS | 主流互联网应用 |
| 极高并发 | 100,000 - 1,000,000 QPS | 头部互联网平台 |
| 超级并发 | > 1,000,000 QPS | 双十一、春晚红包 |

但 QPS 不是越高越好。真正的高并发系统，不只是「跑得快」，而是：

- **请求能被正确处理**：不会超卖、不会数据不一致
- **系统能稳定运行**：不会因为流量激增而崩溃
- **资源能高效利用**：不会因为资源争抢而大量浪费

### 1.2 高并发的三大元凶

高并发系统的问题，归结起来有三个核心矛盾：

**资源有限 vs 需求无限**

服务器资源（CPU、内存、磁盘、网络）是有限的，但请求是无限的。当请求量超过系统的处理能力，就会出现排队、等待、甚至崩溃。

**同步阻塞 vs 异步需求**

传统的同步架构，每个请求占用一个线程/连接，直到处理完成才释放。在高并发场景下，这种模式会导致线程资源耗尽。

**数据一致性 vs 性能**

分布式系统需要在多个节点之间同步数据，这需要时间。如果每次操作都等待数据同步完成，性能会大打折扣；如果不同步，又会面临数据不一致的风险。

### 1.3 从 C10K 到 C10M：并发问题的演进

互联网的并发挑战，有一个经典的演进历程：

| 里程碑 | 并发连接数 | 核心问题 | 解决方案 |
|---|---|---|---|
| **C10K**（1999 年） | 1 万 | select/poll 模型无法高效处理大量连接 | epoll/kqueue |
| **C100K**（2005 年） | 10 万 | 线程创建开销成为瓶颈 | 线程池、异步 I/O |
| **C1M**（2010 年） | 100 万 | 单机资源成为瓶颈 | 多核优化、零拷贝 |
| **C10M**（2015 年） | 1000 万 | 传统 Linux 网络栈成为瓶颈 | DPDK、用户态网络 |

:::tip 并发模型的演进

C10K 时代，大家关心的是「如何用少量线程处理大量连接」——答案是 I/O 多路复用（epoll）。

C100K 时代，大家关心的是「如何让线程处理更多连接」——答案是线程池 + 异步 I/O。

C1M/C10M 时代，大家关心的是「如何突破单机极限」——答案是绕过内核、专用硬件。

作为 Java 工程师，我们主要关注的是 C100K 到 C1M 这个层次——用好现有的异步框架（Netty）和并发工具（线程池、连接池），足够应对大多数高并发场景。

:::

## 二、高并发的核心架构：分层 + 异步 + 缓存

### 2.1 分层架构：高并发的第一原则

高并发系统设计的第一个原则是**分层**。不分层的系统，所有请求都挤在同一个处理链路上，资源争抢严重。

典型的分层架构：

```bash
                        ┌─────────────────────────────────────┐
                        │           用户请求入口               │
                        │        (CDN + DNS + 负载均衡)        │
                        └────────────────┬────────────────────┘
                                         │
                        ┌────────────────▼────────────────────┐
                        │           接入层                    │
                        │      (API 网关 + 限流 + 风控)        │
                        └────────────────┬────────────────────┘
                                         │
                        ┌────────────────▼────────────────────┐
                        │           应用层                    │
                        │     (业务逻辑 + 服务编排)           │
                        └───────┬──────────────┬───────────────┘
                                │              │
               ┌────────────────▼──┐  ┌────────▼───────────────┐
               │     缓存层         │  │      数据访问层        │
               │ (本地 + 分布式)    │  │   (分库分表 + 读写分离) │
               └───────────────────┘  └───────────────────────┘
                                │              │
               ┌────────────────▼──────────────▼───────────────┐
               │              数据存储层                        │
               │      (MySQL + Redis + Elasticsearch)          │
               └───────────────────────────────────────────────┘
```

每一层都有明确的职责：

| 层级 | 职责 | 关键技术 |
|---|---|---|
| **接入层** | 流量清洗、安全防护、负载均衡 | CDN、DNS 智能解析、LVS、Nginx |
| **应用层** | 业务逻辑处理、服务编排 | Spring Cloud、Dubbo、微服务框架 |
| **缓存层** | 热点数据加速、减轻后端压力 | 本地缓存（Caffeine）、分布式缓存（Redis） |
| **数据层** | 数据持久化、查询优化 | MySQL 分库分表、读写分离、ES |

### 2.2 异步处理：解放线程资源

高并发的第二个原则是**异步**。同步处理会让线程阻塞，异步处理可以让线程在等待 I/O 时去处理其他请求。

**异步的三种模式**

**1. 消息队列异步化**

```java
// 同步模式：用户下单后，需要等待库存扣减、支付、发货通知全部完成
public Order placeOrder(OrderRequest request) {
    inventoryService.deduct(request.getItemId(), request.getQuantity()); // 同步
    paymentService.pay(request.getPaymentInfo()); // 同步
    notificationService.notify(request.getUserId()); // 同步
    return order;
}

// 异步模式：下单后，发送消息，由后台服务异步处理
public Order placeOrder(OrderRequest request) {
    Order order = orderService.create(request);
    kafkaTemplate.send("order.created", order.getId()); // 异步
    return order; // 立即返回
}

// 后台服务消费消息
@KafkaListener(topics = "order.created")
public void handleOrderCreated(String orderId) {
    inventoryService.deduct(orderId);
    paymentService.process(orderId);
    notificationService.notify(orderId);
}
```

**2. CompletableFuture 异步编排**

```java
// 同步调用：三个服务串行执行，总耗时 = sum(各服务耗时)
User user = userService.getUser(userId);
List<Order> orders = orderService.getOrders(userId);
List<Product> products = productService.getProducts(orders);

// 异步调用：三个服务并行执行，总耗时 = max(各服务耗时)
CompletableFuture<User> userFuture = CompletableFuture.supplyAsync(() -> userService.getUser(userId));
CompletableFuture<List<Order>> ordersFuture = CompletableFuture.supplyAsync(() -> orderService.getOrders(userId));
CompletableFuture<List<Product>> productsFuture = CompletableFuture.supplyAsync(() -> productService.getProducts(userId));

CompletableFuture.allOf(userFuture, ordersFuture, productsFuture).join();

User user = userFuture.get();
List<Order> orders = ordersFuture.get();
List<Product> products = productsFuture.get();
```

**3. Spring @Async 异步执行**

```java
@Service
public class OrderService {
    
    @Async("asyncExecutor")
    public void sendNotification(String userId) {
        // 异步执行，不阻塞主线程
        notificationService.send(userId);
    }
    
    public Order placeOrder(OrderRequest request) {
        Order order = createOrder(request);
        sendNotification(request.getUserId()); // 异步发送通知
        return order;
    }
}

@Configuration
public class AsyncConfig {
    @Bean("asyncExecutor")
    public Executor asyncExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(10);
        executor.setMaxPoolSize(50);
        executor.setQueueCapacity(100);
        executor.setThreadNamePrefix("async-");
        executor.initialize();
        return executor;
    }
}
```

### 2.3 多级缓存：性能优化的第一利器

高并发的第三个原则是**缓存**。缓存是性能优化的第一利器，能大幅减少数据库压力。

**缓存架构设计**

```bash
用户请求 → 本地缓存（Caffeine）→ 分布式缓存（Redis Cluster）→ 数据库
          ↓命中？              ↓命中？
           0.1ms               1ms                10ms
```

**本地缓存**：访问速度最快（纳秒级），但容量有限，不支持跨进程共享。

```java
// Caffeine 本地缓存
LoadingCache<Long, User> localCache = Caffeine.newBuilder()
    .maximumSize(10000)           // 最大 1 万条
    .expireAfterWrite(10, TimeUnit.MINUTES)  // 写入 10 分钟后过期
    .recordStats()                // 开启统计
    .build(userId -> userMapper.selectById(userId));

// 读取时自动回源
User user = localCache.get(userId);
```

**分布式缓存**：容量大，支持跨进程共享，但有网络开销（毫秒级）。

```java
// Redis 分布式缓存
public User getUser(Long userId) {
    // 先查本地缓存
    User user = localCache.getIfPresent(userId);
    if (user != null) {
        return user;
    }
    
    // 本地缓存 miss，查分布式缓存
    String key = "user:" + userId;
    user = (User) redisTemplate.opsForValue().get(key);
    if (user != null) {
        localCache.put(userId, user);
        return user;
    }
    
    // 分布式缓存也 miss，查数据库
    user = userMapper.selectById(userId);
    if (user != null) {
        redisTemplate.opsForValue().set(key, user, 30, TimeUnit.MINUTES);
        localCache.put(userId, user);
    }
    return user;
}
```

**缓存策略**

| 策略 | 描述 | 适用场景 |
|---|---|---|
| **Cache Aside** | 读：缓存 miss 后查库回填；写：先写库再删缓存 | 大多数读多写少场景 |
| **Read Through** | 缓存负责从数据源加载数据，应用不感知 | 缓存代理数据源 |
| **Write Through** | 写：缓存和数据源同步更新 | 需要强一致性 |
| **Write Behind** | 写：先写缓存，异步写数据源 | 写入性能要求高 |

:::warning 缓存一致性问题

缓存最麻烦的问题是**数据一致性**。常见的问题场景：

**缓存双删**：先删缓存，再更新数据库，再删缓存。但第二步和第三步之间，可能有其他请求把旧数据回填到缓存。

**缓存过期**：缓存过期后，多个请求同时发现 miss，都去查数据库回填。

**解决方案**：

- **延迟双删** + 消息队列异步处理
- **设置较短的过期时间** + **数据更新时主动删除缓存**
- **使用分布式锁**保证只有一个请求回填缓存

:::

## 三、连接池：资源复用是关键

### 3.1 数据库连接池

数据库连接是「重量级」资源，创建和销毁的开销很大（涉及 TCP 三次握手、SSL 握手、认证等）。连接池通过复用连接，大幅提升性能。

```java
// Druid 连接池配置
@Configuration
public class DataSourceConfig {
    @Bean
    public DataSource dataSource() {
        DruidDataSource dataSource = new DruidDataSource();
        dataSource.setUrl("jdbc:mysql://localhost:3306/test");
        dataSource.setUsername("root");
        dataSource.setPassword("123456");
        
        // 连接池核心参数
        dataSource.setInitialSize(10);      // 初始连接数
        dataSource.setMinIdle(10);          // 最小空闲连接数
        dataSource.setMaxActive(100);       // 最大连接数
        dataSource.setMaxWait(60000);       // 获取连接最大等待时间
        
        // 连接有效性检测
        dataSource.setTestWhileIdle(true);
        dataSource.setTestOnBorrow(false);
        dataSource.setValidationQuery("SELECT 1");
        
        // 连接泄漏检测
        dataSource.setRemoveAbandoned(true);
        dataSource.setRemoveAbandonedTimeout(300);
        
        return dataSource;
    }
}
```

**连接池参数调优**

| 参数 | 默认值 | 调整建议 |
|---|---|---|
| `maxActive` | 20 | CPU 核心数的 2-4 倍 |
| `minIdle` | 初始值 | 保持与预估并发数匹配 |
| `maxWait` | -1（无限等待） | 设置合理的超时（如 5-10 秒） |
| `testWhileIdle` | false | 开启，避免连接失效 |

### 3.2 HTTP 连接池

HTTP 请求同样面临连接创建销毁的开销。HTTP Keep-Alive 和连接池是标准解决方案。

```java
// OkHttp 连接池配置
OkHttpClient client = new OkHttpClient.Builder()
    .connectTimeout(10, TimeUnit.SECONDS)   // 连接超时
    .readTimeout(30, TimeUnit.SECONDS)     // 读取超时
    .writeTimeout(30, TimeUnit.SECONDS)    // 写入超时
    .connectionPool(new ConnectionPool(
        100,           // 最大空闲连接数
        5, TimeUnit.MINUTES  // 空闲连接存活时间
    ))
    .retryOnConnectionFailure(true)         // 失败重试
    .build();
```

### 3.3 线程池：服务的「发动机」

线程池是高并发系统的核心组件，负责管理线程资源。

```java
// 线程池配置
@Configuration
public class ThreadPoolConfig {
    
    // IO 密集型任务：线程数 = CPU 核心数 * 2
    // CPU 密集型任务：线程数 = CPU 核心数 + 1
    // 混合型任务：根据比例分配
    
    @Bean("io密集型线程池")
    public ExecutorService ioIntensiveExecutor() {
        return new ThreadPoolExecutor(
            16,                           // 核心线程数
            64,                           // 最大线程数
            60L, TimeUnit.SECONDS,        // 空闲线程存活时间
            new LinkedBlockingQueue<>(1000),  // 队列容量
            new ThreadFactoryBuilder().setNameFormat("io-task-%d").build(),
            new ThreadPoolExecutor.CallerRunsPolicy()  // 拒绝策略：调用者执行
        );
    }
}
```

**线程池参数调优**

| 参数 | 含义 | 调优建议 |
|---|---|---|
| `corePoolSize` | 核心线程数 | 根据任务类型（IO/CPU）计算 |
| `maximumPoolSize` | 最大线程数 | corePoolSize 的 2-4 倍 |
| `queueCapacity` | 队列容量 | 根据峰值预估设置 |
| `keepAliveTime` | 空闲线程存活时间 | 建议 60 秒以上 |

## 四、并发控制：守住数据的底线

### 4.1 乐观锁 vs 悲观锁

并发控制的核心问题是如何在多个线程同时访问数据时，保证数据的一致性。

**悲观锁**：假设并发问题一定会发生，先加锁再操作。

```sql
-- MySQL SELECT FOR UPDATE
BEGIN;
SELECT stock FROM product WHERE id = 1 FOR UPDATE;  -- 加锁
-- 如果 stock >= 1，更新
UPDATE product SET stock = stock - 1 WHERE id = 1;
COMMIT;
```

悲观锁的优点是**保证数据一致性**，缺点是**性能差**（锁等待时间长），适合**写并发多**的场景。

**乐观锁**：假设并发问题不常发生，不加锁，提交时检查是否有冲突。

```sql
-- 版本号控制
UPDATE product 
SET stock = stock - 1, version = version + 1 
WHERE id = 1 AND version = 1;  -- 检查版本号

-- 影响行数为 0 说明有冲突，需要重试
```

乐观锁的优点是**性能好**（无锁等待），缺点是**有冲突时需要重试**，适合**读多写少**的场景。

| 场景 | 推荐方案 | 理由 |
|---|---|---|
| 高并发库存扣减 | 乐观锁 + Redis 原子操作 | 扣减是写操作，但频率高不适合长时间锁 |
| 余额扣款 | 悲观锁 | 金额敏感，不允许并发冲突 |
| 订单状态修改 | 乐观锁（版本号） | 大多数场景读多写少 |

### 4.2 分布式锁：跨进程的并发控制

单机锁只能控制单进程内的并发。分布式系统需要分布式锁。

**Redis 分布式锁**

```java
@Service
public class DistributedLockService {
    
    @Autowired
    private RedisTemplate<String, String> redisTemplate;
    
    private static final String LOCK_PREFIX = "lock:";
    
    // 获取锁（SET NX EX）
    public boolean tryLock(String key, String value, long expireTime) {
        String lockKey = LOCK_PREFIX + key;
        Boolean result = redisTemplate.opsForValue()
            .setIfAbsent(lockKey, value, expireTime, TimeUnit.SECONDS);
        return Boolean.TRUE.equals(result);
    }
    
    // 释放锁（Lua 脚本保证原子性）
    public void unlock(String key, String value) {
        String lockKey = LOCK_PREFIX + key;
        String script = 
            "if redis.call('get', KEYS[1]) == ARGV[1] then " +
            "    return redis.call('del', KEYS[1]) " +
            "else " +
            "    return 0 " +
            "end";
        redisTemplate.execute(
            new DefaultRedisScript<>(script, Long.class),
            Collections.singletonList(lockKey),
            value
        );
    }
    
    // 业务中使用
    public boolean deductStock(Long productId, Integer quantity) {
        String lockKey = "stock:" + productId;
        String lockValue = UUID.randomUUID().toString();
        
        try {
            if (tryLock(lockKey, lockValue, 30)) {
                // 获取锁成功，执行业务
                return stockMapper.deductStock(productId, quantity) > 0;
            }
            return false;  // 获取锁失败
        } finally {
            unlock(lockKey, lockValue);  // 释放锁
        }
    }
}
```

**分布式锁的注意事项**

- **锁的粒度要合适**：锁太粗（锁整个库存）并发度低，锁太细（锁每个 SKU）管理复杂
- **锁的超时要合理**：太短可能业务还没执行完就释放，太长可能影响故障恢复
- **要考虑锁的可靠性**：Redis 单节点不可靠，生产环境用 RedLock

### 4.3 限流：保护系统的最后一道防线

限流是高并发系统的必备保护机制，防止瞬时流量压垮系统。

**限流算法**

| 算法 | 原理 | 优点 | 缺点 |
|---|---|---|---|
| **计数器** | 固定时间窗口内计数 | 简单 | 临界突变 |
| **滑动窗口** | 滑动时间窗口计数 | 精确 | 实现复杂 |
| **漏桶** | 请求以固定速率处理 | 平稳 | 不适合突发流量 |
| **令牌桶** | 令牌以固定速率生成，请求消耗令牌 | 允许突发 | 实现稍复杂 |

**令牌桶实现（Guava RateLimiter）**

```java
// 创建限流器：每秒产生 1000 个令牌
RateLimiter rateLimiter = RateLimiter.create(1000);

public void handleRequest(Request request) {
    // 获取一个令牌（阻塞等待）
    rateLimiter.acquire();
    
    // 执行业务逻辑
    doProcess(request);
}
```

**分布式限流（Redis + Lua）**

```lua
-- 令牌桶算法 Lua 实现
local key = KEYS[1]
local rate = tonumber(ARGV[1])
local capacity = tonumber(ARGV[2])
local now = tonumber(ARGV[3])
local requested = tonumber(ARGV[4])

local fill_time = capacity / rate
local ttl = math.floor(fill_time * 2)

local last_tokens = tonumber(redis.call('get', key .. ':tokens') or capacity)
local last_refreshed = tonumber(redis.call('get', key .. ':time') or now)

local delta = math.max(0, now - last_refreshed)
local filled = math.min(capacity, last_tokens + delta * rate)
local allowed = filled - requested

if allowed >= 0 then
    redis.call('set', key .. ':tokens', allowed)
    redis.call('set', key .. ':time', now)
    return 1
else
    redis.call('set', key .. ':tokens', filled)
    redis.call('set', key .. ':time', now)
    return 0
end
```

## 五、容量规划：兵马未动，粮草先行

### 5.1 如何计算系统容量

容量规划是确定系统需要多少资源（服务器、带宽、存储）来支撑业务目标。

**计算公式**

```
所需服务器数量 = 目标 QPS / 单机 QPS * 冗余系数
```

**示例**

假设：
- 目标 QPS = 100,000
- 单机 QPS = 1,000（根据压测结果）
- 冗余系数 = 1.5（考虑流量波动、故障转移）

```
所需服务器数量 = 100,000 / 1,000 * 1.5 = 150 台
```

### 5.2 压测：验证容量规划

纸上算得再漂亮，也不如实际压测来得真实。

**压测工具**

| 工具 | 适用场景 | 特点 |
|---|---|---|
| **JMeter** | 功能测试、性能测试 | 功能全面，上手简单 |
| **wrk** | 高性能 HTTP 压测 | 轻量，性能高 |
| **ab** | Apache Bench，简单压测 | 单机单协议，简单快速 |
| **Locust** | 分布式 Python 压测 | 脚本化，灵活 |
| **Gatling** | Scala 压测 | 报告漂亮，适合 CI 集成 |

**压测流程**

```bash
# 使用 wrk 进行 HTTP 压测
wrk -t12 -c400 -d30s http://your-api-endpoint

# 参数说明
# -t12: 12 个线程

# -d30s: 持续 30 秒
```

**压测关注指标**

- **QPS/TPS**：系统吞吐量
- **延迟分布**：P50、P95、P99
- **错误率**：请求失败的比例
- **资源利用率**：CPU、内存、IO、网络

## 六、实战案例：京东双十一的高并发架构

### 6.1 流量特征

京东双十一的流量特征是**脉冲式**：零点时刻流量瞬间爆发，峰值是平时的几十倍甚至上百倍。

这种流量模式带来的挑战：

- 瞬时并发极高，服务器必须「弹性扩容」
- 大部分请求集中在前几分钟，热点商品竞争激烈
- 库存扣减、订单创建的压力集中在零点前后

### 6.2 核心架构

**流量分层**

```bash
用户端 → CDN → LVS → Nginx → 应用网关 → 业务服务 → 数据层
         ↓
      静态资源（JS/CSS/图片）
         ↓
      限流防护（提前拦截无效请求）
```

**库存系统设计**

京东的库存系统采用**多层缓存 + 异步确认**的架构：

```bash
1. 用户下单 → 预扣库存（Redis 原子扣减）
                     ↓ 成功
2. 返回下单成功 → 订单服务创建订单
                     ↓ 消息
3. 库存服务异步确认 → 真正扣减数据库库存
                     ↓ 失败
4. 补偿服务 → 回补 Redis 库存，退款
```

这种设计保证了**下单体验的流畅**（Redis 扣减是毫秒级），同时通过**异步确认**保证了**数据的最终一致性**。

**限流与熔断**

- **多级限流**：CDN 限流 → LVS 限流 → 应用限流，逐层拦截
- **热点探测**：实时识别热点商品，提前预热缓存
- **熔断降级**：库存服务不可用时，自动降级为「排队购买」模式

### 6.3 关键设计点

| 设计点 | 方案 | 效果 |
|---|---|---|
| 库存预热 | 提前将热点商品库存加载到 Redis | 零点瞬间爆发时有缓存可查 |
| 订单异步化 | 下单后立即返回，订单创建异步处理 | 用户体验好，峰值压力分散 |
| 流量错峰 | 优惠券发放分批，支付分时段 | 削峰填谷，避免瞬时压力 |
| 多级缓存 | 本地缓存 → Redis → 数据库 | 热点数据查询 < 1ms |

## 七、面试追问

### 问题一：如何设计一个秒杀系统？

**参考答案**：

秒杀系统的核心挑战是**库存的并发扣减**。设计方案的关键点：

**1. 流量拦截**：在网关层限流，拦截大部分无效请求

**2. 热点探测**：识别秒杀热点商品，提前预热缓存

**3. 库存预扣减**：Redis 原子扣减，快速响应

**4. 异步下单**：下单成功不代表秒杀成功，需要后台异步确认

**5. MQ 解耦**：秒杀请求写入消息队列，后台服务慢慢处理

**6. 熔断降级**：库存服务不可用时，返回「系统繁忙」

### 问题二：消息队列如何保证消息不丢失？

**参考答案**：

消息不丢失需要三个环节都做保障：

**生产者端**：

- 发送确认机制（Confirm）
- 失败重试 + 落库记录

**MQ 端**：

- 持久化消息到磁盘
- 主从复制保证高可用

**消费者端**：

- 手动 ACK，确认处理完成后再确认
- 处理失败不确认，消息会被重新投递

```java
// RocketMQ 可靠消息示例
producer.send(message, new SendCallback() {
    @Override
    public void onSuccess(SendResult sendResult) {
        // 消息发送成功
    }
    
    @Override
    public void onException(Throwable e) {
        // 发送失败，保存到数据库，定时重试
        saveToDB(message);
    }
});

consumer.registerMessageListener(new MessageListenerConcurrently() {
    @Override
    public ConsumeConcurrentlyStatus consumeMessage(List<MessageExt> msgs, ConsumeConcurrentlyContext context) {
        for (MessageExt msg : msgs) {
            try {
                processMessage(msg);
                return ConsumeConcurrentlyStatus.CONSUME_SUCCESS;  // 手动确认
            } catch (Exception e) {
                return ConsumeConcurrentlyStatus.RECONSUME_LATER;  // 失败重试
            }
        }
        return ConsumeConcurrentlyStatus.CONSUME_SUCCESS;
    }
});
```

### 问题三：如何处理超卖问题？

**参考答案**：

超卖是秒杀系统的核心问题。解决方案：

**1. 数据库层面加约束**

```sql
UPDATE inventory SET stock = stock - 1 
WHERE product_id = ? AND stock > 0;
-- 影响行数为 0 说明库存不足
```

**2. Redis 原子操作**

```java
// Lua 脚本保证原子性
String script = 
    "local stock = redis.call('get', KEYS[1]) " +
    "if tonumber(stock) >= tonumber(ARGV[1]) then " +
    "    redis.call('decrby', KEYS[1], ARGV[1]) " +
    "    return 1 " +
    "else " +
    "    return 0 " +
    "end";
```

**3. 乐观锁 + 版本号**

```sql
UPDATE inventory SET stock = stock - 1, version = version + 1 
WHERE product_id = ? AND version = ? AND stock >= 1;
```

## 总结

高并发系统设计的核心是**分层 + 异步 + 缓存**，配合**并发控制**和**容量规划**，构建一个能应对百万级请求的系统。

关键要点回顾：

- **分层架构**：接入层 → 应用层 → 缓存层 → 数据层，每层职责明确
- **异步处理**：消息队列、CompletableFuture、Spring @Async，解放线程资源
- **多级缓存**：本地缓存 → 分布式缓存 → 数据库，热点数据快速响应
- **连接池**：数据库连接池、HTTP 连接池、线程池，资源复用
- **并发控制**：乐观锁、悲观锁、分布式锁、限流，守住数据底线
- **容量规划**：计算公式 + 压测验证，兵马未动粮草先行

记住：**高并发不是追求数字，而是追求稳定**。能稳定支撑业务峰值的设计，就是好的设计。
