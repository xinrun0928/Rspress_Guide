# 高可用架构设计：如何保证 7×24 小时可用

2019 年 3 月 3 日凌晨，阿里云宕机，华东地区的多个服务受到影响，大量用户无法访问。事后分析发现，原因是一个服务器集群的磁盘损坏，导致数据丢失。事后复盘：如果是多副本架构，单台磁盘损坏不应该影响服务可用性。

2021 年 7 月，Fastly CDN 发生全球性故障，导致 GitHub、Reddit、Amazon 等大量网站无法访问。事后分析：一次软件升级中的 bug，导致全球范围内的 PoP（Point of Presence）节点同时故障。如果有灰度发布机制，这个问题可能只影响部分用户。

这就是高可用的意义：**在一切正常的时候，你感觉不到它的存在；在故障发生的时候，它是你最后的防线**。

高可用不是「不惜一切代价保证服务正常」，而是在**成本、复杂度、可用性**之间找到平衡。99% 的可用性和 99.9% 的可用性，差距是 8.76 小时/年。但从 99.9% 到 99.99%，成本可能要翻倍。所以，高可用设计的核心问题是：**你的业务需要几个 9？愿意为此付出多少成本？**

这篇文章，我会从高可用的本质讲起，带你掌握高可用设计的核心模式，最后结合 AWS 的实际案例，让你理解大厂是如何做到 99.99% 可用性的。

## 一、高可用的本质：几个 9 的含义

### 1.1 什么是高可用

高可用（High Availability，HA）是指系统在发生硬件故障、软件故障、人为错误等异常情况时，仍然能够正常提供服务的能力。

衡量高可用的核心指标是**正常运行时间（Uptime）**：

| 可用性 | 年停机时间 | 典型场景 |
|---|---|---|
| 99%（两个 9） | 87.6 小时 | 内部系统、开发环境 |
| 99.9%（三个 9） | 8.76 小时 | 普通互联网应用 |
| 99.99%（四个 9） | 52.6 分钟 | 金融支付、核心业务 |
| 99.999%（五个 9） | 5.26 分钟 | 电信级、军事级系统 |

99% 的可用性听起来很高，但换算成停机时间：每天宕机 14 分钟，每年宕机 87 小时。这对于核心业务来说，是不可接受的。

### 1.2 可用性的计算

对于一个由多个组件串联组成的系统，系统的整体可用性等于各组件可用性的乘积：

```
系统可用性 = 组件A可用性 × 组件B可用性 × 组件C可用性 × ...
```

假设每个组件的可用性是 99.9%，三个组件串联后：

```
系统可用性 = 99.9% × 99.9% × 99.9% = 99.7%
```

可以看到，**串联的组件越多，系统的可用性越低**。这是分布式系统的本质挑战。

**提升可用性的方法**

1. **减少组件数量**：简化架构，减少故障点
2. **提升单个组件可用性**：让每个组件更可靠
3. **冗余部署**：多个组件并行，单点故障不影响整体

### 1.3 高可用 vs 高可靠 vs 高一致

三个概念容易混淆：

| 概念 | 定义 | 目标 |
|---|---|---|
| **高可用（Availability）** | 系统能及时响应请求 | 「能提供服务」 |
| **高可靠（Reliability）** | 系统长时间无故障运行 | 「不出问题」 |
| **高一致（Consistency）** | 数据在各节点间保持同步 | 「数据正确」 |

高可用 ≠ 高可靠。一个系统可以随时响应，但数据可能是错的（高可用低一致）。一个系统可以保证数据正确，但响应很慢（高可靠低可用）。

:::tip CAP 定理的启示

CAP 定理告诉我们：在分布式系统中，一致性（Consistency）、可用性（Availability）、分区容错（Partition Tolerance）三者只能同时满足两个。

在真实系统中，分区（网络故障）是不可避免的，所以实际上是在 **C（一致性）** 和 **A（可用性）** 之间做权衡。

- 强一致系统（如金融交易）：选择 CP
- 高可用系统（如社交Feed）：选择 AP

这个权衡，会贯穿整个高可用设计。

:::

## 二、单点故障：识别与消除

### 2.1 什么是单点故障

单点故障（Single Point of Failure，SPOF）是指系统中只有一个实例的组件。一旦这个组件故障，整个系统都会受到影响。

常见的单点故障点：

| 组件 | 风险 | 解决方案 |
|---|---|---|
| **应用服务器** | 单机故障 | 多机部署 + 负载均衡 |
| **数据库** | 单机故障 | 主从复制 + 故障转移 |
| **缓存** | 单机故障 | Redis Sentinel / Cluster |
| **负载均衡器** | 单机故障 | 双机热备（Keepalived） |
| **域名解析** | DNS 故障 | 多 DNS 服务商 |
| **CDN** | 单节点故障 | 多 CDN 厂商 |

### 2.2 消除单点故障的方法

**1. 冗余部署（Redundancy）**

关键组件部署多个实例，单个故障不影响整体：

```bash
# Nginx 负载均衡器 + Keepalived VIP 漂移
┌─────────────────┐     ┌─────────────────┐
│   Nginx Master  │────▶│  Application    │
│   192.168.1.1   │     │  192.168.1.10   │
└────────┬────────┘     └─────────────────┘
         │                      ▲
    Keepalived                  │
    VRRP 协议              ┌────┴────┐
         │                  │ Application│
┌────────┴────────┐         │  192.168.1.11 │
│   Nginx Backup  │────────▶└─────────────────┘
│   192.168.1.2   │
└─────────────────┘
```

**2. 主从复制（Replication）**

数据存储使用主从架构，主库故障时自动切换到从库：

```sql
-- MySQL 主从复制配置
-- 主库
[mysqld]
server-id=1
log-bin=mysql-bin
sync-binlog=1

-- 从库
[mysqld]
server-id=2
relay-log=relay-bin
read-only=1
```

**3. 集群模式（Clustering）**

多个节点组成集群，协同工作，故障自动转移：

```bash
# Redis Sentinel 架构
┌─────────────┐
│   Client    │
└──────┬──────┘
       │
       ▼
┌─────────────────────────────────┐
│        Redis Sentinel           │
│   (监控 + 故障转移 + 配置管理)    │
└──────┬──────────┬──────────────┘
       │          │
       ▼          ▼
┌──────────┐  ┌──────────┐
│ Redis    │  │ Redis    │
│ Master   │  │ Slave    │
│  (写)    │  │  (读)    │
└──────────┘  └──────────┘
```

## 三、高可用架构模式

### 3.1 主备模式（Active-Passive）

主备模式是指一台主机对外提供服务，多台备机随时待命。主机故障时，备机自动切换接管服务。

```
正常状态：
┌──────────┐
│  Active   │ ← 正常工作
└────┬─────┘
     │ 同步
┌────▼─────┐
│ Passive  │ ← 热备/冷备
└──────────┘

故障状态：
┌──────────┐
│  Active   │ ← 故障！停止工作
└────┬─────┘
     │ 失联
┌────▼─────┐
│ Passive  │ ← 自动提升为主机，接管服务
└──────────┘
```

**特点**：

- 优点：简单、可靠、成本低
- 缺点：资源利用率低（备机平时不工作）、切换有延迟

**适用场景**：对可用性要求不是极端高、成本敏感的场景。

### 3.2 双主模式（Active-Active）

双主模式是指两台或多台主机同时对外提供服务，通过负载均衡分发请求。

```
正常状态：
┌──────────┐   请求   ┌──────────┐
│   Nginx   │────────▶│  App #1  │
└──────────┘         └──────────┘
       │
       │ 负载均衡
       │
       ▼
┌──────────┐
│  App #2  │ ← 同时工作
└──────────┘
```

**特点**：

- 优点：资源利用率高、无单点故障、自动负载均衡
- 缺点：数据同步复杂、跨机房部署成本高

**适用场景**：对可用性和性能都有要求的场景。

### 3.3 多活架构（Multi-Active）

多活架构是双主的进一步扩展，在多个地理位置部署数据中心，各数据中心同时对外提供服务。

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│    北京机房       │     │    上海机房       │     │    广州机房       │
│  ┌───────────┐  │     │  ┌───────────┐  │     │  ┌───────────┐  │
│  │  接入层    │  │     │  │  接入层    │  │     │  │  接入层    │  │
│  └─────┬─────┘  │     │  └─────┬─────┘  │     │  └─────┬─────┘  │
│        │         │     │        │         │     │        │         │
│  ┌─────▼─────┐  │     │  ┌─────▼─────┐  │     │  ┌─────▼─────┐  │
│  │   应用层   │  │◀───▶│  │   应用层   │  │◀───▶│  │   应用层   │  │
│  └─────┬─────┘  │     │  └─────┬─────┘  │     │  └─────┬─────┘  │
│        │         │     │        │         │     │        │         │
│  ┌─────▼─────┐  │     │  ┌─────▼─────┐  │     │  ┌─────▼─────┐  │
│  │   数据层   │  │◀───▶│  │   数据层   │  │◀───▶│  │   数据层   │  │
│  └───────────┘  │     │  └───────────┘  │     │  └───────────┘  │
└─────────────────┘     └─────────────────┘     └─────────────────┘
```

**多活的类型**：

| 类型 | 说明 | 适用场景 |
|---|---|---|
| **同城双活** | 两个机房在同一个城市，网络延迟小 | 核心业务容灾 |
| **两地三中心** | 两个城市三个机房 | 城市级故障容灾 |
| **三地五中心** | 三个城市五个机房 | 极端高可用要求 |

## 四、故障检测与自动转移

### 4.1 健康检查

健康检查是故障检测的基础。系统需要定期检测各组件的健康状态，发现故障时及时处理。

**健康检查的类型**

| 类型 | 实现方式 | 检测内容 |
|---|---|---|
| **心跳检测** | 组件定期发送心跳包 | 组件是否存活 |
| **端口检测** | 检测端口是否可达 | 服务是否监听 |
| **HTTP 检测** | 请求健康检查接口 | 服务是否正常响应 |
| **进程检测** | 检测进程是否运行 | 进程是否崩溃 |

**Nginx 健康检查配置**

```nginx
upstream backend {
    server 192.168.1.101:8080 max_fails=3 fail_timeout=30s;
    server 192.168.1.102:8080 max_fails=3 fail_timeout=30s;
    server 192.168.1.103:8080 down;  # 手动标记为下线
}

# 被动健康检查：基于请求结果判断
# max_fails=3: 连续 3 次失败，标记为不可用
# fail_timeout=30s: 30 秒后重新尝试
```

```java
// Spring Boot Actuator 健康检查
@Component
public class CustomHealthIndicator implements HealthIndicator {
    
    @Override
    public Health health() {
        try {
            // 检查数据库连接
            jdbcTemplate.execute("SELECT 1");
            
            // 检查 Redis 连接
            redisTemplate.getConnectionFactory().getConnection().ping();
            
            return Health.up().build();
        } catch (Exception e) {
            return Health.down()
                .withDetail("error", e.getMessage())
                .build();
        }
    }
}
```

### 4.2 故障转移

检测到故障后，系统需要自动将流量切换到健康的实例。

**Keepalived + Nginx 实现**

```bash
# Keepalived 配置
! Configuration File for keepalived

global_defs {
   router_id nginx_master
}

vrrp_script check_nginx {
    script "/etc/keepalived/check_nginx.sh"
    interval 2
    weight -20
}

vrrp_instance VI_1 {
    state MASTER
    interface eth0
    virtual_router_id 51
    priority 100
    advert_int 1
    
    virtual_ipaddress {
        192.168.1.200  # VIP
    }
    
    track_script {
        check_nginx
    }
}
```

```bash
#!/bin/bash
# 健康检查脚本
nginx_status=$(curl -s http://localhost/health)
if [ "$nginx_status" != "OK" ]; then
    systemctl restart nginx
    exit 1
fi
exit 0
```

**Redis Sentinel 自动故障转移**

```bash
# Sentinel 配置
sentinel monitor mymaster 192.168.1.101 6379 2
sentinel down-after-milliseconds mymaster 5000
sentinel failover-timeout mymaster 180000
sentinel parallel-syncs mymaster 1
```

### 4.3 故障恢复

故障转移后，还需要考虑故障恢复的问题。

**1. 自动回切 vs 手动回切**

- **自动回切**：故障恢复后，自动切回原主机
- **手动回切**：故障恢复后，需要人工确认再切回

自动回切的问题是：如果原主机还存在「隐患」（比如磁盘即将损坏），切回后可能再次故障。**建议使用手动回切**，人工确认后再切回。

**2. 数据同步验证**

故障转移前，必须确保从机的数据已经同步完成，否则切换后可能丢数据。

```sql
-- 检查从库同步状态
SHOW SLAVE STATUS\G

-- 关键字段
Slave_IO_Running: Yes       # IO 线程是否运行
Slave_SQL_Running: Yes      # SQL 线程是否运行
Seconds_Behind_Master: 0    # 落后主库的秒数
```

## 五、限流与降级：保护系统的最后防线

### 5.1 限流：控制请求流量

限流是保护系统的第一道防线。当流量超过系统处理能力时，主动拒绝部分请求，保护系统不被压垮。

**限流算法对比**

| 算法 | 原理 | 优点 | 缺点 |
|---|---|---|---|
| **计数器** | 固定时间窗口计数 | 简单 | 临界突变 |
| **滑动窗口** | 滑动时间窗口计数 | 精确 | 实现复杂 |
| **漏桶** | 请求以固定速率处理 | 平稳 | 不适合突发 |
| **令牌桶** | 令牌以固定速率生成 | 允许突发 | 实现稍复杂 |

**Sentinel 限流**

```java
@Configuration
public class SentinelConfig {
    
    @Bean
    public FilterRegistrationBean<SentinelFilter> sentinelFilter() {
        FilterRegistrationBean<SentinelFilter> registration = new FilterRegistrationBean<>();
        registration.setFilter(new SentinelFilter());
        registration.addUrlPatterns("/*");
        registration.setName("sentinelFilter");
        return registration;
    }
}

@Service
public class OrderService {
    
    @Autowired
    private SentinelResourceHandler sentinelHandler;
    
    public Order createOrder(OrderRequest request) {
        // 限流：每秒最多 1000 个请求
        String resourceName = "createOrder";
        double count = 1;
        
        Entry entry = null;
        try {
            entry = SphU.entry(resourceName, EntryType.OUT, count);
            
            // 执行业务逻辑
            return doCreateOrder(request);
            
        } catch (BlockException e) {
            // 被限流了，返回友好提示
            throw new BusinessException("系统繁忙，请稍后重试");
        } finally {
            if (entry != null) {
                entry.exit();
            }
        }
    }
}
```

### 5.2 熔断：防止故障蔓延

熔断器（Circuit Breaker）是防止故障蔓延的核心机制。当下游服务故障时，熔断器「跳闸」，快速失败，避免大量请求积压。

**熔断器状态机**

```
        ┌─────────────────────────────────────────┐
        │                                         │
        ▼                                         │
    ┌───────┐    失败次数>阈值    ┌───────┐    恢复超时    ┌─────────┐
    │ 闭合   │ ─────────────────▶ │ 打开   │ ────────────▶ │  半开   │
    │(正常)  │                   │(熔断)  │               │(探测)   │
    │       │◀────────────────── │       │◀────────────── │         │
    └───────┘    成功次数>阈值    └───────┘    探测失败      └─────────┘
        ▲                                     │
        │                                     │ 探测成功
        └─────────────────────────────────────┘
```

**Hystrix 熔断器**

```java
@Service
public class UserService {
    
    @HystrixCommand(
        fallbackMethod = "getUserFallback",
        commandProperties = {
            @HystrixProperty(name = "circuitBreaker.requestVolumeThreshold", value = "20"),  // 最小请求数
            @HystrixProperty(name = "circuitBreaker.sleepWindowInMilliseconds", value = "5000"),  // 熔断持续时间
            @HystrixProperty(name = "circuitBreaker.errorThresholdPercentage", value = "50")  // 错误率阈值
        }
    )
    public User getUser(Long userId) {
        return userClient.getUser(userId);
    }
    
    // 降级方法
    public User getUserFallback(Long userId) {
        // 1. 尝试从缓存获取
        User cached = cacheService.getUser(userId);
        if (cached != null) {
            return cached;
        }
        
        // 2. 返回默认值
        return User.DEFAULT;
    }
}
```

### 5.3 降级：牺牲功能保证核心

降级是指在系统压力过大或部分功能故障时，主动关闭非核心功能，保证核心功能正常运行。

**降级的策略**

| 策略 | 说明 | 示例 |
|---|---|---|
| **页面降级** | 返回简化版页面 | 首页不显示推荐、排行榜 |
| **请求拦截** | 拒绝非核心请求 | 非会员用户返回「服务繁忙」 |
| **数据降级** | 返回缓存或默认值 | 评论列表返回空 |
| **关闭写接口** | 只提供读服务 | 暂时关闭下单功能 |

**Spring Boot 降级配置**

```java
@Service
public class ProductService {
    
    // 核心接口，不降级
    public Product getProduct(Long productId) {
        return productMapper.selectById(productId);
    }
    
    // 非核心接口，可降级
    @HystrixCommand(fallbackMethod = "getRecommendedProductsFallback")
    public List<Product> getRecommendedProducts(Long userId) {
        return recommendationService.getRecommendations(userId);
    }
    
    public List<Product> getRecommendedProductsFallback(Long userId) {
        // 降级策略：返回空列表或热门商品
        return hotProductMapper.selectTop10();
    }
}
```

## 六、容灾与备份：数据的最后保障

### 6.1 数据备份

数据是公司最重要的资产。数据备份是高可用的最后一道防线。

**备份策略**

| 类型 | 频率 | 保留时间 | 恢复速度 |
|---|---|---|---|
| **全量备份** | 每周/每天 | 1-4 周 | 慢 |
| **增量备份** | 每小时 | 1-7 天 | 中 |
| **binlog 备份** | 实时 | 7-30 天 | 快 |

```bash
# MySQL 全量备份
mysqldump -hlocalhost -uroot -p123456 --single-transaction --master-data=2 --all-databases > backup_full.sql

# MySQL 增量备份（binlog）
mysqladmin -uroot -p123456 flush-logs
cp /var/lib/mysql/mysql-bin.* /backup/binlog/
```

### 6.2 恢复演练

备份只是第一步，定期演练才是关键。

**演练流程**

1. 准备恢复环境（隔离环境）
2. 执行数据恢复
3. 验证数据完整性
4. 记录恢复时间和问题
5. 优化恢复流程

### 6.3 异地容灾

本地备份无法应对城市级灾难（地震、洪水）。异地容灾是应对这类风险的唯一手段。

## 七、混沌工程：主动发现故障

### 7.1 什么是混沌工程

混沌工程（Chaos Engineering）是一种在生产环境中主动注入故障、验证系统韧性的实践。

传统的测试是「确保系统正常运行」，混沌工程是「确保系统在故障时也能正常运行」。

Netflix 是混沌工程的先驱者，他们开发了 **Chaos Monkey**：一个随机关闭生产服务器的「破坏工具」。通过持续地「破坏」，Netflix 验证了系统的容错能力，也倒逼了架构的改进。

### 7.2 混沌工程实验

**常见的混沌实验**

| 实验类型 | 注入故障 | 验证目标 |
|---|---|---|
| **服务器故障** | 随机关闭服务器 | 负载均衡、自动转移 |
| **网络故障** | 模拟网络延迟、丢包 | 超时配置、重试机制 |
| **依赖故障** | 关闭下游服务 | 熔断降级、兜底策略 |
| **资源耗尽** | 消耗 CPU、内存 | 限流、告警 |
| **时钟偏移** | 修改服务器时间 | 时间相关逻辑 |

**ChaosBlade 混沌实验**

```bash
# 安装 ChaosBlade
wget https://github.com/chaosblade-io/chaosblade/releases/download/v1.7.0/chaosblade-1.7.0-linux-amd64.tar.gz
tar -xvf chaosblade-1.7.0-linux-amd64.tar.gz

# 模拟 CPU 满载
blade create cpu load --cpu-percent 80

# 模拟网络延迟
blade create network delay --time 3000 --interface eth0

# 模拟进程 kill
blade create process kill --process-name nginx

# 撤销所有实验
blade destroy
```

### 7.3 混沌实验的注意事项

- **从非核心系统开始**：先在低风险环境验证
- **从小故障开始**：先注入轻微故障，逐步加大
- **准备好回滚方案**：一旦系统不可控，立即停止实验
- **通知相关团队**：让运维、开发、SRE 都知道在做什么
- **选择业务低峰期**：避免影响用户

## 八、监控与告警：故障的早期发现

### 8.1 监控体系

监控是高可用的「眼睛」。没有监控，你不知道系统是否健康；没有告警，你不知道系统何时出了问题。

**监控的四大黄金指标**

| 指标 | 含义 | 告警阈值建议 |
|---|---|---|
| **延迟（Latency）** | 请求响应时间 | P99 > 500ms |
| **流量（Traffic）** | QPS、活跃用户数 | QPS 异常波动 |
| **错误率（Errors）** | 请求失败的比例 | 错误率 > 1% |
| **饱和度（Saturation）** | 资源利用率 | CPU > 80%、内存 > 85% |

**Prometheus + Grafana 监控体系**

```java
// 自定义业务指标
@Service
public class OrderService {
    
    private final Counter orderCounter;
    private final Counter orderFailedCounter;
    private final Timer orderTimer;
    
    public OrderService(MeterRegistry registry) {
        this.orderCounter = Counter.builder("order.created")
            .description("订单创建数量")
            .register(registry);
        
        this.orderFailedCounter = Counter.builder("order.failed")
            .description("订单失败数量")
            .register(registry);
        
        this.orderTimer = Timer.builder("order.process.time")
            .publishPercentiles(0.5, 0.95, 0.99)
            .register(registry);
    }
    
    public Order createOrder(OrderRequest request) {
        orderTimer.record(() -> {
            try {
                Order order = doCreateOrder(request);
                orderCounter.increment();
                return order;
            } catch (Exception e) {
                orderFailedCounter.increment();
                throw e;
            }
        });
    }
}
```

### 8.2 告警策略

告警不是越多越好，太多告警会让人麻木；也不是越少越好，漏报关键问题会出大事。

**告警分级**

| 级别 | 名称 | 响应时间 | 示例 |
|---|---|---|---|
| P1 | 紧急 | 5 分钟内 | 服务不可用 |
| P2 | 严重 | 30 分钟内 | 错误率 > 10% |
| P3 | 警告 | 2 小时内 | CPU > 85% |
| P4 | 提醒 | 24 小时内 | 日志异常 |

**告警抑制**：大量告警时，只发最严重的那个，避免告警风暴。

## 九、案例：AWS 如何做到 99.99% 可用性

### 9.1 AWS 的可用性架构

AWS（Amazon Web Services）对外承诺 S3 的可用性是 **99.999999999%（11 个 9）**，即一年内最多丢失 0.000001% 的数据。

**S3 的高可用设计**

**1. 数据冗余存储**

S3 将数据自动复制到多个可用区（AZ）。即使整个可用区故障，数据也不会丢失。

```
用户上传文件
      │
      ▼
写入主存储节点
      │
      ├──▶ 复制到 AZ2 存储节点
      │
      ├──▶ 复制到 AZ3 存储节点
      │
      └──▶ 元数据写入索引系统
```

**2. 分层故障检测**

- 第一层：存储节点自检（每毫秒）
- 第二层：系统级心跳检测（每几秒）
- 第三层：数据完整性校验（持续）

**3. 自动故障恢复**

- 检测到故障节点 → 自动创建新节点
- 新节点从其他副本同步数据
- 恢复期间，数据仍然可读

### 9.2 核心设计原则

**1. 假设故障会发生**

AWS 的设计哲学是：任何组件都可能故障，所以系统必须能容忍组件故障。

**2. 自动化优先**

故障检测、故障转移、故障恢复，全部自动化。减少人工干预，减少人为错误。

**3. 测量、控制、改进**

- 测量：每个组件的可用性、SLO 达成率
- 控制：严格的变更管理、灰度发布
- 改进：每次故障后必须做复盘，输出改进项

## 十、面试追问

### 问题一：如何设计一个高可用的订单系统？

**参考答案**：

**1. 消除单点**

- 应用服务器：至少 2 台 + Nginx 负载均衡
- 数据库：主从复制 + 自动切换
- 缓存：Redis Sentinel / Cluster
- 消息队列：多副本部署

**2. 流量保护**

- 限流：Sentinel/RateLimiter
- 熔断：Hystrix/Sentinel
- 降级：非核心功能可降级

**3. 数据保护**

- 数据库备份：每天全量 + 实时 binlog
- 异地容灾：主备机房

**4. 监控告警**

- 四大黄金指标监控
- 多级告警（P1-P4）

### 问题二：如何选择主从复制 vs 主主复制？

**参考答案**：

| 对比项 | 主从复制 | 主主复制 |
|---|---|---|
| 写入能力 | 单点写入 | 多个写入点 | 
| 数据一致性 | 强一致（同步） | 最终一致（可能冲突） |
| 复杂度 | 较低 | 较高（冲突解决） |
| 适用场景 | 读多写少 | 写入也需要高可用 |

**主从复制适合**：读多写少的业务，一主多从，读写分离。

**主主复制适合**：需要写入高可用的场景，但需要处理数据冲突（如自增 ID）。

### 问题三：如何设计一个 99.99% 可用的系统？

**参考答案**：

**1. 架构层面**

- 多 AZ 部署，消除单点
- 自动化故障检测和转移
- 定期做混沌实验，验证容错能力

**2. 数据层面**

- 多副本存储，数据不丢失
- 定期备份 + 异地容灾
- 写操作使用分布式事务或可靠消息

**3. 流量层面**

- 多层限流，防止流量冲击
- 熔断降级，保护核心功能
- CDN + 静态资源缓存，减轻源站压力

**4. 运维层面**

- 完善的监控告警体系
- 变更管理，灰度发布
- 故障复盘，持续改进

## 总结

高可用架构设计的核心是**消除单点、自动容错、快速恢复**，让系统在故障发生时也能「活着」。

关键要点回顾：

- **几个 9 的含义**：可用性 99.9% 意味着每年 8.76 小时的停机时间
- **消除单点故障**：每个关键组件都要有备份
- **冗余部署**：主备、双主、多活，根据成本和需求选择
- **健康检查 + 故障转移**：自动化检测和恢复
- **限流 + 熔断 + 降级**：流量保护三剑客
- **备份 + 容灾**：数据的最后保障
- **混沌工程**：主动发现故障，而不是被动挨打
- **监控 + 告警**：故障的早期发现

记住：**高可用不是「不惜一切代价」，而是在成本、复杂度、可用性之间找到平衡**。根据业务需求确定目标可用性，然后针对性地设计。
