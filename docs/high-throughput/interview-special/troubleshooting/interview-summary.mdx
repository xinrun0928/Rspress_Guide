## 一、通用排查框架：五步走方法论

无论遇到什么类型的故障，排查思路都是相通的。这套「五步走方法论」可以应对绝大多数生产环境问题。

### 1.1 五步法的核心逻辑

| 步骤 | 名称 | 核心问题 | 产出 |
|---|---|---|---|
| 第一步 | 监控告警 | 发生了什么？严重程度？ | 初步判断 |
| 第二步 | 复现问题 | 能否在测试环境复现？ | 缩小范围 |
| 第三步 | 定位问题 | 根因在哪一层？ | 问题定位 |
| 第四步 | 分析根因 | 为什么会出问题？ | 根因分析 |
| 第五步 | 修复验证 | 如何修复？如何验证？ | 问题解决 |

### 1.2 五步法的详细说明

**第一步：监控告警**

好的监控系统是排查的第一步。很多故障在发生之前就有预兆，如果监控到位，可以在问题扩大之前介入。

关键动作：
- 查看监控大盘，了解系统整体状态
- 确认告警的真实性（排除误报）
- 评估影响范围和严重程度
- 收集相关信息（时间、范围、趋势）

**第二步：复现问题**

如果能在测试环境复现问题，排查效率会大幅提升。

关键动作：
- 收集生产环境的特征信息（数据量、并发量、配置）
- 在测试环境构造相似场景
- 尝试触发问题
- 如果无法复现，进入第三步定位问题

**第三步：定位问题**

根据问题类型，选择合适的工具和方法，逐步缩小问题范围。

关键动作：
- 确定问题发生在哪一层（JVM、网络、数据库、消息队列）
- 使用诊断工具收集信息（jstack、jmap、netstat）
- 分析日志，寻找错误线索
- 如果仍然无法定位，进入第四步深入分析

**第四步：分析根因**

找到问题点后，需要深入分析为什么会出问题。

关键动作：
- 追溯问题的根本原因
- 区分表象和根因
- 评估是否会影响其他系统
- 制定修复方案

**第五步：修复验证**

修复问题后，必须验证修复效果。

关键动作：
- 在测试环境验证修复
- 灰度上线，持续观察
- 确认没有引入新问题
- 编写故障报告，总结经验

### 1.3 五步法的变体：快速止血模式

有些场景下，排查时间有限，需要先止血再排查。

快速止血 checklist：

- [ ] 确认问题范围和影响
- [ ] 执行紧急止血操作（扩容、重启、限流）
- [ ] 开启监控告警
- [ ] 逐步排查根因
- [ ] 长期修复

## 二、工具矩阵：按问题类型选工具

工具是排查的武器。但更重要的是知道什么场景用什么工具。

### 2.1 JVM 问题工具

| 问题类型 | 首选工具 | 备选工具 | 关键命令 |
|---|---|---|---|
| OOM | MAT | jmap + jhat | `jmap -dump` |
| CPU 飙高 | Arthas trace | jstack + top | `jstack <PID>` |
| GC 停顿 | GCEasy | GCViewer | `jstat -gcutil` |
| 线程问题 | Arthas thread | jstack | `jstack <PID>` |
| 内存泄漏 | MAT | jmap histo | `jmap -histo` |

### 2.2 网络问题工具

| 问题类型 | 首选工具 | 备选工具 | 关键命令 |
|---|---|---|---|
| 连通性 | ping | telnet | `ping <host>` |
| 路由分析 | mtr | traceroute | `mtr <host>` |
| 连接状态 | ss | netstat | `ss -antp` |
| DNS 解析 | dig | nslookup | `dig <domain>` |
| 抓包分析 | tcpdump | Wireshark | `tcpdump -i eth0` |

### 2.3 数据库问题工具

| 问题类型 | 首选工具 | 备选工具 | 关键命令 |
|---|---|---|---|

## 三、思维模式：从表象到根因

### 3.1 症状与根因的区分

面试中，面试官最想看到的是你的根因分析能力。很多人能发现现象，但找不到根因。

**常见症状与可能根因对照**：

| 症状 | 可能根因 |
|---|---|
| CPU 100% | 死循环、正则回溯、GC 压力、JIT 编译 |
| 响应时间长 | 数据库慢、网络延迟、GC 停顿、线程阻塞 |
| 内存持续增长 | 内存泄漏、缓存无限制、流量洪峰 |
| 服务不可用 | OOM、线程池满、连接池耗尽、网络中断 |
| 消息积压 | 消费者崩溃、数据库瓶颈、消费逻辑异常 |

### 3.2 层级排查法

遇到复杂问题时，需要从高层向低层逐步排查。

```
应用层（日志、监控）
    ↓
框架层（Spring、数据库连接池）
    ↓
JVM 层（GC、线程、内存）
    ↓
系统层（CPU、内存、磁盘、网络）
    ↓
网络层（TCP、DNS、负载均衡）
    ↓
基础设施层（云服务、网络设备）
```

### 3.3 相关性 vs 因果性

排查问题时，要区分相关性和因果性。

**例子**：

现象：A 服务 CPU 高、B 服务响应慢
分析：A 和 B 的 CPU 都高
错误判断：B 响应慢是因为 A 的 CPU 高
正确判断：A 和 B 可能都在调用同一个慢的数据库

## 四、面试表达技巧：如何展示排查能力

### 4.1 结构化表达

面试中，结构化表达非常重要。推荐使用「现象 → 排查 → 根因 → 修复 → 预防」的结构。

**回答模板**：

```markdown
「我之前遇到过一个问题，现象是……（现象）

排查过程是这样的……（排查步骤）

最后定位到的根因是……（根因）

修复方案是……（修复）

后来我们还做了……来预防类似问题。」（预防）
```

### 4.2 用数据说话

面试中，如果能给出具体的数字，会大大增加可信度：

- 「当时 CPU 使用率达到了 98%」
- 「Full GC 频率从每分钟 0.5 次飙升到每分钟 5 次」
- 「接口响应时间从 50ms 上升到 3000ms」
- 「消息积压超过了 10 万条」

### 4.3 展示系统性思维

好的工程师不只是解决单个问题，而是能总结规律，举一反三。

**示例**：

「那次问题解决后，我总结了一套 OOM 排查 checklist：
1. 先确认 OOM 类型（Heap/Metaspace/Direct Memory）
2. 查看 GC 日志，看 Full GC 频率
3. 生成 Heap Dump，用 MAT 分析
4. 重点关注 Histogram 中业务类的对象数量
5. 用 Dominator Tree 找内存占用最大的对象
6. 追踪 GC Roots，找到引用链

后来团队把这套流程文档化了，新人上手快多了。」

### 4.4 展示工具链的掌握

面试中展示你对工具的熟练使用，能体现你的实战经验。

**示例**：

「我一般先用 `jstat` 看 GC 趋势，如果 Full GC 频繁，再用 `jmap` 生成 dump。MAT 的 Leak Suspects 功能很好用，能自动定位可疑的泄漏点。定位到对象后，用 Dominator Tree 追引用链，最后用 GC Roots 分析找出为什么对象无法回收。」

## 五、常见问题与参考答案

### 问题一：生产环境出问题，你的第一反应是什么？

**参考答案**：

「我的第一反应是评估影响范围和严重程度。

首先，查看监控大盘，确认问题的真实性，排除误报。然后，评估影响范围——有多少用户受影响？是部分功能不可用还是整体不可用？

接着，我会根据问题类型快速止血——如果是 CPU 飙高，可能需要扩容或重启；如果是 OOM，可能需要重启服务并保留 Heap Dump。

止血之后，再逐步排查根因。」

### 问题二：OOM 和 CPU 100% 有什么区别？

**参考答案**：

| 维度 | OOM | CPU 100% |
|---|---|---|
| 本质 | 内存耗尽 | CPU 被占满 |
| 表现 | 进程崩溃 | 响应变慢 |
| 排查工具 | MAT | jstack、Arthas |
| 根因 | 内存泄漏、分配过大 | 死循环、正则、GC |
| 恢复 | 重启 | 自动恢复或重启 |

「两者有时会相互影响：OOM 可能导致 GC 压力增大，进而导致 CPU 高；CPU 100% 可能导致请求处理变慢，堆积在内存中，最终导致 OOM。」

### 问题三：如何避免生产环境故障？

**参考答案**：

「避免故障需要从多个层面入手：

**架构层面**：
- 高可用设计：多实例部署、负载均衡、自动故障转移
- 限流熔断：防止流量洪峰压垮系统
- 降级预案：核心功能优先，非核心功能降级

**代码层面**：
- 代码审查：避免引入明显的性能问题
- 单元测试：覆盖边界条件和异常场景
- 性能测试：上线前进行压测

**监控层面**：
- 全方位监控：CPU、内存、网络、应用指标
- 告警配置：设置合理的告警阈值
- 日志规范：记录关键操作日志

**运维层面**：
- 灰度发布：小流量验证再全量
- 变更管理：重大变更需审批
- 故障演练：定期进行混沌工程」

### 问题四：排查问题时如何避免影响生产？

**参考答案**：

「生产环境排查要格外谨慎，避免雪上加霜。

**原则一：先评估影响**
使用 `jstack`、`jmap` 命令前，先评估对进程的影响。高并发场景下，`jmap -F` 可能导致进程短暂挂起。

**原则二：优先使用低影响工具**
- `jstat` 几乎无影响，适合在线查看
- `jstack` 影响较小，可以安全使用
- `jmap` 建议只在必要时使用

**原则三：利用自动 dump**
配置 `-XX:+HeapDumpOnOutOfMemoryError`，让 JVM 在 OOM 时自动生成 dump。

**原则四：灰度验证**
先在测试环境复现问题，修复方案先灰度验证，再全量上线。」

### 问题五：如何提升排查效率？

**参考答案**：

「提升排查效率的关键是工具、方法和经验。

**工具层面**：
熟练掌握 jmap、jstack、jstat、Arthas 等工具。搭建完善的监控体系（APM、Grafana）。准备好常用的分析脚本。

**方法层面**：
建立标准化的排查流程。根据现象快速定位问题类型，缩小排查范围。善用搜索和日志。

**经验层面**：
积累常见问题的排查经验。总结常见故障的根因和修复方案。复盘每一次故障，形成知识沉淀。」

## 六、命令速查表

### 6.1 JVM 命令

```bash
# 查看 JVM 进程
jps -l

# 查看 JVM 参数
jinfo <PID>

# 查看堆内存
jmap -heap <PID>

# 生成 Heap Dump
jmap -dump:format=b,file=/tmp/heap.hprof <PID>

# 查看对象统计
jmap -histo <PID> | head -30

# 监控 GC
jstat -gcutil <PID> 1000

# 导出线程栈
jstack <PID> > /tmp/jstack.log

# Arthas 命令
# 启动
java -jar arthas-boot.jar
# 查看线程
thread -n 10
# 追踪方法耗时
trace com.example.service.* * '#cost > 100'
# 生成火焰图
profiler start --duration 30
```

### 6.2 系统命令

```bash
# 查看进程 CPU/内存
top
top -Hp <PID>

# 查看网络连接
netstat -an | grep TIME_WAIT
ss -antp

# 网络测试
ping <host>
traceroute <host>
mtr <host>

# DNS 测试
dig <domain>
nslookup <domain>

# 抓包
tcpdump -i eth0 host <IP> -w /tmp/capture.pcap
```

### 6.3 数据库命令

```bash
# MySQL
SHOW FULL PROCESSLIST;
SHOW ENGINE INNODB STATUS;
EXPLAIN <SQL>;

# Redis
INFO stats
CLIENT LIST
MONITOR
```

### 6.4 消息队列命令

```bash
# Kafka
kafka-topics.sh --describe --topic <name>
kafka-consumer-groups.sh --describe --group <name>

# RabbitMQ
rabbitmqctl list_queues name messages consumers
rabbitmqctl list_consumers
```

### 7.2 常见误区

避免以下常见误区：

| 误区 | 正确做法 |
|---|---|
| 直接重启服务 | 先排查根因，重启只是临时止血 |
| 只看日志，不看监控 | 日志和监控结合，全面了解系统状态 |
| 猜测根因，不验证 | 所有假设都要验证后再下结论 |
| 只修复问题，不复盘 | 复盘才能举一反三 |
| 一个人排查 | 复杂问题需要团队协作 |

## 总结

故障排查是 Java 工程师的核心能力，也是面试中的高频考点。

关键要点回顾：

- **通用框架**：五步走方法论（监控 → 复现 → 定位 → 分析 → 修复）
- **工具矩阵**：按问题类型选择合适的工具
- **思维模式**：从表象到根因，分层排查
- **面试技巧**：结构化表达，用数据说话
- **能力培养**：动手实践，持续积累

记住：**最好的排查是没有排查**。完善的监控、及时的告警、自动化的运维，能让很多问题在发生之前就被发现和解决。但当问题真正发生时，你要能快速定位、根因、修复。
