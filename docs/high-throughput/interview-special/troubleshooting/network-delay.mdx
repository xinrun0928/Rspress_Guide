# 网络延迟排查：从 DNS 到应用层

import { Badge } from '@rspress/core/theme';

周三下午两点，你正在处理一个需求变更，突然收到一堆用户反馈：

```
【用户反馈】
- 订单页面加载很慢，要等 3-5 秒
- 首页图片加载不出来
- 支付页面一直转圈

【运维反馈】
- 服务器 CPU：30%（正常）
- 数据库 CPU：15%（正常）
- 内存使用率：50%（正常）
- 磁盘 IO：正常
- 应用日志：正常，无异常
```

奇怪。服务器和数据库都正常，应用日志也没有报错，但用户却说页面加载很慢。

你开始怀疑：是网络问题？

网络延迟排查是 Java 工程师最容易忽略的领域。我们习惯性地关注 CPU、内存、数据库，却很少想到网络层。但有时候，问题真的就出在网络上——DNS 解析慢、TCP 连接延迟、网络丢包……

这一节，我们来系统性地解决网络延迟问题。

## 一、网络延迟的本质：从用户请求到服务端响应

### 1.1 一次 HTTP 请求的网络之旅

当用户在浏览器中输入一个 URL，到页面完全加载完成，数据经历了漫长的旅程：

```
用户浏览器
    ↓
DNS 解析（域名 → IP）
    ↓
TCP 连接（建立三次握手）
    ↓
TLS/SSL 握手（HTTPS）
    ↓
发送 HTTP 请求
    ↓
负载均衡器
    ↓
Nginx / API Gateway
    ↓
应用服务器
    ↓
数据库（或其他后端）
    ↓
原路返回响应
```

每一个环节都可能产生延迟。网络延迟排查的本质，就是找出这漫长的旅程中，哪一环出了问题。

### 1.2 常见网络延迟原因

| 环节 | 常见问题 | 延迟量级 |
|---|---|---|
| DNS 解析 | DNS 污染、TTL 配置不当 | 10-500ms |
| TCP 连接 | 连接超时、重试 | 100ms-数秒 |
| TLS 握手 | 证书验证、密钥交换 | 50-200ms |
| 负载均衡 | LB 队列等待、连接复用 | 10-100ms |
| 应用层 | 代码逻辑、外部调用 | 毫秒-秒级 |
| 网络传输 | 丢包、重传、拥塞 | 10-500ms |
| 客户端 | 渲染、JS 执行 | 毫秒-秒级 |

### 1.3 网络延迟 vs 应用延迟：如何区分

网络延迟和应用延迟的区分，是排查的第一步。

**应用延迟特征**：
- 响应时间稳定，与并发量相关
- 日志中有明确的处理耗时
- CPU 或内存使用率高
- 数据库或外部服务响应慢

**网络延迟特征**：
- 响应时间波动大，不稳定
- 日志中处理时间很短，但端到端时间长
- 监控显示网络指标异常
- 某些请求超时，某些正常

## 二、网络延迟排查六步法

### 2.1 第一步：基础连通性测试——ping 和 traceroute

网络排查的第一步，是确认基础连通性。

```bash
# ping 测试
ping -c 10 api.example.com

# 输出示例
PING api.example.com (192.168.1.100): 56 data bytes
64 bytes from 192.168.1.100: icmp_seq=0 ttl=64 time=1.234 ms
64 bytes from 192.168.1.100: icmp_seq=1 ttl=64 time=1.456 ms
64 bytes from 192.168.1.100: icmp_seq=2 ttl=64 time=1.678 ms

--- api.example.com ping statistics ---
10 packets transmitted, 10 packets received, 0.0% packet loss
round-trip min/avg/max/stddev = 1.234/1.456/1.678/0.123 ms
```

关键指标：
- **packet loss**：丢包率 > 0% 说明网络有问题
- **time**：延迟，> 100ms 可能是跨地域或网络拥塞

```bash
# traceroute 路由追踪
traceroute api.example.com

# 或 mtr（持续追踪，推荐）
mtr api.example.com
```

mtr 会持续发送数据包并追踪每一跳的延迟，能发现网络延迟在哪一跳。

### 2.2 第二步：端口连通性测试——telnet 和 nc

确认 IP 层面连通后，需要测试端口是否可达。

```bash
# telnet 测试
telnet api.example.com 443

# 或 nc（netcat）测试
nc -zv api.example.com 443

# 输出示例
Connection to api.example.com 443 port [tcp/https] succeeded!
```

如果连接失败，说明端口不通或被防火墙拦截。

### 2.3 第三步：DNS 解析测试——dig 和 nslookup

DNS 是网络请求的第一环，DNS 问题会导致连接延迟或失败。

```bash
# dig 查看 DNS 解析详情
dig api.example.com

# 输出示例
;; QUESTION SECTION:
;api.example.com.               IN      A

;; ANSWER SECTION:
api.example.com.        300     IN      A       192.168.1.100

;; Query time: 5 ms           <-- DNS 解析耗时
;; SERVER: 8.8.8.8#53
;; WHEN: Wed Mar 29 14:00:00 CST 2026
```

关注 `Query time`：如果 DNS 解析时间过长（如 > 100ms），可能是 DNS 服务器问题。

```bash
# 查看 DNS 解析过程
dig +trace api.example.com

# 对比不同 DNS 服务器的解析结果
dig @8.8.8.8 api.example.com   # Google DNS
dig @114.114.114.114 api.example.com  # 腾讯 DNS
```

### 2.4 第四步：连接状态分析——netstat 和 ss

TCP 连接状态能反映很多问题。

```bash
# 查看连接状态统计
netstat -s | grep -i error

# 查看 TIME_WAIT 连接数量
netstat -an | awk '/^tcp/ {print $NF}' | sort | uniq -c

# 输出示例
      123 TIME_WAIT
       10 ESTABLISHED
       5 LISTEN
```

关键指标：
- **TIME_WAIT 过多**：可能导致端口耗尽
- **CLOSE_WAIT 过多**：可能是连接未正确关闭
- **ESTABLISHED 过多**：可能是连接泄漏

```bash
# ss 命令（比 netstat 更快）
ss -s

# 查看所有连接的详细信息
ss -antp

# 查看与特定 IP 的连接
ss -antp | grep 192.168.1.100
```

### 2.5 第五步：应用层延迟分析——APM 和日志

如果基础网络没问题，需要深入到应用层分析延迟。

**APM 全链路追踪**：

```java
// 接入 SkyWalking 或 Pinpoint
// 自动追踪每个请求的耗时分布
// 能清晰看到耗时在哪一环节
```

典型 APM 追踪结果：

```
Total: 2345ms
├── DNS: 5ms
├── TCP: 10ms
├── TLS: 45ms
├── Nginx: 12ms
├── UserService.getUser(): 1200ms
│   ├── DB.selectById(): 1100ms  ← 数据库慢
│   └── Redis.get(): 80ms
├── OrderService.createOrder(): 950ms
│   ├── DB.insert(): 800ms
│   └── MQ.send(): 50ms
└── Response: 100ms
```

从这个追踪可以看到，UserService.getUser() 占了 1200ms，其中数据库查询 1100ms——根因是数据库慢。

**日志分析**：

```bash
# 分析响应时间分布
grep "process time" application.log | awk '{print $NF}' | sort -n

# 查看慢请求日志
grep "slow" application.log

# 分析各环节耗时
grep "timestamp" application.log | jq '.elapsed'
```

### 2.6 第六步：代码层延迟分析——连接池和超时配置

找到延迟环节后，需要检查代码配置。

**连接池配置检查**：

```java
// HikariCP 配置检查
spring:
  datasource:
    hikari:
      maximum-pool-size: 20        // 连接池大小
      minimum-idle: 5              // 最小空闲连接
      connection-timeout: 30000    // 连接超时（毫秒）
      idle-timeout: 600000         // 空闲超时
      max-lifetime: 1800000       // 最大生命周期
      leak-detection-threshold: 60000  // 泄漏检测
```

常见问题：
- `maximum-pool-size` 太小：连接等待
- `connection-timeout` 太短：正常请求超时
- `idle-timeout` 太短：频繁创建连接

**HTTP 客户端配置检查**：

```java
// RestTemplate 配置
RestTemplate restTemplate = RestTemplateBuilder.create()
    .setConnectTimeout(Duration.ofSeconds(5))      // 连接超时
    .setReadTimeout(Duration.ofSeconds(30))        // 读取超时
    .build();

// OKHttp 配置
OkHttpClient client = new OkHttpClient.Builder()
    .connectTimeout(5, TimeUnit.SECONDS)
    .readTimeout(30, TimeUnit.SECONDS)
    .writeTimeout(30, TimeUnit.SECONDS)
    .connectionPool(new ConnectionPool(50, 5, TimeUnit.MINUTES))
    .build();
```

## 三、网络延迟的六大元凶

### 3.1 DNS 解析延迟

DNS 是网络请求的第一环，DNS 问题会导致连接延迟。

**DNS 解析慢的原因**：

| 原因 | 表现 | 解决方案 |
|---|---|---|
| DNS 服务器慢 | 所有请求都慢 | 更换 DNS 服务器 |
| DNS 污染 | 解析结果错误 | 使用可信 DNS |
| TTL 配置不当 | 高峰期 DNS 解析积压 | 合理设置 TTL |
| 域名解析次数多 | 每次都解析 | DNS 缓存 |

**解决方案**：

```java
// DNS 缓存配置
spring:
  cloud:
    inetutils:
      timeout-seconds: 5

// JVM DNS 缓存配置
-Dsun.net.inetaddr.ttl=300
```

### 3.2 TCP 连接延迟

TCP 连接建立需要三次握手，握手延迟不可忽视。

**TCP 连接慢的原因**：

| 原因 | 表现 | 解决方案 |
|---|---|---|
| 网络距离远 | 跨地域延迟高 | 就近部署 |
| 连接超时设置不当 | 失败后重试 | 合理设置超时 |
| 连接池耗尽 | 等待连接 | 增大连接池 |
| 防火墙拦截 | 连接失败 | 检查防火墙规则 |

**解决方案**：

```java
// 使用 HTTP Keep-Alive 复用连接
spring:
  http:
    keep-alive: true

// 使用连接池
CloseableHttpClient client = HttpClients.custom()
    .setMaxTotal(200)
    .setDefaultMaxPerRoute(50)
    .build();
```

### 3.3 TIME_WAIT 连接堆积

服务器主动关闭连接后，会进入 TIME_WAIT 状态，持续等待 2MSL（通常 60 秒）。如果连接关闭过快，TIME_WAIT 连接会堆积，占用大量端口。

```bash
# 查看 TIME_WAIT 数量
netstat -an | grep TIME_WAIT | wc -l

# 如果超过 30000，需要优化
```

**解决方案**：

```bash
# 服务端优化：启用 TIME_WAIT 复用
sysctl -w net.ipv4.tcp_tw_reuse=1

# 服务端优化：缩短 TIME_WAIT 时间
sysctl -w net.ipv4.tcp_fin_timeout=30

# 客户端优化：使用连接池复用连接
```

### 3.4 MTU 和分片问题

MTU（Maximum Transmission Unit）不匹配会导致 IP 分片，影响性能。

```bash
# 查看 MTU 设置
ifconfig

# MTU 通常是 1500
# 如果链路上有更小的 MTU，会导致分片

# 测试 MTU
ping -M do -s 1472 api.example.com
```

**问题表现**：
- 大包请求很慢，小包请求正常
- `ping` 大包失败，`ping` 小包成功
- traceroute 显示某跳延迟突然增大

**解决方案**：
- 设置合理的 MTU
- 使用 TCP MSS 握手

### 3.5 网络拥塞和丢包

网络拥塞会导致丢包和重传，严重影响性能。

```bash
# 查看网络错误统计
netstat -s | grep -i error

# 查看丢包统计
netstat -s | grep -i "packet loss"
```

**问题表现**：
- 延迟不稳定，忽高忽低
- 丢包率 > 0%
- 重传率高

**解决方案**：
- 联系网络服务商
- 就近部署，减少网络跳数
- 使用 CDN 加速静态资源

### 3.6 客户端问题：慢客户端

有时候，问题不在服务端，而在客户端。客户端网络慢、浏览器性能差、JS 执行阻塞等，都会导致页面加载慢。

```java
// 服务端可以做的：减少响应体积
// 1. 启用 GZIP 压缩
spring:
  server:
    compression:
      enabled: true
      mime-types: text/html,text/xml,text/plain,text/css,text/javascript,application/javascript

// 2. 使用 HTTP/2
server:
  http2: true
```

## 四、实战案例：从表象到根因

### 案例一：NTP 服务器导致时钟偏移

**背景**：某服务日志显示请求响应时间正常，但用户反映页面加载很慢。

**排查过程**：

1. APM 分析发现，响应时间确实很长
2. 查看日志，发现时间戳混乱，有些请求的耗时是负数
3. 检查服务器时钟，发现与 NTP 服务器时间不同步
4. 进一步发现，某台服务器时钟快了 30 秒
5. 排查发现 NTP 服务器故障，导致时间同步失败

**影响**：
- 依赖时间的缓存失效（TTL 计算错误）
- 依赖时间的业务逻辑错误
- 分布式系统中时钟不同步导致分布式事务失败

**修复方案**：

```bash
# 修复 NTP 同步
systemctl enable ntpd
systemctl start ntpd

# 手动同步
ntpdate -b ntp.aliyun.com
```

**预防措施**：
- 监控服务器时钟偏移
- 使用分布式时钟服务（如 ZooKeeper）
- 避免在业务代码中依赖绝对时间

---

### 案例二：MTU 不匹配导致请求超时

**背景**：某服务在本地测试正常，但部署到云环境后，某些大请求总是超时。

**排查过程**：

1. 日志显示请求耗时不稳定，小请求正常，大请求超时
2. `ping -M do -s 1472` 失败，`ping -s 1000` 成功
3. `traceroute` 发现某跳 MTU 为 1400
4. 检查网络配置，发现 VPN 隧道的 MTU 设置为 1400

**根因**：大请求被分片后丢失，导致 TCP 重传，影响性能。

**修复方案**：

```bash
# 调整 MTU
ifconfig eth0 mtu 1400

# 或在应用层设置 TCP MSS
iptables -A FORWARD -p tcp --tcp-flags SYN,RST SYN -j TCPMSS --set-mss 1360
```

---

### 案例三：连接池耗尽导致请求阻塞

**背景**：某服务偶发性的响应很慢，但很快又恢复正常。

**排查过程**：

1. APM 分析发现，响应时间在某个时间段突然增加
2. 查看连接池状态，发现连接等待数突然增加
3. `netstat` 发现大量 CLOSE_WAIT 连接
4. 代码审查发现，某个 HTTP 调用没有正确关闭响应流

**根因代码**：

```java
// 错误写法：未关闭响应流
public String callExternalAPI(String url) {
    RestTemplate template = new RestTemplate();
    ResponseEntity<String> response = template.getForEntity(url, String.class);
    return response.getBody(); // 没有关闭连接
}
```

**修复方案**：

```java
// 正确写法：使用 try-with-resources
public String callExternalAPI(String url) {
    RestTemplate template = new RestTemplate();
    ResponseEntity<String> response = template.getForEntity(url, String.class);
    try {
        return response.getBody();
    } finally {
        // ResponseEntity 内部会关闭资源
    }
}

// 或使用 WebClient（Spring 5+）
WebClient client = WebClient.create();
String result = client.get().uri(url).retrieve().bodyToMono(String.class).block();
```

## 五、网络性能优化策略

### 5.1 DNS 优化

| 优化策略 | 说明 |
|---|---|
| 使用本地 DNS 缓存 | 减少 DNS 解析次数 |
| 合理设置 TTL | 平衡缓存和更新 |
| 使用高性能 DNS | 如 DNSPod、阿里云 DNS |
| 预解析关键域名 | `<link rel="dns-prefetch" href="//api.example.com">` |

### 5.2 TCP 连接优化

```bash
# 优化 TCP 参数
sysctl -w net.ipv4.tcp_tw_reuse=1           # TIME_WAIT 复用
sysctl -w net.ipv4.tcp_fin_timeout=30       # 缩短 FIN 超时
sysctl -w net.ipv4.tcp_keepalive_time=600  # TCP 保活
sysctl -w net.core.somaxconn=65535          # 最大连接数
sysctl -w net.ipv4.ip_local_port_range="1024 65535"  # 端口范围
```

### 5.3 HTTP 连接优化

| 优化策略 | 说明 |
|---|---|
| 启用 HTTP Keep-Alive | 复用 TCP 连接 |
| 使用 HTTP/2 | 多路复用，减少连接数 |
| 启用 GZIP 压缩 | 减少传输数据量 |
| 使用 CDN | 加速静态资源访问 |
| 合并请求 | 减少请求次数 |

### 5.4 应用层优化

```java
// 连接池合理配置
spring:
  datasource:
    hikari:
      maximum-pool-size: 50       // 根据数据库性能调整
      minimum-idle: 10            // 保持最小连接
      connection-timeout: 10000   // 10 秒超时
      idle-timeout: 300000        // 5 分钟空闲
      max-lifetime: 1800000       // 30 分钟最大生命周期

// HTTP 客户端优化
@Configuration
public class RestTemplateConfig {
    @Bean
    public RestTemplate restTemplate() {
        SimpleClientHttpRequestFactory factory = new SimpleClientHttpRequestFactory();
        factory.setConnectTimeout(5000);   // 5 秒连接超时
        factory.setReadTimeout(30000);     // 30 秒读取超时
        return new RestTemplate(factory);
    }
}
```

## 七、面试追问：网络延迟相关高频问题

### 问题一：TCP 三次握手和四次挥手的过程？

**参考答案**：

**三次握手**：

```
客户端                    服务端
  │                         │
  │─── SYN ────────────────>│  1. 客户端发送 SYN
  │<─── SYN+ACK ───────────│  2. 服务端返回 SYN+ACK
  │─── ACK ───────────────>│  3. 客户端发送 ACK
  │                         │
  │    连接建立完成          │
```

每次握手延迟约 1 个 RTT（往返时间）。

**四次挥手**：

```
客户端                    服务端
  │                         │
  │─── FIN ────────────────>│  1. 客户端发送 FIN，请求关闭
  │<─── ACK ───────────────│  2. 服务端确认
  │    (客户端不能再发数据)   │
  │<─── FIN ───────────────│  3. 服务端发送 FIN，请求关闭
  │─── ACK ───────────────>│  4. 客户端确认
  │                         │
  │   TIME_WAIT (2MSL)      │
  │    连接关闭完成          │
```

### 问题二：TIME_WAIT 是什么？过多会有什么影响？

**参考答案**：

TIME_WAIT 是 TCP 连接关闭后，本地端进入的等待状态，持续 2MSL（通常 60 秒）。

**作用**：
- 确保对方收到最后的 ACK
- 避免旧连接的延迟数据包被新连接接收

**过多的影响**：
- 占用大量端口（每个 TIME_WAIT 占用一个端口）
- 可能导致端口耗尽，无法建立新连接

**解决方案**：
- `net.ipv4.tcp_tw_reuse=1`：复用 TIME_WAIT 连接
- `net.ipv4.tcp_fin_timeout=30`：缩短 FIN 超时
- 客户端使用连接池复用连接

### 问题三：如何排查网络延迟？

**参考答案**：

**分层排查法**：

1. **网络层**：`ping`、`traceroute`、`mtr` 测试连通性和延迟
2. **传输层**：`netstat`、`ss` 查看连接状态和统计
3. **应用层**：APM 全链路追踪，定位具体耗时环节
4. **代码层**：检查超时配置、连接池配置

**关键命令**：

```bash
# 基础测试
ping api.example.com           # 连通性和延迟
traceroute api.example.com      # 路由追踪
mtr api.example.com            # 持续追踪

# 连接分析
netstat -an | grep TIME_WAIT   # TIME_WAIT 连接
ss -s                          # 连接统计

# 抓包分析
tcpdump -i eth0 host api.example.com -w /tmp/capture.pcap
```

### 问题四：HTTP Keep-Alive 是什么？有什么作用？

**参考答案**：

HTTP Keep-Alive 允许在单个 TCP 连接上发送多个 HTTP 请求/响应，避免每次请求都建立新的 TCP 连接。

**作用**：
- 减少 TCP 连接建立的开销（三次握手）
- 减少服务器资源消耗
- 提高响应速度

**配置**：

```yaml
# Nginx
keepalive_timeout 65;
keepalive_requests 100;

# 客户端
Connection: keep-alive
```

## 总结

网络延迟排查是 Java 工程师的必备技能，排查的核心在于「分层定位 → 对症下药」。

关键要点回顾：

- **延迟层次**：DNS → TCP → TLS → 负载均衡 → 应用层 → 数据库
- **排查工具**：ping、traceroute、mtr、netstat、ss、APM
- **六大元凶**：DNS 解析慢、TCP 连接延迟、TIME_WAIT 堆积、MTU 问题、网络拥塞、客户端慢
- **优化策略**：连接复用、连接池、超时配置、网络参数调优
- **监控指标**：连接数、TIME_WAIT、重传率、丢包率

记住：网络是连接一切的桥梁。网络问题往往不是单一原因造成的，需要系统性地分析和排查。
