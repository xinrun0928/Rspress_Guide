# 典型故障排查案例 - 概述

import { Badge } from '@rspress/core/theme';

凌晨三点，你的手机震动了。告警平台上，一条红色的告警刺眼地跳动着：「服务 A 响应时间 P99 > 5000ms，已有 127 个用户受影响。」

这不是你第一次收到告警，但这次不一样——你下午刚上了一个"优化"。

你打开监控，发现 CPU 在飙升；你查看日志，发现 GC 在疯狂回收；你检查数据库，发现连接数正常。一圈下来，你开始怀疑人生：到底是哪个环节出了问题？

这就是生产环境故障排查的真实写照。

很多工程师学了很多理论知识，却从未真正在生产环境中排查过问题。他们知道 JVM 的内存结构，知道多线程的原理，知道分布式系统的 CAP 理论，但当真正的问题发生时，他们往往束手无策。

**能解决问题的人，才是真正懂技术的人。**

这一章节，就是要把你从「理论派」变成「实战派」。我会带你走进真实的故障场景，手把手教你从告警到根因定位的全流程。

## 一、为什么说故障排查是 Java 工程师的终极考验

面试时，你可以侃侃而谈 HashMap 的原理、synchronized 的锁升级、JVM 的垃圾收集器。但当面试官问一句「生产环境 OOM 了，你如何排查？」，很多人就开始支支吾吾。

这是有道理的。故障排查考察的，不是你知道多少，而是你能解决多少。

### 1.1 理论知识 vs 实战能力：隔着一条鸿沟

知道原理和能排查问题，完全是两码事。

| 维度 | 理论知识 | 实战能力 |
|---|---|---|
| 知识来源 | 书籍、博客、课程 | 真实故障、踩坑经验 |
| 思考方式 | 自上而下，从原理到应用 | 自下而上，从现象到根因 |
| 时间压力 | 无限 | 有限，尤其是生产故障 |
| 工具依赖 | 不需要 | 必备（jmap、jstack、Arthas 等） |
| 判断依据 | 正确性 | 快速定位、可复现 |

理论知识告诉你「是什么」和「为什么」，实战能力告诉你「怎么办」。

举个例子，你知道 Full GC 会回收老年代的对象。但当你看到 GC 日志中 Full GC 越来越频繁，而 Old 区使用率始终居高不下，你知道该怎么做吗？你知道如何生成 Heap Dump 吗？你知道用 MAT 的哪个功能定位泄漏点吗？你知道如何追踪 GC Roots 吗？

这些「隐性知识」，只有在实战中才能获得。

### 1.2 面试官想看到什么

面试中关于故障排查的问题，面试官想看到的不是你背过的知识点，而是你的：

**系统性思维**：你面对一个未知问题，是毫无头绪，还是有章可循？优秀的候选人有一套自己的排查框架，能够系统性地缩小问题范围。

**工具熟练度**：你用过哪些工具？这些工具各自解决什么问题？什么场景下该用什么工具？工具选错了，排查效率会大打折扣。

**根因分析能力**：你能从众多表象中，找到真正的根因吗？很多工程师看到 CPU 高，就认为是 GC 的问题；看到响应慢，就认为是数据库的问题。但真正的根因往往藏在更深的地方。

**举一反三的能力**：你排查完一个问题后，有没有总结出通用规律？下次遇到类似问题，能否快速定位？

:::tip 面试加分项

如果你在面试中能主动说「这个问题我之前遇到过，当时是这样排查的……」，会让面试官眼前一亮。实战经验是最有说服力的证据。

:::

## 二、五步排查法：通用故障排查框架

无论是什么类型的故障，排查思路都是相似的。我总结了五步排查法，可以应对绝大多数生产环境问题。

### 2.1 监控告警：让问题早发现

好的监控体系，是故障排查的第一步。如果你的监控系统足够完善，很多问题在发生初期就能被发现，而不是等到用户投诉才后知后觉。

核心监控指标：

| 类别 | 关键指标 | 告警阈值建议 |
|---|---|---|
| 应用层 | QPS、RT（P50/P95/P99）、错误率 | RT P99 > 1000ms |
| JVM 层 | 堆内存使用率、GC 频率、GC 耗时 | 堆使用率 > 80% |
| 系统层 | CPU 使用率、内存使用率、磁盘 IO | CPU > 80% 持续 5 分钟 |
| 中间件层 | 数据库连接池、Redis 连接数、MQ 消费延迟 | 连接等待 > 100 |

告警的关键是「早发现」。如果你的告警策略是「服务完全不可用了才告警」，那黄花菜都凉了。建议设置梯度告警：先轻度告警（如 70% 使用率），再严重告警（如 85%）。

### 2.2 复现问题：让问题可观测

发现问题后，第一步是复现。无法复现的问题，就像打靶没有靶心，排查效率会很低。

复现的关键是收集信息：

```bash
# 查看进程基本信息
ps -ef | grep java

# 查看进程启动参数
jinfo <PID>

# 查看 JVM 内存使用
jmap -heap <PID>

# 查看当前 GC 状态
jstat -gcutil <PID> 1000
```

如果你能在测试环境复现生产环境的问题，排查效率会大幅提升。很多时候，生产环境的问题在测试环境也能复现，只是数据量和并发量不同。

### 2.3 定位问题：缩小包围圈

这是排查的核心步骤。根据问题类型，选择合适的工具和方法。

常见的问题定位策略：

**CPU 飙高问题**：
- 第一层：`top` 确认是哪个进程
- 第二层：`top -H` 确认是哪个线程
- 第三层：`jstack` 导出线程栈，分析线程状态
- 第四层：Arthas `trace` 命令定位热点方法

**内存问题**：
- `jmap -heap` 查看堆使用概况
- `jstat -gcutil` 监控 GC 趋势
- `jmap -dump` 生成 Heap Dump
- MAT 分析泄漏点

**响应延迟问题**：
- APM 工具（如 SkyWalking）查看全链路调用链
- 日志分析请求耗时分布
- 数据库慢查询日志
- 外部依赖响应时间

### 2.4 分析根因：从表象到本质

找到问题点后，需要深入分析根因。这一步考验的是对技术的深度理解。

举几个例子：

**现象**：CPU 100%，GC 频繁
**表象分析**：`jstat` 显示 Full GC 次数不断增长
**根因分析**：Old 区对象无法回收 → 可能是内存泄漏 → 定位到某个静态 HashMap 在不断累积数据 → 原因是缓存没有清理机制

**现象**：接口响应时间 P99 很高
**表象分析**：APM 显示某个方法耗时异常
**根因分析**：这个方法在循环中调用了数据库 → 没有加批处理 → 改为批量查询后恢复正常

**现象**：消息队列积压
**表象分析**：消费者数量正常，但消费速度跟不上
**根因分析**：消费者线程池配置过小 → 数据库连接等待 → 增加线程池后恢复正常

### 2.5 修复验证：从根除到预防

找到根因后，修复只是第一步。验证修复效果，才是关键。

修复验证 checklist：

- [ ] 在测试环境验证修复是否有效
- [ ] 确认修复没有引入新的问题
- [ ] 上线后持续监控相关指标
- [ ] 编写或更新相关文档
- [ ] 总结经验教训，避免类似问题

:::warning 切忌「头痛医头，脚痛医脚」

修复一个问题时，要确保修复的是根因，而不是表象。否则，你只是在掩盖问题，而不是解决问题。

举个例子，OOM 了只增加堆内存，这不是修复，这是拖延。真正的问题是为什么内存会耗尽。

:::

## 三、各类故障的排查要点

不同的故障类型，有不同的排查侧重点。下面按技术栈分类，介绍各类故障的排查要点。

### 3.1 JVM 层故障

JVM 层故障主要包括 OOM、CPU 飙高、GC 停顿过长等。

**OOM 排查要点**：

| 步骤 | 操作 | 目的 |
|---|---|---|
| 1 | 查看告警，确认 OOM 类型 | 确定是 Heap、Metaspace 还是 Direct Memory |
| 2 | 生成 Heap Dump | 保存现场，便于后续分析 |
| 3 | MAT 分析 Leak Suspects | 自动定位可疑泄漏点 |
| 4 | Dominator Tree 分析 | 找出内存占用的根源 |
| 5 | GC Roots 追踪 | 找到无法回收的原因 |
| 6 | 代码级修复 | 对症下药，修复泄漏代码 |

**CPU 飙高排查要点**：

| 步骤 | 操作 | 目的 |
|---|---|---|
| 1 | `top` 确认进程 CPU 使用率 | 确定是 JVM 进程的问题 |
| 2 | `top -H` 找到高 CPU 线程 | 定位到具体线程 |
| 3 | 线程 ID 转十六进制 | 用于匹配 jstack 输出 |
| 4 | `jstack <PID>` 导出线程栈 | 分析线程状态和调用栈 |
| 5 | Arthas `trace` 定位热点 | 找到耗时最长的方法 |
| 6 | Flame Graph 可视化 | 直观展示 CPU 消耗分布 |

**GC 停顿过长排查要点**：

| 步骤 | 操作 | 目的 |
|---|---|---|
| 1 | 开启 GC 日志 | `-XX:+PrintGCDetails -Xloggc:gc.log` |
| 2 | 使用 GCViewer/gceasy 分析 | 可视化 GC 日志 |
| 3 | 确认 GC 类型和耗时 | Minor GC / Full GC / Mixed GC |
| 4 | 分析 GC 原因 | Allocation Failure / Ergonomics |
| 5 | 调整 GC 参数 | 根据分析结果调优 |
| 6 | 验证调优效果 | 监控 GC 指标变化 |

### 3.2 消息队列故障

消息队列故障主要包括消息积压、消费延迟、消息丢失等。

**消息积压排查要点**：

| 步骤 | 操作 | 目的 |
|---|---|---|
| 1 | MQ 控制台查看积压量 | 确认积压严重程度 |
| 2 | 检查消费者数量 | 是否与分区数匹配 |
| 3 | 检查消费者状态 | 是否正常运行 |
| 4 | 查看消费者日志 | 是否有异常、OOM、超时等 |
| 5 | 检查数据库是否瓶颈 | 消费慢的原因 |
| 6 | 紧急扩容 + 死信队列 | 先止血，再排查 |

### 3.3 网络层故障

网络层故障主要包括连接超时、响应延迟、网络丢包等。

**网络延迟排查要点**：

| 步骤 | 操作 | 目的 |
|---|---|---|
| 1 | `ping` 基础连通性测试 | 确认网络是否可达 |
| 2 | `traceroute` / `mtr` 路由分析 | 定位网络延迟在哪一跳 |
| 3 | `netstat` / `ss` 连接状态 | 是否有大量 TIME_WAIT、CLOSE_WAIT |
| 4 | APM 全链路分析 | 定位慢请求的具体环节 |
| 5 | DNS 解析排查 | DNS 污染或 TTL 配置问题 |
| 6 | MTU 和分片排查 | 分片导致的性能问题 |

## 四、如何在面试中展示排查能力

面试中展示排查能力，有几个关键点。

### 4.1 结构化表达：现象 → 排查 → 根因 → 修复 → 预防

好的故障排查回答，应该有清晰的结构：

```markdown
## 回答模板

「我之前遇到过一个问题，现象是……（现象）

排查过程是这样的……（排查步骤）

最后定位到的根因是……（根因）

修复方案是……（修复）

后来我们还做了……来预防类似问题。」（预防）
```

这个结构能让面试官清晰地看到你的思路，而不是一团乱麻。

### 4.2 用具体数据说话

面试中，如果能给出具体的数字，会大大增加可信度：

- 「当时 CPU 使用率达到了 98%」
- 「Full GC 频率从每分钟 0.5 次飙升到每分钟 5 次」
- 「接口响应时间从 50ms 上升到 3000ms」
- 「消息积压超过了 10 万条」

这些具体数据，比「CPU 很高」「GC 很快」有力得多。

### 4.3 展示工具链的掌握

面试中展示你对工具的熟练使用，能体现你的实战经验：

- 「我一般先用 `jstat` 看 GC 趋势，如果 Full GC 频繁，再用 `jmap` 生成 dump」
- 「Arthas 的 `trace` 命令很好用，能直接看到方法耗时分布」
- 「GCViewer 可以可视化 GC 日志，比看原始日志直观多了」

### 4.4 展示系统性思维

好的工程师不只是解决单个问题，而是能总结规律，举一反三：

```markdown
「那次问题解决后，我总结了一套 OOM 排查 checklist：
1. 先确认 OOM 类型（Heap/Metaspace/Direct Memory）
2. 查看 GC 日志，看 Full GC 频率
3. 生成 Heap Dump，用 MAT 分析
4. 重点关注 Histogram 中业务类的对象数量
5. 用 Dominator Tree 找内存占用最大的对象
6. 追踪 GC Roots，找到引用链

后来团队把这套流程文档化了，新人上手快多了。」
```

## 五、动手实践：搭建自己的实验环境

光看不练假把式。要真正掌握故障排查，必须动手实践。

### 5.1 推荐工具清单

| 类别 | 工具 | 用途 |
|---|---|---|
| JVM 监控 | jstat, jmap, jinfo, jstack | 命令行 JVM 分析 |
| JVM 分析 | MAT (Memory Analyzer) | Heap Dump 分析 |
| Java 诊断 | Arthas | 在线诊断神器 |
| 性能分析 | async-profiler, Flame Graph | CPU 性能分析 |
| GC 分析 | GCViewer, GCEasy | GC 日志分析 |
| APM | SkyWalking, Pinpoint | 全链路追踪 |
| 系统监控 | top, vmstat, iostat, netstat | 系统资源监控 |

### 5.2 动手实验建议

**实验一：模拟 OOM**

```java
public class OOMSimulator {
    public static void main(String[] args) {
        List<byte[]> list = new ArrayList<>();
        int i = 0;
        while (true) {
            list.add(new byte[1024 * 1024]); // 每次分配 1MB
            System.out.println("Allocated " + (++i) + " MB");
        }
    }
}
```

启动参数：`-Xmx256m -XX:+HeapDumpOnOutOfMemoryError`

然后用 MAT 分析生成的 dump 文件，练习定位内存泄漏点。

**实验二：模拟 CPU 飙高**

```java
public class CPUSimulator {
    public static void main(String[] args) {
        while (true) {
            // 模拟正则表达式性能问题
            String pattern = ".*(abc|def|ghi|jkl|mno|pqr|stu|vwx).*";
            String test = "abcdefghijklmnopqrstuvwxyz";
            Pattern.matches(pattern, test);
        }
    }
}
```

启动后用 `top -H` 找到高 CPU 线程，用 `jstack` 分析调用栈。

**实验三：模拟消息积压**

用 Docker 启动一个 Kafka，创建一个 Topic，启动一个消费很慢的消费者，然后往 Topic 里狂发消息，观察积压情况。

### 5.3 复盘总结的习惯

每次排查完问题，都要写一份复盘文档。复盘文档应该包括：

- **问题描述**：发生了什么？
- **时间线**：什么时间发现、什么时间定位、什么时间修复？
- **排查过程**：用了哪些工具？发现了什么线索？
- **根因分析**：真正的根因是什么？
- **修复方案**：如何修复的？
- **经验教训**：下次如何避免？
- **改进项**：有哪些流程、工具、监控可以改进？

## 六、面试追问：故障排查能力考察

### 问题一：如何发现早期故障信号？

**参考答案**：

早期发现故障的关键是完善的监控和告警体系：

**监控层面**：
- 应用层：QPS、RT（P50/P95/P99）、错误率、异常数
- JVM 层：堆内存使用率、GC 频率和耗时、线程数
- 系统层：CPU、内存、磁盘 IO、网络流量
- 中间件层：数据库连接池、Redis 内存、MQ 消费延迟

**告警层面**：
- 设置梯度告警：轻度（60%）→ 中度（75%）→ 严重（90%）
- 趋势告警：堆内存使用率持续上升，即使未达到阈值也要告警
- 同比环比：与昨天/上周同期对比，异常波动也要告警

**日常巡检**：
- 每天检查监控大盘
- 关注 GC 趋势变化
- 关注业务指标异常

### 问题二：排查问题时如何避免影响生产？

**参考答案**：

生产环境排查要格外谨慎，避免雪上加霜：

**原则一：先评估影响**
- 使用 `jstack`、`jmap` 命令前，先评估对进程的影响
- 高并发场景下，`jmap -F` 可能导致进程短暂挂起

**原则二：优先使用低影响工具**
- `jstat` 几乎无影响，适合在线查看
- `jstack` 影响较小，可以安全使用
- `jmap` 建议只在必要时使用

**原则三：利用自动 dump**
- 配置 `-XX:+HeapDumpOnOutOfMemoryError`，让 JVM 在 OOM 时自动生成 dump
- 配置 `-XX:+FlightRecorder` 记录运行时数据

**原则四：灰度验证**
- 先在测试环境复现问题
- 修复方案先灰度验证，再全量上线

### 问题三：如何提升排查效率？

**参考答案**：

提升排查效率的关键是工具、方法和经验：

**工具层面**：
- 熟练掌握 jmap、jstack、jstat、Arthas 等工具
- 搭建完善的监控体系（APM、Grafana）
- 准备好常用的分析脚本

**方法层面**：
- 建立标准化的排查流程（参考五步排查法）
- 根据现象快速定位问题类型，缩小排查范围
- 善用搜索和日志

**经验层面**：
- 积累常见问题的排查经验
- 总结常见故障的根因和修复方案
- 复盘每一次故障，形成知识沉淀

## 延伸思考

故障排查能力不是一朝一夕能练成的。它需要：

1. **扎实的基础知识**：不懂 JVM 原理，怎么排查 OOM？不懂线程模型，怎么排查死锁？
2. **丰富的工具储备**：工具是排查的武器，要熟悉每个武器的用途
3. **大量的实战经验**：只有在真实问题中摸爬滚打过，才能形成直觉
4. **良好的排查习惯**：遇到问题不慌张，有章可循
5. **及时的复盘总结**：每一次排查都是学习机会，要善于总结

记住：**最好的排查是没有排查**。完善的监控、及时的告警、自动化的运维，能让很多问题在发生之前就被发现和解决。但当问题真正发生时，你要能快速定位、根因、修复。
