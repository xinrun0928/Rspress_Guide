
你打开监控面板，看到 GC 停顿时间曲线像过山车一样忽上忽下。每次 Full GC 都会导致服务停顿 2 秒以上，用户开始抱怨页面加载缓慢，甚至出现超时错误。

这不是第一次遇到 GC 停顿问题了。但每次遇到都不免让人头疼——GC 日志太长看不懂，GC 参数调了又调效果不明显，到底问题出在哪里？

别慌。这一节，我们来系统性地解决 GC 停顿过长问题。

## 一、GC 停顿的本质：理解「Stop-The-World」

### 1.1 什么是 GC 停顿

GC（Garbage Collection）在回收垃圾时，需要暂停所有应用线程（Stop-The-World，简称 STW）。这就是为什么 GC 会导致应用停顿。

GC 停顿不是 JVM 的 bug，而是 GC 算法的固有特性。GC 线程在工作时，需要确保应用线程不会修改堆中的对象，否则可能导致对象丢失或重复回收。为了保证这一点，JVM 会在 GC 期间暂停所有应用线程。

### 1.2 GC 停顿对应用的影响

GC 停顿对应用的影响取决于停顿时间和频率：

**对响应时间的影响**：

- 单次停顿 100ms：用户可能感知到轻微延迟
- 单次停顿 500ms：用户明显感知到卡顿
- 单次停顿 2000ms：大量请求超时，用户体验严重下降

**对吞吐量的影响**：

- GC 停顿期间，所有请求都无法处理
- 如果 GC 停顿占总时间 10%，吞吐量就会下降 10%
- 如果 GC 停顿占总时间 50%，吞吐量就会下降 50%

**对 SLA 的影响**：

- 大多数互联网服务的 SLA 要求 P99 延迟 < 200ms
- GC 停顿可能导致 P99 延迟超标，违反 SLA

### 1.3 不同 GC 类型的停顿特点

| GC 类型 | 停顿时间 | 发生频率 | 原因 |
|---|---|---|---|
| Minor GC（Young GC） | 短（毫秒级） | 频繁 | Eden 区满 |
| Major GC（Old GC） | 中等（百毫秒级） | 较少 | Old 区满 |
| Full GC | 长（秒级） | 少 | 空间不足、并发失败 |
| G1 Mixed GC | 可控 | 中等 | Old 区达到阈值 |
| CMS GC | 中等 | 中等 | Old 区达到阈值 |

## 二、GC 停顿排查六步法

### 2.1 第一步：开启 GC 日志——记录 GC 行为

排查 GC 问题，第一步是获取 GC 日志。

```bash
# JVM 启动参数中添加 GC 日志配置
-XX:+UseG1GC
-Xms4g -Xmx4g
-Xloggc:/data/logs/gc.log
-XX:+PrintGCDetails                    # 打印详细 GC 信息
-XX:+PrintGCDateStamps                  # 打印时间戳
-XX:+PrintTenuringDistribution          # 打印年龄分布
-XX:+PrintGCApplicationStoppedTime      # 打印 GC 停顿时间
-XX:NumberOfGCLogFiles=10                # 保留 10 个日志文件
-XX:GCLogFileSize=100M                  # 每个日志文件 100MB
```

:::warning GC 日志的影响

GC 日志本身会消耗一定的 IO 和磁盘空间，但相比排查问题的收益，这点开销是值得的。建议生产环境始终开启 GC 日志。

:::

### 2.2 第二步：分析 GC 日志——GCViewer 和 GCEasy

GC 日志通常很长，直接阅读原始日志效率很低。建议使用可视化工具。

**GCViewer**：

```bash
# 下载 GCViewer
wget https://github.com/chewiebug/GCViewer/releases/download/v1.34.1/GCViewer-1.34.1.jar

# 运行
java -jar gcviewer.jar /data/logs/gc.log
```

GCViewer 会生成关键指标：
- 吞吐量（Throughput）
- GC 停顿次数和平均停顿时间
- 内存使用趋势

**GCEasy**：

GCEasy 是在线工具，支持上传 GC 日志文件自动分析：

```
https://gceasy.io
```

上传日志后，GCEasy 会生成分析报告，包括：
- GC 原因分析
- 内存泄漏风险评估
- GC 调优建议

### 2.3 第三步：识别 GC 类型——是 Minor GC 还是 Full GC

分析 GC 日志的第一步，是识别是哪种类型的 GC 导致停顿。

**典型 Minor GC 日志**：

```
2026-03-29T10:15:23.456+0800: [GC (Allocation Failure) [PSYoungGen: 524288K->4352K(614400K)] 786432K->268435K(2097152K), 0.0234567 secs] [Times: user=0.10 sys=0.01, real=0.02 secs]
```

关键信息：
- `Allocation Failure`：触发原因是 Eden 区空间不足
- `PSYoungGen`：使用的是 Parallel Scavenge 收集器
- `524288K->4352K`：GC 前占用 512MB，GC 后占用 4MB
- `786432K->268435K`：堆内存从 768MB 降到 262MB
- `0.0234567 secs`：停顿时间 23ms

**典型 Full GC 日志**：

```
2026-03-29T10:15:25.789+0800: [Full GC (Ergonomics) [PSYoungGen: 4352K->0K(614400K)] [ParOldGen: 1048576K->786432K(1572864K)] 1052928K->786432K(2187264K), [Metaspace: 524288K->524288K], 2.3456789 secs] [Times: user=10.23 sys=0.45, real=2.35 secs]
```

关键信息：
- `Full GC (Ergonomics)`：Full GC 原因是 GC  ergonomics（自适应调整）
- `ParOldGen: 1048576K->786432K`：Old 区从 1GB 降到 768MB
- `2.3456789 secs`：停顿时间 2.3 秒，非常长

**G1 Full GC 日志**：

```
2026-03-29T10:15:25.789+0800: [Full GC 2097152K->786432K(2097152K), 3.4567890 secs]
   [Eden: 0.0K(1048576K)->0.0K(1048576K)]
   [Survivors: 65536K->0K]
   [Old: 1048576K->786432K(2097152K)]
   [Humongous: 262144K->0K]
```

### 2.4 第四步：分析停顿原因——GC 根因定位

不同类型的 GC 停顿，原因不同。

**Minor GC 停顿过长的原因**：

| 原因 | 表现 | 解决方案 |
|---|---|---|
| Eden 区太小 | Minor GC 频繁 | 增加年轻代大小 |
| 对象分配速率过高 | Minor GC 后存活对象多 | 优化代码，减少对象创建 |
| Survivor 区太小 | 对象过早晋升到 Old | 调整 SurvivorRatio |
| 大对象直接进入 Old | 大对象直接分配在 Old | 调整 PretenureSizeThreshold |

**Full GC 停顿过长的原因**：

| 原因 | 表现 | 解决方案 |
|---|---|---|
| Old 区空间不足 | Full GC 后 Old 区使用率高 | 增加 Old 区大小 |
| 对象晋升失败 | 晋升失败触发 Full GC | 增加年轻代大小 |
| 内存碎片 | Full GC 后空间不连续 | 使用 G1 或调优 CMS |
| 元空间不足 | Metaspace Full GC | 增加 Metaspace 大小 |
| System.gc() 调用 | 显式 Full GC | 避免显式调用 |

### 2.5 第五步：GC 调优——对症下药

根据分析结果，进行针对性的 GC 调优。

**G1 调优参数**：

```bash
# G1 基础配置
-XX:+UseG1GC                    # 使用 G1 收集器
-XX:MaxGCPauseMillis=200        # 最大停顿时间目标 200ms
-XX:G1HeapRegionSize=8m        # Region 大小 8MB

# 调优参数
-XX:G1NewSizePercent=30         # 年轻代占比 30%
-XX:G1MaxNewSizePercent=50      # 最大年轻代占比 50%
-XX:InitiatingHeapOccupancyPercent=45  # 触发并发标记的堆使用率
```

**CMS 调优参数**：

```bash
# CMS 基础配置
-XX:+UseConcMarkSweepGC
-Xms4g -Xmx4g
-XX:NewRatio=2                  # 老年代:年轻代 = 2:1

# 调优参数
-XX:CMSInitiatingOccupancyFraction=70  # Old 区使用率 70% 时开始回收
-XX:+UseCMSInitiatingOccupancyOnly     # 只使用上述阈值
-XX:MaxGCPauseMillis=500               # 最大停顿目标
```

**Parallel GC 调优参数**：

```bash
# Parallel GC 基础配置
-XX:+UseParallelGC
-Xms4g -Xmx4g
-XX:NewRatio=2

# 调优参数
-XX:SurvivorRatio=8             # Eden:Survivor = 8:1
-XX:MaxTenuringThreshold=15     # 最大晋升年龄
```

### 2.6 第六步：验证调优效果——持续监控

调优后，需要持续监控 GC 指标，验证效果。

```bash
# 实时监控 GC 状态
jstat -gcutil <PID> 1000

# 输出示例
  S0     S1     E      O      M     YGC     YGCT    FGC    FGCT     GCT
 12.35   0.00  65.23  71.73  97.45    123   1.234   5     2.345    3.579
```

关键指标：
- `YGC`：年轻代 GC 次数，如果持续增长说明 Minor GC 频繁
- `YGCT`：年轻代 GC 总耗时
- `FGC`：Full GC 次数，如果持续增长说明有问题
- `FGCT`：Full GC 总耗时
- `GCT`：总 GC 耗时

## 三、实战案例：三个典型的 GC 停顿场景

### 案例一：Minor GC 频繁导致停顿累积

**背景**：某服务 QPS 10000，每次 Minor GC 停顿 50ms，GC 频率每分钟 30 次，累计停顿时间占总时间的 25%。

**排查过程**：

1. GC 日志显示 Minor GC 频率非常高（每 2 秒一次）
2. `jstat -gcutil` 显示 Eden 区使用率每次 GC 后都保持在 80% 以上
3. `jmap -histo` 显示大量 `byte[]` 和 `String` 对象
4. 代码审查发现，接口每次请求都创建大量临时对象

**根因代码**：

```java
// 错误写法：每次请求创建大量临时对象
public List<OrderVO> getOrders(Long userId) {
    List<Order> orders = orderMapper.selectByUserId(userId);
    List<OrderVO> result = new ArrayList<>();
    for (Order order : orders) {
        OrderVO vo = new OrderVO();
        vo.setOrderId(order.getOrderId());
        vo.setUserId(order.getUserId());
        vo.setItems(JSON.toJSONString(order.getItems())); // 每次都序列化
        vo.setCreateTime(new Date()); // 每次都创建 Date 对象
        result.add(vo);
    }
    return result;
}
```

**修复方案**：

```java
// 正确写法：复用对象，减少对象创建
public List<OrderVO> getOrders(Long userId) {
    List<Order> orders = orderMapper.selectByUserId(userId);
    List<OrderVO> result = new ArrayList<>(orders.size());
    for (Order order : orders) {
        OrderVO vo = convertToVO(order); // 复用转换方法
        result.add(vo);
    }
    return result;
}

private OrderVO convertToVO(Order order) {
    OrderVO vo = new OrderVO();
    vo.setOrderId(order.getOrderId());
    // ... 设置其他字段
    return vo;
}
```

**JVM 调优**：

```bash
# 增加年轻代大小
-XX:NewSize=1g -XX:MaxNewSize=1g
-XX:SurvivorRatio=6
```

### 案例二：CMS 并发模式失败导致 Full GC

**背景**：某服务 Old 区使用率达到 80% 后，频繁触发 Full GC，每次停顿超过 3 秒。

**排查过程**：

1. GC 日志显示大量 `concurrent mode failure` 或 `promotion failed`
2. `concurrent mode failure`：CMS 收集器在 Old 区回收完成之前，对象晋升导致空间不足
3. `promotion failed`：Minor GC 时，对象晋升到 Old 区失败
4. 原因：Old 区碎片化，空间不连续

**典型 CMS 日志**：

```
2026-03-29T10:15:23.456+0800: [GC (Allocation Failure) [ParNew: 524288K->524288K(614400K), 0.0 secs]2026-03-29T10:15:23.456+0800: [Full GC (Allocation Failure) -- 1048576K->1048576K(2097152K), 4.5678901 secs]
2026-03-29T10:15:28.023+0800: [CMS: 1572864K->1048576K(2097152K), 5.2345678 secs] [Times: user=15.23 sys=0.56, real=5.23 secs]
```

关键信息：
- `Full GC (Allocation Failure)`：`Allocation Failure` 说明对象晋升失败
- CMS 回收后 Old 区使用率仍然很高（1048576K / 2097152K = 50%）

**修复方案**：

```bash
# 方案一：提前触发 CMS 回收
-XX:CMSInitiatingOccupancyFraction=60  # 降低到 60%

# 方案二：增加 CMS 线程数
-XX:ConcGCThreads=4

# 方案三：切换到 G1
-XX:+UseG1GC
-XX:MaxGCPauseMillis=200

# 方案四：增加堆内存
-Xms8g -Xmx8g
```

---

### 案例三：G1 Mixed GC 停顿过长

**背景**：某服务使用 G1 收集器，Mixed GC 停顿时间经常超过 500ms。

**排查过程**：

1. GC 日志显示 Mixed GC 频率很高
2. 每次 Mixed GC 需要回收大量 Old 区 Regions
3. 原因：Old 区使用率过高，G1 需要回收大量分区

**典型 G1 Mixed GC 日志**：

```
2026-03-29T10:15:23.456+0800: [GC pause (G1 Evacuation Pause) (mixed)
   (起点: 4096 regions) 2097152K(4096M)->2097152K(4096M), 0.8567890 secs]
   [Eden: 6144.0M(6144.0M)->0.0B(6144.0M)]
   [Survivor: 512.0M->6144.0M]
   [Old: 3276.8M->3584.0M(4096.0M)]
   [Humongous: 2048.0M->0.0B]
   [Heap: 4096.0M->4096.0M(4096.0M)]
```

关键信息：
- `G1 Evacuation Pause (mixed)`：Mixed GC
- `起点: 4096 regions`：涉及 4096 个 Region
- `0.8567890 secs`：停顿时间 856ms

**修复方案**：

```bash
# 方案一：增加 MaxGCPauseMillis（允许更长停顿，减少 GC 频率）
-XX:MaxGCPauseMillis=500

# 方案二：增加年轻代占比，减少 Old 区回收压力
-XX:G1NewSizePercent=40
-XX:G1MaxNewSizePercent=60

# 方案三：降低 InitiatingHeapOccupancyPercent，提前触发并发标记
-XX:InitiatingHeapOccupancyPercent=35

# 方案四：增加 Heap 大小
-Xms8g -Xmx8g
```

## 四、GC 调优实战指南

### 4.1 GC 调优的核心目标

GC 调优的核心目标是在「吞吐量」、「停顿时间」和「内存占用」之间找到平衡：

| 目标 | 说明 | 衡量指标 |
|---|---|---|
| 高吞吐量 | 单位时间内处理更多请求 | GC 时间占比 < 5% |
| 低停顿 | 单次停顿时间短 | P99 < 200ms |
| 合理内存 | 内存占用合理 | 不 OOM，内存利用率 60-80% |

### 4.2 不同场景的调优策略

**场景一：高并发 Web 服务**

特点：QPS 高，对响应时间敏感

推荐配置：

```bash
# G1 配置
-XX:+UseG1GC
-XX:MaxGCPauseMillis=100     # 严格的停顿目标
-XX:G1HeapRegionSize=4m
-XX:InitiatingHeapOccupancyPercent=45
```

**场景二：离线批处理**

特点：吞吐量优先，对停顿不敏感

推荐配置：

```bash
# Parallel GC 配置
-XX:+UseParallelGC
-XX:+UseParallelOldGC
-XX:ParallelGCThreads=8      # 增加 GC 线程数
-XX:NewSize=2g
-XX:MaxNewSize=2g
```

**场景三：大内存服务**

特点：堆内存大（> 16GB），需要高效回收

推荐配置：

```bash
# ZGC 配置（JDK 11+）
-XX:+UseZGC
-XX:MaxGCPauseMillis=10
-Xms32g -Xmx32g

# 或 G1 配置
-XX:+UseG1GC
-XX:MaxGCPauseMillis=200
-XX:G1HeapRegionSize=16m     # 大 Region
-Xms32g -Xmx32g
```

### 4.3 GC 参数对照表

| 参数 | 默认值 | 说明 | 调优建议 |
|---|---|---|---|
| `-XX:MaxGCPauseMillis` | 200ms | 最大停顿目标 | 根据业务 SLA 设置 |
| `-XX:NewRatio` | 2 | 老年代:年轻代比例 | 调大增加年轻代 |
| `-XX:SurvivorRatio` | 8 | Eden:Survivor 比例 | 调小增加 Survivor |
| `-XX:MaxTenuringThreshold` | 15 | 最大晋升年龄 | 调大延迟晋升 |
| `-XX:PretenureSizeThreshold` | 0 | 大对象阈值 | 调大避免大对象进 Old |
| `-XX:InitiatingHeapOccupancyPercent` | 45% | 触发并发标记的堆使用率 | 调低提前标记 |

## 五、GC 日志解读

### 5.1 GC 日志字段说明

**Minor GC 日志**：

```
2026-03-29T10:15:23.456+0800:  // GC 发生时间
[GC (Allocation Failure)                          // GC 原因
 [PSYoungGen: 524288K->4352K(614400K)]           // 年轻代：GC 前->GC后(总大小)
 786432K->268435K(2097152K),                    // 堆内存：GC 前->GC后(总大小)
 0.0234567 secs]                                 // GC 停顿时间
[Times: user=0.10 sys=0.01, real=0.02 secs]     // GC 线程 CPU 时间
```

**Full GC 日志**：

```
2026-03-29T10:15:25.789+0800:
[Full GC (Ergonomics)
 [PSYoungGen: 4352K->0K(614400K)]              // 年轻代完全回收
 [ParOldGen: 1048576K->786432K(1572864K)]      // 老年代：GC 前->GC后(总大小)
 1052928K->786432K(2187264K),                  // 堆内存：GC 前->GC后(总大小)
 [Metaspace: 524288K->524288K]                  // 元空间
 2.3456789 secs]                                // Full GC 停顿时间
[Times: user=10.23 sys=0.45, real=2.35 secs]
```

### 5.2 常见 GC 原因

| 原因 | 说明 | 是否正常 |
|---|---|---|
| `Allocation Failure` | Eden 区空间不足 | 正常，Minor GC |
| `Ergonomics` | JVM 自适应调整触发 GC | 正常 |
| `GC Cleanup` | CMS 清理阶段 | 正常 |
| `System.gc()` | 显式调用 Full GC | 不正常，应避免 |
| `concurrent mode failure` | CMS 未完成时对象晋升 | 不正常，需调优 |
| `promotion failed` | Minor GC 时晋升失败 | 不正常，需调优 |
| `Metadata GC Threshold` | 元空间不足 | 不正常，需增加 Metaspace |

## 六、监控与告警体系

### 6.1 GC 监控指标

| 指标 | 说明 | 告警阈值 |
|---|---|---|
| Full GC 频率 | Full GC 次数/分钟 | > 1次/5分钟 |
| Full GC 耗时 | 单次 Full GC 停顿时间 | > 1秒 |
| Minor GC 频率 | Minor GC 次数/分钟 | > 10次/分钟 |
| GC 时间占比 | GC 时间/总运行时间 | > 10% |
| Old 区使用率 | 老年代内存使用率 | > 80% |

### 6.2 Prometheus 监控配置

```yaml
# JMX Exporter 配置
jmx_prometheus_javaagent:
  annotations:
    prometheus.io/scrape: "true"
    prometheus.io/port: "9404"
    prometheus.io/path: "/metrics"
```

关键指标：

```
# JVM GC 指标
jvm_gc_collection_seconds_count{gc="PS Scavenge"}  // Minor GC 次数
jvm_gc_collection_seconds_count{gc="PS MarkSweep"} // Full GC 次数
jvm_gc_collection_seconds_sum{gc="PS Scavenge"}    // Minor GC 总耗时
jvm_gc_collection_seconds_sum{gc="PS MarkSweep"}  // Full GC 总耗时
```

### 6.3 告警配置

```yaml
groups:
  - name: gc_alerts
    rules:
      - alert: FullGCMetricsHigh
        expr: rate(jvm_gc_collection_seconds_count{gc="PS MarkSweep"}[5m]) > 0.003
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "Full GC 频率过高"
          description: "Full GC 频率 {{ $value }} 次/秒，超过阈值"

      - alert: GCPausingTimeHigh
        expr: histogram_quantile(0.99, rate(jvm_gc_collection_seconds_sum[5m])) > 1
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "GC 停顿时间过长"
          description: "P99 GC 停顿时间 {{ $value }} 秒"
```

## 七、面试追问：GC 停顿相关高频问题

### 问题一：G1 和 CMS 的区别？如何选择？

**参考答案**：

| 维度 | CMS | G1 |
|---|---|---|
| 收集方式 | 并发标记-清除 | 标记-整理 |
| 停顿时间 | STW 时间长 | 可控（MaxGCPauseMillis） |
| 内存碎片 | 有碎片 | 无碎片（整理） |
| 内存划分 | 新生代/老年代 | 多个 Region |
| 适用场景 | 低延迟应用（已不推荐） | 通用场景 |
| JDK 版本 | JDK 8 及以下 | JDK 9+ 默认 |

**选择建议**：
- JDK 8 及以下：优先选择 CMS（或考虑升级 JDK）
- JDK 9+：默认使用 G1
- 大内存（> 16GB）：G1 更适合
- 追求低延迟：G1 + ZGC

### 问题二：为什么 Full GC 停顿时间那么长？

**参考答案**：

Full GC 停顿时间长的原因：

1. **老年代空间大**：Full GC 需要扫描整个老年代
2. **存活对象多**：需要复制大量存活对象
3. **内存碎片**：CMS 可能产生内存碎片，导致晋升失败
4. **单线程收集**：某些收集器使用单线程回收
5. **跨代引用**：需要扫描年轻代查找老年代引用

解决方案：
- 使用 G1 收集器，控制停顿时间
- 减少对象分配速率
- 避免显式 Full GC
- 适当增加堆内存

### 问题三：GC 调优的一般步骤是什么？

**参考答案**：

1. **确认目标**：吞吐量优先还是延迟优先？
2. **收集数据**：开启 GC 日志，分析 GC 行为
3. **分析问题**：是 Minor GC 频繁还是 Full GC 停顿长？
4. **参数调优**：根据分析结果调整参数
5. **验证效果**：持续监控，观察指标变化
6. **迭代优化**：多次调优，找到最优配置

### 问题四：什么是 ZGC？它有什么优势？

**参考答案**：

ZGC（JDK 11+）是一款低延迟垃圾收集器：

| 特性 | 说明 |
|---|---|
| 停顿时间 | < 10ms，与堆大小无关 |
| 吞吐量 | 吞吐量损失 < 15% |
| 内存占用 | 支持 TB 级堆内存 |
| 并发阶段 | 大部分工作与应用并发进行 |

**适用场景**：
- 对延迟敏感的应用
- 大内存服务（> 16GB）
- JDK 11+

**配置**：

```bash
-XX:+UseZGC
-XX:MaxGCPauseMillis=10
-Xms32g -Xmx32g
```

## 总结

GC 停顿过长是 JVM 性能问题的典型表现，排查的核心在于「日志分析 → 根因定位 → 参数调优」。

关键要点回顾：

- **GC 停顿本质**：STW（Stop-The-World）是 GC 的固有特性
- **排查流程**：GC 日志 → 识别 GC 类型 → 分析停顿原因 → 参数调优 → 验证效果
- **三大根因**：Minor GC 频繁、Full GC 停顿长、G1 Mixed GC
- **调优策略**：G1 适合大多数场景，ZGC 适合大内存低延迟场景
- **监控指标**：GC 频率、停顿时间、吞吐量

记住：GC 调优只是 JVM 性能优化的一环。如果代码层面存在内存泄漏、大对象分配等问题，再好的 GC 参数也无法解决根本问题。
