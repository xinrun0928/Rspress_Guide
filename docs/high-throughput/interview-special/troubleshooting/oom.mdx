# OOM 故障排查：从告警到根因定位

import { Badge } from '@rspress/core/theme';

凌晨两点，你被一阵急促的电话铃声惊醒。告警平台上，一条红色的消息刺眼地跳动着：

```
【严重告警】服务 order-service OOM 了！
进程已重启 1 次，当前状态：重启中
错误类型：java.lang.OutOfMemoryError: Java heap space
```

你揉着惺忪的睡眼，打开监控大盘。堆内存使用率在半小时内从 60% 飙到了 100%，随后进程崩溃、重启。重启后内存使用率又开始攀升——这不是偶发的内存不足，而是有东西在持续泄漏。

这不是你第一次遇到 OOM，但每次遇到都不免让人头疼。代码在测试环境跑得好好的，为什么上了生产就 OOM？是内存泄漏，还是流量太大？是 JVM 参数设错了，还是代码里有隐藏的坑？

别慌。这一节，我们来系统性地解决 OOM 问题。从监控告警到根因定位，从工具使用到代码修复，让你遇到 OOM 不再手足无措。

## 一、OOM 的本质：理解 Java 内存管理的脆弱点

### 1.1 为什么 OOM 是生产环境的头号杀手

OOM（OutOfMemoryError）之所以可怕，有几个原因：

**不可预测性**：内存泄漏往往是渐进式的。在测试环境跑几个小时可能没问题，但在生产环境运行几天后才会暴露。等发现的时候，可能已经影响了大量用户。

**影响范围大**：OOM 导致进程崩溃，所有正在处理的请求都会失败。如果负载均衡没有正确配置，甚至可能导致整个服务不可用。

**排查难度高**：内存泄漏的根因往往藏得很深。一个看似正常的 Map.put() 操作，可能因为持有对象引用而导致内存无法回收。

**恢复时间长**：如果只重启服务而不修复根因，OOM 会反复发生。每次重启都需要时间，期间服务不可用。

### 1.2 OOM 的四种类型：你遇到的到底是哪一种

很多工程师以为 OOM 就是堆内存不够用。实际上，OOM 可能发生在 JVM 的不同内存区域。

| 类型 | 错误信息 | 常见原因 | 排查难度 |
|---|---|---|---|
| Heap Space OOM | `java.lang.OutOfMemoryError: Java heap space` | 内存泄漏、一次性加载过多数据 | 中等 |
| Metaspace OOM | `java.lang.OutOfMemoryError: Metaspace` | 动态类生成、CGLIB 增强、JSP 编译 | 较难 |
| Direct Memory OOM | `java.lang.OutOfMemoryError: Direct buffer memory` | NIO/Netty 堆外内存泄漏 | 较难 |
| 线程创建失败 | `java.lang.OutOfMemoryError: unable to create new native thread` | 线程创建过多 | 较难 |

这篇文章聚焦于 **Heap Space OOM**，因为这是最常见、也是面试最爱问的 OOM 类型。

### 1.3 理解 JVM 内存分配失败的过程

当你在 Java 代码里写 `new Object()` 时，JVM 会在堆内存中找一个足够大的连续空间。如果找到了，对象创建成功；如果找不到，JVM 会尝试触发一次 GC。

```java
public class AllocationDemo {
    public static void main(String[] args) {
        // 这个操作可能导致 OOM
        Object obj = new Object();
    }
}
```

OOM 的触发流程：

1. **新对象分配**：JVM 尝试在 Eden 区分配空间
2. **Minor GC**：如果 Eden 区不够，触发 Minor GC，回收年轻代
3. **对象晋升**：存活对象尝试晋升到 Old 区
4. **Full GC**：如果 Old 区不够，触发 Full GC
5. **分配失败**：Full GC 后仍然不够，抛出 OutOfMemoryError

:::tip 关键认知

OOM 不是突然发生的。在 OOM 之前，往往伴随着频繁的 GC。如果你的监控能捕获这个趋势，就能在 OOM 发生之前介入。

:::

## 二、Heap Space OOM 的四大元凶

在动手排查之前，先了解 OOM 的常见原因，能让你事半功倍。Heap Space OOM 的原因可以归结为四大类：

### 2.1 内存泄漏：对象无法被回收

这是 OOM 最常见的原因。

正常情况下，一个对象不再被引用时，GC 会将其回收。但如果代码错误导致对象被错误地持有引用，这个对象就无法被 GC 回收。随着时间推移，存活对象越来越多，堆内存逐渐耗尽。

常见内存泄漏模式：

| 模式 | 典型代码 | 为什么会泄漏 |
|---|---|---|
| 缓存无清理 | `static Map<K,V> cache = new HashMap<>()` | 数据只进不出 |
| 静态集合 | `static List<Object> list = new ArrayList<>()` | GC 的盲区 |
| 监听器未移除 | `addListener(listener)` | 持有对象引用 |
| ThreadLocal 未清理 | `threadLocal.set(user)` | 线程复用时数据残留 |
| 资源未关闭 | `new FileInputStream(path)` | 文件描述符泄漏 |

### 2.2 一次性加载过多数据

有时候代码没有内存泄漏，但确实需要大量内存。

典型场景：

```java
// 错误写法：一次性加载全表数据
List<User> users = userMapper.selectAll(); // 全表数据，可能百万级

// 正确写法：分页处理
Page<User> page = new Page<>(1, 100);
while (page.hasNext()) {
    List<User> batch = userMapper.selectPage(page);
    process(batch);
    page = page.next();
}
```

### 2.3 JVM 堆设置过小

`-Xmx` 设置的值太小，应用程序的正常内存需求都满足不了。

常见错误：

- 测试环境 2GB 堆，生产环境也是 2GB，但生产数据量是测试的 10 倍
- 容器环境没有正确配置内存限制，导致 JVM 堆被人为压缩
- 对内存消耗预估不足，上线后才发现不够

### 2.4 瞬间大量对象创建

高并发场景下，多个请求同时处理，每个请求都创建大量对象。如果对象创建速度远超 GC 回收速度，堆内存会快速耗尽。

典型场景：

- 秒杀活动，瞬间涌入大量请求
- 数据导入导出，批量处理时并发过高
- 接口没有限流，请求量失控

## 三、排查流程：五步定位 OOM 根因

### 3.1 第一步：监控告警——让问题早发现

真正的 OOM 排查，应该从告警开始，而不是从 OOM 发生开始。

在 OOM 发生之前，堆内存使用率会持续攀升。如果你的监控系统能捕获这个趋势，就能在 OOM 发生之前介入。

生产环境的 JVM 监控告警配置建议：

| 指标 | 轻度告警 | 严重告警 |
|---|---|---|
| 堆内存使用率 | >= 70% | >= 85% |
| Full GC 频率 | >= 1次/5分钟 | >= 1次/分钟 |
| Full GC 耗时 | >= 500ms | >= 1秒 |
| Old 区使用率 | >= 70% | >= 80% |

:::tip 告警策略建议

建议使用 G1 收集器，并配置 `-XX:InitiatingHeapOccupancyPercent=45`，让 JVM 在堆内存使用率达到 45% 时就开始并发标记，提前发现问题。

:::

### 3.2 第二步：获取 Heap Dump——保存现场

OOM 发生时，第一件事是保存堆转储（Heap Dump）。这是后续分析的基础。

```bash
# 方式一：正常生成（推荐，有 pid）
jmap -dump:format=b,file=/tmp/heap.hprof <PID>

# 方式二：强制生成（进程卡住时使用，可能造成进程挂起）
jmap -F -dump:format=b,file=/tmp/heap.hprof <PID>

# 方式三：设置自动生成（OOM 时自动保存）
-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/data/logs/
```

:::warning 生产环境慎用 jmap -F

`-F` 参数会强制执行，可能造成进程短暂挂起。在高并发生产环境中，使用前务必评估影响。如果服务配置了 `-XX:+HeapDumpOnOutOfMemoryError`，让 JVM 在 OOM 时自动生成 dump 是更安全的选择。

:::

### 3.3 第三步：分析 Heap Dump——MAT 三板斧

分析 Heap Dump 的工具有很多，但核心分析方法就三个。

#### 第一板斧：Leak Suspects——自动定位泄漏嫌疑人

MAT（Memory Analyzer Tool）提供了自动分析功能，会列出最可能导致内存泄漏的「嫌疑人」。

使用步骤：

1. 下载 MAT：https://eclipse.dev/mat/downloads.php
2. 打开 Heap Dump 文件
3. 点击「Leak Suspects Report」

MAT 会告诉你：「这个 `HashMap` 占用了 XX% 的内存，而且它一直在增长。」这就是排查的方向。

#### 第二板斧：Histogram——按类统计对象数量

Histogram 视图会列出所有类的对象数量和内存占用，按「Retained Size」排序。

```bash
# 用 jmap 也可以快速查看
jmap -histo <PID> | head -50
```

输出示例：

```
 num     #instances         #bytes  class name
----------------------------------------------
   1:          12345      98765440  [Ljava.lang.Object;
   2:           8901      45678912  com.example.service.UserService
   3:           6789      34567890  java.util.HashMap$Node
```

重点关注：
- 对象数量异常多的类
- 单个对象占用内存特别大的类
- 自己的业务代码中的类（而非 JDK 内部的类）

#### 第三板斧：Dominator Tree——找出内存占用的根源

Dominator Tree 展示了对象之间的引用关系，帮助你找出「谁持有了这些对象，导致它们无法被回收」。

在 MAT 中打开 Dominator Tree，顶部就是占用内存最多的对象。点击任何一个节点，可以看到它的 Retained Heap（这个对象被回收时，能释放多少内存）。

### 3.4 第四步：分析 GC Roots——找到无法回收的根因

找到了占用内存大的对象后，下一步是找出这些对象为什么无法被回收。

GC Roots 是垃圾收集器的根引用。从 GC Roots 出发，沿着引用链往下走，能到达的对象都是「存活」的；无法到达的对象，才会被回收。

在 MAT 中，选择「Path to GC Roots」→「exclude weak/soft references」，可以看到对象的完整引用链。

重点关注：

| GC Root 类型 | 说明 |
|---|---|
| 静态变量 | static 修饰的集合是最常见的泄漏源 |
| ThreadLocal | 线程池复用时的问题 |
| 本地方法栈 | JNI 调用的遗留问题 |
| 监听器 | 事件监听器没有取消注册 |
| 线程栈 | 局部变量持有对象引用 |

### 3.5 第五步：代码级修复——对症下药

找到泄漏点后，下一步就是修复代码。以下是常见内存泄漏的修复方案。

## 四、实战案例：三个典型的 OOM 场景

### 案例一：ThreadLocal 未清理导致的内存泄漏

**背景**：某订单服务运行 3 天后开始频繁 OOM，重启后很快又出现。

**排查过程**：

1. 查看 GC 日志，发现 Old 区使用率持续居高不下，Full GC 频率很高
2. 生成 Heap Dump，用 MAT 分析
3. 在 Histogram 中发现 `ThreadLocal$ThreadLocalMap$Entry` 对象数量异常多
4. Dominator Tree 分析，发现这些 Entry 被某个线程持有
5. 代码审查发现，ThreadLocal 在请求处理中 set 了用户信息，但没有 remove

```java
// 错误写法：线程池复用时，ThreadLocal 数据不会被清理
private static ThreadLocal<User> currentUser = new ThreadLocal<>();

public void handleRequest(HttpServletRequest request) {
    User user = parseUser(request);
    currentUser.set(user); // 设置了用户信息
    try {
        // 处理请求...
        processOrder();
    } catch (Exception e) {
        log.error("处理订单失败", e);
    }
    // 方法结束，但 ThreadLocal 没有清理
}
```

```java
// 正确写法：使用后务必清理
private static ThreadLocal<User> currentUser = new ThreadLocal<>();

public void handleRequest(HttpServletRequest request) {
    User user = parseUser(request);
    try {
        currentUser.set(user);
        // 处理请求...
        processOrder();
    } catch (Exception e) {
        log.error("处理订单失败", e);
    } finally {
        currentUser.remove(); // 务必清理
    }
}
```

**根因**：Tomcat 线程池复用线程，ThreadLocal 中的 User 对象无法被回收，导致内存泄漏。

**修复**：在 finally 块中清理 ThreadLocal。

**预防**：建议使用框架提供的上下文传递机制，而不是 ThreadLocal。

---

### 案例二：缓存没有清理机制导致的 OOM

**背景**：某用户中心服务，缓存了全量用户信息，运行一周后 OOM。

**排查过程**：

1. Heap Dump 分析发现 `ConcurrentHashMap` 占用了 60% 的堆内存
2. Dominator Tree 分析，这个 Map 被一个 static 变量持有
3. 代码审查发现，这是一个用户缓存，写入后从未清理

```java
// 错误写法：缓存永不过期，数据只进不出
public class UserCache {
    private static final Map<Long, User> userCache = new ConcurrentHashMap<>();

    public User getUser(Long userId) {
        // 先查缓存
        User user = userCache.get(userId);
        if (user == null) {
            // 缓存没有，从数据库加载
            user = userMapper.selectById(userId);
            userCache.put(userId, user); // 只进不出
        }
        return user;
    }
}
```

```java
// 正确写法一：使用 WeakHashMap（内存不足时自动回收）
private static final Map<Long, User> userCache = new WeakHashMap<>();

// 正确写法二：使用带过期时间的缓存（Caffeine）
private static final LoadingCache<Long, User> userCache = Caffeine.newBuilder()
    .expireAfterWrite(10, TimeUnit.MINUTES) // 写入后 10 分钟过期
    .maximumSize(10_000) // 最大缓存 1 万条
    .build(userId -> userMapper.selectById(userId));

// 正确写法三：使用 Guava Cache
private static final LoadingCache<Long, User> userCache = CacheBuilder.newBuilder()
    .expireAfterWrite(10, TimeUnit.MINUTES)
    .maximumSize(10_000)
    .build(CacheLoader.from(userId -> userMapper.selectById(userId)));
```

**根因**：用户不断注册，缓存不断增长，最终耗尽堆内存。

**修复**：使用带过期时间和大小限制的缓存。

---

### 案例三：大数据量导入导致的 OOM

**背景**：某数据导入功能，一次性导入 100 万条数据，导入过程中 OOM。

**排查过程**：

1. 查看 GC 日志，发现 Eden 区使用率飙升，然后 Full GC
2. Heap Dump 分析发现大量 `byte[]` 数组和 `String` 对象
3. 代码审查发现，数据导入时先把 Excel 全部读入内存

```java
// 错误写法：一次性加载全部数据
public void importUsers(MultipartFile file) throws IOException {
    // 把整个文件读入内存
    List<String> lines = Files.readAllLines(file.getInputStream().toString());
    // 100万行数据，全部在内存中
    for (String line : lines) {
        User user = parseLine(line);
        userMapper.insert(user);
    }
}
```

```java
// 正确写法：流式处理，边读边写
public void importUsers(MultipartFile file) throws IOException {
    try (BufferedReader reader = new BufferedReader(
            new InputStreamReader(file.getInputStream()))) {
        String line;
        List<User> batch = new ArrayList<>(1000);
        while ((line = reader.readLine()) != null) {
            User user = parseLine(line);
            batch.add(user);
            if (batch.size() >= 1000) {
                // 批量写入，释放内存
                userMapper.insertBatch(batch);
                batch.clear();
            }
        }
        // 处理剩余数据
        if (!batch.isEmpty()) {
            userMapper.insertBatch(batch);
        }
    }
}
```

**根因**：一次性加载太多数据，堆内存无法容纳。

**修复**：使用流式处理，分批读写。

## 五、JVM 参数调优：从源头预防 OOM

### 5.1 堆内存参数配置

```bash
# 基础配置
-Xms4g -Xmx4g         # 初始堆和最大堆设为一致，避免运行时堆伸缩
-XX:NewRatio=2        # 老年代:年轻代 = 2:1
-XX:SurvivorRatio=8   # Eden:Survivor = 8:1

# G1 收集器配置
-XX:+UseG1GC
-XX:MaxGCPauseMillis=200        # 最大 GC 停顿时间目标
-XX:InitiatingHeapOccupancyPercent=45  # 触发并发标记的堆使用率

# OOM 自动 dump
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/data/logs/heap
```

### 5.2 元空间参数配置

```bash
# 元空间配置（避免 Metaspace OOM）
-XX:MetaspaceSize=256m
-XX:MaxMetaspaceSize=512m
```

### 5.3 直接内存参数配置

```bash
# NIO 直接内存配置（避免 Direct Memory OOM）
-XX:MaxDirectMemorySize=1g
```

### 5.4 GC 日志配置

```bash
# GC 日志配置
-Xloggc:/data/logs/gc.log
-XX:+PrintGCDetails
-XX:+PrintGCDateStamps
-XX:+PrintTenuringDistribution
-XX:NumberOfGCLogFiles=10
-XX:GCLogFileSize=100M
```

## 六、排查命令速查

### 6.1 查看堆使用概况

```bash
jmap -heap <PID>
```

输出示例：

```
Heap Configuration:
   MaxHeapSize              = 4294967296 (4096.0MB)
   NewSize                  = 1073741824 (1024.0MB)
   OldSize                  = 3221225472 (3072.0MB)

Heap Usage:
   Eden Space: _capacity=855638016, used=789234000 (92.2%)
   From Space: _capacity=107374182, used=45678900 (42.5%)
   To Space:   capacity=107374182, used=0 (0.0%)
   Old Gen:    capacity=3221225472, used=1987654321 (61.7%)
```

重点看 **Eden Space** 的使用率。如果 Eden 区使用率长期超过 80%，说明对象创建速度太快。

### 6.2 查看对象统计

```bash
jmap -histo <PID> | head -30
```

### 6.3 监控 GC 情况

```bash
# 每秒输出一次 GC 统计
jstat -gcutil <PID> 1000
```

输出示例：

```
  S0     S1     E      O      M     YGC     YGCT    FGC    FGCT     GCT
 42.35   0.00  85.23  61.73  97.45    123   1.234   5     2.345    3.579
```

参数说明：

| 字段 | 含义 |
|---|---|
| S0/S1 | Survivor 区使用率 |
| E | Eden 区使用率 |
| O | Old 区使用率 |
| M | Metaspace 使用率 |
| YGC/YGCT | 年轻代 GC 次数和耗时 |
| FGC/FGCT | Full GC 次数和耗时 |
| GCT | 总 GC 耗时 |

如果 `FGC` 持续增长，而 `FGCT` 也在增加，说明 Old 区压力很大，可能存在内存泄漏或分配不合理。

### 6.4 监控 GC 趋势

```bash
# 每隔 1 秒输出一次 GC 统计，共输出 10 次
jstat -gc <PID> 1000 10
```

这个命令比 `-gcutil` 更详细，能看到具体的内存容量变化。

## 七、预防 OOM 的最佳实践

### 7.1 架构层面

- 对大数据量场景，使用分页、游标、批量处理，避免一次性加载全部数据
- 合理设计缓存策略，使用带过期时间和大小限制的缓存
- 对内存密集型操作，设置合理的超时和熔断机制

### 7.2 代码层面

```java
// 1. 所有资源必须关闭
try (Connection conn = dataSource.getConnection();
     PreparedStatement ps = conn.prepareStatement(sql);
     ResultSet rs = ps.executeQuery()) {
    // 处理结果
} // 自动关闭

// 2. ThreadLocal 必须清理
ThreadLocal<User> userContext = new ThreadLocal<>();
try {
    userContext.set(currentUser);
    // 业务逻辑
} finally {
    userContext.remove();
}

// 3. 避免在循环中创建大量对象
// 错误
for (int i = 0; i < 10000; i++) {
    String temp = new String("temp"); // 每次创建新对象
}

// 正确
StringBuilder sb = new StringBuilder();
for (int i = 0; i < 10000; i++) {
    sb.append("temp"); // 复用对象
}
```

### 7.3 监控层面

- 堆内存使用率告警（建议设置 70% 和 85% 两档）
- Full GC 频率告警
- Old 区使用率告警
- 定期生成 Heap Dump 分析（可使用定时任务）

## 八、面试追问：OOM 相关高频问题

### 问题一：如何避免 OOM？

**参考答案**：

OOM 预防需要从多个层面入手：

**架构层面**：
- 对大数据量场景，使用分页、游标、批量处理
- 合理设计缓存策略，使用带过期时间和大小限制的缓存
- 对内存密集型操作，设置合理的超时和熔断机制

**代码层面**：
- 所有资源必须使用 try-with-resources 或 finally 关闭
- ThreadLocal 使用后务必 remove
- 避免在循环中创建大量对象
- 监听器和回调要在不再需要时取消注册

**JVM 层面**：
- 合理设置 `-Xmx` 和 `-Xms`
- 生产环境务必设置 `-XX:MetaspaceSize` 和 `-XX:MaxMetaspaceSize`
- 开启 `-XX:+HeapDumpOnOutOfMemoryError`
- 配合成熟的监控系统，提前发现内存异常

### 问题二：内存泄漏和内存溢出的区别？

**参考答案**：

| 维度 | 内存泄漏 | 内存溢出 |
|---|---|---|
| 定义 | 对象不再使用，但因错误引用无法被 GC 回收 | 需要的内存超过 JVM 能提供的内存 |
| 特点 | 缓慢发生，随时间累积 | 突然发生，一次性需求过大 |
| 原因 | 代码 bug，对象被错误持有 | 业务需求、JVM 参数不当 |
| 表现 | OOM 前通常 GC 越来越频繁 | 直接 OOM |
| 解决 | 修复代码，移除错误引用 | 增加堆内存、优化内存使用 |

**两者的关系**：内存泄漏是内存溢出的一种原因。泄漏导致对象无法回收，最终堆内存耗尽，触发 OOM。

### 问题三：生产环境 Full GC 频繁，如何排查？

**参考答案**：

**第一步：确认 Full GC 的类型和频率**

```bash
jstat -gcutil <PID> 1000
```

观察 `FGC` 是否持续增长。

**第二步：分析 GC 日志**

重点关注：
- Full GC 的原因：Allocation Failure 还是 System.gc()？
- Full GC 后 Old 区使用率：如果下降很少，说明有大量对象无法回收
- Full GC 耗时：如果很长，说明 Old 区太大或回收效率低

**第三步：生成 Heap Dump 分析**

```bash
jmap -dump:format=b,file=/tmp/heap.hprof <PID>
```

用 MAT 分析 Dominator Tree，找出占用内存最多的对象。

**第四步：定位代码**

找出泄漏对象后，追溯 GC Roots，找出错误持有引用的代码位置。

### 问题四：MAT 分析 Heap Dump 的核心技巧有哪些？

**参考答案**：

| 技巧 | 操作 | 适用场景 |
|---|---|---|
| Leak Suspects | 自动分析 | 快速定位可疑泄漏点 |
| Histogram | 按类统计对象 | 找出占用内存最多的类 |
| Dominator Tree | 引用关系树 | 找到内存占用的根源 |
| Path to GC Roots | 引用链追踪 | 找出为什么对象无法回收 |
| OQL | 对象查询语言 | 按条件查找特定对象 |
| Top Consumers | Top消费者 | 快速查看内存占用 Top |

## 延伸思考

### 思考题一：慢增长 vs 快增长

假设你的服务运行了 3 天后开始 OOM，而同事的服务运行了 3 周才 OOM。两者的 Heap Dump 分析都发现有一个 `HashMap` 占用了 80% 的堆内存。

**问题**：你能从这个现象推断出两者的代码问题有什么区别吗？

**提示**：考虑对象的创建速度和泄漏速度。

### 思考题二：Leak Suspects 之外的线索

MAT 的 Leak Suspects 报告指出了一个 `HashMap` 是内存泄漏的嫌疑人。但这个 `HashMap` 本身是合理的业务代码，它持有对象引用是正常的。

**问题**：如何进一步分析，找出真正持有错误引用的那一层？

**提示**：考虑 GC Roots 和引用链的完整性。

### 思考题三：如何设计一个不会 OOM 的缓存系统

**问题**：如果让你设计一个企业级缓存系统，你会如何从架构层面避免 OOM？

**提示**：考虑缓存容量限制、过期策略、淘汰策略、监控告警等多个维度。

---

## 总结

Heap Space OOM 是生产环境最常见的内存问题，排查的核心在于「保存现场 → 定位泄漏 → 修复代码」。

关键要点回顾：

- **OOM 本质**：堆内存分配失败，Full GC 后仍无法满足
- **四大元凶**：内存泄漏、一次性加载数据、堆设置过小、流量洪峰
- **排查流程**：监控告警 → 获取 Dump → MAT 分析 → GC Roots → 代码修复
- **预防措施**：合理缓存、正确关闭资源、ThreadLocal 清理、监控告警

记住：OOM 不是终点，而是排查的起点。每一次 OOM 都是一次深入理解 JVM 内存管理的机会。
