# 高频场景实战面试总结

恭喜你，已经走过了秒杀系统、高并发计数器、短链系统、信息流系统、搜索系统这五个高频场景。

但学习到这里，只是完成了第一步。

面试中，场景题往往不是让你「背答案」，而是考察你「能不能用自己的话把技术讲清楚」。本文帮你梳理场景题的核心脉络，让你在面试中游刃有余。

## 一、场景题的核心框架

### 1.1 万能答题三步法

无论面试官问哪个场景题，都可以用这个框架来回答：

```
第一步：说清楚「是什么」
  └─ 这个场景要解决什么问题？核心矛盾是什么？

第二步：说清楚「怎么做」
  └─ 技术方案是什么？为什么选这个不选那个？

第三步：说清楚「权衡」
  └─ 这个方案有什么代价？什么时候会出问题？
```

### 1.2 每个场景的本质

| 场景 | 本质 | 核心问题 |
|---|---|---|
| 秒杀系统 | 资源有限 + 流量洪峰 | 如何在高并发下保证不超卖？ |
| 高并发计数器 | 原子操作 + 聚合统计 | 如何支撑 10万+ QPS 的计数？ |
| 短链系统 | 映射服务 + 海量存储 | 如何生成唯一的短码并快速跳转？ |
| 信息流系统 | 推拉模型 + Feed 排序 | 如何在写放大和读放大之间取得平衡？ |
| 搜索系统 | 倒排索引 + 相关性排序 | 如何在大数据量下实现毫秒级全文检索？ |

理解每个场景的「本质问题」，就抓住了答题的核心。

## 三、每个场景的答题模板

### 3.1 秒杀系统答题模板

**问**：如何设计一个秒杀系统？

**答**：

```
秒杀系统的核心矛盾是「流量远超系统处理能力」。

我先说整体架构：
┌────────────────────────────────────────┐
│ CDN层 → 网关层 → 应用层 → 缓存层 → 消息层 → 数据库层 │
└────────────────────────────────────────┘

技术要点有三个：

第一，限流防刷。我用令牌桶算法做全局限流，同时用用户维度的限流防止刷子。
Redis + Lua 脚本实现，每秒可以支撑 10 万+ 请求。

第二，库存扣减。库存提前预热到 Redis，使用 Lua 脚本保证原子性。
核心逻辑是：检查库存 → 扣减库存 → 记录购买记录，三步合一，保证不超卖。

第三，异步下单。用户扣减库存成功后，发送 MQ 消息，异步创建订单。
这样可以把高峰期 QPS 从 10 万降级到数据库可承受的范围。

超卖问题：Redis Lua 是主力方案，数据库唯一约束是兜底方案。
数据一致性：最终一致性 + 定时对账，Redis 和 MySQL 差异过大会告警。
```

### 3.2 高并发计数器答题模板

**问**：如何设计一个支撑 10 万 QPS 的计数器？

**答**：

```
计数器看起来简单，但要在高并发下保证正确性和性能，需要考虑很多细节。

技术方案分三层：

第一层：Redis 原子操作。INCR/DECR 命令是原子的，并发安全。
如果是复杂逻辑（如带检查的库存扣减），用 Lua 脚本。

第二层：防刷保护。同一个用户不能无限增加计数。
我会在 Redis 中用滑动窗口限流，每秒最多 N 次。

第三层：聚合同步。Redis 计数器会丢（Redis 挂了数据就没了），
所以需要定时同步到 MySQL。策略是：写 Redis + 写 pending 队列，
后台任务每分钟批量同步到 MySQL。

不同精度要求用不同方案：
- 精确计数：Redis + MySQL 最终一致
- 大数据量去重：HyperLogLog（误差 0.81%）
- 中等数据量去重：Bitmap（100% 精确，需要连续 ID）
```

### 3.3 短链系统答题模板

**问**：如何设计一个每天生成 1000 万条短链的系统？

**答**：

```
短链系统本质是一个映射服务：长 URL ↔ 短码。

第一，短码生成。我用雪花算法生成 64 位 ID，再 Base62 编码成 6-8 位字符串。
这样可以保证分布式唯一、时间有序。

第二，存储设计。按短码 Hash 分 8 库 64 表，
热点数据放 Redis，全量数据放 MySQL。

第三，跳转优化。多级缓存：本地缓存 → Redis → MySQL。
301 和 302 的选择：需要 SEO 权重传递用 301，正常追踪用 302。

容量规划：1000 万/天 × 365 天 ≈ 36 亿条/年，
每条 ~1KB，总存储 ~3.6TB。分库分表 + 冷数据归档可以支撑。

安全性：频率限制（单用户每小时最多生成 100 条），
黑名单过滤（钓鱼网站、恶意域名），敏感词检测。
```

### 3.4 信息流系统答题模板

**问**：如何设计一个微博 Timeline 系统？

**答**：

```
Timeline 系统的核心问题是「写放大」和「读放大」的矛盾。

我先说三种模型的选择：

推模型（Fan-out on Write）：用户发微博时，主动推给粉丝。
- 优点：读取快
- 缺点：大 V 发一条微博要写 millions 条（写放大）

拉模型（Fan-out on Read）：用户读取时，拉取关注列表的微博。
- 优点：没有写放大
- 缺点：读取时要合并 N 个来源的数据

混合模型（Hybrid）：普通用户用推，大 V 用拉。
这是业界主流方案，Twitter/微博 都用这个。

存储用 Redis Sorted Set，按时间戳排序。
分页用 Cursor-based，不建议用 Offset（深分页性能差）。

排序策略：用 EdgeRank 类算法，综合考虑互动权重、亲密度、时间衰减。
大 V 问题（Kardashian Index）：热点用户内容单独处理，避免刷屏。
```

### 3.5 搜索系统答题模板

**问**：如何设计一个每天支持 1 亿次搜索请求的系统？

**答**：

```
搜索系统的核心是倒排索引：词 → 文档列表。

ES 架构设计：
- 5 个主分片 + 1 副本（可根据数据量调整）
- 冷热分离：热数据多副本、快刷新，冷数据少副本、慢刷新
- 按业务字段 routing，避免跨分片查询

查询优化：
- 用 filter 替代 must（filter 不计算评分，性能更好）
- 深分页用 search_after，不用 from/size
- 只返回需要的字段，减少网络传输

相关性排序：
- 默认 BM25 算法
- 可用 function_score 自定义排序（时间衰减、热度加权）

数据同步：MySQL → ES 用 Canal 监听 binlog，
准实时同步，支持断点续传。
```

## 四、面试高频追问

### 4.1 「如果 Redis 挂了怎么办？」

**参考答案**：

这是考察你系统韧性的问题。要从多个层面回答：

```
第一层：缓存失效后的降级
- 缓存失效时，直接查数据库（限流保护）
- 熔断器打开，避免数据库被打爆

第二层：数据恢复
- Redis 启动时，从 MySQL 加载最新数据
- 关键数据（如库存）持久化到磁盘

第三层：高可用架构
- Redis Cluster / 主从自动切换
- 多机房部署，主备自动切换

第四层：监控告警
- Redis 不可用时立即告警
- 自动化工单触发
```

### 4.2 「如果数据库扛不住怎么办？」

**参考答案**：

```
这个问题的本质是「数据库是系统瓶颈，如何扛住流量」。

方案一：读写分离
- 读请求走从库，写请求走主库
- 延迟问题：主从同步有延迟，对一致性要求高的读走主库

方案二：分库分表
- 按用户 ID 或时间分片
- 跨表查询用 ES 或 NewSQL兜底

方案三：降级非核心功能
- 非核心页面降级，减少数据库压力
- 页面静态化，减少动态请求

方案四：流量控制
- 限流 + 排队，数据库能力有限
- 超出阈值的请求直接返回「系统繁忙」
```

### 4.3 「如果出现超卖，你怎么处理？」

**参考答案**：

```
超卖是秒杀系统的底线问题。

预防：Redis Lua 脚本保证原子扣减，数据库唯一约束作为兜底。

监控：实时监控库存扣减成功率和订单创建成功率，
如果扣减成功但创建失败的比例过高，立即告警。

处理：
1. 立即停止活动，防止损失扩大
2. 查询已超卖的订单数量
3. 联系已超卖用户，退款 + 赔偿
4. 排查问题根因（Redis bug？Lua 脚本逻辑问题？）
5. 修复后重新上架
```

## 五、场景迁移能力

### 5.1 如何把一个场景的知识迁移到另一个场景

核心思路：**找共同点**

| 场景 A | 场景 B | 共同技术 |
|---|---|---|
| 秒杀库存扣减 | 计数器防刷 | Redis 原子操作 + Lua 脚本 |
| 秒杀异步下单 | 信息流 Feed 推送 | 消息队列削峰 |
| 短链跳转 | 信息流加载 | 多级缓存 |
| 搜索结果排序 | 信息流排序 | 相关性算法 |
| 计数器聚合 | 短链统计 | 最终一致性 |

### 5.2 遇到新场景的思考框架

遇到没见过的场景题，用这个框架思考：

```
1. 业务特点是什么？
   - QPS 多高？数据量多大？
   - 一致性要求？实时性要求？

2. 核心矛盾是什么？
   - 流量大？数据多？一致性难？

3. 可以用哪些已知技术解决？
   - 缓存？限流？分库分表？消息队列？

4. 权衡是什么？
   - 性能 vs 一致性
   - 复杂度 vs 可维护性
```

## 七、常见错误与避免

### 7.1 过度设计

:::warning 面试中的大忌

面试官问「如何设计一个秒杀系统」，你上来就说「我们需要 100 台服务器，10 个机房……」

这是减分项。面试官要的是「够用」而不是「炫技」。先回答核心问题，再根据追问扩展。

::::

**正确做法**：先说核心方案，再说可扩展点，给面试官留追问空间。

### 7.2 只背答案，不讲原理

**错误示范**：

> 「库存扣减用 Redis Lua 脚本。」

**追问就死**：为什么用 Lua？Lua 脚本是怎么保证原子性的？

**正确示范**：

> 「库存扣减用 Redis Lua 脚本。因为 Redis 单个命令是原子的，但『检查-扣减-记录』三步不是原子的。Lua 脚本在 Redis 中执行时，会被当作一个整体，不会被其他命令插队，从而保证了原子性。」

### 7.3 忽略业务背景

**错误示范**：

> 「用 Redis 做计数器，因为 Redis 性能高。」

**正确示范**：

> 「要看业务特点。如果是点赞计数，允许 0.81% 的误差，可以用 HyperLogLog；如果像库存扣减一样要求精确，必须用 Redis 原子操作 + 数据库兜底。」

## 八、关联到自己的项目经验

面试中最重要的加分项：**用自己的项目经验回答场景题**。

### 8.1 如何准备

```
1. 梳理自己的项目经历，找出和场景相关的实践
   - 秒杀类：促销活动、大促系统
   - 高并发类：实时统计、排行榜
   - 搜索类：商品搜索、内容检索

2. 准备一个「项目 → 场景」的映射
   - 我的 XX 项目 → 类似秒杀的高并发场景
   - 我们当时的方案是 XX，遇到了 XX 问题

3. 用 STAR 法则描述
   - Situation：什么场景
   - Task：什么任务
   - Action：采取了什么行动
   - Result：取得了什么结果
```

### 8.2 示例回答

> 「我之前做过一个商品详情页的浏览量统计系统，和高并发计数器场景类似。当时遇到的问题是 MySQL 更新频繁，性能跟不上。我们的方案是用 Redis 做计数缓冲：用户浏览页面时，只写 Redis INCR；后台任务每分钟批量同步到 MySQL。后来又遇到 Redis 宕机丢数据的问题，我们加了 checkpoint 机制，定期将计数保存到 MySQL，保证数据不丢。」

## 面试追问

### 问题一：如何在面试中展示自己的深度？

**参考答案**：

1. **知其然，也要知其所以然**。不仅知道「用 Redis」，还要知道「为什么用 Redis」。

2. **主动说坑**。说一个你踩过的坑，以及如何解决的，比背答案加分 10 倍。

3. **画图说明**。能用图画清楚的，就不要用文字说。

4. **量化数据**。不要说「性能很好」，说「QPS 从 1000 提升到 10000，提升了 10 倍」。

### 问题二：如果面试官追问到一个你不熟悉的点怎么办？

**参考答案**：

```
诚实 + 思路 比装懂强 100 倍。

「这个点我了解得不深，但我理解它的作用是 XX。
如果让我来解决这个问题，我会先去查官方文档，然后看看我们项目中的实际情况……」

展现出你的学习能力和解决问题的思路，比答对所有问题更重要。
```

## 总结

场景实战的学习，核心是三点：

1. **理解本质**：每个场景解决什么核心问题
2. **理解权衡**：每个技术选型的代价是什么
3. **理解迁移**：场景之间的共同技术是什么

记住面试不是考试，没有标准答案。面试官要看到的是：
- 你能不能把复杂问题讲简单
- 你有没有系统性思维
- 你有没有踩过坑、总结过经验

把这些展示出来，你就赢了。
