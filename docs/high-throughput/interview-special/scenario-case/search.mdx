# 搜索系统设计：Elasticsearch 与全文检索优化

Google 搜索「Java 面试题」，0.5 秒内返回 1000 万条结果。你有没有想过，这些结果是怎么来的？

传统 SQL 的 `LIKE '%Java%'` 根本扛不住这个数据量。亿级数据下，一次 LIKE 查询可能要几十秒。

答案是倒排索引（Inverted Index）——这是现代搜索引擎的基石。

这篇文章，带你彻底搞懂搜索系统的设计。

## 一、搜索系统的本质

### 1.1 传统数据库查询的局限

```sql
-- 性能杀手：全表扫描
SELECT * FROM articles WHERE title LIKE '%Java%' OR content LIKE '%Java%';

-- 索引失效：无法使用 B+Tree
SELECT * FROM articles WHERE MATCH(title) AGAINST('Java' IN BOOLEAN MODE);
```

| 问题 | 说明 |
|---|---|
| **全表扫描** | LIKE '%keyword%' 无法使用索引，必须扫描全表 |
| **无法排序** | 无法按相关性排序 |
| **不支持分词** | 无法搜索「Java 面试」，只能搜索完整词 |
| **性能差** | 亿级数据下，查询可能需要几十秒 |

### 1.2 倒排索引原理

传统索引（正排索引）：

```
文档1: {title: "Java 面试题", content: "本文介绍 Java..."}
文档2: {title: "Python 教程", content: "Python 入门..."}
```

倒排索引：

```
"Java"   → [文档1]
"面试题" → [文档1]
"Python" → [文档2]
"教程"   → [文档2]
```

倒排索引将「文档 → 词」的反转成「词 → 文档列表」，查询时直接根据词找到文档。

### 1.3 搜索系统的核心组件

```
┌─────────────────────────────────────────────────────────────────┐
│                       搜索请求                                   │
│                    query: "Java 面试题"                          │
└─────────────────────────────────────────────────────────────────┘
                                │
                                ▼
┌─────────────────────────────────────────────────────────────────┐
│                      Query Parser                                │
│              分词 → 语法解析 → 生成查询树                          │
└─────────────────────────────────────────────────────────────────┘
                                │
                                ▼
┌─────────────────────────────────────────────────────────────────┐
│                      Query Rewriter                              │
│                   改写查询（如同义词、纠错）                        │
└─────────────────────────────────────────────────────────────────┘
                                │

┌─────────────────────────────────────────────────────────────────┐
│                       搜索引擎                                    │
│                    倒排索引 + 相关性计算                           │
└─────────────────────────────────────────────────────────────────┘
                                │

┌─────────────────────────────────────────────────────────────────┐
│                    结果处理器                                     │
│                排序 → 聚合 → 高亮 → 分页                          │
└─────────────────────────────────────────────────────────────────┘
```

## 二、Elasticsearch 架构

### 2.1 ES 集群架构

```
┌─────────────────────────────────────────────────────────────────┐
│                       ES 集群                                    │
│                                                                 │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐                       │
│  │ Master   │  │ Master   │  │ Master   │   ← Master 节点（协调） │
│  │ Node 1   │  │ Node 2   │  │ Node 3   │                       │
│  └──────────┘  └──────────┘  └──────────┘                       │
│                                                                 │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐         │
│  │ Data     │  │ Data     │  │ Data     │  │ Data     │         │
│  │ Shard 0  │  │ Shard 1  │  │ Shard 2  │  │ Shard 3  │         │
│  └──────────┘  └──────────┘  └──────────┘  └──────────┘         │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

| 节点类型 | 职责 | 配置要求 |
|---|---|---|
| **Master 节点** | 集群管理、索引创建、Shard 分配 | 高可用，3 个节点 |
| **Data 节点** | 数据存储、文档 CRUD、搜索执行 | 内存大，磁盘快 |
| **Coordinating 节点** | 接收请求、分发到 Data 节点、聚合结果 | 高并发，横向扩展 |
| **Ingest 节点** | 文档预处理（如数据转换） | 可选 |

### 2.2 分片与副本

```json
// 创建索引时指定分片数
PUT /articles
{
  "settings": {
    "number_of_shards": 5,      // 主分片数（创建后不可改）
    "number_of_replicas": 1     // 副本数（可动态调整）
  },
  "mappings": {
    "properties": {
      "title": { "type": "text", "analyzer": "ik_max_word" },
      "content": { "type": "text", "analyzer": "ik_max_word" },
      "author": { "type": "keyword" },
      "created_at": { "type": "date" },
      "view_count": { "type": "long" }
    }
  }
}
```

```
Primary Shard 0 ──副本──▶ Shard 0'
Primary Shard 1 ──副本──▶ Shard 1'
Primary Shard 2 ──副本──▶ Shard 2'
Primary Shard 3 ──副本──▶ Shard 3'
Primary Shard 4 ──副本──▶ Shard 4'
```

### 2.3 分片数规划

| 数据量 | 推荐分片数 | 说明 |
|---|---|---|
| < 1 亿 | 3-5 | 小规模，单节点即可 |
| 1-10 亿 | 5-10 | 中等规模，需要多节点 |
| 10 亿+ | 10-20+ | 大规模，考虑冷热分离 |

:::warning 分片数不是越多越好

每个分片都有开销（内存、文件描述符、Master 管理）。分片数过多会导致集群压力增大。

- 单个分片数据量建议 < 50GB
- 分片数 = 数据总量 / (节点数 × 50GB)

::::

```java
@RestController
public class ArticleSearchController {
    
    @Autowired
    private ElasticsearchRestTemplate elasticsearchTemplate;
    
    @GetMapping("/search")
    public SearchResult search(@RequestParam String keyword) {
        NativeQuery query = NativeQuery.builder()
            .withQuery(q -> q.match(m -> m.field("title").query(keyword)))
            .build();
        
        SearchHits<Article> hits = elasticsearchTemplate.search(query, Article.class);
        return new SearchResult(hits.getTotalHits(), hits.getSearchHits());
    }
}
```

## 五、相关性排序

### 5.1 TF-IDF 算法

**TF（词频）**：词在文档中出现次数越多，文档越相关。

```
TF = 词在文档中出现次数 / 文档总词数
```

**IDF（逆文档频率）**：词在所有文档中出现越少，越能区分文档。

```
IDF = log(总文档数 / 包含该词的文档数)
```

**TF-IDF = TF × IDF**

```json
// TF-IDF 排序
GET /articles/_search
{
  "query": {
    "match": { "content": "Java" }
  }
}
```

### 5.2 BM25 算法

ES 默认使用 BM25（Better Okapi BM25），改进了 TF-IDF 的饱和问题。

```
Score = IDF × (TF × (k1 + 1)) / (TF + k1 × (1 - b + b × dl/avgdl))

其中：
- k1: 词频饱和参数（默认 1.2）
- b: 文档长度归一化参数（默认 0.75）
- dl: 文档长度
- avgdl: 平均文档长度
```

### 5.3 自定义评分


## 六、分页与性能优化

### 6.1 分页方式对比

| 方式 | 实现 | 优点 | 缺点 |
|---|---|---|---|
| **from/size** | `{"from": 0, "size": 10}` | 简单，通用 | 深分页性能差 |
| **search_after** | 游标分页 | 性能好 | 不能随机跳页 |
| **scroll** | 快照分页 | 适合批量导出 | 占用资源 |

### 6.2 from/size 深分页问题

```json
// 性能杀手：请求第 10000 页
GET /articles/_search
{
  "from": 10000,
  "size": 10
}
```

ES 的深分页需要从每个分片获取 `from + size` 条数据，然后汇总排序。

```
假设 5 个分片，要获取第 10000 页（每页 10 条）：
- 每个分片需要获取 10010 条数据
- 总共需要获取 50050 条数据
- 内存和带宽消耗巨大
```

### 6.3 search_after 游标分页

```json
// 第一次查询
GET /articles/_search
{
  "size": 10,
  "sort": [
    { "created_at": "desc" },
    { "_id": "asc" }
  ],
  "query": {
    "match": { "content": "Java" }
  }
}

// 响应中获取最后一条的 sort 值
// "sort": ["2024-01-15T10:30:00Z", "doc_123"]

// 第二次查询，使用 search_after
GET /articles/_search
{
  "size": 10,
  "sort": [
    { "created_at": "desc" },
    { "_id": "asc" }
  ],
  "search_after": ["2024-01-15T10:30:00Z", "doc_123"],
  "query": {
    "match": { "content": "Java" }
  }
}
```

```java
// Java 实现 search_after
public List<Article> searchWithCursor(String keyword, int pageSize) {
    if (lastSortValues == null) {
        // 首次查询
        SearchQuery searchQuery = new NativeSearchQueryBuilder()
            .withQuery(q -> q.match(m -> m.field("content").query(keyword)))
            .withSort(SortBuilders.fieldSort("created_at").order(SortOrder.Desc))
            .withSort(SortBuilders.fieldSort("_id").order(SortOrder.Asc))
            .withMaxResults(pageSize)
            .build();
        
        SearchHits<Article> hits = elasticsearchTemplate.search(searchQuery, Article.class);
        lastSortValues = hits.getSortValues();
        return hits.getSearchHits().stream().map(SearchHit::getContent).collect(Collectors.toList());
    } else {
        // 使用 search_after
        SearchQuery searchQuery = new NativeSearchQueryBuilder()
            .withQuery(q -> q.match(m -> m.field("content").query(keyword)))
            .withSort(SortBuilders.fieldSort("created_at").order(SortOrder.Desc))
            .withSort(SortBuilders.fieldSort("_id").order(SortOrder.Asc))
            .withSearchAfter(lastSortValues)
            .withMaxResults(pageSize)
            .build();
        
        SearchHits<Article> hits = elasticsearchTemplate.search(searchQuery, Article.class);
        lastSortValues = hits.getSortValues();
        return hits.getSearchHits().stream().map(SearchHit::getContent).collect(Collectors.toList());
    }
}
```

## 八、数据同步：MySQL → ES

### 8.1 同步方案对比

| 方案 | 原理 | 优点 | 缺点 |
|---|---|---|---|
| **Logstash** | JDBC 定时同步 | 简单 | 延迟高 |
| **Canal** | 监听 binlog | 准实时 | 复杂 |
| **Debezium** | CDC 变更捕获 | 异构数据库 | 学习成本 |
| **双写** | 应用层同时写两个库 | 简单 | 耦合、高延迟 |

### 8.2 Canal 同步方案

```yaml
# Canal 配置文件
server:
  port: 11111

canal:
  destinations:
    article_sync:
      zookeeper: zk:2181
      elasticsearch:
        host: es:9200
        index: articles
```

```java
// Canal 客户端监听
@Component
public class CanalSyncListener {
    
    @Autowired
    private ElasticsearchRestTemplate elasticsearchTemplate;
    
    @CanalTable(value = "articles")
    @Process波Listener
    public void onMessage(CanalEntry.Entry entry) {
        List<CanalEntry.RowChange> rowChanges = entry.getRowChanges();
        
        for (CanalEntry.RowChange rowChange : rowChanges) {
            CanalEntry.EventType eventType = rowChange.getEventType();
            
            switch (eventType) {
                case INSERT:
                case UPDATE:
                    for (CanalEntry.RowData rowData : rowChange.getRowDatasList()) {
                        Article article = parseRowData(rowData, rowChange.getBeforeColumnsList());
                        elasticsearchTemplate.save(article);
                    }
                    break;
                case DELETE:
                    for (CanalEntry.RowData rowData : rowChange.getRowDatasList()) {
                        Long id = getId(rowData.getBeforeColumnsList());
                        elasticsearchTemplate.delete(String.valueOf(id), Article.class);
                    }
                    break;
            }
        }
    }
}
```

### 8.3 数据一致性保证

```java
@Service
public class DataSyncService {
    
    /**
     * 全量同步 + 增量同步
     */
    @Scheduled(cron = "0 0 2 * * ?")  // 每天凌晨 2 点全量
    public void fullSync() {
        // 1. 获取 ES 最大 ID
        Long maxEsId = getMaxEsId();
        
        // 2. 从 MySQL 增量同步（ID > maxEsId）
        while (true) {
            List<Article> articles = mysqlMapper.selectArticlesByIdRange(maxEsId, 1000);
            if (articles.isEmpty()) break;
            
            elasticsearchTemplate.save(articles);
            maxEsId = articles.get(articles.size() - 1).getId();
        }
    }
    
    /**
     * 数据校验
     */
    public void verifyData() {
        long mysqlCount = mysqlMapper.countArticles();
        long esCount = elasticsearchTemplate.count(new NativeQuery(), Article.class);
        
        if (mysqlCount != esCount) {
            alertService.sendAlert("ES 数据不一致", 
                String.format("MySQL: %d, ES: %d", mysqlCount, esCount));
        }
    }
}
```

## 十、面试高频问题

### 问题一：ES 和 MySQL 的区别？

**参考答案**：

| 维度 | Elasticsearch | MySQL |
|---|---|---|
| **数据结构** | 文档型（JSON） | 关系型 |
| **索引类型** | 倒排索引 | B+Tree 索引 |
| **查询能力** | 全文检索、模糊搜索 | 精确匹配 |
| **事务支持** | 无 | 支持 |
| **JOIN 支持** | 弱 | 强 |
| **一致性** | 最终一致 | 强一致 |
| **写入性能** | 优于写入延迟 | 取决于事务级别 |
| **适用场景** | 搜索、分析、日志 | OLTP 业务 |

### 问题二：如何优化 ES 查询性能？

**参考答案**：

1. **使用 filter 替代 must**：filter 不计算评分，性能更好
2. **避免通配符查询**：如 `keyword*`，性能差
3. **合理设置分片数**：单分片数据量 < 50GB
4. **使用 routing**：针对固定条件查询
5. **预热查询缓存**：热点查询提前加载
6. **控制返回字段**：只返回需要的字段
7. **使用 async search**：大查询异步处理

### 问题三：如何实现拼音搜索？

**参考答案**：

1. **安装拼音分词器插件**：`elasticsearch-analysis-pinyin`
2. **配置拼音字段**：

```json
PUT /articles
{
  "settings": {
    "analysis": {
      "analyzer": {
        "pinyin_analyzer": {
          "tokenizer": "pinyin_tokenizer"
        }
      },
      "tokenizer": {
        "pinyin_tokenizer": {
          "type": "pinyin",
          "lowercase": true
        }
      }
    }
  },
  "mappings": {
    "properties": {
      "title": {
        "type": "text",
        "analyzer": "ik_max_word"
      },
      "title_pinyin": {
        "type": "text",
        "analyzer": "pinyin_analyzer"
      }
    }
  }
}
```

## 总结

搜索系统的设计，核心在于三点：

1. **倒排索引**：词 → 文档列表的映射，支持快速全文检索
2. **相关性排序**：TF-IDF / BM25 算法，支持智能排序
3. **分布式架构**：分片 + 副本 + 协调节点，支撑海量数据

关键优化点：

- **分词器**：选择合适的分词器（IK、拼音等）
- **查询优化**：filter vs must、search_after 分页
- **索引优化**：refresh_interval、段合并
- **数据同步**：Canal / Debezium CDC

理解了这三点和四个优化，你就掌握了搜索系统的精髓。
