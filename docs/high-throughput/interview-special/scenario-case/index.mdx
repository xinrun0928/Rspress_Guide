# 高频场景实战：概述

面试中，你有没有遇到过这种情况——面试官问了一道系统设计题，你脑子里飞速闪过各种技术名词：缓存、消息队列、分库分表、负载均衡……但就是不知道从哪说起，最后东一榔头西一棒槌，回答得毫无章法。

这就是场景实战要解决的问题。

## 一、为什么需要场景实战

很多人学技术是「点状」的——知道 Redis 可以做缓存，知道消息队列可以削峰，知道分库分表可以解决数据量问题。但面试官一问「如何设计一个秒杀系统」，就傻眼了：这些知识点怎么组合？先做什么后做什么？为什么要这样做？

场景实战的价值，就是把「点状知识」串成「线状能力」，让你在面试中能够从容应对任何系统设计题。

:::tip 面试官想看到什么

面试官问场景题，不是要你背诵答案，而是要考察三个层次：

1. **你理解不理解**：对每个技术的原理有清晰认知
2. **你能不能用**：知道在什么场景下用什么技术
3. **你想没想过权衡**：知道每个选择都有代价，能权衡利弊

::::

## 二、场景实战的通用模式

所有高频场景题，都有一个共同的结构：

```
┌─────────────────────────────────────────────────────────┐
│                      业务场景                            │
│              （流量特点、核心矛盾、约束条件）                │
└─────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────┐
│                     技术选型                            │
│      （架构分层、技术组合、为什么选这个不选那个）             │
└─────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────┐
│                     细节落地                            │
│           （核心代码、关键参数、常见坑点）                   │
└─────────────────────────────────────────────────────────┘
```

理解了这个模式，你就有了答题的「骨架」。

## 三、场景分类与对应技术栈

### 3.1 高流量类场景

**典型场景**：秒杀、抢购、突发流量

这类场景的核心矛盾是**流量远超系统处理能力**，技术重心在限流、削峰、防超卖。

| 场景 | 核心技术点 | 关联知识点 |
|---|---|---|
| 电商秒杀 | 令牌桶限流、库存预扣减、异步下单 | 限流算法、Redis 原子操作、消息队列 |
| 限时抢购 | 热点数据隔离、分层缓存、防刷 | 缓存策略、分布式锁、接口幂等 |

### 3.2 高并发读写类场景

**典型场景**：计数器、信息流、实时排名

这类场景的核心矛盾是**读多写多、性能要求高**，技术重心在缓存、异步、读写分离。

| 场景 | 核心技术点 | 关联知识点 |
|---|---|---|
| 高并发计数器 | Redis 原子计数、HyperLogLog、聚合同步 | Redis 数据结构、Pipeline、最终一致性 |
| 信息流系统 | 推拉结合、Timeline 缓存、分页 | 缓存策略、Cassandra、Feed Ranking |
| 实时排行榜 | 有序集合、分段缓存、异步聚合 | Redis Sorted Set、滑动窗口 |

### 3.3 存储与查询类场景

**典型场景**：短链系统、搜索系统、海量存储

这类场景的核心矛盾是**数据量大、查询性能要求高**，技术重心在存储设计、索引优化、缓存。

| 场景 | 核心技术点 | 关联知识点 |
|---|---|---|
| 短链系统 | Base62 编码、301/302 重定向、容量规划 | Hash 碰撞、数据库分片、CDN |
| 搜索系统 | 倒排索引、分片副本、相关性排序 | ES 架构、Query DSL、BM25 |
| 海量数据存储 | 分库分表、冷热分离、归档策略 | ShardingSphere、DataX、容量规划 |

## 四、每个场景的答题框架

### 4.1 第一层：理解业务

面试官出一道题，第一步不是开始画架构图，而是先理解业务。

**必问问题**：

- 每秒请求量是多少？（QPS / TPS）
- 数据规模有多大？（总数据量、增长量）
- 需要保证什么特性？（一致性、可用性、实时性）
- 有什么特殊约束？（预算、技术栈、团队经验）

这些问题的答案决定了你后面的技术选型。没有业务背景的技术方案，都是空中楼阁。

### 4.2 第二层：画出架构

有了业务背景，接下来就是画架构图。

**架构图三要素**：

1. **分层清晰**：前端层、网关层、应用层、数据层
2. **关键组件标注**：每个组件的名字和作用
3. **数据流向标注**：请求从哪来到哪去

:::warning 不要过度设计

面试不是架构设计大赛，面试官要的是「够用」而不是「炫技」。能用单机解决的不上分布式，能用缓存解决的不用分库分表。每一个复杂技术都是有代价的。

::::

### 4.3 第三层：核心技术点深入

架构图画完后，面试官往往会追问细节。这就到了考验你「深度」的时候了。

**常见追问方向**：

- 如果 Redis 挂了怎么办？（缓存策略）
- 如果消息队列积压了怎么办？（消费者扩容）
- 如果数据库慢查询怎么办？（SQL 优化、索引）
- 如果超卖了怎么办？（分布式锁、乐观锁、数据库约束）

追问是你展示深度的机会。把每个技术点的原理、坑点、权衡都讲清楚，面试官想不给你高分都难。

## 五、如何用场景实战备战面试

### 5.1 练习顺序建议

**第一轮：理解每个场景**

先把八个场景全部过一遍，知道每个场景「是什么」。

**第二轮：理解技术关联**

找出场景之间的共同技术点，比如 Redis 在多个场景中都出现了，理解 Redis 为什么是高性能系统的标配。

**第三轮：自己复述**

关上课本，用自己的话把每个场景讲一遍。能讲清楚才是真的理解。

**第四轮：关联到自己的项目**

每个场景找一个自己项目中的对应实践，比如「我的项目里是怎么做缓存的」「我遇到过的 Redis 坑」。

### 5.2 面试中的场景选择

不是每个面试都会问所有场景，常见的高频考察场景：

| 考察频率 | 场景 | 难度系数 |
|---|---|---|
| 最高 | 秒杀系统、分布式 ID | ⭐⭐⭐⭐ |
| 高 | 短链系统、信息流、搜索 | ⭐⭐⭐⭐ |
| 中等 | 高并发计数器、排行榜 | ⭐⭐⭐ |

优先把高频场景练熟。

## 六、自测：你真的理解了吗？

学完场景实战，可以用这几个问题自测：

1. **架构还原**：能不能在白板上画出秒杀系统的完整架构？不用写代码，画架构就行。
2. **权衡思考**：秒杀系统为什么不用分布式事务？什么场景下可以用？
3. **深度追问**：如果面试官问「Redis 库存扣减的原子性怎么保证」，你能不能回答上来？
4. **关联迁移**：秒杀系统的限流方案，能不能迁移到其他高流量场景？

能回答上来，说明你真的理解了；回答不上来，就回到对应的章节继续啃。

## 七、场景之间的关联图谱

八个场景不是孤立的，它们共享很多技术点：

```
                    ┌─────────────┐
                    │   Redis     │
                    │ (核心组件)   │
                    └──────┬──────┘
                           │
     ┌─────────┬───────────┼───────────┬─────────┐
     │         │           │           │         │
     ▼         ▼           ▼           ▼         ▼
┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐
│  秒杀   │ │ 计数器   │ │ 短链    │ │ 信息流   │ │ 搜索    │
└─────────┘ └─────────┘ └─────────┘ └─────────┘ └─────────┘
```

**Redis** 是高频场景中出场最多的组件：库存扣减、计数器、短链映射、Feed 缓存，都离不开它。Redis 学透了，场景实战就通了一半。

## 八、延伸阅读建议

场景实战不是孤立的，每个场景背后都有更深的知识点：

| 场景 | 深入学习方向 | 推荐阅读 |
|---|---|---|
| 秒杀系统 | 分布式锁、消息队列、接口幂等 | 高可用架构 - 限流熔断 |
| 高并发计数器 | Redis 数据结构、Pipeline、持久化 | 高性能优化 - Redis 深度 |
| 短链系统 | 数据库分片、Hash 算法、CDN | 高性能优化 - 数据库 |
| 信息流系统 | Cassandra、缓存策略、Feed Ranking | 高并发设计 - 缓存架构 |
| 搜索系统 | ES 架构、倒排索引、相关性排序 | 高性能优化 - ES 深度 |

## 面试追问

1. **如何判断一个场景应该用缓存还是用数据库？** 考虑读写比例、一致性要求、数据量大小。
2. **如果系统从单机扩展到集群，哪些地方需要改？** 负载均衡、Session 管理、分布式锁、缓存一致性。
3. **如何评估一个架构方案的优劣？** 性能（QPS/RT）、成本、可维护性、扩展性。
