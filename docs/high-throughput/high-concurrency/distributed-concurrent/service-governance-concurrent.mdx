# 服务治理（限流、熔断、负载均衡）

---

## 场景切入：秒杀系统的"雪崩"之夜

凌晨 0 点，准时开始的秒杀活动吸引了 10 万用户同时涌入。订单系统压力骤然飙升，响应时间从 200ms 飙升到 10 秒。一部分请求超时失败，调用方不断重试，订单系统的压力进一步加剧。很快，库存服务也扛不住了，响应时间同样飙升。紧接着，支付服务、用户服务相继宕机……

这，就是传说中的"雪崩效应"——一个服务的故障，引发了连锁反应，最终导致整个系统崩溃。

你有没有想过：**如何阻止这种连锁反应？一个服务故障时，如何保护系统不被拖垮？当流量激增时，如何让系统"优雅地"拒绝部分请求，而不是全面崩溃？**

这就是服务治理要解决的问题。

---

## 一、为什么需要服务治理

### 1.1 微服务架构的核心挑战

微服务架构带来了灵活性和可扩展性，但也引入了新的挑战：

**挑战一：服务数量爆炸**

一个中型的互联网系统可能有几十甚至上百个微服务，它们之间存在复杂的依赖关系。

```java
// 典型的微服务调用链
OrderController
    └── OrderService
            ├── InventoryService（查询库存）
            │       └── WarehouseService（查询仓库）
            ├── PriceService（计算价格）
            │       ├── ProductService（查询商品）
            │       └── PromotionService（查询促销）
            └── PaymentService（支付）
                    ├── AccountService（账户服务）
                    └── BankGateway（银行网关）
```

**挑战二：服务故障的传播**

在依赖链中，如果某个服务故障，调用方会不断重试、加剧故障服务的压力：

```
库存服务故障（响应慢/超时）
        │
        ├── 订单服务调用库存服务
        │       ├── 超时重试
        │       ├── 重试加剧库存服务压力
        │       └── 订单服务资源耗尽
        │
        ├── 价格服务调用库存服务
        │       └── 价格服务也受影响
        │
        └── 整个系统雪崩
```

**挑战三：流量波动不可预测**

流量可能突然爆发（如秒杀活动、热点事件），也可能突然消失。系统需要能够动态应对这种波动。

### 1.2 服务治理的核心目标

服务治理的目标是**在分布式系统中，保证服务的可用性和稳定性**，核心包括：

| 治理方向 | 说明 |
|---|---|
| **限流** | 控制请求速率，避免系统过载 |
| **熔断** | 快速失败，防止故障传播 |
| **负载均衡** | 合理分配请求，充分利用资源 |
| **服务发现** | 自动感知服务实例的变化 |
| **健康检查** | 剔除故障实例，保证服务可用 |

## 二、限流：保护系统的"第一道防线"

### 2.1 限流算法：四大流派

限流的核心问题是：**如何判断当前请求是否应该被拒绝？**

**算法一：固定窗口算法**

将时间划分为固定窗口，在每个窗口内统计请求数：

```java
// 固定窗口限流
public class FixedWindowRateLimiter {
    private final int maxRequests;  // 窗口内最大请求数
    private final long windowSize; // 窗口大小（毫秒）
    private long windowStart;
    private int count;

    public FixedWindowRateLimiter(int maxRequests, long windowSize) {
        this.maxRequests = maxRequests;
        this.windowSize = windowSize;
        this.windowStart = System.currentTimeMillis();
    }

    public synchronized boolean tryAcquire() {
        long now = System.currentTimeMillis();

        // 窗口过期，重置
        if (now - windowStart >= windowSize) {
            windowStart = now;
            count = 0;
        }

        if (count < maxRequests) {
            count++;
            return true;
        }
        return false;
    }
}
```

**固定窗口的问题**：窗口临界点可能出现突发流量。

```
窗口 1：[0-1000ms) → 1000 请求（已满）
窗口 2：[1000-2000ms) → 1000 请求（已满）

问题：在 1000ms 时，同时涌入 2000 个请求
- 窗口 1 最后时刻：0 个请求
- 窗口 2 开始时刻：2000 个请求
- 实际处理：2000 个请求，是限流上限的 2 倍
```

**算法二：滑动窗口算法**

滑动窗口将时间窗口划分为更细的格子，统计最近 N 个格子的请求数：

```java
// 滑动窗口限流
public class SlidingWindowRateLimiter {
    private final int maxRequests;
    private final int windowSize; // 窗口内格子数
    private final long slotSize;   // 每个格子的大小（毫秒）
    private final AtomicInteger[] slots;
    private final AtomicInteger cursor = new AtomicInteger(0);
    private volatile long lastSlotTime;

    public SlidingWindowRateLimiter(int maxRequests, int windowSize, long slotSize) {
        this.maxRequests = maxRequests;
        this.windowSize = windowSize;
        this.slotSize = slotSize;
        this.slots = new AtomicInteger[windowSize];
        for (int i = 0; i < windowSize; i++) {
            slots[i] = new AtomicInteger(0);
        }
        this.lastSlotTime = System.currentTimeMillis() / slotSize;
    }

    public boolean tryAcquire() {
        long now = System.currentTimeMillis();
        long currentSlot = now / slotSize;

        // 清理过期格子
        if (currentSlot > lastSlotTime) {
            int diff = (int) (currentSlot - lastSlotTime);
            for (int i = 0; i < diff && i < windowSize; i++) {
                int idx = (cursor.get() + i) % windowSize;
                slots[idx].set(0);
            }
            cursor.set((cursor.get() + diff) % windowSize);
            lastSlotTime = currentSlot;
        }

        // 统计窗口内请求数
        int total = 0;
        for (int i = 0; i < windowSize; i++) {
            total += slots[i].get();
        }

        if (total < maxRequests) {
            int idx = (int) (currentSlot % windowSize);
            slots[idx].incrementAndGet();
            return true;
        }
        return false;
    }
}
```

**滑动窗口的优势**：解决了固定窗口的临界突变问题。

**算法三：漏桶算法**

漏桶算法模拟水流进漏桶的过程，以固定速率"漏"出请求：

```java
// 漏桶限流
public class LeakyBucketRateLimiter {
    private final long capacity;        // 桶容量
    private final long leakRate;        // 漏出速率（每秒）
    private double water;              // 当前水量
    private long lastLeakTime;         // 上次漏水时间

    public LeakyBucketRateLimiter(long capacity, long leakRate) {
        this.capacity = capacity;
        this.leakRate = leakRate;
        this.water = 0;
        this.lastLeakTime = System.currentTimeMillis();
    }

    public synchronized boolean tryAcquire() {
        leak(); // 先漏水

        if (water < capacity) {
            water++; // 请求入桶
            return true;
        }
        return false; // 桶已满，拒绝
    }

    private void leak() {
        long now = System.currentTimeMillis();
        long elapsed = now - lastLeakTime;
        double leaked = (elapsed / 1000.0) * leakRate;

        water = Math.max(0, water - leaked);
        lastLeakTime = now;
    }
}
```

**漏桶的特点**：

- 请求以任意速率进入
- 请求以固定速率离开
- 无论流量多突发，输出速率恒定
- 桶满时拒绝请求

**算法四：令牌桶算法**

令牌桶以固定速率生成令牌，请求需要获取令牌才能通过：

```java
// 令牌桶限流（Guava RateLimiter 简化版）
public class TokenBucketRateLimiter {
    private final double permitsPerSecond; // 每秒生成的令牌数
    private final double maxPermits;       // 最大令牌数
    private double storedPermits;          // 当前存储的令牌数
    private long lastRefillTime;           // 上次补充时间

    public TokenBucketRateLimiter(double permitsPerSecond, double maxPermits) {
        this.permitsPerSecond = permitsPerSecond;
        this.maxPermits = maxPermits;
        this.storedPermits = maxPermits; // 初始化为满
        this.lastRefillTime = System.currentTimeMillis();
    }

    public synchronized boolean tryAcquire() {
        refill(); // 补充令牌

        if (storedPermits >= 1) {
            storedPermits -= 1;
            return true;
        }
        return false;
    }

    private void refill() {
        long now = System.currentTimeMillis();
        double elapsed = (now - lastRefillTime) / 1000.0;
        double newPermits = elapsed * permitsPerSecond;

        storedPermits = Math.min(maxPermits, storedPermits + newPermits);
        lastRefillTime = now;
    }
}
```

**令牌桶 vs 漏桶**：

| 对比项 | 令牌桶 | 漏桶 |
|---|---|---|
| 输出速率 | 可突发（令牌积累） | 恒定 |
| 适用场景 | 需要应对突发流量 | 需要平滑输出 |
| 典型应用 | API 限流 | 日志限流 |

### 2.3 分布式限流：Redis + Lua

单机限流只能保护单机，在分布式环境下需要分布式限流。

```lua
-- Redis 令牌桶限流（Lua 脚本）
-- KEYS[1] = 限流 key
-- ARGV[1] = 令牌桶容量
-- ARGV[2] = 每秒补充令牌数
-- ARGV[3] = 当前时间戳（毫秒）
-- ARGV[4] = 请求的令牌数

local key = KEYS[1]
local capacity = tonumber(ARGV[1])
local refillRate = tonumber(ARGV[2])
local now = tonumber(ARGV[3])
local requested = tonumber(ARGV[4])

-- 获取当前状态
local data = redis.call('HMGET', key, 'tokens', 'lastRefillTime')
local tokens = tonumber(data[1]) or capacity
local lastRefillTime = tonumber(data[2]) or now

-- 计算应该补充的令牌数
local elapsed = now - lastRefillTime
local delta = elapsed * refillRate / 1000
tokens = math.min(capacity, tokens + delta)

-- 尝试获取令牌
if tokens >= requested then
    tokens = tokens - requested
    redis.call('HMSET', key, 'tokens', tokens, 'lastRefillTime', now)
    redis.call('EXPIRE', key, 60)
    return 1  -- 允许通过
else
    redis.call('HMSET', key, 'tokens', tokens, 'lastRefillTime', now)
    redis.call('EXPIRE', key, 60)
    return 0  -- 拒绝
end
```

## 三、熔断：系统的"保险丝"

### 3.1 熔断器模式：快速失败的艺术

熔断器模式借鉴了电路保险丝的思想：**当某个服务故障率达到阈值时，"熔断"该服务的调用，直接返回失败，避免资源耗尽。**

```java
// 熔断器核心逻辑
public class CircuitBreaker {
    private final String name;
    private final int failureThreshold;  // 失败阈值
    private final long recoveryTimeout;   // 恢复超时

    private State state = State.CLOSED;   // 默认闭合（正常）
    private int failureCount = 0;
    private long lastFailureTime = 0;

    public enum State {
        CLOSED,    // 闭合（正常）：请求通过，失败累计
        OPEN,      // 打开（熔断）：请求拒绝，快速失败
        HALF_OPEN  // 半开：尝试放行一个请求，测试是否恢复
    }

    public boolean allowRequest() {
        switch (state) {
            case CLOSED:
                return true;

            case OPEN:
                // 检查是否超过恢复超时
                if (System.currentTimeMillis() - lastFailureTime > recoveryTimeout) {
                    state = State.HALF_OPEN;
                    return true;  // 放行一个请求测试
                }
                return false;  // 继续熔断

            case HALF_OPEN:
                return true;  // 半开状态下放行

            default:
                return false;
        }
    }

    public void recordSuccess() {
        failureCount = 0;
        state = State.CLOSED;
    }

    public void recordFailure() {
        failureCount++;
        lastFailureTime = System.currentTimeMillis();

        if (failureCount >= failureThreshold) {
            state = State.OPEN;  // 触发熔断
        }
    }
}
```

**熔断器的三种状态**：

```
        ┌─────────────────────────────────────────────────┐
        │                                                  │
        │  CLOSED ──→ (失败次数 ≥ 阈值) ──→ OPEN         │
        │    ↑                                       │     │
        │    │                                       │     │
        │    │ (恢复超时)                          (超时到期) │
        │    │                                       │     │
        │    │           ┌───────────────────────────┘     │
        │    │           │                                 │
        │    │           ▼                                 │
        │    │      ┌─────────────┐                        │
        │    │      │  HALF_OPEN  │                        │
        │    │      └─────────────┘                        │
        │    │           │                                 │
        │    │           ├── 成功 ──→ CLOSED              │
        │    │           │                                 │
        │    │           └── 失败 ──→ OPEN                 │
        │    │                                                 │
        └────┘                                                 │
        │                                                       │
        │   CLOSED：正常，请求通过                              │
        │   OPEN：熔断，请求拒绝                                │
        │   HALF_OPEN：尝试恢复，放行一个请求                   │
        │                                                       │
        └───────────────────────────────────────────────────────┘
```

## 四、负载均衡：流量的"调度员"

### 4.1 负载均衡策略

负载均衡是分布式系统的核心组件，负责将请求合理地分配到多个服务实例。

**策略一：轮询（Round Robin）**

每个请求依次分配给每个服务器，循环往复。

```java
public class RoundRobinLoadBalancer {
    private final List<String> servers;
    private final AtomicInteger index = new AtomicInteger(0);

    public RoundRobinLoadBalancer(List<String> servers) {
        this.servers = servers;
    }

    public String select() {
        int pos = Math.abs(index.getAndIncrement() % servers.size());
        return servers.get(pos);
    }
}
```

**策略二：加权轮询（Weighted Round Robin）**

性能好的服务器分配更多请求。

```java
public class WeightedRoundRobinLoadBalancer {
    private final List<Server> servers;
    private final AtomicInteger current = new AtomicInteger(0);

    public String select() {
        // 计算总权重
        int totalWeight = servers.stream()
            .mapToInt(Server::getWeight).sum();

        // 模拟加权轮询
        int pos = Math.abs(current.getAndIncrement() % totalWeight);

        for (Server server : servers) {
            pos -= server.getWeight();
            if (pos < 0) {
                return server.getUrl();
            }
        }

        return servers.get(0).getUrl();
    }
}
```

**策略三：随机（Random）**

```java
public class RandomLoadBalancer {
    private final List<String> servers;
    private final Random random = new Random();

    public String select() {
        return servers.get(random.nextInt(servers.size()));
    }
}
```

**策略四：一致性哈希（Consistent Hash）**

相同请求特征的请求始终路由到同一个服务器。

```java
public class ConsistentHashLoadBalancer {
    private final TreeMap<Long, String> ring = new TreeMap<>();
    private final int virtualNodes;

    public ConsistentHashLoadBalancer(List<String> servers, int virtualNodes) {
        this.virtualNodes = virtualNodes;
        for (String server : servers) {
            addServer(server);
        }
    }

    public void addServer(String server) {
        for (int i = 0; i < virtualNodes; i++) {
            long hash = hash(server + "#" + i);
            ring.put(hash, server);
        }
    }

    public String select(String key) {
        if (ring.isEmpty()) {
            return null;
        }
        long hash = hash(key);
        Map.Entry<Long, String> entry = ring.ceilingEntry(hash);
        if (entry == null) {
            entry = ring.firstEntry();
        }
        return entry.getValue();
    }

    private long hash(String key) {
        return Math.abs(key.hashCode());
    }
}
```

### 4.2 负载均衡对比

| 策略 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| 轮询 | 简单、均匀 | 无法感知服务器状态 | 服务器性能一致 |
| 加权轮询 | 可根据性能分配 | 需要手动配置权重 | 性能不均的场景 |
| 随机 | 简单 | 不可预测 | 对均匀性要求不高 |
| 一致性哈希 | 相同请求路由到同一节点 | 数据分布可能不均 | 缓存场景、分布式 Session |
| 最少连接 | 负载更均衡 | 需要维护连接数 | 长连接场景 |

### 4.3 客户端负载均衡 vs 服务端负载均衡

```
客户端负载均衡：
┌─────────────────────────────────────────────────────┐
│                                                      │
│   Client ──→ LoadBalancer（客户端）                  │
│                  │                                    │
│                  ├──→ Server A                       │
│                  ├──→ Server B                       │
│                  └──→ Server C                       │
│                                                      │
│   Ribbon、Spring Cloud LoadBalancer                   │
│                                                      │
└─────────────────────────────────────────────────────┘

服务端负载均衡：
┌─────────────────────────────────────────────────────┐
│                                                      │
│   Client ──→ Nginx ──→ LoadBalancer（服务端）        │
│                         │                            │
│                         ├──→ Server A               │

│                                                      │

│                                                      │
└─────────────────────────────────────────────────────┘
```

| 对比项 | 客户端 LB | 服务端 LB |
|---|---|---|
| 复杂度 | 高（客户端需要感知所有服务器） | 低（客户端只需感知 LB） |
| 灵活性 | 高（可自定义策略） | 低 |
| 网络跳数 | 少一跳 | 多一跳 |
| 典型实现 | Ribbon、Nginx | F5、LVS |

```java
// 健康检查示例
@Service
public class HealthCheckService {

    @Scheduled(fixedDelay = 10000)  // 每 10 秒检查一次

}
```

### 5.3 故障剔除与权重动态调整

```java
// 故障剔除
public class FailureDetector {

    private final Map<String, FailureInfo> failures = new ConcurrentHashMap<>();

    public void recordFailure(String server) {
        FailureInfo info = failures.computeIfAbsent(server, k -> new FailureInfo());
        info.incrementFailureCount();

        // 连续失败次数超过阈值，标记为不可用
        if (info.getFailureCount() > MAX_FAILURE_COUNT) {
            markUnhealthy(server);
        }
    }

    public void recordSuccess(String server) {
        FailureInfo info = failures.get(server);
        if (info != null) {
            info.resetFailureCount();
            if (!info.isHealthy()) {
                markHealthy(server);
            }
        }
    }

    // 权重动态调整：根据健康状态调整权重
    public void adjustWeight(String server, boolean healthy) {
        if (healthy) {
            server.increaseWeight();  // 逐步恢复
        } else {
            server.decreaseWeight();  // 逐步降权
        }
    }
}
```

```java
// 第一级：网关限流（全局）
@Bean
public FilterRegistrationBean<RateLimitFilter> rateLimitFilter() {
    FilterRegistrationBean<RateLimitFilter> bean = new FilterRegistrationBean<>();
    bean.setFilter(new RateLimitFilter(10000));  // 整体 10000 QPS

// 第二级：服务限流（单机）
@Aspect
@Component
public class RateLimitAspect {

    @Autowired
    private RateLimiter rateLimiter;

    @Around("@annotation(RateLimited)")
    public Object rateLimit(ProceedingJoinPoint joinPoint) throws Throwable {
        if (!rateLimiter.tryAcquire()) {
            throw new RateLimitException("请求过于频繁，请稍后重试");
        }
        return joinPoint.proceed();
    }
}
```
