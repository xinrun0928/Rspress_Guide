# 分布式并发算法（Paxos、Raft、一致性哈希等）

1989 年，Leslie Lamport 发表了一篇论文，题目叫「The Part-Time Parliament」。

论文讲的是一个关于古希腊 Paxos 小岛上议会如何通过法令的故事。Lamport 用 Paxos 比喻分布式系统中多个节点如何就一个值达成共识。

编辑部拒绝了论文，理由是「太难理解」。Lamport 坚持说「这个比喻让 Paxos 算法变得更容易理解」。

编辑部的回应是：「我们收到过很多用数学方法描述算法的论文，这篇用故事描述的论文让我们很为难。要不你用数学语言重新写一遍？」

Lamport 后来又写了一版，1998 年终于发表。

**这就是 Paxos 算法——一个被埋没了近 10 年的分布式共识协议，直到 Google 在 Chubby 中使用它，才被业界广泛认可。**

分布式系统的核心问题是什么？**如何在不确定的环境中，让多个节点就某个值达成一致。**

这个问题看起来简单，但 FLP 定理告诉我们：在纯异步系统中，如果存在节点可能故障，没有任何算法能保证总是达成共识。Paxos 的贡献是：它提供了一个在大多数情况下能工作的算法。

## Paxos 算法：分布式共识的基石

### Paxos 要解决什么问题？

```
场景：5 个节点的分布式系统

节点 1 提出：值 = A
节点 3 提出：值 = B
节点 5 提出：值 = C

最终结果：所有节点必须接受同一个值
```

这个问题看似简单，但 FLP 定理告诉我们：在存在节点故障和消息延迟的异步系统中，没有任何算法能保证 100% 达成共识。Paxos 的贡献在于：它提供了一个在大多数情况下能工作的算法。

### Paxos 的三个角色

Paxos 算法中有三个核心角色：

| 角色 | 英文 | 职责 |
|------|------|------|
| **提议者** | Proposer | 提出值（value），发起共识流程 |
| **接受者** | Acceptor | 投票决定接受哪个值 |
| **学习者** | Learner | 学习最终被选定的值 |

```
实际系统中的角色映射：

ZooKeeper：
- 每个节点同时是 Proposer、Acceptor、Learner
- 通过 ZAB 协议协调

etcd：
- 每个节点同时是 Proposer、Acceptor、Learner
- 通过 Raft 协议协调

Multi-Paxos 变体（如 Chubby）：
- Leader 节点是唯一的 Proposer
- 其他节点是 Acceptor
- 所有节点都是 Learner
```

### Paxos 的两阶段执行

Paxos 算法的核心是两阶段提交（Two-Phase Commit）：

```java
/**
 * Basic Paxos 核心流程
 * 
 * 这是一个概念性实现，用于理解算法原理
 * 实际生产环境请使用成熟的库，如 Apache Curator
 */
public class BasicPaxos {

    private long proposalNumber = 0;
    private final long nodeId;
    
    // Acceptor 状态
    private long acceptedNumber = -1;
    private String acceptedValue = null;
    
    public BasicPaxos(long nodeId) {
        this.nodeId = nodeId;
    }
    
    /**
     * Proposer：提议一个值
     * 
     * Phase 1：Prepare
     * - 选择一个提案号 N
     * - 向所有 Acceptor 发送 Prepare(N)
     * 
     * Phase 2：Accept
     * - 收到多数派 Promise 后，选择一个值（通常是已接受的值）
     * - 向所有 Acceptor 发送 Accept(N, value)
     */
    public String propose(String value, List<BasicPaxos> acceptors) throws Exception {
        
        // ============ Phase 1: Prepare ============
        proposalNumber++;  // 提案号必须单调递增
        
        // 向所有 Acceptor 发送 Prepare 请求
        List<Promise> promises = new ArrayList<>();
        for (BasicPaxos acceptor : acceptors) {
            Promise promise = acceptor.receivePrepare(proposalNumber);
            if (promise != null) {
                promises.add(promise);
            }
        }
        
        // 如果没有收到多数派 Promise，提议失败（可能存在活锁）
        int majority = acceptors.size() / 2 + 1;
        if (promises.size() < majority) {
            return null;  // 重新尝试
        }
        
        // ============ Phase 2: Accept ============
        // 找出 Promise 中已接受的最大值
        // 如果有值被接受过，使用已接受的值（保证安全性）
        String valueToPropose = value;
        long highestAcceptedNumber = -1;
        
        for (Promise promise : promises) {
            if (promise.hasAcceptedValue() && 
                promise.getAcceptedNumber() > highestAcceptedNumber) {
                valueToPropose = promise.getAcceptedValue();
                highestAcceptedNumber = promise.getAcceptedNumber();
            }
        }
        
        // 向所有 Acceptor 发送 Accept 请求
        List<Accepted> acceptedList = new ArrayList<>();
        for (BasicPaxos acceptor : acceptors) {
            Accepted accepted = acceptor.receiveAccept(proposalNumber, valueToPropose);
            if (accepted != null) {
                acceptedList.add(accepted);
            }
        }
        
        // 如果没有收到多数派 Accepted，提议失败
        if (acceptedList.size() < majority) {
            return null;
        }
        
        return valueToPropose;
    }
    
    /**
     * Acceptor：处理 Prepare 请求
     * 
     * 关键规则：
     * 1. 只承诺大于当前提案号的提案
     * 2. 返回已接受的最高提案号的值
     */
    public Promise receivePrepare(long n) {
        synchronized (this) {
            if (n > proposalNumber) {
                proposalNumber = n;
                // 返回已接受的值，以便 Proposer 选择
                return new Promise(acceptedNumber, acceptedValue);
            }
            return null;  // 拒绝较小提案号的 Prepare
        }
    }
    
    /**
     * Acceptor：处理 Accept 请求
     */
    public Accepted receiveAccept(long n, String value) {
        synchronized (this) {
            // 只有当 n >= 承诺的提案号时，才接受
            if (n >= proposalNumber) {
                acceptedNumber = n;
                acceptedValue = value;
                return new Accepted(true);
            }
            return null;  // 拒绝
        }
    }
    
    // 辅助类省略...
}
```

### 为什么需要多数派（Quorum）？

```
问题：为什么 Paxos 要求多数派，而不是全部节点？

场景：5 节点系统

方案 A：全部同意（5/5）
- 优点：所有人都知道结果
- 缺点：任何一节点故障，系统就不可用
- 可用性：4/5 = 80%

方案 B：多数派同意（3/5）
- 优点：允许 2 个节点故障
- 可用性：5/5 = 100%（只要能凑齐3个节点就能工作）

为什么 Paxos 选择多数派？

Paxos 的核心洞察是「两个多数派必然有交集」：

假设：
- 提案 A 被多数派接受（S1 ∩ S2 = 3 个节点）
- 提案 B 要被多数派接受

证明：
- 如果提案 A 被多数派接受（至少 3 个节点）
- 提案 B 要被多数派接受（也至少 3 个节点）
- 这两个多数派集合至少有一个共同的节点

这个交集保证了：
- 如果提案 A 被选中（被多数派接受）
- 后续任何能被多数派接受的提案，必须包含 A 的值
- 因为提案 B 必须和 A 的接受集合有交集
- 而这个交集节点已经接受了 A
- 所以提案 B 必须使用 A 的值
```

### Multi-Paxos：连续值共识

Basic Paxos 只能就单个值达成共识。但实际系统需要就一系列操作（如日志条目）达成共识。Multi-Paxos 是对 Basic Paxos 的扩展。

```java
/**
 * Multi-Paxos：使用 Leader 优化
 * 
 * 核心思想：选出一个 Leader，Leader 连续提议时可以跳过 Prepare 阶段
 * 
 * 优化效果：
 * - Basic Paxos：每个提案需要 2 轮 RPC
 * - Multi-Paxos with Leader：只需 1 轮 RPC（跳过 Prepare）
 */
public class MultiPaxos {
    
    private final long nodeId;
    private volatile boolean isLeader = false;
    private long currentTerm = 0;
    
    // 日志条目
    private final List<LogEntry> log = new ArrayList<>();
    private long commitIndex = -1;
    
    /**
     * Leader 连续提议：跳过 Prepare，直接 Accept
     * 
     * 只有在以下情况需要重新 Prepare：
     * 1. 选举新 Leader 时
     * 2. 检测到提案冲突时
     * 3. Leader 失联后重新连接
     */
    public boolean accept(LogEntry entry, List<MultiPaxos> acceptors) {
        if (!isLeader) {
            throw new IllegalStateException("我不是 Leader");
        }
        
        // 添加到本地日志
        entry.setIndex(log.size());
        entry.setTerm(currentTerm);
        log.add(entry);
        
        // 直接发送 Accept（跳过 Prepare 阶段）
        int acceptedCount = 1;  // 自身算一个
        for (MultiPaxos acceptor : acceptors) {
            if (acceptor != this && acceptor.receiveAccept(entry)) {
                acceptedCount++;
            }
        }
        
        // 多数派确认后提交
        if (acceptedCount > acceptors.size() / 2) {
            commitIndex = entry.getIndex();
            return true;
        }
        
        return false;
    }
    
    /**
     * Leader 选举：使用 Basic Paxos
     */
    public void leaderElection(List<MultiPaxos> allNodes) {
        BasicPaxos paxos = new BasicPaxos(nodeId);
        
        try {
            String leaderValue = paxos.propose("leader-" + nodeId, allNodes);
            if (leaderValue != null && leaderValue.equals("leader-" + nodeId)) {
                isLeader = true;
                currentTerm++;
            }
        } catch (Exception e) {
            // 选举失败，稍后重试
        }
    }
}
```

## Raft 算法：更易理解的共识协议

### Raft 的设计目标

2014 年，Diego Ongaro 和 John Ousterhout 发表了 Raft 算法。他们的目标很简单：**Paxos 太难理解了，能不能设计一个更容易理解的算法？**

```
Raft 的设计目标（按优先级排序）：

1. 安全性：在非拜占庭条件下，保证正确性
2. 可用性：在多数派节点存活时，系统正常运行
3. 易理解性：易于理解，方便工程实现
4. 高效性：正常的读写操作尽可能高效

Raft 的核心策略：分解问题

Paxos 把所有问题混在一起：
- Leader 选举 + 日志复制 + 安全性

Raft 把问题分解成独立的子问题：
1. Leader 选举（如何选出唯一的 Leader）
2. 日志复制（如何保证日志一致）
3. 安全性（如何保证正确性）
```

### Leader 选举：Term 机制

Raft 使用 **Term（任期）** 来标识逻辑时钟，每个 Term 以一次选举开始。

```java
/**
 * Raft Leader 选举机制
 */
public class RaftNode {
    
    // 持久化状态（必须持久化）
    private long currentTerm = 0;      // 当前任期号
    private String votedFor = null;    // 当前任期内投票给了谁
    private List<LogEntry> log = new ArrayList<>();  // 日志条目
    
    // 易失状态
    private NodeState state = NodeState.FOLLOWER;
    private String leaderId = null;
    private long commitIndex = -1;
    private long lastApplied = -1;
    
    // Leader 选举相关配置
    private static final int ELECTION_TIMEOUT_MIN = 150;  // ms
    private static final int ELECTION_TIMEOUT_MAX = 300;  // ms
    
    /**
     * 选举超时：Follower 在这段时间内没有收到心跳，就发起选举
     * 
     * 随机化超时：避免所有 Follower 同时发起选举导致票数分散
     */
    private int getElectionTimeout() {
        return ELECTION_TIMEOUT_MIN + 
               new Random().nextInt(ELECTION_TIMEOUT_MAX - ELECTION_TIMEOUT_MIN);
    }
    
    /**
     * 开始选举
     */
    public void startElection() {
        state = NodeState.CANDIDATE;
        currentTerm++;  // Term 递增
        votedFor = this.nodeId;  // 给自己投票
        
        // 向所有节点发送 RequestVote 请求
        for (String peerId : peers) {
            sendRequestVote(peerId);
        }
    }
    
    /**
     * 处理 RequestVote 请求
     */
    public RequestVoteResponse handleRequestVote(RequestVoteRequest request) {
        RequestVoteResponse response = new RequestVoteResponse();
        response.setTerm(currentTerm);
        
        // 规则 1：Term 比自己小，拒绝
        if (request.getTerm() < currentTerm) {
            response.setVoteGranted(false);
            return response;
        }
        
        // 规则 2：更新 Term，转为 Follower
        if (request.getTerm() > currentTerm) {
            currentTerm = request.getTerm();
            state = NodeState.FOLLOWER;
            votedFor = null;
        }
        
        // 规则 3：判断是否投票
        // 条件 1：还没投过票，或者投给了候选者
        // 条件 2：候选者的日志至少和自己一样新
        boolean canVote = (votedFor == null || votedFor.equals(request.getCandidateId()));
        boolean logIsAtLeastAsUpToDate = isLogAtLeastAsUpToDate(request.getLastLogTerm(), 
                                                                 request.getLastLogIndex());
        
        if (canVote && logIsAtLeastAsUpToDate) {
            votedFor = request.getCandidateId();
            response.setVoteGranted(true);
        }
        
        return response;
    }
    
    /**
     * 判断候选者的日志是否至少和本节点一样新
     * 
     * 这是 Raft 保证安全性的关键：
     * 只有日志「至少一样新」的节点才能当选 Leader
     */
    private boolean isLogAtLeastAsUpToDate(long candidateLastTerm, long candidateLastIndex) {
        LogEntry lastEntry = log.get(log.size() - 1);
        
        // 如果 Term 不同，Term 大的日志更新
        if (candidateLastTerm != lastEntry.getTerm()) {
            return candidateLastTerm > lastEntry.getTerm();
        }
        
        // Term 相同，索引大的日志更新
        return candidateLastIndex >= log.size() - 1;
    }
}
```

### Raft 的心跳机制

Raft 的 Leader 通过发送心跳（AppendEntries）来维持权威。Follower 如果在选举超时内没收到心跳，就认为 Leader 挂了，发起选举。

```java
/**
 * Leader：发送心跳维持权威
 */
public class RaftLeader {
    
    private final RaftNode node;
    private final ScheduledExecutorService heartbeatScheduler;
    
    public RaftLeader(RaftNode node) {
        this.node = node;
        
        // Leader 定期发送心跳
        this.heartbeatScheduler = Executors.newSingleThreadScheduledExecutor();
        heartbeatScheduler.scheduleAtFixedRate(() -> {
            sendHeartbeat();
        }, 0, HEARTBEAT_INTERVAL, TimeUnit.MILLISECONDS);  // 通常 100-300ms
    }
    
    /**
     * 发送心跳（空的 AppendEntries）
     * 
     * 关键点：心跳不包含任何日志条目
     * 但它携带了 Leader 的当前 Term
     * Follower 收到后可以确认 Leader 还活着
     */
    private void sendHeartbeat() {
        for (String peerId : node.getPeers()) {
            AppendEntriesRequest request = new AppendEntriesRequest();
            request.setTerm(node.getCurrentTerm());
            request.setLeaderId(node.getNodeId());
            request.setPrevLogIndex(-1);  // 无日志条目
            request.setPrevLogTerm(-1);
            request.setEntries(Collections.emptyList());  // 空的心跳
            request.setLeaderCommit(node.getCommitIndex());
            
            // 异步发送，不等待响应
            node.sendAppendEntries(peerId, request);
        }
    }
}
```

### 日志复制：AppendEntries 机制

Raft 的核心是日志复制。所有写请求都必须经过 Leader，Leader 将操作追加到本地日志，然后复制到多数派节点。

```java
/**
 * Raft 日志复制机制
 */
public class RaftLogReplication {
    
    /**
     * Leader：接收客户端请求，追加日志
     */
    public LogEntry appendLog(String command) {
        // 1. 创建日志条目
        LogEntry entry = new LogEntry();
        entry.setTerm(currentTerm);
        entry.setCommand(command);
        entry.setIndex(log.size());  // 日志索引
        
        // 2. 追加到本地日志
        log.add(entry);
        
        // 3. 异步复制到所有 Follower
        replicateToFollowers(entry);
        
        return entry;
    }
    
    /**
     * 异步复制到所有 Follower
     */
    private void replicateToFollowers(LogEntry entry) {
        for (String peerId : peers) {
            // 并行复制，不等待
            CompletableFuture.runAsync(() -> {
                
                // 构建 AppendEntries 请求
                AppendEntriesRequest request = buildAppendEntriesRequest(entry, peerId);
                
                // 发送请求
                AppendEntriesResponse response = sendAppendEntries(peerId, request);
                
                // 处理响应
                if (response.isSuccess()) {
                    // 复制成功，更新 nextIndex 和 matchIndex
                    nextIndex.put(peerId, entry.getIndex() + 1);
                    matchIndex.put(peerId, entry.getIndex());
                    
                    // 检查是否可以提交
                    checkCommit();
                } else {
                    // 复制失败，可能是日志不一致
                    // Leader 会回退 nextIndex 重试
                    nextIndex.decrement(peerId);
                    retryAppendEntries(peerId);
                }
            });
        }
    }
    
    /**
     * Leader 检查是否可以提交日志
     * 
     * Raft 的提交规则：
     * 1. 日志条目必须复制到多数派节点
     * 2. 只能提交当前 Term 的日志
     * 3. 不能直接提交之前 Term 的日志（通过间接方式）
     */
    private void checkCommit() {
        // 统计已复制的节点数（包括 Leader 自己）
        for (int i = commitIndex + 1; i <= log.size(); i++) {
            int count = 1;  // Leader 自己
            for (long matchIndex : matchIndex.values()) {
                if (matchIndex >= i) {
                    count++;
                }
            }
            
            // 如果多数派节点都有这个日志，可以提交
            if (count > (peers.size() + 1) / 2) {
                // 只能提交当前 Term 的日志
                if (log.get(i).getTerm() == currentTerm) {
                    commitIndex = i;
                    applyToStateMachine();
                }
            }
        }
    }
}
```

### Raft vs Paxos：核心区别

| 维度 | Raft | Paxos |
|------|------|-------|
| **强 Leader** | 必须有 Leader | 可以没有 Leader |
| **领导任期** | 明确的 Term 编号 | 提案号递增 |
| **日志追加方向** | Leader → Follower 单向 | 双向都可以 |
| **成员变更 | 联合共识（Joint Consensus）或单节点变更 | 需要额外设计 |
| **可理解性** | 高（分解为独立子问题） | 低（概念抽象） |
| **实际应用** | etcd, TiKV, CockroachDB | Chubby, Spanner |

```
为什么 Raft 要强依赖 Leader？

Paxos 的设计允许任何节点提出提案，这导致：
1. 多个 Proposer 可能相互干扰（活锁）
2. 提案号需要全局协调
3. 实现复杂度高

Raft 的设计选择：
1. 所有写请求必须经过 Leader
2. 读请求可以分散到任何节点（ leases）
3. 简化了问题，提高了效率

代价：
1. Leader 是性能瓶颈（但可以通过 Raedonly Node 缓解）
2. Leader 故障时系统不可写（有短暂的选举期）

但这个代价是值得的：
- Raft 的正确性更容易证明
- Raft 的实现更容易正确
```

### Raft 的成员变更

成员变更（Adding or Removing nodes）是 Raft 中最复杂的部分。一个 naive 的做法是直接切换配置，但可能导致脑裂。

```java
/**
 * Raft 成员变更：单节点变更
 * 
 * 核心思想：每次只变更一个节点
 * 这样可以保证新旧配置之间有多数派交集
 */
public class RaftMembershipChange {
    
    /**
     * 添加一个节点
     */
    public void addNode(String newNode) {
        // 创建一个配置变更日志条目
        ConfigurationEntry entry = new ConfigurationEntry();
        entry.setNewConfiguration(new HashSet<>(currentConfiguration));
        entry.getNewConfiguration().add(newNode);
        
        // 追加到日志，复制并提交
        // 由于只变更一个节点，新旧配置有重叠多数派
        log.append(entry);
        replicateAndCommit(entry);
    }
    
    /**
     * 移除一个节点
     */
    public void removeNode(String removedNode) {
        ConfigurationEntry entry = new ConfigurationEntry();
        entry.setNewConfiguration(new HashSet<>(currentConfiguration));
        entry.getNewConfiguration().remove(removedNode);
        
        log.append(entry);
        replicateAndCommit(entry);
        
        // 注意：被移除的节点可能不知道自己是 Leader
        // 需要在下一次心跳时检测到 Term 变化
    }
}
```

## 一致性哈希：分布式系统的负载均衡利器

### 传统哈希的问题

传统的哈希取模（`hash(key) % N`）在分布式缓存和负载均衡中很常见。但它有一个致命问题：**当节点数量变化时，几乎所有数据的映射关系都会改变。**

```
传统哈希的问题：

场景：3 个节点的缓存集群

数据分布（hash(key) % 3）：
- key1 → 节点0
- key2 → 节点1
- key3 → 节点2
- key4 → 节点0
- key5 → 节点1

现在增加一个节点（节点3）：

数据分布（hash(key) % 4）：
- key1 → 节点1（之前是节点0）❌
- key2 → 节点2（之前是节点1）❌
- key3 → 节点3（之前是节点2）❌
- key4 → 节点0（之前是节点0）✓
- key5 → 节点1（之前是节点1）✓

结果：80% 的数据需要重新映射（cache miss）

在缓存场景下，这意味着：
- 大量的缓存未命中
- 数据库压力骤增
- 可能导致系统雪崩
```

### 一致性哈希的原理

一致性哈希（Consistent Hashing）由 MIT 的 Karger 等人在 1997 年提出。它的核心思想是：**将整个哈希空间映射到一个环上，节点和数据都分布在环上。**

```
一致性哈希环：

                     0
                      ↑
                      │
          ┌───────────┼───────────┐
          │           │           │
          │           │           │
          │     [keyC] │           │
          │           │           │
          │           │           │
          │           │     [NodeA]───→ 90°
          │           │           │
          │           │           │
          │    [NodeB]│           │
          │           │           │
          │           │           │
          └───────────┴───────────┘
                      │
                      ↓
                   360°/0°

数据定位规则：
- keyC 在环上顺时针找到的第一个节点是 NodeA
- 任何节点负责一段弧上的所有数据
```

```java
/**
 * 一致性哈希实现
 */
public class ConsistentHash<T> {
    
    /**
     * 虚拟节点数量
     * 虚拟节点越多，负载分布越均匀
     */
    private static final int VIRTUAL_NODES = 150;
    
    /**
     * 哈希环：哈希值 → 节点
     */
    private final TreeMap<Long, T> ring = new TreeMap<>();
    
    /**
     * 物理节点到虚拟节点的映射
     */
    private final Map<T, Set<Long>> virtualNodeMap = new HashMap<>();
    
    /**
     * 添加节点
     */
    public void addNode(T node) {
        // 创建虚拟节点
        Set<Long> virtualNodes = new HashSet<>();
        for (int i = 0; i < VIRTUAL_NODES; i++) {
            // 虚拟节点的 key = "nodeName#VN" 的哈希值
            String virtualNodeKey = node.toString() + "#VN" + i;
            long hash = hash(virtualNodeKey);
            
            ring.put(hash, node);
            virtualNodes.add(hash);
        }
        virtualNodeMap.put(node, virtualNodes);
    }
    
    /**
     * 移除节点
     */
    public void removeNode(T node) {
        Set<Long> virtualNodes = virtualNodeMap.get(node);
        if (virtualNodes != null) {
            for (Long hash : virtualNodes) {
                ring.remove(hash);
            }
            virtualNodeMap.remove(node);
        }
    }
    
    /**
     * 根据 key 找到对应的节点
     * 
     * 核心逻辑：顺时针方向找到第一个虚拟节点
     */
    public T getNode(String key) {
        if (ring.isEmpty()) {
            return null;
        }
        
        long hash = hash(key);
        
        // 找到第一个 >= hash 的虚拟节点
        Map.Entry<Long, T> entry = ring.ceilingEntry(hash);
        
        // 如果 hash 比环上最大的还大，环绕到最小的节点
        if (entry == null) {
            entry = ring.firstEntry();
        }
        
        return entry.getValue();
    }
    
    /**
     * 哈希函数：使用 MurmurHash 或 KetamaHash
     * 
     * 要求：
     * 1. 均匀分布
     * 2. 高效计算
     * 3. 确定性（相同输入总是得到相同输出）
     */
    private long hash(String key) {
        // 简单实现，实际使用 MurmurHash 或 KetamaHash
        return Math.abs(key.hashCode());
    }
}
```

### 虚拟节点：解决数据倾斜

一致性哈希存在一个问题：**如果节点数量较少，数据可能分布不均匀。**

```
数据倾斜问题：

场景：3 个节点

理想情况：
- 节点A: 33.3%
- 节点B: 33.3%
- 节点C: 33.3%

实际情况（取决于哈希函数）：
- 节点A: 45%
- 节点B: 25%
- 节点C: 30%

甚至更极端的情况：
- 节点A: 60%
- 节点B: 20%
- 节点C: 20%

这会导致：
- 某些节点负载过重
- 某些节点资源浪费
- 系统整体性能下降
```

解决方案是**虚拟节点（Virtual Nodes）**：每个物理节点在环上有多个虚拟节点，虚拟节点负责一部分数据。

```
虚拟节点的工作原理：

物理节点：NodeA, NodeB, NodeC
虚拟节点数：150（每个物理节点）

环上分布：
- NodeA#VN0 (hash=1000)
- NodeA#VN1 (hash=3500)
- NodeA#VN2 (hash=5800)
- ...
- NodeB#VN0 (hash=2000)
- NodeB#VN1 (hash=4200)
- ...
- NodeC#VN0 (hash=1500)
- ...

查找 key 时：
1. 计算 key 的 hash = 4000
2. 在环上顺时针找到第一个节点：NodeB#VN1
3. 返回物理节点：NodeB

虚拟节点的效果：
- 由于有 150 个虚拟节点分布在环上
- 根据大数定律，负载分布接近均匀
- 节点 A 的 150 个虚拟节点可能分布在环的各个位置
```

### 一致性哈希的应用场景

| 场景 | 说明 | 代表系统 |
|------|------|----------|
| 分布式缓存 | Memcached、Redis Cluster | DynamoDB、Cassandra |
| 负载均衡 | Nginx、OpenResty | Consul |
| CDN | 内容路由 | Akamai、Cloudflare |
| 消息队列 | 分区分配 | Kafka（使用改进版） |

```java
/**
 * 一致性哈希 + 虚拟节点的完整示例
 */
public class DistributedCache<T> {
    
    private final ConsistentHash<T> consistentHash;
    private final Map<T, CacheNode<T>> cacheNodes;
    
    public DistributedCache() {
        this.consistentHash = new ConsistentHash<>();
        this.cacheNodes = new HashMap<>();
    }
    
    /**
     * 添加缓存节点
     */
    public void addCacheNode(String nodeId, CacheNode<T> node) {
        cacheNodes.put((T) nodeId, node);
        consistentHash.addNode((T) nodeId);
        System.out.println("添加节点: " + nodeId);
    }
    
    /**
     * 移除缓存节点
     */
    public void removeCacheNode(String nodeId) {
        CacheNode<T> node = cacheNodes.remove((T) nodeId);
        if (node != null) {
            consistentHash.removeNode((T) nodeId);
            System.out.println("移除节点: " + nodeId);
        }
    }
    
    /**
     * 获取数据：根据 key 找到对应节点
     */
    public T get(String key) {
        T nodeId = consistentHash.getNode(key);
        if (nodeId == null) {
            return null;
        }
        
        CacheNode<T> node = cacheNodes.get(nodeId);
        return node.get(key);
    }
    
    /**
     * 设置数据：写入对应节点
     */
    public void set(String key, T value) {
        T nodeId = consistentHash.getNode(key);
        if (nodeId == null) {
            throw new IllegalStateException("No cache node available");
        }
        
        CacheNode<T> node = cacheNodes.get(nodeId);
        node.set(key, value);
    }
}
```

## 面试追问预判

### Q1：Paxos 和 Raft 的核心区别是什么？

**面试官的心理分析：**

这道题考察的是对两种共识算法的理解深度。很多候选人能说出「Raft 比 Paxos 更易理解」，但只有真正理解的候选人能说出「Raft 的强 Leader 化设计如何简化了问题」。

**参考答案：**

```
表面区别：
- Paxos 是理论算法，Raft 是工程实现
- Paxos 可以没有 Leader，Raft 必须有 Leader
- Paxos 的提案号是单一数字，Raft 用 (Term, 索引) 定位日志

核心区别：

1. 强 Leader 化
   - Raft 强制要求有 Leader，所有写请求必须经过 Leader
   - Paxos 允许任何节点发起提案
   - Raft 的设计让问题更简单，但牺牲了一定灵活性

2. 日志追加方向
   - Raft：只有 Leader 能追加日志，日志只能从 Leader → Follower
   - Paxos：任何节点都可能接受来自任何方向的提案

3. 成员变更
   - Raft 支持单节点变更 + joint consensus
   - Paxos 的成员变更需要额外设计

4. 实际选择
   - etcd, TiKV, CockroachDB 选择 Raft
   - Chubby, Spanner 选择 Paxos
   - 区别更多是工程团队的偏好，而非算法优劣
```

### Q2：为什么 Paxos 难以实现？

**面试官的心理分析：**

这道题考察的是工程实践能力。很多候选人知道 Paxos 难，但不知道具体难在哪里。

**参考答案：**

```
Paxos 难以实现的原因：

1. 论文表述晦涩
   - 原始论文用古希腊故事比喻
   - 很多工程细节没有说明
   - 例如：如何选 Leader？如何处理成员变更？

2. 活锁问题
   - Basic Paxos 在高并发下可能活锁
   - 需要额外的优化（如随机退避）

3. Multi-Paxos 不完整
   - 原始论文只描述了 Basic Paxos
   - Multi-Paxos 的实现需要自己设计
   - Chubby 的实现花了 Google 团队大量时间

4. 优化空间大
   - 标准 Paxos 效率不高
   - 实际系统需要大量优化
   - Read-only 优化、Lease 机制等

这就是为什么 Raft 出现了：
- Raft 的目标就是「更易理解」
- 把 Paxos 的核心思想用更清晰的方式表达
- 让工程实现更容易正确
```

### Q3：一致性哈希的虚拟节点数量如何选择？

**面试官的心理分析：**

这是一道偏实践的题目，考察候选人对一致性哈希的理解深度。

**参考答案：**

```
虚拟节点数量的选择：

理论分析：
- 虚拟节点越多，负载分布越均匀
- 但需要维护更多的映射关系

工程实践：
- Memcached 客户端 Ketama：每个物理节点 150 个虚拟节点
- Redis Cluster：每槽 1 个虚拟节点（16384 槽 / N 节点）
- Cassandra：每物理节点 256 个虚拟节点

选择依据：
1. 节点数量
   - 节点越少，需要的虚拟节点越多
   - 节点越多，需要的虚拟节点可以少一些

2. 负载均衡精度要求
   - 对负载敏感的场景（缓存）需要更多虚拟节点
   - 对延迟敏感的场景需要考虑查找开销

3. 运维考虑
   - 虚拟节点越多，数据迁移越平滑
   - 但节点变更时，重新映射的数据量也越大

一般建议：
- 小规模（< 10 节点）：150-200 个虚拟节点
- 中等规模（10-100 节点）：100 个虚拟节点
- 大规模（> 100 节点）：50 个虚拟节点

实际系统如 Cassandra 使用 256 个，DynamoDB 根据节点数动态调整。
```

### Q4：Raft 的日志提交有什么特殊规则？

**面试官的心理分析：**

这是一道深入 Raft 核心安全性的题目。Raft 的「不能提交之前 Term 的日志」是很多候选人忽略但面试官经常追问的点。

**参考答案：**

```
Raft 的日志提交规则（面试重点）：

规则一：只能提交当前 Term 的日志
- Leader 只能提交当前 Term 的日志条目
- 不能直接提交之前 Term 的日志

规则二：之前 Term 的日志通过「间接提交」
- Leader 在当前 Term 创建新日志
- 如果新日志被复制到多数派并提交
- 那么之前 Term 的日志也间接提交了

为什么有这个规则？

考虑这个场景（会导致不正确）：

时间线：
T1: S1 是 Leader，写入 log[2]=v1
T2: S1 复制 v1 到 S2，但还没复制到多数派
T3: S1 宕机
T4: S5 被选为 Leader（Term 3）
T5: S5 写入 log[2]=v2（覆盖了 S1 的日志）
T6: S5 宕机，S1 恢复
T7: S1 被选为 Leader（Term 4）
T8: S1 尝试复制 log[2]=v1

如果 S1 可以提交 log[2]（Term 2 的日志）：
- v1 会被提交
- 但多数派（S1, S2）中的 v1 和多数派（S3, S4, S5）中的 v2 冲突

Raft 的解决方案：
- S1（Term 4）不能直接提交 Term 2 的日志
- 必须先在 Term 4 创建新日志
- 只有 Term 4 的日志被复制到多数派后
- Term 2 的日志才通过「间接提交」被提交

这是 Raft 保证安全性的关键细节。
```

## 总结

分布式并发算法是分布式系统的核心基础设施。理解这些算法背后的设计权衡，你才能在面对实际问题时做出正确的选择。

核心要点回顾：

- **Paxos** 是分布式共识的理论基石，但工程实现困难
- **Raft** 通过强 Leader 化和问题分解，让共识协议更易理解和实现
- **一致性哈希** 通过环结构和虚拟节点，实现了节点变更时的最小数据迁移
- **没有银弹**：每个算法都有其适用场景和局限性

面试中理解这些算法的核心设计思想，比记住具体实现细节更重要。
