双十一零点，你盯着监控大屏，看着 QPS 从平时的 1 万飙升至 50 万。

订单系统开始报警，数据库连接池耗尽，接口响应时间从 50ms 飙升到 5 秒。紧接着，缓存服务雪崩，依赖的下游服务一个接一个倒下。

最后，订单系统完全不可用，持续了整整 15 分钟。

这 15 分钟里，你眼睁睁看着用户下单失败，购物车被清空，直接损失可能超过百万。

**但这一切，本可以避免。**

如果你在流量入口做了限流，在接口层面做了削峰，在系统内部做了降级——这场灾难完全可以控制在萌芽状态。

## 流量控制的本质：让系统优雅地过载

### 什么是流量控制

流量控制（Flow Control），简单来说就是**控制请求的速率**，让系统能够在可承受的范围内处理请求。

```
没有流量控制：

用户请求 → 系统全力接收 → 系统过载 → 系统崩溃

有流量控制：

用户请求 → 限流器判断 → 通过？直接处理
                            ↓ 否
                         拒绝/排队/降级
                         ↓
                    系统稳定运行
```

### 为什么需要流量控制

很多人觉得自己的系统「平时够用」，不需要限流。但流量控制的价值，在以下场景中体现得淋漓尽致：

| 场景 | 风险 | 限流的作用 |
|------|------|-----------|
| 秒杀活动 | 瞬时流量远超系统处理能力 | 控制入口流量，避免压垮系统 |
| 爬虫/恶意请求 | 持续高频请求消耗资源 | 识别并限制异常流量 |
| 下游服务故障 | 上游调用堆积，引发级联故障 | 快速失败，保护下游 |
| 突发行情 | 行情波动引发大量查询 | 削峰填谷，平滑处理 |
| 业务 Bug | 逻辑错误导致请求量暴增 |兜底保护，发现问题 |

### 限流的两个核心维度

```
限流维度：

【按空间维度】
- 入口限流：网关层限流，限制总请求量
- 服务限流：单个服务的 QPS/TPS 限制
- 接口限流：单个接口的并发限制
- 用户限流：单个用户的请求频率限制

【按时间维度】
- 瞬时限流：毫秒级限流
- 秒级限流：QPS 限制
- 分钟级限流：保护稀缺资源
- 小时/天级限流：防爬虫、防薅羊毛
```

## 雪崩效应：为什么你的系统会突然崩溃

在说限流之前，我们必须先理解雪崩效应。因为限流的终极目标，就是防止雪崩。

### 雪崩效应的四个阶段

雪崩效应不是一蹴而就的，而是一个渐进的过程：

```
阶段一：延迟积累
┌─────────────────────────────────────────────────────────┐
│  用户请求 → 服务A → 服务B → 服务C                       │
│           (50ms)  (200ms)  (500ms)                      │
│                                                          │
│  服务C 响应变慢，请求堆积                                │
└─────────────────────────────────────────────────────────┘

阶段二：资源耗尽
┌─────────────────────────────────────────────────────────┐
│  服务C 的线程池被占满                                    │
│  连接池被占满                                            │
│  内存开始堆积未处理的请求                                │
│  服务C 开始出现超时                                      │
└─────────────────────────────────────────────────────────┘

阶段三：服务降级
┌─────────────────────────────────────────────────────────┐
│  服务C 超时，服务B开始降级                              │
│  服务B 降级，服务A也开始降级                            │
│  整个调用链上的服务都在苦苦支撑                          │
│  用户开始看到「系统繁忙」                                │
└─────────────────────────────────────────────────────────┘

阶段四：系统崩溃
┌─────────────────────────────────────────────────────────┐
│  服务B 的资源被耗尽                                      │
│  服务A 也开始超时                                        │
│  最终，整个系统不可用                                    │
│  数据库连接池耗尽                                        │
│  微服务网关熔断                                          │
│  用户看到 502/504 错误                                   │
└─────────────────────────────────────────────────────────┘
```

### 雪崩的常见触发点

```
雪崩触发点的类型：

【服务雪崩】
- 单个服务故障引发连锁反应
- 解决方案：熔断、隔离、超时控制

【资源雪崩】
- 共享资源耗尽（数据库连接池、线程池）
- 解决方案：资源隔离、限流保护

【缓存雪崩】
- 大量缓存同时过期
- 解决方案：过期时间随机化、热点数据永不过期

【流量雪崩】
- 突发流量超过系统承载能力
- 解决方案：入口限流、流量整形
```

### 一个真实的雪崩案例

```java
// 某电商系统的库存服务

public class InventoryService {
    
    @Autowired
    private RestTemplate restTemplate;
    
    @Autowired
    private RedisTemplate redisTemplate;
    
    public boolean deductStock(long productId, int quantity) {
        // 从缓存获取库存
        Integer stock = (Integer) redisTemplate.opsForValue()
            .get("stock:" + productId);
        
        if (stock == null) {
            // 缓存未命中，查询数据库
            stock = inventoryDao.getStock(productId);
            redisTemplate.opsForValue().set("stock:" + productId, stock);
        }
        
        if (stock < quantity) {
            return false;
        }
        
        // 扣减库存
        boolean success = inventoryDao.deductStock(productId, quantity);
        
        if (success) {
            // 更新缓存
            redisTemplate.opsForValue()
                .increment("stock:" + productId, -quantity);
        }
        
        return success;
    }
}
```

这个代码有什么问题？

```
问题分析：

1. 缓存雪崩风险
   - 如果 "stock:" + productId 这个 key 批量过期
   - 所有请求同时打到数据库
   - 数据库被压垮

2. 缓存击穿风险
   - 如果这个 key 正好过期
   - 大量请求同时查询数据库
   - 数据库被打穿

3. 缓存穿透风险
   - 如果 productId 不存在
   - 每次都会查询数据库
   - 恶意用户可以用不存在的 ID 刷接口

4. 调用链超时未控制
   - 如果下游服务响应慢
   - 请求会堆积在当前服务
   - 最终导致资源耗尽
```

## 全链路限流架构设计

理解了为什么需要限流，我们来看看如何设计一个完整的限流体系。

### 三层限流架构

```
全链路限流的三层架构：

【第一层：入口限流（网关层）】
┌─────────────────────────────────────────────────────────┐
│                                                          │
│    用户请求                                              │
│        ↓                                                │
│    [API Gateway]                                         │
│    - 限流维度：总 QPS、IP、用户 ID                       │
│    - 作用：保护整个系统，拦截最外层流量                   │
│                                                          │
└─────────────────────────────────────────────────────────┘
                         ↓
【第二层：链路限流（服务层）】
┌─────────────────────────────────────────────────────────┐
│                                                          │
│    [服务A]  →  [服务B]  →  [服务C]                      │
│      ↓           ↓           ↓                          │
│    限流器      限流器      限流器                       │
│    - QPS       - QPS       - QPS                        │
│    - 并发数    - 并发数    - 并发数                     │
│    - 作用：保护单个服务，防止资源耗尽                     │
│                                                          │
└─────────────────────────────────────────────────────────┘
                         ↓
【第三层：出口限流（资源层）】
┌─────────────────────────────────────────────────────────┐
│                                                          │
│    [数据库]    [Redis]    [外部服务]                    │
│      ↓           ↓           ↓                         │
│    连接池      连接池      HTTP 连接池                   │
│    - 限流：最大连接数                                   │
│    - 限流：最大等待时间                                 │
│    - 作用：保护下游资源，防止被拖垮                      │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

### 限流组件的选择

| 组件 | 适用场景 | 优点 | 缺点 |
|------|----------|------|------|
| Sentinel | 生产级限流熔断 | 功能完善，支持热点参数限流 | 学习成本高 |
| Hystrix | 熔断降级 | 功能稳定 | 已停止维护 |
| Resilience4j | 轻量级限流熔断 | 轻量，适合微服务 | 功能相对简单 |
| Guava RateLimiter | 单机限流 | 简单易用 | 不支持分布式 |
| Nginx limit_req | 网关层限流 | 性能高，配置简单 | 灵活性有限 |

## 令牌桶算法：分布式限流的核心

### 算法原理

令牌桶（Token Bucket）是目前最常用的限流算法之一。它的核心思想是：

```
令牌桶算法：

1. 有一个固定大小的桶
2. 桶里放令牌（Token）
3. 系统以固定速率向桶中放入令牌
4. 每进来一个请求，从桶中取走一个令牌
5. 如果桶空了，请求被拒绝或等待

关键特性：
- 桶的容量：决定最大突发流量
- 放令牌的速率：决定平均流量
```

### 单节点令牌桶实现

```java
import com.google.common.util.concurrent.RateLimiter;

public class RateLimiterExample {
    
    // 创建一个每秒产生 100 个令牌的限流器
    // 第二个参数是预热时间（秒），在这段时间内令牌产生速率会逐渐增加到目标值
    private static final RateLimiter limiter = 
        RateLimiter.create(100.0, 10, TimeUnit.SECONDS);
    
    public void handleRequest(Request request) {
        // 尝试获取一个令牌，阻塞直到获取成功
        // 如果不想阻塞，可以使用 tryAcquire()
        limiter.acquire();
        
        // 处理业务逻辑
        doProcess(request);
    }
    
    public Result fastFail(Request request) {
        // 非阻塞获取，如果获取不到立即返回
        if (limiter.tryAcquire()) {
            return doProcess(request);
        }
        // 获取不到，快速失败
        return Result.rateLimited();
    }
}
```

### 分布式令牌桶：Redis + Lua 实现

单节点的令牌桶无法跨 JVM 共享，在分布式环境下，我们需要借助 Redis 来实现。

```lua
-- 分布式令牌桶：Redis + Lua 实现

-- KEYS[1]: 限流 key
-- ARGV[1]: 桶容量（最大突发）
-- ARGV[2]: 令牌产生速率（每秒产生多少令牌）
-- ARGV[3]: 当前时间戳（毫秒）
-- ARGV[4]: 请求需要的令牌数

local key = KEYS[1]
local capacity = tonumber(ARGV[1])
local rate = tonumber(ARGV[2])
local now = tonumber(ARGV[3])
local requested = tonumber(ARGV[4])

-- 获取桶中当前令牌数和上次更新时间
local bucket = redis.call('HMGET', key, 'tokens', 'lastTime')
local tokens = tonumber(bucket[1])
local lastTime = tonumber(bucket[2])

-- 如果是新桶，初始化
if tokens == nil then
    tokens = capacity
    lastTime = now
end

-- 计算应该新增的令牌数
local elapsed = now - lastTime
local newTokens = math.min(capacity, tokens + elapsed * rate / 1000)

-- 尝试获取令牌
if newTokens >= requested then
    -- 获取成功，更新桶
    newTokens = newTokens - requested
    redis.call('HMSET', key, 'tokens', newTokens, 'lastTime', now)
    redis.call('EXPIRE', key, 60)
    return 1  -- 通过
else
    -- 获取失败，不更新
    return 0  -- 限流
end
```

对应的 Java 调用：

```java
public class RedisRateLimiter {
    
    private final RedisTemplate<String, String> redisTemplate;
    private final DefaultRedisScript<Long> luaScript;
    
    public RedisRateLimiter(RedisTemplate<String, String> redisTemplate) {
        this.redisTemplate = redisTemplate;
        
        // 加载 Lua 脚本
        this.luaScript = new DefaultRedisScript<>();
        this.luaScript.setScriptText(getLuaScript());
        this.luaScript.setResultType(Long.class);
    }
    
    public boolean tryAcquire(String key, int capacity, int rate) {
        long now = System.currentTimeMillis();
        
        Long result = redisTemplate.execute(
            luaScript,
            Collections.singletonList(key),
            String.valueOf(capacity),
            String.valueOf(rate),
            String.valueOf(now),
            "1"  // 每次请求需要 1 个令牌
        );
        
        return result != null && result == 1;
    }
    
    private String getLuaScript() {
        // 上面定义的 Lua 脚本
        return "local key = KEYS[1] ... ";
    }
}
```

### 滑动窗口限流算法

令牌桶虽然常用，但滑动窗口（Sliding Window）算法更加精确，适合对流量控制要求更高的场景。

```lua
-- 滑动窗口限流算法：Redis + Lua

-- KEYS[1]: 限流 key
-- ARGV[1]: 窗口大小（毫秒）
-- ARGV[2]: 窗口内最大请求数

local key = KEYS[1]
local window = tonumber(ARGV[1])
local limit = tonumber(ARGV[2])
local now = redis.call('TIME')
local currentTime = now[1] * 1000 + math.floor(now[2] / 1000)

-- 删除窗口外的旧数据
local windowStart = currentTime - window
redis.call('ZREMRANGEBYSCORE', key, 0, windowStart)

-- 获取窗口内的请求数
local requestCount = redis.call('ZCARD', key)

if requestCount < limit then
    -- 允许通过，添加新请求
    redis.call('ZADD', key, currentTime, currentTime .. ':' .. math.random())
    redis.call('PEXPIRE', key, window)
    return 1
else
    -- 拒绝
    return 0
end
```

### 三种算法的对比

| 算法 | 特点 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|----------|
| 固定窗口 | 最简单 | 实现简单，性能高 | 临界问题 | 简单限流 |
| 滑动窗口 | 更精确 | 临界时段流量均匀 | 实现稍复杂 | 精细化限流 |
| 令牌桶 | 支持突发 | 允许一定突发流量 | 实现复杂 | 流量整形 |
| 漏桶 | 流量平滑 | 输出速率恒定 | 不能突发 | 流量整形 |
| 计数器 | 最简单 | 易于实现 | 精度差 | 粗略限流 |

## Sentinel：生产级流量控制框架

### Sentinel 核心概念

Sentinel 是阿里巴巴开源的流量控制组件，提供了一系列生产级的能力：

```
Sentinel 核心概念：

【资源】
- 任何需要保护的代码块
- 可以是接口、方法、整个服务

【规则】
- 流控规则：控制请求速率
- 降级规则：异常时的降级策略
- 熔断规则：触发熔断的条件

【指标】
- QPS、响应时间、异常比例
- 实时的流量监控
```

### Sentinel 快速上手

```java
// 方式一：try-catch 方式

public class SentinelExample {
    
    public String getUserInfo(long userId) {
        try (Entry entry = SphU.entry("getUserInfo")) {
            // 业务逻辑
            return userService.getUserInfo(userId);
        } catch (BlockException e) {
            // 被限流了，执行兜底逻辑
            return "系统繁忙，请稍后再试";
        }
    }
}

// 方式二：注解方式（推荐）

@SentinelResource(value = "getUserInfo",
    blockHandler = "getUserInfoBlock",
    fallback = "getUserInfoFallback")
public String getUserInfo(long userId) {
    return userService.getUserInfo(userId);
}

// 限流时的处理
public String getUserInfoBlock(long userId, BlockException e) {
    log.warn("getUserInfo 被限流: userId={}, ex={}", userId, e.getMessage());
    return "系统繁忙，请稍后再试";
}

// 异常时的降级处理
public String getUserInfoFallback(long userId, Throwable e) {
    log.error("getUserInfo 异常: userId={}", userId, e);
    return "服务异常，请稍后再试";
}
```

### Sentinel 限流规则配置

```java
// 编程式配置限流规则

List<FlowRule> rules = new ArrayList<>();

FlowRule rule = new FlowRule();
// 资源名
rule.setResource("getUserInfo");
// 设置限流类型：QPS 或并发数
rule.setGrade(RuleConstant.FLOW_GRADE_QPS);
// 每秒允许的 QPS
rule.setCount(100);
// 是否集群限流
rule.setClusterMode(false)
// 限流效果
rule.setControlBehavior(RuleConstant.CONTROL_BEHAVIOR_DEFAULT);

rules.add(rule);
FlowRuleManager.loadRules(rules);
```

| 控制行为 | 效果 |
|----------|------|
| `CONTROL_BEHAVIOR_DEFAULT` | 直接拒绝，超过阈值直接拒绝 |
| `CONTROL_BEHAVIOR_WARM_UP` | 冷启动，逐步增加阈值 |
| `CONTROL_BEHAVIOR_RATE_LIMITER` | 匀速排队，延迟处理 |
| `CONTROL_BEHAVIOR_WARM_UP_RATE_LIMITER` | 冷启动 + 匀速排队 |

### 热点参数限流

Sentinel 的一大特色是支持热点参数限流，可以对携带特定参数值的请求进行精细化限流：

```java
// 热点参数限流示例

@SentinelResource(value = "getGoods",
    blockHandler = "getGoodsBlock",
    paramFlowItemSize = 1)  // 最多允许 1 个热点参数
public String getGoods(long goodsId, int type) {
    return goodsService.getGoods(goodsId, type);
}

// 配置热点参数限流规则
ParamFlowRule rule = new ParamFlowRule("getGoods")
    // 参数索引：0 表示 goodsId，1 表示 type
    .setParamIdx(0)
    // 参数值级别的限流阈值
    .setCount(10);

// 针对特定参数值设置更严格的限制
ParamFlowItem item = new ParamFlowItem()
    .setObject(String.valueOf(goodsId))  // 热门商品 ID
    .setClassType("long")
    .setCount(1);  // 每秒只允许 1 次访问

rule.setParamFlowItemList(Collections.singletonList(item));
ParamFlowRuleManager.loadRules(Collections.singletonList(rule));
```

## 接口级别限流：基于注解的通用方案

### 自定义限流注解

在实际项目中，我们通常希望限流逻辑和业务逻辑分离，通过注解来声明式地配置限流规则：

```java
import java.lang.annotation.*;

@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
@Documented
public @interface RateLimiter {
    
    /**
     * 限流 key，支持 SpEL 表达式
     * 默认使用方法名作为 key
     */
    String key() default "";
    
    /**
     * 限流阈值
     */
    double count() default 100;
    
    /**
     * 时间窗口（秒）
     */
    int window() default 1;
    
    /**
     * 限流后的提示消息
     */
    String message() default "请求过于频繁，请稍后再试";
    
    /**
     * 是否启用
     */
    boolean enabled() default true;
}
```

### 限流切面实现

```java
@Aspect
@Component
public class RateLimiterAspect {
    
    private final RedisTemplate<String, String> redisTemplate;
    private final RedissonClient redissonClient;
    
    public RateLimiterAspect(RedisTemplate<String, String> redisTemplate,
                             RedissonClient redissonClient) {
        this.redisTemplate = redisTemplate;
        this.redissonClient = redissonClient;
    }
    
    @Around("@annotation(rateLimiter)")
    public Object around(ProceedingJoinPoint joinPoint, 
                        RateLimiter rateLimiter) throws Throwable {
        
        if (!rateLimiter.enabled()) {
            return joinPoint.proceed();
        }
        
        // 获取限流 key
        String key = getKey(joinPoint, rateLimiter);
        
        // 尝试获取令牌
        String rateLimiterKey = "rate_limiter:" + key;
        RRateLimiter rRateLimiter = redissonClient.getRateLimiter(rateLimiterKey);
        
        // 设置速率
        rRateLimiter.trySetRate(
            RateType.OVERALL,
            (long) rateLimiter.count(),
            rateLimiter.window(),
            RateIntervalUnit.SECONDS
        );
        
        // 尝试获取令牌，获取不到则限流
        if (!rRateLimiter.tryAcquire()) {
            throw new RateLimitException(rateLimiter.message());
        }
        
        return joinPoint.proceed();
    }
    
    private String getKey(ProceedingJoinPoint joinPoint, RateLimiter rateLimiter) {
        // 如果指定了 key，直接使用
        if (StringUtils.isNotBlank(rateLimiter.key())) {
            return rateLimiter.key();
        }
        
        // 默认使用方法名 + 参数作为 key
        MethodSignature signature = (MethodSignature) joinPoint.getSignature();
        String methodName = signature.getMethod().getName();
        
        Object[] args = joinPoint.getArgs();
        if (args.length > 0) {
            return methodName + ":" + Arrays.toString(args);
        }
        
        return methodName;
    }
}
```

## 限流后的降级策略

限流不是目的，保护系统才是。当请求被限流后，我们需要一套完整的降级策略来处理这些请求。

### 降级策略类型

```
限流后的降级策略：

【直接拒绝】
- 立即返回友好提示
- HTTP 状态码：429 Too Many Requests
- 适用场景：非核心接口，保护系统资源

【排队等待】
- 将请求放入队列，等待处理
- 适用场景：核心接口，用户愿意等待
- 注意：需要设置超时时间

【服务降级】
- 返回简化版本的响应
- 适用场景：非核心功能

【跳转降级】
- 跳转到静态页面或缓存页面
- 适用场景：页面类请求

【熔断】
- 持续限流后触发熔断
- 熔断期间所有请求直接返回
- 适用场景：下游服务故障
```

### 降级策略实现

```java
public class DegradeStrategy {
    
    // 降级策略枚举
    public enum Strategy {
        REJECT,      // 直接拒绝
        QUEUE,       // 排队等待
        FALLBACK,    // 服务降级
        CIRCUIT      // 熔断
    }
    
    // 降级策略配置
    public class DegradeConfig {
        Strategy strategy;
        int queueTimeout;      // 排队超时时间（毫秒）
        String fallbackMethod; // 降级方法名
        int circuitThreshold;   // 熔断阈值
        int circuitWindow;      // 熔断窗口（秒）
    }
    
    // 执行降级
    public Object degrade(DegradeConfig config, Supplier<Object> originalLogic) {
        switch (config.strategy) {
            case REJECT:
                throw new RateLimitException("请求过于频繁");
                
            case QUEUE:
                return queueWithTimeout(originalLogic, config.queueTimeout);
                
            case FALLBACK:
                try {
                    return originalLogic.get();
                } catch (Exception e) {
                    return callFallbackMethod(config.fallbackMethod);
                }
                
            case CIRCUIT:
                if (circuitBreaker.isOpen()) {
                    return getCircuitBreakerResponse();
                }
                try {
                    return originalLogic.get();
                } catch (Exception e) {
                    circuitBreaker.recordFailure();
                    return callFallbackMethod(config.fallbackMethod);
                }
                
            default:
                throw new RateLimitException("请求过于频繁");
        }
    }
}
```

## 流量削峰：让高峰流量平滑处理

限流是被动防御，削峰是主动疏导。两者的结合，才能让系统在面对突发流量时游刃有余。

### 削峰的核心手段

```
流量削峰手段：

【消息队列削峰】
- 将同步请求转为异步处理
- 削峰填谷，平滑处理
- 最常用、最有效的手段

【预约抢购】
- 提前预约，分散流量
- 预约成功后才可抢购
- 适合秒杀场景

【分层过滤】
- 多层校验，逐步筛选
- 层层拦截无效请求
- 减少后端压力

【分批处理】
- 大批量任务分批执行
- 控制并发数
- 避免瞬时压力
```

### 消息队列削峰实战

```java
// 消息队列削峰示例：秒杀系统

@Service
public class SeckillService {
    
    @Autowired
    private RocketMQTemplate rocketMQTemplate;
    
    @Autowired
    private OrderService orderService;
    
    /**
     * 秒杀请求入口：只做快速校验，不做实际处理
     */
    public SeckillResponse seckill(SeckillRequest request) {
        // 1. 限流：每秒最多 10 万请求
        if (!rateLimiter.tryAcquire("seckill")) {
            return SeckillResponse.rateLimited();
        }
        
        // 2. 验证码校验：防刷
        if (!verifyCaptcha(request.getCaptcha())) {
            return SeckillResponse.captchaError();
        }
        
        // 3. 库存预扣减：Redis 原子操作
        Long remaining = redisTemplate.opsForValue()
            .decrement("seckill:stock:" + request.getProductId());
        
        if (remaining == null || remaining < 0) {
            // 库存不足
            return SeckillResponse.soldOut();
        }
        
        // 4. 发送消息，异步处理订单
        // 这是削峰的关键：用户请求立即返回，实际处理在后台
        rocketMQTemplate.asyncSend("seckill:order:topic", request, 
            new SendCallback() {
                @Override
                public void onSuccess(SendResult sendResult) {
                    log.info("秒杀请求发送成功: orderId={}", 
                        sendResult.getMsgId());
                }
                
                @Override
                public void onException(Throwable e) {
                    // 发送失败，回补库存
                    redisTemplate.opsForValue()
                        .increment("seckill:stock:" + request.getProductId());
                    log.error("秒杀请求发送失败: {}", e.getMessage());
                }
            });
        
        // 5. 立即返回，避免用户等待
        return SeckillResponse.success("下单成功，正在处理中");
    }
}

/**
 * 秒杀消息消费者：后台处理订单
 */
@Component
@RocketMQMessageListener(
    topic = "seckill:order:topic",
    consumerGroup = "seckill-order-consumer",
    concurrency = 100  // 100 个并发消费者
)
public class SeckillOrderConsumer implements RocketMQListener<SeckillRequest> {
    
    @Override
    public void onMessage(SeckillRequest request) {
        try {
            // 创建订单
            Order order = orderService.createOrder(request);
            
            // 发送成功通知
            mqTemplate.convertAndSend("seckill:notify:topic", 
                new NotifyMessage(request.getUserId(), order.getId()));
            
        } catch (Exception e) {
            // 处理失败，回补库存
            redisTemplate.opsForValue()
                .increment("seckill:stock:" + request.getProductId());
            log.error("创建订单失败: {}", e.getMessage(), e);
        }
    }
}
```

```java
// 防刷 Token 机制

@Service
public class AntiReplayService {
    
    private static final String TOKEN_PREFIX = "anti_replay:token:";
    private static final long TOKEN_EXPIRE_SECONDS = 300;  // 5 分钟
    
    /**
     * 生成 Token
     */
    public String generateToken(long userId, String action) {
        String token = UUID.randomUUID().toString().replace("-", "");
        String key = TOKEN_PREFIX + token;
        
        Map<String, Object> tokenData = new HashMap<>();
        tokenData.put("userId", userId);
        tokenData.put("action", action);
        tokenData.put("createTime", System.currentTimeMillis());
        
        redisTemplate.opsForHash().putAll(key, tokenData);
        redisTemplate.expire(key, TOKEN_EXPIRE_SECONDS, TimeUnit.SECONDS);
        
        return token;
    }
    
    /**
     * 校验 Token（原子操作，防止并发）
     */
    public boolean validateToken(String token, long userId, String action) {
        String key = TOKEN_PREFIX + token;
        
        // Lua 脚本保证原子性：校验 + 删除
        String luaScript = 
            "local tokenData = redis.call('HGETALL', KEYS[1]) " +
            "if #tokenData == 0 then return 0 end " +  // Token 不存在
            "local tokenUserId = nil " +
            "local tokenAction = nil " +
            "for i = 1, #tokenData, 2 do " +
            "  if tokenData[i] == 'userId' then tokenUserId = tokenData[i+1] end " +
            "  if tokenData[i] == 'action' then tokenAction = tokenData[i+1] end " +
            "end " +
            "if tonumber(tokenUserId) ~= tonumber(ARGV[1]) then return 0 end " +  // 用户不匹配
            "if tokenAction ~= ARGV[2] then return 0 end " +  // 动作不匹配
            "redis.call('DEL', KEYS[1]) " +  // 删除 Token
            "return 1";
        
        DefaultRedisScript<Long> script = new DefaultRedisScript<>(luaScript, Long.class);
        Long result = redisTemplate.execute(script, 
            Collections.singletonList(key),
            String.valueOf(userId), action);
        
        return result != null && result == 1;
    }
}

// 业务中使用
public class SeckillController {
    
    @PostMapping("/seckill")
    public SeckillResponse seckill(@RequestParam String token,
                                   @RequestBody SeckillRequest request) {
        // 1. 校验 Token
        if (!antiReplayService.validateToken(token, 
            request.getUserId(), "seckill")) {
            return SeckillResponse.tokenInvalid();
        }
        
        // 2. 继续秒杀逻辑
        return seckillService.seckill(request);
    }
}
```

```java
public class HttpStatusCode {

    public static final int OK = 200;

public class RateLimitResponse {

    private String message;
```

## 面试追问预判

### Q1：令牌桶和漏桶算法的区别是什么？

**面试官的心理分析：**

这道题考察对限流算法的理解深度。很多人知道两个算法的名字，但说不清楚适用场景的差异。

**参考答案：**

```
令牌桶 vs 漏桶：

【令牌桶（Token Bucket）】
- 以固定速率向桶中添加令牌
- 请求来了从桶中取令牌，取到才处理
- 特点：允许一定程度的突发流量
- 适用：流量整形、API 限流

【漏桶（Leaky Bucket）】
- 请求进入桶中，以固定速率"漏"出
- 无论请求多快，输出速率恒定
- 特点：输出完全平滑，不允许突发
- 适用：网络流量整形、打印机队列

【举例说明】

场景：秒杀活动

令牌桶：
- 桶容量 100，速率 1000/s
- 开始时桶是满的，100 个请求瞬间通过
- 之后每秒通过 1000 个

漏桶：
- 无论多少请求进来
- 始终以每秒 1000 个的速率处理
- 多余的请求排队或拒绝

【结论】
- 想要突发能力 → 令牌桶
- 想要流量平滑 → 漏桶
```

### Q2：如何设计一个每秒支持 100 万请求的限流系统？

**面试官的心理分析：**

这是一道系统设计题，考察综合能力。面试官想看的是你的架构思维，而不是某个具体算法的实现。

**参考答案：**

```
设计思路：

【分层限流】
- 第一层：Nginx 层限流（L4/L7），拦截 80% 恶意流量
- 第二层：网关层限流，IP + UserID 限流
- 第三层：服务层限流，接口级别

【分布式限流】
- 使用 Redis Cluster 做限流计数
- 每个节点本地缓存计数，减少网络开销
- 本地缓存定期同步到 Redis

【异步处理】
- 限流通过 → 放入消息队列
- 消费者异步处理，削峰填谷
- 用户立即返回"请求成功，正在处理"

【资源隔离】
- 核心接口独立线程池
- 避免被非核心接口拖垮

【降级预案】
- 超出系统承载能力 → 服务降级
- 返回简化响应或友好提示
```

### Q3：Sentinel 的滑动窗口是怎么实现的？

**面试官的心理分析：**

这是一道细节题，考察对 Sentinel 内部实现的了解程度。能回答出具体实现的候选人凤毛麟角。

**参考答案：**

```
Sentinel 滑动窗口实现：

【数据结构】
- 使用环形数组（WindowWrap）存储每个时间窗口的统计
- 每个窗口包含：开始时间、结束时间、统计数据（pass/fail/block）

【窗口划分】
- 默认将 1 秒划分为 2 个 500ms 窗口
- 统计时聚合多个小窗口的数据

【线程安全】
- 使用分段锁（Segment）保护不同窗口
- 避免全局锁的性能瓶颈

【具体流程】
1. 计算当前时间戳属于哪个窗口
2. 获取窗口，如果窗口不存在则创建
3. 在窗口上增加计数
4. 统计滑动窗口内的总请求数
5. 超过阈值则限流
```

## 总结

流量控制与削峰是保护系统稳定运行的关键手段。

核心要点：

- **雪崩效应**是一个渐进的过程，限流的目标是阻止雪崩发展到系统崩溃
- **全链路限流**需要入口限流、链路限流、资源限流三层配合
- **令牌桶算法**是分布式限流的核心，支持突发流量
- **Sentinel** 提供了完整的限流、熔断、降级能力
- **削峰**是主动疏导，限流是被动防御，两者结合才能应对极端场景
- **防刷**是限流的延伸，Token + 验证码是常用组合

下一节我们来看消息队列如何进一步提升系统的并发处理能力。
