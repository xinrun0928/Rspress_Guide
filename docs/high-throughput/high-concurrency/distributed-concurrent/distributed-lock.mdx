# 分布式锁（Redis、ZooKeeper、etcd 实现对比）

想象一个场景：双十一零点，百万用户同时抢购一部 iPhone。

你的库存只有 100 台，但技术上如何保证不会出现第 101 个人抢到？

单机环境下，`synchronized` 或 `ReentrantLock` 就能解决问题。但在分布式系统里，请求分布在多台机器上，每个 JVM 进程都有自己的锁，线程 A 在机器 1 加的锁，对机器 2 上的线程 B 毫无约束力。

**这就是分布式锁要解决的问题：让多台机器协调同步，共享同一把锁。**

---

## 为什么需要分布式锁

### 单机锁的局限性

```java
// 单机锁：synchronized 或 ReentrantLock
public class单机库存服务 {
    
    private int stock = 100;
    
    // 这个锁只对当前 JVM 内的线程有效
    public synchronized boolean deductStock() {
        if (stock > 0) {
            stock--;
            return true;
        }
        return false;
    }
}

// 问题：如果是集群部署呢？

// 机器1：运行着 10 个实例
// 机器2：运行着 10 个实例
// 机器3：运行着 10 个实例

// 机器1 的锁，无法阻止机器2 的线程进入临界区
// 结果：超卖！
```

### 分布式锁的核心问题

```
分布式锁需要解决的问题：

1. 互斥性：同一时刻只能有一个客户端持有锁
2. 死锁避免：客户端崩溃时，锁必须能自动释放
3. 饥饿避免：避免锁被某个客户端长期独占
4. 性能保障：加锁和解锁操作要快
5. 可重入性：同一客户端可以多次获取同一把锁

这五个问题，缺一不可。
```

---

## 分布式锁的五个核心要求

### 1. 互斥性：锁的基本属性

互斥性是最基本的要求：同一时刻只能有一个客户端持有锁。

```
互斥性的实现方式：

Redis：SET key value NX
ZooKeeper：创建临时顺序节点
etcd：基于 Lease 的键值对

关键点：互斥性必须由分布式系统保证
不能依赖「先到先得」这种时序假设
因为分布式系统中的时钟并不一致
```

### 2. 死锁避免：锁的自动释放

客户端崩溃时，锁必须能被自动释放，否则整个系统会永久死锁。

```
死锁场景：

客户端 A 获取了锁
客户端 A 崩溃了（还没来得及释放锁）
客户端 A 永远无法释放锁
所有其他客户端永远无法获取锁
系统死锁！

解决方案：
- Redis：TTL 过期
- ZooKeeper：临时节点自动删除
- etcd：Lease 自动过期
```

### 3. 饥饿避免：公平的锁获取

如果锁被同一个客户端长期持有，其他客户端可能永远获取不到锁。

```
饥饿场景：

客户端 A 获取了锁
客户端 A 开始处理一个耗时任务（10 分钟）
客户端 B、C、D 等待 10 分钟

解决方案：
- 公平锁：按请求顺序分配锁
- ZooKeeper：临时顺序节点天然保证公平
- Redis：需要额外实现（如 Redisson 的公平锁）
```

### 4. 性能保障：快速的加解锁

分布式锁的操作通常很频繁，必须保证低延迟。

```
性能对比：

Redis：O(1) 操作，单机可达 10万+ QPS
ZooKeeper：需要服务端交互，约 1000-5000 QPS
etcd：基于 Raft，约 5000-10000 QPS

Redis 的优势在于：
- 纯内存操作
- 单线程模型（无竞争）
- 简单的键值存储
```

### 5. 可重入性：同一客户端多次获取

同一客户端可以多次获取同一把锁，不会死锁。

```
可重入场景：

public void methodA() {
    lock.lock();
    try {
        // 业务逻辑
        methodB();  // 内部也获取同一把锁
    } finally {
        lock.unlock();
    }
}

public void methodB() {
    lock.lock();
    try {
        // 业务逻辑
    } finally {
        lock.unlock();
    }
}

可重入要求：
- 锁需要记录持有者的身份
- 记录重入次数
- 释放锁时重入次数减一，直到 0 才真正释放
```

## Redis 分布式锁：性能之王

### 核心实现

Redis 分布式锁的核心是 `SET key value NX EX` 命令：

```java
/**
 * Redis 分布式锁基本实现
 */
public class RedisDistributedLock {
    
    private final JedisPool jedisPool;
    
    /**
     * 获取锁
     * 
     * 关键点：
     * 1. SET NX：原子性地设置键值，只有键不存在时才成功
     * 2. SET EX：设置过期时间，避免死锁
     */
    public boolean lock(String lockKey, String lockValue, int expireSeconds) {
        Jedis jedis = jedisPool.getResource();
        try {
            // 等价于：SET lockKey lockValue NX EX expireSeconds
            String result = jedis.set(lockKey, lockValue, "NX", "EX", expireSeconds);
            return "OK".equals(result);
        } finally {
            jedis.close();
        }
    }
    
    /**
     * 释放锁
     * 
     * 关键点：必须判断锁的值是否匹配
     * 否则可能释放了其他客户端的锁
     */
    public boolean unlock(String lockKey, String lockValue) {
        Jedis jedis = jedisPool.getResource();
        try {
            // Lua 脚本保证原子性
            String script = 
                "if redis.call('get', KEYS[1]) == ARGV[1] then " +
                "    return redis.call('del', KEYS[1]) " +
                "else return 0 end";
            
            Object result = jedis.eval(script, 1, lockKey, lockValue);
            return 1L.equals(result);
        } finally {
            jedis.close();
        }
    }
}
```

### Redis 分布式锁的三个坑

#### 坑一：锁过期，业务还没完成

```
场景描述：

客户端 A 获取了锁，设置了 10 秒过期时间
客户端 A 开始执行一个耗时任务（15 秒）
10 秒后，锁自动过期
客户端 B 获取了同一把锁
客户端 A 完成任务，释放了客户端 B 的锁
客户端 B 继续执行，这时实际上有竞争问题

问题：锁被提前释放，业务还没完成
```

```java
/**
 * 错误示例：业务执行时间超过锁过期时间
 */
public void wrongExample() {
    String lockKey = "product:1001:lock";
    String lockValue = UUID.randomUUID().toString();
    
    // 获取锁，过期时间 10 秒
    boolean locked = redisLock.lock(lockKey, lockValue, 10);
    if (!locked) {
        throw new RuntimeException("获取锁失败");
    }
    
    try {
        // 业务执行时间：15 秒
        Thread.sleep(15000);  // 模拟耗时操作
        // 此时锁已经过期了！
    } finally {
        redisLock.unlock(lockKey, lockValue);
    }
}
```

**解决方案：看门狗机制（Watchdog）**

```java
/**
 * Redisson 的看门狗机制
 * 
 * Redisson 会自动续期锁：
 * - 默认情况下，看门狗每 10 秒检查一次
 * - 如果锁还被持有，延长过期时间
 * - 续期时间 = originalExpire / 3 = 10 / 3 ≈ 3.3 秒
 * - 最多续期到 originalExpire
 */
public class RedissonWatchdogExample {
    
    public void correctExample() {
        RedissonClient redisson = Redisson.create();
        RLock lock = redisson.getLock("product:1001:lock");
        
        // lock() 会自动续期，看门狗默认工作
        // 如果不调用 unlock()，锁会在 30 秒后自动释放
        // （可以配置 lockWatchdogTimeout）
        lock.lock();
        
        try {
            // 业务执行时间：15 秒
            Thread.sleep(15000);
            // 看门狗会自动续期，不会提前释放
        } finally {
            lock.unlock();
        }
    }
    
    /**
     * lock() vs tryLock()
     * 
     * lock()：阻塞等待，获取到锁才返回
     * tryLock()：立即返回，获取成功返回 true，失败返回 false
     */
    public void tryLockExample() {
        RLock lock = redisson.getLock("product:1001:lock");
        
        // 尝试获取锁，等待 5 秒，锁自动过期 30 秒
        boolean locked = lock.tryLock(5, 30, TimeUnit.SECONDS);
        if (locked) {
            try {
                // 业务逻辑
            } finally {
                lock.unlock();
            }
        }
    }
}
```

客户端 A 获取了锁（值 = "A"）

客户端 B 获取了同一把锁（值 = "B"）

#### 坑三：主从切换导致锁丢失

```
场景描述（主从复制延迟）：

1. 客户端 A 在主节点获取锁（SET key value NX EX 30）
2. 主节点将锁信息同步到从节点（异步复制）
3. 主节点宕机，还没来得及同步到从节点
4. 从节点升级为主节点
5. 锁信息丢失！
6. 客户端 B 获取了同一把锁
7. 客户端 A 和客户端 B 同时在临界区

问题：主从切换导致分布式锁失效
```

```
解决方案：Redlock 算法

Redlock 的思路是：在 N 个 Redis 节点上获取锁
只有多数派节点都成功获取锁，才认为获取成功
这样即使有节点宕机，也不影响锁的正确性
```

### Redlock 算法：多节点分布式锁

```java
/**
 * Redlock 算法实现
 * 
 * 核心思想：在 N 个独立的 Redis 节点上获取锁
 * 只有超过 N/2 + 1 个节点成功，才算获取成功
 */
public class Redlock {
    
    private final List<JedisPool> jedisPools;
    private final int quorum;
    
    public Redlock(List<JedisPool> jedisPools) {
        this.jedisPools = jedisPools;
        this.quorum = jedisPools.size() / 2 + 1;
    }
    
    /**
     * 获取锁
     */
    public RedlockResult lock(String resourceName, long ttlMillis) {
        String lockValue = UUID.randomUUID().toString();
        int failedAttempts = 0;
        List<Long>成功Times = new ArrayList<>();
        
        long startTime = System.currentTimeMillis();
        
        // 并行向所有节点尝试获取锁
        for (JedisPool pool : jedisPools) {
            try (Jedis jedis = pool.getResource()) {
                // SET resourceName lockValue NX PX ttl
                String result = jedis.set(resourceName, lockValue, "NX", "PX", ttlMillis);
                
                if ("OK".equals(result)) {
                    成功Times.add(System.currentTimeMillis());
                }
            } catch (Exception e) {
                failedAttempts++;
            }
        }
        
        // 计算获取锁花费的时间
        long elapsedTime = System.currentTimeMillis() - startTime;
        
        // 检查是否获取了多数派锁
        if (成功Times.size() >= quorum) {
            // 计算锁的有效期
            long validityTime = ttlMillis - elapsedTime - (failedAttempts * 50);  // 留些余量
            return new RedlockResult(true, lockValue, validityTime);
        } else {
            // 获取失败，释放已获取的锁
            for (JedisPool pool : jedisPools) {
                try (Jedis jedis = pool.getResource()) {
                    unlock(jedis, resourceName, lockValue);
                } catch (Exception ignored) {
                }
            }
            return new RedlockResult(false, null, 0);
        }
    }
    
    /**
     * 释放锁
     */
    public void unlock(String resourceName, String lockValue) {
        for (JedisPool pool : jedisPools) {
            try (Jedis jedis = pool.getResource()) {
                unlock(jedis, resourceName, lockValue);
            } catch (Exception ignored) {
            }
        }
    }
    
    private void unlock(Jedis jedis, String resourceName, String lockValue) {
        String script = 
            "if redis.call('get', KEYS[1]) == ARGV[1] then " +
            "    return redis.call('del', KEYS[1]) " +
            "else return 0 end";
        jedis.eval(script, 1, resourceName, lockValue);
    }
}
```

```
Redlock 的争议：

Redis 作者 Antirez 提出 Redlock 后，引发了很多争议：

争议一：时钟跳跃
- Redlock 假设系统时钟是准确的
- 如果某个 Redis 节点时钟跳跃，可能导致锁提前过期

争议二：性能问题
- 需要在 5 个节点上获取锁
- 网络延迟叠加，性能下降

争议三：真正的分布式一致性
- Redlock 是基于多数派的，不是强一致的
- 仍然无法解决分布式系统中的 CAP 约束

Martin Kleppmann 的反驳：
- Redlock 不是严格意义上的分布式锁
- 如果需要强一致锁，应该用 ZooKeeper 或 etcd

实际建议：
- 如果性能是主要需求：Redis 单节点锁 + Redisson 看门狗
- 如果可靠性是主要需求：ZooKeeper 或 etcd
- Redlock 适合对性能和可靠性都有要求，但不极致追求的场景
```

     */

    public boolean tryLockExample() {

        try {

                } finally {
                    if (lock.isHeldByCurrentThread()) {

                    }
                }

        try {

            fairLock.unlock();
        }
    }

        } finally {

        }
    }

        // 库存扣减逻辑
    }
    
    private int getStock() {

        return 100;
    }

## ZooKeeper 分布式锁：高可靠的代名词

### 核心实现原理

ZooKeeper 分布式锁基于**临时顺序节点**和 **Watcher 机制**：

```java
/**
 * ZooKeeper 分布式锁核心原理
 * 
 * 基于临时顺序节点 + Watcher 实现
 */
public class ZooKeeperDistributedLock {
    
    private final ZooKeeper zk;
    private final String lockPath;
    private final String nodeValue;
    private String currentNodePath;
    
    public ZooKeeperDistributedLock(ZooKeeper zk, String lockPath, String nodeValue) {
        this.zk = zk;
        this.lockPath = lockPath;
        this.nodeValue = nodeValue;
    }
    
    /**
     * 获取锁的核心流程
     */
    public boolean lock() throws InterruptedException, KeeperException {
        
        // 1. 创建临时顺序节点
        // ZooKeeper 节点路径格式：/lock/path/node-0000000001
        String createdPath = zk.create(
            lockPath + "/node-",              // 节点名前缀
            nodeValue.getBytes(),             // 节点值
            ZooDefs.Ids.OPEN_ACL_UNSAFE,      // ACL 权限
            CreateMode.EPHEMERAL_SEQUENTIAL   // 临时顺序节点
        );
        currentNodePath = createdPath;
        
        // 2. 获取所有子节点
        List<String> children = zk.getChildren(lockPath, false);
        
        // 3. 对子节点排序
        Collections.sort(children);
        
        // 4. 判断自己是否是最小的节点
        String smallestNode = children.get(0);
        
        if (currentNodePath.endsWith(smallestNode)) {
            // 我是最小的节点，获取锁成功
            return true;
        } else {
            // 我不是最小的，监听前一个节点
            String previousNode = findPreviousNode(children);
            watchPreviousNode(previousNode);
            return false;
        }
    }
    
    /**
     * 找到前一个节点
     */
    private String findPreviousNode(List<String> children) {
        String myNodeName = currentNodePath.substring(currentNodePath.lastIndexOf('/') + 1);
        int myIndex = children.indexOf(myNodeName);
        if (myIndex > 0) {
            return lockPath + "/" + children.get(myIndex - 1);
        }
        return null;
    }
    
    /**
     * 监听前一个节点
     */
    private void watchPreviousNode(String previousNode) throws InterruptedException, KeeperException {
        if (previousNode != null) {
            // Watcher：当节点被删除时，收到通知
            zk.exists(previousNode, event -> {
                if (event.getType() == Watcher.Event.EventType.NodeDeleted) {
                    // 前一个节点被删除了，重新竞争锁
                    synchronized (this) {
                        this.notifyAll();
                    }
                }
            });
        }
    }
    
    /**
     * 释放锁：删除临时节点
     */
    public void unlock() throws InterruptedException, KeeperException {
        if (currentNodePath != null) {
            zk.delete(currentNodePath, -1);
        }
    }
}
```

### ZooKeeper 锁的工作流程

```
ZooKeeper 分布式锁工作流程：

场景：3 个客户端竞争同一把锁

初始状态：/lock 目录下无子节点

客户端 A 请求锁：
1. 创建临时顺序节点：/lock/node-0000000001
2. 获取子节点列表：[node-0000000001]
3. 判断：自己是第一个，获取锁成功 ✓

客户端 B 请求锁：
1. 创建临时顺序节点：/lock/node-0000000002
2. 获取子节点列表：[node-0000000001, node-0000000002]
3. 判断：node-0000000001 在自己前面
4. 监听 node-0000000001

客户端 C 请求锁：
1. 创建临时顺序节点：/lock/node-0000000003
2. 获取子节点列表：[node-0000000001, node-0000000002, node-0000000003]
3. 判断：node-0000000001 在自己前面
4. 监听 node-0000000002

客户端 A 释放锁（删除 node-0000000001）：
1. ZK 通知客户端 B：node-0000000001 已删除
2. 客户端 B 重新检查
3. 现在自己是第一个，获取锁成功 ✓

关键优势：
- 公平锁：按创建顺序获取锁
- 不会死锁：临时节点，客户端崩溃自动删除

```

### etcd 分布式锁实现

```java
/**
 * etcd v3 分布式锁实现
 * 
 * etcd v3 的锁基于 Lease + MVCC + Watch 实现
 */
public class EtcdDistributedLock {
    
    private final KV kvClient;
    private final Lease lease;
    private long leaseId;
    private String lockKey;
    
    /**
     * 获取锁
     */
    public boolean lock(String lockName, long ttlSeconds) throws Exception {
        
        // 1. 创建 Lease（租约）
        LeaseGrantResponse leaseResponse = lease.grant(ttlSeconds);
        leaseId = leaseResponse.getID();
        
        // 2. 在 Lease 下创建 key
        // 格式：lockPrefix/lockName/序列号
        lockKey = "/locks/" + lockName + "/" + UUID.randomUUID().toString();
        
        // 3. 尝试获取锁（乐观锁）
        // 使用 Txn 保证原子性
       Txn txn = kvClient.txn();
        
        // 4. 获取当前最小的序列号
        // 创建一个比较操作
        txn.If(
            // 条件：如果锁不存在，或者自己是序列号最小的
            new Comparator[] {
                compare.lockKey.notExists().or(compare.lockKey.isMinSequence())
            }
        );
        
        // 5. 放入操作
        txn.Then(kvClient.put(lockKey, "locked", PutOption.newBuilder()
            .withLeaseId(leaseId)
            .build()));
        
        // 6. 执行事务
        TxnResponse response = txn.commit();
        
        if (response.isSucceeded()) {
            // 获取锁成功
            return true;
        } else {
            // 获取锁失败，需要等待
            return waitForLock();
        }
    }
    
    /**
     * 等待锁
     */
    private boolean waitForLock() throws Exception {
        
        // 1. 获取所有锁键
        GetResponse getResponse = kvClient.get(
            KeyUtils.getPrefix("/locks/"),
            GetOption.newBuilder()
                .withPrefix()
                .withSortTarget(KeySort.Target.CREATE)
                .build()
        );
        
        List<KeyValue> kvs = getResponse.getKvs();
        
        // 2. 找到自己的位置
        int myIndex = -1;
        for (int i = 0; i < kvs.size(); i++) {
            if (new String(kvs.get(i).getKey()).equals(lockKey)) {
                myIndex = i;
                break;
            }
        }
        
        if (myIndex <= 0) {
            // 我是最小的，应该能获取锁
            return true;
        }
        
        // 3. 监听前一个键
        String prevKey = new String(kvs.get(myIndex - 1).getKey());
        
        // Watch 前一个键的变化
        CountDownLatch latch = new CountDownLatch(1);
        
        Watch watch = kvClient.getWatchClient();
        watch.watch(prevKey, response -> {
            for (WatchEvent event : response.getEvents()) {
                if (event.getEventType() == EventType.DELETE) {
                    latch.countDown();  // 前一个键被删除，我可以重新竞争
                }
            }
        });
        
        // 等待被唤醒
        return latch.await(30, TimeUnit.SECONDS);
    }
    
    /**
     * 释放锁
     */
    public void unlock() throws Exception {
        if (lockKey != null && leaseId != 0) {
            kvClient.delete(lockKey, DeleteOption.newBuilder()
                .withPrevKV(true)
                .build());
            lease.revoke(leaseId);
        }
    }
}
```

```java
/**
 * 数据库唯一索引实现分布式锁
 * 
 * 利用数据库的唯一约束实现互斥
 */
public class DatabaseDistributedLock {
    
    private final DataSource dataSource;
    
    /**
     * 获取锁
     */
    public boolean lock(String lockName, String lockValue) {
        Connection conn = null;
        try {
            conn = dataSource.getConnection();
            
            // 尝试插入一条记录
            // 由于有唯一索引，重复插入会失败
            String sql = "INSERT INTO distributed_locks (lock_name, lock_value, expire_time) " +
                         "VALUES (?, ?, ?)";
            
            PreparedStatement ps = conn.prepareStatement(sql);
            ps.setString(1, lockName);
            ps.setString(2, lockValue);
            ps.setTimestamp(3, new Timestamp(System.currentTimeMillis() + 30000));
            
            ps.executeUpdate();
            return true;
            
        } catch (SQLException e) {
            // 唯一约束冲突，获取锁失败
            return false;
        } finally {
            closeQuietly(conn);
        }
    }
    
    /**
     * 释放锁（只释放自己的锁）
     */
    public void unlock(String lockName, String lockValue) {
        Connection conn = null;
        try {
            conn = dataSource.getConnection();
            
            String sql = "DELETE FROM distributed_locks " +
                         "WHERE lock_name = ? AND lock_value = ?";
            
            PreparedStatement ps = conn.prepareStatement(sql);
            ps.setString(1, lockName);
            ps.setString(2, lockValue);
            ps.executeUpdate();
            
        } catch (SQLException e) {
            // 处理异常
        } finally {
            closeQuietly(conn);
        }
    }
    
    /**
     * 定时清理过期锁
     */
    @Scheduled(fixedRate = 60000)
    public void cleanExpiredLocks() {
        Connection conn = null;
        try {
            conn = dataSource.getConnection();
            
            String sql = "DELETE FROM distributed_locks WHERE expire_time < ?";
            
            PreparedStatement ps = conn.prepareStatement(sql);
            ps.setTimestamp(1, new Timestamp(System.currentTimeMillis()));
            ps.executeUpdate();
            
        } catch (SQLException e) {
            // 处理异常
        } finally {
            closeQuietly(conn);
        }
    }
}

### 方案二：Sharding 策略（无锁设计）

```
有时候，最好的锁就是不用锁。

如果能通过数据分片，让每个分片只有一台机器处理，
就不需要跨机器的分布式锁了。

/**
 * Sharding 策略替代分布式锁
 */
public class ShardingInventoryService {
    
    private final int shardCount = 10;
    private final List<InventoryService> shards;
    
    public ShardingInventoryService() {
        shards = new ArrayList<>();
        for (int i = 0; i < shardCount; i++) {
            // 每个分片使用单机锁即可
            shards.add(new SingleNodeInventoryService());
        }
    }
    
    /**
     * 根据商品 ID 选择分片
     */
    private InventoryService selectShard(long productId) {
        int shardIndex = (int) (productId % shardCount);
        return shards.get(shardIndex);
    }
    
    /**
     * 扣减库存
     */
    public boolean deductStock(long productId) {
        // 直接路由到对应分片，使用单机锁
        InventoryService shard = selectShard(productId);
        return shard.deductStock(productId);
    }
}

## 面试追问预判

### Q1：分布式锁和分布式事务有什么区别？

**面试官的心理分析：**

这道题考察的是对分布式系统中两个核心概念的区分能力。很多候选人容易混淆这两个概念。

**参考答案：**

```
分布式锁 vs 分布式事务：

【分布式锁】
- 目标：互斥访问共享资源
- 解决问题：谁可以进入临界区
- 范围：通常是单个资源
- 失败处理：获取锁失败时等待或放弃
- 一致性级别：通常不保证一致性，只保证互斥

【分布式事务】
- 目标：跨多个服务保证原子性
- 解决问题：多个操作要么全成功要么全失败
- 范围：通常是多个资源
- 失败处理：回滚所有操作
- 一致性级别：强一致性（ACID）或最终一致性（BASE）

类比：
- 分布式锁：解决「谁可以用洗手间」的问题
- 分布式事务：解决「洗手间和厨房的改造必须同时完成」的问题

实际场景：
- 扣库存：分布式锁（保证不超卖）
- 支付 + 扣库存 + 发货：分布式事务（要么全做，要么全不做）
```

### Q2：Redis 锁的 TTL 设置有什么讲究？

**面试官的心理分析：**

这道题考察的是工程实践能力。TTL 设置是 Redis 分布式锁最容易出问题的点之一。

**参考答案：**

```
Redis 锁 TTL 设置的考虑因素：

1. 业务执行时间
   - TTL 必须大于正常业务执行时间
   - 如果业务通常需要 3 秒，TTL 应该设为 10-30 秒

2. 集群规模
   - 节点越多，Redlock 获取锁的时间可能越长
   - 需要留足 buffer

3. 续期机制
   - 如果使用 Redisson 看门狗，可以适当缩短 TTL
   - 看门狗会自动续期

4. 故障恢复时间
   - 如果期望 30 秒内完成故障转移，TTL 应该大于这个时间
   - 避免锁在故障恢复期间自动过期

实际建议：
- 一般场景：TTL = 30 秒，配合看门狗
- 高可靠场景：TTL = 60 秒，手动续期
- Redlock：TTL = 手动计算（考虑网络延迟）

TTL 设置的常见错误：
- TTL 太短：业务还没执行完，锁就过期了
- TTL 太长：节点故障后，锁迟迟不释放
```

### Q3：ZooKeeper 锁为什么性能不如 Redis？

**面试官的心理分析：**

这道题考察的是对 ZooKeeper 和 Redis 内部实现的理解。

**参考答案：**

```
ZooKeeper 锁性能不如 Redis 的原因：

1. 服务端交互次数
   - Redis 锁：通常 2 次操作（SET + DEL）
   - ZooKeeper 锁：至少 3-4 次操作
     - 创建临时节点（1 次）
     - 获取子节点列表（1 次）
     - Watch 通知（网络往返）

2. 网络开销
   - Redis：纯内存操作，响应时间 < 1ms
   - ZooKeeper：每个请求都需要 Leader 处理
     - Leader → Follower 复制
     - 等待多数派确认
     - 网络往返时间增加

3. 数据结构复杂度
   - Redis：简单的键值存储
   - ZooKeeper：维护树形结构和 Watcher 索引
   - Watcher 需要维护和触发

4. 写入路径
   - Redis：单线程命令处理，无竞争
   - ZooKeeper：需要 Raft/ZAB 协议协调
   - 强一致性保证带来额外开销

本质上：Redis 用「弱一致性」换「高性能」
ZooKeeper 用「强一致性」换「可靠性」
```

### Q4：如何设计一个高可用的分布式锁？

**面试官的心理分析：**

这是一道综合性设计题，考察候选人对分布式系统设计的全面理解。

**参考答案：**

```
高可用分布式锁设计要素：

1. 多层架构
   - 第一层：Redis 单节点锁（高性能）
   - 第二层：ZooKeeper/etcd 锁（高可靠）
   - 第三层：数据库唯一索引（兜底）

2. 自动降级
   - 正常情况：使用 Redis 锁（高性能）
   - Redis 不可用：降级到 ZooKeeper 锁
   - ZooKeeper 也不可用：降级到数据库锁

3. 锁续期机制
   - 看门狗自动续期
   - 业务执行期间保持锁的有效性

4. 监控告警
   - 锁获取失败率监控
   - 锁持有时间异常监控
   - 分布式锁健康状态监控

5. 故障转移
   - Redis 主从自动切换
   - ZooKeeper 快速选主
   - 客户端自动重连

    }
    
    return LOCKED;
}

**候选人乙**：

**第四，Lua 脚本保证原子性。**
> 扣减库存必须在 Lua 脚本中执行，保证检查库存和扣减库存的原子性。

**面试官**：Redis 锁有主从切换丢锁的问题，你怎么办？

**方案一：Redlock。**
> 在 5 个独立的 Redis 节点上获取锁，只有多数派成功才算获取成功。这种方案可以容忍部分节点故障。

**方案二：Redisson 的联锁（MultiLock）。**
> 把多个 Redis 节点锁当成一个锁来用，必须所有节点都加锁成功才算成功。

**方案三：换 ZooKeeper 或 etcd。**
> 如果超卖问题非常严重（比如高价值商品），应该用 ZooKeeper 或 etcd，它们的一致性保证更强，虽然性能会低一些。

**面试官点头**：你认为 Redis 锁适合什么场景？

**候选人乙**：
> Redis 锁适合「允许少量超卖、性能优先」的秒杀场景。对于金融级别的库存扣减，我建议用 ZooKeeper 或 etcd。

## 总结

分布式锁是分布式系统中的基础组件，没有银弹，只有权衡。

核心要点回顾：

- **Redis 锁**：性能之王，适合高并发场景，但需要自行保证可靠性
- **ZooKeeper 锁**：可靠卫士，适合强一致场景，但性能是瓶颈
- **etcd 锁**：云原生首选，在 Kubernetes 生态内无可替代

选型的本质是**权衡**：你愿意用可靠性换性能，还是用性能换可靠性？
