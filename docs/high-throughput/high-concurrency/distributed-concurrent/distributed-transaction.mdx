# 分布式事务（Seata、可靠消息、TCC 方案深度对比）

---

## 场景切入：为什么你的下单系统会"丢单"？

凌晨 2 点，你正准备上线新功能，测试反馈了一个诡异的问题：用户下单成功，库存扣了，但订单状态一直是"待支付"。你查了半天日志，发现问题出在两个服务的"跨服务调用"上——库存服务扣减成功，但订单服务插入数据时因为网络抖动失败了。问题是，库存已经扣了啊。

这就是分布式事务要解决的问题。

在单体架构时代，所有的数据操作都在同一个数据库中，事务是"天然的"——本地数据库帮你搞定 ACID。但当你把系统拆成微服务，每个服务独立数据库时，原本的"一句话搞定"变成了"跨服务协作"，事务的边界被彻底打破了。

你有没有想过：**分布式事务的核心矛盾是什么？为什么 CAP 定理决定了我们不可能同时满足强一致性和完美可用性？**

---

## 一、分布式事务的起源：从 ACID 到 BASE

### 1.1 单体架构：事务是"免费的"

在单体架构中，所有业务逻辑和数据都在同一个进程中，数据库事务就是"银弹"。你写一个下单方法：

```java
@Transactional
public void createOrder(OrderDTO orderDTO) {
    // 1. 创建订单
    orderRepository.save(order);
    // 2. 扣减库存
    inventoryService.decreaseStock(order.getSkuId(), order.getQuantity());
    // 3. 扣减余额
    accountService.decreaseBalance(order.getUserId(), order.getTotalAmount());
}
```

三个操作，要么同时成功，要么同时回滚。数据库的本地事务机制帮你处理了一切。

**但这种"免费午餐"的前提是：所有操作都在同一个数据库中。**

### 1.2 微服务架构：事务边界被打破

当你把系统拆成订单服务、库存服务、账户服务，每个服务独立部署、独立数据库时，情况变了：

- 用户下单请求 → 订单服务创建订单（写入订单库）
- 库存服务扣减库存（写入库存库）
- 账户服务扣减余额（写入账户库）

三个操作分布在三个不同的数据库中，本地事务机制管不了了。

**这就是分布式事务的起源：事务的参与者从"同一个数据库"变成了"跨网络的多个服务"。**

### 1.3 CAP 定理：分布式事务的理论边界

理解分布式事务，首先要理解 CAP 定理。CAP 三个字母分别代表：

| 特性 | 含义 |
|---|---|
| **C (Consistency)** | 强一致性：任何时刻，所有节点看到的数据都是一致的 |
| **A (Availability)** | 可用性：每个请求都能在合理时间内得到响应 |
| **P (Partition Tolerance)** | 分区容错：系统在网络分区时仍能运行 |

CAP 定理告诉我们：**在分布式系统中，最多只能同时满足其中两个特性。**

对于分布式事务来说，分区容错（P）是必须满足的——网络分区随时可能发生。那么剩下的选择就是：

- **CP 模式**：选择强一致性，在网络分区时拒绝服务（牺牲可用性）
- **AP 模式**：选择可用性，允许短暂的数据不一致（最终一致性）

绝大多数分布式事务方案，选择的都是 AP 模式——**不追求强一致，而是追求最终一致**。

### 1.4 BASE 理论：分布式事务的实践哲学

BASE 是对 CAP 中一致性和可用性权衡的实践总结：

| 字母 | 含义 | 说明 |
|---|---|---|
| **B (Basically Available)** | 基本可用 | 允许系统在故障时降级，但核心功能可用 |
| **S (Soft State)** | 软状态 | 允许数据存在中间状态，不要求强一致 |
| **E (Eventually Consistent)** | 最终一致 | 系统在一段时间后达到一致状态 |

BASE 理论的核心思想：**牺牲强一致，换取系统可用性。** 这也是绝大多数分布式事务方案的设计哲学。

## 二、X/Open 分布式事务模型：理解事务的抽象

### 2.1 三种角色：AP、TM、RM

X/Open 组织（现在的 The Open Group）定义了分布式事务处理的标准模型，核心角色有三个：

| 角色 | 全称 | 说明 |
|---|---|---|
| **AP (Application Program)** | 应用程序 | 发起分布式事务的业务应用 |
| **TM (Transaction Manager)** | 事务管理器 | 协调整个分布式事务的执行，负责提交或回滚 |
| **RM (Resource Manager)** | 资源管理器 | 管理分布式事务中的资源，如数据库、消息队列 |

```
┌─────────────────────────────────────────────────────────────────┐
│                        Transaction Manager (TM)                │
│                    协调者：决定提交还是回滚                      │
└─────────────────────────────────────────────────────────────────┘
        │                    │                    │
        ▼                    ▼                    ▼
┌─────────────┐      ┌─────────────┐      ┌─────────────┐
│  Resource   │      │  Resource   │      │  Resource   │
│  Manager 1  │      │  Manager 2  │      │  Manager 3  │
│  (订单库)   │      │  (库存库)   │      │  (账户库)   │
└─────────────┘      └─────────────┘      └─────────────┘
        ▲                    ▲                    ▲
        │                    │                    │
   ┌────┴────────────────────┴────────────────────┴────┐
   │              Application (AP)                      │
   │           发起分布式事务的业务逻辑                  │
   └───────────────────────────────────────────────────┘
```

### 2.2 两阶段提交协议（2PC）

2PC 是最经典的分布式事务协议，整个过程分为两个阶段：

**第一阶段：投票阶段（Prepare）**

TM 向所有 RM 发送 Prepare 请求，询问"是否可以提交"。

每个 RM 执行本地事务操作（锁定资源），但不提交，返回 YES 或 NO。

**第二阶段：提交阶段（Commit）**

- 如果所有 RM 都返回 YES，TM 向所有 RM 发送 Commit 请求，所有 RM 提交本地事务。
- 如果任何一个 RM 返回 NO，TM 向所有 RM 发送 Rollback 请求，所有 RM 回滚本地事务。

```java
// 2PC 的简化实现
public class TwoPhaseCommit {

    // 第一阶段：投票
    public boolean prepare() {
        boolean orderPrepared = orderService.prepare();
        boolean inventoryPrepared = inventoryService.prepare();
        boolean accountPrepared = accountService.prepare();

        return orderPrepared && inventoryPrepared && accountPrepared;
    }

    // 第二阶段：提交
    public void commit() {
        orderService.commit();
        inventoryService.commit();
        accountService.commit();
    }

    // 第二阶段：回滚
    public void rollback() {
        orderService.rollback();
        inventoryService.rollback();
        accountService.rollback();
    }
}
```

### 2.3 2PC 的三大问题

2PC 看起来很完美，但有三个致命问题：

**问题一：同步阻塞**

在 Prepare 阶段，所有 RM 都会锁定本地资源。在 TM 发出 Commit 之前，这些资源一直被锁定。

假设有 100 个 RM 都卡在 Prepare 阶段，那么整个系统都处于阻塞状态，其他业务无法使用这些资源。

**问题二：单点故障**

TM 是整个系统的核心，如果 TM 在 Commit 阶段宕机，RM 会处于"等待指令"的状态：

- 部分 RM 已经收到 Commit 并执行了提交
- 部分 RM 还在等待 Commit 指令
- 系统处于"不一致"状态

如果 TM 恢复，需要从日志中恢复事务状态，但日志本身也可能损坏。

**问题三：数据不一致**

当 TM 发出 Commit 请求时，部分 RM 可能因为网络问题没有收到 Commit：

- RM A 收到 Commit，提交成功
- RM B 没有收到 Commit（网络丢包），一直等待
- 最终 RM B 超时回滚

结果是：RM A 提交了，RM B 回滚了，数据彻底不一致。

## 三、三阶段提交协议（3PC）：2PC 的改进

### 3.1 3PC 的核心改进

3PC 在 2PC 的基础上引入了两个改进：

**改进一：增加超时机制**

2PC 中，RM 在 Prepare 阶段会无限等待。3PC 中，RM 增加了超时机制，避免无限阻塞。

**改进二：分阶段确认**

3PC 把 2PC 的投票阶段拆成两个阶段：

| 阶段 | 说明 |
|---|---|
| **CanCommit** | TM 询问 RM 是否有条件提交（不执行实际操作） |
| **PreCommit** | RM 确认可以提交，TM 让所有 RM 执行 Prepare（锁定资源） |
| **DoCommit** | TM 发送真正的 Commit，让所有 RM 提交事务 |

### 3.2 3PC 的执行流程

```
                    TM                        RM
                     │                         │
        ┌────────────┴────────────┐             │
        │                       │             │
   CanCommit?              CanCommit?      CanCommit?
        │                       │             │
        ▼                       ▼             ▼
   ┌─────────┐              ┌─────────┐   ┌─────────┐
   │  timeout│              │  timeout│   │  timeout│
   │  abort  │              │  abort  │   │  abort  │
   └─────────┘              └─────────┘   └─────────┘
        │                       │             │
   ┌────┴────────────────────────┴─────────────┐
   │                                            │
   │              PreCommit                     │
   └────────────────────────────────────────────┘
        │                                        │
        ├──────────────────┐                     │
        ▼                  ▼                     ▼
   ┌─────────┐        ┌─────────┐          ┌─────────┐
   │ ack YES │        │ ack YES │          │ ack YES │
   └────┬────┘        └────┬────┘          └────┬────┘
        │                  │                     │
        └──────────────────┴─────────────────────┘
                             │
                      DoCommit (or Abort)
```

### 3.3 3PC 依然存在的问题

3PC 改进了 2PC 的部分问题，但依然无法完全解决数据不一致：

**问题一：RM 在 PreCommit 后宕机**

假设 RM A 执行了 PreCommit（锁定资源），但没有收到 DoCommit 就宕机了。RM A 会超时自动提交还是回滚？

如果选择提交，可能与其他 RM 不一致；如果选择回滚，与 TM 的决策冲突。

**问题二：网络分区导致数据不一致**

假设在 PreCommit 阶段，TM 发送 DoCommit 时发生了网络分区：

- 部分 RM 收到了 DoCommit，提交成功
- 部分 RM 没有收到 DoCommit，超时后回滚

数据依然不一致。

**3PC 本质上是"乐观"的解决方案**——假设网络分区不会发生，或者发生概率很低。但在分布式系统中，网络分区是常态，不是异常。

## 四、TCC 模式：业务层面的事务补偿

### 4.1 TCC 的核心思想

TCC（Try-Confirm-Cancel）是一种**业务层面的分布式事务实现**，核心思想是"**将两阶段提交上升到业务层面**"。

与 2PC 不同，TCC 的每个阶段都是业务层面的操作：

| 阶段 | 说明 | 对应 2PC |
|---|---|---|
| **Try** | 预留资源，锁定检查 | Prepare 阶段 |
| **Confirm** | 确认执行，真正扣减资源 | Commit 阶段 |
| **Cancel** | 取消预留，释放资源 | Rollback 阶段 |

### 4.2 TCC 的执行流程

以典型的"下单扣库存"场景为例：

```java
// TCC 分布式事务示例
@Service
public class OrderTccService {

    @Autowired
    private InventoryFeignClient inventoryFeignClient;

    @Autowired
    private OrderRepository orderRepository;

    /**
     * Try 阶段：预留资源
     * - 检查库存是否充足
     * - 冻结库存（而非直接扣减）
     */
    @TwoPhaseBusiness
    public boolean try扣库存(Long skuId, Integer quantity) {
        // 冻结库存：status = 'FROZEN'
        return inventoryFeignClient.freezeStock(skuId, quantity);
    }

    /**
     * Confirm 阶段：确认执行
     * - 真正扣减冻结的库存
     */
    public boolean confirm扣库存(Long skuId, Integer quantity) {
        // 确认扣减：冻结 -> 已扣减
        return inventoryFeignClient.confirmDecreaseStock(skuId, quantity);
    }

    /**
     * Cancel 阶段：取消预留
     * - 释放冻结的库存
     */
    public boolean cancel扣库存(Long skuId, Integer quantity) {
        // 释放冻结：冻结 -> 可用
        return inventoryFeignClient.cancelFreezeStock(skuId, quantity);
    }
}
```

**TCC vs 2PC 的核心区别**：

| 对比项 | 2PC | TCC |
|---|---|---|
| 层级 | 数据库层面 | 业务层面 |
| 锁定方式 | 数据库行锁 | 业务冻结字段 |
| 资源占用 | 长事务锁 | 短事务锁 |
| 性能 | 差 | 好 |
| 侵入性 | 低 | 高（需要业务改造） |

### 4.3 TCC 的三大问题

TCC 模式虽然性能好，但引入了三个业务层面的问题：

**问题一：空回滚**

空回滚是指：**Try 方法没有执行，但 Cancel 方法执行了。**

场景：TM 在发送 Try 请求时，部分 RM 没有收到（网络丢包），RM 的 Try 根本没有执行。但 TM 认为 Try 失败了，发送 Cancel 请求。RM 收到 Cancel 时，发现自己根本没有冻结资源，回滚什么？

解决方案：RM 需要记录分布式事务的状态，根据状态决定是否执行 Cancel。

```java
// 空回滚防护
public boolean cancel(Long skuId, Integer quantity) {
    // 查询分布式事务日志
    TransactionLog log = transactionLogRepository.findByXid(xid);

    // 如果 Try 根本没有执行，说明是空回滚
    if (log == null || log.getStatus() != TransactionStatus.TRY_SUCCESS) {
        // 记录空回滚，但不做实际回滚操作
        logEmptyRollback(xid);
        return true;
    }

    // 正常回滚：释放冻结的库存
    return inventoryService.releaseFrozenStock(skuId, quantity);
}
```

**问题二：幂等性**

幂等性是指：**同一个操作执行多次，结果相同。**

在分布式事务中，由于网络重试，Confirm/Cancel 可能被重复调用。如果不做幂等处理，会导致资源被重复扣减或重复释放。

解决方案：每个操作必须携带分布式事务 ID（XID），RM 记录每个 XID 的执行状态，防止重复执行。

```java
// 幂等性防护
public boolean confirm(Long skuId, Integer quantity) {
    // 查询是否已经执行过 Confirm
    TransactionLog log = transactionLogRepository.findByXidAndAction(xid, "CONFIRM");

    if (log != null) {
        // 已经执行过，直接返回成功
        return true;
    }

    // 执行真正的 Confirm 操作
    boolean result = inventoryService.confirmDecreaseStock(skuId, quantity);

    // 记录执行状态
    saveTransactionLog(xid, "CONFIRM", result);

    return result;
}
```

**问题三：悬挂**

悬挂是指：**Cancel 方法比 Try 方法先执行。**

场景：RM 收到 Try 请求后，由于网络问题，Try 执行超时。TM 认为 Try 失败，发送 Cancel 请求。但此时 Try 请求才刚刚到达 RM，Cancel 会先于 Try 执行，导致资源被错误释放。

解决方案：RM 在执行任何操作前，先检查当前状态，确保不会"越界"。

```java
// 悬挂防护
public boolean cancel(Long skuId, Integer quantity) {
    TransactionLog log = transactionLogRepository.findByXidAndAction(xid, "TRY");

    // 如果 Try 根本没有执行过，说明是悬挂（Cancel 先于 Try）
    if (log == null) {
        // 记录悬挂状态
        logSuspension(xid);
        return true;
    }

    // 正常回滚
    return inventoryService.releaseFrozenStock(skuId, quantity);
}
```

## 五、可靠消息最终一致性方案：异步化的妥协

### 5.1 核心思想：消息驱动的事务

可靠消息最终一致性方案的核心思想是：**用消息队列来"记录"业务操作，通过消息的可靠投递来实现最终一致性。**

与 TCC 相比，可靠消息方案不需要业务改造，而是通过消息队列来协调分布式事务。

### 5.2 本地消息表：最朴素的实现

本地消息表是可靠消息方案的最早实现，核心思路是：**把消息存储在业务数据库中，利用本地事务保证消息和业务的原子性。**

```sql
-- 本地消息表结构
CREATE TABLE local_message (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    transaction_id VARCHAR(64),        -- 分布式事务 ID
    message_content TEXT,               -- 消息内容
    message_status INT,                 -- 消息状态：0-待发送，1-已发送，2-已确认
    retry_count INT DEFAULT 0,          -- 重试次数
    create_time DATETIME,
    update_time DATETIME
);
```

```java
// 本地消息表实现
@Service
public class OrderService {

    @Autowired
    private OrderRepository orderRepository;

    @Autowired
    private LocalMessageMapper localMessageMapper;

    @Autowired
    private MessageProducer messageProducer;

    /**
     * 创建订单（本地事务）
     */
    @Transactional
    public void createOrder(OrderDTO orderDTO) {
        // 1. 创建订单
        Order order = new Order();
        order.setUserId(orderDTO.getUserId());
        order.setTotalAmount(orderDTO.getTotalAmount());
        order.setStatus("PENDING");
        orderRepository.save(order);

        // 2. 记录消息（与订单在同一个事务中）
        LocalMessage message = new LocalMessage();
        message.setTransactionId(getXid());
        message.setMessageContent(JSON.toJSONString(orderDTO));
        message.setMessageStatus(0);  // 待发送
        localMessageMapper.insert(message);

        // 3. 事务提交后，消息一定存在于本地表中
        // 事务回滚，消息也被回滚
    }

    /**
     * 定时扫描并发送消息
     */
    @Scheduled
    public void sendPendingMessages() {
        List<LocalMessage> messages = localMessageMapper.findPendingMessages();

        for (LocalMessage message : messages) {
            try {
                // 1. 发送消息到 MQ
                messageProducer.send(message.getMessageContent());

                // 2. 更新消息状态为已发送
                message.setMessageStatus(1);
                localMessageMapper.update(message);

            } catch (Exception e) {
                // 发送失败，增加重试次数
                message.setRetryCount(message.getRetryCount() + 1);
                localMessageMapper.update(message);
            }
        }
    }
}
```

**本地消息表的核心优势**：实现简单，利用本地事务保证了业务和消息的原子性。

**本地消息表的核心缺陷**：消息状态需要轮询扫描，增加延迟；消息表和业务表耦合。

### 5.3 RocketMQ 事务消息：更优雅的实现

RocketMQ 4.3+ 提供了事务消息功能，核心思想是：**把消息发送本身纳入分布式事务，通过 Half Message 机制实现。**

```java
// RocketMQ 事务消息实现
@Service
public class OrderService {

    @Autowired
    private TransactionMQProducer transactionProducer;

    /**
     * 创建订单（半消息）
     */
    public void createOrderTransactional(OrderDTO orderDTO) {
        // 1. 发送半消息（此时消息对消费者不可见）
        TransactionSendResult result = transactionProducer.sendMessageInTransaction(
            "order-topic",
            orderDTO,
            new TransactionListener() {
                /**
                 * 执行本地事务
                 */
                @Override
                public LocalTransactionState executeLocalTransaction(
                        Message msg, Object arg) {

                    try {
                        // 1. 创建订单
                        Order order = createOrder(orderDTO);

                        // 2. 扣减库存
                        inventoryService.decreaseStock(orderDTO.getSkuId(),
                                                      orderDTO.getQuantity());

                        // 本地事务成功，返回 COMMIT
                        return LocalTransactionState.COMMIT_MESSAGE;

                    } catch (Exception e) {
                        // 本地事务失败，回滚
                        return LocalTransactionState.ROLLBACK_MESSAGE;
                    }
                }

                /**
                 * 检查本地事务状态（用于回查）
                 */
                @Override
                public LocalTransactionState checkLocalTransaction(MessageExt msg) {
                    // 查询本地事务是否成功
                    String transactionId = msg.getTransactionId();
                    TransactionLog log = transactionLogMapper.findByTransactionId(transactionId);

                    if (log != null && log.getStatus() == "SUCCESS") {
                        return LocalTransactionState.COMMIT_MESSAGE;
                    } else {
                        return LocalTransactionState.ROLLBACK_MESSAGE;
                    }
                }
            }
        );
    }
}
```

**RocketMQ 事务消息的执行流程**：

```
1. Producer 发送 Half Message（半消息）
       │
       ▼
2. Broker 存储半消息（对 Consumer 不可见）
       │
       ▼
3. Producer 执行本地事务
       │
       ├── 成功 ──→ 发送 Commit/Rollback
       │              │
       │              ▼
       │         Commit ──→ 消息对 Consumer 可见
       │              │
       │              ▼
       │         Rollback ──→ 消息被删除
       │
       └── 失败 ──→ 发送 Rollback
                      │
                      ▼
                 消息被删除

4. 如果 Producer 没有响应（超时），Broker 回查 TransactionListener
       │
       ▼
5. 根据回查结果决定 Commit 或 Rollback
```

### 5.4 可靠消息方案 vs TCC

| 对比项 | 可靠消息 | TCC |
|---|---|---|
| 实现复杂度 | 中等 | 高 |
| 业务侵入性 | 低 | 高（需要改造业务） |
| 性能 | 好 | 最好 |
| 实时性 | 准实时（有延迟） | 实时 |
| 数据一致性 | 最终一致 | 最终一致 |
| 适用场景 | 异步化场景、对延迟不敏感 | 强实时场景 |

## 六、Seata： AT 模式的自动补偿

### 6.1 Seata 简介

Seata（Simple Extensible Autonomous Transaction Architecture）是阿里巴巴开源的分布式事务解决方案，核心提供了四种事务模式：

| 模式 | 说明 | 业务侵入性 |
|---|---|---|
| **AT** | 自动补偿模式，基于undo log | 低 |
| **TCC** | Try-Confirm-Cancel | 高 |
| **SAGA** | 长事务模式 | 中 |
| **XA** | XA 协议模式 | 低 |

### 6.2 AT 模式：自动补偿的核心原理

AT 模式的核心思想是：**在每个业务数据库上记录 undo log，由 TC（Transaction Coordinator）统一管理全局事务。**

```xml
<!-- Seata AT 模式配置 -->
<bean id="dataSource" class="com.alibaba.druid.pool.DruidDataSource">
    <property name="url" value="jdbc:mysql://localhost:3306/order_db"/>
    <property name="username" value="root"/>
    <property name="password" value="password"/>
</bean>

<!-- 配置 Seata DataSource Proxy -->
<bean id="proxyDataSource" class="io.seata.rm.datasource.DataSourceProxy">
    <constructor-arg ref="dataSource"/>
</bean>
```

```java
// Seata AT 模式使用示例
@GlobalTransactional  // 标记全局事务入口
public void createOrder(OrderDTO orderDTO) {

    // 1. 创建订单（自动被 Seata 管理）
    Order order = new Order();
    order.setUserId(orderDTO.getUserId());
    order.setTotalAmount(orderDTO.getTotalAmount());
    orderRepository.save(order);

    // 2. 扣减库存（远程调用，被 Seata 管理）
    inventoryFeignClient.decreaseStock(orderDTO.getSkuId(), orderDTO.getQuantity());

    // 3. 扣减余额（远程调用，被 Seata 管理）
    accountFeignClient.decreaseBalance(orderDTO.getUserId(), orderDTO.getTotalAmount());

    // 整个方法在一个全局事务中
    // 任何一个操作失败，所有操作自动回滚
}
```

### 6.3 AT 模式的工作原理

**阶段一：执行阶段**

TM 向所有 RM 发送执行请求，每个 RM 执行 SQL 并记录 undo log（回滚日志）。

```sql
-- 原始 SQL
UPDATE inventory SET stock = stock - 10 WHERE sku_id = 'SKU001';

-- Seata 自动生成的 undo log
{
    "tableName": "inventory",
    "beforeImage": [{"stock": 100}],
    "afterImage": [{"stock": 90}],
    "sqlType": "UPDATE",
    "condition": "sku_id = 'SKU001'"
}
```

**阶段二：提交阶段**

TM 向所有 RM 发送提交请求，RM 删除对应的 undo log。

**回滚时**：RM 根据 undo log 恢复数据到执行前的状态。

### 6.4 AT vs TCC vs Saga

| 对比项 | AT | TCC | Saga |
|---|---|---|---|
| 业务侵入性 | 低 | 高 | 中 |
| 性能 | 好 | 最好 | 好 |
| 全局锁 | 需要（锁行） | 不需要 | 不需要 |
| 回滚方式 | 自动补偿 | 业务补偿 | 补偿服务链 |
| 适用场景 | 普通业务场景 | 高性能场景 | 长事务场景 |
| 异常处理 | 自动 | 需要处理空回滚、幂等、悬挂 | 补偿服务编排 |

## 七、实战选型指南

### 7.1 选型决策树

面对不同的业务场景，如何选择分布式事务方案？

```
业务场景
    │
    ├── 是否强一致性要求？
    │       │
    │       ├── 是 ──→ 金融交易、账户转账 ──→ Seata AT / 2PC XA
    │       │
    │       └── 否 ──→ 继续判断
    │               │
    │               ├── 性能要求极高？
    │               │       │
    │               │       ├── 是 ──→ TCC
    │               │       │
    │               │       └── 否 ──→ 继续判断
    │               │
    │               └── 异步化是否可接受？
    │                       │
    │                       ├── 是 ──→ RocketMQ 事务消息 / 本地消息表
    │                       │
    │                       └── 否 ──→ Seata AT
```

### 7.2 方案对比总结

| 方案 | 一致性 | 性能 | 侵入性 | 复杂度 | 适用场景 |
|---|---|---|---|---|---|
| 2PC XA | 强一致 | 差 | 低 | 中 | 强一致性要求场景 |
| 3PC | 强一致 | 差 | 低 | 高 | 已被淘汰 |
| TCC | 最终一致 | 好 | 高 | 高 | 高性能、强实时场景 |
| 本地消息表 | 最终一致 | 中 | 低 | 低 | 异步化场景 |
| RocketMQ 事务消息 | 最终一致 | 好 | 低 | 中 | 异步化、强实时场景 |
| Seata AT | 最终一致 | 好 | 低 | 低 | 普通业务场景 |
| Seata Saga | 最终一致 | 好 | 中 | 中 | 长事务场景 |

### 7.3 实战经验

**经验一：优先考虑业务妥协，而非技术方案**

很多业务场景可以通过"业务妥协"来避免分布式事务。比如：

- 异步扣款：先下单，后扣款，通过消息队列解耦
- 补偿机制：允许最终不一致，通过定时任务补偿
- 幂等设计：把"不可重复"变成"可重复"

**经验二：避免分布式事务的最好方式是避免跨库调用**

在设计微服务架构时，尽量避免"跨库关联查询"和"跨库事务"。如果必须跨库，优先考虑：

- CQRS（命令查询职责分离）：写在一处，读在另一处
- 事件驱动：用消息队列解耦
- 聚合根：用领域驱动设计（DDD）的聚合根来封装事务边界

分布式事务是微服务架构的"阿喀琉斯之踵"——没有完美的解决方案，只有在一致性、可用性、性能之间的权衡。

- 如何设计一个"最终一致性"优先的系统？
