# 分布式高并发核心：突破单机瓶颈

> [!WARNING]
> 前置知识：本篇文章是分布式高并发核心的入口，涵盖分布式理论、分布式锁、分布式事务、分库分表等方向。建议按顺序阅读，理解分布式系统的核心挑战与解决方案。

当单机无法满足业务需求时，你需要扩展到多机。

但分布式不是银弹。它带来了 CAP 定理、一致性问题、分布式事务等新挑战。一个在单机环境下简单的问题，在分布式环境下可能变得极其复杂。

比如，单机环境下的 synchronized 锁，在分布式环境下完全失效。你需要分布式锁。
比如，单机环境下的数据库事务，在分布式环境下无法保证。你需要分布式事务。

理解这些问题的本质，才能设计出可靠的分布式系统。

这一讲，我们来掌握分布式高并发的核心知识。

## 二、分布式理论基础：CAP 与 BASE

### CAP 定理

CAP 定理指出：一个分布式系统不可能同时满足一致性（Consistency）、可用性（Availability）和分区容错性（Partition Tolerance）。

```text
┌─────────────────────────────────────────────────────────────────┐
│                        CAP 三角                                   │
│                                                                 │
│                        Consistency                               │
│                            ▲                                     │
│                           ╱ ╲                                    │
│                          ╱   ╲                                   │
│                         ╱     ╲                                  │
│                        ╱   P   ╲                                 │
│                       ╱         ╲                                │
│                      ╱───────────╲                               │
│                    A ◄───────────► C                             │
│                                                                 │
│   在网络分区（P）发生时，必须在 A 和 C 之间二选一                  │
└─────────────────────────────────────────────────────────────────┘
```

| 选择 | 说明 | 代表系统 |
|---|---|---|
| 选择 C | 保证一致性，网络分区时牺牲可用性 | ZooKeeper、etcd、HBase |
| 选择 A | 保证可用性，网络分区时牺牲一致性 | Cassandra、DynamoDB |
| 选择 P | 优先保证分区容错，在 A 和 C 之间权衡 | 大多数分布式数据库 |

### BASE 理论

BASE 理论是对 CAP 中一致性和可用性权衡的总结：

| 特性 | 说明 |
|---|---|
| Basically Available | 保障核心功能可用，允许降级 |
| Soft state | 状态可以是松散的，允许中间状态 |
| Eventually consistent | 最终一致性，不需要实时一致 |

> [!TIP]
> BASE 理论是大多数分布式系统的设计基础。电商秒杀系统、金融支付系统，都是在 BASE 理论指导下设计的。

## 三、分布式锁：跨节点互斥的核心

### 分布式锁的三要素

一个可靠的分布式锁必须满足：

| 要素 | 说明 | 实现方式 |
|---|---|---|
| 互斥 | 同一时刻只有一个客户端持有锁 | SETNX |
| 防死锁 | 锁必须能自动释放，防止死锁 | TTL（过期时间） |
| 防抢锁 | 持有锁的客户端才能释放锁 | 唯一 value |

### Redis 分布式锁的实现

```java
public class RedisDistributedLock {

    private final RedisTemplate<String, String> redisTemplate;

    // 1. 加锁
    public Boolean tryLock(String lockKey, String lockValue, long expireTime) {
        // SET key value NX PX milliseconds
        return redisTemplate.opsForValue()
            .setIfAbsent(lockKey, lockValue, expireTime, TimeUnit.SECONDS);
    }

    // 2. 释放锁（防抢锁）
    public Boolean releaseLock(String lockKey, String lockValue) {
        String script = "if redis.call('get', KEYS[1]) == ARGV[1] then " +
                        "return redis.call('del', KEYS[1]) " +
                        "else return 0 end";
        return redisTemplate.execute(
            new DefaultRedisScript<>(script, Boolean.class),
            Collections.singletonList(lockKey),
            lockValue
        );
    }
}
```

### Redisson：带看门狗的分布式锁

普通 Redis 分布式锁的问题是：如果持有锁的进程在锁过期前崩溃，锁会自动释放，导致其他进程也能获取锁。

Redisson 通过**看门狗机制**（Watchdog）自动续期，解决了这个问题：

```java
// Redisson 自动续期
RLock lock = redissonClient.getLock("order:lock:" + orderId);
try {
    // 等待锁、最长等待 30 秒、锁自动过期时间 60 秒
    // 如果业务执行超过 60 秒，看门狗会自动续期
    lock.tryLock(30, 60, TimeUnit.SECONDS);

    // 执行业务
    doSomething();

} finally {
    lock.unlock();  // 释放锁
}
```

## 四、分布式事务：跨节点数据一致性

分布式事务是分布式系统中最复杂的问题之一。CAP 定理告诉我们，强一致性、可用性、分区容错无法同时满足。

### 常见方案对比

| 方案 | 一致性强度 | 性能 | 复杂度 | 侵入性 | 适用场景 |
|---|---|---|---|---|---|
| 两阶段提交（2PC） | 强一致 | 低 | 高 | 高 | 对一致性要求极高的场景 |
| 三阶段提交（3PC） | 强一致 | 中 | 高 | 高 | 较少使用 |
| TCC | 最终一致 | 中 | 高 | 高 | 需要业务配合 |
| 本地消息表 | 最终一致 | 高 | 中 | 中 | 最常用 |
| Seata AT | 最终一致 | 高 | 中 | 低 | 推荐使用 |
| Saga | 最终一致 | 高 | 中 | 高 | 长事务 |

### Seata AT 模式

Seata AT 是目前最流行的分布式事务解决方案：

```java
// 1. 开启全局事务
@GlobalTransactional
public void placeOrder(OrderDTO orderDTO) {
    // 扣减库存
    inventoryFeignClient.deduct(orderDTO.getProductId(), orderDTO.getCount());

    // 扣减余额
    accountFeignClient.deduct(orderDTO.getUserId(), orderDTO.getAmount());

    // 创建订单
    Order order = new Order();
    order.setUserId(orderDTO.getUserId());
    order.setAmount(orderDTO.getAmount());
    orderRepository.save(order);
}
```

> [!TIP]
> Seata AT 模式对业务零侵入，通过解析 SQL 记录反向 SQL 实现自动补偿。适用于大多数业务场景。

## 五、分库分表：水平扩展的终极方案

### 分库分表的问题

当单库数据量超过千万级时，需要考虑分库分表。但分库分表带来了新问题：

| 问题 | 说明 | 解决方案 |
|---|---|---|
| 跨表查询 | 分片键不在查询条件中，无法定位到具体的库/表 | 数据异构（ES）、冗余查询 |
| 跨表分页 | 分页查询需要聚合多个分片的结果 | 游标分页、禁止深分页 |
| 分片键选择 | 分片键一旦选定，很难更改 | 按业务高频查询选择 |
| ID 唯一性 | 自增 ID 在分片后不唯一 | 分布式 ID（雪花算法） |

### 分片键的选择

分片键的选择至关重要，一旦选择，就很难更改：

| 分片键 | 优点 | 缺点 |
|---|---|---|
| 用户 ID | 用户维度查询快 | 用户数据不均匀（某些用户数据量极大） |
| 时间 | 时序查询快 | 热点数据集中（当前数据访问量大） |
| 地区 | 地理位置查询快 | 地区数据不均匀 |

### 分布式 ID 生成

分库分表后，需要分布式 ID 来保证唯一性：

| 方案 | 特点 | 适用场景 |
|---|---|---|
| UUID | 无中心、简单 | 不推荐（无序、存储大） |
| 数据库自增 | 有序、简单 | 单库单表 |
| 雪花算法 | 高性能、有序 | 推荐 |
| Redis INCR | 简单 | 需要考虑 Redis 持久化 |

## 延伸思考

分布式系统的学习，本质上是在理解**「协作」和「权衡」**。

单机系统是「一个人干活」，分布式系统是「一群人协作」。协作需要协议、需要共识、需要权衡。CAP 定理是权衡的艺术，分布式锁是协作的协议，分布式事务是一致性的代价。

下次设计分布式系统之前，先问自己：**我的业务能接受多大的一致性延迟？我需要跨节点协作吗？我的瓶颈在哪里？**
