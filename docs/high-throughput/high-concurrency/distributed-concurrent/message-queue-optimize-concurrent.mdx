# 消息队列高并发优化（分区/集群/积压处理）

想象一个场景：双十一零点，你准备了 100 台服务器的集群来处理订单。

但问题来了——这 100 台服务器同时收到下单请求，每台都在疯狂处理数据库操作。结果呢？数据库连接池被瞬间耗尽，磁盘 I/O 飙升，整个系统反而比单机时更慢。

**这就是没有消息队列的高并发系统的典型死法。**

而消息队列的价值，就是把这 100 台服务器从「各自为战」变成「有序协作」：订单请求先进入队列，后台消费者按自己的节奏慢慢处理，系统稳如老狗。

但消息队列真的能解决一切问题吗？分区数怎么设置？消息积压了怎么办？如何保证消息不丢失？这些问题，才是真正考验功底的地方。


## 为什么消息队列是高并发系统的核心

### 没有消息队列的系统

```
没有消息队列的架构：

用户请求
    ↓
┌─────────────────────────────────────────────┐
│  订单服务（同步处理）                        │
│  - 接收请求                                  │
│  - 扣减库存                                  │
│  - 创建订单                                  │
│  - 发送通知                                  │
│  - 返回结果                                  │
└─────────────────────────────────────────────┘
    ↓
返回响应

问题：
- 每一步都必须等待完成才能进行下一步
- 任何一个环节慢，整个系统就慢
- 峰值时直接压垮下游服务
```

### 有消息队列的系统

```
有消息队列的架构：

用户请求
    ↓
┌─────────────────────────────────────────────┐
│  订单服务（只做核心校验）                    │
│  - 接收请求                                  │
│  - 快速校验                                  │
│  - 发送消息到队列                            │
│  - 立即返回                                  │
└─────────────────────────────────────────────┘
    ↓
消息队列
    ↓
┌─────────────────────────────────────────────┐
│  后台消费者（异步处理）                      │
│  - 扣减库存                                  │
│  - 创建订单                                  │
│  - 发送通知                                  │
│  - 发送短信                                  │
│  - 更新统计                                  │
└─────────────────────────────────────────────┘
```

## 消息队列的三大核心作用

### 异步处理：让请求飞起来

同步处理和异步处理的对比：

```
同步处理时间线：
t=0ms    请求到达
t=10ms   库存校验完成
t=50ms   订单创建完成
t=100ms  发送通知完成
t=150ms  发送短信完成
t=200ms  返回结果给用户
总耗时：200ms

异步处理时间线：
t=0ms    请求到达
t=5ms    核心校验完成
t=10ms   消息发送完成，返回结果给用户
总耗时：10ms（用户体验）
```

后台处理时间线（对用户不可见）：

```
异步处理（后台）：
t=0ms    收到消息
t=5ms    扣减库存完成
t=10ms   创建订单完成
t=15ms   发送通知完成
t=20ms   发送短信完成
总耗时：20ms
```

### 流量削峰：保护下游服务

```
削峰前后对比：

削峰前：
峰值 100 万 QPS → 直接打满数据库 → 系统崩溃

削峰后：
峰值 100 万 QPS → 消息队列缓冲 → 消费者按 10 万 QPS 处理
                                          ↓
                                   系统稳定运行
```

削峰的关键参数：

| 参数 | 说明 | 影响 |
|------|------|------|
| 队列容量 | 消息最大堆积数量 | 容量不足会丢消息 |
| 消费速率 | 每秒处理的消息数 | 决定系统吞吐量 |
| 积压告警 | 队列堆积多少报警 | 提前发现问题 |

### 系统解耦：让模块独立演化

```
紧耦合系统的问题：

┌────────────┐    ┌────────────┐    ┌────────────┐
│  订单系统  │ →  │  库存系统  │ →  │  物流系统  │
└────────────┘    └────────────┘    └────────────┘

问题：
- 订单系统必须知道库存系统的地址
- 库存系统升级，订单系统可能需要改代码
- 任何一个系统挂了，全部受影响
```

```
解耦后的系统：

┌────────────┐    ┌────────────┐    ┌────────────┐
│  订单系统  │ →  │  消息队列  │    │  库存系统  │
└────────────┘    └────────────┘ ← └────────────┘
                       ↓                 ↑
                  ┌────────────┐    ┌────────────┐
                  │  物流系统  │    │  通知系统  │
                  └────────────┘    └────────────┘

优势：
- 订单系统只管发消息，不关心谁消费
- 库存系统想升级就升级，不影响订单系统
- 任何一个系统挂了，不影响其他系统
```

## Kafka 的高并发设计

### Kafka 为什么这么快

Kafka 是消息队列中的性能标杆，它的快来自于以下几个核心设计：

```
Kafka 高性能的核心：

【顺序写磁盘】
- 追加写模式，磁盘顺序读取速度接近内存
- 写入速度：600 MB/s（普通磁盘）

【页缓存】
- 使用操作系统的页缓存，不自己管理
- 读取时先查缓存，命中则不需要磁盘 I/O

【零拷贝】
- DMA 直接从磁盘到网卡，省去 CPU 拷贝
- 传统方式：磁盘 → 内核缓冲区 → 用户缓冲区 → Socket 缓冲区 → 网卡
- 零拷贝：磁盘 → 内核缓冲区 → Socket 缓冲区 → 网卡

【批量处理】
- 消息批量发送，减少网络往返
- 批量压缩，减少网络带宽

【分区分片】
- 消息分散到多个分区
- 并行消费，充分利用多核
```

### 顺序写的原理

```
磁盘 I/O 的性能对比：

随机写：     0.5 MB/s（机械硬盘）
顺序写：     600 MB/s（机械硬盘）
顺序写：     几 GB/s（SSD）

Kafka 为什么能顺序写？

传统数据库的磁盘操作：
写操作 1：写入位置 A（磁盘臂移动到 A）
写操作 2：写入位置 B（磁盘臂移动到 B）
写操作 3：写入位置 C（磁盘臂移动到 C）
每次都是随机写

Kafka 的磁盘操作：
写操作 1：追加到文件末尾（磁盘臂在末尾）
写操作 2：追加到文件末尾（磁盘臂在末尾）
写操作 3：追加到文件末尾（磁盘臂在末尾）
永远只是顺序追加
```

### 零拷贝技术详解

```java
// 传统方式：需要 4 次拷贝

// 第一次拷贝：从磁盘到内核缓冲区（read() 系统调用）
// 第二次拷贝：从内核缓冲区到用户缓冲区（CPU 拷贝）
// 第三次拷贝：从用户缓冲区到 Socket 缓冲区（CPU 拷贝）
// 第四次拷贝：从 Socket 缓冲区到网卡（DMA 拷贝）

public class TraditionalFileSend {
    public void sendFile(File file, Socket socket) throws Exception {
        // 读取文件到用户缓冲区
        byte[] buffer = new byte[BUFFER_SIZE];
        FileInputStream fis = new FileInputStream(file);
        int bytesRead = fis.read(buffer);
        
        // 发送到 Socket
        socket.getOutputStream().write(buffer, 0, bytesRead);
        // 这里发生了多次 CPU 拷贝
    }
}

// 零拷贝：只需要 2 次拷贝

public class ZeroCopyFileSend {
    public void sendFile(File file, Socket socket) throws Exception {
        FileChannel channel = new FileInputStream(file).getChannel();
        // 直接从内核缓冲区到网卡，CPU 不参与
        channel.transferTo(0, channel.size(), socket.getChannel());
    }
}
```

### 批量处理与压缩

```
Kafka 的批量机制：

【生产端批量】
- 消息不是一条一条发，而是攒一批再发
- batch.size: 16KB（默认）
- linger.ms: 0（默认，攒满就发）

【压缩】
- 支持 snappy、gzip、lz4、zstd 压缩算法
- 压缩比：通常 2-5 倍
- 减少网络带宽，提升吞吐量

【消费端批量】
- 一次拉取多条消息
- max.poll.records: 500（默认）
- 减少网络往返，提高效率
```

## 分区机制：并行消费的核心

### 分区的概念

```
Kafka Topic 的结构：

Topic: order-events
    ↓
┌─────────────────────────────────────────────────────────┐
│  Partition 0  │  Partition 1  │  Partition 2           │
│  [msg1,msg4]   │  [msg2,msg5]   │  [msg3,msg6]          │
│  Broker 0      │  Broker 1      │  Broker 2              │
└─────────────────────────────────────────────────────────┘
```

### 分区与消费者的关系

```
分区与消费者组的关系：

消费者组 A:
┌─────────────┐
│  Consumer 1│ → 消费 Partition 0
└─────────────┘
┌─────────────┐
│  Consumer 2│ → 消费 Partition 1
└─────────────┘
┌─────────────┐
│  Consumer 3│ → 消费 Partition 2
└─────────────┘

消费者组 B:
┌─────────────┐
│  Consumer 4│ → 消费 Partition 0（独立消费）
└─────────────┘
┌─────────────┐
│  Consumer 5│ → 消费 Partition 1（独立消费）
└─────────────┘
┌─────────────┐
│  Consumer 6│ → 消费 Partition 2（独立消费）
└─────────────┘

注意：
- 同一个消费者组内，一个分区只能被一个消费者消费
- 不同消费者组之间，互补干扰
```

### 消费者数量与分区数的关系

```
这是一个经典的面试问题：消费者是不是越多越好？

【分区数 = 消费者数】最优
- 每个消费者正好消费一个分区
- 并行度最大化

【消费者数 < 分区数】部分消费者空闲
- 空闲的消费者浪费了
- 增加消费者可以提升吞吐

【消费者数 > 分区数】部分消费者没有分区
- 多余的消费者永远没有消息可消费
- 浪费资源

【结论】
分区数 = 预期的最大消费者数量
消费者数可以动态调整，但不能超过分区数
```

### 分区策略

```java
// 分区策略决定消息发送到哪个分区

public class PartitionStrategy {
    
    // 1. 默认策略：基于 key 的哈希
    public int defaultStrategy(String key, int numPartitions) {
        // 如果没有指定 key，随机选择一个分区
        if (key == null) {
            return ThreadLocalRandom.current().nextInt(numPartitions);
        }
        // 如果有 key，基于 key 的哈希值选择分区
        return Math.abs(key.hashCode()) % numPartitions;
    }
    
    // 2. 粘性分区策略（Kafka 2.4+）
    // 同一个 Producer 的消息尽量发送到同一个分区
    // 减少请求数，提升批处理效率
    
    // 3. 自定义分区策略
    public class CustomPartitioner implements Partitioner {
        
        @Override
        public int partition(String topic, Object key, byte[] keyBytes,
                           Object value, byte[] valueBytes,
                           Cluster cluster) {
            // 根据业务需求自定义分区逻辑
            if ("vip".equals(key)) {
                // VIP 用户发送到特定分区（可能有更好的硬件）
                return 0;
            }
            // 其他用户按哈希分布
            return Math.abs(key.hashCode()) % (cluster.partitionsForTopic(topic).size() - 1);
        }
    }
}
```

## 消息积压：成因与处理方案

### 消息积压的原因分析

```
消息积压的常见原因：

【消费者故障】
- 消费者进程挂了
- 消费者代码有 bug，处理失败
- 消费者 OOM，被迫重启

【消费速度 < 生产速度】
- 消费者资源不足（CPU、内存）
- 消费者代码效率低
- 消费者数量少于分区数

【下游依赖故障】
- 数据库连接池耗尽
- 下游服务超时
- 消费者主动降级

【业务逻辑复杂】
- 消息处理涉及复杂的数据库操作
- 消息处理涉及外部 API 调用
- 消息处理涉及大量计算
```

### 如何发现消息积压

```java
// Kafka 监控积压的代码

public class KafkaLagMonitor {
    
    @Autowired
    private AdminClient adminClient;
    
    public void monitorConsumerLag(String groupId, String topic) {
        // 获取消费者组信息
        ListConsumerGroupOffsetsResult result = 
            adminClient.listConsumerGroupOffsets(groupId);
        
        Map<TopicPartition, OffsetAndMetadata> offsets = 
            result.partitionsToOffsetAndMetadata().get();
        
        // 获取 Topic 的最新消息偏移量
        for (TopicPartition partition : offsets.keySet()) {
            // 当前消费位置
            long currentOffset = offsets.get(partition).offset();
            
            // 获取最新消息偏移量
            Map<TopicPartition, Long> endOffsets = 
                adminClient.listOffsets(
                    Collections.singletonMap(
                        partition, 
                        OffsetSpec.latest()
                    )
                ).all().get();
            
            long latestOffset = endOffsets.get(partition);
            
            // 积压数量
            long lag = latestOffset - currentOffset;
            
            if (lag > THRESHOLD) {
                // 告警
                sendAlert("消费者积压过多", groupId, partition, lag);
            }
        }
    }
}
```

```
处理积压的策略：

【方案一：增加消费者】
适用场景：消费者数量 < 分区数
操作：启动更多消费者实例
优点：简单直接
缺点：不能超过分区数

【方案二：消费者多线程】
适用场景：单条消息处理时间过长
操作：消费者内部开启多线程处理
优点：充分利用消费者资源
缺点：需要保证线程安全

@KafkaListener(topics = "order-events", concurrency = "3")
public void consumeOrder(String message) {
    // 单个 KafkaConsumer 实例开 3 个线程
    // 但并发数受限于分区数
}

【方案三：消息迁移】
适用场景：分区数已经很高，但仍然积压
操作：将消息迁移到新的 Topic（分区数更多）
优点：彻底解决
缺点：需要迁移工具和停机窗口

【方案四：紧急扩容】
适用场景：突发流量
操作：
1. 增加消费者机器
2. 临时增加消费者实例
3. 降级非核心消费逻辑
4. 监控并恢复
```

```java
// 优先处理积压消息的策略

public class PriorityConsumeStrategy {
    
    private final BlockingQueue<Message> normalQueue;
    private final BlockingQueue<Message> priorityQueue;
    
    public Message take() throws InterruptedException {
        // 优先消费积压的高优先级消息
        Message msg = priorityQueue.poll(100, TimeUnit.MILLISECONDS);
        if (msg == null) {
            // 没有积压消息时，消费正常队列
            msg = normalQueue.poll(100, TimeUnit.MILLISECONDS);

        }
        return msg;
    }
}
```

## 消息消费的幂等性保证

### 幂等性的重要性

```
为什么幂等性重要？

场景：用户支付成功后，需要发送通知消息

消息 1：发送给用户"您的订单已支付"
消息 2：（重试）发送给用户"您的订单已支付"
消息 3：（重试）发送给用户"您的订单已支付"

结果：
- 如果不保证幂等：用户收到 3 条相同的通知
- 如果保证幂等：用户只收到 1 条通知

Kafka 的消息交付语义：

At Most Once（最多一次）
- 消息可能丢失，不会重复
- 消费者先提交 offset，再处理消息
- 处理失败，消息就丢了

At Least Once（至少一次）
- 消息不会丢失，可能重复
- 消费者先处理消息，再提交 offset
- 处理成功但 offset 提交失败，消息会被重复消费

Exactly Once（精确一次）
- 消息不丢失，不重复
- 需要消费者端保证幂等
- Kafka Transactions（0.11+）支持
```

### 方案一：去重表

```java
// 基于去重表的幂等实现

@Service
public class IdempotentService {
    
    @Autowired
    private JdbcTemplate jdbcTemplate;
    
    /**
     * 检查并记录消息处理状态
     */
    public boolean tryProcess(String messageId) {
        // 尝试插入去重表
        String sql = "INSERT INTO idempotent_table " +
            "(message_id, status, create_time) " +
            "VALUES (?, 'PROCESSING', NOW())";
        
        try {
            jdbcTemplate.update(sql, messageId);
            return true;  // 插入成功，可以处理
        } catch (DuplicateKeyException e) {
            // 已存在，说明之前已经处理过
            return false;
        }
    }
    
    /**
     * 更新处理状态
     */
    public void markComplete(String messageId, boolean success) {
        String sql = success ?
            "UPDATE idempotent_table SET status = 'SUCCESS' WHERE message_id = ?" :
            "UPDATE idempotent_table SET status = 'FAILED' WHERE message_id = ?";
        
        jdbcTemplate.update(sql, messageId);
    }
}

// 消费者使用
@KafkaListener(topics = "order-events")
public void consumeOrder(Message message) {
    String messageId = message.getMessageId();
    
    if (!idempotentService.tryProcess(messageId)) {
        log.info("消息已处理过，跳过: {}", messageId);
        return;
    }
    
    try {
        // 处理业务逻辑
        doProcess(message);
        
        // 标记处理成功
        idempotentService.markComplete(messageId, true);
    } catch (Exception e) {
        // 标记处理失败，允许重试
        idempotentService.markComplete(messageId, false);
        throw e;
    }
}
```

### 方案二：唯一 ID + Redis

```java
// 基于 Redis 的幂等实现

@Service
public class RedisIdempotentService {
    
    private static final String IDEMPOTENT_PREFIX = "idempotent:";
    private static final long EXPIRE_SECONDS = 86400;  // 24 小时
    
    private final RedisTemplate<String, String> redisTemplate;
    
    /**
     * 尝试获取处理权（原子操作）
     */
    public boolean tryAcquire(String uniqueId) {
        String key = IDEMPOTENT_PREFIX + uniqueId;
        
        // SETNX + EXPIRE 的原子操作
        Boolean result = redisTemplate.opsForValue()
            .setIfAbsent(key, "1", EXPIRE_SECONDS, TimeUnit.SECONDS);
        
        return Boolean.TRUE.equals(result);
    }
    
    /**
     * 标记为处理中（防止并发处理）
     */
    public boolean tryLock(String uniqueId) {
        String key = IDEMPOTENT_PREFIX + uniqueId + ":lock";
        String value = UUID.randomUUID().toString();
        
        // 设置锁，10 秒过期
        Boolean result = redisTemplate.opsForValue()
            .setIfAbsent(key, value, 10, TimeUnit.SECONDS);
        
        return Boolean.TRUE.equals(result);
    }
    
    /**
     * 释放锁
     */
    public void releaseLock(String uniqueId) {
        String key = IDEMPOTENT_PREFIX + uniqueId + ":lock";
        redisTemplate.delete(key);
    }
}

// 使用示例
@KafkaListener(topics = "payment-events")
public void consumePayment(PaymentMessage message) {
    String messageId = message.getMessageId();
    
    // 1. 检查是否已处理过
    if (!idempotentService.tryAcquire(messageId)) {
        log.info("消息已处理过: {}", messageId);
        return;
    }
    
    // 2. 加分布式锁，防止并发处理
    if (!idempotentService.tryLock(messageId)) {
        log.warn("消息正在被其他实例处理: {}", messageId);
        return;
    }
    
    try {
        // 3. 业务处理
        doProcess(message);
    } finally {
        // 4. 释放锁
        idempotentService.releaseLock(messageId);
    }
}
```

## 消息顺序性问题的解决方案

### 顺序问题的成因

```
消息顺序乱的典型场景：

发送顺序：
消息 1（创建订单）→ 消息 2（支付）→ 消息 3（发货）

期望的消费顺序：
1. 先创建订单
2. 再支付
3. 最后发货

可能的消费顺序：
消息 2（支付）→ 消息 1（创建订单）→ 消息 3（发货）

结果：支付时订单还不存在！
```

### 保证顺序的方案

```
保证顺序的几种方案：

【方案一：单分区 + 单消费者】
- 最简单，最可靠
- 缺点：完全没有并行能力

【方案二：按业务 Key 哈希分区】
- 相同 Key 的消息发送到同一个分区
- 保证同一业务实体的消息有序
- 缺点：热点 Key 会导致分区不均

【方案三：业务端保证幂等】
- 允许消息乱序
- 但保证乱序也不影响业务结果
- 适用场景：最终一致即可
```

### 按 Key 哈希分区实现

```java
// 方案一：使用订单 ID 作为 Key

    new ProducerRecord<>(
        "order-events",      // Topic
        order.getOrderId(),  // Key：订单 ID
        orderMessage         // Value
    );

// 相同订单 ID 的消息会发送到同一个分区
// 订单相关的操作（创建→支付→发货）会有序消费

// 方案二：自定义分区器，热点 Key 打散

public class HotKeyAwarePartitioner implements Partitioner {
    
    private final Set<String> hotKeys = new HashSet<>();
    
    @Override
    public int partition(String topic, Object key, byte[] keyBytes,
                        Object value, byte[] valueBytes, Cluster cluster) {
        String keyStr = (String) key;
        
        if (hotKeys.contains(keyStr)) {
            // 热点 Key，添加随机后缀打散
            int numPartitions = cluster.partitionsForTopic(topic).size();
            String virtualKey = keyStr + ":" + ThreadLocalRandom.current().nextInt(100);
            return Math.abs(virtualKey.hashCode()) % numPartitions;
        }
        
        // 普通 Key，按正常哈希
        return Math.abs(keyStr.hashCode()) % cluster.partitionsForTopic(topic).size();
    }
}
```

## 消息丢失的三个环节

### 消息丢失的完整链路

```
消息可能丢失的三个环节：

【1. 生产者 → Broker】
原因：
- 网络抖动，发送失败
- Broker 正好 Leader 切换
- 发送超时

解决方案：
- 配置 acks=all
- 重试机制
- 事务消息

【2. Broker 内部】
原因：
- Leader 收到消息后，数据还没同步到 Follower
- Leader 挂了，Follower 被选为新 Leader
- 数据还在内存中，机器重启

解决方案：
- 配置 replication.factor >= 3
- 配置 min.insync.replicas >= 2
- 配置 unclean.leader.election.enable=false

【3. Broker → 消费者】
原因：
- 消费者自动提交 offset
- 消息还没处理完，消费者重启

解决方案：
- 手动提交 offset
- 处理完消息后再提交
```

```java
                handleSendFailure(record);

        });
    }
```

## 面试追问预判

### Q1：Kafka 如何保证消息不丢失？

**面试官的心理分析：**

这道题考察对消息队列可靠性的理解深度。很多候选人只知道「配置 acks=all」，但说不清楚为什么这样配置，以及还有哪些环节需要注意。

**参考答案：**

```
Kafka 保证消息不丢失需要三个环节都做好：

【1. 生产端】
- acks=all：等待所有 ISR 副本确认
- retries=3：失败自动重试
- enable.idempotence=true：幂等发送

【2. Broker 端】
- replication.factor >= 3：至少 3 个副本
- min.insync.replicas >= 2：至少 2 个同步副本
- unclean.leader.election.enable=false：不选不同步的副本为 Leader

【3. 消费端】
- 手动提交 offset
- 先处理消息，再提交 offset
- 处理失败不提交 offset，让消息重试

只有三个环节都做好，才能真正保证消息不丢失。
```

### Q2：消费者数量和分区数的关系是什么？

**面试官的心理分析：**

这是一道经典问题，但能说清楚「不是越多越好」这个关键点的候选人不多。

**参考答案：**

```
分区数和消费者的关系：

【最优情况：分区数 = 消费者数】
- 每个消费者正好消费一个分区
- 并行度达到最大

【消费者数 < 分区数】
- 部分消费者空闲
- 增加消费者可以提升吞吐

【消费者数 > 分区数】
- 多余的消费者没有分区可消费
- 浪费资源

【结论】
分区数应该设置为预期的最大消费者数量
消费者可以动态调整，但不能超过分区数

【实际建议】
- 分区数不宜过多（影响选举、性能）
- 建议初期设置 6-12 个分区
- 根据实际吞吐量调整
```

### Q3：如何处理消息积压？

**面试官的心理分析：**

这道题考察生产环境的问题解决能力。面试官想看的是你是否有处理过实际问题的经验。

**参考答案：**

```
处理消息积压的步骤：

【1. 分析原因】
- 消费者故障？代码 bug？
- 消费速度 < 生产速度？
- 下游依赖故障？

【2. 快速止血】
- 增加消费者（如果消费者数 < 分区数）
- 消费者多线程处理（单条处理时间过长）
- 降级非核心处理逻辑

【3. 容量扩容】
- 增加消费者实例
- 扩容消费者机器资源

【4. 长期方案】
- 优化消费者代码
- 增加分区数
- 迁移到新 Topic

【5. 监控告警】
- 设置积压告警阈值
- 提前发现问题
```

### Q4：Kafka 的分区策略有哪些？

**面试官的心理分析：**

这道题考察对 Kafka 分区机制的理解深度。

**参考答案：**

```
Kafka 分区策略：

【1. DefaultPartitioner（默认）】
- 如果有 key，基于 key 的哈希值取模
- 如果没有 key，随机选择（2.4 之前）/ 粘性分区（2.4+）

【2. UniformStickyPartitioner（均匀粘性）】
- 同一批消息发送到同一分区
- 最大化批处理效率

【3. CustomPartitioner（自定义）】
- 根据业务逻辑选择分区
- 例如：VIP 用户发送到特定分区

【4. 按 Key 哈希】
- 保证相同 key 的消息有序
- 但可能导致热点 key 不均

【粘性分区的优势】
- 减少请求数，提升网络效率
- 充分利用批处理
- Kafka 2.4+ 默认使用
```

## 总结

消息队列是解决高并发问题的利器，但要真正用好它，需要深入理解其内部机制。

核心要点：

- **削峰、解耦、异步** 是消息队列的三大核心作用
- **Kafka 的高吞吐** 来自于顺序写、零拷贝、批量处理
- **分区是并行消费的基础**，但消费者数不能超过分区数
- **消息积压** 需要分析原因，对症处理
- **幂等性** 是消费端必须保证的，处理失败的消息要妥善处理
- **顺序性** 可以通过按 Key 分区或业务兜底来解决
- **消息不丢失** 需要生产端、Broker 端、消费端三个环节都做好
