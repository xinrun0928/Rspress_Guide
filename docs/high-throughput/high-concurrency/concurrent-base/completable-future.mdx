# CompletableFuture 异步编程实战

你有没有被这样的代码折磨过？

```java
// 三个接口调用，串行执行
User user = userService.getUser(id);           // 等待 100ms
List<Order> orders = orderService.getOrders(user.getId());  // 等待 200ms
List<Product> products = productService.getProducts(orders); // 等待 150ms

// 总耗时：450ms
```

三个接口调用，串行执行，总耗时等于三个接口耗时的总和。如果能并行调用，理论上只需要 **200ms**（最长那个）。

在 JDK 8 之前，这需要借助 `CountDownLatch`、`CompletableService` 等工具类，代码复杂且容易出错。直到 `CompletableFuture` 的出现。

今天，我们就来彻底掌握 `CompletableFuture`，让你的异步编程从"能用"到"精通"。

## 一、Future 的局限性

在说 CompletableFuture 之前，先看看它的前身 `Future` 有哪些问题。

### 1.1 传统 Future 的用法

`Future` 是 JDK 5 引入的，用于获取异步任务的结果：

```java
ExecutorService executor = Executors.newFixedThreadPool(4);

Future<User> futureUser = executor.submit(() -> {
    return userService.getUser(id);
});

// 主线程可以继续做其他事
doSomethingElse();

// 需要结果时，调用 get()
User user = futureUser.get();  // 阻塞等待
```

### 1.2 Future 的三大缺陷

**缺陷一：无法组合多个 Future**

如果需要等待多个异步任务完成，`Future` 无能为力：

```java
Future<User> futureUser = executor.submit(() -> getUser(id));
Future<List<Order>> futureOrders = executor.submit(() -> getOrders(id));

// 无法直接组合这两个 Future
// 必须手动处理
```

**缺陷二：无法手动完成 Future**

`Future` 的结果只能由异步任务设置，无法从外部手动完成：

```java
// 想根据某个条件手动完成 Future？不支持
Future<String> future = executor.submit(() -> compute());
if (condition) {
    future.complete("result");  // Future 没有 complete() 方法
}
```

**缺陷三：只能通过 get() 阻塞获取结果**

```java
Future<String> future = executor.submit(() -> doSomething());

// get() 是阻塞的
String result = future.get();

// 没有超时的话，如果任务卡住，整个线程就卡住了
```

这三个缺陷让 `Future` 在复杂的异步场景下显得力不从心。

## 二、CompletableFuture 的优势

<Badge text="JDK 8+" type="tip" /> `CompletableFuture` 是 JDK 8 引入的，它解决了 `Future` 的所有问题。

### 2.1 CompletableFuture 的核心能力

```java
CompletableFuture<User> futureUser = CompletableFuture.supplyAsync(() -> getUser(id));

// 支持链式调用
futureUser
    .thenApply(user -> user.getId())           // 转换
    .thenCompose(id -> getOrdersAsync(id))     // 组合
    .thenAccept(orders -> System.out.println(orders))  // 消费
    .exceptionally(ex -> {                     // 异常处理
        log.error("Error", ex);
        return null;
    });
```

相比 `Future`，`CompletableFuture` 提供了：

- **丰富的转换方法**：thenApply、thenAccept、thenRun
- **强大的组合能力**：thenCompose、thenCombine、allOf、anyOf
- **便捷的异常处理**：exceptionally、handle、whenComplete
- **手动完成能力**：complete()、completeExceptionally()

### 2.2 创建 CompletableFuture 的方式

```java
// 方式一：supplyAsync - 有返回值
CompletableFuture<String> future1 = CompletableFuture.supplyAsync(() -> {
    return "Hello";
});

// 方式二：runAsync - 无返回值
CompletableFuture<Void> future2 = CompletableFuture.runAsync(() -> {
    System.out.println("Hello");
});

// 方式三：completedFuture - 直接完成
CompletableFuture<String> future3 = CompletableFuture.completedFuture("Hello");

// 方式四：failedFuture - 直接失败（Java 9+）
CompletableFuture<String> future4 = CompletableFuture.failedFuture(
    new RuntimeException("Error")
);
```

## 三、转换方法：thenApply、thenAccept、thenRun

这三个方法是 CompletableFuture 的基础，它们构成了异步处理的链式调用。

### 3.1 thenApply：转换结果

`thenApply` 用于**转换 CompletableFuture 的结果类型**：

```java
// 原始 CompletableFuture
CompletableFuture<User> futureUser = CompletableFuture.supplyAsync(() -> getUser(id));

// thenApply：把 User 转换成 UserId
CompletableFuture<String> futureUserId = futureUser.thenApply(user -> user.getId());

// 链式调用
CompletableFuture<String> futureResult = futureUser
    .thenApply(user -> user.getId())
    .thenApply(id -> "User ID: " + id);
```

### 3.2 thenAccept：消费结果

`thenAccept` 用于**消费结果，不返回新值**：

```java
futureUser.thenAccept(user -> {
    System.out.println("User: " + user.getName());
    System.out.println("Email: " + user.getEmail());
});  // 返回 CompletableFuture<Void>
```

**场景**：当你只需要执行副作用（打印日志、发送消息）而不需要返回值时，用 `thenAccept`。

### 3.3 thenRun：不依赖结果

`thenRun` 用于**不依赖 CompletableFuture 结果的操作**：

```java
futureUser.thenRun(() -> {
    // 不关心 user 是什么，只想在 futureUser 完成后执行
    System.out.println("User loaded!");
});  // 返回 CompletableFuture<Void>
```

三种方法的对比：

| 方法 | 依赖前一个结果 | 返回值 | 典型用途 |
|---|---|---|---|
| `thenApply` | 是 | 有 | 转换结果 |
| `thenAccept` | 是 | 无 | 消费结果 |
| `thenRun` | 否 | 无 | 执行后续操作 |

### 3.4 异步版本

每个方法都有对应的异步版本（带 Async 后缀）：

```java
// 同步版本：使用上一个任务的线程
futureUser.thenApply(user -> transform(user));

// 异步版本：使用 ForkJoinPool 或自定义线程池
futureUser.thenApplyAsync(user -> transform(user));

// 异步版本 + 指定线程池
futureUser.thenApplyAsync(user -> transform(user), customExecutor);
```

:::warning Async 方法的线程问题

如果多个 `thenApply` 形成链式调用且都是同步版本（无 Async），它们会在**同一个线程**中执行。如果某个是 Async 版本，会切换到其他线程。

这可能导致意想不到的线程上下文问题（如 ThreadLocal 数据丢失）。

:::

## 四、组合方法：thenCompose、thenCombine

组合方法是 CompletableFuture 的精髓，它们让你能够优雅地处理异步依赖。

### 4.1 thenCompose：级联组合

`thenCompose` 用于**处理异步任务的依赖关系**——一个任务的输入是另一个任务的结果。

**场景**：获取用户信息后，再根据用户 ID 获取订单列表：

```java
// 串行写法（低效）
CompletableFuture<User> futureUser = CompletableFuture.supplyAsync(() -> getUser(id));
User user = futureUser.join();  // 阻塞等待
CompletableFuture<List<Order>> futureOrders = CompletableFuture.supplyAsync(() -> getOrders(user.getId()));

// 级联写法（高效）- 使用 thenCompose
CompletableFuture<List<Order>> futureOrders = futureUser
    .thenCompose(user -> getOrdersAsync(user.getId()));
```

**为什么不用 thenApply？**

```java
// 错误：thenApply 期望的是普通值，而不是 CompletableFuture
futureUser
    .thenApply(user -> getOrdersAsync(user.getId()))  // 返回 CompletableFuture<List<Order>>
    // 实际上返回的是 CompletableFuture<CompletableFuture<List<Order>>>>

// 正确：用 thenCompose 扁平化
futureUser
    .thenCompose(user -> getOrdersAsync(user.getId()))
    // 返回 CompletableFuture<List<Order>>
```

`thenCompose` = **扁平化的 thenApply**，用于处理返回 `CompletableFuture` 的函数。

### 4.2 thenCombine：并行组合

`thenCombine` 用于**合并两个独立的 CompletableFuture**。

**场景**：同时获取用户信息和商品列表，然后组合它们：

```java
CompletableFuture<User> futureUser = CompletableFuture.supplyAsync(() -> getUser(id));
CompletableFuture<List<Product>> futureProducts = CompletableFuture.supplyAsync(() -> getProducts());

// 等待两个都完成，然后组合结果
CompletableFuture<OrderPage> futureOrderPage = futureUser
    .thenCombine(futureProducts, (user, products) -> {
        OrderPage page = new OrderPage();
        page.setUser(user);
        page.setProducts(products);
        return page;
    });
```

### 4.3 allOf 和 anyOf：批量处理

**allOf**：等待所有 CompletableFuture 完成

```java
CompletableFuture<User> future1 = getUserAsync(1);
CompletableFuture<User> future2 = getUserAsync(2);
CompletableFuture<User> future3 = getUserAsync(3);

CompletableFuture<Void> allFuture = CompletableFuture.allOf(future1, future2, future3);

allFuture.thenRun(() -> {
    // 所有任务完成后执行
    List<User> users = Arrays.asList(
        future1.join(),  // join() 不抛异常
        future2.join(),
        future3.join()
    );
    System.out.println("All users loaded: " + users);
});
```

:::warning allOf 的返回值问题

`allOf` 返回 `CompletableFuture<Void>`，即使部分任务失败，只要有一个完成就返回。

正确的做法是在之前处理异常：

```java
// 在 allOf 之前完成每个 CompletableFuture 的异常处理
CompletableFuture<User> future1 = getUserAsync(1).exceptionally(ex -> null);
CompletableFuture<User> future2 = getUserAsync(2).exceptionally(ex -> null);
CompletableFuture<User> future3 = getUserAsync(3).exceptionally(ex -> null);

CompletableFuture.allOf(future1, future2, future3);
```

:::

**anyOf**：等待任意一个 CompletableFuture 完成

```java
CompletableFuture<User> future1 = getUserFromCache(id);
CompletableFuture<User> future2 = getUserFromDB(id);

CompletableFuture.anyOf(future1, future2).thenAccept(result -> {
    System.out.println("First available result: " + result);
});
```

## 五、异常处理：exceptionally、handle、whenComplete

异步编程中，异常处理尤为重要。CompletableFuture 提供了多种异常处理方式。

### 5.1 exceptionally：捕获异常

`exceptionally` 用于**捕获异常并返回默认值**：

```java
CompletableFuture<User> future = CompletableFuture
    .supplyAsync(() -> getUser(id))
    .exceptionally(ex -> {
        log.error("Failed to get user", ex);
        return null;  // 返回 null 作为默认值
    });
```

### 5.2 handle：无论成功还是失败都处理

`handle` 用于**无论任务成功还是失败，都执行处理逻辑**：

```java
CompletableFuture<String> future = CompletableFuture
    .supplyAsync(() -> getUserName(id))
    .handle((result, ex) -> {
        if (ex != null) {
            log.error("Failed", ex);
            return "Unknown";  // 失败时返回默认值
        }
        return result;  // 成功时返回原结果
    });
```

### 5.3 whenComplete：完成后回调

`whenComplete` 用于**执行清理操作或记录日志**：

```java
CompletableFuture<User> future = CompletableFuture
    .supplyAsync(() -> getUser(id))
    .whenComplete((result, ex) -> {
        if (ex != null) {
            log.error("Task failed", ex);
        } else {
            log.info("Task succeeded: {}", result);
        }
    });
```

三种异常处理方式的对比：

| 方法 | 处理结果 | 处理异常 | 返回值 |
|---|---|---|---|
| `exceptionally` | 无 | 有 | 必须返回默认值 |
| `handle` | 有 | 有 | 可以返回新值 |
| `whenComplete` | 有 | 有 | 不改变结果 |

## 六、线程池配置：性能的关键

CompletableFuture 默认使用 `ForkJoinPool.commonPool()`，这个线程池是 JVM 全局共享的。理解它的配置，对性能优化至关重要。

### 6.1 默认线程池的问题

```java
// 默认使用 ForkJoinPool.commonPool()
CompletableFuture.supplyAsync(() -> doSomething());

// 问题：commonPool 的并行度 = CPU 核心数 - 1
// 如果 CPU 有 8 核，commonPool 只有 7 个线程
```

`ForkJoinPool.commonPool()` 的特性：

- 并行度 = `Runtime.getRuntime().availableProcessors() - 1`
- 所有 CompletableFuture 共享同一个线程池
- 如果某个任务阻塞，会占用线程资源

### 6.2 自定义线程池

对于 IO 密集型任务，建议使用自定义线程池：

```java
// IO 密集型线程池：线程数 = CPU 核心数 * 2
ExecutorService ioExecutor = new ThreadPoolExecutor(
    20,                                              // corePoolSize
    50,                                              // maximumPoolSize
    60L, TimeUnit.SECONDS,                           // keepAliveTime
    new LinkedBlockingQueue<>(1000),                 // queueCapacity
    new ThreadFactoryBuilder()
        .setNameFormat("io-task-%d")
        .build(),
    new ThreadPoolExecutor.CallerRunsPolicy()        // 拒绝策略
);

// 提交任务时指定线程池
CompletableFuture<String> future = CompletableFuture
    .supplyAsync(() -> doIOOperation(), ioExecutor);
```

### 6.3 线程池选择的最佳实践

```
CPU 密集型：
  - 线程数 = CPU 核心数 + 1
  - 使用 CPU 核心数的线程数，减少线程切换开销

IO 密集型：
  - 线程数 = CPU 核心数 * 2（或根据 IO 等待比例调整）
  - 使用更多线程，因为大部分时间在等待 IO
```

### 6.4 线程池隔离

不同类型的任务使用不同的线程池，避免相互影响：

```java
// CPU 密集型任务线程池
ExecutorService cpuExecutor = Executors.newFixedThreadPool(
    Runtime.getRuntime().availableProcessors()
);

// IO 密集型任务线程池
ExecutorService ioExecutor = Executors.newFixedThreadPool(50);

// HTTP 调用线程池
ExecutorService httpExecutor = new ThreadPoolExecutor(
    10, 100, 60L, TimeUnit.SECONDS, new LinkedBlockingQueue<>(500)
);

// 数据库操作线程池
ExecutorService dbExecutor = new ThreadPoolExecutor(
    5, 20, 60L, TimeUnit.SECONDS, new LinkedBlockingQueue<>(200)
);
```

## 七、实战场景：多个接口并行调用

这是最常见的使用场景：需要调用多个接口，但它们之间没有依赖关系，可以并行执行。

### 7.1 场景一：并行获取用户、订单、商品数据

```java
public OrderDetailVO getOrderDetail(Long orderId) {
    long startTime = System.currentTimeMillis();

    ExecutorService executor = getCustomExecutor();

    CompletableFuture<User> futureUser = CompletableFuture
        .supplyAsync(() -> userService.getUser(orderId), executor);
    CompletableFuture<List<OrderItem>> futureItems = CompletableFuture
        .supplyAsync(() -> orderService.getOrderItems(orderId), executor);
    CompletableFuture<Payment> futurePayment = CompletableFuture
        .supplyAsync(() -> paymentService.getPayment(orderId), executor);
    CompletableFuture<List<Comment>> futureComments = CompletableFuture
        .supplyAsync(() -> commentService.getComments(orderId), executor);

    // 等待所有任务完成
    CompletableFuture.allOf(futureUser, futureItems, futurePayment, futureComments)
        .exceptionally(ex -> {
            log.error("Failed to load order detail", ex);
            return null;
        })
        .join();

    // 组合结果
    OrderDetailVO detail = new OrderDetailVO();
    detail.setUser(futureUser.join());
    detail.setItems(futureItems.join());
    detail.setPayment(futurePayment.join());
    detail.setComments(futureComments.join());

    log.info("Order detail loaded in {}ms", System.currentTimeMillis() - startTime);
    return detail;
}
```

串行执行耗时：100ms + 200ms + 150ms + 80ms = 530ms

并行执行耗时：max(100ms, 200ms, 150ms, 80ms) = 200ms

**性能提升：60% 以上**。

### 7.2 场景二：级联调用（避免回调地狱）

```java
// 需求：获取用户 → 获取用户的店铺 → 获取店铺的商品

// 低效写法：嵌套回调
userService.getUserAsync(userId)
    .thenAccept(user -> {
        storeService.getStoreAsync(user.getStoreId())
            .thenAccept(store -> {
                productService.getProductsAsync(store.getId())
                    .thenAccept(products -> {
                        // 处理结果
                    });
            });
    });

// 高效写法：thenCompose 链式调用
userService.getUserAsync(userId)
    .thenCompose(user -> storeService.getStoreAsync(user.getStoreId()))
    .thenCompose(store -> productService.getProductsAsync(store.getId()))
    .thenAccept(products -> {
        // 处理结果
    })
    .exceptionally(ex -> {
        log.error("Failed to load products", ex);
        return null;
    });
```

### 7.3 场景三：超时控制

```java
CompletableFuture<User> future = CompletableFuture
    .supplyAsync(() -> getUser(id))
    .orTimeout(3, TimeUnit.SECONDS)  // JDK 9+
    .exceptionally(ex -> {
        if (ex instanceof TimeoutException) {
            return getDefaultUser();  // 超时时返回默认值
        }
        throw new CompletionException(ex);
    });
```

或者使用 JDK 8 兼容的方式：

```java
CompletableFuture<User> future = CompletableFuture
    .supplyAsync(() -> {
        try {
            return getUserWithTimeout(id, 3, TimeUnit.SECONDS);
        } catch (TimeoutException e) {
            return getDefaultUser();
        }
    });
```

### 7.4 场景四：重试机制

```java
public CompletableFuture<User> getUserWithRetry(Long userId, int maxRetries) {
    return CompletableFuture
        .supplyAsync(() -> userService.getUser(userId))
        .exceptionallyCompose(ex -> {
            if (maxRetries > 0) {
                return CompletableFuture
                    .delayedFuture(1, TimeUnit.SECONDS)  // 等待 1 秒
                    .thenCompose(v -> getUserWithRetry(userId, maxRetries - 1));
            }
            throw new CompletionException(ex);
        });
}
```

## 八、常见坑点与最佳实践

### 8.1 坑点一：没有指定线程池

```java
// 问题：所有任务都使用默认的 ForkJoinPool.commonPool()
// 在高并发下会造成线程资源竞争
CompletableFuture.supplyAsync(() -> doSomething());

// 解决：始终使用自定义线程池
CompletableFuture.supplyAsync(() -> doSomething(), customExecutor);
```

### 8.2 坑点二：在循环中创建 CompletableFuture

```java
// 问题：大量创建 CompletableFuture，增加线程调度开销
List<Long> userIds = getUserIds();
List<CompletableFuture<User>> futures = new ArrayList<>();
for (Long userId : userIds) {
    futures.add(CompletableFuture.supplyAsync(() -> getUser(userId)));
}

// 解决：使用并行流（内部使用 ForkJoinPool）
List<User> users = userIds.parallelStream()
    .map(id -> userService.getUser(id))
    .collect(Collectors.toList());
```

### 8.3 坑点三：忘记处理异常

```java
// 问题：异常被吞掉，不知道任务失败
CompletableFuture.supplyAsync(() -> doSomething())
    .thenApply(result -> process(result));

// 解决：始终添加异常处理
CompletableFuture.supplyAsync(() -> doSomething())
    .thenApply(result -> process(result))
    .exceptionally(ex -> {
        log.error("Task failed", ex);
        return null;
    });
```

### 8.4 坑点四：阻塞调用 join() 或 get()

```java
// 问题：在异步链中调用 join() 会阻塞
CompletableFuture<User> future1 = CompletableFuture.supplyAsync(() -> getUser(1));
CompletableFuture<User> future2 = CompletableFuture.supplyAsync(() -> getUser(2));

// 阻塞！
List<User> users = Arrays.asList(future1.join(), future2.join());

// 解决：使用 allOf + exceptionally
CompletableFuture.allOf(future1, future2)
    .thenRun(() -> {
        List<User> users = Arrays.asList(future1.join(), future2.join());
        // 处理结果
    });
```

## 九、性能优化建议

### 9.1 选择合适的线程池

| 任务类型 | 推荐线程数 | 理由 |
|---|---|---|
| CPU 密集型 | N+1 或 N*2 | 减少线程切换，充分利用 CPU |
| IO 密集型 | 2N 或更多 | IO 等待期间可以执行其他任务 |
| 混合型 | 根据比例计算 | 参考公式：`线程数 = CPU 核心数 * (1 + IO时间/CPU时间)` |

### 9.2 避免过度异步

异步不是万能药。对于简单操作（如内存计算），同步执行可能更快：

```java
// 简单计算，同步执行更快
int result = calculate();

// 复杂计算或 IO 操作，异步更有优势
CompletableFuture.supplyAsync(() -> callExternalAPI());
```

### 9.3 使用 CompletableFuture 完成任务编排

```java
// 优雅的任务编排
CompletableFuture
    .supplyAsync(() -> fetchDataFromDB(), dbExecutor)
    .thenApplyAsync(this::processData, cpuExecutor)
    .thenComposeAsync(result -> sendToExternalService(result), httpExecutor)
    .whenComplete((result, ex) -> {
        if (ex != null) {
            log.error("Pipeline failed", ex);
        } else {
            log.info("Pipeline succeeded: {}", result);
        }
    });
```

## 十、面试追问预判

### 追问一：CompletableFuture 和 ResponseFuture 的区别？

| 特性 | CompletableFuture | ResponseFuture |
|---|---|---|
| 来源 | JDK 8 | Dubbo RPC 框架 |
| 线程模型 | ForkJoinPool / 自定义线程池 | Netty 异步 IO |
| 异常处理 | exceptionally / handle / whenComplete | try-catch 或 Listener |
| 组合能力 | thenCompose / thenCombine | 嵌套 Future |

CompletableFuture 是 JDK 层面的异步抽象，ResponseFuture 是 RPC 框架层面的封装。

### 追问二：CompletableFuture 的实现原理？

`CompletableFuture` 内部维护了一个 **Completion 链表**，当任务完成后，会依次触发后续的依赖任务：

```
supplyAsync()
    │
    ▼
┌─────────────────────┐
│   CompletableFuture │
│   - result           │  ← 存储结果
│   - callers (链表)   │  ← 依赖当前 Future 的任务
└─────────────────────┘
    │
    ▼
thenApply() 创建新的 CompletableFuture，加入 callers 链表
    │
    ▼
完成时，遍历 callers，触发后续任务
```

### 追问三：如何实现 CompletableFuture 的超时控制？

JDK 9+ 直接提供了 `orTimeout` 方法。JDK 8 可以用以下方式：

```java
public static <T> CompletableFuture<T> withTimeout(
    CompletableFuture<T> future, long timeout, TimeUnit unit) {

    final CompletableFuture<T> timeoutFuture = new CompletableFuture<>();

    final ScheduledFuture<?> scheduledFuture = SchedulorPool
        .scheduledExecutorService
        .schedule(() -> {
            if (future.isDone()) {
                timeoutFuture.complete(future.get());
            } else {
                timeoutFuture.completeExceptionally(
                    new TimeoutException("Operation timed out"));
            }
        }, timeout, unit);

    future.whenComplete((result, ex) -> {
        if (future.isDone()) {
            scheduledFuture.cancel(false);
        }
    });

    return timeoutFuture;
}
```

### 追问四：CompletableFuture 和 Reactor (Flux/Mono) 如何选择？

| 场景 | 推荐 | 理由 |
|---|---|---|
| 简单异步任务 | CompletableFuture | 轻量，API 简洁 |
| 复杂流式处理 | Reactor | 丰富的操作符（map、flatMap、filter、zip...） |
| Spring WebFlux | Reactor | 与 WebFlux 深度集成 |
| JDK 8 环境 | CompletableFuture | 无需额外依赖 |

## 留给你的思考

CompletableFuture 让异步编程变得优雅，但你知道吗？**异步只是性能优化的一环**。

在你的系统中，如果接口返回时间从 300ms 降到了 100ms，用户的体验真的提升了 3 倍吗？

答案是：**不一定**。如果前端渲染还要 200ms，网络延迟 50ms，那后端优化带来的体验提升就不明显了。

真正的性能优化，需要**端到端**考虑：**前端缓存、CDN、网络优化、后端异步处理、数据库优化**……CompletableFuture 只是其中一环。

但这恰恰说明：每个环节都有优化空间，每个环节都值得深入研究。
