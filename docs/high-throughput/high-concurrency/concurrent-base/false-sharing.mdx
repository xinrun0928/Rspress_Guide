# 伪共享（False Sharing）与 @Contended 注解

先看一段代码：

```java
public class FalseSharingDemo {
    public static void main(String[] args) throws InterruptedException {
        long start = System.currentTimeMillis();
        
        Counter c1 = new Counter();
        Counter c2 = new Counter();
        
        Thread t1 = new Thread(() -> {
            for (int i = 0; i < 100_000_000; i++) {
                c1.counter1 = i;
            }
        });
        
        Thread t2 = new Thread(() -> {
            for (int i = 0; i < 100_000_000; i++) {
                c2.counter2 = i;
            }
        });
        
        t1.start();
        t2.start();
        t1.join();
        t2.join();
        
        System.out.println("Time: " + (System.currentTimeMillis() - start) + " ms");
    }
    
    static class Counter {
        long counter1;
        long counter2;
    }
}
```

两个线程分别修改 `counter1` 和 `counter2`，按理说互不干扰。但实际运行时间可能是几百毫秒。

现在给 `Counter` 加上填充：

```java
static class Counter {
    long p1, p2, p3, p4, p5, p6, p7;  // 填充
    volatile long counter1;
    long p8, p9, p10, p11, p12, p13, p14;  // 填充
    volatile long counter2;
    long p15, p16, p17, p18, p19, p20, p21;  // 填充
}
```

同样的代码，运行时间可能降到几十毫秒。

为什么？

这就是我们要讲的**伪共享（False Sharing）**问题。

## 一、CPU 多级缓存架构

在理解伪共享之前，我们需要先了解 CPU 的缓存架构。

### 1.1 为什么需要缓存？

现代 CPU 的主频可以达到 3GHz～5GHz，也就是每秒 30 亿～50 亿个时钟周期。但内存（DDR4/DDR5）的访问延迟大约是几十纳秒，相当于几百个时钟周期。

```
CPU 访问时间：
- L1 Cache：约 1 ns
- L2 Cache：约 3-5 ns
- L3 Cache：约 10-20 ns
- 主内存：约 80-100 ns

速度差距：
- L1 vs 内存：约 100 倍
- L2 vs 内存：约 30 倍
```

如果没有缓存，CPU 大部分时间都在等待内存数据，性能会惨不忍睹。所以，现代 CPU 采用了多级缓存架构：

```
┌─────────────────────────────────────────────────────────────┐
│                         CPU Core                            │
│  ┌─────────────┐                                            │
│  │  Registers │  <- 最快，零等待                            │
│  └──────┬──────┘                                            │
│         │                                                   │
│  ┌──────▼──────┐                                            │
│  │   L1 Cache  │  <- 指令缓存 + 数据缓存，各 32KB           │
│  │   ~1 cycle  │     延迟约 1 个时钟周期                    │
│  └──────┬──────┘                                            │
│         │                                                   │
│  ┌──────▼──────┐                                            │
│  │   L2 Cache  │  <- 统一缓存，通常 256KB-1MB               │
│  │   ~3 cycles │     延迟约 3-5 个时钟周期                  │
│  └──────┬──────┘                                            │
│         │                                                   │
│  ┌──────▼──────┐                                            │
│  │   L3 Cache  │  <- 统一缓存，多核共享，4-32MB             │
│  │  ~10 cycles │     延迟约 10-20 个时钟周期                │
│  └──────┬──────┘                                            │
│         │                                                   │
│  ┌──────▼──────┐                                            │
│  │   Memory    │  <- 主内存，容量大但速度慢                 │
│  │ ~100 cycles │     延迟约 80-100 ns                       │
│  └─────────────┘                                            │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 Cache Line 的概念

CPU 缓存和内存之间的数据交换，不是按「字节」或「字」进行的，而是按**缓存行（Cache Line）**进行的。

一个 Cache Line 的标准大小是 **64 字节**（在大多数现代 x86 和 ARM 处理器上）。

```bash
# 查看 CPU Cache Line 大小
# Linux
$ getconf LEVEL1_DCACHE_LINESIZE
64

# macOS
$ sysctl -a | grep cachesize
```

当 CPU 读取一个变量的值时，它会把这个变量所在的整个 Cache Line（64 字节）都加载到 L1 Cache 中。即使你只读一个字节，下次访问相邻的字节时也可能命中缓存。

### 1.3 CPU 如何知道数据在哪个 Cache Line？

每个 Cache Line 都包含一个**标签（Tag）**，标识这行缓存的是内存中的哪块数据。CPU 通过虚拟地址或物理地址计算标签，来判断数据是否在缓存中：

```
Cache Line 结构：
┌─────────────────────────────────────────────────────────────────┐
│  Tag (用于匹配地址)  │  Valid Bit  │  LRU/Bits  │  Data (64B)   │
└─────────────────────────────────────────────────────────────────┘

地址映射（简化）：
[         Tag          ][    Index      ][   Offset    ]
     用于匹配地址          Cache 组索引      字节偏移
```

## 二、什么是伪共享？

### 2.1 伪共享的定义

**伪共享（False Sharing）** 是指：两个或多个线程分别访问**同一个 Cache Line 上的不同变量**，由于缓存一致性协议，每个线程的写操作都会导致其他线程的 Cache Line 失效，即使它们访问的是完全不同的变量。

这是一个典型的「看似无害实则致命」的性能问题。

### 2.2 伪共享的成因

让我们用图示说明：

```
内存中的数据结构（假设在同一 Cache Line）：

[  long counter1  ][  long counter2  ]
<-------- 16字节 --------><-------- 16字节 -------->  共 32 字节
<-------------------- 64 字节 Cache Line ------------------->

时间线：

Thread 1 (CPU Core 1)            Thread 2 (CPU Core 2)
     │                                │
     │  读取 counter1                  │  读取 counter2
     │  Cache Line 加载到 Core 1 L1    │  Cache Line 加载到 Core 2 L1
     │                                │
     │  修改 counter1                   │  修改 counter2
     │  (写屏障)                       │  (写屏障)
     │  Cache Line 标记为 dirty        │
     │                                │
     │           ──────────── 触发缓存一致性协议 ────────────>
     │                                │
     │                                │  Core 2 的 Cache Line 失效！
     │                                │  必须重新从内存/Core 1 加载
     │                                │
     │                                │  修改 counter2
     │                                │  Cache Line 标记为 dirty
     │                                │
     │           <─────────── 触发缓存一致性协议 ────────────
     │                                │
     │  Core 1 的 Cache Line 失效！    │
     │  必须重新加载                   │
     │                                │
     v                                v

结果：两个线程各自只修改自己的变量，
      却导致对方的高速缓存失效，
      实际效果等同于两个线程在争用同一个锁！
```

### 2.3 伪共享 vs 真共享

| 类型 | 场景 | 特征 |
|---|---|---|
| **真共享（True Sharing）** | 两个线程修改同一个变量 | 需要同步，开发者有预期 |
| **伪共享（False Sharing）** | 两个线程修改同一个 Cache Line 上的不同变量 | 不需要同步，但实际被强制同步 |
| **无共享（No Sharing）** | 两个线程修改不同 Cache Line 上的变量 | 真正并行，无干扰 |

## 三、MESI 缓存一致性协议

### 3.1 什么是 MESI

MESI 是 Modified-Exclusive-Shared-Invalid 的缩写，是一种经典的缓存一致性协议。

每个 Cache Line 有一个状态位：

| 状态 | 说明 | 描述 |
|---|---|---|
| **M（Modified）** | 已修改 | 本核修改过，与主存不一致；其他核的副本已失效 |
| **E（Exclusive）** | 独占 | 只有本核持有，与主存一致 |
| **S（Shared）** | 共享 | 多个核持有相同副本，与主存一致 |
| **I（Invalid）** | 无效 | 本核缓存的数据无效，需要重新加载 |

```
状态转换图：

                    ┌─────────┐
                    │Invalid  │
                    └────┬────┘
                         │
           ┌─────────────┼─────────────┐
           │读本地缓存   │读远程缓存   │写本地
           ▼             │             ▼
     ┌───────────┐        │      ┌───────────┐
     │ Exclusive │        │      │ Modified  │
     └─────┬─────┘        │      └───────────┘
           │              │             ▲
           │共享读取     │写          │
           ▼              │             │
     ┌───────────┐        │             │
     │  Shared   │        └─────────────┘
     └───────────┘         总线嗅探到其他核写
```

### 3.2 MESI 与伪共享

回到前面的例子：

```
初始状态：
Core 1: Cache Line [E] (Exclusive)
Core 2: Cache Line [I] (Invalid)

Core 1 写 counter1：
Core 1: Cache Line [M] (Modified)
Core 2: Cache Line [I] (Invalid) ← 被通知失效

Core 2 要写 counter2：
Core 2: Cache Line [I] → 必须从 Core 1 或内存加载
Core 2: 加载完成后变成 [M]
Core 1: Cache Line [I] ← 被通知失效

Core 1 要写 counter1：
Core 1: Cache Line [I] → 再次失效...

无限循环！
```

每次 Core 1 或 Core 2 写入数据时，都要通知对方「你的缓存失效了」。这导致两个核心反复失效对方的缓存，本质上形成了一种「隐式同步」。

### 3.3 MESI 的性能代价

当一个 Cache Line 在多个核心间来回失效时：

1. **写操作变慢**：写之前需要等待其他核确认失效
2. **缓存未命中**：失效后需要重新加载数据
3. **总线流量增加**：每个写操作都需要广播失效通知
4. **存储stall**：CPU 等待缓存操作完成

```
性能数据（典型值）：

L1 缓存命中：1-2 周期
L2 缓存命中：3-5 周期
跨核 Cache Line 传输：约 30-50 周期
内存访问：约 100 周期

伪共享导致的性能损失：可能高达 10-100 倍！
```

## 四、伪共享的经典案例

### 4.1 Long 类型变量的伪共享

```java
public class LongFalseSharing {
    // 两个独立的计数器，分别被不同线程修改
    static class Data {
        volatile long value1;
        volatile long value2;
    }
    
    public static void main(String[] args) throws InterruptedException {
        Data data = new Data();
        
        Thread t1 = new Thread(() -> {
            for (long i = 0; i < 100_000_000; i++) {
                data.value1 = i;
            }
        });
        
        Thread t2 = new Thread(() -> {
            for (long i = 0; i < 100_000_000; i++) {
                data.value2 = i;
            }
        });
        
        long start = System.nanoTime();
        t1.start();
        t2.start();
        t1.join();
        t2.join();
        long duration = (System.nanoTime() - start) / 1_000_000;
        
        System.out.println("Duration: " + duration + " ms");
        // 典型结果：500-1000 ms（伪共享严重）
    }
}
```

`long` 是 8 字节，两个 `volatile long` 正好 16 字节，很可能落在同一个 Cache Line 上。

### 4.2 LongAdder 的伪共享问题与解决

`LongAdder` 是 JDK 8 引入的高性能计数器，它的核心设计就是为了解决伪共享问题。

```java
// JDK 源码中的 LongAdder 内部类
public class LongAdder extends AbstractLongAdder {
    
    // 关键设计：使用数组 + @Contended 注解
    @sun.misc.Contended
    static final class Cell {
        volatile long value;
        Cell(long initialValue) { value = initialValue; }
        
        final boolean cas(long expect, long update) {
            return UNSAFE.compareAndSwapLong(this, VALUE_OFFSET,
                                              expect, update);
        }
    }
    
    // Cell 数组，每个线程累加不同的 Cell
    transient volatile Cell[] cells;
    
    // base 值，用于低竞争场景
    transient volatile long base;
}
```

`@Contended` 注解确保每个 `Cell` 对象独占一个 Cache Line（周围有填充），从而避免伪共享。

### 4.3 Disruptor 的伪共享解决方案

LMAX Disruptor 是一个高性能消息队列框架，它在早期版本中使用手动填充：

```java
// Disruptor 中的 RingBuffer Pad
abstract class RingBufferPad {
    protected long p1, p2, p3, p4, p5, p6, p7;
}

// 实际数据类继承 Padding
public class RingBufferFields<E> extends RingBufferPad {
    // 8 bytes padding * 7 = 56 bytes
    // 确保后面的字段不会与其他类共享 Cache Line
    
    protected long consumerIndex;
    protected long producerIndex;
    // ... 
}
```

后来 Disruptor 也转向使用 `@Contended` 注解。

## 五、解决方案一：缓存行填充（Padding）

### 5.1 手动填充的原理

通过在变量前后添加无用的 `long` 字段，把变量「撑」到独立的 Cache Line：

```java
public class PaddedCounter {
    // 前置填充：7 个 long = 56 字节
    // 确保 value 在独立的 Cache Line
    private long p1, p2, p3, p4, p5, p6, p7;
    
    // 真正需要缓存对齐的变量
    public volatile long value;
    
    // 后置填充：7 个 long = 56 字节
    // 防止后续变量侵入
    private long q1, q2, q3, q4, q5, q6, q7;
}
```

这样 `value` 变量前后各占 56 字节，确保它独占一个 Cache Line（64 字节）。

### 5.2 填充实现方式

方式一：**继承父类实现填充**

```java
// 父类只包含填充字段
class Padding {
    long p1, p2, p3, p4, p5, p6, p7;
}

// 子类包含实际数据
class Counter extends Padding {
    public volatile long value;
}

// 对象布局：
// [Padding: 56 bytes][value: 8 bytes][Padding: 56 bytes]
```

方式二：**使用继承链隔离**

```java
public class FalseSharingAvoidance {
    // 8 字节填充在父类
    private static class Padding {
        long p0, p1, p2, p3, p4, p5, p6, p7;
    }
    
    // 实际数据在子类
    private static class Counter extends Padding {
        volatile long value;
    }
}
```

方式三：**静态内部类**

```java
public class Example {
    private static class PaddedLong {
        public volatile long value;
        long p0, p1, p2, p3, p4, p5, p6, p7;  // 填充
    }
    
    private final PaddedLong counter1 = new PaddedLong();
    private final PaddedLong counter2 = new PaddedLong();
}
```

### 5.3 填充的代价

| 维度 | 影响 |
|---|---|
| 内存占用 | 每个填充变量额外占用约 120 字节（前置+后置） |
| 代码可读性 | 填充字段看起来像无用代码 |
| 维护成本 | 添加/删除字段时需要重新考虑布局 |
| 适用场景 | 高频竞争字段，值得牺牲内存换性能 |

:::warning 不要过度填充

填充是「用空间换时间」的策略。只对真正存在竞争的字段填充，不要为每个字段都添加填充，这会浪费大量内存。

典型的应用场景：
- 高并发计数器的 Cell
- Disruptor RingBuffer 的游标
- 分段锁的段计数器
- 性能监控的指标采集器

:::

## 六、解决方案二：@Contended 注解

### 6.1 @Contended 的历史

`@Contended` 是 JDK 8 引入的注解，最早应用于 JDK 内部的并发类（如 `LongAdder`、`ConcurrentHashMap`）。

完整路径是 `sun.misc.Contended`，这是一个内部 API：

```java
package sun.misc;

/**
 * Marks a field or class as being contention-prone.
 * The runtime will ensure each instance of the field
 * occupies its own cache line to prevent false sharing.
 */
@Retention(RetentionPolicy.RUNTIME)
@Target({ElementType.FIELD, ElementType.TYPE})
public @interface Contended {
    String value() default "";
}
```

JDK 9+ 将其移到了 `jdk.internal.vm.annotation.Contended`，但 `sun.misc.Contended` 仍然可用。

### 6.2 @Contended 的使用方式

**方式一：注解在字段上**

```java
public class ContendedField {
    @Contended
    volatile long counter1;
    
    @Contended
    volatile long counter2;
}
```

**方式二：注解在类上**

```java
@Contended
public class ContendedClass {
    volatile long field1;
    volatile long field2;
    volatile long field3;
}
```

两种方式的区别：

| 注解位置 | 效果 |
|---|---|
| `@Contended` 在类上 | 类中所有字段彼此隔离，每个字段独占 Cache Line |
| `@Contended` 在字段上 | 只有被注解的字段与其他字段隔离 |
| 无注解字段 + `@Contended` 字段 | 无注解字段可以与其他类共享 Cache Line |

### 6.3 @Contended 的内部实现

JVM 在分配对象时，会为 `@Contended` 字段添加额外的填充：

```
无 @Contended 的对象布局：

┌─────────────────────────────────┐
│ Object Header (12 bytes)       │
├─────────────────────────────────┤
│ field1 (8 bytes)               │
├─────────────────────────────────┤
│ field2 (8 bytes)               │
├─────────────────────────────────┤
│ padding (8 bytes)               │ ← 对象大小对齐
└─────────────────────────────────┘
<--- 可能在同一 Cache Line ------>

有 @Contended 的对象布局：

┌─────────────────────────────────┐
│ Object Header (12 bytes)       │
├─────────────────────────────────┤
│ field1 (8 bytes)               │
├─────────────────────────────────┤
│ padding (56 bytes)              │ ← 前置隔离
├─────────────────────────────────┤
│ field2 (8 bytes)               │ ← 独占 Cache Line
├─────────────────────────────────┤
│ padding (56 bytes)              │ ← 后置隔离
├─────────────────────────────────┤
│ padding (8 bytes)               │ ← 对象大小对齐
└─────────────────────────────────┘
```

### 6.4 @Contended 的组（Group）概念

`@Contended` 支持分组功能，用于优化内存布局：

```java
public class GroupedContended {
    @Contended("reader")
    volatile long readCount;
    
    @Contended("reader")
    volatile long readBytes;
    
    @Contended("writer")
    volatile long writeCount;
    
    @Contended("writer")
    volatile long writeBytes;
}
```

同一组的字段会共享填充空间，不同组之间隔离：

```
布局示意：

┌──────────────────────────────┐
│ Header + field1              │
├──────────────────────────────┤
│ padding (与 field2 隔离)     │
├──────────────────────────────┤
│ field2                       │ ← "reader" 组
├──────────────────────────────┤
│ padding (组间隔离)            │
├──────────────────────────────┤
│ Header + field3               │
├──────────────────────────────┤
│ padding (与 field4 隔离)      │
├──────────────────────────────┤
│ field4                       │ ← "writer" 组
├──────────────────────────────┤
│ padding                       │
└──────────────────────────────┘
```

这样，同一逻辑组内的字段可以更紧凑，不同组之间才需要隔离。

### 6.5 @Contended 的代价

```java
// 使用 @Contended 后的内存变化
public class MemoryTest {
    static class PlainClass {
        long a, b, c, d;  // 4 * 8 = 32 bytes + header
    }
    
    static class ContendedClass {
        @Contended
        long a;
        @Contended
        long b;
        @Contended
        long c;
        @Contended
        long d;
    }
}
```

预估内存占用：

| 类 | 字段大小 | @Contended 填充 | 总计 |
|---|---|---|---|
| PlainClass | 32 bytes | 0 | ~48 bytes（对齐后） |
| ContendedClass | 32 bytes | ~448 bytes（56 * 8） | ~496 bytes |

:::danger @Contended 的内存开销

每个 `@Contended` 字段前后各增加 56 字节填充。如果一个类有 10 个 `@Contended` 字段，仅填充就需要约 1120 字节（56 * 10 * 2）。

因此，**@Contended 只适合高频竞争的字段**，不适合普通业务代码。

:::

## 七、LongAdder 中 @Contended 的应用

### 7.1 LongAdder 的设计背景

`LongAdder` 是 Doug Lea 为解决 `AtomicLong` 的性能问题而设计的。

`AtomicLong` 使用 CAS（Compare-And-Swap）实现原子递增：

```java
// AtomicLong 的实现
public final long incrementAndGet() {
    long prev, next;
    do {
        prev = get();
        next = prev + 1;
    } while (!compareAndSet(prev, next));  // CAS 循环
    return next;
}
```

在高并发场景下，所有线程都在同一个变量上 CAS，大量 CAS 失败导致 CPU 空转。这就是**缓存行颠簸（Cache Line Thrashing）**。

### 7.2 LongAdder 的解决思路

`LongAdder` 采用了分段锁的思路：

```java
public class LongAdder extends AbstractLongAdder {
    // Cell 数组，分散竞争
    transient volatile Cell[] cells;
    
    // base 值，低竞争时直接使用
    transient volatile long base;
    
    public void increment() {
        add(1);
    }
    
    public void add(long x) {
        Cell[] as;
        long b, v;
        int m;
        Cell a;
        
        // 先尝试 CAS base
        if ((as = cells) != null ||
            !casBase(b = base, b + x)) {
            // base CAS 失败，进入分段累加
            boolean uncontended = true;
            int h = (ThreadLocalRandom.getProbe() & AS_MASK);
            if ((as != null && (m = as.length - 1) >= 0 &&
                (a = as[h]) != null &&
                (uncontended = a.cas(v = a.value, v + x))) ||
                // cells 未初始化或为空，尝试创建
                (h = advanceColor(h, cells, m, this)) >= 0)
                // ...
        }
    }
}
```

每个线程根据自己的 `ThreadLocalRandom` probe 值，累加到不同的 Cell 上，最终求和时将所有 Cell 累加。

### 7.3 Cell 中的 @Contended

```java
@sun.misc.Contended
static final class Cell {
    volatile long value;
    
    Cell(long initialValue) {
        value = initialValue;
    }
    
    final boolean cas(long expect, long update) {
        return UNSAFE.compareAndSwapLong(this, VALUE_OFFSET,
                                          expect, update);
    }
}
```

`Cell` 类上的 `@Contended` 确保每个 Cell 的 `value` 字段独占一个 Cache Line。

这样，即使多个线程同时累加，只要它们落在不同的 Cell 上，就不会有伪共享问题。

## 九、面试追问预判

### 追问一：Cache Line 的大小在不同平台上是否一致？

标准答案：

> 大多数现代 x86 和 ARM 处理器使用 **64 字节**的 Cache Line。但这不是绝对的：
>
> - 部分 ARM 处理器使用 32 字节
> - 一些老旧的处理器使用 128 字节
> - 实际测试更可靠
>
> Java 中可以用 `jdk.internal.vm.annotation.Contended` 的填充值（默认 128 字节，包括前后各 56 字节的隔离区）来确保跨平台兼容。

### 追问二：@Contended 为什么需要手动清理？

标准答案：

> `@Contended` 生成的填充数据是 JVM 运行时动态添加的，这些填充字节是对象内存布局的一部分，不是 Java 堆中的实际数据。
>
> 清理一般不是指删除填充，而是指：
> 1. 如果要通过 `jmap` 或 `dump` 分析堆内存，需要理解这些填充不会占用堆
> 2. 如果使用 `-XX:-RestrictContended` 关闭 @Contended 填充，填充会被移除
>
> 真正要「清理」的是：
> ```java
> // 关闭 @Contended 填充
> java -XX:-RestrictContended MyApplication
> ```
>
> 这在测试 @Contended 的性能影响时很有用。

### 追问三：除了填充，还有其他解决伪共享的方法吗？

标准答案：

> 除了填充，还有几种方法：
>
> **1. 调整数据布局**
> 把高竞争的字段和不竞争的字段分开到不同的对象中，确保它们不在同一个 Cache Line。
>
> **2. 使用线程局部变量**
> 让每个线程维护自己的副本，最后再合并。但这只适合最终结果只需要求和的场景（如 LongAdder）。
>
> **3. 使用无锁数据结构**
> 无锁数据结构通过 CAS 和内存序来减少缓存一致性流量。但 CAS 本身也可能导致伪共享。
>
> **4. 降低缓存一致性协议的敏感性**
> 使用 `volatile` 写时使用 `storestore` 屏障，写完立即发布，可以减少缓存行状态的切换。
>
> **填充仍然是目前最通用、最有效的解决方案**。

### 追问四：MESI 协议和总线锁有什么区别？

标准答案：

> **MESI 协议**是缓存一致性协议，用于管理多个 CPU 核心之间的缓存数据。它通过广播失效消息来协调，当某个核心修改了数据，其他核心的对应缓存行会被标记为失效。
>
> **总线锁**是更激进的方式。当 CPU 需要对某个内存地址进行原子操作（如 `lock` 前缀指令），它会通过 LOCK# 信号锁定系统总线，阻止其他 CPU 访问内存。
>
> | 特性 | MESI 协议 | 总线锁 |
> |---|---|---|
> | 粒度 | Cache Line（64 字节） | 任意内存地址 |
> | 性能 | 较轻量 | 非常重，会阻塞所有 CPU |
> | 使用场景 | 普通缓存一致性 | 真正的原子操作（如 xadd） |
>
> Java 中的 `synchronized` 在低竞争时会使用 MESI + 缓存行锁，在高竞争时会升级为总线锁。但随着处理器优化，缓存锁已经成为主流。
