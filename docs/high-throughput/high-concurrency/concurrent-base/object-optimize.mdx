# 高并发场景下的对象创建优化：对象复用与对象头压缩

你知道吗？在 Java 中创建一个普通对象，平均需要消耗 **数千个 CPU 周期**。

这听起来不可思议。我们习惯了 `new Object()` 一行代码搞定一切，觉得对象创建是理所当然的。但当你需要在高并发场景下每秒创建数十万个对象时，对象创建的成本就会变成系统的瓶颈。

更关键的是：对象创建不仅仅是分配内存那么简单。它涉及**对象头设置、构造函数执行、零值初始化**等一系列操作。理解这些底层细节，你才能写出真正高性能的 Java 代码。

今天，我们就来彻底剖析对象创建的成本，以及如何在高并发场景下进行对象优化。

## 一、对象创建的成本：比你想象的贵

### 1.1 对象创建的完整流程

当你执行 `new Object()` 时，JVM 做了以下事情：

```
1. 检查类是否已加载（ClassLoader）
       │
       ▼
2. 分配内存（Eden 区）
       │
       ▼
3. 将分配到的内存空间初始化为零值（零值初始化）
       │
       ▼
4. 设置对象头（Mark Word + Klass Pointer）
       │
       ▼
5. 执行构造函数（<init> 方法）
```

每一步都有成本。让我逐一分析。

### 1.2 分配内存的成本

对象的内存分配有两种方式：**指针碰撞（Bump the Pointer）**和**空闲列表（Free List）**。

**指针碰撞**适用于 Serial、ParNew 等使用标记-整理算法的 GC 收集器：

```
堆内存: [已分配][已分配][已分配][     ] ← 指针
                                        ↑
                               空闲起始位置

分配后: [已分配][已分配][已分配][新对象][     ]
```

JVM 只需要把指针向后移动对象大小的距离，复杂度是 **O(1)**。

**空闲列表**适用于 CMS 等使用标记-清除算法的收集器：

```
空闲列表: Entry(0x1000, 32KB) → Entry(0x2000, 16KB) → null
```

JVM 需要在列表中找到一块足够大的内存，复杂度是 **O(n)**。但实际 JVM 实现会用更高效的数据结构（如堆），接近 O(1)。

:::tip TLAB：减少并发分配的竞争

在高并发场景下，多个线程同时在 Eden 区分配对象会产生竞争。JVM 引入了 **TLAB（Thread Local Allocation Buffer）**：每个线程在 Eden 区预先分配一块线程私有的缓冲区，对象在 TLAB 内分配，无需加锁。

默认情况下，TLAB 大小约占 Eden 区的 1%。可以用 `-XX:TLABSize` 参数调整。

:::

### 1.3 零值初始化的成本

对象分配后，JVM 必须将对象的所有实例字段初始化为零值：

```java
public class User {
    private int id;        // 默认 0
    private String name;   // 默认 null
    private long balance;  // 默认 0L
}
```

这个零值初始化是必须的，因为**对象构造函数可能不初始化所有字段**：

```java
public class User {
    private int id;
    private String name;

    // 没有初始化 balance!
    public User(int id) {
        this.id = id;
    }
}
```

如果 `balance` 没有初始化，其默认值应该是 0，而不是随机值。

零值初始化的成本与对象大小成正比。**一个 1KB 的对象比 100 字节的对象初始化成本高 10 倍**。

### 1.4 对象头的设置

对象头是对象元数据的存储区域，它的设置也是不可忽视的成本。

32 位 JVM 的对象头结构：

```
|-------------------------------------------------------------|
|                      Mark Word (32 bits)                   |
|-------------------------------------------------------------|
|                    Klass Pointer (32 bits)                  |
|-------------------------------------------------------------|
|                      Instance Data                          |
|-------------------------------------------------------------|
```

64 位 JVM 的对象头结构：

```
|-------------------------------------------------------------|
|                      Mark Word (64 bits)                   |
|-------------------------------------------------------------|
|                    Klass Pointer (64 bits, 可能压缩)         |
|-------------------------------------------------------------|
|                      Instance Data                          |
|-------------------------------------------------------------|
```

对象头的大小取决于 JVM 配置，通常为 **12-16 字节**。

## 二、对象头的结构：你不知道的秘密

对象头是 JVM 内部的实现细节，但它对 Java 程序的性能和并发行为有深远影响。

### 2.1 Mark Word 的内容

Mark Word 存储对象的**哈希码、GC 分代年龄、锁状态**等信息：

| 存储内容 | 状态 |
|---|---|
| 对象哈希码 + 年龄 + 分代年龄 | 无锁态 |
| 对象哈希码 + 年龄 + 分代年龄 + 偏向线程 ID + 偏向时间戳 | 偏向锁 |
| 指向栈中锁记录的指针 | 轻量级锁 |
| 指向堆中 Monitor 对象的指针 | 重量级锁 |
| 标记 GC 分代年龄 | GC 标记 |

**Mark Word 只有 8 字节（64 位 JVM）**，却要存储这么多信息。JVM 的做法是：**根据锁状态不同，Mark Word 中存储的内容也不同**。

### 2.2 Mark Word 在不同状态下的结构

<Badge text="64-bit JVM" type="tip" /> 在 64 位 JVM 中，Mark Word 的结构如下：

```
无锁态（Unlocked）:
┌──────────────────────────────────────────────────────────────┐
│                     hashcode (25 bits)                       │
│        age (4 bits)  │   bias_lock (1 bit)  │ 01 (2 bits)   │
└──────────────────────────────────────────────────────────────┘

偏向锁（Biased）:
┌──────────────────────────────────────────────────────────────┐
│            thread (54 bits) │  epoch (2 bits)               │
│        age (4 bits)  │   bias_lock (1 bit)  │ 01 (2 bits)   │
└──────────────────────────────────────────────────────────────┘

轻量级锁（Lightweight Locked）:
┌──────────────────────────────────────────────────────────────┐
│                 指向栈中锁记录的指针 (62 bits)               │
│                                    │ 00 (2 bits)             │
└──────────────────────────────────────────────────────────────┘

重量级锁（Heavyweight Locked）:
┌──────────────────────────────────────────────────────────────┐
│              指向堆中 Monitor 对象的指针 (62 bits)            │
│                                    │ 10 (2 bits)              │
└──────────────────────────────────────────────────────────────┘

GC 标记（Marked for GC）:
┌──────────────────────────────────────────────────────────────┐
│                                    │ 11 (2 bits)             │
└──────────────────────────────────────────────────────────────┘
```

### 2.3 Klass Pointer：类的元信息

Klass Pointer 指向**方法区（MetaSpace）中类的元数据**：

```
┌────────────────────┐      ┌──────────────────────────────┐
│    对象头           │      │      方法区（元数据）          │
│  Klass Pointer ─────┼─────▶│  Klass 元数据                 │
│  (指向类元数据)     │      │  - 类的继承结构               │
└────────────────────┘      │  - 字段信息                    │
                            │  - 方法信息                    │
                            │  - 常量池                     │
                            └──────────────────────────────┘
```

在 32 位 JVM 中，Klass Pointer 是 4 字节；在 64 位 JVM 中是 8 字节。

### 2.4 数组长度：数组对象特有

如果对象是数组，还有一个额外的字段存储数组长度：

```
┌──────────────────────────────────────────────────────────────┐
│                      Mark Word (8 bytes)                     │
├──────────────────────────────────────────────────────────────┤
│                   Klass Pointer (8 bytes)                     │
├──────────────────────────────────────────────────────────────┤
│                     数组长度 (4 bytes)                        │
├──────────────────────────────────────────────────────────────┤
│                                                             │
│                       数组数据                               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

这就是为什么 `new int[0]` 仍然占用几十字节的内存。

## 三、对象头压缩：用更少的内存存更多的信息

64 位 JVM 的对象头比 32 位大 50%，这对内存使用是个挑战。JVM 提供了**对象头压缩**来解决这个问题。

### 3.1 指针压缩：-XX:+UseCompressedOops

<Badge text="JDK 6+" type="tip" /> 从 JDK 6 开始，JVM 引入了**指针压缩（Compressed Oops, Ordinary Object Pointers）**：

```bash
# 开启指针压缩（默认开启）
-XX:+UseCompressedOops

# 关闭指针压缩
-XX:-UseCompressedOops
```

指针压缩的原理：对于 64 位 JVM，理论上可以用 64 位（8 字节）来寻址 2^64 字节的内存空间。但实际上，一个 Java 进程的堆内存很少超过 32GB。

JVM 利用这一点：**把 64 位地址压缩成 32 位使用，通过偏移量计算真正的内存地址**。

```
压缩前（64 位指针）:
0x0000017A: 00000000 00000000 00000001 01111010  (实际 64 位地址)

压缩后（32 位指针）:
0x0000017A: 00000000 00000000 00000001 01111010  (压缩后存 32 位)
                                        ↓
JVM 在运行时通过基址 + 偏移量计算真实地址
真实地址 = Java Heap Base + (压缩指针 << 3)
```

### 3.2 指针压缩的生效条件

指针压缩不是万能的，它有局限性：

| 条件 | 说明 |
|---|---|
| 堆内存 ≤ 32GB | 如果堆内存超过 32GB，压缩指针无法表示所有地址 |
| 开启 UseCompressedOops | 默认开启 |

可以用以下命令查看压缩状态：

```bash
java -XX:+PrintFlagsFinal -version | grep Compressed
```

输出示例：

```
bool UseCompressedOops   = true   {lp64_product}
bool UseCompressedClassPointers = true   {lp64_product}
```

### 3.3 Compressed Class Space

<Badge text="JDK 8+" type="tip" /> JDK 8 引入了 **Metaspace** 替代 PermGen，同时引入了 **Compressed Class Space**：

```bash
# 类指针压缩空间的最大大小（默认 1GB）
-XX:CompressedClassSpaceSize=1G
```

如果类的元数据超过 Compressed Class Space 的容量（但仍在 Metaspace 范围内），类指针将不再压缩，导致对象头变大。

## 四、对象池化：对象复用的艺术

理解了对象创建的成本后，一个自然的想法是：**复用已创建的对象，减少频繁的对象分配和 GC**。

### 4.1 什么时候对象复用有意义

不是所有对象都值得复用。判断标准：

| 对象类型 | 是否值得复用 | 原因 |
|---|---|---|
| 数据库连接 | 是 | 创建成本极高，涉及网络 IO |
| 线程 | 是 | 创建成本高，线程池复用 |
| HTTP 连接 | 是 | 创建成本高，涉及 TCP 握手 |
| 大型缓存对象 | 是 | 减少 GC 压力 |
| 简单 String/Integer | 否 | 创建成本低，JVM 已优化 |
| 基本类型包装类（-128~127） | 否 | JVM 已自动缓存 |

### 4.2 简单对象的创建成本没有那么高

很多人认为"对象创建很慢"，但实际上 JVM 对**小对象的创建做了大量优化**。

对象的分配（在 TLAB 内）是** bump-the-pointer**，只需要移动指针。Eden 区满时，整个 Survivor 区一起复制，分配速度可以达到**每秒数亿次**。

所以，对于普通的业务对象，不要过度追求对象复用。**过早优化是万恶之源**。

真正需要优化的是**重量级对象**，如：

- 数据库连接
- 网络连接
- 大型数据结构（如大数组、大 List）

### 4.3 常用对象池化技术

**技术一：String.intern()**

`String.intern()` 返回字符串的**字符串常量池**版本：

```java
String s1 = new String("hello");           // 堆中创建对象
String s2 = s1.intern();                    // 返回常量池中的引用
String s3 = "hello";                        // 常量池中的字符串

System.out.println(s1 == s2);  // false：s1 在堆，s2/s3 在常量池
System.out.println(s2 == s3);  // true：同一个引用
```

<Badge text="JDK 7+" type="tip" /> 在 JDK 7 之前，字符串常量池在 **PermGen（永久代）**，大小受限且 Full GC 也会清理。JDK 7 开始，**字符串常量池移到了 Heap 区**，可以更放心地使用 `intern()`。

适用场景：大量重复的字符串，比如从数据库读取的固定枚举值。

**技术二：Integer.valueOf() 缓存**

```java
Integer a = 127;    // 自动装箱，实际调用 Integer.valueOf(127)
Integer b = 127;    // 返回同一个对象

System.out.println(a == b);  // true

Integer c = 128;
Integer d = 128;

System.out.println(c == d);  // false：超过缓存范围
```

JVM 默认缓存 **-128 到 127** 的 Integer。可以用参数调整：

```bash
# 调整缓存范围为 -128 到 256
-XX:AutoBoxCacheMax=256
```

**技术三：数据库连接池**

数据库连接的创建涉及：TCP 握手、SSL 认证、数据库认证、协议握手等，耗时可能长达 **100-500ms**。

```java
// HikariCP 配置示例
HikariConfig config = new HikariConfig();
config.setJdbcUrl("jdbc:mysql://localhost:3306/test");
config.setUsername("root");
config.setPassword("password");
config.setMaximumPoolSize(20);       // 最大连接数
config.setMinimumIdle(5);            // 最小空闲连接数
config.setConnectionTimeout(30000);  // 获取连接超时

HikariDataSource ds = new HikariDataSource(config);
Connection conn = ds.getConnection();  // 从池中获取，不是新建
```

**技术四：HTTP 连接池**

HTTP 连接复用同样重要：

```java
// Apache HttpClient 连接池
PoolingHttpClientConnectionManager cm = new PoolingHttpClientConnectionManager();
cm.setMaxTotal(200);                // 最大连接数
cm.setDefaultMaxPerRoute(20);       // 每个路由最大连接数

CloseableHttpClient client = HttpClients.custom()
    .setConnectionManager(cm)
    .build();

HttpGet request = new HttpGet("http://api.example.com/data");
// 复用连接，不需要每次都新建 TCP 连接
CloseableHttpResponse response = client.execute(request);
```

## 五、实战：减少对象创建的建议

### 5.1 StringBuilder 复用

字符串拼接是常见的操作，大量创建 StringBuilder 会增加 GC 压力：

```java
// 低效：每次循环创建新的 StringBuilder
public String buildQuery(List<String> fields) {
    StringBuilder sb = new StringBuilder();
    for (String field : fields) {
        sb.append(field).append(",");
    }
    return sb.toString();
}

// 高效：复用 StringBuilder
public String buildQueryOptimized(List<String> fields) {
    StringBuilder sb = new StringBuilder(estimateSize(fields));
    for (String field : fields) {
        sb.append(field).append(",");
    }
    return sb.toString();
}
```

如果 StringBuilder 在方法内复用，记得**在复用前清空**：

```java
// 清空 StringBuilder 的方法
sb.setLength(0);  // 推荐：设置长度为 0，不重新分配内存
// sb.delete(0, sb.length());  // 也可以，但不推荐
```

### 5.2 批量处理减少对象创建

在高并发场景下，批量处理可以显著减少对象创建：

```java
// 低效：逐条处理，每条数据都创建对象
public void processOrders(List<Order> orders) {
    for (Order order : orders) {
        OrderDTO dto = new OrderDTO();
        dto.setId(order.getId());
        dto.setAmount(order.getAmount());
        // ...
        sendToKafka(dto);
    }
}

// 高效：批量处理，减少对象创建
public void processOrdersBatch(List<Order> orders) {
    List<OrderDTO> batch = new ArrayList<>(BATCH_SIZE);
    for (Order order : orders) {
        batch.add(toDTO(order));
        if (batch.size() >= BATCH_SIZE) {
            sendBatchToKafka(batch);
            batch.clear();
        }
    }
    if (!batch.isEmpty()) {
        sendBatchToKafka(batch);
    }
}
```

### 5.3 避免在循环中创建对象

这是一个常见的性能陷阱：

```java
// 低效：循环内创建对象
for (int i = 0; i < 10000; i++) {
    String key = "user_" + i;        // 每次创建新 String
    String sql = "SELECT * FROM users WHERE id = " + i;  // 每次创建新 StringBuilder
    executeQuery(sql);
}

// 高效：提取不变的部分
String prefix = "user_";
String sqlTemplate = "SELECT * FROM users WHERE id = ";
for (int i = 0; i < 10000; i++) {
    String key = prefix + i;         // 仍然创建新 String，但减少了 StringBuilder
    String sql = sqlTemplate + i;
    executeQuery(sql);
}

// 更高效：使用 StringBuilder 预分配
StringBuilder sb = new StringBuilder(estimateSize);
for (int i = 0; i < 10000; i++) {
    sb.setLength(0);
    sb.append("SELECT * FROM users WHERE id = ").append(i);
    executeQuery(sb.toString());
}
```

### 5.4 使用对象池框架

对于需要频繁创建的对象，可以使用对象池框架：

**Commons Pool2：通用对象池**

```java
GenericObjectPoolConfig<ExpensiveObject> config = new GenericObjectPoolConfig<>();
config.setMaxTotal(100);
config.setMaxIdle(50);
config.setMinIdle(10);

GenericObjectPool<ExpensiveObject> pool = new GenericObjectPool<>(
    new ExpensiveObjectFactory(), config);

// 获取对象
ExpensiveObject obj = pool.borrowObject();
try {
    obj.process();
} finally {
    pool.returnObject(obj);  // 归还对象，不是销毁
}
```

**Spring 的 SimpleBufferingClientHttpRequestFactory：HTTP 连接池**

```java
@Configuration
public class RestTemplateConfig {
    @Bean
    public RestTemplate restTemplate() {
        SimpleClientHttpRequestFactory factory = new SimpleClientHttpRequestFactory();
        factory.setConnectTimeout(5000);
        factory.setReadTimeout(10000);

        // 开启 HTTP 连接池
        HttpComponentsClientHttpRequestFactory httpFactory = new HttpComponentsClientHttpRequestFactory();
        httpFactory.setHttpClient(HttpClients.custom()
            .setMaxTotal(200)
            .setDefaultMaxPerRoute(20)
            .build());

        return new RestTemplate(httpFactory);
    }
}
```

## 六、面试追问预判

### 追问一：对象的内存布局是什么？

对象的内存布局分为三部分：

1. **对象头（Object Header）**
   - Mark Word：锁状态、GC 分代年龄、哈希码
   - Klass Pointer：指向类元数据的指针
   - 数组长度（仅数组对象）

2. **实例数据（Instance Data）**
   - 父类继承的字段
   - 子类定义的字段
   - 按源码顺序排列（子类字段在父类字段后）

3. **对齐填充（Padding）**
   - JVM 要求对象起始地址是 8 字节的倍数
   - 不足的部分用 padding 填充

### 追问二：synchronized 锁的对象头发生了什么？

synchronized 的锁升级过程反映在对象头的 Mark Word 中：

```
无锁 → 偏向锁 → 轻量级锁 → 重量级锁

1. 无锁态：Mark Word 存储对象的 hashCode、GC 年龄
2. 偏向锁：Mark Word 记录偏向线程 ID，后续该线程进入同步块无需 CAS
3. 轻量级锁：Mark Word 复制到线程栈帧的锁记录，替换为指向锁记录的指针
4. 重量级锁：Mark Word 替换为指向 Monitor 对象的指针，线程阻塞等待
```

这就是为什么说**偏向锁适合单线程重复进入的场景，轻量级锁适合多线程交替进入的场景，重量级锁适合多线程竞争的场景**。

### 追问三：如何查看对象的内存布局？

可以使用 JOL（Java Object Layout）工具查看：

```java
// 添加依赖
// <dependency>
//     <groupId>org.openjdk.jol</groupId>
//     <artifactId>jol-core</artifactId>
//     <version>0.16</version>
// </dependency>

import org.openjdk.jol.info.ClassLayout;
import org.openjdk.jol.vm.VM;

public class ObjectLayoutDemo {
    public static void main(String[] args) {
        System.out.println(VM.current().details());

        Object obj = new Object();
        System.out.println(ClassLayout.parseInstance(obj).toPrintable());

        // 开启偏向锁延迟后，查看无锁状态
        // -XX:BiasedLockingStartupDelay=0
    }
}
```

输出示例：

```

# JDK_JAVA_OPTIONS='--add-opens java.base/jdk.internal.misc=ALL-UNNAMED'.
java.lang.Object object internals:
OFF  SZ               TYPE DESCRIPTION
  0   8                    (object header: mark)
  8   4                    (object header: class)
 12   4                    (object header: alignment padding)
Instance size: 16 bytes
```

### 追问四：数组的对象头比普通对象大多少？

数组额外占用 4 字节存储数组长度：

```java
// int[3] 的对象头 = 12 bytes (mark) + 4 bytes (class ptr) + 4 bytes (length) = 20 bytes
// 实际占用（对齐后）= 24 bytes

// int[0] 仍然占用 24 bytes，而不是 0
```

这就是为什么"稀疏数组"不一定节省内存——如果大量元素是 null 或 0，使用 `int[]` 可能比 `Map<Integer, Integer>` 占用更多内存。

对象创建的成本、对象头的结构、对象池化技术……这些知识点看似分散，其实都指向同一个核心：**在 Java 中，"创建对象"不是免费的，你需要理解它的代价**。

但我还想留给你一个问题：如果 JVM 有一天能做到 **Zero-copy 对象分配**（分配时直接复用已释放的内存区域，完全绕过 GC），Java 的并发性能会提升多少？

这个问题涉及到 ZGC、Shenandoah 等新一代 GC 收集器的设计思想。深入理解 JVM 底层，你的并发编程能力会上一个台阶。
