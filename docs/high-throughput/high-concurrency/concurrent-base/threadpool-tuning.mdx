# 线程池调优：核心参数与业务场景匹配

凌晨2点，你正准备上线一个新功能，压测报告显示系统能扛住5000 QPS。代码review通过，测试环境全绿，信心满满。

然后生产环境炸了。

接口响应时间从10ms飙到2秒，CPU直接打满。查日志，发现线程池队列积压了十几万任务。紧急扩容、加服务器，问题依旧——因为瓶颈根本不在机器数量，而在**线程池配置**。

这就是线程池调优的核心矛盾：大多数人对它只停留在「会用」，而不知道它的工作原理和配置边界。

## 一、线程池的七大核心参数

`ThreadPoolExecutor` 是 Java 并发包中最核心的线程池实现，它的构造函数有七个参数：

```java
public ThreadPoolExecutor(
    int corePoolSize,              // 核心线程数
    int maximumPoolSize,           // 最大线程数
    long keepAliveTime,            // 空闲线程存活时间
    TimeUnit unit,                 // 时间单位
    BlockingQueue<Runnable> workQueue,      // 任务队列
    ThreadFactory threadFactory,   // 线程工厂
    RejectedExecutionHandler handler        // 拒绝策略
)
```

### 1.1 corePoolSize：核心线程数

核心线程是线程池的「常驻人口」。即使没有任务执行，这些线程也不会被回收（除非设置了 `allowCoreThreadTimeOut`）。

线程池创建后，默认不会立即创建核心线程。只有当有任务提交时，才会懒加载创建第一个核心线程。这是为了节省资源——如果系统启动后没有任务，何必提前创建线程？

```java
// 创建一个核心线程数为 5 的线程池
ExecutorService executor = new ThreadPoolExecutor(
    5,      // 核心线程数
    10,     // 最大线程数
    60L,    // 空闲存活时间
    TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(1000),
    Executors.defaultThreadFactory(),
    new ThreadPoolExecutor.AbortPolicy()
);

// 懒加载：提交第一个任务时才创建核心线程
executor.execute(() -> {
    System.out.println("第一个任务，触发创建核心线程");
});
```

### 1.2 maximumPoolSize：最大线程数

最大线程数是线程池的「上限」。当核心线程都在忙，且队列满了，新任务会触发创建临时线程。临时线程在空闲超过 `keepAliveTime` 后会被回收。

这里有一个关键问题：**核心线程和临时线程有什么区别？**

答案是：**没有本质区别**。它们都是普通线程，唯一的区别是回收策略不同。核心线程默认不会被回收（除非开启 `allowCoreThreadTimeOut`），而临时线程在空闲时会被回收。

### 1.3 keepAliveTime + unit：空闲线程存活时间

控制临时线程的存活时间。默认情况下，只有超过核心线程数的线程才会受这个参数影响。

```java
// 配置：核心线程 5 个，最大 20 个，空闲线程存活 60 秒
new ThreadPoolExecutor(
    5, 20, 60L, TimeUnit.SECONDS, 
    new LinkedBlockingQueue<>(1000)
);
```

场景解释：当队列积压超过1000个任务时，会创建临时线程（最多到20个）。这些临时线程在空闲60秒后会被回收。

### 1.4 workQueue：任务队列

线程池使用阻塞队列来存储等待执行的任务。常用的队列有：

| 队列类型 | 特点 | 适用场景 |
|---|---|---|
| `LinkedBlockingQueue` | 无界队列（默认 Integer.MAX_VALUE） | 任务量可控，不会突然爆发 |
| `ArrayBlockingQueue` | 有界队列，需要指定容量 | 需要严格控制内存使用 |
| `SynchronousQueue` | 不存储任务，直接交付 | 任务执行速度很快，不希望积压 |
| `PriorityBlockingQueue` | 优先级队列 | 需要按优先级执行任务 |

**重要原则**：`LinkedBlockingQueue` 看似方便，但如果任务生产速度持续大于消费速度，会导致内存无限膨胀，最终 OOM。

### 1.5 threadFactory：线程工厂

用于创建线程。你可以自定义线程工厂，设置线程名前缀、守护线程、优先级等。

```java
ThreadFactory customFactory = new ThreadFactory() {
    private final AtomicInteger count = new AtomicInteger(1);
    
    @Override
    public Thread newThread(Runnable r) {
        Thread thread = new Thread(r);
        thread.setName("custom-pool-" + count.getAndIncrement());
        thread.setDaemon(false);  // 用户线程，不要设为守护线程
        thread.setPriority(Thread.NORM_PRIORITY);
        return thread;
    }
};
```

### 1.6 handler：拒绝策略

当线程池和队列都满了，无法处理新任务时的处理策略。

```java
// 四种内置拒绝策略
new ThreadPoolExecutor.AbortPolicy()          // 抛RejectedExecutionException
new ThreadPoolExecutor.CallerRunsPolicy()      // 由调用线程执行
new ThreadPoolExecutor.DiscardPolicy()         // 直接丢弃
new ThreadPoolExecutor.DiscardOldestPolicy()   // 丢弃队列最旧的任务
```

## 二、线程池的工作原理

### 2.1 任务提交流程图

线程池的任务处理流程是理解线程池的关键：

```
                    ┌─────────────────────────────────────────────────┐
                    │                  提交任务                       │
                    └─────────────────────┬─────────────────────────┘
                                          │
                                          ▼
                    ┌─────────────────────────────────────────────────┐
                    │           是否小于 corePoolSize？               │
                    └─────────────────────┬─────────────────────────┘
                                          │
                     ┌────────────────────┴────────────────────┐
                     │ Yes                                     │ No
                     ▼                                         ▼
    ┌────────────────────────────────┐    ┌────────────────────────────────────┐
    │  创建核心线程执行任务            │    │         队列是否已满？              │
    └────────────────────────────────┘    └─────────────────┬──────────────────┘
                                                           │
                                            ┌───────────────┴───────────────┐
                                            │ Yes                          │ No
                                            ▼                               ▼
                ┌────────────────────────────────┐    ┌────────────────────────────────┐
                │  是否小于 maximumPoolSize？    │    │     加入队列，等待执行          │
                └────────────────┬───────────────┘    └────────────────────────────────┘
                                 │
                ┌────────────────┴────────────────┐
                │ Yes                           │ No
                ▼                                ▼
    ┌───────────────────────────┐    ┌─────────────────────────────────┐
    │  创建临时线程执行任务       │    │       执行拒绝策略                │
    └───────────────────────────┘    └─────────────────────────────────┘
```

### 2.2 线程创建时机

根据流程图，线程的创建遵循以下规则：

1. **第一个任务提交时**：创建第一个核心线程
2. **核心线程满，队列未满**：任务进入队列等待
3. **队列满，线程数未达上限**：创建临时线程
4. **队列满，线程数已达上限**：执行拒绝策略

### 2.3 一个关键问题：先创建线程还是先入队？

答案是：**先尝试创建线程，再入队**。

假设 `corePoolSize=5`，`maximumPoolSize=10`，队列容量100。

当第1-5个任务提交时，会创建5个核心线程执行。
当第6-105个任务提交时，核心线程都在忙，任务进入队列。
当第106个任务提交时，队列满了，此时会创建临时线程（如果未达上限）执行。

```java
// 验证流程
new ThreadPoolExecutor(2, 4, 60L, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(2),
    Executors.defaultThreadFactory(),
    (r, e) -> System.out.println("任务被拒绝")
).execute(() -> {
    try {
        Thread.sleep(1000);
        System.out.println("任务执行");
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
    }
});

// 分析：
// 任务1-2: 创建核心线程执行
// 任务3-4: 进入队列等待
// 任务5: 队列满了，创建临时线程执行
// 任务6: 线程数已达上限(4)，执行拒绝策略
```

## 三、四种拒绝策略详解

### 3.1 AbortPolicy（默认策略）

抛出一个 `RejectedExecutionException`，任务既不执行也不加入队列。

```java
public static class AbortPolicy implements RejectedExecutionHandler {
    public void rejectedExecution(Runnable r, ThreadPoolExecutor e) {
        throw new RejectedExecutionException(
            "Task " + r.toString() + " rejected from " +
            e.toString()
        );
    }
}
```

适用场景：需要严格保证任务不被丢失的场景，通常配合监控告警使用。

### 3.2 CallerRunsPolicy

由提交任务的线程执行。如果调用者线程也被占满，则会阻塞。这是一种「兜底」策略。

```java
public static class CallerRunsPolicy implements RejectedExecutionHandler {
    public void rejectedExecution(Runnable r, ThreadPoolExecutor e) {
        if (!e.isShutdown()) {
            r.run();  // 直接在调用线程执行
        }
    }
}
```

适用场景：想要任务「最终」被执行，但不在乎由哪个线程执行。缺点是会拖慢调用方。

### 3.3 DiscardPolicy

静默丢弃任务，不抛异常，不留痕迹。

```java
public static class DiscardPolicy implements RejectedExecutionHandler {
    public void rejectedExecution(Runnable r, ThreadPoolExecutor e) {
        // 什么都不做
    }
}
```

适用场景：任务可以丢失，但系统不能崩溃。比如日志收集、监控埋点等非核心任务。

### 3.4 DiscardOldestPolicy

丢弃队列中最老的任务，然后重新尝试提交。

```java
public static class DiscardOldestPolicy implements RejectedExecutionHandler {
    public void rejectedExecution(Runnable r, ThreadPoolExecutor e) {
        if (!e.isShutdown()) {
            e.getQueue().poll();  // 丢弃最老的任务
            e.execute(r);         // 重新提交当前任务
        }
    }
}
```

适用场景：优先执行新任务的场景。比如最新行情数据比旧数据更重要。

## 四、四种常见线程池类型

`Executors` 工具类提供了几种预定义的线程池：

### 4.1 FixedThreadPool：固定大小线程池

```java
ExecutorService executor = Executors.newFixedThreadPool(10);
```

核心线程数 = 最大线程数 = 固定值（10），无界队列（`Integer.MAX_VALUE`）。

```java
public static ExecutorService newFixedThreadPool(int nThreads) {
    return new ThreadPoolExecutor(
        nThreads,      // 核心线程数 = 最大线程数
        nThreads,      // 最大线程数
        0L,            // 空闲存活时间（0，因为线程数固定，不会回收）
        TimeUnit.MILLISECONDS,
        new LinkedBlockingQueue<Runnable>()  // 无界队列
    );
}
```

**风险**：如果任务生产速度持续大于消费速度，队列会无限膨胀，最终 OOM。

### 4.2 CachedThreadPool：缓存线程池

```java
ExecutorService executor = Executors.newCachedThreadPool();
```

核心线程数 = 0，最大线程数 = `Integer.MAX_VALUE`，使用 `SynchronousQueue`（不存储任务）。

```java
public static ExecutorService newCachedThreadPool() {
    return new ThreadPoolExecutor(
        0,                          // 核心线程数 0
        Integer.MAX_VALUE,          // 最大线程数无限
        60L,                        // 空闲存活 60 秒
        TimeUnit.SECONDS,
        new SynchronousQueue<Runnable>()  // 不存储任务
    );
}
```

**风险**：如果任务提交速度过快，会创建大量线程，最终 OOM。

### 4.3 ScheduledThreadPool：定时线程池

```java
ScheduledExecutorService executor = Executors.newScheduledThreadPool(4);
```

支持定时任务和周期任务。

```java
ScheduledExecutorService executor = 
    Executors.newScheduledThreadPool(4);

// 延迟 3 秒后执行
executor.schedule(() -> System.out.println("3秒后执行"), 
    3, TimeUnit.SECONDS);

// 延迟 1 秒后开始执行，之后每 2 秒执行一次
executor.scheduleAtFixedRate(() -> System.out.println("周期执行"),
    1, 2, TimeUnit.SECONDS);
```

### 4.4 SingleThreadExecutor：单线程池

```java
ExecutorService executor = Executors.newSingleThreadExecutor();
```

只有一个线程，保证任务顺序执行。

```java
public static ExecutorService newSingleThreadExecutor() {
    return new FinalizableDelegatedExecutorService(
        new ThreadPoolExecutor(
            1, 1, 0L, TimeUnit.MILLISECONDS,
            new LinkedBlockingQueue<Runnable>()
        )
    );
}
```

适用场景：需要保证任务顺序执行的场景，如日志写入、消息处理等。

## 五、CPU密集型 vs IO密集型线程池配置

这是面试超高频问题，也是生产环境配置的核心依据。

### 5.1 核心判断标准

- **CPU密集型（CPU-bound）**：任务主要是计算、逻辑处理，CPU一直是100%，很少等待IO。如加密、压缩、排序、大量计算。
- **IO密集型（IO-bound）**：任务大部分时间在等待IO操作完成（网络、磁盘、数据库），CPU经常处于空闲状态。如文件读取、网络请求、数据库查询。

### 5.2 配置公式

**CPU密集型线程数配置**：

```
线程数 = CPU核心数 + 1
```

为什么 +1？因为计算任务偶尔会因为Page Fault、硬件中断等原因被挂起，多一个线程可以充分利用CPU。

```java
// 获取 CPU 核心数
int cpuCores = Runtime.getRuntime().availableProcessors();

// CPU 密集型线程池
ExecutorService cpuExecutor = new ThreadPoolExecutor(
    cpuCores + 1,    // 核心线程数
    cpuCores + 1,    // 最大线程数（通常不设大）
    0L, TimeUnit.MILLISECONDS,
    new LinkedBlockingQueue<>(1000),
    customFactory,
    new ThreadPoolExecutor.AbortPolicy()
);
```

**IO密集型线程数配置**：

```
线程数 = CPU核心数 × 2（经验值）

或者更精确：
线程数 = CPU核心数 × (1 + IO等待时间 / CPU计算时间)
```

```java
// IO 密集型线程池
ExecutorService ioExecutor = new ThreadPoolExecutor(
    cpuCores * 2,           // 核心线程数
    cpuCores * 4,           // 最大线程数（留有余量）
    60L, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(1000),
    customFactory,
    new ThreadPoolExecutor.AbortPolicy()
);
```

### 5.3 实际配置建议

| 场景 | 核心线程数 | 最大线程数 | 队列类型 |
|---|---|---|---|
| CPU密集型计算 | `CPU核心数 + 1` | 同核心线程数 | 有界队列 |
| IO密集型（低并发） | `CPU核心数 × 2` | `CPU核心数 × 4` | 有界队列 |
| IO密集型（高并发） | `CPU核心数 × 2` | `Integer.MAX_VALUE` | `SynchronousQueue` |
| 混合型 | `CPU核心数 × (1 + IO比)` | 可设大 | 有界队列 |

## 六、为什么不能用 Executors 创建线程池

阿里《Java开发手册》强制规定：**禁止使用 `Executors` 创建线程池，必须使用 `ThreadPoolExecutor` 构造方法。**

### 6.1 FixedThreadPool 和 SingleThreadExecutor 的问题

```java
// 问题：无界队列可能导致 OOM
ExecutorService executor = Executors.newFixedThreadPool(10);
```

核心线程数固定，队列无界。如果任务提交速度持续大于处理速度，队列会无限膨胀，最终导致内存溢出。

### 6.2 CachedThreadPool 的问题

```java
// 问题：最大线程数无限，可能创建过多线程导致 OOM
ExecutorService executor = Executors.newCachedThreadPool();
```

SynchronousQueue 不存储任务，当任务提交速度过快时，会创建大量线程，最终导致系统崩溃。

### 6.3 正确的创建方式

```java
// 推荐：明确指定队列大小
ExecutorService executor = new ThreadPoolExecutor(
    10,      // 核心线程数
    50,      // 最大线程数
    60L,     // 空闲存活时间
    TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(2000),  // 有界队列，控制内存
    new ThreadFactoryBuilder()
        .setNameFormat("business-pool-%d")
        .build(),
    new ThreadPoolExecutor.AbortPolicy()  // 有界拒绝策略
);
```

## 七、Spring Boot 中的线程池配置

### 7.1 使用 @Bean 配置线程池

```java
@Configuration
public class ThreadPoolConfig {
    
    @Bean(name = "taskExecutor")
    public Executor taskExecutor() {
        int cpuCores = Runtime.getRuntime().availableProcessors();
        
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(cpuCores * 2);
        executor.setMaxPoolSize(cpuCores * 4);
        executor.setQueueCapacity(2000);
        executor.setKeepAliveSeconds(60);
        executor.setThreadNamePrefix("spring-pool-");
        
        // 拒绝策略：当队列满了，由调用线程执行
        executor.setRejectedExecutionHandler(
            new ThreadPoolExecutor.CallerRunsPolicy()
        );
        
        // 等待所有任务完成后再关闭线程池
        executor.setWaitForTasksToCompleteOnShutdown(true);
        executor.setAwaitTerminationSeconds(60);
        
        executor.initialize();
        return executor;
    }
}
```

### 7.2 异步任务 @Async 使用

```java
@Service
public class AsyncService {
    
    @Async("taskExecutor")
    public void executeAsyncTask(String taskName) {
        log.info("执行异步任务: {}", taskName);
    }
}

@SpringBootApplication
@EnableAsync  // 开启异步支持
public class Application {
    public static void main(String[] args) {
        SpringApplication.run(Application.class, args);
    }
}
```

## 八、优雅关闭线程池

线程池的关闭比想象中重要。如果关闭不当，可能导致任务丢失或线程被强制中断。

### 8.1 两种关闭方法

```java
// 1. shutdown：拒绝新任务，等待已提交任务完成
executor.shutdown();

try {
    // 等待最多 30 秒
    if (!executor.awaitTermination(30, TimeUnit.SECONDS)) {
        // 超时，强制关闭
        executor.shutdownNow();
    }
} catch (InterruptedException e) {
    executor.shutdownNow();
    Thread.currentThread().interrupt();
}

// 2. shutdownNow：尝试停止所有正在执行的任务，返回等待执行的任务列表
List<Runnable> rejectedTasks = executor.shutdownNow();
```

### 8.2 优雅关闭的最佳实践

```java
public class ThreadPoolHelper {
    
    private static ExecutorService executor;
    
    public static void init() {
        executor = new ThreadPoolExecutor(
            10, 50, 60L, TimeUnit.SECONDS,
            new LinkedBlockingQueue<>(2000),
            r -> {
                Thread t = new Thread(r);
                t.setName("helper-pool-" + t.getId());
                return t;
            }
        );
    }
    
    public static void shutdown() {
        if (executor == null) return;
        
        // 拒绝新任务
        executor.shutdown();
        
        try {
            // 等待 60 秒
            if (!executor.awaitTermination(60, TimeUnit.SECONDS)) {
                // 超时，强制停止
                List<Runnable> tasks = executor.shutdownNow();
                System.out.println("强制停止，未完成任务数: " + tasks.size());
            }
        } catch (InterruptedException e) {
            executor.shutdownNow();
            Thread.currentThread().interrupt();
        }
        
        System.out.println("线程池已关闭");
    }
}
```

## 九、监控指标与最佳实践

### 9.1 核心监控指标

```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(10, 50, 60L, 
    TimeUnit.SECONDS, new LinkedBlockingQueue<>(1000));

// 获取当前状态
int activeCount = executor.getActiveCount();        // 活跃线程数
int queueSize = executor.getQueue().size();          // 队列大小
long completedTaskCount = executor.getCompletedTaskCount();  // 已完成任务数

// APM 埋点示例
public class MonitoredThreadPoolExecutor extends ThreadPoolExecutor {
    
    public MonitoredThreadPoolExecutor(...) {
        super(...);
    }
    
    @Override
    protected void beforeExecute(Thread t, Runnable r) {
        Metrics.counter("threadpool.task.submitted").increment();
        super.beforeExecute(t, r);
    }
    
    @Override
    protected void afterExecute(Runnable r, Throwable t) {
        Metrics.timer("threadpool.task.duration").record(
            System.currentTimeMillis() - startTime
        );
        super.afterExecute(r, t);
    }
}
```

### 9.2 最佳实践清单

```markdown
- [ ] 禁止使用 Executors 创建线程池，必须使用 ThreadPoolExecutor
- [ ] 队列必须设置有界，避免无界队列导致 OOM
- [ ] 根据业务类型（CPU密集/IO密集）合理配置核心线程数
- [ ] 线程池拒绝策略要配合监控告警
- [ ] 应用关闭时必须优雅关闭线程池
- [ ] 给线程池中的线程设置有意义的名称
- [ ] 核心线程数不是越大越好，要考虑上下文切换开销
- [ ] 任务要有合理的超时机制
- [ ] 监控队列大小、活跃线程数、任务拒绝次数
```

## 总结

线程池是 Java 并发编程中最核心的工具之一，也是面试超高频考点。理解七大核心参数的工作原理、掌握CPU密集型和IO密集型的配置区别、知道为什么要禁用 `Executors`，这是每个 Java 工程师的必备技能。

## 面试追问预判

1. **如果线程池的队列设置了有界，但任务处理速度跟不上怎么办？**
   
   这是一个典型的「背压」问题。解决方案包括：
   - 增加消费者线程数
   - 使用 `CallerRunsPolicy` 让调用方帮忙消化
   - 任务降级：非核心任务直接丢弃
   - 限流：从源头控制任务提交速度

2. **线程池的线程为什么不会被GC回收？**
   
   因为线程池持有线程引用，而线程本身持有对象头信息。线程是 JVM 层面的执行单元，不受普通 GC 管理。即使线程池中的任务对象被回收，线程本身（及其栈中的局部变量）仍然存活。

3. **如何动态调整线程池的核心线程数？**
   
   `ThreadPoolExecutor` 提供了 `setCorePoolSize()` 和 `setMaximumPoolSize()` 方法，可以在运行时动态调整。不过需要注意，调整核心线程数会影响是否创建或回收核心线程。
