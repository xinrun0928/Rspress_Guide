# CAS 优化与 LongAdder 高并发性能对比

「synchronized 太重了，我们用 CAS 吧。」

很多开发者把 CAS 当成万能钥匙，认为只要用了 `AtomicInteger`、`AtomicLong`，并发问题就解决了。

但他们不知道，CAS 在高并发下有三大致命问题：ABA 问题、循环开销大、只能保证一个变量。

更不知道的是，JDK 8 引入的 `LongAdder`，用「分段锁」思想彻底解决了这些问题——性能是 `AtomicLong` 的 4 倍。

## 一、CAS 的原理

### 1.1 什么是 CAS？

CAS（Compare-And-Swap）是 CPU 提供的原子指令，用于在多线程环境下实现无锁并发。

```java
// CAS 的语义
// V: 内存地址
// A: 期望值
// B: 新值
// 如果 V 的值等于 A，则将 V 的值更新为 B；否则什么都不做
// 返回是否更新成功
boolean CAS(V, A, B) {
    if (V == A) {
        V = B;
        return true;
    }
    return false;
}
```

### 1.2 Java 中的 CAS

在 Java 中，CAS 被封装在 `Unsafe` 类中：

```java
// AtomicInteger 的实现
public class AtomicInteger extends Number {
    private static final jdk.internal.misc.Unsafe U = ...;
    private static final long VALUE = ...;
    
    // valueOffset: value 字段在对象内存中的偏移量
    private volatile int value;
    
    public final int incrementAndGet() {
        return U.getAndAddInt(this, VALUE, 1) + 1;
    }
}
```

`Unsafe.getAndAddInt` 的实现：

```java
// JVM 源码（简化）
public final int getAndAddInt(Object o, long offset, int delta) {
    int v;
    do {
        v = getIntVolatile(o, offset);  // 获取当前值
    } while (!compareAndSwapInt(o, offset, v, v + delta));  // CAS 直到成功
    return v;
}
```

### 1.3 CPU 层面的实现

CAS 在不同 CPU 架构上有不同实现：

```java
// x86: cmpxchg 指令
// LOCK 前缀保证原子性（多核下缓存锁）
lock cmpxchg [addr], eax, ebx

// ARM: ldaex/stlex 指令
// Load-Exclusive / Store-Exclusive
ldaex r0, [r1]    // 加载并标记
stlex r2, r3, [r1] // 尝试存储
```

**关键点**：在多核 CPU 上，CAS 需要 `LOCK#` 总线锁或缓存锁来保证原子性。即使是原子操作，也不是「免费」的。

```bash
# 查看 CPU 支持的原子指令
# x86/x64: cmpxchg, xadd, xchg 等
# ARM: lda/stl 系列

# CAS 在高并发时的开销
# 1. 总线带宽竞争
# 2. 缓存行失效
# 3. 指令重排序限制
```

## 二、CAS 的三大问题

### 2.1 ABA 问题

**什么是 ABA 问题？**

```java
// 场景描述
// 线程 A：读取 value = A
// 线程 B：将 value 从 A 改成 B，再改回 A
// 线程 A：CAS(value, A, C)，成功！

// 问题：线程 A 不知道 value 被改过
```

ABA 问题在某些场景下是致命的：

```java
// 典型案例：栈的并发操作
Stack<Integer> stack = new ConcurrentStack<>();

// 初始状态：A -> B -> C

// 线程 A：准备 pop A
Node top = stack.top;  // top = A
Node next = top.next;  // next = B

// 线程 B：pop A 和 pop B
stack.pop();  // A 被弹出
stack.pop();  // B 被弹出
stack.push(A); // A 又被推回去

// 线程 A：继续 pop（ABA 问题！）
// top 仍然是 A，但栈的状态已经变了
// CAS(top, A, next) 成功，但结果错误
```

**ABA 问题的解决方案：版本号**

```java
// AtomicStampedReference: 每次修改都带上版本号
public class AtomicStampedReference<V> {
    private static class Pair<T> {
        final T reference;
        final int stamp;  // 版本号
        Pair(T reference, int stamp) {
            this.reference = reference;
            this.stamp = stamp;
        }
    }
    
    private volatile Pair<V> pair;
    
    public boolean compareAndSet(V expectedReference,
                                  V newReference,
                                  int expectedStamp,
                                  int newStamp) {
        Pair<V> current = pair;
        return expectedReference == current.reference &&
               expectedStamp == current.stamp &&
               ((newReference == current.reference &&
                 newStamp == current.stamp) ||
                casPair(current, Pair.of(newReference, newStamp)));
    }
}
```

**使用示例**：

```java
AtomicStampedReference<String> ref = new AtomicStampedReference<>("A", 1);

// 线程 A
String oldValue = ref.getReference();
int oldStamp = ref.getStamp();
ref.compareAndSet(oldValue, "C", oldStamp, oldStamp + 1);

// 线程 B：中间修改了
ref.compareAndSet("A", "B", 1, 2);  // 先改成 B
ref.compareAndSet("B", "A", 2, 3);  // 再改回 A（版本号变成 3）

// 线程 A：CAS 失败！因为版本号不匹配
ref.compareAndSet("A", "C", 1, 2);  // expectedStamp=1, actual=3, 失败
```

**另一种方案：AtomicMarkableReference**

如果只需要知道「是否被修改过」，而不需要版本号：

```java
// AtomicMarkableReference: 使用 boolean 标记
AtomicMarkableReference<String> ref = 
    new AtomicMarkableReference<>("A", false);

String oldValue = ref.getReference();
boolean oldMark = ref.isMarked();
ref.compareAndSet(oldValue, "B", oldMark, !oldMark);
```

### 2.2 循环开销大

**自旋的代价**

CAS 失败时，线程会自旋重试：

```java
// getAndAddInt 的自旋逻辑
public final int getAndAddInt(Object o, long offset, int delta) {
    int v;
    do {
        v = getIntVolatile(o, offset);  // 每次重试都要重新读取
    } while (!compareAndSwapInt(o, offset, v, v + delta));
    return v;
}
```

在高并发下，大量线程同时竞争同一个变量：

```java
// 问题场景：100 个线程同时 increment
// 假设 value 初始值为 0
// 只有 1 个线程能在第一次 CAS 成功
// 其他 99 个线程都失败，需要自旋重试
// 重试过程中，又会有竞争

// 线程数越多，自旋次数越多
// CPU 在空转中浪费大量时间
```

**性能对比**：

```java
// JMH 基准测试
@State(Scope.Thread)
@BenchmarkMode(Mode.Throughput)
public class CASBenchmark {
    
    private AtomicLong atomicLong = new AtomicLong(0);
    
    @Benchmark
    public void atomicIncrement() {
        atomicLong.incrementAndGet();
    }
    
    // 结果（8 线程，1000万次操作）：
    // AtomicLong: ~50万 ops/ms，CPU 100%
    // LongAdder:  ~200万 ops/ms，CPU 80%
}
```

### 2.3 只能保证一个共享变量

**原始 CAS 的限制**

一个 CAS 只能原子地操作一个变量：

```java
// 问题：需要同时更新两个变量
class Node {
    volatile Node next;
    volatile int value;
}

// 线程 A：尝试更新
Node oldHead = head;
Node newHead = oldHead.next;
newHead.value = 10;  // 不是原子操作！

// 问题：如果另一个线程在这之间修改了 head，状态就不一致了
```

**解决方案：AtomicReference**

`AtomicReference` 可以保证引用更新的原子性：

```java
// AtomicReference 保证引用操作的原子性
AtomicReference<Node> headRef = new AtomicReference<>();

// 这样可以原子地更新整个 Node
Node newNode = new Node(value, headRef.get().next);
headRef.compareAndSet(oldHead, newNode);
```

**更复杂的场景：对象多个字段的原子更新**

```java
// 方案 1：缩小临界区
synchronized (this) {
    this.field1 = newValue1;
    this.field2 = newValue2;
}

// 方案 2：不可变对象
// 每次更新都创建新对象
AtomicReference<Data> dataRef = new AtomicReference<>();

// 更新时，创建包含所有字段的新对象
Data newData = new Data(newField1, newField2, ...);
dataRef.compareAndSet(oldData, newData);

// 方案 3：使用 AtomicReferenceFieldUpdater
AtomicReferenceFieldUpdater<Node, Integer> valueUpdater = 
    AtomicReferenceFieldUpdater.newUpdater(Node.class, int.class, "value");
```

## 三、LongAdder 的分段锁思想

### 3.1 为什么需要 LongAdder？

`AtomicLong` 在高并发下性能不佳的根本原因：**所有线程竞争同一个变量的 CAS**。

`LongAdder` 的解决思路：**分段计数，减少竞争**。

### 3.2 LongAdder 的结构

```java
public class LongAdder extends Striped64 {
    
    // 基础值（无竞争时使用）
    transient volatile long base;
    
    // 分段数组（高竞争时使用）
    // 每个 Cell 维护一个局部计数
    @sun.misc.Contended
    static final class Cell {
        volatile long value;
        
        Cell(long x) {
            value = x;
        }
        
        final boolean cas(long cmp, long val) {
            return VALUE.compareAndSwapLong(this, VALUE_OFFSET, cmp, val);
        }
    }
    
    transient volatile Cell[] cells;
}
```

### 3.3 Striped64 的实现原理

`Striped64` 是 `LongAdder` 和 `LongAccumulator` 的基类，实现了分段计数的核心逻辑：

```java
abstract class Striped64 extends Number {
    
    // Cell 数组：类似 ConcurrentHashMap 的桶数组
    @sun.misc.Contended
    volatile Cell[] cells;
    
    // 基础值（无竞争时的快速路径）
    volatile long base;
    
    // cells 数组的 busy 标记（用于扩容）
    volatile int cellsBusy;
    
    // 探针值（ThreadLocalRandom 的前身）
    // 用于将线程映射到对应的 Cell
    final long probe(int x) {
        return x;
    }
    
    // 添加值
    final void add(long x) {
        Cell[] as;
        long b, v;
        int m;
        Cell a;
        
        // 路径 1: cells 未初始化，尝试 CAS 修改 base
        if ((as = cells) == null ||
            (m = as.length - 1) < 0 ||
            (a = as[getProbe() & m]) == null ||
            !(uncontended = a.cas(b = a.value, b + x))) {
            // 进入 longAccumulateCell
            longAccumulateCell(x, null, uncontended);
        }
    }
}
```

### 3.4 分段计数的核心逻辑

```java
// Striped64.longAccumulateCell（简化）
final void longAccumulateCell(long x, LongBinaryOperator fn, boolean wasUncontended) {
    int h;
    if ((h = getThreadProbe()) == 0) {
        // 初始化探针值
        ThreadLocalRandom.current();
        h = getThreadProbe();
        wasUncontended = true;
    }
    
    // 自旋锁标记
    boolean collide = false;
    
    for (;;) {
        Cell[] as;
        Cell a;
        int n;
        long v;
        
        if ((as = cells) != null && (n = as.length) > 0) {
            // 路径 A: Cell 已存在
            if ((a = as[(n - 1) & h]) == null) {
                if (cellsBusy == 0) {
                    Cell r = new Cell(x);
                    if (cellsBusy == 0 && casCellsBusy()) {
                        // 创建新 Cell
                        Cell[] rs = new Cell[n + 1];
                        rs[n] = r;
                        cells = rs;
                        cellsBusy = 0;
                        break;
                    }
                }
                collide = false;
            }
            else if (!wasUncontended) {
                wasUncontended = true;
            }
            else if (a.cas(v = a.value, fn == null ? v + x : fn.applyAsLong(v, x))) {
                break;
            }
            else if (n >= NCPU || cells != as) {
                collide = false;
            }
            else if (!collide) {
                collide = true;
            }
            else if (cellsBusy == 0 && casCellsBusy()) {
                // 扩容：Cell 数组翻倍
                Cell[] rs = new Cell[n << 1];
                for (int i = 0; i < n; ++i)
                    rs[i] = as[i];
                cells = rs;
                cellsBusy = 0;
                collide = false;
                continue;
            }
            cellsBusy = 0;
        }
        else if (cellsBusy == 0 && cells == as && casCellsBusy()) {
            // 路径 B: 初始化 Cell 数组
            Cell[] rs = new Cell[2];
            rs[h & 1] = new Cell(x);
            cells = rs;
            cellsBusy = 0;
            break;
        }
    }
}
```

### 3.5 LongAdder vs AtomicLong 性能对比

```java
// JMH 基准测试
@State(Scope.Thread)
@BenchmarkMode(Mode.Throughput)
@OutputTimeUnit(TimeUnit.MILLISECONDS)
public class LongAdderBenchmark {
    
    private AtomicLong atomicLong = new AtomicLong(0);
    private LongAdder longAdder = new LongAdder();
    
    private int threads = 100;
    private int iterations = 1000000;
    
    @Benchmark
    public void atomicLongMultiThread() throws InterruptedException {
        CountDownLatch start = new CountDownLatch(1);
        CountDownLatch done = new CountDownLatch(threads);
        
        for (int t = 0; t < threads; t++) {
            new Thread(() -> {
                try {
                    start.await();
                    for (int i = 0; i < iterations; i++) {
                        atomicLong.incrementAndGet();
                    }
                } finally {
                    done.countDown();
                }
            }).start();
        }
        
        start.countDown();
        done.await();
    }
    
    @Benchmark
    public void longAdderMultiThread() throws InterruptedException {
        CountDownLatch start = new CountDownLatch(1);
        CountDownLatch done = new CountDownLatch(threads);
        
        for (int t = 0; t < threads; t++) {

                try {
                    start.await();
                    for (int i = 0; i < iterations; i++) {
                        longAdder.increment();
                    }
                } finally {
                    done.countDown();
                }
            }).start();

        done.await();
    }
```

### 3.6 LongAdder 的 sum() 方法

```java
public long sum() {
    Cell[] as = cells;
    long sum = base;
    if (as != null) {
        for (Cell a : as) {
            if (a != null) {
                sum += a.value;
            }
        }
    }
    return sum;
}
```

**注意**：`sum()` 返回的是近似值，因为调用 `sum()` 时：
1. 可能有其他线程正在修改 `base` 或 `cells`
2. `sum()` 没有加锁，返回的是「快照」
3. 如果需要精确计数，可能需要在 `sum()` 前后加锁

## 四、@Contended 注解解决伪共享

### 4.1 什么是伪共享？

在讲解 `LongAdder` 的实现时，你注意到了 `@Contended` 注解吗？

```java
@sun.misc.Contended
static final class Cell {
    volatile long value;
    ...
}
```

这个注解是为了解决**伪共享（False Sharing）**问题。

### 4.2 缓存行与伪共享

CPU 缓存以「缓存行（Cache Line）」为单位工作，标准大小是 64 字节。

```bash
# 查看 CPU 缓存行大小
$ getconf LEVEL1_DCACHE_LINESIZE
64
```

如果两个变量位于同一个缓存行：

```java
class PaddingDemo {
    volatile long value1;
    volatile long value2;  // 与 value1 在同一缓存行
}

// 线程 A 修改 value1
// 线程 B 修改 value2
// 问题：虽然修改的是不同变量，但缓存行被两个 CPU 核心共享
// 一个核心的修改会导致另一个核心的缓存行失效
```

### 4.3 @Contended 的作用

```java
// @Contended 注解会为每个 Cell 填充足够的空间
// 使得每个 Cell 独占一个缓存行

// 效果：Cell[0] 和 Cell[1] 不会在同一个缓存行
// 一个 Cell 的修改不会影响另一个 Cell

// JVM 参数：-XX:-RestrictContended 启用 @Contended
// -XX:ContendedPaddingWidth=128 设置填充宽度
```

## 五、空间换时间策略的适用场景

### 5.1 LongAdder 的适用场景

| 场景 | 推荐 | 原因 |
|---|---|---|
| 高并发计数器 | LongAdder | 分段计数，性能高 |
| 批量求和 | LongAdder | 最后一次性求和 |
| 需要精确 P99 | AtomicLong | LongAdder 求和可能有误差 |
| 低并发（1-2线程） | AtomicLong | LongAdder 额外开销不划算 |

### 5.2 LongAdder 不适用的场景

```java
// 场景 1: 需要立即读取最新值
long current = longAdder.sum();  // 不是原子操作，可能有误差

// 场景 2: 需要 CAS 失败后重试
// LongAdder 内部封装了重试逻辑，不适合自定义重试

// 场景 3: 只关心增量，不需要求和
// AtomicLong.incrementAndGet() 返回增加后的值
// LongAdder.increment() 不返回值
```

### 5.3 扩展：其他 JDK 并发工具

```java
// DoubleAdder / DoubleAccumulator
// 用于浮点数的分段计数
DoubleAdder doubleAdder = new DoubleAdder();
doubleAdder.add(0.1);

// LongAccumulator
// 支持自定义累加函数
LongAccumulator maxAccumulator = 
    new LongAccumulator(Long::max, Long.MIN_VALUE);

// LongAdder vs LongAccumulator
longAdder.increment();          // 只能 +1
maxAccumulator.accumulate(x);   // 可以自定义操作
```

## 六、实战：如何选择并发计数器

### 6.1 选型决策树

```
需要并发计数器？
│
├── 是否高并发（>10线程）？
│   ├── 是否需要精确值？
│   │   ├── 是 → AtomicLong（加锁）
│   │   └── 否 → LongAdder
│   └── ...
│
├── 是否需要立即返回值？
│   ├── 是 → AtomicLong
│   └── 否 → LongAdder
│
└── 是否需要自定义操作？
    ├── 是 → LongAccumulator
    └── 否 → LongAdder
```

### 6.2 性能监控示例

```java
public class ConcurrentCounter {
    
    // 推荐：高并发场景
    private final LongAdder requestCount = new LongAdder();
    private final LongAdder errorCount = new LongAdder();
    
    // 兜底：需要精确值时
    private final AtomicLong exactCount = new AtomicLong(0);
    
    public void recordRequest() {
        requestCount.increment();
        exactCount.incrementAndGet();  // 同步更新
    }
    
    public void recordError() {
        errorCount.increment();
    }
    
    // 监控接口
    public Map<String, Object> getMetrics() {
        return Map.of(
            "requestCount", requestCount.sum(),
            "exactCount", exactCount.get(),
            "errorCount", errorCount.sum(),
            "errorRate", errorCount.sum() / (double) requestCount.sum()
        );
    }
}
```

## 七、总结

CAS 是实现无锁并发的基础，但它有三大局限：

1. **ABA 问题**：用 `AtomicStampedReference` 或 `AtomicMarkableReference` 解决
2. **循环开销大**：用分段计数（LongAdder）解决
3. **单变量限制**：用 `AtomicReference` 或缩小临界区解决

`LongAdder` 的分段锁思想（借鉴自 `ConcurrentHashMap`）是解决高并发计数器问题的经典方案：**空间换时间**。

### 场景模拟

**面试官**：`AtomicLong` 在高并发下有什么性能问题？

**你**：所有线程都在竞争同一个变量的 CAS，在高并发下会有大量自旋重试，浪费 CPU...

**你**：它采用了分段锁的思想...用 base + cells[] 数组...每个线程 CAS 到自己的 Cell 上...最后求和时累加 base 和所有 Cell...

**你**：不是，因为 sum() 没有加锁，返回的是快照...如果需要精确值，可以用 AtomicLong，或者在调用 sum() 前后加锁...

**面试官**：什么是伪共享？@Contended 注解是干什么的？

**你**：伪共享是缓存行级别的性能问题...两个变量在同一缓存行，一个修改会导致另一个的缓存失效...@Contended 注解会在变量前后填充空间，使其独占缓存行...
