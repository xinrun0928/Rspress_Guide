# 锁优化：无锁 → 偏向锁 → 轻量级锁 → 重量级锁

「synchronized 性能很差」，这句话你一定听过。

但如果我告诉你，JDK 6 之后，JVM 团队花了整整三年优化 synchronized，现在它和 Lock 的性能差距已经微乎其微——你信吗？

更关键的是：|synchronized| 的锁升级是**单向不可逆**的。一旦升级到重量级锁，即使锁释放了，下次加锁也从重量级锁开始。

这个「不可逆」的机制，埋藏着很多生产故障的根因。

## 一、synchronized 的底层原理

### 1.1 对象头的秘密

在 JVM 中，每个 Java 对象在内存中分为三部分：对象头（Header）、实例数据（Instance Data）和对齐填充（Padding）。

```java
// 对象内存布局（64位 JVM）
┌─────────────────────────────────────────────────────────────────────┐
│                           Object Header (128 bits)                   │
├─────────────────────────────┬───────────────────────────────────────┤
│  Mark Word (64 bits)        │  Klass Pointer (64 bits,开启指针压缩)  │
│  存储对象的hashcode、锁信息  │  指向类的元数据指针                     │
│  GC 年龄分代信息              │                                       │
└─────────────────────────────┴───────────────────────────────────────┘
│                           Instance Data                              │
│  对象的实例字段（按 align 排列）                                        │
└─────────────────────────────────────────────────────────────────────┘
│                           Padding                                    │
│  对齐填充（8字节对齐）                                                  │
└─────────────────────────────────────────────────────────────────────┘
```

### 1.2 Mark Word 的结构

Mark Word 是 synchronized 的核心，它用不同位表示不同的锁状态：

```java
// 64位 JVM 的 Mark Word 结构（无压缩）
┌───────────────────────────────────────────────────────────────────────────┐
│                           Mark Word (64 bits)                             │
├─────────────────────────────────┬─────────────────────────────────────────┤
│  锁状态        │  内容                          │  说明                    │
├─────────────────────────────────┼─────────────────────────────────────────┤
│  无锁          │  hashcode(31) │ age(4) │ biased(1) │ 01  │  对象未加锁       │
│               │  分代年龄       │ 偏向锁标志 │                        │
├─────────────────────────────────┼─────────────────────────────────────────┤
│  偏向锁        │  thread(54)   │ epoch(2) │ age(4) │ 01  │ 偏向线程ID + 时间戳│
│               │  偏向线程ID                    │                        │
├─────────────────────────────────┼─────────────────────────────────────────┤
│  轻量级锁      │  指向栈中锁记录的指针                         │ 00  │ CAS + 自旋      │
├─────────────────────────────────┼─────────────────────────────────────────┤
│  重量级锁      │  指向互斥量的指针                            │ 10  │ Monitor Object  │
├─────────────────────────────────┼─────────────────────────────────────────┤
│  GC 标记       │  空                                  │ 11  │ 对象被回收      │
└─────────────────────────────────┴─────────────────────────────────────────┘
```

**关键信息**：Mark Word 只有 64 位，要存储 hashcode（31位）、锁信息、线程ID、分代年龄等信息。在不同锁状态下，这些信息的含义会动态变化。这就是为什么锁升级后 hashcode 就丢了——因为那些位被用来存储其他信息了。

## 二、锁升级的四个阶段

|synchronized| 的锁升级是 JDK 6 引入的优化，全称叫「轻量级锁锁膨胀（Lock Coarsening）」机制。它遵循一个原则：**在无竞争的情况下，使用最轻量的加锁方式；在竞争激烈时，逐步升级到重量级锁**。

### 2.2 无锁状态

对象刚创建时，还没有任何线程竞争，Mark Word 处于无锁状态：

```
无锁状态 Mark Word:
┌────────────────────────────────────────────────────────────────┐
│   hashcode(25)      │  age(4)  │  biased(0)  │  01            │
│   对象的 hashcode    │ 分代年龄   │ 未偏向       │ 无锁状态       │
└────────────────────────────────────────────────────────────────┘
```

这里有一个重要细节：**hashcode 只会在对象调用 `hashCode()` 方法后才会被计算并写入 Mark Word**。如果对象已经偏向某个线程，就再也存储不下 hashcode 了——这也是偏向锁撤销后无法直接恢复到无锁状态的原因之一。

## 三、偏向锁的原理

### 3.1 偏向锁的工作机制

偏向锁的核心思想是：**「这个锁偏向于第一个获取它的线程」**。如果后续只有这一个线程访问同步代码块，连 CAS 操作都不需要，直接跳过。

偏向锁的获取流程：

```java
// JVM 源码简化逻辑（偏向锁获取）
void biased_locking_enter(Handle obj) {
    // 1. 检查 Mark Word 是否为可偏向状态（biased=1, lock=01）
    if (obj.mark().is_biased()) {
        // 2. 检查偏向线程是否是当前线程
        if (obj.mark().biased_thread() == current_thread()) {
            // 3. 同一线程，无需任何同步，直接进入
            return;
        }
        
        // 4. 其他线程尝试获取偏向锁，尝试撤销偏向锁
        if (偏向锁未启用 || 需要撤销) {
            // 在安全点进行偏向锁撤销
            revoke_bias(obj);
        }
    }
}
```

### 3.2 偏向锁的 Mark Word

```
偏向锁状态 Mark Word:
┌────────────────────────────────────────────────────────────────┐
│   thread(54)            │  epoch(2)  │  age(4)  │  01            │
│   偏向线程ID             │  偏向时间戳  │ 分代年龄   │ 偏向锁状态     │
└────────────────────────────────────────────────────────────────┘
```

54位存储偏向线程ID，足以区分所有线程。

### 3.3 偏向锁的撤销条件

偏向锁虽然高效，但并非坚不可摧。以下情况会导致偏向锁撤销：

| 撤销条件 | 说明 | 处理方式 |
|---|---|---|
| **hashcode 调用** | 当对象调用 `hashCode()` 或 `System.identityHashCode()` | 偏向锁被撤销，膨胀为轻量级锁 |
| **其他线程竞争** | 其他线程尝试获取已经被偏向的锁 | 在安全点（Safe Point）撤销偏向锁 |
| **wait/notify** | 调用对象的 `wait()` 或 `notify()` | 必须使用重量级锁，偏向锁撤销 |
| **批量重偏向** | JVM 会记录偏向线程，如果某类对象大量偏向同一线程后被另一线程获取 | 批量撤销或重偏向 |

### 3.4 安全点与偏向锁撤销

偏向锁撤销必须在「安全点」进行。安全点是 JVM 在执行 GC、JIT 编译、偏向锁撤销时的「暂停点」。

为什么需要安全点？因为撤销偏向锁需要检查目标对象的 Mark Word，如果对象正在被其他线程访问，直接修改会导致数据不一致。JVM 需要等待所有线程都到达安全点后，才能安全地进行偏向锁撤销。

```java
// 偏向锁撤销过程（简化）
void revoke_bias(Handle obj) {
    // 1. 等待所有线程到达安全点（STW）
    // 2. 遍历所有线程的栈，找到锁记录
    // 3. 如果偏向线程仍在使用该锁，恢复到轻量级锁
    // 4. 如果偏向线程已不再使用，撤销偏向状态
    // 5. 恢复所有线程
}
```

**面试追问**：为什么偏向锁撤销会导致性能抖动？

因为偏向锁撤销需要在安全点进行，而安全点需要所有线程都停下来。如果频繁触发偏向锁撤销，会导致频繁的 STW（Stop The World），影响系统响应时间。这也是 JDK 15 后偏向锁被废弃的原因之一。

## 四、轻量级锁的原理

### 4.1 轻量级锁的工作机制

当偏向锁被撤销，或者对象创建时直接禁用偏向锁，对象会进入轻量级锁状态。

轻量级锁的核心是 **CAS + 自旋**：不阻塞线程，而是让线程在原地「空转」等待锁释放。

轻量级锁的获取过程：

```java
// 线程栈帧中的锁记录（Lock Record）
// 每个线程都有自己的栈，栈中会有锁记录
class LockRecord {
    private Object reference;   // 指向锁对象的指针
    private long   lockRecord;  // 期望值（Displaced Mark Word）
}

// 轻量级锁获取
void light_lock(Handle obj) {
    // 1. 在当前线程栈中创建锁记录
    LockRecord lr = allocate_lock_record();
    lr.set_displaced_header(obj.mark());  // 保存原始 Mark Word
    
    // 2. 使用 CAS 尝试将 Mark Word 替换为指向锁记录的指针
    //    如果成功，说明获取锁成功
    //    如果失败，说明有竞争，自旋重试
    if (obj.mark().compare_and_set(
            obj.mark(),              // 期望值：当前 Mark Word
            lr.address())) {        // 新值：锁记录地址
        return;  // 获取锁成功
    }
    
    // 3. CAS 失败，说明有竞争，自旋重试
    int spin_count = 0;
    while (spin_count < MAX_SPIN) {
        if (obj.mark().compare_and_set(
                obj.mark(), lr.address())) {
            return;  // 获取锁成功
        }
        spin_count++;
    }
    
    // 4. 自旋超过阈值，膨胀为重量级锁
    inflate_to_heavy_lock(obj);
}
```

### 4.2 轻量级锁的 Mark Word

```
轻量级锁状态 Mark Word:
┌────────────────────────────────────────────────────────────────┐
│                    ptr to Lock Record (62 bits)      │  00    │
│                    指向线程栈中锁记录的指针                        │
└────────────────────────────────────────────────────────────────┘
```

低两位是 `00`，表示轻量级锁。62位用来存储指向锁记录的指针。

### 4.3 轻量级锁的释放

```java
// 轻量级锁释放
void light_unlock(Handle obj) {
    // 1. 使用 CAS 将锁记录中的原始 Mark Word 恢复到对象头
    if (obj.mark().compare_and_set(
            lr.address(),            // 当前值：锁记录指针
            lr.displaced_header())) {// 期望值：原始 Mark Word
        return;  // 释放成功
    }
    
    // 2. CAS 失败，说明有竞争，膨胀为重量级锁
    inflate_to_heavy_lock(obj);
}
```

### 4.4 轻量级锁的优缺点

| 优点 | 缺点 |
|---|---|
| 不阻塞线程，避免线程切换 | 自旋消耗 CPU |
| 适用于锁持有时间很短的场景 | 适用于并发度不高的场景 |
| 线程不会进入 BLOCKED 状态 | 自旋失败后会膨胀为重量级锁 |

## 五、重量级锁的原理

### 5.1 Monitor Object

重量级锁依赖操作系统底层的互斥锁（Mutex），通过 Monitor Object 实现。

```java
// Monitor Object 的结构（JVM 源码简化）
class Monitor {
    Object  owner;          // 持有锁的线程
    int     count;          // 重入计数
    WaitSet  waiters;       // 调用 wait() 等待的线程队列
    EntryList entryList;    // 等待获取锁的线程队列
}
```

### 5.2 重量级锁的等待队列

```
┌─────────────────────────────────────────────────────────────────┐
│                     Monitor Object                              │
├─────────────────────────────────────────────────────────────────┤
│  owner ──────────────┐                                           │
│  count: 1            │                                           │
│                      │                                           │
│  ┌───────────────────┴───────────────────┐                      │
│  │                                       │                       │
│  ▼                                       ▼                       │
│  ┌─────────────────┐              ┌─────────────────┐            │
│  │   EntryList     │              │    WaitSet      │            │
│  │  (等待锁的队列)   │              │  (调用wait后等待) │            │
│  │                  │              │                  │            │
│  │  Thread-A ◄──    │   wait()     │  Thread-C        │            │
│  │  Thread-B        │ ──────────►  │  Thread-D        │            │
│  │  Thread-D        │              │                  │            │
│  │                  │              │                  │            │
│  └─────────────────┘              └─────────────────┘            │
└─────────────────────────────────────────────────────────────────┘
```

**EntryList** 和 **WaitSet** 的区别：
- **EntryList**：线程尝试获取锁但失败，进入这个队列等待
- **WaitSet**：线程获取锁后调用 `wait()` 方法，主动释放锁，进入这个队列等待被唤醒

### 5.3 重量级锁的工作流程

```java
// 重量级锁的获取（简化）
void heavy_lock(Handle obj) {
    // 1. 对象头的 Mark Word 指向 Monitor
    Monitor* m = get_or_create_monitor(obj);
    
    // 2. 尝试获取锁
    if (m->owner == NULL) {  // 锁未被持有
        m->owner = current_thread();  // 设置 owner
        m->count = 1;
        return;  // 获取成功
    }
    
    // 3. 如果 owner 是当前线程（重入）
    if (m->owner == current_thread()) {
        m->count++;  // 重入计数 +1
        return;
    }
    
    // 4. 锁被其他线程持有，线程进入 EntryList 等待
    m->add_to_entry_list(current_thread());
    
    // 5. 线程进入 WAITING/BLOCKED 状态（操作系统层面挂起）
    park();
}

// 重量级锁的释放
void heavy_unlock(Handle obj) {
    Monitor* m = obj.monitor();
    
    if (--m->count == 0) {  // 重入计数为0，真正释放锁
        m->owner = NULL;
        
        // 6. 唤醒 EntryList 中的一个线程
        Thread* waiting = m->pick_from_entry_list();
        if (waiting != NULL) {
            unpark(waiting);  // 唤醒等待线程
        }
    }
}
```

### 5.4 wait/notify 的原理

```java
public class WaitNotifyDemo {
    private Object lock = new Object();
    
    public void waitDemo() throws InterruptedException {
        synchronized (lock) {
            while (condition not met) {
                lock.wait();  // 释放锁，进入 WaitSet
            }
            // 处理业务
        }
    }
    
    public void notifyDemo() {
        synchronized (lock) {
            // 修改条件
            condition = true;
            lock.notify();  // 唤醒 WaitSet 中的一个线程
        }
    }
}
```

`wait()` 和 `notify()` 的底层操作：
1. `wait()`：释放锁，将线程从 Monitor 的 EntryList 移动到 WaitSet，然后阻塞
2. `notify()`：从 WaitSet 中选择一个线程唤醒（移到 EntryList）
3. `notifyAll()`：唤醒 WaitSet 中所有线程

**面试追问**：`notify()` 唤醒的是哪一个线程？被唤醒的线程能马上执行吗？

`notify()` 唤醒的是 WaitSet 中「等待时间最长」的线程（不同 JVM 实现可能不同）。被唤醒的线程需要等待 `notify()` 所在线程释放锁后，才能重新竞争锁，不是马上执行。

## 六、锁升级不可逆的原因

这是一个非常重要的知识点，也是很多生产故障的根因。

### 6.1 状态转换的不可逆性

```
无锁 ──► 偏向锁 ──► 轻量级锁 ──► 重量级锁
                    │                   
                    └──► 重量级锁（不可回退）
```

一旦锁升级到重量级锁，Mark Word 的结构就完全改变了（变成了指向 Monitor 的指针）。即使锁释放了，Mark Word 也无法恢复到无锁或偏向锁状态。

### 6.2 为什么不可逆？

1. **Mark Word 空间有限**：Mark Word 只有 64 位，不同状态需要不同的位布局。一旦某个状态占用了某些位，其他状态就再也用不了了。

2. **hashcode 被覆盖**：如果对象曾经调用过 `hashCode()`，hashcode 已经存储在 Mark Word 中。一旦升级到重量级锁，hashcode 的位置被 Monitor 指针占用，无法恢复。

3. **性能考虑**：锁降级需要额外的同步操作，反而增加开销。JVM 认为锁升级更常见，所以只实现了单向升级。

### 6.3 实际影响

在某些场景下，这个问题会导致性能问题：

```java
// 问题场景：锁竞争激烈
synchronized (sharedObject) {
    // 高并发场景，很快膨胀到重量级锁
}

// 之后即使只有单线程访问，也要走重量级锁流程
// 因为无法回退到偏向锁或轻量级锁

// 解决方案：禁用偏向锁，减少锁升级的次数
// JVM 参数：-XX:-UseBiasedLocking
```

## 七、锁消除与锁粗化

### 7.1 锁消除

JIT 编译器在运行时分析，如果发现锁不可能被共享，就会消除这个锁。

```java
// JIT 优化前
public String concatString(String s1, String s2, String s3) {
    StringBuffer sb = new StringBuffer();
    sb.append(s1);
    sb.append(s2);
    sb.append(s3);
    return sb.toString();
}

// JIT 优化后（锁消除）
// StringBuffer 是方法局部变量，不可能被其他线程访问
// 编译器消除所有 synchronized
public String concatString(String s1, String s2, String s3) {
    StringBuffer sb = new StringBuffer();  // 栈上分配
    sb.append(s1);
    sb.append(s2);
    sb.append(s3);
    return sb.toString();
}
```

**触发条件**：JIT 通过「逃逸分析」判断对象是否会发生逃逸。如果对象只在方法内部使用（不逃逸），那么它的锁就可以被安全消除。

### 7.2 锁粗化

如果一系列操作需要对同一个对象加锁，JIT 会将多次加锁合并为一次加锁，减少加锁次数。

```java
// JIT 优化前
StringBuffer sb = new StringBuffer();
synchronized (sb) {
    sb.append(s1);
}
synchronized (sb) {
    sb.append(s2);
}
synchronized (sb) {
    sb.append(s3);
}

// JIT 优化后（锁粗化）
synchronized (sb) {
    sb.append(s1);
    sb.append(s2);
    sb.append(s3);
}
```

## 八、JDK 6 对 synchronized 的大规模优化

JDK 6 是 synchronized 性能的分水岭。JVM 团队花了三年时间，从多个维度进行了深度优化。

### 8.1 自旋锁的优化：自适应自旋

传统自旋锁的缺点是自旋次数固定，无法适应不同场景。

JDK 6 引入**自适应自旋**：根据上一次自旋的成功率动态调整自旋次数。

```java
// JDK 6 之前的自旋（固定次数）
while (spin_count < 10) {
    if (尝试获取锁) return;
    spin_count++;
}

// JDK 6 之后的自适应自旋
// 如果上次自旋成功，延长自旋时间
// 如果上次自旋失败，缩短自旋时间
while (true) {
    if (尝试获取锁成功) return;
    if (自旋时间过长) return;  // 时间动态调整
}
```

### 8.2 偏向锁延迟

JVM 启动后，并不会立即启用偏向锁，而是有一个「偏向锁延迟」期。这是因为启动阶段会有大量 synchronized 代码块，但很快就会被替换。

```
JVM 启动参数：-XX:BiasedLockingStartupDelay=0  // 立即启用偏向锁
JVM 启动参数：-XX:BiasedLockingStartupDelay=5000  // 延迟 5 秒启用偏向锁
```

### 8.3 轻量级锁的优化

JDK 6 优化了轻量级锁的获取和释放路径，减少了 CAS 操作的次数。

### 8.4 锁消除的强化

JDK 6 增强了逃逸分析能力，使得更多锁可以被安全消除。

## 九、实战：如何选择合适的锁

| 场景 | 推荐配置 |
|---|---|
| 单线程 + 无竞争 | 偏向锁（默认启用） |
| 短时竞争 | 轻量级锁（默认启用） |
| 长时竞争或高并发 | 重量级锁（避免自旋浪费 CPU） |
| 高并发 + 低延迟 | 禁用偏向锁，使用轻量级锁起点 |

**JVM 参数建议**：

```bash
# 服务器配置（高并发场景）
-XX:+UseConcMarkSweepGC  # CMS GC
-XX:+UseBiasedLocking    # 启用偏向锁（默认）
-XX:PreBlockSpin=10       # 自旋次数（可调整）

# 延迟敏感型应用（低延迟要求）
-XX:-UseBiasedLocking    # 禁用偏向锁
-XX:PreBlockSpin=6        # 减少自旋次数

# 极端场景
-XX:+AlwaysPreTouch       # 预分配堆内存
```

## 总结

|synchronized| 从 JDK 1.6 开始，经历了脱胎换骨的优化。偏向锁、轻量级锁、自旋锁……这些「花活儿」让 synchronized 从「性能差」的代名词，变成了「高性能」的代名词。

但理解这些优化的原理不是为了面试，而是为了在实际工作中做出正确的配置决策。

**面试官**：synchronized 是如何保证线程安全的？

**你**：synchronized 通过monitorenter和monitorexit指令，对象头的Mark Word会记录锁状态...

**面试官**：那它的锁升级过程是怎样的？

**你**：如果系统的并发特点是「锁竞争激烈但锁持有时间短」，偏向锁会频繁撤销，反而影响性能。这种场景建议禁用偏向锁...
