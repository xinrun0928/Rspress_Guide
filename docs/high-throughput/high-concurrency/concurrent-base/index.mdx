# 并发基础优化：线程安全的底层逻辑

> [!WARNING]
> 前置知识：本篇文章是并发基础优化的入口，建议按顺序阅读。线程池调优、锁优化、CAS 原理是所有并发场景的底层支撑，是面试高频考点。

你有没有遇到过这种情况：代码明明加了 `synchronized`，线上却还是出现了数据错乱？或者线程池配置得很大，但 CPU 利用率却一直上不去？

这些问题的根源，往往不在框架和工具，而在**并发基础**——对线程、锁、内存模型的理解是否到位。

并发基础，是高并发模块的「内功心法」。很多人觉得它只是面试八股文，背一背就能过。但真正的高手，能在生产环境中快速定位并发问题、做出正确的技术选型，靠的正是对这些底层逻辑的深刻理解。

这一讲，我们来打通并发基础的任督二脉。

## 二、线程池调优：并发任务的调度中心

线程池是 Java 并发编程中使用频率最高的工具，但大多数人对它只停留在「会用」的程度。

当你提交一个任务到线程池，线程池会经历这样的决策过程：先看核心线程数满了没，没满就创建新线程；满了就看看队列满了没，没满就入队；队列满了再看最大线程数满了没，没满就创建临时线程；直到所有资源都耗尽，执行拒绝策略。

```java
// 线程池的七参数
public ThreadPoolExecutor(
    int corePoolSize,      // 核心线程数
    int maximumPoolSize,   // 最大线程数
    long keepAliveTime,    // 空闲存活时间
    TimeUnit unit,         // 时间单位
    BlockingQueue<Runnable> workQueue,   // 任务队列
    ThreadFactory threadFactory,         // 线程工厂
    RejectedExecutionHandler handler     // 拒绝策略
)
```

CPU 密集型任务和 IO 密集型任务的配置完全不同：

| 任务类型 | 核心线程数公式 | 原因 |
|---|---|---|
| CPU 密集型 | `CPU 核心数 + 1` | 线程大部分时间在执行 CPU 指令，不需要过多线程 |
| IO 密集型 | `CPU 核心数 × 2` 或更多 | 线程大部分时间在等待 IO，可以容纳更多线程 |

> [!TIP]
> 如何判断任务类型？如果任务中有大量网络请求、数据库查询、文件读写，就是 IO 密集型；如果任务主要是计算、加密、压缩，就是 CPU 密集型。

### 线程池的七种拒绝策略

当线程池资源耗尽时，会触发拒绝策略：

```java
// AbortPolicy（默认）：抛异常
// CallerRunsPolicy：由调用线程执行
// DiscardPolicy：直接丢弃
// DiscardOldestPolicy：丢弃队列中最老的任务

// 实际项目中，往往需要自定义策略
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    8, 16, 60L, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(1000),
    new ThreadFactoryBuilder().setNameFormat("pool-%d").build(),
    (r, pool) -> {
        // 自定义拒绝策略：写入日志 + 告警
        log.error("线程池已满，上报监控");
        Metrics.counter("threadpool_rejected").increment();
    }
);
```

## 三、锁优化：synchronized 的进化史

很多人一提到锁就想到 synchronized 性能差。但从 JDK 6 开始，JVM 团队花了整整三年优化 synchronized，引入了偏向锁、轻量级锁、重量级锁的概念，实现了**锁升级**机制。

### 锁升级的四个阶段

```text
┌──────────────┐    竞争出现    ┌──────────────┐   锁撤销过多   ┌──────────────┐  资源耗尽   ┌──────────────┐
│   无锁状态    │ ────────────▶│   偏向锁     │ ───────────▶│  轻量级锁    │ ──────────▶│  重量级锁    │
│              │              │              │              │              │              │              │
│ 对象头无偏向  │              │ 偏向线程ID   │              │ CAS + 自旋   │              │ OS mutex    │
│  标记        │              │  无需同步    │              │  无锁优化    │              │  系统调用   │
└──────────────┘              └──────────────┘              └──────────────┘              └──────────────┘
                                                                                                │
                     ◀────────────────────────────────────────────────────────────────────────┘
                                        锁升级不可逆（重量级锁释放后不会降级）
```

|synchronized| 的锁升级是一个**不可逆**的过程。从偏向锁一路升级到重量级锁后，即使锁释放了，下次加锁时也不会回到初始状态，而是直接从重量级锁开始。

### 锁优化的核心原则

锁优化的核心不是选择哪种锁实现，而是**控制锁的粒度**：

| 优化方式 | 说明 | 示例 |
|---|---|---|
| 减少锁粒度 | 缩小同步范围 | ConcurrentHashMap 分段锁 |
| 读写分离 | 读不加锁，写才加锁 | ReadWriteLock |
| 写时复制 | 读不加锁，写时复制新副本 | CopyOnWriteArrayList |
| 锁消除 | JVM 逃逸分析后自动消除 | 栈上分配的局部变量锁 |

## 四、CAS：乐观锁的基石

CAS（Compare-And-Swap）是乐观锁的底层实现。Java 中的 `AtomicInteger`、`AtomicLong`、`AtomicReference` 都是基于 CAS 实现的。

```java
// AtomicInteger 的底层是 CAS
public final int incrementAndGet() {
    return unsafe.getAndAddInt(this, valueOffset, 1) + 1;
}

// unsafe.getAndAddInt 底层是 CAS 循环
public final int getAndAddInt(Object o, long offset, int delta) {
    int expected;
    do {
        expected = this.getIntVolatile(o, offset);
    } while (!this.compareAndSwapInt(o, offset, expected, expected + delta));
    return expected;
}
```

### CAS 的三大问题

CAS 看似完美，但有三个必须掌握的问题：

- **ABA 问题**：线程1读取到 A，线程2将 A 改为 B 再改回 A，线程1的 CAS 操作会成功，但实际数据已经被修改过。解决方案是使用 `AtomicStampedReference`（版本号）或 `AtomicMarkableReference`（标记位）。
- **自旋开销**：如果 CAS 操作失败，线程会一直自旋等待，高竞争场景下会消耗大量 CPU。解决方案是设置自旋次数上限，或使用 `LongAdder` 分段锁。
- **原子性问题**：CAS 只能保证单个变量的原子性，多个变量需要使用 `AtomicIntegerFieldUpdater` 等工具类或直接加锁。

### LongAdder vs AtomicLong

高并发计数器场景下，LongAdder 比 AtomicLong 性能高出一个数量级：

```java
// AtomicLong：所有线程竞争同一个变量
AtomicLong count = new AtomicLong(0);
count.incrementAndGet(); // 高竞争下，大量线程 CAS 同一个变量

// LongAdder：分段计数，减少竞争
LongAdder count = new LongAdder();
count.increment(); // 每个线程操作自己的段，最后求和
```

## 五、ThreadLocal：线程封闭的陷阱

ThreadLocal 为每个线程提供独立的变量副本，实现了线程级别的数据隔离。但它有一个致命陷阱——**内存泄漏**。

### 内存泄漏的根源

ThreadLocal 的实现是每个线程持有一个 `ThreadLocalMap`，Entry 的 key 是 **WeakReference** 指向 ThreadLocal 对象：

```text
┌─────────────────────────────────────────────────────────────┐
│                      ThreadLocalMap                          │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  Entry[] table                                        │   │
│  │  ┌─────────────────┐  ┌─────────────────┐            │   │
│  │  │ Entry(key,val)  │  │ Entry(key,val)  │  ...      │   │
│  │  │  key = WeakRef  │  │  key = WeakRef  │            │   │
│  │  │  (指向ThreadLocal)│ │  (指向ThreadLocal)│          │   │
│  │  │  val = Object   │  │  val = Object   │            │   │
│  │  └────────┬────────┘  └─────────────────┘            │   │
│  │           │                                               │   │
│  └───────────┼───────────────────────────────────────────┘   │
│              │                                                    │
│              ▼                                                    │
│         Thread (持有)                                             │
│                                                             │
└─────────────────────────────────────────────────────────────┘

当 ThreadLocal 变量不再被引用时：
  WeakReference.get() → null
  但 Entry.value 仍然被 Entry 持有，不被 GC 回收 → 内存泄漏！

正确做法：在使用完毕后调用 remove()，而非依赖 get() 返回 null
```

### 正确使用 ThreadLocal

```java
// 错误写法：依赖 get() 返回 null 判断
try {
    threadLocal.set(user);
    doSomething();
} finally {
    // 错误：如果 set 之前 threadLocal 已有值，get() 不会返回 null
    if (threadLocal.get() != null) {
        threadLocal.remove();
    }
}

// 正确写法：直接 remove
try {
    threadLocal.set(user);
    doSomething();
} finally {
    threadLocal.remove(); // 无论如何都移除
}

// 最佳实践：使用 try-with-resources 封装
public class ThreadLocalHolder<T> implements AutoCloseable {
    private final ThreadLocal<T> threadLocal = new ThreadLocal<>();

    public void set(T value) {
        threadLocal.set(value);
    }

    public T get() {
        return threadLocal.get();
    }

    @Override
    public void close() {
        threadLocal.remove();
    }
}
```

## 六、虚拟线程：JDK 21+ 的并发新范式

JDK 21 正式发布了虚拟线程（Virtual Thread），这是自 Java 1.0 以来最大的并发变革。

传统线程是操作系统线程（Platform Thread），创建和切换成本极高：

```java
// 传统线程：一万个线程 = 一万个 OS 线程
for (int i = 0; i < 10000; i++) {
    new Thread(() -> doSomething()).start(); // 每个线程占用 1MB 栈空间
}
```

虚拟线程由 JVM 管理，不绑定 OS 线程：

```java
// 虚拟线程：一百万个虚拟线程只需几百个 OS 线程
for (int i = 0; i < 1_000_000; i++) {
    Thread.startVirtualThread(() -> doSomething()); // 每个虚拟线程仅占用少量内存
}
```

### 虚拟线程 vs 传统线程

| 对比项 | 传统线程 | 虚拟线程 |
|---|---|---|
| 创建成本 | 高（需要 OS 线程） | 低（仅 JVM 对象） |
| 内存占用 | ~1MB/线程 | ~200-300 字节/线程 |
| 切换成本 | 高（内核态切换） | 低（用户态切换） |
| 适用场景 | CPU 密集型 | IO 密集型（网络、数据库） |

> [!TIP]
> 虚拟线程不适合 CPU 密集型任务，因为虚拟线程仍然运行在 OS 线程上。对于 CPU 密集型任务，使用传统线程池仍然是最优选择。

## 延伸思考

并发基础的学习，本质上是在理解**资源争用**这个核心问题。

线程池是对 CPU 资源的争用，锁是对临界区的争用，CAS 是对变量的争用，ThreadLocal 是用空间换安全的策略。每一种机制，都是在争用与效率之间做权衡。

下次遇到并发问题，先问自己：**问题出在哪个层面的争用？应该用哪种机制来平衡？**

答案往往不在框架里，而在这些基础原理中。
