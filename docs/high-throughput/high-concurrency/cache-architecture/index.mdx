# 缓存架构：高并发系统的性能支柱

> [!WARNING]
> 前置知识：本篇文章是缓存架构的入口，涵盖多级缓存、缓存策略、Redis 进阶三大方向。缓存是面试超高频模块，建议按顺序阅读，逐步构建完整的缓存知识体系。

---

想象一个没有缓存的电商系统：每秒 10 万次商品查询，每次都要穿透到数据库。数据库在第一秒就会崩溃——不是因为数据库不够快，而是因为它根本承受不了这么高的并发。

**缓存，是高并发系统的第一道防线。** 在数据库前面加一层缓存，可以让 90% 以上的请求根本碰不到数据库。

但缓存不只是「加一层 Redis」这么简单。多级缓存如何设计？缓存和数据库的一致性如何保证？热点数据如何识别？缓存雪崩、穿透、击穿如何处理？这些问题，才是面试真正想考察的。

这一讲，我们来构建完整的缓存知识框架。

---

## 二、多级缓存架构：层层拦截请求

真正的生产级缓存系统，不会只依赖 Redis。一个完整的多级缓存架构是这样的：

```text
┌──────────────────────────────────────────────────────────────────────┐
│                        请求到来                                          │
│                           │                                             │
│                           ▼                                             │
│  ┌────────────────────────────────────────────────────────────────┐   │
│  │  L1 本地缓存（Caffeine / Guava Cache）                          │   │
│  │  访问延迟：纳秒级（~10-100ns）                                 │   │
│  │  容量：受限于 JVM 堆内存                                        │   │
│  │  命中率目标：60-70%                                             │   │
│  └───────────────────────────┬────────────────────────────────────┘   │
│                              │ 未命中                                    │
│                              ▼                                          │
│  ┌────────────────────────────────────────────────────────────────┐   │
│  │  L2 分布式缓存（Redis Cluster）                                 │   │
│  │  访问延迟：微秒级（~0.5-2ms）                                   │   │
│  │  容量：TB 级别                                                  │   │
│  │  命中率目标：90%+                                               │   │
│  └───────────────────────────┬────────────────────────────────────┘   │
│                              │ 未命中                                    │
│                              ▼                                          │
│  ┌────────────────────────────────────────────────────────────────┐   │
│  │  数据库（MySQL）                                                │   │
│  │  访问延迟：毫秒级（~1-10ms）                                    │   │
│  └───────────────────────────┬────────────────────────────────────┘   │
│                              │ 命中后回填                               │
│                              ▼                                          │
│                     L2 缓存 → L1 缓存                                   │
└──────────────────────────────────────────────────────────────────────┘
```

### 本地缓存 vs 分布式缓存

| 对比项 | 本地缓存 | 分布式缓存 |
|---|---|---|
| 访问延迟 | 纳秒级 | 微秒级（网络往返） |
| 容量 | 受 JVM 堆限制（MB 级） | 集群容量（TB 级） |
| 数据一致性 | 节点间不一致 | 节点间一致 |
| 适用场景 | 热点数据、只读数据 | 共享数据、频繁变更数据 |
| 典型实现 | Caffeine、Guava Cache | Redis、Memcache |

> [!TIP]
> 本地缓存不是银弹。由于各节点数据独立，本地缓存只适合**数据一致性要求不高、变更不频繁**的场景。比如商品详情页的分类信息、品牌信息，可以用本地缓存。

## 三、缓存策略与一致性：保证数据不乱

### 四种经典读写模式

| 模式 | 读操作 | 写操作 | 一致性 | 复杂度 |
|---|---|---|---|---|
| Cache Aside | 先读缓存，未命中查 DB 并回填 | 先更新 DB，再删除缓存 | 弱一致 | 低 |
| Read Through | 缓存代理读，未命中由缓存查 DB | 同 Cache Aside | 弱一致 | 中 |
| Write Through | 同步写缓存和 DB | 先写缓存，再写 DB | 强一致 | 高 |
| Write Behind | 只写缓存 | 只写缓存，异步批量写 DB | 最终一致 | 高 |

大多数业务场景使用 Cache Aside 模式，因为它实现简单，在「读多写少」场景下效果最好。

### 缓存三大经典问题

缓存世界里有三个永恒的话题：**缓存穿透**、**缓存击穿**、**缓存雪崩**。

**缓存穿透**：大量请求查询一个不存在的数据，缓存中没有，数据库中也没有，请求直接打到数据库。解决方案是布隆过滤器或缓存空值。

**缓存击穿**：一个热点 key 过期瞬间，大量请求同时涌入，全部打到数据库。解决方案是互斥锁、逻辑过期双检、永久不过期加异步更新。

**缓存雪崩**：大量缓存 key 同时过期，导致大量请求同时击穿到数据库。解决方案是过期时间随机化、多级缓存、熔断降级。

### 缓存与数据库一致性

这是面试超高频问题。常见方案：

| 方案 | 原理 | 一致性强度 | 复杂度 |
|---|---|---|---|
| Cache Aside + 延时双删 | 先删缓存，再更新 DB，延时后再删一次缓存 | 最终一致 | 中 |
| 订阅 binlog | MySQL binlog 异步同步到缓存 | 最终一致 | 高 |
| 分布式锁 | 更新时加锁，串行化读写 | 强一致 | 高 |

## 四、缓存进阶：Redis 的深层能力

### Redis 淘汰策略

当 Redis 内存达到上限（`maxmemory`）时，需要淘汰数据。Redis 提供了 8 种淘汰策略：

| 策略 | 淘汰范围 | 说明 |
|---|---|---|
| noeviction | 不淘汰 | 内存满时返回错误 |
| allkeys-lru | 所有 key | LRU 淘汰 |
| allkeys-random | 所有 key | 随机淘汰 |
| volatile-lru | 带过期时间的 key | LRU 淘汰 |
| volatile-random | 带过期时间的 key | 随机淘汰 |
| volatile-ttl | 带过期时间的 key | TTL 最短的优先淘汰 |

LRU（Least Recently Used）是最常用的策略。但 Redis 实现的是**近似 LRU**，不是精确 LRU。近似 LRU 通过抽样而非全量排序来选择淘汰对象，在性能和精度之间做了权衡。

### Redis 过期策略

Redis 的过期策略是**惰性删除 + 定期删除**的组合：

- **惰性删除**：访问 key 时检查是否过期，过期则删除。优点是不额外消耗资源，缺点是过期 key 可能长时间不被删除。
- **定期删除**：每隔一段时间扫描一定数量的 key，删除过期的。优点是及时清理过期 key，缺点是需要消耗 CPU。

> [!TIP]
> 如果大量 key 同时过期，Redis 会持续触发定期删除扫描，可能导致 CPU 抖动。建议在生产环境中，将 key 的过期时间加上随机偏移量（如 `TTL + rand(0, 3600)`）。

### Redis Lua 脚本

Redis Lua 脚本的优势是**原子性 + 减少网络往返**：

```lua
-- 批量扣减库存，原子操作
local stock = redis.call('GET', KEYS[1])
if stock == nil then
    return -1
end
if tonumber(stock) < tonumber(ARGV[1]) then
    return -2
end
redis.call('DECRBY', KEYS[1], ARGV[1])
return tonumber(stock) - tonumber(ARGV[1])
```

## 延伸思考

缓存的设计，本质上是在**一致性**和**性能**之间做权衡。

越强的一致性，需要越多的同步操作；越多的同步操作，意味着越低的性能。多级缓存、Cache Aside、延时双删……每一种设计，都是在找那个「业务能接受的最优点」。

下次设计缓存方案之前，先问自己三个问题：业务能容忍多大的数据不一致？缓存失效后能承受多大的击穿压力？热点数据的变更频率是多少？

答案决定了你的缓存架构设计。
