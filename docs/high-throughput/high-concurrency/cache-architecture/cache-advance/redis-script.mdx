# Redis 管道、事务与 Lua 脚本的性能深度对比

> [!WARNING]
> 前置知识：本篇文章涉及 Redis 网络协议、Redis 持久化机制、Redis 主从复制等知识，建议先掌握 Redis 数据类型和 Redis 线程模型。

---

凌晨两点，你被报警叫醒。监控显示 Redis 的 QPS 突然下降了 60%，但业务请求量并没有变化。一查慢查询日志，发现大量 `GET` 命令的执行时间从 0.1ms 飙升到了 50ms。

你开始排查：Redis 进程没问题，CPU 不高，网络也没拥塞。但仔细一看日志，发现一个规律——所有慢查询都是**批量操作**，一次请求要获取 100 个 key 的值。

你意识到问题所在：每次 `GET key1 GET key2 GET key3 ...` 都是一次独立的网络往返（RTT），100 个 key 就是 100 次 RTT。如果网络延迟是 0.5ms，那么光 RTT 开销就是 50ms。

这就是 Redis 性能优化的经典问题：**如何减少网络往返次数？**

Redis 提供了三种解决方案：管道（Pipeline）、事务（Transaction）、Lua 脚本。它们都能减少 RTT，但原理和适用场景各不相同。

这一讲，我们深入解析这三种方案，理解它们的差异，选择最适合当前业务的方案。

---

## 一、普通 Redis 命令的执行流程

在讨论优化方案之前，我们先理解普通 Redis 命令的执行流程。

### 1.1 RTT 往返延迟

Redis 使用 TCP 协议通信，每次发送命令都需要：

1. 客户端发送命令到服务器（RTT 1）
2. 服务器执行命令
3. 服务器返回结果到客户端（RTT 2）

```text
客户端                              Redis
  │                                   │
  │───── GET key1 ───────────────────>│  RTT 1
  │<─────────────── "value1" ─────────│  RTT 2
  │                                   │
  │───── GET key2 ───────────────────>│  RTT 1
  │<─────────────── "value2" ─────────│  RTT 2
  │                                   │
  │───── GET key3 ───────────────────>│  RTT 1
  │<─────────────── "value3" ─────────│  RTT 2
```

对于 100 个 GET 操作，如果有 1ms 的网络延迟，总延迟就是 **200ms**。

### 1.2 网络延迟的影响

网络延迟（RTT）是 Redis 性能的重要瓶颈：

| 网络环境 | 单次 RTT | 100 次 GET 延迟 |
|---|---|---|
| 本地（localhost） | < 0.1ms | 10ms |
| 同一机房 | 0.5ms | 50ms |
| 跨机房 | 5ms | 500ms |
| 跨地域 | 50ms | 5000ms |

在高并发场景下，减少 RTT 是提升性能的关键。

## 二、Redis 管道：Pipeline 的原理与使用

### 2.1 管道的工作原理

管道（Pipeline）是 Redis 客户端提供的功能，它允许**一次发送多个命令，然后一次接收所有响应**。

```text
客户端                              Redis
  │                                   │
  │───── GET key1 ───────────────────>│
  │───── GET key2 ───────────────────>│  批量发送
  │───── GET key3 ───────────────────>│
  │───── GET key4 ───────────────────>│
  │───── GET key5 ───────────────────>│
  │<─────────────── response batch ────│  一次接收
```

管道不是 Redis 服务端的功能，而是**客户端的实现**。Redis 服务端只需要能处理批量请求即可。

### 2.2 管道 vs 普通批量操作

你可能会问：Redis 不是有 `MGET`、`MSET` 这样的批量命令吗？为什么还需要管道？

```bash
# 普通批量操作
MGET key1 key2 key3 key4 key5

# 等价于
GET key1
GET key2
GET key3
GET key4
GET key5
```

`MGET` 是 Redis 服务端原生支持的批量操作，**在服务端一次完成多个 key 的读取**，没有 RTT 问题。

但 `MGET` 有局限性：

1. **只能操作同一种类型**：`MGET` 只能读取 STRING 类型，无法混合读取 STRING、HASH、LIST 等
2. **语义受限**：`MGET` 只能批量读取，无法在批量中混入写入操作

管道可以解决这些问题：

```java
// Jedis 管道示例
Jedis jedis = new Jedis("localhost", 6379);
Pipeline pipeline = jedis.pipelined();

for (int i = 0; i < 100; i++) {
    pipeline.get("key:" + i);
    pipeline.hget("hash:" + i, "field");
    pipeline.lpop("list:" + i);
}

List<Object> results = pipeline.syncAndReturnAll();
// results[0] = key:0 的值
// results[1] = hash:0 的 field 值
// results[2] = list:0 的值
// results[3] = key:1 的值
// ...
```

### 2.3 管道的性能优势

管道的性能提升来自**减少网络往返次数**：

| 方式 | 100 次 GET | RTT 次数 |
|---|---|---|
| 普通方式 | 100 个 GET | 100 次 |
| MGET | 1 个 MGET | 1 次 |
| 管道 | 1 次发送 100 个 GET | 1 次 |

但管道和 MGET 的本质区别是：

- **MGET**：服务端一次性读取多个 key，返回多个值
- **管道**：客户端一次性发送多个命令，服务端依次执行，返回多个结果

管道并不减少服务端的执行次数，只是减少了网络通信次数。

### 2.4 管道的正确使用方式

管道不是越多越好，需要注意以下几点：

**管道大小要合理**

```java
// 错误：管道太大
Pipeline pipeline = jedis.pipelined();
for (int i = 0; i < 1000000; i++) {  // 100万次，管道太大
    pipeline.set("key:" + i, "value:" + i);
}
pipeline.sync();

// 正确：分批处理
for (int batch = 0; batch < 10; batch++) {
    Pipeline pipeline = jedis.pipelined();
    for (int i = 0; i < 100000; i++) {
        pipeline.set("key:" + (batch * 100000 + i), "value:" + i);
    }
    pipeline.sync();
}
```

**管道不是原子的**

管道的多个命令是分开执行的，中间可能被其他客户端的命令插入。管道**不保证原子性**。

```java
// 管道中的两个命令不保证原子执行
pipeline.incr("counter");
pipeline.incr("counter");

// 中间可能被其他客户端的操作插入
// 如果需要原子性，使用 Lua 脚本
```

## 三、Redis 事务：Transaction 的原理与使用

### 3.1 事务的基本命令

Redis 的事务通过 `MULTI` 和 `EXEC` 命令实现：

```bash
MULTI              # 开启事务
SET key1 value1    # 命令入队
INCR counter       # 命令入队
GET key1           # 命令入队
EXEC               # 执行队列中的所有命令
```

事务的执行流程：

```text
客户端                              Redis
  │                                   │
  │───── MULTI ─────────────────────>│  开启事务
  │<─────────────── OK ───────────────│
  │───── SET key1 value1 ────────────>│  命令入队
  │<─────────────── QUEUED ───────────│
  │───── INCR counter ────────────────>│  命令入队
  │<─────────────── QUEUED ───────────│
  │───── EXEC ────────────────────────>│  执行事务
  │<─────────────── [OK, 1, "value1"] │  返回所有结果
```

### 3.2 事务的原子性保障

Redis 事务的原子性体现在：**`EXEC` 执行后，所有命令要么全部成功，要么全部失败**。

```bash
MULTI
SET key1 value1
SET key2 value2
INCR key3          # key3 是 STRING，可以执行
EXEC
```

如果 `SET key1 value1` 失败了，`EXEC` 仍然会执行所有命令，最后返回一个结果数组。

**Redis 事务不支持回滚**。如果 `INCR` 一个不存在的 STRING key，它会成功；如果 `LPUSH` 到一个已经是 STRING 的 key，它会失败。但无论成功失败，其他命令都会继续执行。

这与关系型数据库的 ACID 事务不同。Redis 设计者选择不实现回滚的原因是：

1. **性能**：回滚需要额外的状态记录
2. **简化**：Redis 的命令失败通常是编程错误，不应该由 Redis 来弥补
3. **幂等性**：错误命令不应该被执行，而不是执行后回滚

### 3.3 WATCH 命令：乐观锁机制

Redis 的事务不支持 `BEGIN...COMMIT...ROLLBACK` 模式，但提供了 `WATCH` 命令实现乐观锁。

`WATCH` 的语义是：**如果被监视的 key 在 `EXEC` 执行前被其他客户端修改了，整个事务失败**。

```bash
WATCH balance                # 监视 balance key
GET balance                  # 读取当前余额
MULTI
DECRBY balance 100          # 命令入队
DECRBY credit 100            # 命令入队
EXEC                         # 执行事务
# 如果 balance 在 WATCH 后被修改，返回 (nil)
# 如果 balance 未被修改，执行所有命令
```

这个机制解决了「检查-然后-执行」（Check-Then-Act）的并发问题：

```text
客户端A                          客户端B
  │                               │
  │───── WATCH balance ───────────>│
  │───── GET balance ─────────────>│  返回 1000
  │                               │
  │                               │───── WATCH balance ──>│
  │                               │───── GET balance ────>│  返回 1000
  │                               │───── MULTI ──────────>│
  │                               │───── DECRBY balance ─>│
  │                               │───── EXEC ────────────>│  余额变为 900
  │───── MULTI ──────────────────>│
  │───── DECRBY balance 100 ────>│
  │───── EXEC ──────────────────>│  返回 (nil)，事务失败！
```

### 3.4 事务 vs 管道的关键区别

| 特性 | 管道 | 事务 |
|---|---|---|
| RTT | 一次 RTT | 多次 RTT（MULTI 后每个命令一次 RTT） |
| 原子性 | 不保证 | 保证（EXEC 执行所有或全部失败） |
| 隔离性 | 无 | 无（EXEC 时不阻塞其他客户端） |
| WATCH 支持 | 不支持 | 支持 |
| 回滚 | 不支持 | 不支持 |
| 命令队列 | 客户端 | 服务端 |

**重要结论**：

- **管道减少 RTT，但不保证原子性**
- **事务保证原子性，但不减少 RTT**
- **如果要同时减少 RTT 和保证原子性，需要 Lua 脚本**

## 四、Lua 脚本：原子性的终极保障

### 4.1 Lua 脚本的工作原理

Lua 脚本是 Redis 2.6+ 引入的功能，它允许在 Redis 服务端执行一段 Lua 代码。

```bash
EVAL "
    local balance = redis.call('GET', KEYS[1])
    if tonumber(balance) >= tonumber(ARGV[1]) then
        redis.call('DECRBY', KEYS[1], ARGV[1])
        return redis.call('GET', KEYS[1])
    else
        return -1
    end
" 1 "account:balance" 100
```

Lua 脚本的执行过程：

```text
客户端                              Redis
  │                                   │
  │───── EVAL script KEYS[1] ARGV[1] >│
  │                                   │
  │         ┌─────────────────────────┼─────────────────────────┐
  │         │  Lua 脚本在 Redis 单线程中执行                       │
  │         │                                                  │
  │         │  redis.call('GET', KEYS[1])  → 执行 GET 命令       │
  │         │  redis.call('DECRBY', ...)    → 执行 DECRBY 命令   │
  │         │  return ...                                    │
  │         │                                                  │
  │         └──────────────────────────────────────────────────│
  │<─────────────── script result ───────────│
```

关键点：**Lua 脚本在 Redis 单线程中执行，整个脚本是原子的，不会被其他命令插入**。

### 4.2 Lua 脚本 vs 事务的优势

| 特性 | 事务 | Lua 脚本 |
|---|---|---|
| 原子性 |EXEC 时所有命令批量执行，但中间可能被其他命令插入 | 整个脚本原子执行 |
| RTT | 每个命令一次 RTT（入队 + EXEC） | 一次 RTT |
| 条件判断 | 不支持 | 支持 if/else/while |
| 变量 | 不支持 | 支持局部变量 |
| 结果复用 | 不支持 | 可以在脚本中多次使用结果 |
| 服务端负载 | 低 | 低（但复杂脚本可能阻塞） |

Lua 脚本的核心优势是**可编程**。你可以写 if/else 判断、循环、局部变量，这在事务中是不可能的。

### 4.3 EVAL 和 EVALSHA

EVAL 命令每次都会将脚本发送给 Redis，Redis 每次都要计算脚本的 SHA1 摘要。

```bash
# EVAL：每次发送完整脚本
EVAL "
    local balance = redis.call('GET', KEYS[1])
    return balance
" 1 "key1"

# EVALSHA：发送脚本的 SHA1 摘要
EVALSHA "8c6eac3d5f9a1b2c3d4e5f6a7b8c9d0e1f2a3b4" 1 "key1"
```

如果脚本被频繁执行，可以使用 `SCRIPT LOAD` 预加载脚本，然后使用 `EVALSHA`：

```bash
# 预加载脚本，返回 SHA1 摘要
SCRIPT LOAD "
    local balance = redis.call('GET', KEYS[1])
    return balance
"
# 返回: "8c6eac3d5f9a1b2c3d4e5f6a7b8c9d0e1f2a3b4"

# 使用摘要执行
EVALSHA "8c6eac3d5f9a1b2c3d4e5f6a7b8c9d0e1f2a3b4" 1 "key1"
```

### 4.4 Lua 脚本的常见应用场景

**场景一：库存扣减**

```lua title="库存扣减脚本"
-- KEYS[1]: 库存 key
-- ARGV[1]: 扣减数量

local stock = redis.call('GET', KEYS[1])
if not stock then
    return -1  -- 库存不存在
end

local quantity = tonumber(ARGV[1])
local current = tonumber(stock)

if current < quantity then
    return -2  -- 库存不足
end

redis.call('DECRBY', KEYS[1], quantity)
return current - quantity  -- 返回扣减后的库存
```

**场景二：分布式锁**

```lua title="简化版分布式锁脚本"
-- KEYS[1]: 锁 key
-- ARGV[1]: 锁值（UUID）
-- ARGV[2]: 过期时间（秒）

if redis.call('SETNX', KEYS[1], ARGV[1]) == 1 then
    redis.call('EXPIRE', KEYS[1], ARGV[2])
    return 1
else
    return 0
end
```

**场景三：访问频率限制**

```lua title="滑动窗口限流脚本"
-- KEYS[1]: 限流 key（格式: rate:limit:user:123）
-- ARGV[1]: 窗口大小（秒）
-- ARGV[2]: 最大请求数
-- ARGV[3]: 当前时间戳

local key = KEYS[1]
local window = tonumber(ARGV[1])
local limit = tonumber(ARGV[2])
local now = tonumber(ARGV[3])
local windowStart = now - window

-- 删除窗口外的记录
redis.call('ZREMRANGEBYSCORE', key, '-inf', windowStart)

-- 统计当前窗口内的请求数
local current = redis.call('ZCARD', key)

if current < limit then
    redis.call('ZADD', key, now, now)
    redis.call('EXPIRE', key, window)
    return 1  -- 允许
else
    return 0  -- 拒绝
end
```

## 五、管道 + Lua 脚本的组合使用

### 5.1 为什么需要组合

管道用于减少 RTT，Lua 脚本用于保证原子性。有时候我们需要两者的结合。

但实际上，**Lua 脚本本身就是一种「管道」**——一次发送、一次执行、一次返回。你不需要在 Lua 脚本外再包一层管道。

```java
// 错误：Lua 脚本外又套管道
Jedis jedis = new Jedis("localhost", 6379);
Pipeline pipeline = jedis.pipelined();
pipeline.evalsha(scriptSha, ...);  // 多此一举
pipeline.sync();

// 正确：直接使用 EVALSHA
jedis.evalsha(scriptSha, 1, "key", "arg");
```

### 5.2 管道中执行多个 Lua 脚本

如果你真的需要在一次请求中执行多个 Lua 脚本（虽然不推荐），可以这样做：

```java
// 在一个事务（MULTI/EXEC）中执行多个 EVALSHA
Transaction transaction = jedis.multi();
transaction.evalsha(scriptSha1, 1, "key1", "arg1");
transaction.evalsha(scriptSha2, 1, "key2", "arg2");
transaction.exec();
```

但这种情况下，两个脚本的执行是分开的，不是真正的原子操作。

## 六、脚本缓存管理与 SCRIPT LOAD

### 6.1 脚本缓存的生命周期

Redis 会缓存所有加载过的脚本，直到：

1. Redis 重启
2. 执行 `SCRIPT FLUSH` 清空脚本缓存

```bash
# 查看脚本缓存
SCRIPT EXISTS "8c6eac3d5f9a1b2c3d4e5f6a7b8c9d0e1f2a3b4"

# 清空脚本缓存（生产环境慎用）
SCRIPT FLUSH
```

### 6.2 脚本超时与 KILL

Lua 脚本默认没有超时限制。如果脚本陷入死循环，会阻塞 Redis。

Redis 5.0+ 支持脚本超时：

```bash
# 设置脚本最大执行时间（毫秒）
lua-time-limit 5000
```

当脚本执行超时后：

1. Redis 会发送 `SIGALRM` 信号
2. 如果脚本没有响应，Redis 会强制终止脚本
3. 可以使用 `SCRIPT KILL` 手动终止

```bash
# 查看正在运行的脚本
SCRIPT LIST

# 终止正在运行的脚本
SCRIPT KILL
```

## 七、性能对比测试数据

### 7.1 RTT 对比

以下是 1000 次 SET 操作在不同方式下的耗时对比（本地 Redis，网络延迟 < 0.1ms）：

| 方式 | RTT 次数 | 耗时 |
|---|---|---|
| 普通方式 | 1000 | ~100ms |
| MGET/MSET | 1 | ~1ms |
| 管道 | 1 | ~1ms |
| Lua 脚本 | 1 | ~1-2ms |

### 7.2 原子性对比

以下是不同方式在并发场景下的表现：

| 场景 | 普通方式 | 管道 | 事务 | Lua 脚本 |
|---|---|---|---|---|
| 100 个 GET | 并发安全 | 并发安全 | 并发安全 | 并发安全 |
| INCR counter × 100 | 可能丢失更新 | 可能丢失更新 | 可能丢失更新 | **保证原子** |
| 余额扣减 | 检查-执行非原子 | 非原子 | WATCH 可部分保证 | **完全保证** |

### 7.3 吞吐量对比

以下是不同方式在批量操作场景下的吞吐量（本地 Redis）：

```java
// 测试配置
int count = 10000;
String key = "test:key:";

// 普通方式
long start = System.currentTimeMillis();
for (int i = 0; i < count; i++) {
    jedis.set(key + i, "value" + i);
}
System.out.println("普通方式: " + (System.currentTimeMillis() - start) + "ms");

// 管道
start = System.currentTimeMillis();
Pipeline pipeline = jedis.pipelined();
for (int i = 0; i < count; i++) {
    pipeline.set(key + i, "value" + i);
}
pipeline.sync();
System.out.println("管道: " + (System.currentTimeMillis() - start) + "ms");

// Lua 脚本（批量 SET）
start = System.currentTimeMillis();
String luaScript = "for i = 1, #KEYS do redis.call('SET', KEYS[i], ARGV[i]) end return #KEYS";
List<String> keys = new ArrayList<>();
List<String> args = new ArrayList<>();
for (int i = 0; i < count; i++) {
    keys.add(key + i);
    args.add("value" + i);
}
jedis.eval(luaScript, keys, args);
System.out.println("Lua: " + (System.currentTimeMillis() - start) + "ms");

// 预期结果
// 普通方式: ~1000ms
// 管道: ~50ms
// Lua: ~30ms
```

## 八、选型建议与最佳实践

### 8.1 选择决策树

```text
需要批量操作吗？
│
├─ 是：只操作同类型（STRING/HASH）？
│   ├─ 是：使用 MGET/MSET（最高效）
│   └─ 否：使用管道
│
├─ 否：需要原子性吗？
│   ├─ 否：使用普通命令
│   └─ 是：需要条件判断？
│       ├─ 否：使用事务 + WATCH
│       └─ 是：使用 Lua 脚本
```

### 8.2 场景选型建议

| 场景 | 推荐方案 | 原因 |
|---|---|---|
| 批量读取 100 个 STRING key | `MGET` | 服务端原生支持，最高效 |
| 批量写入 100 个 STRING key | `MSET` 或 `MSETNX` | 服务端原生支持 |
| 批量读取混合类型 | 管道 | 支持异构操作 |
| 库存扣减 | Lua 脚本 | 需要原子性 + 条件判断 |
| 分布式锁 | SET NX PX | Redis 原生支持，可配合 Lua |
| 频率限制 | Lua 脚本 | 需要原子性 + 复杂逻辑 |
| 排行榜批量更新 | 管道 | 原子性要求不高 |
| 乐观锁检查 | 事务 + WATCH | Redis 原生支持 |

### 8.3 最佳实践

**实践一：预加载高频脚本**

```java
// 启动时预加载脚本
public class RedisScriptLoader {
    
    private String deductScriptSha;
    
    @PostConstruct
    public void init() {
        deductScriptSha = jedis.scriptLoad(deductScript);

}
```

## 九、总结

Redis 的管道、事务、Lua 脚本是三种不同的批量操作方式，各有优劣。

| 方案 | RTT | 原子性 | 条件判断 | 适用场景 |
|---|---|---|---|---|
| 普通命令 | N 次 | 无 | 无 | 单次操作 |
| MGET/MSET | 1 次 | 无 | 无 | 同类型批量读写 |
| 管道 | 1 次 | 无 | 无 | 异构批量操作，不要求原子 |
| 事务 | N 次 | 有 | 无 | 需要原子但不需要判断 |
| WATCH | N 次 | 部分 | 无 | 乐观锁场景 |
| Lua 脚本 | 1 次 | 有 | 有 | 需要原子 + 判断 |

**核心原则**：

- **减少 RTT**：使用管道或 Lua 脚本
- **保证原子性**：使用事务或 Lua 脚本
- **需要条件判断**：只能使用 Lua 脚本
- **简单批量读写**：优先使用 MGET/MSET

## 面试追问预判

**面试官可能追问的方向：**

1. **Redis 事务为什么不支持回滚**：这和关系型数据库的 ACID 事务有什么区别？为什么 Redis 选择这样设计？

2. **Lua 脚本的性能问题**：如果 Lua 脚本执行时间过长，会影响 Redis 的其他请求吗？Redis 怎么处理这种情况？

3. **管道的客户端实现**：管道是 Redis 服务端的功能还是客户端的功能？不同客户端（Jedis/Lettuce/Redisson）的管道实现有什么区别？

4. **EVALSHA 的问题**：如果脚本被 `SCRIPT FLUSH` 清空后，使用 `EVALSHA` 会返回什么错误？如何处理这种情况？

5. **Lua 脚本的调试**：Lua 脚本写错了怎么办？有没有类似 GDB 的调试工具？

这些问题深入到了 Redis 网络协议、服务端实现、客户端实现等多个层面。如果你能回答清楚，说明你对 Redis 的内部机制有深入的理解。
