# 分布式缓存：Redis Cluster 与 Memcache 对比

双十一零点刚过，你正准备庆祝系统稳定运行，突然监控告警：Redis 主节点 CPU 占用 100%，连接数达到上限，所有请求开始排队。你的第一反应是什么？扩容、加机器、改配置——但问题是：**单机 Redis 的天花板到底在哪？为什么 Redis Cluster 要设计成分布式架构？**

这个问题背后，是所有高性能系统都要面对的核心矛盾：**单机性能再强，也有上限**。分布式缓存的出现，就是为了打破这个上限，让缓存容量和吞吐量可以随着机器数量线性扩展。

## 为什么需要分布式缓存

在说分布式缓存之前，我们先明确一个概念：**本地缓存解决的是「网络延迟」问题，分布式缓存解决的是「数据共享」和「容量扩展」问题**。

本地缓存受限于单机内存，数据无法跨进程共享。当你有 10 台应用服务器时，每台都有一份本地缓存，那「用户 Session」怎么共享？「库存扣减」怎么保证一致性？

分布式缓存的核心价值：

**1. 跨进程数据共享**：所有应用实例访问同一份数据，天然解决了数据一致性问题
**2. 容量可扩展**：通过集群扩展，单集群可支撑 TB 级数据
**3. 故障高可用**：主从复制 + 故障自动转移，单节点故障不影响服务
**4. 集中管理**：缓存数据集中存储，方便监控、运维、清理

但分布式缓存也有代价：

- **网络延迟**：每次访问都要走网络，即使在同一个机房，也有 0.5-1ms 的延迟
- **运维复杂度**：集群部署、节点扩容、数据迁移都比单机复杂
- **一致性问题**：在分布式环境下，强一致性更难保证

:::tip 选型原则
需要跨节点共享的数据（如 Session、分布式锁、计数器），必须用分布式缓存。热点只读数据（如商品详情页），可以用本地缓存 + 分布式缓存的组合。
:::

## Redis vs Memcache：核心区别

这是面试中最常被问到的问题。很多人的回答是「Redis 支持更多数据类型」，但这只是表面。两者真正的区别在于**架构哲学**。

| 特性 | Redis | Memcache |
|---|---|---|
| 数据类型 | String、Hash、List、Set、ZSet 等 9 种 | 仅 String（二进制安全） |
| 持久化 | 支持 RDB、AOF、混合持久化 | 不支持，纯内存 |
| 复制 | 主从复制、哨兵、集群 | 无复制，仅单机 |
| 事务 | 支持 Pipeline + Lua 脚本 | 仅 Pipeline |
| 内存管理 | 自定义内存管理，支持内存碎片优化 | slab allocator |
| 单机性能 | QPS 约 10-20 万 | QPS 约 30-50 万 |
| 客户端 | smart client（直连集群） | 多线程，libevent |

**关键区别解读**：

1. **持久化**：Redis 可以把数据落盘，适合需要「缓存 + 持久化」的场景；Memcache 纯内存，宕机即丢失

2. **多线程 vs 单线程**：
   - Memcache 是多线程模型（使用 libevent），可以充分利用多核 CPU
   - Redis 6.0 之前是单线程模型（6.0 之后支持多线程 IO），但单线程避免了锁竞争，开销更小

3. **内存分配**：
   - Memcache 使用 slab allocator，减少内存碎片，但内存利用率固定
   - Redis 使用自定义内存管理，可以更灵活地控制内存使用

:::warning 性能误解
很多人认为 Memcache 比 Redis 快，这是在「纯 get/set 场景、单线程对比」下的结论。但 Redis 6.0 引入多线程 IO 后，两者差距已经很小。实际项目中，Redis 的丰富功能带来的收益远大于微弱的性能差距。
:::

## Redis Cluster 架构

Redis Cluster 是 Redis 官方提供的分布式解决方案，核心设计目标是：**无中心化、自动故障转移、水平扩展**。

### 槽分片机制

Redis Cluster 把整个 key 空间划分为 **16384 个槽**（slot），每个槽可以存储大量 key-value 数据。当客户端要存取一个 key 时，Redis Cluster 通过以下公式计算这个 key 属于哪个槽：

```java
slot = CRC16(key) % 16384
```

```
┌─────────────────────────────────────────────────────────┐
│                    Redis Cluster                        │
│                                                         │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐   │
│  │ Slot 0  │  │ Slot 1  │  │ Slot 2  │  │ Slot N  │   │
│  │ (0-4095)│  │(4096-8191│  │(8192-12287)│(12288-16383)│
│  └────┬────┘  └────┬────┘  └────┬────┘  └────┬────┘   │
│       │            │            │            │          │
│  ┌────▼────┐  ┌────▼────┐  ┌────▼────┐  ┌────▼────┐   │
│  │Master 1 │  │Master 2 │  │Master 3 │  │Master N │   │
│  │ 6380    │  │ 6380    │  │ 6380    │  │ 6380    │   │

│  └─────────┘  └─────────┘  └─────────┘  └─────────┘   │
└─────────────────────────────────────────────────────────┘
```

**16384 个槽的好处**：

- 槽数量不大，节点间同步开销小（Gossip 协议）
- 2KB 的位图可以表示所有槽的状态
- 计算简单，客户端可以快速定位槽

### 无中心化设计

Redis Cluster 采用 **Gossip 协议**实现去中心化。节点之间定期交换信息，包括：

- 我负责哪些槽
- 其他节点的状态
- 槽的迁移进度

```java
// Gossip 消息类型
GOSSIP_HELLO       // 节点发现
GOSSIP_PING        // 心跳检测
GOSSIP_PONG        // 响应心跳
GOSSIP_MEET        // 新节点加入
GOSSIP_FAIL        // 节点下线
GOSSIP_PUBLISH     // 消息广播
```

每个节点只需要与部分节点通信（大约 O(log n)），就能把信息扩散到整个集群。这种设计的好处：

- **无单点故障**：没有中心节点，整个集群不依赖任何一个节点
- **去中心化**：客户端可以连接任意节点，不需要经过代理
- **自动发现**：新节点加入集群后，自动被其他节点感知

### 故障自动转移

Redis Cluster 的故障转移依靠 **主从复制 + 投票机制**：

**1. 主观下线（Subjectively Down）**：
从节点定期 ping 主节点，如果主节点超过 `cluster-node-timeout` 时间没有响应，从节点就认为主节点「可能挂了」。

**2. 客观下线（Objectively Down）**：
当超过半数以上持有槽的主节点都认为某个主节点下线时，才认定该节点确实下线。

**3. 故障转移**：
从节点通过选举成为新的主节点，其他节点更新映射关系，客户端通过 MOVED 重定向到新主节点。

```
原 Master 宕机
     ↓
从节点发现主节点不可达（主观下线）
     ↓
多个从节点投票，选择一个成为新主节点（选举）
     ↓
新主节点接管原主节点负责的槽
     ↓
客户端通过 MOVED 重定向到新主节点
```

:::tip 故障转移时间
Redis Cluster 的故障转移时间大约是 10-30 秒（取决于 cluster-node-timeout 配置）。对于需要更高可用性的场景，可以考虑 Redis Sentinel 做更快的故障转移。
:::

## Redis Cluster 的槽迁移

当集群需要扩容或缩容时，需要迁移槽。Redis Cluster 支持在线迁移槽，数据可以平滑过渡，不会造成服务中断。

### 迁移过程

```bash
# 1. 新节点加入集群
redis-cli --cluster add-node 192.168.1.10:6379 192.168.1.1:6379

# 2. 开始迁移槽（以槽 5000 为例）
redis-cli --cluster reshard 192.168.1.1:6379 \
  --cluster-from <源节点ID> \
  --cluster-to <新节点ID> \
  --cluster-slots 5000
```

**迁移原理**：

1. 源节点把属于目标槽的所有 keys 扫描出来
2. 对每个 key 执行 `MIGRATE` 命令，把数据迁移到目标节点
3. 源节点标记槽为「迁移中」状态（ASKING）
4. 客户端访问时，如果 key 还在源节点，返回 ASK 响应，引导客户端去目标节点获取

```java
// 迁移中的客户端处理
JedisCluster jedis = new JedisCluster(nodes);

try {
    jedis.get("key");  // 可能返回 MOVED 或 ASK
} catch (JedisMovedDataException e) {
    // MOVED：槽已完全迁移，更新本地槽映射
    // ASK：槽正在迁移，先发送 ASKING，再获取数据
}
```

**ASK vs MOVED**：

| 响应 | 含义 | 处理方式 |
|---|---|---|
| MOVED | 槽永久迁移到新节点 | 更新本地映射，下次直接访问新节点 |
| ASK | 槽正在迁移中 | 先发送 ASKING 命令，再访问新节点 |

### 迁移注意事项

```java
// ❌ 迁移期间批量操作可能出问题
List<String> keys = Arrays.asList("key1", "key2", "key3");
List<String> values = jedis.mget(keys.toArray(new String[0]));
// 如果 key1 在源节点，key2 在目标节点，可能得到 null

// ✅ 迁移期间使用 Pipeline 时需要特殊处理
// 应该分批 Pipeline，确保同一批 keys 都在同一节点
```

:::warning 迁移期间的一致性
槽迁移过程中，如果源节点收到写请求，数据会同时写入源节点和目标节点。但如果在迁移完成前源节点宕机，可能丢失最后一部分数据。对于需要强一致性的场景，建议在迁移期间暂停写入。
:::

## Redis Sentinel vs Redis Cluster

很多人在选型时会纠结：Sentinel 和 Cluster 到底用哪个？

| 特性 | Redis Sentinel | Redis Cluster |
|---|---|---|
| 数据分片 | 无，所有数据在同一节点 | 16384 槽自动分片 |
| 容量 | 单机容量 | 集群总容量（多机） |
| 写能力 | 单机写 | 多机并发写 |
| 故障转移 | Sentinel 投票决定 | 从节点自动选举 |
| 客户端支持 | 任意 Redis 客户端 | 需要支持 Cluster 的客户端 |
| 适用场景 | 高可用、主从复制 | 高可用 + 水平扩展 |

**选型建议**：

- **数据量小（< 10GB）、需要高可用、写请求不多**：Sentinel + 主从复制
- **数据量大（> 10GB）、需要水平扩展**：Redis Cluster

## Memcache 的多线程模型

Memcache 采用多线程模型，使用 **libevent** 实现事件驱动。这与 Redis 6.0 之前的设计形成鲜明对比。

### libevent 事件驱动

```c
// Memcache 的核心架构
struct event_base *base;
struct event *listen_event;

// 初始化 libevent
base = event_base_new();

// 添加监听 socket 事件
listen_event = event_new(base, listen_fd, EV_READ | EV_PERSIST,
                          accept_conn, conns);

// 启动事件循环
event_base_dispatch(base);
```

**多线程处理流程**：

1. 主线程监听端口，接受连接
2. 连接被分配到工作线程（基于连接数负载均衡）
3. 每个工作线程维护自己的 event_base
4. 工作线程独立处理读写事件

```
         ┌─────────────┐
         │  主线程      │
         │  (接受连接)  │
         └──────┬──────┘
                │
       ┌────────┼────────┐
       ▼        ▼        ▼
   ┌───────┐┌───────┐┌───────┐
   │Worker1││Worker2││Worker3│
   │Thread ││Thread ││Thread │
   └───┬───┘└───┬───┘└───┬───┘
       │        │        │
       ▼        ▼        ▼
   ┌───────┐┌───────┐┌───────┐
   │Event  ││Event  ││Event  │
   │Base 1 ││Base 2 ││Base 3 │
   └───┬───┘└───┬───┘└───┬───┘
```

### slab allocator 内存管理

Memcache 使用 **slab allocator** 管理内存，这与 Redis 的内存管理有本质区别。

```c
// slab allocator 的工作原理
// 1. 预分配固定大小的内存块（slab）
// 2. 每个 slab 大小相同（如 1MB）
// 3. 每个 slab 包含多个相同大小的 chunk
// 4. 不同 slab 的 chunk 大小按因子增长（如 1.25 倍）

slab class 1: chunk_size = 96 bytes,  num = 10922
slab class 2: chunk_size = 120 bytes, num = 8738
slab class 3: chunk_size = 150 bytes, num = 6990
slab class 4: chunk_size = 188 bytes, num = 5590
...
```

**优点**：

- 避免了内存碎片
- 分配/释放 O(1) 时间复杂度

**缺点**：

- 内存利用率固定，如果数据大小与 chunk 不匹配，会浪费空间
- 预分配内存，即使不用也会占用

:::tip 内存利用率优化
Memcache 的 `-f` 参数控制增长因子，`-I` 参数控制最大 chunk 大小。合理配置这两个参数可以优化内存利用率。Redis 在这方面更灵活。
:::

## Redis 的演进：从单线程到多线程 IO

Redis 6.0 之前采用单线程模型，很多人对此感到困惑：**单线程怎么做到高并发的？**

### 单线程模型的优势

Redis 的单线程指的是 **命令处理**是单线程，但 **网络 IO 和数据读写** 并不是单线程。Redis 使用 IO 多路复用（select/poll/epoll）处理并发连接。

```java
// Redis 6.0 之前的处理模型
while (true) {
    // 1. 等待事件发生（IO 多路复用）
    aeApiPoll();
    
    // 2. 处理已就绪的事件
    for (j = 0; j < numevents; j++) {
        // 3. 读取命令
        readQueryFromClient();
        
        // 4. 执行命令（单线程，无锁）
        processCommand();
        
        // 5. 写回响应
        sendReplyToClient();
    }
}
```

**单线程的优势**：

1. **无锁竞争**：不需要考虑并发修改数据的问题
2. **原子操作**：每个命令执行期间不会被中断
3. **简单可靠**：代码逻辑简单，不容易出 bug

**单线程的瓶颈**：

1. CPU 利用率低：无法利用多核 CPU
2. 阻塞操作：慢查询会阻塞整个服务器

### Redis 6.0 多线程 IO

Redis 6.0 引入了 **多线程 IO**，将网络读写并行化：

```yaml
# redis.conf
io-threads 4  # 启用 4 个 IO 线程
io-threads-do-reads yes  # IO 线程也处理读操作
```

**处理流程变化**：

1. 主线程 accept 连接
2. 主线程读取请求，放入队列
3. IO 线程并行读取命令、解析命令
4. 主线程执行命令
5. 主线程把响应放入队列
6. IO 线程并行写回响应

```
主线程：accept → 放入读队列 → 执行命令 → 放入写队列
                              ↓
IO 线程1：读取 Client1 → 读取 Client2 → ...
IO 线程2：写回 Client3 → 写回 Client4 → ...
```

:::tip Redis 7.0 的新特性
Redis 7.0 引入了 **多模式集群（Cluster Sharding）** 和 **Function**（替代 Lua 脚本）。这些新特性使得 Redis 在微服务架构中更加好用。
:::

## 客户端路由：smart client vs 代理模式

Redis Cluster 需要客户端知道「key 在哪个节点」，这就涉及到路由策略。

### smart client 模式

smart client（智能客户端）内置槽映射表，可以直接定位 key 所在的节点：

```java
// JedisCluster 的路由逻辑
public class JedisClusterCRC16 {
    // 计算槽号
    public static int getCRC16(String key) {
        int slot = CRC16(key) % 16384;
        return slot;
    }
    
    // 获取槽对应的节点
    public Jedis getSlotNode(int slot) {
        // 本地缓存槽映射表
        // 如果槽已迁移，返回 MOVED 或 ASK
        return slot2node.get(slot);
    }
}

// 客户端直接定位节点
Jedis node = jedisCluster.getSlotNode(slot);
String result = node.get(key);
```

**smart client 的优势**：

- 性能最优：无需经过代理，网络路径最短
- 支持 ASK/MOVED 自动重定向
- 减少代理层的单点瓶颈

**smart client 的劣势**：

- 客户端实现复杂
- 槽迁移时需要更新客户端的槽映射表

### 代理模式

代理模式使用中间件（如 Codis、Redis-Proxy）统一管理槽映射：

```yaml
# 客户端连接代理
Client → Redis-Proxy → Redis Cluster
         (统一路由)
```

**Codis 架构**：

```
┌─────────────────────────────────────────────────────────┐
│                     Codis Proxy                         │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐     │
│  │   Proxy 1   │  │   Proxy 2   │  │   Proxy 3   │     │
│  └──────┬──────┘  └──────┬──────┘  └──────┬──────┘     │
│         │                │                │             │
│         └────────────────┼────────────────┘             │
│                          │                              │
│                   ┌──────▼──────┐                      │
│                   │ Dashboard   │                      │
│                   │ (槽位管理)   │                      │
│                   └──────┬──────┘                      │
└──────────────────────────┼──────────────────────────────┘
                           │
         ┌─────────────────┼─────────────────┐
         ▼                 ▼                 ▼
    ┌─────────┐       ┌─────────┐       ┌─────────┐
    │Redis 1  │       │Redis 2  │       │Redis 3  │
    │(Slot 0) │       │(Slot 1) │       │(Slot 2) │
    └─────────┘       └─────────┘       └─────────┘
```

**代理模式的优势**：

- 客户端无需感知集群结构，兼容单机 Redis 协议
- 统一管理槽映射，迁移更灵活
- 可以做读写分离、流量控制

**代理模式的劣势**：

- 多一跳网络延迟
- 代理本身可能成为瓶颈

:::tip 选型建议
新项目优先使用 smart client（如 JedisCluster、Lettuce）。代理模式适合需要兼容老代码或者需要统一管控的场景。
:::

## 分布式缓存的一致性保证

分布式缓存面临的核心挑战是：**如何保证多节点间的数据一致性？**

### 主从复制的一致性

Redis Cluster 采用 **异步复制**，主节点写入后，立即返回成功，然后异步复制到从节点。

```java
// 主节点写入
redis.set("key", "value");  // 立即返回 OK
// 异步复制到从节点（主节点不等待从节点确认）
// 如果此时主节点宕机，从节点可能没有这条数据
```

**异步复制的风险**：

1. 主节点写入后宕机，数据可能丢失
2. 从节点的数据可能滞后于主节点

**解决方案：WAIT 命令**

```java
// 写入后等待指定数量的从节点确认
redis.set("key", "value");
redis.wait(2, 0);  // 等待 2 个从节点确认，0 表示无限等待
// 注意：WAIT 只是提高可靠性，无法保证绝对一致
```

### 故障转移与数据一致性

Redis Cluster 的故障转移可能导致短暂的数据不一致：

```bash
# 场景：主节点写入后宕机
# 1. Client 写入 Master: key=value
# 2. 异步复制到 Slave（还未完成）
# 3. Master 宕机
# 4. Slave 被选为新 Master
# 5. Client 读取：新 Master 没有 key
```

**处理方式**：

1. **接受短暂不一致**：对于大多数业务场景，异步复制已经足够
2. **使用 WAIT 命令**：要求写操作同步到指定数量的从节点
3. **读主写主**：写操作只读主节点，降低读到旧数据的概率

### 分布式缓存与本地缓存的一致性

这是最容易出问题的场景。当分布式缓存更新时，本地缓存可能还持有旧数据：

```java
// 更新分布式缓存
public void updateUser(Long userId, User newUser) {
    // 1. 更新数据库
    userDAO.update(userId, newUser);
    
    // 2. 更新 Redis
    redis.set("user:" + userId, newUser);
    
    // 3. 通知所有实例清除本地缓存（消息广播）
    messageBroker.publish("user:update", userId);
}

// 各实例收到消息后清除本地缓存
@Subscribe
public void onUserUpdate(Long userId) {
    localCache.invalidate(userId);
}
```

:::warning 一致性 vs 性能
严格的一致性会牺牲性能。实际项目中，应该根据业务需求选择合适的一致性级别：强一致性（分布式锁）、最终一致性（延迟双删）、弱一致性（本地缓存 + TTL）。
:::

## 总结

分布式缓存是现代高并发系统的基础组件，它解决了本地缓存无法跨进程共享的问题，同时也带来了新的挑战。

| 特性 | Redis | Memcache |
|---|---|---|
| 数据类型 | 丰富 | 仅 String |
| 持久化 | 支持 | 不支持 |
| 集群 | Cluster（官方） | 无官方方案 |
| 适用场景 | 通用缓存、分布式锁、消息队列 | 简单 Session、页面缓存 |

**选型建议**：

- **通用场景、丰富功能**：Redis（推荐）
- **简单缓存、高性能**：Memcache（特定场景）
- **需要水平扩展**：Redis Cluster
- **需要高可用但数据量小**：Redis Sentinel

## 面试追问预判

**面试官可能会问：**

1. **Redis Cluster 是如何实现故障转移的？**
   答：Redis Cluster 使用 Raft 协议的简化版本进行主从选举。当主节点下线后，从节点通过投票获得多数支持后成为新主节点，接管原主节点的槽。关键点：需要超过半数的主节点参与投票。

2. **Redis Cluster 为什么用 16384 个槽？**
   答：16384 是 2 的 14 次方，方便用位图表示槽状态；数量不大，节点间心跳包和 Gossip 消息不会太大；又能保证足够的分片粒度。这是一个工程权衡的结果。

3. **Redis Cluster 和 Codis 有什么区别？**
   答：Codis 是第三方代理方案，通过代理转发请求，支持 key 自动迁移；Redis Cluster 是官方方案，客户端直接连接节点，需要 smart client 支持。Codis 更适合需要平滑迁移的场景。

4. **如何保证 Redis Cluster 的高可用？**
   答：主从复制 + 故障自动转移 + 合理配置 cluster-node-timeout + 监控告警。关键配置：`replica-read-only yes`（从节点只读）、`min-replicas-to-write 2`（至少 2 个从节点可用）。

5. **Redis Cluster 能保证强一致性吗？**
   答：不能。Redis Cluster 采用异步复制，存在数据丢失的风险。对于需要强一致性的场景，可以使用 `WAIT` 命令提高可靠性，或者使用外部分布式锁（如 RedLock）。
