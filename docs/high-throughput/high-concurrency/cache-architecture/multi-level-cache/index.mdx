# 多级缓存架构：层层拦截请求

> [!WARNING]
> 前置知识：本篇文章是多级缓存架构的入口，涵盖本地缓存、分布式缓存、CDN 缓存、热点数据四大方向。建议按顺序阅读，理解多级缓存的协同设计思路。

---

你有没有想过，一个每秒 10 万 QPS 的商品查询系统，是怎么做到数据库几乎不被击穿的？

答案就是**多级缓存**——L1 本地缓存拦截 60% 的请求，L2 分布式缓存再拦截 30%，只有不到 10% 的请求会穿透到数据库。数据库只需要承受原来的十分之一压力。

但多级缓存不是简单的「多加几层」。每层缓存的特性不同、定位不同、解决的问题也不同。设计不好，多级缓存反而会成为一致性的噩梦。

这一讲，我们来理解多级缓存的架构设计与协同机制。

---

## 一、知识版图：多级缓存全景

│  │  容量：TB 级                                                  │   │
│  │  命中率：90%+                                                 │   │

### 各层缓存的定位

| 层级 | 组件 | 定位 | 优势 | 劣势 |
|---|---|---|---|---|
| L0 | CDN | 静态资源加速 | 边缘节点、低延迟 | 仅限静态资源 |
| L1 | Caffeine / Guava Cache | 热点数据拦截 | 纳秒级访问、无网络开销 | 节点独立、数据不一致 |
| L2 | Redis Cluster | 共享数据存储 | 集群共享、一致性保证 | 网络延迟 |
| L3 | MySQL | 数据最终来源 | 强一致性 | 延迟高 |

## 二、本地缓存：L1 的极速体验

### 为什么需要本地缓存？

Redis 的访问延迟通常是 0.5-2ms，看起来很快。但对于高频热点数据，这个延迟仍然不可忽视。本地缓存的访问延迟是纳秒级（10-100ns），比 Redis 快 1 万倍以上。

```java
// 没有本地缓存：每次都要访问 Redis
public Product getProduct(Long id) {
    Product p = redisTemplate.opsForValue().get("product:" + id);
    if (p == null) {
        p = productMapper.selectById(id);
        redisTemplate.opsForValue().set("product:" + id, p);
    }
    return p;
}

// 有本地缓存：热点数据直接返回
private final LoadingCache<Long, Product> localCache = Caffeine.newBuilder()
    .maximumSize(10_000)
    .expireAfterWrite(60, TimeUnit.SECONDS)
    .build(id -> null); // 查不到时返回 null，由上层处理

public Product getProduct(Long id) {
    // 先查本地缓存
    Product p = localCache.getIfPresent(id);
    if (p != null) {
        return p;
    }
    // 本地缓存未命中，查 Redis
    p = redisTemplate.opsForValue().get("product:" + id);
    if (p != null) {
        localCache.put(id, p);
    }
    return p;
}
```

### 三大本地缓存对比

| 对比项 | Caffeine | Guava Cache | Ehcache |
|---|---|---|---|
| 性能 | 最高 | 高 | 中 |
| 功能丰富度 | 中 | 高 | 高 |
| 与 Spring 集成 | 需手动集成 | 需手动集成 | 支持 Spring Cache 注解 |
| 过期策略 | write/访问/手动 | write/访问/手动 | write/访问/手动 |
| 淘汰策略 | LRU/LFU/Random | LRU/LFU/TTL | LRU/LFU/FIFO |

> [!TIP]
> Caffeine 是目前 Java 生态中性能最高的本地缓存，推荐优先选择。如果需要更丰富的功能（如磁盘持久化），可以考虑 Ehcache。

### 本地缓存的陷阱：数据不一致

本地缓存最大的问题是**节点间的数据不一致**。当数据变更时，各节点的本地缓存不会同步更新。

解决方案取决于业务场景：

| 场景 | 方案 | 说明 |
|---|---|---|
| 读多写少 | 接受短期不一致 | 变更后等待缓存自然过期 |
| 写操作频繁 | 禁止本地缓存 | 直接使用分布式缓存 |
| 写后立即读 | 主动失效 | 变更后同时删除本地缓存和分布式缓存 |

## 三、分布式缓存：L2 的全局共享

### Redis vs Memcache

| 对比项 | Redis | Memcache |
|---|---|---|
| 数据结构 | String、Hash、List、Set、ZSet 等 | 仅支持 KV |
| 持久化 | 支持 RDB、AOF、混合持久化 | 不支持，重启数据丢失 |
| 集群方案 | Redis Cluster、哨兵、主从复制 | 一致性哈希分片 |
| 性能 | 单机略低于 Memcache | 单机略高 |
| 适用场景 | 复杂数据结构、持久化需求 | 纯 KV、性能要求极高 |

### Redis Cluster 的槽迁移

Redis Cluster 将整个 keyspace 划分为 16384 个槽（slot），每个节点负责一部分槽：

```text
┌──────────────────────────────────────────────────────────────┐
│              Redis Cluster（3 主 3 从）                       │
│                                                              │
│   Slot 0-5460      Slot 5461-10922     Slot 10923-16383     │
│  ┌────────────┐    ┌────────────┐    ┌────────────┐          │
│  │  Node-1   │    │  Node-2   │    │  Node-3   │          │
│  │  (Master) │    │  (Master) │    │  (Master) │          │
│  └─────┬──────┘    └─────┬──────┘    └─────┬──────┘          │
│        │                 │                 │                  │
│  ┌─────▼──────┐    ┌─────▼──────┐    ┌─────▼──────┐          │
│  │  Node-1'  │    │  Node-2'  │    │  Node-3'  │          │
│  │  (Slave)  │    │  (Slave)  │    │  (Slave)  │          │
│  └───────────┘    └───────────┘    └───────────┘          │
└──────────────────────────────────────────────────────────────┘
```

槽迁移过程中，客户端可能会访问到 MOVED 重定向错误，需要客户端支持重试逻辑。

## 四、CDN 缓存：静态资源的加速器

### CDN 的工作原理

CDN（Content Delivery Network）通过在全球部署边缘节点，将静态资源缓存到离用户最近的地方：

```text
┌──────────────────────────────────────────────────────────────┐
│                         CDN 工作流程                          │
│                                                              │
│   用户请求 ──▶  DNS 调度 ──▶ 最近边缘节点                      │
│                                     │                        │
│                                     ▼                        │
│                            缓存命中？                         │
│                              │      │                        │
│                         是   │      │  否                    │
│                              ▼      ▼                        │
│                          直接返回   回源站拉取                 │
│                                     │                        │
│                                     ▼                        │
│                                 缓存到边缘节点                │
│                                 并返回给用户                  │
└──────────────────────────────────────────────────────────────┘
```

### CDN 缓存策略

CDN 的核心是缓存策略，通常使用 HTTP 缓存头控制：

```nginx
# Nginx 配置 CDN 缓存策略
location ~* \.(jpg|png|gif|css|js)$ {
    # 缓存到 CDN 节点，1 天后过期
    expires 1d;
    # 开启 gzip 压缩
    gzip on;
    # 设置缓存 key（按 URL + 参数）
    cache_key $scheme$host$uri$is_args$args;
}
```

## 五、热点数据：识别与应对

### 热点数据识别的三种方式

| 方式 | 原理 | 优点 | 缺点 |
|---|---|---|---|
| 监控 Redis 命令 | 分析 `hotkeys` 或 `redis-cli --hotkeys` | 实时性好 | 影响 Redis 性能 |
| 客户端埋点 | 在应用层统计访问频率 | 无侵入 | 需额外开发 |
| 全链路监控 | 通过 APM 工具（如 SkyWalking）统计 | 精确、完整 | 需要 APM 基础设施 |

### 热点数据的应对策略

识别出热点数据后，有以下应对策略：

**方案一：本地缓存加固**

```java
// 热点数据单独用本地缓存，不走 Redis
private final LoadingCache<Long, Product> hotProductCache = Caffeine.newBuilder()
    .maximumSize(1_000)  // 热点数据量通常不大
    .expireAfterWrite(5, TimeUnit.MINUTES)  // 过期时间短，保证一致性
    .build(id -> redisTemplate.opsForValue().get("product:" + id));
```

**方案二：多副本分散压力**

```
正常数据：key → 1 个节点
热点数据：key → N 个节点（hash:key:1, hash:key:2, ...）
读取时随机选择一个副本
```

**方案三：热点数据预热**

在大促活动开始前，提前将热点数据加载到缓存中：

```java
// 热点数据预热脚本
public void preloadHotData() {
    List<Product> hotProducts = productService.getHotProducts();
    for (Product p : hotProducts) {
        redisTemplate.opsForValue().set("product:" + p.getId(), p);
        localCache.put(p.getId(), p);  // 同时预热本地缓存
    }
}
```

## 延伸思考

多级缓存的设计，本质上是在**访问速度**和**一致性成本**之间做权衡。

L1 缓存最快，但数据不一致风险最高；L3 数据库最慢，但一致性最强。每加一层缓存，都是在「用性能换一致性」。

下次设计多级缓存之前，先问自己：这一层的缓存，数据不一致业务能接受吗？如果接受不了，这层缓存的意义是什么？
