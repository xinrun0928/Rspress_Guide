# 本地缓存（进程缓存）：Caffeine、Guava Cache、Ehcache 对比

凌晨2点，你正在值班。监控大屏上 Redis 的 QPS 突然飙升到 10 万+，RT 从 1ms 暴涨到 50ms。排查后发现：某个热点商品的详情页被大量用户同时访问，而你的缓存层每次都要走网络去 Redis 取数据——**这 1ms 的网络延迟，在高并发下被放大了 100 倍**。

问题出在哪？分布式缓存再快，也快不过「没有网络」。本地缓存，就是把数据直接放在应用进程的内存里，彻底消除网络开销。

## 为什么需要本地缓存

在说本地缓存之前，我们先回顾一下缓存的分层架构：

```yaml
请求链路：
用户 → CDN缓存 → Nginx本地缓存 → 分布式缓存(Redis) → 数据库
```

越靠近用户的位置，延迟越低、吞吐量越高。本地缓存位于应用进程内部，访问延迟通常是**纳秒级**（ns），而 Redis 即使在同一个机房，网络延迟也是**毫秒级**（ms）。两者相差 6 个数量级。

本地缓存的核心价值：

**1. 消除网络开销**：数据就在内存里，不需要 TCP 通信
**2. 降低下游压力**：减少对 Redis/数据库的请求，保护下游服务
**3. 应对突发流量**：秒杀、热点事件时，本地缓存可以扛住第一波流量

但本地缓存也有明显的局限性：

- **进程隔离**：每个实例都有独立的缓存，无法跨进程共享
- **内存有限**：受单机内存限制，不适合存储大量数据
- **数据一致性**：更新时需要处理多节点缓存失效问题
- **内存管理**：需要考虑 OOM 风险，缓存数据不能无限制增长

:::tip 选型原则
本地缓存适合「读多写少、数据量不大、访问极其频繁」的场景。对于数据量超过 100MB 或需要跨节点共享的场景，请选择分布式缓存。
:::

## Guava Cache：简单可靠的经典选择

Guava Cache 是 Google Guava 库的一部分，设计哲学是「实用主义」——API 简洁、功能完备、性能够用。如果你只需要一个本地缓存，不想引入额外依赖，Guava Cache 是首选。

### 核心 API

Guava Cache 提供两种构建方式：

**LoadingCache**：带自动加载功能的缓存，适用于「缓存未命中时需要从源头加载」的场景

```java
LoadingCache<Long, User> userCache = CacheBuilder.newBuilder()
    .maximumSize(10_000)           // 最多存储 10000 条
    .expireAfterWrite(10, TimeUnit.MINUTES)  // 写入后 10 分钟过期
    .build(new CacheLoader<Long, User>() {
        @Override
        public User load(Long userId) {
            // 缓存未命中时，从数据库加载
            return userDAO.findById(userId);
        }
    });

// 使用：自动加载，无需手动处理未命中
User user = userCache.get(1001L);
```

**Cache**：不带自动加载，需要手动检查和填充

```java
Cache<Long, User> userCache = CacheBuilder.newBuilder()
    .maximumSize(10_000)
    .expireAfterWrite(10, TimeUnit.MINUTES)
    .build();

// 手动获取，不存在则返回 null
User user = userCache.getIfPresent(1001L);

// 手动加载并放入缓存
if (user == null) {
    user = userDAO.findById(1001L);
    userCache.put(1001L, user);
}
```

两者的选择很简单：如果你每次都要「先查缓存，没有就查数据库」，用 `LoadingCache`；如果你需要更灵活的控制，用 `Cache`。

### 驱逐策略

Guava Cache 支持四种驱逐策略：

| 策略 | 配置方法 | 适用场景 |
|---|---|---|
| LRU（最近最少使用） | `.maximumSize(N)` | 通用场景，推荐首选 |
| LFU（最不经常使用） | `.maximumWeight(N).weigher(...)` | 访问频率差异大的场景 |
| TTL（时间过期） | `.expireAfterWrite/N/Acess(N, TimeUnit)` | 数据时效性强的场景 |
| 引用回收 | `.weakKeys()/.weakValues()/.softValues()` | 配合 JVM GC 使用 |

**基于大小的驱逐**（LRU）：

```java
Cache<Long, User> cache = CacheBuilder.newBuilder()
    .maximumSize(10_000)  // 超过 10000 条时，移除最近最少使用的
    .build();
```

**基于时间的驱逐**：

```java
Cache<Long, User> cache = CacheBuilder.newBuilder()
    .expireAfterWrite(10, TimeUnit.MINUTES)   // 写入后 10 分钟过期
    .expireAfterAccess(5, TimeUnit.MINUTES)  // 访问后 5 分钟过期（LRU）
    .build();
```

**基于引用的驱逐**：

```java
Cache<Long, User> cache = CacheBuilder.newBuilder()
    .weakValues()   // 值使用弱引用，GC 时可能回收
    .softValues()   // 值使用软引用，内存不足时回收
    .build();
```

:::warning 软引用与内存溢出
使用 `softValues()` 时，如果 JVM 内存不足，缓存可能被大量清除，导致缓存命中率不稳定。对于需要稳定性能的场景，推荐使用 `maximumSize()` + `expireAfterWrite()` 的组合。
:::

### 刷新 vs 过期：两个容易混淆的概念

Guava Cache 有两个容易混淆的 API：

- `expireAfterWrite`：数据过期后，下次访问会阻塞等待加载
- `refreshAfterWrite`：数据过期前，异步刷新数据，访问时可能返回旧值

```java
LoadingCache<Long, User> cache = CacheBuilder.newBuilder()
    .refreshAfterWrite(1, TimeUnit.MINUTES)  // 写入 1 分钟后异步刷新
    .build(new CacheLoader<Long, User>() {
        @Override
        public User load(Long userId) {
            return userDAO.findById(userId);
        }

        @Override
        public ListenableFuture<User> reload(Long userId, User oldValue) {
            // 异步加载，返回 Future
            return executor.submit(() -> userDAO.findById(userId));
        }
    });
```

**两者的核心区别**：

```java
// expireAfterWrite：阻塞等待
User user = cache.get(1001L);
// 如果数据过期，等待加载完成后返回

// refreshAfterWrite：可能返回旧值
User user = cache.get(1001L);
// 如果正在刷新，可能返回旧值，新数据在后台加载
```

:::tip 刷新策略的选择
对于数据一致性要求高的场景，用 `expireAfterWrite`；对于性能优先、数据短暂不一致可接受的场景，用 `refreshAfterWrite`。
:::

## Caffeine：现代高性能缓存

Caffeine 是 Guava Cache 的「精神继承者」，由 Netflix 开源，专门为高性能场景设计。在 benchmark 中，Caffeine 的吞吐量通常是 Guava Cache 的 **3-10 倍**。

Caffeine 的高性能来自于它的核心算法：**W-TinyLFU**（Window TinyLFU）。

### W-TinyLFU 算法

传统的 LRU 算法有一个致命缺陷：**突发流量会污染缓存**。比如一个缓存有 1000 个位置，突然来了 10000 次访问新数据的请求，LRU 会把所有旧数据踢出去，热点数据反而被清掉了。

TinyLFU 的解决思路是：在踢出数据之前，先用频率统计判断「这个数据真的没人访问吗？」。TinyLFU 维护了一个 **Count-Min Sketch** 结构，用很小的空间记录每个 key 的访问频率。

W-TinyLFU 在 TinyLFU 基础上加了 **Window Cache**：

```
┌─────────────────────────────────────────┐
│              TinyLFU Filter              │
│  (Count-Min Sketch，统计访问频率)        │
└─────────────────────────────────────────┘
         ↑                   ↑
    Window Cache        Main Cache
   (5% 容量，近热)     (95% 容量，真正存储)
```

- Window Cache：吸收短期突发流量，保护 Main Cache 不被污染
- Main Cache：存储长期热点数据
- TinyLFU Filter：决定谁应该留在缓存里

### Caffeine 的驱逐策略

**基于大小的驱逐**：

```java
// 基于条目数量
Cache<Long, User> cache = Caffeine.newBuilder()
    .maximumSize(10_000)
    .build();

// 基于权重（自定义每个条目的内存占用）
Cache<Long, User> cache = Caffeine.newBuilder()
    .maximumWeight(100 * 1024 * 1024)  // 最多 100MB
    .weigher((Long key, User value) -> value.getSizeInBytes())
    .build();
```

**基于时间的驱逐**：

```java
Cache<Long, User> cache = Caffeine.newBuilder()
    .expireAfterWrite(10, TimeUnit.MINUTES)   // 写入后过期
    .expireAfterAccess(5, TimeUnit.MINUTES)    // 访问后过期
    .expireAfterAccessAndWrite(10, TimeUnit.MINUTES)  // 访问或写入后过期
    .build();
```

**基于引用的驱逐**：

```java
Cache<Long, User> cache = Caffeine.newBuilder()
    .weakKeys()
    .weakValues()
    .softValues()
    .build();
```

### Caffeine 的异步 API

Caffeine 从设计之初就支持异步操作，这是 Guava Cache 做不到的：

```java
AsyncLoadingCache<Long, User> asyncCache = Caffeine.newBuilder()
    .maximumSize(10_000)
    .refreshAfterWrite(1, TimeUnit.MINUTES)
    .buildAsync(new AsyncCacheLoader<Long, User>() {
        @Override
        public CompletableFuture<User> asyncLoad(Long userId, Executor executor) {
            return CompletableFuture.supplyAsync(() -> userDAO.findById(userId), executor);
        }
    });

// 获取 Future，不阻塞
CompletableFuture<User> futureUser = asyncCache.get(1001L);

// 批量获取
List<CompletableFuture<User>> futures = asyncCache.getAll(Arrays.asList(1001L, 1002L, 1003L));
```

在高性能场景下，异步 API 可以显著提升吞吐量——加载操作在后台执行，不阻塞调用线程。

:::tip Caffeine vs Guava Cache
如果你是新项目，优先选择 Caffeine。Guava Cache 已经进入维护模式，不再活跃开发。Caffeine 完全兼容 Guava Cache 的 API，迁移成本很低。
:::

## Ehcache 3.x：企业级的选择

Ehcache 是 Java 领域历史最悠久的缓存框架，3.x 版本支持**堆内 + 堆外 + 磁盘**三级存储，专门为「需要存储大量数据但又不想全部放堆内存」的场景设计。

### 三级存储架构

```java
CacheManager cacheManager = CacheManagerBuilder.newCacheManagerBuilder()
    .withCache("userCache",
        CacheConfigurationBuilder.newCacheConfigurationBuilder(
            Long.class, User.class,
            ResourcePoolsBuilder.newResourcePoolsBuilder()
                .heap(1_000, EntryUnit.ENTRIES)      // 堆内：1000 条
                .offheap(10, MemoryUnit.MB)          // 堆外：10MB
                .disk(100, MemoryUnit.MB, true)      // 磁盘：100MB
        )
    )
    .build(true);

Cache<Long, User> cache = cacheManager.getCache("userCache", Long.class, User.class);
```

**存储层级**：

| 层级 | 存储位置 | 访问速度 | 容量 | 说明 |
|---|---|---|---|---|
| Heap | JVM 堆内存 | 最快 | 受限于 -Xmx | GC 管理，有 OOM 风险 |
| Offheap | 直接内存 | 快 | 可配置 | 不受 GC 影响，但序列化开销 |
| Disk | 本地磁盘 | 慢 | 可配置 | 持久化，启动时加载 |

### Ehcache 的适用场景

- **Terracotta 集群**：Ehcache 支持 Terracotta 服务器端缓存，可以做分布式缓存
- ** Hibernate 二级缓存**：Java ORM 领域的事实标准
- **需要持久化**：进程重启后缓存数据不丢失

:::warning Ehcache 的复杂性
Ehcache 功能强大，但配置也相对复杂。如果只是简单的本地缓存，Caffeine 是更好的选择。Ehcache 适合「需要多层存储 + 持久化 + 集群」的企业级场景。
:::

## 三者对比

| 特性 | Guava Cache | Caffeine | Ehcache 3.x |
|---|---|---|---|
| 性能 | 中等 | 最高 | 中等（磁盘层较慢） |
| 算法 | LRU/LFU/TTL | W-TinyLFU | LRU/LFU/FIFO |
| 堆外/磁盘 | 不支持 | 不支持 | 支持 |
| 异步 API | 不支持 | 支持 | 支持 |
| 维护状态 | 维护模式 | 活跃开发 | 活跃开发 |
| 依赖大小 | 中等（Guava 整个库） | 小（独立库） | 中等 |
| 适用场景 | 简单场景、快速集成 | 高性能场景 | 企业级、多层存储 |

**选型建议**：

- **简单场景、快速开发**：Guava Cache，API 简洁，文档丰富
- **高性能场景**：Caffeine，性能最优，API 兼容 Guava
- **企业级、超大数据量**：Ehcache 3.x，支持多层存储

## 本地缓存的典型应用场景

### 1. 热点数据缓存

```java
// 商品详情页热点数据
LoadingCache<Long, Product> productCache = Caffeine.newBuilder()
    .maximumSize(10_000)
    .expireAfterWrite(5, TimeUnit.MINUTES)
    .refreshAfterWrite(1, TimeUnit.MINUTES)
    .recordStats()
    .build(id -> productService.getProductFromDB(id));

// 获取商品信息，RT < 1μs
Product product = productCache.get(1001L);
```

### 2. 计算结果缓存

```java
// 复杂报表的计算结果
Cache<String, ReportResult> reportCache = Caffeine.newBuilder()
    .maximumWeight(500 * 1024 * 1024)  // 500MB
    .weigher((key, value) -> value.getSizeInBytes())
    .expireAfterWrite(1, TimeUnit.HOURS)
    .build();

// 报表计算（可能耗时几分钟），结果缓存后秒级响应
ReportResult result = reportCache.get(reportKey, k -> generateReport(k));
```

### 3. 配置数据缓存

```java
// 系统配置，变更频率低
LoadingCache<String, SystemConfig> configCache = Caffeine.newBuilder()
    .maximumSize(1000)
    .expireAfterWrite(1, TimeUnit.HOURS)
    .build(key -> configService.loadConfig(key));

// 路由规则缓存
LoadingCache<String, RouteRule> routeCache = Caffeine.newBuilder()
    .maximumSize(100)
    .refreshAfterWrite(10, TimeUnit.MINUTES)
    .build(key -> routeService.loadRules(key));
```

## 本地缓存的线程安全

本地缓存本身是线程安全的，ConcurrentHashMap 天然支持并发读写。但以下几个场景需要注意：

### 1. 缓存数据本身的线程安全

```java
// 缓存的用户对象可能被多线程修改
Cache<Long, User> cache = Caffeine.newBuilder()
    .maximumSize(10_000)
    .build();

// ❌ 错误：返回的对象可能被其他线程修改
User user = cache.getIfPresent(1001L);
user.setName("new name");  // 危险！其他线程可能也在用这个对象

// ✅ 正确：返回不可变对象或深拷贝
User user = cache.get(1001L);
if (user != null) {
    User copy = User.copyOf(user);  // 返回不可变副本
    copy.setName("new name");       // 修改副本
}
```

### 2. 缓存击穿与雪崩

```java
// ❌ 缓存未命中时，大量请求同时打到数据库
User user = cache.get(1001L, id -> userDAO.findById(id));
// 如果缓存过期，瞬间大量请求都会执行 lambda

// ✅ 正确：使用 synchronized 或锁
LoadingCache<Long, User> cache = Caffeine.newBuilder()
    .maximumSize(10_000)
    .expireAfterWrite(10, TimeUnit.MINUTES)
    .build(key -> {
        // 缓存未命中时执行，这里可能有并发问题
        return userDAO.findById(key);
    });

// ✅ 更好的方式：异步刷新 + 返回旧值
LoadingCache<Long, User> cache = Caffeine.newBuilder()
    .maximumSize(10_000)
    .refreshAfterWrite(1, TimeUnit.MINUTES)  // 后台异步刷新
    .build(key -> userDAO.findById(key));
```

### 3. 缓存与数据库的一致性

```java
// ❌ 先更新数据库，再删除缓存——在高并发下可能产生数据不一致
public void updateUser(Long userId, User newUser) {
    userDAO.update(userId, newUser);
    cache.invalidate(userId);  // 删除缓存
}

// ✅ 更安全的方式：延迟双删
public void updateUser(Long userId, User newUser) {
    userDAO.update(userId, newUser);
    cache.invalidate(userId);  // 第一次删除
    // 异步线程sleep后再次删除（处理并发读）
    CompletableFuture.delayedExecutor(500, TimeUnit.MILLISECONDS)
        .execute(() -> cache.invalidate(userId));
}

// ✅ 更可靠的方式：更新数据库 + 设置短过期时间
public void updateUser(Long userId, User newUser) {
    userDAO.update(userId, newUser);
    // 不删除缓存，而是更新缓存
    cache.put(userId, newUser);
}
```

## 监控与调优

本地缓存需要监控以下指标：

```java
LoadingCache<Long, User> cache = Caffeine.newBuilder()
    .maximumSize(10_000)
    .recordStats()  // 开启统计
    .build(key -> userDAO.findById(key));

// 获取缓存统计
CacheStats stats = cache.stats();
System.out.println("命中率: " + stats.hitRate());           // 0.95
System.out.println("加载次数: " + stats.loadSuccessCount()); // 500
System.out.println("加载失败: " + stats.loadFailureCount()); // 10
System.out.println("驱逐次数: " + stats.evictionCount());    // 1000
```

:::details 关键监控指标

- **hitRate()**：缓存命中率，低于 80% 需要分析原因
- **evictionCount()**：被驱逐的条目数，如果频繁驱逐，考虑扩容
- **loadFailureCount()**：加载失败次数，说明下游服务有问题
- **averageLoadPenalty()**：平均加载时间，如果过长，考虑预加载

:::

## 总结

本地缓存是高性能系统的重要组成部分，它的核心价值是**消除网络开销**，将访问延迟从毫秒级降到纳秒级。

| 缓存 | 核心优势 | 适用场景 |
|---|---|---|
| Guava Cache | 简单可靠、文档丰富 | 快速开发、低性能要求 |
| Caffeine | W-TinyLFU、高性能 | 高并发场景（推荐首选） |
| Ehcache 3.x | 多层存储、持久化 | 企业级、超大数据量 |

选型时记住一个原则：**先用 Caffeine，如果不够再考虑 Ehcache**。

## 面试追问预判

**面试官可能会问：**

1. **Caffeine 的 W-TinyLFU 算法相比 LRU 有什么优势？**
   答：LRU 容易被突发流量污染缓存；W-TinyLFU 通过频率统计判断数据是否真正「热」，Window Cache 吸收突发流量，Main Cache 保护长期热点数据。

2. **本地缓存和分布式缓存如何选择？**
   答：数据量小（`<100MB`）、读多写少、无需跨节点共享，用本地缓存；数据量大、需要共享、有一致性要求，用分布式缓存。实际上往往是两者结合：本地缓存扛热点流量，Redis 扛共享数据。

3. **本地缓存如何保证数据一致性？**
   答：三种方式——（1）短过期时间 + 更新时删除缓存；（2）延迟双删处理并发读；（3）更新数据时同步更新缓存，配合版本号 CAS。

4. **如果本地缓存满了怎么办？**
   答：Caffeine 用 W-TinyLFU 决定驱逐谁；Guava Cache 用 LRU。也可以考虑用堆外内存（Ehcache）或者把数据卸载到 Redis。

5. **缓存预热怎么做？**
   答：系统启动时读取热点数据写入缓存；或者提前触发批量加载；关键数据可以在收到第一个请求时同步加载。
