# 热点数据识别与预热

双十一零点，一位头部主播的直播间突然涌入 50 万并发用户。系统开始告警：Redis 集群单节点 QPS 突破 50 万，CPU 占用 100%，RT 从 1ms 飙升到 500ms。运维紧急扩容，但新节点需要时间同步数据，服务已经雪崩——**问题出在哪？你的系统根本没有「识别热点」的能力，等发现问题时，一切都已经来不及了。**

热点数据问题是高并发系统的核心挑战之一。一个 1000 QPS 的商品详情页，如果不做热点处理，可能导致整个 Redis 集群不可用；但只要把这个商品的信息缓存在本地内存里，Redis 的压力瞬间从 1000 QPS 降到 0。这篇文章，我们来彻底搞懂热点数据的识别、预防和处理。

## 什么是热点数据

### 热点的定义

**热点数据**是指在某个时间段内被大量频繁访问的数据。这些数据的访问频率远高于普通数据，可能达到普通数据的数百倍甚至数千倍。

```yaml
# 典型的热点场景
热点商品：iPhone 15 Pro、茅台飞天——发布即售罄
热点用户：大V、明星——粉丝大量访问
热点帖子：热搜榜、热搜第一——全网关注
热点接口：秒杀、抢购——瞬间涌入大量请求
```

### 热点与普通数据的区别

```
普通数据访问分布（长尾分布）：

访问次数
   ▲
   │     ████
   │   ████████
   │  ████████████
   │ ████████████████
   │████████████████████████████████─────────→ key
   0                                              10000
   │←───── 80% 冷数据 ─────→│←─ 20% 热点数据 ─→│
   │←──────── 99% 访问 ─────→│←─ 1% 访问 ──→│

热点数据访问分布：

访问次数
   ▲
   │                                         ████ ← 超级热点
   │                                   ████████████████
   │                             ████████████████████████████
   │                       ████████████████████████████████████
   │███████████████████████████████████████████████████████████████
   └─────────────────────────────────────────────────────────────────→ key
   0                                                                     10000
   │←──────────────────── 99.9% 访问 ─────────────────────→│← 0.1% ─→│
```

**热点的特征**：

1. **访问集中**：1% 的 key 占了 99% 的访问量
2. **时间集中**：热点往往在特定时间段爆发
3. **跨节点集中**：如果不处理，热点会导致单节点压力过大

### 热点的分类

| 类型 | 说明 | 示例 |
|---|---|---|
| 热点 key | 单个 key 访问量巨大 | 热搜第一的帖子 |
| 热点 slot | 某个槽位的 key 访问集中 | 某个店铺的所有商品 |
| 热点接口 | 某个接口访问量激增 | 秒杀下单接口 |
| 热点用户 | 某个用户的数据被频繁访问 | 大V 的个人主页 |
| 热点数据行 | 数据库某行数据被频繁读写 | 库存扣减 |

### 2. 基于日志的识别

通过分析访问日志，可以发现热点数据的规律：

```python
# 日志分析脚本示例
import re
from collections import Counter

def analyze_hotspot(log_file):
    # 提取访问的 key（以 Redis GET 命令为例）
    pattern = re.compile(r'GET\s+(\S+)')
    
    key_counter = Counter()
    for line in open(log_file):
        match = pattern.search(line)
        if match:
            key = match.group(1)
            key_counter[key] += 1
    
    # 找出 TOP 10 热点 key
    for key, count in key_counter.most_common(10):
        print(f"{key}: {count} 次")
    
    # 统计访问分布
    total = sum(key_counter.values())
    top1_percent_keys = key_counter.most_common(len(key_counter) // 100)
    top1_percent_count = sum(count for _, count in top1_percent_keys)
    
    print(f"TOP 1% key 占比: {top1_percent_count / total * 100:.2f}%")

# 输出示例
# product:10001: 500000 次
# product:10002: 450000 次
# product:10003: 420000 次
# ...
# TOP 1% key 占比: 85.32%
```

### 3. 基于采样的识别

对于实时性要求高的场景，可以使用采样分析：

```java
// 基于采样的热点 key 识别
public class HotKeyDetector {
    // 采样率：每 100 次请求采样 1 次
    private static final int SAMPLE_RATE = 100;
    
    // 计数器（滑动窗口）
    private final ConcurrentHashMap<String, AtomicLong> counters = 
        new ConcurrentHashMap<>();
    
    // 滑动窗口大小：1 分钟
    private final long windowMs = 60_000;
    private final ConcurrentLinkedQueue<LocalDateTime> timestamps = 
        new ConcurrentLinkedQueue<>();
    
    public void recordAccess(String key) {
        // 采样
        if (ThreadLocalRandom.current().nextInt(SAMPLE_RATE) != 0) {
            return;
        }
        
        // 更新计数器
        counters.computeIfAbsent(key, k -> new AtomicLong())
            .incrementAndGet();
        
        // 清理过期数据
        cleanExpired();
    }
    
    public List<Map.Entry<String, Long>> getTopHotKeys(int limit) {
        cleanExpired();
        return counters.entrySet().stream()
            .sorted(Map.Entry.<String, Long>comparingByValue().reversed())
            .limit(limit)
            .collect(Collectors.toList());
    }
    
    private void cleanExpired() {
        LocalDateTime cutoff = LocalDateTime.now().minusNanos(windowMs * 1_000_000);
        while (true) {
            LocalDateTime oldest = timestamps.peek();
            if (oldest == null || oldest.isAfter(cutoff)) {
                break;
            }
            timestamps.poll();
            // 对应的 key 计数也会逐渐被淘汰
        }
    }
}
```

### 3. Redis 哨兵 + 自定义脚本

```python
#!/usr/bin/env python3
# 使用 Redis 集群 API 发现热点 key

import redis
from collections import defaultdict

def find_hot_keys(cluster_nodes, threshold=1000):
    hot_keys = defaultdict(int)
    
    for node in cluster_nodes:
        r = redis.Redis(host=node['host'], port=node['port'])
        # 获取所有 key 的访问信息
        # 使用 OBJECT FREQ 命令（需要开启 LFU）
        for key in r.scan_iter('*'):
            freq = r.object('FREQ', key)
            if freq and freq > threshold:
                hot_keys[key] += freq
    
    # 按频率排序
    return sorted(hot_keys.items(), key=lambda x: x[1], reverse=True)

# 使用
cluster_nodes = [
    {'host': '10.0.0.1', 'port': 6379},
    {'host': '10.0.0.2', 'port': 6379},
]
for key, freq in find_hot_keys(cluster_nodes):
    print(f"{key}: {freq} 次/窗口期")
```

```lua
#register.lua
#!js api_version=1.0 name=hotkey_lib

redis.registerFunction('trackHotKey', function(keys, args)
    local key = keys[1]
    local count = redis.call('INCR', '__hotkey:' .. key)
    redis.call('EXPIRE', '__hotkey:' .. key, 60)  -- 1 分钟窗口

end)

redis.registerFunction('getHotKeys', function(keys, args)
    local threshold = tonumber(args[1]) or 100
    local result = {}
    for _, hotkey in ipairs(redis.call('KEYS', '__hotkey:*')) do
        local count = redis.call('GET', hotkey)
        if tonumber(count) >= threshold then
            table.insert(result, {hotkey:sub(9), count})  -- 去掉前缀
        end
    end
    return cjson.encode(result)
end)

-- 使用
```

## 热点数据对系统的危害

### 1. 单节点压力过大

Redis Cluster 虽然把数据分散到 16384 个槽，但如果热点 key 刚好落在同一个槽，这个槽所在的节点就会承受绝大部分压力：

```
场景：热点 key "product:10001" 被 10 万人同时访问

如果 key 落在 Slot 10000（Master 1）：
Master 1 承受 100000 QPS → CPU 100% → RT 暴涨

其他 Master 可能只有 10 QPS → 完全空闲
```

### 2. 缓存击穿

热点 key 过期时，大量请求同时穿透到数据库：

```java
// 场景：热点商品详情页，TTL 设置为 5 分钟
// TTL 过期瞬间：

// ❌ 问题代码
public Product getProduct(Long productId) {
    // 缓存未命中
    Product product = redis.get("product:" + productId);
    if (product == null) {
        // 10 万并发请求同时执行这里
        product = productDAO.findById(productId);  // 数据库被打爆
        redis.set("product:" + productId, product, 5, TimeUnit.MINUTES);
    }
    return product;
}

// 10 万请求 → 10 万次数据库查询 → 数据库宕机
```

### 3. 缓存雪崩

大量 key 同时过期，或者 Redis 集群故障：

```yaml
# 雪崩场景 1：大量 key 同时过期
# 配置了相同的过期时间
product:1 TTL = 5min
product:2 TTL = 5min
product:3 TTL = 5min
...
# 5 分钟后，10 万个 key 同时过期
# 10 万请求 → 10 万次数据库查询

# 雪崩场景 2：Redis 集群故障
# 主节点宕机，从节点还没完成故障转移
# 10 万请求直接打数据库
```

### 4. 数据倾斜

集群中部分节点数据量远大于其他节点：

```bash
# 检查 slot 分布
redis-cli cluster slots

# 输出
0-5460    Master 1 (数据量 100GB)
5461-10922 Master 2 (数据量 100GB)
10923-16383 Master 3 (数据量 5GB)
# Master 3 数据量异常，可能是热点 key 集中在这里
```

## 热点 key 的处理方案

### 1. 拆分 key：读写分离

把一个热点 key 的读操作分散到多个 key：

```java
// 原始方案：一个 key 存储商品信息
// product:10001 = {name: "iPhone 15", price: 9999}
// 问题：所有读请求都打到一个 key

// 改进方案 1：key 分片
// product:10001:shard1
// product:10001:shard2
// product:10001:shard3
// 读请求随机选择一个分片

// 改进方案 2：主从分离（读从写主）
public class HotKeyHandler {
    @Autowired
    private RedisTemplate<String, Product> masterRedis;
    
    @Autowired
    private RedisTemplate<String, Product> slaveRedis;
    
    public Product getProduct(Long productId) {
        // 读从节点
        return slaveRedis.opsForValue().get("product:" + productId);
    }
    
    public void updateProduct(Long productId, Product product) {
        // 写主节点
        masterRedis.opsForValue().set("product:" + productId, product);
    }
}
```

### 2. 本地缓存 + Redis：多级缓存

热点数据优先从本地缓存读取，减少 Redis 压力：

```java
// 多级缓存：本地缓存 + Redis
public class MultiLevelCache {
    // L1: 本地缓存（Caffeine）
    private LoadingCache<Long, Product> localCache = Caffeine.newBuilder()
        .maximumSize(1000)
        .expireAfterWrite(1, TimeUnit.MINUTES)
        .refreshAfterWrite(30, TimeUnit.SECONDS)
        .build(key -> {
            // L1 未命中，从 Redis 加载
            return redisTemplate.opsForValue().get("product:" + key);
        });
    
    // L2: Redis 缓存
    @Autowired
    private RedisTemplate<String, Product> redisTemplate;
    
    public Product getProduct(Long productId) {
        // 1. 先查本地缓存（纳秒级）
        try {
            return localCache.get(productId);
        } catch (ExecutionException e) {
            // 2. 本地缓存失败，查 Redis（毫秒级）
            Product product = redisTemplate.opsForValue()
                .get("product:" + productId);
            if (product != null) {
                // 3. 回填本地缓存
                localCache.put(productId, product);
            }
            return product;
        }
    }
    
    public void updateProduct(Long productId, Product product) {
        // 1. 更新 Redis
        redisTemplate.opsForValue().set("product:" + productId, product);
        // 2. 清除本地缓存（所有实例）
        localCache.invalidate(productId);
        // 3. 广播通知其他实例清除本地缓存
        messageBroker.publish("cache:invalidate", productId);
    }
}
```

### 4. 热点数据永远不过期

对于极端热点数据，可以设置很长的过期时间甚至永不过期：

```java
public class ExtremeHotKeyHandler {
    // 超级热点：使用本地缓存 + 永不过期
    private final Map<Long, Product> extremeHotCache = new ConcurrentHashMap<>();
    
    public Product getExtremeHotProduct(Long productId) {
        Product product = extremeHotCache.get(productId);
        if (product == null) {
            // 首次加载
            product = productDAO.findById(productId);
            extremeHotCache.put(productId, product);
        }
        return product;
    }
    
    // 定时更新（通过消息队列触发）
    @KafkaListener(topics = "product:update")
    public void onProductUpdate(Long productId) {
        // 更新数据库后，删除本地缓存
        extremeHotCache.remove(productId);
        // 下次访问时重新加载
    }
}
```

### 5. 限流 + 熔断

在应用层对热点 key 的访问做限流：

```java
// 热点 key 限流
public class HotKeyRateLimiter {
    // 使用令牌桶限流
    private final Map<String, RateLimiter> limiters = new ConcurrentHashMap<>();
    
    public void accessHotKey(String key) {
        RateLimiter limiter = limiters.computeIfAbsent(key, 
            k -> RateLimiter.create(10000));  // 每秒 10000 次
        
        if (!limiter.tryAcquire()) {
            // 限流：返回降级响应或使用旧数据
            throw new RateLimitException("hot key rate limited");
        }
    }
}

// Sentinel 热点参数限流
// 配置热点 key 的限流规则
// 当某个 key 的 QPS 超过阈值时，触发限流
```

:::tip 热点处理策略的选择
- **普通热点**：本地缓存 + Redis，适合 80% 的场景
- **超级热点**：本地缓存 + 永不过期 + 异步更新，适合头部商品
- **极端热点**：多副本分散 + 限流熔断，适合大促场景
:::

## 热点数据的预热策略

预热是指在流量高峰到来之前，提前把热点数据加载到缓存中，避免冷启动时的大量回源。

### 1. 启动时预热

系统启动时自动加载热点数据：

```java
@SpringBootApplication
public class Application {
    @Autowired
    private LoadingCache<Long, Product> localCache;
    
    @Autowired
    private RedisTemplate<String, Product> redisTemplate;
    
    @Autowired
    private ProductDAO productDAO;
    
    @PostConstruct
    public void cacheWarmup() {
        // 1. 预加载 TOP 1000 热点商品到 Redis
        List<Product> hotProducts = productDAO.findHotProducts(1000);
        for (Product product : hotProducts) {
            redisTemplate.opsForValue().set(
                "product:" + product.getId(), 
                product,
                24, TimeUnit.HOURS
            );
        }
        
        // 2. 预加载 TOP 100 热点商品到本地缓存
        for (Product product : hotProducts.subList(0, 100)) {
            localCache.put(product.getId(), product);
        }
        
        log.info("Cache warmup completed: {} products in Redis, {} in local", 
            hotProducts.size(), 100);
    }
}
```

### 2. 定时预热

定时任务周期性刷新热点数据：

```java
// 定时预热任务
@Scheduled(cron = "0 */5 * * * ?")  // 每 5 分钟
public void scheduledWarmup() {
    log.info("Starting scheduled cache warmup...");
    
    // 1. 获取最新热点数据
    List<Long> hotProductIds = productService.getHotProductIds(100);
    
    // 2. 批量加载到 Redis
    Map<String, Product> productMap = productDAO.findByIds(hotProductIds)
        .stream()
        .collect(Collectors.toMap(p -> "product:" + p.getId(), p -> p));
    
    redisTemplate.opsForValue().multiSet(productMap);
    
    // 3. 更新本地缓存
    hotProductIds.forEach(id -> {
        Product product = productMap.get("product:" + id);
        if (product != null) {
            localCache.put(id, product);
        }
    });
    
    log.info("Scheduled warmup completed: {} products refreshed", 
        productMap.size());
}
```

基于业务事件触发预热：

```java
// 商品下架/上架事件触发预热
@KafkaListener(topics = "product:onshell")

}
```

## Redis Hash Tag 机制在热点处理中的应用

### Hash Tag 的原理

Redis Cluster 使用 CRC16(key) % 16384 计算 slot，但如果有 hash tag，Redis 会用 hash tag 计算：

```
# 没有 hash tag
key = "product:10001"
slot = CRC16("product:10001") % 16384 = 15421

# 有 hash tag
key = "{product}:10001"
slot = CRC16("product") % 16384 = 1234

# 所有 {product}:* 的 key 都会落在 slot 1234
key = "{product}:10002"  → slot 1234
key = "{product}:10003"  → slot 1234
```

### Hash Tag 的应用场景

**1. 批量操作**

```java
// 用户的 Session 聚合存储
// session:{userId}:token
// session:{userId}:profile
// session:{userId}:permissions
// 使用 hash tag，所有操作落在同一个 slot

redisTemplate.opsForCluster().mget(
    "{user10001}:token",
    "{user10001}:profile", 
    "{user10001}:permissions"
);  // 一次请求完成，不需要跨 slot
```

**2. 分布式锁**

```java
// 分布式锁使用 hash tag 确保同一个锁的所有操作在同一个 slot
// lock:{resource}:holder
// lock:{resource}:timestamp

public boolean acquireLock(String resource, String holder, long expireMs) {
    String key = "{lock:" + resource + "}:owner";
    return redisTemplate.opsForValue().setIfAbsent(key, holder, expireMs, TimeUnit.MILLISECONDS);
}

public boolean releaseLock(String resource, String holder) {
    String key = "{lock:" + resource + "}:owner";
    String currentHolder = redisTemplate.opsForValue().get(key);
    if (holder.equals(currentHolder)) {
        redisTemplate.delete(key);
        return true;
    }
    return false;
}
```

### Hash Tag 导致的热点问题

Hash tag 虽然方便批量操作，但也会导致数据倾斜：

```
场景：所有商品的评论都使用 {productId}:comments

product:10001:comments → slot 1234 (10000 条评论)
product:10002:comments → slot 1234 (8000 条评论)
product:10003:comments → slot 1234 (5000 条评论)
...
product:10050:comments → slot 1234 (10000 条评论)

结果：slot 1234 承载了 50 个商品的评论，约 50 万条数据
其他 slot 只有几千条数据
```

**解决方案：使用随机 tag 分散热点**

```java
// 原始方案：所有评论用 productId 作为 hash tag
// comment:{productId}:{commentId}

// 改进方案：使用随机 tag 分散到不同 slot
// comment:{productId}:random:{commentId}
// 但这样就无法批量获取一个商品的所有评论了

// 更好的方案：分层设计
// 1. 热点评论分散存储
// comment:{commentId} → 使用 commentId 计算 slot
// 2. 索引信息聚合存储
// comment:index:{productId} → SET 存储评论 ID 列表
```

### L1 Caffeine + L2 Redis 架构

```
用户请求

    ▼

│  命中率：60-80%（热点数据）             │
│  容量：100MB（可配置）                 │

                  │ L1 Miss
                  ▼
```

### 完整实现

```java
@Component
public class MultiLevelCacheManager {
    
    // L1: Caffeine（热点数据）
    private final LoadingCache<Long, Product> l1Cache = Caffeine.newBuilder()
        .maximumSize(10_000)
        .expireAfterWrite(30, TimeUnit.SECONDS)
        .refreshAfterWrite(10, TimeUnit.SECONDS)
        .recordStats()
        .build(key -> loadFromRedis(key));
    
    // L2: Redis
    @Autowired
    private RedisTemplate<String, Product> redisTemplate;
    
    // L3: MySQL
    @Autowired
    private ProductDAO productDAO;
    
    // 降级开关
    @Value("${cache.l1.enabled:true}")
    private boolean l1Enabled;
    
    public Product getProduct(Long productId) {
        String redisKey = "product:" + productId;
        
        // 1. 查 L1
        if (l1Enabled) {
            try {
                Product l1Result = l1Cache.get(productId);
                if (l1Result != null) {
                    return l1Result;
                }
            } catch (ExecutionException e) {
                // L1 加载失败，继续 L2
                log.warn("L1 cache load failed for key: {}", productId);
            }
        }
        
        // 2. 查 L2
        Product product = redisTemplate.opsForValue().get(redisKey);
        if (product != null) {
            // 回填 L1
            l1Cache.put(productId, product);
            return product;
        }
        
        // 3. 查 L3
        product = productDAO.findById(productId);
        if (product != null) {
            // 回填 L2
            redisTemplate.opsForValue().set(redisKey, product, 1, TimeUnit.HOURS);
            // 回填 L1
            l1Cache.put(productId, product);
        }
        
        return product;
    }
    
    private Product loadFromRedis(Long productId) {
        String redisKey = "product:" + productId;
        return redisTemplate.opsForValue().get(redisKey);
    }
    
    // 缓存更新：写穿透
    @Transactional
    public void updateProduct(Long productId, Product product) {
        // 1. 更新 L3
        productDAO.update(productId, product);
        
        // 2. 更新 L2
        String redisKey = "product:" + productId;
        redisTemplate.opsForValue().set(redisKey, product, 1, TimeUnit.HOURS);
        
        // 3. 删除 L1（而不是更新，避免所有实例同时写入）
        l1Cache.invalidate(productId);
    }
    
    // 缓存监控
    public CacheStats getL1Stats() {
        return l1Cache.stats();
    }
}
```

### 热点数据的特殊处理

```java
// 热点数据识别与特殊处理
@Component
public class HotDataProcessor {
    
    // 热点数据阈值：每分钟访问超过 10000 次
    private static final int HOT_THRESHOLD = 10000;
    
    // 热点数据 L1 缓存：更长的 TTL
    private final LoadingCache<Long, Product> hotDataCache = Caffeine.newBuilder()
        .maximumSize(100)  // 只缓存 TOP 100 热点
        .expireAfterWrite(5, TimeUnit.MINUTES)
        .refreshAfterWrite(1, TimeUnit.MINUTES)
        .recordStats()
        .build(key -> loadProduct(key));
    
    @Autowired
    private HotKeyDetector hotKeyDetector;
    
    @Autowired
    private MultiLevelCacheManager cacheManager;
    
    public Product getProduct(Long productId) {
        // 1. 检查是否是热点数据
        long accessCount = hotKeyDetector.getAccessCount(productId);
        
        if (accessCount > HOT_THRESHOLD) {
            // 热点数据：使用专用缓存
            return hotDataCache.get(productId);
        }
        
        // 2. 普通数据：使用普通多级缓存
        return cacheManager.getProduct(productId);
    }
    
    // 热点数据预热
    public void warmupHotData(List<Long> hotProductIds) {
        for (Long productId : hotProductIds) {
            hotDataCache.put(productId, cacheManager.getProduct(productId));
        }
    }
}
```

```java
// 热点监控埋点
@Component
public class HotKeyMetrics {
    
    @Autowired
    private MeterRegistry registry;
    
    private final Map<String, Counter> hotKeyCounters = new ConcurrentHashMap<>();
    
    public void recordHotKeyAccess(String key) {
        String normalizedKey = normalizeKey(key);
        Counter counter = hotKeyCounters.computeIfAbsent(normalizedKey,
            k -> Counter.builder("hot_key_access_total")
                .tag("key_prefix", extractKeyPrefix(k))
                .tag("hot_level", determineHotLevel(k))
                .register(registry));
        counter.increment();
    }
    
    private String normalizeKey(String key) {
        // 移除动态参数，保留 key 模式
        // product:10001 → product:{id}
        return key.replaceAll("\\d+", "{id}");
    }
    
    private String extractKeyPrefix(String key) {
        return key.split(":")[0];
    }
    
    private String determineHotLevel(String key) {
        // 根据访问频率确定热点级别
        return "L1";  // 简化版
    }
}
```

## 读写分离在高并发读场景的应用

### 主从架构下的读写分离

```java
@Configuration
public class RedisReadWriteConfig {
    
    @Bean("masterRedisTemplate")
    public RedisTemplate<String, Object> masterRedisTemplate() {
        RedisTemplate<String, Object> template = new RedisTemplate<>();
        template.setConnectionFactory(masterConnectionFactory);
        template.execute(new RedisCallback<Object>() {
            @Override
            public Object doInRedis(RedisConnection connection) {
                // 强制写主节点
                connection.setReadOnly(false);
                return null;
            }
        });
        return template;
    }
    
    @Bean("slaveRedisTemplate")
    public RedisTemplate<String, Object> slaveRedisTemplate() {
        RedisTemplate<String, Object> template = new RedisTemplate<>();
        template.setConnectionFactory(slaveConnectionFactory);
        template.execute(new RedisCallback<Object>() {
            @Override
            public Object doInRedis(RedisConnection connection) {
                // 允许读从节点
                connection.setReadOnly(true);
                return null;
            }
        });
        return template;
    }
}

// 读写分离服务
@Service
public class ReadWrite分离Service {
    
    @Autowired
    @Qualifier("masterRedisTemplate")
    private RedisTemplate<String, Object> masterTemplate;
    
    @Autowired
    @Qualifier("slaveRedisTemplate")
    private RedisTemplate<String, Object> slaveTemplate;
    
    // 写操作：主节点
    public void write(String key, Object value) {
        masterTemplate.opsForValue().set(key, value);
    }
    
    // 读操作：从节点
    public Object read(String key) {
        return slaveTemplate.opsForValue().get(key);
    }
    
    // 热点数据读：从从节点 + 本地缓存
    public Object readHot(String key) {
        // 先查本地缓存
        Product product = localCache.getIfPresent(key);
        if (product != null) {
            return product;
        }
        
        // 查从节点
        product = (Product) slaveTemplate.opsForValue().get(key);
        if (product != null) {
            localCache.put(key, product);
        }
        return product;
    }
}
```

### Redis Cluster 的读写分离

```java
// JedisCluster 的读写分离
public class RedisClusterReadWrite {
    
    private JedisCluster cluster;
    
    public RedisClusterReadWrite(JedisCluster cluster) {
        this.cluster = cluster;
    }
    
    // 写操作
    public void write(String key, String value) {
        // 写操作自动路由到主节点
        cluster.set(key, value);
    }
    
    // 读操作：优先从从节点读
    public String read(String key) {
        // 1. 尝试从从节点读
        JedisSlotBasedConnectionHandler handler = 
            (JedisSlotBasedConnectionHandler) ReflectionUtils.getFieldValue(
                cluster, "connectionHandler");
        
        int slot = JedisClusterCRC16.getCRC16(key);
        JedisPool pool = handler.getSlotPool(slot);
        
        // 从池中获取连接（默认是从节点）
        try (Jedis jedis = pool.getResource()) {
            return jedis.get(key);
        } catch (JedisException e) {
            // 从节点失败，降级到主节点
            log.warn("Slave read failed, fallback to master", e);
            return cluster.get(key);
        }
    }
}
```

:::warning 读写分离的数据一致性
从节点可能有延迟，读到的数据可能是旧数据。对于需要强一致性的场景（如库存扣减），必须读主节点。对于可以容忍短暂不一致的场景（如商品详情页），可以读从节点。
:::

## 总结

热点数据处理是高并发系统的核心能力，需要从识别、预防、处理三个维度综合应对：

**识别**：

- 监控：QPS、RT、CPU、连接数
- 日志分析：TOP N 热点 key
- 采样统计：滑动窗口频率统计
- Redis 内置工具：--hotkeys、OBJECT FREQ

**预防**：

- 多级缓存：本地缓存扛热点流量
- 预热机制：提前加载热点数据
- 随机分散：避免数据倾斜

**处理**：

- 拆分 key：分散单 key 压力
- 限流熔断：保护下游服务
- 读写分离：读从写主

**核心原则**：**热点数据永远不要让它打穿到数据库**。

## 面试追问预判

**面试官可能会问：**

1. **如何发现系统中的热点数据？**
   答：从三个层面监控——（1）Redis 层：QPS、CPU、连接数、慢查询日志；（2）应用层：方法调用频率、接口响应时间；（3）数据库层：TOP SQL、热点行访问。推荐使用 APM 工具（如 SkyWalking）+ Redis 监控（如 RedisInsight）+ 日志分析的综合方案。

2. **热点数据导致 Redis 单节点压力过大怎么处理？**
   答：三种方式——（1）本地缓存 + Redis 多级缓存，减少 Redis 访问；（2）热点 key 分散到多个 key，读请求轮询分散；（3）热点 key 多副本（主从都存），读从写主。三种方式可以组合使用。

3. **缓存击穿和缓存雪崩的区别是什么？**
   答：缓存击穿是单个热点 key 过期导致大量请求穿透；缓存雪崩是大量 key 同时过期或 Redis 集群故障导致所有请求穿透。共同点是都会打爆数据库，区别是影响范围不同。

4. **Redis Cluster 的 slot 倾斜怎么解决？**
   答：检查是否使用了 hash tag 导致大量 key 落在同一个 slot；可以使用 `redis-cli cluster rebalance` 重新分配 slot；或者使用随机 tag 分散热点 key 到不同 slot。

5. **预热数据时如何避免回源风暴？**
   答：使用分布式锁控制只有一个实例执行预热；或者使用消息队列协调预热顺序；或者使用「异步加载 + 返回旧值」模式，让部分请求使用旧缓存，部分请求触发加载。
