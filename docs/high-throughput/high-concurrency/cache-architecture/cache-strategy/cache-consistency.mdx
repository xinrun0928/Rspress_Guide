# 缓存一致性：延时双删 + 订阅 binlog 方案

import { Badge } from '@rspress/core/theme';

> [!WARNING]
> 前置知识：本篇文章涉及 Redis、MySQL、分布式系统等知识，建议先掌握 Redis 基础数据类型与 MySQL 事务机制。

凌晨两点，你被报警电话叫醒——线上出现了一批「超卖」订单。用户下单时显示有库存，扣款也成功了，但发货时却发现库存已经没了。这种诡异的数据不一致问题，在高并发系统中屡见不鲜。

问题的根源往往不在数据库，而在**缓存层**。

当你的系统引入缓存来扛住高并发读流量时，数据就分散在了两个地方：数据库和 Redis。数据库是「真相之源」，缓存只是「加速器」。一旦两者不同步，轻则出现超卖、库存不准，重则导致资损。

这一讲，我们来彻底解决缓存与数据库的一致性问题。

## 一、为什么缓存一致性是难题

### 1.1 CAP 定理在缓存场景的体现

分布式系统中有一个著名定理——CAP 定理：一个分布式系统最多只能同时满足 Consistency（一致性）、Availability（可用性）、Partition tolerance（分区容错性）中的两个。

在缓存场景下，这个定理有了新的含义：

- **数据库**是满足 CP 的——写入后一定能读到最新数据
- **缓存**天生是满足 AP 的——优先保证可用，读取可能返回过期数据
- 当缓存和数据库组合使用时，**无法同时保证强一致性和高可用**

这意味着，只要系统中存在缓存层，数据一致性问题就无法完全消除，我们只能选择：

| 策略 | 一致性等级 | 适用场景 |
|---|---|---|
| 强一致性 | 每次读取都是最新 | 金融交易、库存扣减 |
| 最终一致性 | 允许短暂不一致 | 新闻推荐、商品展示 |

大多数业务场景下，最终一致性已经足够。但像「超卖」这类涉及金钱和库存的场景，必须尽可能缩短不一致窗口。

### 1.2 缓存与数据库的双写困境

缓存和数据库的数据同步，本质上是解决「谁先谁后」的问题。

假设有一个商品表 `products`，包含字段 `id`、`name`、`stock`（库存）。现在有两个并发请求同时修改库存：

- 请求 A：扣减库存 1 件
- 请求 B：扣减库存 2 件

如果不使用缓存，数据库本身有事务和锁保护，不会出问题。但引入缓存后，情况变得复杂：

```text
时序1：请求A读取库存100，缓存中也有100
时序2：请求A扣减库存，写入数据库99
时序3：请求B读取缓存（还是100），扣减2件
时序4：请求B写入缓存98
时序5：请求A写入缓存99（覆盖了B的结果）
```

最终：数据库 99，缓存 99，但实际应该是 97。

这就是经典的**双写并发问题**——两个写操作交叉执行，导致后写入的数据覆盖了先写入的数据。

## 二、四种缓存与 DB 的交互模式

缓存与数据库的交互有四种经典模式：

### 2.1 Cache Aside（旁路缓存）

这是最常用的模式，也是大多数系统默认的选择。

**读操作**：先查缓存，命中则返回；未命中则查数据库，再写入缓存。

```java
public Product getProduct(Long id) {
    // 1. 先查缓存
    Product cached = redis.get("product:" + id);
    if (cached != null) {
        return cached;
    }
    
    // 2. 缓存未命中，查数据库
    Product product = database.findById(id);
    
    // 3. 写入缓存
    redis.set("product:" + id, product, 1, TimeUnit.HOURS);
    
    return product;
}
```

**写操作**：先更新数据库，再删除缓存（而非更新缓存）。

```java
public void updateProduct(Product product) {
    // 1. 先更新数据库
    database.update(product);
    
    // 2. 删除缓存（不是更新）
    redis.del("product:" + product.getId());
}
```

为什么是「删除」而不是「更新」？这涉及到缓存更新的语义问题——

假设更新数据库成功，但更新缓存失败，此时数据库是新数据，缓存是旧数据，下次读取会得到错误数据。而如果是删除缓存，即使删除失败，最多导致一次缓存穿透（回源查数据库），数据本身是正确的。

### 2.2 Read Through（读穿透）

在 Cache Aside 模式中，应用代码需要同时和缓存、数据库打交道。Read Through 模式把这层复杂性封装到了缓存层。

应用只和缓存交互，缓存内部自动处理「未命中时查数据库」的动作：

```text
应用 → 缓存（未命中） → 缓存自动查数据库 → 返回结果
```

Redis 本身不直接支持 Read Through，但可以通过封装实现：

```java
public Product getProduct(Long id) {
    // 应用层不感知缓存实现细节
    return cacheService.getOrLoad("product:" + id, 
        () -> database.findById(id));
}
```

### 2.3 Write Through（写穿透）

写操作同时更新缓存和数据库，缓存层保证两个写入的原子性。

```text
应用 → 缓存 → 缓存同步更新数据库 → 返回成功
```

这种模式保证了强一致性，但牺牲了写入性能。Redis 的主从同步也可以看作一种 Write Through——主库写入后，同步复制到从库。

### 2.4 Write Behind（写回）

写操作只更新缓存，立即返回成功；缓存异步批量写入数据库。

```text
应用 → 缓存（标记脏） → 返回成功
        ↓
   异步批量写入数据库
```

这种模式性能最高，但风险也最大——如果缓存节点宕机，脏数据会丢失。OS 的页面缓存就采用这种策略。

## 三、Cache Aside 的数据不一致场景分析

尽管 Cache Aside 是最推荐的模式，但它并非银弹，仍然存在数据不一致的风险。

### 3.1 经典不一致场景：缓存过期 + 并发更新

这个场景在电商系统中非常常见：

```text
T1：线程A读取商品库存，缓存未命中，查数据库得到 stock=100
T2：线程B更新商品库存 stock=99，更新数据库成功
T3：线程B删除缓存
T4：线程A将 stock=100 写入缓存（基于T1读取的数据）
T5：后续请求从缓存读到 stock=100，但数据库已经是 99
```

结果：数据库 99，缓存 100。**不一致窗口为 T1 到 T4 的时间差**，可能持续几百毫秒到几秒。

### 3.2 另一个风险：删除缓存失败

如果更新数据库后，删除缓存的操作失败了怎么办？

```java
public void updateProduct(Product product) {
    database.update(product);      // 成功
    boolean deleted = redis.del("product:" + product.getId());  // 失败
    if (!deleted) {
        // 这里怎么处理？
    }
}
```

此时数据库是新数据，缓存是旧数据。如果没有补偿机制，这个不一致会一直存在。

### 3.3 为什么 Cache Aside 优先删除而非更新

你可能会问：为什么不能同时更新缓存和数据库？

考虑一个场景：用户快速多次修改同一商品属性（编辑-保存-编辑-保存）。如果每次都同时写缓存和数据库：

```text
T1：请求A更新数据库 name="商品A"
T2：请求B更新数据库 name="商品B"
T3：请求A写缓存 name="商品A"
T4：请求B写缓存 name="商品B"
```

这样看起来没问题。但如果加上网络延迟：

```text
T1：请求A更新数据库 name="商品A"
T2：请求A写缓存成功 name="商品A"
T3：请求B更新数据库 name="商品B"
T4：网络抖动，请求B写缓存失败
T5：请求C读取，得到缓存中的"商品A"，但数据库已经是"商品B"
```

如果采用删除策略：

```text
T1：请求A更新数据库 name="商品A"
T2：请求A删除缓存
T3：请求B更新数据库 name="商品B"
T4：请求B删除缓存
T5：请求C读取，缓存未命中，查数据库得到"商品B"
```

无论并发情况如何，删除操作的结果是确定的——要么成功，要么失败。而更新操作的结果还取决于「谁先写」。删除操作天然具有幂等性。

## 四、延时双删策略

针对 Cache Aside 中「缓存过期 + 并发更新」导致的不一致问题，一个简单有效的解决方案是**延时双删**。

### 4.1 核心思路

```
1. 删除缓存
2. 更新数据库
3. 延时一段时间（如500ms）
4. 再次删除缓存
```

为什么要先删后更？考虑这个顺序：

```text
如果先更新再删除：
T1：线程A读取，缓存未命中，查数据库得到 stock=100
T2：线程B更新数据库 stock=99
T3：线程B删除缓存
T4：线程A写缓存 stock=100（基于T1的旧数据）  ← 问题
```

如果先删后更：

```text
T1：线程A删除缓存
T2：线程B读取，缓存未命中，查数据库（此时B能查到最新值）
T3：线程B写缓存（最新值）
T4：线程A更新数据库
T5：线程A再次删除缓存
```

通过「先删缓存」强制后续请求回源读取，通过「延时」等待正在进行的读操作完成，再通过「二次删除」清理可能写入的旧值。

### 4.2 代码实现

```java
public void updateProduct(Product product) {
    // 1. 删除缓存
    redis.del("product:" + product.getId());
    
    // 2. 更新数据库
    database.update(product);
    
    // 3. 延时后再次删除
   CompletableFuture.delayedExecutor(500, TimeUnit.MILLISECONDS)
        .execute(() -> redis.del("product:" + product.getId()));
}
```

### 4.3 延时双删的问题

延时双删看似简单，但有两个致命缺陷：

**问题一：延时时间难以确定**

500ms 够吗？如果数据库是主从架构，主库写入后数据同步到从库需要时间。如果读操作路由到从库，需要等待主从同步完成。跨机房场景下，这个延迟可能达到几十秒。

延时太短：无法覆盖并发读操作的窗口
延时太长：严重影响业务响应时间

**问题二：异步删除可能失败**

延时删除是在单独的线程中执行的，如果这个线程崩溃或被拒绝，删除操作永远不会执行。

```java
CompletableFuture.delayedExecutor(500, TimeUnit.MILLISECONDS)
    .execute(() -> redis.del("product:" + product.getId()));
// 如果 JVM 在 500ms 内重启，这次删除就丢了
```

**问题三：无法保证跨服务的一致性**

如果系统是微服务架构，多个服务都可能修改同一份数据，单服务的延时双删无法覆盖所有写入方。

## 五、订阅 MySQL binlog 实现缓存一致性

延时双删的局限性促使我们寻找更可靠的方案——**订阅 MySQL binlog**。

### 5.1 binlog 是什么

MySQL 的 binlog（Binary Log）是记录所有数据库变更的文件。每一行数据的增删改操作都会以「事件」的形式写入 binlog。

binlog 有三种格式：

| 格式 | 说明 | 优点 | 缺点 |
|---|---|---|---|
| STATEMENT | 记录 SQL 语句 | 日志量小 | 函数、存储过程结果不确定 |
| ROW | 记录修改前后的行数据 | 准确 | 日志量大 |
| MIXED | 混合模式 | 平衡 | 复杂 |

生产环境推荐使用 **ROW** 格式，因为它是确定性的——无论什么 SQL，执行结果都是可复现的。

### 5.2 Canal 的工作原理

Canal <Badge text="阿里巴巴开源" type="tip" /> 是阿里巴巴开源的 MySQL binlog 增量订阅组件。它的核心原理是**伪装成一个 MySQL 从库（slave），模拟 MySQL 主从复制协议**。

```text
MySQL Master → Canal → 应用（解析 binlog → 更新缓存）
                 ↑
            伪装成 Slave
```

Canal 的工作流程：

1. **伪装从库**：Canal 伪装成 MySQL 的一个从库，向主库发送 `COM_BINLOG_DUMP` 命令
2. **接收 binlog**：主库收到请求后，开始推送 binlog 事件给 Canal
3. **解析事件**：Canal 解析 binlog 事件，转换为 JSON 格式
4. **投递消息**：将解析后的事件投递到消息队列（如 Kafka、RocketMQ）
5. **消费处理**：下游应用消费消息，根据事件类型更新缓存

```java title="Canal 客户端示例"
public class CanalClientExample {
    public void subscribe() {
        CanalConnector connector = Canal.newInstance()
            .withHost("127.0.0.1")
            .withPort(11111)
            .withUser("canal")
            .withPassword("canal")
            .connect();
        
        while (true) {
            Message message = connector.getWithoutAck(batchSize);
            List<Entry> entries = message.getEntries();
            
            for (Entry entry : entries) {
                if (entry.getEntryType() == EntryType.ROWDATA) {
                    // 解析行数据变更
                    RowChange rowChange = RowChange.parseFrom(entry.getStoreValue());
                    for (RowData rowData : rowChange.getRowDatasList()) {
                        processRowChange(entry, rowData, rowChange.getEventType());
                    }
                }
            }
            
            connector.ack(message.getId());
        }
    }
    
    private void processRowChange(Entry entry, RowData rowData, EventType eventType) {
        String tableName = entry.getHeader().getTableName();
        String primaryKey = getPrimaryKey(rowData.getBeforeColumnsList());
        
        switch (eventType) {
            case UPDATE:
                // 更新操作：删除旧缓存
                redis.del(tableName + ":" + primaryKey);
                break;
            case DELETE:
                // 删除操作：删除缓存
                redis.del(tableName + ":" + primaryKey);
                break;
            case INSERT:
                // 插入操作：写入缓存（可选）
                if (shouldCache(rowData)) {
                    Product product = buildProduct(rowData.getAfterColumnsList());
                    redis.set(tableName + ":" + primaryKey, product);
                }
                break;
        }
    }
}
```

### 5.3 Canal vs Maxwell vs Debezium

除了 Canal，业界还有两个主流的 binlog 订阅方案：

| 组件 | 开发语言 | 输出格式 | 特点 |
|---|---|---|---|
| Canal | Java | JSON/Protobuf | 功能全面，阿里有大规模验证 |
| Maxwell | Java | JSON | 轻量级，配置简单 |
| Debezium | Java | JSON/Avro | 支持多数据库，生态丰富 |

对于 Redis 缓存一致性场景，Canal 和 Maxwell 都能满足需求。Canal 在阿里巴巴内部有多年大规模使用经验，稳定可靠；Maxwell 配置更简单，适合中小型项目。

### 5.4 订阅 binlog 的完整流程

结合 Canal，我们可以构建一套完整的缓存一致性方案：

```text
┌─────────┐     binlog      ┌───────┐     解析      ┌────────────┐
│  MySQL  │ ──────────────> │ Canal │ ───────────> │   Kafka    │
└─────────┘                 └───────┘              └────────────┘
                                                          │
                                                          ▼
                                                   ┌────────────┐
                                                   │  Consumer  │
                                                   └────────────┘
                                                          │
                               ┌──────────────────────────┼──────────────────────────┐
                               ▼                          ▼                          ▼
                        ┌───────────┐            ┌───────────┐            ┌───────────┐
                        │ 删除缓存   │            │ 重建缓存   │            │ 记录日志   │
                        └───────────┘            └───────────┘            └───────────┘
```

1. **数据库变更**：业务代码更新 MySQL 数据
2. **binlog 记录**：MySQL 将变更写入 binlog
3. **Canal 推送**：Canal 伪装从库接收 binlog 事件
4. **消息投递**：Canal 将事件投递到 Kafka
5. **消费者处理**：缓存服务消费消息，更新 Redis

### 5.5 Cache Aside + 订阅 binlog 的组合方案

订阅 binlog 解决了「谁来删除缓存」的问题，但完整的方案还需要结合 Cache Aside：

| 场景 | 处理方式 |
|---|---|
| 读操作 | Cache Aside：先读缓存，未命中则查库并写缓存 |
| 写操作（主动） | 1. 更新数据库 2. 发送消息触发缓存更新 |
| 写操作（binlog 触发） | 监听 binlog，删除/更新对应缓存 |

这种组合的优势：

- **读路径简单高效**：命中缓存直接返回，不依赖任何外部服务
- **写路径可靠**：通过 binlog 异步更新缓存，保证最终一致性
- **可扩展**：缓存更新逻辑可以灵活扩展，支持多种缓存策略

```java title="完整的缓存一致性方案"
@Service
public class ProductCacheService {
    
    @Autowired
    private ProductDao productDao;
    
    @Autowired
    private RedisTemplate<String, Product> redisTemplate;
    
    @Autowired
    private KafkaTemplate<String, String> kafkaTemplate;
    
    // ==================== 读操作（Cache Aside） ====================
    public Product getProduct(Long id) {
        String key = "product:" + id;
        
        // 1. 先查缓存
        Product cached = redisTemplate.opsForValue().get(key);
        if (cached != null) {
            return cached;
        }
        
        // 2. 缓存未命中，查数据库
        Product product = productDao.findById(id);
        if (product == null) {
            return null;
        }
        
        // 3. 写缓存（设置过期时间，防止脏数据累积）
        redisTemplate.opsForValue().set(key, product, 1, TimeUnit.HOURS);
        
        return product;
    }
    
    // ==================== 写操作（同步 + 消息） ====================
    public void updateProduct(Product product) {
        // 1. 更新数据库
        productDao.update(product);
        
        // 2. 发送消息（触发 binlog 消费者更新缓存）
        kafkaTemplate.send("product-update", 
            JSON.toJSONString(new ProductUpdateEvent(product.getId())));
    }
    
    // ==================== binlog 消费者 ====================
    @KafkaListener(topics = "product-update")
    public void handleProductUpdate(ConsumerRecord<String, String> record) {
        ProductUpdateEvent event = JSON.parseObject(record.value(), ProductUpdateEvent.class);
        
        String key = "product:" + event.getProductId();
        
        // 删除缓存（让下次读取触发 Cache Aside 重建）
        redisTemplate.delete(key);
        
        // 可选：主动重建缓存（减少一次缓存穿透）
        Product product = productDao.findById(event.getProductId());
        if (product != null) {
            redisTemplate.opsForValue().set(key, product, 1, TimeUnit.HOURS);
        }
    }
}
```

## 六、缓存的版本化与乐观锁

在高并发场景下，即使有了延时双删和 binlog 订阅，仍然可能出现并发更新导致的覆盖问题。这时候需要引入**缓存版本化**机制。

### 6.1 版本化的基本思路

给缓存数据加上一个版本号字段，每次更新都检查版本：

```java
// 缓存数据结构
public class VersionedCache<T> {
    private T data;
    private long version;
    private long lastModified;
}
```

读取时获取版本号，写入时检查版本号：

```java
public boolean updateWithVersion(String key, Product newProduct, long expectedVersion) {
    VersionedCache<Product> cached = redisTemplate.opsForValue().get(key);
    
    if (cached == null) {
        return false;
    }
    
    if (cached.getVersion() != expectedVersion) {
        // 版本冲突，抛出异常或返回失败
        throw new VersionConflictException("Cache version conflict");
    }
    
    // 版本匹配，更新数据并递增版本号
    newProduct.setVersion(expectedVersion + 1);
    redisTemplate.opsForValue().set(key, new VersionedCache<>(newProduct));
    
    return true;
}
```

### 6.2 乐观锁 vs 悲观锁

| 类型 | 特点 | 适用场景 |
|---|---|---|
| 乐观锁 | 假设冲突概率低，更新时检查版本 | 读多写少，低竞争 |
| 悲观锁 | 假设冲突概率高，操作前先加锁 | 写多读少，高竞争 |

缓存层天然适合乐观锁——因为缓存本身就是高性能的，加锁反而会成为瓶颈。使用乐观锁可以在保证数据一致性的同时，不损失缓存的性能优势。

---

## 七、数据一致性等级与选型建议

### 7.1 一致性等级对比

| 一致性等级 | 实现难度 | 性能影响 | 数据风险 | 适用场景 |
|---|---|---|---|---|
| 强一致性 | 高 | 大 | 低 | 金融、库存扣减 |
| 最终一致性（秒级） | 中 | 中 | 中 | 电商商品信息 |
| 最终一致性（分钟级） | 低 | 小 | 大 | 用户画像、日志 |

### 7.2 实际选型建议

**金融支付场景**：不建议使用缓存，数据库直接作为唯一数据源。

**库存扣减场景**：使用分布式锁 + 数据库唯一索引 + 最终一致性。

```java
public boolean deductStock(Long productId, Integer quantity) {
    // 1. 分布式锁
    String lockKey = "stock:lock:" + productId;
    String lockValue = UUID.randomUUID().toString();
    boolean locked = redis.set(lockKey, lockValue, 30, TimeUnit.SECONDS, "NX");
    
    if (!locked) {
        return false;
    }
    
    try {
        // 2. 数据库更新（乐观锁）
        int affected = jdbcTemplate.update(
            "UPDATE products SET stock = stock - ? WHERE id = ? AND stock >= ?",
            quantity, productId, quantity);
        
        if (affected > 0) {
            // 3. 删除缓存
            redisTemplate.delete("product:" + productId);
            return true;
        }
        
        return false;
    } finally {
        // 4. 释放锁
        redisTemplate.delete(lockKey);
    }
}
```

**商品信息展示场景**：使用 Cache Aside + 订阅 binlog，容忍分钟级不一致。

**用户画像、推荐系统**：允许较大不一致窗口，使用异步写入 + 批量更新。

---

## 八、总结

缓存一致性是分布式系统中的经典难题，没有银弹，只有权衡。

| 方案 | 一致性强度 | 实现复杂度 | 适用场景 |
|---|---|---|---|
| Cache Aside | 弱最终一致 | 低 | 大多数读多写少场景 |
| 延时双删 | 较弱一致 | 中 | 简单业务，对一致性要求不高 |
| 订阅 binlog | 较强一致 | 高 | 高并发写场景，需要可靠同步 |
| 分布式锁 + 数据库 | 准强一致 | 高 | 库存扣减等强一致场景 |

选择哪个方案，要根据业务的**一致性要求**、**性能要求**、**实现成本**综合判断。没有最好的方案，只有最适合当前业务的方案。
