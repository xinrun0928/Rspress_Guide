# 缓存策略与一致性：保证数据不乱

> [!WARNING]
> 前置知识：本篇文章是缓存策略与一致性的入口，涵盖 Cache Aside 模式、缓存三大经典问题（穿透/击穿/雪崩）、缓存一致性方案。缓存策略是面试超高频模块，建议按顺序阅读。

---

缓存的世界里，有三个永恒的噩梦：

- **缓存穿透**：请求像水一样穿过缓存，直击数据库
- **缓存击穿**：一个 key 过期，所有请求同时涌入
- **缓存雪崩**：所有 key 同时过期，数据库被击穿

每一个问题，都可能在生产环境中造成灾难性的后果。而解决这些问题的关键，正是缓存策略的设计。

这一讲，我们来彻底解决缓存的三大经典问题。

### 各策略的适用场景

| 策略 | 适用场景 | 核心特点 |
|---|---|---|
| Cache Aside | 读多写少 | 实现简单，一致性够用 |
| Read Through | 读多写少 | 缓存层自动加载 |
| Write Through | 写操作必须立即可见 | 强一致，性能差 |
| Write Behind | 写操作频繁 | 性能高，一致性弱 |

## 二、缓存读写模式：选择正确的姿势

### Cache Aside：最常用的读写模式

Cache Aside（旁路缓存）是大多数业务系统的首选：

```java
// 读操作：先缓存，后数据库
public Product getProduct(Long id) {
    // 1. 先查缓存
    Product p = redis.get("product:" + id);
    if (p != null) {
        return p;  // 缓存命中，直接返回
    }

    // 2. 缓存未命中，查数据库
    p = productMapper.selectById(id);
    if (p == null) {
        return null;
    }

    // 3. 回填缓存
    redis.set("product:" + id, p, 1, TimeUnit.HOURS);
    return p;
}

// 写操作：先数据库，后删缓存
public void updateProduct(Product product) {
    // 1. 先更新数据库
    productMapper.updateById(product);

    // 2. 再删除缓存（不是更新！是删除！）
    redis.delete("product:" + product.getId());
}
```

> [!WARNING]
> 写操作时，**删除缓存而非更新缓存**。如果采用「更新数据库 + 更新缓存」的方式，在并发场景下可能导致数据不一致（缓存被旧值覆盖）。

### 写操作的并发问题

Cache Aside 在写操作时有一个经典的并发问题：

```
时间线：
T1：线程A 更新数据（先更新DB为1）
T2：线程A 删除缓存
T3：线程B 查询缓存（未命中）
T4：线程B 查DB（旧值为1）
T5：线程B 回填缓存（值为1）
T6：线程A 已完成操作

结果：数据已更新为1，但缓存中仍是旧值（如果是更新操作，此时问题暴露）
```

解决方案是**延时双删**：在写操作后延时一段时间，再删除一次缓存。

```java
public void updateProduct(Product product) {
    // 1. 先更新数据库
    productMapper.updateById(product);

    // 2. 删除缓存
    redis.delete("product:" + product.getId());

    // 3. 延时再次删除（300-500ms，足以让读操作完成）
    Thread.sleep(300);
    redis.delete("product:" + product.getId());
}
```

## 三、缓存三大经典问题

### 问题一：缓存穿透

**现象**：大量请求查询一个不存在的数据，缓存中没有，数据库中也没有，请求全部穿透到数据库。

**危害**：数据库被击穿，轻则响应变慢，重则数据库崩溃。

**解决方案一：缓存空值**

```java
public Product getProduct(Long id) {
    String cacheKey = "product:" + id;

    // 1. 查缓存
    String cached = redis.get(cacheKey);
    if (cached != null) {
        // 缓存命中
        if ("NULL".equals(cached)) {
            return null;  // 空值标记，直接返回
        }
        return JSON.parseObject(cached, Product.class);
    }

    // 2. 查数据库
    Product p = productMapper.selectById(id);
    if (p == null) {
        // 空值也缓存，短过期时间（如1分钟）
        redis.setex(cacheKey, 60, "NULL");
        return null;
    }

    // 3. 回填缓存
    redis.setex(cacheKey, 3600, JSON.toJSONString(p));
    return p;
}
```

**解决方案二：布隆过滤器**

对于数据量大、空值查询多的场景，布隆过滤器是更优雅的方案：

```java
// 布隆过滤器判断 key 是否存在
private final BloomFilter<Long> bloomFilter = BloomFilter.create(
    Funnels.longFunnel(), 100_000_000, 0.01  // 1亿条数据，1%误判率
);

// 查询前先判断
public Product getProduct(Long id) {
    // 1. 布隆过滤器判断
    if (!bloomFilter.mightContain(id)) {
        return null;  // 一定不存在，直接返回
    }

    // 2. 查缓存和数据库
    return getProductFromCacheOrDb(id);
}
```

> [!TIP]
> 布隆过滤器有误判率（可能将不存在的 key 判断为存在），但不会漏判（不会将存在的 key 判断为不存在）。适用于「大多数请求都应该命中」的场景。

### 问题二：缓存击穿

**现象**：一个热点 key 过期瞬间，大量请求同时涌入，全部打到数据库。

**危害**：数据库压力骤增，响应时间飙升。

**解决方案一：互斥锁**

```java
public Product getProduct(Long id) {
    String cacheKey = "product:" + id;

    // 1. 先查缓存
    Product p = redis.get(cacheKey);
    if (p != null) {
        return p;
    }

    // 2. 获取分布式锁
    String lockKey = "lock:product:" + id;
    String lockValue = UUID.randomUUID().toString();
    Boolean locked = redis.set(lockKey, lockValue, 30, TimeUnit.SECONDS, "NX");

    if (Boolean.TRUE.equals(locked)) {
        try {
            // 再次查缓存（防止其他线程已加载）
            p = redis.get(cacheKey);
            if (p != null) {
                return p;
            }

            // 查数据库
            p = productMapper.selectById(id);
            if (p != null) {
                redis.setex(cacheKey, 3600, JSON.toJSONString(p));
            }
            return p;
        } finally {
            // 释放锁
            redis.delete(lockKey);
        }
    } else {
        // 未获取到锁，短暂等待后重试
        Thread.sleep(50);
        return getProduct(id);
    }
}
```

**解决方案二：逻辑过期**

```java
public Product getProduct(Long id) {
    String cacheKey = "product:" + id;

    // 1. 查缓存
    String cached = redis.get(cacheKey);
    if (cached == null) {
        return null;
    }

    CacheData<Product> cacheData = JSON.parseObject(cached,
        new TypeReference<CacheData<Product>>() {});

    // 2. 判断是否逻辑过期
    if (cacheData.isExpired()) {
        // 已过期，开启异步加载
        threadPool.submit(() -> {
            reloadProduct(id);
        });
        // 返回旧数据（如果有的话）
        return cacheData.getData();
    }

    return cacheData.getData();
}
```

### 问题三：缓存雪崩

**现象**：大量缓存 key 同时过期，导致大量请求同时击穿到数据库。

**危害**：数据库被集体击穿，可能引发连锁反应。

**解决方案一：过期时间随机化**

```java
// 给过期时间加随机偏移
int expireSeconds = 3600 + RandomUtils.nextInt(300);  // 1小时 + 0~5分钟随机
redis.setex(cacheKey, expireSeconds, JSON.toJSONString(p));
```

**解决方案二：多级缓存**

即使 Redis 缓存全部失效，本地缓存仍然可以拦截大部分请求。

**解决方案三：熔断降级**

```java
public Product getProduct(Long id) {
    try {
        return getFromCacheOrDb(id);
    } catch (Exception e) {
        // 缓存故障，降级到数据库
        log.warn("缓存故障，降级到数据库: {}", id);
        return productMapper.selectById(id);
    }
}
```

## 四、缓存与数据库一致性

### 延时双删

在写操作后延时一段时间再删除缓存：

```java
public void updateProduct(Product product) {
    productMapper.updateById(product);
    redis.delete("product:" + product.getId());
    // 延时 300-500ms
    CompletableFuture.delayedExecutor(300, TimeUnit.MILLISECONDS)
        .execute(() -> redis.delete("product:" + product.getId()));
}
```

### 订阅 binlog

通过 MySQL binlog 异步同步缓存变更：

```java
// 模拟 Canal 订阅 binlog
@CanalTable(tableName = "product")
@Component
public class ProductCanalListener {

    @Autowired
    private RedisTemplate<String, String> redisTemplate;

    @Insert
    public void insert(CanalEntity<Product> entity) {
        Product p = entity.getData();
        redisTemplate.opsForValue().set(
            "product:" + p.getId(),
            JSON.toJSONString(p)
        );
    }

    @Update
    public void update(CanalEntity<Product> entity) {
        // 缓存更新的最佳实践：删除而非更新
        Product p = entity.getData();
        redisTemplate.delete("product:" + p.getId());
    }

    @Delete
    public void delete(CanalEntity<Product> entity) {
        Long id = entity.getData().getId();
        redisTemplate.delete("product:" + id);
    }
}
```

## 延伸思考

缓存三大问题的本质，都是**并发场景下的资源争用**。

缓存穿透是「恶意」并发（请求不存在的数据），缓存击穿是「热点」并发（热点数据过期），缓存雪崩是「批量」并发（大量数据同时过期）。三种问题的根源不同，解决思路也不同。

下次遇到缓存问题，先问自己：**这是哪一种并发场景？应该用哪一种策略？**
