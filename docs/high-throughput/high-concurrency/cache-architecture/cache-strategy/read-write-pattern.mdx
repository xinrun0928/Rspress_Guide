# 缓存读写模式：Cache Aside、Read Through、Write Through、Write Behind

想象这样一个场景：凌晨 2 点，你刚上线了一个新的查询接口，QPS 预计能扛住一万。压测报告漂亮得很，一切看起来都很完美。结果早上 9 点用户涌入，不到十分钟，数据库 CPU 直接打满，系统开始疯狂报警。

问题出在哪？你确实加了缓存，但缓存和数据库的读写顺序没想清楚——写的时候先更新缓存还是先更新数据库？读的时候缓存不存在怎么办？这些问题没处理好，再好的缓存方案也会变成定时炸弹。

|缓存读写模式|就是来解决这个问题的。它不是某一种具体的技术，而是一套经过大量实践验证的**数据同步策略**。今天我们就来聊聊这四种主流模式各自适合什么场景，怎么选才不会踩坑。

---

## 一、为什么需要模式化？

很多开发者在第一次接触缓存时，想法很简单：读的时候先查缓存，有就返回，没有就查数据库然后回填缓存。写的时候更新数据库就行了。

这个思路本身没错，但如果系统规模稍微大一点，问题就来了：

- **并发读写**：两个请求同时更新同一个数据，一个更新数据库，一个更新缓存，谁先谁后？最终数据一致性如何保证？
- **缓存过期**：缓存设置了 TTL，但数据库更新了，缓存里的旧数据还能撑多久？
- **失败处理**：更新数据库成功了，但缓存更新失败了怎么办？数据是不是就乱了？

这些问题在单机、低并发场景下几乎不会发生，但在分布式、高并发的生产环境中，几乎必然遇到。|缓存读写模式|就是把这些场景抽象成标准化的处理流程，让团队在面对缓存问题时有一套统一的答案。

---

## 二、Cache Aside（旁路缓存）

### 2.1 模式原理

|Cache Aside|是业界最常用的缓存模式，也叫「旁路缓存」。它的核心思想是：**缓存是数据库的旁路，数据源始终是数据库**。

这个模式看起来简单，但细节全在流程里。

### 2.2 读流程

```text
请求 → 查缓存 → 命中 → 返回数据
         ↓ 未命中
        查数据库 → 写入缓存 → 返回数据
```

用代码来理解会更清晰：

```java
public String get(String key) {
    // 1. 先查缓存
    String value = redis.get(key);
    if (value != null) {
        return value;
    }

    // 2. 缓存未命中，查数据库
    value = db.query(key);

    // 3. 回填缓存，注意设置过期时间
    redis.setex(key, 3600, value);

    return value;
}
```

这个流程里有几个关键点：

- 缓存未命中时，**必须查数据库再回填**，而不是返回空
- 回填时**一定要设置过期时间**，否则缓存会一直存储永不更新的数据
- 读取操作**始终以数据库为准**，缓存只是加速层

### 2.3 写流程

```text
请求 → 更新数据库 → 删除缓存
```

注意，这里是「**删除缓存**」而不是「**更新缓存**」。这一点非常重要，我见过很多新手会踩这个坑。

为什么是删除而不是更新？因为并发场景下，这样做会导致数据不一致：

```text
# 线程 A 先更新数据库（值为 2）
线程 A: UPDATE user SET age = 2 WHERE id = 1

# 线程 B 后更新数据库（值为 3）
线程 B: UPDATE user SET age = 3 WHERE id = 1
```

如果采用「先更新缓存」的方式：

```java
// 线程 A: 先更新缓存为 2
redis.set("user:1:age", 2);
db.update("UPDATE user SET age = 2 WHERE id = 1");

// 线程 B: 后更新缓存为 3
redis.set("user:1:age", 3);
db.update("UPDATE user SET age = 3 WHERE id = 1");
```

看起来没问题。但如果时序变成这样：

```java
// 线程 A: 先更新缓存为 2
redis.set("user:1:age", 2);

// 线程 B: 立即更新缓存为 3
redis.set("user:1:age", 3);

// 线程 A: 更新数据库
db.update("UPDATE user SET age = 2 WHERE id = 1");  // 数据库变成 2

// 线程 B: 更新数据库
db.update("UPDATE user SET age = 3 WHERE id = 1");  // 数据库变成 3
```

最终数据库是 3，缓存也是 3，看起来没问题。但如果数据库更新失败了呢？缓存里是 3，数据库里还是旧值。这就是「**更新缓存**」的问题——很难保证原子性。

而「**删除缓存**」的方式：

```java
public void update(String key, String value) {
    // 1. 先更新数据库
    db.update(key, value);

    // 2. 删除缓存
    redis.del(key);
}
```

下一次读取时，会重新从数据库加载最新数据到缓存。即使并发，数据库也是唯一数据源，不会有不一致的问题。

### 2.4 Cache Aside 的三个核心问题

|问题|描述|典型场景|解决方案|
|---|---|---|---|
|缓存穿透|大量请求访问不存在的数据，缓存无效，请求直达数据库|恶意刷接口、爬虫爬不存在的 ID|布隆过滤器 + 缓存空值|
|缓存击穿|一个热点 key 过期，瞬间大量请求击穿缓存打到数据库|秒杀商品详情、爆款文章|互斥锁 + 逻辑过期|
|缓存雪崩|大量缓存同时过期或缓存服务宕机，大量请求打到数据库|系统初始化、Redis 宕机|过期时间随机化 + 多级缓存|

这三个问题是 Cache Aside 模式特有的「副作用」，后面三篇文章会逐一展开。

## 三、Read Through（读穿透）

### 3.1 模式原理

|Read Through|的核心是**缓存自动加载数据**。调用方不需要关心数据从哪来，只管问缓存要，缓存自己会去数据库加载。

```text
请求 → 查缓存 → 命中 → 返回数据
         ↓ 未命中
        缓存自己查数据库 → 回填 → 返回数据
```

这个模式把「缓存未命中时查数据库」的逻辑封装到了缓存层，调用方感知不到数据库的存在。

### 3.2 与 Cache Aside 的区别

|对比点|Cache Aside|Read Through|
|---|---|---|
|数据加载|应用代码负责|缓存自己负责|
|代码复杂度|需要写回填逻辑|调用方代码更简单|
|适用场景|细粒度控制缓存策略|通用缓存场景|
|典型实现|Jedis/RedisTemplate|Cache aside with load-aside|

### 3.3 实现示例

以 Spring Cache 为例，使用 `@Cacheable` 注解就是典型的 Read Through 模式：

```java
@Service
public class UserService {

    @Cacheable(value = "users", key = "#userId")
    public User getUser(Long userId) {
        // 这个方法只有在缓存未命中时才会执行
        // 缓存命中时，这个方法根本不会调用
        return userRepository.findById(userId);
    }
}
```

调用 `getUser(1L)` 时，Spring Cache 拦截器会先查 Redis 里有没有 `users::1` 这个 key，有就直接返回，没有就执行方法体里的数据库查询，然后把结果写入缓存。

对于调用方来说，它只看到了缓存，完全不知道数据库的存在。这就是 Read Through 的精髓——**把数据加载的复杂度封装在缓存层**。

## 四、Write Through（写穿透）

### 4.1 模式原理

|Write Through|也叫「同步写」。它的核心是：**写操作必须同时更新缓存和数据库**，两者都成功才算完成。

```text
写请求 → 写缓存 → 写数据库 → 全部成功 → 返回成功
          ↓ 或 ↓
        任一失败 → 回滚 → 返回失败
```

### 4.2 实现示例

```java
public void updateUser(User user) {
    // 1. 先更新缓存
    redis.set("user:" + user.getId(), user);

    // 2. 再更新数据库
    db.update(user);

    // 如果数据库更新失败，需要回滚缓存
}
```

这个模式的好处是**强一致性**——缓存和数据库始终同步。坏处也很明显：**写性能差**，因为要等两次写入完成。

### 4.3 与 Cache Aside 的区别

|对比点|Cache Aside|Write Through|
|---|---|---|
|写入策略|只写数据库，删除缓存|同时写缓存和数据库|
|一致性|最终一致|强一致|
|写入性能|快（一次 IO）|慢（两次 IO）|
|实现复杂度|简单|复杂，需要处理回滚|

Write Through 适合「写少读多」且对一致性要求极高的场景，比如金融系统的账户余额。但对于大多数互联网业务来说，Cache Aside 的最终一致已经足够。

## 五、Write Behind（写回/异步写）

### 5.1 模式原理

|Write Behind|也叫「Write Back」或「异步写」。它的核心是：**写操作只写缓存，然后立即返回**，缓存会在合适时机批量同步到数据库。

```text
写请求 → 写缓存 → 返回成功（异步）
                    ↓
            后台线程定期批量写数据库
```

### 5.2 为什么需要异步写？

数据库的写入 IO 成本远高于内存操作。在高并发写入场景下，每次写入都同步落盘，数据库会成为性能瓶颈。Write Behind 通过「写缓存 + 异步落盘」的方式，大幅提升写入性能。

典型的应用场景：

- **操作系统文件缓存**：写文件时先写 Page Cache，由后台线程定期刷盘
- **消息队列**：生产者发消息先写内存，由 Broker 批量持久化
- **数据库日志**：WAL（Write-Ahead Logging）先写日志再落盘

### 5.3 实现示例

```java
public class AsyncWriteService {

    private Map<String, Object> localCache = new ConcurrentHashMap<>();
    private ScheduledExecutorService executor = Executors.newSingleThreadScheduledExecutor();

    public void write(String key, Object value) {
        // 1. 先写入本地缓存
        localCache.put(key, value);

        // 2. 标记为脏数据
        dirtyKeys.add(key);
    }

    // 后台线程定期同步
    @PostConstruct
    public void init() {
        executor.scheduleAtFixedRate(() -> {
            Set<String> keys = new HashSet<>(dirtyKeys);
            dirtyKeys.clear();

            for (String key : keys) {
                Object value = localCache.get(key);
                db.save(key, value);
            }
        }, 5, 5, TimeUnit.SECONDS);
    }
}
```

### 5.4 数据丢失风险

Write Behind 的致命问题是**数据丢失**。如果缓存写入后还没来得及落盘，系统突然宕机，这部分数据就丢了。

所以 Write Behind 适合的场景是：

- 对丢失有一定容忍度的数据（如日志、指标）
- 有其他备份机制的数据（如消息队列有副本）
- 写入性能比一致性更重要的场景

对于核心业务数据（如订单、支付），Write Behind 基本上是不考虑的。

## 六、四种模式对比

|模式|读流程|写流程|一致性|性能|适用场景|
|---|---|---|---|---|---|
|Cache Aside|缓存命中返回，未命中查库回填|只写库，删除缓存|最终一致|读快写快|大多数互联网业务（推荐）|
|Read Through|缓存未命中时自动查库|只写库，缓存自动更新|最终一致|读快|通用缓存场景|
|Write Through|同 Read Through|同时写缓存和库|强一致|写慢|一致性要求极高的场景|
|Write Behind|同 Read Through|只写缓存，异步落库|弱一致|写极快|日志、指标等容忍丢数据的场景|

这张表的核心结论是：**大多数业务选 Cache Aside，有特殊需求再考虑其他模式**。

## 七、Cache Aside 的实际应用

光说不练假把式，来看看 Cache Aside 在真实业务中是怎么用的。

### 7.1 典型查询场景

```java
@Service
public class ProductService {

    @Autowired
    private RedisTemplate<String, Product> redisTemplate;

    @Autowired
    private ProductMapper productMapper;

    private static final String CACHE_KEY_PREFIX = "product:";
    private static final long CACHE_TTL = 30 * 60; // 30 分钟

    public Product getProduct(Long productId) {
        String cacheKey = CACHE_KEY_PREFIX + productId;

        // 先查缓存
        Product product = redisTemplate.opsForValue().get(cacheKey);
        if (product != null) {
            log.debug("缓存命中: productId={}", productId);
            return product;
        }

        // 缓存未命中，查数据库
        product = productMapper.selectById(productId);
        if (product == null) {
            log.warn("商品不存在: productId={}", productId);
            return null;
        }

        // 回填缓存
        redisTemplate.opsForValue().set(cacheKey, product,
            CACHE_TTL, TimeUnit.SECONDS);
        log.debug("缓存写入: productId={}", productId);

        return product;
    }
}
```

### 7.2 典型更新场景

```java
public void updateProduct(Product product) {
    // 1. 更新数据库
    productMapper.updateById(product);

    // 2. 删除缓存，而不是更新
    String cacheKey = CACHE_KEY_PREFIX + product.getId();
    redisTemplate.delete(cacheKey);

    log.info("更新商品并删除缓存: productId={}", product.getId());
}
```

### 7.3 容易踩的坑

**坑一：先删缓存再更新数据库**

```java
// 错误写法
redis.del(key);
db.update(value);
```

问题：并发时，线程 A 删缓存后还没更新数据库，线程 B 查到旧值回填缓存，导致缓存里是脏数据。

**坑二：缓存过期时间设置过长**

缓存 TTL 设置太长，如果数据库更新了，用户会长时间看到旧数据。建议根据业务特性设置合理的过期时间，一般 30 分钟以内比较常见。

**坑三：不处理缓存失败**

```java
// 错误写法
public void updateProduct(Product product) {
    productMapper.updateById(product);
    redis.delete(key); // 如果这里抛异常，整个事务回滚
}
```

正确的做法是把缓存删除放在 `try-catch` 里，即使失败也不影响主流程。

## 八、如何选择缓存读写模式

说了这么多四种模式，到底怎么选？

### 8.1 选择决策树

```
业务场景
├── 一致性要求极高（如金融、订单）
│   └── Write Through
│
├── 写入量巨大，容忍短暂丢失（如日志、埋点）
│   └── Write Behind
│
└── 大多数互联网业务（读多写少）
    └── Cache Aside（首选）
        或 Read Through（简化代码）
```

### 8.2 决策因素

|因素|推荐模式|原因|
|---|---|---|
|读多写少|Cache Aside|读性能最优，写复杂度低|
|写多读少|Write Through 或直接写库|避免频繁写缓存增加负担|
|一致性要求极高|Write Through|同步写入保证强一致|
|写入性能要求极高|Write Behind|异步写入性能最好|
|代码简洁优先|Read Through|缓存逻辑封装在底层|

### 8.3 实战建议

对于大多数 Java 后端服务，我的建议是：

1. **默认选 Cache Aside**。这套模式经过了无数互联网公司的验证，踩坑成本最低
2. **用 Read Through 简化代码**。Spring Cache + Redis 的组合能让你少写很多缓存逻辑
3. **不要轻易用 Write Behind**。除非你有完善的故障恢复和数据补偿机制
4. **Write Through 只用于核心链路**。比如支付、账户类接口，需要单独设计

---

## 总结

四种缓存读写模式，各有各的适用场景：

- **Cache Aside**：旁路缓存，以数据库为准，是大多数业务的首选
- **Read Through**：缓存自动加载，调用方代码更简单
- **Write Through**：同步写，强一致，但性能较差
- **Write Behind**：异步写，性能最好，但有丢数据风险

没有银弹，只有trade-off。理解每种模式的原理和代价，才能在实际业务中做出正确的选择。

## 面试追问预判

**Q1：为什么 Cache Aside 要「删除缓存」而不是「更新缓存」？**

A：更新缓存存在并发一致性问题。如果先更新缓存再更新数据库，并发时可能出现「缓存是新的、数据库是旧的」情况。而删除缓存后，下次读取会重新从数据库加载最新数据，最终一致性更强。具体原因在 2.3 节的并发分析中已经详细说明。

**Q2：Cache Aside 的「删除缓存」为什么在「更新数据库」之后，而不是之前？**

A：如果先删缓存再更新数据库，并发场景下会出现：线程 A 删缓存 → 线程 B 查库得到旧值 → 线程 B 回填缓存 → 线程 A 更新数据库。最终缓存里是旧值。而先更新数据库再删缓存，线程 B 查到的旧值会重新从数据库加载，不会有脏数据问题。

**Q3：Write Through 和 Write Behind 在 Redis 持久化中有体现吗？**

A：有的。Redis 的 AOF 持久化是典型的 Write Behind 模式——写命令先到 AOF 缓冲区，后台定期刷盘。而 Redis 的 RDB 快照更像是 Write Through——每间隔一段时间同步生成快照文件。

**Q4：Read Through 和 Cache Aside 在实现上有什么区别？**

A：核心区别在于数据加载逻辑的归属。Cache Aside 需要应用代码自己处理「未命中 → 查库 → 回填」的逻辑；Read Through 把这个逻辑封装在缓存层，调用方感知不到数据库的存在。从代码整洁度看 Read Through 更好，从灵活性看 Cache Aside 更可控。
