# 缓存穿透：布隆过滤器 + 缓存空值

凌晨 3 点，监控大屏突然亮起刺眼的红色——数据库 CPU 飙升至 98%，大量慢查询报警纷至沓来。你赶紧打开日志，发现大量请求都在查同一个不存在的用户 ID。

「这个 ID 根本不存在啊，为什么还会有这么多请求？」

你揉了揉眼睛，确认没看错。这些请求的 key 在数据库里也查不到，但它们偏偏绕过了缓存，直接打到了数据库。如果这种情况持续下去，数据库会被活活拖死。

这就是**缓存穿透**——缓存和数据库都找不到数据，请求直达数据库的异常场景。今天我们就来彻底解决这个问题。

## 一、什么是缓存穿透？

### 1.1 定义

|缓存穿透|是指大量请求访问**缓存和数据库中都不存在**的数据。由于缓存中不存在，查询会直接打到数据库；当数据库也查不到时，结果不会回填缓存，导致大量无效请求穿透缓存层直达数据库。

### 1.2 危害

缓存穿透的危害是**数据库压力过大甚至崩溃**。具体表现：

- 数据库 CPU 飙升，响应时间变慢
- 数据库连接被占满，正常业务无法获取连接
- 如果是恶意攻击，可能导致整个服务不可用

更可怕的是，穿透问题往往是「慢性死亡」——一开始只是少量请求，可能没人注意，但随着爬虫、恶意刷接口等行为累积，问题会越来越严重。

### 1.3 与其他缓存问题的区别

|问题|缓存中|数据库中|结果|
|---|---|---|---|
|缓存穿透|不存在|不存在|请求直达数据库|
|缓存击穿|不存在（过期）|存在|请求直达数据库|
|缓存雪崩|大量不存在/过期|存在（但扛不住）|大量请求直达数据库|

简单记忆：

- **穿透**：数据压根不存在
- **击穿**：热点数据过期了
- **雪崩**：大量数据同时过期或缓存宕机

## 二、经典场景

### 2.1 恶意刷接口

最典型的场景是**恶意刷接口**。攻击者使用大量不存在的 ID 不断请求接口，比如：

```
/api/user/100001
/api/user/100002
/api/user/100003
...
/api/user/999999
```

正常用户的 ID 通常是连续的，但攻击者可能会扫描整个 ID 范围。如果业务设计不合理，这些不存在的 ID 每次都会查库。

### 2.2 爬虫爬取

搜索引擎爬虫会尽可能多地抓取网站内容，包括大量不存在的页面。如果网站没有对这些 URL 做缓存，爬虫的请求会直接打到数据库。

### 2.3 错误的数据设计

有时候穿透问题源于**数据设计的缺陷**。比如：

- 用户输入了非法 ID（负数、特殊字符）
- 业务上允许「删除」数据，但删除时只删了数据库，缓存没处理
- ID 生成规则混乱，导致产生了大量「无效 ID」

### 2.4 程序员写错代码

```java
// 错误：查询结果为 null 时，没有回填缓存
public String getUserName(Long userId) {
    String name = redis.get("user:name:" + userId);
    if (name != null) {
        return name;
    }
    User user = db.query(userId);
    if (user != null) {
        redis.set("user:name:" + userId, user.getName());
        return user.getName();
    }
    return null; // 这里 user 是 null，没有回填，下次还是查库
}
```

## 三、解决方案一：缓存空值

### 3.1 原理

缓存空值是最简单的方案：**当数据库也查不到数据时，在缓存中存入一个空值（`null` 或空字符串）**，后续请求直接命中缓存返回空值，不再查库。

```text
请求 → 查缓存 → 命中（空值） → 返回空
         ↓ 未命中
        查数据库 → 数据库也为空
                   ↓
              回填空值 → 返回空
```

### 3.2 代码实现

```java
public String getUserName(Long userId) {
    String cacheKey = "user:name:" + userId;

    // 1. 先查缓存
    String name = redis.get(cacheKey);
    if (name != null) {
        return name; // 命中返回（可能是真实数据，也可能是空值标记）
    }

    // 2. 缓存未命中，查数据库
    User user = db.query(userId);

    // 3. 回填缓存
    if (user != null) {
        redis.setex(cacheKey, 3600, user.getName());
        return user.getName();
    } else {
        // 数据库也查不到，缓存空值，过期时间短一些
        redis.setex(cacheKey, 60, "");
        return null;
    }
}
```

### 3.3 空值标记的选择

|方式|优点|缺点|
|---|---|---|
|`null`|语义清晰|Java 中 null 需要特殊处理|
|`空字符串 ""`|处理简单|无法区分「真正空」和「空字符串值」|

建议使用一个特殊字符串（如 `"NULL"` 或 `"EMPTY"`）作为空值标记，并在读取时特殊处理：

```java
private static final String NULL_MARKER = "__NULL__";

public String getUserName(Long userId) {
    String name = redis.get(cacheKey);

    if (name != null) {
        if (NULL_MARKER.equals(name)) {
            return null; // 空值标记，返回 null
        }
        return name;
    }

    // ... 查库逻辑同上
}
```

### 3.4 过期时间设置

空值的过期时间**不要太长**，否则：

- 真实数据入库后，用户还得等过期才能看到
- 长时间缓存空值浪费内存

建议设置为 **30 秒到 5 分钟**，根据业务对「数据延迟」的容忍度调整。

### 3.5 优点和缺点

|优点|缺点|
|---|---|
|实现简单|浪费缓存空间（存储大量无效数据）|
|见效快|需要维护空值标记|
|对穿透问题有效|空值过期后，穿透问题会再次出现|

## 四、解决方案二：布隆过滤器

### 4.1 什么是布隆过滤器？

|布隆过滤器（Bloom Filter）|是 1970 年由 Burton Howard Bloom 提出的空间高效的概率数据结构。它可以用来判断一个元素**一定不存在**或者**可能存在**。

关键点：

- **一定不存在**：布隆过滤器说没有，那就一定没有
- **可能存在**：布隆过滤器说有，可能有（存在误判）

### 4.2 原理

布隆过滤器的核心是一个**位数组（bit array）**和**多个哈希函数**。

```text
初始状态：所有位都是 0

添加元素 "user:100"：
1. 通过哈希函数 1 计算：hash1("user:100") = 2 → 第 2 位设为 1
2. 通过哈希函数 2 计算：hash2("user:100") = 5 → 第 5 位设为 1
3. 通过哈希函数 3 计算：hash3("user:100") = 8 → 第 8 位设为 1

查询元素 "user:100"：
1. 通过哈希函数 1 计算：hash1("user:100") = 2 → 第 2 位是 1 ✓
2. 通过哈希函数 2 计算：hash2("user:100") = 5 → 第 5 位是 1 ✓
3. 通过哈希函数 3 计算：hash3("user:100") = 8 → 第 8 位是 1 ✓
→ 三位都是 1，返回「可能存在」

查询元素 "user:999"（不存在）：
1. 通过哈希函数 1 计算：hash1("user:999") = 2 → 第 2 位是 1 ✓
2. 通过哈希函数 2 计算：hash2("user:999") = 7 → 第 7 位是 0 ✗
→ 返回「一定不存在」
```

### 4.3 误判率分析

布隆过滤器的误判只存在于「假阳性」，不存在「假阴性」：

- **假阳性**：不存在的元素，被判断为存在（可能）
- **假阴性**：存在的元素，被判断为不存在（不可能）

为什么会误判？因为哈希碰撞——不同的元素可能计算出相同的哈希值。

```
元素 A: hash1=2, hash2=5, hash3=8
元素 B: hash1=2, hash2=7, hash3=9

添加 A 后，第 2、5、8 位为 1。
查询 B 时，第 2 位是 1（但这是 A 的哈希碰撞），第 7 位是 0。
由于第 7 位是 0，直接返回「一定不存在」。

但是如果查询 C: hash1=3, hash2=5, hash3=9
第 3 位是 0，直接返回「一定不存在」。

再查询 D: hash1=2, hash2=7, hash3=8
第 2 位是 1，第 7 位是 1，第 8 位是 1 → 返回「可能存在」
但 D 实际不存在，这就是误判！
```

### 4.4 误判率公式

布隆过滤器的误判率与以下因素相关：

|因素|影响|
|---|---|
|位数组大小|位数组越大，误判率越低|
|哈希函数数量|过多或过少都会增加误判率|
|元素数量|元素越多，误判率越高|

误判率公式：

```
p = (1 - e^(-kn/m))^k
```

其中：

- `p`：误判率
- `m`：位数组大小
- `n`：元素数量
- `k`：哈希函数数量

最优哈希函数数量：`k = (m/n) * ln(2)`

### 4.5 Guava BloomFilter 使用

Google Guava 提供了布隆过滤器的实现，非常方便：

```java
import com.google.common.hash.BloomFilter;
import com.google.common.hash.Funnels;

public class BloomFilterExample {

    // 创建布隆过滤器，预期插入 1000000 个元素，误判率 0.01（1%）
    private static final BloomFilter<Long> bloomFilter = BloomFilter.create(
        Funnels.longFunnel(),
        1_000_000,
        0.01
    );

    // 初始化时加载所有存在的用户 ID
    public void init() {
        List<Long> allUserIds = userMapper.selectAllUserIds();
        allUserIds.forEach(bloomFilter::put);
    }

    public String getUserName(Long userId) {
        // 1. 先检查布隆过滤器
        if (!bloomFilter.mightContain(userId)) {
            // 布隆过滤器说一定不存在，直接返回空
            return null;
        }

        // 2. 布隆过滤器说可能存在，查缓存
        String name = redis.get("user:name:" + userId);
        if (name != null) {
            return name;
        }

        // 3. 缓存未命中，查数据库
        User user = db.query(userId);
        if (user != null) {
            redis.setex("user:name:" + userId, 3600, user.getName());
            return user.getName();
        }

        return null;
    }
}
```

### 4.6 注意事项

布隆过滤器不支持删除操作。如果需要删除元素，可以使用：

- **计数布隆过滤器（Counting Bloom Filter）**：位数组用计数器代替
- **布隆过滤器的变种**：如 Scalable Bloom Filter、Stable Bloom Filter

对于大多数业务场景，标准布隆过滤器就够了。

## 五、Redis 实现布隆过滤器

### 5.1 RedisBloom 插件

Redis 4.0+ 支持通过模块加载布隆过滤器——RedisBloom。

```bash
# 安装（Docker 方式）
docker run -d -p 6379:6379 redislabs/rebloom
```

```bash
# 添加元素
BF.ADD bloom_key "user:100"

# 检查元素是否存在
BF.EXISTS bloom_key "user:100"

# 批量添加
BF.MADD bloom_key "user:101" "user:102" "user:103"

# 批量检查
BF.MEXISTS bloom_key "user:100" "user:999"
```

### 5.2 性能对比

|实现|优点|缺点|
|---|---|---|
|Guava BloomFilter|无需额外依赖，适合单机|JVM 重启后数据丢失，需要重新加载|
|RedisBloom|数据持久化，支持分布式|需要额外部署，增加复杂度|

### 5.3 选择建议

|场景|推荐方案|
|---|---|
|单机应用，数据量不大|Guava BloomFilter|
|分布式系统，多节点共享|RedisBloom|
|数据量极大（亿级以上）|RedisBloom + 分片|

## 六、布隆过滤器的变种

### 6.1 计数布隆过滤器（Counting Bloom Filter）

标准布隆过滤器不支持删除，因为删除可能会影响其他元素的判断。计数布隆过滤器用计数器数组代替位数组：

```text
标准布隆过滤器：
位数组: [0, 1, 1, 0, 1, 0, 0, 1]

计数布隆过滤器：
计数器: [0, 2, 1, 0, 1, 0, 0, 3]
        ↑ 添加了 "A"   ↑ 添加了 "B"
```

删除时，计数器减 1 而不是直接清零。

### 6.2 缩减布隆过滤器（Scalable Bloom Filter）

当元素数量超过预期时，需要扩容。Scalable Bloom Filter 通过多个标准布隆过滤器来实现：

```java
// 初始容量 10000，误判率 1%
ScalableBloomFilter<String> filter = ScalableBloomFilter.create(
    Funnels.stringFunnel(StandardCharsets.UTF_8),
    10_000,
    1.01 // false positive probability
);
```

### 6.3 对比

|类型|支持删除|适用场景|
|---|---|---|
|标准布隆过滤器|否|数据只增不减的场景（如用户 ID 黑名单）|
|计数布隆过滤器|是|需要动态删除元素的场景|
|缩减布隆过滤器|否|数据量不确定，可能扩容的场景|

## 七、实际方案：组合使用

### 7.1 布隆过滤器 + 缓存空值

单一方案都有缺陷，组合使用才是最佳实践：

```java
public String getUserName(Long userId) {
    String cacheKey = "user:name:" + userId;

    // 1. 检查布隆过滤器
    if (!bloomFilter.mightContain(userId)) {
        // 布隆过滤器说一定不存在，直接返回
        // 可以顺便缓存空值，避免下次再走布隆过滤器
        redis.setex(cacheKey, 60, NULL_MARKER);
        return null;
    }

    // 2. 布隆过滤器说可能存在，查缓存
    String name = redis.get(cacheKey);
    if (name != null) {
        if (NULL_MARKER.equals(name)) {
            return null;
        }
        return name;
    }

    // 3. 缓存未命中，查数据库
    User user = db.query(userId);
    if (user != null) {
        redis.setex(cacheKey, 3600, user.getName());
        return user.getName();
    }

    // 4. 数据库也查不到，缓存空值
    redis.setex(cacheKey, 60, NULL_MARKER);
    return null;
}
```

### 7.2 方案对比

|方案|防穿透效果|内存占用|实现复杂度|数据延迟|
|---|---|---|---|---|
|仅缓存空值|✓ 好|高（存储大量空值）|低|小|
|仅布隆过滤器|✓ 好|低|中|无|
|布隆过滤器 + 缓存空值|✓✓ 最好|中|中|小|

### 7.3 注意事项

1. **布隆过滤器需要预热**：系统启动时加载所有有效数据到布隆过滤器，否则新数据会被误判为不存在
2. **空值过期时间要短**：给真实数据留一个快速生效的窗口
3. **布隆过滤器的误判率要控制**：通常设置为 1% 或更低
4. **布隆过滤器不支持删除**：如果业务需要删除数据，可能需要换用计数布隆过滤器

## 八、方案选型决策

### 8.1 决策树

```
穿透问题场景
├── 数据量小（万级以下）
│   └── 直接用缓存空值
│
├── 数据量中等（百万级）
│   └── Guava BloomFilter
│
├── 数据量巨大（千万级以上）
│   └── RedisBloom
│
└── 需要动态删除
    └── 计数布隆过滤器
```

### 8.2 实战建议

1. **先评估数据规模**：如果数据量不大，布隆过滤器可能过度设计
2. **布隆过滤器作为前置过滤**：用布隆过滤器挡住大部分无效请求
3. **空值兜底**：即使布隆过滤器放过来了，数据库查不到也要缓存空值
4. **监控误判率**：定期检查布隆过滤器的实际误判率，调整参数

## 总结

缓存穿透是缓存架构中最容易被忽视的问题，因为它在数据量小的时候几乎不会发生。但一旦被恶意利用或者爬虫触发，后果可能是数据库崩溃。

解决方案的核心思路是「**宁可错杀，不可放过**」：

- **布隆过滤器**：快速判断数据一定不存在，直接返回
- **缓存空值**：即使查库为空，也缓存起来，避免重复查库

两者组合使用，既能高效拦截无效请求，又能兜底处理边缘情况，是生产环境的最佳实践。

## 面试追问预判

**Q1：布隆过滤器的误判率可以降到 0 吗？**

A：理论上不行。布隆过滤器的核心是哈希表，必然存在哈希碰撞。要消除误判，只能使用确定性的数据结构（如 HashMap），但内存消耗会大幅增加。实际工程中，1% 甚至 0.1% 的误判率都是可以接受的。

**Q2：布隆过滤器和缓存空值一起用，会不会浪费内存？**

A：不会。布隆过滤器存储的是「可能存在」的标记，内存占用极低（约 1 bit/元素）。而缓存空值只存储「确定不存在」的数据，过期时间很短（通常 30-60 秒）。两者结合，既高效又安全。

**Q3：布隆过滤器的初始化数据从哪来？**

A：通常有两个来源——（1）系统启动时从数据库全量加载；（2）实时更新，当新增数据时同步添加到布隆过滤器。需要注意的是，布隆过滤器不支持删除，如果业务有删除需求，需要使用计数布隆过滤器。

**Q4：布隆过滤器的空间占用如何计算？**

A：可以使用公式 `m = -n * ln(p) / (ln(2)^2)`，其中 n 是元素数量，p 是期望误判率。例如 100 万元素、1% 误判率，约需要 9.6 Mbit（约 1.2 MB）的空间。

**Q5：RedisBloom 和 Guava BloomFilter 怎么选？**

A：如果系统是分布式的，多个节点需要共享布隆过滤器，用 RedisBloom；如果单机应用，数据可以随 JVM 重启重新加载，用 Guava BloomFilter 更简单。两种方案的空间效率差不多，RedisBloom 多了网络开销。
