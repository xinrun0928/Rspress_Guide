# 缓存雪崩：过期时间随机化 + 熔断降级 + 多级缓存

双十一零点刚过 30 秒，你长舒一口气——系统稳住了，订单量稳步上涨，一切都在掌控之中。

但就在 0 点 31 分，报警声突然响起。数据库 CPU 从 20% 瞬间飙到 100%，所有接口开始超时，大量请求失败。你慌了，赶紧检查 Redis——Redis 没事，连接正常，数据都在。

那问题出在哪？日志显示大量请求同时在查同一个用户数据，key 不存在。用户数据？缓存什么时候过期的？打开缓存管理后台一看——所有用户数据的过期时间都是 0 点 30 分整。

你的心脏漏跳了一拍。

这是**缓存雪崩**——不是一两个 key 的问题，而是成片成片的缓存同时失效，瞬间所有请求都涌向数据库。

## 一、什么是缓存雪崩？

### 1.1 定义

|缓存雪崩|是指大量缓存 key 在**同一时间集中失效**，导致大量请求直接打到数据库，数据库压力瞬间过载而崩溃的现象。

更广义的雪崩还包括：**缓存服务宕机**，导致所有请求无法命中缓存，全部涌向数据库。

### 1.2 危害

缓存雪崩的危害是**系统级联故障**：

- 数据库 CPU 飙升，响应时间从毫秒级变成秒级
- 数据库连接被占满，正常业务无法获取连接
- 上游服务开始超时、熔断
- 整个系统逐渐不可用，形成雪崩效应

雪崩之所以可怕，是因为它往往超出预期。一个缓存 key 过期不会有问题，一百个也不会有太大问题，但当数千、数万个 key 同时过期时，问题就爆发了。

### 1.3 三个成因

|成因|描述|典型场景|
|---|---|---|
|大量 key 集中过期|缓存 key 设置了相同的过期时间，到期后同时失效|系统初始化时批量写入缓存、促销活动预热|
|缓存服务宕机|Redis 等缓存服务不可用，所有请求无法命中缓存|Redis 主从切换、网络分区、机器宕机|
|缓存预热失败|系统启动时缓存预热不充分，大量请求触发回源|重启后第一批请求、突发流量|

## 二、三剑客对比：雪崩 vs 击穿 vs 穿透

|问题|核心特征|影响范围|数据状态|
|---|---|---|---|
|缓存穿透|数据根本不存在|所有非法请求|缓存无，数据库无|
|缓存击穿|单个热点 key 过期|单个 key 的并发请求|缓存无（过期），数据库有|
|缓存雪崩|大量 key 同时失效/宕机|整个系统的所有请求|缓存无（批量），数据库有（但扛不住）|

用一句话总结：

- **穿透**：请求的数据不存在（从根上就没有）
- **击穿**：一个 key 过期了（孤军奋战）
- **雪崩**：成片 key 过期或宕机（全军覆没）

## 三、解决方案一：过期时间随机化

### 3.1 原理

这是最简单、最基础的解决方案。核心思想是：**不要让所有 key 在同一时间过期**。

如果原来是：

```java
// 所有 key 的过期时间都是 1 小时
redis.setex("product:1", 3600, data);
redis.setex("product:2", 3600, data);
redis.setex("product:3", 3600, data);
```

改成：

```java
// 过期时间 = 基础时间 + 随机偏移量
int baseExpire = 3600;
int randomExpire = ThreadLocalRandom.current().nextInt(300); // 0-300 秒随机
int actualExpire = baseExpire + randomExpire;

redis.setex("product:1", actualExpire, data);
redis.setex("product:2", actualExpire, data);
redis.setex("product:3", actualExpire, data);
```

这样，即使有一批 key 在同一时间写入，它们的过期时间也会分散在 1 小时到 1 小时 05 分之间，不会在同一时刻全部失效。

### 3.2 代码封装

```java
public class CacheKeyBuilder {

    private static final int DEFAULT_BASE_EXPIRE = 3600;
    private static final int DEFAULT_RANDOM_BOUND = 300;

    public static long buildExpireSeconds(int baseExpire, int randomBound) {
        return baseExpire + ThreadLocalRandom.current().nextInt(randomBound);
    }

    public static long buildExpireSeconds() {
        return buildExpireSeconds(DEFAULT_BASE_EXPIRE, DEFAULT_RANDOM_BOUND);
    }
}

// 使用
@Service
public class ProductService {

    public void cacheProduct(Product product) {
        long expire = CacheKeyBuilder.buildExpireSeconds(3600, 300);
        redis.setex("product:" + product.getId(), expire, product);
    }
}
```

### 3.3 进阶：过期时间分桶

对于更严格的场景，可以采用「分桶」策略：

```java
public static long buildExpireWithBucket(int baseExpire, int bucketCount) {
    int bucket = ThreadLocalRandom.current().nextInt(bucketCount);
    int bucketSize = 60; // 每个桶 60 秒
    return baseExpire + bucket * bucketSize;
}
```

这样过期时间会更均匀地分布，而不是简单的线性随机。

### 3.4 优缺点

|优点|缺点|
|---|---|
|实现简单|不能完全消除雪崩，只能缓解|
|零额外成本|随机偏移量设多大需要评估|
|立竿见影|对缓存宕机场景无效|

## 四、解决方案二：永不过期 + 逻辑过期

### 4.1 原理

这个方案从根源上解决「集中过期」问题：**热点数据设置超长的过期时间（甚至永不过期）**，通过后台定时任务或逻辑过期机制更新缓存。

```java
@Data
public class CacheData<T> {
    private T data;
    private Long logicalExpireTime; // 逻辑过期时间
}

// 写入时，物理过期时间设长一些
redis.setex("product:1", 86400 * 7, cacheData); // 7 天物理过期

// 读取时，检查逻辑过期时间
public Product getProduct(Long productId) {
    CacheData<Product> cacheData = redis.get("product:" + productId);

    if (cacheData == null) {
        // 缓存不存在，查库
        return db.query(productId);
    }

    // 检查逻辑过期
    if (cacheData.getLogicalExpireTime() > System.currentTimeMillis()) {
        // 未逻辑过期，正常返回
        return cacheData.getData();
    } else {
        // 已逻辑过期，返回旧数据，同时异步刷新
        refreshCacheAsync(productId);
        return cacheData.getData();
    }
}
```

### 4.2 定时刷新

```java
@Scheduled(fixedRate = 300000) // 每 5 分钟执行
public void refreshAllProducts() {
    List<Long> productIds = productMapper.selectAllActiveProductIds();

    for (Long productId : productIds) {
        try {
            Product product = productMapper.selectById(productId);
            CacheData<Product> cacheData = new CacheData<>();
            cacheData.setData(product);
            cacheData.setLogicalExpireTime(System.currentTimeMillis() + 30 * 60 * 1000);

            redis.setex("product:" + productId, 86400, JSON.toJSONString(cacheData));
        } catch (Exception e) {
            log.error("刷新产品缓存失败: productId={}", productId, e);
        }
    }
}
```

### 4.3 优缺点

|优点|缺点|
|---|---|
|从根本上解决集中过期问题|实现复杂度高|
|用户体验好（无感知）|需要维护定时任务|
|数据库压力分散|数据一致性存在延迟|

## 五、解决方案三：多级缓存

### 5.1 原理

|多级缓存|是指在 Redis 之前再加一层**本地缓存**（如 Caffeine、Guava Cache），热点数据优先走本地缓存，只有本地缓存没有才查 Redis。

```
请求 → 本地缓存（L1） → 命中 → 返回
           ↓ 未命中
        Redis 缓存（L2） → 命中 → 回填 L1 → 返回
           ↓ 未命中
        数据库（DB） → 回填 L2 → 回填 L1 → 返回
```

本地缓存的优势是**毫秒级响应**，且完全不需要网络开销。对于热点数据，大部分请求会在 L1 层就被拦截，不会打到 Redis。

### 5.2 代码实现

```java
@Configuration
public class MultiLevelCacheConfig {

    @Bean
    public Cache<Long, Product> localCache() {
        return Caffeine.newBuilder()
            .maximumSize(10000)      // 最多缓存 10000 个商品
            .expireAfterWrite(60, TimeUnit.SECONDS)  // 写入后 60 秒过期
            .recordStats()           // 开启统计
            .build();
    }
}

@Service
public class MultiLevelProductService {

    @Autowired
    private Cache<Long, Product> localCache;

    @Autowired
    private RedisTemplate<String, Product> redisTemplate;

    @Autowired
    private ProductMapper productMapper;

    public Product getProduct(Long productId) {
        // 1. 查 L1 本地缓存
        Product product = localCache.getIfPresent(productId);
        if (product != null) {
            return product;
        }

        // 2. 查 L2 Redis 缓存
        String redisKey = "product:" + productId;
        product = redisTemplate.opsForValue().get(redisKey);
        if (product != null) {
            // 回填 L1
            localCache.put(productId, product);
            return product;
        }

        // 3. 查数据库
        product = productMapper.selectById(productId);
        if (product != null) {
            // 回填 L2 和 L1
            redisTemplate.opsForValue().set(redisKey, product, 3600, TimeUnit.SECONDS);
            localCache.put(productId, product);
        }

        return product;
    }

    // 更新时，清除所有层级的缓存
    public void updateProduct(Product product) {
        // 1. 更新数据库
        productMapper.updateById(product);

        // 2. 清除 L1
        localCache.invalidate(product.getId());

        // 3. 清除 L2
        redisTemplate.delete("product:" + product.getId());
    }
}
```

### 5.3 多级缓存的优势

|层级|响应时间|容量|适用场景|
|---|---|---|---|
|L1 本地缓存|纳秒~微秒级|小（内存有限）|极高热点数据|
|L2 Redis 缓存|毫秒级|大（可水平扩展）|一般热点数据|
|DB|毫秒~秒级|无限|兜底|

### 5.4 缓存一致性问题

多级缓存最大的问题是**一致性**。当数据更新时，需要同时失效所有层级的缓存：

```java
public void updateProduct(Product product) {
    // 1. 更新数据库
    productMapper.updateById(product);

    // 2. 删除 Redis 缓存
    redisTemplate.delete("product:" + product.getId());

    // 3. 删除本地缓存
    localCache.invalidate(product.getId());

    // 4. 广播通知其他节点删除本地缓存
    redisTemplate.convertAndSend("cache:invalidate", product.getId().toString());
}
```

其他节点订阅消息，删除自己的本地缓存：

```java
@PostConstruct
public void subscribeCacheInvalidation() {
    redisTemplate.getConnectionFactory().getConnection()
        .subscribe(
            (message, pattern) -> {
                String productId = new String(message.getBody());
                localCache.invalidate(Long.parseLong(productId));
            },
            "cache:invalidate".getBytes()
        );
}
```

### 5.5 优缺点

|优点|缺点|
|---|---|
|性能极佳（热点数据 L1 拦截）|实现复杂度高|
|抗雪崩能力强（L1 挡住大部分请求）|缓存一致性维护复杂|
|灵活性高（可动态调整层级）|本地缓存占用 JVM 堆内存|

## 六、解决方案四：服务降级与熔断

### 6.1 原理

当缓存雪崩导致数据库压力过大时，**熔断降级**可以从数据库端拦截流量，避免数据库被打垮。

```
正常情况：
请求 → 缓存 → 未命中 → 数据库 → 返回

熔断降级后：
请求 → 缓存 → 未命中 → 熔断器打开 → 返回默认值/友好提示
                              ↑
                         数据库压力过大
```

熔断器的核心是「**快速失败**」——当数据库响应慢或错误率高时，不再等待，直接返回兜底数据。

### 6.2 Sentinel 熔断降级

Sentinel 是 Alibaba 开源的流量控制组件，支持熔断降级：

```java
@RestController
public class ProductController {

    @Autowired
    private ProductService productService;

    @GetMapping("/product/{id}")
    @SentinelResource(value = "getProduct",
        blockHandler = "getProductBlockHandler",
        fallback = "getProductFallback")
    public Product getProduct(@PathVariable Long id) {
        return productService.getProduct(id);
    }

    // 熔断降级处理
    public Product getProductBlockHandler(Long id, BlockException ex) {
        // 熔断触发时，返回兜底数据
        return Product.getDefault();
    }

    // Fallback 处理
    public Product getProductFallback(Long id, Throwable ex) {
        // 异常时，返回兜底数据
        log.error("获取商品异常: id={}", id, ex);
        return Product.getDefault();
    }
}
```

配置熔断规则：

```java
private void initDegradeRule() {
    DegradeRule rule = new DegradeRule("getProduct")
        .setGrade(CircuitBreakerStrategy.ERROR_RATIO.getType())
        .setCount(0.5)  // 错误率超过 50%
        .setMinRequestAmount(10)  // 最小请求数
        .setStatIntervalMs(10000)  // 10 秒内
        .setSlowRatioThreshold(1.0)  // 慢调用比例阈值
        .setMaxResponseTime(1000)  // 最大响应时间
        .setTimeWindow(30);  // 熔断持续时间 30 秒

    DegradeRuleManager.loadRules(Collections.singletonList(rule));
}
```

### 6.3 降级方案设计

降级不是简单的返回错误，而是要**有策略地返回兜底数据**：

|场景|降级策略|兜底数据|
|---|---|---|
|商品详情|返回缓存的历史数据或默认商品|缓存的旧数据|
|用户信息|返回缓存的历史数据|缓存的旧数据或匿名用户|
|排行榜|返回降级期间的静态榜单|上一周期的榜单|
|实时数据|返回友好提示|提示「服务繁忙」|

### 6.4 优缺点

|优点|缺点|
|---|---|
|保护数据库不被压垮|用户看到的是降级数据|
|系统整体可用性提高|降级策略需要精心设计|
|可与限流配合|降级恢复需要人工介入|

## 七、解决方案五：Redis 高可用架构

### 7.1 问题根源

如果是 Redis 宕机导致的雪崩，解决方案就是**让 Redis 本身高可用**。

### 7.2 Redis Sentinel（哨兵模式）

Sentinel 是 Redis 的高可用方案，核心能力：

- **监控**：监控主从节点的健康状态
- **自动故障转移**：主节点宕机后，自动选举从节点为新的主节点
- **通知**：故障转移后通知客户端

```bash
# Docker 部署 Sentinel
docker run -d --name redis-sentinel \
    -v /data/sentinel.conf:/etc/redis/sentinel.conf \
    redis redis-sentinel /etc/redis/sentinel.conf
```

```txt title="sentinel.conf"
sentinel monitor mymaster 127.0.0.1 6379 2
sentinel down-after-milliseconds mymaster 30000
sentinel failover-timeout mymaster 180000
```

### 7.3 Redis Cluster

对于数据量更大、并发更高的场景，使用 Redis Cluster：

- **数据分片**：数据分布在多个节点上
- **高可用**：每个分片有主从复制
- **水平扩展**：可以动态增加节点

```java
// Jedis 连接 Cluster
JedisCluster jedis = new JedisCluster(
    new HostAndPort("127.0.0.1", 7001),
    new HostAndPort("127.0.0.1", 7002),
    new HostAndPort("127.0.0.1", 7003),
    10,
    "password"
);
```

### 7.4 多活架构

更极端的场景，可以使用多活架构：

```
用户请求 → 入口 → 就近访问本地 Redis → 本地没有 → 访问远程 Redis
```

本地 Redis 作为缓存层，即使远程 Redis 宕机，本地缓存也能撑一段时间。

## 八、流量预热与异步加载

### 8.1 问题

系统重启或大批量缓存失效后，第一波请求会直接打到数据库，因为缓存还是冷的。

### 8.2 启动预热

```java
@PostConstruct
public void warmUp() {
    log.info("开始缓存预热...");

    List<Long> hotProductIds = hotProductMapper.selectTopN(1000);
    for (Long productId : hotProductIds) {
        try {
            Product product = productMapper.selectById(productId);
            String cacheKey = "product:" + productId;

            // 随机过期时间
            int expire = 3600 + ThreadLocalRandom.current().nextInt(300);
            redisTemplate.opsForValue().set(cacheKey, product, expire, TimeUnit.SECONDS);

            // 也预热到本地缓存
            localCache.put(productId, product);
        } catch (Exception e) {
            log.warn("预热商品失败: productId={}", productId, e);
        }
    }

    log.info("缓存预热完成，共预热 {} 条", hotProductIds.size());
}
```

### 8.3 异步预热

启动预热可能耗时较长，可以改成异步：

```java
@PostConstruct
public void warmUpAsync() {
    CompletableFuture.runAsync(() -> {
        // 等服务完全启动后再预热
        try {
            Thread.sleep(10000);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
        warmUp();
    });
}
```

### 8.4 限流保护

预热过程中，可以配合限流保护数据库：

```java
public void warmUpWithRateLimit() {
    Semaphore semaphore = new Semaphore(10); // 最多 10 个并发

    for (Long productId : hotProductIds) {
        semaphore.acquire();
        CompletableFuture.runAsync(() -> {
            try {
                // 预热逻辑
                doWarmUp(productId);
            } finally {
                semaphore.release();
            }
        });
    }
}
```

## 九、综合实战：多层防御体系

单一方案都有局限性，生产环境往往是多层防御：

```java
@Service
public class DefensiveProductService {

    // L1 本地缓存
    @Autowired
    private Cache<Long, Product> localCache;

    // L2 Redis
    @Autowired
    private RedisTemplate<String, Product> redisTemplate;

    @Autowired
    private ProductMapper productMapper;

    // 熔断器
    private AtomicInteger failureCount = new AtomicInteger(0);
    private volatile boolean circuitOpen = false;
    private long lastFailureTime = 0;

    public Product getProduct(Long productId) {
        // 1. 检查熔断器
        if (circuitOpen) {
            return getFallbackProduct(productId);
        }

        // 2. 查 L1 本地缓存
        Product product = localCache.getIfPresent(productId);
        if (product != null) {
            return product;
        }

        // 3. 查 L2 Redis
        try {
            String redisKey = "product:" + productId;
            product = redisTemplate.opsForValue().get(redisKey);

            if (product != null) {
                // 回填 L1
                localCache.put(productId, product);
                return product;
            }

            // 4. 缓存未命中，查数据库
            product = queryFromDatabase(productId);

            // 5. 回填缓存（随机过期）
            if (product != null) {
                int expire = 3600 + ThreadLocalRandom.current().nextInt(300);
                redisTemplate.opsForValue().set(redisKey, product, expire, TimeUnit.SECONDS);
                localCache.put(productId, product);
            }

            // 重置熔断计数器
            failureCount.set(0);
            return product;

        } catch (Exception e) {
            // 记录失败，增加熔断计数器
            int failures = failureCount.incrementAndGet();
            lastFailureTime = System.currentTimeMillis();

            if (failures >= 10) {
                circuitOpen = true;
                log.warn("熔断器打开");
            }

            // 降级返回
            return getFallbackProduct(productId);
        }
    }

    private Product queryFromDatabase(Long productId) {
        return productMapper.selectById(productId);
    }

    private Product getFallbackProduct(Long productId) {
        // 尝试从本地缓存获取（即使过期也返回）
        Product product = localCache.getIfPresent(productId);
        if (product != null) {
            return product;
        }

        // 最后兜底：返回默认商品或空
        return Product.getDefault();
    }

    // 熔断恢复检查
    @Scheduled(fixedRate = 10000)
    public void checkCircuitBreaker() {
        if (circuitOpen && System.currentTimeMillis() - lastFailureTime > 30000) {
            circuitOpen = false;
            failureCount.set(0);
            log.info("熔断器关闭");
        }
    }
}
```

### 多层防御体系总结

|层级|防御措施|作用|
|---|---|---|
|L1|本地缓存（Caffeine）|热点数据毫秒级响应|
|L2|Redis + 随机过期|分散过期，降低雪崩概率|
|熔断|快速失败，不拖垮数据库|兜底保护|
|限流|控制请求速率|削峰填谷|
|预热|启动时加载热点数据|避免冷启动|
|高可用|Redis Sentinel/Cluster|防止 Redis 宕机引发雪崩|

---

## 总结

缓存雪崩是缓存架构中最具破坏性的问题之一，因为它会引发连锁反应，导致整个系统崩溃。

核心解决思路是「**分散 + 兜底 + 高可用**」：

- **分散**：过期时间随机化，避免集中失效
- **分层**：多级缓存，减少数据库压力
- **兜底**：熔断降级，保护数据库不被压垮
- **高可用**：Redis 集群，防止缓存服务宕机

没有银弹，只有多层防御。根据业务场景和预算，选择合适的方案组合，才是正确的应对之道。

## 面试追问预判

**Q1：过期时间随机化能完全解决雪崩问题吗？**

A：不能完全解决，只能缓解。随机化只能分散过期时间，但如果 Redis 本身宕机，随机化就无效了。真正的雪崩防御需要多管齐下：随机过期 + 多级缓存 + 熔断降级 + Redis 高可用。

**Q2：多级缓存的一致性问题怎么解决？**

A：核心是「**先更新数据库，再删除所有层级缓存**」，而不是更新缓存。具体做法：更新数据库 → 删除 Redis 缓存 → 删除本地缓存 → 广播通知其他节点删除本地缓存。注意顺序不要反，否则可能导致脏数据。

**Q3：熔断器和限流器有什么区别？**

A：限流器是「**预防**」，在请求进来时控制速率；熔断器是「**补救**」，在系统已经出问题后快速失败。限流器保护的是系统不被过载；熔断器保护的是系统不被拖垮。两者配合使用效果更好。

**Q4：Redis Cluster 和 Sentinel 如何选择？**

A：如果数据量不大（几 GB 以内），需要高可用，选 Sentinel（主从复制）；如果数据量大（几十 GB 以上），需要分片，选 Cluster。Cluster 支持数据分片和水平扩展，但复杂度更高。

**Q5：预热时如何避免打爆数据库？**

A：三个方法：（1）限流 + 异步：控制预热并发度；（2）分级预热：先预热最热的数据，逐步扩展；（3）延迟预热：服务完全启动后再预热，给系统一个缓冲期。
