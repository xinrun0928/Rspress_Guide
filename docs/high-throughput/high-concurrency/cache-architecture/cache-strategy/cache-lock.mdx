# 分布式锁与缓存实现的高并发并发控制

import { Badge } from '@rspress/core/theme';

> [!WARNING]
> 前置知识：本篇文章涉及 Redis、分布式系统、并发编程等知识，建议先掌握 Redis 基础命令与并发编程概念。

双十一零点，你盯着监控大屏，看着请求量曲线像火箭一样飙升。订单服务、库存服务、支付服务——所有系统都在承受着前所未有的压力。

突然，库存服务报警：同一件商品，在同一毫秒内被扣减了三次。三次都返回了「成功」，但库存只剩下一件。

这是典型的**并发控制失效**问题。在单机环境下，我们可以使用 `synchronized` 或 `ReentrantLock` 来保护共享资源。但在分布式环境下，多个服务实例可能同时访问同一份数据，本地锁无能为力。

分布式锁 + 缓存的组合，正是解决这类问题的「黄金搭档」。

## 一、分布式锁在缓存场景的核心作用

### 1.1 为什么需要分布式锁

单机环境下的锁为什么不够用了？

考虑这个场景：一个微服务部署了 4 个实例，每个实例都使用 `ConcurrentHashMap` 做本地缓存。当用户请求到达时：

```text
请求1 → 实例A → 本地缓存未命中 → 查数据库
请求2 → 实例B → 本地缓存未命中 → 查数据库  ← 同一数据，重复查询
请求3 → 实例C → 本地缓存未命中 → 查数据库  ← 再次重复
请求4 → 实例D → 本地缓存未命中 → 查数据库  ← 又一次重复
```

这就是**缓存击穿**问题——大量请求同时访问一个热点数据，缓存失效的瞬间，所有请求都涌向数据库。

分布式锁的作用是：**在缓存失效的瞬间，只允许一个请求回源查库，其他请求等待结果。**

### 1.2 分布式锁的三大核心能力

分布式锁在缓存场景中需要具备：

| 能力 | 说明 | 如果没有会怎样 |
|---|---|---|
| **互斥性** | 同一时刻只有一个客户端能持有锁 | 多个请求同时回源，击穿数据库 |
| **防死锁** | 锁必须能被释放，避免永久阻塞 | 服务宕机后锁无法释放，系统卡死 |
| **可重入** | 同一线程可多次获取同一把锁 | 嵌套调用时发生死锁 |

### 1.3 分布式锁 + 缓存的经典模式

分布式锁和缓存的组合，核心解决两个问题：

1. **缓存击穿**：热点数据过期时，大量请求同时回源
2. **缓存与数据库一致性**：防止并发写入导致的数据覆盖

```java title="分布式锁 + 缓存的经典模式"
public Product getProduct(Long id) {
    String cacheKey = "product:" + id;
    String lockKey = "lock:product:" + id;
    
    // 第一步：尝试获取缓存
    Product cached = redisTemplate.opsForValue().get(cacheKey);
    if (cached != null) {
        return cached;
    }
    
    // 第二步：尝试获取分布式锁
    String threadId = UUID.randomUUID().toString();
    boolean locked = redisTemplate.opsForValue()
        .setIfAbsent(lockKey, threadId, 30, TimeUnit.SECONDS);
    
    if (!locked) {
        // 获取锁失败，说明有其他线程正在加载数据
        // 短暂等待后重试获取缓存
        return retryGetFromCache(cacheKey, 3);
    }
    
    try {
        // 第三步：获取锁成功，再次检查缓存（可能其他线程已经加载）
        cached = redisTemplate.opsForValue().get(cacheKey);
        if (cached != null) {
            return cached;
        }
        
        // 第四步：回源查库
        Product product = database.findById(id);
        
        // 第五步：写入缓存
        redisTemplate.opsForValue().set(cacheKey, product, 1, TimeUnit.HOURS);
        
        return product;
    } finally {
        // 第六步：释放锁
        releaseLock(lockKey, threadId);
    }
}

private void releaseLock(String lockKey, String threadId) {
    // 使用 Lua 脚本保证原子性
    String script = 
        "if redis.call('get', KEYS[1]) == ARGV[1] then " +
        "    return redis.call('del', KEYS[1]) " +
        "else " +
        "    return 0 " +
        "end";
    redisTemplate.execute(
        new DefaultRedisScript<>(script, Long.class),
        Collections.singletonList(lockKey),
        threadId);
}
```

## 二、Redis 分布式锁的七步实现

Redis 实现分布式锁看似简单，但要做得「正确」却需要考虑很多细节。

### 2.1 第一步：SET NX EX（原子性加锁）

这是最基础的加锁命令：

```bash
SET lock_key unique_value NX EX 30
```

- `NX`：只有 key 不存在时才设置成功（互斥性）
- `EX 30`：设置 30 秒过期时间（防死锁）
- `unique_value`：唯一值，用于安全释放锁

为什么必须用 `SET NX EX` 而不是 `SETNX` + `EXPIRE` 分开执行？

```java
// 错误写法
redisTemplate.opsForValue().setIfAbsent(lockKey, threadId);  // 加锁
redisTemplate.expire(lockKey, 30, TimeUnit.SECONDS);          // 设置过期
// 如果在 setIfAbsent 和 expire 之间服务宕机，会死锁！
```

`SET NX EX` 是原子操作，不存在中间状态。

### 2.2 第二步：存储唯一值

为什么锁值必须唯一？

```java
// 场景：线程A获取锁后业务逻辑太慢，锁自动过期
// 此时线程B获取了同一把锁
// 线程A执行完毕，释放锁——但释放的是线程B的锁！

// 正确的释放逻辑
public void releaseLock(String lockKey, String expectedValue) {
    // 只有值匹配才删除，防止误删其他线程的锁
    String currentValue = redisTemplate.opsForValue().get(lockKey);
    if (expectedValue.equals(currentValue)) {
        redisTemplate.delete(lockKey);
    }
}
```

### 2.3 第三步：Lua 脚本释放锁

`GET + DELETE` 不是原子操作，需要用 Lua 脚本：

```lua
-- release_lock.lua
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
```

### 2.4 第四步：防锁超时（续期机制）

如果业务逻辑执行时间超过锁的 TTL 会怎样？

```java
// 场景：锁设置30秒过期，但业务逻辑需要45秒
// 30秒后锁自动释放，其他线程获取锁
// 线程A继续执行，释放锁时删除的是线程B的锁！
```

这就是锁续期问题。Redisson 框架通过「看门狗」机制解决这个问题。

### 2.5 第五步：看门狗续期机制

Redisson 的看门狗每 10 秒检查一次，如果锁还在持有中，就自动延长 TTL：

```java title="看门狗机制示意"
public class Watchdog {
    private static final long LOCK_INTERNAL_TIME = 10_000; // 10秒
    
    private void scheduleExpirationRenewal(String threadId) {
        expirationRenewalMap.put(threadId, 
            CompletableFuture.runAsync(() -> {
                while (true) {
                    try {
                        Thread.sleep(LOCK_INTERNAL_TIME);
                        
                        // 检查锁是否还在持有
                        if (isLocked()) {
                            // 续期
                            renewExpiration();
                        } else {
                            break;
                        }
                    } catch (InterruptedException e) {
                        break;
                    }
                }
            }));
    }
}
```

默认情况下，看门狗的续期间隔是 TTL 的 1/3。如果 TTL 是 30 秒，续期间隔是 10 秒。

### 2.6 第六步：公平锁与非公平锁

Redis 默认的 SET NX 是非公平锁——谁先抢到谁拿锁。但在某些场景下，公平性很重要。

**非公平锁**：线程按抢锁顺序竞争，可能导致「饥饿」。

**公平锁**：线程按请求顺序排队，先到先得。

```java
// Redisson 公平锁
RLock fairLock = redissonClient.getFairLock("myFairLock");
fairLock.lock();
try {
    // 业务逻辑
} finally {
    fairLock.unlock();
}
```

公平锁的实现原理是 Redis 的 `ZSET`——将线程信息按时间戳存入有序集合，抢锁时检查自己是否是集合中的最小元素。

### 2.7 第七步：多节点 RedLock

单节点 Redis 有单点故障风险。RedLock 的思路是：在 N 个独立的 Redis 实例上获取锁，只有超过 N/2+1 个实例成功，才算获取锁成功。

```java
// Redisson 多节点锁
RedissonMultiLock multiLock = new RedissonMultiLock(
    redissonClient.getLock("node1"),
    redissonClient.getLock("node2"),
    redissonClient.getLock("node3")
);
multiLock.lock();
try {
    // 业务逻辑
} finally {
    multiLock.unlock();
}
```

RedLock 的核心假设是：多个独立的 Redis 实例同时宕机的概率远低于单实例。对于一般业务，单节点 Redis + 持久化已经足够。

## 三、分布式锁 + 缓存的组合模式

### 3.1 模式一：单机版分布式锁

最简单的模式，所有服务实例共享同一个 Redis：

```java title="单机分布式锁 + 缓存"
@Service
public class ProductCacheService {
    
    @Autowired
    private RedissonClient redissonClient;
    
    public Product getProduct(Long id) {
        String cacheKey = "product:" + id;
        
        // 1. 先查缓存
        Product cached = redisTemplate.opsForValue().get(cacheKey);
        if (cached != null) {
            return cached;
        }
        
        // 2. 获取分布式锁
        RLock lock = redissonClient.getLock("lock:product:" + id);
        lock.lock(30, TimeUnit.SECONDS);
        
        try {
            // 3. 双重检查
            cached = redisTemplate.opsForValue().get(cacheKey);
            if (cached != null) {
                return cached;
            }
            
            // 4. 查数据库
            Product product = database.findById(id);
            
            // 5. 写缓存
            redisTemplate.opsForValue().set(cacheKey, product, 1, TimeUnit.HOURS);
            
            return product;
        } finally {
            lock.unlock();
        }
    }
}
```

**优点**：实现简单，适合大多数场景
**缺点**：单节点 Redis 有可用性风险

### 3.2 模式二：Redisson 分布式锁

使用 Redisson 简化锁管理：

```java title="Redisson 分布式锁"
@Service
public class ProductCacheService {
    
    private static final TimeUnit UNIT = TimeUnit.SECONDS;
    
    public Product getProduct(Long id) {
        String cacheKey = "product:" + id;
        String lockKey = "lock:product:" + id;
        
        // 1. 先查缓存
        Product cached = redisTemplate.opsForValue().get(cacheKey);
        if (cached != null) {
            return cached;
        }
        
        // 2. 获取锁（自动续期）
        RLock lock = redissonClient.getLock(lockKey);
        boolean locked = lock.tryLock(10, 30, UNIT);
        
        if (!locked) {
            // 获取锁失败，等一会再试
            return retryGet(cacheKey, 3);
        }
        
        try {
            // 3. 双重检查
            cached = redisTemplate.opsForValue().get(cacheKey);
            if (cached != null) {
                return cached;
            }
            
            // 4. 查库写缓存
            Product product = database.findById(id);
            redisTemplate.opsForValue().set(cacheKey, product, 1, UNIT);
            
            return product;
        } finally {
            lock.unlock();
        }
    }
}
```

Redisson 的优势：

- 自动续期（看门狗）
- 可重入
- 公平锁/读写锁
- 多节点 RedLock

### 3.3 模式三：缓存锁 vs 数据库锁

分布式锁保护的是缓存层，但最终写入的还是数据库。数据库本身也有锁机制。

| 场景 | 缓存锁 | 数据库锁 |
|---|---|---|
| 防止缓存击穿 | 必须用分布式锁 | 无法解决（缓存本身不保护） |
| 防止超卖 | 可用，但不可靠 | 必须用数据库唯一索引 |
| 库存扣减 | 辅助作用 | 核心保障 |

对于库存扣减这类场景，正确的做法是：

```java title="库存扣减：数据库 + 分布式锁"
public boolean deductStock(Long productId, Integer quantity) {
    // 1. 分布式锁（可选，主要防止缓存击穿）
    String lockKey = "stock:lock:" + productId;
    RLock lock = redissonClient.getLock(lockKey);
    lock.lock();
    
    try {
        // 2. 数据库乐观锁（必须）
        int affected = jdbcTemplate.update(
            "UPDATE products SET stock = stock - ? " +
            "WHERE id = ? AND stock >= ?",
            quantity, productId, quantity);
        
        if (affected > 0) {
            // 3. 删除缓存（保证后续读取正确）
            redisTemplate.delete("product:" + productId);
            return true;
        }
        
        return false;
    } finally {
        lock.unlock();
    }
}
```

**核心原则**：缓存锁是「辅助工具」，数据库锁是「最后防线」。

## 四、缓存并发控制中的死锁风险与超时处理

### 4.1 死锁的四种场景

分布式环境下的死锁比单机更复杂：

| 场景 | 描述 | 解决方案 |
|---|---|---|
| **锁永不过期** | 业务执行超时时锁未释放 | 必须设置 TTL + 看门狗续期 |
| **锁被误删** | 释放了其他线程持有的锁 | 使用唯一值 + Lua 脚本原子释放 |
| **锁未释放服务宕机** | finally 块未执行 | TTL 自动过期 |
| **重入导致死锁** | 同一线程重复获取锁 | 可重入锁实现 |

### 4.2 超时处理策略

获取锁失败时，有几种常见策略：

**策略一：快速失败**

```java
boolean locked = lock.tryLock(1, TimeUnit.SECONDS);
if (!locked) {
    throw new BusinessException("系统繁忙，请稍后重试");
}
```

适用场景：非核心链路，如推荐列表。

**策略二：等待重试**

```java
public Product getWithRetry(String lockKey, String cacheKey, Long id) {
    int retries = 3;
    while (retries-- > 0) {
        boolean locked = lock.tryLock(10, 30, TimeUnit.SECONDS);
        if (locked) {
            return loadAndCache(id);
        }
        
        // 短暂等待后重试
        try {
            Thread.sleep(100);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            break;
        }
    }
    
    // 最终兜底：直接查数据库
    return database.findById(id);
}
```

适用场景：库存查询、订单确认等核心链路。

**策略三：降级处理**

```java
public Product getProductFallback(Long id) {
    try {
        return getProduct(id);  // 正常逻辑
    } catch (LockException e) {
        // 降级：绕过缓存直接查库
        log.warn("获取锁失败，降级查库: {}", id);
        return database.findById(id);
    }
}
```

适用场景：需要保证可用性的场景。

### 4.3 锁粒度选择

锁粒度越细，并发度越高，但管理复杂度也越高：

```java
// 粗粒度锁：一个商品一把锁
String lockKey = "lock:product:" + productId;

// 细粒度锁：按 SKU 维度
String lockKey = "lock:product:" + productId + ":sku:" + skuId;

// 更细粒度：按库存批次
String lockKey = "lock:product:" + productId + ":batch:" + batchId;
```

选择锁粒度时考虑：

- 数据之间的关联性（同一商品的不同 SKU 是否可以并发修改）
- 业务一致性要求（不同粒度下的一致性保证不同）
- 管理复杂度（锁越多，管理的代码越复杂）

## 五、限流 + 缓存的组合

分布式锁解决的是「谁来查数据库」的问题，限流解决的是「有多少请求能查数据库」的问题。两者配合，是保护数据库的「黄金搭档」。

### 5.1 为什么需要限流

即使使用了分布式锁，同一时刻只有一个请求回源查库。但如果缓存完全失效（比如 Redis 宕机），所有请求都会同时查库。

限流的作用是：**在极端情况下，限制进入数据库的并发量。**

### 5.2 限流的几种算法

**计数器算法**：简单直接，但有临界问题。

```java
public boolean isAllowed(String key, int limit) {
    String countKey = "rate:" + key;
    Long count = redisTemplate.opsForValue().increment(countKey);
    
    if (count == 1) {
        redisTemplate.expire(countKey, 1, TimeUnit.SECONDS);
    }
    
    return count <= limit;
}
```

**滑动窗口算法**：更精确的计数。

```lua
-- sliding_window.lua
local key = KEYS[1]
local now = redis.call('TIME')[1]
local window = 1000  -- 1秒窗口

-- 删除窗口外的旧数据
redis.call('ZREMRANGEBYSCORE', key, 0, now - window)

-- 添加当前请求
redis.call('ZADD', key, now, now .. ':' .. math.random())

-- 统计窗口内请求数
local count = redis.call('ZCARD', key)

-- 设置过期
redis.call('EXPIRE', key, 2)

return count <= 100
```

**令牌桶算法**：允许一定程度的突发流量。

```java
RateLimiter rateLimiter = RateLimiter.create(100); // 每秒100个令牌
if (!rateLimiter.tryAcquire()) {
    throw new RateLimitException("请求过于频繁");
}
```

### 5.3 限流 + 分布式锁的组合方案

```java title="完整的高并发保护方案"
@Service
public class ProtectedProductService {
    
    private static final int DB_QPS_LIMIT = 100;  // 数据库每秒最多处理100个请求
    
    @Autowired
    private RedissonClient redissonClient;
    
    @Autowired
    private RateLimiter rateLimiter;
    
    public Product getProduct(Long id) {
        String cacheKey = "product:" + id;
        String lockKey = "lock:product:" + id;
        
        // 第一层：缓存（高性能）
        Product cached = redisTemplate.opsForValue().get(cacheKey);
        if (cached != null) {
            return cached;
        }
        
        // 第二层：限流（保护数据库）
        if (!rateLimiter.tryAcquire()) {
            throw new RateLimitException("系统繁忙");
        }
        
        // 第三层：分布式锁（防止击穿）
        RLock lock = redissonClient.getLock(lockKey);
        lock.lock(30, TimeUnit.SECONDS);
        
        try {
            // 双重检查
            cached = redisTemplate.opsForValue().get(cacheKey);
            if (cached != null) {
                return cached;
            }
            
            // 查库
            Product product = database.findById(id);
            
            // 写缓存
            redisTemplate.opsForValue().set(cacheKey, product, 1, TimeUnit.HOURS);
            
            return product;
        } finally {
            lock.unlock();
        }
    }
}
```

三层保护的意义：

| 层级 | 作用 | 命中率目标 |
|---|---|---|
| 缓存 | 扛住 99% 的读流量 | 99%+ |
| 限流 | 极端情况保护数据库 | - |
| 分布式锁 | 防止缓存击穿 | - |

## 六、缓存锁在高并发秒杀场景的应用

秒杀是缓存锁最典型的应用场景。

### 6.1 秒杀的三个阶段

| 阶段 | 特点 | 技术挑战 |
|---|---|---|
| **抢券/预热** | 大量用户同时进入页面 | 热点数据、系统预热 |
| **抢购开始** | 瞬时流量峰值 | 高并发扣减、库存一致性 |
| **结果查询** | 大量用户查询结果 | 缓存、异步处理 |

### 6.2 库存扣减的完整方案

```java title="秒杀库存扣减"
@Service
public class SeckillService {
    
    private static final String STOCK_KEY = "seckill:stock:";
    private static final String ORDER_KEY = "seckill:order:";
    
    @Autowired
    private RedissonClient redissonClient;
    
    public SeckillResult seckill(Long userId, Long skuId) {
        String stockKey = STOCK_KEY + skuId;
        String orderKey = ORDER_KEY + skuId + ":" + userId;
        String lockKey = "seckill:lock:" + skuId;
        
        // 1. 检查是否已购买（用 Redis SET 的 SISMEMBER）
        if (redisTemplate.opsForSet().isMember(orderKey, userId.toString())) {
            return SeckillResult.ALREADY_PURCHASED;
        }
        
        // 2. 限流保护（防止黄牛）
        if (!rateLimiter.tryAcquire("seckill:" + skuId, 10000)) {
            return SeckillResult.SYSTEM_BUSY;
        }
        
        // 3. 获取分布式锁（防止超卖）
        RLock lock = redissonClient.getLock(lockKey);
        boolean locked = lock.tryLock(5, 10, TimeUnit.SECONDS);
        
        if (!locked) {
            return SeckillResult.SYSTEM_BUSY;
        }
        
        try {
            // 4. 扣减库存（Redis DECR 的原子操作）
            Long remaining = redisTemplate.opsForValue().decrement(stockKey);
            
            if (remaining == null || remaining < 0) {
                // 库存不足，恢复库存
                if (remaining != null) {
                    redisTemplate.opsForValue().increment(stockKey);
                }
                return SeckillResult.SOLD_OUT;
            }
            
            // 5. 创建订单记录
            redisTemplate.opsForSet().add(orderKey, userId.toString());
            
            // 6. 发送订单消息（异步落库）
            kafkaTemplate.send("seckill-order", 
                new SeckillOrderEvent(userId, skuId, System.currentTimeMillis()));
            
            return SeckillResult.SUCCESS;
        } finally {
            lock.unlock();
        }
    }
}
```

### 6.3 超卖问题的彻底解决

Redis 扣减库存虽然快，但无法保证数据库的最终一致性。需要通过异步消息对账：

```java
@KafkaListener(topics = "seckill-order")
public void handleSeckillOrder(SeckillOrderEvent event) {
    // 1. 落库
    seckillOrderDao.insert(event.toOrder());
    
    // 2. 更新数据库库存（乐观锁）
    int affected = jdbcTemplate.update(
        "UPDATE seckill_stock SET stock = stock - 1 " +
        "WHERE sku_id = ? AND stock >= 1",
        event.getSkuId());
    
    if (affected == 0) {
        // 数据库库存不足，标记异常订单
        handleAbnormalOrder(event);
    }
}
```

**重要提醒**：Redis 库存扣减只是「预扣」，真正的库存以数据库为准。如果 Redis 和数据库出现不一致（概率极低但存在），需要通过后台对账程序修复。

## 七、分布式锁的降级方案

即使做了充分准备，分布式锁也可能失败。降级方案是系统的最后一道防线。

### 7.1 降级策略矩阵

| 锁获取结果 | 缓存状态 | 降级策略 |
|---|---|---|
| 成功 | 未命中 | 查库写缓存 |
| 成功 | 已命中 | 返回缓存 |
| 失败 | 未命中 | 直接查库（加熔断标记） |
| 失败 | 已命中 | 返回缓存 |

### 7.2 熔断标记机制

当锁服务不可用时，系统应该感知到并自动降级：

```java title="熔断降级方案"
@Service
public class CircuitBreakerCacheService {
    
    private volatile boolean circuitOpen = false;
    private AtomicInteger failureCount = new AtomicInteger(0);
    private static final int FAILURE_THRESHOLD = 10;
    
    public Product getProduct(Long id) {
        String cacheKey = "product:" + id;
        String lockKey = "lock:product:" + id;
        
        // 熔断开启时，直接查库
        if (circuitOpen) {
            return database.findById(id);
        }
        
        // 先查缓存
        Product cached = redisTemplate.opsForValue().get(cacheKey);
        if (cached != null) {
            return cached;
        }
        
        try {
            RLock lock = redissonClient.getLock(lockKey);
            boolean locked = lock.tryLock(1, TimeUnit.SECONDS);
            
            if (!locked) {
                failureCount.incrementAndGet();
                
                // 熔断阈值触发
                if (failureCount.get() >= FAILURE_THRESHOLD) {
                    circuitOpen = true;
                    log.error("熔断开启：锁服务连续失败{}次", FAILURE_THRESHOLD);
                }
                
                // 降级：直接查库
                return database.findById(id);
            }
            
            try {
                return loadAndCache(id);
            } finally {
                lock.unlock();
                failureCount.set(0);  // 成功后重置计数
            }
        } catch (Exception e) {
            failureCount.incrementAndGet();
            // 降级：直接查库
            return database.findById(id);
        }
    }
}
```

### 7.3 降级后的恢复

熔断不是永久的，需要有自动恢复机制：

```java
@Scheduled(fixedRate = 30_000)  // 每30秒检查一次
public void checkCircuitBreaker() {
    if (circuitOpen) {
        try {
            // 尝试获取锁
            RLock testLock = redissonClient.getLock("circuit:test");
            if (testLock.tryLock(1, TimeUnit.SECONDS)) {
                testLock.unlock();
                circuitOpen = false;
                failureCount.set(0);
                log.info("熔断恢复");
            }
        } catch (Exception e) {
            log.debug("熔断恢复检查失败");
        }
    }
}
```

## 八、总结

分布式锁 + 缓存的组合，是高并发系统中的「瑞士军刀」。

| 组件 | 职责 | 核心关注点 |
|---|---|---|
| Redis | 缓存层 + 锁存储 | 高可用、持久化 |
| 分布式锁 | 互斥访问 | 原子性、防死锁、可重入 |
| 限流 | 保护数据库 | 算法选择、阈值配置 |
| 熔断 | 系统保护 | 触发条件、恢复机制 |

三个核心原则：

1. **缓存是主战场**：尽可能让请求命中缓存，减少锁竞争
2. **锁是辅助工具**：只在必要时使用，不滥用
3. **降级是最后防线**：任何组件都可能故障，系统必须能优雅降级
