# 高并发设计：系统在高请求量下的稳定运行能力

> [!WARNING]
> 前置知识：本篇文章是高并发模块的入口，涵盖并发基础、缓存架构、异步消息、分布式并发四大方向。建议按顺序阅读，逐步构建高并发知识体系。

---

凌晨三点，你被一条报警短信惊醒——线上系统出现大面积超时。打开监控大盘，QPS 曲线笔直冲向天花板，接口响应时间从正常的 50ms 飙到了 10 秒。运维紧急扩容了两台机器，但于事无补——因为瓶颈根本不在机器数量，而在某个你没注意到的并发细节上。

这种场景，在高并发系统中极为常见。**系统扛不住高并发，往往不是因为硬件不够，而是架构和代码中的并发设计存在缺陷。**

高并发，是 Java 面试的核心主题之一。一个候选人对高并发的理解深度，直接决定了他能匹配什么级别的岗位。初级工程师会「用」，中级工程师会「调优」，高级工程师会「设计」。而面试官想看的，正是你处于哪个层次。

这一讲，我们来构建高并发知识的完整框架。

这四个方向的关系，可以这样理解：

- **并发基础是地基**：线程池、锁、CAS、ThreadLocal，这些是所有并发场景的底层支撑。不懂这些，就像不懂 HTTP 协议却想做好 Web 开发。
- **缓存架构是支撑**：高并发系统的性能瓶颈往往在数据库。缓存是减轻数据库压力的核心手段，也是面试中提问频率最高的模块之一。

- **分布式是扩展**：当单机无法满足需求时，需要扩展到多机。分布式带来了 CAP 定理、一致性、分布式事务等新问题。

## 二、并发基础优化：线程安全的底层逻辑

并发基础优化是高并发模块的「内功」。很多人觉得并发基础就是背八股文，但真正理解这些内容，才能在生产环境中排查问题、做出正确的技术选型。

### 2.1 线程池调优

线程池是 Java 并发编程中最核心的工具之一，但大多数人对它只停留在「会用」的程度。

当你提交一个任务到线程池，线程池会经历这样的决策过程：先看核心线程数满了没，没满就创建新线程；满了就看看队列满了没，没满就入队；队列满了再看最大线程数满了没，没满就创建临时线程；直到所有资源都耗尽，执行拒绝策略。

```java
// 线程池的七参数
public ThreadPoolExecutor(
    int corePoolSize,      // 核心线程数
    int maximumPoolSize,   // 最大线程数
    long keepAliveTime,    // 空闲存活时间
    TimeUnit unit,        // 时间单位
    BlockingQueue<Runnable> workQueue,   // 任务队列
    ThreadFactory threadFactory,         // 线程工厂
    RejectedExecutionHandler handler     // 拒绝策略
)
```

CPU 密集型任务和 IO 密集型任务的配置完全不同。CPU 密集型任务主要消耗 CPU 资源，线程数设置为 `CPU 核心数 + 1` 即可；IO 密集型任务大部分时间在等待 IO，线程数可以设置更大，通常是 `CPU 核心数 × 2` 或更多。

### 2.2 锁优化：synchronized 的进化史

很多人一提到锁就想到 synchronized 性能差。但从 JDK 6 开始，JVM 团队花了整整三年优化 synchronized，引入了偏向锁、轻量级锁、重量级锁的概念，实现了**锁升级**机制。

|synchronized| 的锁升级是一个**不可逆**的过程。从偏向锁一路升级到重量级锁后，即使锁释放了，下次加锁时也不会回到初始状态，而是直接从重量级锁开始。

锁优化的核心不是选择哪种锁实现，而是**控制锁的粒度**。分段锁（ConcurrentHashMap 的实现）、读写锁（ReadWriteLock）、写时复制（CopyOnWriteArrayList），都是针对特定场景的锁优化方案。

### 2.3 CAS：乐观锁的基石

CAS（Compare-And-Swap）是乐观锁的底层实现。Java 中的 `AtomicInteger`、`AtomicLong`、`AtomicReference` 都是基于 CAS 实现的。

CAS 的优势是无锁编程，性能远高于有锁方案。但它有三个必须掌握的问题：

- **ABA 问题**：线程1读取到 A，线程2将 A 改为 B 再改回 A，线程1的 CAS 操作会成功，但实际数据已经被修改过。解决方案是使用 `AtomicStampedReference` 或 `AtomicMarkableReference`。
- **自旋开销**：如果 CAS 操作失败，线程会一直自旋等待，高竞争场景下会消耗大量 CPU。
- **原子性问题**：CAS 只能保证单个变量的原子性，多个变量需要使用 `AtomicIntegerFieldUpdater` 等工具类或直接加锁。

### 2.4 ThreadLocal：线程封闭的陷阱

ThreadLocal 为每个线程提供独立的变量副本，实现了线程级别的数据隔离。但它有一个致命陷阱——**内存泄漏**。

ThreadLocal 的实现是每个线程持有一个 `ThreadLocalMap`，Entry 的 key 是 WeakReference 指向 ThreadLocal 对象。当 ThreadLocal 变量不再被引用时，Entry 的 key 会被 GC 回收，value 却仍然被 Entry 持有，导致内存泄漏。

正确使用 ThreadLocal 的方式是：在使用完毕后调用 `remove()` 方法，而不是依赖 `get()` 返回 null 来判断。

## 三、缓存架构：高并发系统的性能支柱

缓存架构是高并发面试的重中之重。一个没有缓存的系统，在高并发场景下数据库会首先崩溃。

### 3.1 多级缓存：层层拦截请求

真正的生产级缓存系统，不会只依赖 Redis，而是构建**多级缓存**：

- **L1 本地缓存**：Caffeine、Guava Cache，访问延迟极低（纳秒级），但容量有限
- **L2 分布式缓存**：Redis，容量大，支持分布式共享，但有网络延迟
- **L3 数据库**：最终的真相之源

请求到来时，先查 L1，未命中查 L2，再未命中查数据库。多级缓存的目标是让 90% 以上的请求在 L1 层就返回。

### 3.2 缓存策略：读写模式的选择

缓存有四种经典读写模式：

| 模式 | 读 | 写 | 一致性 | 复杂度 |
|---|---|---|---|---|
| Cache Aside | 先缓存，后数据库 | 先数据库，后删缓存 | 弱一致 | 低 |
| Read Through | 缓存代理读操作 | 同 Cache Aside | 弱一致 | 中 |
| Write Through | 同步写缓存和数据库 | 先缓存，后数据库 | 强一致 | 高 |
| Write Behind | 只写缓存，异步写库 | 只写缓存 | 最终一致 | 高 |

大多数业务场景使用 Cache Aside 模式，因为它实现简单，在「读多写少」场景下效果最好。

### 3.3 缓存三大经典问题

缓存世界里有三个永恒的话题：**缓存穿透**、**缓存击穿**、**缓存雪崩**。

**缓存穿透**：大量请求查询一个不存在的数据，缓存中没有，数据库中也没有，请求直接打到数据库。解决方案是布隆过滤器或缓存空值。

**缓存击穿**：一个热点 key 过期瞬间，大量请求同时涌入，全部打到数据库。解决方案是互斥锁、逻辑过期双检、永久不过期加异步更新。

**缓存雪崩**：大量缓存 key 同时过期，导致大量请求同时击穿到数据库。解决方案是过期时间随机化、多级缓存、熔断降级。

### 3.4 Redis 进阶：LRU、LFU 与过期策略

Redis 的内存管理是面试高频考点。

当 Redis 内存达到上限（`maxmemory`）时，需要淘汰数据。Redis 提供了 8 种淘汰策略：

- **noeviction**：不淘汰，内存满时返回错误
- **allkeys-lru/allkeys-random**：所有 key 中 LRU 淘汰或随机淘汰
- **volatile-lru/volatile-random/volatile-ttl**：只淘汰带过期时间的 key

LRU（Least Recently Used）是最常用的策略。但 Redis 实现的是**近似 LRU**，不是精确 LRU。近似 LRU 通过抽样而非全量排序来选择淘汰对象，在性能和精度之间做了权衡。

LFU（Least Frequently Used）是 Redis 4.0 引入的新策略，按访问频率淘汰。LFU 的难点是如何处理「访问频率衰减」——一个曾经热门的 key 很久没被访问了，应该降低其权重。

## 四、异步与消息处理：抗高峰的核心武器

同步调用是高并发系统最大的性能杀手。当系统负载升高时，同步调用会导致级联超时，最终整个系统雪崩。异步处理是解决这个问题的根本方案。

### 4.1 异步编程：从回调地狱到 CompletableFuture

Java 8 引入的 `CompletableFuture` 是异步编程的利器。它支持流式的任务编排，可以优雅地处理依赖关系：

```java
CompletableFuture.supplyAsync(() -> queryUser(userId))
    .thenCompose(user -> queryOrders(user.getId()))
    .thenCombine(CompletableFuture.supplyAsync(() -> queryRecommendations()),
        (orders, recommendations) -> new UserDashboard(user, orders, recommendations))
    .thenAccept(dashboard -> render(dashboard));
```

但 CompletableFuture 只是本地异步。要实现真正的异步解耦，需要消息队列。

### 4.2 消息队列：削峰填谷的利器

消息队列的核心价值有三个：

- **削峰**：将突发的流量洪峰缓存为队列，平稳处理
- **解耦**：生产者和消费者解耦，互不感知
- **异步**：将同步调用改为异步，提升响应速度

Kafka 是高并发场景的首选。它的核心优势是**顺序写磁盘**和**零拷贝**，单节点就能达到百万级 QPS。但 Kafka 的复杂度也更高，需要配置合理的分区数、消费者组、offset 管理策略。

### 4.3 消息可靠性：面试超高频问题

「如何保证消息不丢失？」是面试必问题。

消息的传递有三个环节，每个环节都需要确认：

- **生产者到 Broker**：使用确认机制（acks），`acks=all` 表示所有副本都写入成功才确认
- **Broker 自身**：配置合理的持久化策略，RDB + AOF 混合持久化
- **Broker 到消费者**：消费者手动 ACK，确认处理完成后再提交 offset

但可靠性是有代价的。高可靠性配置会降低吞吐量，需要根据业务场景权衡。

## 五、分布式高并发核心：突破单机瓶颈

当单机无法满足业务需求时，需要扩展到多机。分布式带来了 CAP 定理、一致性问题、分布式事务等新挑战。

### 5.1 分布式锁：跨机器的互斥

单机环境下的 synchronized 或 ReentrantLock 无法跨 JVM 工作。分布式锁是解决跨节点互斥的唯一方案。

Redis 分布式锁是最常用的实现方式：

```java
// 简单的 Redis 分布式锁
String lockKey = "order:lock:" + orderId;
String lockValue = UUID.randomUUID().toString();

// SET key value NX PX milliseconds
Boolean locked = redis.set(lockKey, lockValue, 30, TimeUnit.SECONDS, "NX");
```

但这个简单实现有致命问题——**锁续期**。如果持有锁的进程在锁过期前崩溃，锁会自动释放，导致其他进程也能获取锁。Redisson 通过看门狗机制（Watchdog）自动续期，解决了这个问题。

### 5.2 分布式事务：跨节点的数据一致性

分布式事务是分布式系统中最复杂的问题之一。CAP 定理告诉我们，一致性、可用性、分区容忍性无法同时满足。分布式事务的解决方案，本质上是在一致性和可用性之间做权衡。

常见的分布式事务方案：

| 方案 | 一致性强度 | 性能 | 复杂度 |
|---|---|---|---|
| 两阶段提交（2PC） | 强一致 | 低 | 高 |
| 三阶段提交（3PC） | 强一致 | 中 | 高 |
| TCC | 最终一致 | 中 | 高 |
| 本地消息表 | 最终一致 | 高 | 中 |
| Seata AT | 最终一致 | 高 | 中 |
| Saga | 最终一致 | 高 | 中 |

Seata 是目前最流行的分布式事务解决方案，其中 AT 模式对业务零侵入，是大多数场景的首选。

### 5.3 分库分表：水平扩展的终极方案

当单库数据量超过千万级时，需要考虑分库分表。分库分表带来了新的问题：跨表查询、跨表分页、分片键选择。

分片键的选择至关重要。一旦选择，就很难更改。常见的分片键有用户 ID（适合用户相关查询）、时间（适合时序数据）、地区（适合地理位置相关）。

## 六、高并发面试的核心考察点

面试中对高并发的考察，核心看三个层次：

### 6.1 问题排查能力

面试官喜欢问：「生产环境 CPU 突然飙高，你怎么排查？」

这个问题考察的是你解决问题的系统性思维：

1. **定位问题**：先用 `top` 或 `jstack` 确认是哪个进程、哪个线程
2. **分析原因**：如果是 GC 问题，分析 GC 日志；如果是死循环，分析代码逻辑
3. **解决问题**：从表象到根因，对症下药

### 6.2 方案设计能力

「如何设计一个高并发的秒杀系统？」是经典的系统设计题。

这道题考察的是你对高并发各个知识点的综合运用能力：

- 库存扣减：分布式锁 + 数据库乐观锁
- 流量拦截：多层限流（入口限流、接口限流、服务限流）
- 缓存设计：热点数据预热 + 多级缓存
- 异步处理：消息队列削峰 + 异步下单流程

### 6.3 踩坑经验

面试官真正想看到的，是你有没有在生产环境中真正解决过问题。「遇到过什么印象深刻的技术挑战？」这类问题，是在考察你的实战经验。

## 七、复习路径建议

高并发知识的复习，建议分三个阶段：

### 第一阶段：打牢并发基础

优先掌握线程池调优、synchronized 锁升级、CAS 原理、ThreadLocal。这四个知识点是所有高并发场景的底层支撑，必须理解透彻。

缓存是面试高频区，建议按这个顺序学习：

1. 先理解多级缓存架构和读写模式
2. 再掌握缓存三大经典问题及解决方案

3. 分库分表的设计思路

## 延伸思考

高并发系统的设计，本质上是在**性能**、**一致性**、**可用性**三者之间做权衡。

没有任何系统能做到三者同时最优。你能做的，是根据业务场景，找到当前阶段的最优解。

比如，电商秒杀系统优先保证可用性（宁可超卖，不能宕机）；金融支付系统优先保证一致性（宁可拒绝，不能出错）。

下次设计高并发方案之前，先问自己三个问题：业务能容忍多少不一致？系统能接受多高的延迟？故障时能承受多大损失？

答案决定了你的技术选型。
