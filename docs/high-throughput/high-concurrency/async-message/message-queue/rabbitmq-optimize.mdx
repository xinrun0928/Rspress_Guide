# RabbitMQ 高级特性与性能优化

## 场景切入：双十一的库存危机

凌晨 0 点，双十一抢购开始。

用户的请求像潮水一样涌来：下单、支付、扣库存、核销优惠券。系统运转良好，订单量突破历史新高。

然而，10 分钟后，客服电话被打爆了。

"用户投诉：订单显示已付款，但库存显示还有，根本没扣！"

你赶紧查日志，发现问题所在：订单服务和库存服务之间通过 RabbitMQ 异步通信，但在高并发下，消息的路由出现了混乱——优惠券消息、库存消息、订单消息互相干扰，导致处理逻辑错乱。

更糟糕的是，部分消息处理失败后直接丢失了，用户付款成功但业务没有闭环。

这不是 RabbitMQ 的 bug，而是**你没有正确使用 RabbitMQ 的高级特性**。

本文将深入讲解 RabbitMQ 的核心概念、高级特性和性能优化技巧，让你在高并发场景下真正用好 RabbitMQ。

## 一、RabbitMQ 核心概念

### 1.1 消息模型

RabbitMQ 基于 AMQP（Advanced Message Queuing Protocol）协议，采用**生产者-交换机-队列-消费者**的消息模型：

```
┌──────────┐    ┌───────────┐    ┌────────┐    ┌───────────┐    ┌──────────┐
│ Producer │───▶│ Exchange  │───▶│ Queue  │◀───│ Consumer  │    │ Consumer │
│  生产者  │    │  交换机   │    │  队列  │    │  消费者 A │    │  消费者B │
└──────────┘    └───────────┘    └────────┘    └───────────┘    └──────────┘
                       │
                       ▼
                ┌────────────┐
                │  Binding   │  ← 绑定规则：交换机如何将消息路由到队列
                └────────────┘
```

### 1.2 核心组件

**Producer（生产者）**：发送消息的应用，不直接将消息发送到队列，而是发送到交换机。

**Exchange（交换机）**：接收生产者消息，根据绑定规则将消息路由到一个或多个队列。交换机本身不存储消息。

**Queue（队列）**：存储消息的容器，遵循 FIFO 原则。消息在队列中等待被消费者拉取。

**Binding（绑定）**：定义交换机和队列之间的关系，包含路由键（Routing Key）。

**Consumer（消费者）**：从队列获取消息并处理的应用。

### 1.3 消息流转

一条消息在 RabbitMQ 中的完整生命周期：

```
1. Producer 发送消息到 Exchange
2. Exchange 根据类型和 Binding 规则决定消息路由到哪些 Queue
3. Queue 存储消息，等待 Consumer 拉取
4. Consumer 从 Queue 获取消息并处理
5. Consumer 发送 ACK 确认消息已被处理
6. Queue 删除已确认的消息
```

## 二、Exchange 的四种类型

### 2.1 direct 交换机

**工作原理**：完全匹配路由键。一个队列绑定一个路由键，消息必须精确匹配才能被路由。

```
┌───────────┐
│  Exchange │ ──routing_key="order.created"──▶ Queue: order-queue
│  direct   │                                 
└───────────┘

消息 1: routing_key="order.created" → ✓ 路由到 order-queue
消息 2: routing_key="order.paid"    → ✗ 没有匹配的 Binding，消息被丢弃
```

**适用场景**：点对点消息传递，一个消息对应一个处理逻辑。

**代码示例**：

```java
// 声明交换机
channel.exchangeDeclare("order-exchange", "direct", true);

// 声明队列
channel.queueDeclare("order-queue", true, false, false, null);

// 绑定交换机和队列
channel.queueBind("order-queue", "order-exchange", "order.created");
channel.queueBind("order-queue", "order-exchange", "order.paid");

// 发送消息
channel.basicPublish("order-exchange", "order.created", 
    MessageProperties.PERSISTENT_TEXT_PLAIN,
    "订单创建".getBytes());
```

### 2.2 fanout 交换机

**工作原理**：忽略路由键，将消息广播到所有绑定的队列。每个绑定的队列都会收到消息的副本。

```
┌───────────┐
│  Exchange │ ──▶ Queue: queue-A
│  fanout   │ ──▶ Queue: queue-B
└───────────┘ ──▶ Queue: queue-C

所有队列都收到消息的完整副本
```

**适用场景**：发布-订阅模式，如系统通知、广播消息。

**代码示例**：

```java
// 声明 fanout 交换机
channel.exchangeDeclare("notification-exchange", "fanout", true);

// 声明多个队列（不同的消费者）
channel.queueDeclare("email-queue", true, false, false, null);
channel.queueDeclare("sms-queue", true, false, false, null);
channel.queueDeclare("push-queue", true, false, false, null);

// 绑定队列到交换机（不需要 routing key）
channel.queueBind("email-queue", "notification-exchange", "");
channel.queueBind("sms-queue", "notification-exchange", "");
channel.queueBind("push-queue", "notification-exchange", "");

// 发送消息，所有绑定的队列都会收到
channel.basicPublish("notification-exchange", "", 
    MessageProperties.PERSISTENT_TEXT_PLAIN,
    "系统通知".getBytes());
```

### 2.3 topic 交换机

**工作原理**：通配符匹配路由键。支持两种通配符：
- `*`（星号）：匹配一个单词
- `#`（井号）：匹配零个或多个单词

```
┌───────────┐
│  Exchange │ ──routing_key="order.created"──▶ Queue: order-queue
│   topic   │ ──routing_key="order.*"────────▶ Queue: all-order-queue
└───────────┘ ──routing_key="order.#"────────▶ Queue: order-logs
               ──routing_key="user.created"──▶ (无匹配)

Routing Key 示例：
- "order.created"   → 匹配 "order.*" 和 "order.#"
- "order.paid"      → 匹配 "order.*" 和 "order.#"
- "order.log.info"  → 匹配 "order.#"（但不匹配 "order.*"）
```

**适用场景**：灵活的路由规则，支持多级分类。

**代码示例**：

```java
// 声明 topic 交换机
channel.exchangeDeclare("event-exchange", "topic", true);

// 绑定不同的路由规则
channel.queueBind("order-queue", "event-exchange", "order.*");      // 订单相关
channel.queueBind("user-queue", "event-exchange", "user.*");        // 用户相关
channel.queueBind("all-log-queue", "event-exchange", "#.log");      // 所有日志
channel.queueBind("order-log-queue", "event-exchange", "order.log.#"); // 订单日志

// 发送消息
channel.basicPublish("event-exchange", "order.created", null, "创建订单".getBytes());
channel.basicPublish("event-exchange", "user.login", null, "用户登录".getBytes());
channel.basicPublish("event-exchange", "order.log.error", null, "订单错误".getBytes());
```

### 2.4 headers 交换机

**工作原理**：根据消息头部的属性匹配，而不是路由键。较少使用，但在某些特殊场景下有用。

```
┌───────────┐
│  Exchange │ ──headers={"format":"pdf"}──────▶ Queue: pdf-queue
│  headers  │ ──headers={"format":"excel"}───▶ Queue: excel-queue
└───────────┘ ──headers={"x-match":"all",────▶ Queue: all-queue
                            "format":"pdf",
                            "type":"report"}
```

**x-match 参数**：
- `x-match=all`：所有头部属性必须匹配
- `x-match=any`：任意一个头部属性匹配即可

## 三、消息确认机制

### 3.1 自动确认 vs 手动确认

RabbitMQ 支持两种确认模式：

**自动确认（autoAck=true）**：消息一投递给消费者就自动确认，不管消费者是否真正处理完成。

```java
// 自动确认模式
channel.basicConsume("order-queue", true, deliverCallback, consumerTag -> {});

// 问题：消息投递给消费者后立即删除
// 如果消费者处理失败，消息就丢失了
```

**手动确认（autoAck=false）**：消费者必须显式调用 `basicAck` 确认消息处理成功，或者 `basicNack/basicReject` 拒绝消息。

```java
// 手动确认模式
channel.basicConsume("order-queue", false, deliverCallback, consumerTag -> {});

DeliverCallback deliverCallback = (consumerTag, delivery) -> {
    String message = new String(delivery.getBody());
    try {
        // 业务处理
        processMessage(message);
        // 手动确认
        channel.basicAck(delivery.getEnvelope().getDeliveryTag(), false);
    } catch (Exception e) {
        // 处理失败，拒绝消息
        // requeue=true：重新入队，等待下次处理
        // requeue=false：丢弃消息或发送到死信队列
        channel.basicNack(delivery.getEnvelope().getDeliveryTag(), false, false);
    }
};
```

### 3.2 确认参数详解

```java
// basicAck 参数
channel.basicAck(deliveryTag, multiple);

// deliveryTag: 消息的唯一标识
// multiple: 
//   false - 只确认当前这条消息
//   true  - 确认所有小于等于 deliveryTag 的消息

// basicNack 参数
channel.basicNack(deliveryTag, multiple, requeue);

// requeue:
//   true  - 重新入队，消息会重新投递给其他消费者
//   false - 消息被丢弃或进入死信队列

// basicReject 参数（等同于 basicNack 单条）
channel.basicReject(deliveryTag, requeue);
```

### 3.3 最佳实践

**处理一条确认一条**：

```java
channel.basicConsume("order-queue", false, (consumerTag, delivery) -> {
    try {
        processOrder(new String(delivery.getBody()));
        channel.basicAck(delivery.getEnvelope().getDeliveryTag(), false);
    } catch (Exception e) {
        // 业务异常，尝试重试
        if (isRetryable(e)) {
            // 重新入队，重新投递给当前消费者
            channel.basicNack(delivery.getEnvelope().getDeliveryTag(), false, true);
        } else {
            // 非业务异常，进入死信队列
            channel.basicNack(delivery.getEnvelope().getDeliveryTag(), false, false);
        }
    }
}, consumerTag -> {});
```

## 四、Publisher Confirms（生产者确认）

### 4.1 为什么需要生产者确认

消息发送到 RabbitMQ 后，生产者需要确认消息是否成功到达 Broker。如果没有确认机制：

- 消息可能在网络传输中丢失
- Broker 可能存储失败
- 生产者不知道消息是否真正被消费

### 4.2 开启 Publisher Confirms

```java
// 开启确认模式
channel.confirmSelect();

// 发送消息
channel.basicPublish("order-exchange", "order.created",
    MessageProperties.PERSISTENT_TEXT_PLAIN,
    "订单消息".getBytes());

// 等待确认（同步方式）
boolean success = channel.waitForConfirms();
if (!success) {
    // 确认失败，消息可能丢失，需要重试
    retrySend();
}
```

### 4.3 异步确认

同步等待会降低吞吐量，可以使用异步确认：

```java
channel.confirmSelect();

// 设置监听器
channel.addConfirmListener((deliveryTag, multiple) -> {
    // 消息确认成功
    System.out.println("消息确认成功: " + deliveryTag);
}, (deliveryTag, multiple) -> {
    // 消息确认失败
    System.out.println("消息确认失败: " + deliveryTag);
    // 需要重发消息
    retrySend(deliveryTag);
});

// 发送消息
for (int i = 0; i < 1000; i++) {
    channel.basicPublish("order-exchange", "order.created",
        MessageProperties.PERSISTENT_TEXT_PLAIN,
        ("消息 " + i).getBytes());
}
```

### 4.4 Publisher Returns

当消息无法路由到任何队列时，RabbitMQ 会回调 `ReturnsListener`：

```java
// 开启 Returns
channel.addReturnListener((replyCode, replyText, exchange, routingKey, properties, body) -> {
    System.out.println("消息无法路由: " + new String(body));
    // 处理无法路由的消息
    handleUnroutableMessage(body);
});

// 发送消息时设置 mandatory=true
// 如果消息无法路由，Broker 会返回而不是丢弃
channel.basicPublish("order-exchange", "order.unknown",
    true,  // mandatory: 必须路由到队列
    MessageProperties.PERSISTENT_TEXT_PLAIN,
    "无法路由的消息".getBytes());
```

## 五、死信队列（DLX）

### 5.1 什么是死信队列

死信队列（Dead Letter Queue，DLX）用于存放**无法正常处理的消息**：

- 消息被消费者拒绝（`basicNack/basicReject`）且不重新入队
- 消息超过TTL（生存时间）未被消费
- 队列达到最大长度，新消息被丢弃

### 5.2 配置死信队列

```java
// 创建死信交换机和死信队列
channel.exchangeDeclare("dlx-exchange", "direct", true);
channel.queueDeclare("dlx-queue", true, false, false, null);
channel.queueBind("dlx-queue", "dlx-exchange", "dlx-routing-key");

// 创建普通队列，并配置死信交换机
Map<String, Object> args = new HashMap<>();
args.put("x-dead-letter-exchange", "dlx-exchange");
args.put("x-dead-letter-routing-key", "dlx-routing-key");

channel.queueDeclare("order-queue", true, false, false, args);

// 生产者发送消息
channel.basicPublish("order-exchange", "order.created",
    MessageProperties.PERSISTENT_TEXT_PLAIN,
    "订单消息".getBytes());
```

### 5.3 死信的产生场景

**场景一：消息被拒绝**

```java
@Bean
public SimpleRabbitListenerContainerFactory rabbitListenerContainerFactory() {
    SimpleRabbitListenerContainerFactory factory = new SimpleRabbitListenerContainerFactory();
    factory.setConnectionFactory(connectionFactory);
    factory.setDefaultRequeueRejected(false);  // 拒绝后不重新入队，进入死信队列
    return factory;
}
```

**场景二：消息 TTL 过期**

```java
// 创建队列，设置消息 TTL 为 30 秒
Map<String, Object> args = new HashMap<>();
args.put("x-message-ttl", 30000);  // 30 秒

channel.queueDeclare("order-queue", true, false, false, args);

// 或者在消息上设置 TTL
AMQP.BasicProperties properties = new AMQP.BasicProperties.Builder()
    .expiration("60000")  // 60 秒
    .build();

channel.basicPublish("order-exchange", "order.created", properties, "订单消息".getBytes());
```

**场景三：队列满了**

```java
// 创建队列，设置最大长度
Map<String, Object> args = new HashMap<>();
args.put("x-max-length", 10000);  // 最多存储 10000 条消息
args.put("x-overflow", "reject-publish");  // 新消息拒绝发布到队列（而不是丢弃旧消息）

channel.queueDeclare("order-queue", true, false, false, args);
```

### 5.4 死信队列的处理

```java
// 消费死信队列
channel.basicConsume("dlx-queue", false, (consumerTag, delivery) -> {
    String message = new String(delivery.getBody());
    
    // 查看原始信息
    String originalExchange = delivery.getProperties()
        .getHeaders().get("x-first-death-exchange").toString();
    String originalRoutingKey = delivery.getProperties()
        .getHeaders().get("x-first-death-reason").toString();
    
    System.out.println("死信消息: " + message);
    System.out.println("原始交换机: " + originalExchange);
    System.out.println("死信原因: " + originalRoutingKey);
    
    // 人工处理或补偿
    handleDeadLetter(message);
    
    channel.basicAck(delivery.getEnvelope().getDeliveryTag(), false);
}, consumerTag -> {});
```

## 六、优先级队列

### 6.1 应用场景

优先级队列用于处理**紧急消息优先处理**的场景：

- 会员用户的请求优先处理
- 系统告警优先于普通日志
- 高优先级订单优先发货

### 6.2 配置优先级队列

```java
// 创建最大优先级为 10 的队列
Map<String, Object> args = new HashMap<>();
args.put("x-max-priority", 10);  // 最大优先级 10

channel.queueDeclare("priority-order-queue", true, false, false, args);

// 发送高优先级消息
AMQP.BasicProperties properties = new AMQP.BasicProperties.Builder()
    .priority(10)  // 优先级 0-10
    .build();

channel.basicPublish("order-exchange", "order.priority", properties, 
    "高优先级订单".getBytes());

// 发送普通优先级消息
channel.basicPublish("order-exchange", "order.normal", null, 
    "普通订单".getBytes());
```

### 6.3 注意事项

- 优先级队列需要所有消费者都在线才能生效
- 消息过多时会按优先级排队，高优先级消息会先被消费
- 内存占用较高，不建议设置过大的优先级范围

## 七、延迟队列

### 7.1 延迟队列的应用场景

延迟队列用于**消息在指定时间后才被消费**：

- 订单超时未支付自动取消
- 失败任务延迟重试
- 定时任务触发

### 7.2 实现方式一：延迟插件

RabbitMQ 3.8+ 支持 `rabbitmq-delayed-message-exchange` 插件：

```bash
# 安装延迟插件
rabbitmq-plugins enable rabbitmq_delayed_message_exchange
```

```java
// 声明延迟交换机
Map<String, Object> args = new HashMap<>();
args.put("x-delayed-type", "direct");

channel.exchangeDeclare("delayed-exchange", "x-delayed-message", true, args, args);

// 发送延迟消息（延迟 30 秒）
Map<String, Object> headers = new HashMap<>();
headers.put("x-delay", 30000);  // 延迟 30 秒（毫秒）

AMQP.BasicProperties properties = new AMQP.BasicProperties.Builder()
    .headers(headers)
    .build();

channel.basicPublish("delayed-exchange", "order.cancel", properties, 
    "取消订单".getBytes());
```

### 7.3 实现方式二：TTL + 死信队列

利用消息 TTL 过期后进入死信队列的特性：

```java
// 创建延迟队列（TTL 为 30 秒）
Map<String, Object> args = new HashMap<>();
args.put("x-message-ttl", 30000);        // 消息 TTL
args.put("x-dead-letter-exchange", "order-exchange");  // 死信交换机
args.put("x-dead-letter-routing-key", "order.cancel"); // 死信路由键

channel.queueDeclare("delay-order-queue", true, false, false, args);
channel.queueBind("delay-order-queue", "order-exchange", "order.delay");

// 发送消息到延迟队列
channel.basicPublish("order-exchange", "order.delay", null, 
    "订单消息".getBytes());

// 30 秒后，消息过期，进入死信队列
// 消费者从 order.cancel 队列消费，实现延迟处理
```

### 7.4 两种方式对比

| 方式 | 优点 | 缺点 |
|---|---|---|
| 延迟插件 | 支持任意延迟时间，精度高 | 需要安装额外插件 |
| TTL + DLX | 无需插件，配置简单 | 延迟时间受队列 TTL 限制 |

## 八、集群模式与高可用

### 8.1 普通集群

普通集群（Default Cluster）在多个节点间共享队列定义，但**队列只存在于一个节点**：

```
Cluster:
  Node A: 队列定义 + 队列数据
  Node B: 队列定义（无数据）
  Node C: 队列定义（无数据）
  
消费者连接到 Node B，只能获取到队列定义
消息存储在 Node A
```

**特点**：
- 队列数据只存在于创建队列的节点
- 其他节点只存储队列定义和元数据
- 消费者需要路由到正确的节点才能获取消息

### 8.2 镜像集群

镜像集群（Mirrored Queue）将队列复制到所有集群节点：

```
Cluster:
  Node A: 主节点（master）
  Node B: 镜像节点（slave）- 同步复制
  Node C: 镜像节点（slave）- 同步复制
  
所有节点都有完整的队列数据
```

**配置镜像策略**：

```bash
# 同步所有队列到集群内所有节点
rabbitmqctl set_policy ha-all "^" '{"ha-mode":"all"}'

# 同步指定前缀的队列
rabbitmqctl set_policy ha-orders "^order\." '{"ha-mode":"exactly","ha-params":2}'

# 配置参数说明
rabbitmqctl set_policy <name> <pattern> <definition> [priority]
# ha-mode: all | exactly | nodes
# ha-sync-mode: manual | automatic
```

**Java 配置**：

```java
// 设置 HA 策略
Map<String, Object> policyArgs = new HashMap<>();
policyArgs.put("ha-mode", "all");
policyArgs.put("ha-sync-mode", "automatic");

AMQP.Exchange.DeclareOk exchange = channel.exchangeDeclare(
    "ha-exchange", "direct", true);
```

### 8.3 集群高可用配置

**节点类型**：
- RAM Node：元数据存储在内存，性能高但数据不安全
- Disk Node：元数据存储在磁盘，数据安全

**建议配置**：至少 2 个磁盘节点 + 1 个 RAM 节点。

**故障转移**：
- 当 master 节点故障时，最老的 slave 自动晋升为 master
- 晋升期间短暂不可用
- 建议使用负载均衡器（如 HAProxy）屏蔽节点故障

## 九、消费端限流

### 9.1 prefetch 机制

`prefetch` 控制 Broker 一次性投递给消费者的消息数量：

```java
// 设置 prefetch = 1，每次只投喂 1 条消息
channel.basicQos(1);

// 消费消息
channel.basicConsume("order-queue", false, (consumerTag, delivery) -> {
    processOrder(new String(delivery.getBody()));
    channel.basicAck(delivery.getEnvelope().getDeliveryTag(), false);
}, consumerTag -> {});
```

**prefetch = 1** 的效果：

```
Broker ──msg1──▶ Consumer（处理中...）
          │                  │
          │                  ▼
          │              processOrder()
          │                  │
          │                  ▼
          │◀───────────── ACK
          │
          └──msg2──▶ Consumer（处理中...）
```

### 9.2 限流配置建议

| 场景 | prefetch 值 | 说明 |
|---|---|---|
| 快速处理 | 10-50 | 批处理场景，允许预取多条 |
| 慢速处理 | 1 | 单条确认，确保处理完再获取下一条 |
| 混合场景 | 10-20 | 平衡吞吐量和控制能力 |

```java
// Spring AMQP 配置
@Bean
public SimpleRabbitListenerContainerFactory rabbitListenerContainerFactory() {
    SimpleRabbitListenerContainerFactory factory = new SimpleRabbitListenerContainerFactory();
    factory.setConnectionFactory(connectionFactory);
    factory.setPrefetchCount(10);  // 预取 10 条消息
    factory.setConcurrentConsumers(3);  // 3 个并发消费者
    return factory;
}
```

## 十、内存与磁盘告警

### 10.1 内存告警

RabbitMQ 使用内存超过阈值时会触发告警：

```bash
# 默认阈值为 40% 物理内存
# 可以调整阈值
rabbitmqctl set_vm_memory_high_watermark 0.5  # 设置为 50%
```

**告警时的行为**：
- 生产者被阻塞，无法发送消息
- 消费者继续正常消费
- 需要处理积压或增加内存

### 10.2 磁盘告警

磁盘空间不足时触发告警：

```bash
# 默认阈值为 50MB

```

**告警时的行为**：
- 生产者被阻塞
- 消息无法持久化
- 可能导致数据丢失

## 十一、性能优化总结

### 11.1 生产端优化

```java
// 1. 开启信道复用
Connection connection = connectionFactory.newConnection("producer-pool");
Channel channel = connection.createChannel();

// 2. 开启 Publisher Confirms
channel.confirmSelect();

// 3. 使用批量发送
int batchSize = 100;
for (int i = 0; i < 1000; i++) {
    channel.basicPublish("order-exchange", "order.created", null, 
        ("消息 " + i).getBytes());
    if (i % batchSize == 0) {
        channel.waitForConfirmsOrDie(5000);
    }
}

// 4. 使用事务（谨慎，高开销）
channel.txSelect();
channel.basicPublish("order-exchange", "order.created", null, "消息".getBytes());
channel.txCommit();
```

### 11.2 消费端优化

```java
// 1. 手动确认
channel.basicConsume("order-queue", false, callback);

// 2. 合理设置 prefetch
channel.basicQos(10);

// 3. 批量消费
channel.basicConsume("order-queue", false, (consumerTag, deliveries) -> {
    for (DeliverCallback.Delivery delivery : deliveries) {
        processOrder(new String(delivery.getBody()));
        channel.basicAck(delivery.getEnvelope().getDeliveryTag(), false);
    }
});

// 4. 多线程消费
```

## 面试追问预判

### 追问一：RabbitMQ 如何保证消息不丢失？

需要从三个环节考虑：

**生产者端**：
```java
// 1. 开启确认
channel.confirmSelect();

// 2. 等待确认
channel.waitForConfirms();

// 3. 失败重试
if (!success) {
    retrySend();
}
```

**Broker 端**：
```bash
# 队列持久化
rabbitmqctl set_policy ha-all "^" '{"ha-mode":"all"}'

# 磁盘持久化配置
vm_memory_high_watermark.relative = 0.6
```

**消费者端**：
```java
// 手动确认
channel.basicConsume("queue", false, callback);
channel.basicAck(deliveryTag, false);
```

### 追问二：RabbitMQ 和 Kafka 的区别？如何选择？

| 维度 | RabbitMQ | Kafka |
|---|---|---|
| 协议 | AMQP | 自定义协议 |
| 消息模型 | 队列 + 交换机 | 分区日志 |
| 顺序保证 | 单队列有序 | 分区内有序 |
| 吞吐量 | 万级 QPS | 百万级 QPS |
| 消息持久化 | 队列存储 | 追加写日志 |
| 消息回溯 | 不支持 | 支持（从 offset 回溯） |
| 多租户 | 支持 | 不支持（需独立集群） |

**选择建议**：
- 需要复杂路由 → RabbitMQ
- 需要高吞吐 + 消息回溯 → Kafka
- 需要事务支持 → RabbitMQ
- 需要消息堆积 → Kafka
