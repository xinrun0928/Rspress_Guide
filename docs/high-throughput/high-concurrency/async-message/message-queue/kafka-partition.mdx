# Kafka 分区数与消费者线程数的设置艺术

import { Badge } from '@rspress/core/theme';

> [!WARNING]
> 前置知识：本篇文章涉及 Kafka 的分区机制、消费者组、Rebalance 机制，建议先掌握 Kafka 的基本架构。

---

2018 年双十一，某电商平台的订单系统遇到了一个诡异的问题。

平时系统运行平稳，吞吐量稳定在每秒 5 万订单。但一到高峰期，消息消费严重滞后，订单处理延迟从平时的几百毫秒飙升到几分钟。用户付了钱，却迟迟看不到订单确认。

运维团队排查了整整两天，最后发现原因竟然是：**分区数设置少了**。

只有 8 个分区，却有 20 个消费者实例在消费。大部分消费者分不到分区，只能空转。而少数分到多个分区的消费者，被迫处理了远超预期的消息。

这就是 Kafka 分区数与消费者配置不当导致的典型问题。这一讲，我们来彻底掌握 Kafka 分区的设置艺术。

---

## 一、分区在 Kafka 中的核心作用

### 1.1 分区是 Kafka 并行处理的基石

分区（Partition）是 Kafka 实现**水平扩展**和**并行处理**的核心机制。

```
┌─────────────────────────────────────────────────────────────────────┐
│                         Kafka 分区架构                              │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  Topic: orders                                                      │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  Partition 0          Partition 1          Partition 2        │ │
│  │  [msg1, msg4, msg7]  [msg2, msg5, msg8]  [msg3, msg6, msg9]  │ │
│  │       │                    │                    │            │ │
│  │       ▼                    ▼                    ▼            │ │
│  │  Leader Broker 1      Leader Broker 2      Leader Broker 3   │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  分区分布在不同的 Broker 上，实现数据分片和负载均衡                    │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

每个 Topic 可以划分为多个 Partition，每个 Partition 可以分布在不同的 Broker 上。这种设计带来两个核心能力：

- **并行写入**：多个分区可以并行写入不同的 Broker，充分利用集群的 IO 能力
- **并行消费**：多个消费者可以并行消费不同的分区，实现消费能力水平扩展

### 1.2 分区与数据分布

Kafka 使用**哈希分区**或**轮询分区**来决定消息发送到哪个分区：

```java
// 指定分区发送
producer.send(new ProducerRecord<>("orders", 0, "key", "value"));
// 参数：topic, partition, key, value

// 不指定分区，由 Kafka 根据 key 哈希分配
producer.send(new ProducerRecord<>("orders", "key", "value"));
// 相同 key 的消息会被发送到相同分区

// 不指定 key，轮询分配
producer.send(new ProducerRecord<>("orders", "value"));
// 消息会轮询发送到各个分区
```

**为什么要有 key**：

- 相同 key 的消息保证有序（发送到同一分区）
- 适合需要保证顺序的业务场景（如订单）

**没有 key 的情况**：

- 消息轮询发送到各个分区
- 适合不需要顺序的场景（可以最大化并行度）

## 二、分区数与生产者并发

### 2.1 生产者的写入并行度

生产者的写入并行度**直接由分区数决定**：

```java
// 分区数为 3 时，生产者可以同时向 3 个分区写入
// 每个分区的写入是独立的，可以并行

// 伪代码展示
for (int i = 0; i < 100; i++) {
    // 这些发送请求会并行发送到不同的分区
    producer.send(new ProducerRecord<>("orders", "msg-" + i));
}

// 实际执行顺序可能是：
// Thread 1 ──> Partition 0
// Thread 2 ──> Partition 1
// Thread 3 ──> Partition 2
// Thread 4 ──> Partition 0
// ...
```

### 2.2 生产者的 batch 机制

Kafka 生产者会**批量发送**消息到每个分区：

```java
// 生产者配置
Properties props = new Properties();
props.put("bootstrap.servers", "localhost:9092");
props.put("batch.size", 16384);        // 每个批次最大 16KB
props.put("linger.ms", 10);            // 等待凑满批次的时间
props.put("buffer.memory", 33554432);  // 总缓冲区 32MB
```

```
分区 0 的写入流程：

消息1 ─┐
消息2 ─┼──> [Batch Buffer] ──> 凑满 16KB 或等待 10ms ──> 批量发送到 Broker
消息3 ─┘

优点：减少网络请求次数，提升吞吐量
缺点：增加少量延迟（最多 linger.ms）
```

### 2.3 生产者吞吐量与分区数的关系

```
分区数 vs 吞吐量关系图：

吞吐量 (MB/s)
    │
 60 │                                    ● ● ● ● ●
    │                               ●
 50 │                          ●
    │                     ●
 40 │                ●
    │           ●
 30 │      ●
    │   ●
 20 │ ●
    │
  0  └────────────────────────────────────────────
    0    5    10    15    20    25    30    分区数

注意：分区数过多会带来额外开销，吞吐量不会无限增长
```

**分区数与吞吐量的经验公式**：

```java
// 经验公式
分区数 = max(消费者数量, 期望吞吐量 / 单分区吞吐)

// 单分区吞吐估算
// 普通机械硬盘：50 MB/s
// SSD：100-300 MB/s
// 网络带宽：50-100 MB/s（取决于网卡）
```

## 三、分区数与消费者并发

### 3.1 消费者数与分区数的关系

**核心规则**：一个消费者可以消费多个分区，但一个分区只能被一个消费者消费。

```
分区数 = 6，消费者数 = 3

  Partition 0 ──> Consumer 1
  Partition 1 ──> Consumer 1
  Partition 2 ──> Consumer 2
  Partition 3 ──> Consumer 2
  Partition 4 ──> Consumer 3
  Partition 5 ──> Consumer 3

每个消费者消费 2 个分区，负载均衡
```

```
分区数 = 3，消费者数 = 6

  Partition 0 ──> Consumer 1
  Partition 1 ──> Consumer 2
  Partition 2 ──> Consumer 3
  Consumer 4, 5, 6 分不到分区，只能空转！
```

### 3.2 分区数不足的问题

分区数过少会导致以下问题：

| 问题 | 表现 | 影响 |
|---|---|---|
| 消费者资源浪费 | 多余消费者分不到分区 | 浪费资源 |
| 消费能力受限 | 即使增加消费者也无法提升吞吐 | 瓶颈 |
| 热点分区 | 某些分区消息过多 | 积压 |
| 延迟增加 | 单分区处理压力大 | 延迟飙升 |

### 3.3 如何选择合适的分区数

选择分区数需要综合考虑以下因素：

**因素一：期望吞吐量**

```java
// 假设：
// - 期望吞吐量：10 万 msg/s
// - 单分区吞吐：2 万 msg/s
// - 需要的分区数：10 万 / 2 万 = 5

// 计算公式
分区数 = ceil(期望吞吐量 / 单分区消费吞吐)

// 单分区消费吞吐取决于：
// - 业务处理复杂度
// - 消费者机器配置
// - 网络带宽
```

**因素二：消费者数量**

```java
// 经验法则：分区数 >= 消费者数
// 建议：分区数 = 消费者数 * (1.5 ~ 2)

// 原因：
// 1. 预留扩展空间（消费者可能临时下线或增加）
// 2. 应对 Rebalance（消费者重启时需要重新分配）
```

**因素三：文件句柄限制**

```bash
# 每个 Broker 上的文件句柄数
# 每个分区需要打开多个文件：
# - 数据文件 .log
# - 索引文件 .index
# - 时间索引文件 .timeindex
# - Leader 选举时需要的网络连接

# 假设每个分区需要 3 个文件句柄
# Broker 有 1000 个分区：需要 3000 个文件句柄
# Linux 默认文件句柄限制通常为 65535
```

**因素四：副本数**

```properties
# 副本因子 = 3 时
# 每个分区的 Leader + 2 个 Follower
# 100 个分区 -> 300 个副本需要管理
# 元数据更新、同步压力都会增加
```

## 四、分区数过多的副作用

### 4.1 Leader 选举时间增加

每个分区都有 Leader，当 Broker 宕机时，需要重新选举 Leader：

```bash
# 分区数 vs Leader 选举时间

分区数 = 100:
  选举时间 ≈ 50ms（影响较小）

分区数 = 1000:
  选举时间 ≈ 500ms（明显感知）

分区数 = 10000:
  选举时间 ≈ 5s（可能造成服务抖动）
```

### 4.2 文件句柄增加

```bash
# 假设集群配置：
# - Topic 数：50
# - 每个 Topic 分区数：200
# - 副本因子：3
# - Broker 数：10

# 每个 Broker 上的分区数 = 50 * 200 * 3 / 10 = 3000 个分区

# 每个分区需要：
# - 1 个 Leader 连接（处理读写请求）
# - 副本因子-1 个 Follower 连接（同步数据）
# - 3 个文件（.log, .index, .timeindex）

# 每个 Broker 的网络连接数 = 3000 * 3 = 9000
# 每个 Broker 的文件数 = 3000 * 3 = 9000
```

### 4.3 Rebalance 成本增加

当消费者组成员变化时，需要重新分配分区：

```text
Rebalance 过程：
1. 所有 Consumer 停止消费
2. GroupCoordinator 计算新的分区分配
3. 向所有 Consumer 发送新分配方案
4. Consumer 重新开始消费

分区数越多，分配计算和通信的成本越高
```

### 4.4 控制器压力

Kafka 集群中有一个**控制器（Controller）**，负责管理分区状态和副本：

```text
控制器职责：
- 监控 Broker 存活状态
- 管理分区 Leader 选举
- 维护副本状态
- 处理分区重分配

分区数越多，控制器的压力越大
```

## 五、消费者线程数的设置方式

### 5.1 一个消费者多线程 vs 多个消费者

Kafka 消费者的并行消费有两种方式：

**方式一：一个消费者多线程（不推荐）**

```java
// 消费者内部使用多线程处理消息
KafkaConsumer<String, String> consumer = new KafkaConsumer<>(props);
consumer.subscribe(Arrays.asList("orders"));

while (true) {
    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(1000));

    // 使用线程池并行处理
    executor.submit(() -> {
        for (ConsumerRecord<String, String> record : records) {
            process(record);
        }
    });
}
```

问题：

- 仍然受到分区数限制（单线程拉取，多线程只是处理）
- 消息处理顺序无法保证
- offset 提交时机难以控制

**方式二：多个消费者实例（推荐）**

```java
// 启动多个消费者实例
// 每个实例是一个独立的进程

// 消费者 1
KafkaConsumer<String, String> consumer1 = new KafkaConsumer<>(props);
consumer1.subscribe(Arrays.asList("orders"));

// 消费者 2
KafkaConsumer<String, String> consumer2 = new KafkaConsumer<>(props);
consumer2.subscribe(Arrays.asList("orders"));

// Kafka 自动将分区分配给不同的消费者
```

### 5.2 消费者线程数的最佳实践

**核心原则**：消费者数量 `<=` 分区数量

```java
// 推荐配置
// 分区数 = 消费者数 * 1.5 ~ 2

// 例如：
// 期望并发消费：10 个消费者
// 分区数设置为：15 ~ 20

// 这样即使有 1-2 个消费者下线
// 剩余消费者也能接管，不会完全处理不过来
```

**消费者机器配置与线程数**：

```java
// 线程数配置建议
// CPU 密集型：线程数 = CPU 核心数
// IO 密集型：线程数 = CPU 核心数 * (1 + IO 等待时间 / CPU 时间)

// 示例：8 核机器，处理每条消息需要 10ms 等待 + 5ms 计算
// IO 等待比例：10 / 15 ≈ 67%
// 线程数 ≈ 8 * (1 + 2) = 24（考虑 2 倍 IO 密集度）
```

### 5.3 消费者配置示例

```java
// 消费者完整配置
Properties props = new Properties();
props.put("bootstrap.servers", "localhost:9092");
props.put("group.id", "order-consumer-group");
props.put("key.deserializer", "org.apache.kafka.common.serialization.StringDeserializer");
props.put("value.deserializer", "org.apache.kafka.common.serialization.StringDeserializer");

// 拉取配置
props.put("fetch.min.bytes", 1);              // 最少拉取字节数
props.put("fetch.max.wait.ms", 500);           // 最大等待时间
props.put("max.poll.records", 500);           // 每次最多拉取消息数

// 提交配置
props.put("enable.auto.commit", false);         // 手动提交
props.put("auto.commit.interval.ms", 1000);    // 自动提交间隔

// 心跳配置（影响 Rebalance）
props.put("heartbeat.interval.ms", 1000);      // 心跳间隔
props.put("session.timeout.ms", 30000);        // 会话超时
props.put("max.poll.interval.ms", 300000);    // 拉取间隔（避免频繁 Rebalance）

// 分区分配策略
props.put("partition.assignment.strategy", "cooperative-sticky");  // Kafka 2.4+ 推荐
```

## 六、分区重分配（Reassignment）

### 6.1 触发分区重分配的时机

| 触发时机 | 原因 | 影响 |
|---|---|---|
| Broker 宕机 | 分区的 Leader 不可用 | 自动触发 Leader 选举 |
| 消费者加入/离开 Group | Rebalance | 分区重新分配 |
| 分区数增加 | Topic 配置变更 | 分区重新分配 |
| Broker 上下线 | 集群扩缩容 | 分区自动迁移 |

### 6.2 分区重分配的流程

```bash
# 分区重分配命令
bin/kafka-reassign-partitions.sh --bootstrap-server localhost:9092 \
    --generate \
    --topics-to-move-json-file topics.json \
    --broker-list "0,1,2,3,4,5" \
    --execute

# topics.json 内容
{
    "topics": [
        {"topic": "orders"},
        {"topic": "payments"}
    ],
    "version": 1
}
```

```
分区重分配过程：

┌─────────────────────────────────────────────────────────────────────┐
│                        分区重分配流程                                │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  Phase 1: 副本复制（Replication）                                   │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  Old Broker 1        New Broker 3                          │   │
│  │  Partition 0 ────────> Partition 0'                        │   │
│  │  [正在同步...]        [数据复制中...]                        │   │
│  │                                                             │   │
│  │  期间：两个副本都对外服务                                     │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  Phase 2: ISR 切换                                                  │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  副本同步完成后：                                            │   │
│  │  - 切换 Leader 到新 Broker                                  │   │
│  │  - 旧 Broker 上的副本变为 Follower                          │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  Phase 3: 清理旧副本                                                │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  确认新 Leader 正常工作后：                                   │   │
│  │  - 删除旧 Broker 上的分区副本                                │   │
│  │  - 完成重分配                                                │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 6.3 分区重分配对性能的影响

**问题一：数据迁移期间性能下降**

```bash
# 迁移期间，网络带宽被副本同步占用
# 观察 metrics：

kafka.server:type=FetcherLagMetrics,name=RecordsLag,topic=orders,partition=0
# 值应该逐渐减小，直到为 0

kafka.server:type=FetcherMetrics,name=BytesPerSec,topic=orders
# 值会增加，因为有额外的复制流量
```

**问题二：客户端可能感知到 Rebalance**

```java
// 如果重分配期间有消费者变化，会触发 Rebalance
// 消费者需要重新拉取分配给自己的分区

// 建议：
// 1. 使用 cooperative-sticky 分配策略（最小化 Rebalance）
// 2. 分区重分配和消费者 Rebalance 分开执行
// 3. 在业务低峰期执行分区迁移
```

**优化建议**：

```properties
# 分区重分配配置
# 限制副本同步速度，避免影响正常业务

replica.lag.max.messages=4000          # Follower 落后的最大消息数
replica.lag.time.max.ms=10000          # Follower 落后的最大时间
num.replica.fetchers=1                # 副本同步线程数
```

## 七、Rebalance 机制详解

### 7.1 触发 Rebalance 的条件

| 触发条件 | 说明 |
|---|---|
| 消费者加入 Group | 新消费者启动 |
| 消费者离开 Group | 消费者崩溃或主动关闭 |
| 消费者心跳超时 | 消费者长时间未响应 |
| 分区数变化 | Topic 配置变更 |
| Subscription 变化 | 消费者订阅的 Topic 变化 |
| 消费者 session 超时 | 网络问题导致会话中断 |

### 7.2 Rebalance 的问题

**问题一：Stop The World**

```text
Rebalance 过程：

Consumer 1 ──┐
Consumer 2 ──┼──> [停止消费] --> [重新分配] --> [恢复消费]
Consumer 3 ──┘

期间：所有消费者停止消费
影响：消息积压，延迟增加
```

**问题二：分区分配不均**

某些分配策略可能导致分区分配不均匀：

```text
Range 分配策略示例（可能导致不均）：

Topic A: 3 个分区
Consumer Group: 2 个消费者

Range 分配：
  Consumer 1: Partition 0, 1  （2 个分区）
  Consumer 2: Partition 2   （1 个分区）

RoundRobin 分配：
  Consumer 1: Partition 0, 2
  Consumer 2: Partition 1
```

**问题三：频繁 Rebalance**

如果消费者处理速度慢或频繁超时：

```java
// 问题配置
props.put("session.timeout.ms", 10000);       // 太短，容易超时
props.put("max.poll.interval.ms", 60000);     // 太短，处理不完
props.put("heartbeat.interval.ms", 1000);     // 太频繁

// 正确配置
props.put("session.timeout.ms", 30000);        // 合理
props.put("max.poll.interval.ms", 300000);     // 留足够处理时间
props.put("heartbeat.interval.ms", 3000);      // 适中
```

### 7.3 分区分配策略

| 策略 | 特点 | 适用场景 |
|---|---|---|
| Range | 按 Topic 分配，可能不均匀 | 单 Topic |
| RoundRobin | 轮询分配，较均匀 | 多 Topic |
| StickyAssignor | 粘性分配，最小化 Rebalance 影响 | 通用 |
| CooperativeStickyAssignor | 协作式粘性分配，增量 Rebalance | Kafka 2.4+，最佳 |

**Range 策略**：

```text
Topic A (2 partitions): P0, P1
Topic B (2 partitions): P0, P1

Consumer Group: C1, C2

Range 分配结果：
  C1: Topic A-P0, Topic A-P1, Topic B-P0, Topic B-P1  （4 个分区）
  C2: 无分区！

问题：按 Topic 分别分配，可能导致不均
```

**RoundRobin 策略**：

```text
RoundRobin 分配结果：
  C1: Topic A-P0, Topic B-P0  （2 个分区）
  C2: Topic A-P1, Topic B-P1  （2 个分区）

更均匀的分配
```

**CooperativeStickyAssignor 策略**（Kafka 2.4+）：

```text
协作式 Rebalance 允许增量迁移分区：

Phase 1:
  C1: [P0, P1, P2]  正在消费
  C2: [P3]           正在消费
  C3: 加入 Group

Phase 2: 增量迁移
  C1: [P0, P1]       继续消费 P0, P1
  C2: [P3]           继续消费 P3
  C3: [P2]           开始消费 P2

特点：不需要所有消费者同时停止消费
```

## 八、实战：分区规划与调优

### 8.1 新系统分区规划

```java
// 新系统分区规划流程

// Step 1: 确定吞吐量需求
// 假设：
// - 峰值 QPS：10 万 msg/s
// - 单消息大小：1 KB
// - 期望延迟：< 100ms

// Step 2: 估算单分区吞吐
// 假设单分区吞吐：2 万 msg/s（保守估计）
// 需要分区数 = 10 万 / 2 万 = 5

// Step 3: 考虑消费者数量
// 假设消费者机器：5 台
// 分区数应该 >= 5
// 建议分区数 = 5 * 2 = 10（预留扩展空间）

// Step 4: 考虑副本和机器数
// 副本因子：3
// Broker 数：5
// 每个 Broker 分区数：10 * 3 / 5 = 6

// 最终配置：
// - 分区数：10
// - 副本因子：3
```

### 8.2 分区数调整的注意事项

**增加分区数**：

```bash
# 可以动态增加分区数
bin/kafka-topics.sh --alter --topic orders \
    --partitions 20 \
    --bootstrap-server localhost:9092

# 注意：
# 1. 分区数只能增加，不能减少
# 2. 增加分区后，新分区的数据需要重新分配
# 3. 消息的分区路由会改变（如果按 key 路由）
```

**减少分区数**：

```bash
# Kafka 不支持直接减少分区数
# 方法：创建新 Topic，迁移数据

# 1. 创建新 Topic（目标分区数）
bin/kafka-topics.sh --create --topic orders-v2 \
    --partitions 15 \
    --replication-factor 3 \
    --bootstrap-server localhost:9092

# 2. 使用 MirrorMaker 或手动迁移数据

# 3. 切换消费者到新 Topic

# 4. 删除旧 Topic
bin/kafka-topics.sh --delete --topic orders
```

### 8.3 监控分区与消费者状态


## 面试追问预判

### 追问一：分区数过多会有什么问题？

分区数过多的主要问题：

1. **Controller 压力增大**：分区状态变化需要 Controller 处理
2. **元数据压力**：每个客户端都需要缓存所有分区的元数据
3. **文件句柄增加**：每个分区需要打开多个文件
4. **副本同步开销**：Leader 选举和副本同步时间增加
5. **Rebalance 成本**：消费者组 Rebalance 时间增加

### 追问二：如何动态调整分区数？

Kafka 支持增加分区数，不支持减少分区数。减少分区数的方案是创建新 Topic 并迁移数据。

### 追问三：消费者如何保证消息顺序？

要保证消息顺序，需要：

1. **单分区**：所有消息发送到同一个分区
2. **单消费者**：只有一个消费者实例消费该分区
3. **幂等处理**：处理可能重复的消息

如果需要保证全局顺序，吞吐量会受到限制。
