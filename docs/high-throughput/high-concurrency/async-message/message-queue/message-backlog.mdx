# 消息积压的排查、处理与预防方案

## 场景切入：凌晨的告警

凌晨 4 点，手机突然震动。监控告警显示：Kafka 消费者 Lag 飙升，从平时的几千条直接突破百万。

你赶紧爬起来打开电脑，查了一圈发现：消费者正常运行，但处理速度远远跟不上消息生产速度。消息在队列中越堆越多，消费者的处理延迟从秒级变成分钟级。

这是一个典型的**消息积压**场景。

消息积压是高并发系统中最常见的问题之一。它不像系统崩溃那样明显，但在悄无声息中，你的系统正在慢慢失去实时性。如果不及时处理，最终会导致：

- 业务延迟增加，用户体验下降
- 消息堆积，存储成本上升
- 消费者压力增大，可能引发连锁故障

本文将深入讲解消息积压的成因、排查方法、处理方案和预防措施，帮助你在生产环境中从容应对这一挑战。

---

## 一、什么是消息积压

### 1.1 消息积压的定义

**消息积压**是指消息队列中堆积的消息数量持续增长，消费者的处理速度跟不上生产者的发送速度，导致消息无法被及时消费。

```
正常情况：
生产速度 ≈ 消费速度
Lag 稳定在较低水平

积压情况：
生产速度 > 消费速度
Lag 持续增长
```

### 1.2 消息积压的核心指标：Lag

**Lag** 是消息积压的核心指标，表示**消费者还未处理的消息数量**：

```
Lag = Consumer Offset（已消费位置）- Producer Offset（生产位置）

Kafka 示例：
Topic: orders
  Partition 0
    Producer Offset: 100000  ← 最新消息位置
    Consumer Offset: 95000    ← 已消费位置
    Lag = 5000                ← 积压 5000 条消息
```

| Lag 状态 | 含义 | 处理建议 |
|---|---|---|
| Lag = 0 或很小 | 消费正常，无积压 | 正常监控 |
| Lag 持续增长 | 生产 > 消费，正在积压 | 需要处理 |
| Lag 稳定但较大 | 消费速度匹配生产，但起点就较高 | 评估是否正常 |
| Lag 在下降 | 消费 > 生产，积压在消化 | 继续观察 |

### 1.3 消息积压的危害

**延迟增加**

消息积压最直接的影响是**处理延迟**。用户下单后，本应秒级完成的订单确认，因为积压变成分钟级甚至小时级。

```
正常流程：
用户下单 → MQ → 立即消费 → 订单完成（< 1s）

积压后流程：
用户下单 → MQ → 等待 10 分钟 → 订单完成
```

**存储成本上升**

Kafka 的消息存储依赖磁盘。当消息大量积压时，磁盘占用会持续增长：

- 单条消息平均大小：1KB ~ 10KB
- 积压 1000 万条消息：10GB ~ 100GB 磁盘占用
- 如果积压持续一周：存储成本可能成为灾难

**消费者压力增大**

积压的消息最终还是要被消费。当消费者开始消化积压时：

- CPU 使用率飙升
- 内存占用增加
- 可能触发 OOM
- 如果消费者扛不住，可能直接宕机

**数据时效性丧失**

有些业务对数据的时效性要求很高：

- 实时推荐系统：消息积压导致推荐结果过时
- 监控告警：告警信息延迟到达，错过最佳处理时间
- 金融交易：订单处理延迟，引发用户投诉

## 二、消息积压的成因分析

### 2.1 生产者端原因

**流量突增**

促销活动、热点事件、恶意攻击都可能导致流量突增：

```
常规流量：1000 QPS
促销流量：50000 QPS（50 倍增长）
消费者处理能力：2000 QPS
结果：积压以 48000 QPS 的速度增长
```

**发送频率未控制**

代码中可能存在循环发送、批量发送配置不当等问题：

```java
// 错误的写法：每次查询都发送消息
for (Order order : allOrders) {
    kafkaTemplate.send("order-topic", order);  // 无限制发送
}

// 正确的写法：分批发送，控制频率
int batchSize = 100;
for (int i = 0; i < allOrders.size(); i += batchSize) {
    List<Order> batch = allOrders.subList(i, Math.min(i + batchSize, allOrders.size()));
    kafkaTemplate.sendBatch(batch);
    Thread.sleep(100);  // 控制发送频率
}
```

### 2.2 消费者端原因

**消费者宕机**

最常见的原因。消费者突然宕机，无法继续消费：

```
Consumer 1 宕机 ──> Partition 0, 1 无人消费
Consumer 2 ──> Partition 2, 3 正常消费
结果：2 个分区积压
```

**消费逻辑耗时**

业务逻辑处理时间过长：

```java
@KafkaListener(topics = "order-topic")
public void consume(OrderMessage message) {
    // 同步调用外部服务，耗时 5 秒
    Response response = httpClient.post(externalService, message);

    // 同步写入数据库，耗时 2 秒
    orderService.save(message);

    // 总耗时：7 秒/条
    // 如果每秒来 1000 条消息，消费者完全跟不上
}
```

**消费者线程数不足**

单线程消费无法满足高吞吐需求：

```java
// 默认单线程消费
@KafkaListener(topics = "order-topic")
public void consume(String message) {
    process(message);  // 单线程串行处理
}

// 应该使用多线程消费
@Bean
public ConcurrentKafkaListenerContainerFactory<String, String> kafkaListenerContainerFactory() {
    ConcurrentKafkaListenerContainerFactory<String, String> factory =
            new ConcurrentKafkaListenerContainerFactory<>();
    factory.setConcurrency(10);  // 10 个并发消费者
    return factory;
}
```

**消费者异常导致重试**

消息处理失败后不断重试：

```java
@KafkaListener(topics = "order-topic")
public void consume(OrderMessage message) {
    try {
        process(message);
    } catch (Exception e) {
        log.error("处理失败，重试", e);
        throw e;  // 抛出异常，触发重试
    }
}
```

### 2.3 Broker 端原因

**Broker 性能瓶颈**

Broker 资源耗尽（CPU、内存、磁盘 IO、网络）：

```bash
# 查看 Broker CPU 使用
top -p $(pgrep -f kafka.Kafka)

# 查看磁盘 IO
iostat -x 1

# 查看网络带宽
sar -n DEV 1
```

**网络分区**

Broker 之间的网络分区导致消费者无法连接：

```
Consumer ──✗──> Broker（网络不通）
             │
             └──> 其他 Broker（正常）
             │
             └──> 积压发生
```

### 2.4 其他原因

**消息格式不兼容**

消费者无法解析消息，导致消费失败：

```java
// 生产者发送
kafkaTemplate.send("topic", Json.stringify(newMessage));

// 消费者解析（假设还在用旧格式）
ObjectMapper mapper = new ObjectMapper();
OldMessage msg = mapper.readValue(json, OldMessage.class);  // 解析失败
```

**依赖服务不可用**

消费者依赖的外部服务（数据库、缓存、其他微服务）不可用：

```java
@KafkaListener(topics = "order-topic")
public void consume(OrderMessage message) {
    // 数据库连接池耗尽
    Connection conn = dataSource.getConnection();  // 超时或失败

    // Redis 不可用
    redisTemplate.opsForValue().get("key");  // 抛出异常

    // 外部服务超时
    httpClient.post(url, message);  // 超时
}
```

## 三、消息积压的排查步骤

### 3.1 第一步：确认积压范围

首先确认哪些 Topic、哪些 Partition 出现了积压：

```bash
# 使用 kafka-consumer-groups.sh 查看消费组状态
kafka-consumer-groups.sh --bootstrap-server localhost:9092 \
    --group order-consumer-group \
    --describe

# 输出示例
GROUP                 TOPIC           PARTITION  CURRENT-OFFSET  LOG-END-OFFSET  LAG
order-consumer-group  order-topic     0          50000            55000           5000
order-consumer-group  order-topic     1          48000            48000           0
order-consumer-group  order-topic     2          52000            62000           10000
order-consumer-group  order-topic     3          45000            45000           0
```

从输出中可以看到：

- Partition 0 积压 5000 条
- Partition 1 无积压
- Partition 2 积压 10000 条（最严重）
- Partition 3 无积压

### 3.2 第二步：分析 Lag 变化趋势

查看 Lag 是快速增长还是缓慢增长：

```bash
# 持续监控 Lag 变化
watch -n 5 'kafka-consumer-groups.sh --bootstrap-server localhost:9092 \
    --group order-consumer-group \
    --describe'
```

| Lag 变化 | 原因分析 |
|---|---|
| 快速增长 | 消费能力严重不足 |
| 缓慢增长 | 消费能力略有不足 |
| 稳定 | 消费能力刚好匹配 |
| 下降 | 积压正在消化 |

### 3.3 第三步：检查消费者状态

确认消费者是否正常运行：

```bash
# 查看消费者进程
ps aux | grep kafka

# 查看消费者线程
jstack <pid> | grep "KafkaConsumer"

# 查看消费者 JMX 指标
jconsole <pid>
```

**常见问题**：

- 消费者进程不存在 → 宕机
- 消费者线程全部 blocked → 等待某个资源
- 消费者线程在 sleep → 等待任务
- GC 停顿 → Full GC 时间过长

### 3.4 第四步：分析消费耗时

通过日志或 APM 工具分析单条消息的消费耗时：

```java
@KafkaListener(topics = "order-topic")
public void consume(OrderMessage message) {
    long start = System.currentTimeMillis();

    try {
        process(message);
    } finally {
        long cost = System.currentTimeMillis() - start;
        log.info("消息处理耗时: {}ms, messageId: {}", cost, message.getId());

        // 如果耗时过长，触发告警
        if (cost > 1000) {
            log.warn("消费耗时过长: {}ms", cost);
        }
    }
}
```

### 3.5 第五步：定位瓶颈

根据排查结果，定位具体的瓶颈：

| 瓶颈类型 | 排查方法 | 解决方案 |
|---|---|---|
| CPU 瓶颈 | top 查看 CPU 使用率 | 增加消费者、优化代码 |
| 内存瓶颈 | jstat 查看 GC | 优化内存使用、增加内存 |
| IO 瓶颈 | iostat 查看磁盘 IO | 优化存储、使用 SSD |
| 网络瓶颈 | sar 查看网络带宽 | 优化网络配置 |
| 数据库瓶颈 | 慢查询日志 | 优化 SQL、增加连接池 |
| 外部服务瓶颈 | 调用链路追踪 | 增加超时、优化重试 |

## 五、消息积压的处理方案

### 5.1 方案一：增加消费者（扩容）

最直接的方案，增加消费者实例，提高并行处理能力：

**扩容前**：

```
Topic: order-topic (4 个 Partition)

Consumer Group: order-service
  Consumer 1 → Partition 0, 1, 2, 3（单线程处理所有分区）
```

**扩容后**：

```
Topic: order-topic (4 个 Partition)

Consumer Group: order-service
  Consumer 1 → Partition 0, 1
  Consumer 2 → Partition 2, 3
  Consumer 3 → 新增消费者
```

**扩容步骤**：

1. 确认分区数是否足够（分区数 >= 消费者数）
2. 启动新的消费者实例
3. 观察 Rebalance 过程
4. 确认 Lag 开始下降

```bash
# 如果分区数不足，需要增加分区
kafka-topics.sh --alter --topic order-topic \
    --partitions 8 \
    --bootstrap-server localhost:9092
```

:::warning 注意
增加分区数会改变消息的路由规则，可能影响顺序性。如果业务对顺序有要求，增加分区后需要重新评估。
:::

### 5.2 方案二：消费者多线程消费

单个消费者内部增加并发度：

```java
@Bean
public ConcurrentKafkaListenerContainerFactory<String, String> kafkaListenerContainerFactory() {
    ConcurrentKafkaListenerContainerFactory<String, String> factory =
            new ConcurrentKafkaListenerContainerFactory<>();
    // 设置并发数 = 分区数
    factory.setConcurrency(4);
    // 批量消费
    factory.setBatchListener(true);
    return factory;
}
```

**多线程消费模型**：

```java
@KafkaListener(topics = "order-topic", containerFactory = "kafkaListenerContainerFactory")
public void consume(List<OrderMessage> messages) {
    // 使用线程池并行处理
    List<CompletableFuture> futures = messages.stream()
        .map(msg -> CompletableFuture.runAsync(() -> process(msg), executor))
        .collect(Collectors.toList());

    // 等待所有消息处理完成
    CompletableFuture.allOf(futures.toArray(new CompletableFuture[0])).join();
}
```

### 5.3 方案三：消息迁移

将积压的消息迁移到新的 Topic，逐步消费：

**场景**：积压量太大，短期内无法消化；或者旧消费者有 bug

**步骤**：

1. 创建新 Topic（分区数更多）
2. 写一个临时消费程序，从旧 Topic 消费并发送到新 Topic
3. 使用新消费者处理新 Topic

```java
// 消息迁移程序
@Bean
public KafkaListenerContainerFactory<ConcurrentMessageListenerContainer<String, String>> 
    migrationListenerContainerFactory() {
    ConcurrentKafkaListenerContainerFactory<String, String> factory =
            new ConcurrentKafkaListenerContainerFactory<>();
    factory.setConcurrency(10);
    factory.setBatchListener(true);
    return factory;
}

// 从旧 Topic 消费，发送到新 Topic
@KafkaListener(topics = "old-topic", containerFactory = "migrationListenerContainerFactory")
public void migrate(List<ConsumerRecord<String, String>> records) {
    List<ProducerRecord<String, String>> newRecords = records.stream()
        .map(r -> new ProducerRecord<>("new-topic", r.key(), r.value()))
        .collect(Collectors.toList());

    kafkaTemplate.sendBatch(newRecords);
}
```

### 5.4 方案四：紧急降级（跳过部分消息）

极端情况下，可以选择跳过部分消息：

```java
@KafkaListener(topics = "order-topic")
public void consume(ConsumerRecord<String, String> record) {
    // 检查积压时间
    long lagTime = System.currentTimeMillis() - record.timestamp();
    
    if (lagTime > 30 * 60 * 1000) {  // 积压超过 30 分钟
        log.warn("跳过过期消息: {}", record.key());
        return;  // 跳过消息
    }

    process(record);
}
```

:::danger 警告
跳过消息可能导致业务数据丢失或不一致。使用此方案需要谨慎评估业务影响，并确保有其他补偿机制。
:::

### 5.5 方案五：提高消费并行度

调整消费者并行度：

```java
// 方式一：提高 concurrency
factory.setConcurrency(20);

// 方式二：提高批次大小
props.put("max.poll.records", 1000);  // 每次拉取 1000 条

// 方式三：减少 poll 间隔
props.put("poll.timeout.ms", 500);  // 500ms 无消息就返回
```

**并行度配置原则**：

| 场景 | 建议并发数 |
|---|---|
| CPU 密集型消费 | CPU 核心数 |
| IO 密集型消费 | 2-4 倍 CPU 核心数 |
| 批量消费 | 分区数 |

## 六、消息积压的预防措施

### 6.1 限流保护

在生产者端限流，避免流量突增打垮消费者：

```java
// 使用 RateLimiter 限流
RateLimiter rateLimiter = RateLimiter.create(10000);  // 每秒 10000 条

@PostMapping("/order")
public Response createOrder(Order order) {
    rateLimiter.acquire();  // 获取令牌
    kafkaTemplate.send("order-topic", order);
    return Response.ok();
}
```

**Kafka 生产者端限流配置**：

```java
// 全局限流
props.put("max.block.ms", 60000);  // 缓冲区满时最多阻塞 60 秒

// 租户级别限流（Kafka 2.6+）
props.put("quota.producer.default", "10MB");  // 每秒 10MB
```

### 6.2 监控告警

配置 Lag 告警，及时发现问题：

```yaml
# Prometheus 告警规则
groups:
  - name: kafka-lag-alert
    rules:
      - alert: KafkaLagHigh
        expr: kafka_consumer_lag_sum > 100000
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "Kafka 消费 Lag 过高"
          description: "消费组 {{ $labels.group }} 的 Lag 为 {{ $value }}"

      - alert: KafkaLagCritical
        expr: kafka_consumer_lag_sum > 1000000
        for: 1m
        labels:
          severity: critical
        annotations:
          summary: "Kafka 消费 Lag 严重"
          description: "消费组 {{ $labels.group }} 的 Lag 为 {{ $value }}，请立即处理"
```

### 6.3 容量规划

根据业务量提前规划容量：

```java
// 容量规划公式
// 期望 QPS = 10000
// 单条消息处理耗时 = 100ms
// 单消费者并发 = 10
// 需要的消费者数 = QPS × 处理耗时 / 并发数
//               = 10000 × 0.1 / 10
//               = 100 个消费者

// 分区规划
// 分区数 = max(消费者数, 期望吞吐量 / 单分区吞吐)
// 单分区吞吐 ≈ 50MB/s
// 需要的分区数 = max(100, 100MB/s / 50MB/s) = 100 个分区
```

**容量规划表**：

| 业务场景 | 日消息量 | QPS | 消费者数 | 分区数 |
|---|---|---|---|---|
| 普通订单 | 100 万 | 100 | 10 | 10 |
| 秒杀订单 | 1000 万 | 10000 | 100 | 100 |
| 直播弹幕 | 1 亿 | 100000 | 1000 | 500 |

### 6.4 消费者健康检查

定期检查消费者健康状态：

```java
// 健康检查接口
@GetMapping("/health/consumer")
public Response checkConsumerHealth() {
    try {
        // 检查 Lag
        long lag = getConsumerLag();
        if (lag > threshold) {
            return Response.status(503)
                .entity("Consumer lag too high: " + lag)
                .build();
        }

        // 检查消费速率
        double rate = getConsumeRate();
        if (rate < minRate) {
            return Response.status(503)
                .entity("Consume rate too low: " + rate)
                .build();
        }

        return Response.ok("OK");
    } catch (Exception e) {
        return Response.status(503)
            .entity("Consumer health check failed: " + e.getMessage())
            .build();
    }
}
```

### 6.5 消费逻辑优化

优化消费逻辑，提高处理效率：

**异步化处理**：

@KafkaListener(topics = "order-topic")
public void consume(OrderMessage message) {

@KafkaListener(topics = "order-topic", containerFactory = "batchFactory")

## 七、消费顺序性在积压处理中的注意事项

### 7.1 顺序消息的特点

有些业务对消息顺序有要求（如订单流程）：

```
订单流程必须顺序处理：
1. 创建订单（订单状态 = 新建）
2. 扣减库存（库存减少）
3. 扣减余额（余额减少）
4. 订单完成（订单状态 = 完成）

如果顺序乱了：
1. 扣减余额（订单不存在，失败）
2. 扣减库存（订单不存在，失败）
3. 创建订单（余额已扣，库存已扣，订单失败）
```

### 7.2 顺序消息积压的处理策略

**策略一：保证顺序的分区内顺序处理**

```java
// 同一个 Partition 内的消息顺序处理
// 不同 Partition 之间可以并行
@Bean
public KafkaTemplate<String, String> kafkaTemplate() {
    KafkaTemplate<String, String> template = new KafkaTemplate<>(producerFactory);
    // 使用订单 ID 作为 key，保证同一订单的消息到同一 Partition
    template.setKeyGenerator(message -> message.getOrderId());
    return template;
}
```

**策略二：降级处理**

如果积压严重，可以降级为「最终一致」：

```java
@KafkaListener(topics = "order-topic")
public void consume(OrderMessage message) {
    try {
        // 先检查订单状态，避免重复处理
        Order order = orderService.getById(message.getOrderId());
        if (order.getStatus() >= message.getStatus()) {
            log.info("订单 {} 状态已更新，忽略消息", message.getOrderId());
            return;
        }

        process(message);
    } catch (Exception e) {
        // 处理失败，发送到重试队列
        retryTemplate.send("order-retry-topic", message);
    }
}
```

**策略三：暂停非关键消费者**

优先处理关键消息：

```java
@KafkaListener(topics = "order-critical-topic", concurrency = "10")  // 关键消息高优先级
public void consumeCritical(OrderMessage message) {
    processCritical(message);
}

@KafkaListener(topics = "order-normal-topic", concurrency = "2")  // 非关键消息低优先级
public void consumeNormal(OrderMessage message) {
    processNormal(message);
}
```

## 八、消息过期和死信的处理

### 8.1 消息过期

Kafka 支持消息过期设置：

```java
// 生产者设置消息过期时间
ProducerRecord<String, String> record = new ProducerRecord<>("topic", "key", "value");
record.headers().add("TTL", "86400000".getBytes());  // 24 小时过期
kafkaTemplate.send(record);
```

**过期消息的处理**：

```java
@KafkaListener(topics = "order-topic")
public void consume(ConsumerRecord<String, String> record) {
    // 检查消息是否过期
    long now = System.currentTimeMillis();
    long timestamp = record.timestamp();
    long ttl = 7 * 24 * 60 * 60 * 1000L;  // 7 天

    if (now - timestamp > ttl) {
        log.warn("消息过期，跳过: key={}", record.key());
        // 发送到死信队列
        kafkaTemplate.send("order-dlq-topic", record.value());
        return;
    }

    process(record);
}
```

### 8.2 死信队列（DLQ）

死信队列用于存放处理失败的消息：

**Kafka 配置**：

```yaml
# Kafka 配置
spring:
  kafka:
    consumer:
      config:
        bootstrap.servers: localhost:9092
```

**手动实现 DLQ**：

```java
@KafkaListener(topics = "order-topic")
public void consume(ConsumerRecord<String, String> record) {
    try {
        process(record);
    } catch (Exception e) {
        log.error("处理失败，发送到死信队列", e);

        // 发送到死信队列
        kafkaTemplate.send("order-dlq-topic", record.key(), record.value());

        // 记录原始 Topic 和 Partition
        ProducerRecord<String, String> dlqRecord = new ProducerRecord<>(
            "order-dlq-topic",
            record.key(),
            record.value()
        );
        dlqRecord.headers().add("original-topic", record.topic().getBytes());
        dlqRecord.headers().add("original-partition", String.valueOf(record.partition()).getBytes());
        dlqRecord.headers().add("original-offset", String.valueOf(record.offset()).getBytes());
        dlqRecord.headers().add("error-message", e.getMessage().getBytes());

        kafkaTemplate.send(dlqRecord);
    }
}
```

**死信队列消费**：

```java
@KafkaListener(topics = "order-dlq-topic")
public void consumeDLQ(ConsumerRecord<String, String> record) {
    String originalTopic = new String(record.headers().lastHeader("original-topic").value());
    int originalPartition = Integer.parseInt(new String(record.headers().lastHeader("original-partition").value()));
    long originalOffset = Long.parseLong(new String(record.headers().lastHeader("original-offset").value()));
    String errorMessage = new String(record.headers().lastHeader("error-message").value());

    log.error("死信消息: originalTopic={}, partition={}, offset={}, error={}",
        originalTopic, originalPartition, originalOffset, errorMessage);

    // 处理死信：人工介入或补偿处理
    handleDeadLetter(record);
}
```

## 九、总结

消息积压是分布式系统中常见的问题，处理思路可以总结为：

```
消息积压处理流程：

发现问题（Lag 监控告警）
    │
    ▼
排查原因（生产者 / 消费者 / Broker）
    │
    ├─→ 生产者流量突增 → 限流保护
    │
    ├─→ 消费者能力不足 → 扩容 / 多线程 / 优化逻辑
    │
    ├─→ 消费者宕机 → 重启 / 扩容
    │
    └─→ Broker 瓶颈 → 扩容 / 优化配置
    │
    ▼
选择处理方案
    │
    ├─→ 扩容消费者
    ├─→ 消息迁移
    ├─→ 紧急降级
    └─→ 提高并行度
    │
    ▼
验证效果（Lag 下降）
    │
    ▼
预防措施（限流 + 监控 + 容量规划）
```

## 面试追问预判

### 追问一：如何避免消费者 Rebalance 导致的处理延迟？

消费者 Rebalance 会导致所有消费者停止消费，这是 Kafka 的一个痛点。解决方案包括：

1. **减少 Rebalance 发生**：
   - 合理配置 `session.timeout.ms` 和 `heartbeat.interval.ms`
   - 避免消费者长时间 GC（配置合适的 JVM 参数）

2. **使用静态成员**：
   ```java
   props.put("group.instance.id", "consumer-001");
   ```
   消费者重启后不会触发 Rebalance。

3. **使用协作式再平衡**（Kafka 2.4+）：
   ```java
   props.put("partition.assignment.strategy", "cooperative-sticky");
   ```
   只迁移需要迁移的分区，其他分区继续消费。

4. **分离心跳线程**：
   ```java
   // 独立心跳线程，避免业务处理阻塞心跳
   props.put("heartbeat.thread.name", "kafka-heartbeat");
   ```

### 追问二：Kafka 的消息可靠性如何保证？

Kafka 的可靠性通过多副本机制保证：

1. **副本因子**：设置 `replication.factor >= 3`
2. **最小同步副本**：`min.insync.replicas = 2`
3. **生产者确认**：`acks = all`
4. **消费者手动提交 Offset**

```properties
# Broker 配置
min.insync.replicas=2
replication.factor=3

# 生产者配置
acks=all
retries=3
enable.idempotence=true
```

### 追问三：如何设计一个消息处理系统，保证消息不丢失又不重复消费？

这是面试中的经典问题，需要同时考虑生产者、Broker、消费者三个环节：

**生产者端**：

- 使用确认机制（`acks=all`）
- 开启幂等（`enable.idempotence=true`）
- 消息发送失败重试

**Broker 端**：

- 多副本存储
- 同步刷盘
- 合理配置 `min.insync.replicas`

**消费者端**：

- 手动提交 Offset
- 业务处理成功后提交
- 使用数据库唯一索引实现幂等

```java
@KafkaListener(topics = "order-topic")
public void consume(OrderMessage message) {
    try {
        // 1. 检查是否已处理（幂等）
        if (orderService.exists(message.getOrderId())) {
            log.info("订单已处理，忽略");
            return;
        }

        // 2. 业务处理
        orderService.process(message);

        // 3. 处理成功后才提交 Offset
        // (Spring Kafka 会自动提交，如果抛出异常则不提交)
    } catch (DuplicateKeyException e) {
        // 唯一键冲突，说明已处理，忽略
    }
}
```
