# 消息队列优化：Kafka 高吞吐的秘密

> [!WARNING]
> 前置知识：本篇文章是消息队列优化的入口，涵盖 Kafka、RabbitMQ 的核心原理与实战优化。建议按顺序阅读，理解消息队列在高并发系统中的核心价值。

---

每秒 100 万条消息，Kafka 是怎么做到的？

答案藏在 Kafka 的架构设计里：顺序写磁盘、零拷贝、批量处理、压缩……每一个优化点，都是 Kafka 团队在性能与可靠性之间精心权衡的结果。

但 Kafka 的高吞吐不是银弹。它有复杂度，有延迟，有顺序保证的限制。理解它的原理，才能在生产环境中用好它。

这一讲，我们来深入 Kafka 的核心机制。

## 二、Kafka vs RabbitMQ vs RocketMQ

### 三者的定位差异

| 对比项 | Kafka | RabbitMQ | RocketMQ |
|---|---|---|---|
| 设计定位 | 日志收集、大数据、流处理 | 通用消息队列 | 电商、金融场景 |
| 吞吐量 | 百万级 QPS | 万级 QPS | 十万级 QPS |
| 消息延迟 | 毫秒级 | 微秒级（最低） | 毫秒级 |
| 消息顺序 | 分区内有序 | 单队列有序 | 分区内有序 |
| 消息持久化 | 支持 | 支持 | 支持 |
| 事务消息 | 不支持原生 | 支持 | 支持原生 |
| 延迟消息 | 需插件 | 支持插件 | 支持原生 |
| 消息回溯 | 支持 | 不支持 | 支持 |
| 集群规模 | 支持超大集群 | 百节点级 | 千节点级 |

### 选型建议

| 场景 | 推荐 | 原因 |
|---|---|---|
| 日志收集、大数据分析 | Kafka | 超高吞吐，完美契合 |
| 电商订单、金融交易 | RocketMQ | 原生事务消息，延迟消息 |
| 通用场景、低延迟 | RabbitMQ | Erlang 实现，性能优秀 |
| 微服务解耦 | RabbitMQ / RocketMQ | 延迟消息支持好 |

## 三、Kafka 高吞吐的六大秘密

### 秘密一：顺序写磁盘

传统数据库是随机写，每次写入都需要寻道 + 旋转延迟：

```text
随机写：
┌─────────────────────────────────────────┐
│  Sector 0  │  写入  │  Sector 50 │  写入 │  → 大量磁盘寻道
│  磁头移动 → 旋转 → 写入 → 移动 → 旋转 → 写入
└─────────────────────────────────────────┘

顺序写（Kafka）：
┌─────────────────────────────────────────┐
│  Sector 0 → Sector 1 → Sector 2 → ...   │  → 一次寻道，持续写入
│  磁头移动一次，写入连续
└─────────────────────────────────────────┘
```

Kafka 的写入是追加式顺序写，磁盘顺序写的速度可以接近内存随机读写的速度。

### 秘密二：零拷贝

传统的数据传输需要 4 次拷贝和 2 次上下文切换：

```text
传统方式（4次拷贝）：
磁盘 → 内核缓冲区 → 用户空间 → Socket 缓冲区 → 网卡
     ↑            ↑         ↑            ↑
   DMA拷贝    DMA拷贝    CPU拷贝      DMA拷贝
```

Kafka 使用 `sendfile` 系统调用，只需 2 次拷贝：

```java
// 零拷贝：数据直接从 PageCache 到网卡
fileChannel.transferTo(position, length, socketChannel);
```

### 秘密三：批量处理

Kafka 的批量体现在三个层面：

```java
// 1. 生产者批量发送
Properties props = new Properties();
props.put("batch.size", 16384);       // 批量大小 16KB
props.put("linger.ms", 10);            // 等待时间 10ms
props.put("buffer.memory", 33554432); // 缓冲池 32MB

// 2. 消费者批量消费
ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));
for (ConsumerRecord<String, String> record : records) {
    // 批量处理
}

// 3. 批量压缩
props.put("compression.type", "snappy");  // 或 lz4、zstd
```

### 秘密四：页缓存（PageCache）

Kafka 利用操作系统的 PageCache 加速读写：

```text
写入：Producer → PageCache → 磁盘（异步刷盘）
读取：PageCache（命中） → Consumer（极速）
             ↓ 未命中
           磁盘 → PageCache → Consumer
```

只要消费者的速度跟得上，数据大概率在 PageCache 中就能命中，读取速度接近内存。

### 秘密五：分区（Partition）

Kafka 通过分区实现水平扩展：

```text
Topic: order-events（3个分区）
┌──────────────┐
│  Partition-0  │ → Consumer-0（处理 1/3 数据）
├──────────────┤
│  Partition-1  │ → Consumer-1（处理 1/3 数据）
├──────────────┤
│  Partition-2  │ → Consumer-2（处理 1/3 数据）
└──────────────┘
```

分区数越多，并行消费能力越强。但分区数也不是越多越好——每个分区需要占用文件描述符，过多分区会增加 ZooKeeper/Controller 的负担。

### 秘密六：压缩

Kafka 支持多种压缩算法：

| 压缩算法 | 压缩比 | CPU 开销 | 适用场景 |
|---|---|---|---|
| gzip | 高 | 高 | 网络带宽紧张 |
| snappy | 中 | 中 | 平衡场景（Kafka 默认） |
| lz4 | 中低 | 低 | 低延迟场景 |
| zstd | 高 | 中 | 追求高压缩比 |

## 四、消息可靠性：面试超高频问题

「如何保证消息不丢失？」是面试必问题。消息的传递有三个环节，每个环节都需要确认：

### 环节一：生产者到 Broker

```java
// 使用确认机制
props.put("acks", "all");  // 所有副本都写入成功才确认
props.put("retries", 3);   // 重试 3 次
props.put("enable.idempotence", true);  // 开启幂等
```

| acks 配置 | 可靠性 | 性能 | 适用场景 |
|---|---|---|---|
| acks=0 | 低 | 最高 | 允许少量丢失 |
| acks=1 | 中 | 高 | 普通场景 |
| acks=all | 高 | 低 | 高可靠场景 |

### 环节二：Broker 自身

```properties
# 持久化配置
fsync=always        # 每次写入都刷盘（最可靠，性能最差）
fsync=periodic      # 定期刷盘（推荐，性能好，丢失少量数据）
fsync=disable       # 不刷盘（不推荐）

# 副本配置
replication.factor=3    # 3 副本
min.insync.replicas=2    # 至少 2 个副本同步成功
```

### 环节三：消费者到处理完成

```java
// 手动提交 offset
while (true) {
    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));
    for (ConsumerRecord<String, String> record : records) {
        try {
            process(record);
            consumer.commitSync();  // 处理完成后提交 offset
        } catch (Exception e) {
            // 记录失败，等待重试
            log.error("处理失败", e);
            // 不提交 offset，下次 poll 会重新拉取
        }
    }
}
```

> [!WARNING]
> 自动提交 offset 的问题是：消息可能没处理完就提交了 offset，如果消费者崩溃，消息会丢失。**高可靠场景必须使用手动提交 offset。**

## 五、消息积压：如何处理

### 消息积压的原因

| 原因 | 表现 | 解决方案 |
|---|---|---|
| 消费者能力不足 | 消费速度 < 生产速度 | 增加消费者、优化消费逻辑 |
| 消费者故障 | 某个消费者不工作 | 排查消费者、修复后自动追赶 |
| 消费者死循环 | CPU 100% | 排查代码、紧急重启 |
| 消费者卡住 | 长时间不 poll | 增加 poll 超时监控 |

### 解决方案一：增加消费者

```yaml
# Kafka 消费者组扩容
# 如果当前有 3 个消费者，增加到 6 个
# 每个消费者处理 1/6 的分区
# 消费能力翻倍
```

### 解决方案二：消息迁移

如果积压太多，增加消费者来不及，可以临时迁移到新 Topic：

```java
// 临时将消息迁移到快消费 Topic
public void migrateToFastTopic(String sourceTopic, String targetTopic) {
    // 1. 创建新的快速消费者
    // 2. 消费 sourceTopic，写入 targetTopic
    // 3. 用新的消费者消费 targetTopic
}
```

### 解决方案三：优先处理重要消息

```java
// 消息分级：重要消息优先处理
@KafkaListener(topics = "order-events")
public void consume(ConsumerRecord<String, String> record) {
    if (record.priority() == 1) {
        processHighPriority(record);  // 高优先级
    } else {
        queue.offer(record);  // 低优先级进入队列
    }
}
```

## 延伸思考

Kafka 的设计，本质上是在**可靠性**和**性能**之间做权衡。

顺序写牺牲了随机访问的灵活性，换取了超高的写入性能。批量处理牺牲了实时性，换取了吞吐量。页缓存牺牲了确定性，换取了内存级的读取速度。

下次使用 Kafka 之前，先问自己：**我需要的是高吞吐还是低延迟？我能接受多少消息丢失？我需要顺序保证吗？**
