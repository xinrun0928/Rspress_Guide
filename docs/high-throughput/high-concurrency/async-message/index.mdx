# 异步与消息处理：抗高峰的核心武器

> [!WARNING]
> 前置知识：本篇文章是异步与消息处理的入口，涵盖异步编程、消息队列、线程池三大方向。建议按顺序阅读，理解同步阻塞与异步解耦的本质区别。

---

想象一个电商下单系统：用户点击「立即购买」，然后页面一直转圈，等待 5 秒、10 秒……

这个转圈的背后，是一个同步调用链：前端 → 网关 → 订单服务 → 库存服务 → 支付服务 → 积分服务。每个服务都是同步调用，响应时间是所有服务耗时之和。

但如果把支付和积分改成异步呢？用户点击「立即购买」，系统立即返回「下单成功」，支付和积分在后台处理。用户看到的是 200ms 的极速响应。

**异步，是高并发系统的性能加速器。**

但异步也带来了新问题：消息丢了怎么办？消息重复了怎么办？消息顺序乱了怎么办？

这一讲，我们来理解异步与消息处理的完整体系。

### 三个方向的关系

- **异步编程是基础**：CompletableFuture、Spring @Async，让你从「同步思维」切换到「异步思维」
- **消息队列是核心**：Kafka、RabbitMQ，通过消息中间件实现真正的异步解耦
- **线程池与并发优化是工具**：合理的线程池配置，是异步编程和消息处理的后台支撑

## 二、异步编程：跳出同步思维

### 同步 vs 异步的本质

同步调用：调用方发起请求后，**一直等待**到被调用方返回结果。

```java
// 同步调用：用户下单，等待所有服务处理完成
public OrderResult placeOrder(Order order) {
    // 1. 扣减库存（同步，等待）
    inventoryService.deduct(order.getItems());
    // 2. 扣减余额（同步，等待）
    paymentService.deduct(order.getUserId(), order.getAmount());
    // 3. 增加积分（同步，等待）
    pointsService.add(order.getUserId(), order.getAmount());
    // 4. 创建订单（同步，等待）
    return orderService.create(order);
}
```

异步调用：调用方发起请求后，**立即返回**，被调用方在后台处理，处理完成后通知调用方。

```java
// 异步调用：立即返回，支付和积分在后台处理
public OrderResult placeOrder(Order order) {
    // 1. 扣减库存（同步，必须先完成才能下单）
    inventoryService.deduct(order.getItems());

    // 2. 异步扣减余额（发送消息，立即返回）
    mqTemplate.convertAndSend("payment", order);

    // 3. 异步增加积分（发送消息，立即返回）
    mqTemplate.convertAndSend("points", order);

    // 4. 创建订单
    return orderService.create(order);
}
```

### 异步的三大优势

| 优势 | 说明 | 示例 |
|---|---|---|
| **削峰** | 突发流量可以通过队列缓冲 | 秒杀场景，订单异步处理 |
| **解耦** | 生产者和消费者互不依赖 | 支付完成通知积分、物流、短信 |
| **异步** | 非关键路径异步执行，提升响应速度 | 发送通知邮件、记录日志 |

## 三、消息队列：异步解耦的核心基础设施

### 消息队列的核心价值

消息队列有三个核心价值，面试中经常被问到：

**削峰**：将突发的流量洪峰缓存为队列，平稳处理。

```text
┌─────────────────────────────────────────────────────────────────────┐
│                        削峰示意                                        │
│                                                                      │
│   瞬时流量 10000 QPS                                                    │
│        │                                                               │
│        ▼                                                               │
│  ┌─────────────┐                                                       │
│  │  消息队列   │  ← 峰值时写入队列                                        │
│  │  (100000)  │                                                       │
│  └──────┬──────┘                                                       │
│         │                                                              │
│         ▼ 消费速度 1000 QPS（系统可承受）                                │
│   平稳消费 → 系统稳定运行                                                │
└─────────────────────────────────────────────────────────────────────┘
```

**解耦**：生产者和消费者解耦，互不感知。

```
传统模式：
  支付服务 → 直接调用 → 积分服务
                      → 物流服务
                      → 短信服务
                      → 积分服务
                      → ...（每加一个消费者，修改一次）

消息队列模式：
  支付服务 → 消息队列 ← 积分服务
                   ← 物流服务
                   ← 短信服务
                   ← 积分服务
                   ← ...（新增消费者，无需修改支付服务）
```

**异步**：将同步调用改为异步，提升响应速度。

### Kafka vs RabbitMQ vs RocketMQ

| 对比项 | Kafka | RabbitMQ | RocketMQ |
|---|---|---|---|
| 设计定位 | 日志、大数据 | 通用消息队列 | 电商、金融场景 |
| 吞吐量 | 百万级 QPS | 万级 QPS | 十万级 QPS |
| 消息顺序 | 支持分区顺序 | 支持单队列顺序 | 支持分区顺序 |
| 延迟 | 毫秒级 | 微秒级 | 毫秒级 |
| 事务消息 | 不支持原生 | 支持 | 支持 |
| 延迟消息 | 不支持（需插件） | 支持（插件） | 支持（原生） |

## 四、线程池与并发优化：异步的后台支撑

### 为什么需要线程池

异步编程的背后，是线程池在支撑。没有线程池，每个异步任务都需要创建新线程，创建和切换成本极高：

```java
// 错误方式：每个任务创建新线程
for (int i = 0; i < 1000; i++) {
    new Thread(() -> doSomething()).start();  // 1000 个线程，系统崩溃
}

// 正确方式：使用线程池
ExecutorService executor = Executors.newFixedThreadPool(16);
for (int i = 0; i < 1000; i++) {
    executor.submit(() -> doSomething());  // 复用 16 个线程
}
```

### 线程池的监控

线程池的监控是高并发系统的必备能力：

```java
// 线程池核心指标
public Map<String, Object> getThreadPoolMetrics(ThreadPoolExecutor executor) {
    Map<String, Object> metrics = new HashMap<>();
    metrics.put("activeCount", executor.getActiveCount());  // 活跃线程数
    metrics.put("poolSize", executor.getPoolSize());        // 当前线程数
    metrics.put("corePoolSize", executor.getCorePoolSize());  // 核心线程数
    metrics.put("maximumPoolSize", executor.getMaximumPoolSize());  // 最大线程数
    metrics.put("queueSize", executor.getQueue().size());  // 队列任务数
    metrics.put("completedTaskCount", executor.getCompletedTaskCount());  // 已完成任务数
    return metrics;
}
```

### Disruptor：高性能队列

Disruptor 是 LMAX 交易所开发的高性能队列，性能比 JDK 的 BlockingQueue 高一个数量级：

```java
// Disruptor 的核心配置
Disruptor<TradeEvent> disruptor = new Disruptor<>(
    TradeEvent::new,           // 事件工厂
    1024 * 1024,               // 缓冲区大小
    DaemonThreadFactory.INSTANCE,  // 线程工厂
    ProducerType.SINGLE,        // 单生产者
    new BlockingWaitStrategy()  // 等待策略
);

// 设置处理器
disruptor.handleEventsWith((event, sequence) -> {
    // 处理事件
    process(event);
});

// 启动
disruptor.start();

// 发布事件
RingBuffer<TradeEvent> ringBuffer = disruptor.getRingBuffer();
ringBuffer.publishEvent((event, ctx) -> {
    event.setData(ctx);
}, tradeData);
```

## 延伸思考

异步与消息处理的学习，本质上是在理解**系统间的关系**。

同步调用是紧耦合，一个系统慢，所有系统都慢。异步解耦是松耦合，每个系统按自己的节奏处理。消息队列是它们之间的缓冲器，让快的等慢的成为历史。

下次设计高并发系统之前，先问自己：**哪些调用必须同步完成？哪些可以异步处理？消息的可靠性要求有多高？**
