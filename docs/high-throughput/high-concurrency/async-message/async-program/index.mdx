# 异步编程：从回调地狱到 CompletableFuture

> [!WARNING]
> 前置知识：本篇文章是异步编程的入口，涵盖同步 vs 异步、Spring @Async、CompletableFuture、响应式编程、WebFlux 五大方向。建议按顺序阅读，理解异步编程的演进历程。

---

在 JDK 5 之前，Java 只有两种处理并发的方式：Thread 和 Runnable。

到了 JDK 5，ExecutorService 让线程池走进主流。但真正的异步革命，是 JDK 8 的 CompletableFuture——它让异步编程从「回调地狱」中解脱出来，用流式的 API 编写异步代码。

而 JDK 21 的虚拟线程，则让异步和同步的边界彻底模糊——你可以用同步的方式写代码，享受异步的性能。

这一讲，我们来完整理解 Java 异步编程的演进历程。

## 二、同步 vs 异步：跳出等待的陷阱

### 同步调用的困境

同步调用的特点是「调用方必须等待被调用方返回才能继续」：

```java
// 同步调用：用户信息 → 订单列表 → 订单详情
public OrderDetail getOrderDetail(Long orderId) {
    Order order = orderService.getOrder(orderId);      // 等待
    User user = userService.getUser(order.getUserId()); // 等待
    Product product = productService.getProduct(order.getProductId()); // 等待
    return new OrderDetail(order, user, product);
}

// 如果每个服务耗时 100ms，总耗时 = 300ms
// 如果 10 个服务，每个 100ms，总耗时 = 1000ms（串行）
```

### 异步调用的优势

异步调用的特点是「调用方发起请求后立即返回，被调用方在后台处理」：

```java
// 异步调用：三个查询并行执行
public CompletableFuture<OrderDetail> getOrderDetail(Long orderId) {
    CompletableFuture<Order> orderFuture = CompletableFuture
        .supplyAsync(() -> orderService.getOrder(orderId));

    // orderFuture 和 userFuture 可以并行吗？不行，user 需要 order 的 userId

    return orderFuture.thenCompose(order ->
        CompletableFuture.allOf(
            CompletableFuture.supplyAsync(() -> userService.getUser(order.getUserId())),
            CompletableFuture.supplyAsync(() -> productService.getProduct(order.getProductId()))
        ).thenApply(ignored ->
            new OrderDetail(order, ...)
        )
    );
}

// order 查询：100ms
// user 和 product 查询：并行，100ms
// 总耗时 = 200ms（减少 33%）
```

## 三、Spring @Async：最简单的异步方案

Spring @Async 是最简单易用的异步方案，只需一个注解：

### 基本用法

```java
// 1. 在启动类或配置类上启用异步
@SpringBootApplication
@EnableAsync
public class Application {}

// 2. 在方法上标注 @Async
@Service
public class NotificationService {

    @Async  // 默认使用 SimpleAsyncTaskExecutor
    public void sendEmail(String to, String content) {
        // 异步执行，不阻塞调用方
        emailClient.send(to, content);
    }

    // 指定线程池
    @Async("emailExecutor")
    public void sendSms(String phone, String content) {
        smsClient.send(phone, content);
    }
}

// 3. 自定义线程池
@Configuration
public class AsyncConfig {

    @Bean("emailExecutor")
    public Executor emailExecutor() {
        return new ThreadPoolExecutor(
            8, 16, 60L, TimeUnit.SECONDS,
            new LinkedBlockingQueue<>(1000),
            new ThreadFactoryBuilder().setNameFormat("email-%d").build(),
            new ThreadPoolExecutor.CallerRunsPolicy()
        );
    }
}
```

### @Async 的坑

| 坑 | 说明 | 解决方案 |
|---|---|---|
| 自调用失效 | 在同一个类中调用 @Async 方法，异步失效 | 注入自身 Bean 或使用 AopContext |
| 返回值为 null | @Async 方法的返回值必须为 void 或 Future | 使用 CompletableFuture |
| 异常丢失 | 异步方法的异常不会传播到调用方 | 使用 Future.get() 或 @AsyncExceptionHandler |

> [!WARNING]
> @Async 最常见的坑是**自调用失效**。在同一个类中直接调用标注了 @Async 的方法，Spring AOP 代理无法拦截，方法会同步执行。

## 四、CompletableFuture：函数式异步编程

### 核心 API

CompletableFuture 是 JDK 8 引入的异步编程利器：

```java
// 创建 CompletableFuture
CompletableFuture<String> future = CompletableFuture
    .supplyAsync(() -> "Hello");  // 有返回值

CompletableFuture<Void> voidFuture = CompletableFuture
    .runAsync(() -> doSomething());  // 无返回值

// 任务编排
CompletableFuture<String> result = CompletableFuture
    .supplyAsync(() -> queryUser(userId))                    // 第一步
    .thenApply(user -> user.getName())                     // 变换
    .thenCompose(name -> queryOrdersByName(name))          // 扁平化
    .thenCombine(queryRecommendations(), (orders, recs) ->  // 组合
        new Dashboard(orders, recs))
    .exceptionally(ex -> {  // 异常处理
        log.error("查询失败", ex);
        return DEFAULT_DASHBOARD;
    });
```

### 常用方法速查

| 分类 | 方法 | 说明 |
|---|---|---|
| 创建 | `supplyAsync` / `runAsync` | 异步执行有返回值 / 无返回值的任务 |
| 变换 | `thenApply` | 变换结果 |
| 扁平化 | `thenCompose` | 当上一步返回 CompletableFuture 时使用 |
| 组合 | `thenCombine` | 并行执行两个任务，合并结果 |
| 异常处理 | `exceptionally` / `handle` | 异常时返回默认值 / 处理异常 |
| 最终处理 | `thenAccept` / `thenRun` | 消费结果 / 不关心结果 |
| 等待 | `join` / `get` | 阻塞等待结果 |

### 并行执行：allOf 与 anyOf

```java
// 场景：同时查询用户信息、订单列表、推荐商品
public CompletableFuture<UserDashboard> getUserDashboard(Long userId) {
    CompletableFuture<User> userFuture = CompletableFuture
        .supplyAsync(() -> userService.getUser(userId));

    CompletableFuture<List<Order>> ordersFuture = CompletableFuture
        .supplyAsync(() -> orderService.getOrders(userId));

    CompletableFuture<List<Product>> recsFuture = CompletableFuture
        .supplyAsync(() -> recommendService.getRecommendations(userId));

    // 等待所有查询完成
    return CompletableFuture.allOf(userFuture, ordersFuture, recsFuture)
        .thenApply(ignored ->
            new UserDashboard(
                userFuture.join(),
                ordersFuture.join(),
                recsFuture.join()
            )
        );
}
```

## 五、响应式编程：Reactor 与 Flux/Mono

### 同步 vs 异步 vs 响应式

| 模型 | 调用方式 | 线程状态 | 适用场景 |
|---|---|---|---|
| 同步调用 | 阻塞等待 | 线程等待 | 简单场景 |
| 异步回调 | 回调函数 | 线程不阻塞 | 复杂场景（回调地狱） |
| Future | Future.get() | 线程等待 | 需要等待结果 |
| CompletableFuture | thenApply | 线程不阻塞 | 任务编排 |
| 响应式 | Flux / Mono | 线程不阻塞 | 流式数据、高并发 |

### Flux 与 Mono

Reactor 是响应式编程的核心库，提供两种类型：

```java
// Mono：0 或 1 个元素
Mono<User> userMono = userService.findById(id);
Mono<List<Order>> ordersMono = orderService.findByUserId(userId);

// Flux：0 或 N 个元素
Flux<Order> ordersFlux = orderService.findAll(userId);
Flux<Order> filteredFlux = ordersFlux
    .filter(order -> order.getAmount() > 100)
    .map(order -> order.withDiscount(0.9))
    .take(10);
```

### WebFlux：响应式 Web 框架

Spring WebFlux 是基于响应式栈的 Web 框架：

```java
@RestController
public class UserController {

    @Autowired
    private UserService userService;

    // 传统方式：同步阻塞
    @GetMapping("/user/{id}")
    public User getUser(@PathVariable Long id) {
        return userService.findById(id);  // 阻塞线程
    }

    // WebFlux 方式：异步非阻塞
    @GetMapping("/user/{id}/flux")
    public Mono<User> getUserFlux(@PathVariable Long id) {
        return userService.findByIdMono(id);  // 不阻塞线程
    }
}
```

## 延伸思考

异步编程的演进，本质上是在解决**等待时间**的问题。

Thread 解决的是「不让主线程等待」，ExecutorService 解决的是「复用线程降低成本」，CompletableFuture 解决的是「让异步代码看起来像同步代码」。每一次演进，都是为了让开发者用更简单的方式写出高效的并发代码。

下次写代码之前，先问自己：**这个调用必须同步完成吗？返回值我立即就需要吗？如果不是，试试异步。**
