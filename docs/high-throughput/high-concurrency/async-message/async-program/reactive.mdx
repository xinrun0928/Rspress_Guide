# Reactor 框架与 Flux/Mono 响应式编程

在传统的数据处理模式中，我们通常是这样的：发送一个请求，等待响应，然后处理结果。如果有 1000 条数据需要处理，我们可能需要发送 1000 次请求，然后依次处理 1000 次响应。

但想象另一种场景：数据像水流一样源源不断地到来，我们需要实时处理这些数据，而不是批量处理后再返回。这就是响应式编程（Reactive Programming）的核心思想——**数据应该像水流一样流动，而不是被"拉取"后再处理**。

这就是 Reactor 框架诞生的背景，也是 Spring WebFlux 选择 Reactor 作为底层的原因。

## 一、响应式编程的起源与背景

### 1.1 为什么需要响应式编程？

在传统的同步阻塞编程模式下，处理大量数据时存在几个核心问题：

```java
// 同步阻塞模式：处理大量数据
public List<Result> processData(List<Data> dataList) {
    List<Result> results = new ArrayList<>();
    for (Data data : dataList) {        // 逐个处理
        Result result = httpClient.call(data);  // 阻塞等待
        results.add(result);
    }
    return results;
}
```

**问题一：线程资源浪费**。每个请求占用一个线程直到响应返回，在等待期间线程完全空闲。

**问题二：背压（Backpressure）无法实现**。生产者发送数据的速度远快于消费者处理的速度时，消费者只能被动等待，最终导致内存溢出或请求丢失。

**问题三：无法处理流式数据**。同步模式适合"请求-响应"模式，但不适合持续到来的数据流。

### 1.2 背压：响应式编程的核心机制

背压是响应式编程最重要的概念之一。简单来说，**背压就是消费者告诉生产者"我处理不过来了，请慢一点"**。

```java
// 没有背压：生产者不受限制地发送数据
// dataStream.onNext(data);  // 无限发送，可能压垮消费者

// 有背压：消费者告诉生产者处理能力
dataStream.request(100);  // 我一次只能处理 100 个
// 生产者收到信号后，只发送 100 个数据
```

在没有背压的系统中，如果生产者发送数据的速度远快于消费者处理的速度，通常有几种应对方式：

| 策略 | 说明 | 后果 |
|---|---|---|
| **丢弃** | 直接丢弃新数据 | 数据丢失 |
| **缓冲** | 缓存数据等待处理 | 内存持续增长 |
| **阻塞** | 生产者等待消费者 | 吞吐量大幅下降 |
| **背压** | 消费者告诉生产者处理能力 | 按需发送，最优解 |

### 1.3 响应式宣言

响应式系统通常遵循响应式宣言（Reactive Manifesto）：

| 特性 | 说明 |
|---|---|
| **Responsive（响应性）** | 系统应该能及时响应用户请求 |
| **Resilient（弹性）** | 系统在部分组件失败时仍能保持响应，通过复制、隔离、委托实现 |
| **Elastic（弹性伸缩）** | 系统在不同负载下都能保持响应，通过水平扩展实现 |
| **Message Driven（消息驱动）** | 组件之间通过异步消息进行通信 |

## 二、Reactor 框架概述

### 2.1 什么是 Reactor？

Reactor 是由 Pivotal 团队开发的响应式编程框架，是 Spring 5 引入的 WebFlux 的底层实现。<Badge text="Reactor 3.0" type="tip" /> Reactor 3.0 完全基于 JDK 8 的函数式 API 设计，提供两种核心类型：`Flux` 和 `Mono`。

```xml
<!-- Maven 依赖 -->
<dependency>
    <groupId>io.projectreactor</groupId>
    <artifactId>reactor-core</artifactId>
    <version>3.5.0</version>
</dependency>
```

### 2.2 Flux vs Mono

| 类型 | 含义 | 元素数量 | 典型场景 |
|---|---|---|---|
| **Flux** | 零到多个元素的数据流 | 0 到 N | 列表查询、消息流、文件读取 |
| **Mono** | 零或一个元素的数据流 | 0 或 1 | 单个对象查询、HTTP 响应 |

```java
// Flux：多个元素
Flux<String> flux = Flux.just("A", "B", "C");
Flux<Integer> fluxFromList = Flux.fromIterable(List.of(1, 2, 3));
Flux<String> fluxFromRange = Flux.range(1, 10).map(String::valueOf);

// Mono：单个元素
Mono<String> mono = Mono.just("Hello");
Mono<User> userMono = userRepository.findById(1L);
Mono<Void> asyncOperation = someService.processAsync();

// Mono 可能为空
Mono<User> optionalUser = Mono.justOrEmpty(userRepository.findById(id));
```

### 2.3 Reactor 的优势

| 优势 | 说明 |
|---|---|
| **非阻塞** | 数据流操作全程非阻塞，不占用线程等待 I/O |
| **背压** | 内置背压机制，消费者控制数据流速 |
| **丰富操作符** | 提供 100+ 操作符处理数据流 |
| **异步数据源** | 支持异步数据源（Reactive Repository） |
| **高吞吐量** | 少量线程处理大量并发请求 |

## 三、Flux 的创建方式

### 3.1 just：直接创建

```java
// 从已知元素创建
Flux<String> flux1 = Flux.just("A", "B", "C");

// 从数组创建
Flux<Integer> flux2 = Flux.fromArray(new Integer[]{1, 2, 3});

// 从 Iterable 创建
Flux<String> flux3 = Flux.fromIterable(List.of("X", "Y", "Z"));

// 从 Stream 创建
Flux<Integer> flux4 = Flux.fromStream(Stream.of(1, 2, 3));
```

### 3.2 generate：同步生成

`generate` 用于同步生成数据，每次调用只能产生一个元素：

```java
// 生成 0 到 9 的数字
Flux<Integer> flux = Flux.generate(
    () -> 0,                                        // 初始状态
    (state, sink) -> {                              // 生成函数
        if (state < 10) {
            sink.next(state);                       // 发射元素
            return state + 1;                       // 更新状态
        } else {
            sink.complete();                        // 完成
            return state;                           // 最终状态（忽略）
        }
    }
);

// 使用 MutableHolder 作为状态容器
Flux<String> flux2 = Flux.generate(
    AtomicInteger::new,                             // 状态初始化
    (state, sink) -> {
        int i = state.getAndIncrement();
        if (i < 5) {
            sink.next("value-" + i);
        } else {
            sink.complete();
        }
    }
);
```

### 3.3 create：异步创建

`create` 支持异步发射多个元素，比 `generate` 更强大：

```java
// 模拟异步事件源
Flux<String> flux = Flux.create(sink -> {
    // 可以在任意线程中调用
    for (int i = 0; i < 10; i++) {
        sink.next("event-" + i);
    }
    sink.complete();

    // 或者异步处理
    someAsyncService.getData(new Callback<String>() {
        @Override
        public void onSuccess(String data) {
            sink.next(data);
        }
        @Override
        public void onComplete() {
            sink.complete();
        }
        @Override
        public void onError(Throwable error) {
            sink.error(error);
        }
    });

    // 注意：不需要返回任何值
});
```

### 3.4 push：单消费者异步

`push` 适用于单消费者场景，比 `create` 更轻量：

```java
EventProcessor<String> processor = new EventProcessor<>();

Flux<String> flux = Flux.push(sink -> {
    processor.setConsumer(sink::next);
    processor.setCompletionHandler(sink::complete);
    processor.setErrorHandler(sink::error);
});
```

### 3.5 创建方式对比

| 方法 | 同步/异步 | 元素数量 | 典型场景 |
|---|---|---|---|
| `just` | 同步 | 固定 | 从已知数据创建 |
| `fromIterable` | 同步 | 固定 | 从集合创建 |
| `generate` | 同步 | 可变 | 状态驱动的同步生成 |
| `create` | 异步 | 可变 | 桥接异步 API |
| `push` | 异步 | 可变 | 单消费者异步事件 |

## 四、Flux 常用操作符

### 4.1 转换类操作符

**map：同步转换**

```java
// 将每个元素转换
Flux.range(1, 5)
    .map(i -> i * 2)                    // 2, 4, 6, 8, 10
    .subscribe(System.out::println);

// 异步转换 - 使用 flatMap
Flux.range(1, 5)
    .flatMap(i -> Mono.just(i * 2))     // 异步执行后再发射
    .subscribe(System.out::println);
```

**flatMap：将每个元素展开为新的 Flux**

```java
// 查询用户列表，然后查询每个用户的订单
Flux<User> users = userRepository.findAll();

Flux<Order> orders = users
    .flatMap(user -> orderRepository.findByUserId(user.getId()))
    .flatMap(order -> productRepository.findById(order.getProductId())
        .map(product -> new OrderDetail(order, product)));
```

### 4.2 过滤类操作符

```java
// filter：过滤元素
Flux.range(1, 10)
    .filter(i -> i % 2 == 0)            // 2, 4, 6, 8, 10
    .subscribe(System.out::println);

// distinct：去重
Flux.just(1, 2, 1, 3, 2, 4)
    .distinct()                          // 1, 2, 3, 4
    .subscribe(System.out::println);

// take：取前 N 个
Flux.range(1, 100)
    .take(5)                             // 1, 2, 3, 4, 5
    .subscribe(System.out::println);

// skip：跳过前 N 个
Flux.range(1, 10)
    .skip(5)                             // 6, 7, 8, 9, 10
    .subscribe(System.out::println);
```

### 4.3 组合类操作符

```java
// zip：合并多个 Flux，按位置配对
Flux<String> names = Flux.just("A", "B", "C");
Flux<Integer> ages = Flux.just(18, 20, 22);

names.zipWith(ages)
    .subscribe(tuple -> System.out.println(tuple.getT1() + "-" + tuple.getT2()));
// 输出: A-18, B-20, C-22

// zip 多个
Mono.zip(Mono.just(1), Mono.just(2), Mono.just(3))
    .map(Tuple3::getT1);  // 返回 1

// merge：合并多个 Flux（按发射顺序）
Flux<String> flux1 = Flux.interval(Duration.ofMillis(10)).take(3).map(i -> "A" + i);
Flux<String> flux2 = Flux.interval(Duration.ofMillis(10)).take(3).map(i -> "B" + i);

Flux.merge(flux1, flux2)
    .subscribe(System.out::println);
// 可能输出: A0, B0, A1, B1, A2, B2（顺序不确定）
```

### 4.4 条件判断类操作符

```java
// defaultIfEmpty：空流默认值
Flux.empty()
    .defaultIfEmpty("default")
    .subscribe(System.out::println);  // 输出: default

// switchIfEmpty：空流切换
userRepository.findById(1L)
    .switchIfEmpty(Mono.error(new UserNotFoundException()))
    .subscribe();

// takeUntil：取元素直到条件满足
Flux.range(1, 10)
    .takeUntil(i -> i > 5)             // 1, 2, 3, 4, 5, 6
    .subscribe(System.out::println);
```

## 五、背压（Backpressure）深度解析

### 5.1 什么是背压？

背压是响应式编程中最重要的概念之一。**它是消费者向生产者反馈处理能力的一种机制**，使生产者不会无限制地向消费者发送数据。

```java
// 没有背压的伪代码
producer.onNext(data);  // 无限制发送
producer.onNext(data);
producer.onNext(data);
// ... 可能导致 OOM

// 有背压的伪代码
consumer.request(100);  // 我一次只能处理 100 个
// producer 收到信号，只发送 100 个
```

### 5.2 背压的策略

Reactor 提供了多种背压策略：

| 策略 | 说明 | 适用场景 |
|---|---|---|
| **BUFFER** | 缓冲所有数据直到消费者处理 | 内存充足，允许一定延迟 |
| **DROP** | 丢弃无法处理的数据 | 只关心最新数据 |
| **LATEST** | 只保留最新数据 | 实时数据监控 |
| **ERROR** | 发送溢出错误信号 | 严格要求不丢数据 |

```java
// 配置背压策略
Flux.range(1, 1000)
    .onBackpressureBuffer(100, // 缓冲区大小
        overflow -> System.out.println("溢出: " + overflow),
        BufferOverflowStrategy.DROP_LATEST)  // 丢弃最新
    .subscribe();

// DROP_OLDEST: 丢弃最老的
// ERROR: 抛出异常
```

### 5.3 request 机制

消费者通过 `request()` 方法告诉生产者自己的处理能力：

```java
// 默认情况下，subscribe() 会请求所有数据
flux.subscribe(new BaseSubscriber<Integer>() {
    @Override
    protected void hookOnSubscribe(Subscription subscription) {
        // 默认 request(Long.MAX_VALUE)，请求所有
        subscription.request(100);  // 手动设置，只请求 100 个
    }

    @Override
    protected void hookOnNext(Integer value) {
        // 处理数据
        process(value);
        // 处理完成后，继续请求
        request(1);  // 每处理一个，请求下一个
    }
});
```

### 5.4 背压在生产中的意义

**保护消费者不被压垮**：在高并发场景下，如果所有请求同时涌入，很可能导致消费者内存溢出。背压机制确保消费者按自己的节奏处理数据。

**实现流量控制**：通过动态调整 `request()` 的数量，实现自适应的流量控制。

**优雅降级**：当系统负载过高时，通过背压策略（如丢弃）实现优雅降级，而不是直接崩溃。

## 六、subscribeOn 与 publishOn

### 6.1 subscribeOn：指定订阅线程

`subscribeOn` 指定**整个管道**的订阅线程位置：

```java
// subscribeOn 决定数据从哪里产生
Flux.range(1, 10)
    .subscribeOn(Schedulers.elastic())  // 数据生成在线程 elastic 中
    .map(i -> {
        System.out.println("map in: " + Thread.currentThread().getName());
        return i * 2;
    })
    .subscribe();

// 输出:
// map in: elastic-2  (数据生成在 elastic 线程)
// map in: elastic-2
// ...
```

### 6.2 publishOn：切换处理线程

`publishOn` 切换**后续操作**的线程位置：

```java
Flux.range(1, 10)
    .map(i -> {
        System.out.println("map1 in: " + Thread.currentThread().getName());
        return i;
    })
    .publishOn(Schedulers.parallel())   // 切换后续线程
    .map(i -> {
        System.out.println("map2 in: " + Thread.currentThread().getName());
        return i * 2;
    })
    .subscribe();

/*
 * 输出:
 * map1 in: main              (主线程)
 * map2 in: parallel-1         (切换到 parallel)
 * map1 in: main
 * map2 in: parallel-1
 * ...
 */
```

### 6.3 对比与总结

| 方法 | 作用范围 | 可调用次数 | 典型场景 |
|---|---|---|---|
| `subscribeOn` | 整个管道（数据源位置） | 多次，但只有第一个生效 | 指定数据生成的线程 |
| `publishOn` | 当前位置之后的所有操作 | 任意次数 | 切换处理线程 |

```java
// 典型用法：数据生成在 IO 线程，处理在 CPU 线程
Flux.fromIterable(largeDataList)
    .subscribeOn(Schedulers.elastic())      // IO 操作在线程池
    .publishOn(Schedulers.parallel())       // 计算切换到并行线程池
    .map(this::heavyComputation)
    .filter(this::heavyFilter)
    .subscribe(this::processResult);
```

### 6.4 常用调度器

| 调度器 | 说明 |
|---|---|
| `Schedulers.immediate()` | 当前线程 |
| `Schedulers.single()` | 单线程（可复用） |
| `Schedulers.newSingle()` | 新的单线程 |
| `Schedulers.parallel()` | 并行线程池（CPU 核心数） |
| `Schedulers.elastic()` | 弹性线程池（I/O 密集型，已废弃） |
| `Schedulers.boundedElastic()` | 有界限的弹性线程池（替代 elastic） |
| `Schedulers.fromExecutorService()` | 自定义线程池 |

## 七、错误处理

### 7.1 onErrorReturn：返回默认值

```java
Flux.just("a", "b", "c")
    .map(s -> {
        if ("b".equals(s)) throw new RuntimeException("Oops!");
        return s.toUpperCase();
    })
    .onErrorReturn("DEFAULT")
    .subscribe(System.out::println);
// 输出: A, DEFAULT
```

### 7.2 onErrorResume：切换到备用 Flux

```java
Flux.just("a", "b", "c")
    .flatMap(s -> {
        if ("b".equals(s)) {
            return Mono.error(new RuntimeException("Oops!"));
        }
        return Mono.just(s.toUpperCase());
    })
    .onErrorResume(ex -> {
        System.out.println("错误: " + ex.getMessage());
        return Flux.just("FALLBACK1", "FALLBACK2");
    })
    .subscribe(System.out::println);
// 输出: A, FALLBACK1, FALLBACK2, C
```

### 7.3 retry：重试机制

```java
// 基础重试
httpClient.get(url)
    .retry(3)  // 最多重试 3 次
    .subscribe();

// 带条件的重试
httpClient.get(url)
    .retryWhen(Retry.backoff(3, Duration.ofSeconds(1))
        .filter(ex -> ex instanceof TimeoutException))  // 只重试超时
    .subscribe();

// 指数退避重试
httpClient.get(url)
    .retryWhen(Retry.backoff(3, Duration.ofMillis(100))
        .maxBackoff(Duration.ofSeconds(5))
        .jitter(0.5))  // 添加抖动
    .subscribe();
```

### 7.4 doOn 系列：副作用钩子

```java
flux
    .doOnSubscribe(s -> log.info("开始订阅"))
    .doOnNext(data -> log.debug("收到数据: {}", data))
    .doOnError(ex -> log.error("发生错误", ex))
    .doOnComplete(() -> log.info("完成"))
    .doOnCancel(() -> log.info("取消"))
    .subscribe();
```

## 八、冷流（Cold）与热流（Hot）

### 8.1 冷流（Cold Flux）

冷流是**对每个订阅者都重新执行**的数据流。每次 `subscribe()` 都会重新产生数据。

```java
// 冷流示例
Flux<String> coldFlux = Flux.just("A", "B", "C")
    .map(s -> {
        System.out.println("生成: " + s);
        return s;
    });

System.out.println("第一次订阅:");
coldFlux.subscribe(s -> System.out.println("收到: " + s));

System.out.println("\n第二次订阅:");
coldFlux.subscribe(s -> System.out.println("收到: " + s));

/*
 * 输出:
 * 第一次订阅:
 * 生成: A, 收到: A
 * 生成: B, 收到: B
 * 生成: C, 收到: C
 *
 * 第二次订阅:
 * 生成: A, 收到: A    (重新生成！)
 * 生成: B, 收到: B
 * 生成: C, 收到: C
 */
```

### 8.2 热流（Hot Flux）

热流是**共享数据源**的数据流，多个订阅者共享同一份数据，新订阅者只能收到订阅之后的数据。

```java
// 使用 ConnectableFlux 创建热流
ConnectableFlux<String> hotFlux = Flux.just("A", "B", "C")
    .map(s -> {
        System.out.println("生成: " + s);
        return s;
    })
    .publish();  // 转为热流

hotFlux.subscribe(s -> System.out.println("订阅者1 收到: " + s));

System.out.println("开始连接:");
hotFlux.connect();  // 开始发射数据

// 第二个订阅者只能收到连接后的数据
hotFlux.subscribe(s -> System.out.println("订阅者2 收到: " + s));

/*
 * 输出:
 * 开始连接:
 * 生成: A, 订阅者1 收到: A
 * 生成: B, 订阅者1 收到: B
 * 生成: C, 订阅者1 收到: C
 * 订阅者2 收到: (无，因为订阅时已发射完毕)
 */
```

### 8.3 热流实战：cache 和 share

```java
// cache：将冷流转为带缓存的热流
Flux<Long> tick = Flux.interval(Duration.ofSeconds(1))
    .cache();  // 缓存最新 N 个元素（默认缓存所有）

tick.subscribe(t -> System.out.println("订阅者1: " + t));

// 2 秒后订阅
Thread.sleep(2000);
tick.subscribe(t -> System.out.println("订阅者2: " + t));

// share = publish().refCount()：自动管理连接
Flux<Long> sharedTick = Flux.interval(Duration.ofSeconds(1))
    .share();

// refCount(2)：至少 2 个订阅者才连接，订阅者数量降为 0 时断开
Flux<Long> refCountedTick = Flux.interval(Duration.ofSeconds(1))
    .publish()
    .refCount(2, Duration.ofSeconds(5));  // 2 个订阅者，5 秒无订阅者则断开
```

### 8.4 冷流与热流对比

| 特性 | 冷流 | 热流 |
|---|---|---|
| **订阅行为** | 每次订阅重新执行 | 共享数据源 |
| **新订阅者** | 收到所有历史数据 | 只收到订阅后的数据 |
| **典型场景** | HTTP 请求、数据库查询 | 消息队列、文件流、用户事件 |

## 九、响应式编程的局限性

### 9.1 学习曲线陡峭

响应式编程的思维方式与传统的命令式编程完全不同，需要重新理解数据流、操作符、背压等概念。

```java
// 传统思维：写"如何做"
List<String> result = new ArrayList<>();
for (String s : list) {
    if (s.length() > 3) {
        result.add(s.toUpperCase());
    }
}

// 响应式思维：写"做什么"
Flux.fromIterable(list)
    .filter(s -> s.length() > 3)
    .map(String::toUpperCase)
    .collectList();
```

### 9.2 不适合简单场景

对于简单的同步操作，使用响应式反而增加复杂度：

```java
// 简单场景：同步调用
String result = userService.getUser(1L).getName();

// 响应式版本：过度设计
userService.getUser(1L)
    .map(User::getName)
    .subscribe(System.out::println);
```

### 9.3 需要完整的响应式栈

整个调用链必须是响应式的，否则会阻塞：

```java
// 错误的做法：混合使用
Flux.just(1, 2, 3)
    .flatMap(i -> blockingService.getData(i))  // 阻塞调用！
    .subscribe();

// 正确的做法：全部使用响应式
Flux.just(1, 2, 3)
    .flatMap(reactiveService::getData)
    .subscribe();
```

### 9.4 调试困难

响应式代码的执行流程不直观，调试困难：

```java
// 难以调试：堆栈信息不清晰
Flux.range(1, 100)
    .flatMap(this::complexOperation)
    .filter(this::complexCondition)
    .map(this::complexTransform)
    .subscribe();

// 建议：使用日志钩子辅助调试
Flux.range(1, 100)
    .flatMap(this::complexOperation)
    .doOnNext(data -> log.debug("after flatMap: {}", data))
    .filter(this::complexCondition)
    .doOnNext(data -> log.debug("after filter: {}", data))
    .subscribe();
```

## 十、实战：Spring WebFlux + Reactor

### 10.1 WebFlux 简介

Spring WebFlux 是 Spring 5 引入的响应式 Web 框架，底层使用 Reactor，支持两种开发模式：

| 模式 | 说明 |
|---|---|
| **注解式（类似 MVC）** | 使用 @Controller、@GetMapping |
| **函数式** | 使用 RouterFunction、HandlerFunction |

```java
// 注解式 WebFlux
@RestController
public class UserController {

    @GetMapping("/users/{id}")
    public Mono<User> getUser(@PathVariable Long id) {
        return userRepository.findById(id);
    }

    @GetMapping("/users")
    public Flux<User> getAllUsers() {
        return userRepository.findAll();
    }
}
```

### 10.2 响应式 Repository

```java
// 响应式 Repository
public interface UserRepository extends ReactiveCrudRepository<User, Long> {

    Mono<User> findByUsername(String username);

    Flux<User> findByAgeGreaterThan(int age);

    Flux<User> findByRole(String role);

    @Query("SELECT * FROM users WHERE age > :age")
    Flux<User> findByAgeRange(@Param("age") int age);
}
```

### 10.3 WebFlux vs MVC

| 维度 | MVC | WebFlux |
|---|---|---|
| **线程模型** | 同步阻塞（一个请求一个线程） | 异步非阻塞（事件驱动） |
| **并发数** | 受线程数限制 | 少量线程处理大量并发 |
| **适用场景** | 简单 CRUD、CPU 密集型 | 高并发 I/O 密集型 |
| **学习曲线** | 低 | 高 |
| **调试难度** | 低 | 高 |

## 总结

Reactor 框架提供了 `Flux` 和 `Mono` 两种核心类型来处理响应式数据流。`Flux` 用于处理 0 到 N 个元素的数据流，`Mono` 用于处理 0 或 1 个元素。响应式编程的核心优势在于：非阻塞、背压机制、丰富的操作符和声明式的数据流处理。

但响应式编程也有其局限性：陡峭的学习曲线、不适合简单场景、需要完整的响应式栈、调试困难。在选择响应式编程时，需要权衡这些因素。

## 面试追问预判

**Q1：Flux 和 Mono 的区别是什么？**

答：Flux 用于处理 0 到 N 个元素的数据流（Mono 可以看作 Flux 的特例）。Flux 支持 `map`、`flatMap`、`filter`、`zip` 等操作符，可以发射多个元素；Mono 主要用于表示单个结果或空结果，操作符相对较少。在实际使用中，Mono 更轻量，如果业务只需要返回 0 或 1 个结果，应优先使用 Mono。

**Q2：什么是背压？如何处理？**

答：背压是消费者向生产者反馈处理能力的机制。在响应式编程中，如果生产者发送数据的速度远快于消费者处理的速度，消费者可以通过 `request()` 方法告诉生产者自己一次能处理多少数据，从而避免内存溢出。Reactor 提供了多种背压策略：BUFFER（缓冲）、DROP（丢弃）、LATEST（只保留最新）、ERROR（抛出异常）。

**Q3：subscribeOn 和 publishOn 的区别？**

答：subscribeOn 指定数据生成的线程位置，整个管道都会在指定线程执行（虽然只有第一个生效）；publishOn 切换后续操作的线程位置，可以多次调用切换线程。典型用法是：数据生成使用 IO 线程（subscribeOn），数据处理使用 CPU 线程（publishOn）。

**Q4：冷流和热流的区别？**

答：冷流对每个订阅者都会重新执行数据生成逻辑，类似 HTTP 请求；热流共享数据源，多个订阅者收到同一份数据，新订阅者只能收到订阅后的数据。典型场景：HTTP 请求是冷流，消息队列是热流。使用 `publish()` + `connect()` 或 `share()` 可以将冷流转为热流。

**Q5：响应式编程有什么局限性？什么时候应该用？**

答：局限性包括：学习曲线陡峭、需要完整响应式栈、不适合简单场景、调试困难。适用场景：1）高并发 I/O 密集型服务；2）需要处理大量流式数据；3）需要背压机制控制数据流速；4）微服务间大量异步调用。不适合的场景：简单的 CRUD 操作、CPU 密集型任务、团队对响应式编程不熟悉。
