# Spring @Async：异步执行与线程池配置

你有没有遇到过这样的场景：系统中有一个发送通知的功能，它需要调用第三方短信服务或邮件服务，平均耗时 200ms。用户点击"注册"按钮后，本应该 100ms 完成的注册流程，却因为这个通知服务拖到了 300ms。用户等得焦虑，你也无可奈何——因为在同步模式下，通知发送必须等待完成才能返回结果。

如果我说，只需要加一个注解，就能让通知发送变成异步执行，让注册接口的响应时间从 300ms 降低到 100ms——你会心动吗？这就是 Spring `@Async` 注解的魅力所在。

## 一、为什么需要 Spring @Async？

### 1.1 同步编程的性能瓶颈

在传统的同步编程模式下，方法的调用和返回是串行的：

```java
@Service
public class UserService {

    public void register(User user) {
        // 1. 保存用户信息（本地操作，约 50ms）
        userRepository.save(user);

        // 2. 发送注册通知（远程调用，约 200ms）
        notificationService.sendRegisterNotification(user);

        // 3. 初始化用户积分（本地操作，约 50ms）
        pointsService.initializePoints(user);

        // 总耗时：50 + 200 + 50 = 300ms
        // 但实际上，步骤 2 和步骤 3 并不需要在步骤 1 之后立即执行
    }
}
```

在这个例子中，`sendRegisterNotification()` 是一个耗时操作，但它并不影响注册流程的核心逻辑。如果能把它放到后台异步执行，注册接口的响应时间可以从 300ms 降低到 100ms。

### 1.2 @Async 的核心价值

Spring `@Async` 是 Spring 框架提供的异步方法执行注解，它的核心价值在于：

**无侵入性**：只需添加一个注解，无需改变现有代码结构。

**线程池管理**：自动将方法执行提交到线程池，避免创建过多线程。

**与 Spring 容器的深度集成**：可以注入 Spring Bean，享受依赖注入、AOP 等特性。

```java
@Service
public class UserService {

    // 只需添加 @Async 注解，方法就会异步执行
    @Async
    public void sendNotification(User user) {
        // 这个方法会在独立的线程中执行
        // 主线程调用后立即返回，不阻塞
        notificationService.send(user);
    }
}
```

## 二、@Async 的基本用法

### 2.1 启用异步支持

使用 `@Async` 的第一步是在配置类上添加 `@EnableAsync` 注解：

```java
@Configuration
@EnableAsync
public class AsyncConfig {
    // 可选：自定义线程池配置
}
```

### 2.2 标注异步方法

 `@Async` 注解可以标注在方法上，也可以标注在接口上：

```java
@Service
public class NotificationService {

    // 无返回值的异步方法
    @Async
    public void sendEmail(String to, String subject, String content) {
        // 实际发送邮件的逻辑
        emailSender.send(to, subject, content);
    }

    // 有返回值的异步方法 - 返回 Future
    @Async
    public Future<String> fetchData(String url) {
        String result = httpClient.get(url);
        return new AsyncResult<>(result);
    }

    // JDK 8+ 推荐使用 CompletableFuture
    @Async
    public CompletableFuture<String> fetchDataAsync(String url) {
        String result = httpClient.get(url);
        return CompletableFuture.completedFuture(result);
    }
}
```

### 2.3 调用异步方法

```java
@Service
public class OrderService {

    private final NotificationService notificationService;

    public void createOrder(Order order) {
        // 保存订单
        orderRepository.save(order);

        // 异步发送通知，立即返回
        notificationService.sendEmail(order.getUserEmail(),
            "订单创建成功", "您的订单已创建");

        // 继续执行其他同步逻辑
        updateInventory(order);
        // ...
    }
}
```

## 三、@Async 的实现原理

### 3.1 BeanPostProcessor + AOP 代理

 `@Async` 的实现原理是 Spring AOP（面向切面编程），具体来说是 `AsyncAnnotationBeanPostProcessor`。它会在 Spring 容器初始化时，扫描所有标注了 `@Async` 的 Bean，并为其创建代理对象。

```java
// @Async 的核心处理类简化流程
public class AsyncAnnotationBeanPostProcessor implements BeanPostProcessor {

    @Override
    public Object postProcessAfterInitialization(Object bean, String beanName) {
        // 扫描 Bean 中的 @Async 方法
        if (this.enabled && this.adviseExists && bean instanceof Runnable) {
            // 为 Bean 创建代理对象
            return createAsyncProxy(bean);
        }
        return bean;
    }
}
```

当调用被 `@Async` 标注的方法时，实际上是调用了代理对象的方法。代理对象会：1）将方法调用封装为 `Runnable` 或 `Callable`；2）提交到线程池执行；3）立即返回。

```
│  ┌─────────────────────────────────────────────────────────┐  │
```

## 四、默认线程池的问题

### 4.1 SimpleAsyncTaskExecutor 的缺陷

Spring 默认使用 `SimpleAsyncTaskExecutor` 作为 `@Async` 的执行器。这个执行器的特点是：**每执行一个任务都会创建新线程，不复用线程**。

```java
// SimpleAsyncTaskExecutor 的关键代码
public class SimpleAsyncTaskExecutor {

    @Override
    public void execute(Runnable task) {
        // 每次都创建新线程！
        Thread thread = new Thread(task);
        thread.start();
    }
}
```

在高并发场景下，如果 `@Async` 方法被频繁调用，`SimpleAsyncTaskExecutor` 会创建大量线程，导致：

- 线程创建和销毁的开销巨大
- 线程数量不可控，可能耗尽系统资源
- 上下文切换频繁，导致性能下降

### 4.2 自定义线程池：方式一

第一种方式是实现 `AsyncConfigurer` 接口：

```java
@Configuration
@EnableAsync
public class AsyncConfig implements AsyncConfigurer {

    @Override
    public Executor getAsyncExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(10);           // 核心线程数
        executor.setMaxPoolSize(50);           // 最大线程数
        executor.setQueueCapacity(200);        // 队列容量
        executor.setThreadNamePrefix("async-"); // 线程名前缀
        executor.setKeepAliveSeconds(60);       // 空闲线程存活时间
        executor.setRejectedExecutionHandler(
            new ThreadPoolExecutor.CallerRunsPolicy()); // 拒绝策略
        executor.initialize();
        return executor;
    }

    @Override
    public AsyncUncaughtExceptionHandler getAsyncUncaughtExceptionHandler() {
        return (ex, method, params) -> {
            log.error("异步方法执行异常: {}, 参数: {}", method.getName(), params, ex);
        };
    }
}
```

### 4.3 自定义线程池：方式二

第二种方式是定义 `ThreadPoolTaskExecutor` Bean，并通过 `@Async` 的 `value` 属性指定：

```java
@Configuration
@EnableAsync
public class AsyncConfig {

    @Bean("asyncExecutor")
    public ThreadPoolTaskExecutor asyncExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(10);
        executor.setMaxPoolSize(50);
        executor.setQueueCapacity(200);
        executor.setThreadNamePrefix("async-");
        executor.setWaitForTasksToCompleteOnShutdown(true);
        executor.setAwaitTerminationSeconds(60);
        executor.initialize();
        return executor;
    }
}

// 在 @Async 中指定线程池
@Service
public class NotificationService {

    @Async("asyncExecutor")
    public void sendEmail(String to, String content) {
        // 使用名为 asyncExecutor 的线程池
    }
}
```

### 4.4 线程池参数配置建议

| 参数 | 配置建议 |
|---|---|
| **corePoolSize** | 根据业务线程数 = 核心线程数 + 队列容量 |
| **maxPoolSize** | CPU 密集型：CPU 核数 + 1；IO 密集型：CPU 核数 * 2 |
| **queueCapacity** | 根据任务量设置，拒绝策略建议用 CallerRunsPolicy |
| **threadNamePrefix** | 建议添加，便于日志追踪 |

```java
// IO 密集型任务的线程池配置示例
ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
executor.setCorePoolSize(Runtime.getRuntime().availableProcessors() * 2);
executor.setMaxPoolSize(Runtime.getRuntime().availableProcessors() * 4);
executor.setQueueCapacity(500);
executor.setThreadNamePrefix("io-async-");
```

## 五、@Async 的异常处理

### 5.1 默认异常处理

被 `@Async` 标注的方法如果在执行过程中抛出异常，默认情况下异常会被吞掉，不会传播到调用方。如果需要处理异常，有几种方式：

### 5.2 使用 Future 捕获异常

```java
@Async
public Future<String> callRemoteService(String param) {
    try {
        String result = httpClient.call(param);
        return AsyncResult.forSuccess(result);
    } catch (Exception e) {
        return AsyncResult.forFailure(e);
    }
}

// 调用方处理
public void doSomething() {
    Future<String> future = notificationService.callRemoteService("test");
    try {
        String result = future.get(5, TimeUnit.SECONDS);
    } catch (Exception e) {
        log.error("异步调用失败", e);
    }
}
```

### 5.3 使用 AsyncUncaughtExceptionHandler

```java
@Configuration
@EnableAsync
public class AsyncConfig implements AsyncConfigurer {

    @Override
    public AsyncUncaughtExceptionHandler getAsyncUncaughtExceptionHandler() {
        return new SimpleAsyncUncaughtExceptionHandler() {
            @Override
            public void handleUncaughtException(Throwable ex, Method method, Object... params) {
                log.error("异步方法 {} 执行异常, 参数: {}",
                    method.getName(), Arrays.toString(params), ex);

                // 发送告警通知
                alertService.sendAlert("异步执行异常", ex.getMessage());
            }
        };
    }
}
```

### 5.4 使用 @AsyncExceptionHandler

如果使用 Spring 4.1+ 的注解式异常处理，可以自定义异常处理器：

```java
@Component
public class CustomAsyncExceptionHandler implements AsyncUncaughtExceptionHandler {

    @Override
    public void handleUncaughtException(Throwable ex, Method method, Object... params) {
        log.error("Async method {} threw exception", method.getName(), ex);

        if (ex instanceof ServiceException) {
            // 处理业务异常
            handleBusinessException((ServiceException) ex);
        } else {
            // 处理其他异常
            handleSystemException(ex);
        }
    }
}
```

## 六、@Async 与事务的关系：常见坑点

### 6.1 事务在异步中无效

这是 `@Async` 最常见的坑之一：`@Async` 方法在事务中调用时，事务是无效的！

```java
@Service
public class UserService {

    @Transactional
    public void register(User user) {
        userRepository.save(user);  // 在事务中执行

        // 调用异步方法
        sendNotificationAsync(user);  // 注意：这里的事务对异步方法无效！
    }

    @Async
    public void sendNotificationAsync(User user) {
        // 这个方法在独立的线程中执行
        // 它不会继承外层事务！
        notificationService.send(user);
    }
}
```

**原因**：事务是通过 ThreadLocal 绑定的。当 `@Async` 方法在新线程中执行时，它无法访问主线程的 ThreadLocal，因此事务上下文丢失。

### 6.2 解决方案

**方案一：主动传播事务上下文**

```java
@Async
@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
public void sendNotificationAsync(User user) {
    // 使用 TransactionalEventListener 确保事件在事务提交后才处理
    notificationService.send(user);
}
```

**方案二：在异步方法中开启新事务**

```java
@Async
@Transactional(propagation = Propagation.REQUIRES_NEW)
public void sendNotificationAsync(User user) {
    // 在异步方法中开启新事务
    notificationService.send(user);
}
```

**方案三：使用 @TransactionalEventListener**

```java
// 发布事件
@Service
public class UserService {

    private final ApplicationEventPublisher eventPublisher;

    @Transactional
    public void register(User user) {
        userRepository.save(user);
        // 发布事件（在事务中）
        eventPublisher.publishEvent(new UserRegisteredEvent(user));
    }
}

// 监听事件（在事务提交后执行）
@Component
public class NotificationListener {

    @Async
    @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
    public void handleUserRegistered(UserRegisteredEvent event) {
        // 只在事务成功提交后才发送通知
        notificationService.send(event.getUser());
    }
}
```

## 七、@Async 与 SecurityContext 的关系

### 7.1 子线程无法继承 SecurityContext

和事务类似，`SecurityContext` 也是通过 ThreadLocal 存储的。当 `@Async` 方法在新线程中执行时，无法访问主线程的 SecurityContext。

```java
@Service
public class OrderService {

    @Async
    public void createOrder(Order order) {
        // 获取当前登录用户 - 在异步线程中会返回 null！
        Authentication auth = SecurityContextHolder.getContext().getAuthentication();
        // auth 为 null，因为 SecurityContext 没有被继承
    }
}

public void registerOrder(Order order) {
    // 假设这是 Controller 方法，已登录用户
    order.setUserId(getCurrentUserId());  // 需要手动传递用户 ID

    orderService.createOrder(order);  // @Async 方法
}
```

### 7.2 解决方案

**方案一：手动传递安全上下文**

```java
@Async
public void createOrder(Order order, Long userId) {
    // 传递用户 ID，不依赖 SecurityContext
    order.setUserId(userId);
    // ...
}
```

**方案二：配置 SecurityContext 传播**

```java
@Configuration
@EnableAsync
public class AsyncConfig implements AsyncConfigurer {

    @Bean
    public TaskExecutor taskExecutor() {
        DelegatingSecurityContextAsyncTaskExecutor executor =
            new DelegatingSecurityContextAsyncTaskExecutor(
                new ThreadPoolTaskExecutor());
        return executor;
    }
}
```

**方案三：使用 SecurityContextHolder 配置**

```java
SecurityContextHolder.setStrategyName(
    SecurityContextHolder.MODE_INHERITABLETHREADLOCAL);
```

## 八、@Async 的执行顺序问题

### 8.1 异步方法不保证执行顺序

由于线程池的调度策略，`@Async` 方法的执行顺序是不确定的：

```java
@Service
public class NotificationService {

    @Async
    public void sendStep1() { /* ... */ }

    @Async
    public void sendStep2() { /* ... */ }

    @Async
    public void sendStep3() { /* ... */ }
}

// 调用方
public void process() {
    notificationService.sendStep1();  // 可能最后执行
    notificationService.sendStep2();  // 可能最先执行
    notificationService.sendStep3();  // 可能第二个执行
}
```

### 8.2 保证顺序的方法

**使用 @Order 注解**

```java
@Configuration
@EnableAsync
public class AsyncConfig {
    // 配置同步执行器
    @Bean("syncExecutor")
    public TaskExecutor syncExecutor() {
        return new SyncTaskExecutor();
    }
}

@Async("syncExecutor")  // 使用同步执行器
public void sendStep1() { /* ... */ }
```

**使用 ApplicationEventPublisher + @Order**

```java
@Async
@EventListener
@Order(1)
public void handleStep1(ProcessEvent event) { /* ... */ }

@Async
@EventListener
@Order(2)
public void handleStep2(ProcessEvent event) { /* ... */ }
```

**使用 CompletableFuture 链式调用**

```java
public void process() {
    CompletableFuture.runAsync(() -> step1())
        .thenRun(() -> step2())
        .thenRun(() -> step3());
}
```

## 九、Spring Boot 中的自动配置

### 9.1 自动配置的线程池

Spring Boot 会自动配置一个 `TaskExecutionAutoConfiguration`，如果没有手动定义线程池 Bean，会使用默认配置：

```yaml
# application.yml
spring:
  task:
    execution:
      pool:
        core-size: 8                    # 默认 8
        max-size: 16                    # 默认 Integer.MAX_VALUE
        queue-capacity: 100             # 默认 Integer.MAX_VALUE
        keep-alive: 60s                 # 默认 60s
      thread-name-prefix: task-         # 默认 task-
```

### 9.2 自定义配置

```yaml
spring:
  task:
    execution:
      pool:
        core-size: 20
        max-size: 100
        queue-capacity: 500
      thread-name-prefix: async-
```

## 十、@Async 失效的常见原因

| 原因 | 说明 | 解决方案 |
|---|---|---|
| **同类内部调用** | 在同一个 Bean中调用 @Async 方法，代理不生效 | 注入自身 Bean，或通过 ApplicationContext 调用 |
| **方法不是 public** | 非 public 方法无法被代理 | 将方法改为 public |
| **没有 @EnableAsync** | 没有启用异步支持 | 添加 @EnableAsync 注解 |
| **方法包含锁** | synchronized 与 @Async 冲突 | 使用分布式锁或其他方案 |
| **返回类型问题** | 不支持返回基本类型的包装类 | 使用 Future、CompletableFuture |

```java
// 常见错误：同类内部调用
@Service
public class UserService {

    public void register(User user) {
        // 错误：同类内部调用，@Async 代理不生效！
        sendNotification(user);
    }

    @Async
    public void sendNotification(User user) {
        // 实际是同步执行的
    }
}

// 正确做法：注入自身代理
@Service
public class UserService {

    @Autowired
    private UserService self;  // 注入自身代理

    public void register(User user) {
        // 通过代理调用，@Async 生效
        self.sendNotification(user);
    }

    @Async
    public void sendNotification(User user) {
        // 异步执行
    }
}
```

## 总结

Spring `@Async` 是实现异步编程最简单的方式，只需一个注解就能将方法异步化。但它的"简单"也带来了不少坑：默认线程池的性能问题、事务上下文丢失、SecurityContext 无法继承、执行顺序不确定等。

使用 `@Async` 时，必须注意以下几点：自定义线程池以避免线程资源耗尽、处理异步方法的异常、根据业务需求决定是否需要传播事务上下文、必要时手动传递用户上下文。

## 面试追问预判

**Q1：@Async 的实现原理是什么？**

答：@Async 基于 Spring AOP 实现。核心是 `AsyncAnnotationBeanPostProcessor`，它会在容器初始化时扫描所有标注了 @Async 的 Bean，为其创建代理对象。当调用被代理的方法时，实际是将方法封装为 Runnable 或 Callable，提交到线程池执行，调用方立即返回。

**Q2：@Async 有什么常见坑？如何避免？**

答：常见坑包括：1）同类内部调用导致代理失效，需注入自身 Bean；2）默认使用 SimpleAsyncTaskExecutor 不复用线程，需自定义线程池；3）事务上下文丢失，因为事务通过 ThreadLocal 存储，解决方法是使用 @TransactionalEventListener；4）SecurityContext 无法继承到子线程，可在配置中使用 DelegatingSecurityContextAsyncTaskExecutor。

**Q3：@Async 和线程池的关系是什么？如何选择线程池参数？**

答：@Async 方法默认通过线程池执行。线程池参数选择：CPU 密集型任务，核心线程数设置为 CPU 核数 + 1；IO 密集型任务，核心线程数设置为 CPU 核数 * 2。需要根据任务的并发量和耗时合理设置队列容量和最大线程数。

**Q4：@Async 和 CompletableFuture 有什么区别？**

答：@Async 是 Spring 提供的注解式异步方案，使用简单但控制粒度较粗，适合无侵入地将现有同步方法异步化。CompletableFuture 是 JDK 8 提供的 Future 扩展，支持链式调用、组合操作、异常处理等，更灵活但需要显式编写代码。两者可以结合使用。
