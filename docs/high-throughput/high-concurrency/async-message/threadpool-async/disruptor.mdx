# Disruptor 高性能队列的原理与实践

## 场景切入：每秒百万订单的挑战

2010 年，伦敦金融交易所（LMAX）的架构师们在思考一个问题：如何让单机处理能力突破每秒百万量级？

彼时的技术方案普遍面临瓶颈：

- 数据库：磁盘 IO 是噩梦，事务开销巨大
- 传统 MQ：锁竞争严重，GC 压力山
- 内存队列：JDK 的 ArrayBlockingQueue 在高并发下性能急剧下降

他们决定从零开始，设计一个全新的高性能队列。

18 个月后，Disruptor 诞生了。在普通服务器上，它达到了**每秒 600 万次消息处理**，延迟低至纳秒级。

这个后来成为 LMAX 架构核心的组件，彻底改变了我们对并发处理的认知。

本文将深入剖析 Disruptor 的设计原理和实现机制。

---

## 一、为什么需要 Disruptor

### 1.1 JDK 队列的性能问题

JDK 提供了丰富的队列实现：`ArrayBlockingQueue`、`LinkedBlockingQueue`、`LinkedTransferQueue` 等。但在高并发场景下，它们都面临相同的问题：**锁竞争**和**缓存失效**。

**ArrayBlockingQueue 的实现**：

```java
public class ArrayBlockingQueue<E> implements BlockingQueue<E> {
    final Object[] items;
    final ReentrantLock lock = new ReentrantLock();  // 全局锁
    
    public void put(E e) throws InterruptedException {
        lock.lock();  // 每次入队都要竞争锁
        try {
            items[putIndex] = e;
            putIndex = (putIndex + 1) % items.length;
            count++;
        } finally {
            lock.unlock();
        }
    }
    
    public E take() throws InterruptedException {
        lock.lock();  // 每次出队也要竞争锁
        try {
            E item = items[takeIndex];
            takeIndex = (takeIndex + 1) % items.length;
            count--;
            return item;
        } finally {
            lock.unlock();
        }
    }
}
```

**问题一：入队和出队竞争同一把锁**

当生产者往队列放数据，同时消费者从队列取数据时，两者需要竞争同一把锁。在多核 CPU 上，这意味着：

```
CPU 1 (生产者) ──竞争锁──▶ [队列] ◀──竞争锁── CPU 2 (消费者)
                                    │
                                    ▼
                           等待、唤醒、上下文切换
```

**问题二：伪共享导致缓存失效**

CPU 读取数据时，会将相邻的多个字节加载到缓存行（Cache Line，通常是 64 字节）。如果两个线程访问的数据恰好在同一缓存行，一个线程修改数据会导致另一个线程的缓存行失效。

```java
public class ArrayBlockingQueue<E> {
    Object[] items;
    int putIndex;    // 被生产者频繁修改
    int takeIndex;   // 被消费者频繁修改
    int count;       // 被两者修改
}
```

这三个字段可能在同一缓存行，任何一个被修改都会导致其他线程的缓存失效。

### 1.2 性能对比数据

Disruptor 官方给出了惊人的对比数据：

```
单线程吞吐测试（百万消息/秒）：
ArrayBlockingQueue:     0.4 - 1.0 M ops/sec
LinkedBlockingQueue:    0.6 - 1.2 M ops/sec
Disruptor (单生产者):   4.0 - 6.0 M ops/sec  ← 性能提升 5-10 倍

延迟测试（纳秒）：
ArrayBlockingQueue:     5000 - 15000 ns
LinkedBlockingQueue:    8000 - 20000 ns
Disruptor:              50 - 500 ns  ← 延迟降低 10-100 倍
```

### 1.3 Disruptor 的设计目标

LMAX 团队设定了三个核心目标：

1. **无锁设计**：使用 CAS（Compare-And-Swap）替代锁，消除锁竞争
2. **缓存友好**：消除伪共享，让数据在缓存行中对齐
3. **预分配内存**：避免 GC，通过环形缓冲区复用对象

## 二、RingBuffer 环形缓冲区

### 2.1 环形缓冲区的核心思想

RingBuffer 是 Disruptor 的核心数据结构，本质上是一个**预分配大小的环形数组**：

```java
// RingBuffer 示意图
┌──────────────────────────────────────────────────────┐
│                      RingBuffer                        │
│  ┌────┬────┬────┬────┬────┬────┬────┬────┐         │
│  │ 0  │ 1  │ 2  │ 3  │ 4  │ 5  │ 6  │ 7  │  ...   │
│  └────┴────┴────┴────┴────┴────┴────┴────┘         │
│    ▲                                     │            │
│    │                                     │            │
│    └──────────── 循环回到起始位置 ─────────┘            │
│                                                       │
│  容量: 8 (必须是 2 的幂次: 8, 16, 32, 64, ...)        │
└──────────────────────────────────────────────────────┘
```

### 2.2 为什么是环形

传统的队列需要动态分配内存，每次入队都可能有 GC 开销。环形缓冲区的巧妙之处在于：

1. **预分配**：创建时就分配好所有槽位，不会有动态内存分配
2. **循环覆盖**：新数据覆盖旧数据，不需要清理或释放
3. **序号定位**：通过序号计算位置，实现 O(1) 访问

```java
public class RingBuffer<T> {
    private final Object[] entries;
    private final int mask;  // = size - 1，用于快速取模
    
    public RingBuffer(int size) {
        // size 必须是 2 的幂次
        this.entries = new Object[size];
        this.mask = size - 1;  // 2^4 - 1 = 15，二进制: 01111
    }
    
    // 通过序号计算槽位
    public int index(long sequence) {
        return (int) (sequence & mask);
        // 例如: 17 & 15 = 1, 16 & 15 = 0, 15 & 15 = 15
    }
    
    // 获取槽位数据
    public T get(long sequence) {
        return (T) entries[index(sequence)];
    }
}
```

### 2.3 序号与槽位的映射

序号是递增的，可以很大（远超数组长度），但槽位是通过掩码运算得出的：

```
序号序列: ... 14, 15, 16, 17, 18, 19, 20, 21 ...
          ↓   ↓   ↓   ↓   ↓   ↓   ↓   ↓
槽位映射: ...  6,  7,  0,  1,  2,  3,  4,  5  (假设 size=8)
```

这意味着：
- **序号永远递增**：可以精确追踪消息的处理进度
- **槽位循环复用**：数组空间被高效利用
- **无需垃圾回收**：预分配的对象被复用，不产生 GC

## 三、CAS 与无锁设计

### 3.1 什么是 CAS

CAS（Compare-And-Swap）是 CPU 提供的原子指令：

```java
// CAS 的语义
boolean compareAndSwap(位置, 期望值, 新值) {
    if (内存位置的值 == 期望值) {
        内存位置的值 = 新值;
        return true;
    }
    return false;
}
```

JDK 的 `AtomicLong` 底层使用 CAS 实现：

```java
public class AtomicLong extends Number implements java.io.Serializable {
    private volatile long value;
    
    public final long incrementAndGet() {
        return unsafe.getAndAddLong(this, VALUE_OFFSET, 1L) + 1L;
    }
}

// 等价于：
public long incrementAndGet() {
    long current = value;
    while (!CAS(value, current, current + 1)) {
        current = value;  // CAS 失败，重新读取
    }
    return current + 1;
}
```

### 3.2 Disruptor 的序列号机制

Disruptor 使用 `Sequence` 类管理序号：

```java
public class Sequence extends RhsPadding {
    private volatile long value = INITIAL_CURSOR_VALUE;
    
    // CAS 更新
    public boolean compareAndSet(long expectedSequence, long newSequence) {
        return UNSAFE.compareAndSwapLong(this, VALUE_OFFSET, expectedSequence, newSequence);
    }
    
    // 递增
    public long incrementAndGet() {
        return UNSAFE.getAndAddLong(this, VALUE_OFFSET, 1L) + 1L;
    }
}
```

### 3.3 生产者的写入流程

单生产者写入 Disruptor 的流程：

```java
// 1. 获取下一个可写入序号
long nextSequence = ringBuffer.next();  // CAS: cursor + 1

// 2. 获取序号对应的槽位
Object event = ringBuffer.get(nextSequence);

// 3. 填充数据
OrderEvent orderEvent = (OrderEvent) event;
orderEvent.setOrderId(...);
orderEvent.setAmount(...);

// 4. 发布序号（告诉消费者可以读取了）
ringBuffer.publish(nextSequence);
```

**next() 方法的内部实现**：

```java
public long next() {
    // 当前游标 + 1
    long current = cursor.get();
    long next = current + 1;
    
    // 循环 CAS，直到成功
    while (!cursor.compareAndSet(current, next)) {
        current = cursor.get();  // 失败，重试
        next = current + 1;
        
        // 队列满了，等待
        while (next > availableBuffer) {
            // 忙等待或让出 CPU
            LockSupport.parkNanos(1);
        }
    }
    return next;
}
```

### 3.4 多生产者模式

多生产者时，需要解决序号竞争问题。Disruptor 使用**独享序号槽位**的策略：

```java
public long next() {
    long current;
    long next;
    
    do {
        current = cursor.get();
        next = current + 1;
        
        // 检查队列是否满
        if (next > capacity) {
            // 等待所有可用的槽位被消费
            waitFor(next - availableBuffer);
        }
    } while (!cursor.compareAndSet(current, next));  // 竞争序号
    
    return next;
}
```

多生产者发布时，需要**声明序号是可用的**：

```java
public void publish(long sequence) {
    // 标记这个序号可用（消费端会检查这个标记）
    setAvailable(sequence);
    
    // 唤醒等待的消费者
    waiter.signal();
}
```

## 四、缓存行填充

### 4.1 伪共享问题详解

现代 CPU 的缓存结构：

```
CPU Core 0                    CPU Core 1
┌────────────┐               ┌────────────┐
│  L1 Cache  │               │  L1 Cache  │
│   64 KB    │               │   64 KB    │
└────────────┘               └────────────┘
          │                           │
          ▼                           ▼
┌─────────────────────────────────────────────┐
│               L2 Cache (Shared)               │
│                   256 KB                      │
└─────────────────────────────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────────┐
│               L3 Cache (Shared)               │
│                   8 MB                        │
└─────────────────────────────────────────────┘
                    │
                    ▼
              主内存 (DDR)
```

缓存行是 CPU 与内存交换数据的最小单位，通常是 **64 字节**。

**伪共享场景**：

```java
// 两个线程分别修改这两个变量
class Foo {
    long a;  // Thread 1 修改
    long b;  // Thread 2 修改
}
```

假设 `a` 和 `b` 恰好在同一缓存行。Thread 1 修改 `a` 后，Thread 2 的缓存行失效，必须重新从主内存加载。

### 4.2 Disruptor 的解决方案：缓存行填充

Disruptor 在 Sequence 类前后各填充 7 个 long 变量（每个 8 字节，共 56 字节）：

```java
public class Sequence extends RhsPadding {
    private volatile long value;
}

class RhsPadding {
    protected long p1, p2, p3, p4, p5, p6, p7;  // 前填充
}

class LhsPadding {
    protected long p1, p2, p3, p4, p5, p6, p7;  // 后填充
}
```

**内存布局**：

```
┌────────────────────────────────────────────────────────┐
│ Sequence 对象（至少 64 字节 = 8 字节 * 8）               │
│  ┌──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┐ │
│  │ p1   │ p2   │ p3   │ p4   │ p5   │ p6   │ p7   │value │ │
│  │ (56B)│      │      │      │      │      │      │ (8B) │ │
│  └──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┘ │
│                                                        │
│  前后各 56 字节填充，确保 Sequence 对象独占一个缓存行      │
└────────────────────────────────────────────────────────┘
```

### 4.3 RingBuffer 的缓存友好设计

```java
public final class RingBuffer<E> extends RingBufferFields<E> {
    // 填充 + Sequence 组合
    private final Sequence cursor = new Sequence();      // 生产者位置
    private final Sequence[] gatingSequences;              // 消费者位置数组
    
    // Event 数组（E 的对象引用）
    private final Object[] entries;
}
```

消费者序列和生产者序列**不会在同一个缓存行**，避免互相干扰。

## 五、等待策略

### 5.1 为什么需要等待策略

当消费者处理速度跟不上生产者时，消费者需要等待。有多种等待策略可选，各有权衡：

- **忙等待**：CPU 占用高，延迟最低
- **睡眠**：CPU 占用低，但有延迟
- **阻塞**：CPU 占用低，支持中断唤醒

### 5.2 AvailableBuffer 策略

这是 Disruptor 默认的等待策略，结合了自旋和阻塞：

```java
public class PhasedBackoffWaitStrategy implements WaitStrategy {
    private final long spinTimeInNanos;
    private final long yieldTimeInNanos;
    
    @Override
    public long waitFor(long sequence, Sequence cursorSequence, 
                        Sequence dependentSequence, SequenceBarrier barrier) {
        long availableSequence;
        long startTime = 0;
        int counters = 0;
        
        // 第一阶段：自旋
        while ((availableSequence = dependentSequence.get()) < sequence) {
            if (counters < 4) {
                //忙等待
            } else if (counters < 8) {
                // 让出 CPU
                Thread.yield();
            } else {
                // 阻塞
                LockSupport.parkNanos(spinTimeInNanos);
            }
            counters++;
        }
        return availableSequence;
    }
}
```

### 5.3 常用等待策略对比

| 策略 | CPU 占用 | 延迟 | 适用场景 |
|---|---|---|---|
| BusySpin | 极高 | 最低 | 超低延迟（微秒级） |
| Yielding | 高 | 低 | 低延迟，多线程竞争 |
| Sleeping | 中 | 中低 | 平衡延迟和 CPU |
| Blocking | 低 | 中 | 高吞吐，延迟不敏感 |

**配置示例**：

```java
// 使用 Blocking 策略
Disruptor<OrderEvent> disruptor = new Disruptor<>(
    OrderEvent::new,
    1024 * 1024,
    DaemonThreadFactory.INSTANCE,
    ProducerType.SINGLE,
    new BlockingWaitStrategy()  // 等待策略
);
```

## 六、事件处理器

### 6.1 BatchEventProcessor

`BatchEventProcessor` 是 Disruptor 的核心消费者，一次处理一批可用的消息：

```java
public class BatchEventProcessor<T> implements EventProcessor {
    private final RingBuffer<T> ringBuffer;
    private final SequenceBarrier barrier;
    private final EventHandler<T> handler;
    private long nextSequence;
    
    @Override
    public void run() {
        // 初始化
        nextSequence = sequence.get() + 1;
        
        // 批量处理循环
        while (running) {
            // 获取一批可用的序号
            long availableSequence = barrier.waitFor(nextSequence);
            
            // 批量处理
            while (nextSequence <= availableSequence) {
                T event = ringBuffer.get(nextSequence);
                handler.onEvent(event, nextSequence, nextSequence == availableSequence);
                nextSequence++;
            }
            
            // 更新消费位置
            sequence.set(availableSequence);
        }
    }
}
```

### 6.2 WorkHandler 多消费者模式

使用 `WorkHandler` 可以实现多个消费者并发处理：

```java
// 定义 WorkHandler
public class OrderEventHandler implements WorkHandler<OrderEvent> {
    @Override
    public void onEvent(OrderEvent event) throws Exception {
        // 处理订单
        processOrder(event);
    }
}

// 创建 Disruptor（使用 WorkerPool）
Disruptor<OrderEvent> disruptor = new Disruptor<>(
    OrderEvent::new,
    1024,
    DaemonThreadFactory.INSTANCE,
    ProducerType.MULTI,
    new BlockingWaitStrategy()
);

// 使用 WorkerPool 处理
WorkerPool<OrderEvent> workerPool = disruptor.createWorkerPool(
    new OrderEventHandler(),
    new OrderEventHandler(),
    new OrderEventHandler()
);
workerPool.start();
```

### 6.3 消费者依赖图

Disruptor 支持复杂的消费者依赖关系：

```java
// 创建依赖图
//              ┌── Handler A
// 生产者 ──┬──┼── Handler B
//          │  └── Handler C
//          │
//          └── Barrier ─── Handler D（等待 A、B、C 完成）

// 配置依赖
RingBuffer<OrderEvent> ringBuffer = disruptor.getRingBuffer();

SequenceBarrier barrierAB = ringBuffer.addGatingSequences(
    sequenceA, sequenceB);

EventProcessor processorC = new BatchEventProcessor<>(
    ringBuffer, barrierAB, handlerC);
```

## 七、性能调优

### 7.1 RingBuffer 大小选择

RingBuffer 大小必须是 **2 的幂次**：

```java
// 正确：2 的幂次
bufferSize = 1024;   // 2^10
bufferSize = 4096;   // 2^12
bufferSize = 16384;  // 2^14

// 错误：会导致序号映射错误
bufferSize = 1000;
```

**大小选择原则**：
- 足够大，能容纳生产和消费的峰值差
- 不能太大，否则预分配内存过多
- 通常选择 1024 到 65536 之间

### 7.2 最佳实践：订单处理系统

```java
// 定义 Event
public class OrderEvent {
    private long orderId;
    private BigDecimal amount;
    private String status;
    
    public OrderEvent() {}
    
    public static OrderEvent newInstance() {
        return new OrderEvent();
    }
}

// 定义 Handler
public class OrderEventHandler implements EventHandler<OrderEvent> {
    @Override
    public void onEvent(OrderEvent event, long sequence, boolean endOfBatch) {
        // 业务处理
        processPayment(event);
        updateInventory(event);
        sendNotification(event);
    }
}

// 创建 Disruptor
Disruptor<OrderEvent> disruptor = new Disruptor<>(
    OrderEvent::new,           // 工厂方法
    16384,                     // RingBuffer 大小
    DaemonThreadFactory.INSTANCE,
    ProducerType.SINGLE,       // 单生产者
    new BusySpinWaitStrategy() // 低延迟策略
);

// 注册 Handler
disruptor.handleEventsWith(new OrderEventHandler());

// 启动
disruptor.start();

// 生产消息
RingBuffer<OrderEvent> ringBuffer = disruptor.getRingBuffer();
long sequence = ringBuffer.next();
try {
    OrderEvent event = ringBuffer.get(sequence);
    event.setOrderId(orderId);
    event.setAmount(amount);
} finally {
    ringBuffer.publish(sequence);
}
```

## 八、与其他队列的对比

### 8.1 ArrayBlockingQueue vs Disruptor

| 维度 | ArrayBlockingQueue | Disruptor |
|---|---|---|
| 锁机制 | ReentrantLock（阻塞） | CAS（无锁） |
| 消费通知 | `signal()` 唤醒 | 序列号检查 |
| 内存分配 | 入队时分配 | 预分配，零分配 |
| 缓存行 | 共享 | 填充隔离 |
| 延迟 | 微秒级 | 纳秒级 |
| 吞吐量 | 万级 | 百万级 |

### 8.2 性能测试代码

```java
public class QueueBenchmark {
    public static void main(String[] args) throws Exception {
        int iterations = 1_000_000;
        int bufferSize = 1024;
        
        // 测试 ArrayBlockingQueue
        testArrayBlockingQueue(iterations, bufferSize);
        
        // 测试 Disruptor
        testDisruptor(iterations, bufferSize);
    }
    
    private static void testArrayBlockingQueue(int iterations, int bufferSize) {
        BlockingQueue<Long> queue = new ArrayBlockingQueue<>(bufferSize);
        
        long start = System.nanoTime();
        
        Thread producer = new Thread(() -> {
            for (long i = 0; i < iterations; i++) {
                try {
                    queue.put(i);
                } catch (InterruptedException e) {
                    break;
                }
            }
        });
        
        Thread consumer = new Thread(() -> {
            long sum = 0;
            for (long i = 0; i < iterations; i++) {
                try {
                    sum += queue.take();
                } catch (InterruptedException e) {
                    break;
                }
            }
        });
        
        producer.start();
        consumer.start();
        producer.join();
        consumer.join();
        
        long duration = System.nanoTime() - start;
        System.out.printf("ArrayBlockingQueue: %.2f M ops/sec%n", 
            iterations / (duration / 1_000_000_000.0));
    }
}
```

## 九、总结：Disruptor 核心设计

```
Disruptor 高性能设计总结：

┌─────────────────────────────────────────────────────────────┐
```

## 面试追问预判

### 追问一：Disruptor 的序号为什么用 long 而不是 int？

序号需要表示**无限递增**的消息序列。如果用 int，32 位最多表示 21 亿条消息，远不够用。

```java
// int 的问题
int sequence = Integer.MAX_VALUE;  // 2147483647
sequence++;                         // 溢出，变成 -2147483648

// long 的优势
long sequence = Long.MAX_VALUE;     // 9.22 * 10^18
// 即使每秒处理 1 亿条，也需要 292 年才会溢出
```

### 追问二：RingBuffer 的容量为什么必须是 2 的幂次？

2 的幂次可以用**位运算**快速计算槽位，避免耗时的取模运算：

```java
// 普通取模（慢）
int index = sequence % bufferSize;

// 位运算（快）
int index = (int) (sequence & (bufferSize - 1));

// 例如: bufferSize = 8 (二进制 1000)
// bufferSize - 1 = 7 (二进制 0111)
// 任何数和 7 进行 & 运算，等价于取模 8
// 15 & 7 = 7   等价于 15 % 8 = 7
// 16 & 7 = 0   等价于 16 % 8 = 0
```

### 追问三：Disruptor 适合什么场景？

Disruptor 不是万能的，适合的场景：

- **低延迟要求**：延迟敏感（纳秒级），如高频交易
- **高吞吐要求**：百万级消息处理
- **单进程内通信**：不适合跨进程
- **事件驱动架构**：处理链式事件

不适合的场景：

- 需要跨进程通信
- 消息需要持久化
- 延迟要求不高（毫秒级足够）
