周一早上九点，订单系统开始出现大量超时。

监控系统显示：CPU 使用率正常，内存使用率正常，数据库连接正常。但接口响应时间从平时的 50ms 飙升到 5 秒。

运维团队紧急排查。打开 Arthas，看到一堆线程处于 `RUNNABLE` 状态——不是 CPU 密集型任务，而是**线程池队列积压**。

```
订单处理线程池状态：
- 核心线程数：10
- 最大线程数：20
- 当前活跃线程数：20（全部繁忙）
- 队列大小：10000（积压 10000 个任务）
- 拒绝任务数：500+
```

问题找到了：线程池的队列积压了 1 万个任务，但没有人知道。线程池默默工作了这么久，从没有人监控过它。

这是一个典型的「线程池失控」场景。这一讲，我们来彻底解决线程池监控与调优的问题。

## 一、为什么线程池需要监控

### 1.1 线程池的「黑盒」问题

ThreadPoolExecutor 是 Java 并发编程的核心组件，但它的内部状态对外部是不透明的：

```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    10,     // 核心线程数
    20,     // 最大线程数
    60,     // 空闲线程存活时间
    TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(100)  // 队列容量
);

// 执行任务
executor.submit(() -> {
    // 这个任务何时被执行？
    // 当前队列有多少任务？
    // 有多少线程在忙？
    // 完全不知道！
});
```

当你在线上遇到这些问题时，没有监控的线程池就是黑盒：

- 为什么任务执行这么慢？
- 线程池的队列是否积压了？
- 核心线程数是否设置合理？
- 为什么任务被拒绝了？

### 1.2 监控线程池的核心价值

| 场景 | 没有监控 | 有监控 |
|---|---|---|
| 问题发现 | 用户投诉后才知道 | 提前告警 |
| 问题定位 | 排查几小时 | 几分钟定位 |
| 调优决策 | 靠猜测 | 靠数据 |
| 容量规划 | 拍脑袋 | 有依据 |

## 二、ThreadPoolExecutor 的核心监控指标

### 2.1 如何获取线程池状态

ThreadPoolExecutor 提供了一些方法可以获取状态：

```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    10, 20, 60, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(100)
);

// 获取各种指标
public void printThreadPoolStatus() {
    System.out.println("=== 线程池状态 ===");
    System.out.println("核心线程数: " + executor.getCorePoolSize());
    System.out.println("最大线程数: " + executor.getMaximumPoolSize());
    System.out.println("当前线程数: " + executor.getPoolSize());
    System.out.println("活跃线程数: " + executor.getActiveCount());
    System.out.println("已完成任务数: " + executor.getCompletedTaskCount());
    System.out.println("队列大小: " + executor.getQueue().size());
    System.out.println("队列容量: " + executor.getQueue().remainingCapacity());
}
```

### 2.2 监控线程池的工具类封装

```java
public class ThreadPoolMonitor {

    private final ThreadPoolExecutor executor;
    private final String name;

    public ThreadPoolMonitor(ThreadPoolExecutor executor, String name) {
        this.executor = executor;
        this.name = name;
    }

    public ThreadPoolStats getStats() {
        return new ThreadPoolStats(
            name,
            executor.getPoolSize(),              // 当前线程数
            executor.getActiveCount(),           // 活跃线程数
            executor.getCorePoolSize(),          // 核心线程数
            executor.getMaximumPoolSize(),       // 最大线程数
            executor.getQueue().size(),          // 队列大小
            executor.getQueue().remainingCapacity(),  // 剩余容量
            executor.getCompletedTaskCount(),    // 已完成任务数
            executor.getTaskCount()              // 总任务数
        );
    }

    // 判断是否需要告警
    public boolean shouldAlert() {
        int activeCount = executor.getActiveCount();
        int maxPoolSize = executor.getMaximumPoolSize();
        int queueSize = executor.getQueue().size();
        int queueCapacity = executor.getQueue().remainingCapacity() + queueSize;

        // 活跃线程数达到最大线程数
        if (activeCount >= maxPoolSize && executor.getQueue().size() > 0) {
            return true;
        }

        // 队列积压超过 80%
        if (queueSize > queueCapacity * 0.8) {
            return true;
        }

        return false;
    }
}

@Data
@AllArgsConstructor
public class ThreadPoolStats {
    private String name;
    private int poolSize;
    private int activeCount;
    private int corePoolSize;
    private int maxPoolSize;
    private int queueSize;
    private int queueCapacity;
    private long completedTaskCount;
    private long totalTaskCount;
}
```

### 2.3 使用包装器自动采集指标

```java
public class MonitorableThreadPoolExecutor extends ThreadPoolExecutor {

    private final AtomicLong submittedCount = new AtomicLong(0);
    private final AtomicLong completedCount = new AtomicLong(0);
    private final AtomicLong rejectedCount = new AtomicLong(0);
    private final String name;

    public MonitorableThreadPoolExecutor(int corePoolSize, int maximumPoolSize,
            long keepAliveTime, TimeUnit unit, BlockingQueue<Runnable> workQueue, String name) {
        super(corePoolSize, maximumPoolSize, keepAliveTime, unit, workQueue,
            new ThreadPoolExecutor.AbortPolicy());
        this.name = name;
    }

    @Override
    public void execute(Runnable command) {
        submittedCount.incrementAndGet();
        super.execute(command);
    }

    @Override
    protected void afterExecute(Runnable r, Throwable t) {
        completedCount.incrementAndGet();
        super.afterExecute(r, t);
    }

    @Override
    protected void rejectedExecution(Runnable r, ThreadPoolExecutor executor) {
        rejectedCount.incrementAndGet();
        super.rejectedExecution(r, executor);
    }

    // 获取指标
    public ThreadPoolMetrics getMetrics() {
        return new ThreadPoolMetrics(
            name,
            getPoolSize(),
            getActiveCount(),
            getCorePoolSize(),
            getMaximumPoolSize(),
            getQueue().size(),
            getQueue().remainingCapacity(),
            completedCount.get(),
            submittedCount.get(),
            rejectedCount.get()
        );
    }
}

@Data
@AllArgsConstructor
class ThreadPoolMetrics {
    private String name;
    private int poolSize;
    private int activeCount;
    private int corePoolSize;
    private int maxPoolSize;
    private int queueSize;
    private int queueRemainingCapacity;
    private long completedCount;
    private long submittedCount;
    private long rejectedCount;
}
```

## 三、JMX 暴露线程池指标

### 3.1 通过 JMX 暴露指标

JMX（Java Management Extensions）是 Java 标准的监控接口：

```java
// 创建 MBean 接口
public interface ThreadPoolStatusMBean {
    int getActiveCount();
    int getPoolSize();
    int getQueueSize();
    int getQueueCapacity();
    long getCompletedTaskCount();
    long getRejectedCount();
}

// 实现 MBean
public class ThreadPoolStatus implements ThreadPoolStatusMBean {

    private final ThreadPoolExecutor executor;

    public ThreadPoolStatus(ThreadPoolExecutor executor) {
        this.executor = executor;
    }

    @Override
    public int getActiveCount() {
        return executor.getActiveCount();
    }

    @Override
    public int getPoolSize() {
        return executor.getPoolSize();
    }

    @Override
    public int getQueueSize() {
        return executor.getQueue().size();
    }

    @Override
    public int getQueueCapacity() {
        return executor.getQueue().remainingCapacity() + executor.getQueue().size();
    }

    @Override
    public long getCompletedTaskCount() {
        return executor.getCompletedTaskCount();
    }

    @Override
    public long getRejectedCount() {
        return 0;  // 需要自己维护
    }
}

// 注册 MBean
public void registerMBean(ThreadPoolExecutor executor, String name) {
    ThreadPoolStatus mBean = new ThreadPoolStatus(executor);

    MBeanServer mBeanServer = ManagementFactory.getPlatformMBeanServer();
    ObjectName objectName = new ObjectName("ThreadPool:name=" + name);

    mBeanServer.registerMBean(mBean, objectName);

    System.out.println("MBean 注册成功: " + objectName);
}
```

## 六、线程池的常见问题与处理

### 6.1 问题一：活跃线程数持续高

**症状**：活跃线程数接近最大线程数，队列持续积压

```text
监控指标表现：
- activeCount ≈ maximumPoolSize（持续）
- queueSize 持续增长
- completedTaskCount 增长缓慢

原因分析：
1. 任务执行时间过长
2. 线程数设置不足
3. 数据库连接池瓶颈
4. 外部服务调用超时
```

**排查方法**：

```java
// 使用 Arthas 排查
$ thread  # 查看线程状态

"pool-1-thread-15" Id=45 RUNNABLE
    at com.example.OrderService.processOrder(OrderService.java:45)
    at com.example.OrderService.lambda$0(OrderService.java:30)
    at java.util.concurrent.ThreadPoolExecutor$Worker.run(ThreadPoolExecutor.java:624)

$ thread -n 3  # 查看 CPU 使用最高的 3 个线程

"pool-1-thread-1" Id=31 CPU=45.2% RUNNABLE
    at java.net.SocketInputStream.socketRead0(Native Method)
    at java.net.SocketInputStream.read(SocketInputStream.java:150)
    ...

# 大多数时间在等待 IO（网络调用、数据库查询）
```

**解决方案**：

```java
// 方案一：增加线程数
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    10,     // 核心线程数：10 -> 20
    50,     // 最大线程数：20 -> 50
    60, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(1000)  // 队列容量也增加
);

// 方案二：优化任务执行时间
public void processOrder(Order order) {
    // 优化前：同步调用
    PaymentResult result = paymentService.pay(order);  // 可能耗时 1s

    // 优化后：异步处理
    CompletableFuture.supplyAsync(() -> paymentService.pay(order), paymentExecutor);
}

// 方案三：设置合理的超时时间
@Configuration
public class RestTemplateConfig {

    @Bean
    public RestTemplate restTemplate() {
        SimpleClientHttpRequestFactory factory = new SimpleClientHttpRequestFactory();
        factory.setConnectTimeout(3000);    // 连接超时 3 秒
        factory.setReadTimeout(5000);      // 读取超时 5 秒
        return new RestTemplate(factory);
    }
}
```

### 6.2 问题二：队列持续增长

**症状**：队列大小持续增长，活跃线程数正常

```text
监控指标表现：
- activeCount 正常
- queueSize 持续增长
- rejectedCount 可能为 0

原因分析：
1. 任务产生速度 > 消费速度
2. 消费者线程阻塞（等待数据库连接）
3. 任务中有慢查询
```

**排查方法**：

```java
// 添加监控日志
public class MonitorableThreadPoolExecutor extends ThreadPoolExecutor {

    private final ScheduledExecutorService monitor =
        Executors.newSingleThreadScheduledExecutor();

    public MonitorableThreadPoolExecutor(...) {
        // 每 10 秒输出一次状态
        monitor.scheduleAtFixedRate(() -> {
            System.out.printf("[%s] 线程池状态 - 活跃: %d/%d, 队列: %d/%d%n",
                name,
                getActiveCount(),
                getMaximumPoolSize(),
                getQueue().size(),
                getQueue().remainingCapacity() + getQueue().size()
            );
        }, 0, 10, TimeUnit.SECONDS);
    }
}
```

**解决方案**：

```java
// 方案一：设置队列容量上限 + 拒绝策略
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    10, 50, 60, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(1000),  // 有界队列
    new ThreadPoolExecutor.CallerRunsPolicy()  // 拒绝策略：调用者执行
);

// 方案二：使用监控告警 + 自动扩容
@Configuration
public class DynamicThreadPoolConfig {

    @Autowired
    private MeterRegistry meterRegistry;

    @Bean
    public ThreadPoolExecutor orderExecutor() {
        ThreadPoolExecutor executor = new ThreadPoolExecutor(
            10, 50, 60, TimeUnit.SECONDS,
            new LinkedBlockingQueue<>(1000)
        );

        // 监控并自动扩容
        autoScale(executor);
        return executor;
    }

    private void autoScale(ThreadPoolExecutor executor) {
        // 每分钟检查一次
        Executors.newSingleThreadScheduledExecutor()
            .scheduleAtFixedRate(() -> {
                int active = executor.getActiveCount();
                int max = executor.getMaximumPoolSize();
                int queueSize = executor.getQueue().size();

                // 如果活跃线程数 >= 最大线程数，且队列积压严重，扩容
                if (active >= max && queueSize > 100) {
                    executor.setMaximumPoolSize(Math.min(max + 10, 100));
                    executor.setCorePoolSize(Math.min(
                        executor.getCorePoolSize() + 5, 80));
                    System.out.println("线程池已扩容: " + executor.getMaximumPoolSize());
                }
            }, 1, 1, TimeUnit.MINUTES);
    }
}
```

## 七、线程池参数的动态调整

### 7.1 为什么需要动态调整

静态配置的线程池无法适应业务峰谷变化：

### 7.2 动态调整线程池的实现

```java
@Configuration
public class DynamicThreadPool {

    private volatile int corePoolSize = 10;
    private volatile int maxPoolSize = 50;
    private volatile int queueCapacity = 1000;

    private ThreadPoolExecutor executor;

    @PostConstruct
    public void init() {
        executor = new ThreadPoolExecutor(
            corePoolSize,
            maxPoolSize,
            60, TimeUnit.SECONDS,
            new LinkedBlockingQueue<>(queueCapacity)
        );
    }

    // 动态调整核心线程数
    public void setCorePoolSize(int corePoolSize) {
        this.corePoolSize = corePoolSize;
        executor.setCorePoolSize(corePoolSize);
    }

    // 动态调整最大线程数
    public void setMaxPoolSize(int maxPoolSize) {
        this.maxPoolSize = maxPoolSize;
        executor.setMaximumPoolSize(maxPoolSize);
    }

    // 根据负载自动调整
    public void autoAdjust() {
        int active = executor.getActiveCount();
        int queueSize = executor.getQueue().size();
        int max = executor.getMaximumPoolSize();

        // 如果活跃线程数 >= 最大线程数 * 0.8，且队列积压 > 500
        if (active >= max * 0.8 && queueSize > 500) {
            // 扩容
            int newMax = Math.min(max + 10, 100);
            setMaxPoolSize(newMax);
            setCorePoolSize(Math.min(corePoolSize + 5, newMax - 10));
            System.out.println("线程池扩容: core=" + corePoolSize + ", max=" + maxPoolSize);
        }

        // 如果活跃线程数 < 最大线程数 * 0.3，且队列积压 < 100
        if (active < max * 0.3 && queueSize < 100) {
            // 缩容
            int newMax = Math.max(max - 10, 50);
            setCorePoolSize(Math.max(corePoolSize - 5, 10));
            setMaxPoolSize(newMax);
            System.out.println("线程池缩容: core=" + corePoolSize + ", max=" + maxPoolSize);
        }
    }
}
```

```java

public class ThreadPoolAdminController {

    @Autowired
    private DynamicThreadPool dynamicThreadPool;

    // 查询线程池状态

        return dynamicThreadPool.getStatus();

    // 调整线程池参数
    @PostMapping("/config")
    public Result<Void> updateConfig(

        dynamicThreadPool.setMaxPoolSize(maxPoolSize);
        return Result.success();
    }

    // 触发自动调整
    @PostMapping("/auto-adjust")
    public Result<Void> autoAdjust() {

        return Result.success();
    }
}
```

### 10.2 经验公式与最终配置

```java
// IO 密集型线程池配置公式
int corePoolSize = (线程等待时间 / 线程执行时间 + 1) * CPU 核心数

// 示例：
// - CPU 核心数：8
// - 线程执行时间：50ms（数据库查询 + 业务处理）
// - 线程等待时间：450ms（网络 IO）
// - 核心线程数 = (450 / 50 + 1) * 8 = (9 + 1) * 8 = 80

// 最终配置
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    80,                     // 核心线程数
    150,                    // 最大线程数 = 核心线程数 * 1.5 ~ 2
    60, TimeUnit.SECONDS,   // 空闲线程存活时间
    new LinkedBlockingQueue<>(3000),  // 队列容量 = 核心线程数 * 10 ~ 50
    new ThreadPoolExecutor.CallerRunsPolicy()  // 拒绝策略
);
```

### 10.3 不同场景的配置建议

| 场景 | 核心线程数 | 最大线程数 | 队列容量 | 拒绝策略 |
|---|---|---|---|---|
| 快速接口调用 | CPU * 2 | CPU * 4 | 1000 | CallerRuns |
| 数据库操作 | CPU * 2 | CPU * 4 | 500 | CallerRuns |
| HTTP 远程调用 | CPU * 2 | CPU * 6 | 2000 | CallerRuns |
| 复杂计算任务 | CPU + 1 | CPU * 2 | 100 | Abort |
| 批处理任务 | CPU * 2 | CPU * 4 | 10000 | CallerRuns |

## 面试追问预判

### 追问一：线程池参数如何配置？

线程池参数配置需要根据业务场景：

1. **CPU 密集型**：核心线程数 = CPU 核心数 + 1
2. **IO 密集型**：核心线程数 = CPU 核心数 * (1 + IO 等待时间 / CPU 计算时间)
3. **混合型**：分别设置不同的线程池

### 追问二：线程池队列满了怎么办？

队列满了有几种处理方式：

1. **CallerRunsPolicy**：由调用方线程执行，起到限流作用
2. **增加线程数**：扩容线程池
3. **增大队列容量**：临时缓解
4. **拒绝任务 + 告警**：快速失败，及时处理

### 追问三：如何监控生产环境的线程池？

生产环境监控线程池的方案：

1. **Micrometer + Prometheus**：采集指标
2. **Grafana**：可视化展示
3. **AlertManager**：告警通知
4. **Arthas**：在线排查问题
