# 定时任务调度：ScheduledExecutor 与分布式定时任务

## 场景切入：凌晨的数据灾难

凌晨 3 点，用户李明收到了系统的紧急告警。

"数据同步服务异常：过去 1 小时的数据全部丢失！"

李明赶紧爬起来查日志，发现问题的根源是一个定时任务：每天凌晨 2 点，系统会自动同步一批核心数据。任务本来应该在 5 分钟内完成，但不知为何，它跑了整整 2 个小时。

更糟糕的是，在这 2 个小时里，定时任务的执行节点刚好发生了故障重启。备机上的定时任务自动接管，但两台机器同时运行，导致：

- 数据被重复处理
- 部分数据状态错乱
- 备份数据被覆盖

这不仅仅是定时任务的问题，这是**分布式定时任务**的经典坑。

本文将全面讲解 JDK 定时任务的基础、分布式定时任务的需求，以及 XXL-Job 等主流框架的设计原理。

## 一、JDK 定时任务的演进

### 1.1 Timer 的局限性

JDK 最早提供的定时工具是 `java.util.Timer`。它的设计简单，但存在致命缺陷：

```java
public class TimerDemo {
    public static void main(String[] args) {
        Timer timer = new Timer();
        
        // 调度一个定时任务
        timer.schedule(new TimerTask() {
            @Override
            public void run() {
                System.out.println("定时任务执行");
            }
        }, 0, 1000);  // 延迟 0ms，每秒执行一次
    }
}
```

**Timer 的问题**：

1. **单线程执行**：所有任务共享一个线程，一个任务抛出异常会导致整个 Timer 崩溃
2. **无恢复机制**：线程终止后，所有任务都不会再执行
3. **任务堆积**：如果某个任务执行时间过长，会阻塞后续任务

```java
// 问题演示：TimerTask 抛出异常
Timer timer = new Timer();
timer.schedule(new TimerTask() {
    @Override
    public void run() {
        throw new RuntimeException("任务失败");  // 导致整个 Timer 停止
    }
}, 0, 1000);

timer.schedule(new TimerTask() {
    @Override
    public void run() {
        System.out.println("这个任务永远不会执行");
    }
}, 5000, 1000);
```

### 1.2 ScheduledExecutorService 的诞生

JDK 5 引入了 `ScheduledExecutorService`，解决了 Timer 的多个问题：

```java
public class ScheduledExecutorDemo {
    public static void main(String[] args) {
        // 创建调度线程池
        ScheduledExecutorService executor = 
            Executors.newScheduledThreadPool(4);
        
        // 延迟执行一次
        executor.schedule(() -> {
            System.out.println("延迟 3 秒后执行");
        }, 3, TimeUnit.SECONDS);
        
        // 固定频率执行
        executor.scheduleAtFixedRate(() -> {
            System.out.println("每秒执行一次");
        }, 0, 1, TimeUnit.SECONDS);
    }
}
```

**ScheduledExecutorService 的优势**：

1. **多线程执行**：任务在独立的线程中执行，互不影响
2. **异常隔离**：一个任务抛出异常不影响其他任务
3. **线程复用**：线程可以被多个任务复用

## 二、ScheduledExecutorService 详解

### 2.1 三种调度方法

`ScheduledExecutorService` 提供了三种调度方法：

```java
// 1. 延迟执行一次
ScheduledFuture<?> schedule(Runnable command, long delay, TimeUnit unit);
ScheduledFuture<V> schedule(Callable<V> callable, long delay, TimeUnit unit);

// 2. 固定频率执行（速率固定）
ScheduledFuture<?> scheduleAtFixedRate(Runnable command, long initialDelay, 
                                      long period, TimeUnit unit);

// 3. 固定延迟执行（间隔固定）
ScheduledFuture<?> scheduleWithFixedDelay(Runnable command, long initialDelay, 
                                         long delay, TimeUnit unit);
```

### 2.2 scheduleAtFixedRate vs scheduleWithFixedDelay

这两个方法很容易混淆，但适用场景完全不同：

**scheduleAtFixedRate**：固定**速率**，每次开始时间间隔相同

```java
// 场景：每秒执行一次日志上报
// 任务执行时间：200ms
// 实际执行时间点：0s, 1s, 2s, 3s...
executor.scheduleAtFixedRate(() -> {
    uploadLogs();
}, 0, 1, TimeUnit.SECONDS);
```

**scheduleWithFixedDelay**：固定**间隔**，每次结束到下次开始间隔相同

```java
// 场景：数据库同步，每次同步完成后休息 5 分钟
// 任务执行时间：2 分钟
// 实际执行时间点：0s, 7s, 14s, 21s... (2分钟执行 + 5分钟等待)
executor.scheduleWithFixedDelay(() -> {
    syncDatabase();
}, 0, 5, TimeUnit.MINUTES);
```

**对比图示**：

```
scheduleAtFixedRate（速率固定）：
├─────────────────────────────────────────────────────────────┤
│  任务1  │任务2│ 任务3  │任务4│ 任务5  │任务6│ ...
├─────────────────────────────────────────────────────────────┤
│    0s      1s     2s      3s     4s      5s
│   开始    开始    开始    开始    开始    开始
│  （间隔固定：每秒开始一个新任务）

scheduleWithFixedDelay（间隔固定）：
├─────────────────────────────────────────────────────────────┤
│  任务1  │间隔5分钟│  任务2  │间隔5分钟│  任务3  │间隔5分钟│
├─────────────────────────────────────────────────────────────┤
│    0s-2s    2s-7m      7m-9m    9m-14m    14m-16m
│   执行完成    等待      执行完成    等待     执行完成
│  （间隔固定：每次完成后等待）
```

### 2.3 常见坑与解决方案

**坑一：任务执行时间超过调度间隔**

```java
// 场景：同步任务耗时 2 分钟，但调度间隔是 1 分钟
// scheduleAtFixedRate 行为：任务会排队执行，不会并发
executor.scheduleAtFixedRate(() -> {
    // 耗时 2 分钟
    syncData();  // 120 秒
}, 0, 1, TimeUnit.MINUTES);

// 执行时间线：
// 第 0 分钟：任务 A 开始
// 第 1 分钟：任务 B 开始（排队）
// 第 2 分钟：任务 A 结束，任务 B 开始
// 第 3 分钟：任务 C 开始（排队）
// ...任务堆积
```

**解决方案**：使用 `scheduleWithFixedDelay`，确保任务串行执行

```java
// 每次同步完成后，等待 1 分钟再执行下一次
executor.scheduleWithFixedDelay(() -> {
    syncData();
}, 0, 1, TimeUnit.MINUTES);
```

**坑二：时钟回拨**

系统时钟突然回拨，会导致定时任务执行混乱：

```java
// 场景：NTP 服务将系统时间回拨 1 小时
// 原本 9 点执行的任务，可能被跳过或延迟

// 解决方案：使用相对时间调度
ScheduledFuture<?> future = executor.scheduleAtFixedRate(() -> {
    // 使用业务时间而非系统时间
    LocalDateTime businessTime = getBusinessTime();
}, 0, 1, TimeUnit.HOURS);
```

**坑三：线程池资源泄漏**

```java
// 错误：忘记关闭线程池
public class BadScheduler {
    public void scheduleTask() {
        ScheduledExecutorService executor = 
            Executors.newScheduledThreadPool(10);
        executor.scheduleAtFixedRate(() -> {
            doSomething();
        }, 0, 1, TimeUnit.SECONDS);
        // 如果忘记 executor.shutdown()，线程池永远不会释放
    }
}

// 正确：使用 try-with-resources 或手动关闭
public class GoodScheduler {
    private final ScheduledExecutorService executor = 
        Executors.newScheduledThreadPool(10);
    
    public void shutdown() {
        executor.shutdown();
        try {
            if (!executor.awaitTermination(60, TimeUnit.SECONDS)) {
                executor.shutdownNow();
            }
        } catch (InterruptedException e) {
            executor.shutdownNow();
        }
    }
}
```

## 三、分布式定时任务的需求

### 3.1 单机定时任务的问题

在分布式环境中，单机定时任务面临三个核心问题：

**问题一：任务重复执行**

```
服务器 A：凌晨 2 点执行"数据同步任务"
服务器 B：凌晨 2 点也执行"数据同步任务"

结果：数据被同步两次，可能导致数据重复、状态错乱
```

**问题二：节点故障导致任务漏执行**

```
服务器 A：凌晨 2 点执行"数据同步任务"
服务器 A：凌晨 2 点 01 分宕机
服务器 A：凌晨 2 点的任务永远不会被执行（除非手动触发）
```

**问题三：无法统一管理**

```
运维人员：
  - "任务跑了吗？"
  - "在哪台机器跑的？"
  - "现在状态是什么？"
  
答案：不知道，需要登录每台机器查日志
```

### 3.2 分布式定时任务的核心需求

| 需求 | 说明 | 解决思路 |
|---|---|---|
| 任务不重复 | 同一时刻只有一个节点执行 | 分布式锁 |
| 故障转移 | 节点宕机后自动切换 | 调度中心 + 执行器 |
| 统一管理 | 任务配置、执行日志集中 | 管理平台 |
| 分片执行 | 大任务拆分并行执行 | 分片广播 |
| 失败重试 | 任务失败后自动重试 | 重试策略 |
| 任务依赖 | 任务之间有依赖关系 | DAG 调度 |

## 四、分布式锁在定时任务中的应用

### 4.1 基于 Redis 的分布式锁

最简单的方案是用 Redis 实现分布式锁：

```java
public class DistributedLockScheduler {
    private final StringRedisTemplate redisTemplate;
    private final RedissonClient redisson;
    
    public void scheduleWithLock(String taskName, Runnable task) {
        RLock lock = redisson.getLock("schedule:lock:" + taskName);
        
        try {
            // 尝试获取锁，等待 0 秒，锁自动过期 10 分钟
            boolean acquired = lock.tryLock(0, 10, TimeUnit.MINUTES);
            
            if (acquired) {
                try {
                    task.run();
                } finally {
                    lock.unlock();
                }
            } else {
                log.info("任务 {} 正在其他节点执行，跳过", taskName);
            }
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    }
}
```

### 4.2 完整的定时任务封装

```java
@Component
public class DistributedScheduler {
    private final RedissonClient redisson;
    private final ScheduledExecutorService executor;
    
    public DistributedScheduler(RedissonClient redisson) {
        this.redisson = redisson;
        this.executor = Executors.newScheduledThreadPool(4);
    }
    
    // 分布式定时任务
    public void scheduleAtFixedRateWithLock(String taskName, 
                                              long initialDelay, 
                                              long period, 
                                              TimeUnit unit,
                                              Runnable task) {
        executor.scheduleAtFixedRate(() -> {
            RLock lock = redisson.getLock("schedule:lock:" + taskName);
            
            try {
                // 尝试获取锁，最多等待 1 秒
                if (lock.tryLock(1, 60, TimeUnit.MINUTES)) {
                    try {
                        log.info("节点 {} 开始执行任务 {}", getNodeId(), taskName);
                        task.run();
                        log.info("节点 {} 完成任务 {}", getNodeId(), taskName);
                    } finally {
                        lock.unlock();
                    }
                }
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            } catch (Exception e) {
                log.error("任务执行失败: {}", taskName, e);
            }
        }, initialDelay, period, unit);
    }
    
    private String getNodeId() {
        return InetAddress.getLocalHost().getHostName() + "-" + 
               ProcessHandle.current().pid();
    }
}
```

### 4.3 分布式锁的局限性

Redis 分布式锁方案看起来简单，但有几个问题：

**问题一：锁的可靠性依赖 Redis**

如果 Redis 宕机，所有定时任务都无法执行。

**问题二：任务执行时间超过锁过期时间**

```java
// 任务 A 持有锁，执行时间 30 分钟
// 锁过期时间 10 分钟
// 第 11 分钟，任务 B 获取锁，开始执行
// 第 30 分钟，任务 A 完成后释放锁（但任务 B 已经在执行了）

// 解决方案：锁续期
while (taskRunning) {
    lock.extend();  // 延长锁的过期时间
    Thread.sleep(renewInterval);
}
```

**问题三：没有任务执行状态管理**

锁方案只知道"有没有节点在执行"，但不知道执行到哪一步了。

## 五、XXL-Job 架构设计

### 5.1 整体架构

XXL-Job 是一个轻量级分布式任务调度平台，核心架构：

```
┌─────────────────────────────────────────────────────────────┐
│                        调度中心                             │
│  ┌─────────────┐   ┌─────────────┐   ┌─────────────┐     │
│  │  任务管理    │   │  触发器      │   │  执行日志    │     │
│  └─────────────┘   └─────────────┘   └─────────────┘     │
└─────────────────────────────────────────────────────────────┘
                              │
                              │ RPC 调用（基于 HTTP 或 Netty）
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                       执行器集群                             │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐               │
│  │ 执行器 1  │    │ 执行器 2  │    │ 执行器 3  │    ...      │
│  │ (Node A) │    │ (Node B) │    │ (Node C) │               │
│  └──────────┘    └──────────┘    └──────────┘               │
└─────────────────────────────────────────────────────────────┘
```

**调度中心职责**：
- 任务配置管理（CRUD）
- 触发时间计算（Cron 解析）
- 任务分发（路由到执行器）
- 执行日志收集

**执行器职责**：
- 接收调度中心的任务触发指令
- 在本地线程池执行任务
- 上报执行结果和日志

### 5.3 任务路由策略

调度中心将任务路由到哪个执行器，有多种策略：

| 策略 | 说明 | 适用场景 |
|---|---|---|
| 第一个 | 总是选择第一个注册的节点 | 固定节点 |
| 最后一个 | 总是选择最后一个注册的节点 | 最新节点 |
| 轮询 | 依次选择各节点 | 负载均衡 |
| 随机 | 随机选择节点 | 负载均衡 |
| 一致性哈希 | 根据任务参数哈希选择 | 相同参数同一节点 |
| 最忙 | 选择最近执行次数最多的节点 | 动态负载均衡 |
| 故障转移 | 选择可用的节点 | 容错 |

```java
    return ReturnT.SUCCESS;

// 故障转移示例
public class FailoverRouter {
    public String route(List<String> addressList) {
        for (String address : addressList) {
            // 检查节点是否存活
            if (isAlive(address)) {
                return address;
            }
        }
        throw new RuntimeException("所有节点都不可用");
    }

        try {
            // RPC 健康检查

        } catch (Exception e) {
            return false;
        }
    }
```

## 六、Cron 表达式详解

### 6.1 Cron 表达式格式

XXL-Job 使用标准的 6 位 Cron 表达式（秒 分 时 日 月 周）：

```
┌───────────── 秒 (0-59)
│ ┌───────────── 分 (0-59)
│ │ ┌───────────── 时 (0-23)
│ │ │ ┌───────────── 日 (1-31)
│ │ │ │ ┌───────────── 月 (1-12)
│ │ │ │ │ ┌───────────── 周 (1-7, 1=周一)
│ │ │ │ │ │
* * * * * *
```

### 6.2 常用示例

```java
// 每秒执行
"0 * * * * ?"

// 每分钟执行
"0 0 * * * ?"

// 每小时执行
"0 0 0 * * ?"

// 每天凌晨 2 点执行
"0 0 2 * * ?"

// 每周一凌晨 2 点执行
"0 0 2 ? * MON"

// 每月 1 号凌晨 2 点执行
"0 0 2 1 * ?"

// 每天 9:00-18:00 每小时执行
"0 0 9-18 * * ?"

// 每 5 分钟执行
"0 0/5 * * * ?"

// 工作日每天 9:00 执行
"0 0 9 ? * MON-FRI"
```

### 6.3 Cron 解析原理

```java
public class CronExpressionParser {
    public List<Date> getNextExecutions(String cron, int count) {
        CronExpression expression = new CronExpression(cron);
        List<Date> results = new ArrayList<>();
        
        Date next = new Date();
        for (int i = 0; i < count; i++) {
            next = expression.getNextValidTimeAfter(next);
            results.add(next);
            next = new Date(next.getTime() + 1000);  // 下一秒
        }
        
        return results;
    }
}
```

### 6.4 常见坑

**坑一：日和周不能同时指定**

```java
// 错误：同时指定了日和周
"0 0 2 15 * MON"  // 每月 15 号 AND 每周一
// 应该用 OR，但 Cron 不支持

// 正确：只指定一个
"0 0 2 15 * ?"     // 每月 15 号
"0 0 2 ? * MON"    // 每周一
```

**坑二：L（最后一天）语义**

```java
// L 在日字段：每月最后一天
"0 0 0 L * ?"  // 每月最后一天凌晨执行

// L 在周字段：周六（美国习惯）
"0 0 0 ? * 7L"  // 每周最后一个周六
```

## 七、分布式定时任务的一致性保证

### 7.1 调度中心的 HA

调度中心本身也需要高可用：

```
┌─────────────────────────────────────────────────────────────┐
│                    Nginx 负载均衡                            │
│                 ┌────────────────┐                         │
│                 │   健康检查     │                         │
│                 └────────────────┘                         │
└──────────────────────────┬─────────────────────────────────┘
                           │
         ┌─────────────────┼─────────────────┐
         ▼                 ▼                 ▼
    ┌─────────┐      ┌─────────┐      ┌─────────┐
    │ 调度中心 │      │ 调度中心 │      │ 调度中心 │
    │  主节点  │      │  从节点   │      │  从节点   │
    │ (抢锁)  │      │ (待命)   │      │ (待命)   │
    └─────────┘      └─────────┘      └─────────┘
         │
         ▼
    ┌─────────────────────────────────────┐
    │              MySQL                   │
    │  (任务配置、执行记录、调度日志)        │
    └─────────────────────────────────────┘
```

**HA 实现原理**：

```java
// 使用数据库行锁实现主节点选举
public class LeaderElection {
    private final DataSource dataSource;
    
    public boolean tryElect() {
        Connection conn = dataSource.getConnection();
        conn.setAutoCommit(false);
        
        try {
            // 尝试更新自己的状态为 leader
            String sql = "UPDATE xxl_job_registry SET auto_status=1 " +
                        "WHERE id = ? AND auto_status = 0";
            
            PreparedStatement ps = conn.prepareStatement(sql);
            ps.setString(1, getRegistryId());
            
            int affected = ps.executeUpdate();
            conn.commit();
            
            return affected > 0;  // 更新成功说明获得锁
        } finally {
            conn.setAutoCommit(true);
            conn.close();
        }
    }
}
```

### 7.2 任务执行幂等性

即使有分布式锁，也需要保证任务本身的幂等性：

```java
@XxlJob("orderSyncJob")
public ReturnT<String> orderSyncJob(String param) {
    // 1. 生成任务执行 ID
    String taskId = generateTaskId();
    
    // 2. 检查是否已执行（幂等检查）
    if (taskHasExecuted(taskId)) {
        log.info("任务已执行，跳过: {}", taskId);
        return ReturnT.SUCCESS;
    }
    
    // 3. 记录任务开始
    markTaskStarted(taskId);
    
    try {
        // 4. 执行业务逻辑
        syncOrders();
        
        // 5. 记录任务完成
        markTaskCompleted(taskId);
        
        return ReturnT.SUCCESS;
    } catch (Exception e) {
        // 6. 记录任务失败
        markTaskFailed(taskId, e);
        return new ReturnT<>(500, e.getMessage());
    }
}
```

### 7.3 任务分片

对于数据量大的任务，需要分片并行处理：

```java
@XxlJob("bigDataSyncJob")
public ReturnT<String> bigDataSyncJob(String param) {
    // 获取分片参数
    ShardingUtil.ShardingVO shardingVO = ShardingUtil.getSharding();
    int shardingIndex = shardingVO.getIndex();  // 当前分片编号
    int shardingTotal = shardingVO.getTotal();  // 总分片数
    
    log.info("分片任务开始: index={}, total={}", shardingIndex, shardingTotal);
    
    // 查询本分片需要处理的数据
    List<Order> orders = orderService.getOrdersForShard(
        shardingIndex, 
        shardingTotal
    );
    
    // 处理数据
    for (Order order : orders) {
        syncOrder(order);
    }
    
    log.info("分片任务完成: index={}, total={}, count={}", 
        shardingIndex, shardingTotal, orders.size());
    
    return ReturnT.SUCCESS;
}

// SQL 分片查询
public List<Order> getOrdersForShard(int shardingIndex, int shardingTotal) {
    // 使用取模或范围划分数据
    return jdbcTemplate.query(
        "SELECT * FROM orders WHERE id % ? = ? AND synced = 0",
        (rs, rowNum) -> mapOrder(rs),
        shardingTotal, shardingIndex
    );
}
```

```java
    int batchSize = 1000;
    int totalProcessed = 0;
    
    while (true) {
        List<Order> orders = orderService.getUnsyncOrders(batchSize);
        if (orders.isEmpty()) {
            break;
        }
        
        for (Order order : orders) {
            syncOrder(order);
        }
        
        totalProcessed += orders.size();
        log.info("已处理 {} 条", totalProcessed);
        
        // 每批次后检查是否超时

        }
    }
    
    return ReturnT.SUCCESS;
```

## 面试追问预判

### 追问一：如何设计一个高可用的定时任务系统？

**核心要点**：
1. 调度中心高可用（多节点 + 主备选举）
2. 任务执行幂等性（防止重复执行）
3. 执行状态持久化（任务进度不丢失）
4. 故障自动转移（节点宕机自动切换）

**实现思路**：
```
调度中心：
├─ 多节点部署，通过数据库行锁选举主节点
├─ 主节点负责触发任务，从节点待命
└─ 节点切换时，任务自动重新分发

执行器：
├─ 启动时注册到调度中心
├─ 定期发送心跳
└─ 执行任务前检查是否应该执行（基于数据库记录）

任务状态：
├─ 每次执行前记录"开始"
├─ 执行完成后记录"成功/失败"
└─ 节点宕机重启后，检查是否有未完成任务
```

### 追问二：XXL-Job 和 ElasticJob 有什么区别？

| 维度 | XXL-Job | ElasticJob |
|---|---|---|
| 调度方式 | RPC 调用（HTTP/Netty） | ZK 广播 |
| 依赖 | MySQL | ZK / K8s |
| 分片 | 手动分片 | 原生支持 |
| 失败处理 | 重试 | 重试 + 弹性扩容 |
| 运维复杂度 | 低（仅依赖 MySQL） | 高（依赖 ZK） |
| 功能丰富度 | 中 | 高 |

### 追问三：如何保证定时任务不重复执行？

**方案一：分布式锁（Redis / ZK）**

```java
if (lock.tryLock()) {
    try {
        executeTask();
    } finally {
        lock.unlock();
    }
}
```

**方案二：数据库唯一约束**

```sql
INSERT INTO task_execution (task_id, instance_id, status, start_time)
VALUES (?, ?, 'RUNNING', NOW())
ON DUPLICATE KEY UPDATE status = 'RUNNING';
```

**方案三：任务调度平台（如 XXL-Job）**

平台本身提供了任务分片、路由策略、故障转移等功能，开箱即用。
