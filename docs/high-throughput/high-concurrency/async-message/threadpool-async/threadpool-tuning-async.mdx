# 异步线程池调优与最佳实践

import { Badge } from '@rspress/core/theme';

凌晨 2 点，你被一阵急促的告警电话惊醒。监控大屏上，服务 RT（Response Time）飙升到 5 秒，用户投诉页面打不开。你登录服务器，堆栈信息显示大量线程处于 `WAITING` 状态，线程名称清一色是 `pool-1-thread-*`。这个 `pool-1` 是谁配的？为什么业务线程和异步任务在抢同一批线程资源？

这不是段子，这是生产环境中**线程池配置不当**的经典场景。

## 一、为什么异步场景需要独立的线程池

很多开发者在初学 Java 并发时，会写出这样的代码：

```java
// 全局共用线程池 - 反面教材
public class AppConfig {
    private static final ExecutorService executor = newFixedThreadPool(10);

    public static void executeAsync(Runnable task) {
        executor.execute(task);
    }
}
```

然后在业务代码里这样用：

```java
@Service
public class OrderService {
    public void createOrder(OrderDTO order) {
        // 执行业务逻辑
        processPayment(order);
        deductInventory(order);

        // 异步发送消息通知
        AppConfig.executeAsync(() -> {
            sendNotification(order);
        });
    }
}
```

这个设计看起来没问题——异步任务不会阻塞主线程。但问题在于：|共用线程池|会让业务线程和异步线程相互影响。

假设一个极端场景：下游消息队列出现故障，`sendNotification` 的耗时从 10ms 飙到 5000ms。当每秒有 1000 个订单时，10 个线程很快被占满，后续的业务请求（`processPayment`、`deductInventory`）全部在排队等待线程资源。这就是典型的**异步任务污染业务线程池**的问题。

### 异步线程池的特殊性

异步场景下的线程池和普通业务线程池有本质区别：

| 特性 | 业务线程池 | 异步线程池 |
|---|---|---|
| 任务类型 | CPU 计算为主 | IO 等待为主 |
| 线程数量 | 保守（CPU 核心数 + 1） | 激进（可远超 CPU 核数） |
| 队列选择 | 有界队列，防止内存溢出 | 需根据场景选择 |
| 隔离要求 | 高（直接影响用户体验） | 中等（可容忍一定延迟） |

核心矛盾在于：**IO 密集型任务**的线程数应该远大于 CPU 核心数，而**业务线程池**需要保持较小的线程数以减少线程切换开销。两者混用，必然出问题。

## 二、异步线程池的命名规范

线程池命名看似小事，但在问题排查时至关重要。当你在 `jstack` 输出中看到 `pool-1-thread-3` 时，你根本不知道这个线程是干什么的。但如果看到 `async-email-thread-5` 或 `async-order-thread-2`，问题定位就快多了。

推荐使用 `com.google.common.util.concurrent.ThreadFactoryBuilder` 来构建线程工厂：

```java
import com.google.common.util.concurrent.ThreadFactoryBuilder;
import java.util.concurrent.Executors;
import java.util.concurrent.ThreadPoolExecutor;

public class AsyncThreadPoolFactory {

    public static ThreadPoolExecutor createAsyncPool(String businessName, int corePoolSize, int maxPoolSize, int queueCapacity) {
        ThreadFactory factory = new ThreadFactoryBuilder()
                .setNameFormat(businessName + "-async-%d")
                .setDaemon(false)
                .build();

        return new ThreadPoolExecutor(
                corePoolSize,
                maxPoolSize,
                60L, TimeUnit.SECONDS,
                new LinkedBlockingQueue<>(queueCapacity),
                factory,
                new ThreadPoolExecutor.CallerRunsPolicy()
        );
    }
}
```

这样创建的线程池，线程名会变成 `async-email-0`、`async-email-1`，排查问题时一目了然。

:::tip 命名建议
业务线程池命名建议：`\${business}-async-pool`，如 `order-async-pool`、`email-async-pool`、`sms-async-pool`。
:::

## 三、核心线程数的计算公式

这是线程池配置中最核心的问题。很多面试官喜欢问：「IO 密集型任务，线程池核心线程数设多少合适？」标准答案是：**根据 `waitTime / computeTime` 的比值来计算**。

### 公式推导

假设：
- `C` = CPU 核心数
- `W` = 线程等待 IO 的时间（waitTime）
- `C_t` = 线程执行计算的时间（computeTime）

理想情况下，CPU 利用率 100% 时，线程数 `N` 满足：

```
N = C × (1 + W / C_t)
```

这个公式的物理意义是：当一个线程在等待 IO 时，CPU 可以切换到其他线程继续计算。等待时间越长，需要的线程数越多。

### 实际估算

在生产环境中，精确测量 `waitTime` 和 `computeTime` 比较困难，通常用以下经验值：

| 场景 | waitTime/computeTime 估算 | 线程数建议 |
|---|---|---|
| 纯计算（加密解密） | 0 | C + 1 |
| 轻度 IO（本地文件读写） | 1:1 | 2C |
| 中度 IO（数据库操作） | 2:1 | 3C |
| 重度 IO（HTTP 调用、消息队列） | 4:1 ~ 8:1 | 6C ~ 9C |

对于典型的 Web 服务，HTTP 调用和数据库操作是主要 IO 场景，`waitTime/computeTime` 通常在 4:1 左右。如果你的服务器是 8 核，核心线程数可以设为 **48** 左右。

### 监控调优

但公式只是起点，实际运行时需要根据监控数据调整：

```java
// 典型的异步线程池监控指标
public class ThreadPoolMetrics {
    // 活跃线程数 / 最大线程数 > 0.8 → 考虑扩容
    // 队列大小持续增长 → 队列容量不足或消费速度跟不上
    // 拒绝任务数 > 0 → 需要调整拒绝策略或扩容
}
```

## 四、最大线程数的设置

核心线程数和最大线程数的关系，决定了线程池的弹性能力。

### 配置策略

| 场景 | 核心线程数 | 最大线程数 | 适用情况 |
|---|---|---|---|
| 保守型 | N | N | 任务量稳定，无需弹性 |
| 标准型 | N | 1.5N | 峰值不超过平时的 50% |
| 激进型 | N | 2N | 峰值波动大，需要较强弹性 |

大多数异步场景推荐 **1.5 倍** 的配置：

```java
int corePoolSize = 48;  // 根据公式计算
int maxPoolSize = (int) (corePoolSize * 1.5);  // 72
```

### 为什么要设置最大线程数

如果核心线程都忙，新任务会进入队列。当队列满了，如果还有新任务，线程池会扩容到最大线程数。**最大线程数是应对突发流量的最后一道防线**。

但要注意：最大线程数设置过大，线程创建和销毁的开销会增加；设置过小，突发流量时任务会堆积或被拒绝。

## 五、队列容量的选择

队列是线程池的缓冲地带，选对了能平缓流量高峰，选错了可能引发 OOM。

### 有界队列 vs 无界队列

| 队列类型 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| 无界队列（`LinkedBlockingQueue`） | 永不拒绝任务 | 可能导致 OOM | 任务量可预估、内存充足 |
| 有界队列（`ArrayBlockingQueue`） | 防止内存溢出 | 可能丢失任务 | 任务量不可控、内存敏感 |
| 同步队列（`SynchronousQueue`） | 任务直接交给线程 | 线程不足时立即拒绝 | 需要立即执行的任务 |

### 队列容量计算

队列容量的选择要综合考虑：**最大积压时间**和**内存限制**。

假设：
- 每秒提交任务数 = 100
- 每个任务平均耗时 = 100ms
- 最大可接受延迟 = 10 秒

则队列容量至少需要：100 × 10 = 1000。

但考虑到内存，每个任务对象假设占用 1KB，1000 个任务就是 1MB，10 秒的积压就是 10MB，这是可以接受的。

```java
// 根据任务特性计算队列容量
int queueCapacity = (int) (throughput * maxAcceptableLatency / 1000);
```

:::warning 警惕无界队列
无界队列 `new LinkedBlockingQueue<>()` 看起来省心，但如果任务提交速度持续大于消费速度，队列会无限增长，最终导致 OOM。生产环境**强烈建议使用有界队列**。
:::

## 六、拒绝策略的选择

当线程池达到最大线程数且队列满了，新任务会触发拒绝策略。

### 四种拒绝策略

| 策略 | 行为 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|---|
| `AbortPolicy` | 抛异常 | 明确知道任务被拒绝 | 可能导致任务丢失 | 默认，生产慎用 |
| `DiscardPolicy` | 直接丢弃 | 无额外开销 | 任务静默丢失 | 允许丢任务场景 |
| `DiscardOldestPolicy` | 丢弃最老的 | 优先处理新任务 | 可能丢弃重要任务 | 任务有优先级的场景 |
| `CallerRunsPolicy` | 调用者执行 | 有一定的限流效果 | 可能拖慢主线程 | **推荐** |

### CallerRunsPolicy 的限流机制

`CallerRunsPolicy` 不仅仅是个兜底策略，它还有**流量控制**的作用：

```java
@Service
public class OrderService {
    private final ThreadPoolExecutor asyncPool = new ThreadPoolExecutor(
            48, 72,
            60L, TimeUnit.SECONDS,
            new ArrayBlockingQueue<>(1000),
            new ThreadFactoryBuilder().setNameFormat("order-async-%d").build(),
            new ThreadPoolExecutor.CallerRunsPolicy()  // 关键配置
    );

    public void createOrder(OrderDTO order) {
        asyncPool.submit(() -> {
            sendNotification(order);
        });
    }
}
```

当线程池饱和时，`CallerRunsPolicy` 会让调用线程（通常是业务线程）自己执行任务。这会产生一个有趣的效果：**业务线程变慢 → 提交任务速度下降 → 线程池有机会追上来**。

这就是自适应的限流机制，比直接抛异常或丢弃任务更温和。

## 七、线程池的隔离策略

线程池隔离是高并发系统的核心设计原则之一。为什么要隔离？因为不同业务的流量特征、响应时间要求、资源消耗完全不同。

### 按业务隔离

```java
@Configuration
public class AsyncThreadPoolConfig {

    @Bean("orderAsyncPool")
    public ThreadPoolExecutor orderAsyncPool() {
        return AsyncThreadPoolFactory.createAsyncPool("order", 32, 48, 1000);
    }

    @Bean("emailAsyncPool")
    public ThreadPoolExecutor emailAsyncPool() {
        return AsyncThreadPoolFactory.createAsyncPool("email", 16, 24, 500);
    }

    @Bean("smsAsyncPool")
    public ThreadPoolExecutor smsAsyncPool() {
        return AsyncThreadPoolFactory.createAsyncPool("sms", 8, 16, 200);
    }
}
```

这样设计的好处：
- 订单服务流量大，配置较大的线程池
- 邮件通知延迟容忍度高，可以配置较小的线程池
- 某个业务出问题，不会影响其他业务

### 隔离级别选择

| 隔离级别 | 实现方式 | 适用场景 |
|---|---|---|
| 线程隔离 | 每个业务独立线程池 | 业务间资源消耗差异大 |
| 进程隔离 | 独立服务部署 | 故障隔离要求极高 |
| 舱壁模式 | 信号量限制并发数 | 不想维护多线程池 |

大多数场景下，**线程隔离**已经足够。

## 八、Tomcat 线程池 vs 自定义线程池

很多开发者分不清 Tomcat 线程池和自定义线程池的区别，导致配置混乱。

### Tomcat 线程池

Tomcat 使用自己的线程池处理 HTTP 请求，配置在 `server.xml` 或 `application.yml` 中：

```yaml
server:
  tomcat:
    threads:
      max: 200          # 最大线程数
      min-spare: 10     # 最小空闲线程数
    max-connections: 10000
    accept-count: 100   # 队列长度
```

Tomcat 线程池的特点：
- **CPU 密集型为主**（处理请求逻辑）
- 线程数不宜过大（8~200，根据服务器配置）
- 阻塞操作应在业务线程池中处理

### 自定义线程池

自定义线程池用于异步任务，和 Tomcat 线程池是**完全独立的两个池**：

```java
// 这是异步任务线程池，和 Tomcat 无关
@Bean("asyncTaskExecutor")
public AsyncTaskExecutor asyncTaskExecutor() {
    return new AsyncTaskExecutor();
}
```

:::tip 关键区别
Tomcat 线程池处理同步的 HTTP 请求，阻塞时线程会被挂起；自定义线程池处理异步任务，可以配置更多线程来应对 IO 等待。两者职责不同，配置逻辑也不同。
:::

## 九、异步线程池与 Spring @Async

Spring 提供了 `@Async` 注解来简化异步编程，但很多人不知道它背后用的是哪个线程池。

### @Async 的线程池选择

Spring 默认使用 `SimpleAsyncTaskExecutor`，这个 executor **不会复用线程**——每个任务都会创建新线程，高并发下会创建大量线程，导致 OOM。

正确做法是配置自定义线程池：

```java
@Configuration
@EnableAsync
public class AsyncConfig implements AsyncConfigurer {

    @Override
    public Executor getAsyncExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(48);
        executor.setMaxPoolSize(72);
        executor.setQueueCapacity(1000);
        executor.setThreadNamePrefix("spring-async-");
        executor.setRejectedExecutionHandler(new ThreadPoolExecutor.CallerRunsPolicy());
        executor.initialize();
        return executor;
    }
}
```

### 按业务指定线程池

如果需要不同业务使用不同线程池：

```java
public interface AsyncConfiguration extends AsyncConfigurer {
    @Qualifier("orderAsyncPool") Executor getAsyncExecutor();
}

@Service
public class OrderService {
    @Async("orderAsyncPool")
    public void sendNotification(Order order) {
        // 使用 orderAsyncPool
    }
}
```

## 十、生产环境线程池配置检查清单

上线前，检查清单对照：

- [ ] 线程池名称是否规范（包含业务标识）
- [ ] 核心线程数和最大线程数是否合理（IO 密集型建议 4~9 倍 CPU 核数）
- [ ] 队列容量是否设置（有界队列，防止 OOM）
- [ ] 拒绝策略是否选择 `CallerRunsPolicy`（自适应限流）
- [ ] 是否按业务隔离（不同业务不同线程池）
- [ ] 是否配置监控指标（活跃线程数、队列大小、拒绝次数）
- [ ] Spring @Async 是否使用自定义线程池（避免 SimpleAsyncTaskExecutor）
- [ ] 线程是否为非守护线程（daemon = false）

---

## 总结

异步线程池配置不是拍脑袋的事，它需要考虑：

1. **任务特性**：IO 密集型 vs CPU 密集型，决定了线程数计算方式
2. **隔离策略**：不同业务使用不同线程池，避免相互影响
3. **队列选择**：有界队列防止 OOM，但可能丢任务
4. **拒绝策略**：`CallerRunsPolicy` 提供自适应限流
5. **监控告警**：活跃线程数、队列积压、拒绝次数都要监控

## 面试追问预判

**Q1：线程池的线程数设置为多少合适？**

这道题要分场景回答。CPU 密集型建议 `CPU 核心数 + 1`；IO 密集型建议 `CPU 核心数 × (1 + waitTime/computeTime)`，经验值是 4~9 倍 CPU 核数。还要说明：理论公式只是起点，实际需要根据监控数据调整。

**Q2：线程池拒绝策略用哪个？**

大多数场景推荐 `CallerRunsPolicy`，因为它有自适应限流能力。但要注意：如果是关键业务不能丢任务，也需要配合消息队列做持久化。

**Q3：如何实现动态调整线程池参数？**

可以基于监控指标实现自动扩缩容。例如：活跃线程数 / 最大线程数 > 0.8 时自动扩容；持续低于 0.2 时缩容。Apollo、Nacos 等配置中心支持动态推送配置。

**Q4：线程池的優雅关闭怎么做？**

调用 `shutdown()` 拒绝新任务，等待正在执行的任务完成；调用 `shutdownNow()` 尝试中断正在执行的任务。关键点：`awaitTermination()` 配合超时时间，以及任务执行的幂等性保证。
