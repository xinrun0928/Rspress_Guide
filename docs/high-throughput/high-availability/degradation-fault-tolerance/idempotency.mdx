# 接口幂等性：实现方案与防重设计

你一定遇到过这种情况：在电商 App 上买东西，点击「立即支付」，页面转圈圈，你等不及又点了一次。结果弹出两条扣款通知——你被扣了两次钱。

或者在银行 App 转账，转账成功页面已经显示了，但你想确认一下又刷新了一下，结果又转了一次账。

这些场景的背后，都是「幂等性缺失」在作祟。幂等性是分布式系统的基石之一，不理解幂等性，就无法写出可靠的系统。这篇文章，我会从幂等性的本质讲起，带你理解 HTTP 语义中的幂等性，掌握各种幂等性实现方案，并给出具体的代码实现。

## 一、幂等性的本质：做一次和做多次，结果一样

### 1.1 什么是幂等性

幂等性（Idempotency）是一个数学概念，指的是一个操作执行一次和执行多次，结果是相同的。在计算机科学中，幂等性被广泛应用于分布式系统、API 设计、数据库操作等领域。

举几个生活中的例子：
- **查询余额**：查一次和查多次，余额不会变——天然幂等
- **删除文件**：删除一次和删除多次，结果都是文件不存在——天然幂等
- **ATM 取款**：取一次和取多次，结果可能不一样（可能余额不够了）——不幂等

### 1.2 HTTP 方法的幂等性

HTTP 协议对不同方法的幂等性有明确定义：

| HTTP 方法 | 幂等性 | 说明 |
|---|---|---|
| GET | 幂等 | 查询操作，多次查询结果相同 |
| HEAD | 幂等 | 与 GET 类似，只返回头部 |
| PUT | 幂等 | 更新操作，多次更新相同结果 |
| DELETE | 幂等 | 删除操作，多次删除结果相同 |
| POST | 非幂等 | 创建操作，多次创建产生多个资源 |
| PATCH | 非幂等 | 部分更新，多次可能产生不同结果 |

```http
# PUT 是幂等的，多次调用效果相同
PUT /api/users/123
{
    "name": "张三",
    "email": "zhangsan@example.com"
}

# 第一次调用：更新用户信息
# 第二次调用：结果相同，用户信息仍然是这些

# POST 是非幂等的，多次调用产生多个资源
POST /api/users
{
    "name": "张三",
    "email": "zhangsan@example.com"
}

# 第一次调用：创建用户，返回 201 Created，用户 ID 可能是 123
# 第二次调用：创建另一个用户，返回 201 Created，用户 ID 可能是 124
```

### 1.3 分布式系统中的幂等性问题

在分布式系统中，幂等性问题更加复杂。主要原因：

**网络不可靠**：请求可能超时、丢包、重传
**服务可能重启**：处理到一半的服务可能崩溃
**重试机制**：为了提高成功率，系统会自动重试失败的请求

```java
// 场景：用户下单
public OrderResult createOrder(CreateOrderRequest request) {
    // 1. 扣减库存
    inventoryService.decrease(request.getProductId(), request.getQuantity());

    // 2. 创建订单
    Order order = orderRepository.create(request);

    // 3. 发送消息
    messageService.send("order.created", order);

    return new OrderResult(order.getId());
}
```

如果步骤 2 创建订单后，服务崩溃了，重启后这条消息会被重新消费，可能导致：
- 库存被扣减两次
- 订单被创建两次
- 消息被发送两次

:::tip 幂等性是系统可靠性的基石

在分布式系统中，任何操作都可能失败，任何失败都可能重试。如果系统不具备幂等性，重试就会导致数据重复、数据不一致等问题。因此，设计分布式系统时，必须首先考虑幂等性。

:::

## 二、幂等性的两种类型：天然幂等与设计幂等

### 2.1 天然幂等操作

天然幂等的操作不需要额外的设计，天然具备幂等性：

| 操作类型 | 幂等原因 | 示例 |
|---|---|---|
| 查询 | 不改变数据 | GET /users/123 |
| 删除（按 ID） | ID 唯一，重复删除无影响 | DELETE /users/123 |
| 设置状态 | 状态值确定，多次设置相同 | UPDATE status = 'CLOSED' |
| 覆盖更新 | 覆盖相同值，效果相同 | PUT /users/123 {data} |
| 去重查询 | 结果已确定 | SELECT DISTINCT |

```java
// 天然幂等的操作示例

// 查询：天然幂等
public User getUser(Long userId) {
    return userRepository.findById(userId);
}

// 删除：天然幂等（按 ID 删除，重复删除无影响）
public void deleteUser(Long userId) {
    userRepository.deleteById(userId);
}

// 覆盖更新：天然幂等
public void updateUser(Long userId, UserData data) {
    User user = userRepository.findById(userId);
    user.setName(data.getName());
    user.setEmail(data.getEmail());
    userRepository.save(user);
}

// 设置状态：天然幂等
public void closeOrder(Long orderId) {
    Order order = orderRepository.findById(orderId);
    order.setStatus(OrderStatus.CLOSED);
    orderRepository.save(order);
}
```

### 2.2 需要设计的幂等操作

对于非天然幂等的操作，需要额外设计：

| 操作类型 | 非幂等原因 | 幂等设计方案 |
|---|---|---|
| 创建订单 | 多次创建产生多个订单 | 幂等键 |
| 扣款 | 多次扣款导致余额减少多次 | 幂等键 + 状态检查 |
| 充值 | 多次充值导致余额增加多次 | 幂等键 + 事务 |
| 发送验证码 | 多次发送浪费资源 | 幂等键 + 时间窗口 |
| 库存扣减 | 多次扣减导致库存减少多次 | 乐观锁 |

```java
// 需要设计的幂等操作示例

// 场景一：创建订单（非幂等）
// 幂等方案：使用幂等键
public CreateOrderResult createOrder(IdempotentCreateOrderRequest request) {
    // 1. 检查幂等键是否已存在
    String idempotencyKey = request.getIdempotencyKey();
    Order existingOrder = orderRepository.findByIdempotencyKey(idempotencyKey);
    if (existingOrder != null) {
        // 已存在，直接返回
        return new CreateOrderResult(existingOrder.getId());
    }

    // 2. 执行创建逻辑
    Order order = orderRepository.create(request);
    order.setIdempotencyKey(idempotencyKey);
    orderRepository.save(order);

    return new CreateOrderResult(order.getId());
}

// 场景二：扣款（非幂等）
// 幂等方案：幂等键 + 余额检查
public WithdrawResult withdraw(IdempotentWithdrawRequest request) {
    String idempotencyKey = request.getIdempotencyKey();

    // 检查是否已处理
    TransactionRecord record = transactionRepository.findByIdempotencyKey(idempotencyKey);
    if (record != null) {
        return new WithdrawResult(record.getTransactionId(), record.getStatus());
    }

    // 检查余额
    Account account = accountRepository.findById(request.getAccountId());
    if (account.getBalance() < request.getAmount()) {
        throw new InsufficientBalanceException();
    }

    // 执行扣款（乐观锁）
    int updated = accountRepository.decreaseBalanceWithVersion(
        request.getAccountId(),
        request.getAmount(),
        account.getVersion()
    );

    if (updated == 0) {
        throw new ConcurrentModificationException();
    }

    // 记录事务
    TransactionRecord newRecord = new TransactionRecord();
    newRecord.setIdempotencyKey(idempotencyKey);
    newRecord.setStatus("SUCCESS");
    transactionRepository.save(newRecord);

    return new WithdrawResult(newRecord.getTransactionId(), "SUCCESS");
}
```

## 三、幂等键方案：最通用的幂等设计

### 3.1 幂等键的设计原则

幂等键（Idempotency Key）是保证幂等性的核心机制。设计幂等键需要遵循以下原则：

**唯一性**：全局唯一，避免碰撞。可以使用 UUID、雪花算法、组合键等方式生成。

**可追溯性**：能够从幂等键反推出业务含义，便于排查问题。

**长度适中**：太短可能碰撞，太长浪费存储空间。建议 32-64 位。

**不可预测**：避免被恶意猜测。建议使用随机数或加密算法生成。

```java
public class IdempotencyKeyGenerator {

    /**
     * 生成订单相关的幂等键
     * 格式：ORDER:{userId}:{orderId}:{timestamp}
     */
    public String generateOrderKey(Long userId, Long orderId) {
        return String.format("ORDER:%d:%d:%d", userId, orderId, System.currentTimeMillis());
    }

    /**
     * 生成支付相关的幂等键
     * 格式：PAY:{merchantId}:{outTradeNo}
     */
    public String generatePayKey(String merchantId, String outTradeNo) {
        return String.format("PAY:%s:%s", merchantId, outTradeNo);
    }

    /**
     * 生成退款相关的幂等键
     * 格式：REFUND:{originalPayKey}:{refundId}
     */
    public String generateRefundKey(String originalPayKey, String refundId) {
        return String.format("REFUND:%s:%s", originalPayKey, refundId);
    }

    /**
     * 使用雪花算法生成幂等键
     */
    public String generateWithSnowflake() {
        return String.valueOf(SnowflakeIdGenerator.generate());
    }
}
```

### 3.2 基于 Redis 的幂等实现

Redis 是实现幂等性的首选工具，因为它的 `SETNX` 操作是原子性的。

```java
@Service
public class RedisIdempotencyService {

    private final StringRedisTemplate redisTemplate;

    // 幂等键的默认过期时间（24小时）
    private static final Duration DEFAULT_TTL = Duration.ofHours(24);

    // 幂等键的状态
    private static final String STATUS_PROCESSING = "PROCESSING";
    private static final String STATUS_SUCCESS = "SUCCESS";
    private static final String STATUS_FAILED = "FAILED";

    public RedisIdempotencyService(StringRedisTemplate redisTemplate) {
        this.redisTemplate = redisTemplate;
    }

    /**
     * 尝试获取幂等锁
     * @param idempotencyKey 幂等键
     * @return true 表示首次请求，可以继续处理；false 表示重复请求
     */
    public boolean tryAcquire(String idempotencyKey) {
        Boolean success = redisTemplate.opsForValue()
            .setIfAbsent(idempotencyKey, STATUS_PROCESSING, DEFAULT_TTL);
        return Boolean.TRUE.equals(success);
    }

    /**
     * 标记请求成功
     */
    public void markSuccess(String idempotencyKey, String result) {
        redisTemplate.opsForValue().set(idempotencyKey, STATUS_SUCCESS + ":" + result, DEFAULT_TTL);
    }

    /**
     * 标记请求失败
     */
    public void markFailed(String idempotencyKey) {
        redisTemplate.opsForValue().set(idempotencyKey, STATUS_FAILED, DEFAULT_TTL);
    }

    /**
     * 获取幂等键的状态
     */
    public IdempotencyStatus getStatus(String idempotencyKey) {
        String status = redisTemplate.opsForValue().get(idempotencyKey);
        if (status == null) {
            return IdempotencyStatus.NOT_EXISTS;
        }
        if (status.startsWith(STATUS_PROCESSING)) {
            return IdempotencyStatus.PROCESSING;
        }
        if (status.startsWith(STATUS_SUCCESS)) {
            return IdempotencyStatus.SUCCESS;
        }
        return IdempotencyStatus.FAILED;
    }

    /**
     * 获取成功请求的结果
     */
    public String getSuccessResult(String idempotencyKey) {
        String status = redisTemplate.opsForValue().get(idempotencyKey);
        if (status != null && status.startsWith(STATUS_SUCCESS)) {
            return status.substring(STATUS_SUCCESS.length() + 1);
        }
        return null;
    }

    /**
     * 带超时的等待
     */
    public String waitForResult(String idempotencyKey, long timeoutMs) {
        long startTime = System.currentTimeMillis();
        while (System.currentTimeMillis() - startTime < timeoutMs) {
            String status = redisTemplate.opsForValue().get(idempotencyKey);
            if (status != null) {
                if (status.startsWith(STATUS_SUCCESS)) {
                    return status.substring(STATUS_SUCCESS.length() + 1);
                }
                if (status.startsWith(STATUS_FAILED)) {
                    throw new RuntimeException("幂等请求执行失败");
                }
            }
            try {
                Thread.sleep(50);
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
                throw new RuntimeException("等待被中断", e);
            }
        }
        throw new RuntimeException("等待幂等结果超时");
    }

    public enum IdempotencyStatus {
        NOT_EXISTS,
        PROCESSING,
        SUCCESS,
        FAILED
    }
}
```

### 3.3 完整的幂等执行模板

```java
@Service
public class IdempotentOperationService {

    private final RedisIdempotencyService idempotencyService;
    private final ObjectMapper objectMapper;

    public IdempotentOperationService(
            RedisIdempotencyService idempotencyService,
            ObjectMapper objectMapper) {
        this.idempotencyService = idempotencyService;
        this.objectMapper = objectMapper;
    }

    /**
     * 执行带幂等的操作
     * @param idempotencyKey 幂等键
     * @param operation 需要执行的操作
     * @param resultClass 返回结果类型
     */
    public <T> IdempotentResult<T> execute(
            String idempotencyKey,
            Callable<T> operation,
            Class<T> resultClass) {

        // 1. 检查幂等键状态
        IdempotencyStatus status = idempotencyService.getStatus(idempotencyKey);

        if (status == IdempotencyStatus.SUCCESS) {
            // 已成功处理，直接返回缓存结果
            String cachedResult = idempotencyService.getSuccessResult(idempotencyKey);
            return IdempotentResult.hit(cachedResult);
        }

        if (status == IdempotencyStatus.PROCESSING) {
            // 正在处理中，等待结果
            String result = idempotencyService.waitForResult(idempotencyKey, 5000);
            return IdempotentResult.hit(result);
        }

        if (status == IdempotencyStatus.FAILED) {
            // 之前失败过，可以选择重试或返回失败
            return IdempotentResult.failed("请求之前处理失败");
        }

        // 2. 尝试获取幂等锁
        if (!idempotencyService.tryAcquire(idempotencyKey)) {
            // 获取锁失败，说明有其他请求正在处理
            String result = idempotencyService.waitForResult(idempotencyKey, 5000);
            return IdempotencyResult.hit(result);
        }

        // 3. 执行操作
        try {
            T result = operation.call();
            // 4. 标记成功
            String serializedResult = objectMapper.writeValueAsString(result);
            idempotencyService.markSuccess(idempotencyKey, serializedResult);
            return IdempotentResult.success(result);
        } catch (Exception e) {
            // 5. 标记失败
            idempotencyService.markFailed(idempotencyKey);
            return IdempotentResult.failed(e.getMessage());
        }
    }

    @Data
    public static class IdempotentResult<T> {
        private boolean success;
        private boolean idempotentHit;
        private T data;
        private String message;

        public static <T> IdempotentResult<T> success(T data) {
            IdempotentResult<T> result = new IdempotentResult<>();
            result.success = true;
            result.data = data;
            return result;
        }

        public static <T> IdempotentResult<T> hit(String cachedResult) {
            IdempotentResult<T> result = new IdempotencyResult<>();
            result.success = true;
            result.idempotentHit = true;
            return result;
        }

        public static <T> IdempotentResult<T> failed(String message) {
            IdempotentResult<T> result = new IdempotentResult<>();
            result.success = false;
            result.message = message;
            return result;
        }
    }
}
```

## 四、数据库唯一约束方案

### 4.1 唯一索引实现幂等

对于涉及数据库操作的重试，可以使用数据库的唯一约束来实现幂等：

```sql
-- 创建幂等记录表
CREATE TABLE idempotent_records (
    idempotency_key VARCHAR(64) PRIMARY KEY,
    business_type VARCHAR(32) NOT NULL,
    status VARCHAR(16) NOT NULL,
    result TEXT,
    created_at TIMESTAMP NOT NULL,
    updated_at TIMESTAMP NOT NULL,
    INDEX idx_business_type (business_type),
    INDEX idx_created_at (created_at)
);
```

```java
@Service
public class DatabaseIdempotencyService {

    @Autowired
    private IdempotentRecordRepository repository;

    @Transactional
    public <T> T executeWithIdempotency(
            String idempotencyKey,
            String businessType,
            Supplier<T> operation) {

        // 1. 尝试插入幂等记录
        try {
            IdempotentRecord record = new IdempotentRecord();
            record.setIdempotencyKey(idempotencyKey);
            record.setBusinessType(businessType);
            record.setStatus(IdempotentStatus.PROCESSING.name());
            record.setCreatedAt(new Date());
            record.setUpdatedAt(new Date());
            repository.save(record);
        } catch (DataIntegrityViolationException e) {
            // 唯一键冲突，说明已存在
            return getExistingResult(idempotencyKey);
        }

        // 2. 执行操作
        try {
            T result = operation.get();
            // 3. 更新记录状态
            updateStatus(idempotencyKey, IdempotentStatus.SUCCESS, serialize(result));
            return result;
        } catch (Exception e) {
            updateStatus(idempotencyKey, IdempotentStatus.FAILED, e.getMessage());
            throw e;
        }
    }

    private <T> T getExistingResult(String idempotencyKey) {
        IdempotentRecord record = repository.findByIdempotencyKey(idempotencyKey)
            .orElseThrow(() -> new RuntimeException("幂等记录不存在"));

        if (IdempotentStatus.SUCCESS.name().equals(record.getStatus())) {
            return deserialize(record.getResult());
        }
        if (IdempotentStatus.PROCESSING.name().equals(record.getStatus())) {
            throw new IdempotencyProcessingException("请求正在处理中");
        }
        throw new IdempotencyFailedException("请求处理失败: " + record.getResult());
    }

    private void updateStatus(String idempotencyKey, IdempotentStatus status, String result) {
        repository.updateStatusByIdempotencyKey(idempotencyKey, status.name(), result, new Date());
    }

    private String serialize(Object obj) {
        // 序列化实现
        return JSON.toJSONString(obj);
    }

    @SuppressWarnings("unchecked")
    private <T> T deserialize(String json) {
        return (T) JSON.parseObject(json, Object.class);
    }

    public enum IdempotentStatus {
        PROCESSING,
        SUCCESS,
        FAILED
    }
}
```

### 4.2 乐观锁实现幂等

对于更新操作，可以使用乐观锁来实现幂等：

```java
@Service
public class OptimisticLockIdempotentService {

    @Autowired
    private AccountRepository accountRepository;

    /**
     * 扣款（乐观锁实现幂等）
     */
    public WithdrawResult withdraw(Long accountId, BigDecimal amount, Long version) {
        // 1. 检查版本号（幂等判断）
        Account account = accountRepository.findById(accountId);
        if (!account.getVersion().equals(version)) {
            // 版本号不匹配，说明已经处理过了
            return WithdrawResult.idempotent(account.getVersion());
        }

        // 2. 检查余额
        if (account.getBalance().compareTo(amount) < 0) {
            throw new InsufficientBalanceException();
        }

        // 3. 乐观锁更新
        int updated = accountRepository.decreaseBalanceWithVersion(
            accountId,
            amount,
            version
        );

        if (updated == 0) {
            // 更新失败，说明有并发修改
            throw new ConcurrentModificationException();
        }

        return WithdrawResult.success();
    }
}
```

```java
@Repository
public interface AccountRepository extends JpaRepository<Account, Long> {

    @Modifying
    @Query("UPDATE Account a SET a.balance = a.balance - :amount, a.version = a.version + 1 " +
           "WHERE a.id = :id AND a.version = :version AND a.balance >= :amount")
    int decreaseBalanceWithVersion(
        @Param("id") Long id,
        @Param("amount") BigDecimal amount,
        @Param("version") Long version
    );
}
```

## 五、消息消费的幂等性

### 5.1 消息队列的 At-Least-Once 语义

大多数消息队列（如 Kafka、RocketMQ）提供的是 At-Least-Once 语义，即消息至少被消费一次。这意味着，同一条消息可能被消费多次。

幂等消费者必须能够处理重复消息，保证最终结果的一致性。

```java
@Service
public class IdempotentMessageConsumer {

    private final RedisIdempotencyService idempotencyService;
    private final OrderService orderService;

    @KafkaListener(topics = "order-events")
    public void consumeOrderEvent(ConsumerRecord<String, String> record) {
        String messageId = record.key(); // 消息的唯一 ID
        String payload = record.value();

        // 使用消息 ID 作为幂等键
        idempotencyService.execute(messageId, () -> {
            OrderEvent event = parseEvent(payload);
            processEvent(event);
            return null;
        }, Void.class);
    }

    private void processEvent(OrderEvent event) {
        switch (event.getType()) {
            case "ORDER_CREATED":
                orderService.handleOrderCreated(event);
                break;
            case "ORDER_PAID":
                orderService.handleOrderPaid(event);
                break;
            case "ORDER_CANCELLED":
                orderService.handleOrderCancelled(event);
                break;
            default:
                log.warn("未知事件类型: {}", event.getType());
        }
    }
}
```

### 5.2 消息处理的幂等设计

```java
@Service
public class OrderEventProcessor {

    private final RedisIdempotencyService idempotencyService;
    private final OrderRepository orderRepository;

    /**
     * 处理订单创建事件（幂等）
     */
    public void handleOrderCreated(OrderEvent event) {
        String idempotencyKey = "ORDER_CREATED:" + event.getOrderId();

        idempotencyService.execute(idempotencyKey, () -> {
            // 检查订单是否已存在
            if (orderRepository.existsByOrderId(event.getOrderId())) {
                log.info("订单已存在，忽略重复事件: {}", event.getOrderId());
                return null;
            }

            // 创建订单
            Order order = new Order();
            order.setOrderId(event.getOrderId());
            order.setUserId(event.getUserId());
            order.setAmount(event.getAmount());
            order.setStatus(OrderStatus.CREATED);
            orderRepository.save(order);

            // 扣减库存
            inventoryService.decrease(event.getProductId(), event.getQuantity());

            return null;
        }, Void.class);
    }

    /**
     * 处理订单支付事件（幂等）
     */
    public void handleOrderPaid(OrderEvent event) {
        String idempotencyKey = "ORDER_PAID:" + event.getOrderId();

        idempotencyService.execute(idempotencyKey, () -> {
            Order order = orderRepository.findByOrderId(event.getOrderId());
            if (order == null) {
                log.warn("订单不存在: {}", event.getOrderId());
                return null;
            }

            // 检查订单状态
            if (order.getStatus() != OrderStatus.CREATED) {
                log.info("订单状态不是待支付，忽略事件: {}, status={}",
                    event.getOrderId(), order.getStatus());
                return null;
            }

            // 更新订单状态
            order.setStatus(OrderStatus.PAID);
            order.setPaidAt(new Date());
            orderRepository.save(order);

            return null;
        }, Void.class);
    }
}
```

## 六、事务性发件箱模式

### 6.1 问题背景

在微服务架构中，服务之间通常通过消息队列进行异步通信。如果在处理业务逻辑的同时需要发送消息，常见的做法是：

```java
@Transactional
public void createOrder(Order order) {
    // 1. 保存订单
    orderRepository.save(order);

    // 2. 发送消息
    messageService.send("order.created", order);
}
```

但这种方式存在问题：如果步骤 1 成功，步骤 2 失败（消息发送超时），事务会回滚。但重启后这条消息不会被重新发送，因为订单已经回滚了。

### 6.2 事务性发件箱模式

事务性发件箱（Transactional Outbox）模式通过引入一张「发件箱表」来解决这个问题：

```java
// 1. 保存订单和消息到发件箱（同一个事务）
@Transactional
public void createOrder(Order order) {
    // 保存订单
    orderRepository.save(order);

    // 保存消息到发件箱
    OutboxMessage message = new OutboxMessage();
    message.setAggregateType("ORDER");
    message.setAggregateId(order.getId().toString());
    message.setEventType("ORDER_CREATED");
    message.setPayload(serialize(order));
    message.setStatus(OutboxStatus.PENDING);
    message.setCreatedAt(new Date());
    outboxRepository.save(message);
}

// 2. 定时任务扫描发件箱，发送消息
@Scheduled(fixedDelay = 100)
public void processOutbox() {
    List<OutboxMessage> messages = outboxRepository.findPendingMessages(100);

    for (OutboxMessage message : messages) {
        try {
            // 发送消息
            messageBroker.send(message.getEventType(), message.getPayload());

            // 标记为已发送
            message.setStatus(OutboxStatus.SENT);
            message.setSentAt(new Date());
            outboxRepository.save(message);
        } catch (Exception e) {
            // 记录错误，但不阻塞其他消息
            message.setRetryCount(message.getRetryCount() + 1);
            message.setLastError(e.getMessage());
            outboxRepository.save(message);
        }
    }
}
```

### 6.3 发件箱模式的幂等保证

发件箱模式天然具备幂等性，因为：

1. 订单创建和消息插入在同一个事务中，要么都成功，要么都失败
2. 定时任务只会发送 `PENDING` 状态的消息
3. 消费者可以通过消息 ID（发件箱消息的 ID）来判断是否已处理

```java
@KafkaListener(topics = "order-events")
public void consumeOrderEvent(ConsumerRecord<String, String> record) {
    String messageId = record.key();

    idempotencyService.execute(messageId, () -> {
        OutboxMessage outboxMessage = outboxRepository.findById(Long.parseLong(messageId));
        if (outboxMessage == null) {
            log.warn("发件箱消息不存在: {}", messageId);
            return null;
        }

        OrderEvent event = parseEvent(outboxMessage.getPayload());
        processEvent(event);

        return null;
    }, Void.class);
}
```

## 面试追问：幂等性相关高频问题

### 问题一：如何保证接口的幂等性？

**参考答案**：

保证接口幂等性主要有以下几种方案：

**幂等键方案**：客户端生成一个全局唯一的幂等键，服务端根据幂等键判断是否是重复请求。幂等键可以通过 Redis SETNX 或数据库唯一约束来存储和校验。

**状态机控制**：对于有明确状态流转的操作（如订单状态），通过状态机来保证幂等性。比如，只有「待支付」状态的订单才能转为「已支付」。

**乐观锁/版本号**：在更新操作时检查版本号或乐观锁，只有版本号匹配时才更新。

**去重表**：创建一个去重表，记录已处理过的请求 ID，通过数据库唯一约束来防止重复。

**接口设计**：尽量设计幂等性的接口，比如用 PUT 替代 POST，用 DELETE 替代多次状态更新。

### 问题二：幂等性和事务性有什么区别？

**参考答案**：

幂等性和事务性是两个不同的概念：

| 维度 | 幂等性 | 事务性 |
|---|---|---|
| 关注点 | 多次执行结果相同 | 多个操作要么都成功，要么都失败 |
| 应用场景 | 网络重试、重复提交 | 数据一致性 |
| 实现方式 | 幂等键、状态检查、乐观锁 | 数据库事务、分布式事务 |
| 关系 | 独立概念，可以组合使用 | 独立概念，可以组合使用 |

**两者的关系**：
- 幂等性是为了处理「重试」，事务性是为了保证「一致性」
- 有事务性不一定有幂等性（比如扣款操作在事务中执行多次会扣多次）
- 有幂等性不一定有事务性（比如覆盖更新天然幂等，但不保证原子性）
- 最佳实践是同时保证幂等性和事务性

### 问题三：消息消费如何保证幂等性？

**参考答案**：

消息消费保证幂等性的关键是「让消费者能够识别重复消息」。常用的方案：

**消息 ID 作为幂等键**：每条消息都有唯一的 ID，消费者根据 ID 判断是否已处理。处理前先检查，处理后记录。

**去重表**：消费者在处理消息前，先往去重表插入一条记录（使用消息 ID 作为唯一键）。如果插入成功，说明是首次处理；如果插入失败，说明已处理过。

**状态机检查**：消费者在处理消息前，先检查业务数据的状态，只有在正确状态下才处理。

**业务去重**：在业务逻辑中天然去重。比如扣款时检查余额是否足够，库存扣减时检查库存是否充足。

:::tip 延伸思考

幂等性保证了「重复执行不会产生副作用」，但并没有保证「结果一定是正确的」。比如，用户点击了两次支付，如果两次请求都成功扣款了（因为两次请求几乎同时到达，余额检查都通过了），系统会如何处理？请思考如何在幂等性的基础上，进一步保证结果的正确性。

:::
