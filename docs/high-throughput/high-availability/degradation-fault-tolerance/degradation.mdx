# 服务降级策略与兜底方案

凌晨三点，你被手机铃声惊醒——是监控系统的紧急告警。打开一看，原来是上游供应商的接口全部超时了。下游用户在疯狂刷新页面，订单系统的错误日志像瀑布一样飞流直下。你心想：完了，这是典型的「牵一发动全身」。

但等等，你突然想起来——系统里早就埋好了降级开关。你不慌不忙地打开配置中心，把「启用降级」开关从 `false` 拨到 `true`。瞬间，系统切换到「省电模式」：非核心功能灰掉了、商品详情页改用静态缓存返回、历史订单查询返回聚合数据而不是实时查询。用户的投诉电话开始减少，系统的错误率从 90% 降到了 5%。

这就是服务降级的力量——**不是让系统崩溃，而是让系统在「困难时期」优雅地「节衣缩食」**。

## 一、什么是降级——「断臂求生」的艺术

### 1.1 降级的本质

服务降级（Degradation）是指在系统负载过高、服务异常或依赖不可用时，主动放弃部分功能或降低服务质量，以保证核心功能的可用性。

这听起来像是在「偷工减料」，但实际上恰恰相反。降级是一种**主动选择**，而不是被动崩溃。你可以选择：
- 放弃哪些功能（边缘功能 vs 核心功能）
- 用什么替代方案（缓存 vs 静态数据 vs 默认值）
- 降级到什么程度（部分降级 vs 完全降级）

就像一个手机没电了，你会关掉 GPS、降低屏幕亮度、关闭后台应用——不是为了省电而省电，而是为了**保证核心功能（打电话、发短信）还能用**。

### 1.2 降级 vs 熔断 vs 降级：三个概念的辨析

很多人搞不清这三个概念的关系：

| 概念 | 触发条件 | 目的 | 恢复方式 |
|---|---|---|---|
| **降级（Degradation）** | 服务不可用、功能异常、资源不足 | 保证核心功能，牺牲边缘功能 | 人工恢复或自动恢复 |
| **熔断（Circuit Breaker）** | 下游服务错误率超阈值 | 快速失败，防止雪崩 | 半开状态试探，成功则关闭 |
| **降级（Fallback）** | 请求失败 | 提供兜底响应，避免异常传播 | 下次请求自动重试 |

简单来说：
- **降级**是一种设计理念，是从系统架构层面考虑「如何应对故障」
- **熔断**是一种实现机制，是降级理念在代码层面的具体实现
- **降级**是一种兜底手段，是请求失败时的最后防线

:::tip 一个形象的比喻

把系统比作一艘船：

- **降级**：风暴来了，船长决定「丢掉救生艇、封住非必要舱门」，保证船体不进水
- **熔断**：电流过载了，保险丝自动跳闸，切断电源防止火灾
- **降级**：某个灯泡坏了，工程师用备用手电筒照亮关键区域

:::

### 1.3 为什么要降级

降级的核心价值在于三个「保护」：

**保护核心功能**：当系统资源紧张时，优先保证核心业务流程可用。比如电商系统，核心是下单和支付，商品推荐、用户评论就是可以牺牲的边缘功能。

**保护下游服务**：当下游服务不可用时，如果不降级，本系统会不断重试，最终拖垮自己也拖垮下游。降级可以让下游有喘息的机会。

**保护用户体验**：与其让用户面对一个一直转圈的页面或一个 500 错误，不如给用户一个「功能暂时不可用，但我们正在抢修」的友好提示。

## 二、降级的类型：从页面到数据的全景图

### 2.1 按层级分类

降级可以在系统的不同层级实现：

| 降级层级 | 实现位置 | 降级内容 | 响应速度 |
|---|---|---|---|
| **页面降级** | CDN、网关 | 返回静态页面、缓存页面 | 最快 |
| **请求降级** | 网关、应用层 | 跳过非核心请求、返回默认值 | 快 |
| **数据降级** | 应用层、数据层 | 缓存数据聚合、默认值返回 | 中等 |
| **功能降级** | 代码层面 | 关闭复杂计算、简化逻辑 | 慢 |

### 2.2 页面降级

页面降级是最直观的降级方式，用户能直接感知。

```java
@Controller
public class PageController {

    @Autowired
    private ProductService productService;

    @Autowired
    private DegradationConfig degradationConfig;

    @GetMapping("/product/{id}")
    public String getProductDetail(@PathVariable Long id, Model model) {
        // 检查是否开启降级
        if (degradationConfig.isProductDetailDegraded()) {
            // 返回静态页面
            return "product/static-detail";
        }

        // 正常逻辑
        ProductDetail detail = productService.getDetail(id);
        model.addAttribute("product", detail);
        return "product/detail";
    }
}
```

常见的页面降级策略：
- **静态化降级**：提前生成静态 HTML，发生故障时直接返回
- **缓存降级**：返回 CDN 或 Redis 中的缓存数据
- **友好错误页**：返回「服务繁忙，请稍后再试」的友好页面

### 2.3 请求降级

请求降级是在网关或应用层拦截请求，阻止其访问不可用的下游服务。

```java
@Component
public class RequestFilter {

    @Autowired
    private DegradationConfig degradationConfig;

    @Bean
    public FilterRegistrationBean<RequestDegradationFilter> degradationFilter() {
        FilterRegistrationBean<RequestDegradationFilter> registration = new FilterRegistrationBean<>();
        registration.setFilter(new RequestDegradationFilter(degradationConfig));
        registration.addUrlPatterns("/*");
        return registration;
    }
}

public class RequestDegradationFilter implements Filter {

    private final DegradationConfig degradationConfig;

    @Override
    public void doFilter(ServletRequest request, ServletResponse response, FilterChain chain)
            throws IOException, ServletException {
        HttpServletRequest httpRequest = (HttpServletRequest) request;
        HttpServletResponse httpResponse = (HttpServletResponse) response;

        String path = httpRequest.getRequestURI();

        // 检查该路径是否需要降级
        DegradationRule rule = degradationConfig.getRule(path);
        if (rule != null && rule.isDegraded()) {
            // 返回降级响应
            httpResponse.setStatus(200);
            httpResponse.setContentType("application/json");
            httpResponse.getWriter().write(rule.getFallbackResponse());
            return;
        }

        chain.doFilter(request, response);
    }
}
```

### 2.4 数据降级

数据降级是返回缓存数据、历史数据或计算好的聚合数据，而不是实时查询。

```java
@Service
public class OrderStatisticsService {

    @Autowired
    private OrderRepository orderRepository;

    @Autowired
    private RedisTemplate<String, Object> redisTemplate;

    @Autowired
    private DegradationConfig degradationConfig;

    public OrderStatistics getOrderStatistics(Long userId) {
        // 检查是否开启数据降级
        if (degradationConfig.isStatisticsDegraded()) {
            // 降级策略：返回聚合缓存数据
            return getDegradedStatistics();
        }

        // 正常逻辑：实时查询
        return orderRepository.getStatistics(userId);
    }

    private OrderStatistics getDegradedStatistics() {
        // 从 Redis 获取昨日聚合数据
        String cacheKey = "degraded:order:statistics:yesterday";
        OrderStatistics cached = (OrderStatistics) redisTemplate.opsForValue().get(cacheKey);

        if (cached != null) {
            cached.setDegraded(true);
            cached.setMessage("当前数据为昨日统计，实时数据暂时不可用");
            return cached;
        }

        // 兜底：返回零值
        return OrderStatistics.zero();
    }
}
```

## 三、兜底方案设计：如何设计良好的降级逻辑

### 3.1 降级方案设计原则

设计降级方案时，需要遵循以下原则：

**核心优先原则**：降级是为了保护核心功能，所以要先明确「什么是核心，什么是边缘」。核心功能是「必须可用」的，边缘功能是「可以牺牲」的。

**快速失败原则**：降级逻辑应该尽可能简单和快速。降级代码本身不应该成为新的性能瓶颈。

**有感降级原则**：降级后要给用户明确的提示，让用户知道系统在降级运行，而不是无声无息地返回错误数据。

**可配置原则**：降级开关必须是可配置的，支持动态切换，不需要重新部署。

**可回滚原则**：降级应该能够快速回滚，恢复到正常状态。

### 3.2 常见的降级方案

| 降级场景 | 降级方案 | 示例 |
|---|---|---|
| 依赖服务不可用 | 返回缓存数据 | 秒杀详情页返回昨日缓存 |
| 数据库不可用 | 返回默认值 | 用户积分返回 0 |
| 第三方支付不可用 | 跳转充值页面 | 「支付通道繁忙，请稍后再试」 |
| 推荐服务不可用 | 返回默认推荐 | 返回热销榜单 |
| 搜索服务不可用 | 降级到分类浏览 | 「搜索暂时不可用，请浏览分类」 |
| 消息通知不可用 | 记录日志，后续补发 | 「将在 24 小时内送达」 |

### 3.3 降级代码示例：多级降级策略

```java
@Service
public class ProductDetailService {

    @Autowired
    private ProductApi productApi;

    @Autowired
    private ProductCacheService cacheService;

    @Autowired
    private StaticDataService staticDataService;

    @Autowired
    private DegradationConfig degradationConfig;

    public ProductDetail getProductDetail(Long productId) {
        // 第一级：尝试正常获取
        try {
            if (!degradationConfig.isProductApiDegraded()) {
                ProductDetail detail = productApi.getDetail(productId);
                // 写入缓存
                cacheService.put(productId, detail);
                return detail;
            }
        } catch (Exception e) {
            log.warn("Product API 调用失败，触发降级: {}", e.getMessage());
        }

        // 第二级：尝试从缓存获取
        try {
            ProductDetail cached = cacheService.get(productId);
            if (cached != null) {
                cached.setFromCache(true);
                return cached;
            }
        } catch (Exception e) {
            log.warn("缓存获取失败，继续降级: {}", e.getMessage());
        }

        // 第三级：返回静态数据
        try {
            return staticDataService.getProductDetail(productId);
        } catch (Exception e) {
            log.error("静态数据获取也失败了: {}", e.getMessage());
        }

        // 兜底：抛出业务异常
        throw new ProductUnavailableException("商品暂时无法获取，请稍后再试");
    }
}
```

:::warning 降级层级不要太多

降级层级越多，代码复杂度越高，排查问题越困难。推荐最多三级：「实时 → 缓存 → 静态/默认值」。

:::

## 四、降级触发条件与阈值设计

### 4.1 触发条件类型

降级的触发条件可以分为以下几类：

| 触发类型 | 具体指标 | 阈值示例 | 说明 |
|---|---|---|---|
| **错误率触发** | HTTP 5xx 比例、异常率 | `> 5%` | 下游服务持续报错 |
| **延迟触发** | 平均 RT、P99 RT | `> 500ms` | 下游服务响应变慢 |
| **资源触发** | CPU、内存、连接池 | `> 80%` | 本系统资源不足 |
| **人工触发** | 配置开关 | - | 人工决定降级 |
| **定时触发** | 时间段 | 22:00-6:00 | 低峰期降级 |

### 4.2 阈值设计原则

**不要拍脑袋设阈值**：阈值应该基于历史数据和压测结果来确定，而不是随意写一个数字。

**阈值要有弹性**：设置软阈值和硬阈值，软阈值触发警告，硬阈值触发降级。

**阈值要考虑业务时段**：高峰期和低峰期的阈值可能不同。

```java
@Configuration
public class DegradationThresholdConfig {

    @Value("${degradation.cpu.warning:70}")
    private int cpuWarningThreshold;

    @Value("${degradation.cpu.critical:85}")
    private int cpuCriticalThreshold;

    @Value("${degradation.error-rate.warning:3}")
    private double errorRateWarningThreshold;

    @Value("${degradation.error-rate.critical:5}")
    private double errorRateCriticalThreshold;

    @Value("${degradation.rt.warning:300}")
    private int rtWarningThreshold;

    @Value("${degradation.rt.critical:500}")
    private int rtCriticalThreshold;

    // 根据时段动态调整阈值
    public int getEffectiveThreshold(int baseThreshold) {
        int hour = LocalTime.now().getHour();
        // 低峰期（22:00-6:00）阈值降低 20%
        if (hour >= 22 || hour < 6) {
            return (int) (baseThreshold * 0.8);
        }
        // 高峰期（9:00-11:00, 14:00-18:00）阈值提高 10%
        if ((hour >= 9 && hour < 11) || (hour >= 14 && hour < 18)) {
            return (int) (baseThreshold * 1.1);
        }
        return baseThreshold;
    }
}
```

## 五、自动化降级 vs 人工降级

### 5.1 自动化降级

自动化降级是通过监控系统自动触发的，不需要人工干预。

```java
@Component
public class AutoDegradationTrigger {

    @Autowired
    private DegradationConfig degradationConfig;

    @Autowired
    private MetricsService metricsService;

    @Autowired
    private AlertService alertService;

    // 定时检查是否需要降级
    @Scheduled(fixedRate = 5000)
    public void checkAndTriggerDegradation() {
        // 检查各项指标
        double errorRate = metricsService.getErrorRate("product-api");
        int cpuUsage = metricsService.getCpuUsage();
        int rt = metricsService.getAvgRt("order-service");

        // 检查是否满足降级条件
        if (shouldDegrade(errorRate, cpuUsage, rt)) {
            triggerDegradation(errorRate, cpuUsage, rt);
        } else if (shouldRecover()) {
            recoverFromDegradation();
        }
    }

    private boolean shouldDegrade(double errorRate, int cpuUsage, int rt) {
        // 硬阈值判断
        return errorRate > degradationConfig.getErrorRateCritical()
            || cpuUsage > degradationConfig.getCpuCritical()
            || rt > degradationConfig.getRtCritical();
    }

    private void triggerDegradation(double errorRate, int cpuUsage, int rt) {
        degradationConfig.enableDegradation();

        // 发送告警
        alertService.send("服务降级已自动触发", String.format(
            "errorRate=%f, cpu=%d, rt=%d", errorRate, cpuUsage, rt));
    }

    private boolean shouldRecover() {
        // 需要所有指标都恢复正常一段时间才恢复
        double errorRate = metricsService.getErrorRate("product-api");
        int cpuUsage = metricsService.getCpuUsage();
        int rt = metricsService.getAvgRt("order-service");

        return errorRate < degradationConfig.getErrorRateWarning()
            && cpuUsage < degradationConfig.getCpuWarning()
            && rt < degradationConfig.getRtWarning();
    }
}
```

### 5.2 人工降级

人工降级是通过配置平台手动触发的，通常用于：
- 预判即将到来的流量洪峰（比如双十一前）
- 上线前的保护性降级
- 降级自动化失败时的补救

```java
@Configuration
public class ManualDegradationConfig {

    @Bean
    public DynamicPropertyUpdater degradationDynamicProperty() {
        return new DynamicPropertyUpdater() {
            private volatile boolean degraded = false;

            @Override
            public void setDegraded(boolean degraded) {
                this.degraded = degraded;
            }

            @Override
            public boolean isDegraded() {
                return degraded;
            }
        };
    }
}
```

:::tip 最佳实践：自动化 + 人工双重保障

自动化降级是「守门员」，负责处理突发情况；人工降级是「教练」，负责处理可预见的风险。两者结合，才能应对各种场景。

:::

## 六、降级治理与可观测性

### 6.1 降级状态管理

降级状态需要集中管理，方便查看和操作：

```java
@RestController
@RequestMapping("/admin/degradation")
public class DegradationAdminController {

    @Autowired
    private DegradationConfig degradationConfig;

    @Autowired
    private DegradationHistoryService historyService;

    @GetMapping("/status")
    public DegradationStatus getStatus() {
        return degradationConfig.getStatus();
    }

    @PostMapping("/enable")
    public Result<Void> enableDegradation(@RequestParam String reason) {
        degradationConfig.enableDegradation();
        historyService.record(new DegradationEvent("ENABLE", reason, getCurrentUser()));
        return Result.success();
    }

    @PostMapping("/disable")
    public Result<Void> disableDegradation(@RequestParam String reason) {
        degradationConfig.disableDegradation();
        historyService.record(new DegradationEvent("DISABLE", reason, getCurrentUser()));
        return Result.success();
    }

    @GetMapping("/history")
    public List<DegradationEvent> getHistory(
            @RequestParam(required = false) LocalDateTime startTime,
            @RequestParam @DateTimeFormat(iso = DATE_TIME) LocalDateTime endTime) {
        return historyService.getEvents(startTime, endTime);
    }
}
```

    public void recordNormalRequest() {
        normalRequests.increment();
    }
}

### 7.2 使用 Sentinel 的降级规则

@Configuration
public class SentinelDegradationConfig {

    @PostConstruct
    public void initDegradationRules() {
        // 慢调用比例降级
        DegradeRule slowRatioRule = new DegradeRule("productApi")
            .setGrade(CircuitBreakerStrategy.SLOW_REQUEST_RATIO.getType())
            .setCount(0.5) // 慢调用比例阈值
            .setSlowRatioThreshold(500) // 慢调用阈值：500ms
            .setMinRequestAmount(5) // 最小请求数
            .setStatIntervalMs(10000) // 统计时间窗口
            .setTimeWindow(10); // 熔断时长

        // 异常比例降级
        DegradeRule errorRatioRule = new DegradeRule("orderApi")
            .setGrade(CircuitBreakerStrategy.ERROR_RATIO.getType())
            .setCount(0.3) // 异常比例阈值
            .setMinRequestAmount(5)
            .setStatIntervalMs(10000)
            .setTimeWindow(10);

        DegradeRuleManager.loadRules(Arrays.asList(slowRatioRule, errorRatioRule));
    }
}

## 面试追问：降级相关高频问题

### 问题一：降级和熔断的区别是什么？

**参考答案**：

降级和熔断虽然经常一起出现，但它们是两个不同的概念：

**降级**是一种主动的设计选择，关注的是「功能层面的取舍」——当系统压力过大时，主动关闭一些边缘功能，保证核心功能可用。

**熔断**是一种被动的保护机制，关注的是「调用层面的快速失败」——当下游服务持续失败时，不再继续调用，而是直接返回错误，避免无效请求拖垮系统。

两者的关系是：**熔断是实现降级的一种手段**。熔断后的「快速失败返回」本身就是一种降级策略。

| 维度 | 降级 | 熔断 |
|---|---|---|
| 关注点 | 功能完整性 | 调用成功率 |
| 触发条件 | 资源不足、功能异常 | 下游错误率超阈值 |
| 实现方式 | 关闭功能、返回默认值 | 快速失败、跳过调用 |
| 恢复方式 | 人工或自动恢复 | 半开试探成功后自动恢复 |

### 问题二：如何设计一个好的降级方案？

**参考答案**：

设计降级方案需要考虑以下几个方面：

**明确核心功能**：先梳理业务流程，确定哪些是核心功能，哪些是边缘功能。核心功能是必须保证的，边缘功能是可以牺牲的。

**设计降级路径**：为每个边缘功能设计降级路径，包括：
- 降级后的返回值是什么
- 降级逻辑在哪里实现
- 降级触发条件是什么

**考虑用户体验**：降级后要给用户明确的提示，让用户知道系统在降级运行，而不是无声无息地返回错误数据。

**设置降级阈值**：阈值要基于历史数据和压测结果，不要拍脑袋设置。建议设置软阈值和硬阈值，软阈值触发警告，硬阈值触发降级。

**支持快速切换**：降级开关必须是可配置的，支持动态切换，不需要重新部署。

**建立观测能力**：降级发生时，能够被监控系统捕获，并通知相关人员。

### 问题三：降级会带来哪些风险？

**参考答案**：

降级虽然能保护系统，但也会带来一些风险：

**数据一致性问题**：降级后返回的数据可能不是实时的，可能存在数据滞后。如果业务对数据一致性要求很高，降级需要特别小心。

**降级逻辑本身可能出错**：降级代码也是代码，也可能存在 bug。如果降级逻辑本身有问题，降级反而会导致更严重的问题。

**降级蔓延**：如果多个服务同时降级，可能导致整个系统降级，而不是局部降级。需要设计好降级的范围和层级。

**用户误导**：如果降级后的数据和正常数据差异很大，用户可能会产生困惑或误解。

:::tip 延伸思考

降级和熔断都是系统保护手段，但它们是「事后补救」还是「事前预防」？如何在系统设计阶段就考虑好降级策略，而不是等到故障发生了才手忙脚乱？

:::
