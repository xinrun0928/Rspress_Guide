# 自适应限流：基于系统负载的动态调整

凌晨两点，你的系统突然收到了一波流量洪峰——可能是某个营销活动提前曝光，也可能是被恶意刷接口。静态限流规则告诉你「每秒最多 1000 个请求」，但你的系统其实能扛住 2000 个。结果呢？限流白白拒绝了 1000 个请求，用户怨声载道，你的系统资源还闲置着。

反过来，假设你的限流规则设置得太宽松——「每秒最多 5000 个请求」。系统实际只能处理 3000 个，结果数据库被打爆了，服务开始出现大量超时。

这就是静态限流的困境：**你不知道系统的真实容量，也不知道流量的真实大小**。

如何破局？答案是自适应限流——让系统自己「感知」自己的状态，动态调整限流阈值。

这篇文章，我们来聊聊自适应限流的设计与实现。

---

## 一、为什么需要自适应限流

### 1.1 静态限流的困境

传统的限流规则是静态的——你提前设定一个阈值，系统按这个阈值执行。比如：

- 每秒最多 1000 个请求
- 每个用户每分钟最多 100 次
- 每个 IP 每小时最多 10000 次

静态限流简单直接，但有几个致命问题：

**第一：阈值难以精确设置**

系统的容量不是固定的。它会随着时间、负载、依赖服务状态而变化。早上系统空闲，晚上高峰，可能相差 5 倍甚至 10 倍。静态阈值设置低了，资源浪费；设置高了，系统扛不住。

**第二：流量模式不可预测**

正常情况下系统能扛 5000 QPS，但突然来了流量洪峰，瞬间 10000 QPS，数据库直接被打爆。静态限流无法感知这种突发。

**第三：无法感知下游健康状态**

假设 Redis 突然响应变慢（从 1ms 变成 100ms），系统的实际吞吐量会急剧下降。但静态限流不知道，它还是按照原来的阈值放行请求，结果请求堆积，服务雪崩。

### 1.2 自适应限流的核心思想

自适应限流的核心是**让系统「自己知道自己能扛多少」**。

它通过采集系统的实时指标（CPU、内存、响应时间、队列深度等），动态计算当前的最佳限流阈值。当系统健康时，放宽限流；当系统压力大时，收紧限流。

```
静态限流：
请求 → [固定阈值 1000/s] → 系统

自适应限流：
系统指标采集 → 动态阈值计算 → 限流决策
  ↑
CPU 70% → 阈值 1500/s
CPU 90% → 阈值 800/s
CPU 50% → 阈值 2000/s
```

## 二、系统指标的采集与监控

### 2.1 核心监控指标

自适应限流依赖对系统状态的实时感知。常用的监控指标包括：

**资源类指标**

| 指标 | 说明 | 采集方式 |
|---|---|---|
| CPU 使用率 | 系统 CPU 负载，反映计算能力 | `OperatingSystemMXBean` |
| 内存使用率 | 堆内存/非堆内存占用 | `Runtime` |
| JVM 线程数 | 活跃线程数，反映并发能力 | `ThreadMXBean` |
| 线程池队列深度 | 等待执行的任务数 | 线程池 API |
| GC 频率和耗时 | 内存压力大时 GC 会影响性能 | JMX/G1 日志 |
| 磁盘 I/O | 磁盘繁忙程度 | 系统工具 |

**性能类指标**

| 指标 | 说明 | 采集方式 |
|---|---|---|
| 响应时间（RT） | 接口平均响应时间 | APM/日志 |
| QPS | 每秒请求数 | 请求计数器 |
| 错误率 | 请求失败比例 | 错误计数器 |
| 超时率 | 请求超时的比例 | 超时计数器 |
| 线程池饱和度 | 工作线程忙碌比例 | 线程池状态 |

**依赖服务指标**

| 指标 | 说明 | 采集方式 |
|---|---|---|
| 数据库连接池使用率 | 连接池繁忙程度 | 连接池 API |
| Redis 响应时间 | 缓存服务健康度 | Redis 客户端 |
| 下游服务响应时间 | 依赖服务响应速度 | HTTP 客户端 |
| 下游服务错误率 | 下游服务可用性 | 错误计数器 |

### 2.2 指标采集实现

```java title="系统指标采集器"]
@Service
public class SystemMetricsCollector {
    private final OperatingSystemMXBean osBean;
    private final Runtime runtime;
    private final ThreadMXBean threadBean;

    // 滑动窗口：记录最近 N 秒的数据
    private final SlidingWindow<TimestampedValue> cpuWindow = new SlidingWindow<>(60);
    private final SlidingWindow<TimestampedValue> rtWindow = new SlidingWindow<>(60);
    private final SlidingWindow<TimestampedValue> errorRateWindow = new SlidingWindow<>(60);

    /**
     * 采集系统指标
     */
    public SystemMetrics collect() {
        SystemMetrics metrics = new SystemMetrics();
        metrics.setTimestamp(System.currentTimeMillis());

        // CPU 使用率（0-100）
        double cpuUsage = osBean.getSystemCpuLoad() * 100;
        metrics.setCpuUsage(cpuUsage);
        cpuWindow.add(new TimestampedValue(cpuUsage));

        // 内存使用率
        long totalMemory = runtime.totalMemory();
        long freeMemory = runtime.freeMemory();
        long usedMemory = totalMemory - freeMemory;
        double memoryUsage = (double) usedMemory / totalMemory * 100;
        metrics.setMemoryUsage(memoryUsage);

        // 活跃线程数
        int activeThreads = threadBean.getThreadCount();
        metrics.setActiveThreads(activeThreads);

        // GC 统计
        GCMetrics gcMetrics = collectGCMetrics();
        metrics.setGcMetrics(gcMetrics);

        // 从滑动窗口计算平均指标
        metrics.setAvgCpuUsage(cpuWindow.average());
        metrics.setAvgRt(rtWindow.average());
        metrics.setAvgErrorRate(errorRateWindow.average());

        return metrics;
    }

    private GCMetrics collectGCMetrics() {
        GCMetrics gc = new GCMetrics();
        // 通过 JMX 获取 GC 信息
        List<GarbageCollectorMXBean> gcBeans = ManagementFactory.getGarbageCollectorMXBeans();
        long totalCollectionCount = gcBeans.stream()
                .mapToLong(GarbageCollectorMXBean::getCollectionCount)
                .sum();
        long totalCollectionTime = gcBeans.stream()
                .mapToLong(GarbageCollectorMXBean::getCollectionTime)
                .sum();
        gc.setCollectionCount(totalCollectionCount);
        gc.setCollectionTime(totalCollectionTime);
        return gc;
    }
}
```

### 2.3 滑动窗口算法

为了平滑瞬时波动，自适应限流通常使用滑动窗口来计算指标：

```java title="滑动窗口实现"]
public class SlidingWindow<T> {
    private final int windowSize;           // 窗口大小（秒）
    private final List<Node<T>> buckets;    // 时间桶
    private final AtomicInteger index = new AtomicInteger(0);
    private final long startTime;

    public SlidingWindow(int windowSize) {
        this.windowSize = windowSize;
        this.buckets = new ArrayList<>(windowSize);
        this.startTime = System.currentTimeMillis();

        // 初始化时间桶
        for (int i = 0; i < windowSize; i++) {
            buckets.add(new Node<>());
        }
    }

    /**
     * 添加数据点
     */
    public void add(T value) {
        int idx = (int) ((System.currentTimeMillis() - startTime) / 1000) % windowSize;
        buckets.get(idx).add(value);
    }

    /**
     * 计算滑动窗口内的平均值
     */
    public double average() {
        long now = System.currentTimeMillis();
        double sum = 0;
        int count = 0;

        for (int i = 0; i < windowSize; i++) {
            Node<T> bucket = buckets.get(i);
            long bucketTime = startTime + i * 1000;

            // 跳过过期桶
            if (bucketTime > now) {
                continue;
            }

            Double val = bucket.getAverage();
            if (val != null) {
                sum += val;
                count++;
            }
        }

        return count > 0 ? sum / count : 0;
    }

    /**
     * 获取滑动窗口内的最大值
     */
    public double max() {
        long now = System.currentTimeMillis();
        double max = 0;

        for (int i = 0; i < windowSize; i++) {
            Node<T> bucket = buckets.get(i);
            long bucketTime = startTime + i * 1000;

            if (bucketTime > now) {
                continue;
            }

            Double val = bucket.getMax();
            if (val != null) {
                max = Math.max(max, val);
            }
        }

        return max;
    }

    private static class Node<T> {
        private final List<T> values = new CopyOnWriteArrayList<>();
        private volatile Double cachedAverage;
        private volatile Double cachedMax;
        private volatile long lastUpdate;

        public void add(T value) {
            values.add(value);
            lastUpdate = System.currentTimeMillis();
            cachedAverage = null;  // invalidate cache
            cachedMax = null;
        }

        public Double getAverage() {
            if (values.isEmpty()) return null;
            if (cachedAverage == null) {
                cachedAverage = values.stream()
                        .filter(v -> v instanceof Number)
                        .mapToDouble(v -> ((Number) v).doubleValue())
                        .average()
                        .orElse(0);
            }
            return cachedAverage;
        }

        public Double getMax() {
            if (values.isEmpty()) return null;
            if (cachedMax == null) {
                cachedMax = values.stream()
                        .filter(v -> v instanceof Number)
                        .mapToDouble(v -> ((Number) v).doubleValue())
                        .max()
                        .orElse(0);
            }
            return cachedMax;
        }
    }
}
```

## 三、自适应限流算法：从理论到实现

### 3.1 BBR 算法：TCP BBR 的启发

提到自适应限流，不得不提 BBR（Bottleneck Bandwidth and RTT）算法。这是 Google 在 2016 年提出的 TCP 拥塞控制算法，它的核心理念是：**不要只看丢包率，而是看带宽和 RTT 的真实关系**。

传统的 TCP 拥塞控制基于丢包——丢包了就认为网络拥塞，开始减少发送窗口。但 BBR 认为，丢包不一定是拥塞，也可能是暂时的网络抖动。BBR 通过实时测量**可用带宽**和**最小 RTT**，动态调整发送速率。

自适应限流借鉴了 BBR 的思路：**不要只看请求数，而是看系统的真实吞吐量和响应时间**。

### 3.2  Sentinel 的自适应限流实现

Sentinel 是阿里巴巴开源的流量控制组件，它的自适应限流基于**系统 Load（系统负载）**和**平均 RT（响应时间）**两个维度。

```java title="Sentinel 自适应限流核心逻辑"]
public class AdaptiveFlowRule extends FlowRule {
    // 系统自适应模式
    private String strategy;  // RT / QPS / THREAD / LOAD

    // 阈值配置
    private double highestSystemLoad;      // 最高系统负载（默认 6.0）
    private double highestCpuUsage;        // 最高 CPU 使用率（默认 0.9）
    private double avgRt;                  // 平均响应时间阈值
    private double maxQps;                  // 最大 QPS 阈值
    private int maxThread;                 // 最大并发线程数
}

/**
 * Sentinel 系统自适应限流器
 */
public class SentinelAdaptive limiter {
    private final SystemMetricsCollector metricsCollector;
    private final double highestSystemLoad;
    private final double highestCpuUsage;

    /**
     * 计算自适应限流阈值
     */
    public AdaptiveLimit calculateLimit() {
        SystemMetrics metrics = metricsCollector.collect();
        AdaptiveLimit limit = new AdaptiveLimit();

        // 1. 基于系统 Load 计算
        double loadBasedQps = calculateByLoad(metrics);
        limit.setLoadBasedQps(loadBasedQps);

        // 2. 基于 CPU 使用率计算
        double cpuBasedQps = calculateByCpu(metrics);
        limit.setCpuBasedQps(cpuBasedQps);

        // 3. 基于 RT 计算
        double rtBasedQps = calculateByRt(metrics);
        limit.setRtBasedQps(rtBasedQps);

        // 4. 基于并发线程数计算
        double threadBasedQps = calculateByThread(metrics);
        limit.setThreadBasedQps(threadBasedQps);

        // 5. 取所有阈值的最小值
        double finalQps = Collections.min(Arrays.asList(
                loadBasedQps, cpuBasedQps, rtBasedQps, threadBasedQps,
                maxQpsLimit  // 配置的最大 QPS
        ));

        limit.setFinalQps(finalQps);
        return limit;
    }

    /**
     * 基于系统 Load 计算
     * 公式：highestSystemLoad / currentLoad * 当前 QPS
     */
    private double calculateByLoad(SystemMetrics metrics) {
        double currentLoad = metrics.getSystemLoad();
        if (currentLoad <= 0) {
            return Double.MAX_VALUE;
        }
        // 当 load = 6.0 时，不限制
        // 当 load > 6.0 时，逐渐收紧
        double qps = highestSystemLoad / currentLoad * metrics.getCurrentQps();
        return Math.max(qps, 20);  // 最低保留 20 QPS
    }

    /**
     * 基于 CPU 使用率计算
     * 公式：(1 - cpuUsage) * highestCpuUsage + 一个基准值
     */
    private double calculateByCpu(SystemMetrics metrics) {
        double cpuUsage = metrics.getCpuUsage() / 100;  // 转为小数
        if (cpuUsage >= highestCpuUsage) {
            return minRequestPerSecond;  // CPU 过载，返回最小 QPS
        }
        // CPU 使用率越低，QPS 越高
        return metrics.getCurrentQps() * (1 - cpuUsage) / highestCpuUsage;
    }

    /**
     * 基于平均 RT 计算
     * 公式：RT_threshold / avgRt * currentQps
     */
    private double calculateByRt(SystemMetrics metrics) {
        double avgRt = metrics.getAvgRt();
        if (avgRt <= 0) {
            return Double.MAX_VALUE;
        }
        return avgRtThreshold / avgRt * metrics.getCurrentQps();
    }

    /**
     * 基于并发线程数计算
     */
    private double calculateByThread(SystemMetrics metrics) {
        int activeThreads = metrics.getActiveThreads();
        if (activeThreads >= maxThread) {
            return minRequestPerSecond;
        }
        // 线程数越少，QPS 越高
        return metrics.getCurrentQps() * (1 - (double) activeThreads / maxThread);
    }
}
```

### 3.3 PID 控制器：反馈控制的思路

PID（比例-积分-微分）控制器是经典的控制理论算法，也可以用于自适应限流。

```java title="PID 控制器实现"]
/**
 * PID 限流控制器
 */
public class PidRateLimiter {
    // PID 参数
    private final double kp;  // 比例系数
    private final double ki;  // 积分系数
    private final double kd;  // 微分系数

    // 状态
    private double integral = 0;    // 积分项
    private double lastError = 0;  // 上一次误差
    private long lastTime;

    // 目标值
    private final double targetRt;  // 目标响应时间
    private final double targetErrorRate;  // 目标错误率

    /**
     * 计算下一个限流阈值
     */
    public double calculateThreshold(double currentRt, double currentErrorRate) {
        long now = System.currentTimeMillis();
        double dt = lastTime > 0 ? (now - lastTime) / 1000.0 : 0;
        lastTime = now;

        // 计算误差：RT 误差 + 错误率误差
        double rtError = targetRt - currentRt;
        double errorRateError = targetErrorRate - currentErrorRate;
        double error = rtError + errorRateError * 100;  // 错误率权重放大

        // PID 计算
        double proportional = kp * error;  // 比例项

        integral += error * dt;  // 积分项（有积累效应，消除稳态误差）
        double derivative = dt > 0 ? kd * (error - lastError) / dt : 0;  // 微分项（预测未来）
        lastError = error;

        // 输出：新的 QPS 阈值
        double output = proportional + ki * integral + derivative;

        // 限幅：阈值必须在 [minQps, maxQps] 范围内
        return Math.max(minQps, Math.min(maxQps, output));
    }

    /**
     * 根据 PID 输出调整限流阈值
     */
    public RateLimitResult adjustLimit(double pidOutput, double currentQps) {
        double newQps = currentQps * pidOutput;

        // 防止阈值剧烈波动（阻尼）
        if (Math.abs(newQps - currentQps) > currentQps * 0.5) {
            newQps = currentQps * (1 + Math.signum(pidOutput - 1) * 0.5);
        }

        return new RateLimitResult(newQps, pidOutput);
    }
}
```

## 四、自适应降级：系统的最后一道防线

### 4.1 为什么需要自适应降级

自适应限流保护的是系统不被过载。但有时候，即使限流了，系统仍然可能崩溃——比如依赖的数据库响应变慢，导致大量请求堆积，最终 OOM。

自适应降级（Adaptive Degradation）是在系统濒临崩溃时的**最后一道防线**。它会自动触发降级策略，牺牲部分功能来保护系统核心可用。

```java title="自适应降级决策器"]
@Service
public class AdaptiveDegradationDecider {
    private final SystemMetricsCollector metricsCollector;
    private final DegradationStrategyRegistry strategyRegistry;

    /**
     * 评估是否需要降级
     */
    public DegradationDecision evaluate() {
        SystemMetrics metrics = metricsCollector.collect();
        DegradationDecision decision = new DegradationDecision();

        // 1. 检查 CPU
        if (metrics.getCpuUsage() > 95) {
            decision.setLevel(DegradationLevel.CRITICAL);
            decision.addReason("CPU 使用率 95%");
        }
        // 2. 检查内存
        else if (metrics.getMemoryUsage() > 90) {
            decision.setLevel(DegradationLevel.HIGH);
            decision.addReason("内存使用率 90%");
        }
        // 3. 检查 RT
        else if (metrics.getAvgRt() > 5000) {  // 5 秒
            decision.setLevel(DegradationLevel.MEDIUM);
            decision.addReason("平均响应时间 5 秒");
        }
        // 4. 检查错误率
        else if (metrics.getErrorRate() > 20) {
            decision.setLevel(DegradationLevel.MEDIUM);
            decision.addReason("错误率 20%");
        }
        // 5. 检查线程池饱和度
        else if (metrics.getThreadSaturation() > 0.9) {
            decision.setLevel(DegradationLevel.HIGH);
            decision.addReason("线程池饱和度 90%");
        }

        // 根据决策级别选择降级策略
        if (decision.getLevel() != null) {
            List<DegradationStrategy> strategies =
                    strategyRegistry.getStrategies(decision.getLevel());
            decision.setStrategies(strategies);
        }

        return decision;
    }

    /**
     * 执行降级
     */
    public void executeDegradation(DegradationDecision decision) {
        if (decision.getLevel() == null) {
            // 系统正常，恢复所有功能
            recovery();
            return;
        }

        for (DegradationStrategy strategy : decision.getStrategies()) {
            try {
                strategy.degrade();
                log.info("执行降级策略: {}", strategy.getName());
            } catch (Exception e) {
                log.error("降级策略执行失败: {}", strategy.getName(), e);
            }
        }
    }
}
```

### 4.2 降级策略分类

降级策略可以分为多个层次：

```java title="降级策略定义"]
/**
 * 降级策略接口
 */
public interface DegradationStrategy {
    String getName();
    DegradationLevel getLevel();
    void degrade();    // 执行降级
    void recover();    // 恢复
}

/**
 * 策略一：关闭非核心功能
 */
@Service
@Slf4j
public class NonCoreFeatureDegradation implements DegradationStrategy {
    @Override
    public String getName() {
        return "关闭非核心功能";
    }

    @Override
    public DegradationLevel getLevel() {
        return DegradationLevel.MEDIUM;
    }

    @Override
    public void degrade() {
        // 关闭推荐功能
        featureToggle.disable("recommendation");
        // 关闭评论功能
        featureToggle.disable("comment");
        // 关闭实时通知
        featureToggle.disable("realtime-notification");
    }

    @Override
    public void recover() {
        featureToggle.enable("recommendation");
        featureToggle.enable("comment");
        featureToggle.enable("realtime-notification");
    }
}

/**
 * 策略二：降低数据精度
 */
@Service
@Slf4j
public class DataPrecisionDegradation implements DegradationStrategy {
    @Override
    public String getName() {
        return "降低数据精度";
    }

    @Override
    public DegradationLevel getLevel() {
        return DegradationLevel.HIGH;
    }

    @Override
    public void degrade() {
        // 关闭实时计数，使用缓存计数
        config.set("useCachedCount", true);
        // 关闭详细日志
        config.set("logLevel", "WARN");
        // 降低数据刷新频率
        config.set("refreshInterval", 5000);  // 5 秒

    @Override
    public void recover() {
        config.set("useCachedCount", false);
        config.set("logLevel", "INFO");
        config.set("refreshInterval", 1000);
    }
}

        config.set("skipComputation", false);
    }
}
```

### 4.3 降级策略分级

| 级别 | CPU | 内存 | RT | 错误率 | 降级策略 |
|---|---|---|---|---|---|
| 正常 | < 70% | < 80% | < 500ms | < 5% | 无 |
| 轻度 | 70-85% | 80-90% | 500ms-2s | 5-10% | 关闭非核心功能 |
| 中度 | 85-95% | 90-95% | 2-5s | 10-20% | 降低数据精度 |
| 严重 | > 95% | > 95% | > 5s | > 20% | 返回兜底数据 |
| 危急 | - | - | 超时 | 服务崩溃 | 直接拒绝部分请求 |

## 五、Load Shedding：优雅地丢弃请求

### 5.1 为什么需要 Load Shedding

当系统已经无法处理所有请求时，限流只能控制新请求的进入，但已经进入系统的请求仍然会消耗资源。Load Shedding（负载丢弃）的思路是：**主动丢弃部分请求，以保证剩余请求能够正常处理**。

这听起来很反直觉，但它的哲学是：**与其让所有请求都超时，不如让部分请求直接失败，保证系统整体可用**。

### 5.2 Load Shedding 策略

```java title="Load Shedding 实现"]
@Service
@Slf4j
public class LoadSheddingService {
    private final SystemMetricsCollector metricsCollector;
    private final double criticalCpuThreshold = 0.95;
    private final double criticalRtThreshold = 3000;  // 3 秒

    /**
     * 评估是否需要丢弃请求
     */
    public LoadSheddingDecision shouldDrop() {
        SystemMetrics metrics = metricsCollector.collect();

        // 计算丢弃比例
        double dropRate = calculateDropRate(metrics);

        return new LoadSheddingDecision(dropRate, dropRate > 0);
    }

    /**
     * 计算丢弃比例
     */
    private double calculateDropRate(SystemMetrics metrics) {
        double cpu = metrics.getCpuUsage() / 100;
        double rt = metrics.getAvgRt() / criticalRtThreshold;
        double threadSaturation = metrics.getThreadSaturation();

        // 丢弃比例 = max(各指标超限程度)
        double maxOverload = Math.max(cpu / criticalCpuThreshold, rt);
        maxOverload = Math.max(maxOverload, threadSaturation);

        if (maxOverload <= 1.0) {
            return 0;  // 系统正常，不丢弃
        }

        // 丢弃比例：超载程度越高，丢弃越多
        // 例如：超载 1.2 倍，丢弃 20%；超载 2 倍，丢弃 50%
        double dropRate = Math.min(0.99, (maxOverload - 1.0) * 2);

        log.info("Load Shedding: 超载程度={}, 丢弃比例={}", maxOverload, dropRate);

        return dropRate;
    }

    /**
     * 尝试获取处理资格
     */
    public boolean tryAcquire() {
        LoadSheddingDecision decision = shouldDrop();

        if (!decision.shouldDrop()) {
            return true;
        }

        // 按丢弃比例，随机丢弃
        double random = ThreadLocalRandom.current().nextDouble();
        if (random >= decision.getDropRate()) {
            return true;
        }

        log.debug("请求被 Load Shedding 丢弃");
        return false;
    }
}

/**
 * 基于优先级的 Load Shedding
 */
@Service
@Slf4j
public class PriorityLoadSheddingService {
    private final Map<Integer, Double> priorityDropRates = new ConcurrentHashMap<>();

    /**
     * 计算不同优先级的丢弃比例
     */
    public double getDropRate(int priority) {
        // 优先级越高（数字越小），丢弃概率越低
        // priority 1: 0%, priority 5: 10%, priority 10: 50%
        double baseDropRate = (priority - 1) * 0.1;
        double currentOverload = calculateOverload();
        return Math.min(0.99, baseDropRate * currentOverload);
    }

    /**
     * 优先级队列调度
     */
    public boolean tryAcquire(int priority) {
        double dropRate = getDropRate(priority);
        double random = ThreadLocalRandom.current().nextDouble();
        return random >= dropRate;
    }

    /**
     * 处理请求，按优先级排序
     */
    public void processRequests(List<Request> requests) {
        // 按优先级排序
        requests.sort(Comparator.comparingInt(Request::getPriority));

        // 逐个处理
        for (Request request : requests) {
            if (tryAcquire(request.getPriority())) {
                process(request);
            } else {
                // 高优先级请求被丢弃，记录告警
                if (request.getPriority() <= 2) {
                    log.warn("高优先级请求被丢弃: {}", request.getId());
                    alertService.alert("HighPriorityRequestDropped");
                }
                request.reject();
            }
        }
    }
}
```

```java title="Sentinel 配置类"]
@Component
@Slf4j
public class AdaptiveFlowControlFilter implements Filter {
    private final SystemMetricsCollector metricsCollector;

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, WebFilterChain chain) {
        // 1. 获取当前限流阈值
        AdaptiveLimit limit = calculateAdaptiveLimit();

        // 2. 获取请求特征
        String path = exchange.getRequest().getPath().value();
        int priority = getRequestPriority(exchange);

        // 3. 检查是否需要限流
        if (shouldLimit(path, priority, limit)) {
            log.warn("请求被自适应限流: path={}, priority={}, limit={}",
                    path, priority, limit);
            return chain.filter(exchange);
        }

        // 4. 执行业务逻辑
        return chain.filter(exchange)
                .then(Mono.fromRunnable(() -> {
                    // 5. 记录响应时间和状态
                    recordResponse(exchange);
                }));
    }

    private AdaptiveLimit calculateAdaptiveLimit() {
        // 获取系统指标
        SystemMetrics metrics = metricsCollector.collect();

        // 计算自适应阈值
        AdaptiveLimit limit = new AdaptiveLimit();

        // 基于 CPU 计算
        double cpuUsage = metrics.getCpuUsage() / 100;
        if (cpuUsage > 0.9) {
            limit.setQps(100);  // CPU 过载，极限限流
        } else if (cpuUsage > 0.7) {
            limit.setQps(500 - (cpuUsage - 0.7) * 1000);  // 动态降低
        } else {
            limit.setQps(2000);  // CPU 正常
        }

        // 基于 RT 计算
        if (metrics.getAvgRt() > 2000) {
            limit.setQps(Math.min(limit.getQps(), 500));
        }

        return limit;
    }

    private boolean shouldLimit(String path, int priority, AdaptiveLimit limit) {
        // 高优先级请求不受限
        if (priority <= 2) {
            return false;
        }

        // 获取当前 QPS
        double currentQps = metricsCollector.getCurrentQps(path);

        return currentQps > limit.getQps();
    }
}
```

### 6.2 自定义注解实现

```java title="自适应限流注解"]
/**
 * 自适应限流注解
 */
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
@Documented
public @interface AdaptiveRateLimit {
    // 基础 QPS（系统正常时的阈值）
    double baseQps() default 1000;

    // 最小 QPS（系统压力时的保底阈值）
    double minQps() default 100;

    // CPU 告警阈值（超过此值开始收紧）
    double cpuWarnThreshold() default 0.7;

    // CPU 熔断阈值（超过此值使用 minQps）
    double cpuBreakerThreshold() default 0.9;

    // RT 告警阈值（毫秒）
    double rtWarnThreshold() default 1000;

    // 资源名称
    String resource() default "";

    // 请求优先级提取器（SpEL 表达式）
    String priority() default "0";
}

/**
 * 自适应限流切面
 */
@Aspect
@Component
@Slf4j
public class AdaptiveRateLimitAspect {
    private final SystemMetricsCollector metricsCollector;
    private final ConcurrentHashMap<String, AtomicInteger> qpsCounters = new ConcurrentHashMap<>();

    @Around("@annotation(adaptiveRateLimit)")
    public Object around(ProceedingJoinPoint joinPoint, AdaptiveRateLimit adaptiveRateLimit) throws Throwable {
        String resource = getResourceName(joinPoint, adaptiveRateLimit);

        // 1. 获取自适应限流阈值
        double threshold = calculateThreshold(adaptiveRateLimit);

        // 2. 获取当前 QPS
        double currentQps = getCurrentQps(resource);

        // 3. 检查是否限流
        if (currentQps >= threshold) {
            log.warn("自适应限流触发: resource={}, currentQps={}, threshold={}",
                    resource, currentQps, threshold);

            // 返回限流响应
            return handleLimitExceeded(joinPoint, adaptiveRateLimit);
        }

        // 4. 计数器 +1
        incrementCounter(resource);

        // 5. 执行目标方法
        return joinPoint.proceed();
    }

    private double calculateThreshold(AdaptiveRateLimit config) {
        SystemMetrics metrics = metricsCollector.collect();

        double cpuUsage = metrics.getCpuUsage() / 100;

        if (cpuUsage >= config.cpuBreakerThreshold()) {
            return config.minQps();
        }

        if (cpuUsage > config.cpuWarnThreshold()) {
            // 线性插值
            double range = config.baseQps() - config.minQps();
            double factor = (config.cpuBreakerThreshold() - cpuUsage) /
                    (config.cpuBreakerThreshold() - config.cpuWarnThreshold());
            return config.minQps() + range * factor;
        }

        return config.baseQps();
    }

    private double getCurrentQps(String resource) {
        AtomicInteger counter = qpsCounters.computeIfAbsent(resource, k -> new AtomicInteger(0));
        int count = counter.getAndSet(0);

        // 重置计数器（每秒执行一次）
        scheduleReset(resource, counter);

        return count;
    }

    private void incrementCounter(String resource) {
        qpsCounters.computeIfAbsent(resource, k -> new AtomicInteger(0)).incrementAndGet();
    }

    private Object handleLimitExceeded(ProceedingJoinPoint joinPoint, AdaptiveRateLimit config) throws Throwable {
        // 根据返回类型返回默认值
        MethodSignature signature = (MethodSignature) joinPoint.getSignature();
        Class<?> returnType = signature.getReturnType();

        if (returnType == void.class) {
            return null;
        }

        // 抛出限流异常
        throw new AdaptiveLimitExceededException(
                String.format("请求过于频繁，请稍后重试。当前阈值: %s", config.baseQps()));
    }
}
```

## 八、面试追问：自适应限流高频问题

### 问题一：静态限流和自适应限流如何选择？

**参考答案**：

| 场景 | 推荐方案 | 原因 |
|---|---|---|
| 系统容量稳定 | 静态限流 | 简单直接，配置方便 |
| 流量波动大 | 自适应限流 | 动态调整，避免资源浪费 |
| 下游依赖不稳定 | 自适应限流 | 可感知下游健康状态 |
| 业务优先级明确 | 优先级限流 | 保证核心业务 |
| 极端过载 | Load Shedding | 主动丢弃，保系统可用 |

两者也可以结合使用：**自适应限流作为第一层，静态限流作为兜底**。

### 问题二：自适应限流如何避免「抖动」？

**参考答案**：

自适应限流的常见问题是阈值抖动——系统负载在临界点附近波动，导致限流阈值忽高忽低。

解决方案包括：

**1. 引入「冷却期」**

```java
private long lastAdjustTime = 0;
private static final long COOLDOWN_MS = 10_000;  // 10 秒冷却期

public double adjustThreshold(double newThreshold) {
    if (System.currentTimeMillis() - lastAdjustTime < COOLDOWN_MS) {
        return currentThreshold;  // 冷却期内不调整
    }

    // 平滑过渡
    double adjusted = currentThreshold * 0.7 + newThreshold * 0.3;
    lastAdjustTime = System.currentTimeMillis();
    return adjusted;
}
```

**2. 使用「双阈值」**

设置「告警阈值」和「熔断阈值」，只有超过熔断阈值才触发限流收紧，避免在临界点频繁调整。

**3. 滑动窗口平滑**

使用较长的滑动窗口（如 1 分钟）计算指标，避免瞬时波动的影响。

### 问题三：自适应限流如何保证公平性？

**参考答案**：

自适应限流可能带来公平性问题——某些用户/请求可能被优先限流。

解决方案：

**1. 优先级队列**

按业务重要性给请求分级，高优先级请求享受更高的限流阈值或不受限流影响。

**2. 令牌桶隔离**

不同用户/服务使用独立的令牌桶，自适应调整针对整体，不影响单用户的最小可用性。

**3. 降级策略透明**

降级策略应该对用户透明，告知用户「当前系统繁忙，请稍后重试」，而不是静默丢弃。

### 问题四：如何测试自适应限流的效果？

**参考答案**：

**1. 压力测试**

通过压测工具（如 JMeter、wrk）模拟不同负载，观察限流阈值的变化。

**2. 混沌测试**

模拟系统异常（CPU 高、内存满、下游超时），验证自适应限流的响应。

**3. 灰度发布**

先在部分节点启用自适应限流，观察效果后再全量部署。

**4. 监控告警**

配置完善的监控指标，观察自适应限流触发次数、QPS 变化、用户投诉率等。

## 总结

自适应限流是应对复杂流量场景的利器，它的核心思想是「让系统自己感知自己」。

关键要点回顾：

- **指标采集**：CPU、内存、RT、错误率、队列深度等多维度指标
- **动态阈值计算**：基于实时指标动态调整限流阈值
- **滑动窗口**：使用滑动窗口平滑瞬时波动
- **PID 控制**：引入反馈控制理论，提高限流稳定性
- **自适应降级**：系统濒临崩溃时的最后防线
- **Load Shedding**：主动丢弃部分请求，保护系统整体可用
- **容器集成**：Kubernetes HPA 是云原生时代的自适应扩缩容方案

自适应限流不是银弹，它需要与静态限流、熔断降级、监控系统配合使用。只有建立了完整的防护体系，才能真正保障系统的高可用。
