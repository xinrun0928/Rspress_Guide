# 多级降级：页面降级、请求拦截与数据聚合

想象一下，你正在运营一个电商网站，双十一零点刚过，流量像潮水一样涌来。突然，商品推荐服务挂了。按照传统的降级思路，你可能只能选择：要么让用户看到报错页面，要么整个系统都降级。

但实际上，你可以做得更精细：商品详情页降级到静态版本，购物车服务继续正常运转，推荐列表换成昨日热卖，用户下单功能丝毫无损。这就是多级降级的魅力——不是「全有或全无」，而是「精准施策」。

多级降级是系统韧性的高级形态，它要求我们从全局视角审视系统，在不同层级设置不同的降级策略，确保核心功能始终可用，边缘功能按需牺牲。这篇文章，我会带你理解多级降级的架构设计，掌握各层级的降级实现，并给出具体的代码示例。

## 一、多级降级的架构设计：从页面到数据层

### 1.1 为什么需要多级降级

传统的降级往往是「一刀切」——要么全量降级，要么不降级。但实际上，系统的不同组件有不同的重要程度，不同功能有不同的业务价值。多级降级允许我们对系统进行「精准手术」。

考虑一个典型的电商系统：

| 层级 | 组件/服务 | 业务重要性 | 降级策略 |
|---|---|---|---|
| **页面层** | 首页、商品详情页 | 高 | 静态化、缓存 |
| **网关层** | API 网关 | 极高 | 限流、熔断 |
| **应用层** | 订单服务、用户服务 | 高 | 降级开关 |
| **数据层** | MySQL、Redis | 极高 | 主从切换、缓存 |
| **第三方服务** | 支付、短信、推送 | 中 | 异步、降级 |

如果我们采用多级降级，可以实现：
- 首页可降级，订单流程不能降级
- 商品推荐可降级，库存查询不能降级
- 用户评论可降级，用户登录不能降级
- 短信通知可降级，支付通知不能降级

### 1.2 多级降级的分层架构

```
┌─────────────────────────────────────────────────────────────┐
│                      CDN 层降级                              │
│  - 静态资源缓存     - 边缘计算降级     - 地理就近降级         │
├─────────────────────────────────────────────────────────────┤
│                      网关层降级                              │
│  - 请求过滤        - 参数降级        - 路由降级               │
├─────────────────────────────────────────────────────────────┤
│                      应用层降级                              │
│  - 功能降级        - 服务降级        - 数据降级              │
├─────────────────────────────────────────────────────────────┤
│                      数据层降级                              │
│  - 缓存降级        - 主从降级        - 分库降级              │
└─────────────────────────────────────────────────────────────┘
```

### 1.3 降级优先级的设计原则

设计多级降级时，需要遵循以下优先级原则：

**核心优先原则**：核心功能的降级优先级高于边缘功能。核心功能是「必须可用」的，边缘功能是「可以牺牲」的。

**影响范围原则**：降级时优先考虑影响范围小的方案。能降级单个功能，就不要降级整个服务。

**可逆原则**：降级应该是可逆的，能够快速恢复到正常状态。

**用户感知原则**：降级时应该给用户明确的提示，让用户知道系统在降级运行。

```java
public enum DegradationLevel {
    // 0级：正常
    NORMAL(0, "正常运行"),

    // 1级：边缘功能降级（影响最小）
    EDGE_FUNCTION_DEGRADED(1, "边缘功能降级"),

    // 2级：推荐服务降级
    RECOMMEND_DEGRADED(2, "推荐服务降级"),

    // 3级：部分服务降级
    PARTIAL_SERVICE_DEGRADED(3, "部分服务降级"),

    // 4级：大部分服务降级
    MOST_SERVICE_DEGRADED(4, "大部分服务降级"),

    // 5级：核心功能仅保障
    CORE_ONLY(5, "仅核心功能可用");

    private final int level;
    private final String description;

    DegradationLevel(int level, String description) {
        this.level = level;
        this.description = description;
    }

    public boolean canDegrade(DegradationLevel required) {
        return this.level >= required.level;
    }
}
```

## 二、CDN 层降级：最接近用户的降级

### 2.1 CDN 缓存降级

CDN（Content Delivery Network）是离用户最近的一层降级。通过 CDN，可以缓存静态资源和页面，减少对源站的压力。

```nginx
# nginx 配置示例：CDN 降级策略
server {
    # 正常情况：回源获取动态内容
    location / {
        proxy_pass http://origin_server;
        proxy_cache_valid 200 1m;  # 正常缓存 1 分钟
    }

    # 降级情况：返回缓存内容
    location /degraded {
        # 强制使用缓存，不回源
        proxy_cache_valid 200 60m;  # 降级缓存 60 分钟
        proxy_cache_use_stale error timeout updating;
        proxy_pass http://origin_server;
    }
}
```

### 2.2 边缘计算降级

现代 CDN 支持边缘计算（Edge Computing），可以在 CDN 节点上执行降级逻辑：

```javascript
// Cloudflare Workers 示例：边缘降级
addEventListener('fetch', event => {
    event.respondWith(handleRequest(event.request))
})

async function handleRequest(request) {
    const url = new URL(request.url)

    // 检查是否开启降级
    const degradationEnabled = await checkDegradationStatus()

    if (degradationEnabled) {
        // 降级模式：返回静态内容
        if (url.pathname.startsWith('/api/product/')) {
            return new Response(JSON.stringify({
                status: 'degraded',
                message: '服务暂时不可用，请稍后再试',
                data: await getCachedProduct(url.pathname)
            }), {
                headers: { 'Content-Type': 'application/json' }
            })
        }
    }

    // 正常模式：回源
    return fetch(request)
}

async function checkDegradationStatus() {
    const response = await fetch('https://config.example.com/degradation/status')
    const data = await response.json()
    return data.enabled
}
```

## 三、网关层降级：流量的守门人

### 3.1 Spring Cloud Gateway 降级

网关是流量的入口，在这里进行降级可以最早拦截请求，减轻后端压力。

```yaml
# application.yml
spring:
  cloud:
    gateway:
      routes:
        - id: product-service
          uri: lb://product-service
          predicates:
            - Path=/api/product/**
          filters:
            - name: RequestDegradation
              args:
                enabled: true
                degradedResponse: '{"status": 503, "message": "服务暂时不可用"}'
```

```java
@Component
public class DegradationGatewayFilterFactory
        extends AbstractGatewayFilterFactory<DegradationGatewayFilterFactory.Config> {

    private final DegradationConfig degradationConfig;
    private final ObjectMapper objectMapper;

    public DegradationGatewayFilterFactory(
            DegradationConfig degradationConfig,
            ObjectMapper objectMapper) {
        super(Config.class);
        this.degradationConfig = degradationConfig;
        this.objectMapper = objectMapper;
    }

    @Override
    public GatewayFilter apply(Config config) {
        return (exchange, chain) -> {
            String path = exchange.getRequest().getURI().getPath();

            // 检查是否需要降级
            if (config.isEnabled() && degradationConfig.shouldDegrade(path)) {
                try {
                    DegradedResponse response = degradationConfig.getDegradedResponse(path);
                    ServerHttpResponse httpResponse = exchange.getResponse();
                    httpResponse.setStatusCode(HttpStatus.SERVICE_UNAVAILABLE);
                    httpResponse.getHeaders().add("Content-Type", "application/json");

                    String body = objectMapper.writeValueAsString(response);
                    return httpResponse.writeWith(
                        Mono.just(httpResponse.bufferFactory().wrap(body.getBytes()))
                    );
                } catch (Exception e) {
                    log.error("降级过滤器执行失败", e);
                }
            }

            return chain.filter(exchange);
        };
    }

    @Data
    public static class Config {
        private boolean enabled = false;
        private String degradedResponse;
    }
}
```

### 3.2 请求参数降级

在网关层，可以对请求参数进行降级处理，简化请求复杂度：

```java
@Component
public class ParameterDegradationFilter {

    private final DegradationConfig degradationConfig;

    public ParameterDegradationFilter(DegradationConfig degradationConfig) {
        this.degradationConfig = degradationConfig;
    }

    public ServerHttpRequest.Builder degradeRequest(ServerHttpRequest original, String path) {
        ServerHttpRequest.Builder builder = ServerHttpRequest.builder(original);

        // 检查是否需要参数降级
        if (degradationConfig.shouldDegrade(path)) {
            Map<String, String> queryParams = original.getQueryParams().toSingleValueMap();

            // 简化查询参数
            if (queryParams.containsKey("includeDetails")) {
                // 详情降级：忽略 includeDetails 参数
                queryParams.remove("includeDetails");
            }

            if (queryParams.containsKey("relatedProducts")) {
                // 相关商品降级：忽略
                queryParams.remove("relatedProducts");
            }

            if (queryParams.containsKey("limit")) {
                // 限制数量降级：减少返回数量
                int limit = Integer.parseInt(queryParams.get("limit"));
                queryParams.put("limit", String.valueOf(Math.min(limit, 10)));
            }
        }

        return builder;
    }
}
```

## 四、应用层降级：服务级别的精细控制

### 4.1 功能开关降级

功能开关（Feature Flag）是应用层降级的核心手段。通过功能开关，可以动态开启或关闭某个功能，而不需要重新部署代码。

```java
@Configuration
public class FeatureDegradationConfig {

    private final AtomicBoolean recommendationEnabled = new AtomicBoolean(true);
    private final AtomicBoolean commentEnabled = new AtomicBoolean(true);
    private final AtomicBoolean relatedProductEnabled = new AtomicBoolean(true);

    @Value("${degradation.level:0}")
    private int degradationLevel;

    @PostConstruct
    public void init() {
        // 根据降级级别初始化开关状态
        updateFeaturesByLevel(degradationLevel);
    }

    public void updateFeaturesByLevel(int level) {
        recommendationEnabled.set(level < 1);
        commentEnabled.set(level < 2);
        relatedProductEnabled.set(level < 3);
    }

    public boolean isRecommendationEnabled() {
        return recommendationEnabled.get();
    }

    public boolean isCommentEnabled() {
        return commentEnabled.get();
    }

    public boolean isRelatedProductEnabled() {
        return relatedProductEnabled.get();
    }
}
```

### 4.3 注解式降级

使用自定义注解来实现更灵活的降级控制：

```java
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface Degradable {
    // 降级时的返回值类型
    Class<?> returnType();

    // 降级时返回的默认值
    String defaultValue() default "";

    // 降级时调用的方法名
    String fallbackMethod() default "";
}
```

```java
@Aspect
@Component
public class DegradationAspect {

    private final DegradationConfig degradationConfig;

    public DegradationAspect(DegradationConfig degradationConfig) {
        this.degradationConfig = degradationConfig;
    }

    @Around("@annotation(degradable)")
    public Object around(ProceedingJoinPoint joinPoint, Degradable degradable) throws Throwable {
        String methodName = joinPoint.getSignature().getName();

        // 检查是否需要降级
        if (!degradationConfig.shouldDegrade(methodName)) {
            return joinPoint.proceed();
        }

        // 执行降级逻辑
        log.warn("降级触发: {}", methodName);
        return executeFallback(degradable, joinPoint.getArgs());
    }

    private Object executeFallback(Degradable degradable, Object[] args) {
        Class<?> returnType = degradable.returnType();
        String defaultValue = degradable.defaultValue();

        // 根据返回值类型返回默认值
        if (returnType == String.class) {
            return defaultValue;
        }
        if (returnType == List.class) {
            return Collections.emptyList();
        }
        if (returnType == Map.class) {
            return Collections.emptyMap();
        }
        if (returnType == Boolean.class || returnType == boolean.class) {
            return false;
        }
        if (returnType == Integer.class || returnType == int.class) {
            return 0;
        }

        // 返回 null 或空对象
        return null;
    }
}
```

使用示例：

```java
@Service
public class ProductService {

    @Degradable(returnType = List.class, defaultValue = "[]")
    public List<Product> getRecommendations(Long productId) {

    @Degradable(returnType = List.class, defaultValue = "[]")

    }

}
```

## 五、数据层降级：缓存是最后的防线

### 5.1 多级缓存降级

数据层降级的核心是多级缓存。当主数据源不可用时，逐级降级到缓存：

```java
@Service
public class ProductDataService {

    private final ProductApi productApi;
    private final RedisTemplate<String, Object> redisTemplate;
    private final CaffeineCache localCache;

    public ProductDataService(
            ProductApi productApi,
            RedisTemplate<String, Object> redisTemplate,
            CaffeineCache localCache) {
        this.productApi = productApi;
        this.redisTemplate = redisTemplate;
        this.localCache = localCache;
    }

    public Product getProduct(Long productId) {
        // 第一级：本地缓存
        Product localCached = localCache.getIfPresent(productId);
        if (localCached != null) {
            return localCached;
        }

        // 第二级：Redis 缓存
        try {
            Product redisCached = getFromRedis(productId);
            if (redisCached != null) {
                localCache.put(productId, redisCached);
                return redisCached;
            }
        } catch (Exception e) {
            log.warn("Redis 获取失败: {}", e.getMessage());
        }

        // 第三级：降级数据（静态数据、默认数据）
        try {
            Product degraded = getDegradedProduct(productId);
            if (degraded != null) {
                // 异步写入缓存
                writeToRedisAsync(productId, degraded);
                return degraded;
            }
        } catch (Exception e) {
            log.error("降级数据获取也失败了: {}", e.getMessage());
        }

        // 兜底：抛出业务异常
        throw new ProductUnavailableException("商品暂时不可用");
    }

    private Product getFromRedis(Long productId) {
        String key = "product:" + productId;
        Object cached = redisTemplate.opsForValue().get(key);
        return convertToProduct(cached);
    }

    private Product getDegradedProduct(Long productId) {
        // 从静态配置中获取降级数据
        Map<Long, Product> degradedProducts = getStaticProductMap();
        Product degraded = degradedProducts.get(productId);
        if (degraded != null) {
            degraded.setDegraded(true);
            degraded.setMessage("数据来自降级缓存，可能不是最新");
        }
        return degraded;
    }

    @Async
    protected void writeToRedisAsync(Long productId, Product product) {
        try {
            String key = "product:" + productId;
            redisTemplate.opsForValue().set(key, product, Duration.ofHours(1));
        } catch (Exception e) {
            log.error("写入 Redis 失败: {}", e.getMessage());
        }
    }
}
```

### 5.2 聚合数据降级

当需要聚合多个数据源时，可以采用分批降级策略：

```java
@Service
public class ProductDetailAggregationService {

    private final ProductApi productApi;
    private final RecommendationApi recommendationApi;
    private final ReviewApi reviewApi;
    private final InventoryApi inventoryApi;
    private final DegradationConfig degradationConfig;

    public ProductDetailAggregation getProductDetail(Long productId) {
        ProductDetailAggregation aggregation = new ProductDetailAggregation();

        // 核心数据：必须获取
        Product product = productApi.getProduct(productId);
        aggregation.setProduct(product);

        // 库存数据：必须获取
        Inventory inventory = inventoryApi.getInventory(productId);
        aggregation.setInventory(inventory);

        // 聚合数据：可以降级
        if (degradationConfig.isRecommendationDegraded()) {
            // 推荐降级
            aggregation.setRecommendations(getDegradedRecommendations());
        } else {
            aggregation.setRecommendations(recommendationApi.getRecommendations(productId));
        }

        if (degradationConfig.isReviewDegraded()) {
            // 评价降级
            aggregation.setReviews(getDegradedReviews());
        } else {
            aggregation.setReviews(reviewApi.getReviews(productId));
        }

        // 设置降级标记
        if (degradationConfig.hasAnyDegradation()) {
            aggregation.setDegraded(true);
            aggregation.setDegradedFields(degradationConfig.getDegradedFields());
        }

        return aggregation;
    }

    private List<Product> getDegradedRecommendations() {
        // 返回静态推荐（昨日热卖）
        return hotSaleProductCache.getIfPresent("yesterday_top10");
    }

    private List<Review> getDegradedReviews() {
        // 返回精选评价（缓存数据）
        return featuredReviewCache.getIfPresent("featured");
    }
}
```

## 六、动态配置驱动的降级

### 6.1 配置中心集成

通过配置中心（如 Nacos、Apollo）动态配置降级策略：

```java
@Configuration
public class DegradationConfig {

    @Value("${degradation.enabled:false}")
    private boolean enabled;

    @Value("${degradation.level:0}")
    private int level;

    @Value("#{'${degradation.degradedServices:}'.split(',')}")
    private List<String> degradedServices;

    @Value("#{'${degradation.degradedPaths:}'.split(',')}")
    private List<String> degradedPaths;

    private final Map<String, DegradedResponse> responseCache = new ConcurrentHashMap<>();

    public boolean shouldDegrade(String path) {
        if (!enabled) {
            return false;
        }
        return degradedPaths.contains(path) || shouldDegradeByLevel(path);
    }

    private boolean shouldDegradeByLevel(String path) {
        if (path.contains("/recommendations") && level >= 1) {
            return true;
        }
        if (path.contains("/reviews") && level >= 2) {
            return true;
        }
        if (path.contains("/analytics") && level >= 3) {
            return true;
        }
        return false;
    }

    public DegradedResponse getDegradedResponse(String path) {
        return responseCache.computeIfAbsent(path, this::buildDegradedResponse);
    }

    private DegradedResponse buildDegradedResponse(String path) {
        DegradedResponse response = new DegradedResponse();
        response.setCode(503);
        response.setMessage("服务暂时不可用，请稍后再试");
        response.setDegraded(true);

        // 根据路径设置不同的降级数据
        if (path.contains("/recommendations")) {
            response.setData(getDefaultRecommendations());
        } else if (path.contains("/reviews")) {
            response.setData(Collections.emptyList());
        }

        return response;
    }
}
```

## 七、Spring Cloud 的多级降级实现

### 7.1 Sentinel 的多级降级

Sentinel 提供了多级别的降级能力：

```java
@Configuration
public class SentinelDegradationConfig {

    @PostConstruct
    public void init() {
        // 定义降级规则
        List<DegradeRule> rules = new ArrayList<>();

        // 规则1：慢调用比例降级
        DegradeRule slowRatioRule = new DegradeRule("productService")
            .setGrade(CircuitBreakerStrategy.SLOW_REQUEST_RATIO.getType())
            .setCount(0.5)  // 慢调用比例阈值
            .setSlowRatioThreshold(1000)  // 慢调用阈值（ms）
            .setMinRequestAmount(5)
            .setStatIntervalMs(10000)
            .setTimeWindow(10);

        // 规则2：异常比例降级
        DegradeRule errorRatioRule = new DegradeRule("recommendationService")
            .setGrade(CircuitBreakerStrategy.ERROR_RATIO.getType())
            .setCount(0.3)  // 异常比例阈值
            .setMinRequestAmount(5)
            .setStatIntervalMs(10000)
            .setTimeWindow(10);

        // 规则3：异常数降级
        DegradeRule errorCountRule = new DegradeRule("reviewService")
            .setGrade(CircuitBreakerStrategy.ERROR_COUNT.getType())
            .setCount(10)  // 异常数阈值
            .setMinRequestAmount(5)
            .setStatIntervalMs(60000)
            .setTimeWindow(30);

        rules.add(slowRatioRule);
        rules.add(errorRatioRule);
        rules.add(errorCountRule);

        DegradeRuleManager.loadRules(rules);
    }
}
```

### 7.2 Resilience4j 的多级降级

Resilience4j 提供了更细粒度的降级控制：

```java
@Service
public class ProductServiceWithResilience4j {

    private final CircuitBreaker recommendationCircuitBreaker;
    private final CircuitBreaker reviewCircuitBreaker;
    private final Retry recommendationRetry;
    private final RateLimiter recommendationRateLimiter;

    public ProductServiceWithResilience4j(
            CircuitBreakerRegistry circuitBreakerRegistry,
            RetryRegistry retryRegistry,
            RateLimiterRegistry rateLimiterRegistry) {
        this.recommendationCircuitBreaker = circuitBreakerRegistry.circuitBreaker("recommendation");
        this.reviewCircuitBreaker = circuitBreakerRegistry.circuitBreaker("review");
        this.recommendationRetry = retryRegistry.retry("recommendation");
        this.recommendationRateLimiter = rateLimiterRegistry.rateLimiter("recommendation");
    }

    public List<Product> getRecommendations(Long productId) {
        Supplier<List<Product>> supplier = () -> recommendationApi.getRecommendations(productId);

        // 包装重试
        supplier = Retry.decorateSupplier(recommendationRetry, supplier);

        // 包装熔断
        supplier = CircuitBreaker.decorateSupplier(recommendationCircuitBreaker, supplier);

        // 包装限流
        supplier = RateLimiter.decorateSupplier(recommendationRateLimiter, supplier);

        try {
            return supplier.get();
        } catch (CallNotPermittedException e) {
            // 熔断器打开，返回降级数据
            log.warn("推荐服务熔断打开，返回降级数据");
            return getDefaultRecommendations();
        } catch (Exception e) {
            log.error("推荐服务调用失败", e);
            return getDefaultRecommendations();
        }
    }

    private List<Product> getDefaultRecommendations() {
        // 返回默认推荐
        return hotSaleCache.get("top10");
    }
}
```

## 面试追问：多级降级相关高频问题

### 问题一：如何设计一个好的多级降级方案？

**参考答案**：

设计多级降级方案需要考虑以下几个方面：

**分层设计**：将降级策略按照层级组织，从 CDN 到网关、从网关到应用、从应用到数据，每一层都有对应的降级手段。

**优先级明确**：明确核心功能和边缘功能，确定降级时的优先级。核心功能是「必须可用」的，边缘功能是「可以牺牲」的。

**逐级降级**：采用逐级降级的策略，每一级的降级条件应该比上一级更宽松。比如：边缘功能先降级 → 推荐服务降级 → 详情页降级。

**可配置**：降级策略应该是可配置的，支持动态切换，不需要重新部署代码。

**可观测**：降级发生时，应该能够被监控系统捕获，并通知相关人员。

**可回滚**：降级应该能够快速回滚，恢复到正常状态。

### 问题二：CDN 层降级和应用层降级有什么区别？

**参考答案**：

| 维度 | CDN 层降级 | 应用层降级 |
|---|---|---|
| **位置** | 内容分发网络节点 | 应用服务器 |
| **响应速度** | 最快（边缘节点直接返回） | 较快（需要应用处理） |
| **降级粒度** | 粗（通常是整站或整页面） | 细（可以精确到接口、参数） |
| **数据新鲜度** | 低（缓存数据可能过期） | 高（可以返回最新数据） |
| **适用场景** | 静态资源、页面缓存 | 动态数据、个性化内容 |
| **成本** | 低（利用 CDN 节点） | 中（消耗应用资源） |

**最佳实践**：CDN 层降级作为第一道防线，主要处理大范围的流量拦截；应用层降级作为精细化处理，提供更精准的降级策略。

### 问题三：如何保证降级后的数据一致性？

**参考答案**：

降级后返回的数据可能不是实时的，存在数据滞后。为了保证数据尽量一致，可以采取以下措施：

**标记降级数据**：在返回的数据中标记「降级」状态，让前端知道这是降级数据，可以展示给用户提示。

**设置降级时间**：降级数据应该有明确的过期时间，超过时间后应该强制刷新或告警。

**异步补偿**：降级期间的数据变更应该在降级恢复后进行补偿同步。

**数据版本号**：在数据中加入版本号或时间戳，前端可以据此判断数据是否过期。

**用户提示**：对于重要数据的降级，应该在前端展示友好提示，告知用户数据可能不是最新的。

:::tip 延伸思考

多级降级是一个复杂的话题，涉及架构设计、配置管理、可观测性等多个方面。请思考：如果系统中有多个服务，每个服务都有自己的降级策略，如何统一管理和协调这些降级策略？是否存在「降级冲突」的问题？

:::

### 题目一：降级策略的优先级冲突

假设一个系统中有两个服务 A 和 B，A 依赖 B。在降级时：
- 如果 B 降级，A 的某些功能也应该降级
- 但 A 还有其他不依赖 B 的功能，这些功能不应该降级

**问题**：如何设计降级策略的优先级机制，避免降级冲突？

### 题目二：降级数据的质量保证

降级后，系统会返回缓存数据、历史数据或静态数据。但这些数据可能存在以下问题：
- 数据可能已经过期
- 数据可能与其他数据不一致
- 数据可能是错误的

**问题**：如何设计降级数据的质量保证机制，确保降级数据至少是「合理」的？

### 题目三：降级的自动化决策

目前很多降级决策还是人工触发的。但在某些场景下，需要系统能够自动判断是否需要降级。

**问题**：如何设计一个自动化的降级决策系统？需要收集哪些指标？如何设定自动降级的阈值？
