# 重试机制：指数退避、幂等性保证与重试风暴

你一定遇到过这种情况：点击下单按钮，页面没反应，再点一次，弹出「订单已提交」的提示。你心想，还好没再点第三次——否则可能就下了三单。

这就是重试机制的双刃剑效应。一方面，重试可以提高系统的容错能力，让那些「差一点就成功」的请求最终成功；另一方面，不当的重试策略可能导致重复操作、数据不一致，甚至引发「重试风暴」，把原本只是轻微过载的系统彻底压垮。

重试是分布式系统中最容易被忽视、但又最容易出问题的环节之一。这篇文章，我会从重试的策略讲起，带你理解指数退避的原理，掌握幂等性保证的方法，以及学会如何避免重试风暴。

## 一、重试的本质：给失败一个「再试一次」的机会

### 1.1 什么时候需要重试

不是所有的失败都需要重试。在设计重试策略之前，先要明确哪些情况适合重试：

| 适合重试 | 不适合重试 |
|---|---|
| 网络抖动导致的超时 | 业务逻辑错误（如余额不足） |
| 服务短暂不可用 | 参数校验失败 |
| 依赖服务正在重启 | 资源已耗尽（如库存为 0） |
| 随机性故障 | 幂等性无法保证的操作 |

```java
// 判断是否应该重试
public boolean shouldRetry(Exception e) {
    // 网络超时，可以重试
    if (e instanceof TimeoutException) {
        return true;
    }

    // 服务暂时不可用，可以重试
    if (e instanceof ServiceUnavailableException) {
        return true;
    }

    // 业务错误，不应该重试
    if (e instanceof BusinessException) {
        return false;
    }

    // 默认不重试
    return false;
}
```

### 1.2 重试的分类

根据重试的发生位置和触发条件，重试可以分为几种类型：

| 重试类型 | 触发时机 | 重试次数 | 典型场景 |
|---|---|---|---|
| **客户端重试** | 调用方发起 | 1-3 次 | RPC 调用超时 |
| **服务端重试** | 服务端处理 | 1-5 次 | 消息处理失败 |
| **消息队列重试** | MQ 框架自动 | 可配置 | 消费失败 |
| **网关重试** | 网关层代理 | 1-2 次 | 上游服务超时 |

:::tip 重试层级选择

重试应该在最靠近故障发生点的地方进行。如果 RPC 调用超时，应该由客户端重试；如果消息消费失败，应该由消费者重试。不要在多个层级同时重试，否则会导致大量重复请求。

:::

## 二、重试策略：从固定延迟到指数退避

### 2.1 固定延迟重试

固定延迟重试是最简单的策略：每次重试之间等待相同的时间。

```java
public class FixedDelayRetry {

    public <T> T execute(Callable<T> operation, int maxAttempts, long delayMs) throws Exception {
        Exception lastException = null;

        for (int attempt = 1; attempt <= maxAttempts; attempt++) {
            try {
                return operation.call();
            } catch (Exception e) {
                lastException = e;
                log.warn("Attempt {} failed: {}", attempt, e.getMessage());

                if (attempt < maxAttempts) {
                    Thread.sleep(delayMs);
                }
            }
        }

        throw lastException;
    }
}
```

**固定延迟的问题**：如果服务正在重启，固定延迟可能导致所有客户端同时重试，形成「惊群效应」。

### 2.2 指数退避重试

指数退避（Exponential Backoff）是指每次重试的等待时间呈指数增长：

```java
public class ExponentialBackoffRetry {

    private final long baseDelay;  // 基础延迟
    private final long maxDelay;   // 最大延迟
    private final double multiplier; // 倍数

    public ExponentialBackoffRetry(long baseDelay, long maxDelay, double multiplier) {
        this.baseDelay = baseDelay;
        this.maxDelay = maxDelay;
        this.multiplier = multiplier;
    }

    public <T> T execute(Callable<T> operation, int maxAttempts) throws Exception {
        Exception lastException = null;

        for (int attempt = 1; attempt <= maxAttempts; attempt++) {
            try {
                return operation.call();
            } catch (Exception e) {
                lastException = e;
                log.warn("Attempt {} failed: {}", attempt, e.getMessage());

                if (attempt < maxAttempts) {
                    long delay = calculateDelay(attempt);
                    Thread.sleep(delay);
                }
            }
        }

        throw lastException;
    }

    private long calculateDelay(int attempt) {
        // 指数退避：baseDelay × multiplier^(attempt-1)
        long delay = (long) (baseDelay * Math.pow(multiplier, attempt - 1));
        // 添加随机抖动
        delay += (long) (Math.random() * baseDelay * 0.1);
        // 不超过最大延迟
        return Math.min(delay, maxDelay);
    }
}
```

**指数退避的优势**：
- 避免惊群效应：随着等待时间增加，不同客户端的重试时间会逐渐分散
- 给故障服务恢复时间：服务重启需要时间，退避可以避免在服务还没恢复时就继续请求

### 2.3 抖动的类型

抖动（Jitter）是指数退避的重要组成部分，用于进一步分散重试请求。常见的抖动策略有三种：

| 抖动类型 | 计算公式 | 特点 |
|---|---|---|
| **Full Jitter** | `delay = random(0, base × 2^attempt)` | 随机性最大，效果最好 |
| **Equal Jitter** | `delay = base × 2^attempt / 2 + random(0, base × 2^attempt / 2)` | 平滑且有上界 |
| **Decorrelated Jitter** | `delay = min(maxDelay, random(base, delay × 3))` | 延迟时间相互独立 |

```java
public enum JitterType {
    NONE {
        long calculate(long baseDelay, long currentDelay, int attempt, long maxDelay) {
            return Math.min((long)(baseDelay * Math.pow(2, attempt - 1)), maxDelay);
        }
    },

    FULL {
        long calculate(long baseDelay, long currentDelay, int attempt, long maxDelay) {
            long maxValue = (long)(baseDelay * Math.pow(2, attempt - 1));
            return (long)(Math.random() * maxValue) + 1;
        }
    },

    EQUAL {
        long calculate(long baseDelay, long currentDelay, int attempt, long maxDelay) {
            long power = (long)(baseDelay * Math.pow(2, attempt - 1));
            return power / 2 + (long)(Math.random() * power / 2);
        }
    },

    DECORRELATED {
        long calculate(long baseDelay, long currentDelay, int attempt, long maxDelay) {
            long next = (long)(currentDelay * 3 * Math.random());
            return Math.max(baseDelay, Math.min(next, maxDelay));
        }
    };
}
```

:::tip 推荐使用 Equal Jitter

Full Jitter 在高并发场景下可能导致延迟过长，影响用户体验；Equal Jitter 在保证随机性的同时，有一个可预期的上界，更适合生产环境。

:::

### 2.4 完整的退避策略配置

```java
@Configuration
public class RetryConfig {

    @Value("${retry.base-delay:100}")
    private long baseDelay;

    @Value("${retry.max-delay:30000}")
    private long maxDelay;

    @Value("${retry.multiplier:2.0}")
    private double multiplier;

    @Value("${retry.max-attempts:3}")
    private int maxAttempts;

    @Value("${retry.jitter-type:FULL}")
    private String jitterType;

    @Bean
    public RetryTemplate retryTemplate() {
        // 配置重试策略
        Map<Class<? extends Throwable>, Boolean> retryableExceptions = new HashMap<>();
        retryableExceptions.put(TimeoutException.class, true);
        retryableExceptions.put(ServiceUnavailableException.class, true);
        retryableExceptions.put(IOException.class, true);
        retryableExceptions.put(TransientDataAccessException.class, true);

        // 非重试异常
        Map<Class<? extends Throwable>, Boolean> nonRetryableExceptions = new HashMap<>();
        nonRetryableExceptions.put(BusinessException.class, false);
        nonRetryableExceptions.put(ValidationException.class, false);
        nonRetryableExceptions.put(IllegalArgumentException.class, false);

        ExponentialBackOffPolicy backOffPolicy = new ExponentialBackOffPolicy();
        backOffPolicy.setInitialInterval(baseDelay);
        backOffPolicy.setMultiplier(multiplier);
        backOffPolicy.setMaxInterval(maxDelay);

        SimpleRetryPolicy retryPolicy = new SimpleRetryPolicy();
        retryPolicy.setMaxAttempts(maxAttempts);
        retryPolicy.setRetryableExceptions(retryableExceptions);
        retryPolicy.setNonRetryableExceptions(nonRetryableExceptions);

        RetryTemplate template = new RetryTemplate();
        template.setBackOffPolicy(backOffPolicy);
        template.setRetryPolicy(retryPolicy);

        return template;
    }
}
```

## 三、幂等性保证：重试的前提条件

### 3.1 为什么重试需要幂等性

假设你调用一个「扣款」接口，超时了。你不知道服务端到底有没有扣款成功，于是重试了一次。结果服务端扣了两次款——这就是「幂等性缺失」导致的问题。

**幂等性的定义**：一个操作执行一次和执行多次，结果是一样的。

| 操作类型 | 是否天然幂等 | 重试安全性 |
|---|---|---|
| 查询（GET） | 是 | 安全，可随意重试 |
| 删除（DELETE） | 是 | 安全，多次删除结果相同 |
| 新增（POST） | 否 | 危险，可能重复创建 |
| 更新（PUT） | 是（如果更新相同值） | 安全 |
| 扣款（Debit） | 否 | 危险，可能重复扣款 |
| 充值（Credit） | 否 | 危险，可能重复充值 |

### 3.2 幂等键的设计

幂等键（Idempotency Key）是保证幂等性的核心机制。一个好的幂等键应该具备以下特点：

- **唯一性**：全局唯一，不会重复
- **可追溯性**：能够从幂等键反推出业务含义
- **足够长**：避免碰撞

```java
public class IdempotencyKeyGenerator {

    /**
     * 生成订单支付的幂等键
     * 格式：pay:{userId}:{orderId}:{timestamp}:{nonce}
     */
    public String generateOrderPayKey(Long userId, Long orderId) {
        String timestamp = String.valueOf(System.currentTimeMillis());
        String nonce = generateNonce(8);
        return String.format("pay:%d:%d:%s:%s", userId, orderId, timestamp, nonce);
    }

    /**
     * 生成退款幂等键
     * 格式：refund:{userId}:{orderId}:{refundId}
     */
    public String generateRefundKey(Long userId, Long orderId, Long refundId) {
        return String.format("refund:%d:%d:%d", userId, orderId, refundId);
    }

    /**
     * 生成重试场景的幂等键
     * 格式：retry:{originalKey}:{retryCount}
     */
    public String generateRetryKey(String originalKey, int retryCount) {
        return String.format("retry:%s:%d", originalKey, retryCount);
    }

    private String generateNonce(int length) {
        String chars = "ABCDEFGHIJKLMNOPQRSTUVWXYZ0123456789";
        StringBuilder sb = new StringBuilder();
        Random random = new Random();
        for (int i = 0; i < length; i++) {
            sb.append(chars.charAt(random.nextInt(chars.length())));
        }
        return sb.toString();
    }
}
```

### 3.3 基于 Redis 的幂等实现

Redis 是实现幂等性控制的首选工具，因为它的 `SETNX` 操作是原子性的。

```java
@Service
public class RedisIdempotencyService {

    private final StringRedisTemplate redisTemplate;

    // 幂等键的默认过期时间
    private static final Duration DEFAULT_TTL = Duration.ofHours(24);

    public RedisIdempotencyService(StringRedisTemplate redisTemplate) {
        this.redisTemplate = redisTemplate;
    }

    /**
     * 检查并获取幂等锁
     * @param idempotencyKey 幂等键
     * @return true 表示首次请求，false 表示重复请求
     */
    public boolean tryLock(String idempotencyKey) {
        Boolean success = redisTemplate.opsForValue()
            .setIfAbsent(idempotencyKey, "PROCESSING", DEFAULT_TTL);
        return Boolean.TRUE.equals(success);
    }

    /**
     * 标记请求成功
     */
    public void markSuccess(String idempotencyKey, String result) {
        redisTemplate.opsForValue().set(idempotencyKey, "SUCCESS:" + result, DEFAULT_TTL);
    }

    /**
     * 标记请求失败
     */
    public void markFailed(String idempotencyKey) {
        redisTemplate.opsForValue().set(idempotencyKey, "FAILED", DEFAULT_TTL);
    }

    /**
     * 获取请求结果
     */
    public String getResult(String idempotencyKey) {
        return redisTemplate.opsForValue().get(idempotencyKey);
    }

    /**
     * 检查是否已完成（成功或失败）
     */
    public boolean isCompleted(String idempotencyKey) {
        String status = redisTemplate.opsForValue().get(idempotencyKey);
        return status != null && status.startsWith("SUCCESS");
    }

    /**
     * 带幂等的执行
     */
    public <T> T executeWithIdempotency(String idempotencyKey, Callable<T> operation) {
        // 1. 检查是否已处理
        String cachedResult = getResult(idempotencyKey);
        if (cachedResult != null && cachedResult.startsWith("SUCCESS:")) {
            log.info("幂等命中，直接返回缓存结果: {}", idempotencyKey);
            return parseResult(cachedResult);
        }

        // 2. 尝试获取锁
        if (!tryLock(idempotencyKey)) {
            log.info("其他请求正在处理: {}", idempotencyKey);
            // 等待并获取结果
            return waitForResult(idempotencyKey);
        }

        try {
            // 3. 执行操作
            T result = operation.call();
            // 4. 标记成功
            markSuccess(idempotencyKey, serializeResult(result));
            return result;
        } catch (Exception e) {
            // 5. 标记失败
            markFailed(idempotencyKey);
            throw e;
        }
    }

    private String serializeResult(Object result) {
        // 简单序列化，实际用 JSON
        return result.toString();
    }

    @SuppressWarnings("unchecked")
    private <T> T parseResult(String cachedResult) {
        String json = cachedResult.substring("SUCCESS:".length());
        // 简单反序列化，实际用 JSON
        return (T) json;
    }

    private <T> T waitForResult(String idempotencyKey) {
        int maxRetries = 100;
        int retryCount = 0;
        while (retryCount < maxRetries) {
            String result = getResult(idempotencyKey);
            if (result != null) {
                if (result.startsWith("SUCCESS:")) {
                    return parseResult(result);
                } else if (result.startsWith("FAILED")) {
                    throw new RuntimeException("幂等请求执行失败");
                }
            }
            try {
                Thread.sleep(100);
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
                throw new RuntimeException("等待被中断", e);
            }
            retryCount++;
        }
        throw new RuntimeException("等待幂等结果超时");
    }
}
```

### 3.4 基于数据库唯一约束的幂等实现

对于涉及数据库操作的重试，可以使用数据库的唯一约束来实现幂等：

```java
@Service
public class IdempotentPaymentService {

    @Autowired
    private PaymentRepository paymentRepository;

    @Transactional
    public PaymentResult processPayment(IdempotentPaymentRequest request) {
        try {
            // 尝试插入幂等记录
            IdempotentRecord record = new IdempotentRecord();
            record.setIdempotencyKey(request.getIdempotencyKey());
            record.setStatus("PROCESSING");
            record.setCreatedAt(new Date());

            idempotentRecordRepository.save(record);

            // 执行支付逻辑
            PaymentResult result = doPayment(request);

            // 更新幂等记录状态
            record.setStatus("SUCCESS");
            record.setResult(serialize(result));
            record.setCompletedAt(new Date());
            idempotentRecordRepository.save(record);

            return result;

        } catch (DataIntegrityViolationException e) {
            // 唯一约束冲突，说明已经处理过
            IdempotentRecord existing = idempotentRecordRepository
                .findByIdempotencyKey(request.getIdempotencyKey())
                .orElseThrow();

            if ("SUCCESS".equals(existing.getStatus())) {
                return deserialize(existing.getResult());
            } else if ("PROCESSING".equals(existing.getStatus())) {
                throw new PaymentProcessingException("支付正在处理中");
            } else {
                throw new PaymentProcessingException("支付处理失败，请重试");
            }
        }
    }
}
```

## 四、重试风暴：过度重试的灾难

### 4.1 什么是重试风暴

重试风暴（Retry Storm）是指大量重试请求同时涌向服务，导致服务过载甚至崩溃的现象。

典型的重试风暴场景：

1. 某个服务正常处理能力是 1000 QPS
2. 突然有 100 个请求超时失败
3. 这些请求每个重试 3 次，产生 300 个重试请求
4. 同时，还有新的正常请求涌入
5. 服务处理能力达到 1400 QPS，超过负载
6. 更多的请求超时，产生更多的重试
7. 恶性循环，最终服务崩溃

```
正常请求: 1000 QPS ─┐
                    ├→ 服务 (处理能力 1000 QPS) → 正常响应
重试请求: 300 QPS ─┘
                  ↓
              超载 1400 QPS
                  ↓
           更多超时失败
                  ↓
           更多重试请求
                  ↓
              崩溃
```

### 4.2 如何避免重试风暴

**方法一：限制重试比例**

```java
public class RetryWithLimit {

    private final AtomicInteger currentRetries = new AtomicInteger(0);
    private final int maxRetriesPerSecond;

    public RetryWithLimit(int maxRetriesPerSecond) {
        this.maxRetriesPerSecond = maxRetriesPerSecond;
    }

    public <T> T executeWithLimit(Callable<T> operation, Callable<T> fallback) {
        // 检查重试速率限制
        int current = currentRetries.incrementAndGet();
        if (current > maxRetriesPerSecond) {
            currentRetries.decrementAndGet();
            log.warn("重试速率超限，使用降级策略");
            return fallback.call();
        }

        try {
            return operation.call();
        } finally {
            currentRetries.decrementAndGet();
        }
    }
}
```

**方法二：熔断器保护**

```java
@Service
public class RetryWithCircuitBreaker {

    private final AtomicInteger failureCount = new AtomicInteger(0);
    private final AtomicInteger successCount = new AtomicInteger(0);
    private volatile State state = State.CLOSED;
    private volatile long lastFailureTime = 0;

    private static final int FAILURE_THRESHOLD = 5;
    private static final int SUCCESS_THRESHOLD = 3;
    private static final long RECOVERY_TIMEOUT_MS = 30000;

    public <T> T execute(Callable<T> operation, Callable<T> fallback) {
        switch (state) {
            case OPEN:
                // 熔断打开中，使用降级策略
                if (System.currentTimeMillis() - lastFailureTime > RECOVERY_TIMEOUT_MS) {
                    // 进入半开状态
                    state = State.HALF_OPEN;
                    break;
                }
                return fallback.call();

            case HALF_OPEN:
                try {
                    T result = operation.call();
                    int success = successCount.incrementAndGet();
                    if (success >= SUCCESS_THRESHOLD) {
                        // 连续成功，关闭熔断器
                        state = State.CLOSED;
                        failureCount.set(0);
                        successCount.set(0);
                    }
                    return result;
                } catch (Exception e) {
                    // 半开状态失败，重新打开熔断器
                    state = State.OPEN;
                    lastFailureTime = System.currentTimeMillis();
                    return fallback.call();
                }

            case CLOSED:
            default:
                try {
                    return operation.call();
                } catch (Exception e) {
                    handleFailure();
                    throw e;
                }
        }
    }

    private void handleFailure() {
        failureCount.incrementAndGet();
        lastFailureTime = System.currentTimeMillis();

        if (failureCount.get() >= FAILURE_THRESHOLD) {
            state = State.OPEN;
            log.warn("熔断器打开，连续失败次数: {}", failureCount.get());
        }
    }

    private enum State {
        CLOSED,   // 正常状态
        OPEN,     // 熔断打开，拒绝请求
        HALF_OPEN // 半开状态，允许部分请求通过
    }
}
```

**方法三：死信队列**

将重试多次仍然失败的消息放入死信队列，避免无限重试：

```java
@Service
public class RetryWithDeadLetter {

    private static final int MAX_RETRY_COUNT = 3;

    @Autowired
    private RabbitTemplate rabbitTemplate;

    @Autowired
    private MessageDeadLetterService deadLetterService;

    public void processMessage(Message message) {
        int retryCount = getRetryCount(message);

        try {
            doProcess(message);
        } catch (Exception e) {
            if (retryCount < MAX_RETRY_COUNT) {
                // 重新入队，下次再试
                message.getMessageProperties().setHeader("x-retry-count", retryCount + 1);
                rabbitTemplate.convertAndSend(
                    message.getMessageProperties().getReceivedExchange(),
                    message.getMessageProperties().getReceivedRoutingKey(),
                    message
                );
            } else {
                // 超过最大重试次数，放入死信队列
                deadLetterService.save(new DeadLetterRecord(message, e));
                log.error("消息处理失败，已放入死信队列: {}", message);
            }
        }
    }

    private int getRetryCount(Message message) {
        Object retryCount = message.getMessageProperties().getHeader("x-retry-count");
        return retryCount != null ? (int) retryCount : 0;
    }
}
```

```java
        backoff = @Backoff(delay = 100, multiplier = 2.0),
        fallback = "createOrderFallback"
    )
    public Order createOrderWithFallback(OrderRequest request) {
        return remoteOrderService.create(request);
    }

    public Order createOrderFallback(OrderRequest request, Exception e) {
        log.warn("订单创建失败，使用降级策略: {}", e.getMessage());
        return Order.createPending(request); // 返回待处理订单
    }

    // 排除某些异常不重试
    @Retryable(
        value = Exception.class,
        exclude = {BusinessException.class, ValidationException.class},
        maxAttempts = 2
    )
    public void processBusiness(BusinessRequest request) {
        // 处理逻辑
    }
}
```

### 5.2 Resilience4j Retry

```java
@Configuration
public class Resilience4jRetryConfig {

    @Bean
    public RetryRegistry retryRegistry() {
        RetryConfig defaultConfig = RetryConfig.custom()
            .maxAttempts(3)
            .waitDuration(Duration.ofMillis(500))
            .retryExceptions(Exception.class)
            .ignoreExceptions(
                BusinessException.class,
                ValidationException.class
            )
            .intervalFunction(IntervalFunction.ofExponentialBackoff(500, 2))
            .retryOnResult(response -> {
                // 根据返回结果判断是否需要重试
                if (response instanceof ApiResponse) {
                    return ((ApiResponse<?>) response).getCode() == 500;
                }
                return false;
            })
            .build();

        return RetryRegistry.of(defaultConfig);
    }
}

@Service
public class ProductService {

    private final Retry retry;

    public ProductService(RetryRegistry retryRegistry) {
        this.retry = retryRegistry.retry("productService");
    }

    public Product getProduct(Long id) {
        Supplier<Product> supplier = Retry.decorateSupplier(retry, () -> {
            return productApi.getProduct(id);
        });
        return supplier.get();
    }

    public Product getProductWithFallback(Long id) {
        Supplier<Product> supplier = Retry.decorateSupplier(retry, () -> {
            return productApi.getProduct(id);
        });

        Supplier<Product> fallback = () -> {
            log.warn("获取商品失败，返回默认商品: {}", id);
            return Product.defaultProduct(id);
        };

        return Retry.decorateSupplier(retry, supplier, fallback).get();
    }
}
```

## 面试追问：重试相关高频问题

### 问题一：如何设计一个好的重试策略？

**参考答案**：

一个好的重试策略需要考虑以下几个方面：

**确定重试条件**：不是所有失败都需要重试。应该只对临时性故障（如网络超时、服务暂时不可用）进行重试，而对业务错误（如参数校验失败、余额不足）不重试。

**选择重试间隔**：
- **固定间隔**：简单但可能导致惊群效应
- **指数退避**：间隔时间指数增长，给故障服务恢复时间
- **加随机抖动**：避免多个客户端同时重试

**设置重试次数**：重试次数不宜过多，建议 2-5 次。过多的重试可能导致重试风暴。

**设置总超时**：除了单次重试的超时，还要设置整个重试流程的总超时，避免无限重试。

**配合熔断和降级**：重试应该与熔断器、降级策略配合使用。当失败率达到阈值时，停止重试，启用降级。

### 问题二：重试会导致哪些问题？

**参考答案**：

重试虽然能提高系统的容错能力，但也会带来一些问题：

**重复操作**：如果服务端已经处理了请求，但响应丢失了，客户端重试会导致重复操作。这就是为什么重试需要幂等性保证。

**资源浪费**：大量重试请求会占用系统资源，可能导致重试风暴。

**延迟增加**：重试会增加请求的总延迟，用户需要等待更长时间。

**复杂性增加**：重试逻辑需要考虑超时、幂等性、错误处理等多种情况，代码复杂度增加。

**数据不一致**：如果重试的时机不对，可能导致数据不一致。比如扣款成功了，但响应丢失了，客户端重试又扣了一次款。

### 问题三：如何保证接口的幂等性？

**参考答案**：

保证接口幂等性主要有以下几种方案：

**唯一幂等键**：客户端生成一个全局唯一的幂等键，服务端根据幂等键判断是否是重复请求。幂等键可以通过 Redis SETNX 或数据库唯一约束来存储和校验。

**状态机控制**：对于有明确状态流转的操作（如订单状态），通过状态机来保证幂等性。比如，只有「待支付」状态的订单才能转为「已支付」。

**乐观锁/版本号**：在更新操作时检查版本号或乐观锁，只有版本号匹配时才更新。

**去重表**：创建一个去重表，记录已处理过的请求 ID，通过数据库唯一约束来防止重复。

**接口设计**：尽量设计幂等性的接口，比如用 PUT 替代 POST，用 DELETE 替代多次状态更新。

:::tip 延伸思考

重试、幂等性、熔断是分布式系统容错的三板斧。但这些机制本身也可能出问题——比如重试风暴、幂等键过期、熔断器误判。请思考：如果这些容错机制本身发生故障，应该如何监控系统中的容错机制是否正常工作？

:::
