# 数据库高可用：MySQL MHA、Galera Cluster 与故障切换

想象一下这个场景：你的系统正在处理双十一的秒杀请求，订单数据像潮水一样涌入 MySQL。突然，告警平台弹出红色警告——「主库连接失败」「主库不可达」「数据库写入异常」。紧接着，所有依赖数据库写入的操作全部失败，用户下单、支付、更新库存……一切与写相关的操作都停止了。你意识到，一场灾难正在发生。

数据库是企业最重要的核心资产——它存储着所有的业务数据、用户信息、交易记录。一旦数据库不可用，整个业务系统就会陷入瘫痪。与缓存层、网关层不同，数据库的高可用难度更高，因为它面临的是**持久化数据的一致性问题**。缓存挂了可以回源，数据库挂了，数据可能永久丢失。

这篇文章，我带你深入理解数据库高可用的核心挑战，剖析 MySQL MHA、Galera Cluster 等主流方案的工作原理，让你不仅知道「怎么用」，更理解「为什么这样设计」，以及「不同方案的取舍」。

---

## 一、为什么数据库高可用是「最难的问题」

### 1.1 数据库高可用的独特挑战

数据库高可用之所以是分布式系统中最难的问题，源于以下几个核心矛盾：

**一致性与可用性的矛盾**

这是 CAP 定理的核心困境。在分布式数据库中，网络分区不可避免，但分区的两侧只能二选一：要么保证一致性（牺牲可用性），要么保证可用性（牺牲一致性）。对于金融级业务，一致性是生命线；对于互联网业务，可用性意味着用户体验和收入。

**数据持久化的代价**

缓存可以容忍数据丢失（反正可以重建），但数据库不行。用户下的订单、充值的余额、注册的账号——这些数据一旦丢失，后果不可承受。因此，数据库的复制必须是可靠的，复制延迟必须可接受。

**事务的边界**

MySQL 的 InnoDB 引擎支持 ACID 事务，事务的提交需要在主从之间保持一致。一旦引入复制，事务的提交时机就变得复杂——是主库提交成功就返回，还是要等待从库确认？

**跨机房、跨地域的复杂性**

单一机房内的数据库故障相对容易处理，但跨机房、跨地域的容灾就复杂得多。网络延迟、带宽限制、脑裂风险——每一个都是棘手的问题。

:::tip 数据库高可用的终极目标

数据库高可用的终极目标不是「永远不宕机」，而是「在故障发生时，最大程度减少数据丢失，最快速度恢复服务」。没有任何方案能做到零丢失、零停机，关键是在一致性、性能、成本之间找到平衡点。

:::

### 1.2 故障类型与影响

数据库故障可以分为几个层次：

| 故障类型 | 影响范围 | 恢复难度 | 数据风险 |
|---|---|---|---|
| 从库故障 | 只读不可用，写不受影响 | 低 | 无 |
| 主库软故障 | 写入不可用 | 中 | 可能有少量数据丢失 |
| 主库硬故障 | 写入不可用 | 高 | 可能有多量数据丢失 |
| 机房级故障 | 全量不可用 | 极高 | 取决于备份策略 |
| 数据损坏 | 部分数据不可用 | 极高 | 部分数据丢失 |

## 二、MySQL 主从复制：异步与半同步

### 2.1 异步复制：性能优先

MySQL 默认的复制模式是**异步复制**。主库执行完事务后，立即返回结果给客户端，不等待从库确认。复制过程在后台异步进行。

**工作流程**：

1. 主库事务提交，执行 `COMMIT`
2. 主库将事务写入 binlog
3. 主库返回「提交成功」给客户端
4. 从库的 IO 线程连接主库，请求 binlog
5. 主库的 binlog dump 线程发送 binlog 给从库
6. 从库的 IO 线程接收 binlog，写入 relay log
7. 从库的 SQL 线程读取 relay log，执行事务

```bash
# 主库配置
server-id = 1
log-bin = mysql-bin
binlog-format = ROW        # 推荐使用 ROW 格式
sync-binlog = 1            # 每事务同步 binlog 到磁盘

# 从库配置
server-id = 2
relay-log = relay-bin
read-only = 1
```

**数据丢失风险**：如果主库在事务提交后、binlog 发送给从库前宕机，这个事务的数据就会丢失。从库重启后，发现主库已经切换，会请求新的主库同步数据——但丢失的事务永远找不回来了。

:::warning 异步复制的代价

异步复制性能最好，但数据安全性最低。在主库故障时，可能丢失最后一批事务。对于数据安全性要求高的场景（如金融、订单），不建议使用纯异步复制。

:::

### 2.2 半同步复制：安全与性能的折中

MySQL 5.5 引入了**半同步复制**（Semi-sync Replication），试图在性能和安全之间找到平衡。

**工作原理**：主库执行事务提交后，等待**至少一个从库**确认接收到了 binlog，然后才返回成功。如果在 `rpl_semi_sync_master_timeout` 时间内没有收到确认，主库会切换到异步模式继续工作。

```bash
# 安装半同步插件
INSTALL PLUGIN rpl_semi_sync_master SONAME 'semisync_master.so';
INSTALL PLUGIN rpl_semi_sync_slave SONAME 'semisync_slave.so';

# 启用半同步
SET GLOBAL rpl_semi_sync_master_enabled = 1;
SET GLOBAL rpl_semi_sync_slave_enabled = 1;
```

**无损半同步**：MySQL 5.7 之后，引入 `rpl_semi_sync_master_wait_point = AFTER_SYNC`（默认），确保事务在从库写入 relay log 后才返回成功。相比 `AFTER_COMMIT`，数据安全性更高。

```bash
# 推荐配置
rpl_semi_sync_master_wait_point = AFTER_SYNC
rpl_semi_sync_master_timeout = 1000    # 1000 毫秒超时
```

**局限**：半同步复制仍然不能完全避免数据丢失。如果从库在接收 binlog 后、执行完成前宕机，主库切到新从库时，该事务可能丢失（虽然主库上有，但已经提交了）。

## 三、MHA：MySQL 高可用的事实标准

### 3.1 MHA 架构

MHA（MySQL Master High Availability）是日本DeNA 公司的工程师开发的一款 MySQL 高可用工具，目前在互联网公司中广泛应用。MHA 的设计理念是：**自动化、零数据丢失、快速故障转移**。

**核心组件**：

- **MHA Manager**：管理节点，负责监控主库状态、触发故障转移。通常单独部署一台机器，作为「指挥官」。
- **MHA Node**：运行在每台 MySQL 服务器上，负责主从切换的具体执行。

```bash
# MHA Manager 配置
[server default]
manager_workdir=/var/log/masterha/app1
manager_log=/var/log/masterha/app1/manager.log
remote_workdir=/var/log/masterha/app1
ssh_user=root
ssh_options=-i /root/.ssh/id_rsa
repl_user=repl_user
repl_password=password

[server1]
hostname=192.168.1.100
candidate_master=1
master_binlog_dir=/var/lib/mysql

[server2]
hostname=192.168.1.101
candidate_master=1
master_binlog_dir=/var/lib/mysql

[server3]
hostname=192.168.1.102
no_master=1
```

### 3.2 MHA 如何选择新主库

当主库故障后，MHA 需要从众多从库中选择一个作为新的主库。选择的依据是什么？

**选择标准（按优先级排序）**：

1. **拥有最新的 relay log**：通过 `SHOW SLAVE STATUS` 查看 `Exec_Master_Log_Pos`，值越大，说明数据越新
2. **最近的 master log position**：binlog 位置越靠后，说明与原主库数据差异越小
3. **配置优先级**：`candidate_master=1` 的从库优先
4. **硬件配置**：CPU、内存更好的节点优先
5. **复制延迟**：`SQL_Delay` 小的节点优先

### 3.3 MHA 故障转移流程

MHA 的故障转移分为三个阶段：

**阶段一：VIP 漂移**

通过 SSH 连接到候选主库，执行 `master_ip_failover_script`，将 VIP 从旧主库漂移到新主库。这个过程通常在毫秒级完成。

```bash
# VIP 漂移脚本逻辑
if (is_master()) {
    # 执行 IP 漂移
    system("/sbin/ip addr add $vip/24 dev eth0");
    system("/sbin/arping -q -c 3 -A $vip");
}
```

**阶段二：数据补偿**

如果配置了 `secondary_check_script`，MHA 会尝试通过中转从库（relay master）补齐数据。如果原主库可以 SSH 连接，会尝试获取最后一部分 binlog。

**阶段三：从库重新指向新主库**

让所有从库执行 `CHANGE MASTER TO`，指向新的主库。从库根据 `Exec_Master_Log_Pos` 确定从哪个位置开始复制。

```bash
# MHA Manager 启动
nohup masterha_manager --conf=/etc/app1.cnf &

# 查看状态
masterha_check_status --conf=/etc/app1.cnf
```

### 3.4 MHA 的局限性

MHA 解决了主从复制的自动故障转移问题，但它也有局限性：

- **不支持自动添加从库**：新从库需要手动添加
- **不支持多主复制**：只能有一主多从
- **依赖 SSH**：需要 SSH 到所有 MySQL 节点
- **不支持 GTID**：早期版本不支持，MySQL 5.6+ 的 GTID 需要额外配置

## 四、Galera Cluster：真正的高可用方案

### 4.1 为什么需要同步复制

异步复制的问题在于：主库提交后立即返回，从库的数据可能滞后。当主库故障时，滞后的数据就丢失了。

**同步复制**解决了这个问题：主库提交事务前，必须确保所有节点都完成写入。这样，任何一个节点故障，数据都不会丢失——因为数据在其他节点上已经落盘了。

Galera Cluster 就是 MySQL 的同步复制解决方案，它基于 **Certification-Based Replication（基于认证的复制）**。

### 4.2 Galera 的工作原理

Galera Cluster 不是 MySQL 原生的复制机制，而是一层「复制插件」。它让 MySQL 在提交事务前，先进行集群内的同步。

**写入流程**：

1. **本地执行**：事务在本地节点执行，产生 write set（包含修改的行和主键）
2. **认证（Certification）**：write set 被发送到集群所有节点，各节点验证：如果这个事务与已提交的事务没有冲突（修改了同一行），则认证通过
3. **广播提交**：认证通过后，事务被广播，所有节点同时提交
4. **返回客户端**：本地提交完成，返回「成功」

```bash
# Galera Cluster 配置
wsrep_provider = /usr/lib/galera/libgalera_smm.so
wsrep_cluster_name = "my_cluster"
wsrep_cluster_address = "gcomm://192.168.1.100,192.168.1.101,192.168.1.102"
wsrep_node_name = "node1"
wsrep_node_address = "192.168.1.100"

# 同步模式
wsrep_sync_wait = 1           # 主键冲突时等待同步
wsrep_sst_method = rsync       # 状态快照传输方式
```

### 4.3 IST 与 SST

Galera Cluster 在节点加入集群时，需要同步数据。根据节点当前状态，有两种同步方式：

**SST（State Snapshot Transfer，全量同步）**：新节点加入时，从 Donor 节点（数据提供节点）拉取完整的数据库快照。常用的工具有 `rsync`、`mysqldump`、`xtrabackup`。

**IST（Incremental State Transfer，增量同步）**：如果新节点之前是集群成员，或者 Galera 的 grastate.dat 显示有缓存的写入集（writeset），可以从 Donor 节点的某个位置增量同步。IST 比 SST 快得多，因为只需要同步少量的增量数据。

```bash
# 查看 grastate（Galera 状态）
cat /var/lib/mysql/grastate.dat
# GALERA STATE:
# version: 2.1
# seqno: 156789
# cart_uuid: xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx
# safe_to_bootstrap: 0
```

:::tip Donor 节点选择

SST 期间，Donor 节点无法正常服务（会被锁表），因此应该选择业务低峰期进行，或者选择相对空闲的节点作为 Donor。如果集群压力大，尽量避免触发 SST——这也是为什么建议提前准备好「增量备份」，让节点可以执行 IST。

:::

### 4.4 Galera 的局限性

Galera 解决了同步复制的问题，但它也有自己的局限：

- **写冲突**：如果两个节点同时写入同一行，后写入的事务会回滚
- **DDL 问题**：大表的 ALTER TABLE 会长时间锁定，不适合在 Galera 集群中执行
- **性能开销**：认证过程有额外开销，高并发写入场景性能不如异步复制
- **节点数量**：节点数量影响 quorum 计算，过少的节点可能产生脑裂

## 五、云原生数据库：RDS 与 PolarDB

### 5.1 RDS 的高可用架构

阿里云 RDS（Relational Database Service）是典型的云原生数据库服务。它的主备高可用架构基于「计算与存储分离」的设计：

**架构特点**：

- **一主多备**：主库和备库共享同一份存储（ESSD），数据实时同步
- **快速切换**：主库故障时，备库通过共享存储直接接管，无需数据复制
- **RPO = 0**：因为共享同一份存储，理论上数据零丢失

```bash
# RDS 主备切换
# 通常由云平台自动完成，应用层通过 VIP 访问
# 切换时间通常在 30 秒以内
```

### 5.2 PolarDB：存算分离的革新

PolarDB 是阿里自研的新一代云原生数据库，最大的特点是**存储计算分离**：

**核心架构**：

- **计算层**：多个计算节点（可读写的主节点 + 只读节点）
- **存储层**：分布式共享存储（基于 RDMA 网络）
- **日志流**：Redo Log 通过日志链路实时复制

```bash
# PolarDB 连接
# 通过集群地址访问，自动读写分离
mysql -h polardb-cluster.xxxxx.rds.aliyuncs.com -P 3306 -u user -p
```

**优势**：

- **秒级扩容**：新增只读节点无需复制数据，直接挂载共享存储
- **存储按量计费**：不用为峰值容量付费
- **数据共享**：主节点和只读节点访问同一份数据，一致性有保障

## 六、故障切换：应用无感知

### 6.1 VIP 漂移

故障切换的核心是**让应用层无感知**。最常用的方案是 VIP（虚拟 IP）漂移：应用连接的是 VIP，而不是真实的数据库 IP。当主库切换时，VIP 会「漂移」到新主库，应用层的连接无需修改。

```bash
# Keepalived + MySQL VIP 漂移
vrrp_instance VI_MYSQL {
    state BACKUP
    interface eth0
    virtual_router_id 51
    priority 100
    nopreempt               # 非抢占模式
    
    virtual_ipaddress {
        192.168.1.200/24    # VIP 地址
    }
    
    track_script {
        chk_mysql           # 检查 MySQL 状态的脚本
    }
}
```

### 6.2 连接池重连

VIP 漂移可以解决网络层面的问题，但应用层的数据库连接池需要支持自动重连。

```java
// Druid 连接池配置
DruidDataSource dataSource = new DruidDataSource();
dataSource.setDriverClassName("com.mysql.cj.jdbc.Driver");
dataSource.setUrl("jdbc:mysql://192.168.1.200:3306/db?useSSL=false");
dataSource.setUsername("user");
dataSource.setPassword("password");

// 重连配置
dataSource.setConnectionErrorRetryAttempts(3);      // 重试次数
dataSource.setBreakAfterAcquireFailure(true);        // 获取失败后是否断开
dataSource.setTimeBetweenConnectErrorMillis(3000);  // 重试间隔
```

:::tip 连接池的「预建连接」问题

HikariCP 等高性能连接池会在启动时预建连接。当数据库切换时，预建的连接已经失效，连接池需要重建连接。如果重建过程没有处理好，可能出现短暂的连接失败。

建议在连接池配置中加入「验证查询」，每次获取连接时检查可用性。

:::

### 6.3 优雅关闭

在故障切换前，需要先停止对旧主库的写入，等待复制完成，再进行切换。这个过程叫做「优雅关闭」。

**优雅关闭步骤**：

1. 关闭VIP，避免新流量写入
2. 等待复制延迟归零（`SHOW SLAVE STATUS` 的 `Seconds_Behind_Master` 为 0）
3. 设置主库为只读（`SET GLOBAL read_only = 1`）
4. 确认所有长事务完成
5. 执行主从切换

```bash
# 优雅关闭脚本逻辑
# 1. 设置主库只读
mysql -e "SET GLOBAL read_only = 1;"

# 2. 等待从库追上
while [ $(mysql -e "SHOW SLAVE STATUS\G" | grep Seconds_Behind_Master) -ne 0 ]; do
    sleep 1;
done

# 3. 执行切换
mysql -e "STOP SLAVE; RESET SLAVE ALL;"
```

## 七、主流方案对比

| 方案 | 一致性 | 性能 | 复杂度 | 数据丢失 | 适用场景 |
|---|---|---|---|---|---|
| **异步复制** | 弱一致 | 最高 | 低 | 可能丢数据 | 不重要数据 |
| **半同步复制** | 弱一致 | 中 | 中 | 少量丢失 | 一般业务 |
| **MHA** | 弱一致 | 高 | 中 | 少量丢失 | 互联网业务 |
| **Galera** | 强一致 | 中低 | 高 | 零丢失 | 核心业务 |
| **RDS 主备** | 强一致 | 高 | 低 | 零丢失 | 云服务 |
| **PolarDB** | 强一致 | 高 | 低 | 零丢失 | 云原生业务 |

:::tip 选型建议

- **小公司、单机房**：MHA + 异步/半同步复制，成熟稳定，运维简单
- **中等规模、多机房**：Galera Cluster 或云数据库，自动切换，数据安全
- **大规模、全球化**：云数据库（PolarDB/RDS），免运维，弹性伸缩

:::

## 八、读流量高可用：读写分离

### 8.1 读写分离的架构

对于读多写少的业务，读写分离是提升性能和可用性的有效手段。写操作走主库，读操作走从库。

```bash
# HAProxy 配置读写分离
listen mysql_rw
    bind 0.0.0.0:3306
    mode tcp
    balance leastconn
    
    # 主库（写）
    server master 192.168.1.100:3306 check weight 1
    
    # 从库（读）
    server slave1 192.168.1.101:3306 check weight 2
    server slave2 192.168.1.102:3306 check weight 2
```

### 8.2 延迟问题

读写分离后，从库的数据可能滞后于主库。对于需要强一致性的读取（如读取刚写入的数据），必须读主库。

```java
// 强制读主库注解
@Target({ElementType.METHOD, ElementType.TYPE})
@Retention(RetentionPolicy.RUNTIME)
public @interface Master {
}

// 路由策略
public class DataSourceRouter {
    
    @Around("@annotation(master)")
    public Object routeToMaster(ProceedingJoinPoint point, Master master) throws Throwable {
        DynamicDataSourceHolder.setMaster();
        try {
            return point.proceed();
        } finally {
            DynamicDataSourceHolder.clear();
        }
    }
}
```

## 九、面试追问

### 追问一：主从复制延迟的原因和解决方案？

**参考答案**：

主从复制延迟的原因主要来自三个方面：

**网络延迟**：主从之间的网络带宽不足或延迟过高。解决方案：优化网络拓扑，确保主从之间网络稳定、低延迟。

**从库执行速度慢**：从库的 SQL 线程读取 relay log 后，需要按照顺序执行事务。如果从库负载高，或者大事务执行时间长，就会产生延迟。解决方案：使用并行复制（`slave_parallel_workers`）、优化大事务。

**主库写入速度过快**：主库写入速度远超从库消费速度，积压越来越多。解决方案：控制主库写入速度，或增加从库数量分散读流量。

```bash
# 查看复制延迟
SHOW SLAVE STATUS\G
# 关注 Seconds_Behind_Master 字段

# 启用并行复制
STOP SLAVE;
SET GLOBAL slave_parallel_workers = 8;
START SLAVE;
```

### 追问二：如何防止脑裂？

**参考答案**：

脑裂（Split-Brain）是指集群被分割成两部分，两边都认为自己是主库，都可以写入。防止脑裂的核心是** quorum 机制**。

**quorum 计算**：quorum = floor(节点数 / 2) + 1。超过 quorum 的那部分集群才能继续工作。

**Galera 的处理**：

- 3 节点集群：quorum = 2。最多允许 1 个节点故障。
- 5 节点集群：quorum = 3。最多允许 2 个节点故障。

```bash
# 强制单节点启动（慎用！）
# 只有在确认没有其他节点存活时才能使用
galera_new_cluster

# 查看集群状态
SHOW STATUS LIKE 'wsrep_cluster%';
```

**防止脑裂的配置**：

```bash
# Galera 配置
wsrep_provider_options = "gcache.size=2G;gcs.fc_factor=0.8"
# gcs.fc_factor: 流量控制触发阈值，0.8 表示当队列超过 gcache 的 80% 时触发
```

### 追问三：MySQL 8.0 的复制有哪些改进？

**参考答案**：

MySQL 8.0 在复制方面有多项重要改进：

**Clone Plugin**：可以直接克隆另一个 MySQL 实例的数据，比传统的 mysqldump 快得多，特别适合快速搭建新从库或修复故障节点。

```sql
-- 从远程实例克隆数据
INSTALL PLUGIN clone SONAME 'mysql_clone.so';
SET GLOBAL clone_valid_donor_list = '192.168.1.100:3306';
CLONE FROM INSTANCE 'user'@'192.168.1.100:3306';
```

**影子副本（Replica）替代 Slave**：MySQL 8.0.22 开始，用 `Replica` 替代 `Slave`，用词更中性。

**多线程复制增强**：支持按 Schema 或按逻辑时钟的多线程复制，进一步提升并行度。

**binlog 增强**：binlog 事务压缩，减少网络传输和存储开销。

## 延伸思考

### 思考一：多主复制的可能性

Galera 支持多主写入（Multi-Master），理论上所有节点都可以处理写入请求。但多主写入会引入冲突检测和回滚的问题。

**问题**：在什么业务场景下，多主写入是可行的？什么场景下应该避免？

**提示**：考虑业务的数据隔离性、写入冲突的概率、以及对一致性的要求。

### 思考二：分布式数据库的崛起

传统的 MySQL 复制方案已经相当成熟，但面对 PB 级数据、跨地域部署的需求，传统的分库分表方案运维复杂、成本高昂。

**问题**：TiDB、CockroachDB 等 NewSQL 数据库的崛起，会对传统 MySQL 高可用方案产生什么影响？

## 总结

数据库高可用是分布式系统中最复杂的挑战之一，因为它需要在一致性、可用性和性能之间找到平衡。

关键要点回顾：

- **复制模式**：异步复制性能高但有数据丢失风险，半同步复制是折中方案
- **MHA**：自动化故障转移，基于 binlog 位置选择新主库，适合互联网场景
- **Galera**：同步复制，强一致性，但有写入冲突和性能开销
- **云数据库**：RDS/PolarDB 提供免运维的高可用方案，适合云原生架构
- **故障切换**：VIP 漂移 + 连接池重连 + 优雅关闭，是完整的切换流程
- **读写分离**：读流量分散到从库，但需要注意延迟问题

记住：没有银弹。每个方案都是「一致性、性能、复杂度」的权衡。理解这些权衡，才能在具体业务场景下做出正确的选择。
