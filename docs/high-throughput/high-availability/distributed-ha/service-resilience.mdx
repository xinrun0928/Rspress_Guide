# 服务韧性：限流 + 熔断 + 降级的联动设计

凌晨三点，你被一条告警惊醒：订单服务响应超时，大量请求积压。紧接着，依赖的下游库存服务也开始告警——不是它扛不住，而是订单服务频繁重试把它拖死了。一个局部的超时，最终演变成了整个系统的雪崩。

你开始怀疑人生：限流明明上了，熔断也配了，降级策略也写了，为什么还是没兜住？

这是因为，限流、熔断、降级从来不是三个独立的组件，而是一套需要**联动设计**的系统工程。单独配置某一个，或许能解决部分问题；但真正的高手，懂得让三者协同工作，形成纵深的防御体系。

这篇文章，就是你的服务韧性设计指南。我会从三者的本质讲起，剖析各自的分工与局限，然后深入讲解如何设计一套"1 + 1 + 1 > 3"的联动方案。

## 一、问题的本质：分布式系统的脆弱性

在深入设计之前，我们需要先理解问题的本质。

### 1.1 级联故障：多米诺骨牌的连锁反应

分布式系统的一个根本矛盾是：**系统的稳定性取决于它最薄弱的那个环节**。当你调用一个下游服务时，实际上是把控制权的一部分交给了对方。如果对方响应变慢，你的请求会积压；如果对方超时，你的资源会被耗尽。

想象这样一个场景：

1. 数据库查询变慢（可能是慢查询，也可能是连接池耗尽）
2. 上游服务等待时间过长，开始超时
3. 超时后的重试请求涌入，进一步增加数据库压力
4. 数据库彻底崩溃
5. 所有依赖它的服务全部失败

这就是**级联故障**（Cascading Failure）。每一个环节的错误，最终都会像多米诺骨牌一样传导下去。

:::tip 级联故障的本质

级联故障不是因为某个服务"不够好"，而是因为系统各组件之间存在**隐形的依赖关系**。任何一个环节的异常，都可能通过重试、超时、并发等方式放大，最终拖垮整个系统。

:::

### 1.2 为什么单体防御不够？

你可能会问：我把超时设短一点不行吗？我把限流阈值设低一点不行吗？

不行。原因在于，**每个防护手段都有它的盲区**：

| 防护手段 | 能解决的问题 | 无法解决的问题 |
|---|---|---|
| 限流 | 保护系统不被过载压垮 | 无法处理依赖下游已故障的情况 |
| 熔断 | 快速失败，避免资源耗尽 | 无法提供降级响应 |
| 降级 | 提供兜底，保持核心功能可用 | 无法保护自身不被流量冲垮 |

单独使用任何一种，都像只穿了防弹衣但没戴头盔——你确实有一定的防护，但漏洞依然存在。

### 1.3 服务韧性的三层防线

真正可靠的服务韧性，需要三层防线的**联动设计**：

```
第一层：限流（Rate Limiting）—— 在流量入口拦截
        ↓ 拦截不住的流量
第二层：熔断（Circuit Breaker）—— 在故障点切断传播
        ↓ 熔断后的降级
第三层：降级（Degradation）—— 提供最小可用响应
```

这三层防线各有分工，又相互配合。只有联动设计，才能构建纵深的防御体系。

## 二、限流：流量入口的第一道防线

### 2.1 限流的本质：保护系统的"承压上限"

限流的核心目标是**保护系统不被过载压垮**。无论流量来自正常用户、爬虫、还是攻击者，限流器都会按照预设的规则，决定哪些请求可以通过，哪些需要拒绝。

限流的本质是**流量整形**（Traffic Shaping）——把突发的流量平滑成稳定的流量，或者直接丢弃超过阈值的请求。

### 2.2 限流算法对比：选择适合场景的那一个

限流的算法有很多，每种算法都有它的适用场景和局限性。

| 算法 | 原理 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|---|
| 固定窗口 | 在固定时间窗口内计数 | 实现简单 | 窗口边界可能有突发流量 | 简单场景 |
| 滑动窗口 | 移动时间窗口，统计更平滑 | 精度更高 | 实现相对复杂 | 通用场景 |
| 漏桶 | 以固定速率处理请求 | 流量平滑 | 无法应对突发 | 限速场景 |
| 令牌桶 | 按固定速率放入令牌，请求消耗令牌 | 可以应对突发 | 实现稍复杂 | 高并发 API |
| 计数器 | 简单累计 | 最简单 | 无法平滑流量 | 粗粒度限制 |

:::tip 生产环境推荐：令牌桶

在高并发 API 场景中，令牌桶是最常用的算法。Sentinel 和 Resilience4j 都提供了高质量的令牌桶实现。

令牌桶的核心优势是**允许一定程度的突发流量**——比如平时 QPS 是 1000，但瞬时可以处理 2000 的突发。这对于应对流量峰值非常重要。

:::

### 2.3 限流的局限性：它不是万能的

限流能保护系统不被流量压垮，但它无法解决一个关键问题：**下游服务已经故障了怎么办？**

场景一：你的限流阈值是 10000 QPS，实际流量只有 5000 QPS——限流器放行了所有请求。但下游服务因为其他原因（比如数据库连接泄漏）已经响应缓慢，你的 5000 QPS 请求全部在等待超时。

场景二：下游服务已经宕机，限流器依然在放行请求。每个请求都在等待连接建立或超时，最终耗尽你的连接池资源。

这就是限流的盲区：**它只能控制流量大小，无法判断流量是否有效**。

### 2.4 限流的多维度设计

在联动方案中，限流需要考虑多个维度：

**维度一：单机限流 vs 分布式限流**

单机限流只保护单个节点，分布式限流保护整个集群。对于高可用系统，**必须使用分布式限流**。

分布式限流的实现方案：

- **Redis + Lua**：基于 Redis 的原子操作实现全局计数器
- **Sentinel 的集群流控**：Sentinel 的集群模式，支持服务端的流控规则下发
- **令牌桶的分布式扩展**：将令牌桶的状态同步到分布式存储

**维度二：入口限流 vs 出口限流**

入口限流保护自身，出口限流保护下游。但在实际设计中，我们通常同时做两个方向的限流：

```
外部请求 → [入口限流] → 业务处理 → [出口限流] → 下游服务
```

出口限流的目的是**保护下游不被自己的请求拖垮**。即使你的入口限流放行了一万个请求，但如果下游只能承受一千个并发，你需要主动限制对下游的调用量。

**维度三：限流粒度**

| 粒度 | 说明 | 示例 |
|---|---|---|
| 全局限流 | 对所有请求统一限流 | 整个服务不超过 10000 QPS |
| 接口限流 | 按接口限流 | `/order` 接口不超过 5000 QPS |
| 用户限流 | 按用户限流 | 单个用户不超过 100 QPS |
| IP 限流 | 按 IP 限流 | 单个 IP 不超过 50 QPS |

生产环境通常**组合使用多种粒度**：全局限流做兜底，接口限流做精细化控制，用户/IP 限流做防刷。

## 三、熔断：故障传播的"断路器"

### 3.1 熔断的本质：快速失败的"保险丝"

熔断器的灵感来自电力系统中的断路器。当电路过载时，断路器会自动跳闸，切断电源，保护电器不被烧毁。软件系统中的熔断器，也是同样的原理——当检测到下游服务持续失败时，"跳闸"切断调用，直接返回降级响应，避免资源持续消耗。

### 3.2 熔断器的三种状态

熔断器的核心是一个**状态机**，有三种状态：

```
        ┌─────────────────────────────────────┐
        │                                     │
        ▼                                     │
    ┌───────┐    失败率超阈值    ┌─────────┐  │   成功
    │ Closed │ ───────────────▶  │  Open   │  │   恢复
    │ 闭合   │                   │  打开   │  ◀─────┘
    └───┬───┘                   └────┬────┘
        ▲                              │
        │    探测（放行部分请求）        │
        └──────────────────────────────┘
                    半开状态
```

**Closed（闭合）状态**：正常运行状态，所有请求正常通过。如果失败率在阈值之下，熔断器保持闭合。

**Open（打开）状态**：熔断器打开，所有请求直接失败（或返回降级响应），不再调用下游服务。这避免了持续的资源消耗。

**Half-Open（半开）状态**：熔断器周期性探测是否恢复。如果探测请求成功，说明下游已经恢复，熔断器转为闭合；如果探测失败，熔断器重新打开。

### 3.3 熔断的关键参数

熔断器的行为由三个关键参数控制：

| 参数 | 说明 | 典型值 |
|---|---|---|
| 失败阈值 | 触发熔断的失败率 | 50% |
| 熔断时长 | 熔断器保持打开的时间 | 30 秒 |
| 探测时长 | 半开状态持续的时间 | 10 秒 |
| 最小请求数 | 达到最小请求数后才开始计算失败率 | 10 |

```java
// Sentinel 熔断规则配置
DegradeRule degradationRule = new DegradeRule("orderService")
    .setGrade(CircuitBreakerStrategy.ERROR_RATIO.getType())
    .setCount(0.5)                          // 失败率阈值 50%
    .setSlowRatioThreshold(1.0)             // 最大响应时间阈值
    .setMinRequestAmount(10)                // 最小请求数
    .setStatIntervalMs(10000)              // 统计时间窗口 10 秒
    .setTimeWindow(30);                    // 熔断时长 30 秒
```

### 3.4 熔断策略：RT、异常比例、异常数

Sentinel 支持三种熔断策略，根据不同场景选择：

**RT（平均响应时间）策略**

当资源的平均响应时间超过阈值时触发熔断。适用于下游服务响应变慢的场景。

```java
// RT 熔断：平均 RT 超过 500ms，失败率超过 50% 时熔断
DegradeRule rule = new DegradeRule("下游服务")
    .setGrade(CircuitBreakerStrategy.SLOW_RT_RATIO.getType())
    .setCount(500)                         // 阈值 500ms
    .setSlowRatioThreshold(0.5)           // 慢请求比例 50%
    .setMinRequestAmount(5)                // 最小请求数
    .setTimeWindow(30);                    // 熔断 30 秒
```

**异常比例策略**

当请求的异常比例超过阈值时触发熔断。适用于下游服务返回大量异常的的场景。

```java
// 异常比例熔断：异常比例超过 30% 时熔断
DegradeRule rule = new DegradeRule("下游服务")
    .setGrade(CircuitBreakerStrategy.ERROR_RATIO.getType())
    .setCount(0.3)                         // 异常比例阈值 30%
    .setMinRequestAmount(10)               // 最小请求数
    .setTimeWindow(30);
```

**异常数策略**

当异常的绝对数量超过阈值时触发熔断。适用于对错误容忍度极低的场景。

```java
// 异常数熔断：1 分钟内超过 100 个异常时熔断
DegradeRule rule = new DegradeRule("下游服务")
    .setGrade(CircuitBreakerStrategy.ERROR_COUNT.getType())
    .setCount(100)                         // 异常数阈值
    .setMinRequestAmount(5)
    .setTimeWindow(60);                    // 时间窗口 60 秒
```

### 3.5 熔断的局限性：它需要降级响应

熔断器解决了"快速失败"的问题，但它**不会凭空产生一个响应**。当熔断器打开时，你需要告诉调用方"对不起，服务暂时不可用"——这就是降级的职责。

没有降级的熔断，就像断了保险丝之后家里一片漆黑。没有灯光、没有暖气、甚至没有冰箱——所有东西都停了，你只能等待。

降级响应就是那盏**应急灯**——主电路断了，但至少还有一点光亮。

## 四、降级：故障时的"最小可用"保障

### 4.1 降级的本质：放弃"完美"，保住"核心"

降级的核心思想是：**在系统资源紧张或部分功能不可用时，主动放弃一些非核心功能，保证核心功能的可用性**。

这就像飞机迫降时的"抛掉起落架"——你不可能在所有功能都正常的情况下做这个决定，但在危急时刻，你需要果断取舍。

### 4.2 降级的层次：从页面到数据

降级可以发生在不同的层次：

| 层次 | 说明 | 示例 |
|---|---|---|
| 页面降级 | 整个页面显示静态内容 | 详情页显示"商品信息暂时不可用" |
| 功能降级 | 部分功能暂时关闭 | 评论区暂时关闭，保留核心浏览功能 |
| 数据降级 | 使用缓存/默认值代替实时数据 | 用 Redis 缓存的旧数据代替数据库查询 |
| 链路降级 | 跳过部分调用链路 | 跳过非核心的下游调用 |

### 4.3 降级策略的设计原则

**原则一：核心功能优先**

降级的第一步是识别什么是"核心功能"，什么是"非核心功能"。

以电商系统为例：

| 功能类型 | 示例 | 降级策略 |
|---|---|---|
| 核心功能 | 商品浏览、下单、支付 | 永远不能降级 |
| 重要功能 | 库存查询、优惠券 | 可降级为缓存数据 |
| 非核心功能 | 商品推荐、用户评价、猜你喜欢 | 可直接关闭或返回空 |

**原则二：降级响应要"有温度"**

用户看到"服务不可用"和看到"当前浏览量较高，为您展示缓存数据"，感受是完全不同的。

好的降级响应应该：

- **有解释**：告诉用户发生了什么
- **有预期**：告诉用户什么时候会恢复
- **有替代**：提供力所能及的替代方案

```json
{
  "code": 200,
  "success": true,
  "data": {
    "productId": "12345",
    "name": "iPhone 15",
    "price": 5999,
    "stock": "充足",
    "_degraded": true,
    "_message": "库存信息获取延迟，为您展示缓存数据，实际库存以下单时为准"
  }
}
```

**原则三：降级要可监控

降级不是"偷偷摸摸"的行为。每次降级发生，都应该记录日志和指标，方便事后分析和优化。

```java
@SentinelResource(value = "getStock", fallback = "getStockFallback")
public Stock getStock(String productId) {
    return stockService.getStock(productId);
}

// 降级方法
public Stock getStockFallback(String productId, Throwable t) {
    // 记录降级日志
    log.warn("库存查询降级 productId={}, reason={}", productId, t.getMessage());
    
    // 上报降级指标
    metricsService.increment("stock.degraded");
    
    // 返回缓存数据
    return stockCache.get(productId);
}
```

### 4.4 降级与缓存：最佳拍档

降级响应最常用的数据来源是**缓存**。当核心数据无法实时获取时，缓存可以提供"最后一公里"的保障。

但缓存降级需要注意几个问题：

**问题一：缓存过期了怎么办？**

如果缓存中的数据已经过期，降级返回的是"错误"数据而不是"旧"数据。需要区分：

- 缓存数据"稍旧但可用"：允许降级返回
- 缓存数据"已经严重过期"：返回明确提示，不返回误导性数据

**问题二：缓存也没了怎么办？**

降级需要有"兜底的兜底"。如果连缓存都没有，应该返回一个**静态的默认值**。

```java
public Stock getStockFallback(String productId, Throwable t) {
    // 尝试从缓存获取
    Stock cachedStock = stockCache.get(productId);
    if (cachedStock != null) {
        cachedStock.setFromCache(true);
        return cachedStock;
    }
    
    // 兜底：返回静态默认值
    Stock defaultStock = new Stock();
    defaultStock.setProductId(productId);
    defaultStock.setStock("有货（请以实际下单为准）");
    defaultStock.setDegraded(true);
    defaultStock.setDegradedMessage("库存服务暂时不可用，请稍后再试");
    return defaultStock;
}
```

## 五、联动设计：三者协同的工程实践

### 5.1 联动设计的第一原则：各司其职

限流、熔断、降级各有分工：

- **限流**：在流量入口控制总量
- **熔断**：在故障点切断传播
- **降级**：在故障时提供兜底

三者的调用顺序应该是：**限流 → 熔断 → 降级**

```
外部请求
    ↓
[限流] 拦截超量请求
    ↓ 通过的请求
[业务处理] 调用下游服务
    ↓ 调用结果
[熔断] 判断是否需要熔断
    ↓ 需要降级时
[降级] 返回降级响应
```

### 5.2 联动设计的核心：状态同步

三者联动的关键在于**状态同步**。当熔断器打开时，限流器需要感知到这个状态，并调整自己的行为；当降级发生时，监控系统需要感知到，并通知运维团队。

#### 联动一：限流 + 熔断

当熔断器打开时，下游服务已经不可用。此时，如果限流器继续放行请求，会导致：

1. 大量请求直接失败，用户体验差
2. 即使返回降级响应，也是一种资源浪费

更好的做法是：**熔断器打开时，主动降低限流阈值**，减少无效请求。

```java
// Sentinel 的熔断器状态变化回调
private void onCircuitBreakerStateChanged(String resource, CircuitBreaker circuitBreaker) {
    DegradeRule rule = circuitBreaker.getRule();
    
    if (rule.getState() == CircuitBreaker.State.OPEN) {
        // 熔断打开：降低限流阈值
        FlowRule flowRule = FlowRuleManager.getResourceRule(resource);
        if (flowRule != null) {
            // 将阈值降低为原来的一半
            flowRule.setCount(flowRule.getCount() * 0.5);
            FlowRuleManager.loadRules(Collections.singletonList(flowRule));
        }
    } else if (rule.getState() == CircuitBreaker.State.CLOSED) {
        // 熔断恢复：恢复限流阈值
        restoreFlowRuleThreshold(resource);
    }
}
```

#### 联动二：熔断 + 降级

熔断和降级的联动是最直接的。当熔断器打开时，调用方需要知道"我现在处于降级状态"，以便做出正确的响应。

```java
@SentinelResource(value = "remoteCall", 
                  blockHandler = "blockHandler",
                  fallback = "fallback")
public Result remoteCall() {
    return remoteService.call();
}

// 熔断触发时的处理
public Result blockHandler(BlockException e) {
    if (e instanceof CircuitBrokenException) {
        // 熔断中，返回降级响应
        return getDegradedResponse("service_degraded");
    }
    // 限流中
    return getRateLimitResponse("rate_limited");
}

// 业务异常时的降级
public Result fallback(Throwable t) {
    log.error("远程调用失败", t);
    return getDegradedResponse("downstream_error");
}
```

#### 联动三：降级 + 监控

降级发生时，需要让运维团队知道，以便决定是否需要介入。

```java
public Stock getStockFallback(String productId, Throwable t) {
    // 发送告警
    Alert.send("库存服务降级", 
               "productId=" + productId, 
               "reason=" + t.getMessage());
    
    // 上报降级指标
    metrics.record("stock.degraded", 1);
    
    // 返回降级响应
    return stockCache.getOrDefault(productId, DEFAULT_STOCK);
}
```

### 5.4 实战：Sentinel 的联动配置

Sentinel 是目前 Java 生态中最成熟的流量防卫组件，它天然支持限流、熔断、降级的联动。

```java
// 配置降级规则
List<DegradeRule> degradeRules = Arrays.asList(
    // 库存服务：RT 熔断
    new DegradeRule("stockService")
        .setGrade(CircuitBreakerStrategy.SLOW_RT_RATIO.getType())
        .setCount(500)
        .setSlowRatioThreshold(0.5)
        .setMinRequestAmount(5)
        .setTimeWindow(30),
    
    // 支付服务：异常比例熔断
    new DegradeRule("paymentService")
        .setGrade(CircuitBreakerStrategy.ERROR_RATIO.getType())
        .setCount(0.3)
        .setMinRequestAmount(10)
        .setTimeWindow(60),
    
    // 用户服务：异常数熔断
    new DegradeRule("userService")
        .setGrade(CircuitBreakerStrategy.ERROR_COUNT.getType())
        .setCount(100)
        .setMinRequestAmount(5)
        .setTimeWindow(60)
);
DegradeRuleManager.loadRules(degradeRules);

// 配置限流规则
List<FlowRule> flowRules = Arrays.asList(
    // 全局限流
    new FlowRule()
        .setResource("orderService")
        .setGrade(RuleConstant.FLOW_GRADE_QPS)
        .setCount(10000)
        .setControlBehavior(RuleConstant.CONTROL_BEHAVIOR_DEFAULT),
    
    // 接口限流
    new FlowRule()
        .setResource("createOrder")
        .setGrade(RuleConstant.FLOW_GRADE_QPS)
        .setCount(1000)
        .setControlBehavior(RuleConstant.CONTROL_BEHAVIOR_WARM_UP)
);
FlowRuleManager.loadRules(flowRules);
```

配合 Sentinel 的 `@SentinelResource` 注解，可以实现完整的联动：

```java
@SentinelResource(value = "createOrder",
                  blockHandler = "createOrderBlockHandler",
                  fallback = "createOrderFallback")
public Order createOrder(OrderRequest request) {
    // 调用下游服务
    Stock stock = stockService.getStock(request.getProductId());
    PaymentResult payment = paymentService.pay(request);
    return orderRepository.save(request);
}

// 限流/熔断时的处理
public Order createOrderBlockHandler(OrderRequest request, BlockException e) {
    if (e instanceof DegradeException) {
        // 熔断降级
        return createDegradedOrder(request);
    }
    // 限流拒绝
    throw new RateLimitException("请求过于频繁，请稍后再试");
}

// 业务异常时的降级
public Order createOrderFallback(OrderRequest request, Throwable t) {
    log.error("创建订单失败，降级处理", t);
    return createDegradedOrder(request);
}

// 降级订单：写入延迟队列
private Order createDegradedOrder(OrderRequest request) {
    Order order = new Order();
    order.setId(generateId());
    order.setProductId(request.getProductId());
    order.setStatus("PENDING");
    order.setDegraded(true);
    order.setDegradedMessage("订单正在处理中，请稍后查询");
    
    // 降级写入延迟队列，稍后重试
    delayedQueue.offer(order);
    return order;
}
```

### 5.5 联动设计的调优：踩坑经验

#### 坑一：限流阈值过小导致系统"自我保护过度"

有些同学配置限流时过于保守，把阈值设得很低。结果是：系统明明还有大量余量，却开始拒绝请求。

**解决**：限流阈值应该基于**实际压测结果**设置，而不是"拍脑袋"。建议：

- 使用生产环境的 80% 流量进行压测
- 限流阈值 = 压测 QPS × 0.8
- 预留 20% 作为缓冲

#### 坑二：熔断阈值过低导致"抖动"

如果熔断阈值设置过低（比如失败率 10% 就熔断），在正常波动时就会触发熔断，导致系统不稳定。

**解决**：熔断阈值应该基于业务对错误的容忍度设置。建议：

- 对于核心链路：失败率阈值 50%
- 对于非核心链路：失败率阈值 30%
- 熔断时长不宜过短（至少 30 秒）

#### 坑三：降级数据过于"敷衍"

有些系统的降级响应就是一行文字："服务暂时不可用"。用户看到这样的提示，完全不知道该怎么做。

**解决**：降级响应应该有价值，帮助用户完成核心操作。好的降级响应：

- 提供替代方案（如缓存数据、历史记录）
- 给出预期恢复时间
- 引导用户采取下一步行动

## 六、高可用认证与会话管理的特殊考量

在服务韧性设计中，认证与会话管理有其特殊的重要性——它是几乎所有请求都必须经过的"第一道门"。

### 6.1 认证服务的特殊性

认证服务（或者叫 SSO 服务、OAuth 服务）是整个系统的"守门人"。几乎所有的业务请求都需要先经过认证，一旦认证服务不可用，整个系统都会受到影响。

这意味着：

- **认证服务的限流要更宽松**：限流阈值应该比其他服务更高，避免误伤正常用户
- **认证服务的熔断要更谨慎**：熔断后影响范围最大，需要更精确的判断
- **认证服务的降级要更完善**：降级后的降级方案需要特殊处理

### 6.2 认证降级的安全考量

当认证服务不可用时，降级策略需要格外谨慎。几种可选方案：

| 方案 | 优点 | 缺点 | 风险等级 |
|---|---|---|---|
| 直接放行（危险） | 用户体验好 | 失去认证保护 | 极高 |
| 返回 Token 验证失败 | 简单 | 用户无法使用 | 低 |
| 使用缓存的 Token 验证 | 短时间内可用 | 缓存过期后无效 | 中 |
| 使用备用认证中心 | 保持认证能力 | 架构复杂 | 低 |

:::warning 认证降级不能"简单放行"

认证降级绝对不能选择"直接放行"。即使在服务不可用时，也必须确保请求被正确识别和处理。一种常见的做法是：在降级时将所有请求标记为"未认证"，业务服务根据自身逻辑决定是否允许访问。

:::

### 6.3 会话管理的高可用设计

会话管理（Session）也是认证体系的重要组成部分。在分布式系统中，会话管理的高可用面临特殊挑战：

**挑战一：Session 存储的单点故障**

如果 Session 存储在单机内存中，一旦该机器宕机，所有 Session 都会丢失。

**解决方案**：使用 Redis 集群存储 Session，配合 Session 序列化与反序列化机制。

**挑战二：Session 同步的延迟**

在集群环境中，一个节点写入 Session 后，其他节点可能还没有同步到。

**解决方案**：

- 使用分布式 Session 框架（如 Spring Session）
- Session 数据写入后，等待一小段时间再处理请求
- 对于高敏感操作，使用 Session 验证前先检查 Redis

**挑战三：Token 续期的并发问题**

当多个请求同时使用即将过期的 Token 时，可能会出现"惊群效应"——多个请求同时续期，导致负载过高。

**解决方案**：使用 Token 预刷新机制，提前续期，避免一次性续期压力。

## 七、面试追问：服务韧性设计的高频问题

### 7.1 问题一：限流、熔断、降级的区别是什么？

**参考答案**：

| 维度 | 限流 | 熔断 | 降级 |
|---|---|---|---|
| 目的 | 保护系统不被过载压垮 | 保护系统不被故障传播拖垮 | 在故障时提供兜底响应 |
| 触发条件 | 请求量超过阈值 | 下游失败率超过阈值 | 下游调用失败或熔断触发 |
| 行为 | 拒绝请求 | 快速失败，不再调用下游 | 返回降级响应 |
| 保护对象 | 系统自身 | 自身 + 下游 | 用户体验 |
| 时机 | 流量入口 | 故障传播点 | 故障发生后 |

**核心区别**：限流是在"量"上做控制，熔断是在"质"上做控制，降级是在"结果"上做兜底。

### 7.2 问题二：Sentinel 和 Hystrix 的区别？

**参考答案**：

| 维度 | Sentinel | Hystrix |
|---|---|---|
| 设计理念 | 流量防卫兵，专注流量控制 | 电路保护器，专注延迟容忍 |
| 线程隔离 | 信号量隔离（轻量） | 线程池隔离（重量） |
| 熔断策略 | RT、异常比例、异常数 | 异常比例、滚动窗口 |
| 降级 | 支持 | 支持 |
| 实时性 | 滑动窗口统计，实时性更好 | 依赖 HystrixCommand | 
| 社区活跃度 | 活跃（阿里开源） | 维护停滞（Netflix 已停止维护） |
| 扩展性 | 支持自定义 ProcessorChain | 支持自定义 Command |

**结论**：生产环境推荐使用 Sentinel。Hystrix 已经停止维护，不建议在新项目中使用。

### 7.3 问题三：如何设计一套完整的流量防护体系？

**参考答案**：

完整的流量防护体系需要多层防御：

**第一层：入口限流**

在 API 网关或负载均衡层做全局限流，保护整个集群不被流量冲垮。

**第二层：接口限流**

在每个服务内部，对核心接口做精细化限流。

**第三层：熔断保护**

对每个下游调用配置熔断器，快速切断故障传播。

**第四层：降级兜底**

为每个可能失败的下游调用准备降级响应。

**第五层：监控告警**

对限流触发、熔断打开、降级发生做实时监控和告警。

关键点：

- **联动设计**：限流和熔断要感知彼此的状态
- **分级降级**：不同级别的服务有不同的降级策略
- **可观测性**：每一次防护动作都要记录日志和指标

## 延伸思考

### 思考一：自适应限流

当前的限流策略大多是静态配置的——阈值在配置文件中写死，不会根据系统负载动态调整。

**问题**：能不能设计一个"自适应"的限流器，让它根据系统的实时负载（CPU、内存、响应时间）自动调整限流阈值？

**提示**：可以考虑使用 PID 控制器，根据系统负载和目标负载的差值，动态调整限流阈值。

### 思考二：降级策略的"灰度发布"

目前的降级策略是"全开"或"全关"。但在某些场景下，可能需要"部分降级"——比如只有 10% 的请求走降级路径，其他 90% 继续等待正常响应。

**问题**：如何设计一个支持"灰度降级"的机制，让降级策略可以渐进式生效？

**提示**：可以考虑使用百分比切分的流量分配，让一部分请求走降级路径，一部分继续正常调用，根据降级效果逐步调整比例。

## 总结

服务韧性不是三个独立组件的简单叠加，而是一套需要联动设计的系统工程。

核心要点回顾：

- **限流**：控制流量入口，保护系统不被过载压垮
- **熔断**：切断故障传播，保护自身和下游不被拖垮
- **降级**：提供兜底响应，保证核心功能可用

三者的联动原则：

1. **各司其职**：限流管流量，熔断管故障，降级管响应
2. **状态同步**：限流要感知熔断状态，降级要上报监控
3. **分级防护**：核心链路高保护，非核心链路可降级
4. **可观测**：每一次防护动作都要可监控、可追溯

记住：最好的服务韧性设计，是让用户在不知不觉中度过每一次故障。当用户抱怨"今天网站有点慢"而不是"网站崩了"时，你的韧性设计就成功了。
