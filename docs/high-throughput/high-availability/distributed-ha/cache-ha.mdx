# 缓存高可用：Redis 集群、哨兵与主从复制

想象一下这个场景：凌晨两点，你的手机疯狂震动，告警平台上 Redis 相关的告警铺满了整个屏幕——「Redis 主节点连接失败」「从节点全部断开」「缓存服务不可用」。你揉着眼睛打开监控系统，发现整个缓存层已经彻底瘫痪。用户下单、查询商品、获取 Session——所有依赖 Redis 的操作全部失败。这就是没有做好 Redis 高可用的代价。

Redis 是现代互联网架构的「超级英雄」——它扛住了每秒数十万次的读写请求，让数据库从繁重的查询中解脱出来。但这个超级英雄也有弱点：**一旦倒下，整个系统都会受到牵连**。缓存穿透、缓存雪崩、服务雪崩……一连串的连锁反应会像多米诺骨牌一样，让你的系统在几分钟内彻底崩溃。

这篇文章，我带你深入理解 Redis 高可用的三大支柱：**主从复制**、**哨兵模式**、**集群模式**。我会从原理讲起，让你不仅知道「怎么用」，更知道「为什么这样设计」，以及「什么时候该选哪一种方案」。

## 一、主从复制：Redis 高可用的根基

### 1.1 为什么需要主从复制

单机 Redis 的瓶颈显而易见：

- **容量有限**：单节点内存再大也有上限，大型互联网公司动辄数百 GB 的缓存需求，单机根本扛不住
- **并发有限**：单节点 QPS 再高也有天花板，高并发场景下性能会成为瓶颈
- **可用性脆弱**：单机宕机等于缓存层全灭，后端数据库压力瞬间爆表

主从复制解决的是「数据冗余」和「读写分离」的问题。主节点负责所有写操作，从节点异步复制主节点数据，承担读流量。这种架构让 Redis 从「单打独斗」变成了「团队作战」。

:::tip 主从复制 vs 副本

很多同学搞不清「主从复制」和「副本」的区别。在 Redis 的语境里，从节点就是主节点的「副本」——它完整复制了主节点的数据。主从复制是让副本保持和主节点数据一致的技术手段。

:::

### 1.2 复制原理：SYNC 与 PSYNC

Redis 2.8 之前，只支持 `SYNC` 命令来实现主从同步。这个命令的工作流程是：

1. **从节点发送 SYNC**：向主节点发起全量同步请求
2. **主节点执行 BGSAVE**：在后台生成 RDB 快照文件
3. **主节点发送 RDB 文件**：将快照文件发送给从节点
4. **从节点加载 RDB**：清空本地数据，加载接收到的 RDB 文件

问题来了：**如果从节点在同步过程中断开了怎么办？**

重新同步。这意味着每次从节点重连，都要执行一次全量同步。对于大型数据集来说，这简直是灾难——RDB 文件可能有几个 GB，网络传输耗时可能长达几分钟，期间从节点不可用。

Redis 2.8 引入了 `PSYNC` 命令，解决了这个问题。`PSYNC` 支持两种同步模式：

**完全重同步（Full Resynchronization）**：和 SYNC 一样，生成并发送完整的 RDB 文件。适用于首次同步或复制积压缓冲区（replication backlog）无法满足的情况。

**部分重同步（Partial Resynchronization）**：只同步断开期间丢失的数据。Redis 在主节点维护一个 **replication backlog buffer**（复制积压缓冲区），保存最近执行的写命令。从节点重连后，主节点会检查断开的 master_repl_offset 是否在 backlog 范围内，如果在，就只发送断开的命令；否则，执行完全重同步。

```bash
# 主节点配置
replicaof 192.168.1.100 6379  # 指定主节点
replica-serve-stale-data yes  # 从节点在同步过程中是否响应请求

# 主节点复制相关配置
repl-backlog-size 10mb        # 复制积压缓冲区大小，建议设置大一些
repl-backlog-ttl 3600         # 从节点断开后，backlog 保留时间
```

### 1.3 replication backlog buffer：部分重同步的关键

这个缓冲区是理解部分重同步的核心。它是一个固定大小的环形缓冲区，保存最近执行的写命令。当从节点断开后重连：

1. 主节点根据从节点的 `master_repl_offset`，判断断开期间的数据是否还在 backlog 中
2. 如果在（说明断开时间不长，数据量在 backlog 范围内），执行部分重同步
3. 如果不在，执行完全重同步

:::warning backlog-size 的设置

backlog 太小会导致部分重同步失败，频繁触发全量同步。建议设置为 `second * write_size_per_second` 的 2-3 倍。例如，主节点每秒写入 10MB，断开重连预计在 60 秒内，则 backlog-size 应设置为 60 * 10MB * 2 = 1200MB。

:::

### 1.4 主从复制的几个关键问题

**复制延迟**：主从复制是异步的，主节点执行完写命令后立即返回，不等待从节点确认。这意味着从节点的数据可能滞后于主节点。对于强一致性要求高的场景（如金融类业务），异步复制可能导致数据丢失。

**复制风暴**：如果一个主节点挂了，多个从节点同时尝试重连新主节点，可能导致新主节点被多个全量同步请求压垮。解决方案是使用「树状复制」结构，或确保从节点逐个重连。

**从节点可写吗**？：默认情况下，从节点是只读的，不接受写操作。通过 `replica-read-only yes` 控制。如果强行写入，数据不会同步回主节点，且重启后会丢失——这是新手最容易踩的坑。

## 二、哨兵模式：Redis 的守护者

### 2.1 哨兵解决了什么问题

主从复制解决了数据冗余问题，但**故障转移**仍然需要人工介入。哨兵（Sentinel）就是来解决这个问题的——它是一个独立的进程，负责监控主从节点的健康状态，在主节点宕机时自动完成故障转移。

哨兵的核心职责有三个：

1. **监控（Monitoring）**：持续监控主从节点是否正常工作
2. **自动故障转移（Automatic Failover）**：主节点故障时，自动将从节点升级为新主节点
3. **通知（Notification）**：故障转移后，通过 API 通知应用程序

```bash
# 启动哨兵
redis-sentinel /path/to/sentinel.conf
# 或
redis-server /path/to/sentinel.conf --sentinel
```

### 2.2 主观下线 vs 客观下线

这是理解哨兵故障检测的关键概念。

**主观下线（Subjectively Down，SDOWN）**：单个哨兵认为某个实例不可用。哨兵会向被监控节点发送 PING 命令，如果在 `down-after-milliseconds` 时间内没有收到有效回复，就认为该节点主观下线。

**客观下线（Objectively Down，ODOWN）**：多个哨兵（超过 quorum 配置）都认为主节点主观下线，此时才认为主节点真正下线，触发故障转移流程。

```bash title="sentinel.conf"
sentinel monitor mymaster 192.168.1.100 6379 2  # 监控主节点，quorum 为 2
sentinel down-after-milliseconds mymaster 30000   # 30秒无响应判定为主观下线
sentinel failover-timeout mymaster 180000         # 故障转移超时时间
```

:::tip quorum 的设置

quorum 决定了「多少个哨兵认为主节点下线才触发故障转移」。如果设置太低，可能在网络抖动时误触发故障转移；如果设置太高，可能在真正故障时无法触发。建议设置为 `哨兵总数 / 2 + 1`。

:::

### 2.3 故障转移流程：一场精心策划的「夺位」

当主节点被判定为客观下线后，哨兵开始故障转移流程：

**第一步：选举领导者**

多个哨兵需要选出一个「领导者」，由它负责执行故障转移。选举基于 Raft 算法的思路：向其他哨兵发送「申请成为领导者」的请求，如果收到过半的同意票，则成为领导者。

**第二步：挑选新主节点**

领导者哨兵从所有从节点中挑选新主节点，选择标准是：

1. 优先级：`slave-priority` 配置更高的优先
2. 复制偏移量：与原主节点数据差异最小的优先
3. Run ID：以上都相同时，Run ID 字典序更小的优先

```bash
# 从节点配置
replica-priority 100  # 优先级，数字越大优先级越高
```

**第三步：执行故障转移**

1. 向被选中的从节点发送 `SLAVEOF NO ONE`，让它升级为新主节点
2. 向其他从节点发送 `SLAVEOF` 命令，让它们复制新主节点
3. 更新哨兵监控配置，记录新的主节点信息
4. 原来的主节点如果恢复，会自动变成新主节点的从节点

### 2.4 哨兵与客户端集成

哨兵模式下，客户端连接 Redis 的方式发生了变化。客户端不再直接连接固定地址，而是连接哨兵，由哨兵返回当前的主节点地址。

```java
// Jedis 集成哨兵
JedisSentinelPool pool = new JedisSentinelPool(
    "mymaster",                    // 哨兵监控的主节点名称
    Set.of("192.168.1.101:26379", "192.168.1.102:26379", "192.168.1.103:26379"),  // 哨兵节点地址
    new JedisPoolConfig(),         // 连接池配置
    3000,                          // 连接超时
    3000,                          // 读取超时
    "password"                     // 密码（如果有）
);

// 获取连接
Jedis jedis = pool.getResource();
String value = jedis.get("key");
jedis.close();  // 归还连接到池，而不是关闭连接
```

```python
# Python (redis-py) 集成哨兵
from redis.sentinel import Sentinel

sentinel = Sentinel([
    ('192.168.1.101', 26379),
    ('192.168.1.102', 26379),
    ('192.168.1.103', 26379)
], socket_timeout=0.1)

# 获取主节点连接
master = sentinel.master_for('mymaster', socket_timeout=0.1, password='password')
value = master.get('key')

# 获取从节点连接（用于读操作）
slave = sentinel.slave_for('mymaster', socket_timeout=0.1, password='password')
value = slave.get('key')
```

:::warning 客户端需要支持哨兵

不是所有 Redis 客户端都原生支持哨兵模式。使用前请确认客户端库的支持情况。Java 生态推荐 Jedis 和 Lettuce，两者都支持哨兵。Spring Boot 2.0+ 默认使用 Lettuce。

:::

## 三、Redis Cluster：分片集群

### 3.1 为什么需要 Cluster

哨兵模式解决了「高可用」问题，但**容量**和**并发**的问题仍然存在。哨兵的主从复制仍然是全量复制，所有数据在每个从节点都有一份副本。如果你的数据量是 500GB，单机内存只有 256GB，哨兵模式根本玩不转。

Redis Cluster 的解决方案是**分片（Sharding）**：将数据分散到多个节点，每个节点只负责一部分数据。Cluster 由多个主节点组成，每个主节点可以挂载从节点，实现高可用。

### 3.2 哈希槽：16384 的艺术

Redis Cluster 将整个键空间划分为 16384 个哈希槽（Slot）。每个键通过 CRC16 算法计算哈希值，然后对 16384 取模，决定它属于哪个槽。

```
槽编号 = CRC16(key) % 16384
```

16384 这个数字是怎么来的？Redis 作者 antirez 在官方博客中解释过：

- 节点间的心跳包需要携带每个节点的槽信息，槽越少，心跳包越小
- Redis Cluster 设计时考虑最多 1000 个节点，16384 / 1000 ≈ 16，每个节点平均负责 16 个槽，心跳包大小合理
- 16384 是 2^14，在实际场景中足够覆盖所有数据

:::tip 槽 vs 节点

一个 Redis Cluster 通常有多个主节点（如 6 个），每个主节点负责一部分槽（如 16384 / 6 ≈ 2731 个槽）。当客户端访问某个键时，会先计算槽编号，然后找到负责该槽的节点，直接访问。

:::

### 3.3 集群节点通信：Meet 与心跳

Redis Cluster 的每个节点都与其他所有节点保持 TCP 连接，通过 Gossip 协议交换信息。

**节点发现（MEET）**：新节点加入集群时，向集群中任意节点发送 MEET 命令，被邀请节点再将新节点信息传播给其他节点，最终整个集群都知道新节点的存在。

**心跳机制**：每个节点每秒会随机选择几个节点，发送 PING 消息。收到 PING 的节点需要回复 PONG。如果一个节点超过 `node-timeout` 时间没有响应某个节点的 PING，就被标记为疑似下线（PFAIL）。

**故障检测**：当超过半数的节点认为某个节点疑似下线时，该节点被标记为下线（FAIL），触发故障转移。

```bash title="cluster-enabled yes"
# 节点配置
port 7000
cluster-enabled yes
cluster-config-file nodes-7000.conf
cluster-node-timeout 15000      # 节点超时时间
cluster-replica-validity-factor 10  # 从节点有效性因子
```

### 3.4 集群故障转移：从节点的自我救赎

集群模式下，主节点故障后的故障转移由其从节点自动完成：

**自动故障转移**：当主节点被标记为 FAIL 后，如果有从节点存在，集群会选举一个从节点升级为新主节点。选举过程使用 Raft 算法的变体，从节点向其他节点拉票，获得过半票数的从节点成为新主节点。

**手动故障转移**：`CLUSTER FAILOVER` 命令允许管理员手动触发故障转移。相比自动故障转移，手动故障转移更可控，可以指定是在原主节点停止接受写入时再切换，避免数据丢失。

**集群可用性**：故障转移期间，如果集群有超过半数的主节点存活，集群就能继续工作（处理读写请求）。如果主节点数量不足半数，集群将停止处理请求，进入不可用状态。

### 3.5 集群扩缩容：槽迁移的艺术

集群扩缩容的核心是槽迁移——将数据从一个节点迁移到另一个节点，同时保持服务可用。

**添加新节点**：

1. 启动新节点，此时它不负责任何槽
2. 使用 `CLUSTER ADDSLOTS` 将部分槽迁移到新节点
3. 客户端会自动感知槽的分布变化

**槽迁移过程**：

1. 源节点将槽状态标记为「迁移中」（MOVING）
2. 对迁移中的槽，源节点有两种响应：
   - 如果键存在于源节点：正常返回数据
   - 如果键不存在于源节点：返回 `ASK` 重定向，让客户端去目标节点获取
3. 目标节点收到请求后，正常处理（允许访问迁移中的槽）
4. 迁移完成后，槽被正式分配给新节点

```bash
# 查看槽分布
CLUSTER SLOTS

# 迁移槽
# 1. 先在新节点执行 SETSKIP
CLUSTER SETSLOT 1000 IMPORTING <source-node-id>

# 2. 在源节点执行 SETSKIP
CLUSTER SETSLOT 1000 MIGRATING <target-node-id>

# 3. 使用 CLUSTER GETKEYSINSLOT 获取槽中的键
# 4. 使用 MIGRATE 命令迁移每个键
MIGRATE 192.168.1.100 7000 key 0 1000
```

:::tip ASK vs MOVED

`ASK` 重定向是迁移过程中的临时状态，客户端应该先去源节点尝试，再去目标节点获取。`MOVED` 重定向是槽的永久归属变化，客户端应该更新本地的槽映射表，以后直接访问正确的节点。

:::

## 四、Sentinel vs Cluster：如何选型

这是面试和实战中最常被问到的问题。两种方案各有适用场景，不能简单地说谁更好。

| 维度 | 哨兵模式 | 集群模式 |
|---|---|---|
| **数据分片** | 不支持，所有数据在每个节点都有副本 | 支持，数据分散到多个节点 |
| **容量** | 受单机内存限制 | 可水平扩展，支持 PB 级数据 |
| **并发** | 受单机性能限制 | 多节点并行处理，并发能力线性增长 |
| **复杂度** | 简单，主从 + 哨兵即可 | 复杂，需要管理槽映射、节点通信 |
| **客户端支持** | 几乎所有客户端都支持 | 需要客户端支持集群协议 |
| **事务支持** | 支持（单节点事务） | 限制（跨槽操作需要特殊处理） |
| **适用场景** | 小型系统、数据量 < 100GB | 大型系统、数据量大、高并发 |

**选型建议**：

- **数据量 < 100GB，QPS < 10万**：哨兵模式足够，选择简单可靠的方案
- **数据量 > 100GB，或 QPS > 10万**：必须上集群，考虑未来的扩展性
- **需要强一致性**：谨慎使用集群，异步复制可能导致数据丢失

:::warning 哨兵的局限性

哨兵模式本质上是主从复制的增强版，解决了自动故障转移问题，但无法突破单机内存和性能的瓶颈。如果你的数据量持续增长，迟早要面临迁移到集群的问题。建议在项目初期就规划好架构，避免后期的痛苦迁移。

:::

## 五、Redis 7.0 新特性：流与集群改进

Redis 7.0 带来了多项针对高可用的改进：

**集群功能的正式发布**：Redis 7.0 将 Redis Cluster 标记为生产就绪，之前的实验性标签被移除。所有集群相关 API 稳定可用。

**集群 API 增强**：`CLUSTER SHARDS` 命令可以获取完整的集群分片信息，`CLUSTER SLOTS` 的输出格式更加清晰。

**复制协议优化**：PSYNC2 协议进一步完善，减少了完全重同步的概率，提升了复制效率。

**Memory 命令改进**：`MEMORY STATS` 可以获取更详细的内存使用统计，便于排查问题。

## 六、生产环境最佳实践

### 6.1 连接池配置

高并发场景下，Redis 连接池的配置至关重要。连接池过小会导致连接等待，过大会浪费资源。

```java
JedisPoolConfig poolConfig = new JedisPoolConfig();
poolConfig.setMaxTotal(200);        // 最大连接数
poolConfig.setMaxIdle(50);           // 最大空闲连接
poolConfig.setMinIdle(10);           // 最小空闲连接
poolConfig.setMaxWaitMillis(3000);   // 获取连接最大等待时间
poolConfig.setTestOnBorrow(true);    // 获取连接时检查可用性
poolConfig.setTestOnReturn(false);   // 归还连接时检查可用性
poolConfig.setTestWhileIdle(true);   // 空闲时检测连接可用性
poolConfig.setMinEvictableIdleTimeMillis(60000);  // 空闲连接最小存活时间
poolConfig.setNumTestsPerEvictionRun(10);         // 每次检测的连接数
```

### 6.2 监控指标

生产环境必须监控以下 Redis 指标：

| 指标 | 告警阈值 | 说明 |
|---|---|---|
| 内存使用率 | `> 80%` | 内存不足时可能导致 Redis 拒绝服务 |
| 连接数 | `> maxclients * 80%` | 连接数接近上限时新请求会被拒绝 |
| 复制延迟 | `> 1秒` | 延迟过高可能导致从节点数据陈旧 |
| 命令执行时间 | `> 100ms` | 慢查询可能是性能瓶颈的信号 |
| 主从断开 | 发生即告警 | 主从断开可能是网络问题或节点故障 |

### 6.3 数据一致性：异步复制的代价

无论哨兵还是集群，Redis 的主从复制都是**异步的**。这意味着一旦主节点故障，尚未同步到从节点的数据会丢失。

**可能丢失的数据**：

- 主节点已提交但尚未发送给从节点的操作
- 从节点正在处理的操作（在故障转移发生时）

**减少数据丢失的策略**：

1. **开启 min-replicas-to-write**：要求至少 N 个从节点确认写入后才返回成功
2. **使用半同步复制**：主节点等待至少一个从节点写入成功后返回
3. **合理设置复制积压缓冲区**：确保小范围断开时能执行部分重同步
4. **使用关注延迟的从节点做读操作**：读取对一致性要求高的数据时，读主节点

```bash
# 主节点配置：至少 1 个从节点确认写入
min-replicas-to-write 1
min-replicas-max-lag 10  # 从节点延迟超过 10 秒时，主节点拒绝写入
```

## 七、面试追问

### 追问一：哨兵和集群可以一起用吗？

**参考答案**：

可以，但不推荐。

理论上，你可以在集群的每个主节点上配置哨兵，实现「集群 + 哨兵」的双重高可用。但这会增加系统复杂度，引入更多的故障点和运维负担。

Redis Cluster 本身已经内置了故障检测和自动故障转移机制，不需要额外使用哨兵。Cluster 的故障检测基于 Gossip 协议，当主节点故障时，其从节点会自动发起选举，完成故障转移。这个过程比哨兵更快、更可靠。

如果你的系统使用了 Cluster，就不需要再部署哨兵了。

### 追问二：主节点挂了，从节点会自动提升吗？

**参考答案**：

在哨兵模式下，从节点**不会自动提升**为主节点，必须等待哨兵完成故障转移流程。哨兵会基于优先级、复制偏移量等规则选择一个从节点，发送 `SLAVEOF NO ONE` 命令让它升级为新主节点。

在集群模式下，同样需要等待从节点发起选举并获得过半票数，才能升级为主节点。没有任何一种模式是「瞬间」完成故障转移的，通常需要 10-30 秒。

这 10-30 秒的窗口期，缓存服务不可用，应用程序需要做好降级处理（如回源到数据库）。

### 追问三：Redis Cluster 的槽迁移过程中，客户端如何处理？

**参考答案**：

槽迁移过程中，客户端可能遇到两种重定向：

**MOVED 重定向**：槽已经完成迁移，客户端应该更新本地的槽映射表，以后直接访问新节点。智能客户端（如 Jedis Cluster）会自动处理 MOVED 重定向，并缓存最新的槽映射。

**ASK 重定向**：槽正在迁移中，这是一个临时状态。客户端应该：
1. 先向源节点发送请求
2. 如果收到 ASK 重定向，说明键在迁移中，客户端应该向目标节点发送 ASKING 命令，然后再次发送原请求
3. 目标节点允许客户端访问正在迁入的键

```java
// Jedis Cluster 自动处理 ASK 重定向
public String get(String key) {
    int slot = JedisClusterHashTagUtil.getSlot(key);
    JedisPool pool = cache.getSlotPool(slot);

    try {
        return pool.getResource().get(key);
    } catch (JedisMovedDataException e) {
        // MOVED 重定向，更新槽映射表
        cache.redirectAndRetry(e);
    } catch (JedisAskDataException e) {
        // ASK 重定向，发送 ASKING 命令后重试
        return cache.askingAndRetry(e);
    }
}
```

### 追问四：如何避免主从复制产生数据不一致？

**参考答案**：

首先要明确：**异步复制天然会产生数据不一致**，无法完全避免。我们能做的是尽量减少不一致的范围和时间。

**减少不一致时间的方案**：

1. **提升网络质量**：复制延迟主要来自网络，确保主从节点之间的网络稳定、低延迟
2. **使用快速机器作为主节点**：主节点性能不足时，处理命令变慢，从节点追赶的速度也会变慢
3. **控制从节点数量**：从节点越多，主节点复制压力越大，可能导致复制延迟增加
4. **使用半同步复制**：主节点等待至少一个从节点写入成功后再返回，降低数据丢失风险

**接受不一致的方案**：

如果业务能接受最终一致性，可以采用以下策略：

1. **读取主节点**：对一致性要求高的读取操作，直接读主节点
2. **版本号机制**：写入时携带版本号，读取时检查版本号是否匹配
3. **延迟双删**：写操作后延迟一段时间再删除缓存，期间可能读到旧数据

## 延伸思考

### 思考一：缓存高可用 vs 数据库高可用

如果 Redis 和 MySQL 都挂了，你会先恢复哪个？

这是一个经典的「鸡生蛋还是蛋生鸡」问题。答案取决于业务场景：

- **缓存优先恢复**：如果缓存数据可以从数据库重建，先恢复缓存。因为缓存的职责是保护数据库，缓存恢复后，数据库压力会降低，再处理数据库故障
- **数据库优先恢复**：如果数据库数据是持久化的原始数据，没有缓存可以重建，那必须先恢复数据库。此时所有请求都会直接打到数据库，需要做好限流和降级

### 思考二：多级缓存架构下的高可用

现代互联网架构中，缓存通常不止 Redis 一层。还可能有本地缓存（Caffeine、Guava Cache）、CDN 缓存等。当 Redis 故障时，本地缓存可以作为临时庇护所。

**思考题**：如果本地缓存和 Redis 数据不一致，业务会出现什么问题？如何设计多级缓存的失效策略？

**提示**：考虑缓存穿透、缓存击穿、缓存雪崩的场景，以及它们在高可用架构下如何应对。

## 总结

Redis 高可用是互联网架构的基石。从主从复制的异步同步，到哨兵的自动故障转移，再到集群的分片扩展，Redis 提供了一套完整的高可用解决方案。

关键要点回顾：

- **主从复制**：通过 SYNC/PSYNC 实现数据同步，replication backlog 支撑部分重同步
- **哨兵模式**：基于主观下线/客观下线的故障检测，Raft 风格的领导者选举，自动完成故障转移
- **集群模式**：16384 槽的分片机制，Gossip 协议的节点通信，支持在线扩缩容
- **选型原则**：小规模用哨兵，大规模用集群
- **数据一致性**：异步复制天然存在数据丢失风险，需要根据业务需求权衡

记住：Redis 高可用不是银弹。任何高可用方案都有其局限性，理解这些局限性，才能在故障发生时从容应对。
