# 分布式协调服务：ZooKeeper、etcd、Nacos 对比

凌晨三点，你被一阵急促的告警铃声惊醒。监控大屏上，订单服务的所有实例同时失联，支付系统开始疯狂重试，库存服务的连接池被打满……整个系统在几分钟内陷入瘫痪。

这不是黑客攻击，也不是代码 bug。这是一次典型的「脑裂」故障——多个服务节点对系统状态产生了不一致的认知。

你开始排查：谁应该是主节点？数据有没有丢失？服务注册信息还在吗？

如果你部署过 ZooKeeper，这一切问题都有标准答案。但如果你用的是 etcd 或 Nacos，情况可能完全不同。

这就是分布式协调服务的价值所在——它们是分布式系统的大脑，负责在混乱中建立秩序。这篇文章，我们来聊聊三个最主流的协调服务：ZooKeeper、etcd 和 Nacos，以及它们各自的适用场景。

## 一、分布式协调的本质：为什么需要协调服务

### 1.1 分布式系统的三大困境

在单机系统中，一切都很简单——一个进程，一份数据，一个真相。但在分布式世界里，真相变得模糊起来。

**网络是不可靠的**。消息可能丢失，可能延迟，可能重复到达。你发给 leader 的投票请求，可能在半路迷路。

**节点是会宕机的**。任何一个节点都可能突然crash，也可能恢复后带着脏数据回来。你以为它死了，它只是睡着了。

**时钟是不可同步的**。分布式系统中的节点各自有自己的时钟，「事件 A 发生在事件 B 之前」这种判断，在跨节点场景下变得异常复杂。

这三个困境，造就了分布式系统的核心难题：**如何在不可靠的网络上，让一群不可靠的节点对系统状态达成共识？**

### 1.2 协调服务解决什么问题

分布式协调服务就是为此而生的。它们解决以下核心问题：

| 问题类型 | 协调服务的作用 |
|---|---|
| **选主问题** | 当集群中的主节点宕机时，快速选举出新的主节点 |
| **分布式锁** | 确保同一时刻只有一个节点能操作共享资源 |
| **配置管理** | 集中管理配置，支持动态更新和推送 |
| **服务发现** | 让服务知道有哪些实例可用，它们的地址是什么 |
| **分布式Barrier** | 让节点协调等待，直到所有节点都准备好 |

没有协调服务，分布式系统就像一群没有指挥的乐队——每个人都在演奏，但合在一起就是一团噪音。

## 二、CAP 理论与协调服务选型

### 2.1 CAP 三选二

在讨论具体实现之前，必须理解 CAP 理论。这是分布式系统设计的基石，也是选择协调服务时最重要的参考。

CAP 理论指出：一个分布式系统最多只能同时满足以下三个特性中的两个：

- **Consistency（一致性）**：所有节点看到的数据是一致的
- **Availability（可用性）**：每个请求都能得到响应
- **Partition tolerance（分区容错性）**：系统在网络分区时仍能运行

由于网络分区是不可避免的（参见上文「网络是不可靠的」），所以实际上我们只能在 C 和 A 之间做选择：

- **CP 系统**：优先保证一致性。发生网络分区时，牺牲可用性。
- **AP 系统**：优先保证可用性。发生网络分区时，允许数据短暂不一致。

### 2.2 三种协调服务的定位

理解了 CAP 理论，我们来看三个协调服务的定位：

| 协调服务 | 默认模式 | 核心协议 | 一致性保证 |
|---|---|---|---|
| ZooKeeper | CP | ZAB | 强一致，写请求必须 leader 处理 |
| etcd | CP | Raft | 强一致，Raft 共识算法 |
| Nacos | CP+AP 可切换 | Raft / Distro | 默认 AP，可切换为 CP |

**ZooKeeper 和 etcd 都是 CP 系统**——它们优先保证一致性。这意味着当网络分区发生时，它们可能会拒绝服务，直到分区恢复。

**Nacos 更加灵活**，默认使用 AP 模式，适合注册中心场景；也可以切换到 CP 模式，用于配置管理等需要强一致的场景。

:::tip 选型建议

如果你的业务对数据一致性要求极高（如分布式锁、配置管理），选择 CP 模式。
如果你的业务对可用性要求更高（如服务注册发现），AP 模式更合适。

Nacos 的 CP/AP 切换功能让你可以根据场景灵活选择，这也是它在国内微服务生态中流行的原因之一。

:::

## 三、ZooKeeper：老牌协调服务的工程哲学

### 3.1 ZAB 协议：ZooKeeper 的共识基石

ZooKeeper 使用 ZAB（ZooKeeper Atomic Broadcast）协议来实现分布式一致性。这是一个专门为 ZooKeeper 设计的协议，核心思想是：**所有写请求都经过 leader，leader 通过广播协议将更新同步给 follower**。

ZAB 协议的工作流程分为两个阶段：

**消息广播阶段**：
1. 客户端发起写请求，发送到任意一个 ZooKeeper 节点
2. 该节点将请求转发给 leader
3. leader 为该请求分配一个全局递增的事务 ID（zxid）
4. leader 将提案（proposal）广播给所有 follower
5. 超过半数 follower 确认后，leader 发送提交（commit）指令
6. follower 收到 commit 后应用更新

**恢复阶段**：
当 leader 崩溃时，系统进入恢复阶段。此时需要选举出新的 leader，并且确保之前 committed 的更新不会丢失。

ZAB 的设计哲学是「简单即美」——它不追求通用性，而是针对 ZooKeeper 的实际使用场景做了大量优化。这使得 ZooKeeper 在特定场景下效率极高。

### 3.2 数据模型：Znode 树

ZooKeeper 的数据模型是一个层次化的命名空间，类似于文件系统：

```
/zookeeper
  /config
  /services
    /order
      /0001  {"host": "192.168.1.1", "port": 8080}
      /0002  {"host": "192.168.1.2", "port": 8080}
    /payment
      /0001  {"host": "192.168.1.3", "port": 8080}
  /master
    /lock   (临时节点，表示当前锁持有者)
```

每个节点称为 **Znode**，有以下特性：

| Znode 类型 | 说明 |
|---|---|
| **持久节点** | 创建后一直存在，需要显式删除 |
| **临时节点** | 与客户端会话绑定，会话断开后自动删除 |
| **顺序节点** | 创建时自动追加递增序号 |

Znode 可以存储少量数据（默认 1MB），但通常用于存储元信息（如服务实例的地址），而不是大文件。

### 3.3 Watch 机制：事件通知

ZooKeeper 的 Watch 机制是其最强大的特性之一。它允许客户端订阅节点变化，当节点状态发生变化时，ZooKeeper 会主动通知客户端。

```java
// ZooKeeper Watch 示例
public class ServiceDiscovery {
    private final ZooKeeper zk;
    private final String servicePath;
    
    public List<String> discoverServices() throws KeeperException, InterruptedException {
        // 第一次获取数据
        List<String> children = zk.getChildren(servicePath, event -> {
            // Watch 回调：当子节点变化时，会触发这个回调
            if (event.getType() == Event.EventType.NodeChildrenChanged) {
                // 重新获取服务列表
                try {
                    discoverServices();
                } catch (Exception e) {
                    e.printStackTrace();
                }
            }
        });
        return children;
    }
}
```

Watch 机制的优点是**实时性好**——客户端不需要轮询，变化发生时立即收到通知。缺点是** Watch 是单次触发的**——触发一次后需要重新注册。

:::warning Watch 的单次触发特性

很多开发者在使用 ZooKeeper 时容易踩这个坑：Watch 被触发后，如果你没有重新注册 Watch，后续的变化就收不到了。正确做法是在回调中重新注册 Watch，并且注意处理好并发情况。

:::

### 3.4 Leader 选举：Fast Leader Election

当 leader 节点宕机时，ZooKeeper 会通过 Fast Leader Election（FLE）算法快速选举出新的 leader。

选举过程基于以下规则：

1. **epoch（逻辑时钟）**：每个选举轮次有一个唯一的 epoch 值，epoch 越大越新
2. **zxid**：事务 ID，越大表示处理的事务越多
3. **serverId**：服务器 ID（在配置文件中指定），越大优先级越高

投票时，每个节点会比较自己和对方的这三个值，选择「epoch 最大 → zxid 最大 → serverId 最大」的节点作为 leader。

选举成功的条件是**获得超过半数节点的投票**。这也是为什么 ZooKeeper 集群通常采用奇数节点部署——3 节点集群可以容忍 1 节点故障，5 节点集群可以容忍 2 节点故障。

### 3.5 ZooKeeper 典型应用场景

**分布式锁实现**：

```java
public class ZooKeeperDistributedLock {
    private final ZooKeeper zk;
    private final String lockPath;
    private String currentNode;
    
    public boolean acquire() throws KeeperException, InterruptedException {
        // 创建临时顺序节点
        currentNode = zk.create(lockPath + "/lock-", 
            new byte[0], 
            ZooDefs.Ids.OPEN_ACL_UNSAFE, 
            CreateMode.EPHEMERAL_SEQUENTIAL);
        
        // 获取所有锁节点，按序号排序
        List<String> nodes = zk.getChildren(lockPath, false);
        Collections.sort(nodes);
        
        // 如果当前节点是最小的，获得锁
        String smallestNode = nodes.get(0);
        return currentNode.endsWith(smallestNode);
    }
}
```

**配置管理**：

```java
// 监听配置变化
public void watchConfig(String configPath) throws KeeperException, InterruptedException {
    byte[] data = zk.getData(configPath, event -> {
        if (event.getType() == Event.EventType.NodeDataChanged) {
            try {
                byte[] newData = zk.getData(configPath, this, new Stat());
                // 处理新配置
                processConfig(new String(newData));
            } catch (Exception e) {
                e.printStackTrace();
            }
        }
    }, new Stat());
}
```

## 四、etcd：云原生时代的通用存储

### 4.1 Raft 协议：可理解的共识算法

etcd 使用 Raft 协议实现分布式一致性。Raft 的设计目标是**可理解性**——相比于 Paxos，Raft 更容易理解和实现。

Raft 将共识问题分解为三个子问题：

**Leader 选举**：
- 节点启动时是 Follower 状态
- 如果 Follower 在 election timeout 时间内没收到 Leader 的心跳，就变成 Candidate 并发起选举
- 获得多数节点投票的 Candidate 成为新 Leader

**日志复制**：
- Leader 接收客户端请求，将请求追加到本地日志
- Leader 并行发送 AppendEntries RPC 给 Follower
- 超过半数 Follower 确认后，Leader 提交日志并通知 Follower

**安全性**：
- Leader 必须有所有已提交的日志条目
- 新 Leader 的日志必须比多数节点都新

### 4.2 MVCC 与键值存储

etcd 的核心是一个支持 MVCC（多版本并发控制）的键值存储。这带来两个重要特性：

**版本号与修订历史**：
每次键值变化都会产生一个新的修订版本号（revision），用户可以查询历史版本：

```bash
# 获取当前值
etcdctl get /config/service/timeout

# 获取历史值
etcdctl get /config/service/timeout --rev=3

# 观察变化
etcdctl watch /config/service/timeout --rev=5
```

**事务支持**：
```bash
# 事务：只有在条件满足时才执行
etcdctl txn --cas \
  -success="/a=1" \
  -failure="echo 'Condition failed'"
```

### 4.3 TTL 与租约机制

etcd 支持基于 TTL（Time-To-Live）的自动过期，这是实现服务注册、健康检查等功能的基础。

```bash
# 创建租约（lease），TTL 为 60 秒
etcdctl lease grant 60
# 输出：lease 758107afa29d7a46 granted with TTL(60s)

# 将键绑定到租约
etcdctl put /service/order/instance-1 '{"host":"192.168.1.1"}' --lease=758107afa29d7a46

# 续约（keepalive）
etcdctl lease keep-alive 758107afa29d7a46
```

当租约过期时，所有绑定到该租约的键都会被自动删除。这完美契合了服务注册场景——当服务实例宕机时，租约自动过期，实例自动从注册表中移除。

### 4.4 gRPC 与 Watch API

etcd 提供 gRPC API，这是其性能优秀的原因之一。Watch API 允许客户端订阅键的变化：

```protobuf
// etcd Watch API (Protocol Buffers 定义)
service Watch {
    rpc Watch(stream WatchRequest) returns (stream WatchResponse);
}

message WatchCreateRequest {
    string key = 1;
    int64 start_revision = 2;
    bool progress_notify = 3;
}
```

Java 客户端使用方式：

```java
// etcd Java 客户端示例
public class EtcdServiceRegistry {
    private final EtcdClient client;
    private final String servicePrefix;
    
    public void register(String serviceName, String instanceId, String host, int port) {
        // 创建租约
        Lease lease = client.getLeaseClient().grant(30).join();
        
        // 注册服务实例，绑定租约
        String key = servicePrefix + "/" + serviceName + "/" + instanceId;
        String value = "{\"host\":\"" + host + "\",\"port\":" + port + "}";
        client.getKVClient().put(
            ByteSequence.from(key, StandardCharsets.UTF_8),
            ByteSequence.from(value, StandardCharsets.UTF_8),
            PutOption.newBuilder().withLeaseId(lease.getID()).build()
        ).join();
    }
    
    public void watchServices(String serviceName, Consumer<String> callback) {
        String prefix = servicePrefix + "/" + serviceName + "/";
        client.getWatchClient().watch(
            ByteSequence.from(prefix, StandardCharsets.UTF_8),
            response -> {
                for (WatchEvent event : response.getEvents()) {
                    callback.accept(event.getKey().toString(StandardCharsets.UTF_8));
                }
            }
        );
    }
}
```

## 五、Nacos：面向云原生的一站式方案

### 5.1 CP+AP 双模式

Nacos 最大的特点是**同时支持 CP 和 AP 两种一致性模式**，并且可以在运行时切换。

```yaml
# Nacos 配置示例
nacos:
  consensus:
    # 默认使用 AP 模式（服务注册推荐）
    # 切换为 CP: nacos.cluster.conf: raft
    mode: standalone  # standalone / cluster
```

**AP 模式（默认）**：
- 使用自研的 Distro 协议
- 数据分片存储，每个节点负责一部分数据
- 节点之间异步同步数据
- 分区时可以继续服务，但数据可能短暂不一致

**CP 模式**：
- 使用 Raft 协议
- 所有写入通过 Leader 完成
- 强一致性，但分区时可能不可用

```java
// Java SDK 切换一致性模式
// AP 模式注册（默认）
namingService.registerInstance("order-service", instance);

// CP 模式注册（配置管理推荐）
configService.publishConfig("datasource", "DEFAULT_GROUP", jsonConfig);
```

### 5.2 服务发现与注册

Nacos 的服务发现功能非常完善，支持两种模式：

**临时实例（ephemeral）**：
- 基于心跳保活
- 心跳停止后自动删除
- 对应 AP 模式

**永久实例（persistent）**：
- 基于长连接
- 实例下线需要主动注销
- 对应 CP 模式

```java
// Nacos 服务注册
@NacosInjected
private NamingService namingService;

public void registerService(String serviceName, String ip, int port) {
    Instance instance = new Instance();
    instance.setIp(ip);
    instance.setPort(port);
    instance.setClusterName("DEFAULT");
    instance.setWeight(1.0);
    
    // ephemeral = true 表示临时实例，心跳保活
    // ephemeral = false 表示永久实例
    instance.setEphemeral(true);
    
    namingService.registerInstance(serviceName, instance);
}

// Nacos 服务发现
public List<Instance> getInstances(String serviceName) {
    try {
        // healthyOnly = true 只返回健康实例
        return namingService.selectInstances(serviceName, true);
    } catch (NacosException e) {
        throw new RuntimeException(e);
    }
}
```

### 5.3 配置管理

Nacos 的配置管理支持热更新、配置分组、命名空间隔离等企业级特性：

```java
// Nacos 配置监听
@NacosConfigurationProperties(dataId = "application.yml", autoRefreshed = true)
@Configuration
public class AppConfig {
    // 配置变更时会自动刷新
    private String timeout;
    private int maxConnections;
}

// 手动监听配置变化
public class ConfigListener {
    @NacosConfigListener(dataId = "datasource.json")
    public void onConfigChanged(String config) {
        // 解析并应用新配置
        DataSourceConfig ds = JSON.parseObject(config, DataSourceConfig.class);
    }
}
```

### 5.4 Nacos 的生态优势

Nacos 在国内微服务生态中非常流行，原因不仅是功能完善，更是与 Spring Cloud Alibaba 的深度集成：

```yaml
# Spring Cloud Alibaba + Nacos 集成
spring:
  cloud:
    nacos:
      discovery:
        server-addr: nacos-server:8848
        namespace: ${NACOS_NAMESPACE}
      config:
        server-addr: nacos-server:8848
        file-extension: yml
```

## 六、三方对比：选型决策树

### 6.1 核心能力对比

| 维度 | ZooKeeper | etcd | Nacos |
|---|---|---|---|
| **一致性模型** | CP | CP | CP+AP 可切换 |
| **共识协议** | ZAB | Raft | Distro (AP) / Raft (CP) |
| **数据模型** | Znode 树 | KV 存储 | KV + 服务发现 |
| **Watch 机制** | 一次性 Watch | 持续 Watch | 持续 Watch |
| **客户端支持** | ZooKeeper Java/C/Go | gRPC 多语言 | HTTP/DNS/Java |
| **运维复杂度** | 高 | 中 | 低 |
| **生态集成** | Dubbo、HBase、Kafka | Kubernetes、Prometheus | Spring Cloud Alibaba |

### 6.2 性能对比

| 指标 | ZooKeeper | etcd | Nacos |
|---|---|---|---|
| **写吞吐** | ~10k QPS | ~10k QPS (3节点) | ~15k QPS |
| **读吞吐** | ~100k QPS | ~100k QPS | ~100k QPS |
| **延迟 (P99)** | ~5ms | ~3ms | ~3ms |
| **集群规模** | 推荐 3-7 节点 | 推荐 3-5 节点 | 推荐 3-5 节点 |

:::tip 性能优化建议

无论选择哪个协调服务，以下配置都能提升性能：
- 使用 SSD 存储 WAL 日志
- 合理设置 snapshot 频率
- 监控网络延迟，避免跨地域部署
- 对于 ZooKeeper，避免创建过大的 znode

:::

### 6.3 选型决策树

```
你的使用场景是什么？
│
├─ 通用分布式协调（配置、锁、命名）
│  ├─ 已有 Kubernetes 基础设施 → etcd
│  ├─ 使用 Dubbo/DubboX → ZooKeeper
│  └─ 使用 Spring Cloud Alibaba → Nacos
│
├─ 服务注册与发现
│  ├─ Spring Cloud Alibaba → Nacos（开箱即用）
│  ├─ 需要 Kubernetes 原生 → Consul
│  └─ 高可用要求极高 → ZooKeeper
│
├─ 配置中心
│  ├─ 简单 KV 配置 → Nacos / Apollo
│  ├─ 大规模高并发 → etcd
│  └─ 企业级特性 → Apollo
│
└─ 分布式锁
   ├─ 需要跨语言 → etcd / Redis
   ├─ Java 生态 → ZooKeeper
   └─ 简单场景 → Redis (Redisson)
```

## 七、实战：跨服务迁移与注意事项

### 7.1 从 ZooKeeper 迁移到 Nacos

很多团队在使用 ZooKeeper 做服务注册后，想要迁移到 Nacos。以下是迁移注意事项：

**数据模型映射**：

| ZooKeeper | Nacos | 说明 |
|---|---|---|
| `/services/order/instance-1` | 服务名: `order`, 实例ID: `instance-1` | 概念对应 |
| EPHEMERAL 节点 | ephemeral=true | 心跳保活 |
| Watch | Listener | 功能对应 |

**迁移步骤**：

1. **双注册期**：新旧服务同时注册到两个注册中心
2. **流量切换**：通过网关将流量逐步切换到新服务
3. **观察验证**：确认服务发现正常工作后，停止 ZooKeeper 注册
4. **清理下线**：移除 ZooKeeper 依赖，部署新版本

### 7.2 常见坑与最佳实践

**ZooKeeper 坑点**：

- Watch 是单次触发的，必须在回调中重新注册
- Session timeout 设置要合理，太短容易误判
- 不要用 ZooKeeper 存储大量数据（znode 默认 1MB 限制）

**etcd 坑点**：

- WAL 日志磁盘空间要充足，否则写入会阻塞
- Watch 大量 key 时注意内存使用
- 定期 compact 历史 revision，避免空间膨胀

**Nacos 坑点**：

- 默认配置不适合大规模集群，需要调参
- 切换 CP 模式时，已有数据需要迁移
- 服务实例健康检查依赖心跳，心跳间隔要设置合理

:::warning 生产环境注意事项

1. 协调服务是基础设施的核心，一旦出问题影响整个系统
2. 生产环境必须至少 3 节点部署
3. 监控告警必须覆盖：节点宕机、选主频繁、数据不同步
4. 容量规划要预留 30% 的增长空间

:::

## 面试追问

### 问题一：为什么 ZooKeeper 不适合做服务注册中心？

**参考答案**：

ZooKeeper 本身是 CP 系统，优先保证一致性。但在服务注册场景中，我们更关心可用性。当 ZooKeeper 集群发生网络分区时，整个服务注册功能会不可用——这是不能接受的。

更重要的是，ZooKeeper 的 Watch 机制是「一锤子买卖」，触发一次就失效。如果用 ZooKeeper 做服务发现，每次服务上下线都需要重新 Watch，开发复杂度高。

Nacos 和 Consul 在这方面做得更好：Nacos 默认 AP 模式，Consul 使用 gossip 协议，都能在保证可用性的同时提供服务发现能力。

### 问题二：etcd 和 ZooKeeper 的核心区别是什么？

**参考答案**：

**协议层面**：ZooKeeper 使用自研的 ZAB 协议，etcd 使用通用的 Raft 协议。Raft 更容易理解，etcd 的代码也比 ZooKeeper 更简洁。

**数据模型**：ZooKeeper 使用 Znode 树，适合存储少量元数据；etcd 是 KV 存储，更通用，支持事务和 MVCC。

**API 风格**：ZooKeeper 使用自定义协议，etcd 使用 gRPC + Protocol Buffers，多语言支持更好。

**定位**：ZooKeeper 是专门的协调服务，etcd 是通用的分布式存储（可以理解为「分布式 etcd」）。

### 问题三：Nacos 如何实现 CP+AP 双模式？

**参考答案**：

Nacos 的双模式通过两套不同的协议实现：

**AP 模式（Distro 协议）**：
- 数据分片，每个节点负责部分数据
- 写入时只同步到部分节点，不保证强一致
- 适合服务注册场景，可用性优先

**CP 模式（Raft 协议）**：
- 所有写入经过 Leader，通过 Raft 复制
- 保证强一致性，但分区时可能不可用
- 适合配置管理等强一致场景

用户可以通过 API 动态切换：
```bash
# 切换为 CP 模式
curl -X PUT 'http://nacos-server:8848/v1/ns/operator/switches?entry=serverMode&value=CP'

# 切换为 AP 模式
curl -X PUT 'http://nacos-server:8848/v1/ns/operator/switches?entry=serverMode&value=AP'
```

## 延伸思考

### 思考一：协调服务的边界在哪里？

协调服务很强大，但并不是万能的。有人尝试用 ZooKeeper/etcd 做消息队列、做分布式计算、做分布式数据库……这些场景真的适合吗？

思考一下：协调服务的优势是「小数据、强一致」，如果你的场景需要「大数据、高吞吐」，是不是应该选择更专业的组件？

### 思考二：没有协调服务行不行？

随着云原生理念的普及，有人提出「去中心化」的设计——用 gossip 协议做服务发现，用 Redis 做分布式锁，用数据库做配置存储。

这样做的好处是依赖更少，但代价是数据一致性更难保证。是否有场景真的可以「去协调服务」？这值得深入思考。

---

## 总结

分布式协调服务是微服务架构的基石，选择合适的协调服务需要综合考虑业务场景、一致性需求、运维能力等因素。

**关键要点回顾**：

- **ZooKeeper**：老牌协调服务，适合做配置管理、分布式锁，对 Dubbo 等框架有良好支持
- **etcd**：云原生友好，与 Kubernetes 深度集成，适合做服务配置、分布式存储
- **Nacos**：一站式方案，同时支持服务发现和配置管理，与 Spring Cloud Alibaba 无缝集成

没有最好的协调服务，只有最合适的协调服务。根据你的业务场景和技术栈，做出最适合的选择。
