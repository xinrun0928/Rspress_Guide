# 负载均衡健康检查：主动检查与被动检查

你的服务有 10 台后端服务器，其中一台的数据库连接泄漏了，正在疯狂报错。负载均衡器知道吗？它会把请求继续发到这台故障服务器吗？

答案是：**取决于你配置了什么样的健康检查**。

健康检查是负载均衡系统的「眼睛」。没有健康检查，负载均衡器就是瞎子，它不知道哪些服务器是健康的，只能机械地分发请求；有了健康检查，负载均衡器才能「看见」后端服务器的状态，把请求发给健康的节点。

这篇文章，我带你把健康检查彻底搞懂——从被动检查到主动检查，从 TCP 检测到 HTTP 检测，从单点检测到分布式检测。

## 一、为什么健康检查至关重要

### 1.1 没有健康检查会发生什么

假设有 3 台后端服务器，其中 Server B 宕机了：

```
正常状态：
用户请求 → [负载均衡器] → Server A / Server B / Server C
                         (轮询分发)

Server B 宕机后（无健康检查）：
用户请求 → [负载均衡器] → Server A / Server B / Server C
                              ↑
                         这台已经宕机了
                         
结果：1/3 的请求失败
```

健康检查的作用就是「发现故障服务器并将其摘除」：

```
有健康检查：
用户请求 → [负载均衡器] → Server A / Server C
                              ↑
                         Server B 被摘除
                         
结果：所有请求都打到健康的服务器
```

### 1.2 健康检查解决的核心问题

健康检查要解决的是 **「流量调度准确性」** 问题：

**问题一：服务不可用但服务器存活**

服务器进程在运行，但服务已经不可用了（死锁、OOM、依赖服务故障）。如果只用「服务器 ping 不通」来判断是否存活，会漏掉这类问题。

**问题二：服务恢复后的流量恢复**

故障服务器修复后，需要重新加入流量池。什么时候让它恢复？恢复太快可能导致请求失败，恢复太慢导致负载不均。

**问题三：优雅下线**

服务器要下线，但还有正在处理的请求。如果直接摘除，请求会失败。需要一种机制，让服务器「优雅」地退出。

### 1.3 健康检查的类型

健康检查分为两大类：**主动检查**和**被动检查**。

| 类型 | 说明 | 典型实现 |
|---|---|---|
| 主动检查 | 负载均衡器主动探测后端服务器 | Nginx upstream check module、kube-probe |
| 被动检查 | 通过请求结果判断服务器状态 | Nginx max_fails、 Ribbon failureCount |

## 二、被动检查：基于请求结果的判断

### 2.1 Nginx 的被动检查

Nginx 内置了被动健康检查，通过请求的「失败」来判断服务器是否故障。

```nginx title="nginx.conf"
upstream backend {
    server 192.168.1.101:8080 max_fails=3 fail_timeout=30s;
    server 192.168.1.102:8080 max_fails=3 fail_timeout=30s;
    server 192.168.1.103:8080 backup;
}
```

**参数解析**：

- `max_fails=3`：30 秒内连续 3 次失败，标记为不可用
- `fail_timeout=30s`：统计失败的时间窗口，同时也是不可用状态的持续时间

**工作流程**：

```bash
t=0s:    请求成功，失败计数=0
t=1s:    请求成功，失败计数=0
t=2s:    请求失败（超时），失败计数=1
t=3s:    请求失败（500），失败计数=2
t=4s:    请求失败（连接拒绝），失败计数=3
t=4s+:   标记为不可用，30 秒内不再发送请求

t=34s:   重新尝试一次
         如果成功：恢复为可用
         如果失败：继续不可用 30 秒
```

### 2.2 被动检查的局限性

**局限一：必须先失败才能发现问题**

被动检查的本质是「后知后觉」——必须先有一个失败的请求，才能记录一次失败。如果健康检查间隔是 1 分钟，一台服务器故障后，可能要等 1 分钟才能检测到，期间所有打到这台服务器的请求都会失败。

**局限二：4xx 不算失败**

```bash
# Nginx 默认配置
# 返回 404、403 等 4xx 状态码，不算失败
# 只有 5xx、超时、连接失败才算失败
```

这意味着如果服务器返回大量 4xx，Nginx 仍然认为服务器是健康的。

**局限三：无法区分「暂时故障」和「永久故障」**

如果服务器只是短暂的网络抖动，被动检查可能误判为故障。

:::tip 被动检查的配置建议

- `max_fails` 设置为 2~3：太低容易误判，太高发现故障慢
- `fail_timeout` 设置为 30~60 秒：给服务器足够的恢复时间
- 结合主动健康检查：主动检查发现故障，被动检查统计失败

:::

### 2.3 Ribbon 的被动检查

Ribbon 使用「失败计数」的方式实现被动检查：

```bash title="application.yml"
user-service:
  ribbon:
    # 连续失败多少次后标记为不可用
    # 注意：Ribbon 2.x 之后移除了内置的 Ping
    # 需要配合 HealthCheck 或 Eureka 心跳
```

```java title="NIWSDiscoveryPing.java"
public class NIWSDiscoveryPing implements IPing {
    
    @Override
    public boolean isAlive(Server server) {
        // 从服务注册中心检查实例状态
        InstanceInfo instanceInfo = eurekaClient.getInstanceInfo(
            server.getHost(), 
            server.getPort()
        );
        
        if (instanceInfo == null) {
            return false;
        }
        
        // 检查实例状态
        return InstanceStatus.UP.equals(instanceInfo.getStatus());
    }
}
```

### 2.4 Ribbon + Eureka 心跳机制

Ribbon 通常配合 Eureka 使用，Eureka 的心跳机制提供了被动检查的能力：

```bash
# Eureka 心跳配置（服务端）
eureka:
  instance:
    lease-renewal-interval-in-seconds: 30  # 客户端心跳间隔
    lease-expiration-duration-in-seconds: 90  # 多久没心跳则剔除
```

```bash
# Eureka 心跳流程
客户端 → 每 30 秒向服务端发送心跳
服务端 → 90 秒没收到心跳，剔除实例
Ribbon → 从 Eureka 获取实例列表时，已自动过滤掉不健康的实例
```

## 三、主动检查：定期探测后端状态

### 3.1 主动检查原理

主动健康检查是负载均衡器主动向后端服务器发送探测请求，根据响应判断服务器是否健康。

```bash
主动检查流程：

负载均衡器 ──→ 发送探测请求 ──→ 后端服务器
                 ↓
           HTTP GET /health
                 ↓
           ┌─────┴─────┐
           ↓           ↓
        返回 200    返回错误/超时
           ↓           ↓
        标记健康    连续 N 次失败
           ↓           ↓
        保持可用    标记为不可用
```

### 3.2 TCP 端口检测

最基本的健康检查方式：检测端口是否开放。

```bash
# TCP 检测原理
telnet 192.168.1.101 8080
# 如果能连接，说明端口开放

# 或者用 nc
nc -zv 192.168.1.101 8080
```

**优点**：开销小，不占用应用资源
**缺点**：只能检测端口，无法判断应用是否正常

```nginx title="nginx_upstream_check_module 配置"
upstream backend {
    server 192.168.1.101:8080;
    server 192.168.1.102:8080;
    server 192.168.1.103:8080;
    
    # TCP 检测
    check interval=3000 rise=1 fall=2 timeout=1000 type=tcp;
}
```

### 3.3 HTTP/HTTPS 检测

更可靠的方式：发送 HTTP 请求，检测应用层健康状态。

```nginx title="nginx_upstream_check_module 配置"
upstream backend {
    server 192.168.1.101:8080;
    server 192.168.1.102:8080;
    server 192.168.1.103:8080;
    
    # HTTP 检测
    check interval=5000 rise=2 fall=3 timeout=2000 type=http;
    check_http_send "HEAD /health HTTP/1.0\r\n\r\n";
    check_http_expect_alive http_2xx http_3xx;
}
```

**参数说明**：

| 参数 | 说明 |
|---|---|
| `interval` | 检测间隔（毫秒） |
| `rise` | 连续成功多少次恢复为可用 |
| `fall` | 连续失败多少次标记为不可用 |
| `timeout` | 检测超时时间（毫秒） |
| `type` | 检测类型（tcp、http、ssl_hello、mysql、ajp） |
| `check_http_send` | 发送的 HTTP 请求内容 |
| `check_http_expect_alive` | 期望的 HTTP 状态码 |

### 3.4 健康检查接口设计

后端服务需要提供一个健康检查接口：

```java title="HealthController.java"
@RestController
public class HealthController {
    
    @Autowired
    private DataSource dataSource;
    
    @Autowired
    private RedisTemplate<String, String> redisTemplate;
    
    @GetMapping("/health")
    public ResponseEntity<HealthResponse> health() {
        HealthResponse response = new HealthResponse();
        response.setStatus("UP");
        response.setTimestamp(System.currentTimeMillis());
        
        Map<String, ComponentHealth> components = new HashMap<>();
        
        // 检查数据库
        components.put("db", checkDatabase());
        
        // 检查 Redis
        components.put("redis", checkRedis());
        
        // 检查磁盘空间
        components.put("disk", checkDisk());
        
        response.setComponents(components);
        
        // 如果有任何组件不健康，整体返回 503
        boolean allHealthy = components.values().stream()
            .allMatch(c -> "UP".equals(c.getStatus()));
        
        if (allHealthy) {
            return ResponseEntity.ok(response);
        } else {
            response.setStatus("DOWN");
            return ResponseEntity.status(503).body(response);
        }
    }
    
    private ComponentHealth checkDatabase() {
        try {
            dataSource.getConnection().createStatement().execute("SELECT 1");
            return new ComponentHealth("UP", null);
        } catch (Exception e) {
            return new ComponentHealth("DOWN", e.getMessage());
        }
    }
    
    private ComponentHealth checkRedis() {
        try {
            redisTemplate.opsForValue().get("health-check");
            return new ComponentHealth("UP", null);
        } catch (Exception e) {
            return new ComponentHealth("DOWN", e.getMessage());
        }
    }
    
    private ComponentHealth checkDisk() {
        File disk = new File("/");
        long free = disk.getFreeSpace();
        long total = disk.getTotalSpace();
        double usage = (total - free) * 100.0 / total;
        
        if (usage > 90) {
            return new ComponentHealth("DOWN", "Disk usage: " + usage + "%");
        }
        return new ComponentHealth("UP", null);
    }
}
```

```java title="HealthResponse.java"
public class HealthResponse {
    private String status;
    private long timestamp;
    private Map<String, ComponentHealth> components;
    
    // getters and setters
}

public class ComponentHealth {
    private String status;
    private String error;
    
    // getters and setters
}
```

## 四、健康检查参数设计

### 4.1 检测间隔

检测间隔决定了故障发现的「速度」。

```bash
# 计算故障发现时间
故障发现时间 = 检测间隔 × fall

# 示例
interval=3000ms, fall=3
故障发现时间 = 3秒 × 3 = 9秒
```

**设计原则**：

| 场景 | 推荐间隔 | 原因 |
|---|---|---|
| 核心业务 | 1~3 秒 | 快速发现故障 |
| 非核心业务 | 10~30 秒 | 减少健康检查开销 |
| 特殊场景 | 更短间隔 | 需要更快的故障发现 |

**过短的间隔可能导致的问题**：

- 健康检查流量过大
- 网络抖动误判
- 后端服务器压力增加

### 4.2 失败阈值

`fall`（连续失败多少次标记为不可用）和 `rise`（连续成功多少次恢复为可用）是两个关键参数。

```bash
# 场景：网络抖动
interval=1000ms, fall=3, rise=2

t=0s:  请求失败
t=1s:  请求失败
t=2s:  请求成功（rise=1）
t=3s:  请求成功（rise=2，恢复为可用）

# 网络抖动只持续了 2 秒，被正确处理
```

```bash
# 场景：持续故障
interval=1000ms, fall=3, rise=2

t=0s:  请求失败
t=1s:  请求失败
t=2s:  请求失败
t=3s:  标记为不可用

t=33s: 重新尝试
t=34s:  请求失败
t=35s:  请求失败
t=36s:  请求失败
t=37s:  继续保持不可用
```

**设计建议**：

| 参数 | 推荐值 | 说明 |
|---|---|---|
| `fall` | 2~5 | 太小容易误判，太大发现慢 |
| `rise` | 1~3 | 恢复应该比故障检测快 |
| `timeout` | interval 的 1/3~1/2 | 给足够的响应时间 |

### 4.3 超时设置

超时时间决定了「多慢算失败」。

```bash
# 超时设置
interval=5000ms, timeout=2000ms, fall=3

t=0s:  发送请求
t=2s:  请求超时（2 秒）
t=3s:  记录失败 1 次
t=5s:  发送下一个请求
t=7s:  请求超时
t=8s:  记录失败 2 次
...
```

**超时太短的问题**：

- 服务器短暂繁忙时误判

**超时太长的问题**：

- 故障发现慢
- 连接资源被占用

## 五、分布式健康检查

### 5.1 Kubernetes kubeproxy 健康检查

Kubernetes 的 Service 负载均衡由 kube-proxy 实现，kubeproxy 使用 iptables 或 IPVS 进行负载均衡。

```bash
# Kubernetes 健康检查配置
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  containers:
  - name: my-container
    image: my-image
    ports:
    - containerPort: 8080
    livenessProbe:     # 存活探针
      httpGet:
        path: /health
        port: 8080
      initialDelaySeconds: 30
      periodSeconds: 10
      failureThreshold: 3
    readinessProbe:    # 就绪探针
      httpGet:
        path: /ready
        port: 8080
      initialDelaySeconds: 5
      periodSeconds: 5
      failureThreshold: 3
```

**存活探针（livenessProbe）** vs **就绪探针（readinessProbe）**：

| 类型 | 作用 | 不通过时 |
|---|---|---|
| 存活探针 | 判断容器是否需要重启 | 杀死并重启容器 |
| 就绪探针 | 判断容器是否可以接收流量 | 从 Service 摘除 |

```bash
# 就绪探针示意
t=0s:    容器启动
t=5s:    开始 readinessProbe 检查
t=10s:   检查通过，加入 Service
         ↓
t=20s:   检查失败，从 Service 摘除
t=30s:   检查通过，重新加入 Service
```

生产环境中，通常有多层健康检查协同工作：

## 六、健康检查与会话保持

### 6.1 问题背景

如果启用了会话保持（Sticky Session），健康检查可能会与会话保持产生冲突。

```bash
# 场景：IP 哈希 + 健康检查
upstream backend {
    ip_hash;
    server 192.168.1.101:8080;
    server 192.168.1.102:8080;
}

# 用户 A 的 IP 哈希到 101
用户 A → 101

# 101 健康检查失败，被摘除
用户 A → ???
```

### 6.2 会话保持场景下的健康检查策略

**策略一：使用 Cookie 会话保持**

```nginx title="nginx.conf"
upstream backend {
    server 192.168.1.101:8080;
    server 192.168.1.102:8080;
}

server {
    listen 80;
    
    # 使用 Cookie 会话保持
    ip_hash;
    
    # 或者使用 Cookie
    # sticky cookie srv_id expires=1h domain=.example.com path=/;
    
    location / {
        proxy_pass http://backend;
        
        # 健康检查不影响用户请求
        health_check uri=/health interval=5s fails=3 passes=2;
    }
}
```

**策略二：灰度下线**

```bash
# 灰度下线流程
1. 将服务器权重从 100 降到 50（不再接收新请求）
2. 等待现有请求处理完成（约 30~60 秒）
3. 将权重降到 0（完全摘除）
4. 执行维护操作
5. 完成后，将权重恢复为 100
```

```nginx title="nginx.conf"
upstream backend {
    server 192.168.1.101:8080 weight=50;
    server 192.168.1.102:8080 weight=100;
}
```

### 6.3 优雅下线实现

```java title="GracefulShutdownHandler.java"
@Component
public class GracefulShutdownHandler implements Lifecycle {
    
    private volatile boolean shuttingDown = false;
    
    @Override
    public void start() {
        // 启动时注册关闭钩子
        Runtime.getRuntime().addShutdownHook(new Thread(this::shutdown));
    }
    
    @Override
    public void stop() {
        shutdown();
    }
    
    private void shutdown() {
        shuttingDown = true;
        
        // 1. 将实例从注册中心摘除
        // Eureka:
        applicationInfoManager.setInstanceStatus(InstanceStatus.DOWN);
        
        // Nacos:
        namingService.deregisterInstance("user-service", ip, port);
        
        // 2. 等待现有请求处理完成
        try {
            Thread.sleep(30000);  // 30 秒
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
        
        // 3. 关闭应用
    }
    
    @GetMapping("/health")
    public ResponseEntity<String> health() {
        if (shuttingDown) {
            return ResponseEntity.status(503).body("SHUTTING_DOWN");
        }
        return ResponseEntity.ok("OK");
    }
    
    @GetMapping("/ready")
    public ResponseEntity<String> ready() {
        if (shuttingDown) {
            return ResponseEntity.status(503).body("NOT_READY");
        }
        return ResponseEntity.ok("READY");
    }
}
```

## 七、降级与恢复策略

### 7.1 分级降级

健康检查失败后，可以采用分级降级策略：

| 级别 | 检查项 | 降级措施 |
|---|---|---|
| 1 级 | 基础端口检测 | 摘除服务器 |
| 2 级 | HTTP 健康接口 | 摘除服务器 |
| 3 级 | 依赖服务检测 | 返回降级数据 |
| 4 级 | 响应时间检测 | 降低权重 |

```java title="HealthCheckResult.java"
public class HealthCheckResult {
    private HealthLevel level;
    private String message;
    private DegradeStrategy strategy;
    
    public enum HealthLevel {
        HEALTHY,      // 完全健康
        DEGRADED,     // 降级可用
        UNHEALTHY     // 完全不可用
    }
    
    public enum DegradeStrategy {
        NONE,          // 无需降级
        WEIGHT_DOWN,   // 降低权重
        PARTIAL_TRAFFIC, // 只接受部分流量
        FULL_TRAFFIC  // 接受全部流量（但可能失败）
    }
}
```

### 7.2 渐进式恢复

服务器从故障中恢复后，不应该瞬间接收全部流量，而应该渐进式恢复：

```bash
# 渐进式恢复流程
t=0s:    服务器恢复，开始健康检查
t=5s:    第一次检查通过，设置权重=10%
t=10s:   第二次检查通过，权重=25%
t=20s:   第三次检查通过，权重=50%
t=30s:   第四次检查通过，权重=100%（完全恢复）

# 权重渐进恢复
weight: 0 → 10% → 25% → 50% → 100%
time:   0s  → 5s  → 10s → 20s → 30s
```

```nginx title="nginx.conf"
upstream backend {
    server 192.168.1.101:8080 slow_start=30s;
    server 192.168.1.102:8080;
}
```

### 7.3 熔断模式

健康检查连续失败达到阈值后，可以触发熔断，暂时跳过这台服务器：

```java title="CircuitBreakerHealthCheck.java"
public class CircuitBreakerHealthCheck {
    
    private final AtomicInteger failureCount = new AtomicInteger(0);
    private final AtomicInteger successCount = new AtomicInteger(0);
    private volatile CircuitState state = CircuitState.CLOSED;
    
    private static final int FAILURE_THRESHOLD = 5;
    private static final int SUCCESS_THRESHOLD = 2;
    private static final long BREAK_DURATION = 30000; // 30 秒熔断
    
    public enum CircuitState {
        CLOSED,     // 正常
        OPEN,       // 熔断
        HALF_OPEN  // 半开
    }
    
    public boolean isHealthy() {
        switch (state) {
            case CLOSED:
                return checkHealth();
            case OPEN:
                // 熔断中，等待恢复
                return false;
            case HALF_OPEN:
                // 半开状态，尝试一次请求
                return tryRequest();
        }
        return false;
    }
    
    private void recordSuccess() {
        if (state == CircuitState.HALF_OPEN) {
            if (successCount.incrementAndGet() >= SUCCESS_THRESHOLD) {
                state = CircuitState.CLOSED;
                failureCount.set(0);
                successCount.set(0);
            }
        } else {
            failureCount.set(0);
        }
    }
    
    private void recordFailure() {
        if (failureCount.incrementAndGet() >= FAILURE_THRESHOLD) {
            state = CircuitState.OPEN;
            // 延迟后转为半开状态
            scheduleHalfOpen();
        }
    }
}
```

## 八、面试追问：健康检查高频问题

### 8.1 问题一：健康检查间隔设置多少合适？

**参考答案**：

健康检查间隔需要在「故障发现速度」和「检查开销」之间平衡。

**计算公式**：

```
故障发现时间 = 检测间隔 × 失败阈值
健康检查开销 = 请求数/秒 = 服务器数 × 1/检测间隔
```

**场景化建议**：

| 场景 | 推荐间隔 | 失败阈值 | 故障发现时间 |
|---|---|---|---|
| 核心服务 | 1~3 秒 | 2~3 | 3~9 秒 |
| 普通服务 | 5~10 秒 | 3~5 | 15~50 秒 |
| 非关键服务 | 30~60 秒 | 3 | 90~180 秒 |

**过短的问题**：

- 健康检查流量大，可能导致网络拥塞
- 网络抖动误判
- 后端服务器压力增加

**过长的问题**：

- 故障发现慢
- 用户请求失败时间长

### 8.2 问题二：如何设计一个健康检查接口？

**参考答案**：

一个完整的健康检查接口应该包含以下层级：

**1. 基础检查（必须）**

```java
@GetMapping("/health")
public ResponseEntity<String> health() {
    return ResponseEntity.ok("OK");
}
```

**2. 详细检查（可选，但推荐）**

```java
@GetMapping("/health")
public ResponseEntity<HealthResponse> health() {
    HealthResponse response = new HealthResponse();
    response.setStatus("UP");
    response.setComponents(checkComponents());
    
    if (allComponentsHealthy(response)) {
        return ResponseEntity.ok(response);
    } else {
        return ResponseEntity.status(503).body(response);
    }
}
```

**3. 检查要点**：

| 检查项 | 方法 | 超时 |
|---|---|---|
| 数据库连接 | SELECT 1 | 2~5 秒 |
| Redis 连接 | PING | 1~2 秒 |
| 外部 API | HTTP 请求 | 3~5 秒 |
| 磁盘空间 | File.listRoots() | 1 秒 |
| 线程池 | activeCount | 1 秒 |

**4. 响应时间控制**：

- 健康检查接口本身要有超时
- 依赖项超时后直接标记为不健康，不要等待

```java
@GetMapping("/health")
public ResponseEntity<HealthResponse> health() {
    ExecutorService executor = Executors.newFixedThreadPool(5);
    
    try {
        // 所有检查并行执行，最多等待 5 秒
        CompletableFuture.allOf(
            CompletableFuture.supplyAsync(this::checkDatabase, executor),
            CompletableFuture.supplyAsync(this::checkRedis, executor),
            CompletableFuture.supplyAsync(this::checkDisk, executor)
        ).get(5, TimeUnit.SECONDS);
    } catch (TimeoutException e) {
        return ResponseEntity.status(503)
            .body(new HealthResponse("TIMEOUT"));
    }
    
    return ResponseEntity.ok(response);
}
```

### 8.3 问题三：被动检查和主动检查各有什么优缺点？

**参考答案**：

| 维度 | 被动检查 | 主动检查 |
|---|---|---|
| 检测时机 | 必须先有失败请求 | 主动探测，无需失败请求 |
| 响应速度 | 慢（需要先失败） | 快（定期检测） |
| 开销 | 低（复用用户请求） | 高（额外的探测流量） |
| 准确性 | 低（依赖真实请求） | 高（主动探测） |
| 适用场景 | 非核心服务 | 核心服务 |

**最佳实践**：

- 被动检查 + 主动检查结合使用
- 被动检查处理「显式失败」
- 主动检查处理「隐式故障」（服务活着但不健康）

```nginx title="组合使用"
upstream backend {
    # 被动检查：基于请求失败
    server 192.168.1.101:8080 max_fails=3 fail_timeout=30s;
    
    # 主动检查：定期探测
    check interval=5000 rise=2 fall=3 timeout=2000 type=http;
    check_http_send "HEAD /health HTTP/1.0\r\n\r\n";
}
```

### 8.4 问题四：如何避免健康检查导致的「雪崩」？

**参考答案**：

健康检查设计不当可能导致雪崩效应：

**场景一：健康检查太严格**

```bash
# 太严格的检查
interval=1000ms, fall=2, timeout=500ms

# 如果服务器暂时繁忙（GC、连接池耗尽）
t=0s:  请求超时
t=1s:  请求超时
t=2s:  标记为不可用

# 所有请求被重定向到其他服务器
# 其他服务器压力增大，也开始超时
# 雪崩发生
```

**解决方案**：

1. **放宽失败阈值**：fall=3~5，避免网络抖动误判
2. **使用 TCP 检测代替 HTTP**：开销更小，不容易被服务器压力影响
3. **设置最小成功率**：不只是看失败次数，也看成功率

**场景二：恢复太快**

```bash
# 服务器刚恢复就全量接收流量
t=30s:  服务器恢复
t=31s:  全量流量涌入
t=32s:  服务器再次过载

# 雪崩发生
```

**解决方案**：

1. **渐进式恢复**：权重从低到高
2. **半开状态**：先放一部分流量试探
3. **延迟恢复**：故障恢复后等待一段时间再接入

## 九、留给你的思考题

### 题目一：健康检查的「盲区」

假设服务器进程正常运行，但数据库连接池已耗尽。此时 `/health` 接口能正常返回 200，但实际业务请求会失败。

问题：如何在健康检查中检测到这种「半死不活」的状态？

**提示**：考虑在健康检查中实际执行一次数据库操作，而不是只检查连接存在。

### 题目二：多机房健康检查

假设你的服务部署在北京和上海两个机房，它们之间有专线连接。

问题：如何设计健康检查策略，使得一个机房完全故障时，能自动切换到另一个机房？健康检查的流量应该走哪里？

**提示**：考虑健康检查路径（公网 vs 专线），以及跨机房故障检测的延迟问题。

### 题目三：健康检查与灰度发布

在灰度发布过程中，需要把新版本的服务器逐步接入流量。如果健康检查太严格，新版本刚上线就可能被摘除；如果太宽松，可能会放过有问题的版本。

问题：如何设计一套健康检查策略，在保证可用性的同时，也能保证灰度发布的安全性？

**提示**：考虑版本标识、金丝雀权重、错误率监控等维度。

## 总结

健康检查是负载均衡系统的「眼睛」，没有它，负载均衡器就是瞎子。

**健康检查类型**：

| 类型 | 说明 | 适用场景 |
|---|---|---|
| 被动检查 | 基于请求失败判断 | 非核心服务 |
| 主动检查 | 主动探测 | 核心服务 |
| TCP 检测 | 检测端口开放 | 基础检测 |
| HTTP 检测 | 检测应用状态 | 应用层检测 |
| 自定义检测 | 执行脚本/逻辑 | 特殊场景 |

**关键参数**：

| 参数 | 推荐值 | 说明 |
|---|---|---|
| 检测间隔 | 1~10 秒 | 根据服务重要性调整 |
| 失败阈值 | 2~5 次 | 太低误判，太高发现慢 |
| 超时时间 | 间隔的 1/3~1/2 | 给足够响应时间 |
| 恢复阈值 | 1~2 次 | 恢复应该比失败检测快 |

**最佳实践**：

- 被动 + 主动检查结合
- 健康检查接口要有适当的超时
- 服务器恢复采用渐进式权重
- 核心服务使用多级降级策略

记住：健康检查不是越严格越好，也不是越宽松越好。找到适合自己业务场景的平衡点，才是最好的配置。
