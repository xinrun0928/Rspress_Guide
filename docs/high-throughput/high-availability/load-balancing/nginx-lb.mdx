# Nginx 负载均衡配置与 upstream 模块

你刚刚接手了一个新项目，打开 nginx.conf，发现 upstream 块里密密麻麻写了几十个 server 配置，还有 `max_fails`、`fail_timeout`、`backup` 这些参数——完全看不懂。

这不是你一个人遇到的问题。Nginx 的 upstream 模块配置项很多，参数之间还有关联关系，很多工程师只知道最基本的 `server`，遇到问题就瞎试。

这篇文章，我带你把 Nginx upstream 彻底搞懂。从工作原理到配置细节，从健康检查到性能调优，让你面对任何 upstream 配置都能胸有成竹。

## 一、Nginx upstream 模块工作原理

### 1.1 反向代理架构

在说 upstream 之前，先搞清楚 Nginx 是怎么做反向代理的。

当客户端发送一个请求到 Nginx 时，Nginx 作为反向代理服务器，会把这个请求转发给后端的「上游」服务器，然后将上游服务器的响应返回给客户端。这个「上游」服务器列表，就是通过 upstream 模块定义的。

```bash
客户端请求
    ↓
Nginx（反向代理 / 负载均衡器）
    ↓ 转发请求
上游服务器 A / B / C
    ↓ 返回响应
Nginx
    ↓
客户端
```

### 1.2 连接复用：upstream keepalive

默认情况下，Nginx 每次转发请求都要和上游服务器建立一个新的 TCP 连接。建立 TCP 连接需要三次握手，销毁连接需要四次挥手——在高并发场景下，这个开销不可忽视。

Nginx 的 `keepalive` 指令解决了这个问题：建立少量「持久连接」，重复利用这些连接转发请求。

```nginx title="nginx.conf"
upstream backend {
    server 192.168.1.101:8080;
    server 192.168.1.102:8080;
    server 192.168.1.103:8080;
    
    # 启用连接池，缓存 32 个空闲连接
    keepalive 32;
}

server {
    listen 80;
    
    location /api/ {
        proxy_pass http://backend;
        
        # 关键配置：启用 HTTP keepalive
        proxy_http_version 1.1;
        proxy_set_header Connection "";
    }
}
```

**keepalive 的工作原理**：

- Nginx 维护一个空闲连接池
- 有请求时，从池中取出一个连接使用，用完放回池中
- 超过 `keepalive_timeout` 时间没有被使用，连接才会关闭
- 连接池满了，新请求会建立新连接

:::warning keepalive 的坑

`proxy_set_header Connection ""` 非常重要！如果不设置这个 Header，Nginx 会默认发送 `Connection: close`，导致连接不复用。

:::

### 1.3 upstream 的内存模型

理解 upstream 的内存模型，有助于你理解为什么某些配置会产生特定的行为。

Nginx 为每个 worker 进程维护独立的 upstream 连接池。当配置 `keepalive 32` 时，每个 worker 最多缓存 32 个空闲连接。对于 4 个 worker 的配置，总共最多有 128 个持久连接。

```bash
# 查看 Nginx worker 进程数
ps aux | grep nginx
# worker_processes * worker_connections = 最大并发连接数
```

## 二、所有负载均衡策略详解

### 2.1 轮询（Round Robin）

Nginx 默认的负载均衡策略就是轮询。当 upstream 块中没有指定其他策略时，所有请求会依次分发到每台服务器。

```nginx title="nginx.conf"
upstream backend {
    server 192.168.1.101:8080;
    server 192.168.1.102:8080;
    server 192.168.1.103:8080;
}
```

请求分发顺序：

```
请求 1 → 192.168.1.101
请求 2 → 192.168.1.102
请求 3 → 192.168.1.103
请求 4 → 192.168.1.101
...
```

轮询是「无权重」场景下的默认策略，如果所有服务器配置相同且请求复杂度相近，这是最均衡的分发方式。

### 2.2 加权轮询（Weighted Round Robin）

给每台服务器设置不同的权重，权重越高的服务器分到的请求越多。

```nginx title="nginx.conf"
upstream backend {
    server 192.168.1.101:8080 weight=5;
    server 192.168.1.102:8080 weight=3;
    server 192.168.1.103:8080 weight=2;
}
```

权重分发示意（假设总权重是 10）：

```
192.168.1.101: 50% 的请求
192.168.1.102: 30% 的请求
192.168.1.103: 20% 的请求
```

**权重设置的经验值**：

- 按 CPU 核心数比例设置
- 按内存大小比例设置
- 新上线的服务器设置较低权重，逐步提高

### 2.3 IP 哈希（ip_hash）

根据客户端 IP 地址计算哈希值，确保同一 IP 的请求始终打到同一台服务器。

```nginx title="nginx.conf"
upstream backend {
    ip_hash;
    server 192.168.1.101:8080;
    server 192.168.1.102:8080;
    server 192.168.1.103:8080;
}
```

**适用场景**：

- 服务端本地存储 Session 的场景
- 需要会话保持但不想用 Cookie 的场景

**缺陷**：

- 移动网络用户 IP 经常变化，会导致会话中断
- 公司内网用户 NAT 出口 IP 相同，可能导致负载不均

:::tip ip_hash 的限制

使用 `ip_hash` 时，不能使用 `backup` 参数。如果需要备用服务器，需要用其他方式实现高可用。

:::

### 2.4 最小连接数（least_conn）

优先将请求发送到当前连接数最少的服务器。

```nginx title="nginx.conf"
upstream backend {
    least_conn;
    server 192.168.1.101:8080;
    server 192.168.1.102:8080;
    server 192.168.1.103:8080;
}
```

**适用场景**：

- 请求处理时间差异较大的场景
- 长连接服务（WebSocket、gRPC、长轮询）
- 后端服务器性能不均的场景

**工作原理**：

Nginx 跟踪每台服务器的活跃连接数，选择连接数最少的那台。如果有多台连接数相同，按权重选择。

### 2.5 哈希（hash）

根据指定的 key 做哈希，支持任意 key，不仅仅是 IP 地址。

```nginx title="nginx.conf"
upstream backend {
    hash $request_uri consistent;
    server 192.168.1.101:8080;
    server 192.168.1.102:8080;
    server 192.168.1.103:8080;
}
```

**内置变量作为 key**：

| 变量 | 说明 |
|---|---|
| `$request_uri` | 完整请求 URI（包含参数） |
| `$uri` | 请求 URI（不含参数） |
| `$args` | 请求参数 |
| `$http_cookie` | Cookie 内容 |
| `$remote_addr` | 客户端 IP |

**`consistent` 参数**：启用一致性哈希，减少服务器增减时的重映射范围。

```nginx title="nginx.conf"
# 普通哈希：服务器变化时，75% 的请求会重新路由
hash $request_uri;

# 一致性哈希：服务器变化时，只有 1/n 的请求会重新路由
hash $request_uri consistent;
```

### 2.6 最少时间（least_time）

Nginx Plus（商业版）支持的策略，综合考虑连接数和响应时间。

```nginx title="nginx.conf"
upstream backend {
    least_time header;  # 或 least_time last_byte
    server 192.168.1.101:8080;
    server 192.168.1.102:8080;
}
```

- `least_time header`：计算从收到请求头到收到响应头的平均时间
- `least_time last_byte`：计算完整响应（包含 Body）的平均时间

## 三、server 指令详解

### 3.1 基本参数

```nginx title="nginx.conf"
upstream backend {
    server 192.168.1.101:8080 weight=5 max_fails=3 fail_timeout=10s;
}
```

| 参数 | 说明 | 默认值 |
|---|---|---|
| `address` | 服务器地址（IP 或域名） | — |
| `port` | 端口号 | 80 |
| `weight` | 权重 | 1 |
| `max_fails` | 最大失败次数，超过后标记为不可用 | 1 |
| `fail_timeout` | 失败超时时间 | 10s |
| `backup` | 标记为备用服务器 | — |
| `down` | 标记为永久不可用 | — |
| `max_conns` | 最大并发连接数 | 0（无限制） |

### 3.2 max_fails 和 fail_timeout：健康检查的核心

这两个参数是 Nginx 内置「被动」健康检查的关键。

```nginx title="nginx.conf"
server 192.168.1.101:8080 max_fails=3 fail_timeout=10s;
```

**工作流程**：

1. 正常情况下，Nginx 会把请求转发给这台服务器
2. 如果请求失败（连接超时、连接失败、HTTP 5xx），Nginx 记录一次失败
3. 10 秒内失败次数达到 3 次，标记为「不可用」
4. 接下来 10 秒内，不再向这台服务器发送请求
5. 10 秒后，重新尝试一次，如果成功，恢复为可用

```bash
# fail_timeout 语义解析
fail_timeout=10s 表示：
1. 10 秒内失败 3 次 → 标记为不可用
2. 不可用状态持续 10 秒
3. 10 秒后重新尝试
```

:::warning 失败的定义

只有「连接失败」或「超时」才算失败，返回 4xx 状态码不算失败。所以 Nginx 的被动检查是「不完善」的健康检查。

:::

### 3.3 backup：热备服务器

标记为备用服务器的节点，正常情况下不会收到请求，只有主服务器全部不可用时才会启用。

```nginx title="nginx.conf"
upstream backend {
    server 192.168.1.101:8080;
    server 192.168.1.102:8080;
    server 192.168.1.103:8080 backup;  # 备用服务器
}
```

**常见场景**：

- 主服务器所在的机房出现故障，切换到备用机房
- 新版本发布时，保留旧版本服务器作为回滚点

### 3.4 down：永久下线

标记为永久不可用的服务器，不会参与负载均衡。

```nginx title="nginx.conf"
upstream backend {
    server 192.168.1.101:8080;
    server 192.168.1.102:8080 down;  # 永久下线
    server 192.168.1.103:8080;
}
```

**使用场景**：

- 临时下线某台服务器进行维护
- 通过程序动态控制服务器的上下线

### 3.5 max_conns：连接数限制

限制单台服务器的最大并发连接数，防止某台服务器过载。

```nginx title="nginx.conf"
upstream backend {
    server 192.168.1.101:8080 max_conns=1000;
    server 192.168.1.102:8080 max_conns=500;
    server 192.168.1.103:8080 max_conns=500;
}
```

**使用场景**：

- 后端服务器处理能力不同，限制弱服务器的连接数
- 防止上游服务器连接数过多导致 OOM

:::warning max_conns 的注意事项

`max_conns` 只限制 Nginx 到这台服务器的连接数，不是整台服务器的总连接数。如果 Nginx 有多个 worker，实际并发可能超过 `max_conns`。

:::

## 四、健康检查配置

### 4.1 Nginx 内置的被动检查

Nginx 默认只有被动健康检查，即通过请求的「失败」来判断服务器是否可用。

```nginx title="nginx.conf"
upstream backend {
    server 192.168.1.101:8080 max_fails=3 fail_timeout=30s;
    server 192.168.1.102:8080 max_fails=3 fail_timeout=30s;
}
```

**优点**：无需额外配置，开箱即用。

**缺点**：

- 必须有请求才能检查
- 4xx 错误不算失败，可能导致误判
- 检查不及时，故障恢复慢

### 4.2 主动健康检查：nginx_upstream_check_module

Nginx 官方开源版本不支持主动健康检查，但淘宝开源的 `nginx_upstream_check_module` 提供了这个功能。

```nginx title="nginx.conf"
upstream backend {
    server 192.168.1.101:8080;
    server 192.168.1.102:8080;
    
    # 主动健康检查配置
    check interval=3000 rise=2 fall=3 timeout=1000 type=http;
    check_http_send "HEAD /health HTTP/1.0\r\n\r\n";
    check_http_expect_alive http_2xx http_3xx;
}
```

**参数说明**：

| 参数 | 说明 |
|---|---|
| `interval` | 检查间隔（毫秒） |
| `rise` | 连续成功次数，恢复为可用 |
| `fall` | 连续失败次数，标记为不可用 |
| `timeout` | 检查超时时间（毫秒） |
| `type` | 检查类型（tcp、http、ssl_hello、mysql、ajp） |

```nginx title="nginx.conf"
# TCP 连接检查
check interval=3000 rise=1 fall=2 timeout=1000 type=tcp;

# HTTP 检查（需要后端实现 /health 接口）
check interval=3000 rise=2 fall=3 timeout=1000 type=http;
check_http_send "HEAD /health HTTP/1.0\r\n\r\n";
check_http_expect_alive http_2xx;
```

### 4.3 健康检查接口设计

后端服务器需要提供一个健康检查接口：

```java title="HealthController.java"
@RestController
public class HealthController {
    
    @GetMapping("/health")
    public ResponseEntity<String> health() {
        // 检查数据库连接
        // 检查 Redis 连接
        // 检查磁盘空间
        
        if (isHealthy()) {
            return ResponseEntity.ok("OK");
        } else {
            return ResponseEntity.status(503).body("UNHEALTHY");
        }
    }
    
    private boolean isHealthy() {
        try {
            // 检查依赖的组件是否正常
            return db.isConnected() && redis.isConnected();
        } catch (Exception e) {
            return false;
        }
    }
}
```

```nginx title="nginx.conf"
# 检查健康接口返回 200 才算正常
check_http_expect_alive http_2xx;

# 或者允许 200 和 301 都算正常
check_http_expect_alive http_2xx http_3xx;
```

### 4.4 带权重的健康检查

```nginx title="nginx.conf"
upstream backend {
    server 192.168.1.101:8080 weight=5;
    server 192.168.1.102:8080 weight=3;
    server 192.168.1.103:8080 weight=2;

    check interval=5000 rise=1 fall=3 timeout=2000 type=http;
    check_http_send "HEAD /api/health HTTP/1.0\r\n\r\n";
    check_http_expect_alive http_2xx;

```

## 五、连接池与性能调优

### 5.1 keepalive 配置详解

```nginx title="nginx.conf"
upstream backend {
    server 192.168.1.101:8080;
    server 192.168.1.102:8080;
    server 192.168.1.103:8080;
    
    # 连接池大小
    keepalive 32;
    
    # 空闲连接超时
    keepalive_timeout 60s;
    
    # 单个连接最大请求数
    keepalive_requests 1000;
}
```

**keepalive 数量设置经验**：

- 计算公式：`keepalive = expected_connections * keepalive_time / request_time`
- 预期连接数：QPS × 平均响应时间
- 如果 QPS=10000，平均响应时间=100ms，预期连接数=1000
- `keepalive` 设为 100~200 较合适

### 5.2 proxy 配置优化

```nginx title="nginx.conf"
server {
    listen 80;
    
    location /api/ {
        proxy_pass http://backend;
        
        # 关键性能配置
        proxy_http_version 1.1;
        proxy_set_header Connection "";
        
        # 连接配置
        proxy_connect_timeout 5s;
        proxy_read_timeout 30s;
        proxy_send_timeout 30s;
        
        # 缓冲配置
        proxy_buffering on;
        proxy_buffer_size 4k;
        proxy_buffers 8 4k;
        
        # 错误处理
        proxy_next_upstream error timeout http_502;
    }
}
```

| 参数 | 说明 | 建议值 |
|---|---|---|
| `proxy_connect_timeout` | 连接上游服务器超时 | 5s |
| `proxy_read_timeout` | 读取响应超时 | 30s |
| `proxy_send_timeout` | 发送请求超时 | 30s |
| `proxy_buffering` | 启用缓冲 | on |
| `proxy_buffer_size` | 响应头缓冲大小 | 4k |

### 5.3 完整的 upstream 配置示例

```nginx title="nginx.conf"
# 负载均衡配置
upstream api_backend {
    # 负载均衡策略：加权轮询
    server 192.168.1.101:8080 weight=5 max_fails=3 fail_timeout=30s;
    server 192.168.1.102:8080 weight=3 max_fails=3 fail_timeout=30s;
    server 192.168.1.103:8080 weight=2 backup;
    
    # 连接池配置
    keepalive 100;
    keepalive_requests 10000;
    keepalive_timeout 60s;
}

upstream static_backend {
    server 192.168.1.201:80 weight=10;
    server 192.168.1.202:80 weight=10;
    server 192.168.1.203:80 backup;
    
    keepalive 50;
}

# HTTP 服务器配置
server {
    listen 80;
    server_name example.com;
    
    # API 代理
    location /api/ {
        proxy_pass http://api_backend;
        proxy_http_version 1.1;
        proxy_set_header Connection "";
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        
        proxy_connect_timeout 5s;
        proxy_read_timeout 30s;
        proxy_buffering on;
    }
    
    # 静态资源代理
    location /static/ {
        proxy_pass http://static_backend;
        proxy_http_version 1.1;
        proxy_set_header Connection "";
        
        # 静态资源缓存
        expires 7d;
        add_header Cache-Control "public, immutable";
    }
}

# TCP/UDP 四层负载均衡
stream {
    upstream mysql_backend {
        server 192.168.1.101:3306 weight=5;
        server 192.168.1.102:3306 weight=3;
        server 192.168.1.103:3306 backup;
        
        keepalive 20;
    }
    
    server {
        listen 3306;
        proxy_pass mysql_backend;
        proxy_timeout 10s;
        proxy_connect_timeout 5s;
    }
}
```

## 六、实战技巧

### 6.1 灰度发布

```nginx title="nginx.conf"
upstream stable_backend {
    server 192.168.1.101:8080;
    server 192.168.1.102:8080;
}

upstream canary_backend {
    server 192.168.1.201:8080;
}

server {
    listen 80;
    
    # 10% 的流量走 canary
    location / {
        set $backend stable_backend;
        if ($request_uri ~ "canary=true") {
            set $backend canary_backend;
        }
        # 简单哈希实现百分比分流
        if ($cookie_user_id ~* "^(.*[0-9])$") {
            set $percent $1;
            if ($percent ~ "[012345]") {
                set $backend canary_backend;
            }
        }
        
        proxy_pass http://$backend;
    }
}
```

## 七、面试追问：Nginx upstream 高频问题

### 7.1 问题一：Nginx upstream 和 keepalive 有什么关系？

**参考答案**：

upstream 模块中的 `keepalive` 指令定义了到上游服务器的空闲连接池大小。

没有 keepalive 时，每次请求都会建立新连接；有 keepalive 时，Nginx 维护一个连接池，复用空闲连接，减少 TCP 握手的开销。

关键配置点：

```nginx title="nginx.conf"
upstream backend {
    keepalive 32;  # 连接池大小
}

location / {
    proxy_pass http://backend;
    proxy_http_version 1.1;  # 必须用 HTTP/1.1
    proxy_set_header Connection "";  # 必须清空 Connection Header
}
```

如果不设置 `proxy_http_version 1.1` 和 `proxy_set_header Connection ""`，keepalive 不会生效。

### 7.2 问题二：max_fails 设置多少合适？

**参考答案**：

`max_fails` 的设置需要权衡「灵敏度」和「误判率」。

**设置太低的缺点**：

- 网络抖动时容易误判，导致正常服务器被标记为不可用
- 服务短暂 GC 停顿时容易被误判

**设置太高的缺点**：

- 故障发现不及时，用户请求失败率高
- 健康检查恢复时间长

**经验值**：

| 场景 | max_fails | fail_timeout |
|---|---|---|
| 高可用要求高 | 1~2 | 10s~30s |
| 平衡场景 | 3~5 | 30s~60s |
| 追求稳定 | 5~10 | 60s~120s |

如果是核心业务，建议设置较低的 `max_fails` 和较短的 `fail_timeout`，配合主动健康检查实现快速故障检测。

### 7.3 问题三：upstream 的连接数和后端服务器连接数是什么关系？

**参考答案**：

upstream 的 `max_conns` 限制的是 **Nginx 到单台上游服务器的连接数**，不是上游服务器的「总连接数」。

```nginx title="nginx.conf"
upstream backend {
    server 192.168.1.101:8080 max_conns=1000;
}
```

这表示 Nginx 最多同时向 192.168.1.101:8080 发起 1000 个连接。

如果 Nginx 有 4 个 worker，每个 worker 有 1000 个连接池，总共最多 4000 个连接。

如果 Nginx 后面还有其他 Nginx 或代理，实际并发可能超过 `max_conns` 设置的值。

### 7.4 问题四：如何实现零 downtime 部署？

**参考答案**：

零 downtime 部署的关键是「渐进切换」：

**方案一：灰度权重**

```nginx title="nginx.conf"
upstream backend {
    # 第一阶段：10% 流量到新版本
    server 192.168.1.101:8080;  # 旧版本
    server 192.168.1.201:8080 weight=1;  # 新版本
}
```

逐步调整权重：

```nginx title="nginx.conf"
# 第二阶段：50% 流量到新版本
upstream backend {
    server 192.168.1.101:8080;
    server 192.168.1.201:8080 weight=1;
}
```

**方案二：backup 服务器**

```nginx title="nginx.conf"
upstream backend {
    # 先上线新版本作为 backup
    server 192.168.1.101:8080;
    server 192.168.1.201:8080 backup;
}

# 验证通过后，切换权重
upstream backend {
    server 192.168.1.101:8080;
    server 192.168.1.201:8080;
}
```

**方案三：优雅关闭**

```bash
# 只关闭 worker，不接受新请求，但保持现有连接
nginx -s quit

# 向某个 worker 发送信号，只关闭该 worker 的监听
kill -QUIT <worker_pid>
```

## 总结

Nginx upstream 模块是高性能负载均衡的核心配置。

**关键配置要点**：

- **负载均衡策略**：`round_robin`（默认）、`ip_hash`、`least_conn`、`hash`
- **服务器参数**：`weight`、`max_fails`、`fail_timeout`、`backup`、`down`、`max_conns`
- **连接复用**：`keepalive` + `proxy_http_version 1.1` + `proxy_set_header Connection ""`
- **健康检查**：被动检查（max_fails/fail_timeout）或主动检查（nginx_upstream_check_module）

**性能调优核心**：

- keepalive 值 = QPS × 响应时间 / worker 数
- proxy_buffering 减少阻塞
- 合理的超时时间避免资源浪费

**高可用设计**：

- 至少部署 2 台以上 upstream 服务器
- 合理使用 backup 作为热备
- 配合主动健康检查实现快速故障检测

记住：Nginx 的 upstream 配置看似简单，但里面每一个参数都有其存在的意义。理解原理，才能游刃有余。
