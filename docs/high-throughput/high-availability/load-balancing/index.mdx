# 负载均衡：系统入口的高可用基石

双十一零点，你的服务扛住了 50 万 QPS 的流量冲击。用户下单流畅，页面秒开。你可能不知道，在你看不见的地方，负载均衡器正在以毫秒级的速度，把每一请求精准地分发到 thousands 台后端服务器——

有的服务器 CPU 打满了，它就少分一点；有的服务器刚刚恢复，它就逐步加量；有的服务器彻底宕机，它在 3 秒内完成切换，把流量无缝转移到健康的节点上。

这就是负载均衡的力量。

但如果配置不当呢？你可能遇到过：
- 某台服务器莫名 CPU 飙升，其他机器却「躺平」
- 服务刚上线就触发雪崩，一台故障拖垮整个集群
- 灰度发布时流量忽多忽少，无法精准控制

这些问题，根源往往在负载均衡的设计和配置上。

这一模块，我带你把负载均衡彻底搞懂——从四层与七层的原理对比，到 Nginx 的 upstream 配置；从轮询、哈希等基础算法，到一致性哈希的缓存友好设计；从 DNS 负载均衡的全局调度，到 Keepalived + VRRP 的高可用架构。

:::tip 学习目标

学完这一模块，你应该能够：
- 理解 L4 和 L7 负载均衡的区别与适用场景
- 掌握 Nginx upstream 的所有配置项
- 熟练使用各种负载均衡算法
- 设计一套完整的负载均衡高可用架构

:::

## 模块概述

负载均衡（Load Balancing）是分布式系统的第一道关口，它决定了用户请求如何被分发到后端服务器。

<Badge text="核心价值" type="tip" /> 负载均衡的核心价值体现在三个维度：

| 维度 | 说明 | 解决的问题 |
|---|---|---|
| **高可用** | 故障自动切换 | 单机故障不影响整体服务 |
| **高性能** | 流量均匀分发 | 充分利用每台服务器资源 |
| **可扩展** | 水平扩展能力 | 业务增长时轻松扩容 |

没有负载均衡的系统，是脆弱的——单机故障意味着服务中断。不懂负载均衡的工程师，是危险的——配置错误可能导致雪崩或资源浪费。

## 核心概念与公式

### 核心概念图谱

```
负载均衡架构
├── 架构层
│   ├── L4 负载均衡（传输层，NAT/DSR 模式）
│   └── L7 负载均衡（应用层，反向代理）
├── 算法层
│   ├── 静态算法（轮询、随机、加权）
│   └── 动态算法（最小连接数、一致性哈希）
├── 分发层
│   ├── DNS 负载均衡（全局调度）
│   ├── Nginx upstream（应用层分发）
│   └── 客户端负载均衡（微服务间调用）
└── 高可用层
    ├── 健康检查（主动/被动）
    └── 故障切换（Keepalived + VRRP）
```

### 关键公式

**故障发现时间公式**：

```
故障发现时间 = 检测间隔 × 失败阈值
```

:::tip 公式解读

假设配置 `interval=3000ms, fall=3`，意味着：
- 故障发现时间 = 3 秒 × 3 = **9 秒**
- 健康检查开销 = 1/3 秒 ≈ 0.33 QPS/服务器

在设计健康检查参数时，需要在「发现速度」和「检查开销」之间权衡。

:::

**连接池大小估算公式**：

```
keepalive = QPS × 平均响应时间 / worker 数量
```

```bash
# 示例计算
QPS = 10000
平均响应时间 = 100ms = 0.1s
worker 数量 = 8

keepalive = 10000 × 0.1 / 8 = 125

# 建议设置 keepalive = 150~200
```

**一致性哈希再平衡公式**：

```
节点变化时重新路由的请求比例 ≈ 1 / n（一致性哈希）
节点变化时重新路由的请求比例 ≈ 1（普通哈希）
```

:::warning n 是节点数量

当节点数量较少时（如 3~5 台），一致性哈希的优势不明显。当节点数量达到数十甚至数百台时，一致性哈希能显著减少缓存雪崩的风险。

:::

## 实战配置清单

### Nginx upstream 核心配置检查清单

```nginx title="nginx.conf"
upstream backend {
    # 1. 负载均衡策略
    # 轮询（默认）、加权轮询、最小连接数、IP 哈希、一致性哈希
    # least_conn;
    # ip_hash;
    # hash $request_uri consistent;

    # 2. 服务器配置
    server 192.168.1.101:8080 weight=5 max_fails=3 fail_timeout=30s;
    server 192.168.1.102:8080 weight=3 max_fails=3 fail_timeout=30s;
    server 192.168.1.103:8080 backup;

    # 3. 连接复用
    keepalive 100;
    keepalive_timeout 60s;
    keepalive_requests 10000;
}
```

### Keepalived 高可用核心配置检查清单

```bash title="keepalived.conf"
vrrp_instance VI_1 {
    state BACKUP          # 推荐设置为 BACKUP，避免脑裂
    interface eth0
    virtual_router_id 51  # 同一集群必须相同
    priority 100         # Master > Backup
    advert_int 1         # 广告间隔（秒）
    nopreempt            # 非抢占模式

    # 虚拟 IP
    virtual_ipaddress {
        192.168.1.100/24 dev eth0
    }

    # 单播模式（生产推荐）
    unicast_peer {
        192.168.1.12
    }
}

vrrp_script check_nginx {
    script "/etc/keepalived/check_nginx.sh"
    interval 2
    weight -20
}
```

## 总结

负载均衡是分布式系统的「交通枢纽」，它的重要性怎么强调都不为过。

**本模块的核心知识点**：

| 维度 | 关键点 | 面试高频 |
|---|---|---|
| 原理层 | L4 看端口，L7 看内容 | ✅ L4 vs L7 区别 |
| 算法层 | 一致性哈希解决缓存雪崩 | ✅ 一致性哈希原理与缺陷 |
| 配置层 | Nginx upstream 核心参数 | ✅ max_fails、keepalive 配置 |
| 高可用层 | Keepalived + VRRP 故障切换 | ✅ 脑裂预防、抢占模式 |

**选型决策树**：

```
流量入口选型：
├── 非 HTTP 协议（MySQL、Redis）→ L4 负载均衡
└── HTTP 协议 →
        ├── 追求极致性能 → L4 + L7 组合
        └── 需要精细化路由 → L7（Nginx/Envoy）

内部调用选型：
├── Spring Cloud 项目 → Spring Cloud LoadBalancer
└── 多语言微服务 → Envoy（客户端 + 服务端代理）
```

**行动建议**：

1. **理解原理**：先搞懂 L4/L7 的本质区别，不要死记硬背
2. **掌握配置**：Nginx upstream 配置是实操基础，必须能看懂任何配置
3. **理解算法**：一致性哈希是分布式系统的核心算法，必须深入理解
4. **设计高可用**：Keepalived + VRRP 是经典方案，记住核心配置项

记住：**负载均衡不是「配置一下就完了」，它需要持续监控和调优**。没有最好的配置，只有最适合的配置。
