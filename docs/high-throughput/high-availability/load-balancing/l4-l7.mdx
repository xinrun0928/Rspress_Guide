# 四层负载均衡（L4）vs 七层负载均衡（L7）

凌晨三点，你被一条告警叫醒：「用户反馈下单页面打不开！」你打开监控大屏，发现请求全打到了同一台服务器，而其他三台服务器的 CPU 使用率还不到 10%。原来是负载均衡器把流量全部分发到了某台「明星节点」，其他节点却在「躺平」。

这不是个案。在分布式系统架构中，负载均衡是连接用户请求与后端服务的第一道关口。但同样是负载均衡，有的方案工作在网络层，有的却工作在应用层——这就是我们要聊的 L4 和 L7 负载均衡。

面试官问你「L4 和 L7 负载均衡有什么区别」，如果你只答出「一个工作在四层，一个工作在七层」，那只能拿到基础分。真正优秀的候选人，能从原理、性能、功能、适用场景多个维度，把这个问题讲透彻。

---

## 一、从 OSI 模型说起：为什么叫四层和七层？

要理解 L4 和 L7，首先得搞清楚 OSI 七层模型。这个模型把网络通信划分为七层，从下到上分别是：物理层、数据链路层、网络层、传输层、会话层、表示层、应用层。

<Badge text="OSI 七层" type="tip" /> 记住这个分层很重要，因为负载均衡工作在哪一层，决定了它能看到多少信息，以及能做什么事情。

| OSI 层 | 层名 | 典型协议 | 负载均衡能看到的信息 |
|---|---|---|---|
| 第 7 层 | 应用层 | HTTP、HTTPS、DNS、FTP | 完整请求内容（URL、Header、Cookie、Body） |
| 第 6 层 | 表示层 | TLS/SSL、JPEG、ASCII | 加密后的元数据 |
| 第 5 层 | 会话层 | NetBIOS、SOCKS | 会话状态 |
| **第 4 层** | **传输层** | **TCP、UDP** | **IP + 端口、连接状态** |
| 第 3 层 | 网络层 | IP、ICMP、路由 | IP 地址、路由信息 |
| 第 2 层 | 数据链路层 | Ethernet、ARP | MAC 地址 |
| 第 1 层 | 物理层 | 光纤、双绞线 | 物理信号 |

四层负载均衡（L4）工作在传输层，基于 IP + 端口进行流量分发。它感知不到 HTTP 的 URL、Header 或者 Cookie——在它眼里，所有请求都是 TCP 连接或 UDP 数据报。

七层负载均衡（L7）工作在应用层，能看到完整的 HTTP 请求内容。这意味着它可以根据 URL 路径、请求参数、Header 信息甚至 Cookie 来决定如何分配流量。

:::tip 记忆口诀

L4 看「端口」，L7 看「内容」。四层做「粗筛」，七层做「细选」。

:::

## 二、四层负载均衡：速度快，但眼神不太好

### 2.1 工作原理

四层负载均衡的核心是 **NAT（Network Address Translation，网络地址转换）** 或者 **DSR（Direct Server Return，直接服务器返回）** 两种模式。

**NAT 模式**：客户端请求先到达负载均衡器，负载均衡器修改数据包的目标 IP 和端口，转发给后端服务器。后端服务器的响应先回到负载均衡器，再由负载均衡器返回给客户端。这个过程中，负载均衡器会修改数据包的 IP 地址，所以叫「NAT」。

```bash
# NAT 模式数据包流转示意
客户端 -> [SrcIP:Client, DstIP:LB] -> 负载均衡器 -> [SrcIP:LB, DstIP:Backend] -> 后端服务器
后端服务器 -> [SrcIP:Backend, DstIP:LB] -> 负载均衡器 -> [SrcIP:LB, DstIP:Client] -> 客户端
```

**DSR 模式**：客户端请求到达负载均衡器后，直接转发到后端服务器（不修改目标 IP），但负载均衡器会把响应路径「短路」——让后端服务器直接返回给客户端，省去回程的负载均衡器。这种模式也叫「三角传输」。

```bash
# DSR 模式数据包流转示意
客户端 -> [DstIP:VIP] -> 负载均衡器 -> [DstIP:VIP, 但发送到真实 Backend IP] -> 后端服务器
后端服务器直接返回 -> [SrcIP:VIP] -> 客户端（绕过负载均衡器）
```

### 2.2 核心优势

**快。** 这是四层负载均衡最大的优势。由于工作在传输层，不需要解析应用层协议，CPU 开销极小。很多硬件负载均衡器（如 F5、A10）都能达到线速转发（接近物理带宽极限）。

**稳定。** 不涉及应用层协议解析，所以不会因为某个 HTTP 头的格式问题而出现解析错误。协议无关性意味着它可以转发任何基于 TCP/UDP 的流量——HTTP 当然可以，Redis、MySQL、MongoDB 也都可以。

**简单。** 配置相对简单，只需要指定后端服务器的 IP 和端口即可。

### 2.3 核心劣势

**眼神不好。** 只能看到 IP 和端口，看不到 URL、Cookie 或者其他业务层面的信息。这意味着：

- 无法根据 URL 路径做路由（比如 `/api/*` 走 A 组服务器，`/static/*` 走 B 组服务器）
- 无法根据 Cookie 实现会话保持（Sticky Session）
- 无法做基于 Header 的流量染色（比如根据 `X-Env: blue` 发送到 blue 环境）
- 无法修改请求内容（比如添加或删除 Header）

### 2.4 适用场景

| 场景 | 推荐原因 |
|---|---|
| 数据库读写分离 | 数据库协议不是 HTTP，L4 通用性强 |
| Redis 集群访问 | 基于 TCP 转发，无需感知应用层 |
| 游戏服务器 | UDP 协议场景，L4 支持性好 |
| 对性能要求极高的内部服务 | 追求极致转发速度 |
| 流量单纯，不需要复杂路由 | 只需要均匀分发 |

## 三、七层负载均衡：眼神好，但慢一点

### 3.1 工作原理

七层负载均衡的核心是 **代理（Proxy）**。客户端实际上是在和负载均衡器通信，而不是直接和后端服务器通信。负载均衡器接收到完整请求后，根据规则决定转发到哪个后端服务器。

根据代理方式的不同，七层负载均衡分为两种模式：

**正向代理模式**：客户端明确知道负载均衡器的存在，主动向代理发起请求。浏览器配置代理就是这种模式。

**反向代理模式**：客户端不知道后端服务器的存在，以为负载均衡器就是真正的服务器。Nginx 反向代理就是这种模式，用户完全感知不到代理的存在。

```bash
# 反向代理模式示意
客户端 -> [以为在访问 nginx.com] -> Nginx（负载均衡器）-> [解析 HTTP，发现 /api/user] -> 后端服务器 A
客户端 -> [以为在访问 nginx.com] -> Nginx（负载均衡器）-> [解析 HTTP，发现 /static/logo.png] -> 后端服务器 B（静态资源服务器）
```

### 3.2 核心优势

**智能路由。** 这是七层负载均衡的杀手锏。能看到的请求信息包括：

- URL 路径和查询参数
- HTTP Header（Host、User-Agent、X-Forwarded-For 等）
- Cookie 和 Session
- 请求方法（GET、POST、PUT、DELETE）
- HTTP 版本

基于这些信息，可以实现：

- **路径路由**：`/api/*` → 后端集群 A，`/admin/*` → 后端集群 B
- **A/B 测试**：根据 Cookie 或 Header 分配到不同版本
- **灰度发布**：按百分比或用户 ID 分配流量
- **WAF（Web 应用防火墙）**：检测恶意请求并拦截
- **请求重写**：修改 URL、添加或删除 Header

**缓存优化。** 可以缓存静态资源，减少后端服务器压力。

### 3.3 核心劣势

**慢一点。** 需要解析完整的 HTTP 请求和响应，CPU 开销比四层大。在高并发场景下，CPU 可能成为瓶颈。

**资源消耗。** 需要维护大量的代理连接，内存占用比四层高。

**复杂度。** 配置更复杂，需要理解 HTTP 协议的各种细节。

**协议限制。** 默认只支持 HTTP/HTTPS，其他协议需要额外配置或不支持。

### 3.4 适用场景

| 场景 | 推荐原因 |
|---|---|
| 微服务网关 | 需要根据 API 路径路由到不同服务 |
| CDN 源站 | 需要缓存和加速静态资源 |
| Web 应用防火墙 | 需要检测和拦截恶意请求 |
| A/B 测试 / 灰度发布 | 需要精细化的流量控制 |
| SSL 终结 | 集中管理证书，统一加解密 |

## 四、L4 vs L7：全方位对比

这是面试的重点。把下面的表格记住，比干巴巴地背概念要有效得多。

### 4.1 核心能力对比

| 维度 | 四层负载均衡（L4） | 七层负载均衡（L7） |
|---|---|---|
| OSI 层 | 传输层（IP + 端口） | 应用层（HTTP/HTTPS） |
| 工作模式 | NAT / DSR | 反向代理 |
| 转发速度 | 极快，接近线速 | 较快，有解析开销 |
| 路由依据 | IP + 端口 | URL、Header、Cookie、方法等 |
| 协议支持 | 任意 TCP/UDP | 主要 HTTP/HTTPS |
| 会话保持 | 基于 IP 或端口 | 基于 Cookie、Session |
| SSL 终结 | 不支持 | 支持 |
| 请求修改 | 不支持 | 支持（重写 Header、URL） |
| 缓存能力 | 无 | 有 |
| WAF 能力 | 无 | 有 |
| 配置复杂度 | 低 | 中高 |
| 硬件成本 | 中高（专用硬件） | 中（软件方案成熟） |

### 4.2 性能对比

在实际测试中，L4 和 L7 的性能差距主要体现在 **并发连接数** 和 **每秒请求数（RPS）** 上：

| 指标 | L4 典型值 | L7 典型值 | 说明 |
|---|---|---|---|
| 单机并发连接数 | 百万级 | 十万级 | 七层需要维护更多连接状态 |
| 单机 RPS | 百万级 | 十万~百万级 | 取决于 CPU 和 HTTP 处理效率 |
| 延迟增加 | < 1ms | 1~5ms | 七层解析 HTTP 头部的开销 |
| CPU 占用 | 低 | 中高 | 七层需要解析协议 |
| 内存占用 | 低 | 中高 | 七层需要缓存更多连接信息 |

### 4.3 选型决策矩阵

面试时，面试官可能会问：「那你说说，实际项目里什么时候用 L4，什么时候用 L7？」

下面是一个实用的决策矩阵：

```
需要精细化流量控制（路由、A/B测试、灰度）？
    └── 是 ──> 选 L7（nginx、Traefik、Envoy）
    └── 否 ──> 流量是 HTTP，但追求极致性能？
              └── 是 ──> L4 + L7 组合（L4 做入口，L7 做细筛）
              └── 否 ──> 流量是 MySQL/Redis 等非 HTTP 协议？
                      └── 是 ──> 选 L4
                      └── 否 ──> 流量单纯，只做分发？
                              └── 是 ──> 选 L4
```

### 4.4 实际架构：L4 + L7 组合

在大规模生产环境中，「L4 + L7」组合是常见的架构模式：

```bash
用户请求
    ↓
[L4 负载均衡器] ← 第一道关卡，高并发入口
    ↓ （四层快速分发）
[L7 负载均衡器 / API 网关] ← 精细化路由
    ↓
[微服务集群 A] / [微服务集群 B] / [静态资源]
```

这种架构的优势：

- **L4 做入口**：扛住海量并发，快速分发到多个 L7 节点
- **L7 做路由**：在 L4 后面做精细化控制，实现灰度、A/B 测试等
- **高可用**：任何一层的节点故障，都不会导致服务不可用

:::warning 组合架构的坑

L4 + L7 组合会增加一跳（Hop），每次转发都有延迟。在对延迟极其敏感的场景下（如高频交易），需要评估额外延迟是否可以接受。

:::

## 五、Nginx 的 L4 vs L7 配置对比

作为 Java 工程师，你可能经常和 Nginx 打交道。Nginx 本身就支持 L4 和 L7 两种模式，理解它们的配置差异很重要。

### 5.1 七层配置（最常用）

这是我们最熟悉的 Nginx 反向代理配置：

```nginx title="nginx.conf（HTTP 七层负载均衡）"
upstream backend {
    server 192.168.1.101:8080 weight=5;
    server 192.168.1.102:8080 weight=3;
    server 192.168.1.103:8080 weight=2;
    server 192.168.1.104:8080 backup;  # 备用服务器
}

server {
    listen 80;
    server_name example.com;

    # 七层代理：根据 URL 路径路由
    location /api/ {
        proxy_pass http://backend;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }

    # 静态资源直接返回
    location /static/ {
        root /data/www;
        expires 7d;
    }
}
```

### 5.2 四层配置（stream 模块）

Nginx 从 1.9.0 开始支持 `stream` 模块，可以做 TCP/UDP 的四层负载均衡：

```nginx title="nginx.conf（TCP 四层负载均衡）"
stream {
    upstream mysql_cluster {
        server 192.168.1.201:3306 weight=5;
        server 192.168.1.202:3306 weight=3;
        server 192.168.1.203:3306 backup;
    }

    upstream redis_cluster {
        server 192.168.1.211:6379;
        server 192.168.1.212:6379;
    }

    # MySQL 代理
    server {
        listen 3306;
        proxy_pass mysql_cluster;
        proxy_timeout 10s;
        proxy_connect_timeout 5s;
    }

    # Redis 代理
    server {
        listen 6379;
        proxy_pass redis_cluster;
    }
}
```

:::tip 配置区别

七层配置用 `http {}` 块，四层配置用 `stream {}` 块。七层可以配置 `location` 做路由，四层只能基于 IP + 端口转发。

:::

## 六、面试追问：L4 vs L7 相关高频问题

### 6.1 问题一：为什么 L4 比 L7 快？

**参考答案**：

L4 快的根本原因是 **它做的事情更少**。

四层负载均衡只需要做两件事：修改目标 IP/端口，然后转发数据包。它不需要理解数据包里的内容——在它眼里，一个 TCP SYN 包和一个 HTTP GET 请求没有任何区别，都是一段二进制数据。

七层负载均衡则需要：接收完整的请求（可能包含几 KB 甚至几 MB 的 Body），解析 HTTP 协议（状态行、Header、Body），根据规则决定路由，然后才能转发。这个过程中，CPU 要做字符串解析、协议栈处理、内存拷贝等工作。

类比一下：L4 像快递员，只看收件人地址（IP + 端口），不看里面装的是什么；L7 像海关工作人员，要开箱检查内容（HTTP Header、Body），判断货物类型，然后决定走哪个通道。

### 6.2 问题二：什么场景下 L4 无法替代 L7？

**参考答案**：

主要有三类场景：

**第一类：需要基于 URL 路由**

如果你的微服务架构中，不同的 API 路径需要路由到不同的后端服务，L4 无法实现这个需求。例如 `/order/*` 路由到订单服务，`/user/*` 路由到用户服务，这必须由 L7 来做。

**第二类：需要 SSL 终结**

如果希望在负载均衡器上统一处理 HTTPS 证书，L7 可以直接加解密；L4 无法解析加密流量，只能透传。

**第三类：需要流量分析和修改**

WAF（Web 应用防火墙）需要在应用层检测恶意请求，DDoS 防护需要在应用层识别攻击特征，这些都离不开 L7 的深度包检测能力。

### 6.3 问题三：L4 + L7 组合架构的优势和劣势？

**参考答案**：

**优势**：

- **分层治理**：L4 处理海量并发入口，L7 处理精细化路由，各司其职
- **可扩展性**：可以在 L4 后面水平扩展 L7 节点
- **灵活性**：不同 L7 节点可以承载不同的路由策略

**劣势**：

- **延迟增加**：多一跳意味着多一份延迟
- **架构复杂**：节点更多，排查问题更困难
- **成本增加**：需要维护两套负载均衡体系

### 6.4 问题四：如何选择负载均衡方案？

**参考答案**：

没有银弹，选择取决于业务场景：

| 场景 | 推荐方案 |
|---|---|
| 高并发 HTTP 入口，需要灰度/A/B测试 | L7（nginx、Envoy） |
| MySQL、Redis 等数据库集群 | L4（硬件 F5、LVS） |
| 微服务内部 RPC 调用 | L7（ Envoy、Spring Cloud Gateway） |
| 游戏服务器、实时通信 | L4（UDP 支持） |
| 极致性能要求 | L4 或 L4 + L7 组合 |
| 需要 WAF/防护 | L7 |

## 七、留给你的思考题

### 题目一：架构演进的选择

假设你的系统从单机架构演进到微服务架构，流量从 1 万 QPS 增长到 100 万 QPS。你的负载均衡方案应该如何演进？

**提示**：考虑不同阶段的瓶颈在哪里，是计算密集型还是 IO 密集型？

### 题目二：混合协议的挑战

如果你的系统中同时存在 HTTP API 调用和 gRPC 调用，应该如何设计负载均衡方案？

**提示**：gRPC 基于 HTTP/2 协议，有些 L7 负载均衡器对 HTTP/2 支持不完善，需要评估兼容性。

## 总结

L4 和 L7 负载均衡各有优劣，没有绝对的优劣之分，只有场景的匹配程度。

**L4 的核心特点**：

- 工作在传输层，基于 IP + 端口分发
- 速度快，接近线速转发
- 协议无关，支持任何 TCP/UDP 流量
- 无法感知应用层内容，路由能力有限

**L7 的核心特点**：

- 工作在应用层，基于 HTTP/HTTPS 分发
- 可以看到完整的请求内容，路由能力强大
- 支持 SSL 终结、WAF、缓存等功能
- 性能开销比 L4 大

**实战建议**：

- 单纯做流量分发，追求极致性能：选 L4
- 需要精细化路由，灰度、A/B 测试：选 L7
- 大规模生产环境：L4 + L7 组合

记住：负载均衡是架构的第一道关卡，选对了，后面的路顺很多；选错了，改起来伤筋动骨。
