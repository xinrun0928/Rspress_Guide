# 客户端负载均衡：Ribbon、Spring Cloud LoadBalancer

你写了一段 Java 代码，调用另一个微服务的 API。代码里写的是：

```java
RestTemplate restTemplate = new RestTemplate();
String result = restTemplate.getForObject("http://user-service/api/users", String.class);
```

看起来没问题。但你有没有想过：请求是如何发送到 `user-service` 的后端实例的？Ribbon 怎么知道 `user-service` 有哪些 IP？它用的什么策略选了一台服务器？

这就是客户端负载均衡（Client-Side Load Balancing）。

和 Nginx、LVS 等服务端负载均衡不同，客户端负载均衡是在调用方（Consumer）一侧完成的。服务消费者自己维护一份服务器列表，根据负载均衡算法选择目标服务器。

这篇文章，我带你把客户端负载均衡彻底搞懂，重点讲解 Ribbon 和 Spring Cloud LoadBalancer。

## 一、客户端负载均衡 vs 服务端负载均衡

### 1.1 服务端负载均衡

服务端负载均衡是我们最熟悉的模式：

```
用户请求 → [Nginx/LVS] → 后端服务器 A/B/C
```

Nginx 是独立的进程运行在负载均衡器上，所有的请求都要经过它。客户端不知道有多少台后端服务器，它只知道 Nginx 的地址。

**特点**：

- 负载均衡器是独立的中间层
- 客户端只连接负载均衡器
- 集中管控，配置方便
- 单点（需要高可用）

### 1.2 客户端负载均衡

客户端负载均衡把负载均衡的逻辑下沉到客户端：

```
用户请求 → [客户端] → 后端服务器 A/B/C
              ↑
         负载均衡逻辑
        （在客户端进程内）
```

服务消费者自己维护服务器列表，根据负载均衡算法选择目标服务器。

**特点**：

- 负载均衡逻辑在客户端代码里
- 客户端知道所有后端服务器地址
- 无需额外的负载均衡器
- 更适合微服务架构

### 1.3 对比表格

| 维度 | 服务端负载均衡 | 客户端负载均衡 |
|---|---|---|
| 均衡器位置 | 独立中间层 | 客户端进程内 |
| 服务器列表 | 负载均衡器持有 | 客户端持有 |
| 配置 | 集中配置 | 分散配置（但通常通过配置中心统一） |
| 网络跳数 | 多一跳 | 直接到后端 |
| 服务发现 | 需要配合 | 天然集成 |
| 典型产品 | Nginx、LVS、F5 | Ribbon、Spring Cloud LoadBalancer |
| 适合场景 | 外部流量入口 | 微服务间调用 |

:::tip 为什么微服务推荐客户端负载均衡

在微服务架构中，服务实例数量是动态变化的（扩容、缩容、故障）。如果用服务端负载均衡，需要频繁更新负载均衡器的配置。使用客户端负载均衡，配合服务注册中心（如 Eureka、Nacos），实例列表可以自动同步。

:::

## 二、Ribbon：Netflix 的客户端负载均衡器

### 2.1 Ribbon 概述

Ribbon 是 Netflix 开源的客户端负载均衡器，后被 Spring Cloud 集成，成为 Spring Cloud Netflix 的一部分。

Ribbon 的核心组件：

| 组件 | 说明 |
|---|---|
| ServerList | 服务实例列表（静态或动态） |
| ServerListFilter | 过滤服务实例（如可用区过滤） |
| IRule | 负载均衡策略 |
| LoadBalancer | 协调各组件，执行负载均衡 |
| IPing | 健康检查（Ribbon 2.x 之后不再内置） |

```
Ribbon 架构
┌─────────────────────────────────────────┐
│              LoadBalancer               │
├─────────────────────────────────────────┤
│  ServerList  →  ServerListFilter  →  IRule  │
│  (获取列表)      (过滤)               (选择服务器)  │
└─────────────────────────────────────────┘
```

### 2.2 Ribbon 的负载均衡策略

Ribbon 内置了多种负载均衡策略：

```java
// Ribbon 内置策略
public interface IRule {
    public Server choose(Object key);
}

// 1. RoundRobinRule：轮询
// 2. RandomRule：随机
// 3. WeightedResponseTimeRule：加权响应时间
// 4. BestAvailableRule：选择最小并发数的服务器
// 5. AvailabilityFilteringRule：过滤掉故障和并发高的服务器
// 6. ZoneAvoidanceRule：区域感知（默认）
// 7. RetryRule：带重试的策略
```

**ZoneAvoidanceRule（默认策略）**：

```java title="ZoneAvoidanceRule.java"
public class ZoneAvoidanceRule extends PredicateBasedRule {
    
    private final CompositePredicate compositePredicate = 
        CompositePredicate.withPredicate(
            new ZoneAvoidancePredicate(this)
        )
        .addFallbackPredicate(
            new AvailabilityPredicate(this)
        )
        .build();
    
    @Override
    public Server choose(Object key) {
        List<Server> servers = getPredicate()
            .getEligibleServers(getLoadBalancer().getAllServers());
        
        if (servers == null || servers.isEmpty()) {
            return null;
        }
        
        // 使用 ZoneAwareLoadBalancer 的 chooseZoneServer 方法
        return getEligibleServer(servers);
    }
}
```

### 2.3 Ribbon 配置详解

**方式一：配置文件**

```bash title="application.yml"
# 指定服务名
user-service:
  ribbon:
    # 负载均衡策略
    NFLoadBalancerRuleClassName: com.netflix.loadbalancer.RandomRule
    
    # 服务列表刷新间隔
    ServerListRefreshInterval: 30000
    
    # 连接超时
    ConnectTimeout: 3000
    
    # 读取超时
    ReadTimeout: 5000
    
    # 最大重试次数
    MaxAutoRetries: 3
    
    # 对同一服务器的最大重试次数
    MaxAutoRetriesNextServer: 2
```

**方式二：Java 配置类**

```java title="RibbonConfig.java"
@Configuration
public class RibbonConfig {
    
    @Bean
    public IRule ribbonRule() {
        // 使用加权响应时间策略
        return new WeightedResponseTimeRule();
    }
    
    @Bean
    public ServerList<Server> serverList() {
        // 可以自定义服务列表来源
        return new ConfigurationBasedServerList();
    }
}
```

**全局配置**：

```bash title="application.yml"
ribbon:
  # 全局默认策略
  NFLoadBalancerRuleClassName: com.netflix.loadbalancer.RandomRule
  ConnectTimeout: 3000
  ReadTimeout: 5000
```

### 2.4 Ribbon 与 RestTemplate 集成

Ribbon 最常见的用法是和 RestTemplate 集成，实现服务调用：

```java title="RibbonClientApplication.java"
@SpringBootApplication
@EnableEurekaClient
public class RibbonClientApplication {
    
    @Bean
    @LoadBalanced  // 关键注解：启用 Ribbon 负载均衡
    public RestTemplate restTemplate() {
        return new RestTemplate();
    }
    
    public static void main(String[] args) {
        SpringApplication.run(RibbonClientApplication.class, args);
    }
}
```

```java title="UserService.java"
@Service
public class UserService {
    
    @Autowired
    private RestTemplate restTemplate;
    
    public User getUser(Long id) {
        // 这里的 user-service 会被 Ribbon 解析
        // Ribbon 从 Eureka 获取实例列表，根据策略选择一台
        return restTemplate.getForObject(
            "http://user-service/api/users/" + id,
            User.class
        );
    }
}
```

**@LoadBalanced 的原理**：

`@LoadBalanced` 注解会向 RestTemplate 添加拦截器（LoadBalancerInterceptor），在发送请求前拦截 URL，根据服务名解析 IP。

```java title="LoadBalancerInterceptor.java"
public class LoadBalancerInterceptor implements ClientHttpRequestInterceptor {
    
    @Override
    public ClientHttpResponse intercept(
            HttpRequest request, 
            byte[] body,
            ClientHttpRequestExecution execution) throws IOException {
        
        URI originalUri = request.getURI();
        String serviceName = originalUri.getHost();
        
        // Ribbon 根据服务名选择服务器
        ServiceInstance instance = loadBalancer.choose(serviceName);
        
        // 修改请求 URI
        URI newUri = UriComponentsBuilder
            .fromHttpUrl(originalUri)
            .host(instance.getHost())
            .port(instance.getPort())
            .build()
            .toUri();
        
        request.setURI(newUri);
        return execution.execute(request, body);
    }
}
```

## 三、Spring Cloud LoadBalancer：Ribbon 的继任者

### 3.1 为什么需要 LoadBalancer

Ribbon 已经停止维护（2022 年 12 月宣布进入维护模式），Spring Cloud 推荐使用 Spring Cloud LoadBalancer 作为替代。

| 维度 | Ribbon | Spring Cloud LoadBalancer |
|---|---|---|
| 维护状态 | 维护模式（不推荐新项目使用） | 活跃维护 |
| 依赖 | Netflix Eureka | 通用抽象（支持 Eureka、Nacos、Consul） |
| 集成度 | 需要手动配置 | Spring Cloud 原生集成 |
| 响应式支持 | 不支持 | Reactor 响应式支持 |

### 3.2 Spring Cloud LoadBalancer 架构

```java
// Spring Cloud LoadBalancer 核心接口
public interface LoadBalancer {
    ReactorServiceInstance choose(Request request);
}

public interface ServiceInstanceListSupplier {
    String getServiceId();
    Flux<ServiceInstance> get(request);
}
```

核心组件：

| 组件 | 说明 |
|---|---|
| LoadBalancer | 核心接口，提供 choose() 方法 |
| ServiceInstanceListSupplier | 服务实例列表提供者 |
| LoadBalancerClient | LoadBalancer 的实现 |
| ReactorLoadBalancer | 响应式 LoadBalancer |

### 3.3 LoadBalancer 配置

**默认配置**：

Spring Cloud LoadBalancer 默认使用 **轮询策略**。

**自定义策略**：

```java title="LoadBalancerConfig.java"
@Configuration
@LoadBalancerClient(name = "user-service", 
                    configuration = LoadBalancerConfig.class)
public class LoadBalancerConfig {
    
    @Bean
    public ReactorLoadBalancer<ServiceInstance> randomLoadBalancer(
            ServiceInstanceListSupplier supplier) {
        return new RandomLoadBalancer(supplier);
    }
}

public class RandomLoadBalancer implements ReactorServiceInstanceLoadBalancer {
    
    private final ServiceInstanceListSupplier supplier;
    
    public RandomLoadBalancer(ServiceInstanceListSupplier supplier) {
        this.supplier = supplier;
    }
    
    @Override
    public Mono<Response<ServiceInstance>> choose(Request request) {
        return supplier.get(request)
            .next()
            .map(instances -> {
                ServiceInstance instance = instances.get(
                    ThreadLocalRandom.current().nextInt(instances.size())
                );
                return new DefaultResponse(instance);
            });
    }
}
```

## 四、重试与超时配置

### 4.1 Ribbon 重试配置

```bash title="application.yml"
user-service:
  ribbon:
    # 最大重试次数（不包括首次调用）
    MaxAutoRetries: 3
    
    # 对同一服务器的最大重试次数
    MaxAutoRetriesNextServer: 2
    
    # 是否所有操作都重试
    OkToRetryOnAllOperations: false
    
    # 连接超时
    ConnectTimeout: 3000
    
    # 读取超时
    ReadTimeout: 5000
    
    # 重试间隔
    RetryableStatusCodes: 500,502,503,504
```

**重试公式**：

```
总重试次数 = (MaxAutoRetries + 1) × (MaxAutoRetriesNextServer + 1) - 1
           = 4 × 3 - 1
           = 11
```

**首次调用 + 3 次重试（同一服务器）+ 切换服务器重试 2 次 × 4 次 = 11 次**

### 4.2 Spring Cloud LoadBalancer 重试配置

```bash title="application.yml"
spring:
  cloud:
    loadbalancer:
      retry:
        # 是否启用重试
        enabled: true
        
        # 重试次数
        maxRetries: 3
        
        # 5xx 错误重试
        retryableStatusCodes: 500,502,503,504

# 全局超时配置
hystrix:
  command:
    default:
      execution:
        isolation:
          thread:
            timeoutInMilliseconds: 5000

# Feign 超时配置
feign:
  client:
    config:
      default:
        connectTimeout: 3000
        readTimeout: 5000
```

### 4.3 重试的幂等性问题

重试会放大请求的副作用，必须确保接口是幂等的：

```java
// 非幂等操作（重试危险）
@PostMapping("/api/users")
public void createUser(User user) {
    userMapper.insert(user);  // 重试会创建多条数据！
}

// 幂等操作（重试安全）
@PostMapping("/api/users")
public void createUser(@RequestParam String idempotentKey, User user) {
    if (idempotentService.exists(idempotentKey)) {
        return;  // 重复请求直接返回
    }
    userMapper.insert(user);
    idempotentService.save(idempotentKey);
}
```

## 五、自定义负载均衡策略

### 5.1 基于权重的心跳负载均衡

```java title="HeartbeatWeightedLoadBalancer.java"
public class HeartbeatWeightedLoadBalancer implements ReactorServiceInstanceLoadBalancer {
    
    private final ServiceInstanceListSupplier supplier;
    private final Object provider;
    private volatile int cumulativeWeights = 0;
    
    public HeartbeatWeightedLoadBalancer(
            ServiceInstanceListSupplier supplier,
            String serviceId) {
        this.supplier = supplier;
        this.provider = serviceId;
    }
    
    @Override
    public Mono<Response<ServiceInstance>> choose(Request request) {
        if (this.supplier instanceof DiscoveryClientServiceInstanceListSupplier) {
            ((DiscoveryClientServiceInstanceListSupplier) this.supplier)
                .setServiceId(this.provider);
        }
        
        return supplier.get(request)
            .next()
            .map(instances -> getInstanceResponse(instances, request));
    }
    
    private Response<ServiceInstance> getInstanceResponse(
            List<ServiceInstance> instances, 
            Request<?> request) {
        
        if (instances.isEmpty()) {
            return new EmptyResponse();
        }
        
        // 获取每个实例的响应时间权重
        List<Double> weights = getWeights(instances);
        
        // 使用加权随机算法
        ServiceInstance selected = weightedRandomSelect(instances, weights);
        
        return new DefaultResponse(selected);
    }
    
    private List<Double> getWeights(List<ServiceInstance> instances) {
        // 从心跳数据或元数据获取权重
        return instances.stream()
            .map(instance -> {
                Map<String, String> metadata = instance.getMetadata();
                String weightStr = metadata.getOrDefault("weight", "100");
                return Double.parseDouble(weightStr);
            })
            .collect(Collectors.toList());
    }
    
    private ServiceInstance weightedRandomSelect(
            List<ServiceInstance> instances, 
            List<Double> weights) {
        
        double totalWeight = weights.stream().mapToDouble(w -> w).sum();
        double random = ThreadLocalRandom.current().nextDouble(totalWeight);
        
        double cumulative = 0;
        for (int i = 0; i < instances.size(); i++) {
            cumulative += weights.get(i);
            if (random < cumulative) {
                return instances.get(i);
            }
        }
        return instances.get(instances.size() - 1);
    }
}
```

### 5.2 基于区域感知的负载均衡

```java title="ZoneAffinityLoadBalancer.java"
public class ZoneAffinityLoadBalancer implements ReactorServiceInstanceLoadBalancer {
    
    private final ServiceInstanceListSupplier supplier;
    private final String serviceId;
    
    public ZoneAffinityLoadBalancer(
            ServiceInstanceListSupplier supplier,
            String serviceId) {
        this.supplier = supplier;
        this.serviceId = serviceId;
    }
    
    @Override
    public Mono<Response<ServiceInstance>> choose(Request request) {
        return supplier.get(request)
            .next()
            .map(instances -> selectInZone(instances));
    }
    
    private Response<ServiceInstance> selectInZone(List<ServiceInstance> instances) {
        // 获取当前实例所在的区域
        String currentZone = getCurrentZone();
        
        // 过滤同区域的实例
        List<ServiceInstance> sameZoneInstances = instances.stream()
            .filter(instance -> {
                String zone = instance.getMetadata().get("zone");
                return currentZone.equals(zone);
            })
            .collect(Collectors.toList());
        
        // 如果同区域有可用实例，选择同区域的
        // 否则fallback 到所有实例
        List<ServiceInstance> candidates = sameZoneInstances.isEmpty() 
            ? instances 
            : sameZoneInstances;
        
        ServiceInstance selected = candidates.get(
            ThreadLocalRandom.current().nextInt(candidates.size())
        );
        
        return new DefaultResponse(selected);
    }
    
    private String getCurrentZone() {
        // 从环境变量或配置文件获取当前区域
        return System.getenv("ZONE") != null 
            ? System.getenv("ZONE") 
            : "default-zone";
    }
}
```

## 六、微服务架构中的客户端负载均衡

### 6.1 典型架构

```
┌─────────────────────────────────────────────────────────┐
│                      API Gateway                        │
│                    (服务端 LB)                          │
└───────────────────────┬─────────────────────────────────┘
                        ↓
        ┌───────────────┼───────────────┐
        ↓               ↓               ↓
┌───────────┐   ┌───────────┐   ┌───────────┐
│Order Service│   │User Service│   │Product Svc│
│(Ribbon/LB) │   │(Ribbon/LB) │   │(Ribbon/LB)│
└─────┬──────┘   └─────┬──────┘   └─────┬──────┘
      ↓               ↓               ↓
┌───────────┐   ┌───────────┐   ┌───────────┐
│Order DB  │   │  User DB  │   │Product DB│
└───────────┘   └───────────┘   └───────────┘

Service Registry (Eureka/Nacos)
```

### 6.2 服务注册与发现

客户端负载均衡需要知道所有服务实例的地址，这通过服务注册中心实现：

```java

    service-url:
      defaultZone: http://eureka-server:8761/eureka/

    prefer-ip-address: true
    metadata:
      zone: zone-1  # 可用区

```

```java
// Ribbon 从 Eureka 获取服务列表
@Configuration
public class EurekaClientConfig {

```

### 7.1 问题一：Ribbon 的 ServerList 是如何获取的？

**参考答案**：

Ribbon 的 ServerList（服务实例列表）有以下几种来源：

**1. 静态配置**

```java
@Configuration
public class RibbonConfig {
    @Bean
    public ServerList<Server> serverList() {
        ConfigurationBasedServerList serverList = new ConfigurationBasedServerList();
        serverList.setListOfServers("192.168.1.101:8080,192.168.1.102:8080");
        return serverList;
    }
}
```

**2. Eureka 服务注册中心**

```java
@Configuration
public class RibbonConfig {
    @Bean
    public ServerList<Server> serverList(
            Provider<EurekaClient> eurekaClientProvider) {
        DiscoveryEnabledNIWSServerList serverList = 
            new DiscoveryEnabledNIWSServerList(
                (EurekaClient) eurekaClientProvider.get(),
                "user-service"  // 服务名
            );
        return serverList;
    }
}
```

**3. Nacos**

```java
@Configuration
public class RibbonConfig {
    @Bean
    public ServerList<Server> serverList(
            NacosDiscoveryProperties discoveryProperties) {
        NamingService namingService = discoveryProperties.namingServiceInstance();
        return new NacosServerList(namingService, "user-service");
    }
}
```

**列表刷新**：

```java
// ServerListRefreshInterval 控制刷新间隔
@Configuration
public class RibbonConfig {
    @Bean
    public ServerListRefresh ribbonServerListRefresh() {
        ServerListRefresh refresh = new ServerListRefresh();
        refresh.setServerListRefreshInterval(30000);  // 30 秒刷新一次
```

### 7.2 问题二：Ribbon 和 Spring Cloud LoadBalancer 的区别？

**参考答案**：

| 维度 | Ribbon | Spring Cloud LoadBalancer |
|---|---|---|
| 维护状态 | 进入维护模式 | 活跃开发 |
| Spring Cloud 版本 | Greenwich 及之前版本推荐 | 2020.0 (Spring Cloud 2020) 及之后版本推荐 |
| 响应式支持 | 不支持 | 支持 Reactor |
| 服务发现 | 主要支持 Eureka | 通用抽象，支持 Eureka、Nacos、Consul |
| 配置方式 | XML/Java/yml | Java 配置 |
| 策略扩展 | 需要实现 IRule | 需要实现 ReactorLoadBalancer |

**迁移建议**：

如果项目正在使用 Ribbon，建议逐步迁移到 Spring Cloud LoadBalancer：

1. 移除 Ribbon 依赖，添加 LoadBalancer 依赖
2. 将 `IRule` 实现迁移为 `ReactorLoadBalancer` 实现
3. 将 `ServerList` 实现迁移为 `ServiceInstanceListSupplier` 实现
4. RestTemplate 迁移为 WebClient（可选，推荐响应式）

### 7.3 问题三：客户端负载均衡的优缺点？

**参考答案**：

**优点**：

1. **去中心化**：没有独立的负载均衡器，避免单点故障
2. **性能好**：直接连接后端服务器，少一跳网络
3. **智能路由**：可以根据响应时间、错误率等动态选择
4. **服务感知**：天然和服务注册中心集成

**缺点**：

1. **配置分散**：每个客户端都需要配置，更新成本高（通常通过配置中心解决）
2. **语言绑定**：通常是 SDK 形式，不同语言需要不同实现
3. **资源消耗**：负载均衡逻辑在客户端，占用客户端资源
4. **升级复杂**：SDK 升级需要所有客户端同步升级

### 7.4 问题四：如何实现区域感知的负载均衡？

**参考答案**：

区域感知负载均衡的目的是让请求优先打到同区域的服务器，减少跨区域延迟。

**实现思路**：

1. 服务实例注册时携带可用区信息
2. LoadBalancer 获取当前实例所在的可用区
3. 优先选择同区域的实例
4. 同区域没有可用实例时，fallback 到其他区域

```java title="ZoneAwareLoadBalancer.java"
public class ZoneAwareLoadBalancer implements ReactorServiceInstanceLoadBalancer {
    
    @Override
    public Mono<Response<ServiceInstance>> choose(Request<?> request) {
        // 获取当前可用区
        String currentZone = getCurrentZone();
        
        return supplier.get(request)
            .next()
            .map(instances -> {
                // 过滤同区域的实例
                List<ServiceInstance> sameZone = instances.stream()
                    .filter(i -> currentZone.equals(getZone(i)))
                    .collect(Collectors.toList());
                
                // 同区域有实例，优先使用
                if (!sameZone.isEmpty()) {
                    return selectFromList(sameZone);
                }
                
                // fallback 到所有实例
                return selectFromList(instances);
            });
    }
}
```

```bash title="application.yml"
spring:
  cloud:
    loadbalancer:
      configurations: zone-aware
      zone:
        instance-metadata: ${ZONE:default-zone}
```

## 八、留给你的思考题

### 题目一：客户端负载均衡的「惊群效应」

假设有 10000 个客户端同时重启，它们会同时向服务注册中心拉取服务列表，同时向同一台服务器发送请求。这就是「惊群效应」。

问题：如何设计一个机制，让客户端的请求分散开，避免同时打到同一台服务器？

**提示**：考虑一致性哈希、随机 jitter、退避策略。

### 题目二：Ribbon 迁移到 LoadBalancer

假设你有一个 Spring Cloud Greenwich 项目，使用 Ribbon 实现负载均衡。现在要迁移到 Spring Cloud 2020，使用 LoadBalancer。

问题：迁移过程中可能遇到哪些兼容性问题？如何保证平滑过渡？

**提示**：考虑配置迁移、策略迁移、Feign 兼容性。

### 题目三：本地负载均衡 vs 代理负载均衡

在 Kubernetes 环境中，kube-proxy 负责 Service 的负载均衡（基于 iptables/IPVS）。如果应用层再使用 Ribbon/LoadBalancer，会产生多层负载均衡。

问题：这种情况下，应该如何设计负载均衡策略？是否需要去掉一层？

**提示**：考虑延迟、复杂度、灵活性的权衡。

## 总结

客户端负载均衡是微服务架构的核心组件。

**Ribbon 核心组件**：

- **ServerList**：服务实例列表
- **IRule**：负载均衡策略
- **LoadBalancer**：协调者

**Spring Cloud LoadBalancer 特点**：

- 响应式支持（Reactor）
- 通用抽象（不绑定特定服务发现）
- Spring Cloud 原生集成

**配置要点**：

- 服务发现集成（Eureka/Nacos）
- 负载均衡策略选择
- 超时和重试配置
- 健康检查配置

**选型建议**：

- 新项目：使用 Spring Cloud LoadBalancer
- 老项目：逐步迁移到 LoadBalancer
- 必须使用 Ribbon 的场景：确保 Ribbon 版本与 Spring Cloud 版本兼容

记住：客户端负载均衡的优势在于「智能」和「去中心化」，但配置和维护成本比服务端负载均衡高。理解其工作原理，才能用好它。
