# 高可用架构设计 - 概述

想象一下这个场景：双十一零点，你在某电商平台正准备下单，结果页面显示「系统繁忙，请稍后再试」。你刷新了三次，第四次终于进去了，但购物车里的商品已经被人抢光了。

这只是你一个人的体验。但背后发生的是：零点峰值流量涌入，订单系统撑不住，开始排队；排队时间过长，数据库连接池耗尽；然后是连锁反应——库存服务超时、支付服务超时、整个交易链路雪崩。

最后，平台发布了公告：「由于瞬时流量过大，系统出现短暂异常，正在紧急处理中……」

这就是高可用设计失败的典型代价。

反过来想：如果系统设计时就考虑了高可用呢？流量大？没关系，先在网关层限流，把超过系统处理能力的请求挡在外面。进入系统的请求，通过消息队列削峰填谷，让后端服务按照自己的节奏处理。数据库扛不住？没关系，主从读写分离，写请求打主库，读请求打从库。再不行，同城双活，一个机房出了问题，另一个机房接管。

这就是高可用架构设计的核心思想：**不是让系统不出问题，而是让系统在出问题的时候，依然能够提供服务。**

## 一、什么是高可用——从「几个九」说起

### 1.1 可用性的定义

高可用（High Availability，简称 HA），指的是系统在面对各种故障时，仍能保持正常提供服务的能力。

衡量可用性的指标，通常用「几个九」来表示：

| 可用性等级 | 年度停机时间 | 适用场景 |
|---|---|---|
| 99%（两个九） | 约 87.6 小时（3.6 天） | 内部系统、非核心业务 |
| 99.9%（三个九） | 约 8.7 小时 | 普通互联网应用 |
| 99.99%（四个九） | 约 52 分钟 | 金融支付、核心业务 |
| 99.999%（五个九） | 约 5 分钟 | 电信级、航天级系统 |

<Badge text="五个九" type="tip" /> 99.999% 的可用性，意味着每年只有约 5 分钟的停机时间。这是大多数互联网公司追求的目标，但实现难度极高，需要从架构、运维、监控等各个层面协同配合。

### 1.2 SLA、SLO、SLI：三个关键概念

在讨论高可用时，你一定会遇到这三个概念，它们是 Google SRE 推崇的可用性度量体系：

**SLA（Service Level Agreement，服务级别协议）** 是对外承诺，是和客户签订的「合同」。比如云服务商承诺 99.99% 的可用性，如果做不到，会按合同条款赔偿。

**SLO（Service Level Objective，服务级别目标）** 是内部目标，是团队给自己定的标准。比如团队承诺「我们追求 99.95% 的可用性」，这是比 SLA 更严格的目标，留有一定余量。

**SLI（Service Level Indicator，服务级别指标）** 是具体的测量指标。比如可用率 = 成功请求数 / 总请求数、延迟 P99 < 200ms、错误率 < 0.1%。

三者的关系可以这样理解：

```
SLA = 对外承诺 = SLO + 业务考量
SLO = 内部目标 = SLI 的聚合
SLI = 实际测量 = 数据埋点 + 监控采集
```

:::tip 实战经验

很多团队的问题在于：SLA 是拍脑袋定的，SLO 是空的，SLI 是缺失的。结果就是：系统挂了，客户投诉，才发现 SLA 根本没达到过。

正确做法是反过来的：先有 SLI（能测量），再定 SLO（合理目标），最后对外承诺 SLA（留有余量）。

:::

## 二、高可用架构全景图

高可用不是单一技术，而是一套完整的体系。从架构层面看，高可用设计主要包括以下几块：

### 2.1 负载均衡：流量分发的第一道防线

负载均衡是高可用的基础。它的核心思想是：不让单一服务器承受所有压力，而是把流量分散到多台服务器上。

负载均衡可以分为：

**四层负载均衡（L4）**：工作在 TCP/UDP 层，基于连接级别进行分发。特点是无感知、高效，常见实现有 LVS、HAProxy（四层模式）。

**七层负载均衡（L7）**：工作在应用层，可以基于 HTTP 头部、URL、Cookie 等内容进行分发。特点是功能丰富，可以做更精细的路由，常见实现有 Nginx、HAProxy（七层模式）、Envoy。

<Badge text="L4 vs L7" type="tip" /> 四层负载均衡性能更高，但功能有限；七层负载均衡功能更强，但性能开销略大。在实际架构中，通常是 L4 做入口分发，L7 做细粒度路由。

### 2.2 限流与熔断：流量保护的守门员

当流量超过系统处理能力时，限流和熔断就是保护系统的关键手段。

**限流（Rate Limiting）**：控制单位时间内的请求数量，超过阈值的请求直接拒绝或排队。常见的限流算法有令牌桶、漏桶、固定窗口、滑动窗口。

**熔断（Circuit Breaker）**：当某个下游服务的错误率超过阈值时，「断开」这个服务的调用，快速失败，避免雪崩。常见的实现有 Hystrix、Sentinel、Resilience4j。

两者的区别在于：

| 维度 | 限流 | 熔断 |
|---|---|---|
| 触发条件 | 请求速率超过阈值 | 下游服务错误率过高 |
| 处理方式 | 拒绝/排队多余请求 | 快速失败，不再调用下游 |
| 目的 | 保护系统不被压垮 | 防止故障蔓延 |
| 关系 | 互补，通常同时使用 | — |

### 2.3 降级与容错：优雅地「认输」

降级的本质是：承认自己扛不住，选择性地牺牲部分功能，保证核心功能可用。

举个例子：电商系统在高峰期，可以暂时关闭商品推荐、用户评论等非核心功能，把所有资源集中到下单和支付流程上。这就是「功能降级」。

容错则是：假设故障一定会发生，设计系统时就把故障考虑进去。比如调用远程服务超时，就走本地缓存；数据库主库挂了，就切换到从库继续读。

降级和容错的关系：

- **降级**：主动降低能力，换取核心能力
- **容错**：自动处理故障，保证能力不降级

### 2.4 多活架构：把鸡蛋放在多个篮子里

单机房部署的问题是：如果机房断电、光纤被挖断、自然灾害，所有服务都会受影响。多活架构通过在多个机房部署服务，实现真正的「不依赖单点」。

常见的多活形态：

- **同城双活**：两个机房在同一个城市，延迟低，适合大多数业务
- **两地三中心**：两个城市三个机房，兼顾可用性和数据安全
- **三地五中心**：三个城市五个机房，最高可用性，成本也最高

## 三、高可用的核心原则：三个「凡是」

### 3.1 凡是能出故障的地方，都必须考虑故障

这是高可用设计的第一个原则：**假设一切皆可能故障**。

硬盘会坏、网卡会掉、程序会崩、人会误操作——在系统设计时，必须把每个环节的故障都考虑进去，并给出应对方案。

具体来说：

- **单点故障消除**：任何组件都应有备份，不能有唯一的「命门」
- **故障隔离**：一个模块的问题不能蔓延到其他模块
- **优雅降级**：部分功能不可用时，核心功能仍能工作

### 3.2 凡是能自动处理的，都不要依赖人工

这是高可用设计的第二个原则：**让系统自己「打补丁」**。

想象一下：凌晨三点，系统监控告警说某个服务挂了。如果需要运维工程师爬起来手动重启，那这个系统的可用性天花板就是「运维的响应速度」。

而如果系统有自动故障检测 + 自动重启能力呢？告警发出后，30 秒内服务自动恢复，运维可以安心睡觉。

<Badge text="自动化是关键" type="tip" /> 高可用系统的目标是：无人值守也能正常运行。这个目标说起来简单，做起来需要大量的自动化能力建设。

### 3.3 凡是能提前检测的，都不要等到故障发生

这是高可用设计的第三个原则：**故障预防优于故障处理**。

故障分为两类：**故障前**和**故障后**。故障前可以预防，故障后只能处理。高可用的最高境界，是让故障根本不发生。

预防的手段包括：

- **容量规划**：提前预估业务增长，确保系统有足够容量
- **性能压测**：通过压测发现系统瓶颈，提前优化
- **变更管控**：重要变更必须在低峰期执行，有回滚方案
- **健康检查**：实时监控系统健康状态，异常提前告警

## 四、关键指标：MTTF、MTTR、MTBF

衡量一个系统的高可用水平，有三个关键的时间指标：

### 4.1 MTTF：Mean Time To Failure，平均故障时间

MTTF 是指系统平均能正常运行多久才会出故障。比如某服务器的 MTTF 是 8760 小时（约一年），意味着它平均每运行一年会出一次故障。

MTTF 越大越好，表示系统越可靠。

### 4.2 MTTR：Mean Time To Repair，平均修复时间

MTTR 是指从故障发生到恢复正常的平均时间。比如某个故障从发生到修复用了 30 分钟，那这次故障的修复时间就是 30 分钟。

MTTR 越小越好，表示系统恢复越快。

### 4.3 MTBF：Mean Time Between Failures，平均故障间隔

MTBF = MTTF + MTTR。这个指标衡量的是「两次故障之间的平均间隔时间」。

```
MTBF = 8760 小时（一年） + 0.5 小时（30分钟修复） = 8760.5 小时
```

### 4.4 可用性计算

系统的可用性可以用以下公式计算：

```
可用性 = MTTF / (MTTF + MTTR) × 100%
```

假设一个系统的 MTTF 是 99 天，MTTR 是 1 天，则：

```
可用性 = 99 / (99 + 1) × 100% = 99%
```

这个公式告诉我们：**提高可用性有两个方向：要么增加 MTTF（让系统更可靠），要么减少 MTTR（让系统更快恢复）。** 在现实中，减少 MTTR 通常比增加 MTTF 更容易做到。

:::tip 为什么减少 MTTR 更重要

假设有一个系统，当前 MTTF = 100 天，MTTR = 1 天，可用性 = 99%。

如果把 MTTF 翻倍到 200 天（投入大量成本提升可靠性），可用性 = 200 / 201 ≈ 99.5%。

如果把 MTTR 减半到 0.5 天（通过自动化运维实现），可用性 = 100 / 100.5 ≈ 99.5%。

同样的可用性提升，减少 MTTR 的成本通常更低。所以在大厂的高可用实践中，「故障快速恢复」是比「追求零故障」更优先的目标。

:::

## 五、CAP 理论 vs 高可用设计

分布式系统的高可用设计，绕不开 CAP 理论。

### 5.1 CAP 理论的三选二

CAP 理论指出：分布式系统不可能同时满足一致性（Consistency）、可用性（Availability）、分区容错性（Partition tolerance）这三个特性，最多只能同时满足两个。

- **一致性（Consistency）**：每次读操作都能读到最新写入的数据
- **可用性（Availability）**：每个请求都能在有限时间内得到响应
- **分区容错（Partition tolerance）**：系统在网络分区时仍能运行

由于网络分区是分布式系统中一定会发生的问题（光速有限、光纤会断），所以 CAP 理论实际上是在说：**在网络分区发生时，你必须在一致性和可用性之间二选一。**

| CAP 取舍 | 特点 | 适用场景 |
|---|---|---|
| CP（一致 + 分区容错） | 放弃可用性，分区时可能不可用 | 金融交易、分布式数据库 |
| AP（可用 + 分区容错） | 放弃强一致，分区时仍可用 | 社交 feed、搜索、日志 |

### 5.2 高可用不等于强一致

很多人有一个误区：高可用就是要保证数据强一致。实际上，高可用和强一致是两回事。

**高可用解决的是「系统能不能用」的问题。**

**强一致解决的是「数据准不准」的问题。**

在很多业务场景下，我们可以接受短暂的数据不一致（比如 Feed 流稍微延迟几秒），但绝对不能接受系统不可用。

:::warning CAP 理论不是非此即彼

CAP 理论说的是「三选二」，但在实际系统中，C、A、P 不是布尔值，而是连续的。比如：

- 一致性可以是「强一致」到「最终一致」之间的任何位置
- 可用性可以是「100% 可用」到「99% 可用」之间的任何位置
- 分区容错取决于网络质量和机房布局

现代分布式系统通常是在 CAP 之间的「灰色地带」做权衡，而不是简单地选择 CP 或 AP。

:::

## 六、性能、并发、可用性：三位一体

高性能、高并发、高可用——这是 Java 后端面试的三座大山，也是架构设计的三个核心维度。它们相互关联、相互影响。

### 6.1 性能与可用性的关系

性能优化的目标是最小化响应时间（Latency），最大化吞吐量（Throughput）。

但有时候，过度追求性能反而会损害可用性。比如：

- 为了追求极致的性能，关闭了所有限流和熔断机制
- 为了减少延迟，去掉了所有重试和超时机制
- 为了提升吞吐量，设置了超大的并发连接数

这些做法在低流量时表现优异，但一旦流量超过预期，系统就会快速崩溃，可用性大打折扣。

**正确的做法是：在满足可用性要求的前提下优化性能。** 换句话说，先保证系统「能活着」，再追求「活得好」。

### 6.2 并发与可用性的关系

高并发场景对可用性提出了更高的要求。因为并发意味着多个请求同时竞争资源，更容易触发各种边界条件。

高并发导致的高可用挑战包括：

- **线程池耗尽**：请求太多，线程池被打满，新请求无法处理
- **连接池耗尽**：数据库连接、Redis 连接被占满
- **雪崩效应**：一个服务的超时导致调用方资源耗尽，进而扩散到整个系统

应对高并发的可用性设计手段包括：

- **限流**：控制进入系统的请求量
- **熔断**：阻止故障蔓延
- **降级**：牺牲非核心功能，保证核心功能
- **隔离**：不同业务使用不同的资源池

### 6.3 三者的平衡

在架构设计中，性能、并发、可用性往往需要权衡。没有最优解，只有最适合业务场景的解。

举例来说：

- **金融支付系统**：优先保证可用性和一致性，可以牺牲部分性能
- **社交媒体 Feed**：优先保证可用性和性能，可以接受最终一致
- **秒杀系统**：优先保证高并发下的可用性，一致性可以通过事后对账解决

## 七、本模块章节脉络

高可用架构设计是一个庞大的体系，本模块将从以下几个维度展开：

```
高可用架构设计
├── 主备切换与故障转移（failover）
│   └── 故障转移机制、健康检查、状态同步、脑裂问题
├── 异地多活架构（multi-active）
│   └── 同城双活、两地三中心、三地五中心
├── 数据备份与恢复（backup-restore）
│   └── 全量备份、增量备份、RTO/RPO
├── 灰度发布与金丝雀（canary-deploy）
│   └── 渐进式发布、流量控制、指标分析
├── 熔断后的服务恢复（circuit-recovery）
│   └── 状态机、预热机制、恢复策略
├── 高可用案例：电商秒杀（seckill-design）
│   └── 削峰、限流、库存控制、超卖防护
└── 高可用案例：12306抢票（ticket-design）
    └── 锁座机制、队列设计、公平性保证
```

每个章节都是独立的，但它们共同构成了高可用架构的完整图景。建议按照从上到下的顺序阅读，形成系统性的认知。

## 八、面试追问：可用性设计高频问题

### 问题一：如何设计一个高可用系统？

**参考答案**：

高可用系统设计需要从多个层面入手：

**架构层面**：

- 消除单点故障：所有组件都有冗余备份
- 故障隔离：使用舱壁模式，防止故障蔓延
- 优雅降级：核心功能优先，非核心功能可降级
- 多机房部署：防止单机房故障影响全局

**流量层面**：

- 负载均衡：流量分发到多台服务器
- 限流熔断：防止流量冲击导致系统崩溃
- 异步处理：使用消息队列削峰填谷

**数据层面**：

- 主从复制：读写分离，提高可用性和性能
- 数据备份：定期备份，支持数据恢复
- 异地多活：数据跨机房同步

**运维层面**：

- 自动化运维：故障自动检测、自动恢复
- 灰度发布：降低变更风险
- 监控告警：问题早发现、早处理

### 问题二：「四个九」和「五个九」的实现难度差距有多大？

**参考答案**：

从 99.9% 到 99.99%，可用性提升了 10 倍，但实现难度远不止 10 倍。

| 可用性 | 年度停机时间 | 每小时允许的故障时间 |
|---|---|---|
| 99.9% | 8.7 小时 | 约 6 分钟 |
| 99.99% | 52 分钟 | 约 36 秒 |
| 99.999% | 5 分钟 | 约 3.6 秒 |

99.9% 的系统，允许每 8 小时有一次 6 分钟的故障，这可以通过「人工重启服务」来解决。

99.99% 的系统，允许每 8 小时有一次 36 秒的故障，这需要自动化故障检测和恢复。

99.999% 的系统，允许每 8 小时有一次 3.6 秒的故障，这需要系统具备极其可靠的自我修复能力，以及完善的监控和变更管控体系。

<Badge text="边际效应" type="warning" /> 从 99.9% 到 99.99% 可能是 10 倍的投入，但从 99.99% 到 99.999% 可能是 100 倍的投入。企业在追求更高可用性时，需要权衡成本和收益。

### 问题三：如何衡量一个系统的高可用水平？

**参考答案**：

衡量系统高可用水平，主要从以下维度：

**可用性指标**：

- SLA 达成率：实际可用时间 / 约定可用时间
- 故障频率：平均故障间隔时间（MTBF）
- 故障恢复时间：平均修复时间（MTTR）

**用户体验指标**：

- 错误率：请求失败的比例
- 延迟分布：P50、P90、P99、P999 延迟
- 成功率：下单成功率、支付成功率等业务指标

**运维指标**：

- 监控覆盖率：关键指标是否都有监控
- 告警响应时间：从告警到处理的时间
- 变更成功率：发布、扩缩容等变更的成功率

## 延伸思考

### 思考一：可用性和一致性的边界

在设计高可用系统时，如何判断哪些场景可以接受最终一致，哪些场景必须保证强一致？

提示：考虑业务场景的「容错窗口」，以及数据不一致带来的实际损失。

### 思考二：成本与可用性的权衡

99.999% 的可用性需要付出巨大的成本投入。但对于大多数业务来说，99.9% 的可用性已经足够。

问题：如何评估业务场景对可用性的真实需求？有没有「过度设计」的高可用案例？

## 总结

高可用架构设计是软件工程中最考验综合能力的领域之一。它不是某一项技术的堆砌，而是架构思维、运维能力、业务理解的综合体现。

关键要点回顾：

- **可用性度量**：用「几个九」来衡量，99.99% 是大多数核心系统的目标
- **核心原则**：故障预防 > 故障检测 > 故障恢复，自动化是关键
- **关键指标**：MTTF、MTTR、MTBF 构成可用性评估体系
- **CAP 权衡**：高可用不等于强一致，需要根据业务场景权衡
- **三高平衡**：性能、并发、可用性相互关联，需要整体考虑

高可用不是终点，而是一个持续的过程。再完美的架构设计，也需要运维、监控、变更管理的配合。持续改进，才是高可用的真正内涵。
