# 高可用架构案例：电商秒杀系统设计

双十一零点，你准备好了购物车里的商品，定好闹钟准备抢购。结果倒计时归零的那一刻，页面开始转圈、报错、刷新、重试……经过九九八十一难，终于下单成功，却发现库存已经被抢光了。

这种感觉，每个经历过秒杀的人都懂。

但换个角度思考——如果你是那个电商平台的架构师，面对这种场景，你会怎么设计系统？

零点高峰流量可能是平时的 100 倍甚至 1000 倍，但库存只有那么多。问题是：如何在有限库存的情况下，让尽可能多的用户有好的体验？

这就是秒杀系统要解决的核心问题。

这篇文章，我们来聊聊电商秒杀系统的设计思路，从流量控制到库存扣减，从高并发到高可用，完整覆盖一个秒杀系统的技术架构。

## 一、秒杀场景的特点与挑战

### 1.1 秒杀流量的特征

秒杀场景的流量有几个显著特点：

**流量巨大且集中**：秒杀活动通常在固定时间点开始，所有用户会在那一刻涌入。以双十一为例，零点峰值 QPS 可能达到百万级别。

**有效请求占比极低**：假设库存只有 1000 件，但参与抢购的用户有 100 万人，那么 99.9% 的请求是「无效」的——无论系统多快，这些请求最终都会失败。

**读多写少**：在秒杀开始前，用户会大量刷新页面查看商品详情（读）；秒杀开始后，真正下单的请求才是写操作。但写操作的比例远低于读操作。

**业务逻辑简单**：秒杀的核心逻辑非常单纯——库存够不够，够就扣减，不够就返回失败。没有复杂的业务计算。

### 1.2 秒杀的挑战

基于以上特点，秒杀系统面临的核心挑战是：

**超卖问题**：多个请求同时扣减库存，如果处理不当，可能出现库存变成负数的情况。

**数据库压力**：如果所有请求都打到数据库，数据库会被瞬间击垮。

**用户体验**：用户在秒杀过程中会遇到各种异常（超时、排队、失败），如何让体验尽可能流畅。

**公平性**：如何保证秒杀机会对所有用户公平，不能让脚本党有机可乘。

### 1.3 秒杀 vs 普通下单

| 维度 | 普通下单 | 秒杀下单 |
|---|---|---|
| QPS | 100-1000 | 100,000-1,000,000 |
| 库存 | 充足 | 有限（稀缺） |
| 成功率 | 高 | 极低（>99% 失败） |
| 核心诉求 | 正确性 | 高可用 + 正确性 |
| 技术挑战 | 数据一致性 | 流量洪峰 + 数据一致性 |

## 二、秒杀系统架构设计

### 2.1 分层架构概览

秒杀系统的架构核心是「分层限流」——在不同的层级过滤掉无效请求，减轻后端压力。

```
用户请求
    │
    ▼
CDN 层（静态资源）
    │ → 拦截静态资源请求，减轻带宽压力
    ▼
网关层（流量控制）
    │ → IP 限流、用户限流、黑名单
    ▼
MQ 层（削峰填谷）
    │ → 消息队列缓冲，异步处理
    ▼
秒杀服务层（库存预扣）
    │ → Redis 原子扣减，判断是否进入下单流程
    ▼
订单服务层（最终下单）
    │ → 数据库事务，确保库存不超卖
    ▼
支付层（最终一致性）
    │ → 支付回调，更新最终状态
    ▼
秒杀完成
```

### 2.2 CDN 层：静态资源分离

秒杀页面的静态资源（HTML、CSS、JS、图片）应该部署在 CDN 上，用户就近访问，减少源站压力。

```html
<!-- 秒杀商品详情页 -->
<!DOCTYPE html>
<html>
<head>
    <!-- 静态资源走 CDN -->
    <link rel="stylesheet" href="https://cdn.example.com/seckill.css">
    <script src="https://cdn.example.com/seckill.js"></script>
</head>
<body>
    <!-- 动态数据通过 API 获取 -->
    <div id="product" data-product-id="12345"></div>
    <script>
        // 商品信息通过 API 拉取
        fetch('/api/product/12345')
            .then(res => res.json())
            .then(data => renderProduct(data));
    </script>
</body>
</html>
```

### 2.3 网关层：流量控制

网关层是流量的第一道防线，主要做以下事情：

**IP 限流**：限制单个 IP 的请求频率，防止刷接口。

```nginx title="Nginx 限流配置"]
limit_req_zone $binary_remote_addr zone=ip_limit:10m rate=10r/s;
limit_req_zone $binary_remote_addr zone=user_limit:10m rate=100r/m;

server {
    location /seckill {
        # 限流：每秒最多 10 个请求
        limit_req zone=ip_limit burst=20 nodelay;

        # 用户级别限流：每分钟最多 100 个请求
        limit_req zone=user_limit burst=50 nodelay;

        proxy_pass http://seckill_backend;
    }
}
```

**用户身份校验**：检查用户是否登录、是否有秒杀资格。

**黑名单机制**：将恶意用户、脚本党加入黑名单。

```yaml title="网关限流配置"]
global:
  rate_limit:
    enabled: true

  rules:
    - name: ip_rate_limit
      type: remote_addr
      rate: 10  # 每秒 10 个请求
      burst: 20  # 突发容量

    - name: user_rate_limit
      type: header
      header: X-User-Id
      rate: 100  # 每分钟 100 个请求

    - name: path_rate_limit
      type: path
      path: /api/seckill
      rate: 1000  # 秒杀接口每秒 1000 个请求

  blacklist:
    enabled: true
    source: redis
    update_interval: 60  # 每分钟更新一次
```

## 三、流量削峰：MQ 的作用

### 3.1 为什么需要 MQ

即使做了网关限流，仍可能有大量请求进入后端服务。这些请求如果都直接打到数据库，数据库会瞬间被打爆。

MQ（消息队列）的作用是**削峰填谷**：

- 秒杀开始时，大量请求涌入，MQ 将请求缓冲起来
- 后端服务按照自己的处理能力，从 MQ 中拉取请求
- 即使请求量再大，也不会压垮后端

```
无 MQ：
用户请求 → 直接处理 → 数据库（被打爆）

有 MQ：
用户请求 → 写入 MQ → 后端服务 → 数据库（稳定处理）
```

```java title="秒杀下单接口（写入 MQ）"]
@Service
public class SeckillService {
    private final RedisService redisService;
    private final KafkaTemplate<String, SeckillMessage> kafkaTemplate;

    public Result<?> seckill(SeckillRequest request) {
        String userId = request.getUserId();
        String productId = request.getProductId();

        // 1. 用户幂等检查：同一用户对同一商品只能秒杀一次
        String idempotentKey = "seckill:order:" + userId + ":" + productId;
        if (redisService.exists(idempotentKey)) {
            return Result.error("您已参与过该商品秒杀");
        }

        // 2. 生成请求 ID
        String requestId = UUID.randomUUID().toString();

        // 3. 发送消息到 MQ
        SeckillMessage message = new SeckillMessage();
        message.setUserId(userId);
        message.setProductId(productId);
        message.setTimestamp(System.currentTimeMillis());
        message.setRequestId(requestId);
        message.setTraceId(Tracer.getCurrentTraceId());

        kafkaTemplate.send("seckill-topic", productId, message);

        // 4. 记录用户秒杀状态（用于查询结果）
        redisService.setex("seckill:status:" + userId + ":" + productId,
                3600, "processing");

        // 5. 返回处理中
        return Result.success("秒杀排队中，请稍后查询结果");
    }
}
```

### 3.3 消费端设计与限流

MQ 消费端需要控制消费速度，避免压垮下游服务：

```java title="秒杀消费端限流"]
@Configuration
public class SeckillConsumerConfig {
    @Bean
    public ConcurrentKafkaListenerContainerFactory<String, SeckillMessage> kafkaListenerContainerFactory(
            KafkaTemplate kafkaTemplate) {
        ConcurrentKafkaListenerContainerFactory<String, SeckillMessage> factory =

        factory.setConsumerFactory(consumerFactory());
        // 设置并发数，控制消费速度
        factory.setConcurrency(10);
        // 设置批量拉取
        factory.setBatchListener(true);
        factory.setBatchSize(100);
        factory.setPollTimeout(1000);

        return factory;
    }
}

@Service
public class SeckillOrderConsumer {
    private final SeckillOrderService orderService;

    @KafkaListener(topics = "seckill-topic", groupId = "seckill-order-group")
    public void consume(List<SeckillMessage> messages) {
        for (SeckillMessage message : messages) {
            try {
                // 限流处理
                rateLimiter.acquire();
                orderService.processSeckillOrder(message);
            } catch (Exception e) {
                // 处理失败，记录日志，发送到死信队列
                handleFailure(message, e);
            }
        }
    }

    // 令牌桶限流
    private final RateLimiter rateLimiter = RateLimiter.create(1000); // 每秒处理 1000 个

    private void handleFailure(SeckillMessage message, Exception e) {
        log.error("处理秒杀订单失败: {}", message, e);
        // 发送到死信队列
        kafkaTemplate.send("seckill-dlq", message);
    }
}
```

## 四、库存扣减：Redis 原子操作

### 4.1 为什么用 Redis 预扣库存

秒杀系统的核心瓶颈是**库存扣减**。如果每个请求都直接操作数据库，数据库会被瞬间打爆。

解决方案是**分层扣减**：

- **第一层（Redis）**：快速判断库存是否足够，原子扣减
- **第二层（数据库）**：最终一致性保障，确保不超卖

Redis 的原子操作（如 DECR）是毫秒级的，单机 QPS 可以达到 10 万以上，足以应对秒杀场景。

### 4.2 Redis 库存扣减实现

```java title="Redis 库存扣减"]
@Service
public class RedisStockService {
    private final StringRedisTemplate redisTemplate;

    /**
     * 扣减库存（Lua 脚本保证原子性）
     */
    public boolean decreaseStock(String productId, int quantity) {
        String stockKey = "seckill:stock:" + productId;

        // Lua 脚本：原子判断和扣减
        String script = """
            local stock = tonumber(redis.call('GET', KEYS[1]))
            if stock == nil or stock < tonumber(ARGV[1]) then
                return 0
            end
            redis.call('DECRBY', KEYS[1], ARGV[1])
            return 1
            """;

        RedisScript<Long> redisScript = RedisScript.of(script, Long.class);
        Long result = redisTemplate.execute(redisScript,
                Collections.singletonList(stockKey),
                String.valueOf(quantity));

        return result != null && result == 1;
    }

    /**
     * 回补库存（用于超时未支付等场景）
     */
    public void increaseStock(String productId, int quantity) {
        String stockKey = "seckill:stock:" + productId;
        redisTemplate.opsForValue().increment(stockKey, quantity);
    }

    /**
     * 查询库存
     */
    public int getStock(String productId) {
        String stockKey = "seckill:stock:" + productId;
        String stock = redisTemplate.opsForValue().get(stockKey);
        return stock == null ? 0 : Integer.parseInt(stock);
    }
}
```

## 五、订单创建：最终一致性保障

### 5.1 订单服务设计

MQ 消费端调用订单服务创建订单。订单服务需要确保：

- 订单创建成功
- 库存最终一致性
- 不超卖

```java title="订单服务实现"]
@Service
public class SeckillOrderService {
    private final OrderMapper orderMapper;
    private final ProductMapper productMapper;
    private final RedisStockService redisStockService;

    @Transactional(rollbackFor = Exception.class)
    public Order createSeckillOrder(SeckillMessage message) {
        String productId = message.getProductId();
        String userId = message.getUserId();

        // 1. 查询商品信息
        Product product = productMapper.selectById(productId);
        if (product == null || product.getSeckillStock() <= 0) {
            throw new BusinessException("商品不存在或库存不足");
        }

        // 2. 创建订单
        Order order = new Order();
        order.setId(UUID.randomUUID().toString());
        order.setUserId(userId);
        order.setProductId(productId);
        order.setQuantity(1);
        order.setPrice(product.getSeckillPrice());
        order.setStatus(OrderStatus.PENDING_PAYMENT);
        order.setCreateTime(new Date());
        order.setExpireTime(DateUtils.addMinutes(new Date(), 15));  // 15 分钟支付超时

        orderMapper.insert(order);

        // 3. 扣减数据库库存（乐观锁）
        int rows = productMapper.decreaseSeckillStock(productId);
        if (rows == 0) {
            throw new BusinessException("库存扣减失败");
        }

        // 4. 更新 Redis 状态
        redisStockService.confirmOrder(userId, productId, order.getId());

        return order;
    }
}
```

### 5.3 超时未支付处理

秒杀订单通常有支付时限（如 15 分钟），超时未支付需要释放库存：

```java title="超时订单处理"]
@Scheduled(cron = "0 */1 * * * ?")  // 每分钟执行
public void handleExpiredOrders() {
    log.info("开始处理超时订单...");

    // 1. 查询超时未支付的订单
    List<Order> expiredOrders = orderMapper.selectExpiredOrders(new Date());

    for (Order order : expiredOrders) {
        try {
            // 2. 标记订单为已取消
            order.setStatus(OrderStatus.CANCELLED);
            order.setCancelTime(new Date());
            orderMapper.updateById(order);

            // 3. 回补库存
            productMapper.increaseSeckillStock(order.getProductId());
            redisStockService.increaseStock(order.getProductId(), 1);

            // 4. 记录日志
            log.info("超时订单已取消: orderId={}, productId={}",
                    order.getId(), order.getProductId());

        } catch (Exception e) {
            log.error("处理超时订单失败: orderId={}", order.getId(), e);
        }
    }

    log.info("超时订单处理完成，共处理 {} 个订单", expiredOrders.size());
}
```

## 六、超卖防护：多层保障机制

### 6.1 超卖的原因

超卖（ overselling）是指订单数量超过实际库存数量。产生超卖的原因通常是**并发下的库存扣减不是原子的**：

```java
// 典型的错误写法
Product product = productMapper.selectById(productId);  // stock = 1
if (product.getStock() > 0) {
    product.setStock(product.getStock() - 1);  // stock = 0
    productMapper.updateById(product);  // 扣减成功
    // 如果两个请求同时到达，两个都会扣减成功
}
```

### 6.2 超卖防护策略

**策略一：Redis 原子扣减**

Redis 的 DECR 是原子操作，天然的并发安全：

```lua
-- Redis Lua 脚本：原子扣减
local stock = tonumber(redis.call('GET', KEYS[1]))
if stock and stock >= tonumber(ARGV[1]) then
    redis.call('DECRBY', KEYS[1], ARGV[1])
    return 1
else
    return 0
end
```

**策略二：数据库乐观锁**

```sql
UPDATE product
SET stock = stock - #{quantity}
WHERE id = #{productId}
  AND stock >= #{quantity}
```

**策略三：分布式锁**

```java title="分布式锁实现"]
public boolean tryCreateOrder(String userId, String productId) {
    String lockKey = "order:lock:" + productId;

    // 获取分布式锁
    Boolean acquired = redisTemplate.opsForValue()
            .setIfAbsent(lockKey, userId, 10, TimeUnit.SECONDS);

    if (!acquired) {
        return false;  // 获取锁失败
    }

    try {
        // 执行业务逻辑
        return doCreateOrder(userId, productId);
    } finally {
        // 释放锁
        redisTemplate.delete(lockKey);
    }
}
```

### 6.3 多层防护架构

```
超卖防护多层机制：

Layer 1: 网关层限流
│ └─ 限制单个 IP/用户的请求频率

Layer 2: Redis 库存预扣
│ └─ Lua 脚本原子扣减，判断是否通过

Layer 3: MQ 异步处理
│ └─ 削峰填谷，控制数据库写入速率

Layer 4: 数据库乐观锁
│ └─ 最终兜底，确保不超卖

Layer 5: 定时对账
  └─ 发现异常及时告警和处理
```

## 七、防刷与安全

### 7.1 验证码/人机验证

秒杀开始前，要求用户完成验证码，防止脚本自动抢购：

```java title="验证码服务"]
@Service
public class CaptchaService {
    private final StringRedisTemplate redisTemplate;

    /**
     * 生成验证码
     */
    public String generateCaptcha(String userId, String productId) {
        String captchaId = UUID.randomUUID().toString();
        String captchaText = generateRandomText();

        // 验证码有效期 5 分钟
        redisTemplate.opsForValue().set(
                "captcha:" + captchaId,
                captchaText,
                5, TimeUnit.MINUTES);

        return captchaId;  // 返回验证码 ID，前端用于展示
    }

    /**
     * 验证验证码
     */
    public boolean verifyCaptcha(String captchaId, String captchaText) {
        String key = "captcha:" + captchaId;
        String storedText = redisTemplate.opsForValue().get(key);

        if (storedText == null || !storedText.equalsIgnoreCase(captchaText)) {
            return false;
        }

        // 验证成功后删除验证码
        redisTemplate.delete(key);
        return true;
    }

    private String generateRandomText() {
        // 生成 4 位随机数字
        return String.format("%04d", new Random().nextInt(10000));
    }
}
```

### 7.2 设备指纹与行为分析

```java title="设备指纹识别"]
@Service
public class DeviceFingerprintService {
    private final StringRedisTemplate redisTemplate;

    public String generateFingerprint(HttpServletRequest request) {
        // 组合多个维度生成设备指纹
        String userAgent = request.getHeader("User-Agent");
        String acceptLanguage = request.getHeader("Accept-Language");
        String acceptEncoding = request.getHeader("Accept-Encoding");
        String ip = getRealIp(request);

        // 简单哈希作为指纹
        String raw = userAgent + acceptLanguage + acceptEncoding + ip;
        return String.valueOf(raw.hashCode());
    }

    public boolean isSuspicious(String fingerprint) {
        // 检查是否为可疑设备
        String key = "fingerprint:suspicious:" + fingerprint;
        return redisTemplate.hasKey(key);
    }

    public void markSuspicious(String fingerprint) {
        // 标记为可疑设备
        String key = "fingerprint:suspicious:" + fingerprint;
        redisTemplate.opsForValue().set(key, "1", 24, TimeUnit.HOURS);
    }
}
```

## 九、阿里双十一架构案例

### 9.1 阿里秒杀系统特点

阿里双十一的秒杀系统是业界标杆，其特点包括：

**单元化架构**：全国用户按地域分配到不同单元，每个单元独立处理秒杀请求，减少跨区域流量。

**离架计算**：库存提前分配到各个单元，每个单元只处理自己单元内的用户请求，避免集中式扣减。

**数据分级**：热点数据放在 Tair（阿里自研缓存），冷数据放在数据库。

**流量分层**：CDN → LVS → Nginx → 应用层 → 数据层，每层都有流量控制。

### 9.2 离架计算原理

```
传统架构：
[全国用户] → [中心库存] → 数据库扣减
            ↑
        所有流量集中

离架架构：
[华东用户] → [华东库存] → 华东数据库
[华南用户] → [华南库存] → 华南数据库
[华北用户] → [华北库存] → 华北数据库
           ↑
        流量分散
```

## 十、面试追问：秒杀系统高频问题

### 问题一：如何设计一个秒杀系统？

**参考答案**：

秒杀系统的设计要点：

**流量分层**：

- CDN 层：静态资源分离
- 网关层：IP/用户限流、黑名单
- MQ 层：削峰填谷，异步处理
- Redis 层：库存预扣，原子操作
- 数据库层：最终一致性保障

**核心问题解决**：

- **超卖**：Redis Lua 原子扣减 + 数据库乐观锁双重保障
- **性能**：分层过滤无效请求，减少数据库压力
- **公平性**：验证码、设备指纹、行为分析防刷

**关键指标**：

- QPS：网关限流后的实际处理能力
- 库存扣减成功率：Redis 原子操作成功率
- 订单创建成功率：最终下单成功率

### 问题二：如何防止超卖？

**参考答案**：

超卖防护的多层机制：

**第一层：Redis 原子扣减**

```lua
local stock = tonumber(redis.call('GET', KEYS[1]))
if stock and stock >= tonumber(ARGV[1]) then
    redis.call('DECRBY', KEYS[1], ARGV[1])
    return 1
else
    return 0
end
```

**第二层：数据库乐观锁**

```sql
UPDATE product
SET stock = stock - 1, version = version + 1
WHERE id = #{id} AND stock > 0 AND version = #{version}
```

**第三层：消息队列顺序消费**

保证同一商品的消息顺序处理，避免并发问题。

### 问题三：Redis 库存和数据库库存如何保持一致？

**参考答案**：

Redis 和数据库库存的一致性保证：

**秒杀开始前**：预热库存到 Redis

```java
redis.set("stock:" + productId, product.getStock());
```

**秒杀过程中**：

- Redis 先扣（快速判断）
- 数据库后扣（最终保障）
- 如果数据库扣减失败，回补 Redis

**秒杀结束后**：对账与修正

```java
// 定时检查 Redis 和数据库的库存差异
int redisStock = redis.get("stock:" + productId);
int dbStock = productMapper.selectById(productId).getStock();
if (redisStock != dbStock) {
    // 记录差异，告警处理
    log.error("库存不一致: productId={}, redis={}, db={}",
            productId, redisStock, dbStock);
}
```

## 总结

秒杀系统是高并发场景的典型代表，它综合考验了架构师在流量控制、数据一致性、系统高可用等多个方面的能力。

关键要点回顾：

- **流量分层**：CDN → 网关 → MQ → Redis → 数据库，逐层过滤无效请求
- **Redis 原子扣减**：Lua 脚本保证库存扣减的原子性
- **MQ 削峰**：异步处理，平滑后端压力
- **多层防超卖**：Redis + 乐观锁 + 分布式锁
- **安全防护**：验证码、设备指纹、行为分析

秒杀系统的设计没有银弹，每个细节都需要精心打磨。但核心思想是一致的：**让正确的请求通过，让无效的请求在越早的层级被拦截越好。**
