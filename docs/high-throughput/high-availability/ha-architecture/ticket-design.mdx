# 高可用架构案例：12306 抢票系统设计

春节前三天，你打开 12306 APP，准备买一张回老家的高铁票。页面显示「余票 0 张」，你刷新了一下，还是 0；再刷新，系统提示「当前排队人数较多，请稍后再试」；你等了三分钟再刷，余票突然变成了 2 张——你手速飞快地点击购买，结果页面卡住了两秒，弹出「余票不足」。

这不只是一个买票体验的问题，这是全世界最复杂的并发系统之一在运转。

12306 系统面临着什么量级的挑战？峰值日均访问量超过 1000 亿次，春节高峰期单日售票量超过 1500 万张，核心系统的并发请求量达到每秒数十万级别。这不是简单的「多卖几张票」的问题，而是如何在有限的票务资源下，让数亿用户的购票请求得到公平、高效的处理。

秒杀系统我们讲过了，核心是「库存扣减」。但 12306 比秒杀复杂得多——它不只卖一件商品，而是一张网的车票；不是简单的库存扣减，而是涉及座位、行程、多程组合的复杂查询；不是下单即结束，而是从抢票到支付再到退改签的完整生命周期。

这篇文章，我们来聊聊 12306 抢票系统的设计思路。

## 一、12306 的本质挑战：为什么比秒杀更复杂？

### 1.1 秒杀 vs 12306：根本区别在哪里？

很多人把 12306 等同于「高并发秒杀」，但实际上两者有本质区别：

| 维度 | 电商秒杀 | 12306 抢票 |
|---|---|---|
| **商品类型** | 单品，如一部手机 | 多程组合，如「北京→上海→杭州」 |
| **库存模型** | 集中库存，一件商品一个库存 | 分布式库存，每个座位独立存在 |
| **扣减单位** | 库存数量（整数） | 物理座位 → 逻辑座位 → 预留座位 |
| **查询类型** | 单品查询 | 多程组合查询，涉及中转换乘 |
| **价格模型** | 固定价格 | 里程计价 + 席位差价 + 时段浮动 |
| **约束条件** | 数量约束（库存） | 多维约束（座位类型、余票、时间、身份信息） |
| **取消重试** | 放弃即可 | 可能需要重新抢票 |
| **退改签** | 无 | 全流程退改签，释放座位重新销售 |

秒杀的核心问题是「库存够不够」，答案只有两个：是或否。12306 的核心问题要复杂得多——「这两个人之间的座位能不能卖？」，涉及座位是否已售、是否被锁、是否在本次购票流程中被预留等多种状态。

### 1.2 票务系统的核心矛盾

12306 系统面临三重核心矛盾：

**第一重：查询与扣减的矛盾**

购票流程的第一步是查询余票，但查询量远超扣减量。一个热门车次，可能有 100 万人查询，只有 1000 人能买到票。这意味着 99.9% 的查询是「无效」的，但系统必须处理所有查询。

**第二重：实时性与一致性的矛盾**

用户希望在查询时看到真实的余票信息，但余票信息是分布式存储在多个数据库中的，不同节点的数据同步存在延迟。如果查询返回了「有余票」，但实际扣减时发现「没票了」，用户体验会非常糟糕。

**第三重：公平性与效率的矛盾**

先到先得是最公平的购票方式，但先到先得需要全局排队。在数十万人同时抢票的情况下，全局排队本身就会成为系统瓶颈。

### 1.3 票务系统的特殊性：库存是动态的

普通电商的库存是「静止」的——你卖出一件，库存就少一件。12306 的库存是「动态」的：

- 有人买票，库存减少
- 有人退票，库存增加
- 有人改签，库存重新分配
- 临近开车，部分座位可能「解锁」给其他乘客
- 团体票、学生票、残障票有预留配额

这意味着 12306 不能简单地用 Redis DECR 来扣减库存，而是需要一个能够处理复杂状态变化的库存模型。

## 二、座位模型：从物理座位到逻辑座位

### 2.1 三层座位模型

12306 的座位系统分为三个层次：

**第一层：物理座位**

物理座位是列车上真实存在的座位。每个座位有固定的属性：车厢号、座位号、座位类型（商务座、一等座、二等座）、位置（靠窗、靠过道、中间）。

```java title="物理座位模型"]
public class PhysicalSeat {
    private String trainNumber;      // 车次号
    private String carriageNumber;   // 车厢号
    private String seatNumber;       // 座位号，如 "12A"
    private SeatType seatType;       // 座位类型
    private SeatPosition position;  // 位置：WINDOW/AISLE/MIDDLE
    private int price;              // 基础票价（里程计价）
}
```

**第二层：逻辑座位**

逻辑座位是物理座位在特定日期、特定行程上的映射。同一个物理座位，在不同日期、不同行程上，可能是不同的逻辑座位。

```java title="逻辑座位模型"]
public class LogicalSeat {
    private String seatId;              // 座位 ID
    private String physicalSeatId;      // 物理座位 ID
    private String departureDate;       // 出发日期
    private String departureStation;    // 出发站
    private String arrivalStation;      // 到达站
    private LogicalSeatStatus status;   // 状态
    private String lockUserId;          // 锁定用户 ID
    private LocalDateTime lockTime;     // 锁定时间
}
```

**第三层：预留座位**

预留座位是用户在购票流程中「锁定」的座位。在用户完成支付之前，这个座位对其他人不可见，也不可购买。

```java title="预留座位模型"]
public class ReservedSeat {
    private String reservationId;       // 预留 ID
    private String logicalSeatId;       // 逻辑座位 ID
    private String userId;             // 用户 ID
    private LocalDateTime reservedTime; // 预留时间
    private LocalDateTime expireTime;   // 过期时间（通常 15-30 分钟）
    private ReservationStatus status;   // 状态：LOCKED/PAID/CANCELLED/EXPIRED
}
```

### 2.2 座位状态机

座位的状态转换是一个复杂的状态机：

```
AVAILABLE（可售）
    │
    ├─→ [用户查询] → AVAILABLE
    │
    ├─→ [用户锁定] → LOCKED（锁定中）
    │
    └─→ [系统释放/过期] → AVAILABLE

LOCKED（锁定中）
    │
    ├─→ [用户支付] → RESERVED（已购）
    │
    ├─→ [用户取消] → AVAILABLE
    │
    └─→ [超时释放] → AVAILABLE

RESERVED（已购）
    │
    ├─→ [用户改签] → RESERVED（新座位） + AVAILABLE（旧座位）
    │
    ├─→ [用户退票] → AVAILABLE
    │
    └─→ [开车后] → ARCHIVED（归档）
```

状态机的核心挑战是**状态转换的原子性**。当一个座位从 AVAILABLE 转换到 LOCKED 时，必须保证没有其他并发请求也将其锁定。这需要分布式锁或乐观锁的支持。

## 三、锁座机制：悲观锁 vs 乐观锁

### 3.1 悲观锁：先锁后查

悲观锁的思路是「先占座，再处理」。当用户选择一个座位时，系统立即锁定该座位，然后检查用户是否有购票资格。

```java title="悲观锁实现"]
@Service
public class PessimisticSeatLockService {
    @Transactional
    public SeatLockResult lockSeat(String userId, String seatId) {
        // 1. 获取分布式锁
        String lockKey = "seat:lock:" + seatId;
        Boolean acquired = redisTemplate.opsForValue()
                .setIfAbsent(lockKey, userId, 15, TimeUnit.MINUTES);

        if (!acquired) {
            return SeatLockResult.FAILED_SEAT_LOCKED;
        }

        try {
            // 2. 查询座位状态
            LogicalSeat seat = seatRepository.findById(seatId);
            if (seat.getStatus() != SeatStatus.AVAILABLE) {
                return SeatLockResult.FAILED_SEAT_UNAVAILABLE;
            }

            // 3. 检查用户是否有购票资格（如学生票配额）
            if (!ticketQuotaService.checkQuota(userId, seat.getSeatType())) {
                return SeatLockResult.FAILED_NO_QUOTA;
            }

            // 4. 锁定座位
            seat.setStatus(SeatStatus.LOCKED);
            seat.setLockUserId(userId);
            seat.setLockTime(LocalDateTime.now());
            seatRepository.save(seat);

            // 5. 创建预留记录
            ReservedSeat reserved = new ReservedSeat();
            reserved.setLogicalSeatId(seatId);
            reserved.setUserId(userId);
            reserved.setReservedTime(LocalDateTime.now());
            reserved.setExpireTime(LocalDateTime.now().plusMinutes(15));
            reservedRepository.save(reserved);

            return SeatLockResult.SUCCESS;
        } catch (Exception e) {
            // 释放锁
            redisTemplate.delete(lockKey);
            throw e;
        }
    }
}
```

悲观锁的优点是「简单直接」，用户看到「座位已锁定」就知道自己有机会。缺点是：如果用户锁定后不支付，座位会被锁定 15 分钟白白浪费。

### 3.2 乐观锁：先查后锁

乐观锁的思路是「先检查，再扣减」。不预先锁定座位，而是在提交订单时通过版本号控制实现乐观并发。

```java title="乐观锁实现"]
@Service
public class OptimisticSeatLockService {
    @Transactional
    public OrderResult createOrder(String userId, String seatId, long version) {
        // 1. 查询座位（带版本号）
        LogicalSeat seat = seatRepository.findByIdWithVersion(seatId, version);

        // 2. 检查座位是否可用
        if (seat.getStatus() != SeatStatus.AVAILABLE) {
            return OrderResult.FAILED_SEAT_UNAVAILABLE;
        }

        // 3. 尝试更新（乐观锁）
        int rows = seatRepository.updateSeatStatus(
                seatId,
                SeatStatus.AVAILABLE,
                SeatStatus.RESERVED,
                version
        );

        if (rows == 0) {
            // 版本号不匹配，说明有并发修改
            return OrderResult.FAILED_CONCURRENT_MODIFICATION;
        }

        // 4. 创建订单
        return createOrderInternal(userId, seat);
    }
}
```

乐观锁的优点是「不浪费座位」，只有真正下单成功的用户才会占用座位。缺点是用户可能在提交订单时失败，体验不如悲观锁。

### 3.3 混合锁策略

12306 实际采用的是混合策略：

- **前端展示层**：乐观锁，查询时显示实时余量
- **下单层**：悲观锁，用户点击购买时锁定座位
- **支付层**：超时释放，15 分钟未支付自动释放

```java title="混合锁策略实现"]
@Service
public class HybridSeatLockService {
    private final RedisTemplate redisTemplate;
    private final SeatRepository seatRepository;
    private final OrderRepository orderRepository;

    /**
     * 第一阶段：查询余票（乐观模式）
     */
    public AvailableSeats queryAvailable(String departure, String arrival, String date) {
        // 从 Redis 获取余票缓存
        String cacheKey = String.format("seats:%s:%s:%s", departure, arrival, date);
        Integer available = (Integer) redisTemplate.opsForValue().get(cacheKey);

        if (available == null || available <= 0) {
            // 缓存未命中或无余票，查询数据库
            available = seatRepository.countAvailable(departure, arrival, date);
            redisTemplate.opsForValue().set(cacheKey, available, 5, TimeUnit.MINUTES);
        }

        return new AvailableSeats(available);
    }

    /**
     * 第二阶段：锁定座位（悲观模式）
     */
    public LockResult lockSeat(String userId, String seatId) {
        // Redis 分布式锁
        String lockKey = "seat:lock:" + seatId;
        Boolean acquired = redisTemplate.opsForValue()
                .setIfAbsent(lockKey, userId, 15, TimeUnit.MINUTES);

        if (!acquired) {
            return LockResult.FAILED_ANOTHER_USER_LOCKED;
        }

        // 数据库乐观锁扣减
        int rows = seatRepository.lockSeat(seatId, userId);
        if (rows == 0) {
            redisTemplate.delete(lockKey);
            return LockResult.FAILED_SEAT_UNAVAILABLE;
        }

        // 更新 Redis 缓存
        decrementSeatCache(departure, arrival, date);

        return LockResult.success(seatId);
    }

    /**
     * 第三阶段：支付成功（最终确认）
     */
    public PayResult confirmPayment(String orderId) {
        Order order = orderRepository.findById(orderId);

        // 确认座位状态
        int rows = seatRepository.confirmSeat(order.getSeatId());
        if (rows == 0) {
            // 座位已被释放，订单取消
            order.setStatus(OrderStatus.CANCELLED);
            orderRepository.save(order);
            return PayResult.FAILED_SEAT_RELEASED;
        }

        // 标记订单已支付
        order.setStatus(OrderStatus.PAID);
        order.setPayTime(LocalDateTime.now());
        orderRepository.save(order);

        return PayResult.SUCCESS;
    }
}
```

## 四、查询优化：如何处理每日百亿次查询

### 4.1 查询的特点

12306 的查询量远大于购票量：

- 查询 QPS：可能达到 100 万/秒
- 购票 QPS：可能只有 1 万/秒
- 查询成功率：90% 以上的查询最终发现无票

这意味着查询优化比购票优化更重要。

### 4.2 多级缓存架构

```java title="多级缓存查询"]
@Service
public class TicketQueryService {
    private final RedisTemplate redisTemplate;
    private final SeatRepository seatRepository;
    private final TrainRepository trainRepository;

    /**
     * 三级缓存查询
     */
    public TicketQueryResult query(String from, String to, String date) {
        // 第一级：CDN 缓存（静态数据，如车站信息）
        // 由前端直接访问 CDN

        // 第二级：Redis 缓存（余票数据）
        String redisKey = String.format("query:%s:%s:%s", from, to, date);
        TicketQueryResult cached = (TicketQueryResult) redisTemplate.opsForValue().get(redisKey);

        if (cached != null) {
            return cached;
        }

        // 第三级：本地缓存（Caffeine）
        cached = localCache.getIfPresent(redisKey);
        if (cached != null) {
            redisTemplate.opsForValue().set(redisKey, cached, 30, TimeUnit.SECONDS);
            return cached;
        }

        // 缓存未命中，查询数据库
        cached = queryFromDatabase(from, to, date);

        // 回填缓存
        localCache.put(redisKey, cached);
        redisTemplate.opsForValue().set(redisKey, cached, 30, TimeUnit.SECONDS);

        return cached;
    }

    /**
     * 查询数据库
     */
    private TicketQueryResult queryFromDatabase(String from, String to, String date) {
        // 按车次分组查询
        List<Train> trains = trainRepository.findByRoute(from, to, date);

        List<TrainTicketInfo> result = new ArrayList<>();
        for (Train train : trains) {
            // 统计各座位类型余量
            Map<SeatType, Integer> seatCounts = seatRepository.countByType(
                    train.getId(), from, to, date);

            result.add(new TrainTicketInfo(train, seatCounts));
        }

        return new TicketQueryResult(result);
    }
}
```

### 4.3 异步查询与预加载

对于热门线路，系统会提前预加载数据到缓存：

```java title="热门线路预加载"]
@Service
public class HotRoutePreloader {
    private final RedisTemplate redisTemplate;
    private final SeatRepository seatRepository;

    /**
     * 预加载热门线路余票
     */
    @Scheduled(fixedRate = 60000)  // 每分钟执行
    public void preloadHotRoutes() {
        // 获取未来 7 天的热门线路
        List<HotRoute> hotRoutes = getHotRoutes();

        for (HotRoute route : hotRoutes) {
            try {
                // 预加载到 Redis
                preloadRoute(route);
            } catch (Exception e) {
                log.error("预加载线路失败: {}", route, e);
            }
        }
    }

    private void preloadRoute(HotRoute route) {
        String redisKey = String.format("query:%s:%s:%s",
                route.getFrom(), route.getTo(), route.getDate());

        // 从数据库查询
        TicketQueryResult result = queryFromDatabase(
                route.getFrom(), route.getTo(), route.getDate());

        // 写入 Redis
        redisTemplate.opsForValue().set(redisKey, result, 5, TimeUnit.MINUTES);
    }
}
```

## 五、排队机制：削峰填谷的艺术

### 5.1 为什么需要排队？

当瞬时请求量远超系统处理能力时，如果不排队会发生什么？

- 大量请求直接打到数据库，数据库连接耗尽
- 系统响应时间急剧上升，用户体验崩溃
- 部分请求超时失败，用户重复发起请求，进一步加剧系统压力

排队机制的作用是「以时间换空间」：让用户在页面等待，而不是让系统崩溃。

### 5.2 排队 vs 实时：优缺点对比

| 维度 | 实时处理 | 排队处理 |
|---|---|---|
| 用户体验 | 快（响应即结果） | 慢（需要等待轮询） |
| 系统压力 | 波动大，峰值高 | 平滑，峰值可控 |
| 成功率控制 | 难以保证 | 可以精确控制 |
| 复杂度 | 较低 | 较高（需要状态管理） |
| 适用场景 | 请求量可预期 | 请求量不可预期 |

### 5.3 排队系统设计

```java title="排队系统实现"]
@Service
public class QueueService {
    private final RedisTemplate redisTemplate;
    private final StringRedisTemplate stringRedisTemplate;

    /**
     * 获取排队位置
     */
    public QueuePosition getQueuePosition(String userId, String trainId) {
        String queueKey = "queue:" + trainId;

        // 获取当前排队人数
        Long position = stringRedisTemplate.opsForZSet()
                .rank(queueKey, userId);

        if (position == null) {
            return QueuePosition.NOT_IN_QUEUE;
        }

        // 获取队列总长度
        Long total = stringRedisTemplate.opsForZSet().zCard(queueKey);

        // 获取预估等待时间（假设每秒处理 100 人）
        long waitSeconds = (position / 100) + 1;

        return new QueuePosition(position + 1, total, waitSeconds);
    }

    /**
     * 进入排队队列
     */
    public QueueResult joinQueue(String userId, String trainId) {
        String queueKey = "queue:" + trainId;

        // 检查是否已在队列中
        Boolean exists = stringRedisTemplate.opsForZSet()
                .score(queueKey, userId) != null;

        if (exists) {
            return QueueResult.ALREADY_IN_QUEUE;
        }

        // 获取当前队列长度
        Long currentSize = stringRedisTemplate.opsForZSet().zCard(queueKey);

        // 检查是否超过队列上限
        if (currentSize >= MAX_QUEUE_SIZE) {
            return QueueResult.QUEUE_FULL;
        }

        // 加入队列（使用时间戳作为分数，保证 FIFO）
        double score = System.currentTimeMillis();
        stringRedisTemplate.opsForZSet().add(queueKey, userId, score);

        // 记录用户信息
        redisTemplate.opsForHash().put("queue:user:" + userId, "trainId", trainId);
        redisTemplate.expire("queue:user:" + userId, 1, TimeUnit.HOURS);

        return QueueResult.success();
    }

    /**
     * 批量获取待处理用户
     */
    public List<String> getNextBatch(String trainId, int batchSize) {
        String queueKey = "queue:" + trainId;

        // 获取最早加入的用户
        Set<String> users = stringRedisTemplate.opsForZSet()
                .range(queueKey, 0, batchSize - 1);

        return users != null ? new ArrayList<>(users) : Collections.emptyList();
    }

    /**
     * 从队列移除用户
     */
    public void removeFromQueue(String userId, String trainId) {
        String queueKey = "queue:" + trainId;
        stringRedisTemplate.opsForZSet().remove(queueKey, userId);
    }
}
```

### 5.4 排队公平性保证

排队系统需要保证公平性：先到先得。使用 Redis ZSet 的分数（时间戳）可以保证这一点。

```java title="公平性保证"]
@Service
public class FairQueueService {
    private final StringRedisTemplate redisTemplate;

    /**
     * 严格 FIFO 的入队操作
     */
    public void enqueueFairly(String userId, String trainId) {
        String queueKey = "queue:" + trainId;

        // 使用 Lua 脚本保证原子性
        String script = """
            local key = KEYS[1]
            local userId = ARGV[1]
            local timestamp = tonumber(ARGV[2])

            -- 检查是否已在队列中
            local score = redis.call('ZSCORE', key, userId)
            if score then
                return 0
            end

            -- 添加到队列
            redis.call('ZADD', key, timestamp, userId)
            return 1
            """;

        redisTemplate.execute(new DefaultRedisScript<>(script, Long.class),
                Collections.singletonList(queueKey),
                userId,
                System.currentTimeMillis());
    }

    /**
     * 批量出队（保持 FIFO）
     */
    public List<String> batchDequeue(String trainId, int count) {
        String queueKey = "queue:" + trainId;

        // 按分数（时间戳）从小到大排序，获取最早入队的用户
        Set<String> users = redisTemplate.opsForZSet()
                .range(queueKey, 0, count - 1);

        if (users == null || users.isEmpty()) {
            return Collections.emptyList();
        }

        // 批量移除
        redisTemplate.opsForZSet().remove(queueKey, users.toArray());

        return new ArrayList<>(users);
    }
}
```

## 六、数据分片：按车次和日期分片

### 6.1 为什么要分片？

单表数据量过大会导致查询性能急剧下降。假设有 10000 个车次，每个车次 1000 个座位，运营 365 天，全量数据是 36.5 亿行。

这么大的数据量，必须通过分片来分散到多个数据库节点。

### 6.2 按车次分片

按车次分片是最常见的策略：相同车次的数据存储在同一个分片中。

```java title="按车次分片"]
@Service
public class ShardingService {
    private final List<SeatRepository> seatRepositories;
    private final int shardingCount;

    /**
     * 计算分片索引
     */
    private int getShardingIndex(String trainNumber) {
        // 使用哈希取模，保证同一车次路由到同一分片
        return Math.abs(trainNumber.hashCode()) % shardingCount;
    }

    /**
     * 获取对应分片的 Repository
     */
    private SeatRepository getSeatRepository(String trainNumber) {
        int index = getShardingIndex(trainNumber);
        return seatRepositories.get(index);
    }

    /**
     * 查询某车次的余票
     */
    public List<LogicalSeat> querySeats(String trainNumber, String from, String to) {
        SeatRepository repository = getSeatRepository(trainNumber);
        return repository.findAvailable(trainNumber, from, to);
    }

    /**
     * 锁定座位
     */
    public boolean lockSeat(String seatId) {
        LogicalSeat seat = seatRepository.findById(seatId);
        SeatRepository repository = getSeatRepository(seat.getTrainNumber());
        return repository.lockSeat(seatId) > 0;
    }
}
```

```java title="跨分片查询"]
@Service
public class MultiLegQueryService {
    private final ShardingService shardingService;

    /**
     * 多程查询：查询从 A 到 B 的所有可能路径
     */
    public List<JourneyOption> queryMultiLeg(String from, String to, String date) {
        List<JourneyOption> results = new ArrayList<>();

        // 1. 查询从 A 出发的所有车次
        List<Train> departingTrains = trainService.queryDeparting(from, date);

        for (Train train : departingTrains) {
            // 2. 查询该车次是否经过目的地
            Optional<String> arrivalStation = train.getStations().stream()
                    .filter(s -> s.equals(to))
                    .findFirst();

            if (arrivalStation.isPresent()) {
                // 3. 查询余票
                List<LogicalSeat> seats = shardingService.querySeats(
                        train.getNumber(), from, to);

                if (!seats.isEmpty()) {
                    results.add(new JourneyOption(train, from, to, seats));
                }
            }
        }

        // 4. 中转换乘查询
        List<TransferOption> transfers = queryTransfers(from, to, date);
        results.addAll(transfers);

        // 按时间排序
        results.sort(Comparator.comparing(JourneyOption::getDepartureTime));

        return results;
    }

    /**
     * 查询中转换乘方案
     */
    private List<TransferOption> queryTransfers(String from, String to, String date) {
        // 查询所有在 A 有票且在 B 有票的换乘方案
        List<TransferOption> transfers = new ArrayList<>();

        // 获取所有在 B 到达的车次
        List<Train> arrivingTrains = trainService.queryArriving(to, date);

        for (Train departTrain : departingTrains) {
            for (Train arriveTrain : arrivingTrains) {
                // 检查换乘时间是否合理（30-120 分钟）
                if (isValidTransfer(departTrain, arriveTrain)) {
                    // 查询两个车次的余票
                    List<LogicalSeat> seat1 = querySeats(...);
                    List<LogicalSeat> seat2 = querySeats(...);

                    if (!seat1.isEmpty() && !seat2.isEmpty()) {
                        transfers.add(new TransferOption(departTrain, arriveTrain,
                                seat1, seat2));
                    }
                }
            }
        }

        return transfers;
    }
```

## 七、支付超时与座位释放

### 7.1 超时释放机制

用户锁定座位后，需要在规定时间内完成支付。超时后，座位自动释放给其他用户。

```java title="超时释放实现"]
@Service
public class SeatReleaseService {
    private final SeatRepository seatRepository;
    private final RedisTemplate redisTemplate;
    private final OrderRepository orderRepository;

    /**
     * 处理超时未支付的座位
     */
    @Scheduled(fixedRate = 10000)  // 每 10 秒执行
    public void releaseExpiredSeats() {
        log.info("开始处理超时座位释放...");

        // 1. 查询所有过期的预留
        LocalDateTime now = LocalDateTime.now();
        List<ReservedSeat> expired = reservedSeatRepository
                .findByStatusAndExpireTimeBefore(ReservationStatus.LOCKED, now);

        log.info("发现 {} 个过期预留", expired.size());

        for (ReservedSeat reserved : expired) {
            try {
                releaseSeat(reserved);
            } catch (Exception e) {
                log.error("释放座位失败: {}", reserved.getId(), e);
            }
        }
    }

    /**
     * 释放座位
     */
    @Transactional
    public void releaseSeat(ReservedSeat reserved) {
        // 1. 更新预留状态
        reserved.setStatus(ReservationStatus.EXPIRED);
        reservedSeatRepository.save(reserved);

        // 2. 更新座位状态
        LogicalSeat seat = seatRepository.findById(reserved.getLogicalSeatId());
        seat.setStatus(SeatStatus.AVAILABLE);
        seat.setLockUserId(null);
        seat.setLockTime(null);
        seatRepository.save(seat);

        // 3. 释放分布式锁
        String lockKey = "seat:lock:" + seat.getId();
        redisTemplate.delete(lockKey);

        // 4. 更新缓存
        updateSeatCache(seat);

        // 5. 记录日志
        log.info("座位已释放: seatId={}, userId={}, expireReason=TIMEOUT",
                seat.getId(), reserved.getUserId());
    }
}
```

### 7.2 退款与座位释放

```java title="退票释放座位"]
@Service
public class RefundService {
    @Transactional
    public RefundResult refund(String orderId) {
        Order order = orderRepository.findById(orderId);

        if (order.getStatus() != OrderStatus.PAID) {
            return RefundResult.FAILED_INVALID_STATUS;
        }

        // 1. 检查是否可以退票（如开车后一定时间内不可退）
        if (isAfterDeadline(order)) {
            return RefundResult.FAILED_PAST_DEADLINE;
        }

        // 2. 计算退款金额（根据开车时间扣除手续费）
        BigDecimal refundAmount = calculateRefund(order);

        // 3. 释放座位
        LogicalSeat seat = seatRepository.findById(order.getSeatId());
        seat.setStatus(SeatStatus.AVAILABLE);
        seatRepository.save(seat);

        // 4. 更新缓存
        updateSeatCache(seat);

        // 5. 更新订单状态
        order.setStatus(OrderStatus.REFUNDED);
        order.setRefundAmount(refundAmount);
        order.setRefundTime(LocalDateTime.now());
        orderRepository.save(order);

        // 6. 触发退款流程
        paymentService.processRefund(order.getPaymentId(), refundAmount);

        return RefundResult.success(refundAmount);
    }
}
```

## 八、热点车次预判与资源预分配

### 8.1 为什么需要预判？

春节期间的热门线路（如北京到哈尔滨）会在放票瞬间被抢光。如果系统能提前预判哪些车次是热点，可以提前进行资源准备和流量控制。

### 8.2 热点预测模型

```java title="热点预测服务"]
@Service
public class HotRoutePredictor {
    private final TrainRepository trainRepository;
    private final HistoricalDataRepository historicalRepository;

    /**
     * 预测热点线路
     */
    public List<HotRoute> predictHotRoutes(int daysAhead) {
        List<HotRoute> predictions = new ArrayList<>();

        // 1. 获取历史同期数据
        LocalDate targetDate = LocalDate.now().plusDays(daysAhead);
        List<HistoricalTicketSales> history = historicalRepository
                .findBySimilarPeriod(targetDate);

        // 2. 分析历史抢票模式
        Map<String, Double> routeHotness = new HashMap<>();
        for (HistoricalTicketSales record : history) {
            String route = record.getFrom() + "->" + record.getTo();

            // 综合考虑：抢票速度、售罄时间、候补人数
            double hotness = calculateHotness(record);
            routeHotness.merge(route, hotness, Double::sum);
        }

        // 3. 结合当前预约数据
        Map<String, Integer> reservations = getCurrentReservations(targetDate);

        // 4. 计算最终热度分数
        for (Map.Entry<String, Double> entry : routeHotness.entrySet()) {
            String route = entry.getKey();
            int reservationCount = reservations.getOrDefault(route, 0);

            double finalScore = entry.getValue() * 0.6 + reservationCount * 0.4;

            if (finalScore > HOT_THRESHOLD) {
                predictions.add(new HotRoute(route, finalScore, reservationCount));
            }
        }

        // 5. 按热度排序
        predictions.sort((a, b) -> Double.compare(b.getScore(), a.getScore()));

        return predictions;
    }

    private double calculateHotness(HistoricalTicketSales record) {
        // 售罄速度分数（越快越热）
        double selloutSpeed = 1.0 / (record.getSelloutHours() + 1);

        // 候补人数分数
        double waitingScore = Math.log(record.getWaitingCount() + 1);

        // 查询量分数
        double queryScore = Math.log(record.getQueryCount() + 1);

        return selloutSpeed * 0.5 + waitingScore * 0.3 + queryScore * 0.2;
    }
}
```

## 十、系统演进：从 12306 的架构变迁看设计哲学

### 10.1 12306 的架构演进历程

| 阶段 | 时间 | 架构特点 | 解决的问题 |
|---|---|---|---|
| 1.0 | 2010 年 | 传统单体架构 | 基础功能上线 |
| 2.0 | 2012 年 | 引入 CDN、静态化 | 查询压力分流 |
| 3.0 | 2015 年 | 异步架构、排队系统 | 抢票高峰崩溃 |
| 4.0 | 2018 年 | 云原生、分布式 | 弹性扩展 |
| 5.0 | 2022 年 | AI 预测、实时推荐 | 智能化服务 |

### 10.2 架构设计的核心哲学

**第一：分层防御**

不要依赖单一方案解决问题。12306 的架构是「CDN → 静态页 → 排队 → 缓存 → 数据库」，每一层都承担一部分压力。

**第二：读写分离**

查询量是购票量的 100 倍，但两者可以完全分离。查询走 CDN/缓存/只读库，购票走独立的事务处理链路。

**第三：异步为王**

高峰期让用户等待是可以接受的（排队），但让系统崩溃是不可接受的。异步处理是保持系统稳定的关键。

**第四：提前预判**

12306 现在会提前预测热点线路，并提前进行资源准备。好的系统不是临时抱佛脚，而是提前布局。

## 面试追问：12306 系统高频问题

### 问题一：如何保证购票的公平性？

**参考答案**：

购票公平性的核心是「先到先得」，实现方式包括：

**排队机制**：用户进入购票流程前先进入排队队列，按入队时间（FIFO）顺序处理。Redis ZSet 的分数机制可以保证顺序。

**限流机制**：限制单个用户/IP 的请求频率，防止恶意刷票。

**验证码机制**：在购票关键环节加入验证码，防止自动化脚本。

**设备指纹**：识别同一设备的不同账号，防止黄牛党。

### 问题二：如何设计一个支持中转换乘的查询系统？

**参考答案**：

中转换乘查询的核心是「图搜索」问题：

```java
// 1. 构建车站-车次有向图
Map<String, List<Train>> stationGraph = buildStationGraph();

// 2. 广度优先搜索（BFS）寻找换乘路径
public List<TransferPath> findTransfers(String from, String to) {
    Queue<TransferPath> queue = new LinkedList<>();
    Set<String> visited = new HashSet<>();

    // 初始节点：所有从起点出发的车次
    for (Train train : getTrainsFrom(from)) {
        queue.offer(new TransferPath(train, from));
    }

    while (!queue.isEmpty()) {
        TransferPath current = queue.poll();
        String currentStation = current.getLastStation();

        if (currentStation.equals(to)) {
            // 找到终点，返回路径
            return current.getPath();
        }

        // 遍历所有换乘站
        for (String nextStation : getNextStations(currentStation, current.getLastTrain())) {
            if (!visited.contains(nextStation)) {
                visited.add(nextStation);

                // 换乘到新车次
                for (Train nextTrain : getTrainsFrom(nextStation)) {
                    queue.offer(current.extend(nextTrain, nextStation));
                }
            }
        }
    }

    return Collections.emptyList();
}
```

### 问题三：超卖问题在 12306 中如何解决？

**参考答案**：

12306 的超卖防护比普通秒杀更复杂，因为座位是独立的：

**分布式锁**：使用 Redis 分布式锁锁定座位，保证同一座位同一时间只有一个用户可以操作。

**乐观锁 + 版本号**：数据库操作使用乐观锁，通过版本号控制并发更新。

**状态机约束**：座位状态转换有严格的状态机约束，只能从 AVAILABLE → LOCKED → RESERVED，不能跳过或回退。

**异步对账**：定时检查座位状态和订单状态的一致性，发现异常及时告警。

## 总结

12306 是世界上访问量最大的票务系统之一，它面临的挑战远超普通电商系统。

关键要点回顾：

- **三层座位模型**：物理座位 → 逻辑座位 → 预留座位，每层都有独立的状态管理
- **混合锁策略**：查询用乐观锁，下单用悲观锁，支付后最终确认
- **多级缓存**：CDN → Redis → 本地缓存，逐层减少数据库压力
- **排队机制**：以用户等待换系统稳定，以时间换空间
- **数据分片**：按车次/日期分片，避免单点瓶颈
- **超时释放**：座位锁定后必须在规定时间内支付，否则自动释放

12306 的架构设计没有银弹，每一层优化都是针对具体问题的解决方案。理解这些设计的本质，才能举一反三，应对其他复杂系统。
