# 分布式限流：Redis + Lua 令牌桶实现

单机限流有一个致命的缺陷：**每个实例独立限流，互不知情**。

假设你有 10 个服务实例，每个实例限流 100 QPS。在极端情况下，总 QPS 可能达到 1000——是限流阈值的 10 倍。

这就是分布式限流存在的意义：**把限流决策「集中化」，让所有实例共享同一个限流器**。

Redis 是分布式限流的首选存储，原因很简单：快、支持原子操作、天然支持过期时间。这篇文章，我们来聊聊如何用 Redis + Lua 实现高性能的分布式限流。

## 一、为什么单机限流不够

### 1.1 多实例场景下的「孤岛效应」

单机限流器是「孤岛」——每个实例只知道自己有多少请求，不知道其他实例在干什么。

看一个具体场景：秒杀系统，限流阈值是 10000 QPS，有 100 个服务实例。

- 单机限流：每个实例限流 100 QPS
- 总限流能力：100 × 100 = 10000 QPS
- 实际情况：某些实例可能只收到 50 QPS，另一些收到 150 QPS
- 极端情况：流量不均衡时，总请求量可能远超 10000 QPS

更严重的是：假设某个实例重启，它的限流状态丢失了，请求会瞬间涌入。

### 1.2 分布式限流的核心目标

分布式限流要解决两个问题：

1. **全局一致性**：所有实例共享同一个限流状态
2. **高可用性**：限流器本身不能成为单点故障

分布式限流不是简单的「把计数器放到 Redis 里」，而是需要考虑：

- 限流算法的精度（固定窗口、滑动窗口、令牌桶……）
- 网络延迟的影响
- Redis 集群的分布式特性
- Lua 脚本的原子性保证

### 1.3 Redis 为什么是最佳选择

Redis 是分布式限流的事实标准，原因如下：

| 特性 | 说明 |
|---|---|
| 高性能 | 单实例可达数十万 QPS，延迟 < 1ms |
| 原子操作 | MULTI/EXEC、Lua 脚本保证原子性 |
| 过期机制 | TTL 天然支持滑动窗口的过期清理 |
| 集群支持 | Redis Cluster 支持数据分片 |
| 丰富的数据结构 | String、Hash、ZSet 满足各种限流算法 |

:::tip 分布式限流的常见架构

**中心化架构**：限流逻辑放在 API 网关或独立的限流服务，所有请求都经过它。

**去中心化架构**：限流逻辑嵌入每个服务实例，通过 Redis 共享状态。

两种架构各有优劣：中心化架构精度高但有单点风险；去中心化架构可用性高但精度略低。

:::

## 二、Redis + Lua 限流的原理

### 2.1 为什么用 Lua 脚本

Redis 的 Lua 脚本是**原子执行**的——整个脚本在执行过程中不会被其他命令打断。

这对于限流至关重要。假设不用 Lua：

```
// 伪代码：普通 Redis 命令
current = redis.get(key)      // T1 时刻
if current < limit:
    redis.incr(key)           // T2 时刻
    return allowed
else:
    return denied
```

在高并发下，T1 和 T2 之间可能有其他请求「插队」，导致计数不准确。这叫**竞态条件（Race Condition）**。

用 Lua 脚本：

```lua
-- Lua 脚本：原子执行
local current = redis.call('GET', KEYS[1])
if current and tonumber(current) >= tonumber(ARGV[1]) then
    return 0  -- 拒绝
else
    redis.call('INCR', KEYS[1])
    redis.call('EXPIRE', KEYS[1], ARGV[2])
    return 1  -- 允许
end
```

T1 和 T2 之间的所有操作被打包成一个原子单元，不存在竞态条件。

### 2.2 三种限流算法在 Redis 上的实现

接下来，我们分别实现固定窗口、滑动窗口、令牌桶三种算法的 Redis + Lua 版本。

## 三、固定窗口限流的 Redis 实现

### 3.1 算法原理

固定窗口是最简单的限流方式：

1. 把时间划分为固定大小的窗口（如每秒一个窗口）
2. 每个窗口独立计数
3. 超过阈值就拒绝

Redis 实现思路：使用 `key` = `限流器:窗口ID`，窗口 ID = `当前时间戳 / 窗口长度`。

### 3.2 Lua 脚本实现

```lua
-- fixed_window.lua
-- KEYS[1] = 限流 key
-- ARGV[1] = 限流阈值
-- ARGV[2] = 窗口长度（秒）

local key = KEYS[1]
local limit = tonumber(ARGV[1])
local window_size = tonumber(ARGV[2])

-- 获取当前窗口的计数器
local current = redis.call('GET', key)

if current and tonumber(current) >= limit then
    -- 超过阈值，拒绝
    return 0
end

-- 计数器 +1
local count = redis.call('INCR', key)

-- 设置过期时间（窗口长度 + 1 秒，防止边界问题）
if count == 1 then
    redis.call('EXPIRE', key, window_size + 1)
end

return 1
```

### 3.3 Java 客户端调用

```java
public class FixedWindowRateLimiter {
    private final StringRedisTemplate redisTemplate;
    private final String luaScript;

    public FixedWindowRateLimiter(StringRedisTemplate redisTemplate) {
        this.redisTemplate = redisTemplate;
        // 加载 Lua 脚本
        this.luaScript = loadScript("fixed_window.lua");
    }

    public boolean tryAcquire(String key, int limit, int windowSeconds) {
        String fullKey = "rate:fixed:" + key;
        RedisScript<Long> script = RedisScript.of(luaScript, Long.class);
        Long result = redisTemplate.execute(
            script,
            Collections.singletonList(fullKey),
            String.valueOf(limit),
            String.valueOf(windowSeconds)
        );
        return result != null && result == 1;
    }

    private String loadScript(String filename) {
        try (InputStream is = getClass().getClassLoader().getResourceAsStream(filename)) {
            return new String(is.readAllBytes(), StandardCharsets.UTF_8);
        } catch (IOException e) {
            throw new RuntimeException("Failed to load Lua script", e);
        }
    }
}
```

### 3.4 缺陷：边界突变问题

固定窗口的边界突变问题在分布式环境下依然存在。假设限流阈值是 100 QPS，窗口长度是 1 秒：

- 0:59:00 - 0:59:59：100 个请求（窗口 1 满了）
- 1:00:00 - 1:00:09：100 个请求（窗口 2 也满了）
- 实际 10 秒内处理了 200 个请求

**这是固定窗口的固有缺陷**，无法在算法层面解决。解决方案是使用滑动窗口。

## 四、滑动窗口限流的 Redis 实现

### 4.1 算法原理

滑动窗口的核心思想是：**不再以固定的时间窗口计数，而是精确计算滑动窗口内的请求数**。

Redis 实现思路：使用 **ZSet（有序集合）**，key 是限流 key，member 是请求 ID（时间戳 + 随机数），score 是请求时间戳。

### 4.2 Lua 脚本实现

```lua
-- sliding_window_log.lua
-- KEYS[1] = 限流 key
-- ARGV[1] = 限流阈值
-- ARGV[2] = 窗口长度（毫秒）
-- ARGV[3] = 当前时间戳（毫秒）

local key = KEYS[1]
local limit = tonumber(ARGV[1])
local window_size = tonumber(ARGV[2])
local now = tonumber(ARGV[3])

-- 滑动窗口的起始时间
local window_start = now - window_size

-- 1. 删除滑动窗口之外的记录
redis.call('ZREMRANGEBYSCORE', key, '-inf', window_start)

-- 2. 获取当前窗口内的请求数
local current_count = redis.call('ZCARD', key)

-- 3. 检查是否超限
if current_count >= limit then
    return 0  -- 拒绝
end

-- 4. 添加当前请求（使用时间戳作为 member）
local member = now .. ':' .. math.random(1000000)
redis.call('ZADD', key, now, member)

-- 5. 设置过期时间
redis.call('PEXPIRE', key, window_size)

return 1
```

### 4.3 Java 客户端调用

```java
public class SlidingWindowLogRateLimiter {
    private final StringRedisTemplate redisTemplate;
    private final String luaScript;

    public SlidingWindowLogRateLimiter(StringRedisTemplate redisTemplate) {
        this.redisTemplate = redisTemplate;
        this.luaScript = loadScript("sliding_window_log.lua");
    }

    public boolean tryAcquire(String key, int limit, int windowMillis) {
        String fullKey = "rate:sliding:" + key;
        long now = System.currentTimeMillis();

        RedisScript<Long> script = RedisScript.of(luaScript, Long.class);
        Long result = redisTemplate.execute(
            script,
            Collections.singletonList(fullKey),
            String.valueOf(limit),
            String.valueOf(windowMillis),
            String.valueOf(now)
        );
        return result != null && result == 1;
    }
}
```

### 4.4 优点与代价

**优点**：

- 精度高，精确到每个请求
- 解决了边界突变问题
- 实现相对简单

**代价**：

- 内存消耗：每个请求都要存储一个 ZSet 成员（时间戳 + member）
- ZSet 操作（ZADD、ZREMRANGEBYSCORE、ZCARD）的复杂度是 O(log N)

### 4.5 内存优化：滑动窗口计数器

如果 ZSet 的内存消耗太大，可以用**滑动窗口计数器**代替。

思路：将滑动窗口划分为 N 个子窗口，每个子窗口独立计数。

```lua
-- sliding_window_counter.lua
-- KEYS[1] = 限流 key
-- ARGV[1] = 限流阈值
-- ARGV[2] = 当前时间戳（秒）
-- ARGV[3] = 窗口大小（秒）

local key = KEYS[1]
local limit = tonumber(ARGV[1])
local now = tonumber(ARGV[2])
local window_size = tonumber(ARGV[3])

-- 计算当前子窗口
local current_bucket = now % window_size

-- 1. 清理过期桶
local expired_time = now - window_size
for i = 0, window_size - 1 do
    local bucket_key = key .. ':' .. i
    local last_time = redis.call('GET', bucket_key .. ':time')
    if last_time and tonumber(last_time) < expired_time then
        redis.call('DEL', bucket_key)
        redis.call('DEL', bucket_key .. ':time')
    end
end

-- 2. 检查 + 计数
local bucket_key = key .. ':' .. current_bucket
local count = redis.call('INCR', bucket_key)
if count == 1 then
    redis.call('SET', bucket_key .. ':time', now)
end

-- 3. 设置过期
redis.call('EXPIRE', bucket_key, window_size * 2)

-- 4. 计算窗口内的总数
local total = 0
for i = 0, window_size - 1 do
    local cnt = redis.call('GET', key .. ':' .. i)
    if cnt then
        total = total + tonumber(cnt)
    end
end

return total
```

## 五、令牌桶限流的 Redis 实现

### 5.1 算法原理

令牌桶是最常用的限流算法，需要维护两个状态：

- `tokens`：桶中剩余的令牌数
- `lastRefillTime`：上一次补充令牌的时间

Redis 实现思路：使用 Hash 存储这两个字段。

### 5.2 Lua 脚本实现

```lua
-- token_bucket.lua
-- KEYS[1] = 限流 key
-- ARGV[1] = 桶容量
-- ARGV[2] = 每秒补充的令牌数
-- ARGV[3] = 请求的令牌数
-- ARGV[4] = 当前时间戳（毫秒）

local key = KEYS[1]
local capacity = tonumber(ARGV[1])
local refill_rate = tonumber(ARGV[2])
local requested = tonumber(ARGV[3])
local now = tonumber(ARGV[4])

-- 获取当前状态
local bucket = redis.call('HMGET', key, 'tokens', 'lastRefillTime')
local tokens = tonumber(bucket[1]) or capacity
local last_refill = tonumber(bucket[2]) or now

-- 1. 计算应该补充多少令牌
local elapsed = (now - last_refill) / 1000.0
local new_tokens = math.min(capacity, tokens + elapsed * refill_rate)

-- 2. 检查是否有足够的令牌
if new_tokens >= requested then
    -- 有足够的令牌
    redis.call('HMSET', key, 'tokens', new_tokens - requested, 'lastRefillTime', now)
    redis.call('PEXPIRE', key, 60000) -- 1 分钟过期
    return 1  -- 允许
else
    -- 没有足够的令牌
    redis.call('HMSET', key, 'tokens', new_tokens, 'lastRefillTime', now)
    redis.call('PEXPIRE', key, 60000)
    return 0  -- 拒绝
end
```

### 5.3 Java 客户端调用

```java
public class TokenBucketRateLimiter {
    private final StringRedisTemplate redisTemplate;
    private final String luaScript;

    public TokenBucketRateLimiter(StringRedisTemplate redisTemplate) {
        this.redisTemplate = redisTemplate;
        this.luaScript = loadScript("token_bucket.lua");
    }

    /**
     * 尝试获取令牌
     * @param key 限流 key
     * @param capacity 桶容量
     * @param refillRate 每秒补充的令牌数
     * @param requested 请求的令牌数（通常为 1）
     * @return 是否获取成功
     */
    public boolean tryAcquire(String key, long capacity, double refillRate, int requested) {
        String fullKey = "rate:token:" + key;
        long now = System.currentTimeMillis();

        RedisScript<Long> script = RedisScript.of(luaScript, Long.class);
        Long result = redisTemplate.execute(
            script,
            Collections.singletonList(fullKey),
            String.valueOf(capacity),
            String.valueOf(refillRate),
            String.valueOf(requested),
            String.valueOf(now)
        );
        return result != null && result == 1;
    }

    /**
     * 创建每秒 N 个令牌的限流器
     */
    public static TokenBucketRateLimiter of(StringRedisTemplate redisTemplate,
                                             int permitsPerSecond) {
        return new TokenBucketRateLimiter(redisTemplate);
    }
}
```

### 5.4 分布式令牌桶的精度问题

Redis 令牌桶有一个潜在问题：**分布式环境下的时钟不同步**。

假设 Redis 服务器和业务服务器的时钟有偏差，会导致令牌补充不准确。

**解决方案**：

1. **使用 Redis 服务器的时间**：通过 Lua 脚本中获取 `redis.call('TIME')` 的时间
2. **接受一定误差**：分布式限流的精度本身就无法做到 100%，允许少量误差

## 六、Key 设计最佳实践

### 6.1 Key 的层次结构

限流 Key 的设计需要考虑以下因素：

| 维度 | 说明 | 示例 |
|---|---|---|
| 用户维度 | 按用户 ID 限流 | `rate:user:{userId}` |
| 接口维度 | 按接口限流 | `rate:api:/order/create` |
| IP 维度 | 按 IP 限流 | `rate:ip:{ip}` |
| 组合维度 | 多维度组合 | `rate:user:{userId}:api:{api}` |

```java
// 用户级限流
public boolean limitByUser(String userId, int limit) {
    String key = "rate:user:" + userId;
    return tryAcquire(key, limit);
}

// 接口级限流
public boolean limitByApi(String api, int limit) {
    String key = "rate:api:" + api;
    return tryAcquire(key, limit);
}

// 用户 + 接口组合限流
public boolean limitByUserAndApi(String userId, String api, int limit) {
    String key = "rate:" + userId + ":" + api;
    return tryAcquire(key, limit);
}
```

### 6.2 Key 的过期时间

Key 必须设置过期时间，否则会产生内存泄漏。

| 算法 | 过期时间建议 |
|---|---|
| 固定窗口 | 窗口长度 × 2 |
| 滑动窗口 | 窗口长度 × 2 |
| 令牌桶 | 1 分钟（token 长期有效） |

### 6.3 Key 的前缀隔离

生产环境可能有多个服务共享同一个 Redis，建议使用前缀隔离：

```java
// 不同服务使用不同的前缀
private String getKey(String prefix, String type, String identifier) {
    return String.format("app:%s:rate:%s:%s", appName, type, identifier);
}
```

## 七、Redis Cluster 分布式限流的挑战

### 7.1 哈希槽问题

Redis Cluster 将数据分布在 16384 个哈希槽中。这意味着**同一个 Key 的所有操作必须在同一个节点上**。

Redis Cluster 官方不支持跨槽的 Lua 脚本。如果限流 Key 分散在多个槽，无法用单个 Lua 脚本完成原子操作。

### 7.2 解决方案：Tag 哈希

Redis Cluster 支持 `hash tags`，用于强制将多个 Key 放在同一个槽。

```lua
-- 使用 hash tag
local key = '{rate}:user:' .. userId
```

`{rate}` 告诉 Redis，所有以 `{rate}` 开头的 Key 都哈希到同一个槽。

### 7.3 跨节点限流的妥协方案

如果无法使用 hash tag，有两种妥协方案：

**方案一：单节点限流 + 数据同步**

```
                   +-----------------+
                   |   Redis Proxy   |
                   |  (限流逻辑)      |
                   +-----------------+
                          |
        +-----------------+-----------------+
        |                 |                 |
   +----v----+       +----v----+       +----v----+
   | Node 1  |       | Node 2  |       | Node 3  |
   +---------+       +---------+       +---------+
```

在 Proxy 层做限流，每个 Proxy 实例独立限流，但总限流能力 = N × 单机限流能力。

**方案二：接受分布式误差**

使用多个 Key，每个 Key 独立限流，总限流能力约等于 N × 单机限流能力。

```java
// 把限额分散到多个 Key
for (int i = 0; i < 3; i++) {
    String key = "rate:shard:" + i + ":" + userId;
    if (tryAcquire(key, limit / 3)) {
        return true;
    }
}
return false;
```

### 8.3 限流异常处理

限流器可能出现 Redis 不可用的情况。处理策略：

```java
public boolean tryAcquire(String key, int permits) {
    try {
        return rateLimiter.tryAcquire(key, permits);
    } catch (RedisConnectionException e) {
        // Redis 不可用时，默认放行（fail-open）
        // 也可以选择 fail-closed（默认拒绝）
        log.warn("Redis unavailable, allow request by default", e);
        return true;
    }
}
```

:::warning Fail-Open vs Fail-Closed

**Fail-Open（默认放行）**：Redis 不可用时，允许所有请求通过。优点是用户体验好，缺点是可能冲垮系统。

**Fail-Closed（默认拒绝）**：Redis 不可用时，拒绝所有请求。优点是保护系统，缺点是用户体验差。

**建议**：核心接口（支付、下单）使用 Fail-Closed；普通接口使用 Fail-Open。

:::

## 九、面试追问：分布式限流相关问题

### 9.1 问题一：Redis + Lua 限流有什么优势？

**参考答案**：

Redis + Lua 限流有三个核心优势：

1. **原子性**：Lua 脚本在 Redis 中原子执行，不存在竞态条件
2. **高性能**：网络往返只需一次，减少了 RTT（Round Trip Time）
3. **原子性 + 原子性**：Lua 脚本本身是原子的，可以在一个脚本中完成多个 Redis 命令

### 9.2 问题二：固定窗口、滑动窗口、令牌桶，在 Redis 上如何选择？

**参考答案**：

| 算法 | Redis 数据结构 | 内存消耗 | 精度 | 推荐场景 |
|---|---|---|---|---|
| 固定窗口 | String | 低 | 低（边界问题） | 简单场景、初步限流 |
| 滑动窗口 | ZSet | 高 | 高 | 精确限流 |
| 令牌桶 | Hash | 中 | 高 | 突发流量控制 |

**推荐**：大多数场景使用**令牌桶**，因为它支持突发流量，用户体验最好。

### 9.3 问题三：分布式限流如何保证高精度？

**参考答案**：

分布式限流无法做到 100% 精确，因为存在以下误差来源：

1. **网络延迟**：请求到达限流器的延迟
2. **时钟偏差**：不同服务器的时钟不完全同步
3. **Redis 集群延迟**：跨节点操作有额外延迟

**提高精度的方法**：

- 使用 Redis 服务器的时间（`redis.call('TIME')`）
- 使用滑动窗口代替固定窗口
- 使用 Token Bucket 的「预消费」机制平滑流量

### 9.4 问题四：Redis Cluster 下如何做分布式限流？

**参考答案**：

Redis Cluster 下做分布式限流有两个方案：

1. **使用 Hash Tag**：强制将多个 Key 放在同一个槽，确保 Lua 脚本原子执行
2. **Proxy 层限流**：在 Proxy 层做限流，每个 Proxy 实例独立限流

如果无法使用 Hash Tag，建议在应用层做聚合：将多个请求合并成一个限流请求，减少跨槽操作。

## 延伸思考

### 题目一：分布式限流的精度 vs 可用性

分布式限流存在精度问题：理论上 10000 QPS 的限流，实际可能达到 11000 或 12000 QPS。你能接受这个误差吗？

**提示**：考虑业务场景的容忍度，以及是否需要在应用层做二次限流。

### 题目二：Redis 不可用时的限流策略

当 Redis 不可用时，你的限流策略是「默认放行」还是「默认拒绝」？如何根据业务场景选择？

**提示**：考虑核心接口 vs 普通接口的区别。

## 总结

Redis + Lua 是分布式限流的标准实现：

- **固定窗口**：简单，但有边界问题
- **滑动窗口**：精度高，但内存消耗大
- **令牌桶**：推荐，支持突发且内存消耗适中

在生产环境中，需要考虑：

- **Key 设计**：用户维度、接口维度、组合维度
- **过期时间**：防止内存泄漏
- **集群支持**：Hash Tag 或 Proxy 层限流
- **容错策略**：Fail-Open vs Fail-Closed

分布式限流不是银弹，它解决的是「全局一致性」的问题。在实际项目中，分布式限流 + 单机限流的两层防护是最稳妥的方案。
