# Sentinel 熔断降级：RT、异常比例、异常数策略

你有没有遇到过这种情况：系统正常运行时，某个接口的响应时间从 50ms 突然飙升到 5 秒，用户的请求堆积，服务开始雪崩。

限流只能控制「请求数量」，但控制不了「请求质量」。如果服务本身变慢了，再多的限流也救不了你——你需要的是**熔断降级**。

熔断降级的核心是「快速失败」：当服务出现故障时，不再调用它，而是直接返回降级响应，让服务有时间恢复。

这篇文章，我们来聊聊 Sentinel 的熔断降级功能。

## 一、熔断 vs 限流：两个不同的保护层

在讨论 Sentinel 的熔断降级之前，先搞清楚两个概念的区别：

| 维度 | 限流 | 熔断 |
|---|---|---|
| 保护对象 | 系统容量（不被冲垮） | 服务质量（不被拖垮） |
| 触发条件 | 请求量超过阈值 | 服务响应异常（慢/失败） |
| 行为 | 拒绝请求，排队等待 | 快速失败，不再调用 |
| 恢复 | 请求量降低后自动恢复 | 服务恢复后自动恢复 |

**限流是「守门员」**：告诉你「你不能进来」。

**熔断是「医生」**：告诉你「你需要休息」。

两者配合，构成完整的保护体系：

```
请求 → [限流] → [熔断] → [服务]
        ↓          ↓
     拒绝请求   降级返回
```

### 1.1 什么时候需要熔断

熔断主要用于以下场景：

- 依赖的外部服务响应慢（如第三方支付 API）
- 依赖的数据库查询慢（如复杂 SQL）
- 依赖的服务不可用（如服务宕机）
- 依赖的资源耗尽（如连接池满）

**限流解决不了的问题**：假设下游服务变慢了，限流只能控制请求数量，但如果每个请求都要等待 5 秒超时，线程资源会被耗尽。熔断可以让你不再等待，直接返回降级响应。

## 二、Sentinel 熔断降级的三种策略

Sentinel 支持三种熔断策略：

| 策略 | 说明 | 适用场景 |
|---|---|---|
| **RT 熔断** | 平均响应时间超过阈值时熔断 | 依赖慢服务的场景 |
| **异常比例熔断** | 请求异常比例超过阈值时熔断 | 依赖不稳定服务的场景 |
| **异常数熔断** | 异常数量超过阈值时熔断 | 依赖不可靠服务的场景 |

### 2.1 RT 熔断：平均响应时间

RT（Response Time）熔断策略关注的是**响应时间**。

**触发条件**：当滑动窗口内的**平均响应时间**超过阈值，且**最小请求数**满足时，触发熔断。

```java
// RT 熔断配置
DegradeRule rule = new DegradeRule("orderService")
    .setGrade(RuleConstant.DEGRADE_GRADE_RT)  // RT 熔断
    .setCount(100)                            // 阈值 100ms
    .setSlowRatioThreshold(0.5)                // 50% 请求超过阈值就算慢
    .setMinRequestAmount(5)                    // 最小请求数 5
    .setStatIntervalMs(1000);                 // 统计窗口 1s
```

**熔断过程**：

1. Sentinel 统计最近 1 秒内的所有请求
2. 计算平均响应时间
3. 如果平均 RT > 100ms **且** 50% 的请求 RT > 100ms，触发熔断
4. 熔断持续 `recoveryTimeout`（默认 5 秒）
5. 之后进入半开状态，允许一个请求探测

### 2.2 异常比例熔断：异常请求占比

异常比例熔断策略关注的是**失败率**。

**触发条件**：当滑动窗口内**异常请求的比例**超过阈值，且**最小请求数**满足时，触发熔断。

```java
// 异常比例熔断配置
DegradeRule rule = new DegradeRule("orderService")
    .setGrade(RuleConstant.DEGRADE_GRADE_EXCEPTION_RATIO)  // 异常比例熔断
    .setCount(0.5)                                         // 50% 异常比例
    .setMinRequestAmount(10)                               // 最小请求数 10
    .setStatIntervalMs(60000);                             // 统计窗口 60s
```

**熔断过程**：

1. Sentinel 统计最近 60 秒内的所有请求
2. 计算异常请求的比例
3. 如果异常比例 > 50% **且** 请求数 > 10，触发熔断
4. 熔断持续 `recoveryTimeout`（默认 5 秒）
5. 之后进入半开状态

### 2.3 异常数熔断：异常请求总数

异常数熔断策略关注的是**绝对数量**。

**触发条件**：当滑动窗口内**异常请求的数量**超过阈值时，触发熔断。

```java
// 异常数熔断配置
DegradeRule rule = new DegradeRule("orderService")
    .setGrade(RuleConstant.DEGRADE_GRADE_EXCEPTION_COUNT)  // 异常数熔断
    .setCount(100)                                         // 100 个异常
    .setMinRequestAmount(10)                               // 最小请求数 10
    .setStatIntervalMs(60000);                             // 统计窗口 60s
```

**熔断过程**：

1. Sentinel 统计最近 60 秒内的所有请求
2. 计算异常请求的数量
3. 如果异常数 > 100，触发熔断
4. 熔断持续 `recoveryTimeout`（默认 5 秒）

:::tip 三种策略如何选择

| 策略 | 适用场景 | 推荐配置 |
|---|---|---|
| RT 熔断 | 依赖慢服务（如第三方 API、复杂查询） | RT 阈值 = 正常 RT 的 3-5 倍 |
| 异常比例熔断 | 依赖不稳定服务（如外部支付） | 异常比例 30%-50% |
| 异常数熔断 | 依赖不可靠服务（如老旧系统） | 异常数根据 QPS 调整 |

可以同时配置多个策略，满足不同的保护需求。

:::

## 三、熔断降级的工作原理

### 3.1 熔断器的状态机

Sentinel 的熔断降级本质是一个**断路器**，状态转换与断路器模式一致：

```
CLOSED（闭合） → OPEN（打开） → HALF_OPEN（半开） → CLOSED
```

**状态说明**：

- **CLOSED**：正常状态，请求正常通过，统计异常
- **OPEN**：熔断状态，所有请求被拒绝，立即返回降级响应
- **HALF_OPEN**：探测状态，允许少量请求通过，探测服务是否恢复

**状态转换规则**：

1. **CLOSED → OPEN**：失败次数/失败率/慢调用比例超过阈值
2. **OPEN → HALF_OPEN**：熔断持续时间到达
3. **HALF_OPEN → CLOSED**：探测请求成功
4. **HALF_OPEN → OPEN**：探测请求失败

### 3.2 熔断的生效流程

当一个请求被熔断拦截时，Sentinel 的处理流程：

1. 请求进入 `SphU.entry("resourceName")`
2. `StatisticSlot` 统计请求的响应时间和结果
3. `DegradeSlot` 检查熔断规则
4. 如果当前状态是 OPEN，检查是否超时可以进入 HALF_OPEN
5. 如果当前状态是 CLOSED，检查是否达到熔断条件
6. 如果需要熔断，抛出 `DegradeException`
7. 业务代码捕获异常，执行降级逻辑

### 3.3 半开探测机制

半开状态是熔断恢复的关键。

当熔断持续时间到达后，断路器会进入 **HALF_OPEN** 状态，允许一个请求通过：

- **探测成功**：说明服务恢复了，切换到 CLOSED 状态
- **探测失败**：说明服务还没恢复，切换回 OPEN 状态，继续熔断

```java
// 半开探测配置
DegradeRule rule = new DegradeRule("orderService")
    .setGrade(RuleConstant.DEGRADE_GRADE_EXCEPTION_RATIO)
    .setCount(0.5)
    .setMinRequestAmount(10)
    .setStatIntervalMs(60000)
    .setTimeWindow(10);  // 熔断持续 10 秒
```

## 四、DegradeSlot 的实现原理

### 4.1 核心逻辑

`DegradeSlot` 是 Sentinel 的熔断检查 Slot，它的职责是判断是否需要熔断。

```java
public class DegradeSlot extends SlotChainBuilder {
    @Override
    public void entry(String resourceName, Entry entry, Node node, int count, Object... args)
        throws Throwable {
        // 获取熔断规则
        List<DegradeRule> rules = DegradeRuleManager.getRules();
        for (DegradeRule rule : rules) {
            if (!rule.passCheck()) {
                // 熔断触发，执行降级
                throw new DegradeException(rule.getLimitApp());
            }
        }
    }
}
```

### 4.2 RT 熔断的计算

RT 熔断的核心是**滑动窗口统计**：

```java
// DegradeRule 的 passCheck 逻辑（伪代码）
public boolean passCheck() {
    if (grade != DEGRADE_GRADE_RT) {
        return true;
    }

    // 获取滑动窗口的统计节点
    StatisticNode node = getStatisticNode();

    // 获取最近 1 秒的请求统计
    List<MetricBucket> windows = node.getRecentWindows();

    // 计算平均 RT
    long totalRt = 0;
    long totalCount = 0;
    long slowCount = 0;

    for (MetricBucket bucket : windows) {
        totalRt += bucket.getRT();
        totalCount += bucket.getSuccessCount();
        slowCount += bucket.getSlowCount();
    }

    double avgRt = totalRt / totalCount;
    double slowRatio = slowCount / totalCount;

    // 判断是否触发熔断
    return avgRt <= count && slowRatio <= slowRatioThreshold;
}
```

## 五、SentinelResource 注解：更优雅的降级方式

### 5.1 注解配置

Sentinel 提供了 `@SentinelResource` 注解，简化熔断降级的配置：

```java
@Service
public class UserService {

    @SentinelResource(
        value = "getUser",
        blockHandler = "getUserBlockHandler",
        fallback = "getUserFallback",
        exceptionsToIgnore = {BusinessException.class}
    )
    public User getUser(Long userId) {
        // 调用用户服务
        return userClient.getUser(userId);
    }

    /**
     * 限流/熔断时的处理方法
     */
    public User getUserBlockHandler(Long userId, BlockException e) {
        log.warn("User service blocked: {}", e.getClass().getSimpleName());
        return User.placeholder(); // 返回降级数据
    }

    /**
     * 异常降级方法
     */
    public User getUserFallback(Long userId, Throwable t) {
        log.error("User service error", t);
        return User.placeholder(); // 返回降级数据
    }
}
```

### 5.2 blockHandler vs fallback

`@SentinelResource` 有两个降级方法：

| 属性 | 触发条件 | 异常参数 |
|---|---|---|
| `blockHandler` | 限流、熔断触发 | `BlockException` |
| `fallback` | 业务异常抛出 | `Throwable` |

**执行顺序**：

1. 限流/熔断触发 → `blockHandler`
2. 业务异常（非 `exceptionsToIgnore`）→ `fallback`
3. `exceptionsToIgnore` 中的异常 → 直接抛出

### 5.3 异常忽略

可以使用 `exceptionsToIgnore` 忽略某些异常，不触发降级：

```java
@SentinelResource(
    value = "getUser",
    fallback = "getUserFallback",
    exceptionsToIgnore = {IllegalArgumentException.class} // 忽略参数异常
)
public User getUser(Long userId) {
    if (userId <= 0) {
        throw new IllegalArgumentException("Invalid userId"); // 不触发降级
    }
    return userClient.getUser(userId);
}
```

## 六、生产环境配置实践

### 6.1 RT 熔断配置

RT 熔断适用于**依赖慢服务**的场景，如调用第三方 API。

```java
// 配置 RT 熔断
List<DegradeRule> rules = new ArrayList<>();

DegradeRule rule = new DegradeRule("paymentApi")
    .setGrade(RuleConstant.DEGRADE_GRADE_RT)
    .setCount(2000)                       // 平均 RT 超过 2s 触发
    .setSlowRatioThreshold(0.5)           // 50% 请求超过阈值就算慢
    .setMinRequestAmount(5)               // 最小 5 个请求
    .setStatIntervalMs(1000)              // 统计窗口 1s
    .setTimeWindow(30);                   // 熔断持续 30s

rules.add(rule);
DegradeRuleManager.loadRules(rules);
```

**配置说明**：

- `count = 2000`：如果平均响应时间超过 2 秒，触发熔断
- `slowRatioThreshold = 0.5`：50% 的请求是慢请求就触发
- `timeWindow = 30`：熔断 30 秒后尝试恢复

### 6.2 异常比例熔断配置

异常比例熔断适用于**依赖不稳定服务**的场景。

```java
// 配置异常比例熔断
DegradeRule rule = new DegradeRule("thirdPartyPayment")
    .setGrade(RuleConstant.DEGRADE_GRADE_EXCEPTION_RATIO)
    .setCount(0.3)                        // 30% 异常比例触发
    .setMinRequestAmount(10)              // 最小 10 个请求
    .setStatIntervalMs(60000)            // 统计窗口 60s
    .setTimeWindow(60);                  // 熔断持续 60s

rules.add(rule);
```

### 6.3 多种策略同时生效

可以同时配置多种熔断策略：

```java
// RT 熔断
DegradeRule rtRule = new DegradeRule("paymentApi")
    .setGrade(RuleConstant.DEGRADE_GRADE_RT)
    .setCount(2000)
    .setSlowRatioThreshold(0.5)
    .setMinRequestAmount(5)
    .setTimeWindow(30);

// 异常比例熔断
DegradeRule exceptionRule = new DegradeRule("paymentApi")
    .setGrade(RuleConstant.DEGRADE_GRADE_EXCEPTION_RATIO)
    .setCount(0.5)
    .setMinRequestAmount(10)
    .setTimeWindow(60);

DegradeRuleManager.loadRules(Arrays.asList(rtRule, exceptionRule));
```

**同时生效的效果**：任何一个策略触发，都会熔断。

## 七、降级策略的设计

### 7.1 降级响应的原则

降级响应不是简单的「返回 null」，应该有合理的设计：

| 策略 | 说明 | 适用场景 |
|---|---|---|
| **返回默认值** | 返回一个默认的响应 | 非核心功能 |
| **返回缓存** | 返回最近一次的有效数据 | 有缓存的数据 |
| **返回友好提示** | 返回错误码和提示信息 | 所有场景 |
| **返回简化数据** | 返回精简版的数据 | 复杂查询场景 |

### 7.2 降级策略实现

```java
@Service
public class ProductService {
    private LoadingCache<Long, Product> productCache;

    @PostConstruct
    public void init() {
        // 本地缓存
        productCache = Caffeine.newBuilder()
            .expireAfterWrite(10, TimeUnit.MINUTES)
            .maximumSize(1000)
            .build(key -> loadProductFromRemote(key));
    }

    @SentinelResource(
        value = "getProduct",
        blockHandler = "getProductBlockHandler",
        fallback = "getProductFallback"
    )
    public Product getProduct(Long productId) {
        return productCache.get(productId);
    }

    // 限流/熔断降级：返回缓存
    public Product getProductBlockHandler(Long productId, BlockException e) {
        // 尝试从缓存获取
        Product cached = productCache.getIfPresent(productId);
        if (cached != null) {
            log.info("Product {} served from cache (degraded)", productId);
            return cached;
        }
        return Product.placeholder(productId);
    }

    // 异常降级：返回占位符
    public Product getProductFallback(Long productId, Throwable t) {
        log.error("Failed to get product {}", productId, t);
        return Product.placeholder(productId);
    }
}
```

## 八、与限流的配合使用

### 8.1 两种保护的区别

限流和熔断是两种不同的保护机制：

| 维度 | 限流（Flow） | 熔断（Degrade） |
|---|---|---|
| 触发条件 | 请求量超过阈值 | 服务响应异常（慢/失败） |
| 目的 | 保护系统不被冲垮 | 保护服务不被拖垮 |
| 行为 | 拒绝请求 | 快速失败，不再调用 |
| 统计维度 | QPS、并发线程数 | RT、异常比例、异常数 |
| 恢复 | 请求量降低后自动恢复 | 服务恢复后自动恢复 |

### 8.2 配合使用示例

```java
@Configuration
public class SentinelConfig {
    @PostConstruct
    public void init() {
        // 限流规则：每秒最多 1000 个请求
        FlowRule flowRule = new FlowRule("orderService")
            .setCount(1000)
            .setGrade(RuleConstant.FLOW_GRADE_QPS)
            .setControlBehavior(RuleConstant.CONTROL_BEHAVIOR_DEFAULT);
        FlowRuleManager.loadRules(Collections.singletonList(flowRule));

        // 熔断规则：平均 RT 超过 2s 或 50% 异常，熔断 30s
        DegradeRule degradeRule = new DegradeRule("orderService")
            .setGrade(RuleConstant.DEGRADE_GRADE_RT)
            .setCount(2000)
            .setSlowRatioThreshold(0.5)
            .setMinRequestAmount(5)
            .setTimeWindow(30);
        DegradeRuleManager.loadRules(Collections.singletonList(degradeRule));
    }
}
```

## 九、面试追问：熔断降级相关问题

### 9.1 问题一：Sentinel 熔断的三种策略是什么？

**参考答案**：

Sentinel 支持三种熔断策略：

1. **RT 熔断**：当平均响应时间超过阈值时触发熔断
2. **异常比例熔断**：当异常请求比例超过阈值时触发熔断
3. **异常数熔断**：当异常请求数量超过阈值时触发熔断

三种策略可以同时配置，任一策略触发都会熔断。

### 9.2 问题二：熔断和限流的区别是什么？

**参考答案**：

| 对比维度 | 限流 | 熔断 |
|---|---|---|
| 触发条件 | 请求量超过阈值 | 服务响应异常 |
| 保护对象 | 系统容量 | 服务质量 |
| 行为 | 拒绝请求 | 快速失败 |
| 恢复方式 | 请求量降低后恢复 | 服务恢复后恢复 |

**类比**：限流是「门口的保安」，告诉你不能进；熔断是「医生」，告诉你需要休息。

### 9.3 问题三：RT 熔断的慢调用比例是什么意思？

**参考答案**：

RT 熔断有两个条件：

1. **平均 RT**：滑动窗口内所有请求的平均响应时间
2. **慢调用比例**：超过 RT 阈值的请求占总请求的比例

```
假设：RT 阈值 = 1000ms，统计窗口内 10 个请求
- 5 个请求 RT = 500ms（正常）
- 5 个请求 RT = 2000ms（慢）

平均 RT = (500*5 + 2000*5) / 10 = 1250ms
慢调用比例 = 5/10 = 50%
```

当**平均 RT 超过阈值**且**慢调用比例超过设定值**时，触发熔断。

### 9.4 问题四：半开状态的探测机制是什么？

**参考答案**：

半开状态是熔断恢复的关键：

1. 熔断持续 `timeWindow` 秒后，进入半开状态
2. 半开状态允许一个请求通过，探测服务是否恢复
3. **探测成功**：切换到 CLOSED 状态，正常调用
4. **探测失败**：切换回 OPEN 状态，继续熔断

半开状态只允许少量请求通过，避免服务还没恢复就被大量请求打垮。

## 总结

Sentinel 的熔断降级是保护服务稳定性的关键组件：

- **三种策略**：RT 熔断、异常比例熔断、异常数熔断
- **状态机**：CLOSED → OPEN → HALF_OPEN → CLOSED
