# 熔断工具对比：Hystrix、Sentinel、Resilience4j

你知道吗？Netflix 在 2018 年就宣布 Hystrix 进入维护模式，不再开发新功能。2019 年，Spring Cloud 官方也宣布 Hystrix 进入 end-of-life 状态，推荐使用替代方案。

但很多公司的老系统还在用 Hystrix，很多面试官还在问 Hystrix。这篇文章，我们来聊聊三大熔断工具的对比：**Hystrix**、**Sentinel**、**Resilience4j**，搞清楚它们的优缺点，以及如何选择。

## 一、Hystrix：曾经的王者

### 1.1 Hystrix 的核心设计

Hystrix 是 Netflix 开源的熔断组件，曾经是微服务架构的标配。

它的核心设计包括三个部分：

1. **HystrixCommand**：封装调用逻辑的 Command
2. **线程池隔离**：每个依赖服务独立的线程池
3. **断路器**：基于失败率的状态机

```java
public class GetUserCommand extends HystrixCommand<User> {
    private final Long userId;

    public GetUserCommand(Long userId) {
        super(Setter
            .withGroupKey(HystrixCommandGroupKey.Factory.asKey("UserService"))
            .andCommandKey(HystrixCommandKey.Factory.asKey("GetUser"))
            .andThreadPoolKey(HystrixThreadPoolKey.Factory.asKey("UserPool"))
            .andCommandPropertiesDefaults(
                HystrixCommandProperties.Setter()
                    .withCircuitBreakerRequestVolumeThreshold(20)    // 20 个请求
                    .withCircuitBreakerErrorThresholdPercentage(50)  // 50% 失败率
                    .withCircuitBreakerSleepWindowInMilliseconds(5000) // 5s 后尝试
            )
            .andThreadPoolPropertiesDefaults(
                HystrixThreadPoolProperties.Setter()
                    .withCoreSize(20)     // 核心线程数
                    .withMaxQueueSize(100) // 队列大小
            )
        );
        this.userId = userId;
    }

    @Override
    protected User run() {
        return userService.getUser(userId);
    }

    @Override
    protected User getFallback() {
        return User.placeholder();
    }
}

// 调用
User user = new GetUserCommand(1L).execute();
```

### 1.2 Hystrix 的线程池隔离

Hystrix 使用**线程池隔离**来保护服务。每个依赖服务都有独立的线程池：

```
HystrixCommand      HystrixCommand      HystrixCommand
      │                   │                   │
 ThreadPool A        ThreadPool B        ThreadPool C
  (UserService)       (OrderService)      (PaymentService)
      │                   │                   │
  20 线程            30 线程            50 线程
```

**线程池隔离的优点**：

- 故障隔离：一个服务的故障不会影响其他服务
- 资源控制：每个服务的并发量可精确控制
- 防止雪崩：线程池满时自动拒绝

**线程池隔离的缺点**：

- 线程切换开销：线程上下文切换有性能损耗
- 内存开销：每个线程需要占用栈内存
- 调用延迟：跨线程调用增加延迟

### 1.3 Hystrix 的死亡

Hystrix 停止维护的原因：

1. **线程池开销**：在高并发场景下，线程池隔离的性能损耗明显
2. **维护成本**：Netflix 内部转向其他方案，不再投入资源
3. **Spring Cloud 推荐替代**：Spring Cloud Circuit Breaker 推出

:::warning Hystrix 现状

Hystrix 已经停止维护，但很多公司仍在使用。如果你的系统用 Hystrix，建议：

1. **新项目**：不要使用 Hystrix，选择 Sentinel 或 Resilience4j
2. **老项目**：考虑迁移到 Spring Cloud Circuit Breaker（底层可以是 Resilience4j）

:::

## 二、Sentinel：阿里巴巴的解决方案

### 2.1 Sentinel 的核心能力

Sentinel 是阿里巴巴开源的流量控制组件，提供四大核心能力：

| 能力 | 说明 |
|---|---|
| **流量控制** | QPS 限流、并发线程数限流 |
| **熔断降级** | RT 熔断、异常比例熔断、异常数熔断 |
| **系统自适应** | 根据系统负载自适应限流 |
| **热点参数限流** | 对参数值进行限流 |

```java
@SentinelResource(
    value = "getUser",
    blockHandler = "getUserBlockHandler",
    fallback = "getUserFallback"
)
public User getUser(Long userId) {
    return userService.getUser(userId);
}

public User getUserBlockHandler(Long userId, BlockException e) {
    return User.placeholder();
}
```

### 2.2 Sentinel vs Hystrix

| 对比维度 | Sentinel | Hystrix |
|---|---|---|
| **限流能力** | 支持 QPS、线程数限流 | 不支持限流 |
| **熔断策略** | RT、异常比例、异常数 | 仅异常比例 |
| **线程隔离** | 信号量隔离 | 线程池隔离 |
| **动态配置** | 支持 Dashboard 动态推送 | 静态配置 |
| **监控** | 自带 Dashboard | 依赖 Hystrix Dashboard |
| **学习成本** | 中等 | 较高 |
| **社区活跃度** | 活跃（Spring Cloud Alibaba） | 停止维护 |

### 2.3 Sentinel 的优势

1. **更全面的限流**：不仅支持熔断，还支持限流
2. **更精细的熔断策略**：支持 RT 熔断和慢调用比例
3. **动态配置**：通过 Dashboard 实时调整规则
4. **更轻量**：不使用线程池，性能更好
5. **活跃社区**：与 Spring Cloud Alibaba 深度集成

## 三、Resilience4j：轻量级新选择

### 3.1 Resilience4j 的核心设计

Resilience4j 是一个轻量级的容错库，专为函数式编程设计。

它的核心模块：

| 模块 | 说明 |
|---|---|
| **resilience4j-circuitbreaker** | 断路器 |
| **resilience4j-ratelimiter** | 限流器 |
| **resilience4j-bulkhead** | 舱壁模式 |
| **resilience4j-retry** | 重试 |
| **resilience4j-timelimiter** | 超时控制 |

```java
CircuitBreakerConfig config = CircuitBreakerConfig.custom()
    .failureRateThreshold(50)                      // 50% 失败率
    .minimumNumberOfCalls(10)                      // 最小请求数
    .waitDurationInOpenState(Duration.ofSeconds(30)) // 30s 后尝试
    .permittedNumberOfCallsInHalfOpenState(5)      // 半开状态允许 5 个调用
    .build();

CircuitBreakerRegistry registry = CircuitBreakerRegistry.of(config);
CircuitBreaker circuitBreaker = registry.circuitBreaker("userService");

// 使用装饰器模式
Supplier<User> decoratedSupplier = CircuitBreaker
    .decorateSupplier(circuitBreaker, () -> userService.getUser(userId));

// 执行
User user = Try.ofSupplier(decoratedSupplier)
    .recover(Exception.class, e -> User.placeholder())
    .get();
```

### 3.2 Resilience4j 的特点

**优点**：

1. **轻量级**：没有 Hystrix 的线程池开销
2. **函数式**：支持函数式编程，与 Vavr 深度集成
3. **模块化**：按需引入，不用引入全部依赖
4. **同步友好**：支持同步和异步两种模式

**缺点**：

1. **不支持分布式**：没有 Sentinel 那样的集群限流能力
2. **监控弱**：不自带 Dashboard
3. **文档少**：相对 Hystrix，文档较少

### 3.3 Spring Cloud Circuit Breaker

Spring Cloud Circuit Breaker 是 Spring 官方提供的统一抽象层，可以切换底层实现：

```yaml
spring:
  cloud:
    circuitbreaker:
      resilience4j:
        enabled: true
```

```java
@Service
public class UserService {
    private final CircuitBreakerFactory factory;

    public UserService(CircuitBreakerFactory factory) {
        this.factory = factory;
    }

    public User getUser(Long userId) {
        return factory.create("userService")
            .run(() -> userServiceClient.getUser(userId),
                 throwable -> User.placeholder());
    }
}
```

## 四、三者横向对比

### 4.1 功能对比

| 维度 | Hystrix | Sentinel | Resilience4j |
|---|---|---|---|
| **限流** | 不支持 | 支持 | 支持（RateLimiter） |
| **熔断** | 支持（仅失败率） | 支持（RT、失败率、异常数） | 支持 |
| **隔离方式** | 线程池隔离 | 信号量隔离 | 信号量/线程池 |
| **重试** | 支持 | 不支持（需配合） | 支持 |
| **超时控制** | 支持 | 支持 | 支持 |
| **动态配置** | 不支持 | 支持 | 需配合 Spring Cloud Config |
| **监控** | Hystrix Dashboard | 自带 Dashboard | Prometheus |
| **集群限流** | 不支持 | 支持 | 不支持 |

### 4.2 性能对比

| 维度 | Hystrix | Sentinel | Resilience4j |
|---|---|---|---|
| **线程开销** | 高（有线程池） | 低（无线程池） | 低 |
| **调用延迟** | 较高（线程切换） | 低 | 低 |
| **内存占用** | 高 | 中 | 低 |
| **并发能力** | 受线程池限制 | 高 | 高 |

### 4.3 社区状态

| 组件 | 状态 | 推荐度 |
|---|---|---|
| Hystrix | 停止维护 | 不推荐（新项目） |
| Sentinel | 活跃开发 | **推荐**（Spring Cloud Alibaba 项目） |
| Resilience4j | 活跃开发 | **推荐**（普通 Spring Cloud 项目） |

## 五、迁移路径

### 5.1 从 Hystrix 迁移到 Resilience4j

如果你的项目使用 Hystrix，可以迁移到 Resilience4j：

```java
// Hystrix 代码
public class GetUserCommand extends HystrixCommand<User> {
    @Override
    protected User run() {
        return userService.getUser(userId);
    }
}
User user = new GetUserCommand(1L).execute();

// 迁移到 Resilience4j
Supplier<User> supplier = () -> userService.getUser(userId);
Supplier<User> decorated = CircuitBreaker.decorateSupplier(circuitBreaker, supplier);
User user = Try.ofSupplier(decorated).get();
```

### 5.2 从 Hystrix 迁移到 Sentinel

如果你的项目使用 Spring Cloud Alibaba，可以迁移到 Sentinel：

```java
// Hystrix 代码
@HystrixCommand(fallbackMethod = "getUserFallback")
public User getUser(Long userId) {
    return userService.getUser(userId);
}

// 迁移到 Sentinel
@SentinelResource(
    value = "getUser",
    fallback = "getUserFallback"
)
public User getUser(Long userId) {
    return userService.getUser(userId);
}

public User getUserFallback(Long userId, Throwable t) {
    return User.placeholder();
}
```

### 5.3 迁移注意事项

| 注意事项 | 说明 |
|---|---|
| **配置迁移** | Hystrix 的配置是代码内，Sentinel 支持 Dashboard |
| **监控迁移** | Hystrix Dashboard → Sentinel Dashboard |
| **线程池隔离** | 如果需要线程池隔离，Resilience4j 也有 Bulkhead |
| **服务发现集成** | 确保新框架与你的服务发现组件兼容 |

## 六、如何选择

### 6.1 按项目类型选择

| 项目类型 | 推荐选择 | 原因 |
|---|---|---|
| Spring Cloud Alibaba | Sentinel | 深度集成，开箱即用 |
| 普通 Spring Cloud | Resilience4j | 轻量，Spring Cloud Circuit Breaker 支持 |
| 老项目改造 | Resilience4j | 迁移成本低 |
| 新项目 | Sentinel 或 Resilience4j | 根据集成需求选择 |

### 6.2 按团队规模选择

| 团队规模 | 推荐选择 | 原因 |
|---|---|---|
| 小团队（< 5 人） | Sentinel | Dashboard 开箱即用，减少运维成本 |
| 中型团队（5-20 人） | Sentinel 或 Resilience4j | 根据技术栈选择 |
| 大团队（> 20 人） | Sentinel | 强大的监控和配置管理能力 |

### 6.3 按业务场景选择

| 场景 | 推荐选择 | 原因 |
|---|---|---|
| 需要限流 + 熔断 | Sentinel | 功能最全面 |
| 仅需要熔断 | Resilience4j | 轻量 |
| 高并发项目 | Resilience4j | 性能好，无线程开销 |
| 依赖 Spring Cloud Alibaba | Sentinel | 官方集成 |

## 七、生产环境最佳实践

### 7.1 Sentinel 生产配置

```java
@Configuration
public class SentinelConfig {
    @PostConstruct
    public void init() {
        // 配置限流规则
        FlowRule rule = new FlowRule("orderService")
            .setCount(1000)
            .setGrade(RuleConstant.FLOW_GRADE_QPS)
            .setControlBehavior(RuleConstant.CONTROL_BEHAVIOR_DEFAULT);
        FlowRuleManager.loadRules(Collections.singletonList(rule));

        // 配置熔断规则
        DegradeRule degradeRule = new DegradeRule("orderService")
            .setGrade(RuleConstant.DEGRADE_GRADE_RT)
            .setCount(2000)
            .setSlowRatioThreshold(0.5)
            .setMinRequestAmount(5)
            .setTimeWindow(30);
        DegradeRuleManager.loadRules(Collections.singletonList(degradeRule));
    }
}
```

## 八、面试追问：熔断工具相关问题

### 8.1 问题一：Hystrix 为什么停止维护了？

**参考答案**：

Hystrix 停止维护的主要原因：

1. **性能问题**：线程池隔离在高并发场景下有较大开销
2. **Netflix 内部转型**：Netflix 内部使用其他方案（Adaptive Concurrency Limits），不再投入资源维护
3. **Spring Cloud 替代**：Spring Cloud Circuit Breaker 推出，Hystrix 被标记为 deprecated

**建议**：新项目不要使用 Hystrix，选择 Sentinel 或 Resilience4j。

### 8.2 问题二：Sentinel 和 Resilience4j 哪个更好？

**参考答案**：

两者各有优劣，选择取决于项目需求：

| 维度 | Sentinel | Resilience4j |
|---|---|---|
| 功能 | 更全面（限流 + 熔断 + 热点参数） | 更轻量（仅熔断相关） |
| 性能 | 好 | 更好 |
| 监控 | 自带 Dashboard | 需配合 Prometheus |
| 集群限流 | 支持 | 不支持 |
| Spring Cloud Alibaba | 官方集成 | 需配置 |

**推荐**：

- 使用 Spring Cloud Alibaba → Sentinel
- 普通 Spring Cloud → Resilience4j
- 需要限流 → Sentinel
- 追求极致性能 → Resilience4j

### 8.3 问题三：线程池隔离和信号量隔离的区别是什么？

**参考答案**：

| 对比维度 | 线程池隔离 | 信号量隔离 |
|---|---|---|
| 资源消耗 | 高（每个调用一个线程） | 低（仅计数器） |
| 调用方式 | 跨线程调用 | 同一线程调用 |
| 超时控制 | 线程级别超时 | 整个调用超时 |
| 适用场景 | 依赖慢服务 | 依赖快速服务 |
| 异常传播 | 通过 Future 传递 | 直接传播 |

**选择建议**：

- 依赖服务响应慢 → 线程池隔离
- 依赖服务响应快 → 信号量隔离
- 需要精细超时控制 → 线程池隔离

### 8.4 问题四：如何迁移 Hystrix 到新框架？

**参考答案**：

迁移步骤：

1. **评估依赖**：盘点所有 HystrixCommand 和 @HystrixCommand
2. **选择目标**：Sentinel（功能全）或 Resilience4j（轻量）
3. **逐步迁移**：先迁移非核心服务，积累经验
4. **配置对比**：Hystrix 配置 → 新框架配置
5. **监控验证**：确保监控指标对齐

迁移成本：Hystrix → Sentinel 较低，Hystrix → Resilience4j 较高。

## 总结

三大熔断工具的对比如下：

| 维度 | Hystrix | Sentinel | Resilience4j |
|---|---|---|---|
| 限流 | 不支持 | 支持 | 支持 |
| 熔断 | 支持（仅失败率） | 支持（RT、失败率、异常数） | 支持 |
| 线程隔离 | 线程池隔离 | 信号量隔离 | 信号量/线程池 |
| 性能 | 较低 | 高 | 高 |
| 社区状态 | 停止维护 | 活跃 | 活跃 |
| 推荐度 | 不推荐 | **推荐** | **推荐** |

**选择建议**：

- Spring Cloud Alibaba 项目 → Sentinel
- 普通 Spring Cloud 项目 → Resilience4j
- 新项目不要使用 Hystrix

迁移路径：Hystrix → Resilience4j 或 Hystrix → Sentinel。
