# 限流与熔断：流量防护的核心武器

凌晨三点，你被告警叫醒：「线上服务响应超时，大量请求堆积！」你登录服务器一看，发现某个接口的 QPS 突然暴涨了 10 倍，数据库连接池耗尽，整个系统陷入瘫痪。

如果你提前配置好了限流和熔断策略，这场事故可能根本不会发生。

限流和熔断是保护高并发系统的两大核心武器。**限流**告诉你「请求太多，你需要排队」；**熔断**告诉你「下游服务出问题了，你先休息一下」。两者配合，才能在流量洪峰和下游故障的双重冲击下，保证系统的稳定性。

这一模块，将为你彻底讲透限流算法、分布式限流、单节点限流、Sentinel 流量控制、断路器模式、熔断降级、舱壁隔离，以及 Hystrix、Sentinel、Resilience4j 的对比选择。

## 一、模块定位：为什么这个模块如此重要

在高并发系统中，流量防护不是「锦上添花」，而是「生死攸关」。

你有没有遇到过这种情况：代码里用了 `synchronized`，自认为「线程安全」了，结果上线后系统还是被流量冲垮了。`synchronized` 保护的是代码逻辑，不是系统容量。你用一把锁把 10000 个并发请求拦在门口排队，锁是保住了，但 CPU 全被上下文切换吃光了。

或者这种场景：系统正常运行时一切安好，突然某个下游服务变慢了，你的系统也跟着变慢，然后越来越慢，最后整个系统崩溃了。这不是你的代码有问题，这是**级联故障（Cascading Failure）**——一个服务的故障，引发了连锁反应，最终拖垮了整个系统。

**限流解决的是「量」的问题**：控制请求速率，保护系统不被冲垮。

**熔断解决的是「质」的问题**：检测故障服务，快速失败，避免雪崩。

:::tip 限流 vs 熔断 vs 降级

很多同学搞不清这三者的区别：

- **限流**：控制请求速率，保护系统不被冲垮
- **熔断**：检测故障服务，快速失败，避免雪崩
- **降级**：放弃非核心功能，保证核心功能可用

限流是**预防性**手段，熔断和降级是**应急性**手段。三者配合使用，构成分层防护体系。

:::

## 二、核心概念速览

### 2.1 限流算法的四大天王

| 算法 | 核心思想 | 突发处理 | 精度 | 内存消耗 |
|---|---|---|---|---|
| 固定窗口 | 把时间划分为固定窗口，每个窗口独立计数 | 不支持 | 低（有边界问题） | 低 |
| 滑动窗口日志 | 精确记录每个请求的时间戳 | 不支持 | 高 | 高 |
| 滑动窗口计数 | 将窗口划分为多个子窗口独立计数 | 不支持 | 中（取决于桶数量） | 中 |
| 令牌桶 | 桶里放令牌，令牌以固定速率补充 | **支持** | 高 | 低 |
| 漏桶 | 请求进入漏桶，以固定速率漏出 | 不支持 | 高 | 低 |

**令牌桶是推荐方案**，因为它支持突发流量——平时积累的令牌，可以在关键时刻一次性使用。

### 2.2 断路器的三种状态

断路器灵感来自电路保险丝，有三种状态：

| 状态 | 说明 | 行为 |
|---|---|---|
| **CLOSED（闭合）** | 正常状态，断路器关闭，请求正常通过 | 统计失败次数 |
| **OPEN（打开）** | 故障状态，断路器打开，请求直接拒绝 | 快速失败 |
| **HALF_OPEN（半开）** | 探测状态，允许部分请求通过 | 探测服务是否恢复 |

```
CLOSED → OPEN：失败次数/失败率超过阈值
OPEN → HALF_OPEN：等待超时时间后，允许试探
HALF_OPEN → CLOSED：试探请求全部成功
HALF_OPEN → OPEN：试探请求失败
```

### 2.3 熔断降级的三种策略

| 策略 | 触发条件 | 适用场景 |
|---|---|---|
| **RT 熔断** | 平均响应时间超过阈值 | 依赖慢服务的场景 |
| **异常比例熔断** | 异常请求比例超过阈值 | 依赖不稳定服务的场景 |
| **异常数熔断** | 异常请求数量超过阈值 | 依赖不可靠服务的场景 |

### 2.4 核心公式

```
// 线程池核心线程数计算
核心线程数 = 预期 QPS × 平均响应时间

// 令牌桶容量与突发处理
最大突发量 = 桶容量

// 舱壁大小计算
舱壁大小 = 预期 QPS × 平均响应时间 × 安全系数

// 失败率计算
失败率 = 失败请求数 / 总请求数 × 100%
```

## 五、核心要点提炼

### 5.1 限流的核心要点

1. **令牌桶是推荐方案**：支持突发流量，工业级实现成熟（Guava RateLimiter）
2. **单机限流和分布式限流是互补关系**：分布式限流保护集群，单机限流保护单个实例
3. **Redis + Lua 是分布式限流的标准实现**：原子性保证，高性能，支持多种算法
4. **Fail-Open vs Fail-Closed**：核心接口用 Fail-Closed，普通接口用 Fail-Open

### 5.2 熔断的核心要点

1. **断路器的三种状态**：CLOSED、OPEN、HALF_OPEN，状态转换是关键
2. **三种熔断策略**：RT 熔断（慢服务）、异常比例熔断（不稳定服务）、异常数熔断（不可靠服务）
3. **半开探测机制**：熔断恢复的关键，允许少量请求探测服务是否恢复
4. **降级策略设计**：返回默认值、缓存、友好提示，根据场景选择

### 5.3 舱壁的核心要点

1. **线程池隔离 vs 信号量隔离**：慢服务用线程池，快服务用信号量
2. **舱壁 + 断路器配合**：舱壁限制并发量，断路器检测故障
3. **舱壁大小计算**：核心线程数 = QPS × 响应时间

## 六、生产环境最佳实践

### 6.1 分层防护体系

推荐采用**四层防护**策略：

```
┌─────────────────────────────────────────────────────────────┐
│                      网关层：分布式限流                      │
│         （Nginx / Spring Cloud Gateway + Redis）            │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                      服务层：单机限流 + Sentinel             │
│              （Guava RateLimiter + Sentinel）               │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                      熔断层：断路器 + 熔断降级               │
│              （Sentinel / Resilience4j CircuitBreaker）       │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                      隔离层：舱壁模式                        │
│              （Resilience4j Bulkhead / 线程池隔离）           │
└─────────────────────────────────────────────────────────────┘
```

### 6.2 关键配置建议

| 组件 | 参数 | 推荐值 | 说明 |
|---|---|---|---|
| 令牌桶 | permitsPerSecond | 最大 QPS × 70% | 限流阈值留有余量 |
| 断路器 | failureRateThreshold | 50% | 超过一半失败就跳闸 |
| 断路器 | minimumNumberOfCalls | 10-50 | 太少会导致误判 |
| 断路器 | openToHalfOpenTimeout | 30-60s | 太短频繁探测，太长恢复慢 |
| 舱壁 | maxConcurrentCalls | QPS × RT × 安全系数 | 核心服务配置小一点 |
| RT 熔断 | count | 正常 RT 的 3-5 倍 | 依赖慢服务时设置较高 |
| 异常比例熔断 | count | 30%-50% | 依赖不稳定服务时设置较低 |

### 6.3 监控与告警

建议监控以下关键指标：

| 指标 | 说明 | 告警阈值 |
|---|---|---|
| 通过 QPS | 正常通过的请求 | - |
| 拒绝 QPS | 被限流拒绝的请求 | 持续 > 10% |
| 断路器状态 | CLOSED / OPEN / HALF_OPEN | OPEN 持续 > 5 分钟 |
| 平均响应时间 | 接口的 RT | > 阈值 50% |
| 失败率 | 发生异常的比例 | > 阈值 50% |

:::warning 常见误区

1. **误区一：有了限流就不需要熔断**
   - 限流只能控制请求数量，无法控制请求质量
   - 如果服务本身变慢，限流救不了你

2. **误区二：断路器可以完全替代超时**
   - 断路器是超时策略的补充，不是替代
   - 两者应该同时使用

3. **误区三：舱壁越大越好**
   - 舱壁配置太大，失去隔离的意义
   - 核心服务配置小一点，非核心服务配置大一点

4. **误区四：单机限流可以替代分布式限流**
   - 单机限流只能保护单个实例，无法保护整个集群
   - 两者是互补关系

:::

## 总结

限流与熔断是保护高并发系统的两大核心武器：

**限流**控制的是「量」——通过固定窗口、滑动窗口、漏桶、令牌桶等算法，控制请求速率，保护系统不被冲垮。

**熔断**解决的是「质」——通过断路器状态机，检测故障服务，快速失败，避免级联故障扩散。

**舱壁**隔离的是「资源」——通过线程池隔离或信号量隔离，防止一个服务的故障拖垮整个系统。

在实际项目中，需要**分层防护**：

1. **网关层**：分布式限流，保护整个集群
2. **服务层**：单机限流 + Sentinel，保护单个实例
3. **熔断层**：断路器 + 熔断降级，快速失败
4. **隔离层**：舱壁模式，资源隔离

限流和熔断不是万能药，它们需要配合使用、相互补充。同时，还需要配合**监控告警**、**降级策略**、**超时控制**、**重试机制**等，才能构建完整的高可用系统。

> "系统的稳定性不是靠运气，而是靠设计。"
