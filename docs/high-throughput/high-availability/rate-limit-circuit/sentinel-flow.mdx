# Sentinel 限流规则与流量效果控制器

你有没有想过这个问题：Guava RateLimiter 只能做单机限流，Redis + Lua 只能做分布式限流的中控台。但如果我想要一个**开箱即用的限流熔断框架**，能同时管理数十个服务的限流规则，能实时看到每个接口的 QPS、响应时间、通过率，还能在不重启服务的情况下动态调整限流策略，该怎么做？

这就是 Sentinel 的用武之地。

Sentinel 是阿里巴巴开源的流量控制组件，与 Hystrix 齐名，但比 Hystrix 更轻量、更易用。这篇文章，我们来聊聊 Sentinel 的限流功能。

## 一、Sentinel 架构解析

### 1.1 核心概念：资源与规则

Sentinel 的核心思想很简单：**保护「资源」，遵守「规则」**。

- **资源（Resource）**：一段代码、一个接口、一个方法，都可以是资源
- **规则（Rule）**：限流规则、熔断规则、热点规则、授权规则

当你调用 `SphU.entry("resourceName")` 时，Sentinel 会：

1. 检查是否有对应的规则
2. 根据规则判断是否放行
3. 如果放行，记录统计数据
4. 如果拒绝，抛出 `BlockException`

### 1.2 三大核心组件

Sentinel 有三个核心组件：

| 组件 | 作用 |
|---|---|
| **SphU** | 埋点入口，负责统计和检查 |
| **CtSph** | 上下文持有者，管理调用链 |
| **Slot Chain** | 处理器链，按顺序执行各种检查 |

### 1.3 Slot Chain 执行流程

Sentinel 的处理是链式的，每个 Slot 负责不同的职责：

```
请求进入
    │
    ▼
┌─────────────────┐
│ NodeSelectorSlot│  选择资源对应的 Node
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ ClusterBuilderSlot│  构建集群节点
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ LogSlot        │  记录日志
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ StatisticSlot   │  统计数据（QPS、线程数）
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ FlowSlot        │  限流检查 ← 重点
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ DegradeSlot     │  熔断检查
└────────┬────────┘
         │
         ▼
      请求通过
```

**StatisticSlot** 是所有统计数据的入口，所有的限流和熔断都依赖它提供的数据。

**FlowSlot** 是限流检查的核心，它会根据限流规则决定是否放行。

### 1.4 限流的本质：统计数据 vs 规则匹配

限流的本质是：**用统计数据和规则做比较**。

- 统计数据：过去 1 秒有多少请求？当前有多少并发线程？
- 规则：限流阈值是多少？拒绝策略是什么？

Sentinel 使用**滑动窗口算法**统计 QPS，窗口被划分为 2 个子窗口，每个子窗口 500ms。

## 二、限流规则详解

### 2.1 规则配置

Sentinel 的限流规则通过 `FlowRule` 类定义：

```java
FlowRule rule = new FlowRule();
rule.setResource("orderService");        // 资源名
rule.setCount(100);                       // 阈值
rule.setGrade(RuleConstant.FLOW_GRADE_QPS); // 按 QPS 限流
rule.setControlBehavior(RuleConstant.CONTROL_BEHAVIOR_DEFAULT); // 直接拒绝

// 添加规则
FlowRuleManager.loadRules(Collections.singletonList(rule));
```

### 2.2 限流维度：两种 grade

Sentinel 支持两种限流维度：

| Grade | 说明 | 适用场景 |
|---|---|---|
| `FLOW_GRADE_QPS` | 按每秒请求数限流 | 大多数场景 |
| `FLOW_GRADE_THREAD` | 按并发线程数限流 | 保护线程池、连接池 |

**QPS 维度的限流**：当 1 秒内的请求数超过阈值时，拒绝后续请求。

**线程数维度的限流**：当正在处理的请求数超过阈值时，拒绝新请求。适用于依赖线程池的服务。

```java
// QPS 限流
rule.setGrade(RuleConstant.FLOW_GRADE_QPS);
rule.setCount(100); // 每秒最多 100 个请求

// 线程数限流
rule.setGrade(RuleConstant.FLOW_GRADE_THREAD);
rule.setCount(50); // 同时最多处理 50 个请求
```

:::tip 什么时候用线程数限流

当服务依赖下游服务（如数据库、RPC），且下游服务的并发能力有限时，使用**线程数限流**更合适。

因为 QPS 限流只看请求数量，无法区分「快请求」和「慢请求」。如果每个请求要耗时 10 秒，100 QPS 可能意味着同时有 1000 个并发线程。

线程数限流能更准确地保护下游服务。

:::

### 2.3 拒绝策略：四种 behavior

当请求被限流拦截时，Sentinel 支持四种处理策略：

| Behavior | 值 | 说明 | 适用场景 |
|---|---|---|---|
| 直接拒绝（Default） | 0 | 直接拒绝，超出阈值的请求直接抛出 `FlowException` | 适合纯限流场景 |
| 冷启动（Warm Up） | 1 | 冷启动模式，初始速率很低，逐步增加到阈值 | 依赖冷启动资源的场景 |
| 匀速排队（Queue） | 2 | 让请求匀速通过，超时的请求被拒绝 | 需要严格匀速的场景 |
| 冷启动 + 匀速排队 | 3 | 冷启动后匀速排队 | 结合冷启动和匀速 |

#### 直接拒绝

最简单的策略，阈值是多少，就放行多少请求。

```java
rule.setControlBehavior(RuleConstant.CONTROL_BEHAVIOR_DEFAULT);
```

```
时间轴 →              |<- 1s ->|
请求: ████████████████░░░░░░░░░░░
放行: ████████████░░░░░░░░░░░░░░░░ (假设阈值 100)
拒绝:        ██████████
```

#### 冷启动（Warm Up）

冷启动模式用于处理「突发流量」场景。假设你的系统刚启动，连接池还没建立好，突然涌入大量请求。

冷启动的策略是：

1. 初始阈值很低（通常为阈值的 1/3）
2. 随着时间推移，阈值逐渐增加
3. 达到「预热时长」后，阈值恢复正常

```java
// 冷启动：预热时长 10 秒
rule.setControlBehavior(RuleConstant.CONTROL_BEHAVIOR_WARM_UP);
rule.setWarmUpPeriodSec(10);
```

```
时间轴 → |<- 10s 预热 ->|<- 正常 ->|
阈值:   33 → 67 → 100
```

#### 匀速排队（Queue）

匀速排队让请求匀速通过，超出排队长度的请求被拒绝。

```java
// 匀速排队：超时时间 500ms
rule.setControlBehavior(RuleConstant.CONTROL_BEHAVIOR_QUEUE_WAIT_TIME);
rule.setMaxQueueingTimeMs(500); // 最大排队时间
```

这种策略适合需要「严格匀速」的场景，如调用第三方 API。

```
请求: ████████████████████████████░░░░░░
放行: ██░░░█░░░█░░░█░░░█░░░█░░░█░░░░░░░ (匀速)
拒绝:                      ████████████
```

## 三、限流规则配置方式

### 3.1 代码配置

最简单的方式是代码配置：

```java
// 定义规则
List<FlowRule> rules = new ArrayList<>();
FlowRule rule = new FlowRule("orderService")
    .setCount(100)  // QPS 阈值
    .setGrade(RuleConstant.FLOW_GRADE_QPS)
    .setControlBehavior(RuleConstant.CONTROL_BEHAVIOR_DEFAULT)
    .setMaxQueueingTimeMs(500); // 排队超时

rules.add(rule);

// 加载规则
FlowRuleManager.loadRules(rules);
```

### 3.2 文件配置

可以通过 JSON 文件配置：

```json title="flow-rule.json"
[
  {
    "resource": "orderService",
    "count": 100,
    "grade": 1,
    "controlBehavior": 0,
    "maxQueueingTimeMs": 500
  }
]
```

```java
// 加载文件中的规则
try (InputStream in = new FileInputStream("flow-rule.json")) {
    List<FlowRule> rules = RuleUtil.loadFlowRuleFromFile(in);
    FlowRuleManager.loadRules(rules);
}
```

### 3.3 Sentinel Dashboard 配置（推荐）

Sentinel 提供了一个 Dashboard，可以**动态配置规则**：

```yaml
# application.yml
spring:
  cloud:
    sentinel:
      transport:
        port: 8719  # Sentinel 客户端的 API 端口
        dashboard: localhost:8080  # Dashboard 地址
```

启动 Dashboard：

```bash
java -jar sentinel-dashboard.jar
```

在 Dashboard 上可以：

- 查看所有资源的状态
- 动态添加、修改、删除限流规则
- 查看实时监控数据
- 配置规则生效后实时推送到客户端

:::tip 动态配置原理

Sentinel Dashboard 通过 **HTTP API** 与客户端通信。规则配置在 Dashboard 上后，会通过 `GET /rules/flow` 等接口推送给客户端，客户端收到后调用 `FlowRuleManager.loadRules()` 加载规则。

这种方式**不需要重启服务**，实现了规则的动态调整。

:::

## 四、Sentinel 的埋点方式

### 4.1 SphU 埋点

最基础的埋点方式：

```java
public void order(String userId) {
    Entry entry = null;
    try {
        // 埋点
        entry = SphU.entry("orderService");

        // 业务逻辑
        Order order = orderService.createOrder(userId);
        return order;

    } catch (BlockException e) {
        // 被限流或熔断
        throw new RateLimitException("系统繁忙，请稍后再试");
    } finally {
        // 务必退出
        if (entry != null) {
            entry.exit();
        }
    }
}
```

### 4.2 注解埋点

Sentinel 支持 `@SentinelResource` 注解，简化埋点：

```java
@Service
public class OrderService {

    @SentinelResource(
        value = "createOrder",  // 资源名
        blockHandler = "createOrderBlockHandler",  // 限流/熔断时的处理方法
        fallback = "createOrderFallback"           // 异常降级方法
    )
    public Order createOrder(String userId) {
        // 业务逻辑
        return orderRepository.save(new Order(userId));
    }

    // BlockException 处理方法
    public Order createOrderBlockHandler(String userId, BlockException e) {
        return Order.placeholder("系统繁忙，请稍后再试");
    }

    // 异常降级方法
    public Order createOrderFallback(String userId, Throwable t) {
        log.error("createOrder failed", t);
        return Order.placeholder("服务异常，请稍后再试");
    }
}
```

### 4.3 Spring Cloud Alibaba 集成

如果使用 Spring Cloud Alibaba，Sentinel 可以与 OpenFeign、Sentinel 自动集成：

```yaml
# application.yml
spring:
  cloud:
    sentinel:
      enabled: true
      transport:
        dashboard: localhost:8080
```

```java
@FeignClient(name = "user-service", fallback = UserClientFallback.class)
public interface UserClient {
    @GetMapping("/user/{id}")
    User getUser(@PathVariable("id") Long id);
}

@Component
public class UserClientFallback implements UserClient {
    @Override
    public User getUser(Long id) {
        return User.placeholder(); // 降级返回
    }
}
```

当 `user-service` 被限流或熔断时，会自动触发降级，返回 `User.placeholder()`。

## 五、Sentinel 的滑动窗口统计

### 5.1 为什么需要滑动窗口

Sentinel 的限流统计使用滑动窗口算法，而不是固定窗口。

**固定窗口的问题**（以 1 秒为窗口）：

```
窗口 1: 0:00 - 0:59 → 80 个请求（限流阈值 100，通过）
窗口 2: 1:00 - 1:59 → 80 个请求（限流阈值 100，通过）
```

看起来没问题。但如果请求集中在边界：

```
窗口 1: 0:50 - 0:59 → 80 个请求
窗口 2: 1:00 - 1:09 → 80 个请求
```

实际 10 秒内处理了 160 个请求，是限流阈值的 1.6 倍！

**滑动窗口解决了这个问题**：精确计算任意时间窗口内的请求数。

### 5.2 Sentinel 的滑动窗口实现

Sentinel 使用**两层的滑动窗口**：

1. **第一层**：1 秒窗口划分为 2 个子窗口（各 500ms）
2. **第二层**：每秒更新当前的子窗口

```java
// 伪代码
public class StatisticNode {
    // 当前秒的计数器
    private AtomicInteger currentQps = new AtomicInteger(0);
    // 上秒的计数器
    private AtomicInteger previousQps = new AtomicInteger(0);
    // 当前秒的开始时间
    private volatile long currentSecond = System.currentTimeMillis() / 1000;

    public void addPass() {
        long now = System.currentTimeMillis() / 1000;
        if (now != currentSecond) {
            // 秒切换，更新计数器
            previousQps.set(currentQps.getAndSet(0));
            currentSecond = now;
        }
        currentQps.incrementAndGet();
    }

    public long getQps() {
        // 当前秒的请求 + 上一秒的请求（按比例计算）
        return currentQps.get() + previousQps.get();
    }
}
```

Sentinel 的滑动窗口精度是**秒级**，对于大多数场景足够用了。

## 六、集群限流

### 6.1 为什么需要集群限流

单机限流只能保护单个实例，集群限流可以保护整个集群。

但 Sentinel 的单机限流是分布式的——每个节点独立统计，通过 Token Server 协调。

### 6.2 Token Server 模式

Sentinel 支持两种集群限流模式：

| 模式 | 说明 | 适用场景 |
|---|---|---|
| Token Client | 客户端与 Token Server 通信获取 Token | 大规模集群 |
| Token Server | Token Server 独立部署 | 中等规模集群 |

```
                    ┌─────────────────┐
                    │  Token Server   │
                    │  (独立部署)     │
                    └────────┬────────┘
                             │
           ┌─────────────────┼─────────────────┐
           │                 │                 │
      ┌────v────┐       ┌────v────┐       ┌────v────┐
      │ Client  │       │ Client  │       │ Client  │
      │ Node 1  │       │ Node 2  │       │ Node 3  │
      └─────────┘       └─────────┘       └─────────┘
```

### 6.3 集群限流配置

```yaml
spring:
  cloud:
    sentinel:
      cluster:
        mode: client  # 或 server
        server:
          port: 18730  # Token Server 端口
```

```java
// 开启集群限流
FlowRule rule = new FlowRule("orderService");
rule.setCount(1000); // 集群总阈值
rule.setClusterMode(true);
rule.setClusterConfig(new ClusterFlowConfig()
    .setThresholdType(ClusterThresholdType.avg_global)); // 平均分配
```

## 七、与 Hystrix 的对比

| 对比维度 | Sentinel | Hystrix |
|---|---|---|
| 限流 | 支持 QPS、并发线程数 | 不支持 |
| 熔断 | 支持慢调用比例、异常比例、异常数 | 只支持异常比例 |
| 监控 | 自带 Dashboard，实时监控 | 依赖 Hystrix Dashboard |
| 配置 | 支持动态配置 | 需要重启 |
| 线程隔离 | 信号量隔离（不支持超时） | 线程池隔离、信号量隔离 |
| 社区活跃度 | 活跃（Spring Cloud Alibaba） | 已停止维护 |

Sentinel 相比 Hystrix 更轻量、功能更全面，是目前主流的选择。

## 八、生产环境最佳实践

### 8.1 分层限流

推荐采用**分层限流**策略：

| 层级 | 限流器 | 说明 |
|---|---|---|
| 网关层 | Nginx / Spring Cloud Gateway | 分布式限流，保护整个集群 |
| 服务层 | Sentinel | 单机限流 + 熔断，保护单个实例 |

```java
// 在服务层使用 Sentinel
@SentinelResource(value = "createOrder",
    blockHandler = "handleBlock",
    fallback = "handleFallback")
public Order createOrder(String userId) {
    // 业务逻辑
}

// 限流处理
public Order handleBlock(String userId, BlockException e) {
    log.warn("Rate limited: {}", e.getClass().getSimpleName());
    return Order.placeholder("系统繁忙，请稍后再试");
}

// 降级处理
public Order handleFallback(String userId, Throwable t) {
    log.error("Service error", t);
    return Order.placeholder("服务异常，请稍后再试");
}
```

### 8.2 合理的阈值设置

Sentinel 的阈值设置需要根据压测结果来定：

1. **单接口压测**：确定每个接口的最大 QPS
2. **设置限流阈值**：通常为最大 QPS 的 70%-80%
3. **设置熔断阈值**：通常为最大 QPS 的 90%

```java
// 基于压测结果配置
FlowRule rule = new FlowRule("createOrder")
    .setCount(5000)  // 最大 QPS 7000 的 70%
    .setGrade(RuleConstant.FLOW_GRADE_QPS)
    .setControlBehavior(RuleConstant.CONTROL_BEHAVIOR_DEFAULT);

// 慢调用比例熔断：RT 超过 1s 的比例超过 30% 时熔断
DegradeRule degradeRule = new DegradeRule("createOrder")
    .setGrade(RuleConstant.DEGRADE_GRADE_RT)
    .setCount(1000)  // 阈值 1s
    .setSlowRatioThreshold(0.3)  // 30% 慢调用
    .setMinRequestAmount(5)  // 最小请求数
    .setStatIntervalMs(1000);  // 统计窗口 1s
```

### 8.3 监控与告警

Sentinel Dashboard 提供了实时监控，但你需要配置告警：

```yaml
spring:
  cloud:
    sentinel:
      heartbeat:
        enabled: true  # 开启心跳
      transport:
        dashboard: localhost:8080
```

建议监控以下指标：

- **通过 QPS**：正常通过的请求
- **拒绝 QPS**：被限流拒绝的请求
- **平均响应时间**：接口的平均 RT
- **异常比例**：发生异常的比例

Sentinel 可以导出指标到 Prometheus：

```yaml
spring:
  cloud:
    sentinel:

        enabled: true
```

配合 Grafana Dashboard，可以实现生产级监控。

## 九、面试追问：Sentinel 相关问题

### 9.1 问题一：Sentinel 的限流原理是什么？

**参考答案**：

Sentinel 的限流原理分为三层：

1. **统计数据收集**：StatisticSlot 负责统计每个资源的 QPS、并发线程数等指标
2. **规则匹配**：FlowSlot 根据限流规则判断是否超限
3. **流量控制**：根据控制行为（直接拒绝/冷启动/匀速排队）决定如何处理

核心是**滑动窗口算法**：将 1 秒划分为多个窗口，实时计算当前窗口的 QPS。

### 9.2 问题二：Sentinel 和 Hystrix 有什么区别？

**参考答案**：

| 对比维度 | Sentinel | Hystrix |
|---|---|---|
| 限流能力 | 支持 QPS、线程数 | 不支持 |
| 熔断策略 | RT、异常比例、异常数 | 异常比例 |
| 配置方式 | 支持动态配置 | 静态配置 |
| 监控 | 自带 Dashboard | 需要额外部署 |
| 维护状态 | 活跃 | 已停止 |

Sentinel 更轻量、更易用，是 Spring Cloud Alibaba 的默认熔断组件。

### 9.3 问题三：冷启动模式的原理是什么？

**参考答案**：

冷启动模式通过**渐进式增加阈值**实现：

1. 初始阈值 = `count / coldFactor`（coldFactor 默认 3）
2. 每秒增加 `count / warmUpPeriod` 个单位
3. 达到预热时间后，阈值恢复正常

冷启动适用于依赖冷启动资源的场景（如连接池、缓存预热），可以避免瞬间流量压垮系统。

### 9.4 问题四：如何动态配置 Sentinel 的限流规则？

**参考答案**：

有三种方式：

1. **Sentinel Dashboard**：Web 界面配置，实时推送规则
2. **HTTP API**：调用 Sentinel 客户端的 HTTP 接口
3. **Nacos 配置**：如果使用 Nacos，可以将规则存储在 Nacos，客户端监听变化

推荐使用 Sentinel Dashboard + Nacos 的组合，实现规则的持久化和动态更新。

## 延伸思考

## 总结

Sentinel 是功能全面的限流熔断框架：

- **限流规则**：支持 QPS 和线程数两个维度，四种拒绝策略
- **埋点方式**：支持 SphU 埋点和注解埋点
- **动态配置**：支持 Dashboard 动态推送规则
- **集群限流**：支持 Token Server 模式

在实际项目中，Sentinel 是服务层限流和熔断的首选方案，配合网关层的分布式限流，构建完整的流量防护体系。
