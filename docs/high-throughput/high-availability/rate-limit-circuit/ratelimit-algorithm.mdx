# 限流算法：固定窗口、滑动窗口、漏桶、令牌桶

凌晨三点，你被告警叫醒：「线上服务响应超时，大量请求堆积！」你登录服务器一看，发现某个接口的 QPS 突然暴涨了 10 倍，数据库连接池耗尽，整个系统陷入瘫痪。

如果此时你有一套限流机制，该有多好。

但限流不是简单的「拒绝请求」。不同的限流算法，有不同的精度、资源消耗和突发处理能力。用错了算法，轻则浪费资源，重则影响业务。**这篇文章，就是你的限流算法完全指南。**

## 一、为什么要限流——从「洪水泛滥」到「细水长流」

在高并发系统中，流量洪峰是常态，而不是意外。

想象一下：一个电商系统在双十一零点开抢，瞬间涌入的请求量是平时的 100 倍。如果系统没有保护措施，会发生什么？

- 数据库连接池被瞬间耗尽
- CPU 被大量请求占满
- 内存出现 OOM
- 第三方服务被拖垮
- 最终，整个系统雪崩

限流（Rate Limiting）的核心思想很简单：**在系统可承受的范围内，平稳处理请求；超出范围的请求，要么排队等待，要么直接拒绝。**

:::tip 限流 vs 熔断 vs 降级

很多同学搞不清这三者的区别：

- **限流**：控制请求速率，保护系统不被冲垮
- **熔断**：检测故障服务，快速失败，避免雪崩
- **降级**：放弃非核心功能，保证核心功能可用

限流是**预防性**手段，熔断和降级是**应急性**手段。三者配合使用，构成分层防护体系。

:::

## 二、固定窗口计数器——最简单，也最危险

### 2.1 算法原理

固定窗口（Fixed Window）算法是最简单直观的限流方式。它的核心思想是：**把时间划分为固定大小的窗口，在每个窗口内统计请求次数，超过阈值就拒绝。**

实现思路：

1. 选取一个时间窗口长度（如 1 秒）
2. 在每个窗口开始时，将计数器重置为 0
3. 每次请求进来，计数器 +1
4. 如果计数器超过阈值，拒绝请求

### 2.2 代码实现

```java
public class FixedWindowRateLimiter {
    private final int maxRequests;
    private final long windowSizeMillis;
    private long windowStart;
    private AtomicInteger count;

    public FixedWindowRateLimiter(int maxRequests, long windowSizeMillis) {
        this.maxRequests = maxRequests;
        this.windowSizeMillis = windowSizeMillis;
        this.windowStart = System.currentTimeMillis();
        this.count = new AtomicInteger(0);
    }

    public synchronized boolean tryAcquire() {
        long now = System.currentTimeMillis();
        // 窗口过期，重置计数器
        if (now - windowStart >= windowSizeMillis) {
            windowStart = now;
            count.set(0);
        }
        return count.incrementAndGet() <= maxRequests;
    }
}
```

### 2.3 致命缺陷：边界突变问题

固定窗口算法有一个致命的缺陷——**边界突变问题（Boundary Spike Problem）**。

假设限流阈值是 100 QPS，窗口长度是 1 秒。看下面的时间线：

```
窗口 1 [0s ~ 1s): 0:00 - 0:59 → 收到 100 个请求（达到上限）
窗口 2 [1s ~ 2s): 1:00 - 1:59 → 又收到 100 个请求（达到上限）
```

这看起来没问题。但实际请求可能这样分布：

```
窗口 1 [0s ~ 1s): 0:50 - 0:59 → 收到 100 个请求（达到上限）
窗口 2 [1s ~ 2s): 1:00 - 1:09 → 又收到 100 个请求（达到上限）
```

**问题来了**：在 0:59 到 1:09 这短短 10 秒内，实际上处理了 200 个请求，是限流阈值的 2 倍！

这就是边界突变问题：**窗口边界的两个时刻，请求可以瞬间叠加，造成流量突刺。**

:::warning 固定窗口算法的适用场景

固定窗口算法适合对精度要求不高的场景，或者作为初步的限流手段。如果你的业务能容忍一定的流量波动，可以使用它。

但如果你的系统对流量控制要求严格（比如秒杀系统），请使用滑动窗口或令牌桶算法。

:::

## 三、滑动窗口日志——精度至上，内存买单

### 3.1 算法原理

滑动窗口日志（Sliding Window Log）算法解决了固定窗口的边界问题。它的核心思想是：**记录每次请求的时间戳，精确计算滑动窗口内的请求数。**

实现思路：

1. 维护一个 SortedSet，记录每次请求的时间戳
2. 每次请求时，删除滑动窗口之外（超过窗口长度）的时间戳
3. 如果集合大小小于阈值，允许请求并添加时间戳；否则拒绝

### 3.2 代码实现

```java
public class SlidingWindowLogRateLimiter {
    private final int maxRequests;
    private final long windowSizeMillis;
    private final ConcurrentHashMap<String, ConcurrentLinkedQueue<Long>> requestLogs = new ConcurrentHashMap<>();

    public SlidingWindowLogRateLimiter(int maxRequests, long windowSizeMillis) {
        this.maxRequests = maxRequests;
        this.windowSizeMillis = windowSizeMillis;
    }

    public boolean tryAcquire(String key) {
        long now = System.currentTimeMillis();
        long windowStart = now - windowSizeMillis;

        ConcurrentLinkedQueue<Long> log = requestLogs.computeIfAbsent(
            key, k -> new ConcurrentLinkedQueue<>()
        );

        // 清理过期的请求记录
        while (log.peek() != null && log.peek() <= windowStart) {
            log.poll();
        }

        // 检查是否超限
        if (log.size() >= maxRequests) {
            return false;
        }

        // 记录当前请求
        log.offer(now);
        return true;
    }
}
```

### 3.3 优点与代价

**优点**：

- 精度最高，每个请求的时间都被精确记录
- 解决了边界突变问题

**代价**：

- 内存消耗大。每个请求都要存储一个时间戳（通常是 8 字节），10000 QPS 的系统，每秒需要存储 8 万字节的日志数据。
- 清理操作有额外开销

### 3.4 使用场景

滑动窗口日志适合**对精度要求极高、且 QPS 不是特别高**的场景。比如：

- 付费 API 的严格限流
- 核心接口的精确控制
- 需要审计每个请求的场景

## 四、滑动窗口计数器——精度与性能的折中

### 4.1 算法原理

滑动窗口计数器（Sliding Window Counter）是固定窗口和滑动窗口的折中方案。它不再记录每个请求的时间戳，而是将窗口划分为多个子窗口（通常叫 bucket 或 slot），每个子窗口独立计数。

实现思路：

1. 将窗口划分为 N 个子窗口（如 60 个，每个代表 1 秒）
2. 每次请求时，更新当前子窗口的计数器
3. 计算滑动窗口内的总请求数 = 当前子窗口计数 × 占比 + 历史子窗口计数
4. 如果总数超过阈值，拒绝请求

### 4.2 代码实现

```java
public class SlidingWindowCounterRateLimiter {
    private final int maxRequests;
    private final int bucketCount;
    private final long bucketSizeMillis;
    private final AtomicInteger[] buckets;
    private volatile long windowStart;
    private final AtomicInteger total;

    public SlidingWindowCounterRateLimiter(int maxRequests, int windowSeconds) {
        this.maxRequests = maxRequests;
        this.bucketCount = windowSeconds; // 每秒一个桶
        this.bucketSizeMillis = 1000;
        this.buckets = new AtomicInteger[bucketCount];
        for (int i = 0; i < bucketCount; i++) {
            buckets[i] = new AtomicInteger(0);
        }
        this.windowStart = System.currentTimeMillis();
        this.total = new AtomicInteger(0);
    }

    public synchronized boolean tryAcquire() {
        long now = System.currentTimeMillis();
        long elapsed = now - windowStart;

        // 窗口滑动，需要重置所有桶
        if (elapsed >= bucketSizeMillis * bucketCount) {
            windowStart = now;
            for (AtomicInteger bucket : buckets) {
                bucket.set(0);
            }
            total.set(0);
        } else {
            // 只清理过期的桶
            int expiredBuckets = (int) (elapsed / bucketSizeMillis);
            for (int i = 0; i < expiredBuckets; i++) {
                int idx = (int) ((windowStart / bucketSizeMillis + i) % bucketCount);
                int delta = buckets[idx].getAndSet(0);
                total.addAndGet(-delta);
            }
            windowStart = now - (elapsed % (bucketSizeMillis * bucketCount));
        }

        // 当前桶计数 +1
        int currentBucket = (int) ((now / bucketSizeMillis) % bucketCount);
        buckets[currentBucket].incrementAndGet();
        total.incrementAndGet();

        return total.get() <= maxRequests;
    }
}
```

### 4.3 精度 vs 资源

滑动窗口计数器的精度取决于桶的数量。桶越多，精度越高，但内存消耗也越大。

| 桶数量 | 精度 | 内存开销 |
|---|---|---|
| 60 | 1 秒 | 低 |
| 600 | 0.1 秒 | 中 |
| 6000 | 0.01 秒 | 高 |

实际应用中，60 个桶（1 秒精度）已经能满足大多数场景的需求。

## 五、漏桶算法——流量整形的艺术家

### 5.1 算法原理

漏桶（Leaky Bucket）算法的思想来自现实中的漏桶：**不管流入的水有多快，漏出的水始终是均匀的。**

实现思路：

1. 维护一个队列（桶），队列长度是桶的容量
2. 请求进来时，加入队列尾部
3. 后台有一个恒定速率的消费者，从队列头部取出请求并处理
4. 如果队列满了，新请求被拒绝

### 5.2 代码实现

```java
public class LeakyBucketRateLimiter {
    private final int bucketCapacity;
    private final int leakRate; // 每秒漏出的请求数
    private final Queue<Long> bucket = new ConcurrentLinkedQueue<>();
    private volatile long lastLeakTime = System.currentTimeMillis();
    private final Object lock = new Object();

    public LeakyBucketRateLimiter(int bucketCapacity, int leakRatePerSecond) {
        this.bucketCapacity = bucketCapacity;
        this.leakRate = leakRatePerSecond;
    }

    public boolean tryAcquire() {
        synchronized (lock) {
            leak(); // 先漏桶
            if (bucket.size() >= bucketCapacity) {
                return false; // 桶满了，拒绝
            }
            bucket.offer(System.currentTimeMillis());
            return true;
        }
    }

    private void leak() {
        long now = System.currentTimeMillis();
        long elapsed = now - lastLeakTime;
        // 计算应该漏掉多少请求
        int leakCount = (int) (elapsed * leakRate / 1000);
        for (int i = 0; i < leakCount && !bucket.isEmpty(); i++) {
            bucket.poll();
        }
        lastLeakTime = now;
    }
}
```

### 5.3 特点分析

漏桶算法有两个显著特点：

**优点**：

- 流量整形能力强，输出速率恒定
- 有效平滑突发流量

**缺点**：

- 突发请求无法立即处理，会被放入桶中等待
- 如果桶已满，突发请求会被直接丢弃，而不是暂存
- 无法应对突发流量——无论请求来得多急，输出都是匀速的

### 5.4 使用场景

漏桶算法适合**需要严格流量整形**的场景，比如：

- 第三方 API 调用（对方有严格的 QPS 限制）
- 消息队列的限流（保证消费速率稳定）
- 银行转账等需要匀速处理的业务

## 六、令牌桶算法——突发处理的艺术家

### 6.1 算法原理

令牌桶（Token Bucket）算法是漏桶的「升级版」，核心区别是：**桶里放的是令牌，不是请求。**

实现思路：

1. 系统以固定速率向桶中添加令牌
2. 桶有容量上限，超过上限的令牌被丢弃
3. 每次请求需要从桶中获取一个令牌
4. 如果桶中没有令牌，请求被拒绝或等待

### 6.2 代码实现

```java
public class TokenBucketRateLimiter {
    private final long capacity; // 桶容量
    private final double refillRate; // 每秒添加的令牌数
    private volatile long tokens;
    private volatile long lastRefillTime;

    public TokenBucketRateLimiter(long capacity, double refillRate) {
        this.capacity = capacity;
        this.refillRate = refillRate;
        this.tokens = capacity; // 初始时桶满
        this.lastRefillTime = System.currentTimeMillis();
    }

    public synchronized boolean tryAcquire() {
        refill();
        if (tokens >= 1) {
            tokens -= 1;
            return true;
        }
        return false;
    }

    public synchronized boolean tryAcquire(int permits) {
        refill();
        if (tokens >= permits) {
            tokens -= permits;
            return true;
        }
        return false;
    }

    private void refill() {
        long now = System.currentTimeMillis();
        double elapsed = (now - lastRefillTime) / 1000.0;
        double newTokens = elapsed * refillRate;
        tokens = Math.min(capacity, tokens + newTokens);
        lastRefillTime = now;
    }
}
```

### 6.3 令牌桶 vs 漏桶——核心区别

这是面试中最常问的问题。两者最核心的区别在于**突发处理能力**：

| 特性 | 令牌桶 | 漏桶 |
|---|---|---|
| 突发处理 | 支持，能处理突发流量 | 不支持，输出速率恒定 |
| 令牌积累 | 支持，桶满后令牌可积累 | 不支持，桶满则溢出 |
| 流量整形 | 平滑输出 + 允许突发 | 严格均匀输出 |
| 适用场景 | 大多数限流场景 | 严格流量整形场景 |

**举个例子**：假设令牌桶容量是 100，每秒生成 10 个令牌。

- 系统空闲了 10 秒，桶中积累了 100 个令牌
- 突然来了 80 个请求，这 80 个请求可以一次性通过（因为桶中有足够的令牌）
- 之后，系统以每秒 10 个令牌的速率恢复

这就是令牌桶的**突发处理能力**——平时积累的令牌，可以在关键时刻一次性使用。

:::tip 令牌桶的「预消费」机制

Guava RateLimiter 的令牌桶实现支持「预消费」，即允许一次获取多个令牌，用于处理更大的突发流量。这个机制让 RateLimiter 在应对突发请求时更加灵活。

:::

## 七、四大算法横向对比

| 维度 | 固定窗口 | 滑动窗口日志 | 滑动窗口计数 | 漏桶 | 令牌桶 |
|---|---|---|---|---|---|
| 算法精度 | 低（有边界问题） | 高（精确到请求） | 中（取决于桶数量） | 高 | 高 |
| 内存消耗 | 低（1 个计数器） | 高（每个请求一个时间戳） | 中（桶数量个计数器） | 低（队列长度） | 低（桶容量 + 计数器） |
| 突发处理能力 | 无（严格均匀） | 无（严格均匀） | 无（严格均匀） | 无（严格均匀） | **支持**（允许积累） |
| 实现复杂度 | 低 | 中 | 中 | 中 | 中 |
| 时间序列依赖 | 无 | 有（清理操作） | 部分（滑动计算） | 无 | 部分（令牌补充） |
| 并发友好性 | 好 | 差（需要清理） | 好 | 好（队列） | 好（原子操作） |

### 7.1 算法选择指南

**选择令牌桶**（推荐）：需要处理突发流量的大多数场景。Guava RateLimiter 是成熟的工业级实现。

**选择滑动窗口计数**：对精度有一定要求，且能接受一定的内存开销。Redis 的滑动窗口限流常用这个思路。

**选择漏桶**：需要严格控制输出速率的场景，比如调用第三方 API。

**选择固定窗口**：精度要求不高，作为初步的限流手段，或者在单机内存中使用。

:::warning 不要在分布式环境中自己实现限流算法

单机限流算法在分布式环境下是不准确的，因为每个节点的计数是独立的。除非你使用分布式存储（Redis、数据库）来协调，否则无法保证全局限流的准确性。

:::

## 八、生产级实现：Guava RateLimiter

Java 生态中，Guava RateLimiter 是令牌桶算法的工业级实现，被广泛应用于生产环境。

### 8.1 基本使用

```java
import com.google.common.util.concurrent.RateLimiter;

public class RateLimiterExample {
    public static void main(String[] args) {
        // 创建每秒 100 个令牌的限流器
        RateLimiter limiter = RateLimiter.create(100.0);

        for (int i = 0; i < 10; i++) {
            // 尝试获取令牌，最多等待 1 秒
            double waitTime = limiter.acquire(1);
            System.out.printf("请求 %d 获取令牌，等待 %.3f 秒%n", i, waitTime);
        }
    }
}
```

### 8.2 预消费与突发处理

```java
public class BurstHandling {
    public static void main(String[] args) {
        // 创建每秒 100 个令牌的限流器，预热 5 秒后可达到最大突发量
        RateLimiter limiter = RateLimiter.create(100.0, 5, TimeUnit.SECONDS);

        // 一次性获取 50 个令牌（突发流量）
        limiter.acquire(50);
        System.out.println("突发请求处理完成");

        // 之后按每秒 100 个令牌的速率恢复
        limiter.acquire(1);
    }
}
```

### 8.3 带超时的限流

```java
public class RateLimitWithTimeout {
    private final RateLimiter limiter;

    public RateLimitWithTimeout(double permitsPerSecond) {
        this.limiter = RateLimiter.create(permitsPerSecond);
    }

    public boolean tryAcquire(Duration timeout) {
        return limiter.tryAcquire(1, timeout);
    }

    public CompletableFuture<String> processRequest(String request) {
        return CompletableFuture.supplyAsync(() -> {
            limiter.acquire(); // 阻塞等待令牌
            return handleRequest(request);
        });
    }
}
```

## 九、面试追问：限流算法相关问题

### 9.1 问题一：令牌桶和漏桶的区别是什么？

**参考答案**：

令牌桶和漏桶的核心区别在于**输出方式**：

- **漏桶**：无论请求多急，输出速率始终恒定。突发请求会被放入桶中等待，如果桶满则被拒绝。
- **令牌桶**：允许突发流量。只要桶中有令牌，请求就可以立即通过。桶以固定速率补充令牌。

**形象比喻**：

- 漏桶像是医院的挂号系统，每分钟只叫一个号，不管外面排了多少人。
- 令牌桶像是机场的安检口，平时积累的排队能力，在航班高峰时可以加速消化。

**实际选择**：大多数场景选择令牌桶，因为它能更好地利用系统的瞬时处理能力。

### 9.2 问题二：滑动窗口和滑动窗口计数器的区别？

**参考答案**：

| 对比 | 滑动窗口日志 | 滑动窗口计数器 |
|---|---|---|
| 精度 | 精确到每个请求 | 取决于子窗口数量 |
| 内存 | 每个请求一个时间戳 | 每个子窗口一个计数器 |
| 计算 | 实时统计窗口内请求数 | 统计子窗口计数器的和 |
| 适用 QPS | 较低（< 1000） | 较高（可达数万） |

滑动窗口计数器是滑动窗口日志的「空间换时间」优化。

### 9.3 问题三：如何选择限流算法？

**参考答案**：

选择限流算法需要考虑三个因素：**精度要求、突发处理需求、资源消耗**。

- **高精度 + 突发处理**：选令牌桶（Guava RateLimiter）
- **高精度 + 低 QPS**：选滑动窗口日志
- **中等精度 + 高 QPS**：选滑动窗口计数器
- **严格流量整形**：选漏桶

### 9.4 问题四：单机限流和分布式限流的区别？

**参考答案**：

- **单机限流**：限流器运行在单个节点上，只保护单个实例。实现简单，性能高，但无法保护整个集群。
- **分布式限流**：限流器基于分布式存储（Redis），对整个集群统一限流。精度高，但有网络开销和一致性问题。

**常见实践**：先在网关层做分布式限流（保护整个集群），再在服务内部做单机限流（保护单个实例）。

## 总结

限流算法是保护系统稳定性的第一道防线。不同的算法有不同的特点：

- **固定窗口**：简单但有边界问题，适合作为初步限流
- **滑动窗口日志**：精度最高，但内存消耗大
- **滑动窗口计数**：精度和性能的良好折中
- **漏桶**：严格流量整形，但无法处理突发
- **令牌桶**：推荐方案，支持突发且实现成熟

在实际项目中，令牌桶（Guava RateLimiter 是首选）和基于 Redis 的分布式限流是最佳选择。
