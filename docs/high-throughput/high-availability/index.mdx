# 高可用架构设计 - 概述

想象一下：双十一零点，你正在抢购一部 iPhone。页面流畅、下单秒级完成，你满心欢喜地在朋友圈分享战果。但就在此刻，另一位用户的订单页面转了五分钟圈，最后弹出「系统繁忙，请稍后再试」。

同样的系统，为什么同一时刻的用户体验天差地别？

答案可能就在一个词：**高可用**。

一个系统能不能扛住流量洪峰、在故障时快速恢复、在降级时依然提供核心功能——这些都取决于高可用架构的设计水平。这不是锦上添花，而是现代互联网系统的生死线。

---

## 一、什么是高可用？为什么它比性能更重要？

### 1.1 高可用的定义

高可用（High Availability, HA）是指系统在规定的条件下和规定的时间范围内，持续提供正常服务水平的能力。这里的「规定条件」包括硬件故障、软件升级、网络波动等常见的非灾难性事件；「规定时间范围」则对应了我们常说的几个 9。

### 1.2 几个 9 的含义

| 可用性 | 每日停机时间 | 每年停机时间 | 典型场景 |
|---|---|---|---|
| 99%（2个9） | 14.4 分钟 | 3.65 天 | 个人博客、内部系统 |
| 99.9%（3个9） | 1.44 分钟 | 8.76 小时 | 普通企业应用 |
| 99.99%（4个9） | 8.64 秒 | 52.6 分钟 | 主流互联网应用 |
| 99.999%（5个9） | 0.86 秒 | 5.26 分钟 | 金融支付、电信核心 |

每提升一个 9，难度不是线性增长，而是指数级增长。从 99% 到 99.9%，停机时间从 3.65 天降到 8.76 小时；从 99.9% 到 99.99%，从 8.76 小时降到 52 分钟；从 99.99% 到 99.999%，从 52 分钟降到 5 分钟。每一步都需要在架构设计、运维自动化、故障恢复上投入大量心血。

### 1.3 高可用 vs 高性能 vs 高并发

很多人容易混淆这三个概念。它们之间的关系是：

- **高性能（High Performance）**：系统在给定资源下，能以多快的速度处理请求。核心指标是延迟（Latency）和吞吐量（Throughput）。
- **高并发（High Concurrency）**：系统在任意时刻能同时处理多少请求。核心指标是并发数、QPS、TPS。
- **高可用（High Availability）**：系统在故障条件下能持续提供服务的程度。核心指标是可用性百分比、MTTF、MTTR。

三者的关系可以这样理解：

```
高性能决定了系统能跑多快
高并发决定了系统能扛多少
高可用决定了系统能跑多久

没有高可用，高性能和高并发都是空中楼阁——
流量再大、响应再快，系统崩了就是零。
```

:::tip 三者的优先级

对于大多数互联网系统，建议的优先级是：**高可用 > 高并发 > 高性能**。一个系统即使性能一般，只要稳定可用，就能服务用户；但一个动不动崩溃的系统，性能再好也没有意义。

:::

## 二、高可用的核心度量指标

### 2.1 SLA、SLO、SLI 三者关系

- **SLI（Service Level Indicator）**：服务等级指标，是衡量服务质量的实际量化指标，如延迟、可用率、吞吐量。
- **SLO（Service Level Objective）**：服务等级目标，是 SLI 的目标值，如 p99 延迟 < 200ms，可用率 > 99.95%。
- **SLA（Service Level Agreement）**：服务等级协议，是 SLO 的对外承诺，通常包含补偿条款。

简单来说：**SLI 是测量值，SLO 是目标值，SLA 是承诺值**。SLA 一旦签出，就是法律承诺，违约了要赔钱。

### 2.2 MTTF、MTTR、MTBF

- **MTTF（Mean Time To Failure）**：平均故障前时间，指系统平均能正常运行多久不发生故障。
- **MTTR（Mean Time To Repair）**：平均恢复时间，指从故障发生到恢复正常服务所需的平均时间。
- **MTBF（Mean Time Between Failures）**：平均故障间隔时间，指两次故障之间的平均时间间隔。

```
MTBF = MTTF + MTTR
可用性 = MTTF / (MTTF + MTTR) = MTTF / MTBF
```

从这个公式可以看出，**降低 MTTR（快速恢复）和提升 MTTF（减少故障）是提高可用性的两个方向**。架构设计的重点，往往在降低 MTTR 上——故障不可避免，但快速恢复是可以做到的。

### 2.3 RTO 与 RPO

- **RTO（Recovery Time Objective）**：恢复时间目标，指从故障发生到系统恢复可用的最长时间。
- **RPO（Recovery Point Objective）**：恢复点目标，指系统能容忍的最大数据丢失量，以时间衡量。

```
RTO: "系统最多能宕机多久？"
RPO: "系统最多能丢失多久的数据？"
```

比如，RTO = 4 小时、RPO = 1 小时，意味着：系统宕机不能超过 4 小时，且最多允许丢失 1 小时的数据。这两个指标直接决定了灾备方案的设计。

## 三、高可用架构的三大原则

### 3.1 故障预防：从源头减少故障发生

故障预防的核心思路是**消除单点**和**降低耦合**。

**消除单点**（Single Point of Failure）是指系统中任何一个节点（服务器、数据库、缓存、网关）一旦宕机，整个系统就不可用。消除单点的方式是冗余部署：每个关键组件至少部署两个实例，通过负载均衡或主备切换来保证可用性。

**降低耦合**是指减少组件之间的直接依赖。耦合度越高，一个组件的故障越容易传导到其他组件。降低耦合的方式包括：异步通信、服务拆分、接口隔离、事件驱动。

### 3.2 故障检测：让故障无所遁形

即使做了充分的预防，故障还是会发生。关键是**尽早发现**。

故障检测的手段包括：

- **健康检查**：定期对各服务节点发起探测，及时剔除不健康的实例。
- **心跳机制**：服务定期向注册中心发送心跳，注册中心据此判断服务存活状态。
- **全链路监控**：通过 APM 工具追踪请求全流程，在故障扩散前捕获异常。
- **告警系统**：对关键指标（CPU、内存、延迟、错误率）设置告警阈值，第一时间感知异常。

:::warning 健康检查不是万能的

健康检查只能检测「服务是否存活」，无法检测「服务是否正常」。一个服务可以响应 200 OK，但返回的数据是错误的。这就需要更精细的**业务级健康检查**和**全链路监控**来补充。

:::

### 3.3 故障恢复：快速止血，快速复原

故障恢复是高可用架构的核心竞争力。同样的故障，5 分钟恢复和 50 分钟恢复，给用户和企业带来的损失是天壤之别。

故障恢复的关键能力包括：

- **限流**：在系统濒临过载时，主动拒绝部分请求，保护系统不被压垮。
- **熔断**：当下游服务持续不可用时，主动「熔断」调用，避免无效重试消耗资源。
- **降级**：在核心功能无法满足时，提供备选方案，保证系统依然可用。
- **自动扩容**：根据实时负载动态调整资源，快速响应流量变化。
- **故障转移**：主节点故障时，自动切换到备用节点，实现无感知的故障恢复。

## 四、高可用架构全景图

本模块涵盖高可用架构的五大核心领域：

### 4.1 负载均衡

负载均衡是高可用架构的「交通枢纽」。它将用户请求合理分配到多个服务节点，既能提升系统的处理能力，又能消除单点故障。相关内容包括：

- L4 vs L7 负载均衡的选择
- 一致性哈希算法
- Nginx 高可用（Keepalived + VRRP）
- 客户端负载均衡与 Spring Cloud LoadBalancer

### 4.2 限流与熔断

限流和熔断是保护系统的两大盾牌。限流控制「有多少请求能进来」，熔断控制「有多少请求能出去」。相关内容包括：

- 令牌桶、漏桶、滑动窗口等限流算法
- Guava RateLimiter 单机限流
- Redis + Lua 分布式限流
- Sentinel 限流与熔断
- Hystrix、Sentinel、Resilience4j 三大工具对比

### 4.3 降级与容错

降级是「两害相权取其轻」的智慧。当系统无法提供完整服务时，保留核心功能、放弃非核心功能，是一种务实的选择。相关内容包括：

- 服务降级策略与兜底方案
- 超时处理与最佳实践
- 重试机制与幂等性保证
- 多级降级：页面、请求、数据聚合

### 4.4 高可用架构设计

这一部分聚焦于整体架构设计，涵盖：

- 主备切换与故障转移
- 异地多活：同城双活、两地三中心、三地五中心
- 数据备份与 RTO/RPO
- 灰度发布与金丝雀部署
- 熔断后的服务恢复与预热

### 4.5 分布式高可用核心

在分布式场景下，高可用的挑战更加严峻。相关内容包括：

- ZooKeeper、etcd、Nacos 协调服务对比
- 微服务注册、健康检查、故障剔除
- 全链路监控：SkyWalking、Prometheus
- API 网关高可用
- 数据库高可用：MHA、Galera Cluster

## 五、从案例看高可用架构的演进

### 5.1 从单体到微服务：高可用的代价与收获

在单体架构时代，高可用的核心是「不要宕机」——把所有鸡蛋放在一个篮子里，小心翼翼地维护这个篮子。

进入微服务时代后，高可用的挑战变成了「即使某些篮子倒了，其他篮子依然完好」——服务的拆分天然提供了故障隔离的能力，但同时带来了分布式系统的复杂性。

### 5.2 电商秒杀 vs 12306：高可用设计的两个极端

电商秒杀系统（见 `seckill-design.mdx`）和 12306 抢票系统（见 `ticket-design.mdx`）代表了两种高可用设计的典型挑战：

- **秒杀系统**：流量极大但商品种类少，核心挑战是热点数据和高并发扣减。
- **12306**：商品种类极多（无数车次、座位、时间组合），核心挑战是分布式查询和库存一致性。

两者的共性是：任何一处设计失误，都可能导致系统崩溃。理解这两个案例，就理解了中国互联网高可用架构的深度和复杂度。

## 六、面试中如何回答高可用问题

### 6.1 面试官的考察维度

当面试官问到高可用相关问题时，通常想考察以下几个方面：

1. **知识广度**：你对高可用架构的整体图景了解多少？
2. **实战经验**：你有没有真正处理过生产环境的可用性问题？
3. **分析能力**：面对一个具体场景，你能否给出合理的高可用设计方案？
4. **权衡思维**：你是否理解高可用与成本、性能之间的 trade-off？

### 6.2 答题框架

回答高可用设计类问题，推荐使用这个四步框架：

**第一步：确认需求**
> "我们先明确一下场景：这个系统需要达到多少个 9 的可用性？RTO 和 RPO 的要求是什么？"

**第二步：识别关键路径和单点**
> "我们先梳理一下系统的关键路径，找出所有可能的单点故障点。"

**第三步：给出解决方案**
> "针对每个单点，我建议用以下方案：冗余部署 + 健康检查 + 自动故障转移 + 限流熔断 + 多级降级。"

**第四步：讨论 trade-off**
> "这个方案的代价是运维复杂度增加了一倍，成本上升约 30%。如果预算有限，我们可以考虑简化版方案……"

## 总结

高可用架构不是一套模板，而是一套**思维框架**。它的核心是：

- **预防**：消除单点，降低耦合
- **检测**：早发现，早处理
- **恢复**：快速止血，快速复原

本模块的每一篇文章，都在教你如何实践这三个方向。无论你是准备面试，还是在实际工作中设计系统，这些知识都会成为你最坚实的后盾。

记住：**系统总会出问题，但好的架构让问题变得可控。**
