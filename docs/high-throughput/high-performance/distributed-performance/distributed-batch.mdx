# 分布式批量任务优化

凌晨 3 点，运营同事突然发来消息：「报表系统崩了，导出一亿数据的报表，后台卡死，用户在群里炸锅了。」你赶紧登录服务器查看，发现 CPU 打满、内存溢出、数据库连接池耗尽——一个看似简单的批量导出任务，把整个系统拖下了水。

这不是某个团队的特例，而是分布式批量处理的经典难题：**当数据量突破单机处理极限，如何让任务在合理时间内完成，同时不影响线上服务的稳定性？**

这篇文章，我们从架构设计、任务分片、并行控制、失败处理四个维度，系统讲解分布式批量任务的核心优化思路。

## 一、为什么单机批处理会崩溃

在深入分布式方案之前，先理解单机批处理的瓶颈在哪里。

### 1.1 单机处理的三大困境

**内存困境**：假设每条数据占用 1KB，一亿条数据就是 100GB。哪怕 JVM 堆内存设置为 64GB，一次性加载所有数据也会 OOM。如果为了避免 OOM 而设置很小的批量，每批只处理 100 条，一亿条数据需要处理 100 万次数据库 IO，性能惨不忍睹。

**连接困境**：每个数据库连接都占用服务器资源（内存、文件描述符、线程）。如果每个批次需要 1 个连接，100 万次批次就需要 100 万次连接获取/释放，连接建立的开销可能比实际数据处理还大。

**单点故障困境**：单机处理时，任何一个环节出问题（进程被 kill、机器宕机、网络抖动），整个任务就失败了。没有任何容错机制，要么从头重来，要么永远卡住。

### 1.2 分布式批处理的核心目标

分布式批处理要解决的，就是这三个困境：

- **数据分片**：把一亿条数据拆分成 N 个批次，分布到多台机器上并行处理
- **资源隔离**：批量任务不能影响线上服务，需要独立的资源池
- **容错恢复**：任何一个节点失败，只需要重跑失败的部分，而不是全部

> 面试官问：「单机批处理和分布式批处理的核心区别是什么？」

单机批处理是「把所有鸡蛋放一个篮子」，分布式批处理是「把鸡蛋分到多个篮子，降低单点失败的风险，同时利用多机并行提升处理速度」。

## 二、批量处理架构：三种主流模式

### 2.1 Master-Worker 架构

最直观的分布式批处理架构。Master 负责任务分发，Worker 负责具体执行。

```
┌─────────────┐
│   Master    │  负责任务分发、进度追踪、失败协调
└──────┬──────┘
       │ 任务分片
       ▼
┌─────────────┐  ┌─────────────┐  ┌─────────────┐
│   Worker 1  │  │   Worker 2  │  │   Worker N  │
│  处理分片1  │  │  处理分片2  │  │  处理分片N  │
└─────────────┘  └─────────────┘  └─────────────┘
```

**Master 的核心职责**：
- 任务切分：将大任务拆分成小任务
- 任务分发：把任务分配给 Worker
- 进度追踪：记录每个任务的完成状态
- 失败处理：超时重试、故障转移

**Worker 的核心职责**：
- 任务执行：按照分配的任务执行具体逻辑
- 状态上报：定期向 Master 汇报进度
- 异常处理：任务失败时记录错误信息

```java
public class Master {
    // 任务队列
    private BlockingQueue<Task> taskQueue = new LinkedBlockingQueue<>();
    // Worker 线程池
    private ExecutorService workerPool;
    // 结果收集
    private Map<String, Object> results = new ConcurrentHashMap<>();
    // 任务状态追踪
    private AtomicInteger completedCount = new AtomicInteger(0);
    
    public void submitJob(Job job) {
        // 1. 任务切分
        List<Task> tasks = splitTasks(job);
        taskQueue.addAll(tasks);
        
        // 2. 启动 Worker
        for (int i = 0; i < workerCount; i++) {
            workerPool.submit(new Worker(taskQueue, results, completedCount));
        }
        
        // 3. 等待完成
        waitForCompletion(job.getTotalCount());
    }
    
    private List<Task> splitTasks(Job job) {
        List<Task> tasks = new ArrayList<>();
        long totalRecords = job.getTotalRecords();
        int batchSize = 10000;
        
        for (long start = 0; start < totalRecords; start += batchSize) {
            long end = Math.min(start + batchSize, totalRecords);
            tasks.add(new Task(job.getId(), start, end));
        }
        return tasks;
    }
}
```

```java
public class Worker implements Runnable {
    private BlockingQueue<Task> taskQueue;
    private Map<String, Object> results;
    private AtomicInteger completedCount;
    
    @Override
    public void run() {
        while (true) {
            Task task = taskQueue.poll();
            if (task == null) {
                break; // 队列为空，任务完成
            }
            
            try {
                // 执行任务
                processTask(task);
                completedCount.incrementAndGet();
                // 上报进度
                reportProgress(task);
            } catch (Exception e) {
                // 处理失败，标记重试
                handleFailure(task, e);
            }
        }
    }
    
    private void processTask(Task task) {
        // 从数据库读取分片数据
        List<Record> records = database.query(
            task.getStartId(), 
            task.getEndId()
        );
        
        // 业务处理
        for (Record record : records) {
            processRecord(record);
        }
        
        // 写入结果
        writeResults(record.getId(), record.getProcessedResult());
    }
}
```

### 2.2 MapReduce 思想：分而治之

MapReduce 是 Google 提出的分布式计算模型，核心思想是「分而治之」：

- **Map 阶段**：把大任务拆分成多个小任务，并行处理
- **Reduce 阶段**：把 Map 的结果合并，得到最终结果

```java
// MapReduce 风格的批量处理
public class BatchProcessor {
    
    public void processLargeDataset(List<Record> records) {
        // 1. Map：数据分片 + 并行处理
        int parallelism = Runtime.getRuntime().availableProcessors();
        List<List<Record>> partitions = partition(records, parallelism);
        
        List<Future<List<ProcessedResult>>> futures = 
            executor.invokeAll(partitions.stream()
                .map(part -> Executors.callable(() -> mapPhase(part)))
                .collect(Collectors.toList()));
        
        // 2. Reduce：结果合并
        List<ProcessedResult> finalResults = new ArrayList<>();
        for (Future<List<ProcessedResult>> future : futures) {
            finalResults.addAll(future.get());
        }
        
        // 3. 写入最终结果
        writeFinalResults(finalResults);
    }
    
    private List<ProcessedResult> mapPhase(List<Record> partition) {
        List<ProcessedResult> results = new ArrayList<>();
        for (Record record : partition) {
            results.add(processRecord(record));
        }
        return results;
    }
}
```

MapReduce 的优势在于：**天然支持水平扩展**。数据量增加时，只需要增加机器，Map 和 Reduce 的逻辑无需修改。

### 2.3 消息队列驱动架构

对于需要限流、削峰的场景，MQ 驱动的架构更为合适。

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   Producer  │ ──▶ │   Message   │ ──▶ │   Worker    │
│ (任务生成)   │     │   Queue     │     │ (任务消费)  │
└─────────────┘     └─────────────┘     └─────────────┘
                           ▲
                           │ 消费确认
```

**核心流程**：

1. Producer 将任务切分成小批次，发送到 MQ
2. Worker 从 MQ 消费消息，执行任务
3. Worker 消费成功后 ACK 消息，失败则 NACK 重新入队

```java
// MQ 驱动的批量处理
@Service
public class BatchTaskProducer {
    
    @Autowired
    private RabbitTemplate rabbitTemplate;
    
    public void submitBatchJob(long totalRecords, int batchSize) {
        long totalBatches = (totalRecords + batchSize - 1) / batchSize;
        
        for (long i = 0; i < totalBatches; i++) {
            long startId = i * batchSize;
            long endId = Math.min(startId + batchSize, totalRecords);
            
            BatchTaskMessage message = new BatchTaskMessage();
            message.setBatchId(UUID.randomUUID().toString());
            message.setStartId(startId);
            message.setEndId(endId);
            message.setTimestamp(System.currentTimeMillis());
            
            // 发送消息到队列
            rabbitTemplate.convertAndSend(
                "batch.exchange", 
                "batch.task", 
                message
            );
        }
    }
}

@Component
public class BatchTaskConsumer {
    
    @RabbitListener(queues = "batch.task.queue")
    public void processBatch(BatchTaskMessage message) {
        try {
            // 1. 根据 ID 范围查询数据
            List<Record> records = recordMapper.selectByIdRange(
                message.getStartId(), 
                message.getEndId()
            );
            
            // 2. 业务处理
            for (Record record : records) {
                processRecord(record);
            }
            
            // 3. ACK 消息（自动确认）
        } catch (Exception e) {
            // NACK，消息重新入队
            throw new AmqpRejectAndDontRequeueException(e);
        }
    }
}
```

**MQ 架构的优势**：
- **天然限流**：消费速率由 Worker 数量控制，不会打垮下游
- **失败重试**：消息持久化，进程崩溃不会丢失任务
- **削峰填谷**：高峰期积压的消息，平峰期慢慢处理

## 三、任务分片策略：如何切分是关键

### 3.1 按 ID 范围分片

最简单的分片策略，适用于主键连续的场景。

```java
public class IdRangePartitioner {
    
    public List<Task> partitionByIdRange(long totalRecords, int workerCount) {
        List<Task> tasks = new ArrayList<>();
        long batchSize = (totalRecords + workerCount - 1) / workerCount;
        
        for (int i = 0; i < workerCount; i++) {
            long startId = i * batchSize;
            long endId = Math.min(startId + batchSize, totalRecords);
            
            if (startId < endId) {
                tasks.add(new Task(startId, endId));
            }
        }
        return tasks;
    }
}
```

**优点**：实现简单，分片均匀
**缺点**：如果数据分布不均匀（如某些时间段数据密集），会导致部分 Worker 负载过重

### 3.2 按哈希分片

根据记录的哈希值分配到不同分区，适合需要打散数据的场景。

```java
public class HashPartitioner {
    
    public int determinePartition(Record record, int workerCount) {
        int hash = record.getKey().hashCode();
        return Math.abs(hash % workerCount);
    }
    
    public List<List<Record>> partitionByHash(List<Record> records, int workerCount) {
        List<List<Record>> partitions = new ArrayList<>();
        for (int i = 0; i < workerCount; i++) {
            partitions.add(new ArrayList<>());
        }
        
        for (Record record : records) {
            int partition = determinePartition(record, workerCount);
            partitions.get(partition).add(record);
        }
        return partitions;
    }
}
```

**优点**：数据打散均匀
**缺点**：无法保证每个分片的大小一致，可能出现数据倾斜

### 3.3 动态分片：Worker 数量动态调整

固定分片数的问题是：如果某个 Worker 处理速度快，它完成后就空闲了，而慢的 Worker 还在苦苦挣扎。

动态分片的思路：**Worker 完成后，主动从 Master 领取新任务，而不是等待分配。**

```java
public class DynamicTaskScheduler {
    
    private final BlockingQueue<Task> taskQueue = new LinkedBlockingQueue<>();
    private final AtomicInteger pendingTasks = new AtomicInteger(0);
    
    // Worker 主动拉取任务
    public Task fetchTask() {
        // 先尝试从队列获取
        Task task = taskQueue.poll();
        if (task != null) {
            pendingTasks.decrementAndGet();
            return task;
        }
        
        // 队列为空，等待新任务
        synchronized (this) {
            try {
                wait(1000); // 等待 1 秒
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        }
        return taskQueue.poll();
    }
    
    // Master 添加新任务
    public void addTask(Task task) {
        taskQueue.offer(task);
        pendingTasks.incrementAndGet();
        synchronized (this) {
            notifyAll(); // 唤醒等待的 Worker
        }
    }
}
```

### 3.4 按时间窗口分片

适用于时间序列数据，如日志处理、事件流处理。

```java
public class TimeWindowPartitioner {
    
    public List<TimeWindowTask> partitionByTimeWindow(
            Instant startTime, 
            Instant endTime, 
            Duration windowSize) {
        List<TimeWindowTask> tasks = new ArrayList<>();
        
        Instant current = startTime;
        while (current.isBefore(endTime)) {
            Instant windowEnd = current.plus(windowSize);
            if (windowEnd.isAfter(endTime)) {
                windowEnd = endTime;
            }
            
            tasks.add(new TimeWindowTask(current, windowEnd));
            current = windowEnd;
        }
        return tasks;
    }
}
```

**按时间分片的应用场景**：
- 每天凌晨处理前一天的日志数据
- 按小时聚合统计数据
- 清理 N 天前的历史数据

## 四、并行度控制：太快也是一种罪

批量处理不是越快越好。无限增加并行度，可能压垮下游数据库、触发限流、造成内存溢出。

### 4.1 线程池大小配置

**CPU 密集型任务**：线程数 `=` CPU 核心数 + 1

```java
int cpuCores = Runtime.getRuntime().availableProcessors();
ExecutorService cpuBoundPool = new ThreadPoolExecutor(
    cpuCores,           // 核心线程数
    cpuCores,           // 最大线程数
    0L, TimeUnit.MILLISECONDS,
    new LinkedBlockingQueue<>(),
    new ThreadPoolExecutor.CallerRunsPolicy()
);
```

**IO 密集型任务**：线程数 `=` CPU 核心数 × 期望 CPU 利用率 × (1 + IO 时间/CPU 时间)

```java
// 假设 IO 密集型，期望 CPU 利用率 50%，IO 时间是 CPU 时间的 4 倍
int cpuCores = Runtime.getRuntime().availableProcessors();
int ioThreads = (int) (cpuCores * 0.5 * 5); // = 核心数 * 2.5

ExecutorService ioBoundPool = new ThreadPoolExecutor(
    ioThreads,
    ioThreads * 2,
    60L, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(1000)
);
```

> 实际项目中，IO 密集型的线程数通常设置为 **CPU 核心数的 2-4 倍**。

### 4.2 数据库连接池限制

批量处理最容易踩的坑：**数据库连接池被撑爆**。

```yaml
# application.yml
spring:
  datasource:
    hikari:
      maximum-pool-size: 20        # 连接池最大连接数
      minimum-idle: 5              # 最小空闲连接
      connection-timeout: 30000    # 获取连接超时 30s
      idle-timeout: 600000         # 空闲连接存活时间 10min
      max-lifetime: 1800000        # 连接最大生命周期 30min
```

**核心原则**：批量处理线程数 × 每线程数据库连接数 `<=` 连接池最大连接数

```java
// 连接池配置计算器
public class PoolSizeCalculator {
    
    public int calculateOptimalPoolSize(
            int batchThreads, 
            int connectionsPerThread) {
        
        int required = batchThreads * connectionsPerThread;
        
        // 留 20% 余量给其他业务使用
        int recommended = (int) (required * 1.2);
        
        // 确保不超过数据库最大连接数
        return Math.min(recommended, MAX_DB_CONNECTIONS);
    }
}
```

### 4.3 批量大小调优

每批处理多少条数据，是一个需要仔细权衡的问题。

| 批量大小 | 优点 | 缺点 |
|---|---|---|
| 太小（100 条） | 内存占用低，失败重试代价小 | 数据库 IO 次数多，网络往返频繁 |
| 中等（1000-5000 条） | 平衡点，推荐使用 | - |
| 太大（100000 条） | IO 次数少 | 内存占用高，失败重试代价大 |

```java
public class BatchSizeOptimizer {
    
    // 推荐使用 1000-5000 条
    private static final int DEFAULT_BATCH_SIZE = 2000;
    
    // 根据内存动态调整
    public int calculateBatchSize(Runtime runtime) {
        long freeMemory = runtime.freeMemory();
        long totalMemory = runtime.totalMemory();
        
        // 内存使用超过 70%，减小批量大小
        if (totalMemory > 0 && freeMemory < totalMemory * 0.3) {
            return DEFAULT_BATCH_SIZE / 2;
        }
        
        // 内存充足，可以增大批量
        return DEFAULT_BATCH_SIZE * 2;
    }
}
```

### 4.4 限流策略：保护下游服务

批量处理不能无限制地向下游服务发送请求，需要限流保护。

```java
public class RateLimiterBatchProcessor {
    
    private final RateLimiter rateLimiter;
    private final int batchSize;
    
    public RateLimiterBatchProcessor(int qps) {
        this.rateLimiter = RateLimiter.create(qps);
        this.batchSize = 100;
    }
    
    public void processBatch(List<Record> records) {
        for (int i = 0; i < records.size(); i += batchSize) {
            // 限流：每秒最多 N 个请求
            rateLimiter.acquire();
            
            List<Record> batch = records.subList(
                i, 
                Math.min(i + batchSize, records.size())
            );
            
            process(batch);
        }
    }
}
```

## 五、失败处理与重试机制

### 5.1 重试策略

**固定间隔重试**：每次失败后等待固定时间再重试

```java
public class FixedDelayRetryPolicy {
    
    private final int maxRetries;
    private final long delayMs;
    
    public FixedDelayRetryPolicy(int maxRetries, long delayMs) {
        this.maxRetries = maxRetries;
        this.delayMs = delayMs;
    }
    
    public <T> T execute(Callable<T> operation) throws Exception {
        Exception lastException = null;
        
        for (int attempt = 0; attempt <= maxRetries; attempt++) {
            try {
                return operation.call();
            } catch (Exception e) {
                lastException = e;
                
                if (attempt < maxRetries) {
                    Thread.sleep(delayMs);
                }
            }
        }
        
        throw lastException;
    }
}
```

**指数退避重试**：失败越多，等待时间越长

```java
public class ExponentialBackoffRetryPolicy {
    
    private final int maxRetries;
    private final long baseDelayMs;
    private final double multiplier;
    private final long maxDelayMs;
    
    public ExponentialBackoffRetryPolicy() {
        this(5, 1000, 2.0, 30000);
    }
    
    public <T> T execute(Callable<T> operation) throws Exception {
        Exception lastException = null;
        
        for (int attempt = 0; attempt <= maxRetries; attempt++) {
            try {
                return operation.call();
            } catch (Exception e) {
                lastException = e;
                
                if (attempt < maxRetries) {
                    long delay = calculateDelay(attempt);
                    Thread.sleep(delay);
                }
            }
        }
        
        throw lastException;
    }
    
    private long calculateDelay(int attempt) {
        long delay = (long) (baseDelayMs * Math.pow(multiplier, attempt));
        return Math.min(delay, maxDelayMs);
    }
}
```

> [!TIP]
> 指数退避的重试间隔通常是：1s → 2s → 4s → 8s → 16s，最大不超过 30s。

### 5.2 幂等性保证

重复执行不产生副作用，是分布式批量处理的核心要求。

```java
public class IdempotentProcessor {
    
    @Autowired
    private RedisTemplate<String, Object> redisTemplate;
    
    public void processWithIdempotency(Record record) {
        String idempotentKey = "batch:processed:" + record.getId();
        
        // 1. 检查是否已处理
        Boolean exists = redisTemplate.hasKey(idempotentKey);
        if (Boolean.TRUE.equals(exists)) {
            log.info("Record {} already processed, skip", record.getId());
            return;
        }
        
        // 2. 设置处理中标记（防止并发重复处理）
        Boolean acquired = redisTemplate.opsForValue()
            .setIfAbsent(idempotentKey + ":lock", "1", 10, TimeUnit.SECONDS);
        
        if (!Boolean.TRUE.equals(acquired)) {
            log.warn("Record {} is being processed by another thread", record.getId());
            return;
        }
        
        try {
            // 3. 业务处理
            doProcess(record);
            
            // 4. 标记处理完成
            redisTemplate.opsForValue().set(
                idempotentKey, 
                "done", 
                7, TimeUnit.DAYS
            );
        } finally {
            // 5. 释放锁
            redisTemplate.delete(idempotentKey + ":lock");
        }
    }
}
```

### 5.3 死信队列：超过重试次数的消息处理

对于反复失败的任务，移到死信队列（DLQ）统一处理。

```java
@Configuration
public class DeadLetterQueueConfig {
    
    @Bean
    public DirectExchange deadLetterExchange() {
        return new DirectExchange("batch.dlx.exchange");
    }
    
    @Bean
    public Queue deadLetterQueue() {
        return QueueBuilder.durable("batch.dlx.queue").build();
    }
    
    @Bean
    public Binding deadLetterBinding() {
        return BindingBuilder
            .bind(deadLetterQueue())
            .to(deadLetterExchange())
            .with("batch.dead");
    }
}

@Component
public class BatchMessageHandler {
    
    @Autowired
    private RabbitTemplate rabbitTemplate;
    
    @RabbitListener(queues = "batch.task.queue")
    public void processBatch(BatchTaskMessage message, 
                            @Header(AmqpHeaders.REDELIVERED) boolean redelivered) {
        try {
            doProcess(message);
        } catch (Exception e) {
            if (redelivered) {
                // 已重试过，移到死信队列
                sendToDeadLetterQueue(message, e);
            } else {
                throw e; // 触发重试
            }
        }
    }
    
    private void sendToDeadLetterQueue(BatchTaskMessage message, Exception e) {
        rabbitTemplate.convertAndSend(
            "batch.dlx.exchange",
            "batch.dead",
            message,
            msg -> {
                msg.getMessageProperties().setHeader("error", e.getMessage());
                msg.getMessageProperties().setHeader("failedAt", 
                    System.currentTimeMillis());
                return msg;
            }
        );
    }
}
```

### 5.4 断点续传：记录处理进度

```java
public class CheckpointManager {
    
    private static final String CHECKPOINT_KEY = "batch:checkpoint:";
    
    @Autowired
    private RedisTemplate<String, Object> redisTemplate;
    
    // 保存进度
    public void saveCheckpoint(String jobId, long processedId) {
        String key = CHECKPOINT_KEY + jobId;
        redisTemplate.opsForValue().set(key, processedId);
    }
    
    // 获取进度
    public long getCheckpoint(String jobId) {
        String key = CHECKPOINT_KEY + jobId;
        Object value = redisTemplate.opsForValue().get(key);
        return value == null ? 0 : (Long) value;
    }
    
    // 清除进度
    public void clearCheckpoint(String jobId) {
        String key = CHECKPOINT_KEY + jobId;
        redisTemplate.delete(key);
    }
}

// 使用断点续传
public class BatchProcessorWithCheckpoint {
    
    private final CheckpointManager checkpointManager;
    
    public void processWithCheckpoint(String jobId, long startId, long endId) {
        long checkpoint = checkpointManager.getCheckpoint(jobId);
        long start = Math.max(checkpoint + 1, startId);
        
        for (long id = start; id <= endId; id += BATCH_SIZE) {
            long batchEnd = Math.min(id + BATCH_SIZE - 1, endId);
            
            processBatch(id, batchEnd);
            
            // 保存进度
            checkpointManager.saveCheckpoint(jobId, batchEnd);
        }
        
        // 处理完成，清除 checkpoint
        checkpointManager.clearCheckpoint(jobId);
    }
}
```

```java
public class ProgressReporter {

    private final AtomicLong failedCount = new AtomicLong(0);

                 "Speed: {:.0f}/s | ETA: {}s",
            progress, processed, failed, speed, remaining / 1000);
    }

        processedCount.incrementAndGet();
    }
    
    public void incrementFailed() {
        failedCount.incrementAndGet();
    }
}
```

```java
            }
        }

    }
```

### 7.1 亿级数据迁移

**场景**：将 MySQL 单表数据迁移到分库分表架构，总数据量 1 亿条。

**方案**：

1. **分表策略**：按用户 ID 哈希分到 16 个分片
2. **分批处理**：每批 5000 条，避免长事务
3. **断点续传**：记录已迁移的最后一条 ID
4. **双写**：新数据同时写入新旧库，验证一致性后切换

```java
public class DataMigrationJob {
    
    private static final int BATCH_SIZE = 5000;
    private static final int THREAD_COUNT = 8;
    
    public void migrate() {
        long maxId = getMaxId();
        CheckpointManager checkpoint = new CheckpointManager();
        long startId = checkpoint.getCheckpoint("user_migration");
        
        ExecutorService executor = Executors.newFixedThreadPool(THREAD_COUNT);
        
        for (long id = startId; id <= maxId; id += BATCH_SIZE * THREAD_COUNT) {
            List<Future<?>> futures = new ArrayList<>();
            
            for (int i = 0; i < THREAD_COUNT; i++) {
                final long batchStart = id + i * BATCH_SIZE;
                final long batchEnd = batchStart + BATCH_SIZE;
                
                futures.add(executor.submit(() -> {
                    migrateBatch(batchStart, batchEnd);
                }));
            }
            
            // 等待本轮完成
            for (Future<?> future : futures) {
                try {
                    future.get(5, TimeUnit.MINUTES);
                } catch (Exception e) {
                    log.error("Batch migration failed", e);
                    handleFailure(e);
                }
            }
            
            // 保存 checkpoint
            checkpoint.saveCheckpoint("user_migration", batchEnd);
        }
    }
}
```

## 八、面试实战

### 8.1 场景一：设计定时同步任务

**面试官问**：「如何设计一个每天定时同步数据的任务？」

**回答要点**：

> 定时同步任务的核心挑战有三个：时效性、一致性、性能。

**第一，数据同步的时机选择。**
凌晨业务低峰期同步是最稳妥的选择。但要注意：如果数据量特别大，可以采用增量同步 + 定时全量补齐的策略。比如白天增量同步，晚上全量校验。

**第二，一致性保证。**
同步过程中可能出现数据冲突。我的方案是：记录同步时间戳，优先使用更新时间更近的数据。如果需要强一致性，可以采用双写 + 消息队列的方案。

**第三，性能与隔离。**
同步任务不能影响线上服务。我会使用独立的数据库连接池，设置合理的批量大小（比如 2000-5000 条），同时限制并发线程数。

```java
@Scheduled(cron = "0 0 2 * * ?") // 凌晨 2 点
public void scheduledSync() {
    // 1. 获取增量数据（上次同步时间到现在的变更）
    long lastSyncTime = getLastSyncTime();
    long currentTime = System.currentTimeMillis();
    
    // 2. 分批处理，避免内存溢出
    List<Record> batch;
    while (!(batch = getIncrementalData(lastSyncTime, currentTime)).isEmpty()) {
        processBatch(batch);
        saveProgress(currentTime);
    }
    
    // 3. 发送同步完成通知
    notifySyncComplete();
}
```

### 8.2 场景二：节点故障与数据一致性

**面试官问**：「批量处理过程中出现节点故障，如何保证数据一致性？」

**回答要点**：

> 节点故障是分布式批处理必须面对的问题，核心是「幂等性 + 事务 + 补偿」三板斧。

**幂等性是第一道防线。**
所有处理逻辑必须支持幂等。哪怕同一个任务被执行两次，也不会产生副作用。我的实现是：每条记录处理前先检查是否已处理（通过 Redis 记录），处理成功后再标记。

**事务边界要划清楚。**
如果批量处理涉及多表更新，需要明确事务边界。我通常的做法是：小事务 + 补偿机制。每个小批次是一个事务，失败就回滚并重试，而不是整个任务失败。

**补偿机制处理不可逆故障。**
如果重试超过次数还没成功，移到死信队列，后续人工介入。同时记录详细的错误日志，方便定位问题。

```java
public void processWithFaultTolerance(List<Record> batch) {
    try {
        // 开启事务
        transactionTemplate.execute(status -> {
            for (Record record : batch) {
                // 幂等检查
                if (isProcessed(record.getId())) {
                    continue;
                }
                
                // 业务处理
                doProcess(record);
                
                // 标记已处理
                markAsProcessed(record.getId());
            }
            return null;
        });
    } catch (Exception e) {
        // 重试机制
        retryWithBackoff(() -> {
            processWithFaultTolerance(batch);
        });
        
        // 超过重试次数，进入死信队列
        sendToDeadLetterQueue(batch, e);
    }
}
```

### 8.3 场景三：并发度控制

**面试官问**：「如何控制分布式任务的并发度，避免压垮下游？」

**回答要点**：

> 并发度控制的核心是「感知 + 限流 + 熔断」三层防护。

**第一层：感知下游容量。**
不是简单地设置固定线程数，而是根据下游的响应情况动态调整。比如记录下游的平均响应时间，如果响应变慢，就降低并发。

**第二层：限流保护。**
使用令牌桶或漏桶算法，限制每秒向下游发送的请求数。这个限流值通常根据压测结果来设定，目标是让下游服务保持在 70-80% 的利用率。

**第三层：熔断兜底。**
如果下游错误率突然升高，说明服务可能出现问题了。这时候要快速失败，而不是继续发送请求。熔断器会在错误率恢复正常后自动恢复。

```java
public class AdaptiveConcurrencyController {
    
    private final CircuitBreaker circuitBreaker;
    private final RateLimiter rateLimiter;
    
    public void process(Record record) {
        // 1. 检查熔断器状态
        if (circuitBreaker.isOpen()) {
            throw new CircuitOpenException("Circuit breaker is open");
        }
        
        // 2. 限流
        rateLimiter.acquire();
        
        try {
            // 3. 调用下游
            callDownstream(record);
            
            // 4. 记录成功
            circuitBreaker.recordSuccess();
        } catch (Exception e) {
            // 5. 记录失败
            circuitBreaker.recordFailure(e);
            throw e;
        }
    }
}
```

## 总结

分布式批量任务优化，本质上是在解决三个核心问题：**如何拆分、如何执行、如何容错**。

- **分片策略**决定并行度的上限。ID 范围分片简单但可能倾斜，动态分片更灵活但实现复杂。
- **并行度控制**决定系统稳定性。不是越快越好，要给下游留有余地。
- **失败处理**决定数据安全性。幂等性、重试机制、断点续传缺一不可。

记住一个原则：**批量处理不是一次性工程，而是需要持续监控和迭代的系统**。上线只是开始，监控和优化才是日常。
