# CDN 加速与静态资源缓存

打开一个网页，为什么有些图片加载特别快，有些却要等好几秒？为什么同一张图片在北京和广州的访问速度可能相差巨大？

答案往往在于 CDN（Content Delivery Network，内容分发网络）。

CDN 是互联网基础设施的重要组成部分，它让用户能够「就近」获取内容，大幅提升访问速度。全球排名前 1000 的网站，几乎全部使用了 CDN。如果你的网站还没有接入 CDN，那已经输在了起跑线上。

这篇文章的核心任务，是帮你理解 CDN 的工作原理、掌握 CDN 的使用技巧，以及在实际项目中如何配置和优化 CDN。

---

## 一、CDN 的工作原理

### 1.1 没有 CDN 时的问题

在没有 CDN 的时代，用户访问一个网站的内容，请求路径是这样的：

```
用户（假设在上海）→ 源站服务器（假设在杭州）→ 返回内容
```

问题显而易见：

- **距离远**：跨越半个中国，单程延迟可能就有 50-100ms
- **带宽瓶颈**：所有用户都从源站获取内容，源站带宽压力大
- **稳定性差**：源站挂了，所有用户都访问不了

### 1.2 CDN 的核心思想：就近访问

CDN 的核心思想很简单：**把内容缓存到离用户更近的地方，让用户从最近的节点获取内容。**

```
用户（假设在上海）
  ↓ 请求
CDN 边缘节点（假设在上海 or 华东）
  ↓ 如果缓存命中，直接返回
  ↓ 如果缓存未命中，回源获取
源站服务器（假设在杭州）
```

### 1.3 CDN 的访问流程

一次完整的 CDN 访问流程如下：

```
1. 用户访问 example.com/logo.png

2. DNS 解析
   └─ DNS 服务器返回 CDN 智能调度的边缘节点 IP
   └─ 调度策略：地理位置、网络延迟、负载均衡

3. 用户向边缘节点发起请求

4. 边缘节点检查缓存
   ├─ 命中：直接返回缓存内容
   └─ 未命中：回源获取内容，缓存后返回

5. 返回响应给用户
```

### 1.4 CDN 的三大组件

| 组件 | 功能 | 说明 |
|-----|------|------|
| 边缘节点 | 缓存内容，提供访问 | 部署在全国/全球各处 |
| 源站 | 存储原始内容 | 实际的业务服务器 |
| 调度系统 | 智能分配节点 | DNS、智能解析 |

## 二、CDN 缓存策略

### 2.1 缓存的核心机制

CDN 的核心是缓存。当用户首次访问某个资源时，CDN 边缘节点会从源站拉取内容并缓存起来。之后，同一地区的用户访问相同资源时，直接从边缘节点返回，无需再次回源。

缓存的关键是 **HTTP 缓存控制头**：

| 头部 | 作用 |
|-----|------|
| `Cache-Control` | 控制缓存行为（public/private/no-cache/max-age） |
| `Expires` | 缓存过期时间（绝对时间） |
| `ETag` | 资源版本标识，用于缓存校验 |
| `Last-Modified` | 资源最后修改时间 |

### 2.2 Cache-Control 详解

`Cache-Control` 是最常用的缓存控制头，支持多种指令：

| 指令 | 说明 | 示例 |
|-----|------|------|
| `max-age` | 缓存有效时间（秒） | `max-age=3600`（缓存 1 小时） |
| `public` | 允许 CDN 和浏览器缓存 | 默认即为 public |
| `private` | 只允许浏览器缓存 | 敏感数据，如用户个人信息 |
| `no-cache` | 每次使用前必须验证 | 缓存但需验证 |
| `no-store` | 不缓存 | 敏感数据，如密码 |
| `s-maxage` | CDN 缓存时间（覆盖 max-age） | `s-maxage=86400` |

```bash
# 示例：图片缓存 1 天
Cache-Control: public, max-age=86400

# 示例：HTML 不缓存，必须验证
Cache-Control: no-cache, no-store

# 示例：浏览器缓存 1 小时，CDN 缓存 1 天
Cache-Control: public, max-age=3600, s-maxage=86400
```

### 2.3 缓存校验：ETag vs Last-Modified

当缓存过期后，浏览器需要验证缓存是否还有效。两种验证方式：

**ETag 方式**：

```bash
# 首次请求，服务器返回
HTTP/1.1 200 OK
ETag: "abc123"

# 缓存过期后，浏览器发送
GET /logo.png
If-None-Match: "abc123"

# 服务器检查 ETag，如果没变返回 304 Not Modified
HTTP/1.1 304 Not Modified
# 不返回 Body，节省带宽
```

**Last-Modified 方式**：

```bash
# 首次请求，服务器返回
HTTP/1.1 200 OK
Last-Modified: Tue, 01 Mar 2026 10:00:00 GMT

# 缓存过期后，浏览器发送
GET /logo.png
If-Modified-Since: Tue, 01 Mar 2026 10:00:00 GMT

# 服务器检查时间，如果没变返回 304
HTTP/1.1 304 Not Modified
```

**ETag vs Last-Modified**：

| 特性 | ETag | Last-Modified |
|-----|------|---------------|
| 精度 | 高（基于内容 hash） | 低（秒级） |
| 性能 | 需要计算 hash | 直接比较时间 |
| 分布式问题 | 同一文件不同节点 ETag 可能不同 | 没问题 |

:::tip 推荐实践

在 CDN 场景下，**Last-Modified 更适合**，因为 CDN 边缘节点遍布全球，ETag 在不同节点可能不一致，而 Last-Modified 没有这个问题。

但如果对精度要求高，且能保证源站一致性，可以考虑 ETag。
:::

### 2.4 CDN 缓存时间设置建议

不同类型的内容，缓存时间差异很大：

| 内容类型 | 推荐缓存时间 | 说明 |
|---------|------------|------|
| HTML | 不缓存 or 几分钟 | 变化频繁 |
| CSS/JS（带版本号） | 1 年 | 版本化后可以长期缓存 |
| 图片（logo、图标） | 1 年 | 几乎不变 |
| 图片（用户上传） | 7 天 | 可能删除 |
| 视频/音频 | 1 年 | 静态资源 |
| API 响应 | 不缓存 | 动态内容 |

## 三、CDN 适用场景

### 3.1 适合 CDN 的场景

**静态资源**：这是 CDN 最经典的使用场景。

- 图片：商品图、用户头像、广告图
- 样式文件：CSS
- 脚本文件：JavaScript
- 字体文件：woff2、ttf
- 视频/音频：宣传视频、课程视频

**大文件下载**：软件安装包、ROM 更新包。

**流媒体**：点播视频、直播回放。

### 3.2 不适合 CDN 的场景

**动态接口**：实时性要求高的接口不适合 CDN。

- 用户个人信息接口
- 搜索接口
- 下单/支付接口
- 实时数据接口

**个性化内容**：每个用户看到的内容不同。

- 推荐列表
- 购物车内容
- 消息通知

**高敏感性内容**：涉及隐私或安全的数据。

- 登录凭证
- 金融交易数据
- 个人隐私信息

### 3.3 边缘计算场景

现代 CDN（如 Cloudflare Workers、阿里云 CDN）支持边缘计算，可以：

- 在 CDN 节点进行简单的数据处理
- 实现 A/B 测试
- 做简单的认证和鉴权
- 返回个性化内容（基于地理位置）

## 四、缓存命中率优化

### 4.1 缓存命中率的重要性

CDN 的核心价值在于缓存命中率。命中率越高，说明越多请求被边缘节点处理，无需回源。

```
缓存命中率 = 缓存命中请求数 / 总请求数
```

| 命中率 | 效果 |
|-------|------|
| `>= 95%` | 优秀，绝大部分请求被边缘节点处理 |
| `80% - 95%` | 良好，少量回源 |
| `60% - 80%` | 一般，需要优化 |
| `< 60%` | 较差，大量回源，性能差 |

### 4.2 影响命中率的因素

**命中率低的原因**：

- 缓存时间设置太短
- URL 带随机参数（如 `?t=1234567890`）
- 大量用户访问不同内容（如个性化首页）
- 缓存 key 不统一

**优化策略**：

**1. 合理设置缓存时间**

```bash
# Nginx 配置：静态资源缓存 1 年
location ~* \.(css|js|jpg|jpeg|png|gif|ico|woff2)$ {
    expires 1y;
    add_header Cache-Control "public, max-age=31536000";
}
```

**2. 版本化文件名**

```html
<!-- 使用 hash 版本号，文件变化时 URL 变化 -->
<script src="/js/app.abc123.js"></script>

<!-- 而不是这样 -->
<script src="/js/app.js?v=1.0"></script>
```

**3. 按目录设置不同的缓存策略**

```nginx
# 静态资源：长缓存
location /static/ {
    expires 1y;
    add_header Cache-Control "public, max-age=31536000";
}

# 用户上传：短缓存
location /uploads/ {
    expires 7d;
    add_header Cache-Control "public, max-age=604800";
}

# HTML：不缓存
location / {
    add_header Cache-Control "no-cache, no-store";
}
```

### 4.3 缓存预热

缓存预热是指在业务高峰之前，主动将热点内容推送到 CDN 边缘节点，避免缓存未命中时的回源压力。

```bash
# 阿里云 CDN 预热
aliyun cdn PushObjectCdn --object-urls "https://example.com/logo.png,https://example.com/banner.jpg"

# Cloudflare CDN 缓存预热
curl -X POST "https://api.cloudflare.com/client/v4/zones/{zone_id}/purge_cache" \
    -H "Authorization: Bearer {token}" \
    -H "Content-Type: application/json" \
    --data '{"files":["https://example.com/logo.png"]}'
```

**适合预热的场景**：

- 促销活动开始前（秒杀商品图）
- 版本发布前（静态资源）
- 重大活动前（热点内容）

## 五、CDN 与前端缓存的配合

### 5.1 多级缓存架构

一个完善的缓存体系应该是多级的：

```
用户浏览器 → CDN 边缘节点 → CDN 源站 → 业务源站
```

| 层级 | 缓存位置 | 时效 | 说明 |
|-----|---------|------|------|
| 一级 | 浏览器 | 可控 | 用户本地，最快 |
| 二级 | CDN 边缘 | 中等 | 就近访问 |
| 三级 | CDN 源站 | 较长 | 回源点 |
| 四级 | 源站 | 无 | 原始数据 |

### 5.2 浏览器缓存设置

浏览器缓存是第一道缓存，设置合理的缓存策略：

```nginx
# Nginx 配置：浏览器缓存
server {
    # HTML：no-cache，强制验证
    location = /index.html {
        add_header Cache-Control "no-cache, no-store";
    }

    # CSS/JS：长缓存
    location ~* \.(css|js)$ {
        expires 1y;
        add_header Cache-Control "public, max-age=31536000, immutable";
    }

    # 图片：长缓存
    location ~* \.(jpg|jpeg|png|gif|ico|svg)$ {
        expires 6M;
        add_header Cache-Control "public, max-age=15552000";
    }
}
```

### 5.3 缓存失效策略

当源站内容更新时，需要让 CDN 和浏览器的缓存失效。

**方式一：URL 版本化**

```html
<!-- 文件更新后，修改 URL 版本号 -->
<script src="/js/app.js?v=1.1"></script>
```

**方式二：CDN 缓存刷新**

```bash
# 刷新单个文件
cdn-cli purge https://example.com/static/app.js

# 刷新目录
cdn-cli purge https://example.com/static/

# 批量刷新
cdn-cli purge --file-list urls.txt
```

**方式三：CDN 缓存标签**

```bash
# 给特定标签的所有资源刷新
cdn-cli purge --tag "homepage"
```

## 六、CDN 回源优化

### 6.1 回源的性能问题

当 CDN 缓存未命中时，需要回源获取内容。如果回源量大或回源慢，会影响用户体验。

**回源优化策略**：

**1. 降低回源率**

- 提高缓存命中率
- 合理设置缓存时间
- 使用多级缓存

**2. 优化回源链路**

- 源站和 CDN 同地域部署
- 使用私有协议回源（如 H2 回源）
- 开启回源压缩（回源时启用 gzip）

**3. 回源限速**

```nginx
# 限制单 IP 回源并发
limit_req_zone $binary_remote_addr zone=backorigin:10m rate=10r/s;
location / {
    limit_req zone=backorigin burst=20;
}
```

### 6.2 缓存失效的「惊群效应」

当大量缓存同时失效时，大量请求会同时回源，可能压垮源站。这叫「惊群效应」。

**解决方案**：

- **缓存时间随机化**：设置过期时间时加上随机值

```nginx
# 过期时间 = base + random(0, 3600)
map $request_uri $expire_time {
    ~* \.(css|js)$ 86400;  # 基础 1 天
    ~* \.(jpg|png)$ 432000; # 基础 5 天
    default 3600;
}

# 在过期时间上加随机偏移
set_by_lua_block $expire_time_with_random {
    local expire = tonumber(ngx.var.expire_time)
    local random = math.random(0, 3600)
    return expire + random
}
add_header Cache-Control "public, max-age=$expire_time_with_random";
```

- **主动刷新**：提前刷新即将过期的内容
- **逻辑过期**：返回旧内容同时异步更新

## 七、生产环境配置示例

### 7.1 Nginx 源站配置

```nginx
upstream cdn_origin {
    server 127.0.0.1:8080;
    keepalive 32;
}

server {
    listen 80;
    server_name example.com;

    # 日志格式，添加 CDN 来源标识
    log_format cdn '$remote_addr - $http_x_forwarded_for [$time_local] '
                    '"$request" $status $body_bytes_sent '
                    '"$http_referer" "$http_user_agent" cdn_hit=$upstream_cache_status';

    access_log /var/log/nginx/cdn_access.log cdn;

    # 静态资源：长缓存 + gzip
    location ~* \.(css|js|jpg|jpeg|png|gif|ico|svg|woff2|ttf)$ {
        root /var/www/static;
        expires 1y;
        add_header Cache-Control "public, max-age=31536000";
        add_header ETag "";

        gzip on;
        gzip_types text/css application/javascript image/* font/*;
        gzip_min_length 1k;
        gzip_comp_level 5;
    }

    # API 代理
    location /api/ {
        proxy_pass http://cdn_origin;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;

        # 不缓存
        add_header Cache-Control "no-cache, no-store";
    }

    # 跨域设置（如果需要）
    location ~* \.(woff2|ttf|woff)$ {
        add_header Access-Control-Allow-Origin *;
    }
}
```

## 八、面试实战

### 8.1 场景一：CDN 工作原理

**面试官问**：「请描述一下 CDN 的工作原理？」

**标准回答**：

> CDN（内容分发网络）的核心思想是「就近访问」。当用户请求一个资源时：
>
> 1. DNS 解析返回离用户最近的 CDN 边缘节点 IP
> 2. 用户请求边缘节点
> 3. 边缘节点检查本地缓存：
>    - 命中：直接返回缓存内容
>    - 未命中：回源获取内容，缓存后返回
>
> CDN 的价值在于：减少用户访问延迟、降低源站带宽压力、提升访问稳定性。

### 8.2 场景二：缓存策略设计

**面试官问**：「如果让你设计一个网站的缓存策略，你会怎么做？」

**回答要点**：

> 根据资源类型，我会设计不同的缓存策略：
>
> | 资源类型 | 缓存策略 |
> |---------|---------|
> | HTML | no-cache，每次验证 |
> | CSS/JS（带 hash） | 缓存 1 年 |
> | 图片 | 缓存 6 个月 |
> | API 响应 | 不缓存 |
>
> 关键点：
> 1. 静态资源使用版本化 URL，变更时 URL 变化
> 2. 使用 Last-Modified 进行缓存验证
> 3. 配置合适的 Cache-Control 头
> 4. 大促前进行缓存预热

### 8.3 场景三：CDN 缓存失效问题

**面试官问**：「当源站内容更新后，CDN 上的缓存如何处理？」

**回答要点**：

> 有三种常见方式：
>
> 1. **URL 版本化**：更新文件时修改文件名（如加 hash），用户访问新的 URL，CDN 会缓存新版本。这是推荐做法。
>
> 2. **主动刷新**：通过 CDN 控制台或 API，主动刷新指定 URL 或目录，让 CDN 重新回源获取最新内容。
>
> 3. **设置较短的缓存时间**：适合更新频繁的内容，但会影响 CDN 命中率。
>
> 我推荐 URL 版本化 + 主动刷新结合使用。

---

## 总结

CDN 是提升网站访问速度的利器，核心要点：

1. **就近访问**：CDN 将内容缓存到离用户最近的地方
2. **缓存控制**：通过 Cache-Control、ETag、Last-Modified 控制缓存行为
3. **适用场景**：静态资源（图片/CSS/JS/字体）最适合 CDN
4. **命中率优化**：合理设置缓存时间、版本化文件名、按目录设置策略
5. **回源优化**：避免惊群效应，做好缓存预热

正确使用 CDN，可以让用户访问速度提升 50% 以上，同时大幅降低源站带宽成本。
