# 接口性能优化 Checklist

## 优化之前，先过一遍这个清单

接口是用户直接感知的层面。一个接口快不快，直接影响用户体验和业务转化率。

但接口优化最怕什么？最怕的不是不知道怎么优化，而是优化了半天，效果不明显甚至适得其反。

一个常见场景：团队花了两周优化了接口的 JSON 序列化，用 Kryo 替换了 Jackson，以为能大幅提升性能。结果上线后发现 RT 只降了 5ms，而接口真正的瓶颈在于数据库查询耗时 200ms。优化方向错了，努力白费。

怎么避免这种「优化了个寂寞」的情况？答案是：**在动手之前，先用 Checklist 评估现状，找准优化方向。**

这篇 Checklist 不是告诉你「怎么优化每个细节」，而是告诉你「优化之前要看什么」。它把接口性能优化的关键检查点整理成清单，帮你系统性地审视接口，避免在错误的方向上浪费精力。

---

## 一、序列化与数据传输

### 1.1 序列化方案检查

序列化是接口数据传输的基础，选择合适的序列化方案对性能影响很大。

**序列化方案对比**：

| 方案 | 特点 | 适用场景 |
|-----|------|---------|
| JSON | 可读性好，兼容性最强 | 对外 API、日志 |
| Protobuf | 二进制，高性能，需预定义 Schema | 内部微服务、大数据量 |
| Kryo | Java 专用，高性能 | Java 内部通信 |
| Hessian | 二进制，跨语言 | 异构系统 |

**检查项**：

- [ ] 当前使用的序列化方案是什么？是否符合业务场景？
- [ ] 内部微服务通信是否还在用 JSON？可以考虑 Protobuf 或 Kryo。
- [ ] 序列化后的数据体积是否过大？（可以用 Wireshark 抓包看）
- [ ] 是否需要考虑跨语言兼容性？

**优化建议**：

- 对内接口（大流量、低延迟）：优先考虑 Protobuf 或 Kryo
- 对外接口（需要可读性）：继续用 JSON
- 大数据量传输：考虑 Protobuf + 压缩

### 1.2 数据传输体积检查

很多接口慢，不是因为处理慢，而是因为传输的数据太多。

**常见问题**：

- [ ] SELECT * 导致返回了大量不需要的字段
- [ ] 嵌套层级过深（如返回整个用户对象 + 订单列表 + 商品详情）
- [ ] 分页接口每页返回数据量过大

**优化建议**：

```sql
-- 问题：SELECT * 会返回所有字段，包括大文本字段
SELECT * FROM orders WHERE user_id = ?

-- 正确：只查需要的字段
SELECT order_id, amount, status, create_time
FROM orders WHERE user_id = ?
```

```java
// 问题：返回完整的嵌套对象
Order order = orderService.getOrder(orderId);
// order 里包含 user、items、payment 等完整对象，层级深、体积大

// 正确：使用 DTO 只返回需要的字段
OrderDTO dto = orderService.getOrderSimple(orderId);
// 只包含前端需要的字段
```

## 二、幂等性检查

幂等性是接口设计的基石，特别是在涉及写操作时。

### 2.1 幂等性风险识别

**高风险场景**：

- [ ] POST 请求（创建订单、支付）
- [ ] DELETE 请求（删除数据）
- [ ] PUT 请求（更新数据）
- [ ] 涉及资金、库存等敏感操作

**问题排查**：

- [ ] 接口是否考虑了重复调用的情况？
- [ ] 网络超时重试时是否会重复执行？
- [ ] 前端按钮是否做了防重复点击处理？

### 2.2 幂等性实现方案

根据业务场景选择合适的幂等性方案：

| 方案 | 适用场景 | 实现复杂度 |
|-----|---------|-----------|
| 唯一 Token | 表单提交、按钮操作 | 中 |
| 乐观锁（version） | 数据更新 | 低 |
| 悲观锁（SELECT FOR UPDATE） | 强一致性场景 | 中 |
| 分布式锁 | 分布式环境 | 高 |
| 状态机 | 有明确状态流转的操作 | 低 |

> 详见 [接口幂等性实现方案](/high-throughput/high-performance/api/api-idempotent)。

## 三、连接复用检查

HTTP 连接是接口调用的常见开销。每一次 TCP 握手都需要时间和资源。

### 3.1 HTTP 连接复用检查

**问题识别**：

- [ ] 是否大量使用短连接？每次请求都新建 TCP 连接。
- [ ] HTTP Client 是否使用了连接池？
- [ ] 是否开启了 HTTP Keep-Alive？

**优化建议**：

```java
// 问题：每次请求创建新连接
CloseableHttpClient client = HttpClients.createDefault();

// 正确：使用连接池
PoolingHttpClientConnectionManager cm = new PoolingHttpClientConnectionManager();
cm.setMaxTotal(200);
cm.setDefaultMaxPerRoute(50);

CloseableHttpClient client = HttpClients.custom()
    .setConnectionManager(cm)
    .build();
```

### 3.2 连接池参数检查

连接池参数配置不当也会影响性能：

- [ ] 最大连接数是否满足并发需求？（太小会排队，太大会浪费资源）
- [ ] 连接最大空闲时间是否合理？（太短：频繁创建；太长：浪费连接）
- [ ] 连接获取超时时间是否设置？（防止线程无限等待）

## 四、缓存策略检查

缓存是接口性能优化的杀手锏，但用不好反而适得其反。

### 4.1 缓存命中检查

**检查项**：

- [ ] 接口是否适合缓存？（读多写少、数据量适中）
- [ ] 缓存 key 的设计是否合理？（便于排查、不易冲突）
- [ ] 缓存命中率是多少？（低于 80% 要分析原因）

**缓存命中率计算**：

```bash
# 查看 Redis 统计
redis-cli INFO stats | grep keyspace

# 计算命中率
# 命中率 = keyspace_hits / (keyspace_hits + keyspace_misses)
```

### 4.2 缓存问题排查

| 问题 | 表现 | 解决方案 |
|-----|------|---------|
| 穿透 | 大量请求查询不存在的数据 | 布隆过滤器 + 缓存空值 |
| 击穿 | 热点 key 过期，瞬间大量请求打到 DB | 互斥锁 / 逻辑过期 / 永久不过期 |
| 雪崩 | 大量 key 同时过期 | 过期时间随机化 + 多级缓存 |

> 详见 [缓存穿透/击穿/雪崩](/high-throughput/high-concurrency/cache-architecture/cache-strategy/cache-breakdown)。

### 4.3 缓存粒度检查

- [ ] 是否缓存了过多不需要的数据？（如缓存整个用户对象，只用到昵称）
- [ ] 是否可以用 Hash 替代 String 来聚合相关数据？（减少 key 数量）

## 五、限流与降级检查

在高并发场景下，限流和降级是保护系统的最后一道防线。

### 5.1 限流检查

**检查项**：

- [ ] 是否对接口配置了限流规则？
- [ ] 限流阈值是否合理？（根据历史峰值设置）
- [ ] 限流后的降级方案是什么？

**限流算法选择**：

| 算法 | 特点 | 适用场景 |
|-----|------|---------|
| 固定窗口 | 简单，有临界问题 | 简单场景 |
| 滑动窗口 | 精度高，实现复杂 | 精确限流 |
| 令牌桶 | 支持突发流量 | API 限流 |
| 漏桶 | 流量平滑 | 绝对平滑 |

> 详见 [限流算法](/high-throughput/high-availability/rate-limit-circuit/ratelimit-algorithm)。

### 5.2 超时配置检查

- [ ] 是否对所有外部调用配置了超时？
- [ ] 超时时间是否合理？（太短：容易误判；太长：阻塞链路）
- [ ] 是否考虑了超时传递问题？

**超时配置建议**：

| 配置项 | 推荐值 | 说明 |
|-------|-------|------|
| 连接超时 | 3-5s | 建立连接的最大等待时间 |
| 读取超时 | 5-10s | 等待响应的最大时间 |
| 重试次数 | 1-2 次 | 幂等接口可重试 |

### 5.3 降级方案检查

- [ ] 核心接口是否有兜底方案？
- [ ] 降级后的返回内容是什么？
- [ ] 是否做了熔断配置？

## 六、查询优化检查

### 6.1 数据库查询检查

- [ ] 是否有 N+1 查询问题？（循环中查数据库）

```java
// 问题：N+1 查询
List<Order> orders = orderMapper.selectList();
for (Order order : orders) {
    User user = userMapper.selectById(order.getUserId());  // N 次查询
}

// 正确：批量查询 + 内存关联
List<Order> orders = orderMapper.selectList();
Set<Long> userIds = orders.stream().map(Order::getUserId).collect(toSet());
Map<Long, User> userMap = userMapper.selectBatchIds(userIds)
    .stream().collect(toMap(User::getId, u -> u));
```

- [ ] 是否可以用 JOIN 替代循环查询？（根据数据量权衡）
- [ ] 分页查询是否做了限制？（防止一次查询返回过多数据）

### 6.2 索引使用检查

- [ ] WHERE 条件字段是否有索引？
- [ ] 是否需要覆盖索引以避免回表？
- [ ] 是否存在索引失效的情况？

## 七、日志与监控检查

### 7.1 日志规范检查

- [ ] 生产环境是否关闭了 DEBUG 日志？
- [ ] 日志是否使用了占位符而不是字符串拼接？
- [ ] 高频日志是否考虑异步化？

```java
// 问题：字符串拼接
log.info("userId=" + userId + ", orderId=" + orderId);

// 正确：使用占位符
log.info("userId={}, orderId={}", userId, orderId);
```

### 7.2 监控指标检查

**必监控的接口指标**：

- [ ] QPS / TPS
- [ ] RT（P50/P95/P99）
- [ ] 错误率
- [ ] 慢请求比例

**告警阈值**：

- [ ] P99 RT > SLA 目标是否告警？
- [ ] 错误率 > 1% 是否告警？
- [ ] 接口超时是否立即告警？

## 八、并发处理检查

### 8.1 串行 vs 并行检查

- [ ] 是否存在可以并行的串行调用？

```java
// 串行（慢）
User user = userService.getUser(id);
List<Order> orders = orderService.getOrders(id);
List<Product> products = productService.getProducts(id);

// 并行（快）
CompletableFuture<User> userFuture = userService.getUserAsync(id);
CompletableFuture<List<Order>> ordersFuture = orderService.getOrdersAsync(id);
CompletableFuture<List<Product>> productsFuture = productService.getProductsAsync(id);

CompletableFuture.allOf(userFuture, ordersFuture, productsFuture).join();
```

### 8.2 锁竞争检查

- [ ] 是否存在不必要的全局锁？
- [ ] 是否可以用 ConcurrentHashMap 替代 HashMap？
- [ ] synchronized 的范围是否过大？

---

## 九、总结：检查顺序

接口优化不是一上来就改代码。按照这个顺序检查，效率最高：

```
第一步：定位瓶颈
  └─ APM 追踪、慢日志分析，找到耗时最长的环节

第二步：序列化与传输
  └─ 数据体积是否过大？序列化方案是否合适？

第三步：缓存检查
  └─ 是否适合缓存？命中率是否正常？

第四步：查询优化
  └─ 是否有 N+1？索引是否合理？

第五步：并发与连接
  └─ 串行能否并行？连接是否复用？

第六步：限流与降级
  └─ 限流是否到位？降级方案是否完备？

第七步：监控与日志
  └─ 关键指标是否已覆盖？日志是否影响性能？
```

用好这个 Checklist，让接口优化有方向、有依据、有闭环。

## 面试追问方向

- **「你的接口做过哪些性能优化？」** 可以结合 Checklist 的具体检查项展开。
- **「如何判断接口的瓶颈在哪里？」** 回答：用 APM 工具追踪，结合慢日志分析。
- **「JSON 和 Protobuf 的区别，如何选择？」** 回答：看场景，对内用 Protobuf，对外用 JSON。
