# HTTP 连接池与接口调用优化

> [!TIP]
> 连接池用得好，高并发没烦恼——用不好，那就是一个隐藏的性能杀手

## 场景切入：接口超时，却找不到原因

你遇到过这种情况吗？系统 QPS 不高，数据库查询正常，Redis 也没问题，但接口 RT 突然飙升，甚至超时。

排查了一圈，发现问题出在 **HTTP 客户端**——每次调用下游服务都在建立新连接，三次握手、TLS 握手消耗了大量时间。

更糟糕的是，由于连接建立慢，请求开始排队，后面的请求只能等待，形成了恶性循环。

这就是连接池要解决的核心问题：**复用连接，减少连接建立的开销**。

---

## 一、连接池核心原理

### 为什么连接复用这么重要？

先看一组数据，对比建立 TCP 连接的成本：

| 操作 | 时间消耗 |
|---|---|
| 本地回环（localhost） | < 1ms |
| 同一数据中心 | 1-5ms |
| 跨地域数据中心 | 20-100ms |
| 国际网络 | 100-500ms |

对于一个需要调用下游服务的接口来说，如果每次请求都要建立新连接：

```text
1000 QPS × 10ms（连接建立时间）= 10s CPU 时间等待连接
```

但如果复用连接，同样的 QPS 可能只需要几十个连接，连接建立成本接近 0。

### 连接池的工作原理

连接池的核心思想是**复用**，具体流程：

```text
┌─────────────────────────────────────────────────────────────┐
│                      连接池工作原理                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   请求到来 → 检查池中是否有可用连接                          │
│                ↓                                           │
│         有 → 取出使用 → 使用完毕归还池中                     │
│                ↓                                           │
│         无 → 等待/新建 → 使用完毕归还池中                    │
│                                                             │
│   池中连接 = 活跃连接（正在使用）+ 空闲连接（等待复用）        │
└─────────────────────────────────────────────────────────────┘
```

关键参数的作用：

- **最大连接数（maxTotal）**：池中最多能有多少连接
- **最小空闲（minIdle）**：保持多少空闲连接，避免冷启动
- **最大等待（maxWaitMillis）**：获取连接超时时间
- **空闲回收（minEvictableIdleTimeMillis）**：空闲多久后回收

## 二、HTTP Keep-Alive：长连接的基础

### 短连接的问题

HTTP/1.0 默认是短连接，每次请求都要：

```text
建立连接 → 请求 → 响应 → 关闭连接
    ↓
   重复以上步骤
```

这就是经典的 **短连接问题**：每个请求都要经历三次握手（TCP）和四次挥手（关闭连接）。

### Keep-Alive 的解决方案

HTTP/1.1 引入了 **Keep-Alive** 机制，默认开启：

```http
# 请求头
Connection: keep-alive

# 响应头
Connection: keep-alive
Keep-Alive: timeout=60, max=1000
```

Keep-Alive 的工作方式：

```text
TCP 连接建立 → HTTP 请求1 → HTTP 响应1 → HTTP 请求2 → HTTP 响应2 → ...
```

**一个 TCP 连接上，可以传输多个 HTTP 请求**，这就是连接复用的基础。

### Keep-Alive 的限制

虽然 Keep-Alive 解决了连接复用问题，但 HTTP/1.1 仍然存在**队头阻塞**：

```text
请求1 ──────────────────────────────────────────────────────>
请求2 ───────────>
请求3 ───>
       ↓
    请求2 必须等待请求1 完成才能开始（队头阻塞）
```

这就是为什么 HTTP/2 和 HTTP/3 要解决队头阻塞问题。

## 三、连接池核心参数详解

### Apache HttpClient 参数

```java title="HttpClient 连接池配置"
CloseableHttpClient httpClient = HttpClients.custom()
    // 最大连接数
    .setMaxTotal(200)
    // 每个路由的最大连接数（路由 = 目标主机:端口）
    .setDefaultMaxPerRoute(50)
    // 连接idle时间后回收
    .setIdleAfter(TimeValue.ofMinutes(1))
    // 连接最大存活时间
    .setMaxConnLifetime(TimeValue.ofMinutes(5))
    // 连接验证（获取连接时）
    .setConnectionPoolConcurrencyMode(ConcurrencyPatrol.STRICT)
    // 重试策略
    .setRetryHandler(new DefaultHttpRequestRetryHandler(3, true))
    .build();
```

| 参数 | 说明 | 建议值 |
|---|---|---|
| `MaxTotal` | 连接池最大连接数 | 根据下游服务能力设置 |
| `DefaultMaxPerRoute` | 每个路由最大连接数 | MaxTotal / 下游服务实例数 |
| `maxConnLifetime` | 连接最大存活时间 | 5-10 分钟，避免连接失效 |
| `idleConnection` | 空闲连接回收时间 | 1-2 分钟 |

### OkHttp 参数

```java title="OkHttp 连接池配置"
OkHttpClient client = new OkHttpClient.Builder()
    // 连接池配置
    .connectionPool(new ConnectionPool()
        .Builder()
        .maxIdleConnections(5)           // 最大空闲连接数
        .keepAliveDuration(5, TimeUnit.MINUTES)  // 空闲存活时间
        .build())
    // 连接超时
    .connectTimeout(10, TimeUnit.SECONDS)
    // 读取超时
    .readTimeout(30, TimeUnit.SECONDS)
    // 写入超时
    .writeTimeout(30, TimeUnit.SECONDS)
    // 协议（HTTP/1.1 或 HTTP/2）
    .protocols(Arrays.asList(Protocol.HTTP_1_1, Protocol.HTTP_2))
    .build();
```

:::tip OkHttp 默认配置
- 最大 5 个空闲连接
- 空闲时间 5 分钟
- 超过限制会自动回收最久未使用的连接
:::

### RestTemplate 连接池配置

```java title="RestTemplate 连接池配置"
@Configuration
public class RestTemplateConfig {
    
    @Bean
    public RestTemplate restTemplate() {
        PoolingHttpClientConnectionManager connectionManager = 
            new PoolingHttpClientConnectionManager();
        
        // 最大连接数
        connectionManager.setMaxTotal(200);
        // 每个路由最大连接数
        connectionManager.setDefaultMaxPerRoute(50);
        // 连接空闲超时（毫秒）
        connectionManager.setValidateAfterInactivity(30000);
        
        RequestConfig requestConfig = RequestConfig.custom()
            // 获取连接超时
            .setConnectionRequestTimeout(5000)
            // 连接超时
            .setConnectTimeout(5000)
            // 读取超时
            .setSocketTimeout(30000)
            .build();
        
        CloseableHttpClient httpClient = HttpClients.custom()
            .setConnectionManager(connectionManager)
            .setDefaultRequestConfig(requestConfig)
            // 开启重试
            .setRetryHandler(new DefaultHttpRequestRetryHandler(3, true))
            .build();
        
        return new RestTemplate(httpClient);
    }
}
```

## 四、连接池调优思路

### 核心公式

连接池大小不是拍脑袋定的，有一个计算公式：

```text
最大连接数 = 下游处理能力 / 单连接处理能力
```

具体来说：

| 场景 | 计算方式 |
|---|---|
| **IO 密集型** | 连接数可以设置较大（100-500），因为大部分时间在等待 IO |
| **CPU 密集型** | 连接数 = CPU 核心数 × 2 |
| **下游瓶颈** | 连接数 ≤ 下游服务的最大并发数 |

### 调优步骤

**第一步：确定瓶颈在哪里**

```bash
# 查看 CPU 使用率
top -Hp $(pgrep java)

# 查看网络 IO
sar -n DEV 1

# 查看连接状态
netstat -an | grep TIME_WAIT | wc -l
```

**第二步：计算理论值**

假设：
- 下游服务 RT = 50ms
- 单连接 QPS = 1000ms / 50ms = 20 QPS/连接
- 下游服务支持 1000 QPS

则：最大连接数 = 1000 / 20 = 50

**第三步：压测验证**

```bash
# 使用 wrk 压测验证
wrk -t4 -c100 -d30s --latency http://api.example.com/users
```

## 五、连接池的坑：踩一个悔三天

### 坑 1：连接泄漏

**问题**：获取连接后没有释放，导致连接池耗尽。

```java
// 错误写法：忘记释放连接
public String callApi(String url) throws IOException {
    CloseableHttpResponse response = httpClient.execute(new HttpGet(url));
    return EntityUtils.toString(response.getEntity());
    // 没有 finally 块关闭 response 和 httpClient
}

// 正确写法：使用 try-with-resources
public String callApi(String url) throws IOException {
    try (CloseableHttpResponse response = httpClient.execute(new HttpGet(url))) {
        return EntityUtils.toString(response.getEntity());
    }
}
```

**排查方法**：

```bash
# 查看 TIME_WAIT 连接数
netstat -an | grep -c TIME_WAIT

# 查看连接池状态（添加监控）
log.info("连接池状态: 活跃={}, 空闲={}, 总数={}", 
    pool.getStatistics().getLeased(), 
    pool.getStatistics().getAvailable(),
    pool.getStatistics().getTotal());
```

### 坑 2：连接池耗尽

**问题**：maxWaitMillis 设置过短，导致获取连接超时。

```java
// 错误配置
RequestConfig.custom()
    .setConnectionRequestTimeout(100)  // 只有 100ms，太短了！
    .build();

// 正确配置
RequestConfig.custom()
    .setConnectionRequestTimeout(5000)  // 5 秒，合理
    .build();
```

**症状**：

```
org.apache.http.conn.ConnectionPoolTimeoutException: 
    Timeout waiting for connection from pool
```

### 坑 3：连接超时 vs 读取超时

| 超时类型 | 说明 | 常见原因 |
|---|---|---|
| **connectTimeout** | 三次握手超时 | 网络不通、防火墙 |
| **readTimeout** | 读取数据超时 | 下游服务慢、数据量大 |
| **connectionRequestTimeout** | 从池中获取连接超时 | 连接池耗尽、线程阻塞 |

:::warning 超时配置不当的严重后果
- 超时过短：网络抖动时频繁超时
- 超时过长：请求堆积、线程耗尽
- 建议：connectTimeout = 5-10s，readTimeout = 30-60s
:::

### 坑 4：DNS 缓存问题

**问题**：下游服务 IP 变了，但 DNS 缓存还是旧的。

```java
// 错误：默认会缓存 DNS
OkHttpClient client = new OkHttpClient.Builder()
    // ...
    .build();

// 正确：禁用 DNS 缓存
OkHttpClient client = new OkHttpClient.Builder()
    .dns(hostname -> {
        InetAddress[] addresses = InetAddress.getAllByName(hostname);
        return Arrays.asList(addresses);
    })
    .build();
```

## 六、连接池监控指标

### 必须监控的指标

```java title="连接池监控埋点"
public class ConnectionPoolMonitor {
    
    public void recordMetrics(PoolingHttpClientConnectionManager pool) {
        PoolStats stats = pool.getTotalStats();
        
        // 监控指标
        monitor.gauge("http.pool.leased", stats.getLeased());      // 活跃连接数
        monitor.gauge("http.pool.available", stats.getAvailable()); // 空闲连接数
        monitor.gauge("http.pool.pending", stats.getPending());    // 等待获取连接的请求数
        monitor.gauge("http.pool.max", stats.getMax());            // 最大连接数
        
        // 告警：等待队列过长
        if (stats.getPending() > 100) {
            alert.send("连接池等待队列过长: {}", stats.getPending());
        }
    }
}
```

### 监控维度

| 指标 | 含义 | 告警阈值 |
|---|---|---|
| `leased` | 正在使用的连接数 | > 80% maxTotal |
| `available` | 空闲连接数 | 持续为 0 |
| `pending` | 等待获取连接的线程数 | > 10 |
| `max` | 最大连接数 | — |
| `create.count` | 新建连接数 | 持续很高说明连接泄漏 |

## 七、生产环境最佳配置

### Spring Boot 配置

```yaml title="application.yml"
spring:
  httpclient:
    pool:
      max-total: 200
      max-per-route: 50
      connect-timeout: 5000
      read-timeout: 30000
      connection-request-timeout: 5000

# 或使用 Hutool 的 HttpUtil
hutool:
  http:
    timeout: 5000,30000  # connectTimeout, readTimeout
```

### 微服务间调用配置

```java title="Feign 配置"
@Configuration
public class FeignConfig {
    
    @Bean
    public Client feignClient() {
        return new FeignApacheHttpClient(
            HttpClients.custom()
                .setMaxTotal(100)
                .setDefaultMaxPerRoute(20)
                .setKeepAliveStrategy((response, context) -> 30 * 1000) // 30秒
                .build()
        );
    }
}
```

### Dubbo 连接池配置

```xml title="dubbo.properties"
# Dubbo HTTP 连接池
dubbo.protocol.telnet=120000
dubbo.protocol.connections=10
dubbo.protocol.server=netty4
dubbo.protocol.client=netty4
dubbo.protocol.threadpool=fixed
dubbo.protocol.threads=200
```

## 面试实战

### 面试题 1：连接池的核心参数有哪些？

**参考答案**：

| 参数 | 说明 |
|---|---|
| `maxTotal` | 连接池最大连接数 |
| `maxIdle` | 最大空闲连接数 |
| `minIdle` | 最小空闲连接数（保活） |
| `maxWaitMillis` | 获取连接最大等待时间 |
| `testOnBorrow` | 获取连接时是否验证 |
| `testOnReturn` | 归还连接时是否验证 |
| `testWhileIdle` | 空闲时是否验证 |

建议根据下游服务能力和业务场景调整。

### 面试题 2：连接池耗尽如何排查？

**参考答案**：

1. **检查连接是否释放**：是否有 try-with-resources 或 finally 释放连接
2. **检查超时配置**：connectionRequestTimeout 是否合理
3. **检查下游服务**：是否响应慢导致连接占用时间过长
4. **监控指标**：查看 leased、pending 指标变化趋势

### 面试题 3：OkHttp 和 Apache HttpClient 如何选择？

**参考答案**：

| 维度 | OkHttp | Apache HttpClient |
|---|---|---|
| API 简洁度 | 更简洁 | 较复杂 |
| HTTP/2 支持 | 原生支持 | 需要额外配置 |
| 连接池 | 内置 | 需要手动配置 |
| 成熟度 | Google 维护 | Apache 维护，生态完善 |

**选型建议**：
- Android 开发：OkHttp
- Java 微服务：都可以，Apache HttpClient 生态更丰富
- Spring Cloud：Feign 默认用 HttpURLConnection，可切换为 Apache HttpClient
