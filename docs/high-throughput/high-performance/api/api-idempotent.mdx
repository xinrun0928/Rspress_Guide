# 接口幂等性：让重复调用安全可控

凌晨 2 点，你正准备睡觉，突然收到报警：某用户在秒杀活动中被扣了两次钱。用户投诉到客服，运营紧急调查，发现问题出在接口幂等性没做好——用户点击了两次支付按钮，后端处理了两次。

这不是个案。在分布式系统和网络环境中，接口被重复调用是一个极其常见的问题。原因可能是：

- 用户手抖点了两下
- 前端没有做防重复提交
- 网络超时导致客户端重试
- 消息队列重复消费
- 爬虫或恶意刷单

**如果接口没有做好幂等性，同样的操作执行两次，轻则数据错误，重则资金损失。** 轻则用户投诉，重则公司损失。

这篇文章的核心任务，就是帮你彻底理解幂等性，掌握各种实现方案，以及在不同场景下如何选择。

## 一、什么是幂等性

### 1.1 幂等性的定义

幂等性（Idempotency）是一个数学概念，引入到计算机领域后，指的是：

> **同一个接口，用同样的参数，无论调用一次还是多次，结果都是一样的。**

换句话说，接口重复调用与单次调用的效果一致，这就是幂等性。

### 1.2 幂等性 vs 安全性

很多人容易把幂等性和安全性搞混，这里做一个区分：

| 概念 | 定义 | 关注点 | 例子 |
|-----|------|-------|------|
| 幂等性 | 重复调用结果一致 | 数据一致性 | GET 查询、PUT 更新、DELETE 删除 |
| 安全性 | 防止恶意攻击 | 认证授权 | 登录验证、权限校验 |

**幂等性是保证数据一致性，安全性是防止恶意访问。** 两者是不同维度的问题，需要分别处理。

### 1.3 HTTP 方法的幂等性

HTTP 协议对不同方法的幂等性有明确定义：

| 方法 | 幂等性 | 说明 |
|-----|-------|------|
| GET | 幂等 | 查询操作，多次查询结果相同 |
| HEAD | 幂等 | 与 GET 类似，只返回头部 |
| PUT | 幂等 | 更新操作，用同样的数据更新，效果相同 |
| DELETE | 幂等 | 删除操作，删除已删除的资源，返回结果一致 |
| POST | 非幂等 | 创建操作，每次调用创建新资源 |
| PATCH | 非幂等 | 部分更新，结果可能不同 |

```
幂等：GET、PUT、DELETE
非幂等：POST、PATCH
```

:::tip 为什么 POST 和 PATCH 不是幂等的？

POST 是创建操作，每次调用都创建新资源（比如新订单），结果自然不同。

PATCH 是部分更新，虽然通常是幂等的，但如果更新逻辑涉及「增加」操作（如 balance = balance + 10），多次调用就会产生不同结果。
:::

## 二、为什么要保证幂等性

### 2.1 重复调用的场景

在生产环境中，接口被重复调用的场景非常常见：

**场景一：用户操作**

- 用户点击「提交订单」按钮，因为网络慢没有立即响应，又点了一次
- 用户按 F5 刷新页面，导致表单重复提交
- 用户使用浏览器的「后退」按钮，返回到表单页面，再次提交

**场景二：网络问题**

- 网络超时，客户端自动重试
- TCP 协议重传机制导致重复请求
- 消息队列（RabbitMQ、RocketMQ）投递重复消息

**场景三：系统问题**

- 分布式系统中，服务调用超时重试
- 消息消费者重启导致消息重复消费
- 定时任务重复执行

### 2.2 不保证幂等性的后果

如果没有做好幂等性保护，会出现各种问题：

| 场景 | 后果 |
|-----|------|
| 重复下单 | 用户被扣多次钱，商家被刷单 |
| 重复支付 | 资金损失，财务对账困难 |
| 重复领取优惠券 | 薅羊毛，运营活动失效 |
| 重复扣库存 | 超卖，订单无法履约 |
| 重复发送短信 | 用户投诉，短信费用浪费 |

:::warning 资金类操作必须保证幂等性

涉及资金、库存等核心业务逻辑的接口，幂等性是底线。一旦出问题，轻则用户投诉，重则公司损失。

这不是「锦上添花」，而是「必备能力」。
:::

## 三、幂等性实现方案

### 3.1 方案一：唯一 Token 方案

#### 原理

在用户提交表单之前，先从服务端获取一个唯一的 Token，提交时携带这个 Token。服务端用 Redis 记录已使用的 Token，如果 Token 已使用则拒绝处理。

#### 流程图

```
用户 → 1. 获取 Token（服务端生成唯一 ID，存入 Redis，TTL 有效）→ 返回 Token
用户 → 2. 提交表单（携带 Token）
服务端 → 3. 检查 Token 是否已使用
        ├─ 已使用 → 返回「请勿重复提交」
        └─ 未使用 → 处理业务，标记 Token 已使用 → 返回成功
```

#### 代码实现

```java
@Service
public class IdempotentService {

    @Autowired
    private RedisTemplate<String, String> redisTemplate;

    // 生成唯一 Token
    public String generateToken() {
        String token = UUID.randomUUID().toString();
        // 设置 30 分钟有效期
        redisTemplate.opsForValue().set("idempotent:token:" + token, "0",
            30, TimeUnit.MINUTES);
        return token;
    }

    // 校验 Token
    public boolean checkToken(String token) {
        String key = "idempotent:token:" + token;
        // SETNX：如果 key 不存在则设置，返回 true 表示首次调用
        Boolean result = redisTemplate.opsForValue().setIfAbsent(key, "1",
            30, TimeUnit.MINUTES);
        return Boolean.TRUE.equals(result);
    }
}
```

#### 前端配合

```html
<form action="/api/order/submit" method="POST">
    <!-- 隐藏字段存储 Token -->
    <input type="hidden" name="idempotentToken" th:value="${token}" />
    <button type="submit">提交订单</button>
</form>

<script>
// 防重复提交：按钮点击后禁用
document.querySelector('form').addEventListener('submit', function() {
    const btn = this.querySelector('button');
    btn.disabled = true;
    setTimeout(() => btn.disabled = false, 3000); // 3秒后恢复
});
</script>
```

#### 适用场景

- 表单提交（订单创建、用户注册）
- 按钮操作（支付、领取优惠券）
- 需要用户交互的场景

#### 优缺点

| 优点 | 缺点 |
|-----|------|
| 实现简单，对业务代码无侵入 | 需要前端配合 |
| Token 存在 Redis，天然分布式 | Token 有有效期，需要续期 |
| 可以精确控制每个操作的幂等性 | 并发请求时，第一个请求处理中会放行后续请求 |

### 3.2 方案二：乐观锁方案

#### 原理

在数据库表中添加一个 `version` 字段，每次更新时携带版本号，只有版本号匹配时才更新。

#### 核心 SQL

```sql
UPDATE orders
SET status = 'PAID', version = version + 1
WHERE order_id = ? AND version = ?;
-- 影响行数为 1 表示成功，为 0 表示版本不匹配（已被更新）
```

#### Java 代码实现

```java
// 数据库更新
@Update("UPDATE orders SET status = #{status}, version = version + 1 " +
        "WHERE order_id = #{orderId} AND version = #{version}")
int updateStatus(@Param("orderId") Long orderId,
                 @Param("status") String status,
                 @Param("version") Integer version);

// 业务层处理
public boolean payOrder(Long orderId) {
    // 查询当前订单
    Order order = orderMapper.selectById(orderId);
    if (order == null) {
        throw new BusinessException("订单不存在");
    }
    if ("PAID".equals(order.getStatus())) {
        return true; // 已经是已支付状态，直接返回
    }

    // 乐观锁更新
    int affected = orderMapper.updateStatus(orderId, "PAID", order.getVersion());
    if (affected == 0) {
        throw new BusinessException("订单状态已变更，请重试");
    }
    return true;
}
```

#### 适用场景

- 数据更新操作（订单状态变更、用户信息更新）
- 并发更新不严重的场景
- 对性能要求不高的场景

#### 优缺点

| 优点 | 缺点 |
|-----|------|
| 实现简单，不需要额外存储 | 只能用于更新操作，不能用于创建 |
| 性能好，无锁等待 | 并发更新时失败率较高 |
| 天然支持分布式 | 需要在实体类中维护 version 字段 |

### 3.3 方案三：悲观锁方案

#### 原理

在数据库层面使用行级锁（如 `SELECT FOR UPDATE`），确保同一时刻只有一个请求能处理。

#### 核心 SQL

```sql
SELECT * FROM orders WHERE order_id = ? FOR UPDATE;
-- FOR UPDATE 会锁定这一行，其他事务需要等待
```

#### Java 代码实现

```java
@Transactional
public boolean payOrder(Long orderId) {
    // 使用悲观锁查询
    Order order = orderMapper.selectByIdForUpdate(orderId);
    if (order == null) {
        throw new BusinessException("订单不存在");
    }
    if ("PAID".equals(order.getStatus())) {
        return true; // 已经是已支付状态
    }

    // 扣款逻辑
    accountMapper.deduct(order.getUserId(), order.getAmount());

    // 更新订单状态
    order.setStatus("PAID");
    orderMapper.updateById(order);

    return true;
}
```

```xml
<select id="selectByIdForUpdate" resultType="Order">
    SELECT * FROM orders WHERE order_id = #{orderId} FOR UPDATE
</select>
```

#### 适用场景

- 强一致性要求的场景（资金操作）
- 并发更新比较严重的场景
- 可以接受一定的性能损耗

#### 优缺点

| 优点 | 缺点 |
|-----|------|
| 保证强一致性 | 性能损耗较大，并发高时可能死锁 |
| 实现简单，数据库原生支持 | 锁的范围和时间要控制好 |
| 不会出现更新失败的情况 | 只适用于单节点，分布式环境需配合分布式锁 |

### 3.4 方案四：分布式锁方案

#### 原理

在处理业务之前，先获取一个分布式锁，锁的 key 由业务标识组成（如 `order:pay:123`）。获取锁成功才处理，处理完成后释放锁。

#### 核心代码

```java
@Service
public class DistributedLockIdempotentService {

    @Autowired
    private RedissonClient redissonClient;

    public boolean executeWithLock(String businessKey, Runnable task) {
        String lockKey = "idempotent:lock:" + businessKey;
        RLock lock = redissonClient.getLock(lockKey);

        try {
            // tryLock：尝试获取锁，等待 3 秒，锁自动过期 10 秒
            boolean locked = lock.tryLock(3, 10, TimeUnit.SECONDS);
            if (!locked) {
                throw new BusinessException("请求正在处理中，请稍后重试");
            }

            // 执行任务
            task.run();
            return true;
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            throw new BusinessException("系统繁忙，请稍后重试");
        } finally {
            // 释放锁
            if (lock.isHeldByCurrentThread()) {
                lock.unlock();
            }
        }
    }

    // 支付业务使用
    public void payOrder(Long orderId) {
        executeWithLock("order:pay:" + orderId, () -> {
            // 支付逻辑
            Order order = orderMapper.selectById(orderId);
            // ... 处理逻辑
        });
    }
}
```

#### 适用场景

- 分布式环境下的幂等控制
- 跨服务调用的幂等保证
- 需要精确控制并发访问的场景

#### 优缺点

| 优点 | 缺点 |
|-----|------|
| 天然支持分布式 | 实现复杂度较高 |
| 锁的粒度可控制 | 需要依赖 Redis/ZooKeeper 等中间件 |
| 可以设置锁的有效期，防止死锁 | 性能比本地锁稍差 |

### 3.5 方案五：状态机方案

#### 原理

对于有明确状态流转的操作（如订单），通过状态机的流转限制来保证幂等性。只有在正确的状态下才能执行对应的操作。

#### 订单状态流转

```
待支付 → 已支付 → 已完成
   ↓        ↓
 已取消   已退款
```

```java
// 状态机映射
private static final Map<String, Set<String>> STATE_TRANSITIONS = new HashMap<>();

static {
    STATE_TRANSITIONS.put("待支付", Set.of("已支付", "已取消"));
    STATE_TRANSITIONS.put("已支付", Set.of("已完成", "已退款"));
    STATE_TRANSITIONS.put("已完成", Set.of()); // 终态，不可流转
    STATE_TRANSITIONS.put("已取消", Set.of()); // 终态，不可流转
    STATE_TRANSITIONS.put("已退款", Set.of()); // 终态，不可流转
}

public boolean payOrder(Long orderId) {
    Order order = orderMapper.selectById(orderId);

    // 校验当前状态是否可以流转到目标状态
    Set<String> allowedStates = STATE_TRANSITIONS.get(order.getStatus());
    if (allowedStates == null || !allowedStates.contains("已支付")) {
        throw new BusinessException("订单状态不允许支付");
    }

    // 执行支付逻辑
    // ...

    // 更新状态
    order.setStatus("已支付");
    orderMapper.updateById(order);

    return true;
}
```

#### 适用场景

- 有明确状态流转的业务（订单、工单、审批流）
- 状态流转是单向的、不可逆的
- 需要结合其他方案使用

#### 优缺点

| 优点 | 缺点 |
|-----|------|
| 符合业务逻辑，语义清晰 | 只能用于有状态流转的场景 |
| 可以结合乐观锁使用 | 需要维护状态流转规则 |
| 天然幂等 | 不同业务的状态机不同 |

### 3.6 方案六：去重表方案

#### 原理

利用数据库唯一索引的特性，创建一个去重表（或在业务表中添加唯一索引）。每次操作前尝试插入唯一标识，如果插入成功才执行后续逻辑。

#### 表结构设计

```sql
CREATE TABLE idempotent_record (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    business_type VARCHAR(50) NOT NULL COMMENT '业务类型',
    business_id VARCHAR(100) NOT NULL COMMENT '业务标识',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uk_business (business_type, business_id)
) COMMENT '幂等性去重表';
```

#### Java 代码实现

```java
@Service
public class DeduplicationService {

    @Autowired
    private IdempotentRecordMapper idempotentMapper;

    @Transactional
    public void executeWithDeduplication(String businessType, String businessId,
                                          Runnable task) {
        try {
            // 尝试插入记录，如果唯一键冲突则抛异常
            IdempotentRecord record = new IdempotentRecord();
            record.setBusinessType(businessType);
            record.setBusinessId(businessId);
            idempotentMapper.insertSelective(record);
        } catch (DuplicateKeyException e) {
            // 唯一键冲突，说明已经处理过
            log.info("业务已处理，businessType={}, businessId={}", businessType, businessId);
            return;
        }

        // 执行实际业务逻辑
        task.run();
    }
}

// 使用示例
public void submitOrder(Long userId, Long productId) {
    String businessId = String.format("%d:%d:%d", userId, productId, System.currentTimeMillis());
    deduplicationService.executeWithDeduplication("ORDER_SUBMIT", businessId, () -> {
        // 订单提交逻辑
    });
}
```

#### 适用场景

- 需要精确去重的场景
- 可以用业务标识作为唯一键
- 对一致性要求较高的场景

#### 优缺点

| 优点 | 缺点 |
|-----|------|
| 实现简单，数据库原生支持 | 依赖数据库的唯一索引 |
| 可靠性高，数据持久化 | 去重表会逐渐增大，需要清理 |
| 可以记录处理历史 | 并发插入时失败率较高 |

## 四、方案对比与选型

### 4.1 方案对比表

| 方案 | 适用场景 | 性能 | 实现复杂度 | 可靠性 |
|-----|---------|------|----------|-------|
| 唯一 Token | 表单提交、按钮操作 | 高 | 低 | 中 |
| 乐观锁 | 数据更新 | 高 | 低 | 中 |
| 悲观锁 | 资金操作、强一致性 | 低 | 低 | 高 |
| 分布式锁 | 分布式环境 | 中 | 中 | 高 |
| 状态机 | 有状态流转的业务 | 高 | 中 | 高 |
| 去重表 | 精确去重 | 中 | 低 | 高 |

### 4.2 选型建议

**根据业务场景选择**：

| 场景 | 推荐方案 | 原因 |
|-----|---------|------|
| 表单提交 | Token 方案 | 需要前端配合，粒度可控制 |
| 数据更新 | 乐观锁 + 状态机 | 性能好，符合业务逻辑 |
| 资金操作 | 悲观锁 + 分布式锁 | 强一致性是底线 |
| 消息消费 | 去重表 + 状态机 | 消息可能重复消费 |
| 定时任务 | 分布式锁 | 防止多节点重复执行 |

**组合使用**：

实际生产中，往往需要组合使用多种方案：

- 乐观锁 + 状态机：更新前检查状态，更新时用 version 控制并发
- 分布式锁 + 去重表：锁住处理流程，用数据库唯一键兜底
- Token + 悲观锁：Token 拦截重复提交，悲观锁保证资金安全

## 五、代码实战：订单支付幂等性

### 5.1 完整实现

```java
@Service
public class OrderPayService {

    @Autowired
    private OrderMapper orderMapper;

    @Autowired
    private RedissonClient redissonClient;

    @Autowired
    private IdempotentRecordMapper idempotentMapper;

    /**
     * 支付订单（完整幂等性保护）
     */
    @Transactional
    public PayResult payOrder(Long orderId, String payToken) {
        // 1. 幂等性检查：Token 验证
        if (!verifyPayToken(payToken, orderId)) {
            return PayResult.alreadyProcessed("订单已支付，请勿重复操作");
        }

        // 2. 分布式锁：防止并发重复支付
        String lockKey = "order:pay:" + orderId;
        RLock lock = redissonClient.getLock(lockKey);
        try {
            boolean locked = lock.tryLock(5, 30, TimeUnit.SECONDS);
            if (!locked) {
                return PayResult.processing("订单正在处理中");
            }

            // 3. 悲观锁：查询并锁定订单
            Order order = orderMapper.selectByIdForUpdate(orderId);
            if (order == null) {
                throw new BusinessException("订单不存在");
            }

            // 4. 状态机：检查状态是否允许支付
            if ("PAID".equals(order.getStatus())) {
                return PayResult.alreadyProcessed("订单已支付");
            }
            if ("CANCELLED".equals(order.getStatus())) {
                return PayResult.failed("订单已取消，无法支付");
            }

            // 5. 执行支付逻辑
            boolean paySuccess = executePayment(order);

            // 6. 更新订单状态（乐观锁）
            int updated = orderMapper.updateStatusWithVersion(
                orderId, "PAID", order.getVersion());
            if (updated == 0) {
                throw new BusinessException("订单状态已变更，请重试");
            }

            // 7. 记录幂等性
            saveIdempotentRecord("ORDER_PAY", orderId.toString());

            return PayResult.success("支付成功");

        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            return PayResult.failed("系统繁忙，请稍后重试");
        } finally {
            if (lock.isHeldByCurrentThread()) {
                lock.unlock();
            }
        }
    }

    private boolean verifyPayToken(String token, Long orderId) {
        // Token 校验逻辑
        String key = "pay:token:" + orderId;
        return redisTemplate.opsForValue().setIfAbsent(key, "1", 30, TimeUnit.MINUTES);
    }

    private boolean executePayment(Order order) {
        // 调用支付渠道
        return true;
    }
}
```

### 5.2 关键点总结

1. **Token 校验**：第一道防线，拦截明显重复的请求
2. **分布式锁**：第二道防线，防止多节点并发处理
3. **悲观锁查询**：锁定订单行，确保数据一致性
4. **状态机检查**：业务层面的幂等保障
5. **乐观锁更新**：处理并发更新冲突
6. **去重表记录**：最终兜底，防止极端情况

## 六、面试实战

### 6.1 场景一：幂等性实现方案

**面试官问**：「你们项目中是如何保证接口幂等性的？」

**回答要点**：

> 我们项目根据不同的业务场景，使用了多种幂等性方案：
>
> 1. **表单提交类**：使用唯一 Token 方案，前端获取 Token，后端校验 Token 是否已使用
> 2. **数据更新类**：使用乐观锁，在表中添加 version 字段，更新时携带版本号
> 3. **资金操作类**：使用悲观锁（SELECT FOR UPDATE）+ 分布式锁双重保护
> 4. **消息消费类**：使用去重表，利用数据库唯一键防止重复消费
>
> 实际生产中，我们通常组合使用多种方案，比如支付接口：Token 拦截 + 分布式锁 + 悲观锁 + 乐观锁，四重保障。

### 6.2 场景二：乐观锁 vs 悲观锁

**面试官问**：「乐观锁和悲观锁的区别？分别在什么场景下使用？」

| 维度 | 乐观锁 | 悲观锁 |
|-----|-------|-------|
| 实现方式 | version 字段，无锁 | SELECT FOR UPDATE，有锁 |
| 适用场景 | 并发不严重、更新不频繁 | 并发严重、关键业务 |
| 性能 | 好（无锁等待） | 差（锁等待） |
| 数据一致性 | 最终一致 | 强一致 |

> 我的选择原则是：一般更新操作用乐观锁，资金操作用悲观锁。乐观锁性能好，但在高并发下失败率会上升；悲观锁能保证强一致，但会降低吞吐量。

### 6.3 场景三：分布式锁的使用

**面试官问**：「分布式锁有哪些实现方式？有什么区别？」

| 实现 | 优点 | 缺点 | 适用场景 |
|-----|------|------|---------|
| Redis | 性能高，支持过期时间 | 单机有脑裂风险，需 RedLock | 大多数场景 |
| ZooKeeper | 强一致，有watch机制 | 性能稍差 | 需要强一致 |
| etcd | 高可靠，支持租约 | 配置复杂 | K8s 生态 |

> 我们项目使用的是 Redis 分布式锁（Redisson），因为性能高，且支持自动续期，防止锁过期业务还没执行完的问题。

## 七、总结

幂等性是接口设计的基石，特别是涉及写操作时。核心要点：

1. **幂等性定义**：重复调用与单次调用效果一致
2. **常见场景**：用户重试、网络超时、消息重复
3. **实现方案**：Token、乐观锁、悲观锁、分布式锁、状态机、去重表
4. **选型原则**：根据业务场景和一致性要求选择，必要时组合使用
5. **实战要点**：多层防护，组合使用多种方案

> **思考题**：如果让你设计一个接口，要求保证幂等性且性能最优，你会怎么设计？考虑不同场景（查询、更新、创建、删除）的差异。
