# 序列化方案对比与选型

> [!TIP]
> 接口 RT 100ms，其中序列化耗时 30ms——你注意过这个数字吗？

## 场景切入：一个被忽视的性能杀手

假设你有一个接口，RT 要求 50ms：

```text
数据库查询: 20ms
业务逻辑: 10ms
序列化: 20ms    ← 这个经常被忽视
网络传输: 10ms
─────────────────
总计: 60ms
```

单次看起来还好，但如果是高并发场景：

```text
10000 QPS × 20ms = 200秒 CPU 时间在序列化
```

如果换成更快的序列化方案（比如从 JSON 换成 Protobuf）：

```text
10000 QPS × 3ms = 30秒 CPU 时间
节省: 170秒 = 性能提升 6-7 倍
```

这就是序列化优化的威力。**它不改变业务逻辑，但能让整个系统跑得更快**。

---

## 一、序列化性能的关键指标

### 四大核心指标

评价一个序列化方案，主要看这四个指标：

| 指标 | 说明 | 优化方向 |
|---|---|---|
| **序列化速度** | 对象 → 字节数组的速度 | 减少反射、预计算 |
| **反序列化速度** | 字节数组 → 对象的速度 | 直接内存操作 |
| **字节体积** | 序列化后的数据大小 | 二进制编码、压缩 |
| **跨语言支持** | Java/Python/Go 等能否互相解析 | IDL 定义 |

### 不同场景的侧重点

| 场景 | 关键指标 |
|---|---|
| **微服务内部调用** | 序列化/反序列化速度最重要 |
| **对外 API** | 可读性、跨语言最重要 |
| **大数据量传输** | 字节体积最重要 |
| **日志收集** | 写入速度最重要 |

## 二、常见序列化方案对比

### 一览表

| 方案 | 类型 | 序列化速度 | 反序列化速度 | 字节体积 | 跨语言 | 典型场景 |
|---|---|---|---|---|---|---|
| **JSON** | 文本 | 慢 | 慢 | 大 | 完美 | 对外 API、Web |
| **Protobuf** | 二进制 | 快 | 极快 | 很小 | 完美 | 内部 RPC、微服务 |
| **Hessian** | 二进制 | 中等 | 中等 | 中等 | 支持 | Dubbo 默认 |

| **FST** | 二进制 | 极快 | 极快 | 小 | 否 | Java 内部 |
| **MessagePack** | 二进制 | 快 | 快 | 小 | 支持 | 跨语言场景 |

## 三、JSON：最通用的选择

### 原理

JSON 是纯文本格式，结构简单，易于阅读和调试：

```json
{
    "name": "张三",
    "age": 25,
    "email": "zhang@example.com"
}
```

### 主要实现

| 实现 | 特点 |
|---|---|
| **Jackson** | Spring 默认，功能全面，社区活跃 |
| **Fastjson** | Alibaba 出品，速度快，但安全问题较多 |
| **Gson** | Google 出品，API 友好 |

### Jackson 使用示例

```java
// 对象序列化
ObjectMapper mapper = new ObjectMapper();
User user = new User("张三", 25);

// 序列化
String json = mapper.writeValueAsString(user);

// 反序列化
User parsed = mapper.readValue(json, User.class);
```

### Jackson 性能优化

```java
// 1. 预编译 ObjectMapper（推荐）
private static final ObjectMapper MAPPER = new ObjectMapper();

// 2. 关闭不需要的特性
MAPPER.configure(DeserializationFeature.FAIL_ON_UNKNOWN_PROPERTIES, false);
MAPPER.configure(SerializationFeature.WRITE_DATES_AS_TIMESTAMPS, false);

// 3. 使用缓存
mapper.getFactory().setCache(new BigDecimalRootNodeCache());

// 4. 使用流式 API（大对象优化）
StringWriter writer = new StringWriter();
mapper.writeValue(writer, user);
String json = writer.toString();
```

### JSON 的优缺点

| 优点 | 缺点 |
|---|---|
| ✅ 人类可读，便于调试 | ❌ 速度慢，体积大 |
| ✅ 跨语言支持完美 | ❌ 需要字符串解析 |
| ✅ 生态完善 | ❌ 没有类型信息 |

:::tip 什么时候用 JSON？
- 对外 API（需要可读性）
- 需要人类调试的场景
- 跨语言调用
- 不追求极致性能
:::

## 四、Protobuf：高性能首选

### 原理

Protobuf 是 Google 出品的二进制序列化协议，核心思想是**用字段编号代替字段名**：

```text
JSON: {"name": "张三", "age": 25}
       ──────────────── 冗余的字段名

Protobuf: [0x08, 0x19, 0x12, 0x06, 0x5A, ...]
          ─── ─────── ─── ─── ─── ──────
          字段1  字段1  字段2  字段2  字段2  字段2
          编号  值     编号  值长度 值（张三 UTF-8）
```

同一个对象，Protobuf 体积只有 JSON 的 **30-50%**。

### IDL 定义

```protobuf title="user.proto"
syntax = "proto3";

package user;

option java_package = "com.example.user";
option java_outer_classname = "UserProtos";

message User {
    string name = 1;      // 字段编号 1
    int32 age = 2;       // 字段编号 2
    string email = 3;    // 字段编号 3
    repeated string tags = 4;  // 数组
}

message UserList {
    repeated User users = 1;
}
```

### Maven 编译

```xml title="pom.xml"
<dependencies>
    <dependency>
        <groupId>com.google.protobuf</groupId>
        <artifactId>protobuf-java</artifactId>
        <version>3.25.1</version>
    </dependency>
</dependencies>

<build>
    <plugins>
        <plugin>
            <groupId>org.xolstice.maven.plugins</groupId>
            <artifactId>protobuf-maven-plugin</artifactId>
            <version>0.6.1</version>
            <configuration>
                <protocExecutable>/usr/local/bin/protoc</protocExecutable>
            </configuration>
            <executions>
                <execution>
                    <goals>
                        <goal>compile</goal>
                    </goals>
                </execution>
            </executions>
        </plugin>
    </plugins>
</build>
```

### Java 使用示例

```java
// 序列化
User user = User.newBuilder()
    .setName("张三")
    .setAge(25)
    .setEmail("zhang@example.com")
    .addTags("Java")
    .addTags("微服务")
    .build();

byte[] bytes = user.toByteArray();

// 反序列化
User parsed = User.parseFrom(bytes);

// JSON 互转
String json = JsonFormat.printer().print(user);
JsonFormat.parser().merge(json, userBuilder);
```

### Protobuf 的优势

| 优势 | 说明 |
|---|---|
| **速度极快** | 比 JSON 快 5-10 倍 |
| **体积小** | 比 JSON 小 30-50% |
| **类型安全** | 编译时校验，字段错误提前发现 |
| **向后兼容** | 字段编号机制，增删字段不影响 |
| **跨语言** | 自动生成多语言代码 |

### Protobuf 的缺点

| 缺点 | 说明 |
|---|---|
| ❌ 不可读 | 需要工具才能查看内容 |
| ❌ 需要预定义 | 需要写 .proto 文件 |
| ❌ 学习成本 | 需要了解 IDL 语法 |
| ❌ 字段编号管理 | 需要全局统一管理 |

:::warning 字段编号的坑
字段编号一旦发布，**不要轻易修改或删除**。如果确实不需要，可以将字段标记为 `reserved`：

```protobuf
message User {
    reserved 1, 3;  // 保留编号 1 和 3
    reserved "name";  // 保留字段名
    string new_name = 2;
}
```
:::

## 五、Kryo：Java 极速序列化

### 原理

Kryo 是专门为 Java 设计的序列化框架，核心优化：

1. **不使用反射**：直接操作对象内存
2. **预注册类**：跳过类名写入
3. **缓存优化**：复用 ByteBuffer

### 使用示例

```java
import com.esotericsoftware.kryo.Kryo;
import com.esotericsoftware.kryo.io.Input;
import com.esotericsoftware.kryo.io.Output;

// 创建 Kryo 实例（线程不安全，建议复用）
ThreadLocal<Kryo> kryoHolder = ThreadLocal.withInitial(() -> {
    Kryo kryo = new Kryo();
    kryo.setRegistrationRequired(false);  // 允许未注册的类
    kryo.setReferences(true);  // 支持循环引用
    return kryo;
});

// 序列化
public byte[] serialize(Object obj) {
    Kryo kryo = kryoHolder.get();
    Output output = new Output(1024, -1);
    kryo.writeClassAndObject(output, obj);
    byte[] bytes = output.toBytes();
    output.close();
    return bytes;
}

// 反序列化
public Object deserialize(byte[] bytes) {
    Kryo kryo = kryoHolder.get();
    Input input = new Input(new ByteArrayInputStream(bytes));
    Object obj = kryo.readClassAndObject(input);
    input.close();
    return obj;
}
```

### Kryo 注册优化

```java
// 预注册常用类（减少序列化体积）
Kryo kryo = new Kryo();
kryo.register(User.class, 1);      // ID 1
kryo.register(Order.class, 2);     // ID 2
kryo.register(Product.class, 3);   // ID 3

// KryoPool 复用
KryoPool pool = new KryoPool.Builder(() -> {
    Kryo kryo = new Kryo();
    kryo.setRegistrationRequired(false);
    return kryo;
}).build();

Kryo kryo = pool.borrow();
try {
    // 使用 kryo
} finally {
    pool.release(kryo);
}
```

### Kryo vs JSON 性能对比

| 指标 | JSON (Jackson) | Kryo |
|---|---|---|
| 序列化速度 | 1x（基准） | 3-5x |
| 反序列化速度 | 1x（基准） | 5-10x |
| 字节体积 | 100% | 50-60% |
| 跨语言 | ❌ | ❌ |

## 六、Hessian：Dubbo 默认之选

### 原理

Hessian 是 Caucho 公司出品的二进制序列化协议，设计目标是**简单、快速、紧凑**。

### Dubbo 中的使用

```yaml title="application.yml"
dubbo:
  protocol:
    serialization: hessian2
  provider:
    serialization: hessian2
  consumer:
    serialization: hessian2
```

### Hessian 使用示例

```java
import com.caucho.hessian.io.Hessian2Input;
import com.caucho.hessian.io.Hessian2Output;
import java.io.ByteArrayInputStream;
import java.io.ByteArrayOutputStream;

// 序列化
ByteArrayOutputStream bos = new ByteArrayOutputStream();
Hessian2Output out = new Hessian2Output(bos);
out.writeObject(user);
out.close();
byte[] bytes = bos.toByteArray();

// 反序列化
ByteArrayInputStream bis = new ByteArrayInputStream(bytes);
Hessian2Input in = new Hessian2Input(bis);
User parsed = (User) in.readObject();
in.close();
```

### Hessian 的特点

| 特点 | 说明 |
|---|---|
| **跨语言** | 支持 Java、C++、Python、PHP 等 |
| **简单** | API 简洁，易于使用 |
| **性能中等** | 比 JSON 快，但比 Protobuf 慢 |
| **Dubbo 默认** | Alibaba Dubbo 官方推荐 |

## 七、FST：最快的 Java 序列化

### 原理

FST（Fast Serialization）是 Java 平台最快的序列化库之一，基于 JDK 动态代理实现。

### 使用示例

```java
import com.hierynomus.fst.FSTConf;
import com.hierynomus.fst.FSTEncoder;

// 配置
FSTConf conf = new FSTConf();
conf.registerClass(User.class, Order.class);

// 序列化
FSTEncoder encoder = new FSTEncoder(conf);
byte[] bytes = encoder.encode(user);

// 反序列化
Object obj = encoder.decode(bytes, User.class);
```

:::warning FST 的兼容性问题
FST 依赖 JDK 内部实现，升级 JDK 版本可能导致序列化数据无法解析。使用前请充分测试兼容性。
:::

## 九、选型决策树

### 根据场景选择

```
需要序列化？
├── 对外 API / 需要可读性？
│   └── → JSON（Jackson）
│
├── 内部微服务 / 高性能？
│   ├── Java 内部？
│   │   └── → Kryo 或 FST
│   │
│   └── 跨语言？
│       ├── 追求极致性能？
│       │   └── → Protobuf
│       │
│       └── Dubbo 生态？
│           └── → Hessian
│
├── 大数据量传输？
│   └── → Protobuf + 压缩
│
└── 日志收集？
    └── → Logstash JSON 或 Protobuf
```

### 总结

| 场景 | 推荐方案 | 原因 |
|---|---|---|
| 对外 API | JSON | 可读性好，跨语言 |
| Spring Cloud 微服务 | JSON 或 Protobuf | 生态完善 |
| Dubbo 微服务 | Hessian | Dubbo 默认 |
| 内部高性能场景 | Kryo | Java 最快 |
| 需要跨语言 | Protobuf | 跨语言支持最好 |
| Redis 缓存 | JSON 或 Kryo | 看是否需要跨语言 |

## 面试实战

### 面试题 1：Protobuf 比 JSON 快多少？原理是什么？

**参考答案**：

**性能对比**：
- 序列化速度：Protobuf 比 JSON 快 **5-10 倍**
- 反序列化速度：Protobuf 比 JSON 快 **10-20 倍**
- 字节体积：Protobuf 只有 JSON 的 **30-50%**

**原理**：

1. **二进制编码**：JSON 是文本（ASCII），Protobuf 是二进制
   - 数字 `123`：JSON 需要 3 个字节 `123`，Protobuf 只需要 1-2 个字节

2. **字段编号代替字段名**：
   - JSON：`{"name": "zhang"}` 需要传输 `name` 这个字符串
   - Protobuf：字段 `name = 1`，传输数字 `1` 即可

3. **不需要解析字符串**：
   - JSON 解析需要识别 `:`, `,`, `"` 等符号
   - Protobuf 直接按类型读取字节

4. **预定义结构**：
   - Protobuf 编译时就知道了字段类型
   - JSON 运行时需要反射获取字段信息

### 面试题 2：Protobuf 的缺点是什么？

**参考答案**：

1. **不可读**：二进制格式，需要用工具查看，不利于调试
2. **需要预定义**：需要写 `.proto` 文件，增加了开发成本
3. **字段编号管理**：需要全局统一管理，避免冲突
4. **升级成本**：如果字段编号变了，需要兼容处理
5. **Java 场景学习曲线**：团队需要学习 Protobuf 语法

### 面试题 3：如何选择序列化方案？

**参考答案**：

选择序列化方案，主要考虑三个维度：

**1. 性能要求**：
- 高性能场景（内部 RPC）：Protobuf > Kryo > Hessian > JSON
- 一般场景：JSON 足够

**2. 跨语言需求**：
- 需要跨语言：Protobuf、MessagePack
- Java 内部：Kryo、FST

**3. 可读性需求**：
- 需要调试、可读：JSON
- 不需要可读：Protobuf

**实际建议**：
- 微服务内部：Protobuf（性能最优）
- 对外 API：JSON（可读性）
- Dubbo 生态：Hessian（官方默认）
- 不确定：JSON（最通用）

## 思考题

1. 如果序列化对象中包含 `LocalDateTime` 等日期类型，JSON 和 Protobuf 分别如何处理？性能差异有多大？
2. 在 RPC 调用中，序列化数据会被压缩传输吗？压缩和选择高性能序列化方案，哪个收益更大？
3. Kryo 虽然速度快，但它不支持跨语言。如果未来需要支持跨语言服务，重构成本有多大？

---

> [!TIP]
> 序列化不是银弹，但它是性能优化中最容易实现、收益最明显的手段之一。在做复杂的架构优化之前，先看看序列化能不能帮你省下 50% 的传输时间。
