# 操作系统性能优化

你有没有遇到过这种情况：代码层面一切正常，JVM 参数也调了，但服务就是跑不快。问题可能出在操作系统层面——CPU 调度、内存分配、I/O 调度，每一环都直接影响 Java 应用的性能表现。

这一节，我们从 OS 的视角来审视性能优化，把那些「看不见的」瓶颈逐一拆解。

---

## CPU 调度：线程与进程的爱恨情仇

### 进程 vs 线程：谁才是调度单位

先厘清一个基础概念：线程是 CPU 调度的最小单位，而不是进程。

```
进程（Process）
├── 独立的虚拟地址空间
├── 线程 A（调度单位）
├── 线程 B（调度单位）
└── 线程 C（调度单位）
```

在 Linux 中，线程和进程都使用 `task_struct` 表示，区别在于是否共享地址空间。这就是为什么 Java 中创建一个线程比创建一个进程「便宜」——虽然底层都是调用 clone()，但线程不需要分配独立的内存空间。

### 上下文切换：性能的无声杀手

当 CPU 从执行一个线程切换到另一个线程时，需要保存当前线程的上下文（寄存器、程序计数器、栈指针），然后加载新线程的上下文。这个过程就是「上下文切换」。

**上下文切换的开销**：
- 直接开销：保存/加载寄存器，约 1-5 微秒
- 间接开销：CPU 缓存失效，缓存命中率下降

```bash
# 查看系统上下文切换次数
vmstat 1

# 输出示例
procs -----------memory---------- ---swap-- -----io---- -system-- ------cpu-----
 r  b   swpd   free   buff  cache   si   so    bi    bo   in   cs us sy id wa st
 2  0      0 1234567  12345 234567    0    0     0     0 1000 5000 10  5 85  0  0
```

关注 `cs`（context switch）列，如果每秒上下文切换数万次甚至更多，说明系统调度压力很大。

**如何减少上下文切换**：

1. **减少线程数量**：线程数 = CPU 核心数 / (1 - 阻塞系数)
2. **避免锁竞争**：缩小锁粒度，减少线程阻塞
3. **使用协程**：JDK 21 的虚拟线程天然减少上下文切换

### CPU 亲和性：让线程固定在特定 CPU 上

CPU 亲和性（CPU Affinity）允许我们将进程或线程绑定到特定 CPU 核心，避免线程在核心之间迁移。

```bash
# 查看进程运行的 CPU
ps -eo pid,psr,comm | grep java

# 将 Java 进程绑定到 CPU 0-3
taskset -c 0-3 java -jar app.jar

# 查看当前进程的 CPU 亲和性
taskset -cp <pid>

# Java 代码中设置（使用 jexecutor 库或 JNI）
```

**为什么要绑定 CPU**：
- 避免上下文切换：线程固定在一个核心，减少切换
- 提高缓存命中率：L1/L2 缓存是核心私有的
- NUMA 场景：确保线程访问本地内存

```bash
# 使用 numactl 绑定到特定 NUMA 节点
numactl --cpunodebind=0 --membind=0 java -jar app.jar
```

:::tip
CPU 亲和性是一把双刃剑。在单进程场景下，绑定可以提升性能；但在多进程服务（如 Nginx）中，过度绑定可能影响负载均衡效果。
:::

## 内存管理：看不见的内存战争

### 内存分配器：glibc 的变迁

Linux 默认使用 glibc 的 ptmalloc2 分配器，但它的性能在多线程场景下存在瓶颈——不同线程的内存分配会竞争同一个 arena。

**主流内存分配器对比**：

| 分配器 | 使用者 | 特点 |
|---|---|---|
| ptmalloc2 | glibc 默认 | 多线程锁竞争严重 |
| jemalloc | Redis、Firefox | 多 arena 设计，减少锁竞争 |
| tcmalloc | Chrome、MongoDB | Thread-Local 缓存激进 |
| mimalloc | Microsoft | 微软出品，高性能、安全 |

查看当前使用的分配器：

```bash
# 查看进程加载的分配器
ldd /path/to/java | grep -E "jemalloc|tcmalloc"

# 查看系统默认分配器
ldd --version
```

对于 Java 应用，JVM 有自己的 GC 机制，但 JNI 调用的本地库（如 Netty、Redis 客户端）会直接使用系统分配器。如果你的应用大量使用 Netty，选择支持 jemalloc 的版本会有明显收益。

### 大页（HugePages）：减少 TLB Miss

TLB（Translation Lookaside Buffer）是 CPU 用于加速虚拟地址到物理地址转换的缓存。TLB 容量有限，如果程序访问的内存页过多，TLB Miss 的开销会累积。

**大页的作用**：将多个 4KB 页合并为 2MB 或 1GB 大页，减少 TLB 条目使用。

```bash
# 查看当前大页配置
cat /proc/meminfo | grep -i huge

# 启用大页（需要 root）
sysctl -w vm.nr_hugepages=256  # 分配 256 个 2MB 大页 = 512MB

# 永久配置
echo "vm.nr_hugepages=256" >> /etc/sysctl.conf
```

Java 应用使用大页需要配置 JVM 参数：

```bash
java -XX:+UseLargePages -XX:LargePageSizeInBytes=2m -jar app.jar
```

### Transparent HugePages（THP）：自动化的代价

THP 是 Linux 2.6.38 引入的特性，自动将 4KB 页面合并为大页。

**问题**：THP 的后台合并操作（khugepaged）会消耗 CPU，且合并时机不可控，可能导致性能抖动。

```bash
# 查看 THP 状态
cat /sys/kernel/mm/transparent_hugepage/enabled
cat /sys/kernel/mm/transparent_hugepage/defrag

# 关闭 THP（Oracle 官方建议用于数据库）
echo never > /sys/kernel/mm/transparent_hugepage/enabled
echo never > /sys/kernel/mm/transparent_hugepage/defrag

# 永久配置
echo "never" > /etc/rc.d/rc.local  # 或使用 systemd 配置
```

:::warning
Oracle 官方明确建议：对于 Oracle Database 生产环境，应关闭 THP。这是经过大量生产验证的最佳实践。
:::

### Swap 调优：内存真的够用吗

Swap 是磁盘上的一块空间，当物理内存不足时，OS 将不活跃的内存页交换到磁盘。

**问题**：一旦开始大量使用 Swap，磁盘 I/O 成为瓶颈，服务响应时间会急剧上升。

```bash
# 查看 Swap 使用情况
free -h
swapon -s

# vm.swappiness 参数（0-100）
# 值越高，越倾向于使用 Swap
# 值越低，越倾向于回收内存页而非 Swap
```

**生产环境建议**：

```bash
# 对于有足够内存的服务器，设置为 1-10
sysctl -w vm.swappiness=10

# 对于内存敏感型应用，可以设为 0
# 但注意：如果内存真的耗尽，没有 Swap 会触发 OOM Killer
sysctl -w vm.swappiness=1
```

## 磁盘 I/O 调度：SSD 时代的新命题

### I/O 调度算法演进

磁盘 I/O 调度器的核心任务是合并和排序 I/O 请求，减少磁头移动（在 HDD 时代尤为重要）。

**主流调度算法**：

| 调度器 | 适用场景 | 特点 |
|---|---|---|
| cfq | 通用 | 公平队列，但吞吐量低 |
| deadline | 数据库 | 写优先，减少延迟 |
| noop | SSD、高性能 | 最小化 CPU 开销 |
| mq-deadline | 多核 SSD | 多队列版本的 deadline |
| bfq | 桌面、媒体 | 保证带宽公平 |

```bash
# 查看当前调度器
cat /sys/block/sda/queue/scheduler

# 临时修改
echo noop > /sys/block/sda/queue/scheduler

# 永久修改（通过 udev 规则或内核参数）
```

### SSD 场景：noop 是最优解

SSD 没有机械结构，不存在磁头寻道时间，调度器的合并排序反而增加了不必要的开销。

**推荐配置**：

```bash
# SSD 使用 noop 或 mq-deadline
echo "noop" > /sys/block/sda/queue/scheduler

# 读取预取优化（SSD 场景可以关闭）
echo 0 > /sys/block/sda/queue/read_ahead_kb

# I/O 合并优化
echo 256 > /sys/block/sda/queue/nr_requests
```

## 文件系统：选对格式用对挂载

### 主流文件系统对比

| 文件系统 | 最大文件 | 最大容量 | 适用场景 |
|---|---|---|---|
| ext4 | 16TB | 1EB | 通用 Linux 默认 |
| xfs | 8EB | 8EB | 大文件、高并发 |
| btrfs | 16EB | 16EB | 高级特性（快照、校验） |

**选择建议**：
- 日志存储、数据库数据目录：xfs（高并发、大文件支持好）
- 一般用途：ext4（稳定、成熟）
- 需要快照功能：btrfs（但生产环境需谨慎）

### 挂载选项：细节决定性能

```bash
# /etc/fstab 示例
/dev/sda1 /data xfs defaults,noatime,nodiratime,logbufs=8 0 0

# 参数说明：
# noatime: 不更新文件访问时间（显著减少 I/O）
# nodiratime: 不更新目录访问时间
# logbufs: 日志缓冲区大小（xfs 专用）
```

:::tip noatime 的威力
对于频繁读取的文件（如类库 jar 包），关闭 atime 更新可以减少约 10% 的 I/O 操作。在高并发场景下，这个优化不可忽视。
:::

### 日志模式：data=ordered vs data=writeback

ext4 的日志模式影响数据安全性与性能：

| 模式 | 安全性 | 性能 | 说明 |
|---|---|---|---|
| data=ordered（默认） | 高 | 中 | 元数据写入后，数据再写入 |
| data=writeback | 中 | 高 | 仅保证元数据顺序 |
| data=journal | 最高 | 低 | 数据和元数据都写入日志 |

对于追求极致性能且对短暂崩溃有一定容忍度的场景，可以考虑 `data=writeback`：

```bash
# 临时修改
mount -o data=writeback /dev/sda1 /data

# 永久修改
echo "DATA=writeback" >> /etc/default/tmpfs  # Debian/Ubuntu
```

## NUMA：多路服务器的内存墙

### NUMA 拓扑查看

NUMA（Non-Uniform Memory Access）系统中，CPU 访问本地内存比访问其他 CPU 的内存快得多。

```bash
# 查看 NUMA 拓扑
numactl --hardware

# 输出示例
available: 2 nodes (0-1)
node 0 size: 32768 MB
node 1 size: 32768 MB
node 0 cpus: 0 1 2 3 4 5 6 7
node 1 cpus: 8 9 10 11 12 13 14 15
node 0 free: 16000 MB
node 1 free: 18000 MB
```

### NUMA 亲和性配置

```bash
# 查看进程的 NUMA 节点分布
numastat -p <pid>

# 将进程绑定到特定节点
numactl --cpunodebind=0 --membind=0 java -jar app.jar

# 允许进程自由分配，但优先本地
numactl --preferred=0 java -jar app.jar
```

**JVM 参数配置**：

```bash
# 启用 NUMA 感知
java -XX:+UseNUMA -XX:+AlwaysPreTouch -jar app.jar

# -XX:+AlwaysPreTouch：在启动时预分配所有堆内存
```

## ulimit 调优：被忽视的系统限制

### 核心参数

```bash
# 查看当前限制
ulimit -a

# 关键参数说明：
# open files: 单进程最大打开文件描述符数
# max user processes: 单用户最大进程数
# core file size: 核心转储文件大小
```

**常见问题**：
- Too many open files：文件描述符耗尽
- Cannot create new thread：进程数或线程数达到上限
- Out of memory：虚拟内存限制

### 生产环境配置

```bash
# /etc/security/limits.conf 添加
root soft nofile 655350
root hard nofile 655350
root soft nproc 655350
root hard nproc 655350
root soft core unlimited
root hard core unlimited

# * 表示所有用户，可根据需要指定具体用户
```

对于 Java 应用，文件描述符尤其重要——每个 Socket 连接、每个打开的文件都需要占用一个 fd：

```bash
# JVM 参数：确保使用足够的文件描述符
java -XX:+UnlockExperimentalVMOptions \
     -XX:+UseG1GC \
     -XX:MaxFDLimit=655350 \
     -jar app.jar
```

## 面试实战

### 面试题一：什么情况下会发生频繁的上下文切换？

**参考答案**：
>频繁上下文切换通常由以下原因导致：

>1. **线程数过多**：线程数远超 CPU 核心数，大量时间用于等待和切换。建议通过 `vmstat 1` 观察 cs 列。
>2. **锁竞争激烈**：多个线程争抢同一把锁，未获取锁的线程被挂起。可以通过 `pidstat -w 1` 观察上下文切换次数。
>3. **I/O 阻塞**：线程在等待 I/O 时被挂起，I/O 完成后再恢复。使用异步 I/O 可以减少这类切换。
>4. **定时器过多**：每个定时器到期都可能导致进程被唤醒，增加调度压力。

>排查工具：`vmstat`、`pidstat`、`/proc/<pid>/sched`

### 面试题二：如何排查 Swap 使用过多的问题？

**参考答案**：
>Swap 使用过多的排查步骤：

>1. **确认 Swap 情况**：`free -m`、`swapon -s` 查看 Swap 总量和使用量
>2. **定位进程**：`for i in $(cd /proc && ls -1 | grep "^[0-9]" | sort -n); do awk '/Swap:/{swap+=$2}END{print swap,$i}' /proc/$i/smaps 2>/dev/null; done | sort -rn | head -20` 找出 Swap 占用最多的进程
>3. **分析原因**：
>   - 内存泄漏：进程内存持续增长
>   - 内存配置不当：JVM 堆 + 直接内存超过物理内存
>   - 工作负载过大：正常情况下内存不足
>4. **解决方案**：增加物理内存、优化内存使用、调整 vm.swappiness

### 面试题三：NUMA 对 Java 应用的影响是什么？

**参考答案**：
>NUMA 对 Java 应用的影响主要体现在内存访问延迟上。

>现代 Java 堆通常跨越多个 NUMA 节点，GC 后的对象分配可能不在「本地」节点。启用 `-XX:+UseNUMA` 后，JVM 会尽量将对象分配在当前线程所在的 NUMA 节点上，减少远程内存访问。

>生产建议：
>- 对于 4 节点以上的服务器，强烈建议开启 NUMA 支持
>- 使用 `numactl` 将 Java 进程绑定到特定节点或节点组
>- 监控 `numastat` 确保内存分配符合预期
