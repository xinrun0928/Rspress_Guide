# mmap 与直接 I/O

Kafka 的高性能，离不开顺序写和零拷贝。但很多人不知道的是，Kafka 还大量使用了 mmap（内存映射）来提升写入性能。RocketMQ 同样如此。

那么，mmap 到底是什么？为什么它能提升 IO 性能？它有哪些坑需要避开？这一节，我们深入 mmap 的原理，对比普通 read/write，分析它的适用场景和潜在风险。

## 传统 IO 的问题

在理解 mmap 之前，我们先回顾一下传统 read/write 的 IO 流程：

```java
// 传统 read 操作的流程
FileInputStream fis = new FileInputStream("data.txt");
byte[] buffer = new byte[1024];
int bytesRead = fis.read(buffer);  // 从文件读取数据
```

```bash
# read() 的数据拷贝过程（4 次拷贝）
磁盘
  ↓ DMA 拷贝
内核缓冲区（Page Cache）
  ↓ CPU 拷贝
用户缓冲区（应用内存中的 buffer）
  ↓ CPU 拷贝
应用业务数据
```

```java
// 传统 write 操作的流程
FileOutputStream fos = new FileOutputStream("output.txt");
fos.write(buffer);  // 写入数据到文件
```

```bash
# write() 的数据拷贝过程（4 次拷贝）
应用业务数据
  ↓ CPU 拷贝
用户缓冲区
  ↓ CPU 拷贝
内核缓冲区（Page Cache）
  ↓ DMA 拷贝
磁盘
```

**问题在哪里？**

内核缓冲区（Page Cache）和用户缓冲区是两块独立的内存空间，数据在内核态和用户态之间来回拷贝。CPU 在这个过程中扮演了「搬运工」的角色，不仅要拷贝数据，还要完成上下文切换（用户态 → 内核态 → 用户态）。

## 虚拟内存基础：理解 mmap 的前提

mmap 的本质是**虚拟内存技术**的应用。要理解 mmap，先要理解虚拟内存的几个核心概念。

### 页表与地址映射

现代操作系统使用页表来管理虚拟地址到物理地址的映射：

```bash
# 虚拟地址空间 vs 物理地址空间
# 每个进程有自己的虚拟地址空间（32 位系统：0x00000000 ~ 0xFFFFFFFF）

虚拟地址空间（进程视角）          物理地址空间（硬件视角）
┌─────────────────────┐         ┌─────────────────────┐
│ 0x00000000          │         │ 0x00000000          │
│ 代码段               │         │ 物理内存页 0        │
├─────────────────────┤         ├─────────────────────┤
│ 0x00001000          │  ─────► │ 物理内存页 1        │
│ 数据段               │  页表    │                     │
├─────────────────────┤  映射    ├─────────────────────┤
│ 0x00002000          │  ─────► │ 物理内存页 2        │
│ 堆                   │         │                     │
├─────────────────────┤         ├─────────────────────┤
│ 0xBFFF0000          │         │ 物理内存页 N        │
│ 栈                   │         │                     │
└─────────────────────┘         └─────────────────────┘

# 页表记录：虚拟页号 → 物理页框号
# 虚拟地址 = 页号 × 页大小 + 页内偏移
```

页表由操作系统内核维护，每次地址访问都要查页表。为了加速，CPU 提供了 TLB（Translation Lookaside Buffer）来缓存最近使用的地址映射。

### TLB：地址转换的加速器

TLB 是 CPU 内置的高速缓存，存储最近使用的虚拟地址到物理地址的映射：

```bash
# TLB 命中 vs 未命中
# TLB 命中：地址转换只需要 1 个 CPU 周期
# TLB 未命中：需要访问内存中的页表，几十到上百个周期

CPU
  │
  ▼
┌─────────────┐
│    TLB      │  ◄── 高速缓存，查找虚拟页号 → 物理页框号
└─────────────┘
  │ 命中
  ▼
物理地址 ──► 缓存行

  │ 未命中
  ▼
内存页表 ──► 更新 TLB ──► 物理地址
```

### 缺页中断：Page Fault

当访问的虚拟页不在内存中时，会触发缺页中断（Page Fault），操作系统从磁盘加载对应数据：

```bash
# 缺页中断的处理流程
应用程序访问虚拟地址
        │
        ▼
    TLB 未命中
        │
        ▼
    页表中没有该虚拟页的映射
        │
        ▼
    触发 Page Fault（缺页中断）
        │
        ▼
    操作系统从磁盘读取数据到物理内存
        │
        ▼
    更新页表，建立映射关系
        │
        ▼
    返回应用程序继续执行
```

## mmap 系统调用：文件映射到内存

mmap 的核心思想是**「让文件像内存一样访问」**：

```c
// Linux mmap 系统调用
void *mmap(void *addr, size_t length, int prot, int flags, int fd, off_t offset);

// 参数说明：
// addr:      期望映射到的虚拟地址（通常为 NULL，由内核选择）
// length:    映射的长度
// prot:      保护标志（PROT_READ, PROT_WRITE, PROT_EXEC）
// flags:     映射类型（MAP_SHARED, MAP_PRIVATE, MAP_ANONYMOUS）
// fd:        文件描述符
// offset:    文件偏移量（必须是页对齐的）
```

### Java 中的 mmap 使用

```java
// 使用 FileChannel.map() 进行内存映射
FileChannel channel = new RandomAccessFile("data.bin", "rw").getChannel();
MappedByteBuffer buffer = channel.map(FileChannel.MapMode.READ_WRITE, 0, channel.size());

// 像访问内存一样读写文件
buffer.put(0, (byte) 1);      // 写入
byte b = buffer.get(0);       // 读取

// 修改会自动同步到磁盘（由操作系统决定何时刷盘）
```

### mmap 的内存布局

```bash
# mmap 后的内存布局
虚拟地址空间                      物理地址空间
┌─────────────────────┐         ┌─────────────────────┐
│ 0x00000000          │         │ 物理内存页 0        │
├─────────────────────┤         ├─────────────────────┤
│ 堆                   │         │ 物理内存页 1        │
│                     │         │ ┌─────────────────┐│
├─────────────────────┤         │ │ data.bin 文件   ││
│ mmap 区域            │◄───────►│ │ (映射到内存)    ││
│ 0x7f0000000000      │  页表    │ └─────────────────┘│
│ ┌─────────────────┐ │  映射    ├─────────────────────┤
│ │ mapped region   │ │         │ 物理内存页 2        │
│ │ (文件数据)       │ │         │                     │
│ └─────────────────┘ │         ├─────────────────────┤
└─────────────────────┘         └─────────────────────┘

# 关键点：文件数据直接映射到虚拟地址空间
# 应用读写这段内存，就是读写文件
```

## mmap vs 普通 read/write：核心区别

### 数据拷贝次数对比

**传统 read/write（4 次拷贝 + 2 次上下文切换）：**

```bash
# 读取场景
磁盘 ──DMA──► 内核缓冲区 ──CPU──► 用户缓冲区 ──CPU──► 应用

# 写入场景
应用 ──CPU──► 用户缓冲区 ──CPU──► 内核缓冲区 ──DMA──► 磁盘
```

**mmap（避免了内核到用户的拷贝）：**

```bash
# mmap 读取
磁盘 ──DMA──► 内核缓冲区（Page Cache）
                │
                │ 虚拟地址映射，无需 CPU 拷贝
                ▼
              应用直接访问

# mmap 写入
应用直接写入内核缓冲区（Page Cache）
                │
                │ 操作系统异步刷盘
                ▼
              磁盘
```

| 操作 | 传统 read/write | mmap |
|---|---|---|
| 读取 1 | DMA：磁盘 → 内核缓冲区 | DMA：磁盘 → 内核缓冲区 |
| 读取 2 | CPU：内核缓冲区 → 用户缓冲区 | 无（通过地址映射访问） |
| 写入 1 | CPU：用户缓冲区 → 内核缓冲区 | 直接写入内核缓冲区 |
| 写入 2 | DMA：内核缓冲区 → 磁盘 | DMA：内核缓冲区 → 磁盘（异步） |
| 上下文切换 | 2 次（用户态 ↔ 内核态） | 写入时 0 次（延迟写入） |

### 性能对比：什么时候 mmap 更快

```java
// 测试场景：顺序读写 1GB 文件
// 传统 read/write vs mmap

// 传统方式
long start = System.currentTimeMillis();
FileInputStream fis = new FileInputStream("largefile.bin");
byte[] buffer = new byte[8192];
while (fis.read(buffer) != -1) {
    // 处理数据
}

// mmap 方式
long start = System.currentTimeMillis();
FileChannel channel = new FileInputStream("largefile.bin").getChannel();
MappedByteBuffer buffer = channel.map(MapMode.READ_ONLY, 0, channel.size());
while (buffer.hasRemaining()) {
    buffer.get();  // 像内存访问一样
}
```

```bash
# 性能测试结果（顺序读写大文件）
# 文件大小：1GB
# 机器配置：SSD, 16GB RAM

┌────────────────────────────────────────────────────┐
│ 传统 read/write  │ mmap                              │
├────────────────────────────────────────────────────┤
│ 读取耗时: 2.3s    │ 读取耗时: 0.8s                    │
│                  │ （命中 Page Cache 更快）          │
│ 写入耗时: 2.8s    │ 写入耗时: 1.5s                    │
│                  │ （减少了一次 CPU 拷贝）           │
└────────────────────────────────────────────────────┘
```

### mmap 的陷阱：小文件性能可能更差

mmap 在处理小文件时可能反而更慢，原因是**页表建立的开销**：

```bash
# mmap 的额外开销
# 1. 建立虚拟地址到物理页的映射（页表项）
# 2. 首次访问触发缺页中断（Page Fault）
# 3. 从磁盘加载数据到物理内存

# 对于小文件，这些开销可能超过节省的拷贝收益

┌────────────────────────────────────────────────────┐
│ 文件大小       │ 推荐方式       │ 原因              │
├────────────────────────────────────────────────────┤
│ < 1 页 (4KB)   │ 普通 read()   │ mmap 开销更大     │
│ 4KB ~ 1MB      │ mmap          │ 收益明显          │
│ > 1MB          │ mmap          │ 顺序访问优势明显   │
│ 超大文件       │ mmap + 分段   │ 避免单次映射过大   │
└────────────────────────────────────────────────────┘
```

## mmap 的风险与坑

mmap 虽然性能优秀，但有几个坑需要特别注意。

### 风险一：OOM Killer

mmap 映射的文件数据会占用物理内存（通过 Page Cache）。当物理内存紧张时，操作系统可能会杀死占用大量内存的进程：

```bash
# /var/log/messages 或 dmesg 中的 OOM Killer 日志
[1234567.890123] Out of memory: Kill process 12345 (java) score 888
[1234567.890124] Killed process 12345 (java) total-vm:52345678kB

# 原因：mmap 映射的文件过大，Page Cache 占用过多内存
```

```java
// 危险代码：映射过大的文件
FileChannel channel = new RandomAccessFile("hugefile.bin", "rw").getChannel();
MappedByteBuffer buffer = channel.map(MapMode.READ_WRITE, 0, 100 * 1024 * 1024 * 1024L);
// 100GB 的映射，即使不访问也会占用物理内存

// 建议：分段映射
long fileSize = channel.size();
int segmentSize = 1024 * 1024 * 1024;  // 1GB 一段
for (long offset = 0; offset < fileSize; offset += segmentSize) {
    long remaining = fileSize - offset;
    int size = (int) Math.min(remaining, segmentSize);
    MappedByteBuffer segment = channel.map(MapMode.READ_WRITE, offset, size);
    // 处理这段数据
}
```

### 风险二：多进程同步问题

多个进程映射同一个文件时，写入的同步需要额外处理：

```bash
# 进程 A 和进程 B 同时 mmap 同一个文件
# 操作系统不保证写入的可见性，需要 fsync/msync

# 进程 A 写入
buffer.put(0, (byte) 1);

// 进程 B 读取（可能还是旧值）
byte b = buffer.get(0);  // 不一定是 1

# 必须调用 msync() 才能保证其他进程看到最新数据
```

```java
// 正确的同步写法
MappedByteBuffer buffer = channel.map(MapMode.READ_WRITE, 0, size);

// 写入完成后，强制同步到磁盘
((DirectBuffer) buffer).force();
```

### 风险三：脏页刷新时机不确定

mmap 写入的数据何时刷盘，由操作系统决定：

```java
// mmap 写入后的刷盘时机
buffer.put(0, (byte) 1);  // 写入内存

// 数据可能：
// 1. 立即刷盘（高并发下不会）
// 2. 等待脏页回收（几分钟到几十分钟）
// 3. 进程崩溃时丢失（取决于系统策略）
```

```bash
# Linux 脏页刷新策略
# /proc/sys/vm/dirty_writeback_centisecs: 多久检查一次脏页（厘秒）
# /proc/sys/vm/dirty_expire_centisecs: 脏页多久算过期（厘秒）
# /proc/sys/vm/dirty_ratio: 脏页占比多少时阻塞写入

# 调优建议（数据库场景）
echo 500 > /proc/sys/vm/dirty_writeback_centisecs
echo 500 > /proc/sys/vm/dirty_expire_centisecs
```

### 风险四：内存泄漏

如果忘记 unmap，大文件映射会导致内存泄漏：

```java
// 危险：每次调用 map 都会创建新的映射
for (int i = 0; i < 100000; i++) {
    MappedByteBuffer buffer = channel.map(MapMode.READ_WRITE, 0, size);
    // 没有调用 unmap()
}

// 正确做法：使用 try-with-resources 或手动释放
try (FileChannel channel = new RandomAccessFile(file, "rw").getChannel()) {
    MappedByteBuffer buffer = channel.map(MapMode.READ_WRITE, 0, size);
    // 使用 buffer
}  // channel 关闭时自动 unmap
```

## mmap 适用场景

### 场景一：进程间通信（IPC）

Linux 共享内存使用 mmap 实现：

```c
// 进程 A：创建共享内存
int fd = shm_open("/my_shm", O_CREAT | O_RDWR, 0666);
ftruncate(fd, 1024 * 1024);  // 1MB
void *ptr = mmap(NULL, 1024 * 1024, PROT_READ | PROT_WRITE,
                 MAP_SHARED, fd, 0);

// 进程 B：打开同一块共享内存
int fd = shm_open("/my_shm", O_RDWR, 0666);
void *ptr = mmap(NULL, 1024 * 1024, PROT_READ | PROT_WRITE,
                 MAP_SHARED, fd, 0);

// 现在两个进程可以像访问内存一样共享数据
```

```bash
# 查看共享内存
ipcs -m

# 共享内存的优势：
# 1. 零拷贝：无需内核中转
# 2. 低延迟：直接内存访问
# 3. 简单：像操作内存一样
```

### 场景二：大文件顺序读写

RocketMQ 的消息存储使用 mmap：

```java
// RocketMQ 的 MappedFile 使用 mmap
public class MappedFile {
    private final MappedByteBuffer mappedByteBuffer;
    
    public MappedFile(String filePath, int fileSize) {
        FileChannel fileChannel = new RandomAccessFile(filePath, "rw").getChannel();
        this.mappedByteBuffer = fileChannel.map(
            FileChannel.MapMode.READ_WRITE, 0, fileSize);
    }
    
    public boolean appendMessage(final ByteBuffer buffer) {
        int currentPos = this.position.getAndAdd(buffer.remaining());
        this.mappedByteBuffer.position(currentPos);
        this.mappedByteBuffer.put(buffer);
        return true;
    }
}
```

### 场景三：数据库索引文件

LMDB（Lightning Memory-Mapped Database）使用 mmap：

```bash
# LMDB 的设计哲学：
# 1. mmap 整个数据库文件到内存
# 2. B+ 树直接在 mmap 区域操作
# 3. 操作系统负责处理 Page Fault
# 4. 写入通过 COW（Copy On Write）保证原子性

# LMDB 的优势：
# - 极简实现，代码量少
# - 零拷贝读取
# - 利用操作系统 Page Cache
# - 写入原子性由 COW 保证
```

## 直接 I/O（Direct I/O）

### 什么是 Direct I/O

Direct I/O 绕过 Page Cache，直接访问磁盘：

```c
// 打开文件时指定 O_DIRECT 标志
int fd = open("/path/to/file", O_RDONLY | O_DIRECT);
```

```java
// Java 中使用 Direct I/O（需要 JNI 或库支持）
// 目前 JDK 原生不支持 O_DIRECT，需要使用 JNA 或第三方库

// 示例：使用 jniosd 库
ODirectFileChannel channel = ODirectFileChannel.open("/path/to/file");
ByteBuffer buffer = ByteBuffer.allocateDirect(4096);
channel.read(buffer, 0);  // 直接读取磁盘，绕过 Page Cache
```

### Direct I/O vs Page Cache

```bash
# Page Cache 模式（默认）
应用 ──► Page Cache（操作系统缓存）──► 磁盘
# 优点：读取过的数据可能命中缓存
# 缺点：写入先到 Page Cache，依赖系统刷盘策略

# Direct I/O 模式
应用 ──► 磁盘
# 优点：应用完全控制数据落盘时机
# 缺点：无法利用 Page Cache，需要自己实现缓存
```

| 特性 | Page Cache | Direct I/O |
|---|---|---|
| 数据缓存 | 操作系统自动管理 | 应用自行实现 |
| 读取性能 | 命中缓存时极快 | 每次都从磁盘读取 |
| 写入性能 | 批量合并优化 | 每次都直接写入 |
| 一致性 | 依赖系统刷盘 | 应用完全控制 |
| 适用场景 | 通用场景、文件共享 | 数据库、专用存储 |

### MySQL 的 Direct I/O

MySQL 在某些场景下使用 Direct I/O：

```bash
# MySQL 配置 InnoDB 使用 Direct I/O
# innodb_flush_method = O_DIRECT

# 适用场景：
# 1. 有自己的 Buffer Pool，不需要 Page Cache
# 2. 数据文件和日志文件分离
# 3. 确保数据严格持久化

# 不适用场景：
# 1. 小表频繁访问（无法利用缓存）
# 2. 混合读写场景（Page Cache 读友好）
```

### Hybrid 方案：Page Cache + Direct I/O

现代存储系统通常采用混合策略：

```bash
# 热数据走 Page Cache，冷数据走 Direct I/O

┌─────────────────────────────────────────────────────┐
│                    Hybrid I/O                        │
├─────────────────────────────────────────────────────┤
│ 热数据（频繁访问）  ──► Page Cache ──► 命中即返回    │
│ 冷数据（偶尔访问）  ──► Direct I/O ──► 绕过缓存     │
│ 脏数据（需要持久化）─► fsync/msync ──► 强制刷盘     │
└─────────────────────────────────────────────────────┘
```

```java
// Hybrid 场景示例：Elasticsearch
// 1. 写入：先写 Page Cache（利用顺序写优化）
// 2. 刷盘：定期调用 fsync 持久化
// 3. 读取：优先读 Page Cache，未命中则 Direct I/O

// RocketMQ 的 Hybrid 策略
// 1. 小文件：mmap（利用 Page Cache）
// 2. 大文件：Direct I/O（避免占用过多内存）
```

## mmap 与零拷贝的结合

mmap 是实现零拷贝的重要手段，配合 sendfile 可以实现更高效的传输：

```java
// 场景：静态文件下载服务器

// 方案 1：普通 read/write
while ((bytesRead = fis.read(buffer)) != -1) {
    os.write(buffer, 0, bytesRead);
}
// 拷贝：磁盘 → 内核缓冲区 → 用户缓冲区 → Socket 缓冲区 → 网卡
// 4 次拷贝

// 方案 2：mmap + write
while ((bytesRead = fis.getChannel().map(...)) != null) {
    os.write(buffer);
}
// 拷贝：磁盘 → 内核缓冲区 → Socket 缓冲区 → 网卡
// 3 次拷贝（节省了用户缓冲区拷贝）

// 方案 3：sendfile（完全内核态）
fis.getChannel().transferTo(0, fileSize, socketChannel);
// 拷贝：磁盘 → 内核缓冲区 → 网卡
// 2 次拷贝
```

### 优秀回答示例

**问题：mmap 有什么风险？**

> mmap 有三个主要风险：
>
> 第一，OOM Killer。因为 mmap 的数据会进入 Page Cache，占用物理内存。当内存紧张时，进程可能被杀掉。解决方案是控制映射大小，避免映射过大的文件。
>
> 第二，多进程同步问题。多个进程映射同一文件时，写入的可见性需要额外处理，必须调用 msync 才能保证其他进程看到最新数据。
>
> 第三，脏页刷新时机不确定。mmap 写入的数据何时落盘由操作系统决定，不是立即刷盘。如果需要强持久化保证，必须调用 force() 或 fsync()。
>
> 我在项目中用 mmap 做日志写入时就踩过这个坑——服务器异常重启后，日志丢失了一部分。后来加了 force() 调用才解决。
