# Netty I/O 模型

Tomcat 和 Jetty 都支持 NIO，Undertow 也有出色的性能表现。但为什么在 Java 网络通信领域，Netty 成了事实标准？

Dubbo 用它，gRPC 用它，RocketMQ 用它，Elasticsearch 用它。甚至 Android 的 Square OkHttp 团队，在设计新版本时也在借鉴 Netty 的设计理念。

这一节，我们深入 Netty 的核心设计，看看它凭什么成为 Java 网络通信的「标准答案」。

---

## 为什么是 Netty

在回答这个问题之前，我们先看看 Java NIO 有多难用。

### Java NIO 的真实复杂度

你以为 NIO 的代码是这样的：

```java
// 想象中的 NIO
Selector selector = Selector.open();
channel.register(selector, SelectionKey.OP_READ);
selector.select();
```

实际上，生产级的 NIO 代码是这样的：

```java
// 真实的 NIO：需要处理各种边界情况
public class NioServer {
    private Selector selector;
    private ServerSocketChannel serverChannel;
    private Map<SocketChannel, List<byte[]>> dataCache = new ConcurrentHashMap<>();
    
    public void start() throws IOException {
        serverChannel = ServerSocketChannel.open();
        serverChannel.socket().bind(new InetSocketAddress(8080));
        serverChannel.configureBlocking(false);
        selector = Selector.open();
        serverChannel.register(selector, SelectionKey.OP_ACCEPT);
        
        while (true) {
            int ready = selector.select();
            if (ready == 0) continue;
            
            Set<SelectionKey> keys = selector.selectedKeys();
            Iterator<SelectionKey> iter = keys.iterator();
            
            while (iter.hasNext()) {
                SelectionKey key = iter.next();
                iter.remove();  // 必须手动移除！
                
                try {
                    if (key.isAcceptable()) {
                        handleAccept(key);
                    } else if (key.isReadable()) {
                        handleRead(key);  // 还要处理半包问题
                    } else if (key.isWritable()) {
                        handleWrite(key);
                    }
                } catch (IOException e) {
                    key.cancel();
                    close(key.channel());
                }
            }
        }
    }
    
    private void handleRead(SelectionKey key) throws IOException {
        SocketChannel channel = (SocketChannel) key.channel();
        ByteBuffer buffer = ByteBuffer.allocate(1024);
        int read = channel.read(buffer);
        
        if (read == -1) {
            channel.close();
            return;
        }
        
        // 半包问题：这次读到的可能不是完整消息
        // 需要自己维护读取缓冲区
        byte[] data = new byte[read];
        buffer.flip();
        buffer.get(data);
        
        // 粘包问题：多条消息粘在一起
        // 需要自己拆包
        List<byte[]> messages = splitPackages(data);
        for (byte[] msg : messages) {
            processMessage(msg);
        }
    }
    
    // 还要实现拆包逻辑、编解码、异常处理、断连重连...
}
```

**Java NIO 的问题清单：**

| 问题 | 描述 | 解决方案复杂度 |
|---|---|---|
| ByteBuffer 固定大小 | 无法动态扩容 | 需要自己实现动态扩展 |
| 半包问题 | 一次 read 可能只读了半个包 | 需要维护读取缓冲区 |
| 粘包问题 | 多次 read 可能读了一整个包 + 半包 | 需要实现拆包逻辑 |
| 线程模型 | 单线程处理不了高并发 | 需要自己实现 Reactor |
| 异常处理 | 断连、异常需要处理 | 需要完善的状态管理 |
| 可扩展性 | 业务逻辑和 IO 耦合 | 需要抽象分层 |

### Netty 如何解决这些问题

```java
// Netty 的代码：简洁、优雅、功能完善
public class NettyServer {
    public static void main(String[] args) {
        EventLoopGroup bossGroup = new NioEventLoopGroup(1);
        EventLoopGroup workerGroup = new NioEventLoopGroup();
        
        try {
            ServerBootstrap bootstrap = new ServerBootstrap();
            bootstrap.group(bossGroup, workerGroup)
                    .channel(NioServerSocketChannel.class)
                    .childHandler(new ChannelInitializer<SocketChannel>() {
                        @Override
                        protected void initChannel(SocketChannel ch) {
                            // Pipeline 模式，业务逻辑和 IO 解耦
                            ch.pipeline()
                                .addLast(new LengthFieldBasedFrameDecoder(...))
                                .addLast(new MyDecoder())
                                .addLast(new MyBusinessHandler());
                        }
                    })
                    .option(ChannelOption.SO_BACKLOG, 128)
                    .childOption(ChannelOption.SO_KEEPALIVE, true);
            
            ChannelFuture f = bootstrap.bind(8080).sync();
            f.channel().closeFuture().sync();
        } finally {
            bossGroup.shutdownGracefully();
            workerGroup.shutdownGracefully();
        }
    }
}
```

**Netty 的五大核心优势：**

1. **Pipeline 模式**：业务逻辑和 IO 处理解耦
2. **内置编解码**：解决粘包半包问题
3. **ByteBuf 池化**：高性能 buffer 管理
4. **EventLoop 线程模型**：开箱即用的 Reactor
5. **丰富的协议支持**：HTTP、WebSocket、gRPC 等

## Reactor 模式：Netty 的架构基石

### 什么是 Reactor 模式

Reactor 模式是一种**事件驱动**的并发处理模式。传统的 BIO 是「一个连接一个线程」，而 Reactor 是「一个线程处理多个连接」。

**传统 BIO 的问题：**

```java
// 一个线程处理一个连接
ServerSocket server = new ServerSocket(8080);
while (true) {
    Socket socket = server.accept();  // 阻塞
    new Thread(() -> {
        try {
            // 连接建立后的所有操作
            InputStream in = socket.getInputStream();
            OutputStream out = socket.getOutputStream();
            
            byte[] buffer = new byte[1024];
            while (true) {
                int len = in.read(buffer);  // 阻塞
                if (len == -1) break;
                // 处理请求
            }
        } catch (IOException e) {}
    }).start();
}

// 问题：
// 10000 个连接 = 10000 个线程
// 每个线程栈 1MB = 10GB 内存
// 线程上下文切换 = CPU 大量时间在调度而非计算
```

**Reactor 的解决思路：**

```java
// 核心：单线程（多线程）监听多个连接，只处理就绪的连接
// IO 多路复用：select/poll/epoll

Selector selector = Selector.open();
server.register(selector, SelectionKey.OP_ACCEPT);

while (true) {
    int ready = selector.select();  // 阻塞，直到有连接就绪
    
    Set<SelectionKey> keys = selector.selectedKeys();
    for (SelectionKey key : keys) {
        if (key.isAcceptable()) {
            // 有新连接
            ServerSocketChannel server = (ServerSocketChannel) key.channel();
            SocketChannel client = server.accept();
            client.register(selector, SelectionKey.OP_READ);
        } else if (key.isReadable()) {
            // 有数据可读（一个线程处理所有连接的读事件）
            handleRead(key);
        }
    }
    keys.clear();
}

// 优势：单线程可以处理数万连接
// 10000 连接只需要 1 个线程，内存占用从 10GB 降到几 MB
```

### Reactor 模式的三种形态

**形态一：单 Reactor 单线程**

```java
// Reactor + Handler 都在一个线程
class SingleReactorSingleThread {
    private Selector selector;
    
    public void run() {
        selector.select();
        Set<SelectionKey> keys = selector.selectedKeys();
        for (SelectionKey key : keys) {
            dispatch(key);  // 所有操作都在主线程
        }
    }
    
    private void dispatch(SelectionKey key) {
        // Accept 事件
        if (key.isAcceptable()) {
            accept(key);
        }
        // Read 事件
        if (key.isReadable()) {
            read(key);      // 读取数据
            process(key);   // 处理业务  ← 问题：业务处理阻塞主线程
        }
    }
}

// 适用场景：业务处理极快的轻量级服务
// 问题：业务逻辑和 IO 混在一起，无法利用多核 CPU
```

**形态二：单 Reactor 多线程**

```java
// Reactor 在主线程，Handler 在线程池
class SingleReactorMultiThread {
    private Selector selector;
    private ExecutorService workers = Executors.newFixedThreadPool(8);
    
    public void run() {
        selector.select();
        for (SelectionKey key : selector.selectedKeys()) {
            if (key.isAcceptable()) {
                accept(key);
            } else if (key.isReadable()) {
                // IO 读取在主线程
                read(key);
                // 业务处理在线程池
                workers.execute(() -> process(key));  // ← 改进：业务处理异步化
            }
        }
    }
}

// 适用场景：业务处理较重的中型服务
// 问题：Reactor 仍是瓶颈（单线程处理所有 IO）
```

**形态三：主从 Reactor（Netty 采用）**

```java
// 主 Reactor 处理 Accept，从 Reactor 处理 IO
class MainSubReactor {
    // BossGroup：主 Reactor，处理连接建立
    NioEventLoopGroup bossGroup = new NioEventLoopGroup(1);
    
    // WorkerGroup：从 Reactor，处理读写事件
    NioEventLoopGroup workerGroup = new NioEventLoopGroup();
    
    public void start() {
        ServerBootstrap bootstrap = new ServerBootstrap();
        bootstrap.group(bossGroup, workerGroup)  // ← 主从分离
                 .channel(NioServerSocketChannel.class)
                 .childHandler(new ChannelHandler());
    }
}

// 工作流程：
// Boss 线程：接收连接 → 注册到 Worker → 继续接收
// Worker 线程：处理读写 → 处理业务 → 响应

// 优势：
// 1. 连接建立和 IO 处理完全分开
// 2. 多个 Worker 线程处理 IO，充分利用多核
// 3. 连接处理不会被 IO 阻塞
```

Netty 的线程模型基于主从 Reactor，如下图所示：

## Netty 核心组件

### Channel：Socket 的抽象

Channel 是 Netty 对网络套接字的抽象：

```java
// Channel 的核心方法
public interface Channel extends AttributeMap, ChannelOutboundInvoker, Comparable<Channel> {
    ChannelId id();                    // 唯一标识
    EventLoop eventLoop();             // 绑定的 EventLoop
    ChannelPipeline pipeline();         // 绑定的 Pipeline
    ChannelFuture bind(SocketAddress localAddress);   // 绑定地址
    ChannelFuture connect(SocketAddress remoteAddress); // 连接
    ChannelFuture write(Object msg);   // 写入数据
    ChannelFuture close();             // 关闭连接
}

// 常用的 Channel 类型
// NioSocketChannel: TCP 客户端/服务器 SocketChannel
// NioServerSocketChannel: TCP 服务器 ServerSocketChannel
// NioDatagramChannel: UDP
// EpollSocketChannel: Linux epoll（性能更好）
// KQueueSocketChannel: macOS kqueue
```

### EventLoop：线程与事件的绑定

EventLoop 是 Netty 处理 IO 事件的核心组件：

```java
// EventLoop 的实现
public class NioEventLoop extends SingleThreadEventLoop {
    private final Selector selector;
    private final TaskQueue taskQueue;
    
    @Override
    protected void run() {
        for (;;) {
            try {
                // 1. 监听 IO 事件（select）
                int ready = selectStrategy.calculateStrategy(selectSupplier, hasTasks());
                if (ready == SelectStrategy.CONTINUE) {
                    continue;
                }
                
                // 2. 处理 IO 事件
                processSelectedKeys();
                
                // 3. 处理异步任务（保证及时响应）
                runAllTasks();
            } catch (Exception e) {
                // 异常处理
            }
        }
    }
    
    // EventLoop 绑定一个线程，处理多个 Channel 的事件
    // 这是 Netty 实现高性能的关键：无锁化设计
}
```

**EventLoop 和 Channel 的绑定关系：**

```java
// 绑定过程
// 1. Server 启动：Boss EventLoop 绑定 ServerSocketChannel
// 2. 连接建立：Boss EventLoop 把 SocketChannel 注册到某个 Worker EventLoop
// 3. 后续处理：该 SocketChannel 永远由这个 Worker EventLoop 处理

// 关键保证：Channel 和 EventLoop 的绑定是固定的
// 同一个 Channel 的所有事件都在同一个线程处理
// 无需加锁，保证线程安全
```

### ChannelPipeline：责任链模式

Pipeline 是 Netty 最精妙的设计之一：

```java
// Pipeline 结构
ChannelPipeline pipeline = channel.pipeline();

// Pipeline 包含多个 ChannelHandler，形成责任链
// Inbound（入站）：数据从网络到应用
// Outbound（出站）：数据从应用到网络

// 典型的 Pipeline 配置
pipeline.addLast("decoder", new ByteToMessageDecoder());     // Inbound
pipeline.addLast("encoder", new MessageToByteEncoder());     // Outbound
pipeline.addLast("handler", new MyBusinessHandler());       // Inbound

// 数据流向
// Inbound:  Socket → decoder → handler → 业务
// Outbound: 业务 → encoder → Socket
```

```bash
# Pipeline 示意图
                    Inbound 方向
                    ─────────►
            ┌─────────────────────────────────┐
            │        ChannelPipeline          │
┌───────────┼─────────────────────────────────┼───────────┐
│           │                                 │           │
│     Head  │ ◄────── handler1 ──────►  Tail │           │
│  Context  │    handler2                      │  Context  │
│           │    handler3                      │           │
│           │                                 │           │
└───────────┼─────────────────────────────────┼───────────┘
            │                                 │
            ▼                                 ▼
            │                             ▲
            │         Outbound 方向       │
            │         ◄────────────       │
            └─────────────────────────────────┘

# Handler 的执行顺序
# Inbound: head → handler1 → handler2 → handler3 → tail
# Outbound: tail → handler3 → handler2 → handler1 → head
```

### ByteBuf：Netty 的 ByteBuffer

Java NIO 的 ByteBuffer 有几个严重问题：

```java
// NIO ByteBuffer 的问题
ByteBuffer buffer = ByteBuffer.allocate(1024);

// 问题 1：只有 position，没有 readerIndex 和 writerIndex
// 读写切换需要 flip()
buffer.put(data);     // 写入
buffer.flip();       // 切换模式
buffer.get(result);   // 读取

// 问题 2：无法动态扩容
// 缓冲区满了就没法再写

// 问题 3：需要手动回收
// 没有引用计数，容易内存泄漏
```

Netty 的 ByteBuf 完美解决了这些问题：

```java
// ByteBuf 的优势
ByteBuf buffer = Unpooled.buffer(1024);

// 读写分离：readerIndex 和 writerIndex
buffer.writeBytes(data);    // writerIndex 前移
buffer.readBytes(result);   // readerIndex 前移

// 可重置：读写完成后可以 reset
buffer.markReaderIndex();   // 标记当前位置
buffer.readBytes(temp);    // 读取部分数据
buffer.resetReaderIndex(); // 重置到标记位置

// 可扩容：自动调整容量
buffer.writeBytes(largeData);  // 超出容量时自动扩容

// 引用计数：内存池回收
ByteBuf buffer = ctx.alloc().buffer();
buffer.release();  // 引用计数减一，归还内存池
```

## ByteBuf 池化设计：减少 GC

### 为什么要池化

Java 的 ByteBuffer 每次 allocate 都会触发 GC：

```java
// 普通 ByteBuffer 的问题
ByteBuffer buffer = ByteBuffer.allocate(1024);
// 每秒处理 10 万请求 = 每秒分配 10 万个 ByteBuffer
// GC 压力巨大，Stop-the-World 频繁

// 池化后的 ByteBuf
ByteBuf buffer = ByteBufAllocator.DEFAULT.buffer(1024);
// 从内存池获取，用完后归还
// 复用已有内存，减少 GC
```

### Netty 的内存池架构

```java
// PooledByteBufAllocator：池化分配器
public class PooledByteBufAllocator extends AbstractByteBufAllocator {
    private final PoolArena<?>[] directArenas;  // 堆外内存池
    private final PoolArena<?>[] heapArenas;     // 堆内存池
    
    // Arena：将内存分成多个子池，减少竞争
    // 每个 Arena 独立管理，减少线程间的竞争
    
    @Override
    protected ByteBuf newDirectBuffer(int initialCapacity, int maxCapacity) {
        PoolThreadCache cache = currentThreadCache();
        PoolArena<?> arena = cache.leastUsedArena(...);
        return arena.allocate(cache, initialCapacity, maxCapacity);
    }
}

// PoolArena：内存池的核心组件
class PoolArena {
    // 内存池的核心结构
    // TinySubpagePools: 管理 < 512 字节的小内存块
    // SmallSubpagePools: 管理 512~8KB 的内存块
    // PoolChunkList: 管理不同大小的 PoolChunk
    
    // PoolChunk：实际分配内存的单位
    // 每个 PoolChunk 64MB（16 个 Page，每个 Page 4KB）
}
```

### 内存池的分配算法

```bash
# Netty 内存池的分配策略

# 1. 小内存（< 512B）：TinySubpagePools
#    直接从缓存的 PoolPage 分配
#    避免系统调用

# 2. 中等内存（512B ~ 8KB）：SmallSubpagePools
#    从预分配的 PoolPage 中切分

# 3. 大内存（> 8KB）：PoolChunk
#    不足 8KB 对齐的部分会浪费

┌─────────────────────────────────────────────────────┐
│                 PoolChunk (64MB)                     │
│  ┌─────────┬─────────┬─────────┬─────────┐           │
│  │ Page 4KB│ Page 4KB│ Page 4KB│ Page 4KB│  ... 16个 │
│  └─────────┴─────────┴─────────┴─────────┘           │
│                                                      │
│  小内存请求：直接在 Page 内切分                        │
│  ┌────┬────┬────┬────┬────┬────┬────┐               │
│  │ 16B│ 32B│ 64B│128B│256B│512B│ 剩余 │               │
│  └────┴────┴────┴────┴────┴────┴────┘               │
└─────────────────────────────────────────────────────┘
```

### 内存泄漏的防范

```java
// ByteBuf 泄漏的常见原因
ByteBuf buf = ctx.alloc().buffer();
// ... 使用 buffer
// 如果忘记 release()，就会泄漏

// Netty 提供的泄漏检测级别
// 1. DISABLED：禁用检测（生产环境，性能最好）
// 2. SIMPLE：抽样检测（默认，生产环境推荐）
// 3. ADVANCED：详细检测（测试环境）
// 4. PARANOID：每次分配都检测（调试环境）

// 建议配置
ServerBootstrap bootstrap = new ServerBootstrap();
bootstrap.option(ChannelOption.ALLOCATOR, 
    new PooledByteBufAllocator(PooledByteBufAllocator.DEFAULT.directArenaCount(),
                               PooledByteBufAllocator.DEFAULT.pageSize(),
                               PooledByteBufAllocator.DEFAULT.maxOrder(),
                               PooledByteBufAllocator.DEFAULT.tinyCacheSize(),
                               PooledByteBufAllocator.DEFAULT.smallCacheSize(),
                               PooledByteBufAllocator.DEFAULT.normalCacheSize(),
                               PooledByteBufAllocator.DEFAULT.useCacheForAllThreads(),
                               // 启用泄漏检测
                               ResourceLeakDetector.Level.SIMPLE));
```

## 零拷贝在 Netty 中的实现

Netty 的零拷贝和 Linux 的零拷贝不同，它更侧重于**减少应用层的数据拷贝**。

### 四种零拷贝技术

**技术一：CompositeByteBuf**

将多个 ByteBuf 组合成一个逻辑上的 ByteBuf，无需拷贝：

```java
// 普通方式：需要拷贝
ByteBuf header = Unpooled.buffer();
ByteBuf body = Unpooled.buffer();
ByteBuf combined = Unpooled.buffer(header.readableBytes() + body.readableBytes());
combined.writeBytes(header);
combined.writeBytes(body);  // 数据被拷贝了两次

// Netty 方式：CompositeByteBuf
CompositeByteBuf composite = Unpooled.compositeBuffer();
composite.addComponents(header, body);  // 只是引用，不拷贝
// composite.writerIndex(header.readableBytes() + body.readableBytes());
```

**技术二：FileRegion（sendfile）**

直接传输文件内容，利用 Linux sendfile：

```java
// FileRegion 实现零拷贝文件传输
FileInputStream fis = new FileInputStream("largefile.bin");
FileRegion region = new DefaultFileRegion(fis.getChannel(), 0, file.length());
ctx.write(region);

// 底层实现：
// 1. Linux sendfile 系统调用
// 2. 数据直接从 Page Cache 发送到网卡
// 3. 应用完全不参与数据拷贝
```

**技术三：wrap() 包装**

将已有字节数组包装成 ByteBuf，无需拷贝：

```java
// 普通方式
byte[] bytes = data.getBytes();
ByteBuf buffer = Unpooled.buffer(bytes.length);
buffer.writeBytes(bytes);  // 拷贝了一次

// Netty 方式
byte[] bytes = data.getBytes();
ByteBuf buffer = Unpooled.wrappedBuffer(bytes);  // 直接包装，无拷贝
```

**技术四：slice() 切片**

将一个 ByteBuf 切分成多个视图，无需拷贝：

```java
ByteBuf buffer = Unpooled.buffer(1024);
// ... 写入数据

// 切片：读取前 512 字节
ByteBuf slice = buffer.slice(0, 512);  // 只是视图，不拷贝数据

// 修改 slice 也会影响原始 buffer
// 适合处理协议的粘包半包
```

## 粘包半包：Netty 的解决方案

### 粘包半包的原因

TCP 是面向流的协议，不保留消息边界：

```bash
# 粘包：多条消息粘在一起
发送:  [消息A][消息B][消息C]
接收:  [消息A消息B消息C]  # 粘在一起了

# 半包：一条消息被拆分
发送:  [一条很长的消息]
接收:  [消息的前半部分][消息的后半部分]  # 分开接收
```

### Netty 的五种解决方案

**方案一：固定长度（FixedLengthFrameDecoder）**

```java
// 所有消息都是固定长度
pipeline.addLast(new FixedLengthFrameDecoder(128));

// 发送方必须保证每条消息都是 128 字节
// 不足的用空格补齐
```

**方案二：换行符（LineBasedFrameDecoder）**

```java
// 以 \n 或 \r\n 作为消息分隔符
pipeline.addLast(new LineBasedFrameDecoder(1024));

// 适合基于文本的协议
// 注意：消息内容不能包含换行符
```

**方案三：自定义分隔符（DelimiterBasedFrameDecoder）**

```java
// 自定义分隔符
ByteBuf delimiter = Unpooled.copiedBuffer("$$".getBytes());
pipeline.addLast(new DelimiterBasedFrameDecoder(1024, delimiter));

// 适合私有协议
// 注意：需要转义分隔符
```

**方案四：长度字段前置（LengthFieldBasedFrameDecoder）**

最通用的方案，适合绝大多数自定义协议：

```java
// 协议格式：[长度字段][消息体]
// 长度字段占 4 字节，表示消息体的长度
pipeline.addLast(new LengthFieldBasedFrameDecoder(
    65535,      // 最大帧长度
    0,          // 长度字段偏移量
    4,          // 长度字段长度
    0,          // 长度字段后的字节数（调整值）
    4           // 跳过的字节数
));
```

```bash
# 协议设计示例
# +-------+-------+-------+-------+
# | magic |  ver  |  type | len   |  ← 固定头部
# +-------+-------+-------+-------+
# |           body              |  ← 变长消息体
# +------------------------------+

# 字段说明：
# magic:  4 字节，魔数（校验协议）
# ver:    1 字节，版本号
# type:   1 字节，消息类型
# len:    4 字节，body 长度
# body:   变长，实际消息内容
```

**方案五：Protobuf 特定解码器**

```java
// Protobuf 解码器
pipeline.addLast(new ProtobufDecoder(MyMessage.getDefaultInstance()));
pipeline.addLast(new ProtobufVarint32FrameDecoder());

// Protobuf 自带编解码，不需要手动处理粘包半包
```

## Netty 高性能的关键

### 五大性能优化点

| 优化点 | 实现方式 | 性能收益 |
|---|---|---|
| 异步非阻塞 | EventLoop + Future/Promise | 充分利用 IO 等待时间 |
| 无锁化设计 | Channel 和 EventLoop 绑定 | 避免锁竞争 |
| 内存池 | PooledByteBufAllocator | 减少 GC，提升吞吐量 |
| Reactor 模型 | 主从 EventLoopGroup | 充分利用多核 |
| 零拷贝 | CompositeByteBuf/FileRegion | 减少内存拷贝 |

### 线程数配置建议

```java
// BossGroup：通常 1 个线程即可
NioEventLoopGroup bossGroup = new NioEventLoopGroup(1);

// WorkerGroup：根据 CPU 核数和业务特点调整
// IO 密集型：CPU 核数 × 2
// CPU 密集型：CPU 核数 + 1
NioEventLoopGroup workerGroup = new NioEventLoopGroup(
    Runtime.getRuntime().availableProcessors() * 2
);
```

### 优秀回答示例

**问题：ByteBuf 为什么要池化？**

> ByteBuf 池化的核心目的是**减少 GC**。
>
> 在高并发场景下，如果每次 IO 操作都 allocate一个新的 ByteBuffer，会产生大量对象。以 HTTP 服务为例，每秒 10 万请求，每个请求需要读写 ByteBuffer，GC 压力会非常大。
>
> Netty 的内存池设计有几个关键点：
>
> 第一，PoolArena 分片。将内存分成多个 Arena，每个 Arena 独立管理，减少线程竞争。
>
> 第二，PoolChunk 管理。每个 Chunk 64MB，内部按 Page 切分，小请求直接在同一 Page 内分配。
>
> 第三，PoolThreadCache 缓存。每个线程有自己的缓存，申请过的内存优先从缓存获取，避免每次都到 Arena 申请。
>
> 实际项目中，如果不使用池化 ByteBuf，高并发场景下 Full GC 会非常频繁，严重影响服务延迟。
