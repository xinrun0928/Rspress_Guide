# IO 模型：同步与异步、阻塞与非阻塞

面试官问：「你用过 NIO 吗？」

你回答：「用过。」

面试官点点头：「好，下一题。」

你松了一口气，以为这关过了。但心里隐隐觉得，这个回答太浅了——「用过」两个字背后，藏着多少你可能答不上的问题：NIO 和 BIO 的本质区别是什么？同步和异步到底在说什么？JDK AIO 在 Linux 上真的是异步的吗？

IO 模型的水，比大多数同学想象的深。这篇文章，就是要把这水抽干，让你彻底搞懂 BIO、NIO、AIO 的本质区别。

## 一、先搞清楚：什么是 IO

在说 IO 模型之前，我们需要明确一个基本概念：**IO 的本质是什么？**

IO（Input/Output），本质上是数据在内存和外部设备之间的流动。外部设备可能是磁盘、网络、键盘、显示器。数据流动的过程，涉及两个核心阶段：

- **内核等待数据就绪**：数据从外部设备到达操作系统内核缓冲区
- **数据从内核复制到用户空间**：内核缓冲区 → 用户程序缓冲区

```bash
磁盘/网卡 → 内核缓冲区 → 用户空间 → 返回结果
```

每个阶段都可能阻塞。「IO 模型」，本质上就是描述这两个阶段的不同处理方式。

## 二、四种 IO 模型

Unix 网络编程定义了五种 IO 模型（把「IO 多路复用」单独算作一种），但核心区分维度只有两个：

- **阻塞 vs 非阻塞**：等待数据就绪的过程中，用户线程是否阻塞
- **同步 vs 异步**：数据复制阶段，用户线程是否参与

组合起来，就形成了四种 IO 模型：

```
            等待数据就绪
                  │
        ┌─────────┴─────────┐
        ↓                   ↓
      阻塞                非阻塞
        │                   │
        ↓                   ↓
   ┌────┴────┐         ┌────┴────┐
   ↓         ↓         ↓         ↓
 同步IO    异步IO    同步IO    异步IO
   │                   │
   ↓                   ↓
 BIO(阻塞IO)      NIO(非阻塞IO)  AIO(异步IO)
              + 多路复用
```

### 2.1 阻塞 IO（BIO）

**阻塞 IO**（Blocking IO）是最传统、最简单的 IO 模型。它的特点是：**用户线程在两个阶段都会阻塞。**

```bash
用户线程                    内核
   │                         │
   │──── 请求读取数据 ───────→│
   │                         │
   │     （阻塞等待）         │
   │                         │
   │←─── 数据就绪，复制中 ───│
   │                         │
   │     （阻塞等待）         │
   │                         │
   │←──── 返回数据 ──────────│
   │                         │
```

具体来说：

1. 用户线程调用 `recvfrom`（或 `read`），发起读请求
2. 内核开始准备数据（从磁盘读取或从网络接收）
3. 在数据就绪之前，用户线程**阻塞等待**
4. 数据就绪后，内核开始将数据从内核缓冲区复制到用户缓冲区
5. 在数据复制完成之前，用户线程**继续阻塞**
6. 数据复制完成后，`recvfrom` 返回，用户线程继续执行

**BIO 的问题**：线程在等待数据就绪时完全阻塞，白白浪费 CPU 资源。如果有 1 万个连接，就意味着需要 1 万个线程——线程资源是有限的，每条线程默认占用 1MB 栈空间，1 万线程就是 10GB 内存。

### 2.2 非阻塞 IO（NIO）

**非阻塞 IO**（Non-blocking IO）对第一个阶段做了优化：**内核立即返回，而不是阻塞等待。**

```bash
用户线程                    内核
   │                         │
   │──── 请求读取数据 ───────→│
   │←──── 立即返回 EAGAIN ────│   （数据未就绪）
   │                         │
   │──── 再次请求 ───────────→│
   │←──── 再次返回 EAGAIN ────│   （数据仍未就绪）
   │                         │
   │──── 再次请求 ───────────→│
   │←──── 返回数据 ──────────│   （数据已就绪并复制完成）
   │                         │
```

关键点：

1. 用户线程发起 `recvfrom`，如果数据未就绪，内核**立即返回 EAGAIN**（或 EWOULDBLOCK），不阻塞
2. 用户线程需要**不断轮询**（polling），反复发起请求，直到数据就绪
3. 数据就绪后，用户线程等待数据复制完成（第二个阶段仍是阻塞的）

**NIO 的问题**：用户线程需要不断轮询，造成 CPU 空转，效率并不一定比 BIO 高。这就需要 IO 多路复用来解决。

### 2.3 IO 多路复用（IO Multiplexing）

**IO 多路复用**（IO Multiplexing）是 NIO 的改进版本。核心思想是：**用一个线程管理多个连接，避免每个连接一个线程的问题。**

```bash
用户线程                    内核
   │                         │
   │──── select/poll/epoll ──→│   （注册感兴趣的 fd）
   │←──── 返回就绪的 fd ──────│   （阻塞直到有 fd 就绪）
   │                         │
   │──── recvfrom(就绪的fd)──→│
   │←──── 返回数据 ──────────│
   │                         │
```

关键组件：

- **select/poll/epoll**：操作系统提供的系统调用，可以监听多个文件描述符（fd），返回就绪的 fd 列表
- **一个线程 + 一个 Selector**：管理多个连接

IO 多路复用的详细内容，将在下一篇文章《IO 多路复用：select/poll/epoll》中深入讲解。

### 2.4 异步 IO（AIO）

**异步 IO**（Asynchronous IO，AIO）的核心是：**全程无阻塞，内核完成后通知用户线程。**

```bash
用户线程                    内核
   │                         │
   │────aio_read（异步读）────→│
   │←──── 立即返回 ──────────│
   │                         │
   │  （做其他事情）          │
   │                         │
   │                         │──→ 等待数据就绪
   │                         │──→ 复制数据到用户缓冲区
   │                         │──→ 发送信号/调用回调
   │                         │
   │──── 收到通知，处理数据 ──│
   │                         │
```

关键点：

1. 用户线程发起异步读操作，立即返回，**完全不阻塞**
2. 内核负责等待数据就绪和数据复制
3. 内核完成后，通过**信号或回调**通知用户线程
4. 用户线程直接使用数据，无需等待

真正的异步 IO，必须同时满足：

- 第一阶段（等待数据就绪）**非阻塞**
- 第二阶段（数据复制）**非阻塞**
- 内核完成后**主动通知**应用

## 三、Java IO 的演进

理解完理论模型，我们来看 Java 是如何一步步支持这些 IO 模型的。

### 3.1 Java 1.0~1.3：只有 BIO

最早的 Java IO 是纯 BIO 的。`InputStream`、`OutputStream`、`Reader`、`Writer` 这些类，都是阻塞的。

```java
// Java 1.0 时代的典型代码
ServerSocket serverSocket = new ServerSocket(8080);
while (true) {
    Socket socket = serverSocket.accept();  // 阻塞
    new Thread(() -> {
        InputStream in = socket.getInputStream();
        // 读取数据
        int data = in.read();  // 阻塞
    }).start();
}
```

每个连接一个线程，连接多了，线程就爆了。

### 3.2 Java 1.4：引入 NIO（New IO）

2002 年，Java 1.4 引入了 NIO（New IO，后来改称 Non-blocking IO）。新增了三个核心组件：

- **Channel**：双向数据通道，可以理解为连接
- **Buffer**：读写缓冲区，数据先读入 Buffer
- **Selector**：多路复用器，监听多个 Channel 的事件

```java
// Java 1.4 NIO 示例
ServerSocketChannel serverChannel = ServerSocketChannel.open();
serverChannel.socket().bind(new InetSocketAddress(8080));
serverChannel.configureBlocking(false);  // 非阻塞模式

Selector selector = Selector.open();
serverChannel.register(selector, SelectionKey.OP_ACCEPT);

while (true) {
    selector.select();  // 阻塞，直到有就绪的 Channel
    Set<SelectionKey> selectedKeys = selector.selectedKeys();
    for (SelectionKey key : selectedKeys) {
        if (key.isAcceptable()) {
            // 处理新连接
        } else if (key.isReadable()) {
            // 处理读事件
        }
    }
}
```

### 3.3 Java 7：引入 NIO.2（AIO）

2011 年，Java 7 引入了 NIO.2，也叫 AIO（Asynchronous IO）。新增了异步 Channel：

```java
// Java 7 AIO 示例
AsynchronousServerSocketChannel serverChannel =
    AsynchronousServerSocketChannel.open().bind(new InetSocketAddress(8080));

serverChannel.accept(null, new CompletionHandler<AsynchronousSocketChannel, Void>() {
    @Override
    public void completed(AsynchronousSocketChannel result, Void attachment) {
        // 处理连接
        serverChannel.accept(null, this);  // 继续接受下一个连接
    }

    @Override
    public void failed(Throwable exc, Void attachment) {
        // 处理错误
    }
});
```

但这里有个坑——**JDK AIO 在 Linux 上并不是真正的异步 IO**。

## 四、为什么从 BIO 切换到 NIO

BIO 的问题是**连接数和线程数成正比**。

```
连接数 = 100     → 线程数 = 100
连接数 = 1000    → 线程数 = 1000
连接数 = 10000   → 线程数 = 10000
```

线程是稀缺资源：

- 每条线程默认占用 **1MB** 栈空间（-Xss 参数控制）
- JVM 能创建的线程数量有限，通常几千条就是上限
- 线程切换有上下文切换开销（通常 2~10 微秒）

**BIO 的天花板**：连接数达到几千时，线程资源耗尽，系统崩溃。

### 4.1 线程池优化：治标不治本

有人会说：用线程池不就好了？让线程复用。

```java
// 线程池优化方案
ExecutorService executor = Executors.newFixedThreadPool(100);
ServerSocket serverSocket = new ServerSocket(8080);

while (true) {
    Socket socket = serverSocket.accept();
    executor.submit(() -> {
        // 处理连接
    });
}
```

线程池确实能**复用**线程，减少线程创建销毁的开销。但问题没根本解决：

- 100 个线程，最多同时处理 **100 个并发连接**
- 如果有 1 万个连接，剩余 9900 个连接在排队等待
- 每个连接在「等待数据就绪」时，仍然占用一个线程

**线程池优化的是「线程创建销毁」的成本，但没有解决「连接等待」时线程阻塞的问题。**

### 4.2 NIO 的优势：1 个线程管理 N 个连接

NIO 的核心突破是：**一个线程可以管理多个连接，不需要为每个连接创建一个线程。**

```java
Selector selector = Selector.open();
serverChannel.register(selector, SelectionKey.OP_ACCEPT);

while (true) {
    int readyChannels = selector.select();  // 阻塞，直到有连接就绪
    // 处理就绪的连接
}
```

- 1 个线程 + 1 个 Selector，可以管理 **N 个连接**
- 线程在 Selector.select() 时阻塞，但不占用线程资源
- 连接就绪时，线程被唤醒，处理数据

**对比**：

```
BIO：10000 连接 = 10000 线程 = 系统崩溃
NIO：10000 连接 = 1 线程 + 1 Selector = 稳定运行
```

## 五、NIO 核心概念详解

### 5.1 Channel（通道）

Channel 是数据流通的管道，类似于 BIO 中的 Stream，但更强大：

- **双向**：Channel 既可以读，也可以写（Stream 只能单向）
- **非阻塞**：Channel 可以设置为非阻塞模式
- **配合 Buffer**：Channel 负责传输数据，Buffer 负责存储数据

常用的 Channel：

```java
// 文件 IO
FileChannel fileChannel = new FileInputStream("file.txt").getChannel();

// 网络 IO
SocketChannel socketChannel = SocketChannel.open();
socketChannel.connect(new InetSocketAddress("localhost", 8080));

ServerSocketChannel serverChannel = ServerSocketChannel.open();
serverChannel.socket().bind(new InetSocketAddress(8080));
```

### 5.2 Buffer（缓冲区）

Buffer 是数据的读写缓冲区。使用步骤：

```java
ByteBuffer buffer = ByteBuffer.allocate(1024);  // 分配缓冲区

// 写入数据
buffer.put("Hello".getBytes());

// 切换为读模式（flip）
buffer.flip();

// 读取数据
while (buffer.hasRemaining()) {
    byte b = buffer.get();
    System.out.print((char) b);
}

// 清空缓冲区
buffer.clear();
```

Buffer 的三个关键指针：

```
┌────────────────────────────────────────────┐
│ capacity │    limit    │     position      │
│    0     │    ←         │                   │
│         │   position    │   limit          │
│         │               │                   │
└────────────────────────────────────────────┘
      ↑            ↑           ↑
   capacity      position    limit
```

- **capacity**：缓冲区的容量，不可变
- **position**：当前读写位置，从 0 开始
- **limit**：最大可读写位置

### 5.3 Selector（选择器）

Selector 是 NIO 的核心，它**监听多个 Channel 的 IO 事件**，实现 IO 多路复用。

```java
Selector selector = Selector.open();

// 将 Channel 注册到 Selector
socketChannel.configureBlocking(false);  // 必须是非阻塞模式
SelectionKey key = socketChannel.register(selector, SelectionKey.OP_READ);

// 监听事件
while (true) {
    int readyChannels = selector.select();  // 阻塞
    if (readyChannels == 0) continue;

    Set<SelectionKey> selectedKeys = selector.selectedKeys();
    Iterator<SelectionKey> keyIterator = selectedKeys.iterator();

    while (keyIterator.hasNext()) {
        SelectionKey key = keyIterator.next();

        if (key.isAcceptable()) {
            // 新连接
        } else if (key.isReadable()) {
            // 可读
        } else if (key.isWritable()) {
            // 可写
        }

        keyIterator.remove();  // 处理完后移除
    }
}
```

## 六、为什么 epoll 比 select 更高效

BIO 的问题是每个连接一个线程，NIO 的问题是需要用户线程不断轮询。**IO 多路复用**解决了这个问题，但不同的实现效率相差巨大。

Java NIO 在不同平台上的底层实现：

- **Linux**：默认使用 epoll（Linux 2.6+）
- **macOS**：使用 Kqueue
- **Windows**：使用 IOCP（仅在 Java NIO.2/AIO 中）

select 和 epoll 的区别，将在下一篇文章《IO 多路复用：select/poll/epoll》中详细讲解。这里先预告一下：

| 特性 | select | epoll |
|---|---|---|
| fd 数量限制 | 1024 | 无限制 |
| 时间复杂度 | O(n) | O(1) |
| 内核拷贝 | 每次都要 | 只拷贝一次 |
| 唤醒通知 | 遍历 | 回调 |

## 七、面试追问

### 追问一：NIO 的核心优势是什么？

**普通回答**：NIO 是非阻塞的，可以用一个线程管理多个连接。

**深入回答**：

> NIO 的核心优势是**解决了 C10K 问题**（即如何支持 1 万个并发连接）。
>
> 在 BIO 模式下，连接数和线程数成正比，1 万连接需要 1 万线程，线程资源很快耗尽。NIO 通过 Selector 机制，实现了**一个线程管理多个连接**的能力。
>
> 具体来说：
> - **非阻塞 Channel**：连接读写不会阻塞线程
> - **Selector 多路复用**：一个 Selector 可以监听多个 Channel 的就绪事件
> - **事件驱动**：线程只在有事件就绪时才处理，不需要空转

### 追问二：为什么 epoll 比 select 高效？

**普通回答**：epoll 不用遍历所有 fd，所以更快。

**深入回答**：

> select 的问题是**每次调用都要把 fd 集合从用户态拷贝到内核态**，然后**内核线性扫描所有 fd** 找出就绪的。
>
> epoll 的改进：
> - fd 注册到红黑树，只需一次拷贝
> - fd 就绪时通过**回调机制**加入就绪列表，无需遍历
> - 调用 epoll_wait 时，只需要从就绪列表复制到用户态
>
> 本质区别：select 是**拉取模式**（用户态主动拉取），epoll 是**推送模式**（内核态主动推送就绪事件）。

### 追问三：JDK AIO 在 Linux 上真的是异步的吗？

**普通回答**：是，Java 7 引入了 AIO。

**深入回答**：

> 严格来说，**JDK AIO 在 Linux 上并不是真正的异步 IO**，而是**模拟的异步**。
>
> 原因：
> - Linux 的 AIO 支持不完善，JDK AIO 在 Linux 上底层用的是 **epoll**（JDK 7）或 **io_uring**（JDK 21+）
> - 真正的异步 IO 需要内核同时支持第一阶段（等待数据）和第二阶段（复制数据）的异步化
> - Linux 2.6 引入了 aio 系列系统调用，但支持有限
>
> 实际上：
> - **Windows IOCP 是真正的异步 IO**，内核在两个阶段都完成后才通知应用
> - **Linux epoll 是「IO 多路复用」，不是「异步 IO」**
> - **Java AIO 在 Linux 上的性能优势不明显**，所以实际项目中使用较少

### 追问四：什么场景下用 BIO 比 NIO 更合适？

**普通回答**：连接数少的时候用 BIO。

**深入回答**：

> 这个问题值得深入思考——BIO 并非一无是处。
>
> **BIO 适用的场景**：
> - **连接数少**（几百个以内），线程资源充足
> - **编程模型简单**，不需要处理复杂的状态管理
> - **业务逻辑简单**，每个连接的处理逻辑清晰
>
> NIO 带来的复杂性：
> - **状态管理复杂**：连接从建立到关闭有多种状态，需要正确处理
> - **半包/粘包问题**：需要自己处理数据边界
> - **调试困难**：事件驱动的执行流程比同步代码难理解
>
> 结论：**连接数少、业务简单的场景，BIO 更合适。连接数多、需要高并发的场景，NIO 更合适。** 技术选型要根据业务场景来，不是越「高级」越好。

---

## 总结

这一篇文章，我们从 IO 的本质出发，理解了四种 IO 模型的区别：

| 模型 | 等待数据就绪 | 数据复制 | 线程模型 |
|---|---|---|---|
| BIO | 阻塞 | 阻塞 | 1 连接 1 线程 |
| NIO | 非阻塞（轮询） | 阻塞 | 1 线程 N 连接（需轮询） |
| IO 多路复用 | 非阻塞（Selector） | 阻塞 | 1 线程 N 连接（事件驱动） |
| AIO | 非阻塞 | 非阻塞 | 真正的异步 |

Java 从 BIO 到 NIO 的演进，解决的是**并发连接数增长**的问题。理解这个演进过程，你才能真正回答「NIO 的核心优势是什么」这样的问题。
