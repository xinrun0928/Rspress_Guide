# IO 模型与零拷贝

> 你一定听说过「高性能」这个词，但有没有想过：支撑高性能的底层能力是什么？
>
> 答案是 **IO**。

当我们在谈论一个系统能扛多少 QPS、响应时间有多快时，归根结底，绕不开一个根本问题：**数据是怎么读写、怎么传输的。**

一个接口的响应时间，可能是 10ms，也可能是 100ms。差距不在业务逻辑，而在 IO——磁盘读写是否高效、网络传输是否阻塞、数据拷贝是否冗余。

这一模块，我们来彻底搞懂 IO 模型与零拷贝的原理。这是高性能优化的「底层基础」，也是 Java 面试的高频深水区。

## 模块定位：为什么 IO 能力是根基

很多同学做性能优化，第一反应是「加缓存」「优化 SQL」。这些当然重要，但如果不了解底层 IO 原理，就像盖楼不知道地基——你可以在地上层做很多装饰，但地基不稳，楼就盖不高。

IO 能力决定了什么？

- **并发上限**：IO 模型决定了系统能同时处理多少连接。BIO 模式下 1 万连接 = 1 万线程，线程资源耗尽；NIO 模式下 1 个线程管理 N 个连接。
- **响应延迟**：同步阻塞的 IO 操作会让线程空等；异步 IO 能让 CPU 在等待时做其他事情。
- **吞吐量**：零拷贝技术能减少数据在内存间的拷贝次数，让传输效率翻倍。

理解 IO 原理，不是为了「面试背八股文」，而是为了在实际工作中做出正确的技术选型——什么时候用 NIO，什么时候用 AIO，什么时候需要零拷贝。

## 三个核心主题

本模块围绕三个核心主题展开，层层递进：

### 主题一：IO 模型（同步/异步、阻塞/非阻塞）

这是 IO 能力的基础。理解 BIO、NIO、AIO 的区别，知道每种模型的适用场景，是做出正确技术选型的第一步。

### 主题二：多路复用（select/poll/epoll）

当连接数从几十个增长到上万个时，如何高效地管理这些连接？多路复用是解决方案。epoll 凭什么比 select 高效？这是高性能网络编程的核心。

### 主题三：零拷贝（减少内存拷贝）

数据传输的过程中，有多少是不必要的拷贝？Kafka 为什么这么快？答案就在零拷贝技术里——减少一次拷贝，性能就能提升一个数量级。

## 学习路径

三个主题的关系是递进的，建议按以下顺序学习：

```
第一步：理解 IO 模型

```

## 与 Netty 的关联

说到 IO 模型，不得不提 **Netty**。

Netty 是 Java 世界里最流行的网络通信框架，Tomcat、Dubbo、RocketMQ、Kafka（客户端）等都在用它。为什么？因为 Netty 把 NIO 的能力封装成了一套优雅的 API，让开发者不用直接面对复杂的 Selector、Channel、Buffer。

但理解 Netty 的前提，是理解 NIO。如果不懂 epoll 的原理，不懂 Reactor 模式，看 Netty 源码就是看天书。

本模块的最后一篇文章《Netty I/O 模型》，会从源码层面解析 Netty 的设计，让你知其然更知其所以然。
