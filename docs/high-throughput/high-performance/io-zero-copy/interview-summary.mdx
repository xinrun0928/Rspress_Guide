# IO 相关面试题汇总

本节梳理 IO 模型与零拷贝模块的高频面试题，每道题给出「普通回答」与「优秀回答」的对比，并预判面试追问方向。掌握这些题目，能让你在面试中从容应对 IO 相关的所有问题。

## 1. 「请描述 BIO、NIO、AIO 的区别」

### 普通回答

BIO 是同步阻塞 IO，客户端发起请求后，服务端必须处理完才能响应；NIO 是同步非阻塞，客户端发起请求后，服务端可以先去做别的事；AIO 是异步非阻塞，客户端发起请求后，服务端处理完会主动通知客户端。

### 优秀回答

这个问题要从两个维度来分析：**同步/异步**和**阻塞/非阻塞**。

**第一个维度：用户进程与内核的交互方式**

| IO 类型 | 同步/异步 | 阻塞/非阻塞 | 核心特点 |
|---|---|---|---|
| BIO | 同步 | 阻塞 | 每个连接一个线程，线程在 read/write 时阻塞 |
| NIO | 同步 | 非阻塞 | 一个线程处理多个连接，轮询注册的事件 |
| AIO | 异步 | 非阻塞 | 操作系统完成 IO 后主动回调，无需轮询 |

**第二个维度：数据拷贝的阶段**

一次完整的 IO 操作分为两个阶段：

1. **数据准备阶段**：数据从磁盘加载到内核缓冲区
2. **数据拷贝阶段**：数据从内核缓冲区拷贝到用户缓冲区

```bash
磁盘文件
    ↓
内核缓冲区（Page Cache）
    ↓
用户缓冲区（应用内存）
```

BIO 和 NIO 都是**同步 IO**，数据准备和数据拷贝阶段都需要用户进程参与（等待或主动拷贝）。AIO 是**异步 IO**，整个过程都由操作系统完成，完成后主动通知应用。

**第三个维度：适用场景**

```java
// BIO：适合连接数固定、业务逻辑简单的场景
// 每个连接一个线程，代码最简单
ServerSocket server = new ServerSocket(8080);
while (true) {
    Socket socket = server.accept();  // 阻塞
    new Thread(() -> handle(socket)).start();  // 每个连接一个线程
}

// NIO：适合高并发、连接数多的场景
// 一个线程处理万级连接
ServerSocketChannel server = ServerSocketChannel.open();
server.configureBlocking(false);  // 非阻塞
Selector selector = Selector.open();
server.register(selector, SelectionKey.OP_ACCEPT);
while (selector.select() > 0) {
    // 处理所有就绪的事件
}

// AIO：适合长连接、性能要求高的场景
// Windows 原生支持，Linux JDK 7 以后支持
AsynchronousServerSocketChannel channel = AsynchronousServerSocketChannel.open();
channel.bind(new InetSocketAddress(8080));
channel.accept(null, new CompletionHandler<SocketChannel, Void>() {
    @Override
    public void completed(SocketChannel result, Void attachment) {
        // 操作系统完成 IO 后自动回调
    }
});
```

## 2. 「select、poll、epoll 的区别是什么」

### 普通回答

select 能监听的文件描述符数量有限制，一般是 1024；poll 没有数量限制；epoll 是 Linux 特有的，性能最好，支持边缘触发和水平触发。

### 优秀回答

这个问题要从**底层原理**说起，三者的本质区别在于「如何管理待检测的文件描述符」。

**select 的问题：**

```c
// select 的核心实现
int select(int nfds, fd_set *readfds, fd_set *writefds,
           fd_set *exceptfds, struct timeval *timeout);

// 问题 1：fd_set 大小固定（Linux 默认 1024）
// 问题 2：每次调用都要把 fd_set 从用户态拷贝到内核态
// 问题 3：返回后要遍历所有 fd，找到就绪的那个
```

每次调用 select，都要经历：**用户态 → 内核态拷贝 → 遍历 fd 数组 → 返回 → 再次拷贝**。如果有 10000 个连接，但只有 1 个活跃，select 要拷贝和遍历 10000 次。

**poll 的改进：**

```c
// poll 用动态数组替代固定大小的 fd_set
struct pollfd {
    int fd;        // 文件描述符
    short events;  // 监听的事件
    short revents; // 就绪的事件
};

int poll(struct pollfd *fds, nfds_t nfds, int timeout);

// 解决了数量限制问题，但仍是每次调用都要全量拷贝
```

**epoll 的革命性设计：**

```c
// epoll 三个核心接口
int epoll_create(int size);      // 创建 epoll 实例
int epoll_ctl(int epfd, int op, int fd, struct epoll_event *event);  // 注册事件
int epoll_wait(int epfd, struct epoll_event *events, int maxevents, int timeout);  // 等待事件
```

epoll 的核心思路是**减少无效拷贝和遍历**：

```java
// epoll 的工作原理：红黑树 + 就绪链表
// 1. epoll_create 创建 epoll 实例，同时创建红黑树和就绪链表
// 2. epoll_ctl 往红黑树添加/删除/修改监控的事件
// 3. epoll_wait 只返回就绪链表中的 fd，无需遍历全部

// 关键区别：fd 只注册一次，不用每次调用都拷贝
while (true) {
    int n = epoll_wait(epfd, events, maxevents, -1);
    // 返回的 n 个 events 直接就是已就绪的 fd
    for (int i = 0; i < n; i++) {
        // 处理每个就绪的事件
    }
}
```

**三者的性能对比（10000 连接，1 个活跃）：**

| 操作 | select | poll | epoll |
|---|---|---|---|
| fd 注册 | 每次调用都要拷贝 | 每次调用都要拷贝 | 只注册一次，后续无需操作 |
| 时间复杂度 | O(n) | O(n) | O(1) |
| fd 数量限制 | 1024 | 无限制 | 无限制 |
| 内存拷贝 | 每次全量拷贝 | 每次全量拷贝 | 只拷贝就绪的 |

```java
// Java NIO 中，Selector 的底层就是 epoll（Linux 环境）
Selector selector = Selector.open();
// 这背后的实现就是 epoll_create + epoll_ctl + epoll_wait
selector.select();  // 等价于 epoll_wait
```

## 3. 「epoll 的 LT 和 ET 模式有什么区别」

### 普通回答

LT 是水平触发，只要缓冲区有数据就会通知；ET 是边缘触发，只在状态变化时通知一次。

### 优秀回答

LT 和 ET 是 epoll 的两种触发模式，理解它们的关键是**从「事件通知的时机」角度思考**。

**水平触发（Level Triggered，LT）：**

```java
// 缓冲区状态为"有数据"时，持续通知
// 只要缓冲区有数据，每次 epoll_wait 都会返回

// 场景：客户端发送了 100 字节，服务器只读取了 50 字节
// 再次调用 epoll_wait 时，即使没有新数据，仍会返回这个 fd
while (true) {
    int n = epoll_wait(epfd, events, MAX_EVENTS, -1);
    for (int i = 0; i < n; i++) {
        // 这个 fd 会持续返回，直到缓冲区数据被读完
    }
}
```

**边缘触发（Edge Triggered，ET）：**

```java
// 缓冲区状态从"无数据"变为"有数据"时，只通知一次
// 类似于上升沿触发，只通知变化的那一刻

// 场景：客户端发送了 100 字节
// 第一次 epoll_wait 返回这个 fd
// 如果只读了 50 字节，再次调用 epoll_wait 不会返回
// 必须把 100 字节都读完，下次新数据到来才会再次通知
```

**两者的核心区别：**

| 特性 | LT（水平触发） | ET（边缘触发） |
|---|---|---|
| 通知时机 | 缓冲区有数据就一直通知 | 状态变化时只通知一次 |
| 漏读后果 | 下次还能收到 | 数据可能丢失 |
| 需要循环读取 | 否 | 是（必须循环读到 EAGAIN） |
| CPU 开销 | 可能重复通知 | 通知更精准 |
| 编程难度 | 简单 | 需要完整读取所有数据 |

```c
// ET 模式下的正确读取方式
while (1) {
    int n = recv(fd, buf, sizeof(buf), 0);
    if (n == -1) {
        if (errno == EAGAIN || errno == EWOULDBLOCK) {
            // 数据读完了，下次 EPOLLIN 事件再来
            break;
        }
        // 其他错误
        break;
    }
    // 处理数据
}
```

**适用场景：**

- **LT 模式**：适合**大多数场景**，编程简单，不容易出错。Java NIO 默认使用 LT 模式。
- **ET 模式**：适合**高性能场景**，如 Nginx、Redis，要求必须一次性读完所有数据，避免重复通知带来的开销。

## 4. 「什么是零拷贝？Java 中如何实现零拷贝」

### 普通回答

零拷贝就是减少数据拷贝次数，Java 中可以用 sendfile 和 mmap 实现。

### 优秀回答

零拷贝的核心是**减少甚至消除数据在内核空间和用户空间之间的拷贝次数**。

**传统 IO 的数据拷贝流程（4 次拷贝 + 2 次上下文切换）：**

```bash
磁盘文件
    ↓（DMA 拷贝）
内核缓冲区（Page Cache）
    ↓（CPU 拷贝）
用户缓冲区
    ↓（CPU 拷贝）
Socket 缓冲区
    ↓（DMA 拷贝）
网卡
```

四次拷贝：
1. DMA 将数据从磁盘拷贝到内核缓冲区
2. CPU 将数据从内核缓冲区拷贝到用户缓冲区
3. CPU 将数据从用户缓冲区拷贝到 Socket 缓冲区
4. DMA 将数据从 Socket 缓冲区拷贝到网卡

**零拷贝的实现（2 次拷贝 + 2 次上下文切换）：**

```bash
磁盘文件
    ↓（DMA 拷贝）
内核缓冲区（Page Cache）
    ↓（DMA 拷贝，直接送到网卡）
网卡
```

零拷贝的核心是**数据不需要经过用户缓冲区**，而是直接在内核空间传输。

**Java 中实现零拷贝的三种方式：**

```java
// 方式 1：FileChannel.transferTo()（底层使用 sendfile）
public void transferTo(String sourceFile, String targetFile) throws IOException {
    FileChannel source = new FileInputStream(sourceFile).getChannel();
    FileChannel target = new FileOutputStream(targetFile).getChannel();
    // 内部调用 Linux sendfile 系统调用
    long transferred = source.transferTo(0, source.size(), target);
}

// 方式 2：FileChannel.map()（mmap）
public void mmapCopy(String sourceFile, String targetFile) throws IOException {
    FileChannel source = new RandomAccessFile(sourceFile, "r").getChannel();
    MappedByteBuffer buffer = source.map(FileChannel.MapMode.READ_ONLY, 0, source.size());
    // 文件直接映射到内存，避免了 read/write 的内核到用户拷贝
}

// 方式 3：Netty 的零拷贝
// CompositeByteBuf：组合多个 ByteBuf，无需拷贝
ByteBuf header = Unpooled.buffer();
ByteBuf body = Unpooled.buffer();
CompositeByteBuf composite = Unpooled.compositeBuffer();
composite.addComponents(header, body);  // 无需拷贝，只是引用

// slice()：分割现有 ByteBuf，无需拷贝
ByteBuf sliced = buffer.slice(start, length);

// FileRegion：零拷贝文件传输
FileRegion region = new DefaultFileRegion(file.getChannel(), 0, file.length());
ctx.write(region);  // 直接使用 sendfile
```

**零拷贝的适用场景：**

- 文件传输（静态资源服务、文件下载）
- 消息队列（Kafka 的日志读写）
- 数据库（Redolog、WAL 的顺序写入）

- 「Kafka 是如何利用零拷贝提升性能的？」
- 「什么情况下零拷贝反而不如普通 IO？」

## 5. 「Kafka 为什么这么快」

### 普通回答

Kafka 用了顺序写、零拷贝、批量处理等技术，所以性能很高。

### 优秀回答

Kafka 的高性能是多种技术叠加的结果，我们从 IO 的角度逐层分析。

**第一层：顺序写入**

磁盘的顺序读写速度是随机写读的数百倍。Kafka 将消息追加到日志文件末尾，利用顺序写的特性：

```bash
# 磁盘顺序写 vs 随机写性能对比
# 顺序写：500 MB/s
# 随机写：0.1 MB/s
# 差距：5000 倍
```

Kafka 的写入流程：

```java
// Kafka Producer 发送消息
ProducerRecord<String, String> record = new ProducerRecord<>("topic", "key", "value");
producer.send(record);  // 消息追加到日志文件末尾
```

**第二层：零拷贝**

Kafka 的消息消费使用 sendfile：

```bash
# 传统方式：4 次拷贝
磁盘 → 内核缓冲区 → 用户缓冲区 → Socket 缓冲区 → 网卡

# Kafka（sendfile）：2 次拷贝
磁盘 → 内核缓冲区（Page Cache）→ 网卡
```

```java
// Broker 发送消息给 Consumer
// FileChannel.transferTo() 底层调用 sendfile
fileChannel.transferTo(position, length, socketChannel);
```

**第三层：页缓存（Page Cache）**

Kafka 充分利用操作系统 Page Cache：

```bash
# 写入：先写 Page Cache（内存），异步刷盘
# 读取：先读 Page Cache，如果命中则无需访问磁盘

# 写入流程：
应用 → Page Cache（内存）→ 异步刷盘到磁盘

# 读取流程：
Page Cache 命中 → 直接返回（内存）
Page Cache 未命中 → 磁盘读取 → Page Cache → 返回
```

**第四层：批量处理**

```java
// Producer 批量发送
producer.flush();  // 攒一批再发送，减少网络请求次数

// Consumer 批量消费
// Broker 一次返回一批消息，减少网络往返
```

**第五层：分区并行**

```java
// Kafka Topic 可以有多个 Partition
// 每个 Partition 的写入是顺序的，但多个 Partition 可以并行

// Producer 指定 key，保证同一 key 的消息在同一 Partition
ProducerRecord<String, String> record = 
    new ProducerRecord<>("order-topic", orderId, orderInfo);
```

**Kafka 高性能的核心公式：**

```
高性能 = 顺序写（减少磁盘寻道）+ 零拷贝（减少内存拷贝）+ 页缓存（利用内存）+ 批量处理（减少网络往返）+ 分区并行（横向扩展）
```

## 6. 「Netty 的线程模型是怎样的」

### 普通回答

Netty 有 BossGroup 和 WorkerGroup，BossGroup 接收连接，WorkerGroup 处理 IO 事件。

### 优秀回答

Netty 的线程模型基于**主从 Reactor 模式**，这是目前高性能网络框架的主流选择。

**Reactor 模式的三种形态：**

```java
// 形态 1：单 Reactor 单线程
// 所有连接和 IO 都在一个线程处理，适合简单的 RPC 框架
Reactor reactor = new Reactor();
reactor.register(SelectionKey.OP_ACCEPT);
reactor.run();  // 单线程处理所有事件

// 问题：单线程处理所有 IO，无法利用多核 CPU
```

```java
// 形态 2：单 Reactor 多线程
// IO 处理部分用线程池，但连接处理仍在单线程
Reactor reactor = new Reactor();
reactor.register(SelectionKey.OP_ACCEPT);
reactor.run();

// Worker 线程池处理 IO
ExecutorService workers = Executors.newFixedThreadPool(8);

// 问题：IO 处理是并行了，但 Reactor 仍是瓶颈
```

```java
// 形态 3：主从 Reactor（Netty 采用）
// 主 Reactor 处理连接建立，从 Reactor 处理 IO 事件
ServerBootstrap bootstrap = new ServerBootstrap();
bootstrap.group(new NioEventLoopGroup(1),  // BossGroup：主 Reactor
                new NioEventLoopGroup(8));   // WorkerGroup：从 Reactor

// BossGroup 负责处理连接建立（OP_ACCEPT）
// WorkerGroup 负责处理读写事件（OP_READ、OP_WRITE）
```

**Netty 线程模型的工作流程：**

```java
// BossGroup 的工作（单线程）
NioServerSocketChannel serverChannel = ...;
serverChannel.bind();

// Boss 线程执行：
// 1. 接收客户端连接
// 2. 将连接注册到 WorkerGroup
// 3. 继续等待新的连接

// WorkerGroup 的工作（多线程）
NioEventLoop worker = new NioEventLoop();
worker.register(ch);  // 注册 Channel

// Worker 线程执行：
// 1. 处理 OP_READ：读取数据，触发 ChannelPipeline
// 2. 处理 OP_WRITE：写数据
// 3. 处理 OP_CONNECT：连接建立成功
```

**EventLoop 的工作机制：**

```java
// EventLoop = Selector + TaskQueue
// 一个 EventLoop 绑定一个线程，处理多个 Channel 的 IO 事件

public class NioEventLoop extends SingleThreadEventLoop {
    private final Selector selector;
    private final TaskQueue taskQueue;
    
    public void run() {
        while (!isShuttingDown()) {
            // 1. 阻塞等待 IO 事件就绪
            int ready = selector.select();
            
            // 2. 处理 IO 事件
            processSelectedKeys();
            
            // 3. 处理任务队列（异步任务）
            runAllTasks();
        }
    }
}
```

**一个 EventLoop 处理多个 Channel 的原理：**

```java
// Selector 可以注册多个 Channel
Selector selector = Selector.open();
channel1.register(selector, SelectionKey.OP_READ);
channel2.register(selector, SelectionKey.OP_READ);
channel3.register(selector, SelectionKey.OP_WRITE);

// select() 返回后，遍历所有就绪的 Channel
// 每个 Channel 的处理都是非阻塞的
```

**Netty 线程模型的三个核心优势：**

1. **主从分离**：连接处理和 IO 处理分开，互不阻塞
2. **线程绑定**：EventLoop 和 Channel 绑定，避免锁竞争
3. **任务队列**：IO 事件和异步任务共用线程，提高利用率

## 7. 「Netty 是如何解决粘包半包问题的」

### 普通回答

Netty 提供了多种解码器，比如 FixedLengthFrameDecoder、LineBasedFrameDecoder、LengthFieldBasedFrameDecoder 来解决粘包半包。

### 优秀回答

要回答好这个问题，先要理解**什么是粘包和半包**。

**粘包：一个请求被拆分成多个部分发送**

```bash
# 发送端连续发送两条消息
消息A：Hello
消息B：World

# 接收端可能收到
HelloWorld  # 粘在一起
```

**半包：一个请求被拆分成多次发送**

```bash
# 发送端发送一条长消息
消息A：Hello World This is a long message...

# 接收端可能分多次收到
Hello World    # 第一次收到部分
This is a...   # 第二次收到剩余
```

**为什么会发生粘包半包？**

TCP 是面向流的协议，不保留消息边界。发送方和接收方的缓冲区大小、网络状况、TCP 拥塞控制都会影响数据的分包。

**Netty 的五种解决方案：**

```java
// 方案 1：固定长度解码器
// 所有消息都是固定长度，不足部分用空格补齐
Pipeline.addLast(new FixedLengthFrameDecoder(128));

// 优点：简单
// 缺点：浪费带宽，不灵活
// 适用：自定义协议，二进制消息
```

```java
// 方案 2：换行符解码器
// 以 \n 或 \r\n 作为消息分隔符
Pipeline.addLast(new LineBasedFrameDecoder(1024));

// 优点：方便文本协议
// 缺点：消息内容不能包含分隔符
// 适用：Redis 协议、简单文本协议
```

```java
// 方案 3：分隔符解码器
// 自定义分隔符
Pipeline.addLast(new DelimiterBasedFrameDecoder(1024, 
    Unpooled.copiedBuffer("$$".getBytes())));

// 优点：灵活指定分隔符
// 缺点：需要转义分隔符
// 适用：自定义文本协议
```

```java
// 方案 4：长度字段解码器（最常用）
// 消息格式：[长度字段][实际数据]
Pipeline.addLast(new LengthFieldBasedFrameDecoder(
    1024,    // 最大帧长度
    0,       // 长度字段偏移量
    4,       // 长度字段长度
    0,       // 长度字段后的字节数（调整值）
    4));     // 跳过的字节数

// 优点：最灵活，适用于几乎所有自定义协议
// 适用：HTTP、gRPC、私有协议
```

```java
// 方案 5：自定义解码器（组合拳）
// 实际项目中通常组合使用
Pipeline.addLast(new LengthFieldBasedFrameDecoder(65535, 0, 4, 0, 4));
Pipeline.addLast(new ObjectDecoder());  // 反序列化
Pipeline.addLast(new BusinessHandler()); // 业务处理

// 协议设计最佳实践
public class MyProtocol {
    // +-------+-------+-------+-------+
    // | magic |  ver  |  type | len   |
    // +-------+-------+-------+-------+
    // |           body              |
    // +------------------------------+
    // magic: 4 字节魔数，校验协议
    // ver: 1 字节版本号
    // type: 1 字节消息类型
    // len: 4 字节消息体长度
    // body: 变长消息体
}
```

## 8. 「mmap 和 sendfile 的区别」

### 普通回答

mmap 是把文件映射到内存，sendfile 是直接发送文件，它们都是零拷贝的实现方式。

### 优秀回答

mmap 和 sendfile 都属于零拷贝技术，但它们的**实现原理和适用场景**不同。

**mmap（Memory Map）：**

```java
// Java 中使用 mmap
FileChannel channel = new RandomAccessFile(file, "rw").getChannel();
MappedByteBuffer buffer = channel.map(MapMode.READ_WRITE, 0, channel.size());

// 读取：像访问内存一样访问文件
byte b = buffer.get();

// 写入：直接修改内存，操作系统异步刷盘
buffer.put(0, (byte) 'H');
```

mmap 的原理：

```bash
# 传统 read/write
应用 → 内核缓冲区 → 磁盘（写入）
磁盘 → 内核缓冲区 → 用户缓冲区 → 应用

# mmap
应用 ←→ Page Cache（内存映射）←→ 磁盘
# 没有用户缓冲区和内核缓冲区的区分，共享同一块物理内存
```

**sendfile：**

```java
// Java 中使用 sendfile
FileChannel from = new FileInputStream("file.txt").getChannel();
FileChannel to = new FileOutputStream("to.txt").getChannel();
from.transferTo(0, from.size(), to);  // 底层调用 sendfile
```

sendfile 的原理：

```bash
# sendfile 原理（Linux 2.4+）
磁盘 → Page Cache → 网卡
# 完全在内核空间完成，无需用户态介入
# DMA 引擎直接将 Page Cache 数据送到网卡
```

**两者的核心区别：**

| 特性 | mmap | sendfile |
|---|---|---|
| 用户态是否参与 | 是（需要用户态处理数据） | 否（完全内核态） |
| 适用方向 | 读写（应用需要处理数据） | 传输（应用不需要处理数据） |
| 修改后何时刷盘 | 依赖系统脏页刷新策略 | 立即触发或应用主动 fsync |
| 适合场景 | 需要修改文件内容的场景 | 文件下载、日志传输等只读场景 |
| 多进程安全 | 需要额外同步机制 | 天然支持多消费者 |

**典型应用场景：**

```java
// mmap 的典型场景：数据库索引、日志写入
// RocketMQ 的消息存储使用 mmap
// 消费者拉取消息使用 sendfile（零拷贝传输）

// 文件下载（静态资源服务）
// 用 sendfile，文件数据直接从磁盘到网卡，应用不参与
// 因为下载场景不需要应用处理数据内容
```

## 9. 「Reactor 模式是什么」

### 普通回答

Reactor 模式是一种事件驱动架构，用一个线程处理多个连接，提高并发性能。

### 优秀回答

Reactor 模式是**高性能网络编程的核心范式**，从本质上说，它解决的是「如何高效地处理大量并发 IO」这个问题。

**问题的本质：**

传统 BIO 的问题：

```java
// 每个连接一个线程，10000 个连接就需要 10000 个线程
while (true) {
    Socket socket = server.accept();  // 阻塞
    new Thread(() -> {
        try {
            // 连接占用的线程在整个生命周期中大部分时间都在等待 IO
            while (true) {
                byte[] data = socket.read();  // 阻塞
                // 处理数据
            }
        } catch (IOException e) {}
    }).start();
}

// 问题：线程是稀缺资源，大量线程造成上下文切换开销
// 线程栈占用内存（1MB/线程 × 10000 = 10GB）
// CPU 在线程间切换浪费时间
```

**Reactor 的解决思路：**

不要让线程等待 IO，而是让线程**主动去检查 IO 是否就绪**：

```java
// 核心：单线程监听多个 Channel 的就绪状态
Selector selector = Selector.open();

// 注册所有连接到一个 Selector
for (SocketChannel ch : channels) {
    ch.register(selector, SelectionKey.OP_READ);
}

// 单线程循环检查
while (true) {
    int ready = selector.select();  // 阻塞，直到有 Channel 就绪
    
    Set<SelectionKey> keys = selector.selectedKeys();
    for (SelectionKey key : keys) {
        if (key.isReadable()) {
            // 快速处理就绪的 Channel
            SocketChannel ch = (SocketChannel) key.channel();
            handleRead(ch);
        }
    }
    keys.clear();
}
```

**Reactor 模式的三个组件：**

```java
// 1. Reactor（反应器）：监听和分发事件
class Reactor {
    Selector selector;
    
    void loop() {
        while (true) {
            selector.select();  // 等待事件
            dispatch();          // 分发事件
        }
    }
    
    void dispatch() {
        for (SelectionKey key : selector.selectedKeys()) {
            // 分发给对应的 Handler
        }
    }
}

// 2. Handler（处理器）：处理具体的 IO 事件
interface Handler {
    void handleRead();
    void handleWrite();
}

// 3. Acceptor（接收器）：处理新连接
class Acceptor {
    void accept() {
        SocketChannel client = server.accept();
        // 注册到 Reactor
        client.register(selector, SelectionKey.OP_READ);
    }
}
```

**Reactor 的三种实现：**

| 实现 | 特点 | 适用场景 |
|---|---|---|
| 单 Reactor 单线程 | 简单，但无法利用多核 | 小型服务、学习演示 |
| 单 Reactor 多线程 | Reactor 单线程，Handler 多线程 | 中型服务 |
| 主从 Reactor | 主 Reactor 处理连接，从 Reactor 处理 IO | 大型高并发服务（Netty 采用） |

```java
// Netty 的主从 Reactor 实现
ServerBootstrap bootstrap = new ServerBootstrap();
bootstrap.group(
    new NioEventLoopGroup(1),  // Boss：主 Reactor，处理连接
    new NioEventLoopGroup(8)   // Worker：从 Reactor，处理 IO
);
```

## 10. 「Netty 和 NIO 的区别是什么」

### 普通回答

NIO 是 Java 的原生 API，Netty 是基于 NIO 的框架，Netty 封装得更好，性能更高。

### 优秀回答

NIO 和 Netty 的关系，是**「自行车」和「汽车」的关系**——都能到达目的地，但体验和效率完全不同。

**Java NIO 的核心问题：**

```java
// NIO 的 Selector 操作繁琐，容易出错
Selector selector = Selector.open();
ServerSocketChannel server = ServerSocketChannel.open();
server.bind(new InetSocketAddress(8080));
server.configureBlocking(false);
server.register(selector, SelectionKey.OP_ACCEPT);

while (true) {
    selector.select();
    Set<SelectionKey> keys = selector.selectedKeys();
    Iterator<SelectionKey> iter = keys.iterator();
    
    while (iter.hasNext()) {
        SelectionKey key = iter.next();
        iter.remove();  // 必须手动移除，否则会重复处理
        
        if (key.isAcceptable()) {
            // 处理 Accept
        } else if (key.isReadable()) {
            // 处理 Read
            ByteBuffer buf = ByteBuffer.allocate(1024);
            SocketChannel ch = (SocketChannel) key.channel();
            ch.read(buf);  // 需要手动处理半包
        } else if (key.isWritable()) {
            // 处理 Write
        }
    }
}

// 问题：
// 1. ByteBuffer 需要手动管理，大小固定
// 2. 半包问题需要自己处理
// 3. Selector keySet 操作容易出错
// 4. 线程模型需要自己实现
// 5. 断连重连需要自己处理
```

**Netty 的核心改进：**

```java
// Netty 的代码优雅简洁
ServerBootstrap bootstrap = new ServerBootstrap();
bootstrap.group(new NioEventLoopGroup(), new NioEventLoopGroup())
         .channel(NioServerSocketChannel.class)
         .childHandler(new ChannelInitializer<SocketChannel>() {
             @Override
             protected void initChannel(SocketChannel ch) {
                 // Pipeline 模式，职责链处理
                 ch.pipeline()
                   .addLast(new LengthFieldBasedFrameDecoder(...))
                   .addLast(new ByteBufDecoder())
                   .addLast(new BusinessHandler());
             }
         });

// Netty 解决的问题：
// 1. ByteBuf 池化管理，自动扩容
// 2. 内置多种解码器，解决粘包半包
// 3. Pipeline 责任链，逻辑清晰
// 4. EventLoopGroup 线程模型开箱即用
// 5. Channel 自动处理断连重连
```

**Netty 相对于 NIO 的六大优势：**

| 特性 | Java NIO | Netty |
|---|---|---|
| ByteBuffer | 手动管理，大小固定 | ByteBuf 池化，自动扩容 |
| 粘包半包 | 需要自己处理 | 内置多种解码器 |
| 线程模型 | 需要自己实现 | EventLoopGroup 开箱即用 |
| Channel 生命周期 | 需要手动管理 | 完整的生命周期管理 |
| 异常处理 | 需要自己处理 | 完善的异常处理机制 |
| 性能 | 基础性能 | 高度优化，减少 GC |

**Netty 的 ByteBuf vs NIO 的 ByteBuffer：**

```java
// NIO ByteBuffer 的问题
ByteBuffer buffer = ByteBuffer.allocate(1024);
// 只有一个 position，读写切换需要 flip()
// 无法动态扩容

// Netty ByteBuf 的优势
ByteBuf buffer = Unpooled.buffer(1024);
// 读写有独立的 position
// 支持动态扩容
// 引用计数，内存池回收
```

**为什么生产环境都用 Netty：**

1. **成熟稳定**：经过大量生产环境验证（Dubbo、gRPC、RocketMQ 等都在用）
2. **生态完善**：编解码、协议支持、连接管理一应俱全
3. **社区活跃**：版本迭代快，问题修复及时
4. **性能领先**：经过大量优化，是 Java 网络编程的性能标杆

## 面试实战总结

IO 相关面试题的核心脉络：

```bash
IO 基础（BIO/NIO/AIO）
    ↓
多路复用（select/poll/epoll）
    ↓
Reactor 模式（高性能网络框架的基石）
    ↓
零拷贝（mmap/sendfile）
    ↓
框架实践（Netty、Kafka）
```

掌握这条脉络，你就能在面试中举一反三，从容应对各种 IO 相关的追问。

**核心口诀：**

- BIO 是「一个萝卜一个坑」，每个连接占用一个线程
- NIO 是「一个坑轮流站」，一个线程处理多个连接
- epoll 是「有人来了再叫你」，不用一直问「有人吗」
- 零拷贝是「快递直达」，数据不经过中间商
- Netty 是「把 NIO 封装成框架」，开箱即用
