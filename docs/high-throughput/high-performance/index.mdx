# 高性能优化：让系统跑得更快

import { Badge } from '@rspress/core/theme';

凌晨 2 点，你正准备上线大促活动，突然监控大屏报警：订单接口 RT 从 50ms 飙升至 3000ms，用户怨声载道。你慌忙排查，发现数据库连接池太小、SQL 没有索引、缓存命中率只有 30%——问题一个接一个。

这是每个后端工程师都可能遇到的场景。而高性能优化的核心任务，就是**让系统在各种压力下依然保持「快」的姿态**。

高性能不是某一个知识点，而是一套系统化的工程能力。它涵盖数据库优化、JVM 调优、缓存设计、IO 模型、池化技术、接口优化等多个维度。每个维度都有自己的原理、方法和坑，只有系统性地掌握，才能在面对性能问题时游刃有余。

## 一、高性能优化的核心目标

高性能优化的终极目标可以归结为两个核心指标：**降低 RT（Response Time）** 和 **提升 QPS/TPS（吞吐量）**。

**RT** 是用户最能感知的指标。用户点击一个按钮，1 秒内有响应和 5 秒才有响应，体验天差地别。在电商场景下，RT 每增加 100ms，转化率可能下降 1%。所以降低 RT，是提升用户体验的直接手段。

**QPS/TPS** 是系统吞吐能力的体现。在同样的硬件条件下，QPS 越高，说明系统的处理效率越高。大促期间，流量是平时的 10 倍甚至 100 倍，如果系统 QPS 上不去，就会出现排队、超时、甚至系统崩溃。

```
高性能 = 低 RT + 高 QPS + 低错误率
```

这三个指标是相互关联的。RT 太长会导致超时错误；QPS 上不去会导致请求堆积；任何一个指标崩了，都意味着系统出了问题。

## 二、高性能优化的两大基础主题

在深入各个子模块之前，我们需要理解两个贯穿始终的基础主题：**木桶原理**和**测量驱动**。

### 2.1 木桶原理：永远聚焦最短板

一个木桶能装多少水，不取决于最长的那块板，而取决于最短的那块。这个道理放到系统性能上同样成立。

一个典型的 Web 服务，请求链路可能是这样的：

```
用户请求 → Nginx → 网关 → 应用服务 → Redis → MySQL → 返回
```

每个环节都有自己的处理能力。如果 Redis 能扛 10 万 QPS，MySQL 只能扛 1 万 QPS，那整个链路的实际吞吐上限就是 **1 万 QPS**。花再多精力优化 Redis，都是在做无用功。

**系统的整体性能，由最薄弱环节决定。** 这是木桶原理的核心。

### 2.2 测量驱动：用数据找瓶颈

很多团队做性能优化，有一个共同的问题：**凭经验猜测瓶颈，而不是用数据定位问题。**

「我猜是数据库慢」「我觉得是 GC 问题」「应该是网络延迟」——这些都是假设，不是结论。

测量驱动的核心原则是：**在优化之前，先用数据说话。** 通过 APM 工具（如 SkyWalking）、日志分析、火焰图等手段，找到真正的瓶颈点。只有数据证明优化有效，才算完成。

### 2.3 两者结合：标准优化流程

```
测量 → 定位短板（木桶原理）→ 优化 → 测量验证 → 继续找下一个短板 → ...
```

优化是分层的。先解决最突出的瓶颈，再解决下一个。木桶的短板是动态变化的——解决了一个瓶颈，下一个瓶颈就会浮现。

> 详细内容请参考 [性能优化核心思想：木桶原理与测量驱动](/high-throughput/high-performance/method/principle)。

## 五、与高并发、高可用的关联

高性能（Performance）、高并发（Concurrency）、高可用（Availability）是系统设计的三个核心目标，它们相互关联、相互影响。

### 5.1 三高之间的关系

```
高性能是高并发和高可用的基础

┌─────────────────────────────────────────────┐
│                   高可用                     │
│    ┌─────────────────────────────────┐      │
│    │            高并发                │      │
│    │  ┌─────────────────────────┐    │      │
│    │  │        高性能            │    │      │
│    │  └─────────────────────────┘    │      │
│    └─────────────────────────────────┘      │
└─────────────────────────────────────────────┘
```

**高性能是基础**。一个系统如果连基本性能都做不好（RT 高、QPS 低），高并发和高可用就是空中楼阁。

**高并发是场景**。在实际请求量下的表现，高并发考察的是系统在压力下的稳定性和效率。

**高可用是保障**。系统在故障下的稳定性，高可用考察的是系统的容错能力和恢复能力。

### 5.2 场景举例

以电商秒杀为例：

- **高性能**：SQL 查询在 10ms 以内完成，Redis 操作在 1ms 以内完成
- **高并发**：秒杀瞬间 10 万人下单，系统不崩溃，请求不超时
- **高可用**：即使 Redis 挂了，系统依然能通过降级方案保证核心功能可用

三个目标缺一不可，共同构成一个「又快又稳又能扛」的系统。

## 六、面试实战

### 6.1 场景一：性能优化的完整思路

**面试官问**：「假设你的系统接口 RT 很高，你会怎么排查和优化？」

这道题考察的是对性能优化的整体认知，核心在于「思路」而非「具体命令」。

**标准回答框架**：

> 性能优化的第一步永远是**测量**，不是猜测。我会先建立基准线，记录当前的 RT、QPS、错误率等指标。
>
> 然后通过 APM 工具（如 SkyWalking）做全链路追踪，定位耗时最长的环节——是数据库、Redis、外部服务，还是业务逻辑。根据木桶原理，这个耗时最长的环节就是当前的最短板。
>
> 找到短板之后，再分析原因。数据库慢的话，看是慢查询还是连接池问题；Redis 慢的话，看是命中率低还是连接等待；外部服务慢的话，看是超时还是资源饱和。
>
> 优化之后，用同样的测量手段验证效果。只有数据证明优化有效，才算完成。
>
> 最后还要注意：优化是分层的，解决一个瓶颈后，下一个瓶颈就会浮现。所以这是一个持续迭代的过程，不是做一次就结束了。

### 6.2 场景二：高频误区

**面试官问**：「性能优化中有哪些常见误区？」

**常见误区**：

1. **迷信银弹**：有人说「加了缓存就快了」，有人说「上了微服务就稳了」。每个方案都有适用场景，没有银弹。
2. **过度优化**：追求数字上的极致，而忽视业务实际需求。99 分到 100 分的代价，往往比 60 分到 80 分还高。
3. **只管快不管稳**：把超时时间改短，RT 看起来低了，但错误率飙升。
4. **忽视底层基础**：问题出在机器配置太低、网络带宽不足，而不是代码。

## 总结

高性能优化是一套系统化的工程能力，涵盖数据库、JVM、IO、缓存、接口等多个维度。

学习高性能优化，要抓住两条主线：

1. **测量驱动 + 木桶原理**：先测量、定位短板、优化、验证，形成闭环
2. **原理理解 + 实践应用**：不仅知道怎么做，还要知道为什么这样做

高性能是高并发和高可用的基础。只有先把基础打牢，才能在复杂的分布式系统中游刃有余。
