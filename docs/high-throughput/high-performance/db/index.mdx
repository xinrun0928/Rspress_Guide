# 数据库性能优化

## 模块定位

数据库是整个系统性能优化的核心战场。

你可能花了一周优化了 JVM 参数，GC 停顿从 200ms 降到了 50ms——结果发现接口 RT 根本没变化，因为真正的瓶颈在数据库：一条慢 SQL 跑了 300ms，把所有优化都吞掉了。

反过来，你可能只需要加一个索引，就能让接口 RT 从 500ms 降到 50ms，效果比任何 JVM 调优都来得直接。

本模块覆盖数据库优化的两大方向：**MySQL 调优**和**NoSQL 与搜索引擎**，围绕「如何让数据库跑得更快」这个核心问题展开。

## 核心内容

本模块包含四个子模块，覆盖数据库优化的全链路：

|---|---|
| [慢查询定位](./sql-optimize/sql-slow) | 如何快速找到系统中的慢 SQL？ |
| [EXPLAIN 执行计划](./sql-optimize/explain) | type、key、Extra 字段代表什么含义？ |
| [filesort 与 temporary](./sql-optimize/filesort-temporary) | 看到这两个字段意味着什么？ |
| [JOIN 优化](./sql-optimize/join-optimize) | 多表 JOIN 如何避免大表扫描？ |
| [分页优化](./sql-optimize/pagination) | 深度分页为什么慢？如何优化？ |
| [批量操作优化](./sql-optimize/batch-optimize) | 大批量插入如何提升性能？ |
| [COUNT 优化](./sql-optimize/count-optimize) | COUNT(*) 为什么在大数据量时慢？ |
| [索引优化](./sql-optimize/index-optimize) | 索引失效的场景有哪些？ |

## 学习路径

```
sql-optimize/sql-slow（找问题）→ sql-optimize/explain（分析问题）
→ sql-optimize/index-optimize + join-optimize（解决问题）
→ mysql-tuning/（配置与架构层）
→ redis-optimize/（缓存层）
→ nosql-search/（扩展场景）
```

## 面试考察

数据库优化是 Java 面试中出现频率最高的领域之一：

**SQL 优化方向**：
- 「如何定位慢查询？」
- 「EXPLAIN 中 type=ALL 意味着什么？如何优化？」
- 「分页查询 OFFSET 很大时很慢，怎么解决？」
- 「索引失效的场景有哪些？」

**Redis 优化方向**：
- 「Redis 大 key 怎么处理？」
- 「Redis 的持久化策略选哪个？」
- 「Redis 集群槽迁移的过程？」

掌握本模块，你就掌握了面试中数据库相关问题的半壁江山。
