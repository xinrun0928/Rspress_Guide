# Redis 持久化：RDB 与 AOF 深度剖析

凌晨 3 点，线上 Redis 突然宕机。重启之后，你发现数据全丢了——没有开启持久化，或者 AOF 文件损坏了无法恢复。

这不是故事，这是很多团队踩过的真实坑。Redis 的持久化机制，是生产环境中**最容易被忽视、但一旦出问题影响最大的配置之一**。

为什么？因为 Redis 默认开启 RDB，开发者以为数据安全有保障，但 RDB 是定时快照，两次快照之间的数据可能全部丢失。AOF 每条都写太慢，everysec 折中又最多丢 1 秒数据。怎么选？

这篇文章，带你彻底搞懂 Redis 的两种持久化方式，以及 Redis 4.0 之后的混合持久化方案。

## 一、为什么 Redis 需要持久化？

Redis 的数据存储在内存中，而内存的特点是：**断电即失**。一旦机器重启、Redis 进程崩溃，或者运维误操作，内存中的数据就会全部丢失。

持久化机制的本质是：**将内存数据定期或实时写入磁盘**，在 Redis 重启时从磁盘恢复数据，实现数据的持久保存。

:::tip 持久化 vs 缓存
持久化和缓存是两个不同的概念。缓存是「更快但可能丢失」的数据存储，Redis 做缓存时可以不开持久化；但如果 Redis 是主数据库（存储核心业务数据），持久化是必须的。
:::

## 二、RDB：定时快照

### 2.1 核心原理

RDB（Redis Database Backup）是一种**定时快照**机制。它在指定的时间间隔内，将 Redis 内存中的**整个数据集**写入磁盘，生成一个压缩的二进制文件（`dump.rdb`）。

RDB 的触发方式有两种：

**手动触发**：`BGSAVE` 命令。Redis fork 一个子进程来执行快照操作，父进程继续处理请求，不阻塞。

```bash
127.0.0.1:6379> BGSAVE
Background saving started
```

**自动触发**：通过配置文件中的 `save` 规则自动触发。

```bash title="redis.conf"
# 格式：save <seconds> <changes>
# 在 900 秒内如果有 1 次写操作，则触发 BGSAVE
save 900 1
# 在 300 秒内如果有 10 次写操作，则触发 BGSAVE
save 300 10
# 在 60 秒内如果有 10000 次写操作，则触发 BGSAVE
save 60 10000
```

:::warning 注意
自动触发的 `save` 规则如果配置过多，会导致频繁 fork。生产环境中建议根据业务场景配置 1-2 条规则即可。
:::

### 2.2 RDB 的优缺点

**RDB 的优点**：

1. **恢复速度快**：RDB 文件是紧凑的二进制格式，数据恢复时只需要读取一个文件，不需要逐条执行命令。实测百万级数据恢复只需几秒钟。
2. **适合备份**：RDB 文件是完整的数据快照，适合每日备份、异地备份场景。
3. **fork 后台执行**：BGSAVE 在子进程中执行，不阻塞主进程，对线上请求影响小。
4. **性能高效**：fork + COW（Copy-On-Write）机制保证快照生成的高效性。

**RDB 的缺点**：

1. **可能丢失数据**：RDB 是定时快照，两次快照之间写入的数据会丢失。丢失的数据量取决于 `save` 配置的间隔时间。
2. **fork 开销**：fork 是一个重量级操作，会复制父进程的页表。对于大内存实例（数十 GB），fork 可能造成秒级的停顿。
3. **内存翻倍风险**：fork 瞬间，子进程需要复制父进程的页表结构（不是复制数据），如果在此期间有大量写操作，内存可能翻倍。
4. **无法实时持久化**：无法做到每秒级甚至每条命令级别的持久化。

### 2.3 RDB 的适用场景

| 场景 | 是否适合 RDB |
|---|---|
| Redis 作为缓存，数据可从 DB 重建 | ✅ 适合 |
| 数据允许分钟级丢失 | ✅ 适合 |
| 追求极致的恢复速度 | ✅ 适合 |
| 数据要求绝对不丢失 | ❌ 不适合 |
| 大内存实例（64GB+） | ⚠️ 需要谨慎配置 |

## 三、AOF：日志追加

### 3.1 核心原理

AOF（Append Only File）是**命令日志**机制。它将 Redis 收到的每个**写命令追加到文件末尾**，用日志的方式记录数据的变化过程。

```bash title="redis.conf"
# 开启 AOF
appendonly yes

# AOF 文件名
appendfilename "appendonly.aof"
```

当 Redis 重启时，会重放 AOF 文件中的所有命令来恢复数据。这就像一个「 redo log」，记录了所有让数据从零到达当前状态的操作。

### 3.2 三种刷盘策略

AOF 的核心问题是：**什么时候把日志刷到磁盘？**

```bash title="redis.conf"
# appendfsync 的取值
appendfsync always     # 每次写命令都刷盘
appendfsync everysec   # 每秒刷盘一次（默认）
appendfsync no         # 由操作系统决定何时刷盘
```

**`always`：每条命令都刷盘**

```bash
127.0.0.1:6379> SET name "redis"
# 立即触发 fsync，数据绝对不丢失
```

- **优点**：数据安全性最高，理论上最多丢失一条命令
- **缺点**：每次写操作都要等待磁盘写入，QPS 大幅下降。通常只能达到几百 QPS

**`everysec`：每秒刷盘一次**

- **优点**：最多丢失一秒的数据，兼顾性能和数据安全。Redis 默认推荐
- **缺点**：极端情况下可能丢失 1 秒数据
- **适用场景**：大多数生产环境的选择

**`no`：由操作系统决定**

- **优点**：性能最好
- **缺点**：数据安全性完全取决于操作系统，可能丢失大量数据
- **不推荐**：生产环境慎用

:::tip 生产推荐
生产环境建议使用 `appendfsync everysec`。这是 Redis 官方推荐的配置，在性能和安全性之间取得了最佳平衡。除非有特殊的数据安全要求，否则不要使用 `always`。
:::

### 3.3 AOF 的优缺点

**AOF 的优点**：

1. **数据安全性高**：everysec 模式下，最多丢失 1 秒数据；always 模式下最多丢失 1 条命令
2. **AOF 重写可压缩**：AOF 文件过大会触发重写，生成精简版本的命令日志
3. **命令日志可读**：AOF 文件是文本格式，可以用 `redis-check-aof` 工具修复损坏的文件

**AOF 的缺点**：

1. **AOF 文件通常比 RDB 大**：因为记录的是命令（文本格式），而 RDB 是二进制数据
2. **恢复速度慢**：需要重放所有命令，百万级数据可能需要几分钟
3. **写放大问题**：高并发写入时，频繁的 fsync 可能影响性能

### 3.4 AOF vs RDB 对比

| 特性 | RDB | AOF |
|---|---|---|
| 文件格式 | 二进制，紧凑 | 文本命令日志 |
| 数据安全性 | 较低（定时快照） | 较高（可配置） |
| 恢复速度 | 快（秒级） | 慢（分钟级） |
| 文件大小 | 小（压缩） | 大（包含冗余命令） |
| 资源消耗 | fork 开销 | 磁盘写入开销 |
| 写入方式 | 全量快照 | 增量追加 |

## 四、AOF 重写：让日志「减肥」

### 4.1 为什么要重写？

随着时间推移，AOF 文件会越来越大。原因有两个：

1. **写放大**：同一个 key 被修改多次，AOF 中会记录多次操作命令
2. **过期删除**：已过期的 key 仍然占据 AOF 空间

例如：

```bash
127.0.0.1:6379> SET counter 1    # AOF 记录
127.0.0.1:6379> INCR counter    # AOF 记录
127.0.0.1:6379> INCR counter      # AOF 记录
127.0.0.1:6379> SET counter 100  # AOF 记录
```

4 条命令，最终 counter 的值是 100。**AOF 重写会压缩成一条命令**：

```bash
127.0.0.1:6379> SET counter 100  # 重写后只有一条
```

### 4.2 重写原理

AOF 重写（`BGREWRITEAOF`）的原理是：**fork 子进程读取当前内存数据，生成最小操作集写入新的 AOF 文件**。

```bash
127.0.0.1:6379> BGREWRITEAOF
Background append only file rewriting started
```

重写过程：

1. **fork 子进程**：子进程拥有当前内存数据的快照
2. **读取数据写新文件**：子进程遍历内存数据，为每个 key 生成最简命令
3. **父进程追加新命令**：重写期间，父进程继续接收写命令，追加到 AOF 缓冲区
4. **重写完成**：子进程写完新 AOF 后，父进程将缓冲区的命令追加到新 AOF
5. **替换文件**：用新 AOF 替换旧 AOF

### 4.3 自动触发配置

```bash title="redis.conf"
# 当 AOF 文件大于 64MB 时，且比上次重写后增长了 100%，则触发重写
auto-aof-rewrite-percentage 100
auto-aof-rewrite-min-size 64mb
```

:::warning 自动重写的坑
自动重写触发时，如果 AOF 文件非常大（比如几 GB），重写过程会消耗大量 CPU 和内存，可能导致 Redis 卡顿。建议：
1. 配置合理的 `auto-aof-rewrite-min-size`，避免频繁重写
2. 监控 AOF 文件大小，在业务低峰期手动触发重写
3. 使用 Redis 4.0+ 的混合持久化
:::

## 五、混合持久化：鱼与熊掌兼得

### 5.1 混合持久化的诞生

RDB 恢复快但丢数据，AOF 数据安全但恢复慢。能不能两者兼顾？

Redis 4.0 给出了答案：**混合持久化**。

```bash title="redis.conf"
# 开启混合持久化
aof-use-rdb-preamble yes
```

### 5.2 原理详解

混合持久化的核心思想：**AOF 重写时，先写 RDB 格式的完整数据，再追加 AOF 增量命令**。

```bash title="混合持久化文件结构"
# 文件开头是 RDB 格式（二进制）
# 文件后续是 AOF 格式（文本）
[RDB binary data][AOF text commands...]
```

具体流程：

1. **AOF 重写触发时**，子进程先写入 RDB 格式的完整数据快照
2. **快照完成后**，后续的增量操作以 AOF 命令格式追加
3. **Redis 重启时**，先加载 RDB 部分快速恢复，再重放后续 AOF 命令补全数据

### 5.3 混合持久化的优势

| 优势 | 说明 |
|---|---|
| 快速恢复 | RDB 部分秒级加载，无需重放数万条命令 |
| 数据完整 | AOF 增量部分保证数据不丢失 |
| 文件紧凑 | RDB 压缩 + AOF 增量，比纯 AOF 小 |

### 5.4 兼容性问题

:::danger 重要提醒
混合持久化生成的文件只能被 **Redis 4.0+** 读取。如果将混合文件拷贝到 Redis 3.x 版本，Redis 无法识别，会报错。

如果 Redis 版本混用（如部分节点还在用 3.x），**禁止开启 `aof-use-rdb-preamble`**。
:::

## 六、生产环境持久化配置实战

### 6.1 配置模板

根据不同业务场景，推荐以下配置：

**场景一：Redis 作为缓存**

```bash title="redis.conf"
# RDB 配置
save 900 1
save 300 10
save 60 10000

# AOF 配置（可选，缓存场景可以不开启）
# appendonly no

# 关闭 RDB
# save ""
```

**场景二：Redis 作为主数据库（数据重要）**

```bash title="redis.conf"
# RDB 配置（作为备份）
save 900 1
save 300 10

# AOF 配置（必须开启）
appendonly yes
appendfsync everysec

# 混合持久化（Redis 4.0+）
aof-use-rdb-preamble yes

# AOF 自动重写
auto-aof-rewrite-percentage 100
auto-aof-rewrite-min-size 64mb
```

**场景三：Redis 5.0+ 推荐配置**

```bash title="redis.conf"
# 混合持久化 + AOF everysec
appendonly yes
appendfsync everysec
aof-use-rdb-preamble yes

# RDB 快照作为备份
save 900 1
save 300 10
save 60 10000
```

### 6.2 持久化相关的运维命令

```bash
# 查看持久化状态
127.0.0.1:6379> INFO persistence

# 查看 RDB 是否正在执行
127.0.0.1:6379> LASTSAVE

# 手动触发 RDB 快照
127.0.0.1:6379> BGSAVE

# 手动触发 AOF 重写
127.0.0.1:6379> BGREWRITEAOF

# 检查 AOF 文件完整性
redis-check-aof --fix appendonly.aof

# 检查 RDB 文件完整性
redis-check-rdb dump.rdb
```

### 6.3 监控指标

```bash
# 查看持久化关键指标
127.0.0.1:6379> INFO persistence
aof_current_size: 5242880
aof_base_size: 4194304
aof_pending_rewrite: 0
rdb_changes_since_last_save: 100
```

| 指标 | 含义 |
|---|---|
| `aof_current_size` | AOF 文件当前大小 |
| `aof_base_size` | AOF 重写后的基础大小 |
| `rdb_changes_since_last_save` | 上次 RDB 后累计的写操作数 |

## 七、面试实战

### 7.1 场景一：持久化方式的选择

**面试官问**：「Redis 的持久化方式有哪些？生产环境怎么选择？」

这道题考察的是对持久化机制的全面理解。

**普通回答**：有 RDB 和 AOF 两种。AOF 更安全，生产用 AOF。

**深入回答**：

> Redis 有两种持久化方式：RDB 和 AOF。
>
> **RDB 是定时快照**，将内存数据全量写入磁盘，生成紧凑的二进制文件。优点是恢复快（秒级），缺点是可能丢失最后一次快照后的数据。
>
> **AOF 是命令日志**，将每个写命令追加到文件末尾。可以通过配置 `appendfsync` 控制刷盘频率：`always`（每条都刷，最安全但性能差）、`everysec`（每秒，默认推荐）、`no`（操作系统决定）。
>
> 生产环境的选择：
> - 如果 Redis 是缓存，数据可丢失，用 RDB 就够了
> - 如果 Redis 存储核心业务数据，用 **AOF everysec + RDB 备份**
> - Redis 4.0+ 推荐 **混合持久化**（`aof-use-rdb-preamble yes`），兼顾快速恢复和数据完整
>
> 绝对不能的做法是**不开持久化**，否则 Redis 故障后数据全丢。

### 7.2 场景二：RDB 和 AOF 的区别

**面试官问**：「RDB 和 AOF 的区别是什么？各自的优缺点？」

这道题是高频基础题，重点考察对两种机制的理解深度。

**回答要点**：

| 对比维度 | RDB | AOF |
|---|---|---|
| 持久化方式 | 全量快照 | 增量命令日志 |
| 文件格式 | 二进制压缩 | 文本命令 |
| 数据安全性 | 较低 | 较高（可配置） |
| 恢复速度 | 快 | 慢 |
| 文件大小 | 小 | 大 |
| 性能影响 | fork 开销 | 磁盘 IO 开销 |
| 适用场景 | 缓存、快速恢复 | 核心数据存储 |

### 7.3 场景三：混合持久化原理

**面试官问**：「Redis 的混合持久化是什么原理？有什么优势？」

这道题考察对新特性的理解。

**回答要点**：

> 混合持久化是 Redis 4.0 引入的特性，通过配置 `aof-use-rdb-preamble yes` 开启。
>
> 原理是：AOF 重写时，先将当前数据以 RDB 二进制格式写入文件开头，后续的增量操作继续以 AOF 文本格式追加。这样文件中既有全量数据（RDB 部分），又有增量更新（AOF 部分）。
>
> 优势有两个：
> 1. **恢复快**：Redis 启动时直接加载 RDB 部分即可快速恢复
> 2. **数据完整**：RDB 之后的增量操作通过 AOF 追加，不丢失数据
>
> 需要注意的是，混合持久化的文件只有 Redis 4.0+ 才能识别，老版本不兼容。

## 总结

Redis 的持久化机制，是生产环境的必修课。

**RDB** 是定时快照，恢复快但可能丢数据，适合缓存场景。**AOF** 是命令日志，数据安全但恢复慢，`everysec` 是生产推荐配置。**混合持久化**（Redis 4.0+）用 RDB 快速恢复 + AOF 增量补全，是当前的最佳实践。

无论选择哪种方式，**绝对不能不开持久化**。Redis 故障后数据全丢的代价，远大于持久化带来的性能开销。

理解了持久化机制，你就掌握了 Redis 数据安全的命门。下一个问题：Redis 集群环境下，持久化和高可用如何结合？这个问题，值得你继续深入。
