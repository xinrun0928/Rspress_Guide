# Redis 大 key 问题：发现、危害与解决方案

凌晨 3 点，你被一通电话叫醒：「Redis 集群有个节点 CPU 突然打满，整个服务雪崩了。」

你赶紧登录机器查看，发现一个 key 的内存占用高达 **500MB**，每次查询这个 key 都要消耗 200ms 以上。更糟糕的是，这个大 key 还在持续写入，内存占用越来越大。

这是 Redis 生产环境的经典场景——**大 key 问题**。它不像慢查询那样明显，却能在关键时刻给系统致命一击。今天，我们就来彻底解决它。

## 一、什么是大 key

### 1.1 大 key 的定义

关于大 key，业界没有统一标准，但有两个常用的判断维度：

| 类型 | 阈值 | 说明 |
|---|---|---|
| |string| 类型 | 单个 value 超过 10KB | string 类型的 value 过大，会导致内存分配和序列化开销 |
| |hash|/|list|/|set|/|zset| 类型 | 元素数量超过 5000 | 复合类型包含的元素过多，操作复杂度升高 |

:::tip 阈值不是绝对的

10KB 和 5000 只是经验值。在实际生产环境中，要根据业务场景和 Redis 版本调整。比如在内存紧张的集群环境下，1KB 的 string 也可能成为问题。

:::

### 1.2 大 key 的三大危害

为什么大 key 是 Redis 的「隐形杀手」？我们从三个角度来分析。

**危害一：内存不均，集群倾斜**

在 Redis Cluster 模式下，数据按 slot 分布。如果某个 slot 集中了大量大 key，这个 slot 的内存占用会远高于其他 slot。迁移 slot 时，不仅耗时更长，还可能导致目标节点内存溢出。

更严重的是——**一个节点的内存满了，会触发淘汰策略，淘汰掉本不该淘汰的热数据**。热点数据没了，所有请求都打到后端数据库，系统直接崩溃。

**危害二：操作超时，阻塞主线程**

Redis 是单线程模型（6.0 之前），核心操作是串行执行的。当你执行 `HGETALL` 获取一个包含百万元素的 hash 时，主线程会被这个操作阻塞 几百毫秒甚至几秒。这期间所有其他请求都在排队。

你以为只有复杂操作才会阻塞？错了。**即使是简单的 DEL 操作，删除一个大 key 也会阻塞主线程 几十毫秒。**

**危害三：内存达上限，淘汰失控**

Redis 内存上限（`maxmemory`）是固定的。当大 key 占用了大量内存，可用内存减少，触发淘汰策略。Redis 的淘汰策略是按一定概率执行的，可能淘汰掉正在使用的热数据，导致缓存穿透。

:::warning 淘汰策略不是万能药

很多人以为给 Redis 加了内存上限就安全了。实际上，如果大 key 占比过高，淘汰策略会把热数据和冷数据一起淘汰。正确的做法是：先解决大 key，再配置合理的内存策略。

:::

## 二、如何发现大 key

发现问题比解决问题更重要。这一节介绍几种发现大 key 的方法，从简单到专业。

### 2.1 `redis-cli --bigkeys`：快速扫描

这是 Redis 自带的工具，可以扫描整个实例，按类型找出最大的 key。

```bash
redis-cli --bigkeys
```

输出大概是这个样子：

```
# Scanning the entire keyspace to find biggest keys as well as
# average sizes per key type.  You can use -i 0.1 to sleep 0.1 sec
# per 100 SCAN commands (usually 100 keys, so 10 sec by default)

-------- summary -------
Sampled 100000 keys in the database
Biggest string found: "user:session:abc123"  524288 bytes
Biggest   list found: "feed:timeline:global"  10240 items
Biggest    set found: "hot:products:ids"  5000 members
Biggest   hash found: "user:profile:10001"  128 fields
```

:::tip 优点与局限

优点：零门槛，Redis 自带，一行命令就能跑。

局限：只显示每个类型的最大 key，不显示所有大 key 的完整分布；采样模式，可能漏掉一些 key。

:::

### 2.2 `MEMORY USAGE`：精确计算

知道某个 key 可能有问题，可以用 `MEMORY USAGE` 命令精确计算它的内存占用。

```bash
MEMORY USAGE user:profile:10001
```

```
(integer) 524288
```

返回的是字节数（524288 字节 = 512KB）。

配合 `SCAN` 命令，可以批量扫描所有 key 的内存占用：

```java
// Java 示例：扫描内存占用超过 100KB 的 key
try (Jedis jedis = new Jedis("localhost", 6379)) {
    ScanParams params = new ScanParams().count(100).match("*");
    String cursor = "0";

    do {
        ScanResult<String> result = jedis.scan(cursor, params);
        cursor = result.getCursor();

        for (String key : result.getResult()) {
            Long memory = jedis.memory(key);
            if (memory != null && memory > 100 * 1024) {
                System.out.println(key + ": " + memory + " bytes");
            }
        }
    } while (!"0".equals(cursor));
}
```

### 2.3 `SCAN` + `DBSIZE`：生产安全扫描

**绝对不要在生产环境用 `KEYS` 命令！**

`KEYS` 会遍历整个 keyspace，在百万级 key 的实例上可能阻塞主线程几十秒。而 `SCAN` 是渐进式的，每次只返回一小批 key，可以安全地在生产环境使用。

```bash
# 用 SCAN 配合 Lua 脚本，找出所有大 key
redis-cli --scan | while read key; do
    size=$(redis-cli --no-raw-memory usage "$key" 2>/dev/null)
    if [ ! -z "$size" ] && [ "$size" -gt 102400 ]; then
        echo "$size $key"
    fi
done | sort -n -r | head -20
```

### 2.4 `redis-rdb-tools`：离线分析 RDB 文件

如果你的 Redis 开启了 RDB 持久化，可以用这个工具离线分析 RDB 文件，完全不影响生产环境。

```bash
# 生成内存报告
rdb -c memory /var/redis/dump.rdb --bytes 102400 -f big_keys.csv

# 查看 Top 20 大 key
cat big_keys.csv | head -21
```

输出包含 key 名、类型、内存占用、编码方式等详细信息。

:::info RDB 文件获取

可以用 `BGSAVE` 命令让 Redis 在后台生成 RDB 文件，然后去 `dir` 配置的目录下拿文件。注意：RDB 文件是某个时刻的数据快照，不是实时的。

:::

### 2.5 主动监控：在写入端埋点

以上方法都是「事后发现」，对于新上线的大 key 可能来不及。有条件的话，在写入端埋点统计 key 大小：

```java
// 写入时记录 key 大小
public void setWithMonitoring(Jedis jedis, String key, String value) {
    jedis.set(key, value);

    // 记录 key 大小到监控日志
    long sizeInBytes = Optional.ofNullable(value)
        .map(String::getBytes)
        .map(bytes -> bytes.length)
        .orElse(0);

    if (sizeInBytes > 10 * 1024) {
        log.warn("Big key detected: key={}, size={}KB", key, sizeInBytes / 1024);
        // 告警通知
        alertService.sendAlert("Big key detected: " + key);
    }
}
```

这样新上线的代码如果产生了大 key，可以第一时间告警。

## 三、大 key 的解决方案

发现问题后，关键是解决。这一节按类型讲解大 key 的处理方案。

### 3.1 string 类型大 key

string 类型是最容易出现大 key 的场景。比如有人把整个用户画像 JSON 存进了一个 key。

**方案一：压缩存储**

对于文本类数据，可以用 gzip 或 lz4 压缩后再存储：

```java
// 压缩存储
public void saveCompressed(Jedis jedis, String key, String data) {
    byte[] compressed = compress(data.getBytes(StandardCharsets.UTF_8));
    jedis.set(key.getBytes(), compressed);
}

// 解压读取
public String getDecompressed(Jedis jedis, String key) {
    byte[] compressed = jedis.get(key.getBytes());
    if (compressed == null) return null;
    return new String(decompress(compressed), StandardCharsets.UTF_8);
}
```

压缩比通常能达到 3-5 倍，10MB 的 JSON 压缩后可能只有 2MB。

**方案二：拆分存储**

按业务维度拆分，不要把所有数据存到一个 key：

```java
// 不要这样：一个大 key 存所有用户数据
// jedis.set("user:profile:10001", allUserData);

// 拆分成多个 key
jedis.hset("user:profile:10001", "basic", basicInfo);
jedis.hset("user:profile:10001", "social", socialInfo);
jedis.hset("user:profile:10001", "behavior", behaviorData);
```

拆分的原则是：**按访问频率和业务边界拆分，热数据放一起，冷数据放一起**。

### 3.2 hash 类型大 key

hash 类型大 key 是最常见的场景。比如一个 hash 存了用户的所有行为日志，字段数量上百万。

**方案一：拆分为小 hash**

```java
// 不要这样：一个 hash 包含所有字段
// jedis.hset("user:logs:10001", "field1", "value1");
// jedis.hset("user:logs:10001", "field2", "value2");
// ...

// 拆分成多个小 hash
String userId = "10001";
int totalFields = 100000;
int chunkSize = 5000;
int chunkCount = (totalFields + chunkSize - 1) / chunkSize;

for (int i = 0; i < chunkCount; i++) {
    String chunkKey = String.format("user:logs:%s:chunk:%d", userId, i);
    // 写入 chunk i 的数据
}
```

**方案二：用 `HSCAN` 替代 `HGETALL`**

如果暂时无法拆分存量数据，读取时用 `HSCAN` 分批获取：

```java
// 用 HSCAN 分批读取，避免阻塞
public Map<String, String> hscanAll(Jedis jedis, String key) {
    Map<String, String> result = new HashMap<>();
    ScanParams params = new ScanParams().count(100);
    String cursor = "0";

    do {
        ScanResult<Map.Entry<String, String>> scanResult = jedis.hscan(key, cursor, params);
        cursor = scanResult.getCursor();
        result.putAll(scanResult.getResult());
    } while (!"0".equals(cursor));

    return result;
}
```

### 3.3 list 类型大 key

list 类型常用于消息队列、任务队列等场景。如果队列积压大量消息，就会变成大 key。

**方案一：分段存储**

```java
// 不要这样：一个 list 存所有消息
// jedis.lpush("task:queue", task);

// 分段存储，用 LRANGE 分批消费
public void lpushSegmented(Jedis jedis, String task) {
    // 找到当前最新的 chunk
    long size = jedis.llen("task:queue");
    String chunkKey = "task:queue:" + (size / 1000);

    jedis.lpush(chunkKey, task);
    jedis.expire(chunkKey, 86400); // 24 小时过期
}

// 分批消费
public List<String> lrangeBatch(Jedis jedis, String chunkKey, int batchSize) {
    List<String> batch = jedis.lrange(chunkKey, 0, batchSize - 1);
    if (!batch.isEmpty()) {
        jedis.ltrim(chunkKey, batch.size(), -1); // 删除已读取部分
    }
    return batch;
}
```

**方案二：使用 Stream 类型**

Redis 5.0 引入了 Stream 类型，对大数据量场景更友好，支持消费组、消息 ID 等特性。

### 3.4 zset 类型大 key

zset 常用于排行榜、延时队列等场景。如果排行榜数据量很大，可以用分片策略。

**按分数区间分片**：

```java
// 用户 ID 到分数的映射存到多个 zset，按分数区间分片
public void addScoreSegmented(Jedis jedis, String userId, double score) {
    // 分数在 0-1000 的放 zset1，1000-2000 放 zset2，以此类推
    int shardId = (int) (score / 1000);
    String shardKey = "leaderboard:shard:" + shardId;

    jedis.zadd(shardKey, score, userId);
}

// 查询排名前 N 的用户，需要合并多个分片的结果
public List<Map.Entry<String, Double>> getTopN(Jedis jedis, int n) {
    List<String> keys = new ArrayList<>();
    for (int i = 0; i < 10; i++) {
        keys.add("leaderboard:shard:" + i);
    }

    return jedis.zrevrangeWithScores(keys, 0, n - 1)
        .stream()
        .sorted(Map.Entry.<String, Double>comparingByValue().reversed())
        .limit(n)
        .collect(Collectors.toList());
}
```

### 3.5 大 key 的删除策略

删除大 key 同样有讲究。用错了方法，可能导致主线程阻塞。

**不要用 `DEL`，用 `UNLINK`**

`UNLINK` 是 Redis 4.0 引入的命令，它不会立即释放内存，而是把 key 标记为删除，然后在后台异步释放内存。

```bash
# BAD: 阻塞主线程
DEL big_key

# GOOD: 异步删除，不阻塞
UNLINK big_key
```

**渐进式删除：SCAN + DEL**

如果 key 数量太多，UNLINK 的后台线程也可能压力过大。这时候需要渐进式删除：

```bash
# 用 SCAN 找到 key，循环删除
redis-cli --scan --pattern "big_key_pattern:*" | while read key; do
    redis-cli unlink "$key"
done
```

或者用 Lua 脚本控制删除速度：

```lua
-- 每次删除 100 个，sleep 1 秒，避免对主线程造成压力
local keys = redis.call('SCAN', '0', 'MATCH', KEYS[1], 'COUNT', '100')
for _, key in ipairs(keys[2]) do
    redis.call('UNLINK', key)
end
redis.call('EXPIRE', keys[1], 0) -- 删除游标 key
return keys[1]
```

## 四、生产环境注意事项

### 4.1 删除操作对 AOF/RDB 的影响

大 key 删除时，Redis 会写大量 DEL/UNLINK 命令到 AOF 文件。如果 AOF 重写正好发生，可能导致 AOF 文件膨胀。

:::warning AOF 重写与大 key 删除

在执行 `BGREWRITEAOF` 之前，如果有大 key 删除操作，这些删除命令会被写入 AOF。重写时，新的 AOF 文件会包含当前数据的完整快照。

**最佳实践**：在大 key 删除后，主动触发一次 AOF 重写，保证 AOF 文件的紧凑性。

:::

### 4.2 集群环境下的 slot 倾斜

Redis Cluster 的 slot 倾斜会导致数据分布不均。解决方案：

1. **使用 `redis-cli --cluster rebalance`** 重新平衡 slot
2. **监控 slot 分布**：用 `redis-cli cluster nodes` 或 `CLUSTER SLOTS` 查看分布
3. **从源头避免**：新上线的 key 设计要均匀，不要把所有大 key 写到同一个 slot

## 五、面试追问预判

### 追问一：大 key 删除会阻塞吗？

会，但有办法避免。`DEL` 会阻塞，`UNLINK` 不会阻塞（异步删除）。如果要删除大量 key，用渐进式删除（`SCAN + UNLINK`）。

### 追问二：大 key 会影响 RDB 快照吗？

会。`BGSAVE` 时，如果有大 key 写入，会导致 fork 出来的子进程 COW（Copy-On-Write）内存膨胀。如果大 key 占比超过 Redis 内存的 30%，建议在低峰期做 `BGSAVE`。

### 追问三：如何监控 Redis 实例的大 key？

三个方向：
1. **周期性扫描**：用 `redis-cli --bigkeys` 或自定义脚本定时扫描
2. **慢查询日志**：配置 `slowlog-log-slower-than`，监控超过阈值的操作
3. **主动埋点**：在写入端统计 key 大小，超过阈值告警

## 总结

大 key 是 Redis 生产环境的常见问题，核心是**预防 + 发现 + 治理**三步走。

**预防**：设计阶段就要考虑 key 的大小，按业务维度拆分，不要把所有鸡蛋放一个篮子。

**发现**：用 `redis-cli --bigkeys`、`MEMORY USAGE`、`redis-rdb-tools` 等工具定期扫描，主动埋点监控写入端。

**治理**：string 类型压缩或拆分，hash/list/zset 类型按维度拆分，删除时用 `UNLINK` 替代 `DEL`，大量删除时渐进式执行。

掌握了这些，你就有了应对 Redis 大 key 问题的完整武器库。
