# Redis 内存优化：看不见的内存杀手

某日清晨，运维告警：Redis 内存使用率超过 90%。你登录机器一看，`used_memory` 显示 10GB，但 `INFO memory` 里 `used_memory_rss`（实际物理内存占用）却是 18GB。

**明明只存了 10GB 数据，为什么占用了 18GB 物理内存？**

多出来的 8GB 去哪了？是 Redis 泄露了？还是机器被攻击了？

都不是。多出来的内存，有一个你可能没听说过的名字——**内存碎片（Memory Fragmentation）**。

这是 Redis 内存优化中最容易被忽视的问题，也是面试中常考的细节。这篇文章，我们来彻底拆解 Redis 的内存消耗构成，以及如何对症下药做优化。

## 一、Redis 内存消耗构成

要优化 Redis 内存，首先要搞清楚 Redis 的内存都消耗在哪里。Redis 的内存消耗可以分为五大类。

### 1.1 数据内存

数据内存是 Redis 中存储实际业务数据的空间。这是我们最熟悉的部分，也是大多数人对 Redis 内存的全部认知。

数据内存的大小取决于三个因素：

**key 的数量**：每个 key 本身有固定的开销（RedisObject + key 名字符串）。一个空 key 也要占用几十字节。

**value 的大小**：value 越大，占用越多。这是大头。

**数据类型**：不同数据结构的底层编码不同，内存效率也不同。比如一个只包含几个元素的 Hash，Redis 会用 `ziplist` 编码，比完整的 `hashtable` 节省大量内存。

```bash
# 查看 Redis 内存使用详情
INFO memory
# 输出关键字段：
# used_memory: 1073741824           # 数据申请的内存（字节）
# used_memory_human: 1.00G          # 人类可读格式
# used_memory_rss: 1610612736       # 物理内存占用（RSS）
# mem_fragmentation_ratio: 1.50     # 内存碎片率
# mem_allocator: jemalloc-5.2.1     # 内存分配器
```

### 1.2 额外内存开销

除了存储数据本身，Redis 在管理数据时还会消耗额外的内存，主要来自三个方面：

**过期与淘汰管理**：Redis 使用字典（dict）来管理 key 的过期时间（`expires` dict）。如果大量 key 设置了过期时间，这个 dict 本身就会占用可观的内存。每个 key 的过期信息约占 8 字节。

**主从复制缓冲区**：如果当前 Redis 是主库，它会为每个从库维护一个复制缓冲区（replication backlog buffer）。这个缓冲区用于断点续传，默认大小是 1MB，但可以通过配置调大。`client-output-buffer-limit replica 256mb 64mb 60` 控制从库输出缓冲区。

**Lua 脚本占用**：如果执行 Lua 脚本，Redis 需要为脚本分配额外的内存。这个内存和脚本大小、执行过程中的中间对象数量相关。

```bash
# 查看 keyspace 信息，了解 key 数量和过期 key 占比
INFO keyspace
# db0: keys=100000,expires=50000,avg_ttl=3600000
# 10 万个 key，其中 5 万个有过期时间
```

### 1.3 缓冲区内存

Redis 在处理客户端请求和持久化操作时，会使用各种缓冲区。这些缓冲区也是内存消耗的重要来源。

| 缓冲区类型 | 用途 | 大小 | 控制参数 |
|---|---|---|---|
| `client buffer` | 客户端输入/输出缓冲区 | 动态，默认 `&lt;` 10MB | `client-query-buffer-limit` |
| `aof buffer` | AOF 写缓冲 | 动态，取决于写入量 | 无法直接配置 |
| `rdb buffer` | RDB 生成时的临时缓冲 | 与数据集大小相关 | `client-output-buffer-limit` |
| `replica buffer` | 主库给从库的复制缓冲 | 动态 | `client-output-buffer-limit replica` |

`client buffer` 的大小最需要关注。如果客户端发送的命令特别大（比如一次发送一个 10MB 的字符串），Redis 会分配对应的缓冲区。如果客户端出问题（如发送速度过快或处理速度过慢），缓冲区会持续膨胀，最终导致 Redis OOM。

```bash
# 查看客户端连接和缓冲区信息
INFO clients
# connected_clients: 50
# client_longest_output_list: 0          # 客户端输出缓冲区最大长度
# client_biggest_input_buf: 0              # 客户端最大输入缓冲区
# blocked_clients: 0                       # 被阻塞的客户端数（BLPOP等命令）
```

:::tip 警惕异常客户端

如果监控发现 `client_biggest_input_buf` 数值异常大（如几十 MB），说明某个客户端发送的命令过大，或者处理速度跟不上接收速度。这通常是应用代码 bug 的信号（如循环调用了一个返回大量数据的命令）。

:::

### 1.4 进程本身的开销

Redis 主进程本身也会占用内存，主要包括：

**主进程常驻内存**：Redis 可执行文件和依赖库的内存映射，这部分通常很小（几十 MB）。

**COW（Copy-On-Write）开销**：这是最容易踩坑的地方。当 Redis 触发 `BGSAVE` 或 `BGREWRITEAOF` 时，Redis 会 fork 一个子进程来执行持久化操作。Linux 的 fork 机制使用 COW 策略：**fork 时父子进程共享内存页面，只有当某个进程修改了页面时，才会复制对应的内存页面。**

这意味着 fork 期间，Redis 的内存消耗会短暂翻倍。如果 fork 时 Redis 内存是 10GB，COW 机制下可能瞬间变成 20GB。如果机器物理内存不足，会触发 OOM。

这就是为什么**生产环境 Redis 的 `maxmemory` 不能设置得太接近机器物理内存**。一定要预留足够的空间给 COW 开销，通常建议预留 30%-50%。

```bash
# 查看 fork 操作相关统计
INFO stats
# latest_fork_usec: 12345               # 最近一次 fork 耗时（微秒）
# 如果 fork 耗时过长，说明内存越大或机器负载越高
```

### 1.5 内存碎片

这是本文开头问题的答案。

Redis 使用 `jemalloc` 作为默认内存分配器。jemalloc 的分配策略是按固定大小块（8KB 页面）分配内存。但实际业务中，key 的大小是随意的——有的 key 只存几个字节，有的 key 存几 KB 数据。

当大量 key 被修改大小或被删除后，jemalloc 分配的页面中会出现很多**无法被利用的空洞**。这些空洞累积起来，就是内存碎片。

```
jemalloc 页面分配示意：

分配给 key1（100B）  分配给 key2（50B）  分配给 key3（200B）
[■■■■■■■■■■■......][■■■■■■■..........][■■■■■■■■■■■■■■■....]
                          ↑                        ↑
                       空洞（碎片）              空洞（碎片）

实际占用：100+50+200 = 350B
jemalloc 分配：4096B（1 页）
内存碎片：4096 - 350 = 3746B
```

**内存碎片率** = `used_memory_rss / used_memory`

- **1.0-1.1**：理想状态，几乎无碎片
- **1.1-1.5**：正常范围，可以接受
- **> 1.5**：碎片过多，需要优化
- **> 5.0**：严重碎片，必须处理

## 二、内存碎片问题详解

### 2.1 碎片是如何产生的

内存碎片的产生主要有三个原因。

**原因一：key 的频繁修改和删除**

当你反复 `SET` 一个 key 的 value（新值和旧值大小不同），或者频繁 `DEL` 删除 key，jemalloc 分配的内存块就会出现碎片。新值可能比旧值大，jemalloc 会分配新的内存块，旧块就变成碎片。

**原因二：过期 key 的处理**

Redis 的过期 key 处理有两种策略（详见「Redis 过期策略」章节）：惰性删除和定期删除。惰性删除不立即释放内存，定期删除也只是抽查释放一部分。大量的过期 key 在实际删除前，内存块已经被标记为不可用，形成碎片。

**原因三：数据大小不均匀**

业务数据天然不均匀：有些 Hash 只有几个字段，有些有几十个字段；有些 String 只有几个字节，有些存了一整张图片。jemalloc 的固定页面分配策略会导致这些不同大小的数据在页面中留下碎片。

### 2.2 如何判断碎片是否严重

Redis 提供了 `mem_fragmentation_ratio` 指标，这是判断碎片率的核心依据：

```bash
INFO memory | grep fragmentation
# mem_fragmentation_ratio: 1.65
# mem_fragmentation_ratio: 0.97  （小于 1 的情况，通常是 Redis 刚释放了大量内存）
```

| 碎片率 | 状态 | 处理建议 |
|---|---|---|
| `&lt;` 1.0 | Redis 刚释放内存 | 正常，观察即可 |
| 1.0-1.1 | 接近零碎片 | 理想状态 |
| 1.1-1.5 | 正常范围 | 可以接受 |
| 1.5-3.0 | 碎片偏多 | 考虑优化 |
| `>` 3.0 | 严重碎片 | 必须处理 |
| `>` 5.0 | 极端严重 | 立即处理 |

### 2.3 碎片优化的三种方案

**方案一：MEMORY PURGE（手动触发内存整理）**

```bash
# 手动触发内存碎片整理（Redis 4.0+）
MEMORY PURGE
```

`MEMORY PURGE` 命令会请求 jemalloc 触发内存整理，将碎片页面合并。这个操作会阻塞 Redis（虽然通常很短），在生产环境执行要谨慎。

```bash
# 执行前后对比碎片率
INFO memory | grep mem_fragmentation_ratio
MEMORY PURGE
INFO memory | grep mem_fragmentation_ratio
```

**方案二：开启自动碎片整理（activedefrag）**

Redis 6.4+ 支持自动碎片整理：

```bash
# 开启自动碎片整理
CONFIG SET activedefrag yes

# 配置碎片整理的触发条件（可选）
CONFIG SET active-defrag-ignore-bytes 100mb    # 碎片超过 100MB 时触发
CONFIG SET active-defrag-threshold-lower 10    # 碎片率超过 10% 时触发
CONFIG SET active-defrag-cycle-min 25          # 整理占用 CPU 最低 25%
CONFIG SET active-defrag-cycle-max 75          # 整理占用 CPU 最高 75%
```

:::warning activedefrag 的代价

自动碎片整理会消耗 CPU 资源。如果 Redis 已经负载较高，开启 activedefrag 可能导致性能下降。需要在性能和碎片率之间找平衡。

:::

**方案三：重启 Redis（最彻底但有代价）**

重启 Redis 会清空所有内存分配，重新加载数据，碎片率瞬间归零。但缺点是：

- 持久化数据会丢失（除非用 RDB/AOF 恢复）
- 服务短暂不可用
- 如果碎片是由业务代码引起的（如 key 大小变化频繁），重启后碎片会重新累积

**最佳策略**：日常用 `MEMORY PURGE` 保持碎片率在合理范围，生产环境提前规划好在低峰期重启的计划，同时配合自动碎片整理作为长期手段。

### 2.4 碎片与 maxmemory 的关系

很多人不知道的是：**内存碎片会占用 `maxmemory` 的配额**。

假设你设置了 `maxmemory 10gb`，实际 `used_memory` 是 8GB，但碎片率 1.5，`used_memory_rss` 就是 12GB。这时候 Redis 已经接近 `maxmemory` 限制了，新的写入会被拒绝（触发淘汰策略）。

这就是为什么 `maxmemory` 要设置得保守一点——**不仅要容纳数据，还要容纳碎片**。如果你的数据实际占用 8GB，碎片率预期 1.3，`maxmemory` 至少要设置为 11GB。

## 三、内存优化策略

### 3.1 合理选择数据结构

这是最根本的优化手段。同样的数据，用不同的数据结构存储，内存占用可能差几倍甚至几十倍。

**String vs Hash：批量存储用户属性的选择**

存储一个用户的信息（id、name、age、city），有多种方式：

```bash
# 方式一：String 逐个存储（每个 key 独立，key 名占用大量内存）
SET user:1001:id   "1001"
SET user:1001:name "张三"
SET user:1001:age  "28"
SET user:1001:city "北京"
# 4 个 key，每个 key 名 "user:1001:xxx" 占 12+ 字节，4 个 = 48 字节额外开销
```

```bash
# 方式二：Hash 集中存储（一个 key 包含所有属性）
HSET user:1001 id "1001" name "张三" age "28" city "北京"
# 1 个 key，key 名 "user:1001" 占 9 字节
# 总开销比方式一少约 39 字节/用户
# 如果有 1000 万用户，节省约 390MB
```

但 Hash 也有自己的问题：如果元素太多（几千个字段），Redis 会把 Hash 从 `ziplist` 编码切换到 `hashtable` 编码，内存开销反而更高。

**Redis 5.0 新增的 `LS` 编码**：对于列表和哈希，Redis 引入了 `listpack`（替代 ziplist）和 `LS` 编码，进一步压缩内存。`LS` 是 list 和 stream 的混合编码，对小数据量场景内存效率更高。

### 3.2 选择合适的数据编码

Redis 会根据数据大小自动选择最优的底层编码。理解这个机制，可以帮你在编码层做优化。

| 数据类型 | 编码类型 | 触发条件 | 内存效率 |
|---|---|---|---|
| String | `int` | 存储纯整数 | 最高 |
| String | `embstr` | 字符串 `&lt;` 44 字节 | 较高 |
| String | `raw` | 字符串 `>=` 44 字节 | 普通 |
| Hash | `ziplist` | 字段数 `&lt;` 512，字节 `&lt;` 64B/字段 | 最高 |
| Hash | `hashtable` | 字段数 `>=` 512 | 普通 |
| List | `ziplist` | 元素数 `&lt;` 128，字节 `&lt;` 64B/元素 | 高 |
| List | `linkedlist` | 元素数 `>=` 128 | 普通 |
| Set | `intset` | 全为整数，元素数 `&lt;` 512 | 高 |
| Set | `hashtable` | 包含非整数或元素 `>=` 512 | 普通 |
| ZSet | `ziplist` | 元素数 `&lt;` 128 | 高 |
| ZSet | `skiplist` | 元素数 `>=` 128 | 普通 |

:::tip embstr vs raw 的 44 字节阈值

`embstr` 和 `raw` 的分界线是 **44 字节**（Redis 3.0-4.0 是 39 字节，5.0 后升级到 44 字节）。这是因为 RedisObject 本身占 16 字节，jemalloc 最小分配单位是 64 字节，64 - 16 = 48 字节，减去 4 字节的字符串长度字段，剩余 44 字节可用。

理解这个数字，有助于设计 key-value 大小，避开 `raw` 编码的更高开销。

:::

### 3.3 开启内存淘汰策略

`maxmemory` + 淘汰策略是 Redis 内存控制的核心机制。设置得当，可以避免 OOM，同时最大化利用内存。

```bash
# 设置最大内存为 8GB
CONFIG SET maxmemory 8gb

# 设置淘汰策略为 allkeys-lru
CONFIG SET maxmemory-policy allkeys-lru
```

常用的淘汰策略：

| 策略 | 行为 | 适用场景 |
|---|---|---|
| `noeviction` | 内存满时拒绝写入 | 不推荐，容易 OOM |
| `volatile-lru` | 仅从设置过期时间的 key 中淘汰 LRU | 有明确热冷区分 |
| `allkeys-lru` | 从所有 key 中淘汰 LRU | 无明显热冷区分（**推荐**） |
| `volatile-ttl` | 从设置过期时间的 key 中淘汰 TTL 最短的 | 临时缓存 |
| `volatile-random` | 从设置过期时间的 key 中随机淘汰 | 不推荐 |
| `allkeys-random` | 从所有 key 中随机淘汰 | 不推荐 |
| `volatile-lfu` | 从设置过期时间的 key 中淘汰 LFU | 需要 LFU 支持（Redis 4.0+） |
| `allkeys-lfu` | 从所有 key 中淘汰 LFU | 热点明确（Redis 4.0+） |

:::warning LRU vs LFU 的选择

LRU（Least Recently Used）淘汰最久未使用的，LFU（Least Frequently Used）淘汰使用频率最低的。

LFU 更适合**访问模式稳定的场景**（如排行榜、固定热点数据）。LRU 更适合**访问模式动态变化的场景**。

但 LFU 有一个问题：一旦某个 key 的访问频率高起来，就很难降下来（即使之后不再访问）。在某些业务场景下，LFU 可能反而不如 LRU 灵活。

:::

### 3.4 数据压缩：减少 String 的体积

String 类型是最常用的 Redis 数据结构，但也是内存效率最低的。String 的 value 直接存储，没有额外的结构优化。

如果 String 中存储的是 JSON、XML 或其他文本数据，可以先压缩再存入 Redis：

```java
// 使用 GZIP 压缩后存入 Redis
public void cacheWithCompression(String key, Object value) {
    try {
        ByteArrayOutputStream bos = new ByteArrayOutputStream();
        GZIPOutputStream gzip = new GZIPOutputStream(bos);
        gzip.write(objectMapper.writeValueAsBytes(value));
        gzip.close();
        byte[] compressed = bos.toByteArray();
        
        // 压缩后存入 Redis
        redisTemplate.opsForValue().set(key, compressed);
        
        // 设置压缩标记，便于读取时解压
        redisTemplate.opsForValue().set(key + ":compressed", "true");
    } catch (Exception e) {
        // 压缩失败，降级为普通存储
        redisTemplate.opsForValue().set(key, value);
    }
}
```

压缩的代价是 CPU 开销。对于**读取密集型、value 较大（KB 级以上）**的场景，压缩的内存收益通常大于 CPU 代价。

### 3.5 短字符串的 SDS 优化

Redis 的字符串不是 C 语言的 char 数组，而是自己实现的数据结构 SDS（Simple Dynamic String）。SDS 有几个优化点值得关注：

```c
// SDS 的结构
struct __attribute__ ((__packed__)) sdshdr8 {
    uint8_t len;      // 已使用长度
    uint8_t alloc;    // 总分配长度（不含 header）
    unsigned char flags; // 标志位
    char buf[];        // 实际字符串
};
```

**预分配策略**：当字符串长度小于 1MB 时，Redis 会按 `2 * len + 1` 预分配空间。当字符串长度大于 1MB 时，Redis 按 `len + 1MB + 1` 预分配。这意味着短字符串不会过度预分配，但长字符串会按 1MB 一段一段地扩展。

**惰性释放**：SDS 的字符串缩短操作不会立即释放多余的空间，而是记录新的长度，等后续需要时再释放。这个设计避免了频繁的内存分配，但也会导致碎片。

理解 SDS 的预分配策略，有助于你在业务代码中设计合适的数据大小，避免触发不必要的预分配。

### 3.6 禁止存储大 key

这是最简单也最有效的优化：**不要在 Redis 中存储大 value**。

大 key 的危害：

- **内存倾斜**：一个大 key 可能占据 Redis 内存的 50% 以上
- **操作阻塞**：`DEL`、`HGETALL`、`SMEMBERS` 等命令会遍历整个 key，操作时间与 value 大小成正比
- **集群问题**：在 Redis Cluster 中，大 key 会导致槽分布不均

```bash
# 使用 SCAN 查找大 key（Redis 4.0+）
redis-cli --bigkeys

# 使用 MEMORY USAGE 查看单个 key 的内存占用
redis-cli MEMORY USAGE user:profile:12345
# 上面这个命令会返回 key 加上 value 的总字节数
```

最佳实践是**提前拆分大 key**：把一个大 Hash 按用户 ID 拆成多个小 Hash，把一个大 List 按固定长度拆成多个小 List。

## 四、Redis 6+ 的内存优化新特性

Redis 6 引入了多项内存优化新特性，值得关注。

### 4.1 listpack 替代 ziplist

Redis 5.0 开始，ziplist 被 `listpack` 逐步替代（Redis 7.0 完全移除 ziplist）。`listpack` 是 ziplist 的改进版，消除了 ziplist 的一个关键缺陷：**ziplist 中间元素变更时需要重新排列整个列表尾部**。

`listpack` 的设计使得任意位置的插入和删除都只影响局部，不需要移动大量元素。这对频繁修改的 Hash 和 List 场景内存效率更高。

### 4.2 Hash 的新编码优化

Redis 7.0 引入了 `LISTPACK` 编码的 Hash。当 Hash 的字段数较少时，Redis 会使用 `listpack` 替代原来的 ziplist，进一步减少内存开销。

```
Redis 7.0 Hash 编码决策：
字段数少（默认 &lt; 512）且每字段字节少（默认 &lt; 128） → listpack（最省内存）
字段数多或每字段字节多 → hashtable（O(1) 查找）
```

### 4.3 Stream 的内存优化

Stream 是 Redis 5.0 引入的数据结构，用于实现消息队列。Redis 6 对 Stream 做了内存优化，特别是在 consumer group 场景下，消息确认（ACK）的内部存储从 Radix Tree 改为更紧凑的 `runt` 结构，减少了内存占用。

## 五、监控与排查实战

### 5.1 内存监控的关键指标

```bash
# 完整的内存信息面板
INFO memory
```

| 指标 | 含义 | 关注点 |
|---|---|---|
| `used_memory` | 应用申请的内存总量 | 核心监控指标 |
| `used_memory_rss` | 实际占用的物理内存 | 与 `used_memory` 对比，差值即碎片 |
| `mem_fragmentation_ratio` | 内存碎片率 | `>` 1.5 需要关注 |
| `mem_fragmentation_ratio` | `maxmemory` 设置值 | 应小于机器物理内存 |
| `maxmemory_policy` | 当前淘汰策略 | 确认是否符合业务预期 |
| `used_memory_peak` | 历史峰值内存 | 评估 `maxmemory` 是否够用 |
| `mem_aof_buffer` | AOF 缓冲区大小 | 写入量大时可能较大 |
| `mem_replication_backlog` | 复制积压缓冲区 | 主从环境需要关注 |

### 5.2 内存异常排查流程

```bash
# 第一步：确认内存使用情况
INFO memory | grep -E "used_memory|used_memory_rss|fragmentation"

# 第四步：分析具体 key 的内存占用

INFO clients | grep -E "biggest_input|output_list"
```

## 六、面试实战

### 场景一：Redis 内存满了怎么办

**面试官问**：「Redis 内存使用率达到 90%，如果内存满了会怎样？」

**深入回答**：

> Redis 内存满了取决于 `maxmemory-policy` 的设置。如果设置了淘汰策略，Redis 会按照策略淘汰一些 key，继续提供服务。如果没有设置淘汰策略（默认 `noeviction`），Redis 会**拒绝所有写入操作**，返回 `OOM` 错误。
>
> 解决 Redis 内存满的问题，有几种思路：
>
> **第一，设置合理的 `maxmemory` + 淘汰策略**。根据业务特点选择 LRU/LFU/TTL 策略，确保热点数据留在内存中。
>
> **第二，分析并优化 key 结构**。用 `MEMORY USAGE` 找到占用内存最大的 key，看看是否有过期 key 未清理、大 key、或者可以压缩的数据。
>
> **第三，处理内存碎片**。如果 `mem_fragmentation_ratio` 过高，用 `MEMORY PURGE` 或开启 `activedefrag` 整理碎片。
>
> **第四，扩容**。如果以上都不能解决，考虑水平扩展（Redis Cluster）或垂直扩容（更大内存的机器）。

### 场景二：如何减少 Redis 内存碎片

**面试官问**：「Redis 的内存碎片率很高（2.0 以上），你有什么办法？」

**深入回答**：

> 内存碎片的根本原因是 jemalloc 分配器和业务数据的特性导致的。我的解决方案分三步：
>
> **第一步：确认碎片来源**。检查 `INFO memory` 中的 `mem_fragmentation_ratio`。如果 `>` 1.5，说明碎片偏多。
>
> **第二步：手动触发整理**。用 `MEMORY PURGE` 命令手动触发 jemalloc 整理碎片。这个操作会短暂阻塞 Redis，但通常很快。
>
> **第三步：开启自动整理**。对于长期运行的服务，设置 `CONFIG SET activedefrag yes`，让 Redis 在后台自动整理碎片。
>
> **第四步：治本——优化业务代码**。碎片产生的根源是 key 频繁修改大小和删除。如果业务代码中大量使用了 `SET` 覆盖 key，可以考虑换成 `APPEND` 或分段更新，避免触发大范围的内存重分配。
>
> **最后，如果以上都不行，重启 Redis**。重启是最彻底的碎片清理手段，但需要配合 RDB/AOF 备份来恢复数据。

### 场景三：Redis 的内存消耗由哪些部分构成

**面试官问**：「Redis 的内存消耗由哪些部分构成？COW 是什么？」

**深入回答**：

> Redis 的内存消耗主要分五部分：
>
> **第一，数据内存**。存储业务数据的空间，这是最大的部分。
>
> **第二，额外管理内存**。过期 key 的 expires dict、Lua 脚本占用等。
>
> **第三，缓冲区内存**。客户端输入/输出缓冲区、AOF 写缓冲区、复制缓冲区。
>
> **第四，进程内存**。Redis 主进程本身的开销。
>
> **第五，内存碎片**。jemalloc 分配器产生的碎片空间。
>
> **关于 COW**：当 Redis 执行 `BGSAVE`（后台 RDB 持久化）或 `BGREWRITEAOF`（后台 AOF 重写）时，Redis 会 fork 一个子进程。Linux 的 fork 使用 Copy-On-Write 机制——父子进程共享内存页面，只有在某个进程修改页面时，才会复制对应的页面。在 fork 的瞬间，Redis 的内存消耗会翻倍（如果 10GB，`COW` 开销最多再占 10GB）。这就是为什么生产环境 Redis 的 `maxmemory` 不能设置得太接近机器物理内存，**必须预留 COW 的空间**。

## 七、面试追问预判

### 追问一：jemalloc 和 glibc 的 malloc 有什么区别

Redis 默认使用 jemalloc 而不是 glibc 的 malloc。原因是 jemalloc 在**多线程环境下的锁竞争更少**，内存分配速度更快，而且**内存碎片率更低**。如果用 glibc 的 malloc，高并发下内存分配会成为 Redis 的性能瓶颈。

### 追问二：COW 会导致 Redis OOM 吗

会。如果 Redis 内存占用已经接近机器物理内存，fork 时触发的 COW 会瞬间占用大量物理内存，可能触发系统的 OOM Killer。

**最佳实践**：`maxmemory` 设置为机器物理内存的 **50%-70%**，为 COW 和系统缓存预留足够的空间。

### 追问三：如何发现 Redis 中的大 key

三种方法：

```bash
# 方法一：redis-cli --bigkeys（扫描全库，耗时）
redis-cli --bigkeys

# 方法二：MEMORY USAGE 查看指定 key（精准）
redis-cli MEMORY USAGE big:hash:key

# 方法三：Redis 7.0+ MEMORY STATS（全局分析）
redis-cli MEMORY STATS
```

### 题目一：碎片率的边界判断

假设你的 Redis 集群中，`used_memory` 是 50GB，`used_memory_rss` 是 60GB，`mem_fragmentation_ratio` 是 1.2。如果业务继续增长，`used_memory` 预计半年后翻倍到 100GB。

**问题**：不考虑其他因素，半年后 `used_memory_rss` 可能会到多少？你的 `maxmemory` 设置需要调整吗？

### 题目二：COW 与 fork 的取舍

Redis 的 RDB 持久化需要 `BGSAVE` fork 子进程。如果 Redis 内存是 64GB，机器物理内存是 128GB。

**问题**：此时触发 `BGSAVE`，理论上 COW 最多会占用多少额外内存？如果同时还有从库在做全量同步（`SYNC`），情况会怎样？这种情况下应该如何规划 Redis 的内存使用？

## 总结

Redis 内存优化的核心，是理解内存的**来源**和**去向**，然后对症下药。

**内存消耗的五大来源**：

- 数据内存（业务存储）
- 额外管理内存（过期、复制、脚本）
- 缓冲区内存（客户端、AOF、复制）
- 进程内存（主进程 + COW）
- 内存碎片（jemalloc 分配器产生）

**内存优化的三大手段**：

- **结构优化**：选择合适的数据类型和编码，避免大 key
- **碎片治理**：`MEMORY PURGE` + `activedefrag`
- **容量规划**：合理的 `maxmemory` + 淘汰策略，预留 COW 空间

很多工程师只盯着「存了多少数据」，忽略了看不见的碎片和缓冲区。这篇文章的核心目的，就是让你在排查 Redis 内存问题时，能有一个完整的框架——知道问题在哪，才能对症下药。
