# Redis Pipeline：批量操作背后的性能真相

你有没有遇到过这样的场景：一个接口需要查询 Redis 中的 100 个 key，按常规写法，循环 100 次调用 `GET`，接口响应时间动不动就几十毫秒甚至上百毫秒。你百思不得其解——Redis 本身那么快，为什么加上这个「简单」的查询就慢成这样？

答案很简单：**不是 Redis 慢，是网络往返次数太多了。**

100 次 `GET`，意味着 100 次网络往返（RTT，Round Trip Time）。即使 Redis 本身处理只需要 0.1ms，单是网络往返就可能贡献了几十毫秒。如果应用和 Redis 之间还有额外的网络延迟（如跨机房部署），这个时间还会更长。

这个问题，就是 |Pipeline| 要解决的核心问题。

## 一、普通批量操作 vs Pipeline

### 1.1 普通批量命令是什么

Redis 提供了一批原生批量命令，典型代表是 `mget`、`mset`、`hmget`、`hmset`。这些命令的特点是：**一条命令支持操作多个 key**，服务端原子执行，一次 RTT 返回所有结果。

```redis
MSET key1 value1 key2 value2 key3 value3
MGET key1 key2 key3
```

以 `MGET` 为例，客户端发送一条命令，服务端依次取出 key1、key2、key3 的值，打包在一个响应里返回。**这是原子操作，要么全部成功，要么全部失败**（实际上 Redis 单条命令本身就是原子的）。

适用场景：批量读写**同一种数据类型**的多个 key，且这些 key 可以用同一条命令覆盖。比如批量读取 100 个用户的用户名（`mget user:name:1 user:name:2 ...`）。

### 1.2 Pipeline 是什么

|Pipeline| 是一种**客户端侧的批量优化**机制。它的核心思想是：**把多个 Redis 命令打包成一个请求发送，服务端依次执行后，再打包返回结果**。

```
普通模式（N 次 RTT）：
客户端 → GET key1 → Redis → 返回  →  客户端 → GET key2 → Redis → 返回  → ...
          耗时 RTT                      耗时 RTT

Pipeline 模式（1 次 RTT）：
客户端 → GET key1 + GET key2 + GET key3 + ... → Redis → 批量返回
          耗时 RTT（一次性发送所有命令）
```

注意 Pipeline 和原生批量命令的区别：

| 维度 | 原生批量命令 | Pipeline |
|---|---|---|
| 原子性 | 原子（单命令多 key） | 非原子（命令打包，无事务保证） |
| 命令类型 | 必须是同类型命令（如 mget） | 任意 Redis 命令混合 |
| 网络开销 | 1 次 RTT | 1 次 RTT |
| 服务端处理 | 单命令执行 | 依次执行每个命令 |

**核心区别**：原生批量命令是服务端原生支持的原子多 key 操作；Pipeline 是客户端侧的请求合并，不改变服务端行为，也不保证原子性。

### 1.3 为什么 Pipeline 能提升性能

性能提升的关键是**减少了网络往返次数**。

假设应用和 Redis 之间的 RTT 为 1ms，客户端需要执行 100 次 `GET`：

- **普通模式**：100 次 RTT = **100ms**
- **Pipeline 模式**：1 次 RTT = **1ms**

理论上性能提升 100 倍。实际场景中，这个收益可能没那么夸张（因为服务端处理也有耗时），但 Pipeline 对批量操作场景的性能提升是显著的——特别是跨机房、高延迟网络环境下。

:::warning Pipeline 不是万能的

Pipeline 的收益来自减少 RTT 次数。但有以下限制：

- Pipeline 内所有命令之间**不能有依赖关系**（如 `GET key` 的结果决定下一个命令的参数）
- Pipeline **不保证原子性**：如果第 10 条命令失败，前 9 条不会回滚
- Pipeline 内**不能使用事务**（`MULTI`/`EXEC`），如果需要事务的原子性，要用 Lua 脚本

:::

## 二、Pipeline 的工作原理

### 2.1 命令入队与批量发送

Pipeline 的工作流程分为四步：

```
第一步：命令入队
客户端将命令依次加入本地命令队列（不发送）

第二步：一次性发送
客户端 flush 命令队列，将所有命令打包成一个 TCP 请求发送

第三步：服务端依次执行
Redis 服务端依次接收并执行每个命令

第四步：批量返回
服务端将所有命令的执行结果打包成一个响应返回
```

这个过程对应用代码是透明的——你只需要调用 API「这是一组 Pipeline 命令」，剩下的合并发送由客户端 SDK 完成。

### 2.2 Pipeline 的内部实现

Pipeline 的本质是**客户端侧的请求合并**。Redis 服务端并不感知客户端是否使用了 Pipeline——它收到的是一个 TCP 流，每个命令依次解析执行，结果依次返回。

```
客户端发送的字节流：
*3\r\n$3\r\nGET\r\n$5\r\nkey1\r\n*3\r\n$3\r\nGET\r\n$5\r\nkey2\r\n...
（多个 Redis 协议命令拼接在一起）

Redis 服务端解析：
逐个解析命令 → 依次执行 → 依次写入响应缓冲区
```

理解这一点很重要：Pipeline 不改变 Redis 服务端的行为。它改变的是客户端的发送方式——从「发一条、等一条」，变成「发一批、等一批」。

### 2.3 Pipeline 的性能边界

Pipeline 虽好，但也不是越大越好。有两个约束需要考虑：

**内存约束**：Pipeline 命令队列保存在客户端内存中。如果一次 Pipeline 包含几万条命令，客户端内存占用会显著增加。

**服务端阻塞**：Pipeline 越大，Redis 服务端处理时间越长。Redis 是单线程执行命令的，一个大 Pipeline 会阻塞其他客户端的命令执行（虽然只是毫秒级，但并发量高时会放大影响）。

**最佳实践**：单次 Pipeline 包含 **500-1000 条命令**是一个经验值，既能获得明显的性能收益，又不会对客户端和服务端造成压力。

## 三、Pipeline 的正确使用

### 3.1 Jedis 的 Pipeline 用法

```java
// Jedis Pipeline 使用示例
Jedis jedis = jedisPool.getResource();
Pipeline pipeline = jedis.pipelined();

// 批量发送命令（不执行）
for (int i = 0; i < 100; i++) {
    pipeline.get("user:name:" + i);
    pipeline.hget("user:profile:" + i, "age");
}

// 执行并获取结果
List<Object> results = pipeline.sync();  // 同步执行
// 或
List<Object> results = pipeline.syncAndReturnAll(); // 同步执行并返回结果
```

:::warning 重要：不要在 Pipeline 内使用事务

Pipeline 和事务（`MULTI`/`EXEC`）是两套不同的机制。在 Jedis 中，事务需要单独开启 `multi()`，Pipeline 需要单独开启 `pipelined()`。**两者不能混用**。

```java
// 错误写法：在 Pipeline 中使用事务
Pipeline pipeline = jedis.pipelined();
pipeline.multi();        // 不会生效，Pipeline 会忽略 MULTI
pipeline.get("key1");
pipeline.exec();         // 也不会生效
```

**如果需要原子性 + 批量操作，使用 Lua 脚本。**

:::

### 3.2 Lettuce 的 Pipeline 用法

Lettuce 是 Spring Boot 2.x 默认的 Redis 客户端。相比 Jedis，Lettuce 是**响应式编程友好**的，同时支持同步、异步和 Reactive 模式。

```java
// Lettuce 同步模式 Pipeline
RedisCommands<String, String> commands = connection.sync();
commands.get("key1");
commands.get("key2");
// 实际上 sync() 模式下每条命令都是独立发送的
// 要实现 Pipeline，需要用 execute() 包裹
```

Lettuce 的 Pipeline 更推荐使用 **async 模式**，或者通过 `execute()` 方法手动控制：

```java
// Lettuce 真正的 Pipeline：使用 async + batch
RedisAdvancedClusterAsyncCommands<String, String> async = 
    clusterClient.connectAdvanced().getAdvancedAsync();
AsyncCommand<String, String, String> cmd1 = async.get("key1");
AsyncCommand<String, String, String> cmd2 = async.get("key2");

// 命令在底层被批量发送（由 Lettuce 自动 batching）
CompletableFuture<List<Object>> future = asyncFleet.getValues(
    Arrays.asList("key1", "key2", "key3")
);
```

### 3.3 Spring Data Redis 的 Pipeline

在 Spring Boot 项目中，使用 `StringRedisTemplate` 的 `executePipelined()` 方法是最简洁的方式：

```java
// Spring Data Redis Pipeline 示例
List<Object> results = stringRedisTemplate.executePipelined(
    new RedisCallback<Object>() {
        @Override
        public Object doInRedis(RedisConnection connection) 
                throws DataAccessException {
            for (int i = 0; i < 100; i++) {
                connection.stringCommands().get(
                    ("user:name:" + i).getBytes()
                );
                connection.hashCommands().hGet(
                    ("user:profile:" + i).getBytes(),
                    "age".getBytes()
                );
            }
            // 返回 null 表示没有额外命令
            return null;
        }
    }
);
```

Spring Boot 2.0+ 还支持 `execute` 方法配合 `SessionCallback` 使用 Redis 事务（`MULTI`/`EXEC`），这和 Pipeline 是不同的 API：

```java
// Spring Data Redis 事务：使用 SessionCallback
List<Object> txResults = stringRedisTemplate.execute(
    new SessionCallback<List<Object>>() {
        @Override
        public List<Object> execute(RedisOperations operations) 
                throws DataAccessException {
            operations.multi();              // 开启事务
            operations.opsForValue().get("key1");
            operations.opsForValue().get("key2");
            return operations.exec();         // 执行并返回
        }
    }
);
```

### 3.4 分批策略

当需要批量操作数千甚至数万条命令时，不要把所有命令塞进一个 Pipeline。正确的做法是**分批**：

```java
public void batchGetUsers(List<Long> userIds) {
    // 每 500 条命令分一个 Pipeline
    int batchSize = 500;
    List<List<Long>> partitions = Lists.partition(userIds, batchSize);
    
    List<Object> allResults = new ArrayList<>();
    for (List<Long> batch : partitions) {
        List<Object> batchResults = stringRedisTemplate.executePipelined(
            (RedisCallback<Object>) connection -> {
                for (Long userId : batch) {
                    connection.stringCommands().get(
                        ("user:name:" + userId).getBytes()
                    );
                }
                return null;
            }
        );
        allResults.addAll(batchResults);
    }
    
    return allResults;
}
```

## 四、Lua 脚本 vs Pipeline vs 事务

这是面试中的高频对比题。很多同学分不清这三个概念的区别。

### 4.1 三者的核心区别

| 维度 | Pipeline | 事务（MULTI/EXEC） | Lua 脚本 |
|---|---|---|---|
| 原子性 | 非原子 | 原子（服务端批量执行） | 原子（服务端单线程执行） |
| 网络开销 | 1 次 RTT | N+1 次 RTT（`MULTI` + N 条 + `EXEC`） | 1 次 RTT |
| 服务端并行 | 无（依次执行） | 无（依次执行） | 无（单线程串行） |
| 回滚能力 | 不支持 | 支持 `DISCARD` 放弃事务 | 不支持（但可以判断后返回错误） |
| 条件执行 | 不支持 | 不支持 | 支持（`if/else` 逻辑） |
| 适用场景 | 批量读写，无依赖 | 需要原子性的批量操作 | 原子 + 条件逻辑 + 复杂操作 |

### 4.2 为什么事务的 RTT 是 N+1

Redis 的事务需要手动开启（`MULTI`），之后每条命令入队但不执行，最后 `EXEC` 才触发批量执行：

```redis
MULTI
GET key1
GET key2
GET key3
EXEC
```

这意味着：客户端至少需要 2 次 RTT（`MULTI` + `EXEC`），如果算上命令本身，**实际是 N+1 次 RTT**（`MULTI` 一次，每条命令发送一次，`EXEC` 一次）。在网络延迟高的情况下，性能很差。

### 4.3 Lua 脚本：原子 + 服务端计算

|Lua 脚本| 是 Redis 2.6+ 引入的功能。它的核心优势是：**原子性 + 服务端执行 + 支持条件逻辑 + 1 次 RTT**。

```lua
-- Lua 脚本示例：先判断库存再扣减
local stock = tonumber(redis.call('GET', KEYS[1]) or 0)
local quantity = tonumber(ARGV[1])
if stock >= quantity then
    redis.call('DECRBY', KEYS[1], quantity)
    return stock - quantity   -- 返回扣减后的库存
else
    return -1                -- 库存不足
end
```

Java 中调用 Lua 脚本：

```java
String luaScript = 
    "local stock = tonumber(redis.call('GET', KEYS[1]) or 0) " +
    "local quantity = tonumber(ARGV[1]) " +
    "if stock >= quantity then " +
    "    redis.call('DECRBY', KEYS[1], quantity) " +
    "    return stock - quantity " +
    "else " +
    "    return -1 " +
    "end";

DefaultRedisScript<Long> script = new DefaultRedisScript<>();
script.setScriptText(luaScript);
script.setResultType(Long.class);
Long result = redisTemplate.execute(script, 
    Arrays.asList("stock:1"), "10");
```

### 4.4 选择指南

```
需要批量读写，对原子性无要求
    → Pipeline（性能最优）

需要批量操作，且要保证原子性（要么全成功，要么全失败）
    → Lua 脚本（1 次 RTT + 原子）
    → 事务（MULTI/EXEC）仅作为备选（网络开销大）

需要复杂的条件判断和逻辑分支
    → Lua 脚本（唯一选择）
```

## 五、Pipeline 在生产中的实战场景

### 5.1 场景一：批量查询用户信息

业务场景：展示用户列表，需要批量查询每个用户的昵称、头像、积分。

```java
// 原始写法：100 次网络往返
for (Long userId : userIds) {
    String nickname = redisTemplate.opsForValue().get("user:nickname:" + userId);
    String avatar = redisTemplate.opsForValue().get("user:avatar:" + userId);
    Integer score = (Integer) redisTemplate.opsForHash().get("user:score", userId.toString());
    // ... 组装用户对象
}
// 100 个用户 = 300 次 GET/HGET = 300 次 RTT
```

```java
// Pipeline 优化：1 次 RTT
List<Object> results = redisTemplate.executePipelined(
    (RedisCallback<Object>) connection -> {
        for (Long userId : userIds) {
            connection.stringCommands().get(("user:nickname:" + userId).getBytes());
            connection.stringCommands().get(("user:avatar:" + userId).getBytes());
            connection.hashCommands().hGet(("user:score").getBytes(), userId.toString().getBytes());
        }
        return null;
    }
);

// 结果解析：每 3 个结果为一组用户数据
for (int i = 0; i < results.size(); i += 3) {
    String nickname = (String) results.get(i);
    String avatar = (String) results.get(i + 1);
    byte[] scoreBytes = (byte[]) results.get(i + 2);
    Integer score = scoreBytes == null ? 0 : Integer.parseInt(new String(scoreBytes));
    // ... 组装用户对象
}
```

**性能对比**：假设 RTT 为 0.5ms，100 个用户数据：
- 原始写法：300 × 0.5ms = **150ms**
- Pipeline：1 × 0.5ms = **0.5ms**

### 5.2 场景二：批量更新排行榜分数

游戏排行榜场景：每局游戏结束后，批量更新多个玩家的积分。

```java
// 批量更新多个玩家的积分
List<Object> updateResults = redisTemplate.executePipelined(
    (RedisCallback<Object>) connection -> {
        for (GameResult result : gameResults) {
            connection.zSetCommands().zIncrBy(
                "leaderboard:global".getBytes(),
                result.getDelta(),
                result.getUserId().toString().getBytes()
            );
        }
        return null;
    }
);
```

### 5.3 场景三：Pipeline + 分布式锁的坑

Pipeline 不能用于实现分布式锁，因为分布式锁需要**每一步都检查结果**（获取锁失败就立即返回，不能继续执行后续命令）。Pipeline 的命令是批量发送、批量返回的，无法做到这一点。

```java
// 错误：用 Pipeline 实现分布式锁
Pipeline pipeline = jedis.pipelined();
pipeline.set("lock:order:" + orderId, Thread.currentThread().getId(), 
    SetParams.setParams().nx().ex(30));
pipeline.expire("lock:order:" + orderId, 30);
List<Object> results = pipeline.sync();
// 问题：不知道哪一步失败了，无法做正确的逻辑处理
```

分布式锁必须用单独的命令或 Lua 脚本：

```java
// 正确：Redis SET + NX + EX 原子命令
String result = jedis.set("lock:order:" + orderId, 
    threadId, SetParams.setParams().nx().ex(30));
if ("OK".equals(result)) {
    // 获取锁成功
} else {
    // 获取锁失败
}
```

## 六、面试实战

### 场景一：Pipeline 和普通批量操作的区别

**面试官问**：「Pipeline 和 Redis 的 `MGET`/`MSET` 批量命令有什么区别？各自适用什么场景？」

**普通回答**：Pipeline 快，MGET 慢。

**深入回答**：

> `MGET`/`MSET` 是 Redis **原生提供的原子批量命令**，一条命令可以操作多个 key，服务端保证原子性。而 Pipeline 是**客户端侧的请求合并**，把多个独立命令打包发送，不保证原子性。
>
> 区别在于三点：
>
> **第一，原子性**。`MGET` 是原子的，一条命令要么全部执行，要么全部不执行。Pipeline 里的每条命令是独立执行的，一条失败不会影响其他命令。
>
> **第二，命令类型**。`MGET`/`MSET` 只能操作同一种数据类型（string 类型的多 key）。Pipeline 可以混合任意类型的命令。
>
> **第三，适用场景**。如果业务需要批量读写同类 key，`MGET`/`MSET` 是最优选择。如果需要批量执行不同类型的命令，或者批量操作一个业务场景下的多个 key，Pipeline 更合适。

### 场景二：Pipeline 的原理

**面试官问**：「Pipeline 为什么能提升性能？它的原理是什么？」

**深入回答**：

> Pipeline 提升性能的核心是**减少网络往返次数（RTT）**。
>
> 在一次 Redis 操作中，网络往返时间往往远大于 Redis 服务端的命令执行时间。比如一次 `GET` 命令，服务端处理只需要 0.05ms，但如果网络 RTT 是 0.5ms，那一次完整的 `GET` 交互就是 0.55ms。
>
> Pipeline 把 N 条命令打包成 1 次 TCP 请求发送，Redis 服务端依次执行后打包返回。整个过程只需要 1 次 RTT。
>
> **但是要注意**：Pipeline 不改变 Redis 服务端的行为。Redis 仍然是单线程依次执行每条命令的。Pipeline 的收益完全来自网络优化，而不是服务端并行。

### 场景三：Pipeline 和 Lua 脚本的区别

**面试官问**：「Pipeline 和 Lua 脚本都能减少 RTT，它们有什么区别？什么时候用哪个？」

**深入回答**：

> 两者有三个本质区别：
>
> **第一，原子性**。Lua 脚本在 Redis 服务端是原子执行的——Redis 在执行 Lua 脚本期间不会执行其他命令。Pipeline 不保证原子性，一条命令失败不会回滚其他命令。
>
> **第二，条件逻辑**。Lua 脚本支持 `if/else`、循环等条件判断，可以根据上一条命令的结果决定下一条命令执行什么。Pipeline 的命令是固定的，无法根据执行结果动态调整。
>
> **第三，回滚能力**。Lua 脚本不支持真正的回滚，但可以在脚本内部做条件判断，发现不满足条件时返回错误码，客户端收到后再决定如何处理。Pipeline 完全不支持。
>
> **选择建议**：
> - 纯批量读写，无条件、无依赖 → Pipeline
> - 需要原子性 + 条件判断 → Lua 脚本
> - 需要强一致性回滚 → 使用 WATCH + MULTI + exec（注意 RTT 代价）

## 七、面试追问预判

### 追问一：Pipeline 的命令结果顺序是什么

Pipeline 返回的结果顺序和命令发送顺序完全一致。这是因为 Redis 服务端按接收顺序依次处理，依次返回。

```java
// 发送顺序：GET key1, GET key2, HGET key3 field1
// 返回顺序：result1, result2, result3（与发送顺序一致）
```

### 追问二：Pipeline 会不会丢命令

不会。Pipeline 只是客户端侧的请求合并，底层还是可靠的 TCP 连接。命令打包成字节流后一次性发送，服务端按顺序解析执行。

### 追问三：Pipeline 适合哪些场景，不适合哪些场景

**适合**：批量读取/写入多个 key，且各命令之间无依赖。

**不适合**：

- 命令之间有依赖（如 `GET key1` 的结果决定 `GET key2` 的参数）
- 需要原子性的多步操作（用 Lua 脚本）
- 单次 Pipeline 命令数过多（分批处理）

## 八、留给你的思考题

### 题目一：Pipeline 与长连接的取舍

Pipeline 能减少 RTT，但需要客户端维护一个长连接。如果你的应用和 Redis 之间网络延迟极低（如同机房部署，RTT `&lt;` 0.1ms），Pipeline 的收益会大幅下降。

**问题**：在极低延迟场景下，Pipeline 的收益是否还值得？你能计算出 Pipeline 收益「盈亏平衡」的 RTT 阈值吗？

### 题目二：Pipeline + 批量大小设计

假设每次接口请求需要操作 10000 个 key，单条命令平均处理时间 0.05ms。

**问题**：如果用 Pipeline 一次发送 10000 条，Redis 服务端处理这 10000 条命令需要多久？这个处理时间会不会对其他请求造成影响？你会如何设计分批策略？

---

## 总结

|Pipeline| 的本质是**用一次 RTT 换取批量命令执行**，核心价值是减少网络往返次数对整体耗时的贡献。

关键要点回顾：

- **Pipeline 是客户端优化**，不是服务端特性，不改变 Redis 单线程串行执行的行为
- **Pipeline 不保证原子性**，如果需要原子性 + 条件逻辑，用 Lua 脚本
- **Pipeline 不支持事务**（`MULTI`/`EXEC`），两者是独立的机制
- **最佳实践**：单次 Pipeline 控制在 500-1000 条命令，避免对客户端内存和服务端造成压力

理解 Pipeline 的原理和边界，你就能在面试中清晰地说出「它解决了什么问题，同时带来了什么代价」——这才是面试官真正想听到的答案。
