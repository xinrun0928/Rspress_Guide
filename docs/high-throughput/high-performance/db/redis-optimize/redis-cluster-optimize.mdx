# Redis Cluster 槽迁移与热点扩容

Redis Cluster 是 Redis 官方提供的分布式集群方案，用来实现**数据分片**和**高可用**。

但很多团队用上 Redis Cluster 之后，问题反而更多了：

- 槽迁移时业务抖动怎么办？
- 热点 key 全部打在一个节点上，集群反而成了瓶颈？
- MOVED 重定向和 ASK 重定向有什么区别？
- 集群如何做高可用，节点故障了怎么处理？

这篇文章，带你彻底搞懂 Redis Cluster 的槽迁移机制、热点扩容策略，以及集群高可用的实现原理。

## 一、Redis Cluster 架构回顾

### 1.1 集群核心概念

Redis Cluster 通过 **16384 个槽（slot）** 实现数据分片。每个 key 通过 CRC16 哈希算法映射到某个槽：

```
slot = CRC16(key) % 16384
```

集群中的每个主节点负责一部分槽。例如 3 主 3 从的集群：

| 节点 | 负责的槽 | 从节点 |
|---|---|---|
| 192.168.1.1:6379 | 0-5460 | 192.168.1.4:6379 |
| 192.168.1.2:6379 | 5461-10922 | 192.168.1.5:6379 |
| 192.168.1.3:6379 | 10923-16383 | 192.168.1.6:6379 |

### 1.2 客户端重定向机制

当客户端访问某个 key 时，如果 key 所在的槽不在当前节点，节点会返回**重定向指令**，告诉客户端去哪个节点访问。

```bash
# MOVED 重定向（永久）
127.0.0.1:6379> GET name
(error) MOVED 5798 192.168.1.2:6379

# ASK 重定向（临时，用于槽迁移中）
127.0.0.1:6379> GET name
(error) ASK 5798 192.168.1.2:6379
```

Smart 客户端（如 JedisCluster、Redis-py cluster）会缓存槽到节点的映射关系，遇到 MOVED/ASK 时更新映射并重试请求。

## 二、槽迁移原理

### 2.1 槽迁移的触发条件

槽迁移通常在以下场景触发：

1. **节点添加**：新节点加入集群，需要从其他节点迁移槽
2. **节点移除**：节点下线前，需要将其槽迁移到其他节点
3. **负载不均**：手动调整槽分布，优化数据倾斜
4. **故障转移**：从节点升级为主节点后，需要重新分配槽

槽迁移的核心命令是 `CLUSTER SETSLOT`，配合 `MIGRATING` 和 `IMPORTING` 状态。

### 2.2 迁移过程详解

假设要把槽 5798 从节点 A 迁移到节点 B，完整的迁移流程如下：

**第一步：标记目标节点为 IMPORTING**

```bash
# 在节点 B 上执行
192.168.1.2:6379> CLUSTER SETSLOT 5798 IMPORTING abcd1234
```

`abcd1234` 是节点 A 的节点 ID，表示这个槽正在从节点 A 迁入。

**第二步：标记源节点为 MIGRATING**

```bash
# 在节点 A 上执行
192.168.1.1:6379> CLUSTER SETSLOT 5798 MIGRATING 192.168.1.2:6379
```

此时，节点 A 进入 MIGRATING 状态，节点 B 进入 IMPORTING 状态。

**第三步：迁移 key 数据**

```bash
# 在节点 A 上执行
192.168.1.1:6379> CLUSTER GETKEYSINSLOT 5798 100
1) "user:1001"
2) "user:1002"
...

# 逐个迁移 key 到节点 B
192.168.1.1:6379> MIGRATE 192.168.1.2 6379 "user:1001" 0 5000
```

`MIGRATE` 命令会原子性地将 key 从节点 A 迁移到节点 B。

**第四步：设置槽归属**

```bash
# 在任意节点执行（通常在集群管理节点）
192.168.1.1:6379> CLUSTER SETSLOT 5798 NODE 192.168.1.2:6379
```

此时槽 5798 的归属权正式转移给节点 B。

### 2.3 MOVED vs ASK：两种重定向的区别

这是面试高频题，很多同学分不清两者。

| 对比 | MOVED | ASK |
|---|---|---|
| 性质 | 永久重定向 | 临时重定向 |
| 含义 | 槽已归属其他节点 | 槽正在迁移中 |
| 客户端行为 | 更新本地槽映射，下次直接访问新节点 | 只对当前请求重试一次，后续请求仍访问原节点 |
| 缓存影响 | 客户端会缓存新的槽映射 | 不影响客户端的槽缓存 |

**为什么需要 ASK 重定向？**

考虑迁移过程中的边界情况：槽 5798 正在从 A 迁移到 B，此时客户端请求 `user:1001`：

- 如果 key 还在 A 上：A 正常返回数据
- 如果 key 已迁移到 B：A 返回 ASK，客户端去 B 请求
- 如果 key 不存在：A 返回 nil

ASK 重定向的意义是：**在迁移过程中，保证请求能找到 key，同时不影响客户端的槽缓存**。如果用 MOVED，客户端会更新缓存，导致大量不必要的重定向。

:::tip 迁移期间的数据一致性
槽迁移过程中存在一个**不一致窗口**。假设 `user:1001` 从 A 迁移到 B 后，客户端继续向 A 写入数据，这个写入会丢失（因为槽已不属于 A）。

正确的做法是：迁移期间**暂停向源节点写入**，或者在应用层使用 `ASKING` 命令强制向 IMPORTING 状态的节点写入。
:::

### 2.4 迁移优化的参数配置

```bash title="redis.conf"
# 槽迁移的超时时间
cluster-node-timeout 15000

# 迁移期间批量获取 key 的数量
cluster-migration-barrier 1
```

| 参数 | 说明 |
|---|---|
| `cluster-node-timeout` | 节点超时时间（毫秒），影响故障检测 |
| `cluster-migration-barrier` | 主节点最少从节点数，低于则从其他主迁移从节点 |

## 三、热点扩容与数据倾斜

### 3.1 数据倾斜的根源

Redis Cluster 通过哈希槽实现了数据的均匀分布，但现实中的业务数据往往**天然倾斜**。

**场景一：hash tag 导致的热 key 集中**

```bash
# 这三个 key 都包含 "product"，会落在同一个槽
product:detail:1001
product:detail:1002
product:detail:1003
```

**场景二：业务热点集中**

```bash
# 爆款商品的所有数据都在同一个 key
stock:sku:999999  # 库存
price:sku:999999  # 价格
sales:sku:999999  # 销量
```

**场景三：分片不均**

某些业务场景下，key 的哈希值分布本身就不均匀，导致部分槽的数据量远大于其他槽。

### 3.2 热点 key 在集群下的困境

单节点 Redis 遇到热点 key 时，可以通过**本地缓存 + Redis 分布式缓存**的方式解决。但 Redis Cluster 下，热点 key 的问题更棘手：

1. **单节点扛不住**：热点 key 全部打在一个节点上，该节点的 CPU 和网卡可能成为瓶颈
2. **无法水平扩容**：Redis Cluster 的数据分片是 key 级别的，同一个 key 无法分散到多个节点
3. **热点 key 影响整个 slot**：热点 key 所在的整个 slot 都会受到影响

### 3.3 解决方案：热点 key 打散

**方案一：使用 hash tag 将热点 key 分散到不同 slot**

```bash
# 热点商品 ID
product:999999

# 拆分为多个 key，分散到不同槽
product:{user:001}:999999
product:{user:002}:999999
product:{user:003}:999999
```

优点：实现了热点的水平扩展。缺点：聚合查询变得复杂，需要客户端合并结果。

**方案二：本地缓存 + Redis**

```
用户请求 → 本地缓存（Guava/Caffeine） → Redis → DB
```

对于读多写少的热点数据，本地缓存能拦截大部分请求，减少对 Redis 的压力。

**方案三：热点 key 发现与隔离**

热点 key 发现工具：
- `redis-cli --hotkeys`：发现热 key
- `MONITOR` 命令：实时监控 key 访问频率
- 自定义统计：业务层埋点统计 key 访问量

热点 key 隔离策略：
```bash
# 发现热点 key 后，将其复制到多个 slot
COPY hot:product:999999 slot1
COPY hot:product:999999 slot2
COPY hot:product:999999 slot3
```

客户端通过 hash tag 轮询访问不同的副本：

```java
// 伪代码
String getHotKey(String key) {
    int slot = key.hashCode() % replicaCount;
    return "hot:" + key + ":replica" + slot;
}
```

:::warning 热点 key 的取舍
热点 key 分散会增加业务复杂度，需要权衡：
- 如果热点 key 的访问量确实很大，值得做
- 如果只是为了「以防万一」，可能过度设计
- 优先考虑本地缓存，这是成本最低的方案
:::

## 四、集群参数调优

### 4.1 核心参数配置

```bash title="redis.conf"
# 节点超时时间（毫秒）
# 节点超过此时间未响应 PING，则标记为疑似下线
cluster-node-timeout 15000

# 从节点故障转移有效性因子
# 从节点超过 (node-timeout * replica-validity-factor) 未同步主节点数据，
# 则不会发起故障转移
cluster-replica-validity-factor 10

# 主节点最少从节点数
# 主节点的从节点数低于此值时，会从其他主迁移从节点
cluster-migration-barrier 1

# 集群是否启用
cluster-enabled yes

# 集群配置文件（自动生成，不要手动修改）
cluster-config-file nodes-6379.conf
```

### 4.2 故障检测与转移

Redis Cluster 通过 **PING/PONG 心跳**实现故障检测：

```bash
# 节点定期向其他节点发送 PING
192.168.1.1:6379> CLUSTER INFO
cluster_state:ok
cluster_slots_assigned:16384
cluster_slots_ok:16384
cluster_known_nodes:6
cluster_failures:0
```

**故障转移流程**：

1. **主观下线（pfail）**：节点 A 向节点 B 发送 PING，超过 `cluster-node-timeout` 未收到 PONG，标记 B 为 pfail
2. **客观下线（fail）**：节点 A 通知其他节点，如果集群中超过半数主节点都认为 B 下线，则 B 被标记为 fail
3. **从节点发起选举**：B 的从节点向其他主节点发送 `FAILOVER_AUTH_REQUEST`，申请成为新主节点
4. **主节点投票**：其他主节点根据数据新鲜度、优先级等投票
5. **故障转移执行**：获得多数票的从节点升级为新主节点，其他从节点修改复制关系

```bash
# 查看集群节点状态
192.168.1.1:6379> CLUSTER NODES
abcd1234... master myself,connected 0-5460
efgh5678... slave abcd1234 connected 0-5460
```

### 4.3 读写分离的坑

很多同学以为 Redis Cluster 也支持像主从复制那样的读写分离，其实**不支持**。

Redis Cluster 的读写行为：
- **写操作**：只能路由到 key 所在槽的主节点
- **读操作**：默认路由到 key 所在槽的主节点（也可以配置为随机读从节点，但需要客户端支持）

如果需要在集群下实现读写分离，需要客户端**手动实现**：

```java
// Jedis 实现读从节点的伪代码
public String readFromReplica(String key) {
    JedisSlotBasedConnectionHandler handler = ...;
    // 获取 key 对应的槽
    int slot = JedisClusterCRC16.getSlot(key);
    // 获取槽对应的从节点
    JedisPool pool = handler.getSlotConnection(slot).getReplicaPool();
    try (Jedis jedis = pool.getResource()) {
        return jedis.get(key);
    }
}
```

:::tip 为什么不自动支持读写分离？
Redis Cluster 设计为**客户端智能路由**，而不是服务端代理。客户端更了解业务逻辑（哪些读可以容忍脏数据），可以做出更合理的路由决策。
:::

## 五、集群高可用最佳实践

### 5.1 最小集群配置

生产环境的 Redis Cluster 最小配置：

| 配置 | 推荐值 | 说明 |
|---|---|---|
| 主节点数 | `>= 3` | 保证故障转移能获得多数票 |
| 从节点数 | 每主 `>= 1` | 数据冗余 + 自动故障转移 |
| 跨机房 | 建议 | 避免单机房故障导致数据丢失 |

**推荐的集群拓扑**：

```
# 跨机房 3 主 3 从
机房A: 192.168.1.1:6379 (主) ← 192.168.1.2:6379 (从)
机房B: 192.168.1.3:6379 (主) ← 192.168.1.4:6379 (从)
机房C: 192.168.1.5:6379 (主) ← 192.168.1.6:6379 (从)
```

### 5.2 常见故障场景及处理

**场景一：主节点宕机**

```
处理流程：
1. 从节点检测到主节点下线（cluster-node-timeout 内）
2. 从节点发起故障转移，成为新主节点
3. 客户端自动更新槽映射，继续服务
4. 原主节点恢复后，作为从节点加入集群
```

**场景二：网络分区**

```
处理流程：
1. 网络分区导致部分节点失联
2. 超过半数的节点认为失联节点下线（客观下线）
3. 从节点升级为主节点（需获得多数票）
4. 如果没有节点获得多数票，集群不可用
```

:::danger 脑裂问题
Redis Cluster 可能出现**脑裂**：网络分区时，两个分区的节点各自选出了主节点，都对外提供服务，导致数据不一致。

解决思路：
1. 配置合理的 `cluster-node-timeout`，避免误判
2. 使用 `min-slaves-to-write` 限制写入条件
3. 监控 `cluster_failures`，及时处理

```bash
# 写入条件：至少 2 个从节点可用
min-slaves-to-write 2
min-slaves-max-lag 10
```
:::

**场景三：节点扩缩容**

```bash
# 添加新节点作为主节点
redis-cli --cluster add-node 192.168.1.7:6379 192.168.1.1:6379

# 从主节点迁移槽到新主节点
redis-cli --cluster reshard 192.168.1.1:6379 \
    --cluster-from <源节点ID> \
    --cluster-to <目标节点ID> \
    --cluster-slots 1000

# 删除节点（需先将槽迁移走）
redis-cli --cluster del-node 192.168.1.7:6379 <节点ID>
```

### 5.3 集群监控指标

```bash
# 查看集群整体状态
127.0.0.1:6379> CLUSTER INFO
cluster_state:ok              # ok / fail
cluster_slots_assigned:16384  # 已分配的槽数量
cluster_slots_ok:16384        # 状态正常的槽数量
cluster_slots_pfail:0         # 疑似下线的槽数量
cluster_slots_fail:0           # 已下线的槽数量
cluster_known_nodes:6          # 已知节点数
cluster_size:3                 # 主节点数
cluster_current_epoch:5        # 当前集群版本
cluster_stats_messages_ping_sent:1000
cluster_stats_messages_pong_sent:980
```

| 指标 | 含义 |
|---|---|
| `cluster_state` | `ok` 表示集群正常，`fail` 表示不可用 |
| `cluster_slots_fail` | 非 0 表示有槽不可用，部分 key 无法访问 |
| `cluster_stats_messages_ping_sent` | PING 发送次数，过高可能表示网络问题 |

## 六、面试实战

### 6.1 场景一：槽迁移过程

**面试官问**：「Redis Cluster 槽迁移的过程是怎样的？MOVED 和 ASK 有什么区别？」

这道题考察对槽迁移机制的深度理解。

**回答要点**：

> 槽迁移分四步：
> 1. **标记 IMPORTING**：目标节点标记槽为 IMPORTING 状态
> 2. **标记 MIGRATING**：源节点标记槽为 MIGRATING 状态
> 3. **迁移数据**：使用 MIGRATE 命令逐个迁移 key
> 4. **设置归属**：使用 CLUSTER SETSLOT 正式转移槽归属
>
> MOVED 和 ASK 的区别：
> - **MOVED** 是永久重定向，表示槽已经归属其他节点，客户端会更新本地槽映射
> - **ASK** 是临时重定向，表示槽正在迁移中，客户端只对当前请求重试，不更新映射
>
> ASK 重定向是为了在迁移过程中，保证请求能找到 key，同时不影响客户端的槽缓存。

### 6.2 场景二：热点 key 问题

**面试官问**：「热点 key 在 Redis Cluster 下有什么问题？怎么解决？」

这道题考察对集群模式下热点问题的理解。

**回答要点**：

> 热点 key 在集群下的问题是：**所有请求都打在一个节点上**。由于 key 只能在一个节点上，无法像单节点那样通过增加从节点来分流。
>
> 解决方案：
> 1. **热点 key 打散**：用 hash tag 将热点 key 分散到不同槽，但会增加聚合查询的复杂度
> 2. **本地缓存**：热点数据加一层本地缓存（如 Caffeine），减少 Redis 访问
> 3. **热点监控**：用 `redis-cli --hotkeys` 或 MONITOR 发现热点 key，提前做预案
> 4. **复制热点 key**：将热点 key 复制多份到不同槽，客户端轮询访问
>
> 最推荐的方案是**本地缓存 + Redis**，这是成本最低、效果最好的方案。

### 6.3 场景三：集群高可用

**面试官问**：「Redis Cluster 如何实现高可用？节点故障了怎么处理？」

这道题考察对集群高可用机制的理解。

**回答要点**：

> Redis Cluster 的高可用通过**主从复制 + 自动故障转移**实现。
>
> **故障检测**：节点通过 PING/PONG 心跳检测其他节点的存活状态，超过 `cluster-node-timeout` 未响应则标记为疑似下线（pfail）。
>
> **客观下线**：如果超过半数主节点都认为某节点下线，则该节点被标记为客观下线（fail）。
>
> **故障转移**：从节点发现主节点下线后，向其他主节点申请成为新主节点。获得多数票的从节点升级为新主节点，其他从节点修改复制关系。
>
> **最小配置**：生产环境建议至少 3 主 3 从，且主从跨机房部署，避免单机房故障导致数据丢失。

## 总结

Redis Cluster 的核心是**槽迁移**和**故障转移**。

槽迁移通过 IMPORTING/MIGRATING 状态和 ASK 重定向，在迁移过程中保证请求能找到 key，同时不影响客户端的槽缓存。热点 key 在集群下的问题需要通过热点打散、本地缓存等方案解决。

集群高可用依赖**主从复制 + 自动故障转移**机制，最小配置是 3 主 3 从，且建议跨机房部署。`cluster-node-timeout` 的选择需要在灵敏度和稳定性之间权衡。

理解了 Redis Cluster 的这些核心机制，你就掌握了分布式缓存集群的命门。下一步，可以深入了解 Redis Cluster 在实际业务中的使用，以及与 Codis、Twemproxy 等其他分片方案的对比。
