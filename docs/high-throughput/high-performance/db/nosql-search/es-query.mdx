# Elasticsearch 查询优化：从 500ms 到 50ms 的实战

你的 ES 查询从 500ms 优化到 50ms，QPS 从 1000 提升到 8000。面试官问你：「你做了什么优化？」

你愣了一下——改动有点多，一时不知道从哪说起。是把 `query` 改成 `filter` 的功劳？还是 `doc_values` 优化的功劳？还是 `search_after` 解决了深度分页问题？

如果你答不清楚，说明你对 ES 查询优化的理解还停留在「调参」层面，没有建立系统化的优化思维。

这篇文章，帮你建立 ES 查询优化的完整知识体系。我会从查询流程开始，剖析每个环节的性能损耗，告诉你什么情况下该优化什么，以及如何判断优化效果。

## 一、ES 查询流程深度解析

### 1.1 一次查询的背后

当你执行一个 ES 查询时，ES 内部发生了什么？让我们追踪一下：

```json
GET /products/_search
{
  "query": {
    "bool": {
      "must": [
        { "match": { "category": "电子产品" } }
      ],
      "filter": [
        { "term": { "status": 1 } },
        { "range": { "price": { "gte": 100, "lte": 1000 } } }
      ]
    }
  },
  "from": 0,
  "size": 20,
  "sort": [
    { "sales": "desc" },
    { "_score": "desc" }
  ],
  "_source": ["name", "price", "sales"]
}
```

整个查询流程如下：

```
┌──────────────────────────────────────────────────────────────────┐
│                     ES 查询流程                                    │
│                                                                   │
│  1. 客户端 → 协调节点（Coordinating Node）                        │
│  │   接收请求，解析查询条件                                        │
│  ↓                                                                 │
│  2. 协调节点 → 所有分片（广播查询）                                │
│  │   将查询分发到 Primary Shard（如果配置了 Preferred-node，       │
│  │   会优先发送到对应节点）                                        │
│  ↓                                                                 │
│  3. 各分片并行执行查询                                            │
│  │   - 查询条件缓存（query cache）                                │
│  │   - 索引文件（.fnm, .doc, .pos 等）                            │
│  │   - 排序字段的 doc_values / fielddata                          │
│  ↓                                                                 │
│  4. 各分片 → 协调节点（返回结果）                                  │
│  │   返回 doc_ids + 排序值 + 分数（只返回 size 条）                │
│  ↓                                                                 │
│  5. 协调节点合并结果                                              │
│  │   - 全局排序（如果是多分片查询）                                │
│  │   - 聚合计算                                                   │
│  │   - 返回最终结果                                               │
│  ↓                                                                 │
│  6. 协调节点 → 客户端                                             │
└──────────────────────────────────────────────────────────────────┘
```

### 1.2 查询各阶段的耗时分布

理解查询流程后，我们来看每个阶段的耗时特点：

| 阶段 | 耗时占比 | 可优化点 |
|-----|---------|---------|
| 查询条件匹配 | 40-60% | 索引设计、filter vs query |
| 评分计算 | 10-20% | 减少不必要评分 |
| 排序 | 15-30% | doc_values、index sorting |
| 聚合计算 | 20-40% | 近似聚合、shard size |
| 结果合并 | 5-10% | 分片数量、结果集大小 |

### 1.3 分片与副本的影响

ES 的查询性能与分片策略密切相关：

```json
// 创建索引时指定分片数和副本数
PUT /products
{
  "settings": {
    "number_of_shards": 3,
    "number_of_replicas": 1
  },
  "mappings": {
    "properties": {
      "name": { "type": "text" },
      "price": { "type": "integer" },
      "sales": { "type": "integer" }
    }
  }
}
```

**分片数的影响**：

| 分片数 | 优点 | 缺点 |
|-------|------|------|
| 少 | 结果合并快 | 单分片数据量大，查询慢 |
| 多 | 并行度高，吞吐高 | 结果合并开销大，资源消耗高 |

:::tip 分片数的设计原则

**经验公式**：`分片数 = 数据量 / 20GB`

- 单个分片数据量控制在 20-50GB 为宜
- 单个索引的分片数建议 3-5 个（可根据数据量调整）
- 避免过多小分片（每个分片都有元数据开销）

副本数影响读性能：

- 副本可以分担查询压力，提升读 QPS
- 但副本会占用磁盘空间，且增加数据同步开销
- 读多写少场景：副本数 = 1-2
- 写多读少场景：副本数 = 0 或 1

:::

## 二、分页方式对比：from/size vs search_after vs scroll

### 2.1 三种分页方式概述

ES 支持三种分页方式，各有适用场景：

| 方式 | 原理 | 优点 | 缺点 | 适用场景 |
|-----|------|------|------|---------|
| from/size | 深度分页，从0开始偏移 | 简单易用，支持跳页 | `from + size` `>` 10000 会报错 | 浅分页（`<` 10000） |
| search_after | 基于排序值的游标 | 无深度限制，性能稳定 | 不支持跳页 | 深度分页、排序不变 |
| scroll | 生成数据快照 | 可遍历全量数据 | 消耗资源，不适合实时查询 | 数据导出 |

### 2.2 from/size：简单但有限制

`from/size` 是最直观的分页方式：

```json
// 查询第 1 页
GET /products/_search
{
  "from": 0,
  "size": 20
}

// 查询第 51 页
GET /products/_search
{
  "from": 1000,
  "size": 20
}
```

**`from/size` 的问题**：协调节点需要将所有分片的前 `from + size` 条结果都取出来，然后全局排序。

假设：
- 索引有 5 个分片
- 查询 `from=10000, size=20`
- 每个分片需要返回 10020 条数据
- 协调节点需要合并 5 × 10020 = 50100 条数据，然后丢弃 50080 条

这就是深度分页性能差的根本原因。

:::warning from/size 的限制

ES 默认限制 `from + size` `<=` 10000（由 `index.max_result_window` 控制）。

超过这个限制会报错：

```json
{
  "error": {
    "type": "search_phase_execution_exception",
    "reason": "Result window is too large, from + size must be less than or equal to: [10000]"
  }
}
```

解决方案：使用 `search_after` 或调大 `index.max_result_window`（但调大后深度分页依然很慢）。

:::

### 2.3 search_after：游标分页的原理

`search_after` 使用上一页的排序值作为游标，跳过已读取的数据：

```json
// 第一页查询
GET /products/_search
{
  "size": 20,
  "sort": [
    { "sales": "desc" },
    { "_id": "desc" }
  ],
  "query": {
    "term": { "category": "电子产品" }
  }
}
```

返回结果的最后一条排序值作为下一页的 `search_after`：

```json
// 第一页返回
{
  "hits": {
    "hits": [
      { "_id": "1", "sort": [1000, "1"] },
      { "_id": "2", "sort": [999, "2"] },
      // ... 共 20 条
    ]
  }
}

// 第二页查询（使用第一页最后一条的 sort 值）
GET /products/_search
{
  "size": 20,
  "sort": [
    { "sales": "desc" },
    { "_id": "desc" }
  ],
  "search_after": [999, "2"],
  "query": {
    "term": { "category": "电子产品" }
  }
}
```

**`search_after` 的优势**：

- 不需要全局排序，每个分片独立返回 `size` 条
- 查询时间恒定，与 `from` 无关
- 支持深度分页，不受 `max_result_window` 限制

**`search_after` 的限制**：

- 不支持跳页（无法直接跳到第 50 页）
- 排序字段值必须唯一或包含唯一字段（如 `_id`）
- 查询过程中数据变化，可能导致结果重复或遗漏

### 2.4 scroll：全量遍历

`scroll` 用于全量数据导出，不是实时查询：

```json
// 第一页：指定 scroll context 有效期
GET /products/_search?scroll=5m
{
  "size": 1000,
  "query": { "match_all": {} }
}

// 返回 scroll_id
{
  "_scroll_id": "DXF1ZXJ5QW5kRmV0Y2gBAAAAAAAAAAE...",
  "hits": { ... }
}

// 后续页面：使用 scroll_id
POST /_search/scroll
{
  "scroll": "5m",
  "scroll_id": "DXF1ZXJ5QW5kRmV0Y2gBAAAAAAAAAAE..."
}

// 清理 scroll context（查询完成后）
DELETE /_search/scroll
{
  "scroll_id": "DXF1ZXJ5QW5kRmV0Y2gBAAAAAAAAAAE..."
}
```

:::warning scroll 的问题

1. **占用资源**：`scroll` 会创建快照，保存中间的排序结果，消耗内存和磁盘资源
2. **不适合实时查询**：快照期间的数据变化不会被反映
3. **需要手动清理**：scroll context 不会自动过期，需要定期清理或设置 TTL

实际生产环境中，如果需要深度分页，**推荐 `search_after`**，而不是 `scroll`。

:::

## 三、filter 与 query：性能差异的根源

### 3.1 两者的本质区别

ES 中，查询分为两类：**query** 和 **filter**。

| 类型 | 特点 | 性能 | 使用场景 |
|-----|------|------|---------|
| query | 相关性评分，`_score` | 慢（需要计算评分） | 需要排序的场景 |
| filter | 二值过滤，不评分 | 快（结果可缓存） | 精确过滤 |

```json
// query：需要计算相关性评分
{
  "query": {
    "match": { "title": "手机 苹果" }
  }
}

// filter：只做二值过滤，不计算评分
{
  "query": {
    "bool": {
      "filter": [
        { "term": { "status": 1 } },
        { "range": { "price": { "gte": 1000 } } }
      ]
    }
  }
}
```

### 3.2 filter 更快的原因

**原因一：结果可缓存**

`filter` 的结果是二值的（匹配/不匹配），ES 会自动将结果缓存到 `query cache` 中。同样的 `filter` 第二次执行时，直接从缓存读取，不需要访问索引文件。

```json
// 第一次执行 filter
{
  "filter": [
    { "term": { "status": 1 } }
  ]
}

// 结果被缓存到 query cache
// 第二次执行同样 filter，直接返回缓存结果
```

**原因二：不计算评分**

`query` 需要计算每个文档与查询条件的相关性分数（`_score`），这个计算涉及倒排表查询、TF/IDF 计算等。`filter` 只判断是否匹配，不计算分数，省去了这部分开销。

### 3.3 混合使用：bool 查询

实际查询中，通常需要同时满足「相关性」和「精确过滤」：

```json
{
  "query": {
    "bool": {
      "must": [
        // 需要评分的条件
        { "match": { "title": "手机" } }
      ],
      "filter": [
        // 不需要评分的过滤条件
        { "term": { "status": 1 } },
        { "range": { "price": { "gte": 1000, "lte": 5000 } } },
        { "term": { "category": "电子产品" } }
      ]
    }
  }
}
```

:::tip filter 优化的黄金法则

**能用 filter 的地方，就不要用 query。**

这不仅仅是为了性能，更重要的是语义正确：

- `must`：影响相关性评分，用于「我希望结果匹配这个」
- `filter`：不影响评分，用于「我只需要匹配这个」

比如查询「状态为 1 的手机」：

- `filter: { term: { status: 1 } }` 是正确的——status 是过滤条件，不应该影响「手机」的相关性评分
- `must: { term: { status: 1 } }` 是错误的——把 status 当成相关性条件了

:::

## 四、查询优化实战：10 个实用技巧

### 4.1 技巧一：使用 filter 而非 query

这是最基础也是最重要的优化。检查你的查询，能用 `filter` 的地方都改成 `filter`。

```json
// 优化前：全部用 must
{
  "query": {
    "bool": {
      "must": [
        { "term": { "status": 1 } },
        { "range": { "price": { "gte": 1000 } } },
        { "match": { "title": "手机" } }
      ]
    }
  }
}

// 优化后：精确过滤用 filter
{
  "query": {
    "bool": {
      "must": [
        { "match": { "title": "手机" } }
      ],
      "filter": [
        { "term": { "status": 1 } },
        { "range": { "price": { "gte": 1000 } } }
      ]
    }
  }
}
```

### 4.2 技巧二：减少返回字段（_source filtering）

`"_source"` 字段存储了文档的原始 JSON。返回不需要的字段会增加网络传输和序列化开销。

```json
// 只返回需要的字段
{
  "_source": ["name", "price", "sales"],
  "query": { ... }
}

// 完全禁用 _source（不需要返回文档内容时）
{
  "_source": false,
  "query": { ... }
}

// 通配符过滤
{
  "_source": {
    "includes": ["name", "price*"],
    "excludes": ["description"]
  }
}
```

### 4.3 技巧三：禁用不必要字段的评分

如果某字段不需要参与评分，可以用 `filter` 或 `must_not`：

```json
// 优化前：精确匹配用 match，消耗评分计算
{
  "query": {
    "bool": {
      "must": [
        { "match": { "category_id": "3" } }  // category_id = 3，不需要评分
      ]
    }
  }
}

// 优化后：精确匹配用 term filter
{
  "query": {
    "bool": {
      "filter": [
        { "term": { "category_id": 3 } }
      ]
    }
  }
}
```

### 4.4 技巧四：使用 profile API 分析查询性能

`profile` API 可以展示查询各阶段的耗时：

```json
GET /products/_search
{
  "profile": true,
  "query": {
    "bool": {
      "must": [
        { "match": { "title": "手机" } }
      ],
      "filter": [
        { "term": { "status": 1 } }
      ]
    }
  }
}
```

返回结果中的 `profile` 部分展示了详细耗时：

```json
{
  "profile": {
    "shards": [
      {
        "search_type": "QUERY_THEN_FETCH",
        "id": "[node1][products][0]",
        "Lucene": {
          "query": [
            {
              "type": "BooleanQuery",
              "description": "title:手机",
              "time_in_nanos": "1234567",
              "breakdown": {
                "score": 500000,
                "build_scorer": 300000,
                "match": 100000,
                "bulk_score": 0
              }
            }
          ],
          "rewrite_time": 50000
        }
      }
    ]
  }
}
```

### 4.5 技巧五：合理使用 doc_values

`doc_values` 是 ES 用于排序和聚合的列式存储结构。ES 7.x+ 默认开启。

```json
// 查看 doc_values 是否开启
GET /products/_mapping

// 结果
{
  "properties": {
    "sales": {
      "type": "integer",
      "doc_values": true
    }
  }
}
```

**何时需要 `doc_values`**：

- 需要排序（`sort`）
- 需要聚合（`aggs`）
- 需要脚本访问字段值

**何时可以禁用**：

- 字段只用于全文检索
- 不需要排序和聚合

```json
// 创建索引时禁用 doc_values
{
  "mappings": {
    "properties": {
      "title": {
        "type": "text",
        "doc_values": false  // 只用于搜索，不需要排序
      },
      "sales": {
        "type": "integer",
        "doc_values": true  // 需要排序，必须开启
      }
    }
  }
}
```

### 4.6 技巧六：index sorting 减少排序开销

`index sorting` 是 ES 7.x+ 的特性，允许在写入时预先按指定字段排序，减少查询时的排序开销。

```json
// 创建索引时指定 index sorting
{
  "settings": {
    "index": {
      "sort.field": ["category", "sales"],
      "sort.order": ["asc", "desc"]
    }
  }
}
```

这样查询 `sort: { category: asc, sales: desc }` 时，数据已经预先排序，无需额外排序。

### 4.7 技巧七：使用 routing 直接命中分片

ES 默认根据 `_id` 哈希决定文档所属分片。如果查询有明确的路由条件，可以使用 `routing` 直接定位到目标分片，避免广播查询。

```json
// 创建索引时指定 routing 字段
{
  "settings": {
    "number_of_shards": 5
  },
  "mappings": {
    "_routing": {
      "required": true,
      "path": "user_id"
    }
  }
}

// 写入时指定 routing
POST /user_actions/_doc?routing=1001
{
  "user_id": 1001,
  "action": "click",
  "timestamp": "2024-01-15T10:30:00Z"
}

// 查询时指定 routing，直接定位到目标分片
GET /user_actions/_search?routing=1001
{
  "query": {
    "term": { "user_id": 1001 }
  }
}
```

### 4.8 技巧八：分片数量与副本策略

```json
// 小数据量（< 1GB）：减少分片数
{
  "settings": {
    "number_of_shards": 1,
    "number_of_replicas": 1
  }
}

// 中等数据量（1-10GB）：标准配置
{
  "settings": {
    "number_of_shards": 3,
    "number_of_replicas": 1
  }
}

// 大数据量（> 10GB）：增加分片数
{
  "settings": {
    "number_of_shards": 5,
    "number_of_replicas": 1
  }
}
```

### 4.9 技巧九：warmers 预热查询缓存

对于重要的查询，可以使用 `warmers` 预加载到缓存：

```json
// 创建 warmer
PUT /products/_warmer/my_warmer
{
  "query": {
    "bool": {
      "filter": [
        { "term": { "status": 1 } }
      ]
    }
  },
  "aggs": {
    "top_categories": {
      "terms": { "field": "category" }
    }
  }
}
```

### 4.10 技巧十：控制聚合的分桶数量

ES 的聚合先在每个分片上计算 top N 个桶，再在协调节点合并。如果 `size` 太大，会消耗大量内存。

```json
// 分片返回的桶数，默认为 10
{
  "aggs": {
    "top_categories": {
      "terms": {
        "field": "category",
        "size": 10,  // 只返回前 10 个桶
        "shard_size": 20  // 分片维度返回 20 个桶（比 size 大，以提高精度）
      }
    }
  }
}
```

## 五、聚合查询优化

### 5.1 近似聚合：精度换性能

对于基数（cardinality）和百分位数，可以使用近似算法大幅提升性能：

```json
// 精确基数聚合（慢）
{
  "aggs": {
    "unique_users": {
      "cardinality": {
        "field": "user_id",
        "precision_threshold": 40000  // 超过 40000 时使用 HyperLogLog++
      }
    }
  }
}

// 百分位数聚合（快，近似值）
{
  "aggs": {
    "price_percentiles": {
      "percentiles": {
        "field": "price",
        "percents": [50, 90, 95, 99],
        "tdigest": {
          "compression": 200  // 精度参数，值越大越精确，但越慢
        }
      }
    }
  }
}
```

| 聚合类型 | 近似算法 | 精度 | 性能 |
|---------|---------|------|------|
| cardinality | HyperLogLog++ | `precision_threshold` 以内 100% 准确 | 极快 |
| percentiles | TDigest | 典型误差 1-5% | 快 |
| median_absolute_deviation | TDigest | 典型误差 5-10% | 快 |

### 5.2 深度分桶问题与 shard_size 调优

ES 的聚合是两层聚合：

```
分片维度（shard_size）→ 协调节点合并 → 返回 size 个桶
```

如果某个桶的基数很大，可能导致「本该在前 N 名的桶被丢弃」：

```json
// 解决方案：调大 shard_size
{
  "aggs": {
    "top_products": {
      "terms": {
        "field": "product_id",
        "size": 10,
        "shard_size": 100  // 分片维度返回 100 个桶，合并后取前 10
      }
    }
  }
}
```

### 5.3 预聚合 + 缓存

对于频繁执行的聚合查询，可以预先计算并缓存：

```json
// 使用 named cache
{
  "aggs": {
    "daily_sales": {
      "filter": { "range": { "date": { "gte": "2024-01-01", "lt": "2024-01-02" } } },
      "aggs": {
        "total": { "sum": { "field": "sales" } }
      }
    }
  }
}
```

## 六、排序优化

### 6.1 doc_values vs fielddata

ES 有两种存储排序字段值的数据结构：

| 类型 | 存储位置 | 特点 | 适用场景 |
|-----|---------|------|---------|
| doc_values | 磁盘（列式存储） | 默认开启，大数据量下更高效 | 排序、聚合 |
| fielddata | 内存 | 默认禁用，text 字段可用 | text 字段的排序 |

**doc_values（默认）**：

```json
{
  "properties": {
    "sales": {
      "type": "integer",
      "doc_values": true  // 默认值
    }
  }
}
```

**fielddata（text 字段）**：

```json
{
  "properties": {
    "title": {
      "type": "text",
      "fielddata": true,  // 加载到内存，非常消耗资源
      "fielddata_frequency_filter": {
        "min": 0.001,
        "max": 0.1,
        "min_segment_size": 500
      }
    }
  }
}
```

:::warning fielddata 的问题

`fielddata` 将字段值全部加载到内存，数据量大时会消耗大量堆内存，导致 OOM。除非必要，尽量不要在 text 字段上开启 `fielddata`。

最佳实践：在 text 字段旁加一个 `keyword` 子字段用于排序：

```json
{
  "properties": {
    "title": {
      "type": "text",
      "fields": {
        "keyword": {
          "type": "keyword"
        }
      }
    }
  }
}

// 查询时用 keyword 子字段排序
{
  "sort": [
    { "title.keyword": "asc" }
  ]
}
```

:::

### 6.2 index sorting 与联合排序

如果查询的排序是固定的，可以使用 `index sorting` 预排序：

```json
{
  "settings": {
    "index": {
      "sort.field": ["status", "sales"],
      "sort.order": ["asc", "desc"]
    }
  }
}
```

这样查询 `sort: [{ status: asc }, { sales: desc }]` 时，数据已经预先排序。

## 七、零停机索引重建

### 7.1 Reindex API

ES 支持将数据从一个索引迁移到另一个索引：

```json
// 基本用法
POST /_reindex
{
  "source": {
    "index": "products_old"
  },
  "dest": {
    "index": "products_new"
  }
}
```

### 7.2 Alias 切换实现零停机

```bash
# 1. 创建新索引
PUT /products_v2
{
  "settings": {
    "number_of_shards": 3,
    "number_of_replicas": 1
  },
  "mappings": { ... }
}

# 2. 重建数据
POST /_reindex
{
  "source": { "index": "products" },
  "dest": { "index": "products_v2" }
}

# 3. 添加 alias（原子操作，无停机）
POST /_aliases
{
  "actions": [
    { "remove": { "index": "products", "alias": "products" } },
    { "add": { "index": "products_v2", "alias": "products" } }
  ]
}
```

## 八、面试实战

### 8.1 场景一：ES 分页方式的选择

**面试官问**：「ES 分页方式有哪些？各自的优缺点是什么？」

**深入回答**：

> ES 支持三种分页方式：
>
> **第一种：from/size**，最直观，`from` 指定偏移，`size` 指定返回条数。优点是简单、支持跳页；缺点是 `from + size` 不能超过 10000，深度分页性能差。
>
> **第二种：search_after**，基于排序值的游标分页。上一页返回的 sort 值作为下一页的 search_after。优点是无深度限制，查询时间恒定；缺点是不支持跳页，排序值必须唯一。
>
> **第三种：scroll**，生成数据快照，适合全量导出。优点是可以遍历全量数据；缺点是消耗资源，不适合实时查询，不支持跳页。
>
> **选择建议**：浅分页（`<` 10000）用 from/size；深度分页（无限滚动）用 search_after；数据导出用 scroll。

### 8.2 场景二：深度分页查询优化

**面试官问**：「如何优化 ES 的深度分页查询？」

**深入回答**：

> 深度分页性能差的根本原因是**协调节点需要合并大量数据**。
>
> **优化方向一：使用 search_after**。用排序值作为游标，每次只取 LIMIT 条，不存在 offset 累加问题。适用场景：无限滚动场景。
>
> **优化方向二：减少分片数或使用 preference**。分片越多，合并开销越大。对于不要求高精度的场景，可以用 `preference=_local` 优先查询本地分片。
>
> **优化方向三：减少返回字段**。`_source` 越大，序列化和网络传输越慢。只返回必要的字段。
>
> **优化方向四：使用 filter**。filter 结果可缓存，不消耗评分计算，减少查询时间。
>
> **优化方向五：使用近似聚合**。如果查询包含聚合，可以用 HyperLogLog++ 等近似算法，大幅提升性能。

### 8.3 场景三：search_after 的原理

**面试官问**：「search_after 的原理是什么？」

**深入回答**：

> search_after 的原理是利用上一页查询返回的 sort 值，作为下一页查询的起点。
>
> 传统的 from/size 是**全局定位**：协调节点告诉每个分片「返回第 from 条到第 from+size 条」。每个分片需要扫描前 from+size 条，然后丢弃前 from 条。
>
> search_after 是**局部定位**：上一页最后一条的 sort 值是一个游标，每个分片只需要从游标位置往后扫描 size 条。这个游标通常是排序字段的值（比如销售额）和文档 ID。
>
> 举个例子：按销售额降序分页。
>
> - 第一页查询：sort: [{ sales: desc }, { _id: desc }]，返回最后一条的 sort = [999, "abc"]
> - 第二页查询：search_after: [999, "abc"]，相当于告诉 ES「从销售额 999、ID abc 的后面开始返回」
>
> 这样，每个分片只需要扫描 20 条数据，性能恒定，与页数无关。

## 总结

ES 查询优化是一个系统化工程，需要从多个维度入手：

**分页策略**：

- 浅分页用 `from/size`，深度分页用 `search_after`，数据导出用 `scroll`
- `search_after` 是深度分页的最佳选择

**查询类型**：

- 精确过滤用 `filter`，相关性评分用 `query`
- filter 结果可缓存，性能更高

**返回字段**：

- 只返回必要的字段，减少网络传输和序列化开销

**排序与聚合**：

- 合理使用 `doc_values` 和 `fielddata`
- 近似聚合（HyperLogLog、TDigest）可以用精度换性能

**索引设计**：

- 合理的分片数和副本策略
- 使用 `routing` 直接命中目标分片
- 使用 `index sorting` 预排序

**监控与调优**：

- 使用 `profile` API 分析查询各阶段耗时
- 持续监控查询性能，识别慢查询

记住一句话：**优化不是调参，而是理解原理后有针对性的改进**。理解了 ES 的查询流程和内部机制，才能找到真正的性能瓶颈。
