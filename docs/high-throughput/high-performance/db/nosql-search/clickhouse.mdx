# ClickHouse 性能优化：列式数据库的调优艺术

MySQL 处理一条聚合查询要 5 秒，而 ClickHouse 只需要 50 毫秒。这个 100 倍的性能差距，不是来自硬件，而是来自架构设计的根本差异。

ClickHouse 是 Yandex（俄罗斯搜索引擎）为了解决海量日志分析问题而开发的列式数据库。它的设计目标很明确：**在海量数据上进行高速聚合查询**。如果你的业务场景是数据分析、报表统计、用户行为分析，ClickHouse 是目前最优秀的选择之一。

但 ClickHouse 不是万能药。如果你的场景是点查询（根据主键查单条数据）、频繁写入（每秒上万次单条插入）、事务处理，ClickHouse 可能不是最佳选择。选择合适的工具，和优化工具一样重要。

这篇文章，帮你建立 ClickHouse 性能优化的完整知识体系。

---

## 一、ClickHouse 架构与 MySQL 的本质差异

### 1.1 行式存储 vs 列式存储

MySQL 是行式存储数据库，数据按行组织：

```
[Row 1: id=1, name=张三, age=30, ...]
[Row 2: id=2, name=李四, age=25, ...]
...
```

ClickHouse 是列式存储数据库，数据按列组织：

```
[id列: 1, 2, 3, 4, ...]
[name列: 张三, 李四, 王五, ...]
[age列: 30, 25, 28, ...]
...
```

这个差异带来了巨大的性能优势：

**列式存储的优势：**

- **列聚合只需要读取需要的列**。查询 `SELECT SUM(price) FROM orders`，行式存储需要读取所有列，而列式存储只需要读取 `price` 一列，IO 减少了几十倍。
- **同一列的数据类型相同，压缩率更高**。比如 `age` 列全是整数，可以用 delta encoding、LRZ 等算法压缩，存储空间减少 5-10 倍。
- **向量化执行**。列式存储的数据在内存中连续排列，CPU 可以一次处理多个值（SIMD 指令），充分利用 CPU 的并行能力。

### 1.2 ClickHouse 的核心组件

```
┌─────────────────────────────────────────────────────────────┐
│                         ClickHouse                          │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                    查询层（Query）                    │   │
│  │  ┌──────────┐  ┌──────────┐  ┌──────────┐            │   │
│  │  │ Shard 1  │  │ Shard 2  │  │ Shard 3  │   ...     │   │
│  │  │ (副本1)  │  │ (副本2)  │  │ (副本3)  │            │   │
│  │  └──────────┘  └──────────┘  └──────────┘            │   │
│  └─────────────────────────────────────────────────────┘   │
│  ┌─────────────────────────────────────────────────────┐   │
│  │              协调节点（Keeper / ZooKeeper）            │   │
│  └─────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
```

**表引擎（Table Engine）**：ClickHouse 的核心，决定数据的存储方式和查询性能。常用表引擎包括：

| 表引擎 | 适用场景 | 特点 |
|-------|---------|------|
| MergeTree | 通用场景，高性能 | 支持主键索引、分区、TTL |
| ReplacingMergeTree | 需要去重的场景 | 按主键去重 |
| SummingMergeTree | 需要预聚合的场景 | 按排序键自动聚合 |
| AggregatingMergeTree | 物化视图聚合 | 配合物化视图使用 |
| CollapsingMergeTree | 增量数据同步 | 支持增量 + 撤销 |

**主键索引**：ClickHouse 的主键用于数据排序，而非唯一约束。与 MySQL 的 B+ 树索引完全不同。

**分区（Partition）**：数据按分区组织，查询时可以通过分区裁剪跳过不相关数据。

## 二、MergeTree 表引擎：ClickHouse 的性能基石

### 2.1 MergeTree 的数据结构

MergeTree 表的数据组织如下：

```
┌─────────────────────────────────────────────────────────────┐
│  Partition 202401                                              │
│  ┌─────────────────────────────────────────────────────┐     │
│  │  Part 202401_1_1_0                                    │     │
│  │  ├── primary.idx（主键索引，每隔 8192 行取一个值）       │     │
│  │  ├── price.mrk（列标记文件）                           │     │
│  │  ├── price.bin（压缩后的列数据）                       │     │
│  │  ├── create_time.mrk                                 │     │
│  │  ├── create_time.bin                                 │     │
│  │  └── ...                                             │     │
│  └─────────────────────────────────────────────────────┘     │
│  ┌─────────────────────────────────────────────────────┐     │
│  │  Part 202401_2_2_0                                    │     │
│  └─────────────────────────────────────────────────────┘     │
└─────────────────────────────────────────────────────────────┘
```

关键文件：

- **`*.idx`**：主键索引文件，包含排序键的值，用于快速定位数据块
- **`*.mrk`**：列标记文件，记录列数据在 `.bin` 文件中的偏移量
- **`*.bin`**：压缩后的列数据

### 2.2 主键与排序键的设计

MergeTree 的主键决定数据的物理排序顺序：

```sql
CREATE TABLE orders (
    id UInt64,
    order_no String,
    user_id UInt64,
    price Decimal(10, 2),
    status UInt8,
    create_time DateTime
) ENGINE = MergeTree()
ORDER BY (user_id, create_time)  -- 排序键
PARTITION BY toYYYYMM(create_time)  -- 分区键
PRIMARY KEY user_id;  -- 主键（可选，默认等于 ORDER BY）
```

**排序键的设计原则**：

1. **等值查询字段放前面**：经常用 `WHERE user_id = ?` 查询，就把 `user_id` 放排序键第一位
2. **范围查询字段放后面**：经常用 `WHERE create_time BETWEEN ? AND ?` 查询，`create_time` 放后面
3. **低基数字段优先**：相同值少的字段放前面，索引效率更高

:::tip 排序键 vs 主键

在 MergeTree 中，`ORDER BY` 定义的是**排序键**，决定数据的物理顺序。`PRIMARY KEY` 是可选的，默认等于 `ORDER BY`。

排序键用于：
- 数据块的快速二分查找
- 减少数据扫描范围

主键用于：
- 唯一性约束（可选，不强制）

不要把高基数的随机字段（如 UUID）放在排序键第一位，否则会导致数据写入不均匀、查询效率低下。

:::

### 2.3 分区键与分区裁剪

分区键决定数据的物理分区：

```sql
PARTITION BY toYYYYMM(create_time)  -- 按月分区
PARTITION BY toYYYYMMDD(create_time)  -- 按天分区（数据量大时）
PARTITION BY (user_id % 10)  -- 按取模分区（不推荐）
```

查询时，如果 WHERE 条件包含分区字段，ClickHouse 可以**跳过不相关的分区**：

```sql
-- 查询 2024 年 1 月的数据
SELECT * FROM orders WHERE create_time BETWEEN '2024-01-01' AND '2024-01-31';

-- ClickHouse 执行计划会显示：Only splitting marks from necessary partitions
```

分区数量的选择：

| 数据量级 | 建议分区策略 |
|---------|------------|
| `<` 100 万 | 不分区或按年分区 |
| 100 万 - 1 亿 | 按月分区 |
| `>` 1 亿 | 按天分区 |
| `>` 10 亿 | 按天分区 + 多种粒度 |

:::warning 分区过多的问题

分区太多会导致：
- 元数据压力大（ZooKeeper/Keeper 压力）
- merge 任务碎片化
- 查询时需要合并更多分区

建议单个 ClickHouse 实例的分区数不超过 1000 个。

:::

## 三、数据写入优化：批量、异步、零压力

### 3.1 批量写入：每次至少 1000 行

ClickHouse 的写入优化核心是**批量写入**。单条插入会触发频繁的 merge 操作，严重影响性能：

```sql
-- 错误做法：单条插入
INSERT INTO orders VALUES (1, 'A001', 100, 99.99, 1, now());
INSERT INTO orders VALUES (2, 'A002', 101, 199.99, 1, now());
INSERT INTO orders VALUES (3, 'A003', 102, 299.99, 1, now());

-- 正确做法：批量插入
INSERT INTO orders VALUES
    (1, 'A001', 100, 99.99, 1, now()),
    (2, 'A002', 101, 199.99, 1, now()),
    (3, 'A003', 102, 299.99, 1, now());
```

**批量写入的最佳实践**：

1. **单次批量至少 1000 行**，推荐 5000-50000 行
2. **单次批量大小控制在 5-50MB**，超过 50MB 可能导致内存压力
3. **使用 HTTP 接口批量写入**，比 JDBC 更快

```bash
# 使用 clickhouse-client 批量导入
clickhouse-client --query="INSERT INTO orders FORMAT CSV" < orders.csv

# 使用 curl 批量写入
curl -X POST 'http://localhost:8123/?query=INSERT+INTO+orders+FORMAT+JSONEachRow' \
  --data-binary @data.json
```

### 3.2 异步写入：提升写入吞吐量

ClickHouse 支持异步写入模式，写入请求先写入本地缓冲区，后台线程批量写入：

```sql
-- 开启异步插入
SET async_insert = 1;

-- 设置异步插入的缓冲区大小
SET async_insert_buffer_size = 1048576;  -- 1MB

-- 设置等待数据的最长时间
SET async_insert_max_wait_time = 60000;  -- 60秒
```

```xml title="config.xml"
<clickhouse>
    <async_insert>true</async_insert>
    <async_insert_busy_timeout_ms>60000</async_insert_busy_timeout_ms>
    <async_insert_max_data_size>1048576</async_insert_max_data_size>
</clickhouse>
```

### 3.3 关闭副本写入：初始导入优化

对于初始数据导入，可以关闭副本同步以提升写入速度：

```sql
-- 导入时关闭副本同步
SET insert_distributed_sync = 0;

-- 设置写入等待副本数
SET insert_quorum = 0;  -- 0 = 不等待副本确认
```

:::warning 生产环境注意事项

关闭副本同步只适用于**初始数据导入**。生产环境的实时写入必须保证数据可靠性，`insert_quorum` 应设置为合理的值（如 `1` 或 `majority`）。

:::

## 四、查询优化：从 5 秒到 50 毫秒

### 4.1 分区裁剪：跳过不相关数据

分区裁剪是 ClickHouse 最重要的优化手段之一。如果查询条件包含分区字段，ClickHouse 可以直接跳过不相关的分区：

```sql
-- 查询 2024 年 1 月数据，自动跳过其他分区
SELECT
    user_id,
    count() AS order_count,
    sum(price) AS total_amount
FROM orders
WHERE create_time BETWEEN '2024-01-01' AND '2024-01-31'
GROUP BY user_id;

-- 查看执行计划，确认分区裁剪生效
EXPLAIN indexes = 1
SELECT * FROM orders WHERE create_time BETWEEN '2024-01-01' AND '2024-01-31';
```

**分区裁剪的执行计划特征**：

```
ReadFromMergeTree
├── ExpressionAnalyzer
│   └── PartitionFilter: create_time IN ('2024-01-01' ... '2024-01-31')
└── Ranges: total 128 ranges in 2 partitions (only reading 16 granules)
```

### 4.2 主键索引：快速定位数据块

ClickHouse 的主键索引类似稀疏索引，每隔 8192 行记录一个值。查询时，ClickHouse 先用二分查找定位数据块，再在块内扫描：

```sql
-- 创建表时指定主键
CREATE TABLE user_events (
    user_id UInt64,
    event_type String,
    event_time DateTime,
    properties String
) ENGINE = MergeTree()
ORDER BY (user_id, event_time)
PRIMARY KEY user_id;

-- 查询特定用户的数据，ClickHouse 会用主键索引快速定位
SELECT * FROM user_events WHERE user_id = 12345;
```

**主键索引的稀疏性**：

```
primary.idx 文件内容（每隔 8192 行一个值）：
[12345, 67890, 123456, ...]  -- user_id 的采样值

Granule 结构：
Granule 0: rows 0-8191, primary key value = 12345
Granule 1: rows 8192-16383, primary key value = 67890
...
```

### 4.3 物化视图：预聚合加速查询

对于频繁执行的聚合查询，可以用物化视图预计算结果：

```sql
-- 创建物化视图：按天聚合订单数据
CREATE MATERIALIZED VIEW order_daily
ENGINE = SummingMergeTree()
PARTITION BY toYYYYMM(day)
ORDER BY (day, user_id)
AS
SELECT
    toDate(create_time) AS day,
    user_id,
    count() AS order_count,
    sum(price) AS total_amount
FROM orders
GROUP BY day, user_id;

-- 查询物化视图，秒级响应
SELECT
    user_id,
    sum(order_count) AS total_orders,
    sum(total_amount) AS total_amount
FROM order_daily
WHERE day BETWEEN '2024-01-01' AND '2024-01-31'
GROUP BY user_id;
```

**物化视图的更新时机**：

- 物化视图在数据写入时**自动增量更新**
- 查询物化视图时，数据可能略有延迟（通常秒级）
- 适合对实时性要求不高的统计报表场景

### 4.4 Projection：多种排序方式

Projection 是 ClickHouse 21.3+ 引入的特性，允许为同一张表创建多种排序方式：

```sql
-- 创建主表，按 user_id 排序
CREATE TABLE events (
    user_id UInt64,
    event_type String,
    event_time DateTime,
    properties String
) ENGINE = MergeTree()
ORDER BY (user_id, event_time);

-- 创建 Projection，按 event_time 排序
ALTER TABLE events ADD PROJECTION proj_by_time
(
    SELECT * ORDER BY event_time
);

-- 查询时自动使用最优 Projection
SELECT count() FROM events WHERE event_time BETWEEN '2024-01-01' AND '2024-01-31';
```

**何时使用 Projection**：

- 同一张表有多种查询模式
- 不想维护多张物化视图
- 查询需要返回原始数据（不是聚合结果）

### 4.5 PREWHERE：提前过滤减少数据读取

`PREWHERE` 优化先读取过滤条件字段，再根据结果读取其他字段：

```sql
-- 普通查询：读取所有字段后过滤
SELECT user_id, price, status FROM orders WHERE user_id = 12345;

-- PREWHERE：先读取 user_id，过滤后再读取其他字段
SELECT user_id, price, status FROM orders PREWHERE user_id = 12345;

-- 等效写法（ClickHouse 自动优化）
SELECT user_id, price, status FROM orders WHERE user_id = 12345;
```

ClickHouse 会自动将 `WHERE` 条件转换为 `PREWHERE`，无需手动指定。

## 五、物化视图进阶：预聚合的最佳实践

### 5.1 物化视图的类型

| 类型 | 适用场景 | 特点 |
|-----|---------|------|
| SummingMergeTree | 累加聚合（sum, count） | 自动合并相同键的值 |
| AggregatingMergeTree | 复杂聚合（uniq, quantile） | 需要配合 -State 聚合函数 |
| ReplacingMergeTree | 去重 | 按主键去重 |

### 5.2 聚合函数的使用

```sql
-- 创建聚合物化视图
CREATE MATERIALIZED VIEW user_stats
ENGINE = AggregatingMergeTree()
PARTITION BY toYYYYMM(day)
ORDER BY (day, user_id)
AS
SELECT
    toDate(create_time) AS day,
    user_id,
    uniqState(order_no) AS order_count,  -- 去重计数
    sumState(price) AS total_amount,     -- 求和
    quantileState(0.5)(price) AS price_median  -- 中位数
FROM orders
GROUP BY day, user_id;

-- 查询物化视图
SELECT
    user_id,
    uniqMerge(order_count) AS order_count,
    sumMerge(total_amount) AS total_amount,
    quantileMerge(0.5)(price_median) AS price_median
FROM user_stats
WHERE day = '2024-01-15'
GROUP BY user_id;
```

:::tip -State 和 -Merge 函数

ClickHouse 的聚合物化视图使用两阶段聚合：
1. **写入时**：用 `-State` 函数（如 `sumState`）保存中间状态
2. **查询时**：用 `-Merge` 函数（如 `sumMerge`）合并中间状态得到最终结果

:::

## 七、ClickHouse 的常见坑与避坑指南

### 7.1 坑一：大量小批次写入

**问题**：每分钟写入几百条数据，导致大量小 merge 任务，后台合并压力大，查询性能下降。

**原因**：ClickHouse 的 merge 策略是后台合并小文件成大文件。如果写入太频繁，小文件来不及合并就产生更多小文件，形成「merge 风暴」。

**解决方案**：

1. **攒批写入**：将数据先写入 Kafka 或本地缓冲区，达到一定量后再批量写入 ClickHouse
2. **调整 merge 频率**：`max_merge_list_size` 控制单次 merge 的分区数
3. **使用 Buffer 引擎**：临时缓冲写入，减少 merge 压力

```sql
-- 创建 Buffer 表缓冲写入
CREATE TABLE orders_buffer (
    id UInt64,
    order_no String,
    price Decimal(10, 2)
) ENGINE = Buffer(
    'default',          -- 目标数据库
    'orders',           -- 目标表
    16,                  -- 后台 flush 线程数
    60,                  -- 最长等待时间（秒）
    1000000,             -- 缓冲区行数阈值
    10485760             -- 缓冲区大小阈值（10MB）
);
```

### 7.2 坑二：全表扫描导致内存爆炸

**问题**：查询没有 WHERE 条件，导致全表扫描，内存不足。

**解决方案**：

1. **必须带 WHERE 条件查询**
2. **限制 LIMIT**：`SELECT *` 限制结果集大小
3. **设置查询超时**：`max_execution_time`
4. **使用采样查询**：对于大数据集，先采样查看数据分布

```sql
-- 设置查询超时
SET max_execution_time = 30;  -- 30 秒超时

-- 使用采样查询
SELECT count() FROM orders SAMPLE 0.01;  -- 采样 1%
```

### 7.3 坑三：JOIN 效率低

**问题**：ClickHouse 的 JOIN 效率不如 MySQL，特别是大表 JOIN 大表。

**原因**：ClickHouse 的 JOIN 是基于哈希算法，数据需要全部加载到内存。如果 JOIN 的表都很大，内存压力巨大。

**解决方案**：

1. **避免大表 JOIN**：尽量在写入时预聚合
2. **使用 `GLOBAL IN`**：相当于广播 JOIN，小表广播到大表
3. **使用字典（Dictionary）**：将小表加载为内存字典

```sql
-- 使用 GLOBAL IN（广播小表）
SELECT * FROM large_table
WHERE id GLOBAL IN (SELECT id FROM small_table);

-- 创建字典
CREATE DICTIONARY small_table_dict (
    id UInt64 DEFAULT 0,
    name String DEFAULT ''
)
PRIMARY KEY id
SOURCE(CLICKHOUSE(HOST 'localhost' PORT 9000 TABLE 'small_table'))
LAYOUT(FLAT())
LIFETIME(3600);
```

### 7.4 坑四：NULL 值处理

**问题**：ClickHouse 对 NULL 值的处理比较特殊，可能导致意外结果。

```sql
-- NULL 值在某些聚合函数中被忽略
SELECT sum(price) FROM orders;  -- NULL 不参与求和

-- NULL 在 GROUP BY 中单独一组
SELECT status, count() FROM orders GROUP BY status;
-- 结果：status=0 (100条), status=1 (200条), NULL (50条)

-- 解决方案：使用 coalesce 或显式处理
SELECT coalesce(status, 0), count() FROM orders GROUP BY coalesce(status, 0);
```

## 八、面试实战

### 8.1 场景一：ClickHouse vs MySQL 的选择

**面试官问**：「在什么场景下你会选择 ClickHouse 而不是 MySQL？」

**深入回答**：

> ClickHouse 和 MySQL 的定位完全不同，选择依据主要是**数据规模和查询类型**。
>
> **选择 ClickHouse 的场景**：
>
> - 数据量级在**千万级以上**的日志、行为数据分析
> - 需要**高速聚合查询**（GROUP BY、COUNT、SUM）
> - 查询以**分析为主**，对实时性要求秒级或分钟级
> - 写入模式是**批量导入**，而非频繁单条插入
>
> **选择 MySQL 的场景**：
>
> - 数据量级在**千万级以下**的业务数据
> - 需要**事务支持**（UPDATE、DELETE）
> - 查询以**点查为主**（根据主键查单条）
> - 需要**实时写入**，每秒上万次单条插入
>
> 一个典型的选择案例：电商订单数据，用户量 100 万，订单量 1 亿。用 MySQL 存最近三个月的热数据（满足点查需求），用 ClickHouse 存全量订单数据（满足分析需求）。

### 8.2 场景二：ClickHouse 查询优化思路

**面试官问**：「ClickHouse 上一条查询要 30 秒，你如何优化？」

**深入回答**：

> ClickHouse 30 秒的查询需要系统性地排查。
>
> **第一步，确认是否走了分区裁剪**。执行 `EXPLAIN indexes = 1` 看查询计划，是否只扫描了必要的分区。
>
> **第二步，检查主键设计**。看 WHERE 条件中的字段是否在主键或排序键中。如果不在，需要添加索引或修改查询条件。
>
> **第三步，确认是否需要物化视图**。如果是频繁执行的聚合查询，创建物化视图预计算。
>
> **第四步，检查数据倾斜**。看 `system.parts` 表，确认各分区的数据量是否均匀。
>
> **第五步，检查资源瓶颈**。看 `system.metrics` 中的 CPU、内存、磁盘 IO 是否饱和。
>
> **第六步，看是否有大表 JOIN**。如果是 JOIN 导致的性能问题，考虑用字典或预聚合替代。

### 8.3 场景三：ClickHouse 的写入优化

**面试官问**：「ClickHouse 的写入性能很差，你怀疑是哪些原因？」

**深入回答**：

> ClickHouse 写入性能差，通常是以下几个原因：
>
> **第一，写入批量太小**。单次 INSERT 只有几十行或几行，导致频繁的 merge。解决：攒批写入，每次至少 1000 行。
>
> **第二，索引设计不合理**。排序键选择不当，导致写入时数据倾斜。解决：选择分布均匀的字段作为排序键。
>
> **第三，后台 merge 压力太大**。小文件太多，merge 线程不够用。解决：使用 Buffer 引擎攒批，或调大 `background_pool_size`。
>
> **第四，网络 IO 成为瓶颈**。写入数据量大，网络带宽不足。解决：压缩数据后写入。
>
> **第五，副本同步延迟**。写入时等待所有副本确认。解决：初始导入时关闭副本同步，生产环境设置合理的 `insert_quorum`。

### 题目一：MergeTree 的 merge 机制

ClickHouse 的 MergeTree 表引擎，后台会不断合并小文件成大文件。假设一个分区内有 100 个小文件，每次 merge 合并两个文件，变成 50 个文件，再 merge 变成 25 个……

你能算一下，从 100 个小文件合并成 1 个大文件，最少需要多少次 merge 操作？

这次 merge 过程中，磁盘空间峰值是多少（假设每个文件 100MB）？

### 题目二：物化视图的实时性

物化视图的数据是**增量更新**的，但这个更新不是实时的，而是**异步**的。

如果你的业务需要查询「当前时刻」的聚合数据，物化视图可能无法满足需求。你能想到什么方案，在保证查询性能的同时，实现近实时的数据聚合？

## 总结

ClickHouse 是为分析场景而生的列式数据库，其性能优势来自于：

- **列式存储**：只读取需要的列，IO 效率高
- **向量化执行**：充分利用 CPU 的 SIMD 能力
- **稀疏索引**：快速跳过不相关的数据块
- **分区裁剪**：查询时跳过不相关的分区
- **物化视图**：预聚合计算，查询时直接返回结果

优化 ClickHouse 的核心思路：

1. **写入优化**：批量写入、攒批、关闭不必要的副本同步
2. **查询优化**：充分利用分区裁剪、主键索引、物化视图
3. **设计优化**：合理的排序键、分区策略、索引设计
4. **资源调优**：内存配置、线程池大小、merge 策略

选择 ClickHouse 的关键判断：**数据量级大、查询以聚合分析为主、写入是批量模式**。如果你的场景不满足这三点，MySQL 可能是更好的选择。
