# Using filesort 与 Using temporary 优化思路

你一定见过这样的 EXPLAIN 输出：

```
+----+-------------+--------+------------+------+---------------+------+---------+--------+---------+----------+----------------+
| id | select_type | table  | type       | key  | rows          | rws  | rlen    | Extra                      |
+----+-------------+--------+------------+------+---------------+------+---------+--------+---------+----------+----------------+
|  1 | SIMPLE      | orders | ALL        | NULL | 5,000,000     | 5M   | 1280M   | Using where; Using filesort |
+----+-------------+--------+------------+------+---------------+------+----------+----------------+
```

`Using filesort`——很多同学看到这个词，第一反应是「文件排序，文件排序，那是排序在文件里，肯定慢」。然后就开始折腾 `sort_buffer_size`，调大排序缓冲区。

但问题真的是这样吗？

实际上，`Using filesort` 和「文件」关系不大，它真正的问题是：**MySQL 无法利用索引的有序性，只能自己排序**。这篇文章，带你彻底搞懂 `Using filesort` 和 `Using temporary` 的真正含义，以及如何优化。

---

## 一、Using filesort：不是「文件排序」

### 1.1 名字的误导

`Using filesort` 这个名字非常容易误导人。它的本意是「使用了排序算法进行排序」，至于是用内存还是用磁盘文件，取决于 `sort_buffer_size` 和待排序数据量。

MySQL 的排序过程：

1. **小数据量**：在内存中完成排序，使用 `sort_buffer_size` 分配的缓冲区
2. **大数据量**：内存放不下时，使用磁盘临时文件进行「外部排序」

所以，`Using filesort` 的问题不在于「文件」，而在于**没有利用索引的有序性**——这才是性能杀手。

### 1.2 MySQL 排序的原理

当 MySQL 需要对查询结果排序，但无法利用索引的有序性时，它会走 `filesort` 流程：

```sql
SELECT * FROM orders
WHERE status = 1
ORDER BY create_time DESC;
```

如果 `status` 有索引，但 `create_time` 没有索引，MySQL 的执行过程：

1. **全表扫描**：扫描 `status = 1` 的所有行
2. **回表**：根据主键查所有字段
3. **排序**：把第 2 步的所有结果加载到 `sort_buffer`，按 `create_time` 排序
4. **返回**：返回排序后的结果

问题出在第 3 步：如果有 100 万条 `status = 1` 的数据，MySQL 就要把这 100 万行的数据全部加载到 `sort_buffer` 中排序。这是巨大的内存和 CPU 开销。

:::tip sort_buffer_size 的正确理解

`sort_buffer_size` 是每个连接（线程）独占的排序缓冲区大小，不是全局共享的。

- 默认值：256KB
- 常见配置：2MB - 4MB
- **不是越大越好**：过大的值会导致内存碎片，每个连接独占大内存还会消耗系统资源

:::

### 1.3 Using filesort 的两种算法

MySQL 的 `filesort` 有两种算法：

**双路排序（MySQL 4.1 之前）**：

1. 读取所有行指针和 ORDER BY 列，排序
2. 根据排好序的行指针，回表读取完整数据

**单路排序（MySQL 4.1 之后，默认）**：

1. 读取所有需要查询的列到 `sort_buffer`
2. 在 `sort_buffer` 中直接排序
3. 直接返回结果，不需要回表

单路排序比双路排序减少了回表次数，但如果 `sort_buffer` 不够用，反而会更慢（需要用临时文件）。

## 二、Using temporary：临时表的代价

### 2.1 什么是 Using temporary

`Using temporary` 表示 MySQL 在执行查询时，需要创建**临时表**来存储中间结果。临时表可能是内存表（MEMORY 引擎）或磁盘表（MyISAM/InnoDB）。

```sql
EXPLAIN SELECT customer_id, COUNT(*) as order_count
FROM orders
GROUP BY customer_id;
```

```
+----+-------------+--------+------------+-------+---------------+--------+---------+--------+---------+----------+---------------------------------+
| id | select_type | table  | type       | key   | rows          | rws    | rlen    | Extra                            |
+----+-------------+--------+------------+-------+---------------+--------+---------+--------+---------+----------+---------------------------------+
|  1 | SIMPLE      | orders | index      | idx_cid | NULL          | 50000  | 50000   | Using index; Using temporary; Using filesort |
+----+-------------+--------+------------+-------+---------------+--------+---------+--------+---------+----------+---------------------------------+
```

### 2.2 Using temporary 的常见场景

**场景一：GROUP BY 字段没有索引**

```sql
SELECT customer_id, SUM(amount) as total
FROM orders
GROUP BY customer_id;
```

MySQL 执行过程：

1. 全表扫描 `orders` 表
2. 创建临时表，按 `customer_id` 分组聚合
3. 排序（如果 `GROUP BY` 包含多个字段，且没有索引）
4. 返回结果

如果数据量大，临时表会从内存表溢出到磁盘，性能急剧下降。

**场景二：DISTINCT 去重**

```sql
SELECT DISTINCT customer_id
FROM orders
WHERE status = 1;
```

`DISTINCT` 本质上是分组去重，MySQL 会创建一个临时表来存储已见过的 `customer_id`，遇到重复值就去掉。

**场景三：UNION 查询**

```sql
SELECT id FROM orders WHERE status = 1
UNION
SELECT id FROM orders WHERE status = 2;
```

`UNION` 默认会去重，MySQL 会创建临时表存储两个子查询的结果，再做去重。

### 2.3 临时表的性能问题

`Using temporary` 的性能瓶颈主要来自：

1. **临时表创建开销**：创建和销毁临时表需要时间和资源
2. **内存溢出**：数据量大时，内存临时表会溢出到磁盘（需要 `Using filesort`）
3. **复制成本**：如果临时表在磁盘上，MySQL 需要复制数据

:::tip 临时表相关参数

| 参数 | 说明 | 默认值 |
|-----|------|-------|
| `tmp_table_size` | 内存临时表的最大大小 | 16MB |
| `max_heap_table_size` | MEMORY 引擎表的最大大小 | 16MB |

如果临时表超过 `tmp_table_size`，会转为磁盘临时表，性能急剧下降。

:::

## 三、优化思路：让 MySQL 利用索引

### 3.1 核心原则

`Using filesort` 和 `Using temporary` 的根本问题是：**MySQL 无法利用索引完成排序和分组**。优化的核心思路是**建立合适的索引，让 MySQL 直接利用索引的有序性和聚合能力**。

### 3.2 优化 ORDER BY：利用索引消除 filesort

**优化前**：

```sql
-- orders 表没有 (status, create_time) 联合索引
SELECT * FROM orders
WHERE status = 1
ORDER BY create_time DESC;
```

```
Extra: Using where; Using filesort
```

**优化后**：建立联合索引 `(status, create_time)`

```sql
ALTER TABLE orders ADD INDEX idx_status_create(status, create_time);
```

```sql
-- 现在可以利用索引的有序性，直接返回排序结果
SELECT * FROM orders
WHERE status = 1
ORDER BY create_time DESC;
```

```
Extra: Using index condition; Using filesort
-- 或者
Extra: Using where; Using index; Using filesort
```

等等——怎么还是有 `Using filesort`？

这里有个关键点：**`SELECT *` 会触发回表，回表后破坏了索引的有序性**。如果只查询索引列，就可以完全消除 filesort。

**完全消除 filesort**：

```sql
-- 只查索引列，不需要回表
SELECT status, create_time FROM orders
WHERE status = 1
ORDER BY create_time DESC;
```

```
Extra: Using index condition
-- filesort 消失了！
```

### 3.3 优化多字段 ORDER BY

多字段排序时，索引的顺序至关重要。

```sql
-- 按 status 升序，create_time 降序
SELECT * FROM orders
ORDER BY status ASC, create_time DESC;
```

**联合索引设计**：

| 排序方式 | 正确索引 | 错误索引 |
|---------|---------|---------|
| `ORDER BY status ASC, create_time ASC` | `(status, create_time)` | `(create_time, status)` |
| `ORDER BY status DESC, create_time DESC` | `(status DESC, create_time DESC)` | 只能建升序索引，用 `reverse()` |
| `ORDER BY status ASC, create_time DESC` | `(status, create_time)`，MySQL 支持 DESC | `(status, create_time DESC)` |

:::tip 最左前缀原则与多字段排序

MySQL 的多字段排序遵循**最左前缀原则**：

```sql
-- 索引 (status, create_time) 可以支持的排序：
ORDER BY status;                      -- OK（只用第一列）
ORDER BY status, create_time;         -- OK（按索引顺序）
ORDER BY status DESC, create_time DESC; -- OK（反向扫描）

-- 索引 (status, create_time) 无法支持的排序：
ORDER BY create_time;                 -- NG（跳过第一列）
ORDER BY status ASC, create_time ASC; -- 降序排序，MySQL 8.0+ 以前无法优化
```

如果你的排序涉及多个方向，MySQL 8.0 之前的版本可能无法利用索引。MySQL 8.0+ 支持索引的升降序。

:::

### 3.4 优化 GROUP BY：利用索引消除 temporary

**优化前**：

```sql
-- customer_id 没有索引
SELECT customer_id, COUNT(*) as cnt
FROM orders
GROUP BY customer_id;
```

```
Extra: Using temporary; Using filesort
```

**优化后**：在 `customer_id` 上建立索引

```sql
ALTER TABLE orders ADD INDEX idx_customer_id(customer_id);
```

```sql
SELECT customer_id, COUNT(*) as cnt
FROM orders
GROUP BY customer_id;
```

```
Extra: Using index
-- temporary 和 filesort 都消失了！
```

**多字段 GROUP BY 优化**：

```sql
-- 按 status 和 customer_id 分组
SELECT status, customer_id, COUNT(*) as cnt
FROM orders
GROUP BY status, customer_id;
```

**联合索引设计**：`(status, customer_id)`

```sql
ALTER TABLE orders ADD INDEX idx_status_customer(status, customer_id);
```

## 四、覆盖索引：终极优化手段

### 4.1 什么是覆盖索引

覆盖索引（Covering Index）是指**索引包含了查询所需的所有字段**，MySQL 不需要回表，直接在索引中就能获取数据。

```sql
-- 索引 idx_status_create_time 包含 status, create_time, id
SELECT status, create_time, id FROM orders
WHERE status = 1
ORDER BY create_time DESC;
```

因为 `SELECT`、`WHERE`、`ORDER BY` 的所有字段都在索引中，MySQL 直接在索引中完成查询，不需要回表。

### 4.2 覆盖索引的设计原则

覆盖索引要包含：

1. **WHERE 条件字段**：用于快速定位
2. **ORDER BY 字段**：用于避免 filesort
3. **SELECT 字段**：用于避免回表

```sql
-- 覆盖索引设计
ALTER TABLE orders ADD INDEX idx_cover (
    status,           -- WHERE 条件
    create_time,      -- ORDER BY 排序
    id,               -- 主键（覆盖 SELECT）
    amount            -- SELECT 字段
);
```

```sql
-- 查询可以直接利用覆盖索引
SELECT id, amount, create_time
FROM orders
WHERE status = 1
ORDER BY create_time DESC;
```

### 4.3 覆盖索引的代价

覆盖索引虽好，但不是银弹。它的代价：

1. **索引变大**：索引包含了更多字段，占用更多磁盘空间
2. **写入变慢**：INSERT、UPDATE、DELETE 时需要维护更多索引
3. **内存压力**：大索引可能导致 Buffer Pool 命中率下降

:::tip 覆盖索引的使用建议

- **高频查询**才值得建覆盖索引：如果一个查询执行频率不高，收益有限
- **字段不宜过多**：覆盖索引的字段控制在 3-5 个以内，避免索引过大
- **按需设计**：先分析慢查询，确认 `SELECT`、`WHERE`、`ORDER BY` 的字段，再针对性建索引

:::

## 五、排序优化实战

### 5.1 案例一：订单列表排序优化

**场景**：订单列表页，需要按创建时间倒序展示

```sql
-- 原始 SQL
SELECT * FROM orders
WHERE customer_id = 1001
ORDER BY create_time DESC
LIMIT 20;
```

**分析**：

1. 先看 EXPLAIN，确认是否走了索引
2. 如果 `customer_id` 有索引，但 `create_time` 没有索引，会触发 filesort
3. 需要在 `(customer_id, create_time)` 上建联合索引

**优化**：

```sql
-- 建立联合索引
ALTER TABLE orders ADD INDEX idx_customer_time(customer_id, create_time);

-- 验证优化效果
EXPLAIN SELECT * FROM orders
WHERE customer_id = 1001
ORDER BY create_time DESC
LIMIT 20;
```

```
+----+-------------+--------+------+----------------------+---------+---------+-------+------+-------+
| id | select_type | table  | type | key                  | key_len | rows    | rws   | rlen | Extra |
+----+-------------+--------+------+----------------------+---------+---------+-------+------+-------+
|  1 | SIMPLE      | orders | ref  | idx_customer_time    | 8       | 50      | 1000K | 128K | NULL  |
+----+-------------+--------+------+----------------------+---------+---------+-------+------+-------+
```

`Using filesort` 消失了！

### 5.2 案例二：分组统计优化

**场景**：按商家统计订单数量和金额

```sql
-- 原始 SQL
SELECT merchant_id, COUNT(*) as order_count, SUM(amount) as total_amount
FROM orders
WHERE status = 1
GROUP BY merchant_id;
```

**分析**：如果 `merchant_id` 没有索引，会产生 `Using temporary; Using filesort`

**优化**：

```sql
-- 建立索引
ALTER TABLE orders ADD INDEX idx_status_merchant(status, merchant_id);

-- 验证优化效果
EXPLAIN SELECT merchant_id, COUNT(*) as order_count, SUM(amount) as total_amount
FROM orders
WHERE status = 1
GROUP BY merchant_id;
```

```
+----+-------------+--------+------+----------------------+---------+---------+-------+------+-------------+
| id | select_type | table  | type | key                  | key_len | rows    | rws   | rlen | Extra       |
+----+-------------+--------+------+----------------------+---------+---------+-------+------+-------------+
|  1 | SIMPLE      | orders | ref  | idx_status_merchant  | 4       | 50000   | 50000 | 64M  | Using index |
+----+-------------+--------+------+----------------------+---------+---------+-------+------+-------------+
```

`Using temporary` 和 `Using filesort` 都消失了！

## 六、排序相关参数调优

除了索引优化，MySQL 还有一些参数可以调整排序性能。

| 参数 | 说明 | 调优建议 |
|-----|------|---------|
| `sort_buffer_size` | 排序缓冲区大小 | 2MB - 4MB，不宜过大 |
| `max_sort_length` | 多行数据排序时的最大长度 | 默认 1024，一般够用 |
| `myisam_sort_buffer_size` | MyISAM 表排序缓冲区 | 仅 MyISAM 引擎需要 |
| `tmp_table_size` | 内存临时表大小 | 16MB - 64MB |
| `max_heap_table_size` | MEMORY 引擎表最大大小 | 与 `tmp_table_size` 保持一致 |

:::tip 调参 vs 建索引

调参是**治标**，建索引是**治本**。

- 如果查询没有合适的索引，无论 `sort_buffer_size` 调多大，都无法避免 filesort
- 如果查询有合适的索引，默认的 `sort_buffer_size` 通常够用
- **优先优化索引，在索引无法解决的情况下再考虑调参**

:::

## 七、面试实战

### 7.1 场景一：Using filesort 的含义与优化

**面试官问**：「EXPLAIN 结果里出现 `Using filesort` 是什么意思？如何优化？」

**普通回答**：`Using filesort` 就是文件排序，慢，需要调大 `sort_buffer_size`。

**深入回答**：

> `Using filesort` 的意思是 MySQL 无法利用索引的有序性，需要自己排序。**它和「文件」关系不大**——是否用文件排序取决于数据量和 `sort_buffer_size`。
>
> 优化思路有两个层次：
>
> **第一层：建索引**。这是根本解决之道。如果 `ORDER BY` 的字段有索引，MySQL 直接利用索引的有序性，不需要 filesort。
>
> ```sql
> -- 订单按时间倒序查询，建联合索引
> ALTER TABLE orders ADD INDEX idx_customer_time(customer_id, create_time);
> ```
>
> **第二层：覆盖索引**。如果 `SELECT` 的字段都在索引中，甚至不需要回表，性能最优。
>
> ```sql
> -- 覆盖索引，避免回表和 filesort
> SELECT customer_id, create_time FROM orders
> WHERE customer_id = 1
> ORDER BY create_time DESC;
> ```
>
> `sort_buffer_size` 的调整是辅助手段，不是根本解决之道。

### 7.2 场景二：Using temporary 的产生与优化

**面试官问**：「`Using temporary` 是怎么产生的？如何优化？」

**普通回答**：`Using temporary` 就是用了临时表，需要优化 SQL。

**深入回答**：

> `Using temporary` 表示 MySQL 需要创建临时表来存储中间结果。常见场景有三种：
>
> 1. **GROUP BY 字段没有索引**。MySQL 先扫描全表，再在内存中分组聚合。
> 2. **DISTINCT 去重**。MySQL 创建临时表记录已见过的值。
> 3. **UNION 查询**。MySQL 创建临时表存储合并结果再去重。
>
> 优化的核心是**让 GROUP BY 利用索引**：
>
> ```sql
> -- 在 GROUP BY 字段上建索引
> ALTER TABLE orders ADD INDEX idx_merchant_id(merchant_id);
> ```
>
> 索引建好后，MySQL 可以直接利用索引的有序性完成分组，不需要临时表。

### 7.3 场景三：多字段排序的索引设计

**面试官问**：「如果查询是 `ORDER BY a ASC, b DESC`，索引怎么设计？」

**深入回答**：

> MySQL 8.0 之前，索引只能升序，无法直接支持 `DESC`。需要用 `ASC` 索引，然后 `reverse()` 反转结果，或者在应用层处理。
>
> MySQL 8.0+ 支持**降序索引**：
>
> ```sql
> ALTER TABLE orders ADD INDEX idx_a_b (a ASC, b DESC);
> ```
>
> 这样的索引可以直接支持 `ORDER BY a ASC, b DESC`。
>
> 如果是 MySQL 8.0 之前，只能建 `(a, b)` 升序索引，然后在查询中处理：
>
> ```sql
> SELECT * FROM orders
> ORDER BY a ASC, b DESC;
> -- 这种情况无法利用索引的有序性
> ```
>
> 解决方案：升序索引 + 应用层反转，或者升级到 MySQL 8.0。

## 总结

`Using filesort` 和 `Using temporary` 是 MySQL 性能问题的重要信号。

- **Using filesort**：MySQL 无法利用索引排序，需要自己排序。优化手段是建立合适的索引，让排序在索引中完成。
- **Using temporary**：MySQL 需要创建临时表存储中间结果。优化手段是建立合适的索引，利用索引的有序性和聚合能力。
- **覆盖索引**：索引包含查询所需的所有字段，避免回表和 filesort，是排序优化的终极手段。

调参是辅助手段，建索引是根本解决之道。遇到 `Using filesort` 或 `Using temporary` 时，第一反应应该是**检查索引设计**，而不是去调 `sort_buffer_size`。
