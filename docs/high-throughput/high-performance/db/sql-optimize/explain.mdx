# EXPLAIN 执行计划详解：SQL 优化的指南针

你优化过 SQL 吗？加了索引，查询变快了；删了 JOIN，换成了子查询，RT 降了一半。但你有没有想过——**为什么这样改会有效？**

很多人优化 SQL 是靠「感觉」：听说 JOIN 慢，就不用 JOIN；听说子查询快，就用子查询。但感觉往往不靠谱。生产环境的表结构、数据量、索引情况各不相同，别人的最佳实践可能是你的性能灾难。

正确的做法是：用 |EXPLAIN| 看看 MySQL 到底是怎么执行你的 SQL 的。

EXPLAIN 是 MySQL 提供的 SQL 执行分析工具，它会告诉你：MySQL 打算怎么访问数据、用哪个索引、要不要排序、要不要建临时表。**看懂 EXPLAIN，你就有了 SQL 优化的指南针。**

## 一、EXPLAIN 的基础用法

### 1.1 基本语法

EXPLAIN 的使用非常简单，直接在 SQL 前加上关键字即可：

```sql
EXPLAIN SELECT * FROM users WHERE age > 25 ORDER BY create_time;
```

```json
{
  "id": 1,
  "select_type": "SIMPLE",
  "table": "users",
  "type": "ref",
  "possible_keys": "idx_age",
  "key": "idx_age",
  "key_len": 5,
  "ref": "const",
  "rows": 100,
  "Extra": "Using index condition; Using filesort"
}
```

MySQL 8.0+ 还提供了 `EXPLAIN ANALYZE`，它不仅分析执行计划，还会实际执行 SQL 并报告真实的运行时信息：

```sql
EXPLAIN ANALYZE SELECT * FROM users WHERE age > 25 ORDER BY create_time;
```

```json
{
  "analysis": [
    "-> Sort: users.create_time  (cost=15.25 rows=10) (actual time=0.028..0.032 rows=10 loops=1)",
    "    -> Index lookup on users using idx_age (age>25) (cost=12.25 rows=10) (actual time=0.018..0.025 rows=10 loops=1)"
  ]
}
```

`EXPLAIN ANALYZE` 的关键优势在于：**它会告诉你实际的行数和耗时**，而不仅仅是估算值。这对于判断优化是否有效非常有帮助。

### 1.2 EXPLAIN 的字段概览

| 字段 | 说明 |
|---|---|
| `id` | SELECT 的序号，嵌套查询时用于标识属于哪个 SELECT |
| `select_type` | SELECT 的类型：SIMPLE、PRIMARY、SUBQUERY 等 |
| `table` | 访问的表名 |
| `type` | **核心字段**，访问类型，反映 MySQL 的查找方式 |
| `possible_keys` | 可能使用的索引 |
| `key` | **核心字段**，实际使用的索引 |
| `key_len` | 索引使用的字节数 |
| `ref` | 与索引比较的列或常量 |
| `rows` | 估算需要扫描的行数 |
| `Extra` | **核心字段**，包含大量优化信息 |

下面我们重点讲解面试中最常考察的三个字段：`type`、`key`、`Extra`。

## 二、type 字段：MySQL 的访问方式

### 2.1 访问类型的性能排序

|type| 是 MySQL 决定怎么查找数据的方式，它的值从好到坏依次是：

```
system > const > eq_ref > ref > range > index > ALL
```

| 访问类型 | 含义 | 性能 |
|---|---|---|
| `system` | 表只有一行记录（系统表），等于常量 | **最优** |
| `const` | 通过主键或唯一索引，只匹配一行 | **最优** |
| `eq_ref` | 关联查询时，通过主键或唯一索引关联，只返回一行 | **优** |
| `ref` | 通过普通索引匹配，返回匹配的多行 | **良** |
| `range` | 索引范围扫描，如 BETWEEN、IN、LIKE 等 | **中** |
| `index` | 全索引扫描，扫描整个索引文件 | **差** |
| `ALL` | **全表扫描**，扫描整个数据文件 | **最差** |

记住这个顺序：**越靠左越好，越靠右越需要优化。**

### 2.2 逐个解析访问类型

**`const`：主键或唯一索引的常量查找**

```sql
EXPLAIN SELECT * FROM users WHERE id = 1;
```

```json
{
  "type": "const",
  "key": "PRIMARY",
  "rows": 1
}
```

`const` 是通过主键或唯一索引精确匹配，只返回一行。这是 MySQL 能够达到的最优访问方式。

**`eq_ref`：关联查询中的主键查找**

```sql
EXPLAIN SELECT * FROM orders o JOIN users u ON o.user_id = u.id;
```

```json
{
  "type": "eq_ref",
  "key": "PRIMARY",
  "rows": 1
}
```

在 JOIN 操作中，如果被驱动表通过主键或唯一索引进行匹配，每次匹配只返回一行。

**`ref`：普通索引的等值匹配**

```sql
EXPLAIN SELECT * FROM users WHERE age = 25;
```

```json
{
  "type": "ref",
  "key": "idx_age",
  "rows": 100
}
```

通过普通索引查找，可能返回多行。性能取决于匹配的行数。

**`range`：索引范围扫描**

```sql
EXPLAIN SELECT * FROM users WHERE age BETWEEN 20 AND 30;
EXPLAIN SELECT * FROM users WHERE age IN (20, 25, 30);
EXPLAIN SELECT * FROM users WHERE age > 25;
```

```json
{
  "type": "range",
  "key": "idx_age",
  "rows": 500
}
```

使用索引进行范围查询，比全表扫描好，但仍然需要评估匹配的行数。

**`index`：全索引扫描**

```sql
EXPLAIN SELECT name FROM users ORDER BY name;
```

```json
{
  "type": "index",
  "key": "idx_name",
  "rows": 10000
}
```

扫描整个索引文件。虽然比 `ALL` 快（索引文件比数据文件小），但仍然是性能较差的访问方式。

**`ALL`：全表扫描**

```sql
EXPLAIN SELECT * FROM users WHERE name = '张三';
```

```json
{
  "type": "ALL",
  "key": null,
  "rows": 1000000
}
```

:::danger 全表扫描是最差的访问方式

`type = ALL` 意味着 MySQL 要扫描整张表的全部数据。对于百万级的表，一次全表扫描可能需要几十秒甚至更长。

如果 EXPLAIN 结果中出现 `ALL`，必须想办法优化。
:::

### 2.3 type=ALL 的优化思路

当你发现 EXPLAIN 中出现 `type = ALL` 时，可以从以下几个方向优化：

**方向一：添加合适的索引**

```sql
-- 优化前：全表扫描
EXPLAIN SELECT * FROM users WHERE age = 25;

-- 添加索引
ALTER TABLE users ADD INDEX idx_age (age);

-- 优化后：索引查找
EXPLAIN SELECT * FROM users WHERE age = 25;
-- type = ref, key = idx_age
```

**方向二：利用现有索引**

```sql
-- 假设有 idx_age_name (age, name) 索引
-- 可以改写查询，利用索引覆盖

EXPLAIN SELECT age, name FROM users WHERE age = 25;
-- type = ref, key = idx_age_name
```

**方向三：强制使用索引**

在特殊场景下，可以尝试 `FORCE INDEX`：

```sql
EXPLAIN SELECT * FROM users FORCE INDEX (idx_age) WHERE age = 25;
```

> 注意：`FORCE INDEX` 会让优化器强制使用指定索引，仅在其他方法都无效时使用。

## 三、key 字段：实际使用的索引

### 3.1 possible_keys 与 key 的区别

- `possible_keys`：MySQL 认为可能使用的索引（候选索引）
- `key`：MySQL 实际决定使用的索引

有时候 `possible_keys` 有值，但 `key` 是 `NULL`，说明 MySQL 认为全表扫描更快。

```json
{
  "possible_keys": "idx_age, idx_name",
  "key": null,
  "type": "ALL"
}
```

### 3.2 优化器为什么不选索引？

MySQL 优化器选择索引时会考虑多个因素：

| 因素 | 说明 |
|---|---|
| 索引的基数 | 索引列的唯一值数量，基数越高越容易被选中 |
| 统计数据 | MySQL 根据表统计数据估算每个索引的代价 |
| 查询条件 | `LIKE '%xxx'` 这种模糊查询无法使用索引 |
| 表的大小 | 小表可能全表扫描更快 |

## 四、Extra 字段：执行计划的优化提示

|Extra| 字段包含大量 MySQL 优化决策的信息，是面试中考察的重点。常见的值有：

| Extra 值 | 含义 | 性能影响 |
|---|---|---|
| `Using index` | **索引覆盖**，不需要回表 | **优** |
| `Using where` | 需要在存储引擎后过滤数据 | 需结合场景判断 |
| `Using index condition` | **索引下推**，在索引层面过滤数据 | **良** |
| `Using filesort` | **文件排序**，无法用索引完成排序 | **差** |
| `Using temporary` | **使用临时表**，需要中间结果存储 | **差** |
| `Using index condition; Using MRR` | 使用 MRR 优化随机 IO | **良** |
| `Backward index scan` | 反向索引扫描 | 中性 |

### 4.1 Using index：索引覆盖

```sql
CREATE INDEX idx_age_name ON users(age, name);

EXPLAIN SELECT name FROM users WHERE age = 25;
```

```json
{
  "Extra": "Using index",
  "type": "ref",
  "key": "idx_age_name"
}
```

`Using index` 表示 MySQL 直接从索引中获取数据，**不需要回表**。这是最理想的情况。

索引覆盖的原理：查询所需的列（`name`）恰好在索引 `idx_age_name` 中，无需访问主表。

:::tip 索引覆盖的适用场景

- 查询的列较少且固定
- 需要频繁查询的字段组合
- 对查询性能要求极高的场景

:::

### 4.2 Using index condition：索引下推

```sql
CREATE INDEX idx_age_name ON users(age, name);

EXPLAIN SELECT * FROM users WHERE age > 20 AND name LIKE '张%';
```

```json
{
  "Extra": "Using index condition",
  "type": "range",
  "key": "idx_age_name"
}
```

`Using index condition` 表示 MySQL 使用了**索引下推**（Index Condition Pushdown，ICP）优化。

**索引下推的原理**：

没有索引下推时，查询流程是：
1. 使用索引 `idx_age` 找到 age > 20 的主键值
2. 回表，根据主键获取完整行数据
3. 在 MySQL 服务层过滤 name LIKE '张%'

有索引下推时，查询流程是：
1. 使用索引 `idx_age_name` 找到 age > 20 的记录
2. **在索引层面同时过滤 name LIKE '张%'**，不满足的直接跳过
3. 对满足条件的记录回表

索引下推减少了回表次数，提升了查询性能。

### 4.3 Using filesort：文件排序

```sql
EXPLAIN SELECT * FROM users ORDER BY create_time;
```

```json
{
  "Extra": "Using filesort",
  "type": "ALL"
}
```

`Using filesort` 表示 MySQL 无法使用索引完成排序，需要在内存或磁盘中排序。

:::danger Using filesort 的性能问题

`Using filesort` 不是「使用文件排序」的意思，而是 MySQL 内部的一种排序算法。当数据量小时在内存中排序，数据量大时会使用磁盘临时文件。

排序的成本随数据量增长呈 O(n log n) 级别增长，百万级数据的排序可能需要秒级。
:::

**Using filesort 的常见场景**：

| 场景 | 原因 |
|---|---|
| ORDER BY 字段没有索引 | 无法利用索引的有序性 |
| ORDER BY 字段在索引中但不是最左前缀 | 索引顺序与 ORDER BY 不匹配 |
| 多字段排序但顺序与索引不一致 | 同上 |
| ASC 和 DESC 混用 | 无法利用索引 |

**Using filesort 的解决方案**：

详见《[Using filesort 与 Using temporary 优化思路](./filesort-temporary)》。

### 4.4 Using temporary：临时表

```sql
EXPLAIN SELECT name, COUNT(*) FROM users GROUP BY name;
```

```json
{
  "Extra": "Using temporary; Using filesort",
  "type": "ALL"
}
```

`Using temporary` 表示 MySQL 需要创建临时表来存储中间结果。

**Using temporary 的常见场景**：

| 场景 | 原因 |
|---|---|
| GROUP BY 的列不在索引中 | 无法边读边分组，需要临时表 |
| DISTINCT 多列 | 无法利用索引去重 |
| UNION / UNION ALL | UNION 需要去重，UNION ALL 不需要 |
| ORDER BY 和 GROUP BY 的字段不同 | 无法同时满足两种需求 |

**Using temporary 的解决方案**：

详见《[Using filesort 与 Using temporary 优化思路](./filesort-temporary)》。

### 4.5 Using MRR：多范围读优化

```sql
CREATE INDEX idx_id_name ON orders(id, name);

EXPLAIN SELECT * FROM orders WHERE id IN (1, 3, 5, 7, 9);
```

```json
{
  "Extra": "Using index condition; Using MRR",
  "type": "range",
  "key": "idx_id_name"
}
```

`Using MRR`（Multi-Range Read）优化了随机 IO。当查询需要访问多个不连续的主键值时，MRR 会先将主键值排序，然后按顺序访问数据，减少磁盘随机 IO。

### 4.6 Backward index scan：反向索引扫描

```sql
CREATE INDEX idx_create_time ON orders(create_time DESC);

EXPLAIN SELECT * FROM orders ORDER BY create_time ASC;
```

```json
{
  "Extra": "Backward index scan",
  "type": "index",
  "key": "idx_create_time"
}
```

`Backward index scan` 表示 MySQL 使用反向索引扫描。当 ORDER BY 的顺序与索引顺序相反时，MySQL 可以从索引末尾开始扫描，而不需要文件排序。

## 五、EXPLAIN ANALYZE：MySQL 8.0+ 的增强

### 5.1 为什么需要 EXPLAIN ANALYZE？

`EXPLAIN` 只展示估算的执行计划，但**估算可能与实际执行结果不符**。特别是在数据量变化、统计信息过期时，估算值可能偏差很大。

`EXPLAIN ANALYZE` 则不同——它会**实际执行 SQL**，并报告真实的运行时数据。

### 5.2 输出解读

```sql
EXPLAIN ANALYZE SELECT * FROM users WHERE age > 25 ORDER BY create_time;
```

```json
{
  "execution_time": "0.028 sec",
  "stages": [
    {
      "stage": "Sorting",
      "cost": 15.25,
      "actual_time": "0.028..0.032",
      "rows": 10,
      "loops": 1
    },
    {
      "stage": "Index lookup",
      "cost": 12.25,
      "actual_time": "0.018..0.025",
      "rows": 10,
      "loops": 1
    }
  ]
}
```

关键指标解读：

| 指标 | 含义 |
|---|---|
| `cost` | 优化器估算的代价 |
| `actual_time` | 实际耗时范围（第一次..最后一次） |
| `rows` | 实际处理的行数 |
| `loops` | 该阶段执行的次数 |

通过 `actual_time` 和 `rows`，你可以验证优化器是否做出了正确的选择。如果 `actual_time` 远高于预期，说明执行计划需要调整。

## 六、面试实战

### 6.1 场景一：type=ALL 如何优化

**面试官问**：「你的查询 EXPLAIN 后 type 是 ALL，你知道这意味着什么吗？如何优化？」

**普通回答**：全表扫描慢，加个索引就好了。

**深入回答**：

> `type = ALL` 意味着 MySQL 在做全表扫描，需要读取表的全部数据行。
>
> 优化思路有几个方向：
>
> 第一步，确认是否真的需要优化。如果表只有几百行、查询频率很低，全表扫描可能反而更快。
>
> 如果确实需要优化，先看 WHERE 条件。找出查询中用于过滤的字段，检查是否有合适的索引。如果已经有索引但没被使用，可以用 `FORCE INDEX` 强制指定，或者分析为什么不走索引——可能是统计信息过期，也可能是优化器认为全表扫描代价更低。
>
> 另一个思路是减少查询的数据量。比如分页查询，从「查全部再 LIMIT」改成「基于主键范围分页」。
>
> 最后还要考虑：是不是查询了不必要的字段？只 SELECT 需要的列，利用索引覆盖，可以减少 IO。

### 6.2 场景二：Using filesort 的原因和解决方案

**面试官问**：「EXPLAIN 结果里有 Using filesort，你知道这是什么意思吗？怎么优化？」

**普通回答**：文件排序慢，避免它就好了。

**深入回答**：

> `Using filesort` 不是「使用文件排序」的意思，而是 MySQL 内部的一种排序算法。当排序无法利用索引的有序性时，MySQL 会在内存或磁盘中排序。
>
> 常见原因有几种：
>
> 1. ORDER BY 的字段没有索引
> 2. ORDER BY 的顺序与索引顺序不一致
> 3. 多字段排序但顺序与索引不匹配
>
> 优化方案是根据查询建立合适的索引。比如 `ORDER BY create_time DESC`，如果有 `idx_create_time` 索引，就不需要文件排序了。
>
> 如果无法建立索引，就考虑限制排序的数据量。`LIMIT 100` 和 `LIMIT 100000` 的排序代价差距很大。
>
> 最后，分页查询也可以优化。改成基于主键的范围查询，用上一页的最大 ID 作为起点，避免大偏移量的排序。

### 6.3 场景三：Extra 字段的综合分析

**面试官问**：「Extra 字段有哪些常见的值？分别代表什么？」

**深入回答**：

> Extra 字段包含很多优化信息，常见的：
>
> - `Using index`：索引覆盖，不需要回表，性能最好
> - `Using index condition`：索引下推，在索引层面过滤数据
> - `Using where`：需要在存储引擎读取后进行额外过滤
> - `Using filesort`：需要文件排序，无法利用索引排序
> - `Using temporary`：需要临时表存储中间结果
> - `Using MRR`：使用多范围读优化，减少随机 IO
>
> 面试中常问的是 Using index 和 Using index condition 的区别：
>
> - Using index 表示查询的列都在索引中，完全不需要回表
> - Using index condition 表示利用索引过滤数据，但仍需回表获取其他列

## 总结

EXPLAIN 是 SQL 优化的基础工具，看懂它，你就有了优化 SQL 的方向。

核心要点回顾：

- **type 字段**：反映 MySQL 的访问方式，`system` 到 `ALL` 性能依次下降，`ALL` 是全表扫描，必须优化
- **key 字段**：实际使用的索引，为 `NULL` 时说明走全表扫描
- **Extra 字段**：
  - `Using index`：索引覆盖，性能最优
  - `Using index condition`：索引下推，减少回表
  - `Using filesort`：需要排序，性能差
  - `Using temporary`：需要临时表，性能差

会用 EXPLAIN，你就不再是「凭感觉优化」，而是「用数据说话」。这是每个 DBA 和后端工程师的必备技能。
