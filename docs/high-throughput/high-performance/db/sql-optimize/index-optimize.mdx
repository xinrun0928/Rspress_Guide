# 索引优化：索引下推、覆盖索引、前缀索引

你建了索引，查询还是慢？加了索引，EXPLAIN 依然显示全表扫描？索引没少加，效果却不好——这是很多人在索引优化上踩过的坑。

问题出在哪？**索引不只是「建了就能用」这么简单。** 建索引有讲究：用得不对，索引就是摆设；用对了，才能真正发挥威力。

这篇文章，聚焦索引优化三个核心技巧：**索引下推**、**覆盖索引**、**前缀索引**，以及面试必问的**索引失效场景**。搞懂这些，你的 SQL 优化能力会上一个台阶。

## 一、索引下推：MySQL 5.6+ 的杀手锏

### 1.1 什么是索引下推？

索引下推（Index Condition Pushdown，简称 ICP）是 MySQL 5.6 引入的优化特性。它的核心思想是：**把索引过滤条件下推到存储引擎层执行，而不是全部留给服务层。**

听起来有点抽象，我们用具体例子来说明。

### 1.2 没有索引下推时的查询流程

假设有一张用户表：

```sql
CREATE TABLE users (
    id BIGINT PRIMARY KEY,
    name VARCHAR(50),
    age INT,
    email VARCHAR(100),
    INDEX idx_age_name (age, name)
);
```

执行以下查询：

```sql
SELECT * FROM users WHERE age > 20 AND name LIKE '张%';
```

在 MySQL 5.6 之前（没有索引下推），查询流程是这样的：

1. **索引查找**：使用 `idx_age_name` 索引找到所有 age > 20 的记录的主键值
2. **回表**：根据主键值，回表获取完整的行数据
3. **服务层过滤**：在 MySQL 服务层，用 `name LIKE '张%'` 过滤数据

问题在哪？**步骤 2 回表了太多无用的数据。** 因为 name LIKE '张%' 这个条件，在索引层面就可以过滤，但 MySQL 5.6 之前没有这个能力，它只能先把 age > 20 的记录全部回表，然后在服务层再过滤。

### 1.3 有索引下推时的查询流程

有了索引下推（MySQL 5.6+），查询流程变成：

1. **索引查找 + 过滤**：使用 `idx_age_name` 索引，找到 age > 20 的记录，**同时在索引层面过滤 `name LIKE '张%'`**
2. **回表**：只对满足两个条件的记录回表

**关键区别**：索引下推在索引层面做了部分过滤，减少了不必要的回表操作。

### 1.4 EXPLAIN 中的体现

```sql
EXPLAIN SELECT * FROM users WHERE age > 20 AND name LIKE '张%';
```

```json
{
  "type": "range",
  "key": "idx_age_name",
  "Extra": "Using index condition"
}
```

看到 `Using index condition`，说明索引下推生效了。

如果查询改为只获取索引列：

```sql
EXPLAIN SELECT name FROM users WHERE age > 20 AND name LIKE '张%';
```

```json
{
  "type": "range",
  "key": "idx_age_name",
  "Extra": "Using index condition; Using index"
}
```

`Using index` 表示索引覆盖，完全不需要回表。

### 1.5 索引下推的适用条件

索引下推虽好，但不是所有场景都生效，需要满足以下条件：

| 条件 | 说明 |
|---|---|
| 使用二级索引 | 索引下推只适用于二级索引，不适用于主键索引 |
| WHERE 条件涉及索引列 | 过滤条件必须是索引列的一部分 |
| 存储引擎支持 | InnoDB 和 MyISAM 都支持 |

**不适用场景**：

- 查询条件使用了索引列的函数或表达式（如 `WHERE YEAR(create_time) = 2024`）
- 索引列参与计算（如 `WHERE age + 1 > 21`）
- 使用 OR 连接多个条件，但部分条件不在索引中

### 1.6 索引下推 vs 覆盖索引

这是面试常考的对比题：

| 特性 | 索引下推（ICP） | 覆盖索引 |
|---|---|---|
| 英文名 | Index Condition Pushdown | Covering Index |
| 目的 | **减少回表次数** | **完全避免回表** |
| Extra 显示 | `Using index condition` | `Using index` |
| 查询列要求 | 可以查询索引外的列 | 只能查询索引包含的列 |
| 优化效果 | 减少部分回表 | 零回表 |

简单理解：

- **索引下推**：在索引层面多做一些过滤，减少回表，但不是完全避免回表
- **覆盖索引**：查询的列全部在索引中，**完全不需要回表**

## 二、覆盖索引：性能最优解

### 2.1 什么是覆盖索引？

覆盖索引（Covering Index）是指：**一个索引包含了查询所需的所有列，因此不需要回表，直接从索引中返回数据。**

用 EXPLAIN 查看，会看到 Extra 显示 `Using index`，而不是 `Using index condition`。

### 2.2 覆盖索引的原理

回顾一下 MySQL 的索引结构。InnoDB 使用 B+ 树作为索引结构：

```
B+ 树索引结构
├── 叶子节点存储：索引列的值 + 主键值
└── 非叶子节点：索引列的值 + 指针
```

对于普通索引（如 `idx_age_name(age, name)`）：

- 索引叶子节点存储：(age 的值, name 的值, 主键 id)
- 查询流程：根据 age 找到索引项，获取 name 和主键 id

如果查询只需要 age、name、id 这三列，就可以**直接从索引中返回**，不需要再回表。

### 2.3 覆盖索引的优势

| 优势 | 说明 |
|---|---|
| 避免回表 | 减少一次磁盘 IO |
| 减少数据访问量 | 只读取索引文件，不访问数据文件 |
| 索引有序 | 适合范围查询和排序 |

**性能差异有多大？** 对于大表，磁盘 IO 是主要瓶颈。覆盖索引可以减少 50% 甚至更多的查询时间。

### 2.4 覆盖索引的设计原则

**原则一：经常一起查询的列放在一起**

```sql
-- 用户表常见查询：按年龄范围查询用户名和邮箱
SELECT name, email FROM users WHERE age BETWEEN 18 AND 30;

-- 覆盖索引设计
CREATE INDEX idx_age_name_email ON users(age, name, email);
```

**原则二：高频查询优先考虑覆盖索引**

如果某个查询执行频率很高，优化收益就越大。

```sql
-- 高频查询：获取用户列表（分页场景）
SELECT id, name, email FROM users WHERE status = 1 ORDER BY create_time DESC LIMIT 20;

-- 覆盖索引设计
CREATE INDEX idx_status_create_id ON users(status, create_time, id, name, email);
```

### 2.5 覆盖索引的局限性

**局限性一：索引列数不能太多**

索引列太多会导致索引变大，影响写入性能和存储空间。一般建议索引列不超过 3-4 列。

**局限性二：不是所有查询都能用覆盖索引**

如果查询的列不在索引中，就需要回表。

```sql
-- 查询 mobile 列，但索引中没有它
CREATE INDEX idx_age_name ON users(age, name);

SELECT name, mobile FROM users WHERE age = 25;
-- 需要回表获取 mobile
```

**局限性三：索引维护有代价**

每次 INSERT、UPDATE、DELETE，索引都需要维护。覆盖索引列越多，维护成本越高。

## 三、前缀索引：优化大字段查询

### 3.1 什么时候需要前缀索引？

如果一个字段很长（如 VARCHAR(255) 甚至 TEXT），直接建索引会：

- 占用大量存储空间
- 索引树变高，查询变慢
- 写入性能下降

**前缀索引**允许你只索引字段值的前 N 个字符，而不是全部。

```sql
-- 普通索引：索引整个 email 字段
CREATE INDEX idx_email ON users(email);

-- 前缀索引：只索引 email 前 20 个字符
CREATE INDEX idx_email_prefix ON users(email(20));
```

### 3.2 如何选择前缀长度？

选择前缀长度有两个原则：

**原则一：前缀的基数要足够大**

前缀应该能够区分大部分记录。如果前缀区分度太低，索引效果会很差。

```sql
-- 查看不同前缀长度的基数（唯一值数量）
SELECT
    COUNT(DISTINCT LEFT(email, 10)) AS prefix_10,
    COUNT(DISTINCT LEFT(email, 20)) AS prefix_20,
    COUNT(DISTINCT LEFT(email, 30)) AS prefix_30,
    COUNT(DISTINCT email) AS total
FROM users;
```

| 前缀长度 | 基数 | 区分度 |
|---|---|---|
| 10 | 8000 | 80% |
| 20 | 9800 | 98% |
| 30 | 9950 | 99.5% |
| 全部 | 10000 | 100% |

**原则二：前缀长度要覆盖常见查询**

如果大部分查询是精确匹配 email，要确保前缀长度能唯一确定大部分记录。

```sql
-- 常用查询
SELECT * FROM users WHERE email = 'user@example.com';

-- 前缀长度至少要能唯一标识记录
```

### 3.3 前缀索引的 SQL 示例

```sql
-- 选择合适的前缀长度
CREATE INDEX idx_email_prefix ON users(email(20));

-- 或者用 MySQL 的函数计算最优长度
SELECT
    LENGTH(email) AS total_len,
    COUNT(DISTINCT LEFT(email, 10)) / COUNT(*) AS ratio_10,
    COUNT(DISTINCT LEFT(email, 20)) / COUNT(*) AS ratio_20,
    COUNT(DISTINCT LEFT(email, 30)) / COUNT(*) AS ratio_30
FROM users;
```

### 3.4 前缀索引的局限性

:::warning 前缀索引有三个重要限制

**限制一：不支持 ORDER BY 和 GROUP BY**

```sql
-- 无法利用前缀索引排序
CREATE INDEX idx_email ON users(email(20));

SELECT * FROM users ORDER BY email;
-- Using filesort，无法使用前缀索引
```

因为前缀索引只索引了 email 的前 20 个字符，索引中存储的不是完整的 email 值，无法保证全字段排序的正确性。

**限制二：不支持索引覆盖**

```sql
-- 查询 email 列，但用的是前缀索引，无法索引覆盖
SELECT email FROM users WHERE email LIKE 'user%';
-- Extra 显示 Using index condition; Using where，需要回表
```

**限制三：最左前缀原则的部分失效**

如果查询条件的左侧部分与前缀不匹配，前缀索引可能无法使用。

```sql
-- 前缀索引 idx_email(20)
-- 假设 email 格式是 userid@example.com

-- 可以使用索引（前缀匹配）
SELECT * FROM users WHERE email LIKE 'user%';

-- 无法使用索引（前缀是 middle，右侧匹配）
SELECT * FROM users WHERE email LIKE '%example%';
```

## 四、索引失效场景：面试必问

### 4.1 索引失效的本质

索引失效不是索引「坏了」，而是 MySQL 优化器认为使用索引不会比全表扫描更快。理解这一点，你就知道怎么应对了。

### 4.2 六大索引失效场景

**场景一：隐式类型转换**

```sql
-- phone 字段是 VARCHAR 类型
CREATE INDEX idx_phone ON users(phone);

-- 失效：数字字符串与数字比较，触发隐式转换
EXPLAIN SELECT * FROM users WHERE phone = 13800138000;

-- 生效：使用字符串类型
EXPLAIN SELECT * FROM users WHERE phone = '13800138000';
```

**原理**：MySQL 会把字符串转换成数字进行比较，导致索引列上的比较无法使用索引。

**场景二：最左前缀原则被破坏**

```sql
-- 索引 idx_name_age (name, age)

-- 生效：使用索引的最左前缀
EXPLAIN SELECT * FROM users WHERE name = '张三';

-- 生效：跳过 name，直接用 age（不推荐，索引部分失效）
EXPLAIN SELECT * FROM users WHERE age = 25;

-- 失效：跳过 name，用中间的 age
-- 如果索引是 (name, age, city)，跳过 name 和 age，直接用 city
EXPLAIN SELECT * FROM users WHERE city = '北京';
```

**场景三：LIKE 开头使用通配符**

```sql
-- 索引 idx_name (name)

-- 生效：LIKE 'xxx%' 可以使用索引（前缀匹配）
EXPLAIN SELECT * FROM users WHERE name LIKE '张%';

-- 失效：LIKE '%xxx%' 无法使用索引（中间或后缀匹配）
EXPLAIN SELECT * FROM users WHERE name LIKE '%三%';

-- 失效：LIKE '%xxx' 无法使用索引（后缀匹配）
EXPLAIN SELECT * FROM users WHERE name LIKE '%张三';
```

**场景四：OR 条件中存在非索引列**

```sql
-- 索引 idx_age (age)

-- 失效：OR 条件中有非索引列
EXPLAIN SELECT * FROM users WHERE age = 25 OR name = '张三';

-- 生效：全部使用索引列
EXPLAIN SELECT * FROM users WHERE age = 25 OR age = 30;
```

**场景五：使用函数或表达式**

```sql
-- 索引 idx_create_time (create_time)

-- 失效：对索引列使用函数
EXPLAIN SELECT * FROM users WHERE YEAR(create_time) = 2024;

-- 失效：对索引列使用表达式
EXPLAIN SELECT * FROM users WHERE id + 1 = 100;

-- 生效：函数在比较值的一侧
EXPLAIN SELECT * FROM users WHERE create_time = DATE '2024-01-01';
```

**场景六：NOT NULL 和 NOT EXISTS**

```sql
-- 索引 idx_status (status)

-- 可能失效：IS NOT NULL
EXPLAIN SELECT * FROM users WHERE status IS NOT NULL;

-- 可能失效：NOT IN / NOT EXISTS
EXPLAIN SELECT * FROM users WHERE status NOT IN ('active', 'pending');
```

### 4.3 索引失效的通用排查思路

遇到索引失效，先用 EXPLAIN 确认：

1. **看 type 字段**：如果是 `ALL`，大概率是全表扫描
2. **看 key 字段**：如果是 `NULL`，说明没有使用索引
3. **看 Extra 字段**：结合字段判断失效原因

**排查清单**：

- [ ] 查询条件是否涉及隐式类型转换
- [ ] 是否破坏了最左前缀原则
- [ ] LIKE 是否以通配符开头
- [ ] OR 条件是否包含非索引列
- [ ] 是否对索引列使用了函数或表达式
- [ ] 是否使用了 NOT IN / NOT EXISTS

## 五、联合索引设计原则

### 5.1 联合索引的结构

联合索引是一棵 B+ 树，索引键值按创建顺序排列：

```sql
CREATE INDEX idx_age_name ON users(age, name);
```

索引树按 `(age, name)` 排序：

```
(18, '张三') < (18, '李四') < (20, '王五') < (25, '赵六')
```

### 5.2 联合索引设计三原则

**原则一：把查询频率高的列放在前面**

```sql
-- 高频查询：按年龄查用户名
SELECT name FROM users WHERE age = 25;

-- 索引设计：age 在前
CREATE INDEX idx_age_name ON users(age, name);
```

**原则二：考虑 ORDER BY 和 GROUP BY 的字段**

如果查询有排序或分组，索引列顺序要与 ORDER BY / GROUP BY 一致：

```sql
-- 查询：按状态分组，统计数量
SELECT status, COUNT(*) FROM orders GROUP BY status;

-- 索引设计：status 在前
CREATE INDEX idx_status ON orders(status);
```

**原则三：覆盖更多查询条件**

如果一个索引能覆盖多个查询，就是最优设计：

```sql
-- 常见查询
SELECT * FROM users WHERE age = 25 ORDER BY name;
SELECT * FROM users WHERE name LIKE '张%';
SELECT * FROM users WHERE age BETWEEN 18 AND 30;

-- 联合索引设计：age 和 name 能覆盖上述查询
CREATE INDEX idx_age_name ON users(age, name);
```

### 5.3 联合索引 vs 多个单列索引

| 场景 | 推荐 | 原因 |
|---|---|---|
| 查询条件固定 | 联合索引 | 一个索引覆盖多列，减少索引数量 |
| 查询条件变化大 | 单列索引 | 联合索引的列顺序限制了灵活性 |
| 写入频繁 | 单列索引 | 联合索引维护成本更高 |

## 六、面试实战

### 6.1 场景一：覆盖索引和索引下推的区别

**面试官问**：「覆盖索引和索引下推有什么区别？」

**深入回答**：

> 覆盖索引和索引下推都是为了减少回表，但优化点不同。
>
> 覆盖索引是**索引包含查询所需的全部列**，从索引中直接返回数据，完全不需要回表，EXPLAIN 显示 `Using index`。
>
> 索引下推是**在索引层面先过滤一部分数据**，减少回表次数，但最终还是要回表获取其他列，EXPLAIN 显示 `Using index condition`。
>
> 简单说：覆盖索引是完全不需要回表，索引下推是减少回表次数。两者可以结合使用。

### 6.2 场景二：索引失效的综合分析

**面试官问**：「什么情况下索引会失效？」

**深入回答**：

> 索引失效的场景主要有六种：
>
> 第一，隐式类型转换。如果字段是 VARCHAR 类型，查询时用了数字，MySQL 会做类型转换，导致索引失效。
>
> 第二，破坏了最左前缀原则。联合索引必须从最左边的列开始使用。
>
> 第三，LIKE 开头使用通配符。`LIKE '%xxx'` 无法使用索引，因为不知道前缀是什么。
>
> 第四，OR 条件中有非索引列。MySQL 只能选择全表扫描。
>
> 第五，对索引列使用函数或表达式。索引列参与了计算，索引无法使用。
>
> 第六，使用 NOT IN / NOT EXISTS。优化器认为全表扫描更快。
>
> 遇到索引失效，先用 EXPLAIN 看执行计划，分析 type、key、Extra 字段，找出原因。

### 6.3 场景三：联合索引的设计思路

**面试官问**：「如何设计一个联合索引？」

**深入回答**：

> 设计联合索引要考虑三个原则。
>
> 第一，查询频率。把查询最频繁的条件列放在前面。比如 `WHERE age = 25 AND name = '张三'`，如果 age 过滤掉 90% 的数据，name 只是锦上添花，那就把 age 放前面。
>
> 第二，排序需求。如果查询有 `ORDER BY`，索引列顺序要与排序顺序一致，否则无法利用索引排序。
>
> 第三，覆盖更多查询。尽量让一个索引能覆盖多个查询，减少索引数量。比如 `(age, name)` 可以覆盖 `WHERE age = ?`、`WHERE age = ? AND name = ?`、`ORDER BY age`。

## 总结

索引优化是 SQL 优化的核心，这篇文章讲了三个关键技巧：

**索引下推**：在索引层面提前过滤数据，减少回表次数，EXPLAIN 显示 `Using index condition`。

**覆盖索引**：索引包含查询所需的所有列，完全不需要回表，EXPLAIN 显示 `Using index`。

**前缀索引**：对长字段建索引时，只索引前缀部分，节省空间但有限制——不支持排序和索引覆盖。

**索引失效六大场景**：隐式类型转换、最左前缀原则被破坏、LIKE 开头、OR 非索引列、使用函数/表达式、NOT IN/NOT EXISTS。

索引优化的本质是：**让查询尽可能走索引，让索引尽可能覆盖查询。** 理解这一点，你就能应对各种复杂的索引优化场景。
