# 分页查询优化：延迟关联与游标分页

你一定遇到过这样的场景：用户在商品列表页疯狂往下滑，越滑越慢，最后直接卡死。后台日志里，清一色的慢查询，全是这种 SQL——

```sql
SELECT * FROM orders
ORDER BY create_time DESC
LIMIT 100000, 20;
```

OFFSET 10 万，返回 20 条数据。你知道 MySQL 执行这条 SQL 做了什么吗？**它扫描了 100020 行，然后丢弃掉前 10 万行，只返回最后 20 行。**

这就是深度分页问题——OFFSET 越大，MySQL 浪费的无用功越多。当 OFFSET 是 100 万时，你可能扫描了 100 万零 20 行，只为了返回 20 条数据。这不是优化，这是资源浪费。

今天这篇文章，帮你彻底解决这个问题。

## 一、深度分页的根源：OFFSET 的代价

### 1.1 MySQL 分页的执行过程

MySQL 的分页语法 `LIMIT offset, count` 看起来很简洁，但背后的执行逻辑藏着性能陷阱。

```sql
-- 查第 100001-100020 条数据
SELECT * FROM orders
WHERE status = 1
ORDER BY create_time DESC
LIMIT 100000, 20;
```

当你执行这条 SQL 时，MySQL 的执行过程是这样的：

1. **全表扫描**：从索引或主键索引的第一行开始，逐行扫描
2. **条件过滤**：对每一行判断 `status = 1`
3. **排序**：将符合条件的所有数据按 `create_time` 排序
4. **跳过 OFFSET**：跳过前 10 万行
5. **返回 LIMIT**：返回接下来的 20 行

问题出在第 4 步。MySQL 不是「直接跳到第 10 万行」，而是「从第 1 行开始数，数到第 10 万行才开始返回」。这意味着，如果你的 `orders` 表有 500 万条数据，MySQL 可能扫描了 500 万行，只为了找到符合 `status = 1` 的那 10 万零 20 行。

:::tip OFFSET 的时间复杂度

`LIMIT offset, count` 的时间复杂度可以近似为 **O(offset + count)**。

- `offset = 0, count = 20` → 扫描约 20 行
- `offset = 100000, count = 20` → 扫描约 100020 行
- `offset = 1000000, count = 20` → 扫描约 1000020 行

OFFSET 越大，扫描的行数越多，耗时就越长。而且这个耗时是**线性增长**的，没有优化手段会越来越慢。

:::

### 1.2 用 EXPLAIN 看清分页的成本

光说理论不够直观，我们用 EXPLAIN 来看一个实际案例。

假设有一张 `orders` 表，500 万条数据，需要查第 10 万页（每页 20 条）：

```sql
EXPLAIN SELECT * FROM orders
WHERE status = 1
ORDER BY create_time DESC
LIMIT 1000000, 20;
```

```
+----+-------------+--------+------------+------+---------------+------+---------+--------+---------+----------+----------------+
| id | select_type | table  | type       | key  | rows          | rws  | rlen    | Extra                      |
+----+-------------+--------+------------+------+---------------+------+---------+--------+---------+----------+----------------+
|  1 | SIMPLE      | orders | ALL        | NULL | 5,000,000     | 5M   | 1280M   | Using where; Using filesort |
+----+-------------+--------+------------+------+---------------+------+---------+--------+---------+----------+----------------+
```

几个关键信息：

- `type: ALL`：全表扫描，没有利用任何索引
- `rows: 5,000,000`：MySQL 预估要扫描 500 万行
- `Using filesort`：无法利用索引排序，需要在内存/磁盘中排序

这就是深度分页慢的根本原因——**MySQL 为了找到第 100 万零 20 条数据，不得不扫描前面的所有数据**。

## 二、延迟关联：减少回表的代价

### 2.1 核心思路

延迟关联的优化思路很巧妙：**不要一开始就查所有字段，而是先用索引查出「主键 ID」，再用主键 ID 去查具体数据。**

```sql
-- 优化前：直接查所有字段
SELECT * FROM orders
WHERE status = 1
ORDER BY create_time DESC
LIMIT 100000, 20;

-- 优化后：延迟关联
SELECT o.* FROM orders o
INNER JOIN (
    SELECT id FROM orders
    WHERE status = 1
    ORDER BY create_time DESC
    LIMIT 100000, 20
) t ON o.id = t.id;
```

为什么这样快？我们拆解一下执行过程：

**第一步**：子查询只查询 `id` 字段，利用 `status + create_time` 的联合索引，直接定位到第 10 万条数据的位置，不需要回表。

**第二步**：子查询返回 20 个 ID，INNER JOIN 再根据这 20 个 ID 查具体数据。虽然这一步有 20 次回表，但回表次数从 100 万次降到了 20 次。

:::tip 延迟关联的适用条件

1. 分页字段（`ORDER BY create_time`）和过滤字段（`WHERE status = 1`）都在索引中
2. OFFSET 越大，优化效果越明显
3. 需要有合适的联合索引：`idx_status_create_time(status, create_time)`

如果缺少索引，延迟关联的效果会大打折扣，因为子查询依然要扫描大量数据。

:::

### 2.2 延迟关联的 EXPLAIN 分析

我们来看优化后的执行计划：

```sql
EXPLAIN SELECT o.* FROM orders o
INNER JOIN (
    SELECT id FROM orders
    WHERE status = 1
    ORDER BY create_time DESC
    LIMIT 100000, 20
) t ON o.id = t.id;
```

```
+----+-------------+--------------------+------------+-------+---------------------------+----------------+---------+--------+---------+----------+-------------+
| id | select_type | table              | type       | key   | key_len                   | ref            | rows    | rws    | rlen    | Extra             |
+----+-------------+--------------------+------------+-------+---------------------------+----------------+---------+--------+---------+----------+-------------+
|  1 | PRIMARY     | <derived2>         | ALL        | NULL  | NULL                      | NULL           | 100020  | 2000K  | 0       | NULL               |
|  1 | PRIMARY     | o                  | eq_ref     | PRIMARY| 8                         | t.id           |       1 |  20    | 2560    | NULL               |
|  2 | DERIVED     | orders             | index      | idx_status_create_time    | 9               | NULL    | 100020 | 51200K  | Using where; Backward index scan |
+----+-------------+--------------------+------------+-------+---------------------------+----------------+---------+--------+---------+----------+-------------+
```

关键变化：

- **子查询（DERIVED）**：`type: index`，利用了 `idx_status_create_time` 索引，不需要全表扫描
- **子查询（DERIVED）**：`Extra: Using where; Backward index scan`，说明利用了索引的有序性，直接从第 10 万条开始扫描
- **回表次数**：从原来的 100 万次降到了 20 次（eq_ref 走主键索引）

### 2.3 延迟关联 + 覆盖索引：双剑合璧

如果你的联合索引包含了所有需要查询的字段，那连回表都不需要了：

```sql
-- 创建一个包含所有字段的覆盖索引
ALTER TABLE orders ADD INDEX idx_cover (
    status, create_time, id, order_no, amount, customer_id
);
```

```sql
-- 优化后：子查询直接返回所有字段，不需要回表
SELECT * FROM orders
WHERE status = 1
ORDER BY create_time DESC
LIMIT 100000, 20;
```

但这种方式有个限制：**索引要包含 SELECT 的所有字段**。如果 `SELECT *` 包含了大量字段，索引会变得很大，维护成本上升。实际工作中，更常见的是延迟关联 + 必要的回表。

## 三、游标分页：彻底摆脱 OFFSET

### 3.1 OFFSET 的本质问题

延迟关联优化了回表次数，但 OFFSET 本身的问题没有解决——**不管你怎么优化，OFFSET 100 万还是要从第 1 条开始数到第 100 万条**。

游标分页（keyset pagination）换了一个思路：**不数数，而是「记住位置」**。

### 3.2 游标分页的原理

传统分页是「数到第 N 条开始返回」，游标分页是「从第 N 条的后面开始返回」。

```sql
-- 优化前：OFFSET 分页
SELECT * FROM orders
WHERE status = 1
ORDER BY create_time DESC
LIMIT 100000, 20;

-- 优化后：游标分页
-- 假设上一页最后一条数据的 create_time = 2024-01-15 10:30:00，id = 99999
SELECT * FROM orders
WHERE status = 1
  AND (create_time < '2024-01-15 10:30:00'
       OR (create_time = '2024-01-15 10:30:00' AND id < 99999))
ORDER BY create_time DESC, id DESC
LIMIT 20;
```

游标分页的核心思想：

1. **记住上一页的位置**：上一页返回的数据包含 `create_time` 和 `id`
2. **用条件跳过已读数据**：用 `WHERE` 条件过滤掉已读数据
3. **不依赖 OFFSET**：每次查询都是「从当前位置往后取 N 条」，与总数据量无关

### 3.3 游标分页 vs OFFSET 分页

| 对比项 | OFFSET 分页 | 游标分页 |
|-------|------------|---------|
| 第一页 | 快速 | 快速 |
| 第十页 | 变慢 | 快速 |
| 第一百万页 | 极慢 | 快速（与查第一页一样快） |
| 跳页 | 支持 | 不支持 |
| 实现复杂度 | 低 | 中 |
| 用户体验 | 可跳页 | 只能顺序翻页 |

游标分页最大的优势是：**查询时间与 OFFSET 无关，始终是 O(N)，其中 N 是 LIMIT 的大小**。

### 3.4 游标分页的适用场景

游标分页不是银弹，它有自己的适用场景。

**适合用游标分页的场景：**

- 无限滚动的信息流（微博、抖音评论、商品列表）
- 实时性要求高的数据（排行榜、实时订单）
- 不需要跳页的交互模式

**不适合用游标分页的场景：**

- 用户需要跳页（直接跳到第 50 页）
- 需要随机访问数据（选择任意页码）
- 搜索结果的分页（用户改了搜索词，需要重新开始）

:::tip 游标分页的实现细节

实际生产环境中，游标分页的游标通常用**上一页最后一条记录的唯一标识**（如主键 ID、创建时间等）。

```json
// 前端请求
{
    "pageSize": 20,
    "cursor": "eyJjc3RyIjogIjIwMjQtMDEtMTUgMTA6MzA6MDAiLCAiaWQiOiA5OTk5OX0="
}

// 后端解析 cursor（通常是 Base64 编码的 JSON）
{
    "create_time": "2024-01-15 10:30:00",
    "id": 99999
}
```

游标需要进行加密或签名，防止用户篡改。

:::

## 四、三种分页方案对比

| 方案 | 原理 | OFFSET 100 万耗时 | 适用场景 | 实现复杂度 |
|-----|------|------------------|---------|-----------|
| 直接分页 | `LIMIT offset, count` | 极慢 | 数据量小、OFFSET 不大 | 低 |
| 延迟关联 | 子查询获取 ID，再回表 | 较快 | 中等深度分页、需要回表 | 中 |
| 游标分页 | 记住位置，用 WHERE 跳过 | 最快（恒定） | 无限滚动、不需跳页 | 中 |
| 延迟关联 + 覆盖索引 | 子查询 + 覆盖索引不回表 | 最快 | 深度分页且字段少 | 中 |

## 五、生产环境分页优化清单

### 5.1 分页优化的 checklist

实际工作中，分页优化可以按以下步骤来：

1. **先看 EXPLAIN**：确认是否有全表扫描、`Using filesort`
2. **检查索引**：
   - `WHERE` 条件字段是否有索引？
   - `ORDER BY` 字段是否有索引？
   - 是否能建立联合索引 `(WHERE字段, ORDER BY字段)`？
3. **估算成本**：OFFSET 100 万意味着要扫描 100 万行，这个代价你能接受吗？
4. **选择方案**：
   - 小 OFFSET（`<` 1 万）：直接分页或延迟关联
   - 大 OFFSET（`>=` 1 万）：延迟关联 + 覆盖索引
   - 无限滚动：不需跳页用游标分页
5. **监控慢查询**：上线后持续监控分页 SQL 的执行时间

### 5.2 分页优化的代码示例

以下是几种常见语言/框架中分页优化的实现示例。

```java
// Java + MyBatis-Plus：延迟关联实现
@Service
public class OrderService {

    @Autowired
    private OrderMapper orderMapper;

    // 延迟关联分页
    public Page<Order> selectByDeferredJoin(Long lastId, Integer pageSize) {
        return orderMapper.deferredJoinPage(lastId, pageSize);
    }

    // 游标分页
    public Page<Order> selectByCursor(LocalDateTime lastTime, Long lastId, Integer pageSize) {
        return orderMapper.cursorPage(lastTime, lastId, pageSize);
    }
}
```

```xml
<!-- MyBatis XML：延迟关联查询 -->
<select id="deferredJoinPage" resultType="Order">
    SELECT o.* FROM orders o
    INNER JOIN (
        SELECT id FROM orders
        WHERE status = 1
        <if test="lastId != null">
            AND id &lt; #{lastId}
        </if>
        ORDER BY id DESC
        LIMIT #{pageSize}
    ) t ON o.id = t.id
    ORDER BY o.id DESC
</select>

<!-- MyBatis XML：游标分页 -->
<select id="cursorPage" resultType="Order">
    SELECT * FROM orders
    WHERE status = 1
    <if test="lastTime != null and lastId != null">
        AND (create_time &lt; #{lastTime}
             OR (create_time = #{lastTime} AND id &lt; #{lastId}))
    </if>
    ORDER BY create_time DESC, id DESC
    LIMIT #{pageSize}
</select>
```

## 六、面试实战

### 6.1 场景一：深度分页优化思路

**面试官问**：「你的系统有一张订单表，数据量 5000 万。用户反馈翻到第 100 页之后非常慢，你如何优化？」

**普通回答**：我会用 LIMIT 分页，加个索引。

**深入回答**：

> 这个问题有两个层次：索引层和分页策略层。
>
> **第一层：索引优化。** 先用 EXPLAIN 看执行计划，确认是否走了索引。如果 `ORDER BY` 和 `WHERE` 字段没有联合索引，需要建一个 `(status, create_time)` 的联合索引。这样 ORDER BY 可以直接利用索引的有序性，避免 filesort。
>
> **第二层：分页策略优化。** 如果是深度分页（OFFSET 很大），有两个方向：
>
> - **延迟关联**：先子查询用索引查出主键 ID，再根据 ID 查具体字段。这样可以把回表次数从 OFFSET 次降到 LIMIT 次。
>
> - **游标分页**：如果业务允许不跳页，可以用游标分页。每次返回上一页最后一条记录的 `id` 和 `create_time`，下一页用 `WHERE id &lt; #{lastId}` 来定位。这种方案的查询时间是恒定的，与 OFFSET 无关。
>
> 最终选哪个方案，要看业务场景——如果用户需要跳页，用延迟关联；如果用户只是无限滚动，用游标分页。

### 6.2 场景二：游标分页 vs OFFSET 分页

**面试官问**：「游标分页和传统的 OFFSET 分页有什么区别？各自适用什么场景？」

**普通回答**：游标分页比 OFFSET 分页快。

**深入回答**：

> 两者的核心区别在于**定位方式不同**：
>
> - OFFSET 分页是「数数」：MySQL 从第 1 行开始数，数到第 N 行开始返回。时间复杂度 O(offset + count)。
> - 游标分页是「定位」：MySQL 直接跳到游标位置开始返回。时间复杂度 O(count)。
>
> **OFFSET 分页的适用场景**：数据量不大、用户需要跳页（比如用户想直接看第 50 页）。
>
> **游标分页的适用场景**：无限滚动（微博、抖音）、实时数据（排行榜）、不需要跳页的交互。
>
> **选择建议**：如果业务允许「只能顺序翻页」，游标分页是更好的选择；如果必须支持跳页，那就在大 OFFSET 时加防护（比如限制最大 OFFSET 超过 1 万就返回空），或者用延迟关联优化。

## 总结

分页查询的优化，本质上是解决「如何高效定位第 N 条数据」的问题。

- **延迟关联**：用索引查出 ID，再根据 ID 回表。适合中等深度的分页，减少回表次数。
- **游标分页**：记住位置，用 WHERE 条件跳过已读数据。适合无限滚动场景，查询时间恒定。
- **覆盖索引**：如果索引能覆盖所有查询字段，连回表都省了。

三种方案不是非此即彼，而是要根据业务场景灵活选择。关键是记住一点：**不要让 MySQL 做「数数」这种无用功**。
