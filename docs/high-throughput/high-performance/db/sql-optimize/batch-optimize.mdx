# 批量插入与数据更新优化

你接手了一个数据迁移项目，要把历史订单从老系统导入到新系统。订单总量 500 万条，老同事留下的脚本是这样的：

```java
for (Order order : orders) {
    String sql = "INSERT INTO orders (id, user_id, amount, create_time) " +
                 "VALUES (" + order.getId() + ", " + order.getUserId() + ", " +
                 order.getAmount() + ", '" + order.getCreateTime() + "')";
    jdbcTemplate.execute(sql);
}
```

跑了一晚上，导入了 10 万条。

500 万条数据，按这个速度需要 50 个晚上。这还是在数据库没有被拖垮的前提下。

问题出在哪？**逐条插入，每次都要走一遍连接、解析、执行、关闭的流程。** 这是数据库操作中最经典的性能杀手。

今天聊两个核心场景：**批量插入**和**批量更新**，以及背后的优化思路。

---

## 一、批量插入：让数据库少跑几趟

### 1.1 INSERT VALUES 合并

最简单的方式：把多条 INSERT 合并成一条。

**低效写法**：循环插入 1000 条数据，执行 1000 次 SQL。

```java
for (Product product : products) {
    String sql = "INSERT INTO products (id, name, price) VALUES (?, ?, ?)";
    jdbcTemplate.update(sql, product.getId(), product.getName(), product.getPrice());
}
```

**高效写法**：一条 SQL，插入 1000 条数据，只执行 1 次。

```sql
INSERT INTO products (id, name, price) VALUES
(1, 'Product A', 99.99),
(2, 'Product B', 199.99),
(3, 'Product C', 299.99),
-- ... 一次插入更多行
(1000, 'Product XXX', 999.99);
```

```java
StringBuilder sql = new StringBuilder("INSERT INTO products (id, name, price) VALUES ");
List<Object[]> params = new ArrayList<>();

for (int i = 0; i < products.size(); i++) {
    sql.append("(?, ?, ?)");
    if (i < products.size() - 1) {
        sql.append(", ");
    }
    Product p = products.get(i);
    params.add(new Object[]{p.getId(), p.getName(), p.getPrice()});
}

jdbcTemplate.batchUpdate(sql.toString(), params);
```

| 方式 | SQL 执行次数 | 网络往返次数 | 耗时 |
|---|---|---|---|
| 逐条插入 1000 条 | 1000 | 1000 | 约 10 秒 |
| 合并插入 1000 条 | 1 | 1 | 约 100ms |

差距是 **100 倍**。

:::tip 合并INSERT 的限制

MySQL 对单条 SQL 的长度有限制（`max_allowed_packet`，默认 64MB）。当插入数据量超过这个限制时，需要分批处理。

经验值：每批 1000-5000 条是一个比较安全的范围。

:::

### 1.2 LOAD DATA INFILE：数据库级的高速导入

如果 INSERT VALUES 是「汽车」，那 `LOAD DATA INFILE` 就是「高铁」。

这是 MySQL 服务端提供的高速导入工具，原理是：**直接读取服务器上的文件，跳过 SQL 解析和语义分析，直接写入数据页**。

```sql
LOAD DATA INFILE '/var/lib/mysql-files/products.csv'
INTO TABLE products
FIELDS TERMINATED BY ','
ENCLOSED BY '"'
LINES TERMINATED BY '\n'
(id, name, price);
```

性能对比（导入 100 万条数据）：

| 方式 | 耗时 |
|---|---|
| 逐条 INSERT | 约 30 分钟 |
| 合并 INSERT（每批 1000 条） | 约 3 分钟 |
| LOAD DATA INFILE | 约 10-30 秒 |

|LOAD DATA INFILE| 比普通 INSERT 快 **10-30 倍**，在大数据量导入场景下是必选项。

### 1.3 LOAD DATA INFILE 的注意事项

**第一：文件位置必须在 MySQL 服务端。**

MySQL 读取文件时有两个选项：

```sql
-- 从 MySQL 服务端读取（需要文件在服务器上）
LOAD DATA INFILE '/var/lib/mysql-files/products.csv' INTO TABLE products;

-- 从客户端读取后发送到服务端
LOAD DATA LOCAL INFILE '/local/path/products.csv' INTO TABLE products;
```

`LOAD DATA LOCAL INFILE` 需要服务端开启 `local_infile=on`。

**第二：确保目录权限。**

MySQL 对 `LOAD DATA INFILE` 的文件路径有安全限制，默认只能读取特定目录：

```sql
SHOW VARIABLES LIKE 'secure_file_priv';
```

```
+------------------+-----------------------+
| Variable_name    | Value                 |
+------------------+-----------------------+
| secure_file_priv | /var/lib/mysql-files/ |
+------------------+-----------------------+
```

文件必须放在这个目录下，或者把 `secure_file_priv` 设为空（不推荐，生产环境有安全风险）。

**第三：索引和约束的影响。**

导入前**禁用索引和约束**，导入完成后再开启：

```sql
-- 禁用唯一检查和外键检查
SET UNIQUE_CHECKS = 0;
SET FOREIGN_KEY_CHECKS = 0;

-- 禁用自动提交，提高导入速度
SET AUTOCOMMIT = 0;

-- 执行导入
LOAD DATA INFILE '/var/lib/mysql-files/products.csv' INTO TABLE products;

-- 重新启用
SET AUTOCOMMIT = 1;
SET UNIQUE_CHECKS = 1;
SET FOREIGN_KEY_CHECKS = 1;
```

这样可以再提升 20-30% 的导入速度。

## 二、批量更新：分而治之的艺术

### 2.1 UPDATE ... WHERE IN 的分批处理

假设要给 10 万个用户更新状态：

```sql
-- 低效：一次更新 10 万条
UPDATE users SET status = 'vip' WHERE id IN (1, 2, 3, ..., 100000);
```

问题：这条 SQL 会锁住 10 万行记录，对其他业务造成严重影响。执行时间可能超过 30 秒，期间数据库几乎是不可用的。

**正确做法：分批处理**。

```java
public void updateUsersVip(List<Long> userIds) {
    // 每批 1000 条
    int batchSize = 1000;
    int total = userIds.size();

    for (int i = 0; i < total; i += batchSize) {
        int end = Math.min(i + batchSize, total);
        List<Long> batchIds = userIds.subList(i, end);

        String sql = "UPDATE users SET status = 'vip', update_time = NOW() WHERE id IN (?)";
        jdbcTemplate.batchUpdate(sql, batchIds);

        // 每批之间休息一下，让数据库喘口气
        if (end < total) {
            try {
                Thread.sleep(10);
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        }
    }
}
```

| 方式 | 单次锁行数 | 锁表时间 | 对业务影响 |
|---|---|---|---|
| 一次更新 10 万条 | 10 万 | 30 秒+ | 严重影响 |
| 分 100 批，每批 1000 条 | 1000 | < 1 秒 | 可忽略 |

### 2.2 多值 UPDATE 的妙用

如果需要根据不同条件更新不同值，可以使用 CASE WHEN：

```sql
-- 低效：执行 3 次 UPDATE
UPDATE users SET level = 'gold' WHERE total_amount >= 10000;
UPDATE users SET level = 'silver' WHERE total_amount >= 5000 AND total_amount < 10000;
UPDATE users SET level = 'bronze' WHERE total_amount < 5000;

-- 高效：一次搞定
UPDATE users SET level = CASE
    WHEN total_amount >= 10000 THEN 'gold'
    WHEN total_amount >= 5000 THEN 'silver'
    ELSE 'bronze'
END,
update_time = NOW();
```

这样只需要一次磁盘写入，事务日志也只需要写一次。

### 2.3 避免批量更新的坑

**坑一：长事务。**

批量更新如果在一个事务里，会导致事务时间过长：

- 占用数据库连接时间过长
- 事务日志占用大量磁盘空间
- 主从复制延迟

**正确做法**：每批单独提交，不要把所有批次放在一个事务里。

**坑二：主从复制延迟。**

在 MySQL 主从架构下，大批量更新会导致主从延迟：

```
主库：执行 UPDATE（30 秒）
从库：同步 binlog，执行 UPDATE（30 秒）
```

这段时间内，从库的数据是滞后的。如果业务有读从库的需求，用户可能看到「幻读」。

**解决方案**：

1. 选择业务低峰期执行大批量操作
2. 使用 `SHOW SLAVE STATUS` 监控延迟
3. 如果延迟超过阈值，暂停批量操作，等待同步

## 三、事务控制：节奏与边界

### 3.1 批量操作的事务策略

批量操作的核心矛盾是：**事务越大，性能越好；但事务越大，风险也越大。**

| 事务大小 | 优势 | 风险 |
|---|---|---|
| 小事务（每批 100 条） | 失败影响范围小，回滚快 | 频繁提交，整体性能差 |
| 大事务（每批 10 万条） | 提交次数少，性能好 | 失败回滚慢，占用资源多 |
| 超大事务（100 万条） | — | 可能导致锁表、连接超时、磁盘爆满 |

**推荐的平衡策略**：

```java
public void batchImport(List<Order> orders) {
    final int TX_BATCH_SIZE = 2000;  // 事务内处理 2000 条
    final int COMMIT_BATCH = 5000;    // 每 5000 条提交一次

    int total = 0;
    for (int i = 0; i < orders.size(); i += TX_BATCH_SIZE) {
        int end = Math.min(i + TX_BATCH_SIZE, orders.size());
        List<Order> batch = orders.subList(i, end);

        // 开启小事务
        transactionTemplate.execute(status -> {
            for (Order order : batch) {
                orderMapper.insert(order);
                total++;
            }
            return null;
        });

        // 每 COMMIT_BATCH 条打印进度
        if (total % COMMIT_BATCH == 0) {
            log.info("已导入 {} 条", total);
        }
    }
}
```

### 3.2 错误处理：记录失败的批次

大批量操作中，失败是常态。关键是把失败的批次记录下来，便于后续重试：

```java
public BatchResult importOrders(List<Order> orders) {
    List<Long> successIds = new ArrayList<>();
    List<Long> failedIds = new ArrayList<>();
    List<String> failedReasons = new ArrayList<>();

    for (Order order : orders) {
        try {
            orderMapper.insert(order);
            successIds.add(order.getId());
        } catch (DuplicateKeyException e) {
            // 主键冲突，跳过这条，继续下一条
            failedIds.add(order.getId());
            failedReasons.add("主键冲突: " + order.getId());
        } catch (DataIntegrityViolationException e) {
            // 数据校验失败
            failedIds.add(order.getId());
            failedReasons.add("数据校验失败: " + e.getMessage());
        }
    }

    return new BatchResult(successIds, failedIds, failedReasons);
}
```

## 四、实战案例：导数场景与数据初始化

### 4.1 案例一：历史订单迁移

**背景**：从老系统迁移 500 万条订单到新系统，要求 24 小时内完成，不能影响线上业务。

**方案设计**：

```java
public class OrderMigration {
    private final OrderMapper orderMapper;
    private final DataSource dataSource;

    public int migrateOrders(LocalDate startDate, LocalDate endDate) {
        // 1. 先导出到 CSV（老系统允许只读查询）
        String csvPath = exportToCsv(startDate, endDate);

        // 2. 上传到 MySQL 服务器
        String serverPath = uploadToServer(csvPath);

        // 3. 使用 LOAD DATA INFILE 导入
        return loadDataInfile(serverPath);
    }

    private int loadDataInfile(String serverPath) {
        JdbcTemplate jdbc = new JdbcTemplate(dataSource);

        // 禁用约束检查
        jdbc.execute("SET UNIQUE_CHECKS = 0");
        jdbc.execute("SET FOREIGN_KEY_CHECKS = 0");
        jdbc.execute("SET AUTOCOMMIT = 0");

        String sql = "LOAD DATA INFILE '" + serverPath + "' " +
                     "INTO TABLE orders " +
                     "FIELDS TERMINATED BY ',' " +
                     "ENCLOSED BY '\"' " +
                     "LINES TERMINATED BY '\\n' " +
                     "(id, user_id, amount, create_time)";

        int count = jdbc.update(sql);

        jdbc.execute("COMMIT");
        jdbc.execute("SET AUTOCOMMIT = 1");
        jdbc.execute("SET UNIQUE_CHECKS = 1");
        jdbc.execute("SET FOREIGN_KEY_CHECKS = 1");

        return count;
    }
}
```

**效果**：500 万条数据，约 15 分钟完成。

### 4.2 案例二：初始化配置表

**背景**：系统上线前，需要初始化 1000 条配置数据到配置表。要求幂等（可重复执行），不能重复插入。

**方案**：使用 `INSERT ... ON DUPLICATE KEY UPDATE`。

```sql
INSERT INTO config (id, category, key, value, description, update_time) VALUES
(1, 'system', 'timeout', '3000', '接口超时时间（毫秒）', NOW()),
(2, 'system', 'retry', '3', '重试次数', NOW()),
(3, 'payment', 'min_amount', '1', '最小支付金额（元）', NOW()),
-- ... 更多配置
ON DUPLICATE KEY UPDATE
    value = VALUES(value),
    description = VALUES(description),
    update_time = VALUES(update_time);
```

| 特性 | 说明 |
|---|---|
| 幂等 | 重复执行不会插入重复数据 |
| 更新 | 如果 key 存在，更新 value 和 update_time |
| 高效 | 一条 SQL 完成插入或更新 |

## 五、生产环境注意事项

### 5.1 binlog 格式的影响

批量插入会产生大量 binlog 日志，binlog 格式会直接影响复制效率：

| binlog 格式 | 批量插入的影响 |
|---|---|
| ROW | 记录每一行的变化，binlog 体积大，但安全 |
| STATEMENT | 记录 SQL 语句，binlog 体积小，但可能有不确定因素 |
| MIXED | 混合模式，自动选择 |

在主从复制场景下，**建议使用 ROW 格式**，虽然 binlog 体积大，但数据一致性有保障。

### 5.2 对在线业务的影响

大批量操作对在线业务的影响主要来自：

1. **CPU 和 IO 占用**：导入过程消耗数据库资源
2. **锁竞争**：UPDATE 操作可能锁住其他查询
3. **主从延迟**：binlog 同步到从库需要时间

**最佳实践**：

- 选择业务低峰期执行（如凌晨 2-6 点）
- 控制单次操作的规模，分批执行
- 执行前检查从库延迟，确保在可接受范围内
- 准备回滚方案，一旦出问题能快速恢复

## 六、面试实战

### 6.1 场景一：大批量数据插入优化

**面试官问**：「如果要插入 1000 万条数据，你会怎么做？有哪些优化手段？」

**参考回答**：

> 大批量数据插入有以下几个优化手段：
>
> 1. **合并 INSERT**：把多条 INSERT 合并成一条，减少网络往返和 SQL 解析次数。每批 1000-5000 条比较合适。
>
> 2. **LOAD DATA INFILE**：这是最快的方案，直接读取文件写入数据页，绕过 SQL 解析。1000 万条数据可以在 10-30 分钟内完成。
>
> 3. **禁用约束检查**：导入前执行 `SET UNIQUE_CHECKS = 0` 和 `SET FOREIGN_KEY_CHECKS = 0`，导入完成后再开启。
>
> 4. **禁用自动提交**：`SET AUTOCOMMIT = 0`，每批提交一次，减少 fsync 次数。
>
> 5. **批量更新的分批处理**：如果是 UPDATE 操作，每批处理 1000-2000 条，避免长事务和锁竞争。

### 6.2 场景二：LOAD DATA INFILE 的原理

**面试官问**：「LOAD DATA INFILE 为什么比 INSERT 快那么多？」

**参考回答**：

> INSERT 语句的执行流程是：解析 SQL → 语义分析 → 生成执行计划 → 写入数据 → 更新索引 → 返回结果。
>
> LOAD DATA INFILE 的流程是：读取文件 → 直接写入数据页。
>
> 本质区别在于：**INSERT 需要走 MySQL 的 SQL 引擎层，LOAD DATA INFILE 直接走存储引擎层**。
>
> 具体来说：
>
> 1. **绕过 SQL 解析**：不需要解析 INSERT 语句
> 2. **顺序 IO**：文件本身就是顺序存储的，写入数据页也是顺序 IO
> 3. **批量写入**：一次读取多行，批量写入
> 4. **索引延迟更新**：可以先禁用索引，导入完成后再重建
>
> 综合这些因素，LOAD DATA INFILE 通常比普通 INSERT 快 10-30 倍。

## 总结

批量操作的核心优化思想是：**减少数据库的往返次数，降低每次操作的数据量**。

| 场景 | 优化手段 | 核心价值 |
|---|---|---|
| 批量插入 | INSERT VALUES 合并 | 减少网络往返 |
| 大批量导入 | LOAD DATA INFILE | 绕过 SQL 层，顺序 IO |
| 批量更新 | 分批处理 + 多值 UPDATE | 避免长事务和锁竞争 |
| 生产环境 | 禁用约束检查、自动提交 | 减少日志写入 |

记住两个原则：

1. **能用一条 SQL 搞定的，绝不用多条**
2. **能分批处理的，绝不一次性搞定**

这不只是一个技术问题，更是一个风险控制问题。
