# JOIN 优化：嵌套循环连接与 MRR 优化

凌晨两点，你被报警叫醒：线上接口响应时间飙升至 8 秒。用户投诉页面打不开。

登录服务器，查看慢查询日志，发现一条 SQL 占据了 90% 的执行时间：

```sql
SELECT u.name, o.amount, p.product_name
FROM users u
INNER JOIN orders o ON u.id = o.user_id
INNER JOIN products p ON o.product_id = p.id
WHERE u.status = 'active' AND o.create_time > '2024-01-01';
```

三张表 JOIN，订单表 5000 万条记录，用户表 100 万条记录，商品表 50 万条记录。

explain 看一下，执行计划显示 orders 是驱动表，全表扫描。

等等——100 万条用户的扫描结果，要和 5000 万条订单做 JOIN？这不是把简单问题复杂化了吗？

这就是今天要解决的问题：多表 |JOIN| 优化的核心——**让小表驱动大表**。

---

## 一、嵌套循环连接：JOIN 的底层原理

### 1.1 什么是嵌套循环连接

MySQL 使用 |NLJ|（Nested-Loop Join，嵌套循环连接）作为最基本的 JOIN 算法。原理很简单：

```
FOR each row in t1 (驱动表):
    FOR each row in t2 (被驱动表):
        IF row from t1 matches row from t2:
            RETURN combined row
```

两层循环，外层遍历驱动表，内层遍历被驱动表。如果驱动表有 1000 行，被驱动表有 1000 行，就需要执行 **100 万次**匹配。

这就是为什么驱动表的选择如此重要——**驱动表越大，内层循环次数越多，查询越慢**。

### 1.2 小表驱动大表的原则

假设有以下两张表：

- `users`（用户表）：10 万条记录
- `orders`（订单表）：1000 万条记录

如果用 users 驱动 orders：

```
外层循环：10 万次（users）
内层循环：10 万 × 1000 万 = 1 万亿次匹配
```

如果用 orders 驱动 users：

```
外层循环：1000 万次（orders）
内层循环：1000 万 × 10 万 = 100 万亿次匹配
```

差距是 **100 倍**。这就是小表驱动大表的核心价值。

:::tip 经验法则

**驱动表的选择原则**：让结果集小的表作为驱动表。

在写 SQL 时，你可能觉得无所谓，但执行计划的差异会让你见识到什么叫「失之毫厘，谬以千里」。

:::

### 1.3 什么是 MRR 优化

先看一个常见场景：

```sql
SELECT * FROM orders WHERE user_id IN (1, 2, 3, 4, 5);
```

如果 user_id 有索引，MySQL 会先在索引上找到这 5 个 user_id 对应的记录主键，再回表查询。

问题来了：这 5 次索引查询是**随机 IO**。主键在 B+ 树叶子节点上的物理位置是分散的，每一次回表都是一次磁盘随机读取。

当 IN 列表有 1000 个值时，就是 1000 次随机 IO。

|MRR|（Multi-Range Read，多范围读取）优化登场了。它的核心思路是：

**先把主键 ID 收集起来，排序后批量回表**。

```
1. 在索引上找出所有符合条件的主键 ID：[PK_3, PK_1, PK_5, PK_2, PK_4]
2. 对这些主键排序：[PK_1, PK_2, PK_3, PK_4, PK_5]
3. 按顺序批量回表 —— 变成顺序 IO
```

explain 中的表现：

```sql
EXPLAIN SELECT * FROM orders WHERE user_id IN (1, 2, 3, 4, 5);
```

```
+----+-------------+--------+-------+---------------+-------------+---------+------+------+----------+----------------+
| id | select_type | table  | type  | key           | ref         | rows    | rpp  | Exec | Using MRR | Extra          |
+----+-------------+--------+-------+---------------+-------------+---------+------+------+----------+----------------+
|  1 | SIMPLE      | orders | range | idx_user_id   | const       | 100     | 100  | 100  | Yes      | Using where    |
+----+-------------+--------+-------+---------------+---------------+---------+------+------+----------+----------------+
```

看到 `Using MRR` 就说明 MRR 优化生效了。

### 1.4 MRR 优化的触发条件

MRR 不是你想开就能开的，它需要满足以下条件：

| 条件 | 说明 |
|---|---|
| 范围查询 | `=`、`IN`、`BETWEEN` 等范围查询 |
| 辅助索引 | 查询使用了辅助索引（非主键索引） |
| `mrr=on` | MySQL 配置项，默认开启 |
| `mrr_cost_based=on` | 是否由优化器基于代价决定是否使用 MRR |

```sql
SET optimizer_switch = 'mrr=on,mrr_cost_based=on';
```

大多数情况下，优化器会智能判断是否启用 MRR。但如果你发现明明应该用 MRR 却没有用，可以检查 `mrr_cost_based` 是否被关闭了。

## 二、BKA：MRR 的扩展

### 2.1 BKA 的诞生背景

MRR 优化的是单表回表场景。如果 JOIN 的场景呢？

```sql
SELECT u.name, o.amount
FROM users u
INNER JOIN orders o ON u.id = o.user_id
WHERE u.id IN (1, 2, 3, 4, 5);
```

执行流程是这样的：

```
1. 先在 users 表中找出 id IN (1, 2, 3, 4, 5) 的 5 条记录
2. 遍历这 5 条记录，对每一条去 orders 表做匹配
3. 每次匹配都是一次索引查询 + 回表
```

问题：如果 users 表是驱动表，那步骤 2 中的 5 次查询就是随机的。

**BKA**（Batch Key Access，批量键值访问）就是来解决这个问题的。它将驱动表的结果批量发送给 MySQL Server，由 Server 端统一做查询优化。

### 2.2 BKA 的工作原理

```
1. 驱动表返回一批主键：[1, 2, 3, 4, 5]
2. MySQL Server 将这批主键打包，发送给存储引擎
3. 存储引擎批量获取这些主键对应的记录
4. MRR 优化：对获取到的主键排序，批量回表
```

BKA 让 JOIN 也能享受 MRR 带来的顺序 IO 收益。

### 2.3 启用 BKA

BKA 默认是关闭的，需要手动启用：

```sql
SET optimizer_switch = 'mrr=on,mrr_cost_based=off,batched_key_access=on';
```

```
SET global optimizer_switch = 'batched_key_access=on';
```

:::warning 注意

BKA 依赖 MRR，所以确保 `mrr=on` 是启用 BKA 的前提。

:::

开启后，explain 的 Extra 列会显示 `Using join buffer (Batched Key Access)`：

```sql
EXPLAIN SELECT u.name, o.amount
FROM users u
INNER JOIN orders o ON u.id = o.user_id
WHERE u.id IN (1, 2, 3, 4, 5);
```

```
+----+-------------+-------+--------+---------------+---------+---------+------+----------+--------------------------------+
| id | select_type | table | type   | key           | ref    | rows    | rpp  | Exec     | Extra                          |
+----+-------------+-------+--------+---------------+---------+---------+------+----------+--------------------------------+
|  1 | SIMPLE      | u     | range  | PRIMARY       | NULL   | 5       | 5    | 5        | Using where; Using index       |
|  1 | SIMPLE      | o     | ref    | idx_user_id   | u.id   | 100     | 100  | 100      | Using MRR                      |
+----+-------------+-------+--------+---------------+---------+---------+------+----------+--------------------------------+
```

注意看 orders 表的 Extra 列：`Using MRR`。

## 三、JOIN 的常见坑

### 3.1 强制指定驱动表：STRAIGHT_JOIN

有时候优化器不听话，选了错误的驱动表。这时可以用 `STRAIGHT_JOIN` 强制指定：

```sql
SELECT STRAIGHT_JOIN u.name, o.amount
FROM users u
INNER JOIN orders o ON u.id = o.user_id;
```

`STRAIGHT_JOIN` 的语义是：**按照 SQL 中书写的顺序，从左到右依次作为驱动表**。

在这个例子中，users 是驱动表，orders 是被驱动表。

:::warning 慎用 STRAIGHT_JOIN

STRAIGHT_JOIN 是一种**强制手段**，意味着你完全信任自己的判断而否定优化器的选择。

在以下场景可以使用：

- 你已经通过 explain 确认优化器选错了驱动表
- 业务场景固定，数据分布不会变化

在其他场景，让优化器自己决定。

:::

### 3.2 过多 JOIN 关联

**超过三张表的 JOIN，务必警惕。**

每增加一张表，复杂度指数级上升：

```
2 表 JOIN：O(N × M)
3 表 JOIN：O(N × M × K)
4 表 JOIN：O(N × M × K × L)
```

更重要的是，JOIN 越多，索引失效的概率越高，执行计划越难预测。

实际工作中，我见过一张 SQL JOIN 了 8 张表的惨案。开发人员美其名曰「一次查询搞定」，实际上执行时间 30 秒，还把数据库 CPU 打满。

**解决方案**：拆分成多次查询 + 应用层处理。或者，考虑是否需要反规范化。

### 3.3 警惕 type=ALL 的全表扫描 JOIN

explain 分析时，如果看到 JOIN 的 type 是 ALL（全表扫描），这往往意味着灾难：

```sql
EXPLAIN SELECT u.name, o.amount
FROM users u
INNER JOIN orders o ON u.id = o.user_id
WHERE u.status = 'inactive';
```

```
+----+-------------+-------+------+---------------+-------------+---------+-------+----------+-------------+
| id | select_type | table | type | key           | ref         | rows   | rpp   | Exec     | Extra       |
+----+-------------+-------+------+---------------+-------------+---------+-------+----------+-------------+
|  1 | SIMPLE      | u     | ALL  | NULL          | NULL        | 1000000| 1000000| 1000000  | Using where |
|  1 | SIMPLE      | o     | ref  | idx_user_id   | u.id        | 100    | 1000000| 100000000 |             |
+----+-------------+-------+------+---------------+-------------+---------+-------+----------+-------------+
```

users 表 type=ALL，全表扫描 100 万行。每扫一行，就去 orders 表匹配 100 条记录——**总共 1 亿次匹配**。

如果 u.status 有索引，加上 `USE INDEX(idx_status)` 或者直接改写 SQL。

## 四、多表 JOIN 优化实战

### 4.1 先过滤，再关联

**低效写法**：

```sql
SELECT u.name, o.amount
FROM users u
INNER JOIN orders o ON u.id = o.user_id
WHERE u.status = 'active' AND o.amount > 100;
```

**高效写法**：

```sql
SELECT u.name, o.amount
FROM (SELECT id, name FROM users WHERE status = 'active') u
INNER JOIN (SELECT user_id, amount FROM orders WHERE amount > 100) o
ON u.id = o.user_id;
```

区别在于：**子查询先过滤，减少关联的数据量**。

第一个写法：先做笛卡尔积，再过滤。
第二个写法：先各自过滤，再关联。

### 4.2 不要 SELECT *

很多人写 SQL 顺手就写 `SELECT *`，这在 JOIN 时尤其致命。

假设 orders 表有 50 个字段，你只需要 user_id 和 amount：

```sql
-- 低效
SELECT *
FROM users u
INNER JOIN orders o ON u.id = o.user_id;

-- 高效
SELECT u.name, o.amount
FROM users u
INNER JOIN orders o ON u.id = o.user_id;
```

字段越少，MySQL 需要读取和传输的数据越少，JOIN 的开销越低。

### 4.3 确保关联字段有索引

被驱动表（内层循环的表）的关联字段必须有索引：

```sql
-- orders.user_id 上必须有索引
SELECT u.name, o.amount
FROM users u
INNER JOIN orders o ON u.id = o.user_id;
```

如果 orders.user_id 没有索引，每次匹配都是全表扫描。

索引的设计原则：

| 字段 | 索引类型 | 说明 |
|---|---|---|
| 关联字段 | 单列索引或复合索引 | 确保 JOIN 条件走索引 |
| WHERE 条件字段 | 复合索引 | 考虑最左前缀原则 |
| SELECT 字段 | 覆盖索引 | 如果能覆盖，减少回表 |

### 4.4 控制 JOIN 的数据类型

关联字段类型必须一致：

```sql
-- users.id 是 BIGINT
-- orders.user_id 是 VARCHAR

-- 低效：类型不匹配，索引失效
SELECT * FROM users u
INNER JOIN orders o ON u.id = o.user_id;

-- 高效：类型一致
SELECT * FROM users u
INNER JOIN orders o ON u.id = CAST(o.user_id AS SIGNED);
```

类型不匹配时，MySQL 会做隐式转换，导致索引失效。

## 五、面试实战

### 5.1 场景一：小表驱动大表

**面试官问**：「多表 JOIN 时，如何选择驱动表？为什么要小表驱动大表？」

**参考回答**：

> 驱动表的选择原则是**结果集小的表优先**。
>
> MySQL 使用嵌套循环连接（NLJ），外层循环遍历驱动表，内层循环遍历被驱动表。驱动表的每一行，都要在被驱动表上做一次匹配。如果驱动表有 1000 行，被驱动表有 1000 万行，就需要执行 1000 万次匹配；如果反过来，驱动表 1000 万行，被驱动表 1000 行，就需要执行 1 万亿次匹配。
>
> 差距是指数级的。所以要让小表作为驱动表，减少内层循环的次数。
>
> 在实际 SQL 编写中，我通常会：1）先看 explain 确认优化器选择的驱动表；2）如果不合理，用 STRAIGHT_JOIN 强制指定或调整 SQL 写法；3）确保被驱动表的关联字段有索引。

### 5.2 场景二：MRR 优化原理

**面试官问**：「什么是 MRR 优化？它解决了什么问题？」

**参考回答**：

> MRR（Multi-Range Read，多范围读取）优化的是**随机 IO 问题**。
>
> 当使用辅助索引查询时，MySQL 先在索引上找到主键 ID，再回表获取完整记录。如果 IN 列表有 1000 个值，就是 1000 次随机 IO。
>
> MRR 的核心思路是：先把主键 ID 收集起来，排序后批量回表。这样可以把随机 IO 变成顺序 IO，大幅提升查询性能。
>
> 在 explain 中可以通过 `Using MRR` 关键字确认 MRR 是否生效。

## 六、留给你的思考题

假设有两张表：

- `orders`（订单表）：1 亿条记录，id 是主键，user_id 有索引
- `users`（用户表）：1000 万条记录，id 是主键

业务场景是：查询某个时间段内，下单数量最多的前 10 个用户。

```sql
SELECT u.id, u.name, COUNT(o.id) as order_count
FROM users u
INNER JOIN orders o ON u.id = o.user_id
WHERE o.create_time BETWEEN '2024-01-01' AND '2024-01-31'
GROUP BY u.id, u.name
ORDER BY order_count DESC
LIMIT 10;
```

问题是：

1. 按照小表驱动大表的原则，应该让哪张表作为驱动表？
2. 如果不加任何条件，这张 SQL 的执行计划会是什么样的？
3. 你会如何优化这张 SQL？

## 总结

|JOIN| 优化的核心是**减少无效计算**，而减少无效计算的关键是**让小表驱动大表**。

| 优化点 | 核心价值 |
|---|---|
| 小表驱动大表 | 减少内层循环次数 |
| MRR 优化 | 随机 IO 变顺序 IO |
| BKA 优化 | JOIN 场景下的 MRR 扩展 |
| 先过滤再关联 | 减少 JOIN 的数据量 |
| 确保关联字段有索引 | 避免全表扫描 |

下次写 JOIN 时，先问自己两个问题：**谁驱动谁？关联字段有索引吗？**

回答好这两个问题，你的 JOIN 性能就不会差。
