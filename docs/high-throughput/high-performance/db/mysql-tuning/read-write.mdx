# MySQL 读写分离与延迟优化

想象这样一个场景：凌晨 12 点，你刚发布完新版本，系统一切正常。但早上 9 点一开门，运营同学就炸了——「用户说订单查不到！」

你一查订单数据，发现订单确实入库了。那问题出在哪？

**用户刚下完单，页面刷新，数据没了。再刷新一次，数据又出来了。**

这就是读写分离的经典坑：主从延迟导致的「幻读」问题。

读写分离是 MySQL 提升读性能最常用的方案，但你真的用对了吗？延迟问题怎么处理？哪些场景根本不适合读写分离？这篇文章，把这些问题彻底讲透。

## 一、读写分离的原理

### 1.1 为什么需要读写分离

在互联网业务中，读请求往往是写请求的 5 倍、10 倍甚至更多。一个典型的电商系统，用户浏览商品、查询订单、搜索内容，这些都是读操作。只有下单、支付、修改库存才是写操作。

如果把所有请求都打到主库，主库的压力会非常大。解决方案很直接：**把读请求分散到从库，主库专心处理写操作。**

```
┌─────────┐     ┌─────────┐     ┌─────────┐
│  应用   │────▶│  主库   │     │  从库1  │
│  服务   │     │ (写)    │────▶│ (读)    │
└─────────┘     └─────────┘     └─────────┘
                                  │  从库2  │
                                  │ (读)    │
                                  └─────────┘
```

这就是读写分离的基本架构。

### 1.2 读写分离的路由方式

读写分离的核心是：**写请求路由到主库，读请求路由到从库。** 实现方式有两种。

**方式一：应用层手动路由。** 在代码中手动控制数据库连接。写操作使用主库连接，读操作使用从库连接。

```java
@Service
public class OrderService {

    @Autowired
    @Qualifier("masterDataSource")
    private DataSource masterDataSource;

    @Autowired
    @Qualifier("slaveDataSource")
    private DataSource slaveDataSource;

    // 写操作 - 路由到主库
    public void createOrder(Order order) {
        jdbcTemplate.execute(
            ConnectionFactoryUtils.getConnection(masterDataSource),
            "INSERT INTO orders (...) VALUES (...)"
        );
    }

    // 读操作 - 路由到从库
    public Order getOrder(Long orderId) {
        return jdbcTemplate.queryForObject(
            ConnectionFactoryUtils.getConnection(slaveDataSource),
            "SELECT * FROM orders WHERE id = ?",
            orderId
        );
    }
}
```

这种方式灵活，但问题是：**业务代码和数据源强耦合，改动量大。**

**方式二：中间件路由。** 使用数据库中间件（MyCat、Atlas、ShardingSphere）自动识别 SQL 类型并路由。

```sql
-- 中间件根据 SELECT/INSERT/UPDATE/DELETE 关键字自动路由
-- 读操作自动分发到从库
SELECT * FROM orders WHERE id = 1;  -- 从库

-- 写操作自动路由到主库
INSERT INTO orders (...) VALUES (...);  -- 主库
```

这种方式对应用透明，但增加了运维复杂度，也可能有性能损耗。

**方式三：框架层路由。** 使用 Spring 的 `AbstractRoutingDataSource` 或者 ShardingSphere-JDBC，在框架层面实现读写分离。

```java
public class ReadWriteDataSource extends AbstractRoutingDataSource {

    private static final ThreadLocal<String> dataSourceKey = new ThreadLocal<>();

    public static void setRead() {
        dataSourceKey.set("read");
    }

    public static void setWrite() {
        dataSourceKey.set("write");
    }

    @Override
    protected Object determineCurrentLookupKey() {
        String key = dataSourceKey.get();
        return key != null ? key : "write";
    }
}
```

这是目前最推荐的方式，对业务代码侵入小，配置灵活。

### 1.3 什么场景适合读写分离

不是所有场景都适合读写分离。读写分离的核心优势是**提升读性能**，但它会带来**主从延迟**的问题。

适合读写分离的场景：

| 场景 | 说明 |
|---|---|
| 读多写少 | 读请求占比 80% 以上，写请求相对较少 |
| 对实时性要求不高 | 用户评论、商品详情、历史订单等，允许秒级延迟 |
| 统计分析类查询 | 报表、数据汇总，这类查询本身就耗时，放从库不影响主库 |

不适合读写分离的场景：

| 场景 | 说明 |
|---|---|
| 强一致性要求 | 金融交易、库存扣减，写完必须立刻能读到 |
| 写后立即读 | 注册后查询用户信息、付款后查询订单状态 |
| 实时数据监控 | 仪表盘、数据大屏，必须看到最新数据 |

:::warning 关键判断

如果你在「写后立即读」的流程中用了读写分离，恭喜你，你遇到了文章开头那个「订单消失」的 bug。

:::

## 二、主从复制的原理

读写分离依赖主从复制，理解主从复制是解决延迟问题的基础。

### 2.1 binlog 的三种格式

MySQL 的主从复制核心靠的是 **binlog**（二进制日志）。binlog 记录了所有对数据库的修改操作。binlog 有三种格式，各有优劣。

**Statement 格式：记录 SQL 语句本身。**

```sql
UPDATE orders SET status = 'paid' WHERE id = 1;
```

优点：日志量小，一张表改了 100 万行，只记录一条 SQL。
缺点：部分函数（如 NOW()、RAND()）在从库执行结果可能与主库不一致。

**Row 格式：记录每一行的变化。**

```sql
UPDATE orders SET status = 'paid' WHERE id = 1;
-- 记录：id=1 的行，status 从 'pending' 变成了 'paid'
```

优点：精确记录每行变化，主从完全一致。
缺点：一条 UPDATE 影响 100 万行，日志量巨大（100 万行记录）。

**Mixed 格式：混用 Statement 和 Row。**

默认用 Statement，遇到不确定的函数（如 NOW()）自动切换到 Row。

:::tip 生产环境推荐

生产环境推荐使用 **Row 格式** 或 **Mixed 格式**。虽然 Row 格式日志量大，但保证了主从一致性，出了问题也好排查。

:::

### 2.2 异步复制的完整流程

MySQL 默认的主从复制是**异步**的，整个流程分为五步：

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│   主库      │    │   binlog    │    │   从库      │    │  Relay Log  │
│  (写操作)   │───▶│  (写入磁盘) │───▶│  (IO线程)  │───▶│  (中转日志) │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
                                                                │
                                                                ▼
                                                         ┌─────────────┐
                                                         │  从库重放   │
                                                         │ (SQL线程)  │
                                                         └─────────────┘
```

1. **主库执行写操作**，数据写入 InnoDB，同时写入 binlog（binlog 是独立的日志模块）
2. **binlog 刷新到磁盘**（sync_binlog = 1 时保证持久化）
3. **从库 IO 线程连接主库**，请求拉取 binlog
4. **主库 dump 线程发送 binlog** 给从库，从库写入 Relay Log（中转日志）
5. **从库 SQL 线程读取 Relay Log**，在本地重放执行

异步复制的特点是：**主库执行完就返回客户端，不等待从库执行完成。** 这就是延迟产生的根本原因。

### 2.3 半同步复制：牺牲性能换一致性的方案

异步复制快，但可能丢数据。假设主库写入 binlog 后宕机，从库还没来得及拉取——这台从库就丢了这条数据。

**半同步复制**（Semi-Sync Replication）解决的就是这个问题：

1. 主库执行完写操作，等待**至少一个从库**确认收到 binlog
2. 从库写入 Relay Log 后，回复 ACK 给主库
3. 主库收到 ACK 后，才返回客户端「执行成功」

```sql
-- 安装半同步插件
INSTALL PLUGIN rpl_semi_sync_master SONAME 'semisync_master.so';
INSTALL PLUGIN rpl_semi_sync_slave SONAME 'semisync_slave.so';

-- 启用半同步
SET GLOBAL rpl_semi_sync_master_enabled = 1;
SET GLOBAL rpl_semi_sync_slave_enabled = 1;
```

半同步复制有一个致命问题：`rpl_semi_sync_master_wait_point = AFTER_COMMIT`。

`AFTER_COMMIT` 的流程是：主库 commit 完成后才发送 binlog，收到从库 ACK 后返回客户端。这意味着**主库的事务已经提交了**，如果此时主库宕机，从库没收到，切换到从库后这条数据可能丢失。

**增强半同步**（MySQL 5.7+）解决了这个问题：

```sql
-- 改为 AFTER_SYNC，主库等到从库 ACK 后才 commit
SET GLOBAL rpl_semi_sync_master_wait_point = AFTER_SYNC;
```

`AFTER_SYNC` 的流程：主库写入 binlog，等待从库 ACK，收到 ACK 后才 commit。这保证了**从库没有确认的数据，主库一定不会 commit**。

:::tip 性能影响

半同步复制会增加写操作的延迟，因为要等待从库确认。延迟时间 = 从库 Relay Log 写入 + ACK 返回的网络 RTT。如果从库在异地，这个延迟可能达到几十毫秒。

:::

### 2.4 GTID 复制：让主从管理更简单

GTID（Global Transaction Identifier）是 MySQL 5.6+ 引入的特性。每个事务都有一个全局唯一的 ID，格式是 `server_uuid:transaction_id`。

```sql
-- 开启 GTID 模式
gtid_mode = ON
enforce_gtid_consistency = ON
```

GTID 复制的优势：

1. **自动定位**：从库不需要指定 `MASTER_LOG_FILE` 和 `MASTER_LOG_POS`，MySQL 自动找到需要同步的位置
2. **故障恢复简单**：主从切换时，新主库知道哪些事务已经执行过，不会重复执行
3. **并行复制更精确**：可以基于事务的 GTID 判断哪些可以并行执行

```sql
-- 传统复制需要指定位置
CHANGE MASTER TO
  MASTER_HOST = '192.168.1.100',
  MASTER_LOG_FILE = 'mysql-bin.000001',
  MASTER_LOG_POS = 12345;

-- GTID 复制只需要指定服务器
CHANGE MASTER TO
  MASTER_HOST = '192.168.1.100',
  MASTER_AUTO_POSITION = 1;
```

## 三、读写分离的延迟问题（核心重点）

终于到了最重要的部分。

### 3.1 延迟是怎么产生的

一个写请求的完整生命周期是这样的：

```
主库执行 → binlog 写入 → binlog 传输 → 从库接收 → Relay Log → 从库重放
```

每个环节都有延迟：

| 环节 | 延迟来源 |
|---|---|
| binlog 写入 | 磁盘 I/O、sync_binlog 配置 |
| binlog 传输 | 网络延迟、从库拉取时机 |
| Relay Log | 从库 I/O 负载 |
| 从库重放 | 从库执行效率（CPU、I/O、锁竞争） |

正常情况下，从库延迟可能在毫秒级，用户感知不到。但有些场景下，延迟会飙升到几秒甚至几分钟。

### 3.2 延迟的监控

监控是从库健康度的基础。MySQL 提供了几个关键指标。

**SHOW SLAVE STATUS**：从库状态的核心命令。

```sql
SHOW SLAVE STATUS\G
```

关键字段：

| 字段 | 说明 |
|---|---|
| `Seconds_Behind_Master` | 从库落后主库的秒数，`0` 表示没有延迟，`NULL` 表示从库 IO 线程没运行 |
| `Slave_IO_Running` | IO 线程是否运行 |
| `Slave_SQL_Running` | SQL 线程是否运行 |
| `Last_IO_Error` | 上次 IO 错误信息 |
| `Last_SQL_Error` | 上次 SQL 错误信息 |
| `Relay_Log_Space` | Relay Log 总大小 |
| `Executed_Gtid_Set` | 已执行的 GTID 集合 |

:::warning 注意

`Seconds_Behind_Master` 是估算值，不是精确的。如果从库和主库时间不同步，这个值可能不准确。生产环境建议用 `pt-heartbeat` 等工具监控延迟。

:::

**延迟监控脚本示例**：

```bash
#!/bin/bash
# 监控从库延迟，超过阈值告警

THRESHOLD=5  # 阈值：5秒
DELAY=$(mysql -e "SHOW SLAVE STATUS\G" | grep "Seconds_Behind_Master" | awk '{print $2}')

if [ "$DELAY" != "0" ] && [ "$DELAY" != "NULL" ] && [ "$DELAY" -gt "$THRESHOLD" ]; then
    echo "Alert: Slave lag is $DELAY seconds, exceeds threshold $THRESHOLD"
    # 调用告警脚本
fi
```

### 3.3 延迟的常见原因

从库为什么会落后主库？这里列出最常见的几种原因。

**原因一：大事务**

这是最常见的延迟原因。一个事务里塞了太多操作，从库重放时间就会很长。

```sql
-- 坏例子：一个事务更新 100 万条数据
BEGIN;
UPDATE orders SET status = 'completed' WHERE status = 'pending';
-- 100万行 affected
COMMIT;

-- 好例子：分批处理，每批 1000 条
WHILE (affected_rows > 0) DO
    UPDATE orders SET status = 'completed'
    WHERE status = 'pending' LIMIT 1000;
END WHILE;
```

大事务的问题不仅在于执行时间长，还在于：主库 binlog 传输到一半，从库还没开始重放，主库已经提交了。如果此时主库宕机，从库丢失的数据会很多。

**原因二：缺乏索引导致从库重放慢**

主库的写操作可能走了索引，但从库的查询（用于验证或触发）可能没走索引。

```sql
-- 主库执行：因为 WHERE 有索引，很快
UPDATE orders SET status = 'paid' WHERE user_id = 123;

-- 从库重放时触发的触发器或外键约束
-- 如果缺少索引，全表扫描，导致重放慢
```

**原因三：DDL 操作**

`ALTER TABLE`、`TRUNCATE TABLE` 等 DDL 操作在从库重放非常慢。尤其是大表的加索引、加字段操作，可能导致从库延迟飙升。

解决方案：使用 `pt-online-schema-change` 或 `gh-ost` 做在线 DDL 变更，避免锁表。

**原因四：从库硬件资源不足**

从库的 CPU、I/O 资源被其他进程占用，重放速度跟不上主库写入速度。

解决方案：给从库分配独立磁盘（SSD），避免 I/O 争用；确保从库只跑 MySQL，不要混部其他服务。

**原因五：网络延迟**

主从跨机房部署时，网络延迟直接影响 binlog 传输速度。

解决方案：优先同机房部署；如果必须跨机房，使用压缩（`master_compress_block_size`）减少传输数据量。

## 四、延迟优化方案

### 4.1 方案一：强制路由主库

对于「写后立即读」的场景，最简单粗暴的方案是：**写操作完成后，强制读主库。**

```java
@Service
public class OrderService {

    public Order createOrderAndGet(Order order) {
        // 1. 写操作 - 走主库
        Order created = createOrder(order);  // 路由到主库

        // 2. 写后立即读 - 强制走主库
        DataSourceContextHolder.setDataSource("master");
        return getOrderById(created.getId());
    }

    private Order createOrder(Order order) {
        // 写主库
        ...
    }

    private Order getOrderById(Long id) {
        // 根据 DataSourceContextHolder 决定走主库还是从库
        ...
    }
}
```

:::tip 适用场景

用户注册后查询用户信息、下单后查询订单、付款后跳转结果页——这些场景必须读主库，不要犹豫。

:::

### 4.2 方案二：延迟感知

如果业务能容忍一定的延迟（比如 1 秒），可以在应用层记录写入时间，读从库时判断延迟是否可接受。

```java
@Service
public class UserService {

    // 业务可接受的延迟阈值（秒）
    private static final int ACCEPTABLE_DELAY = 1;

    public User getUser(Long userId) {
        // 记录写入时间
        LocalDateTime writeTime = getWriteTimeFromCache(userId);

        if (writeTime != null) {
            long delaySeconds = Duration.between(writeTime, LocalDateTime.now()).getSeconds();
            if (delaySeconds < ACCEPTABLE_DELAY) {
                // 延迟还在阈值内，强制读主库
                return getUserFromMaster(userId);
            }
        }

        // 延迟超过阈值，正常读从库
        return getUserFromSlave(userId);
    }
}
```

这种方案的优点是：大部分请求走从库，只有刚写入的数据读主库，既保证了实时性，又减少了主库压力。

### 4.3 方案三：选择性走主库

根据业务场景选择是否强制读主库。不同的接口有不同的实时性要求。

```java
@RestController
@RequestMapping("/api/users")
public class UserController {

    // 注册接口 - 必须读主库
    @GetMapping("/check-duplicate")
    public Response checkDuplicate(String username) {
        DataSourceContextHolder.setDataSource("master");
        return userService.checkUsernameExists(username);
    }

    // 用户详情 - 可以读从库，延迟 1 秒可接受
    @GetMapping("/{id}")
    public Response getUserDetail(@PathVariable Long id) {
        DataSourceContextHolder.setDataSource("slave");
        return userService.getUserDetail(id);
    }

    // 用户列表 - 读从库，不关心延迟
    @GetMapping("/list")
    public Response listUsers() {
        DataSourceContextHolder.setDataSource("slave");
        return userService.listUsers();
    }
}
```

### 4.4 方案四：并行复制

并行复制（Parallel Replication）让从库可以并行重放 binlog，充分利用多核 CPU。

```sql
-- 开启并行复制
slave_parallel_workers = 4              -- 启用 4 个 worker 线程
slave_parallel_type = LOGICAL_CLOCK     -- 按逻辑时钟分配事务
slave_parallel_preserve_commit_order = ON  -- 保持提交顺序
```

并行复制的原理：

1. **LOGICAL_CLOCK 模式**：主库上的事务如果是**组提交**（Group Commit），从库可以并行重放它们。因为这些事务在主库上本身就是并行提交的，从库并行执行不会影响数据一致性。

```sql
-- 查看主库的组提交信息
SHOW STATUS LIKE 'Binlog_commit%';
-- Binlog_commit_wait_count: 等待组提交的事务数
-- Binlog_commit_wait_time: 等待时间
```

2. **DATABASE 模式**：按数据库划分，不同数据库的事务可以并行执行。缺点是单库写入为主的场景，收益有限。

:::tip 生产配置建议

```sql
-- 推荐的并行复制配置
slave_parallel_workers = 8              -- CPU 核心数的一半左右
slave_parallel_type = LOGICAL_CLOCK      -- 优先用 LOGICAL_CLOCK
slave_parallel_preserve_commit_order = 1 -- 必须开启，保证数据一致性
```

:::

### 4.5 方案五：从库配置优化

确保从库的硬件和配置不拖累重放速度。

**SSD 是必须的**：机械硬盘的随机 I/O 是并行复制的噩梦。使用 SSD，并行复制的效果才能体现出来。

**调整从库参数**：

```sql
-- 提升从库重放速度
innodb_flush_log_at_trx_commit = 2  -- 降低持久化要求，加快提交
sync_binlog = 0                      -- 从库不需要保证 binlog 不丢
innodb_flush_method = O_DIRECT       -- 绕过文件系统缓存，直接写盘
```

:::warning 风险提示

`sync_binlog = 0` 会导致从库 binlog 丢失。如果从库作为主库的备份，这个配置要慎用。

:::

## 五、读写分离的坑

### 5.1 弱一致性场景才适合读写分离

这是最重要的一条原则。**如果你对数据一致性有强要求，不要用读写分离。**

强一致性场景（必须读主库）：

| 场景 | 原因 |
|---|---|
| 金融交易 | 扣款后立即查余额，必须是最新的 |
| 库存扣减 | 超卖问题，延迟可能导致库存不一致 |
| 分布式事务 | TCC、Saga 等模式，依赖强一致性 |

弱一致性场景（可以读写分离）：

| 场景 | 说明 |
|---|---|
| 内容发布 | 文章发布后几秒看不到，不影响体验 |
| 用户画像 | 统计数据允许分钟级延迟 |
| 日志查询 | 历史日志查询，延迟几秒不影响 |

### 5.2 自增主键的坑

一主多从架构下，如果从库也设置了自增主键，**主从的自增步长必须不同**，否则会冲突。

```sql
-- 主库配置
auto_increment_offset = 1
auto_increment_increment = 2  -- 1, 3, 5, 7...

-- 从库1配置
auto_increment_offset = 2
auto_increment_increment = 2  -- 2, 4, 6, 8...

-- 从库2配置（如果有一主多从）
auto_increment_offset = 3
auto_increment_increment = 2  -- 3, 5, 7, 9...
```

但如果你用了 **GTID + 逻辑时钟并行复制**，这个问题就不存在了，因为并行复制不依赖自增主键分配。

### 5.3 时区问题

如果主库和从库在不同的时区配置下运行，时间函数相关的数据可能不一致。

```sql
-- 主库时区
time_zone = '+08:00'

-- 从库时区（如果配置错误）
time_zone = '+00:00'  -- UTC 时间
```

使用 `NOW()`、`CURDATE()` 等时间函数的语句，在 Statement 格式下，从库执行结果会不同。

**解决方案**：统一使用 `TIMESTAMP` 类型而非 `DATETIME`，或者使用 Row 格式 binlog。

### 5.4 char vs varchar 在不同 binlog 格式下的差异

char 是固定长度的，varchar 是变长的。在不同 binlog 格式下，可能产生不同的结果。

- **Statement 格式**：主库和从库各自根据表结构决定如何存储，可能有尾部空格处理差异
- **Row 格式**：精确记录每行的实际值，不会有这个问题

生产环境建议：统一使用 Row 格式 binlog，避免这类边界问题。

## 六、面试实战

### 6.1 场景一：主从延迟问题排查

**面试官问**：「你的系统从库延迟突然变大，你怎么排查？」

这道题考察的是排查思路和工具使用。

**回答思路**：

1. **第一步：确认延迟现象。** 用 `SHOW SLAVE STATUS` 查看 `Seconds_Behind_Master`，确认延迟确实存在。

2. **第二步：定位哪个环节慢。** 查看从库的 I/O 和 SQL 线程状态：
   - 如果 `Slave_IO_Running = No`，说明 IO 线程挂了，检查 `Last_IO_Error`
   - 如果 `Slave_SQL_Running = No`，说明 SQL 线程挂了，检查 `Last_SQL_Error`

3. **第三步：分析延迟原因。**
   - 看主库有没有大事务：`SHOW PROCESSLIST` 看有没有长时间运行的事务
   - 看 binlog 格式：`ROW` 格式事务较大，传输慢
   - 看从库负载：`SHOW STATUS LIKE 'Threads%'` 看从库连接数和执行效率
   - 看网络延迟：`ping` 主库看 RTT

4. **第四步：处理。**
   - 大事务：拆分事务，分批执行
   - DDL：使用在线 DDL 工具，避免锁表
   - 从库负载高：增加从库数量，分散读压力
   - 网络延迟：同机房部署或压缩传输

### 6.2 场景二：为什么从库会落后主库

**面试官问**：「MySQL 从库为什么会落后于主库？」

这道题考察的是对主从复制原理的理解。

**核心答案**：

> 从库落后主库的根本原因是**异步复制**。主库执行完事务后，不等待从库执行完成，就返回客户端。
>
> 具体来说，延迟产生在以下几个环节：
>
> 1. **主库 binlog 写入**：事务提交后，需要将 binlog 写入磁盘
> 2. **binlog 传输**：主库的 dump 线程将 binlog 发送给从库，网络有延迟
> 3. **从库接收写入 Relay Log**：从库 IO 线程接收并写入 Relay Log
> 4. **从库重放**：SQL 线程读取 Relay Log，在本地执行
>
> 正常情况下，步骤 2 和 3 的延迟只有几毫秒。但以下情况会导致延迟急剧增加：
>
> - **大事务**：一个事务修改 100 万行数据，从库重放需要很长时间
> - **DDL 操作**：大表加索引、添加字段等，操作时间长，锁也长
> - **从库硬件不足**：CPU 打满、I/O 饱和，从库重放速度跟不上
> - **并行复制未开启**：单线程重放，无法利用多核 CPU

## 总结

读写分离是 MySQL 提升读性能的核心手段，但它不是银弹。**主从延迟是读写分离的原罪**，理解延迟产生的原因，才能正确地解决它。

关于读写分离，你需要掌握的核心要点：

| 知识点 | 关键点 |
|---|---|
| 读写分离原理 | 写主库、读从库，通过 binlog 同步 |
| binlog 格式 | Row 格式保证一致性，Statement 格式节省空间 |
| 异步 vs 半同步 | 异步快但可能丢数据，半同步牺牲性能换一致性 |
| GTID 复制 | 自动定位故障恢复简单，支持并行复制 |
| 延迟原因 | 大事务、DDL、从库硬件不足、缺乏索引 |
| 延迟优化 | 强制主库读、延迟感知、并行复制 |
| 避坑指南 | 强一致性场景不用读写分离，注意自增主键、时区问题 |

最后记住一句话：**能用读写分离的场景，读写分离是神器。不能用的场景，强行上读写分离是给自己挖坑。** 选对场景，比优化技巧更重要。
