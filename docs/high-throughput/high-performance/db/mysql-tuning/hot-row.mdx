# MySQL 热点行问题：同一行数据的并发噩梦

2019 年双十一，某电商平台的秒杀接口在 0 点准时开放后，库存从 1000 瞬间变成负数。技术团队排查后发现：一条 SQL `UPDATE inventory SET stock = stock - 1 WHERE id = 1` 被几千个请求同时执行，所有请求全部堆在同一个 |行锁| 上排队等待，最终不仅库存扣成了负数，还把数据库拖垮了。

这就是典型的**热点行问题**。

你可能觉得这是极端案例。但现实开发中，热点行问题远比想象中普遍：**库存扣减、余额扣款、抢票、排行榜积分更新**——任何一个涉及「同一行数据被大量并发更新」的业务场景，都可能踩坑。

这篇文章，我们来彻底拆解热点行问题：从「为什么会这样」，到「本地怎么解决」，再到「分布式场景下怎么破局」，最后配上监控手段和面试套路。

## 一、什么是热点行问题

### 1.1 问题的本质

MySQL 的 |InnoDB| 存储引擎在并发更新同一行数据时，使用的是**行锁（Record Lock）**。行锁的特点是：同一时刻，只有一个事务能持有某行数据的锁，其他事务必须等待。

```
时间线：
T1: 事务A 获取 id=1 行的排他锁，开始更新
T2: 事务B 请求更新 id=1，被阻塞，等待锁
T3: 事务C 请求更新 id=1，被阻塞，等待锁
T4: 事务A 提交，释放锁
T5: 事务B 或事务C 获得锁，继续执行
```

当并发量只有几个的时候，这个锁等待几乎感知不到。但如果同一行数据被**几百甚至几千个请求同时竞争**，问题就来了：

- 大量事务排队等待行锁，RT（响应时间）飙升
- 锁等待占用数据库连接，连接池耗尽
- 等待中的事务积压，数据库 CPU 打满
- 更极端的情况：超时回滚、死锁异常

这就是热点行问题的本质：**高并发 + 同一行 = 锁等待队列 = 系统瓶颈**。

### 1.2 热点行问题的典型场景

不是所有更新操作都会引发热点行问题。典型的热点行场景有以下特征：

**场景一：库存扣减**

```sql
UPDATE products SET stock = stock - 1 WHERE id = ? AND stock > 0
```

秒杀、促销、抢票——几乎所有「限量」场景都涉及库存扣减。而且这类场景还有个特点：**流量集中在活动开始的那几秒钟**。一瞬间涌入的请求，全部打在同一条库存记录上。

**场景二：余额扣款**

```sql
UPDATE accounts SET balance = balance - ? WHERE user_id = ? AND balance >= ?
```

用户余额是强热点数据。特别是余额较少、扣款频繁的业务（如支付、充值），同一用户的扣款请求可能串行化执行。

**场景三：排行榜积分更新**

```sql
UPDATE leaderboard SET score = score + ? WHERE user_id = ?
```

游戏、积分墙、成就系统——排行榜需要频繁更新用户积分。如果按积分排序，top N 的用户永远是最热的热点。

**场景四：乐观锁版本更新**

```sql
UPDATE orders SET status = 'paid', version = version + 1
WHERE order_id = ? AND version = ?
```

很多业务用乐观锁实现幂等性，但在高并发下，冲突会导致大量事务回滚重新尝试，同样会产生热点效应。

### 1.3 热点行问题的量化指标

怎么判断系统是否存在热点行问题？以下几个指标值得关注：

| 指标 | 正常值 | 异常信号 |
|---|---|---|
| `Innodb_row_lock_waits` | 较低 | 持续增长 |
| `Innodb_row_lock_time_avg` | `&lt;` 10ms | `>` 100ms |
| 锁等待数量（`SHOW ENGINE INNODB STATUS`） | `&lt;` 10 | 持续 `>=` 50 |
| 同一行 SQL 的并发执行数 | `&lt;` 10 | 持续 `>=` 50 |
| 事务平均执行时间 | `&lt;` 50ms | `>` 500ms |

如果你在监控上看到这些指标异常，基本可以判定存在热点行问题。

:::tip 热点行 vs 热点键

有时候你会听到「热点键（Hot Key）」和「热点行（Hot Row）」混用。两者有区别：

- **热点键**：被大量请求访问的 key，可能分布在多个 key 上（如缓存 key）
- **热点行**：被大量并发更新的同一行数据

解决热点键可以用缓存分散请求，但解决热点行——即使加了缓存，缓存更新也要写回数据库，根源还是在行锁竞争上。
:::

## 二、本地热点解决方案

所谓「本地热点」，是指**单库单表**范围内，某一行的并发更新成为瓶颈。这时候不需要引入分库分表，通过业务和架构层面的调整来缓解问题。

### 2.1 降低锁粒度：将一行拆成多行

这是解决热点行最直接的思想：**不要把鸡蛋放在一个篮子里**。

以秒杀库存为例。原来只有一条库存记录：

```sql
-- 原来：单行库存
UPDATE inventory SET stock = stock - 1 WHERE id = 1
```

改造后，将库存分散到多条记录上：

```sql
-- 改造后：多条库存记录，随机选一条扣减
UPDATE inventory SET stock = stock - 1
WHERE id IN (SELECT id FROM (SELECT id FROM inventory WHERE stock > 0 ORDER BY RAND() LIMIT 10) t)
```

更工程化的做法：在数据初始化时，按固定规则（如 user_id 哈希）将库存拆分到 N 条记录，扣库存时用哈希定位到其中一条：

```sql
-- 库存表按 sku_id + bucket 分桶，每桶 100 条
UPDATE inventory
SET stock = stock - 1
WHERE sku_id = ? AND bucket = ? AND stock > 0
```

**原理**：将单行锁竞争，分散为 N 行锁竞争。10 个 bucket 的并发能力，理论上提升 10 倍。

**适用场景**：库存数量足够大（几百上千），且允许一定的「随机公平性」。

**注意事项**：

- 扣减后要校验总库存是否正确（所有 bucket 求和）
- 需要额外逻辑处理「某个 bucket 无库存」的重试
- 不适合「一人只能购买一件」的严格顺序场景

### 2.2 异步更新：写请求变事件

热点行的根本问题是「写请求太集中」。如果能把同步写改成异步写，峰值流量就被削平了。

**方案一：记录事件，异步处理**

用户下单时，不直接扣库存，而是写入一条「库存预扣」事件：

```sql
INSERT INTO stock_reserve (order_id, sku_id, quantity, status, create_time)
VALUES (?, ?, 1, 'pending', NOW())
```

库存服务订阅这个事件，异步扣减库存。用户的下单接口可以立即返回「预占成功」，实际库存扣减在后台完成。

**方案二：消息队列削峰**

```
用户请求 → 写入消息队列 → 消费者异步处理 → 更新数据库
```

高并发瞬间，大量大请求堆积在 MQ 中，消费者按自己的能力匀速消费。数据库的压力被控制在可承受范围内。

**适用场景**：允许「最终一致性」，对实时性要求不极端（如秒杀结果可以延迟几秒通知用户）。

**注意事项**：

- 异步化意味着响应延迟，用户体验需要设计兜底方案
- 需要处理消息丢失、重复消费等问题（见「接口幂等性」章节）
- 超时未处理要有补偿机制（如库存超时释放）

### 2.3 Redis 原子扣减：数据库最终一致

这是目前业界最主流的热点行解决方案：**库存预扣到 Redis，用 Redis 的原子操作完成扣减，数据库作为最终的数据源保证一致性**。

**核心流程**：

```
1. 活动开始前：同步库存到 Redis（SET stock:1 1000）
2. 用户下单：Redis DECR 原子扣减（返回值 >= 0 才算成功）
3. 扣减成功：写入订单 + 投递延迟消息
4. 延迟消息：异步扣减数据库库存
5. 数据库扣减失败：补偿 Redis 库存
```

**Redis 扣减代码**：

```java
public boolean deductStock(Long skuId, Long quantity) {
    String key = "stock:" + skuId;
    Long result = redisTemplate.opsForValue().decrement(key, quantity);
    if (result != null && result >= 0) {
        return true;
    }
    // 扣减失败（库存不足），恢复数量
    if (result != null) {
        redisTemplate.opsForValue().increment(key, quantity);
    }
    return false;
}
```

**更安全的 Lua 脚本**：

```lua
local key = KEYS[1]
local quantity = tonumber(ARGV[1])
local stock = tonumber(redis.call('GET', key) or 0)
if stock >= quantity then
    redis.call('DECRBY', key, quantity)
    return 1
else
    return 0
end
```

**数据库最终一致**：

Redis 扣减成功后，写入一条「待扣减」记录到数据库，通过定时任务或消息队列异步完成数据库层面的库存扣减。如果数据库扣减失败（如库存不足），需要补偿 Redis 库存。

**适用场景**：几乎所有「库存」类热点场景，秒杀、抢购、优惠券发放等。

**注意事项**：

- Redis 和 MySQL 的数据一致性需要设计补偿机制
- 超时未完成数据库扣减要有兜底处理
- Redis 故障时要有降级策略（如直接打回数据库）

### 2.4 热点隔离：物理分离热点数据

如果热点数据是某一类特殊数据（如 VIP 用户的账户余额），可以考虑**物理隔离**：将热点数据单独部署、分库分表、或放在独立的数据库实例上。

**方案一：热点表独立实例**

```yaml
# 将热点表放到独立的从库
热点表 → 独立 MySQL 实例（高配机器）
普通表 → 普通从库
```

**方案二：热点数据分片**

将热点数据按某个维度（如用户 ID 范围）拆分到不同表：

```sql
-- 用户表按 user_id % 4 分 4 张表
user_0, user_1, user_2, user_3

-- 扣款时根据 user_id 路由
UPDATE user_1 SET balance = balance - ? WHERE user_id = ?
```

**适用场景**：热点数据明确且集中，有明确的隔离需求。

### 2.5 随机退让：减少锁竞争

在高并发场景下，大量事务同时竞争同一把锁，大部分都在做「无用等待」。如果能在等待时**随机退让**，让一部分请求主动放弃或延迟重试，可以减少锁竞争。

**实现思路**：

```java
public void updateWithRetry(Long id, int maxRetries) {
    Random random = new Random();
    for (int i = 0; i < maxRetries; i++) {
        try {
            // 尝试更新
            int affected = jdbcTemplate.update(
                "UPDATE inventory SET stock = stock - 1 WHERE id = ? AND stock > 0",
                id
            );
            if (affected > 0) {
                return; // 成功
            }
            // 失败（库存不足），直接退出
            return;
        } catch (DuplicateKeyException e) {
            // 锁竞争，随机退让后重试
            try {
                Thread.sleep(random.nextInt(50)); // 0-50ms 随机退让
            } catch (InterruptedException ignored) {}
        }
    }
    throw new BusinessException("库存扣减失败，请重试");
}
```

**原理**：当一个事务因为锁竞争失败时，随机 sleep 一段时间再重试。这样可以让锁请求在时间维度上分散，减少同一时刻的锁竞争。

**适用场景**：并发量不是特别高，可以接受一定的重试延迟。

**注意事项**：

- 退让时间不能太长，否则 RT 会明显增加
- 重试次数不能太多，否则无效等待浪费资源
- 适合与 Redis 扣减等方案配合，不是主力方案

## 三、分布式热点解决方案：分库分表

当单库单表的优化已经到极限，热点行问题仍然无法解决时，就需要**分库分表**：将数据按某个维度分散到多个库/表中，从物理层面打破单行瓶颈。

### 3.1 分库分表的核心思想

分库分表解决热点行的思路很直接：**把同一行的数据，分散到不同的数据库实例或不同的表中**。这样，每行数据的锁竞争只发生在各自的库/表上，并发能力线性扩展。

```
原始架构：
                    [MySQL 单库]
                         │
                 ┌───────┼───────┐
                 │       │       │
               行1     行2     行3    ← 3行数据，共用同一套锁

分库分表后：
    [库1]                 [库2]                 [库3]
      │                     │                     │
    表1                   表2                   表3
      │                     │                     │
   行1-1                行2-1                行3-1
   行1-2                行2-2                行3-2
   行1-3                行2-3                行3-3
```

每行数据的访问分散到不同库，锁竞争被物理隔离。

### 3.2 分片键的选择

分库分表的第一步是**选择分片键（Shard Key）**。分片键决定了数据如何分散，直接影响热点问题的解决效果。

**好的分片键**：

- **访问均匀**：请求能均匀分散到各个分片，避免「热点分片」
- **业务连续**：相关数据尽量在同一分片，减少跨分片查询

**热点行场景下的分片键选择**：

| 场景 | 分片键 | 理由 |
|---|---|---|
| 用户余额扣款 | `user_id` | 按用户 ID 取模或哈希，用户只能操作自己的余额 |
| 商品库存 | `sku_id` 或复合键 | 同 SKU 的库存操作路由到同一分片 |
| 订单处理 | `user_id` + 时间 | 用户查询自己的订单时无需跨分片 |

**常见分片算法**：

```sql
-- 取模分片
分片 = hash(user_id) % 分片数

-- 范围分片
分片 = user_id / 1000  -- 每 1000 用户一个分片

-- 一致性哈希
分片 = hash(user_id) % (分片数 * 副本数)  -- 减少扩容时的数据迁移
```

:::warning 分片键选择的坑

如果分片键选择不当，可能出现「热点分片」问题：大部分请求都路由到某几个分片，其他分片空闲。热点行从一个点变成几个点，但问题没有真正解决。

比如按 `status` 字段分片，把「待支付」订单都放在分片 1，把「已完成」订单放在分片 2。高峰期的「待支付」订单全部压在分片 1，热点转移了，但没消失。
:::

### 3.3 分库分表中间件

常用的分库分表中间件有：

| 中间件 | 特点 | 适用场景 |
|---|---|---|
| ShardingSphere | 社区活跃，功能完善，支持读写分离 | 主流选择 |
| MyCat | 老牌方案，配置复杂 | 遗留系统 |
| Vitess | Google 开源，Kubernetes 原生 | 容器化部署 |
| tdengine | 时序数据专用 | IoT 等时序场景 |

**ShardingSphere 使用示例**（Java 配置）：

```java
@Configuration
public class ShardingConfig {

    @Bean
    public DataSource dataSource() {
        ShardingRuleConfiguration config = new ShardingRuleConfiguration();
        config.getTableRuleConfigs().add(orderTableRule());
        config.setDefaultDatabaseShardingStrategyConfig(
            new InlineShardingStrategyConfiguration("user_id", "ds_${user_id % 2}")
        );
        return DataSourceFactory.getDataSource(config);
    }

    private TableRuleConfiguration orderTableRule() {
        TableRuleConfiguration config = new TableRuleConfiguration();
        config.setLogicTable("t_order");
        config.setActualDataNodes("ds_${0..1}.t_order_${0..15}");
        config.setTableShardingStrategyConfig(
            new InlineShardingStrategyConfiguration("order_id", "t_order_${order_id % 16}")
        );
        return config;
    }
}
```

### 3.4 分库分表后的新问题

分库分表解决了热点行问题，但也引入了新的复杂度：

**问题一：跨分片查询**

按 user_id 分片后，如果要查询「所有商品中库存 `&lt;` 10 的商品」，需要扫描所有分片。

**解决方案**：

- 引入**异构索引表**：库存 `&lt;` 10 的商品 ID 单独建一张表，按库存排序
- 使用**搜索引擎**：用 ES 存储全量数据，支持复杂查询
- **打平汇总**：多分片查询后，在应用层聚合结果（注意分页问题）

**问题二：跨分片聚合**

统计所有分片的库存总量：

```sql
-- 需要查询所有分片后汇总
SELECT SUM(stock) FROM inventory_0
UNION ALL
SELECT SUM(stock) FROM inventory_1
```

**解决方案**：

- 使用**异步汇总表**：定时任务将各分片数据汇总到汇总表
- 引入**Redis 计数**：用 Redis 维护实时库存总量（Redis 作为 buffer）

**问题三：分页排序**

按库存升序分页查询：

```sql
SELECT * FROM inventory ORDER BY stock ASC LIMIT 10 OFFSET 1000
```

分库分表后，每一页都需要从所有分片取数据再排序，效率极低。

**解决方案**：

- **禁止深度分页**：限制最大页数，如只允许查看前 100 页
- **游标分页**：使用上一页最后一条记录的 ID 作为起点
- **ES 承接查询**：分页和复杂排序交给 ES

**问题四：分布式 ID 生成**

分库分表后，主键 ID 不能用自增，需要全局唯一 ID。

**解决方案**：

| 方案 | 原理 | 适用场景 |
|---|---|---|
| 雪花算法 | 时间戳 + 机器 ID + 序列号 | 主流选择 |
| UUID | 随机生成 | 不推荐（无序、存储大） |
| 数据库号段 | 预申请 ID 区间 | 简单场景 |
| ZK / Redis 生成 | 中心化生成 | 有中心化依赖 |

```java
// 雪花算法实现
public class SnowflakeIdGenerator {
    private final long twepoch = 1609459200000L;
    private final long workerIdBits = 5L;
    private final long sequenceBits = 12L;

    private final long maxWorkerId = ~(-1L << workerIdBits);
    private final long workerId;
    private long sequence = 0L;
    private long lastTimestamp = -1L;

    public SnowflakeIdGenerator(long workerId) {
        if (workerId > maxWorkerId || workerId < 0) {
            throw new IllegalArgumentException("worker Id can't be greater than " + maxWorkerId);
        }
        this.workerId = workerId;
    }

    public synchronized long nextId() {
        long timestamp = timeGen();
        if (timestamp < lastTimestamp) {
            throw new RuntimeException("Clock moved backwards");
        }
        if (lastTimestamp == timestamp) {
            sequence = (sequence + 1) & ((1 << sequenceBits) - 1);
            if (sequence == 0) {
                timestamp = tilNextMillis();
            }
        } else {
            sequence = 0;
        }
        lastTimestamp = timestamp;
        return ((timestamp - twepoch) << (workerIdBits + sequenceBits))
                | (workerId << sequenceBits)
                | sequence;
    }
}
```

## 四、热点监控与发现

知道了热点行问题的危害，更重要的是**提前发现**。下面介绍几种热点行的监控手段。

### 4.1 锁等待监控

**查看当前锁等待**：

```sql
SHOW ENGINE INNODB STATUS;
```

输出中的 `LOCK WAITS` 部分显示了当前等待的锁：

```
---TRANSACTION 12345, ACTIVE 5 sec
mysql tables in use 1, locked 1
LOCK WAIT lock struct(s), heap size 400, 2 row lock(s)
LOCK WAIT 2 lock struct(s), heap size 400
MySQL thread id 5, OS thread handle 1234, query id 100 localhost root
UPDATE inventory SET stock = stock - 1 WHERE id = 1
```

**查看锁信息表**：

```sql
-- 当前持有的锁
SELECT * FROM information_schema.INNODB_LOCKS;

-- 锁等待关系
SELECT * FROM information_schema.INNODB_LOCK_WAITS;
```

### 4.2 慢查询日志中的锁等待

锁等待时间会被记录到慢查询日志中。检查慢查询日志：

```sql
-- 查看最近的慢查询
SHOW FULL PROCESSLIST;
```

如果发现大量 SQL 执行时间很长，但 CPU 使用率不高，很可能是锁等待导致的。

### 4.3 Performance Schema 监控

MySQL 5.6+ 提供了 `performance_schema` 表，可以监控锁统计：

```sql
-- 查看最近的锁等待事件
SELECT * FROM performance_schema.events_waits_history
WHERE OBJECT_TYPE = 'LOCK'
ORDER BY TIMER_WAIT DESC LIMIT 10;

-- 统计每个表的锁等待次数
SELECT OBJECT_SCHEMA, OBJECT_NAME, COUNT(*)
FROM performance_schema.events_waits_history
WHERE OBJECT_TYPE = 'LOCK'
GROUP BY OBJECT_SCHEMA, OBJECT_NAME;
```

### 4.4 应用层监控

除了数据库层面的监控，应用层也要埋点：

- **接口 RT 监控**：RT 突增时排查是否锁等待导致
- **线程池队列监控**：等待线程数突增
- **错误率监控**：超时、回滚错误增多

## 五、实战案例分析

### 5.1 案例一：电商秒杀系统

**场景**：双十一秒杀 1000 台 iPhone，库存只有 1000 件。

**问题分析**：

- 峰值 QPS 可能达到 10 万+
- 所有请求都打到同一条库存记录
- 直接扣库存会导致行锁竞争严重

**解决方案**：

1. **活动预热**：秒杀开始前，将库存同步到 Redis
2. **流量过滤**：验证码、答题等方式降低有效请求量
3. **Redis 扣减**：用 Lua 脚本原子扣减，控制并发
4. **异步下单**：扣减成功后，写入订单消息队列，异步生成订单
5. **数据库兜底**：定时任务检查 Redis 和 MySQL 一致性

**关键代码**：

```lua
-- Lua 脚本：原子扣库存
local stock = tonumber(redis.call('GET', KEYS[1]) or 0)
if stock >= tonumber(ARGV[1]) then
    redis.call('DECRBY', KEYS[1], ARGV[1])
    return 1  -- 扣减成功
else
    return 0  -- 库存不足
end
```

**结果**：Redis 单机 QPS 可达 10 万+，热点行问题完全规避。

### 5.2 案例二：12306 抢票系统

**场景**：春运抢票，热门车次余票查询和下单。

**问题分析**：

- 同一车次的余票是热点数据
- 余票数量很少，但查询量巨大
- 扣票涉及多表更新（订单表、座位表、余票表）

**解决方案**：

1. **读请求缓存化**：余票数据缓存到 Redis，减少数据库读压力
2. **写请求串行化**：余票扣减使用 Redis 分布式锁，保证原子性
3. **分桶设计**：将座位按车厢分桶，每节车厢独立扣减
4. **异步确认**：扣减成功即返回「占座成功」，后台异步确认

**关键点**：12306 的核心难点不是单行热点，而是**多表关联 + 强一致性**。解决方案是适当牺牲强一致性，换取高可用。

### 5.3 案例三：积分排行榜

**场景**：游戏积分排行榜，top 100 用户积分频繁更新。

**问题分析**：

- top 100 的用户永远是热点
- 每次积分变化都要更新排行榜

**解决方案**：

1. **分段存储**：将排行榜分成多个段（1-100, 101-200），不同段分散在不同 Redis Sorted Set
2. **延迟更新**：积分变化写入消息队列，批量更新排行榜
3. **本地缓存**：每个节点维护本地排行榜，减少 Redis 访问

**Redis ZSet 实现**：

```java
// 更新用户积分
redisTemplate.opsForZSet().incrementScore("leaderboard", userId, delta);

// 获取 top 100
Set<ZSetOperations.TypedTuple> topUsers =
    redisTemplate.opsForZSet().reverseRangeWithScores("leaderboard", 0, 99);

// 获取用户排名
Long rank = redisTemplate.opsForZSet().reverseRank("leaderboard", userId);
```

## 六、面试实战

### 6.1 场景一：秒杀系统如何保证库存不超卖

**面试官问**：「秒杀系统如何保证库存不超卖，同时又能应对高并发？」

这道题考察的是对热点行问题的综合理解。

**回答思路**：

> 秒杀系统的核心挑战是**高并发 + 库存扣减 + 不超卖**。如果直接用数据库扣库存，所有并发请求都会竞争同一行锁，数据库会直接挂掉。
>
> 我的解决方案是**分层处理**：
>
> **第一层：Redis 原子扣减**
>
> 秒杀开始前，将库存同步到 Redis。用 Lua 脚本保证原子性：
>
> ```lua
> if redis.call('GET', KEYS[1]) >= ARGV[1] then
>     redis.call('DECRBY', KEYS[1], ARGV[1])
>     return 1
> else
>     return 0
> end
> ```
>
> Redis 单机 QPS 可以达到 10 万+，完全扛住秒杀流量。
>
> **第二层：异步下单**
>
> Redis 扣减成功后，立即返回「预占成功」。后台通过消息队列异步创建订单，更新数据库。
>
> **第三层：数据库兜底**
>
> 定时任务检查 Redis 和 MySQL 的一致性。如果 Redis 扣减成功但数据库下单失败，需要补偿 Redis 库存。
>
> **核心思想**：用 Redis 扛住高并发写，数据库保证最终一致性。这不是「不要数据库」，而是「让数据库做它擅长的事」。

### 6.2 场景二：分库分表后如何生成全局唯一 ID

**面试官问**：「分库分表后，自增 ID 不能用了，有什么替代方案？」

**回答思路**：

> 分库分表后，每个表的 ID 只能保证表内唯一，不能保证全局唯一。替代方案有几种：
>
> **方案一：雪花算法（Snowflake）**
>
> 时间戳 + 机器 ID + 序列号，64 位整数，理论上不依赖数据库。主流选择。
>
> **方案二：数据库号段模式**
>
> 从数据库批量申请 ID 区间（如每次申请 1000 个），在内存中分配。用完后再次申请。性能好，但依赖数据库。
>
> **方案三：UUID**
>
> 不推荐。无序（导致索引页分裂）、存储空间大（36 字符 vs 8 字节）。
>
> 我推荐**雪花算法**，因为它不依赖中心化服务，性能高，且 ID 有序（对索引友好）。

### 6.3 场景三：Redis + MySQL 如何实现扣库存

**面试官问**：「Redis + MySQL 如何实现库存扣减，如何保证一致性？」

**回答思路**：

> Redis + MySQL 的核心问题是**分布式一致性**。我的方案分三步：
>
> **第一步：预扣 Redis**
>
> ```java
> Long result = redis.decr("stock:" + skuId);
> if (result >= 0) {
>     // 预扣成功，写入消息队列
>     mq.send("扣库存", skuId);
> } else {
>     // 库存不足，恢复 Redis
>     redis.incr("stock:" + skuId);
>     return "库存不足";
> }
> ```
>
> **第二步：异步扣数据库**
>
> 消费者接收消息，扣减 MySQL 库存。如果 MySQL 扣减失败（库存不足），发送「库存补偿」消息。
>
> **第三步：补偿 Redis**
>
> 补偿消费者收到补偿消息后，将 Redis 库存加回来。
>
> **一致性保证**：通过 MQ 的**可靠消息**机制 + **补偿机制**实现最终一致性。允许短暂的不一致（Redis 和 MySQL 差几秒），但不允许永久不一致。

## 七、面试追问预判

掌握以上内容后，你已经能应对大部分热点行问题。但面试官可能会继续追问：

**追问一**：Redis 扣库存时，如果 Redis 挂了怎么办？

> 这时要评估业务容错要求。几种降级方案：
> - **方案一**：直接打回数据库（单库单表模式），牺牲性能保证可用
> - **方案二**：限流保护，Redis 恢复后再处理积压请求
> - **方案三**：降级为「排队购买」，人工发放购买资格

**追问二**：分库分表后，如何做分页查询？

> 分页查询是分库分表的最大痛点。核心思路：
> - **禁止深度分页**：限制最大页数（如只允许前 20 页）
> - **游标分页**：用上一页最后一条 ID 作为起点
> - **ES 承接**：复杂查询和分页交给 Elasticsearch

**追问三**：如果 MySQL 出现了死锁，怎么排查？

> **排查步骤**：
> 1. `SHOW ENGINE INNODB STATUS` 查看死锁信息
> 2. 分析 `LATEST DETECTED DEADLOCK` 部分，找到两个事务的 SQL
> 3. 检查业务代码中的加锁顺序是否一致
> 4. 优化 SQL，尽量减少锁范围（如使用覆盖索引）

**追问四**：热点行问题和热点 key 问题有什么区别？

> - **热点行**：多个请求并发更新**同一行**数据，竞争行锁
> - **热点 key**：多个请求并发读取**同一个 key**（如缓存 key），造成缓存击穿
>
> 共同点是「单点瓶颈」。区别是：热点行是**写热点**，热点 key 是**读热点**。读热点可以用缓存 + 多级缓存解决，写热点必须分散写请求。

### 题目一：Redis 扣库存的边界情况

假设 Redis 扣库存成功后，消息队列发送失败了（MQ 挂了），导致库存预扣了但没有扣数据库。

### 题目二：分库分表的扩容问题

**问题**：扩容后，`user_id % 8` 和 `user_id % 4` 的映射关系会变化，历史数据如何迁移？有哪些方案可以减少迁移成本？

**提示**：一致性哈希可以在一定程度上减少数据迁移，但无法完全避免。

## 总结

热点行问题的本质是**高并发 + 同一行 = 锁等待**。解决思路分为两个层次：

**本地优化层**（不需要改动架构）：

- 拆分热点行，降低锁粒度
- 异步更新，写请求削峰
- Redis 原子扣减，用内存扛并发
- 物理隔离热点数据

**分布式方案层**（架构级改动）：

- 分库分表，从物理层面隔离热点
- 选择合适的分片键，避免热点分片
- 解决跨分片查询、聚合、分页问题
- 分布式 ID 生成（雪花算法）

没有银弹。每种方案都有适用场景和代价：**Redis 扣减牺牲了强一致性**，**分库分表引入了查询复杂性**。选择哪个方案，取决于业务对一致性、实时性、成本的要求。

理解了这些，你就能在面试和工作中游刃有余地应对热点行问题。
