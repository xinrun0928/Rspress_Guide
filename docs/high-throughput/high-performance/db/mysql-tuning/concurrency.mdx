# MySQL 并发控制：MVCC 与锁优化

两个事务同时更新同一行数据，后执行的事务覆盖了先执行的事务的修改——这是最经典的并发更新丢失问题。

或者换个场景：事务 A 读取了一批数据，事务 B 在这个过程中插入了新的数据，事务 A 再次读取时，发现数据变多了——这就是「幻读」。

MySQL 是如何解决这些问题的？答案就在两个字里：**MVCC** 和 **锁**。两者相辅相成，共同构成了 InnoDB 的并发控制体系。

这篇文章的核心任务，是把这两个机制讲透。不只是「是什么」，更要讲清楚「为什么这样设计」和「在什么场景下用」。

## 一、MVCC：读不加锁的魔法

### 1.1 核心思想：多版本，一个快照

传统的并发控制思路是「读写互斥」：读取数据时加锁，写入数据时加锁，读和写之间互相阻塞。这种方式简单粗暴，但代价是**并发度极低**——读操作和写操作不能同时进行。

MVCC 换了一种思路：**每行数据维护多个版本，每个事务看到一个快照。** 读操作不阻塞写操作，写操作也不阻塞读操作。读写之间不再互斥，并发度大幅提升。

这个设计的核心洞察是：**大多数业务场景中，读操作远多于写操作。** 如果能让读操作完全不加锁，就能显著提升系统的并发能力。

### 1.2 隐藏字段：每一行的「身份证」

InnoDB 存储引擎在每行数据背后，加了三个隐藏字段（以主键表为例）：

```sql
DB_ROW_ID      -- 隐藏主键，如果表没有定义主键则自动生成
DB_TRX_ID      -- 最近修改该行的事务ID
DB_ROLL_PTR    -- 回滚指针，指向undo log中的历史版本
```

| 字段 | 含义 | 作用 |
|------|------|------|
| `DB_ROW_ID` | 隐藏主键 | 6 字节，如果表没有主键则自动生成 |
| `DB_TRX_ID` | 事务ID | 标识最近一次修改该行的事务 |
| `DB_ROLL_PTR` | 回滚指针 | 指向 undo log 中的旧版本，形成版本链 |

:::tip 版本链的工作原理

当你修改一行数据时，InnoDB 会这样做：

1. 把当前数据复制到 undo log 中（旧版本）
2. 修改当前行数据（新版本）
3. 新版本的 `DB_TRX_ID` 记录当前事务ID
4. 新版本的 `DB_ROLL_PTR` 指向 undo log 中的旧版本

多次修改会形成一条 undo log 链：

```
最新版本 → DB_ROLL_PTR → undo log v2 → DB_ROLL_PTR → undo log v1 → NULL
```

:::

### 1.3 ReadView：快照读的眼睛

MVCC 的核心是「快照读」——每个事务在开始时，会生成一个 ReadView（读视图），这个快照决定了事务能看到哪些版本的数据。

ReadView 包含四个关键信息：

```sql
m_ids         -- 当前活跃事务ID列表（已开启但未提交的事务）
min_trx_id    -- 活跃事务中的最小ID
max_trx_id    -- 创建ReadView时最大事务ID + 1
creator_trx_id -- 当前事务的ID
```

事务在读取数据时，通过 ReadView 判断版本的可见性：

- 如果版本的 `DB_TRX_ID` `<` `min_trx_id`：说明这个版本在当前事务开始前就已经提交了，**可见**。
- 如果版本的 `DB_TRX_ID` `>=` `max_trx_id`：说明这个版本是在当前事务之后创建的，**不可见**。
- 如果版本的 `DB_TRX_ID` 在 `m_ids` 列表中：说明这个版本是并发事务创建的，**不可见**。
- 否则：说明这个版本是已提交事务创建的，**可见**。

### 1.4 RC vs RR：Read Committed 和 Repeatable Read 的区别

这是面试中的高频问题。两种隔离级别的本质区别，就在于 **ReadView 的生成时机**。

| 隔离级别 | ReadView 生成时机 | 特点 |
|----------|-------------------|------|
| `READ COMMITTED` | **每次读取数据时**生成新的 ReadView | 能看到其他已提交事务的修改 |
| `REPEATABLE READ` | **事务开始时**生成 ReadView，之后不再更新 | 整个事务看到的是同一个快照 |

用一个场景说明两者的区别：

假设时间顺序是：

1. T1 时刻：事务 A 开始（id = 10）
2. T2 时刻：事务 B 修改了 id = 1 的行，将 age 从 20 改为 25（未提交）
3. T3 时刻：事务 A 读取 id = 1 的行
4. T4 时刻：事务 B 提交
5. T5 时刻：事务 A 再次读取 id = 1 的行

在 `READ COMMITTED` 下：
- T3 时刻：ReadView 中 `m_ids` 包含事务 B（id = 10 之后的某事务），版本不可见，返回 age = 20
- T5 时刻：**重新生成 ReadView**，此时事务 B 已提交，`m_ids` 不包含它，版本可见，返回 age = 25

在 `REPEATABLE READ` 下：
- T3 时刻：ReadView 中 `m_ids` 包含事务 B，版本不可见，返回 age = 20
- T5 时刻：**使用同一个 ReadView**，事务 B 虽然已提交，但版本仍不可见，返回 age = 20

这就是为什么 `REPEATABLE READ` 叫「可重复读」——同一个事务中，多次读取同一行数据，返回的结果是相同的。

:::warning 注意

`REPEATABLE READ` 级别下，MySQL 通过 Next-Key Lock 解决了幻读问题，而不是依赖 MVCC。MVCC 主要解决的是「快照读」的幻读问题。

:::

### 1.5 快照读 vs 当前读

在理解 MVCC 时，必须区分两种读操作：

**快照读（Snapshot Read）**：读取的是历史版本，不加锁。

```sql
SELECT * FROM user WHERE id = 1;  -- 普通SELECT，读取快照
```

**当前读（Current Read）**：读取的是最新版本，加锁。

```sql
SELECT * FROM user WHERE id = 1 FOR UPDATE;  -- 加锁读取
UPDATE user SET age = 30 WHERE id = 1;       -- 更新操作
INSERT INTO user VALUES (...);                -- 插入操作
DELETE FROM user WHERE id = 1;                -- 删除操作
```

当前读通过锁来保证并发安全，快照读通过 MVCC 来保证并发安全。两者配合，实现了「读不加锁，写加锁」的并发模型。

## 二、Next-Key Lock：RR 级别的幻影杀手

### 2.1 三种锁的层次

MySQL 的锁并不是一个单一概念，它分为三个层次：

**Record Lock（记录锁）**：锁住索引记录本身。

```
index:  [1] [3] [5] [7] [9]
lock:    R   R   R   R   R   (R = Record Lock)
```

**Gap Lock（间隙锁）**：锁住索引之间的间隙，防止插入。

```
index:  [1] [3] [5] [7] [9]
lock:    G   G   G   G   G   (G = Gap Lock)
         ↓   ↓   ↓   ↓   ↓
范围:  (-∞,1) [1,3) [3,5) [5,7) [7,9) [9,+∞)
```

**Next-Key Lock（临键锁）**：Record Lock + Gap Lock 的组合。

```
index:  [1] [3] [5] [7] [9]
lock:   [R,G] [R,G] [R,G] [R,G] [R,G]
```

顾名思义，Next-Key Lock 锁住的是「当前索引记录 + 下一个索引之前的间隙」。

### 2.2 RR 级别下防止幻读的原理

幻读（Phantom Read）指的是：同一个事务中，两次相同条件的查询返回了不同的行数。

场景模拟：

```sql
-- 事务A
BEGIN;
SELECT COUNT(*) FROM orders WHERE status = 'pending';  -- 返回 100 条

-- 事务B（同时执行）
INSERT INTO orders VALUES (101, 'pending');  -- 插入一条

-- 事务A
SELECT COUNT(*) FROM orders WHERE status = 'pending';  -- 返回 101 条！幻读
COMMIT;
```

在 `REPEATABLE READ` 级别下，InnoDB 使用 Next-Key Lock 来防止幻读：

```sql
SELECT COUNT(*) FROM orders WHERE status = 'pending' FOR UPDATE;
```

这条语句会对所有 `status = 'pending'` 的记录加 Record Lock，同时对它们之间的间隙加 Gap Lock。这样，事务 B 的 INSERT 操作会被阻塞，直到事务 A 提交。

:::tip 非唯一索引的 Next-Key Lock

对于非唯一索引，Next-Key Lock 的范围会更广。例如：

```sql
SELECT * FROM user WHERE age = 25 FOR UPDATE;
```

假设 age 索引中有 age=25 的记录：[20, 25, 25, 30]。

Next-Key Lock 会锁住：(20, 25] 和 (25, 30) 两个区间。这意味着在这两个区间内的插入都会被阻塞。

:::

### 2.3 什么情况下会加 Gap Lock

Gap Lock 并不是所有查询都会加，以下场景会产生 Gap Lock：

**等值查询（非唯一索引）**：

```sql
SELECT * FROM user WHERE age = 25 FOR UPDATE;
-- 会在 age 索引的 (20, 30) 区间加 Gap Lock
```

**范围查询**：

```sql
SELECT * FROM user WHERE id > 5 AND id < 10 FOR UPDATE;
-- 会在 (5, 10) 区间加 Gap Lock
```

**唯一索引的等值查询不会加 Gap Lock**（只会锁住记录本身）：

```sql
SELECT * FROM user WHERE id = 5 FOR UPDATE;
-- 只会锁住 id=5 这条记录，不会锁间隙
```

### 2.4 索引失效时的锁升级

这是一个生产环境中容易被忽略的坑。

当查询条件涉及的索引失效时（例如 where 子句对索引列做了函数运算），InnoDB 会进行全表扫描。此时，**锁会从行锁升级为表锁**。

```sql
-- 假设 name 有索引
SELECT * FROM user WHERE LEFT(name, 3) = 'Tom' FOR UPDATE;
-- name 索引失效，全表扫描，锁住整张表
```

:::danger 索引失效导致锁升级的后果

线上执行这样的 SQL，配合长事务，会导致大量其他请求被阻塞，甚至造成数据库连接耗尽。

生产环境中务必：
1. 避免在索引列上使用函数
2. 避免类型转换
3. 对涉及锁的 SQL 先在测试环境验证

:::

## 三、死锁：并发下的僵局

### 3.1 死锁的四个必要条件

死锁的发生，必须同时满足以下四个条件（也叫「Coffman 条件」）：

| 条件 | 含义 |
|------|------|
| **互斥** | 一段时间内，资源只能被一个事务占用 |
| **占有并等待** | 事务已经保持至少一个资源，同时请求其他资源 |
| **不可抢占** | 已获得的资源不能被强制释放，只能事务自己释放 |
| **循环等待** | 事务之间形成头尾相接的等待环路 |

破坏其中任何一个条件，就能避免死锁。

### 3.2 典型场景：两事务交叉持锁

最经典的两事务死锁场景：

```sql
-- 事务A
BEGIN;
UPDATE account SET balance = balance - 100 WHERE id = 1;  -- 锁住 id=1
UPDATE account SET balance = balance + 100 WHERE id = 2;  -- 等待 id=2

-- 事务B（同时执行）
BEGIN;
UPDATE account SET balance = balance - 100 WHERE id = 2;  -- 锁住 id=2
UPDATE account SET balance = balance + 100 WHERE id = 1;  -- 等待 id=1
-- 死锁！
```

时间线：

```
T1: 事务A 锁住 id=1
T2: 事务B 锁住 id=2
T3: 事务A 等待 id=2
T4: 事务B 等待 id=1
T5: 死锁检测触发
```

### 3.3 死锁检测与处理

InnoDB 引擎内置了死锁检测机制，通过 `innodb_deadlock_detect` 参数控制（默认 `on`）。

```sql
SHOW VARIABLES LIKE 'innodb_deadlock_detect';
-- +------------------------+---------+
-- | Variable_name          | Value   |
-- +------------------------+---------+
-- | innodb_deadlock_detect | on      |
-- +------------------------+---------+
```

死锁检测的原理：检测是否存在循环等待的图，如果存在，说明发生死锁。检测到死锁后，InnoDB 会选择**回滚代价最小的事务**（通常是根据事务权重决定，修改行数少的事务被回滚）。

死锁处理策略：

1. **超时回滚**：如果死锁检测关闭了，事务会等待 `innodb_lock_wait_timeout` 秒后超时回滚（默认 50 秒）。缺点是等待时间长。
2. **死锁检测回滚**：死锁检测开启时，直接检测并回滚一个事务，代价更小。

:::tip 死锁检测的代价

死锁检测本身是有代价的——检测算法的时间复杂度是 O(n²)。在高并发场景下，如果大量事务持有锁，检测本身可能成为瓶颈。

解决方案：关闭死锁检测（`innodb_deadlock_detect = off`），改用锁超时策略。但这需要业务层做好重试逻辑。

:::

### 3.4 死锁优化思路

既然死锁无法完全避免，那就从减少死锁概率和加快死锁恢复两个角度优化。

**优化一：固定顺序获取锁**

```sql
-- 错误写法：事务A先锁1后锁2，事务B先锁2后锁1
UPDATE account SET balance = balance - 100 WHERE id = 1;
UPDATE account SET balance = balance + 100 WHERE id = 2;

-- 正确写法：所有事务都按 id 从小到大的顺序获取锁
UPDATE account SET balance = balance - 100 WHERE id = 1;
UPDATE account SET balance = balance + 100 WHERE id = 2;
```

**优化二：减少锁的持有时间**

```sql
-- 错误写法：业务逻辑放在锁内
BEGIN;
SELECT * FROM account WHERE id = 1 FOR UPDATE;
-- 复杂的业务计算（耗时 10 秒）
UPDATE account SET balance = balance - 100 WHERE id = 1;
COMMIT;

-- 正确写法：锁内只做必要的操作
BEGIN;
-- 先在锁外完成业务计算
SELECT balance FROM account WHERE id = 1;
-- 计算得出扣款金额
UPDATE account SET balance = balance - 100 WHERE id = 1;
COMMIT;
```

**优化三：合理使用低隔离级别**

在允许的场景下，使用 `READ COMMITTED` 隔离级别。RC 级别下，快照读不加 Gap Lock，减少了锁的范围。

**优化四：批量操作使用锁提示**

```sql
SELECT * FROM account WHERE id IN (1, 2, 3) FOR UPDATE;
-- 一次锁定所有需要的行，而不是分多次锁定
```

## 四、乐观锁与悲观锁：两种并发策略

### 4.1 悲观锁：先锁再说的防御派

悲观锁的核心假设是：**冲突必然发生，所以先加锁再说。**

```sql
SELECT * FROM inventory WHERE product_id = 100 FOR UPDATE;
-- 检查库存
UPDATE inventory SET stock = stock - 1 WHERE product_id = 100;
```

**适用场景**：写多的场景，冲突概率高。

| 优点 | 缺点 |
|------|------|
| 一定能避免冲突 | 锁竞争激烈时，性能下降 |
| 逻辑简单，不容易出错 | 可能导致死锁 |
| 适合强一致性场景 | 锁粒度大时，并发能力受限 |

### 4.2 乐观锁：相信冲突是小概率事件的乐观派

乐观锁的核心假设是：**冲突是小概率事件，先尝试操作，失败再重试。**

实现方式：版本号或 CAS（Compare And Swap）。

**方式一：版本号**

```sql
-- 查询时带上版本号
SELECT stock, version FROM inventory WHERE product_id = 100;
-- 返回 stock=10, version=5

-- 更新时检查版本号
UPDATE inventory
SET stock = stock - 1, version = version + 1
WHERE product_id = 100 AND version = 5;
-- 影响行数 = 1 表示成功，= 0 表示版本已变，重试
```

**方式二：CAS**

```sql
UPDATE inventory
SET stock = stock - 1
WHERE product_id = 100 AND stock >= 1;
-- 如果 stock < 1，影响行数为 0，需要重试
```

**适用场景**：读多写少的场景，冲突概率低。

| 优点 | 缺点 |
|------|------|
| 无锁，性能高 | 冲突多时，重试代价大 |
| 不会死锁 | 实现稍复杂 |
| 适合最终一致性场景 | 可能需要多次重试 |

### 4.3 代价分析

乐观锁和悲观锁的选择，本质上是「锁的开销 vs 重试的开销」之间的权衡。

**悲观锁的代价**：每次操作都要竞争锁，锁竞争激烈时，大量事务等待。

**乐观锁的代价**：冲突时需要重试，重试次数过多时，整体时间可能更长。

公式化表达：

```
悲观锁总耗时 ≈ 锁等待时间 + 操作时间
乐观锁总耗时 ≈ 操作时间 × 重试次数
```

当冲突概率 `P` 较小时（`P < 1/锁等待时间`），乐观锁更快；当冲突概率 `P` 较大时，悲观锁更快。

:::tip 实际选择建议

| 场景 | 推荐策略 |
|------|----------|
| 高并发热点数据更新 | 悲观锁 + 缩短持锁时间 |
| 低并发后台任务 | 悲观锁 |
| 读多写少的数据统计 | 乐观锁 |
| 秒杀场景扣库存 | 乐观锁（但要做好限流 + 队列削峰） |

:::

## 五、面试实战

### 5.1 场景一：MVCC 原理

**面试官问**：「MySQL 的 MVCC 是怎么实现的？RC 和 RR 的区别是什么？」

这道题考察的是对 MVCC 核心机制的理解。

**核心回答框架**：

1. **隐藏字段**：每行数据有三个隐藏字段 `DB_TRX_ID`、`DB_ROLL_PTR`、`DB_ROW_ID`，用于构建版本链。
2. **undo log 版本链**：每次修改都会将旧数据写入 undo log，形成版本链，`DB_ROLL_PTR` 指向历史版本。
3. **ReadView**：事务开始时生成 ReadView，包含 `m_ids`（活跃事务列表）、`min_trx_id`、`max_trx_id`。
4. **可见性判断**：通过 ReadView 判断每个版本的 `DB_TRX_ID` 是否可见。
5. **RC vs RR**：RC 每次读取都重新生成 ReadView，RR 在事务开始时生成一次 ReadView。

**加分回答**：

> 补充一点，MVCC 主要作用于快照读。当前读（如 `SELECT FOR UPDATE`）是通过 Next-Key Lock 来保证并发安全的，不是靠 MVCC。

### 5.2 场景二：Next-Key Lock 与幻读

**面试官问**：「什么情况下会产生间隙锁？Next-Key Lock 是怎么解决幻读的？」

**核心回答框架**：

1. **三种锁的概念**：Record Lock 锁记录，Gap Lock 锁间隙，Next-Key Lock = 两者之和。
2. **产生间隙锁的场景**：非唯一索引的等值查询、范围查询都会加 Gap Lock；唯一索引的等值查询只加 Record Lock。
3. **幻读解决**：RR 级别下，范围查询会对区间加 Next-Key Lock，阻止其他事务在区间内插入新记录。
4. **索引失效风险**：查询条件导致索引失效时，锁会升级为表锁。

**加分回答**：

> 如果面试官追问，可以补充：间隙锁在遇到不存在的记录时会扩大范围。例如 `SELECT * FROM user WHERE id = 10 FOR UPDATE`，如果 id=10 不存在，会锁住 (上一个记录, +∞) 的整个区间。

### 5.3 场景三：死锁排查

**面试官问**：「生产环境发生死锁，你怎么排查和处理？」

这道题考察的是实际问题的解决能力。

**排查步骤**：

```sql
-- 1. 查看死锁日志
SHOW ENGINE INNODB STATUS;

-- 死锁日志示例：
-- TRANSACTION 1 (locker A), OS lock info: ...
--   TABLE LOCK table `test`.`account` trx id 123 lock_mode X
--   RECORD LOCKS space id 1 page no 3 n bits 4 index PRIMARY
--   Record lock, heap no 3 PHYSICAL RECORD: ...
-- TRANSACTION 2 (locker B), OS lock info: ...
--   TABLE LOCK table `test`.`account` trx id 124 lock_mode X
```

**处理流程**：

1. 分析死锁日志，找出两个事务各自锁住了什么
2. 追溯业务逻辑，判断是代码逻辑问题还是 SQL 写法问题
3. 修复代码（例如调整为固定顺序获取锁）
4. 上线后观察是否还有死锁

**预防措施**：

1. 按固定顺序访问资源
2. 减少事务持锁时间
3. 合理设计索引，避免锁升级
4. 高并发场景下考虑关闭死锁检测

## 总结

MySQL 的并发控制，是 MVCC 和锁的协同工作：

- **MVCC** 解决的是「快照读」的并发问题——读不加锁，写不加锁，通过版本链和 ReadView 实现读写分离。
- **锁** 解决的是「当前读」的并发问题——通过 Record Lock、Gap Lock、Next-Key Lock 保证当前数据的一致性。
- **死锁** 是并发控制的副产物——通过死锁检测和回滚机制处理，但更重要的是预防。
- **乐观锁 vs 悲观锁** 是两种并发策略——没有绝对的好坏，只有场景的适配。

理解了这些，你就能回答 MySQL 并发控制的大部分面试问题——不只是「是什么」，更要知道「为什么这样设计」。
