# 线程池原理深度剖析：参数配置与调优

凌晨两点，你被一条告警叫醒：「服务拒绝请求」。登录服务器一看，线程池队列堆积了上万个任务，CPU 使用率却只有 30%——线程池用错了一个参数，整条业务线就崩了。

这不是故事，而是真实的生产事故。

线程池是 Java 并发编程中最核心的组件之一，也是面试中考察频率最高的知识点之一。大多数候选人能说出线程池的参数，但在实际场景中如何配置、如何调优，很多人的理解就模糊了。

这篇文章，帮你把线程池从「会用」提升到「理解原理」。

## 一、线程池的七个核心参数

ThreadPoolExecutor 是 Java 线程池的核心实现，理解线程池的第一步，就是理解它的七个核心参数：

```java
public ThreadPoolExecutor(
    int corePoolSize,          // 核心线程数
    int maximumPoolSize,       // 最大线程数
    long keepAliveTime,         // 空闲线程存活时间
    TimeUnit unit,              // 时间单位
    BlockingQueue<Runnable> workQueue,  // 任务队列
    ThreadFactory threadFactory,// 线程工厂
    RejectedExecutionHandler handler     // 拒绝策略
)
```

### 1.1 corePoolSize：核心线程数

核心线程是线程池的「常驻兵力」。即使没有任务执行，核心线程也不会被销毁（除非设置了 allowCoreThreadTimeOut）。

**核心线程数的配置思路**：

- CPU 密集型任务：corePoolSize = CPU 核心数 + 1
- IO 密集型任务：corePoolSize = CPU 核心数 × 期望 CPU 利用率 × (1 + IO 耗时 / CPU 耗时)

### 1.2 maximumPoolSize：最大线程数

最大线程数是线程池的「兵力上限」。当核心线程都在忙碌、队列已满时，线程池会创建临时线程来处理任务。临时线程用完后，如果空闲时间超过 keepAliveTime，就会被销毁。

### 1.3 keepAliveTime：空闲线程存活时间

临时线程（超过核心线程数部分的线程）在空闲超过 keepAliveTime 后会被销毁。如果设置为 0，表示临时线程用完立即销毁。

### 1.4 workQueue：任务队列

任务队列是存放等待执行任务的缓冲区。常用的队列有：

| 队列类型 | 特点 | 风险 |
|---|---|---|
| LinkedBlockingQueue | 有界/无界可选，默认 Integer.MAX_VALUE | 无界时可能导致 OOM |
| ArrayBlockingQueue | 有界，需指定容量 | 队列满时任务被拒绝 |
| SynchronousQueue | 不存储任务，直接提交给线程 | 每个任务必须有线程接收，否则拒绝 |
| DelayedWorkQueue | 定时任务专用 | — |

**无界队列的风险**：使用 LinkedBlockingQueue 而不指定容量时，队列可以无限增长。当任务生产速度大于消费速度时，队列不断堆积，最终 OOM。

### 1.5 threadFactory：线程工厂

线程工厂负责创建线程。通过自定义线程工厂，可以设置线程名称、设置守护线程、设置线程优先级等。

```java
ThreadFactory factory = r -> {
    Thread t = new Thread(r);
    t.setName("business-pool-" + t.getId());
    t.setDaemon(false);
    return t;
};
```

### 1.6 handler：拒绝策略

当线程池和队列都满时，新任务会被拒绝。JDK 内置四种拒绝策略：

| 策略 | 行为 | 适用场景 |
|---|---|---|
| AbortPolicy（默认） | 抛 RejectedExecutionException | 默认行为 |
| CallerRunsPolicy | 由调用方线程执行 | 限流保护 |
| DiscardPolicy | 静默丢弃任务 | 不关心任务丢失 |
| DiscardOldestPolicy | 丢弃队列最老的任务 | 优先处理新任务 |

## 二、线程池的工作流程

线程池接收任务后的处理流程，是理解线程池行为的关键：

```
新任务到达
    ↓
当前线程数 < corePoolSize？
    ├─ 是 → 创建新核心线程，执行任务
    └─ 否 → 进入下一步
    ↓
队列未满？
    ├─ 是 → 进入任务队列等待
    └─ 否 → 进入下一步
    ↓
当前线程数 < maximumPoolSize？
    ├─ 是 → 创建临时线程，执行任务
    └─ 否 → 执行拒绝策略
```

**一个关键的理解**：线程池不会一开始就把 corePoolSize 个线程全部创建出来。线程是按需创建的——直到有任务到达，才会创建第一个线程。

## 三、线程池的五大状态

ThreadPoolExecutor 用一个 AtomicInteger 型的 ctl 字段，高 3 位存储线程池状态，低 29 位存储工作线程数：

| 状态 | 说明 |
|---|---|
| RUNNING | 接受新任务，处理队列任务 |
| SHUTDOWN | 不接受新任务，但处理队列任务 |
| STOP | 不接受新任务，不处理队列任务，中断正在执行的任务 |
| TIDYING | 所有任务已终止，workerCount 为 0，即将调用 terminated() |
| TERMINATED | terminated() 执行完成 |

状态流转：RUNNING → SHUTDOWN → STOP → TIDYING → TERMINATED。

## 四、Executors 工厂方法的坑

JDK 提供了几个便捷的线程池创建方法，但阿里巴巴规范明确禁止在生产环境中使用：

```java
// 风险：队列无界，可能 OOM
ExecutorService fixedPool = Executors.newFixedThreadPool(10);

// 风险：maxPoolSize 无限大，高并发可能创建海量线程
ExecutorService cachedPool = Executors.newCachedThreadPool();

// 风险：队列无界
ExecutorService singlePool = Executors.newSingleThreadExecutor();
```

**为什么不能用意务 newFixedThreadPool**：

```java
// newFixedThreadPool 的源码
public static ExecutorService newFixedThreadPool(int nThreads) {
    return new ThreadPoolExecutor(
        nThreads, nThreads,
        0L, TimeUnit.MILLISECONDS,
        new LinkedBlockingQueue<Runnable>()  // 无界队列！
    );
}
```

无界队列 + 任务生产速度大于消费速度 = 队列无限增长 → OOM。

**正确做法**：

```java
// 手动创建线程池，指定有界队列
ExecutorService pool = new ThreadPoolExecutor(
    10, 20,                          // 核心10，最大20
    60L, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(1000), // 有界队列
    r -> new Thread(r, "biz-pool-" + r.hashCode()),
    new ThreadPoolExecutor.AbortPolicy()
);
```

## 五、线程池配置策略

### 5.1 CPU 密集型任务

CPU 密集型任务的特点是需要大量 CPU 计算，几乎不阻塞。

**配置思路**：CPU 密集型：核心线程数 = CPU 核心数 + 1。

**理由**：CPU 密集型任务下，线程都在执行计算，几乎没有等待时间。如果线程数大于 CPU 核心数，多出来的线程就会竞争 CPU 资源，反而降低效率。

### 5.2 IO 密集型任务

IO 密集型任务的特点是需要等待 IO 操作完成。

**配置思路**：核心线程数 = CPU 核心数 × 期望 CPU 利用率 × (1 + 平均 IO 时间 / 平均 CPU 时间)

**示例**：8 核 CPU，期望利用率 100%，每个任务的 IO 等待时间是 100ms，CPU 计算时间是 20ms：

```
corePoolSize = 8 × 1 × (1 + 100/20) = 8 × 1 × 6 = 48
```

也就是说，这种场景下，48 个线程才能充分利用 CPU。

### 5.3 混合型任务

如果一个系统同时包含 CPU 密集型和 IO 密集型任务，建议分开创建两个线程池。

## 六、线程池监控

线程池的监控是生产环境运维的核心：

```java
// 获取当前活跃线程数
pool.getActiveCount();

// 获取当前线程数
pool.getPoolSize();

// 获取队列大小
pool.getQueue().size();

// 获取已完成任务数
pool.getCompletedTaskCount();
```

## 七、优雅关闭线程池

线程池的关闭需要优雅，防止任务丢失：

```java
// 停止接收新任务，但执行队列中的任务
pool.shutdown();

// 停止接收新任务，尝试中断正在执行的任务
List<Runnable> tasks = pool.shutdownNow();

// 等待所有任务完成
if (!pool.awaitTermination(60, TimeUnit.SECONDS)) {
    pool.shutdownNow();
}
```

## 八、面试实战

### 场景一：线程池参数与工作流程

**面试官问**：「线程池有哪些参数？线程池的工作流程是什么？」

**深入回答**：

> 线程池有七个核心参数：corePoolSize（核心线程数）、maximumPoolSize（最大线程数）、keepAliveTime（空闲线程存活时间）、workQueue（任务队列）、threadFactory（线程工厂）、handler（拒绝策略）。
>
> 工作流程分三步判断：
> 第一，判断当前线程数是否小于 corePoolSize，如果是则创建核心线程执行任务。
> 第二，如果核心线程满了，则把任务放入队列。
> 第三，如果队列满了，再判断当前线程数是否小于 maximumPoolSize，如果是则创建临时线程执行任务。
> 第四，如果线程数也达到上限，则执行拒绝策略。

### 场景二：CPU 密集型和 IO 密集型

**面试官问**：「CPU 密集型和 IO 密集型线程池分别怎么配置？」

**深入回答**：

> CPU 密集型任务因为几乎不阻塞，线程数 = CPU 核心数 + 1。
>
> IO 密集型任务因为需要等待 IO，线程数 = CPU 核心数 × (1 + IO耗时/CPU耗时)。

### 场景三：队列的选择

**面试官问**：「线程池的队列有哪些类型？各自的特点是什么？」

**深入回答**：

> 常用有三类：LinkedBlockingQueue（有界/无界）、ArrayBlockingQueue（有界）、SynchronousQueue（不存储任务）。
>
> LinkedBlockingQueue 不指定容量时默认无界，风险最大。ArrayBlockingQueue 是有界队列。SynchronousQueue 不存储任务，必须立即交付。

## 总结

线程池是 Java 并发编程的核心组件。理解七个参数的含义和工作流程，是正确使用线程池的基础。

**核心记忆点**：

- 线程池优先创建核心线程，再使用队列，最后才创建临时线程
- CPU 密集型：核心线程数 = CPU 核心数 + 1
- IO 密集型：核心线程数 = CPU 核心数 × (1 + IO耗时/CPU耗时)
- 不要使用 Executors 创建线程池
- 队列必须有界，拒绝策略必须有处理逻辑
