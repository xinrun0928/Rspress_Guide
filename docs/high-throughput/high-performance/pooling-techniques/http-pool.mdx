# HTTP 连接池：连接复用与协议演进

你知道吗，HTTP 请求中最慢的部分，往往不是业务处理，而是网络传输。

建立一个 HTTP 连接需要 DNS 解析、TCP 三次握手、TLS 握手（如果是 HTTPS），整个过程可能需要几十毫秒。如果每个请求都新建连接，高并发下光是建立连接就能让系统崩溃。

HTTP 连接池要解决的，就是这个问题。

## 一、HTTP 连接复用的三层架构

HTTP 连接复用不是单一技术，而是跨越了网络协议栈的多个层次。

### 1.1 TCP 层面的复用：Keep-Alive

HTTP/1.0 时代，每个 HTTP 请求都要建立一个新的 TCP 连接。HTTP/1.0 引入了 Connection: Keep-Alive，请求头声明希望复用连接：

```
Connection: keep-alive
Keep-Alive: timeout=5, max=100
```

服务端返回同样的头，表示同意复用。TCP 连接在发送完响应后不会立即关闭，而是保持一段时间，等待后续请求。

### 1.2 HTTP 层面的复用：持久连接

HTTP/1.1 做出了一个重要改变：**默认开启持久连接**。只要客户端或服务端没有显式关闭连接，TCP 连接就会保持复用。

### 1.3 应用层面的复用：连接池

应用层连接池在 HTTP 持久连接的基础上，进一步确保连接的可靠性和可管理性。

## 二、HTTP/1.1 的队头阻塞问题

HTTP/1.1 虽然支持持久连接，但有一个严重的问题——**队头阻塞（Head-of-Line Blocking）**。

在同一个 TCP 连接上，HTTP 请求必须**串行发送和接收**：

```
请求 A (耗时 1000ms) ──────────────────────────────────────→
请求 B (耗时 50ms)                   ──────────────────>
```

B 和 C 明明可以很快完成，却要等待 A 全部完成才能开始。这就是队头阻塞。

**解决方案**：多开几个连接（但浪费资源，且受浏览器限制每个域名最多 6 个连接）。

## 三、HTTP/2 多路复用

HTTP/2 引入了**多路复用（Multiplexing）**技术，彻底解决了 HTTP/1.1 的队头阻塞问题：

```
TCP 连接（一个）
  ├── Stream 1: 请求 A ──────────────────> 响应 A
  ├── Stream 2: 请求 B ──> 响应 B
  └── Stream 3: 请求 C ────> 响应 C
```

B 和 C 不需要等待 A 完成，可以在 A 传输的过程中就开始。

HTTP/2 还引入了 **HPACK 头部压缩**，减少重复 Header 的传输。

## 四、HTTP/3 与 QUIC

HTTP/3 基于 QUIC 协议（运行在 UDP 上），彻底解决了 TCP 队头阻塞问题：

- **无 TCP 队头阻塞**：QUIC 的每个 Stream 独立可靠传输
- **0-RTT 握手**：第二次连接时可直接发送数据
- **连接迁移**：移动网络切换 IP 时连接不断

## 五、应用层连接池配置

### 5.1 OkHttp 连接池

```java
ConnectionPool pool = new ConnectionPool(
    5,      // 最大空闲连接数
    5,      // 空闲连接存活时间（分钟）
    TimeUnit.MINUTES
);

OkHttpClient client = new OkHttpClient.Builder()
    .connectionPool(pool)
    .connectTimeout(10, TimeUnit.SECONDS)
    .readTimeout(30, TimeUnit.SECONDS)
    .writeTimeout(30, TimeUnit.SECONDS)
    .build();
```

### 5.2 Apache HttpClient 连接池

```java
PoolingHttpClientConnectionManager poolManager = new PoolingHttpClientConnectionManager();
poolManager.setMaxTotal(100);           // 最大连接数
poolManager.setDefaultMaxPerRoute(20);  // 每个路由最大连接数

CloseableHttpClient client = HttpClients.custom()
    .setConnectionManager(poolManager)
    .build();
```

## 六、连接池调优思路

### 6.1 连接数计算

HTTP 连接池的连接数，取决于下游服务的处理能力：

```
最大连接数 = 下游服务处理能力 / 单连接处理能力
```

### 6.2 perRoute 参数的重要性

maxPerRoute（每个路由最大连接数）是容易被忽视但非常重要的参数。如果不设置，所有连接可能被耗尽在某一个域名上，导致其他域名无法访问。

## 思考题

### 题目一：队头阻塞的边界

HTTP/2 解决了 HTTP 层的队头阻塞，但 TCP 层仍然有队头阻塞。HTTP/3 基于 UDP，那 UDP 的队头阻塞在哪里？

### 题目二：连接池与下游能力

你的服务调用下游 API，下游 API 最多支持 200 并发。如果你的服务部署了 10 个实例，每个实例的连接池应该设多大？

---

## 总结

**核心记忆点**：
- HTTP/1.1 默认开启持久连接，但存在队头阻塞
- HTTP/2 通过多路复用彻底解决队头阻塞
- HTTP/3 基于 QUIC/UDP，解决 TCP 层队头阻塞
- 应用层连接池需要配置 maxTotal 和 maxPerRoute
