# 池化技术面试题总结

面试官问：「为什么要用线程池？new Thread() 不是挺好用的吗？」

你微微一笑，心想这个人可能还在用上古时代的写法。但表面上，你还是耐心地解释道：

「因为创建和销毁线程是有成本的。一个线程的创建需要分配栈空间、初始化线程局部存储、注册到 JVM 线程管理系统……这个过程大约需要消耗 1-2ms。」

「如果每个请求都新建线程，1000 QPS 的系统光是创建线程就要消耗 1-2 秒。」

面试官点了点头：「那你继续说说池化技术。」

这篇文章，帮你彻底掌握池化技术的面试要点。

## 一、为什么要用池化技术

池化技术的本质是**以空间换时间，预分配加复用**。

### 1.1 资源创建的成本

在计算机系统中，创建资源通常有以下开销：

| 资源类型 | 创建成本 | 复用价值 |
|---|---|---|
| 线程 | 1-2ms | 高（线程复用避免频繁创建） |
| 数据库连接 | 3-10ms | 极高（网络握手+认证） |
| Redis 连接 | 0.5-2ms | 高 |
| HTTP 连接 | 视情况 | 中等 |
| 内存对象 | 视大小 | 中等 |

### 1.2 池化的核心价值

```
不用池化：请求1 → 创建线程 → 执行 → 销毁线程 → 请求2 → 创建线程 → ...
使用池化：请求1 → 获取线程 → 执行 → 归还线程 → 请求2 → 获取线程 → ...
```

池化解决三个核心问题：

- **时间成本**：复用已有资源，避免重复创建
- **空间成本**：池大小有上限，防止资源耗尽
- **管理成本**：统一管理资源的生命周期

## 二、线程池面试核心问题

### 2.1 线程池有哪些核心参数？

ThreadPoolExecutor 有七个核心参数：

| 参数 | 说明 |
|---|---|
| corePoolSize | 核心线程数，池中保持的最小线程数 |
| maximumPoolSize | 最大线程数，池中允许的最大线程数 |
| keepAliveTime | 空闲线程存活时间 |
| TimeUnit | keepAliveTime 的时间单位 |
| workQueue | 任务队列，用于存放待执行的任务 |
| threadFactory | 线程工厂，用于创建新线程 |
| RejectedExecutionHandler | 拒绝策略，当线程池饱和时的处理方式 |

### 2.2 线程池的工作流程是什么？

线程池的执行流程分为三个阶段：

```
阶段一：线程数 < corePoolSize？
  → 是：创建核心线程，执行任务
  → 否：进入下一阶段

阶段二：队列未满？
  → 是：任务加入队列等待
  → 否：进入下一阶段

阶段三：线程数 < maximumPoolSize？
  → 是：创建临时线程，执行任务
  → 否：执行拒绝策略
```

:::tip
关键点：线程池不会一开始就创建 corePoolSize 个线程，而是按需创建。除非调用 `prestartAllCoreThreads()` 预创建。
:::

### 2.3 线程数量如何设置？

这是面试必问题。

**CPU 密集型任务**：大量计算、很少等待

```java
// 推荐公式
线程数 = CPU 核心数 + 1
```

原因：CPU 密集型任务几乎不阻塞，线程数等于 CPU 核心数即可充分利用 CPU。加 1 是为了处理短暂的线程中断。

**IO 密集型任务**：数据库查询、网络请求、文件读写

```java
// 推荐公式
线程数 = CPU 核心数 × CPU 利用率 × (1 + IO耗时/CPU耗时)
```

举例：4 核 CPU，期望 CPU 利用率 100%，每个任务 IO 等待 100ms、CPU 计算 20ms：

```
线程数 = 4 × 1 × (1 + 100/20) = 4 × 6 = 24
```

**通用公式**：

```
最佳线程数 = (线程等待时间 / 线程CPU时间 + 1) × CPU核心数
```

### 2.4 线程池的队列类型有哪些？

| 队列类型 | 特点 | 风险 |
|---|---|---|
| LinkedBlockingQueue | 可配置容量，默认无界 | 无界队列可能导致 OOM |
| ArrayBlockingQueue | 必须指定容量，有界 | 安全 |
| SynchronousQueue | 不存储任务，每个 put 必须等 take | 任务过多会快速触发拒绝策略 |
| DelayQueue | 延迟执行的任务队列 | 需要实现 Delayed 接口 |

:::warning
生产环境中，LinkedBlockingQueue 必须指定容量。使用 `Executors.newFixedThreadPool()` 存在 OOM 风险。
:::

### 2.5 线程池的拒绝策略

当线程池和队列都满了，新的任务会被触发拒绝策略：

| 策略 | 行为 | 适用场景 |
|---|---|---|
| AbortPolicy | 抛异常 | 默认，需要显式处理 |
| CallerRunsPolicy | 由调用方执行 | 限流保护 |
| DiscardPolicy | 丢弃任务 | 不关心任务丢失 |
| DiscardOldestPolicy | 丢弃最老的 | 优先处理新任务 |

## 三、数据库连接池核心问题

### 3.1 连接池有哪些核心参数？

| 参数 | 说明 | 建议值 |
|---|---|---|
| minimumIdle | 最小空闲连接数 | 与核心线程数匹配 |
| maximumPoolSize | 最大连接数 | 10-50，通常不超过 100 |
| connectionTimeout | 获取连接的最大等待时间 | 30000ms |
| idleTimeout | 空闲连接的最大存活时间 | 600000ms（10min） |
| maxLifetime | 连接的最大生命周期 | 1800000ms（30min） |

### 3.2 连接数如何配置？

数据库连接数的计算公式：

```
最大连接数 = (核心线程数 × 每线程每秒请求数 × 平均RT) / 数据库每秒处理能力
```

举例：核心线程数 20，每秒 1000 请求，平均 RT 50ms，MySQL 处理能力 10000 QPS：

```
最大连接数 = (20 × 1000 × 0.05) / 10000 = 1
```

实际应用中，10-50 通常够用。

:::warning
连接池不是越大越好。数据库连接数有限，连接池太大会因为争抢连接而降低性能。
:::

### 3.3 连接泄漏如何排查？

连接泄漏的原因：获取连接后未正确归还。

**排查方法**：

1. 开启 HikariCP 的泄漏检测：
```yaml
spring:
  datasource:
    hikari:
      leak-detection-threshold: 60000  # 60秒未归还则警告
```

2. 检查代码是否使用了 try-with-resources：
```java
// 错误写法
public void query() {
    Connection conn = dataSource.getConnection();
    if (someCondition) {
        throw new RuntimeException("error");
    }
    conn.close();
}

// 正确写法
public void query() {
    try (Connection conn = dataSource.getConnection()) {
        // 使用连接
    } // 自动关闭
}
```

3. 监控活跃连接数是否持续增长

### 3.4 主流连接池对比

| 连接池 | 性能 | 功能 | 推荐场景 |
|---|---|---|---|
| HikariCP | 最快 | 基础 | 高性能要求（推荐） |
| Druid | 中等 | 最全（监控、SQL防火墙） | 企业级应用 |
| c3p0/DBCP | 较慢 | 一般 | 历史遗留系统 |

HikariCP 性能最优的原因：
- FastList 替代 ArrayList，从后往前遍历更快
- 字节码生成替代反射调用
- 无等待设计，使用 CAS 替代锁

## 四、Redis 连接池核心问题

### 4.1 主流客户端对比

| 客户端 | 线程安全 | 异步支持 | 推荐场景 |
|---|---|---|---|
| Jedis | 否 | 否 | 简单场景 |
| Lettuce | 是 | 是 | 高性能场景（推荐） |
| Redisson | 是 | 是 | 分布式特性（分布式锁等） |

Spring Data Redis 2.0+ 默认使用 Lettuce。

### 4.2 连接池配置

```yaml
spring:
  redis:
    lettuce:
      pool:
        max-active: 50
        max-idle: 20
        min-idle: 5
        max-wait: 3000ms
```

### 4.3 连接数计算

```
最大连接数 = 线程池核心线程数 × (1 + IO等待时间/处理时间)
```

例如：线程池核心线程数 20，每个任务 Redis 操作耗时 10ms、CPU 计算 5ms：

```
最大连接数 = 20 × (1 + 10/5) = 20 × 3 = 60
```

## 五、池化技术的通用设计思想

所有池化技术（线程池、连接池、内存池）都遵循相同的设计模式：

### 5.1 预创建与按需创建

- **预创建**：启动时创建 minSize 个资源，保证基本并发能力
- **按需创建**：收到请求时才创建新资源，避免资源浪费

### 5.2 空闲回收机制

- **idleTimeout**：空闲多久后回收多余资源
- **maxLifetime**：资源最大生命周期，防止资源老化

### 5.3 容量保护

- **有界队列**：防止无限堆积
- **拒绝策略**：资源耗尽时的兜底处理

### 5.4 验证与健康检查

| 验证时机 | 参数 | 性能影响 |
|---|---|---|
| 取出时验证 | testOnBorrow | 高（每次获取都验证）|
| 归还时验证 | testOnReturn | 中 |
| 空闲时验证 | testWhileIdle | 低（后台定期）|

**最佳实践**：testOnBorrow=false（性能考虑）、testWhileIdle=true（平衡可靠性与性能）。

## 六、常见面试追问方向

### 线程池追问

**「线程池的线程如何回收？核心线程也会被回收吗？」**

默认情况下，核心线程不会超时退出。如果调用 `allowCoreThreadTimeOut(true)`，核心线程也会在空闲时被回收。

**「如果一个任务执行时间很长，会影响其他任务吗？」**

会。同一线程执行长时间任务，会导致队列积压。可以使用熔断器模式或隔离池处理。

**「ForkJoinPool 和 ThreadPoolExecutor 的区别？」**

ForkJoinPool 适合分治任务（如归并排序），使用工作窃取算法；ThreadPoolExecutor 适合普通并发任务。

### 连接池追问

**「连接泄漏怎么排查？」**

1. 开启泄漏检测（leak-detection-threshold）
2. 检查代码是否正确关闭连接
3. 监控活跃连接数是否持续增长

**「如何判断连接池大小是否合适？」**

观察 activeConnections 是否经常接近 maximumPoolSize，threadsAwaitingConnection 是否持续大于 0。

### 池化通用追问

**「池化技术有什么代价？」**

- 内存占用：池需要预分配并保持资源
- 调优困难：池太小会限流，太大会浪费资源
- 资源老化：池中的资源可能失效

**「如何监控池化资源？」**

三个核心指标：活跃数、等待数、拒绝次数。

---

## 总结

池化技术是高性能系统的基础设施。面试中，最重要的不是背参数名称，而是理解：

1. **为什么需要池化**：复用资源，避免创建开销
2. **池化参数的含义**：容量参数、时间参数、验证参数
3. **如何配置池化参数**：根据业务场景和资源能力计算
4. **如何排查问题**：泄漏检测、监控指标

记住核心思想——**以空间换时间，预分配加复用**——你就能以不变应万变地应对所有池化相关的问题。
