# Redis 连接池：性能优化的关键环节

Redis 是高并发系统中不可或缺的组件。从缓存数据到分布式锁，从计数器到消息队列，Redis 几乎无处不在。

但你可能没想过：Redis 连接池配置不当，可能会成为系统的性能瓶颈。

这篇文章，帮你把 Redis 连接池从「能用」提升到「用好」。

## 一、为什么需要 Redis 连接池

### 1.1 Redis 连接的成本

建立一次 Redis 连接也有显著的成本：

1. **TCP 连接建立**：约 0.5-1ms
2. **Redis AUTH 认证**（如果有密码）：约 0.1-0.5ms

对于一个要求 RT 小于 10ms 的高并发接口，如果每次 Redis 操作都新建连接，光建立连接就占用了 10% 的时间预算。

### 1.2 连接池的核心价值

Redis 连接池的核心价值有两点：

1. **节省连接建立开销**：复用已建立的连接
2. **控制连接数量**：保护 Redis 服务端不被耗尽

## 二、主流客户端对比

### 2.1 Jedis

Jedis 是最早的 Java Redis 客户端之一。特点是简单易用，但**线程不安全**。

```java
// Jedis 每次操作都需要获取和归还连接
Jedis jedis = jedisPool.getResource();
try {
    jedis.set("key", "value");
} finally {
    jedis.close();
}
```

使用 Jedis 时，必须在每次操作后 close() 归还连接。

### 2.2 Lettuce

Lettuce 是目前推荐使用的 Redis 客户端。核心优势是**支持异步和响应式**，并且**线程安全**。

```java
// Lettuce：同一个连接可以跨线程使用
StatefulRedisConnection<String, String> connection = client.connect();
RedisCommands<String, String> commands = connection.sync();
```

Lettuce 基于 Netty 实现，内部管理连接池，性能优秀。Spring Data Redis 2.0+ 默认使用 Lettuce。

### 2.3 Redisson

Redisson 提供了很多开箱即用的分布式数据结构，如分布式锁、分布式集合等。

### 2.4 客户端对比

| 客户端 | 线程安全 | 异步支持 | 适用场景 |
|---|---|---|---|
| Jedis | 否 | 否 | 简单场景 |
| Lettuce | 是 | 是 | 高性能场景（推荐） |
| Redisson | 是 | 是 | 分布式特性 |

## 三、连接池核心参数

Lettuce 连接池的核心参数：

| 参数 | 说明 | 建议值 |
|---|---|---|
| max-active | 最大连接数 | 根据并发量配置，通常 20-50 |
| max-idle | 最大空闲连接数 | 与平均并发量匹配 |
| min-idle | 最小空闲连接数 | 保证基本并发能力 |
| max-wait | 获取连接最大等待时间 | 3000ms |
| test-while-idle | 空闲时测试连接 | true |

### 3.1 Spring Boot 配置示例

```yaml
spring:
  redis:
    lettuce:
      pool:
        max-active: 50
        max-idle: 20
        min-idle: 5
        max-wait: 3000ms
        test-while-idle: true
```

### 3.2 连接数的计算

Redis 连接数的计算公式和数据库连接池类似：

```
最大连接数 = 线程池核心线程数 × (1 + IO等待时间/处理时间)
```

## 四、Pipeline：批量操作的艺术

### 4.1 普通模式的问题

普通模式下，每次 Redis 操作都是一次网络往返（RTT）：

```java
redis.set("k1", "v1");  // RTT1
redis.set("k2", "v2");  // RTT2
redis.set("k3", "v3");  // RTT3
```

三次 SET 操作，需要三次网络往返。如果 RTT 是 1ms，光网络开销就是 3ms。

### 4.2 Pipeline 的原理

Pipeline 将多个命令打包，一次发送，减少网络往返：

```java
RedisFuture<String> f1 = asyncCommands.set("k1", "v1");
RedisFuture<String> f2 = asyncCommands.set("k2", "v2");
RedisFuture<String> f3 = asyncCommands.set("k3", "v3");

String r1 = f1.get();
String r2 = f2.get();
String r3 = f3.get();
```

### 4.3 Pipeline 的注意事项

Pipeline 命令过多会有问题：客户端内存压力增大。建议每次 Pipeline 的命令数量控制在 100-500 个。

## 五、连接池调优实战

### 5.1 连接池太小

**现象**：获取连接超时，threadsAwaitingConnection 持续大于 0。

**原因**：连接池上限不足以支撑并发量。

### 5.2 连接泄漏

**现象**：活跃连接数持续增长，最终达到 max-active。

**原因**：获取连接后未正确归还。Lettuce 的 Connection 必须在 finally 中 close()。

## 六、监控指标

| 指标 | 含义 | 告警阈值 |
|---|---|---|
| lettuce.pool.active | 活跃连接数 | 接近最大值 |
| lettuce.pool.idle | 空闲连接数 | 持续为 0 |
| lettuce.pool.waiters | 等待连接的线程数 | 持续大于 0 |

## 总结

Redis 连接池是连接应用层和 Redis 服务端的关键组件。

**核心记忆点**：
- 生产环境推荐使用 Lettuce（线程安全，支持异步）
- 连接数 = 线程池大小 × (1 + IO等待时间/处理时间)
- Pipeline 是减少网络往返的关键技术，但不要打包过多命令
- 开启 test-while-idle，防止脏连接
- 监控是生产环境的标配
