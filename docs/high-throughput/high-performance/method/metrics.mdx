# 性能优化度量指标：RT、QPS、TPS、并发数、吞吐率、CPU 利用率

「老板，系统有点慢，怎么办？」

这是很多工程师都听过的抱怨。怎么回答这个问题？

如果你的回答是「我加点缓存试试」，大概率会被追问「加多少？加在哪里？怎么验证效果？」答不上来，优化就是盲人摸象。

如果你的回答是「当前 P99 RT 是 800ms，目标是降到 500ms以内，需要优化 Redis 缓存层，预计可以提升 30% 的 QPS」，那老板至少知道你是有方向、有目标的。

区别在哪？在于你能不能用**数据**说话。

性能优化的第一步永远是「定义清楚指标」。没有指标，就不知道目标是什么；不知道目标是什么，就不知道优化是否成功。这篇文章，把性能优化中最核心的指标讲清楚。

## 一、RT 与分位数：用户感知到的「快」

### 1.1 什么是 RT

RT（Response Time，响应时间）是用户点击操作到收到结果的时间。在 Web 服务中，就是从发起 HTTP 请求到收到响应的时间。

对于前端页面，RT 可能是「白屏时间」「首屏时间」「可交互时间」。对于后端接口，RT 就是从接收到返回的总耗时。

RT 是用户最能感知的指标。用户不关心你的 CPU 利用率多少，不关心你的数据库索引建了几层，只关心「我点的这个按钮，多久能出结果」。

### 1.2 平均值为什么不够用

说起响应时间，很多人第一反应是「平均值」。平均值当然有用，但远远不够。

看一个例子：10000 个请求，9990 个在 10ms 内返回，10 个因为 GC 停顿等了 10 秒。平均值可能是 20ms，看起来还不错。但 10 个用户经历了 10 秒的等待，这在生产环境中是不可接受的。

所以，平均值会掩盖长尾问题。

### 1.3 分位数才是关键

分位数（Percentile）是解决「平均值陷阱」的关键。常见的分位数有：

| 分位数 | 含义 | 用途 |
|--------|------|------|
| P50（中位数） | 50% 的请求在这个时间内完成 | 反映典型用户的体验 |
| P90 | 90% 的请求在这个时间内完成 | 反映大多数用户的最差体验 |
| P95 | 95% 的请求在这个时间内完成 | SLA 常用指标 |
| P99 | 99% 的请求在这个时间内完成 | 反映长尾用户体验 |
| P999 | 99.9% 的请求在这个时间内完成 | 对长尾要求极高的场景 |

为什么要关心 P99 而不是 P50？因为在互联网产品中，1% 的用户（甚至 0.1%）的体验同样重要。一个用户等了 10 秒，可能就流失了，而他在你的指标里只是「0.1% 的异常值」。

:::tip P99 与 SLA

大多数互联网公司的 SLA 承诺是「P99 < Xms」。比如某云服务承诺 P99 < 500ms，意味着 99% 的请求必须在 500ms 内完成。这是可量化的服务承诺，而不是「系统比较快」这种主观描述。

:::

### 1.4 TP999 与最大响应时间

TP999（Tail Percentile 999）是 P999 的另一种说法，指最慢 0.1% 请求的响应时间。

还有最大响应时间（Max RT），指所有请求中最慢的那一个。最大响应时间往往比 P999 更大，因为它只受单次异常影响。最大响应时间高不一定代表系统有问题，但如果持续高位，说明存在未被捕获的异常场景。

**实战中，通常关注 P95 和 P99**，因为这两个指标既能反映长尾问题，又不会因为单次异常波动过大。

## 二、QPS 与 TPS：系统承载了多少请求

### 2.1 QPS：每秒查询数

QPS（Queries Per Second）是系统每秒处理的请求数量。对于只读接口，QPS 就是系统每秒处理的查询数量。

一个接口 QPS 是 1000，意味着每秒有 1000 个请求进来。这是衡量系统吞吐量的核心指标。

QPS 的计算公式是：

```
QPS = 并发数 / 平均响应时间
```

举例：并发数是 100，平均 RT 是 100ms（0.1 秒），QPS = 100 / 0.1 = 1000。

这个公式告诉我们：QPS 和 RT 是此消彼长的关系。在并发数不变的情况下，降低 RT 可以提升 QPS；反之亦然。

### 2.2 TPS：每秒事务数

TPS（Transactions Per Second）是系统每秒完成的事务数量。**事务（Transaction）是一个完整的业务操作，可能包含多个步骤。**

举几个例子：

- 用户下单是一个事务，包含库存扣减、订单创建、支付扣款等多个步骤
- 转账是一个事务，包含扣款、更新余额、入账等多个步骤
- 一个简单的查询接口，请求即事务

QPS 和 TPS 的区别在于：**QPS 衡量的是单个请求的吞吐量，TPS 衡量的是业务层面的吞吐量。** 对于简单的查询服务，QPS ≈ TPS；对于复杂的事务系统，TPS < QPS，因为一个事务包含多个请求。

### 2.3 峰值 QPS 与日常 QPS

系统在不同时间段的 QPS 差异很大。以电商系统为例：

- 日常时段 QPS：1000
- 促销开始瞬间峰值 QPS：50000
- 促销平稳期 QPS：20000

峰值 QPS 是系统设计的重要依据。系统必须能够扛得住峰值流量，否则就会出现服务不可用。

但峰值流量持续时间往往很短。比如大促抢购的高峰期可能只有几分钟到几十分钟。如果按峰值 QPS 来设计系统容量，成本会非常高。所以很多系统会采用**弹性伸缩**的策略：平时用小规模，峰值时自动扩容。

### 2.4 如何计算系统容量

根据业务目标反推需要的系统容量，是一个常见的场景。

**例题**：系统目标是 P99 RT < 200ms，支持 10000 QPS，问需要多少并发线程数？

根据公式：`QPS = 并发数 / 平均 RT`

假设平均 RT = 100ms（考虑 P99 是 200ms，平均可能更低），则：

```
并发数 = QPS × 平均 RT = 10000 × 0.1s = 1000
```

考虑到线程有阻塞时间，实际需要的线程数会更少。这个计算是容量规划的基础。

## 三、并发数：系统同时在处理多少请求

### 3.1 什么是并发数

并发数（Concurrency）是指系统在同一时刻正在处理的请求数量。不是「每秒来了多少请求」，而是「同时有多少请求在处理」。

举个例子：

- 一秒内来了 1000 个请求
- 每个请求处理时间是 100ms
- 如果系统只有 1 个线程，同时只能处理 1 个请求，其他 999 个在排队
- 那并发数就是 1（同一时刻只有 1 个在处理）
- 但实际上，如果系统有 100 个线程，100 个请求同时被处理，并发数就是 100

### 3.2 并发数与 RT 的关系

并发数和 RT 的关系有一个经典公式——**Little 定律**：

```
平均并发数 = QPS × 平均 RT
```

这个公式告诉我们：在 QPS 固定的情况下，RT 越短，并发数越低；在 RT 固定的情况下，QPS 越高，并发数越高。

为什么这很重要？因为系统的并发承载能力是有限的。如果并发数超过了系统的最大处理能力，请求就会排队等待，导致 RT 飙升。

### 3.3 线程数与并发数

Java 服务中，并发数通常受线程数限制。线程池大小是影响并发数的关键因素。

```
最大并发数 ≈ 线程数 × (1 + 等待时间 / 业务处理时间)
```

对于 IO 密集型任务（等待时间长），少量线程可以支撑高并发。对于 CPU 密集型任务（等待时间短），线程数和 CPU 核数相当是最优配置。

:::warning 线程数不是越大越好

线程数过多会导致上下文切换开销增加，每个线程获得的 CPU 时间片变少，反而降低整体吞吐量。线程数的设置需要根据业务类型（CPU 密集 or IO 密集）和系统资源来综合考虑。

:::

### 3.4 并发数过高的危害

并发数超过系统承载能力，会引发一系列连锁反应：

1. **RT 飙升**。请求排队等待时间增加，P99 RT 暴涨。
2. **错误率上升**。超时增多，连接池耗尽，部分请求直接失败。
3. **系统雪崩**。下游服务被拖垮，引发整个链路的崩溃。

这就是为什么限流和熔断如此重要——它们的作用是控制并发数，防止系统过载。

## 四、吞吐率：系统每秒能处理多少数据

### 4.1 吞吐率的定义

吞吐率（Throughput）是指单位时间内系统处理的数据量，通常用「MB/s」或「GB/s」表示。

与 QPS 的区别在于：QPS 衡量的是请求数量，吞吐率衡量的是数据量。

举例：

- 一个文件上传接口，QPS 可能只有 10（每秒 10 个文件），但吞吐率可能是 100MB/s（每秒处理 100MB 数据）
- 一个批量查询接口，QPS 可能很高（每秒 10000 次），但吞吐率可能很低（每次只返回几字节）

### 4.2 吞吐率的应用场景

吞吐率在以下场景特别重要：

**网络传输场景**。服务器之间的数据同步、缓存穿透导致的大量数据传输。吞吐率决定了网络带宽的利用率。

**磁盘 IO 场景**。日志写入、数据库批量导入、大文件处理。磁盘 IO 吞吐率决定了这些操作的速度上限。

**内存处理场景**。大数据量的内存计算、缓存预热。内存带宽决定了数据处理的速度。

### 4.3 吞吐率与 RT 的取舍

有时需要在吞吐率和 RT 之间做取舍。

举例：批量写入数据库。

- 逐条写入：每次 RT 短（10ms），但吞吐率低（每秒 100 条）
- 批量写入：每次 RT 长（100ms），但吞吐率高（每秒 10000 条）

选择哪种方式，取决于业务需求。如果需要实时生效，用逐条写入；如果可以容忍一定延迟，批量写入效率更高。

## 五、CPU 利用率：CPU 有多忙

### 5.1 什么是 CPU 利用率

CPU 利用率（CPU Utilization）是 CPU 执行有用工作的时间占比。100% 表示 CPU 满负荷运行，没有空闲。

CPU 利用率是衡量系统资源消耗的基础指标。CPU 利用率高，说明系统在拼命计算；CPU 利用率低，可能是系统在等待 IO 或者真的没有负载。

### 5.2 CPU 利用率的类型

CPU 利用率通常分为三类：

| 类型 | 含义 |
|------|------|
| User% | CPU 执行用户代码的时间占比（正常业务逻辑） |
| Sys% | CPU 执行内核代码的时间占比（系统调用、IO 操作） |
| Idle% | CPU 空闲的时间占比 |

`User% + Sys% + Idle% = 100%`。

如果 User% 很高，说明业务逻辑在消耗 CPU，常见于计算密集型场景。如果 Sys% 很高，说明系统在频繁做 IO 操作或者系统调用。

### 5.3 CPU 利用率高的原因分析

CPU 利用率高不一定是问题，也可能是系统在高效运行。关键在于：CPU 高是否导致了性能问题（RT 上升、QPS 下降）。

CPU 利用率高的常见原因：

1. **GC 频繁**。垃圾回收会占用大量 CPU 时间。
2. **死循环或密集计算**。代码中存在无限循环或者低效算法。
3. **高并发请求**。请求量大导致 CPU 满负荷，这是正常现象。
4. **频繁的上下文切换**。线程数过多导致 CPU 时间浪费在切换上。

:::tip CPU 利用率 100% 不一定是坏事

如果系统的 QPS 也在预期范围内，CPU 利用率 100% 反而说明资源被充分利用。但如果 CPU 100% 但 QPS 下降，说明存在效率问题。

:::

### 5.4 CPU 使用率与核心数

单核 CPU 100% 使用率，在 8 核服务器上只相当于 12.5% 的总体使用率。所以看 CPU 使用率时，需要区分「单核」和「多核平均值」。

Linux 中：

- `top` 命令显示的是所有核心的平均值
- `top` 后按 `1` 可以看到每个核心的详情
- 多核系统中，如果有 1 个核心 100%，其他核心空闲，平均利用率可能只有 12.5%

## 六、错误率与可用性：系统健康吗

### 6.1 错误率的定义

错误率（Error Rate）是失败请求占总请求的比例。计算公式：

```
错误率 = 失败请求数 / 总请求数 × 100%
```

通常统计的是 HTTP 5xx 错误、接口超时、下游服务调用失败等情况。

### 6.2 错误率的黄金标准

互联网公司通常用「4个9」「5个9」来衡量可用性：

| 可用性 | 每天宕机时间 | 每年宕机时间 |
|--------|------------|-------------|
| 99% | 14.4 分钟 | 3.65 天 |
| 99.9% | 1.44 分钟 | 8.76 小时 |
| 99.99% | 8.64 秒 | 52.6 分钟 |
| 99.999% | 0.86 秒 | 5.26 分钟 |

对于核心链路，错误率通常要求低于 0.1%（P99 正常的情况下）。如果接口超时或错误率飙升，说明系统处于不健康状态，需要立即处理。

### 6.3 RT 与错误率的平衡

RT 和错误率有时候是矛盾的。降低超时阈值可以快速失败，但会增加错误率；提高超时阈值可以减少错误，但会拖慢 RT。

正确的做法是：**根据业务需求设定合理的 SLA，在 SLA 范围内追求最优 RT。** 而不是一味追求 RT 最低或者错误率最低。

## 七、核心指标的综合应用

### 7.1 指标之间的数学关系

几个核心指标之间存在确定的数学关系：

```
QPS = 并发数 / 平均 RT
并发数 = QPS × 平均 RT
最大 QPS = 最大并发数 / 平均 RT
```

这些公式是容量规划的基础。通过已知的指标，可以推算出系统在不同条件下的表现。

### 7.2 一个综合案例

假设一个系统的性能数据如下：

- 当前 QPS：5000
- 平均 RT：50ms
- P99 RT：200ms
- 错误率：0.1%
- CPU 利用率：60%

分析：

1. **当前并发数** = 5000 × 0.05 = 250。系统同时有 250 个请求在处理。
2. **RT 是否健康**？P99 RT 是平均 RT 的 4 倍，说明有长尾问题。需要进一步分析耗时分布。
3. **CPU 是否成为瓶颈**？60% 的利用率说明还有余量，短期内 CPU 不会成为瓶颈。
4. **下一步优化方向**？降低 P99 RT，可能是优化数据库慢查询或缓存命中率。

## 八、面试实战

### 8.1 场景一：指标的选择

**面试官问**：「你们线上用什么指标来衡量系统性能？」

这道题考察的是对性能指标的实战理解。

**参考回答**：

> 我们主要看黄金三指标：QPS、P99 RT、错误率。
>
> QPS 反映系统的吞吐量，决定了我们能扛多少流量。
>
> P99 RT 反映用户的真实体验，因为平均值会被长尾请求拉偏，P99 更能代表大多数用户的最差体验。
>
> 错误率反映系统的健康度。再快的响应，如果大量请求失败，也是没有意义的。
>
> 此外，我们还关注 CPU 利用率、GC 频率、数据库连接池使用率等辅助指标，作为问题排查的依据。

### 8.2 场景二：P99 飙升的排查

**面试官问**：「线上 P99 RT 从 200ms 飙升到 2000ms，你会怎么排查？」

这道题考察的是指标分析和问题定位的能力。

**参考思路**：

1. **先确认指标真实性**。排除采集端的问题，确认数据没有误差。
2. **查看全链路耗时分布**。通过 APM 工具（如 SkyWalking）追踪请求，看是哪个环节耗时增加。
3. **逐层定位**。是数据库慢、缓存命中率下降、外部服务超时、还是 GC 停顿？
4. **找到具体原因后对症下药**。数据库慢就优化 SQL，缓存失效就检查过期策略，GC 问题就调整 JVM 参数。

## 九、留给你的思考题

### 题目一：指标冲突怎么办

假设你的系统在压测时发现：

- 提高并发数，QPS 上升但 P99 RT 飙升
- 降低并发数，P99 RT 改善但 QPS 下降

如果业务需求是「QPS > 10000 且 P99 < 500ms」，你会怎么取舍？需要什么样的硬件资源和代码优化才能同时满足两个目标？

### 题目二：平均值骗了你

一个接口，99% 的请求在 10ms 内返回，1% 的请求因为网络抖动等了 5000ms。平均值看起来不错，但用户体验很差。

你有什么办法让「平均值」也能反映长尾问题？如果让你设计一个告警规则，发现 P99 异常时会自动触发，你会怎么配置？

## 总结

性能优化的核心指标，说白了就几类：

- **速度**：RT、分位数（P50/P95/P99）
- **容量**：QPS、TPS、并发数、吞吐率
- **资源**：CPU 利用率、内存使用率
- **健康**：错误率、可用性

最重要的三个是 P99 RT、QPS、错误率——它们构成性能优化的黄金三角。任何一项异常，都意味着系统出现了问题。

记住：指标不是用来炫技的，是用来指导决策的。知道看什么指标，知道指标背后的含义，知道指标变化意味着什么——这才是一个工程师该有的数据思维。
