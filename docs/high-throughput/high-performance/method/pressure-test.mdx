# 全链路压测：流量构造、数据隔离、回放压测

你有没有遇到过这种情况？

压测报告很漂亮：QPS 50000，P99 RT 80ms，错误率 0%。信心满满上线，一促，崩了。数据库连接池耗尽，服务雪崩，凌晨三点爬起来紧急扩容。

问题出在哪？压测环境和生产环境的差异，可能比你想象的大得多。数据量不对——压测时只有 10 万数据，生产有 10 亿；缓存效果不一样——压测冷数据命中率高，生产热数据集中导致缓存雪崩；依赖服务不同——压测依赖的是 mock，生产调用的是真实服务。

全链路压测（Full-Link Pressure Testing）就是为了解决这些问题——在尽可能真实的生产环境下，验证系统的真实承载能力。

这篇文章，介绍全链路压测的核心方法：流量构造、数据隔离、回放压测。

---

## 一、什么是全链路压测

### 1.1 从单点压测到全链路压测

传统的压测是「单点压测」：只压测某个接口或某个服务，用 mock 数据代替真实的数据库、缓存、外部依赖。单点压测的优点是简单，缺点是——压出来的数据没有参考价值。

全链路压测是对整个请求链路的压测：从用户端到网关到服务到数据库到缓存到外部依赖，全部走真实的请求。

```
用户请求 → CDN → Nginx → 网关 → 应用服务 → Redis → MySQL → 外部服务
```

全链路压测的目标是回答两个问题：

1. **系统的极限在哪里？** 最大能扛多少 QPS？再往上会怎样（RT 飙升、错误率上升、还是直接崩溃）？
2. **系统的薄弱环节在哪里？** 在极限压力下，哪个环节最先成为瓶颈？

### 1.2 全链路压测的三个挑战

全链路压测有三个绕不开的挑战：

**挑战一：数据构造。** 生产环境有真实的用户数据、业务逻辑、数据关联。压测需要构造足够真实的数据，数据量要接近生产，数据分布要符合业务实际。

**挑战二：隔离。** 压测流量不能污染生产数据。压测产生的订单、用户操作，不能出现在真实的业务数据里。但隔离不能影响压测的真实性——不能因为隔离导致依赖服务返回异常数据。

**挑战三：回放。** 线上真实的流量是压测最好的来源。但直接用真实流量压测会有问题：用户 ID、订单号等关键数据会冲突。如何把真实流量「脱敏」后回放，是一个技术活。

## 二、流量构造：让压测数据「以假乱真」

### 2.1 数据构造的核心原则

好的压测数据必须满足三个条件：**数量级接近真实**、**分布符合业务**、**关联关系正确**。

**数量级接近真实**很好理解。如果生产有 1 亿条用户数据，你只压测了 100 万条，那缓存命中率、查询性能都会和真实情况差很远。

**分布符合业务**是关键。很多人压测时随机生成数据，忽略了真实的业务数据分布。举例：电商场景，80% 的流量集中在 20% 的热门商品上。如果压测数据是均匀分布的，测试结果会严重失真。

**关联关系正确**是容易被忽略的。一个订单关联用户、关联商品、关联库存、关联支付。如果这些关联数据不完整或不一致，测试时会出现外键约束失败、数据不一致等奇怪的问题。

### 2.2 用户数据构造

用户数据是压测的基础。构造方式有三种：

**方式一：历史数据放大。** 把生产环境的用户数据复制一份，改掉用户 ID，作为压测用户。这种方式最真实，但需要注意数据脱敏（手机号、身份证等敏感信息必须处理）。

**方式二：规则生成。** 按照用户注册的时间分布、地区分布、活跃度分布，用规则生成用户数据。优点是数据量可控，缺点是数据「太规则」，可能暴露算法。

**方式三：历史数据 + 规则生成混合。** 头部高活跃用户用真实数据，长尾用户用规则生成。兼顾真实性和数量。

```bash
# 用户数据构造示例
# 复制生产用户表，改掉手机号
INSERT INTO users_test (user_id, nickname, phone)
SELECT
    user_id + 100000000,           # 偏移量，避免和生产数据冲突
    CONCAT('test_', nickname),    # 加前缀区分
    CONCAT('138', FLOOR(RAND()*100000000))  # 生成随机手机号
FROM users
LIMIT 1000000;
```

### 2.3 业务数据构造

业务数据的构造比用户数据更复杂，因为涉及大量的关联关系。

以电商场景为例：

```
用户 → 订单 → 订单明细 → 商品 → 库存 → 评价
  ↓
收货地址、购物车、收藏、浏览历史
```

压测数据构造的关键点：

**库存数据要「刚好够用」。** 如果库存数量远超压测并发数，压测就测不出库存扣减的并发问题。通常设置库存数量为预期并发的 1.5-2 倍。

**订单数据要有关联性。** 一个用户有多个订单，订单之间有状态分布（待支付、已支付、已完成、已取消），这个分布要符合真实业务。

**商品数据要突出热点。** 按二八原则，20% 的商品占据 80% 的流量。压测数据中，热点商品的比例要接近真实。

```sql
-- 商品数据构造：热点 + 长尾
INSERT INTO products_test
SELECT
    product_id + 100000 AS product_id,
    category_id,
    CASE
        WHEN RAND() < 0.2 THEN stock * 10   -- 20% 热点商品，库存充足
        ELSE stock                           -- 80% 长尾商品，库存正常
    END AS stock
FROM products;
```

### 2.4 数据构造的常见坑

**坑一：只造关键表数据。** 有些同学只构造了用户表和订单表，忽略了订单明细表、商品表、库存表。压测时外键约束报错，或者库存没扣减，都是因为关联表缺失。

**坑二：数据分布太均匀。** 随机生成的数据太「完美」，和真实业务差距很大。建议用历史数据作为基础，做「放大 + 变形」处理。

**坑三：数据量级差太远。** 压测数据量是生产的万分之一，很多问题不会暴露。建议压测数据至少是生产数据量的 10%。

## 三、数据隔离：让压测流量「看不见」生产数据

### 3.1 隔离的本质

数据隔离的核心问题是：**压测流量产生的数据，不能出现在生产数据里；生产数据，也不能被压测流量意外修改。**

听起来简单，做起来难。如果压测订单号和生产订单号格式一样，就会出现「压测订单混进了生产对账」的问题。如果压测库存扣减了生产真实库存，就会出现「超卖」问题。

隔离有多种方案，各有利弊。

### 3.2 方案一：影子表（Shadow Table）

影子表是指在数据库中创建和生产表结构相同但名称不同的表，压测数据写入影子表，生产数据不受影响。

```
生产表：orders
影子表：orders_test
```

优点：实现简单，不需要改代码逻辑。缺点：需要维护两套表结构，数据不一致风险高。

```sql
-- 创建影子表
CREATE TABLE orders_test LIKE orders;

-- 压测路由：在表名后面加 _test
-- 业务代码中，根据压测标记动态路由
String table = isPressureTest ? "orders_test" : "orders";
```

### 3.3 方案二：压测数据前缀

在数据中加一个前缀或标记字段，区分生产数据和压测数据。

```java
// 订单号构造：压测订单号以 "TEST" 开头
public String generateOrderId(boolean isPressureTest) {
    if (isPressureTest) {
        return "TEST" + UUID.randomUUID().toString().replace("-", "");
    }
    return generateNormalOrderId();
}

// 查询时过滤
SELECT * FROM orders WHERE order_id NOT LIKE 'TEST%';
```

优点：不需要额外的表或数据库。缺点：所有查询都要加过滤条件，容易遗漏。

### 3.4 方案三：压测数据库/集群

为压测创建独立的数据库实例或集群，压测流量完全走压测环境，生产数据零影响。

```
生产集群：10.0.0.1:3306
压测集群：10.0.0.101:3306
```

优点：隔离彻底。缺点：成本高，需要维护两套基础设施。

### 3.5 方案四：流量标记（Traffic Tagging）

在请求 Header 中加一个压测标记，所有服务都识别这个标记，路由到隔离的数据源。

```
Header: X-Pressure-Test: true
```

```java
// Spring Cloud Gateway 压测标记识别
@Component
public class PressureTestFilter {
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        if ("true".equals(exchange.getRequest().getHeaders().getFirst("X-Pressure-Test"))) {
            exchange.getAttributes().put("pressure_test", true);
        }
        return chain.filter(exchange);
    }
}

// 业务服务中读取标记
boolean isPressureTest = exchange.getAttribute("pressureTest");
String dataSource = isPressureTest ? "datasource_test" : "datasource";
```

这是目前最推荐的做法，对业务代码侵入小，扩展性好。

## 四、回放压测：用真实流量验证真实性能

### 4.1 什么是回放压测

回放压测（Replay Testing）是采集生产环境的真实流量，在压测环境中回放，从而验证系统性能。

真实流量是最好的压测来源：它包含了真实的用户行为分布、真实的数据访问模式、真实的并发特征。回放压测比人工构造流量更接近真实。

回放压测的流程：

```
1. 采集：Nginx/网关记录真实请求日志
2. 处理：脱敏、转换格式
3. 回放：用压测工具（如 tcpcopy、goReplay）回放流量
4. 验证：对比压测结果和基准
```

### 4.2 流量采集

流量采集有三种方式：

**方式一：Nginx 日志采集。** Nginx 记录 access log，包含请求时间、URL、IP、响应时间等。优点是零侵入，缺点是信息有限（没有请求体）。

**方式二：网关层镜像。** 在 API 网关层，将请求镜像到压测系统。优点是信息完整，缺点是需要网关支持。

**方式三：分布式追踪日志。** 利用 SkyWalking、Jaeger 等 APM 工具的 trace 日志做回放。优点是包含完整的调用链信息，缺点是数据量大。

```bash
# Nginx access log 格式配置
log_format pressure_log escape=json
    '{'
    '"time":"$time_iso8601",'
    '"remote_addr":"$remote_addr",'
    '"request":"$request",'
    '"status":$status,'
    '"body_bytes_sent":$body_bytes_sent,'
    '"request_time":$request_time'
    '}';

access_log /var/log/nginx/pressure_access.log pressure_log;
```

### 4.3 流量脱敏

真实流量不能直接用于压测，必须先脱敏。

**必须脱敏的字段**：

| 字段类型 | 脱敏方式 |
|---------|---------|
| 用户 ID | 替换为压测用户 ID |
| 订单号 | 加前缀或替换 |
| 手机号 | 部分掩码（138****5678） |
| 身份证 | 完全替换 |
| 银行卡号 | 完全替换 |
| 密码 | 替换为固定测试密码 |

**脱敏的实现方式**：

```java
// 请求脱敏处理器
@Component
public class RequestDesensitizer {
    public HttpRequest desensitize(HttpRequest request, TestUserContext ctx) {
        HttpRequest copy = request.copy();
        // 替换用户 ID
        copy.setHeader("X-User-Id", ctx.getTestUserId());
        // 替换 Body 中的手机号
        String body = copy.getBody();
        body = body.replaceAll("\"phone\":\"\\d{11}\"",
            "\"phone\":\"" + ctx.getTestPhone() + "\"");
        copy.setBody(body);
        return copy;
    }
}
```

:::warning 脱敏要彻底

脱敏不彻底是全链路压测最大的风险。一旦压测数据混入生产，轻则影响数据分析，重则泄露用户隐私。建议脱敏后做一次全面检查，确认没有遗漏。

:::

### 4.4 流量回放工具

常见的流量回放工具：

| 工具 | 特点 | 适用场景 |
|------|------|---------|
| tcpcopy | Linux 内核层，零拷贝，性能高 | Linux 环境，TCP 流量回放 |
| goReplay | Go 语言实现，简单易用 | HTTP 流量回放，推荐 |
| Jeproy | Java 实现 | Java 服务流量回放 |
| Ngrep + 脚本 | 轻量级，适合简单场景 | 临时压测 |

```bash
# goReplay 使用示例
# 1. 启动 goReplay 服务，监听 8080 端口
gor --input-raw :8080 --output-file requests.gor

# 2. 在压测环境回放，1 倍速度
gor --input-file requests.gor --output-http "http://压测环境地址"

# 3. 10 倍放大回放
gor --input-file requests.gor --output-http "http://压测环境地址|10x"

# 4. 时延回放（模拟真实用户思考时间）
gor --input-file requests.gor \
    --output-http "http://压测环境地址" \
    --http-original-host
```

### 4.5 回放压测的注意事项

**注意事项一：时间戳问题。** 真实流量中包含创建时间、过期时间等时间戳字段。直接回放可能导致时间校验失败（如 token 过期、订单超时）。解决方式：回放前将所有时间字段按偏移量调整。

**注意事项二：幂等性问题。** 支付、退款等幂等性要求高的接口，多次回放可能产生问题。建议回放前将这类接口排除，或者在服务层做幂等性跳过。

**注意事项三：放大比例。** 1 倍回放可能无法压出系统极限。常用做法是逐步放大：1x → 5x → 10x → 20x，观察系统的变化趋势。

## 五、压测执行：从基准到极限

### 5.1 压测的三个阶段

全链路压测通常分三个阶段：

**阶段一：基准压测。** 用预期流量的 30% 做基准，记录 RT、QPS、错误率作为基准线。这个阶段主要验证架构是否正确，数据是否准备充分。

**阶段二：梯度压测。** 从基准流量开始，逐步增加并发，每次增加 20-30%，观察系统的变化。

```
30% → 50% → 70% → 90% → 100% → 120% → 150% → 极限
```

**阶段三：极限压测。** 持续压测直到系统崩溃，记录「系统崩溃点」和「崩溃前的表现」。知道系统极限在哪，才能制定合理的限流策略。

### 5.2 压测指标监控

压测过程中，需要实时监控以下指标：

**基础设施层**：CPU、内存、磁盘 IO、网络带宽

**应用层**：QPS、RT（P50/P95/P99）、错误率

**数据库层**：连接池使用率、慢查询数量、锁等待

**缓存层**：命中率、内存使用率、QPS

**JVM 层**：GC 频率、GC 耗时、堆内存使用率

```bash
# 监控脚本示例：压测期间持续采集指标
#!/bin/bash
while true; do
    echo "=== $(date) ===" >> /tmp/pressure_monitor.log
    # CPU 和内存
    top -bn1 | head -5 >> /tmp/pressure_monitor.log
    # MySQL 连接数
    mysql -uroot -e "SHOW STATUS LIKE 'Threads_connected';" >> /tmp/pressure_monitor.log
    # Redis QPS
    redis-cli INFO stats | grep instantaneous >> /tmp/pressure_monitor.log
    sleep 5
done
```

### 5.3 压测终止条件

压测不是压到崩溃才算结束。压测过程中，如果出现以下情况，应立即停止：

- **错误率超过 5%**。说明系统开始出现故障，继续压测会产生大量脏数据。
- **RT 超过 SLA 的 3 倍**。用户已经无法接受这个响应时间，继续压测没有意义。
- **出现 OOM 或 Full GC 频繁**。说明内存已经是瓶颈，继续压测会导致服务崩溃。
- **下游服务被打爆**。压测影响了依赖服务的稳定性。

## 六、压测结果分析与优化

### 6.1 如何从压测数据中找到问题

压测结束后，数据分析是关键。

**看 RT 曲线。** 正常情况下，随着并发增加，RT 会逐步上升但保持平稳。如果 RT 曲线出现「陡峭上升」的拐点，说明系统开始出现排队。

**看错误率曲线。** 正常情况下，错误率保持低位。如果错误率开始上升，说明系统开始拒绝请求。

**看分位数差距。** 如果 P50 和 P99 差距巨大（如 P50=50ms，P99=5000ms），说明存在严重的长尾问题。

**看资源使用率饱和度。** 如果 CPU、内存、连接池等资源接近饱和，说明系统达到了硬件瓶颈。

### 6.2 压测后的优化方向

根据压测结果，确定优化方向：

**数据库层**：慢查询优化、索引优化、连接池调整、分库分表

**缓存层**：命中率提升、热点数据隔离、大 key 拆分

**应用层**：异步化改造、锁粒度优化、线程池调整

**架构层**：服务拆分、读写分离、限流降级

优化后需要再次压测验证，形成闭环。

## 七、面试实战

### 7.1 场景一：压测数据构造

**面试官问**：「全链路压测的数据是怎么构造的？」

这道题考察的是对压测方法论的实战理解。

**参考回答**：

> 全链路压测的数据构造有三个关键点：
>
> **第一，数据量级要接近生产。** 如果生产有 1 亿条数据，压测至少要有 1000 万级别。我们通常用生产数据「复制 + 偏移」的方式构造，避免重新生成带来的分布失真。
>
> **第二，数据分布要符合业务。** 比如电商场景，80% 的流量集中在 20% 的热门商品上。如果压测数据是均匀分布的，缓存命中率、热点竞争等问题都不会暴露。所以我们会按二八原则构造热点数据。
>
> **第三，关联关系要正确。** 压测数据的用户、订单、商品、库存之间必须有正确的关联关系。我们用历史数据作为基础模板，放大后做关联，确保外键约束和业务逻辑都正确。
>
> 同时，压测数据必须和生产数据做隔离，不能污染真实的业务数据。

### 7.2 场景二：压测影响生产

**面试官问**：「全链路压测时，如何避免压测流量影响生产数据？」

这道题考察的是对数据隔离方案的理解。

**参考回答**：

> 隔离方案有几种，根据成本和隔离程度选择：
>
> **流量标记（推荐）**。在请求 Header 中加 `X-Pressure-Test: true` 标记，所有服务都识别这个标记，路由到隔离的数据源。这个方案对业务代码侵入小，扩展性好。
>
> **影子表**。数据库中创建和生产表结构相同的影子表（如 `orders_shadow`），压测数据写入影子表。这个方案实现简单，但需要维护两套表结构。
>
> **压测独立集群**。为压测创建完全独立的数据库和缓存实例。这个方案隔离最彻底，但成本最高。
>
> 我们用的是流量标记方案，配合数据脱敏（替换用户 ID、订单号等关键字段），确保压测数据和生产数据完全隔离。

## 八、留给你的思考题

### 题目一：流量回放的幂等性问题

支付接口在生产被调用了一次，生成了一笔真实的支付流水。如果把这笔请求回放压测，支付接口会被调用第二次，产生第二笔支付流水。**在不做代码改造的前提下，如何避免回放流量触发真实支付？**

### 题目二：压测环境的极限 vs 生产的极限

压测环境通常只有生产 10% 的机器配置。按这个比例，如果压测环境能扛 5000 QPS，是不是意味着生产能扛 50000 QPS？

如果不能，还需要考虑哪些因素？什么情况下压测环境的结论会失真？

## 总结

全链路压测的核心是三个问题：

- **怎么构造真实数据？** 数量级接近、分布符合业务、关联关系正确
- **怎么隔离生产数据？** 影子表、流量标记、独立集群
- **怎么用真实流量验证？** 流量采集、脱敏、回放

全链路压测不是一次性的活动，而是持续验证的过程。系统上线前要做，架构调整后要做，大促前更要反复做。只有用真实的数据和真实的流量，才能发现真实的问题。
