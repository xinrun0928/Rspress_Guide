# 性能优化 Checklist

性能优化最怕什么？最怕的不是不知道怎么优化，而是优化了半天，方向错了。

一个常见场景：团队花了两周优化了接口 A 的缓存策略，命中率从 60% 提升到了 90%，RT 从 200ms 降到了 120ms。开心上线，一促，还是崩了。后来复盘发现，接口 A 的 RT 从来都不是问题，真正的问题在于接口 B 的数据库连接池太小——压测时才发现。

怎么避免这种「优化了个寂寞」的情况？答案是：**在动手之前，先用 Checklist 评估一下现状，找准优化的方向。**

这篇 Checklist 不是告诉你「怎么优化」，而是告诉你「优化之前要看什么」。它把性能优化的关键检查点整理成清单，帮你系统性地审视系统，避免遗漏关键瓶颈。

## 一、基础设施检查

### 1.1 硬件资源是否满足需求

性能问题的根因，有相当一部分出在基础设施上。在优化代码之前，先确认硬件资源是否足够。

**CPU 检查**：

- [ ] 当前 CPU 利用率是多少？平均和峰值分别是多少？
- [ ] CPU 利用率高的原因是业务计算还是 GC？
- [ ] 是否有单核 CPU 打满而其他核空闲的情况（NUMA 问题）？
- [ ] CPU 核心数和线程数是否匹配？

```bash
# 查看 CPU 使用情况
top
# 按 1 查看每核心详情
# 查看 CPU 型号和核数
lscpu | grep -E "Model name|CPU\(s\)"
```

**内存检查**：

- [ ] 堆内存使用率是多少？是否接近 -Xmx 设置的上限？
- [ ] 是否有频繁的 GC？Full GC 的频率和耗时是多少？
- [ ] 非堆内存（Metaspace、Direct Memory）使用情况如何？
- [ ] 应用实际使用的内存是否超过容器/VM 的限制？

```bash
# 查看 JVM 内存使用
jmap -heap <pid>

# 查看 GC 情况
jstat -gcutil <pid> 1000
```

**磁盘 IO 检查**：

- [ ] 磁盘 IO 利用率是多少？是否有 IO 打满的情况？
- [ ] 是顺序写还是随机 IO？是否可以考虑换成 SSD？
- [ ] 日志写入是否在影响 IO 性能？

```bash
# 查看磁盘 IO
iostat -x 1
# %util 接近 100% 说明 IO 打满
```

**网络带宽检查**：

- [ ] 服务器的网络带宽是多少？当前使用率是多少？
- [ ] 是否有过多的短连接导致 TIME_WAIT 堆积？
- [ ] 跨机房调用是否成为瓶颈？

```bash
# 查看网络带宽
sar -n DEV 1
# 查看连接状态
netstat -an | awk '/^tcp/ {++S[$NF]} END {for(a in S) print a, S[a]}'
```

### 1.2 基础设施的常见问题

| 问题 | 表现 | 解决方案 |
|------|------|---------|
| 容器内存限制 < JVM 堆 + 非堆 | OOM Kill | 调整容器内存限制或 JVM 参数 |
| 磁盘 IO 打满 | 日志写入慢、服务响应抖动 | 换 SSD、日志异步化 |
| 网络带宽打满 | 跨机房调用延迟高 | 扩容或压缩数据 |
| CPU 绑核不均 | 单核打满、其他空闲 | 关闭 NUMA 或绑定 CPU |

## 二、JVM 层检查

### 2.1 内存与 GC 检查

GC 问题是最常见的性能杀手之一。很多「莫名慢」的问题，最终根因都是 GC。

**堆内存检查**：

- [ ] -Xms 和 -Xmx 是否一致？不一致可能导致堆动态扩容时的停顿。
- [ ] 堆内存大小是否合理？太小导致频繁 GC，太大导致 GC 停顿时间长。
- [ ] 新生代和老年代比例是否合理？可以通过 -XX:NewRatio 调整。

```bash
# 查看 JVM 启动参数
jinfo -flags <pid> | grep -E "Xms|Xmx|NewRatio|Survivor"
```

**GC 频率检查**：

- [ ] Minor GC 的频率是多少？每次耗时多少毫秒？
- [ ] Full GC 的频率是多少？每次耗时多少毫秒？
- [ ] 对象晋升到老年代的速率是否正常？
- [ ] GC 日志是否开启？分析 GC 日志是排查 GC 问题的第一步。

```bash
# 开启 GC 日志（JDK 8）
-XX:+PrintGCDetails -XX:+PrintGCDateStamps -Xloggc:/tmp/gc.log

# 分析 GC 日志（用 gceasy.io 或 GCViewer）
```

**GC 收集器选择**：

- [ ] 当前使用的是什么垃圾收集器？JDK 8 默认是 Parallel GC（吞吐量优先）。
- [ ] 对于延迟敏感的业务，是否应该换成 G1 或 ZGC？
- [ ] G1 的 -XX:MaxGCPauseMillis 目标是否合理？

| 收集器 | 特点 | 适用场景 |
|--------|------|---------|
| Parallel GC | 吞吐量优先，停顿时间长 | 批处理、大数据 |
| CMS | 低停顿（并发标记） | Web 服务（已废弃） |
| G1 | 可预测停顿，平衡型 | JDK 9+ 默认，推荐 |
| ZGC | 亚毫秒级停顿 | 大内存、低延迟 |

### 2.2 线程与锁检查

**线程数检查**：

- [ ] 线程池的核心线程数和最大线程数是多少？
- [ ] 线程池的任务队列是否有界？无界队列可能导致 OOM。
- [ ] 线程池的拒绝策略是什么？
- [ ] 线程是否有名，便于排查？

```java
// 常见线程池配置问题
// 问题：使用无界队列
new ThreadPoolExecutor(core, max, 0L, TimeUnit.MILLISECONDS,
    new LinkedBlockingQueue<>());  // 无界队列，OOM 风险

// 正确：使用有界队列
new ThreadPoolExecutor(core, max, 60L, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(1000),  // 有界队列
    new ThreadPoolExecutor.CallerRunsPolicy());  // 拒绝策略
```

**锁竞争检查**：

- [ ] 是否有全局锁竞争导致的性能瓶颈？
- [ ] 是否可以用 ConcurrentHashMap 替代 HashMap？
- [ ] synchronized 修饰的范围是否过大？
- [ ] 是否存在死锁风险？

## 三、数据库检查

### 3.1 连接与连接池检查

数据库连接是性能问题的高发区。

**连接池检查**：

- [ ] 连接池的最大连接数是多少？是否满足并发需求？
- [ ] 连接获取等待时间（wait timeout）是否合理？
- [ ] 连接是否有效（做心跳检测）？
- [ ] 连接池的获取和归还是否正常？有无连接泄漏？

```bash
# 查看 MySQL 当前连接数
mysql -e "SHOW STATUS LIKE 'Threads_connected';"
mysql -e "SHOW STATUS LIKE 'Max_used_connections';"
```

**慢查询检查**：

- [ ] slow_query_log 是否开启？
- [ ] long_query_time 设置是多少？是否合理？
- [ ] 最近有哪些慢查询？是否都有索引？
- [ ] 慢查询的执行计划是否正确（用 EXPLAIN 分析）？

```sql
-- 查看慢查询数量
SHOW STATUS LIKE 'Slow_queries';

-- 查看最近的慢查询
SHOW VARIABLES LIKE 'slow_query_log%';
SELECT * FROM mysql.slow_log ORDER BY start_time DESC LIMIT 10;
```

### 3.2 索引检查

索引是 SQL 优化的核心。

**索引覆盖检查**：

- [ ] WHERE 条件字段是否有索引？
- [ ] SELECT 的字段是否在索引中（覆盖索引）？
- [ ] 联合索引的字段顺序是否正确（最左前缀原则）？
- [ ] 是否有索引失效的情况（函数、隐式转换、OR）？

```sql
-- 查看查询执行计划
EXPLAIN SELECT * FROM orders WHERE user_id = 123;

-- 查看表的所有索引
SHOW INDEX FROM orders;
```

**索引失效场景**：

| 场景 | 说明 |
|------|------|
| `WHERE name LIKE '%keyword'` | 前缀通配符导致索引失效 |
| `WHERE age + 1 = 10` | 在索引列上做计算 |
| `WHERE status = 1 OR name = 'test'` | OR 导致索引部分失效 |
| `WHERE CAST(id AS CHAR) = '123'` | 隐式类型转换 |
| `WHERE id IN (SELECT id FROM ...)` | 子查询优化不当 |

## 四、缓存检查

### 4.1 缓存命中率检查

缓存是高性能的关键，但用不好反而适得其反。

**Redis 命中率检查**：

- [ ] Redis 的 keyspace_hits 和 keyspace_misses 是多少？命中率是多少？
- [ ] 命中率低的原因是什么？key 设计问题还是过期时间太短？
- [ ] 是否有大 key 导致访问热点？
- [ ] 是否有过期 key 导致缓存穿透？

```bash
# 查看 Redis 统计信息
redis-cli INFO stats | grep -E "keyspace_hits|keyspace_misses"

# 计算命中率
# 命中率 = keyspace_hits / (keyspace_hits + keyspace_misses)
```

**缓存 key 设计检查**：

- [ ] key 的命名是否规范？是否便于排查？
- [ ] key 是否太大（超过 10KB）？大 key 会影响网络传输和内存分配。
- [ ] key 的过期时间是否合理？是否有过期时间过短导致缓存频繁失效？

```bash
# 扫描大 key
redis-cli --bigkeys

# 查看 key 大小
redis-cli DEBUG OBJECT ENCODING "key_name"
```

### 4.2 缓存一致性检查

- [ ] 缓存和数据库的一致性如何保证？先更新数据库还是先删除缓存？
- [ ] 是否有缓存击穿、穿透、雪崩的问题？
- [ ] 缓存预热是否做好？冷启动时是否有大量请求打到数据库？

## 五、网络与 RPC 检查

### 5.1 HTTP/RPC 调用检查

- [ ] HTTP/RPC 超时时间设置是否合理？超时传递是否处理？
- [ ] 是否有不必要的串行调用？可以并行化的是否并行化了？
- [ ] HTTP 连接是否开启了 Keep-Alive？是否有短连接导致的性能损耗？
- [ ] 是否使用了 HTTP/2 或 gRPC 等高效协议？

**超时配置检查**：

| 配置项 | 推荐值 | 说明 |
|--------|--------|------|
| 连接超时 | 3-5s | 建立连接的最大等待时间 |
| 读取超时 | 5-10s | 等待响应的最大时间 |
| 重试次数 | 1-2 次 | 幂等接口可重试，非幂等接口慎重 |

**串行改并行检查**：

```java
// 串行调用（慢）
User user = userService.getUser(id);
List<Order> orders = orderService.getOrders(id);
List<Product> products = productService.getProducts(id);

// 并行调用（快）
CompletableFuture<User> userFuture = userService.getUserAsync(id);
CompletableFuture<List<Order>> ordersFuture = orderService.getOrdersAsync(id);
CompletableFuture<List<Product>> productsFuture = productService.getProductsAsync(id);

CompletableFuture.allOf(userFuture, ordersFuture, productsFuture).join();
```

### 5.2 连接复用检查

- [ ] HTTP Client 是否使用了连接池？
- [ ] 连接池大小是否合理？
- [ ] 是否开启了 HTTP Keep-Alive？
- [ ] DNS 解析是否有缓存？是否存在 DNS 解析延迟？

## 六、业务逻辑检查

### 6.1 代码层面的性能陷阱

**循环优化**：

- [ ] 是否在循环中做了不必要的数据库或缓存查询？
- [ ] 是否可以用批量操作替代循环中的单条操作？
- [ ] 循环是否有提前终止条件（减少无效计算）？

```java
// 问题：循环中查询数据库
for (Long orderId : orderIds) {
    Order order = orderMapper.selectById(orderId);  // N 次查询
}

// 正确：批量查询
List<Order> orders = orderMapper.selectBatchIds(orderIds);  // 1 次查询
Map<Long, Order> orderMap = orders.stream()
    .collect(Collectors.toMap(Order::getId, o -> o));
```

**对象创建**：

- [ ] 是否在高频路径上创建了大量临时对象？
- [ ] 是否可以用对象池或缓存替代？
- [ ] StringBuilder 是否正确使用？字符串拼接是否优化？

```java
// 问题：循环中拼接字符串
String sql = "";
for (String field : fields) {
    sql += field + ",";  // 每次创建新 String 对象
}

// 正确：使用 StringBuilder
StringBuilder sb = new StringBuilder();
for (String field : fields) {
    sb.append(field).append(",");
}
```

**日志与监控**：

- [ ] 日志级别是否正确？生产环境不应有大量 INFO/DEBUG 日志。
- [ ] 日志是否使用了占位符 `{}` 而不是字符串拼接？
- [ ] 日志输出是否同步？高频日志是否考虑异步化？

```java
// 问题：字符串拼接
log.info("userId=" + userId + ", orderId=" + orderId);

// 正确：使用占位符
log.info("userId={}, orderId={}", userId, orderId);
```

## 七、高可用与容错检查

### 7.1 限流与熔断检查

- [ ] 是否配置了限流策略？限流阈值是否合理？
- [ ] 是否配置了熔断策略？熔断后的降级方案是什么？
- [ ] 限流和熔断是否在网关层和业务层都有配置？

**限流检查清单**：

- [ ] QPS 限流：阈值是否满足业务峰值？
- [ ] 并发限流：线程池大小是否合理？
- [ ] 熔断：触发条件（RT 超时/错误率）是否合理？恢复策略是什么？
- [ ] 降级：熔断后的兜底返回是什么？

### 7.2 超时与重试检查

- [ ] 所有外部调用是否都配置了超时？
- [ ] 超时时间是否合理？是否考虑了网络抖动？
- [ ] 重试是否做了幂等性保护？
- [ ] 重试是否使用了指数退避（避免惊群效应）？

```java
// 正确：指数退避重试
int maxRetries = 3;
long baseDelay = 100;  // 基础延迟 100ms
for (int i = 0; i < maxRetries; i++) {
    try {
        return call();
    } catch (Exception e) {
        if (i == maxRetries - 1) throw e;
        Thread.sleep(baseDelay * (1 << i));  // 100, 200, 400ms
    }
}
```

## 八、监控与告警检查

### 8.1 核心指标是否已覆盖

**必监控的指标**：

- [ ] QPS / TPS
- [ ] RT（P50/P95/P99）
- [ ] 错误率
- [ ] CPU / 内存 / 磁盘 IO / 网络
- [ ] GC 频率和耗时
- [ ] 数据库连接池使用率
- [ ] Redis 命中率
- [ ] 线程池活跃线程数

**告警阈值是否合理**：

- [ ] CPU > 80% 是否告警？
- [ ] 错误率 > 1% 是否告警？
- [ ] P99 RT > SLA 目标是否告警？
- [ ] Full GC 是否立即告警？

### 8.2 日志与链路追踪

- [ ] 关键接口是否开启了链路追踪（SkyWalking/Jaeger）？
- [ ] 慢请求是否记录了详细日志？
- [ ] 异常是否都有上报（ELK/SLS）？

## 九、总结：优化前的思考顺序

性能优化不是一上来就改代码。按照这个顺序检查，效率最高：

```
第一步：基础设施
  └─ CPU、内存、磁盘 IO、网络是否打满？

第二步：JVM 层
  └─ GC 是否正常？线程池配置是否合理？

第三步：数据库层
  └─ 慢查询、连接池、索引是否到位？

第四步：缓存层
  └─ 命中率是否正常？是否存在穿透/击穿/雪崩？

第五步：网络/RPC 层
  └─ 超时、串行调用、连接复用是否合理？

第六步：业务代码
  └─ 是否有循环查询、频繁创建对象等低级问题？

第七步：高可用
  └─ 限流、熔断、降级是否到位？
```

每一步都有关键检查点，都可能成为性能瓶颈。用好这个 Checklist，让优化有方向、有依据、有闭环。

## 面试追问方向

- **「你们的系统做了哪些性能优化，效果如何？」** 可以结合 Checklist 的某个具体优化点展开。
- **「性能优化的收益怎么量化？」** 回答：优化前后对比 P99 RT、QPS、错误率等指标。
- **「遇到过最难排查的性能问题是什么？」** 回答：结合某次用 Checklist 发现「原来问题不在这里」的经历。
