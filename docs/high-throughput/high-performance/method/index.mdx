# 性能优化方法论

## 模块定位

性能优化不是一锤子买卖，也不是靠「经验」拍脑袋。它是一门基于数据的工程学科——用测量找到瓶颈，用数据验证效果，用系统化的方法持续迭代。

「高性能优化」模块的所有子模块（数据库、JVM、缓存、IO 等）都在解决「怎么优化」的问题，而 **method（方法论）** 要解决的问题是「优化前怎么看」和「优化后怎么验证」。

它是高性能优化的底层逻辑，也是整个三高架构体系的通用方法论。

---

## 核心内容

本模块包含 5 个章节，涵盖性能优化的两个核心思想：

### 思想一：往哪优化

性能优化的第一件事，不是动手，而是判断「哪个环节最短板」。

| 章节 | 核心问题 |
|------|---------|
| [木桶原理与测量驱动](./principle) | 性能由最短板决定，不要在长板上浪费精力 |
| [性能优化 checklist](./checklist) | 优化前系统性地过一遍检查项，避免遗漏关键瓶颈 |

### 思想二：怎么量化

找到短板之后，需要用数据定位具体问题，优化后再用数据验证效果。

| 章节 | 核心问题 |
|------|---------|
| [性能优化度量指标](./metrics) | P99 RT、QPS、错误率——黄金三角指标的含义与关系 |
| [性能瓶颈分析方法](./bottleneck) | APM、火焰图、Arthas、jstack——用什么工具找什么问题 |
| [全链路压测](./pressure-test) | 流量构造、数据隔离、回放压测——在生产环境验证真实能力 |

---

## 学习路径

```
principle（框架）→ metrics（量化工具）→ bottleneck（分析方法）
→ checklist（系统检查）→ pressure-test（最终验证）
```

**第一步：理解木桶原理（principle）。** 知道「往哪优化」比「怎么优化」更重要。

**第二步：掌握度量指标（metrics）。** 看不懂指标，就不知道问题在哪、效果如何。

**第三步：学会分析方法（bottleneck）。** 用工具定位，不用猜测——这是工程素养的体现。

**第四步：用 checklist 实践。** 拿到一个系统，能系统性地过一遍检查项，找准瓶颈。

**第五步：掌握全链路压测（pressure-test）。** 用真实流量验证真实能力，是上线前的最后一道关卡。

## 面试考察

性能优化方法论是回答「性能优化思路」类问题的核心框架。

**高频面试题**：

- 「性能优化的思路是什么？」
- 「性能优化的核心指标有哪些？」
- 「如何定位系统性能瓶颈？」
- 「全链路压测怎么做？如何避免影响生产数据？」
- 「线上 CPU 突然飙升，如何排查？」

这些问题都能在 method 模块找到答案。
