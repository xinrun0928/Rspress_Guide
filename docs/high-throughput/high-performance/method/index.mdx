# 性能优化方法论

## 模块定位

性能优化不是一锤子买卖，也不是靠「经验」拍脑袋。它是一门基于数据的工程学科——用测量找到瓶颈，用数据验证效果，用系统化的方法持续迭代。

「高性能优化」模块的所有子模块（数据库、JVM、缓存、IO、池化等）都在解决「怎么优化」的问题，而 **method（方法论）** 要解决的问题是「优化前怎么看」和「优化后怎么验证」。

它是高性能优化的底层逻辑，也是整个三高架构体系的通用方法论。

---

## 核心内容

本模块包含 5 个章节，涵盖性能优化的两个核心思想：

### 思想一：往哪优化

性能优化的第一件事，不是动手，而是判断「哪个环节最短板」。

| 章节 | 核心问题 | 学习目标 |
|-----|---------|---------|
| [木桶原理与测量驱动](./principle) | 性能由最短板决定，不要在长板上浪费精力 | 理解木桶原理和测量驱动思想 |
| [性能优化 checklist](./checklist) | 优化前系统性地过一遍检查项，避免遗漏关键瓶颈 | 掌握系统性排查思路 |

### 思想二：怎么量化

找到短板之后，需要用数据定位具体问题，优化后再用数据验证效果。

| 章节 | 核心问题 | 学习目标 |
|-----|---------|---------|
| [性能优化度量指标](./metrics) | P99 RT、QPS、错误率——黄金三角指标的含义与关系 | 理解核心指标及其关系 |
| [性能瓶颈分析方法](./bottleneck) | APM、火焰图、Arthas、jstack——用什么工具找什么问题 | 掌握瓶颈分析方法 |
| [全链路压测](./pressure-test) | 流量构造、数据隔离、回放压测——在生产环境验证真实能力 | 掌握压测方法论 |

## 学习路径

### 完整学习路线

```
principle（框架）→ metrics（量化工具）→ bottleneck（分析方法）
→ checklist（系统检查）→ pressure-test（最终验证）
```

**第一步：理解木桶原理（principle）**

- 知道「往哪优化」比「怎么优化」更重要
- 理解系统性能由最薄弱环节决定
- 学会用测量数据定位瓶颈，而不是凭感觉猜测

**第二步：掌握度量指标（metrics）**

- 看不懂指标，就不知道问题在哪、效果如何
- 理解 P50/P95/P99 RT 的含义
- 理解 QPS、吞吐量、错误率的关系

**第三步：学会分析方法（bottleneck）**

- 用工具定位，不用猜测——这是工程素养的体现
- 掌握 APM 工具的全链路追踪能力
- 学会使用火焰图分析 CPU 热点

**第四步：用 checklist 实践（checklist）**

- 拿到一个系统，能系统性地过一遍检查项
- 找准瓶颈，而不是眉毛胡子一把抓
- 形成自己的优化 checklist

**第五步：掌握全链路压测（pressure-test）**

- 用真实流量验证真实能力
- 学会流量构造和数据隔离
- 是上线前的最后一道关卡

## 面试考察

性能优化方法论是回答「性能优化思路」类问题的核心框架。

### 高频面试题

| 面试题 | 考察维度 | 答题要点 |
|-------|---------|---------|
| 「性能优化的思路是什么？」 | 整体认知 | 木桶原理 + 测量驱动 |
| 「性能优化的核心指标有哪些？」 | 指标理解 | RT、QPS、错误率 |
| 「如何定位系统性能瓶颈？」 | 分析能力 | APM、火焰图、jstack |
| 「如何排查线上 CPU 高占用？」 | 问题定位 | top → jstack → 火焰图 |
| 「如何排查线上响应慢？」 | 问题定位 | APM 全链路追踪 |
| 「全链路压测怎么做？」 | 工程实践 | 流量构造、数据隔离、回放 |
| 「如何避免压测影响生产数据？」 | 工程实践 | 数据脱敏、影子表 |

### 答题框架

```
面试官问「性能优化思路」类问题，标准回答框架：

1. 原则：性能优化第一步是测量，不是猜测
2. 定位：用 APM 工具定位耗时最长的环节（木桶原理）
3. 分析：根据耗时环节选择分析方法
   - 数据库慢 → EXPLAIN 分析 + 慢查询日志
   - Redis 慢 → 命中率分析 + 连接状态
   - GC 问题 → GC 日志 + 火焰图
   - 网络问题 → tcpdump + ss
4. 优化：根据原因选择优化方案
5. 验证：用同样的测量手段验证效果
6. 迭代：优化是分层的，解决一个瓶颈后继续找下一个
```

## 各章节详细内容

### principle（木桶原理与测量驱动）

这是方法论的基石。

**核心概念**：
- 木桶原理：系统性能由最薄弱环节决定
- 测量驱动：用数据说话，不凭猜测
- 三层瓶颈：资源层 → 交互层 → 架构层

**学习重点**：
- 理解「优化什么」比「怎么优化」更重要
- 学会建立基准线和对比验证
- 理解优化是持续迭代的过程

### metrics（性能度量指标）

这是量化分析的基础。

**核心指标**：
- RT（响应时间）：P50/P95/P99 分位值
- QPS/TPS：系统吞吐能力
- 错误率：5xx、超时比例

**学习重点**：
- 理解为什么要看 P99 而不是平均值
- 理解三个指标之间的关联
- 学会用指标判断系统健康度

### bottleneck（性能瓶颈分析）

这是问题定位的核心技能。

**分析工具**：
- APM：SkyWalking、Pinpoint（全链路追踪）
- JVM：jstack、Arthas、async-profiler（火焰图）
- 数据库：EXPLAIN、慢查询日志
- 网络：tcpdump、ss、netstat

**学习重点**：
- 掌握 APM 全链路追踪的使用
- 学会用火焰图分析 CPU 热点
- 学会用慢查询日志定位问题 SQL

### checklist（性能优化清单）

这是系统性排查的指南。

**检查维度**：
- 数据库：索引、SQL、连接池
- JVM：GC 参数、堆大小、线程数
- Redis：命中率、大 key、过期策略
- 业务代码：循环、锁、同步

**学习重点**：
- 形成自己的 checklist 习惯
- 优化前先过一遍，避免遗漏
- 根据业务特点定制 checklist

### pressure-test（全链路压测）

这是上线前的最终验证。

**核心步骤**：
1. 流量构造：使用日志回放或流量模拟
2. 数据隔离：影子库、影子表、数据脱敏
3. 压测执行：逐步加压，观察指标
4. 结果分析：找到系统瓶颈

**学习重点**：
- 理解压测环境与生产环境的差异
- 掌握数据隔离的方法
- 学会根据压测结果调优系统

## 总结

方法论是高性能优化的「道」，其他模块是「术」。

只有先理解了「为什么要测量」「为什么要找短板」「为什么要验证」，才能在面对具体问题时做出正确的判断。

学习方法论，不要死记硬背，要结合实际项目思考。当你真正理解了这套方法论，你会发现自己面对性能问题时，不再手足无措，而是胸有成竹。
