# 性能瓶颈分析方法：APM、火焰图、Arthas、jstack

性能问题最怕的，不是系统慢，而是**不知道慢在哪里**。

一个接口 RT 从 50ms 涨到了 500ms，可能的原因有几十种：数据库慢查询、Redis 延迟、GC 停顿、网络抖动、线程池耗尽、外部服务超时……每一个环节都可能出问题，但你不确定是哪一个。

有人说「我猜是 GC 问题」，加了堆内存。RT 没降，反而 GC 更频繁了。问题根本不在这里。

这就是典型的「盲人摸象」。凭感觉猜测，十次有九次会错。性能瓶颈分析的核心方法，就是**用工具和数据说话**，把「猜」变成「定位」。

这篇文章，介绍几种生产环境最常用的性能瓶颈分析方法。

## 一、性能分析的基本思路

### 1.1 自顶向下 vs 自底向上

性能分析方法分为两大类：**自顶向下**和**自底向上**。

**自顶向下**是从用户请求的入口开始，逐层往下追踪耗时分布。你不知道瓶颈在哪，就从最外层开始，逐层「剥开」看哪一层耗时最多。

**自底向上**是从系统资源开始，看 CPU、内存、IO、网络哪个先打满，再反推是哪个进程、哪个接口消耗了资源。

两种方法适用场景不同：

- 当你知道问题出在某个链路（如订单服务），但不知道具体是哪个环节，用**自顶向下**
- 当你不知道问题出在哪一层，只感觉「系统变慢了」，用**自底向上**

:::tip 推荐顺序

生产环境中，我推荐**先自顶向下**（APM 全链路追踪），快速定位到出问题的链路；再用**自底向上**（火焰图、profiling）深入分析单个服务的内部瓶颈。这样效率最高。

:::

### 1.2 三类经典瓶颈

性能瓶颈分为三类，每类的排查方法不同：

**第一类：CPU 瓶颈**。CPU 打满或者接近满载，常见于计算密集型场景。排查方法：火焰图（async-profiler/perf）、top + jstack。

**第二类：IO 瓶颈**。磁盘或网络 IO 等待时间长，CPU 大部分时间在等 IO。排查方法：iostat 查看磁盘 IO，sar -n 查看网络 IO。

**第三类：锁/等待瓶颈**。线程在等待锁、等待数据库连接、等待外部服务返回，CPU 空闲但请求不响应。排查方法：jstack 线程栈分析。

三类瓶颈的表现不同：

| 瓶颈类型 | CPU 使用率 | 线程状态 | 常见原因 |
|---------|-----------|---------|---------|
| CPU 瓶颈 | 高 | Runnable | 密集计算、死循环 |
| IO 瓶颈 | 低 | Waiting/Blocked | 磁盘慢、网络延迟 |
| 锁瓶颈 | 低 | Waiting on monitor | 锁竞争、连接池耗尽 |

### 1.3 瓶颈分析的准备工作

在开始排查之前，需要先做一些准备：

1. **开启必要的日志和监控**。GC 日志、慢查询日志、APM 数据采集。
2. **准备分析工具**。火焰图工具、 Arthas、 jstack、MAT 等。
3. **确认问题时间段**。记录问题开始和结束的时间，便于缩小分析范围。

## 二、APM 全链路追踪：快速定位到问题链路

### 2.1 什么是 APM

APM（Application Performance Monitoring，应用性能监控）是一种**自顶向下**的性能分析工具。它能够追踪一个请求从发起到返回的全链路耗时，告诉你每个环节（网关、服务、数据库、缓存）分别花了多少时间。

常见的 APM 工具：

- **SkyWalking**：国产 APM，功能全面，社区活跃，推荐
- **Pinpoint**：韩国团队开发，零侵入，字节码增强
- **Jaeger**：CNCF 项目，Kubernetes 环境集成好
- **Zipkin**：Spring Cloud Sleuth 默认集成，轻量级

### 2.2 SkyWalking 的核心能力

SkyWalking 是目前 Java 生态最流行的 APM 工具。它的核心能力有三个：

**拓扑图（Topology）**：以图形化的方式展示服务之间的调用关系。每个节点是一个服务，每条边是调用链路。节点的颜色反映性能好坏（绿色快，红色慢）。

```
[用户] → [Nginx] → [网关] → [订单服务] → [库存服务]
                              ↓
                           [Redis] [MySQL]
```

通过拓扑图，你可以一眼看出哪个节点是性能洼地（响应最慢的环节）。

**链路追踪（Trace）**：追踪单个请求在各个服务之间的流转，记录每个环节的耗时。

```
Trace ID: abc123
  [Nginx]        2ms
    [网关]        5ms
      [订单服务]  200ms
        [Redis]  150ms   ← 瓶颈在这里
        [MySQL]  45ms
```

链路追踪能帮你精确找到耗时最长的环节——这就是木桶原理中「最短的那块板」。

**指标面板（Metrics）**：聚合 QPS、RT、错误率等指标，支持按服务、按接口分组查看。

### 2.3 APM 的实战用法

**场景：接口 RT 飙升，如何用 APM 定位问题？**

**第一步**：打开 SkyWalking 拓扑图，找到 RT 最长的节点（颜色最红的服务）。

**第二步**：进入链路追踪，过滤 RT 最长的请求，查看耗时分布。如果 Redis 耗时占比 80%，说明瓶颈在缓存层。

**第三步**：在 SkyWalking 中展开 Redis 调用的详情，看是连接等待时间长还是命令执行慢。如果是连接等待，说明连接池太小或者有热点 key 问题。

**第四步**：针对具体问题做优化。

APM 的优势是**零侵入**、**快速定位链路级瓶颈**。缺点是对进程内部的详细分析能力不足（无法看到哪个方法占用 CPU 高），这时候需要用火焰图。

## 三、火焰图：定位 CPU 消耗的「凶手」

### 3.1 火焰图是什么

火焰图（Flame Graph）是一种**自底向上**的性能分析可视化工具。它用图形的形式展示 CPU 时间在各个函数中的分布，每个「火焰」代表一个函数调用栈，火焰越高，说明该函数占用的 CPU 越多。

火焰图看起来像一簇火焰，从下往上追踪可以看到完整的调用链：

```
java.lang.Thread.run()                    [████████████ 50%]
  └─ com.example.Service.process()        [████████ 35%]
       ├─ calculate()                     [███ 20%]
       └─ db.query()                     [██ 15%]
```

:::tip 火焰图阅读技巧

火焰图从下往上看，每一层是其上层函数的子调用。火焰越高，说明该函数占用资源越多。**从火焰图顶端往下找，「平顶」的部分就是性能热点。** 平顶越宽，占用越多。

:::

### 3.2 火焰图的类型

| 类型 | 用途 |
|------|------|
| CPU 火焰图 | 分析 CPU 时间消耗 |
| 内存火焰图 | 分析内存分配（排查 GC 根因） |
| Off-CPU 火焰图 | 分析线程等待时间（排查 IO 等待） |

**CPU 火焰图**是最常用的类型，用于分析 CPU 密集型瓶颈。**Off-CPU 火焰图**用于分析 IO 密集型瓶颈，帮你找到「线程在哪等着」。

### 3.3 async-profiler：生产环境采集火焰图

生产环境推荐用 **async-profiler**（JDK 8+）采集火焰图。它的特点是：

- 基于 JVMTI 和 JVM TI API，**无需对代码做任何改动**
- **采样开销低**（`<5%`），对生产环境影响极小
- 支持 CPU、内存、锁等多种采样模式

采集 CPU 火焰图：

```bash
# 采集 60 秒的 CPU 火焰图
./async-profiler.sh start -d 60 -f /tmp/cpu.html -e cpu <pid>

# -d: 采集时长（秒）
# -f: 输出文件（html 格式，浏览器打开）
# -e: 事件类型（cpu/memory/allocation）
# <pid>: Java 进程 ID
```

生成的是 HTML 文件，用浏览器打开，可以看到交互式的火焰图。

### 3.4 perf +火焰图：Linux 原生工具

Linux 自带的 `perf` 工具也可以采集火焰图，但需要额外步骤生成图形。

```bash
# 采集 60 秒 CPU 数据
perf record -F 99 -p <pid> -g -- sleep 60

# 生成火焰图（需要 Brendan Gregg 的脚本）
git clone https://github.com/brendangregg/FlameGraph.git
perf script | ./FlameGraph/stackcollapse-perf.pl | \
  ./FlameGraph/flamegraph.pl > /tmp/perf.svg
```

async-profiler 比 perf 更适合 Java，因为它能解析 JIT 编译后的方法名，而 perf 在 Java 应用中经常显示的是十六进制地址。

### 3.5 火焰图实战案例

**场景**：一个计算接口，CPU 利用率 100%，但不知道哪个方法消耗最多。

**分析过程**：

1. 用 async-profiler 采集 60 秒 CPU 火焰图
2. 打开 HTML，发现 `com.example.OrderService.calculateDiscount()` 占用了 40% 的 CPU
3. 进入该方法，发现是一个嵌套循环在计算折扣——算法复杂度是 O(n²)
4. 优化为 O(n) 后，CPU 利用率从 100% 降到 30%

## 四、Arthas：Java 诊断神器

### 4.1 Arthas 是什么

Arthas（阿尔萨斯）是阿里巴巴开源的 Java 诊断工具，被称为「Java 诊断领域的瑞士军刀」。它可以**在线诊断**生产环境问题，无需重启 JVM。

Arthas 的核心功能：

- **dashboard**：实时查看 JVM 运行状态（内存、线程、GC、CPU）
- **thread**：查看线程状态，定位死锁和高 CPU 线程
- **trace**：追踪方法调用链路和耗时
- **watch**：观察方法的入参和返回值
- **profiler**：生成火焰图（内置了 async-profiler）

### 4.2 dashboard：JVM 状态一览

```bash
$ dashboard
```

输出：

```
ID     NAME           STATE   %CPU   DELTA_TIM  TIME
1      main           RUNNABLE 45.0   0.090      10:20
2      GC task thread#0  RUNNABLE  5.0   0.010     5:10
3      Reference Handler WAITING  0.0   0.000     2:30
-      Timer-0          TIMED_WA  0.0   0.000     1:00

Memory  used  total  max     usage
heap    512M   1024M  1024M   50.00%
```

dashboard 可以快速看到：

- **CPU 占用最高的线程**：%CPU 列
- **GC 是否频繁**：GC 线程状态和时间
- **堆内存使用率**：是否接近 max，是否频繁 GC

### 4.3 thread：定位高 CPU 线程

```bash
# 查看所有线程
$ thread

# 查看 CPU 占用最高的线程
$ thread -n 5

# 查看是否有死锁
$ thread -b
```

```
Threads click total: 50, new: 0, started: 0, daemon: 45
"http-nio-8080-exec-10" Id=45 CPU=35.0% RUNNABLE
  at com.example.OrderService.calculate(OrderService.java:45)
  at com.example.OrderController.order(OrderController.java:30)
```

`thread -n 5` 找出 CPU 占用最高的 5 个线程，并打印它们的调用栈。直接定位到消耗 CPU 的代码行。

### 4.4 trace：追踪方法调用耗时

```bash
# 追踪方法的调用链和耗时
$ trace com.example.OrderService calculateOrder

# 追踪并过滤耗时超过 100ms 的调用
$ trace com.example.OrderService calculateOrder '#cost > 100'
```

输出示例：

```
`---[6002ms] com.example.OrderService:calculateOrder()
    +---[800ms] com.example.OrderService:getUserInfo()
    |   `---[750ms] com.example.OrderService:queryDatabase()
    +---[200ms] com.example.OrderService:calculateDiscount()
    `---[5002ms] com.example.OrderService:saveOrder()
```

`trace` 的作用类似 APM 的链路追踪，但它是针对单个方法的。适合在 APM 定位到某个服务后，进一步深入分析该服务的内部方法调用。

### 4.5 watch：观察方法调用

```bash
# 观察方法的入参和返回值
$ watch com.example.OrderService calculateOrder '{params, returnObj}' -x 2
```

`-x 2` 表示展开嵌套对象最多 2 层。

`watch` 适合调试：你不知道某个方法的入参是什么，或者想看某个方法的返回值是否符合预期。无需加日志，无需重启，直接观察。

### 4.6 Arthas 实战案例

**场景**：接口 RT 高，怀疑是某个方法慢，但不确定是哪个环节。

**分析过程**：

1. 用 `trace` 追踪接口方法，快速定位到耗时最长的子方法
2. 用 `watch` 观察该方法的入参，看是否因为特殊数据导致耗时增加
3. 用 `thread -n 3` 查看 CPU 最高的几个线程，看是否有死循环
4. 如果需要 CPU 火焰图，用 `profiler` 命令直接生成

Arthas 的优势是**交互式**，可以实时观察生产环境，不用重启服务。但它更适合快速定位「单点问题」，不适合做长时间的性能采样分析。

## 五、jstack：线程Dump 分析

### 5.1 jstack 是什么

jstack 是 JDK 自带的线程分析工具，通过向 JVM 发送信号（`SIGQUIT`），让 JVM 打印出所有线程的栈信息。

```bash
# 打印线程栈
jstack <pid>

# 打印并导出到文件
jstack <pid> > /tmp/threaddump.txt
```

线程栈信息包括：

- 每个线程的名称、ID、状态
- 调用栈（当前执行到哪个类、哪个方法）
- 锁信息（持有的锁、等待的锁）

### 5.2 线程状态解读

线程栈中最重要的信息是**线程状态**：

| 状态 | 含义 | 正常？ |
|------|------|--------|
| RUNNABLE | 正在运行 | 正常 |
| BLOCKED | 等待获取监视器锁 | 异常，说明存在锁竞争 |
| WAITING | 无限期等待 | 异常，需要分析等待原因 |
| TIMED_WAITING | 限时等待 | 视情况而定 |
| NEW / TERMINATED | 新建/已结束 | 正常，不会出现在 Dump 中 |

**BLOCKED** 是最需要关注的。如果大量线程处于 BLOCKED 状态，说明存在严重的锁竞争，需要优化锁的使用。

### 5.3 死锁检测

jstack 有一个特别有用的功能：**自动检测死锁**。

```bash
$ jstack -l <pid> | grep -A 20 "Found one Java-level deadlock"
```

如果有死锁，jstack 会打印出死锁的详细信息：

```
Found one Java-level deadlock:
===========================
"Thread-2":
  waiting for monitor entry at 0x00000000e0b4d3b8
  owned by: Thread-1
"Thread-1":
  waiting for monitor entry at 0x00000000e0b4d4b0
  owned by: Thread-2
```

死锁的危害是致命的：两个或多个线程互相等待对方持有的锁，谁都无法继续执行。**发现死锁必须立即处理。**

### 5.4 线程 Dump 分析实战

**场景**：服务响应变慢，大量请求超时，怀疑是线程阻塞。

**分析过程**：

1. 多次执行 `jstack`（间隔 10-30 秒，至少取 3-5 次），避免偶然因素
2. 统计每次 Dump 中各状态的线程数量，看 BLOCKED 线程是否持续增加
3. 如果是 BLOCKED，找「waiting for monitor entry」的行，看线程在等哪把锁
4. 根据锁的地址（`0x00000000e0b4d3b8`）找到持有锁的线程，看它为什么没有释放

```bash
# 分析 BLOCKED 线程
$ grep -c "waiting for monitor entry" /tmp/threaddump1.txt
$ grep -c "waiting for monitor entry" /tmp/threaddump2.txt

# 找到持有锁的线程
$ grep -B 5 "0x00000000e0b4d3b8" /tmp/threaddump.txt
```

jstack 的优势是**无依赖**，JDK 自带，任何环境都能用。缺点是信息量有限，看不到 CPU 使用率和内存分配细节。

## 六、工具对比与选择

### 6.1 工具对比表

| 工具 | 类型 | 核心能力 | 适用场景 | 开销 |
|------|------|---------|---------|------|
| SkyWalking | APM | 全链路追踪 | 快速定位问题链路 | 低 |
| 火焰图 | Profiling | CPU/内存热点分析 | 定位代码级热点 | `<5%` |
| Arthas | 诊断工具 | 实时诊断、trace/watch | 快速排查单点问题 | 低 |
| jstack | 线程分析 | 线程栈、死锁检测 | 锁竞争、阻塞问题 | 低 |
| top/jstack | 系统工具 | CPU/内存监控 | 快速查看资源使用 | 低 |

### 6.2 排查流程推荐

**第一步：APM 看链路**。通过 SkyWalking 或其他 APM 工具，定位到「哪一段链路」慢。

**第二步：Arthas trace 深入方法**。针对慢的链路，用 Arthas trace 追踪方法调用，找到耗时最长的方法。

**第三步：火焰图看 CPU 热点**。如果怀疑是 CPU 问题，用 async-profiler 采集火焰图，找到消耗 CPU 的代码。

**第四步：jstack 看线程阻塞**。如果有大量线程 BLOCKED，用 jstack 分析锁竞争和死锁。

## 七、面试实战

### 7.1 场景一：CPU 飙升的排查

**面试官问**：「生产环境 CPU 突然 100%，如何定位问题代码？」

这道题考察的是实际的排查能力。

**参考回答**：

> CPU 100% 的排查分为四步：
>
> 第一步，**top 确认进程**。用 `top -p <pid>` 确认是哪个 Java 进程 CPU 高。
>
> 第二步，**jstack 找到高 CPU 线程**。用 `top -H` 找到 CPU 占用最高的线程 ID，转为十六进制，在 jstack 输出中定位到该线程，看它的调用栈。
>
> 第三步，**火焰图确认热点**。用 async-profiler 采集 CPU 火焰图，从顶端往下找平顶，确认是哪个方法占用 CPU 最多。
>
> 第四步，**代码分析和优化**。找到热点代码后，分析原因——是算法复杂度问题、死循环、还是频繁创建对象？针对原因做优化。
>
> 整个过程不需要重启服务，用工具在线采样分析。

### 7.2 场景二：接口慢但 CPU 不高

**面试官问**：「接口 RT 很高，但 CPU 使用率很低，可能是什么原因？」

这道题考察的是对非 CPU 瓶颈的理解。

**参考回答**：

> CPU 不高但 RT 高，说明线程不在 CPU 上执行，而是在等待。可能的原因有几种：
>
> **IO 等待**。线程在等待网络 IO（外部服务调用）、磁盘 IO（日志写入、数据库）。可以用 jstack 看线程状态，是否大量 `WAITING`。
>
> **锁竞争**。线程在等锁，无法执行。用 `jstack -b` 看是否有死锁，用 `jstack | grep "waiting for monitor"` 看锁等待情况。
>
> **连接池耗尽**。数据库连接池或 HTTP 连接池用完了，线程在等待获取连接。可以看应用日志是否有 connection timeout，或者用 Arthas 看连接池使用率。
>
> **GC 停顿**。Full GC 期间所有线程暂停，CPU 反而是空闲的。查看 GC 日志，看是否有长时间的 Full GC。
>
> 不同原因的处理方式不同，所以需要先定位是哪种情况。

## 八、留给你的思考题

### 题目一：APM 和火焰图怎么选

APM 可以看每个方法的耗时占比，火焰图也可以看 CPU 消耗占比。两者有什么区别？什么场景下用 APM？什么场景下必须用火焰图？

### 题目二：在线诊断 vs 离线分析

Arthas 是在线诊断工具，可以实时观察生产环境。火焰图和 jstack 也有在线采集的能力。**在线诊断有什么风险？** 采集过程本身是否会影响系统性能？如果影响严重，该怎么处理？

---

## 总结

性能瓶颈分析的核心方法就三种：

- **APM 全链路追踪**：快速定位到「哪一段链路」慢，是入场的第一选择
- **火焰图**：精准定位「哪个方法/哪行代码」消耗资源，是 CPU 分析的利器
- **jstack / Arthas**：看线程状态和锁竞争，是排查阻塞和死锁的必备工具

三种方法配合使用，覆盖了从链路到方法到线程的全方位分析。

记住：性能问题永远是用工具「找」出来的，不是用脑子「猜」出来的。没有数据支撑的优化，是盲目的优化。
