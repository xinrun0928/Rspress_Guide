# CMS GC 调优实战：从原理到问题排查

你接手了一个 JDK 8 的老项目，GC 日志里清一色是 CMS。团队里有人说 CMS 已经过时了，要换成 G1。但你隐约记得：面试官还经常问 CMS 的问题，项目里还在用 CMS，调优思路到底对不对？

先说结论：**CMS 确实已经被 G1 取代，但 CMS 的调优思路是理解所有 GC 的基础。** 理解了 CMS 的五个阶段和常见问题，你就能举一反三，看懂 G1、ZGC 的设计逻辑。更重要的是——JDK 8 项目还在大量运行，CMS 的调优是刚需，不是历史遗留问题。

这篇文章的目标很明确：**彻底搞懂 CMS 的工作原理，掌握常见问题的排查思路，积累可复用的调优经验。**

## 一、CMS 的五个阶段：从分代设计到并发清理

### 1.1 为什么要并发清理？

在说 CMS 之前，先回顾一下为什么需要 CMS。

Serial GC 和 Parallel GC 在垃圾回收时都需要「Stop The World」——所有用户线程暂停，GC 线程独占 CPU 完成垃圾回收。对于响应时间敏感的应用来说，这个停顿可能是致命的：一次 Full GC 停顿 2 秒，用户就会感知到明显的卡顿。

CMS 的设计目标很清晰：**让垃圾回收的大部分工作与用户线程并发执行，从而减少停顿时间。** 这是 CMS 名字的由来——Concurrent Mark-Sweep，并发标记-清除。

### 1.2 五个阶段详解

CMS 的垃圾回收分为五个阶段，每个阶段都有其特定的行为和停顿特点。

```
┌─────────────────────────────────────────────────────────────────┐
│                        CMS GC 全流程                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  阶段一：初始标记（Initial Mark）          [STW]                 │
│  ├─ 从 GC Roots 直接引用的对象开始标记                             │
│  ├─ 只标记一层，深度有限                                           │
│  └─ 停顿时间短（通常几十毫秒）                                      │
│                                                                 │
│  阶段二：并发标记（Concurrent Mark）         [并发执行]            │
│  ├─ 从初始标记的对象出发，追踪所有存活对象                          │
│  ├─ 用户线程和 GC 线程同时运行                                     │
│  └─ 停顿时间 ≈ 0（但总耗时长）                                     │
│                                                                 │
│  阶段三：并发预清理（Concurrent Preclean）   [并发执行]            │
│  ├─ 修正并发标记期间因用户线程运行导致的标记变动                     │
│  └─ 减少重标记阶段的停顿时间                                       │
│                                                                 │
│  阶段四：重标记（Remark）                  [STW]                  │
│  ├─ 修正并发标记期间的变动对象                                      │
│  └─ 停顿时间通常比初始标记长                                       │
│                                                                 │
│  阶段五：并发清除（Concurrent Sweep）        [并发执行]            │
│  ├─ 清理死亡对象，释放内存                                         │
│  └─ 停顿时间 ≈ 0                                                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**阶段一：初始标记（Initial Mark）**

初始标记是 CMS 整个回收周期中**唯一的 STW 阶段**。它从 GC Roots（栈中的局部变量、方法区中的静态变量、JNI 引用等）出发，只标记这些根对象直接引用的对象。

由于只追踪一层，初始标记的停顿时间通常很短。但有一个细节需要注意：如果老年代中的对象被年轻代的对象引用，初始标记也需要扫描年轻代——这就是为什么年轻代较大时，初始标记的停顿也可能变长。

```bash
# 初始标记阶段的日志
[GC (CMS Initial Mark) [1 CMS-initial-mark: 524288K(786432K)] 612345K(917504K), 0.0456789 secs]
```

这条日志显示：初始标记阶段，老年代已使用 524288K（总容量 786432K），堆已使用 612345K（总容量 917504K），停顿时间 45.7ms。

**阶段二：并发标记（Concurrent Mark）**

并发标记是 CMS 中最耗时的阶段，但它不阻塞用户线程。GC 线程从初始标记的对象出发，像遍历一棵树一样，追踪所有可达的存活对象。

这个阶段的问题是：用户线程在运行过程中，会不断创建新对象和修改对象引用，导致部分对象在标记过程中变成了垃圾——这些垃圾被称为「浮动垃圾」。浮动垃圾只能留到下一次 GC 清理。

```bash
# 并发标记阶段的日志（不显示具体停顿时间，因为是并发的）
[CMS-concurrent-mark-start]
[CMS-concurrent-mark: 0.567/0.567 secs]
```

**阶段三：并发预清理（Concurrent Preclean）**

并发预清理阶段的存在，是为了减少重标记阶段的停顿时间。

在并发标记阶段，用户线程和 GC 线程同时运行。用户线程会不断修改对象的引用关系：某个对象在标记时是存活的，但在重标记前可能变成了死亡对象；某个对象可能是新创建的。这些变化需要在重标记阶段重新处理。

并发预清理会扫描这些发生变动的对象，将它们加入一个「待处理」集合，从而减少重标记的工作量。

```bash
# 并发预清理阶段的日志
[CMS-concurrent-preclean-start]
[CMS-concurrent-preclean: 0.123/0.123 secs]
```

**阶段四：重标记（Remark）**

重标记是 CMS 的第二次 STW 阶段。它的任务是修正并发标记期间的变动对象，确保所有存活对象都被正确标记。

重标记需要扫描的对象包括：

- 并发预清理阶段标记的变动对象
- 并发标记阶段创建的新对象
- 初始标记时漏掉的跨代引用

重标记的停顿时间通常比初始标记长，因为需要扫描的范围更大。CMS 提供了一个优化参数 `-XX:+CMSScavengeBeforeRemark`，在重标记前先执行一次 Minor GC，从而减少需要扫描的对象数量。

```bash
# 重标记阶段的日志
[GC (CMS Final Remark) [YGCT: 123.456] [CMS-remark: 567890K(786432K)] 678901K(917504K), 0.1234567 secs]
```

**阶段五：并发清除（Concurrent Sweep）**

并发清除阶段真正释放死亡对象的内存。与用户线程并发执行，不产生 STW。

清除后，老年代会产生大量不连续的内存空间，这就是「内存碎片」问题。内存碎片会导致两个严重后果：

- **大对象分配失败**：虽然老年代总空闲空间足够，但没有连续的空间容纳大对象
- **Promotion Failed**：年轻代对象晋升时找不到连续空间

```bash
# 并发清除阶段的日志
[CMS-concurrent-sweep-start]
[CMS-concurrent-sweep: 0.789/0.789 secs]
```

### 1.3 五阶段的时间分布

理解每个阶段的耗时，有助于定位 GC 问题。一张典型的阶段耗时分布图：

```
CMS GC 各阶段耗时占比（典型值）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
并发标记（Concurrent Mark）      ████████████████████  60-70%
并发清除（Concurrent Sweep）      ███████████           20-30%
并发预清理（Concurrent Preclean） ███                   5-10%
初始标记（Initial Mark）         ▌                    <1%
重标记（Remark）                 █                     3-5%
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

并发标记和并发清除占了大部分时间，但它们不阻塞用户线程。初始标记和重标记虽然占比小，但它们是真正的停顿来源。

## 二、CMS 的三大问题：理解它才能驾驭它

CMS 之所以被 G1 取代，是因为它存在三个根本性问题。这些问题不是 bug，而是分代设计 + 并发清理的 trade-off。

### 2.1 内存碎片：无解的「遗留问题」

CMS 使用「标记-清除」算法，不进行内存压缩。这意味着每次 GC 后，老年代会产生大量不连续的空闲空间。

想象一个图书馆：读者需要借一本 300 页的书，图书馆总空闲页数超过 300 页，但这些页散落在 100 本不同的书里。结果是无法借出。

内存碎片带来的问题：

**大对象分配失败**：老年代总空闲空间可能还有 2GB，但没有连续 500MB 空间，导致分配失败。

```bash
# Promotion Failed 日志
[GC (Allocation Failure) 
    Desired survivor size 524288 bytes, new threshold 6 (max 6)
     - age 1: 123456 bytes, 123456 total
     - age 2: 78901 bytes, 202357 total
     ...
CMS: promote overflow in CMS collection, 
     will not objects to compact]
```

**频繁的 Full GC**：碎片化严重时，CMS 可能退化到串行 Full GC，进行内存压缩。

CMS 解决这个问题的方式是「并发模式失败后触发 Full GC」——这是一种保护机制，但代价是长时间停顿。

### 2.2 浮动垃圾：并发清理的代价

浮动垃圾是 CMS 并发清理模式下的必然产物。

在并发标记阶段，用户线程和 GC 线程同时运行。用户线程可能创建新对象，也可能修改引用关系：

- **标记过程中死亡的对象**：标记时认为是存活的，但实际已经死亡，只能留到下次 GC
- **新创建的对象**：并发标记期间创建的对象，GC 线程来不及处理

CMS 必须留出足够的空间给这些浮动垃圾。如果老年代空间不足，浮动垃圾可能填满整个老年代，导致「Concurrent Mode Failure」。

CMS 预留空间的比例由 `-XX:CMSInitiatingOccupancyFraction` 控制，默认 92%。这意味着：CMS 必须在老年代达到 92% 之前开始回收，否则预留空间不够。

### 2.3 CPU 敏感：并发是有代价的

CMS 的并发阶段需要占用 CPU 资源。默认情况下，CMS 会占用 `(-XX:ParallelGCThreads + 3) / (4)` 个 CPU 核心。

对于 CPU 敏感型应用（如计算密集型服务），CMS 的并发阶段可能影响业务处理能力。

```bash
# JVM 启动参数示例：CMS 占用 CPU 核心数
-XX:ParallelGCThreads=4    # 并行 GC 线程数
-XX:ConcGCThreads=2       # 并发 GC 线程数（CMS 专用）
```

CMS 会在吞吐量和停顿时间之间做权衡。在 4 核机器上，CMS 可能会占用 1.5 个核心；在 8 核机器上，CMS 可能会占用 2.5 个核心。

## 三、CMS 的致命故障：Promotion Failed 与 Concurrent Mode Failure

这两个问题是 CMS 调优中最常见、也是最棘手的故障。理解它们的成因，才能对症下药。

### 3.1 Promotion Failed：晋升失败

**什么是 Promotion Failed？**

Promotion Failed 发生在 Minor GC 时：年轻代中的存活对象需要晋升到老年代，但老年代没有足够的连续空间。

```
Promotion Failed 场景模拟
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Minor GC 触发：
  年轻代 Eden: [已满] + Survivor From: [部分存活对象]

  需要晋升的对象总大小：300MB

  老年代状态：
  ┌────────────────────────────────────────────────────┐
  │ [已用 400MB] [碎片空间: 200MB] [已用 150MB] ...    │
  │        ↑                        ↑                 │
  │   连续空间不足 300MB           有空间但不连续      │
  └────────────────────────────────────────────────────┘

  结果：Promotion Failed → 触发 Full GC
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**Promotion Failed 的原因**

1. **老年代碎片化严重**：虽然总空闲空间足够，但没有连续空间容纳晋升对象
2. **老年代使用率过高**：CMS 还没来得及清理，老年代就满了
3. **Survivor 区空间不足**：对象在年轻代无法充分 age-up，被迫过早晋升
4. **大对象直接晋升**：大对象直接分配在老年代，占用连续空间

**解决方案**

```bash
# 方案一：降低 CMS 触发阈值，给 CMS 更多回收机会
-XX:CMSInitiatingOccupancyFraction=70

# 方案二：多次 Full GC 后进行内存压缩
-XX:CMSFullGCsBeforeCompaction=5

# 方案三：增大堆内存，给 CMS 更多缓冲空间
-Xmx4g -Xms4g

# 方案四：增大 Survivor 区，减少过早晋升
-XX:SurvivorRatio=8 -XX:MaxTenuringThreshold=15
```

### 3.2 Concurrent Mode Failure：并发模式失败

**什么是 Concurrent Mode Failure？**

Concurrent Mode Failure 是 CMS 最严重的故障。当 CMS 的并发标记还没完成，老年代就满了，CMS 不得不退化到串行 Full GC。

```bash
# Concurrent Mode Failure 日志
[GC (Allocation Failure) 
     524288K->524288K(786432K), 5.6789012 secs]
[CMS-concurrent-mark-abortable-preclean-start]
CMS: abort preclean due to timeout  <-- CMS 放弃了并发预清理
[GC (CMS Final Remark) 
     567890K(786432K)] 678901K(917504K), 0.2345678 secs]
[CMS-concurrent-sweep-abort-]

# 更严重的：直接 Full GC
[Full GC (Allocation Failure) 
     524288K->524288K(786432K), 8.9012345 secs]   <-- 停顿 8.9 秒！
```

**Concurrent Mode Failure 的原因**

1. **CMSInitiatingOccupancyFraction 设置过高**：92% 才触发，但并发阶段来不及完成
2. **对象分配速率过高**：CMS 的并发速度跟不上对象创建速度
3. **并发阶段时间过长**：机器负载高或 GC 线程资源不足，导致并发阶段变慢
4. **浮动垃圾过多**：并发标记期间创建了大量对象

**解决方案**

```bash
# 方案一：降低触发阈值，提前触发 CMS
-XX:CMSInitiatingOccupancyFraction=70
-XX:+UseCMSInitiatingOccupancyOnly   # 只使用设定值，不使用自适应

# 方案二：使用 CMS 作为 Full GC 的备选
-XX:+UseCMSCompactAtFullCollection   # Full GC 后进行内存压缩
-XX:CMSFullGCsBeforeCompaction=3     # 3 次 Full GC 后压缩

# 方案三：切换到 G1
-XX:+UseG1GC

# 方案四：增加堆内存
-Xmx8g -Xms8g
```

### 3.3 两种故障的对比

| 特征 | Promotion Failed | Concurrent Mode Failure |
|---|---|---|
| 本质 | 老年代空间不足（不连续） | 老年代满了，并发阶段无法完成 |
| 触发时机 | Minor GC 时 | CMS 并发阶段 |
| 后果 | 触发 Full GC | 退化到串行 Full GC |
| 停顿时间 | 较长（Full GC 级别） | 很长（串行 Full GC） |
| 根本原因 | 内存碎片 | 对象分配速率过高 / 触发阈值不当 |

## 四、关键参数详解：CMS 调优的核心武器

### 4.1 触发时机控制

**CMSInitiatingOccupancyFraction：CMS 触发阈值**

这是 CMS 最核心的参数，控制老年代达到多少占用率时触发 CMS GC。

```bash
# 默认值：92%
# 老年代占用率达到 92% 时，触发 CMS GC
-XX:CMSInitiatingOccupancyFraction=70   # 推荐值：降低到 70%
```

为什么默认是 92%？因为 CMS 需要预留空间给浮动垃圾。如果设置太低，会浪费老年代空间；如果设置太高，可能导致 Concurrent Mode Failure。

生产环境推荐设置为 70-75%，给 CMS 充足的并发清理时间。

**UseCMSInitiatingOccupancyOnly：禁用自适应触发**

```bash
# 默认：JVM 会根据历史数据动态调整触发阈值
# 禁用自适应，只使用设定的阈值
-XX:+UseCMSInitiatingOccupancyOnly
```

CMS 默认会动态调整触发阈值，根据历史 GC 数据估算「最佳」触发时机。但有时候这种自适应反而会导致问题——你无法预测下一次 GC 何时发生。

启用 `-XX:+UseCMSInitiatingOccupancyOnly` 后，CMS 只按照你设定的阈值触发，更可控。

### 4.2 停顿时间优化

**CMSParallelInitialMarkEnabled：初始标记并行化**

```bash
# 默认：关闭
# 启用后，初始标记使用多线程（ParallelGCThreads）
-XX:+CMSParallelInitialMarkEnabled
```

初始标记通常停顿时间很短（几十毫秒），并行化收益有限。但对于超低延迟场景，可以启用。

**CMSScavengeBeforeRemark：Remark 前先 Minor GC**

```bash
# 默认：关闭
# 启用后，在重标记前先执行一次 Minor GC
-XX:+CMSScavengeBeforeRemark
```

这个参数是 CMS 停顿优化的利器。重标记需要扫描年轻代中的脏卡片（Card），如果年轻代中有大量死亡对象，先清理掉可以减少扫描范围。

```log
# 启用 CMSScavengeBeforeRemark 后的日志
[GC (CMS Final Remark) 
     [YGCT: 123.456] 
     [CMS-remark: 567890K(786432K)] ...
# 注意：YGCT 在增加，说明执行了 Minor GC
```

### 4.3 内存压缩控制

**CMSFullGCsBeforeCompaction：Full GC 后压缩**

```bash
# 默认值：0（每次 Full GC 都压缩）
# 建议值：3-5（几次 Full GC 后压缩一次）
-XX:CMSFullGCsBeforeCompaction=5
```

这个参数控制多少次 Full GC 后进行内存压缩。设置为 0 表示每次 Full GC 都压缩；设置为 5 表示每 5 次 Full GC 压缩一次。

但要注意：内存压缩本身也是 STW。如果你的 Full GC 本身不频繁，可以设置小一些；如果 Full GC 很频繁，可能需要更频繁的压缩。

### 4.4 CPU 资源控制

**ParallelGCThreads：并行 GC 线程数**

```bash
# 默认值：CPU 核心数
# 建议：对于响应时间敏感的应用，可以适当减少
-XX:ParallelGCThreads=4
```

ParallelGCThreads 控制并行 GC（Minor GC 和 Full GC 串行部分）的线程数。

**ConcGCThreads：并发 GC 线程数**

```bash
# 默认值：(ParallelGCThreads + 3) / 4
# 建议：对于 CPU 敏感型应用，可以减少
-XX:ConcGCThreads=2
```

ConcGCThreads 控制 CMS 并发阶段的线程数。增加这个值可以加快并发清理速度，但会占用更多 CPU。

## 五、CMS 调优实战：从问题到方案

### 5.1 典型场景一：CMS 重标记停顿过长

**问题现象**

GC 日志显示，重标记阶段的停顿时间经常超过 100ms，影响接口 P99 响应时间。

```log
[GC (CMS Final Remark) 
     [YGCT: 123.456] 
     [CMS-remark: 567890K(786432K)] 678901K(917504K), 0.2345678 secs]
# 停顿 234ms，需要优化
```

**分析原因**

重标记停顿过长的常见原因：

1. **年轻代对象多**：并发标记期间，大量对象死亡，堆满了需要扫描的「脏卡片」
2. **跨代引用多**：老年代和年轻代之间互相引用多
3. **并发标记期间对象分配率高**：产生了大量需要扫描的新对象

**解决方案**

```bash
# 方案一：启用 Remark 前 Minor GC
-XX:+CMSScavengeBeforeRemark

# 方案二：减少年轻代大小（减少脏卡片数量）
-Xmn512m

# 方案三：增大 Survivor 区（对象在年轻代充分 age-up，减少晋升）
-XX:SurvivorRatio=8 -XX:MaxTenuringThreshold=15

# 方案四：使用 G1（重标记停顿更可控）
-XX:+UseG1GC -XX:MaxGCPauseMillis=100
```

### 5.2 典型场景二：Concurrent Mode Failure 导致 Full GC

**问题现象**

GC 日志频繁出现 Concurrent Mode Failure，每次 Full GC 停顿超过 5 秒。

```log
[CMS-concurrent-mark-abortable-preclean-start]
CMS: abort preclean due to timeout
[Full GC (Allocation Failure) 
     524288K->524288K(786432K), 8.9012345 secs]
```

**分析原因**

1. **CMSInitiatingOccupancyFraction 太高**：老年代占用率达到 92% 才触发，CMS 并发清理来不及
2. **对象分配速率过高**：业务高峰期的对象创建速度超过了 CMS 清理速度
3. **老年代碎片化**：并发清理无法创建连续空间

**解决方案**

```bash
# 方案一：降低触发阈值（最有效）
-XX:CMSInitiatingOccupancyFraction=70
-XX:+UseCMSInitiatingOccupancyOnly

# 方案二：增大并发 GC 线程数（加快并发清理）
-XX:ConcGCThreads=3

# 方案三：增加堆内存（增加缓冲空间）
-Xmx8g -Xms8g

# 方案四：增加 Full GC 后的内存压缩
-XX:+UseCMSCompactAtFullCollection
-XX:CMSFullGCsBeforeCompaction=3

# 方案五：考虑迁移到 G1
-XX:+UseG1GC
```

### 5.3 典型场景三：Promotion Failed 导致 Full GC

**问题现象**

Minor GC 频繁出现 Promotion Failed，每次都触发 Full GC。

```log
[GC (Allocation Failure) 
    Desired survivor size 524288 bytes, new threshold 6 (max 6)
CMS: promote overflow in CMS collection, will not objects to compact
[Full GC (Allocation Failure) 
     524288K->524288K(786432K), 5.6789012 secs]
```

**分析原因**

1. **Survivor 区太小**：对象无法在年轻代充分 age-up，被迫过早晋升
2. **晋升阈值设置不当**：CMS 默认 MaxTenuringThreshold=6，比其他 GC 低
3. **对象年龄分布不均**：大量对象年龄接近阈值

**解决方案**

```bash
# 方案一：增大 Survivor 区
-XX:SurvivorRatio=8   # 默认 2，增大到 8

# 方案二：增大最大晋升年龄
-XX:MaxTenuringThreshold=15   # 默认 6，增大到 15

# 方案三：增大年轻代
-Xmn1g   # 增大年轻代大小

# 方案四：降低 CMS 触发阈值
-XX:CMSInitiatingOccupancyFraction=70

# 方案五：考虑使用 G1（G1 对晋升的控制更精细）
-XX:+UseG1GC
```

## 六、CMS vs G1：何时用哪个？

### 6.1 选择 CMS 的场景

| 场景 | 原因 |
|---|---|
| JDK 8 低延迟服务 | JDK 8 默认 CMS，需要低停顿但堆小于 4GB |
| 老项目迁移 | 已有 CMS 配置，不想大改 |
| 小堆 + 低延迟 | G1 在小堆上优势不明显，CMS 配置简单 |
| 团队 CMS 经验丰富 | CMS 问题排查有成熟经验 |

### 6.2 迁移到 G1 的场景

| 场景 | 原因 |
|---|---|
| JDK 9+ 项目 | JDK 9+ 默认 G1，CMS 已废弃 |
| 堆大于 4GB | G1 对大堆支持更好，停顿更可控 |
| 需要可预测的停顿时间 | G1 支持设置 MaxGCPauseMillis |
| 内存碎片问题严重 | G1 有并发标记+分区，更好的空间整合 |
| 想减少手动调优 | G1 是自适应调优 |

### 6.3 迁移步骤

从 CMS 迁移到 G1，建议按以下步骤：

```bash
# 第一步：保留 CMS 配置作为备选
-XX:+UseG1GC
-XX:+UseCMSInitiatingOccupancyOnly   # 保留，设置高值让 G1 自己判断

# 第二步：设置目标停顿时间
-XX:MaxGCPauseMillis=200

# 第三步：移除 CMS 特有参数
# 删除：-XX:CMSInitiatingOccupancyFraction
# 删除：-XX:CMSParallelInitialMarkEnabled
# 删除：-XX:+CMSScavengeBeforeRemark

# 第四步：观察 G1 行为，调整参数
```

## 七、面试实战

### 7.1 场景一：CMS 的工作流程

**面试官问**：「请描述一下 CMS 垃圾收集器的工作流程。」

这道题考察对 CMS 原理的理解程度。标准答案是描述五个阶段，但更重要的是说明每个阶段的特点和停顿情况。

> CMS（Concurrent Mark-Sweep）垃圾收集器分为五个阶段：
>
> 1. **初始标记**：STW，从 GC Roots 直接引用的对象开始标记，停顿时间短
> 2. **并发标记**：与用户线程并发执行，从初始标记的对象出发追踪存活对象，不停顿但耗时长
> 3. **并发预清理**：修正并发标记期间的变动对象，减少重标记停顿
> 4. **重标记**：STW，修正并发标记期间的变动，停顿时间比初始标记长
> 5. **并发清除**：与用户线程并发执行，清理死亡对象，不停顿
>
> CMS 的核心设计思想是：让大部分 GC 工作与用户线程并发执行，从而减少停顿时间。但这也带来了浮动垃圾和内存碎片问题。

### 7.2 场景二：CMS 的问题与解决方案

**面试官问**：「CMS 有哪些问题？如何解决？」

这道题考察实战经验。

> CMS 有三个主要问题：
>
> 1. **内存碎片**：标记-清除算法导致老年代空间不连续。解决方案是设置 `-XX:CMSFullGCsBeforeCompaction` 定期压缩，或迁移到 G1。
>
> 2. **浮动垃圾**：并发阶段产生的垃圾只能下次 GC 清理。CMS 预留空间给浮动垃圾，由 `-XX:CMSInitiatingOccupancyFraction` 控制。如果浮动垃圾过多，可能触发 Concurrent Mode Failure。解决方案是降低触发阈值到 70-75%。
>
> 3. **Promotion Failed**：Survivor 区对象晋升时找不到连续空间。解决方案是增大 Survivor 区、调整晋升阈值、或降低 CMS 触发阈值。

### 7.3 场景三：CMS 和 G1 的区别

**面试官问**：「CMS 和 G1 的区别是什么？」

> 主要区别有四点：
>
> 1. **内存管理**：CMS 不分代（只有老年代），G1 逻辑分代（将堆划分为多个 Region）
> 2. **算法**：CMS 使用标记-清除（产生碎片），G1 使用标记-整理（无碎片）
> 3. **停顿控制**：CMS 停顿不可控（STW 时间取决于对象图大小），G1 支持设置 MaxGCPauseMillis
> 4. **适用场景**：CMS 适合小堆（`< 4GB`）、低延迟场景；G1 适合大堆、更可预测的停顿时间

## 总结

CMS 虽然已经被 G1 取代，但它代表了一种重要的设计思想：**用并发换停顿**。理解 CMS 的五个阶段、三个问题、两个故障，你就能：

1. **排查 CMS 相关问题**：Promotion Failed、Concurrent Mode Failure 不再是玄学
2. **调优 CMS 参数**：触发阈值、停顿优化、内存压缩都有章可循
3. **理解其他 GC**：G1、ZGC 的设计都有 CMS 的影子

更重要的是，JDK 8 项目还在大量运行。CMS 的调优经验，在未来五年内依然有用武之地。
