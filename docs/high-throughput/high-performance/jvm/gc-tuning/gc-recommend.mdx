# 不同业务场景的 GC 参数推荐

选择 GC 收集器就像选择武器——狙击枪适合远程精确打击，冲锋枪适合近战扫射。你不能让狙击手拿冲锋枪，也不能让突击队员扛狙击枪。

这篇文章的目标：根据不同业务场景，给出经过验证的 GC 参数组合。既有原理分析，也有拿来就用的配置模板。

## 一、GC 收集器全景对比

在推荐参数之前，先看一下各收集器的特点：

| 收集器 | 线程模型 | 停顿时间 | 吞吐量 | 适用场景 | 备注 |
|---|---|---|---|---|---|
| Serial | 单线程 | 长（100ms+） | 低 | 单核、低配环境 | 已被淘汰 |
| Parallel | 多线程 | 较长 | 高 | 批处理、离线计算 | JDK 8 默认 |
| CMS | 多线程并发 | 短（并发阶段无停顿） | 中 | 低延迟服务 | JDK 9 已废弃 |
| G1 | 多线程并发 | 可控（MaxGCPauseMillis） | 高 | 通用场景 | JDK 9+ 默认 |
| ZGC | 多线程并发 | `<` 1ms | 高 | 超低延迟 | JDK 11+ |
| Shenandoah | 多线程并发 | `<` 1ms | 高 | 超低延迟 | JDK 12+ |

```

JDK 1.3    → Serial GC（单线程，低效）
JDK 4      → Parallel GC（多线程，吞吐量优先）
JDK 5      → CMS GC（并发收集，低延迟探索）

JDK 11     → ZGC（亚毫秒级停顿）
```

## 二、场景一：低延迟交易系统

### 业务特征

- 场景：证券交易、支付网关、实时风控
- 核心指标：P99 响应时间 `<` 50ms
- QPS：通常几千到几万
- 用户感知：毫秒级延迟差异

### 痛点

交易系统最怕的不是慢，是「不稳定」。用户可以接受 30ms 的平均响应时间，但不能接受偶尔出现的 500ms 停顿。一次 GC 停顿可能导致订单超时、滑点损失。

### 推荐配置：G1

```bash
# 堆内存配置
-Xms8g -Xmx8g
-XX:NewSize=2g -XX:MaxNewSize=2g

# G1 配置
-XX:+UseG1GC
-XX:MaxGCPauseMillis=30           # 核心参数：目标停顿时间
-XX:G1HeapRegionSize=4m           # Region 大小，可选
-XX:InitiatingHeapOccupancyPercent=45  # 老年代占用 45% 时开始并发周期

# 字符串优化（JDK 11+）
-XX:+UseStringDeduplication

# 日志配置
-XX:+PrintGCDetails
-XX:+PrintGCDateStamps
-Xloggc:/var/log/app-gc.log
```

### 为什么选择 G1？

G1 的停顿时间可控（`MaxGCPauseMillis=30`），而 Parallel GC 无法控制停顿时间。CMS 虽然停顿短，但并发阶段会占用 CPU 资源，可能影响业务线程。

## 三、场景二：高吞吐量批处理

### 业务特征

- 场景：数据报表生成、日志分析、ETL 任务
- 核心指标：吞吐量（单位时间处理数据量）
- 运行模式：离线任务，响应时间不敏感
- 资源特点：通常独占服务器

### 痛点

批处理追求的是「尽快跑完」。GC 停顿虽然长，但只要总体时间短，就可以接受。重点是让 GC 次数少、单次回收多。

### 推荐配置：Parallel GC

```bash
# 堆内存配置（根据数据量调整）
-Xms16g -Xmx16g
-Xmn8g                           # 较大的年轻代，减少 Minor GC 频率

# Parallel GC 配置（关键）
-XX:+UseParallelGC              # 使用 Parallel 收集器
-XX:+UseParallelOldGC           # 老年代也用并行收集器
-XX:ParallelGCThreads=16        # GC 线程数，通常 = CPU 核心数
-XX:+UseAdaptiveSizePolicy     # 自适应调整各代大小

# 吞吐量目标（核心参数）
-XX:GCTimeRatio=19              # GC 时间占比 = 1/(1+19) = 5%，即 95% 吞吐量
-XX:MaxGCPauseMillis=500       # 最大停顿时间（仅供参考）

# 禁用部分优化（节省 CPU）
-XX:-DisableExplicitGC          # 允许 System.gc()

# 日志配置
-XX:+PrintGCDetails
-XX:+UseGCLogFileRotation
-XX:NumberOfGCLogFiles=5
-XX:GCLogFileSize=100m
-Xloggc:/var/log/app-gc.log
```

### 关键参数解读

`-XX:GCTimeRatio=19` 的含义：允许 GC 时间占总时间的 5%。如果设置为 9，则允许 10% 的时间用于 GC，即吞吐量目标 90%。

## 四、场景三：微服务 API

### 业务特征

- 场景：Spring Boot / Dubbo 微服务
- 核心指标：P99 `<` 200ms，QPS 几百到几千
- 部署方式：容器化（Docker + K8s）
- 内存限制：通常 1-4GB

### 痛点

微服务占用资源要「刚刚好」——既不能太小导致频繁 GC，也不能太大浪费资源。GC 配置需要平衡延迟和资源利用率。

### 推荐配置：G1

```bash
# 容器感知配置（JDK 10+ 自动启用）
-Xms2g -Xmx2g
-XX:+UseContainerSupport
-XX:MaxRAMPercentage=75.0      # 容器可用内存的 75%

# G1 配置
-XX:+UseG1GC
-XX:MaxGCPauseMillis=200        # P99 目标内的 GC 停顿
-XX:InitiatingHeapOccupancyPercent=60

# 优化 MetaSpace（如果使用 CMS 或 G1）
-XX:MetaspaceSize=256m
-XX:MaxMetaspaceSize=256m

# GC 日志（生产环境建议输出到独立文件）
-XX:+PrintGCDetails
-XX:+PrintGCDateStamps
-Xloggc:/var/log/gc.log
```

### 容器环境注意事项

```bash
# 错误配置示例
-Xms1g -Xmx1g                    # 固定大小，没问题

# 问题配置
# 容器限制 2GB 内存，JVM 配置 -Xmx4g
# → OOM Kill

# 正确配置
-XX:MaxRAMPercentage=75.0      # 容器内存的 75%
-XX:InitialRAMPercentage=75.0   # 启动时也使用 75%
```

## 五、场景四：超低延迟（金融、游戏）

### 业务特征

- 场景：高频交易、游戏服务器、实时通信
- 核心指标：P99 `<` 1ms，P999 `<` 10ms
- 用户感知：任何可感知的停顿都是问题

### 痛点

传统 GC 即使停顿 50ms，对高频交易来说也是灾难。ZGC 和 Shenandoah 是为此类场景设计的。

### 推荐配置：ZGC

```bash
# ZGC 配置（JDK 11+）
-Xms8g -Xmx8g
-XX:+UseZGC

# ZGC 线程数（通常自动检测）
-XX:ConcGCThreads=8

# GC 日志
-XX:+PrintGCDetails
-Xlog:gc*:/var/log/zgc.log

# 注意：ZGC 不支持 UseStringDeduplication
```

### ZGC 核心原理

```
ZGC 停顿时间 < 1ms 的秘密
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

传统 GC（如 G1）：
  STW → 标记 → 复制 → 更新引用
  ↑
  所有步骤都需要 Stop The World

ZGC：
  并发标记（业务线程运行中）
  并发引用处理（业务线程运行中）
  并发对象重定位（业务线程运行中）

  唯一 STW：初始标记 + 再标记（约 1ms）
  → 停顿时间与堆大小无关！

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### JDK 21+ 分代 ZGC

JDK 21 引入分代 ZGC，进一步降低停顿时间和内存开销：

```bash
# JDK 21+ 分代 ZGC
-Xms8g -Xmx8g
-XX:+UseZGC
-XX:+ZGenerational        # 启用分代

# 堆结构
# Young ZGC = Eden + 2 Survivor
# Old ZGC = Old
```

## 六、场景五：超大型堆（> 64GB）

### 业务特征

- 场景：大型应用服务器、缓存服务器
- 堆内存：64GB 以上
- 特点：堆越大，GC 停顿时间越长（传统 GC）

### 痛点

100GB 堆用 G1，Full GC 可能停顿 10 秒。必须选择能够控制停顿时间的收集器。

### 推荐配置：G1 或 ZGC

```bash
# 配置一：G1（成熟稳定）
-Xms128g -Xmx128g
-XX:+UseG1GC
-XX:MaxGCPauseMillis=500        # 放宽目标，允许较长停顿
-XX:G1HeapRegionSize=8m         # 增大 Region
-XX:InitiatingHeapOccupancyPercent=70

# 配置二：ZGC（更低延迟）
-Xms128g -Xmx128g
-XX:+UseZGC
-XX:ConcGCThreads=16
```

```
堆大小与收集器选择
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

堆大小        推荐收集器      理由
< 4GB         G1              容器环境常用
4-64GB        G1              成熟稳定

> 256GB       ZGC             ZGC 唯一选择

注意：Shenandoah 在 > 256GB 堆时也是选择之一

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 七、场景六：容器化环境（K8s）

### 业务特征

- 部署：Kubernetes 容器
- 资源限制：CPU 和内存都有 Limits
- 弹性伸缩：HPA 自动扩缩容

### 痛点

容器环境中，JVM 需要「知道」自己的资源限制。早期 JDK 版本不了解容器限制，导致资源感知错误。

### 推荐配置：G1 + 容器感知

```bash
# JDK 10+ 自动容器感知
-Xms2g -Xmx2g
-XX:+UseG1GC
-XX:MaxGCPauseMillis=200

# 明确指定容器感知（可选）
-XX:+UseContainerSupport
-XX:MaxRAMPercentage=75.0
-XX:InitialRAMPercentage=75.0

# CPU 限制感知（JDK 10+）
-XX:ActiveProcessorCount=4     # 手动指定（如果需要）

# K8s 健康检查兼容
# 注意：livenessProbe 和 readinessProbe 应该基于应用层指标，
# 而不是 JVM 内部指标
```

### 常见坑

```bash
# 坑 1：JDK 8 不认识容器限制
# JDK 8 需要额外配置
-Xmx=$(cat /sys/fs/cgroup/memory/memory.limit_in_bytes)

# 坑 2：容器 CPU limit 与 JVM 线程数
# JVM 默认线程数 = 容器 CPU limit
# 如果 CPU limit = 2，GC 线程数受限

# 坑 3：内存 limit vs JVM 堆
# 容器 2GB 内存，JVM 配置 -Xmx4g
# → 容器 OOM Killer 杀死进程
```

## 八、GC 对比总结表

### 停顿时间对比

| 收集器 | Min | Max | 平均 | 可控性 |
|---|---|---|---|---|
| Serial | 100ms | 500ms | 200ms | ❌ |
| Parallel | 100ms | 1000ms | 300ms | ❌ |
| CMS | 50ms | 200ms | 100ms | ⚠️ |
| G1 | 30ms | 500ms | 100ms | ✅ |
| ZGC | `<` 1ms | `<` 1ms | `<` 1ms | ✅ |
| Shenandoah | `<` 1ms | `<` 1ms | `<` 1ms | ✅ |

### 吞吐量对比

| 收集器 | 吞吐量目标 | 说明 |
|---|---|---|
| Serial | 低 | 单线程，基本不用 |
| Parallel | 95%+ | 批处理首选 |
| CMS | 80-90% | 并发收集会占用 CPU |
| G1 | 85-95% | 平衡型 |
| ZGC | 90%+ | 并发收集，CPU 占用较高 |
| Shenandoah | 90%+ | 并发收集，CPU 占用较高 |

### 内存占用对比

| 收集器 | 元数据开销 | 碎片化 |
|---|---|---|
| Serial/Parallel | 低 | 有（标记-整理） |
| CMS | 中（Card Table） | 有（并发清理） |
| G1 | 高（Region 机制） | 低（并发整理） |
| ZGC | 中（Color Pointers） | 低（并发整理） |
| Shenandoah | 中（Brooks Pointers） | 低（并发整理） |

## 九、面试实战

### 面试题一：如何选择 GC 收集器？

**答案要点**：

1. **批处理场景**：选择 Parallel GC，追求高吞吐量
2. **Web 服务/API**：选择 G1，平衡延迟和吞吐量
3. **超低延迟**：选择 ZGC 或 Shenandoah
4. **低配/单核环境**：Serial GC（但实际很少用）

**追问**：为什么不默认用 ZGC？

ZGC 虽然停顿时间短，但：
- CPU 占用较高（并发收集需要额外线程）
- 吞吐量略低于 Parallel GC
- JDK 11+ 才支持，部分场景还在用 JDK 8

---

### 面试题二：G1 和 ZGC 的区别是什么？

| 维度 | G1 | ZGC |
|---|---|---|
| 停顿时间 | 可控（P99），但与堆大小相关 | 亚毫秒级，与堆大小无关 |
| 内存结构 | Region | 内存页面（Page） |
| 标记算法 | SATB（Snapshot-At-The-Beginning） | 颜色指针（Color Pointers） |
| JDK 版本 | JDK 7+（默认 JDK 9+） | JDK 11+ |
| 分代支持 | 原生支持 | JDK 21+ 支持分代 |

---

### 面试题三：如何根据业务场景配置 GC 参数？

**回答思路**：

1. **明确目标**：延迟敏感 vs 吞吐量优先
2. **量化指标**：P99 < 50ms 还是 P99 < 200ms
3. **资源限制**：堆大小、CPU 核心数
4. **选择收集器**：根据以上因素确定
5. **调优参数**：重点调整停顿时间目标、堆大小、各代比例
