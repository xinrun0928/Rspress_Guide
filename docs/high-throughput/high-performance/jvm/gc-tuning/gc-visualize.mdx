# GC 日志可视化工具详解

GC 日志是排查 JVM 问题的第一手资料。但面对几百MB的纯文本日志，人眼根本无法识别模式、发现异常。

GC 日志可视化工具的作用，就是把这些冰冷的数字变成直观的图表——让你一眼看出停顿时间趋势、GC 频率异常、内存泄漏信号。

这篇文章，介绍主流的 GC 日志可视化工具，涵盖命令行分析、离线可视化、在线分析和实时监控四个维度。

---

## 一、GC 可视化的价值

### 1.1 纯文本日志的问题

```bash
# GC 日志原始输出示例
2024-01-15T10:30:45.123-0800: [GC (Allocation Failure) [PSYoungGen: 524288K->65536K(786432K)] 1048576K->589824K(2097152K), 0.0256789 secs] [Times: user=0.10 sys=0.01, real=0.03 secs]

2024-01-15T10:30:46.234-0800: [Full GC (System.gc()) [PSYoungGen: 65536K->0K(786432K)] [ParOldGen: 524288K->398540K(1310720K)] 589824K->398540K(2097152K), 1.2345678 secs] [Times: user=2.50 sys=0.10, real=1.24 secs]
```

问题在哪里？

- **信息密度低**：每次 GC 只有一行，但包含十几项数据
- **难以对比**：前后时间点的对比需要手动计算
- **模式难发现**：GC 频率变化、停顿时间趋势无法直观看出

### 1.2 可视化的核心价值

```
GC 可视化能解决的问题
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

✓ 停顿时间趋势：一眼看出 GC 是否在恶化
✓ 频率异常检测：GC 突然变频繁？可视化帮你定位
✓ 内存泄漏信号：老年代持续增长，可能是泄漏
✓ GC 配置效果对比：调参前后的效果对比
✓ 汇报与沟通：图表比日志更易向团队解释问题

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 二、离线分析工具

### 2.1 GCViewer

GCViewer 是最经典的 GC 日志可视化工具，诞生于 2001 年，至今仍在广泛使用。

**下载地址**：[GitHub - cheche/GCViewer](https://github.com/checche/gcviewer)

**支持格式**：
- HotSpot GC 日志（PrintGCDetails）
- IBM GC 日志
- Various 日志格式

**使用方法**：

```bash
# 1. 下载最新版本
wget https://github.com/checche/gcviewer/releases/download/1.34/gcviewer-1.34.jar

# 2. 分析 GC 日志
java -jar gcviewer-1.34.jar /var/log/gc.log

# 3. 或者直接打开（如果 X11 可用）
java -jar gcviewer-1.34.jar
```

**关键指标解读**：

| 指标 | 说明 | 健康值 |
|---|---|---|
| Throughput | 应用运行时间占比，越高越好 | `>` 95% |
| GC Pause Avg | 平均 GC 停顿，越低越好 | `<` 100ms |
| GC Pause Max | 最大停顿时间 | 无硬性标准 |
| Full GC Count | Full GC 次数 | 越少越好 |

### 2.2 gceasy.io

gceasy.io 是在线 GC 日志分析工具，由 Tagtraum Industries 开发。无需安装，直接上传日志文件即可分析。

**访问地址**：https://gceasy.io/

**核心功能**：

1. **智能问题检测**：自动识别 GC 配置问题、内存泄漏信号
2. **AI 问题诊断**：基于多年数据分析经验，给出优化建议
3. **可视化图表**：内存使用趋势、GC 频率、停顿时间分布
4. **对比分析**：支持多次上传，对比不同版本的 GC 表现

**使用流程**：

```
使用 gceasy.io 分析 GC 日志
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 开启 GC 日志
   -XX:+PrintGCDetails -XX:+PrintGCDateStamps -Xloggc:/path/to/gc.log

2. 导出日志文件
   scp user@server:/var/log/gc.log ./

3. 上传到 gceasy.io
   - 打开 https://gceasy.io/
   - 点击 "Upload GC Log"
   - 选择文件

4. 查看分析结果
   - Key Performance Indicators（关键指标）
   - GC Causes（GC 原因分析）
   - Recommendations（优化建议）

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 2.4 GCPlot

GCPlot 是开源的 GC 日志可视化工具，支持本地部署。

**GitHub**：[GitHub - andydote/GCPlot](https://github.com/andydote/GCPlot)

**特点**：

- 支持实时解析
- Web UI 界面
- 支持多种 GC 日志格式

**安装与使用**：

```bash
# 1. 下载编译好的版本
wget https://github.com/andydote/GCPlot/releases/download/2.1.2/gcplot-2.1.2.zip

# 2. 解压
unzip gcplot-2.1.2.zip
cd gcplot-2.1.2

# 3. 分析日志
./gcplot.sh parse /var/log/gc.log

# 4. 启动 Web 服务
./gcplot.sh web
# 访问 http://localhost:8080
```

### 4.2 Java Mission Control (JMC)

JMC 是 JDK Mission Control 的缩写，是比 VisualVM 更专业的监控工具。

**启动**：

```bash
# JDK 11+ 自带
jmc

# 或者使用命令行
jcmd <pid> JFR.start
```

**核心功能**：

1. **飞行记录（JFR）**：记录一段时间内的所有事件
2. **GC 详细分析**：记录每次 GC 的详细信息
3. **内存泄漏检测**：通过对象分配分析检测泄漏

**JFR 启动示例**：

```bash
# 启动飞行记录（持续 60 秒）
jcmd <pid> JFR.start duration=60s filename=/tmp/recording.jfr

# 或者后台持续记录
jcmd <pid> JFR.start settings=profile filename=/tmp/recording.jfr

# 停止记录
jcmd <pid> JFR.stop
```

## 六、面试实战

### 面试题一：如何分析 GC 日志？

**参考答案**：

1. **收集日志**：开启 GC 日志参数
2. **离线分析**：使用 GCViewer 或 gceasy.io
3. **识别问题**：关注吞吐量、GC 频率、停顿时间
4. **定位根因**：根据 GC 类型和频率判断问题类型
5. **优化验证**：调整参数后对比日志

### 面试题二：GC 日志显示 Throughput 只有 80%，意味着什么？

**参考答案**：

Throughput 80% 意味着 20% 的时间花在 GC 上。这通常意味着：

1. **对象分配率过高**：代码层面有问题
2. **堆内存配置不当**：年轻代太小或太大
3. **内存泄漏**：老年代持续增长

正常生产环境，Throughput 应该 `>` 95%。

### 面试题三：GC 日志中 Full GC 频繁，但堆内存还没满，是怎么回事？

**参考答案**：

常见原因：

1. **Metaspace 满**：JDK 8 的 PermGen 空间不足
2. **晋升失败**：Survivor 区空间不足导致对象提前晋升
3. **并发模式失败（CMS）**：CMS 并发清理未完成，老年代已满
4. **System.gc() 调用**：代码中显式调用了 `System.gc()`
