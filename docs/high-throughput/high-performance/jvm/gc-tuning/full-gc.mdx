# Full GC 频繁的原因与排查：生产环境实战指南

凌晨 2 点，你被报警电话叫醒：线上服务接口 P99 响应时间飙到 3 秒。登上服务器一看，jstat 输出让你心头一紧——Full GC 每分钟触发一次，每次停顿 2 秒。用户正在经历「系统卡顿」的噩梦。

这不是演习，这是 Full GC 失控的经典场景。

Full GC 是 Java 应用中最需要警惕的 GC 事件。它意味着整个堆（新生代 + 老年代）都需要回收，通常伴随着长时间的 STW（Stop The World）停顿。一次 Full GC 可能从几十毫秒到几秒不等，如果频繁触发，对用户体验是灾难性的。

这篇文章的目标：**彻底搞懂 Full GC 的触发机制，掌握七大触发原因的分析方法，积累从日志到根因的排查流程。**

## 一、Full GC 基础：从概念到分类

### 1.1 什么是 Full GC？

Full GC（Full Garbage Collection）是回收整个 Java 堆的垃圾收集事件。与 Minor GC（只回收年轻代）不同，Full GC 需要同时扫描和回收年轻代和老年代的所有垃圾对象。

Full GC 的关键特征：

- **回收范围**：整个堆（年轻代 + 老年代 + 可能的 Metaspace）
- **停顿类型**：STW（Stop The World），所有用户线程暂停
- **停顿时间**：通常比 Minor GC 长一个数量级（几十毫秒到几秒不等）
- **触发原因**：多种多样，可能是正常行为，也可能是问题信号

### 1.2 Full GC vs Minor GC vs Mixed GC

不同 GC 类型有不同的触发条件和行为特征：

| 特征 | Minor GC | Full GC | Mixed GC |
|---|---|---|---|
| 回收范围 | 年轻代（Eden + Survivor） | 整个堆 | 年轻代 + 部分老年代 |
| 停顿时间 | 短（通常 < 100ms） | 长（100ms - 数秒） | 中等 |
| 触发条件 | Eden 区满 | 多种原因（见下文） | G1 特定条件 |
| STW | 是 | 是 | 是 |
| 频率 | 高 | 低（正常情况下） | G1 使用 |

```
GC 类型触发时机对比
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Minor GC（Serial/ParNew/PS Scavenge）：
  Eden 区无法为新对象分配空间时触发
  └─ 停顿时间：短
  └─ 频率：高

Mixed GC（G1）：
  老年代占用率达到 IHOP 时触发
  └─ 停顿时间：中等（可通过 MaxGCPauseMillis 控制）
  └─ 频率：取决于对象晋升速率

Full GC（Serial/PS MarkSweep/CMS/Serial Old）：
  多种原因触发
  └─ 停顿时间：长
  └─ 频率：正常情况应极少

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 1.3 为什么要警惕 Full GC？

Minor GC 频繁不会造成严重问题，因为年轻代空间小、存活对象少，停顿时间短。但 Full GC 不同：

**停顿时间长**：Full GC 需要扫描和回收整个堆，包括老年代的死亡对象。如果老年代很大，或者存活对象很多，停顿时间可能达到数秒。

**用户可感知**：一次 2 秒的 Full GC 会导致接口超时、用户请求堆积。如果 Full GC 每分钟触发一次，系统将持续处于「卡顿-恢复-卡顿」的循环中。

**可能引发连锁反应**：Full GC 期间，对象分配被阻塞。如果 Full GC 时间过长，可能导致线程池饱和、连接超时、熔断触发。

```
Full GC 影响链
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Full GC 触发
  └─ STW 停顿（2-5 秒）
      └─ 请求堆积
          └─ 线程池饱和
              └─ 接口超时
                  └─ 用户投诉 / 熔断触发
                      └─ 业务损失

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 二、Full GC 触发的七大原因

理解 Full GC 的触发原因，是排查问题的第一步。根据实践经验，Full GC 触发原因可以分为七大类。

### 2.1 老年代空间不足

**触发机制**：当年轻代中的存活对象需要晋升到老年代，但老年代没有足够的连续空间时，触发 Full GC。

这是最常见的 Full GC 触发原因。老年代空间不足通常发生在以下场景：

- 对象分配速率过高：大量短生命周期对象在 Minor GC 后进入老年代
- 内存泄漏：某些对象被意外持有，生命周期超出预期
- 大对象直接分配：超过 Survivor 区大小的对象直接进入老年代
- 晋升阈值设置不当：对象过早晋升到老年区

```bash
# 老年代空间不足的典型日志
[Full GC (Allocation Failure) 
     524288K->524288K(786432K), 5.6789012 secs]
# Allocation Failure 表示分配失败，老年代空间不足
```

**排查方法**：

```bash
# 1. 查看堆使用情况
jmap -heap <pid>

# 2. 监控老年代使用率
jstat -gcutil <pid> 1000

# 3. 查看对象年龄分布
jmap -heap <pid> | grep -A 20 "tenuring"
```

### 2.2 System.gc() 调用

**触发机制**：代码中显式调用 `System.gc()` 或 `Runtime.getRuntime().gc()`，JVM 为了响应这个调用，会尝试执行 Full GC。

在 JDK 8 中，CMS 默认会尝试 Full GC。虽然可以通过参数优化这个行为，但 `System.gc()` 的调用本身就是一个代码问题。

常见的触发场景：

- **NIO DirectByteBuffer**：没有显式释放堆外内存时，可能触发 Full GC
- **日志框架**：某些日志框架在特定条件下调用 `System.gc()`
- **旧代码遗留**：很多老代码中保留了显式 GC 调用
- **JMX 工具**：通过 JMX 触发 GC

```bash
# System.gc() 触发的 Full GC 日志
[Full GC (System.gc()) 
     123456K->78901K(786432K), 0.2345678 secs]
# System.gc() 表示由显式调用触发
```

**解决方案**：

```bash
# 方案一：禁用显式 GC（不响应 System.gc() 调用）
-XX:+DisableExplicitGC

# 方案二：排查代码中的 System.gc() 调用
# 使用 grep 搜索代码
grep -r "System.gc()" --include="*.java" .

# 方案三：使用 -XX:+ExplicitGCInvokesConcurrent
# 让 System.gc() 触发并发 GC（而非 Full GC）
-XX:+ExplicitGCInvokesConcurrent
```

:::warning 注意

禁用显式 GC 可能导致堆外内存（DirectByteBuffer）无法及时回收。如果你的应用使用了 NIO，需要谨慎处理。

:::

### 2.3 元空间（Metaspace）不足

**触发机制**：当加载的类数量过多，导致 Metaspace 满时，触发 Full GC 尝试回收不再使用的类。

Metaspace 不足通常发生在以下场景：

- **大量动态类生成**：如 Spring、CGLIB、ASM 框架在运行时生成代理类
- **热部署场景**：频繁卸载和加载类
- **反射滥用**：大量使用 Class.forName() 加载类
- **框架使用不当**：如 MyBatis 扫描大量 Mapper 接口

```bash
# Metaspace 不足的 Full GC 日志
[Full GC (Metadata GC Threshold) 
     123456K->78901K(524288K), 0.3456789 secs]
# Metadata GC Threshold 表示元空间不足
```

```bash
# 排查 Metaspace 使用情况
jstat -gc <pid>

# 查看 Metaspace 使用详情
jmap -clstats <pid>
```

**解决方案**：

```bash
# 方案一：增大 Metaspace 大小
-XX:MetaspaceSize=256m -XX:MaxMetaspaceSize=512m

# 方案二：排查类加载问题
# 使用 -verbose:class 打印类加载日志

# 方案三：检查框架配置
# Spring: 减少 component-scan 范围
# MyBatis: 检查 mapper 扫描配置
```

### 2.4 CMS 浮动垃圾满（Concurrent Mode Failure）

这是 CMS 特有的问题，但也是 Full GC 的常见原因。

**触发机制**：CMS 并发标记阶段产生的浮动垃圾过多，导致并发清理无法在老年代满之前完成。此时 CMS 退化到串行 Full GC。

详细分析见 [CMS GC 调优实战](./cms-tuning.mdx) 中的「Concurrent Mode Failure」章节。

```bash
# Concurrent Mode Failure 日志
[CMS-concurrent-mark-abortable-preclean-start]
CMS: abort preclean due to timeout
[Full GC (Allocation Failure) 
     524288K->524288K(786432K), 8.9012345 secs]
```

**解决方案**：

```bash
# 方案一：降低 CMS 触发阈值
-XX:CMSInitiatingOccupancyFraction=70

# 方案二：禁用自适应触发
-XX:+UseCMSInitiatingOccupancyOnly

# 方案三：升级到 G1
-XX:+UseG1GC
```

### 2.5 G1 的 Old 区分配担保失败

这是 G1 GC 特有的 Full GC 触发原因。

**触发机制**：G1 的 Mixed GC 需要回收老年代中的 Region，但老年代空间不足以容纳年轻代晋升的对象。此时触发 Full GC（通常是串行 Full GC，G1 的 Full GC 模式）。

G1 的 Full GC 是单线程的（标记-压缩），停顿时间非常长，是需要极力避免的情况。

```bash
# G1 Full GC 日志
[Full GC (Allocation Failure) 
     524288K->524288K(786432K), 10.2345678 secs]
# Allocation Failure 在 G1 中通常表示 Old 区不足
```

**解决方案**：

```bash
# 方案一：增大堆内存
-Xmx8g -Xms8g

# 方案二：调整 Mixed GC 触发时机
-XX:InitiatingHeapOccupancyPercent=45   # 默认 45%

# 方案三：优化 Humongous 对象（超过 Region 50% 的对象）
-XX:G1HeapRegionSize=4m   # 增大 Region 大小

# 方案四：调整 Mixed GC 回收比例
-XX:G1MaxNewSizePercent=60
```

### 2.6 JVM 切换 GC 策略

**触发机制**：当 JVM 启动参数中指定的 GC 策略发生变化时，可能触发 Full GC。

常见的场景：

- **CMS 退化**：CMS 出现 Concurrent Mode Failure 后，可能退化到 Serial Old
- **GC 策略热更新**：通过 JMX 或工具动态修改 GC 策略
- **参数冲突**：某些 GC 参数组合可能导致 JVM 回退到更保守的 GC

```bash
# GC 策略切换的日志
[Full GC (Allocation Failure) 
     524288K->524288K(786432K), 5.6789012 secs]
[GC (CMS Initial Mark)   # 之后切换到 CMS
```

**解决方案**：

```bash
# 方案一：确保 GC 参数正确且兼容
# 检查是否有参数冲突或回退

# 方案二：避免动态修改 GC 策略
# 生产环境不要通过 JMX 动态修改 GC
```

### 2.7 老年代分配担保失败

**触发机制**：Minor GC 之前，JVM 会检查老年代最大可用连续空间是否大于历次 Minor GC 晋升到老年代的对象平均大小。如果不满足，触发 Full GC（提前担保失败）。

这是 JVM 的保护机制：如果老年代空间不够容纳 Minor GC 晋升的对象，就先 Full GC 清理老年代。

```bash
# 分配担保失败的日志
[Full GC (Allocation Failure) 
     524288K->524288K(786432K), 5.6789012 secs]
# 或
[GC (Allocation Failure) 
     Desired survivor size 524288 bytes, new threshold 6 (max 6)
CMS: promote overflow in CMS collection, will not objects to compact]
```

**解决方案**：

```bash
# 方案一：增大老年代空间
-Xmx4g -Xms4g -XX:NewRatio=2   # 增大老年代比例

# 方案二：增大年轻代
-Xmn1g   # 增大年轻代大小

# 方案三：调整晋升阈值
-XX:MaxTenuringThreshold=15

# 方案四：禁用分配担保（不推荐）
-XX:-HandlePromotionFailure
```

## 三、排查工具与实战流程

### 3.1 GC 日志分析

GC 日志是排查 Full GC 的第一步。开启 GC 日志的参数：

```bash
# 基础配置
-XX:+PrintGCDetails           # 详细日志
-XX:+PrintGCDateStamps        # 时间戳
-Xloggc:/path/to/gc.log       # 日志文件

# 推荐配置（生产环境）
-XX:+PrintGCDetails
-XX:+PrintGCDateStamps
-XX:+PrintGCTimeStamps        # JVM 启动后的相对时间
-XX:+UseGCLogFileRotation     # 日志轮转
-XX:NumberOfGCLogFiles=10     # 保留 10 个文件
-XX:GCLogFileSize=100M        # 每个文件 100MB
-Xloggc:/path/to/gc.log
```

GC 日志中识别 Full GC 的关键词：

```log
[Full GC   # Serial GC / Parallel GC Full GC
[GC (Allocation Failure)    # CMS / G1 Full GC（需要结合上下文）
[Full GC (System.gc())      # 显式 System.gc() 触发
[Full GC (Metadata GC Threshold)  # Metaspace 不足触发
```

### 3.2 jstat 实时监控

jstat 是 JDK 自带的监控工具，可以实时查看 GC 状态：

```bash
# 查看 GC 统计信息（每 1 秒刷新一次）
jstat -gcutil <pid> 1000

# 输出
  S0     S1     E      O      M     CCS    YGC     YGCT    FGC    FGCT     GCT   
 12.45   0.00  67.89  85.67  92.34  88.56     12    0.345    5    2.345   2.690
```

**关键字段解释**：

| 字段 | 含义 |
|---|---|
| S0 / S1 | Survivor 0 / Survivor 1 使用率 |
| E | Eden 区使用率 |
| O | Old 区使用率 |
| M | Metaspace 使用率 |
| YGC | 年轻代 GC 次数 |
| YGCT | 年轻代 GC 总耗时 |
| FGC | Full GC 次数 |
| FGCT | Full GC 总耗时 |

**关注指标**：

- **FGC 增长速率**：如果 FGC 每分钟增长 1 次以上，说明有问题
- **FGCT 累积**：FGCT 持续增长，说明 Full GC 耗时不低
- **O（Old 区使用率）**：持续高于 80% 可能有风险

## 四、每种原因的解决思路

### 4.1 老年代空间不足

**分析思路**：

1. 查看 Old 区使用率是否持续高位
2. 查看对象年龄分布，是否有过早晋升
3. 检查是否有内存泄漏

**解决方案**：

```bash
# 方案一：增大堆内存
-Xmx4g -Xms4g

# 方案二：调整老年代/年轻代比例
-XX:NewRatio=2   # 老年代是年轻代的 2 倍

# 方案三：增大 Survivor 区，减少过早晋升
-XX:SurvivorRatio=8

# 方案四：增大晋升阈值
-XX:MaxTenuringThreshold=15

# 方案五：排查内存泄漏（见下文）
```

### 4.2 System.gc() 调用

**分析思路**：

1. 搜索代码中的 `System.gc()` 调用
2. 检查是否有框架触发的 GC
3. 检查 DirectByteBuffer 使用情况

**解决方案**：

```bash
# 方案一：禁用显式 GC
-XX:+DisableExplicitGC

# 方案二：让 System.gc() 触发并发 GC
-XX:+ExplicitGCInvokesConcurrent

# 方案三：移除代码中的 System.gc() 调用
# 搜索并删除
grep -r "System.gc()" --include="*.java" .
grep -r "Runtime.getRuntime().gc()" --include="*.java" .
```

### 4.3 元空间不足

**分析思路**：

1. 查看 Metaspace 使用率
2. 检查类加载数量
3. 检查动态类生成框架使用情况

**解决方案**：

```bash
# 方案一：增大 Metaspace
-XX:MetaspaceSize=512m -XX:MaxMetaspaceSize=1024m

# 方案二：开启类加载日志，排查问题
-XX:+TraceClassLoading -XX:+TraceClassUnloading

# 方案三：检查框架配置
# Spring: 减少 component-scan 范围
# MyBatis: 检查 mapper.xml 扫描
```

### 4.4 CMS 浮动垃圾问题

**解决方案**：

```bash
# 方案一：降低 CMS 触发阈值
-XX:CMSInitiatingOccupancyFraction=70

# 方案二：强制使用设定值
-XX:+UseCMSInitiatingOccupancyOnly

# 方案三：增加 Full GC 后的压缩
-XX:CMSFullGCsBeforeCompaction=3

# 方案四：升级到 G1
-XX:+UseG1GC
```

### 4.5 G1 Old 区问题

**解决方案**：

```bash
# 方案一：增大堆内存
-Xmx8g -Xms8g

# 方案二：调整 Mixed GC 触发时机
-XX:InitiatingHeapOccupancyPercent=40

# 方案三：增大 Region 大小
-XX:G1HeapRegionSize=4m

# 避免超过 Region 50% 的对象
```

## 五、实战案例分析

### 5.1 案例一：CMS 并发模式失败导致 Full GC

**问题背景**

某电商促销系统，JDK 8 + CMS GC。促销期间 Full GC 每分钟触发一次，每次停顿 5-8 秒。用户反映页面响应慢，购物车操作超时。

**排查过程**：

```bash
# 1. 查看 GC 日志
$ cat gc.log | grep "Full GC"

[Full GC (Allocation Failure) 524288K->524288K(786432K), 8.9012345 secs]
[Full GC (Allocation Failure) 524288K->524288K(786432K), 7.6543210 secs]
[Full GC (Allocation Failure) 524288K->524288K(786432K), 9.0123456 secs]
# 每分钟 Full GC 一次，停顿 7-9 秒
```

```bash
# 2. jstat 监控
$ jstat -gcutil <pid> 1000

  S0     S1     E      O      M     CCS    YGC     YGCT    FGC    FGCT     GCT   
  0.00  12.45  67.89  98.34  92.34  88.56     45    1.234    3    25.678   26.912
# Old 区使用率 98.34%，接近满
```

**根因分析**：

1. 老年代使用率 98%，CMS 触发阈值 92%，理论上 CMS 应该早就触发了
2. 但 CMS 并发阶段无法在老年代满之前完成，导致 Concurrent Mode Failure
3. 促销期间对象分配速率过高，CMS 清理速度跟不上

**解决方案**：

```bash
# 1. 降低 CMS 触发阈值，给 CMS 更多清理时间
-XX:CMSInitiatingOccupancyFraction=70

# 2. 强制使用设定值
-XX:+UseCMSInitiatingOccupancyOnly

# 3. 增大并发 GC 线程数
-XX:ConcGCThreads=3

# 4. 增加堆内存作为缓冲
-Xmx6g -Xms6g
```

**效果**：Full GC 频率从每分钟一次降低到每 10 分钟一次，停顿时间从 8 秒降低到 500ms。

### 5.2 案例二：元空间不足导致 Full GC

**问题背景**

某微服务使用 Spring Boot + MyBatis，启动 2 小时后开始频繁 Full GC。每次 Full GC 后 Metaspace 使用率迅速回升。

## 六、面试实战

### 6.1 场景一：Full GC 的触发原因

**面试官问**：「什么情况下会产生 Full GC？」

这道题是 Java 面试的高频题，考察对 GC 机制的理解深度。

> Full GC 有以下七种触发原因：
>
> 1. **老年代空间不足**：对象晋升时老年代不够
> 2. **显式调用 System.gc()**：代码中显式触发
> 3. **元空间不足**：加载的类过多
> 4. **CMS 并发模式失败**：浮动垃圾过多
> 5. **G1 Old 区分配担保失败**：Mixed GC 无法完成
> 6. **GC 策略切换**：JVM 切换 GC 策略
> 7. **分配担保失败**：Minor GC 前检查老年代空间不足
>
> 正常情况下，Full GC 应该很少发生。如果 Full GC 频繁，需要排查是老年代空间不足、对象过早晋升，还是内存泄漏。

### 6.2 场景二：Full GC 排查思路

**面试官问**：「生产环境 Full GC 频繁，你如何排查？」

这道题考察实战经验。

> 排查 Full GC 的步骤：
>
> 1. **开启 GC 日志**：`-XX:+PrintGCDetails -XX:+PrintGCDateStamps -Xloggc:gc.log`
>
> 2. **分析 GC 日志**：识别 Full GC 类型（System.gc / Allocation Failure / Metadata GC Threshold）
>
> 3. **使用 jstat 监控**：`jstat -gcutil <pid> 1000`，关注 Old 区使用率和 Full GC 次数
>
> 4. **使用 jmap 分析**：`jmap -heap` 查看堆使用，`jmap -histo` 查看对象分布
>
> 5. **定位根因**：
>
> - Old 区持续高位：增大堆内存或减少对象晋升
> - System.gc()：禁用显式 GC 或移除调用
> - Metaspace 不足：增大 Metaspace 或检查类加载
> - CMS 问题：调整 CMS 参数或迁移到 G1
>
> 6. **验证修复**：持续监控，确认 Full GC 频率降低

### 6.3 场景三：Minor GC 和 Full GC 的区别

**面试官问**：「Minor GC 和 Full GC 的区别是什么？」

> 主要区别：
>
> 1. **回收范围**：Minor GC 只回收年轻代，Full GC 回收整个堆
> 2. **停顿时间**：Minor GC 通常几十毫秒，Full GC 可能数百毫秒到数秒
> 3. **触发频率**：Minor GC 频繁（正常情况），Full GC 稀少（异常情况）
> 4. **GC 线程**：Minor GC 可以并行（Parallel New），Full GC 可能是串行
>
> 还有一个重要区别：**Minor GC 不一定触发 Full GC，但 Full GC 会暂停所有应用线程。**

## 总结

Full GC 是 JVM 性能问题的「晴雨表」。正常情况下 Full GC 应该稀少，如果频繁发生，一定是某个环节出了问题。

排查 Full GC 的关键是：

1. **识别类型**：从 GC 日志中判断是哪种 Full GC
2. **定位根因**：根据类型分析可能的原因
3. **针对性解决**：不同原因有不同的解决方案
4. **持续监控**：确认问题不再复发

记住：Full GC 不是洪水猛兽，但它是信号。当你看到 Full GC 日志时，第一反应应该是：「系统在告诉我什么？」
