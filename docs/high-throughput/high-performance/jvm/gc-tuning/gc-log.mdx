# GC 日志分析工具与实操

凌晨 2 点，你被监控告警叫醒：GC 停顿超过 2 秒。用户反馈接口超时 SLA 破标，你打开 GC 日志，满屏的日志信息却不知从何看起——这几乎是每个 Java 开发者都会经历的噩梦。

GC 日志是 JVM 调优的「黑匣子」，它记录了每一次垃圾回收的完整轨迹。但大多数人对 GC 日志的态度是「出事再看」，殊不知 GC 日志的解读能力直接决定了问题定位的效率。本节将从日志配置到工具实操，带你彻底掌握 GC 日志分析。

---

## GC 日志参数配置

GC 日志不是凭空产生的，需要在 JVM 启动时显式开启。不同 JDK 版本推荐的配置方式有所不同。

### JDK 8 及之前版本

```bash
java -XX:+PrintGCDetails \
     -XX:+PrintGCDateStamps \
     -Xloggc:/var/log/myapp/gc.log \
     -XX:+UseGCLogFileRotation \
     -XX:NumberOfGCLogFiles=5 \
     -XX:GCLogFileSize=10M \
     -jar myapp.jar
```

| 参数 | 说明 |
|---|---|
| `-XX:+PrintGCDetails` | 输出详细 GC 信息，包括各区域回收前后大小、耗时 |
| `-XX:+PrintGCDateStamps` | 在每行日志前添加人类可读的时间戳（格式：`2024-01-15T10:30:45.123`） |
| `-Xloggc:/path/to/gc.log` | 指定 GC 日志输出文件路径 |
| `-XX:+UseGCLogFileRotation` | 启用日志轮转，避免单个文件过大 |
| `-XX:NumberOfGCLogFiles=5` | 保留的日志文件数量 |
| `-XX:GCLogFileSize=10M` | 每个日志文件的最大大小 |

### JDK 9+ 统一日志框架

JDK 9 引入了统一的日志框架，语法更加灵活和标准化：

```bash
java -Xlog:gc*:file=/var/log/myapp/gc.log:time,uptime,level,tags:filecount=5,filesize=10M \
     -jar myapp.jar
```

参数解析：

- `gc*` — 匹配所有以 `gc` 开头的日志标签（可细化如 `gc,gc+ref`）
- `file=gc.log` — 输出文件路径
- `time,uptime,level,tags` — 日志包含的字段
  - `time` — 绝对时间
  - `uptime` — JVM 启动后的相对时间（秒）
  - `level` — 日志级别
  - `tags` — 日志标签
- `filecount=5,filesize=10M` — 轮转配置

:::tip 推荐配置

生产环境推荐使用 JDK 9+ 的统一日志框架，输出字段更丰富，便于后续分析：

```bash
-Xlog:gc*=info:file=gc.log:time,uptime,level,tags:filecount=10,filesize=50M
```

:::

## GC 日志格式解读

不同垃圾收集器的日志格式差异很大，理解各收集器的日志格式是分析的第一步。

### Serial GC 日志

Serial GC 是最基础的垃圾收集器，单线程执行，适用于低配环境或客户端应用。

```bash
-XX:+UseSerialGC
```

```
2024-01-15T10:30:45.123: [GC (Allocation Failure) 
 Desired survivor size 524288 bytes, new threshold 1 (max 6)
  - age   1:   1048576 bytes,   1048576 total
  , 0.0123457 secs]
2024-01-15T10:30:45.135: [Full GC (System.gc()) 
  [DefNew: 65536K->0K(65536K), 0.0456789 secs]
  [Tenured: 180224K->165312K(409600K), 0.0789012 secs]
  [Metaspace: 32768K->32768K(32768K)], 0.1245801 secs
```

### Parallel GC 日志

Parallel GC（吞吐量优先）是批处理系统的默认选择，日志中会显示各区域详细信息。

```
2024-01-15T10:30:45.123: [GC (Allocation Failure) 
  [PSYoungGen: 262144K->32768K(306688K)] 
  524288K->262144K(786432K), 
  0.0234567 secs] 
  [Times: user=0.08 sys=0.01, real=0.02 secs]
```

关键字段解读：

- `PSYoungGen` — Parallel Scavenge 年轻代
- `262144K->32768K(306688K)` — 回收前大小->回收后大小(总容量)
- `Times: user/sys/real` — 用户态时间/内核态时间/实际耗时

### CMS GC 日志

CMS（Concurrent Mark Sweep）是低停顿收集器，日志最为复杂，包含多个阶段。

```
2024-01-15T10:30:45.123: [GC (CMS Initial Mark) 
  [CMS-initial-mark: 262144K(524288K)] 262144K, 0.0123456 secs]
   duration: 0.0123456 secs, #0

2024-01-15T10:30:45.135: [GC (CMS-concurrent-mark-start)]
2024-01-15T10:30:46.234: [GC (CMS-concurrent-mark-end), 1.0987654 secs]

2024-01-15T10:30:46.234: [GC (CMS-concurrent-preclean-start)]
2024-01-15T10:30:46.345: [GC (CMS-concurrent-preclean-end), 0.1112345 secs]

2024-01-15T10:30:46.345: [GC (CMS Final Remark) 
  [GC concurrent-remark-start]
  , 0.0234567 secs]
   duration: 0.0234567 secs, #1

2024-01-15T10:30:46.368: [GC (CMS-concurrent-sweep-start)]
2024-01-15T10:30:47.456: [GC (CMS-concurrent-sweep-end), 1.0887654 secs]
```

CMS 的五个阶段：

| 阶段 | 说明 | STW |
|---|---|---|
| Initial Mark | 标记 GC Roots 直接引用的对象 | 是 |
| Concurrent Mark | 并发标记，追踪存活对象 | 否 |
| Preclean | 预处理，修正并发期间的变更 | 否 |
| Final Remark | 最终标记，完成所有追踪 | 是 |
| Concurrent Sweep | 并发清除死亡对象 | 否 |

:::warning CMS 参数说明

JDK 14 已移除 CMS 收集器（JEP 363）。如果还在使用 JDK 8/11，建议逐步迁移到 G1 或 ZGC。但理解 CMS 日志对排查遗留系统仍有价值。

:::

### G1 GC 日志

G1（Garbage First）是 JDK 9+ 的默认收集器，采用分区式架构，日志格式最能反映其工作原理。

```
2024-01-15T10:30:45.123: [GC pause (G1 Evacuation Pause) (young)
  , 0.0234567 secs]
   [Eden: 256.0M(256.0M)->0.0B(256.0M)
    Survivors: 32.0M->32.0M
    Heap: 512.0M(1024.0M)->384.0M(1024.0M)]
   GC workers: 4
    Duration: 20.1 ms, Serial: 0.0 ms
```

年轻代收集（Young GC）关键字段：

- `G1 Evacuation Pause (young)` — 年轻代 GC，将活对象复制到 Survivor 或 Old 区
- `Eden` — 伊甸园区，回收后清空
- `Survivors` —  Survivor 区，对象 age-up
- `Heap` — 整个堆的 before->after(总容量)

Mixed GC 日志：

```
2024-01-15T10:30:50.123: [GC pause (G1 Evacuation Pause) (mixed)
  , 0.0456789 secs]
   [Eden: 128.0M(128.0M)->0.0B(128.0M)
    Survivors: 16.0M->32.0M
    Heap: 768.0M(1024.0M)->512.0M(1024.0M)]
   8 regions are selected
```

Mixed GC 会同时回收年轻代和部分老年代，`X regions are selected` 表示本次选中的老年代 Region 数量。

## GCViewer 使用

GCViewer 是一款开源的 GC 日志可视化工具，支持 JDK 7/8/9+ 各版本格式。

### 安装与运行

```bash
# 下载最新版本
wget https://github.com/chewiebug/GCViewer/releases/download/v1.34.1/ gcviewer-1.34.1.jar

# 直接运行
java -jar gcviewer-1.34.1.jar
```

也可以直接在 IDE 中查看分析结果。

### 核心指标解读

打开 GC 日志后，GCViewer 会展示多个图表：

**GC 次数趋势图（GC Count）**

展示 GC 发生的频率，Y 轴是次数。如果 Young GC 次数持续很高（比如每分钟超过 10 次），说明年轻代配置可能偏小。

**内存使用趋势图（Memory Usage）**

展示堆内存各区域的使用情况。理想状态是年轻代和老年代都呈现「锯齿状」——回收后下降，逐步上升，再回收。如果老年代呈持续上升趋势，说明可能有内存泄漏。

**吞吐量趋势图（Throughput）**

吞吐量 `= (非 GC 时间 / 总运行时间) × 100%`。对于高性能应用，吞吐量应保持在 95% 以上。如果低于 90%，需要分析 GC 对应用的影响。

### 识别常见问题

**GC 过于频繁**

表现：Young GC 次数曲线呈现高密度峰值。

原因：年轻代太小，或者对象分配速率过高。

**内存持续增长**

表现：老年代内存曲线持续上升，无法回到基线。

原因：可能存在内存泄漏，需要结合 heap dump 分析。

**Full GC 过多**

表现：Full GC 次数曲线出现陡增。

原因：老年代空间不足，或 CMS GC 出现并发模式失败。

## gceasy.io 使用

[gceasy.io](https://gceasy.io) 是在线 GC 日志分析工具，界面友好且支持 AI 智能诊断，适合快速定位问题。

### 使用步骤

1. 打开 [gceasy.io](https://gceasy.io)
2. 点击「Upload GC Log」上传 gc.log 文件
3. 自动生成分析报告

### 关键指标

| 指标 | 含义 | 健康值 |
|---|---|---|
| Throughput | 应用程序有效运行时间占比 | `>` 95% |
| GC Cause | GC 触发原因分析 | 避免 Allocation Failure 频繁 |
| Total GC Time | GC 总耗时 | 越低越好 |
| Max GC Pause | 最大单次 GC 停顿 | 满足业务 SLA |
| Avg GC Pause | 平均 GC 停顿 | 满足业务 SLA |

### 问题报告

gceasy.io 会自动识别常见 GC 问题：

- **Long GC Pauses** — 识别出停顿过长的 GC 事件
- **GC Recommendation** — 基于分析结果给出调优建议
- **Memory Leak** — 检测可能的内存泄漏趋势

:::tip gceasy vs GCViewer

gceasy.io 胜在**便捷**，无需安装，直接上传即可分析。GCViewer 胜在**灵活**，支持自定义过滤和本地离线分析。生产环境问题排查建议两者结合使用——先用 gceasy 快速定位问题类型，再用 GCViewer 深入分析细节。

:::

## GC 数学分析

理解 GC 的数学原理，有助于从根本上把握调优方向。

### GC 频率计算

```
GC 频率 = 对象分配速率 / 堆大小
```

例如：应用每秒分配 500MB 对象，堆大小 2GB，则 Minor GC 频率约为：

```
500MB / (2GB × 1/3) = 500MB / 约640MB ≈ 每1.3秒一次
```

### 吞吐量计算

```
吞吐量 = (运行时间 - GC 时间) / 运行时间 × 100%
```

例如：应用运行 1 小时，GC 总耗时 3 分钟，则吞吐量为：

```
吞吐量 = (60分钟 - 3分钟) / 60分钟 × 100% = 95%
```

### 停顿时间分析

```
平均停顿时间 = GC 总耗时 / GC 次数
P99 停顿时间 = 排序后第 99 百分位的停顿时间
```

P99 停顿时间才是关键指标——平均值好看，但如果有一次停顿超过 2 秒，用户仍会感知到卡顿。

## 日志分析实战案例

### 案例一：Minor GC 过频

**现象**

监控系统显示 Young GC 每 3 秒触发一次，累计停顿时间占总运行时间 8%。

**日志分析**

```
2024-01-15T10:30:45.123: [GC (Allocation Failure) 
  [PSYoungGen: 32768K->4096K(49152K)] 
  98304K->32768K(131072K), 
  0.0156789 secs]
```

**问题定位**

- Survivor 区 49152K（48MB），但回收后剩余 4096K（4MB）
- 说明每次 Minor GC 后只有少量对象存活，但 Survivor 区明显偏大
- 年轻代总容量 131072K（128MB），而 Eden 区 98304K（96MB）
- Eden 区使用率达到 100%，说明对象分配速率极高

**优化方案**

增大年轻代，减少 Minor GC 频率：

```bash
-XX:NewRatio=1           # 年轻代:老年代 = 1:1（原值可能是 2:1）
-XX:NewSize=512M         # 直接指定年轻代大小
-XX:MaxNewSize=512M
```

或者调整 Survivor 区比例：

```bash
-XX:SurvivorRatio=6      # Eden:Survivor = 6:1（原值可能是 8）
```

**优化后效果**

Minor GC 频率从每 3 秒一次降低到每 15 秒一次，吞吐量提升至 98%。

### 案例二：CMS 重标记过长

**现象**

CMS GC 的 Final Remark 阶段停顿时间偶尔超过 500ms，影响业务响应。

**日志分析**

```
2024-01-15T10:30:45.135: [GC (CMS Final Remark) 
  [GC concurrent-remark-start]
  , 0.5234567 secs]
   duration: 0.5234567 secs, #1
```

**问题定位**

CMS Final Remark 阶段需要重新扫描 GC Roots，如果老年代对象很多，扫描时间会很长。

**优化方案**

开启 CMSScavengeBeforeRemark，在 Final Remark 前先进行一次 Young GC：

```bash
-XX:+UseCMSInitiatingOccupancyOnly
-XX:CMSInitiatingOccupancyFraction=70
-XX:+CMSScavengeBeforeRemark   # JDK 8 新增参数
```

CMSScavengeBeforeRemark 的作用：

- 在 Final Remark 前先清理年轻代
- 减少需要扫描的老年代对象数量
- 降低 Final Remark 的扫描范围

**优化后效果**

Final Remark 停顿从 523ms 降低到 120ms。

:::tip CMS 已废弃

CMS 收集器在 JDK 14 被移除（JEP 363）。如果使用的是 JDK 8，建议逐步迁移到 G1。G1 的停顿时间可控性更强，且不需要手动处理 CMS 的并发模式失败问题。

:::

## 总结

GC 日志是 JVM 调优的基础数据来源。本节核心要点：

- **日志配置**：`PrintGCDetails` + `PrintGCDateStamps` + 日志轮转是标配
- **格式解读**：不同收集器日志格式差异大，需要逐个掌握
- **工具使用**：gceasy.io 快速定位，GCViewer 深入分析
- **数学基础**：GC 频率、吞吐量、P99 停顿是核心指标

## 面试实战

### 面试题一：如何分析 GC 日志定位性能问题？

**参考回答**：

首先开启详细 GC 日志，记录每次 GC 的时间、类型、各区域变化。然后使用 GCViewer 或 gceasy.io 分析。

关注三个维度：

1. **GC 频率**：如果 Young GC 过于频繁，可能是年轻代太小或对象分配速率过高
2. **内存趋势**：如果老年代持续上升，可能存在内存泄漏
3. **停顿时间**：P99 停顿时间才是关键，关注是否满足业务 SLA

### 面试题二：CMS 和 G1 的日志有什么区别？

**参考回答**：

CMS 日志分为五个阶段（Initial Mark、Concurrent Mark、Preclean、Final Remark、Concurrent Sweep），重点关注 Final Remark 的停顿时间。

G1 日志以 Region 为单位，显示 Eden、Survivor、Old 区的变化。Mixed GC 会显示选中回收的老年代 Region 数量。G1 的停顿时间由 `-XX:MaxGCPauseMillis` 控制。

## 思考题

1. 应用运行 24 小时后，GC 日志显示老年代从 500MB 持续增长到 800MB（堆总大小 1GB），但没有触发 Full GC——这种情况可能是什么问题？如何验证你的判断？

2. G1 的吞吐量目标是 90%，但实际只有 85%，你会从哪些方面入手调优？
