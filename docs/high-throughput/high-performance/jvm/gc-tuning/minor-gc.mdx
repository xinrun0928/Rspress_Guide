# Minor GC 频繁的原因与优化

Minor GC 每次只有几十毫秒，对应用的影响几乎可以忽略。但如果它一分钟触发 100 次呢？

每次停顿虽然短，但累积起来就是几秒钟的 CPU 浪费。更重要的是，Minor GC 频繁往往是更深层问题的信号——可能是年轻代空间设置不合理，可能是对象分配策略有问题，也可能是应用代码在「制造」大量短生命周期对象。

这篇文章，我们从 Minor GC 的执行原理出发，彻底搞清楚：为什么 Minor GC 会频繁触发，以及如何从根本上解决问题。

## 一、Minor GC 执行原理：STW 到底发生了什么

### 1.1 触发条件：Eden 区满

Minor GC 的触发条件很简单：**当 Eden 区无法为新对象分配空间时**。

```
对象分配流程
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

新对象分配请求
      │
      ▼
┌─────────────────────────────────────────────────────────────┐
│  Eden 区有足够空间？                                         │
│                                                              │
│  ├─ 是 → 在 Eden 区分配对象                                  │
│  │                                                          │
│  └─ 否 → 触发 Minor GC                                       │
└─────────────────────────────────────────────────────────────┘

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

这个「足够空间」的判断，本质上是一次内存分配指针检查。HotSpot 采用的是 TLAB（Thread Local Allocation Buffer）机制，每个线程在 Eden 区预分配一小块空间，线程在自己的 TLAB 内分配对象，避免了多线程竞争。

### 1.2 执行过程：STW → 标记 → 复制 → 交换

Minor GC 的执行过程分为四个步骤，全程 STW：

**第一步：Stop The World**

JVM 发起 Safe Point，所有应用线程停止。这是所有 GC 都需要经历的阶段，因为 GC 过程中对象引用关系不能发生变化。

**第二步：GC Roots 标记**

从 GC Roots 出发，遍历年轻代所有存活对象的引用链。GC Roots 包括：

- 虚拟机栈（栈帧中的本地变量表）中引用的对象
- 方法区中类静态属性引用的对象
- 方法区中常量引用的对象
- 本地方法栈中 JNI 引用的对象

**第三步：复制存活对象**

标记完成后，GC 遍历年轻代，将存活对象复制到 Survivor 区。HotSpot 默认使用 Copying 算法，将 Eden 区和 From Survivor 区的存活对象复制到 To Survivor 区。

```
复制算法示意
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

GC 前：
┌─────────────┬─────────────┬─────────────┐
│    Eden     │ From Survivor│ To Survivor │
│  (大部分满)  │  (少量存活)   │    (空)     │
└─────────────┴─────────────┴─────────────┘
         │              │
         ▼              ▼
         全部复制到 To Survivor 区
         │
         ▼
GC 后：
┌─────────────┬─────────────┬─────────────┐
│    Eden     │ From Survivor│ To Survivor │
│    (空)     │    (空)      │ (存活对象)   │
└─────────────┴─────────────┴─────────────┘
         │              │
         ▼              ▼
         From 和 To 交换角色

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**第四步：交换 Survivor 区**

复制完成后，From Survivor 和 To Survivor 交换角色。下一次 Minor GC 时，原本的 To Survivor 就变成了 From Survivor。

### 1.3 Minor GC 的停顿时间由什么决定？

Minor GC 的停顿时间主要受以下因素影响：

- **年轻代大小**：Eden 区越大，Minor GC 频率越低，但每次停顿时间越长
- **存活对象数量**：Survivor 区存活对象越多，复制操作越耗时
- **GC Roots 数量**：跨代引用会增加 GC Roots 遍历的成本
- **垃圾收集器选择**：Serial GC 单线程执行，ParNew/PS Scavenge 多线程并行

```
Minor GC 停顿时间公式（简化版）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

T_minor_gc ≈ T_stop_world + T_mark + T_copy

其中：
- T_stop_world：线程停止时间（通常 1-5ms）
- T_mark：GC Roots 标记时间（与 GC Roots 数量成正比）
- T_copy：存活对象复制时间（与存活对象大小成正比）

关键洞察：
- 年轻代太小 → Eden 区快速填满 → Minor GC 频繁
- Survivor 区太小 → 对象无法充分 age-up → 过早晋升 → Full GC 增多

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 二、Minor GC 频繁的七大原因

Minor GC 频繁的本质是：**Eden 区空间相对于对象分配速率不足**。但具体原因有七种，每种都需要不同的分析思路。

### 2.1 原因一：年轻代太小

**问题描述**：年轻代总空间（Eden + Survivor）配置过小，导致 Eden 区快速填满。

**典型场景**：
- 物理机 8GB 内存，JVM 堆 4GB，年轻代只有 512MB
- 微服务应用，QPS 1000+，每秒创建大量对象

**如何判断**：

```bash
# 查看年轻代配置
jinfo -flag NewSize <pid>
jinfo -flag MaxNewSize <pid>
jinfo -flag NewRatio <pid>

# 示例输出
-XX:NewSize=524288000
-XX:MaxNewSize=524288000
-XX:NewRatio=8
```

`-XX:NewRatio=8` 意味着老年代是年轻代的 8 倍。如果堆是 4GB，年轻代只有 400MB 左右。

### 2.2 原因二：对象分配率高

**问题描述**：应用代码在短时间内创建大量短生命周期对象。

**典型场景**：
- JSON 序列化/反序列化：每次请求创建大量临时对象
- 字符串拼接：大量 `+` 操作产生临时 StringBuilder
- 集合操作：频繁创建 ArrayList、HashMap

**代码层面的问题**：

```java
// 问题代码：每次请求创建新的 StringBuilder
public String buildResponse(Request request) {
    StringBuilder sb = new StringBuilder();
    sb.append("result:");
    sb.append(request.getData());
    return sb.toString();
}

// 问题代码：循环内创建对象
public void process(List<Item> items) {
    for (Item item : items) {
        DTO dto = new DTO();  // 循环内创建，每个元素都新建
        dto.setName(item.getName());
        dto.setValue(item.getValue());
        // ...
    }
}

// 问题代码：方法内不断往 List 添加数据
public void addItems(List<String> list) {
    for (int i = 0; i < 1000; i++) {
        list.add("item" + i);  // String 不断创建
    }
}
```

### 2.3 原因三：Survivor 区太小

**问题描述**：Survivor 区空间不足，导致对象过早晋升到老年代。

Survivor 区的核心作用是**延迟对象晋升**。对象在年轻代需要经历多次 Minor GC 才能确认是「真正的垃圾」还是「长期存活对象」。如果 Survivor 区太小，对象无法在年轻代「待够时间」，就会被过早晋升。

** Survivor 区太小的影响链**：

```
Survivor 区太小
      │
      ▼
对象无法在 Survivor 区充分 age-up
      │
      ▼
年龄达到晋升阈值时强制晋升到老年代
      │
      ▼
老年代空间快速消耗
      │
      ▼
Full GC 频繁
```

### 2.4 原因四：对象 age 阈值设置不当

**问题描述**：MaxTenuringThreshold 设置过大或过小，导致对象晋升策略失衡。

HotSpot 中，对象的年龄（age）存储在对象头的 Mark Word 中。每次 Minor GC 后，对象的 age 加 1。当 age 达到 `MaxTenuringThreshold` 时，对象晋升到老年代。

**默认值**：

| GC 收集器 | MaxTenuringThreshold 默认值 |
|---|---|
| Serial GC | 15 |
| Parallel Scavenge | 15 |
| ParNew GC | 6（最大也是 15） |
| G1 | 动态计算（通过 TargetSurvivorRatio 调整） |

**动态年龄计算**：HotSpot 有一个优化：如果 Survivor 区中相同年龄的所有对象大小之和超过 Survivor 区的 50%，则 age 大于等于该年龄的对象全部晋升到老年代。这是为了应对突发的大量对象。

### 2.5 原因五：大对象直接进入老年代

**问题描述**：超过 `-XX:PretenureSizeThreshold` 阈值的对象直接分配在老年代。

这个参数本身是为了避免大对象在年轻代频繁复制（因为大对象复制成本高）。但如果设置不当，可能导致老年代空间快速消耗。

**典型场景**：
- 分页查询数据库，每页数据 10MB
- 文件上传处理，文件内容作为 byte[] 加载到内存

### 2.6 原因六：晋升阈值被动态调小

**问题描述**：JVM 运行时动态调小晋升阈值，导致对象过早晋升。

G1 以外的收集器（Serial、ParNew、Parallel Scavenge）会根据 Survivor 区的实际使用情况动态调整晋升阈值。如果 Survivor 区空间利用率较低，JVM 会倾向于让对象尽快晋升。

```bash
# 开启详细年龄打印，观察晋升阈值变化
-XX:+PrintTenuringDistribution

# 示例输出
Desired survivor size 5242880 bytes, new threshold 6 (max 15)
Age 1: 1048576 bytes, 1048576 total
Age 2: 524288 bytes, 1572864 total
Age 3: 262144 bytes, 1835008 total
```

`new threshold 6` 表示当前晋升阈值是 6（而不是默认的 15）。这意味着对象 age 达到 6 就会晋升。

### 2.7 原因七：跨代引用问题

**问题描述**：老年代对象持有年轻代对象的引用，导致 Minor GC 时需要扫描老年代。

这原本是一个性能问题——年轻代 GC 需要扫描老年代来找到跨代引用。但 HotSpot 引入 Card Table 机制解决了这个问题。

**Card Table 原理**：

```
Card Table 结构
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

老年代被划分为固定大小的卡片（Card），每张卡片 512 字节。
JVM 为老年代维护一个 Card Table，记录哪些卡片可能包含跨代引用。

Card Table 是一个字节数组，每个元素对应一张卡片：
- 0（clean）：卡片中的对象没有对年轻代的引用
- 1（dirty）：卡片中的对象可能引用了年轻代对象

Minor GC 时：
1. 遍历 GC Roots（不包括老年代）
2. 从 Card Table 找出所有 dirty 卡片
3. 只扫描 dirty 卡片中的对象，避免全表扫描老年代

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

Card Table 的引入，将老年代扫描成本从 O(老年代大小) 降到了 O(脏卡数量)。

## 三、对象晋升机制详解

### 3.1 年龄计数器

HotSpot 在对象头中为每个对象维护一个 age 计数器。对象头结构如下：

```markdown
|-------------------------------------------------------|
| Mark Word (64 bits)                                   |
|-------------------------------------------------------|
| bit 63-62: 对象年龄 (age)                             |
| bit 61: GC 分代年龄                                   |
| ...                                                   |
|-------------------------------------------------------|
```

年龄计数器只有 4 位（最大表示 15），当 age 达到 MaxTenuringThreshold 时，对象晋升到老年代。

### 3.2 晋升条件

对象晋升老年代需要满足以下条件之一：

- **年龄达到阈值**：age `>=` MaxTenuringThreshold
- **Survivor 区空间不足**：To Survivor 区无法容纳存活对象
- **动态年龄判定**：Survivor 区中相同年龄对象总大小 `>` Survivor 区的 50%

### 3.3 晋升失败

晋升失败（Promotion Failed）是 Minor GC 期间常见的问题：

```
晋升失败场景
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Minor GC 触发
      │
      ▼
Eden + From Survivor 的存活对象需要晋升到老年代
      │
      ▼
老年代空间不足或没有连续空间
      │
      ▼
触发 Concurrently Marking（CMS）或 Full GC

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

晋升失败通常意味着老年代空间不足，或者内存碎片化导致没有连续空间。此时系统会退化为更慢的 GC 策略。

## 四、排查方法与工具

### 4.1 -XX:+PrintTenuringDistribution

开启后，每次 Minor GC 会输出对象年龄分布：

```bash
java -XX:+PrintTenuringDistribution -Xms512m -Xmx512m -XX:NewSize=256m -XX:MaxNewSize=256m App

# 示例输出
2024-01-15T10:30:45.123-0800: [GC (Allocation Failure)
Desired survivor size 15728640 bytes, new threshold 6 (max 15)
Age 1: 10485760 bytes, 10485760 total
Age 2: 5242880 bytes, 15728640 total
Age 3: 2097152 bytes, 17825792 total
```

关键信息解读：

- `Desired survivor size`：期望的 Survivor 区大小
- `new threshold 6`：当前晋升阈值是 6（动态调整的结果）
- `Age N: X bytes`：年龄为 N 的对象总大小

### 4.2 jstat 查看 GC 统计

```bash
# 每 1 秒输出一次年轻代统计
jstat -gc <pid> 1000

# 示例输出
 S0C    S1C    S0U    S1U     EC       EU        OC         OU       MC     MU    CCSC   CCSU   YGC     YGCT    FGC    FGCT     GCT
```

关键指标：

- `S0C/S1C`：Survivor 0/1 容量

## 五、调优案例：微服务接口 Minor GC 每秒一次

### 5.1 问题背景

某微服务应用，QPS 约 500，每次请求会进行 JSON 序列化/反序列化。使用 ParNew + CMS 组合。

**监控数据**：

- Minor GC 频率：约每秒 1 次
- Minor GC 停顿：20-30ms
- Full GC 频率：约每 5 分钟 1 次
- Full GC 停顿：200-500ms

### 5.2 问题分析

```bash
# 开启 GC 日志
-XX:+PrintGCDetails -XX:+PrintGCDateStamps -Xloggc:/tmp/gc.log

# 分析日志发现：
# 1. Survivor 区使用率经常接近 100%
# 2. 对象 age 很快达到阈值（6-8）就晋升
# 3. 大量 short-lived 对象（age=1）占据 Eden 区
```

**根本原因**：

1. 年轻代配置过小（-Xmx2g -Xmn512m）
2. Survivor 区太小（-XX:SurvivorRatio=8，Eden:S0:S1 = 8:1:1）
3. JSON 序列化产生大量临时对象

### 5.3 优化方案

```bash
# 优化后的参数
-Xmx4g -Xmn2g                    # 增大年轻代（原来只有 512m）
-XX:SurvivorRatio=4              # 增大 Survivor 区比例（Eden:S0:S1 = 4:1:1）
-XX:MaxTenuringThreshold=15      # 提高晋升阈值
-XX:TargetSurvivorRatio=80       # 提高 Survivor 区目标使用率
```

**关键改动解读**：

- `-Xmn2g`：年轻代从 512MB 增大到 2GB，Eden 区从 400MB 增大到约 1.6GB
- `-XX:SurvivorRatio=4`：Survivor 区从约 50MB 增大到约 250MB
- `-XX:MaxTenuringThreshold=15`：让对象在年轻代停留更长时间
- `-XX:TargetSurvivorRatio=80`：减少动态晋升阈值的触发

### 5.4 优化效果

| 指标 | 优化前 | 优化后 |
|---|---|---|
| Minor GC 频率 | 每秒 1 次 | 每 5 秒 1 次 |
| Minor GC 停顿 | 20-30ms | 30-50ms |
| Full GC 频率 | 每 5 分钟 1 次 | 每 30 分钟 1 次 |
| Full GC 停顿 | 200-500ms | 100-200ms |

Minor GC 频率降低了 80%，Full GC 频率降低了 85%。

## 六、面试实战

### 面试题一：Minor GC 和 Full GC 的区别是什么？

**标准答案**：

Minor GC 回收年轻代，Full GC 回收整个堆。两者的主要区别：

| 维度 | Minor GC | Full GC |
|---|---|---|
| 回收范围 | Eden + Survivor | 年轻代 + 老年代 + Metaspace |
| 停顿时间 | 短（通常 < 100ms） | 长（100ms - 数秒） |
| 触发频率 | 高 | 低 |
| 是否STW | 是 | 是 |
| 常见收集器 | Serial/ParNew/PS Scavenge | Serial Old/PS MarkSweep/CMS |

**追问**：Minor GC 一定会触发 Full GC 吗？

不一定。Minor GC 只在老年代空间不足或对象晋升失败时才会触发 Full GC。如果 Minor GC 后存活对象能正常放入 Survivor 区或老年代，不会触发 Full GC。

### 面试题二：对象什么时候会进入老年代？

**答案要点**：

1. **年龄达到阈值**：对象 age 达到 MaxTenuringThreshold（默认 15，CMS 是 6）
2. **Survivor 区空间不足**：To Survivor 区放不下存活对象
3. **大对象**：超过 PretenureSizeThreshold 的对象直接分配在老年代
4. **动态年龄计算**：Survivor 区中相同年龄对象总大小超过 50%，该年龄以上的对象全部晋升

### 面试题三：如何优化 Minor GC 频繁的问题？

**分析思路**：

1. **确认是否真的频繁**：通过 jstat 观察 YGC 频率
2. **分析原因**：是年轻代太小，还是对象分配率过高
3. **针对性优化**：
   - 年轻代太小 → 增大 `-Xmn` 或调整 `-XX:NewRatio`
   - Survivor 区太小 → 调整 `-XX:SurvivorRatio`
   - 对象晋升太快 → 提高 `-XX:MaxTenuringThreshold`
   - 代码问题 → 优化 JSON 序列化、字符串拼接等
