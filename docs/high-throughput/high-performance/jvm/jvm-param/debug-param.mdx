# GC 日志参数与调试工具

调试 JVM 问题就像在黑暗中摸索——GC 日志和调试参数就是你的手电筒。

没有 GC 日志，你不知道 GC 何时发生、持续多久、回收了多少内存。有了正确的日志配置，你就能看清 JVM 内存管理的每一个细节。

这篇文章，详细讲解 GC 日志参数、OOM 自动导出、诊断参数的正确用法。

## 一、GC 日志参数详解

### 1.1 JDK 8 及之前的日志参数

JDK 8 使用传统的 GC 日志参数：

```bash
# 完整配置示例（JDK 8）
-XX:+PrintGCDetails           # 详细 GC 日志
-XX:+PrintGCDateStamps       # 日期时间戳（可读性更好）
-XX:+PrintGCTimeStamps       # 相对于 JVM 启动的时间戳
-Xloggc:/var/log/gc.log      # 输出到文件
-XX:+UseGCLogFileRotation    # 日志轮转
-XX:NumberOfGCLogFiles=5     # 保留 5 个文件
-XX:GCLogFileSize=100M       # 每个文件最大 100M
```

**参数说明**：

| 参数 | 说明 | 备注 |
|---|---|---|
| `-XX:+PrintGCDetails` | 输出每次 GC 的详细信息 | 包含对象移动、内存池变化等 |
| `-XX:+PrintGCDateStamps` | 输出日期时间（如 2024-01-15T10:30:45.123） | 可读性好 |
| `-XX:+PrintGCTimeStamps` | 输出相对于 JVM 启动的时间（如 123.456） | 便于计算间隔 |
| `-Xloggc:<file>` | GC 日志输出文件路径 | 会覆盖 PrintGCTimeStamps 的行为 |
| `-XX:+UseGCLogFileRotation` | 启用日志轮转 | 防止日志无限增长 |
| `-XX:NumberOfGCLogFiles=5` | 保留 5 个历史日志文件 | 超过后覆盖最早的 |
| `-XX:GCLogFileSize=100M` | 每个日志文件最大 100MB | 需要配合 UseGCLogFileRotation |

**注意**：`PrintGCDetails` 和 `PrintGCDateStamps` 前面的 `+` 表示启用。如果写成 `-XX:-PrintGCDetails` 则表示禁用。

### 1.2 JDK 9+ 统一日志框架

JDK 9 引入了统一的日志框架 `-Xlog`，语法更加规范：

```bash
# JDK 9+ GC 日志配置
-Xlog:gc*:file=/var/log/gc.log:time,uptime,level,tags:filecount=5,filesize=100M
```

**`-Xlog` 语法格式**：

```
-Xlog:<选择器>=<级别>:<输出>:...
```

**选择器（Selectors）**：

| 选择器 | 含义 |
|---|---|
| `gc` | 所有 GC 相关日志 |
| `gc*` | gc 开头的所有日志（包括 gc, gcp, gc cause 等） |
| `gc+class=debug` | GC 和 class 相关的 debug 日志 |

**级别（Level）**：

| 级别 | 说明 |
|---|---|
| off | 关闭日志 |
| trace | 最详细 |
| debug | 调试级别 |
| info | 信息级别（默认） |
| warning | 警告 |
| error | 错误 |

**输出选项**：

| 选项 | 说明 |
|---|---|
| `file=<path>` | 输出到文件 |
| `time` | 包含时间戳 |
| `uptime` | 包含 JVM 启动时间 |
| `level` | 包含日志级别 |
| `tags` | 包含标签 |
| `filecount=5` | 保留文件数 |
| `filesize=100M` | 文件大小限制 |

**对比示例**：

```bash
# JDK 8 写法
-XX:+PrintGCDetails -XX:+PrintGCDateStamps -Xloggc:/var/log/gc.log

# JDK 9+ 等价写法
-Xlog:gc*:file=/var/log/gc.log:time,uptime,level,tags
```

### 1.3 常用日志配置模板

```bash
# 模板 1：开发环境（详细日志）
-XX:+PrintGCDetails
-XX:+PrintGCDateStamps
-Xloggc:/var/log/gc-dev.log

# 模板 2：生产环境（带轮转）
-XX:+PrintGCDetails
-XX:+PrintGCDateStamps
-Xloggc:/var/log/gc-prod.log
-XX:+UseGCLogFileRotation
-XX:NumberOfGCLogFiles=10
-XX:GCLogFileSize=50M

# 模板 3：JDK 9+ 生产环境
-Xlog:gc*:file=/var/log/gc-prod.log:time,uptime,level,tags:filecount=10,filesize=50M
```

## 二、OOM 自动导出堆转储

### 2.1 HeapDumpOnOutOfMemoryError

OOM 是生产环境最常见的问题之一。配置这个参数后，JVM 发生 OOM 时会自动导出堆转储文件，为事后分析提供关键数据。

```bash
# 启用 OOM 自动导出
-XX:+HeapDumpOnOutOfMemoryError

# 指定导出路径
-XX:HeapDumpPath=/var/log/heapdump.hprof

# 完整示例
java -Xmx2g -Xms2g \
     -XX:+HeapDumpOnOutOfMemoryError \
     -XX:HeapDumpPath=/var/log/java_oom.hprof \
     -jar app.jar
```

**重要提示**：

```
OOM 自动导出注意事项
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

⚠️ 堆转储文件可能很大（几GB），需要确保：
1. 导出路径有足够空间
2. 考虑使用符号链接指向专用磁盘
3. 生产环境务必配置！

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 2.2 HeapDumpPath

指定堆转储文件的路径和文件名。如果不指定，默认路径是工作目录。

```bash
# 使用动态文件名（包含时间戳和 PID）
-XX:HeapDumpPath=/var/log/heapdump-%t-%p.hprof

# 时间戳格式说明
# %t - 时间戳 (yyyy-mm-dd_HH-MM-SS)
# %p - 进程 PID
```

### 2.3 手动导出堆转储

除了 OOM 自动导出，还可以通过 jcmd 手动触发：

```bash
# 手动导出堆转储
jcmd <pid> GC.heap_dump /var/log/heapdump_manual.hprof

# 或者使用 jmap
jmap -dump:format=b,file=/var/log/heapdump.hprof <pid>
```

## 三、诊断参数

### 3.1 PrintClassHistogram

打印类实例的直方图，相当于一次轻量级的堆分析。

```bash
# 触发方式
jcmd <pid> GC.class_histogram > /tmp/histogram.txt

# 或者通过信号触发
kill -SIGQUIT <pid>

# Ctrl+\ 也可能触发（取决于 OS 配置）
```

**输出示例**：

```
num     #instances         #bytes  class name
----------------------------------------------
  1:          12345       12345678  [Ljava.lang.String;
  2:           5678        2345678  java.util.HashMap$Node
  3:           4321        1234567  com.example.UserService
  ...
```

### 3.2 PrintFLSStatistics

打印 CMS 和 G1 的并发标记统计信息。

```bash
# 启用 Full GC 后的 CMS 统计
-XX:+PrintFLSStatistics

# 示例输出
FreeListSpaceCMS:       4096K     8192K
MaxThreshold:           8
MinThreshold:           1
Number of Blocks:       1024
Occupancy:              50%
```

### 3.3 +FlightRecorder

JDK 8u40+ 支持的飞行记录功能：

```bash
# 启动时启用 FlightRecorder
-XX:+UnlockCommercialFeatures
-XX:+FlightRecorder
-XX:FlightRecorderOptions=defaultrecording=true,repository=/var/log

# 或者通过 jcmd 启动
jcmd <pid> JFR.start name=test recording=60 settings=profile
```

## 四、远程调试参数

> **警告**：生产环境禁用远程调试！这些参数仅用于本地开发环境。

### 4.1 JDWP（Java Debug Wire Protocol）

```bash
# 启动 JVM 并等待调试器连接
java -agentlib:jdwp=transport=dt_socket,server=y,suspend=n,address=5005 \
     -jar app.jar

# 参数说明
# transport=dt_socket     使用 Socket 传输
# server=y                 作为服务器等待连接
# suspend=n                不暂停启动（设为 y 则等待调试器连接后才启动）
# address=5005             监听端口
```

### 4.2 常用配置组合

```bash
# 开发环境完整配置
java -Xmx1g -Xms1g \
     -XX:+PrintGCDetails \
     -XX:+PrintGCDateStamps \
     -Xloggc:gc.log \
     -XX:+HeapDumpOnOutOfMemoryError \
     -XX:HeapDumpPath=/tmp/heapdump.hprof \
     -agentlib:jdwp=transport=dt_socket,server=y,suspend=n,address=5005 \
     -jar app.jar
```

## 五、NMT（Native Memory Tracking）

NMT 是 JDK 7u40+ 引入的功能，用于追踪 JVM 内部的本地内存使用。

### 5.1 启用 NMT

```bash
# 启动时启用
-XX:NativeMemoryTracking=summary
# 或
-XX:NativeMemoryTracking=detail

# 查看 NMT 输出
jcmd <pid> VM.native_memory summary
```

### 5.2 NMT 输出解读

```
Native Memory Tracking:
Total: reserved=8388608KB, committed=2048000KB
-                 Java Heap (reserved=2097152KB, committed=2048000KB)
-                     Class (reserved=123456KB, committed=123456KB)
-                     Thread (reserved=65536KB, committed=65536KB)
-                     Code (reserved=32768KB, committed=32768KB)
-                     GC (reserved=65536KB, committed=65536KB)
-                     Internal (reserved=8192KB, committed=8192KB)
```

**内存分类说明**：

| 类别 | 说明 |
|---|---|
| Java Heap | 堆内存（-Xms/-Xmx） |
| Class | 类元数据（Metaspace） |
| Thread | 线程栈（Thread Stack） |
| Code | JIT 编译后的代码缓存 |
| GC | GC 收集器使用的内存 |
| Internal | JVM 内部使用（命令行解析等） |

### 5.3 NMT 基准线

```bash
# 标记当前内存使用为基准
jcmd <pid> VM.native_memory baseline

# 稍后对比基准
jcmd <pid> VM.native_memory summary.diff

# 示例输出
Native Memory Tracking:
Total: reserved=8400000KB, committed=2050000KB

Total: reserved=11200KB, committed=1000KB
-                 Thread (reserved=10000KB, committed=800KB)
-                 GC (reserved=1200KB, committed=200KB)
```

## 六、常用调试命令速查

### 6.1 jcmd 常用命令

```bash
# 查看 JVM 信息
jcmd <pid> VM.version
jcmd <pid> VM.flags
jcmd <pid> VM.system_properties

# GC 相关
jcmd <pid> GC.run                    # 手动触发 GC
jcmd <pid> GC.heap_dump /path/file.hprof   # 导出堆转储
jcmd <pid> GC.class_histogram         # 类直方图
jcmd <pid> GC.finalizer_info          # 等待终结的对象

# NMT
jcmd <pid> VM.native_memory summary
jcmd <pid> VM.native_memory baseline

# Flight Recorder
jcmd <pid> JFR.start name=test recording=60
jcmd <pid> JFR.dump name=test filename=/tmp/recording.jfr
jcmd <pid> JFR.stop name=test
```

### 6.2 jinfo 常用命令

```bash
# 查看所有 JVM 参数
jinfo -flags <pid>

# 查看指定参数
jinfo -flag MaxHeapSize <pid>
jinfo -flag PrintGCDetails <pid>

# 设置参数（仅部分参数支持）
jinfo -flag +PrintGCDetails <pid>
jinfo -flag -PrintGCDetails <pid>
jinfo -flag MaxGCPauseMillis=200 <pid>
```

### 6.3 jstat 常用命令

```bash
# GC 统计（每 1 秒输出一次）
jstat -gc <pid> 1000

# 编译统计
jstat -compiler <pid>

# 类加载统计
jstat -class <pid>

# GC 容量统计
jstat -gccapacity <pid>
```

## 七、面试实战

### 面试题一：生产环境如何开启 GC 日志？

**参考答案**：

生产环境开启 GC 日志的关键考虑：

1. **日志输出**：输出到文件，避免占用控制台
2. **日志轮转**：防止日志无限增长
3. **OOM 导出**：配置 HeapDumpOnOutOfMemoryError
4. **磁盘空间**：确保有足够空间

```bash
# 推荐配置（JDK 8）
-XX:+PrintGCDetails
-XX:+PrintGCDateStamps
-Xloggc:/var/log/gc.log
-XX:+UseGCLogFileRotation
-XX:NumberOfGCLogFiles=10
-XX:GCLogFileSize=100M
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/var/log/heapdump.hprof
```

### 面试题二：HeapDumpOnOutOfMemoryError 有什么风险？

**参考答案**：

主要风险：

1. **磁盘空间**：堆转储可能达到几 GB
2. **影响**：导出堆转储会增加停顿时间
3. **生产谨慎**：生产环境建议评估后再开启

建议：配置足够大的磁盘空间，使用符号链接指向专用磁盘。

### 面试题三：NMT 能追踪哪些内存？

**参考答案**：

NMT 可以追踪 JVM 内部的各种内存使用：

1. **Java Heap**：堆内存
2. **Class**：类元数据（Metaspace）
3. **Thread**：线程栈
4. **Code**：JIT 编译后的代码缓存
5. **GC**：GC 收集器使用的内存（如 Card Table）
6. **Internal**：JVM 内部使用

NMT 不追踪：Java 堆外的直接内存（Direct Memory）、JNI 分配的本地内存。
