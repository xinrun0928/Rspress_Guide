# 元空间参数配置

JDK 8 把永久代替换为元空间后，Metaspace OOM 成为了新的「头号杀手」。

很多开发者习惯了「堆内存不够就调大」的经验，但对元空间的理解还停留在「应该够用」的层面。直到某天应用启动时报错：`Metaspace OOM`。

这篇文章，彻底搞清楚元空间的原理和配置方法。

## 一、从永久代到元空间

### 1.1 为什么要替换？

JDK 8 之前使用永久代（PermGen）存储类元数据，存在以下问题：

```
永久代的问题
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

❌ 固定大小：必须在启动时指定，无法动态扩展
❌ 内存溢出：大小受 JVM 进程内存限制，容易 OOM
❌ GC 复杂：需要同时管理堆内存和永久代
❌ 调优困难：需要同时调整 -XX:PermSize 和 -XX:MaxPermSize

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

JDK 8 引入元空间（Metaspace），解决了这些问题：

```
元空间的优势
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

✅ 动态扩展：使用本地内存，可按需分配
✅ 无 OOM 风险（理论上）：受限于机器物理内存
✅ 简化 GC：元空间由本地内存管理，不影响堆 GC
✅ 独立调优：只关注 MetaspaceSize 和 MaxMetaspaceSize

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 1.2 元空间 vs 永久代

| 维度 | 永久代 | 元空间 |
|---|---|---|
| 位置 | JVM 堆内存 | 本地内存（Native Memory） |
| 大小 | 固定，需手动指定 | 动态，可扩展 |
| 溢出条件 | PermGen OOM | Metaspace OOM |
| GC | JVM GC 管理 | 操作系统管理（部分清理） |
| 字符串常量 | 在 PermGen 中 | 在堆中（JDK 7 移入堆） |
| 静态变量 | 在 PermGen 中 | 在堆中 |

### 1.3 Metaspace OOM 的「假象」

元空间使用本地内存，很多人以为「本地内存无限，不会 OOM」。这是一个危险的误解。

```
Metaspace OOM 的真相
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

元空间受以下限制：

1. 进程虚拟地址空间（受 OS 限制）
2. -XX:MaxMetaspaceSize 限制（如果不设置，默认无限制）
3. 操作系统可用内存

实际场景：
- 64 位系统：虚拟地址空间 128GB 或更多
- 元空间满：本地内存耗尽 → SWAP → 性能急剧下降 → OOM Kill

⚠️ 即使不设置 MaxMetaspaceSize，进程也可能因物理内存耗尽而崩溃

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 二、核心参数详解

### 2.1 MetaspaceSize

元空间的初始大小。JVM 启动时，会预分配一块 Metaspace 区域。

```bash
-XX:MetaspaceSize=256m   # 初始元空间大小
```

**重要特性**：

1. **初始分配**：JVM 启动时分配 MetaspaceSize 大小的元空间
2. **Full GC 触发**：当元空间使用达到 MetaspaceSize 时，触发 Full GC
3. **非精确控制**：MetaspaceSize 是触发 Full GC 的阈值，不是精确的初始大小

### 2.2 MaxMetaspaceSize

元空间的最大大小。

```bash
-XX:MaxMetaspaceSize=512m   # 最大元空间大小
```

**重要特性**：

1. **硬限制**：元空间使用不会超过这个值
2. **默认无限制**：如果不设置，理论上可以扩展到进程地址空间上限
3. **生产环境建议**：必须设置，防止元空间无限增长影响其他进程

### 2.3 两者配合使用

```bash
# 推荐配置
-XX:MetaspaceSize=128m    # 初始大小
-XX:MaxMetaspaceSize=512m # 最大大小

# 含义
# 初始分配 128MB
# 达到 128MB 时触发 Full GC 尝试回收
# 如果持续增长，最大允许到 512MB
# 超过 512MB → Metaspace OOM
```

## 三、元空间内存消耗来源

### 3.1 类元数据

这是元空间最主要的消耗来源：

```
类元数据包含内容
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

┌─────────────────────────────────────────────────────────────┐
│ 类元数据                                                     │
├─────────────────────────────────────────────────────────────┤
│ • 类名、方法名、字段名                                        │
│ • 方法字节码（bytecode）                                      │
│ • 常量池（Constant Pool）                                    │
│ • 符号引用（Symbolic Reference）                              │
│ • 注解（Annotations）                                        │
│ • 类加载器信息                                               │
└─────────────────────────────────────────────────────────────┘

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 3.2 JIT 编译产物

JIT 编译器将热点字节码编译为本地机器码，存储在元空间中：

```bash
# JIT 代码缓存大小
-XX:ReservedCodeCacheSize=240m   # 默认 240MB

# JIT 编译阈值
-XX:CompileThreshold=10000       # 方法调用 10000 次后触发编译
```

### 3.3 动态类生成

一些框架会动态生成类，增加元空间消耗：

| 场景 | 框架/工具 | 类生成方式 |
|---|---|---|
| ORM | Hibernate、MyBatis | 动态生成实体类 |
| 代理 | Spring AOP、CGLib | 动态生成代理类 |
| 序列化 | Jackson、Gson | 动态生成序列化类 |
| 模板 | JSP、Thymeleaf | 动态生成模板类 |
| 脚本 | Nashorn、Groovy | 动态生成脚本类 |

### 3.4 元空间消耗分布

```bash
# jcmd 查看元空间使用详情
jcmd <pid> VM.native_memory summary

# 示例输出

  Non-class:      65.43 MB used,   85.23 MB committed,    128.00 MB reserved
  Class:          15.23 MB used,   20.45 MB committed,    64.00 MB reserved
  Code:           25.67 MB used,   30.00 MB committed,    48.00 MB reserved
  Internal:        2.50 MB used,    4.00 MB committed,    8.00 MB reserved
                  ------------------------------------------------
  Total:         108.83 MB used,  139.68 MB committed,  248.00 MB reserved
```

| 类别 | 说明 | 典型大小 |
|---|---|---|
| Non-class | 非类数据（方法、字段等） | 几十 MB |
| Class | 类结构本身 | 十几 MB |
| Code | JIT 编译后的机器码 | 几十 MB |
| Internal | JVM 内部使用 | 几 MB |

## 四、常见配置对比

### 4.1 配置场景

```bash
# 场景 1：小型微服务（Spring Boot）
# 使用轻量级框架，类数量有限
-XX:MetaspaceSize=128m
-XX:MaxMetaspaceSize=256m

# 场景 2：中型应用（Hibernate + MyBatis）
# ORM 框架会动态生成大量类
-XX:MetaspaceSize=256m
-XX:MaxMetaspaceSize=512m

# 场景 3：大型应用（多模块 + 动态类生成）
# 大量动态代理、ORM 实体
-XX:MetaspaceSize=512m
-XX:MaxMetaspaceSize=1024m

# 场景 4：JSP/Java Web 应用
# JSP 编译会产生大量类
-XX:MetaspaceSize=256m
-XX:MaxMetaspaceSize=512m
```

### 4.2 配置建议

| 应用类型 | MetaspaceSize | MaxMetaspaceSize | 说明 |
|---|---|---|---|
| 小型微服务 | 128MB | 256MB | 轻量级框架 |
| 中型应用 | 256MB | 512MB | ORM + 中间件 |
| 大型应用 | 512MB | 1024MB | 多模块 + 动态代理 |
| 超大型应用 | 1024MB | 2048MB | 大量动态类 |

### 4.3 错误的配置

```bash
# 错误 1：不设置 MaxMetaspaceSize
-XX:MetaspaceSize=256m
# → 元空间可能无限增长，导致系统内存耗尽

# 错误 2：MetaspaceSize 设置过大
-XX:MetaspaceSize=2g
# → 启动时分配大量元空间，浪费内存

# 错误 3：MetaspaceSize = MaxMetaspaceSize
-XX:MetaspaceSize=512m
-XX:MaxMetaspaceSize=512m
# → 没有扩容空间，容易触发 Full GC
```

### 5.1 监控命令

```bash
# jstat 查看元空间统计
jstat -gc <pid>

# 示例输出

# MC = Metaspace 容量
# MU = Metaspace 使用
# CCSC = Compressed Class Space 容量
# CCSU = Compressed Class Space 使用
```

### 5.2 jcmd 查看详情

```bash
# 查看类加载统计
jcmd <pid> VM.classloader_stats

# 查看元空间信息
jcmd <pid> VM.native_memory summary
```

### 5.3 Full GC 与元空间

```bash
# 元空间达到 MetaspaceSize 时触发 Full GC

# GC 日志中可以看到
[Full GC (Metadata GC Threshold)]

# 这意味着：
# 元空间使用达到阈值
# JVM 认为需要清理未使用的类元数据
```

```
元空间 Full GC 机制
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

        │
        ▼

        │
        ▼

        │
        ├── 类被卸载 → 元空间释放
        │
        └── 元空间仍满 → 可能导致 Metaspace OOM

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 5.4 类卸载

元空间的一个重要特性是可以卸载不再使用的类：

```bash
# 启用类卸载（默认启用）
-XX:+ClassUnloading

# 如果使用 CMS 或 G1，默认启用
# 可以通过以下参数验证
-XX:+PrintClassHistogram
```

**类卸载条件**：

1. 类的所有实例已被回收
2. 类的 ClassLoader 已被回收
3. 类没有任何引用

## 六、典型问题与解决

### 6.1 Metaspace OOM

**错误信息**：

```
java.lang.OutOfMemoryError: Metaspace
```

**排查步骤**：

```bash
# 1. 查看元空间使用
jcmd <pid> VM.native_memory summary

# 2. 查看类加载统计
jcmd <pid> VM.classloader_stats

# 3. 查看已加载类数量
jstat -class <pid>

# 4. 生成堆转储（如果有堆外内存泄漏）
jcmd <pid> VM.native_memory detail
```

**解决方案**：

```bash
# 方案 1：增加 MaxMetaspaceSize
-XX:MaxMetaspaceSize=1024m

# 方案 2：检查是否类泄漏
# - 大量自定义 ClassLoader 未回收
# - 大量动态代理类未卸载

# 方案 3：切换到 JDK 11+（元空间实现更成熟）
```

### 6.2 元空间持续增长

**问题表现**：
- 元空间使用持续增长，不下降
- 即使触发 Full GC 也无法回收

**可能原因**：
- 类加载器泄漏（最常见）
- 静态变量持有类引用
- 大量动态代理类

**排查命令**：

```bash
# 查看类直方图
jcmd <pid> GC.class_histogram

# 示例输出
num     #instances         #bytes  class name
--------------------------------------------------
  1:          1234       12345678  [Ljava.lang.Class;
  2:           567        2345678  com.example.MyClassLoader
  ...
```

## 七、元空间 vs 永久代表格对照

| 功能 | 永久代参数 | 元空间参数 | 说明 |
|---|---|---|---|
| 初始大小 | `-XX:PermSize` | `-XX:MetaspaceSize` | 元空间使用本地内存 |
| 最大大小 | `-XX:MaxPermSize` | `-XX:MaxMetaspaceSize` | 建议设置 |
| 字符串常量 | PermGen 中 | 堆中 | JDK 7 已移入堆 |
| 静态变量 | PermGen 中 | 堆中 | JDK 7 已移入堆 |
| 类元数据 | PermGen 中 | Metaspace | 元空间更灵活 |
| GC | JVM GC | 操作系统 + 部分清理 | 元空间由 OS 管理 |

## 八、面试实战

### 面试题一：元空间和堆内存有什么区别？

**参考答案**：

| 维度 | 堆内存 | 元空间 |
|---|---|---|
| 存储内容 | Java 对象实例 | 类元数据、JIT 代码 |
| 位置 | JVM 堆 | 本地内存（Native） |
| 大小控制 | -Xms/-Xmx | MetaspaceSize/MaxMetaspaceSize |
| 溢出类型 | Heap Space OOM | Metaspace OOM |
| GC | Full GC 回收 | 类卸载（部分清理） |
| 默认值 | 系统内存的 1/4 | 动态扩展 |

---

### 面试题二：什么情况下会导致 Metaspace OOM？

**参考答案**：

1. **类加载过多**：动态代理、ORM 框架生成大量类
2. **类泄漏**：自定义 ClassLoader 未回收
3. **MaxMetaspaceSize 设置过小**：没有足够的空间
4. **大量 JSP 编译**：JSP 每次访问都会编译生成类

---

### 面试题三：元空间会触发 Full GC 吗？

**参考答案**：

会。当元空间使用达到 `MetaspaceSize` 时，会触发一次 Full GC 来尝试卸载不再使用的类。

但元空间的 Full GC 不会清理堆内存，它只清理类元数据。如果 Full GC 后元空间仍然不足，会抛出 Metaspace OOM。
