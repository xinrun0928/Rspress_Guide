# GC 收集器选择与参数组合：选错 GC 的代价，比你想象的大

凌晨 3 点，监控系统报警：接口 P99 延迟从 50ms 飙升到 2000ms。

你从床上爬起来，打开日志一看：Full GC 停了 1.8 秒。

业务高峰期，一次 Full GC 拖垮了整个系统。用户投诉、退款、老板的电话——这个夜晚彻底毁了。

事后复盘，原因让你吐血：**团队为了「跟上技术潮流」，把 GC 收集器从 Parallel GC 换成了 CMS，但没做任何参数调优。** CMS 的默认参数根本撑不住高并发下的内存压力，频繁的并发模式失败触发了「退化 Full GC」——Single Thread 的 Full GC，停顿时间直接爆表。

**CMS 还是 G1？Parallel 还是 ZGC？选择错误的 GC 收集器，后果比想象中严重得多。**

这篇文章，就是要把六种 GC 收集器讲清楚，帮你做出正确的选择。

## 一、六种 GC 收集器全景图

### 1.1 收集器一览表

Java 发展至今，官方支持（OpenJDK）的 GC 收集器有六种：

| 收集器 | 线程模型 | 核心目标 | JDK 版本 | 适用场景 |
|---|---|---|---|---|
| Serial GC | 单线程 | 低配环境 | 全版本 | 单核、低配机器、客户端程序 |
| Parallel GC | 多线程 | 高吞吐 | 全版本 | 批处理、离线计算、后台任务 |
| CMS GC | 并发标记 | 低停顿 | JDK 7-14（已移除） | 低停顿需求（已被 G1 取代） |
| G1 GC | 分区式并发 | 可预测停顿 | JDK 7+（JDK 9+ 默认） | 通用场景、低延迟服务 |
| ZGC | 着色指针 | 极低停顿 | JDK 11+ 稳定 | 大内存、低延迟场景 |
| Shenandoah | 无分代 | 极低停顿 | OpenJDK 支持 | 大内存、低延迟场景 |

### 1.2 收集器的工作原理分类

从线程模型角度，GC 收集器可以分为三类：

**Stop-The-World（STW）类**：GC 时必须暂停所有应用线程。Serial GC、Parallel GC 属于此类。

**并发类**：GC 线程与应用线程并发执行，不需要完全暂停。CMS、G1、ZGC、Shenandoah 属于此类。

**混合类**：部分阶段并发，部分阶段 STW。G1、ZGC 都是这种模式。

```
┌─────────────────────────────────────────────────────────────────┐
│                        GC 收集器分类                             │
├─────────────────────────────────────────────────────────────────┤
│  STW 类        │  Serial GC、Parallel GC                         │
│  (全部暂停)    │  简单可靠，但停顿时间长                           │
├─────────────────────────────────────────────────────────────────┤
│  并发类        │  CMS、G1、ZGC、Shenandoah                        │
│  (并发执行)    │  停顿短，但实现复杂，有额外开销                  │
├─────────────────────────────────────────────────────────────────┤
│  混合类        │  G1、ZGC                                         │
│  (部分并发)    │  结合两者优点，平衡停顿时间和吞吐量               │
└─────────────────────────────────────────────────────────────────┘
```

## 二、Serial GC：单线程的「老黄牛」

### 2.1 核心原理

Serial GC 是最古老、最简单的 GC 收集器。它的名字来源于「串行」——所有 GC 操作都在单个线程上执行。

```bash
# 开启 Serial GC
-XX:+UseSerialGC
```

**工作原理**：

1. **Minor GC**：使用单线程在年轻代执行「复制算法」。Eden 区的存活对象复制到 S0 或 S1，S0 和 S1 角色互换。
2. **Full GC**：使用单线程在老年代执行「标记-整理算法」。先标记存活对象，再整理碎片。

**为什么叫「复制算法」？** 因为 Minor GC 时，存活对象被「复制」到 Survivor 区，而不是「移动」——Eden 区和原来的 Survivor 区会被整体清空。

### 2.2 特点与适用场景

**优点**：

- 实现简单，稳定性高
- 没有线程切换开销
- 内存占用最小（不需要额外的线程空间）

**缺点**：

- 单线程执行，GC 停顿时间长
- 多核 CPU 利用率低

**适用场景**：

- 单核机器或极低配机器
- 客户端程序（如 Swing 应用）
- 内存极小的嵌入式设备
- 测试环境快速验证

:::warning Serial GC 不是「很差」的收集器

很多人觉得 Serial GC 是「低端货」，不屑一顾。但 Serial GC 在某些场景下表现不差：

1. **内存很小（< 100MB）时**，Serial GC 的单线程复制反而比多线程更快（没有同步开销）
2. **单核 CPU**，Serial GC 是唯一合理的选择
3. **JVM 线程数固定为 1** 时，Parallel GC 也会退化为单线程

不要歧视 Serial GC，根据场景选择才是正确的思路。

:::

## 三、Parallel GC：吞吐量的「战斗机」

### 3.1 核心原理

Parallel GC（又称 Throughput GC）是 JDK 8 及之前的**默认收集器**。它使用多线程并行执行 GC，充分利用多核 CPU 的优势。

```bash
# 开启 Parallel GC（JDK 8 默认）
-XX:+UseParallelGC

# 完整组合：Parallel + Parallel Old
-XX:+UseParallelGC -XX:+UseParallelOldGC
```

**工作原理**：

1. **Minor GC**：多线程并行复制，年轻代 GC 速度大幅提升
2. **Full GC**：多线程并行标记-整理，老年代 GC 速度同样提升

**核心目标**：最大化**吞吐量（Throughput）**，即「GC 时间占比」越低越好。

```
吞吐量 = 运行代码时间 / (运行代码时间 + GC 时间)
```

如果吞吐量目标是 99%，意味着 100 分钟内，只有 1 分钟用于 GC。

### 3.2 关键参数

Parallel GC 最重要的参数是 `-XX:ParallelGCThreads`，它控制 GC 线程数。

```bash
# 默认值 = CPU 核心数
# 16 核机器上，GC 线程数 = 16

# 手动设置（不推荐，除非有特殊理由）
-XX:ParallelGCThreads=8
```

:::tip ParallelGCThreads 的默认值公式

JDK 源码中的计算逻辑（简化版）：

```java
int threads = Math.max(1, (CPU核心数 - 8) * 5 / 8 + 8);
```

| CPU 核心数 | ParallelGCThreads |
|---|---|
| 4 | 4 |
| 8 | 8 |
| 16 | 13 |
| 32 | 21 |
| 64 | 41 |

CPU 核心数越多，额外增加的线程数越少——因为线程数太多反而会因为上下文切换和同步开销降低效率。

:::

### 3.3 特点与适用场景

**优点**：

- 多线程并行，吞吐量高
- 充分利用多核 CPU
- 实现成熟，稳定性高

**缺点**：

- GC 时必须 Stop-The-World，停顿时间长
- 不适合对延迟敏感的应用

**适用场景**：

- **批处理任务**：ETL、数据分析、报表生成
- **离线计算**：机器学习训练、数据仓库
- **后台定时任务**：不需要实时响应
- **高吞吐优先**：不在乎单次停顿时间，只要整体吞吐高

## 四、CMS GC：第一个「并发」探索者

### 4.1 核心原理

CMS（Concurrent Mark Sweep）是第一个实现「并发标记」的收集器，目标是**降低停顿时间**。它试图让 GC 线程和应用线程并发执行，从而减少 STW 时间。

```bash
# 开启 CMS GC
-XX:+UseConcMarkSweepGC

# CMS 通常与 ParNew（年轻代并行收集器）配合
# 使用 CMS 时，年轻代自动使用 ParNew
```

**工作原理（四个阶段）**：

1. **初始标记（Initial Mark）**：STW，只标记 GC Roots 直接引用的对象，停顿时间很短
2. **并发标记（Concurrent Mark）**：与应用线程并发执行，遍历对象图，标记所有存活对象
3. **重新标记（Remark）**：STW，修正并发标记期间因应用线程运行导致的标记变化
4. **并发清除（Concurrent Sweep）**：与应用线程并发执行，清除死亡对象

```
CMS 工作流程：

[初始标记 STW] ──────────────────────────────────────────> 时间
     │
     ↓
[并发标记] (与业务线程并发) ──────────────────────────────> 时间
     │
     ↓
[重新标记 STW] ──────────────────────────────────────────> 时间
     │
     ↓
[并发清除] (与业务线程并发) ──────────────────────────────> 时间
```

### 4.2 CMS 的问题：并发模式失败

CMS 最大的问题是**并发模式失败（Concurrent Mode Failure）**。

当老年代空间不足以容纳年轻代晋升的对象时，CMS 无法继续并发清除，只能触发一次「退化 Full GC」——用 Single Thread 在老年代执行标记-整理。这个停顿时间可能是正常 Full GC 的 5-10 倍。

触发条件：

1. 老年代使用率达到 `-XX:CMSInitiatingOccupancyFraction`（默认 92%）
2. 并发清除速度跟不上对象晋升速度

### 4.3 关键参数

```bash
# 老年代使用率达到 70% 时开始并发标记（默认 92%）
-XX:CMSInitiatingOccupancyFraction=70

# 启用主动 CMS 回收（JDK 8 及以后推荐）
-XX:+UseCMSInitiatingOccupancyOnly

# 并发标记线程数（默认 = CPU 核心数 / 4）
-XX:ConcGCThreads=4

# Full GC 后是否整理碎片（默认 false，会产生内存碎片）
-XX:+UseCMSCompactAtFullCollection
```

### 4.4 CMS 的命运：被 G1 取代

**CMS 在 JDK 9 被标记为废弃，JDK 14 被彻底移除。**

原因：

1. **内存碎片问题**：CMS 使用「标记-清除」算法，会产生内存碎片
2. **浮动垃圾**：并发清除阶段产生的垃圾只能下次 GC 处理
3. **参数复杂**：CMS 有太多难以理解的参数，调优门槛高
4. **设计局限**：CMS 是为「老年代收集器」设计的，年轻代需要另外的收集器配合

**现在，任何考虑使用 CMS 的场景，都应该使用 G1 替代。**

:::danger 不要在新项目中使用 CMS

CMS 已被移除，不再获得任何更新。如果你还在使用 CMS：

1. **JDK 8 用户**：尽快升级 JDK 11+ 并迁移到 G1
2. **JDK 11+ 用户**：G1 是默认收集器，直接使用即可

遗留系统迁移方案：`-XX:+UseG1GC` 即可切换到 G1，大多数情况下不需要额外配置。

:::

## 五、G1 GC：现在的「默认选择」

### 5.1 核心原理

G1（Garbage First）是 JDK 7 引入、JDK 9+ 默认的收集器。它采用了**全新的设计理念**：不再将堆严格划分为年轻代和老年代，而是将整个堆划分为多个大小相等的 **Region**。

```bash
# 开启 G1 GC（JDK 9+ 默认）
-XX:+UseG1GC
```

**Region 的概念**：

- 堆被划分为约 2048 个 Region（数量与堆大小相关）
- 每个 Region 大小 = 堆大小 / 2048，最小 1MB，最大 32MB
- Region 可以动态扮演 Eden、Survivor 或 Old 的角色

```
G1 堆结构：

┌────┬────┬────┬────┬────┬────┬────┬────┬────┬────┬────┬────┬────┬────┬────┬────┐
│ E  │ S  │ O  │ O  │ E  │ E  │ S  │ O  │ O  │ O  │ O  │ E  │ E  │ O  │ O  │ O  │
└────┴────┴────┴────┴────┴────┴────┴────┴────┴────┴────┴────┴────┴────┴────┴────┘
E = Eden Region, S = Survivor Region, O = Old Region

Region 大小相同，但角色动态变化
```

**G1 的核心优势**：

1. **可预测的停顿时间**：通过 `-XX:MaxGCPauseMillis` 设置目标停顿时间
2. **空间整合**：使用「标记-整理」算法，不会产生内存碎片
3. **分代+分区结合**：既有分代概念（Eden/Survivor/Old），又用分区替代固定分代
4. **优先回收价值最大的 Region**：G1 根据「回收价值」选择回收哪些 Region

### 5.2 G1 的工作原理

G1 的 GC 分为两类：**Young GC** 和 **Mixed GC**。

**Young GC（年轻代收集）**：

1. 收集所有年轻代 Region
2. 使用复制算法，将存活对象复制到 Survivor Region
3. 根据 `-XX:MaxGCPauseMillis` 控制收集的 Region 数量

**Mixed GC（混合收集）**：

1. 当老年代占用率达到 `-XX:InitiatingHeapOccupancyPercent` 时触发
2. 收集所有年轻代 Region + 部分老年代 Region
3. 优先收集「回收价值最高」的老年代 Region

### 5.3 G1 专属参数

G1 有大量专属参数，这里先列出核心参数，详细讲解见《G1 GC 专属参数详解》。

```bash
# 目标停顿时间（G1 最核心的参数）
-XX:MaxGCPauseMillis=200

# Region 大小（自动计算，手动设置必须是 2 的幂）
-XX:G1HeapRegionSize=4M

# 触发并发周期的老年代占用比
-XX:InitiatingHeapOccupancyPercent=45

# 年轻代比例范围
-XX:G1NewSizePercent=5
-XX:G1MaxNewSizePercent=60

# 保留内存比例（应对晋升失败）
-XX:G1ReservePercent=15
```

### 5.4 特点与适用场景

**优点**：

- 可预测的停顿时间，适合延迟敏感服务
- 无内存碎片，长期运行稳定
- 适合大堆（> 6GB）
- JDK 9+ 默认，文档和社区资源丰富

**缺点**：

- 吞吐量略低于 Parallel GC
- Region 管理有额外开销
- 参数较多，需要一定调优经验

**适用场景**：

- **延迟敏感服务**：API 服务、游戏后端、实时计算
- **大堆应用**：> 6GB 的堆，G1 比其他收集器更稳定
- **通用场景**：不知道选什么，选 G1 不会错

## 六、ZGC：追求极致的「亚毫秒级」停顿

### 6.1 核心原理

ZGC（Z Garbage Collector）在 JDK 11 作为实验特性引入，JDK 15 成为生产就绪。它是专门为**大内存、低延迟**场景设计的收集器。

```bash
# 开启 ZGC（JDK 11+）
-XX:+UseZGC

# JDK 15+ 可不加 -XX:+UseZGC，ZGC 会自动启用
```

**ZGC 的核心特性**：

1. **停顿时间 < 1ms**：无论堆多大，停顿时间都保持在亚毫秒级
2. **停顿时间可预测**：通过 `-XX:MaxGCPauseMillis` 设置目标
3. **支持大堆**：官方支持 TB 级堆内存
4. **并发执行**：GC 线程与应用线程几乎全程并发

**ZGC 的技术基础：着色指针（Colored Pointers）**

ZGC 在对象的 64 位指针中，使用低 4 位存储标记信息（Marked0、Marked1、Remapped、Finalizable）。通过这些「着色指针」，ZGC 可以在不访问对象的情况下判断对象状态，实现并发移动。

```
64 位对象指针：

┌────────────────────────────────┬────┐
│          42 位对象地址          │ 4 位颜色 │ 18 位元数据
└────────────────────────────────┴────┘

颜色位含义：
- Marked0/Marked1：对象已标记
- Remapped：对象位置已更新
- Finalizable：对象仅通过终结器可达
```

### 6.2 ZGC 的工作原理

ZGC 的工作流程分为三个阶段：

1. **标记（Mark）**：标记所有存活对象（并发）
2. **引用处理（Reference Processing）**：处理软引用、弱引用、虚引用（并发）
3. **重新定位（Relocation）**：移动存活对象，回收旧空间（并发）

关键点：**三个阶段都是并发的，只有短暂的重标记阶段需要 STW。**

### 6.3 关键参数

```bash
# 目标停顿时间
-XX:MaxGCPauseMillis=1

# 并发 GC 线程数（默认 = CPU 核心数）
-XX:ConcGCThreads=8

# 堆内存范围（JDK 12+）
-XX:ZHeapSize=8G

# 显式 GC 触发阈值（JDK 14+）
-XX:ZCollectionInterval=60
```

### 6.4 特点与适用场景

**优点**：

- 停顿时间极短（< 1ms）
- 支持超大内存（TB 级）
- 吞吐表现优秀
- 配置简单

**缺点**：

- JDK 11+ 才稳定（JDK 8 不支持）
- 不支持类卸载（JDK 15+ 支持）
- 内存占用略高（着色指针需要额外空间）

**适用场景**：

- **超低延迟需求**：金融交易、实时游戏、延迟敏感 API
- **大内存应用**：> 32GB 堆内存
- **JDK 11+ 环境**

:::tip ZGC vs G1：怎么选？

| 维度 | G1 | ZGC |
|---|---|---|
| 停顿时间目标 | 200-500ms | < 1ms |
| 最大堆支持 | ~100GB | TB 级 |
| JDK 版本要求 | JDK 7+ | JDK 11+ |
| 配置复杂度 | 中等 | 简单 |
| 吞吐量 | 略低 | 高 |
| 类卸载 | 支持 | JDK 15+ 支持 |

**选择建议**：

- **延迟 < 100ms**：选 ZGC
- **延迟 100-500ms**：选 G1
- **JDK 8 环境**：选 G1（可用 CMS 迁移到 G1）
- **堆内存 > 32GB**：选 ZGC

:::

## 七、Shenandoah：开源社区的「低延迟」方案

### 7.1 核心原理

Shenandoah 是 OpenJDK 社区开发的低延迟 GC收集器，与 ZGC 类似，追求极低停顿时间。**它不是 Oracle JDK 的官方收集器，而是 Red Hat 主导的 OpenJDK 项目。**

```bash
# 开启 Shenandoah（OpenJDK 支持，Oracle JDK 不支持）
-XX:+UseShenandoahGC
```

**Shenandoah vs ZGC 的区别**：

1. **分代设计**：Shenandoah 是无分代的（类似 ZGC），不区分年轻代和老年代
2. **转发指针**：Shenandoah 使用「Brooks Pointer」（转发指针）而非着色指针
3. **跨 JDK 版本**：Shenandoah 在多个 JDK 版本中可用（包括某些 JDK 8 分支）

### 7.2 特点与适用场景

**优点**：

- 停顿时间短（< 10ms）
- 支持多种 JDK 版本
- 与应用线程并发对象移动

**缺点**：

- 不是 Oracle JDK 官方收集器（Oracle JDK 可能不支持）
- 吞吐量略低于 G1
- 社区资源相对较少

**适用场景**：

- 使用 OpenJDK（如 Amazon Corretto、AdoptOpenJDK）
- 需要低延迟但无法升级 JDK 版本
- 对停顿时间有严格要求

## 八、收集器选择决策树

### 8.1 三步决策法

```
第一步：确定延迟需求
│
├─ 延迟 < 100ms？
│   └─ 是 → ZGC（需要 JDK 11+）
│
├─ 延迟 100-500ms？
│   └─ 是 → G1 GC（JDK 9+ 默认）
│
└─ 延迟不敏感？
    │
    └─ 是 → Parallel GC（JDK 8 及之前默认）
```

```
第二步：确定堆内存大小
│
├─ 堆 > 32GB？
│   └─ 是 → ZGC（强烈推荐）或 G1
│
├─ 堆 8-32GB？
│   └─ 是 → G1
│
└─ 堆 < 8GB？
    └─ 是 → G1 或 Serial GC（根据 CPU 核心数）
```

```
第三步：确定 JDK 版本
│
├─ JDK 8？
│   └─ 选 G1（不支持 ZGC）或 Parallel GC
│
├─ JDK 11-14？
│   └─ 选 ZGC（实验特性）或 G1
│
└─ JDK 15+？
    └─ 选 ZGC（生产就绪）或 G1
```

### 8.2 推荐配置速查

| 场景 | 推荐收集器 | JVM 参数 |
|---|---|---|
| 批处理任务 | Parallel GC | `-XX:+UseParallelGC` |
| Web 服务（JDK 8） | G1 GC | `-XX:+UseG1GC` |
| Web 服务（JDK 11+） | G1 GC | 默认（JDK 9+） |
| 低延迟 API | ZGC | `-XX:+UseZGC -XX:MaxGCPauseMillis=1` |
| 大内存服务（> 32GB） | ZGC | `-XX:+UseZGC` |
| 嵌入式/低配机器 | Serial GC | `-XX:+UseSerialGC` |
| OpenJDK 环境低延迟 | Shenandoah | `-XX:+UseShenandoahGC` |

## 九、收集器组合说明

### 9.1 组合的正确理解

很多资料说「Parallel GC + Parallel Old GC」「ParNew + CMS」这样的组合。实际上，这些是**分代收集器的协作组合**。

- 年轻代用一个收集器
- 老年代用另一个收集器
- 两者配合工作

**现代收集器（G1、ZGC）不需要组合，它们是独立工作的。**

### 9.2 组合兼容性表

| 年轻代收集器 | 老年代收集器 | 说明 |
|---|---|---|
| Serial | Serial Old | 客户端组合，低配环境 |
| ParNew | CMS | CMS 的官方组合（JDK 14 已移除） |
| ParNew | Serial Old | 备用组合，不推荐 |
| Parallel | Parallel Old | 吞吐量优先组合，JDK 8 默认 |
| G1 | G1 | 独立工作，JDK 9+ 默认 |
| ZGC | ZGC | 独立工作，TB 级内存 |

:::warning 不要混用不兼容的收集器

如果年轻代用 ParNew，老年代不能选 Parallel Old（不兼容）。ParNew 只能配合 CMS 或 Serial Old 使用。

设置收集器时，通常只需要设置年轻代或老年代其中一个，另一个会自动适配：

```bash
# 设置年轻代为 ParNew，老年代自动选择 CMS
-XX:+UseParNewGC -XX:+UseConcMarkSweepGC

# 设置老年代为 G1，年轻代自动适配 G1
-XX:+UseG1GC

# 设置老年代为 ZGC，年轻代自动适配 ZGC
-XX:+UseZGC
```

:::

## 十、面试实战

### 10.1 场景一：收集器选择

**面试官问**：「如果你要给一个高并发电商系统选 GC 收集器，你怎么选？」

**分析思路**：

1. **高并发电商系统的特点**：
   - 延迟敏感（用户等待体验）
   - 吞吐量要求高（促销期间 QPS 暴涨）
   - 堆内存中等（4-8GB）
   - 通常使用 JDK 8 或 JDK 11

2. **选择依据**：

> 电商系统对延迟有要求，但不像金融系统那样严格到毫秒级。我会选择 **G1 GC**。
>
> 理由：
> - G1 是 JDK 9+ 默认，经过大量生产验证
> - 可预测的停顿时间（设置 200-500ms）
> - 适合中等规模堆（4-8GB）
> - 无内存碎片问题，长期运行稳定
>
> 如果促销期间延迟仍然较高，可以：
> - 调小 `-XX:MaxGCPauseMillis`（比如从 200ms 改为 100ms）
> - 增大年轻代比例，减少对象晋升老年代
> - 如果还是不行，考虑升级 JDK 11+ 并切换到 ZGC

### 10.2 场景二：CMS 到 G1 的迁移

**面试官问**：「公司还在用 CMS，现在要迁移到 G1，怎么做？」

**回答**：

> CMS 到 G1 的迁移相对平滑：
>
> **第一步：升级 JDK**
>
> CMS 在 JDK 14 被移除，如果还在用 JDK 8，需要先升级到 JDK 11+。
>
> **第二步：切换收集器**
>
> ```bash
> # 移除 CMS 参数
> -XX:+UseConcMarkSweepGC  # 删除
> -XX:CMSInitiatingOccupancyFraction=70  # 删除
>
> # 添加 G1 参数
> -XX:+UseG1GC
> -XX:MaxGCPauseMillis=200
> ```
>
> **第三步：调整参数**
>
> CMS 的很多参数（如 `CMSInitiatingOccupancyFraction`）在 G1 中不再适用。G1 有自己的参数体系。
>
> **第四步：压测验证**
>
> 迁移后必须做全链路压测，对比：
> - GC 频率和停顿时间
> - 吞吐量变化
> - 内存使用情况
>
> **注意事项**：
> - CMS 是「老年代收集器」，G1 是「全堆收集器」，行为不同
> - G1 的 GC 日志格式与 CMS 不同，需要调整监控告警
> - 如果 CMS 有特殊调优，迁移后需要重新评估

### 10.3 场景三：GC 停顿分析

**面试官问**：「线上 Full GC 停顿超过 2 秒，怎么排查？」

**回答**：

> Full GC 停顿超过 2 秒，首先要看用的是什么收集器：
>
> **如果是 Serial GC 或 Parallel GC**：
>
> - 说明老年代满了，对象晋升速度 > GC 回收速度
> - 可能原因：内存泄漏、对象年龄过大、Survivor 区太小
> - 排查方法：
>   - `jstat -gcutil` 观察老年代使用率趋势
>   - `jmap -dump` 导出堆转储，MAT 分析对象分布
>   - 观察 `-XX:MaxTenuringThreshold` 是否过大
>
> **如果是 CMS**：
>
> - 很可能发生了「并发模式失败」
> - 老年代使用率达到 92% 时还没回收完，新对象就来了
> - 解决方案：调低 `-XX:CMSInitiatingOccupancyFraction`，或切换到 G1
>
> **如果是 G1**：
>
> - G1 的 Full GC 很少发生，一旦发生说明问题严重
> - 可能原因：元空间不足、对象太大无法疏散、堆太小
> - 解决方案：增大堆、增大 `-XX:G1ReservePercent`
>
> **通用排查步骤**：
>
> ```bash
> # 1. 查看 GC 日志，确认 Full GC 频率和耗时
> jstat -gcutil <pid> 1000
>
> # 2. 查看对象年龄分布
> jstat -gcage <pid>
>
> # 3. 导出堆转储分析
> jmap -dump:format=b,file=heap.hprof <pid>
>
> # 4. 使用 MAT 或其他工具分析
> ```

## 总结

GC 收集器选择，核心是回答三个问题：

1. **你的业务对延迟敏感吗？** 是选 G1/ZGC，否选 Parallel GC
2. **你的堆内存有多大？** 小堆 G1，大堆 ZGC
3. **你的 JDK 版本是什么？** JDK 8 只能用 G1/JVM 8 自带，JDK 11+ 可以选 ZGC

记住：**没有最好的收集器，只有最适合业务场景的收集器。** 选对了收集器只是第一步，正确的参数调优同样重要。

切记：**不要在新项目中使用 CMS。** 已经被淘汰了。
