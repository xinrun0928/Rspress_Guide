# JVM 堆内存参数配置：让你的 GC 不再「抽风」

线上有个服务，内存 8GB，堆设了 4GB，跑得好好的，突然有一天开始频繁 Full GC。每隔几分钟就卡顿一下，用户体验断崖式下跌。

你开始排查：内存没泄漏，代码没改动，流量也没涨。问题出在哪？

最后发现，是有人改了一个参数：`-XX:NewRatio=2`——老年代是年轻代的 2 倍，年轻代只有 1.3GB。促销期间，大量短期对象涌进来，Survivor 区装不下，直接晋升老年代，导致老年代迅速填满，触发频繁 Full GC。

**`-Xms` 和 `-Xmx` 到底该怎么设置？年轻代多大合适？Survivor 区配多少才够用？** 这个问题，搞不清楚的人太多了。

这篇文章，就是要把堆内存参数讲透。

## 一、堆内存的核心结构

### 1.1 堆内存的两大分区

JVM 堆内存分为两大区域：**年轻代（Young Generation）** 和 **老年代（Old Generation）**。

年轻代是对象的「第一战场」。绝大多数对象在这里创建，在这里死亡。年轻代又分为三部分：

- **Eden 区**：对象创建的「出生地」
- **Survivor From 区**（简称 S0）：对象的「中转站」
- **Survivor To 区**（简称 S1）：对象的「另一个中转站」

老年代是对象的「养老院」。经过多次 GC 仍然存活的对象，最终会被移动到这里。

```
┌──────────────────────────────────────────────────────────────┐
│                          Heap Memory                         │
├─────────────────────────────┬────────────────────────────────┤
│        Young Generation      │         Old Generation        │
├─────────────────────────────┼────────────────────────────────┤
│ Eden │   S0    │   S1    │         Tenured                   │
│      │ (From)  │  (To)   │                                  │
└─────────────────────────────┴────────────────────────────────┘
```

为什么要有 Survivor 区？**答案：避免对象过早进入老年代。** 如果对象在 Eden 区熬过几次 Minor GC 后仍然存活，它会在 Survivor 区之间来回复制，直到「成年」再晋升老年代。这个设计大大延缓了老年代的填充速度，减少了 Full GC 的频率。

### 1.2 各区域的内存占比

年轻代和老年代的默认比例由 `-XX:NewRatio` 控制，**默认值为 2**，意味着老年代是年轻代的 2 倍，即年轻代占堆的 1/3，老年代占 2/3。

年轻代内部的 Survivor 区比例由 `-XX:SurvivorRatio` 控制，**默认值为 8**，意味着 `Eden : Survivor = 8 : 1 : 1`，即 Eden 占年轻代的 80%，每个 Survivor 占 10%。

以 4GB 堆为例：

- 年轻代 = 4GB / 3 ≈ 1.33GB
- Eden = 1.33GB × 80% ≈ 1.06GB
- S0 = S1 = 1.33GB × 10% ≈ 133MB
- 老年代 = 4GB - 1.33GB ≈ 2.67GB

这个比例是经验值，适用于大多数场景。但对于某些特殊业务，可能需要调整。

## 二、堆内存核心参数详解

### 2.1 -Xms 和 -Xmx：堆大小的「起止线」

这两个参数是堆内存配置的基础：

- **`-Xms`**（Initial Heap Size）：JVM 启动时分配的初始堆大小
- **`-Xmx`**（Maximum Heap Size）：堆内存的最大上限

```bash
# 设置初始堆为 4GB，最大堆为 4GB（固定大小）
-Xms4g -Xmx4g

# 设置初始堆为 4GB，最大堆为 8GB（弹性扩展）
-Xms4g -Xmx8g
```

生产环境的一个**最佳实践**是：将 `-Xms` 和 `-Xmx` 设置为相同的值。这不是为了性能，而是为了**避免运行时堆伸缩带来的抖动**。

当堆从 4GB 扩展到 8GB 时，JVM 需要向操作系统申请新的内存；当堆从 8GB 收缩回 4GB 时，又需要释放内存。这个过程会触发额外的 GC 操作，且内存分配和回收本身就有开销。固定堆大小虽然牺牲了一点弹性，但换来了更稳定的性能表现。

:::warning 弹性堆的坑

`-Xms4g -Xmx8g` 这种配置，生产环境**强烈不推荐**。原因有三：

1. 运行时堆伸缩本身有性能开销
2. 堆最大不代表能用满，但 JVM 可能会预分配部分内存
3. 如果内存紧张，操作系统可能无法满足扩展请求

如果不知道该设多大，宁可保守一点（设小一点），也不要留太大弹性。

:::

### 2.2 -Xmn：年轻代的「定海神针」

`-Xmn`（New Size = Max New Size）同时设置了年轻代的初始大小和最大大小，**一旦设置，年轻代比例就固定了**，`-XX:NewRatio` 会失效。

```bash
# 固定年轻代为 2GB（老年代自动为 2GB，堆总共 4GB）
-Xms4g -Xmx4g -Xmn2g
```

什么时候需要手动指定年轻代大小？

**第一种情况：需要精确控制 GC 行为。** 比如你知道业务对象的生命周期分布，需要年轻代有足够的空间来容纳短期对象。

**第二种情况：配合特定 GC 收集器。** CMS 和 G1 对年轻代的处理方式不同，有时候需要通过调整年轻代大小来优化 GC 停顿时间。

**第三种情况：大对象为主的业务。** 如果业务中大对象居多，年轻代太小会导致这些对象直接晋升老年代，加速老年代的填充。

### 2.3 -XX:NewRatio：年轻代与老年代的「比例尺」

`-XX:NewRatio` 设置老年代与年轻代的比例，**默认值为 2**。

```bash
# NewRatio=2 表示老年代是年轻代的 2 倍
# 即年轻代占堆的 1/3，老年代占 2/3
-XX:NewRatio=2

# NewRatio=1 表示老年代和年轻代各占一半
# 年轻代更大，Minor GC 更频繁但停顿更短
-XX:NewRatio=1

# NewRatio=3 表示老年代是年轻代的 3 倍
# 年轻代较小，老年代更大，Full GC 较少但Minor GC晋升更快
-XX:NewRatio=3
```

NewRatio 和 `-Xmn` 是**互斥**的。一旦设置了 `-Xmn`，NewRatio 就不再生效。

:::tip NewRatio 的选择依据

| 业务场景 | 推荐 NewRatio | 理由 |
|---|---|---|
| Web 服务（对象生命周期短） | 1 或 2 | 年轻代大一些，减少对象晋升老年代 |
| 批处理（对象生命周期长） | 2 或更大 | 年轻代可以小一些，老年代需要更多空间 |
| 大对象为主 | 1 | 年轻代需要足够大容纳大对象 |
| 延迟敏感 | 1 或更小 | 年轻代越大，Minor GC 越频繁但停顿越短 |

:::

### 2.4 -XX:SurvivorRatio：Survivor 区的「空间分配」

`-XX:SurvivorRatio` 设置 Eden 区与单个 Survivor 区的比例，**默认值为 8**。

```bash
# SurvivorRatio=8 表示 Eden : S0 : S1 = 8 : 1 : 1
# 即每个 Survivor 占年轻代的 1/10
-XX:SurvivorRatio=8

# SurvivorRatio=4 表示 Eden : S0 : S1 = 4 : 1 : 1
# 即每个 Survivor 占年轻代的 1/6，年轻代内部 Survivor 占比更大
-XX:SurvivorRatio=4
```

为什么要调整 SurvivorRatio？**核心目的是防止对象过早晋升老年代。**

如果 Survivor 区太小，对象在 Survivor 区来回复制的次数就会减少，可能还没「成年」就被迫晋升老年代。这会加速老年代的填充，导致 Full GC 越来越频繁。

如果 Survivor 区太大，又会压缩 Eden 区的空间，导致 Minor GC 过于频繁。

:::tip SurvivorRatio 的调整思路

1. **如果对象晋升老年代太快**：调小 SurvivorRatio（增大 Survivor 区）
2. **如果 Minor GC 太频繁**：调大 SurvivorRatio（增大 Eden 区）
3. **如果对象年龄分布偏大**：考虑调整 `-XX:MaxTenuringThreshold`

**监控方法是关键**。用 `jstat -gc` 观察 Survivor 区的使用率，如果 S0 或 S1 经常接近满载，说明 Survivor 区可能太小。

:::

## 三、对象晋升规则：谁决定了你的「退休年龄」

### 3.1 年龄阈值：对象的「法定退休年龄」

JVM 为每个对象维护一个 `age`（年龄）计数器。每次对象在 Survivor 区复制一次，年龄就加 1。当年龄达到 `-XX:MaxTenuringThreshold` 时，对象就会晋升老年代。

**默认值是 15**。这意味着一个对象在年轻代最多经历 15 次 Minor GC 后，无论是否还「年轻」，都会被强制晋升老年代。

```bash
# 设置最大晋升年龄为 8
-XX:MaxTenuringThreshold=8

# 设置最大晋升年龄为 1（对象第二次 Minor GC 后直接晋升）
# ⚠️ 慎用，会导致大量对象快速进入老年代
-XX:MaxTenuringThreshold=1
```

为什么要设置年龄上限？因为 Survivor 区空间有限，不可能让所有对象无限复制下去。设置上限可以强制老年代承担「养老」职责。

### 3.2 动态年龄计算：JVM 的「自适应策略」

JVM 还有一个隐藏的晋升规则：**动态年龄计算**。

> 如果 Survivor 区中，相同年龄的所有对象大小之和超过 Survivor 区的 50%，则年龄大于或等于该年龄的对象可以直接晋升老年代。

这个规则的目的是：**让 Survivor 区始终保持足够的空闲空间。** 如果某个年龄段的对象太多，即使它们的年龄还没到 MaxTenuringThreshold，也会提前晋升，以腾出 Survivor 区的空间。

```bash
# 示例：Survivor 区 100MB
# age=3 的对象共 60MB，超过 50MB 阈值
# 所有 age >= 3 的对象都会直接晋升老年代
```

这个规则往往被忽视。很多时候，对象之所以提前晋升老年代，不是因为年龄到了，而是因为 Survivor 区空间不足。

### 3.3 大对象直接晋升：越过青春直接「退休」

有一个例外情况：**大对象直接进入老年代**，完全跳过年轻代。

这个阈值由 `-XX:PretenureSizeThreshold` 控制。

```bash
# 设置超过 1MB 的对象直接在老年代分配
-XX:PretenureSizeThreshold=1M
```

:::warning PretenureSizeThreshold 的注意事项

1. **只有 Parallel GC 支持这个参数。** CMS 和 G1 可能不生效或行为不同。
2. **这个参数是「直接分配到老年代」，不是「先分配再移动」。** 大对象在老年代分配可能失败，导致 Full GC。
3. **大对象直接进老年代会加速老年代填充。** 谨慎使用。

:::

### 3.4 Survivor 区空间不足：最后的晋升触发器

即使年龄没到阈值、不是大对象，**Survivor 区空间不足时，对象也会直接晋升老年代。**

这是最常见的对象提前晋升原因。当 Survivor 区满了，JVM 没有其他选择，只能把一些对象提前送到老年代「避难」。

```bash
# 观察 Survivor 区使用情况
jstat -gc <pid>

# 输出示例：
# S0C    S1C    S0U    S1U
# 133120.0 133120.0 125000.0 0.0
# S0U 接近 S0C，说明 Survivor 区快满了
```

## 四、容器环境下的堆配置

### 4.1 容器内存限制的挑战

在 Docker/Kubernetes 环境中运行 Java 应用，JVM 默认不知道容器的内存限制。它会按照物理机的内存来估算可用堆大小，可能导致：

- JVM 申请的内存超过容器限制，被 OOM Killer 杀掉
- JVM 申请的内存远小于容器限制，浪费资源

### 4.2 容器支持参数：JVM 的「自适应能力」

JDK 10 引入了 `-XX:+UseContainerSupport`，JDK 11 引入了更完善的容器感知机制。

```bash
# JDK 10+ 自动感知容器内存限制（默认开启）
-XX:+UseContainerSupport

# 按容器内存的百分比设置堆大小
# 初始堆 = 容器内存 × 50%
-XX:InitialRAMPercentage=50

# 最大堆 = 容器内存 × 50%
-XX:MaxRAMPercentage=50
```

```bash
# 示例：容器限制 4GB 内存
# JVM 自动将最大堆设为 4GB × 50% = 2GB
java -XX:MaxRAMPercentage=50 -jar app.jar
```

:::tip 容器内存配置建议

| 容器内存 | 推荐 MaxRAMPercentage | 理由 |
|---|---|---|
| 2GB | 50% - 60% | 堆 1-1.2GB，剩余给元空间、Direct Buffer 等 |
| 4GB | 50% - 70% | 堆 2-2.8GB |
| 8GB | 60% - 75% | 堆 4.8-6GB |
| 16GB+ | 70% - 80% | 堆 11-12.8GB |

容器内存越大，可以给堆的比例越高。元空间、Direct Buffer、CodeCache 等非堆内存通常 1-2GB 就够了。

:::

### 4.3 手动指定容器内存上限

如果 JVM 版本不支持容器感知参数，可以手动指定：

```bash
# 手动指定最大可用内存为 2GB
-XX:MaxRAM=2g
```

但这不如容器感知参数智能，因为它不会自动随容器限制变化。

## 五、常见配置组合与场景分析

### 5.1 标准 Web 服务配置

```bash
# 4核8GB 机器，运行 Spring Boot 应用
-Xms4g -Xmx4g \
-XX:NewRatio=2 \
-XX:SurvivorRatio=8 \
-XX:MaxTenuringThreshold=15 \
-XX:+UseContainerSupport \
-XX:MaxRAMPercentage=50
```

**分析**：

- 固定堆 4GB，避免运行时伸缩
- 年轻代约 1.3GB，Eden 约 1GB，Survivor 各约 130MB
- 适合对象生命周期较短的 Web 服务
- 使用容器感知参数，适配 Kubernetes 环境

### 5.2 批处理任务配置

```bash
# 批处理任务，追求吞吐量
-Xms8g -Xmx8g \
-XX:NewRatio=3 \
-XX:SurvivorRatio=8 \
-XX:MaxTenuringThreshold=20
```

**分析**：

- 年轻代更小（8GB / 4 = 2GB），老年代更大（6GB）
- 对象存活时间长，适合批处理场景
- MaxTenuringThreshold 调大，让对象在年轻代待更久

### 5.3 延迟敏感服务配置

```bash
# 延迟敏感服务（如游戏后端、实时计算）
-Xms4g -Xmx4g \
-Xmn2g \
-XX:SurvivorRatio=4 \
-XX:MaxTenuringThreshold=8
```

**分析**：

- 手动指定年轻代 2GB，占堆的 50%
- SurvivorRatio=4，每个 Survivor 占年轻代的 1/6（约330MB）
- 更频繁的 Minor GC，但每次停顿更短
- 适合追求低延迟的场景

### 5.4 大对象场景配置

```bash
# 处理大文件的场景
-Xms8g -Xmx8g \
-Xmn4g \
-XX:SurvivorRatio=4 \
-XX:PretenureSizeThreshold=2M
```

**分析**：

- 年轻代 4GB，足够容纳大对象
- 大于 2MB 的对象直接在老年代分配
- SurvivorRatio=4，Survivor 区更大，减少对象频繁复制

## 六、堆内存配置的决策框架

### 6.1 配置前的准备工作

在调整堆参数之前，必须先搞清楚几个问题：

1. **当前堆使用情况如何？** 用 `jstat -gc`、`jmap -heap` 或 APM 工具观察。
2. **GC 频率和耗时如何？** Minor GC 多久一次？Full GC 多久一次？每次停顿多长时间？
3. **对象的年龄分布如何？** Survivor 区使用情况怎么样？有没有对象提前晋升？
4. **业务类型是什么？** Web 服务、批处理、实时计算？

### 6.2 决策流程图

```
业务类型是什么？
├─ Web 服务（延迟敏感）
│   └─ 年轻代设大一些（NewRatio=1 或更小）
│       SurvivorRatio 适中（4-8）
│       MaxTenuringThreshold 适中（8-15）
│
├─ 批处理（吞吐优先）
│   └─ 年轻代可以小一些（NewRatio=2-3）
│       MaxTenuringThreshold 可以大一些（15-20）
│
└─ 大对象为主
    └─ 年轻代要大（Xmn 或小 NewRatio）
        PretenureSizeThreshold 考虑设置
```

### 6.3 调整后的验证方法

每次调整参数后，必须验证效果：

```bash
# 1. 启动时观察 GC 日志
-Xlog:gc*:file=gc.log:time,uptime,level,tags

# 2. 运行期间观察 GC 统计
jstat -gcutil <pid> 1000

# 3. 关注的关键指标
# - FGC（Full GC 次数）：应该接近 0 或很少
# - FGCT（Full GC 总耗时）：应该很短
# - NGC（年轻代 GC 次数）：不应该太频繁
# - O（老年代占用率）：应该稳定，不持续增长
```

## 七、面试实战

### 7.1 场景一：堆参数选择

**面试官问**：「线上有个服务，内存 8GB，GC 停顿经常超过 500ms，怎么优化？」

这个问题考察的是对 GC 行为的理解和参数调整能力。

**分析思路**：

1. **先问清楚现状**：用的是什么 GC 收集器？GC 日志显示什么？停顿主要来自 Minor GC 还是 Full GC？

2. **如果是 Minor GC 停顿长**：说明年轻代太大或 Survivor 区不够。考虑减小年轻代、增大 SurvivorRatio。

3. **如果是 Full GC 停顿长**：说明老年代填充太快。考虑增大堆、减小年轻代比例、排查内存泄漏。

4. **给出具体建议**：

> 如果 Minor GC 停顿长，我会：
> - 减小年轻代大小：`-Xmn2g`（从 4g 减小到 2g）
> - Minor GC 会更频繁，但每次停顿更短
> - 同时观察 Survivor 区使用情况，调整 `-XX:SurvivorRatio`
>
> 如果 Full GC 频繁，我会：
> - 首先排查内存泄漏（`jmap -dump` + MAT 分析）
> - 如果不是内存泄漏，考虑增大堆或调整年轻代/老年代比例
> - 考虑切换到 G1 GC，设置 `-XX:MaxGCPauseMillis=200`

### 7.2 场景二：Survivor 区配置

**面试官问**：「年轻代 1GB，SurvivorRatio=8，为什么 Survivor 区还是经常满？」

**回答**：

这个问题有几种可能：

1. **对象年龄分布集中**。如果大量对象的年龄相近（都是 2-3 岁），而这些对象的总大小超过 Survivor 区的 50%，JVM 会触发动态年龄计算，让这些对象提前晋升。这是正常的保护机制。

2. **Survivor 区 1GB × 10% = 100MB，实际可用空间只有 100MB**。对于高并发服务，100MB 可能不够。

3. **Minor GC 间隔太短**。对象在两次 Minor GC 之间累积的速度超过了 Survivor 区的处理能力。

**建议的排查步骤**：

```bash
# 1. 观察 Survivor 区使用率
jstat -gc <pid> 1000

# 2. 如果 S0U 或 S1U 经常接近 S0C 或 S1C
# 说明 Survivor 区确实太小，需要调大

# 3. 调整建议
# 方案 A：增大 SurvivorRatio（减少 Eden，增加 Survivor）
-XX:SurvivorRatio=4  # 每个 Survivor 从 10% 增加到 16.7%

# 方案 B：减小年轻代，增加 Survivor 在年轻代中的占比
-Xmn800m  # 减小年轻代，但比例不变

# 方案 C：直接固定年轻代大小
-Xmn1g -XX:SurvivorRatio=4  # 年轻代 1GB，Survivor 各占 160MB
```

### 7.3 场景三：容器内存配置

**面试官问**：「K8s 容器限制 4GB，Java 进程被 OOM Kill 了，怎么回事？」

**回答**：

这是典型的 JVM 内存配置与容器限制不匹配的问题。

**原因分析**：

1. **JVM 默认堆可能设置过大**。如果 `-Xmx` 设了 4GB，加上元空间、Direct Buffer、线程栈等，进程总内存可能超过 4GB。
2. **JVM 版本不支持容器感知**。JDK 8 及更早版本默认不知道容器内存限制。
3. **元空间等非堆内存挤占了容器内存**。

**解决方案**：

```bash
# JDK 10+
-XX:MaxRAMPercentage=50  # 最大堆占容器内存的 50%

# JDK 8（不支持 MaxRAMPercentage）
-Xmx2g  # 手动指定最大堆
-XX:MaxMetaspaceSize=256m  # 限制元空间
-XX:MaxDirectMemorySize=512m  # 限制直接内存
```

**关键原则**：容器内 Java 进程的总内存使用 = 堆 + 元空间 + 直接内存 + 线程栈 × 线程数 + JVM 自身开销。留 10-20% 的余量。

## 总结

堆内存参数配置，核心就是回答三个问题：

1. **堆多大？** `-Xms` 和 `-Xmx`，生产环境建议设成相同值
2. **年轻代多大？** `-Xmn` 或 `-XX:NewRatio`，根据业务类型选择
3. **Survivor 区多大？** `-XX:SurvivorRatio`，根据对象年龄分布调整

但参数配置只是手段，**监控和验证才是核心**。每次调整后，用 GC 日志和 `jstat` 观察效果，让数据告诉你配置是否合理。

记住：**没有最好的配置，只有最适合的配置。** 脱离业务场景谈参数，都是耍流氓。
