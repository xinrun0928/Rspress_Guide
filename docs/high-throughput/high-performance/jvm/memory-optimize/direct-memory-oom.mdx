# 直接内存溢出：堆外的隐形杀手

你遇到过这种情况吗？

- 堆内存使用正常，GC 很正常
- 但进程占用的物理内存不断飙升
- 最终 OOM killed 或 OutOfMemoryError: Direct buffer memory

这就是直接内存（Direct Memory）泄漏，也叫**堆外内存泄漏**。

它之所以危险，是因为：**你看不见**。JVM 的堆内存监控、各种 GC 工具都监控不到这块区域。如果你不主动配置监控，直接内存泄漏就是"隐形杀手"。

今天，我们来揭开直接内存的面纱。

---

## 一、直接内存是什么

### 1.1 定义与特点

直接内存（Direct Memory）是 JVM 堆外分配的内存，不受 `-Xmx` 堆大小限制。它由操作系统直接管理，用于 Java 程序与外部 I/O 交互。

```
┌─────────────────────────────────────────────────────────────┐
│                         进程内存                            │
│  ┌───────────────────────┐    ┌───────────────────────┐  │
│  │       JVM 堆内存       │    │      直接内存          │  │
│  │   (-Xmx 控制大小)      │    │  (不受 -Xmx 限制)      │  │
│  │                       │    │                       │  │
│  │  ┌─────┐ ┌─────┐     │    │   ┌─────────────┐     │  │
│  │  │ Eden│ │Old  │     │    │   │DirectBuffer │     │  │
│  │  │     │ │ Gen │     │    │   │  (NIO/Netty)│     │  │
│  │  └─────┘ └─────┘     │    │   └─────────────┘     │  │
│  └───────────────────────┘    └───────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

**直接内存的特点**：

- 不受 JVM 堆大小限制
- 直接与操作系统交互，减少 JVM 堆和内核之间的数据拷贝
- 用于 NIO 的 ByteBuffer、Netty 的 ByteBuf、文件映射等
- 需要手动管理或依赖 GC 自动回收

### 1.2 为什么需要直接内存

传统的 JVM I/O 操作是这样的：

```
用户进程 ←→ JVM 堆内存 ←→ 内核缓冲区 ←→ 磁盘/网络
              ↑                  ↑
           数据拷贝            数据拷贝
```

1. 从磁盘/网络读取数据到内核缓冲区
2. 从内核缓冲区拷贝到 JVM 堆内存
3. 应用程序处理数据

**直接内存**改变了这个过程：

```
用户进程 ←→ 直接内存 ←→ 内核缓冲区 ←→ 磁盘/网络
            ↑              ↑
         数据拷贝        数据拷贝（零拷贝跳过这一步）
```

数据可以直接在直接内存和内核缓冲区之间传输，**避免了 JVM 堆和内核之间的数据拷贝**，这就是零拷贝（Zero Copy）的核心思想。

### 1.3 分配方式

```java
// 方式一：NIO DirectByteBuffer
ByteBuffer buffer = ByteBuffer.allocateDirect(1024 * 1024);  // 分配 1MB 直接内存

// 方式二：Netty ByteBuf
ByteBuf byteBuf = Unpooled.directBuffer(1024);  // Netty 堆外内存
```

## 二、直接内存的工作原理

### 2.1 分配与释放机制

直接内存的分配和释放是一个"协作"过程：

```
┌─────────────────────────────────────────────────────────────┐
│                    DirectByteBuffer 分配流程                  │
│                                                             │
│  1. 分配请求                                                │
│     ByteBuffer.allocateDirect(1024)                         │
│                    ↓                                        │
│  2. JVM 调用 native 方法分配内存                            │
│     Unsafe.allocateMemory(size)                             │
│                    ↓                                        │
│  3. 操作系统分配物理内存                                     │
│                    ↓                                        │
│  4. 创建 DirectByteBuffer 对象（堆内）                      │
│     + 记录内存地址和大小                                     │
│     + 创建 Cleaner（虚引用）                                 │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 GC 如何回收直接内存

这是关键问题：**直接内存不在 JVM 堆里，GC 能回收吗？**

答案是：**能，但要依赖 DirectByteBuffer 对象的回收**。

```
┌─────────────────────────────────────────────────────────────┐
│                    直接内存回收机制                          │
│                                                             │
│  DirectByteBuffer 对象（堆内）                              │
│       │                                                     │
│       │ 堆内对象被 GC 回收时                                  │
│       ↓                                                     │
│  Cleaner（虚引用）被加入 ReferenceQueue                      │
│       │                                                     │
│       │  ReferenceHandler 线程检测到                        │
│       ↓                                                     │
│  调用 Cleaner.clean()                                       │
│       │                                                     │
│       ↓                                                     │
│  释放 native 内存（Unsafe.freeMemory）                      │
└─────────────────────────────────────────────────────────────┘
```

**结论**：直接内存的回收依赖于 `DirectByteBuffer` 对象被 GC 回收。如果 `DirectByteBuffer` 对象泄漏了，直接内存也会泄漏。

## 三、直接内存溢出原因

### 3.1 原因一：NIO 使用不当

最常见的问题是：创建了 `DirectByteBuffer` 对象，但没有及时释放。

```java
// 场景一：频繁创建 DirectByteBuffer
public ByteBuffer readData() {
    // 每次调用都分配直接内存
    ByteBuffer buffer = ByteBuffer.allocateDirect(1024 * 1024);
    channel.read(buffer);
    return buffer;  // 如果没有复用，GC 回收前会一直占用
}

// 场景二：直接内存累积
// 高并发场景下，每个请求创建一个 1MB 的 DirectByteBuffer
// 如果 QPS = 1000，这些对象会快速占满直接内存
```

### 3.2 原因二：Netty 堆外内存泄漏（最常见）

Netty 是 Java 最流行的网络框架，它的 ByteBuf 默认使用堆外内存（PooledByteBufAllocator）。

**ByteBuf 泄漏的场景**：

```java
// 场景一：忘记释放 ByteBuf
public void handle(ByteBuf buf) {
    // 处理数据...
    // 如果忘记调用 buf.release()，内存泄漏！
    byte[] data = new byte[buf.readableBytes()];
    buf.readBytes(data);
    // 缺少: buf.release();
}

// 场景二：异常路径未释放
public void process(ByteBuf buf) {
    try {
        // 处理业务逻辑
        doSomething(buf);
    } catch (Exception e) {
        log.error("Process error", e);
        // 缺少: buf.release();
        throw e;
    }
    // 正常路径可能也不会 release
}

// 场景三：ChannelHandler 未正确管理生命周期
public class MyHandler extends ChannelInboundHandlerAdapter {
    @Override
    public void channelRead(ChannelHandlerContext ctx, Object msg) {
        // ByteBuf 没有正确释放
        ctx.fireChannelRead(msg);
    }
}
```

### 3.3 原因三：JNI 调用分配的直接内存

通过 JNI 调用 native 代码时，如果 native 代码分配了直接内存但没有正确释放：

```java
// JNI 侧代码
JNIEXPORT void JNICALL Java_com_example_Native_alloc
    (JNIEnv *env, jobject obj, jlong size) {
    // 分配 native 内存
    void *ptr = malloc(size);
    // 如果忘记调用 free(ptr)，内存泄漏！
}
```

### 3.4 原因四：mmap 文件映射未释放

使用 `FileChannel.map()` 映射大文件时，如果映射未关闭：

```java
// 映射一个大文件（比如 4GB）
RandomAccessFile file = new RandomAccessFile("bigfile.dat", "r");
MappedByteBuffer buffer = file.getChannel().map(
    FileChannel.MapMode.READ_ONLY, 0, file.length()
);

// 如果 file 没有正确关闭，mmap 映射不会被释放
```

## 四、Netty 堆外内存泄漏排查（重点）

### 4.1 堆外内存配置参数

Netty 提供了完善的堆外内存配置：

```bash
# 堆外内存最大限制（不设限制可能导致系统 OOM）
-Dio.netty.maxDirectMemory=1073741824   # 1GB

# 堆外内存分配器类型
# pooled：池化（默认，复用内存块）
# unpooled：非池化（每次分配新内存）
-Dio.netty.allocator.type=pooled        # 推荐 pooled

# 内存泄漏检测级别
-Dio.netty.leakDetection.level=disabled      # 禁用
-Dio.netty.leakDetection.level=simple        # 简单检测（默认）
-Dio.netty.leakDetection.level=advanced      # 高级检测
-Dio.netty.leakDetection.level=paranoid      # 极度检测（每次分配都检测）
```

### 4.2 泄漏检测级别详解

| 级别 | 说明 | 性能影响 |
|---|---|---|
| `disabled` | 不检测 | 无 |
| `simple` | 抽样 1% 的 ByteBuf，检测泄漏 | < 5% |
| `advanced` | 检测所有 ByteBuf 的泄漏点 | < 10% |
| `paranoid` | 每次分配都检测 | 可能有显著影响 |

**生产环境推荐**：`simple` 或 `advanced`

```bash
# 启动时添加参数
java -Dio.netty.leakDetection.level=advanced -jar app.jar
```

### 4.3 泄漏检测报告解读

开启泄漏检测后，Netty 会在控制台输出泄漏报告：

```
LEAK: ByteBuf.release() was not called before it's garbage-collected.
Recent access locations:
    - io.netty.buffer.SimpleLeakAwareByteBuf.<init>(SimpleLeakAwareByteBuf.java:48)
    - io.netty.buffer.PooledByteBufAllocator.newDirectBuffer(PooledByteBufAllocator.java:349)
    - io.netty.buffer.AbstractReferenceCountedByteBuf.retain(AbstractReferenceCountedByteBuf.java:89)
    - com.example.NettyClient.channelRead(NettyClient.java:45)
```

这个报告告诉你：

1. 泄漏的 ByteBuf 在哪里创建
2. 最近一次访问位置在哪里
3. 调用栈是什么

### 4.4 常见泄漏原因与解决

#### 4.4.1 ChannelHandler 未正确释放

```java
// 错误写法
public class BadHandler extends ChannelInboundHandlerAdapter {
    @Override
    public void channelRead(ChannelHandlerContext ctx, Object msg) {
        // ByteBuf 没有释放
        ctx.fireChannelRead(msg);
    }
}

// 正确写法
public class GoodHandler extends ChannelInboundHandlerAdapter {
    @Override
    public void channelRead(ChannelHandlerContext ctx, Object msg) {
        try {
            // 处理消息
            ctx.fireChannelRead(msg);
        } finally {
            // 释放 ByteBuf
            ReferenceCountUtil.release(msg);
        }
    }

    @Override
    public void exceptionCaught(ChannelHandlerContext ctx, Throwable cause) {
        // 异常时也要释放
        ctx.close();
    }
}
```

#### 4.4.2 异常路径未释放

```java
// 错误写法
public void handle(ByteBuf buf) {
    if (!isValid(buf)) {
        throw new IllegalArgumentException("Invalid buffer");
        // buf 没有被释放！
    }
    // 正常处理...
}

// 正确写法
public void handle(ByteBuf buf) {
    try {
        if (!isValid(buf)) {
            throw new IllegalArgumentException("Invalid buffer");
        }
        // 正常处理...
    } finally {
        buf.release();
    }
}
```

#### 4.4.3 使用 try-with-resources 自动释放

```java
// 推荐写法：ByteBuf 实现 AutoCloseable
public ByteBuf process(ByteBuf input) {
    ByteBuf result = null;
    try {
        result = ctx.alloc().directBuffer();
        // 处理逻辑
        return result;
    } catch (Exception e) {
        if (result != null) {
            result.release();
        }
        throw e;
    }
}

// 更好的方式：确保释放
public void process(ByteBuf input) {
    try (ByteBuf buf = input) {
        // 处理逻辑
    }  // 自动释放
}
```

## 五、直接内存监控

### 5.1 配置最大直接内存

```bash
# 方式一：通过 JVM 参数
-XX:MaxDirectMemorySize=1g

# 方式二：通过代码
System.setProperty("io.netty.maxDirectMemory", String.valueOf(bytes));
```

当直接内存超过这个值时，Netty 会主动触发 Full GC 来回收无法回收的 DirectByteBuffer。

### 5.2 NMT（Native Memory Tracking）

JVM 内置的 NMT 可以监控直接内存使用：

```bash
# 启动时开启 NMT
-XX:NativeMemoryTracking=summary   # 摘要级别
# 或
-XX:NativeMemoryTracking=detail   # 详细级别

# 查看 NMT 输出
jcmd <PID> VM.native_memory summary
```

输出示例：

```
Native Memory Tracking:

Total: reserved=1543MB, committed=543MB
-                  Java Heap (reserved=512MB, committed=512MB)
-                      Class (reserved=89MB, committed=45MB)
-                      Thread (reserved=123MB, committed=123MB)
-                        Code (reserved=45MB, committed=25MB)
-                        GC (reserved=234MB, committed=234MB)
-                  Internal (reserved=45MB, committed=25MB)
-                        Direct (reserved=256MB, committed=256MB)   ← 直接内存
-                        Mapped (reserved=0MB, committed=0MB)
```

:::tip 监控建议
- 定期用 NMT 检查直接内存使用
- 如果 `committed` 持续增长，可能是内存泄漏
- 设置合理的 `-XX:MaxDirectMemorySize` 作为保护
:::

### 5.3 其他监控手段

#### 5.3.1 进程内存监控

```bash
# 查看进程总内存
ps -eo pid,rss,vsz,comm | grep java

# rss: 物理内存使用
# vsz: 虚拟内存大小
```

#### 5.3.2 Arthas 监控

```bash
# 使用 Arthas 的 memory 命令
memory

# 输出中包含 direct 区域
# direct: direct memory 使用情况
```

## 七、面试实战

### 面试题一：什么是直接内存？和堆内存有什么区别？

**回答要点**：

```
直接内存是 JVM 堆外分配的内存，不受 -Xmx 限制。

主要区别：

1. 位置不同
   - 堆内存：JVM 进程内，受 -Xmx 控制
   - 直接内存：操作系统层面，不受 JVM 管理

2. 管理方式不同
   - 堆内存：由 GC 管理，自动回收
   - 直接内存：依赖 DirectByteBuffer 对象的 GC 间接回收

3. 用途不同
   - 堆内存：普通对象存储
   - 直接内存：NIO 读写、Netty 通信、文件映射等 I/O 操作

4. 性能不同
   - 直接内存避免了 JVM 堆和内核之间的数据拷贝
   - 对于高 I/O 场景，直接内存性能更好
```

### 面试题二：直接内存泄漏如何排查？

**回答要点**：

```
排查直接内存泄漏，我一般分几步：

第一步：确认是直接内存问题
   - NMT（Native Memory Tracking）查看
   - jcmd <PID> VM.native_memory summary

第二步：如果是 Netty 问题
   - 开启泄漏检测：-Dio.netty.leakDetection.level=advanced
   - 查看泄漏报告定位代码位置

第三步：常见泄漏点检查
   - ByteBuf 是否都 release 了？
   - exceptionCaught 是否关闭连接？
   - 异常路径是否释放资源？

第四步：监控直接内存上限
   - 设置 -XX:MaxDirectMemorySize
   - 超过阈值时 JVM 会主动触发 GC
```

### 面试题三：为什么 GC 能回收直接内存？

**回答要点**：

```
这是很多人会搞混的问题。

直接内存本身不受 GC 管理，但 GC 能回收直接内存的原因在于：

1. DirectByteBuffer 对象在堆内
   - 创建直接内存时，会创建一个对应的 DirectByteBuffer 对象
   - 这个对象在堆内，由 GC 管理

2. Cleaner 机制
   - DirectByteBuffer 对象包含一个 Cleaner（虚引用）
   - 当 DirectByteBuffer 对象被 GC 回收时
   - Cleaner 被加入 ReferenceQueue
   - ReferenceHandler 线程会调用 Cleaner.clean()
   - 最终释放 native 内存

3. 所以
   - 只要 DirectByteBuffer 对象被及时回收，直接内存就会被释放
   - 如果 DirectByteBuffer 对象泄漏（无法被 GC），直接内存也会泄漏
```

## 八、总结

直接内存是 Java 高性能 I/O 的基石，但也是容易被忽略的内存泄漏源。

核心要点：

1. **直接内存不受 -Xmx 限制**，但可以通过 `-XX:MaxDirectMemorySize` 限制
2. **依赖 GC 回收**：直接内存的回收依赖于 `DirectByteBuffer` 对象的回收
3. **Netty 是重灾区**：ByteBuf 泄漏是最常见的直接内存泄漏原因
4. **开启泄漏检测**：`-Dio.netty.leakDetection.level=advanced`
5. **用 NMT 监控**：定期用 `jcmd` 检查直接内存使用
