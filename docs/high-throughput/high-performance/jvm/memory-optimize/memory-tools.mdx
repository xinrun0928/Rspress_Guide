# JVM 内存分析工具三件套：jmap、jhat、MAT 深度指南

工欲善其事，必先利其器。

JVM 内存问题排查，离不开一系列工具的支持。从命令行的 `jmap`、`jstat`，到可视化的 MAT、VisualVM，再到阿里巴巴开源的 Arthas，每一款工具都有其独到的用武之地。

这篇文章，我把 JVM 内存分析的核心工具全部讲透：从命令行工具的基本用法，到可视化工具的深度分析，再到工具之间的配合使用。读完这篇，你就能建立起完整的 JVM 内存分析工具体系。

## 一、工具全景图：选对工具是第一步

### 1.1 命令行工具 vs 可视化工具

JVM 自带了一系列命令行诊断工具，位于 `$JAVA_HOME/bin` 目录下：

| 工具 | 用途 | 特点 |
|---|---|---|
| jmap | 堆内存映像、对象统计 | 快速生成 Dump，适合紧急情况 |
| jhat | Heap Dump 分析（已废弃） | 曾是分析标配，现在用 MAT 替代 |
| jstat | GC 统计、类加载统计 | 实时监控，轻量级 |
| jcmd | 多功能诊断工具 | JDK 7+ 推荐，替代 jmap/jhat |
| jinfo | JVM 配置信息 | 查看和修改运行时参数 |

可视化工具则提供了更友好的交互：

| 工具 | 用途 | 特点 |
|---|---|---|
| MAT | Heap Dump 深度分析 | 内存泄漏分析首选 |
| VisualVM | 实时监控 + Dump 分析 | 轻量级，一站式 |
| JMC | Flight Recorder 录制 | 低开销持续监控 |
| Arthas | 线上诊断神器 | 不需要改代码，attach 到进程 |

### 1.2 工具选择决策树

```
JVM 内存问题
├── 实时监控（想知道现在怎么样）
│   ├── 命令行 → jstat -gcutil
│   └── 可视化 → VisualVM / Arthas dashboard
│
├── 堆转储分析（OOM 了，要分析原因）
│   ├── 小 Dump（< 2GB）→ MAT
│   ├── 大 Dump（> 2GB）→ 保守使用 MAT / jhat
│   └── 在线分析 → Arthas heapdump
│
└── 在线诊断（不知道哪个方法有问题）
    └── Arthas：watch / trace / profiler
```

## 二、jmap 详解：命令行 Heap Dump

### 2.1 jmap 基础用法

`jmap`（JVM Memory Map）是生成堆转储和查看内存布局的命令行工具。

```bash
# 基本语法
jmap [options] <pid>

# 常用选项
jmap -heap <pid>      # 查看堆配置和使用情况
jmap -histo <pid>     # 列出对象统计
jmap -clstats <pid>   # 类加载器统计
jmap -dump <options>  # 生成堆转储
```

### 2.2 查看堆配置和使用情况

```bash
jmap -heap <PID>
```

输出示例：

### 2.3 列出对象统计

```bash
# 查看对象统计（按 Shallow Size 排序）
jmap -histo <PID> | head -50

# 只看前 30 行
jmap -histo <PID> | head -30

# 输出示例
 num     #instances         #bytes  class name
----------------------------------------------
   1:          12345      98765440  [Ljava.lang.Object;
   2:           8901      45678912  com.example.service.UserService
   3:           6789      34567890  java.util.HashMap$Node
   4:           5678      23456789  java.lang.String
   5:           4567      12345678  [B
   6:           3456      9876543   java.util.ArrayList
   ...
```

参数说明：

- `#instances`：对象实例数量
- `#bytes`：对象占用的字节数（Shallow Size）
- `class name`：类的全限定名

```bash
# 进一步筛选：只看自定义类的对象
jmap -histo <PID> | grep com.example

# 按内存占用排序，找出最大的对象
jmap -histo:live <PID> | sort -k3 -n -r | head -20
```

:::warning jmap -histo:live 会触发 Full GC

`-histo:live` 会先触发一次 Full GC 再统计，**生产环境慎用**。

:::

### 2.4 生成堆转储

```bash
# 方式一：正常生成（推荐）
jmap -dump:format=b,file=/tmp/heap.hprof <PID>

# 方式二：设置自动生成（推荐用于生产环境）
# 在 JVM 启动参数中添加
-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/data/logs/heap.hprof

# 方式三：强制生成（进程卡住时使用）
jmap -F -dump:format=b,file=/tmp/heap.hprof <PID>

# 方式四：只 dump 存活对象（减少 dump 大小）
jmap -dump:live,format=b,file=/tmp/heap_live.hprof <PID>
```

Dump 文件大小估算：通常是堆大小的 0.5 - 1 倍。4GB 堆的 Dump 可能达到 2-4GB。

### 2.5 类加载器统计

```bash
jmap -clstats <PID>
```

输出示例：

```
Index         Supercount    Bytes  Class Loader
    1                1       424   <bootstrap>
    2                1      1488   sun.reflect.DelegatingClassLoader@5c8da962
    3               12     13456   com.example.MyClassLoader@6d06d69c
    4                3      2048    sun.misc.Launcher$AppClassLoader@18b4aac2
```

用于分析类加载器的使用情况，特别适合排查 Metaspace 泄漏问题。

## 三、jhat 详解：曾经的标配，如今的鸡肋

### 3.1 jhat 的历史地位

`jhat`（JVM Heap Analysis Tool）在 JDK 8 之前是 Heap Dump 分析的标准工具。它启动一个 HTTP 服务器，让你通过浏览器查看 Dump 内容。

```bash
# 启动 jhat
jhat heap.hprof

# 启动时指定端口
jhat -port 7000 heap.hprof

# 使用自定义 OQL 查询
jhat -parseheapdump heap.hprof
```

启动后，访问 http://localhost:7000 即可查看分析结果。

### 3.2 jhat 的局限性

jhat 有几个致命问题：

1. **无法分析大文件**：只支持小于 2GB 的 Dump
2. **分析速度慢**：启动和查询都很慢
3. **功能有限**：不支持深堆计算、GC Roots 追溯等高级功能
4. **已被标记为废弃**：JDK 9 正式废弃，JDK 11 移除

**现代替代方案**：使用 MAT 代替 jhat，功能更强、速度更快。

### 3.3 jhat 的 OQL 查询

虽然 jhat 已经被淘汰，但它的 OQL 语法被 MAT 继承了下来。了解 jhat 的 OQL，对使用 MAT 的 OQL 有帮助。

```
http://localhost:7000/oql/
```

OQL 查询示例：

```sql
-- 查询所有 HashMap
SELECT x FROM java.util.HashMap x

-- 查询超过 1000 个元素的 Map
SELECT x FROM java.util.HashMap x WHERE x.size > 1000

-- 查询包含特定字段的对象
SELECT {obj: o, field: f} FROM com.example.User o
WHERE o.name.toString().indexOf("admin") >= 0
```

## 四、MAT 深度使用：从入门到精通

### 4.1 MAT 简介

MAT（Memory Analyzer Tool）是 Eclipse 旗下的开源工具，专门用于分析 Heap Dump。它比 jhat 强大得多，是生产环境内存问题排查的首选。

下载地址：https://eclipse.dev/mat/downloads.php

### 4.2 打开 Heap Dump

```bash
# 方法一：MAT GUI
./MemoryAnalyzer heap.hprof

# 方法二：命令行分析
./ParseHeapDump.sh heap.hprof org.eclipse.mat.api:suspects
```

MAT 打开 Dump 后，显示欢迎页面：

- **Leak Suspects**：自动识别内存泄漏嫌疑人
- **Top Consumers**：内存占用最大的对象
- **Histogram**：按类统计对象
- **Dominator Tree**：对象支配树

### 4.4 Histogram：按类统计

Histogram 视图列出所有类的实例数量和内存占用。

**基本操作**：

1. 点击工具栏的「Histogram」图标
2. 按「Retained Heap」列排序
3. 右键某个类，选择操作

**常用操作**：

| 操作 | 说明 |
|---|---|
| 「List objects」→「with outgoing references」 | 查看这个类的对象持有的引用 |
| 「List objects」→「with incoming references」 | 查看持有这个类对象的所有引用 |
| 「Group by package」 | 按包分组 |
| 「Compare Heap objects」 | 比较两个 Heap Dump |

**分组视图**：

### 4.5 Dominator Tree：对象支配树

Dominator Tree 展示对象之间的支配关系，是分析内存占用的利器。

**什么是支配关系**：如果所有到达 B 的路径都必须经过 A，则 A 支配 B。

```java
// 支配关系示例
A
└── B
    └── C
```

这里 A 支配 B 和 C，因为没有 A 就无法到达 B 和 C。

**Dominator Tree 特点**：

- 树根是 GC Roots
- 节点的 Retained Heap = 该节点 + 所有被它支配的子节点的 Shallow Heap
- Retained Heap 越大，说明这个对象越「重要」

**实用技巧**：

1. 按 Retained Heap 排序，顶部就是内存占用的「大户」
2. 点击某个节点，可以看到它的引用链
3. 配合 Path to GC Roots，找出引用链

### 4.6 Path to GC Roots：追溯引用链

当你发现一个可疑对象后，用 Path to GC Roots 找出它为什么无法被回收。

**操作步骤**：

1. 在 Histogram 或 Dominator Tree 中选中对象
2. 右键 → 「Path to GC Roots」→「exclude weak/soft references」
3. 选择「complete」查看完整路径

**为什么排除 weak/soft references**：

| 引用类型 | 说明 | 是否阻止 GC |
|---|---|---|
| Strong（强引用） | 普通引用 `Object obj = new Object()` | 阻止 |
| Soft（软引用） | `SoftReference` | 内存不足时回收 |
| Weak（弱引用） | `WeakReference` | 随时回收 |
| Phantom（虚引用） | `PhantomReference` | 随时回收，不影响 finalize |

排查内存泄漏时，我们关心的是**强引用链**。排除 weak/soft references 后，剩下的就是阻止对象被回收的强引用。

### 4.8 OQL Console：SQL-like 查询

MAT 提供了完整的 OQL 支持，让你用 SQL 语法查询 Dump。

**打开 OQL Console**：菜单 → 「OQL Console」

```sql
-- 查询所有 HashMap
SELECT * FROM java.util.HashMap

-- 查询超过 1000 个元素的 Map
SELECT * FROM java.util.HashMap WHERE size > 1000

-- 查询占用内存超过 1MB 的 String
SELECT * FROM java.lang.String WHERE objectsize > 1048576

-- 查询某个包下的所有类
SELECT * FROM "com\\.example\\..*" WHERE objectsize > 10240

-- 查询包含特定字符串的对象
SELECT * FROM java.lang.String WHERE toString().indexOf("admin") >= 0

-- 查询所有 ThreadLocal
SELECT * FROM java.lang.ThreadLocal
```

**OQL 内置函数**：

| 函数 | 说明 | 示例 |
|---|---|---|
| `objectsize` | 对象浅堆大小 | `WHERE objectsize > 10000` |
| `classof()` | 返回对象类型 | `WHERE classof(o).name LIKE '.*User'` |
| `toHtml()` | HTML 格式化 | `SELECT toHtml(o) FROM ...` |
| `reachables()` | 可达对象 | `SELECT reachables(o) FROM ...` |
| `referrers()` | 引用对象 | `SELECT referrers(o) FROM ...` |

## 六、NMT 本地内存追踪

### 6.1 NMT 简介

NMT（Native Memory Tracking）是 JVM 提供的本地内存监控工具，用于追踪 JVM 内部和直接内存的使用。

```bash
# 启动 JVM 时启用 NMT
-XX:NativeMemoryTracking=summary
# 或详细模式
-XX:NativeMemoryTracking=detail
```

### 6.2 查看 NMT 统计

```bash
# 查看当前进程的 NMT 统计
jcmd <PID> VM.native_memory summary

# 输出示例
Native Memory Tracking:
Total: reserved=5242880KB, committed=5242880KB
-                 Java Heap (reserved=4194304KB, committed=4194304KB)
                      #Java Heap 是堆内存

-                     Class (reserved=262144KB, committed=262144KB)
                      #Class 是类元数据

-                     Thread (reserved=65536KB, committed=65536KB)
                      #Thread 是线程栈

-                     Code (reserved=131072KB, committed=131072KB)
                      #Code 是 JIT CodeCache

-                     GC (reserved=262144KB, committed=262144KB)
                      #GC 是 GC 相关内存

-                 Internal (reserved=32768KB, committed=32768KB)
                      #Internal 是其他本地内存

-                    Direct (reserved=2097152KB, committed=2097152KB)
                      #Direct 是直接内存
```

### 6.3 NMT 详解

```bash
# 对比基线（先记录基线）
jcmd <PID> VM.native_memory baseline

# 一段时间后，对比基线
jcmd <PID> VM.native_memory summary.diff
```

### 6.4 NMT 的局限

- NMT 追踪的是 JVM 内部的内存分配，不包括 native 代码的分配
- 有一定的性能开销（约 5%）
- 需要在 JVM 启动时启用

## 七、工具配合使用实战

### 7.1 场景一：OOM 排查

**目标**：定位 OOM 原因

**工具配合**：

1. **首先**，确保配置了 `-XX:+HeapDumpOnOutOfMemoryError`
2. **OOM 发生后**，用 `jstat -gcutil` 快速确认 GC 情况
3. **然后**，用 MAT 分析 Heap Dump
   - Leak Suspects 找嫌疑人
   - Dominator Tree 看内存大户
   - Path to GC Roots 追溯引用链
4. **最后**，定位到具体代码

### 7.2 场景二：内存持续增长排查

**目标**：找出内存泄漏点

**工具配合**：

1. **定期**用 `jstat -gcutil` 观察内存趋势
2. **当发现增长**时，用 `jmap -histo` 快速查看对象数量变化
3. **问题严重时**，生成 Heap Dump
4. **用 MAT** 对比两个时间点的 Dump
   - 「Compare Heap Objects」
   - 找出数量增长最多的对象

### 7.3 场景三：直接内存泄漏排查

**目标**：找出直接内存泄漏点

**工具配合**：

1. **启用 NMT**
   ```bash
   jcmd <PID> VM.native_memory summary
   ```
2. **关注 Direct 行**的使用量是否持续增长
3. **如果使用 Netty**，启用泄漏检测
   ```bash
   -Dio.netty.leakDetection.level=advanced
   ```
4. **生成 Heap Dump**，搜索 `java.nio.DirectByteBuffer`

## 八、面试追问

### 8.1 问题一：jmap 和 jcmd 的区别？

**参考答案**：

| 维度 | jmap | jcmd |
|---|---|---|
| 功能 | 专门用于堆内存相关操作 | 多功能诊断工具 |
| 推荐度 | 基本可用（部分功能已废弃） | JDK 7+ 推荐 |
| 替代 | 部分功能被 jcmd 替代 | 推荐使用 |

`jcmd` 是 JDK 7+ 推荐的诊断命令，可以替代 `jmap`、`jstack`、`jinfo` 等：

```bash
# jcmd 的常用功能
jcmd <PID> GC.heap_dump /tmp/heap.hprof     # 替代 jmap -dump
jcmd <PID> GC.class_histogram               # 替代 jmap -histo
jcmd <PID> Thread.print                      # 替代 jstack
jcmd <PID> VM.flags                          # 替代 jinfo
```

### 8.2 问题二：MAT 和 jhat 哪个好？

**参考答案**：

**MAT 更好**。原因：

| 维度 | MAT | jhat |
|---|---|---|
| 大文件支持 | 支持 GB 级 | 仅支持 < 2GB |
| 分析功能 | 深堆计算、GC Roots 追溯 | 基础功能 |
| 性能 | 快 | 慢 |
| OQL 支持 | 完整 | 有限 |
| 可视化 | 优秀 | 简陋 |
| 维护状态 | 活跃 | JDK 11 移除 |

生产环境 Dump 通常很大（2GB+），只能用 MAT。

### 8.3 问题三：生产环境如何安全地获取 Dump？

**参考答案**：

生产环境 Dump 需要特别注意：

1. **提前配置自动 Dump**：
   ```bash
   -XX:+HeapDumpOnOutOfMemoryError
   -XX:HeapDumpPath=/data/logs/heap.hprof
   ```
2. **如果需要手动 Dump**：
   - 优先使用 `jcmd`（比 jmap 更安全）
   - 避免使用 `-F` 强制选项
   - 避开业务高峰期
3. **Dump 文件大**：
   - 估算：Dump 大小约为堆大小的 0.5 - 1 倍
   - 确保磁盘空间充足
   - 可以用 `jmap -dump:live` 只 dump 存活对象

## 总结

JVM 内存分析工具是排查内存问题的利器，每种工具都有其适用场景。

关键要点：

- **命令行工具**：`jmap` 生成 Dump、`jstat` 实时监控
- **可视化分析**：MAT 是 Heap Dump 分析的首选，功能强大
- **工具配合**：监控 + Dump + 分析，三步走
- **NMT**：追踪本地内存，排查直接内存泄漏
- **安全第一**：生产环境优先使用自动 Dump，避免手动操作

记住：工具只是手段，关键是分析思路。掌握了工具的使用方法，还要理解内存泄漏的本质，才能快速定位问题。
