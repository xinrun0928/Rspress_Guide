# 堆内存溢出排查：一次生产 OOM 的完整诊断之旅

凌晨两点，你被电话叫醒：「线上服务崩了，OOM 了！」你揉着眼睛打开告警平台，看到一条触目惊心的错误日志：

```
java.lang.OutOfMemoryError: Java heap space
```

这不是演习。这是每一个 Java 工程师迟早会面对的生产事故。

OOM（OutOfMemoryError）是生产环境的头号杀手之一。它来无影去无踪——代码在测试环境跑了一周没问题，上了生产三天就开始 OOM。你开始怀疑人生：到底是内存泄漏，还是内存真的不够用？是 JVM 参数设错了，还是代码里有隐藏的坑？

别慌。这篇文章，就是你的 OOM 排查指南。我会从 OOM 的本质讲起，带你一步步还原生产环境的完整排查流程，最后再给出代码级的修复方案。读完这篇，你不仅能解决当前的问题，还能举一反三，应对未来的各种内存问题。

## 一、OOM 是什么——从「装不下」到「抛异常」

### 1.1 OOM 的本质：JVM 分配失败

要理解 OOM，先要理解 JVM 的内存分配机制。

当你在 Java 代码里写 `new Object()` 时，JVM 会在堆内存中找一个足够大的连续空间存放这个对象。如果找到了，对象创建成功；如果找不到，说明堆内存不足了，此时 JVM 会尝试触发一次 Full GC，试图回收一些不再使用的对象。

问题来了：如果 Full GC 之后，堆内存仍然不足以分配新对象，JVM 就彻底没辙了。它会在抛出 `OutOfMemoryError` 之前，做最后一次尝试——检查是否要退出。如果启用了 `-XX:+ExitOnOutOfMemoryError`，JVM 会直接退出进程；否则，它会抛出这个异常，把「球」抛给应用程序。

```java
public class OOMDemo {
    public static void main(String[] args) {
        // 无限创建对象，直到堆内存耗尽
        List<byte[]> list = new ArrayList<>();
        while (true) {
            // 每次分配 1MB
            list.add(new byte[1024 * 1024]);
        }
    }
}
```

运行这个程序，很快就会看到：

```
Exception in thread "main" java.lang.OutOfMemoryError: Java heap space
```

这就是最典型的 Heap Space OOM。

### 1.2 堆溢出 vs 非堆溢出：两种 OOM，两种病因

很多同学以为 OOM 就是堆内存不够用。实际上，JVM 的内存区域有很多，OOM 可能发生在不同的地方。

| 内存区域 | OOM 类型 | 常见原因 |
|---|---|---|
| 堆内存（Heap Space） | `OutOfMemoryError: Java heap space` | 内存泄漏、一次性加载过多数据、堆设置过小 |
| 元空间（Metaspace） | `OutOfMemoryError: Metaspace` | 动态类生成、CGLIB 增强、JSP 编译 |
| 直接内存（Direct Memory） | `OutOfMemoryError: Direct buffer memory` | NIO/Netty 堆外内存泄漏 |
| 虚拟机栈（Stack） | `OutOfMemoryError: unable to create new native thread` | 线程创建过多 |
| 本地方法栈 | `StackOverflowError` | 递归调用过深 |

这篇文章聚焦于 **Heap Space OOM**——这是最常见、也是面试最爱问的 OOM 类型。

## 二、Heap Space OOM 的四大元凶

在动手排查之前，先了解 OOM 的常见原因，能让你事半功倍。Heap Space OOM 的原因可以归结为四大类：

### 2.1 内存泄漏：对象无法被回收

这是 OOM 最常见的原因。

正常情况下，一个对象不再被引用时，GC 会将其回收，内存会被释放。但如果有代码错误导致对象被错误地持有引用，这个对象就无法被 GC 回收。随着时间推移，存活对象越来越多，堆内存逐渐耗尽。

常见的内存泄漏场景包括：

- **缓存没有清理机制**：`Map` 不断 put 数据，却从未 remove
- **静态集合类持有大量对象**：static 修饰的集合是 GC 的「盲区」
- **监听器和回调未移除**：注册了事件监听，却忘了在不需要时取消注册
- **ThreadLocal 未清理**：线程池复用时，ThreadLocal 中的数据不会被自动清理
- **资源未关闭**：Stream、Connection、Channel 用完不 close

### 2.2 内存溢出：确实需要很多内存

有时候，代码没有内存泄漏，但确实需要大量内存。

典型场景：

- **一次性加载太多数据**：比如一次性从数据库加载全表数据到内存
- **处理大文件**：把一个大文件的内容全部读入内存
- **生成大对象**：比如生成一个很大的 List 或 Map

这种情况的本质是：**业务需求本身就是内存密集型的**，不是代码 bug，而是架构设计问题。

### 2.3 JVM 堆设置过小：巧妇难为无米之炊

`-Xmx` 设置的值太小，应用程序的正常内存需求都满足不了。

常见错误：

- 测试环境 2GB 堆，生产环境也是 2GB，但生产数据量是测试的 10 倍
- 容器环境没有正确配置内存限制，导致 JVM 堆被人为压缩
- 对内存消耗预估不足，上线后才发现不够

### 2.4 瞬间大量对象创建：流量洪峰

高并发场景下，多个请求同时处理，每个请求都创建大量对象。如果对象创建速度远超 GC 回收速度，堆内存会快速耗尽。

典型场景：

- 秒杀活动，瞬间涌入大量请求
- 数据导入/导出，批量处理时并发过高
- 接口没有限流，请求量失控

## 三、生产环境排查标准流程

接下来，是本文的核心内容——如何在生产环境中快速定位和解决 OOM 问题。

### 3.1 第一步：监控告警——让问题早发现

真正的 OOM 排查，应该从告警开始，而不是从 OOM 发生开始。

在 OOM 发生之前，堆内存使用率会持续攀升。如果你的监控系统能捕获这个趋势，就能在 OOM 发生之前介入，而不是等到服务崩溃才手忙脚乱。

生产环境的 JVM 监控告警配置建议：

| 指标 | 告警阈值 | 说明 |
|---|---|---|
| 堆内存使用率 | `>= 80%` | 轻度告警，持续观察 |
| 堆内存使用率 | `>= 90%` | 严重告警，立即处理 |
| Full GC 频率 | `>= 1次/分钟` | 说明堆内存压力很大 |
| Full GC 耗时 | `>= 1秒` | 说明回收效率低 |

建议使用 G1 收集器，并配置 `-XX:InitiatingHeapOccupancyPercent=45`，让 JVM 在堆内存使用率达到 45% 时就开始并发标记，提前发现问题。

### 3.2 第二步：获取 Heap Dump——保存现场

OOM 发生时，第一件事是保存堆转储（Heap Dump）。这是后续分析的基础。

```bash
# 方式一：正常生成（推荐，有 pid）
jmap -dump:format=b,file=/tmp/heap.hprof <PID>

# 方式二：强制生成（进程卡住时使用，可能造成进程挂起）
jmap -F -dump:format=b,file=/tmp/heap.hprof <PID>

# 方式三：设置自动生成（OOM 时自动保存）
-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/data/logs/heap.hprof
```

:::warning 生产环境慎用 jmap -F

`-F` 参数会强制执行，可能造成进程短暂挂起。在高并发生产环境中，使用前务必评估影响。如果服务配置了 `-XX:+HeapDumpOnOutOfMemoryError`，让 JVM 在 OOM 时自动生成 dump 是更安全的选择。

:::

保存好 Heap Dump 后，你的排查工作就成功了一半。接下来，就是分析这个 dump 文件。

### 3.3 第三步：分析 Heap Dump——三板斧

分析 Heap Dump 的工具有很多，但核心分析方法就三个：

#### 第一板斧：Leak Suspects——自动定位泄漏嫌疑人

MAT（Memory Analyzer Tool）提供了自动分析功能，会列出最可能导致内存泄漏的「嫌疑人」。

使用步骤：

1. 下载 MAT：https://eclipse.dev/mat/downloads.php
2. 打开 Heap Dump 文件
3. 点击「Leak Suspects Report」

MAT 会告诉你：「这个 `HashMap` 占用了 XX% 的内存，而且它一直在增长。」这就是你排查的方向。

#### 第二板斧：Histogram——按类统计对象数量

Histogram 视图会列出所有类的对象数量和内存占用，按「Retained Size」排序。

```bash
# 用 jmap 也可以快速查看
jmap -histo <PID> | head -50
```

输出示例：

```
 num     #instances         #bytes  class name
----------------------------------------------
   1:          12345      98765440  [Ljava.lang.Object;
   2:           8901      45678912  com.example.service.UserService
   3:           6789      34567890  java.util.HashMap$Node
```

重点关注：

- 对象数量异常多的类
- 单个对象占用内存特别大的类
- 自己的业务代码中的类（而非 JDK 内部的类）

#### 第三板斧：Dominator Tree——找出内存占用的根源

Dominator Tree 展示了对象之间的引用关系，帮助你找出「谁持有了这些对象，导致它们无法被回收」。

在 MAT 中打开 Dominator Tree，顶部就是占用内存最多的对象。点击任何一个节点，可以看到它的 Retained Heap（这个对象被回收时，能释放多少内存）。

### 3.4 第四步：分析 GC Roots——找到无法回收的根因

找到了占用内存大的对象后，下一步是找出这些对象为什么无法被回收。

GC Roots 是垃圾收集器的根引用。从 GC Roots 出发，沿着引用链往下走，能到达的对象都是「存活」的；无法到达的对象，才会被回收。

在 MAT 中，选择「Path to GC Roots」→「exclude weak/soft references」，可以看到对象的完整引用链。

重点关注：

- **静态变量持有引用**：static 修饰的集合是最常见的泄漏源
- **ThreadLocal 持有引用**：线程池复用时的问题
- **本地方法栈持有引用**：JNI 调用的遗留问题
- **监听器持有引用**：事件监听器没有取消注册

### 3.5 第五步：代码级修复——对症下药

找到泄漏点后，下一步就是修复代码。以下是常见内存泄漏的修复方案。

#### 场景一：缓存未清理

```java
// 错误写法：缓存永不过期
private static Map<String, Object> cache = new HashMap<>();

public void addToCache(String key, Object value) {
    cache.put(key, value); // 永远不清理
}
```

```java
// 正确写法一：使用 WeakHashMap（适合缓存可以被 GC 回收的场景）
private static Map<String, Object> cache = new WeakHashMap<>();

// 正确写法二：使用带过期时间的缓存
private static LoadingCache<String, Object> cache = Caffeine.newBuilder()
    .expireAfterWrite(10, TimeUnit.MINUTES)
    .maximumSize(1000)
    .build(key -> loadFromDB(key));
```

#### 场景二：ThreadLocal 未清理

```java
// 错误写法：线程池复用时，ThreadLocal 数据不会被清理
private static ThreadLocal<User> currentUser = new ThreadLocal<>();

public void handleRequest(User user) {
    currentUser.set(user);
    // 处理请求...
    // 忘记清理
}
```

```java
// 正确写法：使用后务必清理
private static ThreadLocal<User> currentUser = new ThreadLocal<>();

public void handleRequest(User user) {
    try {
        currentUser.set(user);
        // 处理请求...
    } finally {
        currentUser.remove(); // 务必清理
    }
}
```

#### 场景三：资源未关闭

```java
// 错误写法：Stream 未关闭
public void processFile(String path) {
    Files.lines(Paths.get(path))
        .filter(line -> line.length() > 10)
        .forEach(System.out::println);
    // Stream 没有关闭，底层文件描述符可能泄漏
}
```

```java
// 正确写法：使用 try-with-resources
public void processFile(String path) {
    try (Stream<String> lines = Files.lines(Paths.get(path))) {
        lines.filter(line -> line.length() > 10)
             .forEach(System.out::println);
    } // 自动关闭
}
```

## 四、常用排查命令速查

在生产环境中，时间就是生命。以下命令需要烂熟于心。

### 4.1 查看堆使用概况

```bash
jmap -heap <PID>
```

输出示例：

```
Heap Configuration:
   MaxHeapSize              = 4294967296 (4096.0MB)
   NewSize                  = 1073741824 (1024.0MB)
   OldSize                  = 3221225472 (3072.0MB)

Heap Usage:
   Eden Space: _capacity=855638016, used=789234000 (92.2%)
   From Space: _capacity=107374182, used=45678900 (42.5%)
   To Space:   capacity=107374182, used=0 (0.0%)
   Old Gen:    capacity=3221225472, used=1987654321 (61.7%)
```

重点看 **Eden Space** 的使用率。如果 Eden 区使用率长期超过 80%，说明对象创建速度太快。

### 4.2 查看对象统计

```bash
jmap -histo <PID> | head -30
```

### 4.3 监控 GC 情况

```bash
# 每秒输出一次 GC 统计
jstat -gcutil <PID> 1000

# 输出示例
  S0     S1     E      O      M     YGC     YGCT    FGC    FGCT     GCT
 42.35   0.00  85.23  61.73  97.45    123   1.234   5     2.345    3.579
```

参数说明：

- `S0/S1`：Survivor 区使用率
- `E`：Eden 区使用率
- `O`：Old 区使用率
- `M`：Metaspace 使用率
- `YGC/YGCT`：年轻代 GC 次数和耗时
- `FGC/FGCT`：Full GC 次数和耗时
- `GCT`：总 GC 耗时

如果 `FGC` 持续增长，而 `FGCT` 也在增加，说明 Old 区压力很大，可能存在内存泄漏或分配不合理。

### 4.4 生成堆转储

```bash
jmap -dump:format=b,file=/tmp/heap.hprof <PID>
```

## 五、面试追问：OOM 相关高频问题

### 5.1 问题一：如何避免 OOM？

**参考答案**：

OOM 预防需要从多个层面入手：

**架构层面**：

- 对大数据量场景，使用分页、游标、批量处理，避免一次性加载全部数据
- 合理设计缓存策略，使用带过期时间和大小限制的缓存（如 Caffeine、Guava Cache）
- 对内存密集型操作，设置合理的超时和熔断机制

**代码层面**：

- 所有资源（Stream、Connection、Channel）必须使用 try-with-resources 或 finally 关闭
- ThreadLocal 使用后务必 remove
- 避免在循环中创建大量对象，尽量复用对象
- 监听器和回调要在不再需要时取消注册

**JVM 层面**：

- 合理设置 `-Xmx` 和 `-Xms`，建议两者相等避免运行时堆伸缩
- 生产环境务必设置 `-XX:MetaspaceSize` 和 `-XX:MaxMetaspaceSize`
- 开启 `-XX:+HeapDumpOnOutOfMemoryError`，便于事后分析
- 配合成熟的监控系统，提前发现内存异常

### 5.2 问题二：内存泄漏和内存溢出的区别？

**参考答案**：

| 维度 | 内存泄漏 | 内存溢出 |
|---|---|---|
| 定义 | 对象不再使用，但因错误引用无法被 GC 回收 | 需要的内存超过 JVM 能提供的内存 |
| 特点 | 缓慢发生，随时间累积 | 突然发生，一次性需求过大 |
| 原因 | 代码 bug，对象被错误持有 | 业务需求、JVM 参数不当 |
| 表现 | OOM 前通常 GC 越来越频繁 | 直接 OOM |
| 解决 | 修复代码，移除错误引用 | 增加堆内存、优化内存使用 |

**两者的关系**：内存泄漏是内存溢出的一种原因。泄漏导致对象无法回收，最终堆内存耗尽，触发 OOM。

### 5.3 问题三：为什么 OOM 之前不 Full GC？

**参考答案**：

这是一个常见的误解。OOM 发生前，JVM **一定会尝试 Full GC**。

OOM 的触发流程是：

1. 新对象分配，发现 Eden/Old 区没有足够空间
2. 触发 Minor GC，回收年轻代
3. Minor GC 后仍然不够，尝试晋升到 Old 区
4. Old 区空间不足，触发 Full GC（也叫 Major GC）
5. Full GC 后，仍然无法满足内存分配
6. 抛出 `OutOfMemoryError: Java heap space`

所以，OOM 之前往往伴随着频繁的 Full GC。如果你在 GC 日志中看到 Full GC 次数不断增加，而 Old 区使用率持续居高不下，这就是典型的内存泄漏信号。

:::tip 延伸思考

Full GC 后 Old 区使用率下降不明显，说明有大量对象无法回收。这些对象就是泄漏的「嫌疑人」。

:::

### 5.4 问题四：生产环境 Full GC 频繁，如何排查？

**参考答案**：

Full GC 频繁是内存问题的典型表现，排查步骤如下：

**第一步：确认 Full GC 的类型和频率**

```bash
jstat -gcutil <PID> 1000
```

观察 `FGC` 是否持续增长。

**第二步：分析 GC 日志**

```bash
# 查看 GC 日志
cat gc.log | grep Full
```

重点关注：

- Full GC 的原因：Allocation Failure（分配失败）还是 System.gc()（显式调用）？
- Full GC 后 Old 区使用率：如果下降很少，说明有大量对象无法回收
- Full GC 耗时：如果很长，说明 Old 区太大或回收效率低

**第三步：生成 Heap Dump 分析**

```bash
jmap -dump:format=b,file=/tmp/heap.hprof <PID>
```

用 MAT 分析 Dominator Tree，找出占用内存最多的对象。

**第四步：定位代码**

找出泄漏对象后，追溯 GC Roots，找出错误持有引用的代码位置。常见的泄漏源包括：

- 静态集合类未清理
- ThreadLocal 未 remove
- 监听器/回调未取消注册
- 数据库连接、Stream 未关闭

## 六、留给你的思考题

### 题目一：慢增长 vs 快增长

假设你的服务运行了 3 天后开始 OOM，而同事的服务运行了 3 周才 OOM。两者的 Heap Dump 分析都发现有一个 `HashMap` 占用了 80% 的堆内存。

**问题**：你能从这个现象推断出两者的代码问题有什么区别吗？

**提示**：考虑对象的创建速度和泄漏速度。

### 题目二：Leak Suspects 之外的线索

MAT 的 Leak Suspects 报告指出了一个 `HashMap` 是内存泄漏的嫌疑人。但这个 `HashMap` 本身是合理的业务代码，它持有对象引用是正常的。

**问题**：如何进一步分析，找出真正持有错误引用的那一层？

**提示**：考虑 GC Roots 和引用链的完整性。

## 总结

Heap Space OOM 是生产环境最常见的内存问题，排查的核心在于「保存现场 → 定位泄漏 → 修复代码」。

关键要点回顾：

- **OOM 本质**：堆内存分配失败，Full GC 后仍无法满足
- **四大元凶**：内存泄漏、内存溢出、堆设置过小、流量洪峰
- **排查流程**：监控告警 → 获取 Dump → MAT 分析 → GC Roots → 代码修复
- **预防措施**：合理缓存、正确关闭资源、ThreadLocal 清理、监控告警

记住：OOM 不是终点，而是排查的起点。每一次 OOM 都是一次深入理解 JVM 内存管理的机会。
