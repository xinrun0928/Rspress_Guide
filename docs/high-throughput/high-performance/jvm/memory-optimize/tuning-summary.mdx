# JVM 调优经验总结：从理论到实战

JVM 调优是 Java 面试中最考察综合能力的知识点之一。它需要你理解 GC 原理、内存模型、参数体系，还要能在实际场景中灵活运用。

这篇文章，把 JVM 调优的所有知识点串联起来，帮你建立完整的调优知识体系。

## 一、JVM 调优的本质

很多人以为 JVM 调优就是调 GC 参数，这其实是一个误解。

JVM 调优的本质是：**在吞吐量、停顿时间、内存占用三者之间找到当前业务场景的最优平衡点**。

这三个目标之间存在天然的矛盾：

- **吞吐量优先**：意味着减少 GC 次数，但每次 GC 停顿时间可能更长
- **低停顿优先**：意味着频繁 GC，但每次停顿时间更短
- **低内存占用**：意味着频繁 GC 且停顿时间可能更长

没有绝对最优的方案，只有**最适合当前业务场景**的方案。

## 二、JVM 调优的三大原则

### 2.1 不要过早优化

很多人在代码还没写完、基准还没建立之前，就开始研究 GC 参数。这是典型的「过度优化」。

**正确做法**：先上线、监控、收集数据，找到真正的瓶颈，再针对性调优。

没有数据的调优是盲目的调优。

### 2.2 不要过度优化

「这个接口现在 100ms，我要优化到 10ms」——先问一个问题：用户需要 10ms 吗？

99 分到 100 分的代价，往往比 60 分到 80 分还要高。**满足业务需求 + 留有一定余量**才是正确的目标。

### 2.3 监控优先

没有监控的调优是盲目的调优。JVM 调优的每一步都要有数据支撑：

```
监控现状 → 建立基准 → 定位问题 → 调整参数 → 验证效果 → 持续迭代
```

## 三、GC 调优的完整流程

### 3.1 第一步：监控 GC 状态

通过 GC 日志和监控工具，收集当前的 GC 数据：

- GC 频率：Minor GC / 分钟，Full GC / 小时
- GC 耗时：Total GC Time / 测量时间 = GC 占比
- 停顿时间：P99、P999 停顿时间
- 吞吐量：1 - (GC Time / Total Time)

推荐工具：GCViewer、gceasy.io、JMC

### 3.2 第二步：分析 GC 日志

通过 GC 日志，分析以下问题：

- **GC 过于频繁**：Minor GC 每秒多次 → 可能需要增大年轻代
- **Full GC 频繁**：可能是对象晋升过快或内存泄漏
- **GC 停顿时间过长**：可能是 GC 算法选择不当或堆太大
- **内存持续增长**：可能是内存泄漏

### 3.3 第三步：确定调优目标

根据业务场景，确定调优目标：

| 业务场景 | 目标 | 推荐 GC |
|---|---|---|
| Web 服务（延迟敏感）| P99 < 200ms | G1 |
| 批处理（吞吐优先）| 吞吐量 > 95% | Parallel GC |
| 实时计算（极低延迟）| 停顿 < 1ms | ZGC |
| 微服务 API（均衡）| P99 < 500ms | G1 |

### 3.4 第四步：小步快跑

每次只改一个参数，改完后立即对比数据。不要一次改多个参数，否则无法判断是哪个参数起作用。

### 3.5 第五步：迭代优化

一次调优完成后，系统会有新的瓶颈浮现。继续监控、分析、调优，直到达到目标。

## 四、典型调优场景汇总

### 场景一：Minor GC 频繁

**表现**：Minor GC 每秒触发一次，每次 10-20ms

**原因**：
- Eden 区太小，对象稍多就满
- 对象分配率高，大量短生命周期对象

**解决**：
- 增大年轻代：-Xmn 或调大 NewRatio
- 增大 Survivor 区：-XX:SurvivorRatio=8
- 优化代码：减少短期对象创建

### 场景二：Full GC 频繁

**表现**：Full GC 每分钟触发一次，每次 500ms

**原因**：
- 老年代空间不足
- 对象晋升太快（Survivor 区太小）
- 内存泄漏

**解决**：
- 增大堆内存：-Xmx
- 增大 Survivor 区：-XX:SurvivorRatio=8
- 分析 Heap Dump，定位内存泄漏
- 切换到 G1：-XX:+UseG1GC

### 场景三：GC 停顿时间过长

**表现**：P999 停顿超过 500ms

**原因**：
- 堆太大（GC 时间与堆大小成正比）
- GC 算法选择不当
- GC 参数配置不合理

**解决**：
- 切换到低停顿 GC：G1 或 ZGC
- 设置目标停顿时间：-XX:MaxGCPauseMillis=200
- 减小堆大小（牺牲吞吐量换取停顿时间）

### 场景四：内存持续增长

**表现**：堆使用量持续上升，不下降

**原因**：内存泄漏

**解决**：
- Heap Dump 分析
- 使用 MAT 的 Leak Suspects 功能
- 定位泄漏对象和引用链

### 场景五：OOM 频繁

**表现**：OutOfMemoryError: Java heap space

**原因**：
- 堆设置过小
- 内存泄漏
- 一次性加载太多数据

**解决**：
- 增大堆：-Xmx
- 分析 Heap Dump，找内存泄漏
- 优化代码，减少内存占用

## 五、不同业务的调优策略

### 5.1 Web 服务（延迟敏感）

**推荐 GC**：G1

**目标**：P99 < 200ms，吞吐量 > 90%

**典型配置**：

```bash
-Xms4g -Xmx4g
-XX:+UseG1GC
-XX:MaxGCPauseMillis=200
-XX:G1HeapRegionSize=4M
-XX:InitiatingHeapOccupancyPercent=45
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/data/logs/
```

### 5.2 批处理/大数据（吞吐优先）

**推荐 GC**：Parallel GC

**目标**：吞吐量 > 95%

**典型配置**：

```bash
-Xms8g -Xmx8g
-XX:+UseParallelGC
-XX:+UseParallelOldGC
-XX:ParallelGCThreads=8
-XX:+PrintGCDetails
```

### 5.3 实时计算/游戏（极低延迟）

**推荐 GC**：ZGC

**目标**：停顿 < 1ms

**典型配置**：

```bash
-Xms16g -Xmx16g
-XX:+UseZGC
-XX:+UnlockExperimentalVMOptions
```

### 5.4 高并发 API（均衡）

**推荐 GC**：G1

**目标**：P99 < 500ms，吞吐量 > 95%

**典型配置**：

```bash
-Xms4g -Xmx4g
-XX:+UseG1GC
-XX:MaxGCPauseMillis=200
-XX:G1HeapRegionSize=2M
-XX:+PrintGCDetails
-XX:+PrintGCDateStamps
```

## 六、常见 JVM 调优坑

### 坑一：-Xms 和 -Xmx 设置不同

```bash
# 错误：运行时堆伸缩会导致 Full GC
-Xms2g -Xmx8g

# 正确：固定堆大小，避免运行时堆伸缩
-Xms4g -Xmx4g
```

堆伸缩时，JVM 会触发 Full GC 来调整堆大小，这会导致长时间的 STW 停顿。

### 坑二：MetaspaceSize 设置过小

```bash
# 错误：MetaspaceSize 过小导致频繁 Full GC
-XX:MetaspaceSize=64m -XX:MaxMetaspaceSize=128m

# 正确：合理设置，并设置上限
-XX:MetaspaceSize=256m -XX:MaxMetaspaceSize=512m
```

### 坑三：直接内存超过限制

NIO 直接内存不受堆大小限制，但如果超过物理内存，可能被 OOM Killer 杀死：

```bash
# 合理设置直接内存上限
-XX:MaxDirectMemorySize=4g
```

### 坑四：GC 日志参数错误

```bash
# 错误：参数写错了，日志没有输出
-XX:+PrintGCDtailis -XX:+PrintGCTimeStamps

# 正确：使用 JDK 9+ 统一日志框架
-Xlog:gc*:file=gc.log:time,uptime,level,tags:filecount=5,filesize=10M
```

### 坑五：容器环境下参数未适配

容器环境下，JVM 可能没有正确感知容器内存限制：

```bash
# 正确：JDK 10+ 使用容器感知
-XX:+UseContainerSupport
-XX:MaxRAMPercentage=75.0
```

## 七、JVM 调优参数速查表

| 场景 | 核心参数 | 辅助参数 |
|---|---|---|
| 降低停顿 | -XX:MaxGCPauseMillis=200 | -XX:+UseG1GC |
| 提升吞吐 | -XX:+UseParallelGC | -XX:ParallelGCThreads |
| 减少内存占用 | -XX:MaxRAMPercentage | -XX:MetaspaceSize |
| OOM 诊断 | -XX:+HeapDumpOnOutOfMemoryError | -XX:HeapDumpPath |
| GC 监控 | -Xlog:gc*:file=gc.log | GCViewer/gceasy.io |

## 八、生产环境调优 Checklist

上线前检查：

- [ ] -Xms 和 -Xmx 设置相同（固定堆大小）
- [ ] -XX:MaxMetaspaceSize 设置上限
- [ ] -XX:MaxDirectMemorySize 设置上限（NIO 应用）
- [ ] -XX:+HeapDumpOnOutOfMemoryError 开启
- [ ] GC 日志参数正确配置
- [ ] GC 日志轮转配置（避免日志撑爆磁盘）
- [ ] 容器环境下使用 UseContainerSupport

上线后监控：

- [ ] GC 频率和耗时
- [ ] 堆内存使用情况
- [ ] 元空间使用情况
- [ ] 直接内存使用情况（NMT）
- [ ] OOM 事件次数

---

## 总结

JVM 调优是一个持续迭代的过程，不是「一劳永逸」的事情。

**核心记忆点**：
- 调优前先建立基准，没有数据就是盲调
- 不要过早优化，不要过度优化
- G1 是通用场景的首选，ZGC 是超低延迟的首选
- 堆大小固定是生产环境的基本要求
- 监控和调优是分不开的

理解了这套框架，你就有了回答所有 JVM 相关面试题的底气。
