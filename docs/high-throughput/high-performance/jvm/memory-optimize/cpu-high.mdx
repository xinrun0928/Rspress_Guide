# CPU 100% 排查：从现场保留到根因定位

凌晨三点，你正准备睡觉，手机突然响了。监控告警：线上服务 CPU 使用率 100%，接口响应时间从 200ms 飙升到 30 秒，用户开始大量投诉。

你只有十分钟。十分钟后如果还没定位到问题，PG 就会启动预案——重启服务。但重启意味着丢失现场，那个导致 CPU 100% 的根因可能再也找不到了。

这不是演习。这是一次真实的线上故障排查。

**如何在十分钟内定位 CPU 100% 的根因？** 这个问题，几乎是所有 Java 面试中必然会问到的高频考点。但很多人只知道 `top` 和 `jstack`，遇到复杂场景就傻眼了。

今天，我们来系统性地解决这个问题。

## 一、CPU 高占用的常见原因

在动手排查之前，我们需要先了解有哪些"嫌疑人"。CPU 100% 不是无缘无故的，它背后一定有代码在疯狂执行。

### 1.1 死循环与死锁

**死循环**是最直接的 CPU 杀手。当代码进入一个无法退出的循环时，CPU 会在这个循环里空转，直到时间片耗尽。

常见场景：

```java
// 场景一：循环条件写错
while (count != -1) {  // count 永远不会变成 -1
    process(data);
}

// 场景二：正则表达式回溯（稍后详解）
Pattern.matches("(a+)+b", longString);

// 场景三：分布式锁死等
while (!redisLock.tryLock()) {
    // 没有超时，没有退避，直接死等
}
```

**死锁**和死循环不同。死锁是多个线程互相持有对方需要的锁，谁都无法继续执行。但死锁的 CPU 表现通常是：**部分线程忙等待，CPU 不会到 100%**。

真正的死锁排查用 `jstack -l <PID>` 查看 `Waiting for monitor entry` 和 `<deadlock>` 关键字。

### 1.2 频繁 GC

GC 是 Stop-The-World 的。如果 Minor GC 频繁发生，每次 GC 都会让用户线程暂停；如果 Full GC 频繁发生，CPU 会飙升。

典型场景：

- **年轻代太小**：对象存活时间短，但年轻代空间不足，导致频繁 Minor GC
- **大对象直接进入老年代**：一次性分配大数组，直接进入老年代，触发 Full GC
- **内存泄漏**：对象无法回收，堆内存持续增长，GC 越来越频繁

```bash
# 查看 GC 情况
jstat -gcutil <PID> 1000
```

### 1.3 正则表达式回溯

这是最容易忽略的 CPU 杀手。看似普通的正则表达式，在处理特定输入时，可能导致**指数级回溯**，CPU 直接打满。

#### 1.3.1 回溯原理

正则引擎通常采用**回溯算法**（NFA，非确定有限自动机）。当匹配失败时，它会回退到之前的状态，尝试其他路径。

```java
// 危险示例：匹配以 b 结尾的 a 串
String regex = "(a+)+b";
String input = "aaaaaaaaaaaaaaaaaaaaaaaaac";  // 没有 b
```

这个正则的工作过程：

1. `(a+)+` 贪婪匹配所有 `a`
2. `+` 要求至少一个 `a`，但后面是 `b`
3. 匹配失败，回溯
4. 尝试让内层 `a+` 少匹配一个 `a`
5. 外层 `+` 继续尝试匹配
6. ……

每减少一个 `a`，外层都要重新尝试，复杂度是 `O(2^n)`。

对于 20 个 `a`，回溯次数 = 2^20 ≈ 100 万次。

#### 1.3.2 常见危险模式

| 模式 | 危险原因 |
|---|---|
| `(a+)+b` | 嵌套量词 |
| `(a*)*b` | 同上 |
| `(a|a)+b` | 选择 + 量词 |
| `.*.*.*` | 多个贪婪匹配 |
| `(.*)+` | 任意字符 + 嵌套 |

#### 1.3.3 解决方案

**方案一：使用原子组和非贪婪**

```java
// 使用 possessive quantifier（原子组）阻止回溯
// Java 不直接支持，使用非贪婪 + 固化分组
String regex = "(?>a+)+b";  // 原子组，不支持 Java
```

**方案二：优化正则结构**

```java
// 将 "一个或多个 a 后面跟 b" 改为 "a...ab"
String regex = "a+b";

// 如果必须保留原意，用完整匹配替代
if (Pattern.matches("a+b", input)) {
    // ...
}
```

**方案三：使用 Hypersistence Utils 的正则优化库**

```java
// 引入优化库
// io.hypersistence:hypersistence-utils

// 或使用 Google Guava 的正则
Splitter.on(Pattern.compile("a+b"))
        .splitToList(input);
```

### 1.4 加密与解密计算

加解密是 CPU 密集型操作。如果你的服务频繁进行加密（如 MD5、SHA、RSA），CPU 可能会被打满。

典型场景：

- **接口签名验证**：每个请求都要计算签名
- **密码加密存储**：用户注册时 bcrypt 加密
- **数据传输加密**：TLS/SSL 握手

```java
// bcrypt 加密很慢，但很安全
BCryptPasswordEncoder encoder = new BCryptPasswordEncoder();
String hashed = encoder.encode(rawPassword);  // 这会消耗 CPU
```

### 1.5 序列化与反序列化

JSON 库（如 Jackson、Gson）在处理复杂对象或大 JSON 时，会消耗大量 CPU。

```java
// 反序列化大 JSON
ObjectMapper mapper = new ObjectMapper();
User user = mapper.readValue(jsonString, User.class);  // CPU 密集
```

### 1.6 JIT 编译（启动初期）

JVM 在启动时会对热点代码进行编译优化。编译过程本身会消耗 CPU，导致启动初期 CPU 较高。

这是**正常现象**，不是故障。预热后再压测即可。

### 1.7 HashMap 碰撞攻击

攻击者构造大量哈希值相同的 key，导致 HashMap 退化为链表，查询复杂度从 `O(1)` 退化为 `O(n)`。

```java
// 正常情况
HashMap<String, Object> map = new HashMap<>();
map.put("normalKey", value);  // O(1)

// 碰撞攻击
map.put("sameHashCodeKey1", value);
map.put("sameHashCodeKey2", value);  // 退化为链表
```

详见后文"HashMap 碰撞攻击"章节。

## 二、十分钟定位法：标准排查流程

回到凌晨三点的场景。你有十分钟，CPU 100%，服务快撑不住了。**如何在十分钟内定位根因？**

### 2.1 第一分钟：保留现场

**千万不能重启！** 重启之后，现场就没了。

```bash
# 1. 保存进程 ID 和 CPU 使用情况
top -b -n 1 -p <PID> > /tmp/top.log

# 2. 保存线程 CPU 使用情况
top -H -b -n 1 -p <PID> > /tmp/top-thread.log

# 3. 保存进程内存使用
pmap -x <PID> > /tmp/pmap.log
```

### 2.2 第二分钟：找出高 CPU 线程

```bash
# 查看所有线程的 CPU 使用情况（-H 显示线程）
top -H -p <PID>

# 或使用 ps 命令
ps -eLo pid,lwp,nlwp,comm,pcpu --sort=-pcpu | head -20
```

你会看到这样的输出：

```
  PID  LWP NLWP COMMAND         %CPU
12345 12345 50 java             98.5
12346 12346 50 java             35.2
12347 12347 50 java             12.3
```

找到 `%CPU` 最高的线程，记录下它的 **LWP**（Light Weight Process，即线程 ID）。

### 2.3 第三分钟：转换线程 ID 格式

JVM 的线程 ID 是十进制，但 `jstack` 输出的是十六进制。

```bash
# 假设高 CPU 线程的 LWP 是 12345
printf '%x\n' 12345
# 输出：3039
```

这就是我们要找的十六进制线程 ID。

### 2.4 第四分钟：导出线程栈

```bash
# 导出线程栈到文件
jstack <PID> > /tmp/jstack.log

# 直接查找指定线程的堆栈
jstack <PID> | grep -A 30 3039
```

`jstack` 输出中，找到对应的线程：

```
"pool-1-thread-10" #45 prio=5 os_prio=0 tid=0x00007f8a4c01d000 nid=0x3039 runnable [0x00007f8a5c3d4000]
   java.lang.Thread.State: RUNNABLE
    at com.example.service.OrderService.processOrder(OrderService.java:45)
    at com.example.controller.OrderController.createOrder(OrderController.java:23)
    at sun.reflect.NativeMethodAccessorImpl.invoke0(Native Method)
    at java.lang.Thread.run(Thread.java:748)
```

`nid=0x3039` 就是我们之前转换的十六进制线程 ID。

### 2.5 第五分钟：分析堆栈

现在你有了高 CPU 线程的堆栈，分析它：

- **堆栈顶部是当前执行的代码**，这很可能就是问题所在
- **往上看调用链**，找到你写的业务代码
- **忽略 JDK 内部类**，那些通常是正常的

常见的问题堆栈模式：

```
# 问题代码在顶部
at com.example.service.BadService.infiniteLoop(BadService.java:20)

# 数据库查询在顶部
at com.mysql.jdbc.PreparedStatement.executeQuery(PreparedStatement.java:100)

# 正则匹配在顶部
at java.util.regex.Pattern.matcher(Pattern.java:1234)
```

### 2.6 第六~十分钟：使用 Arthas 深入分析

如果 `jstack` 还不够，用 Arthas 做更精细的分析。

```bash
# 1. 进入 Arthas
java -jar arthas-boot.jar

# 2. 查看 CPU 最高的 5 个线程
thread -n 5

# 3. trace 跟踪方法调用耗时
trace com.example.service.OrderService processOrder

# 4. 生成火焰图
profiler start --duration 30
profiler stop --format html -o /tmp/flame.html
```

## 三、火焰图：CPU 问题的终极诊断

火焰图（Flame Graph）是性能分析的终极武器，由 Netflix 的 Brendan Gregg 发明。它可以让你一眼看出**哪个代码路径消耗的 CPU 最多**。

### 3.1 火焰图原理

火焰图是一个倒置的堆栈图：

- **Y 轴**：调用栈深度
- **X 轴**：CPU 时间（宽度代表频率）
- **每个方框**：一个函数调用

```text
                           阔度代表调用频率
                                   ___
                                  |   |
                              ____|   |____
                             |               |
                         ____|               |____
                        |                           |
________________________|                           |________________
     main()                 foo()                      bar()
```

越宽的方块，消耗的 CPU 越多，越需要优化。

### 3.2 生成火焰图：async-profiler

async-profiler 是目前最好的 Java 性能采样工具，它基于 JVMTI 和 JVMTI Agent 实现，对应用影响极小。

#### 3.2.1 安装 async-profiler

```bash
# 下载
wget https://github.com/async-profiler/async-profiler/releases/download/v2.9/profiler.sh

# 添加执行权限
chmod +x profiler.sh
```

#### 3.2.2 采集 CPU 火焰图

```bash
# 开始采样（采集 60 秒）
./profiler.sh start -d 60 <PID>

# 或后台运行
./profiler.sh start -f /tmp/cpu.svg -d 60 <PID>

# 查看采样状态
./profiler.sh status <PID>

# 停止采样并生成火焰图
./profiler.sh stop <PID>
```

#### 3.2.3 生成内存分配火焰图

如果怀疑是内存分配导致的问题（如频繁 GC），可以采集内存分配火焰图：

```bash
./profiler.sh start -e alloc -d 60 <PID>
./profiler.sh stop -f /tmp/alloc.svg <PID>
```

:::warning 注意事项
- `alloc` 采样开销比 `cpu` 大，生产环境慎用
- 建议采样时间 30-60 秒
- 采样期间尽量模拟真实流量
:::

### 3.3 火焰图解读

拿到火焰图后，怎么看？

**第一步：找平顶山**

火焰图的顶部应该相对平坦。如果某个方块特别宽，说明这个函数消耗的 CPU 最多。

**第二步：看调用链**

从顶部往下看，找到你自己写的代码。JDK 内部类（如 `java.util.regex.*`）通常是正常的，但如果它占据了大量宽度，可能需要优化。

**第三步：定位热点**

```
    ┌─────────────────────────────────────┐
    │          processOrder()  35%        │
    │  ┌──────────────────────────────┐   │
    │  │      regexMatch()  30%       │   │ ← 这是热点
    │  └──────────────────────────────┘   │
    └─────────────────────────────────────┘
```

这个例子中，`processOrder` 调用 `regexMatch`，后者消耗了 30% 的 CPU。**优化正则，就是优化性能。**

## 四、常见场景的根因与解决方案

### 4.1 场景一：正则回溯

**特征**：

- CPU 突然打满
- 堆栈中有 `Pattern.*match` 或 `Regex.*`

**解决方案**：

```java
// 优化前：危险正则
String regex = "(a+)+b";

// 优化后：移除嵌套量词
String regex = "a+b";

// 或使用预编译
private static final Pattern PATTERN = Pattern.compile("a+b");
```

### 4.2 场景二：HashMap 碰撞攻击

**特征**：

- 请求量没有明显变化
- CPU 飙升和特定请求相关
- 堆栈中有 `HashMap.get` 或 `HashMap.put`

**解决方案**：

```java
// 方案一：使用 Hashtable（线程安全但性能差）
Hashtable<String, Object> map = new Hashtable<>();

// 方案二：使用 ConcurrentHashMap
ConcurrentHashMap<String, Object> map = new ConcurrentHashMap<>();

// 方案三：使用 TumbleMap（guava）
RangeMap<String, Object> map = new TreeRangeMap<>();

// 方案四：自定义 HashMap（扩容 + 红黑树）
// JDK 8+ 的 HashMap 已经优化了链表转红黑树的逻辑
// 但对于恶意攻击，需要在应用层做防护
public class SecureHashMap<K, V> {
    private final HashMap<K, V> map = new HashMap<>();
    private final Object mutex = new Object();

    public V get(K key) {
        // 简单的防御：限制操作时间
        long start = System.currentTimeMillis();
        try {
            return map.get(key);
        } finally {
            long cost = System.currentTimeMillis() - start;
            if (cost > 100) {
                log.warn("HashMap operation too slow: {}ms", cost);
            }
        }
    }
}
```

### 4.3 场景三：频繁 GC

**特征**：

- `jstat` 显示 GC 次数频繁
- `jstat -gcutil <PID> 1000` 输出中 `FGC` 持续增长

**解决方案**：

```bash
# 查看 GC 详情
jstat -gc <PID> 1000

# 开启 GC 日志分析
-XX:+PrintGCDetails -Xloggc:/tmp/gc.log
```

然后用 gceasy.io 分析 GC 日志。

### 4.4 场景四：JIT 编译

**特征**：

- 发生在服务启动初期
- CPU 高但随着时间推移逐渐下降
- 堆栈中有 JIT 相关的函数

**解决方案**：

```bash
# 预热后再压测
# 或使用 -XX:CICompilerCount 调整 JIT 线程数
java -XX:+TieredCompilation -XX:CICompilerCount=4 -jar app.jar
```

### 4.5 场景五：死循环

**特征**：

- CPU 100%，且不会降低
- 堆栈中能看到循环代码
- 通常是程序 bug

**解决方案**：

1. 如果无法立即修复代码，先扩容抗住流量
2. 联系开发紧急修复
3. 如果必须重启，做好现场保留

```bash
# 重启前必须保留现场
jstack <PID> > /tmp/jstack-final.log
heapdump --live /tmp/heap-final.hprof
```

## 五、面试实战

### 面试题一：线上 CPU 100%，如何排查？

**回答要点**：

```
我一般分五步走：

第一步：保留现场。
千万不能重启。先用 top -H 找到高 CPU 线程，记录线程 ID。

第二步：转换线程 ID。
printf '%x\n' <PID>，将十进制转十六进制。

第三步：导出线程栈。
jstack <PID> | grep <十六进制TID> -A 30。

第四步：分析堆栈。
看堆栈顶部的代码，确认是业务代码还是 JDK 代码。

第五步：定位根因。
如果是正则回溯，优化正则；如果是 HashMap 碰撞，使用安全容器；如果是死循环，紧急修复代码。
```

### 面试题二：死循环和死锁有什么区别？

**回答要点**：

| 区别 | 死循环 | 死锁 |
|---|---|---|
| CPU 表现 | CPU 100%，一个线程打满 | CPU 正常或略高，多线程忙等 |
| 线程状态 | RUNNABLE | BLOCKED / WAITING |
| 影响范围 | 单线程 | 多线程，系统整体阻塞 |
| 排查命令 | `jstack` 找 RUNNABLE | `jstack -l` 找 `<deadlock>` |
| 解决方式 | 修改代码 | 调整锁顺序、使用超时锁 |

**进阶回答**：

死锁通常有四个必要条件：**互斥、占有并等待、非抢占、循环等待**。打破任何一个条件就能解决死锁。实践中常用的方法是：

1. **固定锁顺序**：所有线程都按相同顺序获取锁
2. **锁超时**：使用 `tryLock(timeout)` 替代 `synchronized`
3. **减少锁范围**：尽量只锁必要的代码块

### 面试题三：如何避免正则回溯？

**回答要点**：

```
正则回溯的根本原因是嵌套量词和不恰当的贪婪匹配。

我的经验是：

1. 避免嵌套量词。
不要写 (a+)+b 这种，尽量拆解成正则。

2. 使用独占量词（possessive）或原子组。
Java 不直接支持，但可以用其他方式规避。

3. 性能测试。
上线前用长字符串测试正则性能。

4. 使用成熟的库。
比如 Apache Commons Validator 的正则都是经过优化的。
```

## 六、总结

CPU 100% 不是玄学，它背后一定有代码在疯狂执行。排查的关键是：

1. **不要重启**：保留现场是第一步
2. **找高 CPU 线程**：top -H + jstack 是基本功
3. **分析堆栈**：从顶部开始，找业务代码
4. **火焰图**：复杂问题用火焰图定位热点
5. **根因修复**：正则、HashMap、死循环是最常见的原因

记住这个排查顺序：

```
top -H  →  printf '%x\n'  →  jstack  →  分析堆栈  →  Arthas/profiler
```

十分钟内定位根因，不是梦。

> 下一步思考：定位到 CPU 100% 的根因后，如何彻底解决它？比如正则回溯，修改正则代码后，如何验证修复效果？是否需要做性能测试？
