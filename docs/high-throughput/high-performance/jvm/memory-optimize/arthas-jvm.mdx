# Arthas 实战：线上 JVM 问题诊断神器

凌晨三点，你被告警叫醒：「线上服务 CPU 100%，响应超时。」

你打开电脑，试图复现问题。但线上的数据量、并发量，在你的本地环境根本无法模拟。传统的本地调试方法——加日志、改代码、打包部署——至少需要半小时。

等你改完，服务可能早就崩溃了。

这是传统调试方法的局限性：**本地无法复现线上问题，而线上的问题又难以直接调试。**

Arthas（阿尔萨斯）就是为了解决这个问题而生的。它可以 attach 到正在运行的 JVM 进程，不需要改代码，不需要重启服务，直接在线上环境进行诊断和调试。

这篇文章，我会从 Arthas 的安装讲起，带你掌握核心命令，再通过真实案例演示如何在生产环境中快速定位问题。读完这篇，你就能把 Arthas 打造成你的线上诊断利器。

## 一、Arthas 是什么——阿里开源的诊断利器

### 1.1 Arthas 的定位

Arthas 是阿里巴巴开源的 Java 诊断工具，于 2018 年捐赠给 Apache 基金会，现在是 Apache 的顶级项目。

它的核心能力是：**不需要重启服务，不需要改代码，直接在线上环境进行 JVM 诊断**。

```

│                     Arthas 架构                               │
│                                                              │
│   ┌─────────────┐                    ┌─────────────┐       │
│   │   Web UI   │ ◄─── HTTP ----───► │ Arthas      │       │
│   │  (可选)    │                    │ Server      │       │
│   └─────────────┘                    └──────┬──────┘       │
│                                              │              │
│   ┌─────────────────────────────────────────┴───────────┐  │

│   │  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐ │  │
│   │  │ jvm     │  │ thread  │  │ watch   │  │ trace   │ │  │
│   │  │ 命令    │  │ 命令    │  │ 命令    │  │ 命令    │ │  │
│   │  └─────────┘  └─────────┘  └─────────┘  └─────────┘ │  │
```

### 1.2 Arthas vs 传统工具

| 能力 | 传统方式 | Arthas |
|---|---|---|
| 无侵入 | 需要加日志 | 不需要 |
| 不用重启 | 改代码需要重启 | 不需要 |
| 在线诊断 | 本地无法复现 | 直接在线排查 |
| 热修复 | 不支持 | 支持热更新类 |
| 命令行 | 各工具分散 | 统一命令行 |
| 性能开销 | 低 | 极低 |

### 1.3 Arthas 能做什么

Arthas 的功能非常全面：

- **JVM 监控**：内存、GC、类加载、线程
- **方法调用观测**：入参、返回值、异常
- **性能分析**：方法调用耗时、火焰图
- **热修复**：在线修改类，无需重启
- **反编译**：查看未知类的源码

## 二、安装与启动

### 2.1 快速安装

```bash
# 方式一：下载 arthas-boot.jar
curl -O https://arthas.aliyun.com/arthas-boot.jar
java -jar arthas-boot.jar

# 方式二：使用 as.sh 脚本（推荐）
curl -L https://arthas.aliyun.com/install.sh | sh
./as.sh

# 方式三：使用 Docker
docker run -it --rm \
    -v /tmp/arthas-agent:/root/arthas-agent \
    registry.cn-hangzhou.aliyuncs.com/arthas/arthas:latest
```

### 2.2 启动与 attach 进程

```bash
# 启动 arthas
java -jar arthas-boot.jar

# 输出示例
[INFO] Found existing java process, please choose one and input the serial number:
  [1]: 12345 com.example.MyApplication
  [2]: 23456 org.apache.catalina.startup.Bootstrap
```

输入进程编号，Arthas 会 attach 到对应的 JVM 进程。

```bash
# 或者命令行直接 attach
java -jar arthas-boot.jar <PID>

# attach 后，进入 Arthas 交互界面
```

### 2.3 Web UI 模式

```bash
# 启动时添加 --web-port 指定端口
java -jar arthas-boot.jar --web-port 8080
```

启动后，访问 http://localhost:8563 即可使用 Web UI。

### 2.4 离线使用

生产环境通常无法访问外网，需要离线安装：

```bash
# 在有网络的机器上下载
curl -O https://arthas.aliyun.com/arthas-packaging-zip

# 上传到生产服务器
scp arthas-packaging-zip server:/tmp/

# 解压
unzip arthas-packaging-zip
cd arthas

# 启动
java -jar arthas-boot.jar
```


- **MEMORY POOL**：各内存区详情

- **OPERATING SYSTEM**：系统信息

### 3.4 thread：线程分析

```bash
# 查看所有线程
thread

# 查看 CPU 使用最高的 5 个线程
thread -n 5

# 查看阻塞的线程
thread -b

# 查看某个线程的堆栈
thread <thread-id>

# 查看线程统计
thread -i 1000  # 采样 1000ms
```

**输出示例：thread -n 5**

```
Threads with highest CPU usage:
  Id      Name                      CPU%   Time(s)
  45      pool-1-thread-1           45.2   123.45
  123     http-nio-8080-exec-1      12.3   34.56
  67      pool-1-thread-3           8.9    23.45
```

**重点关注**：

- `CPU%`：CPU 占用百分比
- `WAITING`：等待中的线程
- `BLOCKED`：阻塞中的线程（可能是死锁）

### 3.5 heapdump：生成堆转储

```bash
# 生成堆转储到文件
heapdump

# 生成指定路径
heapdump /tmp/heap.hprof

# 只 dump 存活对象
heapdump --live /tmp/heap_live.hprof
```

生成后，用 MAT 分析：

```bash
# 下载到本地
scp server:/tmp/heap.hprof ./

# 用 MAT 打开
./ParseHeapDump.sh heap.hprof
```

### 3.6 gc：强制触发 GC

```bash
# 强制 Full GC
gc

# 输出

 Eden         256MB           128MB       128MB
 Survivor     32MB             64MB        32MB
 Old Gen      1024MB           400MB       400MB
```

:::warning 慎用

`gc` 会强制触发 Full GC，可能导致服务短暂停顿。生产环境慎用。

### 3.7 vmoption：在线修改 JVM 参数

```bash

# 查看某个选项

# 修改某个选项
```

- Boolean 类型：`-XX:+PrintGCDetails`

## 四、进阶命令：观测与追踪

### 4.1 jad：反编译类

当你看到一行日志里有个类名，但不知道它在哪、源码是什么时，`jad` 可以帮你反编译。

```bash
# 反编译整个类
jad com.example.service.UserService

# 只反编译某个方法
jad com.example.service.UserService getUserInfo

# 反编译并显示行号
jad -c <classloader-hash> com.example.service.UserService
```

### 4.2 watch：观测方法调用

`watch` 是 Arthas 最强大的功能之一，可以观测方法的入参、返回值、异常。

```bash
# 基本用法：观测方法入参和返回值
watch com.example.service.UserService getUserInfo "{params, returnObj}"

# 观测异常
watch com.example.service.UserService * "@java.lang.Exception"

# 观测入参、返回值、异常（全部观测）
watch com.example.service.UserService getUserInfo "{params, returnObj, throwExp}"

# 观测时只显示条件成立的结果（过滤）
watch com.example.service.UserService getUserInfo "{params, returnObj}" "params[0] > 100"
```

**参数说明**：

| 表达式 | 说明 |
|---|---|
| `{params}` | 入参 |
| `{returnObj}` | 返回值 |
| `{throwExp}` | 异常 |
| `{clazz}` | 类对象 |
| `{method}` | 方法对象 |
| `@xxx.xxx` | 访问静态字段 |

**展开层级**：

```bash
# -x 指定展开层级（默认 1）
watch com.example.service.UserService getUserInfo "{params, returnObj}" -x 2
```

### 4.3 trace：方法内部调用耗时

`trace` 可以追踪方法内部各步骤的调用耗时，帮助定位性能瓶颈。

```bash
# 追踪方法内部调用
trace com.example.service.OrderService createOrder

# 只显示耗时超过 100ms 的调用
trace com.example.service.OrderService createOrder '#cost > 100'

# 追踪方法内部多个调用节点
trace com.example.service.OrderService createOrder -n 5

# 输出带时间戳
trace com.example.service.OrderService createOrder --listenerId 1
```

**输出示例**：

```
`---[3.245ms] com.example.service.OrderService:createOrder()
    +---[1.234ms] com.example.service.InventoryService:checkStock()
    |   `---[0.567ms] com.example.repository.ProductMapper:selectById()
    +---[0.890ms] com.example.service.PaymentService:processPayment()
    +---[1.121ms] com.example.service.DeliveryService:scheduleDelivery()
```

这里可以看到 `createOrder` 方法耗时 3.245ms，其中：

- `checkStock` 耗时 1.234ms
- `processPayment` 耗时 0.890ms
- `scheduleDelivery` 耗时 1.121ms

### 4.4 stack：方法调用栈

`stack` 可以打印方法被调用的堆栈信息。

```bash
# 查看 getUserInfo 方法的调用栈
stack com.example.service.UserService getUserInfo

# 只显示符合条件的调用
stack com.example.service.UserService getUserInfo 'params[0] == "admin"'
```

### 4.5 monitor：方法调用统计

`monitor` 统计方法的调用次数、成功率和平均耗时。

```bash
# 统计 getUserInfo 方法
monitor -c 5 com.example.service.UserService getUserInfo

# 输出
 timestamp    class          method       total  success  fail  avg-rt(ms)  fail-rate
 2024-01-01   UserService    getUserInfo   1234    1230     4      12.34       0.32%
```

**参数说明**：

- `total`：总调用次数
- `success`：成功次数
- `fail`：失败次数
- `avg-rt`：平均响应时间
- `fail-rate`：失败率

## 五、性能分析：profiler 与火焰图

### 5.1 profiler：CPU 采样

`profiler` 是 Arthas 内置的 async-profiler，可以生成 CPU 火焰图。

```bash
# 启动 CPU 采样
profiler start

# 或者指定采样时间
profiler start --duration 30

# 查看采样状态
profiler status

# 停止采样并生成火焰图
profiler stop

# 生成 HTML 格式的火焰图
profiler stop --format html --file /tmp/frame.html
```

### 5.2 火焰图解读

火焰图（Flame Graph）是性能分析的可视化利器。

```
┌─────────────────────────────────────────────────────────────────────┐
│                           onApplicationReady() [100%]              │
│  ┌────────────────────────────────────────────────────────────────┐ │
│  │                    handleRequest() [85%]                        │ │
│  │  ┌──────────────────────────────────────────────────────────┐ │ │
│  │  │              processOrder() [60%]                        │ │ │
│  │  │  ┌────────────────────┐  ┌────────────────────────────┐  │ │ │
│  │  │  │  queryDatabase()  │  │  calculatePrice() [45%]    │  │ │ │
│  │  │  │  [15%]            │  │  ┌──────────────────────┐  │  │ │ │
│  │  │  │                   │  │  │  applyDiscount()    │  │  │ │ │
│  │  │  │                   │  │  │  [30%]               │  │  │ │ │
│  │  │  │                   │  │  └──────────────────────┘  │  │ │ │
│  │  │  └────────────────────┘  └────────────────────────────┘  │ │ │
│  │  └──────────────────────────────────────────────────────────┘ │ │
│  └────────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

**解读方法**：

- **宽度**：该函数占用的 CPU 比例，越宽越耗时
- **高度**：调用栈深度
- **顶部**：入口函数
- **从下往上看**：完整的调用链路

### 5.3 生成火焰图的完整流程

```bash
# 1. 启动采样（采样 60 秒）
profiler start --duration 60

# 2. 在采样期间，可以模拟业务请求
# ...

# 3. 停止采样，生成火焰图
profiler stop --format html --file /tmp/flamegraph.html

# 4. 下载到本地查看
scp server:/tmp/flamegraph.html ./

# 5. 用浏览器打开
open flamegraph.html
```

## 六、实战案例

### 6.1 案例一：接口响应慢——trace 定位慢 SQL

**症状**：用户反馈订单接口响应时间达到 3 秒。

**排查步骤**：

1. **首先，用 thread 找到慢请求的线程**
   ```bash
   thread -n 5
   ```
   发现 `http-nio-8080-exec-10` 线程 CPU 占用 80%，处于 `RUNNABLE` 状态。

2. **用 jstack 确认线程堆栈**
   ```bash
   thread 10
   ```
   看到线程卡在 `com.mysql.jdbc.PreparedStatement.executeQuery()`。

3. **用 trace 追踪方法调用**
   ```bash
   trace com.example.service.OrderService createOrder '#cost > 100'
   ```
   发现 `queryDatabase()` 方法耗时 2.5 秒。

4. **用 watch 观测 SQL 参数**
   ```bash
   watch com.example.repository.OrderMapper findByUserId "{params, returnObj}" -x 2
   ```
   发现查询条件缺少索引，导致全表扫描。

**根因**：SQL 查询缺少索引，全表扫描导致慢查询。

### 6.2 案例二：内存持续增长——heapdump + MAT

**症状**：服务运行 3 天后开始 OOM，内存持续增长。

**排查步骤**：

1. **监控内存趋势**
   ```bash
   memory
   ```
   发现 Old Gen 从 400MB 增长到 1.2GB。

2. **生成堆转储**
   ```bash
   heapdump /tmp/heap.hprof
   ```

3. **下载到本地，用 MAT 分析**
   - 打开 Leak Suspects
   - 发现 `com.example.cache.LocalCache` 占用 600MB
   - 用 Path to GC Roots 追溯引用链
   - 发现是静态 `HashMap` 未清理

4. **定位代码**
   ```bash
   jad com.example.cache.LocalCache
   ```
   发现缓存没有过期机制。

**根因**：本地缓存没有大小限制和过期机制，导致内存泄漏。

### 6.3 案例三：CPU 100%——profiler 火焰图

**症状**：CPU 使用率 100%，服务响应超时。

**排查步骤**：

1. **用 profiler 生成 CPU 火焰图**
   ```bash
   profiler start --duration 30
   # 等待 30 秒采样
   profiler stop --format html --file /tmp/flame.html
   ```

2. **下载火焰图分析**
   - 火焰图显示 `calculateDiscount()` 方法占用 60% CPU
   - 点击展开，发现正则表达式回溯导致 CPU 飙升

3. **定位代码**
   ```bash
   jad com.example.service.DiscountService calculateDiscount
   ```
   发现使用了复杂正则：`^(a+)+b$`

4. **检查正则表达式**
   攻击者构造了大量 `a` 开头的字符串，导致正则回溯。

**根因**：正则表达式存在灾难性回溯（Catastrophic Backtracking）。

## 八、面试追问

### 8.1 问题一：Arthas 的原理是什么？

**参考答案**：

Arthas 的核心原理是 **Java Agent**（Java 代理）：

1. Arthas 通过 `attach` 机制连接到目标 JVM 进程
2. 加载一个 Java Agent（`arthas-agent.jar`）
3. Agent 利用 JVM Tool Interface（JTI）提供的功能：
   - ** Instrumentation API**：在运行时修改字节码，注入观测代码
   - ** JVMTI**：虚拟机提供的调试接口
4. 通过 JVMTI，Arthas 可以：
   - 拦截方法调用（watch、trace）
   - 获取线程信息
   - 生成 Heap Dump
5. 所有操作都在目标 JVM 内执行，不影响目标进程

### 8.2 问题二：Arthas 有什么使用限制？

**参考答案**：

1. **JDK 版本限制**：需要 JDK 6+，建议 JDK 8+
2. **权限限制**：需要目标进程的用户权限
3. **网络限制**：需要能访问目标服务器（SSH 或端口）
4. **性能影响**：虽然很小，但观测代码会带来一定开销
5. **安全考虑**：生产环境使用需要谨慎，建议只读操作

### 8.3 问题三：Arthas 和 Arthas MutiTenant 的区别？

**参考答案**：

- **Arthas**：单机版，每个 JVM 进程需要单独启动一个 Arthas
- **Arthas Tunnel**：通过 Tunnel Server 管理多台机器的 Arthas Agent，通过 Web UI 统一管控
- **Arthas Scout**（企业版）：阿里云提供的 Arthas 在线版本，开箱即用

## 总结

Arthas 是线上问题排查的神器，它的核心理念是「**无侵入、无需重启、在线诊断**」。

关键要点：

- **安装启动**：`arthas-boot.jar` 或 `as.sh`
- **核心命令**：`dashboard`、`jvm`、`thread`、`heapdump`、`gc`
- **进阶观测**：`watch`、`trace`、`stack`、`monitor`
- **性能分析**：`profiler` 生成火焰图
- **实战应用**：trace 定位慢 SQL、heapdump + MAT 排查内存泄漏、profiler 排查 CPU 高

记住：Arthas 只是工具，核心还是分析思路。掌握了 Arthas 的命令，还要理解 JVM 原理和常见的性能问题模式，才能快速定位根因。
