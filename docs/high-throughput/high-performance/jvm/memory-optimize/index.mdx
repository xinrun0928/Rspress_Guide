# 内存优化与 OOM 排查

## 模块定位

OOM（OutOfMemoryError）是生产环境的头号杀手之一。

大多数 OOM 不是因为真的内存不够，而是因为**内存泄漏**——对象无法被 GC 回收，越积越多，最终耗尽内存。

OOM 的可怕之处在于：它不会预警，只会突然降临。而且很多 OOM 不是 JVM 的问题，而是代码的问题——一个忘记 close() 的 Connection，一个持续增长没有清理的 Map，就足以让整个服务崩溃。

本模块帮你建立完整的 OOM 排查知识体系，从**堆内存 OOM** 到**元空间 OOM** 再到**直接内存 OOM**，覆盖所有常见 OOM 类型。

## OOM 排查的通用流程

```
告警触发 → 获取堆转储（jmap/自动导出）
→ 分析 Heap Dump（MAT Leak Suspects）
→ 定位问题对象 → 分析 GC Roots
→ 找到引用链 → 修复代码
```

## 学习路径

```
heap-oom（堆 OOM）→ memory-tools（分析工具）
→ memory-leak（泄漏定位）→ metaspace-oom（元空间 OOM）
→ direct-memory-oom（直接内存 OOM）→ cpu-high（CPU 飙升）
→ arthas-jvm（Arthas 工具）→ tuning-summary（调优总结）
```

## 面试考察

OOM 排查是 Java 面试中最考察实战能力的题型：

- 「线上 OOM 了，你如何排查？」
- 「内存泄漏怎么定位？用什么工具？」
- 「CPU 突然 100%，怎么排查？」
- 「Netty 堆外内存泄漏怎么处理？」

这些问题没有标准答案，关键是你能否说清楚排查的思路和每一步用什么工具。
